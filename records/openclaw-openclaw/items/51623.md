## Summary

Keep open: current main still lacks durable pending-followup replay, but this PR has blocking replay, trust, and durability defects plus missing real behavior proof before it can merge.

## What This Changes

The PR adds shutdown persistence of non-empty followup queues to `pending-messages.json`, startup replay through system events, and unit/integration proof scaffolding.

## Best Possible Solution

Adopt one canonical restart-recovery design that persists structured queued or inbound work and replays it through normal dispatch with explicit untrusted semantics and idempotent durable consumption.

## Reproduction Assessment

Yes, at source level. Current main stores accepted followups in a process-local queue, rejects drain-window enqueues with `GatewayDrainingError`, and has no durable pending-message replay path.

## Solution Assessment

No. Persisting queued work is the right direction, but this patch should replay structured work through followup or inbound dispatch rather than flattening user text into trusted system events and deleting recovery state before processing succeeds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Replay recovered messages through normal dispatch:** `src/gateway/server-pending-messages.ts:44-47`
  - body: Recovered followups are flattened into a system event instead of re-entering the followup or inbound dispatch path. That changes routing and wakeup semantics, and can leave the original queued user work unprocessed until unrelated later activity.
  - confidence: 0.9
- **[P1] Mark replayed user text untrusted:** `src/gateway/server-pending-messages.ts:44-47`
  - body: `item.prompt` is channel user text, but this call omits `trusted: false`. Current system events default to trusted and later render as `System:` lines, so a queued user message can gain system-context authority after restart.
  - confidence: 0.95
- **[P1] Make restart snapshots idempotent before replay:** `src/gateway/server-close.ts:132`
  - body: After writing the snapshot, shutdown leaves `FOLLOWUP_QUEUES` live. In the in-process restart fallback, the same queue can survive while startup also consumes the persisted snapshot, producing duplicate recovery unless the persisted items are cleared or replay is otherwise idempotent.
  - confidence: 0.86
- **[P2] Keep recovery files until work is durable:** `src/auto-reply/reply/queue/persist.ts:94-95`
  - body: `consumePersistedQueues()` deletes the file before replayed work has been delivered through a durable processing path. A second restart after consumption but before the next agent turn can still lose the recovered messages this feature is meant to preserve.
  - confidence: 0.86
- **[P2] Remove the hard five-minute recovery expiry:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards persisted queues older than five minutes, so slow reboots, manual downtime, or repeated failed restarts still lose queued work. Recovery should be based on durable consumption or an explicit safer policy instead.
  - confidence: 0.84
- **[P3] Validate persistedAt before age checks:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: `parsed.persistedAt` is cast but not checked. Missing or non-numeric values make the age calculation `NaN`, bypassing the intended stale-file guard for malformed persisted files.
  - confidence: 0.78
