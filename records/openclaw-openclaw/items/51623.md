## Summary

Keep open: current main still lacks durable followup-queue recovery, but this PR is not merge-ready because replay elevates user text into trusted system events, can duplicate recovery on in-process restarts, drops snapshots after a short TTL, and overlaps with open PR #51803.

## What This Changes

The PR adds shutdown persistence for non-empty followup queues, startup replay as system events, targeted tests, and an end-to-end proof script for gateway restart recovery.

## Best Possible Solution

Land one canonical fix for #51620 that persists queued inbound work durably, replays user-originated text through an explicitly untrusted path, is idempotent across process and in-process restarts, targets current startup/shutdown files, avoids fixed short-TTL data loss, and includes the required changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows followup queues are in-memory and shutdown/startup has no durable pending-followup recovery path; the PR-specific trust issue is source-reproducible by tracing `item.prompt` into `enqueueSystemEvent` without `trusted: false`.

## Solution Assessment

No. The Phase 1 direction is useful, but the mergeable solution should be rebased onto current startup code, make replay untrusted and idempotent, avoid the fixed five-minute loss window, and be chosen against #51803 or a replacement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark replayed prompts as untrusted:** `src/gateway/server-pending-messages.ts:44-47`
  - body: `item.prompt` is channel user text, but this call omits `trusted: false`. Current system events default to trusted, so after restart a queued user message is rendered as `System:` instead of `System (untrusted):`, crossing the prompt-trust boundary.
  - confidence: 0.94
- **[P1] Make persisted replay idempotent on in-process restart:** `src/gateway/server-close.ts:130-132`
  - body: After writing a snapshot, the PR leaves `FOLLOWUP_QUEUES` live. In an in-process restart, the old queue can still survive while startup also replays the persisted file, so the same messages can be surfaced twice unless successfully persisted items are cleared or replay is made idempotent.
  - confidence: 0.86
- **[P2] Remove the five-minute snapshot expiry:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards persisted queues older than five minutes, so slow reboots, manual downtime, or repeated failed restarts still lose the queued work this feature is meant to preserve. Use durable consumption or an explicit safer recovery policy instead.
  - confidence: 0.84
- **[P3] Validate persistedAt before age checks:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: `parsed.persistedAt` is cast but not checked. Missing or non-numeric values make the age calculation `NaN`, bypassing the intended stale-file guard for malformed persisted files.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/gateway/server-close.ts:130-132`
  - body: This changes user-visible gateway restart behavior, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a single-line changelog entry for user-facing fixes before merge.
  - confidence: 0.82
