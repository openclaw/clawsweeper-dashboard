## Summary

Keep open: the restart message-loss bug is real, but this PR is not merge-ready because it is conflict-marked against current main, lacks real behavior proof, and replays user prompts through trusted system-event context with durability gaps.

## What This Changes

The branch adds followup-queue persistence on gateway shutdown, startup replay through system events, gateway lifecycle hooks, and restart-persistence tests/scripts.

## Best Possible Solution

Land one maintainer-approved durable recovery path that persists accepted followups and drain-window arrivals, replays them through typed untrusted queue or dispatch semantics with idempotency, and proves exact-once live delivery.

## Reproduction Assessment

Yes, at source level: current main stores accepted followups only in `FOLLOWUP_QUEUES` and rejects drain-window work with `GatewayDrainingError` without durable replay. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. Persisting pending work is the right direction, but this PR's system-event replay is not the safest maintainable fix because it changes trust, routing, wakeup, batching, and durability semantics.

Full review comments:

- **[P1] Replay recovered messages through normal dispatch:** `src/gateway/server-pending-messages.ts:44-47`
  - body: Recovered followups are flattened into `enqueueSystemEvent()` instead of re-entering the followup or inbound dispatch path. That bypasses normal origin routing, queue policy, wakeup, and exactly-once delivery behavior, so recovered user work can sit idle or reply through the wrong surface.
  - confidence: 0.92
- **[P1] Preserve user-message trust during replay:** `src/gateway/server-pending-messages.ts:44-47`
  - body: `item.prompt` is channel user text, but this call does not set `forceSenderIsOwnerFalse` or use an untrusted dispatch path. Current system events drain as `System:` lines, so a queued user message can gain system-context authority after restart.
  - confidence: 0.96
- **[P2] Keep recovery files until replay is durable:** `src/auto-reply/reply/queue/persist.ts:94-95`
  - body: `consumePersistedQueues()` deletes the file before recovered work has been durably requeued or processed. A second restart after consumption but before the next agent turn can still lose the messages this feature is meant to preserve.
  - confidence: 0.88
- **[P2] Remove the hard recovery expiry:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards persisted queues older than five minutes, so slow reboots, manual downtime, or repeated failed restarts still lose queued work. Recovery should be tied to durable consumption or an explicit maintainer-approved policy.
  - confidence: 0.82
- **[P2] Persist summarized queue state:** `src/auto-reply/reply/queue/persist.ts:30-31`
  - body: The serializer only accepts queue `items`, but `FollowupQueueState` can also hold summarized overflow in `droppedCount`, `summaryLines`, and `lastRun`. Capped summarize queues can therefore still lose pending work on restart.
  - confidence: 0.8
