## Summary

Keep open: current main still has the restart-loss path, and this external PR is a plausible implementation candidate, but the proposed replay design has blocking trust, routing, idempotency, and durability problems plus missing real behavior proof.

## What This Changes

The PR adds shutdown persistence for pending followup queues, startup replay via system events, tests, and a proof script for gateway restart recovery.

## Best Possible Solution

Land one canonical restart-recovery design that durably persists queued work, replays through the normal followup or inbound dispatch path, preserves untrusted user semantics and routing metadata, and coordinates the sibling PR with the open restart-loss issue.

## Reproduction Assessment

Yes, at source level. Current main stores FOLLOWUP_QUEUES in process memory and rejects new command enqueues after markGatewayDraining, with no pending-message persistence or durable inbound replay path.

## Solution Assessment

No. Persisting queued work is the right direction, but this PR should replay structured work through the normal followup or inbound dispatch path with explicit untrusted semantics and idempotent durable consumption instead of flattening user prompts into trusted system events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Replay recovered messages through the normal reply path:** `src/gateway/server-pending-messages.ts:44-47`
  - body: Recovered followups are flattened into a system event, so replay does not run the normal followup drain, preserve per-item routing semantics, or necessarily start processing until unrelated later activity. Recovered work should be re-queued or redispatched through the existing reply path.
  - confidence: 0.9
- **[P1] Mark replayed prompts as untrusted:** `src/gateway/server-pending-messages.ts:44-47`
  - body: item.prompt is channel user text, but this call omits trusted: false. Current system events default to trusted and render as System, so a queued user message can gain system-event authority after restart.
  - confidence: 0.95
- **[P1] Make persisted replay idempotent after in-process restart:** `src/gateway/server-close.ts:132`
  - body: After writing the snapshot, shutdown leaves FOLLOWUP_QUEUES live. In an in-process restart, the same queue can survive while startup also replays the persisted file, producing duplicate recovery unless successfully persisted items are cleared or replay is otherwise idempotent.
  - confidence: 0.86
- **[P2] Keep replay files durable until processing succeeds:** `src/auto-reply/reply/queue/persist.ts:94-95`
  - body: consumePersistedQueues deletes the file before replay has been delivered through a durable processing path. A second restart after consumption but before the next agent turn can still lose the recovered messages this feature is meant to preserve.
  - confidence: 0.86
- **[P2] Remove the hard five-minute snapshot expiry:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards persisted queues older than five minutes, so slow reboots, manual downtime, or repeated failed restarts still lose queued work. Use durable consumption or an explicit safer recovery policy instead.
  - confidence: 0.84
- **[P3] Validate persistedAt before age checks:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: parsed.persistedAt is cast but not checked. Missing or non-numeric values make the age calculation NaN, bypassing the intended stale-file guard for malformed persisted files.
  - confidence: 0.78
