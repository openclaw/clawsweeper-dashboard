## Summary

Keep open for maintainer review, but this PR is not merge-ready: it uses a JSON sidecar for runtime queue state, drops persisted messages after a fixed 5-minute window, and replays user text through trusted system-event prompt lines. The paired same-author issue is still open, and the active SQLite-backed replacement path is not merged.

## What This Changes

The branch adds JSON-file persistence and startup replay for in-memory followup queues, plus focused tests and a synthetic proof script.

## Best Possible Solution

Rebuild the fix as a SQLite-backed durable followup queue recovery path on the current gateway runtime-services surface, replaying messages as untrusted inbound/followup work and preserving valid pending entries until they are delivered or explicitly failed.

## Reproduction Assessment

Yes, at source level: current main stores followup queue state in process memory and marks the gateway draining before restart, so pending or drain-window messages can be lost without durable handoff. I did not run a live channel restart in this read-only review.

## Solution Assessment

No. The branch is a plausible phase-one mitigation, but the maintainable fix should use the shared SQLite state store, current startup recovery surfaces, and untrusted replay semantics rather than JSON sidecars and trusted system events.
