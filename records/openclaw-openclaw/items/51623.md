## Summary

Keep open. The underlying gateway restart loss is still relevant on current main, but this PR is not merge-ready because it replays user-originated text as trusted system context, can duplicate recovery after in-process restarts, uses a lossy five-minute snapshot expiry, and overlaps another open candidate for the same issue.

## What This Changes

The PR branch adds disk persistence for non-empty followup queues, startup replay through system events, shutdown/startup hooks, and persistence/replay tests plus a proof script.

## Best Possible Solution

Keep the durable pending-inbound recovery work in core, but route replay through normal untrusted inbound handling or explicitly untrusted system events, make snapshot consumption idempotent across process and in-process restarts, avoid short fixed TTL loss, and coordinate this PR with #51620 and #51803 before landing one canonical fix.

## Reproduction Assessment

Yes. Current main shows followup queues are memory-only, shutdown clears gateway run state without persisting them, and restart draining rejects new enqueues with GatewayDrainingError; the PR-specific trust issue is reproducible by following the added replay path into enqueueSystemEvent's trusted default.

## Solution Assessment

No. The PR is a useful Phase 1 direction, but this is not the safest maintainable implementation until replay is untrusted, snapshot handling is idempotent, the TTL policy is fixed, and the branch is updated for the current gateway startup structure.

Full review comments:

- **[P1] Mark replayed prompts as untrusted:** `src/gateway/server-pending-messages.ts:44-47`
  - body: item.prompt is channel user text, but this call omits trusted: false. enqueueSystemEvent defaults to trusted, so after restart a queued user message is rendered as System context instead of System (untrusted).
  - confidence: 0.94
- **[P1] Clear followup queues after a successful snapshot:** `src/gateway/server-close.ts:132`
  - body: After writing the restart snapshot, FOLLOWUP_QUEUES remains live. In in-process restart paths, the old queue can still drain while startup also replays the persisted file, producing duplicate recovery for the same messages.
  - confidence: 0.86
- **[P2] Preserve valid snapshots beyond five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader drops every snapshot older than five minutes, which reintroduces message loss for slow reboots, manual downtime, or repeated failed restarts. A restart persistence feature should not discard valid queued work on such a short fixed TTL.
  - confidence: 0.84
- **[P3] Validate persistedAt before staleness math:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: parsed.persistedAt is cast but not checked. Missing or non-numeric values make the age calculation NaN, so malformed files bypass the intended staleness guard.
  - confidence: 0.78
