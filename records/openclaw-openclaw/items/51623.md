## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Store queue recovery in SQLite, not a JSON sidecar:** `src/auto-reply/reply/queue/persist.ts:67`
  - body: This writes OpenClaw-owned runtime queue state to `state/pending-messages.json`. The repo's state contract says queues and runtime state belong in SQLite, with legacy sidecars handled by doctor/migration code, so this would create a parallel persistence path without an upgrade owner.
  - confidence: 0.9
- **[P1] Keep replayed prompts out of system-event authority:** `src/gateway/server-pending-messages.ts:32-38`
  - body: `item.prompt` is user/channel text, but this path embeds it in an `enqueueSystemEvent` payload that is rendered as trusted `System:` prompt context. Restore it as untrusted user content or a normal queued followup instead of elevating the message text after restart.
  - confidence: 0.91
- **[P2] Do not expire recoverable messages after five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: A fixed five-minute TTL means a slow reboot, manual downtime, or repeated failed restart drops the saved queue even though the feature exists to prevent silent message loss. Persisted pending messages need a safer recovery policy or an explicit maintainer-approved discard contract.
  - confidence: 0.83
- **[P2] Consume persisted followups atomically for in-process restarts:** `src/gateway/server-close.ts:128-132`
  - body: After writing the snapshot, the in-memory followup map remains live; current gateway restarts can fall back to an in-process restart, so the old queue may still drain while startup also replays the file. Clear, transfer, or otherwise make the snapshot consume exactly once.
  - confidence: 0.78
