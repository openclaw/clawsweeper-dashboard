## Summary

Keep open. Current main still has the gateway restart message-loss gap, but this PR should not merge as-is because replay is routed through trusted system events, can duplicate or lose recovered work around restart races, uses a five-minute durability cutoff, lacks real behavior proof, and overlaps the still-open recovery work at https://github.com/openclaw/openclaw/pull/51803.

## What This Changes

The PR adds disk persistence for non-empty followup queues on gateway shutdown, startup replay as system events, and tests plus a proof script for restart recovery.

## Best Possible Solution

Land one canonical gateway restart recovery design that persists queued work durably, replays user-originated content through an explicitly untrusted and idempotent normal dispatch/followup path, and then resolves https://github.com/openclaw/openclaw/issues/51620 alongside the overlapping PR.

## Reproduction Assessment

Yes, at source level. Current main keeps `FOLLOWUP_QUEUES` process-local and has no startup replay path, and the PR-specific trust issue is reproducible by tracing `item.prompt` into `enqueueSystemEvent` without `trusted: false`.

## Solution Assessment

No. The Phase 1 direction is useful, but this implementation should be reworked to use a durable normal replay path, preserve untrusted user boundaries, avoid duplicate/lost recovery on restart races, and remove the hard five-minute data-loss window.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Replay recovered items through the normal reply path:** `src/gateway/server-pending-messages.ts:44-47`
  - body: Recovered messages are flattened into a system event, so replay does not run the normal followup drain, does not preserve per-item routing semantics, and may wait for unrelated later activity before the agent sees the recovered text. Persisted followups should be re-queued or redispatched through the existing reply path instead of only being summarized as a system event.
  - confidence: 0.9
- **[P1] Mark replayed prompts as untrusted:** `src/gateway/server-pending-messages.ts:44-47`
  - body: `item.prompt` is channel user text, but this call omits `trusted: false`. Current system events default to trusted and render as `System:`, so after restart a queued user message can be elevated above the normal untrusted inbound boundary.
  - confidence: 0.95
- **[P1] Make persisted replay idempotent on in-process restart:** `src/gateway/server-close.ts:130-132`
  - body: After writing the snapshot, shutdown leaves `FOLLOWUP_QUEUES` live. In an in-process restart, the same queue can survive while startup also replays the persisted file, producing duplicate recovery unless successfully persisted items are cleared or replay is otherwise idempotent.
  - confidence: 0.86
- **[P2] Keep replay files durable until processing succeeds:** `src/auto-reply/reply/queue/persist.ts:88-89`
  - body: `consumePersistedQueues` deletes the file before replay has been delivered through a durable processing path. A second restart after consumption but before the next agent turn can still lose the recovered messages this feature is meant to preserve.
  - confidence: 0.86
- **[P2] Remove the five-minute snapshot expiry:** `src/auto-reply/reply/queue/persist.ts:111-116`
  - body: The loader discards persisted queues older than five minutes, so slow reboots, manual downtime, or repeated failed restarts still lose queued work. Use durable consumption or an explicit safer recovery policy instead.
  - confidence: 0.84
- **[P3] Validate persistedAt before age checks:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: `parsed.persistedAt` is cast but not checked. Missing or non-numeric values make the age calculation `NaN`, bypassing the intended stale-file guard for malformed persisted files.
  - confidence: 0.78
