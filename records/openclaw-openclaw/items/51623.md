## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Store queue recovery state in SQLite, not a sidecar:** `src/auto-reply/reply/queue/persist.ts:67`
  - body: This writes OpenClaw-owned runtime queue state to `state/pending-messages.json`. The current storage contract puts runtime state and queues in SQLite-backed stores, so this sidecar would add a second durable state path with no migration or doctor owner.
  - confidence: 0.87
- **[P1] Keep replayed user text out of trusted system events:** `src/gateway/server-pending-messages.ts:31-39`
  - body: `item.prompt` comes from user messages, but the replay path interpolates it directly into a system event. Those events are consumed as `System:` prompt context, so a crafted queued message can be upgraded from user input into trusted restart instructions after recovery.
  - confidence: 0.9
- **[P1] Rebase the startup hook onto current gateway startup code:** `src/gateway/server-startup.ts:193-197`
  - body: This schedules replay from `server-startup.ts`, but current main no longer has that file and routes post-ready services through newer gateway startup/runtime modules. The integration point needs to be rebuilt before recovery can land.
  - confidence: 0.83
- **[P2] Do not discard pending messages after five minutes:** `src/auto-reply/reply/queue/persist.ts:112-117`
  - body: The loader drops the persisted queue when downtime exceeds five minutes, so a slow reboot or manual restart can still lose the exact messages this feature is supposed to preserve. Recovery should be explicit and durable, not tied to a short wall-clock TTL.
  - confidence: 0.84
