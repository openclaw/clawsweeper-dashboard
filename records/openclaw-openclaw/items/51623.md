## Summary

Keep open: current main still lacks durable restart recovery for pending followup work, but this PR should not merge as-is because it replays user prompts as trusted system events, has durability/idempotency gaps, lacks after-fix real behavior proof, and overlaps another open restart-recovery PR.

## What This Changes

The PR adds pending-message persistence for followup queues, startup replay through system events, shutdown/startup hooks, tests, and a proof script.

## Best Possible Solution

Land one canonical restart-recovery design that persists structured queued work and replays through the normal followup or inbound dispatch path with explicit untrusted user semantics and durable, idempotent consumption.

## Reproduction Assessment

Yes, at source level. Current main stores FOLLOWUP_QUEUES in process memory and rejects new command enqueues after gateway draining, with no durable pending-message replay path present.

## Solution Assessment

No. Persisting queued work is the right direction, but this PR should replay structured work through the normal followup or inbound dispatch path instead of flattening user prompts into trusted system events and deleting recovery files before processing succeeds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Replay recovered messages through normal dispatch:** `src/gateway/server-pending-messages.ts:44-47`
  - body: Recovered followups are flattened into a system event instead of going back through the followup or inbound dispatch path. That changes routing semantics and may leave the recovered user turn idle until unrelated later activity rather than processing it as the original queued work.
  - confidence: 0.9
- **[P1] Preserve user-message trust on replay:** `src/gateway/server-pending-messages.ts:44-47`
  - body: item.prompt is channel user text, but this enqueueSystemEvent call omits trusted: false. Current system events default to trusted and later render as System lines, so a queued user message can gain system-context authority after restart.
  - confidence: 0.95
- **[P1] Make restart snapshots idempotent before replay:** `src/gateway/server-close.ts:129`
  - body: After writing the snapshot, shutdown leaves FOLLOWUP_QUEUES live. In an in-process restart fallback, that same in-memory queue can survive while startup also consumes the persisted snapshot, producing duplicate recovery unless persisted items are cleared or replay is otherwise idempotent.
  - confidence: 0.86
- **[P2] Keep replay files until recovered work is durable:** `src/auto-reply/reply/queue/persist.ts:94-95`
  - body: consumePersistedQueues deletes the file before replay has been delivered through a durable processing path. A second restart after consumption but before the next agent turn can still lose the recovered messages this feature is meant to preserve.
  - confidence: 0.86
- **[P2] Do not discard restart recovery after five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards persisted queues older than five minutes, so slow reboots, manual downtime, or repeated failed restarts still lose queued work. Recovery should be based on durable consumption or an explicit safer policy instead.
  - confidence: 0.84
- **[P3] Validate persistedAt before age checks:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: parsed.persistedAt is cast but not checked. Missing or non-numeric values make the age calculation NaN, bypassing the intended stale-file guard for malformed persisted files.
  - confidence: 0.78
