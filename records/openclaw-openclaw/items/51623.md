## Summary

Keep open: current main still lacks pending-followup persistence, and this PR addresses a real source-reproducible restart-loss path, but the proposed replay path is unsafe because it converts queued user text into trusted system events and bypasses normal followup/inbound dispatch semantics.

## What This Changes

The PR adds pending followup queue persistence, startup replay as system events, shutdown/startup hooks, tests, and a proof script for gateway restarts.

## Best Possible Solution

Land one canonical restart-recovery design that persists accepted queued work durably, replays it through normal followup or inbound dispatch paths, preserves untrusted user semantics, and coordinates the remaining drain-window and channel-ack phases through the linked issue.

## Reproduction Assessment

Yes, at source level. Current main stores followup queue items only in process memory and rejects new command enqueues during gateway drain without a durable replay path.

## Solution Assessment

No. Persisting queued work is the right class of fix, but this PR should replay through durable normal dispatch or followup processing with explicit untrusted semantics and idempotency instead of flattening prompts into trusted system events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Replay recovered items through the normal reply path:** `src/gateway/server-pending-messages.ts:44-47`
  - body: Recovered followups are flattened into a system event, so replay does not run the normal followup drain, preserve per-item routing semantics, or reliably start processing without later activity. Recovered work should be re-queued or redispatched through the existing reply path.
  - confidence: 0.9
- **[P1] Mark replayed prompts as untrusted:** `src/gateway/server-pending-messages.ts:44-47`
  - body: item.prompt is channel user text, but this call omits trusted: false. Current system events default to trusted and render as System lines, so a queued user message can gain system-event authority after restart.
  - confidence: 0.95
- **[P1] Make persisted replay idempotent on in-process restart:** `src/gateway/server-close.ts:130`
  - body: After writing the snapshot, shutdown leaves FOLLOWUP_QUEUES live. In an in-process restart, the same queue can survive while startup also replays the persisted file, producing duplicate recovery unless successfully persisted items are cleared or replay is otherwise idempotent.
  - confidence: 0.86
- **[P2] Keep replay files durable until processing succeeds:** `src/auto-reply/reply/queue/persist.ts:94-95`
  - body: consumePersistedQueues deletes the file before replay has been delivered through a durable processing path. A second restart after consumption but before the next agent turn can still lose the recovered messages this feature is meant to preserve.
  - confidence: 0.86
- **[P2] Remove the hard five-minute snapshot expiry:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards persisted queues older than five minutes, so slow reboots, manual downtime, or repeated failed restarts still lose queued work. Use durable consumption or an explicit safer recovery policy instead.
  - confidence: 0.84
- **[P3] Validate persistedAt before age checks:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: parsed.persistedAt is cast but not checked. Missing or non-numeric values make the age calculation NaN, bypassing the intended stale-file guard for malformed persisted files.
  - confidence: 0.78
