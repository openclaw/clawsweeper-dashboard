## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Store queue state in SQLite, not JSON:** `src/auto-reply/reply/queue/persist.ts:63`
  - body: This writes OpenClaw-owned runtime queue state to `state/pending-messages.json`. The repo storage contract says queues and runtime state should use SQLite-backed stores, with old sidecars handled by doctor/migration code, so this would create a parallel source of truth with no upgrade owner.
  - confidence: 0.9
- **[P1] Keep replayed user text out of System events:** `src/gateway/server-pending-messages.ts:31-35`
  - body: `item.prompt` comes from queued channel/user messages, but this code embeds it into an event that current main later formats as `System:` lines. Replay should preserve the text through a normal untrusted message path or wrap it as untrusted prompt data, not elevate it into trusted-looking recovery context.
  - confidence: 0.88
- **[P2] Do not expire restart recovery after five minutes:** `src/auto-reply/reply/queue/persist.ts:106-109`
  - body: The fixed five-minute stale check discards the only persisted copy after slow reboot, manual downtime, or repeated failed restart, which reintroduces the message loss this PR is trying to fix. Recovery needs a safer lifecycle rule, explicit operator cleanup, or a durable queue policy instead of a hard short TTL.
  - confidence: 0.82
- **[P2] Resolve in-process restart duplication before replay:** `src/gateway/server-close.ts:125-130`
  - body: The close hook writes `FOLLOWUP_QUEUES` but does not remove or mark those in-memory items. On SIGUSR1 in-process restarts, process-global queue state can survive into the next gateway iteration while the JSON snapshot is also replayed, so the same pending message can be surfaced twice.
  - confidence: 0.76
