## Summary

Keep open: this PR targets a real P1 restart message-loss bug, but the branch is not merge-ready because it adds JSON sidecar runtime state, replays user text as trusted system events, is now conflicting with current gateway startup code, and lacks real behavior proof.

## What This Changes

This PR adds JSON-file persistence for in-memory follow-up queues during gateway shutdown and replays them as system events after startup, with tests and a synthetic proof script.

## Best Possible Solution

Use the SQLite-backed durable follow-up queue design already being pursued in https://github.com/openclaw/openclaw/pull/82572, or a refreshed equivalent that preserves routing metadata, treats replayed user text as untrusted, and includes current-head real transport proof.

## Reproduction Assessment

Yes, at source level: current main keeps follow-up queues in process memory and has no durable follow-up queue persistence in the shutdown/startup path. I did not run a live channel restart repro in this read-only review.

## Solution Assessment

No. Persisting runtime queue state to a JSON sidecar and replaying user text through trusted system events is not the best fix; the durable recovery path should use SQLite-owned state, preserve routing/execution metadata, and replay user content through an untrusted boundary.

  - body: `item.prompt` is channel/user input, but this builds an `enqueueSystemEvent` payload; current system events are rendered as trusted `System:` lines on the next turn.   - body: This writes OpenClaw-owned runtime queue state to `state/pending-messages.json`. The repository storage contract requires runtime queues/state in SQLite with doctor-owned migration, so this adds an unmigrated parallel source of truth for prompts and routing state.
  - confidence: 0.88
- **[P1] Rebase replay scheduling onto current gateway startup:** `src/gateway/server-startup.ts:193-197`
  - body: The branch schedules replay in `src/gateway/server-startup.ts`, but current main has moved post-ready startup work into `server-startup-post-attach.ts` and `server-runtime-services.ts`, and GitHub reports this branch as conflicting. The replay hook has to be rebuilt on the current startup lifecycle before review can validate it.
  - confidence: 0.86
- **[P2] Do not discard restart recovery after five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: A restart that takes longer than five minutes silently drops the only persisted queue snapshot, which reintroduces the message-loss bug during manual downtime or repeated failed starts. Recovery should be durable until consumed or governed by a maintainer-approved expiry policy.
  - confidence: 0.82
