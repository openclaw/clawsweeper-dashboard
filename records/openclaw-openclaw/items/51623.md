## Summary

Keep open: current main still has the source-level restart-loss path, but this PR is not merge-ready because it replays queued user text through system events, changes trust/routing/durability semantics, lacks real behavior proof, and overlaps https://github.com/openclaw/openclaw/pull/51803 for https://github.com/openclaw/openclaw/issues/51620.

## What This Changes

The PR adds shutdown-time serialization of non-empty followup queues to a state JSON file, startup replay via system events, focused tests, an integration test, and a proof script.

## Best Possible Solution

Land one maintainer-approved durable restart recovery path that persists accepted followups and drain-window arrivals, replays them through typed untrusted dispatch/requeue contracts, and consumes recovery data only after safe processing succeeds.

## Reproduction Assessment

Yes, source-level. Current main keeps accepted followups in process-local FOLLOWUP_QUEUES, rejects drain-window enqueues with GatewayDrainingError, and has no durable pending-message replay path; I did not run a live gateway restart in this read-only pass.

## Solution Assessment

No. Persisting pending work is the right direction, but this PR's system-event replay changes trust, routing, wakeup, batching, and durability semantics; replay through existing typed followup/inbound contracts is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Replay recovered messages through normal dispatch:** `src/gateway/server-pending-messages.ts:43-47`
  - body: Recovered followups are flattened into enqueueSystemEvent() instead of re-entering the followup or inbound dispatch path. That bypasses origin routing and wakeup semantics, and can leave recovered user work idle until unrelated later activity.
  - confidence: 0.91
- **[P1] Preserve user-message trust during replay:** `src/gateway/server-pending-messages.ts:43-47`
  - body: item.prompt is channel user text, but this call omits an explicit untrusted/owner-downgrade option. Current system events render as System lines, so queued user text can gain system-context authority after restart.
  - confidence: 0.96
- **[P1] Make restart snapshots idempotent before replay:** `src/gateway/server-close.ts:132`
  - body: After writing the snapshot, shutdown leaves FOLLOWUP_QUEUES live. In an in-process restart fallback, those same items can survive while startup also consumes the persisted snapshot, producing duplicate recovery unless replay is idempotent or persisted items are cleared safely.
  - confidence: 0.86
- **[P2] Keep recovery files until replay is durable:** `src/auto-reply/reply/queue/persist.ts:94-95`
  - body: consumePersistedQueues() deletes the file before replayed work has been delivered through a durable processing path. A second restart after consumption but before the next agent turn can still lose the recovered messages this feature is meant to preserve.
  - confidence: 0.87
- **[P2] Remove the hard recovery expiry:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards persisted queues older than five minutes, so slow reboots, manual downtime, or repeated failed restarts still lose queued work. Recovery should be based on durable consumption or an explicit safer policy instead.
  - confidence: 0.82
- **[P2] Persist overflow summary state:** `src/auto-reply/reply/queue/persist.ts:31-36`
  - body: The serializer only accepts queue.items and skips queues with no live items, but FollowupQueueState can also hold droppedCount, summaryLines, and summarySources. Capped queues can therefore lose summarized pending messages on restart.
  - confidence: 0.8
- **[P3] Validate persistedAt before age checks:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: parsed.persistedAt is cast but not checked. Missing or non-numeric values make the age calculation NaN, bypassing the intended stale-file guard for malformed persisted files.
  - confidence: 0.78
