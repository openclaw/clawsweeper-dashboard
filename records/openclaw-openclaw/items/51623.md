## Summary

Keep open, but not merge-ready. The PR is a useful Phase 1 attempt for the linked gateway message-loss issue, but live state is conflicting, real behavior proof is still missing, and source review shows blocking trust, durability, and replay-contract problems.

## What This Changes

The branch adds a pending-message JSON snapshot for followup queues on gateway close, startup replay through system events, and restart-persistence tests plus a proof script.

## Best Possible Solution

Land one maintainer-approved durable recovery path that restores followup queue state or routes recovered user messages through normal untrusted inbound/followup dispatch with idempotency, current startup hooks, and exact-once proof.

## Reproduction Assessment

Yes at source level: current main keeps accepted followups only in FOLLOWUP_QUEUES and rejects drain-window work with GatewayDrainingError, so those states have no durable replay path. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. Persisting accepted followups is the right direction, but this PR's notification-style system-event replay is not the safest durable recovery contract; queue restore or normal untrusted dispatch with idempotency and current startup integration is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route recovered messages through untrusted dispatch:** `src/gateway/server-pending-messages.ts:44-47`
  - body: `item.prompt` is channel user text, but replay sends it through `enqueueSystemEvent()`. Current system events drain as `System:` prompt lines, so a crafted queued user message can gain system-context authority after restart instead of following the normal untrusted inbound path.
  - confidence: 0.96
- **[P1] Replay recovered work through the queue contract:** `src/gateway/server-pending-messages.ts:44-47`
  - body: Flattening recovered followups into one system event does not restore the original queue item, wakeup, route, queue policy, or exactly-once processing semantics. Recovered messages can sit until a later turn and be handled as a notification instead of the original user work.
  - confidence: 0.9
- **[P2] Keep recovery snapshots until replay is durable:** `src/auto-reply/reply/queue/persist.ts:94-95`
  - body: `consumePersistedQueues()` unlinks the file before any recovered work has been durably requeued or processed. A crash or second restart after consumption but before the next agent turn can still lose the messages this feature is meant to preserve.
  - confidence: 0.88
- **[P2] Avoid discarding valid recovery after five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The hard five-minute staleness guard drops queued work after slow reboots, manual downtime, or repeated failed restarts. Recovery should be tied to durable consumption or an explicit maintainer-approved expiry policy.
  - confidence: 0.82
- **[P2] Persist complete queue state:** `src/auto-reply/reply/queue/persist.ts:11-14`
  - body: `PersistedQueueEntry` stores only `items`, but `FollowupQueueState` can also hold summarized overflow in `droppedCount`, `summaryLines`, `summarySources`, and `lastRun`. Capped summarize queues can still lose pending work across restart.
  - confidence: 0.8
- **[P2] Use private atomic writes for pending snapshots:** `src/auto-reply/reply/queue/persist.ts:67`
  - body: `pending-messages.json` can contain user prompts and routing metadata, but this direct `fs.writeFile()` relies on the process umask and is not atomic. Use the repo's private atomic JSON/durable queue helpers for this state.
  - confidence: 0.84
