## Best Possible Solution

Keep the canonical issue open and continue with a current-main SQLite recovery design that restores queued messages through an untrusted follow-up/user path, includes doctor/migration coverage, and has real channel proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep replayed user text untrusted:** `src/gateway/server-pending-messages.ts:32-33`
  - body: `item.prompt` is channel-user content, but the replay path embeds it in a system event. Those events are formatted as trusted `System:` prompt lines, so a queued message can gain instruction priority after restart; replay through an untrusted follow-up/user path instead.
  - confidence: 0.94
- **[P1] Store queue recovery in SQLite, not a JSON sidecar:** `src/auto-reply/reply/queue/persist.ts:9-23`
  - body: This adds `state/pending-messages.json` as steady-state runtime queue storage. OpenClaw runtime state and queues are SQLite-owned now, with legacy sidecars limited to doctor/migration code, so this sidecar would create a second persistence contract and upgrade path.
  - confidence: 0.91
- **[P2] Do not expire restart recovery after five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The fixed five-minute cutoff discards persisted queue entries whenever restart downtime is longer than that window, including manual downtime or repeated failed restarts. That reintroduces the message-loss behavior this PR is meant to fix.
  - confidence: 0.86
