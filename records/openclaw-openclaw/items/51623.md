## Summary

Keep open: the user-visible restart message-loss problem is real, but this PR is not merge-ready because it persists queue runtime state as a JSON sidecar, replays user-controlled text through trusted system-event prompt lines, lacks real restart/channel proof, and is currently conflicting with main.

## What This Changes

The PR adds JSON-file persistence for in-memory followup queues on gateway shutdown and replays them on startup as system events, plus mocked persistence/replay tests and a synthetic proof script.

## Best Possible Solution

Land a rebased SQLite-backed queue persistence implementation that rehydrates through normal followup dispatch or explicitly untrusted replay semantics, proves exactly-once live delivery after restart, and keeps the canonical issue as the tracker until that merges.

## Reproduction Assessment

Yes, source-reproducible: current main keeps followup queues in a process-global Map and shutdown clears runtime chat state, so a process restart can lose queued followups. I did not run a live channel restart in this read-only review.

## Solution Assessment

No. Persisting restart queue state is the right problem, but this PR uses the wrong storage boundary and replay trust model; the maintainable fix should be SQLite-backed and replay through normal or explicitly untrusted delivery semantics.

Full review comments:

- **[P1] Move queue persistence out of JSON sidecar state:** `src/auto-reply/reply/queue/persist.ts:67`
  - body: This writes OpenClaw-owned runtime queue state to `state/pending-messages.json`. The repository state contract keeps runtime queues and caches in SQLite-backed stores, with old sidecars handled by doctor or migration code, so this creates a parallel source of truth with no upgrade owner.
  - confidence: 0.91
- **[P1] Replay queued text as untrusted user content:** `src/gateway/server-pending-messages.ts:32-46`
  - body: `item.prompt` comes from user or channel messages, but this path embeds it in a recovery summary and queues that summary as a system event. Current main renders system events as `System:` prompt lines, so a crafted queued message can be elevated after restart instead of being replayed through normal untrusted/user-message handling.
  - confidence: 0.9
- **[P2] Do not expire restart recovery after five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader drops all persisted queue entries older than five minutes, which means a slow update, manual downtime, or repeated restart failure still silently loses exactly the messages this PR is meant to preserve. Durable queue recovery should either restore until consumed or use an explicit product policy with visible handling.
  - confidence: 0.78
