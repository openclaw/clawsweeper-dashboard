## Summary

Keep open: current main still lacks durable followup-queue persistence, but this PR should not merge as-is because it replays user-originated text as trusted system context, has non-idempotent snapshot handling, drops valid snapshots after a short TTL, and overlaps the still-open canonical issue and alternate PR.

## What This Changes

The PR adds followup-queue snapshot persistence to `state/pending-messages.json`, startup replay through system events, shutdown/startup hooks, and restart-persistence tests plus a proof script.

## Best Possible Solution

Land one canonical fix for #51620 that persists queued inbound work durably, replays it through an explicitly untrusted path, is idempotent across process and in-process restarts, avoids short fixed-TTL loss, targets current startup/shutdown files, and includes the required changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows followup queues are process-local and shutdown/startup has no durable pending-message recovery path; the PR-specific trust issue is also source-reproducible by tracing `item.prompt` into `enqueueSystemEvent` without `trusted: false`.

## Solution Assessment

No. The PR is a useful Phase 1 direction, but it is not the best mergeable solution until replay is untrusted, snapshot consumption is idempotent, the TTL policy is fixed, the branch is rebased onto the current gateway startup structure, and maintainers choose it over #51803.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark replayed prompts as untrusted:** `src/gateway/server-pending-messages.ts:44-47`
  - body: `item.prompt` is channel user text, but this call omits `trusted: false`. `enqueueSystemEvent` defaults to trusted, so after restart a queued user message is rendered as `System:` instead of `System (untrusted):`.
  - confidence: 0.94
- **[P1] Clear queues only after a saved snapshot:** `src/gateway/server-close.ts:132`
  - body: After this best-effort write, `FOLLOWUP_QUEUES` remains live. In an in-process restart, the old queue can still drain while startup also replays the persisted file, surfacing the same messages twice; clear or mark only successfully persisted items idempotently.
  - confidence: 0.86
- **[P2] Do not drop valid restart snapshots after five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader unconditionally discards snapshots older than five minutes, so slow reboots, manual downtime, or repeated failed restarts lose the same queued work this feature is meant to preserve. Use durable consumption or an explicit safer recovery policy instead.
  - confidence: 0.84
- **[P3] Validate persistedAt before using it:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: `parsed.persistedAt` is cast but not checked. Missing or non-numeric values make the age calculation `NaN`, so malformed files bypass the intended stale-file guard.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/gateway/server-close.ts:132`
  - body: This changes user-visible gateway restart behavior, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing fixes before merge.
  - confidence: 0.82
