## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Replay restored prompts through untrusted user-followup flow:** `src/gateway/server-pending-messages.ts:32-33`
  - body: `item.prompt` is user/channel text, but this code embeds it in an event that current `drainFormattedSystemEvents()` renders as `System:` on the next prompt. That elevates an untrusted queued message into trusted system context after restart; rehydrate it as a normal follow-up/user turn or wrap it with an explicit untrusted-content envelope before model-visible replay.
  - confidence: 0.96
- **[P1] Actually dispatch or wake restored followups:** `src/gateway/server-pending-messages.ts:44-47`
  - body: The replay path only calls `enqueueSystemEvent`; it never re-enqueues the saved follow-up run, requests a heartbeat, or carries the saved delivery context. A restart can therefore still leave the user message unanswered until an unrelated turn, and the response may not target the original channel/thread.
  - confidence: 0.94
- **[P1] Remove the hard five-minute loss window:** `src/auto-reply/reply/queue/persist.ts:112-116`
  - body: Persisted queues older than five minutes are discarded, but slow restarts, manual downtime, or repeated crash recovery are exactly when durable recovery is needed. This reintroduces silent message loss for valid queued work; use an explicit durable processed/expired policy instead of dropping by wall-clock age.
  - confidence: 0.93
- **[P2] Store restart queues in SQLite instead of a JSON sidecar:** `src/auto-reply/reply/queue/persist.ts:67`
  - body: This writes OpenClaw runtime queue state to `state/pending-messages.json`. Current repository policy and adjacent runtime state use SQLite for persisted state, so this creates a second recovery store with no migration/doctor path and upgrade ambiguity; put the queue entries in the shared state DB.
  - confidence: 0.88
