## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Keep replayed user text out of system events:** `src/gateway/server-pending-messages.ts:32-33`
  - body: `item.prompt` is channel/user input, but this builds it into text passed to `enqueueSystemEvent`; current prompt assembly renders those entries as `System:` lines, so a queued user message can be replayed as trusted-looking restart context. Rehydrate the followup through the normal untrusted queue/inbound path or quote the message in an explicitly untrusted block.
  - confidence: 0.96
- **[P1] Store pending followups in SQLite, not a sidecar:** `src/auto-reply/reply/queue/persist.ts:67`
  - body: This writes OpenClaw-owned runtime queue state to `state/pending-messages.json`. Queue/session recovery state belongs in the shared SQLite state boundary with an explicit migration/doctor owner; a new JSON sidecar creates a competing persistent data contract and upgrade debt.
  - confidence: 0.94
- **[P2] Do not expire saved messages after five minutes:** `src/auto-reply/reply/queue/persist.ts:112-116`
  - body: The fixed five-minute cutoff means a slow reboot, manual downtime, or repeated crash loop discards the messages this feature is supposed to preserve. Recovery should keep the durable entries until they are replayed, explicitly abandoned by policy, or migrated through a tested retention contract.
  - confidence: 0.9
- **[P2] Avoid duplicate replay on in-process restarts:** `src/gateway/server-close.ts:132`
  - body: After a successful snapshot write, this leaves the process-global `FOLLOWUP_QUEUES` map intact. OpenClaw can fall back to in-process restart, so the original live queue may remain while startup also replays the persisted snapshot; the handoff needs an adopt/clear/exactly-once boundary.
  - confidence: 0.84
