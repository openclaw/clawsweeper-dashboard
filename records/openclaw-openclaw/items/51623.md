## Summary

Keep open: the PR targets a real restart-loss bug that current main still has, but the proposed replay path has blocking correctness/security issues and lacks required real gateway restart proof.

## What This Changes

The PR branch adds durable followup-queue persistence before gateway shutdown, startup replay of persisted queued messages as system events, focused tests, an integration test, and a proof script.

## Best Possible Solution

Land one maintainer-approved restart-recovery design that persists structured queued or inbound work, replays it through normal followup or inbound dispatch with explicit untrusted semantics, and consumes recovery files only after the work is safely requeued or processed.

## Reproduction Assessment

Yes, at source level. Current main keeps accepted followups in process-local FOLLOWUP_QUEUES, rejects drain-window enqueues with GatewayDrainingError, and has no durable pending-message replay path.

## Solution Assessment

No. Persisting pending work is the right direction, but this PR should not flatten user text into trusted system events or delete recovery state before durable dispatch succeeds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Replay recovered messages through normal dispatch:** `src/gateway/server-pending-messages.ts:44-47`
  - body: Recovered followups are flattened into a system event instead of re-entering the followup or inbound dispatch path. That bypasses origin routing and wakeup semantics, and can leave recovered user work idle until unrelated later activity.
  - confidence: 0.91
- **[P1] Preserve user-message trust during replay:** `src/gateway/server-pending-messages.ts:44-47`
  - body: item.prompt is channel user text, but this call omits trusted: false. Current system events default to trusted and later render as System lines, so queued user text can gain system-context authority after restart.
  - confidence: 0.96
- **[P1] Make restart snapshots idempotent before replay:** `src/gateway/server-close.ts:132`
  - body: After writing the snapshot, shutdown leaves FOLLOWUP_QUEUES live. In the in-process restart path, those same items can survive while startup also consumes the persisted snapshot, producing duplicate recovery unless the persisted items are cleared or replay is idempotent.
  - confidence: 0.86
- **[P2] Keep recovery files until replay is durable:** `src/auto-reply/reply/queue/persist.ts:94-95`
  - body: consumePersistedQueues deletes the file before replayed work has been delivered through a durable processing path. A second restart after consumption but before the next agent turn can still lose the recovered messages this feature is meant to preserve.
  - confidence: 0.87
- **[P2] Remove the hard five-minute recovery expiry:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards persisted queues older than five minutes, so slow reboots, manual downtime, or repeated failed restarts still lose queued work. Recovery should be based on durable consumption or an explicit safer policy instead.
  - confidence: 0.82
- **[P3] Validate persistedAt before age checks:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: parsed.persistedAt is cast but not checked. Missing or non-numeric values make the age calculation NaN, bypassing the intended stale-file guard for malformed persisted files.
  - confidence: 0.78
- **[P3] Use an explicit persisted run type:** `src/auto-reply/reply/queue/persist.ts:11-14`
  - body: PersistedQueueEntry exposes items as full FollowupRun values even though config and skillsSnapshot are stripped before JSON serialization. A persisted type that omits those fields would prevent replay code from accidentally treating recovered items as live runs.
  - confidence: 0.74
