## Summary

Keep open, but this PR is not merge-ready. It targets a real P1 restart message-loss problem, yet the branch persists queue state through a JSON sidecar, replays user text as trusted system-event prompt content, can duplicate queued work on in-process restarts, and lacks after-fix real behavior proof.

## What This Changes

This PR serializes non-empty followup queues to `state/pending-messages.json` during gateway shutdown and replays them as session system events after startup, with unit/integration tests and a synthetic proof script.

## Best Possible Solution

Use a SQLite-backed queue persistence/recovery design that restores messages through normal queue drain semantics with explicit untrusted-context handling, upgrade tests, and real Telegram/gateway proof.

## Solution Assessment

No. The PR is a plausible phase-1 mitigation, but the best fix is a SQLite-backed restore that preserves queue/runtime semantics and treats replayed user content as untrusted input.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep replay text out of trusted system events:** `src/gateway/server-pending-messages.ts:32-38`
  - body: `item.prompt` comes from channel user messages, but this replay path embeds it in a system event that later renders as trusted `System:` prompt content. Recovery should restore the user turn through the normal untrusted queue/message path, or quote it in an explicitly untrusted block, so restart recovery cannot elevate a crafted user message into system-priority instructions.
  - confidence: 0.91
- **[P1] Avoid duplicating queues on in-process restart:** `src/gateway/server-close.ts:125-130`
  - body: The close handler writes a snapshot but leaves `FOLLOWUP_QUEUES` live. OpenClaw can fall back to an in-process restart, where global singleton queues survive; startup then replays the disk snapshot as well, so the same queued messages can be processed or reported twice unless the live queue and persisted snapshot are reconciled atomically.
  - confidence: 0.86
- **[P2] Persist queue state in shared SQLite:** `src/auto-reply/reply/queue/persist.ts:67`
  - body: This writes OpenClaw-owned runtime queue state to `state/pending-messages.json`, creating a new sidecar store for message text and queue metadata. Current repo policy and the existing state model require runtime state like queues to live in shared SQLite with explicit migration/doctor behavior, not a parallel JSON file.
  - confidence: 0.82
- **[P2] Do not expire restart recovery after five minutes:** `src/auto-reply/reply/queue/persist.ts:107-110`
  - body: A hard 5-minute age check reintroduces the message loss this PR is meant to prevent whenever a restart, upgrade, or manual outage lasts longer than that window. Recovery should either keep durable queued work until it is drained or use a deliberate product policy with visible discard semantics.
  - confidence: 0.83
