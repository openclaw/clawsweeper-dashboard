## Summary

Keep open: this PR targets a real P1 restart message-loss bug, but the submitted branch is not merge-ready because it adds JSON sidecar runtime state, replays user text through trusted system events, lacks real restart proof, and is dirty against current main.

## What This Changes

The PR serializes non-empty `FOLLOWUP_QUEUES` entries to `state/pending-messages.json` on gateway shutdown and replays them after startup as session system events, with mocked persistence/replay tests and a synthetic proof script.

## Best Possible Solution

Land a rebased SQLite/doctor-backed queue persistence implementation that restores through normal followup or explicitly untrusted delivery semantics, proves exact-current-head real restart recovery, and keeps the linked issue as canonical until that lands.

## Reproduction Assessment

Yes, at source level: current main keeps followup queues in a process-global map and enqueue only pushes to memory, so a full process restart drops queued followups. Current main also rejects command-lane enqueues during restart draining with `GatewayDrainingError`.

## Solution Assessment

No. Persisting followup queues is the right problem, but this PR uses the wrong storage boundary and replay trust model; the maintainable path should be SQLite-backed and restore through normal or clearly untrusted delivery semantics.

  - body: This writes OpenClaw-owned runtime queue state to `state/pending-messages.json`. Runtime queues belong in SQLite-backed state with any legacy sidecar handling owned by doctor/migration code, so this creates a parallel source of truth with no upgrade owner.

  - body: `item.prompt` comes from user/channel messages, but this path embeds it in a recovery summary and queues that summary as a system event.   - confidence: 0.9
- **[P2] Do not expire restart recovery after five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader drops all persisted queue entries older than five minutes, which means slow updates, manual downtime, or repeated restart failures can still silently lose exactly the messages this PR is meant to preserve.
