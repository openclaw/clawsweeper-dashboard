## Summary

Keep open: current main and the latest release still have the reported wake-false media delivery asymmetry, and the linked closing PR is an open implementation candidate with unresolved proof and delivery-policy review rather than a merged fix.

## What This Changes

The issue requests a fix for successful generated-media completions where `wakeTaskCompletion` returns `false` without throwing and the outer handler marks the task blocked without attempting direct media fallback.

## Best Possible Solution

Land a reason-aware fix through the linked PR or a successor that retries direct media only for recoverable delivery-loss false returns while preserving requester-abandoned stops.

## Reproduction Assessment

Yes. Source inspection shows current main reaches a blocked terminal result after `wakeTaskCompletion` returns `false` without entering the direct fallback used by the throw path; I did not run a live media-generation repro.

## Solution Assessment

Yes for keeping the issue open. A blanket outer boolean fallback is not the best final fix because it loses delivery reasons; the maintainable path should be reason-aware or return a richer wake result.
