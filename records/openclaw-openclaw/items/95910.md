## Summary

Keep this issue open: current main and v2026.6.10 still have the wake-false media completion asymmetry, and open linked PRs now own the implementation and proof path rather than providing a merged fix.

## What This Changes

The issue asks OpenClaw to recover generated-media completions when `wakeTaskCompletion()` returns `false` after successful generation instead of bypassing direct media fallback.

## Best Possible Solution

Land a reason-aware fallback such as https://github.com/openclaw/openclaw/pull/97275 after proof or maintainer acceptance, use the proof-harness work to validate it, and keep the broader generated-media completion tracker open for separate mechanisms.

## Reproduction Assessment

Yes. Source inspection on current main shows the successful wake false path marks completion delivery blocked without entering the direct fallback used by the thrown wake path; I did not run a live media-generation repro.

## Solution Assessment

Yes for the remaining direction: a reason-aware scheduler fallback is the narrow maintainable fix because it preserves requester-abandoned and generic-false no-send behavior while recovering allowlisted media-loss results.
