## Summary

Keep open: current main and v2026.6.9 still have the reported wake-false asymmetry, but existing tests also say some false-return reasons must not direct-deliver, so this needs a bounded maintainer delivery-contract decision before a fix PR is safe.

## What This Changes

The issue requests a follow-up fix for successful media generation where `wakeTaskCompletion` returns false without throwing and the outer handler marks delivery unconfirmed without trying direct media delivery.

## Best Possible Solution

Define a narrow set of non-terminal successful-media delivery failures that may use direct media fallback, preserve the no-direct behavior for requester abandonment unless maintainers explicitly change that contract, and add focused regression coverage around the chosen reasons.

## Reproduction Assessment

Yes at source level: current main has separate false and throw handling after successful generation, and only the throw path calls direct delivery. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Unclear. Symmetrizing every false return is not currently the best fix because tests intentionally keep direct delivery disabled for requester abandonment and generic handoff failures; maintainers should choose the narrower retry contract first.
