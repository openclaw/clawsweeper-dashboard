Keep this issue open: current main and the latest release still have the reported wake-false media completion asymmetry, and https://github.com/openclaw/openclaw/pull/95945 is an open implementation candidate rather than a merged fix.

## What This Changes

The issue requests a follow-up fix for successful media generation when `wakeTaskCompletion` returns false without throwing, so generated media is not silently lost before direct fallback is attempted.

## Best Possible Solution

Land or revise the linked PR so recovery is reason-aware and retries direct media only for true delivery-loss false returns while preserving requester-abandoned stops.

## Reproduction Assessment

Yes, at source level: construct a successful media completion where `wakeTaskCompletion` returns false and current main marks the task blocked without calling the direct fallback used by the thrown-wake branch. I did not run a live media-generation channel repro in this read-only pass.

## Solution Assessment

Unclear for the exact proposed shape: symmetrizing recovery is the right area, but the implementation should stay reason-aware so policy-stop false returns such as `requester_abandoned` do not direct-send late media.
