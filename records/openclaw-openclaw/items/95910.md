## Summary

Keep this issue open: current main and the latest release still have the wake-false media completion asymmetry, and the linked closing PR is open rather than merged.

## What This Changes

The issue asks to retry direct media delivery when successful media generation reaches `wakeTaskCompletion(false)` instead of marking the completion blocked without sending the artifact.

## Best Possible Solution

Review and either land or revise the linked PR so direct media fallback covers true delivery-loss false returns while preserving the existing requester-abandoned no-send rule.

## Reproduction Assessment

Yes by source inspection: current main and `v2026.6.10` reach a blocked terminal result after `wakeTaskCompletion(false)` without entering the direct fallback used by the throw path. I did not run a live media-generation repro.

## Solution Assessment

Unclear for the submitted fix path: symmetrizing delivery recovery is the right narrow area, but the final implementation should be reason-aware so it does not reopen abandoned requester sessions.
