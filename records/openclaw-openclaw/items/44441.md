## Summary

Keep this PR open for fixes rather than cleanup-close: the docs-only change is focused, but the proposed watchdog still does not match current health-command behavior and misses profile-aware service targeting. Current main does not already include an equivalent macOS watchdog runbook, and the related broader watchdog PRs are still open rather than merged superseders.

## What This Changes

The PR adds an optional macOS LaunchAgent watchdog recovery runbook for Gateway service health and links it from the update guide.

## Best Possible Solution

Keep the runbook docs-only and optional, but change the example so it either uses a supported failing predicate for the advertised unhealthy state or explicitly documents reachability-only recovery, while preserving profile/config/state environment for named services.

## Reproduction Assessment

Yes. Source inspection shows the PR gates recovery on `openclaw health --json`, while current `healthCommand` intentionally treats successful Gateway reachability as a successful command even when channel issues are reported.

## Solution Assessment

No, not as written. The narrow fix is to align the documented watchdog trigger and environment with current CLI/service contracts instead of presenting `health --json` as an alive-but-unhealthy detector.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a predicate that fails for unhealthy Gateway states:** `docs/platforms/mac/bundled-gateway.md:107`
  - body: The runbook promises recovery for a process that is alive but unhealthy, but the example exits whenever `openclaw health --json` can reach the Gateway. Current `healthCommand` treats successful Gateway reachability as success and reports channel degradation without setting a non-zero exit, so the recovery block will not run for the advertised degraded-but-responsive case.
  - confidence: 0.89
- **[P2] Preserve profile environment in the watchdog example:** `docs/platforms/mac/bundled-gateway.md:133-137`
  - body: The page already documents profile-specific LaunchAgents, but this watchdog plist only sets PATH and the script does not pass `--profile`. Named-profile installs will probe and restart the default Gateway instead of the intended service unless the example carries `OPENCLAW_PROFILE` and related config/state env or explicitly limits itself to the default profile.
  - confidence: 0.84
