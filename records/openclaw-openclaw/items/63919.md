## Summary

Keep open: current main still does not materialize coding primitives for direct Gateway tool invocation, and this PR's current shape addresses the earlier hook, leak, loopback, and default-deny concerns. It remains a Gateway execution/data surface expansion that needs maintainer sign-off rather than conservative cleanup.

## What This Changes

The PR adds a raw coding-tool factory, wires core coding primitives into Gateway direct tool resolution, expands the HTTP deny-list docs/changelog, and adds focused Gateway regression tests.

## Best Possible Solution

Land this after maintainer sign-off that direct Gateway tool invocation should expose these coding primitives on the shared invoke path, with the existing deny-list and policy gates kept intact.

## Reproduction Assessment

Yes. Source inspection on current main shows the direct invoke resolver only materializes createOpenClawTools, so a policy-allowed coding primitive such as read has no resolver path until this PR adds the raw coding-tool factory.

## Solution Assessment

Yes for the code shape, subject to maintainer approval. The patch reuses the existing resolver, policy, hook, and deny-list pipeline, avoids double hook execution, and keeps MCP loopback from expanding.
