## Summary

Keep open: the PR is still useful but not merge-ready. It adds a new security-sensitive HTTP coding-tool surface, still needs maintainer scope approval and real behavior proof, and the narrower active exec-only alternative at https://github.com/openclaw/openclaw/pull/79106 has not merged or fully superseded the broader read/write/edit/process work.

## What This Changes

The PR adds an unwrapped raw coding-tool factory and wires `read` plus opt-in `write`/`edit`/`exec`/`process` into Gateway HTTP `/tools/invoke`, with deny-list, docs, changelog, and tests.

## Best Possible Solution

Have Gateway/security owners choose the approved surface; if broad coding tools are accepted, rebase with owner-only gating for exec-capable tools, trusted-proxy negative coverage, updated docs/tests, and real `/tools/invoke` proof.

## Reproduction Assessment

Yes for the source-level feature gap: current main resolves Gateway HTTP tools from `createOpenClawTools` and does not materialize coding primitives for `/tools/invoke`. This is feature work rather than a broken existing behavior with a runtime bug repro.

## Solution Assessment

No, not as currently proposed. The raw factory approach is broad and leaves allowlisted exec-capable tools without owner-only HTTP gating; a safer path is a maintainer-approved broad surface with equivalent admin gating or a narrower exec-only registration like https://github.com/openclaw/openclaw/pull/79106.
