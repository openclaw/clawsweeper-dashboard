## Summary

Keep open. The PR is meaningful but should not merge as-is because the broad HTTP coding-tool wiring exposes allowlisted `exec`/`process` without preserving owner-only semantics for trusted-proxy non-admin callers, and the external PR still lacks inspectable real behavior proof.

## What This Changes

The PR adds a raw coding-tool factory, wires core coding tools into Gateway HTTP `/tools/invoke`, expands deny-list/docs coverage, and adds regression tests plus a changelog entry.

## Best Possible Solution

Choose the intended scope first: either land a narrow owner-only per-tool registration path or make the broad coding-tool registration preserve the same admin/owner boundary for execution tools, then require redacted live HTTP proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the Gateway HTTP resolver builds from `createOpenClawTools` only, so `createOpenClawCodingTools` primitives are not available through `/tools/invoke`; I did not run a live server in this read-only review.

## Solution Assessment

No. The broad raw-factory approach is plausible, but it currently misses owner-only semantics for execution tools; a safer solution is narrow per-tool HTTP registration or explicit owner-only wrappers before broad exposure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve owner-only gating for execution tools:** `src/gateway/tool-resolution.ts:149-156`
  - body: This appends raw coding tools to the HTTP pool, including `exec` and `process`, but those tools are not owner-only today. Because `/tools/invoke` is reachable with `operator.write` and only treats shared-secret/admin callers as owners, a trusted-proxy caller without `operator.admin` can invoke allowlisted command execution. Mark execution tools owner-only or use owner-only HTTP wrappers before exposing them here.
  - confidence: 0.86
