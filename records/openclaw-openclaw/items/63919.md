## Summary

Keep open: the patch is not ready because allowlisted `exec`/`process` can bypass the owner-only boundary for trusted non-admin HTTP callers, the raw factory's plugin-suppression flag is still inert, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a raw coding-tool factory, wires coding tools into Gateway HTTP `/tools/invoke`, updates the default HTTP deny list, and adds docs, changelog, and Gateway tests.

## Best Possible Solution

Have Gateway/security owners choose the intended surface, then either land a narrow owner-only exec registration or fix this broader path so exec-capable tools are owner-gated, plugin suppression is real, docs/tests match, and redacted live `/tools/invoke` proof is supplied.

## Reproduction Assessment

Yes for the review findings: the PR head and current main source show the owner-only and plugin-suppression issues directly. I did not run tests because this review is read-only.

## Solution Assessment

No. The current broad raw-factory approach is not the best merge-ready solution until it preserves the owner/plugin boundaries; a narrower owner-only exec registration or a corrected broad implementation would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep HTTP exec and process owner-only:** `src/gateway/tool-resolution.ts:195-202`
  - body: After `gateway.tools.allow` removes the default deny, the PR appends normal coding `exec`/`process` tools to `/tools/invoke`. Those tools are not `ownerOnly`, and the current fallback only classifies `cron`, `gateway`, and `nodes`, so trusted-proxy callers with `operator.write` but without `operator.admin` can reach command execution. Mark or wrap the HTTP `exec`/`process` entries as owner-only before exposing them.
  - confidence: 0.88
- **[P2] Wire plugin suppression into the raw coding factory:** `src/agents/pi-tools.ts:1156-1159`
  - body: `createOpenClawCodingToolsRaw()` passes `disablePluginTools: true`, but `createOpenClawCodingTools()` never reads that option when building its default construction plan, so channel/OpenClaw/plugin-backed tools are still materialized. Wire the flag into the plan, or pass an explicit core-only plan from the raw factory, so known-core HTTP requests do not re-enter plugin resolution.
  - confidence: 0.9
