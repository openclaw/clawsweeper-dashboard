## Summary

Keep open: current main still lacks the requested coding-tool wiring, and the PR remains relevant, but the latest branch is not merge-ready. The force-pushed version addresses the earlier loopback and double-hook shape, but raw coding-tool construction still re-enters plugin resolution on a sensitive HTTP execution path and `apply_patch` remains impossible to expose as described; the deny-list docs also drift after the new default-denied names.

## What This Changes

The PR adds an unwrapped coding-tool factory, merges coding tools into Gateway HTTP `/tools/invoke`, extends the default HTTP dangerous-tool deny list, and adds Gateway invoke regression tests.

## Best Possible Solution

Revise the existing PR so `/tools/invoke` gets a coding-only, unwrapped tool set that respects core-only plugin suppression, has an explicit and tested `apply_patch` provider decision, keeps HTTP and RPC invoke semantics aligned, and updates the public deny-list docs before merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on the PR head, a known-core HTTP invoke asks the resolver to disable plugin tools, but the added raw coding factory still reaches `createOpenClawTools` without that flag; allowlisting `apply_patch` also cannot surface it because no model provider is passed.

## Solution Assessment

No. The overall direction is useful, but the current implementation should not reuse the full agent coding-tool factory inside the Gateway HTTP resolver unless plugin suppression and provider context are correctly threaded or the raw factory is narrowed to coding primitives only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep raw coding tools from loading plugins:** `src/gateway/tool-resolution.ts:132-139`
  - body: `createOpenClawCodingToolsRaw()` still delegates into `createOpenClawCodingTools()`, whose tool list appends `createOpenClawTools()` without the caller's `disablePluginTools` setting. That means a known-core `/tools/invoke` request can still run plugin resolution through the raw coding path, violating the existing workspace-plugin reload regression. Return only coding primitives here or thread the no-plugin context through the raw factory.
  - confidence: 0.88
- **[P2] Make allowlisted apply_patch reachable:** `src/gateway/tool-resolution.ts:132-139`
  - body: The resolver does not pass a `modelProvider` into the raw coding-tool factory, but `createOpenClawCodingTools()` only creates `apply_patch` for OpenAI-family providers. An operator can therefore remove `apply_patch` from the HTTP deny list and allow it in policy but still get a 404, despite this PR treating the deny list as the gate. Thread the intended provider context or remove/document this tool from the promised surface.
  - confidence: 0.8
- **[P3] Update the HTTP deny-list docs:** `src/security/dangerous-tools.ts:17-21`
  - body: This adds `process`, `write`, and `edit` to the default HTTP deny list, but the public Tools Invoke docs still list the old defaults. Operators reading the docs will not know those canonical coding tools require `gateway.tools.allow`, so update the docs alongside the behavior change.
  - confidence: 0.9
