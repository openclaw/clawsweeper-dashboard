## Summary

Keep this PR open: the feature direction is still useful and current main does not wire coding tools into the HTTP resolver, but the latest PR diff still has blocking correctness and security issues around plugin suppression and provider-gated tool construction.

## What This Changes

The PR adds a raw/unhooked coding-tool factory, wires coding tools into Gateway HTTP `/tools/invoke`, expands the default HTTP deny list with canonical mutating tool names, and adds invoke regression tests.

## Best Possible Solution

Revise the branch so Gateway HTTP coding-tool construction uses a Gateway-specific path that honors core-only plugin suppression, deliberately threads provider/model context for provider-gated tools, keeps loopback behavior unchanged, and updates the public deny-list docs and tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: the PR diff calls the raw coding-tool factory without `disablePluginTools` or provider/model context, while current main’s shared invoke path and coding-tool tests show those inputs materially change the tool set.

## Solution Assessment

No. Reusing the full agent coding-tool factory is reasonable only if the raw factory contract also supports plugin suppression and provider context; otherwise a narrower Gateway-specific coding-primitives factory is the safer implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Thread plugin suppression into raw coding tools:** `src/gateway/tool-resolution.ts:131-139`
  - body: `invokeGatewayTool` intentionally resolves cataloged core tools with `disablePluginTools: true`, but this new raw coding-tool path ignores that flag. Since `createOpenClawCodingToolsRaw()` delegates to `createOpenClawCodingTools()`, whose tool list appends `createOpenClawTools()` without suppression, the core-only HTTP pass can still materialize plugin-backed tools and bypass the intended core-first/fallback split. Thread the suppression through the raw factory or return only coding primitives here.
  - confidence: 0.88
- **[P2] Pass provider context to coding-tool factory:** `src/gateway/tool-resolution.ts:131-139`
  - body: This resolver call does not pass `modelProvider` or `modelId`, but `createOpenClawCodingTools()` only creates `apply_patch` for OpenAI-family providers and model allowlists. Operators can allowlist `apply_patch` over HTTP, yet this surface will still 404 because the tool is never constructed. Thread the intended provider/model context or remove/document `apply_patch` from the promised surface.
  - confidence: 0.84
- **[P3] Update the HTTP deny-list docs:** `src/security/dangerous-tools.ts:17-21`
  - body: The patch adds `process`, `write`, and `edit` to `DEFAULT_GATEWAY_HTTP_TOOL_DENY`, but the public Tools Invoke docs still list the old defaults. Update the docs so operators know these canonical coding tools require `gateway.tools.allow` before they are reachable.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch expands the Gateway HTTP execution surface while the new raw coding-tool path can still re-enter plugin resolution during a core-only lookup.

Concerns:
