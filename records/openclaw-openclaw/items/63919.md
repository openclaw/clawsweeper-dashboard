## Summary

## What This Changes

The branch adds owner-gated `gateway.tools.directInvoke` opt-ins for built-in `read`, `write`, and `edit` coding tools on Gateway HTTP `/tools/invoke` and SDK `tools.invoke`, with config schema/help, audit/deny changes, docs, tests, and SDK surface-budget updates.

## Best Possible Solution

Move built-in-only host-FS deny handling out of production plugin-loader deny filtering, add production plugin-path proof for same-named plugin tools, then land only if maintainers explicitly accept the owner-gated host-FS direct-invoke contract.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR passes `read`/`write`/`edit` through `pluginToolDenylist`, and the production plugin resolver filters denylisted tool names before the new preservation callback can run. I did not run runtime tests because this review is read-only.

## Solution Assessment

No: the triple-key gate is a plausible shape for built-in host-FS tools, but the current plugin-preservation repair is too late in the pipeline. The safer fix is to keep built-in host-FS default deny separate from plugin loader deny filtering and prove the real plugin path.
