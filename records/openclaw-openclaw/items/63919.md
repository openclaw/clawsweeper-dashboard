## Summary

This PR should stay open: the dual-key design avoids silently enabling the new built-in filesystem tools, but same-name plugin tools can win resolution and bypass the documented source assumption, leaving operators with misleading gating and audit behavior. Current-head proof also stops at the compiled resolver instead of exercising a real HTTP or SDK write/edit request.

Likely related people: Vincent Koc (medium confidence, recent shared-invoke and policy-path history), @ai-hpc (high confidence, introduced the merged `tools.invoke` RPC), and @BunsDev (medium confidence, merger/reviewer of that RPC surface).

## What This Changes

The PR adds dual-key configuration opt-ins that expose the built-in read, write, and edit tools through Gateway HTTP and SDK direct-invoke surfaces, with schema, audit, deny-list, test, and documentation updates.

## Best Possible Solution

Give direct-invoke coding tools an explicit, source-aware identity: either reserve these names for the built-in implementations on this surface or enforce the class opt-in after final resolution using tool provenance, then keep generic dangerous-allow warnings for same-name plugin tools and add collision, upgrade, and live transport proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows plugin tools are created before the proposed coding tools, plugin names are not reserved, existing tools win collisions, and the audit exemption checks names only. No live runtime reproduction was run because this review is read-only.

## Solution Assessment

No. Dual-key gating is a reasonable compatibility safeguard for built-in tools, but the proposed implementation is not source-aware and therefore cannot guarantee that the gated name resolves to the documented built-in filesystem tool.
