## Summary

Keep open. Current main still lacks the requested coding-tool wiring, so the PR is not obsolete, but the latest diff still has blocking issues on the Gateway direct tool execution surface: the raw coding factory can re-enter plugin tool resolution despite the known-core fast path, and allowlisted `apply_patch` remains unreachable because no model provider context is supplied.

## What This Changes

The PR adds an unwrapped coding-tool factory, wires coding tools into Gateway HTTP `/tools/invoke`, expands the default HTTP deny list for canonical mutating tools, and adds Gateway HTTP regression tests.

## Best Possible Solution

Revise the branch so the HTTP resolver receives a coding-only, unwrapped tool set that does not re-enter plugin resolution when `disablePluginTools` is requested; then explicitly decide and test whether `apply_patch` belongs on this direct HTTP surface, update the public deny-list docs, and rerun the targeted Gateway/security checks before merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on the PR head: `resolveTools(true)` for a known core HTTP request still constructs `createOpenClawCodingToolsRaw`, which delegates into full OpenClaw/plugin tool construction, and `apply_patch` cannot be produced because no `modelProvider` is passed.

## Solution Assessment

No. The feature direction is useful, but reusing the full agent coding factory inside the HTTP resolver is not the narrowest maintainable solution; a coding-primitives-only raw seam or properly threaded `disablePluginTools` and provider context would be safer.

## Review Findings

Overall confidence: 0.87

Full review comments:

- **[P2] Keep raw coding tools from loading plugins:** `src/gateway/tool-resolution.ts:132-139`
  - body: `createOpenClawCodingToolsRaw()` still delegates to the full coding-tool factory, and that factory appends `createOpenClawTools()` without the caller's `disablePluginTools` setting. This defeats the HTTP handler's known-core fast path and can run plugin resolution for direct core `/tools/invoke` requests; return only coding primitives here or thread `disablePluginTools` through the raw factory.
  - confidence: 0.88

An operator can therefore remove `apply_patch` from the HTTP deny list and still get a 404, contrary to the PR's promised deny-list-gated surface.
  - confidence: 0.8

  - body: The diff adds `process`, `write`, and `edit` to the default HTTP deny list, but `docs/gateway/tools-invoke-http-api.md` still lists the old defaults. Update the public docs so operators know those canonical coding tools require `gateway.tools.allow`.
