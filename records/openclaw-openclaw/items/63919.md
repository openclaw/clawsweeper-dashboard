## Summary

Keep this PR open for gateway/security owner review. The implementation looks technically coherent and the previous code findings appear addressed, but merging would intentionally expand the HTTP tool boundary, so the remaining decision is product/security approval rather than cleanup closure.

## What This Changes

The PR wires raw core coding tools into gateway HTTP `/tools/invoke`, adds HTTP deny-list entries for mutating coding tools, documents the surface, and adds regression coverage.

## Best Possible Solution

Gateway/security owners should explicitly approve the HTTP coding-tool contract before merge; if the full primitive set is too broad, land a narrower approved subset such as read-only first and leave execution/mutation to the broader direct-invoke design.

## Reproduction Assessment

Yes, for the feature gap: current `main` source only materializes gateway/OpenClaw tools in `resolveGatewayScopedTools`, and the PR body includes before/after HTTP evidence for coding-tool availability. I did not run live tests because this review is required to keep the checkout read-only.

## Solution Assessment

Unclear as a product decision, though the technical shape is reasonable: the PR keeps the raw factory HTTP-scoped, avoids double hook execution, extends the deny list, and preserves owner-only filtering. The safer alternative is a staged read-only landing if maintainers do not want mutating/execution tools available by HTTP opt-in yet.
