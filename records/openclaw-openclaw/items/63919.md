## Summary

Keep open. The PR is useful and has live curl proof, but source review still finds merge-blocking issues around the direct HTTP command surface and the raw factory’s plugin-suppression boundary.

## What This Changes

The PR wires raw coding tools into Gateway HTTP `/tools/invoke`, adds `write`/`edit`/`process` to the default HTTP deny list, and updates docs, tests, and the changelog.

## Best Possible Solution

Land only after Gateway/security owners approve the intended surface and the implementation keeps command-execution tools behind the intended owner/admin boundary while truly disabling plugin/channel loading for core-only HTTP resolution.

## Reproduction Assessment

Yes for the review findings: source inspection shows raw coding tools entering `/tools/invoke`, existing exec/process lacking owner-only metadata, and `disablePluginTools` not affecting the coding-tool construction plan. I did not execute tests in this read-only review.

## Solution Assessment

No. The broad raw-factory approach is not merge-ready until command-execution tools preserve owner/admin semantics and plugin suppression is wired into the construction plan; a narrower owner-only direct-invoke path may be safer if maintainers reject the broader surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep HTTP exec and process owner-only:** `src/gateway/tool-resolution.ts:193-202`
  - body: After `gateway.tools.allow` removes the default deny, this path adds normal raw `exec` and `process` tools to `/tools/invoke`. Those tools are not `ownerOnly`, and the owner-only fallback list only covers `cron`, `gateway`, and `nodes`, so a trusted-proxy caller with `operator.write` but without `operator.admin` can reach command execution. Mark or wrap the HTTP entries as owner-only before exposing them.
  - confidence: 0.9
- **[P2] Honor plugin suppression in the raw coding factory:** `src/agents/pi-tools.ts:1156-1159`
  - body: `createOpenClawCodingToolsRaw()` passes `disablePluginTools: true`, but `createOpenClawCodingTools()` never reads that option when building its construction plan, leaving channel/OpenClaw/plugin-capable tool construction enabled. Known-core `/tools/invoke` requests can still materialize plugin/channel tools, defeating the resolver's core-only pass.
  - confidence: 0.93
