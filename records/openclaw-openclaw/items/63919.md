## Summary

Keep open. Current main still does not wire coding primitives into the direct invoke resolver, and this PR has addressed the earlier concrete bot findings; the remaining gate is maintainer product/security approval for expanding the authenticated direct tool surface.

## What This Changes

The PR adds a raw coding-tool factory and wires `read` plus opt-in `write`/`edit`/`exec`/`process` into the gateway direct tool invoke path with deny-list, owner-only, docs, changelog, and tests.

## Best Possible Solution

Have a gateway/security owner decide whether this expanded direct tool surface is acceptable, then merge this guarded implementation if the proof/check state is green.

## Reproduction Assessment

Yes. Source inspection shows current main resolves direct invokes from gateway/OpenClaw tools only, and the PR body provides before/after curl evidence for `read` and the default-denied mutating tools.

## Solution Assessment

Mostly yes. The raw factory, HTTP-only construction guard, default deny-list expansion, `apply_patch` exclusion, and owner-only `exec`/`process` tagging are a maintainable implementation shape; the open question is whether maintainers want this capability exposed.
