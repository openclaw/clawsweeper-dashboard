## Summary

Keep open: current main still lacks the requested HTTP coding-tool wiring, but this patch has merge-blocking source-level issues around plugin suppression and provider-scoped tool availability on a security-sensitive Gateway execution surface.

## What This Changes

The PR adds a raw coding-tool factory, wires coding tools into the Gateway HTTP `/tools/invoke` resolver for HTTP requests, expands the default HTTP deny list, and adds regression coverage.

## Best Possible Solution

Revise the existing branch so HTTP coding-tool construction explicitly honors plugin suppression, receives session provider/model context or excludes provider-gated tools, keeps loopback unchanged, and updates docs/changelog with matching tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: known core HTTP invokes take a `disablePluginTools: true` resolver pass, but the proposed raw coding factory still delegates to plugin-capable tool construction; an allowlisted `apply_patch` also remains unavailable without provider/model context.

## Solution Assessment

No. Wiring coding tools into `/tools/invoke` is useful, but the current raw-factory approach is not complete until plugin suppression and provider-scoped availability are explicit parts of the resolver contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Thread plugin suppression into raw coding tools:** `src/gateway/tool-resolution.ts:129-136`
  - body: `invokeGatewayTool` resolves known core tool requests with `disablePluginTools: true` before falling back to plugins, but this new raw coding-tool path ignores that flag. Because `createOpenClawCodingToolsRaw()` delegates to the normal coding factory, whose inner `createOpenClawTools()` call can still resolve plugin-backed tools, the supposedly core-only HTTP pass can materialize plugins. Thread the suppression flag through the raw factory or build only coding primitives here.
  - confidence: 0.88
- **[P2] Pass provider context to raw coding tools:** `src/gateway/tool-resolution.ts:129-136`
  - body: The PR advertises `apply_patch` as part of the deny-list-gated HTTP coding surface, but this resolver does not pass `modelProvider` or `modelId` into `createOpenClawCodingToolsRaw()`. The coding factory only creates `apply_patch` for OpenAI-family providers, so an operator can allowlist it and still get a 404 from this endpoint. Resolve and pass the session model context, or remove/document `apply_patch` from the promised surface.
  - confidence: 0.84
- **[P3] Align docs and changelog with the new deny list:** `src/security/dangerous-tools.ts:16-20`
  - body: Adding `process`, `write`, and `edit` changes operator-facing `gateway.tools.allow` defaults, but the public Tools Invoke docs still list the old defaults and this user-facing Gateway feature has no changelog entry. Update both before shipping so operators know which tools require explicit opt-in.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch expands a Gateway command/file execution surface and still lets raw coding-tool construction re-enter plugin resolution during core-only HTTP invoke resolution.

Concerns:

- **[medium] Raw coding factory can re-enter plugin resolution:** `src/gateway/tool-resolution.ts:129`
  - body: The proposed `resolveGatewayScopedTools` call would construct raw coding tools for HTTP requests, but that raw factory delegates to the normal coding factory, whose internal `createOpenClawTools()` call does not receive the resolver's `disablePluginTools` flag. This can materialize plugin-backed tools during direct HTTP tool resolution where the handler intentionally requested core-only resolution for known core tools.
