## Summary

Keep open. This is meaningful feature work, but the current branch has a parse/build blocker, security-sensitive authorization and plugin-suppression defects, and no required after-fix real behavior proof.

## What This Changes

The branch adds raw coding-tool construction and Gateway resolver wiring so `read` plus allowlisted `write`/`edit`/`exec`/`process` can be invoked through HTTP `/tools/invoke`, with deny-list, docs, changelog, and tests.

## Best Possible Solution

Have Gateway/security owners choose between a narrow owner-only exec registration and a broader coding-tool HTTP surface; the chosen path should be buildable, owner-gate exec-capable tools, actually suppress plugin-backed tools where intended, update tests/docs, and include redacted live `/tools/invoke` proof.

## Reproduction Assessment

Yes for the review findings: the duplicate `surface` declaration, missing exec/process owner-only classification, and inert plugin-suppression flag are visible from source inspection of the PR head. I did not run tests because this review is read-only.

## Solution Assessment

No. The current broad-factory approach is not ready because it is not buildable and does not preserve the expected owner/plugin boundaries; a safer path is a maintainer-approved narrow registration or a corrected broad surface with equivalent admin gating.

Full review comments:

- **[P1] Remove the duplicate surface declaration:** `src/gateway/tool-resolution.ts:177`
  - body: `resolveGatewayScopedTools` already declares `const surface = params.surface ?? "http"` earlier in this function. Redeclaring it here in the same block makes the PR head fail to parse/compile before any Gateway tests can run; reuse the existing variable instead.
  - confidence: 0.99
- **[P1] Keep allowlisted exec/process owner-only:** `src/gateway/tool-resolution.ts:197-203`
  - body: This appends the normal coding tool set to HTTP `/tools/invoke`, including `exec` and `process` after `gateway.tools.allow` removes the default deny. Those tools are not marked `ownerOnly`, and the current owner-only fallback only covers `cron`, `gateway`, and `nodes`, so trusted-proxy callers with `operator.write` but without `operator.admin` can reach command execution. Wrap or register the HTTP `exec`/`process` entries as owner-only before exposing them.
  - confidence: 0.88
- **[P2] Actually suppress plugin-backed tools in the raw factory:** `src/agents/pi-tools.ts:1159`
  - body: Setting `disablePluginTools: true` here is currently inert: `createOpenClawCodingTools` never reads that option when it builds the default construction plan, so channel, OpenClaw, and plugin-backed tools remain enabled. Wire this into `toolConstructionPlan` or pass an explicit plan from the raw factory, then cover the real factory path instead of only a resolver mock.
  - confidence: 0.9
