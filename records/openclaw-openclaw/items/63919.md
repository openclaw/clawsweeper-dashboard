## Summary

Keep open: the branch still has a blocking plugin-compatibility bug in source, the latest compatibility proof is not real production plugin-path proof, and the protected security label plus host-FS write boundary require maintainer/security acceptance.

## What This Changes

The PR adds direct-invoke host-filesystem read/write opt-ins that materialize built-in `read`, `write`, and `edit` coding tools for Gateway HTTP `/tools/invoke` and SDK `tools.invoke`, with config schema/help/docs, deny/audit changes, SDK surface budget updates, and tests.

## Best Possible Solution

Move the built-in-only host-FS deny handling before plugin loader filtering, add production plugin-loader or live proof for same-named plugin tools, then land only if maintainers explicitly accept the owner-gated host-FS direct-invoke contract.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR passes `read`/`write`/`edit` through `pluginToolDenylist`, and the production plugin resolver filters denylisted tool names before the new preservation callback can run. I did not run runtime tests because this review is read-only.

## Solution Assessment

No: the triple-key gate is a plausible shape for built-in host-FS tools, but the current preservation fix is too late in the pipeline. The safer fix is to keep built-in host-FS default deny separate from plugin loader deny filtering and prove the real plugin path.
