## Summary

Keep open: the proof is useful, but the PR still has blocking security/ownership defects in the HTTP coding-tool path and needs Gateway/security owner judgment before this surface should land.

## What This Changes

The PR wires coding primitive tools into Gateway HTTP `/tools/invoke`, expands default deny-list coverage for mutating tools, and updates docs, tests, and changelog text.

## Best Possible Solution

Keep default-deny behavior, make allowlisted command/process tools preserve owner/admin semantics, wire plugin suppression through the coding-tool construction plan, and then have Gateway/security owners decide whether broad coding primitives belong on HTTP or should be split into narrower surfaces.

## Reproduction Assessment

Yes for the review findings: source inspection on the PR head shows raw coding tools enter `/tools/invoke`, while `exec` and `process` are not owner-only and `disablePluginTools` is not consumed by the construction plan. I did not execute tests in this read-only review.

## Solution Assessment

No. The current raw-factory approach is not merge-ready until command/process owner semantics and plugin suppression are fixed; maintainers may still prefer a narrower exec-only or read-only path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep HTTP exec and process owner-only:** `src/gateway/tool-resolution.ts:195-202`
  - body: After `gateway.tools.allow` removes the default deny, this path adds normal raw `exec` and `process` tools to `/tools/invoke`. Those tool entries are not `ownerOnly`, and the owner-only fallback list only covers `cron`, `gateway`, and `nodes`, so a trusted-proxy caller with `operator.write` but without owner/admin status can reach command execution. Mark or wrap these HTTP entries as owner-only before exposing them.
  - confidence: 0.92
- **[P2] Honor plugin suppression in the raw coding factory:** `src/agents/pi-tools.ts:1156-1159`
  - body: `createOpenClawCodingToolsRaw()` passes `disablePluginTools: true`, but `createOpenClawCodingTools()` never uses that option when building its construction plan. Known-core `/tools/invoke` requests can still materialize channel/OpenClaw/plugin-capable tools through the coding factory, defeating the resolver's core-only pass.
  - confidence: 0.93
