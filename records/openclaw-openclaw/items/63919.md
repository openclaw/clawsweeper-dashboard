## Summary

Keep open: current main still lacks coding-tool HTTP wiring, but this branch is not ready because the raw factory path still bypasses the core-only/plugin suppression contract and cannot expose `apply_patch` as described without provider context.

## What This Changes

The PR adds a raw coding-tool factory, wires coding tools into Gateway HTTP `/tools/invoke`, expands the default HTTP deny list for mutating coding tools, and adds regression tests.

## Best Possible Solution

Revise the existing branch to use a Gateway-specific coding-tool construction path that honors plugin suppression, defines provider/model context for provider-gated tools, and updates docs, changelog, and regression coverage before maintainer security review.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: the PR diff calls `createOpenClawCodingToolsRaw` without plugin-suppression or provider fields, while current `createOpenClawCodingTools` both appends `createOpenClawTools` and gates `apply_patch` on OpenAI-family provider context.

## Solution Assessment

No. The direction is useful, but reusing the full agent coding-tool factory inside the Gateway resolver is not the narrowest maintainable solution unless the raw factory contract explicitly supports core-only resolution and provider-gated tool construction.
