## Summary

Keep this PR open: it contains useful direct-invoke work and strong live proof, but the current patch still changes existing plugin-name behavior and expands the Gateway host-filesystem write security boundary, so it is not merge-ready without repair and maintainer/security review.

## What This Changes

The PR adds `gateway.tools.directInvoke` host-FS read/write opt-ins, deny/audit/docs/tests, plugin-SDK surface budget bumps, and raw `read`/`write`/`edit` coding-tool materialization for Gateway HTTP `/tools/invoke` and SDK `tools.invoke`.

## Best Possible Solution

Preserve existing same-named plugin tool behavior, enforce built-in `read`/`write`/`edit` through source-aware direct-invoke opt-in checks, and land only after maintainer/security approval of the host-FS write boundary on a current base.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: a plugin tool named `write` or `read` without the new `gateway.tools.allow` entry reaches the final name-only `gatewayDenySet` filter after this PR adds those names to the default deny list.

## Solution Assessment

Is this the best way to solve the issue? No; the built-in host-FS opt-ins should be enforced source-aware at materialization/resolution time without globally denying same-named plugin tools that existed before the feature.
