## Summary

Keep open: I found no blocking code findings, current main still lacks coding tools on the HTTP resolver, and the PR now has strong after-fix live output plus green CI. The remaining blocker is maintainer product/security approval because this intentionally expands the authenticated `/tools/invoke` HTTP surface to core file/runtime tools.

## What This Changes

The PR adds a raw unwrapped coding-tool factory, wires `read` plus opt-in `write`/`edit`/`exec`/`process` into Gateway HTTP `/tools/invoke`, expands the default deny list/docs/changelog, and adds focused gateway and agent tests.

## Best Possible Solution

Have a gateway/security owner approve the authenticated HTTP coding-tool boundary, then merge this guarded implementation if CI remains green; if the boundary is too broad, split to a narrower `read`-only surface or keep coding tools out of HTTP.

## Reproduction Assessment

Yes. Current main's resolver only materializes gateway/OpenClaw tools for HTTP, and the PR body supplies before/after live output showing `read` unavailable before the branch and available after it.

## Solution Assessment

Mostly yes. The raw factory, HTTP-only gating, default deny-list expansion, `apply_patch` exclusion, and `exec`/`process` owner-only tagging are a maintainable shape; the open question is whether maintainers want this HTTP surface at all.
