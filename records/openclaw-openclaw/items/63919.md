## Summary

Keep open: the PR addresses a real `/tools/invoke` gap, but the latest head still has blocking Gateway execution-surface issues around plugin suppression and provider-scoped `apply_patch`, plus docs/changelog drift for the new deny-list defaults.

## What This Changes

The PR adds an unwrapped coding-tool factory, wires coding tools into Gateway HTTP `/tools/invoke`, expands default HTTP deny-list entries, and adds invoke regression tests.

## Best Possible Solution

Revise the existing branch so HTTP coding-tool construction honors core-only plugin suppression, receives explicit session provider/model context or deliberately excludes provider-gated tools, keeps loopback unchanged, and updates docs/changelog with matching tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a known core HTTP invoke takes the `disablePluginTools: true` path, but the PR's raw coding factory still delegates to plugin-capable `createOpenClawTools`; an allowlisted `apply_patch` also remains absent because no provider/model context is passed.

## Solution Assessment

No. Wiring coding primitives into `/tools/invoke` is directionally useful, but reusing the full coding factory is not the narrowest maintainable fix until plugin suppression and provider-scoped tool availability are explicit parts of the contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Thread plugin suppression into raw coding tools:** `src/gateway/tool-resolution.ts:132-139`
  - body: `invokeGatewayTool` deliberately resolves known core tools with `disablePluginTools: true`, but this new raw coding-tool path ignores that flag. Since `createOpenClawCodingToolsRaw()` delegates to `createOpenClawCodingTools()`, and that factory appends `createOpenClawTools()` without `disablePluginTools`, a core `/tools/invoke` request can still materialize plugin-backed tools during the supposedly core-only pass. Return only coding primitives here or thread plugin suppression through the raw factory.
  - confidence: 0.88
- **[P2] Pass provider context to raw coding tools:** `src/gateway/tool-resolution.ts:132-139`
  - body: The PR advertises `apply_patch` as part of the deny-list-gated HTTP coding surface, but this resolver does not pass `modelProvider` or `modelId` into `createOpenClawCodingToolsRaw()`. The coding factory only creates `apply_patch` for OpenAI-family providers, so an operator can allowlist `apply_patch` and still get a 404 from this endpoint. Resolve and pass the session model context, or remove/document `apply_patch` from the promised surface.
  - confidence: 0.84
- **[P3] Update docs and changelog for new denied tools:** `src/security/dangerous-tools.ts:16-20`
  - body: This adds `process`, `write`, and `edit` to the default HTTP deny list, but the public Tools Invoke docs still list the old defaults, and this user-facing feature has no changelog entry. Operators need the docs to match the actual `gateway.tools.allow` opt-in set before this ships.
  - confidence: 0.9
