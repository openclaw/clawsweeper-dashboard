## Summary

## What This Changes

This PR adds owner-gated `read`, `write`, and `edit` coding-tool materialization for Gateway HTTP `/tools/invoke` and SDK `tools.invoke`, plus `gateway.tools.directInvoke` config, audit/docs/tests, and SDK surface-budget updates.

## Best Possible Solution

Land a source-aware, owner-gated direct-invoke design after the read-only base is resolved, preserving existing plugin tool-name behavior or documenting and testing an explicitly approved breaking upgrade path.

## Reproduction Assessment

Yes for the review finding: source inspection shows current main applies `DEFAULT_GATEWAY_HTTP_TOOL_DENY` by tool name and falls back to plugins for known core names, so adding `read`/`write`/`edit` there can block same-named plugin tools. The PR itself is a feature, not a current-main bug reproduction.

## Solution Assessment

No: the owner-gated direct-invoke design is plausible, but the current global name-deny shape is not the safest merge path because it changes plugin behavior outside the built-in host-FS opt-in. A source-aware built-in gate or an explicit maintainer-approved migration contract would be safer.
