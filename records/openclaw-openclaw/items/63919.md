## Summary

Keep open: the branch is directionally useful, but the latest diff still has source-confirmed blockers in the HTTP coding-tool resolver, including a security-sensitive plugin-suppression gap.

## What This Changes

The PR adds an unwrapped coding-tool factory, wires coding tools into Gateway HTTP `/tools/invoke`, expands the default HTTP deny list, and adds Gateway invoke regression tests.

## Best Possible Solution

Revise the branch so HTTP coding-tool construction honors plugin suppression, receives explicit session model context or excludes provider-gated tools, keeps loopback unchanged, and updates docs/changelog with matching regression coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a known core `/tools/invoke` request takes the `disablePluginTools: true` path, while the PR's raw coding factory ignores that flag and `apply_patch` remains provider-gated without provider context.

## Solution Assessment

No. Wiring coding primitives into `/tools/invoke` is a useful direction, but reusing the full coding factory is not the narrowest maintainable solution until plugin suppression and provider-scoped tool availability are explicit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Thread plugin suppression into raw coding tools:** `src/gateway/tool-resolution.ts:132-139`
  - body: `invokeGatewayTool` intentionally resolves known core tool requests with `disablePluginTools: true`, but this new raw coding-tool path ignores that flag. Since `createOpenClawCodingToolsRaw()` delegates to the full coding factory, which appends plugin-capable `createOpenClawTools(...)`, a core `/tools/invoke` request can still materialize plugin-backed tools during the supposedly core-only pass. Return only coding primitives here or thread plugin suppression through the raw factory.
  - confidence: 0.88
- **[P2] Pass provider context to raw coding tools:** `src/gateway/tool-resolution.ts:132-139`
  - body: The PR advertises `apply_patch` as part of the deny-list-gated HTTP coding surface, but this resolver does not pass `modelProvider` or `modelId` into `createOpenClawCodingToolsRaw()`. The coding factory only creates `apply_patch` for OpenAI-family providers, so an operator can allowlist `apply_patch` and still get a 404 from this endpoint. Resolve and pass the session model context, or remove/document `apply_patch` from the promised surface.
  - confidence: 0.84
- **[P3] Update HTTP deny-list docs and changelog:** `src/security/dangerous-tools.ts:16-21`
  - body: This adds `process`, `write`, and `edit` to the default HTTP deny list, but the public Tools Invoke docs still list the old defaults, and this user-facing feature has no changelog entry. Operators need the docs and release notes to match the actual `gateway.tools.allow` opt-in set before this ships.
  - confidence: 0.9
