Keep open: current main still lacks the feature and the latest branch appears technically sound after addressing prior bot findings, but it widens Gateway HTTP tool execution and has an exact-head aggregate check failure, so maintainer review/rerun is the right next step.

## What This Changes

The PR adds an unwrapped coding-tools factory and wires it into Gateway HTTP `/tools/invoke`, with deny-list, docs, changelog, and focused Gateway test updates.

## Best Possible Solution

Accept the raw-factory HTTP-only design if maintainers want direct coding-tool invocation over Gateway HTTP, while keeping the deny-list, docs, changelog, and regression tests together.

## Reproduction Assessment

Yes. On current main, a `/tools/invoke` request for an allowlisted coding primitive such as `read` cannot be resolved because the Gateway resolver only materializes Gateway/OpenClaw tools; the PR adds targeted tests for the newly reachable and default-denied paths.

## Solution Assessment

Yes, technically. The PR uses the narrowest maintainable shape I found: reuse the existing Gateway invoke policy/hook path, add an unwrapped HTTP-only coding factory, preserve loopback behavior, and require explicit opt-in for mutating primitives.
