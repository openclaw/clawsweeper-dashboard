## Summary

Keep open: the feature is plausible, but the current PR still has merge-blocking security boundary issues and the latest security-sensitive behavior is not proven in a real run.

## What This Changes

The PR wires coding primitive tools into Gateway HTTP `/tools/invoke`, expands the default HTTP deny list, and updates Gateway docs/tests.

## Best Possible Solution

Have Gateway/security owners choose the intended direct-invoke surface, then land either a narrower owner-only path or a corrected broad implementation that preserves owner/plugin boundaries and includes current-head live proof.

## Reproduction Assessment

Yes for the review findings: the PR head source directly shows raw coding tools entering the HTTP resolver, while current owner-only policy and the PR raw factory show the boundary gaps. I did not run tests because this review is read-only.

## Solution Assessment

No. The broad raw-factory approach is not merge-ready until exec/process are owner-gated, plugin suppression is actually wired into construction, and the latest HTTP behavior is proven in a real Gateway run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep allowlisted exec/process owner-only:** `src/gateway/tool-resolution.ts:193-202`
  - body: After `gateway.tools.allow` removes the default deny, this path adds normal coding `exec`/`process` tools to `/tools/invoke`. Those tools are not `ownerOnly`, and the fallback owner-only names only cover `cron`, `gateway`, and `nodes`, so a trusted-proxy caller with `operator.write` but without `operator.admin` can reach command execution. Mark or wrap the HTTP `exec`/`process` entries as owner-only before exposing them.
  - confidence: 0.9
- **[P2] Wire plugin suppression into the raw coding factory:** `src/agents/pi-tools.ts:1148-1159`
  - body: `createOpenClawCodingToolsRaw()` passes `disablePluginTools: true`, but `createOpenClawCodingTools()` never reads that option when building its default construction plan, so channel/OpenClaw/plugin-backed tools can still be materialized. Wire the flag into the plan, or pass an explicit core-only construction plan from the raw factory, so known-core HTTP requests do not re-enter plugin resolution.
  - confidence: 0.92
