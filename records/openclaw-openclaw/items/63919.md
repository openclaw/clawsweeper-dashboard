## Summary

Keep this PR open. Current main still lacks the HTTP coding-tool wiring this PR targets, and the merged SDK-facing RPC work does not supersede the HTTP surface, but the current patch still has blocking correctness and security issues around plugin suppression and provider-dependent `apply_patch` availability.

## What This Changes

The PR adds an unwrapped coding-tool factory, wires coding tools into Gateway HTTP `/tools/invoke`, expands the default HTTP deny list, and adds Gateway invoke regression tests.

## Best Possible Solution

Revise the existing branch so `/tools/invoke` gets the intended coding primitives through a Gateway-specific construction path that honors `disablePluginTools`, threads or deliberately defines provider context for provider-gated tools, keeps loopback unchanged, and updates public deny-list docs with matching tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: the shared invoke path passes `disablePluginTools: true` for known core tools, while the PR's raw coding factory still reaches `createOpenClawTools` without that suppression and also omits the provider context required for `apply_patch`.

## Solution Assessment

No. The overall direction is useful, but reusing the full agent coding-tool factory inside the Gateway resolver is not the narrowest maintainable solution unless plugin suppression and provider-specific tool creation are explicitly part of the raw-factory contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Thread plugin suppression into raw coding tools:** `src/gateway/tool-resolution.ts:131-139`
  - body: `invokeGatewayTool` intentionally resolves known core tools with `disablePluginTools: true`, but this new raw coding-tool path ignores that flag. Since `createOpenClawCodingToolsRaw()` delegates to `createOpenClawCodingTools()`, and that factory appends `createOpenClawTools()` without `disablePluginTools`, a core `/tools/invoke` request can still materialize plugin-backed tools during the supposedly core-only pass. Return only coding primitives here or thread plugin suppression through the raw factory.
  - confidence: 0.88
- **[P2] Thread provider context for apply_patch:** `src/gateway/tool-resolution.ts:131-139`
  - body: The PR treats the HTTP deny list as the gate for `apply_patch`, but the resolver does not pass `modelProvider` into `createOpenClawCodingToolsRaw()`. `createOpenClawCodingTools()` only creates `apply_patch` for OpenAI-family providers, so an operator can allowlist `apply_patch` and still get a 404 from this HTTP surface. Pass the intended provider context or remove/document `apply_patch` from the promised surface.
  - confidence: 0.82
- **[P3] Update the HTTP deny-list docs:** `src/security/dangerous-tools.ts:17-21`
  - body: This adds `process`, `write`, and `edit` to `DEFAULT_GATEWAY_HTTP_TOOL_DENY`, but the public Tools Invoke docs still list the old defaults. Operators reading the docs will not know those canonical coding tools require `gateway.tools.allow`, so the docs should change with the behavior.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch expands a sensitive Gateway execution surface and still lets the raw coding factory re-enter plugin resolution during core-only HTTP invoke resolution.

Concerns:
