## Summary

Keep open: this PR is still useful and current main does not implement the requested HTTP coding-tool wiring. The latest branch appears to address the prior bot findings, but it intentionally expands a Gateway HTTP execution surface, so it should go through maintainer review rather than cleanup/closure.

## What This Changes

The PR adds an unwrapped core coding-tool factory, wires coding primitives into Gateway HTTP `/tools/invoke`, updates the deny list/docs/changelog, and adds regression coverage.

## Best Possible Solution

Land after maintainer approval of the HTTP execution-surface expansion and exact-head Gateway/changed-surface checks; otherwise keep #37131 as the broader product discussion.

## Reproduction Assessment

Yes. On current main, a `/tools/invoke` request for an allowlisted coding primitive such as `read` cannot resolve because the Gateway resolver only materializes Gateway/OpenClaw tools; the PR adds focused coverage for the intended successful and denied paths.

## Solution Assessment

Yes for the code shape. The branch reuses the existing resolver, policy, owner, hook, and deny-list pipeline while keeping MCP loopback unchanged and avoiding double hook execution through the raw factory.
