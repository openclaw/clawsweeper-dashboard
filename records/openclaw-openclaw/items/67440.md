## Summary

Keep open. Current `main` still documents exec approvals as not being a per-user auth boundary, keeps the approvals file/protocol at a simple version-1 `{id, decision}` model, and has no shipped TOTP/session support in config, CLI, or approval resolution. The discussion’s external references are useful design input, but they do not show an implemented fix on `main` or an existing ClawHub/plugin solution that already covers OpenClaw’s host-side exec approval path.

## Evidence

- **Current docs still describe the reported gap:** The exec-approvals docs explicitly say approvals reduce accidental execution risk but are not a per-user auth boundary, and they document the current default host behavior as `tools.exec.security=full`, `tools.exec.ask=off`, and host `askFallback=full`. That matches the issue’s threat model rather than disproving it.
