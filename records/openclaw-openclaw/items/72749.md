## Summary

Keep this PR open. Current main still has the Telegram/direct metadata bloat that the PR targets, but the proposed patch is not ready to merge because compact-direct prompts remove the parseable sender metadata that the gateway sanitizer uses to recover visible sender labels, and the user-facing config/Telegram behavior lacks a required changelog entry.

## What This Changes

The PR adds optional `messages.inboundMetadataMode: "compact-direct"` config/schema/docs/tests, passes it into inbound prompt construction, strips compact direct metadata from history text, and also changes primary `wiki`/`memory` CLI plugin loading to include the configured memory-slot plugin.

## Best Possible Solution

Land a revised compact-direct implementation that reduces Telegram direct-message prompt noise while preserving the existing user-visible sender attribution contract, keeps default/group/rich-context behavior unchanged, includes focused sanitizer coverage, adds the required changelog entry, and splits the memory CLI loader change if maintainers do not want it coupled to the metadata fix.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to build a compact-direct direct-chat prompt at PR head, then pass that message content through `stripEnvelopeFromMessage` without an explicit `senderLabel`; the sanitizer cannot recover the label because the full sender/conversation blocks were omitted.

## Solution Assessment

No, not as-is. The compact prompt direction is narrow and appropriate for the linked reports, but the implementation should preserve sender labels out of band or teach the sanitizer to recover them from compact metadata, with a regression test covering the gateway display path.
