## Summary

Keep this PR open. Current main still has the reported full direct-chat metadata behavior, but the PR branch is not ready to merge because compact-direct mode removes the only sender label source used by chat-history sanitization, and the user-facing config change is missing the required changelog entry.

## What This Changes

The PR adds optional `messages.inboundMetadataMode: "compact-direct"` config/schema/docs/tests, passes it into inbound prompt construction, strips compact metadata from displayed history, and also adjusts memory-slot plugin loading for `wiki`/`memory` CLI commands.

## Best Possible Solution

Land a revised compact metadata implementation that reduces Telegram/direct prompt noise while preserving user-visible sender attribution, keeps default and group/rich-context behavior unchanged, adds the required changelog entry, and separates the plugin CLI loader change if maintainers do not want it coupled to the metadata fix.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence path: build a compact-direct direct-chat prompt at PR head, pass it through `stripEnvelopeFromMessage` without an explicit `senderLabel`, and the sanitizer cannot recover a sender label because the parseable sender/conversation blocks were omitted.

## Solution Assessment

No, not as-is. The compact prompt direction is narrow and appropriate for the linked reports, but the patch must preserve the existing sender-label display contract, ideally through an out-of-band senderLabel or a compact extraction path covered by a gateway sanitizer regression test.
