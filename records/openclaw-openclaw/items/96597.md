## Summary

Keep open: current main and v2026.6.10 still expose Google Vertex through a static Google catalog, while the existing `models.providers.*.models` config surface already covers manual model rows. The remaining live Vertex discovery work needs maintainer product/API decisions around ADC, endpoint, caching, timeout, and fallback behavior.

## What This Changes

Request dynamic Google Vertex model catalog refresh plus a way to add newer Vertex models when the built-in catalog is stale.

## Best Possible Solution

Keep this as the product/API decision for provider-owned Google Vertex live catalog discovery, and handle broken manual override behavior in the narrower linked issue without adding a second override surface.

## Reproduction Assessment

Yes. Source inspection shows `google-vertex` is still populated from a static catalog hook and the Google provider does not implement a credentialed runtime catalog hook; I did not run a live Google Cloud API call in this read-only review.

## Solution Assessment

Unclear. Dynamic discovery may be the right provider-owned feature, but maintainers should first settle auth, endpoint, cache, and fallback policy and avoid adding a duplicate manual override surface already covered by `models.providers.*.models`.
