## Summary

Keep open: current main still has the Docker/bridge LAN advertisement bug and this PR is the canonical fix path, but the patch is not mergeable as-is because `src/gateway/net.ts` calls `fs.readFileSync` without importing `fs`, and the PR only provides test/CI validation rather than real after-fix behavior proof.

## What This Changes

The PR updates gateway, setup-code, and device-pair LAN IPv4 selection to prefer default-route or physical LAN candidates over Docker/bridge/CNI interfaces while preserving bridge fallback and adding regression tests.

## Best Possible Solution

Fix the missing `fs` import and branch conflict, then land the narrow LAN address priority change after redacted real Linux gateway/QR output shows bridge addresses are de-prioritized.

## Reproduction Assessment

Yes. By source inspection, current main selects `en0`/`eth0` and then the first external/private IPv4 in the gateway, setup-code, and device-pair paths, so an interface snapshot ordered with `br-*` before `wlp*` reproduces the reported advertised Docker bridge URL path.

## Solution Assessment

Yes in direction, no as submitted. De-prioritizing bridge/CNI interfaces while keeping fallback and explicit configuration support is the narrow maintainable fix, but the missing import, conflict, and missing real behavior proof must be resolved first.
