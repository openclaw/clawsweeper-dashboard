## Summary

Keep open: the hook signal is useful, but this PR is not merge-ready because it derives `toolUses` from the final assistant message only, ignores OpenClaw's canonical `toolCall` blocks, and does not show a real hook receiving the new field after an agent run.

## What This Changes

The PR adds a `toolUses` field to plugin hook payload types and tries to populate it for `llm_output` and `agent_end` from assistant `tool_use` blocks.

## Best Possible Solution

Expose a documented additive `toolUses` field populated from canonical current-turn tool-call data, while leaving richer outcomes such as results, durations, and completion reason to https://github.com/openclaw/openclaw/issues/87207.

## Reproduction Assessment

Yes at source level: current main has no typed tool-use collection on `llm_output` or `agent_end`, and the linked issue describes a concrete downstream supervisor need. I did not run a live agent reproduction.

## Solution Assessment

No as submitted. The additive hook field is a reasonable direction, but the implementation should collect canonical current-turn tool calls instead of only raw `tool_use` blocks from the final assistant message.

- **[P1] Collect tool calls from the current turn:** `src/agents/embedded-agent-runner/run/attempt.ts:4925`

Add focused `llm_output`/`agent_end` coverage plus `docs/plugins/hooks.md` wording before exposing this as an SDK-visible contract.
  - confidence: 0.78
