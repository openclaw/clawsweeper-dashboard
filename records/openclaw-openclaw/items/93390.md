## What This Changes

The PR adds a structured `toolUses` hook payload field for assistant tool-use content blocks in CLI and embedded agent hook dispatch.

## Best Possible Solution

Keep the narrow additive hook field only if it is populated from OpenClaw's canonical current-turn tool-call data and documented as a public plugin SDK contract; route richer outcomes to https://github.com/openclaw/openclaw/issues/87207.

## Reproduction Assessment

Yes at source level: current `llm_output` and `agent_end` hook event types expose text/raw assistant data but no typed tool-use collection, and the linked issue describes a real downstream supervisor need.

## Solution Assessment

No as submitted. The additive hook field is a reasonable direction, but extracting only raw `tool_use` blocks from the last assistant message is too narrow for OpenClaw's normalized `toolCall` messages and ordinary multi-step turns.
