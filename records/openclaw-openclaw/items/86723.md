## Summary

## Best Possible Solution

Persist the QQ Bot group route through the channel turn record while either limiting this PR’s new write to group/guild inbounds or adding the sibling main-DM owner-pin guard for direct routes, then merge after redacted real QQ Bot proof confirms group replies deliver.

## Reproduction Assessment

Yes from source inspection: current main records QQ Bot inbound session metadata without `updateLastRoute`, while `sessions_send` announce delivery resolves targets from persisted session delivery context. I did not run a live QQ Bot reproduction in this read-only review.

## Solution Assessment

No: persisting a group route is the right boundary, but this patch applies the write to every inbound and misses the direct-main-session guard used by sibling channels. The safer fix is group-only persistence for this PR, or the full `resolveInboundLastRouteSessionKey` plus `mainDmOwnerPin` pattern with focused tests.
