## Summary

Keep open. Current main still only mitigates part of the reported QMD contention: gateway startup refresh is opt-in and search waits only on the same manager instance, while CLI calls can create a separate transient QMD manager and `qmd update` serialization remains process-local. The related #71784 report is a different memory-search failure mode in the builtin remote embedding/SSRF fetch path, not a replacement tracker for QMD SQLite lock contention.

## What This Changes

The issue requests reliable QMD-backed `openclaw memory search` during normal gateway/background memory activity without intermittent SQLite lock errors.

## Best Possible Solution

Add bounded per-agent cross-process coordination in the memory-core QMD owner path so gateway/background QMD update or embed maintenance and foreground QMD search/doc lookup share a lock, wait, or skip policy, while preserving lazy startup, transient CLI managers, builtin fallback for hard QMD failures, and regression coverage for gateway-process plus CLI-process contention.

## Reproduction Assessment

Unclear. The reporter gives concrete macOS/QMD steps and logs, and current source inspection confirms the race shape, but this read-only review did not run a live two-process QMD reproduction.

## Solution Assessment

Yes. A focused QMD coordination change in `extensions/memory-core` is the narrowest maintainable direction; changing startup defaults, relying on gateway-stop workarounds, or folding this into #71784 would miss the separate QMD SQLite race.
