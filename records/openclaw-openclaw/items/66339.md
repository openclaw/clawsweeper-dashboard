## Summary

Keep open. Current main still coordinates QMD update work only inside one Node process and QMD searches only wait on the same manager instance, while CLI/startup paths can create separate transient managers that race a running gateway against the same QMD SQLite store.

## What This Changes

The issue asks for QMD-backed `openclaw memory search` to avoid intermittent SQLite lock contention between gateway/background memory work and foreground CLI search or indexing.

## Best Possible Solution

Add bounded per-agent cross-process coordination in the memory-core QMD owner path so QMD update/embed/search/doc-lookup work shares a lock, wait, or skip policy without changing the public backend contract.

## Reproduction Assessment

No high-confidence live reproduction was run here. The reporter provides concrete commands and current-main source inspection confirms a plausible cross-process race between gateway/background QMD work and transient foreground CLI managers.

## Solution Assessment

Yes. A focused QMD coordination fix in `extensions/memory-core` is the narrowest maintainable direction; changing startup defaults or folding this into the remote embedding issue would miss the SQLite lock path.
