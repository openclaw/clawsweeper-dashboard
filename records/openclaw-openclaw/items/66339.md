## Summary

Keep open. Current main still leaves QMD gateway and CLI work coordinated only within one manager/process, so the reported gateway-plus-CLI SQLite lock contention remains a source-level gap and is not superseded by the separate remote embedding transport or ARM64 SQLite reports.

## What This Changes

The issue asks OpenClaw to prevent QMD-backed memory search/index/update from intermittently hitting SQLite lock contention during normal gateway-plus-CLI runtime.

## Best Possible Solution

Add bounded per-agent cross-process coordination in the memory-core QMD owner path so update, embed, search, and doc-lookup work share a lock, wait, or skip policy without changing the public memory backend contract.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection still shows the race shape: separate gateway/CLI QMD managers can operate on the same QMD home while update serialization and search waiting are manager/process-local.

## Solution Assessment

Yes. A focused QMD coordination fix is the narrowest maintainable solution; startup-lazy defaults and read busy timeouts reduce some triggers but do not coordinate ordinary foreground/background access to the QMD SQLite store.
