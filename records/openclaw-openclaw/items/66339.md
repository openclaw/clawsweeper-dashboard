## Summary

Keep open. Current main still coordinates QMD update/search only inside one manager/process; embed has a file lock, but QMD update and read/search waiting do not share a cross-process lock, so the reported gateway-plus-CLI SQLite contention remains a real source-level gap and is not superseded by the separate remote-embedding or ARM64 SQLite reports.

## What This Changes

The issue asks for QMD-backed `openclaw memory search` and foreground indexing to coordinate with gateway background QMD work so normal runtime does not hit intermittent SQLite lock errors.

## Best Possible Solution

Add bounded per-agent cross-process coordination in the memory-core QMD owner path so update, embed, search, and doc lookup work share a lock, wait, or skip policy without changing the public memory backend contract.

## Reproduction Assessment

No high-confidence live reproduction was run. The report has detailed runtime symptoms and source inspection still shows the race shape: separate gateway/CLI QMD managers can operate on the same QMD home while update serialization and search waiting are manager/process-local.

## Solution Assessment

Yes. A focused QMD coordination fix is the narrowest maintainable solution; startup-lazy behavior and read busy timeouts reduce some triggers but do not coordinate ordinary foreground/background access to the QMD SQLite store.
