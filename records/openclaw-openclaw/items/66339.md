## Summary

Keep this open. Current main has reduced one startup trigger, but QMD update/search contention is still only coordinated inside one process; a CLI search and the long-lived gateway can still use separate manager state and hit the same QMD SQLite index concurrently.

## What This Changes

The issue asks QMD-backed `openclaw memory search` to coordinate cleanly with gateway/background QMD update work so ordinary runtime searches do not intermittently surface SQLite lock contention.

## Best Possible Solution

Add bounded per-agent cross-process coordination in the memory-core QMD path so gateway/background QMD update or embed maintenance and foreground QMD search/doc lookup share a lock, wait, or skip policy while preserving lazy gateway startup, transient CLI managers, builtin fallback for hard QMD failures, and regression coverage for the gateway-process plus CLI-process race.

## Reproduction Assessment

Unclear. The reporter gives concrete macOS/QMD steps and logs, and source inspection confirms the race shape, but this read-only review did not run a live two-process QMD reproduction against current main.

## Solution Assessment

Yes. A bounded per-agent cross-process coordination layer in the QMD owner module is the narrowest maintainable fix; changing startup defaults, relying on gateway-stop workarounds, or folding this into #71784 would miss the reported QMD SQLite race.
