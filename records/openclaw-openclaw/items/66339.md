## Summary

Keep open. Current main still has only per-process/per-manager QMD coordination, so a foreground CLI manager and the gateway/background manager can still race the same QMD SQLite store; recent startup-lazy fixes reduce one boot path but do not close the reported lock-contention gap.

## What This Changes

The issue asks OpenClaw to prevent QMD-backed memory search from hitting SQLite lock contention when foreground CLI work races gateway/background memory activity.

## Best Possible Solution

Add bounded per-agent cross-process coordination in the memory-core QMD owner path so QMD update/embed/search/doc-lookup work shares a lock, wait, or skip policy without changing the public backend contract.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection still shows the reported race shape: separate gateway/CLI QMD managers can operate on the same QMD home while update serialization and search waiting are manager/process-local.

## Solution Assessment

Yes. A focused QMD coordination fix is the narrowest maintainable path; startup-lazy defaults reduce one boot trigger but do not coordinate ordinary foreground/background access to the QMD SQLite store.
