## Summary

Keep open. Current main still appears to have only partial QMD contention mitigation: searches wait only on the same manager instance, CLI callers can create a separate transient QMD manager, and QMD update serialization remains process-local rather than coordinated between a running gateway and foreground CLI process.

## What This Changes

The issue reports intermittent QMD SQLite lock contention during normal `openclaw memory search` while the gateway or background memory work is running.

## Best Possible Solution

Add bounded per-agent cross-process coordination in the memory-core QMD owner path so gateway/background QMD update or embed maintenance and foreground QMD search/doc lookup share a lock, wait, or skip policy without changing the backend contract.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The reporter gives concrete steps and current source inspection confirms a plausible cross-process race shape between gateway/background QMD work and transient CLI managers.

## Solution Assessment

Yes. A focused QMD coordination fix in `extensions/memory-core` is the narrowest maintainable direction; changing startup defaults or folding this into #71784 would miss the separate SQLite lock contention path.
