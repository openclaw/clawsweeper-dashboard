## Summary

Keep open. Current main still lets the Control UI Sessions tab render Gateway session rows from disk-discovered agent stores, and URL/browser-state restore can still preserve explicit `agent:<id>:*` keys for agents no longer configured.

## What This Changes

The issue asks for the Control UI Sessions tab to exclude sessions belonging to deleted or unregistered agents, matching the configured-agent boundary users see from the CLI.

## Best Possible Solution

Keep broad Gateway/ACP disk-store discovery for recovery and non-UI clients, but make the default Control UI Sessions tab and restored selections respect configured agents, likely through a scoped UI filter or additive `sessions.list` mode plus fallback to the current main session key.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: leave a disk-only `agents/<id>/sessions/sessions.json` while config lists only another agent, then Control UI calls broad `sessions.list` and renders the returned row; the URL/storage restore variant follows from direct `?session=` and persisted key hydration.

## Solution Assessment

Unclear. Removing disk discovery globally, as in the closed related PR, conflicts with current docs and tests; the narrower maintainable direction is a Control UI or additive Gateway listing boundary with restored-key validation.
