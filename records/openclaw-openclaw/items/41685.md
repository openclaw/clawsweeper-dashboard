## Summary

Keep open. Current main still lets the Control UI Sessions tab consume broad Gateway session listings from the combined store, and that combined store intentionally includes disk-discovered agent stores. The merged deleted-agent guards reduce send/resolve hazards, but they do not hide disk-only sessions from the UI or validate restored URL/browser session keys against configured agents.

## What This Changes

The issue asks for the Control UI to stop showing or restoring sessions for agents that are no longer registered in config, matching the narrower CLI session listing behavior after agent deletion.

## Best Possible Solution

Keep broad Gateway/ACP disk-store discovery available for recovery and non-UI clients, but give the default Control UI Sessions tab a configured-agent boundary. Add validation so restored `agent:<id>:*` URL or browser-storage selections fall back to the current main session when the agent is not configured.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to keep a disk-only `agents/<id>/sessions/sessions.json` store while config lists only another agent; current Gateway listing loads that combined store and the Control UI renders the returned row. The restore variant is also source-reproducible through direct URL/session-storage hydration of explicit `agent:<id>:*` keys.

## Solution Assessment

Unclear. The closed related PR’s approach of removing disk discovery globally is not the safest fix because current docs and tests intentionally depend on broad Gateway/ACP discovery; a scoped Control UI filter or additive `sessions.list` mode plus restore validation is the narrower maintainable direction.
