## Summary

Keep open. Current main mitigates deleted-agent targeting paths, but Control UI can still list disk-discovered unregistered agent sessions through the broad Gateway session store, and the remaining UI/API boundary conflicts with documented and tested Gateway/ACP discovery behavior.

## What This Changes

The issue asks Control UI session listing and restored session selection to ignore deleted or unregistered agent namespaces and stay consistent with configured-agent CLI behavior.

## Best Possible Solution

Keep broad Gateway/ACP discovery available for non-UI consumers, but make Control UI default to configured-agent sessions and validate restored `agent:<id>:*` selections against known agents before falling back to the current main session key.

## Reproduction Assessment

Yes, by source inspection and existing tests: a disk-only agent store is included by the combined Gateway store, `sessions.list` returns it to Control UI, and the Sessions view renders returned rows. I did not run a live repro because this review was read-only, but the path is high-confidence.

## Solution Assessment

Unclear. Filtering inside `sessions.list` globally is not clearly the best fix because it conflicts with documented/tested Gateway/ACP discovery; the safer direction is a Control UI default filter or additive configured-only API mode after maintainer boundary confirmation.
