## Label Justifications

- `P2`: This is a concrete hook/session improvement with production impact, but it needs opt-in behavior and security review rather than emergency handling.
- `impact:session-state`: Stable-key webhook sessions lose multi-turn transcript continuity because each hook run is forced into a new session id.
- `impact:data-loss`: The reported behavior makes prior turns unavailable to the agent and can orphan transcript files from the stable session-key mapping.
- `impact:security`: The requested fix changes when hook-selected session keys can reuse existing transcript context.

## Summary

Keep open: current main and the latest release still route stable-key hook-agent calls through forced isolated cron sessions, and an open closing PR already owns the proposed fix path.

## What This Changes

Webhook hook sessions request an explicit way to reuse an existing session when a stable session key is provided, enabling multi-turn context.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918, or an equivalent explicit persistent hook-session mode that keeps isolation as the default and preserves hook session-key policy gates.

## Reproduction Assessment

Yes, source-level. Approved stable-key hook-agent requests become `sessionTarget: "isolated"` cron jobs, which makes `forceNew` true and prevents `resolveCronSession` from reusing the stored session id.

## Solution Assessment

Unclear as a final product choice, but explicit `sessionMode: "isolated" | "persistent"` is the safest evidenced direction. Automatic reuse for every stable key would need a stronger compatibility and security decision.
