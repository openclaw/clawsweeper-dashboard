## Summary

Keep open: current main and the latest release still route hook-agent calls through isolated cron-style runs that force a fresh session, while the linked implementation PR is open but not merged.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing an existing session when a stable sessionKey is supplied, preferably through an explicit backward-compatible persistent session mode.

## Best Possible Solution

Review and land, revise, or close the linked persistent hook-session PR; close this issue only after persistent hook sessions are merged or maintainers explicitly reject the product direction.

## Reproduction Assessment

Yes. Source inspection shows hook-agent requests can pass a stable sessionKey into the cron-style runner, but dispatch still creates an isolated job and the runner passes forceNew for isolated jobs, preventing resolveCronSession from reusing an existing sessionId.

## Solution Assessment

Yes for the explicit sessionMode variant, not for automatic reuse by default. An opt-in persistent hook session mode preserves the secure isolated default while enabling trusted multi-turn webhook integrations.
