## Summary

Keep open. Current main still routes direct and mapped hook-agent requests through the isolated cron runner, so a stable hook `sessionKey` still reaches session resolution with `forceNew: true` and cannot reuse the stored transcript session id. The recent maintainer and ClawSweeper comments match the current source; the related fix PRs are useful prior art but were closed unmerged.

## What This Changes

The issue requests multi-turn webhook hook support by reusing an existing hook session when callers provide a consistent `sessionKey`, preferably through an explicit persistent-session mode.

## Best Possible Solution

Expose a documented, backward-compatible hook option such as `sessionMode: "isolated" | "persistent"` for direct `/hooks/agent` payloads, hook mappings, and transform results. Keep `isolated` as the default, preserve existing session-key policy checks, and only let the explicit persistent hook-agent path resolve sessions without `forceNew`.

## Reproduction Assessment

Yes. A direct or mapped hook with a stable `sessionKey` is normalized and dispatched as an isolated cron-style job, which passes `forceNew: true`; the existing session test confirms forced-new resolution creates a different session id despite a fresh stored entry.

## Solution Assessment

Yes. An explicit persistent-session opt-in is the narrowest maintainable fix because it enables the requested multi-turn webhook behavior without changing default hook or cron isolation; automatic reuse from key shape would be less safe.
