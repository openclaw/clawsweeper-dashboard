## Label Justifications

- `P2`: Stable webhook sessions losing transcript continuity is a normal-priority gateway/session improvement with real user impact and a bounded implementation surface.
- `impact:session-state`: The report is about stable hook `sessionKey` values failing to preserve conversation context across webhook turns.
- `impact:data-loss`: Each hook invocation can orphan the user's expected prior transcript from the active stable session, even though the old transcript files still exist.
- `impact:security`: The requested persistent mode lets external hook inputs influence session continuity and must keep explicit session-key authorization boundaries.

## Summary

Keep open. Current main and the latest release still route hook-agent requests through isolated cron execution with forced fresh sessions, while https://github.com/openclaw/openclaw/pull/75918 is the active closing candidate for the explicit persistent hook-session mode.

## What This Changes

The issue requests opt-in multi-turn webhook hook sessions so a stable hook `sessionKey` can reuse conversation history instead of starting a fresh isolated session every turn.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/75918 with explicit default-isolated `sessionMode` support across direct hook payloads, mappings, transforms, schema/docs, and focused hook/cron regression coverage.

## Reproduction Assessment

Yes. Source inspection shows hook-agent dispatch still forces `sessionTarget: "isolated"`, the runner passes `forceNew: true`, and the reusable session helper is gated behind `!forceNew`; I did not run a live webhook in this read-only review.

## Solution Assessment

Yes. The explicit `sessionMode: "isolated" | "persistent"` path is the best maintainable shape because it preserves the secure isolated default while allowing trusted integrations to opt into reuse; automatic reuse would be a riskier behavior change.
