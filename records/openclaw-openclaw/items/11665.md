## Summary

Keep open. Current main still routes hook-agent dispatch through the isolated cron runner, and the linked closing PR is open rather than merged, so this issue should remain the canonical tracking item until that implementation or an equivalent one lands.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing existing transcript identity when a stable `sessionKey` is supplied.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918, or an equivalent explicit persistent hook-session implementation that defaults to isolation, preserves session-key policy gates, updates config/docs, and closes this issue after merge.

## Reproduction Assessment

Yes, by source inspection on current main. Stable-key hook-agent requests can resolve a `sessionKey`, but dispatch still sets `sessionTarget: "isolated"`, which makes the runner pass `forceNew: true` and prevents session id reuse.

## Solution Assessment

Yes at the direction level. An explicit `sessionMode: "isolated" | "persistent"` opt-in is the narrow maintainable solution because it enables trusted webhook continuity while preserving default isolation and existing session-key gates.
