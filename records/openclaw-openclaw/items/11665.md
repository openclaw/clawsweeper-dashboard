## Label Justifications

- `P2`: This is a normal-priority hook session-continuity improvement with real user impact but bounded gateway/config/cron scope.
- `impact:session-state`: The issue is about stable hook session keys failing to preserve conversation/session continuity across turns.
- `impact:data-loss`: The reported behavior loses usable conversation history for webhook integrations and leaves orphaned transcript files from repeated fresh sessions.
- `impact:security`: The fix would make externally influenced hook session keys control persistent context reuse, so the session-key trust boundary must be reviewed.

## Summary

Keep open: current main still routes direct and mapped hook-agent requests through the isolated cron runner, which forces a fresh session id even when the hook supplies the same stable session key. The linked implementation PR is still open, so this issue should stay open until that PR lands or maintainers choose a different session-mode design.

## What This Changes

The issue requests persistent multi-turn hook-agent sessions so stable webhook session keys can reuse conversation history instead of creating a fresh isolated run on each delivery.

## Best Possible Solution

Review and land a scoped version of https://github.com/openclaw/openclaw/pull/75918, or an equivalent maintainer-approved PR, that keeps isolated as the default and adds explicit persistent sessionMode plumbing across direct hooks, mappings, transforms, config schema, docs, and focused tests.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent requests with sessionTarget isolated, passes forceNew true, and resolveCronSession only reuses an existing session when forceNew is false; I did not run a live webhook repro in this read-only review.

## Solution Assessment

Yes, with maintainer approval. The explicit sessionMode direction is safer than automatic reuse because it preserves default isolation and makes persistent hook context an operator-controlled choice.
