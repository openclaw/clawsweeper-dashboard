## Label Justifications

- `P2`: Stable-key webhook session continuity affects real automations, but the affected surface is bounded to hook-agent session behavior and already has open implementation candidates.
- `impact:session-state`: Hook-agent runs with the same stable session key still roll to a fresh session id, losing transcript continuity.
- `impact:data-loss`: Prior hook conversation history becomes unavailable to later stable-key runs and older transcript files can be orphaned from the latest session-key mapping.
- `impact:security`: The requested fix changes when hook-selected session keys can reuse existing transcript context, which is a security boundary.

## Summary

Keep open: current main and v2026.5.22 still force hook-agent runs through fresh isolated sessions, while open linked PRs already target the root cause and need maintainer product/security review before this can close.

## What This Changes

Webhook hook sessions should support multi-turn reuse when a trusted integration provides a stable session key.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75918 or an equivalent explicit persistent hook-session design, reconcile https://github.com/openclaw/openclaw/pull/70895, keep isolated as the default, and preserve hook session-key security gates.

## Reproduction Assessment

Yes, source-level. Current main routes hook-agent dispatch through an isolated cron job, passes forceNew for isolated runs, and therefore cannot reach the existing resolveCronSession reuse branch for stable hook keys.

## Solution Assessment

Unclear as a final product choice. The explicit sessionMode design is the safest evidenced direction for compatibility and security, while automatic stable-key reuse needs stronger maintainer approval.
