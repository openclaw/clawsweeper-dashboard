## Label Justifications

- `P2`: This has real production workflow impact for webhook agents but is bounded to hook session behavior and has open implementation candidates.
- `impact:session-state`: Stable-key webhook sessions lose transcript continuity because hook-agent runs are forced into fresh isolated session ids.
- `impact:data-loss`: Prior hook turns become unavailable to later agent runs and old transcript files can become orphaned from the latest stable session-key mapping.
- `impact:security`: The requested fix changes when hook-selected session keys can reuse existing transcript context, which is a security boundary.

## Summary

Keep open: current main and `v2026.5.20` still force stable-key hook-agent calls through isolated fresh sessions, and open implementation PRs need maintainer review or landing before this issue can close.

## What This Changes

Allow webhook hook-agent runs with a stable `sessionKey` to reuse session history for multi-turn context, preferably through an explicit persistent session mode.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75918 or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation, reconcile https://github.com/openclaw/openclaw/pull/70895, keep `isolated` as the default, and preserve hook session-key security gates.

## Reproduction Assessment

Yes, source-level. Current main routes hook-agent dispatch through an isolated cron job, passes `forceNew` for isolated runs, and therefore cannot reach the existing `resolveCronSession` reuse branch for stable hook keys.

## Solution Assessment

Unclear as a final product choice, but explicit `sessionMode: "isolated" | "persistent"` is the safest evidenced direction. Automatic reuse for every stable hook key needs stronger compatibility and security approval.
