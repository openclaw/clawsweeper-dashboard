## Summary

Keep open. Current main still forces hook-agent requests with stable session keys through isolated execution, and open PR #75918 already uses closing syntax for the explicit persistent-session mode, so the issue should stay open until that PR or an equivalent implementation lands.

## What This Changes

The issue requests a persistent multi-turn hook-agent session mode so stable webhook `sessionKey` values can reuse conversation history.

## Best Possible Solution

Review and land #75918, or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation that defaults to isolation, scopes replay/idempotency by mode, and documents the trust boundary.

## Reproduction Assessment

Yes, from source inspection. A direct or mapped hook with a stable `sessionKey` still becomes an isolated job and reaches `resolveCronSession` with `forceNew: true`, which mints a fresh session id.

## Solution Assessment

Yes. An explicit persistent-session opt-in is the narrowest maintainable direction because it enables trusted multi-turn webhook threads without changing the default isolated hook behavior.
