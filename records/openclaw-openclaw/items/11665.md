## Label Justifications

- `P2`: Stable-key webhook sessions have real production workflow impact, but the issue is bounded to hook session behavior and has open implementation candidates.
- `impact:session-state`: Stable-key hook-agent runs lose transcript continuity because the current hook path is forced into a fresh isolated session id.
- `impact:data-loss`: Earlier webhook conversation history becomes unavailable to later runs and old transcript files can be orphaned from the latest stable session-key mapping.
- `impact:security`: The requested fix changes when hook-selected session keys can reuse existing transcript context, which is a security boundary.

## Summary

Keep open: current main and the latest release still force stable-key hook-agent runs through fresh isolated sessions, and the open implementation PRs need maintainer product/security review before this issue can close.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing existing transcript state when a stable hook sessionKey is supplied.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75918 or an equivalent explicit persistent hook-session design, reconcile https://github.com/openclaw/openclaw/pull/70895, keep isolated as the default, and preserve hook session-key security gates.

## Reproduction Assessment

Yes, source-level. Current main routes hook-agent dispatch through an isolated cron job, passes forceNew for isolated runs, and therefore cannot reach the existing resolveCronSession reuse branch for stable hook keys.

## Solution Assessment

Unclear as a final product choice. The explicit sessionMode design is the safest evidenced direction for compatibility and security, while automatic reuse needs stronger maintainer approval.
