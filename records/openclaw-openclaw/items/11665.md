## Summary

Keep open. Current main still forces hook-agent runs through isolated session resolution, and #75918 is an open unmerged PR with closing syntax for this issue, so the issue should remain open until that PR lands or is closed.

## What This Changes

Webhook hook-agent calls with a stable `sessionKey` should be able to opt into reusing the same transcript session for multi-turn conversations instead of always starting a fresh isolated run.

## Best Possible Solution

Review and land or close #75918, or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation that preserves default isolation and hook session-key policy checks.

## Reproduction Assessment

Yes. A direct or mapped hook with a stable `sessionKey` still flows through `dispatchAgentHook`, becomes an isolated cron-style job, and reaches `resolveCronSession` with `forceNew: true`; the existing unit test confirms that path mints a new session id.

## Solution Assessment

Yes. An explicit persistent-session opt-in is the narrowest maintainable solution because it enables multi-turn webhook threads without changing the default isolated hook behavior or relying on unsafe key-shape inference.
