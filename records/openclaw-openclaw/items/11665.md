## Summary

Keep open. Current main still forces hook-agent requests with stable `sessionKey` values through isolated session resolution, and #75918 is an open unmerged closing PR for the remaining work.

## What This Changes

The issue requests multi-turn webhook hook sessions by allowing stable hook `sessionKey` values to reuse an existing transcript, preferably through an explicit persistent session mode.

## Best Possible Solution

Review and land or close #75918, or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation that preserves default isolation while allowing trusted, scoped hook sessions to reuse transcripts.

## Reproduction Assessment

Yes. A direct or mapped hook with a stable `sessionKey` still flows through `dispatchAgentHook`, becomes an isolated cron-style job, and reaches `resolveCronSession` with `forceNew: true`; existing unit coverage confirms that path mints a new session id.

## Solution Assessment

Yes. An explicit persistent-session opt-in is the narrowest maintainable solution because it enables multi-turn webhook threads without changing default isolated hook behavior or inferring safety from key shape alone.
