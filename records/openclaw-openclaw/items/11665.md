## Label Justifications

- `impact:data-loss`: Forced-new session identity discards expected prior context and can leave transcript files orphaned.

## Summary

Current main still forces hook-agent requests through fresh isolated session identity, and the issue has a mergeable open closing PR with live proof; it should remain open until maintainers review and either land or reject that canonical implementation.

## What This Changes

Add an explicit persistent hook-session mode so stable webhook session keys can retain multi-turn context while isolated behavior remains the default.

## Best Possible Solution

Use the linked PR as the canonical implementation path, narrowing it if necessary, and land only an explicit persistent mode that preserves isolated defaults, request-key authorization, replay separation, and consistent direct-payload, mapping, transform, schema, docs, and upgrade behavior.

## Reproduction Assessment

Yes, at source level with high confidence: repeated hook-agent requests use an isolated job, which passes `forceNew: true`, so a stable hook key cannot reach the resolver's existing reuse branch. This read-only issue review did not execute a runtime reproduction.

## Solution Assessment

Yes in direction: an explicit opt-in persistent mode with isolated as the default is safer and more maintainable than automatically reusing every stable key. The linked PR still needs maintainer review for its broader diff, security boundary, and compatibility surface.
