## Label Justifications

- `P2`: The issue affects real webhook integrations and conversation continuity, but the blast radius is bounded to hook-agent session handling.
- `impact:session-state`: Stable hook session keys currently fail to preserve multi-turn conversation state.
- `impact:data-loss`: The report describes lost conversation history and accumulating orphaned transcript files when repeated webhook calls mint fresh session ids.
- `impact:security`: The fix direction changes when externally triggered hook traffic can reuse existing agent context, which needs explicit session-key and trust-boundary controls.

## Summary

Keep open: current main and the latest release still force hook-agent runs through isolated session handling, and the remaining work is now owned by open implementation PRs rather than a merged fix.

## What This Changes

Request to add persistent multi-turn webhook hook sessions so stable `sessionKey` values can reuse existing conversation context.

## Best Possible Solution

Land one maintainer-approved persistent hook-session implementation, preferably explicit `sessionMode: "isolated" | "persistent"`, preserving default isolation and documenting the security model.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent requests through an isolated CronJob, passes `forceNew` for that target, and only reuses an existing session id when `forceNew` is false.

## Solution Assessment

Unclear for the issue alone because the open linked PRs still need review. The explicit persistent `sessionMode` approach is the best-supported direction so far because it preserves isolated defaults while giving trusted integrations an opt-in path.
