## Summary

Keep open. Current main still forces stable-key hook-agent requests through isolated cron sessions, and the open closing PR is an implementation candidate rather than a reason to close before merge.

## What This Changes

Allow webhook hook-agent calls with a consistent `sessionKey` to support multi-turn session reuse instead of starting a fresh transcript on every invocation.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918, or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation that defaults to isolation, preserves session-key policy gates, updates generated config/docs, and then closes this issue after merge.

## Reproduction Assessment

Yes, by source inspection on current main. Direct and mapped hook-agent calls can resolve a stable `sessionKey`, but dispatch sets `sessionTarget: "isolated"`, the runner passes `forceNew: true`, and `resolveCronSession` cannot reuse the stored session id.

## Solution Assessment

Yes at the direction level. An explicit persistent-session mode is the narrow maintainable fix because it enables trusted webhook continuity without changing default isolation; automatic deterministic reuse remains a product and security policy choice.
