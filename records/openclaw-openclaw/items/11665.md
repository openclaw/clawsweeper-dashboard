## Label Justifications

- `P2`: This is a concrete gateway/session feature with production workflow impact, but it needs opt-in behavior and review rather than emergency handling.
- `impact:session-state`: Stable-key webhook sessions lose multi-turn transcript continuity because each isolated hook run is forced into a fresh session id.
- `impact:data-loss`: Prior hook turns become unavailable to the agent and old transcript files can be orphaned from the latest stable session-key mapping.
- `impact:security`: The requested fix changes when hook-selected session keys can reuse existing transcript context.

## Summary

Keep open: current main and the latest release still route hook-agent calls with stable session keys through isolated forced-new sessions, and the open implementation PR needs maintainer review/landing before this issue can close.

## What This Changes

Request persistent multi-turn webhook hook-agent sessions when a stable `sessionKey` is supplied, likely through an explicit hook session mode.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75918 or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation that keeps isolation as the default, preserves hook session-key security gates, and reconciles https://github.com/openclaw/openclaw/pull/70895.

## Reproduction Assessment

Yes, source-level. Current main routes hook-agent dispatch through an isolated cron job, passes `forceNew` for isolated runs, and therefore cannot reach the existing `resolveCronSession` reuse branch for stable hook keys.

## Solution Assessment

Unclear as a final product choice, but explicit `sessionMode: "isolated" | "persistent"` is the safest evidenced direction. Automatic reuse for every stable hook key would need stronger compatibility and security approval.
