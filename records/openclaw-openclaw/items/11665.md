## Summary

Keep open: current main still blocks stable-key webhook session reuse, and an open closing PR already targets this issue, so the issue should stay open until maintainers land or close that implementation path.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing conversation state when a hook supplies the same stable `sessionKey`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918, or an equivalent explicit persistent hook-session mode, while preserving default isolation, policy gates, docs, schema, and runtime proof.

## Reproduction Assessment

Yes. By source inspection on current main, approved stable-key hook requests can pass session-key policy, but hook dispatch still sets `sessionTarget: "isolated"`, causing `forceNew: true` and blocking stored session-id reuse.

## Solution Assessment

Unclear as the final product choice. Explicit `sessionMode: "isolated" | "persistent"` is the safer direction if maintainers want persistence to be an opt-in security decision rather than automatic reuse.
