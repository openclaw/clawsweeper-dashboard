## Label Justifications

- `P2`: The missing capability causes real webhook conversation failures and recurring token overhead, but its blast radius is limited to hook integrations requiring multi-turn state.
- `impact:session-state`: Stable webhook identities currently cannot retain the same conversation transcript across turns.
- `impact:data-loss`: Each fresh session identity makes earlier conversation context unavailable to later turns and can orphan prior transcript files from the active mapping.
- `impact:security`: Reusing externally selected session identities requires explicit authorization and tenant-safe key scoping to prevent context exposure.

## Summary

Current main still cannot reuse a stored conversation for stable webhook session keys, while the open linked PR is a viable canonical implementation with positive live proof. Keep this issue open until that PR is reviewed and either merged or explicitly declined.

## What This Changes

Add an opt-in persistent mode so webhook hook calls with a stable session key can continue the same conversation.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918 after rebasing it onto current main, preserving isolated behavior as the default, retaining existing request session-key restrictions, and validating persistent reuse plus default isolation.

## Reproduction Assessment

Yes. Current-main source shows that hook-agent requests remain isolated and that stored session IDs are reusable only when `forceNew` is disabled, but no hook persistence mode can select that path.

## Solution Assessment

Yes. An explicit `sessionMode: "isolated" | "persistent"` is safer and more maintainable than automatic reuse because it preserves existing secure isolation and makes persistence an operator-controlled trust decision.
