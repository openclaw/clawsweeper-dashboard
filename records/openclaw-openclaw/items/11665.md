## Label Justifications

- `P2`: This is a normal-priority webhook session continuity feature with real user impact but bounded hook/config/runtime scope.
- `impact:session-state`: Stable webhook keys currently do not preserve the intended conversation/session state across hook turns.
- `impact:data-loss`: The reported behavior makes previous hook conversation context unavailable to later turns and can leave orphaned transcript/session files.
- `impact:security`: The requested persistent mode changes when external hook callers may intentionally attach to existing session state.

## Summary

Keep open: current main and the latest release still route hook-agent requests through the isolated cron runner without any persistent hook-session mode, and https://github.com/openclaw/openclaw/pull/75918 is the open closing PR that now owns the implementation path.

## What This Changes

Request persistent webhook hook sessions so a stable hook `sessionKey` can preserve multi-turn context instead of creating a fresh isolated session each invocation.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/75918 with an explicit opt-in `sessionMode: "isolated" | "persistent"` path that preserves default isolation and hook session-key policy.

## Reproduction Assessment

Yes, at source level. Current main creates isolated hook-agent cron jobs and passes `forceNew: true`, while the only reuse helper is gated behind `!forceNew`; I did not run a live webhook because this review is read-only.

## Solution Assessment

Yes, the explicit opt-in `sessionMode` direction is the best fit because it preserves current isolated defaults while enabling trusted stable hook keys to reuse sessions. Automatic reuse would be riskier because hook callers and mappings are external input surfaces.
