## Label Justifications

- `P2`: The issue affects webhook hook session continuity in real workflows but has a bounded gateway/cron/config surface.
- `impact:session-state`: Stable webhook session keys currently do not preserve the intended multi-turn session context.
- `impact:data-loss`: Each webhook invocation can start a new transcript/session file, leaving prior conversation history unavailable to the next turn.
- `impact:security`: The requested persistent mode changes when external hook callers may reuse existing context and therefore needs explicit trust and session-key controls.

## Summary

Keep open. Current main still forces hook-agent runs through isolated cron session creation, and the issue is already paired with an open closing PR, so the right cleanup path is to review or replace that PR rather than close this issue now.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing session state when hook calls provide a stable sessionKey.

## Best Possible Solution

Review, narrow if needed, and land a backward-compatible `sessionMode: "isolated" | "persistent"` hook option that preserves isolated defaults and existing hook session-key policy.

## Reproduction Assessment

Yes. Source inspection shows direct and mapped hook-agent dispatch still builds an isolated cron-style job, and isolated cron execution passes `forceNew: true` to `resolveCronSession`, so a stable hook `sessionKey` cannot reuse the previous session on current main.

## Solution Assessment

Yes for the preferred direction, but not yet complete on main. The best solution is the explicit opt-in persistent mode from the linked implementation path, not automatic reuse for every stable hook key because hook session reuse is security-sensitive.
