## Label Justifications

- `P2`: This affects real webhook integrations but is a bounded hook/session continuity issue rather than a core outage.
- `impact:session-state`: Stable hook keys currently do not preserve the expected multi-turn conversation session state.
- `impact:data-loss`: The report describes expected conversation history being effectively lost across turns and orphaned transcript files accumulating.
- `impact:security`: The requested persistent mode touches external hook session-key routing and must keep policy safeguards intact.

## Summary

Keep open: current main and the latest release still route hook-agent runs through isolated cron session handling, so stable hook session keys cannot reuse a fresh existing session id. The issue is now paired with an open implementation PR, so it should stay open until that PR or a successor lands.

## What This Changes

Webhook hook sessions should support multi-turn conversation reuse when callers provide a stable, policy-scoped sessionKey.

## Best Possible Solution

Review, narrow if needed, and land https://github.com/openclaw/openclaw/pull/75918 or a successor that adds explicit persistent hook session mode while preserving default isolation and session-key policy.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the gap: direct and mapped hook-agent runs still use isolated cron jobs, and isolated jobs still force a fresh session id even when the hook sessionKey is stable.

## Solution Assessment

Yes, the explicit `sessionMode: "isolated" | "persistent"` direction is the best known shape because automatic reuse would silently weaken today’s isolated default; the open PR still needs normal maintainer review before it becomes the landed fix.
