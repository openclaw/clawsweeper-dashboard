## Summary

Keep open: current main and the latest release still force hook-agent runs through isolated cron session creation, while an open linked PR now owns the explicit persistent-session implementation path.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing an existing session when a stable hook `sessionKey` is supplied, preferably through an explicit persistent session mode.

## Best Possible Solution

Review and land a focused explicit persistent hook-session mode that keeps isolated as the default, threads the mode through direct payloads, mappings, transforms, schema/help/docs, and tests, and retires competing implicit persistence branches.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent requests as isolated cron jobs, passes `forceNew: true`, and only reuses existing sessions when `forceNew` is false; I did not run a live webhook in this read-only review.

## Solution Assessment

Yes on direction, not automatic closure. Explicit `sessionMode: "isolated" | "persistent"` is safer than automatic reuse because it preserves default isolation while letting trusted hooks opt into context reuse.
