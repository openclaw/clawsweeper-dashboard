## Label Justifications

- `P2`: This is a normal-priority hook session-continuity improvement with real user impact but bounded gateway/config/cron scope.
- `impact:session-state`: Stable webhook `sessionKey` values currently do not preserve the transcript session identity used for multi-turn context.
- `impact:data-loss`: The reported behavior loses usable conversation history between webhook turns and leaves prior generated transcript files orphaned.
- `impact:security`: The requested persistent mode changes how hook-provided or templated session keys select shared transcript context, so the trust boundary needs explicit review.

## Summary

Keep open: current main and the latest release still force hook-agent requests through isolated cron runs, and the remaining work is already paired with the open closing implementation candidate at https://github.com/openclaw/openclaw/pull/75918.

## What This Changes

Request webhook hook sessions with stable `sessionKey` values to support multi-turn transcript reuse instead of minting a fresh session for every hook-agent invocation.

## Best Possible Solution

Review and land a narrow explicit `sessionMode: "isolated" | "persistent"` hook implementation through https://github.com/openclaw/openclaw/pull/75918, keeping isolated as the default and preserving hook session-key security controls.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent requests as isolated cron jobs, passes `forceNew: true`, and only reuses an existing session when `forceNew` is false; I did not run a live webhook in this read-only review.

## Solution Assessment

Yes for the explicit opt-in direction, but not for automatic default reuse. The maintainable fix is a `sessionMode` surface with isolated as the default, scoped session-key policy, docs, schema metadata, and focused hook/cron regression coverage.
