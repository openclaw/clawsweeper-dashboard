## Label Justifications

- `P2`: This is a normal-priority hook session-continuity feature with real user impact but a bounded gateway/config/runtime surface.
- `impact:session-state`: Stable hook session keys currently cannot preserve conversation state because the hook path forces fresh isolated sessions.
- `impact:data-loss`: The reported behavior loses conversational context across webhook turns and leaves orphaned transcript files.
- `impact:security`: The requested persistent mode depends on preserving explicit opt-in and session-key authorization boundaries for externally triggered hooks.

## Summary

Keep open. Current main and the latest release still force hook-agent work through isolated cron sessions with no hook `sessionMode` surface, and the linked implementation PR is open but unmerged.

## What This Changes

Webhook hook sessions should support multi-turn reuse when integrations provide a stable, scoped hook session key.

## Best Possible Solution

Review and land a narrow explicit `sessionMode: "isolated" | "persistent"` implementation that preserves isolated defaults and session-key policy, then close this issue after the linked PR or an equivalent replacement merges.

## Reproduction Assessment

Yes. Source inspection of current main shows hook-agent dispatch uses isolated cron jobs, the runner passes `forceNew: true`, and the resolver only reuses a stored session when `forceNew` is false; I did not run a live webhook in this read-only review.

## Solution Assessment

Yes, the explicit opt-in `sessionMode` direction is the best current path because it preserves default isolation and keeps session reuse behind a trusted hook configuration boundary. This issue should stay open until the linked PR lands or maintainers choose a replacement path.
