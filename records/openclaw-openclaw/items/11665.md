## Label Justifications

- `P2`: This is a meaningful production webhook limitation with bounded scope and an active implementation path.
- `impact:session-state`: Repeated hook invocations with the same logical key currently do not retain the same conversation transcript.
- `impact:data-loss`: Prior conversation context becomes unavailable to later turns and superseded transcript files can accumulate as orphaned state.
- `impact:security`: Persistent caller-selected session identities require trusted scoped keys to prevent cross-caller transcript exposure.

## Summary

Current main still lacks persistent hook sessions, while the open linked PR is a viable implementation candidate with live reuse proof; keep the issue open until that PR is landed or explicitly rejected.

## What This Changes

Add an opt-in persistent session mode for direct and mapped webhook agent hooks so stable, scoped session keys can retain multi-turn context.

## Best Possible Solution

Land a focused version of the linked PR that keeps isolation as the default, permits persistence only for explicit trusted prefix-scoped session keys, and separates unrelated runtime changes unless maintainers intentionally approve the combined scope.

## Reproduction Assessment

Yes, at source level with high confidence: current main's isolated hook-agent path does not expose the resolver's reuse mode, and the linked PR demonstrates live after-change reuse. The path was not executed locally because this review is read-only.

## Solution Assessment

Yes in direction, but not necessarily in the current branch shape: explicit isolated-by-default persistence is safer than unconditional reuse, while unrelated runtime changes should be split or explicitly approved.
