## Label Justifications

- `P2`: This is a normal-priority hook/session continuity problem with real user impact but a bounded gateway/config/session surface.
- `impact:session-state`: Stable hook `sessionKey` values currently do not preserve the conversation session, causing context and transcript continuity loss.
- `impact:data-loss`: The report describes old transcript files becoming orphaned while only the latest session id remains mapped for the hook key.
- `impact:security`: The requested persistent mode crosses a trust boundary where externally influenced session keys can select shared conversation context unless scoped carefully.

## Summary

Current `main` and latest release still route hook-agent requests through isolated cron sessions that force a new session id, and an open closing PR now owns the persistent hook-session implementation, so this issue remains necessary until that PR merges or is rejected.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing the existing conversation when a stable hook `sessionKey` is supplied.

## Best Possible Solution

Review and either land or reject the open persistent hook-session PR; once a reviewed implementation merges, close this issue as implemented with the merge commit and release provenance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: direct or mapped hook-agent dispatch with the same `sessionKey` still runs as `sessionTarget: "isolated"`, which passes `forceNew: true` and prevents `resolveCronSession` from reusing the stored session id; I did not run a live webhook in this read-only review.

## Solution Assessment

Unclear until the open implementation PR is accepted or revised. The explicit `sessionMode: "isolated" | "persistent"` direction appears maintainable because it keeps isolated as the default, but it needs maintainer trust-model review and PR validation before this issue is solved.
