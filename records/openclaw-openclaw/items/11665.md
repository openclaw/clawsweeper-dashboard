## Label Justifications

- `P2`: This is a concrete hook/session improvement with production impact, but it requires opt-in behavior and security review rather than emergency handling.
- `impact:session-state`: Stable-key webhook sessions lose multi-turn transcript continuity because each hook run is forced into a new session id.
- `impact:data-loss`: The reported behavior orphaned prior transcript files from the stable session key mapping and made previous turns unavailable to the agent.
- `impact:security`: The requested fix changes when hook-selected session keys can reuse existing transcript context.

## Summary

Keep open: current main still routes stable-key hook-agent requests through forced isolated cron sessions, and the open closing PR at https://github.com/openclaw/openclaw/pull/75918 is unmerged and currently conflicting.

## What This Changes

Webhook hook-agent requests with a stable `sessionKey` should support multi-turn transcript reuse, likely through an explicit persistent session mode.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918, or an equivalent explicit persistent hook-session mode that keeps isolation as the default and preserves hook session-key policy gates.

## Reproduction Assessment

Yes, source-level. Approved stable-key hook requests become `sessionTarget: "isolated"` cron jobs, which makes `forceNew` true and prevents `resolveCronSession` from reusing the stored session id.

## Solution Assessment

Unclear as a final product choice, but explicit `sessionMode: "isolated" | "persistent"` is the safest direction currently evidenced. Automatic reuse for every stable key would need a stronger compatibility and security decision.
