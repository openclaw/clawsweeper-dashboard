## Summary

Keep open. Current main still source-reproduces the hook-session gap, and a live open PR explicitly closes this issue, so the issue should stay open until that implementation is landed or rejected.

## What This Changes

Request to let webhook hook-agent calls with stable session keys opt into multi-turn session reuse instead of always starting a fresh isolated session.

## Best Possible Solution

Land one reviewed persistent hook session implementation that keeps isolated as the default, preserves hook session-key allowlist checks, updates direct payloads/mappings/transforms/schema/docs/tests, and then let the linked PR close this issue.

## Reproduction Assessment

Yes, source-reproducible: current main routes `/hooks/agent` through an isolated cron-style job, and that runner passes `forceNew: true` for isolated jobs. I did not run a live webhook, but the source path and existing tests establish why stable hook keys still get a new session id.

## Solution Assessment

Yes in direction: explicit `sessionMode: "isolated" | "persistent"` is the best current shape because it preserves secure default isolation while enabling trusted multi-turn hook sessions. The actual merge decision belongs on the open linked PR.
