## Label Justifications

- `P2`: This is a normal-priority feature/repair for webhook session continuity with real user impact but bounded hook/config scope.
- `impact:session-state`: Stable hook session keys currently do not preserve conversation state because the hook path rolls to a fresh session id.
- `impact:data-loss`: The report describes lost conversational history and orphaned transcript files across repeated webhook invocations.
- `impact:security`: The requested persistent mode changes how external hook callers can target stored sessions, so auth and session-key boundaries are central to the fix.

## Summary

Keep open: current main still routes hook-agent requests through isolated cron runs that force fresh sessions, and the issue is already paired with an open mergeable candidate PR that uses closing syntax but has not merged.

## What This Changes

The issue requests persistent multi-turn webhook hook sessions when a trusted hook supplies a stable sessionKey.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/75918, keeping isolated as the default and adding an explicit persistent hook session mode with docs, schema, security, and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows `/hooks/agent` creates an isolated cron-style job, the isolated runner passes `forceNew: true`, and `resolveCronSession` only reuses stored session ids when `forceNew` is false; I did not run a live webhook repro in this read-only review.

## Solution Assessment

Yes, the best visible direction is an explicit `sessionMode: "isolated" | "persistent"` hook option rather than automatic reuse. That keeps the current safe default while giving trusted integrations a documented opt-in path.
