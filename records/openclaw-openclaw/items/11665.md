## Summary

Keep open. Current main still forces hook-agent runs through isolated cron-style execution, and an open unmerged closing PR already targets the persistent hook-session mode requested here.

## What This Changes

Allow hook-agent webhooks with stable sessionKey values to opt into reusing the existing session transcript for multi-turn conversations.

## Best Possible Solution

Review and land #75918, or an equivalent explicit sessionMode: "isolated" | "persistent" implementation that keeps isolation as the default and documents the trust boundary.

## Reproduction Assessment

Yes, from source inspection. A direct or mapped hook with a stable sessionKey still becomes an isolated cron-style job and reaches resolveCronSession with forceNew: true on current main.

## Solution Assessment

Yes. An explicit persistent-session opt-in is the narrowest maintainable direction because it enables trusted multi-turn webhook threads without changing default isolated hook behavior or inferring safety from key shape alone.
