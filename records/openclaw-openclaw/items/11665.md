## Summary

Keep open: current main still forces hook-agent requests through isolated cron runs, so stable hook `sessionKey` values cannot reuse existing fresh session ids, and the issue is already paired with an open closing PR that has not merged.

## What This Changes

The issue requests persistent multi-turn webhook hook sessions so a stable hook `sessionKey` can reuse an existing session instead of starting a new isolated session for each invocation.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/75918 as the canonical fix path, preserving isolated default behavior and adding an explicit, documented persistent hook-session opt-in.

## Reproduction Assessment

Yes. Source inspection on current main shows `/hooks/agent` creates an isolated cron job, the runner passes `forceNew=true`, and `resolveCronSession` only reuses an existing `sessionId` when `forceNew` is false.

## Solution Assessment

Unclear until maintainers finish reviewing the open fix PR. The explicit `sessionMode: "isolated" | "persistent"` direction is the best visible shape because it preserves the safe default while giving trusted hooks a narrow opt-in path.
