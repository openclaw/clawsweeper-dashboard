## Label Justifications

- `P2`: This is a concrete hook/session improvement with production impact, but it needs an opt-in product/security decision rather than emergency handling.
- `impact:session-state`: The reported behavior loses multi-turn transcript continuity for stable-key webhook sessions.
- `impact:data-loss`: Each invocation creates a fresh transcript session and leaves old transcript files orphaned from the stable session key mapping.
- `impact:security`: The requested fix changes how hook-selected session keys can reuse existing transcript context.

## Summary

Keep open: current main still forces hook-agent runs through isolated cron sessions, so stable hook session keys cannot reach the existing session-id reuse branch. The open closing PR at https://github.com/openclaw/openclaw/pull/75918 targets this issue but is unmerged and currently conflicting, so the issue should stay open until maintainers land or close that implementation path.

## What This Changes

Request persistent multi-turn webhook hook sessions for stable `sessionKey` values, preferably through an explicit hook session mode.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918, or an equivalent explicit persistent hook-session mode that keeps isolation as the default and preserves the existing hook session-key policy gates.

## Reproduction Assessment

Yes. Source inspection on current main shows stable-key hook requests can be accepted, but hook dispatch still sets `sessionTarget: "isolated"`, which makes `forceNew` true and prevents `resolveCronSession` from reusing the stored session id.

## Solution Assessment

Unclear as a final product choice, but the safest implementation direction is explicit `sessionMode: "isolated" | "persistent"` with `isolated` as the default. Automatic reuse for all consistent keys would need stronger maintainer security and compatibility approval.
