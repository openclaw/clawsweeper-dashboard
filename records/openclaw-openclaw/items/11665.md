## Summary

Keep open. Current main still forces stable-key hook-agent requests through isolated cron sessions, and the active implementation path is the open closing PR at https://github.com/openclaw/openclaw/pull/75918.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing an existing session when a stable `sessionKey` is supplied, likely through an explicit `sessionMode` option.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918, or an equivalent explicit `sessionMode` implementation that defaults to isolation, preserves session-key policy, and documents the webhook trust boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows a stable-key `/hooks/agent` request is converted to an isolated `CronJob`, then `resolveCronSession` receives `forceNew: true` and cannot reuse the existing session id.

## Solution Assessment

Yes at the direction level. An explicit `sessionMode: "isolated" | "persistent"` option is safer than changing all stable `sessionKey` values to reuse by default because it preserves isolation unless trusted integrations opt in.
