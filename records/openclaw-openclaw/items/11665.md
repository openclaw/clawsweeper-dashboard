## Label Justifications

- `P2`: The missing multi-turn hook behavior affects real integrations but remains bounded to hook configuration, dispatch, and session resolution.
- `impact:session-state`: Repeated hook calls with one logical conversation key currently receive fresh session identities and lose conversation continuity.
- `impact:data-loss`: The active session mapping moves to each new ID while older transcript files become orphaned from the continuing hook conversation.
- `impact:security`: Persistent reuse determines which externally triggered webhook events share trusted agent context and transcript state.

## Summary

Current main still forces hook-agent dispatches into fresh isolated sessions, and the open closing PR provides a concrete opt-in persistent-session implementation with live proof; keep the issue open until that PR is narrowed or explicitly accepted and merged.

## What This Changes

Allow webhook hook-agent calls with a stable session key to opt into reusing an existing session for multi-turn context.

## Best Possible Solution

Land a focused version of the linked PR that keeps isolation as the default, requires trusted scoped session keys for persistent mode, and separates unrelated Codex/harness changes unless maintainers explicitly approve the combined scope.

## Reproduction Assessment

Yes at source level: current main creates an isolated hook job and offers no supported persistent-session mode, while the linked PR demonstrates the corrected reuse behavior in a real deployment. I did not execute a webhook reproduction in this read-only review.

## Solution Assessment

Yes in product direction: an explicit opt-in `isolated | persistent` mode is safer and more maintainable than automatic reuse because it preserves the existing isolation default and exposes the trust boundary. The current linked branch should be narrowed or explicitly approved as combined work before landing.
