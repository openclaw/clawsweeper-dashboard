## Label Justifications

- `P2`: The missing opt-in multi-turn behavior affects production webhook integrations but is bounded to the hook-agent session surface.
- `impact:session-state`: Repeated hook turns using the same stable key cannot retain the same conversation state on current main.
- `impact:data-loss`: Prior conversation context becomes unavailable to later turns and replaced mappings can leave orphaned transcript files.
- `impact:security`: Persistent hook mode changes which externally initiated requests may join existing session state.

## Summary

Keep this issue open. Current main still hard-codes hook-agent dispatch to isolated sessions and exposes no persistent-session option, while the open closing PR https://github.com/openclaw/openclaw/pull/75918 is the active implementation path with substantial live proof.

Likely related people:
- `ryanrhughes` — high confidence; authored the active and prior persistent-hook-session implementations.
- `vincentkoc` — medium confidence; recent contributor to the current hook dispatch and mapping paths.
- `steipete` — medium confidence; reviewed the issue and identified the explicit opt-in design and security boundary.

## What This Changes

Allow trusted hook-agent callers with a stable session key to opt into multi-turn session reuse while preserving isolated behavior by default.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918 if it preserves isolated defaults, enforces existing session-key authorization across direct payloads, mappings, transforms, and replay dedupe, documents the trust model, and proves both persistent reuse and unchanged default isolation.

## Reproduction Assessment

Yes, at source level with high confidence: current hook dispatch always selects isolated mode, and session reuse is disabled when that path requests a fresh session. The linked PR also supplies real-environment evidence demonstrating reuse after its proposed change.

## Solution Assessment

Yes in direction: an explicit `isolated` or `persistent` mode with isolation as the default is safer and more maintainable than automatically reusing every stable key. Final API and security approval should occur in the linked PR review.
