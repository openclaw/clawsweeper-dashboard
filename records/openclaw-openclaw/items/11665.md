## Label Justifications

- `P2`: This is a concrete hook/session improvement with production impact, but it requires opt-in behavior and security review rather than emergency handling.
- `impact:session-state`: Stable-key webhook sessions lose multi-turn transcript continuity because each hook run is forced into a new session id.
- `impact:data-loss`: The reported behavior makes previous stable-key turns unavailable to the agent and leaves old transcript files orphaned from the latest session mapping.
- `impact:security`: The requested fix changes when hook-selected session keys can reuse existing transcript context.

## Summary

Keep open: current main and the latest release still force stable-key hook-agent runs through isolated cron sessions, while the open closing PR is unmerged and conflicting.

## What This Changes

Webhook hook sessions request a persistent multi-turn mode so stable `sessionKey` values can reuse transcript state across hook invocations.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918, or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation that keeps isolation as the default and preserves hook session-key policy gates.

## Reproduction Assessment

Yes, source-level. Approved stable-key hook requests become `sessionTarget: "isolated"` jobs, which makes `forceNew` true and prevents stored session id reuse.

## Solution Assessment

Unclear as a final product choice, but explicit `sessionMode: "isolated" | "persistent"` is the safest evidenced direction. Automatic reuse for every stable key needs stronger compatibility and security approval.
