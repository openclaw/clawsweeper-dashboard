## Label Justifications

- `P2`: The issue affects webhook integrations and session continuity, but it is a bounded hook/config feature rather than an urgent core-runtime outage.
- `impact:session-state`: Stable hook sessionKey values currently do not preserve the conversation session id for hook-agent turns.
- `impact:data-loss`: The reported behavior loses usable transcript continuity and leaves old hook transcript files orphaned from the latest sessionKey mapping.
- `impact:security`: Any fix changes how externally triggered hooks can opt into persistent session context, so the session-key security policy is part of the issue.

## Summary

Keep open. Current main still routes hook-agent requests through isolated cron-style runs that force a fresh session id, and an open linked PR is already the active implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests persistent multi-turn webhook hook sessions when a stable sessionKey is supplied.

## Best Possible Solution

Ship one explicit persistent hook-session mode that preserves isolated as the default, threads through direct payloads, mappings, transforms, schema, docs, and tests, and keeps the hook session-key security policy intact.

## Reproduction Assessment

Yes at source level: current main dispatches hook-agent requests as isolated cron runs, and the isolated runner passes forceNew into the session resolver. I did not run a live webhook reproduction in this read-only sweep.

## Solution Assessment

Yes, the explicit sessionMode direction is the safest visible solution because it preserves the current isolated default while adding a clear operator-controlled persistent path. Automatic reuse by default would be a broader compatibility and security decision.
