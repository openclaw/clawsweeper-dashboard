## Summary

Keep open. Current main still lacks an explicit persistent hook-session mode for stable webhook `sessionKey` values, and the linked closing PR is open, unmerged, and dirty rather than an implemented fix on main.

## What This Changes

The issue requests multi-turn webhook hook sessions by letting stable `sessionKey` values reuse an existing session, preferably through an explicit persistent session mode.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918, or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation that defaults to isolation and documents the webhook trust boundary.

## Reproduction Assessment

Yes, by source inspection on current main. A stable-key hook-agent request becomes an isolated cron job, `resolveCronSession` receives `forceNew: true`, and the existing session id cannot be reused.

## Solution Assessment

Yes at the direction level. An explicit persistent-session mode is the narrow maintainable fix because it enables trusted webhook continuity without changing default isolation; automatic deterministic reuse remains a security/product policy choice.
