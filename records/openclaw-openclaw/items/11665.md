## Summary

Keep open: current main still routes hook-agent requests through isolated cron runs that force fresh session ids, and the issue is already paired with an open closing PR that has not merged.

## What This Changes

The issue requests persistent multi-turn webhook hook sessions when a stable sessionKey is supplied, instead of forcing a new isolated session for every hook-agent run.

## Best Possible Solution

Keep this issue open as the canonical report and resolve it by landing or rejecting the linked persistent hook-session PR, with the narrower overlapping PR redirected as needed.

## Reproduction Assessment

Yes. Source inspection shows /hooks/agent creates an isolated cron-style job, the isolated runner passes forceNew=true, and resolveCronSession only reuses stored session ids when forceNew is false.

## Solution Assessment

Yes on direction, but not complete on current main. An explicit opt-in sessionMode: "isolated" | "persistent" path is safer than automatic reuse because it preserves default isolation while giving trusted integrations a documented persistent mode.
