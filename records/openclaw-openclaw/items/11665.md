## Label Justifications

- `P2`: This is a bounded hook/config improvement with real webhook-session continuity impact, but it is not a core runtime outage.
- `impact:session-state`: Stable hook `sessionKey` values currently fail to preserve conversation/session history across webhook turns.
- `impact:data-loss`: The reported behavior creates new session ids and makes prior webhook conversation state unavailable from the stable key.
- `impact:security`: The requested persistent hook mode affects externally selected session-key trust boundaries and must remain explicitly scoped.

## Summary

Keep open: current main and the latest release still route hook-agent requests through isolated cron session creation, and there is still no explicit persistent hook-session mode in payload, mapping, transform, or strict config schema. The remaining work is already paired with the open implementation candidate at https://github.com/openclaw/openclaw/pull/75918, so this issue should wait for that PR to land or be rejected.

## What This Changes

The issue requests multi-turn webhook hook support by reusing conversation state when a stable hook `sessionKey` is supplied.

## Best Possible Solution

Ship one focused explicit `sessionMode: "isolated" | "persistent"` hook implementation with isolated default, key-prefix/agent policy intact, direct payload plus mapping/transform support, schema/docs updates, and targeted gateway/cron tests.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent requests as isolated cron jobs, passes `forceNew: true`, and only reuses stored session ids when `forceNew` is false; I did not run a live webhook in this read-only review.

## Solution Assessment

Yes on the desired direction, but not as an issue close. Explicit opt-in `sessionMode` is safer than automatic reuse because it preserves the isolated default and keeps hook session-key policy explicit.
