## Label Justifications

- `P2`: This is a normal-priority hook session-continuity feature with real user impact but bounded gateway/config/cron scope.
- `impact:session-state`: The issue is about webhook invocations losing conversation context because stable hook session keys do not currently preserve the session identity used for transcripts.
- `impact:data-loss`: The reported behavior loses usable conversation history between webhook turns and leaves orphaned transcript files for prior generated session IDs.
- `impact:security`: The requested persistent mode changes how externally influenced hook session keys select shared transcript context, so the trust boundary needs explicit review.

## Summary

Keep open: current main still forces hook-agent runs through isolated cron sessions that mint fresh session IDs, and the remaining work is now owned by an open closing PR rather than something to close as implemented.

## What This Changes

Allow hook-agent requests with a stable `sessionKey` to opt into multi-turn session reuse instead of always creating a fresh isolated session.

## Best Possible Solution

Review and land a narrow explicit `sessionMode: "isolated" | "persistent"` implementation through https://github.com/openclaw/openclaw/pull/75918, keeping isolated as the default and preserving hook session-key security controls.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent requests as isolated cron jobs, passes `forceNew: true`, and only reuses an existing session when `forceNew` is false.

## Solution Assessment

Yes for the direction, but not as an implicit default. The maintainable fix is an explicit persistent hook-session mode with secure defaults, not automatic reuse for every stable hook key.
