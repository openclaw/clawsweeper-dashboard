## Label Justifications

- `P2`: This is a normal-priority hook session-continuity feature with real user impact but a bounded Gateway/config surface.
- `impact:session-state`: The issue is about stable hook keys failing to preserve session and transcript continuity across webhook turns.
- `impact:data-loss`: The reported behavior makes prior conversation history unreachable for later hook turns and leaves orphaned transcript files.
- `impact:security`: The requested persistent mode affects which externally triggered hook events can reuse transcript context, making the trust model security-sensitive.

## Summary

Keep open: current main and v2026.6.5 still route hook-agent requests through isolated cron sessions with `forceNew`, so a stable hook `sessionKey` cannot reuse the existing transcript. The open linked pull request is the right implementation candidate to review before this issue can close.

## What This Changes

Request opt-in multi-turn webhook hook sessions so stable hook `sessionKey` values can reuse existing session history instead of creating a fresh isolated session per invocation.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75918 or an equivalent explicit `sessionMode: "isolated" | "persistent"` design with isolated default, scoped session-key policy, direct/mapped/transform coverage, docs, and upgrade proof.

## Reproduction Assessment

Yes. Source inspection shows hook-agent dispatch creates an isolated cron job and the runner passes `forceNew` for isolated jobs, so `resolveCronSession` cannot reuse the existing session id for stable hook keys on current main.

## Solution Assessment

Yes for the preferred direction, but not yet landed. An explicit persistent `sessionMode` that keeps isolated as the default is safer than automatic reuse because it preserves current security and compatibility defaults.
