## Label Justifications

- `P2`: This is a bounded hook/config/cron improvement with real webhook-session continuity impact but not a core runtime outage.
- `impact:session-state`: The issue is directly about stable hook `sessionKey` values failing to preserve conversation/session history.

- `impact:security`: Persistent hook sessions affect externally supplied session-key trust boundaries and need explicit scoped operator control.

## Summary

Keep open: current main and v2026.6.8 still route hook-agent requests through isolated cron session creation, and the remaining work is already owned by an open linked implementation PR with closing syntax.

## What This Changes

The issue requests persistent multi-turn webhook hook sessions when a stable `sessionKey` is supplied.

## Best Possible Solution

Review and land a focused explicit `sessionMode` implementation, or close the linked PR if maintainers reject that product direction, while keeping default isolation and key-prefix policy intact.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent requests as isolated cron jobs, passes `forceNew: true`, and only reuses stored sessions when `forceNew` is false; I did not run a live webhook in this read-only review.

## Solution Assessment

Yes on direction, but not as an automatic issue close. Explicit `sessionMode: "isolated" | "persistent"` is safer than automatic reuse because it preserves default isolation while allowing trusted hooks to opt into context reuse.
