## Label Justifications

- `P2`: Webhook session continuity has real user impact but is bounded to hook, config, and session-runtime behavior rather than an urgent outage.
- `impact:session-state`: The issue is about stable hook `sessionKey` values failing to preserve intended transcript and context across turns.
- `impact:data-loss`: Each webhook invocation can create a new transcript identity, effectively losing conversational history for the active workflow and leaving old transcript files orphaned.
- `impact:security`: Any persistent hook-session fix must preserve external session-key authorization and prefix controls to avoid cross-user or cross-integration context joins.

## Summary

Keep open: current main and the latest release still force hook-agent requests through isolated cron runs, so stable hook `sessionKey` values cannot preserve transcript continuity. The remaining work is already represented by the open implementation candidate at https://github.com/openclaw/openclaw/pull/75918, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests persistent multi-turn webhook hook sessions when callers provide a stable `sessionKey`.

## Best Possible Solution

Land or narrow the open persistent hook-session PR so direct `/hooks/agent`, hook mappings, and transform results get explicit `sessionMode: "isolated" | "persistent"` with isolated as the default and matching schema, docs, and security tests.

## Reproduction Assessment

Yes, at source level. Direct and mapped hook-agent requests still go through isolated cron runs with `forceNew: true`, while the session resolver only reuses existing `sessionId` values when `forceNew` is false.

## Solution Assessment

Yes for the direction, but it is not yet on main. An explicit persistent mode is safer than automatic reuse because it preserves the isolated default and makes the hook session-key trust model operator-controlled.
