## Label Justifications

- `P2`: This is a normal-priority hook session-continuity improvement with real user impact but a bounded gateway/config/session surface.
- `impact:session-state`: Stable webhook `sessionKey` values currently do not preserve the conversation session state needed for multi-turn hook workflows.
- `impact:data-loss`: Each hook turn can move the stored key to a fresh session id, leaving prior transcript context unavailable to later turns.
- `impact:security`: The requested persistent mode changes how externally triggered hooks can join session state, so trust and session-key scoping are part of the issue.

## Summary

This issue should stay open: current main still lacks an explicit persistent hook-session mode for stable webhook `sessionKey` values, and https://github.com/openclaw/openclaw/pull/75918 is an open linked implementation candidate that should be reviewed rather than closing the issue early.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing an existing session when a hook provides a stable `sessionKey`.

## Best Possible Solution

Review and land or explicitly close the linked implementation PR; the desired end state is a backward-compatible explicit `sessionMode: "isolated" | "persistent"` hook surface with isolated as the default, scoped session-key safeguards, docs, schema, and regression coverage.

## Reproduction Assessment

Yes from source inspection, though I did not run a live webhook repro in this read-only pass. Current main still dispatches hook-agent runs as isolated cron jobs, the runner passes `forceNew: true`, and session reuse only happens when `forceNew` is false.

## Solution Assessment

Yes, the explicit opt-in persistent mode is the best current direction. Automatic reuse would be simpler but weaker for compatibility and security because isolated hook behavior is the current default.
