## Label Justifications

- `P2`: This is a concrete hook/session improvement with production impact, but it needs opt-in behavior and security review rather than emergency handling.
- `impact:session-state`: Stable-key webhook sessions lose multi-turn transcript continuity because each hook run is forced into a new session id.
- `impact:data-loss`: Prior hook turns become unavailable to the agent and old transcript files can be orphaned from the stable session-key mapping.
- `impact:security`: The requested fix changes when hook-selected session keys can reuse existing transcript context.

## Summary

Keep open: current main and the latest release still route stable-key hook-agent calls through isolated cron sessions with `forceNew` enabled, and the open closing PR should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing an existing session when a stable `sessionKey` is provided, preferably through an explicit persistent session mode.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/75918 or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation that keeps isolation as the default and preserves hook session-key security gates.

## Reproduction Assessment

Yes, source-level. Approved stable-key hook-agent requests become `sessionTarget: "isolated"` cron jobs, which makes `forceNew` true and prevents `resolveCronSession` from reusing the stored session id.

## Solution Assessment

Unclear as a final product choice, but explicit `sessionMode: "isolated" | "persistent"` is the safest evidenced direction. Automatic reuse for every stable key would need stronger compatibility and security approval.
