## Label Justifications

- `impact:data-loss`: Each forced-new hook run abandons the prior transcript identity, effectively losing conversation continuity and leaving old transcript files orphaned.

## Summary

Keep open: current main and the latest release still route hook-agent requests through isolated cron sessions, and an open linked implementation PR specifically owns the remaining fix path.

## What This Changes

Requests multi-turn webhook hook sessions by reusing an existing session when a stable hook session key is supplied, preferably through an explicit persistent session mode.

## Best Possible Solution

Review and land a scoped, security-reviewed version of https://github.com/openclaw/openclaw/pull/75918, or replace it with the same explicit sessionMode contract while preserving default hook isolation.

## Reproduction Assessment

Yes, at source level. Direct and mapped hook-agent dispatches create isolated cron-style jobs, which pass forceNew into resolveCronSession, so a stable hook sessionKey cannot reuse the existing session id on current main.

## Solution Assessment

Yes, the explicit opt-in sessionMode direction is the best known approach because it preserves the current isolated default while allowing trusted persistent hook sessions. Automatic reuse for every stable key would be riskier without a maintainer-approved trust model.
