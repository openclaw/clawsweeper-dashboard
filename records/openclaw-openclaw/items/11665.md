## Label Justifications

- `P2`: Stable hook sessions losing transcript continuity is a normal-priority webhook/session improvement with real user impact and bounded hook/config/runtime scope.
- `impact:session-state`: The report is about stable webhook `sessionKey` values failing to preserve conversation context across hook turns.
- `impact:data-loss`: Each hook invocation can orphan the expected prior transcript from the active conversation flow, forcing users to reconstruct history externally.
- `impact:security`: The requested persistent mode changes externally reachable session continuity and must preserve explicit session-key authorization boundaries.

## Summary

Keep open: current main and the latest release still route hook-agent work through forced isolated sessions, while the open closing PR is the implementation candidate and should be reviewed or replaced before this issue closes.

## What This Changes

Request stable webhook hook `sessionKey` values to support opt-in multi-turn hook sessions instead of minting a fresh isolated session id on every invocation.

## Best Possible Solution

Review and land, narrow, or replace https://github.com/openclaw/openclaw/pull/75918 with an explicit `sessionMode` hook design that preserves isolated defaults and session-key security gates.

## Reproduction Assessment

Yes, at source level. Current main dispatches hook-agent work as isolated cron jobs, passes `forceNew: true`, and the resolver only reuses an existing `sessionId` when `forceNew` is false; no live webhook run was executed in this read-only review.

## Solution Assessment

Yes, the explicit opt-in `sessionMode: "isolated" | "persistent"` direction is the best-maintainable shape seen so far because it keeps isolation as the default while allowing trusted scoped hook threads to reuse history.
