## Label Justifications

- `P2`: This is a normal-priority hook session continuity problem with real user impact but a bounded gateway/config/cron scope.
- `impact:session-state`: Stable webhook session keys currently fail to preserve conversation context across hook turns.
- `impact:data-loss`: The active stable-key session loses usable conversation history and leaves prior transcript files orphaned from the latest session mapping.
- `impact:security`: The requested persistent mode affects how externally triggered hooks select and reuse long-lived session context.

## Summary

Keep open: current main and the latest release still force hook-agent runs through isolated cron session rollover, and the open linked fix PR has not merged yet.

## What This Changes

The issue requests multi-turn webhook hook sessions so a stable hook `sessionKey` can reuse conversation history instead of minting a fresh session each invocation.

## Best Possible Solution

Review and land a security-reviewed explicit persistent hook session mode that keeps isolated as the default, threads the mode through direct payloads, mappings, transforms, schema, docs, and tests, then close this issue when the PR merges.

## Reproduction Assessment

Yes, source inspection is enough for a high-confidence reproduction path: direct and mapped hook-agent dispatches use isolated cron jobs, the runner passes `forceNew: true`, and the resolver only reuses sessions when `forceNew` is false. I did not run a live webhook in this read-only review.

## Solution Assessment

Yes, the explicit `sessionMode: "isolated" | "persistent"` direction is the best maintainable shape because it preserves safe default isolation while allowing trusted scoped hooks to opt into reuse. Automatic reuse for every stable key would be riskier and needs maintainer security sign-off if chosen instead.
