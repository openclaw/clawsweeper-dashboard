## Summary

Keep open. Current main still forces hook-agent requests with stable `sessionKey` values through isolated session resolution, and #75918 is an open unmerged closing PR for the remaining work.

## What This Changes

Allow hook-agent requests with a stable `sessionKey` to opt into persistent multi-turn session reuse instead of always creating a new isolated transcript.

## Best Possible Solution

Review and land or close #75918, or an equivalent explicit `sessionMode: "isolated" | "persistent"` implementation that keeps isolated as the default.

## Reproduction Assessment

Yes. A direct or mapped hook with a stable `sessionKey` still flows through `dispatchAgentHook`, becomes an isolated cron-style job, and reaches `resolveCronSession` with `forceNew: true`; existing unit coverage confirms that path mints a new session id.

## Solution Assessment

Yes. An explicit persistent-session opt-in is the narrowest maintainable solution because it enables trusted multi-turn webhook threads without changing the default isolated hook behavior.
