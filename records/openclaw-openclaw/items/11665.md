## Label Justifications

- `P2`: The issue is a real hook session-continuity problem with production workarounds, but it is bounded to webhook hook sessions and needs product/security review rather than emergency handling.
- `impact:session-state`: Stable hook session keys currently do not preserve the intended conversation context across webhook turns.
- `impact:data-loss`: Each webhook invocation can abandon the prior transcript identity, effectively losing conversation continuity and leaving old transcript files orphaned.
- `impact:security`: Any persistent hook-session fix changes how externally influenced session keys can join existing context and must preserve authorization and prefix boundaries.

## Summary

Keep open: current main still routes hook-agent requests through isolated cron execution, so a stable hook session key cannot reuse an existing transcript session, and the open linked pull request is the active implementation path.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing an existing session when a stable hook session key is supplied.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/75918 if its security, compatibility, and scope checks pass; otherwise close that PR and pick one explicit persistent-session design that preserves default isolation.

## Reproduction Assessment

Yes, at source level. Direct and mapped hook-agent requests can carry a stable sessionKey, but current main sets the synthesized job to isolated and then passes forceNew: true, so resolveCronSession cannot reuse the stored session id.

## Solution Assessment

Yes, the explicit sessionMode opt-in is the best observed direction. Automatic reuse for every stable hook key would be riskier because isolated hook behavior is the current safe default and session-key joins are security-sensitive.
