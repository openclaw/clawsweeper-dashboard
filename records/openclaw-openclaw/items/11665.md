## Label Justifications

- `P2`: Stable webhook session keys losing context affects real automations, but the fix is bounded to hook/config/session behavior rather than an emergency outage.
- `impact:session-state`: The issue is about webhook turns failing to preserve session and transcript context across a stable sessionKey.
- `impact:data-loss`: Reporters describe lost conversation history and orphaned transcript files when each webhook turn receives a new session id.
- `impact:security`: The requested persistent mode changes how external hook session keys select reusable context, so the trust boundary must stay explicit and default-isolated.

## Summary

Keep open: current main and the latest release still route hook-agent requests through the isolated cron runner, so stable hook session keys cannot reuse the stored session id. The linked open implementation PR owns the remaining work and needs maintainer review rather than closing this issue now.

## What This Changes

Allow webhook hook-agent calls with a stable sessionKey to opt into reusing the existing session so multi-turn webhook conversations keep transcript context.

## Best Possible Solution

Review and land a focused explicit sessionMode implementation that keeps isolated as the default, preserves hook session-key policy checks, updates config/docs/tests, and closes this issue after merge.

## Reproduction Assessment

Yes, at source level. Direct and mapped hook-agent dispatches enter isolated cron jobs, the runner passes forceNew=true for isolated jobs, and the session resolver only reuses an existing session when forceNew is false; I did not run a live webhook in this read-only review.

## Solution Assessment

Yes, the explicit sessionMode direction is the maintainable path, but it is not finished on main. Automatic reuse for every stable hook key would be weaker because it changes the trust boundary without an explicit operator choice.
