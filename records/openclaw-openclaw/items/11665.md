## Label Justifications

- `P2`: The missing capability causes real webhook continuity, token-cost, and transcript-management problems but is limited to hook-agent workflows.
- `impact:session-state`: Repeated hook turns using the same stable key currently cannot retain one multi-turn conversation session.
- `impact:data-loss`: Forced-new session identities discard conversational continuity and can leave prior transcript files orphaned.
- `impact:security`: Reusing externally selected session keys changes a context-isolation boundary and requires explicit trust and scoping controls.

## Summary

Current main still routes hook-agent requests through isolated jobs that force a fresh session, so stable hook session keys cannot retain multi-turn transcript state. The issue should remain open because https://github.com/openclaw/openclaw/pull/75918 explicitly closes it and is still open with a focused implementation plus real-environment proof.

## What This Changes

Add an explicit opt-in persistent session mode for direct and mapped webhook agent runs so stable session keys can reuse conversation state while isolated behavior remains the default.

## Best Possible Solution

Review and land the explicit opt-in persistent mode in the paired PR after confirming isolated-by-default behavior, session-key authorization, replay-scope separation, schema/help/docs alignment, and fresh-install plus upgrade behavior.

## Reproduction Assessment

Yes, at source level. Current main constructs hook-agent jobs as isolated and consequently passes `forceNew: true`, while the resolver would reuse a fresh stored session only when that flag is false.

## Solution Assessment

Yes in direction. An explicit `isolated | persistent` mode is narrower and safer than automatic reuse because it uses the existing resolver, preserves the shipped isolation default, and makes the security tradeoff operator-controlled; the paired PR still needs maintainer review.
