## Summary

Keep open: current main and the latest release still use a fixed cross-origin redirect safe-header allowlist, and the requested opt-in custom-header retention remains a security-sensitive feature/config decision.

## What This Changes

The issue requests an opt-in configuration option that lets operators add custom fetch-guard headers, such as `x-api-key`, to the cross-origin redirect safe-header allowlist.

## Best Possible Solution

Keep strict stripping as the default; if maintainers approve the capability, add a narrowly scoped operator allowlist with explicit security wording and tests for both default stripping and configured retention.

## Reproduction Assessment

Yes, at source level: current code strips `X-Api-Key` on cross-origin redirects and the strict config schema has no fetch-guard extension option. A live integration-specific reproduction is not provided in the issue.

## Solution Assessment

Unclear until maintainers choose the trust model: an opt-in allowlist is plausible, but a broad global escape hatch would be risky without header and redirect guardrails.
