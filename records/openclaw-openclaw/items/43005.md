## Summary

Keep open: current main and v2026.5.12 still use a fixed cross-origin redirect safe-header allowlist, tests assert `x-api-key` is stripped, and the requested opt-in is a new security-sensitive config surface that needs maintainer product/security review.

## What This Changes

Add an opt-in configuration surface for retaining selected custom headers during fetch-guard cross-origin redirects.

## Best Possible Solution

If accepted, design a fail-closed origin- or destination-scoped redirect-header policy shared by fetch guard, media, and MCP paths, with schema/docs/changelog updates and focused regression coverage.

## Reproduction Assessment

Yes, for the current limitation: source inspection on current main shows origin-changing redirects call a fixed safe-header helper, and checked-in tests assert `x-api-key` is stripped. I did not run tests because this review is read-only.

## Solution Assessment

Unclear. The requested global `additionalSafeHeaders` knob is not obviously the safest design; an origin- or destination-scoped trust policy would be a safer maintainable direction if maintainers want this capability.
