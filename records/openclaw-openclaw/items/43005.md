## What This Changes

Requested an opt-in configuration option for preserving additional safe headers during fetch-guard cross-origin redirects.

## Best Possible Solution

Keep strict stripping as the default; if maintainers approve the capability, add a narrowly documented operator allowlist with explicit security wording and tests for default stripping and configured retention.

## Reproduction Assessment

Yes for the current behavior: source inspection shows `fetchWithSsrFGuard` rewrites cross-origin redirects through a fixed allowlist, and checked-in tests assert `x-api-key` is stripped.

## Solution Assessment

Unclear pending maintainer policy. An opt-in allowlist is plausible, but the best shape depends on the credential-forwarding trust model, config validation, and documentation boundary.
