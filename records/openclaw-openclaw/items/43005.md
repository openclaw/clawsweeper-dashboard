## Summary

Keep open: current main and the latest release still enforce a fixed cross-origin redirect safe-header set, while the requested opt-in custom header forwarding changes a credential-forwarding security boundary that needs maintainer product/security approval.

## What This Changes

The issue requests a new configuration surface that lets selected custom headers, such as x-api-key, survive fetch-guard cross-origin redirects.

## Best Possible Solution

Keep strict stripping as the default; if maintainers approve the capability, add a narrowly documented opt-in policy with explicit security wording and regression coverage for default and configured redirect behavior.

## Reproduction Assessment

Yes, by source inspection: current main strips x-api-key on cross-origin redirects, and checked-in SSRF guard tests assert that behavior. I did not run tests because this review is read-only.

## Solution Assessment

Unclear pending maintainer policy: an opt-in allowlist is a plausible solution, but the best shape depends on the credential-forwarding trust model, config validation, and documentation boundary.
