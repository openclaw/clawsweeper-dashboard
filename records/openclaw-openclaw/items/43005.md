## Summary

Keep open: current main and v2026.5.20 still use a fixed cross-origin redirect safe-header allowlist, and retaining credential-like custom headers needs maintainer security/product policy rather than cleanup closure.

## What This Changes

The issue requests an opt-in configuration surface that lets selected custom headers survive fetch-guard cross-origin redirects.

## Best Possible Solution

If accepted, add a fail-closed, source- or destination-scoped redirect-header trust policy shared by fetch guard, media, and MCP, with schema, docs, changelog, and focused regression coverage.

## Reproduction Assessment

Yes, as a source-level reproduction: current main routes origin-changing redirects through a fixed helper, and tests assert x-api-key, private-token, and x-trace are stripped. I did not execute tests because this review is read-only.

## Solution Assessment

Unclear. The proposed global additionalSafeHeaders knob is not clearly the safest final shape; scoped trust by source or destination origin would be safer if maintainers want this capability.
