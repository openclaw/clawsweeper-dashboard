## Summary

Keep open: current main and the latest release still use a fixed cross-origin redirect safe-header allowlist, and the requested custom-header opt-in would add security-sensitive public config rather than cleanup-close material.

## What This Changes

Add an opt-in fetch-guard configuration option that lets selected custom headers survive cross-origin redirects while keeping strict filtering as the default.

## Best Possible Solution

Keep strict stripping as the default; if maintainers approve the capability, add a narrowly documented operator allowlist with explicit security wording and tests for both default stripping and configured retention.

## Reproduction Assessment

Yes for the current limitation: source inspection shows X-Api-Key is stripped on cross-origin redirects and there is no additionalSafeHeaders config surface. I did not run tests because this review is read-only.

## Solution Assessment

Unclear pending maintainer policy: an opt-in allowlist is plausible, but the best shape depends on the redirect credential-forwarding trust model, config validation, and documentation boundary.
