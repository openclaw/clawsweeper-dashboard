## Summary

Keep open: current main and v2026.6.6 still enforce a fixed cross-origin redirect safe-header set in guarded fetch, while the requested opt-in custom header forwarding changes a credential-forwarding security boundary. The item is still necessary as a maintainer product/security decision, not a stale cleanup candidate, and the adjacent bearer-token fallback issue does not supersede this narrower redirect-header request.

## What This Changes

The issue requests an opt-in configuration surface that would let guarded fetch preserve selected custom headers across cross-origin redirects.

## Best Possible Solution

Keep the issue open for maintainers to decide whether OpenClaw should expose a narrow opt-in allowlist for redirect-preserved headers, and if so implement it with schema/docs/tests plus explicit credential-forwarding guardrails.

## Reproduction Assessment

Yes from source inspection: guarded fetch currently strips custom headers like `x-api-key` on cross-origin redirects through the fixed safe-header set, and tests cover that behavior. I did not run a live repro because this read-only review cannot execute tests or create artifacts in the checkout.

## Solution Assessment

Unclear: the requested opt-in allowlist is plausible, but it is not safe to treat as the best solution until maintainers decide the credential-forwarding trust model and config boundary. A narrower API tied to explicit redirect/auth ownership may be safer than a broad global header allowlist.
