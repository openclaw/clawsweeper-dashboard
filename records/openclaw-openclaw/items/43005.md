## Summary

Keep open: current main still has a fixed cross-origin redirect safe-header allowlist, intentionally strips `x-api-key`, and has no config/API surface for opt-in custom headers. This is a security-sensitive feature/config request, not a stale cleanup close.

## What This Changes

The issue requests an opt-in configuration option to preserve selected custom headers across fetch-guard cross-origin redirects.

## Best Possible Solution

If accepted, add a fail-closed redirect-header policy scoped to trusted destinations or origin relationships, with validation, docs, changelog, and regression coverage for fetch guard and media redirects.

## Reproduction Assessment

Yes for the current limitation: source and tests on current main show `X-Api-Key` is stripped on cross-origin redirects, and repository search found no opt-in config surface.

## Solution Assessment

Unclear. The requested global `additionalSafeHeaders` knob is not obviously the safest design; a destination-scoped or origin-bound policy would be more maintainable if maintainers accept the capability.
