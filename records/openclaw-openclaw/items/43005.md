## Summary

Keep open: current main and the latest release still use a fixed cross-origin redirect safe-header allowlist, and the requested opt-in custom-header retention is a security-sensitive feature/config decision rather than cleanup-close material.

## What This Changes

The issue requests a config-backed opt-in that lets selected custom headers survive fetch-guard cross-origin redirects.

## Best Possible Solution

Keep strict stripping as the default; if maintainers approve the capability, add a narrowly scoped operator allowlist with explicit security wording and regression coverage for both default stripping and configured retention.

## Reproduction Assessment

Yes for the missing capability at source level: current code strips X-Api-Key on cross-origin redirects and there is no fetchGuard/additionalSafeHeaders config surface. I did not run a live repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the trust model: an opt-in allowlist is plausible, but it should be scoped to explicit operators and tested rather than added as a broad global escape hatch.
