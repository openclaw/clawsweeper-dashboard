## Summary

Keep open: current main and the latest release still use a fixed cross-origin redirect safe-header allowlist, intentionally strip custom headers like `x-api-key`, and expose no opt-in redirect-header policy surface. This is a security-sensitive feature/config request that needs maintainer security and product judgment rather than cleanup closure.

## What This Changes

The issue requests an opt-in configuration surface that lets callers retain selected custom headers, such as `x-api-key`, across fetch-guard cross-origin redirects.

## Best Possible Solution

If accepted, add a fail-closed redirect-header policy scoped to trusted origins or destinations, with validation, docs, changelog coverage, and regression tests for fetch guard and media redirects.

## Reproduction Assessment

Yes, for the current limitation: source-level inspection on current main shows origin-changing redirects run through the fixed helper, and checked-in tests assert `x-api-key` is stripped. I did not run tests because this review is read-only, but the behavior is directly traceable in source and test expectations.

## Solution Assessment

Unclear. The requested global `additionalSafeHeaders` knob is not obviously the safest design; a destination- or origin-scoped policy would be more maintainable if maintainers accept retaining credential-like headers.
