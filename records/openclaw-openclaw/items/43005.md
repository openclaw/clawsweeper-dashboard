## Summary

Keep open: current main and v2026.6.6 still strip custom headers such as `x-api-key` on cross-origin guarded-fetch redirects, and the requested opt-in changes a shared credential-forwarding security boundary that needs maintainer product/security review. The linked security issue at https://github.com/openclaw/openclaw/issues/57588 is adjacent evidence for the same trust-model area, but it does not supersede this request.

## What This Changes

The issue requests an opt-in configuration surface for retaining selected custom headers across fetch-guard cross-origin redirects.

## Best Possible Solution

Keep strict stripping by default and, if maintainers approve the capability, add a destination-scoped, caller-owned opt-in with focused tests and docs.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main filters cross-origin redirect headers through a fixed safe list, and checked-in tests assert `x-api-key` is stripped. I did not run tests because this was a read-only review.

## Solution Assessment

No, the proposed global name-only config is not the best shape yet. A safer solution would bind allowed custom headers to explicit trusted redirect destinations and the caller that owns the credential.
