## Summary

Keep open: current main and v2026.5.12 still use a fixed cross-origin redirect safe-header allowlist, and the requested opt-in would add a security-sensitive config surface that needs maintainer product/security design rather than cleanup closure.

## What This Changes

The issue requests a configurable way to allow selected custom headers, such as x-api-key, to survive fetch-guard cross-origin redirects while preserving strict defaults.

## Best Possible Solution

If accepted, add a fail-closed, origin- or destination-scoped redirect-header policy shared by fetch guard, media, and MCP redirect paths, with schema/docs/changelog updates and focused regression coverage.

## Reproduction Assessment

Yes, for the current limitation: source inspection on current main shows origin-changing redirects call a fixed safe-header helper, and checked-in tests assert x-api-key is stripped. I did not run tests because this review is read-only.

## Solution Assessment

Unclear. The proposed global additionalSafeHeaders knob is not obviously the safest design; an origin- or destination-scoped trust policy would be a safer maintainable direction if maintainers want this capability.
