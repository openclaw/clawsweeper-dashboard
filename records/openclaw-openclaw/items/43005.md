## Summary

Keep open: current main and v2026.6.6 still intentionally strip custom headers such as `x-api-key` on cross-origin guarded-fetch redirects, and the requested opt-in broadens a shared security boundary that needs maintainer security/product review.

## What This Changes

The issue requests a new `security.fetchGuard.additionalSafeHeaders` configuration option so custom headers can survive fetch-guard cross-origin redirects.

## Best Possible Solution

Keep strict stripping by default and, if approved, add a narrow caller-owned policy that pairs allowed headers with explicitly trusted redirect destinations, with focused tests and docs.

## Reproduction Assessment

Yes, at source level: current main routes cross-origin redirects through the fixed safe-header filter, and checked-in tests assert that `x-api-key` is stripped. No live integration was provided to prove a specific trusted redirect chain that needs an exception.

## Solution Assessment

No, not as proposed: a global list of safe header names is broader than the caller and destination that own the credential. A destination-aware, caller-scoped policy is the safer maintainable direction if maintainers approve the capability.
