## Summary

Keep open: current main still uses a fixed cross-origin redirect safe-header allowlist, and the requested opt-in for custom safe headers is absent. The request is security-sensitive because credential-like headers can leak across redirects, so it needs maintainer/security judgment before any implementation.

## What This Changes

The issue requests a configuration or API option to allow selected custom headers to survive fetch-guard cross-origin redirects while preserving strict defaults.

## Best Possible Solution

If accepted, add a narrow, fail-closed redirect-header policy in the shared fetch-guard/redirect-header path, preferably scoped to trusted redirect destinations or origin relationships rather than a global header-name list. Keep strict defaults, validate header names, deny or explicitly gate credential-like names, wire only the intended config/SDK surface, and cover fetch guard, provider transport, and media redirects with regression tests.

## Reproduction Assessment

Yes. The current behavior is reproducible with the existing SSRF hardening test pattern: send X-Api-Key through a cross-origin redirect and observe that the second request omits it; the absence of an opt-in is also visible in GuardedFetchOptions and provider request config.

## Solution Assessment

Unclear. The issue's proposed global additionalSafeHeaders list is not obviously the best solution because a header-name-only allowlist can leak secrets to attacker-controlled redirect targets; a destination-scoped or origin-bound policy would be safer if maintainers accept the capability.
