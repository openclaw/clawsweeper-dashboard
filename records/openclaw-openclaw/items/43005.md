## Summary

Keep open: current main and the latest release still use a fixed cross-origin redirect safe-header allowlist, and the requested opt-in would change security-sensitive handling of credential-like headers.

## What This Changes

The issue asks for a configuration option that lets selected custom headers survive fetch-guard cross-origin redirects while keeping the default strict filtering.

## Best Possible Solution

If accepted, design a fail-closed source- or destination-scoped redirect-header policy shared by fetch guard, media, and MCP, with schema, docs, changelog, and focused regression coverage.

## Reproduction Assessment

Yes, for the current limitation: source inspection shows origin-changing redirects call a fixed safe-header helper, and checked-in tests assert x-api-key is stripped. I did not run tests because this review is read-only.

## Solution Assessment

Unclear. The requested global additionalSafeHeaders knob is not clearly the safest design; scoped trust by source or destination origin would be a safer maintainable direction if maintainers want this capability.
