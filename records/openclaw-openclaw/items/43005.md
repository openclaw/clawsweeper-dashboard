## Summary

Keep open: current main still uses a fixed cross-origin redirect safe-header allowlist and tests intentionally strip `x-api-key`; adding an opt-in escape hatch would be a new security-sensitive config policy that needs maintainer design review.

## What This Changes

The issue requests a `security.fetchGuard.additionalSafeHeaders`-style option to preserve selected custom headers across fetch-guard cross-origin redirects.

## Best Possible Solution

If accepted, implement a fail-closed redirect-header policy scoped to trusted destinations or origin relationships, with header-name validation, docs, changelog, and regression coverage for both fetch guard and media redirects.

## Reproduction Assessment

Yes for the current limitation: current-main source and tests show `X-Api-Key` is stripped on cross-origin redirects, and no config/API surface exists to opt into preserving it.

## Solution Assessment

Unclear. The requested global `additionalSafeHeaders` knob is not obviously the safest design; a destination-scoped or origin-bound policy would be more maintainable if maintainers accept the capability.
