## Summary

Keep open: current main and v2026.5.20 still use a fixed cross-origin redirect safe-header allowlist, and allowing credential-like custom headers across origin changes needs security and product review rather than cleanup closure.

## What This Changes

The issue requests an opt-in configuration option for preserving selected custom headers, such as `x-api-key`, during fetch-guard cross-origin redirects.

## Best Possible Solution

If accepted, design a fail-closed, source- or destination-scoped redirect-header policy shared by fetch guard, media, and MCP, with schema, docs, changelog, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main routes origin-changing redirects through a fixed safe-header helper, and checked-in tests assert `x-api-key`, `private-token`, and `x-trace` are stripped; no tests were run because this was a read-only review.

## Solution Assessment

Unclear. The integration gap is real, but the proposed global `additionalSafeHeaders` knob is not clearly the safest final shape; scoped trust by source or destination origin would be safer if maintainers want this capability.
