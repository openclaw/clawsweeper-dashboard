## Summary

Keep open: current main and the latest release still use a fixed cross-origin redirect safe-header allowlist, and preserving custom credential-like headers across origin changes is a security/product decision rather than maintainer cleanup.

## What This Changes

The issue requests an opt-in configuration surface that preserves selected custom headers during fetch-guard cross-origin redirects.

## Best Possible Solution

If accepted, design a fail-closed source- or destination-scoped redirect-header policy shared by fetch guard, media, and MCP, with schema, docs, changelog, and focused regression coverage.

## Reproduction Assessment

Yes. Current main routes origin-changing redirects through a fixed safe-header helper, and checked-in tests assert `x-api-key`, `private-token`, and `x-trace` are stripped; I did not run tests because this review is read-only.

## Solution Assessment

Unclear. The integration gap is real, but the proposed global `additionalSafeHeaders` knob is not clearly the safest final shape; scoped trust by source or destination origin would be safer if maintainers want this capability.
