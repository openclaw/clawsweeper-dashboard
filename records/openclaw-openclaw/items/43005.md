## Summary

Keep open: current main and v2026.5.12 still use a fixed cross-origin redirect safe-header allowlist, and the requested opt-in header config changes security-sensitive redirect behavior that needs maintainer product/security review.

## What This Changes

The issue requests a new config option to allow selected custom headers to survive fetch-guard cross-origin redirects.

## Best Possible Solution

If accepted, design a fail-closed source- or destination-scoped redirect-header policy shared by fetch guard, media, and MCP redirect paths, with schema, docs, changelog, and focused regression coverage.

## Reproduction Assessment

Yes, for the current limitation: source inspection shows origin-changing redirects call a fixed safe-header helper, and checked-in tests assert x-api-key is stripped. I did not run tests because this review is read-only.

## Solution Assessment

Unclear. The proposed global additionalSafeHeaders knob is not clearly the safest design; scoped trust by source or destination origin would be a safer maintainable direction if maintainers want this capability.
