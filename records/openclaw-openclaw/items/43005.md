## Summary

Keep open: current main and v2026.6.6 still use a fixed cross-origin redirect safe-header set, so the requested opt-in custom-header forwarding remains a real core security/config decision rather than cleanup-close material.

## What This Changes

The issue requests an opt-in configuration option that would allow selected custom headers to survive fetch-guard cross-origin redirects.

## Best Possible Solution

Keep strict stripping as the default; if maintainers approve the capability, add a narrowly documented operator allowlist with explicit security wording and tests for both default stripping and configured retention.

## Reproduction Assessment

Yes. Source inspection shows current main drops x-api-key during cross-origin redirects, and checked-in SSRF guard tests assert that behavior; I did not run tests because this review is read-only.

## Solution Assessment

Unclear pending maintainer policy. An opt-in allowlist is plausible, but the best shape depends on the credential-forwarding trust model, config validation, and documentation boundary.
