## Summary

Keep open: current main and v2026.5.20 still use a fixed safe-header allowlist for origin-changing redirects, and allowing credential-like custom headers to survive those redirects is a security/product decision rather than cleanup closure.

## What This Changes

Requests an opt-in configuration surface to preserve selected custom headers such as `x-api-key` during fetch-guard cross-origin redirects.

## Best Possible Solution

If accepted, design a fail-closed source- or destination-scoped redirect-header policy shared by fetch guard, media, and MCP, with schema, docs, changelog, and focused regression coverage.

## Reproduction Assessment

Yes, as a source-level reproduction: current main routes origin-changing redirects through a fixed safe-header helper, and tests assert `x-api-key`, `private-token`, and `x-trace` are stripped. I did not run tests because this review is read-only.

## Solution Assessment

Unclear. The proposed global `additionalSafeHeaders` knob is not clearly the safest final shape; scoped trust by source or destination origin would be safer if maintainers want this capability.
