## Summary

The requested capability is still absent: current main and v2026.6.5 deliberately remove custom authentication headers when redirects cross origins. The request is understandable, but the proposed process-wide header-name allowlist would weaken a shared security boundary without constraining which redirect destinations may receive the credentials, so it needs an explicit maintainer security and product decision rather than automatic implementation.

## What This Changes

Add an opt-in configuration that lets named custom headers survive fetch-guard redirects to a different origin while retaining strict defaults.

## Best Possible Solution

Retain strict shared scrubbing and, if the capability is approved, expose a narrow caller-owned policy that pairs permitted headers with explicitly trusted destination origins, with tests proving both trusted forwarding and untrusted stripping.

## Reproduction Assessment

Yes at source level: current main and focused tests deterministically remove non-allowlisted custom headers after an origin change. There is no live integration reproduction identifying the trusted redirect chain that needs an exception.

## Solution Assessment

No, not as proposed: a process-wide list of header names does not constrain the receiving origin and is broader than the credential-owning caller. A destination-aware, caller-scoped policy is the safer maintainable direction if maintainers approve the capability.
