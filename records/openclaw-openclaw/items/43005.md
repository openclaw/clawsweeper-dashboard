## Summary

Keep open: current main still has a fixed cross-origin redirect safe-header allowlist with no opt-in extension point, and the requested change affects credential retention across origins, so it needs maintainer/security judgment rather than conservative cleanup.

## What This Changes

The issue requests a configuration option that lets users preserve selected custom headers across fetch-guard cross-origin redirects while keeping the strict default behavior.

## Best Possible Solution

If accepted, implement a fail-closed redirect-header policy in the shared redirect-header/fetch-guard path, scoped to trusted destinations or origin relationships rather than only global header names, with strict defaults, header-name validation, docs, changelog, and regression coverage for fetch guard and media redirects.

## Reproduction Assessment

Yes for the current limitation. Source inspection and existing tests show `X-Api-Key` is stripped on cross-origin redirects and no current config/API surface exists to opt into preserving it.

## Solution Assessment

Unclear. The requested global `additionalSafeHeaders` list is not obviously the safest design because credential-like headers can leak across attacker-controlled redirects; a destination-scoped or origin-bound policy would be safer if maintainers accept the capability.
