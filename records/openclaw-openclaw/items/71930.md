## Summary

Keep open. Current main still accepts only Mattermost `posted` WebSocket events before the post parser and dispatcher, so `post_edited` never reaches mention detection; open PR #72513 is the active closing implementation path and should land or be replaced before this issue closes.

## What This Changes

Mattermost should evaluate `post_edited` WebSocket payloads so @mentions added by editing an existing message can wake the targeted agent.

## Best Possible Solution

Land PR #72513 or an equivalent Mattermost-plugin-only change that lets `posted` and `post_edited` payloads carrying `data.post` reach the same parsing and mention-gate flow, while preserving self-message filtering and preventing replay dedupe from hiding edited-in mentions.

## Reproduction Assessment

Yes. A synthetic `post_edited` Mattermost WebSocket payload on current main deterministically follows the `payload.event !== "posted"` early return before `onPosted`; the issue also provides concrete Mattermost steps confirmed on April 25, 2026.

## Solution Assessment

Yes, with the replay caveat. The narrow maintainable fix belongs in `extensions/mattermost/src/mattermost/monitor-websocket.ts` plus targeted regression coverage, not in a core channel API change.
