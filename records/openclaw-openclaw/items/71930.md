## Summary

Keep open. Current main still drops Mattermost `post_edited` WebSocket events before post parsing and mention detection, and open PR #72513 is the active closing implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue asks the Mattermost plugin to handle `post_edited` WebSocket payloads so @mentions added by editing a message can reach the normal mention and agent wake flow.

## Best Possible Solution

Keep this issue open until PR #72513, or an equivalent Mattermost-only fix, lands. The desired end state is for `posted` and `post_edited` events carrying `data.post` to reach the same parser and mention-gate flow while preserving self-message filtering and avoiding replay-dedupe suppression of edited-in mentions.

## Reproduction Assessment

Yes. The issue's Mattermost edit sequence is credible, and a synthetic `post_edited` WebSocket payload on current main deterministically returns before `parsePostedPayload` and `onPosted`.

## Solution Assessment

Yes for the requested direction. The narrow maintainable fix belongs in the Mattermost plugin WebSocket parser/dispatcher path, with edit-aware replay handling as needed; I found no evidence that a core channel API change is required.
