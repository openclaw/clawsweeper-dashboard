## Summary

Keep open. Current main still filters Mattermost WebSocket post handling to `posted` only, so `post_edited` payloads never reach post parsing or downstream mention detection; open PR #72513 is the active closing implementation candidate and should be reviewed or landed before this issue closes.

## What This Changes

Mattermost should handle `post_edited` WebSocket events so @mentions added by editing a message are evaluated by the normal mention and wake flow.

## Best Possible Solution

Keep this issue open until PR #72513, or an equivalent Mattermost plugin fix, lands. The desired end state is for both `posted` and `post_edited` events carrying `data.post` to reach the same parser, self-message filter, mention gate, and wake flow with explicit regression coverage for edited-in mentions and replay dedupe.

## Reproduction Assessment

Yes. A synthetic Mattermost WebSocket payload with `event: "post_edited"` deterministically returns before `onPosted` on current main, and the issue provides a plausible live sequence for editing an unmentioned post to add `@agentname`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Mattermost plugin WebSocket parser/dispatcher path, with edit-aware replay validation; I found no evidence that a core channel API change is required.
