## What This Changes

The PR changes `waitForAgentJob` so its fallback timer returns a pending run-timeout snapshot instead of resolving to `null` when a timeout snapshot is already waiting in grace.

## Best Possible Solution

Keep this PR as the active fix path, add redacted real-run proof and preferably a focused fake-timer regression test, then land it to close the linked timeout notification issue.

## Reproduction Assessment

Yes, at source level: current `main` stores pending timeout snapshots but the wait fallback timer only checks pending errors before resolving `null`. I did not run a live sub-agent timeout repro in this read-only review.

## Solution Assessment

Yes, this is the right owner boundary: `waitForAgentJob` already owns pending timeout/error snapshots and the caller needs the snapshot fields. A downstream announce-layer workaround would leave the gateway wait result asymmetric.
