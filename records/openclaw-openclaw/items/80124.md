## Summary

This should stay open: current main still validates raw Codex `thread/start` and `thread/resume` responses against generated schemas that require `thread.sessionId`, and the linked closing PR is still unmerged.

## What This Changes

Normalize Codex app-server thread responses so an otherwise valid `thread.id`-only response is accepted at the protocol boundary before schema validation.

## Best Possible Solution

Land a narrow Codex plugin boundary fix that fills missing `thread.sessionId` from a present `thread.id` before start/resume validation, keeps generated schemas synced, and adds focused regression coverage.

## Reproduction Assessment

Yes, by source inspection: current main feeds the raw `thread/start` response into AJV while the generated schema requires `thread.sessionId`. A focused fixture with `thread.id` and no `thread.sessionId` should fail before `turn/start`.

## Solution Assessment

Yes: the narrow maintainable fix is to normalize the missing `sessionId` from `id` at the Codex app-server response boundary before validation. This issue should remain open until https://github.com/openclaw/openclaw/pull/80137 or a corrected replacement merges.
