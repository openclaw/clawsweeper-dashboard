## Summary

Keep this open: current main still validates raw Codex `thread/start` and `thread/resume` responses against schemas requiring `thread.sessionId`, and the only closing PR is still open with review/proof outstanding.

## What This Changes

The issue asks OpenClaw to tolerate Codex app-server `thread/start` responses that include `thread.id` but omit `thread.sessionId` by normalizing at the response-validation boundary.

## Best Possible Solution

Land or revise one narrow Codex app-server response-boundary normalizer with focused regression coverage, then close this issue when the linked fix is merged.

## Reproduction Assessment

Yes. A focused current-main test that feeds an otherwise valid `thread/start` response with `thread.id` but no `thread.sessionId` into the validator path should fail AJV before `turn/start`; I did not execute it under the read-only review constraints.

## Solution Assessment

Yes for the reported id-only failure: pre-validation normalization at the Codex app-server boundary is the narrow maintainable layer. The safer implementation should be careful about accepting a missing `thread.id` unless the upstream contract proves it is interchangeable with `sessionId`.
