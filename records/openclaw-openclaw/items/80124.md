## Summary

This should stay open: current main still validates raw Codex `thread/start` and `thread/resume` responses against schemas that require `thread.sessionId`, and linked open PRs with closing syntax should be reviewed or merged before the issue is closed.

## What This Changes

The issue requests normalization of Codex app-server thread responses so `thread.id` and `thread.sessionId` are cross-filled before schema validation.

## Best Possible Solution

Review and land one canonical Codex app-server response-boundary normalizer for `thread/start` and `thread/resume`, with focused regression tests and real app-server proof.

## Reproduction Assessment

Yes, by source inspection: current main requires `thread.sessionId` in the generated schema and validates raw thread responses before any identity-field normalization. A `thread/start` payload with `thread.id` and no `thread.sessionId` will be rejected before turn startup.

## Solution Assessment

Yes: the narrowest maintainable fix is to normalize equivalent thread identity fields at the Codex app-server response boundary before AJV validation. Editing generated schemas would drift from upstream, and fallback-to-PI behavior would not address the protocol mismatch.
