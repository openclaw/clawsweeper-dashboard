## Summary

This should stay open: current main still validates Codex `thread/start` and `thread/resume` responses before any thread id/sessionId normalization, and the open linked fix PR is not merged yet.

## What This Changes

Normalize Codex app-server thread responses so an otherwise valid `thread.id` payload without `thread.sessionId` does not fail before the prompt is sent.

## Best Possible Solution

Review and land a corrected version of https://github.com/openclaw/openclaw/pull/80137, with focused regression coverage and real Codex app-server proof before closing this issue.

## Reproduction Assessment

Yes, by source inspection: an id-only `thread/start` response reaches `assertCodexThreadStartResponse` unchanged, and the generated schema requires `thread.sessionId`. I did not run a live Codex app-server reproduction in this read-only review.

## Solution Assessment

Yes for boundary normalization before generated-schema validation; the narrow fix should fill missing `sessionId` from a present `id` and keep generated schemas synced to upstream. A reverse `sessionId` to `id` fill should require upstream proof because OpenClaw persists `thread.id` for future calls.
