## Summary

This should stay open: current main still validates Codex `thread/start` and `thread/resume` responses against generated schemas that require `thread.sessionId`, and the validator does not normalize the reported `id`-only response before schema validation.

## What This Changes

Normalize Codex app-server thread responses so equivalent `thread.id` and `thread.sessionId` shapes remain compatible at the protocol boundary.

## Best Possible Solution

Add a small pre-validation normalizer for Codex thread identity fields in the Codex plugin app-server boundary and cover both start and resume response shapes with focused regression tests.

## Reproduction Assessment

Yes by source inspection, but not by an executed local test here. A `thread/start` or `thread/resume` response whose `thread` has `id` but lacks `sessionId` reaches the generated-schema validator raw, and that schema currently requires `sessionId`.

## Solution Assessment

Yes. The narrowest maintainable fix is to normalize equivalent `thread.id` and `thread.sessionId` fields before `assertCodexShape`; editing generated schemas or changing backend fallback policy would solve the problem at the wrong layer.
