## Summary

This should stay open: current main still validates `thread/start` and `thread/resume` responses against generated schemas that require `thread.sessionId`, and there is no pre-validation normalization for the `id`-only response shape described in the report.

## What This Changes

The issue reports a Codex app-server compatibility regression where OpenClaw rejects `thread/start` responses whose `thread` has `id` but omits `sessionId`.

## Best Possible Solution

Normalize equivalent Codex thread identity fields at the Codex app-server response boundary before generated-schema validation, while keeping the generated protocol files synced to upstream.

## Reproduction Assessment

Yes, by source inspection: a focused current-main test can return a valid `thread/start` payload with `thread.id` and no `thread.sessionId`, and `assertCodexThreadStartResponse` will reject it before `turn/start`. I did not run that test because this review is read-only and the local checkout lacks installed runtime dependencies.

## Solution Assessment

Yes: the narrowest maintainable fix is a small response normalizer before `assertCodexShape` for thread start/resume, with regression coverage for both `id`-only and `sessionId`-only shapes. Editing generated schemas or adding fallback-to-PI behavior would be the wrong layer.

In the Codex plugin, normalize Codex app-server thread response identity fields before generated-schema validation: if `thread.id` is a non-empty string and `thread.sessionId` is missing, set `sessionId` to `id`; if `thread.sessionId` is a non-empty string and `thread.id` is missing, set `id` to `sessionId`. Apply this at the response boundary for `thread/start` and `thread/resume`, preferably in `extensions/codex/src/app-server/protocol-validators.ts`, without editing generated protocol JSON/TS files and without adding fallback-to-PI behavior. Preserve current schema drift checks and generated-protocol sync behavior.
