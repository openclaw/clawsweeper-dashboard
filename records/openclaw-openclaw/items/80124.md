## Summary

Keep open: current main still validates Codex `thread/start` and `thread/resume` responses against generated schemas that require `thread.sessionId`, with no pre-validation normalization for the reported `id`-only response shape. The issue is already linked to an open closing PR, so closure should wait for a merged fix.

## What This Changes

Normalize Codex app-server thread responses so an otherwise valid thread with `id` but no `sessionId` does not fail generated-schema validation.

## Best Possible Solution

Land one narrow Codex plugin boundary fix that fills missing `sessionId` from a present `thread.id` before schema validation, with focused regression coverage and redacted real behavior proof, then let the linked fix close the issue.

## Reproduction Assessment

Yes. By source inspection, current main feeds raw `thread/start` and `thread/resume` responses into AJV while the generated schema requires `thread.sessionId`, so a thread object with `id` and no `sessionId` will throw before turn startup.

## Solution Assessment

Yes on the layer, not yet on main: the maintainable fix belongs at the Codex app-server response boundary before generated-schema validation. The safest implementation should avoid accepting missing `thread.id` unless upstream proves `id` and `sessionId` are interchangeable.
