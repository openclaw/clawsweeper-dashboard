## Summary

Keep open: the reported overflow-ordering bug is credible and the PR moves in the right direction, but the patch can replay a persisted user turn after successful early truncation and the external real-behavior proof gate is still mock-only.

## What This Changes

This PR changes the embedded agent overflow path to try existing tool-result truncation before explicit compaction for tool-heavy overflows and adds focused run-loop tests for the new ordering and fallback.

## Best Possible Solution

Keep the deterministic truncation-first recovery, but route successful early truncation through the same transcript-continuation and persistence-suppression guard used after compaction success, then retain the existing compaction fallback.

## Reproduction Assessment

Yes, from source and the linked live log: current main attempts explicit overflow compaction before the existing tool-result truncation fallback in the tool-heavy overflow path. I did not run a live overflow, but the code path and log evidence line up directly.

## Solution Assessment

No, not as written: trying truncation before compaction is the right narrow direction, but the successful-truncation retry must preserve the persisted-turn replay guard and needs after-fix real behavior proof before merge.

## Review Findings

- **[P2] Preserve the persisted-turn retry guard:** `src/agents/pi-embedded-runner/run.ts:1770`
  - body: When this new early truncation path succeeds, it immediately retries with the original prompt. Please reuse the same continuation/suppression check before continuing.
