## Summary

Keep this PR open: current main and the latest release still use compaction-first overflow recovery, while this branch contains the focused truncation-before-compaction change with sufficient runtime proof. No concrete patch defect surfaced, but the merge changes session-state recovery order, so maintainer acceptance of that tradeoff is still required.

## What This Changes

The branch changes embedded-agent overflow recovery so tool-heavy overflows try deterministic tool-result truncation before explicit LLM compaction, with regression tests for retry ordering and overflow-budget accounting.

## Best Possible Solution

Land this PR only if maintainers accept truncation-first recovery as the default for tool-heavy overflows; otherwise preserve compaction-first behavior and design an explicit shorter-escalation or opt-in policy.

## Reproduction Assessment

Yes. Current main and the latest release still run explicit overflow compaction before the later truncation fallback, and the linked issue/PR body includes a live log showing a 912840 ms compaction timeout followed by immediate truncation of 281 tool results.

## Solution Assessment

Yes technically: using the existing tool-result pressure heuristic and persisted-session truncation helper before explicit compaction is the narrowest code fix I found. The unresolved question is whether maintainers want that session-state policy as the default.
