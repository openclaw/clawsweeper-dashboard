## Summary

Keep open: current main and the latest release still use compaction before the fallback truncation path, while this PR has sufficient proof and no blocking line-level defect; the remaining gate is maintainer acceptance of the default session-state tradeoff.

## What This Changes

The PR changes embedded-agent overflow recovery so tool-heavy overflows try deterministic tool-result truncation before explicit LLM compaction, with focused overflow-loop regression tests.

## Best Possible Solution

Land this only if maintainers accept truncation-first recovery as the default for tool-heavy overflows; otherwise preserve compaction-first behavior and design an explicit shorter-escalation or opt-in policy.

## Reproduction Assessment

Yes. Current main and the latest release still run explicit overflow compaction before the later truncation fallback, while the linked issue includes a live log showing a 912840 ms compaction timeout followed by immediate truncation of 281 tool results.

## Solution Assessment

Yes technically: using the existing tool-result pressure heuristic and truncation helper before explicit compaction is the narrowest code fix. The unresolved question is whether maintainers want that session-state policy as the default.
