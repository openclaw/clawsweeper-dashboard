Keep open for normal maintainer review and merge handling: the bug is source-reproducible on current main, the PR implements the narrow ordering fix with regression coverage, and I found no blocking correctness or security findings.

## What This Changes

The PR makes embedded-agent overflow recovery try existing tool-result truncation before explicit LLM compaction for tool-heavy overflow attempts, then adds focused run-loop regression tests.

## Best Possible Solution

Merge the narrow truncation-first recovery change after normal maintainer gating, preserving the existing compaction fallback and avoiding new timeout or configuration surface.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main code and tests show compaction runs before fallback truncation, and the linked live log shows a 912840ms compaction timeout followed by immediate truncation of 281 tool results. I did not force a live Telegram overflow locally.

## Solution Assessment

Yes. Reusing the existing deterministic truncation path before compaction only when tool-result pressure is detected, then falling back to compaction on a truncation miss, is the narrowest maintainable fix I found.
