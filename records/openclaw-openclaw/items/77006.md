## Summary

Keep open and block merge for now: the override remains relevant to the ACPX Codex auth problem, but the patch lets a shared Codex home become the wrapper credential target, including API-key `auth.json` writes, while the trust-boundary approval and successful live Codex ACP proof are still missing.

## What This Changes

The PR adds ACPX `codexHome` and `OPENCLAW_ACPX_CODEX_HOME` so Codex ACP can launch with an explicit Codex home instead of the isolated default.

## Best Possible Solution

Keep the isolated default, and only land the shared-home opt-in with isolated-only API-key writes, maintainer-approved trust-boundary docs, focused tests, and redacted successful managed Codex ACP proof.

## Reproduction Assessment

Yes at source level for the underlying auth-home issue: current main launches Codex ACP with an isolated `CODEX_HOME`, while upstream Codex reads credentials from the effective `CODEX_HOME/auth.json`; I did not run a live OAuth turn in this review.

## Solution Assessment

No as-is: an explicit shared home may be acceptable only after the credential write target is narrowed and maintainers approve the shared-home trust boundary.
