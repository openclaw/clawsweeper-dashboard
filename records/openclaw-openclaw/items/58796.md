## Summary

Keep open. Current main only implements the registry-row dedupe half of this PR while intentionally preserving duplicate-id diagnostics for bundled/global/workspace distinct-root shadows, and the PR still has blocking functional regressions around allowlisted provenance warnings and installed-index duplicate handling. Related maintainer work in #72409 is the safer place to settle the warning-policy direction before this branch is rebased or replaced.

## What This Changes

The PR dedupes plugin manifest registry rows by id, adds root-dir keyed manifest lookup for loader duplicate handling, suppresses selected bundled-shadow duplicate warnings, updates related tests, and adds changelog entries.

## Best Possible Solution

Keep the PR open for maintainer review, then rebase or replace it with a focused patch that preserves `plugins.allow` provenance semantics, threads installed-plugin ledger state through any suppression helper, aligns warning suppression with #72409, and adds attribution plus targeted registry/loader tests.

## Reproduction Assessment

Yes. Current main has static and test-level reproduction paths for the key regressions: `plugins.allow` currently suppresses untracked-provenance warnings, and registry tests currently require duplicate warnings for bundled/global and bundled/workspace distinct-root shadows; I did not execute tests under the read-only constraint.

## Solution Assessment

No. The branch combines a plausible root-dir manifest lookup change with broader duplicate-warning product policy changes and removes an already-shipped allowlist guard; a narrower rebased fix after the #72409 decision would be safer.
