## Summary

## What This Changes

The branch changes `src/commands/channels/status.ts` so live channel status displays up to five `allowFrom` entries with a `+N more` overflow suffix.

## Best Possible Solution

Land one small formatter/test update that makes truncation explicit outside the sender-token list and treats identifier exposure consistently across status surfaces.

## Reproduction Assessment

Yes, at source level: current main still slices live `account.allowFrom` to two entries, and `channelsStatusCommand` logs that formatter for non-JSON probe output. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

Unclear as proposed. Increasing the cap fixes the central silent truncation, but the overflow marker, identifier exposure policy, and focused proof need to align before this is the best merge shape.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Move the overflow marker out of the sender list:** `src/commands/channels/status.ts:148`
  - body: The overflow label is appended as `,+${remaining} more`, so a phone-number allowlist renders a final comma-separated `+2 more` token that looks like another sender ID. Put the count outside the `allow:` comma list, for example `allow:<ids> (+2 more)`, so diagnostics do not blur real entries with display metadata.
