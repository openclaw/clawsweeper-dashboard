## Summary

Keep open. Current main still has the live `channels status --probe` truncation this PR targets, but the PR overlaps with open PR #51179 and still needs a maintainer decision on allowlist display/privacy policy before landing.

## What This Changes

The PR changes the live gateway channel-status formatter to show up to five `allowFrom` entries and append a `,+N more` overflow suffix.

## Best Possible Solution

Land one consolidated allowlist display helper for live status and config-only summaries, with an explicit cap/redaction/overflow format, regression tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows the live formatter slices `allowFrom` to two entries, so a payload with three entries will omit the third in `channels status --probe` output.

## Solution Assessment

No. The PR fixes the live formatter, but the maintainable solution should coordinate with #51179, use an unambiguous overflow marker, settle the identifier exposure policy, and add focused coverage.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Disambiguate the overflow marker:** `src/commands/channels/status.ts:148`
  - body: The overflow suffix is appended as another comma-separated `+...` token, so phone-number allowlists can render `,+2 more` in the same shape as real sender IDs. Use a parenthesized or bracketed marker so the diagnostic output cannot be mistaken for another allowed sender.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/commands/channels/status.ts:148`
  - body: This is a user-facing CLI diagnostic fix, but the PR only changes the formatter. OpenClaw policy requires user-facing fix changes to add a `CHANGELOG.md` entry, so release notes would miss this behavior change.
  - confidence: 0.88
