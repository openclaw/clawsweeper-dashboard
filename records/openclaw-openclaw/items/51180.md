## Summary

Keep open. Current main still truncates the live `allowFrom` list, and this PR addresses that formatter, but it overlaps with #51179 and still needs an unambiguous overflow marker, changelog coverage, and a maintainer display/privacy decision before merge.

## What This Changes

The PR changes `formatGatewayChannelsStatusLines()` to show up to five `allowFrom` entries and append a `,+N more` suffix when additional entries exist.

## Best Possible Solution

Land one coordinated allowlist display helper for live gateway status and config summaries, with explicit capped or redacted overflow formatting, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows `formatGatewayChannelsStatusLines()` still calls `account.allowFrom.slice(0, 2)`, so a live status payload with three entries will hide the third.

## Solution Assessment

No. The PR fixes the live formatter path, but the best maintainable fix should coordinate with #51179, use an overflow marker that cannot be mistaken for another sender ID, and define the diagnostic exposure policy.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Disambiguate the overflow marker:** `src/commands/channels/status.ts:148`
  - body: The overflow suffix is appended as another comma-separated `+...` token, so phone-number allowlists can render `,+2 more` in the same shape as real sender IDs. Put the marker outside the comma-delimited sender list, such as `allow:a,b,c (+2 more)`.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/commands/channels/status.ts:148`
  - body: This is a user-facing CLI diagnostic fix, but the PR only changes the formatter. OpenClaw policy requires user-facing fix changes to add a `CHANGELOG.md` entry, so release notes would miss this behavior change.
  - confidence: 0.88
