## Summary

Keep open: current main still silently truncates the live `allowFrom` status output, and this PR is a plausible narrow fix for the linked bug, but it still needs real behavior proof and maintainer choice on the overflow/exposure policy before merge.

## What This Changes

The PR changes `src/commands/channels/status.ts` so live channel status output shows up to five `allowFrom` values and appends a comma-prefixed overflow count.

## Best Possible Solution

Land a tested status-formatting fix that removes silent truncation on the live probe path and uses an intentional, unambiguous cap and exposure policy across related status surfaces.

## Reproduction Assessment

Yes, from source inspection: current main's live formatter slices `allowFrom` to two entries, and `channelsStatusCommand()` logs that formatter for non-JSON gateway status. I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear. The PR fixes the live formatter path, but the maintainable solution should settle the overflow marker, identifier exposure, and whether adjacent summary/status surfaces share the same display policy.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Disambiguate the overflow marker:** `src/commands/channels/status.ts:148`
  - body: The suffix is appended as another comma-separated `+...` token, so phone-number allowlists can render `,+2 more` in the same shape as another sender ID. Put the marker outside the sender list, such as `allow:a,b,c (+2 more)`.
  - confidence: 0.82
