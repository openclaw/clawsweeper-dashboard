## Summary

Keep open: current main still truncates live `channels status --probe` `allowFrom` output to two entries, and this PR is a narrow fix for the reported diagnostic bug, but it still lacks contributor after-fix proof and leaves an intentional allowlist exposure/overflow-formatting decision for maintainers.

## What This Changes

The PR changes `src/commands/channels/status.ts` so live channel status output shows up to five `allowFrom` entries with an overflow count instead of always slicing to two entries.

## Best Possible Solution

Land one tested status-formatting fix that clearly indicates truncation without sender-like overflow tokens and intentionally aligns the live/status-summary allowlist exposure policy.

## Reproduction Assessment

Yes. Source inspection shows `formatGatewayChannelsStatusLines()` slices `account.allowFrom` to two, and `channelsStatusCommand()` uses that formatter for non-JSON `channels status --probe` output.

## Solution Assessment

Unclear. The branch fixes the central live truncation, but the overflow marker, display cap, identifier exposure, and adjacent summary/status policy need a maintainer decision before merge.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Put the overflow marker outside the sender list:** `src/commands/channels/status.ts:148`
  - body: The suffix is appended as another comma-separated `+...` token, so phone-number allowlists can render `allow:+491...,+2 more` in the same shape as another sender ID. Put the marker outside the sender list, such as `allow:a,b,c (+2 more)`.
  - confidence: 0.82
