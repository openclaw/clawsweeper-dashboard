## Summary

Keep open: current main still silently truncates the live `allowFrom` status output, and this PR is a focused fix for that path, but it still needs contributor real-behavior proof and maintainer choice on the overflow/redaction policy before merge.

## What This Changes

This PR changes `src/commands/channels/status.ts` so live channel status output shows the first five `allowFrom` entries and appends a `,+N more` suffix for longer lists.

## Best Possible Solution

Land one rebased, tested allowlist-display fix that removes silent truncation from live and summary status paths while using an unambiguous, intentional overflow or redaction format.

## Reproduction Assessment

Yes from source inspection: current main still slices live `allowFrom` output to two entries, and the non-JSON `channels status --probe` path logs that formatter. I did not run the command because this review is read-only.

## Solution Assessment

No. This PR addresses the live formatter path, but the best maintainable fix should also settle the adjacent summary path and use an overflow marker that cannot be mistaken for another sender ID.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Disambiguate the overflow marker:** `src/commands/channels/status.ts:148`
  - body: The overflow suffix is appended as another comma-separated `+...` token, so phone-number allowlists can render `,+2 more` in the same shape as a sender ID. Put the marker outside the sender list, such as `allow:a,b,c (+2 more)`.
  - confidence: 0.82
