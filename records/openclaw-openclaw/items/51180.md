## What This Changes

The PR changes `src/commands/channels/status.ts` so live channel status output shows up to five `allowFrom` entries and appends an overflow count.

## Best Possible Solution

Land one tested status-formatting fix that clearly indicates truncation without ambiguous sender-like tokens and intentionally aligns live/status-summary allowlist exposure policy.

## Reproduction Assessment

Yes, source-reproducible: current main slices `account.allowFrom` to two in `formatGatewayChannelsStatusLines()`, and `channelsStatusCommand()` logs that formatter for non-JSON probe output.

## Solution Assessment

Unclear. The branch removes silent live truncation, but the cap, overflow marker, identifier exposure, and adjacent summary/status policy should be settled before merge.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Put the overflow marker outside the sender list:** `src/commands/channels/status.ts:148`
  - body: The suffix is appended as another comma-separated `+...` token, so phone-number allowlists can render `allow:+491...,+2 more` in the same shape as another sender ID. Put the marker outside the sender list, such as `allow:a,b,c (+2 more)`.
  - confidence: 0.82
