## Summary

Keep open: current main still has the two-entry live `allowFrom` truncation and this PR is a focused partial fix, but it needs contributor real behavior proof plus maintainer choice on overflow/privacy and adjacent summary behavior before merge.

## What This Changes

This PR changes `src/commands/channels/status.ts` so live non-JSON `channels status --probe` output shows up to five `allowFrom` entries and appends `,+N more` when longer.

## Best Possible Solution

Land one rebased, tested status-formatting fix that removes silent `allowFrom` truncation from live and summary paths with an intentional cap, redaction, and unambiguous overflow marker.

## Reproduction Assessment

Yes. Source inspection on current main shows the live formatter slices `allowFrom` to two entries, and `channelsStatusCommand()` logs that formatter for non-JSON `channels status --probe` output.

## Solution Assessment

No. The live formatter change is useful, but the best maintainable fix should also settle the adjacent summary paths and use an overflow/redaction format that cannot be mistaken for another sender ID.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Disambiguate the overflow marker:** `src/commands/channels/status.ts:148`
  - body: The new suffix is appended as another comma-separated `+...` token, so phone-number allowlists can render `,+2 more` in the same shape as another sender ID. Put the marker outside the sender list, such as `allow:a,b,c (+2 more)`.
  - confidence: 0.82
