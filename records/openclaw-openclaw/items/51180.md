## Summary

Keep this PR open: it addresses a real source-reproducible status-output bug on current main, but it still needs contributor real behavior proof and maintainer judgment on allowlist identifier exposure, overflow wording, and adjacent summary/status output before merge.

## What This Changes

The PR changes `src/commands/channels/status.ts` so non-JSON live channel status output displays up to five `allowFrom` entries with an overflow suffix.

## Best Possible Solution

Land one tested status-formatting fix that removes silent `allowFrom` truncation from the live and relevant summary/status surfaces while using an intentional cap, redaction policy, and unambiguous overflow marker.

## Reproduction Assessment

Yes, from source inspection: current main still slices live `allowFrom` output to two entries, and the non-JSON `channels status --probe` path logs that formatter. I did not run the CLI because this review is read-only.

## Solution Assessment

No. The live formatter change is useful, but the best maintainable fix should also settle the adjacent summary/status surfaces and use an overflow format that cannot be confused with another sender ID.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Disambiguate the overflow marker:** `src/commands/channels/status.ts:148`
  - body: The new suffix is appended as another comma-separated `+...` token, so phone-number allowlists can render `,+2 more` in the same shape as another sender ID. Put the marker outside the sender list, such as `allow:a,b,c (+2 more)`.
  - confidence: 0.82
