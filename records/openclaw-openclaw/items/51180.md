## Summary

Keep open: the current main live formatter still truncates `allowFrom`, so the PR addresses a real bug, but it lacks contributor real-behavior proof and still needs maintainer choice on allowlist exposure, overflow formatting, and the adjacent summary path.

## What This Changes

This PR changes `src/commands/channels/status.ts` so live non-JSON `channels status --probe` output shows up to five `allowFrom` entries and appends a comma-prefixed overflow count.

## Best Possible Solution

Land one rebased, tested status-formatting fix that removes silent `allowFrom` truncation from live and summary paths while using an intentional cap, redaction, and overflow marker.

## Reproduction Assessment

Yes from source inspection: current main still slices live `allowFrom` output to two entries, and the non-JSON `channels status --probe` path logs that formatter. I did not run the CLI because this review was read-only.

## Solution Assessment

No. The live formatter change is useful, but the best maintainable fix should also settle the summary path and use an overflow or redaction format that cannot be mistaken for another sender ID.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Disambiguate the overflow marker:** `src/commands/channels/status.ts:148`
  - body: The new suffix is appended as another comma-separated `+...` token, so phone-number allowlists can render `,+2 more` in the same shape as another sender ID. Put the marker outside the sender list, such as `allow:a,b,c (+2 more)`.
  - confidence: 0.82
