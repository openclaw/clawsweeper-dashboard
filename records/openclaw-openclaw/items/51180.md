## Summary

Keep this PR open: current main still has the reported live formatter truncation, and the PR is a plausible narrow fix, but it still needs real behavior proof plus maintainer choice on allowlist display/privacy and the adjacent summary-path behavior.

## What This Changes

The PR changes the live `channels status --probe` formatter to show up to five `allowFrom` entries and append a comma-prefixed overflow suffix.

## Best Possible Solution

Land one rebased, tested allowlist-display fix that removes silent truncation from live and summary status paths while using an unambiguous overflow/redaction format.

## Reproduction Assessment

Yes from source inspection: current main still slices live `allowFrom` output to two entries, and the command path logs that formatter for non-JSON `channels status --probe` output.

## Solution Assessment

No. The PR fixes the live formatter path, but the best maintainable solution should also settle the adjacent summary path and use an overflow marker that cannot be mistaken for another sender ID.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Disambiguate the overflow marker:** `src/commands/channels/status.ts:148`
  - body: The overflow suffix is appended as another comma-separated `+...` token, so phone-number allowlists can render `,+2 more` in the same shape as a real sender ID. Put the marker outside the comma-delimited sender list, such as `allow:a,b,c (+2 more)`.
  - confidence: 0.82
