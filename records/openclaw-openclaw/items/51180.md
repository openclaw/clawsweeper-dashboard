## Summary

Keep open: current main still truncates the live `allowFrom` status output, and this PR is a plausible partial fix, but it needs real behavior proof plus maintainer consolidation with the broader sibling PR before merge.

## What This Changes

The PR changes `channels status --probe` live formatting to show up to five `allowFrom` entries and append a `+N more` overflow suffix.

## Best Possible Solution

Land one consolidated, tested allowlist display change across live gateway status and summary output, with explicit capped or redacted overflow formatting.

## Reproduction Assessment

Yes from source inspection, though I did not live-run a gateway: current main still slices `allowFrom` to two entries in the live formatter, so a payload with three entries hides the third.

## Solution Assessment

No. The PR fixes the live formatter path, but the best maintainable fix should coordinate with the sibling summary-path PR and use an overflow marker that cannot be mistaken for another sender ID.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Disambiguate the overflow marker:** `src/commands/channels/status.ts:148`
  - body: The overflow suffix is appended as another comma-separated `+...` token, so phone-number allowlists can render `,+2 more` in the same shape as a real sender ID. Put the marker outside the comma-delimited sender list, such as `allow:a,b,c (+2 more)`.
  - confidence: 0.82
