## Summary

Keep open: current main still truncates live `channels status --probe` allowlist output, so this PR is not obsolete. The branch is useful, but it needs consolidation with the overlapping summary-path PR and an explicit allowlist display/privacy decision before merge.

## What This Changes

This PR changes the live gateway channel-status formatter to show up to five `allowFrom` entries and append a comma-separated `+N more` suffix when additional entries exist.

## Best Possible Solution

Use one shared allowlist display helper for live gateway status and config-only summaries, choose a clear cap/redaction/overflow policy, render overflow with an unmistakable marker, and cover two-entry, three-entry, exact-limit, and overflow cases in focused tests before landing one consolidated fix.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to pass a live gateway status payload with three `allowFrom` entries into `formatGatewayChannelsStatusLines()` on current main; the formatter slices to two entries. The original linked bug report also provides a WhatsApp `channels status --probe` path with three or more allowlisted numbers.

## Solution Assessment

No, not yet. The PR addresses the live formatter, but the safer maintainable solution should coordinate with #51179, avoid an ambiguous overflow token, and make the display/privacy policy explicit with regression tests.
