## Summary

Keep this PR open: current main still has the reported live `channels status --probe` truncation, and this PR is a plausible narrow fix, but it still needs real behavior proof plus a maintainer choice on allowlist exposure and overflow formatting before merge.

## What This Changes

The PR changes `src/commands/channels/status.ts` so live non-JSON channel status output displays up to five `allowFrom` entries and appends an overflow count.

## Best Possible Solution

Land one rebased, tested status-formatting fix that removes silent `allowFrom` truncation from the live and summary/status surfaces while using an intentional cap, redaction, and unambiguous overflow marker.

## Reproduction Assessment

Yes, from source inspection: current main still slices live `allowFrom` output to two entries, and the non-JSON `channels status --probe` path logs that formatter. I did not run the CLI because this review is read-only.

## Solution Assessment

No. The live formatter change is useful, but the best maintainable fix should also settle the adjacent summary/status surfaces and use an overflow format that cannot be confused with another sender ID.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:
