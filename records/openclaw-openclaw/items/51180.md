## What This Changes

The branch changes `src/commands/channels/status.ts` so live `channels status --probe` output shows up to five `allowFrom` entries and appends a `+N more` overflow marker after that.

## Best Possible Solution

Land one tested status-display fix that clearly indicates truncation, intentionally bounds or redacts identifier exposure, and aligns live and summary status surfaces if maintainers want one policy.

## Reproduction Assessment

Yes. Current main formats non-JSON `channels status --probe` through `formatGatewayChannelsStatusLines()`, and that formatter slices `account.allowFrom` to two entries; a gateway payload with three entries would omit the third.

## Solution Assessment

Unclear. The branch fixes the central truncation, but the overflow marker, five-entry identifier exposure cap, and adjacent summary-status policy need maintainer alignment before merge.
