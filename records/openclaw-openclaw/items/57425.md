## Summary

Keep open: current main has meaningful restart-drain and partial main/subagent recovery work, but the broader recovery contract remains missing and a fresh v2026.5.22 field report shows cron jobs can still be consumed as hard failures during the drain window.

## What This Changes

Request a gateway-managed graceful restart system that records interrupted sessions, cron runs, and subagents, gates or defers restarts, replays drain-window work, and exposes restart recovery configuration.

## Best Possible Solution

Define one maintainer-owned restart recovery contract that composes the existing drain, hook, main-session, subagent, task, cron, and delivery pieces, then split implementation into focused PRs for durable state, replay/idempotency, cron retry, and notices.

## Reproduction Assessment

Yes, at source level: current main still rejects drain-window enqueues, marks cron execution failures as errors, and only partially recovers main/subagent sessions. I did not run a live multi-channel restart E2E in this read-only review.

## Solution Assessment

Unclear. The manifest approach is plausible, but the best maintainable solution needs a maintainer-owned contract across restart lifecycle, durable state, cron retry, replay idempotency, subagent reporting, and channel UX rather than one broad patch.
