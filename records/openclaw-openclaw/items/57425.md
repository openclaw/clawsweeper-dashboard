## Summary

Keep open. Current main covers major pieces of graceful restart, including active-work drain and main/subagent restart recovery, but the remaining cron replay/defer behavior, drain-window inbound message queuing, cross-channel catch-up, and config/product contract are still not fully implemented.

## What This Changes

Request graceful gateway restart behavior that drains or recovers active sessions, subagents, cron runs, and drain-window messages, optionally with new restart configuration.

## Best Possible Solution

Keep this as the broad graceful-restart tracker while maintainers define the durable recovery contract that composes existing restart drain/main-session recovery with cron retry/defer behavior and drain-window inbound message replay.

## Reproduction Assessment

Yes at source level. Current main still marks interrupted cron runs as errors and rejects new drain-window enqueues, while existing recovery covers only part of the broader requested behavior.

## Solution Assessment

Unclear as a concrete implementation path. The manifest/config proposal is plausible, but the best solution needs a maintainer-owned contract across restart drain, session recovery, cron scheduling, durable queues, and channel replay boundaries.
