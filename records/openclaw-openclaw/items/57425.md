## Summary

Keep open: current main has several important restart-drain and recovery pieces, but it still does not provide the requested coherent gateway restart recovery contract for drain-window replay, cron retry, universal interruption propagation, and a dedicated restart config surface. A member explicitly left this issue open after landing the active-run drain fix, and current source still supports that split.

## What This Changes

The issue requests graceful gateway restart behavior with active-session manifests, post-restart recovery notices, cron retry, readiness gating, drain-window message replay, and new gateway restart configuration.

## Best Possible Solution

Define one maintainer-owned restart recovery contract that composes the existing drain, safe-restart, main-session, subagent, task, cron, and delivery pieces, then split implementation into focused PRs for durable state, replay/idempotency, cron retry, and user-visible notices.

## Reproduction Assessment

Yes, at source level. Current main still rejects drain-window enqueues, marks interrupted cron runs failed instead of retrying them, lacks the requested gateway.restart config block, and only partially recovers main and subagent sessions after forced or timed-out drains.

## Solution Assessment

Unclear. The manifest approach is plausible, but the best maintainable solution needs a maintainer-owned contract across restart lifecycle, durable state, cron retry, replay idempotency, subagent reporting, and channel UX rather than a single broad patch.
