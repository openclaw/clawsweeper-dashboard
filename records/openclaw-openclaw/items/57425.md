## Summary

Keep open: current main has meaningful restart-drain, restart-continuation, and main-session recovery work, but the central request is still not fully implemented because cron and subagent recovery remain intentionally limited and newer field reports show drain-window work can still be consumed as failure.

## What This Changes

The issue requests graceful gateway restart behavior that preserves or recovers interrupted sessions, cron runs, subagent state, and drain-window messages.

## Best Possible Solution

Define one maintainer-owned restart recovery contract, then split implementation into focused changes for durable state, replay/idempotency, cron retry, subagent reporting, and user-visible notices.

## Reproduction Assessment

Yes, at source level: current main skips cron/subagent recovery in the main-session path and marks interrupted cron jobs failed on startup, matching remaining scope from the issue. I did not run a live gateway restart scenario in this read-only review.

## Solution Assessment

No, the proposed manifest/config bundle is plausible but not yet proven as the best final shape. A safer direction is a maintainer-owned contract that composes existing SQLite/session-store, restart sentinel, cron, and channel delivery paths.
