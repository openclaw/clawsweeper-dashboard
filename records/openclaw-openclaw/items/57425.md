## Summary

Keep open: current main has partial restart-drain and main/subagent recovery, but it still lacks the requested coherent restart recovery contract for drain-window replay, cron retry/defer semantics, full interruption propagation, and dedicated restart configuration. A member explicitly left this broader scope open after https://github.com/openclaw/openclaw/pull/71465, and a v2026.5.22 field report confirms scheduled cron work can still be consumed as a hard drain error.

## What This Changes

The issue requests graceful gateway restart behavior with active session manifests, interruption/recovery events, cron retry or defer semantics, drain-window message queuing, readiness gating, and restart-related configuration.

## Best Possible Solution

Define one maintainer-owned restart recovery contract that composes existing drain, hook, main-session, subagent, task, cron, and delivery pieces, then split durable state, replay/idempotency, cron retry, and notices into focused PRs.

## Reproduction Assessment

Yes, at source level: current main still rejects drain-window enqueues, marks interrupted cron runs failed instead of retrying them, lacks the requested gateway.restart config block, and only partially recovers main/subagent sessions. I did not run a live multi-channel restart E2E in this read-only review.

## Solution Assessment

Unclear. The manifest approach is plausible, but the best maintainable solution needs a maintainer-owned contract across restart lifecycle, durable state, cron retry, replay idempotency, subagent reporting, and channel UX rather than one broad patch.
