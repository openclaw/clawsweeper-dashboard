## Summary

Keep open: current main has meaningful restart-drain and partial recovery work, but it still lacks the requested coherent gateway restart recovery contract for drain-window replay, cron retry, full interruption propagation, and dedicated restart configuration; a member explicitly left this broader scope open after PR 71465 landed.

## What This Changes

The issue requests graceful gateway restart behavior with pre-restart session manifests, post-restart recovery events, readiness gating, drain-window message replay, cron retry, and gateway.restart configuration.

## Best Possible Solution

Define one maintainer-owned restart recovery contract that composes the existing drain, hook, main-session, subagent, task, cron, and delivery pieces, then split implementation into focused PRs for durable state, replay/idempotency, cron retry, and notices.

## Reproduction Assessment

Yes, at source level: current main still rejects drain-window enqueues, marks interrupted cron runs failed instead of retrying them, lacks the requested gateway.restart config block, and only partially recovers main/subagent sessions. I did not run a live multi-channel restart E2E in this read-only review.

## Solution Assessment

Unclear. The manifest approach is plausible, but the best maintainable solution needs a maintainer-owned contract across restart lifecycle, durable state, cron retry, replay idempotency, subagent reporting, and channel UX rather than one broad patch.
