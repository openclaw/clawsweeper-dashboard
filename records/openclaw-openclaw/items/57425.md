## Summary

Keep open: current main has meaningful restart-drain, main-session, subagent, hook, and restart-sentinel pieces, but it still lacks the requested coherent recovery contract for drain-window replay, cron retry, full interruption propagation, and a dedicated gateway.restart config surface. A maintainer already closed the competing implementation PR in favor of leaving this broader issue open, and current source still supports that split.

## What This Changes

The issue requests graceful gateway restart behavior that preserves interrupted sessions, subagents, cron runs, and drain-window messages across restarts.

## Best Possible Solution

Define one maintainer-owned restart recovery contract that composes existing drain, restart-sentinel, main-session, subagent, task, and cron pieces, then split implementation into focused PRs for durable storage, replay/idempotency, cron retry, and user-visible interruption notices.

## Reproduction Assessment

Yes, at source level. Current main still rejects drain-window enqueues, marks interrupted cron runs failed instead of retrying them, lacks the requested gateway.restart config block, and only partially recovers main and subagent sessions after forced or timed-out drains.

## Solution Assessment

Unclear. The manifest approach is plausible, but the best maintainable solution needs a maintainer-owned contract across restart lifecycle, durable state, cron retry, replay idempotency, subagent reporting, and channel UX rather than a single broad patch.
