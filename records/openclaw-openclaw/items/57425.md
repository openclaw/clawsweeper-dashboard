## Summary

Keep open: current main has partial safe-restart, restart-drain, main-session, subagent, cron accounting, and durable delivery pieces, but the requested manifest, drain-window replay, cron retry, interruption propagation, readiness-gate policy, and `gateway.restart` config contract are still not implemented as one coherent restart-recovery feature.

## What This Changes

Request a first-class graceful gateway restart mechanism that records active sessions, cron runs, and subagents before restart, recovers or replays work afterward, gates unsafe restarts, queues drain-window messages, and adds `gateway.restart` config keys.

## Best Possible Solution

Define a versioned restart recovery contract, then split it into focused changes for manifest/persistence, drain-window replay, cron retry semantics, session/subagent interruption notices, and documented config policy.

## Reproduction Assessment

Yes. Source inspection on current main shows drain-window enqueues reject before replay, requested `gateway.restart` config keys are absent, interrupted cron runs are marked as errors rather than retried, and existing recovery paths are partial rather than a full manifest/replay contract.

## Solution Assessment

Unclear. The proposed manifest approach is plausible, but the maintainable solution needs maintainer-scoped lifecycle, persistence, cron, delivery, idempotency, and UX semantics rather than one broad automatic fix.
