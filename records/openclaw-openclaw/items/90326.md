## Summary

## What This Changes

The branch makes Fireworks manifest catalog IDs defer from the plugin dynamic hook to core static-catalog resolution, adds explicit `reasoning:false` metadata for both bundled Kimi rows, and adds plugin/resolver regression coverage.

## Best Possible Solution

Land the canonical static-catalog path if provider/runtime owners accept the resolution-order compatibility risk and keep the configured-model precedence test in place.

## Reproduction Assessment

Yes, from source inspection: current main and v2026.6.1 clone the default Fireworks template before the static manifest row can supply Kimi K2.6's 262144 limits. I did not run local tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes, this is the best owner-boundary fix: manifest-declared Fireworks IDs should resolve from the core static catalog instead of duplicating catalog metadata in the provider hook. The safer alternative would be another plugin-side copy of the manifest row, but that would preserve the drift this PR removes.
