## Summary

Keep open: the PR fixes the intended Fireworks catalog-limit drift in fallback-enabled runtime paths, but the new manifest-ID early return removes provider-hook resolution for sync and non-static-fallback resolver callers, so exact Kimi catalog refs can become unknown unless the branch preserves those paths or extends core fallback coverage.

## What This Changes

The PR changes Fireworks so manifest-declared Kimi catalog IDs defer from the plugin dynamic resolver to core static-catalog resolution, adds explicit `reasoning:false` metadata to both Kimi catalog rows, and updates Fireworks tests.

## Best Possible Solution

Keep `openclaw.plugin.json` as the canonical Fireworks catalog while preserving exact manifest-ID resolution across both fallback-enabled and non-fallback resolver paths, with regression coverage for each path.

## Reproduction Assessment

Yes, from source. Current main clones the Fireworks default template for Kimi K2.6 in the provider dynamic hook, while the PR diff makes exact manifest IDs return `undefined` and the core sync resolver has no general static-catalog fallback.

## Solution Assessment

No. Deferring manifest IDs to core is a good direction only for fallback-enabled async paths; the best fix must also preserve exact catalog ID resolution for resolver callers that still depend on the provider dynamic hook.
