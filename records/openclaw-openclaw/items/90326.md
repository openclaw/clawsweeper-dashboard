## Summary

This PR is still useful, but the latest head is not merge-ready: exact Fireworks manifest IDs now return `undefined` from the provider hook and only resolve correctly on callers that opt into bundled static-catalog fallback, leaving existing non-fallback resolver paths able to turn those models into unknown models.

## What This Changes

The PR changes the Fireworks provider so manifest catalog IDs defer from `resolveDynamicModel` to core static-catalog resolution, adds `reasoning:false` to both Kimi manifest rows, deep-clones catalog rows, and updates Fireworks tests.

## Best Possible Solution

Keep the Fireworks manifest as the authoritative metadata source, but ensure exact Fireworks catalog IDs resolve correctly in both fallback-enabled async paths and normal resolver paths before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: exact Fireworks manifest IDs now return `undefined` from the provider hook, while sync `resolveModel`/`resolveModelWithRegistry` paths do not enable bundled static-catalog fallback. I did not run a live Fireworks command in this read-only review.

## Solution Assessment

No, this is not yet the best fix because it relies on a fallback that is not available to every resolver caller. The better fix is to keep manifest metadata authoritative while preserving exact catalog ID resolution across both fallback-enabled and non-fallback resolver paths.
