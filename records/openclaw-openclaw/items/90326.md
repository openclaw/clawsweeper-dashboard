## What This Changes

The PR defers Fireworks manifest model IDs to core’s bundled static catalog, preserves plugin synthesis for non-catalog IDs, explicitly marks two Kimi rows as non-reasoning, and adds catalog-resolution and config-precedence coverage.

## Best Possible Solution

Use the manifest as the single source of truth for cataloged Fireworks models, retain plugin-local synthesis only for genuinely dynamic IDs, and coordinate landing order with the companion PR after provider/runtime owner confirmation.

## Reproduction Assessment

Yes. Current main’s Fireworks dynamic resolver clones the default model template for a catalog ID absent from the runtime registry, which can replace that row’s distinct context limits and nested compatibility metadata.

## Solution Assessment

Yes, subject to owner acceptance. Deferring manifest rows to the existing generic core catalog seam removes the competing metadata source while retaining the provider hook for genuinely dynamic IDs.
