## Summary

Keep open: current main still shows a generic existing-config gate before the provider credential path, and the docs do not signpost the post-onboarding provider-switch command clearly enough. The broader stale provider/session-cache behavior is separately tracked at https://github.com/openclaw/openclaw/issues/88372, so this issue remains a narrow discoverability cleanup.

## What This Changes

Make the Windows/post-onboarding provider switch path discoverable when users rerun onboarding with `--auth-choice <provider>`.

## Best Possible Solution

Preserve the safe existing-config gate, but add auth-choice-aware prompt copy and docs that clearly send post-onboarding provider changes through the credential update path or `openclaw configure --section model`.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms the remaining discoverability problem: existing config handling appears before provider auth handling and the current copy/docs do not explain the provider key-update path.

## Solution Assessment

Yes, a narrow copy/docs fix is the best first step. Bypassing config handling would be riskier because that step protects existing config and reset behavior.
