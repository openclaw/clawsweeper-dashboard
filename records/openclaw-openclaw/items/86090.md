## Summary

Keep open: current main still has a plausible plugin heartbeat wake gap, and the report is too concrete to close as fixed, stale, or duplicate. The direct plugin `runHeartbeatOnce` path cannot pass the wake metadata that the runner uses to inspect queued wake payloads, while related fixes cover `hook:wake`, `source:"hook"`, cron, and scheduled wakes rather than the bare `reason:"hook"` call shown here.

## What This Changes

The issue reports that plugin-triggered `runHeartbeatOnce` can return `{status:"ran"}` quickly after enqueueing a system event even though no meaningful model turn consumes the queued event.

## Best Possible Solution

Define the supported plugin wake semantics, then make direct plugin heartbeat runs either carry wake `source`/`intent` into the runner or return an explicit skipped reason when no model turn consumed the queued event, with regression coverage for queued event consumption.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show that the reported plugin call shape cannot express the wake metadata current `runHeartbeatOnce` uses to inspect wake payloads; the exact fast no-model return still needs a targeted test or live log proof.

## Solution Assessment

Unclear. The likely maintainable fix is to align the public plugin helper with the runner's wake contract, but maintainers should first decide whether direct `runHeartbeatOnce` should default to hook/immediate semantics or expose explicit `source`/`intent` fields.
