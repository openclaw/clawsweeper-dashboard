## Summary

Keep open: the PR addresses a real Gateway startup-planning gap for explicitly configured memory embedding providers and follows the existing manifest-contract pattern. The remaining decision is compatibility-sensitive: maintainers should accept that existing `memorySearch.provider` configs can now load an additional provider plugin at startup.

## What This Changes

The PR adds Gateway startup loading for plugins that own explicitly configured `agents.*.memorySearch.provider` memory embedding providers, adds an unregistered-provider warning, expands startup tests, and updates the optional deadcode allowlist.

## Best Possible Solution

Land the startup-planning fix if maintainers accept the earlier provider-plugin loading as the compatibility repair, and track unset or `auto` provider behavior separately if it is confirmed to fail.
