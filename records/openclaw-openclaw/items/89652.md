## Summary

Keep open. Current main still lacks startup planning for configured memory embedding provider owners, and this PR appears to address that startup gap with focused tests and real terminal proof, but merge still needs maintainer compatibility acceptance because explicit memorySearch configs can now activate provider plugins during Gateway startup.

## What This Changes

The PR updates Gateway startup plugin planning and diagnostics so configured `agents.*.memorySearch.provider` and fallback values can load the plugin that owns the matching memory embedding provider contract.

## Best Possible Solution

Land the startup-plan fix after maintainer compatibility signoff, while keeping the runtime fail-fast follow-up in https://github.com/openclaw/openclaw/pull/90336 separate.

## Reproduction Assessment

Yes, source-reproducible: current main does not include memory embedding provider ids in Gateway startup provider collection, while runtime config resolves `memorySearch.provider` such as `openai`. I did not boot a live Gateway in this read-only review.

## Solution Assessment

Yes, this is the best narrow fix for the startup gap: use existing manifest provider contracts and startup activation policy rather than adding a new config surface. The remaining question is maintainer acceptance of the earlier plugin activation behavior, not a different code location.
