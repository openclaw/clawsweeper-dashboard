## Summary

Keep open: current main and the latest shipped release still intentionally clamp Codex `danger-full-access` to `workspace-write` whenever an OpenClaw sandbox is active, and there is no Docker-specific supported opt-out for the requested nested-sandbox failure mode. The request is plausible, but it changes a sandbox boundary and needs maintainer security/product review rather than cleanup closure.

## What This Changes

Request a supported Codex plugin/runtime configuration that lets Docker-sandboxed Codex agents avoid the inner Codex sandbox while preserving OpenClaw's outer Docker sandbox and tool policy.

## Best Possible Solution

Add a narrowly scoped, documented Codex plugin opt-in for already-isolated OpenClaw Docker sandbox runs, likely using the Codex external-sandbox contract or an equivalent guarded mapping, with diagnostics and tests that prove OpenClaw remains the outer sandbox and tool-policy authority.

## Reproduction Assessment

No high-confidence live repro was run for the exact Docker/bwrap failure. Source inspection on current main and v2026.5.12 does confirm the reported configuration is clamped from Codex `danger-full-access` to `workspace-write` under an active OpenClaw sandbox.

## Solution Assessment

Unclear until owners decide the trust model. A narrow opt-in or `externalSandbox`-style mapping with guardrails is safer than globally removing the existing clamp, because current docs frame the clamp as intentional sandbox protection.
