## Summary

Keep open: current main still keeps Codex app-server code-mode turns in Codex `workspaceWrite` whenever an OpenClaw sandbox is active, and the related draft PR only adds doctor/docs mitigation for Ubuntu/AppArmor namespace denials. The remaining decision is the security/product boundary for whether Codex execution should move inside the OpenClaw sandbox or get a guarded no-inner-sandbox opt-in.

## What This Changes

The issue requests a supported configuration path to avoid Codex's inner sandbox when an OpenClaw Docker sandbox is already active.

## Best Possible Solution

Make an explicit maintainer/security decision for the Codex/OpenClaw sandbox trust model, then either move Codex execution into the OpenClaw sandbox or add a narrowly guarded Docker-only opt-in with docs, diagnostics, and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection confirms the reported policy path still maps active OpenClaw sandbox turns to Codex `workspaceWrite`, while the bwrap failure itself depends on host namespace/AppArmor policy and is supported by reporter logs.

## Solution Assessment

Unclear until maintainers decide the trust model. A bare `danger-full-access` opt-out is not obviously the best fix because current docs say the managed app-server still runs on the Gateway host; a guarded opt-in or moving execution into the sandbox would need explicit security review.
