## Summary

Keep open: current main and v2026.5.12 intentionally narrow Codex `danger-full-access` to `workspace-write` when an OpenClaw sandbox is active, and this issue asks for a new security-sensitive config path that is not implemented yet.

## What This Changes

The issue requests a supported Codex/OpenClaw configuration path that keeps Docker sandboxing as the outer boundary while disabling or bypassing Codex's nested bwrap sandbox for app-server turns.

## Best Possible Solution

Add a narrow Codex plugin config path, likely mapping to Codex `externalSandbox` or a guarded `dangerFullAccess` pass-through only after `resolveSandboxContext` confirms the OpenClaw sandbox is active, with docs and Docker smoke coverage.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection does confirm the current-main and v2026.5.12 path that forces `danger-full-access` to `workspace-write` under an active OpenClaw sandbox.

## Solution Assessment

Unclear pending maintainer decision. The requested single-boundary direction is plausible, but the safest implementation shape should be decided around Codex's `externalSandbox` contract and OpenClaw's sandbox/tool-policy guarantees.
