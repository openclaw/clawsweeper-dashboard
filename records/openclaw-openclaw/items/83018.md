## Summary

Keep open: current main preserves Codex network egress after the related fix, but it still deliberately sends a Codex `workspaceWrite` turn sandbox policy whenever an OpenClaw sandbox is active. The requested Docker-specific way to avoid the inner bwrap/filesystem sandbox remains unimplemented and needs sandbox/security product review, especially after the reporter narrowed the repro to managed Codex app-server plus `sessions_spawn`.

## What This Changes

Request a supported Codex configuration or policy so Docker-sandboxed Codex agents can use OpenClaw's Docker sandbox as the single isolation boundary instead of Codex's inner bwrap/workspace-write sandbox.

## Best Possible Solution

Add an explicit, narrowly guarded Codex app-server policy for already Docker-isolated sessions, likely using an external-sandbox-style Codex turn policy, with docs, diagnostics, and live `sessions_spawn` proof that OpenClaw's Docker sandbox and tool policy remain the boundary.

## Reproduction Assessment

No high-confidence live current-main repro was run in this review. Source and comments provide a clear repro shape, and current main still sends workspaceWrite Codex turn policy under active OpenClaw sandboxing, but the bwrap failure should be rechecked after the merged network-egress fix.

## Solution Assessment

Unclear until owners decide the trust model. A global removal of Codex sandboxing is not the best path; a narrow opt-in or external-sandbox policy guarded by active Docker sandboxing is the safer direction to evaluate.
