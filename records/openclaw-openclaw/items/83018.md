## Summary

Keep open: current main still keeps Codex app-server turns on Codex workspaceWrite sandboxing when an OpenClaw sandbox is active, and the related duplicate was closed back to this canonical issue. Recent maintainer discussion is still investigating the Ubuntu/AppArmor/bwrap shape, so there is no implemented fix or maintainer decision to close against.

## What This Changes

The issue requests a supported Codex configuration path to avoid Codex inner bwrap/workspace-write sandboxing when OpenClaw Docker sandboxing is already the isolation boundary.

## Best Possible Solution

Define a guarded Docker/OpenClaw-owned Codex sandbox mode, likely using the Codex externalSandbox/no-bwrap contract only when the OpenClaw sandbox boundary and tool policy remain intact, with diagnostics and docs for hosts where bwrap cannot run.

## Reproduction Assessment

No high-confidence live current-main repro was run in this review. Source inspection confirms current main still sends workspaceWrite sandboxPolicy under an active OpenClaw sandbox, while the actual bwrap failure is supported by reporter and duplicate-issue logs but appears host-environment dependent.

## Solution Assessment

Unclear: disabling or bypassing Codex sandboxing broadly is not proven to be the safest fix. A guarded Docker/OpenClaw-owned externalSandbox path or documented host preflight is safer than removing the current clamp globally.
