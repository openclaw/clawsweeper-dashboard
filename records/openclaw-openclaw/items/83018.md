## Summary

Keep open: current main still sends Codex app-server turns through a Codex `workspaceWrite` sandbox policy whenever an OpenClaw sandbox is active, and there is no supported Docker-specific opt-out. The related network-egress fix in https://github.com/openclaw/openclaw/pull/83502 does not resolve the nested bwrap failure path, and the active discussion is correctly waiting on sandbox/security product review.

## What This Changes

The issue requests a supported configuration option to disable Codex's inner sandbox for Codex-backed agents already isolated by OpenClaw Docker sandboxing.

## Best Possible Solution

Add a narrowly scoped Codex plugin opt-in for externally managed Docker sandbox turns, with docs and regression/live proof showing OpenClaw keeps the outer sandbox and tool policy while Codex avoids the nested bwrap path only in the approved configuration.

## Reproduction Assessment

No, we do not yet have a high-confidence independent live reproduction of the exact bwrap failure. The reporter supplied a narrowed `sessions_spawn` repro, and source inspection confirms current main still sends active-sandbox Codex turns through `workspaceWrite`.

## Solution Assessment

Unclear until sandbox owners decide the trust model. A narrow Docker-only opt-in with guardrails is safer than removing the current Codex `workspaceWrite` policy globally.
