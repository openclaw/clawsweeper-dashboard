- `impact:security`: The request is about which sandbox boundary owns isolation and whether Codex may bypass its inner bwrap/workspace-write sandbox under OpenClaw Docker isolation.

## Summary

Keep open: current main still sends Codex app-server turns an inner `workspaceWrite` sandbox policy whenever an OpenClaw sandbox is active, and the requested Docker-scoped opt-out or externally managed sandbox mode is not implemented. The recent network-egress fix preserved `workspaceWrite`, so this still needs sandbox/security product review rather than conservative cleanup.

## What This Changes

Request a supported Codex configuration path to avoid Codex's inner bwrap/workspace-write sandbox when a Codex agent is already isolated by OpenClaw Docker sandboxing.

## Best Possible Solution

Add a narrow, explicit Codex plugin option or externally managed sandbox mode for Docker-sandboxed `sessions_spawn` runs, with guardrails, diagnostics, docs, and tests that preserve OpenClaw's outer sandbox, tool policy, and egress contract.

## Reproduction Assessment

No high-confidence maintainer reproduction is available yet. Source inspection proves current main still applies an inner Codex `workspaceWrite` policy when OpenClaw sandboxing is active, and the reporter supplied a narrowed `sessions_spawn` bwrap failure path that still needs live maintainer proof.

## Solution Assessment

Unclear until the sandbox trust model is decided. A guarded opt-in or Codex external-sandbox mapping is safer than globally allowing `danger-full-access`, because the app-server path can still be host-side and current docs frame the inner clamp as protective.
