## Summary

Keep open for maintainer/security review: current main still narrows Codex `danger-full-access` to `workspace-write` under an active OpenClaw sandbox, and the later discussion plus https://github.com/openclaw/openclaw/issues/83796 shows the requested opt-out overlaps a larger containment-boundary decision rather than a safe cleanup close.

## What This Changes

The issue requests a supported way for Docker-sandboxed Codex agents to avoid Codex's inner bwrap sandbox while preserving OpenClaw's outer sandbox and tool policy.

## Best Possible Solution

Resolve the trust-boundary decision in the Codex harness: either run Codex-native execution inside the active per-agent Docker sandbox, or fail/warn clearly and document the supported host-policy repair path for bwrap namespace denials.

## Reproduction Assessment

No high-confidence current-main live reproduction was run in this review. The source behavior is clear from prior issue evidence, and multiple commenters supplied concrete Ubuntu/AppArmor and Docker-sandbox observations, but the exact correct runtime fix still depends on the containment model.

## Solution Assessment

Unclear: adding a simple Codex inner-sandbox disable option is probably not the best standalone fix after the discussion in https://github.com/openclaw/openclaw/issues/83796. A boundary-correct solution should first decide whether Codex app-server execution belongs inside the per-agent sandbox or must fail/warn when it cannot be contained.
