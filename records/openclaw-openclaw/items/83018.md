## Summary

Keep open: current main fixed a related sandbox egress regression, but it still intentionally keeps Codex workspace-write filesystem sandboxing for active OpenClaw sandboxes and exposes no Docker-only opt-out. The issue also has active maintainer investigation, and the exact bwrap failure needs a rerun against current main before changing sandbox policy.

## What This Changes

Request a supported Codex/OpenClaw configuration path to avoid Codex's inner bwrap/workspace-write sandbox when an OpenClaw Docker sandbox already isolates the agent.

## Best Possible Solution

Have owners decide a narrow Docker-only policy: either document that Codex inner workspace-write remains required for host-started app-server runs, or add an explicit guarded opt-in after live proof shows PR 83502 did not resolve the bwrap failure.

## Reproduction Assessment

Partially: source inspection gives a high-confidence path for the missing opt-out, because current main still sends a Codex workspaceWrite sandboxPolicy under an active OpenClaw sandbox and has no Docker-only disable key. The exact RTM_NEWADDR failure was not live-reproduced here and should be rerun against current main after PR 83502.

## Solution Assessment

Unclear: disabling the inner sandbox globally is not the best default because the managed app-server runs host-side. A guarded Docker-only opt-in, or an explicit decision to keep the current workspace-write model, is safer than removing the clamp broadly.
