## Summary

Keep open: current main still sends Codex `workspaceWrite` sandbox policy whenever an OpenClaw sandbox is active and exposes no Docker/Ubuntu/AppArmor opt-out or legacy Landlock config. The discussion has narrowed this to managed host app-server plus `sessions_spawn`, with active maintainer investigation and a sandbox/security product decision still needed.

## What This Changes

The issue requests a supported Codex plugin configuration path to avoid nested Codex Linux sandboxing when OpenClaw Docker sandboxing is already the isolation boundary.

## Best Possible Solution

Define a guarded Codex plugin policy for already-isolated OpenClaw Docker sessions, with explicit trust-model docs, diagnostics or preflight for Ubuntu/AppArmor bwrap failures, and regression coverage for default protected behavior plus the opt-in path.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection confirms OpenClaw still sends Codex `workspaceWrite` under active sandboxing, and upstream Codex 0.130.0 uses bubblewrap by default for Linux sandboxed commands, but the Ubuntu/AppArmor failure itself remains reporter- and host-supported rather than locally reproduced.

## Solution Assessment

Unclear. A plain `danger-full-access` override is likely not the best fix because the app-server is host-side and the latest comments suggest Ubuntu/AppArmor may need a legacy Landlock or external-sandbox path with guardrails.
