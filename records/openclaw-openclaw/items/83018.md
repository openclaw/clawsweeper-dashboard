## Summary

Keep open: current main still documents and enforces the Codex app-server sandbox clamp when OpenClaw sandboxing is active, while a maintainer has an active runtime-shape question after failing to reproduce the stock managed path; this remains a security/product decision rather than a cleanup close.

## What This Changes

The issue requests a supported Codex configuration path to disable or override Codex's inner sandbox when the agent is already isolated by OpenClaw's Docker sandbox.

## Best Possible Solution

Add a narrow, explicitly documented Codex plugin opt-in for already-isolated Docker executions only after maintainers confirm the runtime topology and security guardrails for the outer sandbox and tool policy.

## Reproduction Assessment

No high-confidence live reproduction is established for the reported bwrap failure. Source inspection does confirm that current main clamps Codex danger-full-access to workspace-write under active OpenClaw sandboxing, and the latest maintainer check says the stock managed Docker path did not reproduce the failure.

## Solution Assessment

Unclear until maintainers decide the trust model and confirm the failing runtime shape. A narrowly scoped opt-in with diagnostics is safer than removing the existing clamp globally.
