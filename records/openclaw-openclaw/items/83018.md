## Label Justifications

- `P1`: The report describes basic Codex shell/file execution failing inside Docker-sandboxed agents, and the fix affects a high-impact sandbox/runtime workflow.
- `impact:security`: The requested change directly alters the nested Codex/OpenClaw sandbox boundary and must preserve Docker isolation and tool-policy guarantees.

## Summary

Keep open: current `main` still intentionally clamps Codex `danger-full-access` to `workspace-write` whenever OpenClaw sandboxing is active, the requested Docker-specific opt-out is not implemented, and the latest maintainer comment says the stock managed path did not reproduce the exact bwrap failure yet.

## What This Changes

Request a supported Codex configuration path to avoid Codex's inner app-server sandbox when an agent is already isolated by OpenClaw Docker sandboxing.

## Best Possible Solution

Keep the issue open for sandbox/security owners to decide a narrow Docker-only or externally managed Codex sandbox opt-in, backed by live repro proof, docs, and focused tests that preserve OpenClaw's outer sandbox and tool policy.

## Reproduction Assessment

No. Source inspection confirms the clamp from Codex `danger-full-access` to `workspace-write`, but the latest maintainer comment says the stock managed Codex app-server plus Docker sandbox path did not reproduce the reported bwrap failure and still needs runtime-shape details.

## Solution Assessment

Unclear. A narrow opt-in may be the right end state, but removing the clamp globally would conflict with current security docs and needs a trust-model decision plus live proof first.
