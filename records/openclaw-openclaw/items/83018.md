## Summary

Keep open: current main still sends active OpenClaw-sandboxed Codex turns through Codex `workspaceWrite`/bwrap, and the merged fix for https://github.com/openclaw/openclaw/issues/83347 only preserves egress rather than adding the requested inner-sandbox opt-out. The issue also has fresh maintainer follow-up and assignment, so this needs sandbox/security product review rather than cleanup closure.

## What This Changes

The issue requests a supported Codex plugin option or mode to run without Codex's nested workspace-write/bwrap sandbox when OpenClaw Docker sandboxing is already active.

## Best Possible Solution

Decide the trust model explicitly, then either document that the inner Codex sandbox remains required or add a narrowly guarded Codex plugin opt-in for externally managed OpenClaw Docker sandbox runs with tests and live proof.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection and the reporter's follow-up do show the relevant current-main path: an active OpenClaw sandbox causes Codex app-server turns to receive a `workspaceWrite` sandbox policy, which is the inner bwrap path implicated by the failure.

## Solution Assessment

Unclear until owners decide the trust model. A narrow, explicit externally-managed-sandbox opt-in is safer than globally disabling Codex's filesystem sandbox or treating `danger-full-access` as acceptable whenever any OpenClaw sandbox is active.
