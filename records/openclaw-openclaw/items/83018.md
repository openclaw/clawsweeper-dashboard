## Summary

Keep open: current main and v2026.5.12 still intentionally clamp Codex `danger-full-access` to `workspace-write` when an OpenClaw sandbox is active, and the requested Docker-specific opt-out is a sandbox/security product decision rather than an already-implemented cleanup.

## What This Changes

The issue requests a supported Codex-backed agent option to avoid Codex's inner sandbox when the agent is already isolated by an OpenClaw Docker sandbox.

## Best Possible Solution

Add a narrowly scoped Codex plugin opt-in for already-isolated Docker runs, preferably using Codex's external-sandbox semantics if compatible, with docs and diagnostics that preserve OpenClaw's outer sandbox and tool-policy guarantees.

## Reproduction Assessment

No high-confidence live repro was run. Source and `v2026.5.12` inspection do confirm the reported configuration is clamped from Codex `danger-full-access` to `workspace-write` under an active OpenClaw sandbox, but the `bwrap` failure itself is environment-dependent.

## Solution Assessment

Unclear until sandbox/security owners choose the trust model. A narrow Docker or external-sandbox opt-in is safer than globally letting Codex `danger-full-access` bypass the existing clamp.
