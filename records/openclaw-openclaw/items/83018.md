## Summary

Keep open: current main still sends active OpenClaw-sandboxed Codex turns an explicit Codex `workspaceWrite` sandbox policy, and the requested Docker-specific way to avoid the inner bwrap sandbox is not implemented. The latest discussion narrowed the repro to managed Codex app-server plus `sessions_spawn`, while a maintainer's stock managed-path test did not reproduce it, so this needs sandbox/security owner review rather than cleanup closure.

## What This Changes

The issue requests a supported Codex configuration option to disable or override Codex inner sandboxing when an OpenClaw Docker sandbox is already the isolation boundary.

## Best Possible Solution

Add a narrow, explicit Codex plugin opt-in for already-isolated Docker sandbox runs, preferably using a Codex external-sandbox style turn policy where appropriate, with docs, regression coverage, and live proof that OpenClaw's outer sandbox and tool policy remain authoritative.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection confirms active OpenClaw-sandboxed Codex turns still receive a Codex `workspaceWrite` policy, and the reporter supplied a fresh `sessions_spawn` repro, but the exact bwrap failure still needs live trace/proof because the stock managed path did not reproduce for a maintainer.

## Solution Assessment

Unclear until sandbox/security owners decide the trust model. The requested direction is plausible, but the safest implementation is a narrow Docker-scoped opt-in or external-sandbox policy rather than removing the existing workspace-write guard globally.
