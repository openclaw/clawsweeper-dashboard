## Summary

Keep open: current main still intentionally routes active OpenClaw-sandboxed Codex app-server turns through Codex `workspaceWrite`, and the reporter’s clarified `sessions_spawn` repro is recent, concrete, and security-sensitive enough to need owner review rather than cleanup closure.

## What This Changes

The issue asks for a supported Codex configuration path to avoid the inner Codex bwrap sandbox when an OpenClaw Docker sandbox is already intended to be the isolation boundary.

## Best Possible Solution

Resolve the child-task failure with a guardrailed Codex/OpenClaw sandbox contract, such as a true externally managed sandbox mode or running the relevant Codex execution inside the OpenClaw sandbox, rather than globally allowing host-side `danger-full-access`.

## Reproduction Assessment

No high-confidence independent reproduction is available from this review. Source inspection confirms the current OpenClaw-sandboxed Codex app-server path still uses Codex `workspaceWrite`, and the reporter provided a concrete `sessions_spawn` bwrap failure shape that matches the nested-sandbox concern.

## Solution Assessment

Unclear as requested. A supported opt-out may be appropriate only with strong guardrails, because the current managed app-server is host-side and the existing clamp was introduced to prevent workspace and Docker-socket exposure.
