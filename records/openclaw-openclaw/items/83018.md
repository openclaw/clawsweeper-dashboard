## Summary

Keep open: current main still intentionally clamps Codex app-server `danger-full-access` to `workspace-write` under an active OpenClaw sandbox, and the requested Docker/sessions_spawn opt-out is not implemented. The issue has fresh reporter detail and active assignee follow-up; the related open network-egress PR keeps Codex on `workspace-write`, so it does not settle this bwrap opt-out request.

## What This Changes

The issue requests a supported Codex configuration path to avoid nested Codex bwrap sandboxing when OpenClaw Docker sandboxing already provides the outer isolation boundary.

## Best Possible Solution

Add a narrow, documented Codex plugin policy for OpenClaw-managed Docker sandbox runs, with live proof for `sessions_spawn` child tasks and guardrails that preserve OpenClaw tool policy, workspace boundaries, binds, and Docker socket protections.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection confirms the clamp path, and the latest issue comments narrow the failing shape to managed Codex app-server plus `sessions_spawn` for a Docker-sandboxed Codex agent.

## Solution Assessment

Unclear until owners decide the trust model. The safer direction is a narrow opt-in or externally managed sandbox policy for OpenClaw Docker sandboxes, not a global removal of the current clamp.
