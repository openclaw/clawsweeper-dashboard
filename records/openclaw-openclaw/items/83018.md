## Summary

Keep open: current main and the latest shipped release still intentionally clamp Codex app-server `danger-full-access` to `workspace-write` whenever an OpenClaw sandbox is enabled, and no Docker-only opt-out exists. This is a valid-looking request, but it changes sandbox trust boundaries and needs maintainer security/product review rather than conservative cleanup.

## What This Changes

Request a supported Codex plugin configuration path to disable Codex's inner sandbox, or preserve `danger-full-access`, only when OpenClaw Docker sandboxing is already the isolation boundary.

## Best Possible Solution

Define a narrow, explicit Codex plugin opt-in for already-isolated Docker sandbox runs, with diagnostics and docs that preserve OpenClaw's outer sandbox and tool-policy guarantees.

## Reproduction Assessment

No high-confidence live reproduction was run. Source and shipped-release inspection do confirm the reported policy path: a sandboxed Codex app-server session configured as `danger-full-access` is clamped to `workspace-write`, which is the prerequisite for the reported nested bwrap failure.

## Solution Assessment

Unclear until owners decide the trust model. A narrow Docker-only opt-in with explicit guardrails is safer than removing the clamp globally, because the current docs frame the clamp as a security boundary.
