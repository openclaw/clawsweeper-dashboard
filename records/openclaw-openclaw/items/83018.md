## Summary

Keep open: current main still sends active OpenClaw sandboxed Codex turns through a Codex `workspaceWrite` sandbox policy while the managed stdio app-server is spawned from the gateway process. The newer security issue at https://github.com/openclaw/openclaw/issues/83796 broadens the containment question, and https://github.com/openclaw/openclaw/pull/83791 is only a diagnostic PR, so the requested no-inner-bwrap or sandbox-placement decision is not implemented or safely obsolete.

## What This Changes

The issue requests a supported Codex/OpenClaw configuration or policy that avoids Codex inner bwrap/workspace-write sandboxing when an OpenClaw Docker sandbox is already active.

## Best Possible Solution

Define one security contract for Codex with OpenClaw Docker sandboxing: run Codex native execution inside the active per-agent sandbox, fail closed with clear diagnostics when that cannot be honored, or add a tightly guarded opt-in that preserves OpenClaw tool policy and documents the trust boundary.

## Reproduction Assessment

Yes at source level: current main computes and sends a Codex `workspaceWrite` sandbox policy for active OpenClaw sandbox turns, and multiple reporters provided Ubuntu/AppArmor bwrap failure details. I did not run a live current-main Docker/Codex canary in this read-only review, so the host-specific bwrap failure itself remains externally reported.

## Solution Assessment

No, not as originally phrased: simply allowing `danger-full-access` while the Codex app-server remains gateway-hosted could weaken containment. The safer direction is a maintainer-approved sandbox-placement or fail-closed contract, with any opt-in narrowly guarded and documented.
