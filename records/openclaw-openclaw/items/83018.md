## Summary

Keep open: current main still forces Codex app-server turns into a workspace-write sandbox when an OpenClaw sandbox is active, and the requested Docker-specific opt-out is not implemented. The follow-up discussion narrowed the repro to managed Codex app-server plus `sessions_spawn`, but changing this path still needs maintainer security/product review rather than automatic cleanup.

## What This Changes

The issue requests a supported Codex configuration path to avoid Codex's inner bwrap/workspace-write sandbox when the agent is already isolated by OpenClaw Docker sandboxing.

## Best Possible Solution

Add an explicit, narrowly scoped Codex plugin opt-in for already-isolated Docker sandbox sessions, with tests, diagnostics, and docs that preserve OpenClaw's outer sandbox and tool-policy guarantees by default.

## Reproduction Assessment

No, we do not have a high-confidence maintainer-run live reproduction yet. Source inspection confirms the workspace-write clamp, and the reporter supplied a narrowed `sessions_spawn` repro shape, but the exact bwrap failure remains environment-dependent.

## Solution Assessment

Unclear until owners decide the trust model. A narrow Docker-only opt-in with guardrails is more maintainable than removing the current clamp globally, because the clamp is documented and tested as security policy.
