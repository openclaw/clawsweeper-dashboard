## Summary

Keep open: current main still intentionally sends Codex app-server turns through a Codex `workspaceWrite` sandbox whenever an OpenClaw sandbox is active, and the requested Docker-safe no-bwrap/outer-sandbox-only path is not implemented. The newer comments add useful Ubuntu/AppArmor evidence and a broader session-continuity impact, but this remains a sandbox policy change that needs owner security/product review before a fix lane should run.

## What This Changes

The issue requests a supported Codex app-server configuration path that avoids Codex's inner Linux sandbox when OpenClaw Docker sandboxing is already active.

## Best Possible Solution

Add an owner-approved, Docker-scoped Codex app-server sandbox policy such as Codex `externalSandbox` or an equivalent explicit opt-in, with diagnostics, docs, regression tests, and live Ubuntu proof that OpenClaw's outer sandbox and tool policy remain the trust boundary.

## Reproduction Assessment

No high-confidence live current-main repro was run here. Source inspection gives a clear reproduction path for the active-sandbox `workspaceWrite` policy, while the bwrap failure itself is supported by two Ubuntu/AppArmor user reports and remains environment-dependent.

## Solution Assessment

Unclear until owners decide the trust model. A guarded Docker-only external-sandbox/no-bwrap option or documented preflight is more maintainable than globally honoring `danger-full-access` whenever an OpenClaw sandbox is active.
