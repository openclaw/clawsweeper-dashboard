## Summary

Keep open: current main still keeps Codex app-server turns on Codex workspace-write when an OpenClaw sandbox is active, and the latest discussion narrowed the failure to managed Codex app-server plus sessions_spawn child turns. This needs maintainer/security product review rather than cleanup closure.

## What This Changes

Request a supported Codex plugin configuration to let Docker-sandboxed OpenClaw runs use OpenClaw as the sole sandbox boundary instead of forcing Codex workspace-write/bwrap on app-server turns.

## Best Possible Solution

Add a narrow, explicit Codex plugin opt-in for already-isolated Docker sandbox runs, with diagnostics, docs, tests, and live proof for both top-level and sessions_spawn child turns while preserving the current secure default.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection confirms current main still sends Codex workspaceWrite policy under active OpenClaw sandboxing, and the comments provide a narrowed repro shape for a future live check.

## Solution Assessment

Unclear until owners decide the trust model. A narrowly scoped opt-in with Docker-specific guardrails is safer than removing the current workspaceWrite policy globally.
