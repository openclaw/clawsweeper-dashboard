## Summary

Keep open: current main still forces active OpenClaw-sandboxed Codex app-server turns through a Codex `workspaceWrite` sandbox policy, and the requested Docker-specific opt-out remains a security/product decision rather than implemented cleanup.

## What This Changes

The issue requests a supported Codex configuration path to disable or bypass Codex's inner bwrap/workspace-write sandbox when an OpenClaw Docker sandbox is already the isolation boundary.

## Best Possible Solution

Add a narrowly scoped Codex plugin opt-in for already-isolated Docker-backed OpenClaw sandbox sessions, with diagnostics, tests, and docs that preserve OpenClaw's outer sandbox and tool-policy guarantees.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection confirms the current workspaceWrite policy path, and the latest reporter comment narrows the failing runtime to `sessions_spawn` child tasks, but the exact bwrap failure still needs live proof.

## Solution Assessment

Unclear until owners decide the trust model. A narrow Docker/OpenClaw-sandbox opt-in with guardrails is safer than globally removing the workspaceWrite mapping.
