## Summary

Keep open: current main and the latest shipped release still define the root config as JSON5-only, while YAML/YML support would add a second config-format contract with parser, precedence, reload, and write-policy decisions. The discussion usefully reframes part of the pain around comment-safe mutation via `config.patch`/CLI tooling, but I did not find a canonical maintainer decision or merged implementation that supersedes the YAML request.

## What This Changes

Request to support `openclaw.yaml` or `openclaw.yml` as an alternative root config file format alongside the current JSON5 `openclaw.json`.

## Best Possible Solution

Make a maintainer config-contract decision: either keep JSON5-only and strengthen CLI/config.patch editing guidance, or accept YAML with a complete discovery, parsing, validation, reload, write, docs, and test plan.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, tests, and release inspection show current OpenClaw recognizes JSON root config candidates and parses the root config as JSON5.

## Solution Assessment

Unclear. YAML would satisfy the literal request, but the discussion identifies comment-safe mutation as the sharper problem, so maintainers should decide whether the durable solution is a second root config format or stronger config.patch/CLI guidance.
