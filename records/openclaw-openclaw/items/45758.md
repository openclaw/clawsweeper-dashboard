## Summary

Keep open for maintainer product review: current main still treats `openclaw.json` as the canonical JSON/JSON5 config path, and the discussion reframes the main pain as safe config mutation rather than file syntax. This is clear enough not to close as stale or incoherent, but YAML would add a core config-format contract with precedence, parser, write, docs, and upgrade decisions.

## What This Changes

Support `openclaw.yaml` or `openclaw.yml` as an alternative config file format alongside the current JSON5 `openclaw.json`.

## Best Possible Solution

Keep the JSON5 config contract until maintainers explicitly choose a second config format; if they do, design YAML as a core config change with parser, precedence, write, docs, security, and upgrade tests rather than a drive-by parser addition.

## Reproduction Assessment

Not applicable: this is a request for a new config file format, not a report of broken existing behavior. Current-source inspection confirms the requested YAML path is not present on main.

## Solution Assessment

Unclear: YAML is a plausible ergonomics feature, but current code and discussion point to `openclaw config` and `config.patch` as the safer solution for comment-preserving programmatic edits. A maintainer should decide whether a second config format is worth the core compatibility surface.
