Keep open. The MiniMax rate-limit gap remains plausible on current main, but this PR still introduces unsafe failover classifications that should block merge until narrowed.

## What This Changes

The PR adds Chinese/MiniMax rate-limit patterns and scans short non-error assistant text for rate-limit or overload phrases to trigger fallback.

## Best Possible Solution

Land a narrow revision that adds only high-confidence MiniMax/CJK rate-limit markers with positive and negative tests, keeps successful assistant content out of failover matching, and scopes MiniMax-only code through the provider hook or omits it.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main lacks the exact MiniMax markers from the example, while the PR diff demonstrably adds broad text scanning and a generic retry phrase that can misclassify failures.

## Solution Assessment

No. The direction is useful, but this PR is not the best fix because it mixes precise MiniMax markers with unsafe generic retry/content matching instead of a narrowly tested classifier change.

- **[P1] Restrict text matching to provider error payloads:** `src/agents/pi-embedded-helpers/errors.ts:1047-1049`
  - body: The new non-error branch scans any short assistant content with broad rate-limit and overload matchers. - **[P2] Remove generic retry text from rate-limit matching:** `src/agents/pi-embedded-helpers/failover-matches.ts:21`
  - body: `请稍后重试` means "please retry later" and is commonly appended to non-rate-limit failures. Since rate-limit classification runs before timeout, billing, and auth paths, this can force unrelated Chinese provider errors into `rate_limit`.
  - confidence: 0.9

  - body: The content branch returns only `true`, while the runner still derives `assistantFailoverReason` from `errorMessage`, which is usually empty for this new non-error case. These retries can be emitted as `unknown` and skip rate-limit or overload-specific handling.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/agents/pi-embedded-helpers/failover-matches.ts:18-22`
  - body: This changes user-facing provider fallback behavior, but the PR only edits helper code. OpenClaw policy requires a `CHANGELOG.md` entry for user-facing feat/fix/perf changes with appropriate contributor credit.
