## Summary

Keep this PR open: the MiniMax-specific gap is still real because current main shipped broader CJK failover coverage but not the exact `速率限制`, `请求量较高`, or `(2062)` markers. The branch should not merge as-is because it adds broad assistant text scanning and a generic retry phrase that can misclassify normal or unrelated responses.

## What This Changes

This PR adds MiniMax/CN rate-limit patterns to failover matching and extends assistant failover detection to scan short non-error text responses.

## Best Possible Solution

Land a narrow revision or replacement that keeps high-confidence MiniMax rate-limit indicators, avoids standalone generic retry text and broad assistant-content scanning, and adds focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main does not match the exact MiniMax sample markers, and the PR diff clearly introduces content-based matching that can be reasoned about without a live MiniMax account.

## Solution Assessment

No. The safer fix is narrower: add precise MiniMax error-message indicators to the existing classifier, remove `请稍后重试` as a standalone rate-limit signal, and avoid treating ordinary non-error assistant prose as provider failure payloads.

  - confidence: 0.84
