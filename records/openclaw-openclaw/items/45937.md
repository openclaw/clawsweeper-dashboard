## Summary

Keep open. The MiniMax/CN rate-limit problem is real and current main still lacks the exact MiniMax markers, but this PR is not mergeable as written because it adds a generic Chinese retry phrase and broadens failover detection to ordinary short assistant text.

## What This Changes

The PR adds Chinese MiniMax rate-limit markers to shared failover matching and changes assistant failover detection to scan short non-error assistant text for rate-limit or overload phrases.

## Best Possible Solution

Keep the useful MiniMax signal but narrow it to precise provider-error evidence: add 速率限制, 请求量较高, and a scoped MiniMax 2062 matcher with focused positive and negative tests. Do not add standalone 请稍后重试 as a rate-limit marker, and only handle text responses through a known provider error payload path or provider-owned classification hook.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: the current matcher misses the MiniMax-specific payload, while the PR would classify generic Chinese retry text and normal short assistant text through shared failover patterns.

## Solution Assessment

No. The requested direction is valid, but this PR is not the best solution because it combines precise MiniMax markers with a generic retry phrase and scans non-error assistant output without carrying the matched reason; a narrower matcher plus regression tests is safer.

  - confidence: 0.9

  - confidence: 0.84
