## Summary

Keep open. Current main still emits embedded `llm_input` before prompt submission and `llm_output` once after the attempt, while this PR targets a real per-call observability gap tracked by #64268 but has API-contract and correctness blockers before merge.

## What This Changes

The PR moves embedded Pi `llm_input`/`llm_output` plugin hooks from once-per-run attempt emissions to per-assistant-round `message_start`/`message_end` emissions, adds subscription hook helpers/params, updates comments, and adds focused coverage.

## Best Possible Solution

Define the per-LLM-call observability contract alongside #64268, then either preserve the existing run-level hooks and add a per-call hook or deliberately migrate the hooks with prompt-submission input emission, normalized per-call usage, docs, changelog, and focused tests.

## Reproduction Assessment

Yes. Source inspection of the PR path gives a high-confidence reproduction path: fail `activeSession.prompt` before any assistant `message_start` and `llm_input` will not fire; pass usage shaped like `{ totalTokens: 42 }` through the new output path and `usage.total` is not normalized.

## Solution Assessment

No. Per-call observability is a useful feature, but this patch is not the narrowest maintainable solution until the public hook contract is decided and the implementation preserves existing input telemetry and usage normalization guarantees.

## Review Findings

Full review comments:

- **[P2] Emit input hooks before prompt submission:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:74`
  - body: Moving `llm_input` to assistant `message_start` drops telemetry when `activeSession.prompt(...)` fails before the first stream event, such as auth, rate-limit, or network failures. Current main emits before prompt submission, so tracing plugins would lose exactly the failed round they need to diagnose.
  - confidence: 0.92
- **[P2] Normalize assistant usage before emitting output:** `src/agents/pi-embedded-subscribe.ts:407-418`
  - body: `emitLlmOutputHook` forwards `message.usage` through a narrow cast instead of `normalizeUsage`. Provider aliases such as `totalTokens`, `total_tokens`, `prompt_tokens`, and cache-token fields can be dropped, so per-round hook consumers may undercount or miss usage.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/plugins/types.ts:594`
  - body: This changes public plugin hook frequency and payload semantics but leaves `CHANGELOG.md` untouched. Plugin authors need release-note migration guidance if `llm_input`/`llm_output` stop behaving like once-per-run payloads.
  - confidence: 0.86
