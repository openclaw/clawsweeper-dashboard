## Summary

Keep open for maintainer review. Current main still does not provide raw per-LLM-call `llm_output`, but this PR changes public plugin hook semantics with blocking input-boundary and usage-normalization regressions, and it is stale against the current hook type split while related issue #64268 remains open.

## What This Changes

The PR moves embedded `llm_input`/`llm_output` emission from attempt start/end into subscription `message_start`/`message_end`, threads hook context into subscription params, updates hook comments, and adds per-round hook tests.

## Best Possible Solution

Define the per-LLM-call observability contract alongside #64268, keep input telemetry at the prompt-submission boundary, normalize per-call usage through the existing helper, and update hook types, docs, changelog, and focused tests together.

## Reproduction Assessment

Yes. The blocking issues reproduce by inspection and focused test design: make `activeSession.prompt(...)` fail before any assistant `message_start`, and feed assistant usage shaped like `{ totalTokens: 42 }` through the proposed per-round output path.

## Solution Assessment

No. The feature direction is useful, but moving `llm_input` to stream start and bypassing normalized usage are not the narrowest maintainable way to expose per-call hook data.
