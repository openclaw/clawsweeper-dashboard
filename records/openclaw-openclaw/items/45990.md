## Summary

Keep open: the PR targets a real plugin observability gap, but the branch as written should not merge because it replaces aggregate hook semantics, loses pre-stream failure telemetry and current hook metadata/usage normalization, and lacks inspectable real behavior proof.

## What This Changes

The PR moves `llm_input` and `llm_output` plugin hook emission from embedded attempt boundaries into per-assistant-message stream handlers so hooks fire once per LLM round.

## Best Possible Solution

Ship an additive, documented per-call observability contract that preserves aggregate `llm_output` compatibility, current metadata, normalized usage, pre-stream failure telemetry, and raw conversation access gating.

## Reproduction Assessment

Yes. Source inspection on current main shows each assistant `message_end` records and commits usage, while raw `llm_output` still dispatches once later from `runEmbeddedAttempt` with aggregate `getUsageTotals()` data.

## Solution Assessment

No. The PR targets a real observability gap, but replacing the existing aggregate hook semantics with stream-event payloads is not the narrowest maintainable solution until compatibility, metadata, normalized usage, and failure telemetry are preserved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Emit llm_input before prompt submission:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:74`
  - body: Moving `llm_input` emission to `message_start` means rounds that fail before the first stream event, such as auth, rate-limit, network, timeout, or prompt precheck failures during prompt submission, no longer emit input telemetry. Current main emits before prompt execution, so tracing plugins would lose the failing round.
  - confidence: 0.9
- **[P2] Keep aggregate llm_output distinguishable:** `src/agents/pi-embedded-subscribe.ts:405-418`
  - body: The patch reuses `llm_output` for per-round payloads and removes the final aggregate event without a round index, call id, aggregate marker, or separate hook. Existing plugins cannot tell whether `assistantTexts` and `usage` describe one call or the whole turn, and aggregate consumers lose their current event.
  - confidence: 0.88
- **[P2] Normalize assistant usage before emitting llm_output:** `src/agents/pi-embedded-subscribe.ts:407-418`
  - body: The new per-round output path forwards `message.usage` through a narrow cast instead of the existing normalization path. Provider shapes using fields such as `totalTokens`, `prompt_tokens`, `completion_tokens`, cache-token fields, or timing fields can be dropped, so plugins may undercount or miss usage.
  - confidence: 0.88
- **[P2] Preserve current llm_output metadata:** `src/agents/pi-embedded-subscribe.ts:400-418`
  - body: Current main's `llm_output` contract includes context-window metadata, `resolvedRef`, `harnessId`, and prompt data. This subscription-level emitter sends only the older core fields, so rebasing this PR as-is would silently drop metadata for plugins.
  - confidence: 0.84
