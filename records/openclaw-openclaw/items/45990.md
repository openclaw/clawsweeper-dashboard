## Summary

Keep open: this PR targets a real plugin-observability gap tracked by https://github.com/openclaw/openclaw/issues/64268, but the patch currently regresses pre-stream failure observability, bypasses existing usage normalization, changes public hook semantics without a compatibility decision, and lacks inspectable real behavior proof.

## What This Changes

The PR moves `llm_input` and `llm_output` hook emission from the embedded attempt boundary into assistant `message_start` and `message_end` handling, with subscription parameters and tests for per-round hook calls.

## Best Possible Solution

Define an additive or clearly marked per-call hook contract that preserves aggregate/failure observability, normalizes usage through the existing helper, and documents the plugin migration path before merging.

## Reproduction Assessment

Yes, for the current limitation: source inspection shows main emits one `llm_output` at the attempt boundary using accumulated usage, while each assistant `message_end` only records and commits usage. I did not run a live multi-round agent in this read-only review.

## Solution Assessment

No. Moving hooks solely into `message_start` and `message_end` is close to the desired behavior, but the patch should preserve pre-stream failure input telemetry, use existing usage normalization, and settle the aggregate-versus-per-call API contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Emit llm_input before pre-stream failures:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:71`
  - body: Moving `llm_input` emission to `message_start` means a prompt submission that fails before the provider produces a stream event, such as auth, rate-limit, network, or timeout-before-first-byte failures, never emits input telemetry. Current main emits before `activeSession.prompt(...)`, so observability plugins would lose the trace for exactly the failing calls they need to diagnose.
  - confidence: 0.9
- **[P2] Normalize usage before emitting per-round output:** `src/agents/pi-embedded-subscribe.ts:407-418`
  - body: The new `llm_output` path forwards `message.usage` through a narrow cast instead of the existing normalization path. Provider shapes such as `totalTokens`, `prompt_tokens`, `completion_tokens`, cache fields, or `timings.prompt_n` will be dropped from per-round hook payloads, so plugins can undercount or miss usage data.
  - confidence: 0.88
- **[P2] Keep aggregate llm_output distinguishable:** `src/agents/pi-embedded-subscribe.ts:399-418`
  - body: This reuses the existing `llm_output` hook name for per-round events and removes the end-of-run aggregate event without adding a round index, aggregate marker, or separate hook. Existing plugins that treat `assistantTexts` and `usage` as the full-run payload cannot tell whether they are seeing one call or the final aggregate.
  - confidence: 0.78
