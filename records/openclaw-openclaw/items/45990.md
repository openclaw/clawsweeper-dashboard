## Summary

Keep open. The per-round observability goal is useful and overlaps #64268, but this PR changes public plugin hook semantics and still has blocking correctness/API issues around prompt-failure input telemetry, usage normalization, and release-note coverage.

## What This Changes

The PR moves embedded Pi `llm_input`/`llm_output` hooks from attempt-level once-per-run emission into assistant `message_start`/`message_end` handling, adds subscription hook context params, and updates hook comments/tests.

## Best Possible Solution

Define the per-LLM-call observability contract with #64268, preserve prompt-failure visibility, normalize per-call usage through the existing helper, and update hook types, docs, changelog, and focused regression tests together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make `activeSession.prompt(...)` fail before any assistant `message_start` and this PR will not emit `llm_input`; pass provider usage shaped like `{ totalTokens: 42 }` through the new output path and normalized totals can be lost.

## Solution Assessment

No. Per-call observability is reasonable, but this implementation is not the best path because it changes an existing public hook contract while moving input telemetry too late and bypassing the established usage-normalization contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Emit input hooks before prompt submission:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:74`
  - body: Moving `llm_input` to assistant `message_start` drops telemetry when `activeSession.prompt(...)` fails before the first stream event, such as auth, rate-limit, or network failures. Current main emits before prompt submission, so tracing plugins would lose the failed round they need to diagnose.
  - confidence: 0.92
- **[P2] Normalize assistant usage before emitting output:** `src/agents/pi-embedded-subscribe.ts:407-418`
  - body: `emitLlmOutputHook` forwards `message.usage` through a narrow cast instead of `normalizeUsage`. Provider aliases such as `totalTokens`, `total_tokens`, `prompt_tokens`, and cache-token fields can be dropped, so per-round hook consumers may undercount or miss usage.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/plugins/types.ts:594`
  - body: This changes public plugin hook frequency and payload semantics but leaves `CHANGELOG.md` untouched. Plugin authors need release-note migration guidance when `llm_input`/`llm_output` stop behaving like once-per-run payloads.
  - confidence: 0.86
