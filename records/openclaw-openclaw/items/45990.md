## Summary

Keep open: the requested per-call observability gap is real and still tracked by https://github.com/openclaw/openclaw/issues/64268, but this PR needs maintainer API-contract review, compatibility fixes, and contributor real behavior proof before merge. Current main still emits aggregate `llm_output` once, while the branch would change existing hook semantics and drop current metadata/normalization guarantees.

## What This Changes

The PR moves raw `llm_input`/`llm_output` hook dispatch from embedded attempt boundaries into `message_start`/`message_end` stream handling and updates comments/tests for per-round hook semantics.

## Best Possible Solution

Ship an additive, documented per-call observability contract that preserves aggregate `llm_output` compatibility, pre-stream input telemetry, normalized usage, current metadata, and raw conversation access gating.

## Reproduction Assessment

Yes. Source inspection on current main shows per-assistant-message usage is recorded at `message_end`, while raw `llm_output` still dispatches once later with aggregate `getUsageTotals()` data; I did not run a live multi-round agent in this read-only review.

## Solution Assessment

No. The branch targets the right observability gap, but replacing aggregate hook semantics solely with stream-event payloads is not the narrowest maintainable solution until compatibility, metadata, normalized usage, and pre-stream failure telemetry are preserved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Emit llm_input before prompt submission:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:74`
  - body: Moving `llm_input` emission to `message_start` means rounds that fail before the first stream event, such as auth, rate-limit, network, timeout, or prompt precheck failures during prompt submission, no longer emit input telemetry. Current main emits before prompt execution, so tracing plugins would lose the failing round.
  - confidence: 0.9
- **[P2] Normalize usage before per-round output:** `src/agents/pi-embedded-subscribe.ts:407-418`
  - body: The new per-round output path forwards `message.usage` through a narrow cast instead of the existing `normalizeUsage` path. Provider shapes using `totalTokens`, `prompt_tokens`, `completion_tokens`, cache-token fields, or timing fields can be dropped, so plugins may undercount or miss usage.
  - confidence: 0.88
- **[P2] Keep aggregate llm_output distinguishable:** `src/agents/pi-embedded-subscribe.ts:405-418`
  - body: The patch reuses `llm_output` for per-round payloads and removes the final aggregate event without a round index, aggregate marker, or separate hook. Existing plugins cannot tell whether `assistantTexts` and `usage` describe one call or the whole turn, and aggregate consumers lose their current event.
  - confidence: 0.86
- **[P2] Preserve current llm_output metadata:** `src/agents/pi-embedded-subscribe.ts:400-406`
  - body: Current main's `llm_output` contract includes context-window metadata, `resolvedRef`, `harnessId`, and `prompt`. This subscription-level emitter sends only the older core fields, so rebasing this PR as-is would silently drop metadata for plugins.
  - confidence: 0.84
