## Summary

Keep open. The docs gap is real and current main supports the referenced gateway lifecycle and LaunchAgent recovery behavior, but this PR is not merge-ready because the new flow restarts the gateway before checking the case where the gateway is already healthy and only browser state is stale.

## What This Changes

The PR adds a local dashboard after-update troubleshooting section to `docs/gateway/troubleshooting.md` with LaunchAgent recovery, `lsof`/`curl` checks, restart/start steps, root dashboard guidance, and stale browser-state notes.

## Best Possible Solution

Ship the docs-only runbook after making it diagnostic-first: check the listener and root dashboard response, send healthy-gateway users to the root URL and stale-browser path, and reserve restart/start for failed listener or HTTP checks.

## Reproduction Assessment

Not applicable as a docs enhancement. The docs gap is verifiable from current-main docs search, and the PR-specific problem is visible in the proposed command order.

## Solution Assessment

No, not as written. A docs-only troubleshooting update is the right boundary, but the safer flow is to test `lsof`/`curl` first and restart only if the gateway is not listening or not serving the dashboard.
