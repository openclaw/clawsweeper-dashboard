## Summary

Keep this draft open, but it is not merge-ready. The added Slack tests assert unseeded top-level room messages get per-message thread sessions, which conflicts with current routing and guard tests that deliberately keep ordinary top-level room messages on the per-channel session while only real replies or seeded mentions get thread-scoped linkage.

## What This Changes

The PR adds a Slack Vitest file covering session keys and thread context for roomish channel messages when `replyToMode` is `all`.

## Best Possible Solution

Clarify the desired Slack room session semantics; if the current contract stands, replace this with tests for seeded app mentions, real thread replies, and Slack reply metadata while preserving per-channel top-level sessions.

## Reproduction Assessment

Yes. Static inspection of `resolveSlackRoutingContext` with the proposed helper inputs returns the per-channel key for unseeded top-level room messages, and current main already has a guard test for that invariant; I did not run Vitest because this review had to keep the checkout read-only.

## Solution Assessment

No. A test-only patch that encodes the old per-message top-level room session behavior is not the narrowest maintainable fix; a behavior change would need explicit Slack routing design and migration reasoning first.
