## Summary

## What This Changes

Restore gateway startup after upgrade by making the bundled `speech-core/runtime-api.js` public surface resolvable from the published OpenClaw package.

## Best Possible Solution

Stage or bundle the canonical speech runtime artifact into a published location already owned by the packaged plugin/runtime contract, keep the resolver on one canonical path, add generated-fixture resolver coverage, and prove gateway startup after a real npm upgrade.

## Reproduction Assessment

Yes, from source and package artifacts with high confidence: the speech runtime moved to package-local dist, the resolver cannot reach it, and the published tarballs omit that directory. This review did not execute the reporter’s WSL2 service restart, but the exact failure path is independently reproducible from the shipped layout.

## Solution Assessment

No, the currently linked PR is not yet the best demonstrated fix because it adds a fallback to a directory absent from the npm tarball. The maintainable repair is to establish one canonical published runtime location and validate the actual installed gateway path.
