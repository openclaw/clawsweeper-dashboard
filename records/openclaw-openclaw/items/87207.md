## Summary

Keep open. Current main and the latest release still expose `agent_end` as messages plus success/error/duration only; the requested typed per-turn tool outcomes, tool durations, and normalized completion reason remain a public plugin SDK design gap rather than an implemented behavior.

## What This Changes

The issue requests an additive `agent_end` payload or new `turn_end` hook that exposes per-turn tool-call outcomes, per-tool durations, completion reason, and raw provider finish reason for plugin observability.

## Best Possible Solution

Define a small additive turn-outcome contract for plugins, backed by Pi/core and Codex runtime data, with SDK types, focused hook contract tests, and docs steering plugins away from parsing provider-shaped message blocks.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for the current-state check: source inspection shows `agent_end` still lacks the requested aggregate tool outcome and completion-reason fields on current main and in `v2026.6.6`.

## Solution Assessment

Unclear until maintainer API design happens. Additive `agent_end` fields preserve existing subscribers, while a separate `turn_end` hook may be cleaner if maintainers want a new typed turn-outcome contract.
