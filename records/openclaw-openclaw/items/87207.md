## Summary

Keep open. Current main still exposes `agent_end` as a narrow messages/success/error/duration hook, and adjacent `turn_end` or tool-hook surfaces do not provide the requested public plugin SDK aggregate with per-turn tool outcomes, per-tool durations, and completion reason.

## What This Changes

The issue requests an additive `agent_end` payload or new `turn_end` plugin hook exposing per-turn tool-call outcomes, per-tool durations, and completion reason.

## Best Possible Solution

Keep this issue open as the canonical request and design a small additive plugin SDK turn-outcome contract with privacy rules, runner/provider normalization, types, focused tests, and docs.

## Reproduction Assessment

Yes. Source inspection on current main shows the public `agent_end` type and embedded/CLI dispatch paths omit the requested tool outcome and completion-reason fields; I did not run a live plugin repro because this read-only review already proves the current payload shape.

## Solution Assessment

Unclear until maintainer API design happens. Enriching `agent_end` is backward-compatible, while a new public plugin `turn_end` hook may better match the existing internal turn boundary; either path needs privacy and provider-normalization decisions.
