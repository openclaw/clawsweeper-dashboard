## Summary

Keep open. Current main still has a source-backed macOS Voice Wake gap for text-only or zero-timing interim SpeechRecognition updates: the segment gate cannot accept them, the immediate text-only fallback is final-only, and the delayed paths can miss trigger-to-Talk Mode activation while interim text keeps changing.

## What This Changes

Make macOS Voice Wake trigger Talk Mode or wake capture when Apple SpeechRecognition produces trigger text without usable word timings.

## Best Possible Solution

Add a bounded macOS Voice Wake fallback for segmentless or zero-timing interim text, preferably using newly appended transcript text, while preserving word-boundary, filler-prefix, and ordinary-mention rejection behavior.

## Reproduction Assessment

Yes, source-reproducible: a non-final recognition update with trigger text and no usable segment timing leaves `WakeWordGate.match` nil and skips the immediate text-only fallback. I did not live-run Apple SpeechRecognition in this read-only review.

## Solution Assessment

No, current main is not the best solution because it depends on final results or delayed silence for text-only fallback. A narrow delta-aware fallback in the macOS runtime is safer than loosening the shared gate into broad substring matching.

## Repair Work Prompt

Keep the change in the macOS Voice Wake runtime/helper surface unless a small shared SwabbleKit helper is clearly needed. Prefer matching newly appended interim transcript text so trigger-only Talk Mode and trigger-plus-command capture can start before finalization, but preserve existing false-positive protections for ordinary mentions, suffix matches like `computers`, and non-filler prefixes. Add focused regression coverage for no-segment or zero-timing interim trigger text, appended interim wake phrases, trigger-only Talk Mode activation with `voiceWakeTriggersTalkMode=true`, and rejection of ordinary mentions.
