## Summary

Keep this PR open: the warning-status direction matches the linked cron delivery issue and the after-fix proof is sufficient, but the latest head still has a blocking QA wait predicate regression.

## What This Changes

The branch adds a first-class cron `warning` status for delivery-only failures across cron persistence, gateway schema, CLI/UI filters, plugin hooks, docs, changelog, i18n metadata, QA helpers, and focused tests, plus a small OpenAI stream type cleanup.

## Best Possible Solution

Land the additive warning-status model after repairing the QA wait predicate and completing maintainer acceptance of the public cron status contract.

## Reproduction Assessment

Yes. Current main is source-reproducible: successful isolated cron execution can flow into delivery-target or direct-send failure paths that return `status: "error"`; the PR body also supplies after-fix live output for the changed behavior.

## Solution Assessment

No, not merge-ready yet. The additive `warning` status is a maintainable direction, but the QA wait helper must keep the `afterTs` guard around every terminal status before this patch is correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the timestamp guard for terminal QA runs:** `extensions/qa-lab/src/cron-run-wait.ts:45-48`
  - body: This predicate now applies `entry.ts >= params.afterTs` only to `ok` runs because `&&` binds more tightly than `||`. Any older `error`, `skipped`, or `warning` entry for the same job can satisfy the wait before the newly triggered run finishes, making qa-lab scenarios pass or fail against stale run-log data. Keep the timestamp check outside a parenthesized terminal-status test.
