## Summary

Keep open. The linked issue is valid and the warning-status direction is reasonable, but this PR introduces a QA wait regression and still lacks required after-fix real behavior proof.

## What This Changes

The PR adds a cron `warning` status for successful isolated cron executions whose final delivery fails, updating scheduler handling, protocol validation, docs, CLI, QA helpers, and Control UI filters.

## Best Possible Solution

Keep the warning-status implementation path, fix the QA wait predicate, add real OpenClaw proof showing a successful cron run recorded as warning with delivery diagnostics, then rerun the relevant CI gates before merge.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: an isolated `agentTurn` cron job with announce delivery and no resolvable target reaches delivery code that records `status: "error"`, and scheduler error handling then treats it as execution failure state.

## Solution Assessment

No, not as currently written. The warning-status model is a maintainable direction for the linked issue, but the PR needs the QA timestamp regression fixed and real behavior proof before merge.
