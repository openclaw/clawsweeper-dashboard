## Summary

This PR should stay open because it addresses a real open cron status gap and now includes adequate terminal proof, but the current head still has a QA wait-helper regression that can observe stale terminal runs.

## What This Changes

The PR adds a first-class cron `warning` status for successful isolated runs with failed final delivery and threads it through cron persistence, gateway schema, CLI/UI filters, QA helpers, docs, changelog, and tests.

## Best Possible Solution

Keep the additive warning-status design, fix the QA wait predicate so `afterTs` applies to every terminal status, and then review or land this PR as the implementation for https://github.com/openclaw/openclaw/issues/49190.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main lacks `warning`, and delivery-target or direct-send failures in `dispatchCronDelivery` can still return `status: "error"` after successful isolated execution; I did not run a live cron job in this read-only review.

## Solution Assessment

No, not merge-ready yet. The warning-status model is the right maintainable direction and the proof is now sufficient, but the QA wait predicate must keep the timestamp guard around all terminal statuses before this patch is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep timestamp guard around every terminal status:** `extensions/qa-lab/src/cron-run-wait.ts:45-48`
  - body: This predicate now applies `entry.ts >= params.afterTs` only to `ok` runs because `&&` binds more tightly than `||`. Any older `error`, `skipped`, or `warning` entry for the same job can satisfy the wait before the newly triggered run finishes, making qa-lab scenarios pass or fail against stale run-log data. Keep the timestamp check outside a parenthesized terminal-status test.
  - confidence: 0.94
