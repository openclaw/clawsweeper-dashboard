## Summary

Keep this PR open. The warning-status direction matches the linked cron delivery problem and the PR body supplies usable after-fix live output, but the latest head still has a P2 QA wait predicate regression that can make scenarios complete against stale terminal run-log rows.

## What This Changes

The branch adds a cron `warning` status for delivery-only failures across cron dispatch, persistence, gateway schema, CLI/UI/docs/tests, generated UI locale fallback metadata, and a small OpenAI stream type cleanup.

## Best Possible Solution

Repair the QA wait predicate and regression test, then land the additive warning-status model if maintainers accept the public cron status contract.

## Reproduction Assessment

Yes. Current main is source-reproducible: successful isolated cron execution can flow into delivery-target or direct-send failure paths that return `status: "error"`, and the PR body supplies after-fix live output for the changed behavior.

## Solution Assessment

No, not merge-ready yet. The additive `warning` status is a maintainable direction, but the QA wait helper must keep the `afterTs` guard around every terminal status before this patch is correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the timestamp guard for all terminal statuses:** `extensions/qa-lab/src/cron-run-wait.ts:45-48`
  - body: This predicate now applies `entry.ts >= params.afterTs` only to `ok` runs because `&&` binds more tightly than `||`. Any older `error`, `skipped`, or `warning` row for the same job can satisfy the wait before the newly triggered run finishes, so qa-lab scenarios can pass or fail against stale run-log data. Keep the timestamp check outside a parenthesized terminal-status test.
