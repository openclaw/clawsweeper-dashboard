## Summary

Keep open: the warning-status direction matches the linked cron delivery problem, but this PR introduces a QA wait-helper regression and lacks required after-fix real behavior proof, so it is not merge-ready.

## What This Changes

The PR adds a `warning` cron run status for successful isolated cron executions whose final delivery fails, updating cron persistence, gateway schema, run logs, CLI/UI filters, QA helpers, docs, changelog, and tests.

## Best Possible Solution

Keep the additive `warning` status approach, but preserve timestamp filtering in QA waits and require redacted live terminal/log/screenshot proof of an isolated cron delivery failure being recorded as `warning` before merge.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can reach `dispatchCronDelivery` with a successful isolated agent result and an unresolved announce target, then persist `status: "error"` from the delivery-target failure path. I did not run a live cron job in this read-only review.

## Solution Assessment

No, not yet. The warning-status model is the right maintainable direction, but this patch must keep `afterTs` guarding every terminal QA status and include real runtime proof before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the timestamp guard for all terminal statuses:** `extensions/qa-lab/src/cron-run-wait.ts:46-48`
  - body: This predicate now applies `entry.ts >= params.afterTs` only to `ok` runs because `&&` binds more tightly than `||`. Any older `error`, `skipped`, or `warning` entry for the same job can satisfy the wait before the newly triggered run finishes, making qa-lab scenarios pass or fail against stale run-log data. Keep the timestamp check outside a parenthesized terminal-status test.
