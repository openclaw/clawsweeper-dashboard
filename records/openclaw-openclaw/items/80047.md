## Summary

Keep open: the warning-status direction matches the linked cron delivery issue and the PR has sufficient after-fix proof, but the latest head still introduces a QA wait regression where stale non-ok run-log entries can satisfy the completion predicate.

## What This Changes

The PR adds a cron `warning` status for delivery-only failures across cron persistence, gateway schema, CLI/UI filters, plugin hooks, docs, i18n, QA wait logic, and targeted tests, plus small OpenAI stream typing and run-log append changes.

## Best Possible Solution

Land the additive warning-status model after repairing the QA terminal-status predicate and confirming maintainer acceptance of the public cron status contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Current main is source-reproducible through the delivery-target/direct-send failure paths returning `error`, and the PR body supplies after-fix live output from the real cron service path.

## Solution Assessment

Is this the best way to solve the issue? No, not merge-ready yet. The additive `warning` status is a maintainable direction, but the QA wait helper must keep the `afterTs` guard around every terminal status before this patch is correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the timestamp guard for terminal QA runs:** `extensions/qa-lab/src/cron-run-wait.ts:45-48`
  - body: This predicate now applies `entry.ts >= params.afterTs` only to `ok` runs because `&&` binds more tightly than `||`. Any older `error`, `skipped`, or `warning` entry for the same job can satisfy the wait before the newly triggered run finishes, making qa-lab scenarios pass or fail against stale run-log data. Keep the timestamp check outside a parenthesized terminal-status test.
  - confidence: 0.94
