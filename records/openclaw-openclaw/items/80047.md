## Summary

Keep open: the warning-status approach matches the linked cron delivery issue and the PR now supplies real runtime output, but the patch still lets stale non-ok QA run-log entries satisfy waits before the newly triggered run finishes.

## What This Changes

The PR adds a first-class cron `warning` run status for successful isolated runs whose final delivery fails, wiring persistence, gateway schema, CLI/UI filters, plugin hooks, docs, changelog, and tests.

## Best Possible Solution

Land the additive warning-status model after repairing the QA wait stale-entry handling and keeping focused cron, gateway, UI, and qa-lab coverage aligned.

## Reproduction Assessment

Yes. Current main is source-reproducible: successful isolated cron execution can flow into delivery-target or direct-send failure paths that return `status: "error"`; I did not run a live cron job in this read-only review.

## Solution Assessment

No, not merge-ready yet. The additive `warning` status is the right maintainable direction, but the QA wait helper must keep the `afterTs` guard around every terminal status before this patch is correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the timestamp guard for terminal QA runs:** `extensions/qa-lab/src/cron-run-wait.ts:45-48`
  - body: This predicate now applies `entry.ts >= params.afterTs` only to `ok` runs because `&&` binds more tightly than `||`. Any older `error`, `skipped`, or `warning` entry for the same job can satisfy the wait before the newly triggered run finishes, making qa-lab scenarios pass or fail against stale run-log data. Keep the timestamp check outside a parenthesized terminal-status test.
  - confidence: 0.94
