## Summary

Keep open: the PR is a plausible implementation of the linked cron warning-status request and has sufficient real behavior proof, but the latest head still introduces two source-level regressions that should block merge.

## What This Changes

The PR adds a first-class cron `warning` status across cron delivery, scheduler persistence, gateway schema, CLI/UI filters, plugin hook types, docs, changelog, tests, and includes a small OpenAI stream typing fix.

## Best Possible Solution

Land the additive warning-status model after preserving fatal message-presentation semantics, restoring the QA wait timestamp guard, rebasing the head, and keeping focused cron, protocol, UI, and qa-lab coverage green.

## Reproduction Assessment

Yes. Current main is source-reproducible: successful isolated cron execution can reach delivery-target or direct-send failure paths that return `status: "error"`, while the status contract has no `warning` value.

## Solution Assessment

No, not merge-ready yet. The additive `warning` status is a maintainable direction for the linked request, but this patch must preserve existing fatal presentation-warning behavior and fix the QA stale-entry predicate before it is the best solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve fatal presentation warnings on failed delivery:** `src/cron/isolated-agent/run.ts:998`
  - body: Passing `deliveryFailureStatus: hasFatalErrorPayload ? "error" : "warning"` here happens before `failPendingPresentationWarningUnlessDelivered` can promote a pending message-tool presentation warning to fatal. If the agent emits a final report plus a presentation warning and cron delivery then fails, the new result remains `warning`, so a real undelivered presentation failure is persisted as partial success. Recompute or override this status after the pending-warning check and add coverage for that delivery-failure case.
  - confidence: 0.86
- **[P2] Preserve the timestamp guard for terminal QA runs:** `extensions/qa-lab/src/cron-run-wait.ts:45-48`
  - body: Only `ok` entries are protected by `entry.ts >= params.afterTs`; `&&` binds tighter than `||`, so any older `error`, `skipped`, or `warning` run can satisfy the wait immediately. Parenthesize the terminal-status test so all terminal statuses must be newer than `afterTs`, and extend the existing test to cover stale non-ok entries.
  - confidence: 0.95
