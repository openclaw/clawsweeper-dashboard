## Summary

Keep this PR open: the provenance direction matches the linked systemd follow-up, but the patch has a blocking correctness issue in the `.suggested` unit path and the real-behavior proof does not cover that path.

## What This Changes

The PR adds Linux systemd gateway-unit provenance, uses it during update-mode doctor repairs, writes `.suggested` units for legacy or modified units, and documents the behavior.

## Best Possible Solution

Keep the provenance approach, but make `.suggested` output self-contained or write a matching suggested env artifact and apply instructions before merge.

## Reproduction Assessment

Yes for the review finding: the missing-env-file path is source-reproducible from `writeSuggestedSystemdUnit` and the existing systemd env-file contract. The PR's intended provenance behavior also has partial real proof in the PR body.

## Solution Assessment

No: the overall provenance design is a good fit for the linked issue, but the proposed `.suggested` unit implementation is not yet complete for env-file-backed service values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep suggested units self-contained with their env source:** `src/daemon/systemd.ts:822-826`
  - body: `writeSuggestedSystemdUnit` still lets `buildSystemdUnitForInstall` move state-dir `.env` values into `EnvironmentFile=-.../gateway.systemd.env`, but this path passes `writeEnvironmentFile: false` and only writes the `.suggested` unit. The printed apply command copies just that unit, so a legacy/operator-modified install without the generated env file can adopt the suggestion and start without those variables. Either keep those values inline for suggestions or write a matching suggested env artifact and include it in the apply flow.
  - confidence: 0.9
