## Summary

Keep this PR open. The generated-unit provenance direction matches the prior systemd update discussion, but the suggested-unit path can render an `EnvironmentFile` that it never writes, so the printed apply command is unsafe for legacy units with `.env`-backed gateway or provider variables.

## What This Changes

The branch records sha256 provenance for generated Linux systemd gateway units, uses it in update-mode doctor to auto-stage unchanged units, writes `.suggested` units for modified or legacy units, and updates docs, tests, and changelog.

## Best Possible Solution

Keep the provenance tracking approach, but make `.suggested` output self-contained or paired with env-file review/apply commands so applying it cannot silently drop existing state-dir or env-file values.

## Reproduction Assessment

Yes for the PR-introduced blocker: source inspection of the PR head shows suggested units can reference `gateway.systemd.env` while the suggested path does not write that file. A legacy or no-provenance unit with state-dir `.env` values and no existing generated env file would lose those values after the printed apply command.

## Solution Assessment

No. The provenance design is a maintainable direction, but the suggested-unit implementation needs to install or preserve the environment source it tells operators to apply.
