## What This Changes

The PR adds a per-request `Accept: text/event-stream` header for `openai-chatgpt-responses` streaming OpenAI SDK requests and adds regression coverage for the ChatGPT stream path plus negative OpenAI/default paths.

## Best Possible Solution

Keep the fix scoped to the ChatGPT Responses stream request options, then merge after CI and a redacted live ChatGPT-OAuth probe or maintainer acceptance of the live-proof gap.

## Reproduction Assessment

Yes at source level: the linked report gives a concrete ChatGPT-OAuth probe path and current/shipped source shows the SDK stream call lacks the SSE Accept override while the guard rejects missing content type. I did not run a live OAuth probe in this read-only review.

## Solution Assessment

Yes, this appears to be the narrowest maintainable fix: per-request headers override the SDK's default `Accept: application/json` only for ChatGPT Responses streams. Broader alternatives like changing default client headers or all OpenAI Responses streams would affect non-streaming or normal OpenAI API calls unnecessarily.
