## Summary

Keep open. Current main still auto-detects Gemini CLI for audio, and this PR is a useful narrow fix, but the branch needs a refresh because its docs and regression test are stale against the current media auto-detect order and it lacks the required changelog entry.

## What This Changes

This PR skips Gemini CLI auto-detection for audio media understanding while preserving Gemini CLI auto-detect for image/video, and adds docs plus regression coverage for the audio exception.

## Best Possible Solution

Land a refreshed equivalent of the audio-only guard so automatic audio transcription never selects Gemini CLI, while explicit CLI configuration and image/video Gemini auto-detect remain available and docs, tests, and changelog match current main.

## Reproduction Assessment

Yes. Current main gives a source-level reproduction path: with no active/provider/local audio backend and a working `gemini` binary, audio auto-detect can reach `resolveGeminiCliEntry`; the linked #46588 report also includes real Telegram `.ogg` headless failure examples.

## Solution Assessment

Yes, with refresh. The guard in `resolveGeminiCliEntry` is the narrowest maintainable fix because it removes only the unreliable automatic audio fallback and avoids changing explicit user-configured CLI models or image/video behavior.
