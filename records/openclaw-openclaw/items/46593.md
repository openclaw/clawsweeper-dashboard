## Summary

Keep this PR open: current main and the latest release still allow audio auto-detect to fall through to the generic Gemini CLI `read_many_files` entry, so the defensive change is still relevant. It should not merge yet because the added regression test can pass without exercising the Gemini fallback, the docs need a current-main refresh, and the external contributor has not provided real after-fix behavior proof.

## What This Changes

The PR skips Gemini CLI auto-detection for audio media-understanding flows, preserves Gemini CLI auto-detect for image/video, and adds an audio docs/test update.

## Best Possible Solution

Land a refreshed audio-only auto-detect skip that prevents automatic audio transcription from selecting Gemini CLI, with a fallback-exercising regression test and both audio docs pages updated; keep the longer-term binary audio attachment work on https://github.com/openclaw/openclaw/pull/46607 or its replacement.

## Reproduction Assessment

Yes. Source inspection on current main shows audio auto-detect can still reach the generic Gemini CLI entry after provider/local options miss, and the linked reporter issue gives concrete Telegram `.ogg` headless failure examples; I did not run a live flow in this read-only review.

## Solution Assessment

Yes, with refresh. The audio-only guard is the narrowest defensive fix because it removes only unreliable automatic Gemini CLI audio fallback while preserving explicit CLI configuration and image/video Gemini auto-detect.
