## Summary

Keep open. Current main and the latest shipped tag still allow audio auto-detect to fall through to Gemini CLI, so the core fix remains useful; the PR should be refreshed before merge because its docs patch is based on an older auto-detect table and its regression test should cover the actual Gemini-only fallback path.

## What This Changes

This PR adds an audio-only Gemini CLI auto-detect skip, adds regression coverage for audio transcription when a `gemini` binary is present, and updates the audio docs to explain the audio exception.

## Best Possible Solution

Land a rebased equivalent of the audio-only guard so auto-detect never selects Gemini CLI for audio unless the user explicitly configures that CLI entry, while preserving Gemini CLI auto-detect for image/video and aligning docs/tests/changelog with the current provider/local fallback behavior.

## Reproduction Assessment

Yes. Current main provides a source-level reproduction path: with no active/provider/local audio entry and a working `gemini` binary, audio auto-detect reaches `resolveGeminiCliEntry` and can select Gemini CLI; the linked #46588 discussion also includes real Telegram `.ogg` headless failure examples.

## Solution Assessment

Yes, with refresh. The audio-only guard in `resolveGeminiCliEntry` is the narrowest maintainable fix because it removes only the unreliable automatic audio fallback and preserves explicit CLI configuration plus image/video Gemini behavior.
