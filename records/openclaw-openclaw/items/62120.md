## Summary

Keep open. Current main still runs configured-channel plugin preload for `channels login` before the selected-channel auth resolver can invoke the QR login path, and there is no linked closing PR; the broader https://github.com/openclaw/openclaw/issues/62095 remains open but does not replace this narrower host CLI bug.

## What This Changes

The issue reports that `openclaw channels login --channel openclaw-weixin --verbose` hangs before the external Weixin plugin starts QR login on OpenClaw 2026.4.5.

## Best Possible Solution

Land a channel-agnostic CLI startup fix so `channels login` and `channels logout` avoid pre-action configured-channel plugin preload and load only the requested or inferred auth-capable channel inside the existing resolver.

## Reproduction Assessment

Yes for the host-side path by source inspection: current main resolves `channels login` to configured-channel plugin preload during pre-action bootstrap before `runChannelLogin()` can call `plugin.auth.login`. Full Weixin QR behavior still needs live validation after the host policy fix.

## Solution Assessment

Yes. The narrow maintainable solution is a generic CLI startup policy change plus focused channel-auth/startup tests, without special-casing `openclaw-weixin` or changing the external plugin contract.
