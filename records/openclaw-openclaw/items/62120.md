## Summary

Keep open. Current main still applies the broad parent `channels` plugin preload policy before `channels login` reaches the selected channel auth resolver, so the host-side hang path remains source-reproducible and not superseded by related plugin-only or config-block reports.

## What This Changes

The issue reports `openclaw channels login --channel openclaw-weixin --verbose` hanging before the Weixin QR login flow starts on OpenClaw 2026.4.5.

## Best Possible Solution

Add a channel-agnostic CLI startup policy so `channels login` and `channels logout` avoid broad pre-action configured-channel preload and load only the selected or inferred auth-capable channel through the existing resolver.

## Reproduction Assessment

Yes, at source level. Current main still resolves `channels login` to configured-channel plugin preload during pre-action bootstrap before `runChannelLogin()` can call the selected plugin's `auth.login`.

## Solution Assessment

Yes. A generic CLI startup-policy fix with regression coverage is the narrow maintainable solution; a Weixin-specific workaround would leave the same pre-auth preload hazard for other channel plugins.
