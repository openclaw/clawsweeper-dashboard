## Summary

Keep open. Current main still lets `channels login` and `channels logout` inherit the parent `channels` broad plugin-preload policy, so CLI pre-action bootstrap can load all plugins before the selected external channel reaches its QR-login implementation.

## What This Changes

The issue reports that `openclaw channels login --channel openclaw-weixin --verbose` hangs before the Weixin plugin QR-login flow starts on OpenClaw 2026.4.5.

## Best Possible Solution

Add exact, channel-agnostic startup policy for interactive channel auth so `channels login` and `channels logout` avoid broad pre-action plugin preload, then rely on the existing scoped auth resolver to load only the requested or inferred auth-capable channel.

## Reproduction Assessment

Yes for the host-side regression path: current main resolves `channels login` through the parent `channels` `loadPlugins: "always"` policy, and pre-action bootstrap runs before `runChannelLogin()`. The full Weixin QR flow still needs live validation after the host startup-policy fix.

## Solution Assessment

Yes. The narrow maintainable fix is to change CLI startup policy and tests without special-casing `openclaw-weixin`, because the existing channel-auth resolver already provides the generic selected-plugin loading boundary after bootstrap.
