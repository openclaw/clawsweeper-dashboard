## Summary

Keep open. Current main still lets `channels login` inherit the broad configured-channel plugin preload before the selected channel auth flow runs, so the reported host-side hang path remains source-reproducible and is not replaced by the broader stability umbrella.

## What This Changes

The issue reports that `openclaw channels login --channel openclaw-weixin --verbose` hangs before the Weixin plugin can print its QR login prompt.

## Best Possible Solution

Land a channel-agnostic CLI startup fix so interactive `channels login` and `channels logout` avoid broad pre-action configured-channel plugin preload and load only the requested or inferred auth-capable channel through the existing resolver.

## Reproduction Assessment

Yes for the host-side path by source inspection: current main still resolves `channels login` to configured-channel plugin preload during pre-action bootstrap before `runChannelLogin()` can call `plugin.auth.login`. Full Weixin QR behavior still needs live validation after the host fix.

## Solution Assessment

Yes. The best fix is a generic CLI startup policy change plus focused regression coverage, not a Weixin-specific special case or a plugin contract change.
