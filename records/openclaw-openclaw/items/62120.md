## Summary

Keep open. Current main still lets `channels login` inherit the parent `channels` configured-channel plugin preload before the selected plugin auth flow runs, so the reported host-side pre-QR hang remains source-reproducible.

## What This Changes

The issue reports that `openclaw channels login --channel openclaw-weixin --verbose` hangs before the external Weixin QR login flow starts on OpenClaw 2026.4.5.

## Best Possible Solution

Add a channel-agnostic CLI startup policy so `channels login` and `channels logout` avoid broad pre-action configured-channel preload, then load only the selected or inferred auth-capable channel through the existing resolver with regression coverage and live QR proof when available.

## Reproduction Assessment

Yes, at source level. Current main resolves `channels login` to configured-channel plugin preload during pre-action bootstrap before `runChannelLogin()` can call the selected plugin's `auth.login`.

## Solution Assessment

Yes. A generic CLI startup-policy fix is the narrow maintainable solution; a Weixin-specific workaround would leave the same pre-auth preload hazard for other channel plugins.
