## Summary

Keep open. Current main narrowed the parent `channels` preload to configured channel plugins, but `channels login` and `channels logout` still inherit pre-action plugin loading before `runChannelLogin()` reaches the scoped auth resolver, so the reported hang-before-QR host path remains actionable.

## What This Changes

The issue reports that `openclaw channels login --channel openclaw-weixin` hangs before the external Weixin plugin can print its QR-login prompt.

## Best Possible Solution

Add a channel-agnostic startup policy for interactive channel auth so `channels login` and `channels logout` avoid pre-action configured-channel plugin preload, then load only the requested or inferred auth-capable channel inside the existing auth resolver.

## Reproduction Assessment

Yes for the host-side path: current main resolves `channels login` to pre-action configured-channel plugin loading, and bootstrap runs before `runChannelLogin()` can call `plugin.auth.login`. The full Weixin QR flow still needs live validation after the host fix.

## Solution Assessment

Yes. The narrow maintainable fix is to change CLI startup policy and channel-auth tests without special-casing `openclaw-weixin`, because the selected-channel resolver already provides the generic seam.
