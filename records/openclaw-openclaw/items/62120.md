## Summary

Keep open. Current main still lets `channels login` inherit the parent `channels` pre-action configured-channel preload before the selected plugin auth flow runs, so the host-side hang path remains source-reproducible and is not covered by the later Weixin catalog or Undici fetch fixes.

## What This Changes

The issue reports that `openclaw channels login --channel openclaw-weixin --verbose` hangs before the Weixin plugin can print QR-login output.

## Best Possible Solution

Add a channel-agnostic startup policy so `channels login` and `channels logout` avoid broad pre-action configured-channel preload, then load only the selected or inferred auth-capable channel through the existing resolver with regression coverage and live QR proof when credentials are available.

## Reproduction Assessment

Yes, at source level. Current main still resolves `channels login` to configured-channel plugin preload during pre-action bootstrap before `runChannelLogin()` can call the selected plugin's `auth.login`; I did not run a live macOS Weixin QR login in this read-only review.

## Solution Assessment

Yes. A generic CLI startup-policy fix is the narrow maintainable solution; a Weixin-specific workaround would leave the same pre-auth preload hazard for other channel plugins.
