## Summary

This PR should stay open: it adds a new privileged admin HTTP RPC surface, but source review found blocking contract issues and the PR still lacks after-fix real behavior proof beyond a smoke-test claim.

## What This Changes

The PR adds `POST /api/v1/admin/rpc`, an admin HTTP router, a gateway-method allowlist, docs, auth/scope wiring, and focused tests for the new endpoint.

## Best Possible Solution

Land only after the endpoint preserves JSON error behavior, has accurate discovery/docs, includes real behavior proof, and gets appropriate review for the privileged HTTP surface.

## Reproduction Assessment

Yes for the review findings by source inspection: allowlisted handler exceptions escape to the plain-text HTTP fallback, and current `commands.list` is implemented and documented as command inventory rather than admin RPC method discovery. No live branch run was performed.

## Solution Assessment

No; direct dispatch through the existing gateway handler is a plausible direction, but the PR needs to preserve the HTTP JSON contract, clarify or implement discovery, and settle the privileged allowlist before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Handle gateway handler failures as JSON RPC errors:** `src/gateway/admin-http-api/rpc/handlers.ts:228-235`
  - body: `dispatchAdminRpc` lets exceptions from `handleGatewayRequest` escape the admin route. Existing gateway handlers can throw before calling `respond`, and the outer HTTP server then returns a plain `Internal Server Error`, which breaks this endpoint's JSON error contract and status mapping. Please catch handler failures here and convert them to the documented `{ ok: false, error }` response.
  - confidence: 0.86
- **[P2] Fix admin RPC discovery to return allowed methods:** `src/gateway/admin-http-api/rpc/handlers.ts:30`
  - body: The allowlist exposes `commands.list` under `discovery`, and the docs say it lists the HTTP RPC methods allowed by this endpoint. Current `commands.list` returns an agent command inventory, so admin clients calling it for discovery will not receive the HTTP RPC allowlist. Please add a dedicated discovery response/override or remove that claim.
  - confidence: 0.89
- **[P3] Echo or remove the optional RPC id:** `src/gateway/admin-http-api/rpc/handlers.ts:90-92`
  - body: The request parser accepts an optional `id` and builds a Gateway `RequestFrame`, but `RpcResponse` and the `respond` adapter drop that id from both success and error responses. Since the PR body advertises response ids, clients cannot correlate the optional id they sent; either carry `frame.id` into the HTTP response or remove the unused id contract.
  - confidence: 0.73
