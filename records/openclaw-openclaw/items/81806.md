## Summary

Keep open: this is a new privileged gateway feature, and the patch has blocking contract issues around JSON error handling, admin RPC discovery, and the optional request id. Contributor real behavior proof is also still missing before merge.

## What This Changes

The PR adds `POST /api/v1/admin/rpc`, an admin HTTP router and allowlist, shared-token auth/scope wiring, docs, and gateway tests for invoking selected gateway methods over HTTP.

## Best Possible Solution

Land only after the endpoint preserves JSON error behavior, has accurate discovery and id semantics, includes real behavior proof, and gets appropriate review for the privileged allowlist.

## Reproduction Assessment

Yes for the review findings by source inspection: `dispatchAdminRpc` awaits `handleGatewayRequest` without a catch, and current `commands.list` returns command inventory rather than admin RPC discovery. No live branch run was performed.

## Solution Assessment

No; dispatching through existing gateway handlers is a plausible direction, but the PR needs to preserve the HTTP JSON contract, implement or document discovery accurately, and settle the id/allowlist contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Handle gateway handler failures as JSON RPC errors:** `src/gateway/admin-http-api/rpc/handlers.ts:228-235`
  - body: `dispatchAdminRpc` lets exceptions from `handleGatewayRequest` escape the admin route. Existing gateway handlers can throw before calling `respond`, and the outer HTTP server then returns a plain `Internal Server Error`, breaking this endpoint's JSON error contract and status mapping.
  - confidence: 0.86
- **[P2] Fix admin RPC discovery to return allowed methods:** `src/gateway/admin-http-api/rpc/handlers.ts:30`
  - body: The allowlist exposes `commands.list` under discovery, and the docs say it lists the HTTP RPC methods allowed by this endpoint. Current `commands.list` returns an agent command inventory, so admin clients calling it for discovery will not receive the HTTP RPC allowlist.
  - confidence: 0.89
- **[P3] Echo or remove the optional RPC id:** `src/gateway/admin-http-api/rpc/handlers.ts:90-92`
  - body: The request parser accepts an optional `id` and builds a Gateway `RequestFrame`, but `RpcResponse` and the `respond` adapter drop that id from both success and error responses. Since the PR body advertises response ids, clients cannot correlate the optional id they sent.
  - confidence: 0.73
