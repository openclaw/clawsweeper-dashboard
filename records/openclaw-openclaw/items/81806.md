Keep open: the PR now includes acceptable real behavior proof, but the latest diff still has blocking admin HTTP RPC contract bugs around JSON error handling and discovery semantics.

## What This Changes

The branch adds a token-authenticated `POST /api/v1/admin/rpc` route that forwards an allowlisted set of gateway methods through existing gateway handlers, with docs, scope wiring, and focused tests.

## Best Possible Solution

Land only after the endpoint preserves JSON error behavior, exposes accurate discovery or removes that claim, aligns the optional id/response contract, and receives owner review for the privileged allowlist.

## Reproduction Assessment

Yes for the review findings by source inspection: `dispatchAdminRpc` lets `handleGatewayRequest` exceptions escape to the outer HTTP text fallback, and `commands.list` is existing command inventory rather than HTTP RPC allowlist discovery. No live branch run was performed by this review.

## Solution Assessment

No; forwarding through existing gateway handlers is a plausible direction, but the PR needs to preserve the HTTP JSON contract, provide accurate discovery, and settle the response id contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle gateway handler failures as JSON RPC errors:** `src/gateway/admin-http-api/rpc/handlers.ts:228-236`
  - body: `dispatchAdminRpc` lets exceptions from `handleGatewayRequest` escape the admin route. Existing gateway handlers can throw before calling `respond`, and the outer HTTP server then returns a plain `Internal Server Error`, breaking this endpoint's JSON error contract and status mapping. Please catch handler failures here and convert them to the documented `{ ok: false, error }` response.
  - confidence: 0.88
- **[P2] Fix admin RPC discovery to return allowed methods:** `src/gateway/admin-http-api/rpc/handlers.ts:30`
  - body: The allowlist exposes `commands.list` under `discovery`, and the docs say it lists the HTTP RPC methods allowed by this endpoint. Current `commands.list` returns an agent command inventory, so admin clients calling it for discovery will not receive the HTTP RPC allowlist. Please add a dedicated discovery response/override or remove that claim.
  - confidence: 0.9
- **[P3] Echo or drop the HTTP RPC id contract:** `src/gateway/admin-http-api/rpc/handlers.ts:90-92`
  - body: The parser accepts an optional request `id` and builds a Gateway request frame with it, but `RpcResponse` and the HTTP response adapter drop that id from success and error responses. Since the PR body advertises response ids, clients cannot correlate the id they sent; either carry `frame.id` into the HTTP response or remove the unused id contract.
  - confidence: 0.72
