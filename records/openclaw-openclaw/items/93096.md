## Summary

Keep open: the performance problem is real, but this PR's module-level 5 ms cache can return stale task projections after task mutations or maintenance configuration changes, and the supplied proof is only tests/build output rather than a real `openclaw status` run.

## What This Changes

The PR adds a 5 ms module-level cache to `reconcileInspectableTasks()` and clears it from the task-registry maintenance test reset helper.

## Best Possible Solution

Share one reconciled task snapshot inside `getStatusSummary()`, or add explicit dirty invalidation for task upsert/delete/restore and maintenance configuration changes before keeping a module-level cache.

## Reproduction Assessment

Yes by source inspection: current `main` calls the task registry summary and audit paths separately during status, and the PR's cache can be traced through post-mutation task-maintenance reads. I did not run a live large-ledger timing benchmark in this read-only review.

## Solution Assessment

No: the performance issue is real, but a 5 ms module-level cache is broader than the status hot path and lacks freshness invalidation. A per-status shared snapshot, or a dirty-invalidated cache with tests, is the safer fix.
