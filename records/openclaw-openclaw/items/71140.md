## Summary

Keep this open. Current main already has the Telegram media/GIF delivery path, but it still does not parse `[[react...]]` or `[[react_to_current:...]]` into structured outbound data, does not carry `channelData.telegram.reaction`, and still rejects no-text/no-media Telegram replies before a reaction can be applied. Open draft PR #71141 is the active closing implementation candidate, so the issue should remain open until that PR is reviewed, merged, or replaced.

## What This Changes

The issue requests Telegram outbound reply normalization and delivery changes so reaction directives become native Telegram reactions and media directives are consumed as media rather than leaked as chat text.

## Best Possible Solution

Review, land, or replace PR #71141 with a focused implementation that extends shared directive parsing and outbound normalization, permits reaction-only Telegram replies when a valid reply target exists, and invokes the existing Telegram reaction primitive while preserving the current media/GIF behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `[[react_to_current:...]]` is not recognized by the shared directive parser, no structured reaction data is emitted, and Telegram delivery still rejects reaction-only replies before channel data can be used.

## Solution Assessment

Yes. The issue’s proposed shared parser/normalization plus Telegram delivery/adapter wiring is the narrow maintainable fix; a Telegram-only string special case would leave outbound payload behavior inconsistent across delivery lanes.
