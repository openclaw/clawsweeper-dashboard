## Summary

Keep this open. Current main still supports Telegram media extraction and direct Telegram reactions, but it does not implement the requested normal-reply reaction directive path; open draft PR #71141 is still the active closing candidate for this issue.

## What This Changes

The issue asks OpenClaw to parse Telegram reaction directives and media directives into structured outbound behavior instead of leaking raw control text.

## Best Possible Solution

Ship a focused fix that extends shared reply directive parsing, carries Telegram reaction intent through outbound payload normalization, permits reaction-only Telegram replies when a reply target exists, and invokes the existing Telegram reaction primitive with parser, payload, adapter, and delivery coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `[[react_to_current:...]]` is not matched by the directive parser, no reaction channel data is emitted, and reaction-only Telegram replies still fall into the no-text/no-media rejection path.

## Solution Assessment

Yes. The shared-parser plus Telegram adapter/delivery wiring is the narrow maintainable direction; a Telegram-only string special case would leave outbound normalization inconsistent across reply paths.
