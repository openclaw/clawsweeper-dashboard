## Summary

Keep this open. Current main still has the Telegram reaction-directive gap: media directives and native Telegram reaction primitives exist, but `[[react...]]` / `[[react_to_current:...]]` are not parsed into structured outbound data, reaction-only replies are still rejected before delivery, and the Telegram outbound adapter does not execute `channelData.telegram.reaction`. The linked draft PR #71141 remains the active closing implementation candidate, so this issue should stay open until that PR is reviewed, merged, or replaced.

## What This Changes

Request to parse Telegram outbound reaction directives into structured payload data and send native Telegram reactions while preserving existing MEDIA/GIF behavior.

## Best Possible Solution

Review, land, or replace PR #71141 with a focused implementation that extends shared directive parsing and outbound normalization, permits reaction-only Telegram replies when a valid reply target exists, and invokes the existing Telegram reaction primitive without regressing the current media/GIF path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `[[react_to_current:...]]` is not recognized by the shared parser, no structured reaction data is emitted, and Telegram delivery still rejects no-text/no-media replies before a reaction can be applied.

## Solution Assessment

Yes. The shared parser/normalization plus Telegram delivery/adapter wiring described in the issue is the narrow maintainable direction; a Telegram-only string special case would leave outbound payload behavior inconsistent across delivery lanes.
