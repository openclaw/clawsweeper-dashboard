## Summary

Keep this open. Current main still has the Telegram media path, but the reaction directive path requested here is not implemented, and the provided GitHub context shows open draft PR #71141 is the active closing implementation candidate rather than a merged fix.

## What This Changes

Make Telegram outbound replies consume reaction and media directives so native Telegram reactions and GIF/media sends happen instead of leaking control text or rejecting reaction-only replies.

## Best Possible Solution

Review, land, or replace PR #71141 with a focused fix that extends shared directive parsing and outbound normalization, permits reaction-only Telegram replies when a reply target exists, and invokes the existing native Telegram reaction primitive while preserving the existing media behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `[[react_to_current:...]]` is not recognized by the inline directive parser, no reaction channel data is emitted during outbound planning, and Telegram delivery still rejects no-text/no-media replies.

## Solution Assessment

Yes. The shared parser/normalization plus Telegram delivery/adapter wiring described by the issue and draft PR is the narrow maintainable path; a Telegram-only string special case would leave reply payload behavior inconsistent.
