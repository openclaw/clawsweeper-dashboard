## Summary

Keep open: current main still has no cron payload contract or runtime path for per-job `payload.noOutputTimeoutMs`. A shipped change partially mitigates cron `timeoutSeconds` watchdog behavior, and the shared default-cap increase is already tracked separately, but the per-cron override requested here remains a new public API decision.

## What This Changes

Add a bounded per-cron `payload.noOutputTimeoutMs` override for resumed CLI sessions and raise the default resumed CLI watchdog cap beyond 180 seconds.

## Best Possible Solution

Decide the public cron API first; if accepted, add a bounded `noOutputTimeoutMs` field through gateway schema, cron types/normalization/tool guidance, CLI runner params, watchdog resolution, docs, regression tests, and real resumed-cron proof while leaving shared default-cap changes to the existing watchdog-cap thread.

## Reproduction Assessment

Not applicable as a strict bug reproduction because `payload.noOutputTimeoutMs` is not a documented or accepted cron payload field on current main. Source inspection does give high confidence that such a payload value is neither accepted nor forwarded today.

## Solution Assessment

Unclear until maintainers decide the API shape. The closed PR had the right general plumbing direction, but a merge-ready solution needs the public contract/docs and proof gaps resolved, and the default-cap half should stay aligned with the existing broader watchdog-cap item.
