## Summary

Keep open: current main still has no public cron `payload.noOutputTimeoutMs` field or CLI-runner parameter to honor it, and the prior implementation PR was closed unmerged. The shared default-cap part overlaps the broader open watchdog-default work, but the per-cron payload override remains distinct.

## What This Changes

The issue requests a per-cron `payload.noOutputTimeoutMs` override for resumed CLI sessions and a higher default resume no-output cap.

## Best Possible Solution

Decide the cron payload API direction, then if accepted add a bounded `noOutputTimeoutMs` field through the cron protocol/tool docs, normalizer, types, executor, CLI runner params, and resolver with focused regression coverage; coordinate shared default-cap changes with the broader watchdog-default thread.

## Reproduction Assessment

Yes at source level. Current main has no public cron payload field, no normalizer support, no executor forwarding, and no resolver parameter for `payload.noOutputTimeoutMs`; I did not run a live multi-minute cron hang.

## Solution Assessment

Unclear until maintainers approve the new public cron payload knob and default-timeout policy. The narrow implementation shape is straightforward, but it should include docs/tool schema alignment and proof from a resumed CLI cron path.
