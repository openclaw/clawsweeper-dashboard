## Summary

Keep open: current main still has no cron-payload `noOutputTimeoutMs` contract or runtime path into the CLI watchdog resolver, and the resumed watchdog default cap remains 180 seconds. The directly linked fix PR was closed unmerged, while the open watchdog-cap PRs only overlap the default-cap policy and do not implement the per-cron override.

## What This Changes

Request to honor a per-cron no-output watchdog override for resumed CLI sessions and raise the default resumed watchdog cap from 180 seconds to 300 seconds.

## Best Possible Solution

Define a maintainer-approved cron payload watchdog contract, thread it through schema/types/normalization/cron tool/docs/CLI runner params into `resolveCliNoOutputTimeoutMs` with a bounded cap, and reconcile the shared default-cap policy with the open watchdog PRs.

## Reproduction Assessment

Yes, at source level: current main has no cron payload schema/type/normalizer field and no CLI runner parameter, so a per-cron `noOutputTimeoutMs` value cannot be honored. I did not run a live resumed cron CLI job in this read-only review.

## Solution Assessment

Unclear as proposed: the per-cron override is a plausible direction, but it adds public API surface and the default-cap change needs maintainer policy approval. A safer path is a documented bounded override plus focused regression coverage and real resumed-cron proof, with default-cap changes handled together with the overlapping watchdog PRs.
