## Summary

Keep open. Current main includes the older `timeoutSeconds` mitigation for resumed cron runs, but the requested per-cron `payload.noOutputTimeoutMs` API is still absent from the cron protocol, normalizer, CLI runner params, and watchdog resolver.

## What This Changes

The issue requests a bounded per-cron `payload.noOutputTimeoutMs` override for resumed CLI sessions and a higher resume watchdog default.

## Best Possible Solution

Land a documented cron payload API update that adds a bounded `noOutputTimeoutMs` field, preserves it through cron create/update paths, threads it to the CLI watchdog only for resumed sessions, and proves the behavior with focused tests plus real resumed-cron evidence.

## Reproduction Assessment

Yes for the requested behavior's absence: source inspection shows current main has no cron payload field, no normalizer support, no `runCliAgent` parameter, and no resolver input for `payload.noOutputTimeoutMs`. I did not run a live resumed cron session.

## Solution Assessment

Yes, the requested shape is a reasonable maintainable direction if it is treated as an additive cron payload/API feature with docs and bounded validation. The earlier `timeoutSeconds` fix should be preserved rather than replaced.
