## Summary

Keep open: current main still has no public cron payload `noOutputTimeoutMs` field or runtime path into the CLI watchdog, but adding that knob and changing global defaults are product/API decisions rather than a strict existing-contract bug fix.

## What This Changes

The issue asks OpenClaw to honor a per-cron `payload.noOutputTimeoutMs` override on resumed CLI sessions and raise the default resume watchdog cap.

## Best Possible Solution

Decide whether `noOutputTimeoutMs` should become a public cron payload field; if yes, implement bounded end-to-end schema, normalization, executor, resolver, docs, and regression coverage while leaving the global default-cap decision to the broader watchdog PRs.

## Reproduction Assessment

Yes at source level for the missing requested capability: current schema/types omit `payload.noOutputTimeoutMs`, and the executor/resolver have no parameter to use it. This is not a live repro of a documented existing field because the current public cron contract does not expose that field.

## Solution Assessment

Unclear for the combined proposal. The per-cron override shape is plausible if maintainers want that public API, but default cap changes are already being handled in separate watchdog-default PRs and should be reviewed there.
