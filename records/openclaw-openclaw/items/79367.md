## Summary

Keep open: current main still has no public cron `payload.noOutputTimeoutMs` field and no path that forwards such a payload value into the CLI watchdog resolver. A related mitigation for `payload.timeoutSeconds` already exists, and the broad default-cap work is tracked separately, so the remaining request needs maintainer API/reliability judgment rather than cleanup closure.

## What This Changes

Request to add a per-cron payload no-output watchdog override for resumed CLI sessions and raise the default resumed CLI watchdog cap.

## Best Possible Solution

Keep this open until maintainers decide whether to add a bounded, documented `payload.noOutputTimeoutMs` cron payload field and then thread it through the Gateway schema, cron normalization/types, cron tool guidance, CLI runner params, watchdog resolution, docs, and focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main has no `payload.noOutputTimeoutMs` field in cron payload schema/types/tool guidance and no parameter path into the CLI watchdog resolver; I did not run a live resumed cron job in this read-only review.

## Solution Assessment

Unclear. A per-cron payload override is a plausible narrow capability, but the current proposal mixes that new public API with broader default-cap policy already tracked elsewhere; the safer path is a deliberate payload contract plus docs/tests/proof.
