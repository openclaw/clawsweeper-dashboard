## Summary

Keep open: current main still has no cron-payload `noOutputTimeoutMs` contract, and bundled CLI backends still use a configured resume watchdog profile capped at 180s. The requested per-job override plus default-cap change is a public API/default-policy change, so it needs maintainer review rather than cleanup closure.

## What This Changes

Request to add a bounded per-cron `payload.noOutputTimeoutMs` override for resumed CLI watchdogs and raise the default resume watchdog cap to 300s.

## Best Possible Solution

Define a supported, bounded cron payload `noOutputTimeoutMs` contract, thread it through cron CLI runs into watchdog resolution with tests/docs/changelog, and settle the global resume-cap policy against the existing cap-change discussion.

## Reproduction Assessment

Yes, from source inspection: a resumed cron run using a bundled CLI backend reaches `resolveCliNoOutputTimeoutMs` with a configured resume watchdog capped at 180s, and no cron payload `noOutputTimeoutMs` field can affect it. I did not run a live gateway job because this was a read-only review.

## Solution Assessment

Unclear pending maintainer decision: a bounded per-cron override is plausible, but current public cron payload exposes `timeoutSeconds` only and the default-cap portion overlaps an open PR. The maintainable path is a source-level API/docs/test change, not editing built `dist` files.
