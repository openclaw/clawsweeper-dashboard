## Summary

Keep open: current main still has no cron-payload `noOutputTimeoutMs` contract, and bundled CLI backends carry a configured resume watchdog profile capped at 180s, so resumed cron CLI runs cannot use a per-job no-output override today. The requested fix also adds a public cron payload field and changes default watchdog policy, so it needs maintainer review rather than an automatic cleanup close.

## What This Changes

Requests a per-cron agent-turn `noOutputTimeoutMs` override for resumed CLI sessions and a higher default resume watchdog cap.

## Best Possible Solution

Define a supported, bounded cron payload `noOutputTimeoutMs` contract and pass it into CLI watchdog resolution; decide separately whether the global resume default should move to 300s or be handled by the existing cap-change discussion.

## Reproduction Assessment

Yes from source inspection: a resumed cron run using a bundled CLI backend reaches `resolveCliNoOutputTimeoutMs` with a configured resume watchdog capped at 180s, and no cron payload `noOutputTimeoutMs` field can affect it. I did not run a live gateway job because this was a read-only review.

## Solution Assessment

Unclear pending maintainer decision: a bounded per-cron override is plausible, but current public cron contract exposes `timeoutSeconds` only and a broad default-cap PR already overlaps part of the request. The maintainable path is a source-level API/docs/test change, not editing built `dist` files.
