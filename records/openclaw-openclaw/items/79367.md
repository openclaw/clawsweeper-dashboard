## Summary

Keep open. Current main partially lifts cron resume no-output watchdogs when an explicit cron `timeoutSeconds` gives the run a larger budget, but it still does not expose, normalize, or forward a per-cron `payload.noOutputTimeoutMs`; the linked implementation PR was closed unmerged after review found API/docs/proof gaps.

## What This Changes

Request adding a bounded per-cron `payload.noOutputTimeoutMs` override for resumed CLI watchdogs and raising the resume no-output cap above 180 seconds.

## Best Possible Solution

Either document the current `timeoutSeconds` plus backend watchdog behavior as the supported knobs, or add a bounded per-cron no-output field consistently across protocol schema, cron tool/docs, normalizer, executor, CLI resolver, and regression/live proof.

## Reproduction Assessment

Yes, at source level: current main has no cron payload `noOutputTimeoutMs` type/schema/normalizer path, the cron CLI executor does not forward one, and the watchdog resolver has no payload override input. I did not run a live resumed cron job in this read-only review.

## Solution Assessment

Unclear. A per-cron no-output override is a plausible maintainable direction, but current main already partially mitigates the 180s resume cap through `timeoutSeconds`, so maintainers should first confirm that a separate public payload knob is desired.
