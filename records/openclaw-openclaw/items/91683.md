## Summary

This should stay open: current main has partial PDF/media-understanding and OpenResponses input-file handling, but it does not implement the requested first-class provider-native file/document content path across normal sessions, Gateway chat, ACP/subagent turns, channels, and provider routing.

## What This Changes

The issue requests first-class provider-native file/document forwarding for non-image attachments across OpenClaw sessions, Gateway, UI, CLI/API, channels, and subagent paths.

## Best Possible Solution

Add one shared file/document content and capability contract, map it through provider-owned runtime code, and make the current managed-media/tool extraction path an explicit fallback with user-visible status.

## Reproduction Assessment

Yes. Source inspection shows non-image chat attachments are offloaded to managed media paths while the core LLM, harness, and ACP turn contracts only carry text/image-native model input.

## Solution Assessment

Unclear as an implementation plan. The direction is aligned, but the exact API, provider capability metadata, persistence, and fallback semantics need maintainer design before a safe PR.
