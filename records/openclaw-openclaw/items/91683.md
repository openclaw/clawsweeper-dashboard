## Summary

This remains necessary: current main has extraction-based OpenResponses file ingestion and managed-media fallbacks, but no provider-native non-image content path shared by normal sessions, provider streaming, Gateway chat, ACP/Codex, channels, or subagents. No canonical issue or viable PR currently owns the remaining work.

## What This Changes

Requests first-class provider-native forwarding of non-image attachments across sessions, Gateway clients, channels, providers, and subagent flows with explicit fallback reporting.

## Best Possible Solution

Introduce one canonical file/document content and capability contract, implement provider-owned mappings, and expose explicit native, workspace, converted, and unsupported outcomes while retaining managed-media extraction as the fallback.

## Reproduction Assessment

Yes. Current-main source shows non-image Gateway rejection, a text-and-image-only shared LLM contract, runtime rejection of document capability, and extraction-based rather than provider-native OpenResponses file handling.

## Solution Assessment

The requested direction addresses the correct product gap, but the suggested interface is not yet an approved best implementation. The maintainable path starts with a shared typed content and capability boundary plus provider-owned adapters, followed by a deliberate rollout across entry points.
