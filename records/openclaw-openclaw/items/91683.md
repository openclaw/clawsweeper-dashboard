## Label Justifications

- `P2`: This is a real provider/model capability gap for document-heavy workflows, but it is broad feature work rather than an urgent outage or regression.
- `impact:auth-provider`: The request centers on selected-provider/model capability detection and routing file inputs through provider-native formats.
- `impact:security`: Forwarding arbitrary user documents natively to providers affects sensitive data handling, redaction, MIME policy, and provider data-exposure assumptions.

## Summary

Keep open: current main and v2026.6.8 still have image-native model input plus file offload/extraction paths, but not the requested provider-native document/file content contract, runtime capability handling, provider transport mappings, or explicit native-vs-fallback status.

## What This Changes

Request first-class provider-native forwarding for non-image file/document attachments when the selected provider/model supports them, with explicit fallback/status behavior otherwise.

## Best Possible Solution

Keep this as a maintainer-owned design item until OpenClaw chooses one shared document/file content contract, provider capability and MIME/size metadata, native provider adapters, and explicit fallback/status behavior across session, Gateway, channels, and subagents.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: current model/session contracts are text/image-focused, Gateway chat offloads non-images, and OpenResponses input_file is extracted into prompt context rather than forwarded natively.

## Solution Assessment

Is this the best way to solve the issue? Unclear as an implementation plan. The product direction is plausible, but the best fix needs maintainer agreement on the content API, capability metadata, provider adapters, transcript/security behavior, and fallback UX.
