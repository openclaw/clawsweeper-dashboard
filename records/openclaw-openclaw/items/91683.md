## Summary

Keep open: current main and v2026.6.6 still only have partial non-image media staging and OpenResponses extraction, not the requested first-class provider-native file/document content path across sessions, Gateway, providers, ACP/subagents, channels, and Codex-backed flows.

## What This Changes

Requests a provider-agnostic file/document content abstraction that forwards supported non-image attachments natively to model providers with explicit fallback status.

## Best Possible Solution

Define one shared file/document content and capability contract through core LLM/session/Gateway/plugin SDK surfaces, map supported files in provider-owned transports, and keep MediaPaths/tool extraction as an explicit reported fallback.

## Reproduction Assessment

Yes, at source level. Current main shows text/image-only native model contracts while non-image attachments are offloaded, extracted into prompt context, or rejected depending on entrypoint.

## Solution Assessment

Not applicable as a patch yet. The requested direction is aligned, but the best fix needs a maintainer-approved content API, provider capability model, persistence/security policy, and fallback UX before implementation.
