## Summary

This should stay open: current main still has image-native model input plus file extraction/offload paths, but not the requested provider-agnostic document/file content contract, provider capability matrix, explicit native-vs-fallback status, or end-to-end coverage across session, Gateway, channel, ACP/subagent, and provider transports.

## What This Changes

The issue requests first-class provider-native forwarding for non-image attachments, including document/file content types, provider capability checks, native provider mappings, explicit fallback state, and coverage across OpenClaw session, Gateway, UI, CLI/API, channels, and subagent flows.

## Best Possible Solution

Keep this as a maintainer-owned feature/design item until OpenClaw chooses one shared document/file content contract, a provider capability and MIME/size model, native provider adapters, and explicit fallback/status behavior across session, Gateway, channels, and subagents.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection shows current model/session contracts are text/image-focused, Gateway chat offloads non-image files, and OpenResponses input_file is extracted into prompt context rather than forwarded natively.

## Solution Assessment

Is this the best way to solve the issue? Unclear as an implementation plan. The product direction is sound, but the safest solution needs maintainer agreement on the shared content API, provider capability metadata, persistence/transcript handling, and fallback UX before code is queued.
