## Summary

Keep open: current main and the latest release still have text/image-only runtime and Codex input contracts, while the related attachment work stages or extracts files rather than forwarding provider-native document/file content.

## What This Changes

Issue #91683 requests first-class provider-native forwarding for non-image file/document attachments across OpenClaw entrypoints.

## Best Possible Solution

Define a shared document/file content and capability contract, including provider adapter mappings and explicit fallback/status behavior, then implement it across attachment entrypoints with security limits and docs.

## Reproduction Assessment

Yes, at source level: runtime, agent, and Codex contracts accept text plus image content only, while chat.send offloads non-images to workspace media and OpenResponses `input_file` extracts prompt context rather than preserving native provider file content.

## Solution Assessment

Unclear as a concrete implementation: the direction is valid, but a narrow patch would be premature until maintainers choose the document/file contract, provider capability model, fallback status surface, and security policy.
