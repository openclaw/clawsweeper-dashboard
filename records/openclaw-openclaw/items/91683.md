## Summary

Keep open: current main and the latest inspected release still provide image-native model input, chat attachment workspace offload, OpenResponses text/PDF extraction, and a PDF tool with limited native provider mode, but not the requested shared provider-native file/document content contract or cross-provider forwarding behavior.

## What This Changes

Request first-class provider-native forwarding for non-image attachments as file/document model inputs across Gateway, sessions, channels, CLI/UI, and subagents.

## Best Possible Solution

Define a shared document/file content and capability contract, then implement provider adapters and explicit fallback/status behavior across the attachment surfaces.

## Reproduction Assessment

Yes at source level: current runtime/message contracts are text/image-only, chat.send offloads non-images to workspace media, and OpenResponses input_file extracts prompt context rather than forwarding native provider files.

## Solution Assessment

Unclear as an implementation plan; the direction is plausible, but the best fix is a staged maintainer-owned design rather than a narrow repair.
