## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve reranked order before reporting active:** `extensions/memory-core/src/tools.ts:536`
  - body: `MemoryIndexManager.search` can now return results ordered by `rerankScore`, but the tool path later calls `mergeMemorySearchCorpusResults`, which sorts memory-only results by raw `score`. For `corpus="memory"` or `corpus="all"` without supplements, a lower-fusion-score result promoted by the reranker is demoted again while `debug.rerank` still reports `active`; update the tool merge/sort path to respect rerank order or `rerankScore`, and add a tool-level regression.
  - confidence: 0.92
- **[P2] Use the exported source type in reranker docs:** `docs/plugins/memory-reranker.md:38`
  - body: The docs define `MemoryRerankCandidate.source` as `string` and suggest `"sqlite-vec"`, but the exported type is `MemorySource` and core passes result sources such as `"memory"` or `"sessions"`. Plugin authors following this example will write code against a value the API does not provide; make the guide use the exported type and real values, or remove the misleading example.
  - confidence: 0.9
- **[P2] Avoid stale active rerank state for empty pools:** `extensions/memory-core/src/memory/manager.ts:722`
  - body: When a reranker is registered but `merged.length === 0`, this branch emits the previous `rerankState`. After any prior successful search, a later empty search can report `debug.rerank: "active"` even though no scores were applied, contradicting the documented active semantics; emit no rerank debug or a non-active state for empty pools and cover that transition.
  - confidence: 0.84
