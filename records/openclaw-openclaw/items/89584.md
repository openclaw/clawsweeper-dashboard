## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve reranked order at the tool boundary:** `extensions/memory-core/src/tools.ts:536`
  - body: This now exposes `debug.rerank`, but the returned `results` are still assembled through `mergeMemorySearchCorpusResults`, which sorts `surfacedMemoryResults` by raw fusion `score` before returning. A cross-encoder promotion of a lower fusion-score candidate is therefore undone at the user-visible `memory_search` output, so the central feature can report active rerank without delivering reranked order. Preserve the manager's memory order, or sort on an explicit final ranking signal, and add a tool-level test where rerank promotes a lower-score result.
  - confidence: 0.91
- **[P2] Use the exported source type in the reranker docs:** `docs/plugins/memory-reranker.md:38`
  - body: The guide documents `source: string` with a `sqlite-vec` example, but the exported `MemoryRerankCandidate.source` is `MemorySource`, currently `"memory" | "sessions"`. Plugin authors following this page could special-case a backend string they will never receive, so the docs should mirror the exported type and explain the actual values.
  - confidence: 0.96
- **[P2] Stop emitting stale rerank state for empty pools:** `extensions/memory-core/src/memory/manager.ts:722`
  - body: When a reranker is registered but `merged.length === 0`, this emits the previous latched state. After a prior successful or failed search, a no-hit search can therefore report `active` or `degraded` even though no candidates were scored; that contradicts the new docs that say `active` means scores were applied during this search. Emit no per-search rerank state or add an honest skipped/idle state instead, and cover the transition.
  - confidence: 0.88
