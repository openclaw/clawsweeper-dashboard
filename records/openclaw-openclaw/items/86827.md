## Summary

Keep open: the report has current-release live evidence, and current main still contains a credible shared reply-dispatch path where a visible inbound group message can be marked skipped with no queued final reply or user-visible error.

## What This Changes

The issue asks OpenClaw core to recover or visibly report group-channel messages after a prior failed/timeout turn or an active reply-operation lock would otherwise skip the new turn.

## Best Possible Solution

Define one core policy for visible channel messages that arrive while the session is terminal or the reply lane is busy: safely recover terminal entries and either queue/retry or send a visible failure instead of marking the inbound message silently completed.

## Reproduction Assessment

No live high-confidence current-main repro was run in this read-only review. There is a medium-confidence source reproduction path through recordChannelMessageReplyDispatch into dispatchReplyWithBufferedBlockDispatcher, plus reporter live evidence from v2026.5.22.

## Solution Assessment

Unclear: auto-resetting failed or timeout sessions would address only one variant, while the comments also document an active-run busy variant. The best fix should cover both terminal recovery and busy visible-turn handling without creating duplicate replies.
