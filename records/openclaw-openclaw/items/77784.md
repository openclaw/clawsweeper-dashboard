## Summary

Keep open: the PR is a useful, well-documented feature with sufficient Teams proof and green CI, but latest head still has a P2 Teams delegated-auth fallback bug and docs that overstate audience-claim validation.

## What This Changes

The PR adds a policy-gated plugin delegated-auth API, Teams Bot Framework SSO token resolution, hidden auth context propagation through agent/Codex tool execution, docs, and Teams OBO examples.

## Best Possible Solution

Land the feature only after the Teams token fallback continues through all candidate IDs, the audience docs match the implemented claim semantics, and maintainers accept the new delegated-auth API boundary.

## Reproduction Assessment

Yes for the review finding: source inspection shows the resolver iterates multiple Teams user IDs but returns `expired` at the first stale successful token before trying the next candidate. As a bug report against current main this is not applicable, because the PR adds a new feature.

## Solution Assessment

No, not yet: the generic core policy plus Teams-owned SSO design is a maintainable direction, but the implementation should continue fallback candidate checks on expired tokens and align docs with the code before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Continue past expired fallback tokens:** `extensions/msteams/src/delegated-auth.ts:105-107`
  - body: This returns `expired` as soon as the first candidate user ID has a stale token. The resolver already tries both `aadObjectId` and `from.id` because either can hold the Bot Framework token, so a stale token for the first ID prevents a later valid token for the second ID from being used and forces unnecessary re-consent. Continue to the next candidate and only return `expired` after no usable fallback remains.
  - confidence: 0.91
- **[P3] Align audience docs with claim matching:** `docs/plugins/building-plugins.md:318`
  - body: The implementation accepts a multi-valued `aud` claim when at least one value matches the requested or allowed audience, but this text says every `aud` value must match the allowlist. That overstates the enforced guarantee for a token policy feature; update this and the matching Teams doc wording to describe the actual matching rule.
  - confidence: 0.84
