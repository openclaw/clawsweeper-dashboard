## Summary

Keep open: this PR is a plausible compaction bug fix attempt, but the margin-only change is not merge-ready because it still underestimates CJK-heavy content, changes shared pruning behavior, and has only test/mock proof rather than a real compaction run.

## What This Changes

The PR changes the agent compaction `SAFETY_MARGIN` from 1.2 to 1.5, adds regression tests for the new chunk budget, and adds a changelog entry.

## Best Possible Solution

Use content-aware compaction token estimates that reuse `estimateStringChars`, keep separate constants for chunk headroom versus pruning tolerance, and prove the change with CJK-heavy regression coverage plus redacted real-run proof.

## Reproduction Assessment

No high-confidence live reproduction is established. Source inspection does show the underestimation path: OpenClaw feeds messages into a dependency estimator that divides raw character counts by 4, while existing code has a CJK-aware helper for this exact class of undercount.

## Solution Assessment

No. A global 1.5 margin is not the best fix because it still materially undercounts CJK-heavy content and also changes pruning behavior through other `SAFETY_MARGIN` call sites.
