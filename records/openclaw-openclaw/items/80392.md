## Summary

Keep this PR open, but it should not merge as-is: the patch adds the right cleanup primitive but only calls it from the late startup catch, so earlier traced startup failures can still leave the event-loop monitor enabled. The external PR also still lacks required after-fix real behavior proof from a failed gateway startup.

## What This Changes

The PR adds `startupTrace.close()` to disable the startup trace event-loop monitor and calls it from the gateway startup failure catch path.

## Best Possible Solution

Wrap the full traced startup lifetime after `createGatewayStartupTrace()` in cleanup that disables the monitor on every startup failure, with focused early-failure regression coverage and real failed-startup proof.

## Reproduction Assessment

Yes, by source inspection: enable startup tracing or diagnostics event-loop timeline sampling, then force an early measured startup failure before the late `try`, such as TLS runtime unavailable. I did not run a live failing gateway startup in this read-only review.

## Solution Assessment

No. Adding an explicit `close()` primitive is reasonable, but calling it only from the late catch is not the best fix because earlier traced phases still bypass cleanup.
