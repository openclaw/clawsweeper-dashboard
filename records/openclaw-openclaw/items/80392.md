## Summary

Keep this PR open: it addresses a real gateway startup-trace cleanup bug, but the current patch also disables shared event-loop sampling from recoverable measured startup errors, so it needs a narrower cleanup boundary before merge.

## What This Changes

The PR adds gateway startup trace monitor cleanup on startup failure and adjusts Google Chat/Matrix tests for stricter `Response` casts and matcher expectations.

## Best Possible Solution

Keep the fatal startup-failure cleanup, but make monitor closure cover the overall fatal startup path after trace creation without closing the monitor from generic `measure()` span errors.

## Reproduction Assessment

Yes. Source inspection shows current main enables the startup event-loop monitor before config load and only disables it on `ready`; it also shows the PR's generic span-error cleanup would fire for recoverable channel and sidecar startup errors that are caught while startup continues.

## Solution Assessment

No. The intended cleanup is best solved by an overall fatal-startup cleanup boundary, not by closing the monitor inside `startupTrace.measure()` where callers can recover.
