## Summary

Keep open: this is a useful gateway startup cleanup with sufficient terminal proof for the invalid-config failure path, but the patch still has one blocking correctness issue. Cleanup is now tied to any measured span error, and some measured startup errors are recoverable, so successful degraded startups can lose the rest of their event-loop startup trace samples.

## What This Changes

The PR adds startup trace event-loop monitor cleanup on measured span errors and late gateway startup failure, with small Google Chat and Matrix test expectation/type updates.

## Best Possible Solution

Keep startup trace monitor cleanup scoped to the overall failed-startup path, and preserve event-loop sampling for recoverable measured span errors that continue toward readiness.

## Reproduction Assessment

Yes, from source inspection: current main enables the startup event-loop monitor for traced startup and only disables it on the `ready` mark, while invalid config failures occur before readiness. I did not run the reproduction locally because this review is read-only, but the PR body now includes after-fix terminal proof for that path.

## Solution Assessment

No, not yet: cleaning up on failed startup is the right direction, but disabling the monitor inside generic `measure()` error handling is too broad because some measured errors are caught and startup continues. The safer fix is an overall traced-startup failure cleanup scope plus a focused regression for recoverable span errors.

- **[P2] Do not close the trace monitor on recoverable span errors:** `src/gateway/server.impl.ts:398`
