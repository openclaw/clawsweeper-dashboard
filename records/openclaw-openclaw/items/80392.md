## Summary

Keep open: this is a plausible gateway resource-cleanup bug, but the PR only cleans the late startup catch path while current main can enable the event-loop delay monitor and throw during earlier traced phases; it also lacks required real failed-startup proof.

## What This Changes

The PR adds a `startupTrace.close()` helper that disables `monitorEventLoopDelay` and calls it from the gateway startup failure catch path.

## Best Possible Solution

Cover the full traced startup lifetime after `createGatewayStartupTrace()` with trace cleanup, add focused regression coverage for an early measured-phase failure, and require redacted real failed-startup proof before merge.

## Reproduction Assessment

Yes, by source inspection: enable startup tracing or event-loop diagnostics, then trigger an early measured startup failure such as TLS runtime failure before the late catch begins. I did not run a live gateway failure in this read-only review, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

No. Adding an explicit close primitive is useful, but calling it only from the late catch is not the narrow maintainable fix because earlier traced startup failures can still leave the monitor enabled.
