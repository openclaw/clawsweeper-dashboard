## Summary

Keep open: the PR is a plausible bug fix direction, but the current patch only cleans up failures inside the late startup `try` block and still misses earlier traced startup failures where the event-loop monitor can already be enabled.

## What This Changes

The PR adds `startupTrace.close()` to disable the startup `monitorEventLoopDelay` handle and calls it from the gateway startup failure catch path.

## Best Possible Solution

Wrap the full traced startup lifetime after `createGatewayStartupTrace()` in a cleanup path that disables the event-loop monitor on every thrown startup failure, with focused regression coverage for an early measured-phase failure.

## Reproduction Assessment

Yes, by source inspection: enable startup tracing or diagnostics event-loop sampling, then make an early measured startup phase fail before the late catch, such as config/auth/TLS setup. I did not run a live failed gateway startup, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

No. Adding a close primitive is directionally right, but calling it only from the late catch is not the best fix because earlier traced phases can still leak the monitor.
