## What This Changes

The PR adds a `startupTrace.close()` method and calls it from the gateway startup failure catch path to disable `monitorEventLoopDelay`.

## Best Possible Solution

Wrap cleanup in a `finally` or catch/finally scope that covers every traced startup phase after `createGatewayStartupTrace()`, then prove a forced early startup failure disables the monitor.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: enable startup trace or event-loop diagnostics, force a failure before the late `try` block, and the current patch's cleanup will not run. I did not execute a live reproduction in this read-only review.

## Solution Assessment

No. Adding an explicit close method is reasonable, but calling it only in the late catch is not the narrow complete fix because early traced startup phases can still throw before that catch exists.
