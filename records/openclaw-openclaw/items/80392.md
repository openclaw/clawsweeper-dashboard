## Summary

Keep open: current main and v2026.6.10 still only disable the startup trace monitor at `ready`, so this PR addresses a real fatal-startup cleanup gap, but the branch also disables the monitor from every measured span error, including recoverable channel and sidecar startup errors.

## What This Changes

The PR adds startup trace event-loop monitor shutdown in gateway startup failure handling and updates Google Chat and Matrix test casts for stricter `Response` typing.

## Best Possible Solution

Keep the fatal startup cleanup goal, but close the monitor from a fatal-startup boundary that covers early config through later startup failure cleanup while leaving recoverable measured span errors sampled until `ready`.

## Reproduction Assessment

Yes. The PR body gives terminal proof for the fatal invalid-config path, and source inspection shows the remaining PR regression through recoverable measured channel and sidecar startup errors that continue after catching.

## Solution Assessment

No. The late `close()` path is useful, but disabling the monitor inside generic `startupTrace.measure()` errors is too broad; the narrow fix is cleanup at fatal startup boundaries.
