## Summary

Keep open: the PR targets a real source-reproducible gateway startup resource leak, but the patch only cleans up the late startup catch, so early traced startup failures can still leave the event-loop monitor enabled. It also still needs contributor-owned real behavior proof before merge.

## What This Changes

The PR adds a `startupTrace.close()` method in `src/gateway/server.impl.ts` and calls it from the gateway startup failure catch path.

## Best Possible Solution

Wrap the full traced startup lifetime after `createGatewayStartupTrace()` in cleanup that disables the event-loop monitor on every thrown startup failure, while keeping initialized-resource shutdown in the existing late failure path.

## Reproduction Assessment

Yes, by source inspection: enable startup tracing or diagnostics event-loop sampling, then make an early measured phase before the late `try`, such as `tls.runtime`, throw. I did not run a live gateway failure in this read-only review.

## Solution Assessment

No. Adding a `close()` primitive is reasonable, but calling it only from the late catch is not the best fix because earlier traced phases still bypass cleanup.
