## Summary

Keep this PR open: it is a focused gateway lifecycle fix, but the current patch only cleans up failures inside the late startup catch and misses earlier traced startup failures; it also lacks after-fix real behavior proof.

## What This Changes

The PR adds a `startupTrace.close()` method in `src/gateway/server.impl.ts` and calls it from the gateway startup failure catch path.

## Best Possible Solution

Wrap the full traced startup lifetime after `createGatewayStartupTrace()` in a cleanup path that disables the event-loop monitor on every thrown startup failure, with focused regression coverage for an early measured-phase failure.

## Reproduction Assessment

Yes, by source inspection: enable startup tracing or event-loop diagnostics, then make an early measured phase before the line 976 catch throw, such as the TLS runtime check. I did not run a live failing gateway startup in this read-only review, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

No. Adding `close()` is the right primitive, but calling it only from the late catch is not the best fix because earlier traced phases can still leak the monitor; the cleanup scope should cover all traced startup work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

- **[P2] Wrap all traced startup work in cleanup:** `src/gateway/server.impl.ts:1543`
