## Summary

Keep this PR open because source inspection shows the proposed cleanup still misses early traced startup failures, and the external contributor has not provided real after-fix behavior proof.

## What This Changes

The PR adds `startupTrace.close()` to disable the gateway startup trace event-loop monitor and calls it from the gateway startup failure catch path.

## Best Possible Solution

Wrap the full traced startup lifetime after `createGatewayStartupTrace()` in cleanup that disables the event-loop monitor on every startup failure, with focused regression coverage for an early measured-phase throw.

## Reproduction Assessment

Yes, by source inspection: enable startup tracing or event-loop diagnostics, then throw during an early measured phase before the late startup catch begins. I did not run a live failed gateway startup in this read-only review, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

No. Adding `close()` is a useful primitive, but calling it only from the late catch is not the narrow complete fix because earlier traced phases can still bypass cleanup.
