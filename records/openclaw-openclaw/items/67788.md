## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Use the canonical OpenAI auth provider in help:** `src/cli/models-cli.ts:296`
  - body: This new help example tells users to run `openclaw models auth login --provider openai-codex --set-default`, and the docs diff adds a matching `--probe-provider openai-codex` example. Current docs and repo policy route ChatGPT/Codex OAuth through `openai` and reserve `openai-codex` for legacy migration input, so this would publish stale auth setup guidance.
  - confidence: 0.91
- **[P2] Expand help for -h and one-level parent help:** `openclaw.mjs:169`
  - body: The launcher guard only fires for `--help` with more than four argv entries, so common cases like `openclaw models --help` and aliases like `openclaw models auth -h` still skip the eager registration path. Since the PR promises fuller nested help and Commander defines `-h` as the same help option, the fix misses common user invocations.
  - confidence: 0.9
- **[P2] Remove the stop/restart contradiction:** `src/cli/gateway-cli/run.ts:595`
  - body: On a gateway lock failure, the existing error still says a supervised gateway should be stopped, then the new helper immediately says it should be restarted. Users troubleshooting one failure see both instructions for the same condition, so the message should choose the intended service action.
  - confidence: 0.92
- **[P3] Avoid repeating unreachable-target auth diagnostics:** `src/commands/gateway-status/output.ts:244-248`
  - body: `buildGatewayStatusWarnings` already promotes auth diagnostics for unreachable probes into the top-level Warning block. This unconditional inline `Auth hint` prints the first diagnostic again for those same unreachable targets, so it should be gated to reachable/post-connect cases or left in the warning-only path.
  - confidence: 0.84
