## Summary

Keep open. The PR is useful and narrow, but source review confirms it is not merge-ready: the launcher help gate misses common help aliases, gateway lock failures would print conflicting stop/restart advice, and unreachable auth diagnostics can be printed twice.

## What This Changes

The PR updates CLI/docs help text, launcher lazy-subcommand help behavior, gateway run/status guidance, models auth examples, and gateway status config/auth output.

## Best Possible Solution

Preserve the useful help and gateway guidance improvements, but adjust the implementation so all help aliases expand nested command help, lock failures show one consistent status/restart path, and gateway status prints each auth diagnostic in one place.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The three blockers are source-level reproducible from the PR diff plus current main's lazy-registration policy, GatewayLockError handler, and buildGatewayStatusWarnings path.

## Solution Assessment

Is this the best way to solve the issue? No. The direction is sound, but the current patch needs small implementation fixes and targeted tests before it is the narrowest maintainable merge shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Expand help detection for all help aliases:** `openclaw.mjs:166`
  - body: The launcher only disables lazy subcommands when there are more than four argv entries and a literal `--help`. That misses the PR's own `openclaw models --help` example and `-h` aliases such as `openclaw models auth -h`, so those paths can still print lazy placeholder help instead of the real nested command tree. Use the existing help-invocation parsing or cover both help flags and command-root help cases.
  - confidence: 0.9
- **[P2] Remove the stale stop hint from lock failures:** `src/cli/gateway-cli/run.ts:591-595`
  - body: The patch appends restart/status hints after the existing GatewayLockError message, which still tells supervised users to run `openclaw gateway stop`. A lock failure would now print both stop and restart guidance for the same supervised case; keep a single restart/status recommendation and update the old stop-wording test.
  - confidence: 0.94
- **[P3] Avoid duplicating unreachable auth diagnostics:** `src/commands/gateway-status/output.ts:241-245`
  - body: `buildGatewayStatusWarnings` already promotes auth diagnostics for unreachable probes into the top-level Warning block. The new unconditional inline `Auth hint` prints the first same diagnostic again for failed targets, so gate it to reachable probes or otherwise skip diagnostics already shown in warnings.
  - confidence: 0.87
