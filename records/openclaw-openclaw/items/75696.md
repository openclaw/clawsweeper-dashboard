## Summary

Keep open. Current main still has the same conservative boundary described by the issue discussion: only tagged Codex MCP tool approvals are bridged into OpenClaw approvals, while unhandled MCP elicitations fail closed. The Computer Use docs explain setup, OS permissions, and generic tool failures, but they do not yet document the channel-routed app-approval denial symptom or the pre-authorization workaround noted in the discussion.

## What This Changes

The issue requests that OpenClaw either bridge Codex Computer Use MCP app-control approval elicitations from channel-routed sessions, document that those approvals cannot currently be completed from routed chats, or provide a pre-authorization path.

## Best Possible Solution

Add targeted Computer Use troubleshooting documentation now that names the routed-session app approval limitation, explains why direct Codex Desktop chat differs from Discord/OpenClaw harness sessions, and points to pre-authorizing apps in Codex Desktop settings. Keep runtime bridging for a later maintainer-approved design backed by a redacted payload from Codex.

## Reproduction Assessment

Unclear. The issue provides a plausible manual macOS reproduction path and static review confirms unsupported MCP elicitations decline, but this review did not have a live Discord/Codex Desktop run or the exact Computer Use elicitation payload.

## Solution Assessment

Yes for a docs-first fix; unclear for a runtime bridge. The current fail-closed behavior is conservative, so the narrow maintainable next step is documentation plus collecting a redacted upstream payload before changing approval semantics.
