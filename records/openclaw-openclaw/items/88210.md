## Summary

Keep open: the session-store race being targeted is useful to address, but this branch adds a generic short-retry sidecar lock that can throw under normal cross-process contention, and the supplied proof only exercises same-process concurrency.

## What This Changes

The PR wraps the shared session-store writer in an fs-safe sidecar lock, creates the store parent directory before locking, and adjusts session-store regression tests.

## Best Possible Solution

Use session-sized lock semantics around the shared session-store writer, then prove with two real processes that contending writers wait and both updates persist.

## Reproduction Assessment

Yes, source-reproducible rather than executed: hold the PR's sidecar lock longer than its six configured retries, then start another process entering runExclusiveSessionStoreWrite; fs-safe will throw file_lock_timeout.

## Solution Assessment

No. A cross-process lock is a plausible direction, but the best fix should reuse or match the existing session write-lock acquire/stale semantics instead of applying the generic short plugin lock policy.
