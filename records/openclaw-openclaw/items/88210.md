## Best Possible Solution

Use a session-store-sized lock acquisition policy and prove that a contending second process waits and then persists successfully.

## Reproduction Assessment

Yes. From source, hold the branch's sidecar lock longer than the six configured retry attempts, then start a second process that enters runExclusiveSessionStoreWrite; the fs-safe lock manager throws file_lock_timeout when retries are exhausted.

## Solution Assessment

No. A cross-process file lock is the right direction, but this branch uses a short fail-fast acquisition policy and does not prove the two-process wait-and-succeed behavior the fix is meant to provide.
