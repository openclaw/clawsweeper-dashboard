## Summary

Keep open: the PR contains a meaningful session-store concurrency fix idea, but the submitted implementation adds a short generic sidecar-lock policy to the shared session writer, which can turn contention into failed session persistence and is not merge-ready as written.

## What This Changes

The PR wraps file-backed session store writes in a shared file-lock helper and updates session writer/session-store tests.

## Best Possible Solution

Use the existing session write-lock semantics or an owner-approved session-sized acquire/stale policy for any cross-process session-store lock, then prove with two real processes that contending writers wait and preserve both updates.

## Reproduction Assessment

Yes, source-reproducible: hold the PR's sidecar lock longer than its six configured retries, then start another process entering runExclusiveSessionStoreWrite; fs-safe will throw file_lock_timeout. I did not execute that two-process repro in this read-only review.

## Solution Assessment

No. A cross-process lock is a plausible direction, but this implementation uses a short generic plugin file-lock policy instead of the existing session-specific lock semantics and conflicts with current documentation of the normal writer path.
