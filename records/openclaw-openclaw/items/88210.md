## Summary

Keep open: current main still lacks a cross-process session-store writer lock, so the PR addresses a real gap, but it should not merge as-is because the short retry policy can surface `file_lock_timeout` during ordinary contention and the proof only exercises same-process updates.

## What This Changes

The PR wraps `runExclusiveSessionStoreWrite` in the plugin-sdk file lock, creates the session-store parent directory before locking, and adjusts session/script tests.

## Best Possible Solution

Use a session-store-sized wait policy and prove that a second process waits and succeeds after the first writer releases.

## Reproduction Assessment

Yes from source: hold the proposed session-store file lock longer than its six-retry budget, then start another writer from a second process. The OpenClaw wrapper and upstream fs-safe source show exhausted retries surface as `file_lock_timeout`.

## Solution Assessment

No. A cross-process file lock is the right direction, but this implementation copies a short fail-fast retry policy and does not prove that cross-process contention waits and succeeds.
