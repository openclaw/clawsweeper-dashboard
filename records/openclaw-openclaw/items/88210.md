## Summary

Keep open: the PR is a focused session-store concurrency fix, but the current patch is not merge-ready because its short lock retry budget can fail contending writers, real after-fix contention proof is missing, and GitHub reports conflicts with current main.

## What This Changes

The branch wraps session-store writer callbacks in the shared file-lock helper, creates the store directory before locking, and adds a test that observes the sidecar lock file while a writer is active.

## Best Possible Solution

Use a session-store-specific lock policy that waits long enough for normal writes, add contention coverage proving a second writer waits and succeeds, and provide real after-fix proof from a two-process run.

## Reproduction Assessment

Yes from source, though I did not execute it: hold one process's session-store file lock longer than the six short retries, then start a second writer. The fs-safe contract throws `file_lock_timeout` once that retry budget is exhausted.

## Solution Assessment

No. A file lock is a plausible direction for cross-process protection, but this implementation needs session-store-sized wait semantics and proof that contending writers serialize instead of timing out.
