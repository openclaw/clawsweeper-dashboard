## Summary

Keep this PR open, but it is not merge-ready: the cross-process locking direction is useful, yet the implementation uses a generic short-retry file lock where session persistence needs the existing session write-lock policy and two-process proof.

## What This Changes

The PR wraps file-backed session store writes in a generic sidecar file lock and updates session-writer regression tests.

## Best Possible Solution

Use the existing session write-lock policy or an equivalent session-sized acquire/stale policy in the store writer, then prove with two real processes that contending writers wait and preserve both updates.

## Reproduction Assessment

Yes, source-reproducible: hold the PR's sidecar lock longer than its six configured retries, then start another process entering runExclusiveSessionStoreWrite; fs-safe will throw file_lock_timeout. I did not execute that two-process repro locally.

## Solution Assessment

No. A cross-process lock is a plausible direction, but the best fix should reuse the existing session write-lock policy or match its timeout/stale semantics instead of adding a generic short-retry lock to the session-store gate.
