## Label Justifications

- `P2`: This is a normal-priority session-store concurrency fix with important but bounded blast radius.
- `merge-risk: 🚨 session-state`: The diff changes session-store write serialization and can fail contending writes before session metadata is persisted.
- `merge-risk: 🚨 availability`: Lock timeout exceptions from the shared writer can make affected agent or gateway operations fail under concurrent processes.

## Summary

Keep open: the bug-fix direction is useful, but the PR can still fail under normal cross-process session-store contention because the new file lock uses a very short retry budget, and the PR has no real behavior proof beyond unit/lint checks.

## What This Changes

The branch wraps session-store writes in `withFileLock`, creates the store parent directory before locking, and adds a unit test that observes the sidecar lock file while a writer is active.

## Best Possible Solution

Use a session-store-specific lock policy that waits long enough for normal writes, or reuses the dedicated session write-lock semantics, with regression coverage proving a second contending writer waits and then succeeds.

## Reproduction Assessment

Yes from source, though I did not execute it: hold one process's session-store file lock longer than the six short retries, then start a second writer. The dependency code throws `file_lock_timeout` once the retry count is exhausted.

## Solution Assessment

No. A file lock is the right direction for cross-process protection, but this implementation copies a short retry policy that can fail instead of serializing session-store writes; the safer path is a session-store-sized wait policy plus contention proof.
