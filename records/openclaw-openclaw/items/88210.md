## Summary

Keep open: current main does not yet protect session-store writes with a cross-process lock, but this patch is not merge-ready because the new shared writer gate can fail contending writes after a very short retry budget and the external real-behavior proof is still missing.

## What This Changes

The PR wraps the session-store writer in the shared file-lock helper, creates the store directory before locking, and adds a test that asserts a `.lock` sidecar exists while a writer is active.

## Best Possible Solution

Use a session-store-sized lock wait policy and prove with coverage plus redacted real two-process contention output that a second writer waits and succeeds after the first releases the lock.

## Reproduction Assessment

Yes from source for the PR defect: hold the proposed session-store file lock longer than its six-retry budget, then start another writer; @openclaw/fs-safe throws file_lock_timeout when the retry budget is exhausted. I did not run a live two-process repro because this review is read-only.

## Solution Assessment

No. A cross-process file lock is the right direction, but this implementation uses a short fail-fast retry policy for a shared session persistence path and the added test only proves a sidecar exists, not that a contending writer waits and succeeds.
