## Summary

Keep this PR open, but it is not merge-ready: the branch addresses a real session-store concurrency concern, yet the proposed lock policy can turn ordinary contention into `file_lock_timeout`, conflicts with the documented no-runtime-lock session-store contract, and still lacks real after-fix behavior proof.

## What This Changes

The branch adds a file-lock wrapper around the session-store writer, creates the store directory before locking, and adds a unit test that observes the lock sidecar during a write.

## Best Possible Solution

Use a session-store-specific serialization policy that either preserves the current Gateway-owned no-runtime-lock contract or adds a maintainer-approved cross-process lock with session-sized wait semantics, stale-lock handling, and real contention proof.

## Reproduction Assessment

Yes from source, though I did not execute a live run: hold one process's session-store file lock past the configured six retries, then start a second writer. The inspected lock dependency throws `file_lock_timeout` once that retry count is exhausted.

## Solution Assessment

No. A cross-process lock may be a plausible mitigation, but this patch uses a short retry budget and reintroduces a runtime session-store lock without proving it is the safest owner-boundary fix.
