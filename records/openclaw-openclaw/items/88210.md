## Summary

Keep open: this is a focused session-state bug-fix PR, but it is not merge-ready because the new generic file-lock policy can turn cross-process contention into failed session persistence, and the supplied proof still does not exercise the cross-process sidecar-lock path.

## What This Changes

The PR wraps the shared session-store writer with a file lock, creates the store directory before locking, and adjusts session/test coverage.

## Best Possible Solution

Use session-sized acquire/stale/retry semantics around the shared session-store writer, then prove with two real processes that contending writers wait and both updates persist.

## Reproduction Assessment

Yes, from source: hold the PR's .lock sidecar longer than its six retry budget and a second process entering runExclusiveSessionStoreWrite will receive file_lock_timeout. I did not execute that two-process repro in this read-only review.

## Solution Assessment

No. A cross-process lock is a plausible direction, but the best fix should reuse or match the existing session write-lock acquire/stale semantics instead of applying the generic short plugin file-lock policy.
