## Summary

Keep open: the PR targets a real session-store concurrency gap, but the current patch is not merge-ready because the short retry policy can fail contending session writes and no real two-process proof has been provided.

## What This Changes

The PR wraps session-store writer callbacks in the shared file-lock helper, creates the store parent directory before locking, and adds a test that checks for a `.lock` sidecar while a writer is active.

## Best Possible Solution

Keep cross-process serialization only if the lock waits through ordinary session-store writes, docs/tests reflect the new behavior, and a real two-process contention run proves both writes survive.

## Reproduction Assessment

Yes from source, though I did not execute it: hold one process's session-store `.lock` longer than the six short retries, then start a second writer. The inspected `@openclaw/fs-safe@0.3.0` implementation throws `file_lock_timeout` after the retry budget is exhausted.

## Solution Assessment

No. A file lock is a plausible direction for cross-process protection, but this implementation copies a short retry policy that can fail instead of reliably serializing session-store writes.
