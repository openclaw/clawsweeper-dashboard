## Summary

Keep open: this PR targets a real session-store concurrency risk, but it is not merge-ready because it adds a generic six-retry file lock to the shared session persistence gate and the supplied proof only exercises same-process concurrency.

## What This Changes

The PR wraps session-store writes in a file-lock helper, creates the store parent directory before locking, and adjusts session-store tests.

## Best Possible Solution

Use session-sized lock semantics, or another maintainer-approved store policy, and prove with two real processes that contending writers wait and both updates persist.

## Reproduction Assessment

Yes, source-reproducible rather than executed: hold the PR's sidecar lock longer than its six configured retries, then start another process entering runExclusiveSessionStoreWrite; fs-safe will throw file_lock_timeout.

## Solution Assessment

No. A cross-process lock is a plausible direction, but this implementation uses the generic short plugin lock policy instead of the session-sized acquire/stale semantics already used for session write locks.
