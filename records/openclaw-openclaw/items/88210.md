## Summary

Keep open: the PR is aimed at a real session-store concurrency gap, but the current patch still uses a short lock retry budget for broad session persistence paths and has no real two-process after-fix proof.

## What This Changes

The PR wraps `runExclusiveSessionStoreWrite` in `withFileLock`, creates the session-store parent directory before locking, and adds a test that checks for the `.lock` sidecar while one writer is active.

## Best Possible Solution

Use a session-store-sized wait policy and prove with a real two-process contention run that a second writer waits and succeeds after the first releases the lock.

## Reproduction Assessment

Yes from source: hold the proposed session-store file lock longer than the configured six retries, then start another writer. The fs-safe lock code throws `file_lock_timeout` when retries are exhausted.

## Solution Assessment

No. A cross-process file lock is the right direction, but this implementation uses a short fail-fast retry policy for a session persistence path and the added test only proves a sidecar exists, not that a contending writer waits and succeeds.
