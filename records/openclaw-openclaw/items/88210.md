## Label Justifications

- `P2`: This is a normal-priority session-store concurrency fix with bounded but real session persistence impact.
- `merge-risk: 🚨 session-state`: The PR changes the shared session-store serialization gate and can fail contending writes before session metadata is persisted.
- `merge-risk: 🚨 availability`: Lock timeout exceptions from the shared writer can make affected gateway or agent operations fail under multi-process contention.

## Summary

Keep this PR open, but it is not merge-ready: the generic six-retry advisory lock can turn cross-process session-store contention into failed persistence, and the supplied proof only exercises same-process concurrency.

## What This Changes

The PR wraps session-store writer callbacks in a shared advisory file lock, adjusts session-cache and lock-presence tests, and adds one plugin prerelease timeout assertion.

## Best Possible Solution

Use the existing session write-lock policy or an equivalent session-sized acquire/stale policy in the store writer, then prove with two real processes that contending writers wait and preserve both updates.

## Reproduction Assessment

Yes, source-reproducible: hold the PR's sidecar lock longer than its six configured retries, then start another process entering runExclusiveSessionStoreWrite; fs-safe will throw file_lock_timeout. I did not execute that two-process repro because this review is read-only.

## Solution Assessment

No. A cross-process lock is the right goal, but the best fix should reuse the existing session write-lock policy or match its timeout and stale semantics instead of adding a generic short-retry lock to the session-store gate.
