## Summary

Keep this PR open because current main and the latest release still only serialize session-store writes inside one process, but the branch is not merge-ready: its new cross-process lock can time out after a very short retry budget and the proof does not demonstrate two-process contention.

## What This Changes

The PR wraps the session-store writer in the SDK file-lock helper, creates the parent directory before locking, and adjusts session/script tests.

## Best Possible Solution

Use a session-store-sized acquire/wait policy and prove that a contending second process waits and succeeds after the first writer releases.

## Reproduction Assessment

Yes, from source for the PR defect: hold the branch's session-store sidecar lock longer than its six short retry delays, then start a second process that enters the same writer. The fs-safe contract shows that exhausted retries throw file_lock_timeout.

## Solution Assessment

No. A cross-process file lock is the right direction, but this implementation uses a short fail-fast retry policy and does not prove the two-process wait-and-succeed behavior the change is meant to provide.
