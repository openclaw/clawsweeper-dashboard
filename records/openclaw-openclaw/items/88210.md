## Summary

Keep open: the PR targets a real session-store concurrency concern, but it reintroduces a runtime sessions.json sidecar lock despite current main/docs avoiding that path, uses a generic short fs-safe retry budget that can fail contending writes, and still lacks two-process proof of the changed behavior.

## What This Changes

The PR wraps the shared session-store writer in an advisory file lock, creates the store parent directory before locking, and adjusts session-store tests.

## Best Possible Solution

Confirm whether OpenClaw wants cross-process sessions.json writes at all; if yes, use session-sized acquire/stale semantics around the shared writer, update the documented contract, and prove two real processes preserve both updates under contention.

## Reproduction Assessment

Yes, at source level: hold the PR's sidecar lock longer than its six configured retries, then start another process entering runExclusiveSessionStoreWrite. The inspected fs-safe contract throws file_lock_timeout once the retry budget is exhausted.

## Solution Assessment

No. A cross-process lock is a plausible direction, but this PR copies a short generic plugin lock policy onto the shared session-store writer and conflicts with the current documented no-runtime-lock store behavior.
