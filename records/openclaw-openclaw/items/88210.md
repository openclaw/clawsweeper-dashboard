## Summary

## What This Changes

The PR wraps session-store writes in the shared file-lock helper, creates the store parent directory before locking, and adds session-writer/test-project coverage updates.

## Best Possible Solution

Use a session-store-sized locking policy that waits through ordinary persistence contention, then prove with tests and real two-process output that a second writer waits and succeeds after release.

## Reproduction Assessment

Yes from source: hold the proposed session-store file lock longer than the configured retry budget, then start another writer. The OpenClaw wrapper and upstream fs-safe source show exhausted retries surface as `file_lock_timeout`.

## Solution Assessment

No. A cross-process file lock is the right direction, but this implementation uses a short fail-fast retry policy for a session persistence path and only tests sidecar existence, not that a contending writer waits and succeeds.
