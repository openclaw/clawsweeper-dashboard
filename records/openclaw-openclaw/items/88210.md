## Summary

Keep open. The cross-process lock direction is useful, but the patch is not merge-ready because the short retry budget can turn normal session-store contention into `file_lock_timeout`, the PR has no real two-process after-fix proof, and the live branch currently conflicts with main.

## What This Changes

The branch wraps session-store writes in `withFileLock`, creates the store parent directory before locking, and adds a unit test that observes the `.lock` sidecar while a writer is active.

## Best Possible Solution

Use a session-store-specific cross-process lock policy that waits long enough for normal writes, or reuse the existing session write-lock semantics, then prove a second contending writer waits and succeeds.

## Reproduction Assessment

Yes from source. Hold one process's session-store file lock longer than the configured six retries, then start a second writer; `@openclaw/fs-safe` throws `file_lock_timeout` when retries exhaust.

## Solution Assessment

No. File locking is the right direction, but this patch copies a short retry policy onto broad session-store writes instead of using session-store-sized wait semantics with contention proof.
