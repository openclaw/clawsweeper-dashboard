## Summary

Current main already addresses the unbounded stdio MCP child leak with session-scoped runtime reuse, explicit transport closure, process-tree cleanup fallback, idle eviction, session cleanup, and gateway shutdown disposal. The proof commit is on main after the latest proven release, so this can close as fixed on main rather than shipped.

## What This Changes

The issue requests closing or pooling gateway-launched stdio MCP transports so repeated MCP tool calls do not leave child processes alive indefinitely.

## Best Possible Solution

Keep the current pooled session-runtime design with explicit close, process-tree fallback, session/shutdown cleanup, and idle TTL, then ship a release containing the main-branch fix for affected 2026.4.x users.

## Reproduction Assessment

Yes. The reporter supplied a concrete process-count reproduction for 2026.4.9, and current-main source shows the former unbounded per-call lifetime is now bounded by session reuse plus dispose/idle cleanup rather than live-verified locally.

## Solution Assessment

Yes. The current implementation follows the report's acceptable pooled-client alternative: it reuses session-scoped clients, closes transports on disposal, and evicts idle runtimes instead of spawning and leaking one child per call.
