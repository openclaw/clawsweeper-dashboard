## Summary

Close as implemented on main: current main has explicit stdio MCP transport shutdown, session runtime disposal, idle eviction, and gateway/session/cron cleanup paths, so the reported 2026.4.9 unbounded child-process leak is addressed in source.

## What This Changes

The issue reports that stdio MCP server child processes remain alive after tool invocations because gateway-side transports are not closed.

## Best Possible Solution

Keep the current centralized MCP lifecycle cleanup and let the next release carry it to users still on 2026.4.9-era builds.

## Reproduction Assessment

No against current main: I did not run a live process-leak repro in this read-only sweep, but source, docs, and regression coverage now show explicit close paths for stdio transports and session runtimes.

## Solution Assessment

Yes: closing through the shared MCP runtime and transport lifecycle is the narrow maintainable fix because it covers tool calls, idle sessions, session reset/delete, cron cleanup, config reload, and gateway shutdown without per-server special cases.
