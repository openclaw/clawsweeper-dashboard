## Summary

Current main and shipped v2026.5.3 already close gateway-managed stdio MCP transports through shared bundle MCP runtime cleanup, so the leak reported on 2026.4.9 is fixed by upgrading.

## What This Changes

Request is to ensure gateway-managed stdio MCP server processes exit after MCP tool/session completion instead of accumulating under the gateway.

## Best Possible Solution

Keep the shipped v2026.5.3 lifecycle cleanup and direct affected 2026.4.9 users to upgrade.

## Reproduction Assessment

No, not against current main. The old 2026.4.9 report has concrete process-count evidence, but current source, docs, and tests show transports are closed, process trees are reaped, and idle/session cleanup is wired.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: close the MCP transport to signal EOF, fall back to process-tree kill, and route run/session/reload/shutdown cleanup through the shared runtime manager.
