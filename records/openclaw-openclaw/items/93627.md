## Summary

Keep open: this focused memory-core bug-fix PR is not merge-ready because the new `strippedHash` field is written but stripped by `normalizeDailyIngestionState` on the next read, so managed light writes can still re-ingest and advance `lastRecalledAt`; the linked issue remains open.

## What This Changes

Adds a light-phase freshness filter using phase signal timestamps and a stripped daily-note hash intended to avoid re-ingesting managed light dreaming writes.

## Best Possible Solution

Preserve the stripped-content hash through the daily-ingestion read/normalize boundary, then prove a two-cycle managed-light-write scenario no longer refreshes `lastRecalledAt` or repeats the work summary.

## Reproduction Assessment

Yes, source-reproducible from PR-head code: `writeDailyIngestionState` can persist `strippedHash`, but `readDailyIngestionState` immediately normalizes it away, so the new skip predicate cannot match after a persisted cycle. I did not run tests because this review is read-only.

## Solution Assessment

No. The state-boundary fix is the right direction, but this implementation is incomplete until the new hash survives the read/normalize path and is covered by a two-cycle managed-write regression.
