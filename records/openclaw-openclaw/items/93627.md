## Summary

Keep open: this is a focused candidate for a real memory-core bug, but the current patch is not merge-ready because unchanged stripped daily content can still look fresh on the next dreaming day and the supplied proof does not show repeated real dreaming cycles after the fix.

## What This Changes

The PR adds a light-dreaming phase-signal freshness filter, persists a stripped daily-note hash in memory-core daily-ingestion state, and updates memory-core tests.

## Best Possible Solution

Separate light-emission freshness from daily-ingestion recency across dreaming days, keep the calendar-day daily ingestion behavior, and add repeated-cycle proof before merge.

## Reproduction Assessment

Yes, source-level. Current main ranks recent light recall entries without lastLightAt filtering, and deduped daily ingestion still writes lastRecalledAt; I did not run a live overnight repro.

## Solution Assessment

No. The phase-signal filter is in the right area, but the strippedHash skip is same-day only; the safer fix preserves cross-day daily ingestion while using an emission/freshness boundary that cannot mark unchanged content as fresh.
