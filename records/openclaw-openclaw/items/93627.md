## Summary

Keep open: this PR is the current candidate fix for https://github.com/openclaw/openclaw/issues/72096, but the final head still lets unchanged daily content refresh Light Sleep freshness on the next dreaming day and the contributor-supplied proof is still build/static/test output rather than repeated-cycle real behavior proof.

## What This Changes

The PR adds a light-dreaming phase-signal freshness filter, persists a stripped daily-note hash in memory-core daily-ingestion state, and updates memory-core tests.

## Best Possible Solution

Land only after light-emission freshness is separated from daily re-ingestion so unchanged content cannot be emitted again across days, with focused regression coverage and redacted repeated-cycle proof.

## Reproduction Assessment

Yes, source-level: current main ranks all recent light recall entries directly, and daily ingestion can refresh lastRecalledAt even when the content is not new for Light Sleep. I did not run a live overnight multi-cycle repro.

## Solution Assessment

No. The phase-signal filter is in the right area, but the same-day strippedHash guard is partial; the safer fix is a durable non-deduped recall/emission boundary or another freshness source that preserves calendar-day daily ingestion without re-emitting unchanged Light Sleep content.
