## Summary

Keep open: the bug is valid and the PR takes the right direction, but the latest head still returns the new irreducible fail-fast path as resettable `context_overflow`, so auto-reply can replace the actionable error with the generic session-reset recovery. The author’s follow-up fixed the exhausted-overflow loop path and added docs/changelog coverage, but the irreducible path still needs the same non-resettable treatment.

## What This Changes

The PR adds an irreducible-overflow preemptive compaction route, fail-fast embedded runner handling, exhausted-overflow metadata, auto-reply reset suppression for exhausted overflows, tests, docs, and a changelog entry.

## Best Possible Solution

Carry a distinct non-resettable irreducible-overflow classification through embedded run metadata and auto-reply handling, update the user guidance to include oversized current input, then validate the focused runner and auto-reply tests.

## Reproduction Assessment

Yes. Source inspection shows current main routes a zero-history system/user baseline overflow to `compact_only`, then the runner can spend overflow compaction attempts before returning a resettable context-overflow error.

## Solution Assessment

No, not yet. The baseline detector is the right narrow fix, but the PR still classifies irreducible overflow as resettable `context_overflow` and its user-facing copy omits oversized current input as a cause.
