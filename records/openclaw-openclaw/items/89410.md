## Summary

Keep open: the privacy goal is valid, but the patch is not merge-ready because it drops the whole mutating exec/bash failure signal on chat surfaces before the existing false-success guard can run, and the external PR has no real Telegram/chat behavior proof.

## What This Changes

The PR passes message channel/provider context into embedded payload construction, suppresses exec-like and middleware tool-error warnings for hard-coded chat providers/session keys, updates Telegram docs, and adds focused payload/dispatch regression tests.

## Best Possible Solution

Scrub raw command, stderr, stack, and child-output details from chat-delivered tool warnings while preserving a compact failure notice when the assistant has not acknowledged a mutating exec/bash failure; avoid a drifting core channel list, then add real Telegram proof.

## Reproduction Assessment

Yes, source-level: on the PR branch, a Telegram/chat payload with assistant success text and a lastToolError for exec/bash returns showWarning false before the mutating failure policy runs. I did not execute tests in this read-only review.

## Solution Assessment

No. Fail-closed avoids raw diagnostic leaks, but removing the whole warning is too broad; the narrower maintainable fix is to strip raw meta/error details while keeping the compact failure signal for unacknowledged mutating failures.
