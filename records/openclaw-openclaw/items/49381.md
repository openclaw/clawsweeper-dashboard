## Summary

Keep open: current main and the latest release still route Feishu reply errors through streaming closeout, and current tests still encode two visible streaming-card closes after an error then recovered final. The linked candidate fix remains open, conflicting, and proof-gated, while the original streaming-off duplicate case still needs coverage or a split follow-up.

## What This Changes

Prevent Feishu from sending duplicate final replies for one inbound DM when a rate-limited primary model fails over to a fallback model.

## Best Possible Solution

Land or replace a focused fix that guarantees one visible Feishu final per inbound dispatch across failover/retry, with regression coverage for streaming-card recovery and the streaming-off final-reply path before considering new cooldown configuration.

## Reproduction Assessment

Yes, at source level: the issue and follow-up comments provide sanitized one-inbound-message logs, and current main still has tests and code paths that close one Feishu streaming card after an error before a later recovered final can close another. I did not run a live Feishu/provider-failover reproduction in this read-only review.

## Solution Assessment

Yes, the best first fix is a narrow Feishu/final-delivery coordination repair with regression coverage, not a new cooldown opt-out. The cooldown policy ideas may be useful follow-ups, but they should not be required to stop duplicate visible replies.
