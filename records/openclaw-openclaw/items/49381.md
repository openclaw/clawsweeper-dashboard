## Summary

Keep open. Current main has useful Feishu duplicate-final mitigations, but it still leaves the reported failover coordination class unresolved: generic final dispatch still sends every non-reasoning final payload, Feishu non-streaming delivery still preserves distinct final payloads, and the streaming error path still closes the active card before allowing a later recovered final. The related candidate fix remains open rather than merged.

## What This Changes

Prevent a single Feishu DM from producing multiple final user-visible replies when a rate-limited primary model fails over to a secondary model.

## Best Possible Solution

Keep this issue open until #59771 or an equivalent fix lands, then add or confirm coverage for the original non-streaming failover path. The maintainable end state is failover-aware final coordination that suppresses a failed primary attempt's user-visible final when a fallback succeeds, without breaking intentional media/status or multi-part replies.

## Reproduction Assessment

Yes at source/test level: the current Feishu tests model two relevant remaining paths, distinct non-streaming finals and a recovered final after error closeout. A live provider-rate-limit reproduction was not run in this read-only review.

## Solution Assessment

Unclear. The linked PR direction is plausible for the streaming-card variant, but current main still preserves non-streaming multi-final behavior, so maintainers should decide whether the fix belongs in Feishu, generic final dispatch, or both.
