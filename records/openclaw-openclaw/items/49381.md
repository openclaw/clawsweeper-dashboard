## Summary

Keep open: current main and the latest release still route Feishu reply errors through streaming closeout, and current tests encode a later recovered final creating a second streaming card. The linked candidate fix remains open, conflicting, and proof-gated, while the original streaming-off duplicate case may still need broader final-reply coordination coverage.

## What This Changes

The issue requests preventing duplicate Feishu final replies when a rate-limited primary model fails over to a secondary model, with attention to final delivery dedupe and provider cooldown probing.

## Best Possible Solution

Land or replace a focused fix that guarantees one visible Feishu final per inbound dispatch across failover/retry, with regression coverage for streaming-card recovery and the streaming-off final-reply path before considering new cooldown configuration.

## Reproduction Assessment

Yes, at source level: the issue and follow-up comment provide sanitized Feishu logs with one inbound DM and multiple visible replies, and current main still closes the active streaming card on reply errors before a later recovered final can arrive. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

Yes, the best first fix is a narrow Feishu/final-delivery coordination repair with tests, not a new cooldown opt-out. The model-level cooldown ideas may be useful follow-ups, but they are broader provider policy and should not be required to stop duplicate visible replies.
