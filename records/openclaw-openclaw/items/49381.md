## Summary

Keep open: current main still has the Feishu error-closeout path that can finalize an errored streaming card before a recovered final, and the latest release has the same behavior. The linked candidate fix at https://github.com/openclaw/openclaw/pull/59771 remains open, conflicting, and proof-gated, so it is not a landed implementation to close against.

## What This Changes

Prevent one Feishu inbound message from producing duplicate final assistant replies when a rate-limited primary model fails over to a fallback model.

## Best Possible Solution

Land or replace a focused Feishu final-delivery fix that guarantees one visible final per inbound dispatch across failover/retry, with regression coverage for both streaming-card recovery and the streaming-off final path before adding any new cooldown configuration.

## Reproduction Assessment

Yes, at source level: the issue and follow-up comment provide sanitized Feishu logs with one inbound DM and multiple visible replies, and current main still closes streaming on reply errors before a recovered final can arrive. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

Yes for the central bug: the best first fix is a bounded Feishu final-delivery coordination repair, not a new cooldown opt-out. The model cooldown ideas may be useful later, but they should not be required to stop duplicate visible replies.
