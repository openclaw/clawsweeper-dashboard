## Summary

Keep open: current main and v2026.6.10 still have source-level paths where Feishu/final-dispatch can queue multiple final replies for one turn, and the related fixes are either closed unmerged or only cover adjacent subcases.

## What This Changes

Ensure Feishu direct-message turns do not emit duplicate final replies after a rate-limited primary model fails over to a fallback model.

## Best Possible Solution

Land or select a focused fix that prevents one Feishu inbound failover turn from producing more than one visible final reply, while preserving intentional distinct diagnostics and proving it with regression tests plus redacted live Feishu evidence.

## Reproduction Assessment

Yes, at source level: current main still sends every final replyResult entry through final delivery and Feishu tests permit errored/recovered finals to produce multiple visible streaming closes. I did not run a live Feishu/provider-failover replay in this read-only review.

## Solution Assessment

Unclear as stated: a stronger single-final guard is directionally right, but adding model-level cooldown bypasses or new config should be a separate maintainer product decision. The narrowest maintainable path is final-delivery coordination plus focused regression and live proof.
