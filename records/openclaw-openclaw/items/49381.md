## Summary

Keep open. Current main has Feishu duplicate-final mitigations, but source and tests still preserve paths where distinct final payloads or error-closeout recovery can produce multiple user-visible Feishu replies, and the linked candidate fix has not landed.

## What This Changes

Reported Feishu DMs can emit two final assistant replies when a rate-limited primary model fails over to a fallback model.

## Best Possible Solution

Land or replace #59771 for streaming recovery, then add failover-aware final coordination and regression coverage for non-streaming Feishu DMs without breaking intentional multi-part or media replies.

## Reproduction Assessment

Yes at source level: current Feishu tests assert distinct non-streaming finals and two streaming cards after error closeout. A live provider-rate-limit reproduction was not run.

## Solution Assessment

Unclear. #59771 is a plausible streaming-card fix, but the original non-streaming/failover case may need a generic final-coordination guard or a Feishu-specific guard that preserves legitimate multi-part replies.
