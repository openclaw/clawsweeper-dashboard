## Summary

Keep open: current main still has the Feishu reply-error closeout and final-delivery behavior implicated by the reports, and the open candidate fix has not merged.

## What This Changes

The issue asks OpenClaw to prevent a single Feishu inbound message from producing duplicate final assistant replies after model failover from a rate-limited primary model.

## Best Possible Solution

Land a narrow Feishu delivery fix that preserves or resets one reply surface through transient model errors, serializes error cleanup with retry/final delivery, and adds regression coverage for partial/error/recovered-final behavior including the non-streaming duplicate-final guard.
