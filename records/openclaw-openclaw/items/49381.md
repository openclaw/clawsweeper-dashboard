## Summary

Keep open: current main and v2026.6.6 still close a Feishu streaming card from the reply error path, while current tests still encode two visible streaming-card closes after an error followed by a recovered final. The directly linked candidate fix remains open, conflicting, and proof-gated, so there is no landed implementation to close against.

## What This Changes

The issue reports that one Feishu DM can emit duplicate final assistant replies when a rate-limited primary model fails over to a fallback model, including cases where streaming was disabled.

## Best Possible Solution

Land or replace a focused Feishu final-delivery coordination fix that guarantees one visible final per inbound dispatch across failover/retry, with regression coverage for streaming-card recovery and streaming-off final delivery; treat cooldown tuning as a separate follow-up.

## Reproduction Assessment

Yes, at source level: the issue and follow-up comments provide sanitized one-inbound-message logs, and current main still has code/tests that close one Feishu streaming card after an error before a later recovered final can close another. I did not run a live Feishu/provider-failover reproduction in this read-only review.

## Solution Assessment

Yes, the best first fix is a narrow Feishu final-delivery coordination repair with regression coverage, not a new cooldown opt-out. The cooldown policy ideas may be useful follow-ups, but they should not be required to stop duplicate visible replies.
