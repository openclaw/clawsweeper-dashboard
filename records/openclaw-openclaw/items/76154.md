## Summary

Keep open. The report is fresh and concrete, and current main has plausible post-release session-list/history mitigations, but there is still no current-main TTY proof that both Gateway and local launch modes no longer hit the immediate CPU loop; related #75289 overlaps on TUI CPU/RSS symptoms but has a different orphan-process trigger.

## What This Changes

Reports that `openclaw tui`, `openclaw chat`, and `openclaw terminal` immediately spin at 100%+ CPU and never become interactive on v2026.4.29.

## Best Possible Solution

Validate both launch modes on current main under a real TTY, then add a focused regression around the confirmed hot path or close after a shipped fix proves both modes are healthy.

## Reproduction Assessment

No high-confidence current-main reproduction is available. The issue gives concrete v2026.4.29 Linux commands and CPU/RSS observations, but this read-only review did not launch a current-main TTY and the startup/session paths changed after v2026.5.2.

## Solution Assessment

Unclear. The current-main session-list and history bounds are plausible mitigations, but the narrowest maintainable fix depends on live startup proof of the actual hot path.
