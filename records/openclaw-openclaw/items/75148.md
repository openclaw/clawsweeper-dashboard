## Summary

Keep open: this PR is still aimed at a real reply-delivery bug, but the branch does not yet align sandbox availability with the runtime policy key used by the embedded runner, so the visible fallback can still be suppressed when `message` is actually filtered out.

## What This Changes

The PR extends reply dispatch `message` tool availability checks to group, sandbox, and subagent policies and adds fallback regression coverage.

## Best Possible Solution

Compute dispatch-time `messageToolAvailable` from the same effective tool-policy inputs used by the embedded run, especially the runtime policy session key for sandbox policy, then keep focused regression coverage and changelog coverage with no channel default changes.

## Reproduction Assessment

Yes from source inspection. Configure `sandbox.mode: "non-main"` with sandbox `tools.allow` omitting `message`, dispatch an external direct turn through a main session whose runtime policy key derives a non-main peer key, and the PR can still classify `message` as available if it checks `acpDispatchSessionKey`.

## Solution Assessment

No as written. Extending availability to effective policy inputs is the right direction, but sandbox policy must use the runtime policy session key or a shared effective-availability helper, and the user-visible fix needs a changelog entry.
