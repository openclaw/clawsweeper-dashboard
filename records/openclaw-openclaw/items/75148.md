## Summary

Keep open: current main already covers the pre-resolved reply-mode and group-policy parts of this fallback, but the remaining sandbox-policy case is still not correctly handled by the PR as written. The branch’s sandbox availability check uses the dispatch session key instead of the runtime policy key that the embedded runner actually uses, so the visible fallback can still be suppressed in non-main sandboxed external direct turns.

## What This Changes

The PR extends dispatch-time visible-reply fallback coverage by adding message-tool availability checks and regression tests for group and sandbox policy removal.

## Best Possible Solution

Update the PR to compute dispatch-time `messageToolAvailable` from the same runtime policy session key and sandbox policy used by the embedded runner, then keep focused regression coverage and a single-line changelog entry.

## Reproduction Assessment

Yes, from source inspection. A non-main sandboxed external direct turn can have an unsandboxed dispatch key but a sandboxed runtime policy key, and the PR checks the former while the embedded runner filters tools using the latter.

## Solution Assessment

No as written. Extending effective tool availability is the right direction, but the sandbox policy check must use the runtime policy session key or a shared effective-availability helper, and the user-visible change needs changelog coverage.
