## Summary

Keep open: current main already covers the pre-resolved requested-mode fallback and group-policy fallback, but this PR branch still misses the sandbox runtime-policy case because it evaluates sandbox availability with the dispatch session key instead of the runtime policy session key used by embedded execution.

## What This Changes

The PR adds group, sandbox, and subagent tool-policy inputs to dispatch-time message-tool availability and adds group/sandbox fallback regression tests.

## Best Possible Solution

Keep the PR open for a narrow repair or replacement that computes dispatch-time `message` availability from the same runtime policy session key and sandbox tool policy used by embedded execution, while preserving the current main group/subagent/inherited-policy behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path. A direct turn can dispatch through a main key while the runtime policy session key resolves to a non-main sandboxed key whose sandbox `tools.allow` removes `message`, but this PR checks sandbox availability with the dispatch key.

## Solution Assessment

No as written. Extending availability to effective tool-policy inputs is the right direction, but sandbox availability must use the runtime policy session key or a shared effective-availability helper rather than `acpDispatchSessionKey`.
