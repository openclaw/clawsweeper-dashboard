## Summary

Keep open: current main has absorbed the requested-mode and group/subagent parts of this fallback path, but the PR still has a real sandbox-policy gap and its current sandbox implementation checks the dispatch session key instead of the runtime sandbox key.

## What This Changes

The PR branch adds group, sandbox, and subagent policy inputs to dispatch-time `message` tool availability and adds group/sandbox fallback regression tests.

## Best Possible Solution

Keep the PR open for a narrow repair or replacement that computes dispatch-time `message` availability from the same runtime policy session key and sandbox tool policy used by the embedded runner.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a direct external turn can dispatch through a main session while `resolveRuntimePolicySessionKey` derives a sandboxed non-main key whose sandbox `tools.allow` removes `message`. The PR checks the dispatch key, so it can still mark `message` available while the embedded runner filters it out.

## Solution Assessment

No as written. Extending availability to effective tool-policy inputs is the right direction, but sandbox availability must use the runtime policy session key or a shared effective-availability helper rather than `acpDispatchSessionKey`.
