## Summary

This is a valid focused bug fix, and the supplied real payload proof is useful, but the PR should stay open because the proposed wrapper still turns explicit `cacheRetention: "none"` back into a default Anthropic cache marker.

## What This Changes

The PR reorders cache-retention resolution for proxy providers, exports/reuses the Anthropic cache-control resolver, threads retention into the OpenRouter system-cache wrapper, and adds unit coverage for long/short/default markers.

## Best Possible Solution

Land a version that preserves the 1-hour TTL for explicit long retention and also treats explicit `none` as no OpenRouter Anthropic cache marker, with regression coverage for both cases.

## Reproduction Assessment

Yes. The current-main bug is source-reproducible through `resolveCacheRetention({ cacheRetention: "long" }, "openrouter", "openai-completions", ...)` returning before explicit config, and the PR supplies real before/after proxy dumps for the long-retention path.

## Solution Assessment

No, not yet. The direction is right for explicit long retention, but the wrapper must distinguish default/no resolved retention from explicit `cacheRetention: "none"` instead of falling back to a short cache marker.
