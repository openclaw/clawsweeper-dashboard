## Summary

Keep open. The reported OpenRouter Anthropic cache-retention bug is real and the supplied request-payload proof is useful, but this PR currently leaves the production plumbing broken: the caller passes a resolved retention string into a wrapper that reads an object property, so the intended `long`/`none` behavior is not reliably applied from `applyExtraParamsToAgent`.

## What This Changes

The branch updates OpenRouter Anthropic prompt-cache retention handling to preserve `ttl: "1h"`, skip markers for `cacheRetention: "none"`, and adds focused unit coverage.

## Best Possible Solution

Align the wrapper API and caller so the same resolved retention value that drives transport options also drives OpenRouter system/developer marker insertion, then add a production-path regression test for `long` and `none`.

## Reproduction Assessment

Yes. The source path on current main shows explicit OpenRouter Anthropic `cacheRetention` is dropped before marker insertion, and the PR includes captured before/after outbound payloads for the `long` case.

## Solution Assessment

No, not as submitted. The intended fix direction is narrow, but the current patch mixes a resolved string with a wrapper signature that expects an params object; either pass `ctx.effectiveExtraParams` consistently or make the wrapper accept the resolved retention value directly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass cacheRetention through the wrapper correctly:** `src/agents/pi-embedded-runner/extra-params.ts:700`
  - body: This call passes the resolved retention string into `createOpenRouterSystemCacheWrapper`, but the patched wrapper reads `extraParams?.cacheRetention`. A string has no `cacheRetention` property, so the production path falls back to default short markers and `cacheRetention: "none"` is not honored from `applyExtraParamsToAgent`; strict TS should also reject the string as `Record<string, unknown>`. Please either pass the effective params object or change the wrapper to accept the resolved retention value and cover this caller.
  - confidence: 0.93
