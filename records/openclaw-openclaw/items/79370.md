## Summary

Keep open: the PR addresses a real OpenRouter Anthropic cache-retention bug and includes convincing real payload proof, but the current head still lets explicit `cacheRetention` bypass the existing OpenRouter endpoint-class guard for custom proxy base URLs.

## What This Changes

The PR changes OpenRouter Anthropic prompt-cache plumbing so explicit `cacheRetention` and legacy TTL settings can control `cache_control` markers and adds focused regression tests.

## Best Possible Solution

Use one endpoint-class-aware retention decision for both pi-ai stream options and the OpenRouter system/developer marker wrapper, with regression coverage for native OpenRouter, custom proxy base URLs, `long`, `none`, and legacy TTL.

## Reproduction Assessment

Yes, source-reproducible. Current main drops explicit OpenRouter Anthropic retention before stream option insertion, and the PR-head blocker is visible because the patched resolver cannot see the base URL used by the endpoint-class gate.

## Solution Assessment

No, not as submitted. The fix direction is narrow and appropriate, but stream `cacheRetention` eligibility should reuse the same endpoint-class/baseUrl boundary as the OpenRouter marker wrapper.
