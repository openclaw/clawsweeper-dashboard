## Summary

Keep open: this is a real cache-retention bug and the direction is useful, but the latest patch is not mergeable because the production wrapper call no longer matches the new signature and the resolver now contradicts its own added provider-eligibility tests.

## What This Changes

The PR changes OpenRouter Anthropic prompt-cache retention resolution and cache_control marker insertion so explicit `cacheRetention` values can produce 1-hour TTL markers or suppress markers, with focused regression tests.

## Best Possible Solution

Thread the original effective extra params plus provider, model API, and model id into the OpenRouter cache wrapper after endpoint-class gating, keep resolver eligibility aligned with supported cache families/routes, and cover built-in OpenRouter, custom OpenRouter base URLs, non-Anthropic refs, arbitrary proxies, and explicit `none`.

## Reproduction Assessment

Yes. Current main is source-reproducible from `resolveCacheRetention` returning before explicit OpenRouter config is read and the marker helper hardcoding no-TTL markers; the PR discussion also includes before/after outbound request logs for the intended built-in route.

## Solution Assessment

No. The direction is right, but the latest patch does not carry the required prepared provider/model facts into the wrapper and broadens explicit retention before eligibility checks, so it is not the narrow maintainable fix yet.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Pass the real cache params into the wrapper:** `src/agents/pi-embedded-runner/extra-params.ts:694`
  - body: The new wrapper signature expects the original extra params plus provider/model facts, but this call passes the already-resolved retention string and omits the required facts. That should fail typecheck, and if emitted it makes the wrapper recompute retention from missing data so production OpenRouter requests still fall back to short markers. Pass `ctx.effectiveExtraParams`, `ctx.provider`, model API, and `ctx.modelId`, or change the wrapper to accept the resolved retention directly.
  - confidence: 0.97
- **[P2] Keep explicit retention behind provider eligibility:** `src/agents/pi-embedded-runner/prompt-cache-retention.ts:30-33`
  - body: This early return now honors explicit `cacheRetention` for every provider before checking cache-family or route eligibility. That contradicts the new tests expecting unsupported providers and OpenRouter non-Anthropic refs to return `undefined`, and it can pass cache params into providers where docs say the knob has no effect. Gate explicit values to the supported family/verified OpenRouter boundary before returning them.
  - confidence: 0.9
