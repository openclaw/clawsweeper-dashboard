## Summary

Keep open: the reported bug and fix direction are valid, and the supplied payload logs are useful proof, but the latest branch still has a production-path wrapper contract mismatch that keeps the new retention value from reaching `createOpenRouterSystemCacheWrapper` correctly.

## What This Changes

The PR changes OpenRouter Anthropic prompt-cache retention plumbing so explicit `cacheRetention` can add `ttl: "1h"`, suppress markers for `none`, and adds focused cache-marker tests.

## Best Possible Solution

Align the wrapper API and caller so the effective extra params drive OpenRouter system/developer marker insertion, preserve endpoint-class gating, and add a production-path regression for both `long` and `none`.

## Reproduction Assessment

Yes. Source inspection on current main shows explicit OpenRouter Anthropic `cacheRetention` is dropped before marker insertion, and the PR includes before/after outbound payload logs for the `long` case.

## Solution Assessment

No, not as submitted. The intended fix is narrow, but the caller passes a resolved retention string while the patched wrapper expects an extra-params object; pass `ctx.effectiveExtraParams` or make the wrapper accept the resolved value directly and cover that caller.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

- **[P1] Pass the extra params object to the cache wrapper:** `src/agents/pi-embedded-runner/extra-params.ts:700`

Please pass the effective params object or change the wrapper signature to accept the resolved retention value directly.
