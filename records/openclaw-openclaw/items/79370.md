## Summary

Keep open: the PR fixes a real OpenRouter Anthropic cache-retention bug and includes convincing real payload logs, but the latest head still misses one baseUrl-aware retention-resolution call used by prompt-cache metadata.

## What This Changes

The PR updates OpenRouter Anthropic prompt-cache retention plumbing and tests so explicit long, short, none, and legacy TTL settings affect pi-ai stream options and emitted system/developer cache_control markers.

## Best Possible Solution

Land the endpoint-class-aware retention fix after threading baseUrl through all retention-resolution paths and keeping the focused regression coverage around long, none, legacy TTL, and custom proxy behavior.

## Reproduction Assessment

Yes. Current main is source-reproducible from the resolver early return and hardcoded marker helper, and the contributor supplied before/after outbound OpenRouter payload logs for the same configured behavior.

## Solution Assessment

No. The approach is the right narrow fix, but the latest patch should also pass model.baseUrl to the run/attempt retention resolver so custom proxy configs are not misclassified in prompt-cache metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass baseUrl through the new endpoint gate:** `src/agents/pi-embedded-runner/prompt-cache-retention.ts:26`
  - body: `resolveCacheRetention` now relies on `baseUrl` to tell real OpenRouter routes from `provider: "openrouter"` pointed at a custom proxy. Thread the model baseUrl through that caller too.
