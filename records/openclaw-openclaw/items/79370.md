## Summary

Keep open: this is a real source-reproducible cache-retention bug and the PR is in the right area, but the latest head still has blocking correctness defects and the supplied proof does not establish the exact latest head.

## What This Changes

The PR changes OpenRouter Anthropic prompt-cache shaping so explicit `cacheRetention: "long"` can emit `ttl: "1h"`, `"none"` can skip marker insertion, and focused regression tests cover those paths.

## Best Possible Solution

Keep the fix in the agent OpenRouter cache boundary, pass the same validated extra params and endpoint facts through both pi-ai stream options and system-marker generation, then refresh exact-head proxy proof.

## Reproduction Assessment

Yes, source-reproducible. Current main returns before explicit OpenRouter `cacheRetention` is read and then hardcodes no-TTL system/developer markers; the PR discussion also includes before/after proxy logs for the intended route.

## Solution Assessment

No. The requested fix belongs in this boundary, but the latest patch passes a retention string to a wrapper that expects a params object and still does not align all retention forwarding with endpoint-class OpenRouter routing.
