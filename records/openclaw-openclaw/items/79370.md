## Summary

Keep open: the bug is real and the PR targets the right provider/cache boundary, but the latest diff is still incorrect because it passes the wrong value shape into the wrapper and does not consistently use endpoint-class OpenRouter facts for all retention forwarding.

## What This Changes

The branch changes OpenRouter Anthropic cache-retention plumbing so explicit `long` can emit `ttl: "1h"`, `none` can skip marker insertion, and regression tests cover resolver and wrapper payload paths.

## Best Possible Solution

Keep the focused fix, but pass retention data consistently and reuse endpoint-class route facts for both pi-ai cache options and system-marker generation, then refresh exact-head OpenRouter proxy proof.

## Reproduction Assessment

Yes. The issue is source-reproducible: current main returns before reading explicit OpenRouter `cacheRetention` and then hardcodes no-TTL system/developer markers; the PR discussion also provides before/after proxy-log context for the intended route.

## Solution Assessment

No. The PR is aimed at the right boundary, but the latest diff passes a retention string to a params-shaped wrapper and leaves endpoint-class retention forwarding inconsistent; the safer fix is to carry the validated params or resolved enum through one clear contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass a params object or enum consistently:** `src/agents/pi-embedded-runner/extra-params.ts:694`
  - body: `createOpenRouterSystemCacheWrapper` now reads `extraParams?.cacheRetention`, but this call passes `openRouterCacheRetention`, a resolved string/undefined from `resolveCacheRetention`. That does not match the declared `Record<string, unknown>` parameter and, if emitted, means `"long"`/`"none"` are read as no params so system/developer markers still fall back to short. Pass `ctx.effectiveExtraParams` or change the wrapper to accept the resolved enum.
  - confidence: 0.98
- **[P2] Use endpoint facts when resolving OpenRouter retention:** `src/agents/pi-embedded-runner/prompt-cache-retention.ts:42`
  - body: The new OpenRouter eligibility check only accepts `provider === "openrouter"`, while the wrapper and docs treat custom provider ids with `baseUrl` on `openrouter.ai` as verified OpenRouter routes. Because `createStreamFnWithExtraParams` uses this helper to forward `cacheRetention` to pi-ai, those custom-provider user/tool markers still lose `"long"` even though the direct wrapper test passes. Reuse endpoint-class/baseUrl facts for this helper too.
  - confidence: 0.88
