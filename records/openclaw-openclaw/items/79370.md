## Summary

Keep open: the reported bug is real and the patch is in the right area, but latest head still has blocking source-visible defects in its tests and still loses explicit retention for custom provider IDs routed to native OpenRouter.

## What This Changes

The PR threads explicit `cacheRetention` into OpenRouter Anthropic cache marker insertion, exports the marker resolver, and adds regression tests for long/short/none retention.

## Best Possible Solution

Keep the fix in the cache-retention/OpenRouter wrapper boundary, but resolve retention from the same endpoint-class facts used for marker insertion, repair the tests, and refresh proof for the final head.

## Reproduction Assessment

Yes, source-reproducible. Current main returns before explicit OpenRouter `cacheRetention` is read and then hardcodes no-TTL system/developer cache markers, matching the supplied before/after proxy logs for the intended route.

## Solution Assessment

No. The intended fix boundary is right, but the current patch is not the best form because it does not align retention eligibility with endpoint-class OpenRouter routing and its new tests have blocking defects.
