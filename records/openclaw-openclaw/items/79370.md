## Summary

Keep open: the PR fixes the built-in OpenRouter Anthropic `cacheRetention` path, but the latest diff still misses the current endpoint-class route policy for native OpenRouter hosts behind custom provider ids, so explicit `long` or `none` can still be lost on a supported route.

## What This Changes

The PR threads `cacheRetention` into OpenRouter Anthropic cache marker insertion, preserves `ttl: "1h"` for long retention, skips marker insertion for `none`, and adds focused regression tests.

## Best Possible Solution

Make OpenRouter cache-retention resolution share the same verified endpoint/model boundary as marker insertion, then cover built-in OpenRouter, custom provider ids on native OpenRouter, arbitrary proxy URLs, non-Anthropic OpenRouter refs, and explicit `none`.

## Reproduction Assessment

Yes. Current main is source-reproducible from `resolveCacheRetention` returning before explicit OpenRouter config and the marker helper hardcoding no TTL; the PR also supplies before/after outbound request logs for the built-in OpenRouter route.

## Solution Assessment

No. The direction is right, but the latest patch still resolves retention using provider identity while the supported marker route is endpoint-class based, so the fix should reuse or pass that prepared route fact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor retention on custom OpenRouter routes:** `src/agents/pi-embedded-runner/prompt-cache-retention.ts:35`
  - body: This gate still keys explicit OpenRouter retention to the provider id, while `createOpenRouterSystemCacheWrapper` applies to native OpenRouter hosts behind custom provider ids. For a custom provider with `baseUrl: "https://openrouter.ai/api/v1"`, Anthropic model id, and `cacheRetention: "long"` or `"none"`, the wrapper receives `undefined` and emits the default short marker instead of the requested TTL/opt-out. Please reuse the endpoint-class boundary here or resolve the retention inside the wrapper, and add coverage for that route.
