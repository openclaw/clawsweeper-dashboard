## Summary

Keep this PR open, but not merge-ready: it addresses a real OpenRouter Anthropic cache-retention bug and includes sufficient real request-log proof, yet the current patch still widens explicit cache retention beyond supported provider families and bypasses the existing thinking-block cache marker sanitizer when retention is `none`.

## What This Changes

The PR threads `cacheRetention` into OpenRouter Anthropic cache marker insertion, preserves `ttl: "1h"` for long retention, skips marker insertion for `none`, and adds focused regression tests.

## Best Possible Solution

Land a narrow cache fix that honors explicit OpenRouter Anthropic retention only on verified supported routes, preserves unsupported-provider no-ops, keeps the thinking-block cleanup sanitizer active for `none`, and adds regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Current main is source-reproducible through the early `resolveCacheRetention` return for OpenRouter Anthropic models and the hardcoded no-TTL OpenRouter marker helper; the PR body also includes before/after outbound request logs from a real setup.

## Solution Assessment

No, not as currently submitted. The intended fix is the right direction, but the implementation should scope explicit retention to supported OpenRouter Anthropic routes and keep the `none` path's sanitizer instead of returning the unpatched stream.
