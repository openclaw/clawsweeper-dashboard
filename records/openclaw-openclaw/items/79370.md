## Summary

Keep open: current main still has the reported cache-retention bug and this PR contains useful proof and focused changes, but the latest diff still resolves OpenRouter retention by provider id while the supported marker route is endpoint-class based.

## What This Changes

The PR threads explicit `cacheRetention` into OpenRouter Anthropic cache marker shaping so `long` can add `ttl: "1h"` and `none` can suppress new markers, with regression tests.

## Best Possible Solution

Keep the PR focused, but make retention resolution use the same verified OpenRouter endpoint/model boundary as marker insertion and add coverage for built-in OpenRouter, custom provider ids on native OpenRouter, arbitrary proxy URLs, non-Anthropic refs, and explicit `none`.

## Reproduction Assessment

Yes. Current main is source-reproducible because explicit OpenRouter Anthropic `cacheRetention` returns `undefined` before config is read and the marker helper hardcodes no TTL; the PR body also supplies before/after outbound request logs for the built-in route.

## Solution Assessment

No. The PR direction is right, but the latest patch still uses provider identity for retention while the supported OpenRouter marker path is endpoint-class based, so custom provider ids on native OpenRouter keep the wrong default behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor retention on custom OpenRouter routes:** `src/agents/pi-embedded-runner/prompt-cache-retention.ts:35`

In that supported route, explicit `long` or `none` resolves `undefined`, so the wrapper emits the default short marker instead of the requested TTL or opt-out. Please derive retention from the same endpoint-class boundary, or resolve it inside the wrapper, and add coverage for that route.
