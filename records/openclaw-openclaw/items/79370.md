## Summary

Needs changes before merge: the bug is real and the direction is useful, but current head miswires the new wrapper arguments and widens cache-retention resolution beyond supported provider families.

## What This Changes

The PR updates OpenRouter Anthropic prompt-cache retention plumbing and tests so `cacheRetention: "long"` can emit `ttl: "1h"` and `"none"` can suppress markers.

## Best Possible Solution

Keep the route-gated OpenRouter wrapper, pass the original effective extra params plus provider/model facts or a prepared route fact into it, and keep `resolveCacheRetention` scoped to cache-eligible families except for the verified OpenRouter Anthropic route.

## Reproduction Assessment

Yes. Source inspection on current main shows explicit OpenRouter `cacheRetention` is dropped before config is read and system/developer markers are hardcoded short; the contributor also supplied before/after OpenRouter request logs for the built-in route.

## Solution Assessment

No. The current PR direction is right, but the latest head misthreads wrapper arguments and widens shared retention resolution, so the narrow maintainable fix should reuse the verified OpenRouter route boundary and preserve the resolver contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass the real extra params to the cache wrapper:** `src/agents/pi-embedded-runner/extra-params.ts:692-695`
  - body: The production caller now passes `openRouterCacheRetention` as the wrapper's `extraParams` argument and does not pass `provider`, `modelApi`, or `modelId`. With the new signature this is a type mismatch, and at runtime the wrapper's second `resolveCacheRetention(extraParams, provider, ...)` sees no config/provider facts, so `long` and `none` fall back to the default short marker in the real `applyExtraParamsToAgent` path. Pass `ctx.effectiveExtraParams` plus the route/model facts, or resolve retention once inside the already route-gated wrapper.
  - confidence: 0.96
- **[P2] Keep explicit retention behind eligibility checks:** `src/agents/pi-embedded-runner/prompt-cache-retention.ts:30-33`
  - body: This early return accepts explicit `cacheRetention` for any provider before the family/Google eligibility checks. That contradicts the new non-Anthropic Bedrock test and the documented contract that unsupported providers ignore this cache mode, and it can leak cacheRetention into generic stream options for unsupported OpenAI-compatible providers. Handle OpenRouter after endpoint/model verification instead of widening the shared resolver.
  - confidence: 0.9
