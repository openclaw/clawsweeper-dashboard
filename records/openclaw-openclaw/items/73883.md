## Summary

Keep open. Current main does not implement the requested archive opt-in, but this PR is not merge-ready because the proposed archive lookup misses the normal reset lifecycle that rotates to a new session id, and it still needs sequence-metadata, docs, changelog, and reset-archive visibility review.

## What This Changes

The PR adds an optional `includeArchived` parameter to `chat.history` and `sessions_history`, adds reset-archive discovery and aggregation helpers, adds targeted tests, and regenerates Swift protocol models.

## Best Possible Solution

Keep the read-only opt-in direction only if it follows actual reset lineage or recorded archive references, preserves continuous transcript metadata, keeps existing visibility/redaction limits, documents the public behavior, and lands in coordination with the broader archive-recovery work.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: run a normal `sessions.reset`, observe the store entry gets a new session id while the archive is named for the old id, then `chat.history` with this PR's `includeArchived` scan would look for archives under the new id and miss the old transcript. The duplicate-seq issue is also reproducible with two reset archive files in the PR's own multi-archive shape.

## Solution Assessment

No. A read-only explicit archive view is a plausible solution direction, but the current implementation is not the best merge form because it scans by the current session id instead of reset lineage, restarts archive sequence metadata per segment, and leaves public docs/changelog and privacy semantics unresolved.

- **[P1] Use reset lineage for archive lookup:** `src/gateway/session-utils.fs.ts:287`
  - body: `sessions.reset` creates a fresh `entry.sessionId` and archives the previous transcript as `<oldSessionId>.jsonl.reset.<ts>`, but this helper only scans for `<currentSessionId>.jsonl.reset.*`. After a normal reset, `includeArchived=true` on the session key will still miss the archived context this feature is meant to recover. Use recorded predecessor/archive references or a lineage index instead of the current id alone.
  - confidence: 0.91

With two archive segments, multiple messages can carry the same `__openclaw.seq`, which makes cursors, diagnostics, and UI keys ambiguous. Thread a shared counter through all archive segments and cover it in the multi-archive test.
  - confidence: 0.86
