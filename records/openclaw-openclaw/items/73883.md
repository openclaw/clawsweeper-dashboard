## Summary

Keep open. Current main does not implement `includeArchived`, but this PR is not mergeable as-is because its archive path diverges from current bounded `chat.history`, duplicates transcript sequence metadata across archive segments, omits required public docs/changelog coverage, and exposes reset archives through an agent tool without maintainer-approved privacy semantics.

## What This Changes

The PR adds optional `includeArchived` support to `chat.history` and `sessions_history`, with reset-archive discovery/aggregation helpers, tests, and regenerated Swift protocol models.

## Best Possible Solution

Land a coordinated additive archive-read feature after rebasing over current bounded `chat.history`, keeping reset archives archived while exposing only bounded, documented, opt-in recovery views with approved tool privacy semantics.

## Reproduction Assessment

Yes. Static inspection is enough: current main rejects `includeArchived`, while the PR diff shows a full archive reader path and helper-local sequence counter that can duplicate `__openclaw.seq` when two archives are chained.

## Solution Assessment

No. A read-only opt-in is a plausible direction, but the mergeable solution should preserve the bounded async reader path, maintain continuous metadata, document the public contract, add a changelog entry, and get maintainer approval before agent tools can request reset archives.
