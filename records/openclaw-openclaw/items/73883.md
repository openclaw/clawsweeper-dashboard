## Summary

Keep open. Current main still lacks the requested archive selector, so this PR is not obsolete, but it is not merge-ready: it is stale against the current bounded chat.history reader, creates ambiguous archive sequence metadata, omits required changelog/docs coverage, and changes reset-archive exposure semantics for agent tools.

## What This Changes

The PR adds optional includeArchived support to chat.history and sessions_history, reset-archive discovery and aggregation helpers with tests, and regenerated Swift protocol models.

## Best Possible Solution

Land a coordinated, additive archive-read feature only after maintainers approve reset-archive visibility semantics, rebasing it over current bounded chat.history behavior with continuous metadata, public docs, changelog coverage, regenerated clients, and clear interaction with #45003 and sibling archive-recovery PRs.

## Reproduction Assessment

Yes. Static inspection of current main shows includeArchived is rejected by the protocol schema and cannot be forwarded by sessions_history; the PR-specific seq issue is reproducible with two reset archives because readArchivedTranscriptMessages resets messageSeq per archive file.

## Solution Assessment

Unclear. A read-only opt-in is a plausible direction, but this is not the best merge form until it preserves the current bounded reader path, fixes continuous seq metadata, documents the public contract, and gets maintainer approval for agent-tool archive visibility.
