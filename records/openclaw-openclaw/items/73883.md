## Summary

Keep this PR open: the requested archive-history opt-in is still not implemented on current main and remains useful, but the branch has correctness and approval blockers before merge.

## What This Changes

The PR adds an optional `includeArchived` flag to `chat.history` and `sessions_history`, plus reset-archive discovery/aggregation helpers, tests, and regenerated Swift protocol models.

## Best Possible Solution

Land a coordinated additive archive-history mode that uses bounded async transcript reads, stable segment metadata, public docs, and explicit approval for reset-archive visibility semantics.

## Reproduction Assessment

Yes. Static source inspection is enough: current main rejects `includeArchived` and uses bounded `readRecentSessionMessagesAsync`, while the PR branch routes opt-in archive reads through full sync readers and resets sequence counters per archive.

## Solution Assessment

No, not as proposed. A read-only opt-in is plausible, but this implementation should be rebased onto the bounded async reader path, preserve monotonic metadata, document the contract, and get reset-archive visibility approval.

  - body: Current `chat.history` reads only a bounded recent window before projection, but this opt-in path switches to full transcript/archive readers.   - confidence: 0.86
