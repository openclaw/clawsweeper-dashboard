## Summary

Keep open. Current main still lacks the requested `includeArchived` history parameter, and this PR is useful source work for the archive-history gap, but the proposed patch needs maintainer approval for reset-archive visibility and rework for bounded reads, stable metadata, docs, and changelog before merge.

## What This Changes

The PR adds an optional `includeArchived` flag to `chat.history` and `sessions_history`, plus reset-archive discovery and aggregation helpers with Swift protocol regeneration.

## Best Possible Solution

Land a coordinated additive archive-history opt-in that preserves default active-history behavior, bounds archive reads before projection, keeps metadata stable across segments, updates generated clients/docs/changelog, and has maintainer-approved reset-archive visibility semantics.

## Reproduction Assessment

Yes. Static inspection is enough: current main rejects `includeArchived`, while the PR diff shows the new archive read path and its bounded-read and per-archive sequence issues.

## Solution Assessment

No, not as currently proposed. A read-only opt-in can be the right direction, but it should be rebased onto the bounded async history path and approved for reset-archive privacy semantics before merge.
