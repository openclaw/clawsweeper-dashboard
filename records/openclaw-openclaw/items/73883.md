## Summary

Keep open. Current main still lacks an `includeArchived` selector for `chat.history` and `sessions_history`, so the branch is not obsolete, but the proposed merge form remains incorrect against current main because it bypasses the bounded history reader, duplicates archive sequence metadata, misses explicit-session-file archive shapes, and needs public docs/changelog plus maintainer privacy approval for agent-tool access to reset archives.

## What This Changes

The PR adds optional `includeArchived` support to `chat.history` and `sessions_history`, discovers `.reset` transcript archives, aggregates archive segments with synthetic reset markers, regenerates Swift protocol models, and adds unit tests.

## Best Possible Solution

Land a coordinated additive archive-read feature after maintainer approval, rebased over the current bounded `chat.history` path, with continuous metadata, support for canonical and explicit `sessionFile` reset archives, regenerated clients, tests, public docs, and changelog coverage aligned with the rest of the #45003 archive-recovery work.

## Reproduction Assessment

Yes. Static current-main inspection shows `includeArchived` is absent from the schema, handler, generated Swift models, and `sessions_history` tool plumbing; the PR-specific regressions are reproducible by reading the diff against current `readRecentSessionMessages` behavior and by chaining multiple archive files or a topic/custom archived `sessionFile`.

## Solution Assessment

Unclear. A read-only opt-in is a plausible product direction, but this PR is not the best merge form until it preserves bounded reads, covers all reset archive filename shapes, keeps sequence metadata monotonic, and gets maintainer approval for model-facing reset archive visibility.
