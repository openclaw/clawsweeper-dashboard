## Summary

Keep open. Current main still lacks an `includeArchived` selector, but this PR is not mergeable as-is because it bypasses current bounded `chat.history` reads, duplicates transcript sequence metadata across archive segments, lacks the required public docs/changelog coverage, and needs maintainer approval before reset archives are exposed through a model-facing tool.

## What This Changes

The PR adds optional `includeArchived` support to `chat.history` and `sessions_history`, plus reset-archive discovery/aggregation helpers, tests, and regenerated Swift protocol models.

## Best Possible Solution

Land a coordinated additive archive-read feature after rebasing over the bounded reader path, preserving continuous metadata, documenting the public contract, adding a changelog entry, and explicitly approving agent-tool archive visibility.

## Reproduction Assessment

Yes. Static inspection is enough: current main rejects `includeArchived`, while the PR diff shows an unbounded full transcript/archive read path and per-archive `messageSeq` reset that reproduces duplicate sequence metadata with two reset archives.

## Solution Assessment

No. A read-only opt-in archive view is plausible, but the mergeable solution should keep bounded history reads, use monotonic metadata across all returned segments, and make reset-archive privacy semantics explicit before exposing the flag to `sessions_history`.
