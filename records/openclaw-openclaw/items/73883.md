## Summary

Keep open: current main does not already implement the archive-history opt-in, and this PR is recent and tied to active archive-history work. It is not merge-ready because the branch bypasses current bounded transcript reads, duplicates archive sequence metadata, lacks public docs and inspectable real behavior proof, and needs explicit approval for model-facing reset-archive visibility.

## What This Changes

The PR adds an optional `includeArchived` parameter to `chat.history` and `sessions_history`, reset-archive discovery and aggregation helpers, focused tests, and regenerated Swift protocol models.

## Best Possible Solution

Land a coordinated archive-history mode that keeps bounded async reads, stable segment metadata, public docs, generated clients, regression tests, and an approved reset-archive visibility contract.

## Reproduction Assessment

Yes. Static source inspection is enough for the review findings: current main rejects `includeArchived` and uses bounded `readRecentSessionMessagesAsync`, while the PR branch routes opt-in archive reads through full sync readers and resets sequence counters per archive.

## Solution Assessment

No, not as proposed. A read-only opt-in is plausible, but the implementation should preserve the bounded async read contract, keep monotonic segment metadata, document the public contract, and get approval for reset-archive visibility.
