## Summary

Keep this PR open for maintainer review, but do not merge as-is. Current main already solved and shipped the deferred cron timeout half more deeply, while the PR branch arms the timer before the embedded runner's lane wait; the remaining abort/fallback behavior still overlaps active failover-policy work and needs owner direction rather than this stale combined patch.

## What This Changes

The PR changes isolated cron agent-turn execution to use per-attempt abort controllers, defer cron timeout arming through an execution-start callback, and add cron timeout regression tests.

## Best Possible Solution

Keep the shipped deferred-timeout implementation on main, then resolve the remaining fallback-abort problem through a rebased, narrower change in the shared model-fallback or embedded-runner failover contract that distinguishes retryable provider timeouts from terminal cron-budget or user/client aborts.

## Reproduction Assessment

Partial yes: current tests reproduce the queue-wait timeout half and current code inspection shows the shared abort signal still flows through fallback candidates. The poisoned-fallback half still needs a focused current-main regression test or live repro before changing terminal abort policy.

## Solution Assessment

No: this PR is no longer the best direct fix because its deferred-timeout callback fires before the runner lane wait and targets an older execution shape. The safer solution is to drop the already-shipped timeout changes and handle the remaining abort semantics in the current shared failover path with contract tests.
