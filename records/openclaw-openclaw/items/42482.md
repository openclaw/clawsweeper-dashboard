## Summary

Keep this PR open for maintainer follow-up. Current main already has the deferred isolated-cron timeout behavior, but the PR still contains useful abort/fallback context and is now stale against the current run-executor plus shared failover policy; a direct merge would also not correctly defer the timeout past runner lane wait on the submitted branch.

## What This Changes

The PR changes isolated cron agent-turn execution to use per-attempt abort controllers for embedded fallback attempts, defer cron timeout arming through an execution-start callback, and add cron timeout regression tests.

## Best Possible Solution

Keep the shipped deferred-timeout implementation, then land a narrower rebased fix or replacement that targets the current run-executor/shared embedded-runner failover contract and explicitly separates retryable per-attempt provider timeouts from terminal cron-budget or user aborts.

## Reproduction Assessment

Partial yes: current tests reproduce and cover the queue-wait timeout half, and current code inspection shows the remaining shared-abort/fallback policy gap. The fallback-poisoning half still needs a focused current-main regression test or live repro to prove the intended retryable timeout behavior without weakening terminal aborts.

## Solution Assessment

No: the submitted patch is no longer the best direct solution because it arms the deferred timer too early and targets an older execution shape. The safer path is to drop the already-shipped timeout changes and implement the remaining abort semantics in the current run-executor or shared failover policy with contract tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Defer the timer until the lane actually starts:** `src/cron/service/timer.ts:1167`
  - body: The new callback is invoked immediately before `runIsolatedAgentJob`, but the queue wait called out in the PR happens inside that runner path. A cron job can still burn its timeout while waiting for the embedded lane, and the new test resolves immediately rather than simulating that wait. Pass the callback through to the isolated/embedded runner and invoke it after lane acquisition.
