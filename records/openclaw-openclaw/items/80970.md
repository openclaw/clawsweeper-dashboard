## Summary

Keep open: current main and the latest v2026.5.12 release still let `sessions cleanup --fix-missing` pass a key-shaped stored `sessionId` into strict transcript-id validation, so malformed legacy rows can still crash cleanup before they are reported or pruned.

## What This Changes

The issue reports that `openclaw sessions cleanup` crashes when a session-store row stores a session key such as `agent:main:main` in the `sessionId` field.

## Best Possible Solution

Keep transcript filename validation strict, but make cleanup-specific missing-transcript pruning treat invalid or key-shaped stored `sessionId` values as malformed store rows that can be reported or pruned without throwing.

## Reproduction Assessment

Yes, at source level. A row whose stored `sessionId` is `agent:main:main` reaches `resolveSessionFilePath(entry.sessionId, ...)` during `--fix-missing`, then `validateSessionId` rejects the colon-containing value before cleanup can continue.

## Solution Assessment

Yes. The best fix is narrow cleanup-specific malformed-row handling, not relaxing `validateSessionId` or allowing session keys as transcript IDs globally.
