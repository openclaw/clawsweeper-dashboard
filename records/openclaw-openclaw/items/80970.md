## Summary

Keep open: current main still lets cleanup pass a stored key-shaped `sessionId` into strict transcript-id validation, so malformed or legacy rows such as `agent:main:main` can throw before cleanup can report or prune them.

## What This Changes

The issue asks `openclaw sessions cleanup` to handle key-shaped or malformed stored session entries without crashing during transcript-path cleanup.

## Best Possible Solution

Keep transcript filename validation strict, and make cleanup treat invalid or key-shaped stored `sessionId` values as cleanup data errors: use a safe persisted `sessionFile` when possible, otherwise report or prune the malformed row under `--fix-missing` without throwing.

## Reproduction Assessment

Yes, at source level. A store entry with `sessionId: "agent:main:main"` and no usable `sessionFile` reaches `resolveSessionFilePath(entry.sessionId, ...)` in `pruneMissingTranscriptEntries`, then `validateSessionId` rejects the colon-containing value.

## Solution Assessment

Yes. The narrow maintainable direction is cleanup-specific invalid-entry handling, not accepting colon session keys in `SAFE_SESSION_ID_RE` globally; the adjacent cleanup fallback PR at https://github.com/openclaw/openclaw/pull/69954 should be coordinated but does not resolve this crash on main.
