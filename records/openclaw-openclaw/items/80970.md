## Summary

Keep open: current main still lets `sessions cleanup --fix-missing` pass a key-shaped stored `sessionId` into strict transcript-id validation, so malformed legacy rows can crash cleanup before they can be reported or pruned. Prior same-scope fix PRs were closed unmerged, and the remaining open related PR covers stale `sessionFile` fallback rather than this invalid-id crash.

## What This Changes

The issue reports that `openclaw sessions cleanup --fix-missing` crashes when a malformed store row has a key-shaped `sessionId` such as `agent:main:main`.

## Best Possible Solution

Add cleanup-specific handling that treats unresolvable stored session IDs as corrupt or missing metadata under `--fix-missing`, keeps transcript filename validation strict everywhere else, and covers dry-run plus enforced cleanup with regression tests.

## Reproduction Assessment

Yes, at source level. A store row with `sessionId: "agent:main:main"` and no usable `sessionFile` reaches `resolveSessionFilePath(...)` from `pruneMissingTranscriptEntries`, then `validateSessionId` rejects the colon-containing value.

## Solution Assessment

Yes. The narrow maintainable fix is cleanup-specific recovery for malformed legacy rows, not relaxing the global transcript-id validator to accept session keys.
