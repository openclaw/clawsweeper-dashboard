## Summary

Keep open: current main and the latest tagged release still let `sessions cleanup --fix-missing` pass a key-shaped stored `sessionId` into strict transcript-id validation, so malformed legacy rows can crash cleanup before they can be pruned.

## What This Changes

The issue asks for `openclaw sessions cleanup` to tolerate legacy store rows whose stored `sessionId` is a session key such as `agent:main:main` instead of throwing.

## Best Possible Solution

Keep transcript filename validation strict, but make the `--fix-missing` cleanup path treat invalid or key-shaped stored `sessionId` values as corrupt missing-store rows that can be reported and pruned without throwing.

## Reproduction Assessment

Yes, at source level. A store row with `sessionId: "agent:main:main"` reaches `resolveSessionFilePath`, then falls through to `validateSessionId`, which rejects the colon-containing value before cleanup can continue.

## Solution Assessment

Yes. The narrow maintainable fix is cleanup-specific error handling for malformed legacy rows, not broadening `SAFE_SESSION_ID_RE` or accepting session keys as transcript IDs globally.
