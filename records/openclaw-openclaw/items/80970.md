## Summary

Keep open: current main still has a source-reproducible cleanup crash when a key-shaped stored `sessionId` is retained because a corrupt or unsafe `sessionFile` is present, then cleanup falls through to strict transcript-id validation. Prior same-scope fix PRs were closed unmerged, and the remaining open related PR covers stale valid transcript fallback rather than this invalid-id crash.

## What This Changes

The issue reports that `openclaw sessions cleanup --fix-missing` can crash when a session-store row stores a colon-containing session key as `sessionId` instead of a transcript id.

## Best Possible Solution

Add cleanup-specific recovery for unresolvable stored session IDs under `--fix-missing`, keep global transcript filename validation strict, and cover dry-run plus applied cleanup with focused regression tests.

## Reproduction Assessment

Yes, at source level. A row with `sessionId: "agent:main:main"` and a corrupt or unsafe `sessionFile` is retained by store normalization, reaches `pruneMissingTranscriptEntries`, and then `resolveSessionFilePath` falls through into `validateSessionId`.

## Solution Assessment

Yes. The narrow maintainable fix is cleanup-specific handling that treats malformed legacy metadata as missing or corrupt during `--fix-missing`, not relaxing `validateSessionId` to accept session keys everywhere.
