## Summary

Keep open: current main still passes key-shaped stored session IDs into strict transcript path validation during `sessions cleanup --fix-missing`, and the same-scope fix PRs are closed and unmerged, so the cleanup crash remains actionable.

## What This Changes

`openclaw sessions cleanup --fix-missing` should handle legacy store rows whose stored `sessionId` is a session key such as `agent:main:main` without throwing.

## Best Possible Solution

Keep transcript filename validation strict, but make `pruneMissingTranscriptEntries` treat invalid or key-shaped stored `sessionId` values as malformed cleanup rows: recover from a safe existing `sessionFile` when possible, otherwise report and prune them under `--fix-missing` with dry-run and applied regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: a store row with `sessionId: "agent:main:main"` and no usable `sessionFile` reaches `resolveSessionFilePath` during `--fix-missing`, then strict validation rejects the colon-containing value.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is cleanup-specific handling for malformed store rows, not relaxing `SAFE_SESSION_ID_RE` globally or skipping the bad rows without pruning them.
