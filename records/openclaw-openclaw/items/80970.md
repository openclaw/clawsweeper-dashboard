## Summary

Keep open: current main still has a source-reproducible `sessions cleanup --fix-missing` path where a malformed or legacy store row can pass a session-store key such as `agent:main:main` into strict transcript session-id validation and throw before cleanup can prune it. The related open PRs I found are adjacent cleanup work, but they do not specifically close this invalid session-key crash.

## What This Changes

The issue asks for `openclaw sessions cleanup` to handle session-store keys such as `agent:main:main` or malformed legacy `sessionId` values without crashing while pruning missing transcripts.

## Best Possible Solution

Keep transcript filename validation strict, but make cleanup treat malformed legacy store rows as cleanup data problems: skip or prune them safely under `--fix-missing` without letting invalid store keys reach strict transcript-id validation.

## Reproduction Assessment

Yes, at source level: a store entry keyed `agent:main:main` with `sessionId: "agent:main:main"` reaches `pruneMissingTranscriptEntries`, which calls `resolveSessionFilePath` and then `validateSessionId`. I did not establish a live CLI reproduction in this read-only checkout.

## Solution Assessment

Yes. The narrow maintainable fix is cleanup-specific handling of malformed or legacy store entries, not broadening `SAFE_SESSION_ID_RE` to allow colons in generated transcript filenames.
