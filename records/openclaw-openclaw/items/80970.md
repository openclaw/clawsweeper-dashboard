## Summary

Keep open: current main still lets `sessions cleanup --fix-missing` feed a key-shaped `sessionId` such as `agent:main:main` into strict transcript-id validation, so malformed legacy store rows can crash cleanup before they can be pruned or reported.

## What This Changes

Make `openclaw sessions cleanup` handle session-store entries whose `sessionId` is a session key such as `agent:main:main` instead of crashing in transcript-id validation.

## Best Possible Solution

Keep transcript filename validation strict, but make cleanup treat invalid or key-shaped stored `sessionId` values as cleanup data errors: recover from a valid `sessionFile` when safe, otherwise report or prune the malformed row under `--fix-missing` without throwing, with regression coverage.

## Reproduction Assessment

Yes, at source level. A store entry whose stored `sessionId` is `agent:main:main` reaches `resolveSessionFilePath(entry.sessionId, ...)` during `--fix-missing`, then `validateSessionId` rejects the colon-containing value before cleanup can continue.

## Solution Assessment

Yes. The narrow maintainable fix is cleanup-specific handling for malformed legacy store rows, not relaxing transcript filename validation to accept session keys globally.
