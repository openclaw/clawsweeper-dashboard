## Summary

Keep open: current main still has the reported completion-announce failure path for an active requester session whose wake attempt returns `not_streaming`, and the docs/QA scenario define `sessions_yield` completion delivery as push-based expected behavior.

## What This Changes

The issue requests that subagent completion announcements reliably reach the requester session after `sessions_spawn` plus `sessions_yield` instead of failing with `active requester session could not be woken`.

## Best Possible Solution

Repair the subagent completion delivery contract so a yielded or non-streaming requester receives the completion through the requester-agent handoff or a durable follow-up queue, with regression coverage for `not_streaming` and yielded-session paths.

## Reproduction Assessment

Yes, by source inspection: an active requester with a non-streaming embedded run reaches `not_streaming`, returns `active requester session could not be woken`, retries through the registry, and never reaches the requester-agent handoff. I did not execute a live channel repro in this read-only pass.

## Solution Assessment

Yes, the narrow fix is in the subagent announce delivery path, not a queue-mode workaround: fall through to the requester-agent completion handoff when wake/steer cannot accept `not_streaming` or equivalent idle states, while keeping raw external child-output bypasses blocked.
