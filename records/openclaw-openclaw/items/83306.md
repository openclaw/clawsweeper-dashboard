## Summary

Keep open: the Browserbase driver direction is useful and matches the linked Browserbase problem, but the current PR is not merge-ready because the documented profile is still rejected by runtime schema, several shared browser routes still receive an empty CDP URL, signed connect URL material can leak in an error, and no inspectable real Browserbase proof is attached.

## What This Changes

The PR adds a Browserbase browser profile driver that fetches a fresh Browserbase CDP connect URL per attach, updates related browser types/capabilities/docs/tests, and raises remote CDP timeout defaults.

## Best Possible Solution

Land a Browserbase driver only after runtime schema/help/docs/tests, centralized CDP URL resolution, redacted error handling, credential-surface approval, and redacted real Browserbase proof are aligned; otherwise keep the broader work in https://github.com/openclaw/openclaw/issues/43455.

## Reproduction Assessment

Yes for the PR review blockers: the PR head still rejects the documented profile in runtime schema and still passes the unresolved empty cdpUrl through shared route helpers. No live Browserbase flow was reproduced because the PR has no inspectable real-environment artifact.

## Solution Assessment

No, not as implemented: per-attach Browserbase connectUrl resolution is a plausible direction, but the patch misses the canonical runtime schema and shared route boundary and leaks signed URL fragments. A safer solution centralizes URL resolution, aligns schema/docs/tests, redacts sensitive errors, and includes redacted live proof.
