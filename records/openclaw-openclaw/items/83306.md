## Summary

Keep this PR open, but it is not merge-ready: the Browserbase keep-alive driver is a useful direction, while the current patch still leaves the documented profile rejected by runtime config validation, leaves several Playwright routes on the empty placeholder CDP URL, exposes a possible signing-key diagnostic leak, and has no inspectable real Browserbase proof.

## What This Changes

The PR adds a Browserbase keep-alive browser profile driver with per-attach Sessions API connectUrl resolution, related browser profile types/capabilities/docs/tests, and higher remote-CDP timeout defaults.

## Best Possible Solution

Land a smaller integrated Browserbase profile implementation that updates the runtime schema/help/docs together, resolves fresh CDP URLs at one shared attach boundary, redacts URL-bearing diagnostics, and includes focused regression coverage plus redacted real Browserbase proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: the PR head still rejects the documented profile in runtime schema validation, and shared Playwright route paths still receive profileCtx.profile.cdpUrl even though Browserbase profiles resolve it to an empty string. I did not run code or Browserbase live because this review is read-only.

## Solution Assessment

No: per-attach Browserbase connectUrl resolution is the right direction, but this implementation is not central enough, does not update the canonical runtime config contract, and leaks URL material in one diagnostic path. The safer fix is to align schema/help/docs and resolve the live CDP URL once at the shared route/attach boundary.
