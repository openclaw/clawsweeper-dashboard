## Summary

Keep open: the Browserbase direction is useful, but this PR is not merge-ready because the documented profile is still rejected by the runtime schema, shared browser routes still receive the empty placeholder CDP URL, and malformed signed URLs can leak into errors.

## What This Changes

This PR adds a Browserbase browser profile driver that fetches a fresh Browserbase connectUrl per CDP attach, updates browser profile types/capabilities/docs/tests, and raises remote CDP timeout defaults.

## Best Possible Solution

Centralize Browserbase connectUrl resolution at the shared profile/route boundary, update the canonical zod schema/help/docs/tests for the new config contract, redact signed URLs in all errors, and require redacted real Browserbase proof before merge.

## Reproduction Assessment

Yes for the review blockers by source inspection: the PR head still rejects the documented profile in runtime schema and shared browser routes still pass the empty profile cdpUrl. I did not run a live Browserbase flow because this is a read-only review and no inspectable proof artifact was attached.

## Solution Assessment

No: per-attach Browserbase connectUrl resolution is the right direction, but this implementation is not central enough and misses the canonical runtime config contract. The safer solution is a shared CDP-resolution boundary plus schema/docs/test alignment, URL redaction, and real Browserbase proof.
