## Summary

Keep open. Current main has a bundled browser plugin, managed profiles, headless launch support, existing-session attach, remote CDP, and optional Docker browser installation, but the normal managed launch path still depends on finding a host Chromium-family executable and still fails when none is available. The remaining request is a first-class no-host-browser runtime, which needs maintainer product, security, and packaging decisions rather than conservative cleanup.

## What This Changes

The issue requests a first-class bundled headless Chromium runtime so OpenClaw agents can use JS-capable web/browser access without a user-installed Chrome, third-party web APIs, or the old Chrome relay path.

## Best Possible Solution

Define a maintainer-approved browser-runtime packaging and discovery policy, then wire the browser plugin resolver, doctor/fix repair, Docker/package docs, cache persistence, sandbox guidance, and regression tests so the managed `openclaw` profile can start in headless/server/container deployments without requiring a preinstalled host browser.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a host or container with a managed local browser profile but no supported Chrome/Brave/Edge/Chromium executable at the resolver’s explicit/default/system paths; current main returns null, launch throws `No supported browser found`, and doctor emits the missing-executable warning.

## Solution Assessment

Unclear. The requested direction identifies the right remaining gap, but directly bundling Chromium in core is not yet proven to be the narrowest maintainable solution; maintainers should first choose the approved package/download/cache/sandbox policy and then implement that path in the browser plugin.
