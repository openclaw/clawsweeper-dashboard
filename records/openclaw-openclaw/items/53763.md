## Summary

Keep open. Current main and v2026.6.10 ship substantial browser support, but `web_fetch` and `web_search` still remain HTTP/provider paths and do not provide the requested bundled browser-backed web-access layer with rendered extraction, login/session persistence, proxy policy, and security boundaries.

## What This Changes

The issue requests a built-in headless Chromium-backed web access layer that works without user Chrome or third-party web APIs.

## Best Possible Solution

Keep this as the canonical maintainer product/security tracker, then split implementation into browser-runtime packaging, browser-backed extraction fallback, session/proxy isolation, and security-boundary decisions.

## Reproduction Assessment

Yes at source level for the missing capability: `web_fetch` remains guarded HTTP plus extractor/provider fallback and docs explicitly say it does not execute JavaScript. I did not run live benchmarks against zsxq.com, Xiaohongshu, or WeChat articles.

## Solution Assessment

No as one automatic implementation. The direction is valid, but the best fix needs staged maintainer choices on core vs plugin ownership, browser binary/runtime packaging, extraction behavior, session persistence, proxy credentials, and browser security policy.
