## Summary

Keep open. Current main has a bundled browser plugin with managed profiles, headless launch mode, remote CDP, existing-session attach, and opt-in Docker browser installation, but the normal managed launch path still requires an explicit/default/system Chromium-family executable and still errors when none is found. The remaining request is a first-class bundled or managed browser runtime that works without a host-installed browser, which needs maintainer product, packaging, and security review before implementation.

## What This Changes

The issue requests a first-class bundled headless Chromium-style browser runtime so agents can access JS-rendered and login-required pages without depending on host Chrome, third-party extraction APIs, or a Chrome relay.

## Best Possible Solution

Define a maintainer-approved browser-runtime packaging path, then wire executable discovery, doctor/fix repair, Docker/package docs, and update policy around it so the managed openclaw browser profile can start in headless/server/container deployments without requiring a preinstalled host browser.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The remaining gap is high-confidence from source inspection: managed launch still fails when executable discovery returns null, and doctor still directs users to install a Chromium-based browser.

## Solution Assessment

Unclear. A bundled Chromium runtime may be the right product direction, but the best maintainable solution needs an explicit maintainer decision on binary provenance, update cadence, Docker footprint, cache location, and sandbox policy before a code PR is safe.
