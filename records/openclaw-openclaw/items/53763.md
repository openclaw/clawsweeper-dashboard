## Summary

Keep open. Current main and v2026.6.6 already cover managed isolated browser profiles, headless launch, proxy launch flags, and optional Docker images with baked Chromium, but the request still asks for always-available browser distribution plus first-class proxy, user-agent, credential, SSRF, and session-boundary policy that needs maintainer product and security scoping.

## What This Changes

The issue requests a built-in headless Chromium-backed web access path with JS rendering, session persistence, proxy/user-agent support, and no dependency on a user's Chrome or third-party APIs.

## Best Possible Solution

Keep the shipped managed-browser baseline and have maintainers split or sponsor the remaining design around browser distribution, proxy/SecretRef handling, SSRF behavior, user-agent policy, and agent/session profile authority.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/design request. Source confirms current managed/headless browser support exists while the full bundled, policy-defined web access stack remains missing.

## Solution Assessment

No. A single unconditional bundled-browser change is too broad; the maintainable solution is staged design on the existing browser plugin, Docker image, proxy/SecretRef handling, SSRF policy, and session/profile isolation.
