## Summary

Keep open. Current main and the latest release already ship a managed browser plugin with headless local profiles, remote CDP, node proxying, and sandbox-browser support, but this issue still asks for a broader built-in web-access layer that removes installed-browser/external-CDP dependencies and settles extraction, proxy, login-state, and security behavior.

## What This Changes

The issue requests a built-in headless Chromium browser tool so agents can access JS-rendered or login-required webpages without relying on user Chrome, third-party APIs, or browser relay setup.

## Best Possible Solution

Keep the current managed browser as the baseline, then split the remaining work into maintainer-approved browser runtime packaging or backend ownership, extraction fallback behavior, and proxy/session isolation design.

## Reproduction Assessment

Yes for the missing capability at source level: current main has a managed browser plugin, but local managed launch still resolves an installed Chromium-family executable and proxying remains explicit config rather than a bundled all-deployment web-access layer. It is not a single broken existing contract to reproduce with one command.

## Solution Assessment

No as a one-shot implementation request. The maintainable path is a staged product/security design around packaged runtime expectations, sandbox/CDP ownership, extraction fallback behavior, and session/proxy boundaries.
