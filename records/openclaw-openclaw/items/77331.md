## Summary

Keep open: current main still performs the first embedded-run tool build synchronously, and the merged/recorded optimizations only reduce narrower plugin/media factory costs rather than proving the cold-start event-loop stall is gone.

## What This Changes

The issue asks OpenClaw to remove a roughly 21s first-run event-loop block from `createOpenClawCodingTools()` by prewarming, memoizing, lazy-loading, or yielding heavy tool initialization.

## Best Possible Solution

Measure the cold tool-prep breakdown on current main, then move heavy imports and plugin/tool factory work off the first live embedded turn while preserving existing tool policy behavior.

## Reproduction Assessment

Yes. A high-confidence source-level path is cold gateway startup followed by an embedded run that reaches `runEmbeddedAttempt()` and synchronously calls `createOpenClawCodingTools()` before the LLM request; this review did not execute the live timing trace.

## Solution Assessment

Unclear. Prewarming, descriptor/factory memoization, lazy segmentation, and event-loop yielding could all help, but the narrowest maintainable fix needs measured cold-stage data and coordination with #73039 and #76606.
