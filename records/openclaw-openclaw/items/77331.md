## Label Justifications

- `P1`: The report describes first-turn embedded agent startup blocking the event loop long enough to break latency-sensitive channel workflows such as voice/STT WebSockets.
- `impact:message-loss`: The reported event-loop stall causes channel transports to miss heartbeats or time out before a first response can be delivered.
- `impact:other`: The issue is also a core gateway/agent hot-path performance and availability problem that does not fit a narrower issue impact label.

## Summary

Keep open: current main still constructs the embedded run tool surface synchronously on the default path, the merged mitigations only cover narrower cases, and the directly related gateway prewarm candidate remains open and unmerged.

## What This Changes

The issue asks for the first embedded agent run after gateway startup to avoid a long synchronous `createOpenClawCodingTools()` cold-start block that can starve latency-sensitive channels.

## Best Possible Solution

Land or replace the gateway/agent cold-start fix only after a rebased implementation proves first-turn timing, event-loop health, and startup-readiness tradeoffs on current main.

## Reproduction Assessment

No high-confidence live current-main reproduction was established in this read-only review. Source inspection does show the default embedded run still synchronously constructs the broad tool surface, and the issue body provides concrete cold-start timing logs from released builds.

## Solution Assessment

Unclear as a final fix. Prewarming is a plausible mitigation, but maintainers need to choose whether to shift cold work into gateway readiness or pursue deeper lazy/memoized tool construction.
