## Summary

Keep open: current main still performs broad synchronous tool construction on the embedded-run path, and the current cache/denylist work only proves partial mitigations rather than a fix for first cold-start event-loop starvation.

## What This Changes

The issue requests reducing the first cold embedded-agent `createOpenClawCodingTools()` path so it no longer blocks the event loop for about 21s before LLM submission.

## Best Possible Solution

Measure current-main cold first-turn tool prep, then move heavy imports and plugin/tool factory work off latency-sensitive embedded runs without changing tool policy semantics.

## Reproduction Assessment

Yes, source inspection gives a clear path: restart a cold gateway and trigger an embedded run that reaches `runEmbeddedAttempt()`. This review did not re-time the 21s stall, so the exact current-main latency still needs live proof.

## Solution Assessment

Unclear. Prewarming is plausible, but the maintainable fix should be chosen after measuring cold import/factory stages and coordinating with the open memoization/no-tools work.
