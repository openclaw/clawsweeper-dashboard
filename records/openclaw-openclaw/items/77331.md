## Summary

Keep open: current main still builds the embedded-agent tool bundle synchronously on the first live run path, and the merged latency mitigations only reduce narrower factory costs rather than proving the cold-start event-loop stall is gone.

## What This Changes

The issue asks OpenClaw to remove the roughly 21 second first-run cold-start stall in `createOpenClawCodingTools()` for embedded agent runs.

## Best Possible Solution

Measure cold `core-plugin-tools` sub-stages on current main, then move heavy import and plugin/tool factory work out of the first latency-sensitive embedded turn without changing tool policy semantics.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: restart a cold gateway and trigger an embedded run that reaches `runEmbeddedAttempt()`, which still synchronously invokes `createOpenClawCodingTools()` before proceeding. This sweep did not execute the live cold timing trace, so the exact 21 second number still needs runtime proof.

## Solution Assessment

Unclear. Prewarming, descriptor/factory memoization, lazy segment loading, or yielding could each help, but the best maintainable fix should be chosen after current-main cold-stage measurements and coordination with #73039 and #76606.
