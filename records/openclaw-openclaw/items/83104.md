## Summary

Keep this PR open: it targets a real exported-HTML attribute injection issue, but the current patch double-escapes copy-link IDs and the external real-behavior proof is still only a unit/JSDOM-style test claim.

## What This Changes

The PR escapes entry IDs in exported HTML message IDs and copy-link data attributes and adds a security regression test for crafted IDs.

## Best Possible Solution

Escape exactly at HTML attribute boundaries, pass raw `entry.id` through copy-link/share-url logic, add coverage that copied `targetId` remains raw, and include redacted real export proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: render an export whose `entry.id` contains a quote and current main inserts it raw into `id` and `data-entry-id` attributes; I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The escaping direction is right, but the caller should pass raw `entry.id` into `renderCopyLinkButton()` and let the helper be the single attribute-escaping boundary.
