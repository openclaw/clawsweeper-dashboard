## Summary

Keep open: the security fix is useful, but the PR head still double-escapes copy-link IDs and the supplied proof is test-only rather than real export/browser behavior.

## What This Changes

This PR escapes `entry.id` in exported HTML `id` and `data-entry-id` attributes and adds a regression test for crafted IDs.

## Best Possible Solution

Escape exactly at HTML attribute boundaries, pass raw `entry.id` through copy-link/share-url logic, add a regression that copied `targetId` remains raw, and include redacted real export proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main renders raw `entry.id` into HTML attributes, so a crafted double quote can break out; I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The escaping direction is right, but the caller should pass raw `entry.id` into `renderCopyLinkButton()` and let the helper be the single attribute-escaping boundary.
