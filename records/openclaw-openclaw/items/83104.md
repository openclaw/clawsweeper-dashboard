## Summary

Keep open: the attribute-injection sink is real, but this head double-escapes the copy-link entry id, so copied deep links for special-character IDs will not round-trip, and the proof is still linkedom/Vitest rather than a real browser export.

## What This Changes

The PR changes HTML export rendering to escape `entry.id` for message element IDs and copy-link `data-entry-id` attributes.

## Best Possible Solution

Escape raw entry IDs exactly once at each HTML attribute boundary, keep JS/share/deep-link logic on raw decoded IDs, add focused regression coverage, and land after redacted browser/export proof.

## Reproduction Assessment

Yes. Source inspection on current main shows raw `entry.id` interpolation into exported HTML attributes, so a crafted quote-bearing entry id can break the attribute boundary.

## Solution Assessment

No. Escaping at the HTML attribute boundaries is the right direction, but this PR must pass the raw id into `renderCopyLinkButton` so the `data-entry-id` value is escaped exactly once and browser-decoded back to the raw id.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Pass the raw id into the copy-link renderer:** `src/auto-reply/reply/export-html/template.js:1324`
  - body: `renderCopyLinkButton` now escapes its parameter, but this call also pre-escapes `entry.id`. For an id containing `"` or `&`, the emitted `data-entry-id` becomes double-escaped, so `btn.dataset.entryId` feeds `buildShareUrl` an entity string instead of the raw id that `byId`/deep-link lookup uses. Keep the escaped `entryId` for the DOM `id`, but call `renderCopyLinkButton(entry.id)` here.
