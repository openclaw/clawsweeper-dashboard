## Summary

Keep open: the underlying HTML attribute XSS is valid and current main still has the raw interpolation, but this PR still double-escapes copy-link IDs and only supplies test/mock proof for an external security fix.

## What This Changes

The PR escapes exported HTML entry IDs in message element `id` and copy-link `data-entry-id` attributes and adds a regression test for crafted `entry.id` payloads.

## Best Possible Solution

Escape exactly at HTML attribute boundaries, pass raw `entry.id` through copy-link/share-url logic, add a regression that copied `targetId` remains raw, and include redacted real export proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: render an export whose `entry.id` contains a double quote and current main inserts it raw into `id` and `data-entry-id` attributes; I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The escaping direction is right, but the caller should pass raw `entry.id` into `renderCopyLinkButton()` and let the helper be the single attribute-escaping boundary.
