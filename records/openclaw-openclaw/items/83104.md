## Summary

Keep open: the security hardening direction is valid, but this branch double-escapes the copy-link entry id and still needs after-fix real behavior proof before merge.

## What This Changes

The PR escapes exported session entry ids before interpolating them into HTML `id` and `data-entry-id` attributes in the HTML export template.

## Best Possible Solution

Escape entry ids exactly once at each HTML attribute boundary, add regression coverage for a crafted id that proves no injected attribute is created and copy-link deep links still resolve, then provide redacted real export proof.

## Reproduction Assessment

Yes, by source inspection: current `main` places raw `entry.id` into quoted HTML attributes, so an id containing a quote can break attribute boundaries in the generated export. I did not run the browser export flow locally.

## Solution Assessment

No. Escaping at the attribute boundary is the right direction, but the proposed caller/helper combination escapes `data-entry-id` twice and should be collapsed to one escape layer with focused coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid double-escaping the copy-link entry id:** `src/auto-reply/reply/export-html/template.js:1324`
  - body: This caller now passes an escaped entity string into `renderCopyLinkButton`, and that helper also escapes its argument. For an id like `a"b`, the button source becomes `data-entry-id="a&amp;quot;b"`, so `btn.dataset.entryId` is `a&quot;b`; `buildShareUrl` then copies that entity text into `targetId`, while the message DOM id parses back to the raw quote form. The copied deep link will not find or highlight the entry. Pass the raw `entry.id` here and let only one layer escape the attribute.
  - confidence: 0.88
