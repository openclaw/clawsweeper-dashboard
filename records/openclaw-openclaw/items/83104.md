## Summary

Keep open. The PR addresses a real exported-HTML attribute injection sink, but the current head still double-escapes the copy-link ID path and the submitted proof does not exercise the real browser/export behavior that this security fix depends on.

## What This Changes

The PR changes HTML export rendering so entry IDs are escaped before being interpolated into message element IDs and copy-link data attributes.

## Best Possible Solution

Escape raw entry IDs exactly once at each HTML attribute sink, keep share/deep-link logic on browser-decoded raw IDs, add focused regression coverage, and land after redacted real browser/export proof.

## Reproduction Assessment

Yes. Source inspection shows current main puts raw entry IDs into exported HTML attributes, and PR-head inspection shows the proposed copy-link path is double-escaped.

## Solution Assessment

No. Escaping at the HTML attribute boundaries is the right direction, but this implementation should pass the raw ID into renderCopyLinkButton and escape exactly once at the sink.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Pass the raw id into the copy-link renderer:** `src/auto-reply/reply/export-html/template.js:1324`
  - body: `renderCopyLinkButton` now escapes its `entryId` argument, but this call already passes `escapeHtmlAttr(entry.id)`. Pass `entry.id` here and let the helper escape the attribute once.
