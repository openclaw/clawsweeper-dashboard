## Summary

Keep open: the reported attribute-injection problem is real on current main and in the latest release, but this PR is not mergeable as written because it double-escapes the copy-link entry id and the supplied proof does not exercise the patched OpenClaw export path.

## What This Changes

Escapes `entry.id` in the HTML export message element `id` and copy-link `data-entry-id` attributes.

## Best Possible Solution

Escape `entry.id` exactly at HTML attribute interpolation boundaries while keeping raw ids for dataset, share URL, and `byId` matching contracts, then add focused security and copy-link regression coverage plus real export proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main inserts raw `entry.id` into HTML attributes, so a crafted id containing a quote can break out of `id` or `data-entry-id` when the export template renders.

## Solution Assessment

No. The security direction is right, but this patch escapes both before and inside `renderCopyLinkButton()`, so the maintainable fix is a single escaping boundary that preserves raw ids for copy-link and deep-link logic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid double-escaping copy-link ids:** `src/auto-reply/reply/export-html/template.js:1324`
  - body: `renderCopyLinkButton()` now escapes its argument, but this call also passes `escapeHtmlAttr(entry.id)`. For ids containing `"` or `&`, `data-entry-id` becomes an encoded value such as `&quot;`; the click handler copies that into `targetId`, and reload only matches raw `byId` keys, so copy links for those entries stop deep-linking. Pass the raw id here and let the helper be the single attribute boundary.
  - confidence: 0.9
