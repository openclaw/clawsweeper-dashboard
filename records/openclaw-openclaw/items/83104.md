## Summary

Keep open: source inspection supports the reported HTML export attribute-XSS bug, but this patch double-escapes copy-link IDs and only reports existing tests instead of real export proof.

## What This Changes

The PR escapes `entry.id` when rendering HTML export message element IDs and copy-link `data-entry-id` attributes.

## Best Possible Solution

Revise the patch to escape exactly at HTML attribute boundaries while preserving raw `entry.id` for dataset/share URL contracts, then add focused regression coverage and redacted real export proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: render an export whose trajectory entry id contains a quote and the current template inserts it into raw HTML attributes. I did not run the repro in this read-only pass.

## Solution Assessment

No, not as written: the security direction is right, but escaping both before and inside `renderCopyLinkButton()` breaks the raw-ID copy-link/deep-link contract. The narrower fix is to keep raw IDs in logic and escape only when interpolating into attributes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Avoid double-escaping copy-link IDs:** `src/auto-reply/reply/export-html/template.js:1324`
  - body: `renderCopyLinkButton()` now escapes its argument, but this call also passes `escapeHtmlAttr(entry.id)`. For an id containing `"` or `&`, the rendered `data-entry-id` becomes an encoded string such as `&quot;`; the click handler copies that encoded value into `targetId`, and initial load only honors target IDs that match the raw `byId` keys. Pass the raw `entry.id` here and let the helper be the single attribute-escaping boundary.
  - confidence: 0.9
