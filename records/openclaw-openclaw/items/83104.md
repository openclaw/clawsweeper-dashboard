## Summary

Keep open: the PR addresses a real security gap in the HTML export template, but the current diff double-escapes copy-link IDs and the supplied proof does not exercise the changed OpenClaw export path.

## What This Changes

The PR changes the HTML export template to escape `entry.id` when rendering message element IDs and copy-link button `data-entry-id` attributes.

## Best Possible Solution

Escape exactly at HTML attribute boundaries while preserving raw `entry.id` for dataset and share URL logic, then add focused security coverage and redacted real export proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: render an export whose trajectory entry ID contains a double quote, and current main inserts it raw into `id` and `data-entry-id` attributes.

## Solution Assessment

No, not as written. The escaping direction is right, but the caller should pass the raw ID into `renderCopyLinkButton()` and let that helper be the single attribute-escaping boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass raw IDs into the copy-link helper:** `src/auto-reply/reply/export-html/template.js:1324`
  - body: `renderCopyLinkButton()` now escapes its argument, but this call also passes `escapeHtmlAttr(entry.id)`. For IDs containing quotes or ampersands, the DOM dataset value becomes entity text such as `a&quot;...`; copied links then use that encoded target and reloads miss the raw `byId` key. Pass raw `entry.id` here and escape only inside the helper.
  - confidence: 0.9
