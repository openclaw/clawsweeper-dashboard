## Summary

Keep open. Current `main` still lacks a SIGTERM/SIGINT cancellation path for one-shot `openclaw agent` runs: the Gateway-backed CLI call, embedded/local fallback path, Gateway client request wrapper, and connection-close path do not wire an external process signal to the active agent run. Recent Gateway abort and timeout-fallback work helps adjacent surfaces, but it does not solve the external cron/coreutils `timeout` leak described here, and the docs still do not mention the `timeout -k` mitigation.

## What This Changes

This issue requests bounded SIGTERM handling for `openclaw agent` under cron/coreutils `timeout`, or at minimum docs that require a SIGKILL escalation delay such as `timeout -k`.

## Best Possible Solution

Add a focused shutdown path for `openclaw agent`: SIGTERM/SIGINT should trigger one shared AbortController, abort the pending Gateway request, capture the accepted Gateway run ID early enough to send or invoke an authorized abort for that run, pass the same signal into embedded/local runs, exit within a bounded grace window, and document `timeout -k` as the external supervisor safety belt.

## Reproduction Assessment

Yes. The issue includes concrete Linux cron/coreutils steps and observed process chains, and current source supports a focused unit reproduction by holding a Gateway `agent` request or embedded `agentCommand` pending, delivering SIGTERM/SIGINT, and observing that no CLI-owned abort path is wired.

## Solution Assessment

Yes. The primary signal-handling fix is the narrowest maintainable solution because the Gateway and embedded runner already have abort seams; a docs-only `timeout -k` note is useful mitigation but would leave other supervisors vulnerable.
