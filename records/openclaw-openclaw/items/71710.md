## Summary

Keep open. Current main still lacks a CLI-owned SIGTERM/SIGINT cancellation path for one-shot `openclaw agent` runs, and the docs still do not warn external cron/coreutils users to use `timeout -k`; recent related work fixed adjacent Gateway abort and timeout-fallback lock issues but not this process-lifetime bug.

## What This Changes

The issue requests that externally scheduled one-shot `openclaw agent` invocations exit promptly on SIGTERM, or at minimum document SIGKILL escalation for cron wrappers.

## Best Possible Solution

Add a focused one-shot shutdown contract for `openclaw agent`: SIGTERM/SIGINT should create a bounded cancellation path, abort the pending Gateway request, stop any accepted Gateway run through the existing abort registry or owner-disconnect mechanism, pass the same signal into `--local` and embedded fallback runs, preserve session-lock cleanup, and document `timeout -k` as the external-supervisor safety belt.

## Reproduction Assessment

Yes. The issue provides concrete Linux cron/coreutils steps with observed process chains and memory pressure, and current source supports a focused unit reproduction by holding a Gateway agent request or embedded run pending, emitting SIGTERM/SIGINT, and observing that no CLI-owned abort path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort path and `AgentCommandOpts.abortSignal` is the narrowest maintainable root fix; docs-only `timeout -k` guidance is a useful mitigation but would leave the underlying hung-process behavior for other supervisors.
