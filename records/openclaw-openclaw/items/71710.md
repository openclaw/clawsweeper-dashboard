## Summary

Keep open. Current main still has no `openclaw agent` CLI signal bridge from SIGTERM/SIGINT into the pending Gateway request or embedded run, and the docs still do not mention the reporter's `timeout -k` mitigation.

## What This Changes

The issue reports that cron-spawned `openclaw agent` runs ignore SIGTERM from `timeout`, leaving hung process chains that exhaust RAM and swap.

## Best Possible Solution

Wire SIGTERM/SIGINT for `openclaw agent` through a CLI-owned AbortController, abort the accepted Gateway run by captured run id, pass the same signal into embedded/local execution, bound shutdown grace, and document `timeout -k` as an external supervisor backstop.

## Reproduction Assessment

Yes. The issue gives concrete cron/coreutils steps, and current source supports a focused unit reproduction by holding a Gateway final response or embedded `agentCommand` pending, delivering SIGTERM/SIGINT, and observing that no CLI-owned abort path is wired.

## Solution Assessment

Yes. A narrow signal bridge into the existing Gateway and embedded abort seams is the best maintainable fix; docs-only mitigation helps operators but should not be the final behavior.
