## Summary

Keep open. Current main still has no CLI-owned SIGTERM/SIGINT cancellation path for one-shot `openclaw agent` runs, while the issue has concrete cron/coreutils reproduction data and the docs still do not warn external cron users to use `timeout -k` as a mitigation.

## What This Changes

The issue asks for `openclaw agent` runs launched by external cron wrappers to exit predictably on SIGTERM, or at minimum for docs to require a SIGKILL escalation wrapper such as `timeout -k`.

## Best Possible Solution

Add a focused one-shot shutdown contract: process SIGTERM/SIGINT should cancel the pending Gateway request, stop any accepted Gateway agent run through the existing abort registry or disconnect-owned cleanup path, pass the same signal into local and embedded fallback runs, preserve session-lock cleanup, and document `timeout -k` as the external-supervisor safety belt.

## Reproduction Assessment

Yes. The issue gives concrete Linux cron/coreutils steps with observed process chains and memory pressure, and current source supports a focused unit reproduction by holding a Gateway agent request or local embedded run pending, emitting SIGTERM/SIGINT, and observing that no CLI-owned abort path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort path and `AgentCommandOpts.abortSignal` is the narrowest maintainable root fix; docs-only `timeout -k` guidance is useful mitigation but would leave the underlying hung-process behavior for other supervisors.
