## Summary

Keep open. Current main still has the one-shot `openclaw agent` path waiting on a Gateway final response without a process-signal abort bridge, even though the Gateway-side `agent` abort seam exists; the cron-facing CLI docs also still lack a `timeout -k` backstop for external supervisors.

## What This Changes

The issue reports that cron-spawned `openclaw agent` processes can survive SIGTERM from coreutils `timeout`, accumulate hung process trees, and exhaust host memory/swap.

## Best Possible Solution

Add a bounded one-shot `openclaw agent` shutdown path that forwards SIGTERM/SIGINT into Gateway and embedded execution, aborts an accepted run when known, and documents `timeout -k` as a cron supervisor backstop.

## Reproduction Assessment

Yes, at source level: hold a Gateway `agent` request pending, deliver SIGTERM/SIGINT to the CLI process, and current main has no CLI-owned abort signal, accepted-run callback, or embedded fallback signal wiring to observe. I did not run the multi-day cron scenario live.

## Solution Assessment

Yes. The narrow maintainable fix is to bridge process signals into the existing Gateway and embedded abort seams, with docs as an external-supervisor backstop rather than the only fix.
