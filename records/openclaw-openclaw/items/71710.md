## Summary

Keep open. Current main still leaves the one-shot `openclaw agent` CLI without a SIGTERM/SIGINT bridge into the pending Gateway request or embedded fallback path; Gateway-side abort plumbing exists, but the CLI cannot currently drive it when an external supervisor sends SIGTERM.

## What This Changes

The issue reports that cron-spawned `openclaw agent` processes ignore SIGTERM from coreutils `timeout`, leaving hung process chains that can exhaust host memory and swap.

## Best Possible Solution

Add a bounded one-shot `openclaw agent` shutdown path that forwards SIGTERM/SIGINT into Gateway and embedded execution, aborts the accepted run when known, and documents `timeout -k` as a supervisor backstop.

## Reproduction Assessment

Yes for source-level reproduction: hold a Gateway `agent` request pending, deliver SIGTERM/SIGINT to the CLI path, and current main has no CLI-owned abort signal, accepted-run callback, or embedded fallback signal wiring to observe. I did not run the multi-day cron scenario live.

## Solution Assessment

Yes. The narrow maintainable fix is to bridge process signals into the existing abort seams and add external-supervisor documentation; treating the docs backstop as the whole fix would leave the root CLI shutdown behavior broken.
