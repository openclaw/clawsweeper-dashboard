## Summary

Keep open. Current main still lacks a one-shot `openclaw agent` SIGTERM/SIGINT path that can cancel the pending Gateway request, abort the accepted Gateway run, or abort embedded fallback/local execution, and the latest comment adds a related but separable post-completion hang data point rather than resolving the SIGTERM case.

## What This Changes

The issue requests bounded shutdown behavior, or at minimum explicit external-supervisor guidance, for cron-spawned `openclaw agent` processes that survive SIGTERM and accumulate.

## Best Possible Solution

Add a bounded one-shot CLI shutdown path that forwards SIGTERM/SIGINT into Gateway and embedded agent execution, aborts the accepted run when known, and documents `timeout -k` as an external-supervisor backstop.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: hold a Gateway `agent` request pending, send SIGTERM or SIGINT to the one-shot CLI, and current main has no caller AbortSignal, accepted-run callback, top-level signal bridge, or embedded fallback signal wiring to observe.

## Solution Assessment

Is this the best way to solve the issue? Yes for the SIGTERM case: bridge process signals into the existing Gateway `agent` abort controller and local runner abort surfaces; the post-completion no-signal handle leak should remain a narrower follow-up after the Codex raw-completion fix is evaluated.
