## Summary

Keep open. Current main still lacks a one-shot `openclaw agent` SIGTERM/SIGINT cancellation path for both Gateway-backed and embedded/local runs, and the public cron/agent docs still do not document the `timeout -k` mitigation for external cron wrappers.

## What This Changes

This issue requests graceful SIGTERM handling or, at minimum, cron documentation that prevents `timeout openclaw agent ...` processes from accumulating when an agent run blocks past the wrapper budget.

## Best Possible Solution

Add a focused shutdown path for `openclaw agent`: process SIGTERM/SIGINT should trigger a shared abort controller, abort the pending Gateway request, stop any accepted Gateway agent run through the existing abort registry or an equivalent owner-scoped path, pass the same signal to embedded/local runs, and document `timeout -k` as the external supervisor safety belt.

## Reproduction Assessment

Yes. The issue includes concrete Linux cron/coreutils steps and observed process chains, and current source supports a focused unit reproduction by holding a Gateway `agent` request or local embedded run pending, delivering SIGTERM/SIGINT, and observing that no CLI-owned abort path is wired.

## Solution Assessment

Yes. The requested primary fix is the narrowest maintainable direction because the agent runner and Gateway already have AbortSignal seams; a docs-only `timeout -k` warning is useful mitigation but would leave other supervisors vulnerable to the same hung-process behavior.
