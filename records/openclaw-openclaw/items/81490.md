## Label Justifications

- `P1`: A supported sessions_yield/subagent orchestration workflow can silently stall real automated or channel-bound user workflows.
- `impact:session-state`: The central failure is route/session state drifting from the yielded parent session to a new route run.
- `impact:message-loss`: The child completion or parent continuation can be misrouted away from the intended parent workflow.

## Summary

Keep open: current main and the latest release still make yielded requester sessions inactive for embedded-run steering, then fall back to a direct requester-agent handoff keyed by the route; I did not find a durable paused-session route binding that would force the child completion back into the yielded parent session id.

## What This Changes

The issue asks OpenClaw to resume the yielded parent session when a subagent completion arrives on the same external route instead of starting a new route-bound run.

## Best Possible Solution

Land a maintainer-owned yielded-session resume contract that durably binds the external route to the paused parent session and delivers child completion as that session's next event.

## Reproduction Assessment

Yes at source level: current main still makes yielded requesters inactive for embedded steering and then falls through to a route-key direct agent handoff. I did not run a live current-main Telegram/Bedrock reproduction.

## Solution Assessment

No, current main is not the best solution because it has no durable paused-session route binding and can hand completion to a fresh requester-agent run. The safer fix is a shared lifecycle contract for yielded parent resumption, not another one-off direct fallback.
