## Summary

Keep open: current main still has the source-backed yielded-parent/session-state gap, and the latest release has the same inspected behavior. The issue is not fixed by the open queue-announcement PR because the remaining failure is route/session reuse after `sessions_yield`, not only whether a completion announce can queue.

## What This Changes

The issue reports that subagent completion delivery starts a new run on the parent route instead of resuming the yielded parent session.

## Best Possible Solution

Land a coherent yielded-session resume contract that preserves the paused parent session id and route binding, then delivers child completion as the parent session's next event without falling back to a fresh route run.

## Reproduction Assessment

Yes, source-level: current main still makes yielded parents inactive for embedded queue steering and falls through to route-key direct agent handoff. I did not run a live Telegram/Bedrock reproduction in this read-only review.

## Solution Assessment

No, current main is not the best solution because direct route handoff can create a new run instead of resuming the paused parent. Persisting paused yield ownership is plausible, but the final fix should be a maintained lifecycle contract rather than another one-off fallback.
