## Label Justifications

- `P1`: The issue describes a deterministic broken subagent/yield workflow that silently stalls real automated agent work.
- `impact:session-state`: The reported failure orphans the yielded parent session and rewrites the route session-store pointer to a different session id.
- `impact:message-loss`: The child completion announce is consumed by the wrong run instead of waking the parent continuation.

## Summary

Keep open: current main and the latest release still lack a durable paused-session route binding, and the current announce path only steers active requester runs before falling back to a direct requester-agent handoff. The reported yielded-parent orphaning remains source-supported and is not safely owned by the open queueing PR.

## What This Changes

The issue reports that subagent completion announces on an external route start a new parent run instead of resuming the previously yielded parent session.

## Best Possible Solution

Add a maintainer-approved yielded-session route binding or equivalent lifecycle state so subagent completion resumes the yielded parent session id, with regression tests for active, yielded-inactive, and direct fallback cases.

## Reproduction Assessment

Yes at source level, but not by live execution in this read-only review: current code treats only active requester session ids as steerable and otherwise falls back to a direct requester-agent handoff. The reporter also provided concrete transcript and session-store evidence for the yielded parent being orphaned.

## Solution Assessment

Unclear: persisting a paused session UUID on the route is plausible, but the best permanent fix needs maintainer approval because it changes durable session route/liveness semantics. The current main implementation is not the best end state because it has no source-visible paused-session binding.
