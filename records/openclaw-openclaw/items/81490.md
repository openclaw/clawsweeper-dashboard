## Label Justifications

- `P1`: A supported sessions_yield/subagent orchestration workflow can silently stall real automated or channel-bound user workflows.
- `impact:session-state`: The reported failure changes the route/session-store pointer away from the yielded parent session to a new route run.

## Summary

Keep open: current main still treats yielded requester sessions as inactive for subagent completion steering, then falls back to a route-keyed direct agent handoff that can create a fresh run instead of resuming the yielded parent. The latest issue discussion says the failure still reproduces on v2026.6.10, and related open PRs overlap but do not provide a merged, proven route-to-paused-session fix.

## What This Changes

Fix sessions_yield subagent completion routing so a completion on an external route resumes the yielded parent session instead of creating a new route run.

## Best Possible Solution

Land a maintainer-owned yielded-session resume contract that durably binds external routes to paused parent session IDs and wakes that session for child completion events.

## Reproduction Assessment

Yes at source level: current main still makes yielded parent runs inactive for active-run steering, and current completion delivery then falls through to a canonical-route direct agent handoff. I did not run a live current-main Telegram/Bedrock reproduction.

## Solution Assessment

No, current main is not the best solution because it lacks a durable paused-session route binding. The better fix is a shared yielded-parent lifecycle and route-resolution contract, not another one-off direct fallback.
