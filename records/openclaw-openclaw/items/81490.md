## Label Justifications

- `P1`: The original report broke real subagent orchestration by orphaning yielded parent sessions after child completion.
- `impact:session-state`: The central failure was route session state drifting from the paused parent session to a fresh session id.
- `impact:message-loss`: The completion event was consumed by the wrong run, so the parent’s expected follow-up message never reached the workflow.

## Summary

Close as implemented on current main: the inactive-yield fallback now calls the requester route through the normal `agent` path, and that path reuses the stored fresh `sessionId` instead of allocating a new run for the same route. The broader subagent delivery reliability work remains open elsewhere, but the route-to-fresh-run overwrite described here is fixed on main only.

## What This Changes

Report that subagent completion into a yielded external-route parent starts a fresh route run and overwrites the paused session instead of resuming it.

## Best Possible Solution

Keep the current main session-reuse behavior for yielded requester routes, and leave broader exactly-once subagent completion delivery work on the existing canonical reliability issues.

## Reproduction Assessment

No high-confidence current-main failure path remains from source inspection. The old fallthrough now re-enters the canonical requester route, and the `agent` handler reuses the stored fresh `sessionId` instead of creating a new one.

## Solution Assessment

Yes for this narrow bug: using normal session-store reuse is the cleaner fix than adding a separate paused-session pointer. Broader durable delivery and duplicate-prevention semantics remain separate reliability work.
