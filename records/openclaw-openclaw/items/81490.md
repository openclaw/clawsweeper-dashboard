## Label Justifications

- `impact:session-state`: The central failure is route/session state drifting from the yielded parent session to a new route run.

## Summary

Keep open: current main still treats yielded parents as inactive for embedded steering and falls back to a route-key direct requester-agent handoff, while no durable paused-session route binding exists to force the completion back into the yielded parent session id. A fresh reporter comment says the same orphaning remains reproducible on v2026.6.10, and the related draft PR is open/dirty and does not prove this route-to-paused-session problem fixed.

## What This Changes

The issue asks OpenClaw to resume the yielded parent session for a subagent completion on the same external route instead of spawning a new route run and orphaning the paused session.

## Best Possible Solution

Land a maintainer-owned yielded-session resume contract that durably binds an external route to the paused parent session and delivers child completion as that session's next event, with tests for fresh install and upgrade/current-session behavior.

## Reproduction Assessment

Yes at source level: current main still makes yielded requesters inactive for embedded steering and then falls through to a route-key direct agent handoff. I did not run a live Telegram/Bedrock repro, but the issue includes detailed live evidence and a June 24 v2026.6.10 confirmation.

## Solution Assessment

No, current main is not the best solution because it has no durable paused-session route binding and can hand completion to a requester-agent run on the route. The safer fix is a shared yielded-parent resume contract rather than another one-off direct fallback.
