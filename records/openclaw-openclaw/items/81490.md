## Label Justifications

- `P1`: A supported sessions_yield plus subagent orchestration workflow can silently stall real automated or channel-bound workflows.
- `impact:session-state`: The central failure is route/session state drifting from the yielded parent session to a new route run.
- `impact:message-loss`: The child completion and parent continuation can be misrouted away from the intended parent workflow with no user-visible error.

## Summary

Keep open: current main and the latest release still use active-run-only requester wake plus a direct route-key requester-agent handoff, while no durable paused-session route binding exists to force a subagent completion back into the yielded parent session id. The related draft PR remains open, draft, and conflicting, and the latest reporter comment says the issue still reproduces on v2026.6.10.

## What This Changes

A yielded parent session should be resumed by subagent completion on the same external route instead of spawning a new route run and orphaning the paused session.

## Best Possible Solution

Land a maintainer-owned yielded-session resume contract that durably binds the external route to the paused parent session and delivers child completion as that session's next event, with sibling coverage for current route fallback and upgrade behavior.

## Reproduction Assessment

Yes, at source level: current main still makes yielded requesters inactive for embedded steering and then falls through to a route-key direct agent handoff. I did not run a live Telegram/Bedrock repro, but the issue includes detailed live evidence and a June 24, 2026 v2026.6.10 confirmation.

## Solution Assessment

No, current main is not the best solution because the direct fallback can deliver a visible completion without proving that the yielded parent session id is resumed. The safer path is a shared yielded-parent resume contract instead of another one-off fallback.
