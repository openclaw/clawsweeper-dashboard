## Summary

Keep open: current main still deliberately skips OpenClaw memoryFlush for CLI-routed sessions, `claude-cli` declares native compaction ownership, and the current context footer/notice surfaces are not backend-aware enough to answer the reporter's UX request. This is useful, but it needs maintainer product/API direction rather than conservative cleanup.

## What This Changes

Request a `claude-cli`-compatible pre-compaction handoff or memory flush path plus clearer backend-aware context status labels.

## Best Possible Solution

Define a maintainer-approved native-CLI compaction lifecycle contract, then reuse the existing memoryFlush/compaction notice/usage-bar surfaces to expose safe backend-aware handoff and context-stage status without breaking Claude CLI resume ownership.

## Reproduction Assessment

Yes, at source level. Current main excludes CLI-backed followups from `memoryFlush`, documents CLI backends as skipped, and marks `claude-cli` as native-compaction-owned; I did not run a fresh live Claude Code OAuth compaction scenario.

## Solution Assessment

Unclear as requested. A direct gate removal would conflict with the current provider-owned CLI session boundary; the better path is a maintainer-approved CLI/native-compaction hook or status contract that reuses existing compaction notice and usage surfaces.
