## Label Justifications

- `P1`: The bug can block real channel sends with attachments when models populate visible poll fields from the shared message tool schema.
- `impact:message-loss`: Outbound messages can be rejected before channel dispatch because stray poll-looking params trip the send-path guard.

## Summary

Keep open: current main and the latest release still reject `action="send"` before channel dispatch when non-empty poll-looking params are present, and the discussion has a second user confirming the same failure on Telegram. The narrow bug remains valid; broader per-channel/schema decomposition is already tracked in https://github.com/openclaw/openclaw/issues/43015.

## What This Changes

The issue asks for `message` tool `action="send"` to stop rejecting ordinary Feishu or Telegram sends with attachments when stray poll schema fields are present.

## Best Possible Solution

Land one canonical fix that strips or ignores auto-populated poll creation params for `action="send"` while preserving explicit `action="poll"` validation, and keep broader schema decomposition in https://github.com/openclaw/openclaw/issues/43015.

## Reproduction Assessment

Yes, at source level. Calling `runMessageAction()` on current main with `action="send"`, a normal message or file payload, and a non-empty poll field such as `pollQuestion`, `pollOption`, or non-zero `pollDurationSeconds` reaches the existing throw before channel dispatch.

## Solution Assessment

Yes for the narrow bug: action-aware sanitization or a narrower poll-intent check is the maintainable fix. The broader per-channel message tool split is useful design work but belongs in https://github.com/openclaw/openclaw/issues/43015 rather than blocking this specific send-path repair.
