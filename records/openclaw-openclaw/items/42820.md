## Label Justifications

- `P1`: The bug can block real Feishu, Telegram, and other channel sends with attachments when models populate visible poll fields.
- `impact:message-loss`: Outbound messages can be rejected before delivery because stray poll-looking params trip the message tool guard.

## Summary

Keep open: current main and the latest release still reject `action="send"` before channel dispatch when non-empty poll-looking params are present, and the discussion includes a second user confirming the same failure on Telegram. Several open PRs already target this bug cluster, so this should stay open until one canonical fix lands.

## What This Changes

Allow Feishu and other channel `message` tool sends with normal text/file attachments to proceed even when the shared schema or model adds stray poll parameters.

## Best Possible Solution

Land one canonical fix that strips or ignores stray poll creation params for `action="send"` while preserving explicit `action="poll"` validation; keep broader per-channel schema decomposition in https://github.com/openclaw/openclaw/issues/43015.

## Reproduction Assessment

Yes, at source level. Calling `runMessageAction()` on current main with `action="send"`, a normal message/file payload, and a non-empty poll creation field such as `pollQuestion`, `pollOption`, or `pollDurationSeconds` reaches the existing throw before channel dispatch.

## Solution Assessment

Yes for the narrow bug. The maintainable fix is action-aware sanitization or a narrower poll-intent check for stray poll params on `send`, while the broader per-channel or reduced-schema design belongs in https://github.com/openclaw/openclaw/issues/43015.
