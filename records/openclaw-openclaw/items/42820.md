## Summary

Keep open because current main still exposes poll fields on mixed-action message tool schemas and still throws the reported poll-action error before dispatch when a normal send carries non-empty poll-looking params. The related broader issue and an open fork PR are useful context, but neither has landed in openclaw/openclaw yet.

## What This Changes

The issue asks for `message` tool `send` requests, especially Feishu file sends, to stop being rejected by accidental poll creation fields from the shared schema.

## Best Possible Solution

Land a shared message-tool contract fix that keeps poll creation params out of ordinary `send` calls or strips accidental poll params before send dispatch, while preserving explicit `poll` validation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runMessageAction()` with `action: "send"`, a normal target/media payload, and a non-empty poll creation field such as `pollQuestion`; current main throws before Feishu or other channel dispatch.

## Solution Assessment

Yes for the narrow normalization/schema direction. The broader per-channel tool split can remain a follow-up; this bug can be fixed at the shared message-tool boundary without adding a new user-facing mode.
