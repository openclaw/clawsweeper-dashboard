## Summary

Keep open. Current `main` and the latest release still expose poll creation fields through the shared `message` tool schema and still reject `action: "send"` when non-empty poll-looking parameters are present before channel dispatch; an open closing PR remains the implementation path.

## What This Changes

The issue asks for Feishu and other `message` tool `action: "send"` calls to avoid being rejected when poll creation fields are present in the shared tool schema.

## Best Possible Solution

Land a reviewed fix that either strips accidental poll creation params from `send` or narrows the LLM-facing schema by action/channel, while preserving explicit `poll` validation and adding regression coverage for send-with-attachment calls carrying non-empty poll-looking params.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: call the shared message action runner with `action: "send"` plus a non-empty poll creation field such as `pollQuestion` or `pollDurationSeconds`, and current code deterministically throws before Feishu or Telegram handling.

## Solution Assessment

Yes, the best narrow solution belongs in the shared message-tool/outbound boundary rather than Feishu-specific code, because the same guard affects Telegram and related issue #43015. PR #46455 is the current implementation candidate, but it needs review/rebase or replacement before this issue should close.
