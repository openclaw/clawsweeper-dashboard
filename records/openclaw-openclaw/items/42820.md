## Label Justifications

- `P1`: The bug can block real outbound Feishu, Telegram, and related message/file sends before channel dispatch when models populate visible poll fields.
- `impact:message-loss`: Outbound messages can be rejected before delivery because stray poll-looking params trip the message tool guard.

## Summary

Keep open: current main and the latest release still reject `action="send"` before channel dispatch when non-empty poll-looking params are present, and the related repair work is still open rather than landed.

## What This Changes

The issue asks for ordinary message/file sends to ignore or avoid stray poll fields from the shared message tool schema instead of treating them as poll intent.

## Best Possible Solution

Land or replace a narrow upstream repair that makes stray/default poll params non-blocking for ordinary sends while preserving explicit `action="poll"` validation; keep broader schema decomposition tracked in https://github.com/openclaw/openclaw/issues/43015.

## Reproduction Assessment

Yes, at source level. A `runMessageAction()` call with `action="send"`, a normal message/file payload, and a non-empty poll creation field such as `pollQuestion`, `pollOption`, or non-zero `pollDurationSeconds` reaches the existing throw before channel dispatch.

## Solution Assessment

Yes for the narrow bug. The maintainable fix is action-aware sanitization or a narrower poll-intent check for stray/default poll params on `send`, while broader per-channel tool decomposition belongs in https://github.com/openclaw/openclaw/issues/43015.
