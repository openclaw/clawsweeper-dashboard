## Label Justifications

- `P1`: The bug can block real Feishu, Telegram, and other channel sends with attachments when models populate visible poll fields.
- `impact:message-loss`: The issue is about ordinary outbound messages being rejected before delivery because stray poll-looking params trip the message tool guard.

## Summary

Keep open because current main and the latest release still expose poll fields on mixed message tool schemas and still reject `action="send"` before Feishu, Telegram, or other channel dispatch when non-empty poll-looking params are present. The discussion includes a second user confirming the same guard on Telegram, and open repair PRs already target this cluster.

## What This Changes

Requests that `message` `action="send"` ignore or scope poll creation fields so Feishu and other channel file sends are not rejected by the poll guard.

## Best Possible Solution

Review and land one canonical fix that ignores or strips stray poll params for `send` while preserving explicit `poll` validation, and leave broader message-tool schema decomposition to https://github.com/openclaw/openclaw/issues/43015.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runMessageAction()` with `action="send"`, a valid message or file payload, and a non-empty poll creation field such as `pollQuestion`, `pollOption`, or non-zero `pollDurationSeconds`; current main throws before dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is action-aware sanitization or intent detection for stray poll params on `send`, while broader per-channel schema cleanup should stay in the companion issue.
