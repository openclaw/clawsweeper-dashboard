## Label Justifications

- `P1`: The current behavior can block real Feishu, Telegram, and other channel sends with attachments when models populate visible poll fields.
- `impact:message-loss`: The issue is about ordinary channel sends being rejected before delivery because stray poll-looking params trip the message tool guard.

## Summary

Keep open because current main still exposes poll fields on mixed message schemas and still rejects `action="send"` when non-empty poll-looking params are present before channel dispatch. The issue is also paired with open repair PRs, including https://github.com/BingqingLyu/openclaw/pull/708, and the broader companion remains open at https://github.com/openclaw/openclaw/issues/43015.

## What This Changes

The issue requests that plain `message` tool sends, especially Feishu send-with-file calls, ignore or avoid stray poll schema fields instead of failing with the poll-action guard.

## Best Possible Solution

Review, land, or replace one canonical fix that strips or ignores stray poll params for `send` while keeping explicit poll validation strict and tracking broader schema-shape cleanup in https://github.com/openclaw/openclaw/issues/43015.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runMessageAction()` with `action="send"`, a normal message or file payload, and a non-empty poll creation field such as `pollQuestion` or non-zero `pollDurationSeconds`; current main throws before channel dispatch.

## Solution Assessment

Yes. A narrow action-aware sanitizer or schema-scoping fix is the best path for the central bug, while broader per-channel tool-shape work can stay in the companion issue.
