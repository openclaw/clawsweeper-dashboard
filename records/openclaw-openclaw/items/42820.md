## Label Justifications

- `P1`: The bug can block real Feishu, Telegram, and other channel sends with attachments when models populate visible poll fields.
- `impact:message-loss`: The issue is about outbound messages being rejected before delivery because stray poll-looking params trip the message tool guard.

## Summary

Keep open because current main and the latest release still expose poll fields on mixed message tool schemas and still reject `action="send"` before Feishu, Telegram, or other channel dispatch when non-empty poll-looking params are present. The discussion includes a second user confirming the same guard on Telegram, and multiple open PRs already target this bug cluster.

## What This Changes

The issue asks for normal Feishu and other channel `message` sends with attachments to ignore stray poll schema fields instead of being rejected by the poll guard.

## Best Possible Solution

Review and land one canonical fix that strips or ignores stray poll creation params for `action="send"` while preserving explicit `action="poll"` validation; keep the broader per-channel or reduced-schema design discussion in https://github.com/openclaw/openclaw/issues/43015.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runMessageAction()` with `action="send"`, a normal message or file payload, and a non-empty poll creation field such as `pollQuestion`, `pollOption`, `pollDurationSeconds`, or `pollPublic`; current main throws before channel dispatch.

## Solution Assessment

Yes for the narrow bug. The maintainable fix is action-aware sanitization or intent detection for stray poll params on `send`, while broader message-tool schema decomposition should remain in the companion product-design issue.
