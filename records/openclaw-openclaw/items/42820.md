## Summary

Keep open because current main still exposes poll fields through the unified message tool and still rejects normal send requests when non-empty poll-looking params are present; target-repo fix attempts are closed unmerged, and the only open implementation candidate is still an unmerged fork PR.

## What This Changes

The issue asks for `message.send` to stop rejecting normal Feishu and other channel sends when shared poll fields are accidentally present.

## Best Possible Solution

Land or import a narrow shared message-tool/outbound fix that ignores or strips accidental poll creation params on `send`, preserves explicit `poll` validation, and leaves the broader per-channel tool-shape discussion to the related schema issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runMessageAction()` with `action: "send"`, a normal send target, and a non-empty poll creation field such as `pollQuestion`; current main throws before Feishu, Telegram, or plugin dispatch.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared message-action boundary because the same generic schema and guard affect Feishu, Telegram, and plugin send paths; per-channel tool splitting is a broader follow-up, not required for this bug fix.
