## Label Justifications

- `P1`: A shipped cross-channel regression can block ordinary text and attachment sends before they reach the transport.
- `impact:message-loss`: The false-positive poll guard suppresses intended outbound messages and files.

## Summary

Keep open: the latest release and current main partially ignore shared poll modifier defaults, but channel-specific schema-derived fields such as Telegram’s positive `pollDurationSeconds` still trigger the pre-dispatch guard and block ordinary sends. The linked repair PR remains open, conflicting, mixed with unrelated packaging work, and short of real channel proof, so it is not yet a safe resolution.

## What This Changes

Prevent schema-derived poll fields from blocking ordinary `message.send` calls across Feishu, Telegram, and other channels.

## Best Possible Solution

Use a focused shared-outbound intent check that ignores schema-derived poll metadata on `send` unless content-bearing poll intent is present, backed by Telegram-extra regression coverage and real Feishu or Telegram delivery proof.

## Reproduction Assessment

Yes at source level: a `send` payload containing the reported positive `pollDurationSeconds` reaches the current guard and is rejected before channel dispatch, and the June 1 report confirms this behavior on a released Telegram installation.

## Solution Assessment

Yes, a focused action-aware poll-intent check in the shared outbound path is the narrowest maintainable fix; splitting every channel into a separate message tool is a larger product redesign and is unnecessary for this regression.
