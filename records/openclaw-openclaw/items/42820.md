## Summary

Keep open because current main still exposes shared poll creation fields through the generic `message` tool and still rejects `action: "send"` when non-empty poll-looking params are present; open PR #46455 is the implementation path, not a pre-merge close reason.

## What This Changes

The issue requests that Feishu and other `message.send` calls with ordinary text or file attachments not be blocked by accidental poll-creation parameters from the shared tool schema.

## Best Possible Solution

Land a reviewed fix or replacement for #46455 that strips or scopes accidental poll creation params from `send`, preserves explicit `poll` validation, and adds regression coverage for send-with-attachment calls carrying non-empty poll-looking params.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: call `runMessageAction()` with `action: "send"`, a normal message or media target, and a non-empty poll creation field such as `pollQuestion` or `pollDurationSeconds`; current main deterministically throws before Feishu or Telegram dispatch.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared message-tool/outbound boundary rather than Feishu-specific code, because the same schema and guard affect multiple channels and related issue #43015.
