## Label Justifications

- `P1`: The report describes a live agent/channel workflow where ordinary message sends can be rejected before delivery.
- `impact:message-loss`: The affected behavior prevents requested outbound messages or attachments from being sent through the message tool.

## Summary

Keep open: current `main` and `v2026.6.6` still have a source-backed rejection path for normal sends when channel-extra poll fields such as `pollDurationSeconds` or `pollPublic` are present. https://github.com/openclaw/openclaw/pull/89601 shipped the shared-modifier part, but the remaining failure is still active and already has linked repair/canonical follow-up context.

## What This Changes

The issue asks for `message` tool `action: "send"` to stop failing when poll-related schema fields are auto-populated on non-poll sends, especially for Feishu and other channel sends with attachments.

## Best Possible Solution

Land or consolidate a focused action-aware poll-intent fix that keeps real `action: "poll"` validation strict while letting `action: "send"` ignore schema-padded poll extras; leave the broader tool-schema split/components design with https://github.com/openclaw/openclaw/issues/43015.

## Reproduction Assessment

Yes from source, though I did not run a live Feishu/Telegram send in this read-only review. A current test-shaped `runDrySend` payload with `action: "send"`, `pollDurationSeconds`, and `pollPublic` still reaches the guard and is expected to throw before dispatch.

## Solution Assessment

No for current `main`: it is a useful partial fix for shared poll modifiers, not a complete fix for this issue. The maintainable end state is either a narrow action-aware poll-intent guard/sanitizer or a maintainer-approved schema split, with regression coverage for channel-extra poll fields on non-poll sends.
