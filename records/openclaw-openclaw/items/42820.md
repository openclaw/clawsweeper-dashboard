## Summary

Keep open because current main still exposes poll fields on mixed-action message tool schemas and still throws the reported poll-action error before Feishu or other channel dispatch when a normal send carries non-empty poll-looking params. The open fork PR and broader issue are useful implementation/context paths, but neither has landed in openclaw/openclaw.

## What This Changes

The issue requests that ordinary `message` tool `action: "send"` calls with attachments avoid being rejected when poll fields are present in the shared schema.

## Best Possible Solution

Land or replace the shared message-tool normalization fix so stray poll params do not block plain sends while explicit poll creation remains validated; keep the broader per-channel tool split as separate product work if maintainers want it.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runMessageAction()` with `action: "send"`, a normal target/media payload, and a non-empty poll creation field such as `pollQuestion` or `pollDurationSeconds`; current main throws before channel dispatch.

## Solution Assessment

Yes for the narrow normalization/schema direction. The best fix is to keep poll fields out of ordinary send calls or strip accidental poll params before send dispatch while preserving explicit `poll` validation; the per-channel tool split can remain a follow-up product discussion.
