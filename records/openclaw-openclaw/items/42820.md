## Label Justifications

- `P1`: The false-positive poll guard can block ordinary outbound text and attachment sends before they reach active channels.
- `impact:message-loss`: The failure suppresses intended user-visible messages and files at the shared message-tool layer.

## Summary

Keep open: current main and v2026.6.6 still reject normal `send` calls when channel-specific poll metadata such as `pollDurationSeconds` is present, so the Feishu/Telegram/Discord failure class is not fully fixed. The merged shared-modifier fix in https://github.com/openclaw/openclaw/pull/89601 was useful but partial, while the remaining repair PRs are still open and conflicting.

## What This Changes

The issue asks OpenClaw to let `message(action="send")` deliver ordinary messages and files even when shared or channel-specific poll fields are schema-padded into the tool payload.

## Best Possible Solution

Land one rebased, focused send/poll-intent repair that ignores metadata-only poll fields on normal sends, preserves explicit poll creation validation, and proves real Feishu/Telegram/Discord delivery; keep broader schema redesign under https://github.com/openclaw/openclaw/issues/43015.

## Reproduction Assessment

Yes at source level: a current-main `send` payload with channel-specific poll metadata such as `pollDurationSeconds: 60` reaches `hasPollCreationParams` and throws before dispatch. The issue and related comments also provide released Telegram/Discord/Feishu production reports.

## Solution Assessment

Yes for the narrow bug: the best fix is a shared outbound intent check that treats only non-empty poll question/options as send-time poll intent while keeping actual `poll` validation strict. Splitting every channel into separate tools is a larger product redesign, not required to repair this regression.
