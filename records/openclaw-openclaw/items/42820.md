## Summary

Keep open because current main still exposes shared poll creation fields on the generic `message` tool and still rejects `action: "send"` when those fields are non-empty; the linked open PR is the implementation path, not a close reason before merge.

## What This Changes

The issue requests that ordinary `message` sends with attachments stop being rejected when shared poll fields are present in the tool payload.

## Best Possible Solution

Land a reviewed fix or replacement for #46455 that strips or scopes accidental poll params from `send`, preserves explicit `poll` validation, and adds regression coverage for send-with-attachment calls carrying non-empty poll-looking params.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: call `runMessageAction()` with `action: "send"`, a normal message or media target, and a non-empty poll creation field such as `pollQuestion` or `pollDurationSeconds`; current main deterministically throws before Feishu or Telegram dispatch.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the shared message-tool/outbound boundary rather than Feishu-specific code, because the same schema and guard affect multiple channels and related issue #43015.
