## Label Justifications

- `P1`: The bug can block real outbound Feishu, Telegram, and related file/message sends before channel dispatch when models populate visible poll fields.
- `impact:message-loss`: Intended outbound messages or attachments can be rejected before delivery because stray poll-looking params trip the message tool guard.

## Summary

Keep open: current main and the latest release still expose poll fields through the broad message tool schema and still reject normal `send` calls before channel dispatch when non-empty poll-looking params are present; related repair PRs remain open rather than landed.

## What This Changes

The issue asks for Feishu and other channel `message.send` file sends to avoid failing when stray poll parameters appear in the shared message tool payload.

## Best Possible Solution

Land or replace one narrow, maintainer-approved repair that makes stray/default poll params non-blocking for ordinary sends while preserving explicit `action="poll"` validation; keep broader tool-shape work in https://github.com/openclaw/openclaw/issues/43015.

## Reproduction Assessment

Yes, at source level. A `runMessageAction()` call with `action="send"`, normal message/file payload fields, and a non-empty poll creation field such as `pollQuestion`, `pollOption`, or non-zero `pollDurationSeconds` reaches the existing throw before channel dispatch.

## Solution Assessment

Yes for the narrow bug. The maintainable fix is action-aware sanitization or a narrower poll-intent check for stray/default poll params on `send`, while broader per-channel tool decomposition belongs in https://github.com/openclaw/openclaw/issues/43015.
