## Summary

Keep open because current main still exposes poll fields on mixed-action message tool schemas and still rejects ordinary `send` requests before channel dispatch when non-empty poll-looking params are present. The report is source-reproducible, confirmed by related discussion for Telegram as well as Feishu, and there are open fix paths that have not landed.

## What This Changes

Request that `message` tool `action: "send"` for Feishu and other channels not be blocked by accidental poll creation fields from the shared schema.

## Best Possible Solution

Land or replace the normalization fix so stray poll params do not block plain sends, while preserving explicit `poll` validation; keep the broader per-channel schema split under the open broader issue if maintainers want that product direction.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runMessageAction()` with `action: "send"`, a normal message/media payload, and a non-empty poll creation field such as `pollQuestion` or `pollDurationSeconds`; current main throws before channel dispatch.

## Solution Assessment

Yes for the narrow action-aware sanitization or send-schema narrowing direction. The current hard pre-dispatch rejection is the behavior to replace, while the full per-channel tool split is broader follow-up work.
