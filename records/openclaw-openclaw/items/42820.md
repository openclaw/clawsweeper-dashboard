## Label Justifications

- `P1`: The bug can block real Feishu, Telegram, and other channel sends with attachments when models populate visible poll fields.
- `impact:message-loss`: Outbound messages can be rejected before delivery because stray poll-looking params trip the message tool guard.

## Summary

Keep open: current main and latest release still expose poll parameters on mixed message tool schemas, and `action="send"` still rejects before channel dispatch when non-empty poll-looking fields are present. The discussion has a second user confirming the same guard on Telegram, and several open PRs already target this bug cluster.

## What This Changes

The issue requests that normal Feishu and other channel `message(action="send")` calls with files not be blocked by stray poll creation fields from the shared message tool schema.

## Best Possible Solution

Land one canonical fix that strips or ignores stray poll creation params for `action="send"` while preserving explicit `action="poll"` validation; keep broader schema decomposition in https://github.com/openclaw/openclaw/issues/43015.

## Reproduction Assessment

Yes, at source level. Calling `runMessageAction()` on current main with `action="send"`, a normal message/file payload, and a non-empty poll creation field such as `pollQuestion`, `pollOption`, or `pollDurationSeconds` reaches the existing throw before channel dispatch.

## Solution Assessment

Yes for the narrow bug. The maintainable fix is action-aware sanitization or a narrower poll-intent check for stray poll params on `send`, while the broader per-channel or reduced-schema design should stay in the companion product-design issue.
