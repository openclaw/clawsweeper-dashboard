## Summary

Keep open: current main and v2026.6.10 fix the reported metadata-default poll false positive, but Feishu still advertises a multi-action message tool while the shared schema includes poll fields and the send path still rejects content-bearing poll params before delivery. The remaining question is a maintainer-owned schema/runtime contract decision, not a safe cleanup close.

## What This Changes

The issue asks OpenClaw to prevent shared poll schema fields from breaking normal `message` tool `action: "send"` calls with Feishu or other channel attachments.

## Best Possible Solution

Keep the shipped metadata-padding fix, then have maintainers choose and land a durable message-tool contract: action-scoped schemas, send-path sanitization, or an explicit decision that content-bearing poll params should keep failing on `send`.

## Reproduction Assessment

No for a full current-main live user repro: the metadata-default failure is fixed in v2026.6.10. Yes at source level for the remaining contract: Feishu still gets shared poll fields in a multi-action schema, and `send` still rejects non-empty poll content fields before delivery.

## Solution Assessment

Unclear. The shipped validator narrowing is the right fix for metadata padding, but the broader request needs a maintainer choice between narrowing schemas, stripping accidental content fields, or preserving strict poll-intent validation.
