## Label Justifications

- `P1`: The bug can block real Feishu, Telegram, and other channel sends with attachments when models populate visible poll fields.
- `impact:message-loss`: Outbound messages can be rejected before delivery because stray poll-looking params trip the message tool guard.

## Summary

Keep open: current main and the latest release still reject `action="send"` when non-empty poll-looking fields are present before channel dispatch, and the discussion includes corroborating Telegram evidence plus an open linked fix PR.

## What This Changes

The issue asks for normal Feishu `message` send/file delivery to work even when stray poll creation fields are present in the shared message-tool schema.

## Best Possible Solution

Land one canonical narrow fix that strips or ignores stray poll creation params for `action="send"` while preserving explicit `action="poll"` validation; keep the broader per-channel schema decomposition in https://github.com/openclaw/openclaw/issues/43015.

## Reproduction Assessment

Yes, at source level. A `runMessageAction()` send call with normal message/file params and a non-empty poll creation field such as `pollQuestion`, `pollOption`, or `pollDurationSeconds` reaches the existing throw before channel dispatch.

## Solution Assessment

Yes for the narrow bug. The maintainable path is action-aware sanitization or a narrower poll-intent check for stray poll params on `send`, while the broader schema-shape work should remain in https://github.com/openclaw/openclaw/issues/43015.
