## Summary

Keep open: current main still supports only `off`, `non-main`, and `all`, and sandboxed embedded turns resolve the sandbox backend before tool-specific behavior is known. The requested on-demand mode is plausible, but it is a new security-sensitive config/runtime contract rather than a conservative cleanup close.

## What This Changes

Add an on-demand `agents.defaults.sandbox.mode: "needed"` so chat-only turns do not preflight Docker while sandbox-bound tools still fail closed.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and tests so Docker is required only when sandbox-bound execution begins and those tools still fail closed.

## Reproduction Assessment

Yes, at source level: with `mode: "all"`, current main marks the session sandboxed, resolves `resolveSandboxContext` before tool construction, and the Docker backend ensures a container during context creation. I did not perform a live Docker-stop Telegram run.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible direction for the outage, but the exact host-versus-sandbox boundary must be approved before implementation.
