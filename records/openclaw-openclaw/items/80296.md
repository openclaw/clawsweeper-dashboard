## Summary

Keep open: current main and the latest release still lack the requested `needed` sandbox mode, and the linked implementation candidate is open, draft, and conflicting rather than merged. Source inspection still supports the reported failure shape: `all` marks the session sandboxed, embedded runs resolve the sandbox context before model setup, and the Docker backend is created eagerly.

## What This Changes

The issue requests a third `agents.defaults.sandbox.mode` value that skips Docker preflight for chat-only turns while keeping sandbox-bound tools fail-closed when Docker is unavailable.

## Best Possible Solution

Review and either land or replace the linked candidate PR with an explicit lazy sandbox mode that leaves defaults unchanged, skips backend setup for chat-only turns, and keeps exec/apply_patch/sandbox-bound tools fail-closed with focused tests and live Docker-off proof.

## Reproduction Assessment

Yes by source inspection, not by a live Docker-off run. Current main still marks `all` sessions as sandboxed and resolves the sandbox backend before the embedded model turn continues, which makes backend availability part of even chat-only turns.

## Solution Assessment

Unclear as final product direction. A new `needed` mode is plausible and has an open candidate PR, but maintainers still need to confirm the security contract so sandbox-bound tools fail closed instead of falling back to host execution.
