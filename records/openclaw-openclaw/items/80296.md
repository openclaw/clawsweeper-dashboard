## Summary

Keep open: current main and the latest shipped release still only support session-scoped sandbox modes, and `mode: "all"` still resolves a sandbox backend before the embedded attempt proceeds. The requested on-demand mode is plausible, but it is a new security-sensitive config/runtime contract rather than conservative cleanup.

## What This Changes

Add an on-demand sandbox mode so chat-only turns can proceed when Docker is unavailable while sandbox-bound tools still fail closed.

## Best Possible Solution

Define an approved on-demand sandbox contract, then update schema, exported types, runtime routing, docs, and regression tests so Docker is required only when sandbox-bound tool execution begins and those tools fail closed.

## Reproduction Assessment

Yes, at source level: with `mode: "all"`, a non-empty session is sandboxed, the embedded runner calls `resolveSandboxContext` before the attempt, and the Docker backend reports daemon unavailability as a hard sandbox error. I did not run a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear pending maintainer design: a `needed` mode is a plausible narrow fix for the outage, but the security boundary must be approved before implementation. The safer end state is an explicit fail-closed contract for sandbox-bound tools rather than silently falling back to host execution.
