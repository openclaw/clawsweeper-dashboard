## Summary

Keep open: current main still lacks an on-demand sandbox mode, and `mode: "all"` still resolves sandbox context/backend before the embedded model/tool turn. The request is useful and source-reproducible, but it changes a security-sensitive config/runtime contract and needs maintainer/security design rather than conservative cleanup.

## What This Changes

Request adding `agents.defaults.sandbox.mode: "needed"` so chat-only turns can proceed without Docker while sandbox-bound tools still require an available sandbox backend.

## Best Possible Solution

Define an approved on-demand or tool-placement sandbox contract, then update schema, exported types, runtime routing, docs, and regression tests so chat-only turns can proceed without Docker while sandbox-bound tools fail closed.

## Reproduction Assessment

Yes, at source level: with `mode: "all"`, current main marks the session sandboxed, preflights `resolveSandboxContext`, and resolves the backend before the model/tool attempt. I did not run a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear pending maintainer/security design: `needed` directly targets the outage, but the host-vs-sandbox boundary needs approval before implementation. A safe solution should coordinate with broader tool-placement work and make sandbox-bound tools fail closed.
