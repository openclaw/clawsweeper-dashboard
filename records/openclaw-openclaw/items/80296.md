## Summary

Keep open. Current main does not implement `mode: "needed"`; the schema supports `off`, `non-main`, and `all`, and source inspection backs the report that `all` resolves a Docker-backed sandbox before the turn proceeds. Because the requested behavior changes sandbox/security configuration semantics, it needs maintainer product and security review rather than cleanup closure.

## What This Changes

Add `agents.defaults.sandbox.mode: "needed"` so chat-only turns can run without Docker while sandbox-bound tools still require a sandbox.

## Best Possible Solution

If maintainers want this availability tradeoff, add a documented lazy-sandbox mode across schema, types, docs, and embedded-runner/tool setup with regression coverage for chat-only, file-tool, and exec failure paths.

## Reproduction Assessment

Yes, source-reproducible for the reported current behavior: `mode: "all"` marks every session sandboxed, and embedded runs resolve a Docker sandbox context before the model/tool loop. I did not run a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear. `mode: "needed"` is a plausible direction, but it is not a purely mechanical bug fix because it changes the sandbox contract for chat, filesystem, MCP, and tool routing.
