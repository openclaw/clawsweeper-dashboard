## Summary

Keep open: current main and the latest release still only support session-scoped sandbox modes, and sandboxed embedded turns still resolve a backend before the model/tool loop. The requested on-demand mode is a plausible reliability improvement, but it is a new security-sensitive config/runtime contract rather than a conservative cleanup close.

## What This Changes

The issue requests a new `agents.defaults.sandbox.mode: "needed"` mode that defers Docker sandbox use until a sandbox-bound tool is invoked.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and focused tests so Docker is required only when sandbox-bound tools begin and those tools still fail closed.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks the session sandboxed, the embedded runner resolves `resolveSandboxContext` before the model attempt, and the Docker backend ensures a container during context creation. I did not perform a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible narrow direction for the outage, but the fail-closed host-versus-sandbox boundary must be approved before implementation.
