## Summary

Keep open: current main still has only the existing sandbox modes and resolves a Docker-backed sandbox context before the embedded model/tool loop when `mode` is `all`. The requested on-demand mode is plausible, but it is a new security-sensitive config and runtime contract rather than conservative cleanup.

## What This Changes

Request adding `agents.defaults.sandbox.mode: "needed"` so chat-only turns can proceed without Docker while sandbox-bound tools still require Docker and fail closed when unavailable.

## Best Possible Solution

Define an explicit on-demand sandbox threat model, then update schema/types/runtime/docs/tests so Docker is deferred until sandbox-bound tool execution while exec, write, edit, `apply_patch`, process, and sandboxed filesystem paths remain fail-closed.

## Reproduction Assessment

Yes, at source level: with sandbox `mode: "all"`, current main marks the session sandboxed and the embedded runner resolves sandbox context before the model/tool attempt. I did not perform a live Docker-stop Telegram reproduction in this read-only pass.

## Solution Assessment

Unclear pending maintainer/security design: `needed` is a plausible narrow answer to the outage, but host-versus-sandbox routing for filesystem tools, MCP calls, and sandbox-bound tools needs an explicit contract before implementation.
