## Summary

Keep open: current main still has only session-scoped sandbox modes and resolves a sandbox runtime before the embedded model/tool loop for sandboxed sessions. The requested on-demand mode is plausible but is a new security-sensitive config/runtime contract, not a conservative cleanup close.

## What This Changes

The issue requests an `agents.defaults.sandbox.mode: "needed"` mode so chat-only turns can proceed when Docker is down while sandbox-bound tools still fail closed.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and tests so Docker is required only when sandbox-bound tool execution begins and those tools still fail closed.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks the session sandboxed, the embedded runner resolves `resolveSandboxContext` before the model/tool attempt, and the Docker backend ensures a container during context creation. I did not perform a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible narrow direction for the outage, but the exact security boundary must be approved before implementation.
