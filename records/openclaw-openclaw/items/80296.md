## Summary

Keep open: current main still exposes only the existing sandbox modes and eagerly resolves a sandbox runtime for `all` sessions before the embedded turn proceeds. The requested on-demand mode is plausible, but it is a new security-sensitive config/runtime contract rather than a cleanup-close candidate.

## What This Changes

Requests a new `agents.defaults.sandbox.mode: "needed"` that defers sandbox backend setup until sandbox-bound tools are invoked while letting chat-only turns proceed.

## Best Possible Solution

Define an explicit on-demand sandbox contract, then update schema, types, docs, runtime setup, and focused tests so sandbox-bound tools fail closed while chat-only turns can proceed without Docker.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks the session sandboxed, and the embedded runner resolves `resolveSandboxContext` before the turn proceeds to model/tool execution. I did not perform a live Docker-stop Telegram reproduction in this read-only pass.

## Solution Assessment

Unclear pending maintainer/security design: `needed` is a plausible narrow user-facing answer, but the exact host-versus-sandbox routing rules need approval before implementation.
