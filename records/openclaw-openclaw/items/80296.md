## Summary

Keep open: current main still has only `off`, `non-main`, and `all`, and `all` resolves the sandbox backend before the embedded runner builds the model turn, so the reported chat-only outage remains source-reproducible. The requested `needed` mode is plausible but changes security-sensitive config/runtime semantics, so it needs maintainer design review rather than automatic cleanup.

## What This Changes

The issue requests a third sandbox mode that only requires Docker when a sandbox-bound tool is invoked, allowing chat-only turns to continue while Docker is temporarily unavailable.

## Best Possible Solution

Design an explicit lazy sandbox/tool execution mode that preserves secure defaults, never routes sandbox-bound tools to the host on Docker failure, updates config/schema/docs, and covers chat-only success plus sandbox-bound tool failure paths.

## Reproduction Assessment

Yes by source inspection, but not by live execution. With `sandbox.mode=all`, current main resolves the sandbox backend before the embedded model turn, and the Docker backend must create or reuse a container before a chat-only response can be produced.

## Solution Assessment

Unclear as a final design. A `needed` mode is a plausible direction, but it adds a new security-sensitive config behavior and must be designed against existing `off`/`non-main`/`all`, tool host routing, and Codex command-level sandbox contracts.
