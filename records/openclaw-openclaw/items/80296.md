## Summary

Keep open: current main still has only session-scoped sandbox modes and still resolves the sandbox backend before the embedded model/tool loop for sandboxed sessions. The request is a plausible reliability improvement, but it is a new security-sensitive config/runtime contract rather than conservative cleanup.

## What This Changes

Requested behavior is an on-demand sandbox mode so chat-only turns can proceed when Docker is unavailable while sandbox-bound tools still fail closed.

## Best Possible Solution

Define an approved on-demand sandbox contract, then update schema, exported types, runtime routing, docs, and regression tests so Docker is required only when sandbox-bound execution begins and those paths still fail closed.

## Reproduction Assessment

Yes, at source level: with `mode: "all"`, any non-empty session is treated as sandboxed, the embedded runner resolves `resolveSandboxContext` before the model/tool attempt, and the Docker backend initializes a container during context setup. I did not run a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible narrow direction for the outage, but the exact security boundary must be approved before implementation. A safe solution should make sandbox-bound tools fail closed without silently moving execution to the host.
