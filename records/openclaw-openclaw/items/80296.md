## Summary

Keep open: current main still has only session-scoped sandbox modes and still builds the sandbox backend before the embedded turn proceeds for sandboxed sessions. The requested on-demand mode is plausible, but it is a new security-sensitive config/runtime contract rather than a conservative cleanup close.

## What This Changes

Add `agents.defaults.sandbox.mode: "needed"` so chat-only turns can proceed while Docker is down, while sandbox-bound tools still require Docker and fail closed.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and tests so Docker is required only when sandbox-bound tool execution begins and those tools still fail closed.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks the session sandboxed, the embedded runner resolves `resolveSandboxContext` before the model/tool attempt, and the Docker backend ensures a container during context creation. I did not perform a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible direction for the outage, but the exact security boundary must be approved before implementation.
