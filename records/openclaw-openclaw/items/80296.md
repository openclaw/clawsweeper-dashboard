## Summary

Keep open: current main still has only session-scoped sandbox modes and still resolves the configured sandbox backend before an embedded turn reaches model/tool execution. The requested `needed` mode is plausible, but it is a new security-sensitive config/runtime contract rather than conservative cleanup.

## What This Changes

Request adds a third `agents.defaults.sandbox.mode: "needed"` value so chat-only turns can proceed when Docker is down while sandbox-bound tools still require Docker.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and tests so Docker is required only when sandbox-bound tool execution begins and those tools still fail closed.

## Reproduction Assessment

Yes, at source level: current main makes `mode: "all"` sandbox every session and resolves the sandbox backend before the embedded runner reaches model/tool work. I did not run a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible direction for the outage, but the exact security boundary and host fallback semantics must be approved before implementation.
