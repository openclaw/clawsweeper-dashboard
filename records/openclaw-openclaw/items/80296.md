## Summary

Keep open: current main still has only `off`, `non-main`, and `all`, and `all` resolves a sandbox backend before the embedded model turn continues. The requested lazy `needed` behavior is not implemented and needs maintainer security/product design before code work.

## What This Changes

The issue requests a third sandbox mode that only requires Docker when sandbox-bound tools are used, so chat-only turns can continue while Docker is unavailable.

## Best Possible Solution

Design an explicit lazy sandbox/tool execution mode that keeps existing defaults compatible, skips Docker preflight for chat-only turns, and returns clear fail-closed errors for sandbox-bound tools when the backend is unavailable.

## Reproduction Assessment

Yes by source inspection, not by live execution. Current main still resolves a sandbox context before the embedded model turn continues, so `sandbox.mode="all"` makes backend availability part of even a chat-only turn.

## Solution Assessment

Unclear as final product direction. A new `needed` mode is plausible, but maintainers need to define the security contract so sandbox-bound tools fail closed instead of falling back to host execution.
