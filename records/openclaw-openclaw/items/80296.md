## Summary

Keep open: current main still has no on-demand sandbox mode, and `mode: "all"` resolves a sandbox backend before the embedded model/tool loop. The requested behavior is plausible, but it adds a new security-sensitive config/runtime contract rather than a conservative cleanup close.

## What This Changes

Request adding `agents.defaults.sandbox.mode: "needed"` so chat-only turns can proceed when Docker is down while sandbox-bound tools still require sandbox isolation.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema/types/runtime/docs/tests so Docker is required only when a sandbox-bound capability executes and those capabilities fail closed when the backend is unavailable.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks every session sandboxed, and the embedded runner resolves sandbox context before model/tool execution. I did not perform the live Docker-stop Telegram reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible narrow answer to the outage, but host-versus-sandbox routing rules need explicit security/product approval before implementation.
