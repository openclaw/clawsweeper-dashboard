## Summary

Keep open: current main still has only session-scoped sandbox modes and still resolves a sandbox backend before the embedded model/tool loop for sandboxed sessions, while the requested on-demand mode is a new security-sensitive config/runtime contract rather than conservative cleanup.

## What This Changes

The issue requests an on-demand sandbox mode that lets chat-only turns proceed when Docker is unavailable while still sandboxing exec and other sandbox-bound tools.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and tests so Docker is required only when sandbox-bound tool execution begins and those tools still fail closed.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks the session sandboxed, the embedded runner resolves `resolveSandboxContext` before the model attempt, and the Docker backend ensures a container during context creation. I did not perform a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear pending maintainer/security design: `needed` is a plausible direction for the outage, but the exact host-versus-sandbox boundary must be approved before implementation.
