## Summary

Keep open: current main still only supports session-scoped `off`, `non-main`, and `all` sandbox modes, and sandboxed embedded turns still resolve the backend before the model/tool loop. The requested on-demand behavior is plausible, but it is a new security-sensitive config/runtime contract rather than conservative cleanup.

## What This Changes

Request adds an on-demand `agents.defaults.sandbox.mode: "needed"` so chat-only turns can proceed without Docker while sandbox-bound tools still fail closed.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and tests so Docker is required only when sandbox-bound tool execution begins and those tools still fail closed.

## Reproduction Assessment

Yes, at source level. With `mode: "all"`, current main marks the session sandboxed, resolves `resolveSandboxContext` before the embedded turn proceeds, and the Docker backend ensures a container during context creation.

## Solution Assessment

Unclear until maintainers define the security contract. A `needed` mode is a plausible direction for the outage, but the exact host-versus-sandbox boundaries must be approved before implementation.
