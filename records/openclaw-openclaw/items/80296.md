## Summary

Keep open: current main still has no on-demand sandbox mode, and the source path still resolves a sandbox backend before the embedded model/tool loop for sandboxed sessions. This is a plausible outage fix, but it adds a new config value and security-sensitive runtime semantics, so it is not conservative cleanup-close material.

## What This Changes

The issue requests a new `agents.defaults.sandbox.mode: "needed"` that avoids requiring Docker for chat-only turns while preserving sandbox enforcement for sandbox-bound tools.

## Best Possible Solution

Define an approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and regression coverage so Docker is required only when sandbox-bound execution begins and those tools still fail closed.

## Reproduction Assessment

Yes, at source level: `mode: "all"` makes the session sandboxed, the embedded runner resolves `resolveSandboxContext` before the attempt proceeds, and the Docker backend ensures a container during context creation. I did not run a live Telegram/Docker-stop repro in this read-only review.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible narrow direction for the outage, but the security boundary must be defined before implementation. A safe solution should be explicit about which tools may run without Docker and which fail closed when the sandbox backend is unavailable.
