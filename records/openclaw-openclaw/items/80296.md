## Summary

Keep open: current main still has no on-demand sandbox mode and still resolves the configured sandbox backend before an embedded turn proceeds when `mode: "all"` is active. The requested behavior is plausible, but it is a new sandbox config and security contract rather than conservative cleanup.

## What This Changes

Requests a new `agents.defaults.sandbox.mode: "needed"` value so chat-only turns can continue without Docker while sandbox-bound tools still require the sandbox backend.

## Best Possible Solution

Define a reviewed on-demand sandbox contract, then update schema, runtime routing, docs, and regression coverage so sandbox-bound tools fail closed when the backend is unavailable while non-tool chat can proceed.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks the session sandboxed, the embedded runner resolves sandbox context before the attempt proceeds, and the Docker backend ensures a container during context creation. I did not run a live Docker-stopped Telegram turn.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible direction for the outage, but the exact host-versus-sandbox boundary must be approved before implementation.
