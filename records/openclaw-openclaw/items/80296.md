## Summary

Keep open: current main still only supports the existing sandbox modes, and `all` resolves a sandbox runtime before the embedded model/tool loop. The requested on-demand sandbox mode is plausible, but it is a new security-sensitive config and runtime contract, not a conservative cleanup close.

## What This Changes

Add an `agents.defaults.sandbox.mode` value that only requires the sandbox when a sandbox-bound tool is actually invoked, so chat-only turns can continue while Docker is briefly unavailable.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema/types/docs/tests so Docker is required only at sandbox-bound tool execution while preserving fail-closed behavior for exec, write/edit/apply_patch, process, and sandboxed file paths.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks the session sandboxed and the embedded runner resolves the sandbox context before the turn proceeds. I did not perform a live Docker-stop Telegram reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible narrow answer to the outage, but the exact host-versus-sandbox routing rules need explicit security/product approval before implementation.
