## Summary

Keep open: current main still lacks an on-demand sandbox mode, and the reported outage path is credible from source inspection. The requested `needed` mode is a new security-sensitive config/runtime contract, so it needs maintainer design rather than conservative cleanup closure.

## What This Changes

Add an `agents.defaults.sandbox.mode: "needed"` option so chat-only turns can proceed when Docker is down while sandbox-bound tools still fail closed.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and focused tests so Docker is required only when sandbox-bound tool execution begins.

## Reproduction Assessment

Yes, at source level: with `mode: "all"`, the embedded runner resolves a Docker-backed sandbox context before model/tool execution, so stopping Docker can block even a chat-only turn. I did not perform the live Telegram/Docker-stop reproduction.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible direction, but it is only the best solution if maintainers define precise fail-closed host-versus-sandbox semantics first.
