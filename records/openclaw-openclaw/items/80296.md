## Summary

Keep open: current main still only supports the existing sandbox modes and still resolves a sandbox context before the embedded model/tool attempt proceeds. The requested on-demand mode is plausible, but it is a new security-sensitive config and runtime contract, not a conservative cleanup close.

## What This Changes

Request adding `agents.defaults.sandbox.mode: "needed"` so chat-only turns can proceed without Docker while sandbox-bound tools still fail closed when Docker is unavailable.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, exported types, docs, explain/doctor output, and tests so Docker is required only when a sandbox-bound operation is actually executed and those operations still fail closed.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks the session sandboxed and the embedded runner resolves the sandbox context before the model/tool attempt. I did not live-stop Docker or run a Telegram turn in this read-only pass.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible narrow answer to the outage, but adding it changes the public sandbox security contract. The safer path is an explicit on-demand contract with fail-closed tool boundaries and regression coverage.
