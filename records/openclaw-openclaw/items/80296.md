## Summary

Keep open: current main still has only the existing sandbox modes, and `mode: "all"` resolves a sandbox context before the model/tool loop. The requested on-demand mode is plausible, but it is a new security-sensitive config/runtime contract rather than conservative cleanup.

## What This Changes

Request adds an on-demand `agents.defaults.sandbox.mode: "needed"` so chat-only turns can proceed without Docker while sandbox-bound tools still fail closed.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, exported types, runtime resolution, docs, and regression tests so Docker is required only at sandbox-bound tool execution.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks the session sandboxed and the embedded runner resolves Docker-backed sandbox context before the turn proceeds. I did not perform a live Docker-stop Telegram reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible narrow answer to the outage, but the exact routing rules need security/product approval before implementation.
