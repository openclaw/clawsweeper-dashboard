## Summary

Keep open: current main still lacks an on-demand sandbox mode and still resolves the sandbox backend before the embedded model/tool loop for `mode: "all"`. The requested behavior is plausible, but it adds a security-sensitive config/runtime contract rather than a conservative cleanup fix.

## What This Changes

Request adding `agents.defaults.sandbox.mode: "needed"` so chat-only turns can proceed while Docker is unavailable and sandbox-bound tools still fail closed.

## Best Possible Solution

Define an approved on-demand sandbox contract, then update schema/types, runtime tool routing, docs, and regression coverage so sandbox-bound tools fail closed while pure chat avoids Docker preflight.

## Reproduction Assessment

Yes, at source level: with `mode: "all"`, current main marks the session sandboxed and resolves the sandbox backend before the model attempt, so Docker daemon unavailability can reject even a chat-only turn. I did not run a live Telegram/Docker-stopped repro.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible direction, but the exact fail-closed security boundary must be approved before implementation. A safe fix should defer Docker only until sandbox-bound work begins, without silently moving protected tools to the host.
