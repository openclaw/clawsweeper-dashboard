## Summary

Keep open: current main still has only session-scoped sandbox modes, and `mode: "all"` still resolves the sandbox backend before the embedded model/tool attempt. The requested on-demand mode remains a plausible reliability improvement, but it is a new security-sensitive config/runtime contract rather than a conservative cleanup close.

## What This Changes

The issue requests an on-demand sandbox mode so chat-only turns can proceed when Docker is unavailable while sandbox-bound tools still fail closed.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and tests so Docker is required only when sandbox-bound tool execution begins and those tools still fail closed.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks every session sandboxed, the embedded runner resolves `resolveSandboxContext` before the model/tool attempt, and the Docker backend ensures a container during context creation. I did not perform a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible narrow direction for the outage, but the exact security boundary must be approved before implementation.
