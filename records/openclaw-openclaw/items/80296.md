## Summary

Keep open: current main still has no on-demand sandbox mode, and `mode: "all"` still resolves a sandbox backend before the embedded attempt can proceed. The request is valid at source level, but it needs maintainer-approved security semantics for a new config/runtime contract.

## What This Changes

The issue requests a third sandbox mode that only requires Docker when sandbox-bound tools are invoked, so chat-only turns are not blocked by Docker downtime.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and tests so chat-only turns do not preflight Docker while sandbox-bound tools still fail closed.

## Reproduction Assessment

Yes, at source level: current main still resolves a sandbox context before embedded attempts under `mode: "all"`, and the Docker backend ensures a container during that context creation. I did not perform a live Docker-stop Telegram run.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible direction for reducing outages, but the exact host-versus-sandbox boundaries need security review before implementation.
