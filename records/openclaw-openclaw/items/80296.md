## Summary

Keep open: current main still lacks an on-demand sandbox mode, and `all` still resolves the sandbox backend before the embedded runner reaches tool construction. The report is actionable, but it is a new security-sensitive config/runtime contract rather than a conservative cleanup close.

## What This Changes

Request to add `agents.defaults.sandbox.mode: "needed"` so chat-only turns can proceed when Docker is down while sandbox-bound tools still fail closed.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, types, runtime routing, docs, and focused tests so sandbox-bound tools still fail closed when the backend is unavailable.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks the session sandboxed, the embedded runner resolves `resolveSandboxContext` before tool construction, and the Docker backend ensures a container during context creation. I did not run a live Telegram/Docker-stop reproduction.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible direction for the outage, but the exact host-versus-sandbox boundary must be approved before implementation.
