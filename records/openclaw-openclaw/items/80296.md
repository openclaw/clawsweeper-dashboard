## Summary

Keep open: current main still lacks an on-demand sandbox mode, and the current `all` mode still resolves the sandbox backend before the embedded model/tool loop. The request is valid at source level, but it is a new security-sensitive config/runtime contract rather than conservative cleanup.

## What This Changes

The issue requests a new sandbox mode that delays Docker/sandbox backend use until sandbox-bound tools execute, so chat-only turns can continue during Docker downtime.

## Best Possible Solution

Define an approved on-demand sandbox contract, then update schema, exported types, runtime routing, docs, and regression tests so Docker is required only when sandbox-bound execution begins and those paths fail closed.

## Reproduction Assessment

Yes, at source level: with `mode: "all"`, any non-empty session is treated as sandboxed, the embedded runner resolves `resolveSandboxContext` before the model/tool attempt, and the backend is created during context setup. I did not run a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear pending maintainer/security design: `needed` is a plausible direction, but the best solution must first define which host-side work may proceed during sandbox backend downtime and how sandbox-bound tools fail closed.
