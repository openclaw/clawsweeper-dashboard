## Summary

Keep open: current main still has only session-scoped sandbox modes and still resolves a sandbox backend before the embedded model/tool attempt for sandboxed sessions. The requested on-demand mode remains a plausible reliability improvement, but it is a new security-sensitive config/runtime contract rather than conservative cleanup.

## What This Changes

Add an on-demand `agents.defaults.sandbox.mode: "needed"` so chat-only turns can proceed when Docker is unavailable while sandbox-bound tools fail closed.

## Best Possible Solution

Define an approved on-demand sandbox contract, then update schema, exported types, runtime routing, docs, and regression tests so Docker is required only when sandbox-bound execution begins and those tools still fail closed.

## Reproduction Assessment

Yes, at source level: with `mode: "all"`, current main marks the session sandboxed, resolves `resolveSandboxContext` before the model/tool attempt, and the Docker backend creates/verifies a container. I did not run a live Telegram turn with Docker stopped.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible narrow direction for the outage, but the host-vs-sandbox boundary must be approved before implementation. The maintainable solution should keep sandbox-bound tools fail-closed rather than silently moving them to host execution.
