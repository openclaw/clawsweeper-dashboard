## Summary

Keep open: current main still treats sandbox mode as a session-level contract and only exposes `off`, `non-main`, and `all`; the requested on-demand mode would be a new security-sensitive runtime/config behavior, not a conservative cleanup close.

## What This Changes

The issue requests a new `agents.defaults.sandbox.mode: "needed"` mode so chat-only turns can run when Docker is unavailable while sandbox-bound tool execution still fails closed.

## Best Possible Solution

Define a threat-model-approved on-demand sandbox contract, then update schema, exported types, runtime routing, docs, and regression coverage so Docker is required only when sandbox-bound tool execution begins.

## Reproduction Assessment

Yes, at source level: `mode: "all"` marks every session sandboxed, the embedded runner resolves sandbox context before the model/tool attempt, and the Docker backend requires a container during context creation. I did not run a live Telegram turn with Docker stopped in this read-only review.

## Solution Assessment

Unclear pending maintainer design: `needed` is a plausible direction for the outage, but it changes the sandbox security contract and should be specified before implementation.
