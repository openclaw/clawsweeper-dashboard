## What This Changes

The PR replaces backup archive creation with an explicit `tar.c.asyncNoFile()` to `WriteStream` pipeline, adds progress and OOM messaging, and adds stream-error tests.

## Best Possible Solution

Keep the shipped `tar.c({ file })` archive writer with its current retry and cleanup hardening; handle any future progress UI or exclude-pattern work in narrower active items.

## Reproduction Assessment

No. Source and upstream dependency inspection show the current file-mode tar path already streams to a write stream and propagates stream/pack errors, so the reported buffering/stall premise is not reproduced on current main.

## Solution Assessment

No. The PR's explicit pipe is not the best current solution because pinned tar file mode already provides the same streaming/backpressure contract, and the stale branch adds incorrect OOM guidance plus weaker stream cleanup than `pipeline()` would provide.

Full review comments:

- **[P2] Replace the unavailable exclude-flag hint:** `src/commands/backup.ts:369`
  - body: The new ENOMEM handler tells users to retry with `--exclude-logs` or `--exclude-media`, but `backup create` does not register those flags on current main. In the failure path this is meant to help, the suggested command would be rejected, so the message should only reference supported recovery options or a separately landed exclude feature.
  - confidence: 0.9
- **[P2] Destroy the tar stream on destination failures:** `src/commands/backup.ts:361`
  - body: When `destStream` errors, `fail()` destroys only the destination stream while the tar pack can keep walking and compressing input until it later observes the broken pipe. For the large installations this PR targets, use `pipeline()` or explicitly tear down both streams so failures stop work promptly.
  - confidence: 0.82
