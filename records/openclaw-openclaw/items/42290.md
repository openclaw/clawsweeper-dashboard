## Summary

Close as mostly implemented on main: current main already covers the streaming/error-propagation concern through the pinned tar file-mode implementation plus newer backup retry/temp-cleanup hardening, while this stale branch conflicts with the refactored backup path and has flawed leftover guidance.

## What This Changes

The PR replaces the older backup tar write with an explicit readable-stream-to-write-stream pipeline, progress logging, OOM hinting, and two backup error-path tests.

## Best Possible Solution

Keep the current tar file-mode plus retry/volatile-filter implementation; handle any future progress UI or exclude UX in a fresh branch against `src/infra/backup-create.ts` and the canonical exclude-pattern work rather than merging this stale pre-refactor branch.

## Reproduction Assessment

No live large-install reproduction was run. Source and dependency inspection give high confidence that current main already uses node-tar file-mode streaming with error propagation, and the remaining PR defects are reproducible from the diff and CLI registration source.

## Solution Assessment

No. Replacing node-tar file mode with this manual pipe is not the best fix because the dependency already streams to a write stream, current main adds retry/temp cleanup around it, and this branch would need a non-trivial rebase plus corrected error guidance.

Full review comments:

- **[P2] Use supported OOM recovery guidance:** `src/commands/backup.ts:365`
  - body: `backup create` does not register `--exclude-logs` or `--exclude-media`, and the related flag PR was closed unmerged. In the new ENOMEM path this would tell users to run flags the CLI rejects, so the recovery hint needs to point at supported options or land with the actual flag surface.
  - confidence: 0.91
- **[P2] Tear down the tar stream on destination errors:** `src/commands/backup.ts:355-376`
  - body: The shared `fail` handler only destroys `destStream`. If the destination stream fails first, the tar pack can continue walking and compressing the large source tree until it later sees `EPIPE`, preserving the resource waste this change is meant to avoid; use `pipeline()` or explicitly destroy the tar stream on either side's error.
  - confidence: 0.84
