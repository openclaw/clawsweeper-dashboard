## Summary

Close as implemented: current main already satisfies the central streaming/backpressure concern through the pinned tar file-mode writer plus atomic temp cleanup, while this branch is stale against the moved backup implementation and has defects if revived.

## What This Changes

The PR changes backup archive creation from tar file mode to an explicit tar stream piped to a file stream, adds progress/OOM messaging, and adds stream-error tests.

## Best Possible Solution

Keep the shipped tar file-mode atomic writer for backup creation, and route archive-size or progress UX work to a focused follow-up such as the existing smart-exclude work.

## Reproduction Assessment

No. I did not reproduce a current-main stall; source inspection of current main plus upstream `tar@7.5.13` shows the file-mode call already streams to disk and propagates pack/write errors.

## Solution Assessment

Yes for closing, not for merging this branch. The maintainable path is the current shipped writer; reviving this branch would need relocation to `src/infra/backup-create.ts` and fixes for its incorrect OOM guidance and manual stream cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Replace the unavailable OOM recovery flags:** `src/commands/backup.ts:369-372`
  - body: The new OOM handler tells users to retry with `--exclude-logs` or `--exclude-media`, but current `backup create` does not register either flag, so the failure path would direct users to commands the CLI rejects.
  - confidence: 0.94
- **[P2] Destroy the tar stream when the destination fails:** `src/commands/backup.ts:355-376`
  - body: The manual `fail` path destroys only the destination stream. If the file write stream fails, the tar stream can continue walking and compressing input until it hits another error; `pipeline()` or destroying both streams would stop the source promptly.
  - confidence: 0.86
