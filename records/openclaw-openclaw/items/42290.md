## Summary

Close as mostly implemented on main: current main and shipped releases already cover the central streaming/backpressure concern through node-tar file-mode streaming plus temp cleanup, while this older branch is stale against the refactored backup implementation and still has patch-level defects.

## What This Changes

The PR replaces backup archive creation with `tar.c.asyncNoFile()` piped to `createWriteStream()`, adds progress/OOM handling, and adds two backup command tests.

## Best Possible Solution

Keep the current `src/infra/backup-create.ts` file-mode tar implementation; handle any future progress or exclude UX as a separate patch against the current backup infra, with hints limited to flags that actually exist.

## Reproduction Assessment

No for the original large-installation stall. I did not establish a failing current-main path, and dependency/source inspection shows the current tar file-mode path already streams through a `Pack` piped to a write stream; the PR-specific OOM hint defect is source-reproducible from the diff and CLI registration.

## Solution Assessment

No, this PR is no longer the best way to solve the issue. The central behavior is already covered by the shipped implementation, and any remaining progress or recovery-copy work should target `src/infra/backup-create.ts` without adding unavailable flag guidance.

## Review Findings

Overall confidence: 0.88

Full review comments:

- **[P2] Replace the unavailable exclude-flag recovery hint:** `src/commands/backup.ts:369`
  - body: The OOM path tells users to retry with `--exclude-logs` or `--exclude-media`, but this branch does not register either flag for `backup create`. In the failure path this PR adds, users would be directed to options the CLI rejects, so the recovery guidance needs to reference an existing option or be removed.
  - confidence: 0.95
- **[P2] Destroy the tar source when the destination stream fails:** `src/commands/backup.ts:361`
  - body: The shared failure handler only destroys the destination stream. If the destination errors, for example on disk full, the tar pack can keep walking and compressing the source tree until another error propagates; using `pipeline()` or explicitly destroying the tar stream would stop work immediately on the large backups this change targets.
  - confidence: 0.84
