## Summary

Close as implemented on main: the current backup path already writes through tar's file-mode streaming implementation, awaits completion, publishes atomically, and cleans temporary output on failure. The PR is stale against the current `src/infra/backup-create.ts` layout and its branch adds merge-blocking guidance for backup flags that main does not expose.

## What This Changes

The PR rewrites backup archive creation from tar file output to an explicit tar stream piped to a write stream, adds progress and OOM messaging, and adds tests for tar-stream and OOM failures.

## Best Possible Solution

Keep the shipped tar file-mode plus atomic temporary-publish implementation for the streaming/backpressure concern. Leave broader large-backup UX such as progress, smarter excludes, or configurable patterns to the separate #42273/#44288 track instead of merging this stale rewrite.

## Reproduction Assessment

No high-confidence current-main reproduction of tar file-mode buffering is available. Current source and `tar@7.5.13` dependency source show the file-mode path streams through `Pack` into a write stream, and tests cover tar failure cleanup.

## Solution Assessment

No for the PR diff: merging this branch is not the best fix because main already has the maintainable streaming path and the branch is stale. If future progress or OOM UX is added, it should target `src/infra/backup-create.ts` and reference only CLI flags that actually exist.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reference only backup flags that exist:** `src/commands/backup.ts:369-371`
  - body: The OOM error tells users to retry with `--exclude-logs` or `--exclude-media`, but `backup create` does not register either flag on main. In this failure path the suggested recovery command is rejected by the CLI, so the error message needs to point to existing options or wait for the exclude feature to land.
  - confidence: 0.93
- **[P2] Destroy both streams on write failure:** `src/commands/backup.ts:397`
  - body: `destStream.on("error", fail)` rejects and destroys only the destination stream. If the destination fails, the tar pack can keep reading and compressing a large source tree until it later sees the closed pipe; using `pipeline()` or explicitly destroying `tarStream` would clean up both sides promptly.
  - confidence: 0.82
