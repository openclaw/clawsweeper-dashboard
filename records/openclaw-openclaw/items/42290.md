## Summary

Close as mostly implemented on main: the central streaming/backpressure concern is already covered by the shipped `tar.c({ file })` implementation and upstream tar contract, while the stale branch targets the pre-refactor command file and its small remaining pieces are obsolete or tracked by broader backup-exclude work.

## What This Changes

The PR replaces backup archive creation with an explicit tar stream piped to a write stream and adds tests for stream errors and OOM messaging.

## Best Possible Solution

Keep the shipped file-mode tar implementation and handle broader large-backup exclude/progress UX through https://github.com/openclaw/openclaw/issues/42273 and https://github.com/openclaw/openclaw/pull/44288 rather than refreshing this stale branch.

## Reproduction Assessment

No. I did not reproduce a live 4 GB current-main backup stall; source and upstream dependency inspection show the central premise that file-mode tar buffers instead of streaming does not hold for the pinned tar implementation.

## Solution Assessment

No. Replacing the shipped file-mode tar path with this branch's manual `.pipe()` wiring is not the best current fix because the dependency already streams and propagates errors, and any future change would need to target `src/infra/backup-create.ts`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Stop the tar stream when the destination write fails:** `src/commands/backup.ts:355-376`
  - body: The new failure handler destroys only the destination stream. If `createWriteStream` errors, the tar pack can keep walking and compressing a large install until it later observes a broken pipe; use `pipeline()` or destroy the tar stream on any failure so the expensive archive walk stops immediately.
  - confidence: 0.86
- **[P2] Remove recovery advice for flags that do not exist:** `src/commands/backup.ts:369`
  - body: The ENOMEM wrapper tells users to retry with `--exclude-logs` or `--exclude-media`, but `backup create` does not register either flag on current main. In the failure path this would send users to a command the CLI rejects.
  - confidence: 0.95
