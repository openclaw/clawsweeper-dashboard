## Summary

Close as implemented on main: the current backup path already writes through tar's file-mode streaming implementation, publishes atomically only after completion, and has failure-cleanup coverage. The PR is stale against the current `src/infra/backup-create.ts` layout and also adds an OOM hint for flags that `backup create` does not expose on main.

## What This Changes

The PR rewrites backup archive creation to pipe tar's no-file stream into a write stream, adds progress/OOM handling, and adds backup tests for tar-stream and OOM failures.

## Best Possible Solution

Keep the shipped tar file-mode plus atomic-temp implementation for the streaming/backpressure concern. Treat progress UI, smarter default excludes, and configurable large-backup controls as separate work under #42273/#44288 rather than merging this stale rewrite.

## Reproduction Assessment

No current-main reproduction of tar file-mode buffering is available; upstream `tar@7.5.13` source shows the pinned file-mode path already streams through `Pack` into a `WriteStream`. The current test suite also covers tar failure cleanup without leaving a final archive behind.

## Solution Assessment

Yes. Closing this PR is the maintainable outcome because the requested streaming behavior is already provided by the dependency-backed current implementation, while the branch is stale and contains invalid recovery guidance.

Full review comments:

- **[P2] Reference only backup flags that exist:** `src/commands/backup.ts:369`
  - body: The OOM error tells users to retry with `--exclude-logs` or `--exclude-media`, but `backup create` does not register either flag on main. In this failure path the suggested recovery command is rejected by the CLI, so the error message needs to point to existing options or wait for the exclude feature to land.
  - confidence: 0.93
- **[P2] Stop the tar stream when the write stream fails:** `src/commands/backup.ts:397`
  - body: `destStream.on("error", fail)` rejects and destroys only the destination stream. If the destination fails, the tar pack can keep reading and compressing a large source tree until it later sees the closed pipe; using `pipeline()` or explicitly destroying `tarStream` would clean up both sides promptly.
  - confidence: 0.82
