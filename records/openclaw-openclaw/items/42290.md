## Summary

Close as mostly implemented on main: the central memory/backpressure concern is already covered by the shipped `tar.c({ file })` path, whose upstream implementation pipes a `Pack` to a file write stream, while this older branch is now stale, conflicting, and has only minor or separately tracked remnants.

## What This Changes

The PR replaces backup archive creation with an explicit `tar.c.asyncNoFile()` stream piped to `createWriteStream()`, adds progress/OOM handling, and adds stream-error/OOM tests.

## Best Possible Solution

Keep the shipped `tar.c({ file })` writer with current retry/temp cleanup hardening, and handle remaining large-backup exclude or progress UX through https://github.com/openclaw/openclaw/issues/42273 and https://github.com/openclaw/openclaw/pull/44288 rather than merging this stale explicit-pipe branch.

## Reproduction Assessment

No high-confidence current-main reproduction of the claimed memory-buffering root cause. Source inspection plus upstream `tar` source show the shipped file-mode path streams to a write stream, while the PR-specific review blockers are source-visible from the branch diff.

## Solution Assessment

No. Replacing the shipped file-mode tar writer with this stale explicit pipe is not the best fix because the dependency already streams file-mode output and current main has additional retry, volatile-filter, and temp-manifest hardening that the branch does not carry forward cleanly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Point OOM recovery at supported options:** `src/commands/backup.ts:369`
  - body: The new ENOMEM handler tells users to retry with `--exclude-logs` or `--exclude-media`, but this branch's `backup create` registration exposes only `--output`, `--json`, `--dry-run`, `--verify`, `--only-config`, and `--no-include-workspace`. In the failure path being handled, the suggested command would be rejected.
  - confidence: 0.91
- **[P2] Destroy the tar source when the destination fails:** `src/commands/backup.ts:361`
  - body: The error handler destroys only `destStream`. If the file write stream errors, such as `ENOSPC`, the tar pack can keep walking and compressing the large source tree until a later pipe failure, preserving the expensive work this fix is meant to avoid. Use `pipeline()` or explicitly destroy the tar stream on destination errors.
  - confidence: 0.84
