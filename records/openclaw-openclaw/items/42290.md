## Summary

Close as implemented: current main already writes backup archives through tar's file-mode Pack-to-WriteStream path, with atomic temp publishing and failure cleanup; remaining large-backup exclude/progress UX is tracked separately by #42273/#44288.

## What This Changes

The PR replaces backup archive creation with an explicit tar readable stream piped to a file write stream, adds progress/OOM messaging, and adds backup error-path tests.

## Best Possible Solution

Keep the shipped tar file-mode implementation for the streaming/backpressure concern and handle any remaining large-backup exclude/progress work through #42273/#44288.

## Reproduction Assessment

No. I did not run a multi-GB backup, but source and upstream dependency inspection show current main already uses tar's streaming file-mode path rather than an in-memory archive buffer.

## Solution Assessment

No. The PR's manual pipe is not the best current fix because tar@7.5.13 already implements file-mode streaming with error propagation, and the proposed patch also points OOM recovery at unavailable CLI flags.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Destroy both streams on write failures:** `src/commands/backup.ts:355-376`
  - body: If `destStream` errors, the handler destroys only the destination and leaves `tarStream` producing archive data until it eventually notices the broken pipe. For the large backups this change targets, that can keep walking/compressing a large tree after the write has already failed; use `pipeline()` or explicitly destroy the tar stream on either side's error.
  - confidence: 0.83
- **[P2] Avoid suggesting unavailable backup flags:** `src/commands/backup.ts:365-369`
  - body: The OOM error tells users to retry with `--exclude-logs` or `--exclude-media`, but `backup create` does not currently register either option. In this failure path the suggested recovery command would be rejected by the CLI.
  - confidence: 0.88
