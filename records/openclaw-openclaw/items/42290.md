## Summary

Close as mostly implemented on current main: the central streaming/backpressure concern is already covered by the shipped `tar.c({ file })` implementation and atomic temp cleanup, while this stale branch edits the old pre-refactor file and adds incorrect OOM guidance.

## What This Changes

The PR changes `openclaw backup create` to use an explicit tar stream piped to a write stream, adds progress/OOM handling, and adds tests for stream and OOM errors.

## Best Possible Solution

Keep the shipped tar file-mode plus atomic-temp implementation for the core streaming concern, and handle progress or smarter exclude behavior through the narrower backup UX work such as https://github.com/openclaw/openclaw/pull/44288.

## Reproduction Assessment

No. I did not establish a current-main buffering stall; current source plus upstream `tar@7.5.15` source show file-mode archive creation already streams to a write stream and propagates stream/pack errors.

## Solution Assessment

No. Replacing the shipped file-mode tar call with this stale manual pipe is not the best fix, especially because the branch targets the old file location and introduces invalid recovery guidance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Remove the unavailable OOM recovery flags:** `src/commands/backup.ts:369`
  - body: The new OOM message tells users to retry with `--exclude-logs` or `--exclude-media`, but current `backup create` does not register either flag. A user hitting this path would be sent to options the CLI rejects.
  - confidence: 0.91
- **[P2] Tear down the tar stream when the destination fails:** `src/commands/backup.ts:355-361`
  - body: The failure handler destroys only `destStream`; if the destination errors first, the tar pack can keep walking and compressing the source tree until it later sees a broken pipe. Use `pipeline()` or explicitly destroy both streams so the large-backup error path stops promptly.
  - confidence: 0.82
