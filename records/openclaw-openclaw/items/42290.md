## What This Changes

The PR changes the old backup command implementation to pipe `tar.c.asyncNoFile()` into a write stream with progress/OOM handling and adds tests for tar stream failure and OOM messaging.

## Best Possible Solution

Keep the current tar file-mode writer with retry and atomic cleanup, and handle archive-size, progress, and exclude controls through the active canonical backup work rather than merging this stale branch.

## Reproduction Assessment

Unclear for a current-main failing reproduction: I did not run a live 4GB backup, and current main plus the pinned tar source indicate the central buffering claim is already addressed by a streaming file-mode implementation. The broader large-backup size/exclude symptom remains separately reproducible from source as missing CLI capability, but that is tracked elsewhere.

## Solution Assessment

No for merging this PR as-is: current main already has the maintainable streaming/atomic write path, and this branch conflicts with the refactored backup architecture while adding incorrect recovery guidance. The safer solution is to keep the shipped writer and resolve remaining UX through the canonical backup-exclude/progress items.

Full review comments:

- **[P2] Avoid recommending unavailable backup flags:** `src/commands/backup.ts:369`
  - body: The OOM error tells users to retry with `--exclude-logs` or `--exclude-media`, but this PR does not add those flags and the backup CLI at this head still only registers the existing options. In the failure path this adds, users would be directed to commands that Commander rejects instead of an actionable recovery path.
  - confidence: 0.92
- **[P2] Stop the tar stream when the destination fails:** `src/commands/backup.ts:361`
  - body: When the destination stream errors, `fail` only destroys `destStream`; the tar pack stream can continue walking and compressing files until a later pipe error. For the large-installation case this PR targets, the error path should use `pipeline()` or explicitly destroy the tar stream so write failures stop all work promptly.
  - confidence: 0.86
