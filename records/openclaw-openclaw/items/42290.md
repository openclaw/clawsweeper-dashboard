## What This Changes

The PR replaces backup archive creation with an explicit tar stream piped to a file and adds tests for tar stream failures and OOM messaging.

## Best Possible Solution

Keep the shipped tar file-mode writer and handle any future progress UI or smarter exclude behavior as separate current-code work, with https://github.com/openclaw/openclaw/pull/44288 as the relevant exclude-pattern thread.

## Reproduction Assessment

No. Current source plus the pinned tar dependency source show `tar.c({ file })` already streams through a write stream and propagates errors, so I do not have a high-confidence current-main reproduction of the claimed memory-buffering bug.

## Solution Assessment

No. Replacing the current writer with this stale explicit pipe is not the best fix because it is equivalent for the core streaming concern, edits the old command implementation, and adds incorrect recovery guidance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Remove the unavailable exclude-flag hint:** `src/commands/backup.ts:369`
  - body: The OOM path tells users to retry with `--exclude-logs` or `--exclude-media`, but current `backup create` does not register either flag and the related flag PR is closed unmerged. In this failure path the recovery advice would send users to a command the CLI rejects.
  - confidence: 0.92
- **[P2] Stop the tar stream when the write side fails:** `src/commands/backup.ts:355-361`
  - body: `fail()` destroys only `destStream`; when the destination errors, the tar packer can keep walking and compressing input until it hits a later pipe error. Use `pipeline()` or explicitly destroy the tar stream on destination failure so large backups do not continue doing wasted work after the output is gone.
  - confidence: 0.84
