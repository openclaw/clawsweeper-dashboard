## Summary

Keep this PR open, but the current patch is not merge-ready. The linked memory_search bug is real, yet dropping the model predicate from lexical FTS without changing FTS row cleanup can make search return stale or deleted old-model memory/session rows.

## What This Changes

The PR removes provider-model filtering from memory-core lexical FTS search and adds regression coverage for keyword hits across embedding models.

## Best Possible Solution

Align FTS row lifecycle with model-agnostic keyword search before merge, preferably by purging all model FTS rows for refreshed/pruned paths or by otherwise ensuring all returned FTS hits correspond to live chunks.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create old-model FTS rows, refresh or prune the same path under a different active model, then search a keyword that only exists in the old row. The PR's all-model FTS query can return that row because cleanup still removes only the active model's FTS entries.

## Solution Assessment

No. Removing the model predicate is a plausible fix for cross-model lexical recall, but it is not the best safe fix until FTS cleanup or query joins are updated so stale old-model rows cannot be returned.
