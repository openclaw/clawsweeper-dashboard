## What This Changes

The issue asks OpenClaw to give clearer setup/runtime guidance when an OpenAI-compatible custom provider baseUrl omits the required API path such as `/v1`.

## Best Possible Solution

Keep the issue open and land or rework https://github.com/openclaw/openclaw/pull/84703 so OpenClaw rejects successful non-JSON probes and normalizes runtime HTML responses with a baseUrl path hint without auto-rewriting user baseUrl values.

## Reproduction Assessment

Yes, from source inspection: onboarding builds `/chat/completions` from the raw baseUrl and accepts any HTTP 2xx, while runtime uses the same SDK baseURL behavior and can fall through to the generic incomplete-terminal message. A full live provider repro was not run.

## Solution Assessment

Yes. The maintainable fix is targeted custom-provider probe validation and OpenAI-compatible runtime error normalization, not automatically appending `/v1` and risking valid custom prefixes.
