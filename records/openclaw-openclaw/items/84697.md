## Summary

Keep open. Current main still accepts any successful custom-provider probe without validating API-shaped JSON and can still surface the OpenAI-compatible HTML response as the generic incomplete-terminal failure; the linked fixing PR is open and references this issue with closing syntax, so this issue should close only after that fix merges or is replaced.

## What This Changes

The issue asks OpenClaw to guide custom OpenAI-compatible provider users toward the correct API-path baseUrl and report HTML/non-API responses with a clear baseUrl hint instead of an incomplete-terminal fallback.

## Best Possible Solution

Review and land or replace the linked PR so OpenClaw rejects successful-looking non-JSON provider probes and turns OpenAI-compatible HTML responses into a clear baseUrl/API-path hint without auto-rewriting custom baseUrls.

## Reproduction Assessment

Yes from source inspection: current main builds `/chat/completions` from a baseUrl without `/v1`, accepts any HTTP 2xx probe, and can still fall back to the generic incomplete-terminal message. I did not run live provider traffic.

## Solution Assessment

Yes. The best fix is targeted validation/error normalization in the custom-provider probe and OpenAI-compatible transport path, not a blanket `/v1` rewrite that would break valid custom prefixes.
