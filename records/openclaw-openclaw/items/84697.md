## What This Changes

The issue reports that OpenAI-compatible custom providers whose baseUrl omits an API path such as /v1 can receive HTML from the wrong endpoint and surface a generic incomplete-terminal error.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84703 after acceptable proof, or replace it with the same narrow setup-probe validation and runtime HTML/baseUrl error normalization.

## Reproduction Assessment

Yes from source inspection: current main builds the OpenAI-compatible probe and runtime call from the exact configured baseUrl, accepts any 2xx setup response, and can fall through to the generic incomplete-terminal message. I did not run a live spanagent.xyz repro in this read-only review.

## Solution Assessment

Yes. The best fix is targeted validation/error normalization for non-JSON or HTML OpenAI-compatible responses, not auto-appending /v1 to every custom provider baseUrl because valid providers may use other prefixes.
