## Summary

Keep open. Current main and v2026.5.19 still accept successful-looking custom-provider probes based only on HTTP 2xx, build OpenAI-compatible probe/runtime URLs from the exact configured baseUrl, and can still turn the downstream generic empty-response payload into the reported incomplete-terminal fallback. The related open PR at https://github.com/openclaw/openclaw/pull/69346 is useful context for the empty-stream diagnostic, but it does not close this issue and does not cover the custom-provider verification gap.

## What This Changes

The issue asks OpenClaw to guide or validate OpenAI-compatible custom provider baseUrl values and surface a clear content-type/baseUrl diagnostic when an endpoint returns HTML instead of JSON/SSE.

## Best Possible Solution

Reject successful-looking custom-provider probes that return HTML/non-JSON content, and normalize OpenAI-compatible non-JSON/non-SSE runtime failures into a clear baseUrl/content-type hint without automatically appending /v1 to custom prefixes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: custom-provider verification builds /chat/completions from a baseUrl without /v1 and accepts any 2xx, while the runtime uses the same OpenAI SDK baseURL behavior and can convert the generic empty response into the reported incomplete-terminal fallback. A full live spanagent.xyz repro was not run.

## Solution Assessment

Is this the best way to solve the issue? Yes: the maintainable direction is targeted validation and error normalization in the custom-provider probe/OpenAI-compatible runtime path, not a blanket /v1 rewrite that would break valid custom prefixes. The related empty-stream PR should be considered so the final wording has one source of truth.
