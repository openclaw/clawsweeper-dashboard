## Summary

Keep open: current main still has an opt-in cache-trace path that can append full prompt, system, and message snapshots without passing any file or queue cap to the writer. The exact worker StructuredClone OOM was not reproduced here, but the unbounded diagnostic-output path is source-backed and worth a focused fix.

## What This Changes

The issue reports gateway V8 out-of-memory crashes under prompt-cache trace logging and requests bounded cache-trace output plus safer large-payload handling.

## Best Possible Solution

Add a default bounded cache-trace output path that preserves redaction and digests, updates docs/tests, and leaves broader crash-event or heap-size changes for separate follow-up if still needed.

## Reproduction Assessment

No full high-confidence crash reproduction was run. Source inspection does show a clear current-main path where enabling cacheTrace records repeated full diagnostic snapshots without a cache-trace file or queue cap.

## Solution Assessment

Yes for the central failure mode: the narrowest maintainable fix is to bound cache-trace writes and payload growth in the existing diagnostic path. A gateway-wide Node heap bump or generic crash event should not be the first fix unless a follow-up proves the cap is insufficient.
