## Summary

Keep open. Current main still leaves the reported retry/replay path in place, normal repeated-tool loop detection is opt-in, and two open PRs still reference this issue with closing syntax as implementation candidates.

## What This Changes

The issue requests a runtime fix so failed tool calls do not cause OpenClaw to replay the agent's previous tool call indefinitely, especially for Feishu/Qwen embedded runs.

## Best Possible Solution

Land one canonical agents-runtime fix that disables the inner pi-coding-agent retry path only for embedded OpenClaw runs without persisting user Pi settings, preserves prepared compaction overrides and failed-tool context, and adds focused regression coverage for failed exec followed by a changed or no-op assistant action.

## Reproduction Assessment

Unclear. The issue gives a concrete Feishu/Qwen failed-exec loop pattern and dependency inspection confirms a plausible SDK retry replay path, but current main has no dedicated regression test or live replay proof for that exact sequence.

## Solution Assessment

Unclear. A session-scoped in-memory retry disable appears to be the narrowest maintainable fix candidate, while broader thinking-repetition hints need maintainer judgment because they change replay semantics beyond the reported SDK retry loop.
