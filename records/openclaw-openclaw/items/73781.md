## Summary

Keep open: current main still preserves embedded Pi SDK retry in prepared runtime settings, the SDK retry contract still defaults retry on, and two open PRs already use closing syntax for this issue. The newer generic loop blocker is useful mitigation, but it is opt-in for rolling detection and does not replace the narrow retry-ownership fix.

## What This Changes

The issue reports runtime-level replay of prior tool calls after failures, observed with Feishu/Qwen and later a Telegram recovery-path observation.

## Best Possible Solution

Review and land or consolidate a canonical agents-runtime fix that disables Pi SDK auto-retry for OpenClaw-managed sessions, preserves failed-tool context, and adds focused replay-loop regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main keeps Pi SDK retry enabled, the pinned SDK retry defaults are on, and OpenClaw also owns an outer retry loop. I did not run the full Feishu/Qwen or Telegram live channel reproduction.

## Solution Assessment

Is this the best way to solve the issue? Yes for the narrow bug boundary: retry ownership should stay in OpenClaw by disabling embedded Pi SDK auto-retry, while broader circuit-breaker or self-healing behavior should stay in separate trackers unless maintainers choose to include it.
