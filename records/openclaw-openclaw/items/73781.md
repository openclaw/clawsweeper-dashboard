Keep open: current main still preserves the embedded Pi SDK retry path that can recontinue after retryable failures, and two open PRs already reference this issue as candidate fixes.

## What This Changes

The issue requests a fix for runtime-level replay of previous tool calls after tool failures, reported with Feishu/Qwen and reinforced by a later Telegram replay observation.

## Best Possible Solution

Land or consolidate a canonical agents-runtime fix that disables Pi SDK auto-retry for OpenClaw-managed sessions, preserves failed-tool context, and adds focused replay-loop regression coverage; keep broader circuit-breaker policy under the separate open tracker.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was established here; source inspection gives a plausible current-main path because prepared Pi settings keep SDK retry enabled while the SDK removes the error assistant turn and calls continue().

## Solution Assessment

Is this the best way to solve the issue? Yes for the narrow first fix: retry ownership should stay in OpenClaw by disabling the embedded Pi SDK auto-retry, while broader thinking-repetition or circuit-breaker behavior should remain separate unless maintainers choose to include it.
