## Summary

Keep open. Current main still leaves the embedded Pi retry setting untouched, the generic repeated-call guard is disabled by default, and two open PRs already reference this issue with closing syntax as active implementation candidates.

## What This Changes

The issue reports that after a failed tool call, the runtime can replay the previous tool call instead of letting the model reason and choose a new action.

## Best Possible Solution

Ship one canonical agents-runtime fix that disables the embedded pi-coding-agent auto-retry path in OpenClaw-managed runs, keeps failed tool/error context available to the next model turn, and adds focused regression coverage for failed exec followed by a changed or no-op assistant action. Treat broader thinking-repetition hints as a separate maintainer-reviewed behavior change unless the narrow retry fix proves insufficient.

## Reproduction Assessment

Unclear. The report gives a concrete failed-exec loop pattern and dependency inspection shows a plausible retry replay path, but this review did not run the live Feishu/Qwen route and current main has no regression fixture for that exact sequence.

## Solution Assessment

Unclear. The requested no-replay behavior is correct, and disabling the embedded SDK retry appears to be the narrowest first fix, but the broader thinking-repetition detector proposed by one linked PR needs maintainer judgment before becoming part of the canonical solution.
