## Summary

Keep open. Current main still lacks the reported replay-loop fix: embedded Pi settings do not disable the SDK auto-retry path, normal repeated-call loop detection remains opt-in, and two open PRs still reference this issue with closing syntax as implementation candidates.

## What This Changes

The issue reports an OpenClaw 2026.4.25 regression where failed runtime tool calls can cause the same previous tool call to be replayed repeatedly instead of allowing the assistant to reason and choose a new action.

## Best Possible Solution

Land one canonical agents-runtime fix that disables the inner pi-coding-agent retry path for embedded OpenClaw runs without persisting user settings, preserves failed tool-result/error context for the next model turn, and adds focused regression coverage for failed `exec` followed by a changed or no-op assistant action; treat any broader reasoning-loop hint as a separate maintainer-reviewed decision if needed.

## Reproduction Assessment

Unclear. The issue provides a concrete Qwen/Feishu failed-`exec` loop pattern and dependency inspection confirms a plausible inner auto-retry replay path, but this read-only review did not run the live provider/channel path and current main has no regression test for that exact sequence.

## Solution Assessment

Unclear. The narrow in-memory retry disable proposed by the linked PRs appears to be the smallest maintainable fix boundary, while the broader thinking-repetition detector needs maintainer judgment to avoid changing replay semantics too widely.
