## Summary

Keep open. Current main still does not disable the embedded pi-coding-agent retry path, the default tool-loop detector remains opt-in, and two open PRs use closing syntax for this issue, so closing before one canonical fix lands would be premature.

## What This Changes

The issue reports that after a tool failure the runtime automatically repeats the previous tool call instead of preserving the failure context so the model can choose a new action.

## Best Possible Solution

Review and land one canonical fix that disables pi-coding-agent auto-retry for OpenClaw embedded runs without persisting user settings, keeps OpenClaw’s own retry/failover layer intact, and adds regression coverage for a failed `exec` turn followed by a changed or no-op assistant action; treat broader thinking-repetition hints as a separate maintainer scope decision if needed.

## Reproduction Assessment

Unclear. The issue gives a concrete failed-`exec` replay pattern and the pinned dependency source confirms a plausible retry mechanism, but this review did not run the live Feishu/Qwen path and current main has no focused regression test for that exact sequence.

## Solution Assessment

Unclear. The narrow in-memory SDK retry disable appears to be the smallest maintainable fix, while the broader thinking-repetition guard proposed by one open PR needs maintainer review to avoid changing normal replay semantics too widely.
