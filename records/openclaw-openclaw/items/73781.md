## Summary

Keep open: current main still leaves embedded Pi SDK retry enabled in prepared runtime settings, the pinned SDK retry path can re-continue after retryable failures, and two open PRs already reference this issue as candidate fixes.

## What This Changes

The issue reports a regression where failed tool calls are replayed by the runtime instead of letting the agent reason and choose a new action.

## Best Possible Solution

Review and land or consolidate a canonical agents-runtime fix that disables Pi SDK auto-retry for OpenClaw-managed sessions, preserves failed-tool context, and adds focused replay-loop regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main reproduction was established, but source inspection gives a plausible path: prepared Pi settings keep SDK retry enabled while the SDK removes the failed assistant turn and calls continue().

## Solution Assessment

Is this the best way to solve the issue? Yes for the narrow bug boundary: retry ownership should stay in OpenClaw by disabling embedded Pi SDK auto-retry, while broader circuit-breaker behavior should remain separate unless maintainers choose to include it.
