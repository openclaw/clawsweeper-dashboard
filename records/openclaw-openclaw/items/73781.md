## Summary

Keep open. Current main still preserves the embedded pi-coding-agent retry behavior that matches the reported replay path, the default OpenClaw loop detector is not a default mitigation, and two open PRs already reference this issue with closing syntax as active implementation candidates.

## What This Changes

The issue reports a runtime-level replay loop where failed or empty tool calls are automatically repeated after upgrading to OpenClaw 2026.4.25, especially through Feishu with Qwen-compatible routing.

## Best Possible Solution

Ship one canonical agents-runtime fix that disables the embedded pi-coding-agent auto-retry path for OpenClaw-managed runs, keeps failed tool/error context available to the next model turn, and adds focused regression coverage for failed exec followed by a changed or no-op assistant action. Treat broader thinking-repetition intervention as a separate maintainer-reviewed behavior change unless the narrow retry fix proves insufficient.

## Reproduction Assessment

Unclear. The report gives a concrete repeated failed-exec pattern and dependency inspection shows a plausible replay path, but this review did not run the live Feishu/Qwen route and current main has no regression fixture for that exact sequence.

## Solution Assessment

Unclear. Disabling embedded SDK auto-retry appears to be the narrowest first fix for the identified replay path, but the broader thinking-repetition detector proposed by one linked PR should be reviewed as a separate product/runtime behavior decision.
