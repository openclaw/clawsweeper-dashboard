## Summary

Keep open. Current main and the latest shipped release still preserve the embedded Pi retry path implicated by the report, generic tool-loop detection remains opt-in, and open closing PRs already track candidate fixes, so this issue should stay open until a canonical fix lands.

## What This Changes

The issue reports a regression where OpenClaw runtime/session handling replays a prior failed or stale tool call instead of letting the agent continue with fresh reasoning.

## Best Possible Solution

A canonical agents-runtime fix should disable nested Pi SDK retry for OpenClaw-managed sessions, preserve failed-tool context for the next model turn, and add focused regression coverage for failed-tool replay while leaving broader transcript replay work under #69208.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. Source inspection on current main shows the retry/repeat path remains plausible, but this read-only review did not run the live Feishu/Qwen or Telegram restart route and found no current regression fixture for the exact replay loop.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Disabling Pi SDK auto-retry is the narrowest source-backed fix direction, while the broader thinking-repetition hint in #74503 may be useful but changes runtime steering beyond the reported failed-tool replay root cause.
