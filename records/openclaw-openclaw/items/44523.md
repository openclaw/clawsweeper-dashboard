## Summary

Keep this PR open: the reported stale-rollover bug is still source-reproducible on current main, but the branch is no longer the right fix because it copies only raw model/provider fields and bypasses the current source-aware model/auth reset policy.

## What This Changes

The PR moves model/provider override preservation outside the explicit reset guard and adds a fake-timer daily freshness rollover regression test.

## Best Possible Solution

Rebase the branch and route stale rollover through `resolveResetPreservedSelection` or a shared equivalent, with coverage for user, legacy-user, auth-profile, and auto-sourced fallback cases.

## Reproduction Assessment

Yes. By source inspection, a daily-stale entry reaches the new-session branch with `resetTriggered=false`, and `baseEntry` is undefined, so current main does not carry forward the persisted selection fields.

## Solution Assessment

No. The PR targets the correct stale-rollover branch, but copying only raw model/provider values is not the best solution now that reset selection is source-aware and includes auth-profile cleanup.
