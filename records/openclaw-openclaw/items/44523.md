## Summary

Keep this PR open. The reported daily rollover bug still exists on current main, but the branch is stale against the newer source-aware model selection policy; preserving raw model/provider fields unconditionally would carry auto-sourced fallback pins across an implicit reset instead of preserving only user or legacy selections.

## What This Changes

The PR moves `modelOverride` and `providerOverride` preservation outside the explicit reset guard in `initSessionState` and adds a daily freshness rollover regression test.

## Best Possible Solution

Preserve model/provider selection during implicit stale rollover by reusing the existing source-aware selection policy: keep user-driven and legacy model selections, set the preserved source consistently, and continue clearing auto-sourced fallback pins. Add focused coverage for user-sourced, legacy source-less, and auto-sourced daily rollover cases.

## Reproduction Assessment

Yes. A focused `initSessionState` fixture with system time after the 4 AM boundary and a stored session from before the boundary reproduces the path: `isNewSession=true`, `resetTriggered=false`, and current main drops the old model/provider override because preservation is gated behind explicit reset.

## Solution Assessment

No. The PR solves the original older-code bug but is no longer the best current-main fix because it preserves every stored override without consulting `modelOverrideSource`; the safer fix is to apply `resolveResetPreservedSelection` or equivalent policy in the stale rollover path.
