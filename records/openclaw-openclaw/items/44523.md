## Summary

Keep this PR open: the stale daily rollover bug is still present on current main, but the branch copies raw model/provider fields and bypasses the newer source-aware reset-selection policy, so it can preserve auto-sourced fallback state incorrectly. It also still lacks after-fix real behavior proof from a running stale-session rollover.

## What This Changes

The PR moves model/provider override preservation out of the explicit reset guard and adds a fake-timer test for stale daily freshness rollover.

## Best Possible Solution

Preserve stale-rollover user and legacy selections through resolveResetPreservedSelection or an equivalent shared helper, clear auto-sourced model/auth state, and cover user, legacy, and auto-sourced stale rollovers.

## Reproduction Assessment

Yes, by source inspection. A stale stored session with resetTriggered=false enters the new-session branch, has no baseEntry fallback for model/provider/source fields, and current main only runs the preserved-selection helper inside the explicit reset guard.

## Solution Assessment

No. The branch targets the right rollover branch, but the maintainable fix should reuse the shared source-aware preservation policy instead of copying raw model/provider fields.
