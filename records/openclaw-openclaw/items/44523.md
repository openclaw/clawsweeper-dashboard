## Summary

Keep open: current main still drops session model selections on stale daily rollover, but this PR should not merge as-is because it bypasses the source-aware reset-selection policy and lacks after-fix real behavior proof.

## What This Changes

The PR copies `modelOverride` and `providerOverride` from any previous session entry during new-session creation and adds a fake-timer test for stale daily rollover.

## Best Possible Solution

Preserve stale-rollover user and legacy model/auth selections through `resolveResetPreservedSelection` or an equivalent shared policy, clear auto fallback state, and cover user, legacy, and auto-sourced stale rollovers.

## Reproduction Assessment

Yes, by source inspection. A stale stored session with `resetTriggered=false` enters the new-session branch, has no `baseEntry` fallback, and current main only preserves model/provider/source fields for explicit resets.

## Solution Assessment

No. The PR targets the right rollover branch, but the maintainable fix should reuse the shared source-aware preservation policy instead of copying raw model/provider fields.
