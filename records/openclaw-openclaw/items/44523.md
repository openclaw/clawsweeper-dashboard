## Summary

Keep open: the reported daily-rollover bug is still source-reproducible on current main, but this PR needs changes because its raw override copy bypasses the current source-aware user/legacy/auto selection policy.

## What This Changes

The PR moves model/provider override copying out of the explicit reset guard in `initSessionState` and adds a daily freshness rollover regression test.

## Best Possible Solution

Preserve user-driven and legacy source-less model/provider selections during implicit daily/idle rollover through the same source-aware policy used by resets, clear auto-sourced fallback pins, and cover user, legacy, and auto stale-rollover cases.

## Reproduction Assessment

Yes, by source inspection. A stale entry after the 4 AM boundary reaches `isNewSession=true` and `resetTriggered=false`, skips `resolveResetPreservedSelection`, and has no `baseEntry` fallback to restore the previous model/provider selection.

## Solution Assessment

No. The requested direction is right, but the PR should reuse `resolveResetPreservedSelection` or equivalent source-aware logic instead of copying raw override fields.
