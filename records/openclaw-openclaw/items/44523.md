The PR preserves model/provider override fields during stale daily freshness rollover and adds fake-timer coverage for that scenario.

## Best Possible Solution

Preserve stale-rollover user and legacy selections through resolveResetPreservedSelection or an equivalent shared helper, clear auto-sourced model/auth state, and cover user, legacy, and auto-sourced stale rollovers.

Yes, by source inspection. A stale stored session with resetTriggered=false enters the new-session branch, skips resolveResetPreservedSelection, and has no baseEntry fallback for model/provider/source fields.

## Solution Assessment

No. The PR targets the right rollover branch, but the maintainable fix should reuse the shared source-aware preservation policy instead of copying raw model/provider fields.
