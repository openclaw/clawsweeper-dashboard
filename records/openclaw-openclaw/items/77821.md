## Summary

Keep open. Current main still has the restricted memory-flush write guard and generic tool-error warning plumbing, but the memory-flush path runs as `silentExpected`, drops non-voice final payloads during silent turns, and does not forward the resulting tool-error payload from the maintenance caller to a visible chat/TUI error.

## What This Changes

The issue reports that memory-flush write-tool rejections can leave an embedded agent run silent instead of surfacing a visible error.

## Best Possible Solution

Preserve the append-only memory safety rail, but make memory-flush restricted write failures end the maintenance turn promptly and emit a visible diagnostic through the parent reply/TUI path instead of being discarded by silent-turn filtering.

## Reproduction Assessment

No high-confidence live reproduction was established because I did not run the reported macOS/Bedrock/WebChat setup. Source inspection does trace the restricted write through OpenClaw’s error result, `lastToolError`, warning payload, and silent-turn filtering path, so the issue is source-reproducible on current main.

## Solution Assessment

No, current main’s generic tool-error warning path is not enough for this issue because the memory-flush caller and `silentExpected` reply filtering can still suppress the visible error. The narrow fix should handle memory-flush tool rejection/liveness without weakening the write whitelist.
