## Summary

Keep open. Current main still lacks the requested Gemini web_search provider-config API-key and baseUrl fallback, so the PR is not obsolete, but the PR head still has a blocking credential-precedence defect on the auto-detection/runtime-secret path and overlaps the open baseUrl work in #61972.

## What This Changes

The PR changes the Google plugin so Gemini web_search can reuse `models.providers.google.apiKey` and `models.providers.google.baseUrl`, adds `webSearch.baseUrl` manifest/docs metadata, and adds Gemini web-search regression tests.

## Best Possible Solution

Land a rebased Google plugin change that supports Gemini web_search provider API-key and baseUrl fallback while keeping the same credential order in direct execution, auto-detection, and runtime secret resolution: dedicated web-search config first, then `GEMINI_API_KEY`, then `models.providers.google.*`. Reconcile the baseUrl portion with #61972 so one implementation owns the manifest, docs, metadata, cache key, and tests.

## Reproduction Assessment

Yes. A high-confidence static repro exists: on PR head, configure both `GEMINI_API_KEY` and `models.providers.google.apiKey` with no plugin-scoped key, and the configured credential lookup returns the provider key while the runtime resolver treats configured values as higher priority than env.

## Solution Assessment

No. The requested direction is right, but returning the lower-priority provider key through `getConfiguredCredentialValue` is not the best fix unless the runtime can preserve ordered credential sources through auto-detection and SecretRef resolution.

## Review Findings
