## Summary

Keep open: current main still lacks repository-visible merge-queue or synthetic-merge validation for normal main updates, readable main rulesets do not expose required-check enforcement, and no first-bad red-baseline attribution path is present. The generated-protocol drift example is covered by current CI and the latest release, but the central protection and bypass-policy request remains maintainer-owned work.

## What This Changes

The issue asks OpenClaw to prevent red-baseline CI commits on main and add explicit attribution when main CI first goes red.

## Best Possible Solution

Maintainers should choose the main protection, required-check, merge-queue, and emergency-bypass policy first, then add the minimal source-owned attribution workflow if GitHub settings alone do not cover first-bad reporting.

## Reproduction Assessment

Yes. Source and GitHub API evidence reproduce the process gap: current CI lacks merge_group validation, readable main rulesets do not show required-check or merge-queue enforcement, and no first-bad attribution path was found.

## Solution Assessment

No, this is not best solved as an autonomous source-only patch. The narrowest maintainable path is a maintainer policy decision for GitHub rulesets and bypasses, followed by source automation only for attribution or check coverage that remains.
