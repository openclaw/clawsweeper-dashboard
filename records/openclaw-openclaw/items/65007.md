## Summary

Keep open. Current main still matches the reported Google Chat failures: add-on lifecycle payloads are not converted, wildcard routed spaces can retain sender allowlist blocking, and reply delivery can pass a Google Chat message resource where the send API expects a thread resource. The related PR #65058 remains open and only addresses the payload parser subset.

## What This Changes

The issue asks to make Google Chat spaces accept valid add-on message and space lifecycle payloads, honor wildcard group routing without a separate sender allowlist, and avoid invalid thread ids on outbound replies.

## Best Possible Solution

Keep this issue open until the bundled Google Chat plugin has a focused fix covering all three surfaces: add-on lifecycle payload normalization, sender-scoped policy alignment for wildcard routed spaces without groupAllowFrom, and Google Chat-specific reply thread normalization that never sends a message resource as body.thread.name.

## Reproduction Assessment

Yes. A source-level reproduction is clear: an add-on addedToSpacePayload or removedFromSpacePayload without type/eventType reaches the invalid-payload branch, wildcard groups with no sender allowlist keep allowlist sender policy, and generic reply delivery can provide MessageSidFull as replyToId while Google Chat send expects spaces/.../threads/... .

## Solution Assessment

Yes. The narrow maintainable fix belongs in extensions/googlechat, using the existing shared group-access helper and plugin-local parsing/thread validation rather than adding a new core seam.
