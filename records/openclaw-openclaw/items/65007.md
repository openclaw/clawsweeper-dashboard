## Label Justifications

- `P1`: The issue describes a broken Google Chat space workflow where valid events can be rejected, allowed space messages can be blocked, and replies can fail at send time.
- `impact:message-loss`: The reported failures drop inbound space events, block senders before mention gating, or prevent outbound replies from being delivered.

## Summary

Keep open. Current main still has source-level evidence for the add-on payload parser gap, the documented wildcard group pattern can still be blocked by empty sender allowlists, and reply delivery still forwards arbitrary reply ids as Google Chat thread resource names; related open PRs cover only parser or thread subsets and have not landed a full fix.

## What This Changes

Fix Google Chat spaces by accepting valid Workspace Add-on payload variants, honoring wildcard mention-gated group routing, and normalizing reply thread ids before sending.

## Best Possible Solution

Consolidate the active Google Chat PRs and land one plugin-local fix with regression tests for the full Add-on payload union, wildcard routed-space sender policy, and valid thread-resource fallback for replies.

## Reproduction Assessment

Yes, source inspection gives high-confidence reproduction paths: send a lifecycle-only Workspace Add-on payload, configure wildcard Google Chat groups under allowlist policy without `groupAllowFrom`, or let a message-shaped reply id reach `thread.name`. I did not execute a live Google Chat repro or local tests in this read-only review.

## Solution Assessment

Yes, the maintainable fix belongs inside the Google Chat plugin and existing shared reply/access seams; the current open PRs are useful but partial, so maintainers should consolidate rather than close this issue early.
