## Summary

Keep open. Current main still matches the reported Google Chat space failures: add-on lifecycle payloads are not normalized, wildcard routed spaces without sender allowlists can still retain sender allowlist blocking, and reply delivery can pass a message resource as the Google Chat thread name. The related PR #65058 is still only a partial parser fix, so the issue remains actionable.

## What This Changes

Fix Google Chat spaces so add-on lifecycle payloads are accepted, wildcard routed spaces do not require a separate sender allowlist, and replies use a valid thread resource instead of a message id.

## Best Possible Solution

Keep this issue open until the bundled Google Chat plugin has a focused fix covering all three reported surfaces with regression tests: add-on lifecycle payload normalization, sender-scoped policy alignment for wildcard routed spaces without groupAllowFrom, and Google Chat-specific thread-resource validation/fallback for outbound replies.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: send an add-on lifecycle payload without top-level type/eventType, configure `groups: { "*": { requireMention: true } }` with no sender allowlist under allowlist group policy, or produce a reply payload whose `replyToId` is `spaces/.../messages/...`; current main still follows the failing branches.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `extensions/googlechat`, reusing the existing shared group-access helper and adding plugin-local parsing/thread validation instead of changing core contracts; #65058 is useful but incomplete because it only covers the parser subset.
