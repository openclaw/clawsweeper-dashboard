## Summary

Keep open. The PR is active and addresses a real approval UX gap, but current head still has a blocking simple-mode approval safety defect where later high-risk actions can be hidden after the raw command is removed.

## What This Changes

The PR adds configurable plain-language plugin approval rendering across shared approval rendering, forwarding, Telegram, Matrix, config schema/help, docs, tests, and generated baselines, with some incidental adjacent churn.

## Best Possible Solution

Keep the default-preserving, config-gated renderer design, but make simple mode always surface high-risk or preview-worthy actions before hiding raw technical fields.

## Reproduction Assessment

Yes. Source inspection shows a command like `pwd && ls && ps && cat notes.txt && rm -rf /tmp/x` reaches the first-four-action truncation while the delete summary does not force a command preview.

## Solution Assessment

No, not as-is. The shared config-gated renderer is the right direction, but truncation must prioritize high-risk or preview-worthy actions, or force a redacted command preview whenever truncation hides them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Surface hidden high-risk approval steps before truncating:** `src/infra/plugin-approvals.ts:277-281`
  - body: `simple` mode hides raw command details, but this loop renders only the first four action summaries and collapses the rest. A command like `pwd && ls && ps && cat notes.txt && rm -rf /tmp/x` can show `Risk: High` while replacing the delete target with `1 more background step(s)`, and the `rm` summary does not force a command preview. Prioritize high-risk/preview actions or force a preview whenever truncation would hide them.
