## Summary

Keep open: the PR is a useful opt-in feature, but current head still has a concrete approval-safety gap before merge. The simple renderer hides raw command text, yet curl `--url-query` operands can add token-like query data without being classified as credential-bearing or forcing a redacted preview.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` setting for plain-language plugin approval prompts, wires it through Telegram/Matrix/fallback forwarding, and updates docs, config schema, generated baselines, and renderer tests.

## Best Possible Solution

Keep the opt-in plain-language approval feature, but make the command summarizer fail closed for curl query-building operands before simple mode suppresses raw technical details.

## Reproduction Assessment

Yes. Source inspection plus the curl manual gives a high-confidence path: `curl --url-query access_token=s3cr3t https://example.test` is a documented curl form, but PR head only classifies direct URL arguments for credential-bearing queries.

## Solution Assessment

No. The feature direction is maintainable, but the implementation should classify curl `--url-query` operands and file-backed query data as high-risk credential/query sources before hiding the raw command.
