## Summary

Keep open: the feature direction is reasonable and the branch has substantial proof, but the current diff still has a concrete approval-safety bug where `simple` mode can hide sensitive input-redirection reads.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` setting for plain-language plugin approval prompts, wires it through shared approval rendering and Telegram/Matrix/fallback delivery, and updates config, docs, tests, generated baselines, and a Telegram type dependency.

## Best Possible Solution

Keep the opt-in plain-language approval feature, but parse shell input redirection for read/search commands and fail closed or show a redacted preview when the input source is sensitive or ambiguous.

## Reproduction Assessment

Yes for the review finding from source inspection: `cat < .env` reaches the read-command branch with no positional target, while only output redirection is checked before low-risk read rendering. I did not execute the PR branch in this read-only review.

## Solution Assessment

No. The feature direction is maintainable, but the current implementation should account for input-redirection read targets before hiding raw technical command text in `simple` mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Surface input-redirection reads before hiding raw commands:** `src/infra/plugin-approvals.ts:858-864`
  - body: `simple` mode can hide the raw shell command, but the read/search summaries only look at positional args and output redirection. A command like `cat < .env` or `grep TOKEN < .env` has no positional target, so it can render as a low-risk generic read with no preview while printing a secret file. Please treat input redirection as a read target and fail closed or show a preview for sensitive or ambiguous sources.
  - confidence: 0.87
