## Summary

Keep open: the feature is not already on current main and the latest head is clean with green CI, but the new approval summarizer still has a concrete safety bug in its command-wrapper parsing.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` setting for plain-language plugin approval prompts and wires it through config, SDK renderers, Telegram, Matrix, docs, baselines, and tests.

## Best Possible Solution

Land the default-preserving plain-language approval feature after the env wrapper parser handles value-taking options or fails closed for unwrapped commands, with focused renderer regression coverage.

## Reproduction Assessment

Yes for the review blocker by source inspection: `env -u FOO rm -rf /tmp/x` leaves `FOO` as the parsed command because the wrapper loop strips `-u` but not its argument. `env --help` confirms that several env options take required arguments.

## Solution Assessment

No as proposed. The opt-in config and renderer direction is maintainable, but the command parser must handle value-taking `env` flags or conservatively fail closed before simple mode hides the raw technical block.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Parse env option arguments before summarizing wrapped commands:** `src/infra/plugin-approvals.ts:1073-1075`
  - body: The new env-wrapper stripper removes every dash-prefixed `env` option but does not consume option arguments. For `env -u FOO rm -rf /tmp/x`, parsing stops at `FOO`, so simple mode summarizes an unknown medium-risk command instead of the high-risk delete that will actually run. Please parse value-taking env flags such as `-u`, `-C`, and `-S`, or fail closed when the wrapper cannot be unwrapped.
  - confidence: 0.88
