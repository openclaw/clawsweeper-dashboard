## Summary

Keep open: the PR is useful and default-preserving, with real proof supplied, but it is not merge-ready because `simple` mode still hides raw command details for concrete command forms that can leak secrets or execute hidden code.

## What This Changes

The PR adds a configurable `approvals.plugin.language` mode for plain-language plugin approval prompts across shared rendering, forwarding, config schema/help, and Telegram/Matrix fallback payloads.

## Best Possible Solution

Keep the default `original` behavior, and merge the plain-language modes only after command summaries fail closed or show a redacted preview for shell variable output and container/code-execution wrappers, with focused regression coverage.

## Reproduction Assessment

Yes for the review findings: source inspection shows `echo`/`printf` variable arguments and Docker run-style commands flow to low/medium summaries while `simple` mode suppresses the raw technical block. I did not run the PR tests because this review is read-only.

## Solution Assessment

No, not yet. The default-preserving config shape is maintainable, but the simple renderer needs conservative fail-closed behavior for hidden shell variables and container execution before this is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Fail closed on formatter shell variable expansions:** `src/infra/plugin-approvals.ts:898-899`
  - body: With `approvals.plugin.language: "simple"`, commands such as `echo $OPENAI_API_KEY` or `printf '%s\n' "$TOKEN"` reach the formatter branch, get `Risk: Low`, and do not force a command preview while the raw command is hidden. Treat variable expansion in formatter arguments as sensitive/preview-required so chat approvals cannot mask secret-printing commands.
  - confidence: 0.9
- **[P2] Show the Docker command before hiding raw details:** `src/infra/plugin-approvals.ts:1719-1724`
  - body: The generic Docker fallback returns only a medium-risk `change Docker/container state` summary without a preview. A `docker run`/`exec` command can run arbitrary image code, mount the workspace, or read files like `.env`, so simple mode should fail closed or include a redacted command preview for execution/build/unknown Docker subcommands.
  - confidence: 0.86
