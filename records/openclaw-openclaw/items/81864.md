## Summary

This PR should stay open because the opt-in plain-language approval renderer is useful and proof is supplied, but the current patch still has a concrete approval-safety bug in the new git command summarizer.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` setting, plain-language plugin approval rendering, channel fallback wiring for Telegram/Matrix, config/docs/schema updates, and focused approval renderer coverage.

## Best Possible Solution

Land the opt-in shared renderer after the git alias/config parsing case fails closed with a command preview and a focused regression test, while keeping `original` as the default.

## Reproduction Assessment

Yes for the review finding by source inspection and Git proof: `git -c alias.pwn='!printf alias-ran' pwn` executes a shell alias, while the PR parser consumes `-c` as a normal global value option and can summarize the alias invocation as a generic medium-risk git command.

## Solution Assessment

No for the current patch as-is. The shared opt-in renderer is a reasonable boundary, but git config values that can define shell aliases need conservative handling before simple mode hides the raw technical command.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fail closed on git -c shell aliases:** `src/infra/plugin-approvals.ts:1485-1492`
  - body: `git -c alias.pwn='!rm -rf /tmp/x' pwn` is a valid way to define and run a shell-backed git alias for this invocation. The new parser consumes `-c` as a plain global value option, then treats `pwn` as an unknown git subcommand and emits a generic medium-risk source-control summary without a command preview, so simple mode can hide the destructive shell command. Detect `alias.*=!...` config values, or fail closed for alias config before classifying the subcommand, and add a regression.
  - confidence: 0.9
