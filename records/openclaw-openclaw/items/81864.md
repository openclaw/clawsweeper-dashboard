## Summary

Keep open: the feature is useful and current main does not have a plain-language plugin approval renderer, but the latest PR head still has two source-visible simple-mode safety gaps where execution-changing shell context can disappear before raw details are hidden.

## What This Changes

The PR adds `approvals.plugin.language` and a simple/simple-technical renderer for plugin approval prompts, wires it through fallback/Telegram/Matrix delivery, updates config/docs/generated baselines/tests, and includes oc-path formatting plus a `@grammyjs/types` lockfile change.

## Best Possible Solution

Keep the config-gated plain-language renderer with the raw format as the default, but make env assignment prefixes and leading input redirections visible or high-risk before `simple` mode hides technical details, with focused regression tests.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head shows env assignments and leading stdin redirections are stripped before simple-mode text can hide the raw command. I did not execute tests because this was a read-only review.

## Solution Assessment

No, not as-is. The UX direction is maintainable and config-gated, but these shell forms need conservative handling before the plain-language mode can safely omit raw technical details.

- **[P1] Fail closed on env assignment prefixes:** `src/infra/plugin-approvals.ts:1796-1798`
  - body: `resolveCommandWords()` still drops leading `KEY=value` words before classification, and `simple` mode returns before appending raw technical fields. For example, `< .env cat` can reduce to a low-risk `cat` summary with no `.env` target or preview, so carry input targets into read summaries or fail closed.
  - confidence: 0.88
