## Summary

Keep open: the plain-language approval direction is useful, but the current PR head still has a blocking approval-safety issue where env-prefixed execution can be summarized as a benign known command while simple mode hides the raw command text. The PR also needs a branch refresh and current-head Telegram-visible proof before merge.

## What This Changes

The branch adds an `approvals.plugin.language` config and plain-language plugin approval renderer wired through SDK, forwarding, Telegram, and Matrix, with docs/tests/generated baselines plus unrelated oc-path formatting and a `@grammyjs/types` dev-dependency downgrade.

## Best Possible Solution

Keep the feature direction, but make env-prefixed execution fail closed or show a redacted preview before suppressing raw text, refresh the branch, and add current-head Telegram-visible proof.

## Reproduction Assessment

Yes for the blocking review finding. Source inspection at PR head shows leading `KEY=value` words are stripped before known-command classification, while simple mode can return without technical fields or a command preview.

## Solution Assessment

No. The user-facing approval language is plausible, but the current implementation is not conservative enough for env-prefixed execution and should not hide raw command text until that path fails closed or previews safely.
