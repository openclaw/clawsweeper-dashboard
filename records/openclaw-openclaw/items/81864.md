## Summary

Keep open. The PR addresses a real current-main UX gap and has sufficient real-behavior proof, but PR head still has two source-reproducible approval-safety defects where `simple` mode can suppress raw technical details while hiding important command context.

## What This Changes

The PR adds config-gated `original`/`simple`/`simple-technical` plugin approval rendering, wires it through shared, Telegram, Matrix, and forwarding paths, updates config docs/schema/baselines, and adds command-summary tests.

## Best Possible Solution

Keep the config-gated plain-language approval mode, but make `simple` output preserve or preview every hidden medium/high-risk command detail before raw technical fields are suppressed.

## Reproduction Assessment

Yes. Source inspection of PR head shows `pwd && ls && ps && cat notes.txt && rm -rf /tmp/x` can hide the delete target after the four-action cap, and `bash -c 'rm -rf "$1"' sh /tmp/x` drops `/tmp/x` while `simple` mode suppresses raw technical details.

## Solution Assessment

No, not as-is. The shared config-gated renderer is the right direction, but `simple` mode needs to prioritize or preview hidden risky summaries and fail closed when shell-wrapper positional arguments affect the command body.
