## Summary

Keep open. The latest head fixes the simple-mode decision mismatch, but simple-technical still prints the raw default decision list in its technical reply line when the channel view narrows actions, so it can advertise `allow-always` even though the interactive payload only permits `allow-once|deny`.

## What This Changes

Adds opt-in `approvals.plugin.language` modes for plugin approval prompts, wiring them through shared renderers, Telegram/Matrix approval delivery, config schema/help/docs/tests/baselines, plus a Telegram type dependency adjustment.

## Best Possible Solution

Keep the opt-in plain-language approval feature, but make every manual reply line, including simple-technical raw details, render from the same effective allowed-decision list used by buttons/reactions.

## Reproduction Assessment

Yes. From source, render a simple-technical Matrix plugin approval whose view actions are `allow-once` and `deny`; the raw technical `Reply with` line still falls back to `allow-once|allow-always|deny` because it reads only `request.request`.

## Solution Assessment

No. The overall opt-in design is plausible, but the current implementation needs to thread effective allowed decisions through the simple-technical technical reply line before merge.
