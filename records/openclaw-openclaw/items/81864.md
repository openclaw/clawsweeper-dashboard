## Summary

Keep this PR open: the opt-in plain-language approval direction is useful and the default remains unchanged, but the new simple renderer still has a safety blocker where destructive Git commands with global options can be summarized as generic medium-risk Git activity while the raw command is hidden.

## What This Changes

This PR adds an opt-in `approvals.plugin.language` setting, plain-language plugin approval rendering, Telegram/Matrix/fallback forwarding support, config/docs/schema updates, and focused renderer tests.

## Best Possible Solution

Land the opt-in approval language after the command summarizer consumes command-specific global options or fails closed with a redacted preview before simple mode suppresses technical details.

## Reproduction Assessment

Yes for the PR finding: source inspection of PR head shows `git -C repo reset --hard` is parsed with `repo` as the subcommand, while Git's own help confirms `-C <path>` is a global option before the command. This is a PR-head safety regression, not a current-main bug path.

## Solution Assessment

No, not yet. The opt-in renderer and unchanged default are a maintainable direction, but simple mode should not hide raw command text until Git/global-option parsing is command-aware or conservatively high-risk with a preview.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse Git global options before classifying commands:** `src/infra/plugin-approvals.ts:1138`
  - body: The new simple renderer classifies Git commands with `args.find((arg) => !arg.startsWith("-"))`. For `git -C repo reset --hard`, Git consumes `repo` as the value for the global `-C` option, so the real `reset` subcommand is missed and simple mode renders a generic medium-risk Git action while hiding the raw destructive command. Consume Git global option values before subcommand detection, or fail closed with a preview for ambiguous forms.
  - confidence: 0.9
