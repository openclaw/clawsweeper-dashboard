## Summary

Keep this PR open: the opt-in plain-language approval path is useful and has real behavior proof, but the current head still has a credential leak in the new simple-mode command preview redaction.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` config with simple/simple-technical plugin approval rendering across shared forwarding, Telegram/Matrix delivery, docs/schema, and renderer tests.

## Best Possible Solution

Keep the opt-in plain-language approval mode, but land it only after full-header credential redaction, a multi-cookie regression, and green merge checks.

## Reproduction Assessment

Yes for the PR finding: applying the PR-head redaction pattern to a multi-cookie `Cookie:` header leaves `auth=topsecret` in the preview. This is a PR-head defect, not a current-main bug path.

## Solution Assessment

No, not yet. The opt-in renderer direction is maintainable, but simple mode is not safe while credential-bearing cookie headers can leak through the preview.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Redact the full Cookie header value:** `src/infra/plugin-approvals.ts:383`
  - body: The new preview redactor only replaces the first non-whitespace token after `Cookie:`. In simple mode, `curl -H "Cookie: session=s3cr3t; auth=topsecret" https://example.test` still previews `auth=topsecret`, so a credential-bearing approval can leak cookie data into chat. Redact the whole header value, including semicolon-separated cookies, and add a regression for that form.
  - confidence: 0.93
