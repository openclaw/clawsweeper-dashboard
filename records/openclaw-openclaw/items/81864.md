## Summary

Keep this PR open: the plain-language approval direction is useful, but the current patch still has a blocking approval-safety bug where `env` options with operands can hide the real inner command from the simple summary.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` config with `original`, `simple`, and `simple-technical` plugin approval rendering, wired through shared renderers, Telegram/Matrix native approval surfaces, docs, baselines, and tests.

## Best Possible Solution

Land the default-preserving plain-language approval feature after the env wrapper parser consumes value-taking env flags or conservatively fails closed, with focused renderer regression coverage.

## Reproduction Assessment

Yes for the blocking review finding by source inspection: `env -u FOO rm -rf /tmp/x` reaches `resolveCommandWords`, which strips `env` and `-u` but leaves `FOO` as the command before the delete is summarized.

## Solution Assessment

No as proposed. The opt-in config and renderer approach is maintainable, but the command parser must handle value-taking `env` flags or fail closed before simple mode hides raw technical fields.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Parse env option operands before summarizing wrapped commands:** `src/infra/plugin-approvals.ts:1073-1075`
  - body: The new env-wrapper stripper removes dash-prefixed `env` options but does not consume required operands. For `env -u FOO rm -rf /tmp/x`, parsing stops at `FOO`, so simple mode can show a medium-risk unknown command while the actual command deletes files. Please parse value-taking env flags such as `-u`, `-C`, and `-S`, or fail closed when the wrapper cannot be safely unwrapped.
  - confidence: 0.89
