## Summary

Keep this PR open: the plain-language approval direction is useful, but the current renderer can still understate env-prefixed shell commands while hiding the raw command in `simple` mode, and the current-head real behavior proof is not enough for the Telegram-visible path.

## What This Changes

The PR adds configurable plain-language plugin approval rendering, wires it through shared fallback, Telegram, and Matrix approval paths, updates config docs/schema, and adds broad renderer/channel tests.

## Best Possible Solution

Keep the feature path, but make simple approval summaries fail closed or show a redacted command preview whenever env-prefixes can alter execution, then refresh after-fix proof through the affected chat approval path.

## Reproduction Assessment

Yes. Source inspection of the PR head shows leading `KEY=value` words are stripped before classification while `simple` mode can omit the raw command, so an env-prefixed known command can be summarized too narrowly without running the branch.

## Solution Assessment

No. The feature direction is maintainable, but the safe implementation needs conservative handling for env-prefixed commands before hiding raw technical text, plus current-head Telegram-visible proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Fail closed on env-prefixed commands:** `src/infra/plugin-approvals.ts:1796-1798`
  - body: `resolveCommandWords()` drops every leading `KEY=value` word before the simple renderer classifies the command. That means commands such as `LD_PRELOAD=./hook.so ls` or `BASH_ENV=./setup.sh bash -c 'echo ok'` can be shown as a low-risk read/format action while the raw command is hidden in `simple` mode, even though the env prefix can load or source attacker-controlled code. Treat nontrivial env prefixes as high-risk or include a redacted preview before suppressing the technical command.
