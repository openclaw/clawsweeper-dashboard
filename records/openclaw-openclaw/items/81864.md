## Summary

Keep open. The PR is active and has useful proof, but the current head still has a source-reproducible security classification gap in the new simple approval renderer, and CI has a PR-introduced lint blocker.

## What This Changes

The PR adds opt-in `approvals.plugin.language` rendering for plugin approval prompts across shared fallback, Telegram, Matrix, config help/docs/tests, while also carrying oc-path formatting churn and an `@grammyjs/types` package/lockfile change.

## Best Possible Solution

Keep the opt-in shared renderer design, but fail closed for every documented interpreter inline-eval family, preserve the default raw format, and land only after focused approval/channel validation is green.

## Reproduction Assessment

Yes. Source inspection of PR head shows the high-risk interpreter branch omits documented perl/php/lua/osascript forms, so representative inline-eval commands would stay under the generic unknown summary.

## Solution Assessment

No, not yet. The shared opt-in renderer is the right ownership boundary, but the classifier should match the documented security contract before simple mode hides the raw technical approval block.
