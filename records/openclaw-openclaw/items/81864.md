## Summary

Keep open: this is useful active work, but the current PR head still has approval-safety blockers where `simple` mode hides raw command details while dropping important risky context.

## What This Changes

The PR adds `approvals.plugin.language` with plain-language plugin approval rendering and wires it through shared fallback delivery for Telegram, Matrix, and generic plugin approval forwarding.

## Best Possible Solution

Keep the config-gated plain-language approval mode, but make simple output preserve or preview every hidden medium/high-risk command detail before raw technical fields are suppressed.

## Reproduction Assessment

Yes. Source inspection at PR head shows `pwd && ls && ps && cat notes.txt && rm -rf /tmp/x` can hide the delete target after the four-action cap, and `bash -c 'rm -rf "$1"' sh /tmp/x` drops `/tmp/x` when unwrapping the shell body.

## Solution Assessment

No, not as-is. The shared config-gated renderer is the right direction, but it needs to prioritize or preview hidden risky summaries and fail closed when shell-wrapper positional arguments affect the command body.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P1] Show risky actions before truncating:** `src/infra/plugin-approvals.ts:277-281`
  - body: `simple` mode removes raw technical details, but this loop emits only the first four action strings. A command like `pwd && ls && ps && cat notes.txt && rm -rf /tmp/x` can show high risk while hiding the concrete delete target behind `1 more background step(s)` with no preview, so risky summaries need priority or a forced preview before truncation.
  - confidence: 0.92

  - confidence: 0.87
