## Summary

Keep open. The feature direction and supplied proof are reasonable, but latest head still has a concrete approval-safety gap: curl data operands that become URL query data can carry token-like values while `simple` mode suppresses the raw command text.

## What This Changes

The PR adds opt-in `approvals.plugin.language` modes, a plain-English plugin approval renderer, channel/forwarder wiring, config/docs/baseline updates, focused tests, and a dependency lock adjustment.

## Best Possible Solution

Keep the opt-in plain-language approval feature, but fail closed for credential-like curl data operands before hiding raw command text and add focused renderer coverage.

## Reproduction Assessment

Yes. Source inspection plus the curl manual gives a high-confidence path: `curl -G --data-urlencode access_token=s3cr3t https://example.test` uses documented curl query-building behavior, while PR head only treats body flags as file/stdin uploads.

## Solution Assessment

No. The opt-in plain-language approach is maintainable, but the implementation should classify curl `-G` plus `-d`/`--data-urlencode` credential-looking operands as high-risk and force a redacted preview before `simple` mode suppresses the raw command.
