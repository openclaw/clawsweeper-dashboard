## Summary

Keep open: current main does not already provide plain-language plugin approval prompts, and the PR has useful proof, but PR head still has a blocking approval-safety defect where simple mode can hide leading environment assignments before showing a low-risk known-command summary.

## What This Changes

The branch adds `approvals.plugin.language` with plain-English plugin approval rendering and wires it through config, shared forwarding, Telegram/Matrix fallbacks, docs, generated baselines, and tests.

## Best Possible Solution

Keep the config-gated plain-language renderer with `original` as the default, but make simple mode fail closed or show a redacted preview whenever leading environment assignments can affect command behavior before hiding raw technical text.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows leading env assignments are stripped before classification, and `simple` mode returns before raw details are appended. I did not execute the renderer in this read-only review.

## Solution Assessment

No, not as-is. The UX direction is plausible, but the approval summarizer must preserve or conservatively fail closed on env-prefix context before it can safely hide the raw command.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Fail closed on env assignment prefixes:** `src/infra/plugin-approvals.ts:1796-1798`
  - body: `resolveCommandWords()` drops leading `KEY=value` words before command classification, and `simple` mode then returns without the raw command. Prefixes such as `PATH=./bin ls` or `LD_PRELOAD=./hook.so ls` can change command lookup or dynamic loading while rendering as an ordinary known command, so surface a redacted preview or force a high-risk fail-closed summary for prefixed known commands.
