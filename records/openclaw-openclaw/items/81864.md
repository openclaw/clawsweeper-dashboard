## Summary

This PR should stay open: the plain-language approval direction and supplied proof are useful, but the current head still has a security-sensitive renderer gap where shell-capable git forms can be hidden behind a generic simple approval.

## What This Changes

The PR adds `approvals.plugin.language` modes and a plain-English plugin approval renderer, wires it through forwarding/Telegram/Matrix paths, and updates config docs/schema/tests.

## Best Possible Solution

Keep the opt-in shared renderer/config approach, but make opaque or shell-capable git forms fail closed or show a redacted command preview before simple mode hides raw technical fields.

## Reproduction Assessment

Yes for the blocking finding: source inspection of PR head shows unknown git subcommands use the medium-risk no-preview fallback, and Git documents `submodule foreach` as arbitrary shell execution. The feature request itself is not a bug reproduction case.

## Solution Assessment

No, not yet. The opt-in renderer/config direction is maintainable, but simple mode must preserve command context or fail closed for opaque git commands before merge.
