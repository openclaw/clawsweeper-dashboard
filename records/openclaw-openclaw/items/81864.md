## Summary

Keep open: the PR is active and the plain-language approval direction has useful proof, but the current patch still has a concrete approval-safety bug where command targets can be skipped before simple mode hides the raw command.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` config, a plain-language plugin approval renderer, Telegram/Matrix/forwarder wiring, docs, generated baselines, and focused tests.

## Best Possible Solution

Land the default-preserving plain-language approval feature only after command target extraction is command-aware or fail-closed, with unrelated formatting churn split out or removed.

## Reproduction Assessment

Yes for the review finding from source inspection: at PR head, `cat -n .env` is parsed with `-n` as a value-taking option, so `.env` is skipped before sensitive-target checks. I did not run tests because this was a read-only review.

## Solution Assessment

No as proposed. The config and renderer direction is maintainable, but simple mode should not hide raw command text until target parsing is command-specific or conservatively high-risk for ambiguous flags.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Parse command flags per command before hiding targets:** `src/infra/plugin-approvals.ts:1203-1229`
  - body: `getCommandTargetArgs` treats flags such as `-n` and `-f` as value-taking for every command. For commands where those are boolean flags, such as `cat -n .env`, the sensitive path is skipped before target and risk checks, so simple mode can show a low-risk generic read while hiding the raw command; make this command-aware or fail closed for ambiguous flags.
  - confidence: 0.88
