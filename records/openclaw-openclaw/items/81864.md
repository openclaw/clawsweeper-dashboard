## Summary

Keep open: the config-gated plain-language approval UX is useful and the PR has real proof, but the current head still has a P1 approval-safety bug where simple-mode truncation can hide a later high-risk action while raw command details are intentionally removed.

## What This Changes

The PR adds `approvals.plugin.language` with `original`, `simple`, and `simple-technical` plugin approval rendering, wires it through shared forwarding plus Telegram/Matrix payloads, and updates config/docs/tests.

## Best Possible Solution

Keep the default-preserving config-gated renderer shape, but make simple mode always surface high-risk or preview-worthy actions before hiding raw technical fields.

## Reproduction Assessment

Yes for the review finding by source inspection: a command like `pwd && ls && ps && cat notes.txt && rm -rf /tmp/x` can place the delete target behind the generic background-step count because truncation happens before high-risk preservation. Reproduction is not applicable to the feature request itself.

## Solution Assessment

No, not as-is. The shared config-gated renderer is the right direction, but simple mode must promote high-risk or preview-worthy actions before truncating, or force a redacted preview whenever truncation hides them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Surface high-risk actions before truncating simple approvals:** `src/infra/plugin-approvals.ts:277-281`
  - body: `simple` mode hides raw command details, but this loop only renders the first four action summaries and collapses the rest. A command like `pwd && ls && ps && cat notes.txt && rm -rf /tmp/x` can show `Risk: High` while replacing the delete target with `1 more background step(s)`, and `rm` does not force a command preview. Prioritize high-risk/preview actions or force a preview whenever truncation would hide them.
  - confidence: 0.88
