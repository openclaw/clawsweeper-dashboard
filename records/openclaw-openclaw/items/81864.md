## Summary

Keep open: the feature is useful and the branch is currently mergeable with green CI, but the PR still under-classifies commands wrapped by value-taking `env` options, which is security-sensitive for approval copy.

## What This Changes

The branch adds an opt-in `approvals.plugin.language` setting for plain-language plugin approval prompts, wires it through shared renderers plus Telegram/Matrix delivery, updates docs/config baselines/tests, and carries unrelated formatting churn under `extensions/oc-path`.

## Best Possible Solution

Land the opt-in prompt-language feature after the command-wrapper parser consumes value-taking `env` options or fails closed, with a regression test for env-wrapped destructive commands.

## Reproduction Assessment

Yes, by source inspection: at PR head `resolveCommandWords()` strips `env` option tokens without consuming required arguments, so `env -u FOO rm -rf /tmp/x` parses `FOO` as the command. `env --help` confirms `-u`, `-C`, and `-S` take values.

## Solution Assessment

No as proposed. The opt-in config and channel wiring are maintainable, but the simple renderer must parse `env` wrappers correctly or conservatively fail closed before hiding raw technical details.
