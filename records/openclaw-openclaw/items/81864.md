## Summary

Keep open: the opt-in approval language feature has useful proof, but PR head still under-labels documented interpreter inline-eval commands in the security-sensitive simple renderer and CI is red.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` setting with `original`, `simple`, and `simple-technical` plugin approval rendering, wired through shared fallback, Telegram, Matrix, config help, docs, and tests.

## Best Possible Solution

Keep the opt-in shared renderer approach, but fail closed for all documented interpreter inline-eval families, fix the lint issue, preserve the default `original` behavior, and land only with focused approval plus channel validation green.

## Reproduction Assessment

Yes. Source inspection of PR head shows the interpreter high-risk branch omits documented perl, php, lua, and osascript forms, so representative inline-eval commands would not receive the intended high-risk interpreter classification.

## Solution Assessment

No, not yet. The shared opt-in renderer is the right ownership boundary, but it should match the documented interpreter risk family and clear CI before hiding raw technical approval details in simple mode.
