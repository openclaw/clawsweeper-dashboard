## Summary

Keep open: current main does not contain this feature, and the PR is still a useful implementation, but it is not merge-ready because the new approval summarizer under-labels documented interpreter commands and CI is red on a PR-introduced lint failure.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` setting with plain-English plugin approval rendering across shared fallback, Telegram, Matrix, config docs/tests, plus unrelated oc-path formatting and an `@grammyjs/types` downgrade.

## Best Possible Solution

Land the opt-in shared renderer only after the interpreter safety and lint fixes, preserving the `original` default and validating the shared, Telegram, Matrix, config, and renderer paths.

## Reproduction Assessment

Yes. Source inspection of PR head plus current security docs gives a high-confidence reproduction path: representative `perl -e`, `php -r`, `lua -e`, or `osascript -e` approvals fall through the generic medium-risk unknown path instead of the documented high-risk interpreter path.

## Solution Assessment

No, not yet. The opt-in shared renderer is the right ownership boundary, but it should fail closed for all documented interpreter inline-eval families and clear lint before hiding raw technical approval details in `simple` mode.
