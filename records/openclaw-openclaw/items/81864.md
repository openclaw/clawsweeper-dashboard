## Summary

Keep open: the latest PR head still has a blocking approval-safety issue where shell-valid attached redirections can be missed before plain-language mode hides the raw command.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` setting with `original`, `simple`, and `simple-technical` plugin approval rendering across shared reply builders, Telegram, Matrix, docs, config schema/help, and tests.

## Best Possible Solution

Keep the default-conservative opt-in design, but make shell redirection parsing fail closed for attached `<`/`>` forms and cover those cases with focused renderer regressions before merge.

## Reproduction Assessment

Yes for the blocking review finding by source inspection: the regexes require start-of-string or whitespace before redirection operators, while shell accepts attached forms like `cat notes.txt>out.txt`. I did not execute the PR branch because this review is read-only.

## Solution Assessment

No, not as currently proposed: the opt-in default is conservative, but `simple` mode must detect shell-valid attached redirections before it can safely hide raw technical command text.
