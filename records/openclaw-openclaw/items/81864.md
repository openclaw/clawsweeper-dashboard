## Summary

This PR should stay open: the approval-language direction is useful and has supplied proof, but simple mode still under-classifies npm/pnpm initializer aliases that can execute starter packages while hiding raw details.

## What This Changes

The PR adds `approvals.plugin.language` with `original`, `simple`, and `simple-technical` plugin approval rendering, wires it through shared forwarding and Telegram/Matrix fallbacks, and updates docs/schema/tests.

## Best Possible Solution

Keep the centralized configurable renderer, but classify npm/pnpm create/init initializer aliases as high-risk package-provided execution with redacted command previews and focused regressions before merge.

## Reproduction Assessment

Yes for the review finding: source inspection at PR head shows create/init aliases fall through the generic package-manager path, and official npm/pnpm docs define those forms as starter-package execution.

## Solution Assessment

No as-is. The renderer/config direction is maintainable, but initializer aliases need the same fail-closed treatment as exec, dlx, and x before simple mode hides raw details.
