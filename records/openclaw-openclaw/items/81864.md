## Summary

Keep this PR open: the opt-in plain-language renderer is a plausible direction and current CI/proof look strong, but the latest head still has a source-level approval-safety bug in shell splitting for escaped URL query separators.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` config and shared simple/simple-technical plugin approval rendering, wired through fallback, Telegram, Matrix, docs, and tests.

## Best Possible Solution

Land the opt-in shared renderer after preserving escaped shell separators through credential classification and adding a focused regression, while keeping `original` as the default.

## Reproduction Assessment

Yes for the review finding by source inspection: escaped `&` is split before URL credential analysis, so a shell-valid credential URL can be under-classified. I did not run PR-head tests in this read-only review.

## Solution Assessment

No. The shared opt-in renderer is the right ownership boundary, but the current parser should first preserve escaped shell metacharacters or otherwise classify escaped credential-bearing URLs as high-risk before simple mode hides raw technical details.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve escaped ampersands in shell splitting:** `src/infra/plugin-approvals.ts:561-563`
  - body: `splitCommandSegments` treats every non-redirection `&` as a shell separator without honoring a preceding backslash. A normal shell URL such as `curl https://example.test/path?x=1\&token=secret` is therefore split before `hasNetworkUrlCredentials` sees the full URL; the sensitive query fragment becomes a separate assignment-like segment and simple mode can report only medium network risk while hiding the raw credential-bearing command. Keep escaped `&` inside the segment, or conservatively mark this case high-risk and add a regression.
  - confidence: 0.86
