## Summary

Keep open: the latest branch fixes the previous package-manager parser finding, but still under-labels valid pnpm workspace-root installs in the new simple approval renderer.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` setting, plain-language plugin approval rendering, Telegram/Matrix forwarding support, docs/schema/test updates, and related generated/dependency churn.

## Best Possible Solution

Land the opt-in shared renderer after treating `pnpm -w` as a boolean global option, adding a focused regression, and keeping `original` as the default.

## Reproduction Assessment

Yes for the review finding by source inspection: `pnpm -w install` is valid, but the PR parser consumes `install` as the value for `-w` and never classifies the install subcommand. I did not run tests in this read-only review.

## Solution Assessment

No for the latest patch as-is. The shared opt-in renderer is a reasonable boundary, but package-manager parsing should move `-w` to boolean global flags or handle it command-specifically before hiding raw approval text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Treat pnpm -w as a boolean global flag:** `src/infra/plugin-approvals.ts:1377`
  - body: `pnpm -w install` is a valid workspace-root install, but `-w` is currently in `PACKAGE_MANAGER_GLOBAL_VALUE_OPTIONS`. That makes `parsePackageManagerSubcommand()` consume `install` as the option value and fall through to the generic medium-risk package-manager summary in `simple` mode, hiding the fact that dependencies may be installed or updated. Move `-w` to the boolean global flags, or handle it command-specifically, and add a regression for this form.
  - confidence: 0.9
