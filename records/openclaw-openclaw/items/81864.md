## Summary

Keep open: the feature is still useful, but the latest head has a concrete approval-safety parser bug where Bash `|&` pipelines are not treated as pipeline stages before simple mode can hide the raw command text.

## What This Changes

The PR adds opt-in plain-language plugin approval rendering via `approvals.plugin.language`, wires it through shared fallback, Telegram, and Matrix approval paths, and updates config docs, schema, baselines, and tests.

## Best Possible Solution

Keep the opt-in default-conservative approval-language design, but make shell pipeline detection fail closed for Bash `|&` and cover it with a focused renderer regression before merge.

## Reproduction Assessment

Yes. By source inspection, `cat .env |& custom-uploader --send` reaches the bare-ampersand branch before pipeline stage marking, so the unknown uploader is not promoted to the high-risk pipeline path.

## Solution Assessment

No, not yet. The feature direction is reasonable, but the parser needs a narrow `|&` fix and regression before hiding raw command text is safe enough to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat |& as a pipeline operator:** `src/infra/plugin-approvals.ts:561-563`
  - body: Bash treats `|&` as a pipeline operator, but this branch splits on the ampersand before `splitPipelineStages` can mark the right-hand command as `fromPipeline`. A command such as `cat .env |& custom-uploader --send` is therefore summarized as a medium-risk read plus unknown command instead of using the high-risk unknown-pipeline path; handle `|&` as a pipeline and add a regression.
