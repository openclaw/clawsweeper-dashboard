## Summary

Keep open: the opt-in plain-language approval renderer is useful and has real proof, but the current head still lets opaque shell-capable git commands render as medium-risk source-control text without a command preview while `simple` mode hides the raw technical details.

## What This Changes

The branch adds `approvals.plugin.language` with `original`, `simple`, and `simple-technical` plugin approval rendering, wires it through shared/channel fallback paths, updates docs/schema/tests, and includes several main-refresh fixes.

## Best Possible Solution

Land the opt-in shared renderer/config approach after opaque shell-capable git forms fail closed or force a redacted preview, with a focused regression test and a clean refresh against main.

## Reproduction Assessment

Not applicable for the feature request itself, but yes for the blocking finding: source inspection shows `git submodule foreach ...` reaches the unrecognized git fallback without a preview, and Git documents that form as arbitrary shell execution.

## Solution Assessment

No, not yet. The opt-in renderer/config direction is maintainable, but `simple` mode must fail closed or show a redacted preview for opaque shell-capable git commands before raw technical details are hidden.
