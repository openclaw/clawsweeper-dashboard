## Summary

Keep open. Current main does not already implement plain-language plugin approvals, and the active PR still needs changes for a concrete approval-safety parser edge plus refreshed real behavior proof after the latest security-oriented updates.

## What This Changes

The PR adds opt-in `approvals.plugin.language` modes for plain-language plugin approval prompts, wires them through shared, Telegram, and Matrix approval rendering, and updates config/docs/tests with extensive command-summary coverage.

## Best Possible Solution

Land the opt-in shared renderer after `>|` clobber redirections are handled as file writes with regression coverage and the latest head has refreshed real behavior proof for the approval text path.

## Reproduction Assessment

Yes for the review finding: source inspection shows the new pipeline splitter treats the `|` in `>|` as a pipeline separator, and a Bash scratch check confirmed `>|` performs a real overwrite. I did not run the PR test suite in this read-only review.

## Solution Assessment

No, not as-is. The opt-in renderer boundary is reasonable, but clobber redirections need to fail closed as writes before simple mode hides the raw technical block, and latest-head real behavior proof needs a refresh.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle clobber redirections before splitting pipelines:** `src/infra/plugin-approvals.ts:686-687`
  - body: `>|` is a valid shell output redirection; for example, `printf x >| .env` overwrites `.env` even when `noclobber` is enabled. The new splitter treats the `|` in that operator as a pipeline separator, so simple mode can summarize the tail as an unknown medium-risk command with a preview instead of a high-risk sensitive file write. Treat `>|` as redirection in the splitter and target extraction, and add a regression for a sensitive target.
  - confidence: 0.88
