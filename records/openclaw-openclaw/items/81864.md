Keep open: the latest PR head still has a concrete approval-safety bug in `simple` mode. Shell-valid redirects like `cat notes.txt>out.txt` are not detected by the new redirect parser, so a file write can be summarized as a low-risk read while the raw command is hidden.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` setting, plain-language plugin approval rendering, Telegram/Matrix/fallback wiring, config/docs updates, and approval renderer tests.

## Best Possible Solution

Fix the redirect parser and add no-space redirection regressions, then land only if maintainers accept the opt-in approval language and safety model.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR regex requires start-of-string or whitespace before `<`/`>`, while shell accepts attached redirects like `cat notes.txt>out.txt`. I did not execute the PR branch because this review is read-only.

## Solution Assessment

No, not as currently proposed: the opt-in default is conservative, but the parser must detect shell-valid attached redirections before `simple` mode hides the raw technical command.
