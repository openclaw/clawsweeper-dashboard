## Summary

Keep open. The plain-language approval direction is useful and the PR includes substantial proof, but the current head still has a P2 correctness bug: `simple-technical` can advertise the default manual decisions while the actual interactive actions are narrower.

## What This Changes

The PR adds opt-in `approvals.plugin.language` modes for plugin approval prompts, wires them into shared/Telegram/Matrix rendering, expands command summarization and redaction, updates docs/config/tests/baselines, and includes dependency plus oc-path formatting churn.

## Best Possible Solution

Keep the opt-in plain-language feature, but make every rendered manual approval instruction use the same effective allowed-decision list as the channel buttons/reactions before merge.

## Reproduction Assessment

Yes. From source, render a Matrix plugin approval in `simple-technical` mode with view actions `allow-once` and `deny`; the raw technical footer still derives defaults from the synthetic request and can include `allow-always`.

## Solution Assessment

No. The overall design is reasonable, but the PR should thread the effective allowed decisions into the technical/raw footer or otherwise ensure `simple-technical` cannot diverge from the interactive actions.
