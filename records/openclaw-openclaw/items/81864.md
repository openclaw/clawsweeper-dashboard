## Summary

Keep open. This active feature PR adds a useful opt-in plain-language approval path with real behavior proof and green current-head checks, but it still has a blocking approval-safety gap where opaque git subcommands can be summarized without the raw command before `simple` mode hides technical details.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` config, shared plain-language plugin approval rendering/redaction, Telegram/Matrix/fallback wiring, docs/schema/baseline updates, and several test/CI refresh fixes.

## Best Possible Solution

Land the opt-in shared renderer/config approach after fixing the git fail-closed gap, preserving `original` as the default and using redacted previews for opaque or shell-capable commands.

## Reproduction Assessment

Not applicable for the feature request itself, but the blocking review finding is source-reproducible from the PR head: opaque git subcommands fall through to medium-risk text with no preview while Git documents shell-executing aliases and `submodule foreach`.

## Solution Assessment

No, not yet. A shared renderer plus opt-in config is the right direction, but `simple` mode must fail closed or show a preview for shell-capable git forms before it hides raw technical text.
