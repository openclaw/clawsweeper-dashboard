## Summary

Keep open: current main still uses the raw plugin approval message, and this PR adds a real opt-in feature, but the latest head still under-detects shell-valid attached redirections in simple mode, which can make a write look like a low-risk read while hiding the raw command.

## What This Changes

The PR adds an opt-in `approvals.plugin.language` config, plain-language plugin approval rendering, docs/config schema updates, and Telegram/Matrix/fallback wiring.

## Best Possible Solution

Keep the opt-in default-conservative design, but make shell redirection parsing fail closed for attached `<`/`>` forms and cover those cases with focused renderer regressions before merge.

## Reproduction Assessment

Yes for the blocking review finding by source inspection: the PR regexes require start-of-string or whitespace before redirection operators, while shell accepts attached forms like `cat notes.txt>out.txt`. I did not execute the PR branch because this review is read-only.

## Solution Assessment

No, not as currently proposed: the opt-in default is conservative, but simple mode must detect shell-valid attached redirections before it can safely hide raw technical command text.
