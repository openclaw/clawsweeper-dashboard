## Summary

Keep open: the feature direction is useful and current checks/proof are mostly healthy, but the PR still has an approval-safety parser bug where a valid `command -p ...` wrapper is summarized as executable `-p` instead of the wrapped command.

## What This Changes

Adds `approvals.plugin.language` modes and wires plain-language plugin approval rendering through config, SDK reply payloads, Telegram/Matrix/fallback delivery, docs, generated baselines, and tests.

## Best Possible Solution

Keep the default-preserving plain-language approval mode, but make shell wrapper parsing command-aware or conservatively fail closed anywhere simple mode would otherwise understate executable behavior.

## Reproduction Assessment

Yes for the review finding by source inspection: `command -p rm -rf /tmp/x` flows through `resolveCommandWords()`, strips `command`, and leaves `-p` as the command head before the `rm` branch can classify the delete. I did not execute the PR branch locally because this review kept the checkout read-only.

## Solution Assessment

No, not as-is. The config-gated design is maintainable, but the parser should consume supported `command` builtin options or return a high-risk/fail-closed summary before simple mode hides the raw technical block.
