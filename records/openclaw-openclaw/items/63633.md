## Summary

Keep this PR open, but do not merge it as-is: the linked Telegram SOCKS5 media bug is real and source-reproducible, while the branch still misses the guarded media path and overclaims SOCKS variants that undici does not support.

## What This Changes

The PR branch adds SOCKS scheme detection to the shared proxy fetch helper and Telegram explicit-proxy dispatcher, imports undici `Socks5ProxyAgent`, and adds unit tests for SOCKS proxy agent selection and metadata reuse.

## Best Possible Solution

Rework around current main to support dependency-backed `socks5://` and `socks://` for the documented Telegram explicit-proxy media path, allow those schemes through the explicit-proxy guard without widening target SSRF policy, preserve Telegram dispatcher lifecycle/options, and drop unsupported SOCKS4/SOCKS5H claims unless separately implemented.

## Reproduction Assessment

Yes, by source inspection: configure `channels.telegram.proxy` with a `socks5://` URL, send Telegram media, and the media path reaches `fetchWithSsrFGuard()`, where current main rejects the explicit proxy protocol before any SOCKS dispatcher is used.

## Solution Assessment

No. The branch is aimed at the right bug, but the fix belongs in the guarded explicit-proxy media path and must align with undici's actual SOCKS protocol support instead of only swapping dispatcher constructors.

## Review Findings

Full review comments:

- **[P1] Allow SOCKS through the guarded media path:** `extensions/telegram/src/fetch.ts:276`
  - body: This dispatcher branch still does not fix `channels.telegram.proxy: socks5://` media downloads. Telegram media downloads call `fetchRemoteMedia()`, which calls `fetchWithSsrFGuard()` with the explicit proxy policy, and the guard rejects non-HTTP proxy protocols before dispatcher construction. Update the explicit-proxy guard and its SSRF tests along with the Telegram dispatcher, or the reported path will still throw `Explicit proxy URL must use http or https`.
  - confidence: 0.92
- **[P2] Drop unsupported schemes from the shared SOCKS detector:** `src/infra/net/proxy-fetch.ts:20`
  - body: Undici 8.2.0's `Socks5ProxyAgent` only accepts `socks5:` and `socks:`, but this detector routes `socks4:`, `socks4a:`, and `socks5h:` into it. Those configurations will fail immediately despite the PR claiming support, so restrict the shared detector/tests to supported schemes or add a real implementation for the extra protocols.
  - confidence: 0.95
- **[P2] Keep Telegram SOCKS detection aligned with undici:** `extensions/telegram/src/fetch.ts:269`
  - body: The Telegram-specific detector repeats the same unsupported `socks4:`, `socks4a:`, and `socks5h:` schemes before constructing `Socks5ProxyAgent`. Even if the shared helper is fixed, this branch would still fail those Telegram proxy configs at dispatcher creation, so keep this list to `socks5:`/`socks:` or implement the unsupported protocols separately.
  - confidence: 0.95
