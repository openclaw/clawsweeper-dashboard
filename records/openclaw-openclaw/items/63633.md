## Summary

Keep open. The PR is a plausible fix attempt for a documented Telegram SOCKS5 proxy bug, but it still misses the guarded media-download path, overclaims unsupported SOCKS variants, and lacks required real Telegram behavior proof.

## What This Changes

The branch adds SOCKS scheme detection to shared proxy fetch and Telegram explicit-proxy dispatch, imports undici `Socks5ProxyAgent`, and adds unit tests for proxy fetch routing.

## Best Possible Solution

Rework around current main by allowing only dependency-backed `socks5://` and `socks://` through the guarded Telegram media path while preserving proxy-host validation, Telegram target allowlisting, `createHttp1ProxyAgent()` options, and real Telegram proof.

## Reproduction Assessment

Yes. Source inspection shows a documented `channels.telegram.proxy: socks5://...` media download reaches `saveRemoteMedia()` and then `fetchWithSsrFGuard()`, where current main rejects explicit proxy protocols outside `http:` and `https:`; no live Telegram run was performed in this read-only review.

## Solution Assessment

No. The PR is directionally useful, but the maintainable fix should update the guarded explicit-proxy policy and reuse the current ProxyAgent wrapper instead of adding duplicate direct `Socks5ProxyAgent` branches that overmatch unsupported schemes and drop dispatcher options.
