## Summary

Keep open: the documented Telegram SOCKS5 media bug is valid, but this PR still misses the guarded media path, overclaims SOCKS schemes that undici rejects, drops current dispatcher options, and lacks after-fix real behavior proof.

## What This Changes

The PR adds SOCKS scheme detection in shared proxy fetch and Telegram explicit-proxy dispatchers, imports undici `Socks5ProxyAgent`, and adds proxy-fetch unit tests.

## Best Possible Solution

Rework against current main to support only undici-backed `socks5://` and `socks://` for the documented Telegram media proxy path while preserving SSRF validation, HTTP/1 dispatcher options, proxy-host checks, and focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible from current main and the linked reporter log. Configure `channels.telegram.proxy` with `socks5://...`, send Telegram media, and the media path reaches `fetchWithSsrFGuard()`, where `assertExplicitProxyAllowed()` rejects the SOCKS proxy URL before dispatcher construction.

## Solution Assessment

No. The PR changes dispatcher selection but leaves the guard that blocks the reported media path, advertises unsupported SOCKS schemes, and bypasses current Telegram dispatcher option handling; a safer fix should update the guarded explicit-proxy policy and dependency-backed scheme list together.
