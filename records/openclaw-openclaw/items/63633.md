## Summary

Keep open. The PR targets a real documented Telegram SOCKS5 media-download bug, but the branch still misses the guarded media-fetch policy, overclaims SOCKS protocols that undici does not support, bypasses current proxy dispatcher options, lacks real behavior proof, and is now conflicting with current main.

## What This Changes

The PR adds SOCKS scheme detection to shared proxy fetch and Telegram explicit-proxy dispatch, imports undici `Socks5ProxyAgent`, and adds proxy-fetch unit coverage.

## Best Possible Solution

Rework around current main: allow dependency-backed `socks5://` and `socks://` through the guarded Telegram media path, keep proxy-host validation and Telegram allowlisting, reuse `createHttp1ProxyAgent()`, and remove or separately implement unsupported SOCKS4/SOCKS5H claims.

## Reproduction Assessment

Yes, as a source-reproducible path: documented `channels.telegram.proxy: socks5://...` media downloads reach `saveRemoteMedia()` and then `fetchWithSsrFGuard()`, where current main rejects explicit proxy protocols outside `http:` and `https:`. I did not run a live Telegram/SOCKS repro in this read-only review.

## Solution Assessment

No. The branch is directionally useful, but the maintainable fix should update the guarded explicit-proxy policy and reuse the current `ProxyAgent` wrapper instead of adding duplicate direct `Socks5ProxyAgent` branches that overmatch unsupported schemes and drop dispatcher options.
