## Summary

Keep open. The linked Telegram SOCKS5 media-download bug is real and still relevant, but this PR is not correct as-is because it does not update the guarded media fetch policy, overclaims unsupported SOCKS schemes, and still lacks after-fix real Telegram proof.

## What This Changes

This PR adds SOCKS scheme detection to shared proxy fetch and Telegram explicit-proxy dispatcher construction, imports undici `Socks5ProxyAgent`, and adds proxy-fetch unit tests for SOCKS routing and metadata reuse.

## Best Possible Solution

Rework the PR to support dependency-backed `socks5://` and `socks://` for the documented Telegram media proxy path by updating the SSRF explicit-proxy policy, preserving existing Telegram dispatcher options, removing unsupported SOCKS4/SOCKS5H claims, and adding focused guard/media tests plus redacted live Telegram proof.

## Reproduction Assessment

Yes. The issue is source-reproducible on current main: configure `channels.telegram.proxy` with `socks5://...`, send Telegram media, and the media download path reaches `fetchWithSsrFGuard()`, where `assertExplicitProxyAllowed()` rejects the proxy protocol before dispatch.

## Solution Assessment

No. The PR changes dispatcher selection but misses the blocking guard and overstates undici-supported schemes; the narrower maintainable fix is to update the guarded explicit-proxy policy and tests while preserving current Telegram dispatcher behavior.
