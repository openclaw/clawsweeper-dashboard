## Summary

Keep open. Current main still documents `channels.telegram.proxy: socks5://...` while the guarded Telegram media path rejects non-HTTP explicit proxy URLs before any dispatcher is used, so the user problem is real and not implemented on main. The PR is not mergeable as-is because it misses that guard path and routes several SOCKS schemes that undici 8.1.0 does not support.

## What This Changes

The PR branch imports undici `Socks5ProxyAgent`, adds SOCKS scheme detection in the shared proxy fetch helper and Telegram explicit-proxy dispatcher, and adds proxy-fetch unit coverage for SOCKS agent selection and metadata reuse.

## Best Possible Solution

Rework the PR around current main by allowing only dependency-backed `socks5://` and `socks://` explicit proxies through `fetchWithSsrFGuard()` for the documented Telegram media path, while preserving proxy-host validation, Telegram media hostname allowlisting, existing dispatcher pool/TLS/lookup behavior, and removing unsupported SOCKS4/SOCKS5H claims unless real support is added.

## Reproduction Assessment

Yes. The linked issue gives a concrete repro with `channels.telegram.proxy: socks5://...` and a Telegram image, and current source confirms the media path reaches `assertExplicitProxyAllowed()` before dispatcher construction, where `socks5:` is rejected.

## Solution Assessment

No. The PR changes the dispatcher selection but misses the guard that blocks the reported media path and overclaims unsupported protocols; the safer solution is a narrow guard/policy update for undici-supported SOCKS schemes while preserving the existing Telegram proxy dispatcher options.
