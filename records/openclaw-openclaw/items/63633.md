## Summary

Keep this PR open, but do not merge it as-is: the Telegram SOCKS5 media bug is still real on current main, while this branch misses the current SSRF guard that rejects the reported path and overclaims SOCKS schemes undici does not support.

## What This Changes

The PR branch imports undici `Socks5ProxyAgent`, detects SOCKS schemes in shared proxy fetch and Telegram explicit-proxy dispatch, and adds proxy-fetch unit coverage for SOCKS agent selection and metadata reuse.

## Best Possible Solution

Rework against current main by allowing only undici-backed `socks5://` and `socks://` explicit proxies through the guarded Telegram media path while preserving proxy-host validation, Telegram target allowlisting, HTTP/1/pool behavior, and existing dispatcher options.

## Reproduction Assessment

Yes. The linked bug gives a concrete `channels.telegram.proxy: socks5://...` media-download repro and log message, and current main confirms the media path still reaches `assertExplicitProxyAllowed()`, where `socks5:` is rejected.

## Solution Assessment

No. The branch changes dispatcher selection but misses the guard that blocks the user-visible media path and adds unsupported scheme claims; the maintainable fix is a narrow current-main SSRF-policy update with dependency-backed protocol scope and focused media-path tests.
