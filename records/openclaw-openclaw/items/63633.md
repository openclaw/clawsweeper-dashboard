## Summary

Keep open. The SOCKS media bug is still valid, but this branch is not merge-ready because it bypasses current proxy/SSRF architecture, leaves the guarded Telegram media path blocked, overclaims unsupported SOCKS schemes, and lacks usable real behavior proof.

## What This Changes

The branch adds SOCKS scheme detection to the shared proxy fetch helper and Telegram explicit-proxy dispatcher, imports undici `Socks5ProxyAgent`, and adds unit tests for SOCKS proxy fetch routing.

## Best Possible Solution

Rework on current main by allowing only undici-supported `socks5://` and `socks://` through explicit-proxy SSRF validation and the existing `createHttp1ProxyAgent` dispatcher path, with focused tests and live Telegram media proof.

## Reproduction Assessment

Yes, at source level: a `channels.telegram.proxy` SOCKS URL flows into Telegram media `saveRemoteMedia()`, which calls `fetchWithSsrFGuard()`, and current main still rejects non-HTTP(S) explicit proxy protocols. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, this is not the best fix as written. Undici's `ProxyAgent` already delegates `socks5:` and `socks:` to `Socks5ProxyAgent`, so the safer path is to keep OpenClaw's existing proxy wrapper and adjust the SSRF guard for supported SOCKS schemes.
