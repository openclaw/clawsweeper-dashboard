## Summary

Keep open, but do not merge as-is. Current main still has the documented Telegram SOCKS5 media-download gap, while this PR misses the SSRF guard path that blocks media fetches and overclaims SOCKS schemes undici does not support.

## What This Changes

The PR imports undici `Socks5ProxyAgent`, routes SOCKS-looking proxy URLs in shared proxy fetch and Telegram explicit-proxy dispatcher code to it, and adds proxy-fetch unit tests.

## Best Possible Solution

Rework on current main to allow only undici-supported `socks5://` and `socks://` explicit Telegram proxies through the guarded media path while preserving proxy-host validation, target allowlisting, HTTP/1 dispatcher behavior, and focused regression coverage.

## Reproduction Assessment

Yes, from source and the linked reporter log rather than a live run. Configure `channels.telegram.proxy` with `socks5://...`, send Telegram media, and current main reaches `fetchWithSsrFGuard()`, where `assertExplicitProxyAllowed()` rejects the SOCKS proxy URL.

## Solution Assessment

No. The PR changes dispatcher selection but does not update the guard that blocks the media path, and it claims SOCKS4/SOCKS5H schemes that undici's pinned `Socks5ProxyAgent` rejects.
