## Summary

Keep open: this is a real documented Telegram SOCKS media bug, but the branch still misses the guarded media-fetch policy and adds unsupported SOCKS scheme claims.

## What This Changes

The PR adds SOCKS scheme detection to shared proxy fetch and Telegram explicit-proxy dispatcher construction, imports undici `Socks5ProxyAgent`, and adds proxy-fetch unit tests.

## Best Possible Solution

Rework against current main to allow only dependency-backed `socks5://` and `socks://` through the explicit-proxy SSRF guard, preserve the existing `ProxyAgent`/HTTP1 dispatcher path, and add media-path regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `channels.telegram.proxy` with `socks5://...`, send Telegram media, and the guarded media fetch rejects the explicit proxy URL before dispatcher construction.

## Solution Assessment

No. The PR is directionally useful but not the best fix because current undici `ProxyAgent` already delegates `socks5:`/`socks:`, so the safer fix is to update the guarded explicit-proxy policy and tests rather than add parallel SOCKS branches.
