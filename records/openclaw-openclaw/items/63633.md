## Summary

Keep open: current main still has the documented SOCKS5 Telegram media-download gap, but this PR is not mergeable as-is because it misses the guarded media policy path, overclaims unsupported SOCKS schemes, conflicts with current main, and lacks after-fix real Telegram proof.

## What This Changes

The PR adds SOCKS scheme detection to shared proxy fetch and Telegram explicit-proxy dispatcher construction, imports undici `Socks5ProxyAgent`, and adds proxy-fetch unit tests.

## Best Possible Solution

Rework to support only dependency-backed `socks5://` and `socks://` for the documented Telegram explicit-proxy media path by allowing those protocols through the SSRF guard while preserving current ProxyAgent HTTP/1, pool, requestTls, proxy-host validation, and Telegram target allowlist behavior.

## Reproduction Assessment

Yes, from source inspection: configure `channels.telegram.proxy` with `socks5://...`, send Telegram media, and the media path reaches `fetchWithSsrFGuard()` with an explicit proxy policy that still rejects non-http(s) proxy URLs. I did not run a live Telegram/SOCKS setup in this read-only review.

## Solution Assessment

No. The PR is directionally useful, but the narrower maintainable fix is to update the guarded explicit-proxy policy and media-path tests while relying on undici `ProxyAgent` for supported SOCKS delegation instead of adding duplicate branches for unsupported schemes.
