## Summary

Keep open: the reported Telegram SOCKS proxy bug is real, but this PR is conflicting and not merge-ready because it misses the guarded media-download path, overclaims unsupported SOCKS schemes, bypasses current dispatcher behavior, and lacks inspectable real behavior proof.

## What This Changes

The PR adds SOCKS scheme detection to shared proxy fetch and Telegram explicit-proxy dispatch, imports undici `Socks5ProxyAgent`, and adds unit coverage for SOCKS proxy fetch routing.

## Best Possible Solution

Rework the fix around current main: allow dependency-backed `socks5://` and `socks://` through the guarded explicit-proxy media policy, keep proxy-host validation and Telegram allowlisting, reuse the shared proxy-agent wrapper, and remove or separately implement unsupported SOCKS variants.

## Reproduction Assessment

Yes, at source level: documented `channels.telegram.proxy: socks5://...` media downloads reach `saveRemoteMedia()` and then `fetchWithSsrFGuard()`, where current main rejects explicit proxy protocols outside `http:` and `https:`. I did not run a live Telegram/SOCKS repro in this read-only review.

## Solution Assessment

No. The branch is directionally useful, but the maintainable fix should update the guarded explicit-proxy policy and reuse the current shared proxy wrapper instead of adding duplicate direct `Socks5ProxyAgent` branches that overmatch unsupported schemes and drop dispatcher options.
