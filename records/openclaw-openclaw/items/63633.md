## Summary

Keep open. This PR targets a documented Telegram SOCKS proxy bug, but the current branch still misses the guarded media-download path, overclaims unsupported SOCKS schemes, lacks after-fix real Telegram proof, and conflicts with current main.

## What This Changes

The branch adds SOCKS scheme detection to shared proxy fetch and Telegram explicit-proxy dispatch, imports undici `Socks5ProxyAgent`, and adds proxy-fetch unit tests.

## Best Possible Solution

Rework the PR around current main to support only dependency-backed `socks5://` and `socks://` for the documented Telegram explicit-proxy media path while preserving proxy-host validation, Telegram target allowlisting, HTTP/1 pool behavior, request TLS, and pinned lookup settings.

## Reproduction Assessment

Yes. Source inspection shows `channels.telegram.proxy: socks5://...` reaches Telegram media download with an explicit proxy policy, then `fetchWithSsrFGuard()` rejects the proxy URL before the PR's dispatcher branch can make the media path work.

## Solution Assessment

No. The PR is directionally useful, but the maintainable fix needs to update the guarded explicit-proxy policy and media-path coverage while relying on undici's supported SOCKS contracts and preserving existing Telegram dispatcher options.
