## Summary

Keep open. The PR targets a real documented Telegram SOCKS proxy media bug, but the branch still misses the guarded media fetch policy, overclaims unsupported SOCKS variants, lacks after-fix real Telegram proof, and is now conflicting with current main.

## What This Changes

The PR adds undici `Socks5ProxyAgent` routing for SOCKS proxy URLs in shared proxy fetch and Telegram explicit-proxy dispatch, plus proxy-fetch unit tests.

## Best Possible Solution

Rework this around current main to support only dependency-backed `socks5://` and `socks://` for the documented Telegram explicit-proxy media path while preserving proxy-host validation, Telegram target allowlisting, HTTP/1 pool behavior, request TLS, and pinned lookup settings.

## Reproduction Assessment

Yes. From source inspection, `channels.telegram.proxy: socks5://...` reaches Telegram media download with an explicit proxy policy, then `fetchWithSsrFGuard()` rejects the proxy URL before the PR's dispatcher branch can make the media path work.

## Solution Assessment

No. The PR is directionally useful, but the maintainable fix needs to update the guarded explicit-proxy policy and media-path coverage while relying on undici's supported SOCKS contracts and preserving existing Telegram dispatcher options.
