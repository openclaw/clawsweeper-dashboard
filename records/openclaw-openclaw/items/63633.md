## Summary

Keep open: this is a real documented Telegram SOCKS media bug, but the PR is not merge-ready because the guarded media path still rejects SOCKS proxy URLs, the patch advertises schemes undici rejects, Telegram dispatcher options are not preserved, and after-fix real behavior proof is missing.

## What This Changes

The PR imports undici `Socks5ProxyAgent`, routes SOCKS-looking proxy URLs in shared proxy fetch and Telegram explicit-proxy transport to it, and adds proxy-fetch unit tests.

## Best Possible Solution

Rework on current main to support only dependency-backed `socks5://` and `socks://` for the documented Telegram explicit-proxy media path while preserving guard validation, Telegram dispatcher options, and focused regression coverage.

## Reproduction Assessment

Yes, from source and the reporter log rather than a live run: configure `channels.telegram.proxy` with `socks5://...`, send Telegram media, and current main reaches `fetchWithSsrFGuard()`, where `assertExplicitProxyAllowed()` rejects the proxy URL.

## Solution Assessment

No. The PR changes dispatcher selection but leaves the blocking guard in place, routes unsupported SOCKS schemes to undici, and bypasses current Telegram dispatcher options; the safer fix is a focused current-main rework of the guarded explicit-proxy policy and dispatcher construction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Allow SOCKS through the guarded media fetch:** `extensions/telegram/src/fetch.ts:277`
  - body: Adding a `Socks5ProxyAgent` branch here does not fix Telegram media downloads: `downloadAndSaveTelegramFile()` passes the explicit-proxy policy into `fetchRemoteMedia()`, which calls `fetchWithSsrFGuard()`, and the guard still rejects `socks5://` before this dispatcher can be used. Update the guarded explicit-proxy policy path and cover the media download flow.
  - confidence: 0.95
- **[P2] Stop advertising unsupported SOCKS schemes:** `src/infra/net/proxy-fetch.ts:20`
  - body: The scheme detection routes `socks4:`, `socks4a:`, and `socks5h:` into undici `Socks5ProxyAgent`, but undici 8.2.0 accepts only `socks5:` and `socks:`. Those configs will still fail at runtime despite the new tests, so restrict the scheme list or implement real support separately.
  - confidence: 0.93
- **[P2] Preserve Telegram proxy dispatcher options:** `extensions/telegram/src/fetch.ts:277`
  - body: The SOCKS branch constructs a bare `Socks5ProxyAgent(proxyUrl)`, while current Telegram dispatchers carry HTTP/1-only pool behavior plus pinned lookup/TLS settings through the proxy options path. A correct SOCKS path needs to preserve the equivalent transport policy instead of dropping those settings.
  - confidence: 0.86
