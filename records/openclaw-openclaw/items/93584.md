## Summary

Keep open: the PR is a useful fix with strong live proof, but it still leaves a source-level compaction path where repaired messages are not adopted before `summarizeInStages` when pruning is not triggered and recent-turn preservation is disabled.

## What This Changes

The branch applies `repairToolUseResultPairing(..., { erroredAssistantResultPolicy: "drop" })` in compaction planning/safeguard paths and adds regression tests for orphaned tool calls.

## Best Possible Solution

Every built-in compaction summarization path should pass only repaired transcripts to strict providers while keeping the existing aborted-turn drop policy explicit and covered by regression tests.

## Reproduction Assessment

Yes. The source path is clear: the PR-head safeguard only adopts repaired pruned messages inside the prune branch, while `splitPreservedRecentTurns` returns raw messages when preservation is disabled.

## Solution Assessment

No, not yet. Reusing `repairToolUseResultPairing` with the drop policy is the right boundary, but it needs to run unconditionally before built-in LLM summarization rather than only through prune or preservation-dependent paths.
