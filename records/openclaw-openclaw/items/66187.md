## Summary

Keep open: current `main` and the latest release still intentionally classify an `endedAt` parent with pending descendants as active, and `sessions_spawn` uses that same active count for the child quota. The stale-unended orphan variant was fixed separately, but this issue remains the canonical open item for deciding how quota accounting should differ from list/display and cleanup liveness.

## What This Changes

The issue reports that subagent runs with `endedAt` set can still appear active and consume child-run quota when descendants are pending.

## Best Possible Solution

Define separate predicates for display/announce liveness, cleanup-pending tracking, and max-children quota so ended controllers do not permanently consume spawn slots while child cleanup remains observable.

## Reproduction Assessment

Yes, at source level: current predicates and tests preserve ended-parent-with-pending-descendant active behavior, and `sessions_spawn` uses that count for quota gating. I did not run a live nested subagent scenario in this read-only review.

## Solution Assessment

No, the current implementation is not the best complete solution for this report because it uses one active model for display and quota. A cleaner fix would separate user-visible cleanup tracking from child-slot accounting after a maintainer decision.
