## Summary

Keep open. Current main and the latest release still classify an ended orchestrator as active when pending descendants remain, and the same predicate feeds the spawn quota path; related open work argues for even more cleanup-pending liveness, so this needs a maintainer lifecycle/quota decision rather than cleanup closure.

## What This Changes

The issue reports that ended subagent runs remain listed as active and continue consuming spawn quota when descendant cleanup is pending.

## Best Possible Solution

Define separate lifecycle concepts for user-visible active work, cleanup settlement, and spawn quota, then apply that contract consistently across list output, registry counters, native/ACP spawn, archive retention, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: an ended parent with pending cleanup descendants still passes the active-list predicate and active-child quota count on current main. The tests also preserve the ended-parent-with-pending-descendant active count.

## Solution Assessment

Unclear until maintainers choose the lifecycle policy. A list-only change would not fix quota exhaustion; the maintainable fix is to split cleanup-aware liveness from active-list and quota liveness if the reported behavior is accepted.
