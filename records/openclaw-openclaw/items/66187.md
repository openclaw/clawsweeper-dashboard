## Label Justifications

- `P1`: The reported lifecycle accounting can exhaust subagent spawn quota and block nested multi-agent orchestration.
- `impact:session-state`: The issue is about stale or mismatched subagent run lifecycle state being treated as active after `endedAt` is set.

## Summary

Keep open. Current main and v2026.5.18 still classify an ended subagent run as active when pending descendants remain, and that same count feeds native and ACP spawn quota rejection; the open related PR points in the opposite cleanup-pending direction, so this needs a maintainer lifecycle/quota decision rather than cleanup closure.

## What This Changes

Treat subagent runs with `endedAt` set as non-active for list output and spawn quota accounting, with eventual archive/prune behavior.

## Best Possible Solution

Define separate lifecycle concepts for user-visible active work, cleanup settlement, spawn quota, and retention, then apply that contract consistently across list/status output, registry counters, native/ACP spawn, archive cleanup, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: an ended parent with pending descendants still passes the active-list predicate and active-child quota count on current main and v2026.5.18. I did not run a live nested subagent workflow because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the lifecycle policy. A list-only change would not fix quota exhaustion; the maintainable fix is to split cleanup-aware liveness from display and quota liveness if the reported behavior is accepted.
