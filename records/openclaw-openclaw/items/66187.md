## Label Justifications

- `P1`: The reported state leak can block nested subagent orchestration by exhausting the per-session active-child quota.
- `impact:session-state`: The issue is about subagent run lifecycle state drifting between `endedAt`, descendant cleanup, active listing, and quota accounting.

## Summary

Keep open. Current `main` and the latest release still treat an ended parent run as active when pending descendants remain, and the same predicate feeds the per-session spawn quota; the related open PR argues for preserving cleanup-pending liveness, so this needs a maintainer lifecycle/quota decision rather than cleanup closure.

## What This Changes

The issue asks for ended subagent runs to leave the active list and stop consuming the per-session spawn quota even when descendant state remains pending.

## Best Possible Solution

Maintainers should define one lifecycle model that separates terminal parent status, cleanup/announcement pending state, active descendant visibility, and spawn-quota accounting, then update list/status/quota tests together.

## Reproduction Assessment

Yes, from source inspection: seed an ended parent run with a pending descendant lacking `cleanupCompletedAt`; `buildSubagentList` keeps it active and `countActiveRunsForSession` can feed the `sessions_spawn has reached max active children` rejection. I did not run a live gateway reproduction because this review is read-only.

## Solution Assessment

Unclear as requested: making every `endedAt` run inactive would contradict current tests and the linked cleanup-pending PR. The safer direction is a maintainer-approved lifecycle split rather than flipping all active predicates blindly.
