## Label Justifications

- `P1`: Nested subagent orchestration can become unable to spawn more child work when lifecycle state consumes all configured child slots.
- `impact:session-state`: The issue is about subagent registry liveness and session quota accounting diverging from actual run completion.

## Summary

Keep open: current main and v2026.6.10 still classify an ended direct child as active when its child session has pending descendants, and `sessions_spawn` uses that same active count for the child-slot quota. This remains the canonical source-reproducible issue for splitting cleanup/display liveness from spawn quota semantics.

## What This Changes

The issue reports that completed subagent runs with `endedAt` can still appear active and consume subagent concurrency quota.

## Best Possible Solution

Define separate predicates for subagent display/cleanup visibility and spawn quota so ended controllers can remain observable during descendant cleanup without indefinitely consuming new child-spawn slots.

## Reproduction Assessment

Yes, at source level: current list and quota predicates keep ended parents active while descendants remain pending, and focused tests preserve that behavior. I did not run a live nested subagent scenario in this read-only review.

## Solution Assessment

No, the current implementation is not the best complete solution because one active model drives both display/cleanup visibility and spawn quota. The safer repair is a maintainer-approved split of those semantics rather than a broad new config knob or a one-line predicate flip.
