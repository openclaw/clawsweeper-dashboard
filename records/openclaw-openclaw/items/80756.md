## Label Justifications

- `P2`: This is a normal-priority Telegram/plugin command regression with limited blast radius: commands still work, but users receive a misleading extra failure reply.
- `impact:message-loss`: The issue is about duplicate/spurious channel messages after a plugin command has already delivered its intended Telegram reply.

## Summary

Keep this issue open: current main and the latest release still lack a silent `registerCommand` result, the Telegram fallback path is source-reproducible, and open PR https://github.com/openclaw/openclaw/pull/80928 is already the canonical closing implementation path.

## What This Changes

The issue requests a `registerCommand` handler result such as `{ suppressReply: true }` or `{ handled: true }` so Telegram plugins that send their own reply do not also get the empty-response fallback.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/80928 with the SDK contract, Telegram regression coverage, and attached real Telegram proof, then close this issue after that decision.

## Reproduction Assessment

Yes, source-reproducible. Current main maps undefined command results to `{}`, Telegram treats that as non-renderable, and existing coverage asserts the empty-response fallback, although I did not run live Telegram in this read-only review.

## Solution Assessment

Mostly yes. An explicit silent command-result flag is a narrow maintainable fix, but maintainers still need to approve the public SDK naming and channel scope in the linked PR.
