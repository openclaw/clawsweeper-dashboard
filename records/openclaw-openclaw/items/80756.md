## Summary

This should stay open: current main still has no command-result contract for a plugin command that has already delivered its own Telegram reply, and an open fixing PR references this issue with closing syntax. The issue should close only after that PR or an equivalent maintainer-approved fix lands.

## What This Changes

Add a registerCommand result, such as suppressReply or handled, that lets plugin-owned Telegram sends suppress the fallback reply without sending a second OpenClaw message.

## Best Possible Solution

Choose the public no-reply command result contract, then land an SDK type, Telegram native command delivery, docs, and regression-test update through the open fixing PR or an equivalent replacement.

## Reproduction Assessment

Yes, source-reproducible: undefined command results are normalized to an empty object, Telegram maps non-renderable results to the fallback, and a current test expects that fallback. I did not run a live Telegram or Windows reproduction.

## Solution Assessment

Unclear: a first-class suppressReply-style result is plausible, but maintainers should choose the public result name and decide whether direct Telegram transport from plugins is supported or should move through a host-mediated send seam.
