## Summary

This should stay open: current main still has no command-result contract for “handled with no OpenClaw reply,” and Telegram native commands intentionally fall back when a plugin command result has no renderable payload. The report is concrete, but the requested fix is a new public plugin SDK/channel behavior decision rather than a cleanup-close candidate.

## What This Changes

Add a registerCommand handler result that lets a plugin suppress OpenClaw’s fallback reply after the plugin has delivered its own Telegram response.

## Best Possible Solution

Define a small backward-compatible command result contract for terminal no-reply handling, then update SDK types/docs, Telegram native command delivery, and focused regression coverage around undefined, empty payload, continueAgent, and the new silent result.

## Reproduction Assessment

Yes, from current source and tests: an undefined command handler result becomes an empty object, and Telegram native command delivery maps that to the fallback message. I did not run a live Telegram/Windows reproduction in this read-only review.

## Solution Assessment

Unclear: a first-class silent command result is plausible, but maintainers should choose the naming and decide whether direct Telegram transport from plugins is supported or whether this should be a host-mediated send seam instead.
