## Summary

Keep open: current main still lacks a silent `registerCommand` result contract, and the open linked PR is the canonical implementation path. The issue should close only after that PR lands or maintainers explicitly reject the SDK/channel behavior.

## What This Changes

Add a first-class silent `api.registerCommand` handler result so Telegram plugins that send their own response can suppress the empty-response fallback.

## Best Possible Solution

Land a documented SDK command-result contract with focused Telegram fallback coverage and real Telegram proof, then close this issue after merge; if maintainers reject the contract, close with that rationale.

## Reproduction Assessment

Yes. Current source shows `executePluginCommand` returns `{}` for undefined results and Telegram replaces non-renderable payloads with the empty-response fallback; existing coverage asserts that path, and the linked PR's Mantis baseline reproduced the visible Telegram symptom.

## Solution Assessment

Mostly yes. An explicit silent command-result flag is a narrow maintainable solution, but the public SDK semantics and channel scope should be settled in the linked implementation PR before closing this issue.
