## Summary

Keep open: current main and the latest release still lack a silent `registerCommand` result contract, and source/test evidence shows empty Telegram plugin-command results still fall through to the fallback. The open linked PR is the canonical implementation path, so this issue should stay open until that PR lands or maintainers reject the SDK change.

## What This Changes

Request a first-class `registerCommand` result, such as `suppressReply` or `handled`, so Telegram plugin commands that send out of band do not also receive the empty-response fallback.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/80928 with public SDK docs, focused Telegram coverage, and the attached Telegram proof, then close this issue after that decision.

## Reproduction Assessment

Yes, source-reproducible. Current main turns an undefined command result into `{}`, Telegram treats that as non-renderable, and existing Telegram coverage asserts the empty-response fallback; I did not run a live Telegram/Windows repro in this read-only review.

## Solution Assessment

Mostly yes. An explicit `suppressReply` command result is a narrow way to distinguish plugin-owned delivery from an accidental empty result, but maintainers still need to approve the public SDK scope and land the canonical PR.
