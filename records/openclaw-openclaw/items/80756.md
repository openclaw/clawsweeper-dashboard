## Summary

Keep open: current main still has no registerCommand return contract for intentional no-send behavior, and Telegram native commands still turn an empty handler result into the “No response generated” fallback. This is a real, narrow SDK/channel gap, but it needs maintainer API judgment before an automated fix because it adds a new public plugin-command result semantic.

## What This Changes

The issue requests a first-class registerCommand result such as `suppressReply` or `handled` so a plugin that sends its own Telegram response can prevent OpenClaw from sending a fallback message.

## Best Possible Solution

Add an explicit, documented, backwards-compatible plugin-command no-send result in the SDK contract and teach Telegram native command delivery to treat it as terminal without sending the fallback.

## Reproduction Assessment

Yes, source-reproducible: a Telegram native plugin command returning `undefined` becomes `{}` and current test coverage expects the fallback reply. I did not run a live Telegram bot flow in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public API shape; the narrow implementation should live in the plugin command result contract plus Telegram native delivery handling, not in plugin-specific transport workarounds.
