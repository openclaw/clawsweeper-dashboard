## Summary

Current main and the latest release both contain the generic embedded-plugin tool materialization path that the issue asks for: Active Memory forwards the memory tool allowlist into the hidden run, opts into gateway subagent binding, and plugin tool resolution now reuses or cold-loads matching registry entries instead of depending on the provider path.

## What This Changes

The issue reports that Active Memory embedded recall runs can resolve a non-Codex provider model but fail to expose allowlisted memory plugin tools in the hidden runner.

## Best Possible Solution

Keep the shipped generic embedded plugin-tool materialization path and close this report as fixed by v2026.6.6.

## Reproduction Assessment

Yes. The issue discussion gives concrete failed logs, and source inspection traces the reported failure to Active Memory's embedded recall path and the plugin-tool registry materialization path now covered by current/release code.

## Solution Assessment

Yes. The implemented generic fix is the right layer: Active Memory forwards the runtime memory tool allowlist, and plugin resolution reuses or cold-loads matching gateway-bindable plugin tool registries instead of adding provider-specific config or DeepSeek-only handling.
