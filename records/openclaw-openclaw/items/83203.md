## Summary

Keep open: the underlying Gateway streaming error-path bug is still source-reproducible on current main, but this PR is stale against the current usage/tool-call finalizer and lacks after-fix real Gateway SSE proof.

## What This Changes

The PR splits Gateway OpenAI-compatible streaming content chunks from terminal stop chunks and updates focused tests for normal and error streaming.

## Best Possible Solution

Rebase and narrow the fix to current main's finalizer: keep nullable content chunks, emit the generic error text separately from the terminal finish chunk, and preserve usage and tool-call streaming behavior.

## Reproduction Assessment

Yes, source-level. Current main still combines generic streaming error content with a stop-marked choice, and the current test suite still expects that combined shape; I did not run a live Gateway request in this read-only review.

## Solution Assessment

No, not as written. The useful fix should be applied on top of current main's `maybeFinalize()` flow instead of reintroducing a direct lifecycle close path that drops documented usage and tool-call behavior.
