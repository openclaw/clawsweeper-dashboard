## Summary

Keep open: current main still only supports provider request headers at the global model-provider level, and https://github.com/openclaw/openclaw/pull/79990 is an open same-author implementation PR with closing syntax for this issue.

## What This Changes

Request provider request headers to be configurable per agent and per cron-job payload with global-to-agent-to-job precedence.

## Best Possible Solution

Review and, if acceptable, land https://github.com/openclaw/openclaw/pull/79990 or a narrower equivalent with aligned config, protocol, docs, tests, and header policy, then close this issue after merge.

## Reproduction Assessment

Not applicable: this is a feature/config request rather than a broken existing behavior report. Source and docs inspection confirms current main lacks the requested agent-level and cron payload provider-header fields.

## Solution Assessment

Unclear pending maintainer review: typed provider-request override plumbing is the right area, but the final scope should deliberately decide whether per-agent overrides are headers-only or full request overrides while cron remains headers-only with validation.

## Security Review

Status: needs_attention

Summary: Needs attention: the request introduces secret-bearing provider headers into per-agent config and mutable cron-job payloads.

Concerns:

- **[medium] Define cron header validation and redaction policy:** `src/cron/types.ts:152`
  - body: Job-scoped provider headers would be persisted mutable cron payload data; arbitrary names can include auth-like values such as Authorization or Cookie, so an implementation should keep cron overrides headers-only and add maintainer-approved validation/redaction behavior rather than exposing auth, proxy, TLS, or private-network policy through job payloads.
