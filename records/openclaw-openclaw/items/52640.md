## Summary

Keep open. Current main has adjacent task-ledger, Discord typing/preview, channel-health, and Web UI status-chip pieces, but it still lacks the requested persistent channel-turn status surface with one editable authoritative object and explicit done, failed, unknown, or interrupted ownership semantics.

## What This Changes

The issue requests a persistent, session/thread-scoped status surface for long-running channel turns with explicit terminal and interrupted/unknown states.

## Best Possible Solution

Define a generic best-effort run-status transport and ownership model, then ship a Discord-first implementation that reuses existing task/session/run facts without letting status send/edit failures block normal replies or queue finalization.

## Reproduction Assessment

Not applicable: this is a feature request, not a current-main bug with a failing reproduction path. Source inspection verifies the requested persistent channel status surface is still absent.

## Solution Assessment

Unclear until maintainers choose the boundary. A Discord-first status transport is plausible, but the maintainable path is a generic best-effort channel/runtime contract rather than ad hoc Discord-only message edits.
