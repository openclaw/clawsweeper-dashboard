## Summary

Keep open. Current main has adjacent transient visibility mechanisms, but I did not find the requested durable per-turn channel status surface with explicit terminal and unknown/interrupted semantics; related PR #59719 is useful adjacent task-liveness work, not a full substitute for this feature.

## What This Changes

Add a first-class persistent, editable task-status surface for long-running channel turns, starting with Discord, with terminal and unknown/interrupted states and best-effort send/edit behavior.

## Best Possible Solution

Design an additive, optional runtime/channel status transport that can create, update, finalize, and best-effort clear one Discord status message per long-running normal turn, backed by enough run ownership state to degrade to unknown or interrupted after restart, disconnect, cancellation, or uncertain continuity.

## Reproduction Assessment

Not applicable as a feature request. The gap is reproducible by inspection: current main has task records, Discord preview edits, reactions, and busy counters, but no persisted editable channel-turn status object.

## Solution Assessment

Unclear until maintainers define the contract. The requested Discord-first editable status message is plausible, but the maintainable path should be an additive channel lifecycle/status transport rather than overloading task records or draft reply preview.
