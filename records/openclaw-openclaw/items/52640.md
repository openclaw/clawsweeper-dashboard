## Summary

Keep open. Current main has progress drafts, task records, status reactions, and internal busy snapshots, but it still lacks the requested persistent per-channel-turn status surface with explicit done, failed, unknown, and interrupted semantics; related CLI task, Web UI, watchdog, heartbeat, and typing items do not supersede this request.

## What This Changes

Request a first-class, best-effort editable status surface for long-running channel turns, starting with Discord and explicit terminal or uncertain states.

## Best Possible Solution

Define an additive runtime/channel status transport that can create, update, finalize, and best-effort clear one Discord status message per long-running turn, backed by enough ownership state to degrade to unknown or interrupted after restart or uncertain continuity.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main has adjacent progress-draft, task-ledger, reaction, and run-state plumbing, but not the requested persistent per-turn channel status surface.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but maintainers still need to decide how an editable status message relates to progress drafts, task records, status reactions, Web UI status, restart recovery, and channel/plugin SDK contracts.
