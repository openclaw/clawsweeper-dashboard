## Summary

Keep open. Current main and the latest release have adjacent progress drafts, task-ledger terminal/lost states, status reactions, internal run-state accounting, and Web UI run-status chips, but they still do not provide the requested persistent channel-turn status surface with explicit done, failed, unknown, and interrupted ownership semantics.

## What This Changes

The issue requests one persistent, editable, per-run task-status surface for long-running channel turns, Discord first, with best-effort transport and explicit terminal or unknown/interrupted states.

## Best Possible Solution

Design an additive runtime/channel status contract that can create, update, finalize, and best-effort clear one Discord-first status surface per long-running turn, backed by enough ownership state to fail closed after restart, disconnect, or uncertain continuity.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main has adjacent progress, task, reaction, run-state, and Web UI status surfaces, but not the requested persistent channel-turn status contract.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but the best implementation needs an explicit decision about editable status messages versus progress drafts, task records, status reactions, Web UI state, restart recovery, and channel/plugin SDK contracts.
