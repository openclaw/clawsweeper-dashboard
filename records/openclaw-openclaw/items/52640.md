## Summary

Keep open because current main has adjacent progress drafts, task records, status reactions, and run-state accounting, but still does not provide the requested durable editable channel-turn status surface with explicit unknown/interrupted terminal semantics.

## What This Changes

Request to add one persistent, editable, best-effort status surface for long-running channel turns, Discord first, with explicit active and terminal states.

## Best Possible Solution

Define an additive runtime/channel status contract that can create, update, finalize, and best-effort clear one Discord status surface per long-running turn, backed by enough ownership state to fail closed to unknown or interrupted after restart or uncertain continuity.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main has adjacent progress-draft, task-ledger, status-reaction, and run-state plumbing, but not the requested persistent per-turn channel status surface.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but maintainers still need to decide how an editable status message should relate to progress drafts, task records, status reactions, Web UI status, and restart recovery semantics.
