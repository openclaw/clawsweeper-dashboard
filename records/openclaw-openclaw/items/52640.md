## Summary

Keep open. Current main has progress drafts, task records, status reactions, and internal run-state accounting, but it still lacks the requested persistent per-channel-turn status surface with explicit terminal and unknown/interrupted semantics.

## What This Changes

The issue requests a Discord-first persistent status surface for long-running channel turns, with best-effort updates and explicit terminal or unknown states.

## Best Possible Solution

Define an additive runtime/channel status contract that can create, update, finalize, and best-effort clear one Discord status surface per long-running turn, backed by enough ownership state to degrade to unknown or interrupted after restart or uncertain continuity.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main has adjacent progress-draft, task-ledger, status-reaction, and run-state plumbing, but not the requested persistent per-turn channel status surface.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but maintainers still need to choose how an editable status message should relate to progress drafts, task records, status reactions, Web UI status, and restart recovery semantics.
