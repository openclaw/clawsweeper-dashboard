## Summary

Keep open: current main has adjacent progress drafts, Web UI run chips, task-ledger terminal states, and channel busy snapshots, but it still lacks the requested persistent per-channel-turn status surface with explicit done, failed, unknown, or interrupted semantics.

## What This Changes

Request to add a persistent, authoritative task-status surface for long-running channel turns, with Discord first, one editable status object, terminal states, restart-safe unknown/interrupted handling, and best-effort transport failures.

## Best Possible Solution

Define one additive runtime-owned channel-turn status contract that reuses existing progress/session/task signals, persists only the ownership needed to fail closed, and projects first to Discord as best-effort opt-in transport.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request, and source/docs inspection verifies current main lacks the requested persistent channel-turn status contract.

## Solution Assessment

Unclear as a concrete implementation plan: the one-message status model is plausible, but maintainers need to choose the owner, persistence model, restart semantics, and rollout behavior before code work is safe.
