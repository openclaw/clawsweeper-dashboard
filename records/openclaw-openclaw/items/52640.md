## Summary

Keep open. Current main has adjacent task-ledger states, Discord draft-preview edits, status reactions, and internal busy accounting, but it still does not provide the requested single persistent editable status surface for normal long-running channel turns with explicit done, failed, unknown, or interrupted semantics.

## What This Changes

The issue requests a first-class persistent, best-effort task-status surface for long-running channel turns, with Discord as the first rollout target and explicit terminal or uncertain states.

## Best Possible Solution

Design an additive optional channel status transport that can create, update, finalize, and best-effort clear one Discord status message per normal long-running turn, backed by enough run ownership state to fail closed to unknown or interrupted after restart, disconnect, cancellation, or uncertain continuity. Keep task records for detached/background work and draft preview for reply streaming rather than overloading either as the authoritative per-turn status object.

## Reproduction Assessment

Not applicable as a feature request. The gap is high-confidence by inspection: current main exposes task records for detached work, draft preview streaming, reactions, and internal busy patches, but no persisted editable channel-turn status object.

## Solution Assessment

Unclear until maintainers choose the contract. A Discord-first editable status message is plausible, but the maintainable path should be an additive channel/runtime transport with explicit best-effort semantics rather than coupling terminal status to draft reply preview or the detached task ledger.
