## Summary

Keep open. Current main and v2026.5.22 have adjacent progress drafts, task records, status reactions, typing keepalives, and coarse UI/session run status, but they still do not provide the requested persistent channel-turn status surface with explicit done, failed, unknown, and interrupted ownership semantics; the open related PRs at https://github.com/openclaw/openclaw/pull/59719 and https://github.com/openclaw/openclaw/pull/72771 are adjacent and not merged superseding fixes.

## What This Changes

Request a persistent, editable task-status surface for long-running channel turns, Discord-first, with explicit active, terminal, and unknown/interrupted states.

## Best Possible Solution

Design an additive runtime/channel status contract that can create, update, finalize, and best-effort clear one Discord-first status surface per long-running turn, backed by enough ownership state to fail closed after restart, disconnect, or uncertain continuity.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main and v2026.5.22 have adjacent progress, task, reaction, run-state, and UI status surfaces, but not the requested persistent channel-turn status contract.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but the best implementation needs an explicit decision about editable status messages versus progress drafts, task records, status reactions, Web UI state, restart recovery, and channel/plugin SDK contracts.
