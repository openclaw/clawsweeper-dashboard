## Summary

Keep open: current main has adjacent progress drafts, task-ledger terminal/lost states, session lifecycle rows, channel busy patches, and Discord reactions, but it still lacks the requested durable per-channel-turn status object with explicit unknown/interrupted terminal semantics. This remains a product/API design item rather than a conservative cleanup close.

## What This Changes

The issue requests a persistent, editable, session-scoped task status surface for long-running channel turns with explicit terminal and unknown/interrupted states.

## Best Possible Solution

Define an additive runtime-owned channel-turn status contract that reuses existing progress/session/task signals where appropriate, persists only the ownership needed to fail closed to unknown/interrupted, and projects it to Discord as a best-effort opt-in transport first.

## Reproduction Assessment

Not applicable: this is a feature request for a new persistent status surface, not a broken existing-behavior report. Source and docs show adjacent partial surfaces but no current-main implementation of the requested durable channel-turn status contract.

## Solution Assessment

Unclear as an implementation plan: the one-message status model is plausible, but maintainers need to choose the canonical owner and rollout semantics. A Discord-only patch would be too narrow unless it is backed by a runtime/channel contract that avoids status drift.
