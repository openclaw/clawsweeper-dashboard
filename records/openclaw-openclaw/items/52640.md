## Summary

Keep open. Current main has adjacent progress drafts, status reactions, task records, and Web UI run chips, but it still does not provide the requested persistent per-channel-turn status surface with explicit done, failed, unknown, and interrupted semantics.

## What This Changes

The issue requests one persistent, editable status surface for long-running channel turns, with terminal and unknown/interrupted states and best-effort send/edit failure handling.

## Best Possible Solution

Design an additive runtime/channel status transport that can create, update, finalize, and best-effort clear one Discord-first status message per long-running turn, backed by enough ownership state to fail closed to unknown or interrupted after restart or uncertain continuity.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main has adjacent progress, reaction, task, and Web UI status surfaces, but not the requested persistent channel-turn status contract.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but the best implementation needs an explicit decision about how editable status messages relate to progress drafts, status reactions, task records, Web UI state, restart recovery, and channel/plugin SDK contracts.
