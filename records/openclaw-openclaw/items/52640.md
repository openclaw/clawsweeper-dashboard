## Summary

Keep open. Current main has adjacent progress drafts, task-ledger state, channel busy accounting, and a Control UI run-status chip, but it still does not provide the requested persistent channel status surface with explicit done, failed, unknown, or interrupted semantics for normal long-running turns.

## What This Changes

The issue requests a first-class persistent editable status surface for long-running channel turns, starting with Discord, with terminal/unknown states and best-effort send/edit failures.

## Best Possible Solution

Design an additive runtime/channel status transport that can create, update, finalize, and best-effort clear one Discord status message for a normal long-running turn, backed by enough session/run ownership state to fail closed after restart, disconnect, or uncertain continuity.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main has adjacent drafts, task records, and UI status, but not the requested durable channel-turn status surface.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but maintainers need to choose the contract between channel messages, progress drafts, task records, session activity, persistence, and optional rollout before implementation.

## Risks / Open Questions

- The remaining design overlaps with progress drafts, task records, session activity, Control UI status, and watchdog proposals, so implementing one surface without a maintainer-owned contract could create duplicate or conflicting status sources.
- Restart/disconnect behavior and unknown/interrupted semantics still need a clear persistence and ownership model before a channel message can safely claim authoritative status.
- Default-on status messages could add channel noise or misleading terminal states unless rollout, config, delay, and failure handling are decided together.
