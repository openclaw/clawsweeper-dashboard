## Summary

Keep open. Current main has adjacent busy-state accounting, Discord draft preview edits with tool progress, and task-ledger terminal/lost states, but not the requested single persistent editable status surface for normal long-running channel turns with explicit done, failed, unknown, or interrupted semantics; related #59719 is adjacent background CLI/task liveness work rather than a replacement for the channel-turn UX.

## What This Changes

Request a persistent, session/thread-scoped editable task-status surface for long-running channel turns, Discord first, with delayed display, milestone updates, explicit terminal/unknown states, and best-effort non-blocking send/edit behavior.

## Best Possible Solution

Design an additive optional channel status transport that can create, update, finalize, and best-effort clear one Discord status message per long-running normal turn, backed by enough run ownership state to fail closed to unknown or interrupted after restart, disconnect, cancellation, or uncertain continuity. Keep task records for detached/background work and draft preview for reply streaming rather than overloading either as the authoritative per-turn status object.

## Reproduction Assessment

Not applicable as a feature request. The gap is high-confidence by inspection: current main exposes task records for detached work, draft preview streaming, reactions, and internal busy patches, but no persisted editable channel-turn status object.

## Solution Assessment

Unclear until maintainers choose the contract. A Discord-first editable status message is plausible, but the maintainable path should be an additive channel/runtime transport with explicit best-effort semantics rather than coupling terminal status to draft reply preview or the detached task ledger.
