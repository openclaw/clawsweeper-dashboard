## Summary

Keep open. Current main and v2026.5.20 have adjacent progress drafts, task-ledger states, internal run busy snapshots, and a Web UI status chip, but they still do not provide the requested persistent per-channel-turn status surface with explicit done, failed, unknown, and interrupted semantics.

## What This Changes

Request a persistent, editable per-channel-turn task-status surface for long-running chat turns, Discord-first, with terminal and unknown/interrupted states.

## Best Possible Solution

Design an additive runtime/channel status transport that can create, update, finalize, and best-effort clear one Discord-first status message per long-running turn, backed by enough ownership state to fail closed to unknown or interrupted after restart or uncertain continuity.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main has adjacent progress, task, reaction, run-state, and Web UI status surfaces, but not the requested persistent channel-turn status contract.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but the best implementation needs an explicit decision about how editable status messages relate to progress drafts, task records, status reactions, Web UI state, restart recovery, and channel/plugin SDK contracts.
