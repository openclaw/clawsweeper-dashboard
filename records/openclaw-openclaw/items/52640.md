## Summary

Keep open. Current main has substantial adjacent progress-draft, task-ledger, reaction, and busy-state plumbing, but it still does not provide the requested single persistent editable status surface for normal channel turns with explicit done, failed, unknown, or interrupted semantics and restart ownership.

## What This Changes

The issue requests a Discord-first persistent task-status surface for long-running channel turns, with one editable status message, explicit terminal/unknown states, and best-effort transport failures.

## Best Possible Solution

Design an additive runtime/channel status transport that can create, update, finalize, and best-effort clear one Discord status message for a normal long-running turn, backed by enough ownership state to fail closed to unknown or interrupted after restart, disconnect, or uncertain continuity.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main has editable progress drafts and internal busy/task state, but not the requested durable terminal/unknown status surface for normal channel turns.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but maintainers still need to choose whether terminal/unknown state belongs in a separate channel message, task/run record, status reaction layer, session activity API, or some combination of those surfaces.
