## Summary

Keep open. Current main and v2026.5.22 have adjacent progress drafts, task-ledger terminal/lost states, status reactions, and internal busy snapshots, but they still do not provide the requested authoritative per-channel-turn status surface with persistent done, failed, unknown, and interrupted semantics.

## What This Changes

The issue requests a persistent, session/thread-scoped task-status surface for long-running channel turns, with Discord-first editable status messages, explicit terminal or unknown states, and best-effort non-blocking delivery.

## Best Possible Solution

Design an additive runtime/channel status contract that can create, update, finalize, and best-effort clear one Discord-first status surface per long-running turn, backed by enough ownership state to fail closed to unknown or interrupted after restart or uncertain continuity.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: current main has adjacent progress, task, reaction, run-state, and Web UI status surfaces, but not the requested persistent channel-turn status contract.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but the best implementation needs an explicit decision about how editable status messages relate to progress drafts, task records, status reactions, Web UI state, restart recovery, and channel/plugin SDK contracts.
