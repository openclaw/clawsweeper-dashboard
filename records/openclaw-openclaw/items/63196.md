## Summary

Keep open. Current main still skips durable cron recovery for terminal `lost` rows, so a cron task can remain audited as `lost` even when durable cron run history proves the run finished successfully; open PR #75036 is the active fix candidate but has not merged.

## What This Changes

The issue asks for task audit and maintenance to reconcile terminal lost cron task rows when durable cron run history reports the same logical run as finished successfully.

## Best Possible Solution

Review and land or replace #75036 with a narrow reconciliation path that recovers only backing-session false-positive terminal lost cron rows when durable cron state proves the exact execution reached a terminal outcome.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `runtime: "cron"`, `status: "lost"` task whose `runId` matches a finished durable cron run; current main skips recovery because the task is not active, then audit still reports it as lost.

## Solution Assessment

Yes. The best fix direction is the narrow #75036-style maintenance change, constrained to exact durable cron execution identity and backing-session false positives rather than broad job-id-only inference.
