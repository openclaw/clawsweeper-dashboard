## Summary

Keep open. Current main and the latest release still skip durable cron recovery for already-terminal `lost` cron rows, so the reported audit/maintenance disagreement remains; the focused fix path is the open PR at https://github.com/openclaw/openclaw/pull/75036.

## What This Changes

The issue asks task maintenance/audit to reconcile terminal `lost` cron rows when durable cron run history shows the same run finished successfully.

## Best Possible Solution

Review, validate, and land or replace https://github.com/openclaw/openclaw/pull/75036 so terminal `lost` cron rows are recovered only from exact durable cron evidence.

## Reproduction Assessment

Yes, source-level. A `runtime: "cron"`, `status: "lost"` row with a parseable cron run id is skipped by durable recovery because it is not active, then audit reports it as `lost`.

## Solution Assessment

Yes. The narrow maintainable fix is to extend durable cron recovery to terminal `lost` rows only when exact run history or job state proves the same execution reached a terminal result.
