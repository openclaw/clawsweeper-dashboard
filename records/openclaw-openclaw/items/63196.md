## Summary

Keep open. Current main still only applies durable cron recovery to active cron tasks, so a terminal `lost` cron row with matching finished cron run history can remain reported by task audit; open PR #75036 is the active fix candidate but is not merged into this checkout.

## What This Changes

The issue requests reconciliation between cron run history and task audit so finished cron runs do not remain reported as lost backing-session-missing task rows.

## Best Possible Solution

Ship a narrow maintenance/audit reconciliation path that recovers terminal `lost` cron rows only when the row has a parseable `cron:<jobId>:<startedAt>` execution id and durable run log or job state proves that exact execution reached a terminal outcome.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `runtime: "cron"`, `status: "lost"` task with a parseable cron execution run id and matching finished durable run-log entry; current main skips durable recovery because the row is not active, then audit still reports it as `lost`.

## Solution Assessment

Yes. The best fix direction is the narrow one already represented by PR #75036: recover terminal lost cron rows from exact durable cron execution evidence, rather than inferring success from job id alone or broadly suppressing lost audit findings.
