## Summary

Keep open. Current main still limits durable cron recovery to active task rows, so the reported terminal `lost` cron row can continue to audit as `backing session missing`; open PR #75036 targets this exact gap but is still open and unmerged.

## What This Changes

The issue asks task audit and `openclaw tasks maintenance --apply` to reconcile a terminal `lost` cron task when durable cron run history proves the same execution finished successfully.

## Best Possible Solution

Ship PR #75036 or an equivalent narrow maintenance change that recovers terminal `lost` cron rows only when their `cron:<jobId>:<startedAt>` execution id matches durable run-log or job-state proof of a terminal outcome, while leaving unrelated lost rows untouched.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `runtime: "cron"`, `status: "lost"`, `error: "backing session missing"` task with matching durable finished run history; current main skips recovery because the row is not active, then audit still reports it as lost.

## Solution Assessment

Yes, the proposed fix direction is the best narrow path: reuse exact durable cron execution identity and constrain terminal recovery to the backing-session false-positive case. Broad pruning or job-id-only inference would risk hiding genuinely lost cron work.
