## Summary

Keep open. Current main still gates durable cron recovery on active cron tasks, so a terminal `lost` cron row with matching finished cron history can remain auditable as `lost`; #75036 is the active fix candidate but is not merged.

## What This Changes

The issue asks task audit and maintenance to reconcile terminal `lost` cron task rows when exact cron run history proves the matching execution finished successfully.

## Best Possible Solution

Ship a narrow maintenance/audit reconciliation path that recovers terminal `lost` cron rows only when the row has a parseable `cron:<jobId>:<startedAt>` execution id and durable run log or job state proves that exact execution reached a terminal outcome.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `runtime: "cron"`, `status: "lost"` task with matching durable finished run history; current main skips durable recovery because the row is not active, then audit still reports it as `lost`.

## Solution Assessment

Yes. The linked fix direction is the narrow maintainable solution if it is constrained to backing-session false positives and exact execution identity; broad job-id-only inference would be riskier.
