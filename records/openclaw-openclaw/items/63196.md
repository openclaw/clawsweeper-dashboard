## What This Changes

The issue reports that cron run history can show a job finished successfully while task audit still reports the same logical cron task as `lost` with `backing session missing`.

## Best Possible Solution

Review, update, and land or replace https://github.com/openclaw/openclaw/pull/75036 so terminal `lost` cron rows recover only from exact durable cron evidence, then close this issue after the fix reaches `main`.

## Reproduction Assessment

Yes. Source inspection shows a `runtime: "cron"`, `status: "lost"` row is skipped by durable cron recovery because it is not active, then audit still reports it as `lost`.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing durable cron recovery path to terminal `lost` rows only when exact run history or job state proves the same execution reached a terminal result.
