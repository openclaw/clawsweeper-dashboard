## Summary

Keep open. Current main and the latest shipped release still exclude terminal `lost` cron rows from durable cron recovery, so a row with `status: "lost"` and `error: "backing session missing"` can continue to appear in audit even when durable cron run history has the matching successful finish; the active fix candidate is still open at https://github.com/openclaw/openclaw/pull/75036.

## What This Changes

Reconcile terminal cron task rows marked `lost`/`backing session missing` with matching successful cron run history so audit/status and maintenance agree.

## Best Possible Solution

Review, validate, and land or replace the open terminal-lost cron recovery PR so maintenance can recover only exact backing-session false positives from durable cron run history.

## Reproduction Assessment

Yes, source-level. A `runtime: "cron"`, `status: "lost"` task with a parseable `cron:<jobId>:<startedAt>` run id is skipped by durable recovery because it is not active, then audit reports it as `lost`.

## Solution Assessment

Yes. The narrow maintainable fix is to recover terminal `lost` cron rows only when exact durable run history or job state proves that same execution reached a terminal outcome; broad job-id-only inference would be riskier.
