## Summary

Keep open. Current main still only applies durable cron run-log/job-state recovery to active cron task rows, so a terminal `lost` cron row can still audit as `backing session missing`; linked open PR #75036 targets this gap but has not merged.

## What This Changes

The issue requests that task audit and task maintenance reconcile or prune stale terminal `lost` cron task rows when durable cron run history shows the corresponding job run finished successfully.

## Best Possible Solution

Ship a narrow task-maintenance reconciliation path that can recover terminal `lost` cron rows only when the row has a parseable `cron:<jobId>:<startedAt>` execution id and durable run log or job state proves that exact execution reached a terminal outcome.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `runtime: "cron"`, `status: "lost"` task with matching durable finished run history; current main skips durable recovery because the row is not active, then audit still reports it as `lost`.

## Solution Assessment

Yes. The best fix direction is to extend durable cron recovery to terminal `lost` rows constrained to backing-session false positives and exact execution identity; broad job-id-only inference would be less maintainable.
