## Summary

Keep open: current main does not include the requested cross-tool consecutive-error detector, and #75924 is still an open, unmerged PR that explicitly closes this issue.

## What This Changes

Add an opt-in run-scoped `consecutive_errors` detector that blocks the next tool call after a configurable streak of completed error outcomes across different tools.

## Best Possible Solution

Review #75924 as the implementation candidate, making sure the detector, schema, docs, and run-scoped tests stay narrow and do not drift into the broader turn-cap or retry-budget proposals.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: enable loop detection, record different failed tool outcomes under the same `runId`, then call another tool. Current main has no detector that counts that cross-tool error streak.

## Solution Assessment

Unclear until maintainers review #75924. The requested detector is narrower than session caps or retry budgets, but the exact threshold, default, and blocking semantics are product decisions.
