## Summary

Keep open. Current main still lacks the requested run-scoped cross-tool consecutive-error detector/config surface, and the issue is paired with open closing PR #75924, so cleanup should wait for that PR to merge or be closed.

## What This Changes

Add a run-scoped loop detector/config that blocks the next tool call after a threshold of same-run completed tool outcomes whose `resultHash` encodes an error, even across different tools.

## Best Possible Solution

Review and land or close #75924; the accepted change should extend the existing run-scoped loop detector with detector logic, schema/help/docs updates, and same-run plus cross-run isolation coverage.

## Reproduction Assessment

Yes, from source inspection. With loop detection enabled, same-run completed outcomes for distinct tools whose `resultHash` starts with `error:` are recorded, but current `detectToolCallLoop` has no detector that blocks that cross-tool failed tail.

## Solution Assessment

Yes at the issue direction level. Extending the existing run-scoped loop-detection path is narrower than a session cap or retry-budget feature, while #75924 should carry the concrete review for config shape, defaults, and edge cases.
