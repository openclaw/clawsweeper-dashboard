## Summary

Keep open: current main still lacks the requested cross-tool consecutive-error detector, and this issue is already paired with open implementation PR #75924, so closing before that PR is resolved would be premature.

## What This Changes

The issue requests a configurable run-scoped detector that blocks the next tool call after enough consecutive completed tool-call errors across different tools in the same run.

## Best Possible Solution

Review and either land or close #75924; if accepted, the fix belongs in the existing tool-loop detector with aligned config schema, docs, and regression coverage.

## Reproduction Assessment

Yes, the missing guard is source-reproducible: current main records run-scoped tool outcomes, including error hashes, but only evaluates same-pattern, polling, ping-pong, and unknown-tool loops. I did not run tests because this was a read-only review.

## Solution Assessment

Yes, the linked implementation direction is the narrow maintainable path: extend the existing run-scoped tool-loop detector rather than adding a broader turn cap or session retry budget.
