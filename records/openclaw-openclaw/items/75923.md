## Summary

Keep open. Current main has run-scoped loop detection for repeated, polling, ping-pong, unknown-tool, and no-progress patterns, but it does not have the requested `consecutive_errors` detector or `consecutiveErrorThreshold` config, and the issue is paired with open closing PR #75924 from the same author.

## What This Changes

Add a configurable run-scoped detector that blocks the next tool call after enough consecutive completed tool outcomes in the same run are errors, even when the tools differ.

## Best Possible Solution

Review #75924 as the concrete implementation path, and if it is correct, land the narrow detector with schema, docs, and focused unit plus hook-path coverage.

## Reproduction Assessment

Yes. A focused static reproduction is clear: record same-run completed outcomes for different tool names where each outcome has an `error:` result hash, then ask `detectToolCallLoop` about another distinct tool; current main has no cross-tool tail-error detector to block that sequence.

## Solution Assessment

Yes at the issue level. Extending the existing run-scoped loop detector and before-tool-call hook is the narrow maintainable boundary; the open PR still needs normal review for exact defaults and edge cases.
