## Summary

Keep open. Current main still lacks the requested cross-tool `consecutive_errors` detector and threshold config, and the issue is paired with open closing PR #75924, so cleanup should wait for that PR to merge or be closed.

## What This Changes

The issue requests a run-scoped `consecutive_errors` loop detector that blocks the next tool call after a configurable threshold of consecutive failed tool outcomes across different tools.

## Best Possible Solution

Review and land or close #75924; the accepted fix should extend the existing run-scoped loop detector and before-tool-call hook with schema, docs, and focused unit plus hook-path coverage.

## Reproduction Assessment

Yes. Seed same-run completed tool-call history with distinct tools whose outcomes hash as `error:<digest>`, then call `detectToolCallLoop` for another distinct tool; current main has no detector that blocks that cross-tool error tail.

## Solution Assessment

Yes at the issue level. Extending the existing run-scoped loop detector is the narrow maintainable boundary, while broader turn caps or retry budgets belong in their separate threads.
