## Summary

Keep open: current main and the latest release still lack the requested run-scoped detector for consecutive failed outcomes across different tools. Existing loop detection covers same-tool repeats, known polling loops, ping-pong alternation, unknown-tool repeats, and post-compaction repeats, but not the described all-failing cross-tool pivot pattern; the remaining decision is whether maintainers want a dedicated threshold or to fold this into broader native failure-recovery work.

## What This Changes

Request a run-scoped loop detector that blocks the next tool call after a configurable streak of completed tool failures across different tools.

## Best Possible Solution

Decide the product shape first, then either add a small run-scoped tail-streak detector under the existing `tools.loopDetection` owner path or explicitly make the broader native failure-recovery issue own this behavior.

## Reproduction Assessment

Yes, at source level: current loop detection records run-scoped tool outcomes but only evaluates same-tool, unknown-tool, polling, ping-pong, and post-compaction patterns. I did not run a live failing agent loop during this read-only review.

## Solution Assessment

Unclear: a narrow detector is plausible, but it is not proven to be the best product shape until maintainers choose between a new dedicated threshold, reusing existing critical thresholds, or folding the behavior into broader native failure recovery.
