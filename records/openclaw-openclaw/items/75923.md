## Summary

Keep open. Current main still lacks the requested cross-tool `consecutive_errors` detector and `consecutiveErrorThreshold` config, and the issue is paired with open closing PR #75924, so cleanup should wait for that PR to be reviewed and merged or closed.

## What This Changes

Add a run-scoped `consecutive_errors` loop detector that blocks the next tool call after a configurable same-run streak of completed tool outcomes whose result hashes encode errors.

## Best Possible Solution

Review and land or close #75924; the accepted fix should live in the existing run-scoped loop detector and before-tool-call hook with schema, docs, and focused unit plus hook-path coverage.

## Reproduction Assessment

Yes. Record same-run completed outcomes for distinct tools with `error:` result hashes, then call `detectToolCallLoop` for another distinct tool; current main has no detector that blocks that cross-tool error tail.

## Solution Assessment

Yes at the issue level. Extending the existing run-scoped loop detector is the narrow maintainable boundary; the open PR still needs review for exact defaults and edge cases.
