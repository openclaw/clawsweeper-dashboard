## Summary

Keep open: current main has run-scoped tool-loop history and several repeat/no-progress detectors, but it does not implement the requested cross-tool `consecutive_errors` detector. An open same-author PR, #75924, already uses closing syntax for this issue, so the issue should stay open until that PR is reviewed and either landed or closed.

## What This Changes

Add a run-scoped `consecutive_errors` loop detector plus `tools.loopDetection.consecutiveErrorThreshold` config to block the next tool call after a streak of failed outcomes across different tools.

## Best Possible Solution

Review #75924 as the active implementation path; if accepted, keep the change inside the existing tool-loop detector, config schema/help/docs, and focused detector plus hook-path tests while preserving runId scoping.

## Reproduction Assessment

Yes, from source inspection: current main records run-scoped error outcomes but only detects same-pattern, unknown-tool, polling, global same-pattern, and ping-pong loops. A tail of distinct failed tools has no current detector to trip.
