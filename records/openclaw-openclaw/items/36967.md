## Summary

Keep open: the PR targets a real Feishu parity gap, but the branch is stale against current main's plugin-owned allowlist boundary, can shadow inherited Feishu allowlist entries, and still lacks convincing after-fix real behavior proof.

## What This Changes

The PR adds Feishu handling to allowlist commands, markdown-channel detection, envelope stripping, hook channel config metadata, security-fix hardening, tests, and a CI Bun patch-version bump.

## Best Possible Solution

Rebase onto current main, add a Feishu-owned allowlist adapter that preserves inherited top-level/account config on writes, keep the generic hook/security behavior already on main, add Feishu markdown/envelope metadata only where the plugin contract supports it, and provide redacted after-fix proof.

## Reproduction Assessment

Yes for the PR blocker at source level: PR head reads existing entries from the raw account write target while Feishu effective config merges top-level and account config. I did not run live Feishu behavior in this read-only review.

## Solution Assessment

No. The current diff hardcodes Feishu into an older core path; the maintainable fix is a Feishu plugin adapter that preserves inherited config and fits current main's channel boundary.
