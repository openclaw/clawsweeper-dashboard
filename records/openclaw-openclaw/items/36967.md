## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the central Feishu `/allowlist channel=feishu` config edit/listing support and Feishu markdown-capability metadata, while several PR chunks are now superseded by generic main-branch behavior and the branch has security-sensitive regressions around allowlist writes plus an unrelated CI runtime bump.

## What This Changes

The PR adds Feishu handling to `/allowlist`, markdown capability checks, envelope stripping, hook channel validation/help, security-fix hardening, targeted tests, and changes the Bun version in the shared setup action.

## Best Possible Solution

Rework this as a narrow Feishu/plugin-owned change: add a Feishu `allowlist` adapter through the existing plugin SDK contract that preserves effective merged account allowlists and current owner/gateway-scope write gates, set Feishu `markdownCapable` metadata in both runtime and package discovery surfaces, keep hooks/security on the current generic main-branch paths, and split out the Bun runtime change.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: `/allowlist add dm channel=feishu --config ...` reaches the current main `plugin.allowlist.applyConfigEdit` requirement, while the Feishu plugin has no `allowlist` adapter; `isMarkdownCapableMessageChannel("feishu")` also depends on metadata that Feishu does not set.

## Solution Assessment

No. The PR is useful source work, but it hardcodes Feishu behavior into shared command code, carries stale hook/security chunks, misses current owner-only write gating, and mishandles inherited Feishu account allowlists; the narrower maintainable fix belongs in the Feishu plugin adapter plus metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Restore the owner gate before allowlist writes:** `src/auto-reply/reply/commands-allowlist.ts:512`
  - body: This branch proceeds after only `rejectUnauthorizedCommand`, while current main rejects non-owner `/allowlist add/remove` before resolving the target channel. With `commands.config` and channel `configWrites` enabled, an authorized non-owner sender could edit config allowlists, so the non-owner gate needs to be restored before any write path.
  - confidence: 0.9
- **[P2] Preserve inherited Feishu allowlists on account edits:** `src/auto-reply/reply/commands-allowlist.ts:767`
  - body: For account-scoped Feishu edits, the branch reads existing entries only from the account object returned by `resolveAccountTarget`. Feishu accounts inherit top-level `allowFrom`/`groupAllowFrom`, so the first account-local edit can create a new array that overrides and drops the inherited effective entries; seed edits from the merged effective Feishu allowlist or use a Feishu-owned adapter that preserves inherited entries.
  - confidence: 0.86
