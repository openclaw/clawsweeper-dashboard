## Summary

Keep open. Feishu parity remains a valid gap, but this branch is stale against the current plugin-owned allowlist and metadata boundary, lacks after-fix real behavior proof, and still has an access-control edit bug that can shadow inherited Feishu allowlists.

## What This Changes

The branch hardcodes Feishu into core `/allowlist`, hook-channel schema/help, markdown-channel, envelope-stripping, and security-fix paths, bumps the setup Bun version, and adds targeted tests.

## Best Possible Solution

Implement Feishu parity through plugin-owned surfaces: a merged-config-safe allowlist adapter/editor, Feishu markdown metadata, an accepted envelope strategy, focused regression tests, and redacted after-fix behavior proof.

## Reproduction Assessment

Yes, at source level. In the PR head, Feishu list output reads merged config while edits read only the raw account target, so adding to an account that inherits top-level allowlists can shadow inherited entries.

## Solution Assessment

No. The requested Feishu parity is useful, but the branch hardcodes stale core/static paths instead of using the current plugin adapter and metadata boundaries.
