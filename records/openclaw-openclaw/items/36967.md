## Summary

Keep this PR open, but do not merge it as-is. Current main has moved the relevant surfaces toward plugin-owned allowlist adapters and generic security/hooks handling, while this branch still has security-sensitive allowlist regressions and stale hardcoded chunks.

## What This Changes

The PR adds Feishu handling to `/allowlist`, markdown capability checks, envelope stripping, hook channel validation/help, security-fix hardening, targeted tests, and changes the shared setup action's Bun version.

## Best Possible Solution

Rework this around a Feishu-owned allowlist adapter and Feishu metadata: preserve current owner/gateway-scope write gates, edit from the effective merged account allowlists, set markdown capability through plugin metadata, keep current generic hooks/security behavior, and split unrelated CI runtime changes.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main routes config allowlist edits through `plugin.allowlist.applyConfigEdit`, and the Feishu plugin does not expose that adapter; the markdown path also returns false unless Feishu metadata sets `markdownCapable`.

## Solution Assessment

No. The PR is useful source work, but the submitted branch is stale against current main and hardcodes shared behavior while weakening authorization and mishandling inherited account config; the safer direction is a plugin-owned adapter plus metadata-only shared integration.
