Keep this PR open but do not merge it as-is: the Feishu parity goal is useful, yet the branch is conflicting, still has a source-level Feishu allowlist integrity bug, and lacks after-fix real behavior proof.

The branch adds Feishu-specific `/allowlist` config handling, hook-channel schema/help changes, markdown/envelope/security-fix additions, targeted tests, and a Bun setup version bump.

## Best Possible Solution

Implement Feishu parity through Feishu plugin-owned allowlist, metadata, and security-fix adapters that preserve merged account config, then back it with focused regression tests and redacted after-fix behavior proof.

## Reproduction Assessment

Yes for the PR blocker at source level: PR head reads existing entries from the raw write target while current Feishu account resolution merges top-level and account config. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No. Feishu parity is useful, but hardcoding Feishu into core/static paths is not the best current-main shape; the safer path is plugin-owned adapters and metadata with inherited-config preservation.
