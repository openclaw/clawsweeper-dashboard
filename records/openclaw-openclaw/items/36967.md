## Summary

Keep open: the Feishu parity request is still useful, but this PR is not mergeable as-is because it targets the old core-hardcoded allowlist path while current main expects plugin-owned allowlist adapters, and the branch can shadow inherited Feishu allowlists during account edits.

## What This Changes

The branch hardcodes Feishu into `/allowlist` config edits/listing, static markdown/envelope/security/hook lists, and bumps the CI Bun setup version with targeted tests.

## Best Possible Solution

Implement Feishu `/allowlist` support as a Feishu plugin allowlist adapter using the existing SDK helper, add metadata-driven markdown capability if confirmed, preserve inherited top-level/account allowlists on edits, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows the account edit path reads only account-local allowlist entries before writing, and current main source shows Feishu still has no plugin allowlist adapter for `/allowlist channel=feishu`.

## Solution Assessment

No. The desired Feishu parity is valid, but hardcoding Feishu into old core/static lists is no longer the narrowest maintainable solution on current main; the safer path is a Feishu plugin adapter plus plugin metadata.
