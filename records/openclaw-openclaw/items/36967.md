This PR adds Feishu-specific `/allowlist` config handling, hook-channel schema/help changes, markdown/envelope/security-fix additions, targeted tests, and a Bun setup version bump.

## Best Possible Solution

Implement Feishu parity through Feishu plugin-owned allowlist, metadata, and security-fix adapters that preserve merged account config, then back it with focused tests and redacted after-fix behavior proof.

## Reproduction Assessment

Yes at source level for the PR blocker: PR head reads existing entries from the raw account write target while Feishu account resolution merges top-level and account config. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No. Feishu parity is useful, but adding Feishu to core static paths and writing raw account lists is not the best current-main shape; plugin-owned adapters and metadata should preserve inherited account config.
