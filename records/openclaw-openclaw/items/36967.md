## Summary

Keep open: the Feishu parity goal is useful, but current main has moved this surface to plugin-owned allowlist/security contracts and this branch still carries an access-control regression plus insufficient real behavior proof.

## What This Changes

The branch adds Feishu-specific support for `/allowlist` config edits/listing, markdown-capable detection, envelope stripping, hook channel validation/help, and security-fix hardening, plus tests and a Bun setup-action bump.

## Best Possible Solution

Rework this as Feishu plugin-owned allowlist and security-fix adapters plus metadata, preserve inherited account config during edits, keep the generalized hook schema from current main, and drop unrelated CI/runtime changes.

## Reproduction Assessment

Yes. Source inspection of the PR branch and current main shows the Feishu core write path can override inherited account allowlists, while current main requires this behavior to go through `plugin.allowlist`; live after-fix proof is still missing.

## Solution Assessment

No. The narrow maintainable solution is to implement Feishu parity through the current plugin adapter and metadata contracts, not by adding Feishu-specific core branches and an unrelated CI runtime change.
