## Summary

Keep open: the PR still points at real Feishu parity gaps, but the branch is stale and incorrect against current main because it adds channel-specific allowlist behavior in core and can drop inherited account allowlists; after-fix real behavior proof is also missing.

## What This Changes

The PR hard-codes Feishu into allowlist config edits/listing, markdown-capable detection, envelope stripping, hook channel schema/help, and security-fix hardening, with targeted tests and a shared Bun setup-action version bump.

## Best Possible Solution

Rework this against current main as a Feishu plugin change: add a plugin allowlist adapter that preserves inherited account config, expose markdown capability through Feishu metadata, add plugin-owned security-fix mutation, and drop unrelated CI/runtime changes.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current main shows `/allowlist` delegates to `plugin.allowlist`, while the PR adds Feishu to a core account-scoped write path that can omit inherited top-level allowlists for named accounts.

## Solution Assessment

No. The narrow maintainable solution is to implement the Feishu behavior through current plugin adapters and metadata, not through Feishu-specific core branches plus an unrelated CI runtime bump.
