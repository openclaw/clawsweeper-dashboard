## Summary

Keep this PR open for human review: the Feishu parity goal is still partly unresolved on current main, but the branch is conflicting, uses an older core hardcode path, has a P1 allowlist inheritance bug, and still lacks after-fix real behavior proof.

## What This Changes

The branch adds Feishu handling to `/allowlist`, markdown-capability detection, envelope stripping, hook config validation, `security audit --fix`, CI Bun setup, and targeted tests.

## Best Possible Solution

Implement Feishu `/allowlist` through the Feishu plugin's SDK adapter, expose confirmed markdown/envelope capability through plugin metadata or a shared capability seam, and keep the current dynamic hooks/security handling.

## Reproduction Assessment

Yes. Source inspection shows current main still lacks a Feishu plugin allowlist adapter, and the PR head's generic config-edit path can shadow inherited Feishu allowlists when writing account-local entries.

## Solution Assessment

No. The Feishu parity goal is valid, but adding Feishu to old core/static lists is not the narrowest maintainable fix now that current main has plugin-owned allowlist adapters and dynamic hooks/security handling.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Preserve inherited Feishu allowlists on edits:** `src/auto-reply/reply/commands-allowlist.ts:417`
