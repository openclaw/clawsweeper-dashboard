Keep open: the PR addresses real Feishu parity gaps, but current main has superseded several static-list changes and the branch still needs plugin-contract rework, security-sensitive allowlist fixes, and after-fix real behavior proof before merge.

## What This Changes

The PR adds Feishu handling to `/allowlist` config edits/listing, markdown capability checks, envelope stripping, hook channel typing/help, security audit `--fix`, tests, and a Bun setup-action version bump.

## Best Possible Solution

Implement the remaining Feishu parity through a Feishu plugin `allowlist` adapter and plugin metadata, keep current main's generic hook/security paths, add the envelope label only if still needed, and split out the Bun bump.

## Reproduction Assessment

Yes, from source inspection. Current main routes `/allowlist` through plugin allowlist adapters and Feishu does not expose one, while the PR branch's generic write path can override inherited top-level Feishu allowlists.

## Solution Assessment

No. The maintainable fix is to use the current plugin adapter and metadata contracts instead of adding Feishu-specific branches and static list entries to shared core code.
