## Summary

Keep open. The PR still points at real Feishu parity gaps, but the branch is not mergeable as-is because it adds Feishu-specific core allowlist logic that can change access control incorrectly and does not provide after-fix real behavior proof.

## What This Changes

The branch adds Feishu-specific `/allowlist` config editing/listing, markdown detection, envelope stripping, hook channel literals, security-fix hardening, tests, and a Bun setup-action version bump.

## Best Possible Solution

Rework this through Feishu plugin-owned allowlist adapters and channel metadata, preserve inherited account allowlists during writes, keep current main's generic hook/security handling, and drop the unrelated CI runtime bump.

## Reproduction Assessment

Yes, from source inspection. Current main lacks a Feishu `allowlist` adapter while the PR branch routes Feishu through a generic core write path that can override inherited top-level allowlists; I did not run a live Feishu setup.

## Solution Assessment

No. The narrow maintainable fix is to implement Feishu parity through the plugin adapter and metadata contracts already present on current main, not by adding Feishu-specific branches to the shared command handler.
