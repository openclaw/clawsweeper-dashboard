## Summary

Keep open: the PR is useful source work, but the branch should not merge as-is because it still enables Feishu account edits through a generic write path that can shadow inherited top-level allowlists, and it includes an unrelated CI runtime bump.

## What This Changes

The PR adds Feishu support to `/allowlist` config/listing, markdown/envelope/security-fix/hook mapping paths, adds regression tests, and bumps the setup Bun version.

## Best Possible Solution

Port the remaining Feishu behavior onto current main's plugin-owned channel capability and allowlist adapter seams, preserve inherited account allowlists during writes, and drop unrelated CI/runtime changes.

## Reproduction Assessment

Yes. Source inspection of the PR head shows that adding Feishu to the generic edit path makes named-account edits read only account-local entries before writing, so inherited top-level `allowFrom` or `groupAllowFrom` can be lost when the new account-local list shadows it.

## Solution Assessment

No. The better fix on current main is to use plugin-owned capability metadata and the shared/plugin allowlist adapter, not add more Feishu-specific branches in core lists.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve inherited Feishu allowlists on edit:** `src/auto-reply/reply/commands-allowlist.ts:417`
  - body: This line opts Feishu into the generic config edit path, but that path builds the existing list only from the selected account object before writing. In a multi-account config where `accounts.ops` inherits top-level `allowFrom`, `/allowlist add ... channel=feishu --account ops` writes a new account-local list containing only the new entry and shadows the inherited owners, silently dropping access. Merge the effective inherited list before writing or move the edit through the plugin adapter that understands Feishu account inheritance.
  - confidence: 0.88
- **[P3] Remove the unrelated Bun runtime bump:** `.github/actions/setup-node-env/action.yml:64`
  - body: The PR changes the shared setup action from Bun 1.3.9 to 1.3.10 even though the functional change is Feishu allowlist/channel behavior. That affects every workflow using `install-bun` and should be split or justified separately so this fix does not carry unrelated CI runtime risk.
  - confidence: 0.78
