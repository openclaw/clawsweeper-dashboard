## Summary

Keep this PR open: the Feishu parity goal is still useful, but the branch is stale against current main, duplicates the newer plugin-owned `/allowlist` boundary, still has a source-level allowlist integrity bug, and lacks after-fix real behavior proof.

## What This Changes

The branch adds Feishu handling to `/allowlist`, markdown-capable detection, envelope stripping, hook channel schema/help/tests, security-fix hardening, targeted tests, and a CI Bun patch-version bump.

## Best Possible Solution

Add Feishu-owned allowlist and markdown metadata adapters in the plugin, preserve inherited top-level/account config on writes, keep the generic hook/security behavior already on main, and require redacted after-fix proof.

## Reproduction Assessment

Yes for the PR blocker at source level: PR head reads existing allowlist entries from the raw account write target while Feishu account resolution merges top-level and account config. I did not run live Feishu or CLI behavior in this read-only review.

## Solution Assessment

No. The current diff hardcodes Feishu into an older core path; the maintainable fix is to rebase onto current main and implement Feishu plugin-owned allowlist/metadata adapters without dropping inherited config.
