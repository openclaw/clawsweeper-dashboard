## Summary

Keep this PR open, but do not merge it as submitted. Current main still has a real Feishu `/allowlist --config` gap, but the PR is stale against the current plugin-adapter direction, weakens authorization-sensitive allowlist write behavior, mishandles inherited Feishu account allowlists, and includes unrelated CI runtime churn.

## What This Changes

The PR adds Feishu handling to `/allowlist`, markdown capability checks, envelope stripping, hook channel validation/help, security-fix hardening, targeted tests, and a Bun version change in the shared setup action.

## Best Possible Solution

Rework the feature around a Feishu-owned allowlist adapter and Feishu channel metadata: preserve the current owner and config-write gates, read and edit effective merged account allowlists without dropping inherited entries, set markdown capability through plugin metadata/manifest, keep the generic hook and security-fix behavior from current main, and split the Bun setup-action bump into a separate PR if still needed.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main requires `plugin.allowlist.applyConfigEdit` for config-backed `/allowlist` writes, and the Feishu plugin currently has config helpers but no `allowlist` adapter; inherited account behavior is visible in `resolveFeishuAccount` merging top-level and account config.

## Solution Assessment

No. The PR contains useful source work, but the submitted approach hardcodes Feishu into shared paths, is stale against current main's plugin contracts, and introduces security-sensitive regressions; the maintainable solution belongs in Feishu plugin metadata and an inheritance-aware allowlist adapter.
