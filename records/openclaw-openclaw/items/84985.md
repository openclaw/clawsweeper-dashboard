## Label Justifications

- `P2`: The report describes a real upgrade repair bug with limited Codex-plugin blast radius that is already fixed in the latest release.
- `impact:data-loss`: The reported failure deletes operator plugin configuration during `doctor --fix`.
- `impact:auth-provider`: The reported failure disables the Codex plugin and unregisters the Codex agent harness used by model/runtime routing.

## Summary

Current main and the v2026.5.19 release already handle this upgrade repair: the Codex doctor contract removes the retired key before the generic invalid-plugin quarantine can delete the config and disable the plugin, and the regression test preserves the rest of the Codex config.

## What This Changes

The issue asks doctor repair to preserve Codex plugin enablement and migrate or safely remove `plugins.entries.codex.config.codexDynamicToolsProfile` during upgrades.

## Best Possible Solution

Keep the shipped plugin-owned doctor contract path: remove the retired Codex profile key before generic invalid-plugin quarantine, preserve the plugin entry, and rely on `codexDynamicToolsLoading` or its default for the modern behavior.

## Reproduction Assessment

Yes for the reported 2026.5.18 failure mode: source shows the generic invalid-plugin repair deletes config and disables the plugin when a known compatibility migration does not run first. Current main and v2026.5.19 should not reproduce it because the Codex doctor contract removes the retired key before that quarantine path.

## Solution Assessment

Yes. The current plugin-owned doctor contract is the narrow maintainable fix because it keeps Codex-specific compatibility knowledge in the Codex plugin surface and preserves the rest of the config entry.
