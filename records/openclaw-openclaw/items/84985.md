- `impact:data-loss`: The issue is about doctor --fix deleting operator plugin config during an upgrade repair.
- `impact:auth-provider`: The resulting disabled Codex plugin affects agent harness/provider runtime routing for scheduled jobs.

## Summary

Current main and the v2026.5.19 release already handle the reported upgrade repair path: the retired Codex key is removed through the plugin doctor normalizer before invalid-plugin quarantine runs, and disabled Codex entries are repaired when configured routes require the Codex runtime.

## What This Changes

The issue reports that doctor --fix should preserve or migrate a renamed Codex plugin config key instead of deleting it and disabling the plugin.

## Best Possible Solution

Keep the v2026.5.19 doctor behavior: retire codexDynamicToolsProfile, preserve supported Codex config, and re-enable Codex only when configured routes require that runtime.

## Reproduction Assessment

No current-main failing repro remains: source inspection shows current doctor first removes the retired key through the Codex compatibility normalizer and then repairs disabled Codex entries when Codex runtime routes require them. The historical failure is well explained by the old invalid-plugin quarantine and disabled-harness path, but current main takes the fixed path.

## Solution Assessment

Yes. The current solution is the maintainable boundary: Codex owns its retired config key through a plugin doctor contract, while core doctor route repair handles the generic disabled-owner-plugin runtime failure without adding a new config knob.
