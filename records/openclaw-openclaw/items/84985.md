## Summary

Current main and the latest shipped release already handle this upgrade path: the Codex plugin doctor contract removes the retired key before generic invalid-plugin quarantine can disable the plugin, and the release docs describe that removal as intentional rather than a rename migration.

## What This Changes

The issue reports that `openclaw doctor --fix` deletes a retired Codex plugin config key and disables the Codex plugin instead of preserving the plugin entry during upgrade repair.

## Best Possible Solution

Keep the shipped plugin-owned doctor compatibility migration: remove the retired Codex profile key with an explicit note, preserve the plugin entry, and leave `codexDynamicToolsLoading` on its documented default unless explicitly configured.

## Reproduction Assessment

No high-confidence current-main failure path remains. Source and shipped tests show current main removes the retired key through the Codex doctor migration before the generic invalid-plugin quarantine can disable the plugin.

## Solution Assessment

Yes. The current plugin-owned doctor migration is the narrow maintainable fix because the old field is documented as retired, not mapped to a new value, and the migration preserves the rest of the Codex plugin entry.
