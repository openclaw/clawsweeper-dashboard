## Summary

Keep this open: current main still preserves an exact plugin LLM allowlist boundary, and the existing Lossless doctor repair does not cover an already-plugin-style config whose `summaryModel` is bare or legacy while the runtime request is canonical `openai/gpt-5.4-mini`.

## What This Changes

The issue reports that Lossless/LCM deferred maintenance can request `openai/gpt-5.4-mini` while existing Lossless plugin LLM allowlists only permit `openai-codex/gpt-5.4-mini`, causing compaction to fail.

## Best Possible Solution

Add a narrow Lossless doctor/default repair that preserves explicit plugin LLM allowlists while adding the canonical `openai/gpt-5.4-mini` ref when the configured Lossless summary path can request it.

## Reproduction Assessment

Yes by source inspection: a Lossless plugin LLM policy that only allows `openai-codex/gpt-5.4-mini` will not authorize a request resolved as `openai/gpt-5.4-mini`. I did not execute a live Lossless maintenance run because this review is read-only.

## Solution Assessment

Yes, a targeted Lossless config/doctor repair is the best fit; broad runtime aliasing would weaken a security-sensitive allowlist boundary and create confusing policy semantics.
