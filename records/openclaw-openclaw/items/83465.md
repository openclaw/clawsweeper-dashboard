## Summary

Keep this open: the reported fail-closed allowlist mismatch is consistent with current source, and current main does not appear to repair existing plugin-style Lossless configs that allow only the legacy `openai-codex/*` mini ref while the runtime requests canonical `openai/*`.

## What This Changes

The issue asks OpenClaw to prevent Lossless/LCM deferred maintenance from failing when its plugin LLM allowlist contains `openai-codex/gpt-5.4-mini` but the maintenance call requests `openai/gpt-5.4-mini`.

## Best Possible Solution

Add a narrow `openclaw doctor --fix` or generated-default repair for Lossless that preserves explicit allowlists while adding the canonical `openai/gpt-5.4-mini` ref when the configured Lossless summary path can request it.

## Reproduction Assessment

Yes by source inspection: a Lossless plugin LLM policy that only allows `openai-codex/gpt-5.4-mini` will not authorize a request resolved as `openai/gpt-5.4-mini`. I did not execute a live Lossless maintenance run.

## Solution Assessment

The safest fix is not broad runtime aliasing; it is a targeted Lossless config/doctor repair that adds the canonical model ref while keeping the explicit allowlist gate intact.
