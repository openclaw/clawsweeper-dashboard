## Summary

Keep this open: current main still enforces exact canonical plugin LLM allowlists, and it does not repair already-migrated Lossless configs whose allowlist only contains the legacy `openai-codex/gpt-5.4-mini` ref while maintenance requests `openai/gpt-5.4-mini`.

## What This Changes

The issue asks OpenClaw to prevent Lossless/LCM deferred maintenance from failing when legacy Lossless plugin LLM allowlists omit the current canonical mini summary model ref.

## Best Possible Solution

Add a targeted Lossless doctor/default repair that preserves explicit plugin LLM allowlists while adding the canonical `openai/gpt-5.4-mini` ref whenever the configured Lossless summary path can request it.

## Reproduction Assessment

Yes by source inspection: a Lossless context-engine LLM request resolved as `openai/gpt-5.4-mini` will not pass a configured allowlist that only contains `openai-codex/gpt-5.4-mini`. I did not run a live Lossless maintenance cycle.

## Solution Assessment

Yes, the narrow maintainable fix is a Lossless config/doctor repair that adds the canonical model ref while keeping the explicit plugin LLM allowlist gate intact. Broad runtime provider aliasing would be a riskier security-boundary change.
