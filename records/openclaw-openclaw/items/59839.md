## What This Changes

The issue asks for Control UI/WebChat New session to create and select a new chat while keeping the previous chat accessible through session switching.

## Best Possible Solution

Keep the focused current-main behavior: the visible WebChat New session/New Chat affordance should create a real dashboard session, switch to it, and leave the old session selectable, while separate reconnect/reset-archive recovery bugs continue in their narrower reports.

## Reproduction Assessment

Yes. The original behavior has a high-confidence static reproduction path through the old visible button sending `/new` into the reset path, and current main can be checked statically by following the same UI affordance to `sessions.create` instead.

## Solution Assessment

Yes. The current `sessions.create` create-and-switch path is the narrow maintainable fix because it uses an existing gateway API, preserves the old session row, and avoids broad sidebar or reset-archive recovery changes.
