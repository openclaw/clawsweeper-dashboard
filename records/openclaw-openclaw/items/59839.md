## Summary

Current main still reproduces the reported WebChat behavior: the visible New session control sends the reset-backed `/new` command, the gateway treats that as a session reset, and reset archives are not exposed as selectable sessions in the current session picker. The issue should stay open while the focused open maintainer PR #73725, or an equivalent patch, is reviewed and landed.

## What This Changes

Control UI/WebChat New session should create and select a separate chat session while leaving the previous chat accessible from session switching/history UI.

## Best Possible Solution

Make the visible WebChat New session/New Chat affordance use `sessions.create`, switch to the returned dashboard session key, refresh session options, and reserve `/new` or reset behavior for explicit reset flows. Reset archives can remain reset-only, but if users are expected to recover them through the UI that should be a separate first-class recovery/listing path.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: click the WebChat New session button, observe the UI sends `/new`, and follow the gateway path that resets the current session key and archives the prior transcript; the existing UI test asserts the same request shape.

## Solution Assessment

Yes. The narrow `sessions.create` create-and-switch direction in #73725 is the best maintainable fix because it uses an existing gateway API and avoids broad sidebar/session-management churn while preserving the old session as a selectable row.
