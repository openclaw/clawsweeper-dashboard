## Summary

Keep open. Current main has covered major parts of the hardening request, but OAuth-producing setup paths still write through the generic unlocked profile upsert, and duplicate-enrollment protection still relies on provider/profile identity rather than a stable enrollment identity across copied stores or hosts.

## What This Changes

The issue requests harder OpenAI Codex OAuth auth-store persistence: atomic writes, duplicate-enrollment protection, safer Codex CLI sync, and consistent locking for OAuth credential mutations.

## Best Possible Solution

Finish a focused OAuth hardening pass that routes OAuth-producing writes through locked upserts, adds feasible same-store duplicate-enrollment detection or warnings using stable account identity, and documents the remaining single-writer boundary for copied stores and multi-host deployments.

## Reproduction Assessment

Yes for a source-level remaining gap: current OAuth setup persistence still reaches unlocked upsertAuthProfile call sites in provider auth and OAuth helper paths. A live refresh_token_reused reproduction would require duplicated real OpenAI OAuth enrollment state and token rotation, which was not run in this read-only review.

## Solution Assessment

No for closing this as solved. Current main is a partial implementation; the maintainable path is a focused auth-store hardening pass with maintainer review for duplicate identity and multi-writer policy.
