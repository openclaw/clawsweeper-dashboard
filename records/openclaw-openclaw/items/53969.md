## Summary

Keep open. Current main has shipped several important pieces of the request, including atomic JSON writes, Codex CLI bootstrap-only behavior, provider/profile refresh serialization, and refresh_token_reused fallback coverage, but the remaining OAuth mutation hardening is still incomplete because OAuth-producing setup paths still persist through the unlocked generic upsert path and duplicate-enrollment protection remains keyed to provider/profile/store rather than a stable enrollment boundary.

## What This Changes

The issue requests safer OpenAI Codex OAuth auth-profile persistence through atomic auth-store writes, duplicate-enrollment defenses, safer Codex CLI sync, and consistent locking for OAuth credential mutations.

## Best Possible Solution

Keep the existing shipped hardening and finish the remaining auth-store safety work: make OAuth-producing setup writes use a lock-safe mutation path that preserves existing credential normalization, add feasible same-store duplicate-enrollment warnings or rejection based on stable identity metadata, and document the single-writer boundary for copied stores, sibling agents, multiple hosts, and Codex CLI coexistence.

## Repair Work Prompt

Fix the remaining hardening tracked by this issue without changing provider/product policy. Scope the PR to OpenClaw auth-profile persistence: route OAuth-producing writes through a lock-safe mutation path, preserving the existing upsertAuthProfile normalization semantics for api_key/token credentials. In particular, cover applyProviderAuthMethod in src/plugins/provider-auth-choice.ts, persistProviderAuthResult in src/commands/models/auth.ts, and writeOAuthCredentials plus sibling sync in src/plugins/provider-auth-helpers.ts. Prefer adding/reusing a single lock-safe upsert helper that normalizes credentials consistently, or route only OAuth credentials through a lock-safe OAuth-specific helper while leaving non-OAuth behavior unchanged. Add regression tests that would fail if two OAuth setup writes can stale-overwrite each other or if writeOAuthCredentials bypasses the locked updater. Add concise docs to the OpenAI/auth credential docs explaining that copied auth-profiles.json stores and multi-host deployments need one writer per OAuth enrollment, and that OpenClaw can only coordinate within the local store/profile lock boundary. Do not remove existing atomic JSON writes, Codex CLI bootstrap-only behavior, same-profile refresh serialization, stable openai-codex profile IDs, or refresh_token_reused recovery. Changelog: add a user-facing Fixes entry with Thanks @falkenbergo if runtime behavior or docs materially change.
