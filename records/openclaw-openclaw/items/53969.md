## Summary

Keep open. Current main has implemented several important parts of the requested hardening, including atomic JSON writes, bootstrap-only Codex CLI behavior, cross-agent per-profile refresh serialization, and refresh_token_reused recovery, but the request is not fully resolved because OAuth-producing setup writes still use the generic unlocked upsert path and duplicate-enrollment safety is not keyed to a stable enrollment across copied stores, profile ids, or hosts.

## What This Changes

Harden OpenAI Codex OAuth auth-profile persistence around atomic writes, duplicate enrollments, external Codex CLI sync, and locked OAuth credential mutations.

## Best Possible Solution

Keep the issue open and finish a targeted OAuth hardening pass: route OAuth-producing writes through a locked updater, add feasible same-store duplicate-enrollment warnings or rejection using account identity metadata, and keep docs explicit about single-writer limits for copied auth stores, sibling agents, multiple hosts, and Codex CLI coexistence.

## Reproduction Assessment

Yes for a source-level remaining gap: current OAuth setup persistence still reaches unlocked upsertAuthProfile call sites in provider auth and OAuth helper paths. A live refresh_token_reused reproduction would require real duplicated OpenAI OAuth enrollment state and token rotation, which was not run in this read-only review.

## Solution Assessment

No for closing this as solved. Current main is a partial implementation; the maintainable path is a focused auth-store hardening and docs pass with maintainer review for duplicate identity and multi-writer policy.
