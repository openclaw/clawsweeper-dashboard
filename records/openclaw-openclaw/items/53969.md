## Summary

Keep open. Current main implements important parts of the requested hardening, including atomic JSON writes, cross-agent same-profile refresh serialization, Codex CLI bootstrap-only behavior, and refresh_token_reused recovery, but the central issue is not fully resolved because OAuth-producing setup writes still flow through an unlocked upsert path and duplicate-enrollment protection is not keyed to a stable OAuth enrollment across copied stores, profile IDs, or hosts.

## What This Changes

The issue requests harder OpenAI Codex OAuth auth-profile persistence: atomic writes, duplicate-enrollment safeguards, safer Codex CLI sync semantics, and consistent locking for OAuth credential mutations.

## Best Possible Solution

Keep the issue open and narrow the remaining work to lock-safe OAuth-producing writes, feasible same-store duplicate-enrollment detection or warnings using accountId/email identity metadata, and explicit docs for single-writer boundaries across copied auth-profiles.json files, sibling agents, multiple hosts, and Codex CLI coexistence. Preserve the shipped atomic writes, bootstrap-only Codex CLI behavior, refresh serialization for shared provider/profile IDs, and refresh_token_reused recovery.

## Reproduction Assessment

Yes for the remaining source-level gap: current main still traces provider auth-result persistence and writeOAuthCredentials into upsertAuthProfile without the locked updater. A live refresh_token_reused reproduction would require real duplicated OpenAI OAuth enrollment state and token rotation, which was not run in this read-only review.

## Solution Assessment

No for closing this as solved. Current main is a partial implementation; the maintainable solution is a targeted auth-store hardening pass plus docs, with owner review for duplicate identity and multi-writer policy.
