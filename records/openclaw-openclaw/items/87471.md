## Summary

Keep this PR open: it targets a real WebChat replay gap and includes useful terminal proof, but the new immediate `sessions_yield` mirror path can duplicate the same visible status when already-projected history is projected again during later inline appends.

## What This Changes

The PR generalizes WebChat replay mirroring for visible tool output and adds `sessions_yield` assistant-visible mirrors while keeping raw tool results hidden by default.

## Best Possible Solution

Make the visible-tool mirror projection idempotent for already-mirrored `sessions_yield` tool results, then keep the current boundary of projecting durable visible status without rendering generic raw tool results.

## Reproduction Assessment

Yes. The linked issue gives a concrete WebChat replay path where a visible `sessions_yield` status is persisted only as a `toolResult`, and source inspection confirms current default WebChat rendering hides raw tool results.

## Solution Assessment

No, not yet. Mirroring visible tool status at the gateway projection boundary is a good fit, but the implementation needs an idempotency guard for already-projected histories before it is safe to merge.
