## Summary

This should stay open: current main has a broad owner-facing cron tool/RPC and SQLite-backed scheduler, but it does not implement the RFC’s narrower gateway-enforced `schedule_task` style API with non-forgeable creator provenance, playbook allowlists, recurring-job expiry, per-agent quotas, or schedule-create audit entries.

## What This Changes

Request an agent-facing scheduling API that lets agents create or modify bounded cron jobs through gateway-enforced playbook allowlists and non-forgeable provenance.

## Best Possible Solution

Keep this as a maintainer-owned security/product design item for a narrow gateway scheduling contract that preserves operator cron while adding approved playbooks, immutable creator provenance, bounded recurrence, quotas, audit records, and migration semantics if maintainers approve it.

## Reproduction Assessment

Yes for source-level gap proof, not as a failing bug reproduction. Current main shows broad cron CRUD and owner-only mitigations, but no separate playbook/provenance scheduling API in the tool, gateway protocol, persisted schema, or docs.

## Solution Assessment

Unclear until maintainers decide the product/security model. The requested direction is plausible, but the maintainable fix should be a designed gateway contract rather than exposing the existing broad cron tool more widely.
