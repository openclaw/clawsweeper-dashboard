## Summary

Keep open: current main still exposes broad operator/admin cron creation and a narrow self-cleanup exception, but not the requested security-sensitive agent scheduling contract with gateway-derived provenance, playbook allowlisting, quotas, bounded/permanent lifetime policy, and durable schedule mutation audit semantics. The AS76 comments add concrete production context, and the linked recurring-expiration discussion remains open.

## What This Changes

Request an agent-facing Gateway scheduling API that lets agents create constrained cron tasks while the Gateway injects non-forgeable provenance and enforces playbook, quota, lifetime, and audit rules.

## Best Possible Solution

Create a short core design/security decision record, then add an additive Gateway scheduling capability that derives actor and timestamp from authenticated request context, accepts only registered playbooks plus validated params, enforces quota and lifetime policy, records schedule create/update audit events, and keeps unrestricted operator cron separate.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security-sensitive feature/RFC. The high-confidence verification path is current-main source and docs inspection showing no schedule_task/playbook surface, injected createdBy actor, per-agent quota, recurring lifetime policy, or durable schedule-creation audit ledger.

## Solution Assessment

Unclear until maintainers choose the product and security contract. The requested direction is safer than exposing full cron.add to non-owner agents, but the maintainable fix should be a designed Gateway/cron/audit capability rather than a one-off schema patch.
