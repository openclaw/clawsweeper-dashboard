## Summary

Keep open: current main still lacks the requested agent-facing scheduling contract. The existing cron surfaces are broad operator/admin APIs plus an owner-only cron tool, while this RFC asks for new gateway-derived provenance, playbook allowlisting, quotas, recurring lifetime policy, and schedule-mutation audit semantics.

## What This Changes

Add an agent-facing Gateway scheduling API that lets agents create constrained cron jobs with non-forgeable provenance, playbook allowlisting, quotas, expiration policy, and audit records.

## Best Possible Solution

Approve a core Gateway/cron/audit design, then add an additive scheduling capability that derives actor and timestamp from authenticated request context, accepts only registered playbooks with validated params, enforces quota and lifetime policy, records schedule mutations, and keeps unrestricted operator cron separate.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security-sensitive feature/RFC. The high-confidence check is current-main source and docs inspection showing no schedule_task/playbook surface, gateway-injected createdBy actor, per-agent quota, activeUntil/permanent policy, or durable schedule-mutation audit ledger.

## Solution Assessment

Unclear until maintainers choose the product and security contract. The requested direction is safer than exposing broad cron.add to non-owner agents, but the maintainable fix should be a designed Gateway/cron/audit capability rather than a one-off schema patch.
