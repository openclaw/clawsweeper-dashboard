## Summary

Keep open: current main still has broad operator/gateway cron creation and execution history, but no constrained agent-facing scheduling API with gateway-injected provenance, playbook allowlisting, quotas, recurring lifetime policy, or schedule mutation audit. The AS76 comments add concrete production context rather than resolving or superseding the security-sensitive design gap.

## What This Changes

Request an agent-facing scheduler API that lets agents create constrained cron tasks with non-forgeable gateway provenance, playbook allowlists, quotas, expiry policy, and audit logging.

## Best Possible Solution

Design an additive Gateway scheduling capability that derives actor and timestamp from authenticated request context, accepts only registered playbooks with validated params, enforces quota and lifetime policy, records schedule mutation audit events, and keeps unrestricted operator cron separate.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security-sensitive feature/RFC. Source inspection gives high confidence the requested schedule_task, gateway-injected provenance, playbooks, quotas, recurring lifetime policy, and schedule-mutation audit are not present on current main.

## Solution Assessment

Unclear until maintainers choose the policy contract. The requested direction is safer than exposing broad cron.add to non-owner agents, but the maintainable fix should be a designed Gateway/cron/audit capability rather than a narrow schema patch.
