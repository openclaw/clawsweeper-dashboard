## Summary

Keep open: current main still lacks the requested agent-facing scheduling contract, and the follow-up comments add real multi-agent usage context rather than making the RFC obsolete.

## What This Changes

The issue requests a new agent-facing scheduling API with gateway-injected provenance, playbook allowlisting, quotas, lifetime policy, and schedule-mutation audit records.

## Best Possible Solution

Make a core design/security decision, then add an additive Gateway scheduling capability that derives actor and timestamp from authenticated request context, accepts only registered playbooks with validated params, enforces quota and lifetime policy, records schedule mutation audit events, and keeps unrestricted operator cron separate.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security-sensitive feature/RFC. Source and docs inspection on current main gives high confidence that schedule_task, gateway-injected createdBy, playbooks, per-agent quota, recurring lifetime policy, and schedule-mutation audit do not exist.

## Solution Assessment

Unclear until maintainers choose the policy contract. The requested direction is safer than exposing broad cron.add to non-owner agents, but the maintainable fix should be a designed Gateway/cron/audit capability rather than a one-off schema patch.
