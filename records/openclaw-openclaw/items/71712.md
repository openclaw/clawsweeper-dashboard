## Summary

Keep open: current main still exposes cron creation through the existing operator/admin cron surface and owner-only cron tool, but it does not implement the requested constrained agent-facing scheduling API, gateway-injected actor provenance, playbook allowlist, per-agent quota, recurring lifetime policy, or schedule-creation audit ledger. The RFC is recent, security-sensitive, and AS76 added concrete production scheduling context, so this needs maintainer design review rather than cleanup closure.

## What This Changes

The issue requests an agent-facing scheduling API that lets agents create constrained recurring jobs through the gateway with gateway-injected provenance, playbook allowlists, quotas, lifetime limits, and audit entries.

## Best Possible Solution

Create a short core security/design record first, then add an additive Gateway scheduling capability that derives actor and timestamp from authenticated request context, accepts only registered playbook identifiers plus schema-validated params, enforces quota and lifetime policy, records schedule create/update audit events, and keeps the unrestricted operator cron CLI/admin RPC separate.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The high-confidence verification path is source and docs inspection: current main has admin cron.add and an owner-only cron tool, but no constrained schedule_task/provenance/playbook/quota/lifetime contract.

## Solution Assessment

Unclear until maintainers choose the product and security model. The requested direction is safer than exposing the full cron surface to agents, but the maintainable solution should be a designed Gateway/cron/audit capability rather than a one-off schema patch.
