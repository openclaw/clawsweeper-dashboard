## Summary

Keep open: current main has adjacent cron hardening, SQLite-backed persistence, non-owner tool denial, task audit, and creator-tool allowlist caps, but it still lacks the requested agent-facing schedule-authority API with Gateway-injected provenance, playbook allowlists, quotas, expiry policy, and execution lineage.

## What This Changes

The issue requests a Gateway-mediated agent scheduling API with non-forgeable provenance, playbook allowlists, quotas, expiration, and audit lineage for agent-created cron jobs.

## Best Possible Solution

Keep this as a maintainer-owned security/product RFC and design one narrow Gateway schedule-authority contract covering creation provenance, execution lineage, playbook registration/versioning, quotas, activeUntil/permanent policy, audit records, and migration semantics.

## Reproduction Assessment

Yes for source-level gap proof, not as a failing bug reproduction. Current main has broad cron CRUD and mitigations, but no separate playbook/provenance scheduling API in the agent tool, Gateway protocol, persisted cron schema, or docs.

## Solution Assessment

Unclear until maintainers decide the product and security model. The requested direction is plausible, but the best fix should be a designed Gateway scheduling-authority contract rather than widening the existing broad cron tool.
