## Summary

Keep open: current main has a broad cron tool/RPC, but it does not implement the requested non-forgeable agent scheduling contract with gateway-injected provenance, playbook allowlisting, quotas, and lifetime policy. This is security-sensitive product/API design work, not a stale cleanup close or safe autonomous fix PR.

## What This Changes

Requests a gateway-owned agent scheduling API that injects non-forgeable provenance, restricts jobs to allowlisted playbooks, enforces expiration/quota policy, and writes audit entries.

## Best Possible Solution

Define a gateway-owned scheduling surface that derives creator identity from the authenticated session, accepts only operator-registered playbooks with schema-validated params, enforces quota and lifetime policy, records audit/task-ledger provenance, and keeps the unrestricted operator cron CLI/RPC separate.

## Reproduction Assessment

No live failure reproduction is applicable because this is an RFC, but the gap is source-reproducible: the current cron add schema and handler do not include non-forgeable provenance, playbooks, quota, or active-until enforcement.

## Solution Assessment

Unclear until maintainers choose the security/product model. The requested direction is plausible, but the best fix must reconcile cron tool policy, gateway auth scopes, task/audit ledger semantics, existing jobs without provenance, and the related activeUntil proposal.
