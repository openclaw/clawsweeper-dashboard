## Summary

Keep open: current main still has the existing admin/operator cron creation path and an owner-only cron tool, but not the requested constrained agent-facing scheduling API with gateway-injected provenance, playbook allowlists, per-agent quotas, recurring lifetime policy, or schedule-creation audit semantics. The issue is recent, security-sensitive, and the follow-up production-use comments add real deployment context rather than stale noise.

## What This Changes

Requests a Gateway-exposed agent scheduling API that lets agents create or modify scheduled cron work through allowlisted playbooks with non-forgeable provenance, quotas, lifetime controls, and audit records instead of direct `jobs.json` or full admin cron access.

## Best Possible Solution

Create a short core security/design record first, then add an additive Gateway scheduling capability that derives creator identity and timestamp from the authenticated request context, accepts only registered playbook identifiers plus schema-validated params, enforces quota and lifetime policy, records schedule create/update audit events, and keeps the unrestricted operator cron CLI/admin RPC separate.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The high-confidence verification path is source and docs inspection: current main has admin `cron.add` and an owner-only cron tool, but no constrained schedule_task/provenance/playbook/quota/lifetime contract.

## Solution Assessment

Unclear until maintainers choose the product and security model. The requested direction is safer than exposing the full cron surface to agents, but the maintainable solution should be a designed Gateway/cron/audit capability rather than a one-off patch.
