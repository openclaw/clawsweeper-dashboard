## Summary

Keep open: current main still has broad admin/operator cron creation and a narrow isolated-run self-cleanup exception, but not the requested agent scheduling contract with gateway-derived provenance, playbook allowlisting, quotas, recurring lifetime policy, and durable schedule-mutation audit semantics. The AS76 comments add concrete production context, and the related recurring-expiration discussion at https://github.com/openclaw/openclaw/issues/71711 is still open.

## What This Changes

The RFC requests a gateway-exposed agent scheduling API that creates constrained cron jobs with non-forgeable provenance, playbook allowlisting, quotas, bounded recurring lifetimes, and audit entries.

## Best Possible Solution

Create a core design/security decision record, then add an additive Gateway scheduling capability that derives actor and timestamp from authenticated request context, accepts only registered playbooks with validated params, enforces quota and lifetime policy, records schedule create/update audit events, and keeps unrestricted operator cron separate.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security-sensitive feature/RFC. The high-confidence verification path is current-main source and docs inspection showing no schedule_task/playbook surface, injected createdBy actor, per-agent quota, recurring lifetime policy, or durable schedule-creation audit ledger.

## Solution Assessment

Unclear until maintainers choose the product and security contract. The requested direction is safer than exposing full cron.add to non-owner agents, but the maintainable fix should be a designed Gateway/cron/audit capability rather than a one-off schema patch.
