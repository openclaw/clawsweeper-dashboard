## Summary

Keep open: current main still lacks the requested agent-facing scheduling contract, and the recent production-use comments make this a real security/product design item rather than cleanup.

## What This Changes

Request an agent-facing Gateway scheduling API that creates constrained cron jobs with gateway-injected provenance, playbook allowlisting, quotas, lifetimes, and audit records.

## Best Possible Solution

Write a short core security design record, then add an additive Gateway scheduling capability that derives actor/time from authenticated context, accepts only registered playbooks with validated params, enforces quota and lifetime policy, and records durable schedule mutation audit events while keeping unrestricted operator cron separate.

## Reproduction Assessment

Not applicable as a runtime bug. The high-confidence verification path is current-main source and docs inspection showing no schedule_task/playbook surface, injected createdBy actor, per-agent quota, recurring lifetime policy, or durable schedule-creation audit ledger.

## Solution Assessment

Unclear until maintainers choose the security and product contract. The requested direction is safer than exposing full cron.add to non-owner agents, but it should be a designed Gateway/cron/audit capability rather than a one-off schema patch.

## Security Review

Status: needs_attention

Summary: This non-PR issue reports a concrete security-sensitive scheduling design gap; no patch was reviewed.

Concerns:

- **[medium] Persistent schedules lack non-forgeable creator provenance:** `src/cron/types-shared.ts:1`
  - body: Cron jobs store timestamps and optional agent/session routing, but not a gateway-derived creator envelope that distinguishes operator-created jobs from agent-created jobs in a non-forgeable way.
  - confidence: 0.9
- **[medium] Agent-safe scheduling needs a constrained request schema:** `src/gateway/protocol/schema/cron.ts:356`
  - body: The current cron.add schema accepts broad systemEvent or agentTurn job creation, so exposing it to non-owner agents would not enforce playbook allowlists, parameter validation, quotas, or recurring lifetime guardrails.
  - confidence: 0.88
- **[medium] Cron mutation has no narrow agent write grant:** `src/gateway/method-scopes.ts:188`
  - body: Cron mutation methods are scoped to operator.admin, and the agent tool is owner-only except for current-job self-cleanup, so there is no safe middle path for agent-owned scheduling.
  - confidence: 0.86
- **[low] Schedule creation lacks durable audit semantics:** `src/cron/run-log.ts:22`
  - body: Cron run history records finished executions, and the tasks ledger tracks detached work; neither surface currently records schedule_task create/update provenance as durable audit events.
  - confidence: 0.78
