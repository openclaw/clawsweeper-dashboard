## Summary

Keep open: current main still lacks the requested agent-facing scheduling contract. The existing cron surfaces are broad operator/admin APIs plus an owner-only model tool with a self-cleanup exception, while this RFC asks for new gateway-derived provenance, playbook allowlisting, quotas, recurring lifetime policy, and durable schedule-mutation audit semantics.

## What This Changes

The issue requests a gateway-exposed agent scheduling API that constrains agent-created cron jobs through allowlisted playbooks, injected provenance, lifetime limits, quotas, and audit records.

## Best Possible Solution

Create a core design/security decision, then add an additive gateway scheduling capability that derives actor and timestamp from authenticated request context, accepts only registered playbooks with validated params, enforces quota and lifetime policy, records schedule create/update audit events, and keeps unrestricted operator cron separate.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security-sensitive feature/RFC. The high-confidence check is current-main source and docs inspection showing no schedule_task/playbook surface, gateway-injected createdBy actor, per-agent quota, activeUntil/permanent policy, or durable schedule-mutation audit ledger.

## Solution Assessment

Unclear until maintainers choose the product and security contract. The requested direction is safer than exposing broad cron.add to non-owner agents, but the maintainable fix should be a designed Gateway/cron/audit capability rather than a one-off schema patch.
