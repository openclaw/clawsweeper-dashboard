## Summary

Keep open: current main still lacks the requested agent-facing scheduling contract, and the remaining work is security-sensitive product/API design rather than conservative cleanup.

## What This Changes

The issue requests an agent-facing scheduling API that derives non-forgeable provenance in the gateway and constrains agent-created recurring jobs through playbooks, quotas, lifetime policy, and audit records.

## Best Possible Solution

Create a core security/product design, then add an additive gateway scheduling capability that derives actor and timestamp from authenticated request context, accepts only registered playbooks with validated params, enforces quota and lifetime policy, records schedule mutation audit events, and keeps unrestricted operator cron separate.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security-sensitive RFC. The high-confidence check is current-main source and docs inspection showing no schedule_task/playbook surface, gateway-injected createdBy actor, per-agent quota, recurring lifetime policy, or durable schedule-mutation audit ledger.

## Solution Assessment

Unclear until maintainers choose the product and security contract. The requested direction is safer than exposing broad cron.add to non-owner agents, but the maintainable fix should be a designed Gateway/cron/audit capability rather than a one-off schema patch.
