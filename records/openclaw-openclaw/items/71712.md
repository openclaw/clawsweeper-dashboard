## Summary

This should stay open: current main still exposes broad cron CRUD through the agent tool and Gateway RPCs, but it does not provide the RFC's narrower gateway-enforced scheduling API with immutable creator provenance, playbook allowlists, recurring expiry, quotas, or schedule-create audit semantics.

## What This Changes

The issue requests a new agent-facing scheduling API that lets agents create bounded cron work through a gateway-enforced provenance and allowlist boundary.

## Best Possible Solution

Keep this as a maintainer-owned security/product RFC and design a narrow gateway scheduling contract only after the cron trust boundary, playbook registry, provenance model, quota policy, audit records, and migration behavior are decided.

## Reproduction Assessment

Yes for source-level gap proof, not as a failing bug reproduction. Current main has broad cron CRUD and owner-deny mitigations, but no separate playbook/provenance scheduling API in the agent tool, Gateway protocol, persisted cron schema, or docs.

## Solution Assessment

Unclear until maintainers decide the product and security model. The requested direction is plausible, but the best fix should be a designed Gateway contract rather than widening the existing broad cron tool.
