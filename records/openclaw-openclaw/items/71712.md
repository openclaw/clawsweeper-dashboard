## Summary

Keep open: current main still exposes cron creation as a broad operator/admin surface and owner-only cron tool, while this RFC asks for a new security-sensitive agent scheduling contract with gateway-derived provenance, allowlisted playbooks, quotas, lifetimes, and audit semantics.

## What This Changes

The issue requests an agent-facing scheduling API that injects non-forgeable provenance, restricts jobs to allowlisted playbooks, enforces quotas/lifetimes, and audits schedule creation.

## Best Possible Solution

Create a short core design/security decision record, then add an additive Gateway scheduling capability that derives actor and timestamp from authenticated request context, accepts only registered playbooks plus validated params, enforces quota and lifetime policy, records schedule create/update audit events, and leaves unrestricted operator cron separate.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security-sensitive feature/RFC. The high-confidence verification path is source and docs inspection on current main showing no schedule_task/playbook surface, injected createdBy, per-agent quota, recurring lifetime policy, or schedule-creation audit ledger.

## Solution Assessment

Unclear until maintainers choose the product and security model. The requested direction is safer than exposing full cron.add to non-owner agents, but the maintainable fix should be a designed Gateway/cron/audit capability rather than a one-off schema patch.
