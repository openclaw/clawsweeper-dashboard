## Summary

Keep open: current main still has the unrestricted operator/admin cron creation path plus an owner-only agent cron tool, but it does not implement the requested constrained agent-facing scheduling API with gateway-injected provenance, playbook allowlists, quotas, lifetime policy, or schedule-creation audit semantics. The RFC is recent, security-sensitive, and the comments add production-use evidence, so this needs maintainer product/security design rather than cleanup closure.

## What This Changes

The issue requests a documented, hardened agent-facing scheduling API that lets agents create or modify constrained recurring cron jobs while the gateway injects non-forgeable provenance and enforces playbook, quota, lifetime, and audit invariants.

## Best Possible Solution

Write a short core security/design record first, then add an additive gateway scheduling capability that derives creator identity and timestamp from the authenticated session, accepts only registered playbook identifiers plus schema-validated params, enforces per-agent quotas and recurring lifetime or permanent-job policy, records schedule create/update audit events, and leaves the unrestricted operator cron CLI/admin RPC intact.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The high-confidence verification path is source and docs inspection: current main has cron.add and an owner-only cron tool, but no schedule_task/provenance/playbook/quota/lifetime contract.

## Solution Assessment

Unclear until maintainers choose the product and security model. The requested direction is safer than exposing the existing full cron surface to agents, but the maintainable path should be a designed gateway/cron/audit capability rather than a one-off autonomous patch.
