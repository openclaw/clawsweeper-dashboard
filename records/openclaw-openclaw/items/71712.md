## Summary

Keep open: current main still has the operator/admin cron surface and owner-only cron tool, but not the requested constrained agent-facing scheduling API with gateway-injected provenance, playbook allowlists, quotas, lifetime policy, or durable schedule-creation audit. The RFC is recent, security-sensitive, and AS76 added concrete production multi-agent scheduling context, so this needs maintainer product/security design rather than cleanup closure.

## What This Changes

Request an agent-facing Gateway scheduling API that schedules allowlisted playbooks, injects non-forgeable provenance, enforces lifetime and quota policy, and records schedule audit events.

## Best Possible Solution

Create a short core security/design record first, then implement an additive Gateway scheduling capability that derives actor and timestamp from authenticated request context, accepts only registered playbook identifiers with schema-validated params, enforces quota and lifetime policy, records schedule create/update audit events, and keeps unrestricted operator cron separate.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. Source and docs inspection give a high-confidence verification path: current main has admin cron.add plus an owner-only full cron tool, but no schedule_task/provenance/playbook/quota/lifetime contract.

## Solution Assessment

Unclear until maintainers choose the product and security model. The constrained Gateway direction is safer than exposing full cron to agents, but the maintainable fix needs a designed protocol, cron, audit, and docs change rather than a one-off schema patch.
