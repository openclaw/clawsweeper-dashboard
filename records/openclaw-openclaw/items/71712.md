## Summary

Keep open: current main still has only the broad operator/admin cron surfaces plus a narrow self-cleanup exception, while this RFC asks for a new security-sensitive agent scheduling contract with gateway-derived provenance, playbook allowlisting, quotas, lifetimes, and durable audit semantics. AS76's comments add concrete production-use context for provenance, quotas, and permanent recurring jobs, so this should stay with maintainers rather than be swept closed.

## What This Changes

Add an agent-facing scheduling API that creates constrained cron jobs with gateway-injected provenance, playbook allowlists, quotas, bounded recurring lifetimes, and audit records.

## Best Possible Solution

Create a short core design/security decision record, then add an additive Gateway scheduling capability that derives actor and timestamp from authenticated request context, accepts only registered playbooks plus validated params, enforces quota and lifetime policy, records schedule create/update audit events, and leaves unrestricted operator cron separate.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security-sensitive feature/RFC. The high-confidence verification path is current-main source and docs inspection showing no schedule_task/playbook surface, injected createdBy actor, per-agent quota, recurring lifetime policy, or durable schedule-creation audit ledger.

## Solution Assessment

Unclear until maintainers choose the product and security contract. The requested direction is safer than exposing full cron.add to non-owner agents, but the maintainable fix should be a designed Gateway/cron/audit capability rather than a one-off schema patch.
