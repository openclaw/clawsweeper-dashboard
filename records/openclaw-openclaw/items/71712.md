## Summary

Keep open: current main and v2026.6.6 still expose broad cron CRUD plus deny-by-default mitigations, but not the narrower provenance-backed agent scheduling API requested here.

## What This Changes

The issue asks for an agent-facing scheduling API that records gateway-injected provenance, restricts jobs to allowlisted playbooks, and adds quotas, expiry, and audit coverage for agent-created schedules.

## Best Possible Solution

Keep this as a maintainer-owned security/product design item for a narrow gateway scheduling contract that preserves operator cron while adding approved playbooks, immutable creator provenance, bounded recurrence, quotas, audit records, and migration semantics if approved.

## Reproduction Assessment

Yes for source-level gap proof, not as a failing bug reproduction. Current main exposes broad cron CRUD plus deny-by-default security guidance, but lacks the proposed playbook/provenance scheduling API in tool, protocol, gateway handler, persistence, or docs.

## Solution Assessment

Unclear until maintainer product/security review. The requested direction is plausible, but the maintainable fix should be a designed gateway contract rather than widening the existing broad cron tool.
