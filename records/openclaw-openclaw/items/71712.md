## Summary

Keep open: current main still has the existing operator/admin cron surface and owner-only agent cron tool, but not the requested constrained agent-facing scheduling API with gateway-derived provenance, playbook allowlists, per-agent quotas, recurring lifetime policy, or schedule-creation audit semantics. The RFC is recent, security-sensitive, and the AS76 comments add concrete production context, so this should stay with maintainers for product/security design.

## What This Changes

Request an agent-facing scheduling API that lets agents create constrained recurring jobs while the Gateway injects non-forgeable creator provenance and enforces playbooks, quotas, lifetime bounds, and audit records.

## Best Possible Solution

Write a short core security/design record first, then add an additive Gateway scheduling capability that derives creator identity and timestamp from the authenticated session, accepts only registered playbook identifiers plus schema-validated params, enforces per-agent quotas and recurring lifetime/permanent policy, records schedule create/update audit events, and leaves the unrestricted operator cron CLI/admin RPC intact.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The high-confidence verification path is source and docs inspection: current main has cron.add and an owner-only cron tool, but no schedule_task/provenance/playbook/quota/lifetime contract.

## Solution Assessment

Unclear until maintainers choose the product and security model. The requested direction is safer than exposing the existing full cron surface to agents, but the maintainable path should be a designed core gateway/cron/audit capability rather than a one-off autonomous patch.
