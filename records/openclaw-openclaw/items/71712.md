## Summary

Keep open: current main still lacks the requested constrained agent-facing scheduling API, gateway-injected creator provenance, playbook allowlist, per-agent quota, recurring lifetime policy, and schedule-creation audit semantics. The issue is recent, security-sensitive, and the AS76 comments add concrete production context rather than stale noise.

## What This Changes

Request a Gateway-exposed scheduling API that lets agents create or modify limited recurring jobs with non-forgeable provenance, playbook allowlists, quotas, lifetime controls, and audit entries.

## Best Possible Solution

Create a short core security/design record first, then add an additive Gateway scheduling capability that derives creator identity and timestamp from authenticated request context, accepts only registered playbook identifiers plus schema-validated params, enforces quota and lifetime policy, records schedule create/update audit events, and keeps the unrestricted operator cron CLI/admin RPC separate.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The high-confidence verification path is source and docs inspection: current main has admin cron.add and an owner-only cron tool, but no constrained schedule_task/provenance/playbook/quota/lifetime contract.

## Solution Assessment

Unclear until maintainers choose the product and security model. The requested direction is safer than exposing the full cron surface to agents, but the maintainable solution should be a designed Gateway/cron/audit capability rather than a one-off schema patch.
