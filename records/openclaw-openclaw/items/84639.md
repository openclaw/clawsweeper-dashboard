## Summary

Keep open: current main has the realtime relay and `talk.session.submitToolResult`, but it does not expose a session-bound speech-injection RPC. The request is a plausible additive Gateway protocol feature, not something already implemented or safely closeable as ClawHub-only work.

## What This Changes

Add a Gateway `talk.session.speak`-style method so async channel completions can speak through an existing realtime relay session without keeping a tool call open.

## Best Possible Solution

Design an additive Gateway protocol method that is limited to owned realtime relay sessions, uses `operator.write`, validates bounded text, calls a provider-neutral relay text primitive, and emits metadata-only Talk observability events.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that the requested `talk.session.speak`-style method is absent on current main.

## Solution Assessment

Unclear pending maintainer design: the requested direction is plausible because `talk.speak` is a separate TTS path and the provider bridge has a lower-level text primitive. The maintainable solution should land through the typed Gateway protocol, relay runtime, descriptors, docs, and focused tests rather than as a channel-local workaround.
