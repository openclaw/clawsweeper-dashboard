## Summary

Keep open: current main has the unified realtime relay and `talk.session.submitToolResult`, but no session-bound speech-injection RPC exists. This is a plausible additive Gateway/Talk API feature that needs maintainer protocol, authorization, logging/redaction, and realtime turn-semantics review rather than cleanup closure.

## What This Changes

The issue requests an official Gateway method, such as `talk.session.speak`, to speak bounded completion text through an existing realtime relay session without keeping a pending tool call open.

## Best Possible Solution

After maintainer API approval, add a typed Gateway protocol method limited to owned realtime relay sessions, using `operator.write`, bounded text validation, a provider-neutral relay text primitive, metadata-only Talk events, docs, and focused protocol/handler/relay tests.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that the requested `talk.session.speak`-style method is absent on current main.

## Solution Assessment

Unclear pending maintainer design: the direction is plausible because `talk.speak` is a separate TTS path and provider bridges already have lower-level text primitives. The maintainable solution should land through the typed Gateway protocol, relay runtime, descriptors, docs, and tests rather than as a channel-local workaround.
