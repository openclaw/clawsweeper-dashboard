Keep open: current main has unified Talk realtime relay sessions and tool-result continuation, but it still has no session-bound speech-injection Gateway RPC. The request is a plausible additive public protocol feature that needs maintainer API, authorization, logging, and realtime turn-semantics review rather than cleanup closure.

## What This Changes

The issue requests an official `talk.session.speak`-style Gateway method for speaking bounded async completion text through an existing realtime relay session using the active voice/session.

## Best Possible Solution

Design an additive typed Gateway protocol method limited to owned realtime relay sessions, using `operator.write`, bounded text validation, a provider-neutral bridge text primitive, metadata-only Talk events, docs, and focused protocol/relay tests.

## Solution Assessment

Unclear pending maintainer design: the proposed direction is plausible because `talk.speak` is a separate speech-provider path and the bridge has provider text primitives, but the durable solution should be an additive typed Gateway protocol change rather than a channel-local workaround.
