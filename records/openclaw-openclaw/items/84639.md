## Summary

Keep open: current main and the latest shipped tag still expose realtime relay audio, cancel, close, and tool-result controls, but not a session-bound speech-injection RPC. The request is a plausible additive Gateway/Talk protocol feature and overlaps only loosely with the open active-run voice-control PR, so it needs maintainer API/security-shape review rather than cleanup closure.

## What This Changes

Add an official Gateway method for injecting bounded text into an existing realtime relay session so async completions can be spoken by the active realtime voice.

## Best Possible Solution

Design an additive typed Gateway method limited to owned realtime relay sessions, using `operator.write`, bounded text validation, provider-neutral bridge injection, and metadata-only Talk observability events coordinated with active-run voice-control semantics.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken existing behavior report. Source and docs inspection give high confidence that the requested `talk.session.speak`-style method is absent on current main and in `v2026.5.18`.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible because `talk.speak` is a separate TTS path and provider bridges already have a lower-level text primitive, but the maintainable implementation should land through the typed Gateway protocol, relay runtime, descriptors, docs, and focused tests rather than a channel-local workaround.

- Adjacent active-run voice-control work may overlap in turn semantics and queued follow-up behavior, so maintainers should align the two designs before accepting a separate session-speech primitive.
