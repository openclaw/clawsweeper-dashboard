## Summary

Keep open: current main still lacks a `talk.session.speak`-style Gateway RPC, and the request is a plausible additive realtime Talk API change that needs maintainer design rather than cleanup closure.

## What This Changes

The issue requests a Gateway method to inject short speech into an active realtime relay session for async channel completions without keeping a tool call open.

## Best Possible Solution

Design an additive Gateway protocol method limited to owned realtime relay sessions, using `operator.write`, bounded text validation, a provider-neutral relay text primitive, and metadata-only Talk observability while coordinating with active-run voice control work.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that the requested `talk.session.speak`-style method is absent on current main.

## Solution Assessment

Unclear pending maintainer design: the requested direction is technically plausible because the provider bridge has a text primitive and `talk.speak` is a separate TTS path. The durable solution should land through the typed Gateway protocol, relay runtime, descriptors, docs, and focused tests rather than as a channel-local workaround.
