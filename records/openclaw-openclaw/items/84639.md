## Summary

Keep open: current main and the latest release still expose realtime relay creation, audio, cancellation, and `talk.session.submitToolResult`, plus separate `talk.speak`, but no session-bound speech-injection RPC. This is a plausible new Gateway protocol feature that needs maintainer API and security-shape review, not something already implemented or safely moved to ClawHub.

## What This Changes

The issue requests an additive Gateway API for asking an existing realtime relay Talk session to speak bounded completion text after async channel work finishes.

## Best Possible Solution

Design an additive Gateway method limited to owned realtime relay sessions, using `operator.write`, bounded text validation, a provider-neutral relay text primitive, metadata-only Talk observability events, docs, and focused protocol/handler tests.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken existing behavior report. Source inspection and GitHub search give high confidence that the requested session-bound speak method is absent on current main and the latest release.

## Solution Assessment

Unclear pending maintainer API design: the requested direction is plausible because `talk.speak` is a separate TTS path and realtime providers already have lower-level text primitives. The maintainable path should land through the typed Gateway protocol, relay runtime, descriptors, docs, and focused tests rather than a channel-local workaround.
