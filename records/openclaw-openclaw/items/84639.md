## Summary

Keep open: current main has realtime relay sessions, `talk.session.submitToolResult`, and generic `talk.speak`, but it still lacks a session-bound speech-injection Gateway RPC. This is a plausible additive public Gateway/Talk feature and needs maintainer API review rather than cleanup closure.

## What This Changes

The issue requests an official `talk.session.speak`-style Gateway method that can inject bounded text into an existing realtime relay session so async completions use the active realtime voice.

## Best Possible Solution

Design an additive typed Gateway method limited to owned realtime relay sessions, using `operator.write`, bounded text validation, a provider-neutral relay text primitive, and metadata-only Talk observability events.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that the requested `talk.session.speak`-style method is absent on current main.

## Solution Assessment

Unclear pending maintainer design: the requested direction is plausible because `talk.speak` is a separate TTS path and the realtime bridge has a lower-level text primitive. The maintainable path should land through the typed Gateway protocol, relay runtime, descriptors, docs, and focused tests rather than a channel-local workaround.
