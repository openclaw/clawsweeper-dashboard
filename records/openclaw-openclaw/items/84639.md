## Summary

Keep open: current main and the latest release still expose `talk.session.submitToolResult` and `talk.speak`, but not a session-bound speech-injection Gateway RPC. This is a plausible additive core Gateway/Talk protocol feature, not stale cleanup, not a duplicate, and not safely movable to ClawHub because the missing piece is the core session API itself.

## What This Changes

The issue requests an official `talk.session.speak`-style Gateway method that can inject bounded text into an existing realtime relay session for async channel completion speech.

## Best Possible Solution

Design an additive Gateway protocol method limited to owned realtime relay sessions, authorized like `talk.session.submitToolResult`, with bounded text validation, provider-neutral relay text injection, metadata-only Talk events, docs, and focused protocol/relay tests.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than broken existing behavior. Source inspection and the latest release check give high confidence that the requested session-bound speech RPC is absent.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible because `talk.speak` is a separate TTS provider path and realtime bridges already have `sendUserMessage`, but the maintainable fix needs a typed protocol/API decision rather than a channel-local workaround.
