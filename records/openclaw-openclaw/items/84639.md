## Summary

Keep open: current main has realtime relay controls and `talk.session.submitToolResult`, but no session-bound speech-injection RPC. This is a new public Gateway protocol capability that needs maintainer API and security-shape review.

## What This Changes

Request an additive Gateway RPC that lets a channel ask an existing realtime relay session to speak bounded completion text through the active realtime voice without holding a tool call open.

## Best Possible Solution

Design an additive typed Gateway session method for realtime-relay sessions that enforces owner/write scope, validates bounded text, calls a provider-neutral relay text primitive, emits metadata-only observability, and lands with docs plus focused Gateway/protocol/relay tests.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that the requested `talk.session.speak`-style method is absent on current main.

## Solution Assessment

Unclear pending maintainer design: the requested direction is plausible because `talk.speak` is a separate TTS provider path and the realtime bridge has a lower-level text primitive. The maintainable solution should land through the typed Gateway protocol, relay runtime, method descriptors, docs, and focused tests rather than as a channel-local workaround.
