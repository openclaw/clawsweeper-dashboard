## What This Changes

The issue requests a new `talk.session.speak`-style Gateway method for injecting bounded completion text into an existing realtime relay session without keeping a tool call open.

## Best Possible Solution

Design an additive typed Gateway protocol method limited to owned realtime relay sessions, with `operator.write`, bounded text validation, provider-neutral relay support, metadata-only Talk observability, docs, and focused tests.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken existing behavior report. Source inspection gives high confidence that the requested `talk.session.speak`-style method is absent on current main.

## Solution Assessment

Unclear pending maintainer design: the requested direction is plausible because `talk.speak` is a separate TTS path and the realtime bridge already has a lower-level text primitive. The maintainable path should be a typed Gateway protocol and relay-runtime change, not a channel-local workaround.
