## Summary

Keep open: current main and the latest release still lack a session-bound realtime relay speech-injection Gateway method, and the request remains a core Talk/Gateway API and security-shape decision rather than an already-implemented behavior.

## What This Changes

The issue requests an official Gateway method to speak bounded text through an existing realtime relay Talk session without keeping a pending tool call open.

## Best Possible Solution

Design an additive, bounded, redacted session-speech RPC at the Talk session/realtime-relay boundary, or explicitly document that `submitToolResult`, `steer`, and `talk.speak` are the supported paths.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection and the reporter's Gateway probe both confirm the requested `talk.session.speak` method is absent on current main and the latest release.

## Solution Assessment

Unclear until maintainer API decision. The requested shape is plausible because the realtime bridge has `sendUserMessage(text)`, but the permanent solution must define a typed Gateway protocol, relay-only behavior, authorization, text limits, docs, tests, and redacted event semantics.
