## Summary

Keep open: current main and v2026.6.11 still do not expose a session-bound realtime speech-injection Gateway method, and the adjacent `talk.session.submitToolResult`, `talk.session.steer`, and `talk.speak` surfaces do not satisfy the requested no-pending-tool-call behavior.

## What This Changes

Request to add a `talk.session.speak`-style Gateway RPC that speaks bounded text through an active realtime relay session without requiring a pending tool call.

## Best Possible Solution

Expose an additive, bounded realtime-relay session speech request only if maintainers approve the trust and redaction semantics, with protocol schema, handler, metadata-only Talk event, docs, and focused tests.

## Reproduction Assessment

Yes. A caller can attempt `talk.session.speak` and current main should still reject it as unknown because the method registry, descriptor list, protocol schemas, and handlers do not include that method.

## Solution Assessment

Unclear pending maintainer decision. The requested direction is plausible, but the best permanent fix must be a bounded session speech contract rather than repurposing `talk.speak`, `talk.session.steer`, or a tool-result path.
