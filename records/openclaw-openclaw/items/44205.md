## Summary

Keep this PR open, but it is not merge-ready. The stale same-child fallback problem is plausible, but the current patch changes the fallback from aggregating distinct completed child outputs to returning only the newest child, and it applies a broad cron interim-message heuristic to arbitrary child results; both can drop user-visible cron output.

## What This Changes

The PR changes cron isolated-agent descendant fallback behavior to return a single freshest settled descendant reply, updates regression coverage for stale-vs-fresh descendants, and adds a small generic singleton type assertion.

## Best Possible Solution

Keep the useful same-child freshness direction, but revise the fallback to keep one latest usable reply per distinct child and aggregate those child results. The patch should also avoid applying the broad cron acknowledgement heuristic to arbitrary descendant output, add focused coverage for stale same-child history and multi-child fan-out, and include the required changelog entry.

## Reproduction Assessment

Yes. Current main has a focused unit test for readDescendantSubagentFallbackReply that reproduces multi-child fan-out aggregation; the PR changes that case to expect only the freshest child, which demonstrates the output drop. A second static repro is any short legitimate child result containing an interim hint such as "retrying now", which the PR would filter out.

## Solution Assessment

No. Selecting the latest run per child is a useful part of the fix, but returning after the first usable child reply and using the broad cron interim filter on child outputs is not the narrowest maintainable solution; aggregating distinct child outputs with a stricter acknowledgement check is safer.
