## Summary

Keep this PR open, but it is not merge-ready. Current main still treats cron descendant fallback output as user-visible aggregation across distinct child subagents, while the PR changes that fallback to return the first usable freshest child reply and adds a broad interim-message filter that can discard legitimate short child results.

## What This Changes

The PR changes cron isolated-agent descendant fallback behavior to return a single freshest settled descendant reply instead of joining multiple child replies, adds regression coverage, and includes a small shared singleton type cast.

## Best Possible Solution

Preserve run-scoped fallback aggregation by selecting the latest usable reply per distinct child session, filtering stale same-child history by run timing, and only suppressing true acknowledgement replies with narrowly targeted checks. Add regression coverage for stale same-child history, multi-child fan-out aggregation, silent replies, and child result text that contains interim-like substrings.

## Reproduction Assessment

Yes. The current main test for readDescendantSubagentFallbackReply reproduces the multi-child case by expecting two distinct descendant outputs to be joined; the PR changes that same scenario to expect only the freshest output. A second focused repro is a short real child result containing a hint such as "retrying now", which the PR would skip through isLikelyInterimCronMessage.

## Solution Assessment

No. The PR’s latest-per-child dedupe direction is useful, but returning only the first usable child reply and applying the broad cron interim filter to child output is not the narrowest maintainable fix; aggregating one usable latest reply per child with a stricter acknowledgement filter is safer.
