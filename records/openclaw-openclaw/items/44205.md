## Summary

Keep this PR open, but it is not merge-ready. Current main still treats descendant fallback as aggregation across distinct child subagents, while the PR returns only the first freshest usable child reply and applies a broad cron-interim substring filter to arbitrary child output.

## What This Changes

The PR changes cron isolated-agent descendant fallback behavior to return a single freshest settled descendant reply, adds regression coverage for stale-vs-fresh descendants, and includes a small shared singleton type assertion.

## Best Possible Solution

Revise the patch to keep one latest usable reply per distinct child session, filter stale same-child history by the current run boundary, and only suppress true acknowledgement replies with a narrower descendant-output check. Add or preserve coverage for stale same-child history, multi-child fan-out aggregation, silent replies, and short legitimate child output containing interim-like words.

## Reproduction Assessment

Yes. The current main test for readDescendantSubagentFallbackReply reproduces the multi-child fan-out case by expecting two distinct descendant outputs to be joined, and the PR changes that scenario to expect only the freshest output. A second focused repro is a short real child result containing a phrase such as "retrying now", which the PR would skip through isLikelyInterimCronMessage.

## Solution Assessment

No. The latest-per-child dedupe direction is useful, but returning only the first usable child reply and applying the broad cron interim filter to child output is not the narrowest maintainable fix; aggregation plus a stricter acknowledgement filter is safer.
