## Summary

Keep open: current main still has no exact `agentId` filter for `cron.list`, and #77188 plus #77191 are open closing PRs that should be reviewed or deduplicated before this issue is resolved.

## What This Changes

Request adding `agentId` filtering to cron job listing surfaces, including agent-session scoping and a CLI `--agent` flag for multi-agent installs.

## Best Possible Solution

Land one canonical additive `agentId` filter through the cron protocol/service and chosen user surfaces, with an explicit all-jobs path and a documented default visibility policy.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main confirms `cron.list` cannot accept or apply an exact `agentId` filter today.

## Solution Assessment

Unclear until maintainers choose the canonical PR and default visibility policy. The optional server-side filter is narrow, but automatic agent-session scoping changes multi-agent/admin behavior.
