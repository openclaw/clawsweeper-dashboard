## Summary

Keep open: current main still lacks exact `agentId` filtering for `cron.list`, and the requested default agent-session scoping is a new cron protocol/tool/CLI capability plus visibility-policy decision.

## What This Changes

The issue requests `cron list` support for filtering jobs by `agentId`, including CLI/tool support and automatic session-agent scoping for agent calls.

## Best Possible Solution

Add an explicit server-side `agentId` filter through the cron protocol, service, CLI, and tool, then deliberately decide whether agent-session calls default to scoped results or require an explicit filter.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main shows `cron.list` cannot accept an exact `agentId` filter and `listPage` only uses `agentId` in fuzzy query text.

## Solution Assessment

Unclear until maintainers choose the default visibility policy. The explicit filter is narrow, but automatic session scoping and mutation-scope expectations change multi-agent/admin behavior.
