## Summary

Keep open: current main still lacks an exact `agentId` filter for `cron.list`, and open PRs #77188 and #77191 explicitly target this issue, so the issue should stay open until one implementation lands or maintainers choose a replacement path.

## What This Changes

The issue requests an optional `agentId` filter for `cron list` in multi-agent setups, including tool-session scoping and a CLI `--agent` option.

## Best Possible Solution

Ship an additive exact `agentId` filter through the cron protocol, service, tests, and user-facing surfaces, then deliberately choose whether agent tool calls should default to scoped results.

## Reproduction Assessment

Not applicable as a strict bug reproduction, but source inspection gives a high-confidence current-main path: `cron.list` cannot accept or apply an exact `agentId` filter today.

## Solution Assessment

Unclear until maintainers choose the default visibility policy. An optional exact server-side filter is narrow and additive, but automatic agent-session scoping changes multi-agent/admin behavior.
