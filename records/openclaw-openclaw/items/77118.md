## Summary

Keep open: current main still has no exact `agentId` filter for `cron.list`, and the requested default agent-session scoping is a new user-facing policy change rather than a broken documented contract.

## What This Changes

Request adding exact `agentId` filtering to `cron list` across the tool, CLI, gateway protocol, server handler, and cron service, with agent-session calls defaulting to the caller while CLI calls remain all-jobs unless filtered.

## Best Possible Solution

Add an explicit exact `agentId` filter through the cron protocol, service, CLI, and tool, then document an intentional policy for whether agent-session tool calls default to scoped or all-jobs visibility.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main gives high confidence that `cron.list` cannot accept or apply an exact `agentId` filter today.

## Solution Assessment

Unclear until maintainers choose the default visibility policy. An optional exact server-side filter is narrow and maintainable, but automatic agent-session scoping changes multi-agent/admin behavior.
