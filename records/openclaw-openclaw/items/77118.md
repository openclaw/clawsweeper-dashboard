## Summary

Keep open: current main still lacks an exact `agentId` filter for cron listing, and two open PRs already reference this issue with closing syntax, so cleanup should wait for maintainer review or merge of one implementation path.

## What This Changes

Request adds optional `agentId` filtering to `cron.list` and `openclaw cron list`, with agent-session calls defaulting to the calling agent in multi-agent setups.

## Best Possible Solution

Land one additive implementation that threads an exact `agentId` filter through protocol, service, and server, then deliberately choose whether tool and CLI behavior should auto-scope or remain explicit.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for source-level verification: current main has no accepted or forwarded `agentId` parameter for `cron.list`, and the CLI/tool list paths do not provide one.

## Solution Assessment

Unclear until maintainers choose the visibility policy. An exact optional filter is a narrow additive foundation, but automatic agent-session scoping changes multi-agent/admin behavior and should be decided while reviewing #77188 or #77191.
