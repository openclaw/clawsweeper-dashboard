## Summary

Keep open: current main still lacks an exact `agentId` filter for `cron.list`, and this issue is now paired with open closing PRs #77188 and #77191 that should be reviewed before cleanup.

## What This Changes

Add optional `agentId` filtering to `cron list`/`cron.list`, with possible agent-session default scoping for multi-agent setups.

## Best Possible Solution

Review and land one coherent fix from the linked PRs: an additive exact `agentId` filter in the cron protocol/service, plus an explicit maintainer decision on CLI `--agent` and agent-tool default scoping.

## Reproduction Assessment

Not applicable as a bug reproduction, but source inspection gives a high-confidence current-main capability check: `cron.list` cannot accept or apply an exact `agentId` filter today.

## Solution Assessment

Unclear until maintainers choose the visibility policy. The optional server-side filter is narrow and additive, while automatic agent-session scoping changes default multi-agent/admin behavior.
