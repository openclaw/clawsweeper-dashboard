## Summary

Current main now solves the observable fallback failure by moving the retry loop into the image describe CLI/tool-wrapper path, so a skill invoking `openclaw infer image describe` or `describe-many --model` no longer has to rely on the agent to manually choose the configured fallback after a model-call 429.

## What This Changes

The issue reports that a custom image-comparison skill's instructed image-model fallback was skipped after the primary OpenRouter image model returned a 429.

## Best Possible Solution

Keep the current main CLI/tool-wrapper fallback behavior and track any future fail-closed skill-contract design only as a separate, concrete product proposal.

## Reproduction Assessment

No for current main: the v2026.6.10 trajectory supports the old failure, but current source and regression coverage now route explicit image describe model-call failures through configured image fallbacks.

## Solution Assessment

Yes for the central bug: moving retry into `openclaw infer image describe` and `describe-many` is the narrow maintainable fix because it removes the agent's recovery-step judgment from the reported path.
