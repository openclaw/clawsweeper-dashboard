## Summary

Keep open: current main has built-in image-model fallback paths, but it does not solve the reported gap where a custom skill’s prose-prescribed recovery path can be skipped after a provider 429. The remaining work needs maintainer direction on whether this belongs in skill contracts, image CLI/tool behavior, diagnostics, or docs.

## What This Changes

The issue requests stronger enforcement or diagnostics when a custom skill prescribes image-model primary/fallback recovery and the agent skips that recovery after a provider 429.

## Best Possible Solution

Pick one owner-approved path: either expose an enforceable image comparison/fallback tool or skill contract, or document that skills should use the existing no-`--model` image describe path when they want OpenClaw-owned fallback behavior.

## Reproduction Assessment

No high-confidence current-main reproduction was established. The attached trajectory strongly supports the v2026.6.10 behavior, and source inspection shows current skills are prompt instructions while fallback enforcement exists only in OpenClaw-owned image/model paths.

## Solution Assessment

Unclear: the issue points at a valid reliability gap, but the best fix needs product direction. Reusing the existing image fallback helper or adding a typed skill/tool contract would be cleaner than relying on the agent to infer mandatory control flow from prose.
