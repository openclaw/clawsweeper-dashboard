## What This Changes

The PR adds a lifecycle-generation guard to `markRestartAbortedMainSessions` and a regression test so completed current-generation sessions are not reopened by expired abort-controller candidates.

## Best Possible Solution

Land the narrow lifecycle-generation guard after maintainer session-state review and required checks, while leaving the linked issue open until this PR merges or maintainers choose a broader recovery/reset-policy design.

## Reproduction Assessment

Yes at source level: current main can match a completed session when a current-generation candidate has a later observedAt. I did not run a live Telegram replay in this read-only review.

## Solution Assessment

Yes: lifecycleGeneration is the narrow discriminator between current maintenance candidates and stale pre-restart runs. A status-only guard would be too broad because existing stale-generation recovery coverage expects some non-running rows to be markable.
