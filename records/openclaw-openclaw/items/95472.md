## What This Changes

The PR changes `markRestartAbortedMainSessions` so completed sessions are not reopened by current-generation observed active runs, and adds a focused regression test.

## Best Possible Solution

Land the narrow guard after redacted runtime/log proof is added or maintainers explicitly accept unit-boundary proof, while keeping the linked issue open for any broader reset-policy follow-up.

## Reproduction Assessment

Yes at source level: current main's timing branch reopens a completed row when `updatedAt < observedAt`, and the PR's new test models that exact predicate. I did not run tests or live Telegram proof because this review is read-only.

## Solution Assessment

Yes, this is the narrowest maintainable fix I found: it limits only the timing fallback to stale lifecycle generations while preserving running rows and no-observedAt stale recovery candidates. A blanket `entry.status === "running"` fix would break the existing pre-restart queued-run cases covered by adjacent tests.
