## Summary

Keep open. Current main still has a source-level gap for Codex DM turns on Mattermost: Codex defaults source replies to message-tool delivery, the shared dispatcher suppresses progress callbacks unless the channel opts in, and Mattermost does not set that opt-in even though it has draft progress handlers.

## What This Changes

Request Mattermost progress draft wiring for Codex app-server tool activity when `streaming.mode: "progress"` is enabled.

## Best Possible Solution

Let Mattermost opt into suppressed-source progress callbacks only while an editable draft/progress stream owns the turn, and add a regression test for Codex message-tool-only tool events reaching the Mattermost draft handlers.

## Reproduction Assessment

Yes at source level, but not with a live Mattermost server in this read-only pass. A Codex DM defaults to message-tool-only source delivery, and Mattermost's progress callbacks are wrapped but not invoked because it does not set the suppressed-source progress opt-in.

## Solution Assessment

Yes. The narrow maintainable fix is to add the same scoped progress-callback opt-in that Telegram uses to the Mattermost draft path, with coverage that prevents duplicate or standalone progress delivery.

Add a scoped `allowProgressCallbacksWhenSourceDeliverySuppressed` opt-in to the Mattermost draft/progress reply options only when an editable draft/progress stream is active, mirroring Telegram's intent without changing final visible-reply policy or enabling standalone progress chatter. Add focused regression coverage proving Mattermost `streaming.mode: "progress"` forwards `onToolStart`/`onItemEvent` callbacks when `sourceReplyDeliveryMode: "message_tool_only"`, and that callbacks remain disabled when draft streaming/tool progress is off.
