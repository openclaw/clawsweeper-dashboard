## Summary

Keep this open. Current main already has broad tool-use/tool-result pairing repair, but the remaining user-visible failure is still present: Anthropic JSON `invalid_request_error` payloads can still format as `LLM request rejected: <provider message>` instead of a recovery-oriented session-history message.

## What This Changes

The issue asks OpenClaw to recover gracefully from Anthropic orphaned `tool_result` validation errors instead of posting the raw provider error into Discord.

## Best Possible Solution

Keep the existing transcript pairing repair, then add a focused Anthropic orphaned-tool-result classifier that retries only at a safe replay boundary or returns friendly session-recovery text instead of raw provider validation output.

## Reproduction Assessment

Yes for the remaining UX bug: pass the reported JSON Anthropic `invalid_request_error` through `formatAssistantErrorText` and current source shows it returns `LLM request rejected: ...`. No end-to-end current-main reproduction was established for the original intermittent high-tool-volume Discord sequence.

## Solution Assessment

Yes. The narrowest maintainable fix is a classifier and friendly recovery/retry path near provider error formatting and the embedded retry boundary, while leaving the existing pair-aware repair in place.
