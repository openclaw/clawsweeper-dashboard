## Summary

Keep this issue open. Current main has strong tool_use/tool_result pairing repair in replay, truncation, and compaction paths, so the broad compaction root cause from the reported 2026.3.13 build appears largely covered; however, the remaining user-visible failure is still actionable because JSON-wrapped Anthropic `invalid_request_error` payloads can still be formatted as `LLM request rejected: <provider message>` without a specific `unexpected tool_use_id` recovery message or regression test.

## What This Changes

The issue requests recovery from Anthropic `unexpected tool_use_id` validation failures during high-tool-call sessions, including sanitized retry or a user-friendly session-history error instead of posting the raw provider message to Discord.

## Best Possible Solution

Preserve the existing transcript pairing repair, then add a narrow Anthropic `unexpected tool_use_id` / orphaned `tool_result` classifier in the provider error path. When the current attempt can be replayed without duplicating side effects, re-sanitize replay history and retry once; otherwise return a friendly session-history recovery message instead of the raw provider validation string, with focused regression coverage.

## Reproduction Assessment

Partially yes. The remaining UX seam has a high-confidence unit-level reproduction by passing the reported JSON `invalid_request_error` text through `formatAssistantErrorText`, which currently falls to `LLM request rejected: ...`; the original intermittent high-tool-call Discord compaction scenario still lacks a current-main transcript to reproduce end to end.

## Solution Assessment

Yes, for the remaining current-main behavior the narrowest maintainable fix is an error classifier/friendly recovery path near `formatAssistantErrorText` and the embedded runner retry boundary, not another broad compaction rewrite. The existing pair-aware repair should stay in place and be treated as already solving the older root cause tracked by #51423.
