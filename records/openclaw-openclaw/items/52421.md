## Summary

Keep this open. Current main has strong pairing repair in replay, truncation, and compaction paths, but the remaining user-visible Discord failure is still present because JSON-wrapped Anthropic `invalid_request_error` payloads can still be formatted as `LLM request rejected: <provider message>` with no specific `unexpected tool_use_id` classifier or friendly recovery text.

## What This Changes

The issue asks OpenClaw to recover from Anthropic orphaned `tool_result` / `unexpected tool_use_id` failures and avoid posting the raw provider validation string into Discord.

## Best Possible Solution

Keep the existing transcript pairing repair, then add a focused Anthropic orphaned-tool-result classifier that re-sanitizes/retries only at a safe replay boundary and otherwise returns a friendly session-history recovery message instead of the raw provider validation text.

## Reproduction Assessment

Partially yes. The remaining UX bug is source-reproducible by passing the reported JSON `invalid_request_error` through `formatAssistantErrorText`, which currently returns `LLM request rejected: ...`; the original intermittent high-tool-call Discord/compaction path still lacks an end-to-end current-main transcript.

## Solution Assessment

Yes. The narrowest maintainable fix is a classifier and friendly recovery/retry path near provider error formatting and the embedded retry boundary, while leaving the existing pair-aware repair in place rather than rewriting compaction again.
