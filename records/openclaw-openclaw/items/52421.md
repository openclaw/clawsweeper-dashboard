## Summary

Keep this open. Current main has substantial tool-use/tool-result pairing repair, but the remaining user-visible bug still exists: a JSON Anthropic `invalid_request_error` with the reported orphaned `tool_result` wording can still format as `LLM request rejected: <provider message>` and flow into lifecycle/channel error text.

## What This Changes

The issue asks OpenClaw to prevent Anthropic orphaned `tool_result` validation errors from disrupting sessions and surfacing raw provider text to Discord.

## Best Possible Solution

Keep the existing transcript pairing repair, then add a focused orphaned-tool-result classifier and friendly session-recovery message, with retry only at a safe replay boundary if the run has not committed side effects.

## Reproduction Assessment

Yes for the remaining UX bug by source inspection: a JSON `invalid_request_error` carrying the reported Anthropic message still passes through `formatAssistantErrorText` as `LLM request rejected: ...` and then into lifecycle error text. ## Solution Assessment

Yes. The narrowest maintainable fix is to classify the specific orphaned `tool_result` validation shape near provider error formatting/replay-invalid handling while leaving the already-implemented compaction and transcript pairing repairs intact.
