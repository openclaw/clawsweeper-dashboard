## Summary

Keep open. Current main has a partial `waiting` spinner for the pre-token gap, but the TUI still uses free-form activity strings and does not expose the requested Idle/Generating/Working/Waiting/Reconnecting/Disconnected state contract.

## What This Changes

The issue asks for distinct TUI session state labels so connected idle, token generation, non-streaming work, yielded subagent waiting, reconnecting, and disconnected states are no longer conflated.

## Best Possible Solution

Define a typed TUI activity-state contract that derives visible activity from run lifecycle, deltas, tool work, yielded waits, and reconnects while keeping connection status separate.

## Reproduction Assessment

Yes, at source level. Current main still routes TUI activity through free-form strings and lacks yielded/liveness metadata in the visible TUI chat event contract, so the requested distinctions cannot be represented consistently.

## Solution Assessment

Unclear. The direction is sound, but the maintainable fix is a typed state contract plus maintainer-approved wording, not a narrow renderer string rename.
