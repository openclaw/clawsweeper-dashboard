## Summary

Keep this open. Current `main` partially improves the pre-token gap by showing `waiting`, but it still does not implement the requested typed TUI state model or distinguish yielded subagent waits, tool/work activity, generating, reconnecting, and disconnected states cleanly.

## What This Changes

The issue requests clearer TUI session status labels that distinguish idle, token generation, tool/work activity, yielded subagent waits, reconnecting, and disconnected states.

## Best Possible Solution

Define a typed TUI activity-state contract and map existing run metadata into it so idle is reserved for no active work, generation and tool work are separate, yielded parent waits remain visible, and reconnecting/disconnected are connection states.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the remaining gap: a yielded final or paused/liveness result is not represented in the TUI event type and final handling resets active runs to `idle`; tool work similarly stays `streaming`/`running` rather than a distinct Working state.

## Solution Assessment

Unclear. The requested direction is sound, but the best maintainable fix is a typed status model carried through gateway/TUI contracts, and maintainers should choose the final user-facing vocabulary before changing the visible TUI behavior.
