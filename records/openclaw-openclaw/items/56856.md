## Summary

Keep this open. Current main still has the same TUI status gap: deltas are labeled `streaming`, lifecycle work is labeled `running`, post-send waiting is only an acknowledgement state, yielded/liveness metadata exists below the TUI but is not mapped to a visible waiting state, and disconnect recovery does not expose a distinct reconnecting label.

## What This Changes

Request to replace the TUI's ambiguous connected/streaming status with distinct idle, generating, working, waiting, reconnecting, and disconnected labels.

## Best Possible Solution

Introduce a small typed TUI activity-state model and feed it from existing gateway chat/agent metadata: idle when connected with no active run, generating on model token deltas, working on lifecycle/tool execution, waiting when a run yields or is paused for subagent completion, reconnecting while recovery is underway, and disconnected when the gateway is unreachable. Add focused TUI handler coverage and update the user-facing TUI docs/changelog for the new labels.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available by simulating a chat delta followed by a yielded terminal lifecycle/chat final: current TUI handling marks the delta as `streaming` and then returns to `idle` because yielded/liveness metadata is not represented in the TUI state mapping.

## Solution Assessment

No. Current main is not the best solution yet; the narrow maintainable fix is to propagate existing yielded/paused metadata through the chat event or session status path and map it through an explicit TUI status state, rather than only renaming free-form strings.
