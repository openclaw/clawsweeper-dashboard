## Summary

Keep open. Current main and the latest stable release have the partial pre-token `waiting` spinner, but the TUI still uses free-form activity strings and does not provide the requested Idle/Generating/Working/Waiting/Reconnecting/Disconnected state contract; the related open subagent-progress report does not fully replace this TUI-specific state-label request.

## What This Changes

Request clearer TUI session-state labels that distinguish idle, active generation, tool work, yielded subagent waiting, reconnecting, and disconnected states.

## Best Possible Solution

Define a typed TUI activity-state contract, map chat deltas, tool/lifecycle events, yielded/liveness metadata, and reconnects into it, and keep activity state separate from connection state.

## Reproduction Assessment

Yes, at source level. Current main can enter `waiting` after send, but deltas, tool events, lifecycle events, yielded metadata, and disconnect handling still flow through free-form strings rather than the requested typed state distinctions.

## Solution Assessment

Unclear. The direction is sound, but the maintainable solution is a typed state contract plus maintainer-approved wording, not a narrow renderer string rename.
