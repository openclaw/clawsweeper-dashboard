## Summary

Keep this open. Current main still uses the older TUI status vocabulary and does not surface yielded subagent waits, working/tool activity, or reconnecting as the distinct user-visible states requested in the issue.

## What This Changes

The issue asks for the TUI footer/status labels to distinguish connected idle sessions, active token generation, non-streaming work, yielded waits for subagents, reconnecting, and disconnected states.

## Best Possible Solution

Introduce a small typed TUI activity-state model and feed it from existing gateway chat/agent metadata: idle when connected with no active run, generating on token deltas, working during lifecycle/tool execution, waiting when a run yields or is paused for subagent completion, reconnecting during gateway recovery, and disconnected when unreachable. Add focused TUI handler coverage and update the public TUI docs and changelog for the user-facing labels.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available by simulating a chat delta followed by a yielded terminal lifecycle/chat final: current TUI handling marks the delta as `streaming` and then returns to `idle` because yielded/liveness metadata is not represented in the TUI state mapping.

## Solution Assessment

No. Current main is not the best solution yet; the narrow maintainable fix is to propagate existing yielded/paused metadata through the chat event or session status path and map it through an explicit TUI status state, rather than only renaming free-form strings.
