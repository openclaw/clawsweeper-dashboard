## Summary

Keep open. Current main still does not implement the requested TUI state model: the TUI displays raw `streaming`/`running`/`waiting`/`idle` activity strings, yielded subagent waits are not surfaced as a distinct TUI state, and reconnect handling still reports generic disconnected/reconnected states rather than a visible reconnecting state.

## What This Changes

The issue requests clearer TUI session-state labels so connected idle, token generation, yielded waiting, non-streaming work, reconnecting, and disconnected states are visually distinct.

## Best Possible Solution

Add a small typed TUI activity-state model and map it from gateway chat/lifecycle metadata: `Generating` for model deltas, `Working` for active lifecycle/tool execution without token deltas, `Waiting` for yielded or paused subagent waits, `Idle` for connected sessions with no active run, `Reconnecting...` during recovery, and `Disconnected` when the gateway is unreachable. Preserve backward-compatible protocol behavior by adding metadata rather than replacing existing chat fields, and cover the state transitions with focused TUI handler tests plus docs/changelog updates.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to feed `createEventHandlers` a chat delta followed by a lifecycle/chat final carrying yielded or paused metadata: current code sets `streaming` on the delta and then resets to `idle`, while generic disconnects also return idle instead of a reconnecting/disconnected activity state.

## Solution Assessment

Yes, the maintainable fix is a narrow TUI/gateway status-metadata change rather than a broad UI rewrite. The safer path is to introduce explicit state mapping and tests first, then adjust labels and docs, instead of only renaming the current free-form strings.
