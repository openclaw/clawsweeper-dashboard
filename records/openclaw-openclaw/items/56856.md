## Summary

Keep this open. Current main partially improves the slow pre-token gap by showing `waiting` after a sent chat run, but it still does not provide the requested typed TUI state model or clearly distinguish generation, tool work, yielded subagent waits, reconnecting, and disconnected states.

## What This Changes

The issue requests distinct TUI labels for Idle, Generating, Working, Waiting, Reconnecting, and Disconnected session states instead of one ambiguous connected/streaming label.

## Best Possible Solution

Define a typed TUI activity-state contract, carry yielded/liveness/tool metadata into it, reserve idle for no active work, and keep connection state separate from activity state.

## Reproduction Assessment

Yes, at source level. A sent chat can enter `waiting`, but deltas, tool/lifecycle events, yielded metadata, and disconnect handling still flow through free-form status strings rather than the requested state distinctions.

## Solution Assessment

Unclear. The requested direction is sound, but the maintainable fix is a small typed status contract across gateway/TUI events plus maintainer-approved wording, not only a renderer string rename.
