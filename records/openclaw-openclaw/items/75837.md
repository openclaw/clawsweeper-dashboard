## Summary

Keep open. The manifest change is directionally sound for bundled provider id `ollama`, but this PR currently advertises itself as closing #75357 even though the confirmed reproduction uses custom provider names that remain dependent on an explicit `compat.supportsUsageInStreaming` override. Merging as-is would prematurely close the broader custom-provider route gap.

## What This Changes

The PR adds an Ollama `openAICompletions.supportsStreamingUsage` manifest flag, expands OpenAI-completions compat tests, and adds a changelog entry for streaming usage token accounting.

## Best Possible Solution

Keep the Ollama manifest opt-in if maintainers want bundled `ollama` covered, but remove closing references and keep #75357 open for the custom-provider or unconditional-send decision unless the PR is broadened to cover that path with focused tests.

## Reproduction Assessment

Yes. #75357 includes curl evidence and follow-up comments showing custom provider names omit `stream_options.include_usage` until `compat.supportsUsageInStreaming: true` is set; current-main tests also preserve false for generic custom providers.

## Solution Assessment

No as written. The implementation is a narrow maintainable fix for bundled `ollama`, but it is not the best complete fix for the linked report because it leaves custom provider ids unresolved while claiming to close the issue.
