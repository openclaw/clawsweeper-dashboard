## Summary

Keep open: current main has an opt-in `skills.load.allowSymlinkTargets` escape hatch, but the reported default ClawHub managed-skill symlink pattern still rejects outside-root realpaths unless operators configure the target root; the open linked fix PR remains the implementation path.

## What This Changes

Load ClawHub-managed skills installed under `~/.openclaw/skills` even when symlink or junction realpaths resolve outside that managed root.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/59219 or an equivalent narrow source-scoped containment patch that lets managed ClawHub symlinks load by default while keeping untrusted roots and per-skill files fail-closed.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a managed skill directory under `managedSkillsDir` whose realpath points outside that root and is not covered by `skills.load.allowSymlinkTargets`; current main sends it through `resolveContainedSkillPath()` and skips it.

## Solution Assessment

Yes, the linked source-scoped containment direction is the narrow maintainable fix. Current main's opt-in allowlist is useful but does not fully solve the default ClawHub managed install pattern described by this issue.
