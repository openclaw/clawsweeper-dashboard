## Summary

Keep open: current main still routes managed skills through the same realpath containment check that rejects escaped paths, and the issue is already paired with open fix PRs. This should wait for maintainers to land or reject the chosen PR rather than be closed as stale or implemented.

## What This Changes

The issue requests that managed skills installed via ClawHub under `~/.openclaw/skills` load correctly on WSL/symlink-backed layouts instead of being skipped as outside the configured root.

## Best Possible Solution

Land or replace one focused fix that scopes realpath containment to untrusted sources while preserving `SKILL.md` confinement and workspace/extra/bundled hardening, then close this issue and retire duplicate PRs.

## Reproduction Assessment

Yes. A managed skill directory under the configured managed root whose filesystem realpath resolves outside that root will pass through `resolveContainedSkillPath()` with `source: "openclaw-managed"` and be rejected on current main; this was verified by source inspection, not by running a mutating fixture.

## Solution Assessment

Unclear until maintainers choose between the open PRs. The best direction appears to be source-scoped containment rather than WSL-specific normalization, because the docs and discussion identify managed roots as a different trust class from workspace/extra roots.
