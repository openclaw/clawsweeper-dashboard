## What This Changes

This PR caps non-429 browser-control HTTP error body text reads in `fetchHttpJson` at 16 KiB using the shared SDK helper and adds a loopback streaming regression test.

## Best Possible Solution

Land the bounded read and regression test once required CI is green or explicitly dispositioned, keeping the existing 429 discard behavior and dispatcher path unchanged.

## Reproduction Assessment

Yes. Current main clearly reaches an unbounded `res.text()` for non-429 HTTP errors, and the PR body provides a loopback Node 22 before/after path showing the uncapped message and the 16 KiB capped result.

## Solution Assessment

Yes. Reusing the existing `readResponseTextLimited` SDK helper is the narrowest maintainable fix; dropping all body text would reduce useful diagnostics, while adding a browser-local reader would duplicate shared behavior.
