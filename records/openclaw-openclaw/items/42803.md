## Label Justifications

- `P1`: Delayed Feishu control/reset commands break an active channel workflow and can make session reset or status control unusable during long runs.
- `impact:session-state`: The issue concerns stopping, inspecting, and replacing active agent session state from Feishu.

## Summary

Keep open: current main and the latest release still route Feishu `/status` and `/new` onto the normal same-chat lane, so they can wait behind an active run; the open linked fork PR is the implementation path to review rather than a reason to close this issue.

## What This Changes

Restore Feishu text commands such as `/stop`, `/new`, and `/status` so command-only messages execute immediately during active agent runs.

## Best Possible Solution

Review or replace the linked Feishu fix with a current-main patch that routes safe immediate controls and reset commands onto the right non-blocking lane, with focused active-run concurrency tests.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.6` route `/status` and `/new` to the Feishu base chat key, and same-key tasks run FIFO. I did not run a live Feishu account during this read-only review.

## Solution Assessment

No, current main is only a partial fix. A separate Feishu lane is the right mechanism, but maintainers still need to choose the safe command whitelist and `/new` ordering semantics before landing the linked approach.
