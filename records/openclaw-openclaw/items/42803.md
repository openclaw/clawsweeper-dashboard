## Label Justifications

- `P1`: Delayed active-run controls make `/stop` ineffective on affected releases and current main still leaves reported `/status` and `/new` behavior unresolved.
- `impact:session-state`: The issue concerns aborting, inspecting, and replacing active agent session state.

## Summary

Current main only partially addresses the regression: Feishu abort-like text such as `/stop` gets an out-of-band control lane, but `/status` is explicitly tested on the normal per-chat lane and `/new` falls through to that same lane, so both can still wait behind an active run. The issue should remain open while maintainers decide safe reset-command ordering and reconcile the still-open linked fork PR rather than starting competing work.

## What This Changes

Restore immediate Feishu handling for `/stop`, `/new`, and `/status` while an agent run occupies the normal per-chat queue.

## Best Possible Solution

Keep Feishu’s channel-owned queue-key resolver, route abort and safe read-only controls to an immediate lane, and define cancellation-before-reset semantics for `/new` with focused active-run concurrency tests before landing the linked implementation.

## Reproduction Assessment

Yes at source level: current-main code and tests show `/status` uses the normal per-chat lane and `/new` falls through to it, while only abort-like text such as `/stop` gets the control lane. A live Feishu run was not executed during this read-only review.

## Solution Assessment

Not fully. A separate queue lane is the right bounded mechanism, but bypassing every recognized command is not proven safe; `/new` needs explicit cancellation and terminal-state ordering semantics, while read-only status commands can be handled more narrowly.
