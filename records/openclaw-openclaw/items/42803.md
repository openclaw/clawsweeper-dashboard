## Summary

Keep open. Current main still only partially addresses the Feishu regression: abort-like `/stop` can leave the base chat lane, but `/status` and `/new` still resolve to the normal per-chat queue, and an open linked PR remains the active implementation candidate.

## What This Changes

The issue requests restoring immediate Feishu handling for text commands such as `/stop`, `/new`, and `/status` during active agent runs.

## Best Possible Solution

Review and either land or replace the linked Feishu fix with a focused queue-classification change that covers the accepted command-only fast paths and preserves normal-message ordering.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main still awaits the Feishu sequential queue, and the key policy leaves `/status` and `/new` on the base chat key while only abort-like text and `/btw` move out of band.

## Solution Assessment

Unclear for the linked direct-dispatch approach as written. The safer maintainable direction is to extend Feishu’s sequential-key classification with authorization-aware queue and debounce tests after maintainers confirm the exact command scope.
