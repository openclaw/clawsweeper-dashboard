## Summary

Keep open. Current main and the latest shipped tag still route Feishu `/status` and `/new` text commands through the normal per-chat queue, while `/stop` has only a partial abort-lane fix and the full command-only fast-path behavior remains unresolved.

## What This Changes

The issue reports that Feishu text commands like `/stop`, `/new`, and `/status` are queued behind active agent runs instead of executing immediately.

## Best Possible Solution

Review, land, or replace the related Feishu fixes with a focused queue-classification or structured-command fast path that covers the accepted command-only controls while preserving normal-message ordering and authorization checks.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction: current main maps `/status` and `/new` to the base Feishu chat key and then awaits the sequential queue. I did not run a live Feishu/Lark WebSocket repro in this read-only review.

## Solution Assessment

Unclear until maintainers confirm the accepted out-of-band command scope. The maintainable fix is a narrow Feishu queue-key or structured-command fast path with authorization and ordering tests, not an unbounded direct-dispatch bypass.
