## Summary

Keep open. Current main only partially addresses the report: Feishu `/stop` now uses an out-of-band control lane, but `/status` and other command-only text commands still resolve to the normal per-chat lane, and there is an open linked PR that explicitly closes this issue.

## What This Changes

The issue requests restoring Feishu command-only text messages so `/stop`, `/new`, and `/status` bypass active-run queueing as documented.

## Best Possible Solution

Keep the issue open and resolve it by either landing an auth-aware Feishu sequential-key expansion for documented command-only fast paths or explicitly narrowing the docs and product scope for which commands may bypass the queue.

## Reproduction Assessment

Yes. Source inspection on current main shows Feishu still awaits same-key queue work, while tests assert `/status` remains on the base chat lane and only `/stop` moves to `:control`; `/new` follows the same base-lane path unless separately classified.

## Solution Assessment

Unclear for the open PR as-is. The maintainable direction is the current sequential-key seam with explicit command classification, authorization-aware behavior, and regression tests, rather than a broad direct-dispatch bypass.
