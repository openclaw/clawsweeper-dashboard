## Summary

Keep open. Current main only moves Feishu abort-like text to an out-of-band lane; `/status` and `/new` still resolve to the normal per-chat lane, and an open linked PR explicitly targets the remaining behavior.

## What This Changes

Restore immediate Feishu handling for command-only text controls such as `/stop`, `/new`, and `/status` while an agent run is active.

## Best Possible Solution

Resolve this by landing or reworking the linked Feishu PR around the current sequential-key seam, or explicitly narrow the docs/product promise to abort-only fast-path behavior.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main still serializes Feishu `/status` and `/new` on the base chat key while same-key work is awaited; only abort-like text gets the `:control` lane.

## Solution Assessment

Unclear until maintainers decide the command scope. The maintainable path is to classify authorized command-only Feishu fast paths in the sequential-key design, not to add a broad direct-dispatch bypass without reviewing ordering and auth semantics.
