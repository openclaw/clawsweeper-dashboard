## Summary

Keep open: current main and latest release still route Feishu `/status` and `/new` text commands through the normal per-chat queue while the docs promise a command-only fast path; `/stop` now has a control lane, so the remaining fix needs command-scope review alongside the linked PRs.

## What This Changes

The issue asks for Feishu text commands such as `/stop`, `/new`, and `/status` to bypass the per-chat queue during an active agent run.

## Best Possible Solution

Review, land, or replace the linked Feishu fix with a focused queue-classification change that covers the accepted command-only fast paths, preserves normal-message ordering, and adds authorization-aware regression coverage.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main still awaits the Feishu sequential queue, and `/status` plus `/new` remain on the base chat key while only abort-like text and `/btw` move out of band.

## Solution Assessment

Unclear for a broad direct-dispatch fix as written. The maintainable direction is to extend Feishu queue-key or structured fast-path classification for the accepted command-only set with authorization and ordering tests after maintainers confirm scope.
