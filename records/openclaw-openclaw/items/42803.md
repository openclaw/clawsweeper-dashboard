## Summary

Current main only partially addresses the Feishu queue bypass: `/stop` gets a separate control key, but `/status` and `/new` still resolve to the normal per-chat key and can remain blocked behind an active run.

## What This Changes

Feishu command-only text commands such as `/stop`, `/new`, and `/status` should bypass the per-chat queue during an active agent run.

## Best Possible Solution

Finish the Feishu-owned queue-key fix with regression coverage so immediate command-only messages use a non-blocked lane while ordinary same-chat messages remain serialized.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: start a normal Feishu turn that holds the base chat key, then send `/status` or `/new`; both resolve to the same base key and wait behind it. I did not run a live Feishu websocket repro in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable solution is in the Feishu plugin queue-key path plus tests, following the existing Telegram precedent for immediate command lanes. A core rewrite or new configuration option is not needed for the reported behavior.
