## Summary

Keep this open: current main only partially addresses the Feishu fast path by routing abort text such as `/stop` to a control lane, while `/status` and `/new` still resolve to the normal per-chat lane despite the documented command-only bypass behavior.

## What This Changes

The issue requests restoring Feishu text-command queue bypass so `/stop`, `/new`, and `/status` execute immediately during an active agent run.

## Best Possible Solution

Land a Feishu-owned queue-key fix that sends the reported command-only control messages to an out-of-band lane while preserving authorization, debounce, dedupe, and normal message serialization.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: on current main, `/status` and `/new` fall through `getFeishuSequentialKey()` to the same base chat key as normal messages, while `/stop` alone gets `:control`. I did not run live Feishu or tests because this review is read-only.

## Solution Assessment

Yes, the narrowest maintainable direction is a Feishu sequential-key fix with regression coverage, likely by reviewing or importing the linked fork PR. Current main is only partial and should not be treated as implemented.
