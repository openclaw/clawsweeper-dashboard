## Summary

Keep open. Current main only implements the `/stop` slice through a Feishu control lane; `/status` and `/new` still resolve to the normal per-chat lane, which does not satisfy the documented command-only fast path during an active run.

## What This Changes

Feishu text control commands such as `/stop`, `/new`, and `/status` should bypass the active-run queue and execute immediately.

## Best Possible Solution

Keep this issue open until a focused Feishu dispatch or sequential-key fix restores the documented allowlisted command-only fast path without broad unauthenticated direct dispatch.

## Reproduction Assessment

Yes. Source inspection shows `/status` and `/new` still resolve to the normal Feishu chat key and are awaited through the sequential queue, while the docs require command-only allowlisted messages to bypass the queue.

## Solution Assessment

No, current main is only a partial solution. The safer path is a narrow Feishu-owned fix that preserves the shipped `/stop` and `/btw` lanes plus authorization/backpressure, rather than the older broad direct-dispatch branch as-is.
