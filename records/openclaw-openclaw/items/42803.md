## Summary

Keep open: current `main` and `v2026.5.12` still route Feishu `/status` and `/new` text commands through the normal same-chat sequential queue. `/stop` has a shipped control lane, but the broader documented fast-path regression remains source-reproducible.

## What This Changes

Feishu command-only text messages such as `/stop`, `/new`, and `/status` should bypass active same-chat agent queueing.

## Best Possible Solution

Extend the current Feishu-owned sequential-key routing for documented immediate command-only text while preserving authorization, dedupe, and normal handler semantics, then add focused queue/debounce regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a normal Feishu turn that holds `feishu:<account>:<chat>`, then send `/status` or `/new`; current main resolves those commands to the same base key, so they wait behind active work.

## Solution Assessment

Yes, the maintainable fix direction is a narrow Feishu sequential-key/control-lane change; the older direct-dispatch PRs are not the best solution because they bypassed queueing before normal authorization and ordering safeguards.
