## Summary

Keep open. Current main only partially addresses the Feishu regression: abort-like `/stop` now routes out of the base lane, but `/status` and `/new` still resolve to the normal per-chat queue, and an open linked PR remains the active fix candidate.

## What This Changes

The issue asks to restore Feishu text-command queue bypass for `/stop`, `/new`, and `/status` during active agent runs after the 2026.3.8 regression.

## Best Possible Solution

Extend Feishu’s sequential-key policy for the intended authorized command-only fast paths, with focused tests for `/stop`, `/new`, `/status`, authorization, debounce cleanup, and normal-message ordering.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main still awaits the Feishu sequential queue, and the key policy leaves `/status` and `/new` on the base chat key while only abort-like text and `/btw` move out of band.

## Solution Assessment

Unclear for the linked direct-dispatch PR as written. The safer maintainable direction is to extend Feishu sequential-key classification with authorization-aware queue and debounce tests after maintainers confirm the exact command scope.
