## Summary

Keep open: current main and v2026.5.12 still route Feishu `/status` and `/new` command-only text through the normal same-chat sequential queue. `/stop` has a shipped control lane, but the broader documented fast-path regression remains source-reproducible.

## What This Changes

Restore Feishu text command fast-path behavior so `/stop`, `/new`, and `/status` execute during active agent runs instead of waiting behind same-chat queue work.

## Best Possible Solution

Extend Feishu-owned sequential-key routing for safe immediate text commands while preserving authorization and normal handler semantics, then add focused queue/debounce regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a normal Feishu turn that holds `feishu:<account>:<chat>`, then send `/status` or `/new`; current main resolves those commands to the same base key, so they wait behind active work.

## Solution Assessment

Yes. A narrow Feishu sequential-key/control-lane fix is the maintainable direction; the safer alternative to prior direct-dispatch branches is explicit lane classification with existing authorization and handler flow preserved.
