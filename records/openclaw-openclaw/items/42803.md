## Summary

Keep open. Current main still routes Feishu `/status` and `/new` text commands onto the normal per-chat sequential key, so they can wait behind an active Feishu run; only abort-like text and `/btw` are currently split onto out-of-band lanes.

## What This Changes

The issue requests Feishu command-only messages such as `/stop`, `/new`, and `/status` to bypass the per-chat queue during active agent runs.

## Best Possible Solution

Land a narrow Feishu ingress/sequential-key fix, from the existing fork PR or a replacement, that routes authorized immediate command-only `/status`, `/new`, and approved control commands off the busy same-chat lane with regression coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: start a long Feishu task on the base chat key, then send `/status` or `/new`; current main resolves those commands to the same base key, and the sequential queue serializes same-key tasks.

## Solution Assessment

No for current main. The current partial `/stop` and `/btw` split is not enough for the commands named here; the safer approach is a reviewed Feishu command allowlist rather than a blanket bypass for every slash command.
