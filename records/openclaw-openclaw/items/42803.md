## Summary

Keep open. Current main only partially addresses the report: abort-like Feishu text such as `/stop` now uses an out-of-band control lane, but `/status` and `/new` still resolve to the base per-chat lane and can wait behind active same-chat work despite the documented command-only fast path. The issue also has an open linked implementation PR, so it should remain open until that candidate is reviewed, reworked, merged, or closed.

## What This Changes

Feishu text slash commands such as `/stop`, `/new`, and `/status` should bypass the per-chat queue during active agent runs instead of waiting behind the running turn.

## Best Possible Solution

Route the intended authorized Feishu command-only fast paths through explicit out-of-band sequential keys, with focused tests for `/stop`, `/new`, `/status`, authorization, and debounce cleanup semantics.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main awaits the Feishu sequential queue, and the current key policy leaves `/status` and `/new` on the base chat key while only abort-like text and `/btw` move out of band.

## Solution Assessment

Unclear for the linked direct-dispatch PR as written. The safer solution is to extend the current sequential-key classification with authorization-aware queue/debounce tests rather than broadly bypassing dispatch before the command policy is reviewed.
