## Summary

Keep open. Current main still routes Feishu `/status` and `/new` text commands onto the normal per-chat sequential key, so they can wait behind an active Feishu run; only abort-like text and `/btw` are currently split onto out-of-band lanes. The issue also has an open closing PR targeting the same regression, so it should remain open until that implementation is reviewed, landed, or intentionally replaced.

## What This Changes

Feishu command-only text messages such as `/stop`, `/new`, and `/status` should bypass Feishu same-chat queueing and execute immediately during an active agent run.

## Best Possible Solution

Review and land or replace #43512 with a narrow Feishu ingress/sequential-key fix that routes authorized command-only `/status`, `/new`, and maintainer-approved control commands onto an immediate out-of-band key before same-chat serialization, while preserving FIFO for ordinary Feishu messages and adding a busy-lane regression test.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: start a long Feishu task on the base chat key, then send `/status` or `/new`; current main resolves those commands to the same base key, so the sequential queue waits for the active task to finish.

## Solution Assessment

No for current main. The partial `/stop` and `/btw` split does not cover all commands named in the report; the safer fix is a narrow Feishu command allowlist reviewed against ordering semantics, not a blanket bypass for every slash command.
