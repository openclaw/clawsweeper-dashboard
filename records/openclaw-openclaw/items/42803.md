## Summary

Keep open: current main and the latest release still leave `/status` and `/new` on the Feishu base chat lane, while the related open PRs are partial, conflicting, or aimed at adjacent queue behavior.

## What This Changes

Feishu command-only text messages such as `/stop`, `/new`, and `/status` should bypass active same-chat queueing during agent runs.

## Best Possible Solution

Land or replace a current-main Feishu command-lane fix that lets documented allowlisted command-only messages bypass same-chat serialization while preserving command authorization and `/new` ordering semantics.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.6.10` still route `/status` and `/new` through the Feishu base chat key, and that key awaits the full same-chat handler task.

## Solution Assessment

Unclear as a final implementation path. Separate Feishu lanes are the right owner boundary, but maintainers still need to choose the safe command set and `/new` ordering semantics before landing a complete fix.
