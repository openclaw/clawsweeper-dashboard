## Summary

Keep open: current main and the latest tagged release still route Feishu `/status` and `/new` command-only text through the normal same-chat sequential key, so those commands can wait behind an active Feishu run. `/stop` now has a shipped control lane, but the broader reported fast-path regression is not fully fixed in OpenClaw main, and the target-repo fix PRs are closed unmerged while the only open closing PR is in a fork.

## What This Changes

Feishu text command-only messages such as `/stop`, `/new`, and `/status` should bypass the per-chat queue during active agent runs.

## Best Possible Solution

Extend Feishu-owned sequential-key routing for the documented immediate text-command path while preserving normal same-chat FIFO behavior and adding focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a normal Feishu turn that holds `feishu:<account>:<chat>`, then send `/status` or `/new`; current main resolves those commands to the same base key, so they wait behind the active task.

## Solution Assessment

Yes, a narrow Feishu sequential-key/message-handler fix is the maintainable direction. The safer approach is explicit control-lane classification with existing authorization in the normal handler, not the older direct-dispatch bypass pattern from closed PRs.
