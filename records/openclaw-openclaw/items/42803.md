## Summary

Keep open. Current main and the latest shipped tag still leave `/status` and `/new` on the normal Feishu per-chat queue while dispatch awaits the sequential queue; `/stop` now has a control lane, but the reported command set is not fully fixed and related PRs remain open or partial.

## What This Changes

The issue reports that Feishu text commands such as `/stop`, `/new`, and `/status` are queued behind an active agent run instead of taking the documented command fast path.

## Best Possible Solution

Review, land, or replace the Feishu queue-classification fix so the accepted command-only fast paths bypass the per-chat lane with authorization-aware tests and live Feishu proof, while preserving normal message ordering.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main still awaits the Feishu sequential queue, and the current key policy leaves `/status` and `/new` on the base chat key while only abort-like text and `/btw` move out of band.

## Solution Assessment

Unclear for the broader direct-dispatch approach until maintainers confirm the accepted command scope. The maintainable direction is to extend Feishu queue-key classification for the accepted command-only set with authorization and ordering tests.
