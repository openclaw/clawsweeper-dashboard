## Summary

Keep open: current main still only gives Feishu abort-like text a separate control lane; `/status` and `/new` resolve to the normal per-chat key, so they can remain queued behind an active same-chat run. The target-repo fix PRs referenced in the discussion are closed unmerged, and the only open linked implementation in the provided context is in a fork rather than landed on current main.

## What This Changes

Feishu text slash commands such as `/stop`, `/new`, and `/status` should bypass the per-chat queue and execute immediately during an active agent run.

## Best Possible Solution

Finish the Feishu plugin queue-key fix with regression coverage so command-only immediate commands use an unblocked lane while ordinary same-chat messages remain serialized.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a normal Feishu turn that holds `feishu:<account>:<chat>`, then send `/status` or `/new`; both still resolve to the same base key and wait behind it.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin sequential-key/message-handler path with regression tests, following the existing out-of-band lane pattern instead of adding core policy or configuration.
