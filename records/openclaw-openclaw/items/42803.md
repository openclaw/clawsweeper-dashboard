## Label Justifications

- `P1`: The issue blocks Feishu users from timely session control during active runs, including commands that inspect or change session state.
- `impact:session-state`: The affected commands include `/new` and active-run controls, so the failure is about session control/state rather than a cosmetic channel issue.

## Summary

Keep open: current main only gives Feishu abort-like text a separate control lane, while `/status` and `/new` still resolve to the normal per-chat sequential key and can wait behind an active run. There is related open work, but it is either conflicting or intentionally narrower than the full reported behavior.

## What This Changes

Feishu text slash commands such as `/stop`, `/new`, and `/status` should bypass the active per-chat queue and run immediately during an active agent run.

## Best Possible Solution

Have maintainers choose the safe Feishu control-lane whitelist, then land a current-main-compatible fix with regression coverage for busy-run `/status` and the chosen `/new` behavior while preserving normal message serialization.

## Reproduction Assessment

Yes, source-level. `createSequentialQueue` serializes same-key tasks and current `getFeishuSequentialKey` returns the normal per-chat key for `/status` and `/new`; I did not run a live Feishu DM replay.

## Solution Assessment

Unclear until maintainer decision. Mirroring Telegram's read-only control lane looks appropriate for `/status`, but `/new` is stateful and should be an explicit Feishu command-lane policy choice before bypassing an active run.
