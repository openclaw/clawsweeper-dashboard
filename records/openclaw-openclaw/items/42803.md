## Summary

Keep open. Current main still does not implement the full documented Feishu text-command fast path: Feishu dispatch awaits the selected sequential lane, the current key policy only moves abort-like text and `/btw` off the base chat lane, and `/status` plus `/new` remain normal-lane traffic. The open fork PR that says it closes this issue is an implementation candidate, so this issue should stay open until that PR is reviewed, reworked, merged, or closed.

## What This Changes

The issue requests restoring immediate Feishu text-command handling for `/stop`, `/new`, and `/status` during an active agent run after the reported 2026.3.8 regression.

## Best Possible Solution

Route the intended authorized Feishu command-only fast paths through explicit out-of-band lanes while preserving normal message ordering, command authorization, and session-state semantics.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main awaits the Feishu sequential queue, while `/status` and `/new` remain on the base chat key instead of an out-of-band command lane.

## Solution Assessment

Unclear for the linked PR as written. The safer solution is to classify authorized command-only Feishu fast paths in the current sequential-key lane design rather than broadly bypassing dispatch without an ordering and auth review.
