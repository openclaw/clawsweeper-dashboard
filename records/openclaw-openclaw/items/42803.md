## Summary

Keep open. Current main only partially addresses the Feishu queue-bypass regression: abort-like `/stop` now uses an out-of-band lane, but `/status` and `/new` still resolve to the normal per-chat queue, and an open linked PR remains the active implementation candidate.

## What This Changes

Restore the Feishu command-only fast path so `/stop`, `/new`, and `/status` execute immediately during active runs instead of waiting behind per-chat serialization.

## Best Possible Solution

Review and either land, rebase, or replace the linked Feishu queue-classification PR so accepted command-only fast paths leave the per-chat lane while normal messages remain serialized.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main still awaits the Feishu sequential queue, and the key policy leaves `/status` and `/new` on the base chat key while only abort-like text and `/btw` move out of band.

## Solution Assessment

Unclear for the linked direct-dispatch approach as written. The safer maintainable direction is to extend Feishu’s sequential-key classification with authorization-aware queue and debounce tests after maintainers confirm the exact command scope.
