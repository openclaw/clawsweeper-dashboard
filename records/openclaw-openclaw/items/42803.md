## Summary

Current main only partially resolves the report: `/stop` has an out-of-band Feishu lane, while `/status` and `/new` remain on the normal per-chat lane. The linked fork PR is still open but conflicting and implements a broader bypass than the established sibling-channel policy, so this issue remains the canonical place for a maintainer decision.

## What This Changes

Restore immediate Feishu handling for `/stop`, `/new`, and `/status` text commands during an active agent run.

## Best Possible Solution

Match the established channel design by routing abort and safe read-only commands immediately, then define and test an explicit cancel-then-reset sequence for `/new` instead of applying a blanket command bypass.

## Reproduction Assessment

Yes at source level: current Feishu code and tests deterministically show `/status` and `/new` sharing the busy per-chat lane while `/stop` uses the control lane. A live Feishu account was not exercised in this read-only review.

## Solution Assessment

No, the linked blanket bypass is not yet the best proven fix because it conflates abort, read-only, and state-mutating commands. The sibling Telegram implementation provides a safer classification model, with `/new` requiring an explicit ordering decision.
