## Summary

Keep open: current main and v2026.6.10 still only partially address the Feishu active-run command bypass regression. `/stop` is on a Feishu control lane, but `/status` is explicitly tested on the normal same-chat lane and `/new` remains a session command whose lane policy still needs maintainer approval.

## What This Changes

The issue reports that Feishu text commands like `/stop`, `/new`, and `/status` are queued behind active agent runs instead of using the documented immediate command fast path.

## Best Possible Solution

Refresh or replace the linked Feishu work with a current-main patch that keeps `/stop` on the control lane, deliberately handles `/status` and `/new`, and proves active-run behavior with focused Feishu queue tests plus live transport proof when feasible.

## Reproduction Assessment

Yes, at source level: current main routes `/status` to the Feishu base chat key and same-key Feishu work is serialized behind the active handler. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Unclear as a final implementation path. A separate Feishu lane is the right mechanism, but maintainers still need to choose the safe command whitelist and `/new` ordering semantics before landing a fix.
