## Summary

Keep open: current main and the latest release only give Feishu `/stop` and `/btw` separate lanes; `/status` and `/new` still resolve to the base same-chat queue, and related open PRs are partial or adjacent rather than a complete fix.

## What This Changes

The issue reports that Feishu text commands like `/stop`, `/new`, and `/status` no longer bypass the queue during an active agent run.

## Best Possible Solution

Land or replace a focused Feishu command-lane fix that preserves command authorization, keeps `/stop` out-of-band, and defines the intended `/status` and `/new` active-run behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the remaining bug: send `/status` or `/new` in the same Feishu chat while a prior Feishu task occupies the base sequential key; current main routes those commands to that same key.

## Solution Assessment

Unclear as a final implementation path. Separate Feishu lanes are the right owner boundary, but maintainers still need to choose the safe command set and `/new` ordering semantics before landing a complete fix.
