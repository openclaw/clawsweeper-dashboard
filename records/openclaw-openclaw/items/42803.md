## Summary

Keep open: current main still routes Feishu `/status` and `/new` command-only messages onto the normal same-chat sequential key, so they can wait behind an active Feishu run. The target-repo fix PRs linked from the issue are closed unmerged, and the only open closing implementation found in the provided context is in a fork rather than landed on OpenClaw main.

## What This Changes

Feishu text commands such as `/stop`, `/new`, and `/status` should bypass the active same-chat queue and execute immediately during an active agent run.

## Best Possible Solution

Extend Feishu's command-only sequential-key routing so immediate commands use an unblocked control lane, while preserving FIFO serialization for ordinary same-chat messages and adding focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a normal Feishu turn that holds the base `feishu:<account>:<chat>` key, then send `/status` or `/new`; current main resolves those commands to the same base key, so they wait behind the active task.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin's sequential-key/message-handler path, following the existing out-of-band lane pattern instead of adding core policy or configuration.
