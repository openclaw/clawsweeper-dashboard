## Label Justifications

- `P1`: Feishu users still cannot reliably inspect or replace session state during an active run because `/status` and `/new` remain queued behind it.
- `impact:session-state`: The affected commands inspect, abort, or replace active run and session state.

## Summary

Current main partially addresses the regression: Feishu `/stop` uses an out-of-band control lane, but `/status` is explicitly kept on the ordinary per-chat lane and `/new` has no immediate lane. Keep this issue open while maintainers define the safe immediate-command set and review or replace the linked open fork PR. Likely related people are Nimrod Gutman (high confidence, current sequential-key feature history) and Peter Steinberger (medium confidence, recent Feishu dispatch refactor).

## What This Changes

Restore immediate Feishu handling for `/stop`, `/new`, `/status`, and other command-only control messages during an active agent run.

## Best Possible Solution

Define a Feishu immediate-command classifier with explicit semantics, route approved commands to a stable per-chat control lane, and add concurrency tests proving `/stop`, `/status`, `/new`, and ordinary-message ordering without bypassing existing authorization.

## Reproduction Assessment

Yes. Current main explicitly maps `/status` to the ordinary Feishu per-chat queue, and every message on that key awaits the active task; `/stop` alone receives the separate control key.

## Solution Assessment

No, the current abort-only implementation is not the complete fix requested by this issue. The narrow maintainable solution is an explicit, reviewed immediate-command set with race-focused queue tests rather than treating every recognized command identically.
