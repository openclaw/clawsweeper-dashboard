## Summary

Keep open. Current main and the latest release still source-reproduce the Feishu fast-path gap for at least `/status` and `/new`; `/stop` now has a separate control lane, but the reported command set is only partially addressed and an open linked fork PR still uses closing syntax for this issue.

## What This Changes

The issue reports that Feishu text commands such as `/stop`, `/new`, and `/status` should bypass active-run queueing instead of waiting behind the current run.

## Best Possible Solution

Review, land, or replace the linked Feishu fix with a focused queue-classification change that covers the accepted command-only fast paths, preserves normal-message ordering, and adds authorization-aware regression coverage.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main still awaits the Feishu sequential queue, and the current key policy leaves `/status` and `/new` on the base chat key while only abort-like text and `/btw` move out of band.

## Solution Assessment

Unclear until maintainers confirm the accepted out-of-band command scope. The maintainable direction is a narrow Feishu queue-key or structured-command fast path with authorization and ordering tests, not an unbounded direct-dispatch bypass.
