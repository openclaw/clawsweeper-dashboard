## Summary

Keep open: current main still has a source-level path where `session_status({"sessionKey":"current"})` is scoped through the sandbox/runtime-policy direct key instead of the live embedded run key, so a stale direct-session row can win over `agent:main:main`.

## What This Changes

The issue asks for `session_status` to bind `current` to the calling embedded run and report selected model intent instead of stale direct-session or cached runtime metadata.

## Best Possible Solution

Bind `current` to the live run session key while keeping sandbox/runtime-policy keys for policy decisions, and add regression coverage for stale direct rows plus explicit selected-model overrides.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: seed `agent:main:main` and a stale `agent:main:telegram:default:direct:<id>`, construct the tool with the direct runtime-policy key, and call `session_status({"sessionKey":"current"})`. I did not run a live Telegram replay.

## Solution Assessment

Yes, the narrow maintainable fix is to pass the live embedded run session key separately from the sandbox/runtime-policy key and make `current` prefer that live key. Preserve existing runtime-only model diagnostics unless maintainers choose a broader status-contract change.
