## Summary

Keep open: current main still has a source-level path where `session_status({"sessionKey":"current"})` can receive the direct runtime-policy key instead of the live `agent:main:main` run key, and the selected-model versus cached runtime-model status contract still needs maintainer direction.

## What This Changes

Request that `session_status({"sessionKey":"current"})` bind to the live embedded run session and report selected model intent separately from cached last-run/runtime model data.

## Best Possible Solution

Bind `current` to the live embedded run session key while preserving the direct runtime-policy key for sandbox and policy checks, then make status output distinguish selected model intent from active or last-run runtime metadata.

## Reproduction Assessment

Yes for a source-level reproduction: current main threads `runtimePolicySessionKey` into the embedded tool session key and `current` can resolve through that requester key. I did not run a live Telegram replay in this read-only review.

## Solution Assessment

Unclear for the full issue until maintainers confirm the selected-model versus runtime-model status contract. The narrow maintainable fix is to bind `current` to the live run key and retain runtime or last-run model data as secondary diagnostics.
