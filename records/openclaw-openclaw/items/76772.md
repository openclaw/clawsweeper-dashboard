## Summary

Keep open: current main still has a source-level path where `session_status({ sessionKey: "current" })` can resolve through the requester/runtime-policy session key instead of the live merged session, and the selected-model versus runtime/last-run model status contract is still not fully settled.

## What This Changes

The issue reports that `session_status({ sessionKey: "current" })` can show stale model identity because `current` may bind to a legacy direct session and status may display cached runtime model data instead of selected model intent.

## Best Possible Solution

Bind `session_status` `current` to the live run session key while keeping the runtime-policy key for sandbox and policy checks, then expose selected model intent separately from runtime or last-run model diagnostics.

## Reproduction Assessment

Yes for a source-level reproduction: current main still passes the runtime-policy session key into embedded tools, and explicit `current` can fall back through requester-key alias resolution. No live Telegram replay was established in this read-only review.

## Solution Assessment

Unclear for the full issue until maintainers confirm the selected-model versus runtime-model status contract. The narrow maintainable part is to bind `current` to the live run key and retain runtime/last-run model data as secondary diagnostics.
