## Summary

Keep open: current main still has a source-level path where `session_status({"sessionKey":"current"})` can resolve through the requester key before `agent:main:main`, and the no-override status path still intentionally treats cached runtime model fields as the selected card model.

## What This Changes

The issue asks for `session_status` to bind `current` to the live embedded run session and to report selected model intent separately from cached last-run or fallback model metadata.

## Best Possible Solution

Bind `current` to the actual embedded run session key, then display selected model intent separately from last runtime/fallback metadata with regression coverage for stale legacy direct plus live main rows.

## Reproduction Assessment

Yes, at source level: with `opts.agentSessionKey` set to a stale default direct key and both that row plus `agent:main:main` present, the current fallback candidate order selects the requester key before the main key. I did not replay a live Telegram embedded run.

## Solution Assessment

Unclear: the alias repair is narrow, but the selected-model versus cached runtime/fallback display contract needs maintainer judgment because current tests intentionally preserve runtime-as-selected behavior when no override exists.
