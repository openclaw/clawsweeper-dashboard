## Summary

Keep open: current main still has a source-level path where `session_status({"sessionKey":"current"})` can resolve through the runtime-policy requester key instead of the live conversation key, and the selected-model versus runtime-model status contract is still not fully settled.

## What This Changes

The issue asks for `session_status` to bind `current` to the running embedded session and report selected model intent separately from cached last-run model metadata.

## Best Possible Solution

Bind tool-facing `current` to the live run session key while keeping the runtime-policy key only for sandbox/tool policy, then document or expose selected model and last-runtime model as separate status fields.

## Reproduction Assessment

Yes for a source-level reproduction: embedded runs can pass the runtime-policy key into tool construction, and `session_status` can use that requester key when resolving `current`. I did not establish a live Telegram replay in this read-only review.

## Solution Assessment

Unclear for the full issue until maintainers decide the no-override selected-model versus runtime-model contract. The narrow maintainable fix is to pass the live session key to `session_status` while preserving runtime-policy keys for sandbox and tool-policy checks.
