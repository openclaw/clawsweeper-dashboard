## Summary

Keep open: current main still has a source-level path for the embedded tool context to hand `session_status` the runtime-policy/direct session key instead of the live run key, and the status surface still treats cached runtime model fields as displayable session model identity in no-override cases.

## What This Changes

The issue asks for `session_status({"sessionKey":"current"})` to bind to the live embedded run session and report current model intent instead of stale direct-session or last-run model metadata.

## Best Possible Solution

Bind `session_status` `current` to the live embedded run session key while preserving the runtime-policy key for policy/sandbox checks, then split selected model intent from last-run/runtime model diagnostics in the status output.

## Reproduction Assessment

Yes for source-level reproduction: current main still passes the runtime-policy key into embedded tool setup, and `current` can resolve through the requester key. No live Telegram replay was established in this read-only review.

## Solution Assessment

Unclear for the full issue until maintainers confirm the selected-model versus runtime-model status contract. The narrow maintainable first step is to bind `current` to the live run key and keep runtime/last-run model data as secondary diagnostics.
