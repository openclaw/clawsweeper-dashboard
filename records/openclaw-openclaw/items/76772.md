## Summary

Keep open: the report is concrete, filed against v2026.5.2, and the relevant current-main source is unchanged from that release, so there is no main-branch fix to close against. Current code has some requester-session safeguards, but the status path still mixes selected model intent with cached runtime model state and lacks a regression covering the reported stale direct-session plus modelOverride mismatch.

## What This Changes

The issue reports that `session_status({"sessionKey":"current"})` can resolve a stale legacy direct session and report cached last-run model metadata instead of the live session's current model intent.

## Best Possible Solution

Add a regression that seeds `agent:main:main` plus a stale legacy direct row, then make `session_status("current")` bind to the live requester session and report selected model intent while exposing cached last-run/fallback model only as explicit runtime history when it is still valid.

## Reproduction Assessment

No high-confidence live reproduction was run. Source review plus release/current parity gives a medium-confidence reproduction path for the stale model-state portion and a plausible path for the `current` alias portion that should be locked down with a focused regression test.

## Solution Assessment

No, current main is not a sufficient solution because the relevant code is unchanged from the reported release and the exact combined failure mode is not covered. The narrow maintainable fix is in the session_status resolution/model-display path, not a broad session-store migration.
