## Summary

Keep open: current main has partial model-identity improvements, but the combined reported failure is not covered and the `current` resolution path can still fall back through requester/default-direct session keys when the live tool context is not already canonicalized to `agent:main:main`.

## What This Changes

The issue asks for `session_status({"sessionKey":"current"})` to bind to the calling live session and report current selected model intent instead of stale direct-session or cached runtime model data.

## Best Possible Solution

Bind `session_status` `current` to the canonical live requester session, render selected model intent from overrides/defaults, and expose last runtime or fallback model only as explicit history when it is known to be active.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a focused synthetic path for the stale requester/direct fallback and cached runtime-model display, but the Telegram wiring should be locked down with a regression test.

## Solution Assessment

No, current main is not yet a complete solution because it lacks the combined regression and still has fallback behavior that can select requester/default-direct rows. The narrow maintainable fix is in the `session_status` current-resolution and status model-display path, not a broad session-store migration.
