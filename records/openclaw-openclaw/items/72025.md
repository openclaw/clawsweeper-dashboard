## Summary

Keep open. The PR is useful and the latest head fixes the earlier no-final-reply race, but it still does not implement the terminal cleanup contract used by other status-reaction channels, so Signal users can be left with stale done/error reactions and cannot honor `messages.removeAckAfterReply`.

## What This Changes

This PR adds Signal inbound status-reaction controller wiring and regression tests for direct, group, and no-final-reply Signal dispatch cases.

## Best Possible Solution

Land a focused Signal plugin fix that reuses the shared status-reaction controller, adds Signal reaction removal to the adapter, applies done/error hold plus clear/restore behavior consistent with Slack/Discord/Telegram, and updates user-facing docs plus the changelog for the newly supported Signal lifecycle surface.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: a Signal DM or allowed group message with `messages.statusReactions.enabled: true` reaches `dispatchInboundMessage` without status-reaction callbacks, while Signal send/remove reaction helpers already exist.

## Solution Assessment

No. The PR direction is right, but the current patch stops at setting terminal reactions and does not preserve the existing `removeAckAfterReply` cleanup semantics or document the new user-visible Signal support.
