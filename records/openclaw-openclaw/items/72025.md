## Summary

Keep open. Current main still lacks Signal inbound status-reaction lifecycle wiring, so the PR is useful, but the proposed patch is not merge-ready because terminal reactions are left behind instead of following the shared clear/restore contract, and the user-facing config docs/changelog have not moved with the behavior change.

## What This Changes

The PR wires Signal inbound messages into the shared status-reaction controller and adds direct, group, and no-final-reply regression tests for Signal status reactions.

## Best Possible Solution

Land a narrow Signal plugin fix that reuses the shared status-reaction controller, routes direct and group reactions through existing Signal helpers, preserves mention gating and no-final restoration, implements terminal clear/restore behavior consistent with Slack/Discord/Telegram, and updates the changelog plus user-facing config docs.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: a Signal DM or allowed group message with messages.statusReactions.enabled true reaches dispatchInboundMessage without status-reaction callbacks. The PR blocker is also reproducible by following the visible-dispatch setDone/setError path and observing that no clear or restore transition is scheduled.

## Solution Assessment

No, not in the current PR form. The implementation direction is right, but the narrow maintainable fix needs terminal cleanup semantics and docs/changelog updates before merge.
