## Summary

Keep this PR open: it addresses a real Signal status-reaction gap on current main, and the latest head fixes the earlier non-final done-race, but the patch still leaves terminal reactions without the shared cleanup/restore behavior, lacks user-facing docs, and has no real Signal behavior proof.

The PR branch modifies Signal event handling to create a status reaction controller for eligible inbound messages, wire lifecycle callbacks into dispatch, and add direct/group/no-final unit tests.

## Best Possible Solution

Land a corrected Signal plugin patch that reuses sendReactionSignal/removeReactionSignal, mirrors existing terminal cleanup semantics, covers remove/restore cases, updates the Signal/status-reaction docs, and includes redacted real Signal proof.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: Signal inbound dispatch reaches dispatchInboundMessage with no status-reaction controller or lifecycle callbacks, while Signal already has reaction send/remove helpers.

## Solution Assessment

No, not in the current PR form. The direction is narrow and appropriate, but it should wire removeReactionSignal and final clear/restore behavior to match Slack/Discord, plus update docs and provide real Signal proof.
