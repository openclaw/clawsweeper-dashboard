## Summary

Keep open. Current main has addressed the noisy/default-profile warning side of this area, but `apply_patch` is still created only for exact `openai` and `openai-codex` provider ids, so routed or OpenAI-compatible provider cases remain unresolved and need maintainer product/security judgment.

## What This Changes

The issue requests that `apply_patch` stay recognized and available for agent-routed runs instead of being treated as unknown or unavailable during tool-policy resolution.

## Best Possible Solution

Maintainers should decide whether `apply_patch` eligibility should be driven by a provider capability/canonical OpenAI-compatible route rather than exact provider id, then update the gate, warnings, docs, and regression tests without weakening workspace protections.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show the reported routed-provider failure mode: any non-`openai`/`openai-codex` provider id prevents `apply_patch` creation before the policy pipeline can allow it.

## Solution Assessment

No. Treating this as a simple policy-pipeline classification bug is not the best fix; current main points to a provider/model capability gate that needs an explicit maintainer decision before changing a mutating tool surface.
