## Summary

Keep open. Current main has improved the misleading allowlist-warning path for gated core tools, but the reported routed-provider execution problem remains because `apply_patch` is still only constructed for exact `openai` or `openai-codex` provider IDs before the policy pipeline runs.

## What This Changes

The issue requests that `apply_patch` remain available and correctly recognized for eligible agent-routed runs instead of being omitted or treated as unavailable during tool-policy resolution.

## Best Possible Solution

Define the `apply_patch` capability contract for routed/OpenAI-compatible providers, then either add an explicit trusted provider-alias or capability opt-in with focused tests and docs, or intentionally keep the strict provider gate and document the limitation clearly.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to resolve coding tools with `tools.exec.applyPatch` enabled and a non-exact routed provider ID; current main makes `applyPatchEnabled` false and omits `apply_patch` before policy filtering.

## Solution Assessment

No. The issue’s local workaround targets `coreToolNames` classification, but current main shows the tool can be absent before the policy pipeline runs; a safer solution is an explicit provider capability/trusted alias contract or clearer strict-gate diagnostics.
