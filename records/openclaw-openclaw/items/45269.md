## Summary

Keep open: current main still gates `apply_patch` creation to exact `openai`/`openai-codex` provider ids, so routed-provider availability remains unresolved; the original policy-pipeline-only diagnosis has been narrowed by later warning changes and issue discussion.

## What This Changes

The issue asks for `apply_patch` to be available and recognized for OpenAI-compatible agent-routed runs instead of being dropped or reported as unknown.

## Best Possible Solution

Define an explicit trusted routed-provider capability contract for `apply_patch`, or document routed providers as unsupported, then update tool creation, focused tests, and docs to match that decision.

## Reproduction Assessment

Yes, source-reproducible: any provider id other than `openai` or `openai-codex` makes `applyPatchEnabled` false before policy filtering, even when the model and `tools.exec.applyPatch` config otherwise match. I did not run a live routed-gateway session.

## Solution Assessment

No, the proposed local workaround that forces `apply_patch` into `coreToolNames` treats the warning symptom. The maintainable fix belongs at tool creation/provider capability gating, with a fail-closed trust contract for routed providers.
