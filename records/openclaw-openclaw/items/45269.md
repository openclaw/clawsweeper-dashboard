## Summary

Keep open: current `main` has improved the misleading warning path, but `apply_patch` is still constructed only for exact `openai` / `openai-codex` provider ids, so routed/custom provider support remains an unresolved product and security decision.

## What This Changes

The issue requests that `apply_patch` be recognized and executable for agent-routed OpenAI/Codex runs instead of being treated as unavailable or unknown during tool policy resolution.

## Best Possible Solution

Define a fail-closed trusted-provider capability contract for routed `apply_patch` support, or explicitly document routed/custom providers as unsupported for this tool, then align creation logic, warnings, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current `main` shows any `modelProvider` other than `openai` or `openai-codex` makes `applyPatchEnabled` false before the policy pipeline; I did not run a live routed-gateway session.

## Solution Assessment

No. The local policy-pipeline workaround and the closed related PR address warning/classification symptoms, but the maintainable solution needs explicit routed-provider trust semantics or a documented unsupported boundary.
