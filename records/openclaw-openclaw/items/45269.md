## Summary

Keep open. Current main improved the misleading gated-core warning path, but `apply_patch` is still only constructed for exact `openai` or `openai-codex` provider IDs, so routed provider IDs can still omit the tool before policy filtering runs.

## What This Changes

Request to make `apply_patch` recognized and executable for eligible routed OpenAI/Codex agent runs instead of being treated as an unavailable or unknown allowlist entry.

## Best Possible Solution

Define a documented trusted provider alias or capability opt-in for routed OpenAI-compatible providers before constructing `apply_patch`, or explicitly document the strict provider limit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolve coding tools with `tools.exec.applyPatch` enabled and a routed provider ID such as `tailscale-gateway` or `ppq/autoclaw`; `applyPatchEnabled` remains false and the tool is omitted before policy filtering.

## Solution Assessment

No. The issue's local workaround targets `coreToolNames`, but current main shows the remaining behavior belongs in provider/capability gating or explicit documentation of the strict provider gate.
