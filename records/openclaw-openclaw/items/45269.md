## Summary

Keep open: current main has fixed the misleading default-profile warning path, but the routed-provider apply_patch behavior remains unresolved. The tool is still created only for exact openai/openai-codex provider ids before policy filtering, and widening that boundary needs maintainer product/security judgment.

## What This Changes

The issue requests that apply_patch be recognized and usable in agent-routed runs instead of being reported as unavailable or unknown when configured for eligible models.

## Best Possible Solution

Define a fail-closed trusted routed-provider capability contract for apply_patch, or explicitly document routed/custom providers as unsupported, then align tool creation, tests, and docs.

## Reproduction Assessment

Yes, source-reproducible: any resolved provider id other than openai or openai-codex makes applyPatchEnabled false before policy filtering, even when the model/config would otherwise allow the tool. I did not run a live routed-gateway session.

## Solution Assessment

No, the local workaround that forces apply_patch into coreToolNames addresses warning/classification symptoms but not the earlier tool-creation gate. The maintainable fix needs explicit routed-provider capability and trust semantics, or a documented unsupported boundary.
