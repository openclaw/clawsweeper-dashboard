## Summary

Keep open. Current main and the latest stable release still construct `apply_patch` only for exact `openai` / `openai-codex` provider IDs, so routed providers such as a Tailscale gateway or `ppq/autoclaw` still need an explicit trust/capability decision rather than cleanup closure.

## What This Changes

The issue asks for `apply_patch` to be recognized and available for OpenAI-compatible routed provider runs when `tools.exec.applyPatch` enables it.

## Best Possible Solution

Define a documented trusted provider alias or capability opt-in for routed OpenAI-compatible providers before constructing `apply_patch`, or explicitly keep and document the strict direct-provider contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolve coding tools with `tools.exec.applyPatch` enabled and a routed provider ID such as `tailscale-gateway` or `ppq/autoclaw`; the provider predicate fails and `apply_patch` is omitted before policy filtering.

## Solution Assessment

No. The local workaround that forces `apply_patch` into `coreToolNames` is not the best fix because current main shows the remaining behavior belongs in provider/capability gating or explicit documentation of the strict provider contract.
