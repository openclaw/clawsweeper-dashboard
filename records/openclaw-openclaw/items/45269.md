## Summary

Keep open: current main improves the warning path for gated core tools, but it still creates `apply_patch` only when the resolved provider is exactly OpenAI/OpenAI Codex. The issue discussion and related duplicate point to routed/dynamic provider chains where the tool never gets created, so the remaining provider-capability question is not implemented on main.

## What This Changes

Ensure `apply_patch` is available, or at least diagnosed correctly, for supported agent-routed OpenAI/Codex runs instead of being treated like an unknown or plugin-only allowlist entry.

## Best Possible Solution

Define a clear provider-capability contract for when routed providers may expose `apply_patch`, keep incompatible models gated, and make diagnostics distinguish gated core tools from truly unknown tool names.

## Reproduction Assessment

Unclear for the reporter's exact gateway setup because no live routed-provider config or logs are available. Source inspection gives a clear synthetic path: when `tools.exec.applyPatch` is enabled but the resolved provider is not exactly `openai` or `openai-codex`, current main does not create `apply_patch`.

## Solution Assessment

No. Forcing `apply_patch` into `coreToolNames` would only change classification; the remaining execution failure happens at tool creation and needs a provider capability/config decision.
