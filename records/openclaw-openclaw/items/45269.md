## Summary

Keep open: current main no longer matches the original policy-pipeline-only diagnosis, but routed-provider `apply_patch` availability is still unresolved and needs an explicit provider capability decision before code changes.

## What This Changes

The issue requests that `apply_patch` remain recognized and executable for agent-routed runs instead of being dropped as an unknown tool during policy resolution.

## Best Possible Solution

Define an explicit routed-provider capability contract for `apply_patch`, or document that routed providers are unsupported, then gate tool creation on that contract with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: a provider id other than `openai` or `openai-codex` cannot create `apply_patch` even when model configuration otherwise matches. I did not run a live routed gateway reproduction.

## Solution Assessment

No. The local workaround that forces `apply_patch` into `coreToolNames` treats the symptom in policy filtering, while the current source and discussion show the remaining decision belongs at tool creation/provider capability gating.
