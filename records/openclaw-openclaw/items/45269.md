## Summary

Keep open: current main has improved the misleading gated-core warning path, but the remaining reported availability symptom is still a routed-provider capability decision rather than something safe to auto-close as fully implemented.

## What This Changes

The issue asks for `apply_patch` to be treated as a valid built-in tool in agent allowlist resolution so routed OpenAI/Codex runs can execute it.

## Best Possible Solution

Decide whether routed providers may advertise OpenAI-compatible `apply_patch` capability, then implement that as an explicit provider/capability contract with docs and regression coverage instead of forcing the tool through policy allowlists.

## Reproduction Assessment

Yes, partially from source: a provider id other than `openai` or `openai-codex` cannot create `apply_patch`, even if the model behind the route is OpenAI-compatible. I did not establish a live current-main gateway reproduction.

## Solution Assessment

No. The issue’s local workaround that forces `apply_patch` into `coreToolNames` is not the best fix because the current discussion and source point to tool creation/provider capability gating, not allowlist stripping.
