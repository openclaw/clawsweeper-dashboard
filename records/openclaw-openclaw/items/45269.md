## Summary

Keep open. Current main improved the gated-core warning path, but it still constructs `apply_patch` only for exact `openai` or `openai-codex` provider IDs, so routed provider IDs can still omit the workspace-writing tool before policy filtering.

## What This Changes

The issue requests that `apply_patch` be recognized and available for agent-routed OpenAI/Codex runs instead of being treated as unavailable or unknown.

## Best Possible Solution

Define a documented trusted provider alias or capability opt-in for routed OpenAI-compatible providers before constructing `apply_patch`, or explicitly document the strict native-provider limit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolve coding tools with `tools.exec.applyPatch` enabled and a routed provider ID such as `tailscale-gateway` or `ppq/autoclaw`; `isOpenAIProvider` remains false and `applyPatchTool` is never added.

## Solution Assessment

No. The issue's local `coreToolNames` workaround would not create a missing tool; the maintainable fix belongs in provider/capability gating or explicit documentation of the strict provider boundary.
