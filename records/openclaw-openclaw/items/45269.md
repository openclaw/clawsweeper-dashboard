## Summary

Keep open: current main narrowed the misleading baseline-profile warning path, but apply_patch is still created only for exact openai/openai-codex provider ids, so routed/custom provider support remains an unresolved product and security decision.

## What This Changes

Request that apply_patch be recognized and usable for agent runs routed through OpenAI-compatible providers instead of being dropped as unknown or unavailable.

## Best Possible Solution

Add a fail-closed trusted-provider capability contract for routed apply_patch support, or explicitly document routed/custom providers as unsupported for this tool, then align creation logic, warnings, docs, and regression coverage.

## Reproduction Assessment

Yes, source-reproducible: any resolved provider id other than openai or openai-codex makes applyPatchEnabled false before policy filtering, even when apply_patch is configured and the model name would otherwise match. I did not run a live routed-gateway session.

## Solution Assessment

No, the local workaround that forces apply_patch into coreToolNames addresses warning/classification symptoms but not the earlier tool-creation gate. The maintainable fix needs explicit routed-provider capability and trust semantics, or a documented unsupported boundary.
