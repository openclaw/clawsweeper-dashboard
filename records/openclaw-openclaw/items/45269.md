## Summary

Keep open: current main has fixed the misleading default-profile warning path, but routed/custom provider availability for `apply_patch` remains unresolved because tool creation still fails closed to exact OpenAI/OpenAI Codex provider ids before policy filtering.

## What This Changes

The issue asks OpenClaw to keep `apply_patch` recognized and executable for eligible agent-routed OpenAI/Codex runs instead of treating it as unavailable or plugin-only.

## Best Possible Solution

Define an explicit trusted routed-provider capability contract for `apply_patch`, or document routed providers as unsupported, then align tool creation, tests, and docs with that decision.

## Reproduction Assessment

Yes, source-reproducible: any provider id other than `openai` or `openai-codex` makes `applyPatchEnabled` false before policy filtering, even when `tools.exec.applyPatch` and the model allowlist otherwise match. I did not run a live routed-gateway session.

## Solution Assessment

No, the proposed workaround that forces `apply_patch` into `coreToolNames` addresses the warning symptom, not the tool-creation gate. The maintainable fix belongs in provider capability/trust semantics for routed providers, with a fail-closed default.
