## Summary

Close: current main has shipped the policy-pipeline fix for the reported misclassification. `apply_patch` is now treated as a shipped gated core tool in warnings/tests, while remaining custom-relay cases are provider availability rather than plugin-only classification.

## What This Changes

Treat `apply_patch` as a valid built-in/core tool during allowlist resolution so agent-routed runs do not drop it as unknown/plugin-only.

## Best Possible Solution

Keep the shipped gated-core warning behavior; if maintainers want `apply_patch` for custom OpenAI-compatible relays that do not report `openai` or `openai-codex`, track that as a separate provider capability/product decision.

## Reproduction Assessment

No high-confidence current-main failure path for the reported plugin-only warning remains. I did not run a live relay, but current source and tests show `apply_patch` is treated as a gated core tool and default profile warnings are suppressed.

## Solution Assessment

Yes. The shipped fix addresses the misleading policy-pipeline classification without widening the provider/model contract; broader custom-relay support would be a separate feature decision.
