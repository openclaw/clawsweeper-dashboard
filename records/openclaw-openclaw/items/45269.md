## Summary

Keep open: current main has improved the misleading profile-warning path, but `apply_patch` is still constructed only for exact `openai` / `openai-codex` provider ids, so routed/custom provider support remains an unresolved product and security decision.

## What This Changes

The issue asks OpenClaw to recognize and permit the built-in `apply_patch` tool for agent-routed OpenAI-compatible provider chains instead of treating it as unavailable during policy resolution.

## Best Possible Solution

Add a fail-closed trusted-provider capability contract for routed `apply_patch` support, or explicitly document routed/custom providers as unsupported for this tool, then align creation logic, warnings, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows any `modelProvider` other than `openai` or `openai-codex` makes `applyPatchEnabled` false before the policy pipeline; I did not run a live routed-gateway session.

## Solution Assessment

No. The local workaround and closed policy-pipeline PR address warning/classification symptoms, but the maintainable solution needs explicit routed-provider trust semantics or a documented unsupported boundary.
