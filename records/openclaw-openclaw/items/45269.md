## Summary

Keep open. Current main still constructs `apply_patch` only for exact `openai` / `openai-codex` provider IDs, and the remaining routed-provider behavior needs an explicit trust/capability decision rather than cleanup.

## What This Changes

The issue asks for `apply_patch` to remain available and recognized when agent runs use routed OpenAI/Codex provider chains such as `tailscale-gateway` or `ppq/autoclaw`.

## Best Possible Solution

Define a documented trusted provider alias/capability opt-in for routed OpenAI-compatible providers before constructing `apply_patch`, or explicitly keep and document the strict direct-provider contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolve coding tools with `tools.exec.applyPatch` enabled and a routed provider ID such as `tailscale-gateway` or `ppq/autoclaw`; `applyPatchEnabled` remains false and the tool is omitted before policy filtering.

## Solution Assessment

No. The issue's local workaround targets `coreToolNames`, but current main shows the remaining behavior belongs in provider/capability gating or explicit documentation of the strict provider gate.
