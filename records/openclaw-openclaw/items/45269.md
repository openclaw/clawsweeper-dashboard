## Summary

Keep open. Current main has fixed the misleading plugin-only warning path for baseline profile entries, but the reported routed-provider failure still exists at tool construction: `apply_patch` is only created for exact `openai` or `openai-codex` provider IDs, so gateways such as Tailscale or multi-model routers need an explicit trust/capability decision before this can be closed.

## What This Changes

The issue asks for agent-routed OpenAI/Codex runs to receive and execute `apply_patch` instead of having it treated as unknown or unavailable by the tool policy path.

## Best Possible Solution

Define a documented trusted provider alias or capability opt-in for routed OpenAI-compatible providers before constructing `apply_patch`, or explicitly keep and document the strict direct-provider contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolve coding tools with `tools.exec.applyPatch` enabled and a routed provider ID such as `tailscale-gateway` or `ppq/autoclaw`; `isOpenAIProvider` fails and `apply_patch` is omitted before policy filtering.

## Solution Assessment

No. The local workaround that forces `apply_patch` into `coreToolNames` is not the best fix because the remaining failure belongs to provider/capability gating and the security trust model for a mutating workspace tool.
