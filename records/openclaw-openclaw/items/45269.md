## Summary

Keep open. Current main improved the misleading allowlist-warning path, but `apply_patch` is still only constructed for exact `openai` or `openai-codex` provider IDs, so routed providers can still miss the tool before policy filtering runs.

## What This Changes

The issue asks for `apply_patch` to be recognized and executable in agent-routed runs when the operator enables it for a supported runtime/model.

## Best Possible Solution

Define a documented trusted provider alias or capability opt-in for OpenAI-compatible routed providers, with focused creation/warning tests; otherwise document the strict provider limitation clearly.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to resolve coding tools with `tools.exec.applyPatch` enabled and a routed provider ID such as `tailscale-gateway` or `ppq/autoclaw`; current main makes `applyPatchEnabled` false and omits `apply_patch` before policy filtering.

## Solution Assessment
