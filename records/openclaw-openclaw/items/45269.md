## Summary

Keep open. Current main has improved the warning wording, but routed provider IDs still fail the exact `openai`/`openai-codex` gate before `apply_patch` is constructed, so the remaining routed-provider behavior needs maintainer direction rather than cleanup closure.

## What This Changes

The issue requests that `apply_patch` remain recognized and usable for agent-routed OpenAI/Codex-compatible runs instead of being omitted or reported as unavailable during tool-policy resolution.

## Best Possible Solution

Define a security-reviewed trusted provider alias or provider capability opt-in before constructing `apply_patch` for routed OpenAI-compatible providers, or explicitly document that those routed IDs are unsupported for this workspace-writing tool.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolve coding tools with `tools.exec.applyPatch` not disabled and a routed provider ID such as `tailscale-gateway` or `ppq/autoclaw`; `applyPatchEnabled` remains false and `apply_patch` is omitted before policy filtering.

## Solution Assessment

No. The local workaround that changes `coreToolNames` is not the best remaining fix; current main shows the unresolved behavior belongs in provider/capability gating or in explicit documentation of the strict provider limit.
