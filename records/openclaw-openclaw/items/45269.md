## Summary

Keep open: current main has fixed the misleading default-profile warning path, but the remaining routed-provider behavior is still unresolved because apply_patch is only created for exact openai/openai-codex provider ids before policy filtering.

## What This Changes

The issue asks for apply_patch to remain recognized and executable for agent-routed runs when the runtime/model is otherwise eligible, instead of being treated as unavailable or unknown.

## Best Possible Solution

Define a fail-closed trusted routed-provider capability contract for apply_patch, or explicitly document routed/custom providers as unsupported, then align tool creation, tests, and docs.

## Reproduction Assessment

Yes, source-reproducible: any provider id other than openai or openai-codex makes applyPatchEnabled false before policy filtering, even when applyPatch config and the model allowlist otherwise match. I did not run a live routed-gateway session.

## Solution Assessment

No, the workaround that forces apply_patch into coreToolNames addresses a warning/classification symptom, not the earlier tool-creation gate. The maintainable fix needs routed-provider capability and trust semantics with a fail-closed default.
