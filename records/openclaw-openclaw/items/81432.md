## Summary

Keep open: current main still inspects active plugin-skill entries with `fs.readlinkSync()` before checking the entry type, so a real directory at the managed link path follows the reported non-ENOENT warning path.

## What This Changes

The issue asks OpenClaw to tolerate non-symlink directories in the managed plugin-skills directory instead of warning with `EINVAL` during `openclaw skills list`.

## Best Possible Solution

Make `publishPluginSkills()` inspect existing entries with `lstatSync()` first, replace generated Windows directory entries in the owned plugin-skills directory when needed, preserve non-Windows safety behavior, and add focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main calls `fs.readlinkSync(linkPath)` before type-checking the active managed entry, so a preexisting regular directory takes the reported warning path. I did not run a live Windows CLI reproduction in this read-only review.

## Solution Assessment

No, the proposed lstat-before-readlink direction is right but the shown snippet is incomplete because it does not remove or replace real directories before creating the managed link. The safer fix is a type-aware replacement path for generated entries, especially Windows directory/junction-like entries.
