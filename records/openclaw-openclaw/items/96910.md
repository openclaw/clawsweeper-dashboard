- `impact:security`: The issue concerns TLS certificate identity and users may otherwise work around failures by weakening certificate verification.

## Summary

Keep open: current main and v2026.6.10 still generate CN-only gateway self-signed certificates, so standards-strict TLS clients that do not use OpenClaw's fingerprint bypass can reject the gateway by hostname or IP.

## What This Changes

Request to add subjectAltName entries to auto-generated gateway TLS certificates so strict hostname/IP verification works.

## Best Possible Solution

Update the gateway TLS generator to emit a standards-valid SAN extension for the supported local identities, keep existing fingerprint pinning behavior intact, and add regression coverage that inspects the generated certificate SAN.

## Reproduction Assessment

Yes, at source level: the current generator invokes openssl with CN only, and the same command produces a certificate with no subjectAltName extension. I did not run a full gateway plus strict client handshake in this read-only review.

## Solution Assessment

Yes, the right fix is in the gateway TLS certificate generator, not in another client-side fingerprint workaround; existing fingerprint propagation helps OpenClaw-owned clients but does not make the generated certificate standards-valid.
