## Summary

Keep open: current main and the latest release still generate CN-only gateway self-signed certificates, so the SAN-specific strict-TLS problem remains real and unsolved.

## What This Changes

The issue asks for auto-generated gateway TLS certificates to include subjectAltName entries so strict TLS clients can verify localhost and bind-address probes.

## Best Possible Solution

Update the gateway TLS generator to emit standards-valid SANs for supported local identities, keep existing fingerprint pinning intact, and add focused certificate-generation coverage.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 invoke OpenSSL with CN only, and Node's TLS identity check expects IP SANs for IP hostnames. I did not run a full gateway plus strict external-client handshake in this read-only review.

## Solution Assessment

Yes on the likely layer: the durable fix belongs in the gateway TLS certificate generator rather than adding another client-side trust workaround. The identity set still needs maintainer/security confirmation before implementation.
