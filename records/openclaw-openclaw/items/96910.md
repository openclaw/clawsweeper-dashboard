## Summary

Keep open: current main and the latest release still generate CN-only gateway self-signed certificates, and the issue is already paired with the open candidate fix at https://github.com/openclaw/openclaw/pull/97085 rather than being implemented on main.

## What This Changes

The issue asks OpenClaw to add subjectAltName entries to auto-generated gateway TLS certificates so strict TLS clients can verify local gateway hostnames and IPs.

## Best Possible Solution

Land a focused gateway TLS fix that generates standards-valid SANs for OpenClaw-owned self-signed certificates and handles existing CN-only material through a maintainer-approved ownership or doctor-repair path.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 invoke OpenSSL with CN only, while Node's TLS identity check requires IP SAN entries for IP hostnames. I did not run a full gateway plus strict-client handshake in this read-only review.

## Solution Assessment

Yes on the implementation layer: the generator should emit SANs rather than relying on client-side trust workarounds. The exact SAN set and any legacy certificate rewrite path still need maintainer/security review before the candidate fix is the best complete solution.
