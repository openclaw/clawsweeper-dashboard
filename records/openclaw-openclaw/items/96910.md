## Summary

Keep open: current main and v2026.6.10 still generate CN-only gateway self-signed certificates, and the linked fix at https://github.com/openclaw/openclaw/pull/96919 is still open, so it is an implementation candidate rather than completed work.

## What This Changes

The issue reports that auto-generated gateway TLS certificates omit subjectAltName entries, causing strict TLS clients to reject local gateway probes by IP or hostname.

## Best Possible Solution

Update the gateway TLS generator, add a narrow repair or doctor path for existing OpenClaw-generated CN-only certs without touching custom certs, and prove strict TLS clients accept the supported local identities.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 invoke OpenSSL with CN only and no SAN, while Node's TLS identity check expects matching SAN DNS/IP entries. I did not run a full gateway handshake in this read-only review.

## Solution Assessment

Yes on the main layer: the durable fix belongs in gateway certificate generation rather than another client-side trust workaround. The complete solution still needs maintainer confirmation for the SAN inventory and existing-cert repair behavior.
