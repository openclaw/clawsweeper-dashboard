## Summary

Keep open: current main and v2026.6.10 still generate CN-only gateway TLS certificates with no SAN, and open linked pull requests are candidate fixes rather than merged implementation evidence.

## What This Changes

The issue reports that OpenClaw’s auto-generated gateway TLS certificate lacks subjectAltName entries, causing strict TLS hostname/IP verification to fail for gateway clients.

## Best Possible Solution

Ship a reviewed gateway TLS fix that adds appropriate SANs for generated certificates and has an approved, tested strategy for existing OpenClaw-generated CN-only certs while preserving user-provided custom cert material.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 generate a CN-only certificate with no SAN, and Node’s TLS identity check requires IP SAN entries for IP hostnames.

## Solution Assessment

Yes on the main implementation layer, but the complete fix still needs maintainer-approved SAN scope and upgrade behavior. The narrow owner boundary is `src/infra/tls/gateway.ts` plus focused TLS regression coverage.
