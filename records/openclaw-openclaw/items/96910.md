## Summary

Keep open: current main and v2026.6.10 still generate CN-only gateway self-signed certificates, and open pull request https://github.com/openclaw/openclaw/pull/97085 is already the active candidate fix that needs maintainer/security review.

## What This Changes

The issue reports that auto-generated gateway TLS certificates lack subjectAltName entries, causing strict TLS clients that connect by loopback IP or hostname to reject the certificate.

## Best Possible Solution

Ship a standards-valid gateway TLS certificate path with approved generated SANs and an explicit, tested upgrade or doctor strategy for old OpenClaw-generated CN-only certs while preserving custom certificate material.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 generate a CN-only cert with no SAN, and a scratch cert generated with the same arguments fails Node strict identity checking for 127.0.0.1.

## Solution Assessment

Yes on the likely layer, but not fully settled. The fix belongs in the gateway TLS generator plus an upgrade-safe repair path, while the exact SAN set and old-cert replacement policy need maintainer/security acceptance.
