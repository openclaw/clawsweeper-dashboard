## Summary

Keep open: current main and v2026.6.10 still generate CN-only gateway self-signed certificates, and https://github.com/openclaw/openclaw/pull/96919 is an open candidate fix rather than a merged implementation.

## What This Changes

The issue asks for auto-generated gateway self-signed TLS certificates to include subjectAltName entries so strict TLS clients can verify loopback and host identities.

## Best Possible Solution

Fix gateway TLS generation and upgrade repair together, then land a corrected version of https://github.com/openclaw/openclaw/pull/96919 or a replacement with focused SAN and strict-TLS proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 generate a CN-only self-signed gateway certificate, and Node's TLS identity check requires an IP SAN for 127.0.0.1. I did not run a full gateway handshake in this read-only review.

## Solution Assessment

The right layer is the gateway TLS certificate generator plus an upgrade-safe repair path, not another broad client-side trust workaround. The exact SAN identity set still needs maintainer/security confirmation.
