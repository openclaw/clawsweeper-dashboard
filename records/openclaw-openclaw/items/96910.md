## Summary

Keep open: current main and v2026.6.10 still generate CN-only gateway self-signed certificates, and an open candidate PR is already linked with closing syntax, so the issue should wait for that PR's maintainer/security decision rather than be closed or queued again.

## What This Changes

The issue asks OpenClaw to add subjectAltName identities to auto-generated gateway TLS certificates so strict TLS clients can verify local gateway hostnames and IPs.

## Best Possible Solution

Ship a standards-valid gateway TLS certificate path with approved generated SANs and an explicit, tested upgrade or doctor strategy for old OpenClaw-generated CN-only certs while preserving custom cert material.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 generate a CN-only cert with no SAN, and Node's TLS identity check requires IP SAN entries for IP hostnames.

## Solution Assessment

Yes on the likely layer, but not fully settled. The fix belongs in the gateway TLS generator plus an upgrade-safe repair path, while the exact SAN set and old-cert replacement policy need maintainer/security acceptance.
