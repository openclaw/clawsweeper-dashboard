## Summary

Keep open: current main still has a source-reproducible reply-dispatcher path where standalone `MEDIA:` text can reach channel delivery without being promoted to `mediaUrl` or `mediaUrls`. The closed related reports cover parser pollution and generic outbound message delivery, but they do not fix this legacy reply-dispatch boundary.

## What This Changes

The issue reports that standalone `MEDIA:` directives can reach the Weixin channel as plain text when reply payload media fields are absent.

## Best Possible Solution

Normalize reply directives at the shared dispatcher or legacy plugin delivery boundary using `normalizeReplyPayloadDirectives`, preserving existing payload media fields and delivery ordering.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence path: a payload such as `{ text: "caption\nMEDIA:/tmp/a.png" }` sent through `createReplyDispatcherWithTyping` reaches `options.deliver` after `normalizeReplyPayload`, which does not parse `MEDIA:` into media fields.

## Solution Assessment

Yes. The narrowest maintainable fix is channel-agnostic normalization before legacy/plugin delivery, not a Weixin-specific parser in OpenClaw core; existing block and agent-runner paths already use the shared directive normalizer.
