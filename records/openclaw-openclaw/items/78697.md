## Summary

Keep open: current main still has a legacy/plugin reply-dispatch path where a raw standalone `MEDIA:` line can reach `deliver` without being promoted to `mediaUrl` or `mediaUrls`. The normal agent-runner and durable outbound paths already parse media directives, so the remaining gap is narrow and channel-agnostic rather than a Weixin-specific core special case.

## What This Changes

The issue reports that standalone `MEDIA:` directives in assistant reply text can reach the Weixin channel without normalized media fields before delivery.

## Best Possible Solution

Normalize reply directives at the shared dispatcher or legacy plugin delivery boundary before any `deliver` callback, reusing the existing parser and preserving already-normalized payload fields.

## Reproduction Assessment

Yes, source-reproducible: a payload like `{ text: "caption\nMEDIA:/tmp/a.png" }` sent directly through `createReplyDispatcherWithTyping` reaches `deliver` after `normalizeReplyPayload`, which does not parse `MEDIA:` into media fields. The published Weixin deliver callback then sees no `mediaUrl`/`mediaUrls` and takes its text-only path.

## Solution Assessment

Yes. The narrowest maintainable fix is channel-agnostic shared normalization before legacy/plugin delivery, not a Weixin-specific parser in core; the existing agent-runner and outbound helpers show the parser contract to reuse.
