## Summary

Keep open. Current main still renders the Agents page selector as a native `select` with only a `change` handler, and the app-level selection path still relies on receiving a changed value rather than any pointer-open/listbox behavior. The related Agents-page UX trackers remain open and do not prove this automation-specific behavior has been fixed or declared unsupported.

## What This Changes

The issue asks for the Control UI Agents selector to respond to simulated mouse/click events so browser automation can open the selector and change the selected agent.

## Best Possible Solution

Resolve the Agents-page selection UX at the source: either replace the dropdown with an accessible visible agent list/listbox that supports pointer automation, keyboard navigation, and current selection semantics, or explicitly document the supported automation path if maintainers decide native-select click-to-open is outside the support contract.

## Reproduction Assessment

Unclear. The issue gives concrete browser-automation steps and current source still matches the reported native-select/change-only design, but this read-only review did not run a live Control UI browser automation session to observe the popup behavior.

## Solution Assessment

Unclear. Replacing the native dropdown with an accessible list/listbox would solve this issue and align with #57067, but maintainers first need to decide whether simulated click-to-open on a native select is a supported UI contract or whether automation should use value/change/selectOption APIs.
