## Summary

Keep open: current main still has a source-proven Telegram mismatch where documented/config-accepted access group aliases are treated as invalid Telegram sender IDs before runtime authorization can use them.

## What This Changes

The issue reports that Telegram allowFrom/groupAllowFrom rejects documented access group references such as accessGroup:usuarios_basicos.

## Best Possible Solution

Wire Telegram inbound DM and group sender authorization to resolve message.senders accessGroup aliases before numeric Telegram sender matching, while preserving fail-closed behavior for missing groups and arbitrary nonnumeric Telegram entries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure channels.telegram.allowFrom or groupAllowFrom with accessGroup:<name>, then Telegram normalizes that literal as an invalid nonnumeric sender entry before runtime allowlist authorization can match the access group.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing access-group expansion contract in Telegram-owned access checks, not to loosen Telegram raw sender ID validation or move policy into core.
