## Summary

Keep open: current main still leaves the `notify=false` leak source-reproducible, and https://github.com/openclaw/openclaw/pull/80737 is the active closing PR with live Telegram proof.

## What This Changes

The issue asks OpenClaw to treat a trailing standalone `notify=false` in Telegram heartbeat/completion replies as silent-delivery metadata instead of visible message text.

## Best Possible Solution

Land or repair https://github.com/openclaw/openclaw/pull/80737, or an equivalent maintainer branch, so only standalone trailing `notify=false` is stripped, `silent: true` reaches durable delivery, marker-only replies are suppressed, inline text is preserved, and regression coverage covers heartbeat plus completion fallback paths.

## Reproduction Assessment

Yes, source-level: `normalizeHeartbeatReply` only strips heartbeat ack tokens and the heartbeat durable send uses `normalized.text` without `silent`, so text ending in standalone `notify=false` can be sent visibly on current main. I did not run live Telegram in this read-only review.

## Solution Assessment

Yes, the narrow maintainable solution is to keep structured `heartbeat_respond` canonical while adding defensive standalone trailing-marker conversion in the affected fallback path. The linked PR is the right implementation lane, but this issue should remain open until it lands.
