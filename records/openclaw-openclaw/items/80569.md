## Summary

Keep open: current main still has the reported source-level Telegram delivery gap, and every direct fix PR that referenced this issue is closed unmerged. The code strips heartbeat ack tokens and honors structured `silent`, but it does not consume a trailing standalone `notify=false` marker before Telegram delivery.

## What This Changes

The issue requests that Telegram replies ending with a standalone `notify=false` marker not expose that marker and instead deliver silently or be suppressed when appropriate.

## Best Possible Solution

Land one narrow fix that strips only a final standalone marker before Telegram fan-out/native delivery, maps it to existing `silent` metadata for every send in the payload, preserves inline text, and includes focused tests plus redacted live Telegram proof.

## Reproduction Assessment

Yes at source level: current main forwards fallback text unchanged and Telegram only maps structured `silent` to `disable_notification`. I did not run a fresh live Telegram reproduction in this read-only pass.

## Solution Assessment

Yes, a narrow delivery-boundary normalization is the best direction. Patching only one low-level helper is too narrow because media fan-out and native replies can bypass or split the text before every send receives silent metadata.
