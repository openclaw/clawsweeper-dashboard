## Summary

This looks like a valid current-main bug, not a cleanup-close candidate: OpenClaw has structured silent-delivery paths, but a fallback heartbeat/completion final reply ending in `notify=false` is still treated as ordinary visible text and sent without a silent flag.

## What This Changes

The issue asks OpenClaw to stop leaking a trailing `notify=false` delivery-control marker into Telegram messages and to honor silent or no-reply intent for Telegram heartbeat/completion replies.

## Best Possible Solution

Keep structured `heartbeat_respond` and message-tool `silent` metadata as canonical, and add a narrow fallback that strips a trailing standalone `notify=false` from heartbeat/completion final text while applying silent delivery or suppressing no-attention output.

## Reproduction Assessment

Yes, at source level: a heartbeat or completion fallback reply whose text ends with standalone `notify=false` is normalized as ordinary text and sent without `silent`. I did not perform a live Telegram run, but the current code path is clear enough for a focused regression test.

## Solution Assessment

Yes, a narrow repair in the existing heartbeat/completion delivery normalization is the best direction. The safer alternative is not a broad global text convention, but preserving structured metadata and only handling the legacy-looking trailing marker where OpenClaw itself prompts for it.
