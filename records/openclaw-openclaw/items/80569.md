## Summary

Keep open: current main and the latest release still lack fallback `notify=false` normalization, the related fix PRs are closed unmerged, and Codex does not provide an upstream Telegram notification metadata path.

## What This Changes

The issue asks OpenClaw to treat trailing `notify=false` in Telegram completion or heartbeat replies as silent-delivery metadata instead of visible message text.

## Best Possible Solution

Land a narrow OpenClaw heartbeat fallback normalizer that strips only standalone trailing `notify=false`, preserves inline text, suppresses marker-only fallbacks, and forwards existing `silent` delivery metadata.

## Reproduction Assessment

Yes at source level: current main preserves non-`HEARTBEAT_OK` fallback text, `normalizeHeartbeatReply` has no `notify=false` parser, and the durable send does not pass `silent` for that payload. I did not run live Telegram in this read-only review.

## Solution Assessment

Yes. Codex app-server messages are plain text without notification metadata, while OpenClaw already has a structured heartbeat tool and Telegram `silent` send contract, so the narrow OpenClaw fix is to normalize the legacy fallback marker before durable delivery.
