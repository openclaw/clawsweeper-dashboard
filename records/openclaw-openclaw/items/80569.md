## Summary

Keep open: current main still lacks standalone trailing `notify=false` consumption and silent forwarding in the heartbeat fallback path, and the open closing PR remains the canonical implementation path.

## What This Changes

The issue reports Telegram heartbeat/completion replies leaking a trailing `notify=false` marker into visible text instead of applying silent-delivery metadata.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80737 or an equivalent canonical fix that strips only standalone trailing `notify=false`, forwards `silent: true`, suppresses marker-only replies, preserves inline text, and covers heartbeat plus exec-completion fallback tests.

## Reproduction Assessment

Yes, as a source-level reproduction: current main's heartbeat fallback normalizer only strips heartbeat ack tokens, and the durable send builds payloads without a `silent` flag, so text ending in standalone `notify=false` would be delivered as normal text. I did not run a live Telegram current-main repro.

## Solution Assessment

Yes: the narrow maintainable direction is to keep structured `heartbeat_respond` canonical and add defensive standalone trailing-marker conversion in the affected fallback path. The open linked PR already targets that boundary and should be reviewed, repaired, or landed before this issue closes.
