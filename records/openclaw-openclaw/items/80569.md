## Summary

Keep open: current main has structured Telegram silent-delivery plumbing, but a plain trailing `notify=false` heartbeat/completion marker is still not stripped or converted before Telegram delivery, so the reported user-visible leak remains plausible from source inspection.

## What This Changes

The issue requests that Telegram heartbeat/completion replies honor silent notification intent by suppressing delivery or converting a trailing `notify=false` marker into silent-send metadata instead of showing it in the message body.

## Best Possible Solution

Keep structured `heartbeat_respond` metadata as canonical, and add a narrow defensive normalizer that strips only a standalone trailing `notify=false` from affected heartbeat/completion output and either suppresses the no-op reply or sends the remaining Telegram reply with `silent: true`.

## Reproduction Assessment

No high-confidence live reproduction was run. The issue has a concrete observed Telegram symptom, and current-main source shows a medium-confidence path where a plain heartbeat/completion reply ending in `notify=false` would remain normal deliverable text.

## Solution Assessment

Yes, the narrow repair direction is the best fit. The fix should preserve structured delivery metadata as the primary contract and only treat a standalone trailing `notify=false` as a defensive legacy/control marker in the implicated delivery path.
