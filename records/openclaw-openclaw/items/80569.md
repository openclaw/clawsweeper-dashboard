## Summary

Keep open: current main still lets heartbeat fallback text carry a standalone trailing `notify=false` marker into durable delivery without a silent flag, and the open closing PR is the active implementation path.

## What This Changes

The issue reports that Telegram heartbeat or completion replies ending with a standalone `notify=false` marker are sent with normal notifications and show the marker in the visible message body.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80737 or an equivalent canonical fix that strips only standalone trailing `notify=false`, forwards `silent: true`, suppresses marker-only replies, preserves inline text, and covers heartbeat plus exec-completion fallback tests.

## Reproduction Assessment

Yes, as a source-level reproduction: current main's fallback normalizer does not parse `notify=false`, and the durable send path forwards only text/media without `silent`, so a standalone trailing marker remains visible and loud. I did not run a live Telegram repro against current main.

## Solution Assessment

Yes, the best direction is a narrow fallback-boundary fix that keeps structured `heartbeat_respond` canonical while defensively converting only a final standalone legacy marker into silent delivery metadata. The linked PR already owns that path and should be reviewed, repaired, or replaced before closing this issue.
