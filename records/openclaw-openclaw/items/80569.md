## Summary

Current main has structured Telegram silent-delivery plumbing, but the reported plain-text `notify=false` marker is still only present in prompts/tests and is not consumed before delivery, so this looks like a valid open bug rather than cleanup-close material.

## What This Changes

Telegram replies should consume a silent notification intent instead of displaying a trailing `notify=false` marker and sending a normal notification.

## Best Possible Solution

Keep structured `heartbeat_respond` and message-tool metadata as the primary contract, and add a narrow defensive path that strips a standalone trailing `notify=false` from heartbeat/completion output and either suppresses a no-op report or sends the remaining Telegram reply with `silent: true`.

## Reproduction Assessment

No live bot reproduction was run in this read-only review. Source inspection gives a medium-confidence reproduction path: current main prompts models with `notify=false`, supports structured silent delivery, but has no delivery normalizer for a plain trailing marker.

## Solution Assessment

Yes, a narrow source-level repair is the right direction. The fix should preserve structured metadata as canonical and only treat a standalone trailing `notify=false` as a legacy/defensive control marker in the affected heartbeat or completion delivery path.
