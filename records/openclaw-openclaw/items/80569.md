## Summary

Keep open: current main still has the source-level gap where a plain heartbeat/completion reply ending in `notify=false` is normalized as text and sent without `silent`, and the linked fix PR is still open with closing syntax for this issue.

## What This Changes

The issue reports Telegram heartbeat/completion replies leaking a trailing `notify=false` marker into chat text instead of converting it to silent delivery metadata.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80737 after normal CI and maintainer review, keeping structured `heartbeat_respond` canonical and limiting fallback parsing to standalone trailing control markers.

## Reproduction Assessment

Yes, by source-level reproduction: make the heartbeat fallback/plain reply return text ending in a standalone `notify=false`; current main strips only heartbeat ack tokens and sends the batch without `silent`. I did not run a live current-main Telegram replay in this read-only review.

## Solution Assessment

Yes. The linked PR’s approach is the narrowest maintainable fix because it consumes only a standalone trailing marker in the implicated heartbeat/completion fallback path and reuses the existing durable `silent` delivery contract instead of adding a global text parser.
