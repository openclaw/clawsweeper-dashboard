## Summary

Keep open: current main still lacks marker consumption and silent forwarding for heartbeat fallback text, and an open linked fix PR is the active implementation path. This issue should remain open until https://github.com/openclaw/openclaw/pull/80737 lands or maintainers choose another canonical fix.

## What This Changes

Telegram heartbeat/completion replies should treat a standalone trailing `notify=false` as silent-delivery metadata instead of visible message text.

## Best Possible Solution

Land one canonical fix through https://github.com/openclaw/openclaw/pull/80737 or an equivalent maintainer branch that strips only standalone trailing `notify=false`, forwards `silent: true`, suppresses marker-only replies, preserves inline text, and covers heartbeat plus exec-completion fallback tests.

## Reproduction Assessment

Yes, as a source-level reproduction: current main's heartbeat fallback normalizer only strips heartbeat ack tokens, and the durable send builds payloads without a silent flag, so text ending in standalone `notify=false` would be delivered as normal text. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is to keep structured `heartbeat_respond` as canonical and add a defensive standalone trailing-marker conversion only in the affected heartbeat/completion fallback path. The open linked PR already targets that boundary, but it should be tightened and landed before this issue closes.
