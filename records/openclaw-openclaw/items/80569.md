## Summary

Keep open: current main still has the source-level Telegram heartbeat fallback gap, and the open canonical fix PR references this issue with closing syntax, so the issue should stay open until that PR lands or is closed.

## What This Changes

The issue requests that Telegram heartbeat or completion replies consume a standalone trailing `notify=false` marker as silent-delivery metadata instead of sending it as visible text.

## Best Possible Solution

Land one canonical fix through https://github.com/openclaw/openclaw/pull/80737 or a replacement that strips only standalone trailing markers, forwards `silent` through durable delivery, covers exec-completion and terminal-newline cases, and preserves inline user text.

## Reproduction Assessment

No high-confidence live Telegram reproduction was run here. The current-main source path is clear: heartbeat fallback text can keep a trailing `notify=false` marker and then be sent without `silent` metadata.

## Solution Assessment

Yes, the canonical repair direction is the right owner-boundary fix. Keep structured `heartbeat_respond` as primary and add a narrow heartbeat/completion fallback normalizer instead of parsing all Telegram text broadly.
