## Summary

Keep open: current main still leaves the heartbeat fallback `notify=false` marker as ordinary text and sends the durable Telegram reply without silent metadata. The issue already has an open closing PR, https://github.com/openclaw/openclaw/pull/80737, so this should stay open until that canonical fix or an equivalent maintainer branch lands.

## What This Changes

The issue asks OpenClaw to consume a trailing standalone `notify=false` intent for Telegram heartbeat/completion replies instead of rendering it in the message body and sending a normal notification.

## Best Possible Solution

Land one canonical fix through https://github.com/openclaw/openclaw/pull/80737 or an equivalent maintainer branch that strips only standalone trailing `notify=false`, forwards `silent: true`, suppresses marker-only replies, preserves inline text, and covers heartbeat plus exec-completion fallback tests.

## Reproduction Assessment

Yes, as a source-level reproduction: current main's heartbeat fallback normalizer leaves a standalone trailing `notify=false` in `normalized.text`, and the durable send call does not pass `silent`. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable path is to keep structured `heartbeat_respond` as canonical and add a defensive fallback conversion only for standalone trailing markers on the affected heartbeat/completion delivery path. The open linked PR is the right canonical branch to review and land after any remaining parser fixes.
