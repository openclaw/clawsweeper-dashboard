## Label Justifications

- `P2`: The issue describes a real session-memory persistence gap with bounded blast radius and active implementation PRs awaiting maintainer choice.
- `impact:data-loss`: Automatic daily or idle rollovers can discard prior conversation context without writing the expected session-memory snapshot.
- `impact:session-state`: The affected behavior is session rollover state, previous transcript handling, and memory persistence across session boundaries.

## Summary

Keep open because current main still preserves and archives stale daily/idle rollover state but does not route automatic rollovers through the reset persistence hook surface; the open linked PRs need maintainer contract selection rather than issue cleanup.

## What This Changes

Requests a reset hook or equivalent persistence surface so session-memory and custom hooks can save previous session context during automatic daily or idle rollovers.

## Best Possible Solution

Ship one canonical reset-persistence contract that persists bundled session-memory exactly once for automatic daily and idle rollovers, preserves manual reset and transcript archive behavior, documents the selected hook shape, and adds focused daily plus idle regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure daily or idle reset, let an existing session become stale, then send a normal message; `initSessionState` creates a new session and emits lifecycle hooks only, while session-memory ignores those events.

## Solution Assessment

Unclear. The persistence gap is real, but the fix changes the hook contract and two open PRs propose overlapping approaches, so maintainers should choose the canonical public/internal surface before merge.
