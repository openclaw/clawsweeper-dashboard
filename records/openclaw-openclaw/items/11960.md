## Label Justifications

- `impact:data-loss`: Automatic daily or idle rollovers can discard prior conversation context without writing the expected session-memory snapshot.

## Summary

Keep open because current main still has the reported reset-persistence gap: automatic daily/idle rollovers capture and archive prior session state, but only emit lifecycle hooks that session-memory does not consume. The linked implementation PRs are open and conflicting, so the next step is maintainer contract selection rather than closing this issue.

## What This Changes

The issue requests a session reset or before-reset hook for automatic daily and idle rollovers so session-memory and custom hooks can persist context before the old session is cleared.

## Best Possible Solution

Ship one canonical reset-persistence contract that persists session-memory exactly once for automatic daily and idle rollovers, preserves manual reset and transcript archive behavior, documents the selected hook shape, and adds focused daily plus idle regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure daily or idle reset, let an existing session become stale, then send a normal message; initSessionState creates a new session and emits lifecycle hooks only, while session-memory ignores those events.

## Solution Assessment

Unclear. The persistence gap is real, but the solution changes the hook/session lifecycle contract and two open PRs propose overlapping approaches, so maintainers should select the canonical public/internal surface before merge.
