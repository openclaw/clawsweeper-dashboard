- `impact:session-state`: The request concerns transcript, memory, and context preservation when session state rotates automatically.

## Summary

Keep open: current main and the latest release still leave bundled session-memory on the command-only reset path, while automatic daily/idle rollovers only reach session lifecycle hooks. The open linked implementation PRs should be reviewed, landed, replaced, or rejected before this issue closes.

## What This Changes

The issue requests a reset-time hook surface so session-memory and custom hooks can persist previous session context before automatic daily or idle resets clear it.

## Best Possible Solution

Land or replace one canonical implementation that lets automatic daily and idle rollovers persist previous session context exactly once without surprising broad command hooks or changing manual reset semantics.

## Reproduction Assessment

Yes, at source level. Current main can roll over stale daily/idle sessions and emit session_end/session_start, while bundled session-memory still accepts only command:new and command:reset.

## Solution Assessment

Unclear as a final API shape. The persistence gap is real, but maintainers need to choose whether the durable contract is before_reset, session_end, a new reset event, or a narrow internal session-memory dispatch.
