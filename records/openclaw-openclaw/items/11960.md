## Summary

Keep open. Current main and the latest release still leave the primary auto daily/idle reset persistence hook gap in place, and https://github.com/openclaw/openclaw/pull/71792 explicitly closes this issue while https://github.com/openclaw/openclaw/pull/61675 overlaps, so the cleanup path is maintainer selection of an existing implementation rather than closing this issue now.

## What This Changes

Add a persistence-capable reset lifecycle notification for automatic daily and idle session rollovers, including the previous session context and reset reason.

## Best Possible Solution

Ship one maintainer-approved implementation, from https://github.com/openclaw/openclaw/pull/71792, https://github.com/openclaw/openclaw/pull/61675, or a replacement, that persists bundled session-memory exactly once for automatic daily and idle rollovers, exposes the chosen reset hook contract, and includes daily plus idle regression coverage and real rollover proof.

## Reproduction Assessment

Yes at source level, but not from a live run in this review. Current main can roll a stale daily/idle session through `initSessionState`, while bundled session-memory still only accepts `command:new` and `command:reset` events.

## Solution Assessment

No final implementation is on main yet. The narrowest maintainable path is for maintainers to choose one reset hook contract and land one of the existing PRs or a replacement with no double-fire behavior and focused daily/idle coverage.
