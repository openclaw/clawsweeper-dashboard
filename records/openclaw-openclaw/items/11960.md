## Summary

Keep open. Current main still reproduces the core gap: automatic daily/idle rollover has the previous session state available and emits lifecycle hooks, but it does not invoke the reset persistence surface that session-memory listens to; open implementation PRs already target the issue, so this should stay with maintainer review rather than be closed or queued again.

## What This Changes

The issue asks for a session reset or before-reset hook so session-memory and custom hooks can persist context during automatic daily/idle resets, plus a hook-config schema allowance for handler-specific keys.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71792, https://github.com/openclaw/openclaw/pull/61675, or a maintainer-approved replacement that persists session-memory exactly once for automatic daily and idle rollovers, documents the chosen hook contract, and adds regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an existing session to become stale by daily or idle policy and send the next inbound message; current main starts a new session and lifecycle hooks, while session-memory remains limited to command:new and command:reset.

## Solution Assessment

Unclear. The requested persistence gap is real, but the best maintainable fix needs maintainer selection because the open branches propose different reset hook/API shapes.
