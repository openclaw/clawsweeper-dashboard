## Label Justifications

- `P2`: The missing persistence callback is a normal-priority session behavior gap with meaningful but bounded impact.
- `impact:session-state`: Automatic rollover replaces the active session without invoking the persistence surface used at manual session boundaries.
- `impact:data-loss`: Daily or idle rollover can silently discard context that the bundled memory hook has not persisted.

## Summary

Current main still does not route automatic daily or idle rollover through the reset-persistence surface used by manual resets. The hook-config schema mismatch is fixed, but an open closing PR already owns the remaining work, so this issue should stay open pending maintainer selection and landing of a canonical implementation.

## What This Changes

Add a session reset lifecycle surface so automatic daily and idle rollovers can persist prior-session context, and permit hook-specific configuration fields.

## Best Possible Solution

Land a focused maintainer-approved implementation that routes daily and idle rollover through one canonical pre-reset persistence surface, passes the previous entry and stable transcript context, preserves typed lifecycle hooks, and proves both automatic modes without duplicating manual resets.

## Reproduction Assessment

Yes, from current-main source with high confidence: a stale daily or idle session follows transcript archival plus session_end/session_start, while bundled session-memory remains attached to manual command events. This read-only review did not execute a live rollover.

## Solution Assessment

Unclear as proposed: the desired persistence behavior is well supported, but adding a brand-new session:reset event is not yet proven better than extending the existing before_reset or internal reset surfaces. Maintainers should choose one canonical contract and avoid parallel hook semantics.
