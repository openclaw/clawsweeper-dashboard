## Label Justifications

- `P2`: The issue describes a real session memory persistence gap with meaningful but bounded impact and active implementation PRs.
- `impact:data-loss`: Automatic daily/idle rollovers can discard prior conversation context without a session-memory file being written.
- `impact:session-state`: The affected behavior is session rollover state, transcript archival, and memory persistence across session boundaries.

## Summary

Keep open because current main still captures automatic daily/idle rollover state but does not route that rollover through the reset persistence hook surface, and an open PR explicitly closes this issue while another open PR overlaps the same contract work.

## What This Changes

The issue requests a session reset hook or equivalent persistence surface so bundled and custom hooks can save memory when automatic daily or idle session rollovers occur.

## Best Possible Solution

Ship one canonical implementation that persists session-memory exactly once for automatic daily and idle rollovers, preserves transcript archival/manual reset behavior, documents the selected hook contract, and adds focused daily plus idle regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure daily or idle reset, let an existing session go stale, then send the next user message; initSessionState captures previousSessionEntry but only emits session_end/session_start while session-memory ignores non-command events.

## Solution Assessment

Unclear. The persistence gap is real, but the open implementation PRs propose overlapping contracts, so maintainers should choose the canonical reset hook surface before merge.
