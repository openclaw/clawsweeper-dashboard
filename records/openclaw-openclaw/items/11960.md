## Summary

Keep open. Current main still leaves automatic daily/idle rollovers on the session lifecycle path while bundled session-memory only runs for command:new and command:reset; the config-schema mismatch is fixed, but the reset-persistence hook/API contract remains unresolved and an open candidate PR still needs maintainer selection.

## What This Changes

The issue requests a session reset hook or equivalent pre-reset persistence surface so automatic daily and idle session rollovers can flush bundled session-memory before context is cleared, plus schema support for per-hook configuration.

## Best Possible Solution

Land, replace, or explicitly reject one canonical implementation that persists session-memory on automatic daily/idle rollover without surprising existing command hooks or losing lifecycle-hook behavior.

## Reproduction Assessment

Yes, at source level. Current main can mark daily/idle sessions stale and create a new session while session-memory only reacts to command:new and command:reset, so the automatic rollover path has no memory persistence hook.

## Solution Assessment

Unclear pending maintainer product/API choice. The source gap is real, but maintainers still need to choose whether the durable contract should be synthetic command events, before_reset on automatic rollover, a new typed session reset event, or session_end-based memory behavior.
