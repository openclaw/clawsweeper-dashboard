## Label Justifications

- `P2`: This is a normal-priority session-memory persistence gap with meaningful but bounded impact on users relying on automatic daily or idle rollovers.
- `impact:data-loss`: Automatic rollovers can discard previous conversation context from the memory persistence path without writing the expected memory snapshot.
- `impact:session-state`: The affected behavior is session rollover state, previous transcript handling, and continuity across session boundaries.

## Summary

Keep open: current main and v2026.6.5 still leave automatic daily/idle rollovers outside the reset persistence surface used by manual resets, while open implementation PRs already target the gap and need maintainer contract selection.

## What This Changes

Request to add reset-hook coverage for automatic daily/idle session rollovers so session-memory and custom hooks can persist previous context before reset, plus a secondary hook-config schema mismatch report.

## Best Possible Solution

Ship one canonical reset-persistence contract that persists bundled session-memory exactly once for automatic daily and idle rollovers, preserves manual reset and transcript archive behavior, documents the chosen hook shape, and adds focused daily plus idle regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure daily or idle reset, let an existing session become stale, then send a normal message; initSessionState creates a new session and emits lifecycle hooks only while session-memory ignores those events.

## Solution Assessment

Unclear. The persistence gap is real, but the requested hook/API surface has overlapping open PRs with different contract choices, so maintainers should select the canonical approach before merge.
