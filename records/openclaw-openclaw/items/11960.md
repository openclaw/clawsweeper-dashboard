## Summary

Keep open. Current main and v2026.6.5 still leave automatic daily/idle rollovers on the typed session lifecycle path while bundled session-memory remains command-only; the secondary hook-config schema mismatch is fixed, but the requested reset persistence surface is still unlanded. The issue is paired with open implementation PRs at https://github.com/openclaw/openclaw/pull/71792 and https://github.com/openclaw/openclaw/pull/61675, so maintainers should choose, land, replace, or reject that path before closing it.

## What This Changes

Request to add a reset or before-reset hook for automatic daily/idle session resets so session-memory and custom hooks can persist previous context before rollover, plus allow hook-specific config keys.

## Best Possible Solution

Land or replace the open fix PRs with one maintainer-approved reset-persistence contract that saves session-memory exactly once on automatic daily/idle rollovers, carries the previous transcript/context, preserves typed lifecycle hooks, and adds daily plus idle regression coverage.

## Reproduction Assessment

Yes, source-level: stale daily/idle rollovers capture and archive the old session, then emit typed session lifecycle hooks, while session-memory only handles command new and reset. I did not run a live reset scenario because this review is read-only, but the current-main path is clear.

## Solution Assessment

Unclear as a product/API decision: adding a named reset event is plausible, but current code also has before_reset and session_end surfaces that overlap. The best fix should choose one maintainable reset-persistence contract and avoid duplicate or double-fired hooks.
