## Label Justifications

- `P2`: The issue describes a real session-memory persistence gap with bounded blast radius and active implementation PRs awaiting maintainer choice.
- `impact:data-loss`: Automatic daily or idle rollovers can discard prior conversation context without writing the expected session-memory snapshot.
- `impact:session-state`: The affected behavior is session rollover state, previous transcript handling, and memory persistence across session boundaries.

## Summary

Keep open: current main and the latest release still route automatic daily/idle rollovers through session lifecycle hooks only, while bundled session-memory still listens only to command:new and command:reset. An open PR directly closes this issue and another open PR overlaps, so the next step is maintainer selection of the hook contract rather than cleanup closure or a new fix lane.

## What This Changes

The issue requests a session reset hook or equivalent pre-reset persistence surface so bundled session-memory and custom hooks can save context before automatic daily or idle session resets clear the previous session.

## Best Possible Solution

Ship one canonical reset-persistence contract that persists bundled session-memory exactly once for automatic daily and idle rollovers, preserves manual reset and transcript archive behavior, documents the chosen hook shape, and adds focused daily plus idle regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure daily or idle reset, let an existing session become stale, then send a normal message; initSessionState creates a new session and emits lifecycle hooks only, while session-memory ignores those events.

## Solution Assessment

Unclear. The persistence gap is real, but solving it changes the hook/session lifecycle contract and two open PRs propose overlapping approaches, so maintainers should select the canonical surface before merge.
