## Label Justifications

- `P2`: Automatic rollover can repeatedly lose memory continuity for affected users, but the problem has a bounded session-lifecycle scope.
- `impact:data-loss`: Conversation context can be discarded at automatic rollover without the bundled memory artifact being written.
- `impact:session-state`: The missing persistence surface occurs while replacing stale daily and idle sessions.

## Summary

Current main and the latest release still rotate stale daily or idle sessions without invoking the bundled session-memory persistence path or plugin `before_reset`; the hook-config schema mismatch is fixed, but the primary gap remains and two open conflicting PRs require maintainer selection.

## What This Changes

Add a reset lifecycle surface so automatic daily and idle session rollovers can persist the previous session context, and allow hook-specific configuration keys.

## Best Possible Solution

Refresh one canonical implementation that makes daily and idle rollover invoke the maintainer-approved pre-reset persistence contract with the previous entry and transcript, while preserving typed lifecycle hooks and adding daily, idle, and no-double-fire regression coverage.

## Reproduction Assessment

Yes, at source level with high confidence: configure a stale daily or idle session and trigger lazy rollover on the next user message; current main archives and replaces the session while bypassing both the internal reset persistence event and plugin `before_reset`.

## Solution Assessment

Unclear as proposed: persisting automatic rollover is the right outcome, but maintainers must choose whether the canonical contract reuses the internal command hook, extends plugin `before_reset`, introduces a dedicated typed reset event, or combines narrowly compatible surfaces.
