## Summary

Keep open. Current main still captures the previous session on automatic daily/idle rollover but only emits session lifecycle hooks, while bundled session-memory still accepts only command:new and command:reset; https://github.com/openclaw/openclaw/pull/71792 explicitly closes this issue and https://github.com/openclaw/openclaw/pull/61675 overlaps, so the issue should stay open until maintainers land, replace, or reject one implementation.

## What This Changes

Requests a session reset hook so automatic daily/idle resets can persist session memory, and notes a hook-specific config schema mismatch for session-memory messages.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71792, https://github.com/openclaw/openclaw/pull/61675, or a maintainer-approved replacement that persists session-memory exactly once for automatic daily and idle rollovers and documents the chosen hook contract with regression coverage.

## Reproduction Assessment

Yes, by source inspection. Configure an existing session to become stale by daily or idle policy and send the next inbound message; current main creates a new session and lifecycle hooks while bundled session-memory remains limited to command:new and command:reset.

## Solution Assessment

Unclear. The requested persistence gap is real, but the best fix needs maintainer selection because the open branches propose different reset hook/API shapes.
