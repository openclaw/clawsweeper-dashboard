## Label Justifications

- `P2`: The issue affects session-memory continuity for daily/idle rollover users, but the blast radius is bounded to session lifecycle and hook behavior.
- `impact:data-loss`: Automatic daily or idle rollover can leave useful conversation context unsaved before the previous session is replaced.

## Summary

Keep open: current main and latest release still keep bundled session-memory scoped to command-driven resets, while automatic daily/idle rollovers only emit typed session lifecycle hooks; the open implementation PRs are candidates, not merged fixes.

## What This Changes

The issue requests a session reset hook or before-reset lifecycle surface so automatic daily and idle session rollovers can persist session memory before context is cleared.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71792, https://github.com/openclaw/openclaw/pull/61675, or a maintainer-approved replacement that persists session-memory once for automatic daily and idle rollovers while preserving typed lifecycle hooks and manual reset semantics.

## Reproduction Assessment

Yes, at source level with high confidence: current main can roll over stale daily/idle sessions and emit session_end/session_start, while bundled session-memory still accepts only command:new and command:reset. I did not run a live rollover because this was a read-only cleanup review.

## Solution Assessment

Unclear as a final API shape: the persistence gap is real, but maintainers still need to choose whether the durable contract is before_reset, a new session reset event, synthetic command actions, or lifecycle-hook-based persistence.
