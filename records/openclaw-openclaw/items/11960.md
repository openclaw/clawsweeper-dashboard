## Summary

Keep open. Current main and v2026.6.5 still leave automatic daily/idle rollovers on the typed session lifecycle path while bundled session-memory remains command-only; the secondary hook-config schema mismatch is fixed, but the requested reset persistence surface is not. The issue is already paired with open implementation PRs at https://github.com/openclaw/openclaw/pull/71792 and https://github.com/openclaw/openclaw/pull/61675, so maintainers should choose, land, replace, or reject that path before closing this issue.

## What This Changes

Request a `session:reset`/`session:before-reset` hook or equivalent reset-persistence surface so automatic daily and idle session resets can save memory before clearing context.

## Best Possible Solution

Land or replace the open fix PRs with a maintainer-approved reset-persistence contract that saves session-memory once on automatic daily/idle rollovers, carries the previous transcript/context, preserves typed lifecycle hooks, and adds daily plus idle regression coverage.

## Reproduction Assessment

Yes, from source inspection: stale daily/idle rollovers capture and archive the old session, then emit typed session lifecycle hooks, while session-memory only handles `command:new` and `command:reset`. I did not run a live reset scenario because this review is read-only, but the current-main path is clear.

## Solution Assessment

Unclear as a product/API decision: adding a named reset event is plausible, but current code already has `session_end` and `before_reset` surfaces that overlap. The best fix should choose one maintainable reset-persistence contract and avoid duplicate or double-fired hooks.
