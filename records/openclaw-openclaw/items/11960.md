## Summary

Keep open. Current main and the latest release still leave automatic daily/idle rollovers on the typed session lifecycle path while bundled session-memory remains command-only; the secondary hook-config schema mismatch is fixed, but the requested reset persistence surface is not. The issue is already paired with open implementation PRs, so maintainers should review, land, replace, or reject that path rather than close the report now.

## What This Changes

Request a reset lifecycle hook or equivalent persistence surface for automatic daily/idle rollovers, plus hook-specific config schema support for session-memory options.

## Best Possible Solution

Land or replace the open fix PRs with a maintainer-approved reset persistence surface that makes automatic daily/idle rollovers save session-memory once, carries the previous transcript/context, preserves typed lifecycle hooks, and adds daily plus idle regression coverage.

## Reproduction Assessment

Yes, from source inspection: stale daily/idle rollovers capture and archive the old session, then emit only typed session lifecycle hooks, while session-memory only handles `command:new` and `command:reset`. I did not run a live reset scenario because this review is read-only, but the current-main path is clear.

## Solution Assessment

Unclear as a product/API decision: adding a named reset event is plausible, but current code already has `session_end` and `before_reset` surfaces that overlap. The best fix should choose one maintainable reset-persistence contract and avoid duplicate or double-fired hooks.
