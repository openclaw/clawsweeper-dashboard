## Summary

Keep open. Current main and v2026.5.12 still leave automatic daily/idle rollovers on the session_start/session_end lifecycle path while bundled session-memory only handles command:new and command:reset; the open closing PR and overlapping PR make this maintainer contract-selection work, not cleanup closure.

## What This Changes

Request to add a session reset or before-reset hook surface so session-memory and custom hooks can persist previous session context on automatic daily/idle resets, plus allow hook-specific config keys.

## Best Possible Solution

Ship one canonical implementation that persists session-memory exactly once for automatic daily and idle rollovers, preserves transcript archival and manual reset behavior, documents the selected hook contract, and adds focused daily plus idle regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure an existing session to become stale by daily or idle policy and send the next inbound message; current main creates a new session and lifecycle hooks while bundled session-memory remains limited to command:new and command:reset.

## Solution Assessment

Unclear. The persistence gap is real, but maintainers need to choose between the overlapping synthetic command reset, before_reset, and new session-event shapes before landing a durable API.
