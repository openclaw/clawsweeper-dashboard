## Summary

Keep open: this is a plausible TTS feature, but the branch is now stale against the speech-core/plugin TTS architecture, has blocking correctness gaps, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a `persist=true` TTS directive that stores a selected voice in user prefs per provider and reuses it across later TTS responses, with Discord voice and tests updated on the old core TTS paths.

## Best Possible Solution

Implement persistent voice selection centrally in speech-core using the current provider override/result contract, cover file, stream, Discord voice, and telephony flows, and add focused tests plus real redacted runtime proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows current providers such as MiniMax parse voice overrides through `providerOverrides`, while the PR helper only persists OpenAI and ElevenLabs voices. The feature itself has no real runtime proof attached.

## Solution Assessment

No. The old-core implementation should be ported to the current speech-core/provider plugin architecture and should persist the resolved provider voice generically instead of hardcoding provider IDs.

## Review Findings
