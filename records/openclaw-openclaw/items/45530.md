## Summary

Keep open. The feature is still not implemented on current main, and the PR is not merge-ready because it patches the pre-reorg core TTS and Discord paths while current runtime behavior lives in the speech-core plugin, the plugin SDK speech parser, and the bundled Discord plugin voice path. The earlier review discussion was useful: the branch appears to have addressed the first-round telephony/policy/write-failure comments, but current-main inspection confirms the later architecture-drift blockers remain.

## What This Changes

The PR adds provider-keyed persisted TTS voice preferences, parses `persist=true` in model-emitted TTS directives, applies persisted voices during normal/telephony synthesis and Discord voice replies, and adds focused TTS tests.

## Best Possible Solution

If maintainers want this capability, rebuild it on the current speech-core/plugin architecture: add an explicit persistence signal to the provider-aware directive parser/result type, add provider-keyed voice prefs and facade/runtime APIs in speech-core, resolve persisted voices for normal and telephony synthesis, write preferences only after successful synthesis as best-effort, wire `extensions/discord/src/voice/tts.ts`, and update focused tests plus TTS docs.

## Reproduction Assessment

Not applicable as a user bug report. The PR defects are high-confidence by source inspection: current main has no `persist` signal, no voice-pref runtime API, and the active Discord voice path differs from the files changed by the branch.

## Solution Assessment

No. The proposed patch is not the best way to solve the feature now because it targets obsolete core and Discord paths; the maintainable path is an additive speech-core/plugin-SDK implementation after maintainer agreement on durable model-written preferences.
