## Summary

Keep open: the feature is useful, but the branch is stale against current main. It patches legacy/facade TTS and old Discord paths while the active speech-core provider override runtime lacks persisted-voice support, and contributor real behavior proof is still missing.

## What This Changes

The PR adds a `persist=true` TTS directive, provider-keyed voice preferences, OpenAI/ElevenLabs voice read/write wiring, Discord voice handling, and legacy `src/tts` tests.

## Best Possible Solution

Retarget persistence to the active speech-core runtime: parse a persist signal, store provider-keyed voices in TTS prefs, merge persisted voices into provider overrides for file, stream, telephony, and Discord voice paths, and prove the behavior in a real setup.

Not applicable. This is a feature PR; source inspection shows current main does not implement persisted TTS voice directives, but there is no existing bug contract to reproduce.

## Solution Assessment

No. The requested behavior should be implemented in `src/tts/directives.ts`, `extensions/speech-core/src/tts.ts`, and the active provider override contract instead of the legacy `src/tts/tts.ts` and old Discord manager paths.
