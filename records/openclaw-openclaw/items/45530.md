## Summary

Keep open, but do not merge this PR as-is. Current main does not implement durable `persist=true` TTS voice preferences, and the branch is stale against the active speech-core/plugin architecture: the parser, runtime facade, prefs storage, and Discord voice path it needs to change now live elsewhere.

## What This Changes

The branch adds a `persist` flag to TTS directive parsing, provider-keyed voice preference helpers, persisted voice reads for OpenAI/ElevenLabs synthesis, Discord voice-session persistence writes, and focused tests.

## Best Possible Solution

If maintainers want this capability, rebuild it on the active speech-core/plugin-SDK design: add an explicit persistence signal to the provider-aware directive result, store provider-keyed voice prefs in speech-core, expose narrow runtime facade helpers if needed, resolve persisted voices for normal and telephony synthesis, write prefs only after successful synthesis as best-effort, wire `extensions/discord/src/voice/tts.ts`, and update focused TTS docs and tests.

Not applicable as a user bug report. The PR defects have a high-confidence source-inspection path: current main routes parsing through `src/tts/directives.ts` and Discord voice synthesis through `extensions/discord/src/voice/tts.ts`, while the branch changes older core and manager paths.

## Solution Assessment

No. The proposed patch is not the best way to solve the feature on current main because it targets obsolete code paths and does not extend the active speech-core runtime contract.
