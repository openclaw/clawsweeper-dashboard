## Summary

Keep open: current main still contains and tests the doctor rewrite from `openai-codex/*` to `openai/*`, while provider docs and validation still define `openai-codex/gpt-5.x` as a valid PI OAuth route, so the reported auth-losing upgrade path remains plausible and source-reproducible.

## What This Changes

The issue asks doctor repair to preserve valid `openai-codex/*` model references instead of rewriting them to unauthenticated `openai/*` refs during the 2026.5.4 to 2026.5.5 update path.

## Best Possible Solution

Narrow doctor repair so it does not move a usable PI Codex OAuth route to `openai/*` plus PI; canonicalize to `openai/*` only when selecting the native Codex runtime, and align the conflicting doctor/provider docs and tests.

## Reproduction Assessment

Yes, source-level. Current tests directly assert that `openai-codex/gpt-5.5` is rewritten to `openai/gpt-5.5` with PI when Codex is not ready, and the issue logs show that route fails for a user with only `openai-codex` OAuth and no `OPENAI_API_KEY`.

## Solution Assessment

No, the current repair is not the best fix because it can convert a valid PI OAuth route into a direct OpenAI API-key route. A safer fix is to preserve `openai-codex/*` when PI remains the selected runtime and only rewrite when the native Codex harness will own the turn.
