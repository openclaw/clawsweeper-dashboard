## Summary

Keep open: current main still contains the doctor config and session repair paths that rewrite documented `openai-codex/*` PI Codex OAuth routes to `openai/*`, and the discussion includes multiple independent 2026.5.5 live confirmations of auth lockout or unexpected direct OpenAI billing. Related open PRs are useful but not merged, and the current fix PR only appears to cover part of the session/config surface.

## What This Changes

The issue reports that `openclaw doctor --fix` rewrites subscription-backed `openai-codex/*` model references to `openai/*` during upgrade, leaving PI-runtime users without the intended Codex OAuth route.

## Best Possible Solution

Make doctor repair auth- and runtime-aware across both config and session stores: preserve intentional `openai-codex/*` PI Codex OAuth routes, or migrate only when the target `openai/*` route is guaranteed to run through Codex OAuth rather than PI/direct API billing.

## Reproduction Assessment

Yes. Source inspection shows a current-main path from `openai-codex/*` config/session refs through doctor repair to `openai/*` plus PI, and the issue discussion includes multiple independent 2026.5.5 live confirmations of the resulting auth failure or billing route change.

## Solution Assessment

No. Current main’s repair policy is not the best solution because it can turn a documented PI Codex OAuth route into a direct OpenAI PI route; the safer fix is to preserve or migrate only with matching runtime/auth proof and session-store coverage.
