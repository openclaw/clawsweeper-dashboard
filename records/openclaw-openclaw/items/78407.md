## Summary

Keep open: current main still has a doctor repair path that rewrites `openai-codex/*` refs to `openai/*`, while other current docs and runtime guidance still describe `openai-codex/*` as the supported PI route for Codex OAuth users. This needs maintainer resolution rather than cleanup closure.

## What This Changes

The issue reports that `openclaw doctor --fix` rewrites `openai-codex/*` model IDs to `openai/*` during a 2026.5.4 to 2026.5.5 update, breaking ChatGPT OAuth-only configurations.

## Best Possible Solution

Align the route policy across doctor, runtime auth, tests, and docs so `doctor --fix` never migrates a Codex OAuth-only PI setup into an `openai/*` PI route that requires an OpenAI API key; either preserve `openai-codex/*` for PI OAuth or migrate only to a guaranteed Codex runtime route.

## Reproduction Assessment

Yes from source inspection, but not from a live macOS upgrade run. A config with `openai-codex/gpt-5.5` and only `openai-codex` OAuth will pass through `maybeRepairCodexRoutes`, can be rewritten to `openai/gpt-5.5` with `agentRuntime.id: "pi"`, and the direct OpenAI provider then reports missing API-key auth.

## Solution Assessment

Unclear until maintainers resolve the route-policy conflict. Preserving `openai-codex/*` matches several current user-facing docs and error hints, while current doctor tests and doctor docs assert the rewrite behavior.
