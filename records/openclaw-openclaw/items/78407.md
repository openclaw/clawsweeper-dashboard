## Summary

Keep open: current main still contains the doctor repair path that rewrites `openai-codex/*` into `openai/*` and can select PI, while provider docs and multiple live reports confirm `openai-codex/*` remains the valid Codex OAuth through-PI route.

## What This Changes

The issue asks OpenClaw to stop `doctor --fix` from rewriting ChatGPT/Codex OAuth `openai-codex/*` config and session routes into `openai/*` PI routes during update repair.

## Best Possible Solution

Make doctor fail closed for Codex subscription routes: preserve `openai-codex/*` config and session state whenever repair would produce `openai/* + pi` or an auth/billing change that is not proven safe, then align doctor docs, provider docs, changelog, and regression tests.

## Reproduction Assessment

Yes, from source inspection and multiple live reports, though I did not run a live update. Current main runs `maybeRepairCodexRoutes`, rewrites `openai-codex/gpt-5.5` to `openai/gpt-5.5`, and can select PI when the Codex plugin readiness check fails.

## Solution Assessment

Unclear until maintainers choose the route policy. The narrowest safe direction appears to preserve existing `openai-codex/*` PI OAuth routes unless doctor can prove a native Codex OAuth route will be used without changing billing/auth semantics.
