## Summary

Keep open: current main still contains the doctor repair path that rewrites `openai-codex/*` routes to `openai/*` and can select PI, while current provider docs and live reports show `openai-codex/*` remains the working PI OAuth route for Codex subscription users.

## What This Changes

The issue reports that `openclaw doctor --fix` rewrites Codex OAuth model references to direct OpenAI references during update, causing auth lockouts or unintended API-key billing.

## Best Possible Solution

Align doctor, runtime auth, tests, docs, and changelog so `doctor --fix` never migrates a Codex OAuth PI setup into an `openai/*` PI route without a valid direct OpenAI auth/billing path.

## Reproduction Assessment

Yes, from source inspection and multiple live reports. A config containing `openai-codex/gpt-5.x` with only `openai-codex` OAuth can pass through `maybeRepairCodexRoutes`, be rewritten to `openai/gpt-5.x` with PI, and then hit the direct OpenAI missing-key path.

## Solution Assessment

No for current main: the existing rewrite is not the safest fix because it can orphan auth or silently change billing. The maintainable direction is to preserve `openai-codex/*` when PI remains the runtime, or migrate only when the target route has proven auth/runtime support.
