## Summary

Keep open: current main still contains the doctor repair path that rewrites `openai-codex/*` to `openai/*` and can force `agentRuntime.id: "pi"`, which matches the reported lockout when the user only has Codex OAuth and no OpenAI API key.

## What This Changes

The issue asks OpenClaw to stop `openclaw doctor --fix` from converting Codex OAuth `openai-codex/*` model refs into direct `openai/*` refs that require `OPENAI_API_KEY`.

## Best Possible Solution

Align doctor, docs, runtime auth, and tests so `doctor --fix` never converts a Codex-OAuth-only PI setup into `openai/*` running through PI without OpenAI API-key auth.

## Reproduction Assessment

Yes from source inspection and live reporter comments, but not from a local macOS upgrade run. Current main calls `maybeRepairCodexRoutes`, rewrites `openai-codex/gpt-*` to `openai/gpt-*`, can set `agentRuntime.id` to `pi`, and the OpenAI provider then requires API-key auth.

## Solution Assessment

Unclear until maintainers resolve the route-policy conflict. The maintainable outcome is to prevent the bad `openai/*` plus PI no-auth state, then update the doctor/provider docs and regression tests to match the chosen policy.
