## Summary

Current main resolves the reported supply-chain path: ordinary bare plugin specs no longer go to ClawHub first, while explicit `clawhub:` specs still use ClawHub.

## What This Changes

The issue asks OpenClaw to prevent trusted bare plugin names from being replaced by name-squatted ClawHub packages.

## Best Possible Solution

Keep the npm-default bare-spec resolver, reserve ClawHub installation for explicit or trusted catalog flows, and handle signatures or trust UI as separate defense-in-depth work if maintainers want it.

## Reproduction Assessment

No against current main. Source inspection and existing regression coverage show a bare spec such as `demo` reaches npm without calling the ClawHub installer.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix for this issue: ordinary package specs stay npm-default, and ClawHub requires an explicit `clawhub:` locator or trusted catalog path.
