## Summary

Close as implemented on current main: bare plugin package specs now route to npm by default, while ClawHub installation requires an explicit `clawhub:` locator or trusted catalog flow, so the reported ClawHub-first name-squatting path is no longer present.

## What This Changes

The issue asks OpenClaw to avoid letting trusted-looking bare plugin names be silently resolved from ClawHub before the intended trusted package source.

## Best Possible Solution

Keep the npm-default bare-spec resolver, reserve ClawHub for explicit or trusted catalog flows, and ship the current-main fix in the next release while treating any broader signatures/trust UI as separate defense-in-depth work.

## Reproduction Assessment

No against current main. The source path and regression test show `openclaw plugins install demo` goes to npm without calling the ClawHub installer, so the reported current-main failure path no longer reproduces.

## Solution Assessment

Yes. The implemented policy change is the narrow maintainable fix for the reported substitution path: ordinary package specs stay npm-default, explicit `clawhub:` still works, and official catalog trust remains separate from community name lookup.
