## Summary

Keep open: current main still implements the reported bare-spec ClawHub-first plugin install behavior. Existing controls reduce exposure, but a community/non-official ClawHub package can still satisfy a trusted-looking bare name before npm is tried, so this remains a security/product design issue rather than cleanup-close material.

## What This Changes

The issue asks OpenClaw to prevent bare plugin installs from silently resolving trusted-looking npm package names to potentially name-squatted ClawHub packages.

## Best Possible Solution

Keep this open for a security/plugin-install design fix. The maintainable end state should prevent community ClawHub packages from silently winning over trusted npm names, for example by requiring explicit clawhub: for non-official packages, letting npm win unless ClawHub provenance is official/trusted, adding durable trust/signature policy, or making install prompts fail closed until the operator explicitly accepts the source.

## Reproduction Assessment

Yes. The source-backed reproduction is `openclaw plugins install demo`: current main converts it to `clawhub:demo`, installs a found community ClawHub package first, and the contract test asserts npm is not called.

## Solution Assessment

No current fix is in place. Existing warnings and verification are useful defense in depth, but they do not resolve the initial name-squatting/substitution path; the safer solution needs an explicit trust/provenance policy for bare specs.
