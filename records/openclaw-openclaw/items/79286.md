## Summary

Keep open. Current main still builds the Gateway HTTP response bodies from `agentCommandFromIngress` results/events with `deliver: false`, bypassing the hook-aware outbound delivery path that applies `message_sending` and emits `message_sent`; the related umbrella remains open, but this endpoint-specific surface has distinct files and streaming semantics.

## What This Changes

The issue requests documented plugin lifecycle hooks to fire for assistant responses returned by the Gateway HTTP `/v1/chat/completions` and `/v1/responses` endpoints.

## Best Possible Solution

Add an HTTP response finalization seam that applies documented outbound hooks exactly once for both Gateway OpenAI-compatible endpoints, with explicit streaming behavior and regression tests.

## Reproduction Assessment

Yes for the `message_sending`/`message_sent` gap: register a rewrite hook and call either HTTP endpoint; current source shows `deliver: false` and direct response assembly outside the hook-aware delivery path. The `agent_end` claim is less certain because the agent runner already has an `agent_end` hook path for normal turns.

## Solution Assessment

Unclear. A per-endpoint fix is plausible for non-streaming responses, but streaming rewrite/cancel and exactly-once emission semantics need maintainer direction before treating this as a fully mechanical repair.
