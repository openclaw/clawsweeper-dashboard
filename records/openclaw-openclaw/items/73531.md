## Summary

Keep open. Current main still has the duplicate `openclaw status` task-reconciliation path, and open PR #73650 is already the linked implementation candidate with closing intent, so this issue should remain open for review, revision, landing, or closure of that PR.

## What This Changes

The issue requests reducing `openclaw status` latency by avoiding repeated `reconcileInspectableTasks()` work between the task registry and task audit summaries.

## Best Possible Solution

Review and land PR #73650 if its per-status-call shared reconciled snapshot is correct, or merge an equivalent narrow fix that reconciles once inside `getStatusSummary()` while preserving the existing zero-argument helper contract for other callers.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main calls both task summary helpers during one status summary build, and each helper independently calls `reconcileInspectableTasks()` before reaching the clone-and-sort task list path.

## Solution Assessment

No for the issue's suggested module-level process cache as the first fix. A per-status-call shared snapshot is narrower, avoids dirty-flag and shared-reference hazards, and matches the linked PR direction.
