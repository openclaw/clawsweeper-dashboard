## Summary

Keep open. Current main still builds the status task registry summary and task audit summary through two independent helper calls, and the issue is paired with open, unmerged PR #73650 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests removing redundant `reconcileInspectableTasks()` work during `openclaw status`, preferably with an in-process cache or an equivalent shared reconciled snapshot.

## Best Possible Solution

Review and land PR #73650 if its per-status-call shared snapshot is correct, or merge an equivalent narrow fix that reconciles tasks once inside `getStatusSummary()` while preserving the zero-argument helper behavior for other call sites.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main calls both status task helpers in one summary build, and each helper independently calls `reconcileInspectableTasks()` before reaching the clone-and-sort task list path.

## Solution Assessment

No for the issue’s module-level process cache as the first fix. A per-status-call shared snapshot is narrower, avoids invalidation and shared-reference hazards, and matches the existing open PR direction.
