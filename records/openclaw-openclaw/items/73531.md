## Summary

Keep open: current main and v2026.6.6 still run the status task registry summary and task audit through separate reconciliation paths, and an open closing PR now owns the implementation path.

## What This Changes

The issue requests removing duplicate `reconcileInspectableTasks()` work from `openclaw status` so registry and audit summaries share one reconciled task snapshot.

## Best Possible Solution

Land a narrow current-main fix that computes one reconciled task snapshot for the status path, or proves an equally safe cache invalidation strategy, then let the linked PR close this issue.

## Reproduction Assessment

Yes by source inspection: `getStatusSummary()` calls the registry summary and audit findings helpers separately, and both helpers call `reconcileInspectableTasks()` before reaching `listTaskRecords()`. I did not run a live large-ledger timing test in this read-only review.

## Solution Assessment

Yes, removing the duplicate status-path reconciliation is the narrow maintainable fix. The shared-snapshot shape avoids broader cache invalidation risk, while the current open cache PR needs review before it can be treated as the final solution.
