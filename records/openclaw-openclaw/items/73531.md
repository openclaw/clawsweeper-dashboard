## Summary

Keep open. Current main still has the duplicate status-task reconciliation path, and open PR #73650 directly references this issue with GitHub closing syntax, so the issue should remain open until that PR is reviewed, merged, or closed.

## What This Changes

The issue requests reducing `openclaw status` latency by reusing one reconciled task snapshot across the registry and audit summaries instead of reconciling the same task list twice.

## Best Possible Solution

Review and land PR #73650 if its per-status-call shared snapshot contract is correct, or merge an equivalent narrow fix that reconciles once inside `getStatusSummary()` while preserving the zero-argument helper behavior used by other call sites.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main calls both status task helpers in one summary build, and each helper independently calls `reconcileInspectableTasks()` before reaching the clone-and-sort task list path.

## Solution Assessment

No for the issue's suggested module-level process cache as the first fix. A per-call shared reconciled snapshot is narrower, avoids invalidation and shared-reference hazards, and matches the existing open PR direction.
