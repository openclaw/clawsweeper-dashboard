## Summary

Keep open. Current main still builds the status task registry summary and audit summary through two separate helper calls, and each helper still performs its own task reconciliation; linked PR #73650 remains the active implementation path and should be reviewed, landed, or closed before this issue is resolved.

## What This Changes

Request to remove redundant `openclaw status` task reconciliation by sharing or caching the reconciled task snapshot between registry and audit summaries.

## Best Possible Solution

Review and land PR #73650 if its per-status-call shared snapshot is correct, or merge an equivalent narrow change that reconciles inspectable tasks once inside `getStatusSummary()` and derives both summaries from that snapshot while preserving the zero-argument helper contract for other call sites.

## Reproduction Assessment

Yes. Current main provides a high-confidence source-level reproduction: one status summary build calls both task summary helpers, and both helpers independently call `reconcileInspectableTasks()` before reaching the clone-and-sort task list path.

## Solution Assessment

No for the module-level process-lifetime cache as the first fix. A per-status-call shared reconciled snapshot is narrower, avoids invalidation and shared-reference hazards, and matches the linked implementation direction.
