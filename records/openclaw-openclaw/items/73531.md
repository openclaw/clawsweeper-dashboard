## Summary

Keep open. Current main still has the duplicate status task-reconciliation path, and open PR #73650 already targets this exact bug with this issue as its closing target.

## What This Changes

The issue asks to avoid duplicate `reconcileInspectableTasks()` work during `openclaw status`, especially on large task registries.

## Best Possible Solution

Review and land PR #73650, or an equivalent narrow fix that shares one reconciled task snapshot inside the status summary path while preserving existing helper behavior for other callers.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: one `getStatusSummary()` call still reaches both task summary helpers, and each helper independently calls `reconcileInspectableTasks()` and the full `listTaskRecords()` clone/sort path.

## Solution Assessment

Yes for the linked per-status-call shared-snapshot direction. It is narrower than a module-level process cache because it avoids registry invalidation and shared-reference hazards outside the status command.
