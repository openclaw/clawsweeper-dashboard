## Summary

Keep open: current `main` and the latest release still run the two status task summaries through separate `reconcileInspectableTasks()` calls, and the direct implementation PR was closed unmerged rather than landed or superseded.

## What This Changes

The issue asks for `openclaw status` to avoid running `reconcileInspectableTasks()` twice by sharing or caching a reconciled task snapshot.

## Best Possible Solution

Implement a per-status shared reconciled task snapshot with focused regression coverage, preserving zero-argument helper behavior for other callers and avoiding a process-wide cache unless a separate invalidation contract is designed.

## Reproduction Assessment

Yes, from source inspection: current `main` calls both task summary helpers from `getStatusSummary()`, and each helper calls `reconcileInspectableTasks()` before reaching the cloned-and-sorted `listTaskRecords()` path.

## Solution Assessment

Yes for a per-status shared snapshot, not for a module-level dirty cache as the first repair. That narrower shape removes the duplicate work without adding shared invalidation semantics.
