## Summary

Keep open: current main and the latest shipped release still compute the status task registry summary and task audit findings through separate `reconcileInspectableTasks()` paths, and the focused fix attempts are closed unmerged rather than canonical replacements.

## What This Changes

The issue requests removing duplicate task reconciliation work from `openclaw status` by sharing or caching the reconciled task snapshot used by task registry and audit summaries.

## Best Possible Solution

Implement the shared-snapshot shape on current main: compute the reconciled task list once per `getStatusSummary()` call and pass it to registry/audit helpers while preserving existing zero-argument callers.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current `getStatusSummary()` still calls two helpers that each reach `reconcileInspectableTasks()` and then `listTaskRecords()`. I did not run a live large-ledger timing repro because this review is read-only.

## Solution Assessment

Yes, the shared-snapshot status-path fix is the best narrow solution. It removes the duplicate work without introducing a broader time-based cache or invalidation policy that could make task-state freshness ambiguous.
