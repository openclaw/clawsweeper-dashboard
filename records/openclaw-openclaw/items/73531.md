## Summary

Keep open: current main and the latest release still run independent task reconciliation for the status task summary and audit path, and the two matching fix PRs are closed unmerged, so the reported latency bug still needs a narrow fix.

## What This Changes

The issue requests reducing `openclaw status` task reconciliation from two independent clone/sort passes to one shared in-process snapshot per status invocation.

## Best Possible Solution

Compute one reconciled task snapshot in `getStatusSummary()` after configuring maintenance, derive both registry summary and audit findings from it, and keep single-purpose helpers cheap for other callers.

## Reproduction Assessment

Yes for the source-level path: current main calls task registry summary and audit findings from `getStatusSummary()`, and each path reaches `reconcileInspectableTasks()` and `listTaskRecords()`. I did not run a live 10K-task timing benchmark in this read-only review.

## Solution Assessment

The reported optimization target is valid, but the best fix is narrower than a process-lifetime cache: share one per-status-call reconciled snapshot and add regression coverage for the single-reconcile contract.
