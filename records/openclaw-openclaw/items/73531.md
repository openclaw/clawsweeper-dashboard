## Summary

Keep open. Current main still builds the status task registry and audit summaries through two independent helper calls, and each helper still reconciles the task list internally, so the reported duplicate reconciliation remains present. The issue is also paired with open unmerged PR #73650, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests removing duplicate task reconciliation during `openclaw status`, preferably by reusing a single in-process reconciled task snapshot for the registry and audit summaries.

## Best Possible Solution

Land PR #73650 after review if its per-status-call shared snapshot is correct, or merge an equivalent narrow fix that reconciles tasks once inside `getStatusSummary()` and derives both status task summaries from that snapshot while preserving existing helper behavior for other consumers.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main calls both task summary helpers in one `getStatusSummary()` invocation, and each helper independently calls `reconcileInspectableTasks()` before reaching the clone-and-sort task list path.

## Solution Assessment

No for the issue's suggested module-level process cache as the first fix. A per-status-call shared snapshot is narrower, avoids cache invalidation and shared-reference hazards, and matches the open PR direction.
