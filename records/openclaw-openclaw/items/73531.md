## Summary

Keep open: current main still has the duplicate status reconciliation path, and the only direct fix PRs found are closed unmerged rather than landed or superseding this issue.

## What This Changes

The issue asks to remove redundant task reconciliation from `openclaw status`, preferably by reusing a reconciled task snapshot or cache within a single status invocation.

## Best Possible Solution

Implement a per-status-call shared reconciled task snapshot with regression coverage, preserving zero-argument helper behavior for other callers and avoiding a process-wide cache until a broader contract is needed.

## Reproduction Assessment

Yes. Source inspection on current main shows `getStatusSummary()` calls both task summary helpers, each helper calls `reconcileInspectableTasks()`, and reconciliation reaches the clone-and-sort `listTaskRecords()` path; I did not run a live large-ledger timing benchmark.

## Solution Assessment

Yes for a shared per-status snapshot, not for a broad module-level cache as the first step. The closed implementation in https://github.com/openclaw/openclaw/pull/73650 is the narrower maintainable shape because it removes the duplicate work without shared-cache invalidation hazards.
