## Summary

Keep open. Current main still performs two independent status task-summary calls that each reconcile the inspectable task list, and open PR #73650 is already linked with closing syntax for the same bug.

## What This Changes

The issue requests avoiding duplicate task reconciliation during `openclaw status` by sharing or caching the reconciled task snapshot.

## Best Possible Solution

Review and land PR #73650, or an equivalent narrow fix that reconciles once inside the status summary path while preserving zero-argument helper behavior for other callers.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: one `getStatusSummary()` invocation calls both task summary helpers, and each helper independently reaches `reconcileInspectableTasks()` and the clone-and-sort `listTaskRecords()` path.

## Solution Assessment

Yes for the linked per-status-call shared snapshot direction. It is narrower than a module-level process cache because it avoids invalidation complexity and shared-reference hazards outside the status command.
