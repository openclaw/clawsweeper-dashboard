## Summary

Keep open: current main and the latest shipped release still have the duplicate task-reconciliation status path, and the direct implementation PR is closed unmerged rather than landed or superseding this issue.

## What This Changes

The issue requests eliminating redundant `reconcileInspectableTasks()` work during `openclaw status` by sharing or caching the reconciled task list within one status invocation.

## Best Possible Solution

Implement a per-status shared reconciled task snapshot with focused regression coverage, preserving zero-argument helper behavior for other callers and avoiding a process-wide cache unless a separate invalidation contract is designed.

## Reproduction Assessment

Yes, from source inspection. Current main still has `getStatusSummary()` call both task helpers, each helper still calls `reconcileInspectableTasks()`, and reconciliation still reaches the cloned-and-sorted `listTaskRecords()` path.

## Solution Assessment

Yes for a per-status shared snapshot, not for a module-level dirty cache as the first repair. The closed implementation in https://github.com/openclaw/openclaw/pull/73650 demonstrates the narrower maintainable shape without adding shared invalidation semantics.
