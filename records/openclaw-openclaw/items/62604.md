## Summary

Keep open. Current main still only intercepts clipboard items whose MIME starts with `image/`, so `text/plain` or `text/html` clipboard payloads containing `data:image/...;base64,...` fall through to the composer; the related wrong-image report is useful context but does not replace this distinct text-injection bug.

## What This Changes

The issue asks OpenClaw to harden Control UI image paste so data-image clipboard text is handled as an attachment or rejected cleanly, while keeping send failure restoration centralized.

## Best Possible Solution

Add focused Control UI paste hardening for data-image text/html/plain payloads, preserve file-backed paste and existing send restoration behavior, and leave installed-package hotpatch/source-map policy as a separate support or docs decision if maintainers want it.

## Reproduction Assessment

Yes, from source inspection. A focused DOM test can dispatch a paste event against the chat textarea with only `text/plain` or `text/html` `data:image/...;base64,...` clipboard data and assert that current main neither prevents default nor creates an attachment.

## Solution Assessment

Yes for the central bug. The narrow maintainable solution is in the Control UI paste path with regression tests; draft restoration should remain in the existing send pipeline, and package hotpatch/source-map policy should not be bundled into the runtime bug fix.
