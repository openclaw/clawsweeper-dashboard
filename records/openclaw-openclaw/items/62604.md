## Label Justifications

- `P2`: This is a normal-priority Control UI paste bug with a clear, limited implementation surface and user-visible composer impact.
- `impact:session-state`: The issue affects composer draft/message state by allowing a large pasted image payload to become raw text instead of attachment state.

## Summary

Keep open. Current main and the latest release still let `handlePaste` fall through when the clipboard only contains `text/plain` or `text/html` data-image content, so the browser can insert the raw base64 string into the composer instead of the attachment path.

## What This Changes

Harden Control UI image paste so data-image clipboard text is not inserted as raw base64, while keeping send/draft restoration centralized in the send pipeline.

## Best Possible Solution

Add focused `handlePaste` hardening for data-image `text/plain` and `text/html` clipboard payloads, preserve the existing file-backed attachment and send-restoration paths, and handle dist hotpatch/source-map policy separately.

## Reproduction Assessment

Yes, at source level. A focused DOM test can dispatch a paste event with only `text/plain` or `text/html` containing `data:image/...;base64,...` against the composer textarea and assert current main does not prevent default insertion or convert it to an attachment.

## Solution Assessment

Yes for the central bug. The maintainable fix belongs in `handlePaste` with regression coverage, while draft restoration should remain in the existing send pipeline and packaging/source-map policy should stay separate.
