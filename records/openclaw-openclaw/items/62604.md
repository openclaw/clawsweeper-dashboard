## Summary

Keep open. Current main still lets text/plain or text/html `data:image/...;base64,...` clipboard payloads fall through to the composer because the paste handler only intercepts clipboard items whose MIME starts with `image/`; https://github.com/openclaw/openclaw/issues/74960 is useful related media-pipeline context, but it does not replace this distinct data-URL text-injection case.

## What This Changes

Harden the Control UI composer so pasted data-image clipboard text is converted to an attachment or safely rejected, while keeping draft restoration in the send pipeline and treating dist hotpatch/source-map policy separately.

## Best Possible Solution

Add focused Control UI paste hardening for data-image text/html/plain payloads, preserve file-backed paste and existing send restoration behavior, and leave package hotpatch/source-map policy out of the bug fix unless docs need a separate clarification.

## Reproduction Assessment

Yes, from source inspection. A focused DOM test can dispatch a paste event against the chat textarea with only text/plain or text/html `data:image/...;base64,...` clipboard data and assert that current main neither prevents default nor creates an attachment.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is in the Control UI paste path with regression tests; draft restoration should remain in the existing send pipeline and installed-package hotpatching should be treated as a separate support/docs concern.
