## Summary

Keep open. Current main still lets clipboard payloads that are only `text/plain` or `text/html` data-image strings fall through to the textarea default paste path, so the central composer base64-injection bug is not fixed or superseded.

## What This Changes

The issue asks OpenClaw to harden Control UI image paste handling so data-image clipboard payloads become safe attachments or are rejected without dumping raw base64 into the composer.

## Best Possible Solution

Add focused Control UI paste hardening for data-image text/html/plain clipboard payloads, preserve existing file-backed image paste and send restoration behavior, and leave source-map or hotpatch policy to a separate docs/support decision.

## Reproduction Assessment

Yes, from source inspection. A focused DOM test can dispatch a paste event with only `text/plain` or `text/html` containing `data:image/...;base64,...` against the chat textarea and assert current main does not prevent the default text insertion.

## Solution Assessment

Yes for the central bug. The maintainable fix belongs in the Control UI paste path with regression coverage; draft restoration should remain in the existing send pipeline.
