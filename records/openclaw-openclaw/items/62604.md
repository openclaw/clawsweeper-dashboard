## Summary

Keep open. Current main still lets `data:image/...;base64,...` clipboard text fall through to the textarea because the paste handler only intercepts clipboard items whose MIME starts with `image/`; the related #74960 report is useful clustering context but does not fully replace this distinct text-data-URL injection case.

## What This Changes

Harden Control UI image paste so data-image clipboard text is converted to an attachment or rejected instead of entering the chat draft, while keeping send/draft restoration centralized.

## Best Possible Solution

Add focused Control UI paste hardening for data-image text/html/plain payloads, preserve file-backed paste and existing send restoration behavior, and leave package hotpatch/source-map policy out of the bug fix.

## Reproduction Assessment

Yes. A focused DOM test can dispatch a paste event against the chat textarea with only text/plain or text/html `data:image/...;base64,...` clipboard data and assert current main neither prevents default nor creates an attachment.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Control UI paste path with regression tests; draft restoration should remain in the existing send pipeline and package dist debugging should be handled separately.
