## Summary

Keep open: current main still only supports global session cleanup limits, while per-label retention would add new cleanup semantics and public config/CLI/Gateway API surface that needs maintainer design approval.

## What This Changes

The issue requests per-label session cleanup retention so labelled cron/job sessions can keep the latest N sessions per group instead of only using global age/count limits.

## Best Possible Solution

Define an additive `session.maintenance` retention policy first, then implement it in the shared session maintenance path with CLI, Gateway protocol, docs, and regression tests aligned.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics rather than a report of broken documented behavior. Source inspection confirms current main has cron labels but no cleanup grouping by label.

## Solution Assessment

Unclear; per-label retention is plausible, but the exact CLI/config/protocol shape and ordering with existing cleanup policies are maintainer product/API decisions before a narrow implementation should start.
