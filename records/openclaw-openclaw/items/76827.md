## Summary

Keep open: current main still only supports global session cleanup limits, while this request adds new per-label retention semantics that need maintainer API/product review before implementation.

## What This Changes

Add per-label retention to `openclaw sessions cleanup` so users can keep the last N sessions per cron/job label instead of only applying a global count cap.

## Best Possible Solution

Define an additive `session.maintenance` retention shape first, then implement grouped capping in the shared session maintenance path with CLI, Gateway, docs, schema, and tests kept in sync.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics, not a report of broken documented behavior. Source inspection confirms current main lacks per-label grouping while cron sessions already have usable labels.

## Solution Assessment

Unclear; per-label retention is plausible, but the exact CLI/config contract and how it composes with existing cleanup policies is a maintainer product/API decision before implementation.
