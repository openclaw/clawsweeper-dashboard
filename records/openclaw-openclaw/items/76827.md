## Summary

Keep open: current main and the latest release still only support global session cleanup age/count limits plus cron-specific retention, and the broader per-pattern issue does not fully cover this request's keep-last-N-per-label behavior. This is a real session-state ergonomics/config design item, but it needs maintainer product direction before implementation.

## What This Changes

Add per-label retention to `openclaw sessions cleanup` so it keeps the last N sessions per job label instead of relying only on global age/count thresholds.

## Best Possible Solution

Approve a single core session-maintenance design that covers typed per-label or per-pattern retention/count rules, safe dry-run visibility, active/durable session protections, docs, and regression tests.

## Reproduction Assessment

Yes for source inspection: current main exposes global `session.maintenance.pruneAfter` and `maxEntries`, and `capEntryCount` keeps one global most-recent list rather than keeping N per label. I did not run a live cleanup because this is a read-only feature review.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested behavior is plausible, but a raw label-matching flag/config should be weighed against a typed `session.maintenance.rules` design and the cron-specific retention path before implementation.
