## Summary

Keep open: current main and v2026.6.6 still expose global session-maintenance age/count/disk settings, not per-label keep-N retention. The related retention issues overlap, but none fully owns count-based grouped retention by label, so this remains a maintainer product-design item rather than cleanup-closeable.

## What This Changes

The issue requests per-label keep-N retention for `openclaw sessions cleanup` so cron-style session groups retain recent history independently instead of competing under one global cap.

## Best Possible Solution

Define one additive session retention policy that can express label or pattern keep-N, then wire schema, CLI/Gateway cleanup, docs, and pruning tests through that single contract.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence that current main cannot express per-label keep-N cleanup.

## Solution Assessment

Unclear until maintainers choose the public retention contract. Per-label keep-N is plausible, but it should be folded into one retention-rule design rather than added as a parallel destructive cleanup knob.
