## Summary

Keep open: current main still only supports global session cleanup limits, and per-label retention is a new cleanup/config/API behavior that needs maintainer design before implementation.

## What This Changes

Request add per-label session retention so cleanup can keep the newest N sessions per label or cron job instead of only applying global age/count limits.

## Best Possible Solution

Decide the additive `session.maintenance` CLI/config/Gateway contract first, then implement label grouping in the shared cleanup path with docs and focused regression tests.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics, not a report of broken documented behavior. Source inspection gives high confidence that current main lacks per-label cleanup while cron-created sessions already carry labels.

## Solution Assessment

Unclear; per-label retention is plausible, but the proposed API is not settled and the config naming conflicts with the current `session.maintenance` contract. The maintainable path is an additive design across CLI, config, Gateway, docs, and tests.
