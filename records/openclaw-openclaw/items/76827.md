## Summary

Keep open: current main still only supports global session age/count/disk-budget cleanup, while per-label retention would add new cleanup semantics plus CLI/config/Gateway API surface that needs maintainer design review.

## What This Changes

Add per-label retention to `openclaw sessions cleanup` so cleanup can keep the last N sessions per job label instead of only applying one global store cap.

## Best Possible Solution

Decide the additive `session.maintenance`/CLI/Gateway contract first, then implement grouped count retention in the shared session maintenance path with dry-run output, docs, changelog, and focused regression coverage.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics, not a broken existing behavior report. Source inspection does confirm current main lacks per-label cleanup grouping while cron sessions already have labels that could support it.

## Solution Assessment

Unclear; per-label retention is plausible, but the exact CLI/config/protocol design is a maintainer product/API decision before a narrow implementation should start.
