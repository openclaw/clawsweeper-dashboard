## Summary

Keep open: current main still only supports global age/count/disk session cleanup, while this request asks for new per-label retention semantics. That is a plausible feature, but it adds CLI/config/protocol behavior and needs maintainer API/product design before implementation.

## What This Changes

Request per-label retention for `openclaw sessions cleanup`, such as keeping the last N sessions per cron/job label instead of applying only global cleanup limits.

## Best Possible Solution

Design an additive `session.maintenance` retention model first, then implement grouped retention in the shared session cleanup path with CLI, schema, Gateway, docs, dry-run output, and focused regression tests kept in sync.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics, not a report of broken documented behavior. Source inspection confirms current main lacks per-label grouping while cron session labels exist.
