## Summary

Keep open: current main still only supports global session cleanup limits plus cron age retention, while per-label count retention would add new core cleanup semantics and config/CLI surface that need maintainer API review.

## What This Changes

Add per-label retention to `openclaw sessions cleanup` so the last N sessions can be kept for each job label instead of only applying global caps.

## Best Possible Solution

Decide the additive retention API first, then implement label grouping in the shared cleanup preview/apply path with config schema, CLI/Gateway behavior, docs, and regression tests kept in sync.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics, not a report of broken documented behavior. Source inspection confirms current main lacks per-label fields or flags and applies the count cap globally.

## Solution Assessment

Unclear; per-label retention is a plausible direction, but the exact config/CLI shape and interaction with cron and disk-budget cleanup are maintainer product/API decisions before implementation.
