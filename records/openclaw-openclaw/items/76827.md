## Summary

Keep open: current main still has only global session age/count/disk cleanup, while this asks for new per-label retention semantics that need API and product design before implementation.

## What This Changes

Add per-label retention to `openclaw sessions cleanup` so cleanup can keep the most recent N sessions for each job label instead of only applying global thresholds.

## Best Possible Solution

Approve an additive retention policy under `session.maintenance`, then implement it in the shared session maintenance path with CLI, Gateway schema, docs, and regression coverage kept aligned.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics, not a broken documented behavior. Source inspection confirms current main has cleanup and cron labels but no per-label retention path.

## Solution Assessment

Unclear; per-label retention is plausible, but the API shape and interaction with existing retention policies need maintainer design before a narrow code change is the right solution.
