## Summary

Keep open: current main still only has global age/count/disk cleanup, and the requested group-by-label retention is a new session-maintenance/config feature that needs maintainer design.

## What This Changes

Add per-label session cleanup retention so `openclaw sessions cleanup` can keep the last N sessions per cron/job label instead of only applying global limits.

## Best Possible Solution

Define an additive `session.maintenance` grouped-retention contract first, then implement it in the shared session maintenance path with CLI, schema/help, docs, Gateway cleanup behavior, and regression tests kept in sync.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics, not a broken documented behavior. Source inspection confirms current main has only global session cleanup caps while cron-created sessions do carry labels.

## Solution Assessment

Unclear; per-label retention is plausible, but the proposed CLI/config shape needs maintainer API judgment before implementation. A safer direction is an additive schema under the existing `session.maintenance` surface rather than the requested `sessions.maintenance` spelling.
