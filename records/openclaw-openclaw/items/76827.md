## Summary

Keep open: current main still only supports global session cleanup limits, while this asks for new per-label retention semantics. Related #76826 and #77021 cover dry-run label summaries, but they do not implement retaining N sessions per label/job.

## What This Changes

The issue asks for `openclaw sessions cleanup` to keep the newest N sessions per label or cron job type instead of only applying global age/count cleanup.

## Best Possible Solution

Choose an additive `session.maintenance` API first, then implement grouped retention in the shared session maintenance path with CLI, Gateway protocol, docs, changelog, and regression tests kept in sync.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken documented behavior. Source inspection confirms current main lacks per-label cleanup semantics while already storing cron labels that could be used for grouping.

## Solution Assessment

Unclear; per-label retention is plausible, but the issue proposes both CLI and config APIs and changes retention semantics. Maintainers should settle the API and interaction order before implementation.
