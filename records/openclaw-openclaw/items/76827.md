## Summary

Keep open: current main still supports only global session age/count/disk cleanup, while this asks for new per-label retention semantics that require maintainer API/product choice before implementation.

## What This Changes

Add per-label retention to `openclaw sessions cleanup` so operators can keep the newest N sessions for each cron/job label instead of only applying a global cap.

## Best Possible Solution

Define an additive `session.maintenance` retention contract first, then implement label grouping in the shared session maintenance path with CLI, Gateway RPC, schema, docs, and focused regression tests kept in sync.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics, not a broken documented behavior. Source inspection does confirm current main lacks per-label grouping while cron labels exist.

## Solution Assessment

Unclear; per-label retention is plausible, but the exact CLI/config/protocol shape is a maintainer product/API decision before a narrow implementation should start.
