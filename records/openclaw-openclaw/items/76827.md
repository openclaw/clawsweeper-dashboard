## Summary

Keep open: current main still lacks grouped keep-N retention by session label. Recent model-run cleanup and label-aware dry-run summaries are adjacent, but they do not implement the requested destructive retention behavior, so this needs maintainer product/API direction.

## What This Changes

Request adds grouped session cleanup retention that keeps the newest N sessions per label/job type instead of relying only on global age/count caps.

## Best Possible Solution

Adopt one maintainer-approved session-retention contract that either supports grouped keep-N by label directly or folds it into the broader retention-rule design, then update schema, CLI/Gateway cleanup, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main exposes only global session-maintenance age/count/disk-budget fields and cleanup functions, while grouped keep-N by label is absent. This is a feature-gap reproduction rather than a failing bug repro.

## Solution Assessment

Unclear until maintainers choose the retention contract. Per-label keep-N is plausible, but it should not be added as a parallel destructive cleanup knob without coordinating with the broader retention-rule design.
