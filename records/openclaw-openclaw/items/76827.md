## Summary

Keep open: current main still supports only global session cleanup limits, while per-label retention would add new retention semantics plus CLI/config/Gateway API surface that needs maintainer design first.

## What This Changes

Add per-label session cleanup retention so the last N sessions can be kept for each session label, especially cron job labels.

## Best Possible Solution

Decide an additive `session.maintenance` and CLI/Gateway contract, then implement grouping in the shared session maintenance path with docs and focused regression tests kept in sync.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics, not broken documented behavior. Source inspection confirms current main lacks per-label cleanup fields while cron sessions already carry labels that could support the feature.

## Solution Assessment

Unclear; per-label retention is a plausible direction, but the exact CLI/config/protocol shape and ordering with existing cleanup rules is a maintainer product/API decision before implementation.
