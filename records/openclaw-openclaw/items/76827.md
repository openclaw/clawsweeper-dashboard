## Summary

Keep open: current main still only supports global session cleanup limits, while per-label retention would add new cleanup semantics plus CLI/config/Gateway surface that needs maintainer API and product review.

## What This Changes

Request to add per-label retention for `openclaw sessions cleanup`, keeping the newest N sessions per label or cron job instead of only applying global age/count limits.

## Best Possible Solution

Choose the additive `session.maintenance`, CLI, and Gateway protocol shape first, then implement label grouping in shared session maintenance with docs and regression coverage kept in sync.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics, not broken documented behavior. Source inspection does confirm current cleanup lacks label grouping while cron-created sessions already carry `Cron:` labels.

## Solution Assessment

Unclear; per-label retention is plausible, but the exact CLI/config/protocol design is a maintainer product/API decision before a narrow implementation should start.
