## Summary

Keep open: current main still implements global session cleanup age/count limits, while this request adds new per-label keep-N retention semantics that are not covered by the related per-pattern age-retention issue or the open per-agent maintenance PR.

## What This Changes

The issue requests per-label retention for `openclaw sessions cleanup`, so cron-style labels can keep the newest N sessions per label instead of only using global cleanup limits.

## Best Possible Solution

Decide the additive retention-policy contract first, then implement per-label or rule-based grouping in the shared session maintenance path with CLI, Gateway schema, config schema/help, docs, and focused pruning tests kept in sync.

## Reproduction Assessment

Not applicable; this is a feature request for new retention semantics, not a broken documented behavior. Source inspection confirms current main still lacks per-label grouping while cron sessions already carry labels that could support it.

## Solution Assessment

Unclear; per-label keep-N retention is plausible, but the CLI/config/API shape and its relationship to broader per-pattern retention need maintainer design approval before implementation.
