Keep open. Current main still has the docs ambiguity, and the issue is already linked to open closing PRs, so it should remain open until a fix such as https://github.com/openclaw/openclaw/pull/80678 lands.

## What This Changes

Clarify the browser docs so `openclaw browser wait --load networkidle` is documented as unsupported only for existing-session profiles rather than universally unsupported.

## Best Possible Solution

Land the focused docs clarification in https://github.com/openclaw/openclaw/pull/80678 or an equivalent patch that states the unsupported case is existing-session/Chrome MCP while preserving Playwright-backed managed and remote-CDP support.

## Reproduction Assessment

Yes. Reading current main reproduces the ambiguity: the browser-control reference presents `--load networkidle` without a caveat, while the browser overview and CLI docs list the unsupported case under existing-session limits.

## Solution Assessment

Yes. A docs-only clarification is the narrowest maintainable fix; changing runtime support would be a separate browser feature request, and the wording should not incorrectly exclude raw remote-CDP profiles.
