Keep open: current main appears to handle managed dreaming cron recreation, but the reported zero-promotion state is still source-plausible and overlaps unresolved memory-core promotion semantics plus the still-open active-memory lane issue.

The issue reports that memory-core dreaming does not recreate its managed cron after re-enable and that short-term recall entries remain stuck below promotion thresholds despite active-memory now completing.

## Best Possible Solution

Keep the current managed cron reconciliation model, then define one memory-core recovery policy for stale low-signal session-corpus entries, including operator-visible reset/backfill guidance and regression coverage.

## Reproduction Assessment

Partly: source inspection gives a high-confidence zero-candidate path when session-ingested entries stay at dailyCount=1, score=0.58, and one query/day. I did not establish a live Windows reproduction for the cron recreation symptom, and current main has tests for that path.

## Solution Assessment

Unclear: automatic age- or dailyCount-based fallback may recover the reporter's stuck state, but it changes promotion quality semantics and should be decided alongside the existing promotion-signal threads.
