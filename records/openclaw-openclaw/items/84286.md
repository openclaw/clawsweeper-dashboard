## Summary

Keep open: current main has strong scheduler reconciliation for managed dreaming cron jobs, but the zero-promotion path remains plausible for low-signal session-ingested recall entries and is not resolved by the open active-memory lane-deadlock report alone.

## What This Changes

The issue reports that memory-core dreaming does not recreate its managed cron job after re-enable and that recall entries with active-memory now completing still never reach durable promotion.

## Best Possible Solution

Keep the current cron reconciliation model, then define a bounded memory-core recovery path for stale low-signal session-corpus entries that preserves promotion quality while giving operators a documented reset or backfill route.

## Reproduction Assessment

Partly: source inspection gives a high-confidence path for zero candidates when session-ingested entries stay at signalCount=1, score=0.58, and one query/day. I did not establish a live Windows reproduction for the cron recreation symptom, and current main has tests for that path.

## Solution Assessment

Unclear: automatic age/dailyCount fallback may solve the reporter's stuck state, but it changes promotion policy and needs maintainer judgment; cron reconciliation itself already appears to be the right current-main shape.
