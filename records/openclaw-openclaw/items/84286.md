## What This Changes

The issue reports that memory-core dreaming does not recreate its managed cron after re-enable and that short-term session recall entries remain unpromoted despite active-memory running.

## Best Possible Solution

Keep the current cron reconciliation model, then settle one memory-core promotion/recovery policy that coordinates the open active-memory and promotion-signal work and gives operators a documented reset or backfill route if maintainers want one.

## Reproduction Assessment

Yes at source level for the zero-promotion part: current ranking filters one-signal, one-query session-ingested entries before promotion. No for the cron recreation part: current main has source and test coverage for managed job creation, but I did not run a live Windows reproduction.

## Solution Assessment

Unclear as an automatic fix: a fallback based on age or dailyCount could unblock this state, but it changes promotion quality policy and overlaps the open signal-contract and active-memory lanes.
