## Summary

The PR addresses a real CLI-to-Discord chunking gap, but it introduces a generic core resolver for plugin-owned account configuration and still lacks after-fix live Discord proof. Current Discord code already owns effective account merging, including default-account selection and top-level fallback, so the proposed second resolver is not the best permanent boundary and can drift from channel behavior.

## What This Changes

The PR resolves `maxLinesPerMessage` from channel and account config in `sendMessage`, adds it to durable-delivery formatting, and adds five unit tests.

## Best Possible Solution

Expose or invoke a channel-owned outbound-formatting resolver after effective account resolution, use it for durable CLI sends, and verify that a configured 20-line Discord send remains one message in real readback.

## Reproduction Assessment

Yes, from source with high confidence: current `sendMessage` omits `maxLinesPerMessage`, while Discord consumes that formatting field and otherwise falls back to its default. The linked issue also supplies a concrete released-version CLI/readback reproduction, though this review did not execute it.

## Solution Assessment

No. Forwarding effective formatting is the right fix, but generic core should not duplicate Discord’s plugin-owned account/default/fallback rules; the channel owner should resolve the value and core should forward it.
