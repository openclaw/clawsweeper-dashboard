## Summary

Keep open: current main has a guarded live catalog SDK and official OpenAI live filtering, but custom OpenAI-compatible base URLs still intentionally skip `/models`, and the remaining work needs maintainer decisions on provider API shape, merge/replace behavior, cache scope, refresh lifecycle, and credential safety.

## What This Changes

The issue requests an opt-in provider catalog refresh that fetches configured OpenAI-compatible provider `/v1/models`, merges or replaces bundled rows, and refreshes picker/catalog state.

## Best Possible Solution

Settle the custom OpenAI-compatible live-catalog contract, then implement it through the existing guarded provider-catalog SDK with auth-scoped caching, static fallback, picker/list refresh behavior, and SecretRef-safe generated catalog handling.

## Reproduction Assessment

Yes, source-level: current main intentionally skips live `/models` discovery for custom OpenAI-compatible base URLs and has a test locking in that behavior.

## Solution Assessment

No, the proposed new `resolveDynamicCatalog` seam is not the best fix as-is; the safer path is to extend the existing guarded provider catalog runtime after maintainers decide semantics and security policy.
