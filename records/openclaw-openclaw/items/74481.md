## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and the linked same-author PR is open but intentionally leaves the catalog-resolution and UI refresh plumbing undecided.

## What This Changes

Request an opt-in dynamic provider catalog path that refreshes OpenAI-compatible model picker entries from the configured provider `/v1/models` endpoint, with cache, merge/replace, and Control UI refresh behavior.

## Best Possible Solution

Choose the provider-catalog contract first, then land provider-owned OpenAI-compatible discovery that performs guarded credentialed `/models` fetches, defines deterministic merge/replace and cache invalidation semantics, updates Gateway and Control UI refresh behavior, and includes focused tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from current-main source inspection: configured `openai` `baseUrl`/`apiKey` without explicit `models` rows has no path that refreshes picker entries from upstream `/v1/models`.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but current docs and contracts suggest extending the existing provider `catalog` and explicit refresh/cache path may be narrower than adding a separate `resolveDynamicCatalog` hook.
