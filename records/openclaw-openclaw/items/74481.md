## Summary

Keep open. Current main still lacks a configured OpenAI-compatible `/models` catalog refresh path for the Control UI picker, and the same-author open PR #74488 is a first implementation slice rather than a merged fix.

## What This Changes

Request an opt-in provider-backed dynamic model catalog refresh from configured OpenAI-compatible `/v1/models` endpoints, with cache, merge/replace behavior, and Control UI refresh plumbing.

## Best Possible Solution

Define the provider catalog contract first, then integrate provider-owned OpenAI-compatible discovery into Gateway and Control UI catalog refresh with deterministic merge/replace behavior, credential/base URL cache invalidation, tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. The absence is high-confidence from current-main source inspection: a configured `baseUrl` and `apiKey` without explicit `models` does not trigger a provider `/models` fetch or add discovered rows to the picker.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but the existing provider `catalog`/`buildProvider` path already supports credentialed provider-owned discovery, so that may be narrower than introducing a separate `resolveDynamicCatalog` hook.
