## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and same-author PR #74488 is open but intentionally stops before the catalog plumbing and design questions.

## What This Changes

The issue requests an opt-in dynamic model catalog refresh from configured provider `/v1/models` endpoints so OpenAI-compatible upstreams can drive the Control UI model picker.

## Best Possible Solution

Choose the provider-catalog contract first, then land a provider-owned OpenAI-compatible discovery path that performs guarded credentialed `/models` fetches, defines deterministic merge/replace behavior, invalidates on baseUrl/auth changes, updates Gateway and Control UI refresh behavior, and includes focused tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from current-main source inspection: a configured OpenAI `baseUrl`/`apiKey` without explicit `models` rows has no path to refresh picker entries from upstream `/v1/models`.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but the narrowest maintainable solution may be to extend the existing provider `catalog` discovery path rather than introduce a new `resolveDynamicCatalog` hook.
