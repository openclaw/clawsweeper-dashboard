## Summary

Keep open. Current main still lacks a configured OpenAI-compatible `/models` catalog refresh for the picker, and the linked open PR only adds a first discovery-helper slice without the provider contract, Gateway, or Control UI integration.

## What This Changes

Request dynamic model-catalog discovery from configured provider `/v1/models` endpoints, especially OpenAI-compatible proxies, plus picker refresh and merge/replace behavior.

## Best Possible Solution

Choose a provider-owned catalog seam, then implement bounded credentialed discovery with deterministic merge behavior, cache invalidation, Gateway/Control UI refresh wiring, tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request rather than a broken documented behavior. Source inspection gives high confidence that current main does not fetch configured OpenAI-compatible `/models` endpoints for the picker.

## Solution Assessment

Unclear until maintainers choose the seam. The feature direction is valid, but reusing existing provider-owned catalog surfaces may be cleaner than adding the proposed `resolveDynamicCatalog` hook exactly as written.
