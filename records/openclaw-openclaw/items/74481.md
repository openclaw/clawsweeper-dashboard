## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and the linked PR #74488 is an open partial implementation that intentionally does not wire discovery into catalog resolution or UI refresh.

## What This Changes

The issue requests an opt-in dynamic model-catalog refresh from configured OpenAI-compatible provider `/v1/models`, including provider plumbing, Control UI refresh behavior, caching, and merge/replace semantics.

## Best Possible Solution

Define the provider-catalog contract first, then implement provider-owned OpenAI-compatible discovery that uses guarded credentialed fetches, deterministic merge/replace behavior, cache invalidation on base URL/auth changes, Gateway and Control UI refresh behavior, focused tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from current-main source inspection: OpenAI catalog rows are static/synthetic, configured models are only read from config, and `models.list` has no refresh parameter or provider discovery trigger.

## Solution Assessment

Unclear until maintainers choose the seam. The requested direction is valid, but the narrowest maintainable solution may extend the existing provider catalog/cache contracts rather than add a brand-new hook name and lifecycle.
