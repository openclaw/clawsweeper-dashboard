## Summary

Keep open because current main has a guarded live-provider catalog SDK and uses it for official OpenAI catalogs, but OpenAI-compatible custom base URLs still intentionally fall back to static manifest rows and the linked implementation PR remains open with design, security, and proof blockers.

## What This Changes

Request dynamic provider model-catalog discovery and refresh from configured `/v1/models` endpoints, including OpenAI-compatible proxy catalogs and Control UI picker refresh.

## Best Possible Solution

Decide the custom OpenAI-compatible live-catalog contract, then implement it through the shared guarded provider-catalog SDK with explicit auth-scoped caching, fallback, UI/list refresh, and #88562-compatible credential redaction.

## Reproduction Assessment

Yes for source-level reproduction: custom OpenAI-compatible base URLs still skip live fetch in `buildOpenAILiveProviderConfig`, and the test suite asserts that skip.

## Solution Assessment

Not yet; the requested direction is plausible, but the best fix should reuse the merged shared live-catalog SDK rather than the older standalone #74488 helper, with maintainer approval for custom-upstream, cache, and security semantics.
