## Summary

Keep open: current main has shared live-catalog infrastructure and picker/list hooks, but OpenAI-compatible custom base URLs still intentionally skip live `/models` discovery, and the remaining refresh behavior overlaps product, auth-provider, and generated-catalog security decisions.

## What This Changes

The issue requests opt-in dynamic model catalog refresh from configured provider `/v1/models` endpoints, starting with OpenAI-compatible providers and extending to picker/list refresh behavior.

## Best Possible Solution

Settle the custom OpenAI-compatible live-catalog contract, then implement it through the guarded provider-catalog SDK with auth-scoped caching, static fallback, picker/list semantics, and SecretRef-safe generated catalog re-derivation.

## Reproduction Assessment

Yes. Source inspection shows current main still skips live `/models` discovery for custom OpenAI-compatible base URLs, and a focused OpenAI provider test asserts that behavior.

## Solution Assessment

No, not as proposed. A new standalone `resolveDynamicCatalog` hook is not proven best; the safer direction is extending the existing guarded provider-catalog runtime after maintainer decisions on trust, cache scope, refresh lifecycle, and generated catalog safety.
