## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and the same-author paired PR #74488 remains open as an implementation/design discussion rather than a merged fix.

## What This Changes

The issue requests an opt-in dynamic model catalog refresh path that uses configured provider `/v1/models` endpoints, merges or replaces the static catalog, and refreshes the Control UI picker when provider configuration changes.

## Best Possible Solution

Settle the provider catalog contract first, then land a provider-owned OpenAI-compatible discovery path that reuses the existing catalog/discovery architecture, guards credentialed fetches, scopes cache keys to base URL plus auth or tenant context, defines deterministic merge/replace behavior, wires Gateway and Control UI refresh behavior, and adds focused tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from source inspection: OpenAI has only static catalog rows plus synthetic forward-compatible augmentation, and `models.list` exposes no configured provider refresh surface.

## Solution Assessment

Unclear until maintainers choose the seam. The requested provider-owned discovery direction matches existing patterns, but the maintainable fix may be extending the existing `catalog`/discovery path instead of adding a separate `resolveDynamicCatalog` hook.
