## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and the paired open PR #74488 is explicitly only a first slice, so this remains feature/API design work rather than cleanup-closeable work.

## What This Changes

Request an opt-in provider-backed dynamic model catalog refresh that fetches configured OpenAI-compatible `/v1/models` and feeds the Control UI picker with merge/replace semantics.

## Best Possible Solution

Define the provider-owned catalog discovery contract first, then wire OpenAI-compatible discovery into gateway and Control UI listing with deterministic merge/replace behavior, cache invalidation, tests, docs, and changelog coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature/API request; source inspection gives high confidence that current main does not fetch configured OpenAI `/models` or refresh the picker from `baseUrl` plus API key.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the seam; the behavior is valid, but the narrowest maintainable path may extend the existing provider `catalog` flow rather than add `resolveDynamicCatalog` exactly as proposed.
