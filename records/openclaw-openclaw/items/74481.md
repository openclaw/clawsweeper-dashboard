## Summary

Keep open. Current main still lacks a configured OpenAI-compatible `/models` refresh path for the Control UI picker, and the linked implementation PR is still only a first slice, so this remains feature/API design work rather than cleanup-closeable work.

## What This Changes

Request an opt-in provider-owned dynamic catalog refresh that fetches OpenAI-compatible `/v1/models`, merges or replaces static rows, and refreshes the Control UI picker when provider config changes.

## Best Possible Solution

Define the provider-owned catalog contract first, then wire OpenAI-compatible discovery into Gateway and Control UI listing with deterministic merge/replace behavior, cache invalidation, tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. Source inspection gives high confidence that configured `baseUrl` plus API key does not currently make main fetch upstream `/models` or add discovered rows to the picker.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but the narrowest maintainable path likely needs to build on the existing provider `catalog` and explicit catalog-cache flow rather than adding the proposed hook shape exactly as written.
