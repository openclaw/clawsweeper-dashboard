## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and the paired same-author PR #74488 is open and intentionally leaves the catalog plumbing/design questions unresolved.

## What This Changes

Request to add opt-in dynamic model catalog refresh for configured OpenAI-compatible providers by fetching `/v1/models`, mapping discovered metadata, caching it, and surfacing it in the Control UI model picker.

## Best Possible Solution

Choose the provider-catalog contract first, then land a provider-owned OpenAI-compatible catalog refresh that performs guarded credentialed discovery, defines deterministic merge/replace behavior, invalidates cache on baseUrl/auth changes, updates Gateway and Control UI refresh behavior, and adds focused tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from current-main source inspection: direct OpenAI catalog data is static/synthetic, while `models.list` has no configured OpenAI `/v1/models` refresh path.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but the narrowest maintainable fix may be to extend the existing provider catalog/discovery hooks rather than add a brand-new hook name and lifecycle.
