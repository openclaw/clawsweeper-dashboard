## Summary

Keep open: current main has a shared guarded live-catalog SDK and some provider discovery, but custom OpenAI-compatible base URL discovery and ordinary Control UI configured-picker refresh remain intentionally limited; the linked implementation PR is still open, conflicting, and blocked on product/security/API decisions.

## What This Changes

The issue requests opt-in dynamic model catalog refresh from configured provider `/v1/models` endpoints so model pickers and provider catalogs reflect the configured upstream instead of only bundled static rows.

## Best Possible Solution

Decide the provider catalog contract first, then implement any approved custom-upstream refresh through the existing guarded live-catalog SDK with explicit cache, fallback, UI refresh, and SecretRef-safe persistence behavior.

## Reproduction Assessment

Yes for source-level behavior: current main explicitly skips live OpenAI discovery for custom OpenAI-compatible base URLs, and tests lock that behavior. I did not run a live gateway reproduction for the later sidecar-staleness comments.

## Solution Assessment

Unclear until maintainers decide the product/security contract. A standalone new hook is only plausible; the existing `provider-catalog-live-runtime` SDK path is the better owner boundary if this feature is approved.
