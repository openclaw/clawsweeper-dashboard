## Summary

Keep open. Current main still has no configured OpenAI-compatible `/v1/models` catalog refresh path for the Gateway/Control UI picker, and the related same-author PR #74488 is still an open first-slice helper rather than integrated catalog/UI plumbing.

## What This Changes

Request an opt-in provider-backed dynamic model catalog refresh that fetches configured OpenAI-compatible `/v1/models` data, merges or replaces static entries, and updates Control UI pickers.

## Best Possible Solution

Settle the additive provider catalog contract first, then implement provider-owned discovery with deterministic merge/replace behavior, cache invalidation on auth/base URL changes, existing fetch guards, tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from source inspection: `baseUrl` plus `apiKey` does not trigger a `/models` fetch, and the Control UI has no provider refresh parameter.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but the maintainable route may be extending the existing provider catalog/augmentation flow rather than adding a separate `resolveDynamicCatalog` hook.
