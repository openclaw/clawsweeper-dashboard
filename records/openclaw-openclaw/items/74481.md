## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and the same-author paired PR #74488 is still the active implementation discussion rather than a merged fix.

## What This Changes

Request an opt-in dynamic catalog refresh for configured OpenAI-compatible providers from `/v1/models`, with caching, metadata mapping, merge/replace behavior, and Control UI picker refresh.

## Best Possible Solution

Define the provider catalog contract first, then land a provider-owned OpenAI-compatible refresh path that reuses existing catalog/discovery conventions where possible, applies guarded baseUrl/API-key-scoped caching, documents merge behavior, wires Gateway and Control UI refresh behavior, and adds focused tests plus changelog/docs coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from current-main source inspection: OpenAI remains static plus synthetic forward-compat rows, while `models.list` and the Control UI have no configured OpenAI `/v1/models` refresh path.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but the best maintainable fix may be extending the existing provider catalog/discovery contract rather than adding a separate `resolveDynamicCatalog` hook.
