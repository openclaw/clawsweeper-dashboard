## Summary

Keep open. Current main still lacks an OpenAI-compatible configured-provider `/models` refresh path for the Control UI picker, and the linked same-author PR is still open and intentionally does not wire discovery into catalog resolution.

## What This Changes

Request an opt-in dynamic catalog refresh that fetches configured OpenAI-compatible `/models` endpoints and updates model picker rows with merge or replace behavior.

## Best Possible Solution

Define the provider-owned catalog refresh contract, then wire OpenAI-compatible discovery into the Gateway and Control UI listing path with deterministic cache invalidation, tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from source inspection: configured `baseUrl` and `apiKey` do not currently cause main to fetch upstream `/models` or add discovered rows to the picker.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but the narrowest maintainable solution may extend the existing provider `catalog`/cache path rather than adding the proposed `resolveDynamicCatalog` hook exactly as written.
