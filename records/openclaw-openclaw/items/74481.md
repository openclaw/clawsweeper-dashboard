## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and same-author PR #74488 is open but only adds an unmerged discovery helper without catalog or UI plumbing.

## What This Changes

Request opt-in dynamic model catalog refresh from configured provider `/v1/models` endpoints, starting with OpenAI and later Anthropic, with merge/replace and cache semantics.

## Best Possible Solution

Choose the catalog contract first, then land guarded provider-owned OpenAI-compatible discovery that integrates with Gateway/Control UI catalog refresh, has deterministic merge or replace semantics, invalidates on auth/base URL changes, and includes tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from source inspection: with only `models.providers.openai.baseUrl` and `apiKey`, current main has no path that fetches upstream `/models` or adds discovered rows to the configured picker.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but the narrowest maintainable path may be extending the existing `ProviderPlugin.catalog`/static catalog flow rather than adding a separate `resolveDynamicCatalog` hook.
