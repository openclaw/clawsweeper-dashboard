## Summary

Keep open. Current main still lacks a configured OpenAI-compatible `/models` refresh path for the Control UI picker, and open PR #74488 is only a first slice, so the remaining work is a feature/API design decision rather than cleanup-closeable work.

## What This Changes

Request an opt-in dynamic model catalog refresh that fetches configured provider `/v1/models` responses and uses them to update the Control UI model picker.

## Best Possible Solution

Define the provider-owned catalog contract first, then wire OpenAI-compatible discovery into Gateway and Control UI listing with deterministic merge/replace behavior, cache invalidation, tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from source inspection: configured `baseUrl` and API key do not currently cause main to fetch upstream `/models` or add discovered rows to the picker.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but the narrowest maintainable path may be extending the existing provider `catalog` flow rather than adding the proposed `resolveDynamicCatalog` hook exactly as written.
