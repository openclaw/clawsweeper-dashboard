## Summary

Keep open: current main has a guarded live catalog SDK and picker/list hooks, but custom OpenAI-compatible base URLs still intentionally skip provider `/models` discovery, and the remaining work needs maintainer decisions on endpoint trust, cache scope, refresh behavior, and generated catalog credential safety.

## What This Changes

Request to add opt-in dynamic provider model catalog refresh from configured provider `/v1/models`, including OpenAI-compatible gateways and picker refresh behavior.

## Best Possible Solution

Settle the custom OpenAI-compatible live-catalog contract, then implement it through the guarded provider-catalog SDK with auth-scoped caching, static fallback, picker/list refresh behavior, and SecretRef-safe generated catalog re-derivation.

## Reproduction Assessment

Yes, source-level: current main intentionally skips live `/models` discovery for custom OpenAI-compatible base URLs, and a focused test asserts the live fetch guard is not called for that case. I did not run a live proxy or Control UI flow.

## Solution Assessment

No, the proposed new `resolveDynamicCatalog` hook is not proven as the best solution. The safer path is to extend the existing guarded provider-catalog runtime after maintainers decide endpoint trust, cache scope, merge/replace behavior, and refresh lifecycle.
