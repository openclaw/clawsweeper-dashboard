## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and the same-author paired PR #74488 remains the active implementation discussion rather than a merged fix.

## What This Changes

Request an opt-in dynamic catalog refresh path so OpenAI-compatible configured provider base URLs can populate and update model picker entries from `/v1/models` instead of relying only on bundled static catalog rows.

## Best Possible Solution

Choose the provider-catalog contract first, then land a provider-owned OpenAI-compatible catalog refresh that reuses existing catalog/discovery patterns where possible, applies guarded credential/baseUrl-scoped caching, defines deterministic merge or replace behavior, wires Gateway and Control UI refresh behavior, and adds focused tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from current-main source inspection: OpenAI remains static plus forward-compatible synthetic model resolution, while `models.list` has no configured OpenAI `/v1/models` refresh trigger.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but the best maintainable fix may be extending the existing provider catalog/discovery path rather than adding a brand-new `resolveDynamicCatalog` hook.
