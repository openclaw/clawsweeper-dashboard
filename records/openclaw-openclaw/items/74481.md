## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and the same-author open PR #74488 is only a first slice that intentionally does not wire discovery into catalog resolution or the UI.

## What This Changes

The issue requests an opt-in provider-backed dynamic catalog refresh so configured OpenAI-compatible upstream `/v1/models` responses can populate the Control UI model picker.

## Best Possible Solution

Settle the provider-catalog contract first, then land a provider-owned OpenAI-compatible discovery path that uses the agreed cache/refresh mechanism, defines deterministic merge or replace behavior, invalidates on baseUrl/auth changes, updates Gateway and Control UI refresh behavior, and adds SDK docs, provider docs, tests, and a changelog entry.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from current-main inspection: OpenAI catalog data is static/synthetic, configured provider models are config-only, and `models.list` has no provider refresh parameter or configured OpenAI `/v1/models` fetch path.

## Solution Assessment

Unclear. The requested behavior is valid, but maintainers should choose whether the narrowest durable solution is a new `resolveDynamicCatalog` hook, the existing provider `catalog`/explicit cache path, or another additive SDK seam before implementation proceeds.
