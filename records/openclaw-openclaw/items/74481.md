## Summary

Keep open. Current main still lacks a configured OpenAI-compatible `/models` refresh path for the Control UI picker, and the linked same-author PR is an open first slice that explicitly does not wire discovery into catalog resolution.

## What This Changes

Request an opt-in provider catalog refresh path that fetches configured OpenAI-compatible `/models` endpoints and updates the model picker with merge or replace semantics.

## Best Possible Solution

Define the provider catalog contract first, then land provider-owned OpenAI-compatible discovery wired into Gateway and Control UI listing with deterministic merge or replace behavior, cache invalidation, tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from source inspection: configured `baseUrl` and `apiKey` do not currently cause main to fetch upstream `/models` or add discovered rows to the picker.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but the narrowest maintainable solution may be extending the existing provider `catalog` path rather than introducing a separate `resolveDynamicCatalog` hook exactly as proposed.
