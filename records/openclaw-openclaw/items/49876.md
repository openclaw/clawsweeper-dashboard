## Label Justifications

- `P1`: The issue describes false success delivery in unattended and interactive agent workflows, which can mislead real operators when tool-backed work did not happen.
- `impact:session-state`: The failure is drift between final assistant text and the actual tool-call, delivery, and session ledger state.
- `impact:security`: The examples include fabricated external operations and destructive instructions after failed or missing tool calls, making this a trust-boundary concern.

## Summary

Keep open: current main has partial warning and fatal-denial hardening, but it still does not implement a universal fail-closed or ledger-backed finalization policy for unsupported cron/session success output after failed or missing tool work.

## What This Changes

Issue requests fail-closed or ledger-backed handling for cron and chat sessions so agents cannot deliver fabricated operational output when required tool calls did not actually succeed.

## Best Possible Solution

Define a maintainer-approved core finalization policy that uses structured tool-result and run-state evidence to suppress, replace, or clearly mark unsupported cron/session success output without relying only on brittle text markers.

## Reproduction Assessment

Yes at source level: current main still has paths where final assistant text can win after selected non-fatal tool warnings or recoverable tool errors, though the original Telegram/Gemini setup was not rerun live.

## Solution Assessment

No complete solution exists on current main; warning payloads and narrow fatal-denial signals are useful partial hardening, but the remaining fix needs maintainer policy for fail-closed finalization and unsupported success claims.
