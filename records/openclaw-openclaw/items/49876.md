## Summary

Keep open: current main has some cron/agent error hardening, but it still lacks a general finalization boundary that prevents successful-looking assistant output after failed, missing, or unsupported tool work from being delivered as success.

## What This Changes

Request fail-closed handling for cron and agent sessions so failed or unverified tool work cannot be announced as successful output.

## Best Possible Solution

Add a core finalization and delivery policy that treats unsupported success after failed or unattempted tool work as an explicit failed or suppressed run, with maintainer-approved semantics and regression coverage across cron and interactive paths.

## Reproduction Assessment

Yes by source inspection: current main still has paths where successful-looking visible output can be terminal after tool errors, while existing safeguards cover only narrower structured-error, warning, or planning-only cases.

## Solution Assessment

No, the current partial hardening is not the best complete solution; a structural finalization check tied to tool/error ledger state is safer than only wording changes or text-pattern filters.
