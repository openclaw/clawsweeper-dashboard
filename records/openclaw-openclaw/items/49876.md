## Summary

Keep open: current main has adjacent cron hardening for structured denials, run-level errors, NO_REPLY, and some mutating-tool warnings, but it still lacks the requested fail-closed delivery policy for fabricated final text, ordinary tool failures, and unsupported success claims.

## What This Changes

Request fail-closed validation for isolated cron delivery so tool failures, incomplete runs, and fabricated or placeholder output are suppressed rather than announced.

## Best Possible Solution

Keep this open for a maintainer-owned output-integrity policy, likely starting in warn or opt-in strict mode, that combines run/tool failure state, required-resource preconditions, suspicious draft-language suppression, and ledger-backed success-claim checks without breaking existing NO_REPLY, denial, failure-alert, and message-tool delivery behavior.

## Reproduction Assessment

Yes for the current source gap: a non-empty fabricated-looking assistant payload without runLevelError, fatal failureSignal, denial markers, or structured error status is still selected and can reach delivery. The original live incidents were not replayed here.

## Solution Assessment

No for a marker-filter-only fix; it would catch some examples but miss clean unsupported success claims and tool-never-attempted cases. A structured policy tied to run/tool evidence is the narrower maintainable direction.
