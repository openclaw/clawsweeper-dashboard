## Summary

Keep open. Current main has warning and incomplete-turn safeguards, but it still lacks the requested fail-closed or ledger-backed delivery contract for cron output after ordinary tool failures, and related live reports show the failure is broader than the original Gemini cron examples.

## What This Changes

The issue requests fail-closed cron/session delivery so outputs after failed or missing tool calls are suppressed or reported as clean errors instead of delivered as model-generated success or fabricated data.

## Best Possible Solution

Define a maintainer-approved unattended-output contract that uses structured tool/run metadata for cron delivery, documents default versus strict behavior, and handles broader claim verification without relying on phrase filters alone.

## Reproduction Assessment

No high-confidence live current-main repro was run. Source inspection and current tests show a source-reproducible path where final assistant text remains deliverable or recoverable after ordinary tool-warning payloads when current fatal checks do not match.

## Solution Assessment

Unclear. The requested hard-coded phrase filter is not sufficient by itself; the maintainable direction is a structured run/tool status policy with explicit cron defaults, opt-in strictness, and a decision about which generic claim-verification policies belong in core versus plugins.
