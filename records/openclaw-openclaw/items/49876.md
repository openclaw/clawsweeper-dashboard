## Summary

Keep open: current main has partial safeguards that surface some mutating tool failures and fatal cron errors, but it does not implement the requested fail-closed or ledger-backed final-output boundary for cron/unattended hallucinated success after tool failures.

## What This Changes

The issue requests fail-closed handling for cron or unattended agent output when tool calls fail or output appears fabricated.

## Best Possible Solution

Keep this open for a maintainer-approved core finalization policy that prevents unattended cron/session delivery from presenting unsupported success as real output while preserving legitimate recovered runs and plugin-owned optional claim verification.

## Reproduction Assessment

Yes from source inspection: current tests exercise success-looking final output combined with tool-error payloads, and current cron code can treat some warning/error payloads as recovered by final assistant text. I did not run a live model reproduction in this read-only pass.

## Solution Assessment

Unclear. The current warning-based hardening is useful but not the full fix; the best solution needs a maintainer decision on default fail-closed behavior, opt-in policy/config, and how to distinguish real recovery from unsupported success claims.
