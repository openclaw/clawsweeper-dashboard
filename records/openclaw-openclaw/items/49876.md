## Summary

Keep open. Current main has useful narrow safeguards for cron silent tokens, structured denials, run-level errors, and mutating-tool warnings, but it still lacks the general fail-closed or ledger-backed output policy requested here; the live discussion also adds non-cron and non-Gemini examples that a simple marker filter would miss.

## What This Changes

Request fail-closed delivery behavior for cron or unattended agent output when tool calls fail, required data is missing, or completion cannot be verified.

## Best Possible Solution

Define a maintainer-approved unattended-output contract that uses structured tool/run metadata where possible, documents default and opt-in semantics, and leaves generic claim-verification policy to plugins when core hooks are sufficient.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. Source inspection gives a medium-confidence path: final assistant text can remain deliverable after ordinary tool failures when current fatal-denial checks do not match.

## Solution Assessment

Unclear. The requested hard-coded marker filter is not sufficient by itself; the maintainable direction is a structured run/tool status policy with explicit core-vs-plugin ownership and compatibility semantics.
