## Summary

Keep open. Current main has adjacent cron hardening for structured errors, denial markers, silent tokens, stale interim text, and mutating-tool warnings, but it still lacks the requested fail-closed delivery policy for fabricated final text, placeholder/mock/simulated output, or ledger-backed unsupported success claims.

## What This Changes

The issue requests fail-closed cron/session delivery when tool calls fail, data is missing, or final assistant output is fabricated, self-referential, or unsupported by real tool evidence.

## Best Possible Solution

Add a maintainer-owned unattended/finalization policy that combines explicit run/tool failure state, required-resource preconditions, fabrication-marker suppression, and ledger-backed success-claim verification, with focused regression coverage and a documented rollout mode.

## Reproduction Assessment

Yes for current source behavior: a final assistant text payload can remain deliverable after existing error/denial checks, and the tests show unsupported success text is preserved alongside a warning rather than fail-closed. The original production incidents were not replayed in this read-only pass, but the missing guard is visible in the current code path and supported by fresh discussion repros.

## Solution Assessment

Unclear for the issue’s marker-filter-only proposal; it is too narrow for the later unsupported success-claim cases. The safer maintainable direction is a structured policy that uses run/tool evidence and explicit failure signals before delivery, not only text heuristics.
