## Summary

Keep open: current main has partial cron and embedded-runner hardening for fatal denial signals and warning payloads, but it still lacks a general fail-closed or claim-vs-tool-ledger boundary for fabricated final output after tool failures.

## What This Changes

The issue requests fail-closed delivery or suppression when cron or session tool failures can lead to fabricated user-facing output.

## Best Possible Solution

Maintainers should choose the core/plugin boundary, then implement a reliable finalization policy that correlates tool results, final assistant text, and delivery before user-facing output is sent.

## Reproduction Assessment

Yes, source-reproducible: current main can still treat non-fatal tool failures or final assistant text as recoverable output rather than enforcing a structural failure/claim-verification gate.

## Solution Assessment

No complete best fix is selected yet; the current partial hardening is useful but does not settle fail-closed semantics, plugin ownership, or claim-verification behavior.
