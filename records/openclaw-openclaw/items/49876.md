## Summary

Keep open. Current main has partial guardrails for cron run-level errors, structured execution denials, and some mutating-tool failure warnings, but the broader failure remains unresolved: assistant text can still be treated as deliverable output unless a narrow error/denial/failure signal is present, and the related fabrication/tool-boundary reports remain open.

## What This Changes

The issue asks for isolated cron sessions to fail closed or suppress delivery when tool calls fail, required data is missing, or the model emits fabricated, placeholder, or self-referential output.

## Best Possible Solution

Define a maintainer-approved agent finalization policy that can use session type, tool-result state, and claim-vs-ledger evidence before outbound delivery, with plugin/API extension only where core should stay policy-neutral.

## Reproduction Assessment

Yes, at source level: current cron finalization only fails closed for structured error payloads, fatal denial metadata, recognized denial text, or run-level errors, while issue comments provide concrete live cases where failed tools are followed by fabricated final text. I did not execute a live model repro in this read-only review.

## Solution Assessment

Unclear. The requested regex suppression would catch some placeholder/mock-text cases, but comments show clean unsupported success claims too, so the safer solution is a ledger-aware finalization policy rather than a text-marker-only filter.
