## Summary

Keep open: current main has adjacent cron hardening, but it still lacks the requested fail-closed delivery policy for fabricated final text, marker phrases, ordinary tool failures, and unsupported success claims.

## What This Changes

The issue requests that cron/session delivery suppress or cleanly fail output when tool calls fail, required data is missing, or assistant text appears fabricated or unsupported by the run ledger.

## Best Possible Solution

Keep this open for a maintainer-owned output integrity policy, likely starting opt-in or warn-mode, that combines tool/run failure state, required-resource preconditions, fabricated-output marker suppression, and ledger-backed success-claim checks without breaking existing NO_REPLY, denial, failure-alert, and message-tool delivery behavior.

## Reproduction Assessment

Yes for the current source gap: a non-empty fabricated-looking assistant payload without runLevelError, fatal failureSignal, denial markers, or structured error status is still selected and can reach delivery. The original live Gemini/Qwen/Hermes incidents were not replayed here.

## Solution Assessment

No for a marker-filter-only fix; that would catch some examples but miss clean unsupported success claims and tool-never-attempted cases. A structured policy tied to run/tool evidence is the narrower maintainable direction.
