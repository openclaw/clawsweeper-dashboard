## Summary

Keep open. Current main has useful narrow safeguards for silent tokens, structured denials, run-level errors, and tool-error warnings, but it still does not define the requested fail-closed or ledger-backed delivery contract for cron output after ordinary tool failures.

## What This Changes

The issue requests fail-closed cron delivery so failed, incomplete, or unverifiable agent runs do not deliver fabricated or placeholder output.

## Best Possible Solution

Define a maintainer-approved unattended-output contract that uses structured tool/run metadata for cron, documents default and opt-in behavior, and leaves optional generic claim-verification policies to plugins where current hooks are sufficient.

## Solution Assessment

Unclear. The requested hard-coded phrase filter is not sufficient by itself; the maintainable direction is a structured run/tool status policy with explicit cron defaults, opt-in strictness, and core-vs-plugin ownership.
