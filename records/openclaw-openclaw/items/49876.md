## Summary

Keep open. Current main has warning and recovery paths for failed tools, but it still permits success-looking assistant text to remain deliverable after ordinary tool failures, and the discussion adds fresh non-cron evidence showing the failure is broader than a Gemini cron phrase-filter problem.

## What This Changes

The issue requests fail-closed or clean-error handling for cron/session output when required tool work fails or cannot be verified.

## Best Possible Solution

Define a maintainer-approved unattended-output contract that uses structured tool/run metadata for cron delivery, documents default versus strict behavior, and handles broader claim verification without phrase filters alone.

## Reproduction Assessment

No high-confidence live current-main repro was run. Source inspection and current tests show a source-reproducible path where success-looking final assistant text remains deliverable or recoverable after ordinary tool-warning payloads when current fatal checks do not match.

## Solution Assessment

Unclear. The requested hard-coded phrase filter is not sufficient by itself; the maintainable direction is a structured run/tool status policy with explicit cron defaults, opt-in strictness, and a decision about which generic claim-verification policies belong in core versus plugins.
