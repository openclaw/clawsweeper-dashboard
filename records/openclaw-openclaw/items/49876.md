## Summary

Keep open: current main has adjacent cron hardening for denial markers, run-level errors, silent replies, and stale interim text, but it still does not implement the requested fail-closed policy for fabricated normal-looking output or unsupported success claims after ordinary tool failures.

## What This Changes

Request a fail-closed cron/session delivery policy that suppresses or errors on unverified fabricated output after tool failures or incomplete task execution.

## Best Possible Solution

Define a maintainer-owned output-integrity policy, likely warn or opt-in strict first, that uses run/tool evidence and delivery preconditions rather than only keyword filters, then document the cron and broader agent-session behavior.

## Reproduction Assessment

Yes for the current source gap: a non-empty assistant payload that is not a structured error, fatal denial, run-level error, silent token, or stale interim update can still be selected and delivered. I did not replay the original live incidents in this read-only review.

## Solution Assessment

No for the marker-filter-only direction in the issue body, because recent examples include clean unsupported success claims with no placeholder/mock language. The maintainable fix should tie delivery decisions to structured run and tool-call evidence, with an explicit product decision on warn versus strict behavior.
