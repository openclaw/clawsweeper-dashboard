## Summary

Keep open. Current main has partial cron safeguards for silence tokens, stale interim text, structured error payloads, and narrow execution-denial markers, but it still lacks the requested general fail-closed delivery policy for ordinary tool failures, missing data, and unsupported success claims.

## What This Changes

The issue requests fail-closed cron/session delivery when tools fail, required data is missing, or assistant output is not backed by real tool results.

## Best Possible Solution

Define a maintainer-approved unattended-output contract that uses structured run/tool metadata, documents cron versus interactive defaults, and treats text phrase filters as supplemental rather than the core safeguard.

## Reproduction Assessment

Yes at source level: current main can still carry non-error assistant text through cron delivery after ordinary tool-failure conditions unless they match narrow fatal/denial paths. I did not run a live current-main reproduction against the reporter's model stacks.

## Solution Assessment

No, the hard-coded phrase-filter suggestion is not sufficient by itself because recent examples contain clean-looking unsupported success claims. The safer direction is a ledger/session-status delivery policy with maintainer-approved strictness and plugin/core boundaries.
