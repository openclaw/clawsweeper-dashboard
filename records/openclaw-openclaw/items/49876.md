## Summary

Keep open. Current main has adjacent cron hardening for explicit denial metadata, run-level failures, stale acknowledgement text, and NO_REPLY suppression, but it still lacks the requested fail-closed delivery policy for fabricated non-empty final assistant text, missing required tool evidence, or unsupported success claims. The latest discussion also broadens the failure mode beyond cron/Gemini, so this needs maintainer policy and architecture judgment rather than conservative cleanup closure.

## What This Changes

The issue requests a fail-closed delivery policy for unattended cron/session outputs when tool calls fail, required data is missing, or final assistant text appears fabricated or unverified.

## Best Possible Solution

Keep this open for a maintainer-owned unattended delivery policy. A good end state would separate core cron/session fail-closed behavior for explicit tool/run failure and unavailable required resources from optional ClawHub claim-verification plugins, while preserving existing NO_REPLY, denial-token, message-tool, stale-ack, and --no-deliver behavior.

## Reproduction Assessment

Yes for the current source-level gap: a non-empty final assistant payload without runLevelError, fatal failureSignal, structured error payload, or known denial token can still be classified as ok and delivered after only silent-token filtering. The supplied comments add live production and deterministic interactive examples, though those exact sessions were not replayed here.

## Solution Assessment

Unclear for the issue's marker-filter-only suggestion; it would catch some placeholder/mock/simulate incidents but miss the clean unsupported success claims described later. The narrower maintainable direction is structured run/tool-failure gating for unattended delivery, with any broader claim-vs-ledger policy designed explicitly and coordinated with plugin-scope work.
