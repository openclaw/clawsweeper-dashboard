## Summary

Keep open. Current main has adjacent hardening for cron denials, run-level errors, stale acknowledgements, silent tokens, and mutating-tool warnings, but it still lacks the requested fail-closed delivery policy for fabricated final assistant text, placeholder/mock/simulated output, and unsupported success claims backed by the tool-call ledger.

## What This Changes

The issue requests a fail-closed cron/session delivery policy so tool failures, missing required data, incomplete tasks, fabricated text, and internal draft language are not delivered as real output.

## Best Possible Solution

Keep this open for a maintainer-owned output-integrity policy at the unattended/final-message boundary. The durable end state should combine explicit run/tool failure state, exact silent-token behavior, marker filtering for self-referential draft/fabrication language, and ledger-backed checks for side-effect or data claims, with defaults chosen deliberately for cron versus interactive sessions.

## Reproduction Assessment

Yes for the current source behavior: a non-empty success-looking payload without runLevelError or fatal failureSignal is selected by resolveCronPayloadOutcome and can flow into dispatchCronDelivery. The original external Gemini incidents were not replayed locally, but the missing guard is visible in current main.

## Solution Assessment

Unclear for the issue body's marker-filter-only suggestion; it is too narrow for the later unsupported-success-claim reports. The better fix is a structured delivery/finalization policy that uses run/tool evidence first and text heuristics only as a secondary guard.
