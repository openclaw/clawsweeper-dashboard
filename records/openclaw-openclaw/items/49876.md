## Summary

Keep open. Current main has narrow cron handling for silent tokens, stale acknowledgements, and execution-denial markers, but it does not provide the general fail-closed output policy requested here; recent comments add concrete non-cron and non-Gemini examples, so this is still an unresolved delivery-policy problem rather than a cleanup close.

## What This Changes

Request for unattended cron output to fail cleanly or suppress delivery when tool failures, missing data, incomplete work, fabricated success claims, or placeholder/draft language appear.

## Best Possible Solution

Define a maintainer-approved unattended-output contract that uses structured run/tool metadata where possible, supports plugin-based policy when sufficient, and documents any core fail-closed mode or suppression semantics before implementation.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. Source inspection shows the missing gate: current code can still carry final assistant text through delivery when ordinary tool failures exist, while the issue thread supplies concrete production and deterministic model-run examples.

## Solution Assessment

Unclear. A hard-coded phrase filter is not the best standalone fix; the safer direction is a maintainer-approved ledger/session-status policy, probably with plugin hooks or a carefully scoped core cron/unattended mode.
