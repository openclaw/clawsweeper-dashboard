## Summary

Keep open. Current main contains adjacent cron hardening, but not the requested delivery-layer fail-closed policy for fabricated final assistant text, placeholder/mock/simulated output, missing required tool evidence, or unsupported success claims.

## What This Changes

The issue asks for isolated cron/session delivery to suppress or cleanly fail outputs that are fabricated, incomplete, or unsupported after tool failures instead of announcing them to users.

## Best Possible Solution

Keep this open for a maintainer-owned unattended delivery policy, likely opt-in first, that suppresses or cleanly fails cron/session delivery when required resources or tool calls failed, final output contains self-referential draft/fabrication markers, or success claims are unsupported by turn/run evidence. Preserve existing NO_REPLY, stale-ack, denial-token, run-level-error, message-tool, and --no-deliver behavior with focused regression coverage.

## Reproduction Assessment

Yes for the current source behavior: a non-empty fabricated-looking final payload without failureSignal/runLevelError is selected by resolveCronPayloadOutcome and can proceed through dispatchCronDelivery. The original live Gemini incidents were not replayed, but the missing guard is visible in the current code path.

## Solution Assessment

Unclear for the exact proposed marker-filter-only fix; that would catch some examples but miss clean unsupported success claims. The better direction is a structured unattended delivery policy that combines explicit run/tool failure state, marker filtering, and ledger-backed claim evidence rather than relying only on text heuristics.
