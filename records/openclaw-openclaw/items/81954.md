## Summary

Keep open: this maintainer-labeled member PR addresses a real missing Agents tab creation flow, but source review found blocking correctness issues in model value qualification and post-create refresh handling.

## What This Changes

Adds a Control UI Agents tab New Agent dialog backed by `agents.create`, with styling, i18n updates, controller/view tests, and a changelog entry.

## Best Possible Solution

Reuse the existing model qualification path and make the post-create list reload independent of any in-flight refresh, then validate with focused UI/controller tests and maintainer UI proof.

## Reproduction Assessment

Yes for the review findings by source inspection: the new model select submits raw catalog ids, and the new create controller can skip its verification reload when `agentsLoading` is already true. I did not run a live browser/Gateway repro in this read-only pass.

## Solution Assessment

No, not yet; the overall direction is right, but the implementation should preserve the existing provider-qualified model contract and avoid treating a successful create as failed during an overlapping list refresh.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Qualify model options before creating agents:** `ui/src/ui/views/agents.ts:466`
  - body: The create dialog stores `entry.id` as the submitted model, but catalog entries normally carry the provider separately and the existing Agents model picker uses `buildQualifiedChatModelValue(entry.id, provider)`. With a real catalog entry like `{ id: "gpt-5.5", provider: "openai" }`, this flow persists `gpt-5.5` instead of `openai/gpt-5.5`, making the new agent's initial model ambiguous and inconsistent with the existing editor. Reuse the existing model option helper or equivalent qualification here, and cover the real catalog shape in the test.
  - confidence: 0.9
- **[P2] Force the post-create list refresh:** `ui/src/ui/controllers/agents.ts:152`
  - body: After `agents.create` succeeds, this calls `loadAgents(state)` and then requires the created id to be present, but `loadAgents` returns immediately if another Agents refresh is already in progress. Since the New Agent action is still usable while the list is loading, that race can create the agent on disk/config and then show `Created agent ... was not returned by agents.list` without selecting it. Make this path wait for or force a fresh `agents.list` response before verifying the created agent.
  - confidence: 0.82
