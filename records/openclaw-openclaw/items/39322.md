## Summary

Keep open, but do not merge as-is. Current main still lacks native iMessage tapback forwarding, and the PR is useful source work, but the branch does not match the current imsg RPC contract and still needs an explicit maintainer decision for group tapback wake-up semantics; the same author also has the paired open feature issue #39031, so this PR should remain attached to that work until it is repaired, replaced, or closed by a maintainer.

## What This Changes

The PR adds an opt-in iMessage `includeReactions` config/schema/type field, subscribes to reaction events, parses reaction notifications, converts added tapbacks into synthetic `IMessagePayload` messages, and adds parser tests.

## Best Possible Solution

Refresh the implementation against current main and current imsg: enable `watch.subscribe` with `include_reactions`, parse reaction metadata from normal message notifications, keep all access-control and echo/loop protections in the existing inbound pipeline, add explicit DM and group-policy tests, then document and changelog the config only after the group wake-up behavior is intentional.

## Reproduction Assessment

Yes. A code-level reproduction is sufficient: with the PR head, `includeReactions` sends a subscription flag current imsg does not read and waits for a notification method current imsg does not emit; if those are corrected, current group mention tests show `[emoji reaction]` bodies are dropped as `no mention` under mention-gated group config.

## Solution Assessment

No. Reusing the existing iMessage inbound pipeline is the right direction, but this patch is not the best mergeable solution until it follows the current imsg RPC contract, preserves current monitor retry behavior, and deliberately defines group reaction policy.
