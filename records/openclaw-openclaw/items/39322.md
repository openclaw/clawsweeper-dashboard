## Summary

Keep this PR open, but do not merge it as-is: current main still lacks iMessage tapback ingress, and the branch’s watcher contract does not match current `imsg`; it also needs a maintainer decision for mention-gated group tapbacks.

## What This Changes

The PR adds an opt-in `channels.imessage.includeReactions` config path, a reaction parser, and synthetic iMessage inbound messages for tapback reactions.

## Best Possible Solution

Refresh against current main and current `imsg`: subscribe with `include_reactions`, parse reaction metadata from normal message notifications, define explicit DM/group forwarding policy, preserve monitor retry/sanitization behavior, and update tests, docs, changelog, and generated config metadata.

Yes. Source inspection is enough: the PR sends a subscription flag current `imsg` does not read and waits for a notification method current `imsg` does not emit; current iMessage group policy also shows sanitized reaction bodies are dropped under mention-gated group config.

## Solution Assessment

No. Reusing the existing iMessage inbound pipeline is the right direction, but this branch is not the best mergeable solution until it follows the current `imsg` RPC shape, preserves current monitor behavior, and deliberately defines group tapback semantics.
