## Summary

Keep open. The branch is a plausible feature implementation, but it still targets an obsolete imsg reaction RPC shape, lacks after-fix real behavior proof, and the paired feature issue plus the related tapback-filtering bug are still open, so closing or merging would be premature.

## What This Changes

The PR adds an opt-in iMessage `includeReactions` config path, a separate reaction parser/handler, and synthetic inbound iMessage payloads for tapback reactions.

## Best Possible Solution

Refresh or replace the branch against current main, align it with imsg's `include_reactions`/message-field contract, and resolve the paired reaction-forwarding and tapback-filtering policy before merge.

## Reproduction Assessment

Yes for the review findings. Source inspection plus the imsg RPC docs show the PR sends the wrong subscription flag and listens for a notification method current imsg does not emit.

## Solution Assessment

No. Reusing the iMessage inbound pipeline is directionally useful, but this branch is not the best mergeable solution until it follows the current imsg RPC shape and maintainers decide how reactions should behave under DM/group policy.
