## Summary

Keep open: current main still lacks inbound iMessage tapback forwarding, but this PR is not mergeable as-is because it targets the wrong current imsg reaction contract and leaves group reaction policy unresolved.

## What This Changes

The PR adds an opt-in iMessage `includeReactions` config/schema/type field, reaction parser tests, a reaction notification handler, and synthetic inbound messages routed through the existing iMessage pipeline.

## Best Possible Solution

Refresh the implementation against current main and current imsg by subscribing with `include_reactions`, parsing reaction metadata from normal message notifications, preserving the current retry/sanitized-error monitor behavior, adding explicit DM and group-policy tests, and updating docs/changelog/generated config metadata after the policy is chosen.

## Reproduction Assessment

Yes. Source inspection is enough: the PR sends a subscription flag current imsg does not read and waits for a notification method current imsg does not emit; current iMessage group tests also show sanitized reaction bodies without mentions would be dropped under mention-gated group config.

## Solution Assessment

No. Reusing the existing iMessage inbound pipeline is the right direction, but this branch is not the best mergeable solution until it follows current imsg RPC shape, preserves current monitor retry behavior, and deliberately defines group tapback semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use imsg's include_reactions subscription flag:** `extensions/imessage/src/monitor/monitor-provider.ts:587`
  - body: Current imsg reads `include_reactions` from `watch.subscribe`, not `reactions`. With this payload, enabling `includeReactions` leaves the watcher in normal message-only mode, so tapbacks will not be emitted.
  - confidence: 0.95
- **[P1] Parse reactions from message notifications:** `extensions/imessage/src/monitor/monitor-provider.ts:566`
  - body: Current imsg sends reaction events as `method: "message"` with metadata such as `is_reaction`, `reaction_type`, `reaction_emoji`, `is_reaction_add`, and `reacted_to_guid`. This `method === "reaction"` branch never runs against that contract.
  - confidence: 0.95
- **[P2] Define group reaction mention behavior:** `extensions/imessage/src/monitor/monitor-provider.ts:512`
  - body: After stripping target text, every synthetic group reaction body is `[emoji reaction]`. The existing iMessage group decision checks mentions against that text and drops mention-gated group traffic, so opt-in reactions from allowed groups will not reach the agent unless users weaken group mention policy.
  - confidence: 0.9
- **[P3] Document the public includeReactions option:** `src/config/types.imessage.ts:57`
  - body: The branch adds a user-facing `channels.imessage.includeReactions` option, but it does not update docs, changelog, or generated config metadata. Operators and schema/UI consumers will not discover the new option, and config-contract checks are likely to flag drift.
  - confidence: 0.82
