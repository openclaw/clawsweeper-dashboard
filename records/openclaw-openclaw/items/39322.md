## Summary

Keep open: current main still lacks iMessage tapback ingress, but this branch is not mergeable as-is because it targets the wrong imsg RPC shape and leaves group reaction policy unresolved.

## What This Changes

The PR adds an opt-in iMessage `includeReactions` config field, reaction parsing tests, and monitor handling that synthesizes tapbacks into inbound iMessage messages.

## Best Possible Solution

Refresh the implementation against current main and current imsg by subscribing with `include_reactions`, parsing reaction metadata on normal message notifications, preserving monitor retry behavior, defining group tapback policy, and updating docs/changelog/generated config metadata.

## Reproduction Assessment

Yes. Source inspection is enough: the PR sends a subscription key current imsg ignores and waits for a notification method current imsg does not emit, while current main has no reaction ingress path.

## Solution Assessment

No. Reusing the iMessage inbound pipeline is a good direction, but this branch needs the current imsg RPC contract, current monitor retry behavior, explicit group semantics, and config/docs follow-through before it is the right solution.

## Review Findings

Overall confidence: 0.94

Full review comments:

- **[P1] Use imsg's include_reactions subscription flag:** `extensions/imessage/src/monitor/monitor-provider.ts:587`
  - body: Current imsg reads `include_reactions` from `watch.subscribe`, not `reactions`. With this payload, enabling `includeReactions` still subscribes in normal message-only mode, so tapbacks will not be emitted.
  - confidence: 0.95
- **[P1] Parse reactions from message notifications:** `extensions/imessage/src/monitor/monitor-provider.ts:566`
  - body: Current imsg emits reaction events as `method: "message"` with reaction metadata on the message payload. This separate `method === "reaction"` branch and `raw.reaction` parser will not run against that contract.
  - confidence: 0.95
- **[P2] Define group reaction mention behavior:** `extensions/imessage/src/monitor/monitor-provider.ts:512`
  - body: The synthetic group reaction body is `[emoji reaction]`, so the existing iMessage mention gate drops it as `no mention` in default mention-gated groups. The PR needs an explicit policy for whether opt-in group tapbacks should bypass or satisfy mention gating.
  - confidence: 0.9
- **[P3] Document the public includeReactions option:** `src/config/types.imessage.ts:57`
  - body: This adds a user-facing `channels.imessage.includeReactions` option without docs, changelog, or generated config metadata updates, so users and schema/UI consumers will not discover the new setting and config contract checks are likely to drift.
  - confidence: 0.82
