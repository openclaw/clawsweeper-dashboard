## Summary

Keep open. The PR is a plausible feature direction, but it is not mergeable against current iMessage behavior because it uses the wrong imsg RPC reaction contract, leaves group reaction routing policy unresolved, and still lacks real behavior proof.

## What This Changes

The PR adds an opt-in iMessage `includeReactions` config option, a separate tapback reaction parser/handler, and synthetic inbound iMessage payloads for reactions.

## Best Possible Solution

Define the iMessage tapback contract first, then refresh or replace this branch so it follows imsg's `include_reactions` message metadata and the chosen DM/group routing policy.

## Reproduction Assessment

Yes for the PR review findings. Source inspection plus upstream imsg source show the branch sends the wrong subscription flag and waits for a notification method current imsg does not emit; live iMessage behavior proof is still absent.

## Solution Assessment

No. Reusing the inbound pipeline is directionally useful, but this branch is not the best mergeable solution until it matches imsg's current RPC shape and maintainers choose the reaction forwarding/filtering policy.
