## Summary

Keep open. The contributor addressed earlier guard/target-text feedback, but the branch still does not match the current imsg RPC reaction contract, lacks real behavior proof, and is paired with an open same-author feature issue whose policy decision remains unresolved.

## What This Changes

The PR adds an opt-in iMessage `includeReactions` config path, parses a separate reaction RPC payload, and synthesizes tapbacks into inbound `IMessagePayload` text.

## Best Possible Solution

Refresh or replace the branch with an owner-local iMessage implementation that uses imsg message reaction fields and explicitly defines whether tapbacks dispatch, filter, or become structured events under DM/group policy.

## Reproduction Assessment

Yes for the review findings: source inspection plus current imsg RPC docs show the branch sends the wrong subscription flag and waits for a notification method imsg does not emit. Not applicable as a bug reproduction for the feature request itself.

## Solution Assessment

No. Reusing the inbound pipeline is directionally useful, but this branch is not mergeable until it follows `include_reactions`/message metadata and maintainers choose the reaction forwarding policy.
