## Summary

Keep open. Current main still lacks legacy imsg tapback forwarding, and the paired feature issue remains open. The PR is useful source work, but the branch is currently stale/mergeable=false and the implementation does not match the current imsg RPC reaction contract; group reaction mention semantics also need to be settled before merge.

## What This Changes

The PR adds an opt-in iMessage includeReactions config/schema/type field, subscribes to reaction events, parses reaction notifications, converts added tapbacks into synthetic IMessagePayload entries, and adds parser tests.

## Best Possible Solution

Refresh the work against current main and current imsg, parse reaction metadata from message notifications using the include_reactions subscription flag, keep reactions inside the existing access-control/routing pipeline, add explicit tests for DM and group-policy behavior, and document the new config only after the group wake-up semantics are intentional.

## Reproduction Assessment

Yes. A code-level reproduction is enough: with includeReactions enabled, the PR sends the wrong imsg subscribe parameter and waits for a notification method the current CLI does not emit, so handleReaction is not reached; if that is corrected, the current mention gate still drops synthetic group reaction bodies that contain no mention.

## Solution Assessment

No. Reusing the existing iMessage inbound pipeline is a good direction, but this branch is not the best mergeable solution until it follows the current imsg RPC contract and handles group reaction mention policy deliberately.

- **[P1] Use imsg's include_reactions subscription flag:** `extensions/imessage/src/monitor/monitor-provider.ts:587`
  - body: The current imsg RPC handler reads `include_reactions`, not `reactions`, when enabling reaction events. With this payload, the watcher stays in normal message-only mode, so `includeReactions: true` will not cause tapbacks to be emitted.
  - confidence: 0.95
- **[P1] Parse reactions from message notifications:** `extensions/imessage/src/monitor/monitor-provider.ts:566`
  - body: Current imsg watch still sends `method: "message"`; reaction events are represented as message payload metadata (`is_reaction`, `reaction_type`, `is_reaction_add`, `reacted_to_guid`). This `method === "reaction"` branch never runs against that CLI contract, so inbound tapbacks are dropped.
  - confidence: 0.95
- **[P2] Handle group reaction mention behavior:** `extensions/imessage/src/monitor/monitor-provider.ts:512`
  - body: After removing target text, every synthetic group reaction has body `[emoji reaction]`. The existing iMessage inbound decision checks mentions against that text and skips group traffic when `requireMention` is true, so opt-in reactions from allowed groups do not reach the agent under normal mention-gated setups.
  - confidence: 0.9
