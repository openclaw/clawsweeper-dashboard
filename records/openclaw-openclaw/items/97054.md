## What This Changes

The PR changes session cost usage scanning to estimate known-priced zero API cost totals, bumps the usage-cost cache semantic version, and adds a DeepSeek V4 fixture test.

## Best Possible Solution

Normalize the parsed entry once in `scanTranscriptFile` so known-priced zero totals use the estimated value consistently across direct summaries, durable cache entries, session summaries, and model/day breakdowns.

## Reproduction Assessment

Yes for a source-level reproduction: a nonzero-token transcript with known-priced DeepSeek V4 and `usage.cost.total: 0` leaves `costBreakdown.total` at zero, and the summary paths prefer that breakdown over the new estimate. I did not run a live authenticated DeepSeek API call in this read-only review.

## Solution Assessment

No. `scanTranscriptFile` is the right owner boundary, but this PR must also clear or rebuild the stale breakdown so every direct, cache, and session consumer sees the same estimated cost.
