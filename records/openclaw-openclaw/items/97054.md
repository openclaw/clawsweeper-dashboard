The PR changes session cost usage scanning to estimate known-priced zero API cost totals, bumps the usage cost cache version, and adds a DeepSeek V4 regression test.

## Best Possible Solution

Normalize the parsed entry once in `scanTranscriptFile` so known-priced zero totals use the estimated value consistently across direct summary, durable cache, session summary, and model/day breakdowns.

## Reproduction Assessment

Yes for a source-level reproduction: a nonzero-token transcript with `usage.cost.total: 0` creates `costBreakdown.total = 0`, and the PR still leaves downstream aggregators that prefer that breakdown over the new estimate. I did not run a live DeepSeek API call in this read-only review.

## Solution Assessment

No. `scanTranscriptFile` is the right owner boundary, but the PR must also clear or rebuild the stale breakdown so every summary/cache/session consumer sees the same estimated cost.
