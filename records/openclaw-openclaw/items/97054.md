## Summary

Keep open: the PR targets the right DeepSeek V4 spend bug, but the patch is not merge-ready because the zero `costBreakdown` still wins during aggregation and existing durable cache entries would remain fresh across upgrade. The PR body also only provides fixture/test evidence, not real after-fix behavior proof.

## What This Changes

The branch changes session cost scanning so recorded zero cost totals with known pricing are estimated from token counts, and adds a DeepSeek V4 regression test.

## Best Possible Solution

Treat known-priced zero totals as missing by clearing the parsed breakdown and rebuilding old cache entries, with regression coverage for direct and cached Control UI summary paths.

## Reproduction Assessment

Yes. Source inspection shows a transcript row with nonzero usage, known DeepSeek pricing, and `cost.total: 0` still has `costBreakdown.total === 0`, and the aggregators prefer that breakdown over the newly estimated `costTotal`; I did not run tests in this read-only checkout.

## Solution Assessment

No. The scanner is the right owner surface, but this implementation is incomplete because it leaves the stale breakdown intact and does not invalidate durable cache entries created under the old semantics.
