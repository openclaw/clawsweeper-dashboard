Keep open for maintainer review: the latest head addresses the prior non-OK body-cap gap and the SDK fetch-hook contract checks out, but this changes existing Azure Responses transport behavior and the current lint/type jobs are failing without published logs yet.

## What This Changes

The PR injects `buildGuardedModelFetch` with the resolved Azure base URL into both Azure OpenAI Responses SDK client paths, adds an Azure-scoped non-OK body cap wrapper, and adds provider transport tests.

## Best Possible Solution

Land the Azure guarded-fetch wiring only after maintainers accept the compatibility and availability rollout risk, required checks are green, and provider-path regression coverage is tightened if they want stronger future protection.

## Reproduction Assessment

Yes, source-reproducible: current main creates Azure Responses SDK clients without a custom fetch, while OpenAI v6.39.1 accepts the fetch hook and buffers non-OK error bodies with `response.text()`. I did not run a live OOM scenario in this read-only review.

## Solution Assessment

Yes, with caveats: injecting the SDK fetch hook after resolving the Azure base URL is the right owner-boundary fix, and the Azure-scoped non-OK wrapper limits cross-provider blast radius. A shared guard change would be broader, while the remaining concern is rollout compatibility and tighter provider-level regression coverage.
