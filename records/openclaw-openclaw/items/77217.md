## Summary

Keep open; current main still lacks base bottom safe-area padding for the sticky Control UI chat compose, and this PR is a narrow useful fix for the linked iOS standalone PWA bug. I found one small CSS regression to address before landing: the new rule replaces the existing 4px bottom padding with 0px on non-inset viewports instead of adding the safe-area inset on top of it.

## What This Changes

The PR adds bottom safe-area padding to the Control UI chat compose CSS and an Unreleased changelog entry for the iOS standalone PWA toolbar clipping fix.

## Best Possible Solution

Land a version that adds bottom safe-area space to base `.shell--chat .chat-compose` while preserving the existing bottom padding, then let the linked issue close after merge.

## Reproduction Assessment

Yes, by source inspection and the linked issue steps: current main has `viewport-fit=cover`, `display: standalone`, and a sticky non-focus compose at `bottom: 0` without bottom safe-area padding. I did not run a live iPhone PWA session.

## Solution Assessment

Mostly yes. The CSS target is the narrow maintainable surface and matches the existing focus-mode approach, but the padding should be `calc(4px + env(safe-area-inset-bottom, 0px))` or equivalent so non-inset layouts keep their current bottom padding.
