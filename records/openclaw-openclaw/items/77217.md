## Summary

This PR addresses a valid current Control UI iOS PWA bug, but it should not merge as written because the proposed CSS drops the existing non-inset bottom padding and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds bottom safe-area padding to the Control UI sticky chat compose and adds a changelog fix entry for #76505.

## Best Possible Solution

Keep the selector but add the safe-area inset on top of the existing 4px bottom padding, retain the changelog entry, and merge only after real iOS PWA proof or equivalent maintainer validation.

## Reproduction Assessment

Yes, by source inspection: current main has `viewport-fit=cover`, a standalone manifest, and a sticky non-focus chat compose pinned to `bottom: 0` without bottom safe-area padding. I did not run a live iOS PWA session.

## Solution Assessment

No, not as written: the PR targets the right base compose selector, but `padding-bottom: env(...)` overrides the existing 4px baseline bottom padding. The safer fix is `calc(4px + env(safe-area-inset-bottom, 0px))` or the equivalent existing safe-area token.
