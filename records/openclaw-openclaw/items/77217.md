## Summary

Keep open: the PR targets a valid Control UI iOS PWA safe-area bug, but the current CSS value would remove the existing 4px bottom padding in non-inset browsers because the new selector is more specific than the later compose padding shorthand.

## What This Changes

Adds bottom safe-area padding to the sticky Control UI chat compose and an Unreleased changelog fix entry for #76505.

## Best Possible Solution

Preserve the existing 4px compose bottom padding while adding the safe-area inset, keep the changelog credit, and merge after targeted CSS validation and normal UI review.

## Reproduction Assessment

Yes, source-backed but not live-device reproduced: current main has the standalone PWA surface and a non-focus sticky compose without bottom safe-area padding, while the focus-mode compose already accounts for the inset.

## Solution Assessment

No, not as written: the selector is the right place, but the value should add the inset to the existing 4px bottom padding instead of replacing it with `env(...)`.

## Review Findings

Overall correctness: patch is incorrect
