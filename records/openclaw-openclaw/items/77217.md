## Summary

Keep open: the PR targets a valid iOS PWA safe-area bug, but the current CSS value drops the existing non-PWA compose bottom padding because the new more-specific rule overrides the later base compose padding.

## What This Changes

The PR adds bottom safe-area padding to the sticky Control UI chat compose and an Unreleased changelog fix entry for #76505.

## Best Possible Solution

Add the safe-area inset on top of the existing 4px compose bottom padding, then merge after normal UI review and targeted style validation.

## Reproduction Assessment

Yes, source-reproducible: current main has `viewport-fit=cover`, a standalone PWA manifest, and a sticky base compose pinned to the bottom without bottom safe-area padding. I did not run a live iOS PWA session.

## Solution Assessment

No, not as written: the selector is right, but `padding-bottom: env(...)` drops existing baseline spacing. The safer fix is `calc(4px + env(safe-area-inset-bottom, 0px))` or an equivalent value that preserves the current 4px padding.
