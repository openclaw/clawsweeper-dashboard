## Summary

Keep open: this targets a real Control UI iOS PWA safe-area bug, but the current diff still drops the normal 4px compose bottom padding in zero-inset browsers and the external PR lacks after-fix real iOS PWA proof.

## What This Changes

Adds a `.shell--chat .chat-compose` bottom safe-area padding declaration and an Unreleased changelog fix entry for #76505.

## Best Possible Solution

Keep the safe-area fix on the base sticky compose, but add the inset on top of the existing bottom spacing and merge only after real iOS PWA proof or maintainer device validation.

## Reproduction Assessment

Yes, by source inspection: current main has `viewport-fit=cover`, a standalone manifest, and a sticky non-focus compose at `bottom: 0` without bottom safe-area padding. I did not run a live iOS PWA session.

## Solution Assessment

No, not as written: the PR targets the right selector, but `padding-bottom: env(...)` should preserve the existing 4px baseline, for example `calc(4px + env(safe-area-inset-bottom, 0px))` or the existing safe-area token.
