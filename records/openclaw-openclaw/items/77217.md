## Summary

Keep open: the PR targets a real current-main iOS PWA safe-area bug, but the proposed CSS value would remove existing bottom spacing in non-inset browsers because it overrides the later base compose padding.

## What This Changes

This PR adds bottom safe-area padding to `.shell--chat .chat-compose` in `ui/src/styles/components.css` and adds an Unreleased changelog fix entry for #76505.

## Best Possible Solution

Adjust the new base compose rule to add the safe-area inset on top of the existing 4px bottom padding, then merge after normal UI review and targeted validation.

## Reproduction Assessment

Yes, by source inspection: current main has `viewport-fit=cover`, a standalone PWA manifest, and a sticky non-focus chat compose without bottom safe-area padding. I did not run a live iOS PWA session.

## Solution Assessment

No, not as written: the selector is right, but `padding-bottom: env(...)` drops the existing non-PWA 4px bottom padding. The safer fix is `calc(4px + env(safe-area-inset-bottom, 0px))` or an equivalent value that preserves baseline spacing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

In regular Safari and desktop browsers where the safe-area inset resolves to 0, the compose loses its existing 4px bottom padding, so the non-PWA layout is not unchanged; add the inset to the existing 4px instead.
