Full review comments:

- **[P1] Keep raw probe headers out of the public client API:** `src/gateway/client.ts:79`
  - body: `GatewayClient` is exported through the plugin SDK gateway runtime barrel on current main, so adding a generic `headers` constructor option exposes raw handshake header injection beyond this self-probe. Because trusted-proxy identity is header-based, keep this internal to the probe path or get explicit API/security approval with docs and boundary tests before merging.
  - confidence: 0.78
- **[P2] Do not treat synthetic headers as trusted-proxy proof:** `src/cli/gateway-cli/run.ts:562`
  - body: This direct probe only injects trusted-proxy headers, but current auth still checks the socket source and rejects loopback by default unless `allowLoopback` is enabled. For normal non-loopback reverse-proxy deployments, the probe will not travel through the actual trusted proxy and can still report a healthy gateway as unhealthy, leaving the EADDRINUSE restart-loop behavior unresolved.
  - confidence: 0.9
- **[P2] Bracket IPv6 custom bind hosts before probing:** `src/cli/gateway-cli/run.ts:169`
  - body: `resolveGatewayBindUrl()` formats a custom host directly into `ws://<host>:<port>`, so an IPv6 host such as `::1` becomes an invalid URL and is filtered before probing. A gateway bound only to that IPv6 address can still be treated as unhealthy and exit non-zero on port conflict.
  - confidence: 0.86
