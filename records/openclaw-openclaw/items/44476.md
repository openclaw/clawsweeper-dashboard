## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve systemd exit-78 semantics:** `src/cli/gateway-cli/run.ts:568`
  - body: Current main documents and tests healthy systemd lock conflicts as exit 78 so `RestartPreventExitStatus=78` stops `Restart=always` loops. This successful-probe path exits 0 for every supervisor, so a rebased version needs to keep the systemd-specific exit code or carry an explicit maintainer-approved policy change.
  - confidence: 0.87
- **[P1] Probe trusted-proxy listeners through a trusted source:** `src/cli/gateway-cli/run.ts:562`
  - body: In trusted-proxy mode this self-probe only injects identity headers, but the server also requires the socket source to match `gateway.trustedProxies`. A healthy gateway behind a normal reverse proxy can fail this probe and the catch path exits 1, reintroducing the restart loop this PR is trying to prevent.
  - confidence: 0.86
- **[P1] Keep plaintext LAN/custom probe targets reachable:** `src/cli/gateway-cli/run.ts:155-156`
  - body: Filtering configured non-loopback `ws://` bind URLs on this PR head means a healthy gateway bound only to `gateway.bind=lan` or `customBindHost` may not be probed unless the break-glass env var is set; the loopback fallback will not reach that listener, so startup can still exit 1 on EADDRINUSE.
  - confidence: 0.8
- **[P2] Bracket IPv6 bind hosts before probing:** `src/cli/gateway-cli/run.ts:168-170`
  - body: `resolveGatewayBindUrl` emits custom hosts directly, so `customBindHost: "::1"` becomes an invalid `ws://::1:<port>` candidate and is filtered out before the probe runs. For an IPv6-only loopback bind, the fallback to `127.0.0.1` cannot confirm the healthy gateway.
  - confidence: 0.76
