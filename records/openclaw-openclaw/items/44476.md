- **[P1] Route trusted-proxy probes through the trusted boundary:** `src/cli/gateway-cli/run.ts:562`
  - body: The patch sends fabricated trusted-proxy headers from the local CLI probe. Current trusted-proxy auth still requires the request source to be a configured trusted proxy and rejects loopback sources unless `allowLoopback` is explicitly enabled, while the docs tell internal Gateway clients to use password auth instead. That means common non-loopback trusted-proxy deployments still fail the EADDRINUSE health check, and loopback-enabled deployments blur the proxy identity boundary.
  - confidence: 0.88

  - body: For `gateway.bind=custom` with an IPv6 host, `resolveGatewayBindUrl` emits an unbracketed URL such as `ws://::1:18789`, and this code passes it straight to the probe URL filter. The candidate is invalid and gets dropped, so an existing healthy IPv6-bound gateway can still be treated as unhealthy and the restart-loop behavior remains.
  - confidence: 0.84
