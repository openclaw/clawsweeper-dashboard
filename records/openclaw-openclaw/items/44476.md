## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Verify the listener before sending gateway credentials:** `src/cli/gateway-cli/run.ts:193-200`
  - body: This probe sends `auth` and trusted-proxy headers through `probeGateway()` to whatever process owns the port. `GatewayClient` puts token/password auth into the connect frame after a peer sends a challenge, so a fake local listener can harvest the gateway secret before this code decides it is not OpenClaw. Use a no-secret health/identity check first, or require a TLS-pinned peer before sending credentials.
  - confidence: 0.84
- **[P2] Probe trusted-proxy gateways through a trusted source:** `src/cli/gateway-cli/run.ts:562`
  - body: Passing synthetic trusted-proxy headers is not enough for this new EADDRINUSE probe: `authorizeTrustedProxy` also rejects requests whose socket address is not in `gateway.trustedProxies`. A direct `probeGateway()` from the starter process will still report deployments whose trusted proxy is a different host as unhealthy, so supervised restarts can keep exiting non-zero in the auth mode the PR says it handles.
  - confidence: 0.82
- **[P2] Normalize IPv6 bind hosts before building probe URLs:** `src/cli/gateway-cli/run.ts:169`
  - body: `resolveGatewayBindUrl()` can return an unbracketed IPv6 host such as `ws://::1:18789`, and this new probe path passes that string straight into the URL filter. That malformed URL gets dropped, leaving only the IPv4 loopback fallback, so an already-running gateway bound to an IPv6 custom host can still be treated as unhealthy on EADDRINUSE.
  - confidence: 0.8
