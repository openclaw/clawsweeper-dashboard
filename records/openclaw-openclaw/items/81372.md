## Summary

Keep open: the direction is plausible, but the PR currently introduces CI-confirmed regressions in reply diagnostics and status usage display, leaves a new reply-turn state open on one handled-hook path, and lacks required real behavior proof.

## What This Changes

The PR changes auto-reply and subagent announcement paths to suppress acknowledgement-only replies, track reply-turn state, add stale-turn recovery/status output, and add related prompt/test coverage.

## Best Possible Solution

Keep the acknowledgement-suppression goal, but first repair the lifecycle/status/diagnostic regressions, restore green targeted CI, and add redacted real Telegram or forum-channel proof showing the changed visible behavior.

## Reproduction Assessment

Yes for the PR regressions: CI on the PR head reproduces failures in dispatch diagnostics and status usage, and source inspection shows the unclosed reply_dispatch handled path. Real Telegram/forum behavior is not reproduced because the PR has no real behavior proof.

## Solution Assessment

No: suppressing low-value acknowledgements is a reasonable direction, but this patch is not the best mergeable form until it closes every new reply-turn path, preserves active-provider usage accounting, avoids duplicate diagnostic progress, and proves the visible behavior in a real channel.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Close reply-turn state for handled reply_dispatch hooks:** `src/auto-reply/reply/dispatch-from-config.ts:997-1001`
  - body: The PR marks the session `replyTurnState` as `running` before `reply_dispatch`, but the handled `replyDispatchResult` path returns without `markReplyTurnClosed`. Any plugin hook that handles a turn can leave the session permanently running, and the new stale-turn recovery can later send a false interruption message.
  - confidence: 0.89
- **[P2] Restore active runtime usage lookup in status:** `src/status/status-text.ts:280-292`
  - body: This now resolves usage from the selected provider and `selectedModelAuth` only. For OpenAI models running through the Codex harness, the active runtime provider is `openai-codex`, so the status command stops loading the usage summary; the current CI failure confirms the expected `Usage: 5h 91% left` line disappears.
  - confidence: 0.93
- **[P2] Avoid double-marking routed progress payloads:** `src/auto-reply/reply/dispatch-from-config.ts:666`
  - body: `onToolResult` already calls `markProgress()` before routing a visible tool payload, and this helper now marks the same payload again when it routes to the originating channel. The PR head fails the existing diagnostic test because progress is recorded three times instead of once.
  - confidence: 0.86
