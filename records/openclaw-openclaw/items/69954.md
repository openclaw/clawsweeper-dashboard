## Summary

Keep open. Current main still has the cleanup bug, so this PR remains useful, but it needs changes before merge because the repair branch rewrites `entry.sessionFile` without reporting a mutation, and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR changes `sessions cleanup --fix-missing` to preserve and repair sessions whose stored `sessionFile` path is stale when the canonical `<sessionId>.jsonl` transcript exists, with focused cleanup tests.

## Best Possible Solution

Port the canonical fallback into current `src/config/sessions/cleanup-service.ts`, return both removed and repaired counts, include repairs in dry-run and applied `wouldMutate` accounting, and cover stale-path repair plus true-missing pruning in focused tests.

## Reproduction Assessment

Yes, at source level. Current main resolves one transcript path from `entry.sessionFile` and deletes when it is missing, so a store entry with a stale stored path and an existing canonical `<sessionId>.jsonl` follows the bad branch.

## Solution Assessment

No, not as-is. The canonical fallback-and-repair direction is the right narrow fix, but the implementation must report repaired paths as mutations and be ported to the current cleanup-service owner path.
