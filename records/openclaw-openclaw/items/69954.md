## Summary

Keep this PR open. Current main still has the unsafe cleanup behavior linked to #50248, and the PR is aimed at the right bug, but the patch is not merge-ready because repair-only runs can rewrite `sessionFile` metadata while JSON summaries still report `wouldMutate: false`.

## What This Changes

The PR changes `sessions cleanup --fix-missing` to check the canonical `<sessionId>.jsonl` transcript before pruning a stale stored `sessionFile` path, and adds dry-run/applied regression tests for that repair path.

## Best Possible Solution

Land a focused cleanup fix that preserves valid sessions when the canonical transcript exists, records repaired `sessionFile` metadata as a mutation in both dry-run and applied summaries, and keeps true missing-transcript pruning unchanged.

## Reproduction Assessment

Yes. The stale metadata case is deterministic: create a store entry whose `entry.sessionFile` points to a missing file while `<sessionId>.jsonl` exists beside the store, then run `sessions cleanup --fix-missing`; current main follows only the stored-path check. The PR finding is also reproducible from the diff because the repair branch sets `entry.sessionFile` while the returned missing count remains zero.

## Solution Assessment

No, not as-is. The canonical fallback is the right narrow direction, but the implementation must also count or otherwise report repair-only mutations so JSON output remains accurate for operators and automation.

## Review Findings

Overall correctness: patch is incorrect
