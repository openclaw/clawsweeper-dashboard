## Summary

Keep open. The linked cleanup bug is real and still present on current main, but this PR is not mergeable as-is because the active cleanup logic has moved to the cleanup service, repair-only runs are not counted as mutations, the new fallback can throw for invalid legacy session IDs, and the external PR still lacks inspectable real behavior proof.

## What This Changes

The PR adds a canonical transcript fallback for `sessions cleanup --fix-missing`, repairs stale `sessionFile` metadata when that canonical transcript exists, and adds command-level tests.

## Best Possible Solution

Port the fallback-and-repair behavior into `src/config/sessions/cleanup-service.ts`, count repaired paths in preview and applied summaries, guard invalid fallback IDs, and add focused service-level regression coverage.

## Reproduction Assessment

Yes, at source level. A store entry with a valid `sessionId`, a missing stale `sessionFile`, and an existing canonical `<sessionId>.jsonl` still follows current main's single resolved-path cleanup check and would be pruned; I did not run a mutating fixture because this review is read-only.

## Solution Assessment

No, not as-is. The fallback-and-repair direction is the right narrow fix, but this branch needs current-main porting, repaired-path mutation accounting, invalid-ID guarding, and real behavior proof before merge.
