## What This Changes

The PR changes `sessions cleanup --fix-missing` to check the canonical `<sessionId>.jsonl` transcript before pruning and to repair stale `sessionFile` metadata when that canonical transcript exists.

## Best Possible Solution

Port the fallback-and-repair behavior into `src/config/sessions/cleanup-service.ts`, track repaired-path mutations in dry-run and applied summaries, keep true-missing pruning unchanged, and add focused regression tests.

## Reproduction Assessment

Yes, at source level. A store entry with `sessionId=<id>`, a stale missing `sessionFile`, and an existing canonical `<id>.jsonl` still follows the single resolved-path cleanup check on current main and would be pruned.

## Solution Assessment

No, not as-is. The fallback-and-repair direction is the narrow maintainable fix, but this branch needs current-main porting plus repaired-path mutation accounting before it is safe to merge.
