## Summary

Keep open: current main still has the stale-sessionFile pruning bug, and this PR is the right narrow direction, but it is not mergeable as-is because the active cleanup code has moved to the service layer, repair-only runs are not counted as mutations, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a canonical transcript fallback for `sessions cleanup --fix-missing`, repairs stale `sessionFile` metadata when that transcript exists, and adds command-level tests.

## Best Possible Solution

Port the fallback-and-repair behavior into `src/config/sessions/cleanup-service.ts`, count repaired paths in preview and applied summaries, preserve true-missing pruning, and add focused service-level regression coverage.

## Reproduction Assessment

Yes, at source level. A store entry with a valid `sessionId`, a missing stale `sessionFile`, and an existing canonical `<sessionId>.jsonl` still follows the single resolved-path cleanup check on current main and would be pruned.

## Solution Assessment

No, not as-is. The fallback-and-repair direction is the right narrow fix, but this branch needs current-main porting, repaired-path mutation accounting, and real behavior proof before it is safe to merge.
