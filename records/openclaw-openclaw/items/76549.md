## Summary

Keep open. The fake-IP model-provider failure is valid and this PR touches the right transport seam, but the proposed `allowedHostnames` policy bypasses all resolved-IP safety checks for the configured provider hostname, so it needs maintainer security/product review before merge.

## What This Changes

This PR adds a provider `baseUrl` hostname-derived SSRF policy to `buildGuardedModelFetch` and merges it with the existing `allowPrivateNetwork` policy when calling `fetchWithSsrFGuard`.

## Best Possible Solution

Support fake-IP model-provider calls with a hostname-scoped exemption limited to RFC2544 and IPv6 ULA answers, while preserving RFC1918, loopback, metadata, and other special-use blocks unless private networking is explicitly enabled, and add focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current model transport reaches `fetchWithSsrFGuard` without fake-IP SSRF flags, while current SSRF code and tests block 198.18.0.0/15 and fc00::/7 unless explicitly exempted.

## Solution Assessment

No. The PR is aimed at the correct provider transport surface, but `allowedHostnames` is not the narrowest maintainable fix because it skips all resolved-IP checks for that host instead of only the fake-IP ranges.
