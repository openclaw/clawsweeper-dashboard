## Summary

Keep open. The fake-IP provider transport failure is source-reproducible and the PR touches the right model fetch seam, but the patch should not merge as-is because `allowedHostnames` broadens the SSRF trust boundary beyond the reported RFC2544/IPv6 ULA case and the user-facing fix still needs changelog and regression coverage.

## What This Changes

The PR modifies `buildGuardedModelFetch` to derive an `allowedHostnames` SSRF policy from `model.baseUrl` and merge it with the existing `allowPrivateNetwork` provider request policy.

## Best Possible Solution

Add a hostname-scoped fake-IP-only exemption for model provider transport that permits RFC2544 and IPv6 ULA DNS answers without bypassing other private or metadata ranges, then add regression coverage and a changelog entry.

## Reproduction Assessment

Yes, from source inspection. Current main sends model provider requests through the guarded fetch without fake-IP SSRF flags, the model request schema exposes no fake-IP escape hatch, and SSRF tests show RFC2544/IPv6 ULA answers are blocked unless explicitly exempted.

## Solution Assessment

No. The PR changes the right model transport surface, but `allowedHostnames` is broader than the reported fake-IP failure because the SSRF guard treats it as a full resolved-IP private/special-use bypass for that hostname.

## Review Findings
