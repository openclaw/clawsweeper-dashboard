## Summary

Keep open. The PR targets a plausible fake-IP model-transport bug, but the current approach widens the SSRF policy from fake-IP ranges to a full resolved-IP bypass for every configured provider hostname, so it needs maintainer security review before merge.

## What This Changes

The PR adds a model `baseUrl` hostname-derived SSRF policy to `buildGuardedModelFetch` and merges it with the existing `allowPrivateNetwork` policy before calling `fetchWithSsrFGuard`.

## Best Possible Solution

Preserve provider calls through fake-IP DNS proxies with a hostname-scoped exemption for RFC2544 and IPv6 ULA answers, while keeping ordinary RFC1918, loopback, metadata, and other special-use DNS answers blocked unless `allowPrivateNetwork` is explicit.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current model transport calls the guarded fetch without fake-IP SSRF policy, while the shared SSRF tests show 198.18.0.0/15 and fc00::/7 are blocked without explicit flags. I did not run a live fake-IP proxy test in this read-only review.

## Solution Assessment

No. The PR touches the right model transport surface, but using `allowedHostnames` is broader than the fake-IP problem because current SSRF code treats that as a complete private/special-use resolved-IP bypass for the configured host.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve resolved-IP checks for provider hostnames:** `src/agents/provider-transport-fetch.ts:316-322`
  - body: The new `allowedHostnames` policy makes `resolvePinnedHostnameWithPolicy` skip all private/special-use DNS answer checks for `model.baseUrl`'s host. That fixes fake-IP ranges, but it also allows that same provider hostname to resolve to RFC1918, loopback, metadata, or other blocked addresses. Please use a narrower fake-IP exemption that keeps ordinary private/metadata DNS answers blocked unless `allowPrivateNetwork` is explicit.
