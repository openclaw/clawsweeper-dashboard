## Summary

Keep open. The PR targets a valid model-provider fake-IP failure and correctly points at existing plugin hostname-allowlist precedent, but the implementation uses `allowedHostnames`, which skips all resolved-IP private/special-use checks for the configured provider host and widens the SSRF boundary beyond the narrower fake-IP exemption already present in the guard.

## What This Changes

The PR imports `ssrfPolicyFromHttpBaseUrlAllowedHostname`, derives a policy from `model.baseUrl`, and merges it with `allowPrivateNetwork` when `buildGuardedModelFetch` calls the SSRF-guarded fetch.

## Best Possible Solution

Add a hostname-scoped fake-IP-only model transport policy that permits RFC2544 and IPv6 ULA answers for the configured provider host while preserving loopback, RFC1918, metadata, and other reserved-address blocks, then add regression coverage and a changelog entry.

## Reproduction Assessment

Yes from source inspection. Current main routes model provider calls through the guarded fetch with only `allowPrivateNetwork`, while the SSRF guard and tests show RFC2544 and IPv6 ULA fake-IP DNS answers are blocked unless the narrower fake-IP flags are explicitly present.

## Solution Assessment

No. The PR changes the right model transport surface, but `allowedHostnames` is broader than the reported fake-IP failure because it disables resolved-IP private/special-use checks for the configured provider hostname instead of only exempting RFC2544 and ULA answers.
