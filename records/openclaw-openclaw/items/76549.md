## Summary

Keep open. The fake-IP provider transport failure is source-reproducible and this PR targets the right shared model fetch path, but the current diff uses `allowedHostnames`, which bypasses all resolved-IP private/special-use checks for the configured provider host.

## What This Changes

The PR threads the configured model `baseUrl` hostname into the provider transport SSRF policy so guarded model API requests on fake-IP DNS proxies can bypass special-use IP blocking.

## Best Possible Solution

Add a provider-transport fake-IP-only exemption that covers RFC2544 and IPv6 ULA answers while preserving the existing private/metadata blocks, then add regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Current main sends model provider requests through `buildGuardedModelFetch` without fake-IP SSRF flags, so a fake-IP DNS answer such as `198.18.0.1` or `fc00::1` will be rejected by the SSRF resolver before the provider call is sent.

## Solution Assessment

No. The PR changes the right transport surface, but `allowedHostnames` is broader than the reported fake-IP failure because it disables all resolved-IP private/special-use checks for that provider hostname.
