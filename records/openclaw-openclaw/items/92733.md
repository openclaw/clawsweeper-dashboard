## Summary

Keep open: the managed-skill Workshop direction is useful, but this patch is not merge-ready because it resolves managed skills through a different root than the existing global skills CLI/status path and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR expands Skill Workshop update/revise/apply support from workspace skills to shared OpenClaw-managed skills, and updates the agent prompt, CLI copy, docs, and tests for that wider writable surface.

## Best Possible Solution

Unify Skill Workshop managed-skill updates with the existing global skills root used by install/list/status, add a regression for custom config-path setups, then provide real setup proof for proposing and applying a managed skill update.

## Reproduction Assessment

Yes, source-level: with `OPENCLAW_CONFIG_PATH` set and no `OPENCLAW_STATE_DIR`, existing global skills resolve through `CONFIG_DIR/skills` while the PR's Workshop path resolves through `resolveStateDir()/skills`. I did not run the command path because this review is read-only.

## Solution Assessment

No as submitted. The feature should reuse the existing managed-skill root contract rather than introducing a second root, and it still needs real behavior proof before merge.

Full review comments:

- **[P1] Use the existing managed skills root:** `src/skills/workshop/service.ts:320`
  - body: This adds a second definition of the managed skills directory. Existing global skill commands target `CONFIG_DIR`, and status loading defaults managed skills to `CONFIG_DIR/skills`; with only `OPENCLAW_CONFIG_PATH` set, `resolveStateDir()/skills` is a different directory. That means Workshop updates can fail to find the managed skill the user installed with `--global`, or update a same-named skill under the wrong root. Please reuse/thread the existing `CONFIG_DIR` managed root contract and cover the custom config-path case.
