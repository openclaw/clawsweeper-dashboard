## Summary

Keep open: the managed-skill direction is useful, but this patch is not merge-ready because it resolves managed skills through a different root than the existing loader/watcher for `OPENCLAW_CONFIG_PATH` setups, leaves adjacent prompt-contract tests stale, and has no after-fix real behavior proof.

## What This Changes

The PR extends Skill Workshop update/revise/apply flows to `openclaw-managed` skills, changes agent/CLI/docs wording from workspace-only to writable user-owned skills, and adds service coverage for managed targets.

## Best Possible Solution

Reuse the same canonical managed skills root as discovery/loading/watching, add coverage for custom config-path setups and updated prompt semantics, then require redacted real managed update/apply proof before merge.

## Reproduction Assessment

Not applicable as a user bug, but the PR blockers are source-reproducible: current main loads managed skills from `CONFIG_DIR/skills`, while the PR writes through `resolveStateDir()/skills`, and current prompt tests still assert the old approval wording.

## Solution Assessment

No. Supporting managed updates is plausible, but the best fix must reuse the canonical managed-skill root, update the prompt contract/tests, and settle same-turn apply policy explicitly.
