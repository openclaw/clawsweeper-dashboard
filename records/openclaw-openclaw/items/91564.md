## Summary

Keep open. Two independent installations report persistent topic-scoped Telegram message loss, current main and v2026.6.5 retain the implicated guarded-lane behavior, and no merged fix or canonical report fully explains the restart-surviving, empty-spool case.

## What This Changes

Investigate and fix a Telegram forum topic-specific inbound message-loss state that persists after stuck-session recovery and gateway restarts.

## Best Possible Solution

Give isolated-ingress handlers an explicit terminal cancellation lifecycle, trace fresh-message versus reply/command dispatch before inbound logging, and release the topic lane only after late output and state mutation are impossible; validate the complete restart scenario in a real Telegram forum.

## Reproduction Assessment

Yes at the source level: current main and focused tests show that an unsettled timed-out handler keeps later same-topic updates blocked. The full-process restart persistence and reply/command asymmetry still need a controlled live reproduction.

## Solution Assessment

Unclear. Session deletion and timeout tuning do not address the proven ingress ownership boundary, while unconditional lane release would violate the existing late-handler safety contract.
