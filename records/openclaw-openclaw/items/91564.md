## Summary

Keep open. The restart-persistent forum-topic behavior and reply/command bypass are not fully explained by the related DM report, so no existing fix or duplicate safely closes this issue.

## What This Changes

Fix a Telegram forum topic-specific inbound delivery failure that persists after stuck-session recovery and gateway restarts while other topics and outbound delivery continue working.

## Best Possible Solution

Reproduce the failure on current main with isolated polling ingress, instrument the update from durable spool claim through sequentialization and pre-inbound filtering, then make one canonical recovery path release or quarantine irrecoverable topic lanes without concurrent duplicate handling; cover fresh messages, reply/command bypass, restart recovery, and live Telegram behavior.

## Reproduction Assessment

Yes at source and production-signature level: two independent reports describe the same topic-local loss, and current code contains explicit indefinite guarded-lane branches. A controlled live reproduction on current main is still required to identify the restart-persistent owner.

## Solution Assessment

No proposed fix is yet proven best. Clearing session data or the process-local guard is insufficient; the repair should follow the acknowledged update through spool ownership, sequentialization, and fresh-message filtering before selecting safe release semantics.
