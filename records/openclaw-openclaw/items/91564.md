## Summary

Keep open. The latest release adds meaningful Telegram isolated-ingress timeout recovery, but shipped source and tests still intentionally leave later same-topic updates blocked when the old handler does not settle, so the central message-loss failure is not conclusively fixed.

## What This Changes

Investigate and fix a Telegram forum-topic failure where inbound updates are acknowledged but permanently stop reaching the agent after stuck-session recovery.

## Best Possible Solution

Make every Telegram spooled handler participate in a lifecycle-owned cancellation path that reliably settles before the lane resumes, retain failed-update accounting, and prove both same-process timeout recovery and full gateway-restart recovery with a real forum topic.

## Reproduction Assessment

Yes at source/test level: current main has a focused test that wedges one topic handler and verifies the next same-topic update remains pending after timeout. The stronger full-process-restart persistence claim still needs a live v2026.6.5/current-main reproduction with ingress-queue evidence.

## Solution Assessment

No complete fix is established yet. The shipped dead-letter, reply-fence abort, and restart flow is directionally correct, but the safest solution must make the old handler settle before resuming the lane rather than either blocking forever or permitting concurrent topic handlers.
