Keep open. The cleanup idea is still useful, and Greptile plus the related fork reference support the intended direction, but the PR head is stale against current main and would regress current redirect-header security and configurable media-size contracts if rebased as shown.

The PR changes media URL downloads to add guarded settlement, response-body draining, write-stream cleanup on abort/pipeline errors, temp-file cleanup on failed URL saves, and two media-store cleanup tests.

## Best Possible Solution

Rebase or replace the PR with a narrow media-store cleanup on current main that keeps the stream/socket/temp-file cleanup ideas while preserving retainSafeHeadersForCrossOriginRedirect, maxBytes threading, and focused regression tests for both cleanup and existing redirect/limit contracts.
