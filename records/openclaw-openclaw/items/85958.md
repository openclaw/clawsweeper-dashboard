## Summary

Keep open: the Codex boundary direction is coherent and the live proof is strong, but the patch still reports a pending native Codex compaction start as `ok && compacted`, which leaks into the gateway session compaction path and records a false completed compaction. The fail-closed compatibility and availability behavior also still needs maintainer acceptance before merge.

## What This Changes

The branch removes OpenClaw-owned automatic/context-engine Codex compaction paths, starts explicit Codex compaction asynchronously via `thread/compact/start`, blocks `/compact` gateway fallback, and updates related tests, docs, and changelog.

## Best Possible Solution

Keep Codex automatic compaction native-owned, but expose explicit native-start as pending across all OpenClaw callers and land only after owners accept the fail-closed fallback removal.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the PR defect: PR head returns pending Codex native starts as `ok && compacted`, and the unchanged gateway session compact path treats that shape as completed and mutates session state. I did not run the app, but the call contracts are direct and covered by the cited lines.

## Solution Assessment

No, not yet. The ownership boundary is maintainable, but the result shape needs a shared pending-state contract or caller updates before this is the best fix.
