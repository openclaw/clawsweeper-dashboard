## Summary

Keep open. Current main still drops bundle `activation` metadata before bundle registry records are built, and this PR targets the right parser/registry path, but the branch is dirty against current main and still needs the required changelog entry before merge.

## What This Changes

This PR exports manifest activation normalization, parses `activation` from Codex/Cursor/Claude bundle manifests, stores it on bundle registry records, and adds Claude bundle coverage plus a gateway CLI test adjustment.

## Best Possible Solution

Rebase or repair this PR on current main, keep the bundle activation metadata pass-through, add the single-line changelog fix entry, and land after focused plugin/gateway validation.

## Reproduction Assessment

Yes. On current main, loading a Claude/Codex/Cursor bundle manifest with `activation.onStartup: false` loses that field because `loadBundleManifest` and `buildBundleRecord` never carry activation into the registry record.

## Solution Assessment

Yes. Exporting the existing manifest activation normalizer and threading the normalized value through bundle parsing and registry assembly is the narrowest maintainable fix; the repair should avoid changing unrelated gateway test infrastructure beyond resolving the current-main conflict.
