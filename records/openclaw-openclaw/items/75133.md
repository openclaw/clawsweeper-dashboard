## Summary

Keep open. Current main still drops bundle manifest activation metadata before registry records are built, while this PR narrowly threads the existing activation normalizer through the bundle parser and registry path; the patch has no actionable review findings, is reported mergeable, and all fetched PR-head check runs are complete without failures.

## What This Changes

The PR exports manifest activation normalization, preserves activation metadata for Codex, Cursor, and Claude bundle manifests, copies it into bundle registry records, adds Claude bundle coverage for `activation.onStartup: false`, and isolates ambient service-mode environment state in a gateway CLI test.

## Best Possible Solution

Land the narrow metadata-preservation fix after maintainer review, keeping activation as manifest/control-plane metadata for bundle records rather than adding compatibility-warning exceptions or requiring bundle authors to convert inert content bundles into native plugins.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: put `activation.onStartup: false` in a bundle manifest, then `loadBundleManifest` ignores it, `buildBundleRecord` leaves `record.activation` undefined, and inert bundle records remain eligible for the legacy implicit startup sidecar classification.

## Solution Assessment

Yes. Reusing the existing manifest activation normalizer and threading the normalized value through bundle parsing and registry assembly is the narrowest maintainable fix for the observed behavior.
