## Summary

Keep open. Current main still drops bundle `activation` metadata before registry records are built, so the PR is addressing a real bundle/status compatibility gap; the branch needs a rebase over current-main test changes plus the required changelog entry before merge.

## What This Changes

The PR exports manifest activation normalization, reads `activation` from Codex, Claude, and Cursor bundle manifests, copies it into bundle registry records, adds Claude bundle coverage, and adjusts a gateway CLI environment-isolation test.

## Best Possible Solution

Land the narrow metadata-preservation fix after rebasing onto current main, keep activation as bundle manifest/control-plane metadata on registry records, add the required changelog note, and validate the bundle registry plus installed-index/status compatibility paths.

## Reproduction Assessment

Yes. On current main, a bundle manifest with `activation.onStartup: false` is parsed into a bundle manifest object with no activation field, then `buildBundleRecord` creates a registry record with activation undefined, which can trigger the legacy implicit startup sidecar classification.

## Solution Assessment

Yes. Reusing the existing native manifest activation normalizer and threading the normalized value through bundle parsing and registry assembly is the narrowest maintainable fix; the remaining work is rebase, changelog, and validation, not a different design.
