## Summary

Keep open: the reported durable-session duplication is still not solved on current main. Current code has outward assistant-text dedupe and Telegram prompt-context recording changes, but the session append path still accepts identical assistant text as a new linked leaf entry, and the related Moonshot/Gemini issues do not own this exact pure assistant.text persistence bug.

## What This Changes

The issue requests one durable assistant text record per Gemini native Telegram turn, plus clearer outbound dedupe logging for sent versus suppressed dispatches.

## Best Possible Solution

Add idempotency at the embedded runner finalization or session append boundary so duplicate final assistant text is not persisted, with Telegram dedupe/logging left as a delivery backstop.

## Reproduction Assessment

Yes at source/report level: the reporter supplied concrete duplicate JSONL excerpts, and current main still has no durable identical assistant.text append guard; no live Gemini/Telegram repro was run.

## Solution Assessment

Yes for the primary bug: preventing duplicate final assistant persistence is the right fix direction, while strict-mode/provider-list changes should stay secondary unless the implementation proves they are required.
