## Summary

Keep open: current main and the latest release still append ordinary successful assistant text without an unkeyed same-turn/same-text persistence guard, and the related visible-duplicate, retry, and overflow trackers do not own this exact Gemini-native assistant.text double-append path.

## What This Changes

The issue requests preventing duplicate assistant.text records from being persisted during Gemini native streaming turns and improving outbound dedupe logging so suppressed duplicate sends remain diagnosable.

## Best Possible Solution

Keep this issue open until maintainers isolate the duplicate producer and land a narrow idempotency fix at the correct finalization or persistence boundary, with logging that distinguishes sent from suppressed outbound duplicates.

## Reproduction Assessment

No high-confidence live current-main reproduction was established in this read-only review. The reporter supplied concrete v2026.5.22 JSONL evidence, and current main plus v2026.6.9 still have append paths that would persist a second identical successful assistant message if one is emitted.

## Solution Assessment

Unclear. A same-parent/content storage guard is a plausible backstop, but the best fix needs maintainer choice on whether finalization, fallback, hook invocation, or transcript persistence owns the idempotency boundary.
