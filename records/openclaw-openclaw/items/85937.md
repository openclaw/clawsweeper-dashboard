## Summary

Keep open: current main has dedupe for live assistant payloads and delivery-mirror gap-fills, but I did not find a core session-persistence guard that would prevent the reported consecutive identical Gemini assistant text rows from being appended to JSONL.

## What This Changes

The issue requests that Gemini native streaming over Telegram persist exactly one final assistant text record per turn and log sent versus suppressed outbound dispatches clearly.

## Best Possible Solution

Identify the double-finalization source first, then add a narrowly scoped transcript idempotency guard for duplicate terminal assistant text and make Telegram dispatch logs distinguish sent from suppressed outcomes.

## Reproduction Assessment

No, not from this review alone. The reporter provides concrete production JSONL evidence and current source shows no matching storage guard, but I did not reproduce the live Gemini native plus Telegram path on current main.

## Solution Assessment

Unclear. A storage fingerprint guard is a plausible backstop, but the best fix should first locate whether Gemini streaming finalization, fallback, or hook handling is producing the second assistant message.
