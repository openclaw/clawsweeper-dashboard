## Summary

Keep open: current main still runs the expensive provider path during plain `openclaw memory status`, while the docs say plain status should stay fast and leave live embedding/vector checks to `--deep`.

## What This Changes

Request that plain `openclaw memory status` avoid `provider=auto` embedding-provider initialization and reserve expensive provider/vector probes for `--deep`.

## Best Possible Solution

Keep plain memory status on a cheap read-only path that reports provider/vector as unknown or not probed, and reserve provider auto-discovery plus live embedding/vector checks for `--deep` or indexing.

## Reproduction Assessment

Yes, source-reproducible on current main: the non-deep status branch calls `probeVectorAvailability()`, and that method initializes the embedding provider before vector readiness. I did not run the live command because this review is read-only and the slow timing depends on configured provider credentials/network behavior.

## Solution Assessment

Yes, the requested direction matches the existing docs: avoid the expensive probe in plain status and keep the deeper provider checks behind `--deep` or `--index`. A narrow fix should add regression coverage that plain status does not call provider initialization or embedding probes.
