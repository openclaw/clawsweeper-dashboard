## Summary

Keep open: current `main` still sends plain `openclaw memory status` through the provider-initializing vector probe path, despite docs reserving live vector and embedding probes for `--deep`.

## What This Changes

The issue asks for normal `openclaw memory status` to avoid expensive embedding provider auto-discovery when memory search uses `provider=auto`.

## Best Possible Solution

Make plain `memory status` use cheap status data without `probeVectorAvailability()` or provider creation, keep live vector and embedding checks behind `--deep`/`--index`, and add regression coverage for both paths.

## Reproduction Assessment

Yes from source inspection, but not from a live run: on current `main`, non-deep `runMemoryStatus()` still calls `manager.probeVectorAvailability()`, which calls `ensureProviderInitialized()` and can invoke `createEmbeddingProvider(provider=auto)`.

## Solution Assessment

Yes: the narrowest maintainable fix is to move the vector/provider probe out of the non-deep status path and leave expensive readiness checks to `--deep`/`--index`, matching the existing CLI docs without adding config.
