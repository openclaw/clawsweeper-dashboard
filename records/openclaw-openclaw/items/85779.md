## Summary

Keep open: current main still has a real mismatch between Discord guild-channel memory docs and normal-session bootstrap behavior, and the status-window side is only partly reduced by a current-main follow-up that has not shipped in the latest release. The remaining decision is product-level: either Discord guild/group sessions should stop auto-loading long-term memory, or the docs and status language need to be updated to describe that behavior clearly.

## What This Changes

The issue asks OpenClaw to reduce or clarify large repeated Discord turn context, especially MEMORY.md/workspace context injection, and to make status context-window reporting reliable.

## Best Possible Solution

Decide the intended Discord guild/group memory policy, then either add channel-aware bootstrap filtering with focused Discord/Codex coverage or update the Discord docs and status copy so loaded workspace memory is explicit.

## Reproduction Assessment

Partially yes: source inspection reproduces the docs/runtime mismatch because ordinary sessions keep MEMORY.md and Codex leaves it in turn context. No live Discord/Codex run reproduced the reported context growth or status-window flip end to end.

## Solution Assessment

Unclear until maintainers choose the product behavior. If guild channels should not auto-load memory, the narrow fix is session/channel-aware bootstrap filtering; if they should, the narrow fix is docs/status clarification plus keeping the current runtime behavior.
