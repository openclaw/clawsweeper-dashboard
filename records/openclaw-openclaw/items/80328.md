## Summary

Keep open: current main has durable final-send queueing and startup recovery, but source inspection does not show a Telegram polling-restart drain or operator-visible live failure path for the reported transport degradation.

## What This Changes

The issue reports that Telegram assistant replies can fail after polling or transport stalls and become invisible to the user/operator.

## Best Possible Solution

Reuse the existing durable delivery queue and add Telegram-owned recovery/visibility around polling or transport recovery, while preserving the current safe-send retry policy.

## Reproduction Assessment

No; there is no high-confidence deterministic live reproduction here. The issue has strong live logs and source shows a plausible gap: failed final sends can remain in the durable queue while Telegram polling restarts without draining that queue.

## Solution Assessment

Yes; the best narrow direction is to reuse the existing queue drain/status surfaces on Telegram recovery. It should not add blind sendMessage retries or cross-channel failover as part of the first fix.
