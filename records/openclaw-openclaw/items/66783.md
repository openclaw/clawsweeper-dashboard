Keep open. Current main already shipped the immediate microphone unblock, but it still lacks the configurable `permissionsPolicy` config surface; this PR remains useful but is currently dirty against main and needs docs/changelog cleanup before it is merge-ready.

The PR adds `gateway.http.securityHeaders.permissionsPolicy` as a string-or-false runtime config, threads it through gateway HTTP handling and schema metadata, changes the default Permissions-Policy behavior, and adds focused gateway tests.

## Best Possible Solution

Land a rebased, scoped version that keeps the existing `securityHeaders` pattern, exposes `permissionsPolicy?: string | false` through types, zod and generated schema metadata, runtime resolution, and HTTP handling, preserves the current microphone-safe default unless maintainers intentionally change it, and adds public docs plus a changelog entry.

## Solution Assessment

Yes with cleanup. Extending the existing gateway `securityHeaders` path is the narrow maintainable solution, but the PR should be rebased, documented, changeloged, and aligned with the current `microphone=(self)` default before merge.
