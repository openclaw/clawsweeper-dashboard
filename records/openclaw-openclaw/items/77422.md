## Summary

Keep open: current main still has a source-level path where queued inter-session follow-up runs can be registered as not Control-UI-visible, and the websocket broadcast layer then suppresses live UI pushes even though the durable transcript can still be updated.

## What This Changes

Ensure subagent completion announce runs for webchat-bound orchestrator sessions push live Control UI websocket events even when the announce arrives through inter-session or queue-drained follow-up routing.

## Best Possible Solution

Add a small shared visibility helper that treats either the resolved run origin or the persisted session channel/lastChannel/origin.provider as webchat-visible, use it at both run-context registration sites, and cover the inter_session announce-to-webchat-bound-session case.

## Reproduction Assessment

Yes, by source inspection: a queued follow-up with a webchat-bound session but non-webchat inter-session origin registers isControlUiVisible as false, and server-chat then skips the live websocket broadcasts. I did not run a live Control UI websocket repro in this read-only review.

## Solution Assessment

Likely yes in direction, but the safest implementation is a small shared helper rather than duplicating fallback logic inline. The fix should preserve suppression for truly external/channel-routed runs while allowing persisted webchat sessions to surface live updates.
