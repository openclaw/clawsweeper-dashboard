## Summary

Keep open: current main still has a plausible WebChat live-delivery gap for queued/internal subagent announce runs because Control UI visibility is registered from per-run origin/provider metadata, while server-chat suppresses websocket agent events when that flag is false.

## What This Changes

The issue requests live Control UI/WebChat websocket pushes for subagent completion announcements that are processed in a WebChat-bound parent session after the parent run is idle or queue-drained.

## Best Possible Solution

Add one shared visibility resolver for agent run registration that treats a WebChat-bound session entry as Control UI visible for internal queued/announce runs, then cover it with focused followup, direct-run, and server-chat regression tests.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only pass. The source path is reproducible by inspection: a non-WebChat per-run origin/provider can register isControlUiVisible=false, and server-chat then suppresses live agent broadcasts even when the durable session history later contains the reply.

## Solution Assessment

Unclear whether the proposed inline fallback is the best exact implementation. The narrow maintainable fix is likely a shared helper used by both run-registration call sites, backed by tests that prove WebChat-bound sessions stay visible without leaking external-channel hidden runs.
