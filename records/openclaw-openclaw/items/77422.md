## Summary

Keep open: current main still derives Control UI visibility from the per-run origin/provider, so an inter-session subagent announce can write durable history while `server-chat` suppresses the live WebSocket agent event for a WebChat-bound parent session.

## What This Changes

Ensure subagent completion announce responses pushed into an idle WebChat-bound orchestrator session reach the Control UI WebSocket without requiring refresh.

## Best Possible Solution

Centralize the Control UI visibility decision so WebChat-bound sessions remain visible for internal announce/followup runs while external-channel hidden-run suppression remains intact.

## Reproduction Assessment

Yes, source-reproducible with high confidence: the issue's inter-session announce path reaches run registration with non-WebChat provenance, and current main can register `isControlUiVisible=false` before `server-chat` suppresses live broadcasts. I did not execute a live WebChat run in this read-only pass.

## Solution Assessment

The proposed session-bound fallback is directionally correct, but the best implementation is likely a small shared helper used by both visibility registration call sites. That keeps the fix narrow and reduces the chance of weakening external-channel privacy routing.
