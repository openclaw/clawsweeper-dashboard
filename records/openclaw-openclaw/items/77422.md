## Summary

Keep open: current main still gates live Control UI/WebChat agent broadcasts on per-run origin/provider visibility, and I did not find a session-bound WebChat fallback or regression coverage for announce-driven queued runs.

## What This Changes

The issue requests that subagent completion announcements into a WebChat-bound parent session push live WebSocket updates without requiring a UI refresh.

## Best Possible Solution

Centralize the Control UI visibility decision so WebChat-bound sessions remain visible for internal announce/followup runs while external-channel hidden-run suppression stays intact.

## Reproduction Assessment

No live reproduction was established in this read-only pass. The source path is still reproducible by inspection: an inter-session queued run can register isControlUiVisible=false, and server-chat then suppresses the WebSocket agent broadcast even though history is later durable.

## Solution Assessment

Unclear whether the exact proposed patch is the best implementation. The best fix is likely a small shared helper used by both run-registration call sites, with regression coverage proving session-bound WebChat visibility without weakening external-channel privacy routing.
