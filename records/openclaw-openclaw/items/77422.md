## Summary

Keep open: current main still computes Control UI/WebChat live visibility from per-run origin/provider in both agent run setup paths, while `server-chat` suppresses live WebSocket broadcasts when that context is false. The docs promise deterministic WebChat replies and push-based subagent completion, so this remains a narrow source-backed bug.

## What This Changes

Ensure subagent completion announcements on WebChat-bound orchestrator sessions are pushed live over WebSocket even when the queued run origin is `inter_session`.

## Best Possible Solution

Centralize the Control UI visibility decision so WebChat-bound sessions stay live-visible for internal announce/followup runs while external or intentionally hidden runs remain suppressed.

## Reproduction Assessment

Yes for a source-level reproduction: an inter-session subagent announce can register `isControlUiVisible: false`, and `server-chat` then suppresses the live agent broadcast even though the session history remains durable. I did not execute the live UI flow in this read-only pass.

## Solution Assessment

Unclear whether the exact proposed patch is the best implementation. The maintainable fix is likely a shared helper used by both run-registration paths, with regression coverage proving WebChat-bound visibility and preserving hidden-run suppression.
