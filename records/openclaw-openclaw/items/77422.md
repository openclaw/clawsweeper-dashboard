## Summary

Keep open: current main still derives live Control UI visibility from per-run origin/provider at the run-registration points, while the WebChat/subagent docs describe push-based completion and live WebSocket delivery; the reported failure remains a narrow source-reproducible bug.

## What This Changes

The issue requests real-time WebChat websocket delivery for subagent completion announcements when the parent orchestrator session is idle and the response is already durable in chat history.

## Best Possible Solution

Centralize Control UI visibility resolution so WebChat-bound sessions stay live-visible for internal announce and followup runs while hidden external-channel run suppression remains intact.

## Reproduction Assessment

Yes for source-level reproduction: an announce-driven run can register isControlUiVisible=false from non-webchat per-run provenance, and server-chat then suppresses the live WebSocket broadcasts. I did not establish a live UI reproduction in this read-only pass.

## Solution Assessment

The proposed direction is right, but duplicating session-entry fallback logic at both call sites is probably not the best implementation. A shared helper used by both registration paths, with tests preserving hidden-run suppression, is the narrowest maintainable fix.
