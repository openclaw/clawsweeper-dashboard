## Summary

Keep open: this member-authored PR still has a blocking action-scoping bug. The latest branch claims delivery-only hashing, but the helper ignores `params.action`, so all `message` actions would strip ids/timestamps and can make real non-send progress look like a critical no-progress loop.

## What This Changes

The PR changes agent tool-loop result hashing for `message` and `sessions_send` outcomes and adds regression tests for volatile delivery IDs in repeated send loops.

## Best Possible Solution

Land the delivery-result hashing only after `sessions_send` stays covered and `message` is scoped to `send` or `broadcast`, with the non-send regression, core typecheck, focused test, and formatter check passing.

## Reproduction Assessment

Yes. Source inspection shows the latest branch ignores `params.action`, and the added non-send regression would hash two `read` outcomes identically after stripping `messageId` and `ts`.

## Solution Assessment

No. The targeted delivery-result projection is the right fix shape, but the current implementation is too broad; the safer fix is to gate `message` by `send`/`broadcast` and leave all other actions on full result hashing.
