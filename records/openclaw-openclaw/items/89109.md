## Summary

Keep open: the branch targets a real P1 message-spam bug, but the current patch is still not merge-ready because it strips volatile IDs from `details` while real message-tool sends also serialize those IDs into result text, so the critical no-progress paths can still be bypassed.

## What This Changes

The PR changes message-tool loop detection to hash message send outcomes without volatile delivery IDs and adds regression coverage for repeated message sends.

## Best Possible Solution

Normalize message send outcomes from the actual structured payload, removing volatile delivery IDs from both details and serialized text while preserving argsHash as the target/content guard.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: record repeated `message` sends with identical args and a `jsonResult`-shaped payload whose serialized text contains changing `messageId` values. Current main and this PR both hash extracted text in the no-progress result hash path.

## Solution Assessment

No, this is only a partial fix. The narrow maintainable fix is still in `hashToolOutcome`, but it needs to stabilize the actual message-tool payload/text shape, not only top-level `details`.
