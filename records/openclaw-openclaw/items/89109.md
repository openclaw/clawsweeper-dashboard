## Summary

Keep open. The PR fixes the central `message.send`/`sessions_send` volatile-result hash problem at the right shared layer, but it omits the sibling `upload-file` delivery action even though current channel implementations can return fresh message IDs there too, so merging as-is leaves part of the message-spam loop surface unblocked.

## What This Changes

The PR adds stable result hashing for selected message delivery tool outcomes by stripping volatile delivery identifiers, plus regression tests for repeated sends, broadcasts, replies, sessions_send, and non-send reads.

## Best Possible Solution

Land the shared stable-hash projection for all outbound delivery actions that can return volatile delivery IDs, at least including `upload-file`, while preserving full-result hashing for non-delivery message actions.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main hashes full tool result details/text while `jsonResult` puts volatile delivery IDs in both places, and critical loop paths require stable `resultHash`. I did not run a live Feishu or gateway spam loop in this read-only review.

## Solution Assessment

No, not quite as submitted. `hashToolOutcome` is the right shared layer, but the send-like action set should include `upload-file` or have a tested maintainer-approved reason for excluding it.
