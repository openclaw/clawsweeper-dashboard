## Summary

Keep open: the reported message-tool spam loop is real and the PR targets the right shared hashing layer, but the patch still hashes volatile ids through the result text, so the central Feishu/core path can continue to evade critical blocking.

## What This Changes

The PR adds message-tool result hashing that drops volatile delivery-id fields from `details` and adds loop-detection regression tests.

## Best Possible Solution

Normalize message-tool outcomes from one stable send payload that strips volatile delivery ids from both `details` and JSON result text, with Feishu/core serialized-result regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: repeated `message` sends whose result text is JSON-stringified details with changing `messageId` will still produce changing `resultHash` values under this patch; I did not run a live Feishu loop in this read-only review.

## Solution Assessment

No. The PR is at the right layer, but stripping only `details` is incomplete because current core and Feishu result text can carry the same volatile ids.
