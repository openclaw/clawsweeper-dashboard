## Summary

Keep open: the PR fixes the reported send-loop hash shape, but the current implementation is too broad because it applies volatile-id stripping to every `message` action, including non-send reads and listing actions where ids or timestamps can be real progress.

## What This Changes

The branch changes tool-loop result hashing to strip volatile delivery IDs for `message` and `sessions_send` results and adds regression tests for duplicate send loops.

## Best Possible Solution

Keep the stable send-result hash, but scope it to `sessions_send` and actual delivery actions on `message`; preserve full result hashing for non-send `message` actions before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence deterministic reproduction: current main hashes `jsonResult` details and text for `message`, so changing per-send ids prevent stable no-progress hashes. I did not run tests in this read-only review.

## Solution Assessment

No. The proposed fix is in the right layer for send loops, but it is not the best merge shape until it is narrowed to delivery actions and leaves non-send `message` result hashing unchanged.
