## Summary

Keep open. Current main still routes Windows Scheduled Task-marked config/SIGUSR1 restarts through a schtasks full-process handoff, and the maintainer discussion explicitly says PR #71465 did not cover this Windows stale-lock/EADDRINUSE failure mode; existing stale-listener and supervised-lock recovery are partial mitigation, not proof that the hot-reload crash loop is fixed.

## What This Changes

Editing openclaw.json while the Windows gateway is running should not leave the Scheduled Task/SIGUSR1 restart path in a stale-lock or EADDRINUSE crash loop.

## Best Possible Solution

Make ordinary Windows config/SIGUSR1 restarts deterministic, either by using in-process restart for non-update reloads or by adding a health-checked schtasks handoff that cleans stale lock/listener state and fails boundedly.

## Reproduction Assessment

Yes at code-path level. The report gives concrete Windows 11 steps, and current main can be regression-tested by setting Windows Scheduled Task markers, triggering a config/SIGUSR1 restart, and asserting stale lock/EADDRINUSE recovery rather than an unbounded crash loop.

## Solution Assessment

No current solution is complete. The existing stale-listener cleanup and supervised-lock retry logic are useful, but they do not specifically prove the Windows config hot-reload handoff cannot race into stale lock/EADDRINUSE; the safer fix should preserve update hard-respawn while making ordinary config reload deterministic.
