## Summary

Keep open. Current main still routes ordinary Windows Scheduled Task-marker config/SIGUSR1 restarts through the schtasks full-process handoff, and the newer adjacent fixes do not prove the reported Windows stale-lock/EADDRINUSE hot-reload crash loop is fixed.

## What This Changes

Editing openclaw.json while a Windows gateway is running should not leave the gateway in a stale-lock or EADDRINUSE crash loop after config hot-reload.

## Best Possible Solution

Make ordinary Windows non-update config/SIGUSR1 restarts deterministic, likely by using the in-process restart path or an equivalent safe handoff, while preserving update-specific fresh-process replacement and existing Scheduled Task cleanup behavior.

## Reproduction Assessment

Yes at source level: with Windows Scheduled Task markers, current main still sends a non-update config/SIGUSR1 restart through `restartGatewayProcessWithFreshPid()` into the schtasks supervised full-process path and exits the running gateway. ## Solution Assessment

No, current main is not complete for this report. The narrow maintainable direction is to fix the ordinary non-update Windows config restart path without changing update restarts or the already-merged duplicate schtasks `/Run` guard.
