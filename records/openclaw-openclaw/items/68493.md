## Summary

Keep open. Current main still routes non-update Windows config/SIGUSR1 restarts with Scheduled Task markers through the schtasks full-process handoff, and the maintainer comment already distinguished this from the restart-drain work in https://github.com/openclaw/openclaw/pull/71465; later lock and stale-listener hardening is adjacent but not issue-specific proof.

## What This Changes

The issue reports a Windows gateway hot-reload restart crash loop after editing openclaw.json, involving stale lock state and EADDRINUSE during Scheduled Task/SIGUSR1 handoff.

## Best Possible Solution

Make ordinary Windows non-update config/SIGUSR1 restarts deterministic, either in-process or through a bounded health-checked Scheduled Task handoff that cleans stale lock/listener state while preserving update restarts that need a fresh process.

## Reproduction Assessment

Yes, at source level: with Windows Scheduled Task markers, a non-update config/SIGUSR1 restart still follows restartGatewayProcessWithFreshPid to triggerOpenClawRestart/schtasks and then exits as a supervised full-process restart. I did not run a live Windows Scheduled Task repro in this Linux checkout.

## Solution Assessment

No current solution is complete. The existing stale-listener cleanup and supervised-lock retry bounds are useful hardening, but they do not prove that ordinary Windows config hot-reload avoids the schtasks race and stale-lock/EADDRINUSE recovery failure.
