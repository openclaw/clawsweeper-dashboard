## Summary

Keep open. Current main still sends ordinary Windows Scheduled Task-managed config restarts through the schtasks full-process handoff, and the live maintainer comment explicitly says the merged restart-drain work did not cover this stale-lock/EADDRINUSE failure mode.

## What This Changes

Editing `openclaw.json` while a Windows Scheduled Task-managed gateway is running should not leave the gateway in a stale-lock/EADDRINUSE restart loop.

## Best Possible Solution

Make ordinary non-update Windows Scheduled Task config restarts deterministic while preserving update-specific process replacement and existing schtasks cleanup behavior.

## Reproduction Assessment

Yes at source level: a restart-required config edit reaches `emitGatewayRestart()`, and Windows Scheduled Task markers still route the non-update SIGUSR1 path through schtasks full-process handoff and process exit. I did not establish live native Windows crash-loop proof in this checkout.

## Solution Assessment

No, current main is not complete for this report. The narrow maintainable fix is to change the ordinary non-update Windows config restart path without weakening update restarts or the scheduled-task `/Run` guard.
