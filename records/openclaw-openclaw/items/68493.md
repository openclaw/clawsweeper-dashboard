## Summary

Keep open. Current main still routes Windows Scheduled Task-marked non-update config/SIGUSR1 restarts through a schtasks full-process handoff, and the maintainer comment says https://github.com/openclaw/openclaw/pull/71465 did not cover this Windows stale-lock/EADDRINUSE failure mode.

## What This Changes

Fix Windows gateway config hot-reload after editing openclaw.json so it does not crash-loop with a stale lock file and EADDRINUSE.

## Best Possible Solution

Make ordinary Windows config/SIGUSR1 restarts deterministic, preferably in-process for non-update reloads or through a bounded health-checked schtasks handoff that cleans stale lock/listener state while preserving update fresh-process restarts.

## Reproduction Assessment

Yes at source level: set Windows Scheduled Task markers, trigger a non-update config/SIGUSR1 restart, and current code still goes through schtasks/full-process exit instead of a proven in-process or health-checked recovery path. I did not run a live Windows Scheduled Task repro in this read-only Linux checkout.

## Solution Assessment

No current solution is complete. The stale-listener cleanup and systemd loop fix are useful adjacent hardening, but they do not prove the Windows config hot-reload schtasks handoff cannot race into stale lock/EADDRINUSE recovery failure.
