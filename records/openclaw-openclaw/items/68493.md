## Summary

Keep open. Current main still routes ordinary Windows service config/SIGUSR1 restarts through the schtasks full-process handoff, while the maintainer discussion explicitly says the merged restart-drain work did not cover this Windows stale-lock/EADDRINUSE failure mode.

## What This Changes

Editing `openclaw.json` while a Windows managed gateway is running should hot-reload or restart cleanly without leaving a stale lock, racing the gateway port, or requiring manual lock deletion.

## Best Possible Solution

Make ordinary Windows config/SIGUSR1 restarts deterministic: either keep non-update config reloads in-process when the gateway is already running, or add a Windows schtasks handoff that health-checks startup, recovers stale lock/listener state, and fails with bounded, actionable behavior instead of looping.

## Reproduction Assessment

Yes at a code-path level. The report gives concrete Windows 11 steps, and current main can be exercised by setting Windows Scheduled Task markers, triggering a config/SIGUSR1 restart, and observing the schtasks supervised full-process path plus lock/listener recovery behavior.

## Solution Assessment

No current solution is complete. PR #71465 improved restart draining, and main has stale listener cleanup, but the remaining Windows schtasks hot-reload restart path still needs targeted behavior and regression coverage for stale lock/EADDRINUSE recovery.
