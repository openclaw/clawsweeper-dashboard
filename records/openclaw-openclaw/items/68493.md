## Summary

Keep open. Current main still routes ordinary Windows Scheduled Task config/SIGUSR1 restarts through the full-process schtasks handoff, and the live maintainer comment explicitly distinguishes this stale-lock/EADDRINUSE failure from the restart-drain fix that already landed.

## What This Changes

The issue reports that editing `openclaw.json` while a Windows gateway is running can trigger a hot-reload restart loop with a stale lock file and EADDRINUSE.

## Best Possible Solution

Make ordinary non-update Windows Scheduled Task config restarts deterministic while preserving update-specific process replacement and existing schtasks cleanup behavior.

## Reproduction Assessment

Yes at source level: with Windows Scheduled Task markers, current main still sends a non-update config/SIGUSR1 restart through `restartGatewayProcessWithFreshPid()` into the schtasks supervised full-process path and exits. I did not establish a live Windows crash-loop reproduction in this checkout.

## Solution Assessment

No, current main is not complete for this report. The narrow maintainable fix is to change the ordinary non-update Windows config restart path without changing update restarts or the already-merged scheduled-task `/Run` guard.
