## Summary

Current main and the v2026.5.18 stable release implement the missing ordinary unmanaged restart fix, so the Windows stale-lock/EADDRINUSE crash-loop path reported here is no longer the current behavior; the separate Windows Scheduled Task `/Run` diagnostics/fallback issue remains tracked elsewhere.

## What This Changes

The issue reports that editing openclaw.json while the Windows gateway is running can trigger a restart crash loop involving stale locks, EADDRINUSE, and WebSocket 1006 failures.

## Best Possible Solution

Keep the shipped in-process ordinary restart behavior; continue tracking remaining Windows Scheduled Task `/Run` diagnostics and half-state cleanup in https://github.com/openclaw/openclaw/issues/90158.

## Reproduction Assessment

Yes for the original failure at source level: pre-fix ordinary unmanaged SIGUSR1/config restarts could detach-spawn and exit the tracked PID. Current main no longer reproduces that path because the helper returns disabled and the run loop restarts in-process.

## Solution Assessment

Yes. Reusing the existing in-process fallback for ordinary unmanaged restarts is the narrowest maintainable fix, while update restarts still keep their fresh-process replacement path.
