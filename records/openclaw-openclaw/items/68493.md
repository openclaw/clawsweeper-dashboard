## Summary

Keep open. Current main still sends ordinary Windows Scheduled Task-marker config/SIGUSR1 restarts through the schtasks full-process handoff and exits the running gateway; the merged adjacent fixes harden restart drain, duplicate schtasks runs, and stale-listener cleanup, but do not prove this hot-reload stale-lock/EADDRINUSE loop is fixed.

## What This Changes

The issue reports that editing `openclaw.json` while a Windows gateway is running can trigger a config/SIGUSR1 restart loop with a stale lock file, `EADDRINUSE`, and failed WebSocket handshakes.

## Best Possible Solution

Make ordinary Windows non-update config/SIGUSR1 restarts deterministic with focused regression coverage, while preserving update restarts that need a fresh process and existing Scheduled Task cleanup behavior.

## Reproduction Assessment

Yes, at source level: with Windows Scheduled Task markers, a non-update config/SIGUSR1 restart still follows `restartGatewayProcessWithFreshPid` into the schtasks supervised full-process path and exits the running gateway. I did not establish a live Windows Scheduled Task repro in this checkout.

## Solution Assessment

No, current main is not complete. The narrow maintainable fix is to change ordinary non-update Windows config/SIGUSR1 restarts or their handoff so the gateway does not enter the stale-lock/port-rebind loop, while keeping update-specific process replacement intact.
