## Label Justifications

- `P1`: The report describes a Windows gateway restart crash loop after config edits that could leave the local gateway unusable until manual cleanup.

## Summary

Close as implemented: current main and shipped releases now keep ordinary non-update gateway restarts in-process when no trusted supervisor is detected, which removes the config/SIGUSR1 detached-child path that caused stale gateway locks and EADDRINUSE loops.

## What This Changes

The issue asks for Windows config hot-reload restarts to complete cleanly instead of leaving a stale gateway lock and EADDRINUSE crash loop after editing openclaw.json.

## Best Possible Solution

Keep the shipped in-process ordinary restart behavior; use https://github.com/openclaw/openclaw/issues/49036 only for the separate doctor-facing stale gateway lock cleanup UX.

## Reproduction Assessment

No current-main failing reproduction remains: source inspection shows the historical detach-spawn path now returns the in-process fallback for ordinary unmanaged restarts. The original failure path is still well understood from the related report and the merged fix provenance.

## Solution Assessment

Yes. The implemented fix is the narrow maintainable solution because ordinary non-update restarts no longer detach-spawn, while trusted supervisor handoff and update.run fresh-process replacement remain separate.
