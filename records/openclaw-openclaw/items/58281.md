## Summary

Keep open. Current main still only resolves `~` and would keep `${XDG_CONFIG_HOME}` literal through the shared `resolveUserPath` path, so the PR is not obsolete; however, the proposed helper should not merge as-is because unset placeholders are collapsed to an empty string, which can redirect configured state/workspace paths to the filesystem root, and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR adds an `expandEnvVars` helper to `src/infra/home-dir.ts`, wires it into `resolveHomeRelativePath` and `resolveOsHomeRelativePath`, and adds tests for `$VAR` and `${VAR}` path expansion.

## Best Possible Solution

Land a narrow core path-resolution fix that expands `$VAR` and `${VAR}` only when the variable has a usable value, preserves or fails closed for unresolved placeholders, keeps `$HOME` aligned with the existing `OPENCLAW_HOME > HOME > USERPROFILE > os.homedir()` home contract, and covers the skill workspace path flow plus a changelog entry before closing #53628.

## Reproduction Assessment

Yes. On current main, a focused call to `resolveHomeRelativePath("${XDG_CONFIG_HOME}/workspace", { env: { XDG_CONFIG_HOME: "/home/node/.openclaw" } })` follows the non-tilde branch and resolves the literal placeholder; the PR regression can be reproduced with a leading unset placeholder such as `${MISSING}/workspace`.

## Solution Assessment

No. The PR changes the right shared helper, but the empty-string fallback is not the safest maintainable behavior for configured paths; preserving unresolved placeholders or failing closed, with a home-specific fallback for `$HOME`, is safer and matches the existing home-resolution contract.
