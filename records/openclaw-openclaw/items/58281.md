## Summary

Keep open. Current main still leaves environment placeholders literal in shared user path resolution, and this PR targets the right core helper, but the proposed empty-string fallback would turn missing or empty leading placeholders into root-relative paths and the user-facing fix still needs a changelog entry. The overlapping open PR #53718 is useful context, but this PR remains a narrow repair candidate rather than something to close as implemented or obsolete.

## What This Changes

The PR adds an exported `expandEnvVars` helper to `src/infra/home-dir.ts`, wires it into `resolveHomeRelativePath` and `resolveOsHomeRelativePath`, and adds tests for `$VAR` and `${VAR}` path expansion.

## Best Possible Solution

Land one canonical shared path-resolver fix that expands `$VAR` and `${VAR}` only when a variable has a usable own value, preserves or fails closed for unresolved placeholders, handles tilde-valued env values through the existing home contracts, covers both resolver helpers plus the active-workspace ClawHub install path, and adds the required changelog entry before closing #53628 and retiring the unchosen overlapping PR.

## Reproduction Assessment

Yes. Current main has a code-level reproduction: a configured workspace string such as `${XDG_CONFIG_HOME}/workspace` flows through `resolveUserPath` into `resolveHomeRelativePath`, which only expands `~`; the PR regression is reproduced by `${MISSING}/workspace`, which its helper turns into `/workspace`.

## Solution Assessment

No. The PR changes the right shared helper, but replacing missing or empty variables with `""` is not the safest path semantics; preserving the token or failing closed, with explicit handling for home-like values and tilde-valued env vars, is the narrower maintainable fix.
