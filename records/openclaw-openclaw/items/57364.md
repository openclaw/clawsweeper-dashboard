## Summary

Keep open. Current main still leaves stale MS Teams FileConsentCards on decline, expired/not-found, conversation-mismatch, and upload-failure paths, and this PR is now stale because the live invoke handler moved from the older inlined `monitor-handler.ts` code into `file-consent-invoke.ts`.

## What This Changes

The PR adds best-effort FileConsentCard deletion through `context.deleteActivity(activity.replyToId)` in the MS Teams file-consent flow and extends tests for success, decline, expiry, upload failure, and delete-failure resilience.

## Best Possible Solution

Move the cleanup into `extensions/msteams/src/file-consent-invoke.ts`, target the stored consent-card activity id after conversation validation when available, keep deletion best-effort, preserve or intentionally replace the current successful-accept `updateActivity` behavior, and add focused tests for decline, expired/not-found, conversation mismatch, upload failure, and delete failure.

## Reproduction Assessment

Yes. Against current main, the static reproduction is to follow the active `fileConsent/invoke` paths for decline, missing upload, conversation mismatch, or `uploadToConsentUrl` failure: those branches send or clean state without any `deleteActivity` call.

## Solution Assessment

No. The intended cleanup is the right product direction, but this patch is not the best current implementation because it targets stale inlined handler code instead of the active `file-consent-invoke.ts` route and should use the stored consent-card activity id when possible.
