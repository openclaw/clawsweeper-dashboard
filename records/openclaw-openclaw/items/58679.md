## What This Changes

The PR adds config provenance metadata to config snapshots and wires a new `openclaw config explain` CLI command plus docs for config path, include, env, and selected key-source diagnostics.

## Best Possible Solution

Rebase the implementation onto current main, compute provenance from the authoritative include/env read pipeline, keep provenance on invalid snapshots, render failed includes with accurate neutral status, and add focused tests plus real command proof.

## Reproduction Assessment

Yes. Static source inspection of PR head reproduces the blockers: invalid snapshots omit provenance, env provenance scans the root parsed tree, and failed includes are rendered with misleading status text.

## Solution Assessment

No. The product direction is reasonable, but this branch is not the best mergeable form until provenance follows the same include/env semantics as config loading and current main's config CLI surface is preserved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Populate provenance on invalid snapshots:** `src/config/io.ts:2320-2332`
  - body: The schema-validation failure branch builds a snapshot without `provenance`, so `config explain` reports empty provenance exactly when an operator is debugging a bad but parsed config. Build the same provenance snapshot here that the valid path returns.
  - confidence: 0.9
- **[P2] Scan the include-resolved tree for env provenance:** `src/config/io.ts:1902`
  - body: `collectEnvReferences` only traverses `params.parsed`, but actual config reads resolve `$include` before applying `${ENV}` substitution. Env refs that exist only inside included files can be substituted by the loader while `config explain` reports no env provenance.
  - confidence: 0.88
- **[P2] Track failed ancestors for nested include failures:** `src/config/io.ts:2275-2284`
  - body: For non-circular nested include failures, `ConfigIncludeError` only carries the inner failed path while the root parsed config only lists the outer `$include`. That outer include can still be reported as applied even though its merge never completed.
  - confidence: 0.84
- **[P2] Use neutral status text for unapplied includes:** `src/commands/config-explain.ts:36`
  - body: The renderer labels every unapplied include as `(not found)`, but `applied: false` also covers circular chains, parse errors, guarded reads, and ancestor failures. Use neutral text or carry a failure reason so diagnostics do not point to the wrong cause.
  - confidence: 0.86
