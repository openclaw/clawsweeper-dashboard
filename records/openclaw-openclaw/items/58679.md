## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Populate provenance for schema-invalid snapshots:** `src/config/io.ts:2325`
  - body: In this validation-failure return path, the snapshot is built without `provenance`. `config explain` then falls back to empty entries/keySources for configs that parsed and resolved but failed schema validation, which removes the diagnostic output exactly when operators are investigating an invalid config.
  - confidence: 0.9
- **[P2] Scan include-resolved config for env provenance:** `src/config/io.ts:1902`
  - body: `collectEnvReferences` walks `params.parsed`, and the valid read path passes the root `effectiveParsed`, while `${ENV}` substitution runs after `$include` resolution on `resolved`. Any env token that exists only in an included file is substituted by OpenClaw but omitted from `config explain`, so the new command underreports env provenance.
  - confidence: 0.88
- **[P2] Mark failed include branches for non-circular errors:** `src/config/io.ts:2281-2284`
  - body: `failedIncludeAncestors` is populated only for `CircularIncludeError`. If the root config includes `a.json5` and `a.json5` fails to read or parse a nested include, the root include is still reported as applied even though the merge never completed.
  - confidence: 0.86
- **[P2] Use neutral status text for unapplied includes:** `src/commands/config-explain.ts:36`
  - body: The formatter prints every `applied: false` entry as `(not found)`, but the provenance builder also uses that state for circular include chains and other failed include branches. That points operators at the wrong root cause for parse, security, or circular failures.
  - confidence: 0.87
