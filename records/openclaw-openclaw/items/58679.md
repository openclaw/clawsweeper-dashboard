## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Populate provenance on invalid snapshots:** `src/config/io.ts:2320-2332`
  - body: The schema-validation failure branch builds a snapshot without `provenance`, so `config explain` reports empty provenance exactly when an operator is debugging a bad but parsed config. Build the same provenance snapshot here that the valid path returns.
  - confidence: 0.9
- **[P2] Scan the include-resolved config for env provenance:** `src/config/io.ts:2350`
  - body: `buildConfigProvenance` traverses the pre-include `effectiveParsed` tree, but actual config reads resolve `$include` before applying `${ENV}` substitution. Env refs that exist only inside included files can be substituted by the loader while `config explain` reports no env provenance.
  - confidence: 0.88
- **[P2] Mark failed parent includes on nested failures:** `src/config/io.ts:2275-2284`
  - body: For non-circular nested include failures, `ConfigIncludeError` only carries the inner failed path while the root parsed config only lists the outer `$include`. That outer include can still be reported as applied even though its merge never completed.
  - confidence: 0.84
- **[P2] Use neutral status text for unapplied includes:** `src/commands/config-explain.ts:36`
  - body: The renderer labels every unapplied include as `(not found)`, but `applied: false` also covers circular chains, parse errors, guarded reads, and ancestor failures. Use neutral text or carry a failure reason so diagnostics do not point to the wrong cause.
  - confidence: 0.86
- **[P2] Use toSorted for include path ordering:** `src/config/io.ts:1853`
  - body: CI's `pnpm lint` fails on this `Array#sort()` call with the repository rule requiring `Array#toSorted()`. Switch the copied include path array to `toSorted()` so the PR can pass lint.
  - confidence: 0.92
