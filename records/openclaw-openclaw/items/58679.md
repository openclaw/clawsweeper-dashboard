## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Populate provenance for invalid snapshots:** `src/config/io.ts:2320-2325`
  - body: The schema-invalid `readConfigFileSnapshotInternal` branch returns a snapshot without `provenance`, so `openclaw config explain` falls back to empty entries for parsed-but-invalid configs. Build the same provenance snapshot here that the valid branch returns.
  - confidence: 0.9
- **[P2] Scan include-resolved config for env provenance:** `src/config/io.ts:1902`
  - body: `buildConfigProvenance` scans the root parsed config, but loading resolves `$include` before applying `${ENV}` substitutions. Env refs that exist only inside included files can be substituted while `config explain` reports no env provenance.
  - confidence: 0.88
- **[P2] Mark failed include ancestors for all nested errors:** `src/config/io.ts:2281-2284`
  - body: `failedIncludeAncestors` is only populated for circular includes, so a root include whose nested include fails with a missing, parse, or security error can still be rendered as applied. Track the failed branch or mark it conservatively unapplied.
  - confidence: 0.84
- **[P2] Use neutral text for unapplied includes:** `src/commands/config-explain.ts:36`
  - body: The renderer labels every `applied: false` include as `(not found)`, but the flag can also mean circular chains, parse/security failures, or ancestor failures. Use neutral text such as `not applied` or carry the actual reason.
  - confidence: 0.86
- **[P2] Use non-mutating include ordering:** `src/config/io.ts:1853`
  - body: CI's `pnpm lint` rejects this new `Array#sort()` call. Switch the copied include array to `toSorted()` or another non-mutating ordering so the check lane can pass.
  - confidence: 0.92
