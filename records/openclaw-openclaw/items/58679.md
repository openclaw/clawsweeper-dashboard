Full review comments:

- **[P2] Add provenance for invalid configs:** `src/config/io.ts:2325`
  - body: In the schema-invalid return path, the snapshot is built without `provenance`, so `config explain` falls back to empty entries exactly when an operator is troubleshooting a malformed config. Add the same `buildConfigProvenance` call used by the valid path before returning this snapshot.
  - confidence: 0.92
- **[P2] Detect env references from included files:** `src/config/io.ts:1902`
  - body: `buildConfigProvenance` checks `params.parsed`, which is the root config payload before `$include` expansion on successful reads. If `${ENV}` appears only inside an included file, the command reports no env provenance even though the active config used env substitution.
  - confidence: 0.88
- **[P2] Mark failed include branches, not only terminal includes:** `src/config/io.ts:2275-2284`
  - body: For non-circular nested include failures, the catch path passes only the terminal `includePath`; the provenance builder lists root `$include` entries, so a parent include can still be shown as applied even though its nested include aborted resolution. Track the include stack or conservatively mark the affected root branch unapplied.
  - confidence: 0.84
- **[P3] Use a neutral label for unapplied includes:** `src/commands/config-explain.ts:36`
  - body: `applied: false` is used for more than missing files, including circular, parse, security, and ancestor failures, but the CLI always prints `(not found)`. That points users at the wrong fix for several real include errors; render a neutral status such as `not applied` or carry the actual reason.
  - confidence: 0.9
- **[P2] Track skipped and ancestor includes as unapplied:** `src/config/io.ts:1865-1869`
  - body: `applied` only checks the terminal failed path and circular ancestors. Because include resolution aborts on the first thrown include error, later siblings in an include array and non-circular parent includes can still be printed as applied even though their merge never completed.
  - confidence: 0.87
- **[P2] Scan include-resolved config for env references:** `src/config/io.ts:1902-1908`
  - body: This scans only the root parsed payload, but env substitution runs after `$include` resolution. A `${VAR}` that exists only inside an included file affects runtime config but will not produce the `env` provenance entry, making `config explain` under-report substitutions.
  - confidence: 0.84
- **[P2] Use neutral status text for unapplied includes:** `src/commands/config-explain.ts:35-37`
  - body: Every unapplied include is rendered as `(not found)`, but the builder also marks entries false for circular chains and other include-resolution failures. That sends operators toward the wrong root cause for parse, security, depth, or ancestor failures.
  - confidence: 0.86
