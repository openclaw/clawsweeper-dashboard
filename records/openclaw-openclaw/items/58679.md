Full review comments:

- **[P2] Add provenance to invalid-schema snapshots:** `src/config/io.ts:2320-2325`
  - body: The schema-invalid return path builds a snapshot without `provenance`, while the valid and include-error paths attach it. `config explain` will therefore hide include/env/key-source context for parsed-but-invalid configs, which is one of the main troubleshooting cases for this command.
  - confidence: 0.9
- **[P2] Derive env provenance from the resolved config:** `src/config/io.ts:1902`
  - body: `buildConfigProvenance` scans `params.parsed`, but config loading resolves `$include` before applying `${ENV}` substitutions. Env references that exist only inside included files can be substituted while `config explain` reports no env provenance.
  - confidence: 0.88
- **[P2] Mark non-circular failed include ancestors:** `src/config/io.ts:2281-2284`
  - body: Only circular include errors populate `failedIncludeAncestors`. If a root include loads a file whose nested include is missing, invalid, or blocked, the root include can still render as applied even though its merge never completed.
  - confidence: 0.84
- **[P2] Use neutral text for unapplied includes:** `src/commands/config-explain.ts:36`
  - body: The CLI renders every `applied: false` include as `(not found)`, but that state also covers circular, parse, security, and ancestor failures. This can point operators at the wrong cause while debugging include trees.
  - confidence: 0.86
- **[P2] Use toSorted for include ordering:** `src/config/io.ts:1853`
  - body: The new `[...includePaths].sort()` violates the repo lint rule requiring `toSorted()`, and the PR's CI `check` job fails on that lint error before the branch can merge.
  - confidence: 0.92
