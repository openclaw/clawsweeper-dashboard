## Summary

Keep open. Current main still matches the reported visibility gap: native subagent work is tracked as a detached child run, `sessions_yield` deliberately ends the parent turn, and the Control UI/TUI chat paths do not render a requester-scoped waiting state once the parent run finalizes. The prior ClawSweeper comment is still consistent with current source and docs; the issue is a valid narrow repair candidate if scoped to a progress/wait indicator rather than full native child stream forwarding.

## What This Changes

The issue requests visible parent-session progress, and ideally subagent streaming, while a main session has yielded and is waiting for native subagent work to complete.

## Best Possible Solution

Add a generic requester-scoped native subagent waiting/progress signal that begins when the parent yields while child work is active and clears on child completion, error, kill, pause, or parent resume. Render that state in Control UI and TUI; leave full native streamTo="parent" forwarding as a separate product/API decision.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: spawn a native subagent, call sessions_yield in the parent, and observe that current main finalizes the parent run while Control UI/TUI ignore the child run for the parent chat status.

## Solution Assessment

Yes for the minimum repair. A requester-scoped lifecycle/progress event is narrower and safer than forwarding raw native child deltas into the parent session, while still solving the frozen-looking UI reported here.
