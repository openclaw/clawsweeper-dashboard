## Label Justifications

- `P1`: A yielded parent session can appear frozen across primary user surfaces during active subagent work, breaking a real agent workflow until completion or manual investigation.
- `impact:session-state`: The issue is about parent session state becoming dormant or indistinguishable from idle while child work remains active.
- `impact:message-loss`: Intermediate progress and waiting-state events are not surfaced to attached parent-session clients, so user-visible delivery is suppressed or delayed.

## Summary

Keep open. Current main still has only partial yielded-subagent visibility: native subagent `streamTo` is ACP-only/ignored, yielded metadata reaches lifecycle/wait plumbing, but TUI and Control UI still lack a parent-visible yielded waiting state or native child progress relay.

## What This Changes

The issue asks for parent-session surfaces to show progress or forwarded child output while `sessions_yield` waits on a spawned subagent.

## Best Possible Solution

Add a structured parent-yield wait lifecycle/session state consumed by TUI and Control UI, then design native child stream forwarding as an explicit redacted, policy-backed extension.

## Reproduction Assessment

Yes, at source level. A native `sessions_spawn` plus `sessions_yield` path ignores native `streamTo`, emits yielded/liveness metadata only at terminal lifecycle boundaries, and current TUI/Control UI contracts do not expose a yielded waiting state.

## Solution Assessment

Unclear until maintainers choose the event contract. The narrow maintainable first step is a parent-visible yielded wait state; full native child streaming should stay behind a separate redaction- and verbosity-backed design.
