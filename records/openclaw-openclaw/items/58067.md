## Label Justifications

- `P1`: A yielded parent session can appear frozen across primary user surfaces during active subagent work, breaking a real agent workflow until completion or manual investigation.
- `impact:session-state`: The issue is about parent session state becoming dormant or indistinguishable from idle while child work remains active.
- `impact:message-loss`: Intermediate progress and waiting-state events are not surfaced to attached parent-session clients, so user-visible delivery is suppressed or delayed.

## Summary

Keep open. Current main still source-reproduces the visibility gap: native subagent `streamTo` is ignored, yielded metadata is terminal/wait plumbing rather than a parent-visible surface state, and TUI/Control UI do not yet share a yielded-wait/progress contract.

## What This Changes

Request a parent-visible progress or waiting state, and ideally native subagent progress streaming, while `sessions_yield` waits for child completion.

## Best Possible Solution

Add a structured parent-yield wait lifecycle/session state consumed by TUI and Control UI, then design native child progress streaming as a separate explicit redacted policy-backed extension.

## Reproduction Assessment

Yes, at source level. A native `sessions_spawn` plus `sessions_yield` path ignores native `streamTo`, records yielded/liveness state only as terminal metadata, and current TUI/Control UI code does not expose that as a parent-visible waiting state.

## Solution Assessment

Unclear until maintainers choose the event contract. The narrow first step is a structured yielded-wait state; full native child streaming needs separate redaction, verbosity, and trust policy.
