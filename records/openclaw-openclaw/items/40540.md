## Summary

Keep open: current main still activates npm global updates by renaming the live package tree from the running updater process, so the Windows `EBUSY` self-update report remains valid; open PR #75649 is the active fix path.

## What This Changes

Windows npm users report `openclaw update` failing with an `EBUSY` rename when the global OpenClaw package tree is locked by the running CLI or Gateway process.

## Best Possible Solution

Land a reviewed Windows-only detached staged-swap flow, currently represented by PR #75649, so the active process exits before replacing the npm package tree while preserving staged verification, restart status, docs, and regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: on Windows with a global npm install, `openclaw update` still reaches an in-process `fs.rename` of the live package root after staging the new package. I did not execute a live Windows run in this checkout.

## Solution Assessment

Yes directionally: a Windows-only detached handoff after staged verification is narrower and more maintainable than trying to rename files held by the current Node process. The exact PR should be reviewed and landed rather than closing the issue now.
