## Summary

Keep open. Current main still updates npm-global installs by staging a package and renaming it into the live global package root from the active update flow, so the reported Windows EBUSY self-lock remains plausible and unresolved. The open detached-updater PR #44614 is the right active implementation candidate to review or replace before this issue closes.

## What This Changes

The issue reports Windows npm-global self-updates failing with EBUSY when `openclaw update` tries to replace the active OpenClaw package tree.

## Best Possible Solution

Keep this as the canonical Windows npm-global self-update EBUSY report until OpenClaw has a Windows-safe update handoff that releases the active CLI/Gateway process before replacing the live package tree, while preserving service restart verification, diagnostics, npm fallback behavior, and regression coverage.

## Reproduction Assessment

Yes. The issue includes concrete Windows Server npm-global steps and an EBUSY rename log, and current main still contains the live package-root rename path that matches that failure mode.

## Solution Assessment

No. The current service-stop mitigation helps one case but does not release the active OpenClaw process before replacing the package tree; a reviewed detached or exit-and-relaunch updater such as #44614 is the narrower complete fix.
