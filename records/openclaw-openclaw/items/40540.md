## Summary

Current main and the latest stable release now prevent the reported Windows update lock by stopping the managed Gateway before shell-driven package replacement and using a detached managed-service handoff for Gateway-driven updates.

## What This Changes

Fix Windows npm-global self-updates so `openclaw update` can replace the installation without a running Gateway locking the package directory.

## Best Possible Solution

Keep the shipped pre-update service stop and detached control-plane handoff as the canonical Windows updater lifecycle, with the Windows npm-global upgrade smoke guarding update and restart behavior.

## Reproduction Assessment

Yes. The reported npm `EBUSY` rename maps to the historical Windows path where a managed Gateway remained active while its global package directory was replaced, and the Windows upgrade scenario explicitly covers stopping that process first.

## Solution Assessment

Yes. Stopping the managed service before shell updates and delegating Gateway-originated updates to a detached process directly removes the locking process instead of relying on retries or recovery hints.
