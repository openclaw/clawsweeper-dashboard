## Summary

Keep open: current main and v2026.5.22 still leave the `openclaw update` CLI package-manager path on the staged global install plus in-process package-root swap that matches the Windows `EBUSY` rename report. Current main added a managed-service handoff for Gateway control-plane `update.run`, but that does not close the CLI self-update failure, and the active detached Windows handoff remains open at https://github.com/openclaw/openclaw/pull/75649.

## What This Changes

The issue asks for Windows npm-global `openclaw update` to update OpenClaw without failing on an `EBUSY` rename of the active package directory.

## Best Possible Solution

Rebase, review, prove, and land or replace the detached Windows package-swap handoff so the locked package-tree replacement happens outside the invoking OpenClaw process while preserving staged verification and restart status reporting.

## Reproduction Assessment

Yes, at source level. The report includes concrete Windows npm-global `EBUSY` rename logs, and current main still performs the staged package-root swap from the CLI update path that matches the failure surface.

## Solution Assessment

No, current main is not enough. The maintainable fix is to complete the detached Windows handoff for the CLI/package-swap path and prove it on a real Windows npm-global install.
