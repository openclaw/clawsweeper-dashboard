## Summary

Keep open: current main still performs the npm-global package swap from the `openclaw update` CLI path inside the invoking process, which matches the Windows `EBUSY` rename report. The Gateway control-plane `update.run` path now has a managed-service handoff, but the remaining CLI/package-swap fix is still owned by the open implementation PR at https://github.com/openclaw/openclaw/pull/75649.

## What This Changes

The issue reports Windows npm-global `openclaw update` failing with `EBUSY` during package-root rename because the running OpenClaw process locks the install directory.

## Best Possible Solution

Rebase, review, prove, and land or replace the detached Windows package-swap handoff so the locked package-tree replacement happens outside the invoking OpenClaw process while preserving staged verification and restart status reporting.

## Reproduction Assessment

Yes, at source level. The report includes concrete Windows npm-global `EBUSY` rename logs, and current main still performs the staged package-root swap from the CLI update path that matches the failure surface.

## Solution Assessment

No, current main is not enough. The maintainable fix is to complete the detached Windows handoff for the CLI/package-swap path and prove it on a real Windows npm-global install.
