## Summary

Keep open: current main still has the Windows self-update failure path, and an open implementation PR is already carrying the detached Windows update handoff needed to resolve it.

## What This Changes

Native Windows npm global installs should update successfully instead of failing with EBUSY when `openclaw update` replaces the package it is running from.

## Best Possible Solution

Review and land #75649 or an equivalent detached Windows npm swap/restart implementation, then close this issue as implemented once it reaches main and a release.

## Reproduction Assessment

Yes, source-reproducible: on native Windows with an npm global install, `openclaw update` reaches the staged `global install swap` path that renames the live package root while the CLI/Gateway process can still have files loaded. I did not run a live Windows reproduction in this read-only review.

## Solution Assessment

Yes. The detached-helper direction in #75649 is the narrow maintainable fix because it keeps staged verification in the current flow but moves only the Windows-locked package-tree swap and restart out of the live process; the hint-only path from #40724 was not enough to make updates succeed.
