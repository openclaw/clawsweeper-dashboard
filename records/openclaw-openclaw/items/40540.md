## Summary

Keep open. Current main still performs the npm-global package-tree swap from the active update flow, so the reported Windows EBUSY self-lock remains unresolved; #75649 is the active follow-up PR to review before this issue can close.

## What This Changes

Running `openclaw update` on Windows npm-global installs can fail with EBUSY when OpenClaw replaces the live package tree that the active process is using.

## Best Possible Solution

Land or replace #75649 with a reviewed Windows-safe update handoff that keeps staged verification but delegates the locked package-tree swap and restart outside the active OpenClaw process.

## Reproduction Assessment

Yes. The issue includes concrete Windows Server npm-global steps and an EBUSY rename log, and current main still contains the same live package-root rename path from the active update flow.

## Solution Assessment

No. Current main's managed-service stop helps one class of lock, but it does not release the active CLI/Gateway process before replacing the live package tree; the detached staged handoff in #75649 is the narrower complete direction to review.
