## Summary

Keep this open: current main improved the Gateway control-plane update handoff, but the reported `openclaw update` CLI path still performs the npm-global package-tree swap from the running OpenClaw process. The open replacement PR at https://github.com/openclaw/openclaw/pull/75649 is still the canonical path, but it has not landed and still needs Windows proof/revision.

## What This Changes

The issue asks OpenClaw to stop failing Windows npm-global self-updates with `EBUSY`/`rename` when `openclaw update` replaces the installed package it is running from.

## Best Possible Solution

Land a focused Windows npm-global update path that preserves staged verification, releases the invoking OpenClaw process before the locked package-tree swap, reconciles restart/status, and includes real Windows npm-global proof.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here. The report gives concrete Windows npm-global `EBUSY` rename logs, and current main still swaps the active npm package root from the running CLI update path.

## Solution Assessment

No, closing as implemented is not the best path. The maintainable route is to finish https://github.com/openclaw/openclaw/pull/75649 or an equivalent detached Windows CLI handoff that covers the reported `openclaw update` path without losing staged verification.
