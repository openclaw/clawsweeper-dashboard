## What This Changes

The issue requests that Windows npm-global `openclaw update` avoid EBUSY self-update failures when replacing the active OpenClaw package tree.

## Best Possible Solution

Land a focused detached Windows npm-global update flow that stages and verifies the package, releases the invoking CLI process before swapping the locked package tree, preserves restart/status behavior, and includes redacted Windows npm-global proof.

## Reproduction Assessment

Yes, at source level. The issue includes concrete Windows npm-global EBUSY rename logs, and current main still maps `openclaw update` to in-process `runGlobalPackageUpdateSteps(...)` and `swapStagedNpmInstall(...)`.

## Solution Assessment

No, current main is not enough. Service-stop and control-plane handoff safeguards cover the live Gateway process, but the CLI still swaps the active npm package root from the invoking `openclaw update` process.
