## Summary

Keep open: current main still performs an in-process npm package-tree swap that can hit the Windows self-lock reported here, and the active repair PR #75649 has not merged or shipped yet.

## What This Changes

The issue asks for npm-global `openclaw update` on Windows to avoid EBUSY rename failures when replacing the OpenClaw package tree while OpenClaw is running.

## Best Possible Solution

Review and land or replace #75649 so Windows package updates stage and verify first, then hand the locked tree swap and restart to a detached helper with focused tests, docs, and changelog coverage.

## Reproduction Assessment

Yes, source-reproducible with high confidence: run npm-global `openclaw update` on Windows from the installed package tree while OpenClaw is loaded from that tree. This review did not execute Windows live, but current main still reaches the same live `fs.rename` package-root swap path described by the field logs.

## Solution Assessment

No, current main is not the complete fix because staged install still activates via an in-process package-root rename. The narrow maintainable direction is the active detached Windows handoff in #75649, preserving staged verification before the process releases file locks.
