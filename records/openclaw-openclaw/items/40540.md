## Summary

Keep open: current main and the latest release still route npm-global updates through the staged install plus in-process package-root swap that matches the Windows `EBUSY` report, while the detached Windows handoff remains an unmerged, conflicting implementation PR at https://github.com/openclaw/openclaw/pull/75649.

## What This Changes

The issue reports that Windows npm-global `openclaw update` fails with `EBUSY` during self-update because the running OpenClaw process locks the package root being replaced.

## Best Possible Solution

Rebase, review, prove, and land or replace the detached Windows npm-global update handoff so staging and verification stay intact while the locked package-tree swap happens outside the invoking OpenClaw process.

## Reproduction Assessment

Yes, at source level. The issue includes concrete Windows npm-global `EBUSY` rename logs, and current main still performs the staged package-root swap from the update path that matches the reported failure surface.

## Solution Assessment

No, current main is not enough. The narrow maintainable solution is the detached Windows npm-global handoff already being attempted in the linked PR, with conflicts resolved and live Windows proof before closure.
