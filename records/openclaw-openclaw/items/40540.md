## Summary

Keep open: current main still has a source-reproducible Windows npm-global self-update lock risk because the CLI update path stages and swaps the npm package root from the invoking process, while the active detached-handoff fix remains in an open implementation PR rather than merged behavior.

## What This Changes

The issue asks for `openclaw update` to succeed on Windows npm-global installs instead of failing with an npm `EBUSY` rename while the OpenClaw package tree is locked.

## Best Possible Solution

Review, rebase, prove, and either land or replace the detached Windows npm-global update handoff so the staged package is verified before the locked package-tree swap happens outside the invoking OpenClaw process.

## Reproduction Assessment

Yes, at source level. The issue includes concrete Windows npm-global `EBUSY` rename logs, and current main still routes `openclaw update` through the in-process staged package-root swap that matches the reported failure surface.

## Solution Assessment

No, current main is not enough. The maintainable fix is a focused detached Windows npm-global handoff that keeps staging and verification intact while releasing the invoking OpenClaw process before replacing the locked package tree.
