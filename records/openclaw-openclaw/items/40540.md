## Summary

Keep open: current main still performs the Windows npm-global package-root swap from the active `openclaw update` CLI process, so the reported EBUSY self-update failure remains source-reproducible. The related open fix path is not merged and still needs CLI coverage, rebase/mergeability work, and real Windows proof.

## What This Changes

The issue reports that `openclaw update` on Windows npm-global installs fails with npm `EBUSY`/`rename` while replacing the active OpenClaw package directory.

## Best Possible Solution

Implement or land a focused detached Windows npm-global update flow that stages and verifies the package, releases the invoking CLI process before the locked package-tree swap, preserves restart/status behavior, and includes redacted Windows npm-global proof.

## Reproduction Assessment

Yes. The issue has concrete Windows npm-global `EBUSY` rename logs, and current main still calls `runGlobalPackageUpdateSteps(...)` from the CLI update path before swapping the active package root.

## Solution Assessment

No, current main is not enough. Service-stop and control-plane handoff safeguards are partial because the CLI self-update package-root swap still happens from the active process; the fix needs a detached swap path that covers `openclaw update` itself.
