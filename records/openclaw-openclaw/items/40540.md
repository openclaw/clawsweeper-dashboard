## Summary

Keep open: current main and v2026.6.5 mitigate managed Gateway and dashboard update locks with staged swaps and managed-service handoff, but the standalone Windows npm-global `openclaw update` path is still not proven clear of the CLI self-lock reported here.

## What This Changes

Running `openclaw update` from a Windows npm-global install can fail with npm `EBUSY`/`rename` while replacing the OpenClaw package it is running from.

## Best Possible Solution

Keep the shipped Gateway handoff and staged-swap protections, then close this only after real Windows npm-global shell-update proof passes or the external updater boundary lands.

## Reproduction Assessment

Yes at source/log level, not as a live current-main run: the report has concrete Windows npm `EBUSY` rename output, and current source still performs package-root swaps from the standalone CLI path.

## Solution Assessment

Partially: current managed-service handoff is the right fix for dashboard and supervised Gateway updates, but it is not proven sufficient for the standalone CLI self-update lock; the external-updater boundary remains the better long-term fix.
