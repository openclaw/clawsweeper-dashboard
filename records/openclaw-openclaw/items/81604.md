## Summary

Keep open: current main still has the source-reproducible Telegram CLI bridge bug and this PR is the right narrow direction, but the external PR lacks required real Telegram proof before merge and has a small adapter-forwarding gap to polish.

## What This Changes

The PR adds `resolveCliActionRequest` to the exported Telegram action wrapper and tests the internal `thread-create` to `topic-create` remap.

## Best Possible Solution

Merge a runtime-consistent wrapper forwarding fix with regression coverage on `telegramPlugin.actions` after real Telegram topic-creation proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, at source level. Current main's CLI asks the exported channel plugin for a `thread-create` remap, the exported Telegram wrapper lacks that hook, and the Telegram handler only maps `topic-create`.

## Solution Assessment

Mostly yes. Forwarding the existing resolver through the exported wrapper is the narrow maintainable fix, but it should follow the runtime-override pattern and be proven against a real Telegram topic flow before merge.
