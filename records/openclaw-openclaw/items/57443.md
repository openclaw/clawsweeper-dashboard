## Summary

Keep open: current main and v2026.6.8 still route BytePlus tool schemas through the active normalizer without stripping `patternProperties` unless provider compat declares it, and the same-repo fix attempts are closed while the only open closing PR is a conflicting fork branch.

## What This Changes

The issue asks OpenClaw to prevent BytePlus Ark/OpenAI-compatible tool requests from sending JSON Schema keywords such as `patternProperties` that the provider rejects.

## Best Possible Solution

Add provider-owned schema compatibility for BytePlus and Volcengine/Ark models that strips `patternProperties`, with focused regression tests and live Ark proof; avoid global non-Anthropic cleaning unless maintainers explicitly choose that policy.

## Reproduction Assessment

Yes, source-level: `exec.env` can produce a record schema while the active normalizer only strips Gemini or compat-declared keywords, and BytePlus currently has no compat declaration. I did not run a live BytePlus Ark request in this read-only review.

## Solution Assessment

No, the requested universal scrubber is not clearly the best fix because it changes all non-Anthropic provider schemas. The narrower maintainable path is provider-owned compatibility metadata for BytePlus/Volcengine unless maintainers approve a broader policy.
