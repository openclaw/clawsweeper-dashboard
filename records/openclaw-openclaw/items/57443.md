## Summary

Keep open. Current main still lacks BytePlus/Ark `patternProperties` stripping unless model compat metadata supplies it, BytePlus does not currently supply that metadata, and the open closing PR has not landed.

## What This Changes

The issue asks OpenClaw to strip unsupported JSON Schema keywords such as `patternProperties` for BytePlus Ark/doubao tool schemas so provider requests do not fail with HTTP 400.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/57480 or a replacement lands BytePlus/Ark unsupported-keyword handling in the active schema helper or BytePlus compat metadata, with focused regression coverage and redacted live provider proof.

## Reproduction Assessment

Yes, at source level: `exec.env` still produces the map-style schema surface, current normalization only strips via Gemini or explicit compat metadata, and BytePlus has no compat keyword entry. I did not run a live BytePlus Ark API request.

## Solution Assessment

No, not necessarily as a universal default for every non-Anthropic provider. The narrower maintainable path is to use the active helper or provider-owned BytePlus/Ark compat metadata while preserving Anthropic and existing Gemini behavior.
