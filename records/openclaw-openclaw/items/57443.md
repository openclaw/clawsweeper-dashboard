## Summary

Keep open. Current main still does not strip `patternProperties` for BytePlus/Ark unless the resolved model carries explicit compat metadata, BytePlus does not currently supply that metadata, and https://github.com/openclaw/openclaw/pull/57480 already references this issue but remains open and previously reviewed as needing a port to the active helper plus real provider proof.

## What This Changes

Make tool schema cleanup handle provider-unsupported JSON Schema keywords such as `patternProperties` for BytePlus Ark/doubao tool calls instead of only Gemini-specific paths.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/57480 or a replacement lands with BytePlus/Ark unsupported-keyword handling in the active schema helper or provider compat metadata, focused regression coverage, and redacted live provider proof.

Yes, at source level: current main still routes BytePlus tools through the active normalizer without BytePlus `patternProperties` compat, while exec still exposes the reported map-style `env` schema. I did not run a live BytePlus Ark API request.

## Solution Assessment
