## Summary

Keep open. Current main still leaves BytePlus Ark schemas uncleaned unless explicit compat metadata supplies unsupported keywords, and open PR https://github.com/openclaw/openclaw/pull/57480 already targets this issue with closing syntax, so the issue should wait for that PR to be reviewed, replaced, or closed.

## What This Changes

Ensure BytePlus Ark/doubao tool schemas do not send `patternProperties` so tool-bearing requests stop failing with HTTP 400.

## Best Possible Solution

Ship a focused fix that removes `patternProperties` from outgoing BytePlus/Ark tool schemas, preserves Anthropic full JSON Schema behavior, and adds regression coverage for the `exec.env` path.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: `exec.env` still uses a record schema, current normalization strips `patternProperties` only for Gemini or explicit compat metadata, and BytePlus has no such metadata. I did not run a live Ark request.

## Solution Assessment

Unclear. Universal non-Anthropic cleaning would address the report, but provider-owned BytePlus/Ark compat metadata plus a regression test is the narrower maintainable fix unless maintainers deliberately choose the broader policy.
