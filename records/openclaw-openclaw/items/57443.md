## Summary

Keep open: current main and v2026.6.8 still route BytePlus/Volcengine Ark tool schemas through provider metadata that does not declare `patternProperties` unsupported, while the same-repo fix attempts are closed and the only open closing PR is a conflicting fork branch against obsolete `pi-tools` files.

## What This Changes

The issue asks for BytePlus Ark and similar OpenAI-compatible providers to stop receiving unsupported JSON Schema keywords such as `patternProperties` in tool parameter schemas.

## Best Possible Solution

Add provider-owned schema compatibility for BytePlus and Volcengine/Ark models that strips `patternProperties`, with focused regression tests and live Ark proof; use a global non-Anthropic scrubber only if maintainers explicitly choose that policy.

## Reproduction Assessment

Yes, source-level: `exec.env` uses a record schema, BytePlus exposes Ark OpenAI-compatible routes without a schema-compat hook, and the active normalizer only strips Gemini or declared unsupported keywords. I did not run a live BytePlus Ark request in this read-only review.

## Solution Assessment

No, the universal scrubber proposed in the issue is not clearly the best fix because it changes all non-Anthropic provider schemas. The narrower maintainable path is provider-owned compatibility metadata for BytePlus/Volcengine unless maintainers approve a broader policy.
