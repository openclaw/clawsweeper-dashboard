## Summary

Keep open: current main contains a Mattermost base URL prompt, but the interactive wizard still validates and applies the bot token before the URL answer is collected, matching the reopened maintainer reproduction.

## What This Changes

The issue asks Mattermost onboarding to prompt for and persist the Mattermost server URL during setup instead of relying on `MATTERMOST_URL`.

## Best Possible Solution

Keep Mattermost's non-interactive `--bot-token` plus `--http-url` validation intact, and change the interactive wizard path so it collects the bot token and base URL before validating or persisting the complete Mattermost setup input.

## Reproduction Assessment

Yes from source inspection and maintainer reproduction: `openclaw setup --wizard` can fail immediately after the Mattermost token answer because the wizard applies `{ botToken }` before collecting `httpUrl`. I did not run the live wizard in this read-only environment, so the structured status is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes, the narrow maintainable fix is to stage or apply the Mattermost wizard answers together in the interactive flow. Setting `stepOrder: "text-first"` alone is not sufficient because default text input application would still validate only `{ httpUrl }` against an adapter that requires both fields.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76670. On current main, `mattermostSetupWizard.textInputs` defines the `httpUrl` prompt, but `src/channels/plugins/setup-wizard.ts` runs credential steps first and the default credential apply calls the Mattermost setup adapter with only `{ botToken }`; `extensions/mattermost/src/setup-core.ts` then rejects the incomplete input because it requires both `botToken`/`token` and `httpUrl`. Implement a narrow fix so interactive `openclaw setup --wizard` can collect the Mattermost bot token and base URL and persist both without throwing before the URL prompt. Prefer a plugin-owned staged/finalize apply or a backwards-compatible generic wizard path that validates the combined candidate input; do not weaken non-interactive validation for `openclaw channels add --bot-token --http-url`, do not add a new config option or environment variable, and do not special-case Mattermost in core beyond a generic setup seam if needed. Check https://github.com/openclaw/openclaw/pull/61521 before editing `src/channels/plugins/setup-wizard.ts`; if it has landed, adapt to its seam instead of duplicating deferred-validation work. Add regression coverage that simulates entering a Mattermost token and then a base URL through the wizard and asserts both `botToken` and `baseUrl` are written. Add a single-line user-facing `CHANGELOG.md` fix entry if behavior changes.
