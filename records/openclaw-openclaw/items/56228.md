## Summary

Keep open. Current main still routes Discord reply context through `resolveReplyContext()` and `resolveDiscordMessageText()`, but the text resolver does not inspect Components v2 component trees, so component-only referenced messages can still produce no `ReplyToBody`. The related #56248 patch is useful prior art, but it was closed unmerged and had a noted test defect, so there is no merged fix to close against.

## What This Changes

Request that Discord reply context extract TextDisplay text from Components v2 `referenced_message` payloads when normal content and embeds are empty.

## Best Possible Solution

Implement the fix in the Discord plugin by adding a small recursive Components v2 text extractor for raw/direct message components and forwarded snapshot component trees, wire it into the text fallback only after normal visible text/media/embed paths are empty, and add focused regression tests for a Components v2 referenced message plus forwarded snapshot coverage.

## Reproduction Assessment

Yes. A mocked Discord message whose `referenced_message` has an author, empty `content`/`embeds`, and only Components v2 TextDisplay content will flow through `resolveReplyContext()` into `resolveDiscordMessageText()` and currently return null because components are never inspected.

## Solution Assessment

Yes. A Discord-plugin-local text fallback is the narrowest maintainable solution; it avoids core changes, matches the owner boundary, and should cover both referenced messages and forwarded snapshots rather than only the direct message path.
