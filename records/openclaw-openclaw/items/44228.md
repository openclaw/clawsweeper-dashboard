## Summary

Keep open: current main still compares Windows drive-letter local paths and file URLs as different dedupe keys, and this PR is useful but incomplete because it misses drive-letter case folding and is stale against the current helper split.

## What This Changes

The PR adds Windows drive-letter file URL/local path normalization and a regression test for reply-media dedupe, while also carrying unrelated push/config protocol label updates.

## Best Possible Solution

Land a narrow current-main fix in `src/auto-reply/reply/reply-payloads-dedupe.ts` that canonicalizes Windows drive-letter file URLs and local paths to one stable key, including drive-letter case and slash normalization, with focused tests and a single user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: on current main, `sentMediaUrls` containing `C:\tmp\photo.jpg` will not match a reply media URL of `file:///C:/tmp/photo.jpg` or `file:///c:/tmp/photo.jpg` because the helper returns different normalized strings.

## Solution Assessment

No. The PR's target area is right, but the patch is incomplete for Windows case-insensitive drive letters and targets the pre-split helper shape; the safer solution is a focused current-main patch in `reply-payloads-dedupe.ts` with lowercase/uppercase drive-letter coverage.
