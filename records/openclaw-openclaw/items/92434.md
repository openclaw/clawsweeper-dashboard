## Summary

Current `main` already searches session labels/display names and supports individual deletion through the gateway and Control UI, but the broader CLI/TUI management request remains unresolved; existing rename issues and an open Control UI PR cover only part of this scope.

## What This Changes

Add user-controlled session naming and expose targeted session deletion through consistent CLI, slash-command, TUI, and Control UI workflows.

## Best Possible Solution

Reuse `sessions.patch` labels for user-controlled names and `sessions.delete` for removal, land the existing narrow Control UI work if suitable, and define one canonical follow-up for CLI/TUI operations with safe confirmation and durable-session semantics.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken documented behavior. Source inspection clearly distinguishes the already-supported search and Control UI deletion paths from the missing CLI/TUI management surfaces.

## Solution Assessment

No, not as a single all-surfaces implementation: the best path is to reuse the existing label and delete RPC contracts, let the active Control UI work remain narrow, and separately design safe CLI/TUI exposure.
