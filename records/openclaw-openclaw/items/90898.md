## Summary

Keep open: the patch targets the right Mattermost thread-context gap, but the shared message-tool runner can still write the current child post into `params.replyTo` before the new plugin fallback runs, so existing-thread sends can miss the Mattermost root id. The supplied proof is still mocked/local rather than a live Mattermost post.

## What This Changes

The PR adds Mattermost `threading.buildToolContext`, changes Mattermost send actions to inherit same-channel tool thread context, and adds direct action tests.

## Best Possible Solution

Make the shared message-tool path preserve the Mattermost thread root when both `currentThreadTs` and `currentMessageId` are present, then prove that path with a runner-level regression and redacted live Mattermost evidence.

Yes, source-level. Combine the PR's Mattermost `buildToolContext` output with current `runMessageAction`: the shared helper writes `currentMessageId` into `params.replyTo` before the new Mattermost fallback can use `currentThreadTs`.

No. Adding Mattermost thread context is the right direction, but the best fix must cover the shared runner path and prove that real Mattermost sends use the thread root.
