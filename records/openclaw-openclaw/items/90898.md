## Summary

Keep open: this targets a valid Mattermost thread-delivery problem, but the patch only proves the direct plugin adapter path. In the normal shared message-tool runner, current `main` pre-fills `params.replyTo` from `toolContext.currentMessageId` before Mattermost `handleAction` runs, so the new fallback to `currentThreadTs` can be bypassed when both ids are present.

## What This Changes

The PR adds Mattermost thread-context construction and direct send-action fallback logic so `message(action="send")` can inherit an active Mattermost thread, with regression tests around reply precedence and replyToMode behavior.

## Best Possible Solution

Make the normal shared message-tool path prefer the Mattermost thread root when `MessageThreadId` is present, then cover that path with a runner-level regression test and redacted live Mattermost thread proof.

## Reproduction Assessment

Yes, from source inspection: combine the PR's `buildToolContext` output with current `handleSendAction`, which writes `params.replyTo` from `currentMessageId` before dispatching to Mattermost. I did not run a live Mattermost setup in this read-only review.

## Solution Assessment

No: the direct plugin fallback is plausible, but it is not yet the best fix because the real message-tool runner mutates reply metadata before the plugin sees it. The safer solution is to fix the runner/thread-root precedence and test through `runMessageAction`.
