## Summary

Keep this open: current `main` and the latest release still expose `channel.inbound.*` without the legacy `channel.turn.*` alias, while published older channel plugins can still call the removed alias and lose inbound delivery. A focused open PR already closes this issue, so the right cleanup path is maintainer review of that PR rather than closing or queuing a duplicate fix.

## What This Changes

The issue requests restoring the legacy `core.channel.turn.*` plugin runtime alias after the channel runtime surface moved to `core.channel.inbound.*`.

## Best Possible Solution

Keep this issue open while maintainers review and either land or explicitly reject the linked compatibility-alias PR; if rejected, document the migration/support decision for old channel plugins.

## Reproduction Assessment

Yes, from source and package inspection: current main and `v2026.6.8` expose `channel.inbound` without `channel.turn`, while published older channel packages call `core.channel.turn.run`. I did not run a live Feishu or BlueBubbles credentialed setup.

## Solution Assessment

Yes, if maintainers accept the compatibility contract: restoring `channel.turn` as the same object as `channel.inbound` is the narrow core-boundary fix. Updating only one affected plugin would leave other old compiled channel plugins exposed.
