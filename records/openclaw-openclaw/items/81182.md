## Summary

Keep open: current main still has the reported wait-before-truncate recovery ordering, and the issue is already paired with an open closing PR that should be reviewed rather than closing the issue early.

## What This Changes

The issue requests truncating or pruning accumulated tool results before waiting for full overflow auto-compaction in tool-heavy sessions.

## Best Possible Solution

Land a narrow run-loop fix that tries existing tool-result truncation before explicit overflow compaction for tool-heavy snapshots, while preserving normal compaction fallback for non-tool-heavy overflows.

## Reproduction Assessment

Yes: source inspection on current main shows the overflow recovery branch calls explicit compaction before checking whether tool results are reducible. I did not run a live Telegram overflow because the issue log and current source establish the ordering path.

## Solution Assessment

Yes: reusing the existing tool-result reduction heuristic before explicit compaction is the narrowest maintainable fix. Racing compaction would be a larger concurrency change and is not necessary for the first repair.
