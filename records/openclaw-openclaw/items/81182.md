## Summary

Keep open: current main still has the overflow recovery ordering reported here, and the linked open fixing PR should be reviewed and landed before this issue closes.

## What This Changes

The issue requests changing overflow recovery so tool-heavy contexts try deterministic tool-result truncation before waiting for full LLM auto-compaction timeout.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81190 if normal CI and maintainer checks stay clean, then close this issue after the merge.

## Reproduction Assessment

Yes. Source inspection on current main shows overflow compaction is attempted before the tool-result truncation fallback, and the supplied live logs show a 912840ms compaction timeout followed immediately by truncation of 281 tool results.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing tool-result truncation heuristic before explicit overflow compaction for tool-heavy snapshots, then fall back to compaction when truncation is unavailable or insufficient.
