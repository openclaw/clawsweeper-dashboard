## Label Justifications

- `P1`: The report has real logs for a 15-minute Telegram direct-session stall caused by context overflow recovery ordering.
- `impact:session-state`: The affected behavior is session transcript/context overflow recovery and tool-result pruning.

## Summary

Keep open: current main still tries explicit overflow compaction before the fallback tool-result truncation branch, and https://github.com/openclaw/openclaw/pull/81190 is the open fixing PR with closing syntax and supplied runtime proof.

## What This Changes

Overflow recovery in tool-heavy sessions should try deterministic tool-result truncation before waiting for full LLM auto-compaction timeout.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81190 if normal checks stay clean, then close this issue after the merge.

## Reproduction Assessment

Yes. Source inspection on current main shows compaction runs before the tool-result truncation check, and the supplied live logs show a 912840ms compaction timeout followed immediately by truncation of 281 tool results; I did not rerun a live Telegram overflow.

## Solution Assessment

Yes for the linked fix direction. The narrow maintainable solution is to reuse the existing tool-result truncation heuristic before explicit overflow compaction for tool-heavy snapshots, then fall back to compaction when truncation is unavailable or insufficient.
