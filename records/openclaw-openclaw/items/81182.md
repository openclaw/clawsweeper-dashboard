## Summary

Keep open: current main still reaches explicit overflow compaction before the ordinary tool-result truncation fallback, and the open linked pull request is the correct implementation path to review before closing this issue.

## What This Changes

The issue asks OpenClaw to try cheap tool-result truncation earlier for tool-heavy context overflows instead of waiting for the full auto-compaction timeout first.

## Best Possible Solution

Review and either land or close https://github.com/openclaw/openclaw/pull/81190; the desired end state is truncation-first recovery for tool-heavy overflows with compaction preserved as the fallback.

## Reproduction Assessment

Yes from source: current main calls explicit overflow compaction before the normal tool-result truncation fallback, matching the reported log sequence. I did not run a live Telegram overflow reproduction in this read-only review.

## Solution Assessment

Yes: reusing the existing tool-result pressure and truncation helpers before explicit overflow compaction is the narrow maintainable fix. Racing compaction is a larger concurrency change and is not needed for the first repair.
