## Summary

Keep open: current main still tries explicit overflow compaction before checking whether accumulated tool results can be truncated, so the reported 900s wait-before-truncate path is still plausible and actionable. The related context/compaction issues are adjacent, but none clearly owns this specific recovery-order bug.

## What This Changes

Request that tool-heavy context overflow recovery try cheap tool-result truncation before waiting for the full LLM auto-compaction timeout.

## Best Possible Solution

Make the ordinary overflow-recovery branch reuse the existing tool-result reduction estimate and truncate/retry first when the snapshot is tool-result-heavy, then fall back to compaction if truncation is unavailable or insufficient.

## Reproduction Assessment

Yes from source, though I did not run a live Telegram reproduction. Current main has a clear harness-testable path where a context overflow calls compaction before checking reducible tool results, matching the reported timeout-before-truncation logs.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing tool-result truncation heuristic before explicit overflow compaction for tool-heavy snapshots. Racing compaction is a larger concurrency change and is not needed for the first repair.

Fix https://github.com/openclaw/openclaw/issues/81182. In the embedded Pi overflow recovery path, avoid waiting for the full LLM compaction timeout when the overflow snapshot is dominated by reducible tool results. Reuse the existing tool-result reduction/truncation helpers in src/agents/pi-embedded-runner/tool-result-truncation.ts and the existing precheck routing shape where practical. In src/agents/pi-embedded-runner/run.ts, before explicit overflow compaction for ordinary promptError/assistantError overflows, detect whether attempt.messagesSnapshot has reducible oversized or aggregate tool-result content; if truncation succeeds, retry promptly without calling contextEngine.compact. If there is no reducible tool-result content or truncation does not help, preserve the existing compaction fallback and retry behavior. Do not add a new config option, do not change Telegram transport behavior, and do not weaken non-tool conversation overflow compaction. Add/update focused regression coverage in the overflow compaction loop tests showing truncation happens before compactDirect for tool-heavy overflows and compaction remains the fallback.
