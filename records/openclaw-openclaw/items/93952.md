## Summary

Keep open: the PR targets a real, source-reproducible auth deadlock class, but the current Promise-race deadline shape is not merge-ready because abandoned OAuth/runtime auth work can still mutate credentials or runtime auth state after timeout, and the PR still lacks real gateway/auth-refresh proof.

## What This Changes

The PR adds hard deadlines around runtime auth refresh, cold-start auth preparation, and OAuth refresh critical sections to prevent agent gateway auth deadlocks.

## Best Possible Solution

Keep the deadline/backstop direction, but make timed-out work unable to mutate auth state or credentials after timeout or lock release, give the outer deadline tested headroom, and require redacted real gateway proof before merge.

## Reproduction Assessment

Yes from source: current main awaits runtimeAuthState.refreshInFlight directly, and a never-settling prepareProviderRuntimeAuth leaves the single-flight pending with later waiters blocked. I did not run a live gateway reproduction because this cleanup review is read-only.

## Solution Assessment

No. Hard deadlines are the right mitigation layer, but this implementation uses non-canceling Promise races that can release ownership while abandoned work still mutates auth state; the best fix needs ownership guards or restructuring plus real headroom.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep timed-out OAuth work inside the lock:** `src/agents/auth-profiles/oauth-manager.ts:488-492`
  - body: Wrapping the whole critical section in `withRefreshCallTimeout` only rejects the returned race. The async body can keep running after `withFileLock` releases and the in-process gate opens, so a later `refreshCredential`, `saveOAuthCredentialWithStoreLock`, or mirror write can overlap another refresher and reintroduce the token-rotation race the lock is meant to prevent.
  - confidence: 0.92
- **[P1] Suppress late runtime auth writes after deadline:** `src/agents/embedded-agent-runner/run/auth-controller.ts:228-230`
  - body: `withAuthDeadline(refreshOperation, ...)` rejects the outer promise but does not cancel `refreshOperation`. If the provider hook later resolves, the operation can still set the runtime API key, apply model overrides, and replace `runtimeAuthState` after the single-flight handle was cleared or a retry started.
  - confidence: 0.9
- **[P2] Give the outer auth deadline real headroom:** `src/agents/runtime-auth-refresh.ts:26`
  - body: The PR adds a 150,000 ms held-lock OAuth deadline but leaves the outer runtime auth deadline at 300,000 ms. One caller can wait behind another full in-lock attempt and then need its own full in-lock attempt, leaving no margin for scheduling, release, or jitter before a legitimate contended refresh times out.
  - confidence: 0.86
