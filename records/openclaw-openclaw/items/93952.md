## Summary

Keep open: current main still has the auth-refresh deadlock class and the PR targets the right path, but the patch remains unsafe because its non-cancelling deadlines can let abandoned auth work mutate credentials or runtime state after timeout; it also still lacks real behavior proof.

## What This Changes

The PR adds hard-deadline wrappers around embedded-runner runtime auth refresh, cold-start auth preparation, and OAuth refresh critical sections, with fake-timer and invariant tests.

## Best Possible Solution

Keep the deadline/backstop approach, but make timed-out OAuth and runtime auth work side-effect safe, add deadline headroom, and require redacted real gateway proof before merge.

## Reproduction Assessment

Yes from source, though not with a live gateway run: current main awaits runtimeAuthState.refreshInFlight without a timeout, so a never-settling prepareProviderRuntimeAuth leaves later waiters blocked. The PR's tests model that path, but I did not execute tests in this read-only review.

## Solution Assessment

No. A hard deadline is the right mitigation layer, but this implementation must prevent late side effects after timeout or lock release and add real headroom before it is the best fix.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P1] Prevent timed-out OAuth work from writing after lock release:** `src/agents/auth-profiles/oauth-manager.ts:488-492`
  - body: Wrapping the whole locked callback in withRefreshCallTimeout only rejects the promise returned to withFileLock; it does not cancel the callback. After the rejection releases the global OAuth lock, the abandoned callback can still continue into credential persistence or mirroring after another refresher owns the lock.
  - confidence: 0.92
- **[P1] Suppress late refreshOperation mutations after deadline:** `src/agents/embedded-agent-runner/run/auth-controller.ts:228-230`
  - body: withAuthDeadline(refreshOperation, ...) rejects the single-flight promise but does not cancel refreshOperation. If the provider hook later resolves while generation/profile/source still match, the old operation can still update authStorage, request overrides, and runtimeAuthState after the handle was cleared or a retry is active.
  - confidence: 0.9
- **[P2] Give the outer auth deadline real headroom:** `src/agents/runtime-auth-refresh.ts:26`
  - body: The PR adds a 150,000 ms OAuth in-lock deadline but keeps the outer runtime auth deadline at 300,000 ms. A caller can wait behind one full in-lock attempt and then spend its own full in-lock budget, leaving no margin for scheduling, lock release, or jitter before a legitimate refresh is timed out.
  - confidence: 0.86
