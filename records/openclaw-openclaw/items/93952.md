## Summary

## Best Possible Solution

Keep the single-flight deadline idea, but make timed-out refresh operations unable to mutate auth state later, add strict headroom over the OAuth lock-plus-call budget, and require redacted after-fix gateway proof.

## Reproduction Assessment

Yes from source: current main awaits refreshInFlight without a timeout, and a never-settling prepareProviderRuntimeAuth leaves every later waiter blocked. I did not run a live gateway repro in this read-only review.

## Solution Assessment

No, not yet. A hard deadline is the right layer for the single-flight wedge, but this implementation must suppress late side effects and add real headroom above the OAuth lock/call budget before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Prevent timed-out refreshes from applying late:** `src/agents/embedded-agent-runner/run/auth-controller.ts:221-224`
  - body: `withRuntimeAuthRefreshDeadline` only rejects the outer promise; it does not cancel `refreshOperation`. If the provider hook finally resolves after the deadline, this async body continues and can still call `setRuntimeApiKey`, update the model overrides, and write `runtimeAuthState` after the in-flight handle was cleared or a retry already refreshed the same profile. Add an identity/abandoned check before applying side effects, or otherwise make timed-out work unable to mutate auth state.
  - confidence: 0.9
- **[P2] Leave headroom above the OAuth lock budget:** `src/agents/runtime-auth-refresh.ts:26`
  - body: The new default is exactly `120_000 + 180_000`, but that is the OAuth refresh call timeout plus the stale-lock window, with no scheduling, lock-release, or jitter margin. A legitimate contended refresh that waits near the stale window and then uses the full call budget can lose this race even though it is still within the documented OAuth budget. Make this strictly greater than the combined budget and test that margin.
  - confidence: 0.86
