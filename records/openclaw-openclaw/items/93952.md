## Summary

## What This Changes

## Best Possible Solution

Keep the hard-deadline mitigation only after timeout ownership is side-effect safe, the outer budget has real headroom, and redacted live gateway/auth-refresh proof is added.

## Solution Assessment

No, not yet. A hard deadline is the right mitigation layer, but this implementation must prevent late side effects after timeout or lock release and add deadline headroom before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep timed-out OAuth work from writing after lock release:** `src/agents/auth-profiles/oauth-manager.ts:488-492`
  - body: Wrapping the whole locked callback in `withRefreshCallTimeout` only rejects the promise returned to `withFileLock`; it does not cancel the callback. After that rejection releases the global OAuth lock, the abandoned callback can still continue into credential persistence or mirroring after another refresher owns the lock.
  - confidence: 0.92
- **[P1] Block timed-out refresh operations from mutating auth state:** `src/agents/embedded-agent-runner/run/auth-controller.ts:228-230`
  - body: `withAuthDeadline(refreshOperation, ...)` rejects the single-flight promise but leaves `refreshOperation` running. If the provider hook later resolves while the generation/profile/source still match, the old operation can still update authStorage, request overrides, and runtimeAuthState after the handle was cleared or a retry is active.
  - confidence: 0.9
- **[P2] Give the outer auth deadline real headroom:** `src/agents/runtime-auth-refresh.ts:26`
  - body: The PR adds a 150,000 ms OAuth in-lock deadline but keeps the outer runtime auth deadline at 300,000 ms. A caller can wait behind one full in-lock attempt and then spend its own full in-lock budget, leaving no scheduling, lock-release, or jitter margin before a legitimate refresh is timed out.
  - confidence: 0.86
