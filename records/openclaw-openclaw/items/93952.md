## Label Justifications

- `P1`: The PR targets an agent/gateway auth deadlock that can stop replies while the process still appears healthy.
- `merge-risk: 🚨 auth-provider`: The diff changes OAuth/provider runtime auth refresh timing and can write or reject auth credentials in edge cases.

## Summary

Keep open: current main still has unbounded auth-refresh waits and the PR is a useful fix direction, but the patch is not merge-ready because the timeout wrappers abandon uncancellable work that can later mutate auth state or OAuth stores after handles and locks are released, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds hard-deadline wrappers around embedded-runner runtime auth refresh, cold-start auth preparation, and OAuth refresh critical sections, plus focused regression tests.

## Best Possible Solution

Keep the deadline/backstop direction, but make timed-out work unable to mutate auth state after the deadline or lock release, add tested deadline headroom, and require redacted real gateway proof before merge.

## Reproduction Assessment

Yes from source: current main awaits refreshInFlight without a timeout, and a never-settling prepareProviderRuntimeAuth leaves later waiters blocked. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. A hard deadline is the right mitigation layer, but this implementation must prevent abandoned work from mutating state after timeout or lock release and must add real headroom before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep timed-out OAuth work from writing after lock release:** `src/agents/auth-profiles/oauth-manager.ts:488-491`
  - body: Wrapping the whole `withFileLock` callback in `withRefreshCallTimeout` only rejects the returned promise; the callback keeps running. Once the race rejects, `withFileLock` releases the global OAuth lock, but the abandoned callback can later continue into `saveOAuthCredentialWithStoreLock` or `mirrorRefreshedCredentialIntoMainStore` after another refresher owns the lock, which reintroduces the refresh-token race the lock is supposed to prevent.
  - confidence: 0.92
- **[P1] Suppress late runtime auth mutations after deadline:** `src/agents/embedded-agent-runner/run/auth-controller.ts:228-230`
  - body: `withAuthDeadline(refreshOperation, ...)` rejects only the outer promise; it does not cancel `refreshOperation`. If the provider hook eventually resolves, the operation can still call `setRuntimeApiKey`, apply model overrides, and replace `runtimeAuthState` after the in-flight handle has been cleared or a retry has already refreshed the same profile.
  - confidence: 0.9
- **[P2] Give the outer auth deadline real headroom:** `src/agents/runtime-auth-refresh.ts:26`
  - body: The PR adds a 150,000 ms held-lock OAuth critical-section deadline but leaves the outer runtime auth deadline at 300,000 ms. One caller can wait behind another full in-lock attempt and then need its own full in-lock attempt, leaving 0 ms for scheduling, release, or jitter before a legitimate contended refresh loses the race.
  - confidence: 0.86
