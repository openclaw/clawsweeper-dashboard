## Summary

Keep open: the reported gateway/auth deadlock is source-reproducible on current main, but this PR is not merge-ready because it lacks real behavior proof and the timeout wrappers can still let abandoned auth work mutate credential/session state after the deadline boundary.

## What This Changes

Adds hard-deadline wrappers around embedded-runner runtime auth refresh, cold-start auth preparation, and the OAuth refresh critical section, plus fake-timer and invariant tests.

## Best Possible Solution

Keep the deadline/backstop concept, but make timed-out OAuth and runtime auth work side-effect safe, add deadline headroom, and require redacted real gateway proof before merge.

## Reproduction Assessment

Yes from source: current main awaits a shared runtimeAuthState.refreshInFlight promise without a timeout, so a never-settling provider runtime auth hook blocks later refresh callers. The PR's fake-timer tests model that path, but this read-only review did not execute tests or a live gateway repro.

## Solution Assessment

No. A hard deadline is the right mitigation layer, but this implementation is not yet the best fix because timed-out work is abandoned without cancellation or side-effect suppression and the outer budget has no headroom.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Prevent timed-out OAuth work from writing after lock release:** `src/agents/auth-profiles/oauth-manager.ts:488-492`
  - body: Wrapping the whole locked callback in `withRefreshCallTimeout` makes the promise given to `withFileLock` reject, but it does not cancel the callback. `withFileLock` then releases the global lock while the abandoned callback can continue to save or mirror refreshed credentials after another refresher owns the lock, reopening the refresh-token race the lock protects.
  - confidence: 0.92
- **[P1] Suppress late refreshOperation mutations after deadline:** `src/agents/embedded-agent-runner/run/auth-controller.ts:228-230`
  - body: `withAuthDeadline(refreshOperation, ...)` rejects `refreshPromise`, but it does not cancel `refreshOperation`. If the provider hook later resolves while generation/profile/source still match, the old operation can still update authStorage, request overrides, and runtimeAuthState after the handle was cleared or a retry is active.
  - confidence: 0.9
- **[P2] Give the outer auth deadline real headroom:** `src/agents/runtime-auth-refresh.ts:26`
  - body: The PR adds a 150,000 ms OAuth in-lock deadline but keeps the outer runtime auth deadline at 300,000 ms. A caller can wait behind one full in-lock attempt and then spend its own full in-lock budget, leaving no margin for scheduling, lock release, or jitter before a legitimate refresh is timed out.
  - confidence: 0.86
