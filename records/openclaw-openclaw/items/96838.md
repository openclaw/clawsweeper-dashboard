## Label Justifications

- `P2`: This is a normal user-facing WebChat/channel Stop bug with limited blast radius, but the current PR still has merge-blocking targeting and proof gaps.
- `merge-risk: 🚨 session-state`: The diff changes which hidden active and pending session runs can be stopped and can mis-associate a broad Stop with a channel session.

## Summary

Keep open: the latest head still introduces broad-session Stop fallbacks that can target hidden channel runs without an explicit selected channel session, and the external PR still has only unit/static validation rather than real WebChat/WeChat behavior proof.

## What This Changes

The PR changes WebChat and Gateway `chat.abort` targeting so explicit channel-scoped sessions, and some broad Stop requests with a single hidden channel run, can abort hidden active or pending channel runs.

## Best Possible Solution

Keep the explicit channel-key Gateway exception, remove or narrow both broad-session fallback paths unless maintainers explicitly accept that policy, then add redacted WebChat/WeChat runtime proof and coordinate with the broader Stop recovery PR.

## Reproduction Assessment

Yes, at source level: current `main` filters hidden active and pending runs before runId-less session-key abort matching, while WebChat can send runId-less aborts. I did not run a live WebChat/WeChat reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The explicit channel-key exception is the right boundary for the narrow bug, but broad UI/Gateway guessing should be removed or explicitly accepted by maintainers before merge.
