## Label Justifications

- `P2`: This is a normal user-facing WebChat/channel abort bug with limited blast radius, but the current PR still has a session-targeting blocker.
- `merge-risk: 🚨 session-state`: The diff changes which hidden active and pending session runs can be stopped and can mis-associate a broad Stop with a channel session.
- `merge-risk: 🚨 message-delivery`: If the broad-session fallback chooses the wrong channel session, merging the PR can interrupt an unrelated in-flight channel response.

## Summary

Keep open: the gateway exact-session-key fix is plausible, but the new UI fallback can retarget a broad main-session Stop to a channel session the user did not explicitly select, and the PR still lacks real WebChat/WeChat proof.

## What This Changes

The PR changes WebChat and Gateway `chat.abort` targeting so explicit channel-scoped session keys, and one UI single-active-channel fallback, can abort hidden active and pending channel runs.

## Best Possible Solution

Keep the gateway-side explicit-key exception, remove or narrow the broad-session UI guessing to an explicit selected channel-session signal, then add redacted real WebChat/WeChat Stop proof and coordinate with the broader Stop recovery path.

## Reproduction Assessment

Yes at source level: current main filters hidden registered and pending runs before session-key abort matching, while WebChat can send a runId-less `chat.abort`. The new review finding is also source-reproducible from the PR test that starts at `agent:main` and expects a WeChat channel key abort.

## Solution Assessment

No, not as submitted. The gateway explicit-key exception is the right owner boundary for the narrow bug, but the UI fallback should not guess a channel session from a broad main session without an explicit selected-session signal.
