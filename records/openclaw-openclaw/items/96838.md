## Summary

Keep open: the gateway-side explicit channel-key exception is a plausible narrow fix, but the PR still retargets a broad WebChat Stop to a guessed channel session and lacks real WebChat/WeChat behavior proof.

## What This Changes

The PR changes WebChat abort targeting and Gateway `chat.abort` resolution so explicit channel-scoped session keys can abort hidden active and pending channel runs, with focused unit coverage.

## Best Possible Solution

Keep the gateway explicit-key exception, remove or narrow broad-session UI guessing to an explicit selected-channel signal, then add redacted WebChat/WeChat proof and coordinate with the broader Stop recovery path.

## Reproduction Assessment

Yes, source-level: current main filters hidden active and pending runs before runId-less session-key abort matching, while WebChat can send runId-less aborts. The PR finding is also source-reproducible from the new test that starts at `agent:main` and expects a channel-key abort.

## Solution Assessment

No, not as submitted. The Gateway explicit-key exception is the right boundary for the narrow bug, but the UI should not guess a channel session from a broad main session without an explicit selected-session signal.
