## Summary

Keep open: this is a useful Gateway feature, but it is not merge-ready because the endpoint advertises every configured TTS provider while only some providers actually honor the OpenAI `response_format` override.

## What This Changes

The branch adds a gated Gateway `POST /v1/audio/speech` OpenAI-compatible TTS endpoint, lists configured TTS providers as `tts/<provider>` models when enabled, updates OpenAI-compatible speech providers for per-request formats, and adds docs/tests.

## Best Possible Solution

Make the audio speech endpoint advertise and accept only provider/format combinations it can enforce, or add a provider capability/mapping contract that translates OpenAI formats before listing those providers.

## Reproduction Assessment

Yes for the review finding by source inspection: a listed non-OpenAI provider can receive `response_format` as `responseFormat` while its implementation reads `outputFormat`, so the requested format is not enforced. I did not run the PR branch because this review is read-only.

## Solution Assessment

No: the endpoint shape is plausible, but the current provider listing and format override contract are too broad for what the provider implementations can guarantee. The safer fix is to gate/list only enforceable providers or add an explicit provider format capability mapping.
