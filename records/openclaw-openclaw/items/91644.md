## Summary

Keep open: the opt-in Gateway TTS endpoint is useful and the proof covers the OpenAI-compatible happy path, but the patch still advertises all configured TTS providers without a settled provider capability contract for OpenAI speech parameters.

## What This Changes

The PR adds a disabled-by-default Gateway `POST /v1/audio/speech` endpoint, lists configured TTS providers as `tts/<provider>` models when enabled, adds per-request speech format overrides for OpenAI-compatible providers, and updates docs/tests.

## Best Possible Solution

Merge only after the endpoint's model listing, request validation, and provider overrides share one explicit capability contract so every advertised `tts/<provider>` either honors or rejects OpenAI speech parameters consistently.

## Reproduction Assessment

Yes. Source inspection shows the endpoint forwards `responseFormat`, while listed sibling providers such as Azure Speech and ElevenLabs read `outputFormat`, so requested formats can be ignored without running the branch.

## Solution Assessment

No. The Gateway endpoint is a plausible layer, but the best fix needs an explicit provider capability contract before listing all configured TTS providers as OpenAI-compatible models.

- **[P2] Enforce the requested format for every listed TTS provider:** `src/gateway/audio-speech-http.ts:176`

  - confidence: 0.91
