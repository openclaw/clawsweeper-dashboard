## Summary

Keep this PR open, but do not merge it yet: the implementation is substantial and CI is green, while the advertised OpenAI compatibility is broken because standard speech model values are interpreted as provider IDs; the new public plugin capability also needs an explicit maintainer contract decision.

## What This Changes

The PR adds a disabled-by-default `/v1/audio/speech` gateway endpoint, advertises compatible TTS providers through `/v1/models`, extends the public speech-provider contract, forwards request overrides, and adds tests and documentation.

## Best Possible Solution

Keep OpenAI `model` semantics intact, resolve the provider from the request agent's canonical TTS configuration, pass the requested speech model through as a provider override when supported, and add any provider-selection extension separately only after maintainers approve the public plugin capability design.

## Reproduction Assessment

Yes for the blocking compatibility defect: source inspection shows a standard request with `model: "tts-1"` or `model: "gpt-4o-mini-tts"` deterministically follows provider lookup and returns the unavailable-provider error.

## Solution Assessment

No. Reusing the existing TTS runtime is sound, but replacing the standard model contract with provider selection is not the narrowest maintainable compatibility bridge; provider selection should remain canonical config and the model should reach the chosen provider.
