## Summary

Keep this open: current main still treats fallback as provider/model candidate selection for the same agent run state, and this issue is the canonical thread for choosing a fallback context/workspace contract.

## What This Changes

Request configurable fallback attempts that can use a different agent/workspace or model-specific context profile instead of reusing the primary agent context.

## Best Possible Solution

Define one first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests together.

## Reproduction Assessment

Yes for the current limitation by source inspection: config schemas lack fallbackAgent/fallbackWorkspace/context profiles, ModelCandidate only carries provider/model, and fallback runs reuse the original workspace/session/agent state. This is a feature gap, not a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should first choose whether the durable contract is agent switching, workspace override, context profiles, context budgets, or fallback runbook injection.
