## Summary

Keep open. Current main still treats fallback as provider/model candidate selection, with no first-class fallback agent, workspace, skills, session, or context-profile switch; recent issue discussion also explicitly kept this as maintainer design work.

## What This Changes

Request a per-agent fallback mechanism so model failover can switch to another agent/workspace or use a smaller model-specific context profile.

## Best Possible Solution

Define a first-class fallback-context contract, then update config schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests around alternate context without corrupting original agent state.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the proposed fallback agent/workspace/context-profile fields are absent from strict config surfaces, and fallback execution only carries provider/model while reusing the original run context.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should first choose between agent switching, workspace override, model-specific context profiles, and plugin-hook semantics before implementation.
