## Label Justifications

- `P2`: Fallback quality and graceful degradation are materially affected, but this is a bounded feature/design issue rather than an urgent core outage.
- `impact:session-state`: Changing workspace or agent context during fallback can alter transcript, memory, skills, and session-state association.
- `impact:auth-provider`: The requested behavior is selected as part of provider/model fallback routing and may interact with auth/provider choice.

## Summary

Keep open: current main and the latest release still implement fallback as provider/model candidate selection within one caller-supplied agent, workspace, agentDir, skills snapshot, and session context. The requested behavior remains useful but needs a maintainer product decision before implementation.

## What This Changes

The issue requests agent-level or model-aware fallback context so smaller fallback models can use a separate workspace, bootstrap profile, or agent instead of inheriting the primary agent prompt.

## Best Possible Solution

Define a typed fallback context profile that can reduce bootstrap files, prompt budgets, skills, and tools while preserving the original agent/session identity by default; add cross-agent handoff only if maintainers explicitly define state, auth, transcript, and delivery ownership.

## Reproduction Assessment

Yes, at source level. Current-main types, strict schemas, docs, and fallback execution show retries vary provider/model only while reusing the caller's agent, workspace, agentDir, and session context.

## Solution Assessment

No implementation is under review yet. The issue is valid, but maintainers should choose a narrower fallback context-profile contract before considering cross-agent handoff semantics.
