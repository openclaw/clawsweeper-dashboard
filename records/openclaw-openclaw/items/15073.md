## Summary

Keep this open. Current main still implements fallback as provider/model candidate selection while reusing the same agent run context, and the recent maintainer/Codex comments kept this as design work rather than cleanup closure.

## What This Changes

Request an agent-level or model-fallback context mechanism so fallback attempts can use a different agent, workspace, or model-specific context profile instead of the primary agent context.

## Best Possible Solution

Define a first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests for alternate fallback context without corrupting the original agent state.

## Reproduction Assessment

Yes. Static source reproduction is high confidence: strict config surfaces lack the proposed fields, and fallback execution only carries provider/model while reusing the original run context.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should first choose between agent switching, workspace override, and model-specific context-profile semantics.
