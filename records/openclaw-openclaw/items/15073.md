## Summary

Keep open: current main and v2026.6.6 still treat fallback as provider/model candidate selection inside the original agent/session context. The request remains valid, but the next step is a maintainer API/product decision, not cleanup closure or an autonomous fix PR.

## What This Changes

The issue requests fallback behavior that can switch to a different agent or reduced workspace/context profile when the primary model fails.

## Best Possible Solution

Keep this issue open for a first-class fallback context design: choose a typed context-profile/budget mechanism or explicitly define cross-agent handoff ownership for state, auth, transcript, workspace, skills, and delivery.

## Reproduction Assessment

Yes, at source level. Current-main types, strict schemas, docs, and fallback execution show retries vary provider/model only while reusing the caller's agent, workspace, agentDir, and session context.

## Solution Assessment

No implementation is under review yet. The issue is valid, but maintainers should choose a fallback context-profile contract before considering cross-agent handoff semantics.
