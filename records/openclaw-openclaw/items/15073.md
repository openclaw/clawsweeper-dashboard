## Summary

Keep this open: current main still implements fallback as provider/model candidate selection, with no first-class way to switch fallback agent identity, workspace, session file, skills snapshot, or model-specific context profile. The related context-budget/profile issues were closed as duplicates of this issue, and recent maintainer/Codex discussion kept this as the canonical design thread.

## What This Changes

Request an agent- or model-specific fallback context/workspace so small fallback models do not receive the primary agent's full prompt and workspace context.

## Best Possible Solution

Define a first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests for alternate fallback context without corrupting original agent state.

## Reproduction Assessment

Yes. Static source inspection on current main shows strict config surfaces lack the requested fields, fallback candidates carry only provider/model, and retry callers reuse the original workspace/session/agent state.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should first choose between agent switching, workspace override, and model-specific context-profile or budget semantics.
