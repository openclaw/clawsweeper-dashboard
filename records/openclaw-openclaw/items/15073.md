## Summary

Current main still models fallback as provider/model candidate selection and does not provide fallback-specific agent, workspace, skills, or context-profile selection. The recent issue discussion also explicitly kept this as design work rather than cleanup closure.

## What This Changes

Request a per-agent fallback mechanism so model fallback can switch to a different agent/workspace or otherwise use a smaller model-specific context profile.

## Best Possible Solution

Design a first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests so fallback retries can use alternate context without corrupting original agent state.

## Reproduction Assessment

Yes. Static reproduction is high confidence: the proposed fallback agent/workspace/context-profile fields are absent from strict config surfaces, and fallback execution only carries provider/model while reusing the original run context.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should first choose between agent switching, workspace override, and model-specific context-profile semantics before implementation.
