## Label Justifications

- `P2`: This is a valid multi-model fallback improvement with real user impact, but it is a design-level feature gap rather than an urgent broken workflow.
- `impact:session-state`: The requested behavior changes how fallback attempts use agent identity, workspace context, session files, skills snapshots, and transcript state.
- `impact:auth-provider`: Fallback attempts currently vary provider/model and auth profile behavior, and any agent-switching design must preserve provider and credential routing semantics.

## Summary

Keep this open: current main still implements fallback as provider/model candidate selection inside the same agent/session/workspace context, and the issue remains a public config/API design request rather than cleanup-close material.

## What This Changes

Request an option for model fallback to use an alternate agent, workspace, or model-specific context profile instead of reusing the primary agent context.

## Best Possible Solution

Define one first-class fallback-context contract, then update config types, strict schemas, docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests together.

## Reproduction Assessment

Yes for a source-level reproduction: the proposed fields are absent from strict schemas, fallback candidates carry only provider/model, and queued runs reuse one agent/session/workspace context. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose the durable contract across agent switching, workspace overrides, context profiles, context budgets, and runbook semantics before implementation.
