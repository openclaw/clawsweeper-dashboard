## Summary

Keep this open: current main still treats fallback as provider/model selection on the same agent/session context, and the linked context-profile/runbook reports have been closed into this issue as the canonical design thread. This needs maintainer API and session-state design rather than cleanup closure.

## What This Changes

Feature request for model fallback attempts to switch to fallback-specific agent, workspace, or context profile instead of reusing the primary agent context.

## Best Possible Solution

Define one first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests together.

## Reproduction Assessment

Yes for the current limitation: source inspection shows the proposed fallback agent/workspace/context fields are absent, fallback candidates only carry provider/model, and context injection has no fallback model-change input. This is a feature gap, not a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose the durable contract across agent switching, workspace override, context profiles, context budgets, fallback runbooks, and plugin hooks before implementation.
