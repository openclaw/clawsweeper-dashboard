## Summary

Keep this open: current main still treats fallback as provider/model candidate selection on the same agent run state, with no fallback-specific agent, workspace, or context profile. The related context-profile and runbook issues have been closed back into this issue, so this remains the canonical design thread rather than cleanup material.

## What This Changes

The issue requests fallback behavior that can switch to a different agent workspace/context, or an equivalent per-model bootstrap profile, when a primary model falls back to a smaller model.

## Best Possible Solution

Define one first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests together.

## Reproduction Assessment

Yes for the current limitation: source inspection shows config schemas lack fallbackAgent/fallbackWorkspace/context profiles, ModelCandidate only carries provider/model, and fallback attempts reuse the original workspace/session/agent state. This is a feature gap, not a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose the durable contract across agent switching, workspace override, context profiles, context budgets, fallback runbook injection, and plugin hooks before implementation.
