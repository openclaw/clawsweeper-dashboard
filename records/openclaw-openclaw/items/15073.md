## Summary

Keep open: current main and the latest release still implement fallback as provider/model candidate retry inside the original agent run context, while this issue asks for a maintainer-approved fallback agent/workspace/context-profile contract. The live discussion already kept it open for design, and the source shape has not materially changed since that review.

## What This Changes

The issue requests a way for model fallback to switch to a different agent, workspace, or model-specific context profile instead of reusing the primary agent's full context.

## Best Possible Solution

Keep this issue open for a first-class fallback context design that chooses fallbackAgent/fallbackWorkspace semantics or a narrower model-specific context profile/budget mechanism, then updates config, strict schemas, docs, candidate metadata, session plumbing, and regression tests together.

## Reproduction Assessment

Yes, at source level. Current-main types, strict schemas, docs, fallback candidate resolution, and attempt execution show retries vary provider/model only while reusing the caller's agent/workspace context.

## Solution Assessment

No implementation is under review yet. The issue is valid, but maintainers should choose a fallback context-profile or cross-agent handoff contract before any repair PR picks public config and session semantics.
