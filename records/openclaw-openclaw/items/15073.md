## Summary

Keep open. Current main still models fallback as provider/model candidate selection that reuses the existing run context, and the recent issue discussion explicitly keeps this as design work rather than cleanup closure.

## What This Changes

Allow fallback attempts to use a different per-agent or model-specific workspace/context so smaller local fallback models are not fed the primary agent's full bootstrap prompt.

## Best Possible Solution

Design a first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests so fallback retries can use alternate context without corrupting the original agent state.

## Reproduction Assessment

Yes. Static reproduction is high confidence: the proposed fallbackAgent, fallbackWorkspace, contextProfiles, bootstrapFiles, and per-model bootstrap-budget fields are absent from strict config surfaces, and fallback execution only carries provider/model while reusing the original run context.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be valid, but maintainers should first choose between agent-switching, workspace override, and model-specific context-profile semantics before implementation.
