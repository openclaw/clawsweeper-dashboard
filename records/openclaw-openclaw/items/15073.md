## Summary

Keep this open: current main still treats fallback as provider/model candidate selection on the same agent/session context, and the related context-budget/profile/runbook reports have been closed into this issue as the canonical design thread. This needs maintainer API and session-state design, not cleanup closure.

## What This Changes

Request adding fallback-specific agent/workspace/context selection so model fallback can use a smaller or different agent context instead of the primary agent context.

## Best Possible Solution

Define one first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests together.

## Reproduction Assessment

Yes for the source-level limitation: current schemas reject the proposed fields, fallback candidates only carry provider/model, and the reply runner keeps the same run context. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose the durable contract across agent switching, workspace override, context profiles, context budgets, fallback runbooks, and plugin hooks before implementation.
