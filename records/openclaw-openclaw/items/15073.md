## Label Justifications

- `P2`: This is a normal-priority resilience and output-quality enhancement for heterogeneous fallback chains, not an emergency or confirmed urgent regression.
- `impact:session-state`: The request changes which workspace, prompt, skills, and potentially agent identity are associated with an in-progress session retry.
- `impact:auth-provider`: The behavior is activated by provider/model failover and a cross-agent design could change which agent-scoped credentials are available.

## Summary

Keep open: current main and v2026.6.5 still select fallback attempts by provider/model only, while the requested fallback-specific workspace/context behavior remains the canonical unresolved product design. Existing per-agent bootstrap and context limits can reduce prompt size globally for an agent, but they do not change automatically when a fallback candidate activates.

## What This Changes

Add fallback-specific agent workspace or reduced context controls so smaller fallback models do not inherit the primary model’s full prompt and tool context.

## Best Possible Solution

Define one canonical fallback-context contract at the retry boundary that reuses existing bootstrap, skills, tool, and context-budget primitives while preserving the original agent/session identity by default; add cross-agent fallback only if maintainers explicitly specify credential, transcript, delivery, and session ownership semantics.

## Reproduction Assessment

Yes at the source level: current main has model-only fallback candidates and agent-scoped context controls, so a fallback attempt cannot select alternate workspace or context. A live Claude-to-Ollama fallback was not executed in this read-only review.

## Solution Assessment

No single proposed shape is yet proven best. A fallback-specific context profile is narrower and safer than switching full agent identity, but maintainers must decide whether the contract covers workspace files, tools, skills, budgets, a runbook, or a deliberate combination.
