## Summary

Keep this open. Current main still implements fallback as provider/model candidate selection, while the request needs a new fallback-context contract; the recent maintainer/Codex discussion also explicitly kept this as design work rather than cleanup closure.

## What This Changes

The issue requests an agent-level fallback mode that can switch to a different agent/workspace/context profile when the primary model fails.

## Best Possible Solution

Define a first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests for alternate fallback context without corrupting original agent state.

## Reproduction Assessment

Yes. Static reproduction is high confidence: strict config surfaces lack the proposed fields, and fallback execution only carries provider/model while reusing the original run context.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should first choose between agent switching, workspace override, and model-specific context-profile semantics.
