## Summary

Keep this open: current main still implements model fallback as provider/model candidate selection on the same agent run state, with no fallback-specific agent, workspace, or context profile; related context-profile issues have been closed into this canonical design thread rather than replacing it.

## What This Changes

The issue requests fallback-specific agent/workspace/context configuration so a fallback model can use a smaller dedicated context instead of the primary agent’s prompt and workspace.

## Best Possible Solution

Define one first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests together.

## Reproduction Assessment

Yes for the current limitation: source inspection shows config schemas lack fallbackAgent/fallbackWorkspace/context profiles, ModelCandidate only carries provider/model, and agent-command reuses the original workspace/session/agent state. This is a feature gap, not a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose whether the durable contract is agent switching, workspace override, context profiles, context budgets, or fallback runbook injection before implementation.
