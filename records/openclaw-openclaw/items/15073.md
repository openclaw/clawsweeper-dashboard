## Summary

Keep this open: current main still treats fallback as provider/model candidate selection on the same agent/session context, while related context-profile and runbook requests have been closed into this as the canonical design thread.

## What This Changes

The issue requests an agent-level fallback mode so a fallback model can run with its own agent, workspace, bootstrap context, and simpler instructions.

## Best Possible Solution

Define one first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests together.

## Reproduction Assessment

Yes for the source-level limitation: current schemas reject the proposed fields, fallback candidates only carry provider/model, and queued runs keep one agent/session/workspace context. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose the durable contract across agent switching, workspace override, context profiles, context budgets, fallback runbooks, and plugin hooks before implementation.
