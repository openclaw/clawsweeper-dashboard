## Summary

Keep this open: current main and the latest release still implement model fallback as provider/model candidate selection, while the requested fallback-agent/workspace/context behavior remains a missing core design decision.

## What This Changes

Request to let model fallback attempts use a different agent, workspace, or context profile instead of reusing the primary agent context.

## Best Possible Solution

Keep this as the canonical fallback-context design issue, then implement one first-class contract across config types, strict schemas, docs, fallback metadata, reply-runner/session plumbing, and regression tests.

## Reproduction Assessment

Yes for the missing capability at source level: current model config, strict schemas, fallback candidate records, docs, and the fallback runner only carry provider/model selection. I did not run a live outage or rate-limit scenario to reproduce degraded model output.

## Solution Assessment

Unclear until maintainers choose the public contract. The maintainable path is a designed fallback-context feature rather than a narrow config-only patch that could blur agent, workspace, session, skills, and auth boundaries.
