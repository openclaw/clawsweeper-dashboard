## Summary

Keep open: current main and v2026.6.6 still treat fallback as provider/model retry selection inside the original agent/session context. The request remains valid, but the next step is a maintainer API/product decision, not cleanup closure or an autonomous fix PR.

## What This Changes

The issue requests a fallback mode that can switch to another agent or model-specific context/workspace profile when a configured model fallback is used.

## Best Possible Solution

Keep this open for a first-class fallback context design that explicitly chooses typed context-profile/budget semantics or cross-agent handoff ownership for state, auth, transcript, workspace, skills, and delivery.

## Reproduction Assessment

Yes, at source level. Current-main types, strict schemas, docs, and fallback execution show retries vary provider/model only while reusing the caller's agent, workspace, agentDir, session file, and skills snapshot.

## Solution Assessment

No implementation is under review yet. The issue is valid, but maintainers should choose a fallback context-profile contract before considering cross-agent handoff semantics.
