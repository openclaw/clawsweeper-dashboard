## Summary

Keep this open: current main and the latest shipped tag still model fallback as provider/model selection only, while this issue remains the canonical design thread for alternate fallback context after related context-budget/profile requests were closed into it.

## What This Changes

The issue requests a first-class fallback mechanism that can switch to another agent, workspace, or context profile when a primary model falls back to a smaller model.

## Best Possible Solution

Define a first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests for alternate fallback context without corrupting original agent state.

## Reproduction Assessment

Yes for the current limitation: static source inspection on current main and v2026.5.7 shows strict config surfaces lack the requested fields, fallback candidates carry only provider/model, and retry plumbing reuses the original workspace/session/agent state.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should first choose between agent switching, workspace override, and model-specific context-profile or budget semantics.
