## Label Justifications

- `P2`: This is a valid multi-model fallback improvement with real user impact, but it is a design-level feature request rather than an urgent broken existing contract.
- `impact:session-state`: The requested behavior changes fallback use of agent identity, workspace context, session files, skills snapshots, and transcript state.
- `impact:auth-provider`: Fallback attempts already change provider/model routing, and any agent-switching design must preserve auth-profile and credential routing semantics.

## Summary

Keep open: current main still treats fallback as provider/model candidate selection while reusing the same agent/session/workspace context, and related reports have been closed in favor of this issue as the canonical fallback-context design thread.

## What This Changes

The issue requests fallback attempts that can switch to a different agent, workspace, or context profile instead of sending the primary agent’s full prompt context to the fallback model.

## Best Possible Solution

Define one first-class fallback-context contract and implement it consistently across config schema/help/docs, fallback candidate metadata, run/session context, auth routing, and regression coverage.

## Reproduction Assessment

Yes for a source-level reproduction: current schemas lack the proposed fields, fallback candidates carry only provider/model, and the runner reuses the same session/workspace/agent context. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose the durable contract across agent switching, workspace overrides, context profiles, context budgets, and runbook semantics before implementation.
