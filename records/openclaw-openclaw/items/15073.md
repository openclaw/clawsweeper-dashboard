## Summary

Keep open: current main and v2026.6.8 still model fallback as provider/model candidate selection inside the original agent/session context, with no supported fallback agent, workspace, or model-specific context profile. The request remains valid, but the next step is a maintainer API/product decision rather than cleanup closure or an autonomous fix PR.

## What This Changes

Request an option for model fallback to switch to a different agent/workspace or use model-specific context limits/files so smaller fallback models are not given the primary agent's full context.

## Best Possible Solution

Keep this issue open for one first-class fallback context contract that explicitly chooses typed context-profile/budget semantics or cross-agent handoff ownership for state, auth, transcript, workspace, skills, and delivery.

## Reproduction Assessment

Yes, at source level. Current-main types, strict schemas, docs, and fallback execution show retries vary provider/model only while reusing the caller's agent, workspace, agentDir, session file, and skills snapshot.

## Solution Assessment

No implementation is under review yet. The issue is valid, but maintainers should choose between fallbackAgent, fallbackWorkspace, and model-context-profile semantics before code is queued.
