## Summary

Keep this open. Current main and the latest release still implement fallback as provider/model candidate selection only, with no config or runtime contract for switching agent identity, workspace, agentDir, session file, skills snapshot, or context profile; the recent maintainer/Codex discussion also kept this open for design follow-up.

## What This Changes

The issue requests a way for model fallback to use a different agent/workspace/context profile instead of reusing the primary agent’s full context.

## Best Possible Solution

Choose a first-class fallback-context contract, then update config types, strict schemas, docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests so fallback retries can use alternate context without corrupting the original session.

## Reproduction Assessment

Yes for the missing capability: source inspection shows current main only supports provider/model fallback strings and rejects the proposed fallback-agent/workspace/profile config shape. A live degraded-output reproduction was not run, but the config/runtime gap is clear from current source.

## Solution Assessment

Unclear until maintainers choose the public contract. The safest direction is not a narrow config-only patch, but a designed fallback-context feature that keeps agent, workspace, session, skills, and auth boundaries coherent.
