## Summary

Keep this open: current main and the latest release still treat fallback as provider/model candidate selection on the same agent run state, and this issue remains the canonical design thread for fallback-specific context/workspace behavior.

## What This Changes

Request an option for model fallback to switch to a different agent, workspace, or context profile instead of reusing the primary agent context.

## Best Possible Solution

Define one first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests together.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main and v2026.5.12 shows fallback candidates carry only provider/model and attempts reuse the original workspace/session/agent state. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should first choose whether the durable contract is agent switching, workspace override, context profiles, or context-budget policy.
