## Label Justifications

- `P2`: Heterogeneous fallback chains can produce unusable output for affected users, but the problem is limited to configured multi-model setups and is not a core outage.
- `impact:session-state`: The requested fallback behavior directly affects which workspace, prompt, skills, transcript, and agent state are associated with an active session.
- `impact:auth-provider`: The requested context transition is triggered by provider/model fallback selection and must remain aligned with the selected model and its runtime.

## Summary

Current main still retries a fixed agent/session context across provider/model candidates, so the requested capability remains necessary; the unresolved choice is the public contract for reduced context versus full agent switching.

## What This Changes

Add a fallback mechanism that can use a different agent workspace or model-specific reduced context when the primary model fails.

## Best Possible Solution

Adopt one first-class, attempt-scoped fallback context profile that preserves the originating session identity while explicitly controlling bootstrap files, prompt budgets, skills, and tools for the selected fallback model; add full agent delegation only if maintainers intentionally define its separate session semantics.

## Reproduction Assessment

Yes, at source level: current main has a concrete provider/model-only fallback path and strict schemas with no fallback-specific agent, workspace, or context selector. The reported output quality depends on model behavior, but the missing capability itself is high-confidence.

## Solution Assessment

No, the issue's broad fallbackAgent shape is not yet proven to be the best solution because it risks crossing session and credential ownership. A narrower attempt-scoped model context profile is the safer default, subject to maintainer product approval.
