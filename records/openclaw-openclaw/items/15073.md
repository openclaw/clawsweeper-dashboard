## Summary

Keep this open: current main still implements model fallback as provider/model candidate selection on the same agent/session/workspace context, and related requests have been closed into this issue as the canonical design thread.

## What This Changes

The issue requests a fallback-context mechanism, such as fallbackAgent, fallbackWorkspace, or model-specific context profiles, so fallback models can use a smaller agent workspace and prompt context.

## Best Possible Solution

Define one first-class fallback-context contract, then update config types, strict schemas, docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests together.

## Reproduction Assessment

Yes for a source-level reproduction: the proposed fields are absent from strict schemas, fallback candidates carry only provider/model, and queued runs keep one agent/session/workspace context. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose the durable contract across agent switching, workspace overrides, context profiles, context budgets, and runbook semantics before implementation.
