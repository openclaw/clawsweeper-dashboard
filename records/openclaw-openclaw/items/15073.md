## Label Justifications

- `P2`: This is a valid multi-model fallback improvement with real user impact, but it is a design-level feature request rather than an urgent broken existing contract.
- `impact:session-state`: The requested behavior changes fallback use of agent identity, workspace context, session files, skills snapshots, and transcript state.
- `impact:auth-provider`: Fallback attempts already change provider/model routing, and any agent-switching design must preserve auth-profile and credential routing semantics.

## Summary

Keep open: current main still implements fallback as provider/model candidate selection while preserving the same agent, session, workspace, agentDir, and skills snapshot; the related closed context/runbook reports point back to this issue as the canonical fallback-context design thread, and the adjacent open notification/re-injection thread is not a full replacement.

## What This Changes

Request an agent-level or model-specific fallback context mechanism so fallback models can use a separate workspace or reduced context instead of the primary agent's context.

## Best Possible Solution

Define one first-class fallback-context contract and implement it consistently across config schema/help/docs, fallback candidate metadata, run/session context, auth routing, and regression coverage.

## Reproduction Assessment

Yes for a source-level reproduction: current schemas lack the proposed fields, fallback candidates carry only provider/model, and the embedded run parameters reuse the existing session/workspace/agent context. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose the durable contract across agent switching, workspace overrides, context profiles, context budgets, and runbook semantics before implementation.
