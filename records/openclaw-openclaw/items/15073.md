## Label Justifications

- `P2`: This is a valid multi-model fallback improvement with real user impact, but it is a design-level feature request rather than an urgent broken existing contract.
- `impact:session-state`: The requested behavior would change fallback use of agent identity, workspace context, agentDir, skills snapshots, session files, and transcript continuity.
- `impact:auth-provider`: Fallback attempts already change provider/model routing, and any agent-switching design must preserve auth-profile and credential routing semantics.

## Summary

Keep open: current main still treats fallback as provider/model candidate selection inside the active agent context, and the remaining work needs a maintainer product decision on fallback agent, workspace, context-profile, runbook, and session/auth semantics.

## What This Changes

The issue requests a configurable fallback path that can use a different agent/workspace or model-specific context profile when the primary model falls back to a smaller or local model.

## Best Possible Solution

Define one first-class fallback-context contract and carry it through config schema/help/docs, fallback candidate metadata, run/session context, auth routing, and regression coverage without corrupting the original agent session.

## Reproduction Assessment

Yes for a source-level reproduction: current schemas lack the requested fields, fallback candidates carry only provider/model, and the runner resolves context from the same session agent/workspace. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose the durable contract across agent switching, workspace overrides, context profiles, runbook injection, and auth/session semantics before implementation.
