## Label Justifications

- `P2`: This is a valid multi-model fallback improvement with real user impact, but it is a design-level feature request rather than an urgent broken existing contract.
- `impact:session-state`: The requested behavior changes fallback use of agent identity, workspace context, session files, skills snapshots, and transcript state.
- `impact:auth-provider`: Fallback attempts already change provider/model routing, and any agent-switching design must preserve auth-profile and credential routing semantics.

## Summary

Keep open: current main still treats fallback as provider/model candidate selection while reusing the same agent, session, workspace, agentDir, and skills context, and related closed threads point back to this issue as the canonical fallback-context design item.

## What This Changes

Request configurable fallback behavior that can switch to a different agent, workspace, or model-specific context profile when a primary model falls back.

## Best Possible Solution

Define one first-class fallback-context contract, then update config types, strict schemas, docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests together.

## Reproduction Assessment

Yes for a source-level reproduction: the proposed fields are absent from strict schemas, fallback candidates carry only provider/model, and queued runs reuse one agent/session/workspace context. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose the durable contract across agent switching, workspace overrides, context profiles, context budgets, and runbook semantics before implementation.
