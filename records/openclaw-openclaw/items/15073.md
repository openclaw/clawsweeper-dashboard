## Label Justifications

- `P2`: This is a valid multi-model fallback improvement with real user impact, but it is a design-level feature request rather than an urgent broken existing contract.
- `impact:session-state`: The requested behavior changes fallback use of agent identity, workspace context, session files, skills snapshots, and transcript state.
- `impact:auth-provider`: Fallback attempts already change provider/model routing, and any agent-switching design must preserve auth-profile and credential routing semantics.

## Summary

Keep open: current main still falls back across provider/model candidates while reusing the active agent, workspace, session, and bootstrap context, and related fallback-context requests have been closed into this issue as the canonical design thread.

## What This Changes

Request a fallback mode that can switch to an alternate agent/workspace or model-specific context profile when a fallback model is selected.

## Best Possible Solution

Define one first-class fallback-context contract and implement it consistently across config schemas/help/docs, fallback candidate metadata, run/session context, auth routing, and regression coverage.

## Reproduction Assessment

Yes for a source-level reproduction: current schemas lack the proposed fields, fallback candidates carry only provider/model, and the runner resolves context from the same session agent/workspace. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should choose the durable contract across agent switching, workspace overrides, context profiles, runbook injection, and auth/session semantics before implementation.
