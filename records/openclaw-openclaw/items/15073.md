## Summary

Keep this open. Current main still treats fallback as provider/model candidate selection while reusing the same agent run context, and the recent maintainer/Codex comments explicitly kept this as design work rather than cleanup closure.

## What This Changes

Request to let model fallback switch to a different agent, workspace, or model-specific context profile so smaller fallback models do not receive the primary agent's full context.

## Best Possible Solution

Define a first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests for alternate fallback context without corrupting original agent state.

## Reproduction Assessment

Yes. Static source inspection on current main shows strict config surfaces lack the requested fields, fallback candidates carry only provider/model, and fallback retry paths reuse the original workspace/session/agent state.

## Solution Assessment

Unclear. The requested fallbackAgent shape may be viable, but maintainers should first choose between agent switching, workspace override, and model-specific context-profile or budget semantics.
