## Label Justifications

- `P2`: Fallback quality and graceful degradation are materially affected, but this is a bounded feature rather than an urgent core outage.
- `impact:session-state`: Changing workspace or agent context during fallback can alter transcript, memory, skills, and session-state association.
- `impact:auth-provider`: The requested context behavior is selected as part of provider and model fallback routing.

## Summary

Current main still performs fallback as provider/model candidate selection within one caller-supplied agent and session context. The requested capability remains missing, but its configuration and state-transfer semantics require an explicit product decision before implementation.

## What This Changes

Add fallback-specific agent or context selection so smaller fallback models can use a reduced workspace, prompt, skills, and tool surface.

## Best Possible Solution

Define a typed fallback context profile that can reduce bootstrap files, prompt budgets, skills, and tools while preserving the original agent and session identity by default; add cross-agent handoff only if maintainers explicitly define state, auth, transcript, and delivery ownership.

## Reproduction Assessment

Yes, at source level. Current-main types, strict schemas, and fallback execution prove that retries vary provider/model only while reusing the caller's agent and session context.

## Solution Assessment

Unclear as proposed. The user problem is valid, but a fallback-specific context profile is narrower and safer than switching complete agent identity; maintainers must choose the intended public contract.
