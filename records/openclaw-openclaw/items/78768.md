## Summary

Keep open: the staged-output refactor is directionally consistent with the new fs-safe primitive, but it introduces a blocking Mantis visual-task regression by dropping a recording when `crabbox record` exits nonzero after producing driver output, which current code treats as recoverable.

## What This Changes

This PR bumps `@openclaw/fs-safe`, exports `writeExternalFileWithinRoot` through OpenClaw security facades, and routes browser, media, channel, and QA external output writers through staged finalization.

## Best Possible Solution

Keep the fs-safe staged-output hardening, but preserve Mantis' current ability to publish an already-produced recording before surfacing an expected record-command failure.

## Reproduction Assessment

Yes, source-reproducible. Current main writes `crabbox record` directly to `videoPath` and continues when the driver result exists, while the PR finalizes staged output only after the record command resolves successfully.

## Solution Assessment

No. `writeExternalFileWithinRoot` is the right primitive for successful external writers, but this specific `crabbox record` call needs an opt-in publish-on-expected-error path so the refactor does not drop existing debug artifacts.

## Repair Work Prompt

On PR branch `refactor/fs-safe-external-output`, fix the Mantis visual task recording regression introduced by staging `crabbox record` output through `writeExternalFileWithinRoot`. Current main writes `crabbox record` directly to `videoPath`, catches a nonzero exit when `driverResultPath` exists, records `recordingError`, and still uses a produced `visual-task.mp4`. The PR's helper only publishes after the command resolves, so a written staged video is cleaned up on that expected error path. Preserve the staged external-output hardening for successful paths and do not loosen other writers. Likely approach: add an opt-in publish-on-error path to `runCommandWithExternalOutput`, or handle the record call locally by catching the command error inside the write callback, finalizing only when `driverResultPath` exists, then rethrowing after finalization so existing `recordingError` behavior remains. Add/extend `extensions/qa-lab/src/mantis/visual-task.runtime.test.ts` with a case where the record runner writes the temp output and throws while the driver result exists, asserting the final video is present/readable and the summary preserves the current failure/reporting behavior. No changelog is expected for this internal PR repair unless maintainers request one.
