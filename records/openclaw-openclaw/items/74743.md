## Summary

Keep open. Current main still matches the reported custom-session cron transcript gap: cron direct delivery can send to the destination channel, but the direct delivery call does not pass mirror metadata, and transcript persistence remains gated on that mirror option. The issue is also paired with open PR #74913 using closing syntax, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

Cron jobs created with `--session session:<id>` should persist their delivered assistant response into the destination session transcript so later channel turns remember it.

## Best Possible Solution

Review and land PR #74913, revise it, or replace it with an equivalent focused patch that passes direct cron delivery mirror metadata into the existing outbound transcript mirror contract with idempotency and regression coverage for `session:<id>` jobs.

## Reproduction Assessment

Yes. The reporter's steps map directly to current main: `session:<id>` resolves to a stable session key, direct cron delivery sends the payload without `mirror`, and the only transcript append path in `deliverOutboundPayloads` requires `params.mirror`.

## Solution Assessment

Yes. Reusing the existing outbound mirror contract is the narrowest maintainable fix because it preserves the shared transcript idempotency and delivery-success gating already used by other outbound sends.
