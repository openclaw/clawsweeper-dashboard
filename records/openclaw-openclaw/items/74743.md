## Label Justifications

- `P2`: This is a bounded cron/session-state persistence bug with limited blast radius but real conversation-continuity impact.
- `impact:session-state`: The affected behavior is drift between a cron message delivered to a channel and the session context/history used by later turns.

## Summary

Keep open: current main still treats `session:<sessionId>` as a literal session key, so the reporter's persisted sessionId path can run and mirror under the wrong cron/custom session instead of the destination channel session. The merged target-awareness work helps the next-turn context slice, but the later session-id resolver PR that directly targeted this issue was closed unmerged.

## What This Changes

The issue reports that cron jobs targeted with `--session session:<id>` deliver to a channel but do not persist the delivered response into the intended destination session history.

## Best Possible Solution

Use one store-backed cron session-target resolver that preserves exact session keys first, resolves persisted `sessionId` values second, and feeds execution, delivery context, previews, transcript mirroring, and failure notifications consistently.

## Reproduction Assessment

Yes by source inspection: current main extracts the `session:` suffix literally and never maps a persisted `sessionId` from the session store to its session key. I did not run the live Discord cron flow in this read-only review.

## Solution Assessment

No completed fix is on main yet. The best fix is a shared store-backed resolver applied to all cron session-target consumers, not another one-call-site patch or a broader proactive-context redesign.
