## Summary

Keep open. Current main still matches the reported cron transcript gap: explicit `session:<id>` cron targets are persistent by contract, but direct cron delivery calls the shared outbound delivery helper without the `mirror` payload that triggers transcript persistence. The issue is also paired with open PR #74913 using closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that cron jobs targeting an explicit persistent session deliver the response to the channel but do not persist that assistant response into the destination session transcript.

## Best Possible Solution

Land PR #74913 or an equivalent focused patch that passes direct cron delivery mirror metadata into the existing outbound transcript mirror contract, using the destination session key and delivery idempotency key with regression coverage for `session:<id>` cron jobs.

## Reproduction Assessment

Yes. The reporter's path maps directly to current main: `session:<id>` resolves to a persistent run session, direct cron delivery sends the outbound payload without `mirror`, and `deliverOutboundPayloads` only appends to a transcript when `params.mirror` is present.

## Solution Assessment

Yes. Reusing the existing outbound mirror contract is the narrowest maintainable fix because it keeps transcript writes gated on successful delivery and preserves the existing idempotency/deduplication behavior instead of adding a cron-only append path.
