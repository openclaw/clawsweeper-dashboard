## Summary

Keep open: current main partially addresses cron target-session awareness, but it still treats the `session:` suffix literally and does not resolve the reporter's `session:<sessionId>` target back to the destination session key, so the persistence bug remains.

## What This Changes

The issue reports that cron jobs targeted with `--session session:<sessionId>` deliver to a channel but do not persist the delivered response in the intended destination session.

## Best Possible Solution

Use one store-backed resolver so `session:<sessionId>` maps to the existing destination session key consistently across cron execution, announce and failure delivery, delivery context, and previews, then prove it with a real cron delivery plus destination-history readback.

## Reproduction Assessment

Yes from source inspection and the issue steps: current main extracts the `session:` suffix literally, so `session:<sessionId>` does not select the existing destination session key. I did not run a live Discord cron repro because this review is read-only.

## Solution Assessment

Yes, the best fix appears to be one shared store-backed session-target resolver, not another delivery-only workaround. The closed fix candidate pointed in that direction but still needed sibling-consumer alignment and real after-fix proof.
