## Label Justifications

- `P2`: This is a bounded cron/session-state persistence bug with limited blast radius but real conversation-continuity impact.
- `impact:session-state`: The affected behavior creates drift between a cron response visible in a channel and the session context/history used by later agent turns.
- `impact:data-loss`: The delivered cron response can be absent from the intended destination session log/context.

## Summary

Keep open: current main still treats `session:<sessionId>` as a literal cron session key, so the destination channel session can miss the delivered response in durable session history. The merged target-awareness work in https://github.com/openclaw/openclaw/pull/93580 helps the next-turn context slice, but durable `sessionId` resolution remains unmerged and the exact later candidate at https://github.com/openclaw/openclaw/pull/96522 was closed unmerged.

## What This Changes

The issue requests that cron jobs created with `--session session:<sessionId>` persist delivered cron responses into the intended destination channel session history/context.

## Best Possible Solution

Use one store-backed cron session-target resolver that preserves exact session keys first, resolves persisted `sessionId` values second, and feeds execution, delivery context, previews, transcript mirroring, and failure notifications consistently.

## Reproduction Assessment

Yes. Source inspection shows current main extracts the `session:` suffix literally and never maps a persisted `sessionId` from the session store back to its destination session key; I did not run the live Discord cron flow.

## Solution Assessment

Yes. A shared store-backed resolver is the narrowest maintainable fix; merged ephemeral awareness and plugin transcript APIs do not replace consistent cron sessionId resolution.
