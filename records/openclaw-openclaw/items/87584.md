## Summary

Keep open: current main and v2026.6.6 still hard-disable active-run steering for non-internal room events, and the queue config/schema has no opt-in for trusted group rooms. The request is plausible but needs maintainer product and security direction before implementation.

## What This Changes

The issue requests a default-off configurable policy that lets trusted group room events steer an active run when queue mode is `steer`.

## Best Possible Solution

Add a default-off room-event steering policy in the canonical messages/queue configuration, with schema/help/docs/tests and the narrowest trusted-room scope maintainers approve.

## Reproduction Assessment

Yes. Source and existing tests show the current-main path: a `room_event` with queue mode `steer` produces `shouldSteer: false` and `shouldFollowup: true`; I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the policy boundary. The maintainable direction is a documented default-off opt-in, not removing the `room_event` guard globally.
