## Summary

Keep open: current main and latest release still hard-disable active-run steering for room events, and no queue/config schema knob exists to opt trusted group rooms into steering. The request is valid, but it needs maintainer product and security direction before implementation.

## What This Changes

The issue requests an opt-in configuration path that allows trusted group room events to steer active runs when queue steering is enabled, while preserving the current safe default.

## Best Possible Solution

Keep the current room-event follow-up/message-tool-only behavior as the default, then add a documented default-off steering opt-in scoped to trusted operator rooms if maintainers approve the security model.

## Reproduction Assessment

Yes, source-level. Current main has a direct `!isRoomEvent` gate in the steering predicate, and an adjacent test asserts active `room_event` input with queue mode `steer` is queued as follow-up instead of steering.

## Solution Assessment

Unclear until maintainers choose the policy boundary. A default-off, documented opt-in is the maintainable direction; removing the room-event guard globally would be the wrong security shape.
