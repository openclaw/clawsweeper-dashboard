## Summary

Keep open: current main and the latest release still hard-disable active-run steering for room events, and the requested opt-in is a new security-sensitive queue/config behavior that needs maintainer product direction.

## What This Changes

The issue requests a default-off configuration path that lets trusted group room events steer an active run when `messages.queue.mode` is `steer`.

## Best Possible Solution

Keep room-event steering disabled by default, and add a documented default-off opt-in for trusted operator rooms only after maintainers choose the trust model and config scope.

## Reproduction Assessment

Yes, source-level. Current main has a direct `!isRoomEvent` steering gate, and an adjacent test asserts an active `room_event` with queue mode `steer` is queued as a follow-up instead of steering.

## Solution Assessment

Unclear until maintainers choose the policy boundary. A default-off, documented opt-in scoped to trusted rooms looks maintainable; removing the room-event guard globally would be the wrong security shape.
