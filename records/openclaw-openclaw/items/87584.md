## Summary

Keep open: current main and the latest release still hard-disable active-run steering for room events, while the requested opt-in would add a new security-sensitive queue/config surface that needs maintainer product direction before implementation.

## What This Changes

The issue requests a default-off configuration path that lets trusted group room events steer an active run when `messages.queue.mode` is `steer`, instead of always routing them as followups.

## Best Possible Solution

Keep room events as followup/message-tool-only by default, and add a documented default-off opt-in for trusted operator rooms only if maintainers approve the security model and config scope.

## Reproduction Assessment

Yes, source-level. Current main has a direct `!isRoomEvent` steering gate and an adjacent test asserts an active `room_event` with queue mode `steer` is queued as followup instead of steering.

## Solution Assessment

Unclear until maintainers choose the policy boundary. A default-off, documented opt-in scoped to trusted rooms is the maintainable direction; removing the room-event guard globally would be the wrong security shape.
