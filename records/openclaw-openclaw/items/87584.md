## Summary

Keep open: current main and the latest release still hard-disable active-run steering for room events, and there is no queue/config schema knob to opt trusted group rooms into steering. The request is valid, but it is a feature/config/security decision rather than a strict bug fix.

## What This Changes

Request a default-off configuration option that lets selected group room events steer an active run when queue mode is `steer`.

## Best Possible Solution

Keep the current room-event follow-up default, then add a documented default-off steering opt-in scoped to trusted operator rooms if maintainers approve the security model.

## Reproduction Assessment

Yes, source-level. Current main has a direct `!isRoomEvent` gate in the steering predicate, and an adjacent test asserts active `room_event` input with queue mode `steer` is queued as follow-up instead of steering; no tests were run because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the policy boundary. A default-off, documented opt-in is the maintainable direction; removing the room-event guard globally would be the wrong security shape.
