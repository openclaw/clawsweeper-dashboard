## Summary

Keep open: current main still hard-disables active-run steering for room events, and no existing config surface provides the requested opt-in. The request is plausible but needs maintainer product/security direction because it would let trusted group-room traffic affect an active tool-enabled session.

## What This Changes

Request to add an opt-in configuration path that allows group room events to steer active runs in `messages.queue.mode: "steer"` while preserving the current followup default.

## Best Possible Solution

Add a default-off room-event steering policy in the canonical messages/queue configuration, with schema/help/docs/tests and the narrowest scope maintainers approve, ideally channel or room scoped for trusted operator rooms.

## Reproduction Assessment

Yes. Source and existing tests show the current-main path: a `room_event` with queue mode `steer` produces `shouldSteer: false` and `shouldFollowup: true`; I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the policy boundary. The maintainable direction is not to remove the `room_event` guard outright, but to add a documented default-off opt-in with focused coverage for default and enabled behavior.
