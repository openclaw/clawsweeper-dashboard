## Summary

Keep open: current main still hard-disables active-run steering for room events, and the queue config/schema/docs do not provide a default-off room-event steering opt-in. The request remains valid but needs maintainer product/security direction because it changes whether selected group-room traffic can affect an active tool-enabled session.

## What This Changes

Requested configurable opt-in behavior for steering group room events when `messages.queue.mode=steer` is enabled.

## Best Possible Solution

Add a documented, default-off room-event steering policy only after maintainers choose the allowed scope, such as global, per-channel, or per-room trusted operator rooms.

## Reproduction Assessment

Yes. Source and existing tests show the current-main path: active `room_event` input with queue mode `steer` produces `shouldSteer: false` and `shouldFollowup: true`; I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the policy boundary. The maintainable direction is a default-off, documented opt-in with focused coverage for disabled and enabled behavior, not removing the room-event guard globally.
