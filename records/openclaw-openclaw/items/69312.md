## Summary

Keep open: the underlying parser bug is valid, but this PR head still changes existing lowercase media directive behavior, leaves streaming buffering inconsistent with the new indented-code parser rule, and lacks after-fix real behavior proof.

## What This Changes

The PR updates media and fence parsing to skip MEDIA extraction from CommonMark indented code blocks, tighten fence closing validation, adjust cleanup around retained code lines and audio tags, and add parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips CommonMark indented code blocks while preserving existing lowercase MEDIA handling, matching streaming buffering, and using code-aware cleanup after media/audio tag removal.

## Reproduction Assessment

Yes, source-level. Current main skips fenced spans but not CommonMark indented code before the trimmed MEDIA gate, so a blank-preceded 4-space or tab-indented MEDIA line still reaches extraction; this read-only review did not execute tests.

## Solution Assessment

No. The parser is the right place to fix the false positive, but this PR head is not the best mergeable solution because it changes the lowercase directive contract and leaves streaming/audio cleanup out of sync.

  - body: Current main intentionally parses lowercase `media:` in block replies. Removing the regex `i` flag and adding a case-sensitive guard makes `media: ./report.pdf` remain visible text instead of producing `mediaUrls`, which regresses the existing block-reply contract.

  - confidence: 0.9

  - body: After `[[audio_as_voice]]` is stripped, this branch always applies the non-code leading-normalization regex. If the first remaining content is an indented code line, the cleanup can remove its tab or up to three spaces, changing visible code formatting and potentially exposing `MEDIA:` as a directive later.
  - confidence: 0.88
