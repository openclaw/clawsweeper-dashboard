## Summary

Keep open: the reported parser bug is real and the PR targets the right surface, but the current branch still has blocking parser/streaming/audio regressions and lacks required real behavior proof for an external PR.

## What This Changes

The PR changes media/fence parsing so indented `MEDIA:` lines are treated as code text, tightens fence closing validation, adjusts media/audio cleanup whitespace handling, and adds parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips CommonMark indented code blocks while preserving current lowercase MEDIA, streaming buffering, and audio cleanup semantics, with focused regressions and real behavior proof.

## Reproduction Assessment

Yes, source-level: current main excludes fenced spans but has no indented-code guard before the trimmed `MEDIA:` gate, so a blank-preceded 4-space or tab-indented `MEDIA:` line still reaches extraction. I did not run tests because this review is read-only.

## Solution Assessment

No. The parser is the narrow maintainable place to fix this, but this branch should preserve current lowercase directive behavior and keep streaming/audio cleanup aligned with the new code-aware parsing rules before merge.

  - body: Current main intentionally parses lowercase `media:` in block replies. Removing the regex `i` flag and making the entry guard case-sensitive turns `media: ./report.pdf` into visible text instead of `mediaUrls`, regressing the existing block-reply contract.
