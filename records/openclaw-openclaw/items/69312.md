## Summary

This PR is a focused fix for a real media-parser bug, but it is not merge-ready: the final parser changes are still inconsistent with streaming tail buffering, the audio-tag cleanup can still damage tab-indented code, and the contributor has not supplied usable real behavior proof.

## What This Changes

The PR changes `splitMediaFromOutput` to preserve blank-preceded 4-space/tab-indented `MEDIA:` lines as text, tightens markdown fence closing validation, and adds media/fence regression tests.

## Best Possible Solution

Unify the final parser, streaming accumulator, and audio-tag cleanup around one code-aware directive classification, keep the focused regression tests, and require redacted real OpenClaw or Telegram proof before merge.

## Reproduction Assessment

Yes, from source inspection: current `main` skips fenced code only and then trims leading whitespace before matching `MEDIA:`, which is enough to reproduce the indented-code false positive without adding new product semantics.

## Solution Assessment

No, not yet. The direction is right, but the mergeable solution needs the streaming accumulator and audio-tag cleanup to share the same indented-code behavior before real transport proof.

- **[P2] Align streaming buffering with indented MEDIA code:** `src/media/parse.ts:563-569`
  - body: After this parser preserves blank-preceded 4-space/tab `MEDIA:` lines as text, `splitTrailingDirective` still buffers any `^\s*MEDIA:` tail. A streamed reply ending a chunk with `\n\n    MEDIA:/tmp/example.png` can move that code line into `pendingTail` and lose it if the stream item resets before final flush, so streaming and final parsing disagree. Update the accumulator guard/test to mirror the new indented-code classification.
  - confidence: 0.86
- **[P2] Preserve tab-indented code after audio tags:** `src/media/parse.ts:745-748`
  - body: The post-`[[audio_as_voice]]` cleanup always uses the non-code leading-normalization regex. For `[[audio_as_voice]]\n\n\tMEDIA:/tmp/code.png`, the first surviving line was marked as code, but this branch strips the tab and returns `MEDIA:/tmp/code.png`, regressing visible formatting and making the code example look like a directive. Reuse the code-aware leading regex here and add a tab regression.
  - confidence: 0.83
