## Summary

Keep open: the reported bug is source-reproducible and the parser is the right area, but the current PR head still regresses established lowercase media directive support, leaves streaming/audio cleanup out of sync with the new code-aware rules, and lacks real behavior proof.

## What This Changes

The PR changes MEDIA/fence parsing to skip MEDIA extraction from indented code blocks, tighten fence closing validation, adjust media cleanup around code lines/audio tags, and add parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips indented code blocks while preserving lowercase directive support and keeping streaming/audio cleanup rules aligned, with focused regressions and redacted real behavior proof.

## Reproduction Assessment

Yes, source-level: current main excludes fenced spans but not CommonMark indented code blocks before the trimmed `MEDIA:` gate, so the reported false extraction path is clear. I did not run tests because this review was read-only.

## Solution Assessment

No. The parser is the narrow maintainable place to fix this, but this PR should not merge until lowercase media semantics, streaming buffering, and audio-tag cleanup follow the same directive/code rules.
