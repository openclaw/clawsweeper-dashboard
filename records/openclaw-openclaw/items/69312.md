## Summary

Keep open: this PR targets a real source-reproducible parser bug, but the latest head still regresses established lowercase MEDIA directive behavior, leaves streaming/audio cleanup mismatched with the new parser rules, and lacks after-fix real behavior proof.

## What This Changes

The PR updates media and markdown fence parsing so indented-code `MEDIA:` lines are preserved as text, malformed fence closers stay open, and parser/fence regressions are added.

## Best Possible Solution

Land a focused parser fix that preserves indented-code `MEDIA:` text while retaining lowercase directives, aligning streaming/audio cleanup, and adding redacted real CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main skips only fenced spans and then checks `trimStart().toUpperCase().startsWith("MEDIA:")`, so blank-preceded 4-space or tab-indented code lines can be extracted; I did not run tests because the review must remain read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the patch should preserve lowercase media directives and keep streaming plus audio-tag cleanup semantics aligned.
