## Summary

Keep open: this is a valid targeted parser fix, but the latest head still needs contributor real behavior proof and has a P2 audio-tag cleanup regression that can strip a leading tab from preserved code text.

## What This Changes

The PR adds indented-code-block detection to `splitMediaFromOutput`, code-aware whitespace cleanup, stricter markdown fence closing validation, and regression tests.

## Best Possible Solution

Have the contributor apply the same code-aware leading normalization after audio tag stripping, add a tab-indented `audio_as_voice` regression test, and then add redacted after-fix runtime proof before landing with green checks.

## Reproduction Assessment

Yes. Source inspection shows current main trims indentation before the `MEDIA:` prefix check and only protects fenced spans, so a 4-space or tab-indented `MEDIA:` line after a blank line has a clear failing path; I did not run tests because this review is read-only.

## Solution Assessment

Mostly yes, but not as currently written. The indented-code guard is the right narrow parser fix, while the post-audio-tag cleanup should reuse the code-aware leading-normalization decision and the PR still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve tab-indented code after audio tag stripping:** `src/media/parse.ts:747-748`
  - body: The main cleanup path now avoids stripping indentation when the first surviving line is code, but the `parseAudioTag` path still applies the non-code leading-normalization regex unconditionally. Reuse the code-aware `leadingNormRegex` here or add an equivalent code-aware branch, plus a tab-indented audio-tag regression test.
