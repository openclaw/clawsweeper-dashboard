## Summary

Keep open: this is a real parser bug and the branch is useful, but the latest PR head is not merge-ready because it regresses the existing lowercase media directive contract, leaves streaming and audio-tag cleanup paths inconsistent with the new rule, and still lacks real behavior proof.

## What This Changes

The PR adds indented-code MEDIA suppression, code-aware cleanup, stricter markdown fence closing validation, and parser/fence regression tests.

## Best Possible Solution

Land a focused parser update that treats indented code blocks consistently across batch and streaming paths, preserves the lowercase `media:` contract, reuses code-aware cleanup after audio stripping, validates shared fence consumers, and includes redacted CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main only excludes fenced spans before `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so a blank-preceded 4-space or tab-indented MEDIA-looking code line can enter extraction.

## Solution Assessment

No, not as submitted. The indented-code guard is the right narrow direction, but the branch should preserve existing lowercase media behavior and update the streaming/audio cleanup paths that share the same parsing contract.
