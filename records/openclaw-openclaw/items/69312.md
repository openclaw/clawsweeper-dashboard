## Summary

Keep open: this PR targets a real source-reproducible media parsing bug, but the latest head still regresses lowercase media directives, leaves streaming buffering out of sync with the new indented-code rule, can strip retained code indentation after audio-tag cleanup, and lacks real behavior proof.

## What This Changes

The PR updates media and markdown parsing to avoid extracting MEDIA-looking lines from indented or malformed fenced code blocks, adjusts cleanup/case handling, and adds media/fence parser tests.

## Best Possible Solution

Land a focused parser fix that skips indented-code MEDIA false positives while preserving lowercase media support and sharing the same code-aware rules across batch parsing, streaming buffering, and audio-tag cleanup.

## Reproduction Assessment

Yes, source-level. Current main excludes fenced spans only, then trims leading indentation before a case-insensitive MEDIA gate, so an indented-code `MEDIA:` line can be extracted without needing a new product decision.

## Solution Assessment

No, not as submitted. The indented-code and fence-validation direction is narrow, but the branch should preserve lowercase media semantics and align the same code-aware parsing rules across streaming and audio cleanup before merge.

  - confidence: 0.88
