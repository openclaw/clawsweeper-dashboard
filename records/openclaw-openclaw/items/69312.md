## Summary

Keep open. Current main still has the reported indented-code MEDIA extraction bug, and this PR is useful source work, but the latest head is not merge-ready because it changes established lowercase MEDIA directive behavior and still has a non-code-aware audio-tag cleanup path that can strip retained code indentation.

## What This Changes

The branch updates `splitMediaFromOutput` to skip MEDIA extraction from indented code blocks, preserve retained code indentation during cleanup, tighten `parseFenceSpans` closing-fence validation, make MEDIA matching case-sensitive, and add media/fence parser tests.

## Best Possible Solution

Land a focused version that preserves indented-code `MEDIA:` lines and retained code indentation, keeps lowercase MEDIA semantics consistent across parser, block replies, and streaming unless a coordinated product change updates all surfaces, reuses code-aware cleanup after audio-tag stripping, validates shared fence consumers, and adds the required changelog fix entry.

## Reproduction Assessment

Yes. Current main is source-reproducible with output containing a blank line followed by a 4-space or tab-indented `MEDIA:` line; the PR regressions are reproducible from the existing lowercase block-reply test and the `[[audio_as_voice]]` plus indented-code cleanup path.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow direction, but the case-sensitivity change and post-audio cleanup path are not the safest maintainable solution without coordinated updates across the other directive surfaces.
