## Summary

Keep open: the core indented-code fix is plausible, but the current head still breaks established lowercase MEDIA parsing, leaves streaming tail buffering inconsistent with the new indented-code rule, and lacks required real behavior proof.

## What This Changes

The PR updates media parsing to skip MEDIA directives in indented code blocks, protect retained code whitespace, tighten fence closing validation, remove case-insensitive MEDIA matching, and add parser tests.

## Best Possible Solution

Land a focused parser and streaming fix that preserves the existing lowercase MEDIA contract or intentionally migrates all consumers together, applies the indented-code rule consistently, preserves code indentation after audio tags, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main only excludes fenced spans and then case-insensitively treats line.trimStart() MEDIA prefixes as directives, so indented code lines can be extracted. I did not execute tests because this review was constrained to read-only inspection.

## Solution Assessment

No. The indented-code guard belongs in splitMediaFromOutput, but this PR is not yet the best solution because it changes tested lowercase behavior, leaves streaming buffering on old directive rules, and applies non-code cleanup after audio tag stripping.
