## Summary

Keep this PR open. Current main still has the reported indented-code MEDIA false positive, but the PR head is not merge-ready because it makes lowercase MEDIA parsing inconsistent with existing block-reply and streaming behavior, still has one tab-indentation cleanup gap after audio-tag removal, and lacks the required user-facing changelog entry.

## What This Changes

The PR updates media and markdown fence parsing to preserve indented-code MEDIA lines, protect retained code indentation during cleanup, reject invalid fence closers with trailing text, make MEDIA token matching case-sensitive, and add media/fence regression tests.

## Best Possible Solution

Land a focused version that keeps the indented-code MEDIA protection and code-aware cleanup, preserves lowercase MEDIA compatibility unless every directive surface is intentionally migrated together, reuses the code-aware normalization path after audio-tag stripping, validates shared fence consumers, and adds the active-version changelog fix entry.

## Reproduction Assessment

Yes. Current main is source-reproducible by passing output with a blank line followed by a four-space or tab-indented MEDIA line to splitMediaFromOutput; the current line gate trims indentation and extracts it unless it is inside a fenced span.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow direction, but the parser case change and post-audio cleanup path need repair before this is the best maintainable fix.
