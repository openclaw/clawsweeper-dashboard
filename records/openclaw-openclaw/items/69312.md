## Summary

Keep open: the underlying MEDIA indented-code bug is source-reproducible on current main, but this PR is not merge-ready because it regresses existing lowercase MEDIA behavior, can strip tab indentation after audio-tag removal, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds indented-code detection to MEDIA parsing, preserves code-line whitespace during cleanup, tightens fence closing validation, removes MEDIA regex case-insensitivity, and adds parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips MEDIA extraction inside CommonMark indented code while preserving current lowercase and streaming contracts, keeping tab-indented code after audio tags, and adding regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `splitMediaFromOutput` only excludes fenced spans before `trimStart().toUpperCase().startsWith("MEDIA:")`, which gives a concrete path for indented `MEDIA:` false extraction.

## Solution Assessment

No. The indented-code guard is in the right area, but the PR also changes established lowercase MEDIA behavior and does not apply code-aware leading normalization after audio tag stripping.
