## Summary

Keep open: this is a real parser bug and the central indented-code fix is useful, but the current branch regresses the existing lowercase MEDIA contract, leaves streaming/audio cleanup semantics misaligned, and still lacks real behavior proof.

## What This Changes

This PR adds indented-code protection and code-aware cleanup to MEDIA parsing, tightens markdown fence closing validation, changes MEDIA matching to case-sensitive, and adds media/fence tests.

## Best Possible Solution

Land a focused parser fix that skips indented-code MEDIA false positives while preserving lowercase MEDIA compatibility and aligning streaming plus audio-tag normalization with the same rules.

## Reproduction Assessment

Yes, source-level: current main only skips fenced spans and then extracts any trimmed 4-space/tab MEDIA line via the uppercase-normalized line gate. I did not run tests because this review was read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should preserve the existing lowercase MEDIA contract and keep streaming/audio cleanup behavior aligned.
