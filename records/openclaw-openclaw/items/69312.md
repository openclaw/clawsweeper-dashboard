## Summary

## What This Changes

The PR adds indented-code-block handling to `splitMediaFromOutput`, tightens markdown fence closing detection, and adds media/fence regression tests.

## Best Possible Solution

Land a focused parser fix that applies code-aware leading normalization after audio-tag stripping and includes redacted real CLI or Telegram proof that indented-code `MEDIA:` is preserved while real media still attaches.

## Reproduction Assessment

Yes, source-level. Current main only excludes fenced spans before the trimmed `MEDIA:` gate, so a blank-preceded 4-space or tab-indented `MEDIA:` line can still be extracted from an indented code block.

## Solution Assessment

No, not yet. The indented-code guard is the right direction, but the audio-tag cleanup path needs to reuse the code-aware normalization and the PR still needs real after-fix proof.
