## Summary

Keep open: current main still has the reported source-level bug, but this PR should not merge yet because it changes established lowercase MEDIA semantics and lacks real behavior proof.

## What This Changes

The PR updates media/fence parsing and tests to avoid extracting MEDIA directives from CommonMark indented code blocks, preserve retained code indentation, validate fence close lines more strictly, and treat lowercase media prefixes as plain text.

## Best Possible Solution

Land a focused parser fix that skips MEDIA extraction in indented code blocks while preserving the established lowercase MEDIA contract across media parsing, block reply parsing, and streaming buffering, with redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: splitMediaFromOutput only excludes fenced spans, then trim-starts indented lines before a case-insensitive MEDIA gate, so indented code lines can be extracted.

## Solution Assessment

No, not as patched. The indented-code guard belongs in the right area, but the PR changes MEDIA case semantics without updating the existing block-reply and streaming contract, and it still needs real behavior proof.

## Review Findings

- **[P2] Keep MEDIA case semantics aligned:** `src/media/parse.ts:17`
  - body: Removing the case-insensitive flag here makes splitMediaFromOutput stop extracting lowercase media:, but block-reply auto parsing still opts in for /media:/i and existing coverage expects lowercase directives to work. Streaming buffering at the PR head also still treats lowercase MEDIA tails as directive candidates, so lowercase media can be left visible or buffered inconsistently instead of delivered.
  - confidence: 0.92

When an audio tag is removed before a tab-indented code line, that tab can be stripped, so the code line is returned unindented even though the PR classified tab-indented lines as code.
  - confidence: 0.8
