## Summary

Keep open: current main still has the indented-code MEDIA false positive and this PR is the active canonical fix, but the current branch is not merge-ready because it regresses established lowercase `media:` support, leaves streaming/audio cleanup semantics inconsistent, and lacks required real behavior proof.

## What This Changes

The PR changes media parsing to skip `MEDIA:` lines in indented code blocks, preserve code indentation during cleanup, tighten markdown fence closing validation, and add parser/fence regression tests.

## Best Possible Solution

Update this PR to preserve lowercase MEDIA directives, align streaming and audio cleanup with the code-aware parser, then add redacted CLI or Telegram proof and land with focused parser/fence coverage.

## Reproduction Assessment

Yes, source-level. Current main trims leading indentation before the case-insensitive `MEDIA:` gate, so a blank-preceded 4-space or tab-indented code line can enter extraction; I did not run tests in this read-only pass.

## Solution Assessment

No, not as submitted. The indented-code guard and fence validation are the right direction, but the branch should preserve lowercase directive support and reuse aligned code-aware cleanup rules for streaming and audio paths.
