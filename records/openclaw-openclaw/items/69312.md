## Summary

Keep open: the central indented-code bug is valid, but the current PR head still introduces parser regressions around lowercase MEDIA semantics and audio-tag cleanup, and it lacks required real behavior proof.

## What This Changes

The PR adds an indented-code heuristic to `splitMediaFromOutput`, tightens markdown fence closing detection, removes the MEDIA regex case-insensitive flag, and adds parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips MEDIA extraction inside indented code while preserving lowercase/streaming semantics and code indentation after audio tags, with regression tests and redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `splitMediaFromOutput` excludes fenced spans only, then uses a case-insensitive `trimStart()` MEDIA prefix check, which gives a concrete current-main path for indented `MEDIA:` false extraction.

## Solution Assessment

No. The indented-code guard is in the right owner path, but the current PR also changes established lowercase MEDIA behavior and applies non-code leading normalization after audio-tag stripping.
