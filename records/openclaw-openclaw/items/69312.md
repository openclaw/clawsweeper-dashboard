## Summary

Keep open. The linked issue https://github.com/openclaw/openclaw/issues/69311 is a real parser bug and this PR touches the right surface, but the current head still regresses established lowercase media directives, diverges from streaming buffering, can strip tab indentation after an audio tag, and lacks after-fix real behavior proof.

## What This Changes

The PR adds indented-code detection to `splitMediaFromOutput`, tightens fence parsing, adjusts media cleanup, and adds parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips indented code blocks while preserving lowercase directive support and keeping streaming/audio cleanup rules aligned, with focused regressions and redacted real behavior proof.

## Reproduction Assessment

Yes, source-level. On current main, `splitMediaFromOutput` only excludes fenced spans before a case-insensitive trimmed `MEDIA:` gate, so a blank-preceded 4-space/tab indented `MEDIA:` line reaches extraction; I did not run tests because this was a read-only review.

## Solution Assessment

No. The parser is the right narrow area, but this PR should not merge until lowercase semantics, streaming buffering, and audio-tag cleanup all follow the same directive/code rules.
