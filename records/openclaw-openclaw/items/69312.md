## Summary

Keep open: the reported indented-code MEDIA false positive is a valid bug, but this PR still has blocking parser regressions around lowercase directives, streaming buffering, and audio-tag cleanup, plus it lacks required real behavior proof.

## What This Changes

The PR updates MEDIA parsing to skip MEDIA-looking lines in indented code blocks, tightens markdown fence closing validation, adjusts cleanup, and adds media/fence regression tests.

## Best Possible Solution

Land a focused parser fix that preserves lowercase MEDIA support, aligns streaming buffering with the batch indented-code rules, reuses code-aware cleanup after audio tag removal, and includes redacted CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main only excludes fenced spans before a case-insensitive MEDIA line gate, so a blank-preceded four-space or tab-indented MEDIA-looking code line can be parsed as an attachment.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the patch should preserve lowercase directive behavior and keep streaming/audio cleanup semantics aligned with the new parser behavior.

  - confidence: 0.87

  - confidence: 0.86
