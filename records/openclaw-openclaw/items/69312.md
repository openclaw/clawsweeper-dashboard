## Summary

Keep open: this PR targets a real media-parser bug, but the current head still introduces parser compatibility regressions and lacks the required real behavior proof for an external contribution.

## What This Changes

The PR changes media and markdown fence parsing so indented-code `MEDIA:` lines are preserved, malformed fence closers stay open, cleanup tries to preserve retained code indentation, and parser tests are added.

## Best Possible Solution

Land a focused parser fix after preserving the lowercase directive contract, aligning streaming and audio cleanup with the same code-aware rules, validating shared fence consumers, and adding redacted real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main only skips fenced spans before a case-insensitive `MEDIA:` line gate, so a blank-preceded 4-space or tab-indented `MEDIA:` line can be extracted; no tests were run in this read-only review.

## Solution Assessment

No. The parser is the right narrow surface, but this patch should not merge while lowercase media parsing, streaming buffering, and audio-tag cleanup use inconsistent directive rules.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve lowercase media directive parsing:** `src/media/parse.ts:17`
  - body: Current block-reply parsing intentionally enters directive parsing for `/media:/i`, and the existing test expects `media: ./report.pdf` to become `mediaUrl`/`mediaUrls`. Removing the regex `i` flag and making `splitMediaFromOutput` case-sensitive turns that supported directive form into visible text instead of an attachment.
  - confidence: 0.95
- **[P2] Align streaming buffering with parser rules:** `src/media/parse.ts:573`
  - body: The parser now rejects lowercase `media:` and skips indented-code directive lines, but `splitTrailingDirective` still buffers `/^\s*MEDIA:/i` tails and partial prefixes case-insensitively. Streamed lowercase or indented-code text can be held and later dropped instead of following the non-streaming parser behavior.
  - confidence: 0.9
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:745-748`
  - body: After `parseAudioTag` removes a leading `[[audio_as_voice]]`, this cleanup always applies the non-code leading-normalization regex. A first surviving tab-indented code line can lose its tab, breaking the PR's own indentation-preservation goal for audio-tagged replies.
  - confidence: 0.88
