## Summary

Keep open. The reported bug is source-reproducible on current main, but this PR head still regresses the existing lowercase `media:` contract, leaves streaming directive buffering out of sync with the new indented-code rule, applies non-code audio cleanup to code-aware text, and lacks required real behavior proof.

## What This Changes

The PR changes media and markdown fence parsing to skip MEDIA extraction in 4-space/tab indented code blocks, preserve code indentation while stripping media/audio directives, tighten fence closing validation, and add parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips CommonMark indented code blocks while preserving current lowercase directive semantics and keeping streaming/audio cleanup aligned, backed by regression tests and real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main skips fenced spans but has no indented-code guard before `trimStart().toUpperCase().startsWith("MEDIA:")`, so a blank-preceded 4-space or tab-indented `MEDIA:` line can still reach extraction.

## Solution Assessment

No. The parser is the right owner boundary, but this PR should preserve existing lowercase behavior and update streaming/audio cleanup to match the new code-aware parsing rules before merge.
