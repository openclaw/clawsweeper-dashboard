Full review comments:

- **[P2] Preserve code indentation after audio tag removal:** `src/media/parse.ts:747`
  - body: After the new code-aware cleanup preserves an indented code line, this `[[audio_as_voice]]` branch re-runs a non-code-aware leading regex. For input like `[[audio_as_voice]]\n\n\tMEDIA:/tmp/code.png`, `parseAudioTag` can return a leading newline plus the tab-indented code line, and this regex then consumes the newline and the tab, turning the preserved code text back into an unindented directive-shaped line. Reuse the `firstLineIsCode`/`keptLineIsCode` normalization here and add tab and 4-space regressions for the audio-tag path.
  - confidence: 0.9
- **[P2] Keep streaming buffering in sync with indented code:** `src/media/parse.ts:563`
  - body: This branch makes blank-preceded indented `MEDIA:` lines plain text in final parsing, but `splitTrailingDirective` still buffers any optional-whitespace `MEDIA:` last line as a directive tail. A streamed reply ending a chunk with an indented code example can be withheld from partial delivery, and if that accumulator is reset before final flush the text can be dropped; update the streaming classifier and add a streamed regression for the new preserved-code case.
  - confidence: 0.86
- **[P2] Correct the lowercase-prefixed media expectation:** `src/media/parse.test.ts:425`
  - body: The new test expects `media: note MEDIA:/tmp/a.png` to produce `mediaUrls`, but `MEDIA_TOKEN_RE` is greedy from the lowercase prefix and captures `note MEDIA:/tmp/a.png` as one payload, so the current parser does not produce a valid URL from that input. Decide whether that line should be text or a directive, then update the parser and assertion together so the added test does not fail or encode a mismatched contract.
  - confidence: 0.88
