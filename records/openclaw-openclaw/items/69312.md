## Summary

Keep this PR open, but it is not merge-ready. The core indented-code MEDIA false-positive is real and the PR takes the right general direction, but the latest head still regresses established lowercase `media:` behavior and applies non-code-aware leading whitespace cleanup after `[[audio_as_voice]]` removal.

## What This Changes

The PR changes media and markdown fence parsing to skip indented-code `MEDIA:` lines, preserve retained code indentation during cleanup, reject invalid fence closers with trailing text, make MEDIA token matching case-sensitive, and add parser regression tests.

## Best Possible Solution

Land a focused version that preserves indented-code `MEDIA:` lines without changing established lowercase media directive compatibility, reuses the same code-aware normalization after audio-tag stripping, keeps or drops the fence-closing tightening based on downstream validation, and adds the required active-version changelog entry.

## Reproduction Assessment

Yes. On current main, a blank line followed by a four-space or tab-indented `MEDIA:` line reaches `splitMediaFromOutput` because only fenced spans are skipped; the PR defects are also source-reproducible with `media: ./report.pdf` and `[[audio_as_voice]]` before an indented code line.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow repair, but the case-sensitivity change and post-audio cleanup path need to be corrected before this is the best maintainable fix.
