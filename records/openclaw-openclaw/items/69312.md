## Summary

Keep this PR open, but it is not merge-ready: the reported bug is valid on current main, while the branch still regresses the existing lowercase `media:` directive contract, leaves streaming/audio cleanup mismatches, and lacks required real behavior proof.

## What This Changes

The PR changes MEDIA and fence parsing to skip MEDIA extraction from indented code blocks, preserve retained code whitespace, make MEDIA matching case-sensitive, tighten fence close validation, and add parser tests.

## Best Possible Solution

Land a focused parser fix that preserves the lowercase media directive contract, aligns streaming buffering with code-aware parsing, keeps audio-tag cleanup code-aware, and includes redacted real OpenClaw proof.

## Reproduction Assessment

Yes, source-level: current main only skips fenced spans before MEDIA extraction and then checks `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so a blank-preceded 4-space/tab MEDIA line can be extracted. I did not execute local tests because this review was read-only.

## Solution Assessment

No. The indented-code guard is the right direction, but the current branch is not the narrowest maintainable fix until lowercase semantics, streaming buffering, and audio-tag cleanup match the existing parser contracts.
