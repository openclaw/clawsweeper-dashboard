The PR changes media and fence parsing to preserve indented-code `MEDIA:` examples, tighten closing-fence recognition, and add parser regression tests.

## Best Possible Solution

Land a focused parser fix that preserves indented code examples without changing established lowercase directive compatibility, applies code-aware cleanup consistently after audio-tag removal, and records the user-facing fix in the active changelog.

Yes. Current main source inspection shows indented `MEDIA:` lines reach extraction unless they are inside fenced spans; the PR regressions are source-reproducible with lowercase `media: ./report.pdf` and an audio tag followed by a tab-indented code line.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the case semantics and post-audio cleanup need to stay aligned with existing parser, block-reply, and streaming behavior.
