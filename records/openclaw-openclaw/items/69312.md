## Summary

Keep open. Current main still has the reported parser bug and this PR targets the right surface, but the current head is not mergeable because it changes the existing lowercase MEDIA contract, leaves streaming buffering inconsistent with the new indented-code rule, still strips indentation after audio-tag cleanup, and lacks required real behavior proof.

## What This Changes

The PR changes media and Markdown fence parsing to skip MEDIA directives in indented code blocks, tighten closing-fence validation, alter MEDIA case matching, and add parser regression tests.

## Best Possible Solution

Land a narrowed parser fix that preserves the existing lowercase contract or migrates it deliberately across callers, applies the indented-code rule consistently to streaming, keeps code indentation through audio-tag cleanup, and includes focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main only excludes fenced spans before a case-insensitive MEDIA line gate, so a blank-preceded 4-space or tab-indented MEDIA line can be extracted; I did not execute tests because this review was constrained to read-only inspection and `pnpm` is unavailable in the checkout.

## Solution Assessment

No. Guarding `splitMediaFromOutput` is the right boundary, but this PR is not yet the best mergeable fix because it breaks lowercase compatibility, does not align streaming buffering, and uses non-code cleanup after audio tag stripping.
