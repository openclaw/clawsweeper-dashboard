## Summary

Keep open. The PR addresses a real media-parser bug, but the current head still regresses established lowercase `media:` directive support, leaves streaming tail buffering out of sync with the new indented-code parser semantics, and lacks required real behavior proof from the contributor.

## What This Changes

The PR changes media parsing to skip 4-space/tab indented `MEDIA:` lines, preserve code indentation during cleanup, validate fenced-code closing lines, remove case-insensitive media matching, and add parser/fence tests.

## Best Possible Solution

Land the indented-code and fence parsing fixes only after preserving lowercase directives, aligning streaming tail buffering with the parser contract, and adding focused parser/reply coverage plus real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main skips only fenced spans and then applies a case-insensitive `trimStart()` `MEDIA:` gate, so indented code lines can be extracted as media; I did not run tests under the read-only review policy.

## Solution Assessment

No, not as submitted. The indented-code guard and fence validation are directionally right, but the PR should preserve lowercase media directives and update streaming buffering before merge.
