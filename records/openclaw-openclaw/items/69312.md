## Summary

Keep open. The original bug is source-reproducible and the indented-code direction is useful, but latest head still regresses existing lowercase MEDIA semantics, leaves streaming buffering inconsistent with the parser, applies non-code cleanup after audio tags, and lacks required real behavior proof.

## What This Changes

The branch updates MEDIA/fence parsing to treat 4-space/tab-indented MEDIA lines as code, adjust whitespace and case handling, and add media/fence parser tests.

## Best Possible Solution

Land a focused parser fix that skips MEDIA extraction from indented code blocks while preserving established lowercase directives, keeping streaming buffering in sync, reusing code-aware cleanup after audio tags, and adding redacted real behavior proof.

## Reproduction Assessment

Yes, source-level. On current main, splitMediaFromOutput only excludes fenced spans and then applies a trimStart case-insensitive MEDIA gate, so a blank-preceded 4-space/tab indented MEDIA line can be extracted; I did not run tests because this review was read-only.

## Solution Assessment

No. The parser guard is the right narrow area, but the proposed patch is not the best merge candidate until lowercase MEDIA semantics, streaming buffering, and audio-tag cleanup are aligned with existing contracts.
