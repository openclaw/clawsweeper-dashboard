## Summary

Keep open. This is a valid current-main parser bug fix path, but the latest PR head still introduces a streaming text-loss regression because media parsing becomes case-sensitive while the streaming tail buffer remains case-insensitive; the PR also still lacks required after-fix real behavior proof.

## What This Changes

The PR adds indented-code protection and code-aware whitespace normalization to `splitMediaFromOutput()`, tightens markdown fence closing validation, removes case-insensitive `MEDIA:` matching, and adds parser/fence regression tests.

## Best Possible Solution

Land a revised parser fix that also aligns `streaming-directives.ts` with the chosen `MEDIA:` case contract and includes redacted real behavior proof from an installed OpenClaw CLI/channel run; keep the separate fenced-`MEDIA:` product contract in https://github.com/openclaw/openclaw/issues/41966.

## Reproduction Assessment

Yes by source inspection. Current main has a direct parser path where 4-space/tab indented `MEDIA:` lines are not treated as code before media extraction, and the PR finding is also source-reproducible from the case-sensitive parser diff plus case-insensitive streaming buffer regexes.

## Solution Assessment

No, not as currently written. The indented-code guard is the right narrow direction, but the case-sensitivity cleanup must either update the streaming accumulator and tests or be reverted so parser and streaming behavior stay aligned.

## Review Findings
