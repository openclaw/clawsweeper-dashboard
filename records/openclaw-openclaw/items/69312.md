## Summary

Keep open: this is a real parser bug, but the current PR head is not merge-ready because it regresses established lowercase MEDIA behavior, leaves streaming/audio cleanup inconsistent, and lacks required real behavior proof.

## What This Changes

The branch updates MEDIA and fence parsing to skip MEDIA extraction from indented code blocks, preserve retained code whitespace during cleanup, validate closing fences more strictly, make MEDIA matching case-sensitive, and add parser tests.

## Best Possible Solution

Land a focused parser fix that preserves existing MEDIA directive semantics, aligns streaming buffering with code-aware parsing, keeps audio-tag cleanup code-aware, and includes redacted real behavior proof before merge.

## Reproduction Assessment

Yes, source-level: current main only excludes fenced spans before MEDIA extraction and then checks line.trimStart().toUpperCase().startsWith("MEDIA:"), so a blank-preceded 4-space/tab MEDIA line can be extracted. I did not execute tests because this review was read-only.

## Solution Assessment

No. The indented-code guard is the right direction, but this patch is not the best merge candidate until case semantics, streaming buffering, and audio-tag cleanup are aligned with existing contracts.
