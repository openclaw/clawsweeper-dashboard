Full review comments:

- **[P2] Preserve code indentation after audio tag removal:** `src/media/parse.ts:747`
  - body: After the new code-aware cleanup preserves an indented code line, this `[[audio_as_voice]]` branch re-runs a non-code-aware leading regex. For input like `[[audio_as_voice]]\n\n\tMEDIA:/tmp/code.png`, the regex can consume the leading newline and tab, turning preserved code text back into an unindented directive-shaped line; reuse the `firstLineIsCode`/`keptLineIsCode` normalization here.
  - confidence: 0.9
- **[P2] Keep streaming buffering in sync with indented code:** `src/media/parse.ts:563`
  - body: This branch makes blank-preceded indented `MEDIA:` lines plain text in final parsing, but `splitTrailingDirective` still buffers any optional-whitespace `MEDIA:` last line as a directive tail. A streamed reply ending a chunk with an indented code example can be withheld from partial delivery, and if the accumulator is reset before final flush the text can be dropped.
  - confidence: 0.86
- **[P2] Correct the lowercase-prefixed media expectation:** `src/media/parse.test.ts:425`
  - body: The new test expects `media: note MEDIA:/tmp/a.png` to produce `mediaUrls`, but `MEDIA_TOKEN_RE` is greedy from the lowercase prefix and captures `note MEDIA:/tmp/a.png` as one payload, so the current parser does not produce a valid URL from that input. Decide the intended contract and update the parser and assertion together.
  - confidence: 0.88
