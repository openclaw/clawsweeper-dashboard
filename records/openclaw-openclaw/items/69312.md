## Summary

Keep this PR open, but it is not merge-ready: current main still has the indented-code MEDIA false-positive bug, while the branch introduces parser-contract regressions and still lacks required real behavior proof.

## What This Changes

The PR changes media and markdown fence parsing plus parser tests to ignore MEDIA directives in indented code blocks, preserve retained code indentation during cleanup, tighten closing fence validation, and make MEDIA token matching case-sensitive.

## Best Possible Solution

Land a focused parser fix that preserves the existing lowercase MEDIA contract, keeps streaming directive buffering aligned with batch parsing, reuses code-aware cleanup after audio-tag stripping, validates shared fence consumers, and includes redacted real CLI or Telegram proof.

## Reproduction Assessment

Yes. Source inspection of current main shows only fenced spans are skipped before MEDIA line extraction, so a 4-space or tab-indented MEDIA-looking code line can be parsed as an attachment; I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should not regress lowercase media directives or leave streaming and audio cleanup paths inconsistent with the new parser contract.
