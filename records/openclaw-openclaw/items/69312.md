## Summary

Keep open: this is a real parser bug and the indented-code direction is useful, but the current PR still regresses the existing lowercase `media:` directive contract and lacks required after-fix real behavior proof.

## What This Changes

The PR adds indented-code detection to `splitMediaFromOutput`, tightens `parseFenceSpans` closing validation, changes MEDIA matching to uppercase-only, and adds media/fence parser tests.

## Best Possible Solution

Land a focused parser fix that skips indented code blocks and preserves code indentation without changing lowercase MEDIA support, then validate the media parser plus shared fence consumers and add real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main only excludes fenced spans before `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so a 4-space or tab-indented `MEDIA:` line outside fences can be extracted as media.

## Solution Assessment

No, not as submitted. The indented-code guard is a reasonable narrow fix, but the branch also changes lowercase media directive semantics and leaves streaming buffering/tests inconsistent; the safer solution preserves case-insensitive media directives.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep lowercase MEDIA directives parseable:** `src/media/parse.ts:17`

Removing the `i` flag and using a case-sensitive line gate makes lowercase directives remain text while streaming buffering still treats media tails case-insensitively, regressing an existing directive contract.
