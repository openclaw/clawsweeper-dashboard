## Summary

Keep open. The reported false-positive extraction path is real and the core direction is useful, but the latest PR head still introduces blocking parser regressions and lacks external real behavior proof.

## What This Changes

The PR changes media parsing to skip `MEDIA:` directives in indented code blocks, preserve retained code indentation during cleanup, tighten markdown fence closing validation, and add parser/fence tests.

## Best Possible Solution

Land a focused parser fix that preserves indented-code text while maintaining lowercase `MEDIA:` support, aligning streaming buffering and audio cleanup, and adding redacted real CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main skips only fenced spans and then trims indentation before a case-insensitive `MEDIA:` check, so blank-preceded 4-space or tab-indented code lines can be extracted.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the patch should preserve lowercase directive support and keep streaming plus audio-tag cleanup semantics aligned.
