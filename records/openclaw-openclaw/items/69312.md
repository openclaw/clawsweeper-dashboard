## Summary

Keep open: this is a real MEDIA parser bug and the PR touches the right area, but the latest head still regresses existing lowercase MEDIA behavior, leaves streaming/audio cleanup paths inconsistent, has failing checks, and lacks required real behavior proof.

## What This Changes

The PR changes MEDIA and fence parsing to preserve indented-code MEDIA lines as text, protect retained code whitespace during cleanup, tighten closing-fence validation, remove case-insensitive MEDIA matching, and add parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips MEDIA extraction from indented code blocks while preserving current lowercase directive behavior, aligning streaming buffering, keeping audio cleanup code-aware, and validating shared fence consumers.

## Reproduction Assessment

Yes, source-level: current main trims indentation before MEDIA detection and only excludes fenced spans, so a blank-preceded 4-space/tab MEDIA line can be extracted. I did not execute tests under the read-only review constraint.

## Solution Assessment

No. The indented-code guard belongs in splitMediaFromOutput, but this PR is not yet the best solution because it changes established lowercase behavior, leaves streaming buffering inconsistent, and applies non-code cleanup after audio tag stripping.
