## Summary

Keep open. Current main still has the reported indented-code MEDIA extraction bug and this PR targets the right shared parser area, but the latest head still regresses established lowercase MEDIA directive behavior, leaves streaming buffering out of sync with the new parser rule, reintroduces non-code cleanup after audio tags, and lacks after-fix real behavior proof.

## What This Changes

The PR adds indented-code-block skipping and code-aware cleanup to media parsing, tightens markdown fence closing validation, and adds media/fence parser tests.

## Best Possible Solution

Land a focused parser fix that preserves indented-code examples without changing lowercase media directive semantics, aligns streaming buffering and audio cleanup with the same code-aware rules, and includes redacted real CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main only skips fenced spans before `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so a blank-preceded 4-space or tab-indented `MEDIA:` line can be extracted.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the patch should preserve lowercase directive support and reuse the same code-aware rules in streaming and audio-tag cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Preserve lowercase media directives:** `src/media/parse.ts:17`
  - body: Current main intentionally parses lowercase `media:` directives; `reply-delivery.test.ts` covers `media: ./report.pdf`, and commit f350bb4dfc62 introduced the behavior. Removing the regex `i` flag makes those block-reply attachments visible text instead of media.
  - confidence: 0.94

  - body: The parser now preserves blank-preceded indented `MEDIA:` lines as text, but streaming still holds any whitespace-prefixed `MEDIA:` tail with a case-insensitive regex. If reset happens before final flush, the text this PR is trying to preserve can be dropped.
  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:745-748`
  - body: The main cleanup skips leading whitespace stripping when the first surviving line is code, but after `parseAudioTag` this branch always applies the non-code leading regex. A `[[audio_as_voice]]` tag before a tab or 4-space code line can still strip retained code indentation.
  - confidence: 0.86
