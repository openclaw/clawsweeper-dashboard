## Summary

Keep this PR open: it targets a real source-reproducible MEDIA parsing bug, but the latest head still introduces blocking parser-contract regressions and lacks required after-fix real behavior proof.

## What This Changes

The PR changes media directive parsing to skip indented code-block lines, tightens markdown fence closing detection, and adds media/fence parser tests.

## Best Possible Solution

Land a focused parser fix that preserves lowercase MEDIA semantics, aligns streaming buffering and audio-tag cleanup with the same code-line rule, and validates shared fence consumers.

## Reproduction Assessment

Yes, source-level: current main only excludes fenced spans before `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so a 4-space or tab-indented `MEDIA:` line outside fences can be parsed as media. I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should preserve lowercase MEDIA behavior, update streaming buffering to the same boundary, and reuse code-aware cleanup after audio tags.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve lowercase MEDIA directives:** `src/media/parse.ts:17`
  - body: Current main has block-reply coverage that `media: ./report.pdf` becomes `mediaUrl`/`mediaUrls`. Dropping the `i` flag and switching the line gate to case-sensitive `startsWith("MEDIA:")` makes that supported directive visible text instead, while callers still pre-detect `/media:/i`; keep the case semantics consistent or update every caller and test deliberately.
  - confidence: 0.94
- **[P2] Align streaming buffering with indented-code parsing:** `src/media/parse.ts:562-563`
  - body: The new `isIndented` branch makes 4-space/tab `MEDIA:` lines after a blank plain text, but `createStreamingDirectiveAccumulator` still buffers whitespace-prefixed `MEDIA:` and partial `MEDIA` tails under the old rule. A streamed indented code example can be held, dropped, or parsed differently across chunks unless streaming boundary tests use the same code-line rule.
  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:745-747`
  - body: After `parseAudioTag`, this unconditional non-code leading regex can remove up to three spaces or tabs from the first surviving line even when `keptLineIsCode` marked it as code. Inputs with `[[audio_as_voice]]` before an indented code line can lose structural indentation and expose a retained `MEDIA:` example as top-level text.
  - confidence: 0.86
