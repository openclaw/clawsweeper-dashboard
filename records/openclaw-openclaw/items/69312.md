## Summary

Keep open. Current main still has the indented-code false-positive path, and this PR is useful source work for linked issue #69311, but the latest PR head is not merge-ready because it regresses existing lowercase `media:` directive behavior and still applies non-code-aware cleanup after removing `[[audio_as_voice]]`.

## What This Changes

The PR changes media parsing, shared Markdown fence parsing, and parser tests to preserve indented-code `MEDIA:` lines while tightening fence closing and directive case handling.

## Best Possible Solution

Land a focused parser fix that preserves indented-code `MEDIA:` text, keeps existing lowercase directive compatibility unless every consumer is deliberately migrated together, reuses one code-aware cleanup path after audio-tag removal, validates shared fence consumers, and adds the required active-version changelog entry.

## Reproduction Assessment

Yes. On current main, pass `splitMediaFromOutput` output containing a blank line followed by a four-space or tab-indented `MEDIA:` line; the source shows only fenced spans are skipped before the indented line is trimmed and parsed as a directive.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow direction, but the branch should preserve current lowercase `media:` behavior and apply code-aware normalization after audio-tag stripping before it is the best maintainable fix.
