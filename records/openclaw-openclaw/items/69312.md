## Summary

## What This Changes

The PR adds indented-code-block guards to MEDIA parsing and streaming buffering, tightens markdown fence closing validation, and adds media, fence, and streaming regression tests.

## Best Possible Solution

Align streaming buffering with the final parser's full indented-code state, then land after real OpenClaw or Telegram delivery proof and maintainer acceptance of the compatibility and fence-parser behavior.

## Reproduction Assessment

Yes, at source level. Current main trims indentation before MEDIA matching and only protects fenced code; I did not execute tests in this read-only review.

Not yet. The parser direction is narrow, but the streaming guard must mirror the final parser's indented-code state before this is the safest fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep streaming indented-code detection in sync:** `src/auto-reply/reply/streaming-directives.ts:81-83`
  - body: The final parser treats an indented line as code when the previous line was blank or another indented-code line, but this streaming guard only checks start-of-text or a blank-line predecessor. For a chunk ending with `Example:\n\n    first\n    MEDIA:/tmp/code.png`, this buffers the MEDIA-looking code line as a directive tail; if the stream resets before final flush, that visible code line can be dropped, and even normal continuation no longer matches final parsing. Carry enough previous-line state here and add a regression test for a MEDIA line after another indented code line.
