## Summary

Keep open: the underlying indented-code media extraction bug is valid on current main, but this PR is not merge-ready because the latest head still makes lowercase `media:` handling inconsistent across parser, block-reply, and streaming surfaces, and one cleanup path can still strip tab indentation from retained code after an audio tag.

## What This Changes

The PR updates media and fence parsing to preserve indented-code `MEDIA:` lines, protect retained code indentation, tighten fence closing validation, change media token case matching, and add media/fence regression tests.

## Best Possible Solution

Land a focused version that preserves indented-code `MEDIA:` lines and retained code indentation while keeping existing lowercase directive semantics consistent across parser, block-reply, and streaming paths unless a coordinated product change updates all surfaces and tests. Add the required Unreleased changelog fix entry and validate the shared fence-parser consumers before merge.

## Reproduction Assessment

Yes. Current main has a source-level reproduction with output containing a blank line followed by a four-space or tab-indented `MEDIA:` line: the parser checks only fenced spans, then trims indentation and treats the line as a directive.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow direction, but the case-sensitivity change and the post-audio cleanup path are not the safest maintainable implementation without coordinated updates across the other directive surfaces.
