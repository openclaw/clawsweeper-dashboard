## Summary

Keep open. The PR targets a real media-parser bug, but the latest head is not merge-ready because it regresses existing lowercase `media:` directive behavior and still has a code-indentation cleanup gap after audio-tag stripping.

## What This Changes

The PR adds indented-code protection to media parsing, tightens markdown fence closing validation, adjusts MEDIA token case handling/cleanup, and adds media/fence parser tests.

## Best Possible Solution

Land a focused parser fix that preserves indented-code MEDIA examples without changing lowercase directive semantics unless every consumer is migrated, reuses code-aware cleanup after audio-tag stripping, and includes the required changelog plus targeted shared-parser validation.

## Reproduction Assessment

Yes. Current main can reproduce the original bug through `splitMediaFromOutput` with a blank line followed by an indented `MEDIA:` line, and the latest PR head has a CI annotation reproducing the lowercase directive regression in `reply-delivery.test.ts`.

## Solution Assessment

No, not as submitted. The indented-code guard is the right narrow direction, but the case-semantics change and second cleanup path need repair before this is the maintainable fix.
