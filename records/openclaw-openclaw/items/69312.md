## Summary

Keep open: current main still has the source-level false-positive path for indented `MEDIA:` code examples, but the latest PR diff still regresses the existing lowercase `media:` directive contract and lacks required real behavior proof.

## What This Changes

The PR updates `splitMediaFromOutput` and `parseFenceSpans` to skip `MEDIA:` extraction from indented code blocks, preserve some code indentation, tighten fence closing validation, and add parser tests.

## Best Possible Solution

Keep the indented-code protection, preserve the existing lowercase media directive contract across parser and streaming paths, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, source-level. Current main only guards fenced code spans and then trims indentation before a case-insensitive `MEDIA:` prefix check, so a 4-space or tab-indented code-block line can still be extracted as media.

## Solution Assessment

No, not yet. The indented-code heuristic is a reasonable narrow fix, but the case-sensitivity cleanup should be removed or handled as an explicit contract change because lowercase media directives are already tested behavior.
