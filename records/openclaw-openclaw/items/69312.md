## Summary

Keep open: the PR targets a real MEDIA parsing bug, but the current patch still regresses established lowercase directive behavior, leaves streaming buffering out of sync with the new parser semantics, has an audio-tag indentation edge case, and only provides automated test proof.

## What This Changes

The PR changes MEDIA parsing to skip 4-space/tab indented code lines, preserve code indentation during cleanup, tighten fence closing validation, remove case-insensitive MEDIA matching, and add parser/fence regression tests.

## Best Possible Solution

Land a focused parser fix that preserves lowercase MEDIA support, aligns streaming tail buffering with the batch indented-code rules, reuses code-aware cleanup after audio tag removal, and includes redacted real CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main only excludes fenced spans before a case-insensitive MEDIA line gate, so a blank-preceded four-space or tab-indented MEDIA-looking code line can be parsed as an attachment.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the patch should preserve lowercase directive behavior and keep streaming/audio cleanup semantics aligned with the new parser behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve lowercase MEDIA directive support:** `src/media/parse.ts:17`
  - body: Current main has block-reply coverage for `media: ./report.pdf`, and commit f350bb4dfc62 introduced case-insensitive MEDIA parsing. Removing the `i` flag and switching the line gate to `startsWith("MEDIA:")` makes lowercase directives visible text instead of attachment metadata.
  - confidence: 0.94
- **[P2] Align streaming buffering with the new parser rules:** `src/media/parse.ts:563`
  - body: Batch parsing now keeps blank-preceded indented `MEDIA:` lines as text and treats lowercase `media:` as text, but `splitTrailingDirective` still buffers any whitespace-prefixed `MEDIA:` tail case-insensitively. If such a tail lands at a stream boundary and the stream resets before final flush, text that should display can be dropped.
  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:745-748`
  - body: The main cleanup chooses a code-aware leading regex, but after `parseAudioTag` this branch always applies the non-code regex. With `[[audio_as_voice]]` before a tab-indented code line, the first code line loses its tab indentation even though the parser kept it as code.
  - confidence: 0.86
