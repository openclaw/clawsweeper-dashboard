## Summary

Keep this PR open: it targets a real current-main bug, but the latest head is still not merge-ready. The indented-code protection is the right direction, but the patch still makes lowercase `media:` handling inconsistent with existing block-reply and streaming surfaces, can strip protected code indentation after `[[audio_as_voice]]` cleanup, and lacks the required user-facing changelog entry.

## What This Changes

The PR changes media parsing to preserve indented-code `MEDIA:` lines, protect retained code indentation during cleanup, tighten markdown fence closing validation, make `MEDIA:` token matching case-sensitive, and add media/fence regression tests.

## Best Possible Solution

Land a focused parser fix that preserves indented-code `MEDIA:` lines while keeping existing directive semantics consistent across `splitMediaFromOutput`, block replies, and streaming buffering; apply the same code-aware cleanup after audio-tag stripping; keep the fence-parser change only with shared-consumer validation; and add the required Unreleased changelog fix entry.

Yes. Current main is source-reproducible with output containing a blank line followed by a 4-space or tab-indented `MEDIA:` line, and the PR regressions are source-backed by existing lowercase block-reply coverage plus the PR head's post-audio normalization path.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow approach, but the case-sensitivity change and non-code-aware audio cleanup are not the safest maintainable way to solve the reported bug.
