## Summary

Keep this PR open: it addresses a real current-main parser bug, but the live patch still regresses established lowercase `media:` behavior and has no after-fix real behavior proof.

## What This Changes

The PR changes media and fence parsing so indented-code `MEDIA:` lines are preserved, fence closers require whitespace-only tails, parser `MEDIA:` matching becomes case-sensitive, and regression tests cover those cases.

## Best Possible Solution

Land a focused parser fix that protects indented-code `MEDIA:` lines and code indentation while preserving current lowercase media semantics across parser, block replies, streaming buffering, and tests.

## Reproduction Assessment

Yes, by source inspection. Current main has no indented-code guard before the `MEDIA:` line check and still globally folds/trims kept lines, which matches the reported false-positive extraction and indentation-loss paths.

## Solution Assessment

No, not as currently proposed. The indented-code direction is narrow, but the case-sensitivity and audio-tag normalization changes need to be corrected before this is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

  - body: Removing the case-insensitive parser match makes lowercase `media:` stay as text, but block-reply tests still expect lowercase directives to produce media and streaming buffering still treats lowercase tails as directives.
