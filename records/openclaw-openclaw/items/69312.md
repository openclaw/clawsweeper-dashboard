## Summary

Keep open: the reported bug is valid and the PR has a useful direction, but the current head still introduces blocking parser/streaming regressions and lacks required real behavior proof.

## What This Changes

The PR updates MEDIA parsing to skip 4-space/tab indented code blocks, tighten fenced-code closing validation, remove case-insensitive MEDIA matching, and add parser/fence tests.

## Best Possible Solution

Land a narrower parser fix that treats true indented code blocks as text while preserving existing lowercase directive semantics and sharing boundary/cleanup rules with streaming and audio-tag paths.

## Reproduction Assessment

Yes, source-level: current main only checks fenced spans before `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so a 4-space or tab-indented `MEDIA:` line outside fences can be parsed as media. I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should preserve lowercase MEDIA semantics, update streaming buffering to the same boundary, and reuse code-aware cleanup after audio tags.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve lowercase MEDIA directives:** `src/media/parse.ts:17`
  - body: Current main has regression coverage that lowercase `media: ./report.pdf` in block replies becomes `mediaUrl`/`mediaUrls`. Removing the regex `i` flag turns that supported directive into visible text instead of an attachment, so keep case-insensitive MEDIA parsing unless the broader contract is deliberately changed.
  - confidence: 0.94
- **[P2] Align stream buffering with code-line detection:** `src/media/parse.ts:563`
  - body: The parser now preserves indented `MEDIA:` lines as code, but the streaming accumulator still buffers whitespace-prefixed MEDIA tails under the old rule. A streamed indented code example can be held, dropped on reset, or parsed differently across chunks unless `streaming-directives.ts` and its tests use the same code-line boundary.
  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:747`
  - body: The main cleanup chooses a safe leading regex when the first surviving line is code, but the `audio_as_voice` post-processing always applies the non-code regex. With `[[audio_as_voice]]` before a tab- or four-space-indented code line, this can strip structural indentation and expose retained MEDIA-shaped text incorrectly.
  - confidence: 0.86
