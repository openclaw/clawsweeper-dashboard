## Summary

Keep open. This is a plausible fix for a real parser bug, but the current PR head still breaks the established lowercase `media:` contract, leaves streaming buffering out of sync with the new parser rules, and still lacks contributor real behavior proof.

## What This Changes

The PR changes media parsing to skip indented-code `MEDIA:` lines, preserve retained code indentation during cleanup, tighten fence closing validation, make `MEDIA:` matching case-sensitive, and add media/fence parser tests.

## Best Possible Solution

Land a focused parser fix that skips indented-code false positives while preserving lowercase directive behavior and keeping streaming, audio cleanup, and shared fence-parser validation aligned.

## Reproduction Assessment

Yes, source-level: current main skips only fenced spans and then extracts any trimmed line whose uppercase prefix is `MEDIA:`, so blank-line-preceded 4-space or tab-indented code can be extracted. I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the patch should preserve the existing lowercase MEDIA contract and update streaming/audio cleanup paths to match the new parser rules.

## Review Findings

- **[P2] Preserve lowercase MEDIA directive parsing:** `src/media/parse.ts:17`
  - body: Current main has block-reply coverage that `media: ./report.pdf` becomes `mediaUrl` and `mediaUrls`. Removing the `i` flag and making the gate case-sensitive turns that supported directive into visible text, so keep parser case semantics aligned unless the whole contract is deliberately changed.
  - confidence: 0.94
- **[P2] Align streaming buffering with the new parser rules:** `src/media/parse.ts:562-563`
  - body: The parser now treats some whitespace-prefixed `MEDIA:` lines as code/plain text and uppercase-only, but `splitTrailingDirective` still buffers `^\s*MEDIA:` case-insensitively. A streamed lowercase or indented-code tail can be held in `pendingTail` and delayed or dropped on reset instead of flushing as ordinary text.
  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:745-748`
  - body: The first cleanup pass preserves indentation when `firstLineIsCode` is true, but after `parseAudioTag` the branch applies an unconditional non-code leading regex. Inputs with `[[audio_as_voice]]` before an indented code line can lose up to three leading spaces or tabs from visible code text.
  - confidence: 0.87
