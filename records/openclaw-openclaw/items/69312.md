## Summary

Keep this PR open: it is the canonical fix for a real media-parser bug, but the current patch is not merge-ready because streaming directive buffering and post-audio-tag cleanup still do not share the new code-aware semantics, and the external PR still lacks real behavior proof.

## What This Changes

The PR changes media parsing and fence parsing to preserve `MEDIA:` text inside indented code blocks, tighten malformed fence handling, and add parser/fence regression tests.

## Best Possible Solution

Keep this PR as the canonical path, align streaming buffering and audio-tag cleanup with the same code-aware parser contract, add focused regressions for those paths, then require redacted real OpenClaw runtime proof before merge.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction path: `splitMediaFromOutput` trims indentation before matching `MEDIA:` and only skips fenced code spans, so 4-space/tab indented code examples can be extracted and removed.

## Solution Assessment

No, not yet. The parser-side indented-code guard is the right narrow direction, but the patch should also update streaming tail buffering and post-audio-tag cleanup to use the same code-aware semantics.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Keep streaming buffering in sync with code blocks:** `src/media/parse.ts:562-563`
  - body: The new indented-code skip makes blank-line 4-space/tab `MEDIA:` lines visible text, but `splitTrailingDirective` still buffers any `/^\s*MEDIA:/i` final line. A streamed chunk ending with `\n\n    MEDIA:/tmp/code` can stay in `pendingTail` and then be dropped by `reset()` instead of being emitted, so the same code example can still disappear in streamed replies.
  - confidence: 0.86
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:745-748`
  - body: The main cleanup chooses `leadingNormRegex` based on whether the first surviving line is code, but this branch hard-codes the non-code leading normalizer after `parseAudioTag()`. For `[[audio_as_voice]]\n\n\tMEDIA:/tmp/code.png`, the tab-indented line is retained as code, then tag removal makes it first visible text and this regex strips the tab.
  - confidence: 0.88
