## Summary

Keep open: current main still has a source-reproducible indented-code MEDIA false positive, but the PR is not mergeable as-is because it regresses existing lowercase media directive behavior, leaves streaming buffering out of sync with the parser, strips code indentation after audio-tag cleanup, and still lacks real behavior proof.

## What This Changes

The PR changes media/fence parsing and tests to skip `MEDIA:` extraction from CommonMark-style indented code blocks, tighten malformed fence closing, adjust cleanup whitespace, and make MEDIA token matching case-sensitive.

## Best Possible Solution

Keep the focused parser fix, but align parser and streaming directive predicates, preserve the established lowercase directive contract unless it is intentionally retired everywhere, reuse code-aware cleanup after audio tags, and add redacted real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main only excludes fenced spans before a case-insensitive `MEDIA:` line gate, so a blank-preceded 4-space/tab indented `MEDIA:` line can be extracted; I did not run tests because this was a read-only review.

## Solution Assessment

No. The parser is the right narrow area, but the current patch should not merge until lowercase semantics, streaming buffering, and audio-tag cleanup all follow the same directive/code rules.
