## Summary

Keep open: the bug is valid on current main, but the latest PR head still changes the established lowercase media directive contract, leaves parser-adjacent edge cases, and lacks required real behavior proof.

## What This Changes

The PR changes media parsing to skip `MEDIA:` extraction from indented code blocks, tightens markdown fence closing validation, changes directive case handling, and adds parser/fence regression tests.

## Best Possible Solution

Land a focused parser fix that prevents indented-code false positives while preserving the existing lowercase media directive contract and keeping streaming plus audio-tag cleanup semantics aligned.

## Reproduction Assessment

Yes, source-level: current main only excludes fenced spans before parsing a trimmed, case-insensitive media line, so an indented code example can still be treated as a media directive. I did not execute tests because this review is read-only.

## Solution Assessment

No, not as submitted. The indented-code and fence-validation direction is plausible, but the branch should not retire lowercase `media:` semantics without a deliberate contract change across parser, streaming, docs, and tests.

Full review comments:

- **[P2] Preserve lowercase media directive support:** `src/media/parse.ts:17`
  - body: Current main intentionally supports lowercase `media:` directives, including outbound reply coverage and a merged `fix(media): parse lowercase media directives` change.   - confidence: 0.95
- **[P2] Align streaming buffering with indented-code parsing:** `src/media/parse.ts:563`
  - body: This branch preserves indented-code `MEDIA:` lines as text, but the streaming accumulator still buffers whitespace-prefixed media-looking tails. A streamed 4-space or tab-indented code example at a chunk boundary can sit in `pendingTail` and be delayed or dropped on reset, so the streaming path needs the same code-aware rule or a conservative flush path.
  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:747`
  - body: The main cleanup path preserves leading indentation when the first surviving line is code, but the `parseAudioTag` branch applies an unconditional leading-normalization regex. With `[[audio_as_voice]]` before a tab-indented code line, that tab can still be stripped even though the line was retained as code.
  - confidence: 0.87
