## Summary

Keep open: the reported indented-code MEDIA bug is valid and the PR is useful, but the current head regresses existing lowercase media directive behavior, leaves related streaming/audio cleanup paths inconsistent, has failing CI, and still lacks real behavior proof.

## What This Changes

The PR adds indented-code MEDIA guards, fence closing validation, whitespace-preservation tests, and changes MEDIA directive matching to be case-sensitive.

## Best Possible Solution

Land a focused parser fix that prevents indented-code false positives while preserving lowercase MEDIA support and aligning streaming plus audio-tag cleanup semantics.

## Reproduction Assessment

Yes, source-level: current main only skips fenced spans before case-insensitive MEDIA extraction, so a 4-space or tab-indented code example reaches extraction. The PR's lowercase regression is also reproduced by the current GitHub CI shard.

## Solution Assessment

No, not as submitted. The indented-code and fence validation direction is plausible, but the branch should preserve the existing lowercase directive contract and apply the same code-aware rules to streaming and audio-tag cleanup.

- **[P2] Preserve lowercase media directive support:** `src/media/parse.ts:17`
  - body: Current main has block-reply coverage that parses `media: ./report.pdf` into `mediaUrl` and `mediaUrls`.   - confidence: 0.96

  - body: This branch keeps 4-space/tab indented `MEDIA:` lines as text, but `splitTrailingDirective` still buffers any whitespace-prefixed `MEDIA:` tail. A streamed code example at a chunk boundary can sit in `pendingTail` and be dropped on reset instead of flushing as ordinary text, so the streaming guard needs the same indented-code rule or a conservative flush path.
  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:745-748`
  - body: The main cleanup path chooses a code-safe leading regex, but the `parseAudioTag` branch always uses prose normalization. With `[[audio_as_voice]]` before a tab-indented code line, that tab can be stripped even though the line was retained as code, so reuse the first-line code decision after audio-tag removal too.
  - confidence: 0.87
