## Summary

Keep open: current main still has the source-level MEDIA false-positive path, but this PR should not merge as submitted because it changes tested lowercase MEDIA behavior, leaves streaming/audio cleanup inconsistent, and lacks real behavior proof.

## What This Changes

The PR updates media parsing to skip `MEDIA:` inside indented code blocks, tightens fenced-code closing detection, and adds media/fence parser regression tests.

## Best Possible Solution

Land a focused parser fix that skips indented-code `MEDIA:` false positives while preserving existing lowercase MEDIA semantics and keeping streaming, audio cleanup, and fence consumers aligned.

## Reproduction Assessment

Yes, source-level: current main only excludes fenced spans before `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so a blank-line-preceded 4-space/tab `MEDIA:` line can enter extraction. I did not run the repro or tests because this review is read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should preserve the lowercase directive contract, align streaming buffering, and reuse code-aware cleanup after audio-tag stripping.
