## Summary

Keep this PR open: it targets a real current-main MEDIA parsing bug, and the contributor has actively addressed several earlier whitespace review comments, but the latest head still regresses the tested lowercase MEDIA contract, leaves streaming buffering out of sync with the new indented-code rule, mishandles tab-indented code after audio tags, and lacks required real behavior proof.

## What This Changes

The PR changes MEDIA parsing to skip indented-code MEDIA lines, preserve retained code-line whitespace, tighten markdown fence closing validation, remove case-insensitive MEDIA matching, and add media/fence parser tests.

## Best Possible Solution

Land a focused parser fix that preserves the existing lowercase MEDIA contract or updates all consumers together, aligns streaming tail buffering with indented-code detection, uses code-aware cleanup after audio tags, validates shared fence consumers, and includes redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows splitMediaFromOutput excludes fenced spans only, then uses a case-insensitive trimStart MEDIA check, giving a concrete path for 4-space or tab-indented MEDIA lines to be extracted.

## Solution Assessment

No. The indented-code guard belongs in splitMediaFromOutput, but this patch is not yet the best fix because it changes tested lowercase behavior, leaves streaming buffering misaligned, and applies non-code leading normalization after audio tag removal.

Overall confidence: 0.91

Full review comments:

- **[P2] Keep lowercase MEDIA directives supported:** `src/media/parse.ts:17`
  - body: Current main has block-reply coverage expecting `media: ./report.pdf` to populate `mediaUrl` and `mediaUrls`. Removing the `i` flag and requiring `startsWith("MEDIA:")` makes lowercase directives visible text while reply delivery still pre-scans `/media:/i`, so this breaks established behavior unless every caller and test is deliberately migrated together.
  - confidence: 0.93
- **[P2] Preserve code indentation after audio tags:** `src/media/parse.ts:747`
  - body: `firstLineIsCode` is computed before `parseAudioTag`, so an opening `[[audio_as_voice]]` tag makes the first real content line look non-code. The audio cleanup then applies the non-code leading-normalization regex; `[[audio_as_voice]]\n\n\tMEDIA:/tmp/code.png` loses the tab and can be reformatted as a top-level line.
  - confidence: 0.88
- **[P2] Align streaming buffering with indented-code parsing:** `src/media/parse.ts:563-569`
  - body: This new branch preserves indented MEDIA lines as text, but `splitTrailingDirective` still buffers whitespace-prefixed `MEDIA:` tails and `reset()` drops `pendingTail`. A streamed indented code example ending at a chunk boundary can disappear, so the streaming path needs the same indented-code/context rule or a safe flush path.
  - confidence: 0.83
