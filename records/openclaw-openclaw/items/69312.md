## Summary

Keep open: this is a focused bug-fix PR for a real current-main parser gap, but the latest branch still changes covered MEDIA case behavior, leaves streaming buffering out of sync with the new indented-code rule, and does not reuse the code-aware cleanup after audio-tag stripping.

## What This Changes

The PR changes MEDIA extraction and markdown fence parsing to skip 4-space/tab indented code blocks, preserve retained code indentation, tighten fence close validation, alter MEDIA case matching, and add parser tests.

## Best Possible Solution

Land a focused parser fix that preserves indented-code MEDIA text without regressing lowercase directives, streaming behavior, or audio-tag cleanup, then validate the shared fence consumers and add redacted real CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main trims indentation before the MEDIA gate after only excluding fenced spans, so a 4-space or tab-indented MEDIA-looking line can enter extraction; I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right narrow direction, but the branch must keep MEDIA case semantics aligned, update streaming buffering for the new code-aware rule, and reuse code-aware normalization after audio-tag removal.
