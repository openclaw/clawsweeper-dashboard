## Summary

Keep this PR open: it targets a valid current-main media parsing bug, but the latest head still has blocking parser regressions and lacks the required real behavior proof for the user-visible media path.

## What This Changes

The PR updates shared media/fence parsing to skip `MEDIA:` extraction from indented code blocks, tighten markdown fence closing validation, adjust media cleanup whitespace handling, and add parser tests.

## Best Possible Solution

Land a focused parser fix that protects indented code blocks while preserving the established lowercase media directive contract and reusing one code-aware cleanup rule across normal and audio-tag paths.

## Reproduction Assessment

Yes, source-level. Current main skips fenced spans but then trims indentation and uses a case-insensitive `MEDIA:` gate, so a blank-preceded 4-space or tab-indented code line can be extracted as media.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the patch should preserve lowercase directive support, keep streaming buffering aligned, and reuse the code-aware leading cleanup after audio tags.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

  - confidence: 0.88
