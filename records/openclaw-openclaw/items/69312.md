## Summary

Keep open. This is a valid focused bug fix for media-parser message loss, but the latest PR head still has a blocking audio-tag normalization edge case and no after-fix real behavior proof.

## What This Changes

The PR updates media directive parsing and markdown fence parsing so `MEDIA:` lines inside indented code blocks are preserved as text, with regression tests for media parsing and fence spans.

## Best Possible Solution

Keep this as the canonical fix path, reuse the same code-aware leading normalization after audio-tag stripping, add the tab-indented audio regression, and require redacted real runtime proof before merge.

## Reproduction Assessment

Yes, by source inspection. Current main trims indentation before the `MEDIA:` check and only skips fenced spans, so the issue-body input with a blank line followed by a 4-space or tab-indented `MEDIA:` line has a clear failing path.

## Solution Assessment

Mostly yes, but not as currently written. The indented-code guard is the right narrow parser fix, while the post-audio-tag cleanup should reuse the same code-aware normalization decision before this can merge.

- **[P2] Reuse code-aware normalization after audio tags:** `src/media/parse.ts:747`
  - body: The main cleanup path chooses `leadingNormRegex` based on whether the first surviving line is code, but this audio-tag branch always applies the non-code normalizer.
