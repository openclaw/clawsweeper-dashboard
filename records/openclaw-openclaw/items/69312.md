## Summary

Keep open. Current main still has the reported source-level media parsing bug, and this PR is a plausible fix path, but the latest head regresses the shipped lowercase media directive contract, leaves streaming buffering out of sync with the new parser rules, and still lacks contributor real behavior proof.

## What This Changes

This PR updates media parsing to skip MEDIA directives in indented code blocks, preserve code indentation during cleanup, tighten markdown fence closing validation, and add parser/fence regression tests.

## Best Possible Solution

Land a focused parser fix that preserves indented-code MEDIA text while retaining lowercase directive support and keeping streaming buffering plus audio-tag cleanup aligned with the same code-aware rules.

## Reproduction Assessment

Yes, source-level. Current main only exempts fenced spans before a case-insensitive trimmed MEDIA gate, so 4-space/tab indented code-block MEDIA lines can enter extraction; the PR regressions are also visible from the head diff and existing tests.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but changing lowercase directive semantics, not updating streaming buffering, and reapplying non-code cleanup after audio tags make this patch too broad and internally inconsistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve lowercase MEDIA directive support:** `src/media/parse.ts:17`
  - body: Current main and v2026.5.12 have regression coverage expecting lowercase `media:` in block replies to produce media fields, introduced by f350bb4dfc62. Removing the `i` flag turns that shipped directive form into visible text instead of an attachment.
  - confidence: 0.94
- **[P2] Align streaming buffering with indented-code parsing:** `src/media/parse.ts:562-563`
  - body: The parser now preserves blank-preceded indented `MEDIA:` lines as text, but the streaming accumulator still buffers any whitespace-prefixed `MEDIA:` tail under the old rule. A streamed code example split at a chunk boundary can sit in `pendingTail` and be dropped on reset instead of being displayed.
  - confidence: 0.87
- **[P2] Reuse code-aware leading cleanup after audio tags:** `src/media/parse.ts:745-748`
  - body: The first cleanup chooses `leadingNormRegex` based on whether the first content line is code, but the `parseAudioTag` branch always reapplies the non-code regex. With `[[audio_as_voice]]` before a tab-indented code line, the tab can be stripped and visible code formatting changes.
  - confidence: 0.86
