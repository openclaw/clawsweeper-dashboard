## Review Findings

Full review comments:

- **[P2] Align streaming buffering with indented-code parsing:** `src/media/parse.ts:562-569`
  - body: This new final-parser guard preserves blank-preceded 4-space or tab MEDIA lines as code, but `splitTrailingDirective` still buffers any trailing line that matches optional whitespace plus `MEDIA:`. A streamed reply ending at an indented code example can therefore withhold that visible text as a pending directive, and the existing reset path can drop pending tails before a final flush. Please update the streaming classifier and add a streamed regression for indented MEDIA code.
  - confidence: 0.82
- **[P2] Preserve tab-indented code after audio tag stripping:** `src/media/parse.ts:745-748`
  - body: After `[[audio_as_voice]]` is removed, this cleanup always applies the non-code leading-normalization regex. For `[[audio_as_voice]]\n\n\tMEDIA:/tmp/code.png`, the earlier parser state marked the tab-indented line as code, but this post-audio cleanup strips the leading tab and returns an unindented MEDIA-looking line, regressing visible code formatting and risking later reinterpretation. Reuse the code-aware leading regex here and cover the tab case.
  - confidence: 0.84
