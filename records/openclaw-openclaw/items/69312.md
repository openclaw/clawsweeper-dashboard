## Summary

Keep this PR open, but do not merge the current head. The core bug is valid and the direction is useful, but the branch still regresses an existing lowercase media contract, leaves streaming/audio cleanup rules inconsistent with the new parser behavior, and lacks required real behavior proof.

## What This Changes

The PR updates shared media and markdown fence parsing to skip `MEDIA:` extraction from indented or malformed code-block contexts, adjusts cleanup/case handling, and adds parser regression tests.

## Best Possible Solution

Land a focused parser fix that preserves lowercase media semantics, applies the indented-code rule consistently in batch parsing, streaming buffering, and audio-tag cleanup, then includes redacted real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main excludes fenced spans only and then trims leading indentation before a case-insensitive `MEDIA:` gate, so a 4-space or tab-indented code line can be extracted as media.

## Solution Assessment

No, not as submitted. The indented-code and fence-validation direction is narrow, but the current branch needs to preserve lowercase media behavior and share the same code-aware rules across streaming and audio cleanup before merge.
