## Summary

Keep open. Current main still has the reported indented-code MEDIA extraction bug, but the PR head is not merge-ready because it changes lowercase MEDIA compatibility while related directive surfaces remain case-insensitive, and its post-audio cleanup can still strip tab indentation from retained code.

## What This Changes

The PR changes media and markdown fence parsing to skip indented-code MEDIA directives, preserve retained code indentation during cleanup, reject invalid fence closers with trailing text, make MEDIA token matching case-sensitive, and add parser regression tests.

## Best Possible Solution

Land a focused version that preserves indented-code MEDIA lines, keeps lowercase MEDIA semantics consistent across splitMediaFromOutput, block-reply parsing, and streaming buffering, reuses code-aware normalization after audio-tag removal, validates shared fence consumers, and adds the active-version changelog fix entry.

## Reproduction Assessment

Yes. A source-level reproduction is passing output with a blank line followed by a four-space or tab-indented `MEDIA:` line to splitMediaFromOutput on current main; the current code trims indentation and extracts it unless it is inside a fenced span.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow direction, but the case-semantics change and post-audio cleanup path need repair before this is the best maintainable fix.
