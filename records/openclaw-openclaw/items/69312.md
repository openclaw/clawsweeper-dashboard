## Summary

Keep open: current main still has the indented-code false-positive path, so the PR is useful, but latest head still removes the established lowercase `media:` contract, leaves streaming buffering misaligned with the new code-aware parser, and lacks real behavior proof.

## What This Changes

The PR changes media parsing to skip `MEDIA:` extraction from indented code blocks, tightens markdown fence closing validation, changes directive case handling, and adds parser/fence regression tests.

## Best Possible Solution

Land a focused parser fix that prevents indented-code false positives while preserving lowercase directive support and aligning batch, streaming, and audio-tag cleanup semantics before merge.

## Reproduction Assessment

Yes, source-level. Current main only excludes fenced spans and then calls `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so a 4-space or tab-indented `MEDIA:` line outside a fence can still be extracted; I did not execute tests because the review is read-only.

## Solution Assessment

No, not as submitted. The indented-code and fence-validation direction is narrow enough, but the branch should preserve the lowercase media contract and reuse the same code-aware rules in streaming and audio-tag cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve lowercase media directive support:** `src/media/parse.ts:17`
  - body: Current main intentionally parses lowercase `media:` in block replies. Removing the regex `i` flag and adding tests that reject lowercase directives turns `media: ./report.pdf` into visible text instead of an attachment, so keep the case-insensitive contract unless it is deliberately changed across parser, streaming, docs, and tests.
  - confidence: 0.95
- **[P2] Align streaming buffering with indented-code parsing:** `src/media/parse.ts:563`
  - body: The batch parser now retains 4-space/tab indented `MEDIA:` lines as text, but the streaming accumulator still buffers whitespace-prefixed media-looking tails. A streamed indented code example at a chunk boundary can sit in `pendingTail` and be dropped on reset or a phase boundary, so the streaming path needs the same code-aware rule or a conservative flush path.
  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:747`
  - body: The main cleanup path preserves leading indentation when the first surviving line is code, but the `parseAudioTag` branch applies non-code leading normalization unconditionally. With `[[audio_as_voice]]` before a tab-indented code line, that tab can still be stripped even though the line was retained as code.
  - confidence: 0.87
