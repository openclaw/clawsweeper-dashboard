## Summary

Keep this PR open, but do not merge it as submitted: the indented-code guard is the right fix direction, while the current branch regresses the existing lowercase `media:` contract and leaves streaming plus audio-tag cleanup inconsistent with the new parser rules.

## What This Changes

The PR updates media parsing to skip `MEDIA:` inside indented code blocks, tightens markdown fence closing validation, changes media directive case handling, and adds parser/fence regression tests.

## Best Possible Solution

Land a focused parser fix that skips indented-code `MEDIA:` false positives while preserving lowercase media support and keeping streaming plus audio-tag normalization aligned.

## Reproduction Assessment

Yes, source-level: current main only skips fenced spans before extracting any trimmed `MEDIA:` line, so a 4-space or tab-indented code example can be stripped and turned into media. I did not run tests because this review was read-only.

## Solution Assessment

No, not as submitted. The indented-code and fence-validation pieces are promising, but the branch should preserve the existing lowercase `media:` contract and update streaming/audio cleanup to match the parser rules.

Full review comments:

- **[P2] Preserve lowercase media directive support:** `src/media/parse.ts:17`
  - body: Current main explicitly supports lowercase `media:` in block replies, including coverage that maps `media: ./report.pdf` to `mediaUrl` and `mediaUrls`. Removing the `i` flag and changing the gate to `startsWith("MEDIA:")` turns that supported directive into visible text, so keep the parser case semantics aligned unless this contract is deliberately retired across all callers and tests.
  - confidence: 0.94
- **[P2] Align streaming buffering with indented-code parsing:** `src/media/parse.ts:563-569`
  - body: The parser now keeps indented-code `MEDIA:` lines as text, but `splitTrailingDirective` still buffers any whitespace-prefixed `MEDIA:` tail. A streamed tab/4-space code example at a chunk boundary can sit in `pendingTail` and be delayed or dropped on reset instead of flushing as ordinary text, so the streaming accumulator needs the same indented-code rule or a conservative flush path.
  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:745-748`
  - body: The main cleanup path preserves leading indentation when the first surviving line is code, but the `parseAudioTag` branch applies an unconditional leading-normalization regex. With `[[audio_as_voice]]` before a tab-indented code line, that tab can be stripped even though the line was retained as code, so reuse the `firstLineIsCode` decision after audio-tag removal too.
  - confidence: 0.87
