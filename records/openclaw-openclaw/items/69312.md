Full review comments:

- **[P2] Align streaming buffering with indented code:** `src/media/parse.ts:563`
  - body: The new final parser preserves blank-preceded 4-space/tab `MEDIA:` lines as code, but streaming still buffers any last line matching optional whitespace plus `MEDIA:`. If a streamed reply ends on an indented code example and the accumulator is reset before a final flush, that text can still be withheld and dropped, so update `splitTrailingDirective` and its regressions to use the same indented-code decision.
  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:745-748`
  - body: After `parseAudioTag`, this path applies a generic leading-normalization regex instead of the `firstLineIsCode` check above. A reply like `[[audio_as_voice]]` followed by a 4-space or tab-indented code line can lose part or all of its indentation after the tag is stripped, which reintroduces the formatting corruption this PR is trying to prevent.
  - confidence: 0.86
- **[P2] Fix the mixed-case MEDIA regression expectation:** `src/media/parse.test.ts:421-425`
  - body: This new test expects `media: note MEDIA:/tmp/a.png` to yield media, but the restored case-insensitive regex greedily matches from the lowercase prefix through the end of the line, leaving no valid media candidate and stripping the line. Either make the parser intentionally extract the later token or assert the preserved-text behavior so the test matches the implementation.
  - confidence: 0.78
