## Summary

Keep open: this is a valid media-parser bug, but the current PR branch still regresses lowercase MEDIA support, leaves streaming/audio cleanup paths inconsistent, and lacks required real behavior proof.

## What This Changes

The PR changes media and markdown fence parsing to skip MEDIA extraction from indented code and malformed fence contexts, adjust cleanup/case handling, and add parser regression tests.

## Best Possible Solution

Land a focused parser fix that keeps lowercase MEDIA compatibility, applies code-aware rules consistently in batch parsing, streaming buffering, and audio-tag cleanup, validates shared fence consumers, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main excludes fenced spans only and then trim-starts lines before the case-insensitive MEDIA gate, so a 4-space or tab-indented code line can enter extraction.

## Solution Assessment

No, not as submitted. The indented-code direction is narrow, but the branch must preserve existing lowercase media semantics and align streaming/audio cleanup before it is the maintainable fix.
