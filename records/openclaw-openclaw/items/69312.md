## Summary

Keep open. Current main still has the reported indented-code false positive, and the PR is useful, but the latest head is not merge-ready because it makes `MEDIA:` parsing case-sensitive while current block-reply and streaming directive surfaces still expect lowercase `media:` compatibility; it also still has a non-code-aware post-audio cleanup path and lacks the required user-facing changelog entry.

The PR changes media parsing to preserve indented-code `MEDIA:` lines, protect retained code indentation during cleanup, tighten shared markdown fence closing validation, make `MEDIA:` token matching case-sensitive, and add media/fence regression tests.

## Best Possible Solution

Land a focused version that preserves indented-code `MEDIA:` lines and retained code indentation, keeps lowercase directive semantics consistent across `splitMediaFromOutput`, block replies, and streaming buffering unless a coordinated product change updates all surfaces and tests, reuses code-aware cleanup after audio-tag stripping, validates shared fence consumers, and adds the required Unreleased changelog fix entry.

Yes. Current main is source-reproducible with output containing a blank line followed by a 4-space or tab-indented `MEDIA:` line, and the PR regression is source-backed by existing lowercase block-reply coverage plus PR-head case-sensitive parser changes.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow direction, but the case-sensitivity change and post-audio cleanup path are not the safest maintainable solution without coordinated updates across the other directive surfaces.
