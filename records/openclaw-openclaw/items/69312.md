## Summary

Keep open: the central bug is real, but the branch still regresses current lowercase media support, leaves streaming buffering on the old directive boundary, and can strip code indentation after audio tag cleanup; external contributor proof is also missing.

## What This Changes

This PR changes shared MEDIA parsing to skip CommonMark-style indented code blocks, tightens fenced-code closing detection, alters MEDIA case handling, and adds parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips true indented code blocks while preserving existing lowercase MEDIA semantics, code-aware cleanup, and the same directive boundary in streaming paths.

## Reproduction Assessment

Yes, source-level: current main only excludes fenced spans before applying the trimStart/toUpperCase MEDIA gate, so 4-space or tab-indented code lines can be parsed as media. I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. The indented-code direction is reasonable, but the branch should preserve lowercase directives, update streaming buffering to the same boundary, and keep audio-tag cleanup code-aware.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve lowercase MEDIA directives:** `src/media/parse.ts:17`
  - body: Current main has regression coverage that lowercase `media: ./report.pdf` is parsed into mediaUrl/mediaUrls before path normalization. Removing the regex `i` flag and making the line gate uppercase-only turns that supported directive into visible text, so this breaks existing block-reply media behavior.
  - confidence: 0.94
- **[P2] Align streaming buffering with code-line detection:** `src/media/parse.ts:563`
  - body: The parser now preserves indented `MEDIA:` lines as code, but the streaming accumulator still buffers whitespace-prefixed MEDIA tails under the old boundary. A streamed indented code example can therefore be held, dropped on reset, or parsed differently across chunks unless streaming-directives.ts and its tests are updated with the same code-line rule.
  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:747`
  - body: The main cleanup chooses a safe leading regex when the first surviving line is code, but the `audio_as_voice` post-processing always uses the non-code regex. With `[[audio_as_voice]]` before a tab- or four-space-indented code line, this can strip code indentation and re-expose a retained `MEDIA:` example as ordinary directive-shaped text.
  - confidence: 0.86
