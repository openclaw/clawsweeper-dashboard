## Summary

Keep open: the indented-code-block bug is real on current main, but this PR still changes the established lowercase media directive contract and only has test/CI proof rather than real behavior proof.

## What This Changes

The PR adds indented-code-block protection, stricter markdown fence close validation, whitespace normalization changes, uppercase-only MEDIA parsing, and parser tests.

## Best Possible Solution

Land a narrow parser fix that skips indented code blocks and tightens fence closes without changing the existing lowercase MEDIA directive contract, with regression coverage for both behaviors.

## Reproduction Assessment

Yes, source-level: current main only excludes fenced spans, then uses line.trimStart() before MEDIA extraction, so a 4-space or tab-indented MEDIA line outside a fence can be extracted.

## Solution Assessment

No, not as currently submitted. The indented-code guard is the right fix boundary, but lowercase media directive support should be preserved or any uppercase-only change needs a separate documented maintainer decision across parser, reply delivery, and streaming.
