## Summary

Keep open: current main still has the source-level MEDIA extraction bug, but this PR is not mergeable yet because the latest diff changes established lowercase MEDIA behavior, leaves streaming buffering inconsistent with parser rules, keeps audio cleanup non-code-aware, and lacks required real behavior proof.

## What This Changes

The PR updates MEDIA and fence parsing to skip MEDIA extraction from indented code blocks, preserve retained code whitespace, tighten closing-fence validation, make MEDIA matching case-sensitive, and add parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips MEDIA extraction from indented code blocks while preserving or deliberately migrating MEDIA case semantics across block and streaming paths, reuses code-aware cleanup after audio tags, and validates shared fence consumers.

## Reproduction Assessment

Yes, source-level: current main trims indentation before MEDIA detection and only excludes fenced spans, so a blank-preceded 4-space/tab MEDIA line can be extracted. I did not execute tests under the read-only review constraint.

## Solution Assessment

No. The indented-code guard is the right direction, but this PR is not the best current patch until case semantics, streaming buffering, and audio-tag cleanup are aligned with existing contracts.
