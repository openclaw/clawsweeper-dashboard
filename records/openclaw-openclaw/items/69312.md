## Summary

Keep open. The PR targets a real current-main parser bug, but the current branch still regresses existing lowercase `media:` support, leaves streaming directive buffering out of sync with the new parser semantics, applies less code-aware cleanup after audio-tag stripping, and lacks required real behavior proof.

## What This Changes

The PR changes media/fence parsing to skip indented-code `MEDIA:` lines, preserve retained code indentation, make media token matching case-sensitive, validate fence closers, and add parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips indented-code `MEDIA:` false positives while preserving lowercase media support, aligning streaming buffering, and using code-aware cleanup after audio-tag stripping.

## Reproduction Assessment

Yes, source-level. Current main only skips fenced spans before extracting any trimmed `MEDIA:` line, so a 4-space or tab-indented code example can be stripped and turned into media; I did not run tests because this review was read-only.

## Solution Assessment

No, not as submitted. The indented-code and fence-validation pieces are promising, but the branch should preserve the existing lowercase `media:` contract and keep streaming plus audio-tag cleanup aligned with the parser rules.

  - confidence: 0.88

After an audio tag is removed before a tab- or space-indented code line, that indentation can be stripped from visible output.
  - confidence: 0.87
