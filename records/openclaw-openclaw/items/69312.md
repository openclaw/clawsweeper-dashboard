## Summary

Keep open: the parser fix is directionally useful, but the latest patch still leaves streaming buffering out of sync with the final parser for multi-line indented code, and the proof is still unit-test/mock-only.

## What This Changes

The branch updates MEDIA parsing, fence span parsing, streaming directive buffering, and regression tests so MEDIA-looking lines inside indented code or malformed fence contexts are preserved as text instead of attachments.

## Best Possible Solution

Align streaming buffering with the final parser's full indented-code state, add a reset/drop regression for multi-line indented code, and land only after redacted live delivery proof plus maintainer acceptance of the compatibility shift.

## Reproduction Assessment

Yes, at source level. Current main only protects fenced spans before trimStart MEDIA matching, and the PR's streaming guard still visibly differs from the final parser for multi-line indented code; I did not run tests in this read-only review.

## Solution Assessment

No, not yet. The final parser direction is maintainable, but the streaming guard must track previous indented-code lines too, preferably through a shared helper or equivalent state, before merge.
