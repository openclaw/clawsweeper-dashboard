## Summary

Keep this PR open. Current main still has the reported indented-code `MEDIA:` false positive, and the PR contains useful targeted work, but the latest head still regresses existing lowercase media-directive behavior and leaves the audio-tag cleanup path able to strip tab-indented code.

## What This Changes

The PR updates media and fence parsing plus tests to preserve indented-code `MEDIA:` lines, make cleaned output more code-aware, tighten malformed fence closing, and make `MEDIA:` token matching uppercase-only.

## Best Possible Solution

Land a focused version that preserves indented-code `MEDIA:` lines without changing existing lowercase directive compatibility, makes the post-audio cleanup reuse the same first-surviving-code-line protection, keeps or separately validates the shared fence-parser change, and adds the required Unreleased changelog entry.

## Reproduction Assessment

Yes. The main-branch bug is source-reproducible with output containing a blank line followed by a 4-space or tab-indented `MEDIA:` line, and the PR regressions are source-backed by current lowercase block-reply coverage plus the PR head's post-audio normalization regex.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow direction, but the uppercase-only parser change and non-code-aware audio cleanup are not the safest maintainable fix for the reported bug.
