## Summary

Keep open: the reported bug is source-reproducible and the indented-code direction is useful, but the current PR still breaks established lowercase MEDIA behavior, leaves streaming/audio cleanup paths misaligned, and lacks contributor real behavior proof.

## What This Changes

The PR updates MEDIA and markdown fence parsing to ignore MEDIA tokens in CommonMark-style indented code blocks, tighten fence-close validation, alter MEDIA case handling, and add parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips true indented code blocks while preserving lowercase MEDIA support and aligning streaming/audio cleanup with the same directive boundary and regression coverage.

## Reproduction Assessment

Yes, source-level. Current main only excludes fenced spans before the trimStart-based MEDIA gate, so a 4-space or tab-indented code line can still enter media extraction; I did not run repo tests because this review is read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should preserve lowercase MEDIA directives and update the streaming and audio cleanup boundaries before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Preserve lowercase MEDIA directive support:** `src/media/parse.ts:17`
  - body: Current main has regression coverage that lowercase `media: ./report.pdf` in block replies is parsed before path normalization. Removing the regex `i` flag and making the line gate case-sensitive turns that supported directive into visible text, so this breaks existing MEDIA behavior.
  - confidence: 0.94
- **[P2] Align streaming buffering with indented-code detection:** `src/media/parse.ts:563-569`
  - body: The parser now preserves 4-space/tab indented `MEDIA:` lines as code, but the streaming accumulator still buffers whitespace-prefixed MEDIA tails under the old boundary. A streamed indented code example can therefore be held and dropped on reset, or handled differently across chunks, unless streaming-directives.ts and its tests get the same code-line rule.
  - confidence: 0.88
- **[P2] Reuse code-aware leading cleanup after audio tags:** `src/media/parse.ts:747`
  - body: The main cleanup chooses a safe leading regex when the first surviving line is code, but the `audio_as_voice` post-processing always uses the non-code regex. With `[[audio_as_voice]]` before a tab-indented code line, that strips the tab and exposes the retained code example as ordinary directive-shaped text.
  - confidence: 0.86
