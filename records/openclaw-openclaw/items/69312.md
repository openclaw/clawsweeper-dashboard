## Summary

Keep open: the reported parser bug is valid and this PR is the canonical fix path, but the latest head still regresses the lowercase media directive contract, leaves streaming and audio-tag cleanup inconsistencies, and lacks real behavior proof.

## What This Changes

The PR adds an indented-code MEDIA guard, adjusts media cleanup and fence parsing, and adds media/fence parser tests.

## Best Possible Solution

Land a focused parser fix that prevents indented-code false positives while preserving lowercase media directives and applying code-aware parsing/cleanup consistently across batch, streaming, and audio-tag paths.

## Reproduction Assessment

Yes, source-level. Current main only excludes fenced spans and then trims leading indentation before checking `MEDIA:`, so a 4-space or tab-indented code line can still be parsed as media; I did not run tests in read-only mode.

## Solution Assessment

No, not as submitted. The indented-code and fence-validation direction is narrow, but the branch should preserve lowercase media support and align streaming and audio-tag cleanup with the new code-aware parser rule.

## Review Findings

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve lowercase media directive support:** `src/media/parse.ts:17`
  - body: Current main intentionally parses lowercase `media:` in block replies. Removing the `i` flag and making the line gate case-sensitive turns `media: ./report.pdf` back into visible text instead of an attachment, so keep the case-insensitive contract unless the parser, streaming path, docs, and tests are changed together.
  - confidence: 0.95
- **[P2] Align streaming buffering with code-aware MEDIA parsing:** `src/media/parse.ts:562-569`
  - body: This batch parser now treats 4-space/tab indented `MEDIA:` lines as text, but the streaming accumulator still buffers whitespace-prefixed `MEDIA:` tails. A streamed indented code example at a chunk boundary can sit in `pendingTail` and be dropped on reset or a phase boundary, so the streaming path needs the same code-aware rule or a conservative flush path.
  - confidence: 0.88
- **[P2] Reuse code-aware leading trim after audio tags:** `src/media/parse.ts:747`
  - body: The main cleanup path preserves leading indentation when the first surviving line is code, but the `parseAudioTag` branch applies non-code leading normalization unconditionally. With `[[audio_as_voice]]` before a tab-indented code line, that tab can still be stripped and the code example can be reinterpreted as a top-level directive downstream.
  - confidence: 0.87
