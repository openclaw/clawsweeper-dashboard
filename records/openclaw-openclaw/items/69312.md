## Summary

Keep open: the indented-code MEDIA false positive is a real source-level bug, but the current PR head still regresses established lowercase MEDIA parsing, leaves streaming/audio cleanup paths inconsistent, and lacks contributor real behavior proof.

## What This Changes

The PR adds an indented-code guard and fence-closing validation to media parsing, changes MEDIA case handling, and adds media/fence parser regression tests.

## Best Possible Solution

Land a narrower parser fix that skips indented-code MEDIA false positives while preserving lowercase directive behavior and keeping streaming plus audio-tag normalization aligned.

## Reproduction Assessment

Yes, source-level: current main skips only fenced spans and then extracts any trimmed line whose uppercase prefix is `MEDIA:`, so blank-line-preceded 4-space or tab-indented code can be extracted. I did not run tests because this review was read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should preserve the existing lowercase MEDIA contract and update streaming/audio cleanup paths to match the new parser rules.
