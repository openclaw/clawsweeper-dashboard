## Summary

Keep open: the original MEDIA false-positive bug is valid, but the latest PR head still regresses established lowercase media directive support, leaves streaming directive buffering out of sync with the new indented-code semantics, lacks required real behavior proof, and currently has a CI failure on the lowercase contract.

## What This Changes

The PR changes MEDIA parsing and fence parsing so indented-code MEDIA lines are preserved as text, closing fences require trailing whitespace only, lowercase media directives are rejected, and parser/fence tests are added.

## Best Possible Solution

Revise the PR to skip MEDIA extraction from indented code blocks while preserving lowercase directive compatibility, aligning streaming buffering with the same parser rules, keeping audio-tag cleanup code-aware, and adding redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: splitMediaFromOutput only excludes fenced spans and then removes indentation before the MEDIA prefix check, so blank-preceded 4-space/tab code lines can be extracted; I did not run local tests because pnpm is unavailable in this checkout.

## Solution Assessment

No. The indented-code guard belongs in splitMediaFromOutput, but this PR is not yet the best solution because it rejects a tested lowercase directive contract, does not update streaming buffering to the new directive shape, and still needs real behavior proof.
