## Summary

Keep open: the PR addresses a valid source-reproducible media parsing bug, but the latest head still introduces lowercase `media:` delivery regressions and an audio-tag indentation regression, with the required changelog entry still missing.

## What This Changes

The PR changes media parsing and markdown fence parsing to avoid extracting `MEDIA:` directives from indented code blocks, tightens closing-fence detection, and adds media/fence regression tests.

## Best Possible Solution

Land a repaired parser/fence patch that preserves lowercase media directive compatibility, keeps code indentation after audio-tag stripping, and records the user-facing fix in the active changelog.

## Reproduction Assessment

Yes, source-reproducible. Current main skips fenced code only, then uses `trimStart()` before a case-insensitive `MEDIA:` gate, which can treat a 4-space or tab indented code-block line as a real media directive.

## Solution Assessment

No, not as submitted. The indented-code and fence-close approach is narrow, but the patch should preserve existing lowercase media behavior and reuse code-aware cleanup after audio tag stripping.
