## Summary

Keep open: the indented-code bug is real, but this PR still regresses the existing lowercase `media:` directive contract and lacks after-fix real behavior proof from an OpenClaw setup.

## What This Changes

The PR changes `splitMediaFromOutput` and `parseFenceSpans` to skip `MEDIA:` extraction from indented code blocks, preserve retained code indentation, tighten fence closing validation, and add media/fence parser tests.

## Best Possible Solution

Land a focused patch that preserves the existing case-insensitive media directive contract, keeps streaming buffering aligned, protects indented code blocks, and validates shared fence-parser consumers.

## Reproduction Assessment

Yes, source-level. Current main only skips fenced spans before media extraction and then tests `line.trimStart()` for `MEDIA:`, so four-space or tab-indented code-block lines can still enter extraction.

## Solution Assessment

No, not as currently proposed. The indented-code protection is a reasonable narrow fix, but the case-sensitivity cleanup should be removed or coordinated across block-reply parsing, streaming buffering, and tests.
