## Summary

Keep open. Current main has the reported source-level MEDIA false-positive path, but this PR head still introduces parser regressions around lowercase directives, streaming tail buffering, and audio-tag cleanup, and the external PR only has test claims instead of real behavior proof.

## What This Changes

The PR changes shared MEDIA parsing to skip MEDIA-looking lines in indented code blocks, preserve selected code indentation during cleanup, tighten fence close parsing, and add parser/fence regression tests.

## Best Possible Solution

Land a focused parser fix that preserves the existing lowercase MEDIA and streaming contracts, reuses code-aware cleanup after audio tag stripping, keeps the fence validation tests, and includes redacted CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main checks only fenced spans before `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so blank-preceded 4-space or tab-indented code lines can be parsed as attachments.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should preserve lowercase directives, align streaming behavior, and reuse the code-aware cleanup after audio tag stripping before merge.

Removing `[[audio_as_voice]]` before an indented code line can still damage indentation and expose a code example as a directive-shaped line.
  - confidence: 0.86
