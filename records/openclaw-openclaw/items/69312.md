## What This Changes

The PR updates media and fence parsing so indented-code `MEDIA:` lines are preserved, fence closes require whitespace-only suffixes, MEDIA token case handling changes, and regression tests cover those paths.

## Best Possible Solution

Land a focused parser fix that skips indented-code `MEDIA:` lines without changing lowercase block/streaming MEDIA semantics, shares code-aware cleanup through audio-tag stripping, records the changelog entry, and includes redacted real setup proof.

## Reproduction Assessment

Yes, by source inspection. Current main only protects fenced spans, then checks `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so blank-preceded 4-space or tab-indented MEDIA lines can enter extraction.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should preserve existing lowercase/streaming MEDIA semantics, reuse code-aware cleanup after audio-tag stripping, add a changelog entry, and provide real behavior proof before merge.
