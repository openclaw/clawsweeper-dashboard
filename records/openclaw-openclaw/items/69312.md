## Summary

Keep this PR open: it targets a real current-main media parsing bug, but the latest branch still changes `MEDIA:` case semantics without updating the block-reply and streaming directive surfaces, and it still lacks required real behavior proof before merge.

## What This Changes

The PR changes media parsing to skip `MEDIA:` directives in indented code blocks, tighten markdown fence closing validation, adjust media-token case handling, and add parser/fence regression tests.

## Best Possible Solution

Land a focused version that preserves indented-code `MEDIA:` text and code indentation while keeping lowercase media directive semantics aligned across `splitMediaFromOutput`, block-reply parsing, and streaming buffering, then require real setup proof before merge.

## Reproduction Assessment

Yes. Source inspection of current main shows `splitMediaFromOutput` only protects fenced spans before trimming indentation and matching `MEDIA:`, so the linked indented-code false-positive path is clear without executing tests.

## Solution Assessment

No, not as currently proposed. The indented-code guard is a reasonable narrow fix, but the case-sensitivity cleanup must either be dropped or carried through block-reply parsing, streaming buffering, and tests.
