## Summary

This PR addresses a real parser bug linked from https://github.com/openclaw/openclaw/issues/69311, but it should stay open because the current branch still breaks established lowercase MEDIA handling, leaves streaming buffering out of sync, and lacks contributor-supplied real behavior proof.

## What This Changes

The branch adds an indented-code guard to `splitMediaFromOutput`, tightens fenced-code closing detection, and adds media/fence parser regression tests.

## Best Possible Solution

Land a focused parser fix that skips indented-code MEDIA false positives while preserving lowercase directive semantics and keeping streaming, audio cleanup, and regression tests aligned.

## Reproduction Assessment

Yes, source-level: current main only excludes fenced spans before checking each trimmed line for `MEDIA:`, so a blank-line-preceded 4-space or tab-indented code line can be extracted. I did not run tests because this review was read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should preserve the existing lowercase MEDIA contract and update the streaming/audio cleanup paths to match the new parser rules.
