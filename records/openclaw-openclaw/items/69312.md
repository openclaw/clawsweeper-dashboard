## Summary

Keep open: the reported bug is source-reproducible and the indented-code direction is useful, but the current branch still breaks covered lowercase MEDIA behavior, leaves streaming/audio cleanup inconsistent, and lacks real after-fix proof.

## What This Changes

The PR updates MEDIA parsing, markdown fence parsing, and parser tests so MEDIA-looking lines in indented code blocks are preserved as text instead of extracted as attachments.

## Best Possible Solution

Keep the indented-code protection, preserve lowercase MEDIA support, align streaming buffering and audio-tag cleanup with the same code-aware parsing contract, then add targeted tests plus redacted real CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main only excludes fenced spans, then checks `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so a 4-space or tab-indented code line can be parsed as media.

## Solution Assessment

No, not as submitted. The indented-code fix is narrow, but the branch should not remove covered lowercase MEDIA support or leave streaming and audio-tag cleanup on older parsing assumptions.
