## Summary

Keep open. The PR targets a real source-reproducible media-parser bug, but the latest head still changes existing lowercase MEDIA semantics, leaves streaming and audio-tag cleanup inconsistent with the new parser boundary, and lacks contributor-supplied real behavior proof.

## What This Changes

This PR changes MEDIA parsing to skip 4-space/tab indented code blocks, protect retained code indentation, tighten fence closing validation, remove case-insensitive MEDIA matching, and add parser/fence tests.

## Best Possible Solution

Land a focused parser fix after preserving existing lowercase directive semantics, aligning streaming and audio-tag cleanup with the same code-line rules, and validating the shared fence parser consumers if that change remains.

## Reproduction Assessment

Yes, source-level: current main only checks fenced spans before the trimStart/toUpperCase MEDIA gate, so a 4-space or tab-indented MEDIA line outside fences can be parsed as media. I did not run tests because this review is read-only.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should preserve lowercase MEDIA semantics, align streaming buffering, and reuse code-aware cleanup after audio tags.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep MEDIA case semantics aligned:** `src/media/parse.ts:17`
  - body: Removing the `i` flag makes `splitMediaFromOutput` ignore lowercase `media:`, but current block-reply coverage expects lowercase directives to become `mediaUrl`/`mediaUrls`, and streaming still buffers lowercase MEDIA-like tails. Keep the existing case-insensitive contract or update every caller and test that depends on it.
  - confidence: 0.9
- **[P2] Align streaming buffering with indented-code rules:** `src/media/parse.ts:563`
  - body: The parser now preserves indented `MEDIA:` lines as code, but the streaming accumulator still buffers whitespace-prefixed MEDIA tails under the old rule. A streamed indented code example ending at a chunk boundary can be held, dropped on reset, or parsed differently unless `streaming-directives.ts` uses the same boundary.
  - confidence: 0.86
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:747`
  - body: After `parseAudioTag`, cleanup always applies the non-code leading regex instead of the `firstLineIsCode`-aware regex. With `[[audio_as_voice]]` before a tab- or four-space-indented code line, this can strip structural indentation from retained code text.
  - confidence: 0.86
