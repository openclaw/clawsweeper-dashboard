## Summary

Keep this PR open: current main still has the reported source-level media-parser bug, but the branch is not mergeable as submitted because it changes established lowercase `media:` behavior, leaves streaming buffering out of sync with the new indented-code semantics, and lacks after-fix real behavior proof.

## What This Changes

The PR adds indented-code-block protection and stricter fence closing validation to media parsing, adjusts cleanup/case handling, and adds media/fence parser tests.

## Best Possible Solution

Land a focused parser fix that preserves indented-code `MEDIA:` lines as text while keeping existing lowercase directive semantics and aligning streaming buffering; validate shared fence consumers if the fence change stays.

## Reproduction Assessment

Yes, source-level: current main skips only fenced spans and then applies a `trimStart()` case-insensitive `MEDIA:` gate, so indented code lines can be extracted as media. I did not run tests under the read-only review policy.

## Solution Assessment

No, not as submitted. The indented-code guard is a reasonable narrow fix, but the branch changes an established lowercase directive contract and does not align streaming tail buffering with the new parser semantics.

A stream chunk ending on that preserved code line can move it into `pendingTail` and lose it on reset, so streaming needs the same indented-code exclusion and coverage.
  - confidence: 0.84
