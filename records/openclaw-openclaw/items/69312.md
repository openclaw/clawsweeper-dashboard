## What This Changes

The PR changes MEDIA and fence parsing so indented code-block MEDIA lines are preserved, invalid fence closers stay open, and parser regressions are covered by tests.

## Best Possible Solution

Land a focused parser fix that preserves MEDIA code examples in batch and streaming paths, keeps audio-tag cleanup code-line-aware, and includes redacted real-run proof.

## Reproduction Assessment

Yes, source-reproducible: current main skips only fenced spans and then uses trimStart plus a case-insensitive MEDIA prefix check, so a blank-line-prefixed 4-space or tab-indented MEDIA line reaches extraction. I did not run tests in this read-only sweep.

## Solution Assessment

No, not yet. The main indented-code guard is a reasonable narrow fix, but the branch should align streaming MEDIA casing and make post-audio cleanup reuse the code-line-aware leading normalization before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Align streaming directive casing:** `src/media/parse.ts:17`
  - body: This makes splitMediaFromOutput treat lowercase `media:` as text, but the streaming accumulator still buffers MEDIA lines and partial prefixes case-insensitively. A lowercase streaming chunk can be held in `pendingTail` and dropped on reset instead of emitted as text, so streaming and non-streaming paths disagree.
  - confidence: 0.92
- **[P2] Preserve code indentation after audio tags:** `src/media/parse.ts:747`
  - body: After `parseAudioTag` strips `[[audio_as_voice]]`, this normalization no longer checks whether the first surviving line was tracked as code. Inputs like `[[audio_as_voice]]\n\n\tMEDIA:/tmp/code.png` can lose the tab and make visible code look like a top-level directive.
  - confidence: 0.9
