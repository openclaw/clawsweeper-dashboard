## Summary

Keep open: the central indented-code MEDIA fix is plausible and addresses a source-reproducible current-main bug, but the latest PR head still has two blocking parser regressions and lacks required real behavior proof from an external contributor.

## What This Changes

The PR updates MEDIA extraction and markdown fence parsing to avoid extracting MEDIA directives from indented code blocks, changes MEDIA case handling, and adds parser regression tests.

## Best Possible Solution

Keep the focused indented-code and fence fixes, align streaming case handling, make audio-tag cleanup code-aware, reconcile overlap with https://github.com/openclaw/openclaw/pull/41222, then merge after redacted real-run proof.

## Reproduction Assessment

Yes. Source inspection on current main shows splitMediaFromOutput only excludes fenced spans and then trimStart/toUpperCase checks MEDIA:, so a blank-line-prefixed four-space or tab-indented MEDIA line can be parsed as a directive.

## Solution Assessment

No, not yet. The core approach is narrow, but the branch should also update streaming directive case handling and preserve tab-indented code after audio tag stripping before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep streaming MEDIA case handling aligned:** `src/media/parse.ts:17`
  - body: This PR makes MEDIA_TOKEN_RE and the line guard case-sensitive, but splitTrailingDirective still buffers `media:` and partial prefixes with `/i` in `src/auto-reply/reply/streaming-directives.ts`. A lowercase streaming chunk can be held as a directive and then dropped on reset instead of emitted as text, so update the streaming guard and add coverage.
  - confidence: 0.9
- **[P2] Preserve tab code indentation after audio tags:** `src/media/parse.ts:747`
  - body: After `parseAudioTag` removes `[[audio_as_voice]]`, this generic leading-normalization regex no longer knows that the first surviving line was marked as code. `[[audio_as_voice]]\n\n\tMEDIA:/tmp/code.png` can lose the tab and become a top-level-looking directive, so reuse the code-aware leading normalization for the post-audio path.
  - confidence: 0.88
