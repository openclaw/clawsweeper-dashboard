## What This Changes

The PR updates MEDIA and fence parsing plus parser tests so indented code-block MEDIA lines are preserved, invalid fence closers stay open, and MEDIA matching becomes case-sensitive.

## Best Possible Solution

Merge the focused parser fix after aligning streaming casing, making post-audio cleanup code-line-aware, and adding redacted real-run proof.

## Reproduction Assessment

Yes, from source inspection rather than executed tests. Current main only skips fenced spans and then uses trimStart plus a case-insensitive MEDIA prefix check, so a blank-line-prefixed 4-space or tab-indented MEDIA line reaches extraction.

## Solution Assessment

No, not yet. The main indented-code approach is narrow, but the branch should keep streaming case semantics aligned and preserve code indentation after audio-tag stripping before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep streaming MEDIA case handling aligned:** `src/media/parse.ts:17`
  - body: This makes `splitMediaFromOutput` treat lowercase `media:` as text, but the streaming accumulator still buffers directive lines and partial prefixes case-insensitively. A lowercase streaming chunk can sit in `pendingTail` and be dropped on reset instead of being emitted as text, so streaming and non-streaming paths disagree.
  - confidence: 0.9
- **[P2] Preserve tab code indentation after audio tags:** `src/media/parse.ts:747`
  - body: After `parseAudioTag` strips `[[audio_as_voice]]`, this normalization pass no longer knows whether the first surviving line was tracked as code. For `[[audio_as_voice]]\n\n\tMEDIA:/tmp/code.png`, the tab can be stripped, changing visible code formatting and making the line look like a top-level directive.
  - confidence: 0.87
