## Summary

Keep open. Current main still has the reported indented-code MEDIA extraction bug, and this PR is a useful targeted fix, but the latest head is not merge-ready because it changes lowercase MEDIA directive semantics without updating the other directive surfaces and still has code-aware cleanup/streaming gaps.

## What This Changes

The PR updates media parsing to skip MEDIA extraction from indented code blocks, preserve retained code indentation during cleanup, tighten markdown fence close validation, make MEDIA matching case-sensitive, and add media/fence regression tests.

## Best Possible Solution

Land a focused version that keeps the indented-code MEDIA protection while preserving existing lowercase MEDIA directive behavior, aligns streaming tail buffering with the same directive rules, makes post-audio cleanup respect retained code indentation, validates shared fence-parser consumers, and adds the required Unreleased changelog fix entry.

## Reproduction Assessment

Yes. Current main is source-reproducible with an output containing a blank line followed by a 4-space or tab-indented MEDIA line; the parser only checks fenced spans before trimStart-based MEDIA extraction. The PR regressions are also source-backed by existing lowercase block-reply coverage and the unchanged streaming accumulator.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow direction, but the case-sensitivity change, streaming-buffer mismatch, and post-audio cleanup path are not yet the safest maintainable implementation.
