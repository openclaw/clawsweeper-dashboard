## Summary

Keep open: the bug is real, but the current PR still changes the established lowercase MEDIA contract, leaves streaming/audio cleanup mismatches, and lacks contributor-provided real behavior proof.

## What This Changes

The PR changes MEDIA parsing to skip MEDIA-looking lines inside indented code blocks, tightens markdown fence closing detection, and adds parser/fence regression tests.

## Best Possible Solution

Land a focused parser fix that skips true indented-code MEDIA lines without changing lowercase directive behavior, keeps streaming buffering aligned, reuses code-aware cleanup after audio tag stripping, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main only excludes fenced spans before the case-insensitive MEDIA line gate, so blank-preceded 4-space or tab-indented MEDIA-looking code lines can be parsed as attachments.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the patch should preserve lowercase directives, align streaming buffering with the new parser behavior, and apply the same code-aware cleanup after audio tag stripping.

## Review Findings

  - body: Current main has block-reply coverage expecting `media: ./report.pdf` to parse into `mediaUrl` and `mediaUrls`. Removing the regex `i` flag and requiring an uppercase line gate makes that existing lowercase directive visible text instead of attachment metadata.
  - confidence: 0.94

  - confidence: 0.88
- **[P2] Reuse code-aware cleanup after audio tags:** `src/media/parse.ts:745-748`
  - body: The main cleanup chooses a safe leading-normalization regex when the first retained content line is code, but the post-audio cleanup always applies the non-code regex. Removing `[[audio_as_voice]]` before a tab-indented code line can still damage indentation and expose a code example as a directive-shaped line.
  - confidence: 0.86
