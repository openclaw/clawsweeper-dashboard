## Summary

Keep open. The reported bug is real and the PR is active, but the current patch still regresses existing lowercase MEDIA directive behavior, leaves streaming/audio cleanup semantics inconsistent with the parser change, and is missing external real behavior proof.

## What This Changes

The PR changes media/fence parsing to skip MEDIA extraction from indented code blocks, preserve code indentation during cleanup, validate closing fence lines more strictly, and add media/fence regression tests.

## Best Possible Solution

Land a focused parser fix that preserves indented-code text, retains the current case-insensitive MEDIA contract unless maintainers intentionally change it, aligns streaming/audio cleanup, and includes redacted real CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main skips only fenced spans and then uses `trimStart().toUpperCase().startsWith("MEDIA:")`, so a blank-preceded 4-space or tab-indented `MEDIA:` code line can be extracted; I did not run tests in this read-only review.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the patch should preserve lowercase MEDIA support and align streaming plus audio cleanup before merge.
