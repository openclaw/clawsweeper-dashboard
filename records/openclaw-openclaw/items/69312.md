## What This Changes

The PR changes media and fence parsing to skip MEDIA extraction from indented code blocks, preserve retained code indentation, tighten fence closing validation, and add parser regressions.

## Best Possible Solution

Land a focused parser fix after preserving current lowercase MEDIA behavior across parser, block-reply, and streaming paths, reusing code-aware cleanup after audio tags, adding the active changelog entry, and validating shared fence-parser consumers.

Yes. Source inspection of current main shows `splitMediaFromOutput` only protects fenced spans before trimming and matching MEDIA lines, so blank-preceded 4-space or tab-indented code examples can still be extracted as media.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the patch should preserve existing lowercase MEDIA semantics, align streaming behavior, reuse code-aware audio cleanup, add changelog coverage, and include real setup proof.
