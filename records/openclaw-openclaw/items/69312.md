## Summary

Keep open: the reported bug is valid, but the patch still regresses current lowercase `media:` directive behavior, leaves streaming tail buffering inconsistent with the new indented-code semantics, and lacks required real behavior proof for an external PR.

## What This Changes

The PR adds an indented-code-block guard and fence closing validation to media parsing, updates whitespace cleanup, removes case-insensitive MEDIA parsing, and adds parser/fence tests.

## Best Possible Solution

Land the indented-code-block and fence parsing fixes after preserving existing lowercase media directives, aligning streaming tail buffering with parser semantics, and adding real behavior proof.

## Reproduction Assessment

Yes, source-level: current main skips only fenced spans and then applies a `trimStart()` case-insensitive `MEDIA:` gate, so indented code lines can be extracted as media. I did not run tests under the read-only review policy.

## Solution Assessment

No, not as submitted. The indented-code guard and fence validation are reasonable, but the branch should preserve current lowercase media directive support and update streaming buffering before merge.

- **[P2] Preserve lowercase media directive support:** `src/media/parse.ts:17`
  - body: Current main treats lowercase `media:` as a valid reply media directive, with regression coverage in `reply-delivery.test.ts` and case-insensitive streaming buffering.   - confidence: 0.94

In block streaming, a chunk ending after such an indented code line can move visible code into `pendingTail` and a reset can drop it, so the streaming guard needs the same indented-code exclusion and coverage.
  - confidence: 0.86
