## Summary

Keep open: current main still has the indented-code false-positive path, but this PR is not mergeable because it changes established lowercase MEDIA parsing, leaves code-aware normalization unused after audio tag stripping, lacks a changelog entry, and has no real behavior proof.

## What This Changes

The PR adds indented-code-block skipping and code-aware cleanup to `splitMediaFromOutput`, tightens fence closing validation, removes case-insensitive MEDIA parsing, and adds parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips indented-code MEDIA extraction while preserving lowercase MEDIA and streaming contracts, sharing code-aware normalization after audio tag stripping, documenting the user-facing fix in the changelog, and adding real setup proof.

## Reproduction Assessment

Yes, by source inspection. Current main only guards fenced spans before using `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so a blank-preceded four-space or tab-indented `MEDIA:` code line reaches extraction.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the branch should preserve current lowercase MEDIA semantics and reuse code-aware normalization after audio-tag stripping before merge.
