## Summary

Keep open: this PR targets a real source-reproducible parser bug, but current head still breaks established lowercase `MEDIA:` behavior, leaves streaming case handling inconsistent, can strip tab indentation after `[[audio_as_voice]]`, and lacks external real behavior proof.

## What This Changes

The PR changes media and markdown fence parsing to skip `MEDIA:` extraction from indented code blocks, preserve retained code indentation during cleanup, tighten fence closing validation, and add parser tests.

## Best Possible Solution

Land a focused parser fix that skips indented-code `MEDIA:` false positives, preserves code indentation through both media and audio cleanup, keeps or deliberately updates `MEDIA:` case semantics across parser/block/streaming paths, and includes targeted tests plus redacted real-run proof.

## Reproduction Assessment

Yes, from source inspection: current main only excludes fenced spans before a `trimStart().toUpperCase().startsWith("MEDIA:")` gate, so a blank-line-prefixed 4-space or tab-indented `MEDIA:` line can be extracted. I did not execute tests because this review is read-only.

## Solution Assessment

No, not yet. The indented-code direction is maintainable, but the patch should first align `MEDIA:` case semantics across parser, block replies, and streaming, and reuse code-aware normalization after audio-tag stripping.
