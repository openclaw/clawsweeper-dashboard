## Summary

Keep open. The PR addresses a real current-main MEDIA parsing bug, but the current head is not merge-ready because it changes lowercase MEDIA directive semantics while adjacent block-reply and streaming surfaces remain case-insensitive, leaves the post-audio cleanup path outside the new code-aware indentation handling, and lacks the required user-facing changelog entry.

## What This Changes

The PR changes media parsing and shared markdown fence parsing/tests to skip MEDIA extraction from indented code blocks, preserve retained code indentation, reject invalid fence closers with trailing text, and add media/fence regression coverage.

## Best Possible Solution

Land a focused version that preserves MEDIA lines in indented code blocks, keeps lowercase MEDIA directive behavior consistent across parser, block-reply normalization, and streaming buffering unless a coordinated product change updates all surfaces, reuses code-aware cleanup after audio tag stripping, validates shared fence consumers, and adds the active-version changelog fix entry.

## Reproduction Assessment

Yes. Current main is source-reproducible with output containing a blank line followed by a 4-space or tab-indented `MEDIA:` line: only fenced spans are skipped, then `trimStart().toUpperCase().startsWith("MEDIA:")` sends the line into media extraction.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow direction, but the case-sensitivity change, post-audio cleanup path, and missing changelog entry need repair before this is the best maintainable fix.
