## Summary

Keep open. The bug is real on current main, and the PR is moving in the right direction, but the latest head still regresses established lowercase `media:` handling and strips indentation again after `[[audio_as_voice]]` removal; it also lacks the required user-facing changelog entry.

## What This Changes

The PR changes media and markdown fence parsing to preserve indented-code `MEDIA:` lines, protect retained code indentation, reject invalid fence closers with trailing text, make media token matching case-sensitive, and add parser regression tests.

## Best Possible Solution

Land a focused parser fix that preserves indented-code `MEDIA:` examples while keeping lowercase directive compatibility consistent across `splitMediaFromOutput`, block-reply normalization, and streaming buffering. Reuse the same code-aware cleanup after audio-tag stripping, validate the shared fence-parser change or split it out, and add an active-version changelog Fixes entry.

Yes. The PR defects are also reproducible from source with `media: ./report.pdf` and `[[audio_as_voice]]` before an indented code line.

## Solution Assessment

No, not as submitted. The indented-code guard is the right narrow direction, but the case-sensitivity change and post-audio cleanup need to be corrected before this is the maintainable fix.
