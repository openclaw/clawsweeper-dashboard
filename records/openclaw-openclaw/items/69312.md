## Summary

This PR should stay open: it targets a real parser bug in the right area, but the latest external branch still lacks required after-fix real behavior proof and has blocking parser-contract regressions around lowercase `media:`, streaming buffering, and audio-tag cleanup.

## What This Changes

The branch updates MEDIA/fence parsing to skip 4-space/tab-indented MEDIA lines as code, tighten fence closing validation, adjust whitespace/case handling, and add parser tests.

## Best Possible Solution

Land a focused parser fix that skips MEDIA extraction from indented code blocks while preserving or explicitly migrating the lowercase directive contract across parser, streaming, reply tests, and audio cleanup, with redacted real behavior proof before merge.

## Reproduction Assessment

Yes, source-level. On current main, `splitMediaFromOutput` only excludes fenced spans and then uses a trim-start MEDIA gate, so a blank-preceded 4-space or tab-indented `MEDIA:` line can be parsed as an attachment.

## Solution Assessment

No. The parser guard is the right narrow area, but this branch is not yet the best merge candidate because it changes lowercase directive semantics, leaves streaming/audio cleanup misaligned, and lacks required real behavior proof.
