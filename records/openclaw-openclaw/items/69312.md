## Summary

Keep open. The central bug is valid and the PR is active, but the latest head still regresses established lowercase `media:` directive handling, leaves streaming buffering out of sync with the new parser rules, and lacks external real-behavior proof.

## What This Changes

The PR changes MEDIA parsing to skip MEDIA-looking lines in indented code blocks, preserves code indentation during cleanup, tightens markdown fence closing validation, and adds media/fence regression tests.

## Best Possible Solution

Land a focused parser fix that skips indented-code false positives while preserving lowercase directive support, keeping streaming/audio cleanup semantics consistent, and attaching redacted CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main skips only fenced spans and then trims indentation before case-insensitive `MEDIA:` matching, so a blank-preceded four-space or tab-indented `MEDIA:` code line can be extracted; I did not run tests in this read-only review.

## Solution Assessment

No, not as submitted. The indented-code guard is the right direction, but the patch should preserve lowercase directives and align streaming buffering plus audio cleanup with the new code-aware parser behavior.
