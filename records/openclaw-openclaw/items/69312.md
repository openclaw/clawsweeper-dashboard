## Summary

Keep open: current main still has the original indented-code false positive, so the PR is addressing a real parser bug, but the latest head is not merge-ready. It breaks existing lowercase media-directive compatibility, CI confirms that regression, and the audio-tag cleanup path still strips tab indentation from preserved code text.

## What This Changes

The PR updates media and fence parsing plus tests to skip `MEDIA:` extraction from indented code blocks, preserve code indentation during media cleanup, tighten malformed fence-closing handling, and make `MEDIA:` token matching uppercase-only.

## Best Possible Solution

Land a focused version that preserves indented-code `MEDIA:` lines while keeping existing lowercase directive compatibility across parser, block-reply, and streaming paths. Reuse the code-aware first-surviving-line normalization after audio-tag removal, keep the fence-parser change only with shared-consumer validation, and add the required Unreleased changelog entry.

## Reproduction Assessment

Yes. The main bug is reproducible by calling `splitMediaFromOutput` with a blank line followed by a 4-space or tab-indented `MEDIA:` line, and the PR regression is reproduced by the existing lowercase block-reply test that fails on the exact PR head.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right narrow direction, but the uppercase-only parser change and non-code-aware audio cleanup are not the safest maintainable fix.
