## Summary

Keep open: current main still has the indented-code MEDIA false-positive, but this PR still introduces parser compatibility regressions, leaves streaming buffering inconsistent with the new rule, and lacks required real behavior proof.

## What This Changes

The PR updates MEDIA parsing and fence parsing to avoid extracting MEDIA directives from indented code blocks, preserve retained code whitespace, make MEDIA matching case-sensitive, and add parser/fence tests.

## Best Possible Solution

Land a focused parser fix that skips MEDIA extraction from indented code blocks while preserving lowercase compatibility, updating streaming buffering consistently, keeping audio-tag cleanup code-aware, and adding redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: splitMediaFromOutput only excludes fenced spans and then trims indentation before MEDIA detection, so 4-space/tab indented code lines can be extracted; I did not execute tests in this read-only review.

## Solution Assessment

No. The indented-code guard belongs in splitMediaFromOutput, but this PR is not yet the best solution because it changes established lowercase behavior, leaves streaming buffering on the old directive rules, and applies non-code cleanup after audio tag stripping.
