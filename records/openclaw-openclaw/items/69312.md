## Summary

Keep open: the PR addresses a real source-reproducible media parsing bug, but current head still breaks established lowercase `media:` directive behavior, leaves streaming buffering inconsistent with the new code-aware parsing rule, and lacks after-fix real behavior proof.

## What This Changes

The PR changes shared media parsing and markdown fence parsing so indented-code `MEDIA:` lines are preserved as text, with focused parser regression tests.

## Best Possible Solution

Land a focused parser fix that preserves indented-code `MEDIA:` text while keeping lowercase directive compatibility, streaming buffering, and audio-tag cleanup aligned, then add redacted real CLI or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main excludes fenced spans only, then trims indentation before a case-insensitive `MEDIA:` gate, so a 4-space or tab-indented `MEDIA:` code line can enter extraction.

## Solution Assessment

No, not as submitted. The indented-code heuristic is a plausible narrow direction, but the branch must preserve existing lowercase media behavior and apply the same code-aware contract to streaming and audio-tag cleanup before merge.
