## Summary

## What This Changes

The PR adds an indented-code-line guard to `splitMediaFromOutput`, updates cleanup for preserved code lines, tightens fence close validation, and adds media/fence parser tests.

## Best Possible Solution

Keep the parser-side indented-code guard, but make streaming buffering and audio-tag cleanup share the same code-aware contract, correct the lowercase regression test, and require redacted real-run proof before merge.

## Reproduction Assessment

Yes, from source inspection. Current main only protects fenced code spans and then matches `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so blank-line 4-space/tab code examples can be extracted and removed.

## Solution Assessment

No, not yet. The parser-side guard is the right narrow direction, but the branch still needs the streaming and audio cleanup paths brought into the same contract and the lowercase test corrected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Fix the lowercase MEDIA regression test:** `src/media/parse.test.ts:421-425`
  - body: This test expects `media: note MEDIA:/tmp/a.png` to produce `mediaUrls`, but the branch keeps the greedy case-insensitive `MEDIA_TOKEN_RE`. The lowercase prefix consumes the whole line as one invalid payload, so no media is emitted unless the parser contract is intentionally changed.
  - confidence: 0.94
- **[P2] Keep streaming buffering code-aware:** `src/media/parse.ts:562-563`
  - body: The PR now preserves 4-space/tab `MEDIA:` lines as visible code, but streaming still buffers any whitespace-indented `MEDIA:` tail. A streamed reply ending with an indented code example can sit in `pendingTail` and be cleared by `reset()` before final flush, losing the text this patch is meant to preserve.
  - confidence: 0.88
- **[P2] Reuse the code-aware normalizer after audio tags:** `src/media/parse.ts:745-748`
  - body: After removing `[[audio_as_voice]]`, this branch always applies the non-code leading normalizer. If the tag is followed by a preserved tab or 4-space code line, it can strip indentation from the first visible code line and change the text the fix is meant to preserve.
  - confidence: 0.88
