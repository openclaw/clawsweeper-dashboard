## Summary

Keep open: the PR targets a real, source-reproducible parser bug, but latest head regresses current lowercase `media:` directive support, still strips indentation after audio-tag cleanup, and lacks required real behavior proof.

## What This Changes

The PR changes MEDIA and markdown fence parsing to skip `MEDIA:` extraction from indented code blocks, tighten fence closing validation, and add parser regression tests.

## Best Possible Solution

Land a focused parser fix that preserves indented code examples while keeping current lowercase directive support and using code-aware normalization after audio-tag stripping.

## Reproduction Assessment

Yes. Source inspection of current main shows `splitMediaFromOutput` skips only fenced spans, then trims indentation before case-insensitive MEDIA matching, so a blank-line-prefixed 4-space or tab-indented `MEDIA:` code line is extracted; I did not run tests in this read-only sweep.

## Solution Assessment

No, not yet. The indented-code guard is the right narrow direction, but the branch should preserve existing lowercase media semantics and reuse code-aware leading normalization after audio tags before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve lowercase MEDIA directives:** `src/media/parse.ts:17`
  - body: Current main deliberately supports lowercase `media:` in block replies, with a regression test expecting `media: ./report.pdf` to become `mediaUrl`/`mediaUrls`. Removing the regex `i` flag and making the entry guard case-sensitive turns that payload back into plain text, breaking the merged Telegram media delivery behavior.
  - confidence: 0.94
- **[P2] Preserve code indentation after audio tags:** `src/media/parse.ts:747`
  - body: The first pass chooses a code-aware leading-normalization regex, but after `parseAudioTag` strips `[[audio_as_voice]]` this line applies the generic regex again. Inputs such as `[[audio_as_voice]]\n\n    MEDIA:/tmp/code.png` lose leading code indentation after the tag is removed, so the PR's audio-tag regression case does not actually preserve the code block formatting it promises.
  - confidence: 0.9
