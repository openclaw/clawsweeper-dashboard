## Summary

Keep open. Current main still returns a hard `sessions_send` timeout after the target run has started and before launching the A2A announce flow, while related open PRs overlap on the timeout/delivery contract and need maintainer consolidation.

## What This Changes

The issue asks for `sessions_send` to start the A2A announce flow asynchronously after a post-accept wait timeout so delayed target replies can still be delivered.

## Best Possible Solution

Land one focused, backward-compatible fix that starts the existing A2A announce path after a true post-accept wait-window expiry, preserves the requester-owned child skip guard and reply baseline, distinguishes terminal errors from delayed replies, and adds regression coverage for delayed reply, no-new-reply, second-wait-timeout, and terminal-error cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `waitForAgentRunAndReadUpdatedAssistantReply` returning `timeout` after `startAgentRun`; current main then returns at `src/agents/tools/sessions-send-tool.ts:423` before any A2A announce launch.

## Solution Assessment

Yes, reusing the existing A2A flow with the accepted `runId` is the narrowest maintainable direction. The exact result status after the caller's wait window expires should be consolidated across the open PRs before landing.
