## Summary

Keep open. Current main still returns a hard `sessions_send` timeout after the target run starts and before launching the A2A announce flow, while the docs describe peer `sessions_send` delivery as an announce-backed path; the latest release has the same timeout branch. Related PRs are open and overlapping, so this needs maintainer consolidation rather than closure or another automated repair.

## What This Changes

Request that `sessions_send` still start the A2A announce flow asynchronously when the caller's wait window times out after a target run has been accepted.

## Best Possible Solution

Land one focused, backward-compatible fix that starts the existing A2A announce path after a true post-accept wait-window expiry, carries the pre-send reply baseline into any deferred reply read, preserves the owned-ACP-child skip guard, and adds regression coverage for delayed reply, no-new-reply, second-wait-timeout, and terminal-error cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `waitForAgentRunAndReadUpdatedAssistantReply` returning `timeout` after `startAgentRun`; current main then returns at `src/agents/tools/sessions-send-tool.ts:379` before any A2A announce launch.

## Solution Assessment

Yes, reusing the existing A2A flow with the accepted run id is the narrow maintainable direction. The safer implementation should explicitly distinguish post-accept wait expiry from a terminal failure and keep the existing delivery guards intact.
