## Summary

Current main and the shipped v2026.5.3 tag already keep A2A announce delivery alive after a non-terminal `sessions_send` wait timeout, with regression coverage for the delayed reply path and terminal-timeout guard.

## What This Changes

The issue requests that `sessions_send` start the A2A announce flow asynchronously after a wait-window timeout so delayed target replies can still reach the user.

## Best Possible Solution

Keep the shipped v2026.5.3 behavior: post-accept wait-window timeouts keep announce delivery pending while terminal run timeouts remain hard failures.

## Reproduction Assessment

No current-main failure path remains. Source inspection shows non-terminal wait timeouts now launch A2A delivery, and regression coverage exercises the original delayed-reply timeout sequence.

## Solution Assessment

Yes. The shipped implementation reuses the existing A2A helper with the accepted `runId` and preserves the terminal-timeout branch, which is the narrow maintainable fix for this bug.
