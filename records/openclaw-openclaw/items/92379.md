## Summary

Current source does not confirm the reported next-message failure because every embedded attempt constructs a fresh AgentSession from the resolved model. However, Telegram’s picker does omit the live-switch signal used by other model-change surfaces, so a narrower active-run state bug remains plausible and deserves focused maintainer follow-up.

## What This Changes

Investigate and fix stale in-memory session model metadata after a Telegram model switch so compaction uses the active model’s context window.

## Best Possible Solution

Add a focused regression test that changes the persisted Telegram model, starts the next embedded turn, and verifies both the newly created AgentSession and threshold compaction use the selected registry model; only add Telegram live-switch signaling if maintainers intend picker changes to interrupt or replace an already active run.

## Reproduction Assessment

Unclear. The report is detailed, but current source creates a fresh AgentSession with the resolved model for each next attempt, and no focused test or log demonstrates stale state surviving that boundary.

## Solution Assessment

Unclear. Using the existing pending live-switch contract may be appropriate for active runs, but the reported next-turn path should first be captured in a regression test; adding registry lookup inside compaction is not the preferred solution.
