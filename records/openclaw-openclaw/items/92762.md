## Label Justifications

- `P1`: The PR targets and can regress a documented outbound interception hook used by safety, rewrite, and cancellation plugins on primary channel reply paths.
- `merge-risk: 🚨 compatibility`: The diff changes where an existing documented plugin hook runs and removes delivery-layer context that plugins may already depend on.
- `merge-risk: 🚨 message-delivery`: A `message_sending` cancel or rewrite can be skipped for Telegram direct and fallback deliveries, causing messages to be sent when plugins expected suppression or modification.
- `merge-risk: 🚨 security-boundary`: `message_sending` is a safety boundary for DLP and policy plugins, and the PR can make Telegram sends fail open on paths outside the dispatcher.

## Summary

Keep open: the reported hook bypass is real, but this PR is not merge-safe because it removes Telegram's delivery-owned `message_sending` enforcement while direct native-command, fallback, media, and voice delivery paths still call `deliverReplies` outside the dispatcher hook chain.

## What This Changes

The PR adds `message_sending` to configured shared dispatcher `beforeDeliver` chains and deletes Telegram `deliverReplies`' direct `message_sending` invocation.

## Best Possible Solution

Keep the shared dispatcher repair, but preserve Telegram delivery-owned hook coverage through an explicit exactly-once ownership design that covers direct `deliverReplies`, native commands, fallbacks, media, voice fallback, and streamed finalization.

## Reproduction Assessment

Yes. Source inspection on current main shows configured `beforeDeliver` omits `message_sending`, and the existing test still asserts `runMessageSending` is not called for a custom `beforeDeliver` path.

## Solution Assessment

No. The dispatcher addition is a plausible part of the fix, but removing Telegram's delivery-owned hook is not the best solution because direct delivery, fallback, media, and spokenText paths still need exactly-once `message_sending` coverage.
