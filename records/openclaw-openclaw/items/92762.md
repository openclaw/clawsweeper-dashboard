## Label Justifications

- `P1`: The PR targets and can regress a documented outbound interception hook used by safety, rewrite, and cancellation plugins on primary channel reply paths.
- `merge-risk: 🚨 compatibility`: The diff changes where an existing documented plugin hook runs and removes delivery-layer context that plugins may already depend on.

## Summary

Keep this PR open but not merge-ready: it fixes the shared dispatcher bypass in part, but the patch removes Telegram's delivery-owned `message_sending` hook while direct Telegram delivery paths still call `deliverReplies` outside the dispatcher hook chain.

## What This Changes

The PR adds `message_sending` to configured auto-reply `beforeDeliver` chains, updates one dispatcher test, and removes Telegram `deliverReplies`' direct `message_sending` hook.

## Best Possible Solution

Keep the shared dispatcher repair, but preserve or rehome Telegram delivery-owned hook coverage with an explicit exactly-once model covering direct `deliverReplies`, native/fallback/media/voice paths, and streamed finalization.

## Reproduction Assessment

Yes. Current-main source shows configured `beforeDeliver` excludes `message_sending`, Telegram supplies configured `beforeDeliver`, and existing delivery tests show `deliverReplies` owns richer Telegram hook context.

## Solution Assessment

No. Adding the dispatcher hook is a plausible part of the fix, but deleting Telegram's delivery-owned hook is not the best solution because direct delivery, fallback, media, and spokenText paths still need exactly-once `message_sending` coverage.
