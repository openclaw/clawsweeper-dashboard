## Label Justifications

- `P1`: The PR targets and can regress a documented outbound interception hook used by safety, rewrite, and cancellation plugins on primary channel reply paths.

## Summary

Keep open: the PR addresses a real dispatcher hook bypass, but the branch is not merge-ready because it removes Telegram's delivery-level `message_sending` owner and only supplies unit-test proof.

## What This Changes

The PR adds `message_sending` to configured auto-reply `beforeDeliver` chains, updates one dispatcher test, and removes Telegram `deliverReplies` direct `message_sending` handling.

## Best Possible Solution

Keep the shared dispatcher repair, but preserve or rehome Telegram delivery-owned hook coverage with an explicit exactly-once model covering direct `deliverReplies`, native/fallback/media/voice paths, and streamed finalization.

## Reproduction Assessment

Yes. Current-main source shows configured `beforeDeliver` excludes `message_sending`, Telegram supplies configured delivery hooks, and current tests document delivery-layer hook context; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. Adding `message_sending` to configured dispatcher chains is part of the fix, but deleting Telegram's delivery-level hook is too narrow because direct delivery and fallback paths still need exactly-once hook coverage.
