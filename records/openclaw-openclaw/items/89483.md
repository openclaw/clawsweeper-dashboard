- `P1`: The linked bug affects Control UI session history after failed agent runs, and the PR currently has a blocking duplicate-history edge in that same workflow.
- `merge-risk: 🚨 session-state`: Merging as-is can persist an extra assistant error row into the session transcript when runtime persistence already wrote a visible error.
- `merge-risk: 🚨 message-delivery`: The changed Gateway delivery/persistence path can make refreshed chat.history deliver duplicate assistant error messages for one failed run.

## Summary

Keep open: the bug is real and the PR is aimed at the right Gateway/session boundary, but it has a blocking duplicate-history edge because the returned-error branch forces a Gateway assistant-error append even after runtime assistant-error persistence was reported.

## What This Changes

The PR adds an internal assistant-error persistence callback, forwards it through runner paths, and appends a Gateway-injected assistant error transcript entry for failed started Control UI agent runs with regression tests.

## Best Possible Solution

Land a Gateway fix that persists exactly one display-safe assistant error for failed started runs, while skipping or replacing visible runtime-persisted errors and still covering hidden/internal stubs.

## Reproduction Assessment

Yes: current main and v2026.6.6 compute returnedAgentErrorPayloads and call broadcastChatError without an assistant transcript append, and the PR body supplies before/after Gateway/WebSocket proof. I did not run tests because this review is read-only.

## Solution Assessment

No: the implementation is in the right layer, but forcing the append after runtime persistence is not the best fix because the callback contract covers any persisted assistant stopReason:error message. It should distinguish visible runtime rows from hidden stubs before appending another Gateway row.

- **[P1] Avoid forcing duplicate persisted error rows:** `src/gateway/server-methods/chat.ts:4210`
  - body: `onAssistantErrorMessagePersisted` fires after any assistant `stopReason: "error"` row is persisted, not only after hidden/internal stubs.
