## Best Possible Solution

Keep the reconciliation in the shared Skill Workshop service, but run stale marking through the same target-lock and fresh pending-record guard used by lifecycle mutations.

## Reproduction Assessment

Yes, source-level: current main list/inspect only returns persisted proposal state, while gateway, tool, CLI, and UI surfaces all rely on those shared service methods. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not as submitted: the shared-service location is right, but the mutation must be lock-protected and based on a fresh pending record. The safer fix is a small locked reconciliation helper plus a stale-manifest or race regression test.
