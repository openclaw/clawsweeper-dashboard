## Summary

Keep open: the PR fixes the stale-create symptom, but it persists proposal status from `operator.read` list/inspect paths even though those Gateway methods are documented and declared as read-only, so it needs a maintainer/API decision or a derived-status rewrite before merge.

## What This Changes

The PR adds Skill Workshop stale reconciliation for pending create proposals whose target `SKILL.md` already exists and covers list, inspect, and terminal-status preservation in service tests.

## Best Possible Solution

Keep the shared-service reconciliation, but either return derived stale status from read paths or explicitly move durable stale marking behind an admin/lifecycle boundary with maintainer-approved contract updates and tests.

## Reproduction Assessment

Yes, source-reproducible: current main's scoped list/inspect paths return persisted pending status after the target `SKILL.md` appears, while apply later throws when the create target already exists. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written: service-level reconciliation is the right layer, but persisting from read-scoped list/inspect methods is not the safest contract. A derived-status read path or admin-scoped durable reconciliation is a cleaner fix.
