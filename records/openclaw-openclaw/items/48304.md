## Summary

Keep open. Current main has adjacent owner-only and per-sender whole-tool policy pieces, but it still does not provide trusted per-action/per-sender authorization or owner confirmation for `message.send`, and the related action-policy/session-policy threads are still open.

## What This Changes

The issue requests gateway-enforced tool authorization that validates sender identity for protected actions such as `message` `send`, with an optional owner-confirmation flow.

## Best Possible Solution

Define a maintainer-approved typed action/sender-aware authorization contract, enforced before multiplexed tool actions execute, with docs and regression coverage for non-owner attempts to trigger `message.send`.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/security policy request. The source-level gap is high confidence because current policy admits the single `message` tool by name before reading `action`, and the public hook path does not expose trusted sender identity.

## Solution Assessment

Unclear until maintainer design is done. Sender/action-aware enforcement is the right direction for the reported safety gap, but the narrow maintainable fix should be a shared policy or trusted hook contract rather than a one-off `message` config branch.
