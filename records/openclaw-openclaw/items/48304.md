## Summary

Keep open. Current main still has whole-tool policy, group-scoped per-sender policy, and cross-context message controls, but no generic per-action/per-sender gate for protected message sends; related open trackers cover parts of the design but do not fully supersede this security-sensitive request.

## What This Changes

Requests gateway-enforced sender and action authorization for protected message tool actions, with optional owner confirmation and a generic tool authorization model.

## Best Possible Solution

Define a maintainer-approved action-aware and sender-aware tool authorization contract that uses verified inbound sender identity, composes with DM toolsBySender and action-level deny, and is enforced before message/gateway side effects execute.

## Reproduction Assessment

Yes by source inspection, though I did not run a live Slack setup. Current main passes trusted sender metadata through the message path but has no configured per-action sender gate before message sends execute.

## Solution Assessment

Unclear until maintainer design is done. Per-action sender enforcement is the right direction for the reported gap, but the implementation should coordinate with https://github.com/openclaw/openclaw/issues/13948 and https://github.com/openclaw/openclaw/issues/53760 rather than adding an ad hoc message-only check.
