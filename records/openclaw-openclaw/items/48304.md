## Summary

Keep open. Current main still lacks a typed per-action, per-sender authorization contract for `message(action="send")`; related action-level policy work is open but does not fully cover this owner/sender validation request.

## What This Changes

Request gateway/tool-level policy that blocks or confirms protected message actions based on trusted sender or owner identity.

## Best Possible Solution

Define a maintainer-approved, typed action-aware and sender-aware authorization contract that can block or require owner approval before `message` send dispatch, while aligning with the broader action-level policy tracker.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible, but not live-reproduced here: current main admits tools by name, then `message` reads `action` during execution with no shared action-plus-sender policy gate before dispatch.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer design is done; the requested direction is valid, but config grammar, precedence, confirmation UX, and relation to https://github.com/openclaw/openclaw/issues/13948 should be settled before implementation.
