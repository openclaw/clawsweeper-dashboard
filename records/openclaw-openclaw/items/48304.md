## Summary

Keep open. Current main has sender metadata, owner-only tool filtering, group-scoped `toolsBySender`, and `before_tool_call` hooks, but it still lacks a built-in policy that can enforce `message` actions such as `send` by trusted sender/owner before execution; the related action-level tracker is complementary but does not cover the sender/owner authorization part.

## What This Changes

The issue requests gateway-enforced per-action, per-sender authorization for protected message/tool actions in multi-user deployments.

## Best Possible Solution

Define a maintainer-approved hard authorization contract that evaluates trusted sender/owner identity plus tool action before execution, preferably sharing grammar and enforcement with the action-level policy work rather than adding a Slack-only or prompt-only workaround.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a security-feature request for a new authorization contract. Source inspection gives high confidence that current main lacks the requested per-action, per-sender enforcement even though it carries sender metadata in parts of the message path.

## Solution Assessment

Unclear until maintainer design is done. Hard enforcement is the right direction, but the safest solution should be a shared policy/hook contract with explicit precedence and approval semantics, not an ad hoc check inside only the Slack or message send path.
