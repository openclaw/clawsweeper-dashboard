## Summary

Keep open. Current main has adjacent sender identity, owner-only tool filtering, and group per-sender tool policy plumbing, but it still lacks a configured gateway-enforced sender/action gate for `message.send` or `message.poll`; the two related guardrail PRs were closed unmerged and do not supersede the remaining security-sensitive design work.

## What This Changes

Add gateway-enforced per-action/per-sender authorization for protected `message` tool actions, with possible owner confirmation or a generic tool authorization framework.

## Best Possible Solution

Add a core/API-backed authorization check before `message.send` and `message.poll` execute, using trusted requester identity and a documented denial result; broaden to generic policy providers or owner confirmation only after the stable action/requester contract is chosen.

## Reproduction Assessment

Yes, source-reproducible. The issue provides concrete Slack DM steps, and current main forwards trusted requester identity through message actions without a sender-based denial gate for `send` or `poll`.

## Solution Assessment

Yes for the narrow gateway-enforced message action boundary, but the exact Option A/B/C API needs maintainer judgment. Prompt rules, whole-tool deny, DM allowlists, and cross-context blocking are not equivalent to a trusted sender/action authorization check.
