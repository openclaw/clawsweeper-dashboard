## Summary

Keep open for maintainer product/security review: current main has plugin hooks that can support an external authorization plugin, but the requested built-in sender/action policy and owner-confirmation semantics are not implemented and are not fully covered by the open action-level deny tracker.

## What This Changes

The issue requests gateway-level per-sender, per-action authorization for protected tool calls, especially outbound `message` sends from shared workspace assistants.

## Best Possible Solution

Maintainers should choose one canonical direction: a ClawHub policy plugin using existing hooks, or a core action-aware policy design that extends https://github.com/openclaw/openclaw/issues/13948 with sender/owner semantics.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature/security policy request. Source inspection does confirm that built-in policy is still whole-tool/name based and does not implement sender-conditioned `message.send` authorization.

## Solution Assessment

Unclear. A plugin-based policy aligns with the current architecture, but the requested built-in config and owner-confirmation behavior require maintainer product/security judgment before implementation or closure.
