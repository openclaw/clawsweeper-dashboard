## Summary

Keep this issue open. Current main has adjacent protections, including owner-only filtering for some core tools, trusted/bundled pre-tool policies, channel allowlists, and forwarding of requester identity through message actions, but it still lacks a configured per-action sender gate for `message.send` or `message.poll`. The related guardrail PRs were closed unmerged, so they do not supersede the remaining work.

## What This Changes

Request a gateway-enforced tool authorization policy that validates the trusted sender before allowing protected message actions such as sends, polls, DMs, or channel posts.

## Best Possible Solution

Add a core/API-backed authorization path checked before `message.send` and `message.poll`, using trusted requester identity and returning a configured denial reason. Then document and test the contract, and only broaden to generic policy providers or owner-confirmation once action classification and requester identity are part of the stable pre-tool policy surface.

## Reproduction Assessment

Yes. The issue gives concrete Slack DM reproduction steps, and current source inspection confirms that sender identity is forwarded through the message action path without a sender-based denial gate for `send` or `poll`.

## Solution Assessment

Yes for the narrow core-enforced message action policy; prompt rules, whole-tool deny, DM allowlists, and cross-context blocking are not equivalent. The safer path is to enforce the first protected action at the trusted message boundary, then generalize the hook/policy API deliberately.
