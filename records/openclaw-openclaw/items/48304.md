## Summary

This is useful security policy work, but current main already exposes plugin hooks that can inspect `message` tool params, correlate inbound sender metadata, block execution, or require approval before the tool runs. Under the current OpenClaw scope guidance, this should move to a ClawHub/community policy plugin rather than adding another core guardrails/config subsystem.

## What This Changes

Request sender-aware, per-action authorization for protected tool calls, especially blocking or confirming `message.send` when the requester is not the agent owner.

## Best Possible Solution

Build and publish a ClawHub policy plugin that records inbound sender identity, checks `before_tool_call` for `message` plus protected actions such as `send`, and either blocks fail-closed or uses plugin approvals for owner confirmation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction: this is a feature request for a new authorization policy surface, and source inspection confirms the exact built-in per-action sender config is not present in core.

## Solution Assessment

Is this the best way to solve the issue? Yes for repository cleanup: the existing plugin hook and message-hook APIs can support this policy outside core, while the separate core action-deny grammar remains tracked at https://github.com/openclaw/openclaw/issues/13948.
