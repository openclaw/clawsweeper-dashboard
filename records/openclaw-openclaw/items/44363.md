## Summary

Keep open. Current main has cron run history, failure counters, failure alerts, and stuck-marker cleanup, but it does not include this PR's doctor reliability report; the proposed patch is also stale against current cron and doctor contracts and is not merge-ready.

## What This Changes

The PR adds cron reliability helpers and a doctor hook, plus memory-search validation/onboarding and agent lifecycle/watchdog utilities with tests.

## Best Possible Solution

Replace or rework this as a focused cron diagnostics change that uses current CronJobState, schedule.expr, durable run logs, and the doctor health contribution registry. Keep memory-search onboarding/config validation and agent watchdog work in separate purpose-specific PRs, and coordinate broader health endpoint/status-warning scope with #13593.

## Reproduction Assessment

Yes. A static reproduction is enough for the blocking issues: the PR diff references job.state.status and job.schedule.cron, while current main defines lastRunStatus/runningAtMs and schedule.expr instead.

## Solution Assessment

No. The cron diagnostics direction is useful, but this PR is too broad and stale; the maintainable path is a narrower current-main cron patch plus separate PRs for unrelated memory-search and watchdog changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the real cron state field in doctor output:** `src/commands/doctor-cron.ts:219`
  - body: The new doctor hook formats problematic jobs with `job.state.status`, but current `CronJobState` has no `status` field; it uses `lastRunStatus`, `runningAtMs`, and related fields. This will fail typechecking after rebase and prevents the reliability diagnostic from compiling.
  - confidence: 0.94
- **[P1] Use schedule.expr for cron recommendations:** `src/cron/reliability.ts:232`
  - body: Cron schedules in this codebase store the expression as `schedule.expr`, not `schedule.cron`. This condition targets a non-existent field, so the recommendation logic is stale against the current cron contract and will not typecheck cleanly.
  - confidence: 0.92
- **[P2] Reset the global watchdog after stopping it:** `src/infra/agent-watchdog.ts:194-198`
  - body: `stopGlobalWatchdog()` stops the singleton but leaves `globalWatchdog` set. A later `getGlobalWatchdog(newConfig)` returns the stopped old instance and ignores the new configuration, which breaks reconfiguration and test isolation for the new watchdog module.
  - confidence: 0.88
- **[P2] Wire cron health through the current doctor registry:** `src/commands/doctor.ts:226`
  - body: This PR wires the reliability check through the old `doctor.ts` flow, but current main runs doctor checks from `src/flows/doctor-health-contributions.ts`. The cron reliability check needs to be registered as a doctor health contribution after rebase, or it will be unreachable.
  - confidence: 0.83
