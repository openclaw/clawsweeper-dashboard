- **[P1] Use the real cron state field in doctor output:** `src/commands/doctor-cron.ts:224`
  - body: The new doctor hook formats problematic jobs with `job.state.status`, but current `CronJobState` has no `status` field; it uses `lastRunStatus`, `runningAtMs`, and related fields. This will fail typechecking after rebase and prevents the diagnostic from compiling.
  - confidence: 0.94
- **[P1] Use schedule.expr for cron recommendations:** `src/cron/reliability.ts:232-233`
  - body: Cron schedules in this codebase store the expression as `schedule.expr`, not `schedule.cron`. This condition targets a non-existent field, so the recommendation logic is stale against the current cron contract and will not typecheck cleanly.
  - confidence: 0.93
- **[P2] Register cron diagnostics in the current doctor flow:** `src/commands/doctor.ts:229`
  - body: This PR wires the reliability check through the old `doctor.ts` flow, but current main runs doctor checks from `src/flows/doctor-health-contributions.ts`. The cron reliability check needs to be registered as a doctor health contribution after rebase, or it will be unreachable.
  - confidence: 0.86
- **[P2] Reset the global watchdog singleton:** `src/infra/agent-watchdog.ts:194-198`
  - body: `stopGlobalWatchdog()` stops the singleton but leaves `globalWatchdog` set. A later `getGlobalWatchdog(newConfig)` returns the stopped old instance and ignores the new configuration, which breaks reconfiguration and test isolation for the new watchdog module.
  - confidence: 0.9
