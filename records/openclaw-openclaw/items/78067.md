## Summary

Keep open: the PR targets a real source-level bug, but the patch rehydrates omitted web config for every refresh, which can keep intentionally removed web provider metadata active until restart. It also still lacks after-fix real behavior proof from a gateway/config-write run.

## What This Changes

The PR preserves omitted `tools.web` and plugin web config during secrets runtime refresh and adds a fast-path regression test for a stripped refresh source.

## Best Possible Solution

Preserve web metadata only for confirmed stripped-source refreshes or repair the writer/source projection boundary, while honoring explicit removal or disablement and covering both cases with regression tests plus real gateway proof.

## Reproduction Assessment

Yes, source-level: activate a snapshot with configured `tools.web` or plugin web metadata, then refresh with a source config that lacks that surface; current main can prepare and activate empty web metadata through the fast path. I did not run a live gateway repro in this read-only review.

## Solution Assessment

No: the proposed approach preserves omitted web surfaces for every refresh, but normal config writes can omit those paths because the user intentionally removed them. A safer fix needs a partial-source signal or writer-side repair plus explicit-removal coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Do not rehydrate explicitly removed web config:** `src/secrets/runtime.ts:526-530`
  - body: The refresh hook now restores omitted web-tool config from the active snapshot before every refresh. Since config writes reread and publish the canonical persisted source config, omission can be an intentional removal; this can keep old web provider or plugin web config active in memory until restart. Distinguish stripped-source refreshes from explicit removals, or fix the writer/projection path that loses the surface.
