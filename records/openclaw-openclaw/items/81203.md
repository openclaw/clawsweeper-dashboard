## Summary

Keep this PR open, but it is not ready to merge: the reported auto-TTS trust gap still exists on current main, the one-line patch only fixes the direct return path, and the contributor has not supplied inspectable after-fix proof.

## What This Changes

The PR adds `trustedLocalMedia: true` to auto-TTS media payloads returned by `maybeApplyTtsToPayload`.

## Best Possible Solution

Carry the internal trust signal through every OpenClaw-generated auto-TTS local audio payload that WebChat may embed, including reconstructed TTS-only final payloads, while preserving the existing local-root and assistant-media checks.

## Reproduction Assessment

Yes, at source level. Current main produces local auto-TTS media without `trustedLocalMedia`, while the WebChat audio embedding helper rejects local audio unless that flag is true; I did not run a live WebChat session in this read-only review.

## Solution Assessment

No. Adding the flag in `maybeApplyTtsToPayload` is necessary for the direct path, but the best fix also propagates it through the streamed-final and ACP TTS-only payload reconstruction paths with focused regression coverage.
