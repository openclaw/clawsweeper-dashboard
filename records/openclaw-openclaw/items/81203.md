## Summary

Keep this PR open, but it should not merge as-is: the one-line change is directionally correct for direct auto-TTS payloads, yet source inspection shows reconstructed TTS-only dispatch payloads can still drop the same trust flag, and the external PR proof is prose-only.

## What This Changes

The PR adds `trustedLocalMedia: true` to the auto-TTS media payload returned by `maybeApplyTtsToPayload`.

## Best Possible Solution

Carry the internal local-media trust signal through every auto-TTS local audio payload that webchat may persist or broadcast, including reconstructed TTS-only dispatch payloads, while keeping the existing local-root and assistant-media safeguards.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows current main returns auto-TTS media without the internal trust flag while the webchat media resolver requires `trustedLocalMedia === true`; the exact user flow still needs real webchat proof.

## Solution Assessment

No. Adding the flag in `maybeApplyTtsToPayload` is necessary for the direct path, but it is not the complete fix because later TTS-only dispatch paths reconstruct media payloads and still omit the flag.
