## Summary

Keep open. The bug diagnosis matches current source, but the patch as written fails production type checking because `speech-core` returns the public plugin SDK `ReplyPayload`, which intentionally omits `trustedLocalMedia`; the PR also still needs artifact-backed real behavior proof.

## What This Changes

Adds `trustedLocalMedia: true` to the auto-TTS payload returned by `maybeApplyTtsToPayload` in `extensions/speech-core/src/tts.ts`.

## Best Possible Solution

Preserve the narrow auto-TTS behavior fix, but route the returned payload through an internal reply-payload seam or type that can legally carry `trustedLocalMedia`, then verify it with focused tests and real Control UI playback proof alongside the companion webchat rendering change.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path even though I did not run a live TTS setup: current main auto-TTS returns a local `mediaUrl` without `trustedLocalMedia`, and webchat embedding rejects local audio unless that flag is true.

## Solution Assessment

No. The direction is right, but the implementation needs to use an internal payload contract instead of adding an internal-only field to the public plugin SDK `ReplyPayload`; https://github.com/openclaw/openclaw/pull/81204 also needs to be considered for visible webchat playback.
