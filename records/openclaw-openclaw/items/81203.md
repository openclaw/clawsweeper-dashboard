## Summary

Keep open: the reported root cause is plausible on current main, but the patch only fixes the direct auto-TTS return path and still misses reconstructed final TTS-only payloads; the PR also lacks inspectable after-fix real behavior proof.

## What This Changes

This PR adds the internal `trustedLocalMedia` flag to successful auto-TTS payloads returned from `extensions/speech-core/src/tts.ts`.

## Best Possible Solution

Carry the internal local-media trust signal through every OpenClaw-generated auto-TTS local audio payload that webchat can persist or broadcast, including TTS-only dispatch and ACP finalization, while preserving the existing localRoots and file-safety checks.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows current main omits `trustedLocalMedia` from auto-TTS while the webchat resolver and tests reject untrusted local audio paths.

## Solution Assessment

No. Adding the flag in `maybeApplyTtsToPayload` is necessary for the direct path, but the fix is incomplete because later final-TTS dispatch paths reconstruct payloads and drop the trust signal.
