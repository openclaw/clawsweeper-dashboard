## Summary

Keep open: the PR addresses a real auto-TTS webchat trust gap for the direct payload path, but the patch is incomplete because TTS-only dispatch paths still reconstruct local audio payloads without the trust flag, and the external PR still lacks inspectable after-fix behavior proof.

## What This Changes

The PR adds `trustedLocalMedia: true` to the successful auto-TTS payload returned by `maybeApplyTtsToPayload`.

## Best Possible Solution

Carry the internal local-media trust signal through every OpenClaw-generated auto-TTS local audio payload that webchat may persist or broadcast, including reconstructed TTS-only dispatch payloads, while keeping the existing local-root and assistant-media safeguards.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows current main returns auto-TTS local media without the trust flag while webchat embedding requires `trustedLocalMedia === true`; the exact webchat flow still needs real proof.

## Solution Assessment

No. Adding the flag in `maybeApplyTtsToPayload` is necessary for the direct path, but the fix should also carry the trust signal through reconstructed final TTS-only payloads in dispatch paths.
