## Summary

Keep open: the PR addresses a real source-level auto-TTS/webchat gap, but the patch is incomplete because TTS-only dispatch paths still reconstruct local audio payloads without the trust flag, and the contributor proof is prose-only rather than inspectable after-fix evidence.

## What This Changes

Adds `trustedLocalMedia: true` to the auto-TTS media payload returned by `maybeApplyTtsToPayload`.

## Best Possible Solution

Carry the internal local-media trust signal through every OpenClaw-generated auto-TTS local audio payload that webchat may embed, including reconstructed TTS-only dispatch payloads, with focused regression coverage and inspectable webchat proof.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a clear current-main path: auto-TTS local audio lacks `trustedLocalMedia`, webchat rejects local audio without that flag, and the proposed patch still loses the flag in reconstructed TTS-only dispatch payloads.

## Solution Assessment

No. Adding the flag in `maybeApplyTtsToPayload` is necessary for the direct path, but it is not the complete fix because `dispatch-from-config.ts` and `dispatch-acp.ts` rebuild payloads that omit the same internal trust signal.
