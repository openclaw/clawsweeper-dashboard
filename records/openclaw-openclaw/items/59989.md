## Summary

Keep open, but the patch is not mergeable as-is: the cleanup direction is reasonable, yet this branch changes the unsubscribe contract without updating callers, can still unsubscribe before the final compaction-end event, uses stale event names in new tests, and lacks after-fix real behavior proof.

## What This Changes

The branch makes embedded Pi session unsubscribe async, waits up to five seconds after aborting compaction before removing the session listener, stores a cleanup resolver, and adds unsubscribe cleanup tests.

## Best Possible Solution

Land a narrow compaction-teardown fix that preserves or consistently awaits the unsubscribe contract, waits for the final current `compaction_end` before listener removal, and proves timeout/retry behavior with deterministic tests plus real runtime evidence.

## Reproduction Assessment

Yes, at source level. Current main aborts compaction and immediately removes the session listener, and the PR-introduced retry bug follows by processing a `compaction_end` with `willRetry: true` before the final end event.

## Solution Assessment

No. Waiting for cleanup is the right direction, but this implementation is not the safest fix because it changes `unsubscribe` to async without updating teardown callers and resolves cleanup before the final retry sequence is complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Await unsubscribe cleanup before continuing teardown:** `src/agents/pi-embedded-subscribe.ts:735`
  - body: Changing `unsubscribe` to `async` changes the subscription completion contract, but the embedded runner still calls it synchronously in cleanup. Listener removal can now happen after teardown has moved on, and a later `sessionUnsubscribe()` throw becomes an unhandled rejection instead of hitting the existing `try/catch`; either preserve a synchronous API or update callers to await an idempotent cleanup promise.
  - confidence: 0.9
- **[P1] Resolve cleanup only on the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The cleanup resolver runs for every compaction-end event, including `willRetry=true`. A retry end means another final end is still pending, so this can remove the listener before the final notification the PR is trying to preserve.
  - confidence: 0.88
- **[P2] Use current compaction event names in the new tests:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:19-31`
  - body: Current main and the published Pi SDK use `compaction_start` and `compaction_end`, but these tests emit the old `auto_compaction_start` and `auto_compaction_end` names. After rebase, they will not exercise the handler path that needs coverage.
  - confidence: 0.86
- **[P3] Clear the cleanup timeout after a successful wait:** `src/agents/pi-embedded-subscribe.ts:720-731`
  - body: When the cleanup promise wins the race, the timeout remains pending until it fires five seconds later. Store the timeout handle and clear it in `finally` so successful cleanup does not leave a needless timer alive.
  - confidence: 0.78
- **[P3] Drive the timeout test with fake timers:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:53-55`
  - body: The timeout test awaits the real five-second cleanup timeout, adding fixed latency and CI flake risk. Use fake timers and advance past the timeout deterministically.
  - confidence: 0.82
