## Summary

Keep open: this PR targets a real compaction teardown race, but the branch is not mergeable as written because it changes unsubscribe completion semantics, can still unsubscribe before the final compaction end during retries, and is stale against the current compaction event names.

## What This Changes

The branch makes embedded Pi unsubscribe async, waits up to 5 seconds for compaction cleanup after abort, stores a cleanup resolver in subscription state, and adds unsubscribe cleanup tests.

## Best Possible Solution

Land a rebased, narrow compaction-teardown fix that preserves unsubscribe completion semantics, waits through the final compaction_end event, uses current event names, and includes deterministic regression coverage plus real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main aborts in-flight compaction and immediately removes the session listener, so a later compaction_end event can be lost before the frontend sees the final notification.

## Solution Assessment

No, not as written. Waiting before listener removal is the right direction, but this patch changes unsubscribe to async without updating the production caller contract and can still resolve cleanup before the final retry completion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the unsubscribe completion contract:** `src/agents/pi-embedded-subscribe.ts:699`
  - body: Making `unsubscribe` async changes the cleanup contract, but production cleanup still calls it synchronously inside a try/catch. Timeout teardown can continue before `sessionUnsubscribe()` runs, and async failures bypass that catch; either keep the API synchronous or update callers to await a shared cleanup promise.
  - confidence: 0.9
- **[P1] Resolve cleanup only on the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The added resolver runs for both retry and final compaction-end events. When `willRetry` is true, another completion is still pending, so this can remove the listener before the final notification this PR is trying to preserve.
  - confidence: 0.9
- **[P2] Update tests to current compaction event names:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:18-31`
  - body: Current main dispatches `compaction_start` and `compaction_end`, but the new tests emit the old `auto_compaction_start` and `auto_compaction_end` names. After rebase, these tests will not exercise the handler path they are meant to prove.
  - confidence: 0.88
- **[P3] Clear the cleanup timeout after success:** `src/agents/pi-embedded-subscribe.ts:684-686`
  - body: When the cleanup promise wins the race, the timeout created here is left pending until it fires. Clear the handle in `finally` so successful unsubscribe cleanup does not keep a needless timer alive for five seconds.
  - confidence: 0.78
- **[P3] Use fake timers for the timeout test:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:52-56`
  - body: This test waits for the real 5000 ms cleanup timeout, which adds fixed latency and avoidable CI flake risk. Advance the timeout with Vitest fake timers instead.
  - confidence: 0.82
