## Summary

The main/code-path pass confirms this is still a live agents teardown problem, but the PR branch is not merge-ready: it is conflicting, lacks contributor real-behavior proof, and the proposed async unsubscribe path has correctness holes around teardown ordering and retry compaction events.

## What This Changes

The PR changes embedded Pi session unsubscribe to wait up to five seconds for compaction cleanup, adds a cleanup resolver to subscription state, and adds regression tests for unsubscribe during compaction.

## Best Possible Solution

Land a rebased teardown fix that preserves the existing unsubscribe completion contract or updates all callers deliberately, waits through the final `compaction_end`, uses current event names, and includes deterministic regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current main aborts in-flight compaction and immediately removes the session listener, while the handler path can still receive a later `compaction_end`. I did not establish a live failing setup in this read-only review.

## Solution Assessment

No, not as written. Waiting before listener removal is the right direction, but this patch changes a synchronous cleanup contract without updating callers and can still stop listening before the final retry completion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the unsubscribe completion contract:** `src/agents/pi-embedded-subscribe.ts:699`
  - body: Making `unsubscribe` async changes completion semantics, but the embedded runner cleanup still calls it synchronously inside a try/catch. Timeout or abort cleanup can continue before `sessionUnsubscribe()` runs, and async failures bypass the local catch, so either keep this API synchronous or update callers to await a shared cleanup promise.
  - confidence: 0.9
- **[P1] Resolve cleanup only after the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The resolver runs for `willRetry=true` as well as the final end event. When a retry is pending, this can remove the listener before the final `compaction_end` notification that the PR is trying to preserve.
  - confidence: 0.91
- **[P2] Use current compaction event names in tests:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:19-31`
  - body: The new tests emit `auto_compaction_start` and `auto_compaction_end`, but current main and the documented handler path use `compaction_start` and `compaction_end`, so the rebased tests would not exercise the intended handler path.
  - confidence: 0.88
- **[P2] Register the cleanup waiter without a missed-end window:** `src/agents/pi-embedded-subscribe.ts:673-681`
  - body: The state check happens before storing `compactionCleanupResolve`; if `compaction_end` fires in that gap, unsubscribe waits for the timeout even though cleanup already completed. Register the waiter before the vulnerable state transition or make the state check and resolver registration atomic in this subscription state.
  - confidence: 0.8
- **[P3] Use fake timers for timeout coverage:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:54-59`
  - body: The timeout test awaits the real five-second cleanup timeout on every run, adding fixed latency and CI flake risk. Use Vitest fake timers to advance this path deterministically.
  - confidence: 0.82
