## Summary

Keep open: the underlying compaction teardown race is still present on current main and in the latest release, but this PR is not merge-ready because it changes the unsubscribe completion contract, can still unsubscribe before a retry-final compaction event, uses stale event names in added tests, conflicts with main, and lacks real behavior proof.

## What This Changes

The PR makes embedded Pi session unsubscribe async, adds a compaction cleanup waiter/resolver, and adds tests for waiting before removing the session listener.

## Best Possible Solution

Land a rebased fix that preserves synchronous unsubscribe or audits every caller to await a shared teardown promise, waits through the final current `compaction_end`, closes the missed-end window, and includes real timeout or abort proof.

## Reproduction Assessment

Yes, at source level: current main aborts in-flight compaction and immediately removes the session listener, while `compaction_end` can be delivered through that listener afterward. I did not run a live 48-hour timeout reproduction in this read-only review.

## Solution Assessment

No, not as written. Waiting before listener removal is the right direction, but this patch changes the unsubscribe API contract and can still stop listening before retry-final compaction completion.

Full review comments:

- **[P1] Preserve the unsubscribe completion contract:** `src/agents/pi-embedded-subscribe.ts:735`
  - body: Making `unsubscribe` async changes cleanup semantics, but production teardown still calls it without `await` inside a synchronous try/catch. Teardown can continue before `sessionUnsubscribe()` runs, and async failures bypass the existing cleanup error handling.
  - confidence: 0.92
- **[P1] Resolve cleanup only after the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The new resolver runs for `willRetry=true` as well as the final end event. When a retry is pending, unsubscribe can remove the listener before the final `compaction_end` notification arrives, preserving the event-loss race.
  - confidence: 0.91
- **[P2] Use current compaction event names in tests:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:19-31`
  - body: The added tests emit `auto_compaction_start` and `auto_compaction_end`, but current main handles `compaction_start` and `compaction_end`. Rebasing these tests as-is would not exercise the intended handler path.
  - confidence: 0.89
- **[P2] Register the cleanup waiter without a missed-end window:** `src/agents/pi-embedded-subscribe.ts:710-717`
  - body: The code checks compaction state before storing `compactionCleanupResolve`. If the end event fires in that gap, the handler cannot resolve the waiter and unsubscribe waits for the timeout even though cleanup already completed.
  - confidence: 0.82
- **[P3] Avoid the real five-second timeout in the test:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:55`
  - body: The timeout test waits for the real 5000 ms fallback on every run, which slows the focused suite and adds timing noise under CI load. Use fake timers to advance the cleanup timeout deterministically.
  - confidence: 0.78
