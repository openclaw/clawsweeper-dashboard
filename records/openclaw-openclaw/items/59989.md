## Summary

Keep open. The underlying teardown race is plausible and current main still unsubscribes synchronously after aborting compaction, but this PR is not merge-ready: it changes the subscription cleanup contract to async without updating the runner cleanup path, resolves the cleanup waiter on retry compaction-end events, and is stale against the current compaction event names.

## What This Changes

The PR makes `subscribeEmbeddedPiSession().unsubscribe` async, waits up to five seconds after `abortCompaction()` for compaction cleanup, stores a cleanup resolver in subscription state, and adds unsubscribe cleanup tests.

## Best Possible Solution

Preserve the frontend compaction-end delivery guarantee without changing the subscription cleanup contract accidentally: either keep `unsubscribe` synchronous through a state-machine approach or update every teardown caller to await a shared, idempotent cleanup promise, resolve only on the final compaction end, and rebase the tests onto `compaction_start`/`compaction_end`.

## Reproduction Assessment

Yes. The single-end case can be exercised with the added harness, while the blocking retry case is reproduced by emitting `compaction_end` with `willRetry: true` followed by a final end and observing that the PR would unsubscribe after the first event.

## Solution Assessment

No. Waiting for compaction cleanup is a reasonable direction, but this patch is not the best implementation because it changes `unsubscribe` to async without updating callers and resolves on retry events before compaction is actually finished.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Await the new async unsubscribe everywhere:** `src/agents/pi-embedded-subscribe.ts:735`
  - body: Changing `unsubscribe` to `async` changes the cleanup contract, but the runner still calls it in a synchronous `finally` without `await`. In timeout/abort paths, cleanup can proceed before `sessionUnsubscribe()` runs and any later rejection bypasses the existing `try/catch`; keep this API synchronous or update callers to await a shared idempotent cleanup promise.
  - confidence: 0.9
- **[P1] Resolve cleanup only after the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The cleanup resolver runs after both retry and final compaction-end events. When `willRetry` is true, another compaction completion is still pending, so this can remove the session listener before the final end event and drop the notification the PR is trying to preserve.
  - confidence: 0.88
- **[P2] Use current compaction event names in the added tests:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:19-31`
  - body: Current main dispatches `compaction_start` and `compaction_end`, but the new tests emit the old `auto_compaction_start` and `auto_compaction_end` names. After rebasing, these tests will not exercise the handler path they are meant to prove.
  - confidence: 0.86
- **[P3] Avoid sleeping five seconds in the timeout test:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:53-56`
  - body: The timeout test awaits the real 5000 ms cleanup timeout. That adds fixed latency and CI flake risk; use fake timers to advance the timeout deterministically.
  - confidence: 0.82
