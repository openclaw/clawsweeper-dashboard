## Summary

Keep open: current main still has the source-level listener-removal ordering this PR is trying to address, but the branch is not merge-ready because it changes the unsubscribe contract, can still unsubscribe before the final compaction end on retry, uses stale event names in new tests, and lacks real behavior proof.

## What This Changes

The PR makes embedded Pi session unsubscribe wait for compaction cleanup after aborting before removing the session listener, and adds cleanup tests.

## Best Possible Solution

Land a narrow current-main compaction-teardown fix that preserves final compaction-end delivery, keeps unsubscribe completion semantics consistent, updates tests to current event names, and includes deterministic regression coverage plus real setup proof.

## Reproduction Assessment

Yes, at source level. Current main aborts in-flight compaction and immediately removes the session listener, and the PR-introduced retry bug is visible because cleanup resolves outside the final !willRetry branch.

## Solution Assessment

No, not as written. Waiting before listener removal is the right direction, but this patch changes unsubscribe to async without updating the production caller contract and can still unsubscribe before the final compaction-end notification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the unsubscribe completion contract:** `src/agents/pi-embedded-subscribe.ts:699`
  - body: Making `unsubscribe` async changes the cleanup contract, but production cleanup still calls it synchronously inside a try/catch. Timeout teardown can continue before `sessionUnsubscribe()` runs, and async failures bypass that catch; either keep the API synchronous or update callers to await a shared cleanup promise.
  - confidence: 0.9
- **[P1] Resolve cleanup only on the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The added resolver runs for both retry and final compaction-end events. When `willRetry` is true, another completion is still pending, so this can remove the listener before the final notification this PR is trying to preserve.
  - confidence: 0.9
- **[P2] Update tests to current compaction event names:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:18-31`
  - body: Current main dispatches `compaction_start` and `compaction_end`, but the new tests emit the old `auto_compaction_start` and `auto_compaction_end` names. After rebase, these tests will not exercise the handler path they are meant to prove.
  - confidence: 0.88
- **[P3] Clear the cleanup timeout after success:** `src/agents/pi-embedded-subscribe.ts:686`
  - body: When the cleanup promise wins the race, the timeout created here is left pending until it fires. Clear the handle in `finally` so successful unsubscribe cleanup does not keep a needless timer alive for five seconds.
  - confidence: 0.78
- **[P3] Use fake timers for the timeout test:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:53-56`
  - body: This test waits for the real 5000 ms cleanup timeout, which adds fixed latency and avoidable CI flake risk. Advance the timeout with Vitest fake timers instead.
  - confidence: 0.82
