## Summary

Keep open: the underlying event-loss path is still plausible on current main, but this PR is not mergeable as written because it changes unsubscribe completion semantics, can still resolve before the final retry completion, uses stale compaction event names in tests, and lacks contributor real behavior proof.

## What This Changes

The PR changes embedded Pi subscription teardown to wait for compaction cleanup before removing the session listener and adds unsubscribe-during-compaction tests.

## Best Possible Solution

Land a rebased teardown fix that preserves or deliberately updates the unsubscribe completion contract, waits through the final `compaction_end`, uses current event names, and includes deterministic regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current main aborts in-flight compaction and immediately removes the session listener, while the dependency can still emit a later `compaction_end`. I did not establish a live failing setup in this read-only review.

## Solution Assessment

No, not as written. Waiting before listener removal is the right direction, but this patch changes a synchronous cleanup contract without updating callers and can still stop listening before the final retry completion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the unsubscribe completion contract:** `src/agents/pi-embedded-subscribe.ts:735`
  - body: Making `unsubscribe` async means existing finally cleanup that calls `unsubscribe()` synchronously will not wait for `sessionUnsubscribe()`, and async failures bypass the local catch. Keep this API synchronous or update callers to await a shared cleanup promise.
  - confidence: 0.9
- **[P1] Resolve cleanup only after the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The resolver runs for `willRetry=true` as well as the final end event. When a retry is pending, this can remove the listener before the final `compaction_end` notification that the PR is trying to preserve.
  - confidence: 0.9
- **[P2] Use current compaction event names in tests:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:19-31`
  - body: The new tests emit `auto_compaction_start` and `auto_compaction_end`, but current main and the upstream dependency use `compaction_start` and `compaction_end`, so the rebased tests would not exercise the intended handler path.
  - confidence: 0.88
- **[P2] Register the cleanup waiter without a missed-end window:** `src/agents/pi-embedded-subscribe.ts:710-717`
  - body: The state check happens before storing `compactionCleanupResolve`; if `compaction_end` fires in that gap, unsubscribe waits for the timeout even though cleanup already completed.
  - confidence: 0.82
- **[P3] Clear the cleanup timeout after success:** `src/agents/pi-embedded-subscribe.ts:720-721`
  - body: When the cleanup promise wins the race, the timeout remains scheduled for five seconds. Store the handle and clear it in `finally` so successful cleanup does not leave a needless timer alive.
  - confidence: 0.78
- **[P3] Use fake timers for timeout coverage:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:55-59`
  - body: The timeout test waits for the real five-second timeout, adding fixed latency and CI flake risk. Use Vitest fake timers to advance this path deterministically.
  - confidence: 0.82
