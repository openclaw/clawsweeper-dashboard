## Summary

Keep this PR open, but do not merge it as-is. Current main still removes the embedded Pi session listener immediately after aborting compaction, so the targeted race is not already fixed; however, the PR changes the unsubscribe contract to async without updating the runner caller, resolves cleanup on retry compaction-end events, and its added tests use stale event names after #67713.

## What This Changes

The PR makes `subscribeEmbeddedPiSession().unsubscribe` async, waits up to five seconds after `abortCompaction()` for compaction cleanup, stores a cleanup resolver in subscription state, and adds unsubscribe cleanup tests.

## Best Possible Solution

Land a narrow compaction-teardown fix that preserves frontend compaction-end delivery, updates every production teardown path consistently, resolves cleanup only on the final `compaction_end`, uses current event names in tests, and avoids real-time sleeps in the timeout coverage.

## Reproduction Assessment

Yes. The current-main race is visible from the teardown path that aborts compaction and immediately removes the session listener, and the PR-introduced retry bug is reproducible by emitting `compaction_end` with `willRetry: true` followed by a final end event.

## Solution Assessment

No. Waiting for cleanup is a reasonable direction, but this implementation is not the safest solution because it changes `unsubscribe` to async without updating callers and can still drop the final compaction-end event on retries.

- **[P1] Await the async unsubscribe cleanup everywhere:** `src/agents/pi-embedded-subscribe.ts:735`
  - body: Changing `unsubscribe` to `async` changes its completion contract, but the embedded runner still calls it in a synchronous `finally` block without `await`.   - confidence: 0.9

  - body: The added cleanup resolver runs after both retry and final compaction-end events. When `willRetry` is true, another compaction completion is still pending, so this can remove the session listener before the final end event and drop the notification this PR is trying to preserve.
  - confidence: 0.88

  - body: Current main dispatches `compaction_start` and `compaction_end`, but the new tests emit the old `auto_compaction_start` and `auto_compaction_end` names. After rebasing, those tests will not exercise the handler path they are meant to prove.
  - confidence: 0.86
