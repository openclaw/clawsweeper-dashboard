## Summary

Keep open: current main still has the listener-removal race this PR targets, but the branch is not merge-ready because it changes unsubscribe's completion contract, resolves cleanup on retry events, uses stale compaction event names in tests, and lacks after-fix real behavior proof.

## What This Changes

The branch makes embedded Pi session unsubscribe async, waits up to five seconds for compaction cleanup after aborting, stores a cleanup resolver in subscription state, and adds unsubscribe cleanup tests.

## Best Possible Solution

Land a narrow current-main compaction-teardown fix that preserves final compaction-end delivery, keeps the unsubscribe completion contract consistent, resolves cleanup only after the final compaction_end, and includes deterministic regression tests plus real setup proof.

## Reproduction Assessment

Yes, at source level. Current main aborts compaction and immediately removes the session listener, and the PR-introduced retry bug is evident from resolving cleanup when willRetry is true before the final compaction_end.

## Solution Assessment

No. Waiting before listener removal is the right direction, but this patch changes unsubscribe to async without updating the production caller contract and can still unsubscribe before the final compaction-end notification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await async unsubscribe cleanup everywhere:** `src/agents/pi-embedded-subscribe.ts:735`
  - body: Changing unsubscribe to async changes its completion contract, but production teardown still calls it synchronously inside a try/catch. Timeout cleanup can continue before sessionUnsubscribe runs, and any later rejection bypasses that catch; either preserve a synchronous API or update callers to await a shared idempotent cleanup promise.
  - confidence: 0.9
- **[P1] Resolve cleanup only after the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The added cleanup resolver runs after both retry and final compaction-end events. When willRetry is true, another completion is still pending, so this can remove the listener before the final notification this PR is trying to preserve.
  - confidence: 0.88
- **[P2] Use current compaction event names in the new tests:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:18-31`
  - body: Current main dispatches compaction_start and compaction_end, but the new tests emit the old auto_compaction_start and auto_compaction_end names. After rebase, these tests will not exercise the handler path they are meant to prove.
  - confidence: 0.86
- **[P3] Clear the cleanup timeout on success:** `src/agents/pi-embedded-subscribe.ts:720-722`
  - body: When the cleanup promise wins, the timeout created for the race is left pending until it fires. Clear the handle in finally so successful unsubscribe cleanup does not keep a needless timer alive for five seconds.
  - confidence: 0.78
- **[P3] Avoid sleeping five seconds in the timeout test:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:53-56`
  - body: The timeout test awaits the real 5000 ms cleanup timeout, adding fixed latency and CI flake risk. Use fake timers to advance the timeout deterministically.
  - confidence: 0.82
