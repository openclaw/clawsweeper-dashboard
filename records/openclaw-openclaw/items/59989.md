## Summary

Keep this PR open: the underlying compaction teardown race is still source-reproducible on current main, but this branch is not mergeable as written because it changes unsubscribe completion semantics without updating callers, can resolve cleanup on retry events, is stale after the compaction event rename, and lacks required real behavior proof.

## What This Changes

The PR makes embedded Pi unsubscribe async, waits up to 5 seconds after aborting compaction before removing the session listener, stores a compaction cleanup resolver, and adds unsubscribe cleanup tests.

## Best Possible Solution

Land a rebased, narrow teardown fix that preserves or explicitly updates the unsubscribe completion contract, waits through the final `compaction_end`, uses current event names, and includes deterministic regression coverage plus redacted real behavior proof.

## Reproduction Assessment

Yes, source-reproducible. Current main aborts in-flight compaction and immediately removes the session listener, so a later dependency-defined `compaction_end` event can be emitted after OpenClaw has stopped listening.

## Solution Assessment

No, not as written. Waiting before listener removal is the right direction, but this patch changes a synchronous cleanup contract without updating production callers and can still remove the listener before the final retry completion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the unsubscribe completion contract:** `src/agents/pi-embedded-subscribe.ts:735`
  - body: Making `unsubscribe` async changes cleanup semantics, but production cleanup still calls it synchronously inside a `try`/`catch`. Timeout teardown can continue before `sessionUnsubscribe()` runs, and async failures bypass that catch; keep the API synchronous or update callers to await a shared cleanup promise.
  - confidence: 0.9
- **[P1] Resolve cleanup only on the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The added resolver runs for retry and final compaction-end events. When `willRetry` is true, another completion is still pending, so this can remove the listener before the final notification this PR is trying to preserve.
  - confidence: 0.9
- **[P2] Update tests to current compaction event names:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:19-31`
  - body: Current main and the upstream dependency use `compaction_start` and `compaction_end`, but the new tests emit the old `auto_compaction_*` names. After rebase, these tests will not exercise the handler path they are meant to prove.
  - confidence: 0.88
- **[P2] Register the cleanup waiter before the state check:** `src/agents/pi-embedded-subscribe.ts:710-717`
  - body: There is a race between checking compaction state and storing `compactionCleanupResolve`: if the end event fires in that window, unsubscribe waits for the full timeout even though cleanup already completed. Register the waiter or recheck state in a way that cannot miss the event.
  - confidence: 0.82
- **[P3] Clear the cleanup timeout after success:** `src/agents/pi-embedded-subscribe.ts:720-721`
  - body: When the cleanup promise wins the race, the timeout is left pending until it fires. Keep the timeout handle and clear it in `finally` so successful cleanup does not leave a needless timer alive for five seconds.
  - confidence: 0.78
- **[P3] Use fake timers for timeout coverage:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:55-59`
  - body: The timeout test waits for a real five-second timeout, adding fixed latency and CI flake risk. Use Vitest fake timers to advance this path deterministically.
  - confidence: 0.82
