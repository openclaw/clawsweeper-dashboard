## Summary

Keep this PR open, but it still needs changes: current main still has the underlying teardown race, while the PR introduces async unsubscribe contract issues, resolves cleanup on retry events, and is stale against the merged compaction event rename.

## What This Changes

The PR changes embedded Pi session unsubscribe to wait for compaction cleanup before removing the session listener and adds tests for that teardown path.

## Best Possible Solution

Land a narrow compaction-teardown fix that preserves final compaction-end delivery, keeps unsubscribe cleanup semantics consistent for all callers, uses current compaction_start/compaction_end events, and covers timeout behavior with deterministic tests.

## Reproduction Assessment

Yes, at source level. Current main aborts compaction and immediately removes the listener, and the PR-introduced retry bug follows from resolving cleanup on a willRetry=true compaction_end before the final event.

## Solution Assessment

No. Waiting for compaction cleanup is the right direction, but this patch is not the best solution until it preserves the unsubscribe contract or updates callers, resolves only on the final compaction_end, and rebases onto the current event names.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await the async unsubscribe cleanup everywhere:** `src/agents/pi-embedded-subscribe.ts:699`
  - body: Changing unsubscribe to async changes its completion contract, but production teardown still calls it synchronously. Timeout cleanup can continue before sessionUnsubscribe runs, and any rejection from the async path bypasses the existing try/catch; either preserve a synchronous API or make callers await a shared idempotent cleanup promise.
  - confidence: 0.9
- **[P1] Resolve cleanup only after the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The cleanup resolver runs after both retry and final compaction-end events. When willRetry is true, another completion is still pending, so this can remove the listener before the final notification this PR is trying to preserve.
  - confidence: 0.89
- **[P2] Use current compaction event names in the new tests:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:19-31`
  - body: Current main dispatches compaction_start and compaction_end, but the new tests emit the old auto_compaction_start and auto_compaction_end names. After rebase, these tests will not exercise the handler path they are meant to prove.
  - confidence: 0.88
- **[P3] Clear the cleanup timeout handle on success:** `src/agents/pi-embedded-subscribe.ts:684-686`
  - body: When the cleanup promise wins, the timeout created for the race is left pending until it fires. Clear the handle in finally so successful unsubscribe cleanup does not keep a needless timer alive for five seconds.
  - confidence: 0.78
- **[P3] Avoid sleeping five seconds in the timeout test:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:53-56`
  - body: The timeout test awaits the real 5000 ms cleanup timeout, adding fixed latency and CI flake risk. Use fake timers to advance the timeout deterministically.
  - confidence: 0.82
