## Summary

Keep open: the PR addresses a plausible compaction teardown race, but it is not merge-ready because it changes unsubscribe completion semantics, can still miss the final retry compaction_end, uses stale event names in tests, conflicts with main, and lacks real behavior proof.

## What This Changes

The PR makes embedded Pi session unsubscribe async, waits up to five seconds for compaction cleanup after abort, adds a cleanup resolver, and adds unsubscribe-during-compaction tests.

## Best Possible Solution

Land a rebased fix that preserves or explicitly audits the unsubscribe contract, waits only through the final compaction_end without a missed-end window, updates tests to current event names, and includes real timeout/abort proof.

## Reproduction Assessment

Yes, at source level: current main aborts in-flight compaction and immediately removes the session listener, while compaction_end can be a later event. I did not establish a live long-running timeout reproduction in this read-only review.

## Solution Assessment

No, not as written. Waiting before listener removal is the right direction, but this patch changes the unsubscribe API contract and can still stop listening before retry-final compaction completion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the unsubscribe completion contract:** `src/agents/pi-embedded-subscribe.ts:735`
  - body: Making unsubscribe async changes the cleanup API, but production finally cleanup still calls it without await inside a synchronous try/catch. Teardown can continue before sessionUnsubscribe() runs, and async failures bypass the existing cleanup error handling.
  - confidence: 0.9
- **[P1] Resolve cleanup only after the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The cleanup resolver runs for willRetry=true as well as the final end event. When a retry is pending, unsubscribe can remove the listener before the final compaction_end notification arrives, preserving the event-loss race.
  - confidence: 0.9
- **[P2] Use current compaction event names in tests:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:19-31`
  - body: The added tests emit auto_compaction_start and auto_compaction_end, but current main handles compaction_start and compaction_end. Rebasing these tests as-is would not exercise the intended handler path.
  - confidence: 0.88
- **[P2] Register the cleanup waiter without a missed-end window:** `src/agents/pi-embedded-subscribe.ts:710-717`
  - body: The code checks compaction state before storing compactionCleanupResolve. If compaction_end fires in that gap, the handler cannot resolve the waiter and unsubscribe waits for the timeout even though cleanup already completed.
  - confidence: 0.82
