## Summary

This PR still targets a real source-level teardown race, but it is not safe to merge as written: current main still removes the listener immediately after aborting compaction, while the PR changes a synchronous cleanup contract, resolves the wait before retry-final compaction completion, uses stale event names in new tests, and lacks contributor-supplied real behavior proof.

## What This Changes

The PR makes embedded Pi subscription unsubscribe wait up to five seconds for compaction cleanup after abort, adds a cleanup resolver to subscription state, and adds unsubscribe-during-compaction tests.

## Best Possible Solution

Land a rebased teardown fix that preserves the cleanup contract or updates every caller to await a shared cleanup promise, waits through the final compaction_end, uses current event names, and includes deterministic regression tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current main aborts in-flight compaction and immediately removes the session listener, while the dependency can still emit compaction_end afterward. I did not establish a live long-running timeout reproduction in this read-only review.

## Solution Assessment

No, not as written. Waiting before listener removal is the right direction, but this patch changes the unsubscribe completion contract and can still stop listening before retry-final compaction completion.

Full review comments:

- **[P1] Preserve the unsubscribe completion contract:** `src/agents/pi-embedded-subscribe.ts:735`
  - body: Making unsubscribe async changes the returned cleanup API, but production finally cleanup still calls it without await inside a synchronous try/catch. That lets teardown continue before sessionUnsubscribe() runs and lets async failures bypass the existing cleanup error handling.
  - confidence: 0.9
- **[P1] Resolve cleanup only after the final compaction end:** `src/agents/pi-embedded-subscribe.handlers.compaction.ts:75-79`
  - body: The cleanup resolver runs for willRetry=true as well as the final end event. When a retry is pending, unsubscribe can remove the listener before the final compaction_end notification arrives, preserving the same event-loss race.
  - confidence: 0.9
- **[P2] Use current compaction event names in tests:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:19-31`
  - body: The added tests emit auto_compaction_start and auto_compaction_end, but current main and @earendil-works/pi-coding-agent use compaction_start and compaction_end. Rebasing these tests as-is would not exercise the intended handler path.
  - confidence: 0.88
- **[P2] Register the cleanup waiter without a missed-end window:** `src/agents/pi-embedded-subscribe.ts:710-717`
  - body: The state check happens before compactionCleanupResolve is stored. If compaction_end fires in that gap, the handler cannot resolve the waiter and unsubscribe sits until the five-second timeout even though cleanup already completed.
  - confidence: 0.82
- **[P3] Clear the cleanup timeout after success:** `src/agents/pi-embedded-subscribe.ts:720-721`
  - body: When the cleanup promise wins the race, the timeout remains scheduled for five seconds. Store the handle and clear it in finally so successful cleanup does not leave a needless timer alive.
  - confidence: 0.78
- **[P3] Use fake timers for timeout coverage:** `src/agents/pi-embedded-subscribe.unsubscribe-compaction-cleanup.test.ts:55-59`
  - body: The timeout test waits for the real five-second timeout on every run, adding fixed latency and CI flake risk. Use Vitest fake timers to advance this path deterministically.
  - confidence: 0.82
