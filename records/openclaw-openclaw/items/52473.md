## Summary

Keep open. Current main still resolves explicit skill slash commands, but ordinary SKILL.md-backed skills are rewritten into a normal model prompt instead of loading and injecting the selected SKILL.md content. The prior ClawSweeper comment remains consistent with the current code, docs, and tests: only `command-dispatch: tool` is deterministic today.

## What This Changes

The issue requests deterministic SKILL.md loading and context injection when a user explicitly invokes a skill through `/skill <name>` or a direct skill slash command.

## Best Possible Solution

Implement an explicit-invocation-only activation path in core: after `/skill <name>` or a direct skill command resolves against the visible eligible skill set and authorization passes, load that selected normal skill through the existing skills subsystem and inject its SKILL.md instructions into the agent run context before the user's args. Leave natural-language skill matching model-mediated, preserve deterministic tool dispatch, and update regression tests plus user-facing docs.

## Reproduction Assessment

Yes. A focused current-main check can pass a normal `SkillCommandSpec` without `dispatch` into `handleInlineActions` and observe `ctx.Body` being rewritten to `Use the "<skill>" skill for this request.` with no SKILL.md content injected.

## Solution Assessment

Yes. The requested explicit slash-command-only injection is the narrow maintainable boundary; the safer implementation is to reuse the existing visible/eligible skills snapshot and an existing model-context message mechanism rather than broadening natural-language autoloading.

## Repair Work Prompt

Implement deterministic SKILL.md loading for explicit skill slash commands. Observable bug: when a user invokes a normal user-invocable skill via `/skill <name> [args]` or a direct skill command, OpenClaw currently rewrites the request to `Use the "<skill>" skill for this request.` and relies on the model to voluntarily read SKILL.md. Expected behavior: for explicit skill invocation only, after command resolution plus authorization and visibility gates pass, load the selected normal SKILL.md-backed skill through the existing skills subsystem and inject its full instructions into the agent run context before the user args. Do not change natural-language skill matching. Keep core extension-agnostic and avoid bundled skill IDs. Add focused regression tests for direct skill aliases, `/skill <name>`, skills hidden from available_skills via `disable-model-invocation`, tool-dispatch unaffected behavior, transcript/provider-compatible context shape, and docs updates.
