## Summary

Keep open: current main and v2026.6.8 still handle explicit channel skill slash commands by rewriting non-tool skills into advisory model text, not by deterministically injecting SKILL.md content. A new CLI-style `/skill:name` expansion path exists, but it does not cover the documented `/skill <name>` or direct native/channel skill-command path, and the remaining fix needs a maintainer product/security decision.

## What This Changes

Requested deterministic loading of SKILL.md content when a user explicitly invokes a non-tool skill through a slash command.

## Best Possible Solution

Define a maintainer-approved explicit skill invocation and trust-boundary contract, then implement one core path that deterministically loads SKILL.md for non-tool slash-invoked skills across channel and session entry points with focused security, prompt, command, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: current main recognizes explicit channel skill slash commands, but ordinary non-tool skills are rewritten into advisory model text instead of having SKILL.md injected. I did not run a live model transcript reproduction because this review is read-only.

## Solution Assessment

Unclear as a final implementation path: deterministic injection is a plausible fix for explicit user intent, but the best solution must first settle the instruction isolation, security precedence, and shared channel/session contract.
