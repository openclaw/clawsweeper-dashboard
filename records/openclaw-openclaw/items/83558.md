## Label Justifications

- `P1`: The reported regression blocks Codex provider authentication for affected remote/headless installs on the latest shipped release.
- `impact:auth-provider`: The issue concerns OpenAI Codex provider auth method selection and login routing.

## Summary

Current main restores a documented headless Codex auth path by mapping `models auth login --provider openai-codex --device-code` to the provider device-code method, while keeping the provider-owned device-code registration and tests in place. This is current-main-only evidence; it is not present in the latest shipped release tag.

## What This Changes

Issue reports that OpenAI Codex device-code authentication is no longer offered for remote/headless installs after the 2026.5.12 externalized plugin release.

## Best Possible Solution

Keep the current explicit `--device-code` headless auth path and ensure the next release carries it so v2026.5.12 users no longer need a workaround.

## Reproduction Assessment

Yes for the shipped-release symptom: source comparison shows v2026.5.12 lacks the current `--device-code` CLI flag and the auth picker defaults to OAuth when no method is requested. No current-main failing path remains for headless login because main maps `--device-code` to the provider device-code method.

## Solution Assessment

Yes for the central headless-auth problem: current main makes device-code selection explicit and documented rather than relying on the old chooser behavior. Reintroducing a prompt would be a separate UX choice, not needed to restore the reachable auth path.
