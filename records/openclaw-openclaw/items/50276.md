## Summary

Keep open: the requested Slack threading behavior is still not on current main, but this PR head is not mergeable because the prior correctness findings still apply and real Slack after-fix proof is missing.

## What This Changes

The PR adds Slack-only message-action handling that maps parsed `[[reply_to_current]]` directives to `threadId` through Slack auto-thread resolution and adds same-channel and cross-channel threading tests.

## Best Possible Solution

Implement explicit current-thread resolution through the Slack or channel threading contract, align off-mode docs/metadata/tests with the intended policy, refresh the branch, and prove actual Slack threaded delivery.

## Reproduction Assessment

Yes for source-level reproduction: current main parses and strips `[[reply_to_current]]` but never maps `parsed.replyToCurrent` to `threadId` before Slack dispatch. I did not run live Slack proof in this read-only review.

## Solution Assessment

No. The current PR is not the best mergeable solution because it redeclares an imported helper, adds tests with undefined helpers, and hard-codes a Slack off-mode override in shared core instead of resolving the policy through the Slack/channel threading contract.
