## Summary

Keep open: this is a focused Slack threading PR, but the current head still has merge-blocking source/test defects, lacks required real Slack behavior proof, and needs maintainer policy judgment because it overrides documented Slack `replyToMode="off"` semantics.

## What This Changes

The PR adds Slack-only handling that maps `[[reply_to_current]]` in message-action sends to `threadId` through the Slack auto-thread resolver and adds same-channel and cross-channel threading tests.

## Best Possible Solution

Route explicit current-thread resolution through the Slack or channel threading contract, align docs/tests with the intended off-mode policy, then refresh the branch and prove a real Slack threaded reply.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main strips `[[reply_to_current]]` in message-action sends and never maps `parsed.replyToCurrent` to a Slack `threadId` before dispatch; I did not run live Slack proof in this read-only review.

## Solution Assessment

No. The current PR is not mergeable as-is because it redeclares an imported helper, adds tests with undefined helpers, and encodes a Slack off-mode override in shared core rather than in a clear Slack/channel threading policy seam.
