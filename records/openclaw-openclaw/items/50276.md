## Summary

Keep open: current main still does not implement the Slack `[[reply_to_current]]` mapping and this PR is the focused open implementation thread, but the current PR head is not mergeable as-is because it introduces source/test blockers, conflicts with Slack off-mode threading policy, and lacks after-fix real Slack proof.

## What This Changes

This PR maps Slack `[[reply_to_current]]` in outbound message-action sends to the active Slack `thread_ts` and adds same-channel and cross-channel threading tests.

## Best Possible Solution

Move explicit current-thread routing through the Slack or channel threading contract, keep docs and off-mode semantics aligned, fix the compile/test blockers, and require redacted Slack workspace proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main parses and strips `[[reply_to_current]]` but never maps `parsed.replyToCurrent` to Slack `threadId` before dispatch. I did not run live Slack proof in this read-only review.

## Solution Assessment

No, the PR is not currently the best mergeable solution because it hardcodes Slack off-mode override policy in shared core and introduces compile/test blockers. A safer fix should go through the Slack/channel threading seam and intentionally update policy/docs if off-mode behavior changes.
