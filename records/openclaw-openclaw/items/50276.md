## Summary

Keep open, but do not merge as-is: current main still lacks the requested Slack `[[reply_to_current]]` mapping, while this PR head is conflicting and has concrete correctness blockers plus a missing real Slack proof gate.

## What This Changes

The PR adds Slack-only message-action handling that maps `[[reply_to_current]]` to the active Slack thread ID and adds same-channel/cross-channel threading tests.

## Best Possible Solution

Resolve explicit current-thread replies through the Slack/channel threading contract, update Slack off-mode policy/docs/tests only if maintainers choose that behavior, and require redacted real Slack proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main strips `[[reply_to_current]]` and only maps explicit `[[reply_to:<id>]]` into `replyTo`, while Slack dispatch would honor a resolved `threadId` or `replyTo`. I did not establish a live Slack workspace reproduction in this read-only review.

## Solution Assessment

No. This PR is not the best mergeable solution as written because it redeclares an imported helper, adds tests that cannot run, and forces Slack `replyToMode` policy from shared outbound code.
