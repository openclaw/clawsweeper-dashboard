## Summary

Keep open: current main still does not map Slack message-tool `[[reply_to_current]]` to a thread target, but this PR is not mergeable as written because it has concrete compile/test blockers, lacks real Slack proof, and changes Slack off-mode policy from shared outbound code.

## What This Changes

The PR adds Slack-only outbound message-action handling to resolve `[[reply_to_current]]` into a Slack thread ID, plus same-channel and cross-channel threading tests.

## Best Possible Solution

Resolve explicit current-thread replies through the Slack/channel threading contract, update Slack off-mode docs and tests only if maintainers choose that behavior, then refresh the branch and verify with redacted real Slack proof.

## Reproduction Assessment

Yes for source-level reproduction: current main strips `[[reply_to_current]]` and only maps explicit `[[reply_to:<id>]]` into `replyTo`, while Slack dispatch would honor a resolved `threadId` or `replyTo`. I did not establish a live Slack workspace reproduction in this read-only review.

## Solution Assessment

No. The PR is not the best mergeable solution as written because it redeclares an imported helper, adds tests that cannot run, and forces Slack off-mode policy from shared outbound code instead of the Slack/channel threading contract.
