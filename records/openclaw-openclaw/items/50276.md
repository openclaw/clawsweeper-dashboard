## Summary

Keep open: the requested Slack threading behavior is still not implemented on current main, but this branch cannot merge as-is because it has blocking TypeScript/test defects, lacks real Slack behavior proof, and changes the documented Slack `replyToMode="off"` contract.

## What This Changes

The PR adds Slack-specific `[[reply_to_current]]` handling in the message-action runner and threading tests for same-channel and cross-channel sends.

## Best Possible Solution

Implement the directive through the generic channel threading adapter contract, decide Slack off-mode semantics explicitly, keep focused runner/Slack coverage, and require redacted real Slack proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows `[[reply_to_current]]` is stripped but not mapped to `threadId`, and Slack auto-thread resolution returns no thread in `replyToMode="off"`; I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The behavior should be routed through the channel/plugin threading seam and reconciled with Slack off-mode policy instead of hardcoding Slack and forcing `replyToMode: "all"` in shared core.
