## Summary

Keep open: the underlying Slack threading request is still source-reproducible on current main, but this PR is not mergeable as-is because it introduces a duplicate ESM binding, broken tests, a top-level opt-out regression, missing real Slack behavior proof, and an unresolved Slack off-mode policy/docs question.

## What This Changes

This PR adds Slack-only `[[reply_to_current]]` handling in outbound message-action sends and adds same-channel and cross-channel threading tests.

## Best Possible Solution

Decide the Slack off-mode explicit-tag policy, then implement the accepted behavior through the existing outbound threading helper path with opt-outs preserved, focused tests, docs alignment, and redacted real Slack proof.

## Reproduction Assessment

Yes, at source level: current main strips `[[reply_to_current]]` in `handleSendAction` but never maps `parsed.replyToCurrent` into `threadId` or `replyTo`, while Slack dispatch only threads when those fields are set. I did not run a live Slack repro in this read-only review.

## Solution Assessment

No, not as patched. The feature direction may be reasonable, but the branch needs to avoid the duplicate helper, define the new test harness, preserve explicit top-level opt-outs, align the Slack off-mode policy, and include real Slack proof.
