## Summary

Keep open: current main still does not implement Slack `[[reply_to_current]]` handling, but this PR is not merge-ready because the patch introduces blocking module/test defects, bypasses existing threading opt-out semantics, and lacks real Slack behavior proof.

## What This Changes

The PR adds Slack-only `[[reply_to_current]]` resolution to `threadId` in outbound send actions and adds same-channel/cross-channel threading tests.

## Best Possible Solution

Decide whether Slack explicit reply tags should bypass `replyToMode="off"`, then implement that policy through the existing outbound threading helper with top-level, `threadId: null`, and first-mode semantics preserved, matching docs/tests, and redacted real Slack proof.

## Reproduction Assessment

Yes, at source level: current main strips `[[reply_to_current]]` via `parseReplyDirectives` but does not map `parsed.replyToCurrent` into Slack `threadId` or `replyTo`. I did not establish a live Slack reproduction in this read-only review.

## Solution Assessment

No, not as patched. The feature direction may be useful, but this implementation duplicates an imported helper, leaves undefined test helpers, bypasses top-level/first-mode threading policy, and needs a clear Slack off-mode docs decision.
