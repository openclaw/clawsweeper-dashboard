## Summary

Keep open: current main still does not map Slack `[[reply_to_current]]` from message-action sends into a Slack thread target, but this PR is not mergeable as-is because the branch has blocking source/test defects, conflicts, missing real Slack proof, and now crosses a documented Slack off-mode policy point.

## What This Changes

The PR adds Slack-only `[[reply_to_current]]` handling in outbound message-action sends and adds same-channel and cross-channel threading tests.

## Best Possible Solution

Decide the Slack off-mode explicit-tag policy, then implement the accepted behavior through the existing outbound threading helper path with top-level opt-outs preserved, focused tests, matching docs, and redacted real Slack proof.

## Reproduction Assessment

Yes, at source level: current main strips `[[reply_to_current]]` in `handleSendAction` but never maps `parsed.replyToCurrent` into `threadId` or `replyTo`, while Slack dispatch only threads when those fields are set. I did not run a live Slack repro in this read-only review.

## Solution Assessment

No, not as patched. The feature direction may be reasonable, but the branch needs to reuse or extend the existing threading helper, preserve explicit top-level opt-outs, define its test harness, align Slack docs/policy, and include real Slack proof.
