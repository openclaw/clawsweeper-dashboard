## Summary

Keep open for maintainer review: the report is source-reproducible on current main because browser CLI management commands all route through admin-scoped `browser.request`, so a paired device without `operator.admin` can hit the reported scope-upgrade wall. The right fix overlaps security/product policy already tracked in related scope-upgrade and headless-token discussions.

## What This Changes

The issue asks for browser CLI and browser tool usage on a headless Hostinger install to work instead of failing on a pending operator scope upgrade.

## Best Possible Solution

Preserve explicit approval for privileged browser control, but make the browser CLI failure actionable and decide whether read-only browser status/doctor should use a narrower scope or whether headless installs should rely on a bounded token-scope trust surface.

## Reproduction Assessment

Yes, at source level. A paired device without `operator.admin` running `openclaw browser status`, `doctor`, or `start` will call admin-scoped `browser.request`, which matches the reported scope-upgrade failure path; I did not run a live Hostinger reproduction.

## Solution Assessment

Unclear. Lowering scope for read-only browser paths could help status and doctor, but browser start/tool control is privileged and the headless bypass/config direction needs the same security decision as the related token-scope issue.
