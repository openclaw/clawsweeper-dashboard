## Summary

Keep open. Current main still appears to let browser CLI calls rely on inferred gateway scopes while the gateway requires `operator.admin` for `browser.request`, which can explain the reported scope-upgrade loop for paired CLI devices.

## What This Changes

The request is to make browser CLI commands and agent browser usage work when `browser.request` currently triggers a gateway scope-upgrade approval loop.

## Best Possible Solution

Have browser CLI requests ask for the existing `operator.admin` browser scope explicitly, with regression coverage, while preserving the current `browser.request` admin authorization boundary.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a credible current-main path: browser CLI calls `browser.request` without explicit scopes while standalone plugin CLI loading may not activate the method-scope registry used for least-privilege inference.

## Solution Assessment

Yes for a narrow repair. The least risky fix is to make browser CLI RPCs request `operator.admin` explicitly, not to broaden global defaults or weaken the admin requirement for `browser.request`.
