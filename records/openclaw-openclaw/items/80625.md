## Summary

Keep open. Current main still has a plausible Anthropic Claude 4.6 cache-miss failure path: replay policy preserves latest signed/redacted thinking blocks, the existing exact-error retry helper is only defined and tested, and production request wrapping is gated off when `dropThinkingBlocks` is false.

## What This Changes

The issue requests automatic recovery for Anthropic `thinking`/`redacted_thinking` sessions that become stuck after prompt cache invalidation.

## Best Possible Solution

Wire a bounded Anthropic thinking-block recovery retry into the production Claude replay path, preserving normal signed-thinking replay and stripping thinking blocks only once after the exact cache-miss rejection.

## Reproduction Assessment

No. The issue has a concrete real provider log and source inspection shows the current Claude 4.6 path can replay preserved thinking blocks without a production retry, but I did not establish a live Anthropic cache-miss reproduction.

## Solution Assessment

Yes. A one-shot retry on the exact Anthropic thinking-block modification error is narrower than stripping all thinking at persistence time or changing the general Claude 4.6 replay policy.
