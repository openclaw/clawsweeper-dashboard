## Summary

Keep this PR open, but it should not merge as-is: the patch broadens an exact model allowlist into a different provider ID without declared alias metadata, and the PR still lacks real runtime proof for the claimed hyphen-stripping path.

## What This Changes

The PR adds hyphen-stripped provider/model allowlist variants so a configured hyphenated provider entry also authorizes the same model under a provider ID with hyphens removed.

## Best Possible Solution

Keep provider IDs exact by default; if a hyphen-stripped spelling is valid, resolve it through declared provider/plugin alias metadata or fix the runtime path that loses the configured provider ID, then prove that flow in a real setup.

## Reproduction Assessment

Unclear for the reported bug because the PR does not identify the current runtime path that strips provider hyphens. The PR-introduced authorization broadening is source-reproducible from the diff adding stripped variants to both allowlist key sets.

## Solution Assessment

No. Unconditional hyphen folding is broader than the stated mismatch and should be replaced by declared alias resolution or by preserving the original provider ID through the runtime path.
