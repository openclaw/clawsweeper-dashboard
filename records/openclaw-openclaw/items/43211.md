## Summary

Keep open, but do not merge as written: the patch addresses a plausible allowlist mismatch, but its unconditional hyphen stripping broadens authorization to a distinct provider ID and the PR lacks after-fix real behavior proof.

## What This Changes

The PR adds a helper and unit tests so a hyphenated configured provider/model key also allows the provider ID with hyphens removed.

## Best Possible Solution

Keep exact allowlist semantics unless a specific provider alias is proven; if aliasing is needed, implement it through an explicit provider normalization or alias contract with regression coverage for coexisting hyphenated and unhyphenated provider IDs.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection shows the patch would authorize a hyphen-stripped provider key while current provider normalization keeps those provider IDs distinct. The original user-facing mismatch is not proven with real runtime output.

## Solution Assessment

No. Unconditional hyphen stripping is not the best fix because it widens the model allowlist across distinct provider IDs; a safer solution should use explicit provider aliases or provider-owned normalization only when the IDs are known to be equivalent.
