## Summary

Keep open: this PR is a plausible attempt at the reported allowlist mismatch, but it should not merge as-is because it broadens model authorization from an explicitly allowlisted hyphenated provider ID to a distinct hyphen-stripped provider ID. Current main keeps provider IDs exact except for explicit compatibility aliases, docs describe `agents.defaults.models` as the exact allowlist, and prior review comments already identified the same authorization regression.

## What This Changes

The PR adds hyphen-stripped provider/model key variants to model allowlist construction and unit tests covering `buildConfiguredAllowlistKeys` and `buildAllowedModelSet`.

## Best Possible Solution

Keep provider IDs exact by default, and model intentional equivalence through explicit provider/plugin alias metadata or by fixing the resolver that loses the configured provider ID. Regression coverage should prove declared aliases work while `my-provider` and `myprovider` remain separately allowlisted unless explicitly linked.

## Reproduction Assessment

Unclear for the original bug: the PR describes a runtime path that strips provider hyphens, but the inspected current-main code and provided discussion do not identify that path. Yes for the PR-introduced regression: adding the stripped key makes `myprovider/some-model` pass `allowedKeys.has(key)` when only `my-provider/some-model` was configured.

## Solution Assessment

No. The proposed unconditional folding is broader than the reported mismatch and changes an authorization boundary; explicit alias metadata or preserving the provider ID at the resolver boundary is the narrower maintainable fix.
