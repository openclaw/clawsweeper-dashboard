## Summary

Keep open: the PR targets a plausible allowlist problem, but the current patch broadens authorization across distinct provider IDs without a declared alias contract, lacks a proven runtime repro, and needs contributor proof plus human provider/security judgment.

## What This Changes

The PR adds hyphen-stripped provider/model variants to configured model allowlist sets and adds unit tests for those variants.

## Best Possible Solution

Keep provider IDs exact by default; fix any proven mismatch by preserving the configured provider ID or resolving only declared provider/plugin aliases, then prove that flow in a real setup.

## Reproduction Assessment

Unclear for the reported bug: the PR does not identify a current path that strips provider hyphens. The PR-introduced authorization broadening is source-reproducible from the diff adding stripped variants to both allowlist builders.

## Solution Assessment

No. Unconditional hyphen stripping is broader than the stated mismatch; declared alias resolution or preserving the original provider ID is the narrower maintainable direction.
