## Best Possible Solution

Keep provider IDs exact by default; if a hyphen-stripped spelling is valid, resolve it through declared provider/plugin alias metadata or preserve the original provider ID through the runtime path, then prove that flow in a real setup.

Unclear for the reported bug because the PR does not identify the current runtime path that strips provider hyphens. The PR-introduced authorization broadening is source-reproducible from the diff adding stripped variants to both allowlist key sets.

No. Unconditional hyphen folding is broader than the stated mismatch and should be replaced by declared alias resolution or by preserving the configured provider ID.
