## Summary

Keep open for human review: the PR targets a real model allowlist surface, but the reported hyphen-stripping path is not proven on current main and the patch broadens exact provider authorization in a security-sensitive way.

## What This Changes

The PR adds hyphen-stripped provider/model variants to the model allowlist builders and adds unit tests for that behavior.

## Best Possible Solution

Keep provider IDs exact by default; if a real stripped-provider path exists, fix that runtime path to preserve the configured ID or honor only declared provider/plugin aliases.

## Reproduction Assessment

No high-confidence way to reproduce the reported current-main permission failure exists from the PR evidence; current source preserves arbitrary hyphens in provider IDs. The PR-introduced authorization broadening is source-reproducible from the diff and allowlist call path.

## Solution Assessment

No. Unconditional hyphen stripping is broader than the claimed mismatch; explicit provider aliases or preserving the original provider ID is the narrower maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restrict stripped provider aliases to declared aliases:** `src/agents/model-selection.ts:232`
  - body: This adds `myprovider/model` for every configured `my-provider/model`, turning an exact allowlist entry into permission for a distinct provider ID.
