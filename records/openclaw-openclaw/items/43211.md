## Summary

Keep open: the PR targets a plausible allowlist mismatch, but its unconditional hyphen-stripping changes a provider authorization boundary while current main treats provider IDs as exact except for explicit aliases.

## What This Changes

The PR adds hyphen-stripped provider/model key variants to model allowlist construction and adds unit tests for configured allowlist and allowed model set behavior.

## Best Possible Solution

Keep provider IDs exact by default; if an alias is needed, resolve it through declared provider/plugin alias metadata or preserve the configured provider ID at the resolver that strips it.

## Reproduction Assessment

Unclear for the original reported bug because the discussion does not identify the current runtime path that strips provider hyphens. Yes for the PR-introduced regression by source inspection: the diff makes `myprovider/some-model` pass the allowlist when only `my-provider/some-model` was configured.

## Solution Assessment

No. Unconditional hyphen folding is broader than the reported mismatch and changes an authorization boundary; declared aliases or fixing the resolver that loses the provider ID is narrower and maintainable.

## Review Findings
