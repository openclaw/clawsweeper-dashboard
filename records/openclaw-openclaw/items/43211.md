## Summary

Keep this PR open, but do not merge it as-is. Current main still uses exact normalized provider/model allowlist keys, so the intended fix is not implemented there; however, the PR’s unconditional hyphen-stripped variant broadens authorization to potentially distinct custom provider IDs.

## What This Changes

The PR adds hyphen-stripped provider key variants to model allowlist construction and adds unit coverage for `buildConfiguredAllowlistKeys` and `buildAllowedModelSet`.

## Best Possible Solution

Keep provider IDs exact by default and model intentional equivalence through explicit provider/plugin alias metadata, or fix the runtime resolver to preserve the configured provider ID. Regression coverage should prove declared aliases work while `my-provider` and `myprovider` remain separately allowlisted unless explicitly linked.

## Reproduction Assessment

Unclear for the original bug: the PR describes a runtime path that strips provider hyphens, but the discussion and inspected current code do not identify that path. The PR-introduced authorization regression is reproducible by allowlisting `my-provider/some-model` and selecting `myprovider/some-model`, because the patch adds that stripped key to `allowedKeys`.

## Solution Assessment

No. Expanding all hyphenated provider allowlist entries is broader than the reported mismatch and conflicts with current exact provider-ID semantics; explicit alias metadata or preserving the provider ID at the resolver boundary is the safer maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restrict stripped provider aliases to declared aliases:** `src/agents/model-selection.ts:231`
  - body: Adding the stripped provider key for every hyphenated provider turns one allowlist entry into permission for a different provider ID. Gate this on explicit alias metadata or fix the resolver that drops the hyphen.
