## Summary

Keep open: current main keeps model allowlist keys exact, but this PR adds global hyphen-stripped provider aliases at an authorization boundary without evidence that those provider IDs are always equivalent.

## What This Changes

The PR adds hyphen-stripped provider/model key variants to model allowlist construction and tests that both canonical and stripped provider forms are accepted.

## Best Possible Solution

Keep provider IDs exact by default; if an alias is needed, resolve it through declared provider/plugin alias metadata or fix the resolver that loses the configured provider ID.

## Reproduction Assessment

Unclear for the original report because no current runtime path that strips provider hyphens is identified. The PR-introduced authorization broadening is source-reproducible from the diff adding stripped variants to `allowedKeys`.

## Solution Assessment

No. Unconditional hyphen folding is broader than the reported mismatch and changes an authorization boundary; declared aliases or preserving the original provider ID would be narrower.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restrict stripped provider aliases to declared aliases:** `src/agents/model-selection.ts:231-239`
  - body: Adding `${stripped}/${model}` for every hyphenated allowlist key makes a config that permits only `my-provider/x` also authorize `myprovider/x`. Current provider normalization preserves arbitrary hyphens and provider config keys are trimmed rather than canonicalized, so these can be distinct providers; gate this on declared provider aliases or preserve the original provider ID instead.
  - confidence: 0.91
