## Summary

Keep this PR open but do not merge it as-is: the patch broadens a model allowlist authorization boundary by treating hyphenated and hyphen-stripped provider IDs as equivalent without an explicit alias contract.

## What This Changes

The PR adds hyphen-stripped provider/model key variants to model allowlist construction and unit tests for `buildConfiguredAllowlistKeys` and `buildAllowedModelSet`.

## Best Possible Solution

Keep provider IDs exact by default, then support intentional equivalence through declared provider/plugin alias metadata or by fixing the resolver that loses the configured provider ID.

## Reproduction Assessment

Unclear for the original reported bug because the discussion does not identify the current runtime path that strips provider hyphens. Yes for the PR-introduced regression by source inspection: adding the stripped key makes `myprovider/some-model` pass `allowedKeys.has(key)` when only `my-provider/some-model` was configured.

## Solution Assessment

No. The proposed unconditional folding is broader than the reported mismatch and changes an authorization boundary; explicit alias metadata or preserving the provider ID at the resolver boundary is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91
