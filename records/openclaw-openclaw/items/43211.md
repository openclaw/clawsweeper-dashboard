## Summary

Keep open: the proposed patch is not mergeable as-is because it broadens an exact model allowlist at an authorization boundary without tying the alternate provider spelling to declared provider alias metadata or showing a real runtime path that strips hyphens.

## What This Changes

The branch adds hyphen-stripped provider/model allowlist key variants and unit tests so a configured hyphenated provider entry also authorizes the same model under a provider ID with hyphens removed.

## Best Possible Solution

Keep provider IDs exact by default; if a hyphen-stripped spelling is valid, resolve it through declared provider/plugin alias metadata or fix the resolver that loses the configured provider ID, then prove the real runtime flow.

## Reproduction Assessment

Unclear for the reported bug because no current runtime path that strips provider hyphens is identified. The PR-introduced authorization broadening is source-reproducible from the diff adding stripped variants to the allowlist key sets.

## Solution Assessment

No. Unconditional hyphen folding is broader than the stated mismatch and should be replaced by declared alias resolution or by preserving the configured provider ID through the runtime path.
