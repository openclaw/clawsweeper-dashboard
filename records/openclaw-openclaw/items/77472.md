## Summary

Keep open: the production direction is narrow, but this PR is not mergeable because the new Windows tests fail on the PR head when the resolver reads the real registry SystemRoot casing.

## What This Changes

This PR changes `.cmd`/`.bat` process-wrapper launches to derive `cmd.exe` from `getWindowsInstallRoots()` instead of `process.env.ComSpec`, adds Windows wrapper regression tests, and updates the changelog.

## Best Possible Solution

Keep the production hardening direction, but update the new Windows tests to account for registry-preferred SystemRoot casing or stub registry lookup when the test is specifically exercising env/default fallback behavior.

## Reproduction Assessment

Yes. Source inspection shows current main still routes batch shims through `process.env.ComSpec`, and PR-head Windows CI gives a high-confidence reproduction of the introduced test failure.

## Solution Assessment

No, not as submitted. The production change is the right narrow shape, but the test assertions need to avoid hard-coded Windows root casing or explicitly disable registry roots before this is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

- **[P2] Make the Windows root expectation registry-safe:** `src/process/exec.windows.test.ts:202`
