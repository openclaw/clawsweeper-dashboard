## Summary

The issue appears valid on current main: Xiaomi advertises TTS models/voices and its synthesizer already consumes provider overrides, but the standard CLI/Gateway override resolver rejects Xiaomi because the provider does not implement `resolveTalkOverrides`.

## What This Changes

Support request-scoped Xiaomi MiMo TTS `voice` and `model` overrides through the existing OpenClaw TTS CLI and Gateway convert paths.

## Best Possible Solution

Add Xiaomi's provider-owned `resolveTalkOverrides` mapping for `voiceId` and `modelId`, cover the CLI/Gateway resolver path and request payload in tests, and leave the separate VoiceDesign feature tracked by #72927.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: `--voice` or `--model xiaomi/...` selects Xiaomi, `resolveExplicitTtsOverrides` requires `resolveTalkOverrides`, and Xiaomi currently does not provide that hook.

## Solution Assessment

Yes. The narrowest fix is provider-local: expose Xiaomi's existing override fields through `resolveTalkOverrides` rather than changing shared CLI/Gateway override semantics.
