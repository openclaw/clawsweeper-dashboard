## Summary

Keep open: current main still accepts inline `buffer` attachment fields for `message.send`, but the send hydration and Discord delivery paths do not turn those fields into a Discord multipart upload. The path/filePath upload route appears implemented and tested, so the remaining bug is the inline-buffer false-success path and overlapping open fix PRs need maintainer consolidation.

## What This Changes

Make Discord `message.send` deliver advertised attachment inputs instead of returning success with empty Discord attachments.

## Best Possible Solution

Land one canonical fix that either hydrates inline `buffer`/`filename`/`contentType` into Discord attachments through the existing media resolver and multipart upload path, or rejects unsupported inline fields before reporting success.

## Reproduction Assessment

Yes, at source level. A Discord `message.send` call with `message`, `buffer`, `filename`, and `contentType` follows a current-main path where the schema accepts the fields, `send` hydration returns early, and Discord only uploads when `mediaUrl` exists.

## Solution Assessment

Unclear until maintainers consolidate the open fix candidates. Supporting inline buffers matches the advertised schema, while explicit rejection would be the narrower false-success fallback if cross-channel inline attachment support is not desired.
