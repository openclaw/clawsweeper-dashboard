## Summary

Keep open. Current main still has the direct buffer-backed Discord `message.send` gap: the public send schema accepts buffer attachment fields, but ordinary `send` is excluded from attachment hydration, the shared runner and Discord bridge only forward media path/URL hints, and Discord multipart upload is only reached when `mediaUrl` is present. The path/filePath upload route appears implemented and tested, so the remaining work is narrowed to direct buffer delivery or explicit fail-fast rejection; open PRs #57335 and #63578 are still unmerged candidates.

## What This Changes

Discord `message.send` should deliver local or buffer-backed attachments as actual Discord attachments instead of returning success with an empty `attachments[]` array.

## Best Possible Solution

Align the advertised `send` contract with runtime behavior by either hydrating and delivering `buffer`/`filename`/`contentType` through the existing attachment pipeline and Discord multipart upload path with regression coverage, or rejecting unsupported attachment fields with a clear error.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is `message.send` for Discord with `message`, `buffer`, `filename`, and `contentType`: schema validation accepts it, hydration returns early for `send`, and no later layer maps the buffer into Discord media upload.

## Solution Assessment

Yes. Reusing the existing attachment hydration and multipart upload path for `send` when a buffer is present is the narrowest maintainable fix; explicit rejection is the safer alternative if `send` is not meant to support those fields.
