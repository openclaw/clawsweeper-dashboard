## Summary

Keep open. Current main still exposes `buffer`/filename/MIME attachment fields for ordinary Discord `message.send`, but that path does not hydrate or forward direct buffers into Discord multipart upload; two related fix PRs are open and conflicting, so this should wait for maintainer consolidation rather than be closed.

## What This Changes

Fix Discord `message.send` so advertised local image or buffer attachments are delivered, or fail loudly instead of returning success with no attachment.

## Best Possible Solution

Land one consolidated fix that either hydrates direct `buffer`/`filename`/MIME fields through the shared attachment pipeline into Discord multipart upload, or rejects those advertised fields explicitly with regression coverage and redacted live Discord proof.

## Reproduction Assessment

Yes, at source level. Current main accepts direct `buffer`/`filename`/MIME fields for `message.send`, but ordinary `send` is excluded from attachment hydration and Discord upload only runs when a path or URL becomes `mediaUrl`.

## Solution Assessment

Yes for keeping this open. The maintainable path is to review or consolidate the open attachment-hydration PRs, then close this issue only after the selected fix is merged and proven against Discord.
