## Summary

Keep open: current main and v2026.5.22 still advertise direct buffer attachment fields for ordinary `message.send`, but the shared attachment hydrator skips `send` and the Discord bridge only uploads when a path/URL becomes `mediaUrl`. The path/filePath multipart route now appears implemented and tested, so the remaining bug is direct buffer delivery or fail-fast rejection, with https://github.com/openclaw/openclaw/pull/63578 still open for maintainer consolidation.

## What This Changes

The issue requests that Discord `message.send` deliver local path and buffer attachments, or fail instead of returning a successful message id with no Discord attachments.

## Best Possible Solution

Land one consolidated fix that either hydrates direct `buffer`/filename/MIME fields for ordinary Discord `message.send` through the shared attachment pipeline and multipart upload, or fails loudly before returning success.

## Reproduction Assessment

Yes, source-level. Current main accepts direct `buffer`/filename/MIME fields for ordinary `message.send`, but `send` is excluded from attachment hydration and Discord upload only runs when `mediaUrl` is present.

## Solution Assessment

Yes for keeping this open. The narrow maintainable path is to review or consolidate the open send-media hydration work, then close this only after the selected fix is merged and proven against Discord.
