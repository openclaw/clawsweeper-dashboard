- `impact:message-loss`: The issue is about Discord message attachments being silently omitted while the send reports success.

## Summary

Keep open. Current main still has the Discord `message.send` false-success path for direct buffer attachments: the public send schema accepts attachment fields, but ordinary `send` is not hydrated into a deliverable attachment and the Discord runtime only enters multipart upload when a media URL/path is present; the open fix PRs still need maintainer review or consolidation.

## What This Changes

The issue reports Discord `message.send` returning success and a message id for attachment inputs while Discord receives a message with no uploaded attachment.

## Best Possible Solution

Land one consolidated fix that sends `buffer`/`filename`/`contentType` through the shared attachment hydration and Discord multipart path with regression coverage, or reject unsupported fields instead of returning false success.

## Reproduction Assessment

Yes, at source level. A `send` call with `buffer`/`filename`/`contentType` is accepted by the schema, but current main does not hydrate `send` attachments and Discord only uploads when a media URL/path reaches `sendMessageDiscord`.

## Solution Assessment

Yes on direction, not yet on the exact PR. The maintainable fix is to reuse the existing attachment hydration and multipart upload path for ordinary `send`, or fail fast if those fields are intentionally unsupported.
