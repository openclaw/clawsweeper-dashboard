- `impact:message-loss`: The reported behavior drops the attachment portion of a message while returning a successful message id.

## Summary

Keep open. Current `main` and the latest shipped tag still have the Discord `message.send` buffer false-success path: the tool schema accepts buffer attachment fields, but ordinary `send` is not hydrated and the Discord bridge/runtime only enter multipart upload when a path or URL media reference is present. Open pull requests https://github.com/openclaw/openclaw/pull/57335 and https://github.com/openclaw/openclaw/pull/63578 target this bug class but are not merged.

## What This Changes

Discord `message.send` with a local image or image buffer should upload the file instead of returning success with empty Discord attachments.

## Best Possible Solution

Land or consolidate one fix that either hydrates ordinary Discord `message.send` buffer attachments through the shared media policy and multipart upload path, or rejects those advertised fields loudly with regression coverage and live Discord proof.

## Reproduction Assessment

Yes, at source level. The current schema accepts `buffer`/`filename`/MIME fields for `send`, but hydration returns before `send`, and the Discord bridge/runtime only upload when a path or URL becomes `mediaUrl`.

## Solution Assessment

Yes for keeping this issue open. The maintainable path is to review or consolidate the open attachment-hydration PRs, then close this issue only after the chosen fix is merged and proven against Discord.
