- `impact:message-loss`: The reported behavior drops the attachment portion of a message while returning a successful message id.

## Summary

Keep open. Current `main` and the latest shipped tag still advertise direct `buffer` attachments for ordinary Discord `message.send`, but the send hydration gate excludes `send`, the shared runner only promotes URL/path media hints, and the Discord bridge only enters multipart upload when `mediaUrl` is present. Open pull requests https://github.com/openclaw/openclaw/pull/57335 and https://github.com/openclaw/openclaw/pull/63578 target this bug class but are not merged and both currently conflict with main.

## What This Changes

Discord `message.send` should not return success while dropping supplied attachment fields such as `buffer`, `path`, or `filePath`.

## Best Possible Solution

Land one consolidated fix that either hydrates direct `buffer`/`filename`/MIME fields through the shared attachment policy into Discord multipart upload, or rejects those advertised fields loudly with regression coverage and redacted live Discord proof.

## Reproduction Assessment

Yes, at source level. Current main accepts direct `buffer`/`filename`/MIME fields for `message.send`, but ordinary `send` is excluded from attachment hydration and the Discord upload path only runs when a path or URL becomes `mediaUrl`.

## Solution Assessment

Yes for keeping this issue open. The maintainable path is to review or consolidate the open attachment-hydration PRs, then close this issue only after the chosen fix is merged and proven against Discord.
