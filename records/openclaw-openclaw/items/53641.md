## Summary

Keep open: current main still advertises `buffer`/`filename` on `send`, but the send path only turns media/path aliases into `mediaUrl`/`mediaUrls`, so Discord can return success for a text/card message while no file is uploaded. Related PR work exists, but it is either open/conflicting or closed unmerged.

## What This Changes

The issue reports Discord `message.send` returning success and a message id while dropping attachments supplied through `filePath`, `path`, or direct `buffer` inputs.

## Best Possible Solution

Make the `message.send` attachment contract explicit, then either hydrate direct buffer attachments into real Discord uploads or reject unsupported direct-buffer sends with focused regression tests and live Discord proof.

## Reproduction Assessment

Yes: source inspection gives a high-confidence current-main reproduction path for `send` with `buffer`/`filename` because the schema accepts the fields while hydration and payload assembly ignore them as media. I did not run a live Discord upload in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract: preserving the advertised buffer support requires hydrating it into an upload, while removing support should fail loudly instead of sending a successful no-attachment message.
