## Summary

Keep open. Current main and v2026.5.18 still have the direct buffer-backed Discord message.send false-success gap: the schema advertises buffer attachment fields, but ordinary send does not hydrate those fields and the Discord upload branch only runs when a media URL/path is present. Open pull requests https://github.com/openclaw/openclaw/pull/57335 and https://github.com/openclaw/openclaw/pull/63578 already target this bug class.

## What This Changes

The issue requests that Discord message.send deliver local image or buffer attachments as real Discord uploads, or fail instead of returning a successful message id with no attachments.

## Best Possible Solution

Land or consolidate one fix that makes ordinary Discord message.send hydrate and deliver buffer-backed attachments through the existing multipart upload path, or explicitly reject unsupported buffer fields with regression coverage and Discord proof.

## Reproduction Assessment

Yes, at source level. Current main accepts buffer attachment fields for send, but the ordinary send path does not hydrate or forward those fields into the Discord multipart upload branch.

## Solution Assessment

Yes for keeping this issue open. The maintainable path is to review or consolidate the open attachment-hydration PRs, then close this issue only after the chosen fix is merged and proven against Discord.
