## Summary

Keep open: current main still has a source-level false-success path for direct buffer-backed Discord sends, and the two related fix candidates are still open rather than merged.

## What This Changes

The issue reports Discord `message.send` returning success and a message id while advertised attachment inputs are silently omitted from the delivered Discord message.

## Best Possible Solution

Land one consolidated fix that either hydrates direct `buffer`/`filename`/MIME fields through the shared attachment pipeline into Discord multipart upload, or rejects those fields explicitly so callers cannot receive a false success.

## Reproduction Assessment

Yes, at source level. Current main accepts direct `buffer`/`filename`/MIME fields for `message.send`, but `send` is excluded from attachment hydration and Discord upload only runs when path/URL media becomes `mediaUrl`.

## Solution Assessment

Yes for keeping this open. The maintainable path is to review or consolidate the open attachment-hydration PRs, then close this issue only after the selected fix is merged and proven against Discord.
