## Summary

Keep open. Current `main` still has the advertised Discord `message.send` buffer-attachment gap: the schema accepts buffer attachment fields, but ordinary `send` is excluded from attachment hydration and the Discord action bridge only forwards path/URL media into the multipart upload path. The related fix PRs are open, not merged, so this issue should stay open for review/consolidation rather than be closed.

## What This Changes

Fix Discord `message.send` so advertised local or buffer attachments upload instead of returning success with no Discord attachment.

## Best Possible Solution

Land one consolidated fix that makes ordinary Discord `message.send` deliver advertised buffer/file attachment inputs through the existing multipart path with regression coverage, or reject unsupported attachment fields loudly instead of returning false success.

## Reproduction Assessment

Yes, at source level. Current `message.send` exposes `buffer`, `filename`, `contentType`, and `mimeType`, but ordinary `send` is excluded from attachment hydration and the Discord bridge only forwards media/path/filePath as `mediaUrl`.

## Solution Assessment

Unclear for the open PRs pending maintainer consolidation. The best direction is to reuse the existing attachment hydration/media policy path without creating a false success or bypassing gateway/media size boundaries.
