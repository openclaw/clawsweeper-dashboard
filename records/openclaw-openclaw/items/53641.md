## Summary

Keep open. Current main still has the direct `buffer` false-success gap for Discord `message.send`: the public send schema accepts buffer attachment fields, but ordinary `send` is excluded from attachment hydration, the shared runner and Discord bridge only promote path/URL media references, and the Discord upload branch is gated on `mediaUrl`. The path/filePath multipart route is implemented and tested, so the remaining work is narrowed to direct buffer delivery or explicit fail-fast rejection; open PRs #57335 and #63578 already target that area but are not merged.

## What This Changes

The issue reports that Discord `message.send` returns success and a message id while attachments provided through `filePath`, `path`, or `buffer` do not arrive in Discord.

## Best Possible Solution

Align the advertised `send` contract with runtime behavior by either hydrating and delivering `buffer`/`filename`/`contentType` through the existing attachment pipeline and Discord multipart upload path with focused regression coverage, or rejecting unsupported attachment fields with a clear error before returning success.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call ordinary Discord `message.send` with `buffer`/`filename`/`contentType`; current main accepts those fields in the schema but never routes them through hydration or Discord multipart upload.

## Solution Assessment

Yes for the keep-open outcome, but unclear which open PR is the best implementation. The narrow maintainable fix should reuse the existing attachment hydration/upload path or explicitly fail unsupported `send` buffer fields, rather than adding a separate Discord-only buffer path.
