## What This Changes

The issue reports that Discord `message.send` can return success and a message id while silently omitting a requested attachment.

## Best Possible Solution

Land one consolidated fix that either hydrates direct `buffer`/`filename`/MIME fields through the shared attachment pipeline into Discord multipart upload, or rejects unsupported fields explicitly, with regression coverage and redacted live Discord proof.

## Reproduction Assessment

Yes, at source level. Current main accepts direct `buffer`/`filename`/MIME fields for `message.send`, but `send` is excluded from attachment hydration and Discord upload only runs when path/URL media becomes `mediaUrl`.

## Solution Assessment

Yes for keeping this open. The maintainable path is to review or consolidate the open attachment-hydration PRs, then close this issue only after the selected fix is merged and proven against Discord.
