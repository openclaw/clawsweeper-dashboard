## Label Justifications

- `P2`: The remaining bug is a normal-priority Discord/message-tool delivery failure with limited but real user impact.
- `impact:message-loss`: The attachment portion of a channel message can be silently dropped while OpenClaw reports success.

## Summary

Keep open: current main still has the direct buffer-backed Discord `message.send` false-success gap. Path/filePath media references now have a multipart route, but direct `buffer`/filename/MIME fields are still accepted by the public send schema without being hydrated or forwarded to Discord upload.

## What This Changes

The issue reports Discord `message.send` returning success while dropping attachment payloads from `filePath`, `path`, or `buffer` inputs.

## Best Possible Solution

Adopt one maintainer-approved `message.send` attachment contract: either hydrate direct `buffer`/filename/MIME fields through the shared attachment pipeline and Discord multipart upload with regression coverage, or reject those fields before returning success.

## Reproduction Assessment

Yes, source-level: current main accepts direct `buffer`/filename/MIME fields for ordinary `message.send`, but that path is excluded from hydration and Discord upload only runs when a `mediaUrl` reaches the runtime. I did not run a live Discord send in this read-only review.

## Solution Assessment

Unclear until maintainers choose the final send-attachment contract. The safest direction is to consolidate the open send-media hydration work and close this only after the selected behavior is merged and proven against Discord.
