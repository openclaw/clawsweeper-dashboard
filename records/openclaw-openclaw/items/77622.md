## Summary

Keep this PR open. The maintainer label already makes it a human-handled policy change, and the current proof classifier has two blocking mismatches with the stated contract: fenced mock/test output can pass, while ordinary linked artifact proof can fail.

## What This Changes

Adds an external-contributor real-behavior-proof PR gate with a new workflow, proof-policy scripts, Barnacle labels/actions, PR template and contributor docs updates, changelog entry, and tests.

## Best Possible Solution

Land a maintainer-approved version of the gate that keeps the trusted base-checkout workflow design while making the proof policy match the documented evidence contract.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: put `pnpm test` or CI output inside a fenced block and it can pass as real evidence, while a normal artifact URL without an image/video extension can be rejected.

## Solution Assessment

No. The policy direction is maintainable, but the current classifier is not the best implementation because it both accepts supplemental-only proof in one format and rejects a documented valid proof format.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reject fenced mock-only proof before passing the gate:** `scripts/github/real-behavior-proof-policy.mjs:226-227`
  - body: Because `hasMockOnlyEvidence` is suppressed whenever `hasRealEvidence` is true, and any multi-line fenced block sets `hasRealEvidence`, an external PR can paste `pnpm test` or CI output in a code fence and still pass the required check. That undercuts the promised mock-only rejection.
  - confidence: 0.91
- **[P2] Accept the documented artifact proof format:** `scripts/github/real-behavior-proof-policy.mjs:59-60`
  - body: The template and error text say linked artifacts count, but the real-evidence regex only accepts GitHub user attachments or media URLs with image/video extensions. A normal GitHub Actions artifact link without those tokens will be rejected as insufficient, blocking valid proof on the new required check.
  - confidence: 0.84
