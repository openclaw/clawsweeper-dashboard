## Summary

Keep open: the bug path appears valid and current main has not implemented the fix, but this PR should not merge yet because the sanitizer can still return key/depth-heavy details that the existing validator rejects, and the after-fix real behavior proof is mock-only.

## What This Changes

The PR fast-paths agent tool-result middleware when no handler exists, sanitizes incoming tool-result details before handlers run, adds harness regression tests, isolates two provider-registry test modules, and adds a changelog entry.

## Best Possible Solution

Keep the fix at the harness boundary, but make the sanitizer return details that satisfy the existing byte, key, and depth caps, add regressions for those caps, and then supply redacted runtime proof from a tokenjuice-enabled setup.

## Reproduction Assessment

Yes for the source-level path: current main validates the unmodified result after a registered no-op middleware, and tokenjuice returns undefined when it leaves a result unchanged. I did not establish a live Discord or gateway reproduction in this read-only review.

## Solution Assessment

No, not yet. The harness boundary is the right place to fix this bug class, but the current sanitizer needs to mirror the validator's key and depth caps before the patch is safe to merge.
