## Summary

Keep open: this targets a real source-reproducible harness bug, but the PR still allows JSON-safe details that exceed the validator key/depth caps to fail after a no-op middleware, and the external after-fix proof is only tests plus pre-fix logs.

## What This Changes

The PR fast-paths tool-result middleware when no handler is registered, sanitizes incoming tool result `details` before middleware handlers run, adds harness regression tests, updates two provider registry tests, and adds a changelog entry.

## Best Possible Solution

Keep the fix at the harness boundary, but make the incoming sanitizer always return a value that satisfies the existing validator caps, add regressions for key/depth caps, then supply redacted tokenjuice-enabled runtime proof.

## Reproduction Assessment

Yes for the source-level path: current main validates the unmodified result after a registered no-op middleware, and tokenjuice returns `undefined` when it leaves a result unchanged. I did not run a live Discord roundtrip in this read-only review.

## Solution Assessment

No, not yet. The harness boundary is the right place to fix this bug class, but the sanitizer needs to mirror the validator's key and depth caps before the patch is safe to merge.

## Review Findings

Overall correctness: patch is incorrect
