## Summary

Keep open: this PR targets a real source-reproducible harness bug, but the current patch still has a blocking sanitizer edge case and lacks after-fix real runtime proof.

## What This Changes

The PR adds a no-middleware fast path, sanitizes incoming tool-result details before registered middleware runs, adds harness regression tests, and updates the changelog.

## Best Possible Solution

Keep the fix at the harness boundary, but make sanitized incoming details always satisfy the existing validator bounds and require redacted after-fix runtime proof before merge.

## Reproduction Assessment

Yes for the source-level path: current main validates an unmodified result after a registered no-op middleware, and tokenjuice returns `undefined` when it leaves the result unchanged. I did not establish a fresh live Discord roundtrip in this read-only review.

## Solution Assessment

No, not yet. The harness boundary is the right maintainable place to fix this bug class, but the sanitizer must mirror or call the existing validator so key and depth caps cannot still fail closed.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0

Full review comments:

- **[P2] Ensure sanitized details satisfy validator caps:** `src/agents/harness/tool-result-middleware.ts:138`
  - body: `sanitizeMiddlewareDetailsValue` only checks serialized size before returning `JSON.parse(serialized)`. JSON-safe details with 1001 small keys or nesting past depth 20 remain invalid under `isValidMiddlewareDetails`, so a registered no-op middleware can still swap the real tool result for the failure sentinel. Validate the parsed value with the same details validator or collapse it to a bounded marker before returning it.
