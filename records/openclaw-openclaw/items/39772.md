## Summary

Keep open: the Telegram reply-gating direction is useful, but this PR still lacks after-fix live Telegram proof and its current head regresses current-main streaming/topic behavior that maintainers explicitly marked as intentional.

## What This Changes

The PR gates Telegram streaming preview `reply_to_message_id` use, passes `replyToMode` into draft-stream creation, adds regression tests, and updates the changelog.

## Best Possible Solution

Port only the single-use reply gating onto current main while preserving message/edit streaming, DM topic fail-closed sends, overflow chaining, and `replyToMode: "all"`, then attach redacted live Telegram proof.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection shows current main can reuse reply metadata after `forceNewMessage()`, and the linked report includes a Telegram screenshot, but the after-fix Telegram UI path still needs live proof.

## Solution Assessment

No for the branch as-is. The reply-gating approach is the right narrow direction, but this PR should be re-ported onto current main without the topic retry, native draft transport, or overflow chaining regressions.

  - body: PR head retries `message thread not found` by deleting `message_thread_id`. Current main deliberately does not do this for DM topics; retrying without the topic id can send the preview outside the intended topic instead of surfacing the routing error.
  - confidence: 0.9

  - body: PR head stops streaming when rendered text exceeds `maxChars`. Current main splits and continues in chained preview messages, with tests covering this, so long Telegram streaming replies would stop instead of continuing.
  - confidence: 0.88
- **[P2] Keep streaming on message/edit transport:** `extensions/telegram/src/draft-stream.ts:151-157`
  - body: PR head reintroduces native `sendMessageDraft` selection for DM-scoped `auto` preview streams. Maintainer notes and current main keep Telegram answer streaming on persistent `sendMessage` plus `editMessageText`, because native drafts are ephemeral and still need a later real message.
  - confidence: 0.78
