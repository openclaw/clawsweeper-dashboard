## Summary

Keep open: the central Telegram streaming bug still has a plausible narrow fix, but this PR head is stale relative to current main and would reintroduce later Telegram stream regressions/security-sensitive routing behavior. It also still lacks required live Telegram proof for a visible reply-header change.

## What This Changes

This PR gates Telegram stream-preview reply metadata so single-use reply modes apply `reply_to_message_id` only to the first rotated preview, passes `replyToMode` into draft streaming, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Port the single-use reply gating onto current main’s message/edit draft-stream implementation while preserving DM-topic fail-closed behavior, over-limit preview chaining, and `replyToMode: "all"`, then add redacted live Telegram proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows current main still reuses `replyParams` across `forceNewMessage()` rotations, and the linked bug report includes a Telegram screenshot of the visible `Deleted message` header.

## Solution Assessment

No for this branch as-is. The reply-gating idea is the right narrow direction, but the current branch must be rebased/ported without restoring native draft transport, threadless DM-topic retry, or over-limit preview stopping.
