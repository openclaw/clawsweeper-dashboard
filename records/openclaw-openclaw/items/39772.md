## What This Changes

The PR gates Telegram streaming preview reply metadata so single-use `reply_to_message_id` is consumed only by the first preview while adding reply-mode coverage, tests, and a changelog entry.

## Best Possible Solution

Port the narrow reply-consumption fix onto current main and make draft preview thread fallback match the existing DM-only final-delivery policy, preserving regression coverage and the changelog entry.

## Reproduction Assessment

Yes, at source level. On current main, create a Telegram draft stream with `replyToMessageId`, flush once, call `forceNewMessage()`, then flush again; the second `sendMessage` still derives params from the same `replyParams` object.

## Solution Assessment

Yes in direction, but not as currently mergeable. Gating single-use reply metadata inside `createTelegramDraftStream` is the narrow maintainable fix, but the branch needs a current-main port and draft preview fallback should fail closed for forum topics like final delivery already does.
