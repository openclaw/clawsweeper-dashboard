## Summary

Keep open: the specific Telegram `forceNewMessage()` reply-rotation path is still source-reproducible on current main and in the latest release, but this PR is stale and not mergeable as-is because it would restore removed stream behavior and still lacks live Telegram proof.

## What This Changes

This PR gates Telegram stream preview `reply_to_message_id` so single-use reply modes apply it only to the first rotated preview, passes `replyToMode` into draft streaming, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Port the single-use reply gating onto current main’s draft-stream implementation while preserving message/edit transport, DM-topic fail-closed behavior, over-limit preview chaining, and `replyToMode: "all"`, then add redacted live Telegram proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main and v2026.5.12 reuse `replyParams` after `forceNewMessage()`, and the linked bug report includes a Telegram screenshot of the visible `Deleted message` header.

## Solution Assessment

No for this branch as-is. The reply-gating idea is the right narrow fix, but it must be ported without restoring native draft transport, threadless DM-topic retry, or over-limit preview stopping.
