## Summary

Keep open: the branch is a useful Telegram streaming fix, but current main still has the reply-reuse bug and the PR head remains incorrect because draft-preview thread fallback still drops `message_thread_id` on thread-not-found, unlike final delivery's DM-only policy.

## What This Changes

The branch passes Telegram `replyToMode` into draft streams, consumes single-use `reply_to_message_id` only on the first preview send, adds regression coverage, and adds a changelog entry.

## Best Possible Solution

Port the reply-consumption fix to current main and align draft preview thread fallback with final delivery's DM-only threadless retry policy.

## Reproduction Assessment

Yes, at source level. On current main, create a Telegram draft stream with `replyToMessageId`, flush once, call `forceNewMessage()`, then flush again; the second `sendMessage` is still built from the same `replyParams` object.

## Solution Assessment

Yes in direction, but no as currently mergeable. Gating single-use reply metadata inside `createTelegramDraftStream` is narrow and maintainable, but the branch must be ported to current main and the preview thread fallback must fail closed for forum topics.

## Review Findings
