## Summary

Keep open. Current main still reuses one Telegram draft-stream reply parameter object across rotated preview sends, so the user-visible bug is not implemented on main; the PR is directionally useful and has focused regression coverage, but it still needs maintainer review for current-main drift and the existing threadless retry behavior in the touched send path.

## What This Changes

The PR gates Telegram draft-stream `reply_to_message_id` so single-use reply modes only apply it to the first preview/materialized send, passes `replyToMode` into draft streaming, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Port the reply-gating logic onto current main's message-only Telegram draft stream, preserve `message_thread_id` and `replyToMode: "all"` semantics, keep the changelog/test coverage, and make an explicit maintainer decision on whether draft-stream thread-not-found fallback should fail closed or be limited to an opt-in/DM-only case.

## Reproduction Assessment

Yes. A source-level reproduction is to create a Telegram draft stream with `replyToMessageId`, flush once, call `forceNewMessage()`, then flush again; current main reuses the same `replyParams`, so the second new preview can still carry `reply_to_message_id`.

## Solution Assessment

Yes in direction, but not ready as-is. Gating reply consumption inside `createTelegramDraftStream` is the narrowest maintainable fix, but the branch needs a current-main port and explicit maintainer handling of the thread fallback behavior before merge.
