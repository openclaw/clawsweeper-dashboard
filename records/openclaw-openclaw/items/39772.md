## Summary

Keep open: current main still reuses the same Telegram draft-stream reply params after preview rotation, so the linked behavior is not implemented on main. The PR is directionally useful, but it needs a current-main port and explicit maintainer/security handling before merge.

## What This Changes

The PR gates Telegram draft-stream `reply_to_message_id` use so single-use reply modes only quote the first streaming preview/materialized send, passes `replyToMode` into the draft stream, adds regression coverage, and updates the changelog.

## Best Possible Solution

Land a narrow current-main patch that consumes single-use draft replies only after a delivered first preview, preserves `message_thread_id` and `replyToMode: "all"`, and aligns draft-stream thread fallback with the safer DM-only delivery helper policy.

## Reproduction Assessment

Yes, at source level. On current main, create a Telegram draft stream with `replyToMessageId`, flush once, call `forceNewMessage()`, then flush again; the second new `sendMessage` still derives params from the same `replyParams` object.

## Solution Assessment

Yes in direction, but not merge-ready as-is. Gating reply consumption inside `createTelegramDraftStream` is the narrow fix, but the branch needs a current-main port and a maintainer decision on the thread fallback behavior.
