## What This Changes

The PR gates Telegram streaming preview `reply_to_message_id` so single-use reply modes only attach it to the first preview send, while passing `replyToMode` into draft-stream creation and adding tests plus a changelog entry.

## Best Possible Solution

Port the single-use reply gating onto current main's message-preview-only Telegram stream while preserving `replyToMode: "all"`, over-limit preview chaining, private-topic fail-closed behavior, and requiring redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, at source level: current main creates `replyParams` once, reuses it for new preview sends, and `forceNewMessage()` resets preview state without changing those params. I did not establish a live Telegram reproduction of the visible UI artifact.

## Solution Assessment

No, not as the branch currently stands. The fix direction is narrow, but the branch must be rebased onto current main and must preserve message-preview-only streaming plus DM/topic fail-closed fallback behavior.
