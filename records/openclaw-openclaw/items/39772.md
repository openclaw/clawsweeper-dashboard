## Summary

Keep open: the PR targets a plausible Telegram streaming reply bug that is still visible in current-main source, but the branch is not mergeable as-is because it reintroduces older streaming behavior, has a topic-routing security concern, and lacks live Telegram proof.

## What This Changes

The PR gates Telegram draft-stream reply metadata so only the first rotated streaming preview uses `reply_to_message_id`, passes `replyToMode` into preview creation, and adds regression tests plus a changelog entry.

## Best Possible Solution

Port only the single-use reply gating onto current main while preserving persistent message previews, topic fail-closed behavior, over-limit chaining, and `replyToMode: "all"`, then attach redacted live Telegram proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows current main can still reuse the same reply params after `forceNewMessage()` rotations, and the linked report included a Telegram screenshot of the visible `Deleted message` header.

## Solution Assessment

No for this branch as-is. The reply-gating direction is the right narrow fix, but this branch needs a current-main port that avoids native draft transport, threadless topic retry, and lost overflow preview chaining.
