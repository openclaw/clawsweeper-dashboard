## Summary

Keep open. Current main still reuses one Telegram draft-stream reply parameter object across rotated preview sends, while this PR contains the intended single-use reply gating and regression coverage. The branch now needs maintainer follow-up because it is dirty against current main after the native draft preview transport removal, and the thread-fallback security concern should be resolved or explicitly split before landing.

## What This Changes

The PR adds Telegram draft-stream reply gating so single-use `reply_to_message_id` is applied only to the first successful preview/materialized send, preserves thread IDs on later previews, preserves `replyToMode: "all"`, and adds tests plus a changelog entry.

## Best Possible Solution

Land a current-main version of the branch: add `replyToMode` to the draft-stream call, apply `reply_to_message_id` only to the first successful/ambiguous preview for single-use modes, restore the reply target on safe first-send failures, preserve `message_thread_id` on later previews, keep `replyToMode: "all"` behavior, and update the tests after the native draft transport removal.

## Reproduction Assessment

Yes. The focused source-level reproduction is to create a Telegram draft stream with `replyToMessageId`, flush once, call `forceNewMessage()`, and flush again; current main would reuse the same `replyParams` for the second send. The live Telegram visual artifact was not re-run in this read-only review, but the send-parameter behavior is directly visible in current code.

## Solution Assessment

Yes in direction: gating reply state inside `createTelegramDraftStream` while preserving thread parameters and `replyToMode` is the narrowest maintainable fix. The exact branch needs to be rebased/ported to current main and the thread-fallback security concern needs an explicit maintainer decision before merge.
