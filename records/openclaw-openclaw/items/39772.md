## Summary

Keep open, but do not merge as-is: the reported Telegram reply artifact is plausible and the reply-gating idea is useful, but this stale branch conflicts with current Telegram stream policy, regresses DM-topic and long-preview behavior, and lacks required live Telegram proof.

## What This Changes

This PR changes Telegram draft streaming to consume `reply_to_message_id` once for single-use reply modes after rotated previews, passes `replyToMode` into the stream, and adds regression tests plus a changelog entry.

## Best Possible Solution

Port the single-use reply gating onto current main’s `sendMessage`/`editMessageText` draft-stream path while preserving `replyToMode: "all"`, DM-topic fail-closed behavior, over-limit preview chaining, and live Telegram proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main still reuses one preview `replyParams` object across `forceNewMessage()` rotations, and the linked bug report includes a Telegram screenshot, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No for this branch as-is. The single-use reply gating direction is narrow, but the implementation must be rebased or ported without restoring native draft transport, threadless DM-topic retry, or over-limit preview stopping.
