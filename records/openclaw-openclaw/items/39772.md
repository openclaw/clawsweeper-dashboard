## What This Changes

The PR gates Telegram draft-stream reply metadata so single-use previews keep `reply_to_message_id` only for the first delivered preview while preserving `replyToMode: "all"`, tests, and a changelog entry.

## Best Possible Solution

Port the narrow reply-consumption fix onto current main and align draft preview thread fallback with the existing DM-only `sendTelegramWithThreadFallback` policy, with regression coverage and the changelog entry preserved.

## Reproduction Assessment

Yes, at source level. On current main, create a Telegram draft stream with `replyToMessageId`, flush once, call `forceNewMessage()`, then flush again; the second new `sendMessage` still derives params from the same `replyParams` object.

## Solution Assessment

Yes in direction, but not merge-ready as this PR currently stands. Gating reply consumption inside `createTelegramDraftStream` is the narrow maintainable fix, but it must be ported to current main and the draft preview thread fallback should match the existing DM-only delivery helper policy.
