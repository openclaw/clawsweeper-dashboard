## Summary

Keep open. The PR targets a real current-main Telegram streaming bug: current main reuses the same `reply_to_message_id` draft-stream send params across `forceNewMessage()` rotations, while the branch adds directionally useful reply gating, tests, and a changelog entry. It still needs maintainer review because the branch is dirty against current main after native draft-preview transport removal, and the touched Telegram thread-fallback path has an unresolved disclosure concern.

## What This Changes

The PR adds Telegram draft-stream reply gating, passes `replyToMode` into streamed previews, adds regression coverage for rotation/threading/failure cases, and adds a user-facing changelog entry.

## Best Possible Solution

Port the reply-gating logic onto current main's message-only Telegram draft stream: consume single-use `reply_to_message_id` for the first successful or ambiguous preview/materialized send, preserve `message_thread_id`, keep `replyToMode: "all"` behavior, add focused regression tests, keep the changelog entry, and make an explicit maintainer decision on the thread-fallback behavior before landing.

## Reproduction Assessment

Yes. A source-level reproduction is to create a Telegram draft stream with `replyToMessageId`, flush once, call `forceNewMessage()`, then flush again; current main reuses the original `replyParams`, so the second new preview can still carry `reply_to_message_id`.

## Solution Assessment

Yes in direction, but not as-is. Gating reply state inside `createTelegramDraftStream` is the narrowest maintainable fix, but this branch needs a current-main port and an explicit decision on the forum-thread fallback before merge.
