## Summary

Keep open: the fix is in the right Memory Wiki extraction path, but the current fence regex can still leak `[[...]]` text from a valid fenced block when a shorter fence-looking line appears before the real closing fence, and the PR still lacks user-facing `openclaw wiki lint` after-fix proof.

## What This Changes

The PR masks fenced code blocks and inline code before Memory Wiki wikilink extraction, exports the extractor for tests, and adds markdown/lint regressions for code-origin broken-wikilink warnings.

## Best Possible Solution

Keep the fix at the Memory Wiki extractor boundary, replace the fence regex with stateful markdown-region scanning or an equivalent scanner that skips invalid close candidates, add the missing regression, and show redacted CLI lint proof.

## Reproduction Assessment

Yes. Current main is source-reproducible because code-bearing markdown flows into `linkTargets`, and a read-only probe against the PR regex shows a remaining valid-fence case still leaks a code-region wikilink target.

## Solution Assessment

No. The Memory Wiki extractor is the right layer, but the current lazy regex plus rejection callback is not the best fix because it stops at invalid fence-looking lines instead of scanning to the real close.
