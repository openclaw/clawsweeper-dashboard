## Summary

Keep this open: current main still accepts inline attachment fields on `message.send`, but the ordinary send hydration, payload building, and Telegram runtime path do not carry direct `buffer`/`filename` through to a delivered attachment or an explicit failure.

## What This Changes

Make Telegram `message.send` either deliver advertised inline `buffer`/`filename` attachments or reject them instead of returning success.

## Best Possible Solution

Review and land one consolidated fix that either honors `buffer`/`filename` in Telegram `message.send` end to end or rejects unsupported inline attachment fields before success.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with text plus `buffer`, `filename`, and `mimeType`; current main accepts those fields but does not hydrate or forward the buffer to Telegram delivery.

## Solution Assessment

Unclear for any one open patch. The maintainable endpoint is a single contract decision and implementation that either honors the advertised inline attachment fields end to end or rejects them before success.
