## Summary

Keep this open: current main still advertises inline attachment fields on `message.send`, while the shared send path and Telegram runtime only route URL/path media. The latest related comment is useful because https://github.com/openclaw/openclaw/pull/84596 covers only the narrower multi-URL path and explicitly does not solve the broader buffer/filename hydration bug.

## What This Changes

The issue asks for Telegram `message.send` to either deliver inline `buffer`/`filename` attachments or reject unsupported fields instead of returning `ok=true`.

## Best Possible Solution

Land one consolidated boundary-correct fix that either preserves inline buffers through the actual delivery runtime or rejects unsupported inline fields before a send can report success.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with text plus `buffer`, `filename`, and `mimeType`; current main accepts those fields but does not hydrate or forward the direct buffer to Telegram delivery.

## Solution Assessment

Not applicable to a specific patch yet. The maintainable fix is a single shared outbound/Telegram boundary change that either honors the advertised inline attachment fields or rejects them explicitly before success.
