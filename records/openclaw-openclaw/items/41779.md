## Summary

Keep this open: current main still accepts `buffer`/`filename` on `message.send`, but the ordinary send path does not hydrate or forward those fields to Telegram, so the reported false-success attachment loss remains source-reproducible. Related open PRs cover the central and adjacent media paths, including a narrower URL-backed fan-out PR, so this needs consolidation rather than closure.

## What This Changes

The issue asks for Telegram `message.send` to either deliver inline `buffer`/`filename` attachments or reject those fields instead of returning success after a text-only send.

## Best Possible Solution

Land one consolidated contract fix that either carries `buffer`/`filename` through Telegram `message.send` end to end or rejects unsupported inline attachment fields before returning success.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with text plus `buffer`, `filename`, and `mimeType`; current main accepts those fields but does not hydrate or forward the buffer to Telegram delivery.

## Solution Assessment

Unclear for any one open patch. The maintainable endpoint is a single contract decision and implementation that either honors the advertised inline attachment fields end to end or rejects them before success.
