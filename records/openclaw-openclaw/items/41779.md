## Summary

Keep this open: current main still advertises inline attachment fields on `message.send`, but the ordinary send path and Telegram runtime do not hydrate or forward direct `buffer`/`filename` data, so the false-success bug remains valid. Open implementation PRs exist, but none has merged.

## What This Changes

The issue requests that Telegram `message.send` either deliver advertised inline buffer attachments or reject unsupported inline attachment fields instead of returning success.

## Best Possible Solution

Land one consolidated fix that either hydrates and forwards inline buffer attachments through the correct outbound and Telegram plugin boundary, or rejects unsupported inline attachment fields before any send succeeds.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with `message`, `buffer`, `filename`, and `mimeType`; current main accepts those fields but only hydrates and forwards URL/path media.

## Solution Assessment

Unclear. Supporting the advertised buffer delivery and explicitly rejecting unsupported buffer fields would both fix the false-success bug; the maintainable path is to consolidate the competing PRs into one boundary-correct fix.
