## Summary

Keep this open: current main and the latest release still advertise inline attachment fields on `message.send`, while the ordinary send hydration and Telegram dispatch paths do not carry direct `buffer`/`filename` through to delivery. The issue is source-reproducible, but overlapping open PRs and the support-versus-reject contract choice make this maintainer consolidation work rather than a new automated fix branch.

## What This Changes

The issue requests that Telegram `message.send` either deliver inline `buffer`/`filename` attachments or reject those advertised fields instead of returning success while dropping the attachment.

## Best Possible Solution

Land one consolidated contract fix that either hydrates and forwards inline `buffer`/`filename` through Telegram `message.send` end to end or rejects those fields before success, with focused regression tests and live Telegram proof.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with text plus `buffer`, `filename`, and `mimeType`; current main accepts those fields but does not hydrate or forward the buffer to Telegram delivery.

## Solution Assessment

Unclear for any one open patch. The maintainable endpoint is a single contract decision and implementation that either honors the advertised inline attachment fields end to end or rejects them before success.
