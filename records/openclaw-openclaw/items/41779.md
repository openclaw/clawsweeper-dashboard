## Summary

Current main still has the reported schema/runtime mismatch: `message.send` advertises inline attachment fields, but the outbound and Telegram paths only route URL/path media. This should stay open because active implementation candidates at https://github.com/openclaw/openclaw/pull/42674 and https://github.com/openclaw/openclaw/pull/63578 already cover overlapping parts of the fix and need maintainer consolidation.

## What This Changes

The issue asks OpenClaw to stop returning success when Telegram `message.send` is called with inline attachment fields that are not delivered.

## Best Possible Solution

Keep this issue open until maintainers choose and land one canonical path: either support inline Telegram/plugin-channel attachments through `send` with media limits and live proof, or reject unsupported inline fields before reporting success.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with `message`, `buffer`, `filename`, and `mimeType`; current main accepts the fields in the schema, skips send-action hydration, and reaches Telegram with only URL/path media support.

## Solution Assessment

Unclear. Inline attachment support matches the advertised schema, while explicit rejection is the narrower false-success fix; the best current path is maintainer consolidation of the existing fix candidates rather than another duplicate repair lane.
