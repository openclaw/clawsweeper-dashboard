## Summary

Keep this open. Current main still matches the reported schema/runtime mismatch: `message.send` advertises inline attachment fields, but shared hydration skips `send`, the shared send runner only forwards URL/path media, and the Telegram plugin/transport path has no inline buffer option. The linked open PRs are implementation candidates, not a reason to close the issue before one canonical fix lands.

## What This Changes

The issue asks Telegram `message.send` to deliver or explicitly reject inline attachment fields such as `buffer`, `filename`, and MIME type instead of returning `ok=true` while dropping them.

## Best Possible Solution

Keep this issue open and land or replace #42674/#63578 with a focused fix that either hydrates inline attachment fields through `message.send` into Telegram media delivery or rejects those fields explicitly for `send`, with regression coverage across the shared outbound runner and Telegram plugin path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main accepts `buffer`/`filename` in the `message.send` schema, skips hydration for `send`, and reaches Telegram with only URL/path media fields.

## Solution Assessment

Unclear. Both supported delivery and explicit rejection would satisfy the report, but the best current path is one maintainer-reviewed canonical PR rather than another duplicate implementation.
