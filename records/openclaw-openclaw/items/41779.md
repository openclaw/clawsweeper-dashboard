- `impact:message-loss`: The reported path can silently drop the intended outbound attachment while returning a successful message result.

## Summary

Keep this open: current main and the latest release still advertise `buffer`/`filename` on `message.send`, while the send hydration and Telegram runtime paths only carry media URL/path-style inputs, so the false-success attachment drop remains source-reproducible.

## What This Changes

The issue requests that Telegram `message.send` either deliver advertised inline `buffer`/`filename` attachments or fail instead of returning `ok=true` while dropping them.

## Best Possible Solution

Land one consolidated contract fix that either carries `buffer`/`filename` through Telegram `message.send` end to end or rejects unsupported inline attachment fields before returning success.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with text plus `buffer`, `filename`, and `mimeType`; current main accepts those fields but does not hydrate or forward the buffer to Telegram delivery.

## Solution Assessment

Unclear for any one open patch. The maintainable endpoint is a single contract decision and implementation that either honors the advertised inline attachment fields end to end or rejects them before success.
