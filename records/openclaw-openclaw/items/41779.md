## Label Justifications

- `P2`: This is a valid channel message-delivery bug with a limited attachment-send blast radius and open implementation candidates.
- `impact:message-loss`: The reported behavior can silently drop the intended attachment while returning a successful message result.

## Summary

Keep this open: current main and the latest shipped tag still advertise inline attachment fields on `message.send`, while the shared send path and Telegram runtime only forward URL/path media, so `buffer`/`filename` can still be accepted without producing the expected attachment. Open PRs already target the mismatch, but none has merged.

## What This Changes

Telegram `message.send` with `buffer`, `filename`, and `mimeType` should either deliver a usable attachment or fail instead of returning a successful text-only send.

## Best Possible Solution

Land one consolidated boundary-correct fix that either hydrates inline buffer attachments through the shared outbound/media and Telegram plugin path, or rejects unsupported inline fields before any send can report success.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with text plus `buffer`, `filename`, and `mimeType`; current main accepts those fields but only hydrates and forwards URL/path media.

## Solution Assessment

Unclear. Both honoring the advertised buffer delivery and explicitly rejecting unsupported inline fields would fix the false-success bug; the maintainable next step is to choose and land one consolidated implementation.
