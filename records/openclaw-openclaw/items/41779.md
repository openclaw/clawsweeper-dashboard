## Summary

Keep this open: current main still advertises inline attachment fields on `message.send`, but the send runner and Telegram runtime do not hydrate or forward direct `buffer`/`filename` data. Open PRs already cover the central and adjacent attachment paths, so this issue should remain the canonical tracker until one consolidated fix lands.

## What This Changes

The issue reports that Telegram `message.send` accepts `buffer`, `filename`, and MIME attachment inputs but silently drops them while returning `ok: true`.

## Best Possible Solution

Review and land one consolidated fix that either delivers direct `buffer`/`filename` through the Telegram send path or rejects unsupported inline fields before returning success.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with text plus `buffer`, `filename`, and `mimeType`; current main accepts those fields but does not hydrate or forward the buffer to Telegram delivery.

## Solution Assessment

Unclear for any one open patch. The maintainable endpoint is a single contract decision and implementation that either honors the advertised inline attachment fields end to end or rejects them before success.
