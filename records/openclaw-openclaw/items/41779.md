## Label Justifications

- `P2`: The original bug caused a normal attachment-send workflow to report success while dropping the user-visible file.
- `impact:message-loss`: The affected behavior silently lost outbound attachment content while returning a successful message result.

## Summary

Current main and the latest release now materialize buffer-only `message.send` attachments into normal outbound media paths before Telegram delivery, so the reported false-success/drop path is fixed.

## What This Changes

Telegram `message.send` was reported to accept `buffer`/`filename`/MIME inputs but drop the attachment while returning `ok=true`.

## Best Possible Solution

Keep the shipped generic outbound buffer materialization path; any remaining provider-neutral attachment design should stay in narrower current follow-ups.

## Reproduction Assessment

Yes for the historical bug: the issue payload matches the schema/runtime mismatch, and current source shows the previous missing `send` hydration path is now present. I did not establish a current-main failure because the relevant source path now normalizes the buffer before dispatch.

## Solution Assessment

Yes. The shipped fix solves this at the generic outbound boundary by converting `buffer`/metadata into existing media paths, so Telegram can keep using its normal `mediaUrl` delivery path instead of gaining a one-off buffer contract.
