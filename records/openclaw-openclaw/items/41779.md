## Summary

Keep this open: current main and the latest release still accept inline attachment fields on `message.send`, but the send path does not hydrate or forward `buffer`/`filename` to Telegram delivery. Related open PRs cover overlapping attachment paths, so this needs maintainer consolidation rather than closure.

## What This Changes

The issue asks OpenClaw to either deliver Telegram `message.send` inline `buffer`/`filename` attachments or reject unsupported fields instead of returning success while dropping the attachment.

## Best Possible Solution

Land one consolidated contract fix that either carries inline `buffer`/`filename` through Telegram `message.send` end to end or rejects unsupported inline attachment fields before returning success.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with text plus `buffer`, `filename`, and `mimeType`; current main accepts those fields but does not hydrate or forward the buffer to Telegram delivery.

## Solution Assessment

Unclear for any one open patch. The maintainable endpoint is a single contract decision and implementation that either honors the advertised inline attachment fields end to end or rejects them before success.
