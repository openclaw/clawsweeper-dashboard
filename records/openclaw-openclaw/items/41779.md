## Summary

Keep this open: current main and v2026.5.22 still expose inline attachment fields on `message.send`, but the send hydration and Telegram dispatch paths do not carry direct `buffer`/`filename` through to delivery. Related open PRs cover overlapping attachment paths, so the next step is maintainer consolidation rather than closure.

## What This Changes

The issue asks OpenClaw to either deliver Telegram attachments for `message.send` `buffer`/`filename`/MIME inputs or reject those unsupported fields before returning success.

## Best Possible Solution

Land one consolidated contract fix that either hydrates and forwards inline `buffer`/`filename` through Telegram `message.send` end to end or rejects those fields before success, with focused regression tests and live Telegram proof.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with text plus `buffer`, `filename`, and `mimeType`; current main accepts those fields but does not hydrate or forward the buffer to Telegram delivery.

## Solution Assessment

Unclear for any one open patch. The maintainable endpoint is a single contract decision and implementation that either honors the advertised inline attachment fields end to end or rejects them before success.
