## Summary

Keep open: current main and v2026.5.18 still have the advertised `message.send` buffer fields without a Telegram delivery path for direct buffers, and the latest related URL fan-out PR explicitly leaves buffer/filename hydration out of scope.

## What This Changes

The issue reports that Telegram `message.send` accepts `buffer`, `filename`, and MIME fields but can ignore them while still returning `ok: true`.

## Best Possible Solution

Land one consolidated fix that either honors the advertised buffer attachment contract through the actual Telegram delivery boundary or rejects unsupported inline fields before any successful send result is returned.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `message.send` accepts `buffer`/`filename`/MIME fields, but current main excludes ordinary `send` from attachment hydration and Telegram only forwards `mediaUrl`.

## Solution Assessment

Unclear until maintainer consolidation. Supporting the advertised buffer delivery is the cleanest contract match, but an explicit rejection before success would also fix the false-success bug if maintainers do not want inline buffers on `send`.
