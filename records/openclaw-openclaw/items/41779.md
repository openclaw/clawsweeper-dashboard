## Summary

Keep this open: current main still exposes inline attachment fields on `message.send`, but the ordinary send path and Telegram runtime do not hydrate or forward direct `buffer`/`filename` data. The related open PRs cover overlapping or narrower pieces, so the issue should remain the canonical tracking point until one consolidated fix lands.

## What This Changes

The issue asks for Telegram `message.send` to either deliver advertised inline `buffer`/`filename` attachments or reject those fields instead of returning success while dropping the attachment.

## Best Possible Solution

Land one consolidated boundary-correct fix that either preserves inline buffers through the actual Telegram delivery runtime or rejects unsupported inline fields before a send can report success.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with text plus `buffer`, `filename`, and `mimeType`; current main accepts those fields but does not hydrate or forward the direct buffer to Telegram delivery.

## Solution Assessment

Not applicable to a specific patch yet. The maintainable fix is a single shared outbound/Telegram boundary change that either honors the advertised inline attachment fields or rejects them explicitly before success.
