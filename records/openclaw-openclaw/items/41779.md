## Summary

Keep this open: current main still advertises `buffer`/`filename` on `message.send`, but the ordinary send path neither hydrates nor forwards direct inline attachment data for Telegram, so the false-success bug remains valid.

## What This Changes

The issue requests that Telegram `message.send` either deliver `buffer`/`filename` attachments advertised by the schema or fail explicitly instead of returning success while dropping them.

## Best Possible Solution

Keep this as the canonical tracker until one consolidated fix lands that either safely supports inline buffer attachments through the correct Telegram/plugin boundary or rejects unsupported fields loudly, with regression coverage and redacted real Telegram or plugin-channel proof.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call Telegram `message.send` with `message`, `buffer`, `filename`, and `mimeType`; current main accepts those fields but only builds and forwards URL/path media.

## Solution Assessment

Unclear. Supporting buffer delivery matches the advertised schema, while explicit rejection would also fix the false-success behavior; the safest path is to consolidate the overlapping PRs into one boundary-correct fix.
