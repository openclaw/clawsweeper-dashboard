## Summary

Keep open: the central UX fix is useful, but the patch can hide plaintext stored in config env values and still lacks external real behavior proof.

## What This Changes

The PR adds a `MIGRATION_RECOMMENDED` secrets-audit finding/count, CLI/docs output, and regression tests for exact `${ENV_VAR}` strings on SecretRef-capable config fields.

## Best Possible Solution

Keep the UX improvement, but make audit source-aware so exact env templates become migration guidance only when the resolved value is not sourced from plaintext stored in OpenClaw config, then require redacted CLI proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `env.vars.OPENAI_API_KEY` can populate `${OPENAI_API_KEY}` before config substitution, and this PR branch would classify the authored target as `MIGRATION_RECOMMENDED` while top-level config env remains outside the audit target surface.

## Solution Assessment

No. Comparing only the authored target with the resolved target is not the narrowest safe fix because it does not attribute whether the env value came from shell/dotenv versus plaintext stored in OpenClaw config.
