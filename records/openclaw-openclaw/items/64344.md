## Summary

Keep open: the PR targets a real secrets-audit false positive, but the current patch can suppress the only plaintext finding when an exact `${ENV_VAR}` target resolves from plaintext stored in `openclaw.json` `env`/`env.vars`. It also lacks after-fix real behavior proof, so it is not ready for merge.

## What This Changes

The PR adds a `MIGRATION_RECOMMENDED` secrets-audit finding/count, CLI/docs output, and regression tests for exact `${ENV_VAR}` strings on SecretRef-capable config fields.

## Best Possible Solution

Keep the UX improvement, but only emit migration guidance after the audit can attribute whether the resolved value came from shell/dotenv versus plaintext stored in OpenClaw config.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path with `env.vars.OPENAI_API_KEY` containing a secret and `models.providers.openai.apiKey` set to `${OPENAI_API_KEY}`; the config reader applies `env.vars` before substitution, and the PR branch would classify the target as migration guidance.

## Solution Assessment

No. Comparing only the authored and resolved target value is not the narrowest safe fix because it loses whether the env value came from shell/dotenv or plaintext stored in the same config file.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Preserve plaintext findings for config env templates:** `src/secrets/audit.ts:266-286`
