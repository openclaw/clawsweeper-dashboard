## Summary

Keep open: the PR addresses a real secrets-audit false positive, but it is not merge-ready because real behavior proof is still missing and the patch can downgrade a plaintext-secret finding when an env template resolves from `config.env` or `env.vars`.

## What This Changes

The branch adds a `MIGRATION_RECOMMENDED` secrets-audit finding/count, CLI/docs output, and regression tests for exact env interpolation on SecretRef-capable config fields.

## Best Possible Solution

Emit migration guidance only when audit can prove the resolved value did not come from plaintext stored in OpenClaw config, while preserving plaintext findings and coordinating the source-provenance contract with https://github.com/openclaw/openclaw/issues/62438.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path. Current config reads apply `config.env` before substitution, and the PR classifies the authored exact template as migration guidance even when that env value came from plaintext config storage.

## Solution Assessment

No. Comparing only the authored target string with the resolved target is not source-aware enough; the safer fix must distinguish shell/dotenv env values from plaintext values injected by OpenClaw config.
