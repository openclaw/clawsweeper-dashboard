## Summary

Keep this PR open, but the current patch is not ready to merge. It fixes a real audit UX problem, yet the reclassification is too broad and can suppress the only plaintext finding when an exact env template is backed by plaintext stored in `config.env` or `env.vars` inside `openclaw.json`.

## What This Changes

The PR adds a `MIGRATION_RECOMMENDED` secrets-audit code/count, CLI/docs output, and tests so exact `${ENV_VAR}` strings on SecretRef-capable config fields are reported as migration guidance instead of `PLAINTEXT_FOUND`.

## Best Possible Solution

Keep the focused UX improvement, but revise the audit logic to preserve a plaintext finding or emit a distinct source-specific finding when an exact env template resolves from plaintext stored in `config.env` or `env.vars`; only shell/runtime/dotenv-backed exact templates should become migration guidance. Add regression coverage for both config env shapes and a user-facing changelog entry before merge.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `openclaw.json` with `env.vars.OPENAI_API_KEY: "sk..."` and `models.providers.openai.apiKey: "${OPENAI_API_KEY}"` while the host env lacks `OPENAI_API_KEY`; current config reading applies `config.env` before substitution, and the PR branch would classify the authored template as migration guidance instead of plaintext.

## Solution Assessment

No. The proposed approach compares authored and resolved target values but does not attribute where the env value came from, so it is not the narrowest safe fix for a security audit path.
