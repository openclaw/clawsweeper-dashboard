## Summary

Keep open: this is a focused bug-fix PR, but the current patch can suppress a real plaintext audit finding when an exact `${ENV_VAR}` target resolves from plaintext stored in OpenClaw config, and the external real-behavior proof gate is still unmet.

## What This Changes

The PR adds a `MIGRATION_RECOMMENDED` secrets-audit code/count, CLI/docs output, and regression tests for exact `${ENV_VAR}` strings on SecretRef-capable config fields.

## Best Possible Solution

Classify exact authored env templates only when audit can prove the resolved value did not come from plaintext stored in OpenClaw config, then include redacted CLI proof for the real `openclaw secrets audit` path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `config.env` is applied before `${VAR}` substitution, and the PR then classifies the authored exact template as `MIGRATION_RECOMMENDED` even when the resolved value came from config-owned plaintext.

## Solution Assessment

No. Comparing only the authored target with the resolved target is not source-aware enough; the safer fix must distinguish shell or dotenv env values from plaintext values injected by OpenClaw config, or explicitly report that config-owned plaintext source.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve plaintext findings for config env templates:** `src/secrets/audit.ts:277-286`

If `openclaw.json` also stores `env.vars.OPENAI_API_KEY` as the real secret, config reads apply that value before substitution and `config.env` is not otherwise audited, so the only plaintext finding is downgraded.
