## Summary

Keep open: current main has not implemented this audit distinction, and the PR still has a blocking audit-safety issue because config-stored plaintext can be downgraded to migration guidance through an authored `${ENV_VAR}` template. The external PR also lacks after-fix real behavior proof for the CLI path.

## What This Changes

The branch adds a `MIGRATION_RECOMMENDED` secrets-audit code/count, CLI/docs output, and regression tests for exact `${ENV_VAR}` strings on SecretRef-capable config fields.

## Best Possible Solution

Keep the UX improvement only after the audit can prove the resolved env value did not come from plaintext stored in OpenClaw config, then align the result with the related env-ref audit work.

## Reproduction Assessment

Yes. Source inspection shows `config.env` is applied before `${VAR}` substitution, and the PR would then classify the authored exact template as `MIGRATION_RECOMMENDED` instead of preserving the plaintext finding.

## Solution Assessment

No. Comparing only the authored target with the resolved target is not source-aware enough; the safer fix must distinguish shell/dotenv env values from plaintext values injected by OpenClaw config.
