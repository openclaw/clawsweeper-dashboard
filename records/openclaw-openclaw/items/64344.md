## Summary

Keep open: the PR addresses a real audit UX bug, but the proposed classification can suppress the only plaintext finding when the referenced env value is itself stored in `openclaw.json` under `env` or `env.vars`.

## What This Changes

The PR adds a `MIGRATION_RECOMMENDED` secrets-audit code/count, CLI/docs output, and tests so exact `${ENV_VAR}` strings on SecretRef-capable config fields are reported as migration guidance instead of `PLAINTEXT_FOUND`.

## Best Possible Solution

Keep the focused UX improvement, but only emit migration guidance after attributing the env value source; plaintext stored in config `env` or `env.vars` should still produce a plaintext finding or a distinct source-specific finding, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an `openclaw.json` with `env.vars.OPENAI_API_KEY: "sk..."` and `models.providers.openai.apiKey: "${OPENAI_API_KEY}"` while the host env lacks that key; current config reading applies `config.env` before substitution, and the PR branch would classify the authored template as migration guidance.

## Solution Assessment

No. Comparing only the authored and resolved target value is not the narrowest safe fix because it loses whether the env value came from shell/dotenv or from plaintext stored in the same config file.
