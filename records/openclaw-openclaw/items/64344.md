## Summary

Keep open: the PR addresses a real secrets-audit false positive, but the current patch can suppress the only plaintext finding when an exact `${VAR}` target resolves from plaintext stored in top-level config `env`/`env.vars`; it also lacks after-fix real behavior proof.

## What This Changes

The PR adds a `MIGRATION_RECOMMENDED` secrets-audit finding/count, CLI/docs output, and regression tests for exact `${ENV_VAR}` strings on SecretRef-capable config fields.

## Best Possible Solution

Keep the UX improvement, but make secrets audit source-aware so exact env templates become migration guidance only when the resolved value is not sourced from plaintext stored in OpenClaw config, then add regression coverage and redacted CLI proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `env.vars.OPENAI_API_KEY` can populate `${OPENAI_API_KEY}` before config substitution, and the PR branch would classify the authored target as `MIGRATION_RECOMMENDED` while top-level config env remains outside the audit target surface.

## Solution Assessment

No. Comparing only the authored target with the resolved target is not the narrowest safe fix because it does not attribute whether the env value came from shell/dotenv versus plaintext stored in OpenClaw config.
