## Summary

Keep open: current main still has the unsafe-reindex/sqlite-vec state mismatch, and the latest release snapshot also still drops `chunks_vec` while later sync setup can prepare vector DELETE statements against the missing table. The related Windows `EBUSY` and sqlite-vec loader/status threads are adjacent but do not supersede this narrower crash.

## What This Changes

The issue reports that env-gated unsafe memory reindex drops `chunks_vec` while `vector.available` remains true, causing sqlite-vec DELETE preparation to crash before the vector table is recreated.

## Best Possible Solution

Repair unsafe reindex so vector table existence and vector availability/readiness stay consistent before sync setup prepares vector DELETE statements, with focused regression coverage and the Windows safe-reindex fallback kept separate.

## Reproduction Assessment

Yes, at source level: the reporter’s env-gated command maps to current main’s `runUnsafeReindex()` path, which drops `chunks_vec` while keeping vector availability true and then prepares `DELETE FROM chunks_vec` before table recreation. I did not run a live Windows/sqlite-vec command in this read-only review.

## Solution Assessment

Mostly yes: the right fix boundary is the unsafe reindex/vector-table state handling in memory-core. The proposed row-delete approach is plausible, but the implementation should first validate sqlite-vec virtual table behavior and avoid mixing in the separate Windows atomic-swap fallback.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/70559. On current main, `OPENCLAW_TEST_FAST=1` plus `OPENCLAW_TEST_MEMORY_UNSAFE_REINDEX=1` makes forced memory sync call `runUnsafeReindex()`. With sqlite-vec loaded, `resetIndex()` drops `chunks_vec` but leaves `this.vector.available` true, so `syncMemoryFiles()`/`syncSessionFiles()` prepare `DELETE FROM chunks_vec ...` before `indexFile()` can recreate the table.
