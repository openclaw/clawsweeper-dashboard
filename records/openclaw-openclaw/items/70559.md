## Summary

Keep open: current main still has the unsafe-reindex/sqlite-vec state mismatch, where `resetIndex()` drops `chunks_vec` while vector availability remains true and later sync setup can prepare `DELETE FROM chunks_vec` before the table is recreated. The adjacent Windows rename and sqlite-vec loader/status reports do not supersede this narrower crash.

## What This Changes

The issue asks OpenClaw to make the env-gated unsafe memory reindex path work when sqlite-vec vector indexing is enabled.

## Best Possible Solution

Repair the unsafe reindex vector reset path so vector availability/readiness and `chunks_vec` schema existence stay consistent before sync setup prepares vector DELETE statements, with focused regression coverage and the Windows safe-reindex fallback kept separate.

## Reproduction Assessment

Yes, at source level: the env-gated forced reindex path drops `chunks_vec`, leaves `this.vector.available` true, and then prepares vector DELETE statements before table recreation. I did not run a live Windows/sqlite-vec command in this read-only review.

## Solution Assessment

Yes, the best fix boundary is the memory-core unsafe reindex/vector-table state handling. The reporter's row-delete approach is plausible, but the implementation should validate sqlite-vec virtual table semantics and avoid folding in the separate Windows atomic-swap fallback.
