Keep open: current main and the latest release still have the unsafe-reindex/sqlite-vec state mismatch, and the related Windows EBUSY fix and adjacent reindex-recovery PR do not supersede this narrower crash.

## What This Changes

The issue reports that the env-gated unsafe memory reindex path drops `chunks_vec` and then prepares vector DELETE statements while sqlite-vec still appears available.

## Best Possible Solution

Repair the unsafe reindex vector reset path so vector availability/readiness and `chunks_vec` schema existence stay consistent before sync setup prepares vector DELETE statements, with focused regression coverage and the Windows safe-reindex fallback kept separate.

Yes, at source level: the env-gated forced reindex path drops `chunks_vec`, leaves `this.vector.available` true, and then prepares vector DELETE statements before table recreation. I did not run a live Windows/sqlite-vec command in this read-only review.

## Solution Assessment

Yes, the best fix boundary is the memory-core unsafe reindex/vector-table state handling. The reporter's row-delete approach is plausible, but the implementation should validate sqlite-vec virtual-table semantics and avoid folding in the separate Windows atomic-swap fallback.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/70559 in memory-core. Current main's env-gated unsafe reindex path calls `resetIndex()`, which drops `chunks_vec` and clears only `vector.dims`; because `vector.available` can remain true, `syncMemoryFiles()` and `syncSessionFiles()` can prepare `DELETE FROM chunks_vec ...` before `ensureVectorReady(sample.length)` recreates the table. Keep the fix scoped to `extensions/memory-core/src/memory/manager-sync-ops.ts` and focused tests. Make the vector reset path internally consistent, for example by preserving/deleting vector rows when sqlite-vec is available or by resetting readiness/availability so no statement is prepared against a missing table. Validate dimension-change behavior and missing-table fallback. Do not fold in the separate Windows safe-reindex atomic-swap fallback and do not attempt the broader rollback/cache work from https://github.com/openclaw/openclaw/pull/59137. Add a user-facing changelog entry if the maintainer lane expects one for this fix.
