## Summary

Keep open. Current main and the latest release still treat generic polluted auto-fallback origin rows as active stored overrides, and the open linked fix PR is the right path to review rather than closing this issue now.

## What This Changes

The issue reports that auto fallback session origin metadata can be polluted so the primary snap-back probe never runs and sessions stay pinned to a fallback model/provider.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92819, or an equivalent fix, so stale auto-fallback origin rows are classified consistently and cleared safely at the run boundary without overwriting newer user or session state.

## Reproduction Assessment

Yes, source-reproducible: an auto fallback session entry with fallback override fields but origin fields that do not match the current primary makes resolveAutoFallbackPrimaryProbe return undefined before the primary retry path can run. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

No, current main is not yet the best end state because generic polluted-origin rows remain active stored overrides. The best fix is the linked PR or equivalent source change that classifies stale origins and clears/persists them with concurrency guards.
