## Summary

Keep open. Current main and the latest stable release still honor auto-created session model pins whose recorded fallback origin differs from the current primary on ordinary turns, while the primary-probe resolver returns no probe for that state.

## What This Changes

The issue requests fixing auto fallback session pins whose polluted fallback-origin fields prevent recovery to the configured primary model.

## Best Possible Solution

Add a narrow session-state repair so auto-owned mismatched-origin fallback pins are treated as stale or recoverable on normal, direct-agent, and queued follow-up paths without changing user overrides or the five-minute probe cadence.

## Reproduction Assessment

Yes. Source inspection shows an auto session entry with provider/model overrides and origin fields that differ from the current primary is honored as the stored model, while `resolveAutoFallbackPrimaryProbe` returns `undefined`.

## Solution Assessment

Yes, if scoped narrowly. The best fix is a focused cleanup or probe repair for auto-owned mismatched-origin pins, not a broad new fallback policy or config surface.
