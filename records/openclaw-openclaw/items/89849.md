## Summary

## What This Changes

The PR removes Node readline from the iMessage RPC client, buffers stdout until LF delimiters, and adds focused U+2028 parse-fragment regression tests.

## Best Possible Solution

Land the LF-only iMessage RPC stdout parser after redacted live or real-environment evidence shows a U+2028-containing iMessage response is received without parse-error flooding or pending-request timeout.

## Reproduction Assessment

Yes at source/dependency level: current main and v2026.5.28 use Node readline on `imsg rpc` stdout, and Node's readline source plus a local probe show U+2028 splits a JSON string before `handleLine` can parse it. I did not establish a live iMessage reproduction in this checkout.

## Solution Assessment

Yes: replacing only the iMessage RPC stdout framing with an LF-only buffer matches the documented one-JSON-object-per-line contract and an existing stdio JSON-RPC sibling. The remaining gap is proof of the live iMessage path, not a better code location.
