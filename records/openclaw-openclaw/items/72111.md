## Summary

Keep open: current main still silently resolves a mixed MCP `command` plus `url` server as stdio before the `streamable-http` branch can run, while the docs say that shape should be rejected. The open closing PR remains the right path until it lands or is superseded.

## What This Changes

Fix mixed MCP server configs that set both `command` and `url` for `streamable-http` so they do not silently fall back to stdio and time out.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/72515 so mixed MCP transport configs fail fast with a redacted diagnostic and regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, at source level: a server config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio before HTTP is considered. I did not run the reporter's live OpenSpace setup in this read-only review.

## Solution Assessment

Yes. Rejecting the ambiguous shape in `resolveMcpTransportConfig` is the narrowest maintainable fix because that seam sees both transport fields and the docs already require one transport per server.
