## Summary

Keep open: current main still resolves a mixed non-empty `command` plus `url` MCP server as stdio before the HTTP branch, and the closing PR #72515 remains open and unmerged.

## What This Changes

The issue reports a spawned MCP server configured with `transport: "streamable-http"` and a URL timing out after OpenClaw starts the child process.

## Best Possible Solution

Land or revise PR #72515 so mixed `command` plus `url` MCP configs fail fast with redacted diagnostics and tests aligned to the documented config contract.

## Reproduction Assessment

Yes. A config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` follows the current stdio-first resolver path and can reach the existing 30000ms connection timeout.

## Solution Assessment

Yes, if maintainers agree with the existing docs contract. Early rejection is narrower and safer than adding a new managed HTTP child-process lifecycle for a shape the docs already say is invalid.
