## Summary

Keep open. Current main still resolves a mixed `command` plus non-empty `url` MCP server through the stdio path before HTTP handling, so the reported timeout remains plausible; the linked closing PR is open and unmerged, so this issue should stay open until that PR lands or maintainers choose a broader managed spawned-HTTP lifecycle design.

## What This Changes

Report that an MCP server configured with `command`, `url`, and `transport: "streamable-http"` starts its HTTP process but OpenClaw times out after 30000ms instead of connecting or clearly rejecting the mixed transport config.

## Best Possible Solution

Make MCP transport resolution unambiguous: reject non-empty mixed `command` plus `url` configs with a clear redacted diagnostic and regression coverage, unless maintainers intentionally add a documented managed spawned-HTTP-server mode.

## Reproduction Assessment

Yes. A config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` can be traced on current main into the stdio resolver branch before the HTTP branch, which can then hit the existing 30000ms connection timeout when the child process only serves HTTP.

## Solution Assessment

Unclear. Early rejection is the narrowest maintainable fix and matches current docs, but it is only the best solution if OpenClaw does not intend to manage spawned HTTP server processes for MCP.
