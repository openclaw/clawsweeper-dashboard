## Summary

Keep open: current main still chooses the stdio path before HTTP for mixed `command` plus `url` MCP configs, and the open closing PR #72515 has not landed yet.

## What This Changes

The report asks OpenClaw to handle or clearly reject MCP configs that combine a spawned command with `transport: "streamable-http"` and `url` instead of timing out after startup.

## Best Possible Solution

Align source with the documented registry contract: reject non-empty `command` plus `url` configs at transport resolution with redacted diagnostics and tests, unless maintainers explicitly design managed HTTP child-process support.

## Reproduction Assessment

Yes. The reported config has non-empty `command`, `url`, and `transport: "streamable-http"`; current main resolves stdio first, ignores the URL, and can then hit the existing 30000ms connection timeout.

## Solution Assessment

Unclear until maintainer product intent is confirmed. Early rejection is the narrowest maintainable fix and matches current docs, while spawned HTTP lifecycle support would be a larger runtime design.
