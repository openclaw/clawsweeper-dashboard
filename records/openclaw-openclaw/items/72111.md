## Summary

Keep open: current main still resolves mixed non-empty `command` plus `url` MCP server configs as stdio before HTTP, which can lead to the reported 30000ms timeout; the linked closing PR #72515 is open and unmerged, so this issue should wait for that PR or an equivalent maintainer-approved fix.

## What This Changes

The issue reports that an MCP server configured with both a spawned command and `transport: "streamable-http"` plus `url` starts listening but OpenClaw times out after 30000ms instead of connecting clearly or rejecting the unsupported mixed shape.

## Best Possible Solution

Review and land or revise PR #72515, or an equivalent change, so mixed `command` plus `url` configs fail fast with redacted diagnostics and focused resolver tests aligned to the documented MCP config contract.

## Reproduction Assessment

Yes. A non-empty `command`, non-empty `url`, and `transport: "streamable-http"` still follows the resolver's stdio-first branch on current main, then can hit the existing `MCP server connection timed out after 30000ms` path.

## Solution Assessment

Yes for the documented contract: early rejection is the narrowest maintainable fix unless maintainers explicitly decide to support OpenClaw-managed spawned HTTP child processes as a new runtime design.
