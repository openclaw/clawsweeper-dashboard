## Summary

Keep open: current main still resolves a mixed MCP server config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` as stdio before the HTTP branch can run, while the docs say that shape is rejected. The open linked fix PR uses closing syntax for this issue, so this should remain open until that PR lands or is superseded.

## What This Changes

The issue reports that an MCP server configured with both a spawned command and `transport: "streamable-http"` times out instead of connecting to the configured HTTP endpoint.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/72515 so mixed MCP transport configs fail fast with a redacted diagnostic and focused resolver coverage, then close this issue after merge.

## Reproduction Assessment

Yes, at source level: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio before HTTP is considered. I did not run the reporter's live OpenSpace setup in this read-only review.

## Solution Assessment

Yes. Early rejection in `resolveMcpTransportConfig` is the narrowest maintainable fix because that seam sees both transport fields and already returns `null` for skipped servers.
