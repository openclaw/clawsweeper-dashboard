## Summary

Keep open: current main still resolves mixed `command` plus `url` MCP server entries through the stdio path before HTTP handling, and PR #72515 is open with closing syntax for this issue. The issue should remain open until that PR lands or maintainers choose a broader managed HTTP child-process design.

## What This Changes

The issue requests that an MCP `streamable-http` server configured with a spawned command and URL connect successfully instead of timing out after OpenClaw starts the child process.

## Best Possible Solution

Make mixed `command` plus `url` behavior unambiguous at the MCP transport boundary: either reject the ambiguous config with a clear diagnostic and regression tests, or explicitly add and document managed HTTP child-process lifecycle support.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a server config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio on current main and can reach the existing 30000ms connection timeout path.

## Solution Assessment

Unclear. Rejecting mixed `command` plus `url` configs is the narrowest maintainable fix that matches current docs, but if maintainers want OpenClaw to manage spawned HTTP server processes then the better solution is a broader lifecycle design.
