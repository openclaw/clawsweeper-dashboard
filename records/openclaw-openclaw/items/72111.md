Keep open: current main and the latest release still route a mixed MCP `command` plus `url` server through stdio before streamable-http, while the same-repo linked PR remains the focused candidate fix. The reporter's follow-up supports that the adjacent Accept-header issue is fixed, but it does not prove the mixed-config resolver path is fixed.

## What This Changes

The issue reports that an MCP server configured with both a spawned command and a streamable-http URL starts but times out instead of connecting or failing with a clear config diagnostic.

## Best Possible Solution

Land or replace the resolver-level mixed `command`+`url` guard with clear diagnostics and focused tests, then close this issue after the fix merges.

## Reproduction Assessment

Yes, source-level: current main resolves a non-empty `command` before considering `url` or `transport`, so the reported mixed streamable-http config takes the stdio path and can hit the 30-second connect timeout. I did not run the external OpenSpace setup.

## Solution Assessment

Yes, the best fix belongs at `resolveMcpTransportConfig`, where both `command` and `url` are visible before transport construction. A spawn-timeout-only change would leave the silent transport-selection mismatch in place.
