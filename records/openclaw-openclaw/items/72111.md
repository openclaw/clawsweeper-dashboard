## Summary

Keep open: current main still resolves the reported mixed MCP `command` plus `url` config as stdio before HTTP is considered, and the open closing PR should remain the path to resolution until it lands or is superseded.

## What This Changes

The issue reports a streamable-http MCP server timing out when its config includes both a spawned `command` and an HTTP `url`.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/72515 so mixed MCP transport configs fail fast with a redacted diagnostic and focused resolver coverage, then close this issue after merge.

## Reproduction Assessment

Yes, at source level: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio before HTTP is considered. I did not run the reporter's live OpenSpace setup in this read-only review.

## Solution Assessment

Yes. Rejecting the ambiguous shape in `resolveMcpTransportConfig` is the narrowest maintainable fix because that seam sees both transport fields and already returns `null` for skipped servers.
