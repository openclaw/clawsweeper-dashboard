## Summary

Keep open: current main still resolves a mixed MCP `command` plus `url` server as stdio before the `streamable-http` path can run, and the live closing PR is still open rather than merged.

## What This Changes

The issue reports a spawned MCP server configured with `transport: "streamable-http"`, `command`, and `url` timing out after 30000ms even though the HTTP endpoint is listening.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/72515 so mixed MCP transport configs fail fast with a redacted diagnostic and regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, at source level: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio before HTTP is considered on current main. I did not run the reporter's live OpenSpace setup in this read-only review.

## Solution Assessment

Yes. Rejecting the ambiguous shape in `resolveMcpTransportConfig` is the narrowest maintainable fix because that seam sees both transport fields and the docs already require one transport per server.
