## Summary

Keep open: current main still resolves a mixed MCP config with `command`, non-empty `url`, and `transport: "streamable-http"` as stdio before HTTP is considered, so the reported timeout path remains reachable. The linked PR #72515 is open and uses closing syntax, so this issue should stay open until that PR or a maintainer-selected alternative lands.

## What This Changes

The issue requests that OpenClaw handle or clearly reject MCP servers configured with a spawned command plus `streamable-http` URL transport instead of timing out after 30000ms.

## Best Possible Solution

Keep this issue open and resolve the product boundary: either land PR #72515 to reject mixed `command`+`url` configs with a clear diagnostic matching the docs, or replace it with an explicit managed HTTP child-process design if OpenClaw should own that mode.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio on current main and can hit the existing MCP connection timeout path.

## Solution Assessment

Unclear. Rejecting mixed `command`+`url` is the narrowest maintainable fix under the current docs, but if maintainers want spawned HTTP process management then the best solution is a broader lifecycle design rather than only changing resolver validation.
