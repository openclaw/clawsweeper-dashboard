## Summary

Keep open: current main still lets a mixed MCP server config with non-empty `command` and `url` resolve through stdio before HTTP handling, and the provided GitHub context shows open PR #72515 uses closing syntax for this issue. The issue should stay open until that PR lands or maintainers choose a broader managed HTTP child-process design.

## What This Changes

The issue requests reliable handling or diagnostics for MCP server configs that combine a spawned command with `streamable-http` URL transport instead of timing out after 30000ms.

## Best Possible Solution

Make the MCP transport boundary unambiguous: reject mixed `command` plus non-empty `url` configs with a clear diagnostic and regression coverage if current docs remain the contract, or explicitly design and document managed HTTP child-process lifecycle support if that workflow should be supported.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio on current main and can hit the existing 30000ms connection timeout path.

## Solution Assessment

Unclear. Rejecting the mixed config is the narrowest maintainable fix under current docs, but supporting spawned HTTP servers would require a broader lifecycle design rather than a resolver-only change.
