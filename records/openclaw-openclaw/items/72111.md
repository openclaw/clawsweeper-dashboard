## Summary

Keep open: current main and the latest shipped tag still resolve a mixed MCP `command` plus `url` server as stdio before the `streamable-http` branch can run, while the docs say that shape should be rejected. The open closing PR at https://github.com/openclaw/openclaw/pull/72515 is the right path to review and land or supersede.

## What This Changes

The issue asks OpenClaw to avoid a 30-second timeout for an MCP server config that combines `command`, `url`, and `transport: "streamable-http"`.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/72515 so mixed MCP transport configs fail fast with a sanitized diagnostic and regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, at source level: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio before HTTP is considered. I did not run the reporter's live OpenSpace setup.

## Solution Assessment

Yes. Rejecting the ambiguous mixed transport shape in `resolveMcpTransportConfig` is the narrowest maintainable fix because that seam sees both fields and the docs already require one transport per server.
