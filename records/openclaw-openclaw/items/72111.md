## Summary

Keep open: current main still resolves a mixed MCP server config with both non-empty `command` and non-empty `url` as stdio before streamable-http handling can run, which matches the reported timeout path. Open PR #72515 is the active closing implementation candidate, so this issue should stay open until that PR or a maintainer-selected alternative lands.

## What This Changes

The issue reports that MCP `streamable-http` configs combining a spawned `command`/`args` with a `url` hang for 30000ms instead of connecting or producing a clear config diagnostic.

## Best Possible Solution

Keep explicit MCP config semantics: either reject mixed `command` plus `url` definitions early with a clear diagnostic and regression tests, or intentionally add managed HTTP child-process lifecycle support. The silent stdio fallback should not remain the user-visible behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio, then the runtime can hit the existing 30-second MCP connection timeout.

## Solution Assessment

Unclear until maintainers choose the boundary. If the docs are authoritative, PR #72515 is the narrow maintainable fix; if OpenClaw should manage spawned streamable-http servers, that needs a broader lifecycle design instead of a resolver-only change.
