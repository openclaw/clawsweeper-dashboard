## Summary

Keep open: current main still silently resolves mixed non-empty `command` plus non-empty `url` MCP server config as stdio before HTTP handling, which matches the reported 30000ms timeout path, and #72515 is an open unmerged PR with closing syntax for this issue.

## What This Changes

The issue reports that a `streamable-http` MCP server configured with both a spawned command and an HTTP URL starts listening but OpenClaw times out instead of connecting or diagnosing the config.

## Best Possible Solution

Review and land or revise #72515 so mixed `command` plus `url` MCP configs fail fast with redacted diagnostics and focused resolver coverage.

## Reproduction Assessment

Yes for a source-level reproduction: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio first on current main and can hit the existing 30000ms connect timeout path. I did not run a live OpenSpace server because this review was read-only.

## Solution Assessment

Yes. Early rejection of mixed `command` plus `url` is the narrowest maintainable fix because it matches the documented config contract; spawned HTTP subprocess management should be handled as a separate feature if maintainers want it.
