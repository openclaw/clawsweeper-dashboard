## Summary

Keep open: current main still selects stdio before HTTP for mixed `command` plus `url` MCP configs, and the linked closing PR #72515 remains open and unmerged.

## What This Changes

The report asks OpenClaw to handle or reject an MCP server config that combines a spawned command with `transport: "streamable-http"` and a URL instead of timing out after 30 seconds.

## Best Possible Solution

Land or otherwise supersede #72515 with resolver-level rejection of non-empty `command` plus non-empty `url`, matching docs; only design managed HTTP child-process support if maintainers intentionally want that larger feature.

## Reproduction Assessment

Yes. A config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` currently resolves as stdio before HTTP and then reaches the existing 30000ms MCP connection timeout path.

## Solution Assessment

Yes for the documented contract. Early rejection is the narrowest maintainable fix because docs already declare mixed `command`/`url` entries invalid, while managed spawned HTTP lifecycle support would be a separate product design.
