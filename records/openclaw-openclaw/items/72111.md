## Summary

Keep open: current main still silently selects stdio for a mixed `command` plus `url` MCP config, and https://github.com/openclaw/openclaw/pull/72515 references this issue with closing syntax but has not merged.

## What This Changes

The issue asks OpenClaw to handle or clearly diagnose streamable-http MCP configs that combine a spawned command with a URL instead of timing out after 30 seconds.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/72515 so mixed `command` plus `url` configs fail fast with a redacted diagnostic and focused resolver coverage, then close this issue after merge.

## Reproduction Assessment

Yes, at source level: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio before HTTP is considered. I did not run the reporter's live OpenSpace setup in this read-only review.

## Solution Assessment

Yes. Early rejection in the resolver is the narrowest maintainable fix under the documented one-transport-per-server contract; spawned HTTP subprocess lifecycle support would be separate feature work.
