## Summary

Keep open: current main still resolves a mixed `command` plus `url` MCP server through the stdio path before HTTP handling, so the reported 30-second timeout remains plausible. The issue also has an open closing PR, #72515, so it should stay open until that PR lands or maintainers choose a broader managed HTTP child-process design.

## What This Changes

The issue asks OpenClaw to stop timing out when an MCP `streamable-http` server config includes both a spawned `command`/`args` pair and a `url` endpoint.

## Best Possible Solution

Make MCP transport selection unambiguous at the resolver boundary: reject mixed `command` plus non-empty `url` configs with a clear, redacted diagnostic and regression tests, unless maintainers intentionally design and document managed spawned HTTP server lifecycle support.

## Reproduction Assessment

Yes. A config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` is enough to trace the current-main resolver into the stdio branch, which can then hit the existing 30000ms connection timeout when the child process only serves HTTP.

## Solution Assessment

Unclear. Early rejection is the narrowest maintainable fix and matches current docs, but it is only the best solution if OpenClaw does not intend to manage spawned HTTP server processes for MCP.
