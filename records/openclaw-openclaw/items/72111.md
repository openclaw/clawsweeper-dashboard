## Summary

Keep open: current main still misroutes a mixed MCP config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` through the stdio path before HTTP transport resolution can run. The linked open PR #72515 is the active closing candidate, so the issue should remain open until that PR or a maintainer-selected alternative lands.

## What This Changes

The issue asks OpenClaw to handle an MCP streamable-http configuration that launches a server process and connects to its HTTP endpoint without timing out after spawn.

## Best Possible Solution

Keep MCP config semantics explicit: either reject non-empty `command` plus non-empty `url` with a clear diagnostic and resolver regression tests, matching current docs, or intentionally design managed HTTP child-process lifecycle support as a broader feature.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio on current main, and the runtime still has the reported 30-second MCP connection timeout path.

## Solution Assessment

Unclear as a product boundary. If the docs are authoritative, PR #72515 is the narrow maintainable fix; if OpenClaw should manage spawned streamable-http child processes, that needs a separate lifecycle design rather than a resolver-only change.
