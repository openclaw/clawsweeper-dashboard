## Summary

Keep open: current main still selects stdio for a mixed MCP `command` plus `url` config before `streamable-http` is considered, and the open closing PR remains the right resolution path until it lands or is superseded.

## What This Changes

The issue asks OpenClaw to handle or diagnose MCP `streamable-http` configs that combine a spawned command with an HTTP URL and currently time out.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/72515 so mixed MCP transport configs fail fast with a redacted diagnostic and regression coverage, then close this issue after merge.

## Reproduction Assessment

Yes, at source level: a config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` resolves as stdio before HTTP is considered. I did not run the reporter's live OpenSpace setup in this read-only review.

## Solution Assessment

Yes. Rejecting the ambiguous `command` plus `url` shape in `resolveMcpTransportConfig` is the narrowest maintainable fix because that seam sees both transport fields and the docs already define one transport per server.
