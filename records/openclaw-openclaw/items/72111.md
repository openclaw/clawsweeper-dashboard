## Summary

Keep open: current main still resolves a mixed non-empty `command` plus `url` MCP server as stdio before the HTTP branch, and PR #72515 explicitly closes this issue but is still open and unmerged.

## What This Changes

The issue asks OpenClaw to handle `streamable-http` MCP configs that include both a child-process command and an HTTP URL without timing out after spawning the server.

## Best Possible Solution

Land or revise PR #72515 to enforce the documented command/url mutual exclusion with a clear diagnostic, or explicitly design managed spawned-HTTP lifecycle support if maintainers want that config shape.

## Reproduction Assessment

Yes. A config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` follows the stdio-first resolver path; a spawned process that does not speak JSON-RPC over stdio can then reach the existing 30000ms timeout.

## Solution Assessment

Yes, if the documented contract stands: early rejection of the ambiguous mixed shape is the narrowest maintainable fix. Supporting spawned HTTP servers would be a broader runtime design, not the same resolver bug fix.
