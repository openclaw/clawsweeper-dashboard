## Summary

Keep open: current main and v2026.6.10 still accept arbitrary configured model strings structurally and only catch static suppressed model refs or the hook-specific doctor path, while an open linked fork PR already targets the issue and needs maintainer policy on catalog authority and warning-versus-error behavior.

## What This Changes

Requests validation or warnings for configured model refs that do not resolve to provider/catalog evidence, including defaults, per-agent values, and fallbacks.

## Best Possible Solution

Land or replace the linked validation-only PR with a shared analyzer used by config validate and doctor that resolves refs like runtime does, distinguishes trusted catalog absence from custom forward-compatible setups, and covers default, per-agent, fallback, alias, and legacy default-style entries.

## Reproduction Assessment

Yes. Source inspection shows current main accepts arbitrary model strings structurally and only rejects statically suppressed refs or hook-specific catalog misses, so an unsuppressed nonexistent configured model can still pass upfront validation.

## Solution Assessment

No complete solution is on main. The best fix needs maintainer-approved catalog authority and warning/error semantics, and should proceed through the open linked PR or a narrow replacement.
