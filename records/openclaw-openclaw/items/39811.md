## Summary

Keep open: current main and v2026.6.8 still lack the requested generic warning for arbitrary invalid configured model refs. The remaining fix needs maintainer policy for trusted catalog evidence, aliases, custom providers, and forward-compatible model names, and the open linked fork PR is conflicting rather than landed.

## What This Changes

The issue requests config/doctor validation and user-visible warnings for unrecognized configured model names so invalid provider/model routing is caught before silent runtime fallback or failure.

## Best Possible Solution

Land a focused warning-first doctor/config validation path that uses runtime-equivalent model resolution, distinguishes trusted catalog evidence from user-declared rows, covers default/per-agent/fallback refs, and preserves intentional custom or forward-compatible setups.

## Reproduction Assessment

Yes, source-reproducible. Current schema accepts arbitrary model strings, config validation only rejects static manifest suppressions, and doctor only has a catalog-backed check for hooks.gmail.model rather than default/per-agent configured model refs.

## Solution Assessment

Unclear as an implementation path. Advisory validation is the right direction, but maintainers still need to decide trusted catalog semantics and warning behavior for aliases, custom providers, and forward-compatible model refs.
