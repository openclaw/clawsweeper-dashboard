## Summary

Keep open: current main and v2026.6.10 now have fallback visibility, but they still only reject statically suppressed model refs, not arbitrary configured model names against trusted provider/catalog evidence. The downstream `openclaw-extensions doctor` workaround and the still-open conflicting fork PR show the core validation gap remains active.

## What This Changes

The issue asks OpenClaw to flag unverified configured model names and visibly report fallback when invalid primary models fail.

## Best Possible Solution

Add a shared, maintainer-approved model-ref validation analyzer used by `config validate`, doctor, and save/reload paths, with runtime-equivalent alias resolution and clear rules for trusted catalog evidence versus custom or forward-compatible model refs.

## Reproduction Assessment

Yes, source-reproducible: current schema and config validation accept arbitrary model strings unless they match a static suppression, so a nonexistent but unsuppressed ref can still pass config validation. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

No, the current main behavior is only a partial solution: fallback notification is implemented, but generic invalid configured-model validation remains missing. The best fix needs maintainer-approved validation semantics before another autonomous patch is safe.
