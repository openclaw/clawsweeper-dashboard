## Summary

Keep open: current main and v2026.5.22 still only reject statically suppressed model refs, while arbitrary configured provider/model strings and user-declared catalog rows can pass validation; the prior target-repo fix PR was closed unmerged and the discussion includes a later in-the-wild workaround report.

## What This Changes

The issue requests upfront validation or warning for configured model refs that cannot be verified against trusted provider/catalog data, plus visibility when fallback hides a bad primary model.

## Best Possible Solution

Add shared advisory validation for config validate and doctor that resolves refs with runtime-equivalent alias/provider logic, compares against trusted provider/static/runtime catalog evidence while excluding user-declared rows as proof, covers defaults, per-agent refs, fallbacks, hooks, and channel overrides, and preserves intentional custom provider setups.

## Reproduction Assessment

Yes, by source inspection. Current main accepts configured model refs structurally and only rejects static manifest suppressions, while user-declared provider model rows can make a bad ref look present in the loaded catalog.

## Solution Assessment

No current fix is sufficient. The maintainable solution needs maintainer-approved trusted catalog semantics and warning scope before implementation so custom and forward-compatible providers are not broken.
