## Summary

Keep open: current main still accepts arbitrary configured model strings unless they match a static manifest suppression, and the linked fork PR that targets this issue is open but conflicting and unmerged. The downstream doctor workaround reported in the discussion is useful evidence that the core validation gap remains real.

## What This Changes

The issue requests upfront validation or doctor warnings for invalid configured model refs, plus user-visible notice when model fallback hides a primary-model failure.

## Best Possible Solution

Add a shared, maintainer-approved model-ref validation analyzer used by config validation and doctor, with clear trusted-catalog versus custom-model semantics and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current config validation accepts arbitrary model strings unless they match a static manifest suppression, so a nonexistent unsuppressed ref can still pass upfront validation.

## Solution Assessment

No current solution is complete. The best fix needs maintainer-approved validation semantics before another automated patch is safe, especially for custom and forward-compatible model refs.
