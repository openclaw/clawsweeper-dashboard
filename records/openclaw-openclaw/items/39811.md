## Summary

Keep open: current main and v2026.6.6 still lack the requested generic warning for arbitrary invalid configured model refs. The fallback-notification half is implemented, but the remaining validation path needs maintainer policy for trusted catalog evidence, custom providers, aliases, and forward-compatible model names, and an open linked fork PR is still conflicting rather than landed.

## What This Changes

The issue requests model configuration validation or warnings for invalid primary and fallback model refs so bad provider/model choices fail early and visibly instead of only at runtime.

## Best Possible Solution

Land a focused warning-first doctor/config validation path that uses runtime-equivalent model resolution, distinguishes trusted provider/static/discovered catalog evidence from user-declared rows, covers default/per-agent/fallback refs, and preserves custom or forward-compatible setups without fail-closing upgrades unexpectedly.

## Reproduction Assessment

Yes, source-reproducible. Current schema accepts model refs structurally, config validation only rejects manifest-suppressed refs, and current tests still cover unknown models failing later at runtime.

## Solution Assessment

Unclear as an implementation path. Advisory validation is the right direction, but maintainers still need to decide trusted catalog semantics and how warnings should behave for custom, alias, and forward-compatible model refs.
