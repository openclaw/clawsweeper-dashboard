## Summary

Keep open: current main and v2026.6.10 still lack the requested generic validation or doctor warning for arbitrary invalid configured model refs. The fallback-notification half is covered elsewhere, but the remaining config-validation behavior needs maintainer policy for trusted catalogs, aliases, custom providers, and forward-compatible model names.

## What This Changes

The issue requests early config/doctor validation for configured model refs that cannot be verified against provider catalogs, plus visible notification when fallback hides a failed primary model.

## Best Possible Solution

Land a focused warning-first doctor/config validation path that uses runtime-equivalent model resolution, distinguishes trusted catalog evidence from user-declared rows, and covers default, per-agent, and fallback refs without breaking intentional custom setups.

## Reproduction Assessment

Yes, source-reproducible: the schema accepts arbitrary model strings, config validation only rejects static suppressions, and doctor catalog warnings are scoped to hooks.gmail.model rather than default/per-agent/fallback refs.

## Solution Assessment

Unclear as an implementation path. Advisory validation is the right direction, but maintainers still need to choose trusted catalog semantics and warning behavior for aliases, custom providers, and forward-compatible model refs.
