## Summary

Keep open. Current main still validates configured model refs structurally rather than against the runtime/provider catalog, `openclaw config validate` only reports the config snapshot's schema validity, and the direct implementation candidate PR #40355 is still open and unmerged with closing syntax for this issue. The closed fallback-notification report #39818 does not resolve the remaining config-time catalog warning request.

## What This Changes

Request to warn or validate when configured primary or fallback model names are not present in the provider/runtime catalog, and to make fallback from an invalid primary visible to users.

## Best Possible Solution

Keep the issue open until PR #40355 or an equivalent catalog-aware warning change is reviewed and merged. The desired end state is a non-blocking, provider/catalog-aware diagnostic for configured primaries and fallbacks across default and per-agent model surfaces, with fallback visibility handled by the separate fallback-status path.

## Reproduction Assessment

Yes. The static current-main path is high-confidence: a config containing an unknown ref such as `openai/gpt-5.3` satisfies the model schemas, and raw/config CLI validation does not load the catalog before declaring the config valid.

## Solution Assessment

Unclear. A warning-based approach is the maintainable direction because hard rejection would break forward-compatible and custom providers, but the open PR still needs review for full coverage of fallback refs and the expected CLI/doctor warning surface.
