## Summary

Keep open: current main still lacks generic validation for invalid configured default/per-agent model refs; it only rejects statically suppressed model refs and has a narrower doctor check for hooks.gmail.model. The discussion adds useful downstream workaround evidence, and the linked fork PR is open but conflicting and unmerged.

## What This Changes

Request warnings or validation for configured model refs that do not resolve to trusted provider/catalog evidence, plus clearer fallback visibility.

## Best Possible Solution

Add a shared, maintainer-approved model-ref validation analyzer for config validate and doctor that resolves refs like runtime does, distinguishes trusted catalog evidence from user-declared rows, and covers defaults, per-agent refs, fallbacks, and aliases.

## Reproduction Assessment

Yes. Source inspection shows current main accepts arbitrary model strings structurally and only rejects statically suppressed refs, so an unsuppressed nonexistent configured model can still pass upfront validation.

## Solution Assessment

No current solution is complete. The best fix needs maintainer-approved warning/error and trusted-catalog semantics before another automated patch is safe.
