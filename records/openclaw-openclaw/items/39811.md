## Summary

Keep open: current main still validates configured model refs only structurally or against static manifest suppressions, while catalog-backed doctor warnings are scoped to `hooks.gmail.model`. The explicit closing PR is still open, and recent comments add in-the-wild evidence plus a downstream `openclaw-extensions doctor` workaround.

## What This Changes

Warn or validate when configured primary or fallback model refs cannot be verified against trusted provider/catalog evidence instead of accepting arbitrary model strings until runtime.

## Best Possible Solution

Land or replace the linked diagnostic with a focused advisory validation path that follows runtime-equivalent model resolution, distinguishes trusted catalog evidence from user-declared rows, covers primary and fallback refs, and leaves broader fallback-notification behavior to its own tracker.

## Reproduction Assessment

Yes, by source inspection. Current main accepts model refs structurally, only rejects manifest-suppressed refs, and does not provide a generic catalog-backed `config validate` or doctor warning for configured primary/fallback refs.

## Solution Assessment

No, current main is not sufficient. Advisory validation is the right direction, but the final shape needs maintainer-approved catalog-authority semantics and should be coordinated through the linked PR or a focused replacement.
