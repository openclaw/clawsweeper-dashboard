## Summary

Current main still permits configured agent model references to reach runtime resolution without an equivalent early doctor/catalog diagnostic. This issue should remain open while maintainers review the linked implementation and decide how warnings distinguish known-invalid legacy values from intentional custom or forward-compatible models.

Likely related people: @steipete (medium confidence, recent doctor/model runtime work) and Ayaan Zaidi (low confidence, current model-selection history visible in this checkout).

## What This Changes

Warn users early when default, fallback, or per-agent configured model references cannot be resolved, including invalid legacy values such as bare `default`.

## Best Possible Solution

Extend the existing doctor model-status machinery into one shared advisory diagnostic for all configured agent model references, using runtime-equivalent normalization and dynamic catalog preparation, while giving known legacy-invalid values a stronger doctor repair path and leaving intentional custom models supported.

## Reproduction Assessment

Yes at source level: current tests prove unresolved model references can reach an `Unknown model` runtime result, and current doctor checks do not cover agent defaults and overrides. This read-only review did not run a live gateway scenario.

## Solution Assessment

Partly. Advisory doctor warnings are the safest direction, but the best implementation should reuse existing model-status and runtime-resolution machinery, cover every configured agent model surface, and avoid treating catalog absence as authoritative for custom or forward-compatible models.
