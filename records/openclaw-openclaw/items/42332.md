## Label Justifications

- `impact:session-state`: The reported failure is overlapping turns for the same logical sessionKey, which can duplicate work and drift session or agent state.

## Summary

Keep open: current main and the latest release still accept repeated direct or mapped webhook agent deliveries for the same resolved sessionKey when the caller does not reuse an idempotency key. The gap is real, but the remedy needs maintainer policy because a default 409 response would change the public webhook contract.

## What This Changes

Add a webhook delivery overlap guard keyed by resolved sessionKey so recurring webhook sources can avoid concurrent agent turns for the same logical session.

## Best Possible Solution

Define a documented, compatibility-safe overlap policy for resolved webhook sessionKey values, then enforce it consistently for direct and mapped hooks with regression coverage and docs.

## Reproduction Assessment

Yes, source-level: direct or mapped webhook POSTs with the same allowed sessionKey and no reused idempotency key can pass replay checks, dispatch a fire-and-forget run, and receive 200 while the earlier async run can still be active.

## Solution Assessment

Unclear as proposed: returning 409 is plausible, but maintainers should first choose whether overlap handling is default, opt-in, queued, or status-reporting because this changes the public webhook contract.
