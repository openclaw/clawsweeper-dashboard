## Label Justifications

- `P2`: Recurring deliveries can accumulate duplicate queued LLM turns and resource pressure, but the impact is limited to webhook integrations that reuse a session key without idempotency keys.
- `impact:session-state`: The requested behavior governs admission of active and queued turns associated with the same session identity.

## Summary

Current main and v2026.6.5 still accept every non-idempotent webhook delivery, then serialize same-session turns by queueing them; the requested skip/reject policy remains useful, but its default and HTTP contract require maintainer product judgment.

## What This Changes

Add session-key-based overlap handling so recurring webhook deliveries can skip or reject a turn when the same session already has active or queued work.

## Best Possible Solution

Add an explicit hook overlap policy at the gateway admission boundary, backed by atomic session-lane admission and covering both active and queued turns; preserve current parallel/queue behavior by default unless maintainers deliberately approve a breaking default.

## Reproduction Assessment

Yes, at source level. Submit two webhook agent requests with the same resolved session key and no shared idempotency key: both are accepted, and the second is queued behind the first on the session lane.

## Solution Assessment

No, the proposed unconditional active-only 409 is not the best complete solution. The policy should account for active and queued work, reserve admission atomically, and be opt-in unless maintainers choose to change the existing webhook contract.

## Risks / Open Questions

- Making HTTP 409 the default would replace the shipped accept-and-queue contract and could break integrations that treat non-2xx responses as delivery failures or retries.
- Checking only active execution would still admit duplicates already queued on the same session lane, leaving the reported compounding behavior partially unresolved.
- A check performed before asynchronous dispatch must reserve admission atomically; a snapshot-only check can race when two deliveries arrive together.
