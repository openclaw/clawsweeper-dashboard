## Label Justifications

- `P2`: Recurring webhook deliveries can enqueue duplicate LLM turns and create resource pressure, but the impact is bounded to hook integrations that reuse session keys without idempotency keys.
- `impact:session-state`: The issue concerns admission and duplication of work associated with the same session identity.

## Summary

Keep open: current main and the latest release still accept duplicate non-idempotent webhook deliveries for the same resolved session key and queue them instead of rejecting or skipping them. The remaining work is real, but the public default, HTTP response contract, and active-plus-queued admission semantics need maintainer product/API review before implementation.

## What This Changes

Request a sessionKey-based webhook admission guard that rejects overlapping webhook deliveries, likely with HTTP 409, so polling integrations do not enqueue duplicate LLM turns for the same session.

## Best Possible Solution

Add an explicit hook overlap policy at the gateway admission boundary, backed by active-plus-queued session-lane admission and preserving the current accept-and-queue default unless maintainers deliberately approve a breaking default.

## Reproduction Assessment

Yes, at source level. Submit two direct or mapped webhook agent requests with the same resolved sessionKey and no shared idempotency key: both are accepted with HTTP 200/runId and the runtime queues them on the same session lane instead of rejecting the second delivery.

## Solution Assessment

No, the proposed unconditional active-only 409 is not yet the best complete solution. A maintainable fix should define opt-in/default behavior, include active plus queued work, and reserve admission atomically enough to avoid races.

## Risks / Open Questions

- Changing the default to HTTP 409 would replace the current accept-and-queue contract and could break integrations that treat non-2xx hook responses as delivery failures or retry triggers.
- An active-only check would still admit duplicates that are already queued on the same session lane, so the policy needs to define active plus queued behavior.
- A snapshot-only preflight can race when two deliveries arrive together; admission likely needs an atomic reserve/enqueue decision.
