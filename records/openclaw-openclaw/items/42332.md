## Summary

Keep open. Current main still has the webhook ingress gap: direct and mapped agent hooks replay idempotency keys, otherwise dispatch and return 200 without checking whether the normalized hook sessionKey is already active. Downstream session lanes serialize work after acceptance, which avoids simultaneous transcript mutation but does not give polling callers the requested 409 signal.

## What This Changes

The issue requests webhook delivery conflict handling keyed by sessionKey so repeated deliveries for an active session can return 409 instead of starting or queueing another turn.

## Best Possible Solution

Add a backward-compatible hook ingress concurrency policy for direct and mapped agent hooks. The guard should run after agent rebinding and session-prefix validation, use the normalized dispatch sessionKey, preserve idempotency replay semantics, return 409 only under the explicit policy, and update schema, docs, tests, and changelog together.

## Reproduction Assessment

Yes. Send two non-idempotent POST /hooks/agent or mapped hook requests with the same allowed sessionKey while the first run is still active; current source accepts and queues both rather than returning 409.

## Solution Assessment

Unclear for the issue’s unconditional default 409 proposal, because that would change existing webhook behavior. The narrow maintainable fix is an opt-in documented conflict-on-active policy that defaults to the current accept-and-queue behavior.

## Repair Work Prompt

Implement an opt-in sessionKey concurrency guard for webhook agent deliveries. Scope it to direct /hooks/agent and mapped agent hooks in the Gateway hook ingress path. Preserve current default behavior unless the new documented config policy is enabled. The guard must evaluate the normalized dispatch sessionKey after agent rebinding and allowed-prefix checks, preserve idempotency replay semantics so repeated idempotency keys still return the cached 200 runId, and return 409 Conflict only under the enabled policy when same-session hook work is active or already pending according to the chosen session-lane/active-run signal. Do not change cron job overlap behavior, hook auth, token handling, external-content wrapping, or generated per-request sessionKey behavior. Update config schema/types/docs and add a user-facing changelog entry crediting @ethanbeard.
