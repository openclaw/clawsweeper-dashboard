- `impact:session-state`: The reported problem is overlapping turns for the same logical sessionKey, which can drift session and agent state.

## Summary

Keep open: current main still has only idempotency-key replay protection for webhook deliveries, not an active-session guard keyed by resolved sessionKey, and the requested 409 behavior needs a maintainer API policy decision.

## What This Changes

Request support for rejecting or otherwise guarding overlapping webhook agent deliveries that resolve to the same sessionKey.

## Best Possible Solution

Define and document a lifecycle-aware webhook overlap policy for resolved sessionKey values, then implement it consistently for direct and mapped hooks with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Repeat direct or mapped webhook POSTs with the same resolved sessionKey and no reused idempotency key pass replay checks, dispatch a fire-and-forget run, and receive 200 while the async run can still be active.

## Solution Assessment

Unclear as proposed. A 409 active-session response is plausible, but the maintainable path should first define whether overlap handling is default, opt-in, queued, or only reported because this changes the public webhook contract.
