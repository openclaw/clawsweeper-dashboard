## Label Justifications

- `P2`: This is a normal-priority webhook/session improvement with real duplicate-turn and resource impact, but it requires API policy direction rather than emergency handling.
- `impact:session-state`: The reported failure is overlapping turns for the same logical sessionKey, which can duplicate work and drift session or agent state.

## Summary

Keep open: current main and the latest release still dedupe only by idempotency key, while direct and mapped webhook agent deliveries with the same resolved sessionKey can accept another 200 response while the previous async run is still active.

## What This Changes

Add a webhook delivery overlap guard keyed by resolved sessionKey, with the proposed behavior returning 409 Conflict when a previous turn for that sessionKey is still running.

## Best Possible Solution

Define a documented, compatibility-safe hook overlap policy for resolved sessionKey values, then implement it consistently for direct and mapped hooks with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Repeat direct or mapped webhook POSTs with the same resolved sessionKey and no reused idempotency key pass replay checks, dispatch a fire-and-forget run, and receive 200 while the async run can still be active.

## Solution Assessment

Unclear as proposed. Returning 409 is plausible, but maintainers should first choose whether overlap handling is default, opt-in, queued, or only reported because this changes the public webhook contract.
