## Summary

Keep open: current main and v2026.6.10 still admit same-session hook-agent deliveries with `200`/`runId`; idempotency replay and session-lane queueing are adjacent protections, but they do not provide the requested `sessionKey`-scoped 409 admission guard.

## What This Changes

The issue requests a sessionKey-scoped webhook concurrency guard that returns HTTP 409 when a delivery for the same hook sessionKey is already active.

## Best Possible Solution

Define an explicit hook overlap policy at the gateway admission boundary, likely opt-in first, that preserves today’s default `200`/`runId` behavior unless maintainers approve a breaking 409 default.

## Reproduction Assessment

Yes, from source. Direct and mapped hook-agent requests with the same resolved sessionKey and no shared idempotency key take the dispatch path and return `200`/`runId` without checking active or queued same-session work first.

## Solution Assessment

No, not as an unconditional 409 default. The maintainable solution needs a maintainer-chosen overlap policy and compatibility posture, likely opt-in first, with active-plus-queued admission semantics.

- The open persistent-session PR may reduce transcript forking for trusted hooks, but it does not settle reject versus skip versus queue versus parallel behavior for overlapping deliveries.
