## Summary

Keep open: current main and v2026.6.8 still dedupe hook retries only by idempotency key, then accept same-session webhook deliveries with HTTP 200/runId and queue them on the session lane. The related persistent-session work is useful adjacent context, but it does not implement the requested 409 overlap guard.

## What This Changes

The issue asks OpenClaw to reject webhook agent deliveries with 409 Conflict when another run for the same sessionKey is already active.

## Best Possible Solution

Add an explicit hook overlap policy at the gateway admission boundary, backed by active-plus-queued session-lane admission, while preserving the current accept-and-queue default unless maintainers deliberately approve a breaking default.

## Reproduction Assessment

Yes, at source level. Submit two direct or mapped hook-agent requests with the same resolved sessionKey and no shared idempotency key; current main accepts both and queues the work instead of returning a conflict.

## Solution Assessment

No, not as the unconditional active-only 409 behavior proposed in the issue body. The maintainable fix needs default/opt-in behavior, active plus queued admission, and an atomic enough reserve/enqueue point.

## Risks / Open Questions

- Changing the default to HTTP 409 would replace the current accept-and-queue hook contract and may break integrations that treat non-2xx hook responses as delivery failures.
- An active-only check would still admit duplicates that are already queued on the same session lane, so the policy needs active-plus-queued semantics.
- A snapshot-only preflight can race when two deliveries arrive together; the final design likely needs an atomic reserve/enqueue admission point or equivalent lane-level guard.
