## Label Justifications

- `P2`: This is a normal-priority webhook/session improvement with real duplicate-turn and resource impact, but it needs API policy direction rather than emergency handling.
- `impact:session-state`: The reported failure is overlapping turns for the same logical sessionKey, which can duplicate work and drift session or agent state.

## Summary

Keep open: current main and v2026.5.20 still provide idempotency-key replay protection only, not an active-run guard keyed by resolved webhook sessionKey or a 409 overlap response. The source path is clear, but the public overlap behavior needs maintainer policy direction before implementation.

## What This Changes

The issue requests rejecting or skipping overlapping webhook deliveries for the same resolved sessionKey, such as returning 409 Conflict while a prior webhook turn is still active.

## Best Possible Solution

Define a documented, compatibility-safe webhook overlap policy for resolved sessionKey values, then implement it consistently for direct and mapped hooks with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Repeat direct or mapped webhook POSTs with the same resolved sessionKey and no reused idempotency key pass replay checks, dispatch a fire-and-forget run, and receive 200 while the async run can still be active.

## Solution Assessment

Unclear as proposed. Returning 409 is plausible, but maintainers should first choose whether overlap handling is default, opt-in, queued, or only reported because this changes the public webhook contract.
