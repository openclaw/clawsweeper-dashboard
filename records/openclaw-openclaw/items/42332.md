- `impact:session-state`: The reported failure is overlapping turns for the same logical sessionKey, which can duplicate work and drift session or agent state.

## Summary

Keep open: current main and v2026.5.22 still accept same-sessionKey webhook deliveries when no idempotency replay key is reused. Adding a 409, skip, queue, or status policy would change the public webhook contract, so this needs maintainer product/API direction.

## What This Changes

The issue requests a sessionKey-scoped active-run guard for direct and mapped webhook deliveries, returning 409 Conflict when a previous turn for the same resolved sessionKey is still running.

## Best Possible Solution

Define a documented, compatibility-safe overlap policy for resolved webhook sessionKey values, then enforce it consistently for direct and mapped hook-agent deliveries while preserving idempotency replay and session-key prefix gates.

## Reproduction Assessment

Yes, source-level. Direct or mapped POSTs with the same allowed sessionKey but different or missing idempotency keys pass request policy and replay checks, dispatch a fire-and-forget agent run, and return 200 while an earlier async run can still be active.

## Solution Assessment

Unclear as proposed. A 409 guard is plausible, but maintainers should choose default versus opt-in behavior and 409 versus skip, queue, or status semantics because this changes webhook API behavior for existing integrations.
