## Summary

Keep open: current main has hook idempotency replay and cron job active tracking, but no sessionKey-scoped active webhook run guard or 409 conflict contract. The request is a real feature/API policy change rather than an already-defined bug fix.

## What This Changes

Add a sessionKey-scoped webhook concurrency guard that returns 409 when a previous delivery for the same sessionKey is still active.

## Best Possible Solution

Define an explicit webhook overlap policy for resolved sessionKey values, then implement it at hook ingress with lifecycle-aware release plus docs and regression coverage for direct and mapped hooks.

## Reproduction Assessment

Yes, from source inspection. Repeated webhook POSTs with the same resolved sessionKey and no reused idempotency key both reach dispatch and return 200 before the first isolated run completes.

## Solution Assessment

No, not as an unconditional default without maintainer API direction. The safer maintainable solution is an explicit overlap policy, likely opt-in or mapping-level, wired to actual run lifecycle rather than only request replay.
