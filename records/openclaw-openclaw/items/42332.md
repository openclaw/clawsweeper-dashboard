## Label Justifications

- `P2`: This is a normal-priority gateway/session feature with real duplicate-turn and resource impact but no emergency runtime outage.
- `impact:session-state`: The reported failure is overlapping agent turns for the same logical sessionKey, which can duplicate work and drift session or agent state.

## Summary

Keep open: current main and v2026.5.20 still only dedupe webhook replays by idempotency key, not by active resolved sessionKey, and they do not return 409 for overlapping direct or mapped hook deliveries. The remaining work is a webhook API/session policy decision rather than a cleanup close.

## What This Changes

The issue requests a sessionKey-based active-run guard for webhook POST deliveries, with overlapping deliveries returning 409 Conflict.

## Best Possible Solution

Define a documented overlap policy for resolved hook sessionKey values, then implement it consistently for direct and mapped hooks with compatibility-safe defaults and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Repeat direct or mapped webhook POSTs with the same resolved sessionKey and no reused idempotency key pass replay checks, dispatch a fire-and-forget run, and receive 200 while the async run can still be active.

## Solution Assessment

Unclear as proposed. Returning 409 is plausible, but maintainers should first choose whether overlap handling is default, opt-in, queued, or reported because that changes the public webhook contract.
