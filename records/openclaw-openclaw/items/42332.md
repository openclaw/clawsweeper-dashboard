## Label Justifications

- `P2`: The issue describes duplicate webhook-triggered agent turns with real operational impact, but no current evidence of an emergency outage or data-loss incident in core.
- `impact:session-state`: The requested guard is keyed by hook session identity and is meant to prevent overlapping work for the same logical session.

## Summary

Keep open: current main still accepts direct and mapped hook-agent deliveries with 200/runId and no same-session active/queued conflict path, while the open persistent-session PR is only partial overlap and does not settle the requested webhook overlap policy.

## What This Changes

Request to add a sessionKey-based concurrency guard for webhook deliveries so overlapping deliveries for an active session can be rejected with 409 instead of starting duplicate turns.

## Best Possible Solution

Define an explicit, compatibility-safe hook overlap policy at the gateway admission boundary, likely opt-in, that covers active and queued same-session work while coordinating with the persistent hook-session design.

## Reproduction Assessment

Yes. Source inspection shows two direct or mapped hook-agent requests with the same resolved sessionKey and no shared idempotency key are accepted with 200/runId instead of a caller-visible conflict.

## Solution Assessment

No for an unconditional default 409. The best path is a maintainer-approved overlap policy that decides reject, skip, queue, compatibility defaults, and active-plus-queued semantics before implementation.
