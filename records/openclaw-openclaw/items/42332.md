## Label Justifications

- `P2`: Same-session webhook polling can duplicate or queue redundant LLM work and increase resource pressure, but the remaining work is bounded to hook/session admission policy.

## Summary

Keep open: current main and v2026.6.10 still accept same-session webhook deliveries with `200`/`runId`; idempotency replay and session-lane queueing are related safeguards but do not provide the requested `sessionKey`-scoped 409/skip admission contract.

## What This Changes

The issue requests a webhook admission guard that rejects a POST with 409 Conflict when the resolved `sessionKey` already has active work.

## Best Possible Solution

Define an explicit webhook overlap policy at the Gateway hook admission boundary, likely opt-in first, covering active and queued same-session work while preserving today’s `200`/`runId` behavior unless maintainers approve a breaking default.

## Reproduction Assessment

Yes, from source. Submit two direct or mapped hook-agent requests with the same resolved `sessionKey` and no shared idempotency key; current main accepts both with separate `200`/`runId` responses instead of a conflict.

## Solution Assessment

No, not as an unconditional 409 default. The maintainable path needs a maintainer-chosen overlap policy, compatibility behavior, and active-plus-queued admission semantics.
