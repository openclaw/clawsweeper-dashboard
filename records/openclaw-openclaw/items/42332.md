## Summary

Keep open: current main and v2026.6.8 still admit same-`sessionKey` webhook deliveries with `200`/`runId`; related persistent-session work is useful context but does not implement a skip/reject overlap guard.

## What This Changes

The issue requests a `sessionKey`-scoped webhook overlap guard that rejects a new delivery while a prior turn for the same key is still active.

## Best Possible Solution

Add an explicit hook overlap policy at the Gateway admission boundary, preserving today’s accept behavior by default unless maintainers deliberately approve a breaking default.

## Reproduction Assessment

Yes, from source. Submit two direct or mapped hook-agent requests with the same resolved `sessionKey` and no shared idempotency key; current main accepts both and returns separate `200`/`runId` responses instead of a conflict.

## Solution Assessment

No, not as an unconditional 409 default. The maintainable path needs a maintainer-chosen overlap policy, compatibility behavior, and active-plus-queued admission semantics.

## Risks / Open Questions

- Changing the default to HTTP 409 would replace today’s accepted `200`/`runId` hook contract and may break integrations that treat non-2xx as delivery failure.
- An active-only check would still allow duplicates already queued for the same session, so the policy needs active-plus-queued semantics or another atomic admission point.
- Persistent-session work in https://github.com/openclaw/openclaw/pull/75918 may reduce duplicate simultaneous execution by queueing, but it does not settle whether webhook callers should see 409, skip, queue, or parallel behavior.
