## Summary

Keep open: current main and the latest release still accept overlapping webhook agent deliveries for the same resolved sessionKey unless the caller reuses an idempotency key, and changing that to a 409 response is a webhook API policy decision. The prior ClawSweeper review still matches the current source, and the related persistent-session PR did not implement an active-run overlap guard.

## What This Changes

The issue requests a sessionKey-scoped webhook ingress guard that returns 409 Conflict when a previous delivery for the same sessionKey is still running.

## Best Possible Solution

Define a documented webhook overlap policy for resolved sessionKey values, then implement a lifecycle-aware ingress guard with regression coverage for both direct and mapped hooks.

## Reproduction Assessment

Yes, source-level. Repeated direct or mapped webhook POSTs with the same resolved sessionKey and no reused idempotency key pass the replay checks, dispatch a fire-and-forget run, and receive 200 while the async run continues.

## Solution Assessment

No, not as an unconditional 409 default without maintainer API direction. The safer maintainable path is an explicit documented overlap policy, likely opt-in or mapping-level, wired to actual run lifecycle rather than request replay only.
