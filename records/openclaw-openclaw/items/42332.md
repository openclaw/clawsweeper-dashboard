## Label Justifications

- `P2`: This is a normal-priority webhook/session improvement with real duplicate-turn and resource impact, but it needs API policy direction rather than emergency handling.
- `impact:session-state`: The reported failure is overlapping turns for the same logical sessionKey, which can duplicate work and drift session or agent state.

## Summary

Keep open: current main and v2026.5.20 still dedupe webhook deliveries only by idempotency key, not by active resolved sessionKey, so repeated deliveries can dispatch independent async runs and return 200. The source path is clear, but default 409 versus an opt-in skip/queue/report policy is a public webhook contract decision.

## What This Changes

Add a webhook overlap guard that rejects a POST with 409 Conflict when another run for the same resolved sessionKey is active.

## Best Possible Solution

Define a documented, compatibility-safe overlap policy for resolved webhook sessionKey values, then enforce it consistently for direct and mapped hooks with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Direct or mapped webhook POSTs with the same allowed sessionKey and no reused idempotency key can pass replay checks, dispatch a fire-and-forget run, and receive 200 while the earlier async run can still be active.

## Solution Assessment

Unclear as proposed. Returning 409 is plausible, but maintainers should first choose whether overlap handling is default, opt-in, queued, or only reported because this changes the public webhook contract.
