## Summary

Keep open: current main and the latest release still accept overlapping webhook agent deliveries for the same resolved sessionKey unless the caller reuses an idempotency key, and adding a 409 overlap contract is a new webhook API policy decision.

## What This Changes

Request adds a sessionKey-scoped webhook overlap guard that returns 409 Conflict when a previous hook run for the same sessionKey is still active.

## Best Possible Solution

Define an explicit webhook overlap policy for resolved sessionKey values, then implement it at hook ingress with lifecycle-aware release, docs, and regression coverage for direct and mapped hooks.

## Reproduction Assessment

Yes, from source inspection. Repeated direct or mapped webhook POSTs with the same resolved sessionKey and no reused idempotency key pass replay checks, call dispatchAgentHook, and receive 200 while the async run continues.

## Solution Assessment

No, not as an unconditional default without maintainer API direction. The safer maintainable solution is an explicit overlap policy, likely opt-in or mapping-level, wired to actual run lifecycle rather than only request replay.
