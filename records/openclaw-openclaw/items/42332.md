## Summary

Keep open: current main and v2026.6.6 still accept same-session webhook deliveries after idempotency-only replay checks, then queue the runs on the session lane instead of returning a conflict. The user problem remains real, but the HTTP 409 contract, default compatibility behavior, and active-plus-queued admission semantics need maintainer product/API review before implementation.

## What This Changes

Request to reject or skip overlapping webhook deliveries for the same resolved sessionKey, such as returning HTTP 409 while a prior turn is still active.

## Best Possible Solution

Add an explicit hook overlap policy at the gateway admission boundary, backed by active-plus-queued session-lane admission, while preserving the current accept-and-queue default unless maintainers deliberately approve a breaking default.

## Reproduction Assessment

Yes, at source level. Submit two direct or mapped webhook agent requests with the same resolved sessionKey and no shared idempotency key: current main accepts both and queues the work instead of returning a conflict.

## Solution Assessment

No, the proposed unconditional active-only 409 is not yet the best complete solution. A maintainable fix should define default versus opt-in behavior, active plus queued admission, and an atomic enough reserve/enqueue point.

## Risks / Open Questions

- Changing the default from accept-and-queue to HTTP 409 could break existing webhook integrations that treat non-2xx responses as delivery failures.
- An active-only check would still admit duplicates that are already queued for the same session lane, so the design needs active-plus-queued semantics.
- A snapshot-only preflight can race when two deliveries arrive together; the final design likely needs an atomic reserve/enqueue boundary or equivalent session-lane admission contract.
- Related persistent-session work in https://github.com/openclaw/openclaw/pull/75918 is useful context but does not itself provide the requested same-session concurrency guard.
