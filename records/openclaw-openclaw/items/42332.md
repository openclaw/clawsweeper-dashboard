## Summary

Keep open. Current main still accepts same-session hook-agent webhook deliveries immediately, and the related persistent-session work at https://github.com/openclaw/openclaw/issues/11665 and https://github.com/openclaw/openclaw/pull/75918 addresses transcript reuse rather than a 409 active-turn rejection policy.

## What This Changes

Add a server-side sessionKey concurrency guard that returns 409 Conflict for webhook deliveries when the same sessionKey already has an active turn.

## Best Possible Solution

Define an explicit hook overlap policy at the hook dispatch boundary, keyed by the normalized dispatch sessionKey, preserving idempotency replay and session-key gates, with focused tests and docs for direct and mapped webhook deliveries.

## Reproduction Assessment

Yes, by source inspection on current main. Repeated direct or mapped hook-agent requests with the same resolved sessionKey reach `dispatchAgentHook` and return 200 without any active-sessionKey check; I did not run a live gateway because this review is read-only.

## Solution Assessment

Unclear at the final API level. A 409 guard is plausible and matches the report, but maintainers should decide how it interacts with idempotency, persistent hook sessions, retries, and possible queue-or-skip semantics before implementation.
