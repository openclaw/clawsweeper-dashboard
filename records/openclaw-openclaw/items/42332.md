## Summary

Keep open: current main and the latest release still accept overlapping webhook agent deliveries for the same resolved sessionKey unless the caller reuses an idempotency key, and changing that to a conflict response is a webhook API policy decision. Related persistent-session work does not implement an active-run overlap guard.

## What This Changes

Request to guard overlapping webhook agent deliveries for the same resolved sessionKey, preferably by returning 409 Conflict while a prior run is active.

## Best Possible Solution

Define a documented webhook overlap policy for resolved sessionKey values, then implement a lifecycle-aware ingress guard with regression coverage for both direct and mapped hooks.

## Reproduction Assessment

Yes, source-level. Repeated direct or mapped webhook POSTs with the same resolved sessionKey and no reused idempotency key pass replay checks, dispatch a fire-and-forget run, and receive 200 while the async run continues.

## Solution Assessment

No, not as an unconditional 409 default without maintainer API direction. The safer maintainable path is an explicit documented overlap policy, likely opt-in or mapping-level, wired to actual run lifecycle rather than request replay only.
