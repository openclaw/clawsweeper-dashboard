## Label Justifications

- `P2`: Same-session webhook polling can duplicate queued LLM work and increase resource pressure, but the remaining work is a bounded hook/session admission policy decision.
- `impact:session-state`: The issue is about preventing overlapping or repeated work for the same hook session identity.

## Summary

Keep open: current main and v2026.6.10 still accept same-sessionKey hook deliveries with 200/runId, while the open persistent-session work is partial overlap and does not implement a 409/skip overlap guard.

## What This Changes

The issue requests a webhook admission guard that rejects a delivery with 409 Conflict when the same sessionKey already has active work.

## Best Possible Solution

Add an explicit webhook overlap/admission policy at the Gateway boundary, preserving today's accept behavior by default unless maintainers approve a breaking 409/skip contract, and cover both active and queued same-session work.

## Reproduction Assessment

Yes, at source level. Submit two direct or mapped hook-agent requests with the same resolved sessionKey and no shared idempotency key; current main accepts both and returns 200/runId rather than a conflict.

## Solution Assessment

No, not as an unconditional 409 default. The maintainable fix needs a maintainer-chosen overlap policy, compatibility behavior, and active-plus-queued admission semantics.
