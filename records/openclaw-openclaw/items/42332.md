## Summary

Keep open: current main and v2026.6.8 still accept same-session hook deliveries with 200/runId unless the caller reuses an idempotency key. The requested sessionKey overlap/409 policy remains a real feature/API decision, and the open persistent-session work only partially overlaps rather than implementing this guard.

## What This Changes

The issue requests a webhook delivery admission policy that rejects overlapping deliveries with the same sessionKey, preferably with HTTP 409 so polling integrations can skip duplicate work.

## Best Possible Solution

Add an explicit hook overlap policy at the gateway admission boundary, backed by active-plus-queued session-lane admission, while preserving the current accept-and-queue default unless maintainers deliberately approve a breaking default.

## Reproduction Assessment

Yes, at source level. Submit two direct or mapped webhook agent requests with the same resolved sessionKey and no shared idempotency key: current main accepts both and queues the work instead of returning a conflict.

## Solution Assessment

No, the proposed unconditional active-only 409 is not yet the best complete solution. A maintainable fix should define default/opt-in behavior, active plus queued admission, and an atomic enough reserve/enqueue point.
