## Summary

Keep open: current main still accepts overlapping direct and mapped webhook deliveries for the same resolved `sessionKey` unless the caller reuses an idempotency key, and changing that behavior to a conflict response is a Gateway webhook API policy decision. The closed persistent-session PR and the cron exclusive-run issue do not implement this webhook active-run guard.

## What This Changes

Request adds a `sessionKey`-scoped webhook overlap guard that returns 409 Conflict when a webhook delivery targets an already-running session.

## Best Possible Solution

Define a documented webhook overlap policy for resolved `sessionKey` values, then implement a lifecycle-aware guard with regression coverage for both direct and mapped hooks.

## Reproduction Assessment

Yes, source-level: repeated direct or mapped webhook POSTs with the same resolved `sessionKey` and no reused idempotency key pass the replay checks, dispatch a fire-and-forget run, and receive 200 while the async run continues.

## Solution Assessment

No, not as an unconditional 409 default without maintainer API direction. The safer maintainable path is an explicit documented overlap policy, likely opt-in or mapping-level, wired to actual run lifecycle rather than request replay only.
