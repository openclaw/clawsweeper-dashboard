## Summary

Keep open: current main and the latest release still dedupe hook retries only by idempotency key, then accept same-session webhook deliveries with HTTP 200/runId and queue them on the session lane. The remaining work is real, but the default behavior, HTTP 409 contract, config surface, and active-plus-queued admission semantics need maintainer product/API review before implementation.

## What This Changes

Add a sessionKey-based webhook delivery guard so polling integrations can avoid enqueueing duplicate LLM turns when work for the same hook session is already active.

## Best Possible Solution

Add an explicit hook overlap policy at the gateway admission boundary, backed by active-plus-queued session-lane admission, preserving the current accept-and-queue default unless maintainers deliberately approve a breaking default.

## Reproduction Assessment

Yes, at source level. Submit two direct or mapped webhook agent requests with the same resolved sessionKey and no shared idempotency key: current main accepts both and queues the work instead of returning a conflict.

## Solution Assessment

No, the proposed unconditional active-only 409 is not yet the best complete solution. A maintainable fix should define default/opt-in behavior, active plus queued admission, and an atomic enough reserve/enqueue point.
