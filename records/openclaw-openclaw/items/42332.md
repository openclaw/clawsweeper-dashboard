## Summary

Keep open: current main still has idempotency-key replay protection, but no active-run guard keyed by resolved webhook sessionKey and no 409 overlap response for direct or mapped hook deliveries.

## What This Changes

Request a webhook concurrency guard that rejects or otherwise handles deliveries while another run for the same sessionKey is active.

## Best Possible Solution

Define a documented hook overlap policy for resolved sessionKey values, then implement it consistently for direct and mapped hooks with focused regression coverage and preserved current behavior unless maintainers choose a default 409 contract.

## Reproduction Assessment

Yes, source-level. Repeat direct or mapped webhook POSTs with the same resolved sessionKey and no reused idempotency key pass replay checks, dispatch a fire-and-forget run, and receive 200 while the async run can still be active.

## Solution Assessment

Unclear as proposed. Returning 409 is plausible, but maintainers should first choose whether overlap handling is default, opt-in, queued, or only reported because this changes the public webhook contract.
