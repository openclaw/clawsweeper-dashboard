## Summary

Keep open: current `main` and the latest release still accept same-session webhook deliveries with `200`/`runId`; idempotency replay and the open persistent-session PR are adjacent but do not provide the requested `sessionKey`-scoped 409/skip admission contract.

## What This Changes

The issue requests a `sessionKey`-scoped webhook concurrency guard that returns `409 Conflict` when a previous webhook delivery for the same logical session is still running.

## Best Possible Solution

Define an explicit Gateway hook overlap policy, likely opt-in first, that covers active and queued same-session deliveries while preserving today’s `200`/`runId` default unless maintainers approve a breaking public contract.

## Reproduction Assessment

Yes, from source. Submit two direct or mapped hook-agent requests with the same resolved `sessionKey` and no shared idempotency key; current main accepts both with separate `200`/`runId` responses instead of a conflict.

## Solution Assessment

No, not as an unconditional 409 default. The maintainable path needs a maintainer-chosen overlap policy, compatibility behavior, and active-plus-queued admission semantics.

- The open persistent-session PR may improve session continuity, but it does not settle whether hook callers should see reject, skip, queue, or parallel behavior for overlapping deliveries.
