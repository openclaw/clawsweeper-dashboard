## Summary

Keep open: current main and the latest release still accept same-session webhook deliveries with 200/runId, and the related persistent-session work does not implement the requested 409/skip overlap guard.

## What This Changes

Request to add a sessionKey-scoped concurrency guard for webhook deliveries, returning HTTP 409 when a prior turn for the same sessionKey is still active.

## Best Possible Solution

Add an explicit hook overlap policy at the Gateway admission boundary, preserving today’s accept behavior by default unless maintainers deliberately approve a breaking default, and cover active-plus-queued same-session behavior in tests.

## Reproduction Assessment

Yes, at source level. Submit two direct or mapped hook-agent requests with the same resolved sessionKey and no shared idempotency key; current main admits both and returns 200/runId instead of a conflict.

## Solution Assessment

No, not as an unconditional 409 default. The maintainable solution needs a maintainer-chosen overlap policy, compatibility behavior, and active-plus-queued admission semantics.

## Risks / Open Questions

- Changing the default webhook response to HTTP 409 would be a compatibility change for integrations that treat non-2xx as delivery failure.

- The persistent-session work in https://github.com/openclaw/openclaw/pull/75918 may change same-session execution behavior but does not decide whether webhook callers should see reject, skip, queue, or parallel semantics.
