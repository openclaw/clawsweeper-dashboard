## Summary

Keep open: current main has idempotency replay for repeated hook requests, but no sessionKey-based active-run guard, and the requested 409 behavior is a new webhook contract that needs maintainer product/API direction.

## What This Changes

The issue requests returning 409 Conflict when a webhook delivery arrives for a sessionKey that already has an active hook turn.

## Best Possible Solution

Add an explicit hook overlap policy tied to the resolved dispatch sessionKey and actual run lifecycle, with docs and regression coverage for core `/hooks/agent` and mapped hooks.

## Reproduction Assessment

Yes, from source inspection. Two hook POSTs with the same allowed sessionKey before the first isolated run settles will both pass the current handler and receive 200 unless they intentionally reuse the same idempotency key.

## Solution Assessment

No, not as an unconditional default yet. A maintainable fix should first define an overlap policy, likely opt-in or mapping-level, then implement a sessionKey-scoped in-flight guard that releases on real run completion.

## Risks / Open Questions

- Maintainers need to choose whether the conflict behavior is default, opt-in per hook/mapping, or a broader overlap mode such as skip/queue/parallel.
- A process-local in-flight map would not cover multi-process gateways unless the contract explicitly scopes it to one gateway runtime or uses durable state.
- The release path must define how locks clear for timed-out, aborted, crashed, or long-stalled hook runs.
