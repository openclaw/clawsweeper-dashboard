## Summary

Keep open: current main still accepts repeated webhook deliveries for the same resolved sessionKey unless the caller supplies and reuses an idempotency key, and adding a 409 active-session response changes the public Gateway webhook API contract.

## What This Changes

Requested support for rejecting overlapping webhook deliveries that resolve to the same sessionKey while an earlier webhook turn is still active.

## Best Possible Solution

Define and document a lifecycle-aware webhook overlap policy for resolved sessionKey values, then implement it consistently for direct and mapped hooks with focused regression coverage.

## Reproduction Assessment

Yes, source-level: repeat direct or mapped webhook POSTs with the same resolved sessionKey and no reused idempotency key pass replay checks, dispatch a fire-and-forget run, and receive 200 while the async run can still be active.

## Solution Assessment

No, not as an unconditional 409 default without maintainer API direction. The safer path is an explicit documented overlap policy, likely opt-in or mapping-level, wired to actual run lifecycle rather than request replay only.
