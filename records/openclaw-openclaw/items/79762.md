## Summary

Keep open. The sender normalizer gap from the earlier review is fixed and real behavior proof is now supplied, but the patch still turns provenance `sourceChannel` into `OriginatingChannel`, which current session code treats as delivery-route state rather than prompt-only metadata.

## What This Changes

This PR adds optional sender fields to `systemInputProvenance`, preserves them during normalization, and uses trusted operator provenance to populate `chat.send` sender/channel context.

## Best Possible Solution

Separate trusted author/provenance metadata from delivery-route fields, add focused gateway/session regression coverage, and land the additive protocol only after maintainer agreement on the author-attribution contract.

## Reproduction Assessment

Not applicable for the feature request itself. For the patch defect, source inspection gives a high-confidence path: the PR writes provenance `sourceChannel` into `MsgContext.OriginatingChannel`, and current session initialization persists that field into route state.

## Solution Assessment

No. The sender field normalization is now aligned, but routing metadata should not be overridden through `sourceChannel`; use a prompt/provenance-only field or the existing explicit originating-route path for delivery state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep provenance sourceChannel out of delivery routing:** `src/gateway/server-methods/chat.ts:2262`
  - body: The new `OriginatingChannel` fallback uses `systemInputProvenance.sourceChannel` even when `chat.send` did not provide an explicit route. `OriginatingChannel` is later persisted as session delivery state, and `resolveLastToRaw` can pair it with the previous `lastTo`, so an operator metadata-only send can corrupt future reply routing. Keep this as prompt/provenance metadata, or only override route fields through the existing explicit originating-route path.
  - confidence: 0.86
