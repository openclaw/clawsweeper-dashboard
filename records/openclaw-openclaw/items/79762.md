## Summary

Keep open. Current main does not already implement trusted per-message sender metadata for operator chat.send calls, and the PR is recent with supplied real behavior proof; the remaining blocker is maintainer protocol/trust-model review, not cleanup closure.

## What This Changes

The PR adds senderId, senderName, and senderRole to input provenance and lets operator-admin chat.send calls use that metadata for gateway sender attribution plus source-channel Provider/Surface metadata.

## Best Possible Solution

Choose one additive gateway attribution contract, then land it with schema, runtime trust semantics, docs, and focused tests aligned across provenance, chat history/events, and command/provider metadata.

## Reproduction Assessment

Not applicable for the feature request itself. Source inspection confirms current main lacks the new sender provenance fields, while the PR body and follow-up comment provide after-fix terminal proof for the branch behavior.

## Solution Assessment

Unclear as a product direction. The patch is mechanically narrow after the OriginatingChannel correction, but maintainers should choose whether trusted per-message identity belongs in systemInputProvenance or a first-class author contract before merge.
