## Summary

Keep open: the PR is a focused implementation of trusted sender metadata for operator-authenticated `chat.send`, but it changes gateway protocol/trusted-attribution semantics and should get maintainer security/product review before merge.

## What This Changes

The PR adds optional sender fields to `InputProvenance`, normalizes them, and uses operator-admin `systemInputProvenance` to populate chat dispatch sender identity plus Provider/Surface metadata.

## Best Possible Solution

Define one additive gateway sender-attribution contract, then land schema, runtime behavior, trust semantics, docs, and focused tests consistently across provenance, prompts, history, and events.

## Reproduction Assessment

Not applicable for a feature PR. Source inspection confirms current main lacks the proposed sender provenance fields, and the PR body/comment provide after-fix terminal proof for the branch behavior.

## Solution Assessment

Unclear as the final product direction. The patch is mechanically narrow and admin-gated, but maintainers should decide whether trusted per-message identity belongs in `systemInputProvenance` or in the broader first-class author contract before merge.
