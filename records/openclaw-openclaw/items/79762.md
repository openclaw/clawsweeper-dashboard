## Summary

Keep open. Current main does not already implement trusted sender metadata in `systemInputProvenance`, but the PR still needs maintainer protocol/trust review and has a concrete correctness issue around reusing `sourceChannel` as runtime `Provider`/`Surface`.

## What This Changes

The PR adds optional sender fields to input provenance and uses operator-admin `systemInputProvenance` to populate chat.send sender context.

## Best Possible Solution

Land one additive attribution contract that keeps gateway runtime routing/auth fields separate from provenance display metadata, with schema, trust semantics, docs, and focused tests aligned.

## Reproduction Assessment

Not applicable for the feature request itself. Source inspection confirms current main lacks the new sender provenance fields, and the PR body/comment provide after-fix terminal proof for the branch behavior.

## Solution Assessment

No. The sender passthrough is mechanically narrow, but using `sourceChannel` as `Provider`/`Surface` is not the narrowest maintainable solution because those fields drive authorization, routing, hooks, and model selection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep sourceChannel out of Provider and Surface:** `src/gateway/server-methods/chat.ts:2260-2265`
  - body: The PR describes `sourceChannel` as prompt metadata, but assigning it to `ctx.Provider` and `ctx.Surface` changes runtime decisions. Current main uses those fields for command authorization, channel model override, hooks, and reply routing, so an operator `chat.send` carrying `sourceChannel` can stop behaving like an internal gateway turn. Keep `Provider`/`Surface` as `INTERNAL_MESSAGE_CHANNEL` and expose the provenance channel through provenance or a separate prompt-metadata field.
  - confidence: 0.88
