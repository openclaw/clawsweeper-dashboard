## Summary

Keep open: this is a recent external feature PR with supplied gateway proof, and current main does not already implement the sender provenance fields. The patch still needs maintainer protocol/product review because it changes public gateway attribution, and the current `sourceChannel` handling crosses into `Provider` state that existing runtime code uses beyond prompt metadata.

## What This Changes

The PR adds senderId, senderName, and senderRole to InputProvenance and uses operator-admin systemInputProvenance to override chat.send MsgContext sender and source-channel metadata.

## Best Possible Solution

Define one additive gateway attribution contract, keeping routing/auth state separate from display or prompt metadata, then land schema, runtime, docs, and focused tests together.

## Reproduction Assessment

Not applicable for a feature PR as a bug reproduction. Source inspection confirms current main lacks the new sender fields, and the PR body/comment provide after-fix terminal proof for the branch behavior.

## Solution Assessment

No for the current patch as-is. Passing trusted sender metadata is plausible, but writing provenance sourceChannel into Provider is not the narrowest maintainable solution because Provider is existing routing/auth context, not only prompt metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep sourceChannel out of Provider routing state:** `src/gateway/server-methods/chat.ts:2257-2264`
  - body: `Provider` is used outside prompt rendering: current runtime falls back to it for `messageProvider` when `OriginatingChannel` is absent and also checks it for internal-channel owner behavior. Setting it from `systemInputProvenance.sourceChannel` means an operator-supplied sourceChannel can change command/tool/reply context even after `OriginatingChannel` was fixed to preserve delivery state. Carry this as prompt/display provenance without overwriting `Provider`, or require an explicit origin route for runtime channel changes.
  - confidence: 0.84
