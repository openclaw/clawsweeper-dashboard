## Summary

Keep open; current main does not implement trusted sender metadata through systemInputProvenance, and this active feature PR still needs maintainer protocol/product review plus a routing fix because Provider/Surface are used for delivery decisions, not only prompt metadata.

## What This Changes

The PR adds optional senderId, senderName, and senderRole fields to input provenance and uses operator-admin systemInputProvenance to populate chat.send sender identity and source-channel Provider/Surface metadata.

## Best Possible Solution

Define one additive gateway attribution contract, then land it with schema, runtime, trust semantics, docs, and focused tests aligned across provenance, prompt metadata, routing, chat history, and event attribution.

## Reproduction Assessment

Not applicable as a feature PR rather than a current-main bug report. Source inspection confirms current main lacks the new sender provenance fields, and the PR body plus follow-up comment provide after-fix terminal proof for the branch behavior.

## Solution Assessment

Unclear as a product direction. The sender override is narrow, but sourceChannel currently flows into routing-sensitive Provider/Surface fields and maintainers should choose whether trusted per-message identity belongs in systemInputProvenance or a first-class author contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep sourceChannel out of routing surfaces:** `src/gateway/server-methods/chat.ts:2258-2264`
  - body: `Provider` and `Surface` are not prompt-only metadata: dispatch routing uses them as the current surface before deciding whether to route replies to `OriginatingChannel`. With this change, an operator `chat.send` using `deliver: true`, an explicit `originatingChannel`/`originatingTo`, and a matching provenance `sourceChannel` makes the current surface equal the origin, so route-reply is skipped and the response stays on the Gateway dispatcher instead of being delivered externally. Keep these fields internal for chat.send or introduce a separate prompt-metadata field.
  - confidence: 0.87
