## Summary

Keep this PR open: the latest head fixed the earlier route-state bug and the remaining change is a plausible additive operator-gateway feature, but it changes the protocol/provenance contract and overlaps an open author-attribution design thread that needs maintainer review before merge.

## What This Changes

The PR adds optional sender metadata fields to `InputProvenance`, normalizes them, and uses operator-admin `systemInputProvenance` to populate `chat.send` sender and source-channel context.

## Best Possible Solution

Land an additive protocol only after maintainers decide whether trusted per-message identity belongs in `systemInputProvenance` or the first-class author-attribution contract tracked at https://github.com/openclaw/openclaw/issues/68353.

## Reproduction Assessment

Not applicable for the feature request itself. For the previously reported patch defect, source inspection now shows `OriginatingChannel` remains tied to the explicit route value rather than provenance `sourceChannel`.

## Solution Assessment

Unclear as a product direction: the latest patch looks mechanically narrow after the route fix, but maintainers should choose whether this belongs in trusted provenance or in the first-class `author` contract before merge.
