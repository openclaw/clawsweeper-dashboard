## Summary

Keep open: current main still has the linked cron/tool-allowlist bug, and this PR is aimed at the right area, but the merge result is incomplete because the wildcard override is applied only at the final policy filter while `allowBackground` is still derived from the restrictive profile. The supplied proof is unit-test-only, and the intended behavior broadens a tool-authorization boundary that needs maintainer review before merge.

## What This Changes

The PR makes runtime `toolsAllow: ["*"]` skip static profile/provider-profile filters in `createOpenClawCodingTools` and adds a focused regression test for `exec` visibility with the `messaging` profile.

## Best Possible Solution

Move the wildcard override into one effective-policy calculation used by both `allowBackground` and the final policy pipeline, add regression coverage for background/yield behavior, and require redacted real cron or isolated runtime proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main passes runtime `toolsAllow` into tool construction and then applies wildcard filtering after profile filtering, so `applyEmbeddedAttemptToolsAllow` cannot restore an `exec` tool already removed by the `messaging` profile. I did not run tests because this review was read-only.

## Solution Assessment

No; the PR is a plausible fix location but not the best complete fix yet. The wildcard decision needs to be applied before both `allowBackground` derivation and final policy filtering, with regression coverage and real cron/isolated runtime proof.
