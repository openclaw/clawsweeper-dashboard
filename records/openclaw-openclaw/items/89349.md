## Summary

Keep this PR open, but do not merge it as written: it fixes a real source-visible cron tool-construction failure mode by making `exec` survive `toolsAllow: ["*"]`, but the implementation bypasses `tools.profile` and `tools.byProvider.profile` for every wildcard runtime allowlist, which changes a documented security boundary beyond the cron case.

## What This Changes

The PR skips static profile/provider-profile filtering when `runtimeToolAllowlist` contains `*` and adds focused agent/cron tests for wildcard allowlists with the messaging profile.

## Best Possible Solution

Preserve profile/provider-profile caps by default, then fix the cron isolated-session case through a narrow explicit widening path or a maintainer-approved wildcard precedence change with docs, upgrade/security rationale, and real cron proof.

## Reproduction Assessment

Yes from source inspection: current main builds shell tools for wildcard cron allowlists, then filters them through the messaging profile before returning the final tool list. I did not run tests because this review is read-only.

## Solution Assessment

No as written: the patch is a plausible Bug B workaround, but it is too broad because it changes wildcard-vs-profile precedence for every embedded caller. A narrower cron-owned explicit widening path or a maintainer-approved contract change would be safer.
