## Summary

Keep open. The runtime change is plausibly the right layer for the linked cron exec regression, but this PR still needs maintainer acceptance of the wildcard security-boundary change, real behavior proof from the cron or isolated-runtime path, and a corrected background regression test.

## What This Changes

The PR makes `createOpenClawCodingTools` skip static profile and provider-profile filters when `runtimeToolAllowlist` contains `*`, and adds regression tests for exec visibility/background behavior under the `messaging` profile.

## Best Possible Solution

Land a version that keeps global, agent, group, sandbox, subagent, and inherited denies authoritative while allowing an explicit runtime wildcard to bypass only profile/provider-profile filters, with live isolated-cron proof and a real exec yield/background assertion.

## Reproduction Assessment

Yes from source inspection: the cron executor passes payload `toolsAllow` to `runEmbeddedAgent`, the embedded runner passes it into `createOpenClawCodingTools`, and current main still applies the `messaging` profile before the final wildcard filter. I did not run a live cron repro in this read-only review.

## Solution Assessment

The implementation layer is likely right because final `toolsAllow` filtering cannot restore a tool removed during construction or change `exec` background behavior. The submitted PR is not the best complete merge shape until the background test observes real exec execution and maintainers accept the wildcard security-boundary precedence.
