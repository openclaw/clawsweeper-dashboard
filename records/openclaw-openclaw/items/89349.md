## Label Justifications

- `P1`: The PR addresses a user-facing isolated cron regression where configured jobs can lose `exec` despite an explicit wildcard tool allowlist.
- `merge-risk: 🚨 compatibility`: Existing cron jobs that relied on a messaging profile to cap wildcard `toolsAllow` would see a broader shell-tool surface after merge.
- `merge-risk: 🚨 security-boundary`: The changed precedence can expose `exec` and `process` in trusted cron runs, so maintainers need to accept the scoped shell-tool boundary.

## Summary

Keep open: the patch is still useful and appears correct for the linked tool-injection bug; current main still lets the messaging profile filter out cron wildcard shell tools, while this PR scopes the override to cron and preserves non-cron profile caps.

## What This Changes

The PR lets cron-triggered embedded runs with `runtimeToolAllowlist` containing `*` bypass static/provider profile filtering in `createOpenClawCodingTools`, and adds focused cron/non-cron regression tests.

## Best Possible Solution

Land this narrow Bug B fix after maintainers accept the scoped cron wildcard precedence, and keep the remaining scheduler routing/status work tracked in https://github.com/openclaw/openclaw/issues/89228.

## Reproduction Assessment

Yes for the narrow Bug B path from source: cron forwards `trigger: "cron"` plus `toolsAllow`, wildcard builds shell tools, and current main then filters them through the messaging profile before the runtime allowlist can keep `exec`. I did not run a live current-main cron repro in this read-only review.

## Solution Assessment

Yes for Bug B: the patch changes the owner boundary where the loss occurs, keeps the override scoped to cron wildcard runs, and preserves non-cron profile caps. It is not the full solution for the broader linked issue's routing and status-classification bugs.
