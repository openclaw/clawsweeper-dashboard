## Summary

Keep this PR open, but it is not merge-ready: the cron exec symptom is source-reproducible, yet the patch changes shared runtime wildcard precedence in a way that can widen profile-capped tool access beyond cron.

## What This Changes

The PR skips static and provider profile filtering whenever `runtimeToolAllowlist` contains `*` and adds focused agent/cron tests for wildcard allowlists with the messaging profile.

## Best Possible Solution

Restore `exec` for cron isolated `toolsAllow: ["*"]` through a cron-owned or explicitly approved precedence path while keeping profile caps as the default security boundary.

## Reproduction Assessment

Yes. Source inspection shows current main carries cron `toolsAllow` into `runtimeToolAllowlist` but still applies the messaging profile before the final runtime allowlist filter, so `exec` is filtered out for that shape.

## Solution Assessment

No as written. The cron symptom is real, but this patch fixes it by changing shared profile precedence; the safer solution is a cron-scoped override or explicit maintainer approval of global wildcard-over-profile semantics.
