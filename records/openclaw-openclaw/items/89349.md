## Summary

Keep this PR open: it addresses a real cron tool-construction bug, but the patch changes wildcard runtime allowlists into a global profile/provider-profile bypass across shared agent tool construction, which needs a narrower fix or explicit maintainer policy approval before merge.

## What This Changes

The PR skips `tools.profile` and `tools.byProvider.profile` filtering when `runtimeToolAllowlist` contains `*`, then adds focused agent and cron tests for wildcard allowlists with the messaging profile.

## Best Possible Solution

Restore `exec` for cron isolated `toolsAllow: ["*"]` without changing global profile precedence, or make the runtime-wildcard-overrides-profile rule an explicit maintainer-approved and documented policy change.

## Reproduction Assessment

Yes, from source inspection: current main carries cron `toolsAllow` into `runtimeToolAllowlist` but still applies the messaging profile before the final runtime allowlist filter. I did not run a live repro because this review is read-only.

## Solution Assessment

No as written. The cron symptom is real, but the patch fixes it by changing a shared profile-precedence rule used beyond cron; the safer fix is cron-owned narrowing or explicit maintainer approval of the new global precedence.
