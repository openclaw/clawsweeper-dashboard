## Summary

Keep open. The normalization bug remains plausible on current main, but this PR branch is stale and unsafe as submitted because its selector change does not preserve the current endpoint-class guard or the OpenAI Codex HTTP/boundary-aware transport boundary.

## What This Changes

The PR normalizes provider/modelApi matching for OpenAI WebSocket transport selection and adds embedded-runner regression coverage for normalized OpenAI values and non-OpenAI providers.

## Best Possible Solution

Normalize provider and modelApi inside the current selector while keeping the endpoint-class guard, keeping `openai-codex` out of this native WebSocket selector, and adding focused tests for casing/whitespace plus custom baseUrl and Codex fallback behavior.

## Reproduction Assessment

Yes. A focused helper case with provider `"  OpenAI  "`, modelApi `" OpenAI-Responses "`, and no custom baseUrl reproduces the strict-comparison problem on current main; this review did not execute tests because the task is read-only.

## Solution Assessment

No. The normalization direction is useful, but the submitted patch is not the best fix because it drops current main's endpoint guard and Codex transport boundary; the safer fix is to normalize before the existing guarded decision.
