## Summary

This PR should stay open for maintainer decision: current main still has the missing-target delivery gap, and the patch is focused with real Telegram proof, but it changes last-mile fallback targeting by using same-channel requester-session state when the completion origin lacks `to`.

## What This Changes

The PR imports `deliveryContextFromSession`, backfills missing completion delivery fields from the requester session entry before computing `effectiveDirectOrigin`, and adds same-channel recovery plus cross-channel isolation regression tests.

## Best Possible Solution

Land this narrow fallback only after a delivery-path owner explicitly accepts same-channel requester-session `lastTo` as the final missing-target fallback; otherwise require a bounded alternative such as task-bound destination capture, freshness limits, or a clearer no-target failure.

## Reproduction Assessment

Yes, at source level: current main computes deliverability before any requester-entry `lastTo` fallback, and the linked report shows persisted requester origins with channel/accountId but no `to`. The Mantis Telegram Desktop recording also demonstrates visible candidate delivery, though I did not run a fresh live repro in this read-only review.

## Solution Assessment

Conditionally yes. If maintainers accept current-session same-channel fallback routing, this is the narrowest owner-local fix because it reuses `deliveryContextFromSession` and `mergeDeliveryContext`; if they want immutable task destinations, this PR is only a plausible mitigation.
