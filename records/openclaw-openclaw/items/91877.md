The PR adds a Discord voice receive priming helper that plays a short Opus silence burst through the joined voice player and wires it into `manager.join` with unit tests.

## Best Possible Solution

Prime Discord receive without using or blocking the normal playback capture gate, add coverage for speaker-start during the prime window, and attach redacted live Discord voice proof before merge.

## Reproduction Assessment

Yes for the blocking PR defect by source inspection: `player.play` can put the shared Discord voice player in Playing, and current `handleSpeakingStart` returns before subscribing non-realtime capture during Playing. I did not establish a live Discord reproduction for the original SFU issue because no inspectable live artifact is attached.

## Solution Assessment

No. The op-5 priming direction is plausible, but sending the prime through the same capture-gated player is not the best fix because it can suppress the exact first speaker event the PR is meant to recover.
