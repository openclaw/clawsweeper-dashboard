## Summary

Keep open: current main still lacks Anthropic advisor support, but this PR should not merge as-is because it uses an incompatible documented default advisor model and does not preserve required advisor result blocks through Anthropic replay.

## What This Changes

The PR adds Anthropic advisor-tool beta header/tool injection, server-side tool-result display handling, stop-reason hardening, user docs, live validation scripts, and focused transport/wrapper/chat-content tests.

## Best Possible Solution

Rebase or replace this PR with a maintainer-reviewed implementation that uses pair-aware advisor config, preserves raw advisor/server-side result blocks through transcript storage/replay and pruning, and documents the supported config contract.

## Reproduction Assessment

Yes. Enable params.advisor: true for an Anthropic Sonnet 4.6 model and inspect the outgoing advisor payload, then replay an assistant turn containing advisor_tool_result through convertAnthropicMessages and verify the next request lacks the raw advisor_tool_result block.

## Solution Assessment

No. The feature direction is useful, but display-only server-tool text is not the narrow maintainable fix for Anthropic's replay contract; a raw/opaque server-block representation integrated with pruning is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Anthropic requires that executor to use Opus 4.7 as advisor, and invalid pairs return 400, so the documented common config will fail until the default is pair-aware or users must configure a compatible advisor explicitly.

Anthropic expects full assistant content, including advisor result blocks, on later turns; store and replay the raw server-side blocks instead of replacing them with `serverToolDisplay` text.
  - confidence: 0.89
