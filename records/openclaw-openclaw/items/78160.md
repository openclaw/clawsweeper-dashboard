## Summary

Keep open. This PR targets a real, source-reproducible bundled MCP timeout problem, but it is not merge-ready because after-fix real behavior proof is still missing and the patch leaves two concrete contract gaps around progress reset and config schema coverage.

## What This Changes

The PR adds a per-server bundled MCP `requestTimeoutMs`, passes it to SDK `callTool` with `resetTimeoutOnProgress`, and updates MCP docs plus focused tests.

## Best Possible Solution

Complete the explicit `requestTimeoutMs` contract across runtime, SDK progress handling, public schema, docs, and tests, then merge only after real long-running MCP proof and maintainer acceptance of the default/progress policy.

## Reproduction Assessment

Yes, source-reproducible: current main executes materialized bundled MCP tools through `Client.callTool` without `RequestOptions`, and SDK v1.29.0 defaults omitted request timeouts to 60000 ms. I did not run a live 60s-plus MCP tool in this read-only review.

## Solution Assessment

No, not merge-ready as submitted. Passing SDK `RequestOptions.timeout` is the right narrow code direction, but the progress-reset wiring, public schema field, missing real proof, and timeout-policy choice still need follow-up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Request progress before relying on progress resets:** `src/agents/pi-bundle-mcp-runtime.ts:368`
  - body: The SDK only adds a progress token and stores a progress handler when `RequestOptions.onprogress` is provided. This PR passes `resetTimeoutOnProgress: true` without `onprogress`, so bundled MCP servers are not asked to send progress for this request and progress cannot reset the timer; either pass an appropriate progress callback or remove the promised reset behavior from code/docs/tests.
  - confidence: 0.9
- **[P2] Add requestTimeoutMs to the config schema:** `src/config/types.mcp.ts:20-21`
  - body: The PR adds `requestTimeoutMs` to the exported type, docs, and runtime, but `McpServerSchema` still does not declare it. That leaves the new public config option out of schema-derived config/help/editor surfaces; please add the positive finite-number field to `src/config/zod-schema.ts` with focused schema coverage.
  - confidence: 0.88
