## Summary

Keep open: the PR targets a real source-reproducible bundled MCP tool timeout path, but it is not merge-ready because the new public config field is not added to the config schema and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds a bundled MCP `requestTimeoutMs` setting, forwards it to SDK `Client.callTool` with progress timeout resets, and updates docs plus focused tests.

## Best Possible Solution

Complete the bundled MCP request-timeout contract across runtime, public type, config schema, docs, and focused tests, then merge only after real long-running tool proof and maintainer acceptance of the timeout policy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: bundled MCP tools execute through `params.runtime.callTool`, which reaches SDK `Client.callTool` without `RequestOptions`, and SDK v1.29.0 defaults omitted request timeouts to 60000 ms. I did not run a live long-running MCP tool in this read-only review.

## Solution Assessment

No, not merge-ready as submitted. Passing SDK `RequestOptions` is the right narrow code direction, but the public config schema gap, missing real behavior proof, and timeout-policy choice still need follow-up.
