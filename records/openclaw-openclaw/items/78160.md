## Summary

Keep open. Current main and the latest release still inherit the SDK's 60-second MCP tool-call timeout, and this PR is a plausible fix path, but it is not merge-ready because the public config schema is incomplete and after-fix real behavior proof is missing.

## What This Changes

The PR adds a per-server bundled MCP `requestTimeoutMs`, forwards it to SDK `callTool` with `resetTimeoutOnProgress`, and updates docs plus focused tests.

## Best Possible Solution

Complete the bundle MCP request-timeout contract across runtime, type, public schema, docs, and focused tests, then merge after real long-running tool proof and maintainer acceptance of the default/progress policy.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.5.7` call SDK `Client.callTool` without `RequestOptions`, and SDK 1.29.0 defaults omitted request timeouts to 60000 ms. I did not run a live 60s+ MCP tool in this read-only review.

## Solution Assessment

No, not merge-ready as submitted. Passing SDK `RequestOptions` is the right narrow code direction, but the public config schema gap, missing real behavior proof, and timeout-policy decision still need follow-up.
