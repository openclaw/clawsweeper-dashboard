## Summary

Keep this PR open. Current main still drops Codex item-event `meta` before Telegram can render it, so the core behavior is still valid, but the latest PR head introduced blocking review issues in the expanded Codex/docs surface: the Codex resume-param test now expects an unsupported `summary` field, and the docs check is already failing on the head SHA.

## What This Changes

The PR forwards Codex item-event `meta` through auto-reply callbacks, formats Telegram draft progress from structured item name/meta, adds optional Codex app-server `reasoningSummary` config/docs/baseline updates, adjusts Crabbox workflow input handling, and adds focused Telegram/Codex tests plus a changelog entry.

## Best Possible Solution

Land the Telegram meta-forwarding and turn-start reasoning-summary support after fixing the Codex test and docs formatting. Keep `summary` off `thread/resume` unless the generated Codex protocol adds that field, and keep the Crabbox workflow hardening only if maintainers accept that extra CI change in this PR.

## Reproduction Assessment

Yes. A current-main Codex item event shaped like `{ kind: "tool", title: "Tool", name: "read", meta: "from README.md" }` reproduces the user-visible Telegram problem because `meta` is emitted by Codex, dropped by auto-reply, and Telegram then falls back to `Tool`. The PR’s new test issue is also reproducible by evaluating `buildThreadResumeParams`, which does not return `summary`.

## Solution Assessment

No, not as submitted. Forwarding `meta` and formatting Telegram progress is the narrow right fix, and sending `reasoningSummary` on `turn/start` matches the generated protocol, but the resume-param expectation and failing docs check need repair before merge.
