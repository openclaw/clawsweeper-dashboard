## Summary

Keep open; this is a meaningful feature PR that is not implemented on main, but the latest head still has a process-lifecycle/security contract gap for shell-backed commands and lacks real behavior proof before merge.

## What This Changes

The PR adds an opt-in `ui.statusLine` TUI config that runs a shell command on a timer, renders stdout in the TUI, updates config schema metadata, adds docs, and adds focused status-line runner tests.

## Best Possible Solution

Land only after maintainers choose the status extension model and the PR either enforces full process-tree cleanup or narrows the public cleanup contract, with real TUI proof and focused tests.

## Reproduction Assessment

Yes for the review finding. PR head launches with `shell: true`; timeout/stop signal only the returned `ChildProcess`, and Node’s child_process contract says shell descendants can survive parent termination on Linux.

## Solution Assessment

No, not as-is. The schema/TUI wiring is plausible, but the process cleanup guarantee, trusted-local-command security contract, and product choice between raw shell config and plugin status providers need settlement before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Align cleanup with shell-backed commands:** `src/tui/tui-statusline.ts:83-84`
  - body: This runner starts the command with `shell: true`, so the later `killChild(child)` calls signal the shell wrapper only. On Linux, shell descendants can survive parent termination, so a status script that backgrounds work can outlive timeout or TUI exit while the docs claim it cannot leak; either kill the full process group/tree or narrow that public contract.
