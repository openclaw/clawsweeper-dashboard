## Summary

Keep open. Current main still lacks this TUI status extension, the PR is recent and active, and the remaining blockers are real behavior proof plus maintainer product/security judgment around trusted shell execution and overlapping footer/plugin-status work.

## What This Changes

Adds an opt-in `ui.statusLine` config/schema/docs surface plus TUI runner, wiring, and tests to periodically execute a shell command and render stdout in the TUI.

## Best Possible Solution

Choose one TUI status extension model: either accept a trusted-local command feature with accurate cleanup/output contracts and real TUI proof, or route status contributions through the plugin/footer provider track coordinated with the open footer tracker.

## Reproduction Assessment

Yes for the PR review findings: PR-head source shows `shell: true` cleanup only targets the returned `ChildProcess`, and the output cap compares decoded string length. For the requested feature itself, reproduction is not applicable because current main has no `ui.statusLine` surface.

## Solution Assessment

No, not as currently patched. The implementation direction is plausible, but it still overclaims process cleanup, misstates the byte cap, lacks real TUI proof, and needs maintainer choice between raw shell commands and the plugin/footer API path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Narrow the descendant cleanup guarantee:** `docs/web/tui.md:55`
  - body: This docs line promises that a SIGTERM-ignoring script cannot leak processes, but the runner starts commands with `shell: true` and only signals the returned `ChildProcess`. A command that backgrounds a child can still leave a descendant running after timeout or stop, so either kill a process group/tree or narrow the documented guarantee.
  - confidence: 0.92
- **[P3] Cap buffered bytes or document character limits:** `src/tui/tui-statusline.ts:105-108`
  - body: The code and docs describe a 4 KB cap, but this appends decoded UTF-8 and compares `stdout.length`, which counts JavaScript string code units. Multibyte output can exceed the promised byte cap, so cap bytes before decoding or document this as a character/code-unit limit.
  - confidence: 0.81

## Security Review
