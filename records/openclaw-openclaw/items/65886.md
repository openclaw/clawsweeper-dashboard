## Summary

Keep open: current main does not implement this new TUI status-line config, and the PR is active with unique feature work, but it still needs real behavior proof plus maintainer judgment on the trusted local shell/process-cleanup contract.

## What This Changes

The PR adds an opt-in `ui.statusLine` config block, generated schema/help/docs entries, a TUI status-line command runner, lifecycle tests, and TUI wiring to render command stdout.

## Best Possible Solution

Decide whether OpenClaw wants a raw trusted-local shell status line or a plugin/status-provider footer API; if this PR proceeds, make the cleanup/output-cap contract accurate and require real TUI proof.

## Reproduction Assessment

Yes for the PR review findings: PR-head source plus a focused Node probe show shell descendants can survive cleanup of the returned shell process. For the requested feature itself, reproduction is not applicable because current main has no `ui.statusLine` surface.

## Solution Assessment

No, not as currently patched. The schema and TUI wiring are plausible, but the cleanup guarantee remains inaccurate for shell descendants, the output cap is not byte-based, and the raw shell extension point needs maintainer product/security direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Narrow the cleanup guarantee for shell descendants:** `src/tui/tui-statusline.ts:83-84`
  - body: This starts the configured command with `shell: true`, but `killChild()` only signals the returned shell process. A status script that backgrounds a child can still leave that descendant running after timeout or stop, so either kill a process group/tree or narrow the docs/help guarantee that ignored SIGTERM cannot leak processes.
  - confidence: 0.93
- **[P3] Cap bytes instead of decoded string length:** `src/tui/tui-statusline.ts:105-108`
  - body: The constant and docs promise a 4 KB output cap, but this appends decoded UTF-8 and compares `stdout.length`, which counts JavaScript string code units. Multibyte output can exceed the documented byte cap; cap buffered bytes or document this as a character/code-unit limit.
  - confidence: 0.81
