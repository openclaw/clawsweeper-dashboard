## Summary

Keep open for maintainer review. The contributor has addressed earlier schema/generated-artifact issues, but the current head still lacks real TUI proof and leaves the shell-backed cleanup/security contract unresolved.

## What This Changes

The branch adds `ui.statusLine` config/schema/docs plus a TUI runner that periodically executes a configured shell command and renders stdout below the editor.

## Best Possible Solution

Settle whether OpenClaw wants a raw trusted local shell status line, a plugin status-provider/footer API, or both, then land a branch with accurate cleanup semantics, docs, tests, and real TUI proof.

## Reproduction Assessment

Yes for the review finding: PR head spawns through `shell: true` and cleanup signals only the returned `ChildProcess`, while Node's docs say shell descendants are not killed with the parent on Linux. For the feature itself, reproduction is not applicable because current main has no `ui.statusLine` surface.

## Solution Assessment

No, not as-is. The schema and TUI wiring are plausible, but the trusted-shell process-tree contract, public docs claim, real behavior proof, and relationship to the plugin status-provider path need settlement first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Kill the shell process group or narrow cleanup docs:** `src/tui/tui-statusline.ts:83-84`
  - body: This runner starts the configured command with `shell: true`, but timeout/stop cleanup still signals only the returned shell process. Node documents that Linux shell descendants are not killed with their parent, so a status command can leave child work running despite the new SIGKILL fallback; either kill the process group/tree or narrow the cleanup guarantee in code/docs.
  - confidence: 0.91

  - body: The docs and constant describe a 4 KB output cap, but the runner trims `stdout.length` after UTF-8 decoding, which counts JavaScript string code units rather than bytes. Multibyte output can exceed the documented byte cap; cap buffered bytes or document this as a character/code-unit cap.
  - confidence: 0.78
