## Summary

Keep open: current main does not implement a configurable TUI status line, and the PR is a plausible feature branch, but it still needs real behavior proof plus maintainer judgment on the trusted shell/process-cleanup contract before merge.

## What This Changes

The PR adds an opt-in `ui.statusLine` config/schema/docs surface plus a TUI runner that periodically runs a shell command and renders stdout below the editor.

## Best Possible Solution

Land only after maintainers choose the status extension model and the PR either enforces full process-tree cleanup or narrows the public cleanup contract, with real TUI proof attached.

## Reproduction Assessment

Yes for the review finding: PR head uses `shell: true` and cleanup signals only the returned `ChildProcess`, while Node documents that Linux shell descendants are not killed with their parent. For the feature request itself, reproduction is not applicable because current main simply lacks the requested config surface.

## Solution Assessment

No, not as-is. The schema and TUI wiring are plausible, but the trusted local command model, process-tree cleanup guarantee, and relationship to plugin status providers need maintainer settlement before this is the narrowest maintainable solution.
