## Summary

Keep open. Current main still lacks the requested configurable TUI status-line command surface, and the PR is active, but the latest patch still needs a process-tree timeout fix plus required changelog/rebase validation before it is merge-ready.

## What This Changes

The PR adds an opt-in `ui.statusLine` config block, schema/help/docs metadata, a TUI runner that periodically executes a local shell command and renders stdout, lifecycle wiring, and focused runner tests.

## Best Possible Solution

Rebase the branch onto current main, keep the config/schema/docs artifacts aligned, change the runner so timeout and stop reliably terminate the command process tree or narrow the command contract to avoid shell descendants, add the required Unreleased changelog credit, and then have a maintainer decide whether this belongs in core TUI config or should wait for a plugin/status-render seam.

## Reproduction Assessment

Yes. The current-main feature gap is reproducible by inspecting the strict `ui` schema and TUI layout, and the process-leak finding has a high-confidence contract reproduction path with a shell-backed status command that starts a long-lived child.

## Solution Assessment

No, not as-is. The direction is plausible and mostly narrow, but the current timeout implementation does not satisfy the documented cleanup guarantee for `shell: true` commands; a process-group kill strategy or safer non-shell command contract is needed first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Kill the whole status command process tree:** `src/tui/tui-statusline.ts:83-84`
  - body: Because the runner uses `shell: true`, `child.kill()` targets the shell process, not necessarily the command it launched. Node documents that children of child processes can survive killing their parent on Linux, so `sleep`, scripts, or tools spawned by the status command can leak past timeout/stop even with the SIGKILL fallback.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `docs/web/tui.md:42`
  - body: This is a user-facing TUI/config feature, but the PR does not update `CHANGELOG.md`. Please add a single-line Unreleased entry with appropriate `Thanks @nanami-he` credit so the release notes cover the new status-line command surface.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds an explicit trusted-local-command surface, but the timeout/stop implementation does not reliably terminate shell-descendant processes.

Concerns:

- **[medium] Shell descendants can survive timeout cleanup:** `src/tui/tui-statusline.ts:83`
  - body: `ui.statusLine.command` auto-runs through `shell: true`; killing only the returned shell process can leave descendant commands running after timeout or TUI exit, weakening the safety boundary documented for this feature.
