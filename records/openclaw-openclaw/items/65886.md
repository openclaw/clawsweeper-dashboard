## Summary

Keep open for maintainer review: current main still lacks this feature, and the PR remains a new security-sensitive TUI config surface with missing real behavior proof plus a shell-descendant cleanup contract bug.

## What This Changes

The PR adds an opt-in `ui.statusLine` config block, TUI runner/wiring that periodically executes a shell command and renders stdout, generated config metadata/docs, and fake-spawn lifecycle tests.

## Best Possible Solution

Settle the product/security boundary between a raw trusted-local shell status line and the plugin status-provider/footer API; if this PR lands, make cleanup and output-cap contracts accurate and require real TUI proof.

## Reproduction Assessment

Yes for the PR review findings: PR-head source shows `shell: true` cleanup that signals only the returned child, and a focused Node probe showed a shell descendant alive after its shell was killed. For the requested feature itself, reproduction is not applicable because current main has no `ui.statusLine` surface.

## Solution Assessment

No, not as currently patched. The schema and TUI wiring are plausible, but the cleanup guarantee remains inaccurate for shell descendants, the output cap is not byte-based, and the feature overlaps the plugin/footer status-provider path that needs maintainer decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Narrow the cleanup guarantee for shell descendants:** `src/tui/tui-statusline.ts:83-84`
  - body: This starts the configured command with `shell: true`, but `killChild()` only signals the returned shell process. A status script that backgrounds a child can still leave that descendant running after timeout or stop, so either kill the process group/tree or narrow the docs/help guarantee that ignored SIGTERM cannot leak processes.
  - confidence: 0.91
- **[P3] Cap bytes instead of decoded string length:** `src/tui/tui-statusline.ts:105-108`
  - body: The constant and docs promise a 4 KB output cap, but this appends decoded UTF-8 and compares `stdout.length`, which counts JavaScript string code units. Multibyte output can exceed the documented byte cap; cap buffered bytes or document this as a character/code-unit limit.
  - confidence: 0.78
