## Summary

Keep open. Current main still lacks this status-line extension, and the PR is useful, but it still needs real TUI proof and a fix or narrower contract for shell-backed process cleanup before merge.

## What This Changes

This PR adds an opt-in `ui.statusLine` config/schema/docs surface and TUI runner that periodically executes a shell command and renders stdout below the editor.

## Best Possible Solution

Settle the TUI status extension model, then either enforce full process-group/tree cleanup or narrow the public cleanup contract, with real TUI proof attached before merge.

## Reproduction Assessment

Yes for the review finding: PR head uses `shell: true` and only signals the returned `ChildProcess`, while Node documents Linux shell descendants surviving parent termination. For the feature request itself, reproduction is not applicable because current main lacks this config/status surface.

## Solution Assessment

No, not as-is. The schema and rendering path are plausible, but the auto-started shell command cleanup contract and relationship to plugin status providers need maintainer settlement first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep shell descendants inside the cleanup contract:** `src/tui/tui-statusline.ts:83-84`
  - body: This runner starts the configured command with `shell: true`, but timeout and `stop()` cleanup only signal the returned shell `ChildProcess`. On Linux, grandchildren can survive killing that shell, so a status script that starts background work can outlive the timeout or TUI exit while the docs promise it cannot leak; either kill a process group/tree or narrow the cleanup guarantee and tests.
