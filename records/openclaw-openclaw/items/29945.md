## Summary

Keep open: current main still locks in branded pairing replies for newly created unpaired DM pairing requests, and the open PR that explicitly tracks configurable unpaired responses has not merged.

## What This Changes

The issue requests a silent or configurable unpaired DM response policy so unknown senders do not receive a branded OpenClaw pairing reply by default.

## Best Possible Solution

Land or supersede PR #39065 with a centralized unpaired-response policy wired through the shared pairing helper, config schema, docs, and DM-capable channel call sites.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a DM-capable channel using `dmPolicy: "pairing"` reaches `issuePairingChallenge`, `buildPairingReply` creates branded text, and the channel send callback delivers it.

## Solution Assessment

Unclear on the requested silent default because that changes product behavior; the maintainable shape is a shared configurable policy rather than per-channel suppression.
