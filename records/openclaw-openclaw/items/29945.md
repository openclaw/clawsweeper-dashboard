## Summary

Keep open: current main still hard-codes branded pairing replies for newly created unpaired DM pairing requests, the requested `unpairedResponse` config is absent, and PR #39065 is open and unmerged.

## What This Changes

The issue requests changing unpaired DM pairing behavior to default silent mode with optional `silent`, `code-only`, and `branded` response modes.

## Best Possible Solution

Land or supersede PR #39065 with a centralized, documented unpaired-response policy across the shared pairing helper and DM-capable call sites, including schema, docs, tests, and an explicit default-policy decision.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with a DM-capable channel using `dmPolicy: "pairing"`, an unknown sender reaches `issuePairingChallenge`, `buildPairingReply` creates branded text, and the channel send callback delivers it.

## Solution Assessment

Unclear on the requested silent default, but the centralized config-based helper direction is the maintainable shape. Per-channel suppression would be weaker than one shared policy wired through the pairing contract and documented config schema.
