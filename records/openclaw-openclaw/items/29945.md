## Summary

Keep open: current main still sends branded pairing replies for newly unpaired DM senders and has no `unpairedResponse` config surface. The issue is already paired with an open maintainer-labeled fix PR, so it should stay open until that PR or a replacement lands.

## What This Changes

Request to make unknown DM pairing responses silent or configurable so OpenClaw branding and pairing codes are not exposed by default.

## Best Possible Solution

Land a coherent response-mode contract across shared pairing helpers, config schemas, docs, and channel call sites, with an explicit maintainer decision on the default and any identifier-redaction follow-up.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: an unknown sender under `dmPolicy: "pairing"` reaches `issuePairingChallenge`, which sends `buildPairingReply` containing the OpenClaw brand. I did not run a live channel in this read-only review.

## Solution Assessment

Unclear as proposed: the open PR direction is plausible, but review feedback shows schema/runtime gaps and it keeps the backward-compatible branded default. Maintainers should decide the default posture and land the paired PR or a tighter replacement.
