## Summary

Keep this issue open. Current main still sends branded pairing replies to newly created unpaired DM requests, the requested `unpairedResponse` policy is absent from config/docs/tests, and open PR #39065 uses closing syntax for this work but has not merged.

## What This Changes

The issue asks for a configurable unpaired DM response policy so unknown senders can receive no reply, a code-only reply, or the existing branded OpenClaw pairing reply.

## Best Possible Solution

Land or supersede PR #39065 with a shared, documented `unpairedResponse` policy wired through the pairing helper and all DM pairing call sites, with regression tests for `silent`, `code-only`, and `branded`, config schema/metadata/docs updates, and an explicit decision on sender-identifier logging.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure a DM-capable channel with `dmPolicy: "pairing"`, send a DM from an unapproved sender, and the pairing branch calls `issuePairingChallenge`, which builds the branded reply and sends it through the channel API.

## Solution Assessment

Unclear on the default, but the shared config direction is the right fix shape. The maintainable solution should be centralized in the pairing helper/config contract rather than per-channel ad hoc suppression, while maintainers decide whether security posture or backward compatibility controls the default.
