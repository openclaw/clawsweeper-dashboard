## Summary

Keep open: current main still sends branded pairing-code replies to newly unpaired DM senders and has no `unpairedResponse` policy, while https://github.com/openclaw/openclaw/pull/39065 remains open with closing syntax and unresolved review concerns.

## What This Changes

The issue asks for a configurable unpaired DM response policy, preferably silent by default, to avoid fingerprinting OpenClaw to unknown senders.

## Best Possible Solution

Review, revise, and land or supersede https://github.com/openclaw/openclaw/pull/39065 with one centralized unpaired-response policy wired through config schemas, pairing helpers, docs, tests, and all DM-capable channel call sites.

## Reproduction Assessment

Yes. Source inspection on current main shows `dmPolicy: "pairing"` reaches `issuePairingChallenge`, `buildPairingReply` includes OpenClaw branding and a pairing code, and representative WhatsApp/Discord paths send that text.

## Solution Assessment

Unclear on silent-by-default because that changes existing pairing behavior. The maintainable direction is a centralized configurable policy through shared pairing helpers and channel config contracts, not per-channel suppression patches.
