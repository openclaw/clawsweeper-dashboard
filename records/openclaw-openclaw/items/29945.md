## Summary

Keep open: current main still sends a branded OpenClaw pairing-code response to newly unpaired DM senders and does not expose an `unpairedResponse` or equivalent suppression policy. The issue is paired with open fix PR https://github.com/openclaw/openclaw/pull/39065 via closing syntax, so this should remain open until that PR lands or is superseded.

## What This Changes

The issue requests suppressing or de-branding unpaired DM pairing replies, preferably by adding `channels.unpairedResponse` with `silent` as the default.

## Best Possible Solution

Review, fix, and land or supersede https://github.com/openclaw/openclaw/pull/39065 with one centralized unpaired-response contract wired through schemas, docs, pairing helpers, and all DM-capable channel call sites.

## Reproduction Assessment

Yes. Source inspection on current main shows `dmPolicy: "pairing"` reaches the shared branded pairing reply, representative WhatsApp/Discord/Telegram paths send that text, and no `unpairedResponse` config exists.

## Solution Assessment

Unclear as proposed because making silence the default is a security/product decision. The maintainable direction is a centralized configurable response policy rather than per-channel suppression patches.
