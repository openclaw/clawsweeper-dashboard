## Summary

Keep this open: current main still emits branded pairing replies for new unpaired DM pairing requests, lacks the requested response policy config, and the linked implementation PR is still open and unmerged.

## What This Changes

Add a configurable unpaired DM response policy, preferably silent by default, so unknown DM senders do not receive branded OpenClaw pairing replies.

## Best Possible Solution

Land or supersede #39065 with a centralized, documented unpairedResponse policy across the shared pairing helper and DM-capable call sites, including config schema, docs, regression tests, and a decision on raw sender logging.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with a DM-capable channel using dmPolicy="pairing", an unknown sender reaches issuePairingChallenge, buildPairingReply creates branded text, and the channel send callback delivers it.

## Solution Assessment

Unclear on the requested silent default, but the shared config direction is the right fix shape. The maintainable solution should be centralized in the pairing helper/config contract rather than per-channel suppression.
