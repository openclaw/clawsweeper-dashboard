## Summary

Keep open: current main still sends the branded pairing-code response for newly unpaired DM senders and has no `unpairedResponse` or equivalent suppression policy. The issue is also paired with open PR https://github.com/openclaw/openclaw/pull/39065 via closing syntax, so it should remain open until that PR lands or is superseded.

## What This Changes

Request a configurable unpaired-DM response policy, preferably silent by default, to avoid exposing OpenClaw branding or pairing codes to unknown senders.

## Best Possible Solution

Review, fix, and land or supersede https://github.com/openclaw/openclaw/pull/39065 with one centralized unpaired-response contract wired through config schemas, docs, pairing helpers, and all DM-capable channel call sites.

## Reproduction Assessment

Yes. Source inspection on current main shows `dmPolicy: "pairing"` reaches the shared pairing challenge, the shared reply includes OpenClaw branding and a pairing code, and representative WhatsApp/Discord paths send that text.

## Solution Assessment

Unclear as proposed because silent-by-default is a security/product default change. The maintainable solution is a centralized configurable response policy with a maintainer decision on the default, not per-channel suppression patches.
