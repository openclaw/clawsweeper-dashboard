## Summary

Keep open: current main still sends branded OpenClaw pairing replies for newly created unpaired DM pairing requests, and the linked fix PR #39065 is still open rather than merged.

## What This Changes

Request silent, code-only, or branded handling for unpaired DM pairing replies to reduce platform and OpenClaw fingerprinting.

## Best Possible Solution

Land or supersede PR #39065 with a centralized, documented unpaired-response policy wired through shared pairing helpers, config schema/types, and DM-capable channel call sites.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `dmPolicy: "pairing"` reaches `issuePairingChallenge`, `buildPairingReply` includes OpenClaw branding, and channel callbacks send that text.

## Solution Assessment

Unclear on the silent default because it changes documented pairing behavior. The narrow maintainable direction is a shared configurable policy, not per-channel suppression.
