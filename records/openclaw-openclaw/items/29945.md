## Summary

Keep this issue open. Current main still sends branded pairing replies for newly created unpaired DM requests, has no `unpairedResponse` config surface, and the open unmerged PR #39065 is the active closing implementation path for this exact work.

## What This Changes

Request that unpaired DM/pairing flows avoid leaking OpenClaw/platform identity by supporting silent, code-only, and branded response policies, with silent proposed as the default.

## Best Possible Solution

Keep this issue open until PR #39065 is reviewed and either merged or superseded by an equivalent maintainer-approved change that adds a documented unpaired-response policy, wires it through shared pairing helpers and all DM pairing call sites, updates config schema/docs/tests, and makes an explicit security decision on the default and nearby identifier logging.
