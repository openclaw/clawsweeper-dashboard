## Summary

Keep open: current main still has the branded pairing-code reply path and no `unpairedResponse` policy, while https://github.com/openclaw/openclaw/pull/39065 is still open with closing syntax and unresolved review concerns.

## What This Changes

The issue requests a silent or configurable unpaired-DM response policy so unknown senders do not receive an OpenClaw-branded pairing reply.

## Best Possible Solution

Review, revise if needed, and land or supersede https://github.com/openclaw/openclaw/pull/39065 with one shared unpaired-response policy wired through config schemas, pairing helpers, docs, and DM-capable channel call sites.

## Reproduction Assessment

Yes. Source inspection on current main shows `dmPolicy: "pairing"` reaches `issuePairingChallenge`, `buildPairingReply` includes OpenClaw branding and a pairing code, and representative WhatsApp/Discord paths send that text.

## Solution Assessment

Unclear on silent-by-default because that changes documented pairing behavior. The maintainable direction is a centralized configurable policy through the shared pairing helper and channel config contracts, not per-channel suppression patches.
