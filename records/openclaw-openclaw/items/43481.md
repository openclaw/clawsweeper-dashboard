## Summary

Keep open: current main and the latest release still lack the requested provider-level media trust policy, and current source still has a prompt/media-reference path that can load images during fallback without a fallback trust decision. A linked open PR already uses closing syntax for this issue, so the issue should remain open until maintainers review, repair, or replace that implementation path.

## What This Changes

The issue requests a provider trust boundary for model fallback so text fallback trust can be separated from image or attachment trust.

## Best Possible Solution

Adopt a maintainer-approved fallback media trust design that covers explicit images, prompt-detected images, and attachment-style media references with upgrade-safe defaults and focused tests.

## Reproduction Assessment

Yes, from source inspection: prompt-referenced and attachment-style image refs can still be detected and loaded in the embedded runner without a fallback/provider-trust parameter. I did not run a live cross-provider repro, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear. A provider-level media trust policy is plausible, but the default behavior, config shape, migration/docs impact, and the linked conflicting PR need maintainer security/product review before this is the best fix.
