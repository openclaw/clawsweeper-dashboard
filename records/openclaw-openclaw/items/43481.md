## Summary

Keep open: current main strips explicit `images` on every fallback retry, but it still lacks provider-level trust configuration and still has a source-visible prompt-image path that can reload local image references during fallback. The issue is also explicitly paired with an open implementation PR that closes it, so it should stay open until that PR is landed or rejected.

## What This Changes

The issue asks OpenClaw to respect provider trust boundaries for user-supplied images and attachments during cross-provider model fallback retries.

## Best Possible Solution

Keep this issue paired with https://github.com/openclaw/openclaw/pull/57959 or a replacement PR, and make one canonical retry media decision cover explicit images, prompt-detected images, and any documented attachment path with config/docs aligned if trust controls are added.

## Reproduction Assessment

Yes, by source inspection for the remaining prompt-detected-image trust gap: current main strips explicit fallback `images`, but still detects prompt-local image references without a fallback suppression flag. The broader provider trust-policy request is not implemented as a current config surface.

## Solution Assessment

Unclear. A fallback-safety patch is the right area, but maintainers still need to decide whether the best solution is configurable provider-level trust or a simpler fail-closed cross-provider media rule.
