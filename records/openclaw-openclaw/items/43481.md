## Label Justifications

- `P1`: The issue concerns potential unintended disclosure of user media to a different model provider during fallback.
- `impact:security`: The core concern is a provider trust boundary for sensitive image and attachment data.
- `impact:auth-provider`: The behavior depends on model provider routing and cross-provider fallback choices.

## Summary

Keep open: current main still lacks a provider media-trust policy, prompt-detected image and attachment refs can still be loaded for fallback attempts, and the linked open PR at https://github.com/openclaw/openclaw/pull/57959 is the active implementation venue but remains conflicting and review-blocked.

## What This Changes

Request provider-level fallback media trust controls so images and attachments are not sent across provider boundaries without explicit trust.

## Best Possible Solution

Resolve the linked fallback-safety PR or a narrower replacement with one shared provider-media decision covering explicit images, prompt-detected refs, attachment refs, config/docs, and upgrade behavior.

## Reproduction Assessment

Yes. Source inspection shows fallback retries still lack provider media-trust policy, while prompt-detected image and attachment refs can be loaded for fallback-capable models; I did not run live providers in this read-only review.

## Solution Assessment

Unclear. A configurable trust policy is plausible, but maintainers need to choose between default-compatible trust configuration, fail-closed cross-provider media stripping, or both with an explicit upgrade path.
