## Label Justifications

- `P1`: The issue concerns potential unintended disclosure of user media to a different model provider during fallback.
- `impact:security`: The core concern is a provider trust boundary for sensitive image and attachment data.
- `impact:auth-provider`: The behavior depends on model provider routing and cross-provider fallback choices.

## Summary

Keep open: current main still lacks a provider media-trust policy, explicit retry images are stripped only by a blanket fallback-retry rule, and prompt-detected image or attachment refs can still be loaded for the fallback attempt. The linked open PR is the right implementation venue for now, but it is conflicting and still needs product/security judgment before this issue can close.

## What This Changes

The issue requests a provider-level fallback media trust policy so image and attachment data is not unintentionally forwarded across provider boundaries during model fallback.

## Best Possible Solution

Resolve the linked fallback-safety PR or a narrower replacement with one shared provider-media decision covering explicit images, prompt-detected refs, attachment refs, config/docs, and upgrade behavior.

## Reproduction Assessment

Yes. Source inspection shows fallback retries still strip explicit images by a blanket retry boolean while prompt-detected image and attachment refs can still be loaded, and no provider media-trust config exists; I did not run live providers in this read-only review.

## Solution Assessment

Unclear. A configurable trust policy is plausible, but maintainers need to decide whether the durable boundary is default-compatible trust configuration, fail-closed cross-provider media stripping, or both with an explicit upgrade path.
