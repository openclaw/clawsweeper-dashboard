## Summary

Keep open: current main still lacks a provider media-trust policy, fallback retries still use blanket image stripping rather than a provider-aware decision, and prompt/attachment image refs can still be loaded for a fallback-capable target. An open closing pull request already owns the implementation path, but it is conflicting and still needs product/security judgment.

## What This Changes

Requests provider-level trust controls so fallback retries do not send user-supplied images or attachment media across provider boundaries without explicit trust.

## Best Possible Solution

Resolve the linked PR path with one shared fallback-media decision covering explicit images, prompt-detected images, attachment refs, config, docs, and upgrade behavior.

## Reproduction Assessment

Yes. Source inspection shows a fallback-capable model can still load prompt-local image and attachment refs through `detectAndLoadPromptImages`, while current config/docs provide no provider media-trust boundary; I did not run live providers.

## Solution Assessment

Unclear. A configurable trust policy may be the right product shape, but maintainers need to decide whether the durable fix is explicit trust configuration or fail-closed cross-provider media stripping at the fallback boundary.
