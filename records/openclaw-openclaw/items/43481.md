## Label Justifications

- `P1`: The issue concerns possible unintended disclosure of user media to a different configured model provider during fallback.
- `impact:security`: The requested behavior is a trust boundary for sensitive image and attachment data.
- `impact:auth-provider`: The behavior depends on provider routing and configured cross-provider model fallback choices.

## Summary

Keep open: current main and the latest release still do not provide a provider media-trust policy for fallback retries, and the linked closing PR remains open. Source inspection shows explicit images are stripped only by retry state, while prompt-detected image and attachment refs still load without a cross-provider trust decision.

## What This Changes

Add provider-level media trust controls so images and attachments do not cross model-provider fallback boundaries unless that target is trusted for media.

## Best Possible Solution

Review or replace the linked PR with one provider media-trust policy covering explicit images, prompt-detected refs, attachment refs, skipped candidates, config/docs, tests, upgrade behavior, and security review.

## Reproduction Assessment

Yes, for source-level reproduction: current main and v2026.5.22 have cross-provider fallback paths while prompt/attachment image loading lacks a provider media-trust gate. I did not run a live provider failover in this read-only review.

## Solution Assessment

Unclear. A provider media-trust policy is the right direction, but maintainers still need to choose the compatibility default, config surface, docs, and upgrade behavior before the implementation path is settled.
