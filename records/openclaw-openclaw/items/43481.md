- `impact:security`: The issue concerns a sensitive-data disclosure boundary between independently operated model providers.
- `impact:auth-provider`: The affected decision is provider fallback routing and which request data may be forwarded to the selected provider.

## Summary

Current main strips explicitly supplied images on every fallback retry, but prompt-referenced local images are still detected inside the embedded runner, and no provider trust policy exists. This issue should remain open because https://github.com/openclaw/openclaw/pull/57959 explicitly closes it but is still open, conflicting, proof-gated, and awaiting a compatibility/security policy decision.

## What This Changes

Add user control over whether images and attachments may cross provider boundaries during model fallback.

## Best Possible Solution

Define one explicit core fallback data policy covering both supplied and prompt-detected images, preserve or migrate existing behavior according to the chosen compatibility contract, and land it with same-provider/cross-provider tests plus redacted real-provider proof.

## Reproduction Assessment

Yes from source at medium confidence: cross-provider fallback is a supported path, explicit images are currently stripped on retries, and prompt image detection remains active inside the embedded runner. A controlled live run is still needed to confirm the complete outbound payload.

## Solution Assessment

Unclear. A provider-level trust policy matches the requested compatibility goal, while fail-closed cross-provider stripping offers a stronger default; maintainers must select the permanent policy before the linked implementation can be judged the best fix.
