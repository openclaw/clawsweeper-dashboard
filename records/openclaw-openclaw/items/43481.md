## Summary

Keep open: current main does not implement provider-level fallback trust controls, and the remaining media boundary is only partially mitigated because explicit retry images are stripped while prompt-referenced images can still be loaded by the embedded runner. The linked open PR is an implementation candidate, not a merged fix.

## What This Changes

The issue requests provider trust controls so cross-provider model fallback does not unintentionally forward user image or attachment data to a provider trusted only for text fallback.

## Best Possible Solution

Keep this issue as the canonical security/product tracker and resolve it by landing a maintainer-approved fallback trust-boundary design or explicitly choosing a different policy for the linked candidate PR.

## Reproduction Assessment

Yes, at source level: current main strips explicit retry images but still lets the embedded runner load prompt-referenced images from the retry prompt, and no provider trust-policy config exists. I did not run a live provider repro in this read-only cleanup review.

## Solution Assessment

Unclear until maintainer decision: the issue’s trust-policy proposal fits the security goal, but choosing compatibility-preserving forwarding, stricter stripping, or an explicit trust knob is a product/security policy call. The linked open PR is a plausible candidate, not a merged best-solution proof.
