## Summary

Keep open: current main still has no provider-level media trust policy, and a source-visible prompt-image path can still load local image references during fallback even though explicit `images` are stripped on fallback retries. The issue is also paired with an open closing PR, so it should stay open until that PR or a replacement is landed or rejected.

## What This Changes

The issue asks OpenClaw to let users distinguish text-fallback provider trust from image/attachment trust when model fallback crosses provider boundaries.

## Best Possible Solution

Keep this issue paired with https://github.com/openclaw/openclaw/pull/57959 or a replacement PR, and make one canonical retry-media decision cover explicit images, prompt-detected images, and documented attachment paths with aligned config/docs if trust controls are added.

## Reproduction Assessment

Yes, by source inspection: explicit fallback images are stripped, but prompt-local image detection still runs and there is no provider media-trust config surface on current main. I did not execute a live provider fallback run in this read-only review.

## Solution Assessment

Unclear. A fallback-safety patch belongs at the `runAgentAttempt` / embedded-runner boundary, but maintainers still need to decide whether the best policy is configurable media trust or fail-closed cross-provider media stripping.
