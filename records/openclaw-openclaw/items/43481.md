## Summary

Keep open: current main and the latest release still lack a provider-level media trust policy, and prompt-detected images can still be loaded during fallback without a provider-boundary guard. The issue is also linked to an open, conflicting implementation PR, so it should not be closed before that path is reviewed or replaced.

## What This Changes

Requests provider trust boundaries so cross-provider model fallback does not forward user-supplied images or attachments to an untrusted fallback provider.

## Best Possible Solution

Review, repair, and either land or replace the linked fallback-safety PR with a maintainer-approved trust-boundary design that covers explicit images, prompt-detected images, and attachment-like media paths.

## Reproduction Assessment

Yes from source inspection: current main strips explicit images on fallback retry but still lets the embedded runner detect and load prompt-referenced images without a fallback/provider-trust guard. I did not run a live cross-provider repro, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear. A provider-trust config may be the right permanent solution, but repository policy treats fallback/config/security behavior as product-sensitive and the open implementation PR still needs maintainer review and real behavior proof.
