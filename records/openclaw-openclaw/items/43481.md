## Label Justifications

- `P1`: The issue concerns possible unintended disclosure of user media to a different model provider during fallback.
- `impact:security`: The core concern is a trust boundary for sensitive image and attachment data.
- `impact:auth-provider`: The behavior depends on provider routing and configured cross-provider model fallback choices.

## Summary

Keep open: current main still lacks a provider media-trust policy for model fallback, and the open closing PR remains unmerged. Source inspection shows retry media handling is based on retry state, while prompt-detected image and attachment refs can still be loaded without a cross-provider trust decision.

## What This Changes

Request provider-level trust configuration so cross-provider model fallback can distinguish text fallback trust from image and attachment forwarding trust.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/57959 or a narrower replacement with one provider media-trust policy covering explicit images, prompt-detected refs, attachment refs, config/docs, tests, security review, and upgrade behavior.

## Reproduction Assessment

Yes, for source-level reproduction. Current main and v2026.5.20 have cross-provider fallback paths and still call prompt image/attachment loading without a provider media-trust gate; I did not run a live provider failover in this read-only review.

## Solution Assessment

Unclear. A configurable provider media-trust policy is plausible, but maintainers still need to choose the compatibility default, fail-closed behavior, docs/config surface, and upgrade path before this is settled.
