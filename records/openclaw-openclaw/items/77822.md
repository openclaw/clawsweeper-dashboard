## Summary

Keep open. The report is plausible and includes an external mem0 workaround, but current source shows this touches the plugin lifecycle contract rather than a cleanup-close case: OpenClaw has multiple registration modes plus tool-discovery loading, while the SDK also exposes session/run context seams that may be the intended fix path.

## What This Changes

The issue requests a plugin architecture state-sharing mechanism or single-register lifecycle guarantee because an external mem0 plugin observed hook/tool closure state isolation.

## Best Possible Solution

Define the public SDK contract for this case: either document and test request-scoped state via tool factory context/session extensions/run context, or add a generic shared-state seam if maintainers want plugins to coordinate across registration instances.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows separate full and tool-discovery registration paths, and the reporter supplied an external patch, but there is no local mem0 fixture or live run proving the exact failure on current main.

## Solution Assessment

Unclear. Ensuring register(api) is called only once per session is probably not the narrowest maintainable answer because current architecture intentionally separates discovery and runtime modes; a safer path is to clarify or extend the official SDK state/context seams after maintainer review.
