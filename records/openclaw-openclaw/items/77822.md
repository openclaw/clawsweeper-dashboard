## Label Justifications

- `P2`: This is a normal-priority plugin runtime bug with real external memory-plugin impact but limited blast radius to hook-plus-tool plugin state.
- `impact:data-loss`: The reported mem0 symptoms include auto-capture not saving memories and ghost-write behavior where memory data does not land where users expect.
- `impact:session-state`: The reported root cause is per-session state splitting between hook and tool registration instances.

## Summary

Keep this issue open: current main and the latest release still resolve cached descriptor-backed plugin tools before the live registry path, while the linked implementation PR remains open and unmerged.

## What This Changes

The issue asks the core plugin runtime to preserve shared state between hook and tool registrations, or expose a supported state mechanism, for hook-plus-tool plugins such as @mem0/openclaw-mem0.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/78075; if it is rejected, document the supported state-sharing contract for mixed hook-and-tool plugins.

## Reproduction Assessment

Yes, at source level: current main can return cached descriptor-backed tools before the live registry path while hooks are stored on registry entries. I did not run a live mem0 gateway reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer API-contract judgment. The linked resolver fix is narrow for preserving same-registration closure state, while an explicit state API remains the safer alternative if closure sharing should not be a plugin contract.
