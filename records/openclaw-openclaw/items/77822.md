## Summary

Current main already provides first-class plugin context and state seams for this use case, so plugin code no longer needs cross-register closure sharing to correlate agent/session/run state.

## What This Changes

The issue asks for a shared plugin state mechanism or single-register lifecycle so external memory plugins can correlate prompt hooks, tools, and post-turn capture.

## Best Possible Solution

Keep the current plugin API contract and steer external memory plugins to use hook/tool context, run context, or session extensions instead of relying on closure sharing across registration calls.

## Reproduction Assessment

No high-confidence end-to-end reproduction of `@mem0/openclaw-mem0` is available in this repo. Source inspection does confirm the relevant lifecycle and shows current main already exposes supported context/state surfaces for the reported correlation problem.

## Solution Assessment

Yes. First-class hook/tool context and namespaced run/session state are a narrower and more maintainable solution than promising singleton `register()` semantics across discovery, activation, and request-scoped registry loads.
