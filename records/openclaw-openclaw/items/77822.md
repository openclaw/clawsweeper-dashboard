- `impact:data-loss`: The reported mem0 symptoms include auto-capture not saving memories and writes not landing where users expect.

## Summary

Keep this issue open: current main and v2026.6.8 still have the cached-descriptor-first plugin tool path that can split hook and tool closure state, and the linked implementation candidate remains open and unmerged.

## What This Changes

The issue reports that repeated plugin `register()` calls split closure state between `before_prompt_build` hooks and plugin tools, breaking @mem0/openclaw-mem0 session/run memory behavior.

## Best Possible Solution

Review and land or supersede https://github.com/openclaw/openclaw/pull/78075; if maintainers reject closure preservation, document and implement the supported shared-state contract for mixed hook-and-tool plugins.

## Reproduction Assessment

Yes, at source level: current main can return cached descriptor-backed tools before the live registry path while hooks are registered on registry entries. I did not run a live mem0 gateway reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer API-contract judgment. The linked resolver fix is narrow for preserving same-registration closure state, while an explicit state API remains the safer alternative if closure sharing should not be a plugin contract.
