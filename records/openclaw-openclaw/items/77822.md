## Summary

Keep this open: current main and v2026.6.10 still have the cached-descriptor-first plugin tool path that can split hook and tool closure state, and the linked implementation PR remains open and unmerged.

## What This Changes

The issue reports that repeated plugin `register()` calls split closure state between `before_prompt_build` hooks and `memory_add` tools, breaking `@mem0/openclaw-mem0` session/run memory behavior.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/78075; if closure identity is rejected as a contract, define a supported shared-state API for mixed hook-and-tool plugins instead.

## Reproduction Assessment

Yes, at source level: current main can return cached descriptor-backed tools before the live registry path whose hooks own closure state. I did not run a live mem0 gateway reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer API-contract judgment. The linked resolver fix is a plausible narrow repair, but maintainers still need to decide whether closure identity or an explicit shared-state API is the best supported plugin contract.
