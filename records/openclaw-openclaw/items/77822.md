- `P2`: This is a normal-priority plugin runtime bug with concrete impact on one memory plugin path and a narrow open fix candidate.
- `impact:data-loss`: The report says auto-captured memories are not saved after conversations and related comments describe successful writes leaving expected storage empty.
- `impact:session-state`: The reported failure is closure/session state splitting between `before_prompt_build` and `memory_add`, leaving `run_id` and per-agent memory isolation wrong.

## Summary

Current main and the latest release still resolve cached plugin-tool descriptors before the live registry path, so the reported hook/tool closure split is not implemented away. This issue should stay open because https://github.com/openclaw/openclaw/pull/78075 explicitly owns the fix with closing syntax, but that PR is still open, conflicting, and missing real behavior proof.

## What This Changes

The issue asks OpenClaw to prevent repeated plugin `register()` materialization from separating hook state from tool state for hook-and-tool plugins such as `@mem0/openclaw-mem0`.

## Best Possible Solution

Land or supersede the linked resolver fix with a narrow hook-and-tool live-registry path that preserves one live `register()` closure while keeping descriptor caching for stateless plugins.

## Reproduction Assessment

Yes at source level: current main resolves cached descriptor-backed tools before live registry materialization, matching the reported split between a hook closure and a later tool factory closure. No live mem0 gateway reproduction was run in this read-only review.

## Solution Assessment

Likely yes: the best direction is a resolver-side fix that uses an already-live hook-and-tool registry instead of adding a new public shared-state API. The open PR appears to target that boundary, but it needs conflict resolution and real behavior proof before this issue can close.
