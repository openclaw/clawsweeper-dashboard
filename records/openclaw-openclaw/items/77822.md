## Summary

Keep open: current main still has the cached descriptor tool path implicated by the report, and https://github.com/openclaw/openclaw/pull/78075 is an open implementation candidate that uses closing syntax for this issue. The issue should remain open until that PR lands, is superseded by a replacement, or maintainers decide the mem0-specific behavior belongs elsewhere.

## What This Changes

Plugin hook and tool registrations can end up using separate `register()` closures, breaking closure-held state for plugins such as `@mem0/openclaw-mem0`.

## Best Possible Solution

Review, validate, and land or supersede https://github.com/openclaw/openclaw/pull/78075 with real plugin behavior proof, keeping any mem0-specific database fallback fix in the plugin if it proves separate.

## Reproduction Assessment

Yes at source level, but not from a live mem0 setup in this review. The current resolver can serve cached descriptor wrappers before the live hook+tool registry path, and the open PR adds a focused regression test for a warmed cache plus a live registry with `before_prompt_build` and a tool.

## Solution Assessment

Yes for triage: keeping the issue open behind the active fix PR is the right path. The narrowest maintainable code direction is to preserve a shared live `register()` closure for hook+tool plugins rather than adding mem0-specific core policy.
