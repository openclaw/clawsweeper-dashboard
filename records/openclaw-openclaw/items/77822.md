## Label Justifications

- `P2`: The report describes a real plugin-runtime bug affecting external memory plugin behavior, but the blast radius is limited to plugins that share state between hooks and tools.
- `impact:data-loss`: The reported mem0 behavior includes auto-capture not saving memories and memory writes not landing where users expect.
- `impact:session-state`: The root problem is per-run/session state split between hook and tool registration instances.

## Summary

Keep this issue open: the reported hook/tool closure split is not fixed on current main, and the live implementation PR that explicitly closes this issue remains open and unmerged. Current main still resolves cached descriptor-backed tools before falling back to live plugin registry materialization, which is the path implicated by the mem0 hook-plus-tool state report.

## What This Changes

The issue reports that repeated plugin `register(api)` materialization splits closure state between plugin hooks and tools, breaking the external mem0 memory plugin.

## Best Possible Solution

Review and either land or explicitly reject https://github.com/openclaw/openclaw/pull/78075; if rejecting it, document that hook-plus-tool plugins should use `api.runContext` instead of relying on shared closure state.

## Reproduction Assessment

Yes at source level: current main can use cached descriptor-backed tools before live registry materialization, while hooks are registered on live plugin registry entries. I did not run a live mem0 reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer API-contract judgment. The linked resolver fix is a narrow way to preserve same-registration closure state, while current `api.runContext` is the existing supported state-sharing path if maintainers do not want closure sharing to be a plugin contract.
