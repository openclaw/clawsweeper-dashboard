## Summary

Keep this issue open: current main still has the cached descriptor-backed plugin tool path that can bypass the live hook registry, and an open implementation PR already uses closing syntax for this issue. The issue should close only after that PR, or a superseding fix, lands with suitable validation.

## What This Changes

The issue asks OpenClaw to preserve shared plugin state between hooks such as `before_prompt_build` and tools such as `memory_add` when plugin registration is reached more than once.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/78075 with a narrow resolver fix that keeps hook handlers and tool factories on the same live plugin registration path, plus regression coverage and real plugin behavior proof.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection gives a clear reproduction path: warm the plugin tool descriptor cache, keep a live registry with the same plugin's hooks and tools, then resolve tools again and observe the cached wrapper path winning before the live registry path.

## Solution Assessment

Yes in direction: the linked resolver-side PR is the narrow maintainable fix because it preserves the live hook/tool closure instead of adding mem0-specific core policy. It should still be reviewed and proven against a real plugin setup before merge.
