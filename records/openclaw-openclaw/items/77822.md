## Summary

Keep open: current main and the latest release still have the cached descriptor-backed tool path implicated by the report, and https://github.com/openclaw/openclaw/pull/78075 is an open implementation PR with closing syntax for this issue.

## What This Changes

The issue asks OpenClaw to stop repeated plugin `register()` calls from splitting closure state between `before_prompt_build` hooks and tools such as `memory_add`.

## Best Possible Solution

Keep this issue open while maintainers review, prove, land, or supersede https://github.com/openclaw/openclaw/pull/78075; the fix should preserve one live `register()` closure for hook handlers and tool factories without adding a broad shared-state SDK unless maintainers choose that contract.

## Reproduction Assessment

Yes, at source level: current main can return cached descriptor-backed tools before using the live registry, which matches the reported split between a live `before_prompt_build` hook and a later tool factory. I did not run a live mem0 gateway reproduction in this read-only review.

## Solution Assessment

Yes for the current direction: the open resolver-side fix path in https://github.com/openclaw/openclaw/pull/78075 is narrower than adding a new shared-state SDK. This issue should close only after that PR, or a superseding fix, lands with adequate proof.
