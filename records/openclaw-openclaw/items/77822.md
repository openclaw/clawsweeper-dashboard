## Summary

Keep this open: current main and the latest release still have the cached-descriptor-first plugin tool path that can split hook and tool closure state, and the linked implementation candidate remains open and unmerged.

## What This Changes

The issue reports repeated plugin register() calls splitting closure state between hooks and tools, breaking mem0 memory/session behavior.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/78075; if closure identity is rejected as a contract, define the supported shared-state API for mixed hook-and-tool plugins instead.

## Reproduction Assessment

Yes, at source level: current main can return cached descriptor-backed tools before the live runtime registry path while hooks are stored on registry entries. I did not run a live mem0 gateway reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer API-contract judgment. The linked resolver fix is narrow for same-registration closure preservation, while an explicit shared-state API may be better if closure sharing should not be guaranteed.
