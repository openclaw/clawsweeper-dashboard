## Summary

Keep this issue open: current main and v2026.6.6 still have the cached-descriptor-first plugin tool path implicated by the hook/tool closure split, and the only PR that explicitly closes this issue remains open and unmerged. The later mem0 comments add useful evidence that the failure may also surface as wrong-path or ghost memory writes, but they do not make the core runtime question resolved.

## What This Changes

The issue requests a plugin-runtime contract fix or shared-state mechanism for plugins whose hooks and tools currently observe different `register()` closure instances.

## Best Possible Solution

Review and either land or explicitly reject https://github.com/openclaw/openclaw/pull/78075; if rejecting it, document that hook-plus-tool plugins should use `api.runContext` instead of relying on shared closure state.

## Reproduction Assessment

Yes at source level: current main can use cached descriptor-backed tools before live registry materialization, while hooks are stored on live registry entries. I did not run a live mem0 reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer API-contract judgment. The linked resolver PR is a narrow implementation fix, while `api.runContext` is the existing documented state-sharing path if closure sharing should not become a plugin contract.
