## Summary

Keep open. Current main contains partial OpenAI-Codex transport/auth cleanup, but the broader stale-route problem is still source-reproducible: stored direct or parent session overrides can still win over a changed default when no model allowlist rejects them, and the current docs explicitly treat both PI `openai-codex/*` and native Codex runtime routes as valid routes needing policy-sensitive handling.

## What This Changes

The issue reports stale Codex/OpenAI-Codex provider and model routes being selected in message-channel or embedded fallback runs after configuration has moved to another model route.

## Best Possible Solution

Keep this as the umbrella unless maintainers split it, and land a policy-backed cleanup/regression suite that prevents stale automatic or legacy Codex state from silently winning after a default/runtime change while preserving explicit supported Codex routes.

## Reproduction Assessment

Yes, source-level. Seed a direct or parent session entry with `providerOverride: "openai-codex"` and `modelOverride: "gpt-5.4"` while the configured default points elsewhere and has no rejecting allowlist; current selection code will still choose the stored route.

## Solution Assessment

Unclear. There is no patch under review, and the safest fix is not blanket Codex deletion or alias normalization but a maintainer-approved stale-state policy with targeted model-selection, reset/doctor, and provider-routing coverage.
