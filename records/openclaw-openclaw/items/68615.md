## Summary

Keep open. Current main has partial hardening, but the broader stale route/state behavior is still present in code: persisted session or parent overrides can still win over a changed configured default, reset cleanup is intentionally source-limited, and the disabled-plugin loader concern remains observable.

## What This Changes

The issue requests that message-channel and embedded fallback runs stop selecting stale `codex` or `openai-codex` routes after the configured/default model has moved to a proxy or other provider.

## Best Possible Solution

Keep this as the umbrella unless maintainers split it. The desired end state is a focused regression suite and policy-backed cleanup that prevents stale auto/legacy Codex state from winning after a default route change while preserving explicit user selections and the documented `openai-codex/*` versus native Codex runtime split.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can seed a session or parent session with a stale `providerOverride: openai-codex` and `modelOverride: gpt-5.4` while the configured default is a different provider; current selection code will still choose that override when no allowlist rejects it, matching the later embedded-fallback report.

## Solution Assessment

Unclear. There is no proposed patch under review, and the complete fix needs a narrower maintainer-approved policy for stale auto, legacy, and user overrides rather than a broad provider-name normalization or wholesale state deletion.
