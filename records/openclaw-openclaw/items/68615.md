## Summary

Keep open. Current main has useful partial fixes for OpenAI-Codex transport normalization, auto-fallback reset cleanup, and observability, but it still does not close the broader stale route/state problem reported in the issue discussion.

## What This Changes

The issue asks OpenClaw to stop message-channel and embedded fallback runs from selecting stale `codex` or `openai-codex` model/provider/auth routes after the active configured route has moved elsewhere.

## Best Possible Solution

Keep this as the umbrella unless maintainers split it. The end state should include a focused regression where a session with stale `codex` or `openai-codex` state and a new proxy/Copilot default does not attempt the stale route first, plus explicit cleanup semantics for auto overrides, auth-profile ordering, legacy `codex/*` aliases, and non-activating plugin discovery side effects.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can seed a session row with `providerOverride: openai-codex`, `modelOverride: gpt-5.4`, and `modelOverrideSource: auto` while the configured default is a different provider; current selection code will still choose the stored override when no allowlist rejects it, matching the later issue comment’s embedded fallback report.

## Solution Assessment

Unclear. There is no single proposed patch under review, and current main’s partial fixes are not the narrow complete solution; the safest path is to split policy-sensitive stale-state cleanup from loader discovery correctness and transport normalization.
