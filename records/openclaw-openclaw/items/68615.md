## Summary

Keep open. Current main still has source-backed stale-route paths: stored session or parent model overrides can select a prior Codex/OpenAI-Codex route when no allowlist rejects it, reset cleanup intentionally preserves legacy/user overrides, and non-activating plugin loads still execute `register()` before restoring side effects.

## What This Changes

Message-channel runs should stop selecting stale `codex` or `openai-codex` provider/model/auth routes after Codex is disabled and the configured default is moved to another provider route.

## Best Possible Solution

Land a policy-backed cleanup and regression suite that prevents stale automatic or legacy Codex state from winning after default-route changes while preserving explicit supported Codex/OpenAI-Codex selections.

## Reproduction Assessment

Yes. A high-confidence source reproduction can seed a session or parent session with `providerOverride: "openai-codex"` and `modelOverride: "gpt-5.4"` while the configured default points elsewhere; current selection code will still choose that stored override unless an allowlist rejects it.

## Solution Assessment

Unclear. There is no patch under review, and the narrowest maintainable fix needs maintainer-approved stale-state policy rather than blanket normalization or deletion of all Codex/OpenAI-Codex state.
