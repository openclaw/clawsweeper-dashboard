## Summary

Keep open. Current main has partial Codex/OpenAI-Codex hardening, but the umbrella stale-route behavior is still supported by source and tests: stored session or parent overrides can still beat a changed default, non-activating plugin loads still execute `register()` before restoration, and the discussion includes a later concrete reproduction on `openclaw@2026.4.25`.

## What This Changes

The issue asks OpenClaw to stop message-channel and embedded-agent runs from selecting stale `codex` or `openai-codex` routes after the configured model route changes away from Codex and the Codex plugin is disabled.

## Best Possible Solution

Keep this as the umbrella unless maintainers split it, and land a policy-backed cleanup/regression suite that prevents stale automatic or legacy Codex state from winning after a default route change while preserving explicit supported Codex routes.

## Reproduction Assessment

Yes. A high-confidence reproduction can seed a session or parent session with `providerOverride: "openai-codex"` and `modelOverride: "gpt-5.4"` while the configured default points elsewhere; current selection code will still choose the stored override when no allowlist rejects it, and the issue discussion includes a later live reproduction.

## Solution Assessment

Unclear. There is no patch under review, and the narrowest maintainable fix needs maintainer-approved stale-state policy rather than broad `codex`/`openai-codex` normalization or wholesale state deletion.
