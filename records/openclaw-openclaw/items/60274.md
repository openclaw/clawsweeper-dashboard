## Summary

Keep open. Current main still lacks an iMessage plugin tapback filter in the imsg RPC inbound decision path, and open PR #60677 explicitly targets this issue, so conservative cleanup should wait for that PR or an equivalent fix to land.

## What This Changes

Filter imsg RPC tapback/reaction messages before they are delivered to the iMessage agent as normal text.

## Best Possible Solution

The iMessage plugin should have a small owner-local guard that recognizes imsg RPC tapback/reaction payloads and drops them before pairing or agent dispatch, with regression tests for explicit metadata and Loved/Liked-style quoted text; full reaction-event forwarding should remain tracked separately by #39031.

## Reproduction Assessment

Yes. A code-level reproduction is clear: pass a valid inbound imsg RPC message with sender text like `Loved "hello"` through `resolveIMessageInboundDecision()` under an allowed DM policy and current main can dispatch it because there is no tapback branch; live iMessage testing was not run.

## Solution Assessment

Yes. Keeping this issue open is the best cleanup action because #60677 is an active implementation candidate that explicitly targets the bug; the narrow maintainable fix belongs in `extensions/imessage`, not in core or in the broader reaction-event feature issue.
