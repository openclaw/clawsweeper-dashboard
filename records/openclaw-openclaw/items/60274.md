## Summary

Keep open. Current main still lets imsg RPC tapback/reaction inputs enter the ordinary iMessage inbound dispatch path, and two open implementation PRs already reference this issue with closing syntax, so the issue should wait for one of those fixes or an equivalent owner-local replacement to land.

## What This Changes

Request that iMessage imsg RPC tapback reactions stop reaching the agent as ordinary messages, either by filtering them or by routing them as reaction events.

## Best Possible Solution

Land or replace the protected reaction-event PR with an iMessage plugin fix that classifies imsg reaction metadata and legacy tapback text before ordinary dispatch; keep broader inbound reaction policy aligned with https://github.com/openclaw/openclaw/issues/39031.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: an allowed imsg RPC reaction payload is converted into normal message text in monitor-provider, and resolveIMessageInboundDecision has no reaction branch before returning dispatch; a legacy body such as Loved "hello" follows the same terminal dispatch path.

## Solution Assessment

Unclear as a final product choice. Filtering tapbacks would fix the immediate bug, while https://github.com/openclaw/openclaw/pull/80713 attempts the fuller reaction-event path; maintainers should land or replace one path rather than closing this issue early.
