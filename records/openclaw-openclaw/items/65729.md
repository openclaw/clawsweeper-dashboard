## Summary

Keep this open. The Mattermost WebSocket keepalive slice is implemented and shipped, but current main still keeps Mattermost DMs non-threaded by contract, suppresses DM thread reply/history/mention paths, and has no Mattermost thread-history backfill; the remaining request needs an owner/product decision rather than conservative cleanup closure.

## What This Changes

The issue requests native Mattermost threading reliability improvements covering WebSocket keepalive, configurable DM threading parity, persisted/backfilled thread history, and bare mention handling.

## Best Possible Solution

Keep the shipped keepalive behavior, then split the remaining work into an explicit opt-in Mattermost DM threading/history contract and a separate bare-mention wake-up fix with docs, schema, and focused tests.

## Reproduction Assessment

Yes. Source inspection on current main is a high-confidence path: direct chats still force replyToMode off, suppress effective reply targets/history/mention detection, and drop a mention-only normalized body; I did not run a live Mattermost server.

## Solution Assessment

No current implementation fully solves the issue. Applying the workaround as-is would override a documented and tested flat-DM contract; the maintainable route is an explicit opt-in DM threading/backfill design plus a narrow bare-mention fix.
