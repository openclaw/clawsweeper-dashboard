## Summary

Keep this open. Current main implements the Mattermost WebSocket ping/pong keepalive portion, but the broader threading request remains unresolved: direct messages are still documented and tested as non-threaded, DM history is disabled, thread-history backfill is absent, and mention-only posts still return early after normalization.

## What This Changes

The issue requests native Mattermost threading reliability improvements: WebSocket keepalive, DM replyToMode parity with thread-scoped history, thread-history backfill after restarts, and bare-mention wake-up handling.

## Best Possible Solution

Keep this as Mattermost-plugin-owned reliability work: retain flat DMs by default, add an explicit opt-in DM/threading configuration if maintainers want that mode, define access-control rules before any thread-history backfill, accept mention-only wake-up invocations, and update schema, tests, docs, and changelog together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: direct chats still force replyToMode/effectiveReplyToId/historyKey off regardless of config, and a post containing only the bot mention normalizes to an empty body and returns early. The WebSocket keepalive subcase no longer reproduces because current code sends protocol pings and terminates on missed pongs.

## Solution Assessment

No current implementation fully solves this issue. The fixed keepalive is the right direction for one subcase, but the remaining DM threading and backfill work should be solved with an explicit opt-in contract and access-control review rather than by silently changing existing DM defaults.
