## Summary

Keep open: current main still has a source-level path where manifest model-id normalization can reject a valid workspace-scoped Gateway metadata snapshot and fall back to repeated plugin metadata loading, and fresh 2026.5.7 comments report the same latency family on Gateway and status paths.

## What This Changes

The issue reports repeated plugin metadata rescans during gateway model normalization instead of reusing a prepared workspace-scoped metadata snapshot.

## Best Possible Solution

Land a narrow snapshot-reuse fix for manifest model-id normalization and, if needed, the broader Gateway cold-slot refill/provider-runtime snapshot threading from the existing related PRs while preserving the documented no-persistent-metadata-cache boundary.

## Reproduction Assessment

Yes, source-level with high confidence: publish a workspace-scoped current plugin metadata snapshot, leave the active plugin registry workspace unset, and call `normalizeProviderModelIdWithManifest()` without an explicit workspace. Current main rejects that snapshot and falls back to `loadPluginMetadataSnapshot()`.

## Solution Assessment

Yes for the requested direction: using the existing `allowWorkspaceScopedSnapshot`/single-slot snapshot handoff is the narrow maintainable fix, while the broader open PRs should be reviewed for the additional cold-slot and provider-runtime paths before introducing any new cache layer.
