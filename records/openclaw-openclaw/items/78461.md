## Summary

Keep open: current main still has a source-level model-normalization path that can miss a compatible Gateway-owned metadata snapshot and fall back to cold plugin metadata loading; the exact narrow fix was closed unmerged, and the overlapping open fix has not landed.

## What This Changes

The issue asks Gateway model-id normalization to reuse the Gateway-owned plugin metadata snapshot instead of reloading plugin/index metadata during embedded agent attempts.

## Best Possible Solution

Land or supersede the open normalization snapshot-reuse work, preserving CLI freshness and the documented single-slot snapshot boundary, then close this issue after the fix merges and release provenance is known.

## Reproduction Assessment

Yes, source-level. Store a workspace-scoped current snapshot, leave the active plugin runtime workspace unset, and call `normalizeProviderModelIdWithManifest()` without `workspaceDir`; current main omits `allowWorkspaceScopedSnapshot` and falls back to `loadPluginMetadataSnapshot()`.

## Solution Assessment

Yes, the maintainable direction is to reuse the existing Gateway-owned snapshot handoff for model normalization and cover the no-active-workspace case with focused regression tests. A broad process-level TTL cache would conflict with the documented plugin cache boundary and should not be the first fix.
