## Summary

Keep open: current main still has a source-level path where manifest model-id normalization can reject a compatible Gateway-owned workspace-scoped metadata snapshot and fall back to `loadPluginMetadataSnapshot()`. The issue also has fresh 2026.5.7 reproductions and overlapping open PRs, so it should stay open until the fix path lands rather than being swept as implemented.

## What This Changes

The issue asks OpenClaw to stop reloading plugin metadata during Gateway model normalization by reusing compatible Gateway-owned metadata snapshots.

## Best Possible Solution

Land a narrow fix that makes manifest model-id normalization reuse compatible workspace-scoped current snapshots and keeps Gateway-flow snapshot handoff warm without adding persistent plugin-discovery caches.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: store a workspace-scoped current plugin metadata snapshot, leave the active plugin-registry workspace unset or clear the single slot after an installed-index write, then call manifest model-id normalization and observe the cold `loadPluginMetadataSnapshot()` fallback path.

## Solution Assessment

Yes for the issue’s direction, but it is not implemented on current main. Reusing the existing compatible snapshot handoff matches adjacent model-catalog and manifest-contract readers; the open PRs should be reviewed or consolidated rather than adding a new duplicate repair branch.
