## Summary

Keep open: current main still has the manifest model-normalization path that can reject a workspace-scoped current metadata snapshot and fall back to `loadPluginMetadataSnapshot()`. Recent issue comments report the same class of regression on 2026.5.7, and the open related PR https://github.com/openclaw/openclaw/pull/78589 is the active fix path rather than a reason to close this issue early.

## What This Changes

Gateway model normalization should reuse the Gateway-owned plugin metadata snapshot instead of rescanning plugin manifests and indexes on hot paths.

## Best Possible Solution

Keep this issue as the canonical bug until a compatibility-preserving snapshot reuse fix, likely the open related PR or a narrower replacement, lands with targeted regression coverage and runtime proof.

## Reproduction Assessment

Yes, source-level: store a workspace-scoped current metadata snapshot, leave the active runtime workspace unset, then call manifest model normalization without an explicit workspace; current main rejects the snapshot and selects the metadata-loader fallback. Live comments add current-release evidence on 2026.5.7, but I did not run the live performance repro in this read-only pass.

## Solution Assessment

Yes, the maintainable direction is to reuse the existing current-snapshot compatibility contract for unscoped manifest normalization and keep explicit workspace freshness checks intact. The open related PR should be reviewed as the active implementation path before starting a duplicate repair branch.
