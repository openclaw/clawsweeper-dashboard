## Summary

Keep open: current main still has the manifest model-id normalization path that can reject a compatible workspace-scoped current plugin metadata snapshot and fall back to `loadPluginMetadataSnapshot()`. The exact fix PR was closed unmerged, while the open related PRs address adjacent provider-runtime or cold-slot refill paths rather than the unscoped workspace-snapshot miss itself.

## What This Changes

The issue requests reusing Gateway-owned plugin metadata snapshots during manifest model-id normalization to avoid repeated plugin index and manifest discovery.

## Best Possible Solution

Land or recreate the narrow manifest-normalization workspace-snapshot reuse fix, then handle the broader cold-slot refill and provider-runtime scan reductions in their separate related PRs.

## Reproduction Assessment

Yes, source-level: store a workspace-scoped current metadata snapshot, leave active plugin-registry workspace state unset, and call `normalizeProviderModelIdWithManifest()` without `workspaceDir`. Current main does not pass `allowWorkspaceScopedSnapshot`, so the compatible snapshot is rejected and the cold metadata loader path remains reachable.

## Solution Assessment

Yes, the best narrow fix is to apply the existing unscoped-reader opt-in pattern from model catalog to manifest model-id normalization while preserving explicit workspace checks and fallback freshness. A broad process-level TTL cache would be a riskier solution for this specific miss.
