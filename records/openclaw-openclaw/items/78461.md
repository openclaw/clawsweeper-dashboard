## Summary

Keep open: current main still has a source-level metadata snapshot miss in manifest model-id normalization, and recent issue comments show the same performance problem broader than the original narrow fallback path.

## What This Changes

The issue asks OpenClaw to stop repeated plugin metadata/index rescans during Gateway and status model normalization by reusing compatible plugin metadata snapshots or prepared registry data.

## Best Possible Solution

Consolidate the open snapshot-reuse work so Gateway/status/provider model paths reuse compatible prepared metadata while preserving the documented CLI freshness boundary.

## Reproduction Assessment

Yes, source-level: store a current snapshot with `workspaceDir`, leave active plugin runtime workspace unset, then call manifest model normalization without an explicit workspace. Current main rejects that snapshot and selects the cold metadata load path; live comments corroborate the performance symptom on 2026.5.7.

## Solution Assessment

Yes for the narrow path: use the existing workspace-scoped snapshot opt-in for unscoped normalization and add regression coverage. The broader reported latency should be solved by carrying or republishing prepared metadata through Gateway/status flows, not by adding ad hoc TTL caches.
