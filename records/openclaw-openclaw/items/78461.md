## Summary

Keep open: current main still has source-level model-normalization paths that can miss the current plugin metadata snapshot and fall back to cold metadata loading, and the later comments broaden the same performance problem across provider and status/session surfaces. Related PRs cover pieces of the cluster, but no merged current-main change closes the central issue yet.

## What This Changes

The issue requests eliminating repeated plugin metadata, manifest-registry, and installed-index rescans during gateway model normalization and related status/session model resolution paths.

## Best Possible Solution

Keep this as the canonical performance tracker and consolidate the active narrow fixes so model/provider/status paths either reuse explicit current metadata snapshots or avoid manifest normalization on shallow status paths, without adding broad persistent discovery caches.

## Reproduction Assessment

Yes, source-level: publish a workspace-scoped current metadata snapshot, leave the caller unscoped or without an active plugin-registry workspace, then call manifest model-id normalization; current main rejects the compatible snapshot and selects the loadPluginMetadataSnapshot fallback. I did not run a live timing/profile reproduction in this read-only review.

## Solution Assessment

No single landed solution exists yet. The unscoped manifest-normalization fix is narrow, but the newer comments and open PRs show the best maintainable fix likely needs coordinated landing across manifest normalization, provider runtime snapshot threading, and status/session cold paths.
