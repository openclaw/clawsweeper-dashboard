## Summary

Keep open: current main still has a source-level model-normalization path that can miss a compatible workspace-scoped current plugin metadata snapshot and fall back to plugin metadata loading. The gateway setter side is now workspace-aware, but the unscoped manifest-normalization reader still lacks the existing `allowWorkspaceScopedSnapshot` fast-path used by adjacent readers, and recent production comments show active related latency/CPU reports.

## What This Changes

The issue asks OpenClaw to reuse Gateway-owned plugin metadata snapshots during manifest model-id normalization instead of repeatedly reloading plugin metadata/index state.

## Best Possible Solution

Keep this as the canonical tracker until manifest model-id normalization consistently reuses compatible Gateway-owned metadata snapshots and the remaining status/idle-gateway scan reports are either fixed or split into narrower linked issues.

## Reproduction Assessment

Yes, source-level: publish a workspace-scoped current metadata snapshot, leave the active runtime workspace unset, then call `normalizeProviderModelIdWithManifest()` without `workspaceDir`; current main does not pass `allowWorkspaceScopedSnapshot`, so the helper rejects the snapshot and the cold metadata loader path remains reachable.

## Solution Assessment

No for closure: current main only contains the Gateway setter-side fix, while the unscoped manifest-normalization reader and related scan hot paths remain unresolved. The narrow existing-reader opt-in is likely right for the original miss, but the broader issue should be coordinated with the open related PRs.
