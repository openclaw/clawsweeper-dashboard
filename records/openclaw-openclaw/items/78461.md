## Summary

Keep open: current main still has a source-level manifest model-id normalization path that can miss the Gateway-owned workspace-scoped metadata snapshot, and newer 2026.5.7 comments show related gateway/status rescans still affecting users.

## What This Changes

The issue asks OpenClaw to stop repeatedly reloading plugin metadata, manifest registries, and installed indexes during gateway model normalization and related status/model-resolution paths.

## Best Possible Solution

Consolidate the open snapshot-reuse fixes, prefer explicit per-flow snapshot or manifest-registry threading over persistent TTL caches, and land focused coverage for gateway normalization plus status session-summary cold paths.

## Reproduction Assessment

Yes, source-level: store a workspace-scoped current metadata snapshot, leave the active plugin registry workspace unset, then call `normalizeProviderModelIdWithManifest()` without `workspaceDir`; current main rejects the snapshot and falls back to `loadPluginMetadataSnapshot()`. Live commenters also reproduced 2026.5.7 gateway/status slowness around the same metadata rescan family.

## Solution Assessment

Unclear as one isolated patch: adding the unscoped workspace-snapshot opt-in to manifest normalization is still needed, but comments and open PRs show additional provider, status, and inventory hot paths that should be coordinated before landing.
