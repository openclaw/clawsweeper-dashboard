## Summary

Keep open: current main still has the source-level model-normalization snapshot miss, and this issue is also being used as the canonical tracker for the broader plugin metadata rescan family.

## What This Changes

The issue asks Gateway model normalization to reuse the Gateway-owned plugin metadata snapshot instead of repeatedly falling back to plugin metadata discovery.

## Best Possible Solution

Fold the narrow unscoped manifest-normalization snapshot reuse fix into the active plugin metadata performance work, then keep broader discovery, realpath, and status-specific issues tracked separately where their triggers differ.

## Reproduction Assessment

Yes, source-level: store a workspace-scoped current metadata snapshot, leave the active plugin runtime workspace unset, and call normalizeProviderModelIdWithManifest without workspaceDir. Current main calls getCurrentPluginMetadataSnapshot without allowWorkspaceScopedSnapshot, rejects that snapshot, and can fall back to loadPluginMetadataSnapshot.

## Solution Assessment

Yes for the original miss: the closed PR’s approach of normalizing omitted config, passing allowWorkspaceScopedSnapshot only for unscoped callers, and avoiding workspaceDir: undefined matches existing model-catalog and manifest-contract readers. The open lazy-republish PR is useful adjacent work, but it does not by itself cover the no-active-workspace snapshot miss.
