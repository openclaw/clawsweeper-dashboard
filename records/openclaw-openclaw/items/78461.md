## Summary

Keep open: current main still has a source-level model-normalization path that can reject a compatible Gateway-owned workspace-scoped plugin metadata snapshot and fall back to a cold metadata scan; recent comments add 2026.5.7 live impact rather than resolving it.

## What This Changes

The issue asks OpenClaw to reuse Gateway-owned plugin metadata snapshots during manifest model-id normalization instead of repeatedly re-scanning plugin metadata on hot paths.

## Best Possible Solution

Land a focused metadata snapshot reuse fix for manifest model-id normalization, coordinating with the open lazy-republish and provider-runtime snapshot PRs rather than adding a broad persistent cache.

## Reproduction Assessment

Yes, source-level: store a workspace-scoped current metadata snapshot, leave the active runtime workspace unset, then call `normalizeProviderModelIdWithManifest()` without `workspaceDir`; current main rejects the snapshot and selects `loadPluginMetadataSnapshot()`. The 2026.5.7 comments provide live impact, but I did not rerun a Gateway profile here.

## Solution Assessment

Yes for the known missing caller: reusing the existing current-snapshot compatibility path for unscoped manifest model-id normalization is the narrow maintainable fix. Broader process-level memo caches should stay out unless maintainers choose a wider plugin metadata strategy.
