## Summary

Keep open: current main still has a source-level path where manifest model-id normalization can miss the Gateway-owned workspace-scoped plugin metadata snapshot and fall back to a cold plugin metadata scan.

## What This Changes

The issue asks OpenClaw to reuse the Gateway plugin metadata snapshot during model alias and manifest model-id normalization to avoid repeated plugin index and manifest discovery.

## Best Possible Solution

Add the existing unscoped-reader snapshot reuse pattern to manifest model-id normalization, preserve explicit workspace compatibility checks, and cover the no-active-workspace/current-workspace-snapshot case with a focused regression test.

## Reproduction Assessment

Yes, source-level: store a current metadata snapshot with `workspaceDir`, leave the active plugin runtime workspace unset as Gateway startup can before runtime plugin load, then call `normalizeProviderModelIdWithManifest()` without an explicit workspace. Current main rejects the snapshot and selects `loadPluginMetadataSnapshot()`.

## Solution Assessment

Yes, with a small refinement: the narrow maintainable fix is the manifest-normalization opt-in to `allowWorkspaceScopedSnapshot` for unscoped callers; passing `workspaceDir` at Gateway setter sites alone is not sufficient on current source because the snapshot already carries its workspace.
