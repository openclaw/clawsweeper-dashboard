## Summary

Keep open: current main still has a source-level path where manifest model-id normalization can miss the Gateway-owned workspace-scoped metadata snapshot and fall back to a cold plugin metadata scan.

## What This Changes

The issue requests reusing the current Gateway plugin metadata snapshot during model alias and manifest model-id normalization instead of rediscovering plugin metadata on embedded agent attempts.

## Best Possible Solution

Add the existing unscoped-reader snapshot reuse pattern to manifest model-id normalization, keep compatibility checks intact, and cover the no-active-workspace/current-workspace-snapshot case with a focused regression test.

## Reproduction Assessment

Yes, source-level: store a current metadata snapshot with `workspaceDir`, leave the active plugin runtime workspace undefined as Gateway startup can before runtime plugin load, then call `normalizeProviderModelIdWithManifest()` without an explicit workspace. Current main rejects the snapshot and selects `loadPluginMetadataSnapshot()`.

## Solution Assessment

Yes for the manifest-normalization opt-in: using `allowWorkspaceScopedSnapshot` only when the caller did not explicitly request a workspace matches existing model-catalog and manifest-contract readers. Passing `workspaceDir` at the Gateway setter call sites is not sufficient by itself on current source because the setter already fingerprints with `snapshot.workspaceDir`.
