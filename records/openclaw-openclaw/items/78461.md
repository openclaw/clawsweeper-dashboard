## Summary

Keep open: current main still has a source-level path where manifest model-id normalization can miss a compatible Gateway-owned workspace-scoped metadata snapshot and fall back to cold plugin metadata loading.

## What This Changes

The issue asks Gateway model-id normalization to reuse compatible current plugin metadata snapshots instead of rescanning plugin metadata on embedded-agent/model-resolution hot paths.

## Best Possible Solution

Apply the existing unscoped-reader snapshot reuse pattern to manifest model-id normalization, then coordinate the broader provider/runtime snapshot reuse PRs without adding persistent metadata caches.

## Reproduction Assessment

Yes, source-level: store a workspace-scoped current metadata snapshot, leave active runtime workspace unset, then call `normalizeProviderModelIdWithManifest()` without `workspaceDir`; current main rejects the snapshot and selects `loadPluginMetadataSnapshot()`. I did not run a live Gateway profile in this read-only review.

## Solution Assessment

Yes for the narrow miss: manifest model-id normalization should mirror the existing model-catalog and manifest-contract `allowWorkspaceScopedSnapshot` pattern for unscoped callers. Broader cold-slot or provider-runtime rescans should be handled by reviewing the related open PRs rather than adding a new cache layer.
