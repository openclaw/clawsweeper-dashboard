## Summary

Keep open: current main still has a source-level metadata snapshot miss in manifest model-id normalization, and newer 2026.5.7 reports show broader discovery/index rescans that are not covered by a shipped fix.

## What This Changes

The issue asks OpenClaw to stop repeatedly reloading plugin metadata/index data during Gateway model normalization by reusing the Gateway-owned metadata snapshot and related registry data.

## Best Possible Solution

Have plugin/Gateway owners choose architecture-safe slices that reuse prepared metadata snapshots and remove repeated discovery/index scans while preserving manifest freshness and CLI edit-detection behavior.

## Reproduction Assessment

Yes, source-level: a workspace-scoped current snapshot with no active runtime workspace is still rejected by manifest model-id normalization on current main, causing the cold metadata loader path. Multiple live 2026.5.7 reports also identify related discovery/index rescan paths, but I did not run those profiles locally.

## Solution Assessment

Unclear as one automated fix. The original unscoped snapshot reuse is narrow, but the latest evidence spans manifest normalization, provider runtime, status/session rows, installed-index rebuilds, and discovery realpath/fingerprint behavior, so the maintainable repair should be split or sequenced by maintainers.
