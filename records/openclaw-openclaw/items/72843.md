## Summary

Keep open: current main still has the reported gateway context-cache warmup bug, but this PR is not mergeable yet because its latest `index.js` guard misses documented relative/container gateway launches and the user-facing fix lacks a changelog entry.

## What This Changes

The PR changes agent context-cache startup gating so gateway invocations can eager-warm context-window metadata and adds focused argv recognition tests.

## Best Possible Solution

Revise `shouldEagerWarmContextWindowCache` so documented npm/global, dev-checkout, Docker/Compose, Podman, Fly, and packaged gateway daemon entries warm the cache while arbitrary plugin-sdk `index.js` scripts still do not, then add regression coverage and a single-line changelog fix.

## Reproduction Assessment

Yes. Source inspection on current main shows `node .../dist/index.js gateway` is rejected by the CLI-entry check and `gateway` is skipped, while session/status paths use synchronous lookups that fall back to 200k when the cache is cold.

## Solution Assessment

No, not as-is. Removing `gateway` from the skip list and guarding generic `index.js` consumers is the right direction, but the current ancestor-only match misses supported container and relative daemon entries and the PR needs a changelog entry.
