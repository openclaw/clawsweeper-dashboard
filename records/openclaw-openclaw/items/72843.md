## Summary

Keep open. Current main still has the reported gateway warmup gap, and this PR targets a real startup/status context-window bug, but the current patch still misses documented Docker/Fly daemon launches because the new `index.js` recognition requires an exact `openclaw` ancestor directory.

## What This Changes

The PR updates `src/agents/context.ts` to recognize selected `index.js`/`index.mjs` OpenClaw entries, removes `gateway` from the eager-warmup skip list, and adds unit coverage in `src/agents/context.test.ts`.

## Best Possible Solution

Revise the PR so eager context-window warmup recognizes all documented OpenClaw packaged gateway daemon launches while keeping generic Node/plugin-sdk consumers cold, then add regression coverage for npm-global, dev checkout, Docker/Fly relative `dist/index.js`, `/app/dist/index.js`, and generic `index.js` false-positive cases.

## Reproduction Assessment

Yes. On PR head, `shouldEagerWarmContextWindowCache(["node", "dist/index.js", "gateway"])` and the `/app/dist/index.js` variant follow the new ancestor check and return false, while Docker and Fly docs use those daemon launch shapes.

## Solution Assessment

No, not as-is. Removing `gateway` from the skip list is the right direction and the second commit fixes the bare-`index.js` false positive, but the exact-ancestor requirement is narrower than supported deployments; a packaged OpenClaw entry signal should cover container/cloud `dist/index.js` launches too.
