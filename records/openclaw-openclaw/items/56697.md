## Summary

Keep open: current main still does not include tool-specific parameter examples, so the idea is not obsolete, but this PR is not merge-ready because its edit example teaches a payload current OpenClaw rejects, it lacks the required changelog entry, and it carries dead code while the validation helper has drifted on main.

## What This Changes

The PR adds hard-coded read/write/edit usage examples to tool parameter validation errors and threads the tool name through required-parameter validation.

## Best Possible Solution

Rebase or replace the small example-guidance patch on current main, preserve the newer received-parameter hints, use canonical read/write/edit schemas, add focused regression tests and a single-line changelog entry, and leave retry-budget behavior to #68070 or its successor.

## Reproduction Assessment

Yes for the validation-message surface: source inspection shows assertRequiredParams still emits only the generic retry suffix on empty required-parameter failures. The live provider self-correction claim is not reproducible from the static PR alone.

## Solution Assessment

No, not as currently written. The example-guidance direction is narrow and maintainable, but the patch must use the current edit({"path":"...","edits":[...]}) contract, preserve current-main received hints, remove dead code, and add changelog/tests before merge.
