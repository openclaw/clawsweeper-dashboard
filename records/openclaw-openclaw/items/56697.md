## Summary

Keep open. Current main still does not add tool-specific examples for empty required-parameter failures, so the PR addresses a real remaining diagnostics gap, but the proposed patch is not merge-ready against the current tool contract: the edit example teaches a rejected payload shape, the diff introduces unused dead code, and the user-facing fix still needs a changelog entry and focused coverage.

## What This Changes

The PR changes `src/agents/pi-tools.params.ts` to append hard-coded read/write/edit usage examples to parameter validation errors and thread `toolName` into the validation-error helper.

## Best Possible Solution

Keep the narrow example-hint idea, but update it against current main's canonical schemas, especially `edit({"path":"...","edits":[{"oldText":"...","newText":"..."}]})`, add regression coverage for empty-object validation failures, and keep retry-budget/non-retryable behavior in the separate retry-loop work.

## Reproduction Assessment

Yes for the validation-message surface: `assertRequiredParams({}, REQUIRED_PARAM_GROUPS.read, "read")` on current main emits the generic missing-parameter guidance without an example. The live provider self-correction claim is still unverified and should not be treated as reproduced by this review.

## Solution Assessment

No, not as currently written. Adding examples is a plausible small fix, but the PR must match the current runtime edit schema, remove or use the dead accepted-key accumulator, and add the required changelog/test coverage before merge.
