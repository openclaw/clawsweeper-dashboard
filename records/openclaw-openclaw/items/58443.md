## Summary

Keep open. Current main contains meaningful partial fixes for this duplicate-delivery class, including generic inbound dedupe, Telegram accepted-offset persistence, Slack seen-message guards, and bounded subagent orphan recovery. But the subagent announce path still keys announce idempotency to child session plus child run id, while the strongest later reproduction describes orphan recovery/remap minting new run ids for the same logical completion, so the umbrella report is not proven fixed.

## What This Changes

The issue asks OpenClaw to prevent gateway/channel/session replay from injecting the same external inbound message or logical subagent completion more than once into a session.

## Best Possible Solution

Keep the existing Telegram, Slack, and auto-reply dedupe hardening, then design a stable logical completion identity for subagent announce/remap/recovery so retries stay idempotent without collapsing genuinely distinct child runs. Add focused regression coverage for orphan recovery plus duplicated external message ids.

## Reproduction Assessment

Yes for triage: the discussion gives multiple high-signal production reproductions, including repeated Telegram message_ids, Slack transcript entries for the same message, and a subagent orphan-resume timeline with the same childSessionKey but new run ids. I did not live-reproduce a channel gateway in this read-only sweep, but current-main source still matches the run-id-scoped announce concern.

## Solution Assessment

Unclear. Current main is the right direction for several narrower replay paths, but it is not the complete solution for the issue because the remaining announce identity change needs a maintainable logical-completion key rather than a simple dedupe-key swap.
