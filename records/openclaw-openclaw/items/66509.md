## Summary

Keep open: current main and the latest release still source-reproduce the QQBot silence path because text-only tool deliveries are buffered and a later block reply clears the only fallback timer before those texts are sent.

## What This Changes

The issue reports that QQBot gateway tool-kind progress text is buffered instead of delivered when partial streaming is configured.

## Best Possible Solution

Add a QQBot-local progress delivery path for text-only tool progress when streaming is active, with focused regression coverage, while preserving existing media and official C2C stream behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: deliver a text-only `{ kind: "tool" }` payload followed by `{ kind: "block" }` through QQBot outbound dispatch; the tool text is buffered, then the block reply clears the fallback timer before it can send.

## Solution Assessment

Yes. The narrow maintainable fix is in the QQBot outbound/progress boundary, not a new config option or global reclassification of all tool results as block replies.
