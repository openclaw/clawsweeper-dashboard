## Summary

Keep open: current main still buffers text-only QQBot tool deliveries and clears the only fallback timer when a later block reply arrives, so the reported silence before the final answer is still source-reproducible.

## What This Changes

QQBot gateway should deliver pre-tool/tool-kind progress text under partial streaming instead of buffering and discarding it when a final block reply arrives.

## Best Possible Solution

Add a QQBot-local progress delivery path for text-only tool progress when partial streaming is active, while preserving final block delivery, media fallback, tool-only fallback semantics, and official C2C stream behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: deliver a text-only `{ kind: "tool" }` payload followed by `{ kind: "block" }` through QQBot outbound dispatch; the first text is buffered and the block reply clears the fallback timer.

## Solution Assessment

Yes. The narrow maintainable fix is in the QQBot outbound/progress boundary, not a global reclassification of all tool results as block replies or a new config option.

On current main, QQBot `dispatchOutbound()` buffers text-only `info.kind === "tool"` payloads in `toolTexts`, returns without sending, then clears `toolOnlyTimeoutId` when a later block reply arrives; because `hasBlockResponse` is true, final cleanup never flushes the buffered text. Add a QQBot-local progress delivery path for safe text-only tool progress when partial streaming/block delivery is active. Add focused regression coverage in `extensions/qqbot/src/engine/gateway/outbound-dispatch.test.ts` for a text-only tool delivery followed by a block delivery, and include an off/official-stream guard if the implementation branches on those modes. Add a changelog entry for a user-visible fix; update QQBot/streaming docs only if behavior or config documentation changes.
