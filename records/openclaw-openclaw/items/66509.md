## Summary

Keep this issue open. Current main still buffers QQBot `kind: "tool"` text, clears the fallback when a block reply arrives, and has no QQBot preview/tool-progress path or regression test that would make `streaming.mode: "partial"` deliver the intermediate text promptly.

## What This Changes

The issue requests that gateway tool/progress text emitted before a tool call be delivered immediately when partial streaming is enabled instead of being buffered and discarded.

## Best Possible Solution

Implement the behavior in the QQBot plugin boundary: surface useful tool/progress text promptly through QQBot preview/streaming or a controlled immediate-delivery path when partial streaming is active, keep final reply delivery single-path, and preserve existing media fallback behavior.

## Reproduction Assessment

Yes. A focused source-level reproduction is available: call the QQBot outbound `deliver` callback with `{ kind: "tool", text: "Working..." }` followed by `{ kind: "block", text: "Done" }`; current code buffers the first text, clears the timeout on the block reply, and only sends the final block.

## Solution Assessment

No, current main is not the best solution because `streaming.mode: "partial"` only affects block/partial delivery and leaves tool/progress text on a fallback buffer. The narrow maintainable fix is QQBot-specific preview/tool-progress handling, not a global reclassification of all tool results as blocks.
