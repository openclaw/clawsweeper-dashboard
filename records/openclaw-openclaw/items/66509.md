## What This Changes

The issue asks for QQBot gateway tool-kind progress text to deliver immediately, or to honor `streaming.mode: "partial"`, instead of buffering it until a fallback timeout.

## Best Possible Solution

Add a QQBot-local progress delivery path for text-only tool deliveries when visible progress is active, with regression coverage, while preserving media fallback and official C2C streaming behavior.

## Reproduction Assessment

Yes, source-level. A text-only `{ kind: "tool" }` delivery followed by a `{ kind: "block" }` delivery buffers the text and then clears the fallback before it can send.

## Solution Assessment

Yes. The narrow maintainable fix is in QQBot outbound progress delivery; adding a new global config option or reclassifying all tool results would be broader and riskier.
