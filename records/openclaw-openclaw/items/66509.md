## Summary

Keep this open: current main still buffers QQBot `kind: "tool"` text, only sends that text through a delayed fallback when no block reply arrives, and has no QQBot tool-progress preview path tied to `streaming.mode: "partial"`. The prior ClawSweeper comment remains accurate against the newer main snapshot.

## What This Changes

The issue requests immediate delivery of user-facing intermediate QQBot/Telegram gateway text that currently arrives as tool-kind delivery while partial streaming is enabled.

## Best Possible Solution

Add a QQBot-local progress/preview delivery path for text-only tool progress when preview/partial streaming is active, while preserving the existing final block path and media fallback semantics. The fix should include a focused dispatcher regression test, docs/changelog updates if user-visible configuration or behavior changes, and avoid reclassifying all tool results globally as block replies.

## Reproduction Assessment

Yes. A source-level reproduction is clear: invoke the QQBot dispatcher delivery callback with a text-only `{ kind: "tool" }` payload followed by a `{ kind: "block" }` payload; the first text is buffered, the block clears the fallback timeout, and only the block is delivered.

## Solution Assessment

Yes for a narrow QQBot-boundary fix; the current implementation is not the solution because partial streaming only affects block/partial replies. A safer repair is to surface tool-progress text through QQBot preview/streaming or a controlled immediate path rather than changing shared tool-result classification.

## Repair Work Prompt

Current `extensions/qqbot/src/engine/gateway/outbound-dispatch.ts` buffers text-only `kind: "tool"` deliveries in `toolTexts`, schedules `TOOL_ONLY_TIMEOUT`, and drops that text when a later block reply arrives first. Implement the fix inside the QQBot plugin boundary: when QQBot preview/partial streaming is active, surface useful tool/progress text promptly through the QQBot streaming/preview path or a controlled immediate-delivery path. Preserve final block delivery as the single final-answer path, preserve existing media fallback behavior, and do not globally reclassify all shared tool results as block replies. Add or extend `extensions/qqbot/src/engine/gateway/outbound-dispatch.test.ts` with a regression that sends `{ kind: "tool", text: "Working..." }` followed by `{ kind: "block", text: "Done" }` and asserts the progress text is delivered promptly instead of silently discarded. Update `docs/concepts/streaming.md`, `docs/channels/qqbot.md`, `extensions/qqbot/src/config-schema.ts`, and `CHANGELOG.md` only if the fix changes documented user-facing behavior or config. Follow repo policy: use OpenClaw/plugin wording, no core owner IDs, and keep the fix QQBot-local unless a generic SDK seam is strictly necessary.
