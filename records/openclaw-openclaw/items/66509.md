## Summary

Keep open: current main and v2026.5.7 still source-reproduce the QQBot silence path, because text-only tool deliveries are buffered and a later block reply clears the fallback timer before those texts can be sent.

## What This Changes

Request immediate delivery of intermediate text classified as tool progress in the QQBot gateway when partial streaming is enabled.

## Best Possible Solution

Implement a QQBot-local progress delivery path for text-only tool progress when partial/block streaming is active, while preserving media fallback and official C2C streaming behavior, with regression coverage.

## Reproduction Assessment

Yes, source-level. A text-only `kind: "tool"` delivery followed by a `kind: "block"` delivery through QQBot outbound dispatch buffers the text, then the block reply clears the fallback timer before the text can send.

## Solution Assessment

Yes. The narrow maintainable fix is in the QQBot outbound/progress boundary; adding a new config option or globally reclassifying all tool results would be broader and riskier.
