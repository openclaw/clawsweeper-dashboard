## Summary

Keep this issue open: current main still buffers QQBot `kind: "tool"` text and returns before user delivery, so `streaming.mode: "partial"` does not surface pre-tool progress text when a later block reply arrives.

## What This Changes

The issue requests immediate delivery of QQBot gateway text emitted before tool calls when partial streaming is enabled instead of buffering it as tool-only fallback content.

## Best Possible Solution

Add a QQBot-local progress/preview delivery path for text-only tool progress when partial streaming is active, while preserving final block delivery and the existing media/tool-only fallback semantics.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: deliver a text-only `{ kind: "tool" }` payload followed by `{ kind: "block" }` through the QQBot outbound dispatcher and observe that the first text is buffered while the block reply clears the fallback timer and is sent.

## Solution Assessment

Yes for a narrow QQBot-boundary fix. The maintainable path is to surface tool-progress text through QQBot preview/partial delivery or a controlled immediate path, not to globally reclassify all tool results as block replies.
