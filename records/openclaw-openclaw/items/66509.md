## Summary

Keep open: current main still buffers QQBot `kind: "tool"` text, clears the fallback timer when a later block reply arrives, and has no QQBot tool-progress delivery path tied to `streaming.mode: "partial"`.

## What This Changes

The issue requests immediate delivery of QQBot text-only tool progress under partial streaming instead of buffering and discarding it before the final block reply.

## Best Possible Solution

Add a QQBot-local progress delivery path for text-only tool progress when partial streaming is active, while preserving final block delivery and existing media/tool-only fallback semantics.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: deliver a text-only `{ kind: "tool" }` payload followed by `{ kind: "block" }` through the QQBot outbound dispatcher and observe that the first text is buffered while the block reply clears the fallback timer.

## Solution Assessment

Yes. The narrow maintainable fix is in the QQBot outbound/progress boundary, not a global reclassification of all tool results as block replies or a new config surface.
