## Summary

Keep open. Current main still drops wake mapping routing before dispatch, and PR #70268 is an open, unmerged closing implementation, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks wake-mode hook mappings to honor configured `agentId` and `sessionKey`, or reject/warn on those fields instead of silently routing to the main session.

## Best Possible Solution

Land PR #70268 or an equivalent narrow fix that threads wake mapping `agentId` and `sessionKey` through the existing hook policy/session resolution path while preserving the current main-session fallback when routing is omitted.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence static reproduction: configure a wake mapping with `agentId` or `sessionKey`, then the mapping builder and request handler drop those fields before `dispatchWakeHook` enqueues into the main session.

## Solution Assessment

Yes. Reusing the existing hook target-agent and session-key policy path is the narrowest maintainable fix; the active closing PR should be reviewed or replaced rather than racing it with a separate repair.
