## Label Justifications

- `P2`: This is a valid multi-agent gateway routing bug with limited blast radius and a clear focused fix path.
- `impact:message-loss`: Wake events can be silently routed to the wrong agent/session, so the intended target never receives the event.
- `impact:session-state`: The event can be appended to the main heartbeat session instead of the configured agent/session, causing session state drift.

## Summary

Keep open. Current main and the latest shipped tag still accept `agentId` and `sessionKey` on hook mappings but drop those fields on the mapped `wake` path, and the linked fixing PR is still open rather than merged.

## What This Changes

The issue asks for `action: "wake"` hook mappings to honor configured `agentId` and `sessionKey` routing instead of silently enqueueing every wake into the default main session.

## Best Possible Solution

Land a rebased narrow fix, likely https://github.com/openclaw/openclaw/pull/70268 or an equivalent replacement, that threads mapped wake targets through the existing hook agent/session policy path with focused regressions.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: configure a mapped wake hook with `agentId` or `sessionKey`, then the mapping builder and mapped wake branch drop those fields before dispatching to the main session; I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes for the reported mapped wake bug. Threading `agentId` and `sessionKey` through the existing hook session-policy routing is the narrow maintainable solution; this issue should stay open until the linked PR or an equivalent fix lands.
