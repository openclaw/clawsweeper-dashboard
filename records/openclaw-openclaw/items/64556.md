## Label Justifications

- `P2`: This is a valid gateway hook routing bug with limited blast radius to configured webhook wake flows in multi-agent or explicit-session setups.
- `impact:session-state`: Wake events are mis-associated with the default main heartbeat session instead of the configured agent or session state.
- `impact:message-loss`: The intended agent/session never receives the wake event even though the gateway returns success.

## Summary

Keep open: current main and v2026.6.8 still drop wake-mapping `agentId` and `sessionKey` before dispatch, while the open closing PR at https://github.com/openclaw/openclaw/pull/70268 is unmerged and conflicting. The merged cron wake work is useful adjacent evidence, but it fixes the cron/tool path rather than this HTTP hook mapping path.

## What This Changes

Wake-mode webhook mappings should honor configured `agentId` and `sessionKey` routing instead of silently enqueueing every wake into the default main session.

## Best Possible Solution

Review, rebase, prove, and land or replace https://github.com/openclaw/openclaw/pull/70268 so wake mappings route through the same policy-checked agent/session path as agent hooks.

## Reproduction Assessment

Yes, source-level: current main drops mapped wake `agentId` and `sessionKey` before dispatch and always enqueues through `resolveMainSessionKeyFromConfig()`. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Yes for the current routing decision: keeping this issue open and landing the linked HTTP hook fix or a narrow replacement is the maintainable path. The cron wake merge is adjacent evidence, not the webhook mapping fix.
