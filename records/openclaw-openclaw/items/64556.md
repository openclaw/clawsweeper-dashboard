## Summary

Keep open: current `main` and the latest release still route mapped wake hooks through the main-session-only wake dispatcher, and the linked fixing PR is still open and conflicting rather than merged.

## What This Changes

The issue reports that `action: "wake"` hook mappings accept `agentId` and `sessionKey` but ignore them at dispatch time.

## Best Possible Solution

Rebase, prove, and land the linked targeted-wake PR, or replace it with a narrow equivalent that threads mapped wake routing through the existing hook agent/session policy helpers.

## Reproduction Assessment

Yes from source inspection: a mapped wake with `agentId` or `sessionKey` reaches only the `{ text, mode }` dispatch path, and that dispatcher enqueues to the main session. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Yes, the best fix is to thread mapped wake routing through the existing hook session/agent policy path. Removing the fields or adding a second routing mechanism would be less maintainable than making the documented mapping contract work.
