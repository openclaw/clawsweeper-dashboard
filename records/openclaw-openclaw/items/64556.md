## Summary

Keep open. Current main still accepts `agentId` and `sessionKey` on hook mappings but drops them for `action: "wake"`, and the dispatcher still enqueues wake events into the configured main session. PR #70268 is an open, unmerged implementation candidate with closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that wake-mode webhook mappings silently ignore configured `agentId` and `sessionKey` routing and asks OpenClaw to either honor those fields or reject/warn on them.

## Best Possible Solution

Land PR #70268 or an equivalent narrow fix that threads `agentId` and `sessionKey` through wake mapping resolution, applies the existing hook agent/session policy guardrails, preserves the current main-session fallback when routing is omitted, and adds regression coverage for mapped wake routing. The separate reload-mode bug raised in the comments should be tracked separately because it is a different gateway reload path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure a wake mapping with `agentId` or `sessionKey`; the mapping normalizer preserves those fields, but the wake action builder and request handler drop them before `dispatchWakeHook` enqueues into `resolveMainSessionKeyFromConfig()`.

## Solution Assessment

Yes. Honoring wake mapping routing through the same hook policy path used by agent mappings is the narrowest maintainable fix; the direct `/hooks/wake` session-key expansion in PR #70268 should receive normal PR review, but it does not make this issue obsolete before merge.
