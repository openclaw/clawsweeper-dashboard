## Label Justifications

- `P2`: This is a real gateway hook routing bug with limited blast radius to configured webhook wake flows in multi-agent/session setups.
- `impact:message-loss`: Wake events can be accepted with 200 OK but delivered to the wrong session, making the intended agent/session miss the message.
- `impact:session-state`: The bug mis-associates webhook wake events with the main heartbeat session instead of the configured agent/session state.

## Summary

Current main and the latest release still drop hook wake routing fields, so the issue is valid and should stay open while the linked implementation PR is reviewed, rebased, proven, or replaced.

## What This Changes

Webhook wake mappings should honor configured `agentId` and `sessionKey` instead of always routing wake events to the default main session.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/70268 with routing, prefix-policy, global-scope, and regression coverage for direct and mapped wake hooks, then close this issue as implemented.

## Reproduction Assessment

Yes, from source inspection: current main accepts hook routing fields on mappings, but the wake mapping and dispatcher path only forwards `text` and `mode` and resolves the main session. I did not run a live gateway curl repro in this read-only review.

## Solution Assessment

Yes, the best fix is to wire the existing accepted `agentId`/`sessionKey` contract through the hook wake path with the same policy checks used by agent hooks. The already-linked PR is the right work item, but it needs conflict resolution and proof before merge.
