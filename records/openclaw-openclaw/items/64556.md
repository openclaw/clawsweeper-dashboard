- `P2`: This is a valid multi-agent gateway bug with limited blast radius and an open focused fix path.
- `impact:message-loss`: Wake events can be silently routed to the wrong agent/session, so the intended target never receives the event.
- `impact:session-state`: The event is appended to the main heartbeat session instead of the configured agent/session, causing session state drift.

## Summary

Keep open. Current main still accepts hook mapping `agentId` and `sessionKey` but drops them on the mapped `wake` path, and the linked fixing PR remains open and conflict-marked rather than merged.

## What This Changes

The issue requests that `hooks.mappings[*]` with `action: "wake"` honor configured `agentId` and `sessionKey` instead of silently routing every wake event to the default main session.

## Best Possible Solution

Land a rebased narrow fix, likely https://github.com/openclaw/openclaw/pull/70268 or an equivalent replacement, that threads mapped wake targets through the existing hook session-policy path with focused regressions.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: configure a mapped wake hook with `agentId` or `sessionKey`, then the mapping builder and mapped wake branch drop those fields before dispatching to the main session.

## Solution Assessment

Yes for the reported mapped wake bug. Threading `agentId` and `sessionKey` through the existing hook session-policy routing is the narrow maintainable solution; the issue should stay open until that lands.
