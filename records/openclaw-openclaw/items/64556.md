## Summary

Keep open: current main and latest v2026.6.10 still ignore `agentId`/`sessionKey` on mapped wake hooks, while the linked closing PR remains open, conflicting, and unmerged.

## What This Changes

The issue asks for `action: "wake"` hook mappings to honor configured `agentId` and `sessionKey` instead of silently routing all wake events to the default main session.

## Best Possible Solution

Rebase, repair, and prove https://github.com/openclaw/openclaw/pull/70268, or land a narrow replacement that routes HTTP wake hooks through policy-checked agent/session resolution and documents the operator contract.

## Reproduction Assessment

## Solution Assessment

Yes for triage: the best path is to keep this issue open until the linked HTTP hook fix or a narrow replacement lands. The adjacent cron wake merge is not sufficient for the reported webhook mapping behavior.
