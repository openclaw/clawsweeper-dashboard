## Label Justifications

- `P2`: This is a valid gateway hook routing bug with limited blast radius to configured webhook wake flows in multi-agent/session setups.
- `impact:session-state`: Wake events are mis-associated with the default main heartbeat session instead of the configured agent/session state.
- `impact:message-loss`: The intended agent/session never receives the wake event even though the gateway returns success.

## Summary

Current main and v2026.6.8 still drop `agentId` and `sessionKey` on wake-mode webhook mappings, so this issue is still necessary. The open closing PR at https://github.com/openclaw/openclaw/pull/70268 owns the intended fix path, while the merged cron wake PR only covers the cron/tool path, not the HTTP hook mapping path reported here.

## What This Changes

The issue asks for wake-mode webhook mappings to honor configured `agentId` and `sessionKey`, or reject/warn when those fields are unsupported.

## Best Possible Solution

Review, rebase, prove, and land or replace https://github.com/openclaw/openclaw/pull/70268 so wake mappings route through the same policy-checked agent/session path as agent hooks.

## Reproduction Assessment

Yes, with source-level confidence: current main drops wake mapping `agentId`/`sessionKey` before dispatch and always enqueues through `resolveMainSessionKeyFromConfig()`. I did not run a live gateway repro because this review is read-only, but the dispatch path is direct and covered by current source/tests.

## Solution Assessment

Yes, keeping this open and letting the linked fix PR or a narrow replacement wire wake routing through the existing hook policy/session helpers is the maintainable path. The cron wake merge is useful adjacent evidence but is not the same HTTP hook mapping fix.
