## Summary

Keep open. Current main still drops wake-mapping routing before dispatch and always enqueues wake events into the configured main session; #70268 is an open, unmerged PR that explicitly targets this issue, so the issue should remain open until that or an equivalent fix lands.

## What This Changes

The issue reports that `hooks.mappings[]` accepts `agentId` and `sessionKey` for `action: "wake"`, but wake dispatch ignores those fields and routes all mapped wake events to the default main session.

## Best Possible Solution

Review and land #70268 or an equivalent narrow fix that threads `agentId` and `sessionKey` through mapped wake action resolution and dispatch, reuses the existing hook session-key and agent allowlist guardrails, preserves the main-session fallback when routing is omitted, and adds regression coverage for mapped wake routing.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure a wake mapping with `agentId` or `sessionKey`, and the mapping builder plus request handler drop those fields before `dispatchWakeHook` enqueues into `resolveMainSessionKeyFromConfig()`. I did not run live gateway tests because this review is read-only.

## Solution Assessment

Yes, honoring mapped wake routing through the existing hook policy path is the narrowest maintainable fix. The direct `/hooks/wake` session-key expansion in #70268 should get normal PR review, but it does not make this issue obsolete.
