## Label Justifications

- `P2`: The issue is a normal-priority WhatsApp channel lifecycle bug with a clear restart workaround and limited blast radius to dynamic account enablement changes.
- `impact:session-state`: A running WhatsApp account remains connected after config state says the account is disabled.
- `impact:other`: The operator-facing soft-disable control is not honored until a full gateway restart, which is meaningful channel lifecycle impact outside the more specific labels.

## Summary

This issue should stay open because current main still classifies `channels.whatsapp.*` changes as reload no-ops, while the running channel manager only honors disabled accounts during startup or explicit stop paths. Source inspection supports the reported stale-running-session behavior, and I did not find a merged fix or a narrower canonical issue that owns the disable-on-reload case.

## What This Changes

The issue requests dynamic reload behavior that stops a running WhatsApp account when `channels.whatsapp.accounts.<id>.enabled` flips from true to false without deleting credentials or requiring a gateway restart.

## Best Possible Solution

Teach the reload path to treat account enabled transitions as channel account lifecycle changes: stop the affected running account when it becomes disabled, start it when it becomes enabled and configured, and leave credentials intact.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a WhatsApp account enabled flip is classified as a no-op reload, while disabled-account handling only runs on channel start or explicit stop. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to wire account enabled changes into the existing channel account lifecycle rather than require a new config option or a full gateway restart. The safer implementation should target account enabled transitions instead of broadly restarting on all WhatsApp config edits.
