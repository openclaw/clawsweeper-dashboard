## Summary

Keep open. Current `main` still has the reported mismatch: sandbox diagnostics are policy-only, the Claude CLI runner reports and prepares an unsandboxed run, and the browser tool still emits the generic no-bridge error; the broader sandbox and isolated-browser threads are related context but do not precisely own this Claude CLI runtime mismatch.

## What This Changes

The issue asks OpenClaw to make Claude CLI OAuth runs either honor sandboxed execution or clearly report that the selected CLI runner cannot provide a sandbox/browser bridge.

## Best Possible Solution

Make sandbox diagnostics distinguish requested sandbox policy from actual runner containment, and make the browser sandbox error name the missing bridge/current unsandboxed runtime condition while maintainers decide whether real Claude CLI sandbox execution is supported.

## Reproduction Assessment

Yes at source level: current `main` has policy-only sandbox reporting, a CLI runner prompt report that is explicitly unsandboxed, and the same no-bridge browser error. I did not run the live Claude CLI OAuth reproduction from the report.

Unclear because there is no patch and the main policy question needs owner input. The narrow maintainable step is clear diagnostics; actual sandboxed Claude CLI execution would need an explicit runtime/security decision.
