## Summary

Keep open. Current main and v2026.6.8 still show the reported mismatch: sandbox explain is config/session-derived, while CLI runner metadata is explicitly unsandboxed and the browser tool still fails `target="sandbox"` on a missing bridge with config-focused wording.

## What This Changes

The issue reports that Claude CLI OAuth runs can be reported as sandboxed by `openclaw sandbox explain` while the actual CLI runner disables sandboxing and cannot provide a sandbox browser bridge.

## Best Possible Solution

Make sandbox explain, run metadata, and browser `target="sandbox"` failures reflect actual runner bridge availability unless maintainers explicitly decide to support sandboxed Claude CLI execution.

## Reproduction Assessment

Yes, source-level. Current main computes sandbox explain from config/session policy, hard-codes CLI run metadata as unsandboxed, and keeps the browser `target="sandbox"` failure path when no bridge URL exists.

## Solution Assessment

Unclear for full sandbox support. A diagnostics-first fix is the narrow maintainable path if CLI backends remain unsandboxed; actually sandboxing Claude CLI needs explicit security/product direction.
