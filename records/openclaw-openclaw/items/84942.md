## Summary

Keep open. Current main and the latest release still have the reported mismatch: sandbox explain is derived from sandbox policy, while the Claude CLI path reports unsandboxed execution and browser target=sandbox still fails on missing bridge context.

## What This Changes

The issue asks OpenClaw to make sandbox diagnostics and browser errors match the actual Claude CLI runner behavior when sandbox policy says a session should be sandboxed.

## Best Possible Solution

Make sandbox explain, CLI run metadata, and browser target=sandbox errors derive from actual runner and bridge availability; if Claude CLI cannot run inside an OpenClaw sandbox, report that explicitly under a maintainer-approved security contract.

## Reproduction Assessment

Yes. Source inspection on current main shows sandbox explain is policy-derived, CLI run reporting is hard-coded unsandboxed, and target=sandbox still fails when no bridge URL is present; no live Claude CLI OAuth run was executed in this read-only pass.

## Solution Assessment

Unclear for full sandbox support. A diagnostics-first fix is the narrow maintainable path if CLI backends remain unsandboxed, while actually sandboxing Claude CLI needs explicit security/product direction.
