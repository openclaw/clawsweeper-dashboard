## Summary

Keep open. Current main and the latest stable tag still show the reported mismatch: sandbox explain reports sandboxed from config/session policy, while Claude CLI preparation reports the actual CLI run as unsandboxed and the sandbox-browser error still blames browser config rather than the missing sandbox bridge.

## What This Changes

The issue reports that Claude CLI-backed sessions can be reported as sandboxed by `openclaw sandbox explain` while the actual CLI runner disables sandboxing and `target="sandbox"` browser access fails.

## Best Possible Solution

Make sandbox diagnostics and `target="sandbox"` browser failures reflect the actual prepared runner unless maintainers explicitly decide to add sandboxed Claude CLI execution.

## Reproduction Assessment

Yes, at source level: current main computes `sandbox explain` from sandbox config/session keys while CLI preparation hard-codes `sandboxed: false`. A live Claude CLI OAuth/Docker turn was not run in this read-only review.

## Solution Assessment

Unclear for full sandbox support. A diagnostics-first fix is the narrow maintainable path if CLI backends remain unsandboxed; actually sandboxing Claude CLI needs explicit security/product direction.
