## Summary

Keep open. Current main still has a source-level mismatch: sandbox explain reports the session sandboxed from config alone, while the CLI runner reports its actual execution as unsandboxed and the sandbox browser error still points at browser config instead of the missing runner bridge.

## What This Changes

The issue asks OpenClaw to align sandbox explain and browser diagnostics with Claude CLI's unsandboxed runner behavior, or to make that runner actually provide a sandbox bridge.

## Best Possible Solution

Make sandbox status and target="sandbox" browser failures reflect the actual prepared runner: CLI backend turns should clearly report that no OpenClaw sandbox bridge exists unless a supported sandboxed CLI runtime is intentionally added.

## Reproduction Assessment

Yes, at source level: sandbox explain computes sandboxed from config while CLI preparation hard-codes sandbox off for CLI backend runs. A live Claude CLI OAuth/Docker run was not performed in this read-only review.

## Solution Assessment

Unclear for full sandbox support. A diagnostics-first fix is the narrow maintainable path if CLI backends remain unsandboxed; actually sandboxing Claude CLI would need an explicit runner/security decision.
