## Summary

Keep open. Current main and v2026.6.8 still show the reported mismatch: sandbox explain derives `runtime: sandboxed` from config/session policy, while CLI runner preparation records CLI-backed runs as unsandboxed and browser `target="sandbox"` still fails on a missing bridge with config-focused wording.

## What This Changes

The issue reports that Claude CLI OAuth sessions can be reported as sandboxed while the actual CLI runner executes unsandboxed and cannot provide a sandbox browser bridge.

## Best Possible Solution

Make sandbox explain, run metadata, and browser `target="sandbox"` failures reflect actual runner bridge availability unless maintainers explicitly support sandboxed Claude CLI execution.

## Reproduction Assessment

Yes. Source inspection on current main shows `sandbox explain` can report sandboxed from config/session policy while CLI preparation hard-codes unsandboxed run metadata and the browser tool lacks a sandbox bridge.

## Solution Assessment

Unclear for full sandbox support. A diagnostics-first fix is the narrow maintainable path if CLI backends remain unsandboxed; actually sandboxing Claude CLI needs explicit security/product direction.
