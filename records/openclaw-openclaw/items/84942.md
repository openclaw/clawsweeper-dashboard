The issue reports that Claude CLI OAuth runs can be reported as sandboxed by sandbox policy while actual CLI execution is unsandboxed and the sandbox browser bridge is unavailable.

## Best Possible Solution

Make sandbox explain, CLI run metadata, and browser target=sandbox errors derive from actual runner and bridge availability under a maintainer-approved security contract.

Yes. Source inspection on current main shows the mismatch path without a live Claude OAuth run: policy-derived sandbox explain, hard-coded unsandboxed CLI metadata, and no sandbox browser bridge error handling.

## Solution Assessment

Unclear for full sandbox support. A diagnostics-first fix is the narrow maintainable path if CLI backends remain unsandboxed, while actually sandboxing Claude CLI needs maintainer security/product direction.
