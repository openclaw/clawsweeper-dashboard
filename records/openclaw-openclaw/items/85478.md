## Summary

Keep this PR open: the Slack UX direction is useful and the supplied after-fix proof covers the intended no-match path, but the new classifier can also hide unrelated real command failures that merely mention `rg`.

## What This Changes

Adds exec outcome classification for benign `rg` no-result exits, forwards that metadata through command-output events, and renders Slack progress as `No matches found`.

## Best Possible Solution

Keep the Slack-facing no-result label, but classify it only when the failed command is actually the `rg` search or the `xargs`-launched `rg`, with regression coverage for unrelated pipelines that mention `rg`.

## Reproduction Assessment

Yes for the review finding: source inspection shows exit code 1 plus an `rg` token and empty aggregate output is enough to return `benign_no_result`. I did not run tests in this read-only review, but the false-positive path is directly traceable in the proposed classifier.

## Solution Assessment

No, not yet: the overall shape is maintainable, but the classifier is too broad for a user-visible failure/status path. The safer fix is a narrower command-shape check plus explicit negative tests.
