## Summary

Keep open. Current main still blanket-drops final reply payloads when a same-target messaging-tool send is recorded, and open PR #59752 explicitly closes this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Preserve a legitimate final text reply when the message tool previously sent media-only content to the originating channel in the same turn.

## Best Possible Solution

Land #59752 or an equivalent narrow fix that replaces the blanket same-target final-payload drop with content-level text/media dedupe and regression coverage for same-target media-only send followed by distinct final text.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A focused call to `buildReplyPayloads` with a distinct final text payload, same-origin `messagingToolSentTargets`, sent media telemetry, and no duplicate sent text will set `suppressReplies` and return no final payloads on current main.

## Solution Assessment

Is this the best way to solve the issue? Yes at the issue level. Keeping this issue open while the linked closing PR is reviewed is the right path; content-level dedupe is the narrowest maintainable fix direction.
