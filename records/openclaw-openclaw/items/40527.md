## Summary

Keep open. Current main still has a source-reproducible connect-ordering bug: the gateway can start the automatic macOS remote bin probe before the Swift app-backed node has entered normal event dispatch, and the related open churn PR does not prove or implement the readiness fix.

## What This Changes

Fix remote macOS skill bin probes so app-backed nodes on the gateway host or over LAN do not time out during startup and reconnect.

## Best Possible Solution

Move the automatic remote bin refresh behind a confirmed node readiness point after hello-ok delivery, or buffer and dispatch pre-hello node events in the Swift channel, while preserving command policy and macOS system.run/system.which support.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is a macOS app-backed Darwin node plus at least one Darwin skill requiring a binary: current main can issue node.invoke.request for system.which before the Swift client starts normal event dispatch after hello-ok.

## Solution Assessment

Yes. A readiness or buffering fix is the narrow maintainable direction; loopback-only skips, broad deny-command workarounds, or a global disable flag would mask legitimate remote macOS nodes rather than fixing the race.

## Repair Work Prompt

Fix the macOS app-backed remote skill bin probe readiness bug tracked at https://github.com/openclaw/openclaw/issues/40527. Current main starts refreshRemoteNodeBins from the node-role connect path before hello-ok is sent, while the Swift GatewayChannel only starts normal event listening after sendConnect returns, so node.invoke.request for system.which can be sent before the app node is ready to dispatch events. Implement the narrow readiness fix: move the automatic remote bin refresh behind a confirmed node readiness point after hello-ok delivery, or buffer and dispatch pre-hello node events in the Swift channel. Preserve existing node command policy and macOS system.run/system.which support. Do not solve this primarily with a global disable flag, denyCommands workaround, or loopback-only skip. Avoid duplicating the separate cache/backoff work in https://github.com/openclaw/openclaw/pull/79438 if it lands first. Add focused regression coverage for the gateway ordering and/or Swift event buffering/readiness path. Include a CHANGELOG.md entry if the fix changes user-visible startup/reconnect behavior.
