## Summary

Keep open. Current main still has a source-reproducible connect ordering bug: the gateway can send the automatic remote bin probe before `hello-ok`, while the Swift macOS node starts normal event dispatch only after the connect response.

## What This Changes

The issue asks OpenClaw to stop timing out automatic remote skill binary probes for macOS app-backed nodes, especially when the app node is co-located with the gateway.

## Best Possible Solution

Move the initial remote bin refresh behind a confirmed node readiness point, or buffer and dispatch pre-`hello-ok` node events in the Swift channel, so app-backed macOS nodes can answer `system.which`/`system.run` without weakening command policy.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is a macOS app-backed Darwin node plus at least one Darwin skill requiring a binary: current main can send `node.invoke.request` before `hello-ok`, while the Swift client is still waiting for the connect response and not dispatching those events.

## Solution Assessment

Yes. A narrow handshake/readiness ordering fix is the best direction; loopback-only skips, broad deny-command workarounds, or a global disable flag would mask legitimate remote macOS nodes instead of fixing the race.

## Repair Work Prompt

Fix the remote skill binary probe timeout for macOS app-backed nodes tracked by https://github.com/openclaw/openclaw/issues/40527. The current gateway connect path starts `refreshRemoteNodeBins` before sending `hello-ok`, while `GatewayChannelActor` does not enter normal event dispatch until after the connect response, so `node.invoke.request` for `system.which` can be dropped/ignored until the gateway times out. Implement the narrow readiness/ordering fix in the gateway and/or Swift channel: either defer the initial remote bin refresh until after node handshake/readiness, or buffer and dispatch pre-`hello-ok` node invoke events safely. Preserve node command policy, pairing auth, exec approvals, and legitimate remote macOS node support. Do not add a global config flag or disable remote skills as the primary fix. Add regression coverage for the pre-hello invoke ordering and any adjusted probe scheduling. Add a user-facing `CHANGELOG.md` fix entry if shipped behavior changes.
