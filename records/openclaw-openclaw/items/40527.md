- `impact:session-state`: Failed probes can clear or stale remote node bin eligibility while the node is otherwise reported connected, creating node/skill state drift.
- `impact:other`: The issue also affects gateway startup/reconnect latency and warning quality, which are meaningful maintainer-visible impacts outside the specific owned labels.

## Summary

Keep open: current main no longer grants desktop host commands from platform defaults alone, but it still automatically probes any connected macOS node that advertises `system.which` or `system.run`, and the later discussion points to probe readiness/timing rather than a fixed issue.

## What This Changes

The issue reports remote macOS app node bin probing timing out during gateway startup or reconnect, especially when the app node is co-located with the gateway.

## Best Possible Solution

Preserve the macOS app-node `system.run` and `system.which` contract, then fix automatic remote-bin probe readiness/churn in `src/skills/runtime/remote.ts` with targeted tests and real macOS app-node proof; refresh or replace https://github.com/openclaw/openclaw/pull/79438 if that remains the chosen path.

## Reproduction Assessment

Yes from source for the automatic probe path, but not from a live macOS setup in this sweep. Current main still starts a connect-time remote-bin probe for macOS nodes that advertise `system.which`/`system.run`, and user comments provide real timeout reports.

## Solution Assessment

No, the closed allowlist-removal approach is not the best fix because it breaks the documented app-node exec surface. The best path is a narrower readiness, co-location, or churn-control fix that preserves valid declared node commands.
