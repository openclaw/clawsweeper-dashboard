## Summary

Keep open: current main still has a source-reproducible connect-order path for this timeout, and this issue is paired with an open closing PR that should be reviewed, revised, or rejected before the issue is closed.

## What This Changes

The issue reports automatic remote skill binary probe timeouts for macOS app-backed nodes during gateway startup/reconnect, including co-located gateway/app nodes and first-connect readiness timing.

## Best Possible Solution

Fix the node readiness contract so automatic remote-bin refresh runs only after app-backed nodes can receive normal events, or skip only a proven co-located non-exec case, while preserving declared macOS system.run and system.which support.

## Reproduction Assessment

Yes, as a source-level reproduction: current main can start the automatic remote-bin probe before the Swift app starts its normal receive loop. The April 2 live comment also confirms automatic probe timeouts while manual system.which succeeds.

## Solution Assessment

Yes: a readiness/order fix is the narrow maintainable solution. Loopback-only skips, deny-command workarounds, or global command removal would hide or break valid remote macOS node execution instead of fixing the race.
