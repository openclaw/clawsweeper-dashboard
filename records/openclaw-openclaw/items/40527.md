## Summary

Keep open: current main still has a source-reproducible connect-order path where the gateway starts automatic remote-bin probing before the Swift app begins its normal event receive loop, and this issue has an open closing PR that should be reviewed, revised, or rejected before the issue closes.

## What This Changes

The issue reports timeout spam and startup delay from automatic remote skill binary probing against macOS app-backed node connections.

## Best Possible Solution

Fix the node readiness contract so automatic remote-bin refresh runs only after app-backed nodes can receive normal events, or skip only a proven co-located non-exec case, while preserving declared macOS system.run and system.which support.

## Reproduction Assessment

Yes. Current main can start the automatic remote-bin probe before the Swift app starts its normal receive loop, and the April 2 live comment confirms automatic probe timeouts while manual system.which succeeds.

## Solution Assessment

Yes for the issue direction: a readiness/order fix is the narrow maintainable solution. Loopback-only skips, deny-command workarounds, or global command removal would hide or break valid remote macOS node execution instead of fixing the race.
