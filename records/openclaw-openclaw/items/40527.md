## Summary

Keep open: current main and the latest release still start remote-bin refresh immediately on macOS app-backed node connect when the node advertises `system.which` or `system.run`, and the related attempted command-removal fix was closed as the wrong direction.

## What This Changes

The issue reports remote skill binary probe timeouts for macOS app-backed nodes, especially when the app node is co-located with the gateway.

## Best Possible Solution

Preserve the macOS app-node exec command contract, then fix the automatic remote-bin probe readiness path with live macOS app/gateway proof and an explicit maintainer decision on any defer/backoff or co-located-skip policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live; source inspection shows the current connect-time probe path and the issue discussion supplies real logs, but a current macOS app plus gateway run is still needed for high-confidence reproduction.

## Solution Assessment

Is this the best way to solve the issue? No final fix is on current main; the rejected blanket command-removal approach was not the best solution, and the better path is a narrow probe readiness/timing fix that preserves app-node exec.
