## Summary

Keep open: current main and the latest release still start automatic remote bin probing for macOS nodes that advertise `system.which` or `system.run`, while the overlapping open PR is conflicting/stale and the earlier command-removal PR was closed as the wrong direction.

## What This Changes

The issue asks OpenClaw to stop remote skill binary probes from timing out for macOS app-backed nodes, especially when co-located with the gateway, without blocking legitimate remote node execution.

## Best Possible Solution

Preserve the macOS app node `system.run`/`system.which` contract and land a narrow readiness, deferral, backoff, or co-location-aware probe fix with live macOS app-node proof.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows the same automatic probe path on current main and v2026.6.8, and the issue comments include live macOS setups where the automatic probe timed out while manual `system.which` could succeed.

## Solution Assessment

Unclear as a final implementation path. The safer direction is not a broad config flag or disabling app-node exec commands, but a readiness/order/backoff or co-location-specific probe fix that preserves declared macOS app node execution.
