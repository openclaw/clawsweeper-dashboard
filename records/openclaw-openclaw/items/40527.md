## Summary

Keep open: current main and v2026.6.8 still run automatic remote-bin refresh for macOS nodes that advertise `system.which` or `system.run`, and the related PRs have not landed a safe fix for the reported probe timing/co-location problem.

## What This Changes

The issue asks OpenClaw to stop noisy, slow remote skill binary probes when macOS app-backed nodes are co-located with or newly connected to the gateway.

## Best Possible Solution

Preserve the macOS app-node exec contract and fix the automatic remote-bin refresh/readiness behavior in the current runtime, with redacted live macOS app-node proof for co-located and remote reconnect cases.

## Reproduction Assessment

No high-confidence live current-main reproduction was run here. Source inspection shows the current connect path still starts automatic probes for macOS nodes advertising `system.which` or `system.run`, and reporter comments provide concrete live timeout logs.

## Solution Assessment

No, disabling app-node exec commands is not the best solution because current app code and docs treat them as supported. The better path is a narrow readiness, deferral, backoff, or co-location-specific probe fix that keeps valid remote node execution working.
