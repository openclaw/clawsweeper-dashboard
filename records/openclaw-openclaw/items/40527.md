## Summary

Keep open: current main still starts the automatic macOS remote-bin probe during node-role connect before the Swift client starts its normal receive loop, and the latest release has the same ordering. The user follow-up confirms the automatic probe can time out while manual `system.which` succeeds, and the related open PR only targets probe churn/backoff rather than the first-connect readiness bug.

## What This Changes

The issue requests fixing remote macOS node bin-probe timeouts during gateway startup or reconnect, especially when the macOS app node is co-located with the gateway or not yet ready for automatic probes.

## Best Possible Solution

Fix the node readiness contract so automatic remote-bin refresh runs only after an app-backed node can receive normal events, or make the Swift channel buffer and dispatch handshake-time events safely, while keeping valid remote macOS node probing supported.

## Reproduction Assessment

Yes, as a source-level reproduction: current main can emit `node.invoke.request` for the remote bin probe before the Swift app starts its normal receive loop. The April 2 user comment also confirms automatic probe timeouts while manual `system.which` succeeds.

## Solution Assessment

Yes: a readiness/order fix is the narrow maintainable solution. Loopback-only skips, deny-command workarounds, or a global disable flag would hide useful remote macOS nodes rather than fixing the race.
