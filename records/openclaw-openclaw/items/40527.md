## Summary

Keep this open: current main and v2026.6.8 mitigate the original platform-default command-policy problem, but they still run automatic connect-time remote bin probing for macOS nodes that advertise `system.which` or `system.run`. The related backoff PR is stale against the moved runtime, and a maintainer already rejected the blanket app-node command-removal approach.

## What This Changes

The issue reports remote skill binary probe timeouts for macOS app-backed nodes, especially when a node is co-located with the gateway host.

## Best Possible Solution

Preserve the macOS app-node command contract and land a narrow, proof-backed probe readiness/backoff fix in the current remote-skill runtime if the timeout still reproduces.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows the connect-time refresh path still probes macOS nodes that advertise `system.which` or `system.run`, and related live reports still show the same timeout family.

## Solution Assessment

No, not as blanket app-node command removal. The best direction is a narrower readiness/order/backoff fix that keeps declared macOS app-node exec commands working.
