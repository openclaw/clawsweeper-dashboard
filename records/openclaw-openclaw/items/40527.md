## Label Justifications

- `impact:session-state`: The failure arises when registered connection state and actual node command readiness temporarily disagree during startup or reconnect.
- `impact:other`: Failed binary discovery causes misleading warnings, startup delays, and suppression of otherwise eligible macOS-dependent skills.

## Summary

Current main and the latest release still start remote-bin discovery immediately after node registration, so the reported readiness race remains plausible and no linked fix has merged. The original claim that macOS app nodes cannot handle shell RPC is outdated, but the narrower automatic-probe timing problem remains actionable.

## What This Changes

Fix remote skill binary probe timeouts for macOS app-backed nodes during startup and reconnect, including co-located and early-connect cases.

## Best Possible Solution

Trigger the initial bin probe only after the registered node is demonstrably ready to service commands, then coalesce and back off transient retries while preserving declared macOS app shell capabilities and refreshing promptly when capabilities or required bins change.

## Reproduction Assessment

Yes at source level: current main starts the probe immediately after registration with a short readiness preflight, and multiple reports describe the matching automatic-probe failure while later manual commands can succeed. A fresh v2026.6.5 real-node reconnect run is still needed to classify it as fully reproduced.

## Solution Assessment

No, categorically disabling system.run and system.which for macOS app nodes is no longer the best approach because current app nodes legitimately implement them. A readiness-aware deferred probe with bounded retries is the narrower fix.
