## Summary

Keep this open: current main preserves the macOS app-node command contract and has mitigations, but it still starts automatic remote bin probing on node connect and related live reports/PRs show the readiness/probe-churn problem is not resolved enough to close.

## What This Changes

The issue asks OpenClaw to stop macOS app-backed nodes, especially co-located app-plus-gateway setups, from causing remote skill binary probe timeouts during startup or reconnect.

## Best Possible Solution

Keep this issue open as the remote-bin-probe report, preserve the macOS app-node system.run/system.which contract, and land a narrow proof-backed readiness, defer, co-location, or backoff fix coordinated with https://github.com/openclaw/openclaw/issues/83958 if live evidence shows the same root cause.

## Reproduction Assessment

No, not with high-confidence live proof in this review. The source path is clear and the issue/comments include concrete macOS setups and logs, but current-main behavior still needs a real macOS app plus gateway reproduction to separate probe timing from broader node readiness.

## Solution Assessment

No, the broad command-removal or new config-flag directions are not the best fix. The better path is to preserve the declared app-node exec contract and fix probe readiness, deferral, backoff, or only a proven co-located non-exec case.
