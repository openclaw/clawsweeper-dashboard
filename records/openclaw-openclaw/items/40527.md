## Summary

Keep open. Current main still has a source-reproducible connect-ordering bug: the gateway can send the automatic remote bin probe before the macOS app-backed node starts normal event dispatch, and the related open churn PR does not prove that readiness path fixed.

## What This Changes

The issue asks OpenClaw to stop timing out automatic remote skill binary probes for macOS app-backed nodes, especially when the app node is co-located with the gateway.

## Best Possible Solution

Move the automatic remote bin refresh behind a confirmed post-hello node readiness point, or buffer/replay pre-hello Swift events, while preserving declared command policy and legitimate macOS system.run/system.which nodes.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is a macOS app-backed Darwin node plus at least one Darwin skill requiring a binary: current main can issue node.invoke.request for system.which before the Swift client starts normal event dispatch after hello-ok.

## Solution Assessment

Yes. A readiness/order fix is the narrow maintainable direction; loopback skips, broad deny-command workarounds, or a global disable flag would mask legitimate remote macOS nodes instead of fixing the race.

## Repair Work Prompt

Fix the source-reproducible remote skill bin probe connect-readiness bug for https://github.com/openclaw/openclaw/issues/40527. Current main registers node sessions and starts refreshRemoteNodeBins before sending hello-ok, while Swift GatewayChannel.waitForConnectResponse ignores non-matching event frames before listen starts; a node.invoke.request for system.which can therefore be consumed before normal event dispatch and time out. Make the narrowest maintainable ordering/readiness fix: either move the initial refreshRemoteNodeBins and related node startup events until after hello-ok is sent, or buffer/replay pre-hello events in GatewayChannel. Do not add a global disable flag or loopback-only skip, and preserve declared command allowlist behavior plus macOS system.run/system.which support. Add focused regression coverage for a node connect where the automatic remote bin probe cannot be delivered before the connect response path is ready. Treat https://github.com/openclaw/openclaw/pull/79438 as related churn/backoff work only; do not duplicate unrelated cache/backoff changes. Add a CHANGELOG.md fix entry if the final behavior change is user-visible.
