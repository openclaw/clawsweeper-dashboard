## Summary

Keep this open. Current main and v2026.6.8 have useful partial mitigations, but they still start remote-bin refresh on macOS node connect when the node advertises `system.which` or `system.run`; related live reports and maintainer discussion show the remaining problem is a readiness/probe policy issue, not safely solved by disabling the macOS app-node exec surface.

## What This Changes

The issue reports startup/reconnect timeout warnings from automatic remote skill binary probing against macOS app-backed nodes, especially when the app and gateway are co-located.

## Best Possible Solution

Keep this as the narrow remote-bin-probe/readiness tracker: preserve the macOS app-node `system.run`/`system.which` contract, then land a refreshed, proof-backed probe timing/backoff or co-location-specific fix on `src/skills/runtime/remote.ts`.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows the connect-time refresh path still probes macOS nodes that advertise `system.which` or `system.run`, and related live comments report the same macOS app-node timeout family after later releases.

## Solution Assessment

No, not as a close or blanket allowlist-removal fix. The maintainable direction is a readiness/order, backoff, or co-location-specific probe change that preserves valid app-node execution commands.
