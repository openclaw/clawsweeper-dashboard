## Summary

Keep open. Current main still starts the automatic remote bin probe before `hello-ok`, while the Swift app channel starts normal event dispatch only after the connect response, so the reported app-node timing bug remains source-reproducible.

## What This Changes

The issue asks OpenClaw to avoid `skills-remote` probe timeouts for macOS app-backed nodes, especially when the macOS app is co-located with the gateway.

## Best Possible Solution

Move the initial remote bin probe behind a real post-handshake/readiness point, or make the Swift channel safely buffer and dispatch pre-`hello-ok` node events, while preserving node command policy and exec approval semantics.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is a macOS app-backed Darwin node plus at least one Darwin skill requiring a binary: current main can send `node.invoke.request` before `hello-ok`, while the Swift client is still waiting for the connect response.

## Solution Assessment

Yes for a narrow handshake/readiness ordering fix. A global remote-skills disable flag, loopback-only skip, or blanket `denyCommands` workaround would mask legitimate remote macOS nodes and is not the best end state.

## Repair Work Prompt

Fix the remote skill bin probe timing bug for macOS app-backed nodes tracked by https://github.com/openclaw/openclaw/issues/40527. Current main registers node connections and starts `refreshRemoteNodeBins` before sending `hello-ok`, while `GatewayChannelActor` waits for the connect response before starting normal event dispatch; this can drop or miss the automatic `node.invoke.request` probe and produce `remote bin probe timed out`. Make the initial remote bin probe wait until the node is actually ready to receive events, or make the Swift channel buffer/dispatch early event frames safely. Preserve node command declarations, gateway allow/deny command filtering, device-token pairing, `system.run` approval checks, and existing remote skill eligibility semantics. Do not add a global remote-skills disable flag, do not blanket-deny `system.run`/`system.which`, and do not special-case only loopback nodes unless it is part of a more general readiness fix. Add a single-line `CHANGELOG.md` Fixes entry because this is user-facing log/skill behavior. Include focused regression tests for the gateway ordering and/or Swift early-event handling.
