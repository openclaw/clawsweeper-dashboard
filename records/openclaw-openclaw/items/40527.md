## Summary

Keep open: current main and the latest release still start automatic remote skill bin probes for macOS nodes that advertise `system.which` or `system.run`, and newer related reports still show the same probe/invoke timeout family. The closed attempted fix at https://github.com/openclaw/openclaw/pull/83429 was rejected for removing the app-node exec contract, while https://github.com/openclaw/openclaw/pull/79438 is open but stale/conflicting and not a safe resolution yet.

## What This Changes

The issue asks OpenClaw to stop automatic remote skill binary probes from timing out and spamming logs when a macOS app node is co-located with the gateway, while preserving valid remote node exec.

## Best Possible Solution

Land a narrow, live-proven macOS app-node probe/readiness fix that preserves declared `system.run` and `system.which`, defers or backs off automatic bin probes when the node is not ready, and covers both co-located and real remote macOS app-node flows.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows the connect-time automatic probe path and the issue comments provide real macOS probe-timeout logs, but the co-located setup still needs fresh live proof.

## Solution Assessment

Unclear as a final implementation path. The best direction is not a blanket command removal or new config flag; it should preserve app-node exec support and fix probe readiness, deferral, or backoff with live macOS proof.
