## Summary

Keep open. Current main still has a source-reproducible ordering bug: the gateway can send the automatic remote bin probe before `hello-ok`, while the Swift app node starts normal event dispatch only after the connect response; a follow-up comment also confirms newer 2026.4.2 setups still hit this family of timeouts even when manual `system.which` succeeds.

## What This Changes

The issue asks OpenClaw to stop timing out automatic remote skill binary probes for macOS app-backed nodes without blocking legitimate `system.run`/`system.which` remote nodes.

## Best Possible Solution

Move the initial remote bin refresh behind a real post-handshake/readiness point, or make the Swift channel buffer and dispatch pre-`hello-ok` node events, while preserving command policy and exec approvals.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is a macOS app-backed Darwin node plus at least one Darwin skill requiring a binary: current main can send `node.invoke.request` before `hello-ok`, while the Swift client is still waiting for the connect response and not dispatching those events.

## Solution Assessment

Yes, a narrow handshake/readiness ordering fix is the best direction. A loopback-only skip, global remote-skills disable flag, or broad `denyCommands` workaround would hide legitimate remote macOS nodes instead of fixing the race.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40527 on current main. The gateway currently calls `refreshRemoteNodeBins` from `src/gateway/server/ws-connection/message-handler.ts` before it sends `hello-ok`; that probe can synchronously reach `NodeRegistry.invoke` and send `node.invoke.request` while `GatewayChannel.waitForConnectResponse` is still consuming frames and before `listen()` dispatches events. Implement a narrow readiness-ordering fix: defer the initial remote bin probe until after the node handshake/listener is actually ready, or buffer and dispatch pre-`hello-ok` node events in OpenClawKit. Preserve node command allowlist/declared-command behavior, remote skill eligibility semantics, and macOS exec approval enforcement. Do not add a global remote-skills disable flag or solve this by blanket-denying `system.run`/`system.which`. Add regression coverage for the ordering path and update `CHANGELOG.md` because this is user-facing log/startup behavior.
