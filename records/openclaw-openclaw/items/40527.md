## Summary

Keep open: current main and the latest release still connect the macOS app as a node advertising `system.which`/`system.run`, and the gateway still starts the remote bin probe for connected macOS nodes; the related open PR is conflicting and not a safe canonical fix as submitted.

## What This Changes

The issue asks OpenClaw to stop remote skill binary probe timeouts for macOS app-backed nodes, especially when the app node is co-located with the gateway.

## Best Possible Solution

Keep this as the focused tracker until maintainers choose and prove a narrow macOS app-node probe strategy: skip co-located app nodes, defer/back off early probes, or otherwise stabilize readiness without adding a broad config flag by default.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection does show that macOS app nodes advertise `system.which`/`system.run` and the gateway still starts the remote bin probe on node connect.

## Solution Assessment

Unclear as a final fix path. Current main has partial allowlist hardening and connectivity preflight, but the best remaining fix needs maintainer choice plus live macOS app-node proof rather than another speculative config knob.
