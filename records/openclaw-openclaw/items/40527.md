## Summary

Keep open: current main still has a source-reproducible automatic remote-bin probe/readiness path, and this issue is explicitly paired with the open closing pull request at https://github.com/openclaw/openclaw/pull/83429 rather than a merged fix.

## What This Changes

The issue reports startup/reconnect timeout warnings from automatic remote skill binary probing against macOS app-backed nodes, including a co-located app node on the gateway host.

## Best Possible Solution

Review, land, or replace https://github.com/openclaw/openclaw/pull/83429 with a readiness-aware gateway/macOS-node fix that prevents premature automatic probes without breaking valid remote macOS shell execution.

## Reproduction Assessment

Yes. Current main can start automatic remote-bin probing before the Swift app starts its normal receive loop, and the April 2 live comment confirms automatic probe timeouts while manual `system.which` succeeds.

## Solution Assessment

Unclear for the exact open PR path. The maintainable fix should address node readiness or a narrow co-located non-exec case rather than globally removing macOS app-backed `system.run`/`system.which` support.
