## Summary

Keep open: current main has reduced stale-node warning behavior, but the central macOS app-node connect-time remote-bin probe/readiness problem is not proven fixed. The automatic refresh path still runs on node connect, app-backed macOS nodes still legitimately advertise and handle `system.which`/`system.run`, and the related PR paths either remain open/conflicting or were closed because removing app-node exec commands was the wrong direction.

## What This Changes

The issue reports timeout warnings and startup/reconnect delay when automatic remote skill binary probing targets a macOS app-backed node, especially when the app node is co-located with the gateway.

## Best Possible Solution

Validate current main with a real co-located macOS app-backed node and a remote macOS app-backed node; if the timeout still appears, fix probe readiness, co-location skipping, or retry/backoff in `src/infra/skills-remote.ts` and the node connect path while preserving declared `system.which`/`system.run` support.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. Source inspection still shows the same automatic connect-time probe path, and issue comments plus related reports provide real macOS timeout evidence that maps to that path.

## Solution Assessment

Unclear. Removing app-node exec commands has already been rejected; the best fix is likely a narrow readiness, co-location, or probe-throttling change that preserves the documented macOS node execution contract.
