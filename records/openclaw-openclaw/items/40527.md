## Summary

Keep open: current main has related improvements, but the central macOS app-node remote-bin probe timing problem is not proven fixed. The gateway still starts automatic remote-bin refresh from the node connect path, the probe can still invoke `system.which`/`system.run`, and the attempted PR that removed app-node exec commands was closed as the wrong fix direction.

## What This Changes

The issue asks OpenClaw to stop macOS app-backed nodes from causing remote skill binary probe timeouts during gateway startup or node reconnect, especially when the app node is co-located with the gateway.

## Best Possible Solution

Validate current main with a real co-located and remote macOS app-backed node; if the timeout still appears, gate or defer automatic remote-bin refresh until node readiness is proven while preserving declared system.run/system.which support.

## Reproduction Assessment

No high-confidence live current-main reproduction is available from this read-only review. The source path still maps to the reported behavior, and older live reports confirm the timeout family, but current-main/macOS validation is needed after recent preflight and exec-routing changes.

## Solution Assessment

Unclear. The rejected command-removal PR was not the right fix; the maintainable path is a narrow readiness, co-location, or probe-throttling fix that preserves the documented macOS app-node execution contract.
