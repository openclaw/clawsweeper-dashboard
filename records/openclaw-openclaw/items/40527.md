## Summary

Keep open: current main and v2026.6.10 still run automatic remote macOS skill-bin probes for connected app nodes that advertise `system.which` or `system.run`, and the remaining fix needs maintainer judgment on readiness, co-location, and retry/backoff semantics with live macOS proof.

## What This Changes

The issue asks OpenClaw to stop remote skill binary probe timeouts/noisy startup delays for macOS app-backed nodes, especially when the app node is co-located with the gateway.

## Best Possible Solution

Implement the active runtime fix in `src/skills/runtime/remote.*`: probe only ready app nodes, avoid noisy co-located app-node startup behavior, add accepted retry/backoff or cache invalidation semantics, and prove it with a real macOS app node.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows current main still automatically probes connected macOS nodes that advertise `system.which` or `system.run`, and issue comments include real macOS app-node timeout observations.

## Solution Assessment

Unclear: the reported fixes are plausible, but the best solution requires a maintainer choice among co-located skip logic, readiness deferral, and cache/backoff policy rather than a broad disable flag.
