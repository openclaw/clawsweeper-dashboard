## Summary

Keep this open: current main still probes connected macOS nodes from the remote skills runtime, the related cache/backoff PR is open but conflicting/stale, and the closed PR that tried to remove app-node exec commands was explicitly rejected as the wrong direction.

## What This Changes

The issue reports automatic remote skill binary probes timing out for macOS app-backed nodes, especially when co-located with the gateway, causing startup/reconnect delay and misleading log warnings.

## Best Possible Solution

Land a focused current-runtime probe readiness/backoff fix that preserves declared macOS app-node system.run/system.which, with live macOS gateway proof; use the broader macOS node stability issue for general invoke flapping if that proves to be the root cause.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: the reported timeout path maps to current connect-time refreshRemoteNodeBins source and later live comments, but this review did not run a current-main macOS app-node repro.

## Solution Assessment

Is this the best way to solve the issue? The issue direction is valid, but the safest fix is not the config flag or blanket command removal suggested in the body; it should be a readiness/backoff/co-location probe fix that keeps the macOS node exec contract intact.
