## Summary

Keep open: current main and the latest release still start remote bin refresh on macOS node connect and probe any macOS node that advertises `system.which` or `system.run`; related live reports and the stale/conflicting open PR show the remaining timing/probe-churn problem is not safely resolved yet.

## What This Changes

The issue reports gateway startup/reconnect delays and warning spam from remote skill binary probes against macOS app-backed nodes, especially when the app node is co-located with the gateway.

## Best Possible Solution

Land or sponsor a narrow remote-bin readiness/probe-throttling fix in the current `src/skills/runtime/remote.ts` and node-connect path that preserves the macOS app `system.run`/`system.which` contract and is proven with a real macOS app node run.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection and linked live reports show the same connect-time macOS node probe path and timing-sensitive failure family remain present enough to keep the issue open.

## Solution Assessment

Unclear as a final fix path. The safest direction is not a new config flag or removing app-node exec commands, but a readiness/order, retry, or throttling change that preserves declared macOS app node execution and has live proof.
