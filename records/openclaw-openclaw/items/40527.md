## Summary

Keep open. Current main still has a source-reproducible startup/connect ordering bug: the gateway can send the automatic remote bin probe before the macOS app-backed node has started normal event dispatch, and the newer related PR only reduces probe churn rather than proving this readiness path fixed.

## What This Changes

The issue reports that macOS app-backed nodes can time out during automatic remote skill binary probing when the node connects on the same host as the gateway or early in the macOS app-node connect lifecycle.

## Best Possible Solution

Move the automatic remote bin refresh behind a confirmed node readiness point after hello-ok delivery, or buffer and dispatch pre-hello node events in the Swift channel, while preserving command policy and macOS exec approval behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is a macOS app-backed Darwin node plus at least one Darwin skill requiring a binary: current main can issue node.invoke.request for system.which before the Swift client starts normal event dispatch after hello-ok.

## Solution Assessment

Yes. A readiness/order fix is the narrow maintainable direction; loopback skips, broad deny-command workarounds, or a global disable flag would mask legitimate remote macOS nodes instead of fixing the race.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40527 on current main. The bug is that node-role connect currently starts refreshRemoteNodeBins before hello-ok is sent, while the Swift macOS app node starts normal event dispatch only after sendConnect returns, so automatic system.which/system.run remote skill bin probes can time out even though direct node invocations work. Keep command policy and macOS exec approval behavior intact. Prefer moving the initial automatic remote bin refresh behind successful hello-ok delivery and node readiness, or adding a narrow Swift buffering/readiness fix if that is the better boundary. Do not add a broad skipRemoteBinProbe config, loopback-only policy, or deny system.run/system.which by default. Do not replace the cache/backoff work from https://github.com/openclaw/openclaw/pull/79438; if it has landed, integrate with it. Add focused regression coverage showing the initial Darwin node remote-bin probe cannot be sent before hello-ok/readiness, and add Swift coverage if GatewayChannel buffering/readiness changes. Add a changelog entry because this changes user-visible startup/probe behavior.
