## Summary

Keep open: current main and the latest release still run the automatic remote bin probe for macOS nodes that advertise `system.which` or `system.run`, while the related open PR is stale against the moved runtime and still needs maintainer policy/proof work.

## What This Changes

The issue asks OpenClaw to avoid connect-time remote skill binary probe timeouts for macOS app-backed nodes, especially when the app node is co-located with the gateway.

## Best Possible Solution

Land or sponsor a narrow remote-bin readiness/probe-throttling fix in the current `src/skills/runtime/remote.ts` and node-connect path, preserving macOS app `system.run`/`system.which` and proving it with a real macOS app node run.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection and linked live reports show the same connect-time macOS node probe path and timing-sensitive failure family remain present enough to keep the issue open.

## Solution Assessment

Unclear as a final fix path. The safer direction is not a new config flag or removing app-node exec commands, but a readiness/order, retry, or throttling change that preserves declared macOS app node execution and has live proof.
