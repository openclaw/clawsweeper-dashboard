## Summary

Keep open. Current main still has a source-reproducible connect-ordering path where the gateway can send the automatic remote bin probe before a Swift app-backed node has started normal event dispatch, and the related open PR only addresses probe churn/backoff rather than this readiness race.

## What This Changes

Avoid remote skill bin probe timeouts and warning spam when macOS app-backed nodes connect, especially co-located gateway/app nodes and early connect timing.

## Best Possible Solution

Move the automatic remote bin refresh behind a confirmed node readiness point after hello-ok delivery, or buffer pre-hello node events in the Swift channel, while preserving command policy and macOS exec approval behavior.

## Reproduction Assessment

Yes, as a high-confidence source reproduction path: a macOS app-backed Darwin node plus at least one Darwin skill requiring a binary can receive node.invoke.request for system.which before normal Swift event dispatch starts. I did not live-run the macOS setup from this read-only Linux checkout.

## Solution Assessment

Yes. A readiness/order fix is the narrow maintainable direction; loopback skips, broad deny-command workarounds, or a global disable flag would mask legitimate remote macOS nodes instead of fixing the race.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40527 as a narrow bug. Current main registers node sessions and starts refreshRemoteNodeBins before hello-ok is sent, while the Swift app starts normal listen/event dispatch only after sendConnect returns; automatic system.which probes can time out even when manual system.which works. Implement a readiness/order fix: defer the automatic remote bin refresh until after the node can receive normal events, or buffer and dispatch pre-hello node events in the Swift channel. Do not add a global skipRemoteBinProbe config, do not remove macOS system.run/system.which support, and do not broaden node command policy. Add focused regression coverage for the ordering/readiness behavior and preserve existing remote skill eligibility behavior.
