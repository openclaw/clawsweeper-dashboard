## Summary

Keep open. Current main still allows the automatic remote skill bin probe to run during the node connect handler before the macOS app-backed node has entered its normal event listener; the app now advertises and handles `system.which` and `system.run`, so the remaining bug is handshake/readiness ordering rather than a missing command implementation.

## What This Changes

The issue requests that remote skill binary probing stop timing out for macOS app-backed nodes, especially when the app node is co-located with the gateway or still completing its connect lifecycle.

## Best Possible Solution

Move the initial remote bin probe behind a real post-handshake/readiness point, or make the Swift channel buffer and dispatch pre-`hello-ok` node events safely. The fix should preserve node command declaration, allowlist filtering, device-token pairing, and exec approval semantics.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is a macOS app-backed Darwin node plus at least one Darwin skill requiring a binary: current main can send the automatic `system.which` probe before `hello-ok`, while the Swift client is still waiting for the connect response and not dispatching normal events.

## Solution Assessment

Yes for a narrow handshake/readiness ordering fix; no for blanket `denyCommands`, loopback-only skipping, or a global remote-skills disable flag. The maintainable fix should make automatic probes wait until the node can receive events, or make early event handling reliable.

## Repair Work Prompt

Fix the macOS app-backed node remote skill bin probe timeout described by https://github.com/openclaw/openclaw/issues/40527. Current main can call `refreshRemoteNodeBins` during the gateway node connect handler before `hello-ok` is sent, while `GatewayChannelActor` only starts normal event dispatch after the connect response, so an early `node.invoke.request` can be dropped and the automatic `system.which` probe times out. Implement a narrow readiness/order fix: either defer the initial remote bin probe until after the node handshake is complete and the node can receive events, or make the Swift channel safely buffer and dispatch pre-`hello-ok` node events. Preserve `system.which`/`system.run` declaration, node command allowlist filtering, device-token pairing, and exec approval behavior. Do not add a global remote-skills disable flag or solve this by denying `system.run`. Add regression coverage around the gateway connect/probe ordering and the Swift node session/channel behavior. Add a user-facing changelog entry because this affects gateway/macOS node behavior.
