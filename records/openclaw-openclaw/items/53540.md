## Summary

Keep open. Current main has timeout plumbing that can mitigate explicitly configured providers, but it does not prove a complete fix for the default OpenRouter-to-Anthropic large tool-parameter path: OpenRouter still uses the OpenAI-compatible transport, native Anthropic tool-argument streaming is on a different route, and no regression covers a silent OpenAI-compatible stream while large tool arguments are generated.

## What This Changes

The issue requests that embedded LLM calls survive or clearly handle large tool-call parameter generation, especially OpenRouter-to-Anthropic calls that can stay silent long enough to surface as `Network connection lost`.

## Best Possible Solution

Keep this as the canonical tracker for OpenRouter/Anthropic large tool-parameter generation failures. The end state should add a synthetic OpenAI-compatible streaming regression that withholds tool-call argument chunks long enough to exercise the watchdog path, then implement the chosen provider/runtime policy: supported provider-side tool-argument streaming, a safe timeout policy for silent tool-argument generation, or a tool-loop strategy that avoids requiring one huge `Write` payload.

## Reproduction Assessment

Yes: the issue gives a precise OpenRouter/Anthropic Opus `Write` repro, and current source still has a chunk-idle watchdog that can abort if the OpenAI-compatible route emits no chunks while tool arguments are generated. I did not run the provider live in this read-only review.

## Solution Assessment

No current-main solution is proven complete. Provider timeout plumbing is a useful mitigation for explicitly configured setups, but default OpenRouter still needs a maintainer decision on timeout defaults, OpenRouter-specific streaming, or large tool-payload behavior.
