## Label Justifications

- `P1`: The issue describes a real embedded-runner/provider workflow failure where large tool-call responses abort and the agent loses the produced work.

## Summary

Keep open. Current main and the latest shipped release still expose the reported OpenRouter-to-Anthropic large tool-parameter path: OpenRouter resolves through the OpenAI-compatible transport, native Anthropic fine-grained tool streaming is only wired on the Anthropic Messages route, and the embedded runner still aborts when no stream event arrives inside the single idle-watchdog window.

## What This Changes

The issue asks OpenClaw to stop losing embedded-runner LLM calls when a slow OpenRouter-routed Anthropic model spends longer than the request/idle timeout generating large tool-call parameters.

## Best Possible Solution

Keep this as the canonical tracker and land a provider/runtime-policy fix with a regression that simulates an OpenAI-compatible stream withholding tool-call arguments long enough to exercise the watchdog.

## Reproduction Assessment

Yes, source-backed reproduction is high confidence: the reporter provides precise OpenRouter/Anthropic Opus large-`Write` steps, and current main still has the OpenRouter OpenAI-compatible route plus a no-chunk idle watchdog. I did not run a live provider replay in this read-only review.

## Solution Assessment

No complete solution is on current main. The best fix needs maintainer/provider-policy agreement and regression proof rather than closing this based on adjacent timeout plumbing or the still-open related PR.
