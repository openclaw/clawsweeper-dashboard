## Summary

Keep open. Current main and the latest release still leave the reported OpenRouter-to-Anthropic large tool-parameter path exposed: OpenRouter model refs use the OpenAI-compatible transport, the native Anthropic fine-grained tool-streaming header is only on the separate Anthropic Messages route, and the embedded runner still aborts a stream when no event arrives inside the idle watchdog window.

## What This Changes

The issue requests fixing embedded runner `Network connection lost` failures when a provider withholds large tool-call parameters long enough to exceed the LLM stream idle window.

## Best Possible Solution

Keep this as the canonical tracker and land a provider/runtime-policy fix with a synthetic OpenAI-compatible streaming regression that withholds tool-call arguments long enough to exercise the watchdog.

## Reproduction Assessment

Yes for a high-confidence source-backed path: the reporter provides precise OpenRouter/Anthropic Opus large-`Write` steps, and current source still has the OpenRouter OpenAI-compatible route plus chunk-idle watchdog. I did not run the live provider replay in this read-only review.

## Solution Assessment

No complete solution is on current main. Provider timeout configuration can mitigate some setups and related pull request https://github.com/openclaw/openclaw/pull/65898 explores split first-token/idle handling, but neither currently proves the default OpenRouter large tool-parameter path fixed.
