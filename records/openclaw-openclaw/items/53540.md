## Summary

Keep this open: current main still routes OpenRouter text models through the OpenAI-compatible stream path and still relies on a single idle watchdog, so the reported large tool-argument failure is not proven fixed. The merged provider `timeoutSeconds` work is a useful mitigation for configured installs, but it does not resolve the default OpenRouter/Anthropic path or remove the need for live provider proof.

## What This Changes

The issue reports embedded runner failures when OpenRouter/Anthropic generates a large tool-call parameter payload slowly enough that no response chunks arrive before the request or idle timeout.

## Best Possible Solution

Keep this as the canonical tracker, run a current-main live OpenRouter large-Write replay, then implement the chosen provider/runtime policy with focused regression coverage for a silent OpenAI-compatible stream while tool-call arguments are generated.

## Reproduction Assessment

No high-confidence live current-main reproduction was established in this read-only review. The source path remains reproducible enough to keep open because OpenRouter still uses OpenAI-compatible streaming, the watchdog still times out chunk silence, and the issue provides a concrete OpenRouter/Opus large-Write reproducer.

## Solution Assessment

No proven best fix yet. A larger timeout is only a mitigation; maintainers should first confirm current OpenRouter behavior, then choose between provider-side tool-argument streaming, timeout policy, or a large-payload tool strategy.
