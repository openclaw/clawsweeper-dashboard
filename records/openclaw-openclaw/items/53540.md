## Label Justifications

- `P1`: The report describes a reproducible provider/runtime failure that stops an active agent turn and loses the subagent's intended output.
- `impact:message-loss`: The failed model/tool-call response produces no final output or file, so the user-visible agent result is lost.

## Summary

Keep open. Current main still leaves the reported OpenRouter-to-Anthropic large tool-parameter path exposed: OpenRouter resolves through the OpenAI-compatible transport, the native Anthropic fine-grained tool-streaming header is on a different transport path, and the embedded runner still aborts when no stream event arrives inside the idle window.

## What This Changes

The issue asks OpenClaw to avoid or clearly recover from embedded-runner failures when a model spends longer than the underlying timeout generating a large tool-call parameter payload.

## Best Possible Solution

Keep this as the canonical tracker and land a tested provider/runtime policy that either streams OpenRouter Anthropic tool arguments where supported or handles silent large tool-argument generation without losing the run.

## Reproduction Assessment

Yes, we have a high-confidence source-backed reproduction path: the reporter provided precise OpenRouter/Anthropic large-Write steps, and current main still has a chunk-idle watchdog on the OpenRouter OpenAI-compatible route. I did not run the live provider replay in this read-only review.

## Solution Assessment

No complete fix is on current main. Provider timeout configuration can mitigate some setups, but the default remote OpenRouter path remains capped by the idle watchdog and does not enable OpenRouter's Anthropic fine-grained tool-streaming header.
