## Summary

Keep open. Current main still has the affected OpenRouter path: OpenRouter model refs use the OpenAI-compatible transport, native Anthropic fine-grained tool streaming is not applied there, and the embedded runner still aborts streams when no chunk arrives within the idle watchdog window.

## What This Changes

The issue reports embedded-runner `Network connection lost.` failures when a slow OpenRouter Anthropic route generates a large tool-call parameter payload before emitting response chunks.

## Best Possible Solution

Keep this as the canonical tracker and land a provider/runtime-policy fix with a synthetic OpenAI-compatible streaming regression that withholds tool-call argument chunks long enough to exercise the watchdog.

## Reproduction Assessment

Yes for a live provider setup: the reporter gives a precise OpenRouter/Anthropic Opus large-`Write` repro, and source inspection confirms the affected route still has a chunk-idle watchdog. I did not run the live OpenRouter replay in this read-only pass.

## Solution Assessment

No complete solution is on current main. Provider timeout configuration can mitigate some setups, and native Anthropic fine-grained tool streaming helps the native route, but neither proves the default OpenRouter OpenAI-compatible route is fixed.
