## Summary

Keep this open: current main and the latest release still route OpenRouter Anthropic models through the OpenAI-compatible stream path while the embedded runner uses a chunk-silence idle watchdog. The existing provider `timeoutSeconds` path is a useful opt-in mitigation, but it does not prove the default large tool-parameter failure is fixed or settle the provider/runtime policy.

## What This Changes

The issue asks OpenClaw to stop losing embedded-runner work when large generated tool-call parameters create a long silent LLM response window and surface as `Network connection lost.`

## Best Possible Solution

Keep this as the canonical tracker, run a current-main live OpenRouter large-Write replay, then implement the chosen provider/runtime policy with focused regression coverage for silent tool-argument generation.

## Reproduction Assessment

No high-confidence live current-main reproduction was established in this read-only review. The source path remains reproducible enough to keep open because OpenRouter still uses OpenAI-compatible streaming, the watchdog still times out chunk silence, and the issue provides concrete OpenRouter/Opus large-Write steps.

## Solution Assessment

No proven best fix yet. Raising timeouts is only a mitigation; maintainers should confirm current OpenRouter behavior, then choose between provider-side tool-argument streaming, timeout policy, or a large-payload tool strategy.
