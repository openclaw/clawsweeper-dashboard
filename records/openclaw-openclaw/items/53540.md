## Summary

Keep this open: current main and v2026.6.6 still route OpenRouter text models through the OpenAI-compatible path and the embedded runner still uses a chunk-progress idle watchdog. Existing timeoutSeconds support is a mitigation, not proof that the default OpenRouter/Anthropic large tool-argument failure is fixed.

## What This Changes

The issue reports embedded runner LLM calls failing with a generic network error when slow providers generate large tool-call parameters before emitting any streamed chunk.

## Best Possible Solution

Keep this as the canonical tracker, run a current-main live OpenRouter large Write replay, then implement the chosen provider/runtime policy with a focused regression for a silent OpenAI-compatible stream while tool-call arguments are generated.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. The source path remains plausible because OpenRouter still uses OpenAI-compatible streaming and the idle watchdog still times out chunk silence, but current provider behavior needs live confirmation.

## Solution Assessment

No proven best fix yet. A timeout increase is only a mitigation; maintainers should first confirm current OpenRouter behavior, then choose between provider-side tool-argument streaming, timeout policy, or a large-payload tool strategy.
