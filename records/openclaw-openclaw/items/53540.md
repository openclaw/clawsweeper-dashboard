## Summary

Keep open. Current main still has the affected default route and no complete source or test proof that large OpenRouter-to-Anthropic tool-call parameter generation survives the silent streaming gap described in the updated report.

## What This Changes

The issue requests that embedded LLM runs avoid losing OpenRouter/Anthropic tool calls when large tool-call parameters take longer than the current request or idle timeout, or at least retry and report the failure more clearly.

## Best Possible Solution

Keep this as the canonical tracker for OpenRouter/Anthropic large tool-parameter failures. The durable end state should add a synthetic OpenAI-compatible stream regression that withholds tool-call argument chunks long enough to exercise the watchdog, then implement the selected runtime/provider policy for supported tool-argument streaming, timeout behavior, or large tool-payload avoidance.

## Reproduction Assessment

Yes. The issue gives a precise OpenRouter/Anthropic Opus large-Write repro, and current source inspection still shows the affected OpenAI-compatible route plus a single idle watchdog; this pass did not run the live provider.

## Solution Assessment

No complete current-main solution is proven. Provider timeout configuration is a mitigation and native Anthropic fine-grained streaming helps only the native Anthropic route, while the OpenRouter route still needs a maintainer-approved fix path.
