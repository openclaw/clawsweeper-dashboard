## Summary

Keep open. The updated report is still plausible on current main: OpenRouter is routed through the OpenAI-compatible transport, the native Anthropic fine-grained tool streaming path does not apply to that route, and the embedded runner still has a single chunk-idle watchdog that can abort a silent stream while large tool-call parameters are generated. The overlapping timeout PR is still open in the provided context, so this should remain the canonical tracker until a fix lands.

## What This Changes

The issue requests that OpenClaw avoid or clearly handle embedded-runner network failures when slow LLMs generate large tool-call parameter payloads, especially OpenRouter-to-Anthropic large `Write` calls.

## Best Possible Solution

Keep this as the canonical tracker for OpenRouter/Anthropic large tool-parameter generation failures. The end state should add a synthetic OpenAI-compatible streaming regression that withholds tool-call argument chunks long enough to exercise the watchdog, then implement the selected runtime/provider policy for safe timeout behavior or supported tool-argument streaming.

## Reproduction Assessment

Yes. The issue gives a precise OpenRouter/Anthropic Opus large-`Write` repro, and current main still has the 120-second chunk-idle watchdog on the affected OpenAI-compatible route; I did not run the live provider in this read-only pass.

## Solution Assessment

No complete current-main solution is proven. Provider timeout configuration and native Anthropic fine-grained tool streaming can help some setups, but they do not by themselves cover the default OpenRouter OpenAI-compatible route without a maintainer policy decision.
