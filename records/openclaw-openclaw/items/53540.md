## Summary

Keep open. Current main and v2026.5.22 still have the OpenRouter Anthropic path on the OpenAI-compatible transport plus a 120s per-chunk idle watchdog; provider timeout configuration can mitigate opted-in setups, but it does not prove the default large tool-argument path is fixed.

## What This Changes

Fix embedded-runner LLM calls that fail with `Network connection lost.` when a slow model generates large tool-call parameters before any stream chunk is emitted.

## Best Possible Solution

Keep this as the canonical bug tracker, choose the provider/runtime policy for silent large tool-argument generation, add a synthetic regression that withholds OpenAI-compatible tool-call argument chunks, and prove the final behavior with real OpenRouter Anthropic evidence.

## Reproduction Assessment

Yes, at source level. The issue gives precise OpenRouter Anthropic large-`Write` steps, and current main still has the default OpenRouter OpenAI-compatible route plus the per-chunk idle watchdog; I did not run a live provider replay.

## Solution Assessment

No complete fix is on current main. Explicit provider timeout configuration can mitigate opted-in setups, but the default OpenRouter Anthropic path still needs a chosen timeout, streaming, or large-tool-payload policy with regression proof.
