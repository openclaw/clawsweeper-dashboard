## Summary

Keep open: the PR targets a real MCP stdio bug, but the current patch routes logs inside the command action, after Commander preAction/config preflight can already write warnings to stdout, and the PR still lacks inspectable real stdio proof.

## What This Changes

The PR adds `routeLogsToStderr()` to the channel MCP server, adds an in-memory `tools/list` capability test, and adds a changelog entry.

## Best Possible Solution

Route or suppress `openclaw mcp serve` startup output before CLI/config preflight, then keep/adjust the MCP server defense and add strict stdio proof for `tools/list`.

## Reproduction Assessment

Yes, at source level: `openclaw mcp serve` is documented as a stdio server, the shared preAction/config guard can run before the MCP action, and the current `mcp` command policy does not request stdout suppression. I did not run a live Hermes or raw stdio session in this read-only review.

## Solution Assessment

No, not as currently proposed. `routeLogsToStderr()` is the right mechanism, but calling it inside `serveOpenClawChannelMcp` is too late for preAction/config warnings; the safer fix is to reserve stdout in the MCP serve startup path before the config guard runs.
