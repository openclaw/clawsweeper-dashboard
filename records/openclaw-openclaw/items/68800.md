## Summary

Keep open. Current main still lacks stderr routing on the channel MCP stdio entrypoint, so this PR is not obsolete; config warnings can still be emitted during config load before the MCP transport is connected. The production change is the right narrow direction, but the PR needs a changelog entry and the added test should either directly cover stdio isolation or be renamed because it currently uses an in-memory transport.

## What This Changes

The PR imports `routeLogsToStderr`, calls it at the start of `serveOpenClawChannelMcp`, and adds a channel MCP `tools/list` capability test.

## Best Possible Solution

Land the narrow channel MCP stderr-routing fix on current main, with `routeLogsToStderr()` executed before config resolution, plus a regression test that proves the stdio/log-routing order and a user-facing changelog entry.

## Reproduction Assessment

Yes: the source path is high-confidence. `openclaw mcp serve` calls the channel MCP server, resolves runtime config through `loadConfig()`, and config validation warnings can be logged before any current-main stderr routing exists on that entrypoint.

## Solution Assessment

Yes for the production direction: routing logs to stderr at the top of `serveOpenClawChannelMcp()` mirrors the existing MCP tools stdio helper and is the narrowest maintainable fix. The PR is incomplete as merge-ready because its test does not prove that behavior and the changelog is missing.
