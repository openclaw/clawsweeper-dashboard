## Summary

Keep open. Current main still converts parameter-validation failures into ordinary tool errors and the final payload policy can still emit synthetic warning payloads for mutating `write`/`message`-style tools, so the requested chat-surface suppression is not implemented. The adjacent open PR #68070 is useful retryability work, but its patch only adds `retryable: false` metadata and does not change final user-facing payload suppression.

## What This Changes

The issue requests that missing or invalid tool-parameter validation failures stay internal to the agent/runtime instead of being forwarded as raw or synthetic warning messages to chat surfaces.

## Best Possible Solution

Carry typed validation-only metadata from the parameter-validation wrapper through adapter normalization, tool-error summary storage, and final payload policy. Suppress normal chat warning payloads only for validation-only failures while still logging them, returning them to the model/runtime, and preserving visible warnings for real mutating side-effect failures and verbose diagnostics.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: pass a mutating `write` or `message` `lastToolError` with a required-parameter validation string into `buildEmbeddedRunPayloads`; the current tests assert a visible warning payload. I did not run tests because this review is read-only.

## Solution Assessment

Yes. Typed validation-only classification through the existing tool-error pipeline is the narrowest maintainable fix; keyword-only suppression or disabling all mutating warnings would be less safe.
