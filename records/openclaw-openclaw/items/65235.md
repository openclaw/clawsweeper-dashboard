## Summary

Keep open. Current main still routes validation-shaped mutating tool failures through the generic mutating-tool warning path, and the adjacent retryability PR addresses loop metadata rather than suppressing final chat-surface warning payloads.

## What This Changes

The issue asks OpenClaw to keep missing or invalid tool-parameter failures internal instead of sending raw validation failures as chat warnings.

## Best Possible Solution

Add a typed parameter-validation classification carried from validation through adapter normalization and ToolErrorSummary, then suppress only that class in final chat payload policy.

## Reproduction Assessment

Yes. By source inspection, calling buildEmbeddedRunPayloads with a mutating lastToolError such as { toolName: "message", meta: "reply", error: "text required" } and no assistant reply follows the visible warning path, and current tests assert that behavior.

## Solution Assessment

Yes. A typed validation-only signal is the narrowest maintainable fix; text matching or globally hiding mutating warnings would risk masking real failed side effects.
