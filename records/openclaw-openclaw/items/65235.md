## Summary

Keep open. Current main still lacks an end-to-end parameter-validation classification, so mutating validation failures can become user-facing warning payloads; related PR #68070 addresses retryability, not final chat-surface suppression.

## What This Changes

Request that tool parameter validation failures be handled internally instead of forwarding raw or summary failure warnings to chat channels such as Feishu/DM.

## Best Possible Solution

Mark parameter validation failures as typed internal input errors end-to-end and suppress only that validation class from chat payloads while preserving warnings for real mutating side-effect failures.

## Reproduction Assessment

Yes. A static reproduction exists on current main: pass buildEmbeddedRunPayloads a mutating lastToolError such as write missing content or message text required, and the current policy emits a user-facing warning.

## Solution Assessment

Yes. A typed validation-only signal propagated into ToolErrorSummary and final payload policy is the narrowest maintainable path; global mutating-warning suppression or message-text matching would hide real failed side effects.
