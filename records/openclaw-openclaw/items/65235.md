## Summary

Keep open. Current main still routes mutating validation-shaped tool failures through the final user-facing warning path, and the related closed retryability PR did not implement chat-surface suppression for this UX bug.

## What This Changes

The issue requests suppressing missing or invalid tool-parameter validation failures from Feishu/DM and other user chat surfaces.

## Best Possible Solution

Carry a typed validation/internal-only classification from parameter validation and adapter normalization through ToolErrorSummary, then suppress only that class in final chat payload policy.

## Reproduction Assessment

Yes, source-reproducible with high confidence. On current main, calling buildEmbeddedRunPayloads with a mutating lastToolError such as `{ toolName: "message", meta: "reply", error: "text required" }` and no user-facing acknowledgement follows the visible warning branch, and current tests assert that payload.

## Solution Assessment

Yes. A typed validation/internal-only signal is the narrowest maintainable fix; text matching, retryability-only metadata, or globally hiding mutating warnings would either miss the chat policy or mask real failed side effects.

Current main still sends mutating tool parameter-validation failures to user chat as warning payloads because validation failures are serialized as generic tool errors and ToolErrorSummary has no validation/internal-only classification. Add a typed classification for missing/malformed tool-parameter validation failures, thread it through the adapter/tool execution end state into ToolErrorSummary, and suppress only that class in buildEmbeddedRunPayloads. Cover write/read/edit required-param failures and exec missing-command or equivalent malformed-parameter failures where the current path can be classified safely. Preserve visible warnings for real unresolved mutating side-effect failures, preserve verbose details where appropriate for non-validation failures, and do not implement the broader configurable pre-dispatch validation feature from https://github.com/openclaw/openclaw/issues/56771. Do not revive the ad-hoc retryable-only approach from https://github.com/openclaw/openclaw/pull/68070 unless it becomes a typed contract that also drives chat visibility. Add focused regression tests and a CHANGELOG.md Fixes entry unless maintainers decide this is release-note-exempt.
