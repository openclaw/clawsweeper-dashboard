## Summary

Keep open. Current main still lacks a typed parameter-validation signal in the tool-error path, so validation failures for mutating tools can still be converted into final user-facing warning payloads; the adjacent open PR #68070 addresses retryability for #68026, not chat-surface suppression for this issue.

## What This Changes

The issue requests suppressing tool parameter validation failures from end-user chat surfaces while still exposing them internally to the agent/runtime.

## Best Possible Solution

Add a typed parameter-validation classification from validation through adapter normalization and stored tool-error summaries into final payload policy, then suppress only that class from normal chat warnings while preserving real mutating-failure warnings.

## Reproduction Assessment

Yes. A static reproduction exists on current main: pass buildEmbeddedRunPayloads a mutating lastToolError such as message/text required or write/missing content, and the current policy produces a user-facing warning summary.

## Solution Assessment

Yes. A typed validation-only signal is the narrowest maintainable fix; global mutating-warning suppression or message-text matching would risk hiding real failed side effects.
