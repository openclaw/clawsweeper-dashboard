## Summary

Keep open. Current main still collapses tool parameter validation failures into generic tool errors and final payload policy still sends mutating recoverable failures as user-facing warnings; the adjacent open PR #68070 only targets retry loops for #68026, not this chat-surface suppression request.

## What This Changes

Suppress missing or invalid tool-parameter validation failures from normal user chat surfaces while still letting the agent/runtime handle them internally.

## Best Possible Solution

Add a typed parameter-validation classification from required-parameter checks through adapter normalization and ToolErrorSummary into final payload policy, then suppress only that class from normal chat warnings.

## Reproduction Assessment

Yes. A static reproduction exists by calling buildEmbeddedRunPayloads with a mutating lastToolError such as { toolName: "message", meta: "reply", error: "text required" } and no assistant reply; current tests assert it produces a user-facing warning payload.

## Solution Assessment

Yes. A typed validation-only signal is the narrowest maintainable solution; message-text matching or globally hiding mutating warnings would risk masking real failed side effects.

Add a typed validation classification at the required-parameter error source and propagate it through tool adapter normalization, stored ToolErrorSummary, and buildEmbeddedRunPayloads. Suppress only that validation class from normal chat warning payloads; do not globally suppress mutating tool warnings or hide real side-effect failures. Coordinate with open PR #68070 if it has landed or overlaps, but do not treat retryability alone as sufficient.
