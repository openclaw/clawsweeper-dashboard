## Summary

Keep open. Current main still routes validation-shaped mutating tool failures through the final chat-warning policy, current tests assert that visible warning, and https://github.com/openclaw/openclaw/pull/68070 addresses retryability metadata rather than chat-surface suppression.

## What This Changes

The issue requests suppressing missing or invalid tool-parameter failures from user-visible chat surfaces while keeping them available to the agent runtime and logs.

## Best Possible Solution

Carry a typed validation/internal-only classification from parameter validation through adapter normalization and ToolErrorSummary, then suppress only that class in final chat payload policy while preserving real mutating failure warnings.

## Reproduction Assessment

Yes. By source inspection, calling buildEmbeddedRunPayloads with a mutating lastToolError such as { toolName: "message", meta: "reply", error: "text required" } and no user-facing acknowledgement follows the visible warning branch, and current tests assert that behavior.

## Solution Assessment

Yes. A typed validation-only signal is the narrowest maintainable fix; adding only retryable metadata, text matching, or globally hiding mutating warnings would miss the chat-surface policy or mask real failed side effects.
