## Summary

Keep open. Current main still lacks a typed parameter-validation signal in the tool-error path, so mutating validation failures can become user-facing warning payloads; the adjacent open PR #68070 only addresses retryability, not final chat-surface suppression.

## What This Changes

Suppress user-facing chat warnings for tool parameter validation failures while leaving the agent runtime, retry behavior, and logs able to handle the error internally.

## Best Possible Solution

Add a typed parameter-validation classification from validation through adapter normalization and stored summaries into final payload policy, then suppress only that class from normal chat warnings.

## Reproduction Assessment

Yes. A static reproduction exists on current main: pass buildEmbeddedRunPayloads a mutating lastToolError such as message/text required or write/missing content and the current policy produces a user-facing warning summary.

## Solution Assessment

Yes. A typed validation-only signal is the narrowest maintainable fix; global mutating-warning suppression or message-text matching would risk hiding real failed side effects.
