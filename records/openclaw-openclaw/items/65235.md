## Summary

Keep open. Current main still sends validation-shaped mutating tool failures through the user-facing warning path, and the related retryability PR does not implement chat-surface suppression for this issue.

## What This Changes

The issue asks to suppress raw tool-parameter validation failures from user-facing chat surfaces while still handling them inside the agent runtime.

## Best Possible Solution

Carry a typed validation/internal-only classification from parameter validation through adapter normalization and ToolErrorSummary, then suppress only that class in final chat payload policy while preserving real mutating failure warnings.

## Reproduction Assessment

Yes by source inspection. Calling buildEmbeddedRunPayloads with a mutating lastToolError such as { toolName: "message", meta: "reply", error: "text required" } and no user-facing acknowledgement follows the visible warning branch, and current tests assert that payload.

## Solution Assessment

Yes. A typed validation-only signal is the narrowest maintainable fix; adding only retryable metadata, text matching, or globally hiding mutating warnings would miss the chat-surface policy or mask real failed side effects.
