## Summary

Keep open. Current main still has no typed parameter-validation signal and still surfaces mutating validation-shaped tool failures as user-facing warning payloads; the adjacent open PR only handles retryability for #68026, not chat-surface suppression for this issue.

## What This Changes

Suppress tool parameter validation failures from user-facing chat surfaces while keeping them available to the agent/runtime.

## Best Possible Solution

Carry a typed parameter-validation classification from required-parameter checks through adapter normalization and ToolErrorSummary, then suppress only that class in final chat warning payloads.

## Reproduction Assessment

Yes, source-backed. A call to buildEmbeddedRunPayloads with lastToolError like { toolName: "message", meta: "reply", error: "text required" } and no assistant reply follows the mutating warning branch, and current tests expect a visible Message warning.

## Solution Assessment

Yes. A typed validation-only signal is the narrow maintainable fix; text matching or globally hiding mutating warnings would risk masking real failed side effects.
