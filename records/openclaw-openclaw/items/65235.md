## Summary

Keep open. Current main still carries tool parameter validation failures as generic tool errors, and the final payload policy still emits mutating validation-shaped failures as user-facing warning payloads; the adjacent open PR only targets retry-loop metadata for a different report, not chat-surface suppression.

## What This Changes

The issue requests that missing or invalid tool parameters be handled inside the agent runtime instead of being forwarded as raw warning text to chat channels.

## Best Possible Solution

Add a typed parameter-validation classification from required-parameter checks through adapter normalization and ToolErrorSummary into final payload policy, then suppress only that class from normal chat warnings.

## Reproduction Assessment

Yes, by source-level reproduction: call buildEmbeddedRunPayloads with a mutating lastToolError such as { toolName: "message", meta: "reply", error: "text required" } and no assistant reply, and current tests assert it emits a visible warning payload. I did not run tests because this review is read-only.

## Solution Assessment

Yes. A typed validation-only signal is the narrowest maintainable fix; matching error text or globally hiding mutating warnings would risk masking real failed side effects.
