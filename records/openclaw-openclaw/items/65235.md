## Summary

Keep open. Current main still carries tool parameter validation failures as generic tool errors, and the final payload policy can still emit mutating validation-shaped failures as user-facing warning payloads; the related retry-loop PR does not implement chat-surface suppression.

## What This Changes

The issue requests suppressing raw tool parameter validation failures from end-user chat surfaces while keeping them available to the agent runtime and logs.

## Best Possible Solution

Carry a typed parameter-validation/internal-only signal from required-parameter checks through adapter normalization and ToolErrorSummary, then suppress only that class from final chat warning payloads while preserving real side-effect failures.

## Reproduction Assessment

Yes, by source-level reproduction: call `buildEmbeddedRunPayloads` with a mutating `lastToolError` such as `{ toolName: "message", meta: "reply", error: "text required" }` and no assistant reply; current tests assert this produces a visible warning payload. I did not run tests because this review is read-only.

## Solution Assessment

Yes. A typed validation-only signal is the narrowest maintainable fix; matching error text or globally hiding mutating warnings would risk masking real failed side effects.
