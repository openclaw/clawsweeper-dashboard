## Summary

Keep open. The PR targets a real current-main Gateway mismatch, but it is not merge-ready because it predates the Windows loopback guard, needs Gateway locality/auth documentation approval, and lacks inspectable after-fix proof.

## What This Changes

The PR changes `resolveGatewayListenHosts()` and its tests so tailnet/custom bind hosts also add a bindable `127.0.0.1` listener for internal gateway calls.

## Best Possible Solution

Choose one canonical same-host Gateway locality model; if server-side dual-bind is approved, rebase this patch over current main with the Windows guard preserved, docs updated, focused tests, and live tailnet/custom proof.

## Reproduction Assessment

Yes, by source inspection: current main listens only on the resolved tailnet/custom bind host while implicit local call paths still target `127.0.0.1`. I did not run a live tailnet setup in this read-only review.

## Solution Assessment

Unclear. Server-side dual-bind is a plausible narrow fix and avoids relaxing plaintext non-loopback client rules, but it must preserve the Windows guard, update docs, and be approved against the Gateway locality/auth model; https://github.com/openclaw/openclaw/pull/50751 remains an alternate client-side path.
