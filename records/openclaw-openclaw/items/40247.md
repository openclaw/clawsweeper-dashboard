## Summary

Keep open. Current main still has the tailnet/custom loopback-listener gap this PR targets, but the branch is not merge-ready because it predates the Windows loopback guard, changes gateway locality/auth behavior without docs alignment, and lacks inspectable real behavior proof.

## What This Changes

The PR changes `resolveGatewayListenHosts()` and its tests so specific tailnet/custom bind hosts also add a bindable `127.0.0.1` listener for internal gateway calls.

## Best Possible Solution

Choose one canonical same-host Gateway locality model; if server-side dual-bind is accepted, land it with the Windows guard preserved, docs/auth guidance updated, and focused live tailnet/custom proof.

## Reproduction Assessment

Yes, by source inspection. Current main starts listeners only from `resolveGatewayListenHosts()`, that resolver returns only the tailnet/custom bind host, and implicit local clients still target `127.0.0.1`.

## Solution Assessment

Unclear. Server-side dual-bind is a plausible narrow fix and avoids plaintext non-loopback client targeting, but it needs maintainer approval for the locality policy plus a rebase that preserves the Windows guard and updates docs.
