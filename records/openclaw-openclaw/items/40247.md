## Summary

Keep open: the tailnet/custom bind bug is still present on current main, but this external PR is not ready to merge because it would drop the merged Windows loopback guard, lacks inspectable real behavior proof, and needs Gateway locality/auth policy alignment.

## What This Changes

This PR changes Gateway listen-host resolution and tests so specific tailnet/custom bind hosts also try to listen on `127.0.0.1` for internal calls.

## Best Possible Solution

Choose the approved Gateway local-call model; if listener-side dual-bind is accepted, preserve the Windows guard, document the listener/auth semantics, update tests, and include redacted live tailnet/custom proof.

## Reproduction Assessment

Yes, source-level: current main listens only on the specific non-loopback bind host while local Gateway callers and tests still target `127.0.0.1`. I did not run a live tailnet setup in this read-only review.

## Solution Assessment

Unclear. Listener-side dual-bind is plausible because it preserves existing loopback local callers, but this branch as-is is not the best form because it drops the Windows guard and lacks docs, proof, and locality/auth approval.

## Review Findings

Overall correctness: patch is incorrect

  - body: Current main skips `::1` probing on Windows because binding both `127.0.0.1` and `::1` can hang Gateway HTTP requests. If dual-bind is accepted, update the Gateway docs so operators understand the added local listener behavior.
  - confidence: 0.84
