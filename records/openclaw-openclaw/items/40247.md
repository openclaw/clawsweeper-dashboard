## Summary

Keep this PR open: it targets a real source-level gateway mismatch, but it is not merge-ready because contributor proof is missing, the branch predates the Windows loopback guard, and listener-side dual-bind needs gateway locality/security policy alignment.

## What This Changes

The branch changes `resolveGatewayListenHosts()` and its tests so specific non-loopback gateway bind hosts also attempt a `127.0.0.1` listener for internal calls.

## Best Possible Solution

Adopt one canonical Gateway local-call model; if listener-side dual-bind is accepted, preserve the Windows guard, update docs/tests, reconcile the overlapping client-side fix, and require redacted runtime proof.

Yes, at source level. Current main constructs local gateway URLs as `127.0.0.1` while non-loopback listeners return only the configured bind host; I did not run a live tailnet/custom gateway in this read-only review.

## Solution Assessment

No, not as-is. Listener-side dual-bind is plausible, but the mergeable solution must preserve Windows behavior, document and approve the listener-surface change, and reconcile the competing client-side approach.
