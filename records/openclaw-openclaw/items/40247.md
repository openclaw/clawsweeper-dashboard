## Summary

Keep open. The PR addresses a real, source-visible Gateway tailnet/custom bind reachability problem, but the branch is not ready because it drops the current Windows loopback guard, changes listener locality semantics without docs/policy alignment, lacks inspectable real behavior proof, and overlaps an open client-side alternative.

## What This Changes

The PR changes `resolveGatewayListenHosts()` and `src/gateway/net.test.ts` so specific non-loopback bind hosts also attempt a `127.0.0.1` listener for internal Gateway calls.

## Best Possible Solution

Choose the approved Gateway local-call model; if listener-side dual-bind is accepted, rebase this PR over current main with the Windows guard preserved, docs/tests updated, and redacted live tailnet/custom proof.

## Reproduction Assessment

Yes, from source inspection. Current main listens only on the specific non-loopback bind host while local Gateway connection details and tests still target `127.0.0.1`; I did not run a live tailnet setup in this read-only review.

## Solution Assessment

Unclear. Listener-side dual-bind is plausible because it preserves implicit local loopback callers, but this branch must preserve the Windows guard and be approved/documented against Gateway locality and auth policy before merge.

## Review Findings

Overall correctness: patch is incorrect

  - body: Current main skips `::1` probing on Windows because binding both `127.0.0.1` and `::1` can hang Gateway HTTP requests. This branch rewrites the loopback branch and probes `::1` unconditionally, so rebasing it as-is would reintroduce that Windows hang.
  - confidence: 0.9

If dual-bind is accepted, update the Gateway docs so operators understand the added local listener behavior.
