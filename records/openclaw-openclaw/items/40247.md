## Summary

Keep open. Current main still listens only on the selected tailnet/custom bind host while local gateway calls target loopback, and this PR is a plausible server-side fix, but it changes the documented listener and backend-auth locality model and overlaps the open #50751 alternative, so maintainer policy selection is still needed before merge.

## What This Changes

This PR changes `resolveGatewayListenHosts()` and its unit tests so specific non-LAN bind hosts also try to listen on `127.0.0.1` for internal loopback calls.

## Best Possible Solution

Make one gateway locality model canonical, then land either this server-side dual-bind approach with docs, changelog, and focused runtime coverage, or close it in favor of the selected alternative.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `gateway.bind` to `tailnet` or a custom IP, current main listens only there, and local gateway calls still resolve to `ws(s)://127.0.0.1:<port>`, so same-host internal calls have no listener to reach. I did not run live tests because the review is read-only.

## Solution Assessment

Unclear. The PR is a narrow server-side fix and avoids #50751's plaintext non-loopback target problem, but it should not merge until maintainers explicitly accept the new loopback listener policy and update operator-facing docs.
