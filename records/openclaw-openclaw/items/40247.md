## Summary

Keep this PR open because current main still has the reported tailnet/custom listener mismatch, but the proposed branch is not merge-ready: it is stale against the merged Windows loopback guard, lacks inspectable real behavior proof, and needs Gateway locality/auth policy alignment.

## What This Changes

The PR changes `resolveGatewayListenHosts()` and its tests so specific tailnet/custom bind hosts also attempt a `127.0.0.1` listener for internal Gateway calls.

## Best Possible Solution

Choose the approved Gateway local-call model; if listener-side dual-bind is accepted, rebase this PR over current main with the Windows guard preserved, docs/tests updated, and redacted live tailnet/custom proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main listens only on a specific non-loopback bind host while local Gateway connection details still target `127.0.0.1`; I did not run a live tailnet setup in this read-only review.

## Solution Assessment

Unclear. Server-side dual-bind is plausible because it keeps implicit local clients on loopback, but this branch must preserve the Windows guard and be approved/documented against Gateway locality and auth policy before merge.
