## Summary

Keep open: current main still has the source-visible tailnet/custom bind breakage, but this PR is not merge-ready because it predates the Windows loopback guard, changes Gateway listener locality without docs/policy alignment, and lacks inspectable real behavior proof.

## What This Changes

The PR changes `resolveGatewayListenHosts()` so specific tailnet/custom bind hosts also attempt a `127.0.0.1` listener for internal Gateway calls.

## Best Possible Solution

Choose and document the approved Gateway local-call model; if listener-side dual-bind is accepted, rebase this PR over current main with the Windows guard preserved, docs updated, focused resolver/call coverage, and redacted live tailnet/custom proof.

## Reproduction Assessment

Yes, by source inspection: current main listens only on the resolved tailnet/custom bind host while implicit local Gateway call paths still target `127.0.0.1`. I did not run a live tailnet setup in this read-only review.

## Solution Assessment

Unclear. Server-side dual-bind is a plausible narrow fix that avoids plaintext non-loopback client targeting, but it must preserve the Windows guard, update listener/auth docs, and be reconciled with https://github.com/openclaw/openclaw/pull/50751.
