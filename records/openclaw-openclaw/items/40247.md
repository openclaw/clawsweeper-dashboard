## Summary

Keep open: current main still has the source-reproducible tailnet/custom bind mismatch, and this PR carries useful Gateway listener work, but the branch is not merge-ready because it predates the Windows loopback guard, changes documented listener policy, and lacks inspectable real behavior proof.

## What This Changes

The PR changes `resolveGatewayListenHosts` and resolver tests so tailnet/custom specific-IP binds also attempt a `127.0.0.1` listener for internal Gateway calls.

## Best Possible Solution

Choose and document the approved Gateway local-call model; if server-side dual-bind is accepted, rebase this PR over current main with the Windows guard preserved, docs updated, focused resolver/call coverage, and redacted live tailnet/custom proof.

## Reproduction Assessment

Yes, by source inspection: current main listens only on the resolved tailnet/custom bind host while implicit local Gateway call paths still target `127.0.0.1`. I did not run a live tailnet setup in this read-only review.

## Solution Assessment

Unclear. Server-side dual-bind is a plausible narrow fix that avoids plaintext non-loopback client targeting, but it must preserve the Windows guard, update the listener/auth docs, and be reconciled with the overlapping client-side approach in https://github.com/openclaw/openclaw/pull/50751.
