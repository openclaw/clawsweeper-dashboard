## Summary

Keep open: current main still has the tailnet/custom bind mismatch at source level, but this branch is not ready to merge because it drops the merged Windows loopback guard, needs Gateway listener/auth documentation alignment, and lacks inspectable real behavior proof.

## What This Changes

The PR changes Gateway listen-host resolution and tests so specific non-loopback bind hosts also attempt to listen on `127.0.0.1` for local internal calls.

## Best Possible Solution

Choose the approved Gateway local-call model; if listener-side dual-bind is accepted, preserve the Windows guard, update docs and tests, and attach redacted live tailnet/custom proof.

## Reproduction Assessment

Yes, source-level: current main returns only the selected non-loopback listener host while implicit local Gateway callers still target `127.0.0.1`. I did not run a live tailnet/custom gateway in this read-only review.

## Solution Assessment

Unclear. Listener-side dual-bind is plausible because it keeps internal callers on loopback, but this branch is not the best form until it preserves the Windows guard, documents listener/auth semantics, and includes real setup proof.
