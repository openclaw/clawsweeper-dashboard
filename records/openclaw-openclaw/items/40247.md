## Label Justifications

- `P1`: The PR targets a broken gateway workflow where tailnet/custom binds can break CLI probes and session spawning for remote/headless users.
- `merge-risk: 🚨 compatibility`: The diff changes tailnet/custom bind semantics by adding a loopback listener where docs currently describe Tailscale-IP-only behavior.

- `merge-risk: 🚨 availability`: The stale branch drops the current Windows `::1` guard and could reintroduce a gateway HTTP hang if rebased incorrectly.

## Summary

Keep open: current main and the latest release still have the source-level tailnet/custom listener versus loopback local-caller mismatch, but this stale PR is not merge-ready because it predates the Windows loopback guard, lacks inspectable real behavior proof, and needs maintainer approval of the listener-side locality model.

## What This Changes

This PR changes `resolveGatewayListenHosts()` and its tests so specific non-loopback bind hosts also try to listen on `127.0.0.1` for internal gateway calls.

## Best Possible Solution

Adopt one approved Gateway local-call model; if listener-side dual-bind is accepted, rebase on current main, preserve the Windows guard, update docs/tests, reconcile the overlapping client-side PR, and require redacted runtime proof.

## Reproduction Assessment

Yes at source level. Current main local callers target `127.0.0.1` while non-loopback listeners return only the configured bind host; I did not run a live tailnet/custom gateway in this read-only review.

## Solution Assessment

No, not as-is. Listener-side dual-bind is plausible, but the mergeable solution must preserve Windows behavior, document and approve the listener-surface change, and reconcile the competing client-side approach.
