## Summary

Keep open: current main still binds only the resolved tailnet/custom address while implicit local gateway calls target loopback, and this PR’s server-side dual-bind approach matches the existing listener structure with no blocking review findings.

## What This Changes

The PR changes `resolveGatewayListenHosts()` so specific tailnet/custom bind hosts also attempt a `127.0.0.1` listener, with focused gateway net tests for those branches.

## Best Possible Solution

Land a refreshed server-side dual-bind fix after normal CI/rebase, preserving the existing auth and non-loopback exposure checks without broad client-side RPC target rewrites.

## Reproduction Assessment

Yes, from source inspection: non-loopback listen hosts do not include loopback, while implicit local gateway calls target `127.0.0.1`. I did not run a live tailnet repro in this read-only review.

## Solution Assessment

Yes. Adding a secondary loopback listener at the listen-host layer is narrower than changing each local call target, and current listener code already supports multiple hosts with non-fatal alias failures.
