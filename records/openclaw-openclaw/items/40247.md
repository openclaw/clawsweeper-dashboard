## Summary

## What This Changes

The PR updates `resolveGatewayListenHosts()` so specific non-loopback bind hosts also try to listen on `127.0.0.1` for internal gateway calls, with focused resolver tests.

## Best Possible Solution

Adopt one approved gateway local-call model; if listener-side dual-bind is accepted, rebase on current main, preserve the Windows guard, update gateway docs/tests, reconcile the overlapping client-side PR, and require redacted runtime proof.

## Reproduction Assessment

Yes, source-level: current main local callers target `127.0.0.1` while non-loopback listeners return only the configured bind host. I did not run a live tailnet/custom gateway in this read-only review.

## Solution Assessment

Unclear as-is. Listener-side dual-bind is plausible, but the mergeable solution must preserve Windows behavior, document and approve the listener-surface change, and reconcile the competing client-side approach.
