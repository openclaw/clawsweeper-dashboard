## Summary

Keep open. ## What This Changes

The PR changes `resolveGatewayListenHosts()` and tests so specific non-loopback Gateway bind hosts also try to listen on `127.0.0.1` for internal calls.

## Best Possible Solution

Adopt one canonical Gateway local-call model; if listener-side dual-bind is accepted, preserve the Windows guard, document the new listener policy, reconcile the overlapping client-side PR, and require redacted runtime proof.

## Reproduction Assessment

Yes, at source level: current `main` constructs implicit local Gateway URLs with `127.0.0.1` while non-loopback listeners return only the configured bind host. I did not run a live tailnet/custom Gateway in this read-only review.

## Solution Assessment

No, not as-is. Listener-side dual-bind is plausible, but the mergeable solution must preserve Windows behavior, document and approve the listener-surface change, and reconcile the competing client-side approach.
