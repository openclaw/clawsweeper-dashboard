## Summary

Keep this PR open. Current main still single-homes non-loopback gateway bind hosts while local gateway calls resolve to loopback, so the reported tailnet/custom-bind failure is still real and this PR is not obsolete. The patch is a plausible narrow server-side fix, but it changes the documented gateway locality/auth model and overlaps the alternate open client-side approach in #50751, so it needs maintainer selection before merge.

## What This Changes

This PR changes `resolveGatewayListenHosts()` to add `127.0.0.1` as a secondary listener for specific non-LAN bind hosts and adds resolver unit cases for tailnet/custom dual-bind behavior.

## Best Possible Solution

Choose one coherent gateway locality model and make it canonical: either land this server-side dual-bind approach with docs, changelog, and focused runtime coverage, or land a client-side target-resolution approach like #50751 while preserving the documented listener semantics. The chosen fix should cover the linked tailnet-bind reports through one merged path.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: configure `gateway.bind` to `tailnet` or a specific custom IP, observe current main listening only on that bind host, then run a local gateway operation whose connection details resolve to `ws(s)://127.0.0.1:<port>`. I did not run live tests because this review was required to stay read-only.

## Solution Assessment

Unclear. The PR is a narrow server-side fix for the observed connection-refused failure, but maintainers should explicitly choose it over the #50751 client-side model because it changes documented tailnet/custom listener locality and the loopback backend-auth path.

## Risks / Open Questions

- The PR changes the documented locality model: direct `gateway.bind=tailnet` currently says loopback will not work, while this patch intentionally makes loopback work for tailnet/custom binds.
- The direct-loopback backend shared-auth path can omit device identity for `gateway-client` backend calls, so the dual-bind behavior should be explicitly approved and documented with that auth policy.
- The PR has resolver unit coverage only; it does not add docs, changelog, or an end-to-end check for `sessions_spawn` or CLI management calls through a started tailnet/custom-bound gateway.
- Open #50751 proposes a different client-target model for the same family of failures, so merging both directions without a maintainer choice could leave inconsistent gateway locality semantics.
