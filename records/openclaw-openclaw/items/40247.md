## Summary

Keep this PR open because it targets a real tailnet/custom bind breakage, but it is not merge-ready: the branch conflicts with current main, drops the merged Windows loopback guard, lacks inspectable real behavior proof, and changes documented gateway listener locality.

The PR changes gateway listen-host resolution so specific non-loopback bind hosts also try to listen on 127.0.0.1, with focused tests for tailnet/custom IP host resolution.

## Best Possible Solution

Adopt one approved gateway local-call model, likely listener-side dual-bind if accepted, while preserving the Windows guard, documenting tailnet/custom locality semantics, reconciling the overlapping PR, and requiring redacted runtime proof.

## Reproduction Assessment

Yes, source-level: current main local callers target 127.0.0.1 while non-loopback listeners return only the configured bind host. I did not run a live tailnet/custom gateway in this read-only review.

## Solution Assessment

Unclear as-is. Listener-side dual-bind is plausible and aligns with current local caller tests, but the mergeable solution must preserve Windows behavior, document/approve the listener-surface change, and reconcile the competing client-side approach.
