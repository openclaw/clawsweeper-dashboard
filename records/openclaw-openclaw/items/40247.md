## Summary

Keep open. The reported tailnet/custom bind breakage is source-reproducible on current main, but this external PR still needs a rebase that preserves the current Windows loopback guard, docs/auth-locality agreement, and inspectable real behavior proof before merge.

## What This Changes

The PR changes gateway listen-host resolution so specific tailnet/custom bind hosts also add 127.0.0.1 for internal gateway calls, with focused resolver tests.

## Best Possible Solution

If server-side dual-bind is the approved locality model, rebase this patch over current main with the Windows guard preserved, docs updated, focused tests, and redacted live tailnet/custom proof; otherwise route the fix through the chosen client/service-identity path.

## Reproduction Assessment

Yes, by source inspection: current main listens only on the resolved tailnet/custom bind host while implicit local call paths still target `127.0.0.1`. I did not run a live tailnet setup in this read-only review.

## Solution Assessment

Unclear. Server-side dual-bind is a plausible narrow fix and avoids plaintext non-loopback client targeting, but this branch must preserve the Windows guard, update the accepted listener/auth docs, and align with the open service-identity discussion before it is the best merge path.

  - body: Current main deliberately skips `::1` probing on Windows because the dual loopback bind can hang Gateway HTTP requests. This branch probes `::1` unconditionally in loopback mode, so the rebase needs to keep the `process.platform === "win32"` guard.
