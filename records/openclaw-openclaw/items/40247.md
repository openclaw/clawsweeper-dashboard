## Summary

Keep open: current main still has the tailnet/custom bind versus loopback internal-call mismatch, and this PR is a plausible server-side fix, but it predates current gateway listener changes, needs docs/security-policy alignment, and lacks inspectable real behavior proof.

## What This Changes

The branch changes `resolveGatewayListenHosts()` and its resolver tests so non-loopback specific gateway bind hosts also try `127.0.0.1`.

## Best Possible Solution

Adopt one canonical same-host gateway locality model; if dual-bind is accepted, rebase it onto current main with the Windows guard preserved, docs updated, targeted tests, and live tailnet/custom runtime proof.

## Reproduction Assessment

Yes, by source inspection: tailnet/custom bind resolves to a non-loopback listener while implicit local gateway clients still connect to `127.0.0.1`. I did not run a live tailnet gateway in this read-only review.

## Solution Assessment

Unclear. Server-side dual-bind is narrower than client-side plaintext non-loopback targeting, but it changes listener locality and needs maintainer approval plus docs/security alignment.
