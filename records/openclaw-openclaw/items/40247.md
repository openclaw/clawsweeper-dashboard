## Summary

Keep this PR open: it addresses a real current-main gateway bug, but it is not ready to merge because the branch is stale/conflicting, needs a maintainer decision on the dual-bind locality model, and lacks contributor-provided real behavior proof.

## What This Changes

The PR changes gateway listener host resolution so tailnet/custom IP binds also add `127.0.0.1` when bindable, with resolver unit cases for tailnet/custom hosts.

## Best Possible Solution

Adopt one canonical same-host gateway locality model; if dual-bind is accepted, rebase it onto current main with the Windows guard preserved, docs updated, and real tailnet/custom runtime proof.

## Reproduction Assessment

Yes, by source inspection: tailnet/custom bind resolves to a non-loopback listener, while implicit local gateway clients still connect to `127.0.0.1`. I did not run a live tailnet gateway in this read-only review.

## Solution Assessment

Unclear. The server-side dual-bind approach is narrow and avoids plaintext non-loopback client targeting, but it needs maintainer approval for the locality policy plus a current-main rebase that preserves the Windows guard.
