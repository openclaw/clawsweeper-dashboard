## Summary

Keep open: current main still shows the tailnet/custom loopback reachability problem, but this branch is not merge-ready because it would drop the current Windows loopback guard, changes Gateway listener locality without docs/auth policy alignment, lacks inspectable real setup proof, and overlaps another open Gateway PR.

## What This Changes

The PR changes `resolveGatewayListenHosts()` and its tests so specific tailnet/custom bind hosts also attempt a `127.0.0.1` listener for internal Gateway calls.

## Best Possible Solution

Choose the approved Gateway local-call model; if listener-side dual-bind is accepted, rebase this PR over current main with the Windows guard preserved, docs/tests updated, and redacted live tailnet/custom proof.

## Reproduction Assessment

Yes. Source inspection is high confidence: current main listens only on the resolved tailnet/custom bind host while implicit local Gateway call paths still target `127.0.0.1`; I did not run a live tailnet setup in this read-only review.

## Solution Assessment

Unclear. Server-side dual-bind is plausible because it avoids implicit plaintext non-loopback client URLs, but this branch must preserve the Windows guard, document the listener/auth policy, and be reconciled with https://github.com/openclaw/openclaw/pull/50751.
