## Summary

Keep this PR open. Current main still has the reported tailnet/custom-bind mismatch, and the PR is a plausible narrow server-side fix, but it changes gateway locality semantics and overlaps with the open client-side approach in #50751, so it needs maintainer architecture/security-policy review before merge rather than cleanup closure.

## What This Changes

This PR changes `resolveGatewayListenHosts()` to add `127.0.0.1` as a secondary listener for specific non-LAN bind hosts and adds net unit cases for tailnet/custom dual-bind behavior.

## Best Possible Solution

Pick one coherent model for local operations under tailnet/custom binds: either land the server dual-bind approach with docs, changelog, and end-to-end coverage for `sessions_spawn` plus CLI management calls, or land the client-side target-resolution approach in #50751 and keep listener semantics as documented. The chosen implementation should close the linked tailnet-bind reports through one canonical path.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: configure `gateway.bind` to tailnet or a specific custom IP, observe current main listening only on that bind host, then call a local operation whose connection details resolve to `ws(s)://127.0.0.1:<port>`, which has no listener in that mode.

## Solution Assessment

Unclear. This PR is a narrow server-side fix for the observable failure, but maintainers should explicitly choose it over the #50751 client-side model because it changes documented locality and the direct-loopback backend auth path.
