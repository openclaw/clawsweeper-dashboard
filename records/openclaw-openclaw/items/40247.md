## Summary

Keep open: current main still has the loopback-client versus non-loopback-listener mismatch, and this PR is the canonical open server-side fix path, but it needs a current-main rebase, docs/policy alignment, and inspectable real behavior proof before merge.

## What This Changes

The PR changes `resolveGatewayListenHosts()` so specific non-loopback gateway binds also add `127.0.0.1` when bindable, with resolver tests for tailnet/custom hosts.

## Best Possible Solution

Rebase the server-side dual-bind fix onto current main, preserve the Windows guard, update gateway/security docs, and require redacted live tailnet/custom proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main starts listeners from `resolveGatewayListenHosts(bindHost)`, non-loopback hosts return only themselves, and implicit local clients still connect to `127.0.0.1`. I did not run a live tailnet gateway in this read-only review.

## Solution Assessment

Unclear. The server-side dual-bind approach is a plausible narrow fix that avoids plaintext non-loopback client targeting, but it needs maintainer approval for the gateway locality model plus docs, the Windows guard, and live proof.
