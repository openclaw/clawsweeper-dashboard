## Summary

Keep this PR open. Current main and the latest release still keep non-loopback gateway binds single-homed while local gateway calls target loopback, so the proposed dual-bind fix is not obsolete; the remaining blocker is maintainer choice between this server-side model and the open client-side host-resolution approach in #50751, plus docs/security alignment.

## What This Changes

This PR changes `resolveGatewayListenHosts()` to add `127.0.0.1` as a secondary listener for specific non-LAN bind hosts and expands gateway net tests for tailnet/custom dual-bind cases.

## Best Possible Solution

Choose one coherent model for tailnet/custom internal calls, then land it with regression coverage for `sessions_spawn` and CLI management calls, update the Tailscale/security/protocol docs to match the chosen locality model, and close the linked tailnet-bind reports through the merged implementation.

## Reproduction Assessment

Yes. The source-level path is clear: configure a tailnet/custom bind so the gateway listens only on that specific host, then use a local operation whose connection details resolve to `ws(s)://127.0.0.1:<port>`; current main has no loopback listener for that case.

## Solution Assessment

Unclear. This PR is a narrow and plausible server-side fix, but #50751 proposes a client-side target-resolution model for overlapping symptoms, so maintainers should choose the preferred architecture before landing either path.
