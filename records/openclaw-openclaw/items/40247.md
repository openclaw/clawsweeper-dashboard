## Summary

Keep open: current main still has the reported loopback/tailnet listener mismatch at source level, and this PR targets the right Gateway surface, but the branch is not mergeable as-is because it predates the Windows loopback guard, lacks inspectable real behavior proof, and changes Gateway locality policy without docs/security alignment.

## What This Changes

The PR changes `resolveGatewayListenHosts()` and its tests so specific non-loopback bind hosts also try to listen on `127.0.0.1` for internal Gateway calls.

## Best Possible Solution

Choose the approved Gateway local-call model; if listener-side dual-bind is accepted, rebase this PR over current main with the Windows guard preserved, docs/tests updated, and redacted live tailnet/custom proof attached.

## Reproduction Assessment

Yes, source-level: current main still listens only on the specific non-loopback bind host while implicit local Gateway callers and tests target `127.0.0.1`. I did not run a live tailnet setup in this read-only review.

## Solution Assessment

Unclear. Listener-side dual-bind is plausible and narrower than changing every client callsite, but this branch must preserve the Windows guard and be approved/documented against Gateway locality and auth policy before merge.
