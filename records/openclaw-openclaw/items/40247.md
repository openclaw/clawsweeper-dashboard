## Summary

Keep this PR open: current main still has the reported loopback/self-call failure path for tailnet/custom binds, and this PR is a plausible narrow server-side fix, but it changes the documented gateway listener/locality model and needs maintainer policy acceptance plus docs/changelog alignment before merge.

## What This Changes

The PR changes `resolveGatewayListenHosts()` and its unit tests so specific non-LAN bind hosts also try to listen on `127.0.0.1` for internal loopback calls.

## Best Possible Solution

Make one gateway locality model canonical, then land the selected fix with docs, changelog, focused unit coverage, and a runtime smoke for tailnet/custom bind plus local internal calls.

## Reproduction Assessment

Yes. Source inspection shows current main listens only on the specific tailnet/custom bind host while local gateway calls still resolve to `ws(s)://127.0.0.1:<port>`, so same-host internal calls have no listener to reach.

## Solution Assessment

Unclear. The PR is a narrow server-side fix, but it changes listener locality and should be accepted only after maintainers choose it over the client-target alternative and align docs/security guidance.
