## Summary

Keep open: current main still has the reported tailnet/custom-bind loopback mismatch, and the PR is a plausible narrow fix, but it changes the gateway listener/locality model and overlaps with open alternative #50751, so it needs maintainer selection before merge.

## What This Changes

The PR changes `resolveGatewayListenHosts()` so specific non-LAN bind hosts also bind `127.0.0.1` when available, with unit coverage for tailnet and custom-IP cases.

## Best Possible Solution

Make one gateway locality model canonical, then land the chosen fix with docs, changelog, and focused runtime coverage for tailnet/custom bind plus local management calls.

## Reproduction Assessment

Yes, by source inspection rather than a live run: current main listens only on the resolved tailnet/custom host while local gateway clients still target `127.0.0.1`. That gives a high-confidence failure path for same-host internal calls under `gateway.bind=tailnet` or a specific custom IP.

## Solution Assessment

Unclear. The PR is a narrow server-side fix and avoids relaxing plaintext tailnet WebSocket targeting, but it should not be treated as the canonical solution until maintainers accept the dual-bind listener policy over #50751's client-side approach.
