## Summary

Keep open: the tailnet/custom-bind failure is source-reproducible and current main still does not implement the server-side loopback listener, but this PR needs a current-main rebase, gateway locality/docs alignment, and inspectable real behavior proof before merge.

## What This Changes

The PR changes `resolveGatewayListenHosts()` so tailnet/custom-specific bind hosts also add `127.0.0.1` when bindable, with resolver tests for the new cases.

## Best Possible Solution

Adopt one canonical same-host gateway locality model; if server-side dual-bind is accepted, rebase it onto current main with the Windows guard preserved, update gateway docs/auth-locality guidance, and require focused live tailnet/custom proof.

## Reproduction Assessment

Yes, by source inspection. Current main starts listeners only from `resolveGatewayListenHosts()`, that resolver returns only the tailnet/custom bind host, and implicit local clients still target `127.0.0.1`.

## Solution Assessment

Unclear. Server-side dual-bind is a plausible narrow fix and avoids plaintext non-loopback client targeting, but it needs maintainer approval for the locality policy plus a rebase that preserves the current Windows guard and updates docs.
