## Summary

This PR should stay open: source review confirms the current bug path, but the branch chooses a server-side dual-bind locality model that should be explicitly accepted against the existing tailnet/loopback docs and the open client-side alternative before merge.

## What This Changes

The PR changes `resolveGatewayListenHosts()` and its unit tests so specific non-LAN bind hosts also try to listen on `127.0.0.1` for internal loopback calls.

## Best Possible Solution

Make one gateway locality model canonical, then land this dual-bind fix with docs, changelog, and focused runtime proof if maintainers accept server-side loopback compatibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `gateway.bind` to `tailnet` or a custom IP, current main listens only there, and same-host gateway clients still resolve to `ws(s)://127.0.0.1:<port>`.

## Solution Assessment

Unclear. The PR is a narrow server-side fix that matches the current loopback client assumption, but it should be selected explicitly over the related client-target approach and aligned with docs/security wording.
