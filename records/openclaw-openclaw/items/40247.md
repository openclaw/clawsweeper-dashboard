## Summary

Keep open: current main still has a source-visible Gateway loopback/listener mismatch, and this PR is a plausible fix path, but it cannot merge as-is because it would drop current Windows loopback protection, changes listener locality policy without docs alignment, and lacks inspectable real-environment proof.

## What This Changes

The PR updates `src/gateway/net.ts` and `src/gateway/net.test.ts` so specific tailnet/custom bind hosts also attempt a `127.0.0.1` listener for internal Gateway calls.

## Best Possible Solution

Choose and document the approved Gateway local-call model; if listener-side dual-bind is accepted, preserve the Windows guard, update Gateway configuration/remote-access docs, keep focused resolver/call coverage, and require redacted live tailnet/custom proof.

## Reproduction Assessment

Yes, by source inspection: current main listens only on the resolved tailnet/custom bind host while implicit local Gateway call paths still target `127.0.0.1`. I did not run a live tailnet setup in this read-only review.

## Solution Assessment

Unclear. Server-side dual-bind is a plausible narrow fix that avoids plaintext non-loopback client targeting, but this branch must preserve the Windows guard, document the listener/auth model, and be reconciled with https://github.com/openclaw/openclaw/pull/50751.
