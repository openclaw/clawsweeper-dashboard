## Label Justifications

- `P2`: This is a normal-priority stability bug in a bounded memory-wiki bridge/status workflow with no data loss or core runtime outage.
- `impact:security`: The reported failure and any fix center on fs-safe path identity checks, which are part of OpenClaw's symlink/swap security boundary.

## Summary

Keep open: current main and the latest release still run imported-source sync before `wiki_status` and still convert non-symlink `FsSafeError` write failures, including `path-mismatch`, into a fatal status error. No related merged PR or canonical duplicate currently owns the remaining fix.

## What This Changes

Make memory-wiki `wiki_status` tolerate transient `path-mismatch` failures during concurrent bridge source page rewrites instead of aborting the whole status call.

## Best Possible Solution

Handle transient fs-safe `path-mismatch` during memory-wiki imported-source sync locally with a bounded retry and/or per-page warning, while leaving fs-safe guard semantics intact and adding focused regression coverage.

## Reproduction Assessment

No live high-confidence race reproduction was run. Source inspection shows a high-confidence failing path: `wiki_status` runs bridge sync, bridge sync awaits each source-page write, and a `path-mismatch` `FsSafeError` from that write is wrapped and allowed to reject the whole status call.

## Solution Assessment

Yes, the best fix should be local to memory-wiki source sync rather than changing fs-safe. Bounded retry plus per-page isolation is narrower than adding cross-process locks or making `wiki_status` fully read-only in this cleanup pass.
