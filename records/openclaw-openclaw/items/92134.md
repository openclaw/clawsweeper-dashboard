## Summary

Keep open: current main and v2026.6.6 still run imported-source sync before `wiki_status` returns, and the write path still lets transient fs-safe path failures abort the whole call. The direct fix PR is open but unmerged, so this issue should stay open until that PR or a replacement lands.

## What This Changes

The issue asks memory-wiki `wiki_status` to tolerate transient fs-safe path failures during concurrent bridge source-page re-exports instead of aborting the status call.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92375 or a replacement that handles transient fs-safe rename races without weakening symlink, hardlink, or path validation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level path: current `wiki_status` runs bridge source sync, and the imported-source writer has no retry when fs-safe reports a transient path failure during the write.

## Solution Assessment

Is this the best way to solve the issue? The best path is not a second repair lane; the linked PR or a narrow replacement should add bounded transient handling while preserving hard failures for real path-safety violations.
