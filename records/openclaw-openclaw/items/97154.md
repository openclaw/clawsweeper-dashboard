## Summary

Keep open. This is a maintainer-labeled Slack feature PR, current main does not implement the alternate API-root surface, and review still finds a security/compatibility blocker because the new `SLACK_API_URL` default can be loaded from workspace `.env` and redirect Slack bearer-token Web API calls.

## What This Changes

Threads a `slackApiUrl`/`SLACK_API_URL` alternate Web API root through Slack WebClient creation paths, including send, actions, probes, scopes, directory, allowlist, monitor, and write-client caching.

## Best Possible Solution

Keep the Slack SDK `slackApiUrl` threading, but make `SLACK_API_URL` a trusted operator/global override by blocking workspace `.env` injection and covering the upgrade boundary before merge.

## Reproduction Assessment

Yes. Source inspection shows PR head reads `process.env.SLACK_API_URL`, while current main's workspace dotenv filter lacks `_API_URL` and writes accepted workspace `.env` entries into `process.env`.

## Solution Assessment

No, not as submitted. The Slack SDK `slackApiUrl` option is the right low-level seam, but the process-env default must be protected from lower-trust workspace env sources before it is safe.
