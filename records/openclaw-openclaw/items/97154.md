- `P2`: This is an unmerged feature PR with a concrete security-boundary defect, not an active production outage.
- `merge-risk: 🚨 compatibility`: A newly honored generic env var can change Slack routing for existing environments that already define `SLACK_API_URL`.
- `merge-risk: 🚨 security-boundary`: The new env default can redirect Slack bearer-token API requests and is currently allowed from workspace `.env` files.

## Summary

Keep open: the Slack SDK seam is valid, but the patch adds an unprotected `SLACK_API_URL` process-env default that current workspace `.env` filtering can inherit, allowing Slack bearer-token requests to be redirected by workspace state.

## What This Changes

The PR threads an alternate Slack Web API root through Slack WebClient creation paths using explicit `slackApiUrl` options or `SLACK_API_URL`.

## Best Possible Solution

Keep the Slack SDK `slackApiUrl` threading, but treat `SLACK_API_URL` as a trusted operator endpoint override by blocking workspace `.env` injection, documenting the trusted sources, and adding focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows PR head reads `process.env.SLACK_API_URL`, while current workspace dotenv filtering writes allowed `.env` entries into `process.env` and does not block `_API_URL`.

## Solution Assessment

No, not as submitted. The Slack SDK `slackApiUrl` option is the right low-level seam, but the process-env default must be protected from lower-trust workspace env sources before it is safe.
