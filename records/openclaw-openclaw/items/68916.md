## Summary

Keep open. Current main has partial ACP task-maintenance cleanup for terminal or orphaned parent-owned ACP sessions, but the specific reset path still closes only the reset target session and does not enumerate child ACP sessions owned by that parent.

## What This Changes

The issue requests reliable cleanup for ACP `sessions_spawn` one-shot runs, especially closing child ACP sessions and their processes when the parent session is reset.

## Best Possible Solution

Add reset/delete-time enumeration of ACP session entries whose `spawnedBy` or `parentSessionKey` matches the reset parent, close each child through the existing ACP manager cleanup path, and cover parent-plus-child reset with a focused gateway regression test. Leave PID/process-tree fallback work at the ACPX/runtime boundary unless live proof shows logical close still leaks descendants.

## Reproduction Assessment

Yes for the core reset gap: seed a parent session plus an ACP child entry whose `spawnedBy` or `parentSessionKey` points to the parent, then reset the parent and observe that current main only closes the target key. The full process-table memory leak still needs live ACPX validation.

## Solution Assessment

Yes, the narrowest maintainable core fix is to reuse `listAcpSessionEntries()` and `closeAcpRuntimeForSession()` from `src/gateway/session-reset-service.ts` rather than adding process-kill policy to core reset logic. Process reaping should remain in the ACPX/runtime workstream.

## Repair Work Prompt

Fix the parent-reset ACP child cleanup gap for https://github.com/openclaw/openclaw/issues/68916. In current main, `cleanupSessionBeforeMutation()` in `src/gateway/session-reset-service.ts` only calls `closeAcpRuntimeForSession()` for the reset target. Add a narrow reset/delete cleanup step that enumerates ACP session entries via the existing `listAcpSessionEntries()` seam and closes entries whose `entry.spawnedBy` or `entry.parentSessionKey` matches the canonical reset parent key. Reuse `closeAcpRuntimeForSession()` and existing timeout/error behavior; do not add PID/process-table reaping or ACPX-specific process policy in core. Add a regression test in `src/gateway/server.sessions.reset-cleanup.test.ts` that seeds a parent plus at least one ACP child and verifies resetting the parent closes both target and owned child ACP sessions. Preserve existing direct ACP reset behavior and persistent reset metadata handling. No changelog entry is required unless the maintainer wants user-facing release notes for this fix.
