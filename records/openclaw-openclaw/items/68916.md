## Summary

Keep open. Current main still closes only the reset target ACP runtime, while spawned ACP child sessions can record the parent in `spawnedBy` or `parentSessionKey`; related open PRs cover adjacent run-completion, pressure, or output surfaces rather than this exact parent-reset cleanup gap.

## What This Changes

The issue asks for ACP run/session cleanup so parent resets close child ACP session keys and prevent orphaned ACP backend processes.

## Best Possible Solution

Add reset/delete cleanup that enumerates ACP entries owned by the target parent and closes each child through the existing bounded ACP cancel/close helper, leaving completion cleanup, PID exposure, and startup reaping to their separate work streams.

## Reproduction Assessment

Yes, source-level. Seed a parent session and a child ACP entry whose `spawnedBy` or `parentSessionKey` points to that parent, invoke `sessions.reset` for the parent, and current main only closes the target ACP key.

## Solution Assessment

Yes. The narrowest maintainable fix is to enumerate parent-owned ACP child entries during reset/delete and close them with the existing bounded ACP manager cleanup path; broader PID fallback and startup orphan reaping should remain separate.
