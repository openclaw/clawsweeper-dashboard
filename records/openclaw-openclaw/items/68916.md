## Summary

Keep open. Current main still closes only the reset target ACP session during `/new` or `/reset`; it has adjacent ACP task-maintenance and acpx process cleanup, but not synchronous child ACP session cleanup for the parent reset path.

## What This Changes

The issue asks for ACP child sessions spawned from a parent to be closed when that parent session is reset, preventing orphaned ACP processes from accumulating.

## Best Possible Solution

Add synchronous reset-path cleanup that enumerates child ACP session entries whose `spawnedBy` or `parentSessionKey` matches the reset parent and closes them through the existing ACP close/cancel helper, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Seed a parent session plus a child ACP entry whose `spawnedBy` or `parentSessionKey` points to that parent, then reset the parent; current main closes only the target ACP key and leaves the child row untouched.

## Solution Assessment

Yes. The narrow reset-path child ACP enumeration is the best immediate fix for this issue’s specific gap; completed-run cleanup, PID exposure, and startup process reaping should remain in their related work streams.
