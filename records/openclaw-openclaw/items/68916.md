## Summary

Keep open. Current main still closes only the reset target ACP runtime; it does not enumerate ACP child session entries whose `spawnedBy` or `parentSessionKey` points to the reset parent, and the related open PRs cover adjacent completion, pressure, and PID-observability work rather than this reset/delete cleanup gap.

## What This Changes

The issue requests ACP oneshot/run child sessions to be cleaned up when their parent session is reset, including child ACP session keys recorded through `spawnedBy` or `parentSessionKey`.

## Best Possible Solution

Add reset/delete cleanup that lists ACP entries owned by the reset parent and closes each through the existing bounded ACP cancel/close helper, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Seed a parent session plus a child ACP entry whose `spawnedBy` or `parentSessionKey` matches that parent, invoke `sessions.reset` for the parent, and current main closes only the target ACP key rather than the child row.

## Solution Assessment

Yes. The narrow maintainable fix is to enumerate parent-owned ACP child entries in the reset/delete cleanup path and reuse the existing ACP manager cancel/close helper; completed-run cleanup, PID exposure, and startup reaping should remain separate work streams.
