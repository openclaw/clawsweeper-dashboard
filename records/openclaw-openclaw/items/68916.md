## Label Justifications

- `P1`: The reported orphaned ACP process accumulation can exhaust memory and break ACP-backed agent workflows during normal use.
- `impact:crash-loop`: The issue is about orphaned ACP backend processes accumulating until the host can OOM or become unavailable.
- `impact:session-state`: The reset/delete lifecycle leaves child ACP session state associated with the old parent lifecycle instead of closing it.

## Summary

Keep open. Current main and the latest release still close only the target ACP session during reset/delete, while parent-owned child ACP sessions are persisted with lineage metadata that the reset path does not enumerate.

## What This Changes

The issue requests cleanup of ACP `mode: "run"` child sessions and processes when their parent session is reset or deleted.

## Best Possible Solution

Add reset/delete cleanup that enumerates ACP entries owned by the mutated parent and closes each child through the existing bounded ACP cancel/close helper, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Seed a parent session and an ACP child entry whose `spawnedBy` or `parentSessionKey` matches that parent, then call `sessions.reset` or `sessions.delete`; current main only closes the target key.

## Solution Assessment

Yes. The narrow maintainable fix is to enumerate parent-owned ACP rows during reset/delete cleanup and reuse the existing close helper; broader orphan reapers and PID observability should remain separate work.
