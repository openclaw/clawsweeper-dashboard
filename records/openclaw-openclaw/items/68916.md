## Summary

Keep open. Current main still closes only the reset/delete target ACP runtime and does not enumerate ACP child sessions whose `spawnedBy` or `parentSessionKey` points at the parent, so the specific reset cleanup leak remains valid.

## What This Changes

The issue requests cleanup of ACP `mode: "run"` child sessions when a parent session is reset, plus stronger orphan-process cleanup around failed ACP close paths.

## Best Possible Solution

Add reset/delete cleanup that enumerates ACP entries owned by the reset parent and closes each through the existing bounded ACP cancel/close helper, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Seed a parent session plus a child ACP entry whose `spawnedBy` or `parentSessionKey` points to that parent, invoke `sessions.reset` for the parent, and current main closes only the target ACP key.

## Solution Assessment

Yes. The narrow maintainable fix is to enumerate parent-owned ACP rows in the reset/delete cleanup path; completion cleanup, PID observability, and startup reaping should remain in their related work streams.
