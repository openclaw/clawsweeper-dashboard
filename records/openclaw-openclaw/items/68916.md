## Summary

Keep open. Current main has adjacent ACP task-maintenance and acpx process cleanup, but the parent reset path still closes only the reset target ACP key and does not enumerate child ACP sessions owned by that parent.

## What This Changes

The issue asks OpenClaw to terminate ACP `mode: "run"` child processes after completion and close child ACP sessions when the parent session is reset.

## Best Possible Solution

Add reset-path cleanup that finds child ACP session entries whose `spawnedBy` or `parentSessionKey` matches the reset parent and closes them through the existing ACP cancel/close helper, with focused regression coverage.

## Reproduction Assessment

Yes, for the source-level reset gap: seed a parent session plus a child ACP entry whose `spawnedBy` or `parentSessionKey` points to that parent, then reset the parent. Current main only closes the reset target ACP key; I did not live-run the ACP/acpx process leak symptom.

## Solution Assessment

Yes. The narrow reset-path child ACP enumeration is the best immediate fix for this issue’s specific additional gap; PID-level fallback, completed-run cleanup, startup reaping, and liveness contracts should stay in their related work streams.
