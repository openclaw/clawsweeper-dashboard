## Label Justifications

- `P1`: The reported ACP child process accumulation can exhaust memory and break real ACP-backed agent workflows.
- `impact:session-state`: The reset/delete lifecycle leaves parent-owned ACP child session state outside the parent cleanup boundary.
- `impact:crash-loop`: The orphaned process trees described by the issue can OOM or otherwise make the gateway host unavailable.

## Summary

Keep open. Current main still closes only the reset/delete target ACP session key, while ACP child sessions can be stored with parent lineage metadata that this cleanup path never enumerates.

## What This Changes

The issue asks for ACP parent session reset/delete cleanup to terminate child `sessions_spawn({ runtime: "acp", mode: "run" })` sessions and their process trees.

## Best Possible Solution

Add a bounded reset/delete cleanup step that enumerates ACP entries owned by the parent and closes each child through the existing ACP cancel/close helper, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Seed a parent session plus an ACP child entry whose `spawnedBy` or `parentSessionKey` points to that parent, then invoke reset/delete for the parent; current main only calls ACP close for the target key.

## Solution Assessment

Yes. The narrow maintainable fix is to enumerate parent-owned ACP session rows during reset/delete cleanup and reuse the existing close helper; PID exposure and startup process reaping should remain separate work.
