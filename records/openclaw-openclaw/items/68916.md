## Summary

Keep open. Current main still cleans up only the reset target ACP session during `/new` or `/reset`; child ACP session rows that point back to the reset parent are not enumerated or closed, while related open PRs cover adjacent completion, reaping, PID, and subagent-stop surfaces rather than this exact reset gap.

## What This Changes

The issue requests that parent session reset/delete cleanup also close ACP child sessions spawned from that parent so ACP adapter and child-agent processes do not remain orphaned.

## Best Possible Solution

Add synchronous reset/delete cleanup that enumerates child ACP entries whose spawnedBy or parentSessionKey matches the reset parent and closes them through the existing bounded ACP cancel/close helper, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Seed a parent session plus a child ACP entry whose spawnedBy or parentSessionKey points to that parent, invoke sessions.reset for the parent, and current main closes only the target ACP key rather than the child row.

## Solution Assessment

Yes. The narrow reset-path child ACP enumeration is the best immediate fix for this issue’s specific gap; completed-run cleanup, PID exposure, and startup process reaping should remain in their related work streams.
