## Label Justifications

- `P1`: The issue describes orphaned ACP process trees causing progressive memory exhaustion and a broken spawned-agent cleanup workflow for real users.
- `impact:crash-loop`: Accumulated orphan ACP processes can exhaust memory and cause OOM or severe process-level availability failure.
- `impact:session-state`: Parent reset leaves child ACP runtime/session state associated with the old parent instead of cleaning it up.

## Summary

Keep open. Current main still closes only the reset/delete target ACP runtime and does not enumerate ACP child session entries whose `spawnedBy` or `parentSessionKey` points at the parent, so the reported reset cleanup leak remains source-reproducible.

## What This Changes

The issue requests that parent session reset/delete clean up ACP child session keys spawned from the parent so ACP `mode: "run"` processes do not remain orphaned.

## Best Possible Solution

Add reset/delete cleanup that enumerates ACP entries owned by the target parent and closes each child through the existing bounded ACP cancel/close helper, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Seed a parent session plus a child ACP entry whose `spawnedBy` or `parentSessionKey` points at that parent, invoke parent `sessions.reset` or delete cleanup, and current main only closes the target ACP key.

## Solution Assessment

Yes. The narrow maintainable fix is to enumerate parent-owned ACP rows in the reset/delete cleanup path and close them through the existing ACP manager helper, leaving completion cleanup, PID exposure, and startup reaping to their separate work streams.
