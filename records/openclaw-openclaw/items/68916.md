## Label Justifications

- `impact:session-state`: The bug is about parent session lifecycle cleanup failing to include child ACP session state.
- `impact:crash-loop`: The reported orphaned process accumulation can exhaust memory and make the gateway host unavailable.

## Summary

Keep open. Current main still cleans up only the reset/delete target ACP session key, while ACP child sessions are created with parent lineage metadata that this path does not enumerate; related open work covers adjacent runtime pressure/PID observability but not this parent-reset cleanup gap.

## What This Changes

The issue asks for ACP `mode: "run"` child sessions to be closed when their parent session is reset or deleted, with stronger fallback cleanup for orphaned ACP processes.

## Best Possible Solution

Add a bounded reset/delete cleanup step that lists ACP session entries owned by the parent and closes each child through the existing ACP cancel/close helper; keep PID exposure and broad startup reaping tracked separately unless needed for the narrow regression.

## Reproduction Assessment

Yes, source-level. Seed a parent session plus an ACP child row whose `spawnedBy` or `parentSessionKey` points to that parent, then reset/delete the parent; current main only closes the target ACP key.

## Solution Assessment

Yes for the narrow fix. The maintainable path is to enumerate parent-owned ACP session entries in reset/delete cleanup and reuse existing ACP close/cancel behavior, while leaving PID observability and global orphan reaping to narrower follow-ups.
