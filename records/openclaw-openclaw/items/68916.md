## Summary

Keep open. Current main has adjacent terminal/orphaned parent-owned ACP task maintenance and direct ACP-session reset cleanup, but the synchronous parent reset path still does not enumerate ACP child sessions owned by the parent; the related open PRs cover completion or process-reaper surfaces rather than this reset-specific gap.

## What This Changes

The issue requests cleanup for ACP `sessions_spawn(mode: "run")` children, parent-reset cleanup of child ACP session keys, and startup reaping so leaked ACP processes do not accumulate.

## Best Possible Solution

Add focused reset-path cleanup that lists ACP session entries owned by the resetting parent, closes each child through the existing ACP cancel/close helper, and covers it with a parent-plus-child regression test while leaving broad process reaping to the runtime/acpx work.

## Reproduction Assessment

Yes. Seed a normal parent session and an ACP child session whose `spawnedBy` or `parentSessionKey` matches the parent, call `sessions.reset` for the parent, and current main will not call ACP cancel/close for the child key.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse the existing ACP session listing and close helper during parent reset; PID fallback and startup orphan reaping are safer as runtime/acpx follow-ups if live validation still shows leaked descendants.
