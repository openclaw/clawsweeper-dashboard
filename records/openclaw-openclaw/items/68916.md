## Summary

Keep open. Current main has adjacent ACP task-maintenance cleanup for terminal or orphaned parent-owned sessions, but the parent reset path still closes only the target ACP session key and does not enumerate ACP child sessions owned by that parent, so the specific reset leak remains actionable.

## What This Changes

The issue asks OpenClaw to reliably clean up ACP `mode: "run"` child sessions and their runtime processes when the run completes or when the parent session is reset.

## Best Possible Solution

Add a focused reset-path cleanup in `src/gateway/session-reset-service.ts` that finds ACP session entries whose `spawnedBy` or `parentSessionKey` matches the reset parent and closes them through the existing ACP cancel/close helper, with regression coverage for parent reset closing child ACP keys. Keep PID/process-group reaping in the acpx/runtime boundary unless live validation shows the generic ACP close path still leaks descendants.

## Reproduction Assessment

Yes for the core reset gap: seed a parent session plus a child ACP session whose `spawnedBy` or `parentSessionKey` points at that parent, reset the parent, and current main will only call ACP cleanup for the target parent key. The live orphaned-process symptom still needs an ACP/acpx runtime run to validate process behavior after the store-level fix.

## Solution Assessment

Yes, the narrowest maintainable fix is to enumerate owned ACP child session entries during parent reset and reuse `closeAcpRuntimeForSession`; the current task-maintenance sweep is delayed and terminal/orphan-oriented, so it is not a substitute for reset cleanup.
