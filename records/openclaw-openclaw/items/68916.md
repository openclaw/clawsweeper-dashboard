## Summary

Keep open. Current main has adjacent ACP process-tree and task-maintenance cleanup, but the parent reset path still closes only the reset target ACP key and does not synchronously enumerate child ACP sessions owned by that parent.

## What This Changes

The issue asks for ACP run/oneshot cleanup to terminate child processes and for parent `/new` or `/reset` to close child ACP session keys.

## Best Possible Solution

Add reset-path cleanup that finds child ACP session entries whose `spawnedBy` or `parentSessionKey` matches the reset parent and closes them through the existing ACP cancel/close helper, with focused regression coverage.

## Reproduction Assessment

Yes for the source-level reset gap: seed a parent session plus a child ACP entry whose `spawnedBy` or `parentSessionKey` points to that parent, then reset the parent; current main only closes the reset target ACP key. I did not live-run acpx to validate the OS process leak symptom.

## Solution Assessment

Yes. The narrow reset fix is the best immediate solution for this issue’s specific additional gap; broader completed-run cleanup, PID exposure, and startup reaping are already separate related work.
