## Summary

Keep open. Current main still has the reset-specific ACP child cleanup gap: parent reset closes only the target ACP session key, while adjacent maintenance cleanup is delayed and does not make `/new` or `/reset` synchronously close child ACP sessions.

## What This Changes

The issue requests cleanup for ACP `mode: "run"` child sessions, especially closing child ACP session keys owned by a parent when that parent is reset.

## Best Possible Solution

Add reset-path cleanup that finds ACP session entries whose `spawnedBy` or `parentSessionKey` matches the reset parent and closes them through the existing ACP cancel/close helper, with regression coverage for parent reset closing child ACP keys.

## Reproduction Assessment

Yes for the source-level reset gap: seed a parent session plus a child ACP entry whose `spawnedBy` or `parentSessionKey` points to that parent, then reset the parent; current main only closes the reset target ACP key. I did not live-run acpx to validate the OS-level orphaned process symptom.

## Solution Assessment

Yes, the narrow reset fix is the best immediate solution for this issue’s specific additional gap. Broader PID/process-group reaping belongs in the ACP/acpx runtime boundary or the related process-reaper PRs, not in the parent reset patch.
