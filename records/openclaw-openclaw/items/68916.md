## Label Justifications

- `P1`: The reported orphaned ACP process accumulation can exhaust memory and break ACP-backed agent workflows for real users.
- `impact:session-state`: The reset/delete path leaves child ACP session state attached to the old parent lifecycle instead of closing it during parent mutation.
- `impact:crash-loop`: Accumulated orphaned ACP process trees can OOM or make the gateway host unavailable.

## Summary

Keep open. Current main still closes ACP runtime state for only the reset/delete target session key, while parent-owned ACP child entries are stored with lineage metadata that the cleanup path does not enumerate.

## What This Changes

The issue requests cleanup of ACP child sessions owned by a parent when that parent is reset or deleted so ACP process trees do not remain orphaned.

## Best Possible Solution

Add reset/delete cleanup that enumerates ACP rows owned by the reset parent and closes each child through the existing bounded ACP cancel/close helper, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Seed a parent session plus a child ACP entry whose spawnedBy or parentSessionKey points to that parent, invoke sessions.reset or sessions.delete for the parent, and current main only closes the target ACP key.

## Solution Assessment

Yes. The narrow maintainable fix is to enumerate parent-owned ACP session rows in the reset/delete cleanup path and reuse the existing close helper; broader process reaping and PID exposure should remain separate work.
