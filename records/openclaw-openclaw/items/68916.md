## Label Justifications

- `P1`: The reported leak can progressively exhaust memory and break ACP-backed agent workflows on real user machines.
- `impact:crash-loop`: The issue is about orphaned ACP backend processes accumulating until the host can OOM or become unavailable.
- `impact:session-state`: The reset path leaves child ACP session state associated with the old parent lifecycle instead of closing it during parent mutation.

## Summary

Keep open. Current main still performs ACP reset/delete cleanup for only the target session key, while the reported parent-owned child ACP sessions remain discoverable only through metadata that the reset path does not enumerate; adjacent ACP cleanup PRs address manager pressure, process reaping, or PID observability but not this reset ownership gap.

## What This Changes

The issue requests ACP reset/delete cleanup that closes child ACP sessions created from a parent session so their backend processes are not orphaned after `/new` or `/reset`.

## Best Possible Solution

Add reset/delete cleanup that enumerates ACP entries owned by the reset parent and closes each child through the existing bounded ACP cancel/close helper, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. Seed a parent session plus a child ACP entry whose `spawnedBy` or `parentSessionKey` points to that parent, invoke `sessions.reset` or `sessions.delete` for the parent, and current main only calls ACP close for the target key.

## Solution Assessment

Yes. The narrow maintainable fix is to enumerate parent-owned ACP rows in the reset/delete cleanup path and reuse the existing close helper; broader process reaping and PID observability should remain separate work.
