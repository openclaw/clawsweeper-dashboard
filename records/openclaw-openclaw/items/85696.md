## Summary

Keep open: the PR addresses a real Moonshot embedded-runner latency path, but the current patch is not merge-ready because it makes skip-discovery resolution static-first across providers and skips provider dynamic preparation/runtime-preferred metadata for exact static rows.

## What This Changes

The PR enables bundled static catalog fallback from `runEmbeddedAgent` and changes `resolveModelAsync` to try static catalog rows before provider dynamic hooks during skip-agent-discovery resolution.

## Best Possible Solution

Rebase onto current main, keep provider dynamic/runtime-preferred resolution first, and add a narrow static fallback path for Moonshot/static-only rows with regression coverage for both Moonshot and runtime-preferred providers.

## Reproduction Assessment

Yes for the PR-introduced regression: the diff itself shows a static catalog row can return before `prepareProviderDynamicModel`, and current tests/source show runtime-preferred dynamic metadata is an existing invariant. I did not run the original Moonshot Discord stall live in this read-only review.

## Solution Assessment

No: static fallback is a plausible mitigation for the Moonshot stall, but making skip-discovery static-first across providers is broader than the safest fix. The better path is a rebased dynamic-first resolver with a narrow static fallback for the reported Moonshot/static-catalog gap.
