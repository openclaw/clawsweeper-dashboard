## Summary

Keep open: the PR targets a real embedded-runner slowdown and the static-catalog-first direction is useful, but the current diff is not merge-ready because skip-PI static hits bypass provider model normalization and can regress existing provider compat behavior.

## What This Changes

The PR changes embedded PI model resolution to try bundled static catalog rows before dynamic provider preparation when `skipPiDiscovery` is set, passes that opt-in from `runEmbeddedPiAgent`, and updates focused tests.

## Best Possible Solution

Keep the static catalog lookup before dynamic model preparation, but preserve lightweight target-provider model normalization or move required compat metadata into static manifests with focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live Discord/Moonshot run in this read-only pass, but the source path is clear: `runEmbeddedPiAgent` uses `skipPiDiscovery`, and the PR resolves static rows before dynamic prep while disabling provider normalization.

## Solution Assessment

Is this the best way to solve the issue? No. The ordering change is the right shape, but using `STATIC_PROVIDER_RUNTIME_HOOKS` for returned rows drops established provider compat normalization; a lightweight target-provider normalization path is safer.
