## Summary

Keep open: current main and the latest release still have the linked Moonshot model-resolution problem, but this PR is not merge-ready because it moves opt-in static catalog rows ahead of provider dynamic hooks and still lacks after-fix embedded dispatch timing proof.

## What This Changes

The PR lets the embedded agent runner opt into bundled static catalog fallback during skip-discovery model resolution and changes the resolver to try static catalog rows before provider dynamic hooks.

## Best Possible Solution

Keep provider dynamic hooks first, enable a targeted bundled-static fallback before broad `models.json` discovery for the Moonshot/native fast path, and require before/after embedded dispatch timing proof.

## Reproduction Assessment

No high-confidence live reproduction was run here. The linked issue has concrete shipped-release timing logs, and current source still shows native Moonshot can miss the skip-discovery static path and fall into broad model discovery.

## Solution Assessment

No as written. The static-catalog fallback direction is plausible, but static-first resolution is broader than needed; the safer fix keeps provider dynamic hooks first and uses static fallback only before broad discovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep provider dynamic hooks ahead of static rows:** `src/agents/embedded-agent-runner/model.ts:1659-1660`
  - body: This returns an opt-in static catalog model before `resolveDynamicAttempt()`. Because the runner now opts all embedded runs into static fallback, providers with static rows and dynamic preparation can skip their provider-owned capability load; for example, OpenRouter's `prepareDynamicModel` loads per-model capabilities before `resolveDynamicModel`, and current tests assert that call for `openrouter/auto`. Keep the static fallback after dynamic resolution, or narrow static-first to providers without dynamic hooks and add regression coverage.
