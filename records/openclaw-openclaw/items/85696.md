## Summary

Keep open: the PR targets the right Moonshot model-resolution latency path and supplies useful terminal proof, but the new static fast path bypasses target provider normalization hooks and can drop provider-owned compat metadata for other static-catalog providers.

## What This Changes

The PR lets the embedded PI runner resolve bundled static catalog models during skip-PI discovery before calling provider dynamic hooks, and updates focused embedded-runner tests.

## Best Possible Solution

Resolve bundled static catalog rows before broad PI discovery, but still run target-provider normalization/compat hooks without calling dynamic prepare/run hooks.

## Reproduction Assessment

Source-level yes, live no. Current main’s embedded runner can fall from skip-PI resolution into broad models.json preparation, matching the linked issue logs, and the PR body shows after-fix isolated model resolution.

## Solution Assessment

Not yet. Static catalog-before-discovery is the right direction, but the patch should preserve target provider normalization while skipping dynamic discovery work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve provider normalization for static fast paths:** `src/agents/pi-embedded-runner/model.ts:1215-1217`
  - body: The new skip-PI static-catalog path normalizes rows with `STATIC_PROVIDER_RUNTIME_HOOKS`, so target provider `normalizeResolvedModel` hooks no longer run before returning. Static providers such as Mistral rely on that hook to add compat like `supportsReasoningEffort` and `maxTokensField`; an embedded PI run for `mistral/mistral-medium-3-5` can now skip broad discovery but lose metadata that current main applies through the target provider hook. Keep dynamic prepare/run out of the fast path, but still run target-provider normalization and cover a compat case.
