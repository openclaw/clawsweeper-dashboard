## Summary

Current main still exposes the reported data-retention bug: the maintenance paths can delete idle topic/thread session pointers, and the related items are adjacent rather than an exact merged replacement.

## What This Changes

The issue asks session maintenance to stop pruning, capping, or disk-evicting durable Telegram topic and generic thread session entries.

## Best Possible Solution

Add a shared session-maintenance predicate for durable external conversation sessions and apply it consistently across save-time prune/cap, load-time prune/cap, disk-budget entry deletion, and cleanup preview/reporting. Use existing typed session/thread parsing where possible instead of raw substring checks, and keep per-pattern retention rules in the separate #50900 design track.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a store with an older Telegram topic or generic thread key plus enough newer ephemeral entries, age, count pressure, or disk pressure; current helpers will delete the old non-active key. I did not run mutating tests because this review is read-only.

## Solution Assessment

Yes. The best narrow fix is a shared helper used by every maintenance path, with tests for topic/thread/group protection and for non-protection of ephemeral maintenance candidates. Ad hoc `:topic:`/`:thread:` substring checks would be riskier because plugin-owned session grammars already need typed parsing.
