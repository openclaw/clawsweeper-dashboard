## Summary

Keep open. The PR targets a real hot-path performance problem, but the current guard is too broad: it reuses any active registry before checking the requested workspace/runtime compatibility, and it also misses the required user-facing changelog entry.

## What This Changes

The PR adds an active-registry early return to `ensureRuntimePluginsLoaded` and updates the focused unit test to expect no runtime resolver call when an active registry exists.

## Best Possible Solution

Keep the performance direction, but make the fast path compatibility-aware: normalize the requested workspace, reuse the active registry only for the same workspace/runtime surface, fall through for mismatches or placeholder state, and add focused regression coverage plus the changelog entry.

## Reproduction Assessment

Yes. A focused unit test can set an active registry for workspace A or default runtime mode, call `ensureRuntimePluginsLoaded` for workspace B or gateway binding, and assert `resolveRuntimePluginRegistry` is still called.

## Solution Assessment

No. The proposed early return is narrower than changing the shared cache-key matcher, but it must still honor the existing loader compatibility contract before skipping resolution.
