## Summary

Keep open. Current main still has the reported dispatch-time registry reload path, so the PR is relevant, but the proposed early return is too broad because it skips the existing registry compatibility contract before checking config, workspace, runtime mode, or placeholder state. The branch also lacks the required changelog entry for a user-facing performance fix.

## What This Changes

The PR adds an active-registry fast path to `ensureRuntimePluginsLoaded` and updates its unit tests to assert that dispatch does not call `resolveRuntimePluginRegistry` when an active registry is present.

## Best Possible Solution

Keep the performance direction, but implement a compatibility-preserving fast path: reuse the gateway boot registry for dispatch only when it is semantically compatible, fall through for mismatched workspace/config/runtime/scope or placeholder registries, cover both cases with regression tests, and add the required changelog entry.

## Reproduction Assessment

Yes. The reported reload path is reproducible from source: inbound dispatch calls `ensureRuntimePluginsLoaded`, which builds a smaller load options object than gateway boot, while the loader currently requires strict cache-key compatibility. The PR's regression can be reproduced with an active registry keyed to one workspace or scope followed by an `ensureRuntimePluginsLoaded` call for another, which current loader tests already define as incompatible.

## Solution Assessment

No. A fast path is the right shape, but raw `getActivePluginRegistry()` truthiness is not the safest implementation because it bypasses the loader's compatibility checks. A narrower helper or explicit gateway-loaded marker should preserve compatibility while avoiding the dispatch cache miss.
