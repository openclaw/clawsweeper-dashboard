## Summary

Needs changes before merge: the latest head fixes the earlier unconditional pre-hook staging, but it still pre-stages cloned remote iMessage media whenever any global `message_received` hook exists, which can preserve the duplicate-SCP availability risk for ordinary non-plugin turns in setups with bundled startup hooks. The PR body proof also appears stale after the latest force-push because it describes the old “stage before first hook-context derivation” order rather than the current lazy prepare path.

## What This Changes

The PR stages remote iMessage media into a local remote cache for plugin hook/claim metadata before Codex receives turn input, keeps the original reply context available for get-reply staging, and adds staging/Codex regression tests.

## Best Possible Solution

Keep the plugin-bound Codex claim metadata on a staged remote-cache path, but remove the broad global `message_received` staging trigger or make it channel/media-aware, then prove both the Codex-bound path and an ordinary remote iMessage path with a global ignored hook.

## Reproduction Assessment

Yes. Source inspection shows current main builds plugin-bound Codex claim metadata from raw remote iMessage paths before staging; PR-head source also shows the remaining broad `message_received` hook trigger that can reproduce the duplicate-SCP availability risk without a live remote Mac.

## Solution Assessment

No. The owner-boundary direction is right for the Codex-bound bug, but the current implementation is still too broad; the best fix stages for the plugin-bound claim path, or for a hook consumer proven to need media for this channel.
