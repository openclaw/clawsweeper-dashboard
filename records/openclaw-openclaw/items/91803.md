## Summary

Keep open: current main still builds plugin-bound Codex claim metadata from raw remote iMessage media before dispatch-side staging, and this PR is the open candidate fix for that remaining bug. ## What This Changes

This PR stages remote iMessage media into the local cache before plugin-bound Codex inbound claims, adds pending/original media hook metadata for unstaged remote media, and covers dispatch, staging, hook mapper, and Codex input paths with tests.

## Best Possible Solution

Land this dispatch-layer staging fix once maintainers explicitly accept the pending-media hook contract and the pre-claim SCP staging tradeoff; keep Codex consuming staged localImage paths and leave ordinary get-reply staging single-owner for non-plugin turns.

## Reproduction Assessment

Yes. Current-main source shows plugin-bound Codex claim metadata is built from raw remote iMessage media before dispatch-side staging, and Codex reads localImage paths from the local filesystem.

## Solution Assessment

Yes, with maintainer acceptance of the hook contract. Staging the dispatch hook copy before plugin-bound Codex claims is the right layer because Codex expects readable localImage paths; teaching Codex about remote macOS Messages paths would be channel-specific and less maintainable.
