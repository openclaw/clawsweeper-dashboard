## Summary

Keep open: this PR fixes the right linked bug path, but the current patch stages remote media too broadly and can add an extra SCP copy to ordinary remote iMessage turns that do not need plugin-bound Codex claim metadata.

## What This Changes

The PR stages remote iMessage media before plugin hook and plugin-bound Codex inbound-claim metadata is built, while keeping the original reply context unstaged for later agent processing.

## Best Possible Solution

Keep the plugin-bound Codex claim metadata on a local remote-cache path, but gate pre-hook remote staging to consumers that actually need it so ordinary remote iMessage agent turns still use the existing single get-reply staging path.

## Reproduction Assessment

Yes. Source inspection shows current main builds plugin-bound claim metadata from raw remote iMessage media paths before staging, and the PR source shows the non-plugin path can now run cloned pre-hook staging plus later original-context get-reply staging.

## Solution Assessment

No. The core fix direction is right, but the implementation is too broad; the best fix is to stage only for the plugin-bound or hook metadata path that needs a shared absolute local path.
