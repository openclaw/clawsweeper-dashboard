## Summary

Keep open: current main still has source-visible paths that classify accepted-before-hello Gateway closes as unreachable/not running, which matches the Windows report and the later Linux protocol-mismatch transcript.

## What This Changes

The issue asks for `gateway probe` and `doctor` to stop reporting a healthy/listening Gateway as unreachable or not running, and to surface the real auth/protocol/handshake failure instead.

## Best Possible Solution

Implement a tri-state Gateway diagnostic state for no listener, accepted-but-handshake/read/auth/protocol failed, and fully reachable, then use it consistently in `gateway probe`, status JSON/text, and doctor panels.

## Reproduction Assessment

Yes, source-reproducible on current main: an accepted WebSocket that closes before `hello-ok`, such as protocol mismatch or auth rejection, leaves `connectLatencyMs` null and is then rendered as unreachable/not running. I did not run the Windows npm install path live.

## Solution Assessment

Yes, the narrow maintainable fix is to carry accepted-before-hello failure state through probe/status/doctor diagnostics while preserving the existing auth and protocol fail-closed behavior.
