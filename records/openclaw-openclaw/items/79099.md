## Summary

Keep open. Current main still has source-visible paths where a gateway can accept a WebSocket or have a listening port, but the probe and doctor diagnostics collapse pre-hello handshake failures into unreachable or not-running output.

## What This Changes

The issue reports misleading `gateway probe` and `doctor` diagnostics when a healthy or listening gateway fails during probe/auth/protocol negotiation.

## Best Possible Solution

Carry an explicit connection-stage result through gateway probe, status, and doctor so no listener, accepted-but-handshake/auth/protocol-failed, and fully healthy states render differently.

## Reproduction Assessment

Yes. A live current-main run was not performed, but source inspection gives a high-confidence path: pre-hello closes leave `connectLatencyMs` null, status treats that as unreachable, and doctor can still show a separate port-in-use panel.

## Solution Assessment

Yes. The narrow maintainable fix is diagnostic state propagation in probe/status/doctor, not a change to gateway auth policy or protocol compatibility.
