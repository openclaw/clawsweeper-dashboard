## Summary

Keep open. Current main still has source-visible paths where a WebSocket is accepted and then closed before `hello-ok`, but `gateway probe`, status output, and doctor still collapse that into unreachable or not running instead of reachable-but-errored diagnostics.

## What This Changes

The issue asks `gateway probe` and `doctor` to report a listening, health-ok Gateway as reachable or to surface the specific auth/protocol/probe-path failure instead of saying it is unreachable or not running.

## Best Possible Solution

Carry a tri-state Gateway connectivity result through probe, status, and doctor: no listener, listener accepted but handshake/auth/protocol/read failed, and fully healthy/reachable.

## Reproduction Assessment

Yes at source level: force an accepted Gateway WebSocket to close before `hello-ok`, and current main keeps `connectLatencyMs` null, causing status/probe to render unreachable. I did not live-run the Windows timeout or Linux two-version protocol-skew setup.

## Solution Assessment

Yes. The narrow maintainable fix is to add an accepted-but-errored diagnostic state and render the underlying auth/protocol/handshake reason without weakening Gateway access policy.

Fix the Gateway probe/status/doctor diagnostic contradiction reported in https://github.com/openclaw/openclaw/issues/79099. Current main treats a WebSocket accepted by the Gateway but closed before `hello-ok` as unreachable because `probeGateway` only records `connectLatencyMs` after `hello-ok`; doctor also maps `gateway closed` to `Gateway not running.` even when the port scan proves a Gateway listener exists. Do not loosen auth, pairing, origin, TLS, or protocol validation. Add focused regression coverage for pre-hello close/protocol-mismatch classification, status text/JSON warning output, and doctor health/daemon wording.
