## Summary

Keep open. Current main and the latest release still collapse pre-hello WebSocket closes, protocol mismatches, and generic `gateway closed` doctor failures into unreachable or not-running diagnostics, even though the docs define accepted WebSocket connect as reachable and the report has multiple real-world reproductions.

## What This Changes

The issue asks for `gateway probe` and `doctor` to report a live-but-failing gateway as reachable-but-errored, not unreachable or not running.

## Best Possible Solution

Carry a tri-state gateway connectivity result through probe, status, and doctor: no listener, listener accepted but handshake/auth/protocol/read failed, and fully healthy/reachable.

## Reproduction Assessment

Yes, at source level: simulate a gateway accepting a WebSocket and closing before `hello-ok`, or a protocol-mismatch handshake close, and current main keeps `connectLatencyMs` null so probe/status treat it as unreachable. I did not live-run the reported Windows environment.

## Solution Assessment

Yes. The narrow maintainable fix is diagnostic state propagation for accepted-but-errored connections, not weakening gateway auth/protocol checks or adding a Windows-specific timeout workaround.
