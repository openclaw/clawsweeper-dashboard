## Summary

Keep open. Current main still has a source-visible path where reachable handshake failures can be surfaced as “Gateway not running,” and the landed auth-none fix does not cover the broader probe/doctor diagnostic contradiction added in the latest issue comment.

## What This Changes

The issue asks OpenClaw to stop reporting a listening gateway as unreachable or not running when health, port ownership, or handshake behavior shows the gateway is alive but the probe path failed.

## Best Possible Solution

Use the existing reachability contract to distinguish no listener, socket accepted but handshake/RPC failed, and fully healthy states, then make doctor and gateway probe report the specific close reason without changing gateway auth boundaries.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not by live execution in this review; the issue gives concrete Windows and Linux command sequences, and current source shows the reachable-handshake-close path that can still be rendered as not running.

## Solution Assessment

Is this the best way to solve the issue? Yes for the remaining diagnostic bug: a narrow reachability-state fix in probe/doctor is more maintainable than changing auth-none policy or treating every gateway close as a stopped gateway.
