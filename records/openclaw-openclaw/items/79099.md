Keep open. Current main has fixed part of the overlapping auth-none backend pairing issue, but the remaining probe/doctor diagnostic problem is still source-visible: pre-hello handshake closes can still be rendered as unreachable/not running even when the gateway process accepted a connection or the port scan finds the listener.

## What This Changes

Report that `gateway probe` and `doctor` can say a healthy/listening gateway is unreachable or not running on Windows auth-none and version-skew protocol-mismatch cases.

## Best Possible Solution

Add focused probe/doctor state handling that distinguishes no listener from accepted-but-errored handshakes, with regression coverage for auth-none and protocol-mismatch diagnostics.

## Reproduction Assessment

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to preserve existing auth semantics while making probe/doctor report accepted-but-errored handshakes as reachable/degraded or explicitly failed, not not-running.
