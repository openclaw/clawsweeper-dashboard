## Summary

Keep open. The no-auth backend subcase has been fixed, but current main still has source-visible paths where a WebSocket connection that fails before `hello-ok` remains classified as unreachable/not-running instead of reachable-but-errored.

## What This Changes

The issue asks for gateway probe and doctor diagnostics to distinguish an alive listener with failed auth/protocol/probe handshakes from a truly unreachable or stopped gateway.

## Best Possible Solution

Add an explicit probe/doctor diagnostic state for listener accepted but handshake, auth, protocol, or read diagnostics failed, and render that as reachable-but-errored instead of not running.

## Reproduction Assessment

Yes for a source-level reproduction, but not a live replay. Running a newer CLI against an older gateway protocol path or any pre-hello close should follow the current `onConnectError`/`onClose` path that leaves `connectLatencyMs` null and renders the target unreachable.

## Solution Assessment

Yes. The narrow fix is to carry a tri-state probe result through gateway probe, status, and doctor without changing gateway auth policy or treating failed handshakes as successful operator access.
