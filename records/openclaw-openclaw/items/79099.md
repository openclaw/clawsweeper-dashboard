## Summary

Keep open. Current main still has source-visible paths where an accepted-but-failed gateway handshake can be rendered as unreachable/not running, and the newer protocol-mismatch comment broadens the report beyond the already-fixed auth-none subcase.

## What This Changes

The issue asks for `gateway probe` and `doctor` to distinguish a live gateway listener with handshake/auth/protocol failure from a truly unreachable or stopped gateway.

## Best Possible Solution

Carry an explicit probe/doctor diagnostic state for no listener, listener accepted but handshake/auth/protocol/read failed, and fully healthy, then render only the first state as not running.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: a protocol mismatch closes an accepted WebSocket before hello-ok, `probeGateway` leaves `connectLatencyMs` null, and status/doctor can render that as unreachable or not running while the port scan proves a listener.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is diagnostic-state propagation in probe/status/doctor, not a change to gateway auth or protocol policy.
