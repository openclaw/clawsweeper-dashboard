## Summary

Keep open. The adjacent auth-none device-identity bug is fixed on current main, but the central diagnostic problem remains source-visible: an accepted Gateway socket that closes during auth/protocol handshake can still lose its reachable state and flow into unreachable/not-running diagnostics while the port scan proves a listener exists.

## What This Changes

The issue asks `gateway probe` and `doctor` to report a live listener with failed auth/protocol/probe handshakes as reachable-but-errored instead of unreachable or not running.

## Best Possible Solution

Add a narrow probe/status/doctor diagnostic state for no listener, listener accepted but handshake/auth/protocol/read failed, and fully healthy RPC success, while preserving existing auth and protocol policy.

## Reproduction Assessment

No live current-main reproduction was run. The issue has concrete Windows and Linux transcripts, and current source shows the accepted-but-pre-hello failure path still collapses to unreachable/not-running diagnostics.

## Solution Assessment

Yes. A focused diagnostic-state fix in `probeGateway`, gateway-status rendering, and doctor health/daemon flow is the narrow maintainable path; changing auth policy or adding a config knob would be the wrong fix.
