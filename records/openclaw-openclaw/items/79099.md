## Summary

Keep open. Current `main` still has source-visible paths where a Gateway socket can be accepted and then fail before `hello-ok`, while `gateway probe` and `doctor` classify that as unreachable or not running instead of a reachable-but-errored Gateway.

## What This Changes

The issue requests clearer `gateway probe` and `doctor` diagnostics when the Gateway listener is alive but probe/auth/protocol handshakes fail.

## Best Possible Solution

Carry a tri-state Gateway connectivity result through probe/status/doctor: no listener, accepted connection with handshake/auth/protocol/read failure, and fully healthy/reachable.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: force an accepted Gateway WebSocket to close before `hello-ok`, and current `probeGateway` returns `connectLatencyMs: null` while status/doctor render unreachable or not running. I did not run a live Windows or two-version Linux smoke.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable fix is to add a distinct accepted-but-errored diagnostic state and render the underlying auth/protocol/handshake reason without weakening Gateway access policy.
