## Summary

Keep open. This is a fresh current-stable report with a new Linux protocol-mismatch reproduction, and current `main` still has source-visible paths that can collapse accepted-but-failed gateway handshakes into unreachable/not-running diagnostics while a separate port scan can prove a listener exists.

## What This Changes

The issue asks for `gateway probe` and `doctor` to distinguish a listening/healthy gateway with auth, protocol, or handshake failures from a truly unreachable or not-running gateway.

## Best Possible Solution

Teach the gateway probe and doctor health flow to carry a tri-state result: no listener, listener accepted but handshake/read/auth/protocol failed, and fully healthy/reachable.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was run, but the issue provides concrete Windows and Linux command transcripts and current source shows the accepted-but-pre-hello failure path still renders as unreachable/not running.

## Solution Assessment

Is this the best way to solve the issue? Yes, the maintainable fix is a narrow diagnostic-state change in probe/status/doctor that preserves auth policy while surfacing protocol/auth/handshake failures distinctly from no listener.
