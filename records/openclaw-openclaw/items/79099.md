## Summary

Keep open. Current main still has source-visible gateway probe/status/doctor paths that can report a listening Gateway as unreachable or not running, and the latest issue comment confirms the contradiction on v2026.5.22 after the partial protocol-mismatch diagnostics work shipped.

## What This Changes

The issue asks for Windows Gateway probe and doctor diagnostics to agree with a healthy/listening Gateway or report the specific probe/auth/handshake failure instead of saying unreachable or not running.

## Best Possible Solution

Carry an accepted-but-errored Gateway state through probe/status/doctor, or finish the bounded local fallback in https://github.com/openclaw/openclaw/pull/73724 if maintainers decide that is the right path, while keeping security-sensitive auth behavior unchanged.

## Reproduction Assessment

Yes at source level: force an accepted Gateway WebSocket to close before `hello-ok`, or a local probe timeout while direct status/health succeeds, and current main still classifies the null-latency probe as unreachable. I did not run the exact Windows commands during this read-only review.

## Solution Assessment

Unclear whether the existing open fallback PR is the complete solution. The maintainable end state is a narrow diagnostic classification fix that distinguishes no listener from accepted-but-handshake/auth/protocol/read failed without relaxing Gateway auth policy.
