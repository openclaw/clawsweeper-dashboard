## Summary

Keep open. Current main still has a source-visible path where a Gateway connection that is accepted and then fails before `hello-ok` is rendered as unreachable/not running, and the issue has a fresh 2026.5.22 Windows confirmation rather than stale evidence.

## What This Changes

The issue asks for `gateway probe` and `doctor` diagnostics to agree with a healthy/listening Gateway or report the specific auth, timeout, or protocol failure instead of saying the Gateway is unreachable or not running.

## Best Possible Solution

Carry an explicit Gateway connectivity state through probe, status, and doctor: no listener, listener accepted but handshake/auth/protocol/read failed, and fully healthy/reachable.

## Reproduction Assessment

Yes at source level: an accepted Gateway WebSocket that closes or errors before `hello-ok` can leave `connectLatencyMs` null, and current status/doctor paths still render that as unreachable or not running. I did not live-run the Windows timing case in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve the existing access checks while adding a reachable-but-errored diagnostic state and surfacing the underlying timeout/auth/protocol reason.
