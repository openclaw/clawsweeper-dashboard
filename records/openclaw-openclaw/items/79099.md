## Summary

Keep open. The auth-none backend subcase appears addressed by the merged fix for https://github.com/openclaw/openclaw/pull/75781, but current main still has source-visible paths where a WebSocket accepted then closed before `hello-ok` is rendered as unreachable/not running, which matches the later protocol-mismatch comment and the documented reachability contract.

## What This Changes

Report that Windows `gateway probe`/`doctor` can claim a healthy listening gateway is unreachable/not running, with a later Linux protocol-mismatch case showing the same diagnostic contradiction.

## Best Possible Solution

Add a narrow gateway probe and doctor diagnostic state that distinguishes no listener from accepted-but-closed handshake/auth/protocol failures, then render the latter as reachable-but-errored with the close reason.

## Reproduction Assessment

No live reproduction was run. The issue and follow-up comment provide concrete command transcripts, and current source shows the accepted-before-hello close path still renders through the same unreachable/not-running diagnostics.

## Solution Assessment

Yes. The best fix is a narrow diagnostic-state change in probe/status/doctor that preserves auth and protocol policy while surfacing handshake, auth, or protocol failures separately from no listener.
