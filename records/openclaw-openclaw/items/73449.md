## Summary

Keep open. The PR addresses a real Codex app-server gap, but the latest diff still publishes new normalized tool events with unsanitized args and full-output payloads, unlike the existing Pi tool stream path; the user-facing fix also still lacks the required changelog entry.

## What This Changes

The PR maps Codex app-server tool-like ThreadItems into normalized `stream: "tool"` start, update, and result events and adds Codex projector regression tests.

## Best Possible Solution

Keep the Codex tool-stream bridge, but redact all new start/update/result payloads with the same semantics as existing Pi tool streams, preserve the full-output gates, add secret-bearing regression coverage, and record the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. On the PR head, instantiate `CodexAppServerEventProjector`, send a command/MCP/dynamic item with secret-like args or full-output result data, and inspect the emitted `stream: "tool"` events; the new code forwards those payloads without the existing tool-stream sanitizer.

## Solution Assessment

No. Bridging Codex app-server items into normalized tool events is the right narrow direction, but this implementation is not the best safe version until producer-side redaction and changelog policy are handled.
