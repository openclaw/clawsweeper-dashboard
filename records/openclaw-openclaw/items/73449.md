## Summary

Keep this PR open. Current main still lacks the requested Codex normalized tool stream bridge, so the contribution is not obsolete, but the latest PR head is not merge-ready because it publishes unsanitized Codex tool args/update/full-output result payloads and still lacks the required user-facing changelog entry.

## What This Changes

The PR maps Codex app-server tool-like ThreadItems into normalized `stream: "tool"` start, update, and result events and adds projector tests for command lifecycle, result dedupe, and non-full-output redaction of bulky payloads.

## Best Possible Solution

Keep the Codex tool-stream bridge, but publish only sanitized tool args, partial results, and full-output final results through an SDK-safe shared sanitizer or equivalent extension-local sanitizer; keep the existing bulky-output gates, add regression coverage with secret-bearing command/MCP/dynamic/image payloads, and add the user-facing changelog entry.

## Reproduction Assessment

Yes. On the PR head, instantiate `CodexAppServerEventProjector` with `onAgentEvent`, send a `commandExecution` or MCP/dynamic item containing a secret-looking arg/output under full output, and inspect the emitted `stream: "tool"` event; the new code builds and emits those fields without applying the tool payload sanitizer.

## Solution Assessment

No. Bridging Codex app-server items into normalized tool events is the right narrow direction, but this implementation is not the best safe version until it matches the existing Pi stream's producer-side sanitization and restores changelog coverage.
