## Summary

Keep open. The bridge is directionally useful and the author addressed the earlier diff/prompt bulk-output comments, but the latest PR head still publishes Codex tool args/update/result payloads without the sanitizer used by existing Pi tool streams, and the user-facing fix still lacks a changelog entry.

## What This Changes

The PR maps Codex app-server tool-like ThreadItems into normalized `stream: "tool"` start, update, and result events and adds projector regression tests.

## Best Possible Solution

Keep the Codex tool-stream bridge, but route args, partial results, final results, and errors through an extension-safe redaction layer matching the Pi tool stream, preserve the existing full-output gates, add secret-bearing regression tests, and add the required changelog entry.

## Reproduction Assessment

Yes. On the PR head, instantiate `CodexAppServerEventProjector`, send an MCP or dynamic tool item with a secret-like argument or error/result, and inspect `onAgentEvent`; the new `stream: "tool"` payload is emitted without the Pi sanitizer.

## Solution Assessment

No. Bridging Codex app-server items into normalized tool events is the right narrow direction, but the implementation is not the best safe version until it matches existing producer-side redaction and release-note policy.
