## Summary

Keep open. The PR targets a real Codex app-server gap on current main, but the latest head still publishes new normalized tool events with raw args and full-output/update payloads instead of the existing producer-side tool payload sanitizer, and it still lacks the required user-facing changelog entry.

## What This Changes

This PR maps Codex app-server tool-like ThreadItems into normalized `stream: "tool"` start, update, and result events and adds projector regression tests.

## Best Possible Solution

Keep the Codex tool-stream bridge, but apply the same producer-side sanitizer used by the Pi tool stream to args, partial results, and final full-output results, preserve the existing full-output gates, add secret-bearing regression cases, and record the fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. On the PR head, instantiate `CodexAppServerEventProjector`, send a command/MCP/dynamic item containing a secret-like arg or full-output result, and inspect the emitted `stream: "tool"` event; the new code forwards those payloads without the Pi sanitizer.

## Solution Assessment

No. Bridging Codex app-server items into normalized tool events is the right narrow direction, but this implementation is not the best safe version until it matches existing tool-stream redaction and changelog policy.
