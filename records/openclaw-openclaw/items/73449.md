## Summary

Keep open. The Codex tool-stream bridge is useful and focused, but the PR still emits new global tool event args, partial results, and full-output results without the existing Pi tool payload sanitizer, and it is missing the required user-facing changelog entry.

## What This Changes

The PR adds Codex app-server projection of tool-like ThreadItems into normalized `stream: "tool"` start, update, and result events with projector regression tests.

## Best Possible Solution

Keep the Codex tool-stream bridge, but apply the same producer-side redaction semantics used by Pi to start args, update chunks, and final results, preserve the existing full-output gates, add secret-bearing Codex projector tests, and record the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. On the PR head, instantiate `CodexAppServerEventProjector`, send command/MCP/dynamic or image items containing secret-like args or full-output results, and inspect emitted `stream: "tool"` payloads; the new code passes those payloads through without the Pi sanitizer.

## Solution Assessment

No. Bridging Codex app-server items into normalized tool events is the right narrow direction, but this implementation is not the best safe version until it uses a boundary-correct shared sanitizer and includes the required changelog entry.
