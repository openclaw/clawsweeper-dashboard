## Summary

Keep this PR open for now, but it should not merge as submitted: the source-backed crash is real on current main and the latest release, while this patch converts malformed required token/text prompt results into empty strings that can silently keep or skip required setup values. The stricter replacement at https://github.com/openclaw/openclaw/pull/90284 is the safer open candidate, but it is not merged yet.

## What This Changes

This PR changes channel setup token and generic text-input prompts to coerce non-string `WizardPrompter.text()` results to empty strings and adds token prompt unit coverage for undefined results.

## Best Possible Solution

Preserve the defensive setup-wizard guard while keeping required credential/text inputs strict, then land a proven strict fix such as https://github.com/openclaw/openclaw/pull/90284 or update this branch to match that behavior with real CLI proof.

## Reproduction Assessment

Yes for source-level reproduction: current main and `v2026.6.8` still call `.trim()` directly on awaited setup prompt text in the token replacement and generic text-input paths. I did not run the full interactive macOS onboarding sequence in this read-only review.

## Solution Assessment

No, not as submitted. Guarding the shared setup prompt boundary is the right direction, but required token/text prompt failures should reject or re-prompt instead of becoming keep/skip outcomes; https://github.com/openclaw/openclaw/pull/90284 shows the safer shape but is not merged.
