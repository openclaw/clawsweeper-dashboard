## Summary

Keep open. The PR targets a real current-main and latest-release setup crash path, but it is not merge-ready because its empty-string fallback can bypass required token/text validation, and the supplied proof is still mocked unit-test proof rather than a real setup run.

## What This Changes

The PR replaces direct `.trim()` calls on setup prompter text results with `normalizeOptionalString(...) ?? ""` in the channel token and generic text-input setup paths, plus unit tests for undefined token prompt results.

## Best Possible Solution

Keep the narrow setup-wizard guard, but preserve required-input semantics: reject or re-prompt on nullish required token/text results, only keep an existing credential after an explicit keep path, and add real CLI setup proof.

## Reproduction Assessment

Yes for source-level reproduction: current main still calls `.trim()` directly on awaited setup prompt text in the token-replace and generic text-input paths. I did not run the full interactive macOS onboarding sequence in this read-only review.

## Solution Assessment

No. Guarding the two crash sites is the right boundary, but required token/text inputs should reject or re-prompt on nullish results instead of converting them into keep/skip outcomes.
