## Summary

Keep this PR open, but it is not merge-ready: it targets a real setup-wizard crash that still exists on current main and the latest release, but the submitted fallback can silently keep or skip required credential/setup values and the external contributor proof is still mock-only. The stricter replacement at https://github.com/openclaw/openclaw/pull/90284 is related, but it is not merged and still has proof/maintainer handling pending, so it is not a safe reason to close this branch yet.

## What This Changes

The PR changes channel setup wizard token and text-input prompt handling to normalize non-string `prompter.text()` results to an empty string, plus unit tests for undefined token prompt results.

## Best Possible Solution

Preserve the setup-wizard guard while rejecting or re-prompting malformed required credential/text input, then merge after redacted real CLI proof or let the stricter replacement PR land.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.6.10 still call `.trim()` directly on awaited setup prompt text in the token replacement and generic text-input paths. I did not run the full interactive macOS onboarding sequence in this read-only review.

## Solution Assessment

No as submitted. Guarding the setup prompt boundary is the right direction, but required token/text prompt failures should reject or re-prompt instead of becoming keep or skip outcomes.
