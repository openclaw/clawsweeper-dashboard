## Summary

Keep open. Current main still forwards OpenShell exec command text into the SSH shell wrapper without the requested generated-command or malformed-shell preflight, and generic repeated tool calls still remain warning-level until the separate global no-progress breaker fires.

## What This Changes

Request to reject malformed/generated OpenShell commands before shell execution and hard-abort repeated identical tool-call loops after threshold.

## Best Possible Solution

Add a focused preflight in the OpenShell/SSH exec path that rejects the reported non-runnable placeholders and malformed shell syntax before remote shell invocation, then resolve generic-repeat critical blocking through the existing loop-detection work so one canonical policy ships with tests and docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a command such as `workflow install <name>` or an unclosed quote still flows from `extensions/openshell/src/backend.ts` into `src/agents/sandbox/ssh.ts` and then `/bin/sh -c`; loop tests also assert generic repeats remain warning-level below the global breaker.

## Solution Assessment

Yes, with scoping. The maintainable direction is a narrow OpenShell/SSH exec preflight plus coordinated loop-detector escalation, rather than a broad shell filter across unrelated exec modes or a parallel loop policy.

## Repair Work Prompt

Fix the OpenShell malformed/generated command preflight gap for this issue. Add a focused guard before OpenShell SSH exec can invoke the remote shell, rejecting obvious non-runnable generated placeholders such as `workflow install <name>` / `<workflow-id>`, unclosed single or double quotes, trailing backslash escapes, and unterminated backticks or command substitutions. Keep the boundary narrow: prefer OpenShell/SSH exec validation near `extensions/openshell/src/backend.ts` and/or `src/agents/sandbox/ssh.ts`; do not broaden unrelated host exec behavior unless a shared helper is clearly appropriate. Add regression tests proving invalid commands fail before SSH shell execution and ordinary valid shell commands still pass. Do not duplicate the generic-repeat loop policy work from #58726, #60248, or #62775; only touch loop detection if the maintainer explicitly asks to take over that canonical policy. Add a user-facing `CHANGELOG.md` fix entry crediting `Thanks @Kaspnov`.
