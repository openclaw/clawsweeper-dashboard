## Summary

Keep open. Current main still lacks OpenShell/generated-command preflight before the SSH `/bin/sh -c` wrapper, and the generic repeated-call detector still warns until the separate global no-progress breaker. The linked loop-detection issue and PRs overlap the hard-abort half, but they do not cover the OpenShell malformed-command validation gap.

## What This Changes

The issue requests OpenShell command preflight rejection for malformed/generated shell commands and hard-abort behavior for repeated identical tool calls.

## Best Possible Solution

Add a focused preflight before OpenShell/SSH remote shell invocation that rejects the reported generated placeholders and malformed shell syntax with a clear tool-visible error, then resolve the repeated identical-call hard-abort through the existing loop-detection tracking so one canonical threshold behavior ships with tests and docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a command such as `workflow install <name>` or an unterminated quote flows from `extensions/openshell/src/backend.ts` into `src/agents/sandbox/ssh.ts` and then `/bin/sh -c`; repeated generic tool calls are covered by tests that still assert warning-only behavior below the global breaker.

## Solution Assessment

Yes. The narrow maintainable direction is OpenShell/SSH exec preflight plus coordinated loop-detector escalation, not a broad shell-obfuscation policy or a duplicate generic loop PR.
