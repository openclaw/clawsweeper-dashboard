## Summary

Keep open. Current main still passes OpenShell exec text through to the SSH `/bin/sh -c` wrapper without the requested generated-command preflight, and the generic repeated-call detector still warns until the separate no-progress circuit breaker. The related loop-detection items overlap the hard-abort half, but they remain open and do not cover the OpenShell malformed-command validation gap.

## What This Changes

Reject malformed or generated OpenShell commands before shell execution and hard-abort repeated identical tool-call loops after the configured threshold.

## Best Possible Solution

Add a focused OpenShell/SSH exec preflight that rejects the reported generated placeholders and malformed shell syntax with a clear tool-visible error before any SSH shell invocation, then resolve the generic repeat hard-abort through the existing loop-detection tracking so one threshold/default policy ships with tests and docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `workflow install <name>` or an unterminated quote reaches `buildExecRemoteCommand` from OpenShell and is wrapped for `/bin/sh -c`; generic repeated non-poll calls are still covered by tests that assert warn-only behavior below the global breaker.

## Solution Assessment

Yes. The narrow maintainable direction is an OpenShell/SSH generated-command preflight plus a coordinated loop-detector escalation, not a broad host-exec command-obfuscation policy or a duplicate uncoordinated loop PR.
