## Summary

Keep open. Current main still has the OpenShell generated-command validation gap: OpenShell passes raw exec text into the shared SSH `/bin/sh -c` wrapper without rejecting literal placeholders or malformed shell syntax first. The repeated-loop half overlaps the already-open loop-detection items, but those do not cover the OpenShell command preflight request.

## What This Changes

The issue requests OpenShell preflight rejection for malformed/generated shell commands and hard-abort behavior for repeated identical tool calls.

## Best Possible Solution

Add a focused OpenShell/shared SSH remote-exec preflight that rejects obvious generated placeholders and malformed shell syntax before `/bin/sh -c` is invoked, with regression tests. Keep the generic repeated-tool hard-abort policy coordinated through the existing loop-detection items so one threshold/default policy ships consistently.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: an OpenShell command such as `workflow install <name>` or an unterminated quote flows from `prepareExec` into `buildExecRemoteCommand`, which wraps it for `/bin/sh -c` without rejection on current main.

## Solution Assessment

Yes, with a split boundary. The maintainable fix is a narrow generated-command preflight near the OpenShell/shared SSH exec construction path; the generic repeat-abort behavior should remain with the existing loop-detection cluster rather than being duplicated in this repair.

## Repair Work Prompt

Fix the OpenShell malformed/generated command preflight gap from https://github.com/openclaw/openclaw/issues/72373. Add a narrow validation step before OpenShell/shared SSH remote exec invokes `/bin/sh -c`, rejecting literal generated placeholders such as `<name>` / `<workflow-id>` and obvious malformed shell syntax such as unclosed single/double quotes, trailing backslash escapes, unterminated backticks, and unterminated `$(` substitutions. Return a clear tool-visible error before creating/running the remote shell command. Keep the fix scoped to command validation; do not change the generic repeated-tool-loop policy, which is tracked by #58726, #56403, and #60248. Add regression tests for the rejected commands and keep existing valid wrapping behavior. Add a user-facing changelog fix entry; do not change unrelated sandbox, approval, or loop-detection defaults.
