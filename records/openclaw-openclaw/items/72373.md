## Summary

Keep open. Current main still has the reported OpenShell validation gap: OpenShell exec text is passed into the SSH `/bin/sh -c` wrapper without preflight rejection for placeholder tokens or malformed shell syntax. The repeated-tool-loop half overlaps active loop-detection items, but those do not cover the OpenShell command validation request.

## What This Changes

The issue requests fail-fast validation for malformed/generated OpenShell commands and hard-aborting repeated identical tool-call loops after a threshold.

## Best Possible Solution

Add a narrow OpenShell/SSH exec preflight that rejects the reported generated placeholders and malformed shell syntax with a clear tool-visible error before remote shell invocation, with focused regression tests. Resolve generic repeat hard-abort behavior through the existing loop-detection items so one threshold/default policy ships consistently.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `workflow install <name>` or an unterminated quote flows from OpenShell `prepareExec` into `buildExecRemoteCommand`, which wraps it for `/bin/sh -c` without rejection on current main.

## Solution Assessment

Yes, with a split boundary. The maintainable fix is a targeted generated-command preflight near OpenShell/SSH exec construction, while the repeated-tool abort policy should be coordinated through the existing loop-detection cluster instead of duplicated here.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/72373 for the OpenShell malformed/generated-command preflight gap only. Add a focused validator before OpenShell/SSH exec invokes the remote `/bin/sh -c` wrapper, rejecting obvious generated placeholders such as standalone `<name>` / `<workflow-id>`, unclosed single or double quotes, trailing backslash escapes, and unterminated backticks or command substitutions with a clear tool-visible error. Prefer a small helper at the OpenShell/SSH exec boundary and avoid broad shell parsing or banning legitimate shell redirection/XML-like arguments. Do not change generic tool-loop defaults, thresholds, or critical escalation in this PR; #58726, #56403, and #60248 already track that policy. Add regression tests for the rejected command patterns and preserved valid commands, update `CHANGELOG.md` because this is user-facing reliability behavior, and keep extension/core boundaries intact.
