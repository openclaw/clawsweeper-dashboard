## Summary

Keep open. Current main covers the generic repeated no-progress loop half through the shared loop detector, but the OpenShell/shared SSH exec path still forwards malformed or placeholder command text into the remote `/bin/sh -c` body without a preflight rejection.

## What This Changes

OpenShell should reject malformed generated shell commands before shell execution and hard-abort repeated identical tool loops.

## Best Possible Solution

Add a narrow OpenShell/shared SSH exec preflight that rejects obvious generated-command hazards before remote shell invocation, while preserving the current shared loop detector’s no-progress boundary.

## Reproduction Assessment

Yes for source reproduction: `workflow install <name>` or an unterminated quote flows from OpenShell `prepareExec` into `buildExecRemoteCommand` and then into `/bin/sh -c` without a preflight rejection. I did not run a live OpenShell session because this review is read-only.

## Solution Assessment

Yes, with a split boundary. The maintainable repair is an exec preflight at the OpenShell/shared SSH boundary; the repeated no-progress loop behavior is already covered on current main and broader abort policy should not be mixed into this fix.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/72373 for the remaining OpenShell/SSH exec preflight portion only. Current main already handles generic repeated no-progress tool loops via the shared detector, so do not change loop-detection defaults, thresholds, or broader abort policy. Add a small validation seam before remote `/bin/sh -c` invocation, likely in `src/agents/sandbox/ssh.ts` or the OpenShell backend wrapper, that rejects clearly generated placeholders such as literal `<name>` / `<workflow-id>` in runnable command text and malformed shell syntax such as unclosed quotes, trailing escapes, and unterminated backticks/substitutions before opening the SSH exec.
