## Summary

Keep open. Current main still lacks OpenShell/SSH exec preflight for generated placeholders or malformed shell syntax, while the repeated-tool hard-abort half remains covered by related open loop-detection items.

## What This Changes

Request fail-fast validation for malformed generated OpenShell exec commands and hard-abort behavior for repeated identical tool loops.

## Best Possible Solution

Add narrow OpenShell/SSH exec-command validation before remote shell invocation, and resolve generic repeated-tool hard-abort policy through the existing loop-detection cluster.

## Reproduction Assessment

Yes. A static reproduction is enough: `workflow install <name>` or an unterminated quote flows from OpenShell `prepareExec` into the SSH `/bin/sh -c` wrapper without rejection on current main.

## Solution Assessment

Yes, with a split boundary. The maintainable path is targeted exec-command validation near the OpenShell/SSH wrapper, while #58726, #56403, and #60248 should carry the broader repeated-tool abort/default-policy decision.

## Repair Work Prompt

Fix issue #72373 by adding a narrow OpenShell/SSH exec command preflight before remote `/bin/sh -c` execution. Reject obvious generated placeholders such as `<name>` and `<workflow-id>` when they appear as literal command/argument tokens, unclosed single or double quotes, trailing backslash escapes, and unterminated backticks or command substitutions. Keep the change scoped to OpenShell/SSH exec or a small shared sandbox helper; do not change generic tool-loop defaults, thresholds, or hard-abort policy in this PR because #58726, #56403, and #60248 track that. Add regression tests for rejected commands and preserve valid shell redirection/comparison syntax where feasible. Add a user-facing `CHANGELOG.md` fix entry. Validate with the targeted OpenShell/SSH tests and formatter, then run `pnpm check:changed` in Testbox before handoff if code/tests changed.
