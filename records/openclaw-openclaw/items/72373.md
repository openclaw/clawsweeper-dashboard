## Summary

Keep open. Current main still lets OpenShell exec commands flow into the SSH `/bin/sh -c` wrapper without generated-placeholder or malformed-shell preflight, and the repeated-tool hard-abort behavior remains only partially covered by existing loop-detection work.

## What This Changes

Request OpenShell command preflight rejection for generated or malformed shell commands and hard-abort behavior for repeated identical tool loops.

## Best Possible Solution

Add targeted OpenShell/SSH exec preflight validation for generated-command hazards, and leave generic repeated-tool abort policy to the active loop-detection items so both concerns are fixed at the right boundary.

## Reproduction Assessment

Yes. Static source inspection gives a high-confidence reproduction path: a command such as `workflow install <name>` or an unterminated quote flows through OpenShell `prepareExec` into the SSH `/bin/sh -c` wrapper without repository-side rejection.

## Solution Assessment

Yes, with a split boundary. The narrow maintainable fix is OpenShell/SSH command validation near the exec wrapper, while #58726, #56403, and #60248 should carry the broader repeated-tool hard-abort and default-policy decision.

## Repair Work Prompt

Fix the OpenShell generated/malformed command fail-fast gap from https://github.com/openclaw/openclaw/issues/72373. Add narrow validation before OpenShell/SSH invokes the remote shell so clearly generated non-runnable commands are rejected before `/bin/sh -c`, including literal placeholder tokens such as `<name>` or `<workflow-id>` in executable or argument positions, unclosed quotes, trailing backslash escapes, and unterminated backticks or command substitutions. Keep the change generic to the OpenShell/SSH exec boundary; do not change bundled skill copy, do not implement broad shell parsing beyond the listed hazards, and do not fold in generic repeated-tool loop policy because #58726, #56403, and #60248 already track that. Add focused tests for OpenShell/SSH rejection and preserve existing valid shell-command behavior. Add a user-facing CHANGELOG.md fix entry if prod behavior changes.
