## Summary

Keep open. Current main still passes OpenShell exec text through to the SSH `/bin/sh -c` wrapper without the requested malformed/generated-command preflight, and generic repeated tool calls still warn until the separate no-progress global breaker. The open loop-detection cluster overlaps the hard-abort half, but it does not cover the OpenShell validation gap.

## What This Changes

Request preflight rejection of malformed/generated OpenShell commands and hard-abort escalation for repeated identical tool-call loops.

## Best Possible Solution

Add a focused OpenShell/SSH exec preflight that rejects the reported generated placeholders and malformed shell syntax with a clear tool-visible error before SSH shell invocation, with regression tests. Resolve generic repeat hard-abort behavior through the existing loop-detection tracking so one threshold/default policy ships with aligned tests and docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a command such as `workflow install <name>` or an unterminated quote flows from `prepareExec` into `buildExecRemoteCommand` and then `/bin/sh -c`, while existing loop tests assert disabled-by-default and warn-only generic-repeat behavior.

## Solution Assessment

Yes. The maintainable direction is a narrow OpenShell generated-command preflight plus a coordinated loop-detector escalation, not broad shell parsing or a duplicate uncoordinated loop policy PR.
