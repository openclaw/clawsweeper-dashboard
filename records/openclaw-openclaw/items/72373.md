## Summary

Keep open. Current main still passes OpenShell exec command text into the SSH `/bin/sh -c` wrapper without rejecting generated placeholders or malformed shell syntax, and the repeated-tool hard-abort part is still split across related open loop-detection items.

## What This Changes

Requests OpenShell fail-fast validation for generated or malformed shell commands and hard aborts for repeated identical tool loops.

## Best Possible Solution

Add conservative OpenShell/SSH exec-command validation near the shared remote exec builder, and keep generic repeated-tool abort/default-policy changes in the existing loop-detection cluster.

## Reproduction Assessment

Yes. A static path is enough: `buildExecRemoteCommand` composes `cd ... && ${params.command}` and sends it through `/bin/sh -c`, so `workflow install <name>` or an unterminated quote is not rejected before shell dispatch.

## Solution Assessment

Yes for the OpenShell half if scoped to a small SSH/OpenShell preflight helper with focused tests. The generic repeated-tool abort policy is better handled through #58726, #56403, and #60248 rather than folded into the same repair.

## Repair Work Prompt

Fix the OpenShell malformed generated-command fail-fast gap from https://github.com/openclaw/openclaw/issues/72373. Scope the repair to OpenShell/shared SSH exec preflight only: reject obviously non-runnable generated placeholders such as literal `<name>` / `<workflow-id>` tokens in executable positions or args, and reject malformed shell syntax hazards called out in the issue such as unclosed quotes, trailing escapes, and unterminated backticks/command substitutions before invoking remote `/bin/sh`. Prefer a small helper near `src/agents/sandbox/ssh.ts` or the OpenShell SSH exec path so both `extensions/openshell/src/backend.ts` and shared SSH exec behavior stay consistent. Add focused unit tests for accepted normal shell commands plus rejected examples in `src/agents/sandbox/ssh.test.ts` and/or `extensions/openshell/src/openshell-core.test.ts`. Do not change `tools.loopDetection.enabled`, `generic_repeat` critical escalation, thresholds, docs for loop defaults, or the unrelated loop policy tracked by #58726, #56403, and #60248. Add a single-line user-facing `CHANGELOG.md` fix entry if code behavior changes.
