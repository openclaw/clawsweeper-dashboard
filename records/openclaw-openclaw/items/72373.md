## Summary

Keep open. Current main implements the repeated no-progress loop hard block when loop detection is enabled, but the remaining OpenShell/SSH exec path still builds a `/bin/sh -c` body from raw generated command text without rejecting obvious placeholders or malformed shell syntax first.

## What This Changes

The issue asks OpenShell to reject clearly malformed generated shell commands before execution and to hard-abort repeated identical failing tool loops.

## Best Possible Solution

Add a focused validator at the shared SSH/OpenShell exec boundary that rejects clearly generated placeholders and malformed shell quoting or escaping before remote `/bin/sh -c` invocation, with regression coverage and no unrelated loop-detection policy changes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for source-level reproduction: `workflow install <name>` or an unterminated quote still flows through OpenShell `prepareExec` into `buildExecRemoteCommand` and then into `/bin/sh -c` on current main; I did not run a live OpenShell session.

## Solution Assessment

Is this the best way to solve the issue? Yes. A narrow OpenShell/SSH command preflight is the maintainable fix for the remaining behavior, while the repeated no-progress loop-abort behavior should stay in the existing loop detector.
