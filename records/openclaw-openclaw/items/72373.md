## Summary

Keep open. Current main still passes OpenShell exec command text into the SSH `/bin/sh -c` wrapper without a generated-placeholder or malformed-shell preflight, while the repeated-tool-loop half overlaps existing open loop-detection items.

## What This Changes

Request fail-fast validation for malformed/generated OpenShell commands and hard aborts for repeated identical tool-call loops.

## Best Possible Solution

Add a narrow OpenShell/SSH exec preflight that rejects generated placeholders and malformed shell syntax before remote shell invocation, and leave generic repeat-abort policy to the existing loop-detection cluster.

## Reproduction Assessment

Yes. A static reproduction is enough: a command such as `workflow install <name>` or an unterminated quote flows from OpenShell `prepareExec` into the shared SSH `/bin/sh -c` wrapper on current main without rejection.

## Solution Assessment

Yes, with a split boundary. The maintainable fix is targeted exec-command validation near the OpenShell/SSH wrapper, while the generic repeated-tool hard-abort policy should be resolved through #58726, #56403, or #60248.

## Repair Work Prompt

Implement a narrow fix for https://github.com/openclaw/openclaw/issues/72373 focused on OpenShell/SSH exec command validation. Add preflight rejection before remote shell invocation for obvious generated-command hazards from the report: literal placeholder tokens such as `<name>` and `<workflow-id>` in executable/argument positions, unclosed single or double quotes, trailing backslash escapes, and unterminated backticks or command substitutions. Return a clear tool-visible error before creating/running the remote shell command. Prefer the shared SSH exec construction path only if the behavior is safe for both OpenShell and generic SSH sandbox backends; otherwise keep the OpenShell-specific check in the plugin boundary. Add focused regression tests in `extensions/openshell/src/openshell-core.test.ts` and/or `src/agents/sandbox/ssh.test.ts`, including false-positive coverage for legitimate shell redirection or quoted text if the validator could match angle brackets. Do not change `tools.loopDetection` defaults or generic repeat hard-abort behavior in this PR; those are tracked by #58726, #56403, and #60248. Add a user-facing changelog entry if the fix changes runtime behavior.
