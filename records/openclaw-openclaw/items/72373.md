## Summary

Keep open. Current main still forwards raw OpenShell exec text into the shared SSH `/bin/sh -c` wrapper without a malformed-command preflight, while the repeated-tool hard-abort portion overlaps existing open loop-detection work.

## What This Changes

Request OpenShell preflight rejection for obviously malformed/generated shell commands and hard-abort handling for repeated identical tool calls.

## Best Possible Solution

Add narrow OpenShell/SSH exec-command validation before remote shell invocation, and keep the generic repeated-tool abort/default-enable policy with the existing loop-detection cluster.

## Reproduction Assessment

Yes for source reproduction: commands like `workflow install <name>` or an unterminated quote flow from OpenShell `prepareExec` into the SSH `/bin/sh -c` wrapper without preflight. I did not run a live OpenShell sandbox because this review is read-only.

## Solution Assessment

Yes, with a split boundary. The maintainable repair is focused OpenShell/SSH exec preflight, while loop-detection hard-abort behavior should be resolved through #58726, #56403, and #60248.

## Repair Work Prompt

Implement a narrow fix for openclaw/openclaw#72373: add OpenShell/SSH exec preflight before building the remote `/bin/sh -c` command. Reject clearly non-runnable generated command hazards such as literal placeholder tokens like `<name>` and `<workflow-id>`, unclosed single/double quotes, trailing backslash escapes, and unterminated backticks or command substitutions. Return a clear tool-visible error before creating or invoking the SSH exec command. Add focused regression tests for the helper/path that OpenShell uses. Do not change generic loop-detection defaults, thresholds, or escalation behavior; #58726, #56403, and #60248 cover that separately. Add a user-facing CHANGELOG.md fix entry.
