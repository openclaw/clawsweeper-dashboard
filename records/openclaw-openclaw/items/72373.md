## Summary

Keep open. Current main still passes raw OpenShell/shared SSH exec command text into `/bin/sh -c`, and generic repeated-tool detection remains warning-only with loop detection disabled by default; the related loop-abort items are still open rather than shipped.

## What This Changes

Requests OpenShell/SSH exec to reject malformed generated shell command text before remote shell execution and to hard-abort repeated identical tool loops after the configured threshold.

## Best Possible Solution

Add narrow OpenShell/shared SSH exec-command validation for obvious generated placeholders and malformed shell syntax, while resolving generic repeated-tool abort behavior through the existing loop-detection cluster.

## Reproduction Assessment

Yes. By source inspection, `workflow install <name>` or an unterminated quote flows from OpenShell `prepareExec` into the shared SSH `/bin/sh -c` wrapper without preflight rejection on current main.

## Solution Assessment

Yes, with a split boundary. The maintainable path is targeted validation at the OpenShell/shared SSH exec boundary, while broader loop-detection defaults and critical escalation should be handled in the related loop-detection items.

## Repair Work Prompt

Fix the OpenShell/shared SSH exec-command preflight gap for https://github.com/openclaw/openclaw/issues/72373. Add validation before remote shell invocation so obvious generated-command hazards fail fast with a clear error: literal placeholder tokens such as `<name>` or `<workflow-id>` in executable/argument positions, unclosed quotes, trailing backslash escapes, and unterminated backticks or command substitutions. Prefer a shared helper near `buildExecRemoteCommand` so both OpenShell and generic SSH sandbox exec paths are covered. Add regression coverage for accepted normal commands and rejected malformed/generated commands. Do not change global loop-detection defaults, thresholds, or generic repeated-tool abort policy in this PR; keep that work with #58726, #56403, and #60248. Add a user-facing changelog fix entry unless maintainers decide this is purely internal.
