## What This Changes

The PR adds an optional `deliveryThreadId` fallback to routed thread resolution and passes `preparedSessionState.sessionEntry.deliveryContext.threadId` from `runPreparedReply`.

## Best Possible Solution

Wire a trusted stored thread id into the ACP dispatch event route before visible delivery, preserve the stale-route guard, then add focused regression coverage plus redacted real Discord/ACP proof.

## Reproduction Assessment

Yes, source-level: current main's ACP dispatch event still receives `originatingThreadId` from `dispatch-from-config`, while the canonical report has an ACP key without `:thread:` and a stored `deliveryContext.threadId`. I did not run a fresh live Discord/ACP reproduction, so this is source-reproducible rather than locally reproduced.

## Solution Assessment
