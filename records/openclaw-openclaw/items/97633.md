## Summary

Keep open: this is a real bug-fix attempt, but the current patch is not merge-ready because it wires the stored thread fallback only into `runPreparedReply`; ACP `reply_dispatch` can handle visible output earlier and still receives `routeReplyThreadId` without `deliveryContext.threadId`.

## What This Changes

The PR adds an optional `deliveryThreadId` fallback to routed thread resolution and passes `preparedSessionState.sessionEntry.deliveryContext.threadId` from `runPreparedReply`.

## Best Possible Solution

Wire a trusted ACP session thread id into the ACP `reply_dispatch` route before visible delivery, preserve the stale-route guard, then add focused regression coverage and redacted real Discord/ACP proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-level: the linked report has an ACP key without `:thread:` and a stored `deliveryContext.threadId`, while current ACP `reply_dispatch` still receives a route thread id computed before this PR's new fallback.

## Solution Assessment

Is this the best way to solve the issue? No. The helper fallback is plausible, but wiring it only through `runPreparedReply` misses the earlier ACP dispatch hook path that can deliver visible output and return before that code runs.
