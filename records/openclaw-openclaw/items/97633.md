## Summary

Keep open: the reported ACP/Discord delivery bug is real, but this patch is too narrow because the ACP dispatch hook still gets its `originatingThreadId` from `dispatch-from-config` before `runPreparedReply` runs, so the new `followupRun.originatingThreadId` value does not fix the routed ACP delivery path.

## What This Changes

The PR adds an optional `deliveryThreadId` fallback to `resolveRoutedDeliveryThreadId` and passes `preparedSessionState.sessionEntry.deliveryContext.threadId` from `runPreparedReply`.

## Best Possible Solution

Wire a trusted `deliveryContext.threadId` into the actual ACP dispatch `originatingThreadId` path, preserve the stale-route guard in `dispatch-from-config`, then add focused regression coverage and redacted real ACP Discord delivery proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level, yes: the canonical report supplies an ACP key without a `:thread:` suffix plus stored `deliveryContext.threadId`, and current main's ACP dispatch event path still does not use that field. I did not run a fresh live Discord/ACP reproduction in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No. The helper fallback is plausible, but wiring it only through `runPreparedReply` misses the earlier ACP dispatch hook path that supplies `originatingThreadId` to routed ACP delivery.
