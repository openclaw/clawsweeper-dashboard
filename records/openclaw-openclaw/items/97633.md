## Summary

Keep open: the patch is a plausible narrow fix for the ACP/Discord thread delivery bug tracked by https://github.com/openclaw/openclaw/issues/96665, but this external PR currently has only unit-test output and no real ACP/Discord after-fix proof.

## What This Changes

The PR adds an optional `deliveryThreadId` fallback to routed delivery thread resolution and wires it from `preparedSessionState.sessionEntry.deliveryContext.threadId` in `get-reply-run`.

## Best Possible Solution

Land the narrow fallback only after a focused regression test and redacted ACP/Discord thread proof show stored `deliveryContext.threadId` routes replies without reviving cleared thread routes.

## Reproduction Assessment

Yes at source level, but not with a live run: current main returns no routed thread for an ACP-shaped session key without inbound thread ids, while the canonical issue shows the session entry already carries `deliveryContext.threadId`. I did not run a real Discord/ACP reproduction.

## Solution Assessment

Yes, this appears to be the narrowest source-level fix because it uses already-loaded session state at the `get-reply-run` call site and leaves the sibling `dispatch-from-config` stale-route guard untouched. It still needs a regression test and real behavior proof before merge.
