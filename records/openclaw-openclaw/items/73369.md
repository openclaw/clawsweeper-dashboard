## Summary

Keep open. The PR is scoped to the right surface, but the latest wording still treats `autoThread: true` itself as proof that the final assistant text is automatically delivered. Current `main` resolves Discord guild/channel turns to `message_tool_only` unless the visible-replies mode is automatic, so this prompt can suppress the only visible reply in the default Discord guild/channel path.

## What This Changes

The PR adds a Discord skill guideline and changelog entry telling agents not to repeat conversational replies with `message.send` when auto-thread runtime delivery will post the final reply to the created thread.

## Best Possible Solution

Revise the touched skill and changelog text so the no-repeat rule applies only when the resolved turn context says final replies are automatically delivered to the source or created thread. Keep `message.send` as the required visible-output path for the default Discord guild/channel and autoThread configuration.

## Reproduction Assessment

Yes. The PR-specific regression is reproducible from current source and tests: Discord guild messages resolve as `ChatType: "channel"`, default to `message_tool_only`, and dispatch tests assert the final reply is not sent in that mode.

## Solution Assessment

No in its current wording. A scoped Discord skill clarification is a maintainable fix boundary, but the instruction must key off automatic source delivery or the per-turn prompt, not `autoThread: true` alone.
