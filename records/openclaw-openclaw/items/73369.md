## Summary

Keep open. The PR targets a real Discord autoThread duplicate-reply problem and the Discord skill is a reasonable scope, but the revised wording still overstates current behavior by saying `autoThread: true` means the runtime will deliver final assistant text to the created thread. Current `main` still defaults Discord guild/channel turns to message-tool-only delivery unless automatic visible replies are configured, so the prompt could make agents skip the only visible `message.send` path.

## What This Changes

This PR adds a bundled Discord skill guideline and changelog entry saying `message.send` should not repeat conversational replies when runtime delivery posts the final reply to a Discord auto-created thread.

## Best Possible Solution

Revise the skill and changelog so the duplicate-avoidance rule applies only when the resolved turn context says final replies are automatically delivered to the source thread/channel, while preserving `message.send` as the visible-output path for default Discord guild/channel tool-only turns.

## Reproduction Assessment

Yes. The patch-level regression is reproducible from current source and tests: Discord guild messages resolve as `ChatType: "channel"`, default source delivery resolves to `message_tool_only`, and automatic delivery is only expected after explicit visible-reply configuration.

## Solution Assessment

No. A Discord-scoped skill clarification is the right boundary, but the current text should key off resolved automatic source delivery rather than `autoThread: true` alone.
