## Summary

Keep open. The PR is aimed at a real Discord autoThread duplicate-reply problem, but the current wording still overstates what `autoThread: true` does on current main: Discord guild/channel turns default to `message_tool_only`, and automatic final delivery only happens when visible replies are explicitly configured as automatic or the message tool is unavailable. As written, the skill prompt could make an agent skip `message.send` in the default Discord autoThread path and produce no visible reply.

## What This Changes

The PR adds a Discord skill guideline reserving `message.send` for sideband traffic in autoThread duplicate-reply cases and appends a matching `CHANGELOG.md` fix entry for #73278.

## Best Possible Solution

Revise the skill and changelog so duplicate-avoidance applies only when the per-turn/runtime context says final replies are automatically delivered to the source thread or channel. Preserve `message.send` as the required visible-output path for default Discord guild/channel and autoThread turns that resolve to `message_tool_only`.

## Reproduction Assessment

Yes. The PR-level regression is reproducible from current source and tests: Discord guild messages resolve as `ChatType: "channel"`, default delivery resolves to `message_tool_only`, and the tests only expect automatic delivery after explicit visible-reply configuration.

## Solution Assessment

No. A scoped Discord skill prompt clarification is the right boundary for #73278, but this wording should key off resolved automatic source delivery or explicit `messages.groupChat.visibleReplies: "automatic"`, not `autoThread: true` alone.
