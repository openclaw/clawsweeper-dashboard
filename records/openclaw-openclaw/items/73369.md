## Summary

Keep open. The PR is scoped well, but its current skill and changelog wording still overstates current `main` behavior: Discord auto-thread creation changes the reply target/thread context, while guild/channel final replies still resolve to `message_tool_only` by default unless visible replies are explicitly configured as automatic.

## What This Changes

The PR adds a Discord skill guideline and changelog entry telling agents not to duplicate final conversational replies with `message.send` when Discord auto-thread runtime delivery will post to the created thread.

## Best Possible Solution

Revise the skill and changelog so duplicate-avoidance applies only when the runtime or per-turn context says final replies are automatically delivered. Preserve guidance that default Discord guild/channel turns, including auto-threaded turns under tool-only visible replies, still need `message.send` for visible output and should keep the final answer private after sending.

## Reproduction Assessment

Yes. The PR-specific problem is reproducible from current source and tests: default Discord guild/channel turns resolve to `message_tool_only`, and automatic delivery is asserted only after `messages.groupChat.visibleReplies: "automatic"` is configured.

## Solution Assessment

No in the current wording. A scoped Discord skill clarification is the right boundary, but the no-repeat rule must be keyed to resolved automatic source delivery rather than `autoThread: true` alone.
