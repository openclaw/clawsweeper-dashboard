## Summary

Keep open: current main still has a source-backed path where Telegram's internal temporary-unhealthy transport cooldown is not classified as recoverable, so isolated polling can exit or churn under health-monitor restarts without surfacing a useful user-visible error.

## What This Changes

The issue reports Telegram long polling entering a silent restart/crash loop after a transient `api.telegram.org` timeout, requiring a full gateway restart to recover.

## Best Possible Solution

Fix the Telegram polling recovery path so the temporary-unhealthy transport cooldown is treated as a bounded recoverable poll/setup condition, logged or reflected in channel status, and retried or transport-rebuilt within the Telegram session instead of relying on gateway-wide restart.

## Reproduction Assessment

Yes at source level, but not with a live Telegram run. Repeated transport fallback failures can produce the exact `temporarily unhealthy` error from current source, and that error is not recoverable under the current isolated polling/session handling.

## Solution Assessment

Unclear because there is no proposed patch yet. The best fix should be narrow inside the Telegram transport/polling recovery path, preserving fatal auth/API failures and existing 409 conflict handling without adding a new config surface.
