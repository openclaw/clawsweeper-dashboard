## Summary

Keep open. Current main and v2026.6.8 still contain the Telegram lease and gateway health-monitor behavior implicated by the report, and the open closing PR is not yet merged or safe enough to make this issue obsolete.

## What This Changes

The issue reports that Telegram polling can enter a silent stopped/crash-loop state after transient network timeouts, leaving health-monitor restarts unable to recover until the gateway process is restarted.

## Best Possible Solution

Keep this issue open until a reviewed fix lands that preserves the one-active-poller invariant while making Telegram polling or gateway recovery recover from the reported transient-timeout state, with focused gateway/Telegram regression tests and live or real-runtime proof.

## Reproduction Assessment

Yes at source level, but not as a full live current-main reproduction. The current code still contains the lease, retry, and health-monitor paths described by the issue comments, while the exact transient Telegram outage needs credentialed live proof.

## Solution Assessment

Unclear. A lease-only age timeout is not the best fix as currently proposed; the safer direction is lifecycle-aware recovery that does not permit competing same-token pollers and also addresses the gateway health-monitor state machine if that is the blocker.
