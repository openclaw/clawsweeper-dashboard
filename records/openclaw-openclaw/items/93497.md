## Label Justifications

- `P2`: This is a real Slack threading bug with limited blast radius to media/generated completion sends on internal wake turns.
- `impact:message-loss`: Affected media replies are misrouted to the Slack channel root, where they can be missed or notify the wrong audience.

## Summary

Keep open: current main still drops the persisted thread id on system-event reply routes, the latest release tag also lacks that thread propagation, and open fixing PRs explicitly reference this issue with closing syntax.

## What This Changes

The issue asks Slack media/generated-completion sends on internal wake turns to inherit the active Slack thread instead of posting at the channel root.

## Best Possible Solution

Land a narrow route-layer fix that carries a trusted persisted threadId into system-event reply routes, then close this issue after the merged PR proves Slack media wakes stay in-thread.

## Reproduction Assessment

Yes at source level: current main's system-event route omits persisted threadId, and Slack media/tool threading can only use the envelope fields it receives. I did not run a live Slack workspace reproduction in this read-only pass.

## Solution Assessment

No, current main is not yet the best solved state. The better fix belongs in the generic reply-route inheritance path so all Slack action kinds receive the same active thread context, rather than only patching one Slack media path.
