## Label Justifications

- `P2`: The report describes a real Slack media delivery bug with limited channel/workflow blast radius rather than a core outage.
- `impact:message-loss`: Generated media replies are delivered to the wrong Slack location, making threaded updates easy to miss or misroute.
- `impact:session-state`: The failure is caused by active thread/session context not carrying through internal wake turns to media delivery.

## Summary

Keep open: current main and the latest release still omit persisted thread inheritance in the system-event reply route, and open PRs already reference this issue with closing syntax. The focused next step is to review the linked fix PRs rather than close this issue or start a duplicate repair lane.

## What This Changes

The issue asks for Slack generated media/upload replies on internal wake turns to inherit the active thread instead of posting at the Slack channel root.

## Best Possible Solution

Review and land the focused route-level thread inheritance fix, preferably https://github.com/openclaw/openclaw/pull/94213 if it passes maintainer review and proof, then close this issue as implemented.

## Reproduction Assessment

No live current-main Slack run was performed, but the source path is high-confidence: system-event route resolution still omits threadId, and Slack media/upload only threads when the tool context resolves a current thread timestamp.

## Solution Assessment

Yes, the best fix is to populate the generic effective reply route with the persisted threadId before Slack builds its tool context; a Slack-only fallback is less complete when the missing value never reaches the envelope.
