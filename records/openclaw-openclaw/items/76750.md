## Summary

Current main has the plugin-tool planning path needed for wildcard allowlists: explicit `tools.allow` entries, including `*`, are passed into plugin tool resolution, and enabled browser/memory plugins are selected from manifest `contracts.tools` before the final deny pass removes only denied tools.

## What This Changes

The issue requests that `tools.allow: ["*"]` plus `tools.deny` expose enabled plugin-backed tools such as `browser`, `memory_search`, and `memory_get`.

## Best Possible Solution

Keep the current main implementation and let the next release carry these plugin-tool availability fixes; any remaining catalog source-label cleanup can be tracked separately if users still need it.

## Reproduction Assessment

No, I do not have a high-confidence current-main failing reproduction. Source inspection shows current main now threads wildcard allowlists into plugin tool resolution and selects the browser and memory-core manifest tool contracts before deny filtering.

## Solution Assessment

Yes. The implemented path is the narrow maintainable fix: keep plugin tool ownership in the plugin resolver, honor manifest contracts and wildcard allowlists there, then apply the existing tool-policy deny pipeline.
