## Summary

Keep open. Current main has source-level evidence for the reported policy/catalog gap: plugin tools are materialized from profile/global allowlists, but the profile step is applied before the explicit `tools.allow` step, and `tools.catalog` also suppresses plugin-owned browser/memory entries by treating static catalog ids as existing tool-name conflicts.

## What This Changes

The issue asks for `tools.allow: ["*"]` plus `tools.deny` to expose enabled plugin tools such as `browser`, `memory_search`, and `memory_get` while still hiding denied tools.

## Best Possible Solution

Align policy and catalog behavior with the intended denylist model: explicit wildcard allow should load and retain all enabled plugin tools except denied entries, while `plugins.allow`, plugin disablement, and activation rules remain authoritative.

## Reproduction Assessment

No live high-confidence current-main reproduction was run, but the issue is source-reproducible: the reporter supplied concrete runtime output, and current code shows profile filtering before explicit wildcard allow plus catalog de-duping against static browser/memory ids.

## Solution Assessment

Yes, a narrow policy/catalog fix is the best path. Avoid broad plugin activation changes; the fix should preserve plugin allow/deny semantics and add regression coverage for wildcard allow plus deny with enabled browser and memory plugins.
