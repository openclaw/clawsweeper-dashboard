## What This Changes

The PR changes `src/gateway/session-utils.ts` so `loadSessionEntry` prefers a direct requested or canonical store row before falling back to freshest duplicate resolution.

## Best Possible Solution

Scope exact-key preference to explicit non-canonical archived or compound keys, preserve freshest duplicate selection for canonical lookups, add focused regression coverage, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes for the PR-introduced regression: source inspection and current tests show canonical requests must keep the freshest duplicate row, while the diff returns an exact canonical row first. The original sidebar symptom is source-plausible but still lacks real setup proof.

## Solution Assessment

No. Exact-first lookup is plausible for explicit archived or compound keys, but this implementation is too broad because canonical exact hits must not outrank fresher duplicate rows.
