## Summary

Keep this PR open for rework: the intended archived-session fix is plausible, but the current exact-first lookup regresses the existing freshest-duplicate session-row contract and still lacks real behavior proof.

## What This Changes

The PR changes `loadSessionEntry` to return an exact requested or canonical store row before falling back to the freshest `updatedAt` session match.

## Best Possible Solution

Scope exact-key preference to explicit non-canonical requested keys or the specific archived-session case, preserve freshest duplicate selection for canonical logical keys, add focused regression coverage, and include redacted real behavior proof.

## Reproduction Assessment

Yes for the PR regression: source and current tests show stale canonical plus fresh case-variant rows, while the diff would return `store[trimmedKey]` first. The original sidebar symptom is plausible from the code path but still lacks a focused test or real setup artifact.

## Solution Assessment

No. Exact-first lookup is plausible for explicit archived keys, but this implementation is too broad because canonical exact hits must not outrank fresher duplicate rows.
