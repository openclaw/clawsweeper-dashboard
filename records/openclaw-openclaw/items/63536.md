## Summary

This PR should stay open because the proposed exact-first lookup introduces a concrete regression: canonical lookups can return a stale exact row before the existing freshest duplicate selection runs. The branch also lacks after-fix real behavior proof, so it is not ready for merge or cleanup close.

## What This Changes

The PR changes `src/gateway/session-utils.ts` so `loadSessionEntry` returns an exact requested or canonical store row before falling back to the freshest matching session row.

## Best Possible Solution

Preserve freshest duplicate selection for canonical logical keys, add a narrowly scoped exact-key path only for explicit non-canonical archived or compound keys, and cover both behaviors with focused regression tests.

## Reproduction Assessment

Yes for the PR regression: the patch returns exact canonical rows before `freshestMatch`, while current tests define stale canonical plus fresher duplicate rows as returning the fresher entry. The original sidebar symptom still lacks focused current-main reproduction or real setup proof.

## Solution Assessment

No. Exact-first lookup is plausible for explicit archived keys, but this implementation is too broad because canonical exact hits must not outrank fresher duplicate rows.
