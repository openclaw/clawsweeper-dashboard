## Summary

Keep this PR open. Current main still lacks the exact non-canonical session-key preference, but the submitted shortcut would regress the canonical freshness behavior already protected by gateway tests.

## What This Changes

The PR changes `src/gateway/session-utils.ts` so `loadSessionEntry` returns an exact requested or canonical store entry before falling back to the freshest `updatedAt` match.

## Best Possible Solution

Keep the useful source work but repair it narrowly: prefer an exact on-disk requested key only when the normalized requested key differs from the resolved canonical key, preserve freshest duplicate selection for canonical logical keys, and cover the user-facing history path with regression tests.

## Reproduction Assessment

Yes. Existing current-main tests reproduce the PR regression with stale canonical and fresh case-variant rows; a new focused test should reproduce the original archived/non-canonical exact-key bug by seeding an older exact requested row plus a newer canonical row.

## Solution Assessment

No. The proposed approach is close but too broad because it lets canonical exact hits outrank fresher logical duplicates; the safer solution is to scope exact-key preference to explicit non-canonical requested keys and reuse that rule on the relevant history load surface.
