## What This Changes

The PR changes `loadSessionEntry` in `src/gateway/session-utils.ts` to return an exact requested or canonical store row before falling back to the freshest matching session row.

## Best Possible Solution

Scope exact-key preference to explicit non-canonical archived or compound keys, preserve freshest duplicate selection for canonical lookups, and add focused regression coverage plus real behavior proof.

## Reproduction Assessment

Yes for the PR regression: the diff returns exact canonical rows before `freshestMatch`, while current tests define stale canonical plus fresher duplicate rows as returning the fresher entry. The original sidebar symptom is not yet reproduced with a focused test or real setup artifact.

## Solution Assessment

No. Exact-first lookup is plausible for explicit archived keys, but this implementation is too broad because canonical exact hits must not outrank fresher duplicate rows.
