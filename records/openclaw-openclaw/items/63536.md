## Summary

Keep this PR open. Current main still resolves `loadSessionEntry` through the freshest matching store row, so the exact legacy/non-canonical key problem is not implemented on main, but the submitted patch is too broad and would regress the current freshest-duplicate behavior protected by tests.

## What This Changes

The PR changes `src/gateway/session-utils.ts` so `loadSessionEntry` returns an exact requested or canonical store entry before falling back to the freshest `updatedAt` match.

## Best Possible Solution

Request changes or create a narrow replacement that prefers an exact store entry only for explicit non-canonical requested keys, while preserving freshest duplicate/case-variant selection for canonical logical keys and adding regression coverage for both behaviors.

## Reproduction Assessment

Yes. Existing current-main tests provide a high-confidence reproduction of the PR regression by seeding a stale canonical row and a fresher case variant; a focused new test can reproduce the underlying requested exact-key bug with a non-canonical legacy key and a newer canonical row.

## Solution Assessment

No. The proposed approach is close but too broad because it treats canonical exact hits as higher priority than fresher logical duplicates; the safer fix is to scope exact-key preference to requested keys that differ from the resolved canonical key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve freshest canonical duplicate selection:** `src/gateway/session-utils.ts:407-415`
