## Summary

Keep this PR open, but it is not mergeable as-is: the exact-first lookup regresses the existing freshest duplicate session-row contract, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes gateway `loadSessionEntry` to return an exact requested or canonical store row before falling back to the freshest `updatedAt` session match.

## Best Possible Solution

Scope exact-key preference to explicit non-canonical requested keys, preserve freshest duplicate selection for canonical logical keys, add focused regression coverage, and include redacted real behavior proof.

## Reproduction Assessment

Yes for the PR regression: source and tests show stale canonical plus fresh case-variant rows, while the diff would return `store[trimmedKey]` first. The original sidebar bug is source-reproducible, but still lacks a focused test or real setup artifact.

## Solution Assessment

No. Exact-first lookup is plausible for explicit archived compound keys, but this implementation is too broad because canonical exact hits must not outrank fresher duplicate rows.
