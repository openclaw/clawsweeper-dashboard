## Summary

Keep open: the PR targets a plausible session-history bug, but the exact-first lookup is too broad and would return stale canonical rows ahead of fresher logical duplicates already covered by current main tests; it also lacks required real behavior proof.

## What This Changes

The branch changes `loadSessionEntry` to return an exact requested or canonical session-store row before falling back to the freshest `updatedAt` match.

## Best Possible Solution

Scope exact-key preference to explicit non-canonical archived keys, preserve freshest duplicate selection for canonical logical keys, add focused regression coverage, and include redacted real behavior proof.

## Reproduction Assessment

Yes for the PR regression: current tests seed stale canonical and fresh case-variant rows and the diff would return `store[trimmedKey]` first. The original archived-key bug is plausible from source inspection but is not yet proven with a focused test or real setup artifact.

## Solution Assessment

No. Exact-first lookup is close for explicit non-canonical archived keys, but the current implementation is too broad because canonical exact hits must not outrank fresher duplicate rows.
