## Summary

Keep open. The resolver-side fix is focused and the supplied proof covers the intended config behavior, but the diff adds a new path for top-level CDP endpoints into Chrome MCP attach failures while those errors still echo the endpoint verbatim.

## What This Changes

The PR applies normalized top-level `browser.cdpUrl` to the selected existing-session default profile when no profile-local endpoint is set, and adds resolver precedence tests.

## Best Possible Solution

Keep the resolver inheritance behavior, but redact Chrome MCP attach failure labels for inherited and profile-local CDP endpoints while preserving the new precedence rules.

## Reproduction Assessment

Yes. Source inspection shows current main leaves `defaultProfile: "user"` on an existing-session profile without the top-level endpoint, while the PR proof shows the patched resolver passes that endpoint into Chrome MCP args.

## Solution Assessment

No, not quite. The resolver location is the right boundary, but the patch should also redact sensitive CDP endpoints before they are included in Chrome MCP failure messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Redact inherited CDP endpoints before failure text:** `extensions/browser/src/browser/config.ts:349`
  - body: By copying top-level `browser.cdpUrl` onto an existing-session profile, this path can now feed tokenized endpoints into Chrome MCP attach failures. Those failures currently interpolate `options.browserUrl` verbatim, while existing tests and docs treat CDP URLs as secret-bearing, so failed Browserless or Browserbase attaches can expose credentials. Redact the endpoint before formatting the failure message.
  - confidence: 0.88
