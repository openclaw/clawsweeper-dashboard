## Summary

Keep open: this targets a real browser config routing bug and the supplied proof demonstrates the intended resolver/Chrome MCP argument behavior, but the patch creates a new path where sensitive top-level `browser.cdpUrl` values can be echoed unredacted in Chrome MCP attach failures.

## What This Changes

The PR applies normalized top-level `browser.cdpUrl` to the selected existing-session default profile and adds resolver tests for inheritance and explicit per-profile precedence.

## Best Possible Solution

Keep the resolver-side inheritance, but merge only after Chrome MCP attach failures redact inherited and profile-local CDP endpoints while preserving the new precedence rules.

## Reproduction Assessment

Yes. Source inspection shows current main lets `defaultProfile: "user"` select an existing-session profile whose `cdpUrl` comes only from the profile-local field, while the PR proof shows the patched resolver passes the top-level endpoint into Chrome MCP args.

## Solution Assessment

No, not quite. The resolver location and behavior are appropriate, but the patch should also redact sensitive CDP endpoints before they are surfaced in Chrome MCP failure messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Redact inherited CDP endpoints before failure text:** `extensions/browser/src/browser/config.ts:349`
  - body: By copying top-level `browser.cdpUrl` onto an existing-session profile, this path now feeds tokenized endpoints into Chrome MCP. Attach failures currently format `options.browserUrl` verbatim, and CDP URLs are treated as sensitive elsewhere, so a failed Browserless/Browserbase attach can expose credentials in CLI or Gateway errors. Please redact the endpoint before formatting Chrome MCP failure messages.
  - confidence: 0.88
