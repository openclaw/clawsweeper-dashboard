## Summary

Keep open: current main and the latest release still leave the top-level `browser.cdpUrl` out of the selected `user` existing-session profile, while this focused PR adds the resolver inheritance and tests. The earlier endpoint-redaction concern is covered on current main, so I do not see a blocking correctness finding now.

## What This Changes

The PR copies normalized top-level `browser.cdpUrl` into the selected existing-session default profile when no profile-local endpoint is set, and adds regression tests for inheritance and precedence.

## Best Possible Solution

Land this resolver fix with current Chrome MCP endpoint redaction preserved, then let the linked browser issue close after merge.

## Reproduction Assessment

Yes. Source inspection shows `defaultProfile: "user"` selects an existing-session profile whose `cdpUrl` remains empty, so Chrome MCP falls back to `--autoConnect` instead of using the configured endpoint.

## Solution Assessment

Yes. Applying the normalized top-level endpoint only to the selected existing-session default profile without a profile-local endpoint is the narrowest maintainable fix and preserves explicit profile precedence.
