## Summary

Keep open: this PR addresses a valid source-reproducible browser config bug, but it should not merge until inherited CDP endpoints are redacted from Chrome MCP attach failure text.

## What This Changes

The branch copies top-level `browser.cdpUrl` onto the selected existing-session default profile when no profile-local `cdpUrl` exists, and adds resolver precedence tests.

## Best Possible Solution

Land the resolver inheritance with Chrome MCP attach error redaction and regression coverage for secret-bearing CDP endpoint labels.

## Reproduction Assessment

Yes. Source inspection of current main shows `defaultProfile: "user"` selects an existing-session profile without the top-level endpoint, while the PR proof shows the patched resolver feeds that endpoint into Chrome MCP args.

## Solution Assessment

No, not quite. The resolver boundary is the right place to preserve the configured endpoint, but the patch also needs to redact sensitive CDP URLs before formatting Chrome MCP failure messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Redact inherited CDP endpoints before failure text:** `extensions/browser/src/browser/config.ts:349`
  - body: By copying top-level `browser.cdpUrl` onto an existing-session profile, this path can now feed tokenized endpoints into Chrome MCP attach failures. Those failures currently interpolate `options.browserUrl` verbatim, while existing tests and docs treat CDP URLs as secret-bearing, so failed Browserless or Browserbase attaches can expose credentials.
  - confidence: 0.88
