## Summary

Keep open: the PR addresses a source-reproducible browser config bug, but it should not merge until Chrome MCP attach failure text redacts inherited CDP endpoints that may contain credentials.

## What This Changes

The PR makes the selected existing-session default browser profile inherit top-level `browser.cdpUrl` when no profile-local endpoint exists, with resolver tests for inheritance and precedence.

## Best Possible Solution

Keep the resolver-side inheritance, but merge only after Chrome MCP attach failures redact inherited and profile-local CDP endpoints while preserving explicit profile precedence.

## Reproduction Assessment

Yes. Source inspection on current main shows `defaultProfile: "user"` selects an existing-session profile whose endpoint comes only from profile-local `cdpUrl`, so top-level `browser.cdpUrl` falls through to Chrome MCP auto-connect behavior.

## Solution Assessment

No, not quite. The resolver location and precedence rule are the narrow maintainable fix, but the PR should also redact sensitive CDP endpoints before user-visible Chrome MCP failure messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Redact inherited CDP endpoints before failure text:** `extensions/browser/src/browser/config.ts:349`
  - body: By copying top-level `browser.cdpUrl` onto an existing-session profile, this path now feeds tokenized endpoints into Chrome MCP. Attach failures format `options.browserUrl` verbatim, and CDP URLs are treated as sensitive elsewhere, so a failed Browserless/Browserbase attach can expose credentials in CLI or Gateway errors. Redact the endpoint before formatting failure text and cover it.
  - confidence: 0.88
