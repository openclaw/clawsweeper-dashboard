## Summary

Keep open: the PR fixes a real browser config bug and includes convincing after-fix live output, but it still needs one blocking security repair because inherited CDP endpoints can reach Chrome MCP attach error text without redaction.

## What This Changes

The PR copies normalized top-level `browser.cdpUrl` onto the selected existing-session default browser profile when that profile has no profile-local endpoint, with precedence tests.

## Best Possible Solution

Keep the resolver inheritance behavior, but redact CDP endpoints in Chrome MCP failure labels and add focused coverage for inherited and profile-local credential URLs before merge.

## Reproduction Assessment

Yes. Source inspection shows current main leaves `defaultProfile: "user"` on an existing-session profile without the top-level endpoint, and the PR body's live output shows the patched resolver passes that endpoint into Chrome MCP args.

## Solution Assessment

No, not quite. The resolver boundary is appropriate, but the patch should also redact sensitive CDP endpoints before they are included in Chrome MCP failure messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Redact inherited CDP endpoints before failure text:** `extensions/browser/src/browser/config.ts:349`
  - body: By copying top-level `browser.cdpUrl` onto an existing-session profile, this path can now feed tokenized endpoints into Chrome MCP attach failures. Those failures still format `options.browserUrl` verbatim, while current docs and redaction tests treat CDP URLs as secret-bearing, so failed Browserless or Browserbase attaches can expose credentials. Redact the endpoint before composing the failure label.
  - confidence: 0.88
