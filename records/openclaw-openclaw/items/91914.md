## Label Justifications

- `P1`: The issue describes a broken external-channel workflow where a follow-up message can stop an active same-session reply from reaching the originating channel.
- `impact:message-loss`: The reported symptom is assistant output remaining visible in the session UI while later source delivery is suppressed before the plugin/channel receives it.
- `impact:session-state`: The root cause is a mismatch between same-session admission state and foreground delivery generation state.

## Summary

Keep open: current main and the latest release still create the foreground reply fence at buffered-dispatch entry before deeper same-session admission, and the linked lazy-fence fix paths are open rather than merged.

## What This Changes

The issue asks that queued same-session inbound messages not make an active source reply stale before the queued turn is actually admitted or starts visible delivery.

## Best Possible Solution

Land a corrected lazy foreground-fence fix, such as https://github.com/openclaw/openclaw/pull/91963 or https://github.com/openclaw/openclaw/pull/93977 after review, with tests preserving both queued-turn delivery and newer-visible stale suppression.

## Reproduction Assessment

Yes at source level: current main creates the foreground fence before same-session reply admission, and the guard can cancel older source delivery behind a newer generation. I did not run a live channel replay in this read-only review.

## Solution Assessment

Yes in direction: deferring foreground fence creation until visible/source delivery begins is the narrowest maintainable fix. It still needs maintainer review through the open linked PRs because stale-suppression invariants must be preserved.
