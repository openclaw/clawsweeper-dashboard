## Label Justifications

- `P1`: The issue describes a broken external-channel workflow where user follow-up messages can cause an active reply to stop delivering.
- `impact:message-loss`: The reported symptom is assistant output visible in the session UI but missing from the originating plugin/channel.
- `impact:session-state`: The root cause is a mismatch between same-session admission state and foreground delivery generation state.

## Summary

Keep open: current main and v2026.6.6 still create the foreground reply fence before same-session reply admission, and the linked closing fix at https://github.com/openclaw/openclaw/pull/91963 is still open rather than merged.

## What This Changes

The issue asks OpenClaw to keep an active same-session source delivery fresh until a queued follow-up turn is actually admitted or begins visible/source delivery.

## Best Possible Solution

Keep this issue open, repair and land https://github.com/openclaw/openclaw/pull/91963 or an equivalent lazy-fence fix that preserves newer-visible stale suppression, then close from the merged implementation.

## Reproduction Assessment

Yes, at source level: current main creates the foreground fence before reply-operation admission, and the guard can cancel older source delivery behind a newer generation while the newer same-session turn is still waiting. I did not run a live channel scenario in this read-only review.

## Solution Assessment

Yes in direction: deferring foreground fence creation until visible/source delivery begins is the narrowest maintainable fix. The linked PR still needs to preserve the existing newer-visible stale-suppression invariant before it is the best landable implementation.
