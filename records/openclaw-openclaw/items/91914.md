## Summary

Keep open: current main and v2026.6.8 still create the foreground reply fence before deeper same-session admission, and the linked fix PRs are open rather than merged.

## What This Changes

The issue reports that a queued same-session inbound can advance foreground delivery freshness early and stop an active source reply from reaching the originating plugin/channel.

## Best Possible Solution

Land a corrected lazy foreground-fence fix, such as the linked lazy-fence PR path after review, with tests preserving both queued-turn delivery and newer-visible stale suppression.

## Reproduction Assessment

Yes at source level: current main creates the foreground fence before same-session reply admission, and the guard can cancel older source delivery behind a newer generation. I did not run a live channel replay in this read-only review.

## Solution Assessment

Yes in direction: deferring foreground fence creation until visible/source delivery begins is the narrowest maintainable fix. The open PRs still need maintainer review because the final implementation must preserve stale-suppression invariants.
