## Label Justifications

- `P1`: The issue describes a broken external-channel workflow where a follow-up message can stop an active same-session reply from reaching the originating channel.
- `impact:session-state`: The root cause is a mismatch between same-session admission state and foreground delivery generation state.
- `impact:message-loss`: The reported symptom is assistant output remaining visible in the session UI while later source delivery is suppressed before the plugin/channel receives it.

## Summary

Keep open: current main and v2026.6.8 still create the foreground reply fence at buffered-dispatch entry before deeper same-session admission, and the linked lazy-fence repair PRs remain open rather than merged.

## What This Changes

The issue reports that a queued same-session inbound can advance foreground reply freshness before it is admitted, causing an active turn's later plugin/channel source delivery to be suppressed.

## Best Possible Solution

Land a corrected lazy foreground-fence fix through the existing candidate PR path, preserving same-session serialization and newer-visible stale-suppression invariants with focused freshness coverage.

## Reproduction Assessment

Yes, at source level: current main creates the foreground fence before same-session reply admission, and the guard can cancel older source delivery behind a newer generation. I did not run a live external-channel replay in this read-only review.

## Solution Assessment

Yes in direction: deferring foreground fence creation until visible/source delivery begins is the right layer for queued-before-admission turns. The final implementation still needs maintainer review because it must preserve newer-visible stale suppression while fixing queued-turn loss.
