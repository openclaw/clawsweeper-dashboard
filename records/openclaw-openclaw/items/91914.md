## Label Justifications

- `P1`: The reported bug can make active external-channel replies stop delivering when a user sends a same-session follow-up during a long response.
- `impact:message-loss`: The core symptom is assistant output visible in the session UI but not delivered to the originating plugin/channel.
- `impact:session-state`: The suspected failure is a mismatch between same-session admission state and foreground delivery generation state.

## Summary

Keep open: current main still creates the foreground reply fence before same-session reply admission, so a queued inbound can become the newer foreground generation while the active turn is still delivering. I found no merged fix or canonical duplicate that already owns this exact admission/fencing mismatch.

## What This Changes

The issue asks OpenClaw to keep delivering an active channel turn to its originating plugin/channel when a second same-session inbound is only queued behind it.

## Best Possible Solution

Align foreground generation advancement with actual reply-operation admission, preserving stale-delivery suppression only for newer turns that are actually admitted or visibly delivered.

## Reproduction Assessment

Yes, at source level: start a buffered channel dispatch, enter a second dispatch with the same fence/session key before the first finishes delivery, and the second dispatch can create a newer foreground generation before `admitReplyTurn` owns the session lane. I did not execute a live plugin reproduction in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to defer or gate foreground generation advancement until the queued turn is admitted, rather than adding a plugin-specific workaround or a new config option. The fix should also keep coverage for the existing accepted-newer-turn stale suppression behavior.
