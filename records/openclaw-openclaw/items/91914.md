## Label Justifications

- `P1`: The issue describes a broken user-facing agent/channel workflow where a follow-up message can stop an active reply from reaching the originating channel.
- `impact:session-state`: The root cause is a mismatch between same-session admission state and foreground delivery generation state.
- `impact:message-loss`: The symptom is assistant output remaining visible in the session UI while later source delivery is suppressed before the plugin/channel receives it.

## Summary

Keep open: current main and v2026.6.9 still create the foreground reply fence before same-session admission, and the open candidate PRs that target this delivery-loss race have not merged.

## What This Changes

Prevent a queued same-session inbound from advancing foreground reply freshness before it is admitted, so the active turn continues delivering to the originating plugin/channel.

## Best Possible Solution

Land a reviewed foreground-fence fix through the existing PR path that aligns fence creation with admitted/visible delivery while preserving newer-visible stale suppression.

## Reproduction Assessment

Yes, at source level: current main creates the foreground fence before same-session reply admission, and the guard can cancel older source delivery behind a newer visible generation. I did not run a live external-channel replay in this read-only review.

## Solution Assessment

Yes in direction: deferring foreground fence creation until visible/source delivery begins is the right layer for queued-before-admission turns. The landable fix still needs maintainer review to preserve newer-visible stale suppression and focused freshness coverage.
