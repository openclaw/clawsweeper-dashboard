## Label Justifications

- `P1`: The issue describes a broken external-channel workflow where a user follow-up can prevent an active reply from reaching the originating channel.
- `impact:message-loss`: The reported symptom is assistant output visible in the session UI but missing from the plugin/channel delivery path.

## Summary

Keep open: current main and v2026.6.6 still create the foreground reply fence before same-session admission, and the linked closing PR is still open rather than merged.

## What This Changes

The issue reports that a queued same-session inbound can make an active source delivery stale before the queued turn is admitted.

## Best Possible Solution

Keep this issue as the canonical same-session queued-turn tracker, review and land https://github.com/openclaw/openclaw/pull/91963 if exact-head validation remains clean, then close from the merged implementation.

## Reproduction Assessment

Yes, at source level. Current main starts the foreground fence before dispatchReplyFromConfig/admitReplyTurn admission, and the freshness guard can hold or cancel older source delivery behind a newer queued generation.

## Solution Assessment

Yes. Deferring fence creation until the first visible/source delivery attempt is the narrowest maintainable direction because it aligns freshness with the delivery behavior it governs without adding channel-specific policy or a config knob.
