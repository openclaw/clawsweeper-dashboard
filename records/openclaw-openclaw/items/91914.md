## Label Justifications

- `P1`: The issue describes a broken channel workflow where user follow-up messages can cause active external-channel replies to stop delivering.
- `impact:message-loss`: The reported symptom is assistant output visible in the session UI but missing from the originating plugin/channel.
- `impact:session-state`: The root cause is a mismatch between same-session admission state and foreground delivery generation state.

## Summary

Keep open: current main and the latest release still create the foreground reply fence at buffered-dispatch entry, before same-session reply admission, and the linked fix PR that closes this issue is still open rather than merged.

## What This Changes

Queued same-session inbounds should not advance foreground freshness before admission in a way that makes the active turn stop delivering to its originating channel/plugin.

## Best Possible Solution

Keep this issue open as the canonical tracker, review and land the linked lazy-foreground-fence fix if exact-head checks stay clean, then close from the merged implementation.

## Reproduction Assessment

Yes, at source level. Current main creates the foreground fence before reply-operation admission, and the fence guard can hold or cancel older source delivery behind a newer generation while the newer same-session turn is still waiting.

## Solution Assessment

Yes. Deferring foreground fence creation until visible/source delivery begins is the narrowest maintainable direction because it aligns freshness with the delivery behavior it governs without adding channel-specific policy or a new config option.
