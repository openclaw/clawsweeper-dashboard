## Label Justifications

- `P1`: The reported bug can make active external-channel replies stop delivering when a user sends a same-session follow-up during a long response.
- `impact:message-loss`: The core symptom is assistant output visible in the session UI but not delivered to the originating plugin/channel.
- `impact:session-state`: The suspected failure is a mismatch between same-session admission state and foreground delivery generation state.

## Summary

Keep open: current main still starts foreground reply fencing before the same-session reply turn is admitted, and source inspection shows that a queued second inbound can become a newer foreground generation while the first turn is still delivering. I found no merged fix or canonical duplicate that already owns this exact admission/fencing mismatch.

## What This Changes

The issue asks for queued same-session inbound turns not to make an active turn's originating channel delivery stale before the queued turn is actually admitted to run.

## Best Possible Solution

Align foreground reply fence generation with actual reply-operation admission, while preserving the existing stale-reply suppression behavior for newer turns that are truly admitted or accepted.

## Reproduction Assessment

Yes, at source level: start a buffered channel dispatch, enter a second dispatch with the same fence/session key before the first finishes delivery, and the second dispatch can create a newer foreground generation before `admitReplyTurn` owns the session lane. I did not execute a live plugin reproduction in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to defer or gate foreground generation advancement until the queued turn is admitted, rather than adding a plugin-specific workaround or a new config option. The PR should also keep tests for the existing accepted-newer-turn stale suppression behavior.
