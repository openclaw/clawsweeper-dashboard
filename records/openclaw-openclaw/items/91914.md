## Label Justifications

- `P1`: The issue describes a broken external-channel workflow where a follow-up message can stop an active same-session reply from reaching the originating channel.
- `impact:session-state`: The root cause is a mismatch between same-session admission state and foreground delivery generation state.
- `impact:message-loss`: The reported symptom is assistant output remaining visible in the session UI while later source delivery is suppressed before the plugin/channel receives it.

## Summary

Keep open: current main and v2026.6.8 still create the foreground reply fence at buffered-dispatch entry before deeper same-session admission, and the linked pull requests are open candidate fixes rather than merged implementation.

## What This Changes

The issue requests that queued same-session inbound messages stop making an active source reply stale before the queued turn is admitted to run.

## Best Possible Solution

Land a corrected lazy foreground-fence repair through the existing candidate PR path, preserving newer-visible stale suppression and adding focused freshness coverage.

## Reproduction Assessment

Yes, at source level: current main creates the foreground fence before same-session reply admission and the guard can cancel older source delivery behind a newer generation. I did not run a live external-channel replay in this read-only review.

## Solution Assessment

Yes in direction: deferring foreground fence creation until visible/source delivery begins is the right layer. The best fix still has to preserve newer-visible stale suppression, so the open candidate PRs need review or repair rather than closing this issue.
