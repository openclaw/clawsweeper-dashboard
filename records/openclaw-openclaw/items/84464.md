## Best Possible Solution

Keep this open and route it to the Discord plugin/runtime owners for a live current-main reproduction, then fix the READY observation or reconnect path in `extensions/discord` with regression coverage for startup wait, opened-but-not-ready sockets, and external plugin install behavior.

No high-confidence current-main reproduction from this review. The issue body provides strong v2026.5.18 runtime evidence and raw-WebSocket control proof, but I did not run a live Discord bot setup and the old registration-race explanation is not enough for current main.

No, the suggested `await registerClient` direction is not the best complete fix by itself because current startup already awaits OpenClaw's gateway registration promise and assigns the client before metadata fetch.
