## Summary

Keep open. The PR addresses a valid iMessage RPC stdin EPIPE bug in the right owner module, but the new stdin error path can bypass normal client teardown by clearing the child reference and resolving the close promise before the child process is actually cleaned up.

## What This Changes

The PR adds iMessage RPC stdin error and write-callback handling, focused regression tests for those write failures, and an Unreleased changelog entry.

## Best Possible Solution

Keep the local iMessage RPC stdin handling approach, but revise the stdin error path so it either tears down the captured child/readline immediately or leaves the child close state for `stop()` to clean up, then merge after focused tests and CI are green.

## Reproduction Assessment

Yes. Current main has an unguarded `child.stdin.write(line)` in the iMessage RPC request path, and the linked issue describes the EPIPE failure; the PR's cleanup regression is reproducible by emitting a stdin error without a child `close` event and then calling `client.stop()`.

## Solution Assessment

No, not as currently written. The PR is the right narrow fix direction, but the new error handler should not mark the client closed and drop the child reference without also closing the reader and terminating or preserving the child process for cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Tear down the child on stdin errors:** `extensions/imessage/src/client.ts:109-111`
  - body: The new stdin error handler clears `this.child` and resolves the close promise, but it never closes the readline interface or terminates the captured child. Callers such as send, probe, and monitor still call `client.stop()` in `finally`; after this assignment `stop()` returns immediately, so a process that closed only fd 0 can keep running with its stdout reader left open. Keep the child available for `stop()` or close/kill it in this handler.
  - confidence: 0.83
