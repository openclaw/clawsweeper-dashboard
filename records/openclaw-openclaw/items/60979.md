## Summary

Keep open. Current main still lacks a caller-specified ACP `sessions_spawn` delivery target: the tool exposes `streamTo: "parent"` and thread/current-conversation ACP binding paths, but not the requested `delivery` object for routing a run to a chosen channel conversation or topic.

## What This Changes

Request a `delivery` option for `sessions_spawn` ACP runs so spawned ACP output can stream or announce directly to a specified Zulip/Discord-style channel topic without the parent agent relaying it.

## Best Possible Solution

Define a generic, policy-checked ACP `sessions_spawn` delivery contract that reuses `DeliveryContext`, current conversation/session binding adapters, and task-completion delivery rather than adding a parallel channel-specific path. The finished design should document streaming versus completion semantics and enforce account/channel/thread authorization before any outbound post.

## Reproduction Assessment

Yes. This has a high-confidence static reproduction path: inspect the `sessions_spawn` schema and ACP dispatch path on current main, then observe that the requested `delivery` field is absent while direct channel target aliases are explicitly rejected and run-mode ACP dispatch keeps `deliver` false.

## Solution Assessment

No, not as a direct cron-schema copy alone. The requested direction is plausible, but the maintainable solution needs a generic authorized delivery design that fits existing binding and task-completion paths.
