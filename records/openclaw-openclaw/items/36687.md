## Summary

Current main contains guardrails for the observed WebChat/TUI stale-route leakage symptom, but the narrower claim that the 2026.3.2 update path reset or dropped an explicit `session.dmScope` value is still unverified. Because that remaining claim is privacy-sensitive and maintainer comments deliberately kept it separate from the merged routing fixes, this should stay open for targeted maintainer follow-up.

## What This Changes

The issue reports that upgrading to 2026.3.2 reset effective `session.dmScope` to `main`, causing WebChat/TUI replies to leak to an iMessage contact.

## Best Possible Solution

Keep the shipped stale-route guardrails, then verify the 2026.3.2 update/config-write path with before/after config evidence; if an explicit isolated `dmScope` can be dropped, repair that narrow preservation or doctor path with regression coverage.

## Reproduction Assessment

No: the WebChat stale-route symptom has source and test coverage, but the remaining update/reset allegation lacks config snapshots, migration logs, or a current-main upgrade reproduction showing `session.dmScope` being changed or dropped.

## Solution Assessment

Unclear: the merged routing guardrails are the right narrow fix for the observed leakage symptom, but the remaining reset claim should be solved by proving and repairing a specific config preservation path rather than broadly changing the documented omitted-value fallback without maintainer approval.
