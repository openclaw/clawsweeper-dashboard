## Summary

Keep open: current main has shipped guardrails for the observed WebChat/TUI stale-route leakage symptom, but the narrower claim that the 2026.3.2 update reset or dropped `session.dmScope` remains unverified and is not implemented as a tested migration/doctor repair.

## What This Changes

The issue asks OpenClaw to verify and fix a reported 2026.3.2 upgrade path where `session.dmScope` was reset or effectively treated as `main`, causing WebChat/TUI replies to leak to iMessage.

## Best Possible Solution

Keep the shipped stale-route guardrails and verify the specific 2026.3.2 update/config path. If an update, onboarding, doctor, or config-write flow can drop an existing isolated `session.dmScope`, fix that narrow path with regression coverage; if omitted `dmScope` intentionally remains `main`, close only after maintainer judgment and documented guidance.

No: we do not have a high-confidence reproduction for the update/reset part of the issue. Current source inspection confirms the documented missing-`dmScope` fallback and the shipped route guardrails, but the report still lacks config snapshots or migration logs showing an explicit value being reset during the 2026.3.2 update.

## Solution Assessment

Unclear: the merged routing fixes are the right narrow solution for the observable stale-route leakage symptom, but the proposed unmerged default flip in #36864/#38338 would be a broader compatibility change. If the reset path is real, a targeted config migration/doctor repair that preserves explicit `dmScope` is safer than changing every omitted-value fallback.
