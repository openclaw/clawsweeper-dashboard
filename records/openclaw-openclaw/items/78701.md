## Summary

Keep this PR open: the additive CLI JSON feature is plausible, but the proposed helper can report legacy `lastStatus`-only jobs as `idle`, and the external PR still needs real behavior proof plus the required changelog entry.

## What This Changes

The PR adds a cron JSON enrichment helper and routes `cron list --json` and `cron show <id> --json` through it so job objects get a computed top-level `status`.

## Best Possible Solution

Use one shared cron status derivation for human and JSON output, preserving `lastRunStatus ?? lastStatus ?? "idle"`, then cover list/show JSON with focused tests, add the changelog entry, and attach redacted real CLI output proof.

## Reproduction Assessment

Yes. Source inspection of current main shows `cron list --json` and `cron show <id> --json` print raw gateway job data directly, while human output computes status separately.

## Solution Assessment

No. The PR is close, but the best fix should share or match the existing status derivation and preserve the deprecated `lastStatus` alias before adding the JSON field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve the legacy lastStatus fallback:** `src/cli/cron-cli/shared.ts:65`
  - body: `CronJobState.lastRunStatus` is preferred, but current human status still falls back to `state.lastStatus`, and persisted or legacy jobs can have only that deprecated alias. With this helper, a completed, errored, or skipped job that only has `lastStatus` gets JSON `status: "idle"`, so `--json` can disagree with the table it is meant to mirror. Use the shared status function or fall back to `state.lastRunStatus ?? state.lastStatus ?? "idle"`.
  - confidence: 0.86
