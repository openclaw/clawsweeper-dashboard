## Summary

Keep open: current main still has only private Claude live-session internals for stream-json stdin writes, while the requested external-process wake injection API and optional UDS listener remain a security-sensitive product/API decision. The related wake rendering, durability, and telemetry issues reinforce that this is an unresolved wake-delivery feature cluster rather than an implemented core behavior.

## What This Changes

The issue requests a supported live-session API, optionally backed by a local Unix-domain socket, for external orchestrators to inject user-side stream-json messages into the active Claude TUI session.

## Best Possible Solution

Design a narrow, authenticated, opt-in Gateway or live-runtime wake-delivery contract with explicit busy, replay, consumed-state, TUI visibility, and audit semantics, or document that heartbeat/system-event paths remain the supported orchestration mechanism.

## Reproduction Assessment

No for the reported `openclaw agent --message` hang: the issue does not include runnable steps, logs, or a current-main failure transcript. The missing injection API itself is source-confirmed by the private live-session registry and internal stdin serializer.

## Solution Assessment

Unclear: reusing the existing stream-json serializer is technically plausible, but exposing it through an API or UDS listener needs maintainer product and security review first. A safer solution would be an authenticated, opt-in runtime contract with explicit delivery, busy, replay, visibility, and audit semantics.
