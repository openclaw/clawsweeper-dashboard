## Summary

Keep open: current main and the latest release still only have private Claude live-session stdin plumbing, while the requested external-process injection API and optional UDS listener remain a security-sensitive product/API decision.

## What This Changes

Expose a supported API, and optionally an explicitly enabled local UDS listener, so external same-machine orchestrators can inject user messages into an active Claude live session through the existing stream-json stdin path.

## Best Possible Solution

Design a narrow, authenticated, opt-in Gateway or live-runtime contract for wake delivery, then align wake-inbox durability and wake telemetry with that approved contract.

## Reproduction Assessment

No for the reported hang: the issue lacks runnable steps, logs, version details, or a current-main repro for `openclaw agent --message` hanging. The missing supported injection API itself is source-confirmed, but that is a feature/API gap rather than a strict bug reproduction.

## Solution Assessment

Unclear: reusing the existing stream-json serializer is plausible, but exposing it through an API or UDS listener needs a product and security decision first. A generic authenticated runtime/Gateway contract is safer than exposing a private Claude-only helper ad hoc.
