## Summary

Keep open: current main still has only a private Claude live-session registry and private stream-json stdin serializer. Existing Gateway `sessions.send`/`sessions.steer` and wake paths are adjacent, but they do not expose a supported external-process API that writes a user message into the active Claude live child, and the requested UDS/API surface needs product and security review.

## What This Changes

The issue requests a supported live-session API, optionally backed by an opt-in Unix-domain socket, for external processes to inject stream-json user messages into an active Claude session.

## Best Possible Solution

Design a narrow, authenticated, opt-in Gateway or core runtime contract for live-session wake delivery, or explicitly keep existing `sessions.steer`/wake/heartbeat paths as the supported orchestrator surface instead of exposing a Claude-specific private stdin hook.

## Reproduction Assessment

No high-confidence reproduction exists for the reported `openclaw agent --message` hang because the issue lacks exact steps and logs. Source inspection does confirm the requested supported live-session injection API and UDS listener are absent on current main.

## Solution Assessment

Unclear as proposed. Reusing the existing serializer is plausible, but exposing it through a local UDS/API is a new security-sensitive public surface and should first settle auth, opt-in, replay, busy-turn, and Gateway versus Claude-specific ownership.
