## Summary

Keep open: current main still has only a private Claude live-session registry and internal stream-json stdin write path, while the requested external-process user-message injection API remains a new security-sensitive product surface. The related wake rendering and inbox-durability issues depend on this contract rather than superseding it.

## What This Changes

The issue requests a supported live-session API, and optionally a local UDS listener, for external processes to inject stream-json user messages into an active Claude live session.

## Best Possible Solution

Design a narrow, authenticated, opt-in Gateway or live-runtime contract for orchestrator wake delivery, or explicitly document that system events, heartbeat, and plugin next-turn injections are the supported path.

## Reproduction Assessment

No for the reported hang: the issue lacks runnable steps, logs, or a current-main failing path for `openclaw agent --message`. Source inspection does confirm the requested injection API is not present on current main, so the feature gap is clear.

## Solution Assessment

Unclear: reusing the existing serializer is plausible, but exposing it as an API or UDS listener needs a product and security decision first. The safer direction is a documented, authenticated, opt-in runtime contract rather than exposing a private Claude-only helper ad hoc.
