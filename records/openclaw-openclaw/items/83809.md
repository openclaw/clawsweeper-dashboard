## Summary

Keep open: current main has the internal Claude live-session registry and stream-json stdin serializer, but it does not expose a supported API or local IPC surface for external user-message injection. This is a new security-sensitive orchestration feature, not an implemented fix or a ClawHub-only item, because the requested surface reaches a private core live-session process.

## What This Changes

The issue asks for a supported live-session user-message injection API, optionally backed by an explicitly enabled Unix-domain socket, so external orchestrators can wake an active Claude stream-json session.

## Best Possible Solution

Design a narrow, authenticated, opt-in core or Gateway runtime contract for live-session injection, or explicitly keep system events and heartbeat delivery as the supported orchestrator wake path.

## Reproduction Assessment

No; this is primarily a feature/API request rather than a reproducible bug. Source inspection confirms the internal live-session path exists, but the reported `openclaw agent --message` hang does not include enough steps or logs to reproduce as a current-main defect.

## Solution Assessment

Unclear; reusing the existing serializer is plausible, but exposing it through a UDS listener is a new security-sensitive public surface. The maintainable path should first settle auth, opt-in configuration, busy-turn behavior, and whether this belongs in Gateway/runtime rather than a Claude-specific helper.
