## Summary

Keep open: current main still has the private Claude live-session registry and internal stream-json stdin write path, but no supported external-process injection API; the existing Gateway send/steer and heartbeat wake paths are related but not equivalent.

## What This Changes

The issue requests a supported, opt-in API, possibly backed by a local UDS listener, for external processes to inject user-side wake messages into an active Claude live stream-json session.

## Best Possible Solution

Design a narrow, authenticated, opt-in Gateway or live-runtime contract for orchestrator wake delivery, or explicitly keep heartbeat/system events as the supported path.

## Reproduction Assessment

No for the reported hang: the issue lacks runnable steps, logs, and a current-main live reproduction. Yes for the feature gap: source inspection confirms current main has no supported external live-session injection API.

## Solution Assessment

Unclear: reusing the existing serializer is plausible, but exposing it through an API or UDS listener needs product and security design first. A documented authenticated opt-in runtime contract is safer than exporting a private Claude-only helper ad hoc.
