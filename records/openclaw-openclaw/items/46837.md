## Summary

Keep open: current main still lacks the requested doctor warning, and the PR is useful, but the patch has a source-backed correctness bug that can make the new diagnostic disagree with the memory runtime.

## What This Changes

The branch adds an `openclaw doctor` memory-search warning for missing `memorySearch.extraPaths` entries and unit tests for missing, existing, and tilde-prefixed paths.

## Best Possible Solution

Land a current-main version of the doctor diagnostic that mirrors `normalizeExtraMemoryPaths`, covers stale absolute and workspace-relative paths, and includes the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `extraPaths: ["~/notes"]`, `$HOME/notes` present, and `<workspace>/~/notes` missing, this PR suppresses the warning while the runtime still normalizes and indexes the workspace-relative path.

## Solution Assessment

No. The diagnostic is the right direction, but the implementation should reuse or mirror `normalizeExtraMemoryPaths` instead of adding a separate `resolveUserPath` tilde-expansion contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Align extraPaths checks with runtime resolution:** `src/commands/doctor-memory-search.ts:271-273`
  - body: The new helper treats `~` entries as home-relative via `resolveUserPath`, but the memory runtime resolves every non-absolute `extraPaths` entry under the workspace. With `extraPaths: ["~/notes"]`, doctor can report healthy when `$HOME/notes` exists even though indexing looks under `<workspace>/~/notes`, hiding the stale path this warning is meant to catch.
  - confidence: 0.9
