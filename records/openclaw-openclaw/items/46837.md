## Summary

Keep open: current main still lacks the requested doctor warning, but this PR should not merge as-is because its `~` handling can make the new health check disagree with the memory runtime.

## What This Changes

The PR adds an `openclaw doctor` warning for configured memory-search `extraPaths` that do not exist, plus targeted tests for present and missing paths.

## Best Possible Solution

Land a rebased diagnostic that checks missing `extraPaths` with the same absolute/workspace-relative normalization used by the memory runtime, with regression coverage for `~` as workspace-relative and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows the patch resolves `~` paths differently from `normalizeExtraMemoryPaths`, which is the helper used by runtime `extraPaths` reads.

## Solution Assessment

No. The diagnostic direction is good, but the implementation should reuse or exactly mirror runtime path normalization instead of adding home-relative behavior for this setting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Match extraPaths checks to runtime path normalization:** `src/commands/doctor-memory-search.ts:267-270`
  - body: The new check resolves `~/notes` with `resolveUserPath`, but runtime indexes `extraPaths` through `normalizeExtraMemoryPaths`, where non-absolute values are resolved against the workspace. If `$HOME/notes` exists, doctor will suppress the warning while runtime still looks at `<workspace>/~/notes` and indexes nothing. Use the same normalization helper or workspace-relative rule and cover this case in the test.
  - confidence: 0.88
