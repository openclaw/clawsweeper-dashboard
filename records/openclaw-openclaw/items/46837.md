## Summary

Keep this PR open, but it is not merge-ready: the diagnostic fills a real doctor gap, yet the current patch can hide a missing-path warning for builtin memory `extraPaths` that start with `~`, and it lacks after-fix real behavior proof.

## What This Changes

The PR adds an `openclaw doctor` warning for configured `agents.defaults.memorySearch.extraPaths` entries that do not exist, with focused tests for absolute and tilde paths.

## Best Possible Solution

Keep the diagnostic, but resolve missing `extraPaths` through the same canonical path contract used by the active memory backend and demonstrate the warning with a real doctor run.

## Reproduction Assessment

Yes, source-level: with builtin memory, `extraPaths: ["~/notes"]`, an existing home path, and no workspace `~/notes`, the PR's doctor helper would suppress the warning while builtin runtime still resolves the entry under the workspace. I did not run the command because this review is read-only and the PR provides no real behavior proof.

## Solution Assessment

No. The direction is useful, but the patch should use the active backend's canonical extraPaths resolver or first unify the runtime contract; hard-coding `~` expansion in doctor creates a false-negative for builtin memory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve extraPaths the same way the runtime does:** `src/commands/doctor-memory-search.ts:267-270`
  - body: The new helper expands `~` with `resolveUserPath`, but builtin memory still resolves non-absolute `extraPaths` through `normalizeExtraMemoryPaths`, which treats `~/notes` as workspace-relative. If the user's home `notes` path exists but `<workspace>/~/notes` does not, doctor will skip the new warning even though builtin sync/watch indexes nothing. Use the active backend's canonical resolver, or make the runtime path contract consistent before adding this warning.
