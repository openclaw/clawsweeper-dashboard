## Summary

Close as superseded: the same Google Chat thread-escape bug is already tracked by https://github.com/openclaw/openclaw/issues/80995 and active fix PRs https://github.com/openclaw/openclaw/pull/80996 and https://github.com/openclaw/openclaw/pull/74235, while this branch uses a process-global fallback that can misroute messages and lacks branch-specific proof.

## What This Changes

The branch caches the most recent Google Chat room thread per space on `process` and uses it when `sendText` lacks an explicit `threadId` or `replyToId`.

## Best Possible Solution

Close this branch in favor of the existing Google Chat threading PRs that route the inbound thread through the channel threading contract with tests and proof, rather than adding hidden process-global thread state.

- **[P1] Preserve explicit top-level sends:** `extensions/googlechat/src/channel.adapters.ts:222-229`
  - body: This fallback runs whenever adapter inputs lack `threadId` and `replyToId`, but that is also what the core path produces after an intentional `topLevel: true` or `threadId: null` opt-out. A cached room thread would re-thread sends that the caller explicitly asked to keep top-level.
  - confidence: 0.9
- **[P1] Keep room thread state scoped to the active turn:** `extensions/googlechat/src/monitor.ts:308-316`
  - body: The cache is keyed only by `spaceId` and lives on `process`, so a later room event in the same Google Chat space can overwrite the stored thread before an earlier agent turn sends its reply. The thread context needs to travel through the turn/tool context instead of shared mutable process state.
  - confidence: 0.88

  - body: The new guard reads `process.__gchatRoomThreads` directly before the later casted assignments. `NodeJS.Process` does not declare this plugin-specific property, so strict TypeScript can reject the read unless it goes through a typed helper or declaration.
  - confidence: 0.82
