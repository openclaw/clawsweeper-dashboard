## Summary

Keep open: the reported Google Chat thread bug is plausible, but this PR is not merge-ready because the process-global space thread fallback can misroute later or explicitly top-level sends, and no attached real-behavior proof was provided.

## What This Changes

The PR stores the last Google Chat room thread per space on `process` and falls back to it when Google Chat text sends lack an explicit `threadId` or `replyToId`.

## Best Possible Solution

Carry the inbound Google Chat thread through the canonical `MessageThreadId`/message-tool threading contract or a Google Chat `threading.buildToolContext`/`resolveAutoThreadId` implementation, while preserving explicit `replyTo`, `threadId`, `topLevel`, and `threadId: null` semantics with focused tests and redacted live proof.

## Reproduction Assessment

Source-reproducible, not live reproduced: current `room_event` dispatch uses `message_tool_only`, and Google Chat populates `ReplyToId` but not `MessageThreadId`, so message-tool replies do not get a canonical ambient thread id unless the model passes one explicitly.

## Solution Assessment

No; the branch uses hidden process-global state below the routing layer. The maintainable fix should route the thread through existing channel threading/tool-context contracts so explicit caller intent and concurrency remain visible to core.
