## Summary

Keep open: current main still delivers the unscheduled-reminder note, but this branch has a P2 correctness issue and lacks required real behavior proof. Maintainers also need to choose between overlapping reminder-guard approaches before merging one UX path.

## What This Changes

The PR adds a helper to strip the unscheduled-reminder note from reply payload text before typing/final delivery and updates focused runReplyAgent tests to expect clean text.

## Best Possible Solution

Settle the reminder-guard UX, then land one implementation that keeps internally generated guard notes out of delivered chat without deleting legitimate text, backed by focused regression coverage and redacted live channel proof.

## Reproduction Assessment

Yes, at source level: current main appends the hardcoded note when reminder-like reply text has no successful cron add and no same-session active cron. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. The PR fixes the common leak, but stripping every matching suffix after payload construction is broader than guard-injected text; the safer path is to track the payload modified by the guard or avoid adding the note to deliverable payloads.
