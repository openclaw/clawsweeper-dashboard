## Summary

Keep open: current main still has the visible reminder-note path, but this PR has a blocking over-strip bug and lacks external real behavior proof before merge.

## What This Changes

The branch adds a helper to strip the unscheduled-reminder note from reply payload text before typing/final delivery and updates reminder-guard tests to expect clean user-visible text.

## Best Possible Solution

Keep guard-generated text out of delivered chat payloads without deleting legitimate content, then back the chosen reminder-guard UX with focused tests and real channel proof.

## Reproduction Assessment

Yes, by source inspection: a non-error reminder-like payload with successfulCronAdds set to 0 and no same-session active cron flows through appendUnscheduledReminderNote and is delivered. I did not run tests because this review is read-only, but current tests assert that same visible note path.

## Solution Assessment

No. The PR fixes the common leak, but stripping every matching suffix after payload construction is broader than guard-injected text; a safer fix should track the modified payload or avoid injecting the note into delivery payloads at all.
