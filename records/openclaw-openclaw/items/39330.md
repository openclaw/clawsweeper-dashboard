## Summary

Keep open: current main still appends the unscheduled-reminder note on the final auto-reply delivery path, and this PR targets that real user-visible problem, but the branch is not merge-ready because the sanitizer can remove legitimate reply text, after-fix proof is mock-only, the branch conflicts with current main, and maintainers still need to choose the reminder-guard UX.

## What This Changes

The PR adds a `stripUnscheduledReminderNote` helper, applies it to auto-reply payloads before typing/final delivery, and updates reminder-guard tests to expect clean visible text.

## Best Possible Solution

Land one chosen reminder-guard design that keeps internal annotation text out of delivered chat without deleting legitimate content, with focused regression coverage and redacted real channel proof.

## Reproduction Assessment

Yes, at source level: current main appends the note when a reminder-like reply has zero successful cron adds and no same-session cron coverage, and focused tests assert that visible text. I did not run a live Discord or Telegram repro in this read-only review.

## Solution Assessment

No. The PR fixes the common leak path, but stripping every non-error payload is broader than the guard mutation; a safer solution should avoid adding the annotation to deliverable payloads or strip only payloads the guard actually mutated after maintainers choose the UX.
