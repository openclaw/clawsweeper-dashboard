## Summary

Keep open. Current main still sends the appended reminder-guard note through typing and final delivery, but this PR needs real behavior proof and has a blocking sanitizer-scope bug; maintainers also need to choose between this delivery-stripping approach and the related reminder-guard UX/API directions.

## What This Changes

The PR adds a helper to strip the unscheduled reminder note from reply payload text before typing/final delivery and updates reminder-guard tests to expect clean reply text.

## Best Possible Solution

Prevent only OpenClaw-injected reminder guard text from reaching delivery, preserve legitimate model-authored or quoted matching text, add regression coverage for that provenance boundary, and collect redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection of current main shows `runReplyAgent` appends the guard note and then uses those payloads for typing and final delivery, and current tests assert the visible note in no-cron cases.

## Solution Assessment

No. The PR fixes the common leak but is not the narrowest maintainable solution because it strips any matching payload suffix, including legitimate model-authored or quoted text; the removal should be tied to the payload OpenClaw modified or the note should stay out of delivery payloads entirely.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Restrict stripping to guard-injected note text:** `src/auto-reply/reply/agent-runner.ts:532-540`
  - body: The sanitizer runs over every non-error text payload after `guardedReplyPayloads` is built. When the guard did not append anything, a legitimate model response that ends with the same `\n\nNote: ...` text is still stripped, so quoted or model-authored content can be deleted. Track which payload was modified by the guard, or avoid injecting the note into delivery payloads, so only OpenClaw-injected text is removed.
