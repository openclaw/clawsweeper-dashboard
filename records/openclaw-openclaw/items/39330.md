## Summary

Keep open. Current main still exposes the unscheduled-reminder guard note on the reply delivery path, so this PR addresses a real bug, but the current patch should not merge as-is because it strips any matching model-authored suffix and still has only mocked test proof.

## What This Changes

The PR adds `stripUnscheduledReminderNote`, applies it to reply payloads before typing/final delivery, and updates reminder-guard tests to expect clean user-visible text.

## Best Possible Solution

Prevent only OpenClaw-injected reminder guard text from reaching user-visible delivery, preserve legitimate model-authored matching text, add regression coverage for that provenance boundary, and collect redacted real delivery proof.

## Reproduction Assessment

Yes. Source inspection of current main shows `runReplyAgent` appends the guard note and then uses the guarded payloads for typing and final delivery; current tests assert the visible note in no-cron cases.

## Solution Assessment

No. The PR fixes the common leak, but the sanitizer is broader than the source of the injected text; the safer fix is to track the payload OpenClaw modified or avoid putting the guard note into delivery payloads in the first place.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Restrict stripping to guard-injected note text:** `src/auto-reply/reply/agent-runner.ts:532-540`
  - body: The sanitizer runs over every non-error text payload after `guardedReplyPayloads` is computed. When the guard did not append anything, a legitimate response that ends with the same `\n\nNote: ...` text will still be stripped, so quoted or model-authored content can be deleted. Track which payload was modified by the guard, or avoid injecting the note into delivery payloads before sanitizing.
