The PR adds a helper that strips the unscheduled reminder note from auto-reply text payloads before typing/final delivery and updates reminder-guard tests to expect clean delivered text.

## Best Possible Solution

Land one selected reminder-guard design that prevents unintended delivered note text without deleting legitimate quoted content or losing the intended no-reminder safety signal, backed by focused regression tests and redacted real channel proof.

## Reproduction Assessment

Yes, source-reproducible: current main appends the note on reminder-like reply payloads with zero cron adds and no same-session cron coverage, and focused tests still assert that visible text. I did not run a live Discord or Telegram path in this read-only review.

## Solution Assessment

No, not as-is: broad post-processing fixes the common leak but is not the narrowest maintainable solution because it can remove legitimate payload text. A safer fix would tag guard-injected text, avoid inserting the annotation into deliverable payloads, or settle a maintainer-approved reminder-guard UX.
