## Summary

Keep open: the underlying guard-note delivery path is still present on current main, but this PR is not mergeable as-is because the sanitizer can delete matching user-authored text and the guard-note visibility policy needs maintainer confirmation against current tests and changelog history.

## What This Changes

The PR adds a helper to remove the unscheduled-reminder note from reply payload text before typing/final delivery and updates reminder-guard tests to expect clean reply text.

## Best Possible Solution

Decide the intended guard-note UX first, then implement a provenance-preserving fix that either avoids injecting delivery text or removes only text known to be internally injected, with regression tests and a changelog entry.

## Reproduction Assessment

Yes, source-reproducible. Current main appends the note for reminder-like replies with `successfulCronAdds: 0`, returns `guardedReplyPayloads`, and has tests asserting that visible output.

## Solution Assessment

No. The PR's suffix stripping is not the best current solution because it lacks provenance for the text being removed and changes the visible warning semantics without a maintainer decision.
