## Summary

Keep open: the PR targets a real visible reminder-note leak, but the current patch can strip legitimate content because it removes the exact note suffix from every non-error reply payload, not only text injected by the guard, and after-fix real channel proof is still missing.

## What This Changes

The branch adds `stripUnscheduledReminderNote`, applies it before typing/final reply delivery, and updates reminder-guard tests to expect clean text when no cron was scheduled.

## Best Possible Solution

Keep guard-generated annotations out of delivered chat payloads by tracking or omitting only internally injected text, then back the chosen reminder-guard UX with focused tests and live channel proof.

## Reproduction Assessment

Yes, by source inspection: a non-error reminder-like payload with `successfulCronAdds` set to 0 and no same-session active cron flows through `appendUnscheduledReminderNote` and is delivered. I did not run tests in this read-only review, but current focused tests assert the same visible note path.

## Solution Assessment

No. The PR fixes the common leak, but stripping every matching suffix after payload construction is broader than guard-injected text; a safer fix should track which payload the guard modified or avoid injecting the note into delivery payloads at all.
