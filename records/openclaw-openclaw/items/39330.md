## Summary

Keep open: the underlying reminder-note leak is still present on current main, but this PR is not merge-ready because it can silently remove legitimate reply text, has only mocked test proof, and is currently merge-conflicting.

## What This Changes

The PR adds a helper to remove the hardcoded unscheduled-reminder note from auto-reply payload text before typing/final delivery and updates reminder-guard tests to expect clean replies.

## Best Possible Solution

Settle one reminder-guard UX, then land a path that either keeps the note out of deliverable payloads entirely or strips only text known to have been injected by the guard, with focused tests and live channel proof.

## Reproduction Assessment

Yes for the focused source path: current main appends and returns the note for unbacked reminder-style replies, and current tests still assert that delivered text includes it. I did not run a live Discord or Telegram reproduction in this read-only review.

## Solution Assessment

No: stripping the suffix from every text payload is broader than the guard-owned mutation and can delete legitimate reply text. A safer solution should tag or avoid the injected note rather than sanitize unrelated payloads.
