## Summary

Keep open: current main still delivers the reminder guard note on the unbacked-reminder path, but this PR is not merge-ready because its sanitizer can remove legitimate suffix text, real behavior proof is missing, and the branch now conflicts with current main.

## What This Changes

The PR adds `stripUnscheduledReminderNote`, applies sanitized payloads before typing and final reply delivery, and updates focused runReplyAgent tests to expect clean reminder text when no cron was scheduled.

## Best Possible Solution

Land one selected reminder-guard UX that keeps internal annotations out of delivered chat without deleting legitimate model/user text, backed by focused regression tests and live Telegram or equivalent proof.

## Reproduction Assessment

Yes, by source inspection: current main appends the hardcoded note when a non-error reminder-like payload has zero successful cron adds and no same-session active cron. The focused current tests also assert that visible-note path, though I did not run tests in this read-only review.

## Solution Assessment

No. The PR fixes the common leak, but stripping every matching suffix after payload construction is broader than guard-injected text; a safer fix should track the payload the guard modified or avoid injecting the note into delivery payloads at all.
