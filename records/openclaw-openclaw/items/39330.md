## Summary

Keep open: current main still exposes the unscheduled-reminder note in the reply path, but this PR is not merge-ready because the sanitizer can delete legitimate model-authored text and the contributor has not supplied after-fix real behavior proof. The overlapping reminder-guard PRs also need maintainer UX consolidation before one approach lands.

## What This Changes

The PR adds a helper to strip the unscheduled-reminder note from reply payload text before typing/final delivery and updates focused runReplyAgent tests to expect clean text.

## Best Possible Solution

Choose one reminder-guard UX and land a narrow implementation that keeps internal guard annotations out of delivered chat without deleting legitimate text, backed by focused regression coverage and redacted live channel proof.

## Reproduction Assessment

Yes, at source level: current main appends the hardcoded note when reminder-like reply text has no successful cron add and no same-session active cron. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. The PR fixes the common leak, but stripping every matching suffix from every non-error text payload is broader than guard-injected text; the safer path is to track the payload modified by the guard or avoid adding the note to deliverable payloads.
