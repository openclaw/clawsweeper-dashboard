## Summary

Keep open: current main and the latest release still deliver the unscheduled-reminder note, but this external PR is not ready because it can strip legitimate matching text and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a reply-payload sanitizer that removes the hardcoded unscheduled-reminder note before typing/final delivery and changes reminder-guard tests to expect clean text.

## Best Possible Solution

Keep guard-generated text out of delivered chat payloads without deleting legitimate user-visible content, then align focused tests and real channel proof with the chosen reminder-guard UX.

## Reproduction Assessment

Yes for the current behavior by source inspection: a non-error reminder-like payload with successfulCronAdds set to 0 and no same-session active cron flows through appendUnscheduledReminderNote and is delivered. I did not run tests because this review is read-only, but current tests assert that same visible note path.

## Solution Assessment

No. The PR fixes the common leak, but stripping every matching suffix after payload construction is broader than guard-injected text; a safer fix should track the modified payload or avoid injecting the note into delivery payloads at all.
