## Summary

Keep open: current main still has the user-visible reminder note path, but this PR is not mergeable as-is because the sanitizer can remove legitimate model-authored text and the external contributor has not provided real behavior proof. The overlapping reminder-guard PRs also need maintainer consolidation before one UX direction lands.

## What This Changes

The PR adds `stripUnscheduledReminderNote`, applies it to reply payloads before typing/final delivery, and changes focused reminder-guard tests to expect clean reply text.

## Best Possible Solution

Keep the note out of delivered chat only for text the guard actually injected, preferably by avoiding deliverable-payload mutation or by tracking the modified payload, then validate with focused tests and redacted live channel proof.

## Reproduction Assessment

Yes, at source level: current main appends the hardcoded note when reminder-like reply text has no successful cron add and no same-session active cron coverage. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. The PR fixes the common leak, but stripping every matching suffix from every non-error text payload is broader than guard-injected text; the safer path is to track the payload modified by the guard or avoid putting the note into deliverable payloads.
