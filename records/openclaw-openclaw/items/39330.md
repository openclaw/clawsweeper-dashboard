## Summary

Keep open: current main and the latest release still append the hardcoded unscheduled-reminder note into delivered reply payloads, so the underlying user-visible problem is real. This external PR should not merge as-is because the sanitizer can strip legitimate model-authored text and the contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds a helper that strips the hardcoded unscheduled-reminder note from reply payload text before typing/final delivery and updates focused `runReplyAgent` tests to expect clean user-visible text.

## Best Possible Solution

Keep guard-generated text out of delivered chat payloads without deleting legitimate user-visible content, then align tests and real channel proof with the chosen reminder-guard UX.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-error reminder-like payload with `successfulCronAdds: 0` and no same-session active cron flows through `appendUnscheduledReminderNote`, and current tests assert the hardcoded note in the returned text.

## Solution Assessment

No. The PR fixes the common leak, but stripping every matching suffix after payload construction is broader than the guard-injected text; a safer fix should track the guard-modified payload or avoid injecting the note into delivery payloads at all.
