## Summary

Keep open. Current main has narrowed the original `help` failure, but the generic fallback still treats an unregistered non-plugin primary token such as `tool` as a plugin ID under a restrictive `plugins.allow`, so the canonical CLI-token/plugin-ID confusion is still valid.

## What This Changes

The issue asks OpenClaw to stop treating built-in, framework, or synthetic CLI command tokens as plugin IDs when `plugins.allow` is restrictive, especially for `openclaw tool` and `openclaw help`.

## Best Possible Solution

Add an explicit CLI command classification layer before plugin allowlist diagnostics: framework/core/reserved non-plugin names should never suggest adding themselves to `plugins.allow`, manifest CLI aliases should resolve to the owning plugin ID, runtime slash aliases should explain chat-only usage or point to the related CLI command, and truly unknown commands should fall through to Commander-style unknown-command handling.

## Reproduction Assessment

Yes. The issue gives a concrete restrictive `plugins.allow` config and command sequence, and current source still routes an unregistered primary token such as `tool` into the missing-plugin fallback; `help` is partially fixed, but the remaining non-plugin-token path is reproducible by code inspection.

## Solution Assessment

Yes. The narrow maintainable fix is to classify command tokens before the plugin allowlist fallback, not to add pseudo command IDs to `plugins.allow` or restore an undocumented `openclaw tool` command surface.
