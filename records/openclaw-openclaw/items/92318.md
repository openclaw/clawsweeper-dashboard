## Label Justifications

- `P1`: The current behavior can mark a cron run delivered while suppressing the fallback that would actually send the scheduled message to the configured channel.
- `merge-risk: 🚨 message-delivery`: The new fallback behavior fixes observed message loss but could duplicate delivery if a pending nested announce later reaches the same target.

## Summary

The PR remains necessary on current main and applies the existing explicit-target contract at the correct cron boundary. Source inspection and the live Telegram proof support the fix; because this is a draft, maintainer-authored PR with a protected label, it should remain open for explicit maintainer handling, with delayed nested-announce duplication as the only residual delivery question.

## What This Changes

The PR requires cron announce and no-deliver source plans to have explicit message-tool target metadata before treating a send as delivery to the configured target, and adds focused regression coverage.

## Best Possible Solution

Keep the explicit-target requirement at the cron source-delivery boundary, preserve legacy implicit-target behavior for other owners, and confirm that the pending `sessions_send` scenario results in exactly one configured-target delivery before landing.

## Reproduction Assessment

Yes. On current main, `didSendViaMessageTool=true` with an empty target list synthesizes the configured cron target, making source delivery appear satisfied and skipping direct fallback without explicit target evidence.

## Solution Assessment

Yes, subject to one delivery-semantics check. Reusing the existing explicit-target contract at the cron owner boundary is narrower and safer than treating a pending inter-session send as a synchronous channel acknowledgement or changing all message-tool-owned flows.
