## Summary

Keep open. Current main has not implemented this policy, and PR head still has two P2 correctness defects in the new allowlist filtering while the new config mode remains a maintainer-reviewed compatibility decision.

## What This Changes

The PR adds `plugins.bundledDiscovery`, makes bundled provider and web-provider discovery honor restrictive `plugins.allow` by default, migrates legacy allowlist configs to `compat` in doctor, and updates docs, schema, changelog, and regression tests.

## Best Possible Solution

Fix the alias-normalization and master-toggle edge cases, keep the doctor compatibility migration, then have maintainers approve the new default/compat policy before landing.

## Reproduction Assessment

Yes for the PR defects by source inspection. The new public-artifact filter uses raw allow entries before alias normalization, and the compat helper can flip `plugins.enabled: false` after allowlist filtering rejects all requested plugin ids.

## Solution Assessment

No as written. The high-level allowlist/compat direction is plausible, but the implementation should normalize aliases and preserve the plugin master toggle before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Normalize allowlist aliases before filtering providers:** `src/plugins/web-provider-public-artifacts.ts:41-42`
  - body: The new public-artifact filter compares raw `plugins.allow` strings with bundled plugin ids. Supported aliases such as `google-gemini-cli` and `minimax-portal` are normalized by the regular plugin config path, so allowlist-mode web search/fetch inventories can disappear on this fast path even though the same config is accepted elsewhere.
  - confidence: 0.87
- **[P2] Avoid re-enabling plugins after allowlist rejection:** `src/plugins/bundled-compat.ts:51-53`
  - body: When `plugins.enabled` is `false` and allowlist mode filters out every requested bundled provider, `changed` remains false but the later `forcePluginsEnabled` return path can still set `enabled: true`. Preserve the original config unless at least one requested plugin survives the allowlist or explicit compat mode warrants the override.
  - confidence: 0.9
