## Summary

Keep open: this draft, protected-label PR is useful but still has a blocking correctness issue because allowlist mode filters bundled entries while the compat helper can still rewrite the global plugin disable from `false` to `true`.

## What This Changes

The branch adds `plugins.bundledDiscovery`, wires it into bundled provider and web-provider discovery, adds doctor warnings, and updates config schema, docs, changelog, generated baselines, and regression tests.

## Best Possible Solution

Land the opt-in bundled discovery policy after preserving `plugins.enabled: false` as the global plugin disable in allowlist mode.

## Reproduction Assessment

Yes. Source inspection of PR head shows `withBundledPluginEnablementCompat` can still return `plugins.enabled: true` from a config that set `plugins.enabled: false` when allowlist discovery is enabled.

## Solution Assessment

No as written. The opt-in policy is maintainable, but the helper should gate the global-enable compat override off in allowlist mode and add disabled-master-toggle regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the plugin master toggle:** `src/plugins/bundled-compat.ts:45-46`
  - body: In allowlist mode this helper filters which bundled entries get injected, but `forcePluginsEnabled` still runs independently. A config with `plugins.enabled: false` and `plugins.bundledDiscovery: "allowlist"` can therefore be rewritten with `plugins.enabled: true`, allowing bundled provider discovery/load work even though the documented master toggle is off. Gate that global-enable compat override off in allowlist mode and add regression coverage for this config shape.
  - confidence: 0.86
