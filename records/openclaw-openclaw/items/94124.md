## Label Justifications

- `P1`: The issue reports active run-mode subagents being terminated mid-work by the idle watchdog, which breaks a real agent workflow.
- `impact:session-state`: The reported failure is premature termination or takeover of an active subagent session while it is collecting tool results.

## Summary

Keep open: current main and v2026.6.8 still reset the provider idle watchdog only on LLM stream/request activity, not on completed tool calls, so the reported subagent timeout path is not implemented yet. The linked candidate PR at https://github.com/openclaw/openclaw/pull/94133 is open but currently blocked by ClawSweeper review findings and missing real behavior proof.

## What This Changes

The issue asks OpenClaw to keep run-mode subagent sessions alive when tool calls make progress so the 120-second LLM idle watchdog does not terminate active data-collection work.

## Best Possible Solution

Land a repaired implementation that records run-scoped tool activity across shared direct and cataloged tool execution, preserves the idle watchdog for truly stalled providers, adds focused idle-timeout coverage, and includes redacted real run-mode subagent proof.

## Reproduction Assessment

Yes at source level: current main arms the LLM idle watchdog on provider stream waits and has no tool-completion activity signal in the shared tool execution path. I did not run a live 120-second subagent timeout reproduction in this read-only review.

## Solution Assessment

Yes for the requested boundary: an internal activity signal is better than adding config for this narrow bug. The current candidate PR is not the best implementation as-is because the existing ClawSweeper review found heartbeat lifecycle gaps and missing real behavior proof.
