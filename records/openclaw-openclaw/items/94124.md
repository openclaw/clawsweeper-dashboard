## Label Justifications

- `P1`: The report describes active subagent runs terminating during long tool work, which can break agent workflows for real users.
- `impact:message-loss`: The reported timeout can discard the final subagent report after tool work has already completed or nearly completed.
- `impact:session-state`: The issue concerns idle tracking, takeover behavior, and lifecycle state for active subagent sessions.

## Summary

Keep open: current main still refreshes the LLM idle watchdog only from provider stream progress or LLM request activity, while normal direct tools execute outside that stream progress path. The linked open PR is the active candidate fix, but it has unresolved source-level review blockers and missing real-behavior proof, so this issue remains the canonical report until a corrected fix lands.

## What This Changes

The issue asks OpenClaw to refresh run-mode subagent idle tracking when tool calls complete so active data-collection subagents are not killed by the LLM idle timeout.

## Best Possible Solution

Keep this issue open as the canonical narrow report, then land a corrected and proven fix through https://github.com/openclaw/openclaw/pull/94133 or a replacement that records run-scoped tool activity across both direct and cataloged tool completion paths.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection provides a medium-confidence reproduction path: current main waits on provider stream progress while direct tool execution happens outside that progress path, with no tool-completion activity hook feeding the idle watchdog.

## Solution Assessment

No for the currently linked PR as reviewed: the idea belongs in the right area, but the best fix should use a run-scoped activity boundary that survives across provider waits and covers shared direct tools as well as cataloged tools.
