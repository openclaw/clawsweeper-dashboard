## Summary

Keep open: current main has caps, compact prompt fallback, and session snapshot reuse, but it still has the reported nested-directory truncation path and no INDEX/lazy discovery contract for very large skill bundles.

## What This Changes

The issue requests indexed, lazy, or configurable skill discovery so very large nested skill collections do not get scanned, truncated, or overrepresented in the model-facing skills catalog.

## Best Possible Solution

Design one canonical skills discovery contract for large bundles, preferably an indexed or manifest-backed path that preserves bounded traversal, prompt-budget limits, snapshot reuse, and public docs without adding broad fallback stacks.

## Reproduction Assessment

Yes for the discovery warning by source inspection and existing tests: current main still logs the reported nested-directory truncation when a grouped skill folder exceeds the candidate limit. No for the full 700-skill live performance claim; I did not run that scenario in this read-only pass.

## Solution Assessment

Unclear: an INDEX or lazy-loading mechanism is plausible, but OpenClaw already has caps, compact prompt fallback, and snapshot reuse, so maintainers should choose the permanent discovery contract before implementation.
