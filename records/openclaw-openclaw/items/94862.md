- `merge-risk: 🚨 session-state`: The diff can mis-associate successful cron run state with failed embedded-agent runtime state.

## Summary

Keep open and block merge: the PR targets the right cron delivery bug, but its current predicate is too broad and would let generic run-level retry/provider failures be recorded as successful when later or final output exists.

## What This Changes

The PR changes `resolveCronPayloadOutcome` and adds helper tests so recovered early cron tool errors with a run-level error can be treated as nonfatal for delivery.

## Best Possible Solution

Narrow the recovery rule to proven non-terminal tool-warning cases, preserving fatal run-level provider/retry/context errors and keeping regression coverage in the canonical cron helper and run-meta suites.

## Reproduction Assessment

Yes at source level: the linked issue gives a scheduled repro, and current main gates recovered payload detection on the absence of `runLevelError`. The PR-introduced regression is also source-reproducible against existing run-level error tests; I did not run a live cron/channel job in this read-only review.

## Solution Assessment

No: the resolver is the right layer, but this implementation clears too many run-level failures. The safer fix should classify the reported recovered tool-warning shape without overriding arbitrary provider, retry, context, or incomplete-turn errors.
