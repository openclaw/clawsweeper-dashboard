## Summary

Keep open: the linked cron delivery bug is real and source-reproducible, but this PR's recovery rule is too broad and can turn genuine terminal `meta.error` states into successful delivery candidates.

## What This Changes

The PR changes cron isolated-agent payload outcome classification so a run-level error no longer makes a recovered error-then-success payload fatal, and adds helper tests for the recovery and fatal-control cases.

## Best Possible Solution

Narrow the recovery condition to the reported non-terminal tool-warning shape while preserving fatal handling for terminal embedded-run metadata errors.

## Reproduction Assessment

Yes at source level: current main has the reported run-level-error gate in `resolveCronPayloadOutcome`, and the linked issue supplies a reduced scheduled-run trajectory that reaches that helper before dispatch.

## Solution Assessment

No: the PR fixes the reported trajectory but also broadens recovery to unrelated terminal `meta.error` states; the safer fix is to recognize recovered non-terminal tool warnings without disabling all run-level errors.
