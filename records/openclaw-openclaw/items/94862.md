## Summary

Keep open: this is a valid cron bug-fix PR, but the patch is not merge-ready because it marks recovered run-level errors nonfatal without preserving the recovered final output and it broadens existing fatal run-level error semantics.

## What This Changes

The PR changes cron payload outcome classification so later successful payloads or final text can override a run-level error, and adds helper tests for recovered early tool errors.

## Best Possible Solution

Narrow recovery to proven non-terminal tool-warning cases, preserve generic run-level failures as fatal, and ensure recovered runs deliver the final assistant text or successful payload rather than the run-level error.

## Reproduction Assessment

Yes, at source level: the linked issue provides a scheduled repro and current main gates recovered payload detection on the absence of `runLevelError`. The PR-introduced regressions are also source-reproducible from the helper predicate and current run-level error tests; I did not run a live cron/channel job in this read-only review.

## Solution Assessment

No: the resolver is the right layer, but this implementation is too broad and does not preserve recovered output after marking a run nonfatal. The safer fix should target recovered non-terminal tool-warning evidence while keeping provider, retry, context, and incomplete-turn errors fatal.
