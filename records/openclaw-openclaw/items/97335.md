## Summary

Keep open. The report is a plausible current bug: cron fallback selection is documented and wired, but current main still caps remote cron LLM idle time at 60 seconds even when the job has an explicit 600 second run timeout, which matches a fallback run that progresses and then aborts with `Request was aborted`.

## What This Changes

The issue reports isolated `agentTurn` cron jobs aborting mid-run after switching from a failing primary model to a working fallback model.

## Best Possible Solution

Keep this issue open and verify the isolated-cron fallback idle-timeout path; the likely fix is to make explicit cron/provider timeouts govern progressing fallback turns or surface a precise idle-timeout failure instead of a generic abort.

## Reproduction Assessment

Yes at source level, not live. Current main routes cron fallback attempts through the embedded runner with the explicit run timeout, but remote cron idle timeout still clamps a 600 second run to a 60 second idle watchdog.

## Solution Assessment

Unclear until maintainer policy is confirmed. The narrowest fix is likely in the cron LLM idle-timeout policy and its tests, with live proof that a progressing fallback turn no longer aborts before the job timeout.
