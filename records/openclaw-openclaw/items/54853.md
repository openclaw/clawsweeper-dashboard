## Summary

Keep open. Current main still lacks a per-agent target-channel account binding, and Feishu tool account selection can still fall back to the configured Feishu default when the originating channel account id is not a configured Feishu account.

## What This Changes

Request support for binding an agent to a target-channel accountId so cross-channel tools, such as DingTalk-originated Feishu operations, use the intended account instead of a channel default.

## Best Possible Solution

Add a generic, documented target-channel account resolver keyed by active agentId and target channel, preserving explicit tool accountId first, then agent binding, then valid same-id context, then channel default fallback.

## Reproduction Assessment

Yes, source-level reproduction is high confidence. The proposed per-agent keys are rejected by the strict agent schema, and a non-Feishu originating account id reaches Feishu tool resolution as context before falling back to channels.feishu.defaultAccount.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the safest maintainable fix is a generic target-channel resolver and documented precedence rather than a Feishu-only key.
