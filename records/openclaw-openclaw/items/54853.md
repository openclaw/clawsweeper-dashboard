- `impact:security`: The reported behavior can route Feishu tool execution through another user's credentials.

## Summary

Keep open. Current main still lacks a generic per-agent target-channel account binding; the shipped Feishu session-account fix only covers same-channel or same-account-id context and does not solve the reported DingTalk-to-Feishu mismatched-account isolation case.

## What This Changes

Request to bind an agent to a target channel accountId so cross-channel tool calls use that agent's intended credentials instead of a channel default.

## Best Possible Solution

Define a generic, documented target-channel account resolver keyed by active agentId plus target channel, wire Feishu tools through it, and add schema, SDK/docs, and regression coverage for mismatched DingTalk and Feishu account ids.

## Reproduction Assessment

Yes, source-level. The proposed per-agent keys are still rejected by the strict schema, and Feishu falls back to channels.feishu.defaultAccount when the contextual account id is not a configured enabled Feishu account.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the best fix needs an approved generic config/SDK contract and security precedence rather than a Feishu-only field.
