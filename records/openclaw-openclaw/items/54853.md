## Summary

Keep open. Current main and the latest release still lack a per-agent target-channel account binding, so the reported DingTalk-to-Feishu isolation case can still fall back to Feishu default-account behavior when the origin account id is not a Feishu account.

## What This Changes

The issue requests an agent-level way to bind a target-channel accountId so cross-channel tool calls use the intended Feishu account.

## Best Possible Solution

Define a generic, documented target-channel account binding contract and thread it through plugin tool context so Feishu can resolve explicit tool accountId first, then the bound target account, then valid same-channel context, then configured default fallback.

## Reproduction Assessment

Yes at source level. Current schema rejects the requested per-agent binding fields, and Feishu tool resolution has no per-agent target-channel lookup; I did not run a live DingTalk plus Feishu setup in this read-only review.

## Solution Assessment

Unclear. A generic channelBindings-like contract is the maintainable direction, but the exact schema and precedence need maintainer product/security approval before implementation.
