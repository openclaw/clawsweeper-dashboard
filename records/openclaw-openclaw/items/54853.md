## Summary

Keep open. Current main still lacks a generic per-agent target-channel account binding, and Feishu tool account resolution can still fall back to the configured Feishu default when the active channel account id is not an enabled Feishu account.

## What This Changes

Request support for binding an agent to a specific accountId on another channel, especially for DingTalk-originated agents invoking Feishu tools.

## Best Possible Solution

Define a generic, documented target-channel account resolver keyed by active agentId plus target channel, wire Feishu tools through it with explicit override precedence, and add schema/docs/regression coverage for mismatched DingTalk and Feishu account ids.

## Reproduction Assessment

Yes. Source inspection shows the proposed per-agent keys are still rejected by the strict agent schema, while Feishu tool resolution still falls back to defaultAccount when ctx.agentAccountId is not an enabled Feishu account.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the maintainable fix needs an approved generic SDK/config contract and security precedence instead of a Feishu-only feishuAccountId field.
