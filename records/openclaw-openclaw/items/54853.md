## Summary

Keep open. Current main and v2026.6.8 still lack a target-channel account binding for Feishu tools, so a DingTalk-originated agent whose DingTalk account id is not a Feishu account can still fall back to Feishu default-account behavior.

## What This Changes

Request support for per-agent cross-channel account binding so a DingTalk-originated agent can select the intended Feishu account when invoking Feishu tools.

## Best Possible Solution

Define a generic, documented target-channel account resolution contract and thread it through plugin tool context so Feishu can resolve explicit tool accountId first, then a bound target account, then valid same-channel context, then configured default fallback.

## Reproduction Assessment

Yes at source level. Current schema rejects the requested per-agent binding fields, and Feishu tool resolution has no per-agent target-channel lookup; no live DingTalk plus Feishu setup was run in this read-only review.

## Solution Assessment

Unclear. A generic channelBindings-like contract is the maintainable direction, but a Feishu-only config key is not the best fit and the exact precedence needs maintainer product/security approval.
