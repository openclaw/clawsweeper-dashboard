## Summary

Keep open. Current main still lacks a target-channel account binding contract for plugin tools, and the Feishu resolver still falls back to the configured Feishu default when the active `agentAccountId` is not an enabled Feishu account; the prior ClawSweeper comments remain current.

## What This Changes

Request support for binding an agent to a target channel accountId so DingTalk-originated agents can invoke Feishu tools under the intended Feishu account instead of the Feishu default.

## Best Possible Solution

Add a generic, documented target-channel account resolver keyed by active `agentId` plus target channel, then wire Feishu tools through it with precedence of explicit tool `accountId`, agent binding, valid same-id context, and finally Feishu default fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: per-agent binding keys are rejected by the strict agent schema, and a DingTalk-originated `agentAccountId` that is not an enabled Feishu account falls through to Feishu default-account selection.

## Solution Assessment

Unclear. The requested `channelBindings` shape is plausible, but the narrow maintainable fix should be a generic target-channel resolver and documented precedence rather than a Feishu-only field.
