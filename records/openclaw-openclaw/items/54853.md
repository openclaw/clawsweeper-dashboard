- `impact:security`: The report concerns cross-user or cross-tenant tool execution through the wrong Feishu credentials.

## Summary

Keep open. Current main and the latest release still lack a generic per-agent target-channel account binding; the shipped Feishu fix covers same-channel/session account precedence, but cross-channel tool calls still fall back to the Feishu default when the contextual account is not a configured enabled Feishu account.

## What This Changes

The issue requests a per-agent cross-channel account binding so a DingTalk-originated agent can select the correct Feishu account for Feishu tool execution.

## Best Possible Solution

Define a generic documented target-channel account resolver keyed by active agentId plus target channel, wire Feishu tools through it, and add schema, SDK/docs, and regression coverage for mismatched DingTalk and Feishu account ids.

## Reproduction Assessment

Yes, from source inspection. The proposed per-agent keys are still rejected by the strict agent schema, and the Feishu resolver falls back to the Feishu default when the contextual account is not a configured enabled Feishu account; I did not run a live DingTalk-to-Feishu scenario in this read-only review.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the best fix needs an approved generic config/SDK contract and account-isolation precedence rather than a Feishu-only field.
