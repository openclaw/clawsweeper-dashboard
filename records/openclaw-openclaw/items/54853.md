## Summary

Keep open. Current main still does not provide a generic per-agent target-channel account binding, and the remaining issue is a real account-isolation/product-contract gap rather than a stale Feishu default-account bug already covered by the related PR.

## What This Changes

The issue requests a per-agent cross-channel account binding so a DingTalk-originated agent can select the intended Feishu accountId when invoking Feishu tools.

## Best Possible Solution

Define a generic documented target-channel account resolver keyed by active agentId and target channel, wire Feishu tools through that contract, and cover the cross-channel mismatch case with schema, SDK/docs, and regression tests.

## Reproduction Assessment

Yes, from source inspection. The requested per-agent binding keys are still rejected by the strict agent schema, and Feishu tool account resolution still falls back to the Feishu default when the contextual account is not a configured enabled Feishu account.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the best fix needs maintainer approval for a generic config and SDK contract rather than a Feishu-only agent field.
