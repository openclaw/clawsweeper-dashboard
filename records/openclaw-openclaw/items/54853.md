- `impact:security`: The reported behavior can route Feishu tool execution through another user's configured credentials.

## Summary

Keep open. Current main and v2026.5.22 still lack the requested per-agent target-channel account binding: same-channel Feishu account context is handled, but a DingTalk-origin agent can still reach Feishu tool resolution with only the origin account id and then fall back to the configured Feishu default account.

## What This Changes

Add per-agent cross-channel account binding so a DingTalk-origin agent can select the intended Feishu account for Feishu tools instead of using the Feishu default account.

## Best Possible Solution

Add a generic documented target-channel account resolution contract, expose the needed SDK/runtime seam, and wire Feishu tools through explicit accountId first, approved bound Feishu account second, valid same-channel context third, then Feishu default fallback.

## Reproduction Assessment

Yes, at source level. The proposed per-agent keys are rejected by the strict agent schema, inbound runs pass only the originating account id as agentAccountId, and Feishu tool resolution falls back to channels.feishu.defaultAccount when that id is not a configured enabled Feishu account.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the maintainable fix needs an approved generic config/SDK contract and account-selection precedence rather than a Feishu-only field or plugin-internal route-binding read.
