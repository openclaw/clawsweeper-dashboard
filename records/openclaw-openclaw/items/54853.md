## Summary

Keep open. Current main and the latest release still lack a generic per-agent target-channel account binding for the reported DingTalk-origin agent calling Feishu tools, even though same-channel Feishu session-account precedence is now covered.

## What This Changes

The issue requests a per-agent cross-channel accountId binding so a non-Feishu-origin agent can choose the correct Feishu account for Feishu tool execution.

## Best Possible Solution

Add a generic documented target-channel account resolution contract, then wire Feishu tools to use explicit accountId first, an approved per-agent Feishu binding next, and only then existing contextual/default fallbacks.

## Reproduction Assessment

Yes, at source level: the proposed per-agent binding key is rejected by strict agent schema, and Feishu tools only receive an origin account plus Feishu default fallback. I did not execute a live DingTalk-to-Feishu reproduction.

## Solution Assessment

Unclear. The requested shape is plausible, but the best fix needs a maintainer-approved generic config/SDK contract rather than a Feishu-only one-off field.
