## Summary

Keep open. Current main and v2026.6.6 still lack a per-agent target-channel account binding, so the reported DingTalk-to-Feishu isolation case can still fall back to the configured Feishu default account when the originating account id is not a real Feishu account.

## What This Changes

Request support for binding an agent to a target channel accountId so a DingTalk-origin agent can execute Feishu tools with the intended Feishu account instead of the default account.

## Best Possible Solution

Define a generic, documented target-channel account binding contract and pass it through plugin tool context to Feishu account resolution while preserving explicit accountId, valid contextual account, and defaultAccount precedence.

## Reproduction Assessment

Yes at source level. Current schema rejects the requested per-agent account binding fields, and the Feishu resolver has no agent-to-target-channel account lookup; I did not run a live DingTalk plus Feishu setup in this read-only review.

## Solution Assessment

No, a Feishu-only feishuAccountId field is not the best long-term shape. The safer direction is a generic target-channel binding or plugin SDK context contract with explicit precedence and security review.
