## Summary

Keep open. Current main still has no per-agent target-channel account binding, and Feishu tool resolution can still fall back to the configured Feishu default when the originating channel account id is not a Feishu account.

## What This Changes

Request support for per-agent cross-channel account binding so a DingTalk-originated agent can invoke Feishu tools with the intended Feishu accountId.

## Best Possible Solution

Add a documented target-channel account resolver keyed by active agentId plus target channel, preserving explicit tool accountId first, then agent binding, then valid same-id context, then channel default fallback.

## Reproduction Assessment

Yes. Source inspection shows the proposed per-agent keys are rejected by the strict agent schema, and a DingTalk-originated agentAccountId that is not a configured Feishu account falls through to Feishu default account selection.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the narrow maintainable solution should be a generic target-channel resolver and documented precedence rather than a Feishu-only per-agent field.
