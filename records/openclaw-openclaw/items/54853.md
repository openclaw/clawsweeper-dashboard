## Summary

Keep open. Current main still lacks a per-agent target-channel account binding for the reported DingTalk-to-Feishu case; Feishu tools can use an explicit tool accountId, a same-id contextual agentAccountId, or the Feishu default account, but there is no configured agent-to-Feishu account map when those ids differ.

## What This Changes

The issue requests per-agent cross-channel account binding so a DingTalk-originated agent can select the intended Feishu account for Feishu tool calls.

## Best Possible Solution

Add a generic, documented target-channel account resolver keyed by active agentId plus target channel. Feishu should preserve explicit tool accountId first, then use the agent's bound Feishu account, then valid same-id contextual routing, then channels.feishu.defaultAccount/runtime fallback; update config types/schema/help/docs and add DingTalk-to-Feishu regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a proposed agents.list[].feishuAccountId or channelBindings key is rejected by the strict agent schema, and a DingTalk-originated agentAccountId that is not a configured Feishu account falls through to Feishu default account selection.

## Solution Assessment

Unclear. The requested channelBindings shape is a reasonable direction, but the best maintainable fix should be a generic target-channel resolver and documented precedence rather than a Feishu-only per-agent field.
