## Summary

Keep open: current main and v2026.6.10 still lack a target-channel account binding path for Feishu plugin tools, so the reported DingTalk-to-Feishu account isolation problem remains a security-sensitive product/API decision.

## What This Changes

Requests cross-channel account binding so an agent invoked from one channel can use the correct Feishu account when executing Feishu tools.

## Best Possible Solution

Define a maintainer-approved generic target-channel account resolution contract, preferably reusing documented bindings[] where possible, and make Feishu tools consult it before global default while preserving explicit and valid contextual account precedence.

## Reproduction Assessment

Yes at source level. Current schema still rejects the requested per-agent binding fields, and Feishu tool account resolution has no agent-to-target-channel binding lookup; no live DingTalk plus Feishu setup was run.

## Solution Assessment

Unclear. A Feishu-only feishuAccountId key is not the best fit; the maintainable path is a generic target-channel account contract or reuse of bindings[] with maintainer-approved precedence.
