## Summary

Keep open. Current main and the latest release still lack a per-agent target-channel account binding, so the reported DingTalk-to-Feishu isolation case remains a real product/security decision rather than a cleanup-close candidate.

## What This Changes

The issue requests a per-agent cross-channel accountId binding so an agent originating on one channel can choose which Feishu account its Feishu tools use.

## Best Possible Solution

Define a generic, documented target-channel account resolution contract and thread it through plugin tool context so Feishu resolves explicit tool accountId first, then a bound Feishu account for the active agent, then valid same-channel context, then configured default fallback.

## Reproduction Assessment

Yes at source level. Current schema still rejects the requested per-agent binding fields, and Feishu tool resolution has no per-agent target-channel lookup; no live DingTalk plus Feishu setup was run.

## Solution Assessment

Unclear. A generic channelBindings-like contract is the maintainable direction, but a Feishu-only config key is not the best fit and the exact precedence needs maintainer product/security approval.
