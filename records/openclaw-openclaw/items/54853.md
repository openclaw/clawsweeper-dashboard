## Summary

Keep open. Current main and the latest release cover same-channel Feishu session-account priority, but they still do not provide a generic per-agent target-channel account binding for the reported DingTalk-origin agent invoking Feishu tools, so a non-Feishu origin can still fall back to the Feishu default account.

## What This Changes

The issue requests support for binding an agent to a target-channel accountId so cross-channel tool calls use the intended account credentials.

## Best Possible Solution

Add a generic documented target-channel account resolution contract, wire Feishu tools through explicit accountId first and an approved bound target account before default fallback, and cover schema, docs, SDK expectations, and mismatched DingTalk-to-Feishu account tests.

## Reproduction Assessment

Yes, at source level. The proposed per-agent keys are rejected by the strict agent schema, Feishu tools only receive origin account context, and non-Feishu origin ids fall back to Feishu default selection.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the maintainable fix needs an approved generic config/SDK contract and account-selection precedence rather than a Feishu-only field.
