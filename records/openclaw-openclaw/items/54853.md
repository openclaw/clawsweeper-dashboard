## Summary

Keep open. Current main still lacks a generic per-agent target-channel account binding, and Feishu tool account resolution can still fall back to the configured Feishu default when the active channel account id is not a Feishu account.

## What This Changes

Request support for binding an agent to a target channel accountId so a DingTalk-originated agent can call Feishu tools with that agent's intended Feishu account.

## Best Possible Solution

Add a documented generic target-channel account resolver keyed by active agentId plus target channel, wire Feishu tools through it, and cover the DingTalk-to-Feishu mismatch with schema, docs, and regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: proposed per-agent keys are rejected by the strict agent schema, and Feishu tools fall back to the configured Feishu default when the origin agentAccountId is not a configured Feishu account. I did not run a live DingTalk/Feishu setup because this review is read-only.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the maintainable fix should be a generic plugin-facing target-account resolver with documented security precedence rather than a Feishu-only field.
