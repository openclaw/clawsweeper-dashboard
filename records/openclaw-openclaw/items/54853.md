## Summary

Keep open. Current main and v2026.5.22 still lack a per-agent target-channel account binding for Feishu tools; the related session-before-default Feishu work solved same-channel context precedence, but not the reported DingTalk-to-Feishu account isolation case.

## What This Changes

The issue requests a per-agent cross-channel binding so an agent invoked from one channel can choose the correct Feishu account when using Feishu tools.

## Best Possible Solution

Add a generic, documented target-channel account resolution contract keyed by active agent and target channel, then wire Feishu tools through explicit accountId first, approved bound Feishu account second, valid same-channel context third, and Feishu default fallback last.

## Reproduction Assessment

Yes, at source level. The proposed per-agent keys are rejected by strict agent config, origin runs only pass the source accountId into plugin tools, and Feishu tool resolution falls back to channels.feishu.defaultAccount when that id is not an enabled Feishu account.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the maintainable fix needs an approved generic config/SDK contract and account-selection precedence rather than a Feishu-only field.
