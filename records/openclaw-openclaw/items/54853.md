## Summary

Keep open. Current main still lacks a generic per-agent target-channel account binding, and the remaining fix requires a public config/SDK contract plus security-sensitive precedence rules rather than a narrow bug patch.

## What This Changes

The request asks for per-agent cross-channel accountId binding so a DingTalk-originated agent can choose the correct Feishu account for Feishu tool calls.

## Best Possible Solution

Define a generic documented target-channel account resolver keyed by active agentId plus target channel, wire Feishu tools through it, and add schema/docs/regression coverage for mismatched DingTalk and Feishu account ids.

## Reproduction Assessment

Yes. Source inspection shows the proposed per-agent keys are still rejected by the strict agent schema, and Feishu tool account resolution still falls back to the configured Feishu default when the contextual account is not an enabled Feishu account.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the maintainable fix needs an approved generic SDK/config contract and security precedence rather than a Feishu-only field.
