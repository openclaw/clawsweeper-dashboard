## Summary

Keep open. Current main and the latest release still lack a generic per-agent target-channel account binding; the shipped Feishu session-account fix only helps when the contextual account id is itself a configured enabled Feishu account, so the reported DingTalk-to-Feishu mismatched-account case can still fall back to the Feishu default.

## What This Changes

Request support for binding an agent to a target-channel accountId so a DingTalk-originated agent can call Feishu tools with the intended Feishu account instead of the Feishu default.

## Best Possible Solution

Define a generic documented target-channel account resolver keyed by active agentId plus target channel, wire Feishu tools through it, and add schema, SDK/docs, and regression coverage for mismatched DingTalk and Feishu account ids.

## Reproduction Assessment

Yes, from source inspection. The proposed per-agent keys are still rejected by the strict agent schema, and the Feishu resolver falls back to the Feishu default when the contextual account is not a configured enabled Feishu account.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the best fix needs an approved generic config/SDK contract and account-isolation precedence rather than a Feishu-only field.
