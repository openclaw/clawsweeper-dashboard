## Summary

Keep open. Current main still lacks a generic per-agent target-channel account binding, and the Feishu tool resolver can still fall back to the configured Feishu default when the active channel account id is not an enabled Feishu account.

## What This Changes

Request support for binding an agent to a target channel accountId so a DingTalk-originated agent can use its intended Feishu credentials when invoking Feishu tools.

## Best Possible Solution

Add a documented generic target-channel account resolver keyed by active agentId plus target channel, wire Feishu tools through it with explicit override precedence, and cover the cross-channel mismatched-account case in schema, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection shows the proposed per-agent keys are rejected by the strict agent schema, while Feishu tool account resolution falls back to defaultAccount when ctx.agentAccountId is not an enabled Feishu account; no live DingTalk/Feishu setup was run in this read-only review.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the best fix needs a maintainer-approved generic SDK/config contract and security precedence rather than a Feishu-only feishuAccountId field.
