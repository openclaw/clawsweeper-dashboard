## Summary

Keep open. Current main still lacks a generic per-agent target-channel account binding; the related Feishu session-account and inbound Feishu binding fixes do not cover a DingTalk-origin agent that needs an explicitly bound Feishu target account.

## What This Changes

Request to bind each agent to a target-channel accountId so cross-channel Feishu tool calls use the correct Feishu credentials instead of the Feishu default.

## Best Possible Solution

Define a generic documented target-channel account resolver keyed by active agentId plus target channel, wire Feishu tools through it, and add schema, SDK/docs, and regression coverage for mismatched DingTalk and Feishu account ids.

## Reproduction Assessment

Yes, from source inspection. The proposed per-agent keys are still rejected by the strict agent schema, tool context carries the origin account only, and Feishu falls back to the configured default when that origin account is not a configured enabled Feishu account.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the durable fix needs an approved generic config/SDK contract and account-isolation precedence rather than a Feishu-only field.
