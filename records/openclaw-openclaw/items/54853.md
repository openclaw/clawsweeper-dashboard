## Summary

Keep open. Current main fixes same-channel Feishu session-account priority, but it still lacks a generic per-agent target-channel account binding for a DingTalk-origin agent invoking Feishu tools, so a non-Feishu origin account can still fall back to the configured Feishu default.

## What This Changes

Requests a way to bind an agent to a target-channel accountId so a DingTalk-origin agent can invoke Feishu tools with the correct Feishu account instead of the configured default.

## Best Possible Solution

Define a documented generic target-channel account resolver keyed by active agentId and target channel, wire Feishu tools through explicit accountId first and the approved bound target account before default fallback, and cover the schema/docs/SDK contract with mismatched DingTalk and Feishu account tests.

## Reproduction Assessment

Yes, at source level. The proposed per-agent binding keys are rejected by the strict agent schema, Feishu tools only receive the origin account context, and non-Feishu origin ids fall back to the configured Feishu default.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the maintainable fix needs an approved generic config/SDK contract and account-selection precedence rather than a Feishu-only field.
