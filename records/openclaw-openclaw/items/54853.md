## Summary

Keep open. Current main and v2026.5.22 still do not provide a per-agent target-channel account binding for Feishu tools; existing code only handles explicit tool accountId, valid same-channel context, then Feishu default fallback.

## What This Changes

Request support for binding an agent to a target channel accountId so a DingTalk-origin agent can use the intended Feishu account for Feishu tools.

## Best Possible Solution

Add a generic documented target-channel account resolver, likely keyed by active agentId plus target channel, and wire Feishu tools through explicit accountId first, approved bound Feishu account second, valid same-channel context third, then Feishu default fallback.

## Reproduction Assessment

Yes, at source level. The proposed per-agent keys are rejected by strict agent config, origin sessions pass only the source accountId into plugin tools, and Feishu tool resolution falls back to channels.feishu.defaultAccount when that id is not an enabled Feishu account.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the maintainable fix needs an approved generic config/SDK contract and account-selection precedence rather than a Feishu-only field.
