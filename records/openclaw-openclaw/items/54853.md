## Summary

Keep open. Current main and v2026.5.20 still lack a generic per-agent target-channel account binding; the shipped Feishu session-account resolver only helps when the origin account id is also a configured enabled Feishu account, so the reported DingTalk-to-Feishu mismatch can still fall back to the Feishu default.

## What This Changes

Request support for binding an agent to target-channel accountIds so a DingTalk-origin agent can use the intended Feishu account when invoking Feishu tools.

## Best Possible Solution

Define a generic, documented target-channel account resolver keyed by active agentId plus target channel, wire Feishu tools through it, and add schema, SDK/docs, and regression coverage for mismatched DingTalk and Feishu account ids.

## Reproduction Assessment

Yes, from source inspection. The proposed per-agent keys are still rejected by the strict agent schema, and the Feishu resolver falls back to channels.feishu.defaultAccount when the contextual origin account is not a configured enabled Feishu account.

## Solution Assessment

Unclear. The requested channelBindings shape is plausible, but the maintainable fix needs an approved generic config/SDK contract and account-isolation precedence rather than a Feishu-only feishuAccountId field.
