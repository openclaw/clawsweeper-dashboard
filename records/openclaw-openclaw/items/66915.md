## Summary

Keep open. Current main still suppresses ordinary verbose-off `edit` exact-match mismatch details behind the generic failed-tool warning, and the focused closing implementation at https://github.com/openclaw/openclaw/pull/67276 is still open and conflicting rather than merged.

## What This Changes

The issue asks for clearer user-facing `edit` mismatch errors and mentions disabling `edit` for deployments that prefer `write`-only file changes.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/67276 with a sanitized payload-layer summary and regression coverage; keep broader edit normalization/read-positioning under https://github.com/openclaw/openclaw/issues/62863 and use existing `tools.deny` for disabling `edit`.

## Reproduction Assessment

Yes, at source level. Build an embedded-run payload with an `edit` exact-match mismatch `lastToolError` and verbose mode off; current main will show the mutating-tool failure warning while suppressing the mismatch reason.

## Solution Assessment

Yes for the narrow bug. A sanitized payload-layer summary is the narrowest maintainable fix because it improves the visible warning without changing `edit` semantics or leaking raw file contents.
