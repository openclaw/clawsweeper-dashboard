## Summary

Keep open. Current main still hides ordinary verbose-off `edit` mismatch details behind a generic mutating-tool warning, while the provided GitHub context and remote refs show #67276 is still the open implementation candidate for the narrow warning fix.

## What This Changes

The issue asks for clearer user-facing `edit` failure messaging on exact-text whitespace/newline mismatches and a way to avoid the `edit` tool for reliability-focused deployments.

## Best Possible Solution

Review and land or replace #67276 with a focused sanitized payload-layer exact-match mismatch summary, while keeping `edit` semantics unchanged and using existing `tools.deny` for deployments that want to disable `edit`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an `edit` exact-match mismatch can produce richer lower-level context, but verbose-off payload generation suppresses the detail and emits a generic mutating-tool warning.

## Solution Assessment

Yes. The narrow payload-layer approach in #67276 is the maintainable direction because it can summarize the known mismatch case without changing edit semantics or exposing raw file contents; broader normalization belongs to #62863.
