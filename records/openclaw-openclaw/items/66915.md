## Summary

Keep open. Current main still suppresses ordinary verbose-off `edit` mismatch details behind a generic mutating-tool warning, and PR #67276 remains the open implementation candidate for the narrow payload-layer fix.

## What This Changes

The issue requests clearer user-facing guidance when the `edit` tool fails because `oldText` does not exactly match current file contents, plus a way to disable `edit` in favor of `write`.

## Best Possible Solution

Review and land or replace PR #67276 with sanitized payload-layer exact-match mismatch guidance, keep `edit` semantics unchanged, and use existing `tools.deny` for deployments that want to disable `edit`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an `edit` exact-match mismatch can produce richer lower-level context, but verbose-off payload generation suppresses the detail and emits a generic mutating-tool warning.

## Solution Assessment

Yes. The narrow payload-layer approach in PR #67276 is the maintainable direction because it can summarize the known mismatch case without changing edit semantics or exposing raw file contents; broader edit normalization belongs to #62863.
