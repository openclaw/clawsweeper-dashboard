## Label Justifications

- `P1`: A beta regression can silently perform a state-changing session reset in a real Mattermost chat workflow.
- `impact:message-loss`: The user-visible failure is a missing reset acknowledgement message in the source Mattermost conversation.
- `impact:session-state`: The missing message follows a `/new` or `/reset` command that changes session state.

## Summary

Keeping this open: it is a current beta-blocker with a concrete Mattermost direct-message report, and current source still shows reset acknowledgements can be ordinary final replies that are suppressed in message-tool-only source delivery.

## What This Changes

The issue asks OpenClaw to make Mattermost `/new` and `/reset` always produce a visible acknowledgement or failure message after changing session state.

## Best Possible Solution

Patch the reset command acknowledgement path so authorized bare `/new` and `/reset` replies are explicitly deliverable through source-reply suppression, with focused tests for Mattermost-style message-tool-only delivery and no broader change to verbose reply policy.

## Reproduction Assessment

Yes, at source level. Current main returns normal reset acknowledgement payloads, and dispatch drops unmarked final replies when source delivery is message-tool-only; this matches the reported Mattermost silent-reset path, though I did not run a live Mattermost client.

## Solution Assessment

Yes, the narrowest maintainable direction is to mark reset acknowledgements as deliverable despite source-reply suppression. That should be paired with regression coverage proving `/new` and `/reset` acknowledgements remain visible while ordinary automatic replies stay suppressed.
