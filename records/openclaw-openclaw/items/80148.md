## Summary

Keep open: current main still has the reported stale external-CLI overlay path, but this PR is not safe to merge because it introduces a runtime import cycle and writes OAuth refresh material from read-only and inherited overlay paths; real behavior proof is still insufficient.

## What This Changes

The PR adds a `shouldPersist` marker for external CLI OAuth overlays and writes fresh non-bootstrapOnly CLI credentials back to `auth-profiles.json` from the external-auth overlay helper, with mocked unit coverage.

## Best Possible Solution

Persist fresher matching external CLI OAuth credentials only from write-authorized auth resolution under the owning auth-store lock, while preserving read-only/status/secrets and read-through inheritance contracts.

## Reproduction Assessment

Yes, source-reproducible: current main logs the external CLI bootstrap DEBUG line and returns a runtime-only overlay when a stored `anthropic:claude-cli` OAuth profile is expired and external credentials are fresh. I did not run a live Claude CLI expiry cycle.

## Solution Assessment

No, not as written: persisting fresh external credentials is a plausible narrow fix, but this implementation puts writes in a generic overlay helper that is used by read-only and inherited-profile paths and it breaks the import-cycle gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Remove the runtime auth import cycle:** `src/agents/auth-profiles/external-auth.ts:9`
  - body: Importing `upsertAuthProfileWithLock` here creates a runtime value cycle: `external-auth.ts -> upsert-with-lock.ts -> store.ts -> external-auth.ts`. The PR's `check:import-cycles` job is failing on this exact cycle, so move the write-back into an existing store-owned path or a lower-level helper that does not import back through `store.ts`.
  - confidence: 0.96
- **[P2] Keep read-only auth loads read-only:** `src/agents/auth-profiles/external-auth.ts:78-88`
  - body: This write-back runs inside `overlayExternalAuthProfiles`, but `loadAuthProfileStoreForRuntime(..., { readOnly: true })` and `loadAuthProfileStoreForSecretsRuntime` still call that overlay. A read-only status or secrets load with an expired external CLI profile can now mutate `auth-profiles.json`; thread an explicit write-allowed flag through the path or move persistence to a non-read-only caller.
  - confidence: 0.9
- **[P2] Write inherited OAuth updates to the owning store:** `src/agents/auth-profiles/external-auth.ts:84-87`
  - body: For non-main agents, runtime loading merges the main store into the child store before this overlay runs, but the new upsert writes to `params.agentDir`. If the stale profile was inherited from the main agent, this copies OAuth material into the child store and leaves the main profile stale; resolve the persisted owner or skip persistence for inherited profiles.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The patch broadens OAuth refresh-token writes beyond the existing owner-aware, write-authorized auth path.

Concerns:

- **[medium] Read-only auth loads can persist OAuth credentials:** `src/agents/auth-profiles/external-auth.ts:78`
  - body: Because write-back is triggered from the generic overlay helper, callers that load auth profiles for read-only/status/secrets use can persist fresh external CLI OAuth material into `auth-profiles.json`. That broadens credential writes beyond write-authorized auth paths.
  - confidence: 0.88
- **[medium] Inherited OAuth credentials can be copied into child stores:** `src/agents/auth-profiles/external-auth.ts:84`
  - body: When a child agent inherits a stale main-agent OAuth profile, the PR writes the fresh external credential to the child agent directory. That duplicates refresh-token material across stores despite the read-through inheritance policy for non-portable OAuth profiles.
  - confidence: 0.8
