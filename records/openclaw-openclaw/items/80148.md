## Summary

Keep open. The linked bug is source-reproducible, but the latest PR still writes OAuth material from a generic overlay path used by read-only/secrets and inherited-agent loads, and the claimed runtime proof does not match an emitted log line.

## What This Changes

The PR adds a `shouldPersist` flag for external CLI OAuth overlays and fire-and-forget writes fresh CLI credentials to `auth-profiles.json`, with mocked unit coverage.

## Best Possible Solution

Persist fresher matching external CLI OAuth credentials only from write-authorized auth resolution, under the owning auth store lock, while preserving read-only/status/secrets and inherited-agent behavior.

## Reproduction Assessment

Yes, source-reproducible: current main logs the external CLI bootstrap DEBUG line and returns a runtime-only overlay when a stored `anthropic:claude-cli` OAuth profile is expired and external credentials are fresh. I did not run a live Claude CLI expiry cycle.

## Solution Assessment

No, not as written: persisting the fresh credential is the right narrow direction, but this patch places writes in the generic overlay helper used by read-only and inherited-profile paths. The safer fix should thread write permission and persisted-owner resolution into the write-back path or move persistence to a write-authorized caller.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate write-back out of read-only auth loads:** `src/agents/auth-profiles/external-auth.ts:78-88`
  - body: The new write-back runs inside `overlayExternalAuthProfiles`, but `loadAuthProfileStoreForRuntime(..., { readOnly: true })` and `loadAuthProfileStoreForSecretsRuntime` still call that overlay. A read-only status/secrets load with an expired external CLI profile can now mutate `auth-profiles.json`; thread a write-allowed flag through this path or move persistence to a non-read-only caller.
  - confidence: 0.89
- **[P2] Refresh the owning persisted store for inherited profiles:** `src/agents/auth-profiles/external-auth.ts:84-87`
  - body: For non-main agents, runtime loading merges the main store into the child store before this overlay runs, but the new upsert writes to `params.agentDir`. If the stale profile was inherited from the main agent, this copies OAuth material into the child store and leaves the main profile stale; resolve the persisted owner or skip persistence for inherited profiles.
  - confidence: 0.81

## Security Review

Status: needs_attention

Summary: The patch broadens where OAuth refresh-token material can be written, including read-only and inherited-agent auth paths.

Concerns:

- **[medium] Read-only auth loads can persist OAuth credentials:** `src/agents/auth-profiles/external-auth.ts:78`
  - body: Because write-back is triggered from the generic overlay helper, callers that load auth profiles for read-only/status/secrets use can persist fresh external CLI OAuth material into `auth-profiles.json`. That broadens credential writes beyond write-authorized auth paths.
  - confidence: 0.86
- **[medium] Inherited OAuth credentials can be copied into child stores:** `src/agents/auth-profiles/external-auth.ts:84`
  - body: When a child agent inherits a stale main-agent OAuth profile, the PR writes the fresh external credential to the child agent directory. That duplicates refresh-token material across stores despite the read-through inheritance policy for non-portable OAuth profiles.
  - confidence: 0.78
