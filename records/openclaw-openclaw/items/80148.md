## Summary

Keep open: the PR targets a source-reproducible auth bug, but the latest patch still writes OAuth credentials from a generic overlay path that read-only and inherited-profile callers use, and the supplied runtime proof does not actually demonstrate the patched behavior.

## What This Changes

The PR persists fresh non-bootstrapOnly external CLI OAuth credentials back into `auth-profiles.json` when the stored profile is expired and adds stale/current external OAuth tests.

## Best Possible Solution

Persist fresher matching external CLI OAuth credentials only from write-authorized runtime auth paths, under the owning auth-store lock, while preserving read-only loads and read-through inheritance.

## Reproduction Assessment

Yes, source-reproducible: current main logs the external CLI bootstrap DEBUG line and returns a runtime-only overlay when a stored `anthropic:claude-cli` OAuth profile is expired and external credentials are fresh. I did not run a live Claude CLI expiry cycle in this read-only review.

## Solution Assessment

No, not as written: persisting the fresh credential is the right narrow direction, but this patch places writes in the generic overlay helper used by read-only and inherited-profile paths. A safer fix should thread write permission and persisted-owner resolution into the write-back path or move persistence to a write-authorized caller.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate write-back out of read-only auth loads:** `src/agents/auth-profiles/external-auth.ts:100-111`
  - body: The new persistence runs inside `resolveExternalAuthProfileMap`, which is reached by `loadAuthProfileStoreForRuntime(..., { readOnly: true })` and `loadAuthProfileStoreForSecretsRuntime`. A read-only status/secrets load with an expired external CLI profile can now write `auth-profiles.json`; thread write permission through this path or move persistence to a write-authorized caller.
  - confidence: 0.9
- **[P2] Write back to the owning auth store:** `src/agents/auth-profiles/external-auth.ts:107-110`
  - body: For child agents, runtime loading merges the main store into the child store before this overlay and then passes the child `agentDir`. If the stale profile came from main, this writes refresh-token material into the child store and leaves the main profile stale; resolve the persisted owner or skip inherited profiles.
  - confidence: 0.85

## Security Review

Status: needs_attention

Summary: The diff broadens OAuth credential writes from a runtime overlay path and needs auth-profile security review before merge.

Concerns:

- **[medium] Read-only auth loads can persist OAuth credentials:** `src/agents/auth-profiles/external-auth.ts:100`
  - body: Because write-back is triggered from the generic overlay helper, callers that load auth profiles for read-only/status/secrets use can persist fresh external CLI OAuth material into `auth-profiles.json`. That broadens credential writes beyond write-authorized auth paths.
  - confidence: 0.87
- **[medium] Inherited OAuth credentials can be copied into child stores:** `src/agents/auth-profiles/external-auth.ts:107`
  - body: When a child agent inherits a stale main-agent OAuth profile, the PR writes the fresh external credential to the child agent directory. That duplicates refresh-token material across stores despite the read-through inheritance policy for non-portable OAuth profiles.
  - confidence: 0.82
