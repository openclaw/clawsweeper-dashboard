## Summary

Keep open. The linked bug is source-reproducible on current main, but this PR still writes OAuth material from generic overlay paths that include read-only and inherited-agent loads, and its supplied proof does not convincingly show the real after-fix behavior.

## What This Changes

The PR adds a `shouldPersist` flag for non-bootstrapOnly external CLI OAuth overlays, fire-and-forget writes fresh CLI credentials back to `auth-profiles.json`, and adds mocked unit coverage.

## Best Possible Solution

Persist fresher matching external CLI OAuth credentials only from write-authorized runtime auth paths, under the owning auth-store lock, while preserving read-only/secrets loads and read-through inheritance semantics.

## Reproduction Assessment

Yes, source-reproducible: current main logs the external CLI bootstrap DEBUG line and returns a runtime-only overlay when a stored `anthropic:claude-cli` OAuth profile is expired and external credentials are fresh. I did not run a live Claude CLI expiry cycle.

## Solution Assessment

No, not as written: persisting the fresh credential is the right narrow direction, but this patch places writes in the generic overlay helper used by read-only and inherited-profile paths. A safer fix should thread write permission and persisted-owner resolution into the write-back path or move persistence to a write-authorized caller.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate write-back out of read-only auth loads:** `src/agents/auth-profiles/external-auth.ts:100-111`
  - body: The new persistence runs from `overlayExternalAuthProfiles`, but read-only status/secrets paths still call this overlay through `loadAuthProfileStoreForRuntime(..., { readOnly: true })`. A read-only auth load with an expired external CLI profile can now mutate `auth-profiles.json`; pass a write-allowed flag through this path or move persistence to a non-read-only caller.
  - confidence: 0.89
- **[P2] Write back to the owning auth store:** `src/agents/auth-profiles/external-auth.ts:107-110`
  - body: For child agents, runtime loading merges the main store into the child store before this overlay runs, but this call writes to `params.agentDir`. If the stale profile was inherited from the main agent, the fresh OAuth material is copied into the child store and the main profile remains stale; resolve the persisted owner or skip persistence for inherited profiles.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces credential-write boundary concerns by persisting OAuth refresh material from generic overlay paths.

Concerns:

- **[medium] Read-only auth loads can persist OAuth credentials:** `src/agents/auth-profiles/external-auth.ts:100`
  - body: Because write-back is triggered from the generic overlay helper, callers that load auth profiles for read-only/status/secrets use can persist fresh external CLI OAuth material into `auth-profiles.json`. That broadens credential writes beyond write-authorized auth paths.
  - confidence: 0.86
- **[medium] Inherited OAuth credentials can be copied into child stores:** `src/agents/auth-profiles/external-auth.ts:107`
  - body: When a child agent inherits a stale main-agent OAuth profile, the PR writes the fresh external credential to the child agent directory. That duplicates refresh-token material across stores despite the read-through inheritance policy for non-portable OAuth profiles.
  - confidence: 0.8
