## Summary

Keep open: the reported bug is valid and the persistence direction is plausible, but this patch writes OAuth material from shared overlay paths that current docs/tests treat as read-only and read-through, so it needs changes before merge.

## What This Changes

The PR persists fresh non-bootstrap-only external CLI OAuth credentials back to `auth-profiles.json` when the stored local profile is expired and adds focused unit coverage.

## Best Possible Solution

Persist fresher matching external CLI OAuth credentials only from write-authorized auth resolution, target the owning persisted auth store, preserve read-only/status/secrets and inherited-agent behavior, and add regression coverage plus redacted real runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main overlays a fresh Claude CLI credential when the stored `anthropic:claude-cli` OAuth profile is expired, but the overlay remains runtime-only so the next auth resolution can repeat the bootstrap DEBUG path. I did not run a live Claude CLI refresh cycle in this read-only review.

## Solution Assessment

No, not as written: persisting the fresh credential is the right narrow direction, but placing the write in the generic overlay helper bypasses read-only and persisted-owner boundaries. The safer fix should thread write permission and owner-agent resolution into the write-back path or move persistence to a write-authorized caller.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate write-back out of read-only auth loads:** `src/agents/auth-profiles/external-auth.ts:78-87`
  - body: The new write-back runs inside `overlayExternalAuthProfiles`, but that helper is also used by `loadAuthProfileStoreForRuntime(..., { readOnly: true })` and `loadAuthProfileStoreForSecretsRuntime`. A read-only status/secrets load with an expired external CLI profile can now call `upsertAuthProfileWithLock` and mutate `auth-profiles.json`, violating the read-only external-overlay contract. Thread a write-allowed flag through this path or move persistence into a non-read-only caller.
  - confidence: 0.89
- **[P2] Refresh the owning persisted store for inherited profiles:** `src/agents/auth-profiles/external-auth.ts:84-86`
  - body: For non-main agents, `loadAuthProfileStoreForRuntime` merges the main store into the child store before calling this overlay helper, but the new upsert writes to `params.agentDir`. If the stale profile was inherited from the main agent, this copies OAuth material into the child store instead of refreshing the owner store, leaving the main profile stale and breaking the read-through inheritance contract. Resolve the persisted owner or skip persistence for inherited profiles.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff broadens where OAuth refresh-token material can be written and duplicated.

Concerns:

- **[medium] Read-only auth loads can persist OAuth credentials:** `src/agents/auth-profiles/external-auth.ts:78`
  - body: Because write-back is triggered from the generic overlay helper, callers that load auth profiles for read-only/status/secrets use can persist fresh external CLI OAuth material into `auth-profiles.json`. That broadens credential writes beyond write-authorized auth paths.
  - confidence: 0.86
- **[medium] Inherited OAuth credentials can be copied into child stores:** `src/agents/auth-profiles/external-auth.ts:84`
  - body: When a child agent inherits a stale main-agent OAuth profile, the PR writes the fresh external credential to the child agent directory. That duplicates refresh-token material across stores despite the documented read-through inheritance policy for non-portable OAuth profiles.
  - confidence: 0.78
