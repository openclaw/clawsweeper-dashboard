## Summary

Keep this PR open: the latest head still puts OAuth write-back in the shared external-auth overlay path, which can mutate read-only/status loads and copy inherited refresh material into child agent stores, and the PR still lacks contributor-supplied real behavior proof.

## What This Changes

The PR persists fresher non-bootstrap-only external CLI OAuth credentials back into `auth-profiles.json` when a stored profile is expired and adds focused unit coverage.

## Best Possible Solution

Persist fresher matching non-bootstrap-only external CLI OAuth credentials only from write-authorized auth resolution paths, target the owning persisted auth store, preserve read-only/status/secrets and inherited-agent behavior, and include regression coverage plus redacted real runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main takes the external CLI bootstrap branch when a stored `anthropic:claude-cli` OAuth profile is expired and the Claude CLI credential is fresh, logs the DEBUG bootstrap message, and returns only a runtime overlay. I did not run a live Windows/Claude CLI refresh cycle in this read-only review.

## Solution Assessment

No, not as written: persisting fresh matching credentials is a plausible narrow fix, but doing it inside the generic overlay helper writes from read-only paths and does not respect persisted-profile ownership. The safer solution should thread write permission and owner-agent resolution into the persistence path or move persistence to a write-authorized caller.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate write-back out of read-only auth loads:** `src/agents/auth-profiles/external-auth.ts:78-87`
  - body: The new write-back runs inside `overlayExternalAuthProfiles`, but that helper is also used by `loadAuthProfileStoreForRuntime(..., { readOnly: true })` and `loadAuthProfileStoreForSecretsRuntime`. A read-only status/secrets load with an expired external CLI profile can now call `upsertAuthProfileWithLock` and mutate `auth-profiles.json`, violating the existing read-only external-overlay contract. Thread a write-allowed flag through this path or move the persistence into a non-read-only caller.
  - confidence: 0.89
- **[P2] Refresh the owning persisted store for inherited profiles:** `src/agents/auth-profiles/external-auth.ts:83-86`
  - body: For non-main agents, `loadAuthProfileStoreForRuntime` merges the main store into the child store before calling this overlay helper, but the new upsert writes to `params.agentDir`. If the stale profile was inherited from the main agent, this copies OAuth material into the child store instead of refreshing the owner store, leaving the main profile stale and breaking the read-through inheritance contract. Resolve the persisted owner or skip persistence for inherited profiles.
  - confidence: 0.78

## Security Review
