## Summary

Keep open: the PR targets a real source-reproducible auth log-flood bug, but the current head still persists OAuth material from the generic external-overlay path used by read-only/secrets/status loads and lacks contributor-supplied real behavior proof.

## What This Changes

The PR writes fresher non-bootstrap-only external CLI OAuth credentials back to `auth-profiles.json` when an existing stored profile is expired, and adds unit coverage for Claude CLI stale-profile write-back.

## Best Possible Solution

Persist fresher matching non-bootstrap-only external CLI OAuth credentials only from write-authorized auth resolution paths, target the owning persisted auth store, preserve read-only/status/secrets and inherited-agent behavior, and include regression coverage plus redacted real runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main takes the external CLI bootstrap branch when a stored `anthropic:claude-cli` OAuth profile is expired and the Claude CLI credential is fresh, logs the DEBUG bootstrap message, and returns only a runtime overlay. I did not run a live Windows/Claude CLI refresh cycle.

## Solution Assessment

No, not as written: persisting fresh matching credentials is a plausible narrow fix, but doing it inside the generic overlay helper writes from read-only paths and does not respect persisted-profile ownership. The safer solution should thread write permission and owner-agent resolution into the persistence path or move persistence to a write-authorized caller.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Skip write-back from read-only auth loads:** `src/agents/auth-profiles/external-auth.ts:78-87`
  - body: The new write-back runs inside `overlayExternalAuthProfiles`, but that helper is also used by `loadAuthProfileStoreForRuntime(..., { readOnly: true })` and `loadAuthProfileStoreForSecretsRuntime`. A read-only status/secrets load with an expired external CLI profile can now call `upsertAuthProfileWithLock` and mutate `auth-profiles.json`, violating the existing read-only external-overlay contract. Thread a write-allowed flag through this path or move the persistence into a non-read-only caller.
  - confidence: 0.89
- **[P2] Write inherited profile refreshes to the owner store:** `src/agents/auth-profiles/external-auth.ts:83-85`
  - body: For non-main agents, `loadAuthProfileStoreForRuntime` merges the main store into the child store before calling this overlay helper, but the new upsert writes to `params.agentDir`. If the stale profile was inherited from the main agent, this copies OAuth material into the child store instead of refreshing the owner store, leaving the main profile stale and breaking the read-through inheritance contract. Resolve the persisted owner or skip persistence for inherited profiles.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes OAuth secret persistence and currently writes credential material from paths that are expected to be read-only or inherited at runtime.

Concerns:

- **[medium] Read-only auth loads can persist OAuth credentials:** `src/agents/auth-profiles/external-auth.ts:78`
  - body: Because the write-back is triggered from the generic overlay helper, callers that load auth profiles for read-only/status/secrets use can persist fresh external CLI OAuth material into `auth-profiles.json`. That broadens where credential material is written and should be constrained to write-authorized paths.
  - confidence: 0.86

  - body: When a child agent inherits a stale main-agent OAuth profile, the PR writes the fresh external credential to the child agent directory. That duplicates refresh-token material across stores despite the documented read-through inheritance policy for non-portable OAuth profiles.
  - confidence: 0.76
