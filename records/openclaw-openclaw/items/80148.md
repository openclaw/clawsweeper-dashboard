## Summary

Keep open. The linked bug is source-reproducible, but this PR is not merge-ready because it introduces an auth-profile runtime import cycle, can write inherited credentials to the wrong auth store, and only provides unit-test proof instead of real behavior proof.

## What This Changes

The PR persists fresh external CLI OAuth credentials back into the local auth profile store when an expired stored profile is refreshed from the external CLI.

## Best Possible Solution

Persist fresh matching non-bootstrapOnly external CLI OAuth credentials through an acyclic auth-store write-back seam that resolves the persisted profile owner, then merge after real proof shows the repeated bootstrap DEBUG log stops.

## Reproduction Assessment

Yes, source-reproducible: an expired stored `anthropic:claude-cli` profile plus fresher Claude CLI credentials takes the current `resolveExternalCliAuthProfiles` bootstrap path and returns a runtime-only overlay. I did not run a live Windows/Claude CLI refresh cycle.

## Solution Assessment

No as submitted. Persisting fresher matching credentials is a plausible fix, but the implementation needs an acyclic write-back boundary, correct profile-owner targeting, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Break the new auth-profile import cycle:** `src/agents/auth-profiles/external-auth.ts:11`
  - body: Importing `upsertAuthProfileWithLock` here makes `external-auth.ts -> upsert-with-lock.ts -> store.ts -> external-auth.ts` a runtime value cycle, and CI's `check:import-cycles` is failing on that exact path. Move the write-back seam so `external-auth.ts` does not depend back on the store module.
  - confidence: 0.98
- **[P2] Persist refreshed credentials to the owning store:** `src/agents/auth-profiles/external-auth.ts:84`
  - body: Runtime auth loads can merge the main auth store with an agent-local store before calling this overlay path. If the stale profile is inherited from the main store, passing the requested `agentDir` writes the fresh OAuth credential into the child agent store, leaving the main profile stale and duplicating the secret; resolve the persisted owner before writing back.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no dependency or CI supply-chain changes, but the OAuth write-back can duplicate inherited credentials into the wrong per-agent auth store.

Concerns:

- **[medium] Avoid copying inherited OAuth tokens into child stores:** `src/agents/auth-profiles/external-auth.ts:84`
  - body: When the runtime store inherits a profile from the main auth store, writing with the requested `agentDir` can persist the refreshed OAuth tokens into an agent-local store instead of updating the canonical owner. That broadens local secret copies and leaves the original stale credential in place.
  - confidence: 0.76
