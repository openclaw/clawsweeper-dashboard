## Summary

Keep this PR open for maintainer review. Current main still validates omitted plugin config in the affected non-activating paths, while the PR is a plausible focused fix but intentionally relaxes required-field validation for discovery/CLI metadata loads and needs explicit maintainer approval of that security/design boundary before merge.

## What This Changes

The PR defers validation for `undefined` plugin config during non-activating snapshot and CLI metadata loads, preserves normalized/defaulted `api.pluginConfig`, adds focused loader regression tests, and includes two SwiftFormat-only macOS brace edits.

## Best Possible Solution

Land this PR or an equivalent loader change only after maintainer review confirms the discovery/CLI metadata boundary is safe: omitted config should not create false invalid-config noise, activation and explicit-config paths should remain strict, schema defaults should still populate `api.pluginConfig`, and the sibling log-spam root cause in #64946 should stay separately tracked.

## Reproduction Assessment

Yes. A focused source-level reproduction is a plugin with a strict `configSchema.required` field and no `plugins.entries.<id>.config`, loaded through `loadOpenClawPlugins({ activate: false, loadModules: false })` or `loadOpenClawPluginCliRegistry`; current main validates `undefined` as `{}` before returning metadata.

## Solution Assessment

Unclear pending maintainer security review. Functionally, the PR is a narrow fix for the false positive and preserves activation/explicit-config validation, but maintainers should decide whether discovery registrations may proceed with omitted required runtime config represented by `{}` plus safe defaults.
