## Summary

Keep this PR open. Current main still has the reported false-positive validation path, and the PR is a plausible focused fix, but it intentionally relaxes required-field config validation for non-activating discovery/CLI metadata loads and should get maintainer approval for that boundary before merge.

## What This Changes

The PR defers config schema validation when plugin config is undefined during non-activating snapshot and CLI metadata loads, preserves normalized/defaulted pluginConfig, adds focused loader regression tests, and includes two SwiftFormat-only macOS brace changes.

## Best Possible Solution

Land this PR or an equivalent loader change only after maintainer review confirms the discovery/CLI metadata boundary is safe: omitted config should not create false invalid-config noise, activation and explicit-config validation should remain strict, schema defaults should still populate pluginConfig, and the sibling log-spam root cause in #64946 should stay separately tracked.

## Reproduction Assessment

Yes. A focused reproduction is a plugin with a strict configSchema.required field and no plugins.entries.<id>.config loaded through loadOpenClawPlugins({ activate: false, loadModules: false }) or loadOpenClawPluginCliRegistry; current main validates undefined as {} before returning metadata.

## Solution Assessment

Unclear pending maintainer security review. Functionally, the PR is a narrow fix for the false positive and preserves activation plus explicit-config validation, but maintainers should decide whether discovery registrations may proceed with omitted required runtime config represented by {} plus safe defaults.
