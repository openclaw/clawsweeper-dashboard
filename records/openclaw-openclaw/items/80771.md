## Summary

Keep open. Current main and v2026.5.18 still clamp an exhausted image-description budget to a 1ms timeout, and the two focused fix PRs for this behavior are closed without merging, so the reported bug remains a narrow repair candidate.

## What This Changes

The issue requests that image understanding stop converting an exhausted per-call timeout budget into repeated 1ms aborts during retries.

## Best Possible Solution

Keep the total image-description budget, but make an exhausted budget fail before starting provider work with a clear terminal timeout and focused regression coverage for setup and completion retry exhaustion.

## Solution Assessment

## Repair Work Prompt

Fix the image-understanding exhausted timeout budget bug reported in https://github.com/openclaw/openclaw/issues/80771. Keep the total image-description deadline semantics, but stop converting a negative remaining budget into `1ms`; when the budget is exhausted, fail before starting provider work with a clear terminal timeout that preserves the original budget. Be careful that provider work is currently passed eagerly into `withImageDescriptionTimeout`; the fix must not construct or start `complete()`/provider-stream work after the deadline has already expired. Add focused regression coverage in `src/media-understanding/image.test.ts` for an exhausted budget during the shared startedAtMs path, and update `src/agents/tools/image-tool.test.ts` only if the tool-facing contract changes. Add a user-facing `CHANGELOG.md` fixes entry unless maintainers direct otherwise; do not change retry policy broadly, provider selection, media model config semantics, or unrelated timeout helpers.
