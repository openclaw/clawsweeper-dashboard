## Summary

Keep open. ## What This Changes

The issue reports that image-understanding retries inherit an exhausted timeout deadline and repeatedly fail with `image description timed out after 1ms`.

## Best Possible Solution

Land or replace the linked open timeout-semantics PR with a narrow fix that separates setup/run deadlines from per-image request retries and adds regression coverage for exhausted budgets.

## Reproduction Assessment

Yes from source, though I did not run a live slow vision call: current main still passes a shared `startedAtMs` into image description and clamps exhausted budget to `1`, and the colocated test asserts that expired budgets resolve to `1`.

## Solution Assessment

Yes, the best fix is still narrow code work in the image timeout path: either make exhausted shared deadlines terminal/unrecoverable or give intentional retries a fresh request budget, rather than preserving the 1ms floor.
