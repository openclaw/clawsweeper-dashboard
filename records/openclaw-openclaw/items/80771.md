## Summary

Keep open. Current main still clamps an exhausted image-description deadline to `1ms` and uses that resolved value in the timeout error/provider call path; the full multi-hour production retry loop was not live-reproduced, but the source-level 1ms collapse is real and narrow enough to fix.

## What This Changes

Fix image-understanding timeout handling so an exhausted per-call deadline does not collapse to a 1ms timeout and trigger repeated immediate failures.

## Best Possible Solution

Keep the total image-description budget, but make an exhausted budget produce a clear terminal deadline/timeout result instead of scheduling a 1ms provider call, with regression coverage for the exhausted-budget internal retry path.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows a deterministic path where elapsed time beyond `timeoutMs` is coerced to `1ms`, but the full reported multi-hour agent loop still needs runtime proof or a focused regression test.

## Solution Assessment

Yes, keeping this open for a narrow code fix is appropriate. The safest repair is to avoid the `1ms` clamp for exhausted deadlines and add targeted tests, rather than changing broader agent retry policy first.
