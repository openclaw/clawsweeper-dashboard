## Summary

Keep open. Current main still clamps an exhausted image-description timeout budget to `1ms` and then uses that value for the abort timer/provider timeout path; the two referenced fix PRs are closed unmerged, and no open canonical replacement currently owns the fix.

## What This Changes

The issue reports that exhausted image-understanding retry deadlines are coerced to a 1ms timeout, producing repeated `image description timed out after 1ms` failures.

## Best Possible Solution

Update `src/media-understanding/image.ts` so an exhausted image-description budget produces a clear terminal deadline/timeout result before scheduling a 1ms timer or starting a new provider call, with focused regression coverage for the exhausted-budget retry path.

## Reproduction Assessment

Yes, for the core bug at source level: if elapsed time exceeds `timeoutMs`, the current helper returns `1`, and the timeout wrapper schedules a 1ms abort/error. I did not live-reproduce the reported multi-hour Slack/Anthropic loop.

## Solution Assessment

Yes, a narrow media-understanding timeout fix is the best path. The fix should preserve positive-budget behavior and configured timeout plumbing while making exhausted budgets terminal, rather than changing broad agent retry policy first.
