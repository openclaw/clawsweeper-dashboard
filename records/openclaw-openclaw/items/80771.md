## Summary

Keep open: current main and v2026.5.7 still contain the reported 1ms exhausted-deadline path, and the related closed timeout issues addressed configured timeout propagation rather than this retry-budget collapse.

## What This Changes

The issue requests a fix for image-understanding retries that recompute an exhausted timeout budget as 1ms and then fail immediately in a retry loop.

## Best Possible Solution

Patch the image-description deadline helper so an exhausted operation budget fails with a stable original-timeout/deadline error before starting a retry, and add regression coverage for a retry attempted after the budget is exhausted.

## Reproduction Assessment

Yes, by source inspection: a fake-timer test can make the first image completion return a reasoning-only response after the configured budget has elapsed, then observe the retry resolving to 1ms. I did not run tests because this review is read-only, but the current code path is direct.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/media-understanding/image.ts`: preserve the existing operation-level timeout semantics but throw when the deadline is exhausted instead of clamping to 1ms.

## Repair Work Prompt

Fix the image-understanding deadline-budget bug reported in https://github.com/openclaw/openclaw/issues/80771. In `src/media-understanding/image.ts`, stop `resolveImageDescriptionTimeoutMs` from returning `1` after the original budget has elapsed. Preserve the existing operation-level timeout behavior, but throw a stable timeout/deadline error using the original configured timeout before creating the provider/runtime task when the deadline is exhausted. Keep configured timeout propagation to provider `timeoutMs` options intact. Add focused coverage in `src/media-understanding/image.test.ts` for a reasoning-only first image response where the retry is attempted after the timeout budget is exhausted; assert the retry does not call the provider with `timeoutMs: 1` and the user-facing error is clear. Do not change media config schema, agent retry policy, or unrelated provider behavior. Changelog entry is expected if landing as a user-facing fix.
