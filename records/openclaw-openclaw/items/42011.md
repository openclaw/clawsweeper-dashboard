## Summary

Keep open: current main still has a source-reproducible timeout/teardown path where an embedded run can abort and unsubscribe before a terminal lifecycle event reaches Control UI, leaving the Stop state uncleared.

## What This Changes

Ensure Control UI chat runs always leave the Stop/running state after embedded run timeout or abort by delivering a terminal lifecycle/chat event.

## Best Possible Solution

Add a backend subscription teardown fallback that emits exactly one terminal lifecycle event on timeout/abort teardown, with idempotency coverage against normal `agent_end`.

## Reproduction Assessment

Yes for source-level reproduction: force an embedded timeout/abort so cleanup unsubscribes before `agent_end`; current main has no fallback terminal emission on that teardown path. No live Windows run was executed in this read-only pass.

## Solution Assessment

Yes: the narrowest maintainable fix is at the embedded subscription lifecycle seam, enforcing exactly one terminal event before teardown. A UI `agent.wait` watchdog can help related stuck states but should not replace the backend lifecycle contract.

## Repair Work Prompt

Fix the embedded timeout/abort lifecycle race behind this issue. Current main can call `abortRun(true)`, then run cleanup `unsubscribe()` before the pi-agent subscription receives `agent_end`; `subscribeEmbeddedPiSession.unsubscribe()` must not remove the listener without ensuring exactly one terminal lifecycle event reaches `emitAgentEvent` and `params.onAgentEvent`. Implement the fix in the agents subscription/teardown seam, not as a broad Control UI watchdog. Preserve idempotency with normal `handleAgentEnd`, preserve existing terminal metadata from `setTerminalLifecycleMeta`, and avoid changing run timeout policy, chat protocol shape, or provider behavior. Add focused regression coverage for unsubscribe without `agent_end`, normal `agent_end` plus unsubscribe not duplicating terminal events, and timeout/abort metadata if needed. Add a single-line user-facing changelog fix entry because this affects Control UI behavior. Coordinate with related #73105 as source context and with #42258/#42213 so the backend fix does not conflict with the open UI recovery work.
