## Label Justifications

- `P2`: Scheduled live backups can fail intermittently, but the affected surface is the local backup CLI rather than the core runtime loop.
- `impact:session-state`: The failing paths live under agent session stores and involve backup handling of volatile session-state files.

## Summary

Keep open: current main and the latest release still have a source-backed live-backup ENOENT race. The volatile filter covers the reported path classes only after node-tar successfully stats an entry, while the retry wrapper handles EOF-class tar races rather than ENOENT from a disappearing file.

## What This Changes

`openclaw backup create` should tolerate session-state files disappearing between backup enumeration and archive entry creation.

## Best Possible Solution

Make archive creation tolerate expected volatile per-entry ENOENT before or during tar entry creation, keep manifest/verify metadata coherent, and add regression coverage for disappearing session transcript and usage-cost temp paths.

## Reproduction Assessment

Yes, source-reproducible: node-tar can emit ENOENT from lstat before OpenClaw’s volatile filter runs, and current retry logic does not catch ENOENT. The June 4 shipped-version report matches that path with a disappearing `.usage-cost-cache...tmp` file.

## Solution Assessment

Yes, if scoped narrowly: the best fix is to skip expected volatile per-entry ENOENT without swallowing missing stable backup inputs. Current main’s filter/retry combination is not enough because filtering happens after node-tar stats the entry.
