## Label Justifications

- `P2`: The issue describes intermittent backup failures for live installations with a focused CLI/state blast radius.
- `impact:session-state`: The failing paths are session-store files under agent session directories and affect backup coverage of session state.

## Summary

Keep open: current main still has a plausible ENOENT gap for live backup races. The merged volatile-filter work in https://github.com/openclaw/openclaw/pull/72251 skips known live-mutation files after tar can stat them, but node-tar emits lstat errors before applying the filter, matching the June 4 report against v2026.6.1 for disappearing `.usage-cost-cache*.tmp` files.

## What This Changes

The issue asks for `openclaw backup create` to tolerate disappearing session or volatile files during live backup instead of aborting the whole archive with ENOENT.

## Best Possible Solution

Keep the existing volatile-path policy, but make archive creation skip expected volatile ENOENT races before or during tar entry creation and add regression coverage for `.usage-cost-cache*.tmp` under agent session directories.

## Reproduction Assessment

Yes from source: node-tar can emit ENOENT from lstat before OpenClaw's filter runs, and `writeTarArchiveWithRetry` only retries EOF-class errors. The June 4 report against v2026.6.1 supplies a matching live path class under `agents/<agent>/sessions/.usage-cost-cache*.tmp`.

## Solution Assessment

No: the existing volatile filter is useful but not sufficient for a path that disappears before tar reaches the filter. The narrow maintainable fix is to prefilter or catch expected volatile ENOENT per entry without changing broader backup semantics.
