## Label Justifications

- `P2`: The issue is a normal-priority reliability bug: scheduled live backups can intermittently fail, but the blast radius is limited to backup creation.
- `impact:session-state`: The failing path class lives under agent session stores and affects backup handling of session-state files.

## Summary

Keep open: current main and the latest release still have a source-backed ENOENT race. OpenClaw’s volatile-file filter only runs after node-tar successfully lstats a queued entry, while the retry wrapper only handles EOF-class tar races; the June 4 report confirms the same failure class for disappearing `.usage-cost-cache*.tmp` files on a shipped build.

## What This Changes

The issue asks for `openclaw backup create` to skip session-state files that disappear during archive creation instead of failing the whole backup.

## Best Possible Solution

Make backup archive creation tolerate ENOENT for expected volatile state entries before or during tar entry creation, and add a regression for disappearing usage-cost temp/session transcript paths.

## Reproduction Assessment

Yes, source-reproducible: node-tar can emit ENOENT from lstat before OpenClaw’s volatile filter runs, and current retry logic does not catch ENOENT. I did not run a live race repro in this read-only review, but the June 4 shipped-version report matches the source path.

## Solution Assessment

Yes, if scoped narrowly: the best fix is to skip expected volatile per-entry ENOENT without swallowing unrelated missing stable backup inputs. The current volatile filter alone is not sufficient because it runs after node-tar has already statted the path.
