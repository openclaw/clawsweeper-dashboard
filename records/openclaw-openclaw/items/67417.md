## Summary

Keep open. Current main and the latest release still route `openclaw backup create` through a single `node-tar` archive operation with no per-entry ENOENT skip path, while session maintenance can still rename transcript files during cleanup; open PR #67472 and broader PR #72251 remain unmerged, so this issue should wait for maintainer PR selection or a merged replacement fix.

## What This Changes

The issue asks for `openclaw backup create` to skip session transcript files that disappear during archive creation instead of aborting with ENOENT.

## Best Possible Solution

Land one coherent backup archiver fix in `src/infra/backup-create.ts` that treats expected deleted-file ENOENT during archive creation as non-fatal, reports skipped files clearly, and adds regression coverage for a planned session transcript removed or renamed during the archive walk.

## Reproduction Assessment

Yes. I did not execute the live race in this read-only review, but current source gives a high-confidence path: backup planning includes the state tree, `tar@7.5.13` propagates missing-file lstat errors, and session maintenance can rename transcript files between planning and archive walking.

## Solution Assessment

No, current main is not the best way to solve the issue because it still lets one missing file abort the whole archive. The narrow maintainable fix is per-entry ENOENT handling in the archiver, or a broader volatile-file design that explicitly covers this deleted-transcript case.
