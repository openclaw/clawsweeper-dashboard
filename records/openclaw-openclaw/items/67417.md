## Summary

Keep open. Current main still archives the planned backup sources through one `node-tar` create operation with no ENOENT soft-skip path, while session maintenance can still rename deleted transcript files during the backup window; an open closing PR and a broader open backup-race PR should be reviewed or superseded before this issue closes.

## What This Changes

The issue asks for `openclaw backup create` to continue when a session transcript disappears during archive creation, reporting skipped files instead of aborting with ENOENT.

## Best Possible Solution

Land one coherent archiver fix that treats expected per-entry deleted-file ENOENT as non-fatal, reports skipped files clearly, and adds focused regression coverage for a planned session transcript removed or renamed during archive creation.

## Reproduction Assessment

Yes. I did not execute the live race in this read-only review, but the current source gives a high-confidence reproduction path: backup planning includes the state tree, `tar.c` walks it later, and session maintenance can rename transcript paths in between.

## Solution Assessment

Yes. A per-entry ENOENT soft-skip in the archiver is the narrow maintainable fix; if maintainers choose the broader live-backup approach, it should still explicitly cover this deleted-transcript case.
