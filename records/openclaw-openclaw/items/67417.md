- `P2`: This is a normal-priority backup reliability bug with limited blast radius but real impact for scheduled backups on live installs.
- `impact:session-state`: The failure is triggered by session transcript cleanup racing with backup archive creation.

## Summary

Keep open. Current main and the latest release still route backup creation through `tar.c` with a post-stat volatile filter and an EOF-only retry wrapper, so a session transcript that disappears before node-tar's `lstat` remains a fatal ENOENT path. The issue is also paired with the open closing PR at https://github.com/openclaw/openclaw/pull/67472, so cleanup should wait for that PR or a replacement to land.

## What This Changes

Handle the race where `openclaw backup create` aborts if a session transcript disappears during archive creation.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/67472 with a narrow archive fix that treats expected per-entry ENOENT/ENOTDIR as non-fatal and proves a planned session transcript can disappear during backup creation.

## Reproduction Assessment

Yes, by source inspection: backup create passes state roots to node-tar, node-tar emits lstat ENOENT before the filter can run, and session cleanup can rename transcript files while tar walks the tree. I did not run a live race in this read-only pass.

## Solution Assessment

No, current main is not the complete solution. The merged volatile-filter and EOF retry work handles live append races, but the deleted-before-lstat ENOENT path still needs the linked PR or a safer replacement implementation.
