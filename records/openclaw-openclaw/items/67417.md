## Summary

Keep open. Current main has the adjacent live-backup volatile-file filter and EOF retry, but it still uses `tar.c(...)`; `tar@7.5.15` performs `lstat` before applying the filter, so a session transcript deleted between directory enumeration and stat can still surface the reported ENOENT. The exact closing PR remains open and conflicting, so this issue should wait for that PR to be repaired, replaced, or deliberately superseded.

## What This Changes

Make `openclaw backup create` tolerate session transcript files disappearing during archive creation instead of aborting with ENOENT.

## Best Possible Solution

Repair, rebase, or replace https://github.com/openclaw/openclaw/pull/67472 with a current-main archiver fix that treats expected per-entry ENOENT from disappearing volatile session transcripts as non-fatal, reports skipped files, and adds a regression test for a file removed between directory enumeration and archive stat.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: session maintenance can rename an agent session transcript while `tar.c` is walking the state tree, and `tar@7.5.15` emits stat ENOENT before OpenClaw's filter can skip that entry.

## Solution Assessment

No, current main is not the complete fix for this issue. The merged volatile filter is useful for entries that survive to filtering, but the narrow maintainable solution still needs ENOENT-aware archive handling or a corrected replacement for the open closing PR.
