## Label Justifications

- `P2`: This is a normal-priority backup reliability bug with real impact for scheduled backups but limited blast radius and no reported data loss.
- `impact:session-state`: The race is triggered by session transcript cleanup/rename during backup and prevents reliable backup of session state.

## Summary

Keep open. Current main and the latest release still route backup creation through `tar.c` with only EOF retry/volatile filtering, and node-tar reports a deleted entry at `lstat` before OpenClaw's filter can skip it; the open closing PR at https://github.com/openclaw/openclaw/pull/67472 remains unmerged and conflicting.

## What This Changes

Make `openclaw backup create` skip session transcript files that are deleted or renamed during archive creation instead of aborting with ENOENT.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/67472 with a narrow archiver fix that treats expected per-entry ENOENT/ENOTDIR as non-fatal, reports skipped files, keeps manifest/verify behavior coherent, and adds regression proof for a transcript removed during archive creation.

## Reproduction Assessment

Yes by source inspection. A file listed during node-tar directory traversal and removed before `lstat` emits ENOENT before OpenClaw's volatile filter runs, and current session maintenance can rename transcript files during cleanup.

## Solution Assessment

Yes, fail-open handling for expected per-entry deleted files is the right direction. Current main is not enough, and the open PR needs rebase/rework or replacement plus regression proof before this issue can close.
