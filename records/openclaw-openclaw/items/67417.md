## Label Justifications

- `P2`: This is a normal-priority backup reliability bug with intermittent failures and limited but real operator impact.
- `impact:session-state`: The failure is triggered by volatile session and agent-state files disappearing while backup is archiving them.

## Summary

Current `main` does not fully solve the reported ENOENT backup race. The merged EOF/volatile-filter work reduced adjacent live-backup failures, but the archiver still fails on non-EOF ENOENT errors and the latest reported shell-snapshot path is not covered by the current volatile filter.

## What This Changes

The issue reports that `openclaw backup create` can fail when volatile session or agent-state files disappear while the backup archive is being created.

## Best Possible Solution

Keep this open until backup archive creation either safely tolerates disappearing known-volatile entries or the volatile filter gains a tested policy for all live scratch paths that can disappear during archiving.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-filtered file under the backed-up state tree, such as `agents/<agent>/codex-home/shell_snapshots/*.sh`, can disappear during `tar.c`; the current wrapper treats the resulting ENOENT as non-EOF and fails the backup.

## Solution Assessment

Unclear. The user-visible expectation is valid, but maintainers should choose between a narrow volatile-path policy and a carefully scoped per-entry ENOENT skip that cannot mask missing durable backup content.
