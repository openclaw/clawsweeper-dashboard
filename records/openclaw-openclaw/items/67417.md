## Summary

Keep open. Current main and the latest release still route `openclaw backup create` through one `node-tar` archive operation without an ENOENT-specific skip path, and the issue is already paired with open implementation PRs that should be reviewed or superseded before closure.

## What This Changes

Make `openclaw backup create` continue when a session transcript disappears during archive creation instead of failing the whole backup with ENOENT.

## Best Possible Solution

Land one coherent backup archiver fix in `src/infra/backup-create.ts` that treats expected deleted-file ENOENT during archive creation as non-fatal, reports skipped files clearly, and adds a focused regression test for a planned transcript removed or renamed during the archive walk.

## Reproduction Assessment

Yes. I did not execute the live race in this read-only sweep, but current source gives a high-confidence path: backup planning includes the state tree, `tar.c` later walks it, and session maintenance can rename a transcript path in between.

## Solution Assessment

No, current main is not the best way to solve the issue because it still lets `node-tar` abort the whole archive on missing entries. The narrow maintainable fix is to handle expected per-entry ENOENT in the archiver, or to land a broader volatile-file design that explicitly covers this deleted-transcript case.
