## Label Justifications

- `P2`: This is a normal-priority reliability bug: scheduled live backups can fail, but the core runtime remains usable and the workaround is manual backup/quiescing.
- `impact:session-state`: The failing surface is backup of local agent/session state and volatile session-adjacent files under the OpenClaw state tree.

## Summary

Keep open: current main still has the ENOENT race. `node-tar` stats entries before OpenClaw's volatile filter runs, OpenClaw only retries EOF-class tar errors, and the latest release/current main have the same backup code while the latest issue comment confirms the failure on `v2026.6.6`.

## What This Changes

The issue asks for `openclaw backup create` to tolerate session and other volatile state files disappearing during live archive creation instead of aborting the whole backup.

## Best Possible Solution

Own the archive walk enough to treat ENOENT for expected live volatile state as a reported skip, clean partial temp archives on failure, and add regression coverage for deletion between directory enumeration and lstat, including usage-cost temp files and Codex shell snapshots.

## Reproduction Assessment

Yes, source-reproducible: a file deleted after node-tar directory enumeration but before `fs.lstat` will emit ENOENT before OpenClaw's filter runs, and current retry logic ignores non-EOF errors. I did not run a live command because this review is read-only.

## Solution Assessment

Yes for keeping the issue open; the current filter-plus-EOF-retry implementation is only a partial fix. Adding more retry attempts would not be the best fix because ENOENT is not classified as retryable and node-tar can hit it before OpenClaw gets a filter callback.
