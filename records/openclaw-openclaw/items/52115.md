## Summary

Keep open. Current main still starts dirty search sync in the background and immediately queries existing rows, so the first agent-facing search after memory file create/change/delete can still return stale or ghost-path results; open PR #72748 targets this path but has not landed.

## What This Changes

Make agent-facing `memory_search` refresh long-lived gateway memory indexes so new, changed, renamed, and deleted memory files are reflected without returning stale paths.

## Best Possible Solution

The end state should be a focused memory-core fix that awaits dirty builtin sync before querying, adds first-search create/update/delete and ghost-path coverage, and preserves the existing clean-search and QMD paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use the cached default manager, mark memory dirty via a watched file create/change/unlink, then call agent-facing `memory_search` before the scheduled sync completes; current `search()` starts sync without awaiting and can query stale rows.

## Solution Assessment

Yes. Awaiting dirty builtin sync before the search query is the narrow maintainable fix because watcher state already marks the manager dirty and stale-row pruning already lives in sync; a broader gateway snapshotting feature would not fix the core read path.
