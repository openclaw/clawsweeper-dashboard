## Summary

Keep open: current main and the latest release still start dirty builtin memory search sync in the background, then query the existing index, so the reported first-search stale or ghost-path behavior remains a valid memory-core freshness bug.

## What This Changes

The issue requests that long-running gateway `memory_search` results reflect created, renamed, and deleted memory files before returning paths to the agent.

## Best Possible Solution

Fix memory-core search freshness at the manager boundary so dirty on-search sync completes, or is explicitly serialized within the existing tool deadline, before querying and returning stale paths.

Yes at source level: with an already-populated dirty builtin index, current `search()` starts sync in the background and immediately queries old rows. I did not run the full live gateway timing repro in this read-only pass.

## Solution Assessment

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/52115 in memory-core, not in gateway or CLI wrappers. Current builtin `MemoryIndexManager.search()` only awaits `sync({ reason: "search", force: true })` for an empty index, then calls `startAsyncSearchSync()` which fire-and-forgets dirty sync before querying existing rows; this lets the first `memory_search` after memory file create/delete/rename return stale or ghost paths. Make dirty on-search memory/session sync complete, or otherwise serialize safely within the existing memory_search deadline, before querying results. Preserve existing provider fail-closed behavior, paused index identity handling, search timeout cancellation, and QMD-specific behavior unless a focused sibling test proves it should share the same invariant. Update the async-search test that currently encodes non-awaited sync and add regression coverage for first search after a memory file change and delete. Coordinate with adjacent refs https://github.com/openclaw/openclaw/issues/14770, https://github.com/openclaw/openclaw/issues/56862, https://github.com/openclaw/openclaw/issues/40088, and https://github.com/openclaw/openclaw/issues/90361, but do not add a new config option or broad memory lifecycle refactor. No CHANGELOG.md edit for a normal fix PR.
