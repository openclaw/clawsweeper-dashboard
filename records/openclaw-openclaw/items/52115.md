## Label Justifications

- `P2`: The reported stale memory_search behavior is a normal-priority session-state correctness issue with limited blast radius.
- `impact:session-state`: The issue concerns drift between long-running gateway memory/session state and persisted workspace memory files.

## Summary

Keep open: current main and the latest release still allow the builtin long-lived memory manager to launch dirty search sync in the background and query the old index immediately, so the stale/ghost-path issue remains source-reproducible. The related open memory PRs are adjacent but do not yet merge or own this exact first-search freshness fix.

## What This Changes

The issue reports that gateway-invoked memory_search can serve stale or deleted-path results from a long-lived memory index while the CLI search sees current files.

## Best Possible Solution

Make search-triggered dirty sync a freshness gate for the builtin memory manager, then add regression coverage that the first search after a changed/deleted memory file sees the updated index.

## Reproduction Assessment

Yes at source level: current search-triggered sync is still detached when the manager is dirty, so the first query can read the old index before sync removes stale rows or indexes new files. I did not run a live gateway repro because this is a read-only cleanup review.

## Solution Assessment

Yes, the best repair appears to be in memory-core search freshness semantics rather than CLI or gateway wrappers, because the long-lived manager owns the dirty state and index sync.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/52115. The builtin MemoryIndexManager currently starts search-triggered sync asynchronously when dirty/session-dirty and immediately queries the old index; make the on-search dirty sync path a freshness gate for changed/deleted memory files so the first memory_search after a file write/delete observes the updated index. Keep the fix in memory-core search/sync semantics, do not add config options or broad watcher/reindex policy, and do not change QMD behavior except where the same public manager contract requires it. Add/update focused regression coverage proving a long-lived manager finds a newly changed memory file on the first search and stops returning a deleted/renamed path. If open memory-core PRs touching manager.ts or manager-sync-ops.ts have landed first, rebase against their canonical behavior rather than duplicating their reindex-handle fixes. No CHANGELOG.md edit.
