## Label Justifications

- `P2`: This is a real memory freshness bug with a focused memory-core fix path, but it is limited to stale search results rather than a core outage.
- `impact:session-state`: The issue is about gateway memory/index state drifting from persisted workspace memory files during a long-running session.

## Summary

Keep open: current main and v2026.5.20 still start built-in dirty memory search sync as detached background work before querying, so the first agent-facing search after memory file changes can still observe stale rows. The open linked PR targets this exact fix path but has not merged, so this issue should remain open until that PR lands or is superseded.

## What This Changes

The issue asks for agent-facing `memory_search` to refresh long-lived gateway memory indexes before querying so newly created files appear and deleted paths stop surfacing without a gateway restart.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/72748 with dirty-search sync awaiting, stale-row pruning, and focused create/update/delete first-search regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use the cached default gateway manager, let the file watcher mark memory dirty, then call agent-facing `memory_search` before the scheduled sync completes.

## Solution Assessment

Yes. The narrow maintainable fix belongs in memory-core search semantics: complete dirty sync before querying and cover first-search create/update/delete behavior in regression tests.
