## Summary

Keep open: current main and the latest release still leave the default builtin memory backend able to query a populated but dirty long-lived index while search-triggered sync runs in the background. The KeepMyClaw comment is useful diagnostic context, but the source-visible OpenClaw bug remains; the broad related runtime PR was closed unmerged and the open replacement PR is docs-only wording alignment.

## What This Changes

The issue reports that agent-facing `memory_search` can return stale or deleted file paths after the gateway has been running and workspace memory files change.

## Best Possible Solution

Add a builtin memory-core search freshness guard that completes dirty/session-dirty catch-up or safely reports a transient unavailable state before returning results, with stale-row pruning and first-search create/update/delete regression coverage.

## Reproduction Assessment

Yes by source inspection. Use the cached default builtin gateway manager, let the watcher mark memory dirty after a create/update/delete, then call agent-facing `memory_search`; current code starts sync in the background and queries before catch-up/pruning completes.

## Solution Assessment

Yes. The narrow maintainable fix belongs in builtin memory-core search semantics, not docs or external state snapshots: complete dirty sync before querying, or return an explicit bounded transient state instead of stale paths.

## Repair Work Prompt

Fix the source-reproducible builtin `memory_search` freshness bug tracked by this issue and the overlapping https://github.com/openclaw/openclaw/issues/56862. Current builtin default managers are cached for gateway use; after file watcher dirty state, `MemoryIndexManager.search()` starts `startAsyncSearchSync()` without awaiting it, so the current query can return stale rows or deleted paths. Keep the fix inside the builtin memory manager/search sync boundary; do not change QMD behavior except to preserve existing tests, and do not duplicate the docs-only wording work in https://github.com/openclaw/openclaw/pull/85899. Add regression coverage for first search after create/update/delete of memory files and for stale-row pruning before results are returned. If a full await can hang on slow embedding/reindex work, use existing bounded timeout or transient-unavailable semantics rather than silently returning stale results. Add a changelog entry only if the final PR is user-visible runtime behavior and repo policy requires it.
