## Label Justifications

- `impact:session-state`: The issue is about gateway memory/index state drifting from persisted workspace memory files during a long-running session.

## Summary

Keep open: current main and v2026.5.19 still start dirty memory search sync as detached background work before querying, so the first agent-facing search after memory file changes can still observe stale rows.

## What This Changes

Request that long-running gateway `memory_search` refresh and prune its memory index before returning results so new files appear and deleted or moved paths stop surfacing.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/72748 with dirty-search sync awaiting, stale-row pruning, and focused create/update/delete first-search coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use the cached default gateway manager, let the file watcher mark memory dirty, then call agent-facing `memory_search` before the scheduled sync completes.

## Solution Assessment

Yes. The narrow maintainable fix belongs in memory-core search semantics: complete dirty sync before querying and cover first-search create/update/delete behavior in regression tests.
