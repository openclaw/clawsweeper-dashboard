## Summary

Keep open. Current main still lets the default builtin gateway memory manager start dirty search sync in the background and immediately query existing index rows, so the first agent-facing search after a create/change/delete can still observe stale results; the related PR #72748 is open and targets this path but has not landed.

## What This Changes

The issue asks for agent-facing `memory_search` to refresh long-lived gateway memory indexes before queries so new files appear and deleted or moved files stop returning ghost paths.

## Best Possible Solution

Land PR #72748 or an equivalent focused memory-core change that awaits dirty builtin search sync before querying, adds first-search create/update/delete and ghost-path coverage, preserves clean-search speed, and keeps the already-awaited QMD path intact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with an already-populated builtin index, create/change/unlink a watched memory file, then call agent-facing `memory_search` before the debounced/background sync completes; current `search()` starts sync without awaiting and can query stale rows.

## Solution Assessment

Yes. Awaiting dirty builtin sync before the search query is the narrow maintainable fix because watcher state already marks the manager dirty and stale-row pruning already lives in sync; broader gateway session-state snapshotting may help diagnose drift but is not the core fix here.
