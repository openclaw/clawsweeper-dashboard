## Summary

Keep open. Current main still leaves the builtin/default memory manager path vulnerable to first-search stale results: the agent tool uses a cached default manager, builtin search starts dirty sync in the background, and stale-row pruning happens only inside sync. QMD search awaits dirty sync, but config-missing/default and fallback paths remain builtin, and PR #72748 is open but not landed.

## What This Changes

Agent-facing `memory_search` should refresh a long-running gateway memory index before querying so new, moved, or deleted memory files are reflected without restarting the gateway.

## Best Possible Solution

Land PR #72748 or an equivalent focused memory-core change that awaits dirty builtin search sync before querying, covers first-search create/update/delete and ghost-path behavior, preserves the QMD awaited path, and avoids broad gateway session-state changes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: populate the builtin index, create/change/delete a watched memory file, then call agent-facing `memory_search` before the debounced/background sync completes; current builtin search can query stale rows first.

## Solution Assessment

Yes. Awaiting dirty builtin sync before querying is the narrow maintainable fix because watcher state already marks the manager dirty and stale-row pruning already lives in sync; a broader session snapshot system would diagnose drift but not fix this search freshness path.
