## Summary

Keep open. Current main still starts dirty memory search sync in the background and immediately queries existing index rows, so the first agent-facing search after a watched memory-file create/change/delete can still return stale or deleted-path results; https://github.com/openclaw/openclaw/pull/72748 targets this path but remains open and unmerged.

## What This Changes

The issue asks for agent-facing memory_search to refresh long-running gateway indexes so newly created files appear and deleted or moved files stop appearing without a gateway restart.

## Best Possible Solution

Land a focused memory-core fix that awaits dirty builtin sync before querying, prunes stale rows before results are returned, and adds first-search create/update/delete plus ghost-path regression coverage while preserving clean-search latency.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use the cached default manager, mark memory dirty via a watched file create/change/unlink, then call agent-facing memory_search before the scheduled sync completes; current search starts sync without awaiting it.

## Solution Assessment

Yes. Awaiting dirty builtin sync before the query is the narrow maintainable fix because watcher state already marks the manager dirty and stale-row pruning already lives in the sync path; a broader snapshotting workaround would not fix the core read path.
