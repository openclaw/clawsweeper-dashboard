## Summary

Keep this open. The reported stale-result behavior is still present in the checked-out main and in the newer read-only remote main ref I inspected: the agent-facing tool uses the cached default memory manager, watcher events mark the index dirty, but builtin `search()` starts dirty sync in the background and immediately queries the old rows. GMTekAI's comment accurately narrows this to an async dirty-search refresh race, and open PR #72748 already targets that same path.

## What This Changes

The issue requests that agent-facing `memory_search` refresh long-running gateway memory indexes before querying so created, moved, or deleted memory files are reflected without a gateway restart.

## Best Possible Solution

Land PR #72748 or an equivalent focused memory-core change that awaits dirty builtin sync before querying, covers first-search create/update/delete and ghost-path cases, preserves clean-search performance, and leaves QMD's already-awaited dirty-search path intact.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the default builtin memory backend after indexed content exists: create/change/unlink a memory file so the watcher sets `dirty`, invoke agent-facing `memory_search`, and observe that current `search()` starts sync in the background before querying the old index rows.

## Solution Assessment

Yes. Awaiting dirty builtin sync before querying is the narrowest maintainable fix because watcher dirty state and stale-row pruning already exist; the missing piece is the await boundary before the first dirty search reads the index.
