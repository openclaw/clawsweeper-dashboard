- `impact:session-state`: The issue is about gateway memory/index state drifting from persisted workspace memory files during a long-running session.

## Summary

Keep open. Current main and v2026.5.18 still trigger dirty builtin memory sync asynchronously and then query existing rows; the open fix PR that awaits dirty search sync has not merged.

## What This Changes

Requests that agent-facing `memory_search` refresh long-lived gateway memory indexes before querying so created, deleted, or moved memory files do not produce stale or ghost results.

## Best Possible Solution

Review, land, or replace https://github.com/openclaw/openclaw/pull/72748 so dirty builtin memory sync completes before querying, stale rows are pruned, and first-search create/update/delete coverage protects the behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use the cached default builtin manager, mark memory dirty via file watcher changes, then call agent-facing `memory_search` before the scheduled sync completes.

## Solution Assessment

Yes. The narrow maintainable fix is in memory-core builtin search semantics: complete dirty sync before querying and cover create/update/delete first-search behavior in regression tests.
