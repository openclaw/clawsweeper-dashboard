## Summary

Keep open. Current main still lets the default builtin memory manager start dirty search sync in the background and immediately query existing rows, while the linked fix PR remains open and unmerged.

## What This Changes

The issue asks for agent-facing `memory_search` to avoid stale and deleted-path results in long-running gateway sessions after memory files change.

## Best Possible Solution

Review, land, or replace https://github.com/openclaw/openclaw/pull/72748 so dirty builtin memory sync is awaited or otherwise completed before querying, stale rows are pruned before results return, and first-search create/update/delete coverage proves the behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use the cached default builtin manager, mark memory dirty via watcher add/change/unlink, then call agent-facing `memory_search` before the scheduled sync completes.

## Solution Assessment

Yes. The narrow maintainable fix is in memory-core search semantics: complete dirty builtin sync before querying and rely on the existing sync path to index new files and prune stale rows.
