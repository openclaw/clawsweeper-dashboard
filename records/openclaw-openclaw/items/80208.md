## Summary

Keep open. The SDK import gap is real and related to https://github.com/openclaw/openclaw/issues/73274, but this PR promotes a durable cross-session transcript writer as a raw public SDK subpath without the security/product contract, docs/API-surface alignment, or package-subpath runtime proof this surface needs.

## What This Changes

The PR adds `openclaw/plugin-sdk/transcript.runtime` to the package exports and SDK entrypoint list, then re-exports existing assistant transcript append helpers.

## Best Possible Solution

Design a constrained, documented transcript-append SDK contract with explicit authority semantics, assistant-attributed default writes, idempotency guarantees, update-event behavior, API/export checks, and follow-through for https://github.com/openclaw/openclaw/issues/73274.

## Reproduction Assessment

Yes for the import gap by source inspection: current main omits `transcript.runtime` from both `package.json` exports and `scripts/lib/plugin-sdk-entrypoints.json` while the internal append helper exists. I did not run a package import because this review is read-only.

## Solution Assessment

No. A raw re-export is the shortest path, but durable cross-session transcript writes should be exposed only through a constrained, documented SDK/runtime contract with explicit authority and audit semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate transcript appends before exporting them:** `src/plugin-sdk/transcript.runtime.ts:6-9`
  - body: This re-export makes the internal append helpers a supported public SDK import. The helper can persist assistant messages to any known `sessionKey` or caller-provided `storePath`, and the related API request is explicitly about cross-session transcript writes; expose this only through a constrained, documented capability or authority-gated contract so plugin authors do not get an unaudited durable write path.
  - confidence: 0.86
- **[P3] Document the new SDK subpath:** `scripts/lib/plugin-sdk-entrypoints.json:295`
  - body: Adding a public subpath needs the plugin docs and API/baseline surfaces kept aligned. This PR updates the export map and entrypoint list, but `docs/plugins/sdk-subpaths.md` still has no transcript append contract or guidance for when third-party plugins may write session transcripts.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces a public durable transcript-write SDK surface without explicit authority, consent, or audit semantics.

Concerns:

- **[medium] Gate durable transcript writes:** `src/plugin-sdk/transcript.runtime.ts:7`
  - body: `openclaw/plugin-sdk/transcript.runtime` would make the internal append helpers a supported public import. Because the helper writes persistent transcript content for caller-supplied sessions, this needs an explicit trusted capability, operator opt-in, or equivalent authority model before it becomes a stable SDK contract.
  - confidence: 0.88
- **[medium] Avoid exposing exact-message spoofing as the default path:** `src/plugin-sdk/transcript.runtime.ts:8`
  - body: `appendExactAssistantMessageToSessionTranscript` accepts a full assistant message object, including provider/model/timestamp-like metadata. Even with the assistant-role check, public exact-message writes need attribution and replay semantics before being exposed to arbitrary plugins.
  - confidence: 0.82
