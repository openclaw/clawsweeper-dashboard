## Summary

Keep open: current main and v2026.5.12 still keep the default resumed CLI no-output watchdog capped at 180 seconds, and linked open PRs already overlap the implementation path. The remaining work needs maintainer timeout-policy review rather than cleanup closure.

## What This Changes

The issue asks to raise or disable the CLI no-output watchdog caps so long-running CLI requests are not terminated after about three minutes.

## Best Possible Solution

Land or replace the active watchdog-default PR with maintainer-approved timeout values, focused regression coverage, and redacted real CLI proof, while tracking no-cap and Monitor expected-idle semantics separately if needed.

## Reproduction Assessment

Yes, at source level. Current main still resolves default resumed non-cron CLI runs through CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000 and passes that value into timeout enforcement paths that produce the reported 180-second failure text.

## Solution Assessment

Unclear as a full solution. Raising the default cap is a plausible mitigation, but disabling the cap and coordinating expected-idle semantics require maintainer timeout-policy judgment.
