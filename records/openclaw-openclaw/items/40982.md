## Label Justifications

- `P1`: The issue reports and current tests preserve a user-facing CLI watchdog termination path that can break real long-running agent turns.
- `impact:session-state`: The watchdog can terminate an in-flight resumed CLI session before the agent finishes a long silent turn.
- `impact:crash-loop`: The runtime deliberately kills the backend CLI process on the no-output timeout, creating an availability failure for legitimate silent work.

## Summary

Keep open: current main and the latest release still preserve the ordinary resumed CLI no-output watchdog path that resolves to 180s, while the broader disable/configurable behavior needs maintainer timeout-policy approval before an automated fix should proceed.

## What This Changes

The issue asks to raise, remove, or make configurable the CLI no-output watchdog cap that terminates long-silent resumed CLI sessions after about three minutes.

## Best Possible Solution

Keep this issue as the canonical CLI no-output watchdog policy tracker, with the Monitor and cron reports linked as evidence, until maintainers approve and land the intended default/config behavior.

## Reproduction Assessment

Yes, at source level: call `resolveCliNoOutputTimeoutMs` with `useResume: true`, a 600s timeout, no backend watchdog config, and a normal user trigger; current coverage expects 180s. I did not run a live 180-second CLI stall in this read-only review.

## Solution Assessment

Unclear as proposed. Raising the default is a plausible mitigation, but removing or disabling the cap changes safety/default semantics and should be decided against the existing positive per-backend watchdog override surface.
