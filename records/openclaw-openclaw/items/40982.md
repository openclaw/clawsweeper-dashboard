## Label Justifications

- `P1`: The default can terminate active CLI-backed agent requests and surface as failed or lost sessions for real users.
- `impact:session-state`: The issue concerns in-flight resumed CLI sessions being killed before the agent can complete or preserve the expected conversation flow.
- `impact:crash-loop`: The affected runtime path forcibly terminates the CLI subprocess through the no-output watchdog and reports a timeout failure.

## Summary

Keep open: current main and the latest release still carry the 180-second resumed CLI no-output cap, and open linked PRs already target the timeout policy. The report is source-reproducible, but the remaining choice between raising, disabling, or specializing the watchdog needs maintainer review rather than cleanup closure.

## What This Changes

The issue requests raising or disabling the CLI no-output watchdog cap so long-running resumed CLI-backed agent requests are not killed after about three minutes.

## Best Possible Solution

Review and land or replace the active watchdog PR with maintainer-approved defaults or disable semantics, focused regression coverage, and redacted real CLI proof; keep Monitor expected-idle and closed-stdin behavior explicit if they are separate fixes.

## Reproduction Assessment

Yes, at source level. Current main resolves default resumed non-cron CLI runs through CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000, passes that timeout into runtime paths, and emits the reported 180-second no-output termination text.

## Solution Assessment

Unclear as a complete solution. Raising defaults is a plausible mitigation, but disabling the cap or exempting expected-idle states is a timeout-policy choice that should be reconciled with the open PRs before landing.
