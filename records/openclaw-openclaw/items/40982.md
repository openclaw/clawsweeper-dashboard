## Label Justifications

- `P1`: The shipped watchdog can terminate active CLI workflows at a fixed three-minute silence boundary, and multiple users reported real session-ending failures.
- `impact:session-state`: Termination destroys an in-flight resumed agent session before it can consume tool results or complete the turn.
- `impact:crash-loop`: The watchdog failure can cascade through model fallback and leave the agent workflow unavailable with all candidates failed.

## Summary

Current main and v2026.6.5 still cap resumed CLI silence at 180 seconds, while the resolver can already accept per-backend overrides; multiple reports describe active long-running work being terminated at that exact boundary. This issue should remain open because the user-facing failure remains, the narrower related Monitor report is also open, and the linked implementation PR is open but conflicting and mixes a separate TUI policy change.

## What This Changes

Raise or remove the resumed CLI no-output watchdog ceiling so legitimate long-running silent requests can complete instead of being terminated after about three minutes.

## Best Possible Solution

Preserve the overall request timeout and existing per-backend watchdog policy, then narrowly adjust resumed-session or proven expected-idle handling so legitimate silent work survives while truly stuck processes remain bounded, with focused fresh/resume and live CLI proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: run a resumed CLI request with an overall timeout above three minutes and no stdout for 180 seconds; the resolved watchdog reaches the termination boundary. The issue comments and https://github.com/openclaw/openclaw/issues/71803 provide matching logs, though this read-only review did not wait through a live three-minute run.

## Solution Assessment

No, blanket removal or another generic config key is not yet the best demonstrated fix because current main already supports per-backend overrides. The safer direction is a narrow default or expected-idle correction that preserves the overall timeout and proves both legitimate silence and genuinely stuck-process behavior.
