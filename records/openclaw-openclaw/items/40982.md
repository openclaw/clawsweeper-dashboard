## Summary

Keep open: current main and the latest release still carry the 180-second resumed CLI no-output cap, and active linked PRs are already competing over the timeout policy. The report is source-reproducible, but raising, disabling, or specializing the watchdog needs maintainer policy review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to raise, remove, or make configurable the hard 3-minute CLI no-output watchdog cap that can terminate long silent CLI-backed requests.

## Best Possible Solution

Review and land or replace the active watchdog PR with maintainer-approved defaults or disable semantics, focused regression coverage, and redacted real CLI proof; keep Monitor expected-idle and closed-stdin behavior explicit if they are separate fixes.

## Reproduction Assessment

Yes, at source level. Current main resolves default resumed non-cron CLI runs through CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000, passes that timeout into runtime paths, and emits the reported 180-second no-output termination text.

## Solution Assessment

Unclear as a complete solution. Raising the defaults is a plausible mitigation, but disabling the cap and coordinating expected-idle behavior are timeout-policy choices that should be reconciled with the open PRs before landing.
