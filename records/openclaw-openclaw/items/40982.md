## Label Justifications

- `P1`: The current default can terminate active CLI-backed agent requests and surface as failed or lost sessions for real users.
- `impact:session-state`: The issue concerns in-flight resumed CLI sessions being killed before the agent can complete or preserve the expected conversation flow.
- `impact:crash-loop`: The affected runtime path forcibly terminates the CLI subprocess through the no-output watchdog and reports a timeout failure.

## Summary

Keep open: current main and v2026.5.20 still resolve default resumed non-cron CLI runs through a 180000 ms no-output cap, and open overlapping PRs already target this timeout surface. The report is source-reproducible, but the remaining choice between raising defaults, disabling the cap, or modeling expected-idle states needs maintainer timeout-policy review.

## What This Changes

Raise or remove the 180-second CLI no-output watchdog cap, with optional tuning or disable semantics, so long-running silent CLI-backed requests can complete.

## Best Possible Solution

Land one maintainer-approved watchdog policy update that coordinates the open timeout PRs with the expected-idle reports, adds focused regression coverage, and includes redacted real CLI proof.

## Reproduction Assessment

Yes, at source level. Current main resolves default resumed non-cron CLI runs through `CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000` and emits the reported no-output termination path when that watchdog fires.

## Solution Assessment

Unclear as a complete solution. Raising constants is a plausible mitigation, but removing the cap or adding no-cap and expected-idle semantics changes timeout policy and should be reconciled with the overlapping open PRs first.
