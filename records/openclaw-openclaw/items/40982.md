## Label Justifications

- `P1`: The reported default can terminate active CLI-backed agent requests and surface as failed or lost sessions for real users.
- `impact:session-state`: The issue concerns in-flight resumed CLI sessions being killed before the agent can complete or preserve the expected conversation flow.
- `impact:crash-loop`: The affected runtime path terminates the CLI subprocess through the no-output watchdog and reports a timeout failure.

## Summary

Keep open: current main still carries the default 180-second resumed CLI no-output cap, the latest release has the same cap, and open linked PRs are already competing to change the watchdog policy. The issue is source-reproducible, but the exact default, disable semantics, and relationship to expected-idle workflows need maintainer policy review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to raise, remove, or make configurable the CLI no-output watchdog cap that currently terminates long silent CLI-backed agent requests around three minutes.

## Best Possible Solution

Review and land or replace the active watchdog PR with maintainer-approved timeout defaults, focused regression coverage, and redacted real CLI proof; track no-cap and Monitor expected-idle semantics explicitly if they remain separate decisions.

## Reproduction Assessment

Yes, at source level. Current main resolves default resumed non-cron CLI runs through CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000, passes that timeout into the live-session/supervisor paths, and emits the reported 180-second no-output termination text.

## Solution Assessment

Unclear as a full solution. Raising the defaults is a plausible mitigation, but removing or disabling the cap and coordinating expected-idle behavior are timeout-policy choices that should be reconciled with the open PRs before landing.
