## Label Justifications

- `P1`: The current default can terminate active CLI-backed agent requests and surface as failed or lost sessions for real users.
- `impact:session-state`: The issue concerns in-flight resumed CLI sessions being killed before the agent can complete the expected conversation flow.
- `impact:crash-loop`: The affected runtime path forcibly terminates the CLI subprocess through the no-output watchdog and reports a timeout failure.

## Summary

Keep open: current main and v2026.5.20 still keep the default resumed CLI no-output watchdog at 180000 ms, and real user logs plus source inspection show that it can terminate active CLI-backed sessions. The remaining fix is a timeout-policy decision because open related PRs propose different remedies: raise the caps, disable the watchdog for closed stdin, or add broader watchdog configuration.

## What This Changes

Raise or remove the CLI no-output watchdog's 180-second default cap so resumed CLI-backed requests can stay silent longer without being terminated.

## Best Possible Solution

Adopt one maintainer-approved CLI watchdog policy that covers default caps, no-cap or expected-idle semantics, regression tests, docs/config contract updates, and redacted real CLI proof.

## Reproduction Assessment

Yes, at source level. Current main resolves default resumed non-cron CLI runs through `CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000` and emits the reported no-output termination path when that watchdog fires.

## Solution Assessment

Unclear as a complete solution. Raising constants is a plausible mitigation, but removing the cap or adding no-cap and expected-idle semantics changes timeout policy and should be reconciled with the overlapping open PRs first.
