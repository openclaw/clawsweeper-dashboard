## Label Justifications

- `P1`: The current default can terminate active CLI-backed agent requests and surface as failed or lost sessions for real users.
- `impact:session-state`: The issue concerns in-flight resumed CLI sessions being killed before the agent can complete or preserve the expected conversation flow.
- `impact:crash-loop`: The affected runtime path forcibly terminates the CLI subprocess through the no-output watchdog and reports a timeout failure.

## Summary

Keep open: current main and the latest release still carry the 180-second resumed CLI no-output cap, and an open linked PR already uses closing syntax for this issue. The report is source-reproducible, but the remaining choice between raising defaults, disabling the cap, or specializing expected-idle states needs maintainer timeout-policy review.

## What This Changes

The issue asks to raise or remove the hard 180-second CLI no-output watchdog cap and optionally add no-cap tuning for CLI backends.

## Best Possible Solution

Reconcile and land one maintainer-approved watchdog policy PR with focused regression coverage and redacted real CLI proof, while keeping Monitor expected-idle and closed-stdin behavior explicit if those remain separate fixes.

## Reproduction Assessment

Yes, at source level. Current main resolves default resumed non-cron CLI runs through CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000 and emits the reported 180-second no-output termination text when that watchdog fires.

## Solution Assessment

Unclear as a complete solution. Raising defaults is a plausible mitigation, but no-cap semantics and expected-idle exemptions are timeout-policy choices that should be reconciled with the open PRs before landing.
