## Summary

Keep open: current `main` and v2026.5.12 still enforce the default resumed CLI no-output watchdog through a 180-second cap, and the linked implementation PRs are open or unmerged. This needs maintainer timeout-policy review rather than cleanup closure.

## What This Changes

The issue asks to raise or remove the default three-minute no-output watchdog cap for CLI requests so long silent agent runs are not terminated prematurely.

## Best Possible Solution

Review and land or replace the active watchdog-default PR with maintainer-approved timeout values, focused regression coverage, and redacted real CLI proof, while tracking no-cap and Monitor expected-idle semantics separately if needed.

## Reproduction Assessment

Yes, at source level. Current main still resolves default resumed non-cron CLI runs through `CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000` and passes that value into timeout enforcement paths that produce the reported 180-second failure text.

## Solution Assessment

Unclear as a full solution. Raising the default cap is a plausible mitigation, but disabling the cap and coordinating expected-idle semantics require maintainer timeout-policy judgment.
