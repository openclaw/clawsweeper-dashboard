## Label Justifications

- `P1`: The issue describes active CLI-backed agent sessions being terminated mid-run and cascading into end-user all-model failure.
- `impact:session-state`: The reported watchdog kill loses or interrupts in-flight resumed CLI agent session state and transcript continuity.
- `impact:crash-loop`: The no-output watchdog terminates the CLI child process and can surface as a process-level agent failure.

## Summary

Keep open: current main and the latest release still preserve the ordinary resumed CLI no-output path that can resolve to 180s, and the open implementation path is still unmerged/conflicting with product-sensitive default/config tradeoffs.

## What This Changes

The issue requests raising, removing, or making configurable the CLI no-output watchdog cap so long silent CLI requests are not killed after about three minutes.

## Best Possible Solution

Keep this issue as the canonical CLI no-output watchdog policy tracker, then either refresh/split the open PR or land a narrower maintainer-approved change with bounded defaults, explicit operator-controlled semantics, docs, and regression proof.

## Reproduction Assessment

Yes, at source level: current main's resolver test proves an inherited resumed user run with a 600s outer timeout still resolves to the 180s no-output cap. The issue discussion also includes matching user logs from openclaw 2026.4.26.

## Solution Assessment

No, not as a blanket no-cap change yet: removing the watchdog entirely could hide wedged CLI processes. The better direction is a maintainer-approved bounded default or explicit operator-controlled mode with regression proof for both legitimate silence and real stalls.
