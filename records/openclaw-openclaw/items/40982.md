## Summary

Keep open: current main and v2026.5.22 still route default resumed CLI runs through a 180000 ms no-output ceiling, and there are still open related PRs/issues targeting the same timeout policy surface.

## What This Changes

Raise or remove the CLI no-output watchdog's three-minute default cap, with optional no-cap/tuning semantics for long silent CLI-backed agent requests.

## Best Possible Solution

Land one maintainer-approved watchdog policy update that coordinates the open timeout PRs with expected-idle behavior, preserves a bounded stuck-process guard or explicit operator override, and adds focused regression coverage plus redacted real CLI proof.

## Reproduction Assessment

Yes, at source level. Current main resolves default resumed non-cron CLI runs through `CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000` and emits the reported no-output termination path when that watchdog fires.

## Solution Assessment

Unclear as a complete solution. Raising constants is a plausible mitigation, but removing the cap or adding no-cap and expected-idle semantics changes timeout policy and should be reconciled with the overlapping open PRs first.
