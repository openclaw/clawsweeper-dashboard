## Summary

Keep open: current main and the latest release still keep ordinary inherited resumed CLI runs on the 180s no-output cap, while the merged timeout-intent work only covers cron or explicit run timeouts. The remaining default/config policy is still unresolved and the related implementation path at https://github.com/openclaw/openclaw/pull/79997 is open but conflicted and marked for maintainer review.

## What This Changes

The issue requests raising, removing, or making configurable the 3-minute CLI no-output watchdog cap so long silent CLI-backed requests are not killed mid-run.

## Best Possible Solution

Keep this issue as the canonical watchdog policy tracker, and land or replace the linked implementation with a maintainer-approved default/config shape plus focused resolver coverage and real behavior proof for long silent CLI work.

## Reproduction Assessment

Yes, at source level: current main still defines `CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180_000`, and a resolver test asserts an inherited resumed user run with a 600s outer timeout returns `180_000`. I did not run a live 180-second silent CLI scenario in this read-only review.

## Solution Assessment

Unclear as a product decision. Raising the inherited resume cap is plausible, but removing or disabling the watchdog, or adding a new null/0 disable mode, changes availability and stuck-process policy; the safest path is maintainer-approved defaults/config plus proof rather than an automatic cleanup close.
