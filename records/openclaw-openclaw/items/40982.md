Keep open: current main and the latest release still retain the 180-second resumed CLI no-output cap, and a live closing PR is already the active implementation path. This issue should stay open until maintainers land, replace, or close that PR and reconcile the overlapping watchdog-policy work.

## What This Changes

Request to raise or remove the CLI no-output watchdog defaults so long-running resumed CLI requests are not killed at about 180 seconds.

## Best Possible Solution

Review, fix, and land or replace the active watchdog-default PR with updated tests and redacted real CLI proof, while tracking no-cap and expected-idle semantics as separate timeout-policy work if needed.

Unclear as a full solution. Raising the default cap is a plausible narrow mitigation, but removing or disabling the cap and coordinating Monitor expected-idle semantics need maintainer timeout-policy judgment.
