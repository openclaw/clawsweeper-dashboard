## Summary

Keep open: current main and the latest release still preserve the reported 180-second no-output cap for ordinary resumed CLI user runs. A partial merged fix now lifts cron/explicit-timeout resume paths, but the central default-cap policy remains unresolved and has open/unmerged fix candidates.

## What This Changes

The issue asks to raise, remove, or make configurable the CLI no-output watchdog cap that terminates resumed CLI sessions after about 180 seconds of silence.

## Best Possible Solution

Keep this canonical issue open and have maintainers choose a refreshed, proof-backed watchdog default/config fix, likely by repairing or replacing the current open PR rather than spawning another duplicate branch.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main still resolves an ordinary resumed user run with a 600000 ms overall timeout to a 180000 ms no-output watchdog, and that value is passed into the runtime timeout path. I did not run a live 180-second CLI repro in this read-only review.

## Solution Assessment

No complete solution is on main yet. A bounded default increase using the existing per-backend watchdog surface is the narrowest likely fix, while null/0 disable behavior or new config semantics require maintainer approval and upgrade proof.
