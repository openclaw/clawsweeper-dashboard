## Summary

Keep open: current main and v2026.5.12 still keep the resumed CLI no-output watchdog capped at 180 seconds, and an open PR with closing syntax remains the active implementation path.

## What This Changes

The issue requests raising or removing the CLI no-output watchdog's 180-second cap and optionally allowing operators to tune or disable that cap.

## Best Possible Solution

Review, land, or replace the active watchdog-default PR with real CLI proof, while handling no-cap semantics and Monitor expected-idle behavior as explicit follow-up decisions.

## Reproduction Assessment

Yes, at source level. Current main resolves default resumed non-cron CLI runs through CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000 and enforces that value in the no-output termination path; reporter logs match the 180-second timeout.

## Solution Assessment

Unclear as a full solution. Raising the default cap is a narrow mitigation, but disabling the cap or changing idle semantics needs maintainer policy review and should not be hidden inside stale cleanup.
