## Summary

Keep this open: current main and v2026.6.6 still keep ordinary inherited resumed CLI runs on the 180-second no-output cap. A merged partial fix now honors configured or explicit long timeout intent, but the default/disablement policy remains unresolved and the related implementation PR is still open, conflicting, and proof-sensitive.

## What This Changes

The issue asks OpenClaw to raise, remove, or make configurable the 180-second no-output watchdog cap for CLI-backed resumed requests so long silent tasks are not terminated.

## Best Possible Solution

Keep this issue open as the canonical policy item until maintainers choose the default/disablement behavior, then land a focused implementation with resolver coverage, config/schema/docs alignment if needed, and real long-silent CLI proof.

## Reproduction Assessment

Yes, at source level: current main still resolves an ordinary inherited resumed user run with a 600-second outer timeout to a 180-second no-output cap. I did not run a live 180-second CLI stall because this review is read-only and the resolver/test path already proves the remaining behavior.

## Solution Assessment

Unclear: raising or disabling the cap is plausible, but the right default and escape-hatch behavior is an availability/product decision. The safer path is to review or replace the open implementation PR with a narrower maintainer-approved change that preserves stuck-process protection where needed.
