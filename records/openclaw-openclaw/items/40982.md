## Summary

Keep this issue open: current main still has the 180s resumed-session no-output cap, and an open PR with closing syntax directly targets the same watchdog-default change, so the issue should wait for that PR to be reviewed, landed, or closed.

## What This Changes

The issue requests raising or disabling the CLI no-output watchdog cap so long-running resumed CLI sessions are not terminated after roughly three minutes.

## Best Possible Solution

Review and either land or close PR #41022; if maintainers want a disabled-watchdog mode, track that as a narrower follow-up after the default-cap decision.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a resumed CLI backend with a long global timeout still resolves to a 180_000 ms no-output watchdog, and the runner turns that supervisor timeout into the reported termination error.

## Solution Assessment

Unclear as a final product decision. Raising defaults in the linked PR is a narrow, plausible mitigation, but disabling the watchdog entirely needs explicit maintainer agreement and should not be implied by closing this issue early.
