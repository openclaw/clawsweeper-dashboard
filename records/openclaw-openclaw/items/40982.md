## Summary

Keep this open: current main and the latest release still preserve the 180-second no-output cap for ordinary inherited resumed CLI runs. A merged partial fix now lets deliberate/configured long timeouts scale, but the broader default/config policy remains unresolved and the related implementation PR is still open and conflicting.

## What This Changes

The issue requests raising, disabling, or making configurable the CLI no-output watchdog so resumed CLI requests are not killed after about 180 seconds of silence.

## Best Possible Solution

Keep this issue open as the canonical policy item until maintainers decide the CLI watchdog default/disablement behavior, then land a focused implementation with resolver coverage and real long-silent CLI proof.

## Reproduction Assessment

Yes, at source level: current main still resolves ordinary inherited resumed user runs with a 600-second outer timeout to a 180-second no-output cap. I did not run a live 180-second CLI stall because this review is read-only and the resolver/test path already proves the remaining behavior.

## Solution Assessment

No final fix is selected yet: raising or disabling the cap is plausible, but the right default and escape-hatch behavior is an availability/product decision. The safest path is to review the open PR or replace it with a narrower maintainer-approved change that preserves stuck-process protection where needed.
