## Summary

Keep open. Current main and latest v2026.4.29 still compute and enforce a 180s resumed CLI no-output timeout, and #41022 is an active closing-syntax PR rather than a reason to close before merge.

## What This Changes

Raise or remove the CLI no-output watchdog cap so long-running fresh or resumed CLI requests are not terminated after about 180 seconds of silence.

## Best Possible Solution

Ship a narrow CLI watchdog fix that removes the practical 180s cap for long-running CLI sessions, keeps defaults/schema/types/tests aligned, and explicitly scopes no-cap or Monitor idle exemptions into the patch or a linked follow-up.

## Reproduction Assessment

Yes. With a resumed CLI backend, default watchdog settings, and an overall timeout above 600000ms, the resolver computes 180000ms and both supervisor and Claude live-session paths enforce that as a no-output timeout; the Apr 29 user log in the discussion matches this path.

## Solution Assessment

Yes for triage. Keeping the issue open while reviewing #41022 is the narrow maintainable route; the final implementation should deliberately decide whether disabling the cap belongs in the same fix or a follow-up.
