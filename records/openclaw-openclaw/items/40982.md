## Label Justifications

- `P1`: The issue describes real long-running CLI-backed agent sessions being terminated at 180 seconds and surfacing as failed user work.
- `impact:session-state`: The watchdog termination can destroy in-flight CLI-backed agent sessions before pending work results are consumed.
- `impact:crash-loop`: The reported failure mode terminates the CLI process during otherwise healthy long-running work.

## Summary

Keep open. Current main and v2026.6.8 still preserve the inherited resumed CLI 180-second no-output cap; the merged timeout-intent work only fixes explicit/configured timeout paths, and the broader linked implementation PR remains open, conflicting, and product/config-sensitive.

## What This Changes

The issue asks OpenClaw to raise, remove, or make configurable the CLI no-output watchdog cap that kills long silent resumed CLI agent requests at about 180 seconds.

## Best Possible Solution

Keep this issue as the canonical watchdog policy tracker and resolve it with a refreshed, maintainer-approved fix that either raises the inherited resume default or adopts an explicit state/config policy without bundling unrelated TUI work unless maintainers choose that broader scope.

## Reproduction Assessment

Yes, at source level. Current main still resolves an inherited resumed user CLI run with a 600000 ms outer timeout to a 180000 ms no-output watchdog, and the runtime emits the reported termination text when that watchdog fires.

## Solution Assessment

Unclear as a final implementation choice. The bug is real, but maintainers still need to choose the policy tradeoff between raising defaults, honoring explicit overrides more broadly, disabling the watchdog in selected states, or adding a no-cap mode.
