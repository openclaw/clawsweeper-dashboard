## Label Justifications

- `P1`: The issue reports real CLI-backed agent turns being terminated mid-request at the watchdog cap, causing user-visible model-run failure.
- `impact:session-state`: Killing a resumed CLI process mid-turn can strand or lose in-flight session context before the agent can finish.
- `impact:crash-loop`: The reported timeout terminates the CLI process and can cascade into all model candidates failing for the prompt.

## Summary

Keep open: current main and the latest stable release still preserve the 180s default cap for ordinary inherited resumed CLI runs. Recent code lifts cron and explicit-timeout paths, but the remaining default-policy/config work is not implemented and the open implementation PR still needs maintainer review and real behavior proof.

## What This Changes

The issue requests raising, removing, or making configurable the CLI no-output watchdog cap so long silent resumed CLI requests are not killed after about three minutes.

## Best Possible Solution

Decide the supported CLI no-output timeout policy, then land a focused fix that deliberately adjusts fresh/resume defaults or existing CLI watchdog config semantics, documents any disable behavior, and proves ordinary resumed runs plus cron/explicit timeout paths.

## Reproduction Assessment

Yes, at source level: current main still resolves ordinary inherited resumed CLI runs with `CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000` and passes that timeout into the no-output termination path. I did not run a live 180-second silent CLI session in this read-only review.

## Solution Assessment

Unclear as a final implementation: raising the cap is a plausible mitigation, but the safest permanent answer needs maintainer choice on default stall-detection latency, disable semantics, and upgrade proof.
