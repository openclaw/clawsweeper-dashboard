## Label Justifications

- `P1`: The issue affects real CLI-backed agent runs by terminating long-running quiet sessions and surfacing model-fallback failures to users.
- `impact:session-state`: A no-output watchdog kill can drop an in-flight resumed CLI session before the agent completes the turn.
- `impact:crash-loop`: The reported failure terminates the supervised CLI process and can cascade into repeated fallback or apparent runtime failure.

## Summary

Keep open: current main and the latest release still keep the default resumed CLI no-output cap at 180 seconds for inherited user resumes, while the merged timeout work only covers explicit cron/run-timeout cases and the open broader PR is not yet a clean landed fix.

## What This Changes

The issue requests raising, removing, or making configurable the CLI no-output watchdog caps so long-running silent CLI requests are not killed after about three minutes.

## Best Possible Solution

Land a maintainer-approved narrow fix that defines the intended CLI watchdog default/config behavior, covers inherited user resumes plus explicit-timeout cases in resolver tests, and coordinates with https://github.com/openclaw/openclaw/pull/79997 and https://github.com/openclaw/openclaw/issues/68596 rather than closing this report early.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current tests assert inherited user resumed runs still resolve to 180_000 ms, and the resolver/defaults show why. I did not run a live 180-second silent CLI process in this read-only review.

## Solution Assessment

No, the issue is not fully solved on current main; PR https://github.com/openclaw/openclaw/pull/90912 only lifted explicit timeout paths. The best fix needs a maintainer decision on safe default ceilings and whether disabling/tuning semantics should be exposed or narrowed to existing backend watchdog config.
