## Label Justifications

- `P1`: The issue reports real CLI-backed agent turns being terminated mid-run and surfacing as failed model/session workflows.
- `impact:session-state`: The watchdog termination can kill the in-flight CLI session before the agent completes or preserves the turn state.
- `impact:crash-loop`: The failure is a process-level watchdog termination that can surface as unavailable or failed agent runs.

## Summary

Keep open: current main and the latest release still keep the ordinary resumed-user CLI no-output watchdog capped at 180 seconds, while only cron or explicit-run-timeout resume paths now lift that ceiling. The linked open PR is a possible fix path, but it is broader, conflicting, and not merged.

## What This Changes

The issue requests raising, removing, or making configurable the CLI no-output watchdog cap that terminates long silent CLI-backed agent requests after about three minutes.

## Best Possible Solution

Keep this canonical issue open until maintainers choose and land a proof-backed CLI watchdog default/config policy, likely by narrowing or repairing the open fix candidate rather than creating another duplicate branch.

## Reproduction Assessment

Yes, at source level: current main resolves an ordinary resumed user run with a 600000 ms overall timeout to a 180000 ms no-output watchdog and passes that value into runtime timeout paths. I did not run a live 180-second CLI repro in this read-only review.

## Solution Assessment

No complete solution is on main yet. A bounded default increase using the existing per-backend watchdog surface is the narrowest likely mitigation, while null/0 disable behavior or a broader user-facing config knob needs maintainer product approval and upgrade proof.
