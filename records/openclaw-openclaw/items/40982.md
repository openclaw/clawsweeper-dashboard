## Summary

Keep open. Current main still has the 180-second resumed CLI no-output cap, the latest release tag shows the same behavior, and open PR #41022 is an active closing-syntax implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests raising, removing, or making configurable the CLI no-output watchdog defaults that cap resumed CLI requests at about three minutes.

## Best Possible Solution

Ship a narrow CLI watchdog fix that removes the practical 180-second cap for long-running CLI sessions, keeps defaults, schema/types, tests, and plugin-exported backend behavior aligned, and deliberately scopes no-cap or Monitor expected-idle behavior either into that patch or a clearly linked follow-up.

## Reproduction Assessment

Yes. With a resumed CLI backend, no watchdog override, and an overall timeout above 600000ms, the current resolver computes 180000ms and the runner/live-session paths enforce it as a no-output timeout; the Apr 29 production log matches that path.

## Solution Assessment

Yes for triage: the issue should stay open while #41022 is reviewed, landed, revised, or replaced. The implementation direction is plausible, but the final patch should explicitly decide whether disabling the cap belongs in the same fix or a follow-up.
