## Label Justifications

- `P1`: The reported watchdog timeout can terminate live CLI-backed agent runs during legitimate long silent work and surface as failed model/agent runs.
- `impact:crash-loop`: The issue concerns process-level availability because the CLI runner terminates the backing process on a no-output timeout.

## Summary

Keep open: current main and v2026.6.8 include a partial fix for configured or explicit resumed CLI timeouts, but ordinary inherited resumed CLI runs are still source-proven to resolve to the 180000 ms no-output cap, and the related open PR is still dirty and unproven.

## What This Changes

The issue asks to raise, remove, or make configurable the CLI no-output watchdog cap that can terminate silent long-running CLI requests after roughly three minutes.

## Best Possible Solution

Keep this issue open as the canonical CLI no-output watchdog policy bug until maintainers land a refreshed, proven fix that preserves stuck-process protection while allowing legitimate long silent CLI work.

## Reproduction Assessment

Yes, at source level. Current main still resolves ordinary inherited resumed user runs through the 180000 ms resume watchdog ceiling, and the runtime emits the same no-output termination error when that timer fires.

## Solution Assessment

No complete solution is on main yet. The shipped partial fix handles explicit/configured timeout provenance, but the remaining default cap and any disable/tuning semantics need a maintainer-owned timeout policy with proof.
