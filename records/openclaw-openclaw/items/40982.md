## Label Justifications

- `P1`: The issue reports real CLI-backed agent turns being killed mid-run and surfacing as failed model/session workflows.
- `impact:session-state`: The watchdog termination can kill an in-flight CLI session before the agent completes or preserves the turn state.
- `impact:crash-loop`: The failure is a process-level watchdog termination that makes agent runs unavailable or failed during long silent work.

## Summary

Keep open: current main and v2026.6.8 still preserve the reported 180-second no-output cap for ordinary resumed CLI user runs, while related work only partially fixes cron/explicit-timeout paths or remains an open blocked PR.

## What This Changes

Raise or remove the hard 180-second CLI no-output watchdog cap so long silent CLI-backed agent requests are not killed prematurely.

## Best Possible Solution

Keep this canonical issue open until maintainers choose a proof-backed watchdog policy, ideally by refreshing or replacing the open PR with a narrow default/config change and upgrade evidence.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main resolves an ordinary resumed user run with a 600_000 ms overall timeout to a 180_000 ms no-output watchdog, and that value is passed into the runtime timeout paths. I did not run a live 180-second CLI stall in this read-only review.

## Solution Assessment

No complete solution is on main yet. A bounded default increase using the existing per-backend watchdog surface is the narrowest likely code fix, while disable/null semantics or a new public knob need maintainer approval and upgrade proof.
