## Label Justifications

- `P1`: The default can terminate active CLI-backed agent requests and surface as failed or lost sessions for real users.
- `impact:session-state`: The issue concerns in-flight resumed CLI sessions being killed before the agent can complete or preserve the expected conversation flow.
- `impact:crash-loop`: The affected runtime path forcibly terminates the CLI subprocess through the no-output watchdog and reports a timeout failure.

## Summary

Keep open: current main and the latest tagged release still keep the default resumed CLI no-output watchdog capped at 180000 ms, and open related PRs are already trying to change the same timeout policy surface.

## What This Changes

The issue asks OpenClaw to raise, remove, or make configurable the 3-minute no-output watchdog cap for long-running CLI-backed agent requests.

## Best Possible Solution

Land one maintainer-approved watchdog policy update that coordinates the open timeout PRs, preserves a bounded stuck-process guard or explicit operator override, and adds focused regression coverage plus redacted real CLI proof.

## Reproduction Assessment

Yes, at source level. Current main resolves default resumed non-cron CLI runs through CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000 and emits the reported no-output termination path when that watchdog fires.

## Solution Assessment

Unclear as a complete solution. Raising constants is a plausible mitigation, but no-cap or expected-idle semantics change timeout policy and should be reconciled with the overlapping open PRs first.
