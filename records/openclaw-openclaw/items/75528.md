## Summary

Keep open. The PR takes the right narrow approach by adding a stable caller-selected live model default, but the latest head still omits two fallback-capable embedded runner entrypoints, so direct agent commands and queued follow-up runs can still compare a pending live switch against the active fallback candidate instead of the caller-selected model.

## What This Changes

The branch adds live-model default parameters to embedded PI agent runs, uses them in live-session switch checks, threads them through cron and auto-reply runner paths, adds regression tests, and updates the changelog.

## Best Possible Solution

Complete the stable live-default contract across every fallback-capable embedded runner entrypoint, then keep the current helper-level approach with focused tests for cron, heartbeat/auto-reply, queued follow-ups, direct agent commands, and the core live switch helper.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a direct `openclaw agent --model` run or queued follow-up where the first fallback candidate differs from the caller-selected model while `liveModelSwitchPending` is set; the omitted callsites still pass only the active fallback provider/model into runEmbeddedPiAgent.

## Solution Assessment

No. The new liveModelDefaultProvider/liveModelDefaultModel fields are the right narrow seam, but this PR is not the best complete fix until every fallback-capable caller passes the original caller-selected provider/model separately from the active fallback candidate.
