## Summary

Keep open. The patch is moving in the right direction, but it still leaves the isolated cron embedded-runner path using the active fallback candidate as the live-session default, so one of the PR's stated cron cases can still clear a pending switch instead of returning to the caller-selected run model.

## What This Changes

The PR adds live-model default threading through embedded agent runs, updates focused live-switch and auto-reply fallback tests, and adds a changelog fix entry for explicit cron/heartbeat model handling.

## Best Possible Solution

Keep the PR open and finish the fix by threading a stable caller-selected live default through every fallback-capable embedded runner entrypoint, especially isolated cron, with a regression test proving reset-to-default during a fallback attempt returns to the selected cron model.

## Reproduction Assessment

Yes. A deterministic code-level reproduction is an isolated cron run whose selected model falls back to another candidate while `liveModelSwitchPending` is set after a reset-to-default; with the current PR shape, the cron caller does not pass stable defaults, so the switch check can treat the fallback candidate as the default and clear the pending switch.

## Solution Assessment

No. The auto-reply path is closer, but the safest maintainable fix is to pass the original run default separately from transient fallback candidates for cron as well, or make the new embedded-runner default contract explicit enough that missing callers cannot silently regress.
