## Summary

Keep open. The PR targets a real live-session model propagation bug, but the branch still leaves the active embedded-run caller passing product defaults, so the new resolver can mistake `openai/gpt-5.5` for an explicit caller override when an agent primary differs from the product default.

## What This Changes

The PR changes `src/agents/live-model-switch.ts` so live-session model resolution compares caller defaults with agent defaults and prefers caller defaults when they differ.

## Best Possible Solution

Thread the actual initial run provider/model from `src/agents/pi-embedded-runner/run.ts` into `shouldSwitchToLiveModel`, make `resolveLiveSessionModelSelection` honor those explicit caller defaults without regressing normal agent-primary behavior, and add focused coverage plus a changelog entry for the user-visible fix.

## Reproduction Assessment

Yes. A focused unit path can mock an agent primary that differs from `openai/gpt-5.5`, set `liveModelSwitchPending`, call the switch check the way the embedded runner currently does, and verify that the caller-selected heartbeat or cron model is not resolved back to the product default.

## Solution Assessment

No. The PR's resolver-only heuristic is not the best fix because the active caller still passes product defaults; the safer maintainable fix is to pass the real run selection into the switch check and test that contract.
