## Summary

Keep open. The reported bug is plausible, but this patch is not merge-safe because the active live-switch caller still supplies product defaults, so the new resolver branch can treat `openai/gpt-5.5` as an intentional caller override for custom-agent sessions.

## What This Changes

The PR changes `src/agents/live-model-switch.ts` to compare caller-provided defaults with agent defaults and prefer caller defaults when they differ during live-session model resolution.

## Best Possible Solution

Thread the actual initial run provider/model into the live-switch check, make the resolver honor those explicit caller defaults without regressing agent-specific primary defaults, and lock the behavior with focused unit coverage for normal message flow, heartbeat/cron explicit models, and pending user `/model` switches.

## Reproduction Assessment

Yes. A focused unit path can mock an agent primary that differs from `openai/gpt-5.5`, set `liveModelSwitchPending`, and verify that a caller-selected heartbeat/cron model is not resolved back to the product default.

## Solution Assessment

No. A resolver-only heuristic is not the best fix because the active caller still passes product defaults; the safer solution is to pass the real initial run selection into `shouldSwitchToLiveModel` and then simplify the resolver around that contract.
