## Summary

Keep open: the PR is a valid bug-fix direction, but the supplied latest diff is still incomplete because two fallback-capable entrypoints can omit the stable caller-selected live defaults and keep comparing pending live switches against the active fallback candidate.

## What This Changes

The PR adds liveModelDefaultProvider/liveModelDefaultModel plumbing to embedded agent runs, threads it through cron and one auto-reply execution path, updates focused live-switch tests, and adds a changelog entry.

## Best Possible Solution

Complete the stable live-default contract across every fallback-capable embedded runner entrypoint, with regression coverage for cron, heartbeat/auto-reply, queued follow-ups, direct agent commands, and the core live switch helper.

## Reproduction Assessment

Yes, at source level. A queued follow-up or direct `openclaw agent --model` run that falls back while `liveModelSwitchPending` is set can still pass only the active fallback provider/model into the embedded runner, because the latest supplied diff leaves those callsites unthreaded.

## Solution Assessment

No. The new live-default fields are the right narrow mechanism, but the patch is not the best complete fix until the original caller provider/model is passed through the remaining fallback callsites and covered by focused tests.
