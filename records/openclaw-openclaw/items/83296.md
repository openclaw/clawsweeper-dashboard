## Summary

Keep open: the updated PR fixes the fresh-generation path and the merge-mode path with model rows, but it still leaves a matching plaintext auth-profile key in generated models.json when the provider entry has no model rows because the strip lookup is gated on `hasProviderModels`.

## What This Changes

The PR changes model provider normalization, models.json planning, and regression tests so plaintext auth-profile keys are not persisted into generated provider config entries.

## Best Possible Solution

Finish the PR by scrubbing profile-backed plaintext values from any generated provider entry that already has an `apiKey`, including merge-preserved entries without model rows, while keeping unrelated user-managed keys intact.

## Reproduction Assessment

Yes, by source inspection: current main can copy plaintext profile keys into generated provider config, and the updated PR still skips the strip path for matching provider entries without model rows. I did not execute a repro because this review was read-only.

## Solution Assessment

No; the PR is the right direction, but the strip is still too narrowly gated. The safer fix is to apply matching plaintext-profile scrubbing after merge for provider entries with an existing `apiKey`, independent of whether `models` is non-empty.
