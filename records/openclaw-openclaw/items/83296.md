## Summary

Keep this PR open: it addresses the fresh provider-normalization path and includes usable terminal proof, but the patch leaves the merge-mode upgrade path able to preserve an existing plaintext `models.json` `apiKey` after the new stripping step runs.

## What This Changes

The PR changes model provider normalization and tests so plaintext auth-profile API keys are not added to or retained in generated provider config entries.

## Best Possible Solution

Revise the PR so plaintext profile-backed values are scrubbed from both fresh generation and existing `models.json` merge mode, while preserving unrelated user-managed provider keys.

## Reproduction Assessment

Yes, by source inspection: current main can copy plaintext profile keys through `resolveMissingProviderApiKey`, and the remaining PR blocker is source-reproducible because merge mode preserves existing non-marker `apiKey` values after normalization. I did not execute the repro because the review was read-only.

## Solution Assessment

No; the PR is the right general direction, but the strip runs too early to cover merge-mode regeneration from an existing `models.json`. The safer fix is to apply matching-profile plaintext scrubbing at or after the merge step and add a regression test for `planOpenClawModelsJsonWithDeps` with existing parsed contents.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Strip profile-backed keys after merge-mode preservation:** `src/agents/models-config.providers.normalize.ts:203-208`
  - body: The new strip runs before `planOpenClawModelsJsonWithDeps` merges existing `models.json`. In merge mode, `mergeWithExistingProviderSecrets` preserves any existing non-marker `apiKey` unless the provider is in `secretRefManagedProviders`; plaintext auth-profile matches are not added to that set, so an existing `models.json` containing the same profile key is written back unchanged. Please cover the plan/merge path and strip matching profile-backed plaintext after merge without dropping unrelated user-managed keys.
  - confidence: 0.94
