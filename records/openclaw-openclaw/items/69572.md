## Summary

Keep open. Current main and v2026.5.12 still implement Feishu typing feedback as reactions, and the requested native typing/status path needs Feishu API, permissions, and UX fallback validation before a safe implementation.

## What This Changes

Request Feishu typing feedback use a native typing/status API instead of the current `Typing` message reaction emoji.

## Best Possible Solution

Confirm the native Feishu/Lark per-chat typing or status API and permission model, then update the Feishu plugin, docs, tests, and live proof while retaining reaction feedback only as an explicit fallback if native typing is unavailable.

## Reproduction Assessment

Yes for the source-level behavior: current main routes Feishu typing feedback through `im.messageReaction.create` and reaction endpoints. The native client `正在输入...` claim still needs live-client/API proof.

## Solution Assessment

Unclear. Replacing reactions may be the right UX direction, but the checked SDK surface only proves message reactions and tenant/user system status with a `Typing` icon, not a confirmed per-chat bot typing API.
