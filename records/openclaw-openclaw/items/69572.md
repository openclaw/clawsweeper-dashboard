## Summary

Keep open: current main and the latest release still implement Feishu typing feedback as a persistent `Typing` message reaction, so the requested native Feishu/Lark typing status is not implemented and still needs maintainer API/product judgment.

## What This Changes

The issue asks OpenClaw to replace Feishu's current Typing emoji reaction indicator with a native Feishu/Lark typing status while an agent processes a message.

## Best Possible Solution

Decide the Feishu plugin UX/API contract and, if a supported native endpoint exists for bot messages, implement it behind the existing typing lifecycle with cleanup and live Feishu/Lark proof; otherwise document the reaction fallback as the supported behavior.

## Reproduction Assessment

Partially. Source inspection proves current main and v2026.6.8 still use Feishu message reactions for typing, but live Feishu/Lark proof is still needed to verify the exact native client-status behavior and API availability.

## Solution Assessment

Unclear. A native endpoint would be the cleanest solution if Feishu/Lark supports it for bot messages, but current code and the pinned SDK only prove the reaction path, so maintainers need to choose the permanent UX/API direction.
