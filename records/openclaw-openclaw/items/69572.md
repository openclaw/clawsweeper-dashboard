## Summary

Keep open. Current main and the latest release still implement Feishu typing feedback as Feishu/Lark reactions, and the remaining native typing/status behavior needs Feishu API and live-client confirmation before a safe repair can be queued.

## What This Changes

Replace Feishu typing feedback from message-reaction emoji calls with native Feishu/Lark typing or status behavior.

## Best Possible Solution

Confirm the Feishu/Lark native typing or status API and permission model, then replace or deliberately augment the Feishu plugin's reaction lifecycle with tests, docs, and live proof; keep reactions only as a documented fallback if native typing is not available.

## Reproduction Assessment

Yes for the source-level behavior: with Feishu typing enabled, reply start calls `addTypingIndicator()`, which sends `im.messageReaction.create`. I did not establish live-client proof for whether any native Feishu typing/status API shows the requested `正在输入...` text.

## Solution Assessment

Unclear. Replacing reactions is likely the right direction for the reported UX, but the maintainable fix depends on confirming whether Feishu exposes a bot-safe per-chat typing API or only broader user/system status APIs.
