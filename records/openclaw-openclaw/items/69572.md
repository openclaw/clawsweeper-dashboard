## Summary

Keep open. Current main and v2026.4.29 still implement Feishu typing feedback through Feishu/Lark reaction endpoints, and the related open PRs are reaction-target or customization changes rather than native typing-status replacements.

## What This Changes

The issue requests that Feishu message processing use a native Feishu/Lark typing-status API instead of adding a `Typing` emoji reaction.

## Best Possible Solution

Keep this as a bundled Feishu plugin behavior bug and replace the reaction-based message typing lifecycle with a confirmed native typing/status implementation, while documenting any deliberate fallback if Feishu/Lark does not expose a supported bot API.

## Reproduction Assessment

Yes. With Feishu typing enabled, the source-level path is deterministic: reply start calls `addTypingIndicator()`, which sends `im.messageReaction.create` with the `Typing` emoji instead of a native typing-status call.

## Solution Assessment

No current fix is present. The safest path is to confirm the official Feishu/Lark native typing API and then update the Feishu-owned typing helper, dispatcher tests, comment behavior if applicable, and docs around the chosen semantics.
