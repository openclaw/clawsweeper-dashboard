## Summary

Keep open: current main and the latest release still implement Feishu processing feedback as a persistent `Typing` message reaction, while the requested native typing-status path remains an external Feishu/Lark API and product-boundary decision.

## What This Changes

The issue requests replacing Feishu's `Typing` message-reaction feedback with a native Feishu typing-status indicator.

## Best Possible Solution

Keep the issue open until maintainers confirm the supported Feishu/Lark typing API contract; then implement native typing in the Feishu plugin where supported, with an intentional documented fallback for DM, group, reply, and thread surfaces that cannot use it.

## Reproduction Assessment

Partially yes: source inspection proves current main and v2026.6.10 still use Feishu message reactions for typing feedback. No live Feishu/Lark client repro was run to prove the native-status rendering gap or a supported replacement endpoint.

## Solution Assessment

Unclear until maintainers choose the product/API boundary. A native Feishu typing endpoint would be the cleanest fix if it is supported for bot messages across the affected surfaces; otherwise the safer path is a documented channel-owned fallback UX.
