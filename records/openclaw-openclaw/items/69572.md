## Summary

Keep open. Current main and the latest release still implement Feishu typing feedback as Feishu/Lark message and comment reactions, and the related open PRs continue to work within reaction-based behavior rather than replacing it with native chat typing status.

## What This Changes

The issue requests replacing Feishu's `Typing` emoji reaction-based feedback with native Feishu/Lark typing-status behavior.

## Best Possible Solution

Keep this as a bundled Feishu plugin behavior bug. A maintainer should confirm the official native typing/status API and permission requirements, then update the Feishu-owned typing lifecycle and docs/tests so native typing is distinct from any reaction fallback.

## Reproduction Assessment

Yes. With Feishu typing enabled, the source-level path is deterministic: `createFeishuReplyDispatcher()` calls `addTypingIndicator()`, which sends `im.messageReaction.create` with the `Typing` emoji instead of a native typing-status call.

## Solution Assessment

No current fix is present. The safest direction is a Feishu-owned native typing/status implementation after API and permissions are confirmed; guessing from adjacent SDK APIs would be less maintainable.
