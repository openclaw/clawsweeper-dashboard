## Summary

Keep open: current `main` and v2026.6.10 still implement Feishu processing feedback as a persistent `Typing` reaction, while the native-status API and fallback UX choice remain unconfirmed maintainer product/API decisions.

## What This Changes

The issue requests replacing Feishu's current reaction-based typing feedback with a native Feishu/Lark typing status.

## Best Possible Solution

Keep the issue open until maintainers confirm the Feishu/Lark typing API contract; then either implement the native status inside the Feishu plugin typing lifecycle with cleanup/live proof, or document the reaction/streaming-card fallback as intentional.

## Reproduction Assessment

Partially yes: source inspection reproduces the central current behavior because Feishu typing feedback still goes through `im.messageReaction.create`. No: this review did not live-reproduce the native Feishu/Lark client-status gap or prove the supported native endpoint.

## Solution Assessment

Unclear: using a supported native typing endpoint would be the cleanest solution if Feishu/Lark supports it for bot messages. If not, the maintainable answer is to keep the reaction or streaming-card fallback and document it as the official Feishu plugin UX.
