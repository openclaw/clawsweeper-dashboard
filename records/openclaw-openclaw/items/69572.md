## Summary

Keep open: current main and the latest release still implement Feishu typing as a persistent `Typing` message reaction, not a native Feishu/Lark typing status, and no merged or open canonical item owns this exact native-typing decision.

## What This Changes

The issue requests changing the Feishu channel working indicator from a `Typing` message reaction to a native Feishu/Lark typing status or an intentional documented fallback.

## Best Possible Solution

Decide the Feishu plugin UX contract first: use a supported native typing endpoint behind the existing `typingIndicator` behavior if Feishu/Lark supports it, otherwise document and implement the best visible fallback without reaction notification spam.

## Reproduction Assessment

Partially. Source and docs prove current main and the latest release use a `Typing` message reaction, but a high-confidence live Feishu reproduction of the missing native client status still needs real client/API proof.

## Solution Assessment

Unclear. A native typing API would be the cleanest fix if Feishu/Lark supports it for bot messages, but the current SDK surface does not expose one and maintainers need to choose the supported fallback behavior if no native endpoint is available.
