## Summary

Keep this PR open. Current main still does not expose a Microsoft Teams `defaultAzureCredential` auth mode, but the branch is not merge-ready because it conflicts with the shipped Teams auth config contract, needs identity-modeling changes, and changes security-sensitive credential selection behavior.

## What This Changes

The PR adds a Microsoft Teams `defaultAzureCredential` auth option with an `@azure/identity` token provider, schema/type updates, setup reset behavior, tests, and lockfile changes.

## Best Possible Solution

Rework the feature as an additive Teams auth option on top of current main: preserve the shipped `secret`/`federated` config contract, model any separate Azure identity client ID explicitly, constrain or document DAC fallback behavior, and wire it through the current Teams SDK App token path with docs and tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `defaultAzureCredential` is absent from the config type, strict schema, and credential resolver; the PR-specific blockers are reproducible from the provided branch diff and review discussion.

## Solution Assessment

No. The requested direction is plausible, but this branch is not the narrow maintainable solution because it changes shipped config values, pins DAC identity selection to the bot app ID, and targets an older Teams SDK integration path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve shipped Teams auth config values:** `src/config/types.msteams.ts:77`
  - body: Current main and docs expose `authType: "secret" | "federated"` with `certificatePath`, `useManagedIdentity`, and `managedIdentityClientId`. This branch switches to `clientSecret`/`certificate`/`federatedCredential` and different field names, so existing federated configs would fail strict schema validation or stop resolving after merge. Add DAC without removing the shipped names, or add an explicit migration/compat layer.
  - confidence: 0.88
- **[P1] Use a separate identity client ID for DAC:** `extensions/msteams/src/sdk.ts:35-38`
  - body: The DAC wrapper forces both `managedIdentityClientId` and `workloadIdentityClientId` to the Teams bot `appId`. User-assigned managed identity and workload identity deployments can use a different client ID, which current main already models separately as `managedIdentityClientId`; this can acquire tokens for the wrong principal or fail.
  - confidence: 0.87
- **[P2] Mirror env-backed app and tenant checks:** `extensions/msteams/src/token.ts:69-70`
  - body: `resolveMSTeamsCredentials` accepts `MSTEAMS_APP_ID` and `MSTEAMS_TENANT_ID`, but the branch's configured check only uses config fields for those IDs. Env-only DAC or federated deployments can look unconfigured to setup/doctor flows even though token resolution would work, so the configured check should mirror resolver fallback.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR changes Teams authentication and introduces DefaultAzureCredential fallback behavior that needs explicit maintainer security review before merge.

Concerns:

- **[medium] Constrain DAC credential fallback:** `extensions/msteams/src/sdk.ts:35`
  - body: The new DAC mode accepts only appId and tenantId and then lets DefaultAzureCredential use its default chain unless the environment constrains it. That chain can include developer credentials such as Azure CLI, PowerShell, or Azure Developer CLI after production credentials, which is useful locally but risky for a production Teams bot if it silently authenticates as an unintended ambient principal.
  - confidence: 0.78
