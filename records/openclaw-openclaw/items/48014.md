## Summary

Keep open: current main has overlapping passwordless Teams auth from a merged federated-auth implementation, but this PR's specific DefaultAzureCredential/local-CLI mode is not implemented on main and the branch still has blocking correctness and proof gaps.

## What This Changes

The PR adds a `defaultAzureCredential` MS Teams auth type, wires it through token acquisition, config schema/types, setup reset behavior, tests, and the extension dependency lockfile.

## Best Possible Solution

If maintainers still want DefaultAzureCredential, rebase it onto the current Teams SDK app/federated-auth surface, keep identity selection separate from the bot app ID or env-driven, document Azure Container Apps/AKS behavior, and require redacted live Teams/Azure proof.

## Reproduction Assessment

Yes for the code-level review findings: the PR branch and Azure identity source show the identity-ID mismatch, and the setup configured check mismatch is visible from source. No live Azure/Teams reproduction was established.

## Solution Assessment

No. The PR is not the best merge path as-is because it pins DAC identity options to the bot app ID, misses env-backed setup detection, lacks real behavior proof, and predates the current main Teams auth architecture.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Do not pin DAC identity IDs to the bot app ID:** `extensions/msteams/src/sdk.ts:36-37`
  - body: `managedIdentityClientId` and `workloadIdentityClientId` are selectors for the user-assigned managed/workload identity, but this passes the Teams bot app ID into both. In deployments where the bot app registration and managed identity differ, DAC will target the wrong principal and token acquisition can fail even though the environment is configured correctly. Leave these unset for env-driven DAC or add a separate identity client ID option.
  - confidence: 0.91
- **[P2] Mirror env fallback in the configured-credentials check:** `extensions/msteams/src/token.ts:68-71`
  - body: `resolveMSTeamsCredentials` accepts `MSTEAMS_APP_ID` and `MSTEAMS_TENANT_ID`, but `hasConfiguredMSTeamsCredentials` only reads `cfg.appId` and `cfg.tenantId`. The setup wizard uses this configured check before prompting, so an env-backed `defaultAzureCredential` config can be treated as missing and then overwritten with the client-secret prompt path.
  - confidence: 0.87
