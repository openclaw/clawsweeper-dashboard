## Summary

Keep open. The PR adds a useful DAC-style Teams auth idea, but it is not merge-ready: current main already ships certificate and managed-identity passwordless auth through a different `authType: "federated"` surface, while this branch adds a distinct stale `@microsoft/agents-hosting` path with identity-selection defects and no real Teams/Azure runtime proof.

## What This Changes

The branch adds a `defaultAzureCredential` Microsoft Teams auth mode around `@azure/identity`, routes token call sites through a provider factory, extends config/schema/tests, adds the dependency, and resets stale auth fields in setup.

## Best Possible Solution

If maintainers want DAC convenience, implement it on the current Teams SDK auth surface with explicit identity-client selection, docs, tests, and redacted real Teams/Azure proof; otherwise keep the shipped certificate and managed-identity path.

## Reproduction Assessment

No high-confidence live reproduction was established. The review findings are source-reproducible from the PR diff and Azure Identity source, but the feature needs real Teams/Azure runtime proof.

## Solution Assessment

No. The direction may be useful, but this branch is stale against current main and pins DAC identity selection too narrowly; a current-main implementation should use the shipped Teams SDK auth path with explicit identity selection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Allow a separate DAC identity client ID:** `extensions/msteams/src/sdk.ts:36-38`
  - body: This passes the Teams bot `appId` as both the managed-identity and workload-identity client ID. Azure Identity treats those options as the user-assigned/workload identity client ID, so deployments where the bot app registration and managed identity differ will request tokens for the wrong principal. Leave DAC to read `AZURE_CLIENT_ID` or add an explicit identity-client config field.
  - confidence: 0.86
- **[P2] Mirror env-backed IDs in the configured check:** `extensions/msteams/src/token.ts:69-71`
  - body: `resolveMSTeamsCredentials` accepts `MSTEAMS_APP_ID` and `MSTEAMS_TENANT_ID`, but `hasConfiguredMSTeamsCredentials` only checks `cfg.appId` and `cfg.tenantId`. Env-backed passwordless configs can therefore look unconfigured in setup/status flows and fall back into client-secret prompting.
  - confidence: 0.82
