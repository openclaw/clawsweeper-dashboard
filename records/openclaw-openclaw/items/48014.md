## Summary

Keep open: the enterprise Teams passwordless use case is still real and current main does not expose `defaultAzureCredential`, but this PR needs rework before merge because it diverges from the shipped Teams auth config, pins DAC identity selection to the bot app ID, and touches security-sensitive auth fallback behavior.

## What This Changes

The PR adds a Microsoft Teams `defaultAzureCredential` auth mode with an `@azure/identity` token provider, config/schema changes, setup reset behavior, tests, and lockfile updates.

## Best Possible Solution

Rework DAC as an additive Teams auth option on the current Teams SDK App path, preserving the shipped `secret`/`federated` config contract and explicitly modeling any separate Azure identity client ID with docs and tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `defaultAzureCredential` is absent from the public type, strict schema, and credential resolver, and the PR-specific blockers are visible from the provided branch diff and review discussion.

## Solution Assessment

No. The requested direction is useful, but this branch is not currently the best solution because it changes shipped config values, pins DAC identity selection to the bot app ID, and is stale against the current Teams SDK App token path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve shipped Teams auth config values:** `src/config/types.msteams.ts:77`
  - body: Current main exposes `authType: "secret" | "federated"` with `certificatePath`, `useManagedIdentity`, and `managedIdentityClientId`. This branch adds DAC by switching the public union to `clientSecret`/`certificate`/`federatedCredential`, so existing federated configs would fail strict validation or stop resolving after merge. Add DAC without removing the shipped names, or provide an explicit compatibility layer.
  - confidence: 0.88
- **[P1] Model the DAC identity client ID separately:** `extensions/msteams/src/sdk.ts:35-38`
  - body: The DAC provider forces both `managedIdentityClientId` and `workloadIdentityClientId` to the Teams bot `appId`. User-assigned managed identity and workload identity deployments can use a different client ID, which current main already models as `managedIdentityClientId`; forcing the app ID can acquire tokens for the wrong principal or fail.
  - confidence: 0.87
- **[P2] Mirror env-backed app and tenant checks:** `extensions/msteams/src/token.ts:69-70`
  - body: `resolveMSTeamsCredentials` accepts `MSTEAMS_APP_ID` and `MSTEAMS_TENANT_ID`, but the branch's configured check only uses config fields for those IDs. Env-only DAC or federated deployments can look unconfigured to setup and doctor flows even though token resolution would work.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a new Teams authentication path whose ambient credential fallback behavior needs explicit maintainer security review before merge.

Concerns:

- **[medium] Constrain DAC credential fallback:** `extensions/msteams/src/sdk.ts:35`
  - body: The new DAC mode lets `DefaultAzureCredential` use its default chain unless the environment constrains it. That can include developer credentials such as Azure CLI or Azure Developer CLI, which is useful locally but risky for a production Teams bot if it silently authenticates as an unintended ambient principal.
