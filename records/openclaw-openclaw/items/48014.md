## Summary

Keep this PR open. Current main still does not support a `defaultAzureCredential` Teams auth mode, but the proposed branch is not merge-ready because it changes the shipped Teams auth config contract, pins DAC to the bot app ID instead of a separate managed identity, and needs maintainer security review for ambient credential fallback.

## What This Changes

The PR adds a Microsoft Teams `defaultAzureCredential` auth mode with an `@azure/identity` token provider/factory, schema/type updates, setup reset behavior, tests, and lockfile dependency changes.

## Best Possible Solution

Rework the feature on top of current main as an additive Teams auth option: preserve the shipped `secret`/`federated` config contract, model any separate Azure identity client ID explicitly, constrain or document DAC fallback behavior, update docs/changelog/tests, and wire it through the current `@microsoft/teams.apps` App token path.

## Reproduction Assessment

Yes. On current main, a config with `authType: "defaultAzureCredential"` is outside both the TypeScript union and strict Zod enum, and the resolver has no DAC branch; the PR-specific blockers are reproducible by inspecting the provided branch diff and review discussion.

## Solution Assessment

No. The feature direction is plausible, but this branch is not the narrow maintainable solution because it changes shipped config values, pins identity selection incorrectly for supported managed-identity deployments, and targets an older Teams SDK integration path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Preserve shipped Teams auth config values:** `src/config/types.msteams.ts:77`
  - body: Current main and docs expose `authType: "secret" | "federated"` with `certificatePath`, `useManagedIdentity`, and `managedIdentityClientId`. This branch replaces that public contract with `clientSecret`/`certificate`/`federatedCredential` and different field names, so existing federated configs would fail strict schema validation or stop resolving after merge. Add DAC without removing the shipped names, or add an explicit migration/compat layer.
  - confidence: 0.88
- **[P1] Use a separate identity client ID for DAC:** `extensions/msteams/src/sdk.ts:35-38`
  - body: The DAC wrapper forces both `managedIdentityClientId` and `workloadIdentityClientId` to the Teams bot `appId`. User-assigned managed identity and workload identity deployments can use a different client ID, which current main already models separately as `managedIdentityClientId`; this can acquire tokens for the wrong principal or fail. Leave DAC unpinned by default or add a separate identity client ID field.
  - confidence: 0.87
- **[P2] Honor env-backed app and tenant IDs in configured checks:** `extensions/msteams/src/token.ts:69-70`
  - body: `resolveMSTeamsCredentials` accepts `MSTEAMS_APP_ID` and `MSTEAMS_TENANT_ID`, but the branch's configured check only uses config fields for these IDs. Env-only DAC or federated deployments can look unconfigured to setup/doctor flows even though token resolution would work, so the configured check should mirror resolver fallback.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR changes Teams authentication and introduces DefaultAzureCredential fallback behavior that needs explicit maintainer security review before merge.

Concerns:

- **[medium] Constrain DAC credential fallback:** `extensions/msteams/src/sdk.ts:35`
  - body: The new DAC mode accepts only appId and tenantId and then lets DefaultAzureCredential use its default chain unless the environment constrains it. That chain can include developer credentials such as Azure CLI/PowerShell/Azure Developer CLI after production credentials, which is useful locally but risky for a production Teams bot if it silently authenticates as an unintended ambient principal.
  - confidence: 0.78
