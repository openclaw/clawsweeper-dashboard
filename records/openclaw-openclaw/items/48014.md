## Summary

Keep open: the remaining `DefaultAzureCredential` convenience mode is not on current main, but this branch is not mergeable as-is because it is stale against the current Teams SDK auth implementation, has source-reproducible auth/config issues, and still lacks real Azure/Teams behavior proof.

## What This Changes

The PR adds a `defaultAzureCredential` Microsoft Teams auth mode, routes Teams token acquisition through a provider factory, updates config schema/types/tests, resets stale setup auth fields, and adds `@azure/identity` to the Teams plugin dependency set.

## Best Possible Solution

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Allow DAC identity to differ from the bot app ID:** `extensions/msteams/src/sdk.ts:35-38`
  - body: This passes the Teams bot `appId` as both `managedIdentityClientId` and `workloadIdentityClientId`. Azure Identity uses those options to choose the managed/workload identity, which can legitimately be different from the bot app registration; in that setup DAC requests the wrong principal. Leave these unset or add a separate identity-client setting.
  - confidence: 0.86
- **[P2] Mirror env-backed IDs in the configured check:** `extensions/msteams/src/token.ts:69-71`
  - body: `resolveMSTeamsCredentials` accepts `MSTEAMS_APP_ID` and `MSTEAMS_TENANT_ID`, but `hasConfiguredMSTeamsCredentials` only checks config fields before accepting `defaultAzureCredential`. Setup finalize uses that configured check, so env-backed DAC configs can still be treated as unconfigured and prompt for client-secret credentials.
  - confidence: 0.82
