## Summary

Keep this PR open, but it should not merge as-is: current main has since shipped explicit Microsoft Teams certificate/managed-identity auth through a different Teams SDK path, while this branch still proposes a distinct DefaultAzureCredential mode on the older agents-hosting path and has unresolved identity-selection and setup-detection issues.

## What This Changes

The PR adds a `defaultAzureCredential` Microsoft Teams auth type, wires token-provider creation through the Teams token call sites, updates config types/schema/tests, and adds `@azure/identity` to the Teams plugin dependency set.

## Best Possible Solution

Decide whether OpenClaw should add a DAC-style convenience mode on top of the current Teams SDK integration, and if so implement it against the current `authType: "federated"` model with explicit identity selection, docs, tests, and real Azure/Teams proof.

## Reproduction Assessment

No high-confidence live reproduction was established. The review findings are source-reproducible from the PR diff and Azure Identity source, but the new Teams/Azure auth behavior still needs real runtime proof.

## Solution Assessment

No. The direction may be useful, but this branch is stale against the current Teams SDK auth path and pins DAC identity selection too narrowly; a current-main implementation should reuse the shipped federated-auth surface or make the new DAC mode an explicit product decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Allow a separate DAC identity client ID:** `extensions/msteams/src/sdk.ts:37`
  - body: This constructs `DefaultAzureCredential` with both managed-identity and workload-identity client IDs forced to the Teams bot `appId`. Azure Identity treats those options as the user-assigned/workload identity client ID, so deployments where the bot app registration and managed identity differ will ask DAC for the wrong principal. Leave DAC to read `AZURE_CLIENT_ID` or add a separate identity-client config field instead of pinning both values to `appId`.
  - confidence: 0.86
- **[P2] Mirror env-backed IDs in the configured check:** `extensions/msteams/src/token.ts:71`
  - body: `resolveMSTeamsCredentials` accepts `MSTEAMS_APP_ID` and `MSTEAMS_TENANT_ID`, but `hasConfiguredMSTeamsCredentials` only checks `cfg.appId` and `cfg.tenantId` before accepting `defaultAzureCredential`. In setup/finalize flows that use this configured check, env-backed passwordless credentials can look unconfigured and push the wizard toward client-secret prompts. Read the same env fallbacks here that the resolver uses.
  - confidence: 0.82
