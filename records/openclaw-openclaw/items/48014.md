## Summary

Keep this PR open, but not as merge-ready cleanup. Current main still lacks the proposed `defaultAzureCredential` Teams auth mode and local Azure CLI-style flow, while the PR branch is stale against the current Teams SDK runtime and still has blocking auth-contract and identity-selection issues.

## What This Changes

The PR adds a Microsoft Teams `defaultAzureCredential` auth mode with an `@azure/identity` token provider, schema/type updates, setup reset behavior, tests, and lockfile/dependency changes.

## Best Possible Solution

Keep the feature under maintainer review and rework it on top of current main: add DAC as an additive Teams auth option without breaking the shipped `secret`/`federated` config contract, model any separate identity client ID explicitly, constrain or document DAC fallback behavior, and adapt token acquisition to the current `@microsoft/teams.apps` `App` path.

## Reproduction Assessment

Yes for the current-main gap: a config using `authType: "defaultAzureCredential"` is outside the current TypeScript union and strict Zod enum, so it will not validate as a supported Teams auth mode. The PR-specific blockers are reproducible by code inspection of the branch diff: DAC pins identity IDs to `appId`, and the configured check ignores env-backed app/tenant IDs.

## Solution Assessment

No, not as currently proposed. The direction is plausible, but the branch should preserve current public config values, use a separate or unset Azure identity client ID for DAC, and be rebased to the current Teams SDK integration before it can be considered the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Preserve shipped Teams auth config values:** `src/config/types.msteams.ts:77`
  - body: Current main and docs already expose `authType: "secret" | "federated"` with `certificatePath`/`managedIdentityClientId`. This branch replaces that public contract with `clientSecret`/`certificate`/`federatedCredential` and new field names, so existing federated configs would fail strict schema validation or stop resolving after merge. Add DAC without removing the shipped names, or provide an explicit migration/compat layer.
  - confidence: 0.86
- **[P1] Use a separate identity client ID for DAC:** `extensions/msteams/src/sdk.ts:35-38`
  - body: The DAC wrapper forces both `managedIdentityClientId` and `workloadIdentityClientId` to the Teams bot app ID. User-assigned managed identity and workload identity deployments can use a different client ID, which current main models separately as `managedIdentityClientId`; this will acquire tokens for the wrong principal or fail. Leave DAC unpinned by default or add an explicit identity client ID field.
  - confidence: 0.88
- **[P2] Honor env-backed app and tenant IDs in configured checks:** `extensions/msteams/src/token.ts:69-70`
  - body: `resolveMSTeamsCredentials` accepts `MSTEAMS_APP_ID` and `MSTEAMS_TENANT_ID`, but `hasConfiguredMSTeamsCredentials` only checks config fields. Env-only DAC or federated deployments will look unconfigured to setup/doctor flows even though token resolution would work.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff is an authentication change and introduces broad DefaultAzureCredential fallback behavior that needs maintainer security review before merge.

Concerns:

- **[medium] Constrain DAC credential fallback:** `extensions/msteams/src/sdk.ts:35`
  - body: The PR's DAC mode accepts only appId and tenantId, then lets DefaultAzureCredential use its default chain unless the environment constrains it. That chain includes developer credentials such as Azure CLI, PowerShell, Azure Developer CLI, and VS Code after production credentials, which is useful locally but risky for a Teams bot if a production deployment silently authenticates as an unintended ambient principal. Require or document a constrained production mode before merge.
  - confidence: 0.78
