## Summary

Keep this PR open, but it is not merge-ready: the feature is useful, the exact DefaultAzureCredential mode is not on current main, and the branch still has correctness, proof, and rebase blockers.

## What This Changes

This PR adds a `defaultAzureCredential` Microsoft Teams auth mode with @azure/identity token acquisition, schema/type updates, setup reset behavior, tests, and lockfile/dependency metadata.

## Best Possible Solution

Rebase the idea onto the current Teams SDK auth surface, preserve shipped `authType: "federated"` behavior, and add DAC only with an explicit identity-selection contract and live Teams/Azure proof.

## Reproduction Assessment

Yes for the source-level review findings, but no live reproduction was established. The diff clearly pins DAC identity IDs to `appId` and has an env/config mismatch in the configured check, while the PR supplies no live Azure/Teams proof.

## Solution Assessment

No. The current patch is not the best solution because it targets an obsolete Teams SDK surface and needs a maintainer decision on whether DAC should infer, pin, or explicitly configure the Azure identity client ID.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use a separate identity client ID for DAC:** `extensions/msteams/src/sdk.ts:35-37`
  - body: The DAC wrapper passes the Teams bot `appId` as both `managedIdentityClientId` and `workloadIdentityClientId`. Azure treats those options as user-assigned identity client IDs, and Teams configs already support a separate managed-identity client ID, so valid deployments where the bot app registration and managed identity differ will request tokens for the wrong principal. Leave these unset by default or add an explicit identity field.
  - confidence: 0.88
- **[P2] Mirror env fallback in configured checks:** `extensions/msteams/src/token.ts:69-70`
  - body: `resolveMSTeamsCredentials` accepts `MSTEAMS_APP_ID` and `MSTEAMS_TENANT_ID`, but `hasConfiguredMSTeamsCredentials` only checks `cfg.appId` and `cfg.tenantId`. An env-backed `defaultAzureCredential` setup can therefore be treated as unconfigured by setup/status paths even though the resolver can start it.
  - confidence: 0.83
