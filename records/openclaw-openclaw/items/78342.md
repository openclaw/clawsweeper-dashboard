## Summary

Keep open: the CLI admin-scope fix matches the current gateway authorization boundary, but this PR is not merge-ready because it lacks real behavior proof and reverses a recent macOS LaunchAgent PATH hardening change.

## What This Changes

The PR makes `openclaw agent` request `operator.admin` for `/new`, `/reset`, and `--model` gateway runs, updates the agent CLI note, and changes macOS service PATH generation to include user tool directories by default.

## Best Possible Solution

Keep the `openclaw agent` admin-scope fix scoped to CLI/gateway tests and docs, and handle any macOS service PATH expansion separately with daemon/security owner review.

## Reproduction Assessment

Yes, source-backed but not live-proven: current main connects CLI `agent` calls with `operator.write` by default, while the gateway requires `operator.admin` for `/new`, `/reset`, and provider/model overrides. The contributor still needs to add after-fix real setup proof.

## Solution Assessment

No. The admin-scope change is the narrow direction, but merging it together with a macOS LaunchAgent PATH policy reversal is not the best fix; split or drop that part and prove the gateway behavior in a real setup.
