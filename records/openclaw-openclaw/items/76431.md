## Summary

Keep this PR open, but do not merge it as-is. The feature is reasonable and scoped to the acpx plugin, but the implementation flattens `args` with a plain space join, so array elements containing whitespace or quoting characters will not survive the existing command-line parsing path; it also lacks the required user-facing docs/changelog coverage for a new plugin config field.

## What This Changes

The PR adds `agents.<id>.args` to the acpx plugin config schema/types and resolves each configured agent override by concatenating `command` with the args array.

## Best Possible Solution

Land the config shape only after encoding each `args` element with the same quote-safe command-string convention used by the existing acpx wrapper path, adding regression coverage for whitespace/quoted args, and documenting the new field with a changelog entry.

## Reproduction Assessment

Yes for the PR-introduced defect: the diff builds `fullCommand` with `cmdArgs.join(" ")`, and the current runtime later splits command strings on whitespace unless args are quoted. A config such as `args: ["/path/with space/adapter.mjs"]` would not preserve that as one argv element.

## Solution Assessment

No. The config addition belongs in the acpx plugin, but flattening args with a raw space join is not the best implementation; reuse or add a small quote-safe helper and cover it in config/runtime tests.
