## Summary

Keep open. Current main still accepts trimmed non-JSON single-id exec stdout verbatim, while this PR adds a narrow fail-closed guard and regression test; merge should wait for secops/maintainer approval plus changelog/docs for the new plaintext contract.

## What This Changes

The PR changes exec SecretRef parsing so single-id `jsonOnly: false` stdout must be valid JSON or exactly one non-empty plaintext line, and adds a POSIX resolver regression test.

## Best Possible Solution

Land a secops-approved fail-closed parser contract with a changelog entry and secrets-doc guidance, while keeping #64821 open until same-line corruption is separately fixed or ruled out.

## Reproduction Assessment

Yes. On current main, a `jsonOnly: false` single-id exec resolver that emits two non-empty non-JSON lines reaches the catch path and returns `stdout.trim()`; the PR's test exercises that parser boundary. This read-only pass did not execute tests.

## Solution Assessment

Unclear. Rejecting ambiguous multi-line plaintext is a narrow maintainable hardening for the parser, but it is not proven to solve same-line hot-reload corruption in #64821; secops should approve the contract and docs before merge.
