## Summary

Keep open. Current main still accepts the ambiguous `jsonOnly: false` single-value exec stdout path, and this PR is a narrow hardening candidate, but it changes CODEOWNERS-protected secrets behavior and still needs secops/maintainer review for the fail-closed contract, related #64821 scope, docs, changelog, and validation.

## What This Changes

The PR changes exec SecretRef parsing so a single-id `jsonOnly: false` provider must emit valid JSON or exactly one non-empty plaintext line, and adds a POSIX regression test for polluted multiline stdout.

## Best Possible Solution

Land a resolver-boundary hardening change after secops agrees with the fail-closed contract, keep the focused regression coverage, document that plaintext `jsonOnly: false` exec output must be one clean non-empty line or JSON, and keep #64821 open until the same-line corruption question is resolved or separately tracked.

## Reproduction Assessment

Yes. A code-level reproduction is clear: a single-id exec SecretRef with `jsonOnly: false` whose command prints two non-empty non-JSON lines reaches the current fallback and returns the full trimmed stdout. The PR’s added POSIX test exercises that exact parser boundary, though it was not executed in this read-only pass.

## Solution Assessment

Unclear. Rejecting ambiguous multiline plaintext is a narrow and maintainable fail-closed direction, but it is not yet proven to fully solve the linked #64821 same-line corruption report; secops should confirm scope before merge.
