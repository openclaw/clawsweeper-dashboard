## Summary

## What This Changes

This PR changes `src/secrets/resolve.ts` so single-id `jsonOnly: false` exec SecretRef stdout must be valid JSON or exactly one non-empty plaintext line, and adds a regression test in `src/secrets/resolve.test.ts`.

## Best Possible Solution

Land this or an equivalent resolver-boundary guard after secops approves the fail-closed behavior, record the one-clean-line-or-JSON plaintext exec contract, and keep https://github.com/openclaw/openclaw/issues/64821 open until same-line corruption is separately resolved or scoped.

## Reproduction Assessment

Yes, from source inspection. Current main returns the whole trimmed stdout for a single-id `jsonOnly: false` exec provider after JSON parsing fails, so a two-line stdout payload is accepted as one polluted value; I did not run tests in this read-only review.

## Solution Assessment

Unclear. Rejecting ambiguous multi-line plaintext at the parser boundary is a narrow security hardening path, but secops should approve the compatibility change and decide how it relates to same-line corruption in the linked security issue.
