## Summary

Keep open. Current main still has the parser behavior this PR targets, and the branch is narrowly scoped, but it changes security-owned secret-resolution semantics and needs secops/maintainer review plus user-facing docs or changelog before merge.

## What This Changes

This PR makes single-id `jsonOnly: false` exec SecretRef stdout resolve only from valid JSON or exactly one non-empty plaintext line, with a POSIX regression test for polluted multiline stdout.

## Best Possible Solution

Land a resolver-boundary fail-closed change after secops approves the plaintext stdout contract, document the `jsonOnly: false` requirement in changelog and public secrets docs, and keep #64821 open until the same-line corruption path is resolved or separately tracked.

## Reproduction Assessment

Yes. Current main has a direct code-level reproduction: a single-id exec SecretRef with `jsonOnly: false` and two non-empty non-JSON stdout lines reaches the fallback and returns `params.stdout.trim()`. The PR adds a POSIX unit test for that parser boundary, but this read-only review did not execute it.

## Solution Assessment

Unclear. Rejecting ambiguous multiline plaintext is safer and narrower than guessing which line is the secret, but it is not proven to fix #64821's same-line hot-reload corruption and should be paired with secops approval plus release-note/docs updates.
