## Summary

Keep open. Current main still accepts multi-line non-JSON stdout as a single exec secret value, and this PR is a focused secrets hardening patch rather than an obsolete or superseded branch; it still needs security-owner review and contributor real behavior proof before merge.

## What This Changes

This PR changes `parseExecValues()` so single-id `jsonOnly: false` exec stdout must be valid JSON or exactly one non-empty plain-text line, and adds a POSIX regression test for polluted stdout.

## Best Possible Solution

Land this or an equivalent resolver-boundary guard after secops approves the fail-closed plaintext contract, document the `jsonOnly: false` one-line-or-JSON requirement, and keep https://github.com/openclaw/openclaw/issues/64821 open until same-line corruption is resolved or separately tracked.

## Reproduction Assessment

Yes, from source inspection. On current main, a single-id exec SecretRef with `jsonOnly: false` and two non-empty non-JSON stdout lines reaches the fallback in `src/secrets/resolve.ts` and returns the full trimmed stdout.

## Solution Assessment

Unclear. The fail-closed parser guard is narrow and safer than last-line extraction for newline-separated noise, but security owners should approve the compatibility change and it does not address the same-line corruption shown in the related security issue.
