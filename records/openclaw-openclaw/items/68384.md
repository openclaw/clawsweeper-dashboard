## Summary

Keep open. Current main still accepts polluted non-JSON plaintext exec stdout for single-id `jsonOnly: false` SecretRefs, and this PR is a narrow hardening change rather than obsolete work; it still needs secops/maintainer review plus docs/changelog for the user-facing parser contract.

## What This Changes

The PR changes `src/secrets/resolve.ts` to reject multi-line non-JSON stdout for single-id exec SecretRefs and adds a POSIX regression test in `src/secrets/resolve.test.ts`.

## Best Possible Solution

Land a resolver-boundary fail-closed change only after secops approves the compatibility contract, document `jsonOnly: false` plaintext stdout as one clean non-empty line or JSON, and keep #64821 open until same-line corruption is resolved or separately tracked.

## Reproduction Assessment

Yes. A code-level reproduction is clear on current main: a single-id exec SecretRef with `jsonOnly: false` and stdout like `log\nfull\n` reaches the fallback and returns the full trimmed stdout; this pass did not execute tests because the review is read-only.

## Solution Assessment

Unclear. Rejecting ambiguous multiline plaintext is narrow and maintainable, but it is not proven to solve the same-line corruption reported in #64821 and should be paired with secops approval plus docs/changelog.
