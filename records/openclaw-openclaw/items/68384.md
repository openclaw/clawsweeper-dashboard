## Summary

Keep this PR open: the patch is focused and plausibly correct for newline-separated polluted exec stdout, but current main still has the old behavior and the remaining blockers are contributor real behavior proof plus security-owner approval for a secrets parser compatibility change.

## What This Changes

The PR changes `parseExecValues()` so single-id `jsonOnly: false` exec stdout must be valid JSON or exactly one non-empty plain-text line, and adds a resolver regression test for polluted stdout.

## Best Possible Solution

Land this or an equivalent resolver-boundary guard after secops approves the fail-closed plaintext contract, add the public docs/changelog wording for the new contract, and keep https://github.com/openclaw/openclaw/issues/64821 open until same-line corruption is fixed or separately scoped.

## Reproduction Assessment

Yes, from source inspection. On current main, a single-id exec SecretRef with `jsonOnly: false` and two non-empty non-JSON stdout lines reaches the fallback in `src/secrets/resolve.ts` and returns the full trimmed stdout.

## Solution Assessment

Unclear. Rejecting ambiguous multi-line plaintext is narrow and safer than last-line extraction, but security owners should approve the compatibility contract and the related same-line corruption remains outside this exact guard.
