## Summary

Keep open. Current main still accepts polluted plaintext exec stdout, and this PR is a narrow hardening step with focused coverage, but it changes CODEOWNERS-protected secrets behavior and still needs secops review plus release-note/docs treatment before merge.

## What This Changes

This PR changes exec SecretRef parsing so a single-id `jsonOnly: false` provider must emit valid JSON or exactly one non-empty plaintext line, and adds a POSIX unit test for polluted multiline stdout.

## Best Possible Solution

Land a resolver-boundary fail-closed change only after secops agrees with the plaintext stdout contract, add changelog and public secrets-doc guidance for `jsonOnly: false`, and keep the broader #64821 investigation open until same-line corruption is either fixed or explicitly tracked separately.

## Reproduction Assessment

Yes. A focused code-level reproduction is clear: on current main, a single-id exec SecretRef with `jsonOnly: false` whose command prints two non-empty non-JSON lines reaches the fallback and returns the full trimmed stdout. The PR adds a POSIX unit test for that parser boundary, but this read-only pass did not execute it.

## Solution Assessment

Unclear. Rejecting ambiguous multiline plaintext is a narrow, maintainable hardening direction, but it is not proven to solve the same-line corruption in #64821 and should be paired with secops approval plus release-note/docs updates.
