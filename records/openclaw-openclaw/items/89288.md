## Summary

Keep open: this is a focused fix for a source-reproducible session cleanup gap, but the current PR still needs maintainer judgment because it duplicates archive-retention policy instead of sharing the established cleanup owner, and its real behavior proof is not a full current-head CLI run.

## What This Changes

The PR extends session cleanup to prune stale deleted/reset archive files, threads `resetArchiveRetention` into cleanup sweeps, and adds regression tests for longer and disabled reset retention.

## Best Possible Solution

Share one archive-expiration/removal contract between normal store maintenance and explicit `sessions cleanup`, preserve separate deleted/reset retention semantics, and prove dry-run/apply in a full current-head checkout.

## Reproduction Assessment

Yes, at source level: current explicit cleanup delegates to an unreferenced-artifact sweep that excludes archive names, while reset archive retention is separately defined for normal store maintenance. I did not run a live CLI filesystem repro in this read-only review.

## Solution Assessment

No, not in its current shape. The PR fixes the immediate reset-retention semantics, but the best maintainable fix is to reuse a single archive cleanup contract instead of adding a parallel predicate.
