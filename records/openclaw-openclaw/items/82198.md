## Summary

Keep open. The report has a concrete release log and is security-sensitive around local media fetching, but current main does not show a confirmed Feishu path that should be auto-closed; it overlaps the existing open image-tool managed-media work that should be consolidated rather than duplicated.

## What This Changes

The issue reports that Feishu image messages are saved locally but image analysis receives a localhost media URL that the SSRF guard blocks.

## Best Possible Solution

Keep SSRF blocking intact and route Feishu inbound images through the shared managed-media resolver or a canonical `media://inbound`/saved-path contract before any HTTP fetch path is considered.

## Reproduction Assessment

No high-confidence current-main reproduction yet. The reported log is concrete and source shows localhost media fetches fail closed by design, but current Feishu source maps downloaded media to saved paths, so the handoff that produces `localhost:18888` needs confirmation.

## Solution Assessment

Unclear for an immediate patch. Allowlisting localhost would be the wrong direction; the maintainable path is to preserve trusted local media references through the shared media resolver and consolidate with the open media URI/image-tool fixes.
