## Summary

Current main already handles the reported Feishu path by passing saved inbound media as local media-store paths, while keeping the localhost/private-network remote fetch block intact; the fix/proof is on main after v2026.5.12 and appears unreleased.

## What This Changes

The issue reports Feishu image messages being saved locally but exposed to the agent as localhost media URLs that are later blocked by OpenClaw’s internal-host fetch policy.

## Best Possible Solution

Keep the current main behavior: trusted Feishu downloads should reach agents as local media-store paths, and localhost/private-network HTTP URLs should continue to be blocked by the SSRF guard.

## Reproduction Assessment

No against current main from source inspection: Feishu now surfaces saved inbound media as local paths, and the image/media-understanding path reads those local attachments before remote URL fetching. The reported localhost error remains reproducible only if a localhost HTTP URL is supplied, which current main should not need for Feishu inbound media.

## Solution Assessment

Yes: the maintainable fix is to preserve the SSRF block and route trusted channel-downloaded media through the local media-store path contract, which current main does.
