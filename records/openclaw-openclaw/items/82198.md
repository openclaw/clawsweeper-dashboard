## Summary

Keep open: the reported Feishu image failure is plausible and security-sensitive, but current main does not show Feishu handing the image tool a localhost URL, so the remaining work is to identify the exact localhost producer before changing SSRF behavior.

## What This Changes

The issue reports that Feishu image attachments are downloaded locally but image analysis later tries to fetch them through a localhost gateway URL that the SSRF guard blocks.

## Best Possible Solution

Find the component that turns Feishu managed media into a localhost URL, then route that handoff through an existing managed media reference or staged local path while keeping the generic HTTP SSRF guard strict.

## Reproduction Assessment

No. The issue has a concrete external log and steps, but current main source shows Feishu passing managed local paths and the image tool only blocks localhost when it is given an HTTP URL directly; a live Feishu/Windows path or the exact generated URL is still needed.

## Solution Assessment

No. The suggested broad localhost interception is not the best default; the safer direction is to preserve SSRF blocking and convert only OpenClaw-owned managed media references through a bounded media-store or staging path.
