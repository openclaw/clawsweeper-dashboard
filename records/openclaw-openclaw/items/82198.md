## Summary

Keep open: Feishu documents inbound image support, and the source path shows Feishu downloads media for agent turns while the image/media loader still sends `http(s)` inputs through the SSRF-guarded fetch path that blocks localhost. The report is actionable, but the fix crosses SSRF and local-file access boundaries, so it needs security-aware maintainer review rather than cleanup closure.

## What This Changes

Feishu inbound images downloaded to local media should remain analyzable by image/native vision tooling without being blocked as localhost SSRF.

## Best Possible Solution

Keep strict SSRF defaults and add a narrow OpenClaw-owned media-reference path so Feishu inbound images reach native image injection or the image tool as `media://` or safe local files, not generic localhost HTTP.

## Reproduction Assessment

No high-confidence live reproduction was run. The reported failure is source-reproducible for the localhost URL shape because current media loading sends `http(s)` image references through the SSRF-guarded fetch path and localhost is explicitly blocked.

## Solution Assessment

Unclear for the commenter’s draft implementation: broadly allowlisting or string-rewriting localhost would be unsafe. The safer fix is a narrow parser for OpenClaw-owned assistant-media or `media://inbound` references that reuses existing local-root/media-reference guards and leaves all other localhost/private URLs blocked.
