## Summary

## What This Changes

The PR pins plugin HTTP route registries during gateway bootstrap/runtime loading and changes gateway route lookup to prefer the process-global active registry before the server-local registry.

## Best Possible Solution

Replace the active-registry shortcut with gateway-owned deferred runtime attachment that updates server-local route/method registries and plugin service start/stop, then prove it with focused tests and real gateway output.

## Reproduction Assessment

Yes. Source inspection shows deferred runtime plugin prewarm loads plugins after gateway attachment, while current HTTP dispatch and service startup remain tied to the gateway-attached registry; I did not run a live curl/service reproduction in this read-only pass.

## Solution Assessment

No. This is a plausible route-only workaround, but the maintainable fix belongs in gateway-owned deferred attachment so routes, methods, and services share the same server-local lifecycle.
