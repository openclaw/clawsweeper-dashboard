## Summary

Keep open: the PR targets a real gateway/plugin bug, but the patch is not merge-ready because it makes HTTP route dispatch consult process-global active registry state, pins routes from the generic active runtime surface, and still lacks real runtime proof for the linked route/service lifecycle issue.

## What This Changes

The PR pins HTTP-route registries during gateway bootstrap and active standalone runtime loading, then makes gateway route lookup return the process-global active plugin registry when present.

## Best Possible Solution

Attach deferred runtime-loaded plugin registries through the gateway-owned server-local route, method, and service lifecycle, with focused regression tests and redacted runtime proof.

## Reproduction Assessment

Yes, at source level: deferred runtime plugin prewarm loads plugins after gateway attachment, while route dispatch is designed around server-local registry/context callbacks. I did not run a live packaged curl/service reproduction in this read-only review.

## Solution Assessment

No. This is a plausible route-only workaround, but the best fix belongs in gateway-owned deferred attachment so routes, methods, and services share the same server-local lifecycle.
