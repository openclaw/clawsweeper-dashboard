The branch adds HTTP-route registry pinning in gateway bootstrap and active runtime loading, then changes gateway route lookup to prefer process-global active registry state.

## Best Possible Solution

Yes at source level: the deferred runtime plugin path loads registries after gateway attachment, while current source and tests show HTTP dispatch is expected to use the server-local registry/context. I did not run a packaged curl/service reproduction in this read-only review.

## Solution Assessment

No. The active-registry lookup is a route-only workaround in the wrong boundary; the maintainable fix belongs in gateway-owned deferred attachment that keeps routes, methods, and services on the same server-local lifecycle.
