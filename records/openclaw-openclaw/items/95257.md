## Summary

Keep open: this PR is a focused attempt to mitigate https://github.com/openclaw/openclaw/issues/94572, but the final shape bypasses the server-local gateway route/context boundary, can replace pinned startup routes with the process-global active registry, leaves the linked service-lifecycle symptom unresolved, and still lacks real behavior proof.

## What This Changes

The branch adds HTTP-route registry pinning to gateway bootstrap and the active standalone runtime surface, then changes gateway route lookup to return the process-global active plugin registry when present.

## Best Possible Solution

Attach deferred runtime-loaded plugin registries through the gateway-owned server-local route, method, and service lifecycle, with focused regression tests and redacted curl/log proof.

## Reproduction Assessment

Yes, at source level: the deferred prewarm path loads runtime plugins after gateway attachment, while HTTP dispatch reads the server-local route registry/context. I did not run a live packaged curl/service reproduction in this read-only review.

## Solution Assessment

No. The active-registry lookup is a route-only workaround in the wrong boundary; the maintainable fix belongs in gateway-owned deferred attachment that keeps routes, methods, and services on the same server-local lifecycle.
