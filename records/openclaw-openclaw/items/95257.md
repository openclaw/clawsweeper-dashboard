## Summary

Keep open: the PR is not merge-ready because it routes plugin HTTP dispatch through process-global active registry state, can replace the gateway-owned pinned route registry from a generic active runtime load, and still lacks real gateway proof for the linked runtime-plugin bug.

## What This Changes

The PR pins plugin HTTP route registries during gateway bootstrap and active runtime loading, and changes gateway route lookup to prefer the process-global active plugin registry.

## Best Possible Solution

Replace the active-registry shortcut with gateway-owned deferred runtime attachment that updates server-local route/method registries and plugin service start/stop, then prove HTTP dispatch and service lifecycle behavior.

## Reproduction Assessment

Yes, at source level: deferred runtime plugin prewarm loads plugins after gateway attachment, while route dispatch and service startup are tied to the gateway-attached registry. I did not run a live packaged curl/service reproduction in this read-only review.

## Solution Assessment

No. This is a plausible route-only workaround, but the best fix belongs in gateway-owned deferred attachment so routes, methods, and services share the same server-local lifecycle.
