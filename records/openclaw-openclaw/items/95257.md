## Summary

Keep open: this PR is a candidate fix for the linked runtime-loaded workspace plugin bug, but the current patch is not merge-ready because route lookup switches to process-global active registry state, can drop this gateway's attached startup routes, leaves the service lifecycle half of the linked bug unresolved, and has only static proof.

## What This Changes

The PR pins HTTP-route registries during gateway/plugin runtime loading and changes gateway plugin route lookup to prefer the process-global active plugin registry over the server's attached registry.

## Best Possible Solution

Attach deferred runtime-loaded plugin registries through the gateway-owned server-local route, method, and service lifecycle, then prove HTTP dispatch and service start/stop with focused tests plus redacted curl/log output.

## Reproduction Assessment

Yes at source level: the deferred runtime plugin path loads registries after gateway attachment, while current source and tests show HTTP dispatch is expected to use the server-local registry/context. I did not run a packaged curl/service reproduction in this read-only review.

## Solution Assessment

No. The active-registry lookup is a route-only workaround in the wrong boundary; the maintainable fix belongs in gateway-owned deferred attachment that keeps routes, methods, and services on the same server-local lifecycle.
