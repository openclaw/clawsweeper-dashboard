## What This Changes

## Best Possible Solution

Replace the global active-registry lookup with gateway-owned deferred runtime attachment that updates routes, methods, and services for the accepting gateway, with focused tests and real runtime proof.

Yes, at source level: deferred runtime plugin prewarm loads plugins after gateway attachment, while route dispatch is designed around server-local registry/context callbacks. I did not run a live packaged curl/service reproduction in this read-only review.

## Solution Assessment

No. This is a plausible route-only workaround, but the best fix belongs in gateway-owned deferred attachment so routes, methods, and services share the same server-local lifecycle.
