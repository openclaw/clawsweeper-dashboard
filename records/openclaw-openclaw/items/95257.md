## Summary

Keep open: this PR targets a real gateway/plugin bug, but the current head is not merge-safe because it routes plugin HTTP dispatch through process-global active registry state and still lacks service lifecycle coverage and real gateway proof.

## What This Changes

The PR adds HTTP route registry pinning to gateway bootstrap and active runtime loading, then changes gateway plugin route lookup to prefer the process-global active plugin registry.

## Best Possible Solution

Replace the active-registry shortcut with gateway-owned deferred runtime attachment that updates route/method registries and plugin service start/stop for the accepting gateway, then prove the real HTTP and service behavior.

## Reproduction Assessment

Yes, at source level: deferred runtime plugin prewarm loads plugins after gateway attachment, while route dispatch and service startup are tied to the gateway-attached registry. I did not run a live packaged curl/service reproduction in this read-only review.

## Solution Assessment

No. This is a plausible route-only workaround, but the best fix belongs in gateway-owned deferred attachment so routes, methods, and services share the same server-local lifecycle.
