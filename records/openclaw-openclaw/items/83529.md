## Summary

Keep open: the patch targets a real Codex native-bound tool gap, but source review found an upgrade path that can rotate existing bound threads, channel context that can mis-target message-capable tools, and proof that still does not show a real bound conversation tool call.

## What This Changes

The branch wires OpenClaw plugin dynamic tools into native Codex bound conversations by passing dynamic tools to thread/start, refreshing bindings, dispatching item/tool/call through the dynamic tool bridge, and adding regression tests.

## Best Possible Solution

Reuse the existing dynamic-tool bridge in bound conversations, but refresh bindings in an upgrade-safe way that preserves old threads and derives tool channel context from the bound conversation/session target, with real bound-conversation proof before merge.

## Reproduction Assessment

Yes from source inspection: current main's native bound item/tool/call handler returns a fixed unsupported response and thread/start omits dynamicTools for this binding path. I did not run a live bound Codex conversation in this read-only review.

## Solution Assessment

No: the bridge reuse is the right direction, but this branch should preserve or explicitly migrate existing thread bindings and derive channel context through the established target-normalization path before merge.
