## Best Possible Solution

Reuse the existing Codex dynamic-tool bridge for bound conversations while preserving existing bound-thread context during upgrades and deriving tool channel context from the bound conversation/session target.

## Reproduction Assessment

Yes from source inspection: current main returns a fixed unsupported response for `item/tool/call` in native bound Codex conversations. I did not run a live bound Codex conversation because this review is read-only.

## Solution Assessment

No: reusing the existing dynamic-tool bridge is the right direction, but this patch is not yet the best implementation because it can rotate existing bindings and mis-scope channel context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Preserve existing bound threads when tools appear:** `extensions/codex/src/conversation-binding.ts:440`
  - body: For any existing sidecar without `dynamicToolsFingerprint`, this starts a fresh thread as soon as plugin tools are available. That means upgraded or manually attached bound conversations can lose their existing Codex thread history/session state on the next message, even though the current binding is still valid. Please make this upgrade-safe by preserving/resuming/forking existing context or requiring an explicit rebind, and cover the old-sidecar path.
  - confidence: 0.86
- **[P2] Use the bound conversation for tool channel context:** `extensions/codex/src/conversation-binding.ts:619`
  - body: `ctx.channelId` is the channel/plugin id, while channel-aware tools and hook context need the target conversation id resolved from the session/conversation target. Passing `telegram` or `slack` here makes the new tool bridge scope message/action discovery and hook metadata to the provider name instead of the bound conversation, so plugin tools can get wrong delivery defaults. Reuse the existing hook-channel helper or prefer `event.conversationId`/`ctx.conversationId`, and add a regression that asserts the tool factory context.
  - confidence: 0.82
