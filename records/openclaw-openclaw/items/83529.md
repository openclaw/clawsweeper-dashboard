## Summary

Keep open: the PR targets a real native Codex binding gap, but the new dynamic-tool request handler is not scoped to the active thread/turn and the supplied proof is still mocked/local rather than a real bound conversation invoking a plugin tool.

## What This Changes

The branch updates the bundled Codex plugin's native conversation binding to project OpenClaw plugin tools into Codex dynamic tools, dispatch `item/tool/call` back to plugin handlers, refresh stale sidecars, and add regression tests.

## Best Possible Solution

Scope native-bound dynamic tool handling to the active thread and turn using the existing app-server run-attempt pattern, then add live redacted proof for a real bound conversation invoking a plugin tool before merge.

## Reproduction Assessment

Yes, from source inspection: current main starts native bound Codex turns without usable OpenClaw dynamic tool dispatch and returns the fixed unsupported-tool response for `item/tool/call`. The PR's new scoping defect is also source-reproducible from the shared request-handler contract.

## Solution Assessment

No, not as written: projecting and dispatching plugin tools is the right direction, but the handler must first decline nonmatching thread/turn requests. The safer path is to mirror the scoped guard used by the existing Codex app-server run-attempt dynamic-tool handler.
