Full review comments:

- **[P1] Fix the CLI status warmup path:** `src/commands/status.summary.runtime.ts:12`
  - body: `getStatusSummary` still calls this resolver with `allowAsyncLoad: false` for default and session rows, while the new `skipRuntimeConfigLoad` warmup branch only fires when that flag is not false. That means the `openclaw status` path can still read a cold dynamic catalog cache and fall back to 200K unless some other lifecycle happened to warm it first. Add a status-summary warmup/prepared-metadata path and a regression that exercises `getStatusSummary` from a cold dynamic catalog.
  - confidence: 0.9
- **[P2] Warm with the caller configuration:** `src/agents/context.ts:263-265`
  - body: When a caller passes `cfg` and allows async loading, this new branch calls `ensureContextWindowCacheLoaded()` without passing that cfg, so the background cache load uses `getRuntimeConfig()` instead of the configuration being resolved. In explicit-config status/test paths that can populate the shared cache from the wrong provider catalog; either pass the caller config through the warmup path or keep warming lifecycle-owned only.
  - confidence: 0.82
