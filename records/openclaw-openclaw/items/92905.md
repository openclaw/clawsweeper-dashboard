## Summary

Keep open: the latest branch fixed the earlier CLI `allowAsyncLoad: false` blocker, but it still relies on fire-and-forget cache warming that cannot affect the first status render for dynamic/persisted catalog rows, and the external real-behavior proof gate is still missing.

## What This Changes

The PR warms context-window metadata from gateway startup and status/context lookups, delegates CLI status context resolution to the shared agent resolver, and adds context lookup regression tests.

## Best Possible Solution

Resolve the exact read-only model context metadata before status summary rendering, keep cache warmup scoped to the active or caller config, and require redacted real `openclaw status` proof before merge.

## Reproduction Assessment

Yes for the remaining PR defect from source inspection: dynamic/persisted catalog metadata is populated by an async warmup after `getStatusSummary` has already computed the displayed context value. I did not run a live command because this review is read-only.

## Solution Assessment

No. Delegating CLI status to the shared resolver is a good direction, but fire-and-forget warmup is not the best fix for a one-shot status read; the status owner should prepare or await the needed read-only metadata before rendering.
