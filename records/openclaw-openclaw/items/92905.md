## Summary

Keep open: the latest force-push removed the explicit no-warmup flags, but status still renders from the synchronous return value before the fire-and-forget cache warmup can affect dynamic catalog rows, and the external PR still has no real after-fix `openclaw status` proof.

## What This Changes

The PR delegates CLI status context lookup to the shared context resolver, enables background context-cache warming from status and gateway startup paths, and adds context-cache warmup regression tests.

## Best Possible Solution

Resolve the exact read-only model context metadata before status rendering, keep cache warming scoped to the active lifecycle or caller config, and require redacted real `openclaw status` proof before merge.

## Reproduction Assessment

Yes for the remaining PR defect from source inspection: the PR's own added tests show the first cfg/status-style call still returns 200K before async warmup, and status stores the synchronous value immediately. I did not run a live command because this review is read-only.

## Solution Assessment

No. Delegating CLI status to the shared resolver is a good direction, but fire-and-forget warmup is not the best fix for a one-shot status read; the status path should prepare or await the needed read-only metadata before rendering.
