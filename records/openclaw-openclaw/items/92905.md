## Label Justifications

- `P2`: This is a normal-priority status/context-window correctness fix with limited user-facing blast radius.
- `merge-risk: 🚨 compatibility`: The PR changes read-only status/context lookup semantics by allowing status paths to trigger model-catalog cache warming.
- `merge-risk: 🚨 availability`: The PR adds background context catalog loading to gateway startup and status paths, which can affect startup or command latency if the read-only load is slow or faulty.

## Summary

Keep open: the latest patch fixes the earlier caller-config propagation gap, but it still does not make the first one-shot status render use warmed catalog context data, leaves adjacent status-summary tests inconsistent with the new call contract, and lacks required real behavior proof.

## What This Changes

The PR changes status and gateway paths so context-window cache warming can start before status context-window values are read, and delegates CLI status context lookup to the shared resolver.

## Best Possible Solution

Have status resolve the read-only context metadata it needs before rendering, while preserving a clear lifecycle owner for any background catalog warmup.

## Reproduction Assessment

Yes. Source inspection and the PR's own tests show the first status-style resolver call still returns 200K before async warmup, while status stores that synchronous value immediately.

## Solution Assessment

No. Delegating CLI status to the shared resolver is useful, but fire-and-forget warmup is not the best fix for a one-shot status read; status should prepare the needed read-only metadata before rendering.
