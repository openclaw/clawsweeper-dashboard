## Summary

Keep this PR open. The CLI resolver delegation is useful, but the current patch still relies on fire-and-forget cache warming after status has already computed its returned context-token values, and it still lacks real after-fix status proof.

## What This Changes

The PR changes CLI and Gateway status context-window lookup so status paths can trigger asynchronous context-window cache warming and CLI status delegates to the shared context resolver.

## Best Possible Solution

Make status compute the needed read-only context metadata before returning the summary, then keep any background cache warmup lifecycle-owned, non-blocking, and covered by startup proof.

## Reproduction Assessment

Yes. Source inspection shows current main's CLI status runtime imports the raw cache resolver and status summary/status text pass `allowAsyncLoad:false`, so discovery-cache-only context windows are not warmed from those paths; I did not run a live CLI status command in this read-only review.

## Solution Assessment

No. Delegating the duplicate CLI resolver is the right direction, but background-only warmup is not the best fix for one-shot status output because the summary computes before that warmup completes.
