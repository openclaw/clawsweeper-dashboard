## Summary

Keep open: the latest head is focused but still not merge-ready because the status warmup is fire-and-forget, adjacent status summary tests still assert the old read-only resolver contract, and external real behavior proof is missing.

## What This Changes

The PR changes agent context lookup, CLI/Gateway status paths, and gateway startup so model context-window cache warming can run before status displays context limits.

## Best Possible Solution

Make status deterministic before rendering: either prepare the needed read-only context metadata before computing the summary, or keep status read-only and move warming to a lifecycle-owned startup/cache-refresh path with focused tests and real proof.

## Reproduction Assessment

Yes. Source inspection of the PR head shows status computes the displayed value synchronously while the new warmup runs asynchronously, and the added test still observes 200K before warmup completes.

## Solution Assessment

No. Delegating CLI status to the shared resolver is useful, but fire-and-forget warming is not the best fix for a one-shot status read unless maintainers deliberately accept the side effect with proof.
