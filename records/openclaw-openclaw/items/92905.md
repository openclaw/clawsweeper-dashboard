## Summary

Keep open: this is a focused bug-fix PR, but the patch still does not guarantee the first returned status output uses the warmed context metadata, and the contributor has not added real after-fix behavior proof.

## What This Changes

The PR delegates CLI status context resolution to the shared agent context resolver, removes read-only async-load guards from status rendering, starts background context-window warming at Gateway startup, and adds context-cache regression tests.

## Best Possible Solution

Make status compute the needed read-only context metadata before returning, then keep any background cache warmup lifecycle-owned, non-blocking, and covered by startup proof.

## Reproduction Assessment

Yes. Source inspection shows current main's CLI status runtime uses a cache-only resolver while status paths pass allowAsyncLoad:false, and the PR's own added test shows the first status-style result remains 200k before warmup completes.

## Solution Assessment

No. Delegating the duplicate CLI resolver is the right direction, but fire-and-forget warmup is not the best fix for one-shot status output because the summary computes before that warmup completes.
