## Summary

The PR has a useful direction, especially delegating CLI status context resolution to the shared resolver, but it is not merge-ready because the changed status calls still compute their returned value synchronously while the new cache warmup runs in the background. The external PR also still lacks after-fix real `openclaw status` or Gateway `/status` proof.

## What This Changes

The PR changes status and Gateway startup paths to trigger model context-window cache warming, delegates CLI status context resolution to `src/agents/context.ts`, and updates context/status regression tests.

## Best Possible Solution

Keep the shared resolver delegation if it is needed, but make status prepare the needed context metadata before returning and keep any startup warmup lifecycle-owned, non-blocking, and proven with real status/startup output.

## Reproduction Assessment

Yes, source-reproducible. The PR's own added test shows the first status-style resolver result remains `200_000` until async warmup finishes, and the changed status code still returns the synchronous resolver value immediately.

## Solution Assessment

No. Delegating the duplicate CLI runtime resolver is a good direction, but fire-and-forget warmup is not the best fix for a one-shot status command because it does not guarantee the returned status output is corrected.
