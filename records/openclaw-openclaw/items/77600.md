## Summary

Keep open: current main documents the native Codex runtime route and generic OAuth portability rules, but the OpenAI/Codex setup docs still do not explicitly warn against manually copying the same Codex OAuth refresh-token set across isolated Codex homes.

## What This Changes

The issue asks for documentation clarifying native Codex backend OAuth ownership for isolated agents and distinguishing `openai/gpt-5.5`, the `codex` runtime, `openai-codex` auth, and direct `openai-codex/gpt-*` routing.

## Best Possible Solution

Add a short OAuth ownership note to the OpenAI provider and Codex harness docs: isolated native Codex backends need one refresh owner, using per-agent login or OpenClaw-managed auth-profile inheritance/locking rather than manual token-file copies.

## Reproduction Assessment

Not applicable as a docs clarification. Source and tests show the `refresh_token_reused` class is real, but the remaining actionable item is missing setup guidance rather than a current-main runtime repro.

## Solution Assessment

Yes, a docs-only update is the narrowest maintainable solution. Current code already avoids default OAuth profile copying and has per-profile refresh coordination, so the missing piece is clearer user-facing setup text.
