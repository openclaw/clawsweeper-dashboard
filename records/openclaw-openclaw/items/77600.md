## Summary

Keep open: current docs already explain the native Codex runtime route, but they do not explicitly document the OAuth refresh-token ownership rule for manually copied isolated Codex homes.

## What This Changes

Clarify native Codex app-server OAuth setup for isolated agents, including route layering and refresh-token ownership.

## Best Possible Solution

Add a short OAuth ownership note to the OpenAI and Codex harness docs: each isolated native Codex backend needs one refresh owner, using either per-agent login or OpenClaw-managed profile inheritance/locking, not manual token-file copies.

## Repair Work Prompt

Update the docs for https://github.com/openclaw/openclaw/issues/77600. In `docs/providers/openai.md` and `docs/plugins/codex-harness.md`, add a concise OAuth ownership note for native Codex app-server isolated agents. Preserve the existing provider/model/runtime distinction: `openai/gpt-5.5` plus `agentRuntime.id: "codex"` is native Codex app-server execution, while `openai-codex/gpt-*` is Codex OAuth through PI. State that Codex OAuth refresh tokens rotate, so users must not manually copy one Codex OAuth token/account set into multiple isolated Codex homes and let each backend refresh independently. Safe setups are per-agent `openclaw models auth login --provider openai-codex` or OpenClaw-managed profile inheritance/mirroring through a single refresh owner/lock. Touch `docs/cli/agents.md` only if a cross-link from the existing OAuth seeding note is needed. Do not change runtime behavior, config schema, generated files, or add a changelog entry for this docs-only clarification.
