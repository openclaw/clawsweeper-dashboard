## Label Justifications

- `P2`: The issue is a normal-priority WebChat behavior bug with misleading model routing and limited blast radius.
- `impact:session-state`: The reported mismatch is caused by child session state inheriting or omitting model override state differently from what the UI displays.
- `impact:auth-provider`: The effective model/provider choice can silently differ from the displayed default, affecting provider routing and auth/cost expectations.

## Summary

This should stay open: current main still has the WebChat/session surfaces that can make the picker’s displayed default diverge from session model state, and no merged fix or narrower canonical item owns this exact parent-session override case.

## What This Changes

The issue asks WebChat `+ New Session` to align the displayed model picker default with the effective model used for the first child-session send when the parent session has a non-default model override.

## Best Possible Solution

Align WebChat’s displayed model selection and effective send-time model for child sessions, while preserving intentional dashboard lineage/hooks and adding regression coverage for parent override plus explicit default selection.

## Reproduction Assessment

Yes, at source level: the issue gives deterministic WebChat steps, and current source shows the child session is created with `parentSessionKey`, the picker can display the configured default, and `chat.send` does not send that displayed model. A live gateway run would still be useful before choosing the fix shape.

## Solution Assessment

Unclear until maintainers choose the intended semantics for dashboard child sessions. The safest end state is one shared resolver/contract where the picker, persisted session row, and send-time model agree, rather than a UI-only label fix.
