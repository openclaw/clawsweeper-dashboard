## Label Justifications

- `P2`: The original bug caused misleading WebChat model routing with moderate but bounded user impact.
- `impact:session-state`: The issue is about child session state inheriting model selection from a parent session.
- `impact:auth-provider`: The mismatch could route a turn through a different provider/model than the UI indicated.

## Summary

Current main and release `v2026.6.5` now make WebChat-created child sessions carry the parent runtime model selection directly, so the picker has an authoritative child session model to display before the first send. The broader reset/default model policy remains tracked separately, but the central silent WebChat new-session mismatch is fixed.

## What This Changes

The issue reports that WebChat’s new-session flow can display the configured default model while routing the first child-session turn through a parent session model override.

## Best Possible Solution

Keep the shipped behavior that makes WebChat child sessions visibly carry the inherited model, and handle broader cross-channel reset/default-model semantics in the related policy issue.

## Reproduction Assessment

Yes, source-level for the original bug: the old report mapped to WebChat creating a parent-linked child while runtime model resolution could inherit the parent. Current main and `v2026.6.5` now store the inherited runtime model selection on the child entry, so I did not establish a failing current-main path.

## Solution Assessment

Yes for the central WebChat new-session mismatch: the fix uses the existing session creation and session-row model surfaces, so the UI can show the effective inherited model before the first send. The separate question of clearing inherited or persisted model pins on default/reset should stay with the related open product-policy issue.
