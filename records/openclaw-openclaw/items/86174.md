## Summary

Keep open: current main partially addresses the initial `+ New Session` mismatch by making the inherited parent model explicit on the child session, but the source still shows no child-level “use configured default, do not inherit parent” state. After a user selects Default on a parent-linked child, the UI can again resolve to the configured default while runtime stored-model resolution can fall back to the parent override.

## What This Changes

The issue asks WebChat new sessions to keep the displayed model picker state and the effective runtime model routing in sync when parent sessions have model overrides.

## Best Possible Solution

Define one canonical child-session model contract where WebChat can select the configured default without inheriting the parent, while preserving intentional parent inheritance for the flows that still need it.

## Reproduction Assessment

Yes. The source path is high-confidence: after a parent-linked child clears its direct model override through the Default selection, runtime stored-model resolution can still fall back to the parent override while the picker resolves from the child/default row state.

## Solution Assessment

No. Current main is a useful partial fix for initial session creation, but it is not the complete fix because selecting Default still lacks a no-inherit child-session state.
