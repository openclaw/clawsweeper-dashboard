## Summary

Keep open: current main and v2026.6.9 still have the source-level path where WebChat creates a parent-linked child session and reply-time model resolution can use the parent's stored override while the picker can show Default. The linked PRs only cover narrower stale-reset diagnostics or are unmerged competing attempts, so the remaining work needs a maintainer decision about child-session inheritance versus explicit Default/no-inherit semantics.

## What This Changes

The issue asks OpenClaw to make WebChat new-session model picker state match the model that will actually be used when the session is parent-linked.

## Best Possible Solution

Define one supported child-session model contract and make the UI, sessions.create, sessions.patch Default handling, and reply-time resolver follow it consistently without regressing thread/channel inheritance.

## Reproduction Assessment

Yes, at source level. Current main still creates parent-linked WebChat sessions and reply-time model selection can resolve the parent's stored model override when the child has no direct override.

## Solution Assessment

Unclear until maintainers choose the child-session contract. A complete fix should either show inherited parent selection everywhere or add a narrow Default/no-inherit state without breaking intentional thread/channel inheritance.
