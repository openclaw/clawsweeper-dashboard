## Summary

Keep open: current main and v2026.6.8 still leave a source-level path where a WebChat child session can display or select Default while reply-time model resolution falls back to the parent session's override. The remaining fix is not a stale cleanup close; it needs a maintainer decision about child-session Default/no-inherit semantics versus intentional parent inheritance.

## What This Changes

The issue requests that WebChat new sessions either route through the displayed default model or visibly show and control inherited parent model overrides.

## Best Possible Solution

Define one child-session model contract: show inherited parent selection everywhere, or add a narrow child-level Default/no-inherit state that prevents parent fallback without breaking channel/thread inheritance.

## Reproduction Assessment

Yes, at source level. A parent-linked WebChat child can clear its direct override via Default, after which `resolveStoredModelOverride` can still fall back to the parent override in the auto-reply model-selection path.

## Solution Assessment

No, current main is not a complete solution. It makes initial child creation inheritance explicit, but the best fix still needs a maintainer-chosen Default/no-inherit contract that does not regress intentional parent fallback for threads/channels.
