## Label Justifications

- `P2`: The bug is deterministic and can silently route WebChat requests to the wrong model/provider, but it is limited to dashboard child-session model state.
- `impact:session-state`: The issue is caused by child session state inheriting a parent session model override that the child picker does not expose.
- `impact:auth-provider`: The mismatch can send the first child-session request through a different provider/model than the UI displays.

## Summary

Keep open: current main still has the WebChat mismatch where dashboard `+ New Session` creates a parent-linked child, the picker shows the configured default, and runtime model resolution can still inherit the parent session override. The related open items cover adjacent default-label and per-turn-override behavior, but they do not own this specific child-session routing mismatch.

## What This Changes

The issue asks WebChat `+ New Session` to make the displayed default model agree with the model used when a child dashboard session can inherit a parent override.

## Best Possible Solution

Define the dashboard child-session model contract, then make the picker and first run agree by either creating independent dashboard sessions for model selection or representing inherited/default state explicitly.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: WebChat creates a parent-linked child, the picker renders default from child/default rows, and runtime model selection inherits the parent override. I did not execute a live WebChat run in this read-only review.

## Solution Assessment

Unclear: the report identifies a real bug, but the best fix requires a maintainer choice between independent dashboard sessions, explicit inherited/default model state, or a visible inherited-model picker state.
