## Label Justifications

- `P2`: The bug is deterministic and can silently route WebChat requests to the wrong model, but the affected surface is limited to dashboard child-session creation and model override state.
- `impact:session-state`: The issue is caused by child session state inheriting a parent session model override that is not visible in the child picker.
- `impact:auth-provider`: The mismatch can route the first child-session request through a different provider/model than the UI displays.

## Summary

Keep open: current main still has the WebChat mismatch where `+ New Session` creates a parent-linked child that displays the configured default while runtime model selection can inherit the parent session override. The related open items cover adjacent model-label and per-turn override behavior, but they do not own this specific dashboard child-session routing bug.

## What This Changes

Fix Control UI WebChat `+ New Session` so the displayed model picker state matches the model used for the child session's first send, including inherited-parent override cases.

## Best Possible Solution

Preserve intentional thread parent inheritance, but make dashboard `+ New Session` either independent for model selection or persist/display an explicit inherited/default model state so the picker and first run agree.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: WebChat creates a parent-linked child, the picker renders default from the child/default rows, and runtime model selection inherits the parent override. I did not execute a live browser run in this read-only review.

## Solution Assessment

Unclear: the report identifies a real bug, but the best fix needs maintainer choice between independent dashboard sessions, an explicit child-level default sentinel, or displaying inherited model state. The fix should avoid weakening documented parent inheritance for thread sessions.
