## Summary

Keep open: current main and v2026.6.9 still have a source-level path where a WebChat child session can show or select Default while reply-time model resolution falls back to the parent session's override. The current remaining fix is a maintainer decision about the child-session model contract, not a stale cleanup close.

## What This Changes

This issue asks for WebChat new child sessions to keep the visible Default model selection and effective reply model routing in sync when a parent session has a model override.

## Best Possible Solution

Define one child-session model contract: either display inherited parent model state everywhere, or add a narrow child-level Default/no-inherit state that prevents parent model fallback without regressing intentional channel/thread inheritance.

## Reproduction Assessment

Yes, at source level. A parent-linked WebChat child can have its direct override cleared by the Default picker, after which reply-time model resolution can still inherit the parent override through parentSessionKey.

## Solution Assessment

No, current main is not a complete solution. It now makes create-time inheritance explicit, but the best fix still needs a maintainer-chosen Default/no-inherit or inherited-display contract that does not regress intentional parent fallback for threads/channels.
