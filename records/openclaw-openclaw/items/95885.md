## What This Changes

The PR records byte-triggered compaction retry metadata on SessionEntry and suppresses repeated byte-triggered preflight compactions for the same active transcript, threshold, and non-material growth while preserving token-triggered compaction behavior.

## Best Possible Solution

Land a narrow internal byte-trigger retry guard after maintainer review and real runtime proof, leaving broader public compaction rate-limit policy to the existing product discussion.

## Reproduction Assessment

Yes at source level: current main rechecks the same active transcript byte threshold on each preflight without remembering a prior byte-triggered compaction result. I did not establish a live runtime reproduction path in this read-only review.

## Solution Assessment

Yes, this appears to be the narrowest maintainable fix for the reported loop because it adds internal same-file hysteresis without changing token compaction or adding public rate-limit configuration. The remaining gap is real behavior proof from an actual session.
