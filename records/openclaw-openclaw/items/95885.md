## Summary

Keep open for maintainer review: the patch appears correct and the updated PR body now provides runtime-path terminal proof, but it intentionally adds persistent session retry metadata and a new byte-compaction hysteresis policy that maintainers should accept before merge.

## What This Changes

The PR adds internal SessionEntry retry metadata and a byte-growth hysteresis guard so byte-triggered preflight compaction is not repeated for the same active transcript until material growth, file change, or threshold change.

## Best Possible Solution

Merge after maintainers accept the internal session-state hysteresis policy and the staged runtime proof, while leaving broader `maxActiveTranscriptBytes` semantics to the existing product discussion.

## Reproduction Assessment

Yes at source level: current main checks the same active transcript bytes against `maxActiveTranscriptBytes` on each preflight without remembering a prior byte-triggered result. I did not run a live long-session reproduction in this read-only review.

## Solution Assessment

Yes: this is a narrow owner-boundary fix because it keeps the retry decision inside the preflight compaction/session-entry path and preserves token-triggered compaction. The main alternative, adding a public rate-limit option, would increase config surface for an internal loop guard.
