## Label Justifications

## Summary

Keep open for maintainer review: current main still has the fixed 60-second pre-runner setup watchdog, and this PR is the current canonical rebased fix with no blocking code findings, but it intentionally changes recovery timing and needs maintainer acceptance of that availability tradeoff.

## What This Changes

The PR derives the isolated cron setup watchdog from the job timeout budget, reports the derived timeout through setup-timeout telemetry, and adds a regression test for setup lasting past 60 seconds.

## Best Possible Solution

Land the watchdog-owned setup-window fix after maintainer acceptance of the longer bounded recovery delay, keeping setup-timeout telemetry aligned with the actual timeout used.

## Reproduction Assessment

Yes. Current main source arms a fixed 60-second setup timer before runner start even when the resolved job timeout is much larger; I did not run the regression test because this review is read-only.

## Solution Assessment

Yes, with a maintainer tradeoff check. The setup timeout is owned by the cron watchdog and this PR updates both the timer and setup-timeout telemetry in that owner path; the only remaining question is whether the 10-minute cap is the desired recovery policy.
