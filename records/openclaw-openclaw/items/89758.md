## Summary

Keep open: current main and the latest release still rotate/fall back on overloaded provider errors without a same-profile overload retry, and a second external commenter confirmed the cascade behavior. Prior merged work deliberately capped overload rotations, so this needs a maintainer policy decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to retry transient overloaded provider responses with configurable backoff before rotating profiles or falling back to another model.

## Best Possible Solution

Have maintainers decide whether overload retry should remain immediate-fallback oriented or gain a bounded same-profile retry lane, then implement that policy consistently with docs, config schema, and failover tests.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main and `v2026.6.6` show no same-profile overload retry, and current tests assert cross-provider fallback after an overloaded primary. I did not run a live provider repro in this read-only review.

## Solution Assessment

Unclear: a bounded retry-before-fallback lane is plausible for transient overloads, but the suggested unconditional 2-3 retries would change documented fallback timing and conflicts with prior overload policy. A maintainer-approved default or opt-in config is the safer direction.
