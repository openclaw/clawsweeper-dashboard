## Summary

Keep open: current OpenClaw pins Claude ACP 0.39.0, whose advertised `effort` option is model-aware, but OpenClaw still fails a turn when a persisted/default ACP `thinking` option exists and the active backend/model advertises no thinking/effort key.

## What This Changes

The issue asks OpenClaw ACP runtime controls to stop failing Claude Haiku sessions when a configured thinking default maps to an unsupported `effort` option.

## Best Possible Solution

Make ACP thinking controls model/capability-aware so inherited or default thinking does not break models that advertise no thinking option, while preserving an intentional strict path for explicit controls if maintainers want that contract.

## Reproduction Assessment

Yes, from source inspection rather than a live run: an ACP session with persisted `thinking` and advertised config keys that omit every thinking alias will build a config pair and then fail before the turn. The current Claude ACP pin can omit `effort` for models without supported effort levels.

## Solution Assessment

No, current main does not fully solve the issue. The dependency pin is model-aware, but OpenClaw still treats `thinking` as a required runtime config option instead of a model-aware/default-only hint.
