## Summary

Keep open: current `main` and the latest release still treat ACP `thinking`/`effort` as a required pre-turn config control, and the only direct fix PR was closed unmerged, so the reported Claude Haiku failure remains actionable.

## What This Changes

The issue reports that ACP sends the mapped `effort` config option to Claude Haiku, which rejects it and fails the session.

## Best Possible Solution

Make ACP pre-turn thinking controls model/capability-aware so inherited or default thinking does not abort models that reject effort, while preserving explicit config-command strictness unless maintainers intentionally change that contract.

## Reproduction Assessment

Yes, from source inspection rather than a live run: a persisted/default ACP `thinking` value can still be mapped to `effort`, and current `main` only tolerates rejected timeout config writes. The current Claude ACP pin can omit `effort` for models without supported effort levels.

## Solution Assessment

No, current `main` is not the best end state because it still keeps pre-turn thinking strict when the backend rejects the mapped key. The narrow code path is clear, but maintainers should define default/inherited thinking tolerance versus explicit control strictness before landing a fix.
