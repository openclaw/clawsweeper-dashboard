## Summary

Keep open: the race is still source-visible on current main, and this PR is a plausible fix direction, but it lacks contributor after-fix proof and now overlaps a newer retry-based PR for the same user-visible `agents add` race.

## What This Changes

The PR adds advisory config-write locking, exports a transform-style `updateConfigFile`, and moves non-interactive `agents add` config mutation inside the locked fresh-state transform.

## Best Possible Solution

Land one canonical `agents add` config-race fix on current main, with focused regression coverage and redacted real CLI proof that parallel adds preserve every agent.

## Reproduction Assessment

Yes, source-level: current main reads a config/base hash once in `agentsAddCommand`, computes the new agent config, then commits with that stale hash while `replaceConfigFile` rejects changed hashes. I did not run the live concurrent CLI flow because this review is read-only.

## Solution Assessment

Unclear: computing each mutation against fresh config state under a lock is a sound lost-update fix, but current main now has newer config mutation flows and https://github.com/openclaw/openclaw/pull/76601 offers a narrower retry-based path with tests. Maintainers should select and rebase one canonical fix.
