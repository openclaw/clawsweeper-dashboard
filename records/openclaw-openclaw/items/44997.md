## Summary

Keep open: this is a plausible focused bug fix for a current-main concurrent config mutation race, but the PR lacks after-fix real behavior proof and now overlaps with a newer open implementation path that maintainers need to choose between.

## What This Changes

The PR adds an advisory config write lock, exports a transform-style `updateConfigFile` API, and switches non-interactive `agents add` to compute its mutation against freshly read locked config state.

## Best Possible Solution

Land one canonical concurrent `agents add` fix on current main, with focused regression coverage and redacted real CLI proof that parallel adds preserve every agent.

## Reproduction Assessment

Yes, source-level: current main reads a config/base hash once in `agentsAddCommand`, computes the new agent config, then commits with that stale base hash, while `replaceConfigFile` rejects changed hashes. I did not run the concurrent CLI flow because this review is read-only.

## Solution Assessment

Mostly yes: applying each agent mutation to fresh config state under a lock is a maintainable shape for the lost-update race. The final path should be rebased onto current config mutation flows or consciously superseded by https://github.com/openclaw/openclaw/pull/76601.
