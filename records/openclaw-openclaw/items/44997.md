## Summary

Keep open. Current main still builds the non-interactive `agents add` config from a pre-read snapshot and only guards the later write with a base-hash conflict check, so it does not provide the PR's intended queued read-modify-write behavior where all concurrent agent additions persist. The PR addresses the right race after the review discussion, but it is now dirty against current main and needs a rebase or replacement that preserves the current config writer and plugin-install contracts.

## What This Changes

The PR adds an advisory lock around config writes, exposes an `updateConfigFile(transform)` API, and switches non-interactive `agents add` to update `agents.list` from a fresh locked config snapshot.

## Best Possible Solution

Rebase or replace the PR with a current-main implementation that serializes the config read-modify-write for `agents add` while preserving the existing config writer behavior: plugin install record commits, env-ref restoration, include-file handling, runtime snapshot notifications, directory permissions, audit/destructive-write guards, and persisted hash results. Add a focused concurrent `agents add` regression test that proves all independently named agents persist.

## Reproduction Assessment

Yes. The PR body and #43367 give a high-confidence reproduction path: run four or more `openclaw agents add ... --non-interactive --workspace ... --json` commands against the same config and then inspect `openclaw agents list`. Current main confirms the risky shape because each process computes `nextConfig` from its own initial snapshot before committing.

## Solution Assessment

No, not as this PR currently stands. The transform-under-lock direction is the right narrow solution, but this branch is stale and should be rebuilt on top of the current config mutation/writer contracts rather than merging the old extracted write path.
