## Summary

Keep open: the transform-under-lock direction addresses a real config race tied to #43367, but this branch is stale/dirty and would drop newer current-main config writer and plugin-install contracts if merged as-is.

## What This Changes

The PR adds an advisory lock-backed `updateConfigFile` API and changes non-interactive `agents add` to run agent and binding mutations inside a locked config transform.

## Best Possible Solution

Land a current-main implementation that adds a locked transform-based config commit path for read-modify-write cases, updates non-interactive `agents add`, preserves runtime writer and plugin-install contracts, and leaves the broader multi-agent failures in #43367 for separate fixes.

## Reproduction Assessment

Yes, by source inspection. Current main reads one `agents add` config snapshot, computes the agent list outside any cross-process lock, then commits with an optimistic `baseHash`, so concurrent adds from the same base are not queued and replayed against the latest file.

## Solution Assessment

No, not as this branch currently stands. The transform-under-lock design is the narrow maintainable fix, but it needs to be rebuilt on current main without replacing newer writer behavior or bypassing pending plugin-install commits.
