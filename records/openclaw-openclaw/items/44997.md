## Summary

Keep open: the concurrent config-write bug is source-reproducible and the transform-under-lock direction is useful, but this stale branch would regress current main config-writer/plugin-install behavior and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a config advisory lock, exports `updateConfigFile(transform)`, and changes non-interactive `agents add` to compute agent and binding mutations inside that locked transform.

## Best Possible Solution

Land a current-main implementation that adds a locked transform-based config commit path for read-modify-write callers, integrates with `commitConfigWithPendingPluginInstalls`, preserves the current writer/runtime notification contracts, and keeps the broader work in https://github.com/openclaw/openclaw/issues/43367 separate.

## Reproduction Assessment

Yes, by source inspection. Current main reads one config snapshot for non-interactive `agents add`, computes the agent list outside any cross-process lock, and commits with the original `baseHash`, so concurrent adds from the same base are not serialized and replayed.

## Solution Assessment

No, not this branch as it stands. The transform-under-lock approach is the narrow maintainable direction, but the patch needs to be rebuilt on current main without dropping newer config writer and plugin-install commit contracts.
