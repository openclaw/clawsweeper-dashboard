## Summary

Keep open: current main still lacks a transform-under-lock path for non-interactive `agents add`, but this stale PR is not mergeable as-is because it would replace newer config writer behavior and bypass current plugin-install commit handling.

## What This Changes

The PR adds an advisory config file lock, exports `updateConfigFile(transform)`, and changes non-interactive `agents add` to compute agent and binding mutations inside a locked transform.

## Best Possible Solution

Land a current-main implementation that adds a locked transform-based config commit path for read-modify-write cases, updates non-interactive `agents add`, preserves runtime writer and plugin-install contracts, and leaves the broader multi-agent failures in #43367 for separate fixes.

## Reproduction Assessment

Yes, by source inspection. Current main reads one config snapshot for `agents add`, computes the new agent list outside any cross-process lock, and commits with an optimistic `baseHash`, so concurrent adds from the same base can conflict instead of all being serialized and persisted.

## Solution Assessment

No, not as this branch currently stands. The transform-under-lock design is the narrow maintainable fix, but the patch needs to be rebuilt on current main without replacing newer config writer behavior or bypassing pending plugin-install commits.
