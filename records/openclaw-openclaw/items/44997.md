## Summary

Keep open: this targets a real source-visible `agents add` config race, but the external PR still lacks after-fix real behavior proof and is stale relative to the current config mutation design. The newer related PR is also open and not merged, so this PR should not be closed as implemented or superseded yet.

## What This Changes

The branch adds an advisory config-file lock, exports an `updateConfigFile` transform API, and switches non-interactive `agents add` to mutate a freshly-read config under that lock.

## Best Possible Solution

Choose one current-base fix for concurrent `agents add` writes, preserve the current config mutation contracts, add focused regression coverage, and require redacted parallel CLI proof showing all added agents persist.

## Reproduction Assessment

Yes, at source level. Current main computes `agents add` from one config snapshot and commits with that original hash, while the writer throws `ConfigMutationConflictError` if another process changed the config before commit.

## Solution Assessment

No, not as-is. The lock/transform direction is plausible, but this branch should be reworked against current `mutateConfigFile`/`replaceConfigFile` contracts or superseded by a corrected current-base retry PR with real behavior proof.
