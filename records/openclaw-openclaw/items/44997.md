## Summary

## What This Changes

The PR adds a locked `updateConfigFile` read-modify-write helper and moves non-interactive `agents add` mutations into that transform so concurrent agent additions see fresh config state.

## Best Possible Solution

Rebase the fix onto current `main`, preserve the existing config commit/plugin-install contract, handle fresh-state binding errors, add a focused concurrent-add regression test, and attach redacted real CLI proof; alternatively continue with the current-main retry PR after its blockers are fixed.

Yes, at source level. Current `main` reads a config hash before building the `agents add` mutation and `replaceConfigFile` throws on a later hash mismatch, which matches the reported concurrent config race; I did not run a live race in this read-only review.

## Solution Assessment

No, not as currently submitted. The locked transform direction addresses the stale-array problem, but the branch must be rebased onto the current config mutation surface and fix fresh binding parse handling before it is the maintainable merge path.
