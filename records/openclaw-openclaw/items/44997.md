## Summary

Keep open: the PR targets a real `agents add` config race, but the branch is now stale/conflicting against current main and lacks after-fix real behavior proof. A newer open PR, https://github.com/openclaw/openclaw/pull/76601, also tracks the current-main retry path, so maintainers should choose one canonical fix rather than close this contribution prematurely.

## What This Changes

This PR adds advisory config write locking, exports an `updateConfigFile` transform API, and changes non-interactive `agents add` to compute its config mutation against a fresh on-disk config inside the lock.

## Best Possible Solution

Land one canonical current-main fix that re-applies `agents add` mutations to a fresh valid config while preserving `commitConfigWithPendingPluginInstalls`, then cover it with a concurrent-add regression test and real CLI proof.

## Reproduction Assessment

Yes, at source level. Current main reads a base config hash before the `agents add` mutation and the config writer throws `ConfigMutationConflictError` if another writer changes that hash before commit; I did not run a live race in this read-only review.

## Solution Assessment

No, not as currently submitted. The fresh-state transform idea is sound, but this branch is based on an obsolete write path and should be ported or consolidated with the current `replaceConfigFile`/`mutateConfigFile` commit flow.
