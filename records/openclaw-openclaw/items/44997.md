## Summary

Keep this PR open, but do not merge it unchanged: it targets a real current config race, while the branch is stale/dirty against current main and would regress newer config writer and plugin-install commit contracts if rebased mechanically.

## What This Changes

The PR adds an advisory config write lock, exports `updateConfigFile(transform)`, and switches non-interactive `agents add` to compute agent and binding changes from a freshly locked config snapshot.

## Best Possible Solution

Land a current-main implementation that applies read-modify-write transforms inside a config file lock for `agents add`, preserves all current config writer and plugin-install contracts, adds regression coverage, and leaves the separate session/OAuth/detached-work parts of #43367 for their own fixes.

## Reproduction Assessment

Yes. Source inspection shows current main still computes non-interactive `agents add` from one initial config snapshot and then commits that precomputed config with a stale-hash guard, so concurrent appends are not replayed against the latest file.

## Solution Assessment

No, not as this branch currently stands. The transform-under-lock design is the right narrow solution, but the PR must be rebuilt on current main without replacing newer config writer behavior or bypassing pending plugin-install commits.
