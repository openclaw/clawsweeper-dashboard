## Summary

Keep open. Current main contains a partial hot-path improvement for runtime plugin loads, but source inspection still shows remaining per-turn plugin metadata/runtime resolution paths in embedded tool construction and tool-result middleware, and the report now has independent confirmations across ARM, macOS, WSL2, and Windows.

## What This Changes

The issue reports a regression where embedded chat turns repeatedly rebuild plugin runtime metadata and reload bundled plugins, adding 30-60s or more of prep latency per turn.

## Best Possible Solution

Keep this issue open and land a narrow fix that threads the gateway-owned metadata snapshot, lookup table, manifest registry, or compatible active registry into embedded-runner tool and middleware construction. The fix should preserve plugin install/config freshness and avoid hidden TTL or process-wide metadata caches.

## Reproduction Assessment

Yes at the report level: the issue gives concrete gateway steps, trace lines, repeated-load logs, strace/profiler evidence, and multiple independent confirmations. This review did not live-run the ARM/Telegram setup, but the source paths and cross-platform comments make the reproduction path credible.

## Solution Assessment

Yes, the explicit snapshot/registry-threading direction is the best maintainable fix. Current main's runtime-deps and gateway-bindable active-registry shortcuts are useful but incomplete because not every hot-path caller receives or passes the metadata objects that the architecture docs require.
