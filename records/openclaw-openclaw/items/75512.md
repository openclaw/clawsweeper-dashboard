## Summary

Keep open. The report is recent, detailed, and source inspection of current main does not show a complete fix for the reported per-turn plugin metadata/runtime work in embedded chat prep; the existing code still rebuilds or compatibility-checks plugin registry inputs from hot-path tool and middleware entry points instead of consistently receiving a caller-owned snapshot or manifest registry.

## What This Changes

The issue requests eliminating per-turn plugin runtime/metadata re-evaluation that makes embedded chat prep take 30-60+ seconds by reusing caller-owned plugin metadata or manifest registry state across turns.

## Best Possible Solution

Thread the existing gateway-owned metadata snapshot, lookup table, or manifest registry into embedded-runner tool construction and middleware resolution, and reuse the active gateway-bindable plugin registry where valid. Keep the plugin cache boundary explicit: no hidden TTL or process-wide metadata cache, and preserve plugin install/config freshness.

## Reproduction Assessment

No high-confidence live reproduction in this checkout because the ARM SBC, Telegram setup, and release binaries were not exercised. The report itself provides a detailed reproduction path, and current-main source inspection validates the named hot paths still lack complete caller-owned metadata threading.

## Solution Assessment

Yes, the reporter's proposed direction matches the documented plugin cache boundary: reuse explicit caller-owned snapshot/manifest objects in hot flows. A broad persistent metadata cache would be the less maintainable fix because plugin installs and manifest edits must remain visible on the next explicit read.
