## Summary

Keep open. The report is detailed, current main still routes embedded-turn tool setup through request-time plugin runtime resolution and metadata/cache-key reconstruction, and I did not find a merged fix or canonical open tracker that fully supersedes this specific per-turn plugin-runtime latency problem.

## What This Changes

Reports a regression where embedded chat turns spend 30-60+ seconds re-evaluating bundled plugin runtime/tool metadata after v2026.4.23, especially on ARM hardware.

## Best Possible Solution

Thread the Gateway-owned plugin lookup/manifest/runtime registry objects through embedded run startup, tool construction, system prompt/tool-result middleware, and provider/tool resolution so subsequent turns avoid discovery/index/package-root work while still rebuilding snapshots on config, plugin inventory, install-record, or workspace-scope changes.

## Reproduction Assessment

Yes. The issue includes a concrete ARM setup, repeated-turn steps, trace-stage timings, plugin load logs, strace, and profiler output; I did not live-run that hardware path, but the current source still contains the hot paths named in the report.

## Solution Assessment

Yes, the reporter's proposed direction is aligned with the docs: pass an explicit caller-owned `PluginMetadataSnapshot`, `PluginLookUpTable`, manifest registry, or compatible runtime registry through the current flow rather than adding a hidden wall-clock cache. The narrowest maintainable fix should include a regression test that subsequent embedded turns do not rebuild plugin discovery/index state when the gateway snapshot is unchanged.
