## Summary

Current main now addresses the reported per-turn plugin-runtime reload path: embedded runs pre-load runtime plugins through the shared resolver, plugin tool construction reuses the active gateway-bindable registry, and regression coverage asserts that the tool hot path does not call the runtime resolver or loader again.

## What This Changes

The issue reports 30-60s+ chat-turn latency after v2026.4.23 from repeated per-turn plugin runtime and metadata work during embedded-runner preparation.

## Best Possible Solution

Keep the explicit gateway-bindable registry reuse and hot-path runtime-deps skip now on main, then let the next release carry the fix to packaged users without adding hidden wall-clock plugin metadata caches.

## Reproduction Assessment

Yes. The issue gives a concrete affected-release reproduction path with trace, strace, and profiler evidence; current main was not live-benchmarked here, but source and regression-test inspection verify the named plugin reload path is no longer used for gateway-bindable tool construction.

## Solution Assessment

Yes. Reusing the active gateway-owned runtime registry and keeping dependency repair out of per-turn fallback loads is narrower and safer than adding a broad process-wide metadata cache, while preserving explicit reload boundaries for plugin/config changes.
