## Summary

Current main and the v2026.5.20 tag already implement the requested cron TTS voice-note propagation: cron announce delivery applies TTS with the delivery channel/account, speech-core derives audioAsVoice from the channel capability, and Feishu consumes that flag for native voice/transcode delivery.

## What This Changes

The issue reports that isolated cron announce delivery fails to propagate voice-note intent for TTS audio, so Feishu receives generated speech as a file attachment instead of a native voice note.

## Best Possible Solution

Keep the existing channel-capability-driven TTS path; reopen only with artifact-specific evidence that a packaged v2026.5.20 build diverges from the tagged source or that Feishu transcode fallback is firing unexpectedly.

## Reproduction Assessment

No high-confidence failing current-main reproduction path is available from source inspection; current main and the v2026.5.20 tag already propagate audioAsVoice through the reported cron/TTS/Feishu path. I did not run a live Feishu cron job in this read-only review.

## Solution Assessment

Yes. The current implementation keeps core generic by passing delivery channel/account context into TTS, lets speech-core derive voice intent from plugin capabilities, and leaves Feishu-owned native/transcode behavior inside the Feishu plugin.
