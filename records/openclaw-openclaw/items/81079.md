## Summary

Keep open: the PR is a plausible additive ContextEngine API change, but the new token estimate is computed before known final prompt additions/current-turn context, the new docs misstate current precheck ordering, and external real behavior proof is still mock/test-only.

## What This Changes

Adds optional `currentTokenCount` to `ContextEngine.assemble`, threads token estimates through PI and Codex harness assembly paths, re-exports `estimatePrePromptTokens`, and updates docs/tests.

## Best Possible Solution

Define one supported `currentTokenCount` contract based on the prompt/system state the model will actually see, keep the public docs accurate about runtime precheck ordering, and merge only after redacted real long-session proof is added.

## Reproduction Assessment

Yes for the review findings via source inspection: the PR head computes `currentTokenCount` from `params.prompt` before later current-turn context and prompt context are added. The downstream live overflow scenario itself has not been reproduced; the PR body says live provider proof is pending.

## Solution Assessment

No, not yet. The API direction is plausible, but the implementation should count known final prompt additions/current-turn context or clearly narrow the contract, and the docs should match the current precheck order.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Count the resolved PI prompt before exposing headroom:** `src/agents/pi-embedded-runner/run/attempt.ts:2549-2553`
  - body: The new PI `currentTokenCount` is based on `params.prompt`, but the run later applies prompt hooks, provenance/orphan repairs, runtime-context splitting, and `buildCurrentTurnPrompt(... params.currentTurnContext)` before precheck/model submission. A large current-turn context is therefore invisible to context engines, so they can compute too much remaining headroom and still over-inject. Please compute the estimate from the same prompt text the model/precheck will see, or move assembly until that prompt is resolved.
  - confidence: 0.86
- **[P2] Include Codex current-turn context in the estimate:** `extensions/codex/src/app-server/run-attempt.ts:626-629`
  - body: The Codex path has the same undercount: it estimates with `params.prompt` here, then prepends `params.currentTurnContext` after assembly. Existing coverage expects that context at the front of the Codex prompt, so engines will see a smaller `currentTokenCount` than the real request uses. Include that known context before estimating and add coverage for this case.
  - confidence: 0.84
- **[P3] Correct the precheck ordering note:** `docs/concepts/context-engine.md:131`
  - body: This new docs sentence says OpenClaw's preemptive compaction runs before `assemble`, but the embedded runner assembles context first and calls `shouldPreemptivelyCompactBeforePrompt` later with the post-assemble system prompt. Please rewrite this note so plugin authors do not learn the wrong runtime order while adopting the new field.
  - confidence: 0.9
