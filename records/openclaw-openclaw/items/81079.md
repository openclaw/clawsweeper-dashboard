## Summary

Keep this PR open, but it is not merge-ready: the current head still leaves the PI tool-loop assemble path without a production `currentTokenCount`, and the external proof is unit-test output rather than a real long-session run.

## What This Changes

The PR adds optional `currentTokenCount` to `ContextEngine.assemble`, threads estimates through PI and Codex assemble paths, re-exports `estimatePrePromptTokens`, and updates focused docs/tests.

## Best Possible Solution

Define one supported pre-assembly token-count contract, populate it in every production assemble path including mid-tool-loop assembly, document the public SDK/API seam, and merge only after real long-session proof is added.

## Reproduction Assessment

Yes for the review finding: source inspection shows the actual embedded-runner loop callback still builds runtime context without `currentTokenCount`, so the new loop-hook pass-through remains undefined there. The downstream long-session overflow itself was not live-reproduced here.

## Solution Assessment

No, not yet. The API direction is plausible, but the patch should populate the real tool-loop runtime context or derive the estimate inside the hook before merge.
