## Summary

The PR is still useful but not merge-ready: it adds the new assemble token-count API and covers the main PI/Codex assemble callers, but the production context-engine tool-loop path still never supplies the estimate that the new hook reads.

## What This Changes

The PR adds optional `currentTokenCount` to `ContextEngine.assemble`, threads pre-assembly token estimates through PI and Codex assemble paths, re-exports `estimatePrePromptTokens`, and updates docs/tests.

## Best Possible Solution

Define one supported pre-assembly token-count contract, populate it in every production assemble path including mid-tool-loop assembly, document the SDK/API seam, and merge after real long-session proof is added.

## Reproduction Assessment

Yes for the review finding: source inspection shows the real embedded-runner loop callback still builds runtime context without `currentTokenCount`, so the new loop-hook pass-through remains undefined there. The downstream provider overflow itself was not live-reproduced here.

## Solution Assessment

No, not yet. The API direction is plausible, but the patch should populate the real tool-loop runtime context or derive the estimate inside the hook before merge.
