## Summary

Keep open: this is a plausible SDK feature, not cleanup-close material, but the patch still needs a token-estimate fix, real behavior proof, and maintainer approval of the public ContextEngine contract.

## What This Changes

The PR adds optional currentTokenCount to ContextEngine.assemble, threads pre-prompt token estimates through PI and Codex harness assemble paths, re-exports estimatePrePromptTokens, and updates docs/tests.

## Best Possible Solution

Land one additive token-count contract only after the estimate covers the prompt blocks the model will actually see on PI and Codex paths and redacted live long-session proof is added.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR estimates from params.prompt, while Codex and PI later include currentTurnContext in the final prompt. No high-confidence live reproduction of the downstream overflow scenario is supplied.

## Solution Assessment

No, not yet. The direction is reasonable, but the current estimate can undercount real prompt usage; the safer solution is to estimate against the final prompt shape before exposing it as an engine headroom signal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Include current-turn context in the estimate:** `extensions/codex/src/app-server/run-attempt.ts:629`
  - body: `currentTokenCount` is meant to let engines size `systemPromptAddition` against remaining headroom, but this estimate only uses `params.prompt`. The Codex path later prepends `params.currentTurnContext` before `thread/start`, so reply/channel context tokens are invisible to the engine and a large context block can still make an apparently fitting addition overflow. Estimate against the final prompt text or otherwise include current-turn context before passing `currentTokenCount`.
  - confidence: 0.86
- **[P2] Include current-turn context in the PI estimate:** `src/agents/pi-embedded-runner/run/attempt.ts:2552`
  - body: The PI runner also computes `currentTokenCount` from the raw prompt before the later prompt build adds `params.currentTurnContext` to the model prompt. That makes the new field lower than the request the model actually sees, so engines can over-inject on reply-context turns. Include the same current-turn prompt context in this estimate or document and test a deliberately conservative alternative.
  - confidence: 0.78
