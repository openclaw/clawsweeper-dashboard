## Summary

The prior Codex harness gap appears addressed, but the re-review found a remaining source-level wiring gap: the PR adds a loop-hook pass-through, while the actual embedded-runner callback feeding that hook still does not populate `currentTokenCount`, so context engines can still see the field as absent during mid-tool-loop assembly. The branch also still has only test-layer proof despite the external-PR real behavior proof requirement.

## What This Changes

The PR adds optional `currentTokenCount` to `ContextEngine.assemble`, threads a pre-assembly estimate through the main PI and Codex harness assemble paths, exposes `estimatePrePromptTokens` from the agent harness SDK, and updates docs and focused tests.

## Best Possible Solution

Define one supported pre-assembly token-count contract, pass it through every production context-engine assemble path including mid-tool-loop assembly, and land only after contributor-supplied real long-session proof is added.

## Reproduction Assessment

Yes for the review finding: source inspection shows the actual embedded-runner `getRuntimeContext` callback still builds runtime context without `currentTokenCount`, so the new loop-hook pass-through has no production value on that path. The underlying downstream overflow scenario itself was not reproduced live here.

## Solution Assessment

No, not yet. The added field is a reasonable API direction, but the patch should also populate the tool-loop runtime context from the real embedded-runner caller or compute the estimate inside the hook before merge.
