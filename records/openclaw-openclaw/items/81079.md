## Summary

Keep open: the API direction is plausible, but the patch still leaves the mid-tool-loop assemble path without a production `currentTokenCount`, and the external PR has only test-level proof for a live overflow scenario.

## What This Changes

The PR adds optional `currentTokenCount` to `ContextEngine.assemble`, threads token estimates through PI and Codex assemble callers, exports `estimatePrePromptTokens` from the harness runtime SDK, and documents the new contract.

## Best Possible Solution

Define one supported pre-assembly token-count contract and make every production assemble path, including mid-tool-loop assembly, supply it consistently before merging the public SDK/API change.

## Reproduction Assessment

Yes for the review finding: source inspection shows the production loop caller builds runtime context without `currentTokenCount`, while the PR's hook only forwards that field if it is already present. The downstream long-session overflow itself was not live-reproduced here.

## Solution Assessment

No, not yet. The added field is a reasonable direction, but the patch should also populate the mid-tool-loop path from the real caller or compute the estimate inside the hook before merge.
