## Summary

Keep open: the API direction is plausible, but the latest patch still leaves the production PI tool-loop assemble path without an actual currentTokenCount because the hook only forwards a value the real caller never supplies, and the external PR still has only test-layer proof.

## What This Changes

This PR adds optional currentTokenCount to ContextEngine.assemble, threads pre-assembly token estimates through PI/Codex harness assembly, re-exports estimatePrePromptTokens, and updates context-engine docs/tests.

## Best Possible Solution

Populate one maintainer-approved pre-assembly token-count contract on every production context-engine assemble path, including mid-tool-loop assembly, and land it with contributor-supplied live long-session proof.

## Reproduction Assessment

Yes for the review finding: source inspection of current main and the PR diff shows the embedded-runner getRuntimeContext callback builds runtime context without currentTokenCount while the new hook only forwards that optional field. The downstream overflow scenario itself has not been live-reproduced here or in the PR body.

## Solution Assessment

No, not yet. The optional field is a reasonable API direction, but the patch should populate or derive the estimate for the real loop-hook caller and receive maintainer approval for the new public SDK semantics.
