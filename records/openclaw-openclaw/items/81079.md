## Summary

Keep open: the API direction is plausible, but the patch still leaves the real PI mid-tool-loop assemble path without a populated `currentTokenCount`, and the external PR proof remains test-only with no live long-session run.

## What This Changes

The branch adds optional `currentTokenCount` to `ContextEngine.assemble`, threads estimated pre-prompt tokens through PI and Codex assemble paths, re-exports the estimator, and updates docs/tests.

## Best Possible Solution

Populate `currentTokenCount` consistently for initial PI, Codex, and mid-tool-loop assemble, keep the estimator behind an approved SDK seam, and land only after redacted real-session proof is added.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection shows the changed loop hook forwards only a value from `getRuntimeContext()`, while the production caller builds that context without `currentTokenCount`. No live reproduction of the downstream overflow scenario is supplied.

## Solution Assessment

No, not yet. The optional field is a reasonable approach, but the patch should also populate the real mid-tool-loop path or derive the estimate in the hook, and the SDK export needs maintainer signoff.
