## Summary

Keep open: the PR is a useful API direction, but it is not merge-ready because the new assemble token-count field is not wired through all production harness callers, and the external-PR proof is only unit/integration-test output.

## What This Changes

The branch adds an optional `currentTokenCount` field to `ContextEngine.assemble` and threads it through the harness helper, embedded runner, and context-engine tool-loop hook with tests.

## Best Possible Solution

Land this only after all production context-engine assemble paths pass a consistent, documented token estimate and the contributor supplies redacted real behavior proof from a long-session context-engine setup.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a high-confidence review reproduction for the patch gap: the Codex harness call to `assembleHarnessContextEngine` still has no `currentTokenCount` argument, while the PR only computes the value in the PI runner path.

## Solution Assessment

Is this the best way to solve the issue? Not yet; the proposed field is reasonable, but the maintainable fix is to define the estimate semantics once and pass it consistently through every production assemble caller that exposes the public context-engine contract.
