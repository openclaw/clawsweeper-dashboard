## Summary

Keep open: the field is useful, but the proposed wiring is incomplete for production assemble paths and the supplied proof is only mock/unit output.

## What This Changes

The PR adds an optional `currentTokenCount` field to `ContextEngine.assemble`, wires it through embedded attempt/harness and loop-hook paths, and adds focused pass-through tests.

## Best Possible Solution

Make `currentTokenCount` a documented optional assemble parameter supplied by every production context-engine assemble path from a shared host-owned estimate, then verify it with a real downstream context-engine run.

## Reproduction Assessment

Yes for the review findings by source inspection: the loop-hook reads a value the only production callback does not provide, and the Codex harness caller still omits the new parameter. I did not run tests because this review is required to keep the checkout read-only.

## Solution Assessment

No; the API direction is likely right, but this patch only fully handles the main embedded assemble path and helper-level tests. The safer solution is to compute or carry the host estimate once and reuse it for embedded, owned-compaction loop, and Codex harness assemble calls, with docs and real proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Provide a real count for loop-hook assemble:** `src/agents/pi-embedded-runner/tool-result-context-guard.ts:330-334`
  - body: This reads `currentTokenCount` from `getRuntimeContext`, but the only production caller builds that context without passing a count. As a result, owned-compaction engines still see no headroom signal during post-tool-call assemble calls; compute or thread the estimate in the production callback before relying on this field.
  - confidence: 0.88

  - body: Adding the wrapper parameter does not make the bundled Codex harness supply it: `extensions/codex/src/app-server/run-attempt.ts` still calls `assembleHarnessContextEngine` without `currentTokenCount`. Context-engine plugins running through that documented harness path will still get `undefined` and cannot use the new headroom guard there.
  - confidence: 0.82
