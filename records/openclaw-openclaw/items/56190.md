## Summary

Keep open. Current main still only supports plain `${VAR}` config substitution, and the PR is useful, but its latest head still leaves fallback-bearing `env.vars` entries on the unresolved-reference path before substitution.

## What This Changes

The PR adds `${VAR:-default}` config env substitution support, preservation round-trip parsing, a brace scanner helper, and focused unit coverage.

## Best Possible Solution

Land the feature after defaulted refs are treated as safely resolvable in config env injection, with chaining regression coverage plus public docs and changelog coverage.

## Reproduction Assessment

Yes. From the PR head source, `env.vars.API_KEY = "${UPSTREAM:-demo}"` is parsed as a substitution, `containsEnvVarReference` returns true, and `applyConfigEnvVars` skips it before a later `${API_KEY}` can resolve.

## Solution Assessment

No. The parser/preservation approach is reasonable, but the implementation needs a small injection-path repair so fallback-resolvable refs are not treated like unresolved required refs, and the new public syntax needs docs/changelog follow-through.
