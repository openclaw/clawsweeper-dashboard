## Summary

Keep open: current main does not implement fallback env syntax, and the PR head still has a source-proven P2 defect in config env injection plus missing real behavior proof for the external feature PR.

## What This Changes

This PR adds `${VAR:-default}` config env fallback parsing, write-back preservation, a shared brace scanner, and focused unit coverage under `src/config`.

## Best Possible Solution

Land a revised config-contract feature that treats fallback templates consistently across read, config env injection, write-back, and docs, with redacted real config-load proof attached.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: `${VAR:-default}` remains a normal `substitution`, and `applyConfigEnvVars()` skips any value where `containsEnvVarReference()` is true before substitution. Not applicable as a current-main bug because fallback syntax is a new feature not present on main.

## Solution Assessment

No. The parser and preservation direction is plausible, but the patch should distinguish fallback-resolvable refs during config env injection, update the public config docs, and include real config-load proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Distinguish defaulted refs before unresolved-ref checks:** `src/config/env-substitution.ts:82`
  - body: `parseEnvTokenAt()` returns `${VAR:-default}` as a normal `substitution`, so `containsEnvVarReference()` still treats it as unresolved. Since `applyConfigEnvVars()` runs before substitution and skips those values, `env.vars.API_KEY = "${UPSTREAM:-demo}"` will not populate `API_KEY` for a later `${API_KEY}` consumer even though the fallback can resolve safely.
  - confidence: 0.9
