## Summary

Keep open. Current main still only supports plain `${VAR}` config substitution, and the PR is a plausible feature, but the latest PR head still classifies defaulted tokens as unresolved so `env.vars` fallback chaining can be skipped before substitution; public docs and changelog coverage are also missing.

## What This Changes

The PR adds `${VAR:-default}` config env substitution and env-reference preservation support, plus a brace scanner and focused unit tests.

## Best Possible Solution

Land the feature after fallback-bearing env refs are distinguished from truly unresolved refs for config env injection, with regression coverage plus public docs and an active changelog entry.

## Reproduction Assessment

Yes, from source inspection of the PR head: configure `env.vars.API_KEY = "${UPSTREAM:-demo}"` and another config field as `${API_KEY}`; the fallback token is still reported as an env reference, so `applyConfigEnvVars` skips injecting `API_KEY` before substitution.

## Solution Assessment

No. The parser direction is reasonable, but the merge shape should either resolve fallback-bearing config env values before injection or expose a helper that only skips truly required unresolved refs; the public syntax also needs docs and changelog coverage.
