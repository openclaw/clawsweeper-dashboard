## Summary

Keep open: current main has not implemented fallback syntax, and the PR still has a source-reproducible env.vars chaining bug plus no real after-fix proof. This is a new public config feature that needs maintainer agreement before merge.

## What This Changes

The PR adds `${VAR:-default}` parsing and write-back preservation for config env substitution, a brace-aware parser helper, and unit tests.

## Best Possible Solution

Land the feature only after fallback-resolvable refs work through config env injection, public docs cover the syntax, and the contributor provides redacted real config-load proof.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: PR head still classifies `${UPSTREAM:-demo}` as a substitution reference, and config env injection skips it before later `${API_KEY}` substitution. The requested fallback syntax itself is a new feature, not a current-main bug reproduction.

## Solution Assessment

No. The parser and preservation direction is reasonable, but the patch needs to distinguish fallback-resolvable refs from unresolved required refs during env.vars injection and document the new public syntax before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Resolve defaulted env refs before injection:** `src/config/env-substitution.ts:82`
  - body: `${VAR:-default}` is still returned as a normal `substitution`, so `containsEnvVarReference()` treats it as unresolved. Because `applyConfigEnvVars()` skips unresolved refs before `resolveConfigEnvVars()` runs, `env.vars.API_KEY = "${UPSTREAM:-demo}"` will not populate `API_KEY` for a later `${API_KEY}` consumer even though the fallback can resolve safely.
  - confidence: 0.9
