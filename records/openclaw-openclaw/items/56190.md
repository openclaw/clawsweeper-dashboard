## Summary

Keep this PR open. The feature is useful and current main still lacks fallback syntax, but the latest PR head still has a blocking config-env chaining bug: fallback tokens are classified as ordinary unresolved references, so `env.vars` values that could resolve from a default are skipped before substitution. The PR also needs the required user-facing docs and changelog entry before merge.

## What This Changes

The PR adds a brace-aware env-token scanner and updates config env substitution, env-reference preservation, and unit tests to support `${VAR:-default}` fallbacks.

## Best Possible Solution

Land fallback syntax after the parser/helper API can distinguish required unresolved refs from fallback-resolvable refs, with a regression test for config-defined env chaining plus public docs and an active changelog entry for the new config syntax.

## Reproduction Assessment

Yes. On the PR head, configure `env.vars.API_KEY = "${UPSTREAM:-demo}"` and another config string as `${API_KEY}`; `applyConfigEnvVars` skips `API_KEY` because `containsEnvVarReference` returns true, so the later consumer remains unresolved even though the upstream fallback exists.

## Solution Assessment

No. The parser direction is reasonable, but the merge shape is not the best solution until fallback-bearing tokens either bypass unresolved-ref checks or are resolved before `applyConfigEnvVars`; a required-ref helper or distinct token kind would be safer.
