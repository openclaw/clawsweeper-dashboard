## Best Possible Solution

Land a revised config-contract change that resolves fallback templates consistently across substitution, `env.vars` injection, write-back preservation, docs, and focused tests.

## Reproduction Assessment

Yes for the PR-introduced defect by source inspection: fallback tokens are still ordinary `substitution` tokens, while current config read skips `env.vars` values reported by `containsEnvVarReference()` before substitution runs. A current-main bug repro is not applicable because the fallback syntax is not on main.

## Solution Assessment

No. The parser and preservation direction is plausible, but the branch should handle fallback-capable `env.vars` values, document the new config contract after maintainer agreement, and include redacted real config-load proof before merge.
