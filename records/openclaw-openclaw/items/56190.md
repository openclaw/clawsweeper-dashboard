## Summary

Keep open: this is a plausible public config feature, but it is not on current main and the PR still has a correctness blocker where fallback-resolved `env.vars` values are treated as unresolved references. It also needs contributor real behavior proof and maintainer agreement on the public config syntax before merge.

## What This Changes

The PR adds `${VAR:-default}` fallback parsing and preservation for config env substitution, a shared brace scanner, and unit coverage for the new syntax.

## Best Possible Solution

Land a revised feature that treats fallback-resolvable refs correctly through `env.vars` injection, documents the syntax, and includes redacted real config-load proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: `env.vars.API_KEY = "${UPSTREAM:-demo}"` is still classified as an unresolved env reference before later `${API_KEY}` substitution. The requested fallback syntax itself is a new feature, not a current-main bug reproduction.

## Solution Assessment

No. The parser and preservation direction is reasonable, but the patch needs to distinguish fallback-resolvable refs from required unresolved refs during config env injection and add docs/proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Resolve defaulted env refs before injection:** `src/config/env-substitution.ts:82`
  - body: `env.vars` values can feed later `${API_KEY}` config references because `resolveConfigForRead()` calls `applyConfigEnvVars()` before substitution. This patch still returns `${UPSTREAM:-demo}` as `kind: "substitution"`, and `containsEnvVarReference()` returns true for every substitution token, so `applyConfigEnvVars()` skips `API_KEY: "${UPSTREAM:-demo}"`; a later `${API_KEY}` remains unresolved even though the fallback is safe to compute.
  - confidence: 0.91
