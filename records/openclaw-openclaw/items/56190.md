## Summary

Keep this PR open. Current main still only supports plain `${VAR}` config substitution, so the feature is useful, but the current PR head is not merge-ready because fallback-bearing tokens are still classified as ordinary substitutions and therefore conflict with the existing config-defined env var injection path. The previous ClawSweeper P2 still applies to the current head, and the user-facing syntax also needs docs and changelog coverage.

## What This Changes

The PR adds a brace-aware env-token scanner and updates config env substitution, env-reference preservation, and unit tests to support `${VAR:-default}` fallbacks.

## Best Possible Solution

Land the fallback syntax only after the helper contract distinguishes required unresolved refs from fallback-resolvable refs, add a regression test for config-defined env chaining, and document the new public config syntax with an active changelog entry.

## Reproduction Assessment

Yes. On the PR head, `env.vars.API_KEY = "${UPSTREAM:-demo}"` plus a later `${API_KEY}` consumer follows the static read path where `applyConfigEnvVars` runs before substitution and skips values reported by `containsEnvVarReference`, so the fallback-resolvable env var is not injected.

## Solution Assessment

No. The parser direction is reasonable, but the current merge shape is not the narrowest maintainable solution until fallback tokens either use a distinct required-ref classification or the env-var injection path resolves fallback-bearing values before deciding to skip them.
