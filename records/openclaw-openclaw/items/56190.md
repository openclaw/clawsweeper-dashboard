## Summary

Keep open. The PR is useful, but the latest head still leaves fallback-bearing config env values incompatible with the pre-substitution `env.vars` injection path, and it is missing the required public docs and changelog entry for a user-facing config syntax change.

## What This Changes

The PR adds brace-aware env-token parsing plus substitution/preservation tests to support `${VAR:-default}` fallbacks in config strings.

## Best Possible Solution

Land the feature after fallback-bearing env values are resolved safely for config env injection, with chaining regression coverage plus public docs and an active changelog entry.

## Reproduction Assessment

Yes. On the PR head, configure `env.vars.API_KEY = "${UPSTREAM:-demo}"` and another config field as `${API_KEY}`; the static read path shows `applyConfigEnvVars` skips `API_KEY` before substitution because the fallback token is still reported as an env reference.

## Solution Assessment

No. The parser direction is reasonable, but the merge shape should resolve fallback-bearing config env values, or otherwise distinguish required unresolved refs from fallback-resolvable refs, before the skip path; docs and changelog also need to define the public contract.

## Review Findings
