## Summary

Keep open: current main and the latest release do not implement `${VAR:-default}` config fallback syntax, and the PR branch still leaves a blocking config env chaining path unresolved while also lacking real after-fix behavior proof.

## What This Changes

The PR adds `${VAR:-default}` fallback parsing to config env substitution and write-back preservation, with unit coverage for defaults, escaping, and brace handling.

## Best Possible Solution

Land the feature only after defaulted refs are distinguished from truly unresolved refs in config env injection, with chaining regression coverage and public docs for the new syntax.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `env.vars.API_KEY = "${UPSTREAM:-demo}"` is classified as an env reference and skipped before substitution, so a later `${API_KEY}` consumer cannot see the fallback-resolved value.

## Solution Assessment

No. The parser/write-back direction is reasonable, but the branch needs a small contract change or helper so fallback-resolvable tokens are not treated like unresolved required refs during config env injection.
