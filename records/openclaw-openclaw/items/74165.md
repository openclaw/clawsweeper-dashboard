## Summary

Keep open. Current main still lacks the requested live abort-source diagnostic, while the embedded runner still has the shared abort path and the named abort triggers. The related implementation PR is closed and unmerged, and the requested stack-frame logging is a core operator diagnostic that needs maintainer product/security approval before automation should implement it.

## What This Changes

Add opt-in structured logging for embedded runner abort sources, gated by an abort-source env flag, from the shared `abortRun` path.

## Best Possible Solution

If maintainers accept the diagnostic, add a small default-off helper in the embedded runner, classify the existing abort sources at the shared `abortRun` entry point without changing abort behavior, bound or redact stack-frame output, document the sensitivity under diagnostics, and cover the helper plus nearby timeout paths with focused tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source check is high-confidence: current main has the shared abort path and abort-state metadata, but the requested env gate, classifier, `[abort-source]` marker, stack capture, and tests are absent.

## Solution Assessment

Unclear until maintainers approve the core logging surface. A single helper called from `abortRun` is technically narrow, but fitting it into the existing diagnostics flag/redaction model may be safer than adding a separate env-only diagnostic.
