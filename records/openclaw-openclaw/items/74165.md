## Summary

Keep open. Current main still lacks the requested env-gated abort-source diagnostic, the related implementation PR is closed and unmerged, and the proposed stack-frame logging is a new core operator diagnostic surface that needs maintainer product/security approval before it is safe cleanup or automation work.

## What This Changes

Request to add a default-off `OPENCLAW_LOG_ABORT_SOURCES` diagnostic that logs classified embedded-run abort sources and caller stack frames from the shared `abortRun()` path.

## Best Possible Solution

If maintainers accept the diagnostic surface, implement a narrow helper owned by the embedded runner, call it once from `abortRun()`, classify only the existing abort sources without changing abort behavior, and document the stack-frame sensitivity alongside focused helper tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug reproduction. The source check is high-confidence: the described abort paths exist on current main, while the requested env gate, marker, classifier, stack capture, and test file do not.

## Solution Assessment

Unclear until maintainers approve the new core diagnostic surface. Technically, a single helper at `abortRun()` is the narrowest maintainable implementation, but using the existing diagnostics flag system and documenting or bounding stack output may be safer than a new standalone env var.
