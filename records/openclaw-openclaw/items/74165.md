## Summary

Keep open. Current main still lacks the requested live abort-source diagnostic, the linked implementation PR is closed unmerged, and the remaining decision is not purely mechanical because opt-in stack-frame logging is a core diagnostics/security surface that needs maintainer approval before implementation.

## What This Changes

The issue requests an opt-in embedded-run abort diagnostic gated by `OPENCLAW_LOG_ABORT_SOURCES=1` that logs a classified abort source and caller stack frames from the shared embedded runner abort path.

## Best Possible Solution

If maintainers accept the diagnostic, add a small default-off embedded-runner helper at the shared abort entry point, use the established diagnostics flag model or an explicitly approved env var, classify the existing abort sources without changing abort behavior, bound and redact stack-frame output, document the sensitivity, and cover the helper plus timeout/compaction paths with focused tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source check is high-confidence: current main has the shared abort path and abort-state metadata, but the requested env gate, `[abort-source]` marker, classifier, stack capture, and tests are absent.

## Solution Assessment

Unclear until maintainers approve the core diagnostic surface. A single helper called from `abortRun` is technically narrow, but using the existing `diagnostics.flags`/`OPENCLAW_DIAGNOSTICS` model and explicit stack redaction would be safer than landing a new env-only stack logger without product/security sign-off.
