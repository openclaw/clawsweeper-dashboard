## Summary

Keep open. Current main still lacks the requested opt-in abort-source diagnostic, the linked implementation PR is closed unmerged, and stack-frame abort logging is a core diagnostics/security surface that needs maintainer approval before implementation.

## What This Changes

Add default-off structured abort-source logging for embedded run aborts, gated by `OPENCLAW_LOG_ABORT_SOURCES`, with source classification and bounded caller stack frames.

## Best Possible Solution

If maintainers accept the diagnostic, implement it narrowly at the shared `abortRun` entry point using the approved diagnostics flag surface, bounded/redacted stack output, docs, and focused abort-source tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. Source inspection is high-confidence: current main has the shared abort path and post-run abort metadata, but not the requested env gate, classifier, `[abort-source]` marker, stack capture, or tests.

## Solution Assessment

Unclear until maintainers approve the diagnostic surface. A single helper at `abortRun` is technically narrow, but aligning with `diagnostics.flags`/`OPENCLAW_DIAGNOSTICS` and explicit stack redaction is safer than adding a standalone env-only stack logger without review.
