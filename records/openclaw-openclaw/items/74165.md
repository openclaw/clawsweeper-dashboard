## Summary

Keep open. Current main still lacks the requested live abort-source diagnostic, the embedded runner still has the shared abort path and named abort triggers, and the related implementation PR is closed unmerged. The code change is narrow, but stack-frame logging is a core diagnostic surface that needs maintainer product/security approval before automation should implement it.

## What This Changes

Request an opt-in `OPENCLAW_LOG_ABORT_SOURCES=1` diagnostic that logs a classified `[abort-source]` line with caller stack frames whenever the embedded runner aborts.

## Best Possible Solution

If maintainers accept this diagnostic, add a small default-off helper in the embedded runner, classify the existing abort sources at the shared `abortRun` entry point without changing abort behavior, bound or redact stack-frame output, document the sensitivity under diagnostics, and cover the helper plus nearby timeout paths with focused tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source check is high-confidence: current main has the shared abort path and abort-state metadata, but the requested env gate, classifier, `[abort-source]` marker, stack capture, and tests are absent.

## Solution Assessment

Unclear until maintainers approve the core logging surface. A single helper called from `abortRun` is technically narrow, but aligning it with existing diagnostics flags and explicit stack redaction is likely safer than adding an env-only diagnostic as proposed.
