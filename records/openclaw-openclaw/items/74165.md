## Summary

Keep open. Current main still lacks the requested live abort-source diagnostic, and the related implementation PR is closed unmerged. The change is narrow in code, but logging caller stack frames from a core embedded-run abort path is a security-sensitive diagnostic surface that needs maintainer approval on the flag shape and redaction model before automation should implement it.

## What This Changes

Add a default-off structured embedded-run abort diagnostic that classifies abort sources and logs caller stack frames when enabled by an operator flag.

## Best Possible Solution

If maintainers accept the diagnostic, implement it at the shared abortRun entry point, classify the existing abort sources without changing abort behavior, bound and redact any stack output, document the sensitivity under diagnostics, and cover the helper plus timeout paths with focused tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source check is high-confidence: current main has the abort path and abort-state metadata, but the requested env gate, classifier, [abort-source] marker, stack capture, and tests are absent.

## Solution Assessment

Unclear until maintainers approve the diagnostic surface. A small helper called from abortRun is technically narrow, but using the existing diagnostics flag conventions with explicit redaction is safer than adding raw env-only stack-frame logging as proposed.
