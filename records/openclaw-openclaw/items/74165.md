## Summary

Keep open. Current main still lacks the requested live abort-source diagnostic, the related implementation PR is closed unmerged, and opt-in caller stack logging is a core diagnostics/security decision that needs maintainer approval before automation should implement it.

## What This Changes

The issue requests a default-off embedded-runner diagnostic that classifies abort sources and logs bounded caller stack frames when enabled.

## Best Possible Solution

If maintainers accept the diagnostic, implement it as a narrow default-off helper at the shared abort entry point using the approved diagnostics flag surface, bounded/redacted stack output, docs, and focused timeout/compaction tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source check is high-confidence: current main has the shared abort path and abort-state metadata, but not the requested env gate, classifier, `[abort-source]` marker, stack capture, or tests.

## Solution Assessment

Unclear until maintainers approve the diagnostic surface. A single helper called from `abortRun` is technically narrow, but the existing diagnostics-flag model with explicit stack redaction is safer than adding a new env-only stack logger without review.
