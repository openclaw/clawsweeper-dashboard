## Summary

Keep open. Current main still lacks the requested live abort-source diagnostic, while the embedded runner still has a shared `abortRun` path and the named abort triggers. The related implementation PR is closed and unmerged, and adding opt-in stack-frame logging in core needs maintainer product/security approval before automation should implement it.

## What This Changes

The issue requests a default-off `OPENCLAW_LOG_ABORT_SOURCES=1` diagnostic that logs a structured `[abort-source]` line with abort classification and caller stack frames from embedded runner aborts.

## Best Possible Solution

If maintainers accept the diagnostic, add a small default-off embedded-runner helper at the shared `abortRun` entry point, classify the existing abort sources without changing abort behavior, use the approved diagnostics flag surface, bound and redact stack output, document the sensitivity, and cover classification/gating plus nearby timeout paths with focused tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source check is high-confidence: current main has the shared abort path and abort-state metadata, but the requested env gate, classifier, `[abort-source]` marker, stack capture, and tests are absent.

## Solution Assessment

Unclear until maintainers approve the core logging surface. A helper called from `abortRun` is technically narrow, but fitting it into the existing diagnostics flag and redaction model is safer than adding a separate env-only diagnostic by default.
