## Summary

Current main still does not contain the requested abort-source diagnostic, and the only linked implementation was closed unmerged in favor of maintainer discussion. This should stay open because it is a new core diagnostics/config surface with stack and abort-reason logging, which needs product/security approval before implementation.

## What This Changes

Add an opt-in structured embedded-runner abort diagnostic that logs abort source and caller frames when enabled by `OPENCLAW_LOG_ABORT_SOURCES=1`.

## Best Possible Solution

If maintainers accept the idea, implement it as a narrow approved diagnostics flag with bounded/redacted stack output, documented behavior, and focused tests for abort-source priority.

## Reproduction Assessment

Not applicable: this is a feature request for a new diagnostics surface, not a broken existing contract. Source inspection gives high-confidence proof that current main lacks the requested env flag, log marker, classifier, stack capture, and tests.

## Solution Assessment

Unclear until maintainers approve the diagnostic surface. A single helper at `abortRun` is technically narrow, but fitting it into the existing `diagnostics.flags`/`OPENCLAW_DIAGNOSTICS` model with explicit redaction would be safer than adding a standalone stack logger.
