## Summary

Keep open. Current main still lacks the requested abort-source diagnostic, the embedded runner still has the shared abort path and named abort triggers, and the only implementation PR was closed unmerged so maintainers still need to decide whether this core diagnostic/logging surface is acceptable.

## What This Changes

The issue requests an opt-in structured abort-source diagnostic for embedded runner aborts, gated by `OPENCLAW_LOG_ABORT_SOURCES=1`, with source classification and bounded caller stack frames.

## Best Possible Solution

If maintainers accept the diagnostic, add a default-off helper in the embedded runner, call it once from `abortRun`, classify the existing abort sources without changing run behavior, bound or redact stack/reason output, document the sensitivity, and cover the classifier and gate with focused tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source verification path is high-confidence: current main has the shared abort path and named triggers, but lacks the requested env gate, source classifier, marker log line, stack capture, and tests.

## Solution Assessment

Unclear until maintainer approval. A single helper at `abortRun` is technically narrow, but the maintainable design should probably align with existing diagnostics flags and redaction rules before adding a new core logging surface.
