## Summary

Keep open. Current main still lacks the requested live abort-source diagnostic, the shared embedded-run abort path is still uninstrumented, and the related implementation PR was closed without merge. The change is technically narrow, but logging caller stack frames from a core abort path needs maintainer product/security approval before it should be implemented automatically.

## What This Changes

Add an opt-in `OPENCLAW_LOG_ABORT_SOURCES` diagnostic that classifies embedded-run `abortRun()` triggers and logs a structured `[abort-source]` line with bounded caller stack frames.

## Best Possible Solution

If maintainers accept the diagnostic, add a default-off embedded-runner helper at the shared `abortRun` entry point, classify the existing abort sources without changing abort behavior, emit bounded and sanitized structured data through the approved diagnostics/logging surface, document the sensitivity, and cover classifier/gating plus timeout and compaction paths with focused tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source check is high-confidence: current main has the shared abort path and abort-state metadata, but the requested env gate, classifier, `[abort-source]` marker, stack capture, and tests are absent.

## Solution Assessment

Unclear until maintainers approve the logging surface. A single helper called from `abortRun` is technically narrow, but the safer direction is to align it with diagnostics flags and explicit stack redaction instead of adding raw stack-frame logging by env var alone.
