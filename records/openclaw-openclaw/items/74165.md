## Summary

Keep open. Current main still lacks the requested abort-source diagnostic, the only matching implementation PR is closed unmerged, and stack-frame logging is a core diagnostics/security surface that needs maintainer approval before automation should implement it.

## What This Changes

The issue requests a default-off structured diagnostic for embedded runner aborts, gated by `OPENCLAW_LOG_ABORT_SOURCES=1`, that classifies the abort source and logs bounded caller stack frames.

## Best Possible Solution

If maintainers accept the diagnostic, add a small default-off helper at the shared abort entry point, classify existing abort sources without changing abort behavior, use the approved diagnostics flag/env surface, bound and redact stack output, document sensitivity, and cover the helper plus timeout/compaction paths with focused tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source check is high-confidence: current main has the shared abort path and abort-state metadata, but not the requested env gate, classifier, `[abort-source]` marker, stack capture, or tests.

## Solution Assessment

Unclear until maintainers approve the diagnostics surface. A single helper called from `abortRun` is technically narrow, but using the existing diagnostics-flag model and explicit stack redaction would be safer than landing a new env-only stack logger without product/security sign-off.
