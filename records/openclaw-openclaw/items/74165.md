## Summary

Keep open. Current main still lacks the requested abort-source diagnostic, the linked implementation PR is closed unmerged, and adding stack/reason abort logging is a core diagnostics surface that needs maintainer product/security approval before implementation.

## What This Changes

The issue requests an opt-in structured diagnostic for embedded runner aborts, gated by `OPENCLAW_LOG_ABORT_SOURCES=1`, that classifies abort sources and logs caller stack frames.

## Best Possible Solution

If maintainers accept the idea, implement it as a narrow core diagnostics flag using the approved diagnostics surface, bounded/redacted stack output, docs, and focused abort-source tests.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing contract. Source inspection does give high-confidence proof that current main lacks the requested env flag, classifier, marker, stack capture, and tests.

## Solution Assessment

Unclear until maintainers approve the diagnostic surface. A single helper at `abortRun` is technically narrow, but using the existing `diagnostics.flags`/`OPENCLAW_DIAGNOSTICS` model with explicit stack redaction is safer than adding a standalone env-only stack logger without review.
