## Summary

Keep open. Current main still lacks the requested abort-source diagnostic, while the embedded runner retains the shared abortRun path and the named abort triggers. The related implementation PR is closed and unmerged, and the proposed stack-frame logging is a core operator diagnostic that should get maintainer product/security approval before repair automation picks it up.

## What This Changes

Add an opt-in structured diagnostic for embedded run aborts, gated by `OPENCLAW_LOG_ABORT_SOURCES=1`, that logs the classified abort source and caller stack frames.

## Best Possible Solution

If maintainers accept the diagnostic surface, add a narrow default-off helper in the embedded runner, call it once from `abortRun`, classify the existing abort sources without changing run behavior, bound or redact stack-frame output, document the sensitivity, and cover the helper with focused tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source check is high-confidence: current main has the shared abort path and named triggers, but the requested env gate, marker, classifier, stack capture, and tests are absent.

## Solution Assessment

Unclear until maintainers approve the core operator diagnostic. A single helper called from `abortRun` is technically narrow, but using the existing diagnostics flag surface and documenting/redacting stack output may be safer than adding another standalone env var.
