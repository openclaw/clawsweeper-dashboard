## Summary

Keep open: current main still lacks the requested per-abort source diagnostic, and the linked implementation PR is closed unmerged. Adding stack and abort-reason logging would be a new core diagnostics/config surface, so this needs maintainer product and security review before implementation.

## What This Changes

The issue requests an opt-in structured abort-source diagnostic for embedded runner aborts, gated by `OPENCLAW_LOG_ABORT_SOURCES`.

## Best Possible Solution

If maintainers accept the idea, implement it as a bounded diagnostics flag using the existing diagnostics model, with redacted stack/reason output and focused priority tests.

## Reproduction Assessment

Not applicable: this is a feature request for a new diagnostics surface, not a broken existing contract. Source inspection on current main confirms the requested env flag, log marker, classifier, stack capture, and tests are absent.

## Solution Assessment

Unclear until maintainers approve the diagnostic surface. The single `abortRun` hook is technically narrow, but routing it through `diagnostics.flags`/`OPENCLAW_DIAGNOSTICS` with explicit redaction would be safer than adding an unreviewed standalone stack logger.
