## Summary

Keep open. Current main still lacks the requested abort-source diagnostic, while the embedded runner still has the shared abortRun entry point and the named abort triggers. The related implementation PR was closed unmerged, and the request is a core operator diagnostic that should get maintainer product/security approval before an automated repair lane implements stack-frame logging.

## What This Changes

Request an opt-in embedded-run abort diagnostic, gated by OPENCLAW_LOG_ABORT_SOURCES=1, that classifies abortRun sources and logs bounded caller stack frames.

## Best Possible Solution

If maintainers accept the diagnostic, add a narrow default-off helper in the embedded runner, call it once from abortRun, classify existing abort sources without changing run behavior, bound and redact stack-frame output, document the sensitivity, and cover the helper plus the nearest timeout paths with focused tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source check is high-confidence: current main has the shared abort path and named triggers, but the requested env gate, marker, classifier, stack capture, docs, and tests are absent.

## Solution Assessment

Unclear until maintainers approve the core operator diagnostic. A single helper at abortRun is technically narrow, but using the existing diagnostics flag system with bounded/redacted stack output is the safer maintainable direction.
