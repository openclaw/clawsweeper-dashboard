## Summary

Keep open: current main only supports `systemEvent` and `agentTurn` cron payloads across the protocol, runtime, CLI, UI, and docs, so the requested script target is a new security-sensitive capability rather than an already-implemented fix.

## What This Changes

Request adding a native `kind: "script"` cron payload that runs a Gateway subprocess with argv/env/timeout/stdout capture while keeping those jobs visible in `openclaw cron list`.

## Best Possible Solution

Design a hardened first-class cron script execution contract, or explicitly choose lifecycle hooks as the approved surface, then update protocol/schema, runtime, CLI/UI, docs, and tests together with clear trust-boundary documentation.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection does confirm current main lacks a script payload path in schema, validation, CLI, runtime dispatch, UI types, and docs.

## Solution Assessment

Unclear until maintainer/security design review. A native script payload could solve the unified-inventory and token-cost problem, but the safest implementation needs explicit subprocess, environment, output, and permission semantics before code work starts.
