## Summary

Close as duplicate/superseded. The same zero-LLM cron script/exec payload work is already tracked by the open canonical issue at https://github.com/openclaw/openclaw/issues/18160, and this issue adds useful macOS TCC context but not a distinct source-level fix path.

## What This Changes

The issue requests an additive `script` cron payload so scheduled jobs can run shell commands under the OpenClaw gateway instead of using agent turns or external launchd plists.

## Best Possible Solution

Consolidate this macOS/TCC datapoint into https://github.com/openclaw/openclaw/issues/18160 and design one additive script/exec cron contract there with explicit host-command security policy, protocol/schema, runtime, CLI/UI, docs, delivery, and tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main cannot express the requested payload because cron accepts only `systemEvent` or `agentTurn`.

## Solution Assessment

Yes for cleanup. Closing this duplicate in favor of the open canonical tracker is the narrowest maintainable triage outcome; implementation should be designed once on the canonical issue rather than split across near-identical script/exec requests.

## Security Review

Status: needs_attention

Summary: The underlying feature is security-sensitive because it would persist scheduled host-command execution under the gateway context.

Concerns:

- **[medium] Define persisted script execution trust boundaries:** `src/cron/service/jobs.ts:254`
  - body: A native script cron runner would let stored cron definitions launch host commands, so the canonical design needs explicit rules for approval, sandboxing, host routing, cwd/env binding, timeout, cancellation, retry, and delivery.
  - confidence: 0.93
- **[medium] Account for exec approval bypass risk:** `docs/security/THREAT-MODEL-ATLAS.md:252`
  - body: The security threat model already calls out exec approval bypass and path/alias manipulation risks; a zero-LLM scheduled command path must not bypass those controls by persistence or scheduler context.
  - confidence: 0.86
- **[medium] Specify output and environment redaction:**
  - body: The proposed `env` and stdout capture fields could expose secrets or private data through stored jobs, run logs, delivery, or UI/API responses unless redaction and size limits are part of the contract.
  - confidence: 0.86
