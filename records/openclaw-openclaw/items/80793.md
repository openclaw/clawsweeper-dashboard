## Summary

Keep open. Current main still limits cron payloads to `systemEvent` and `agentTurn`, and a native script cron runner would be a new security-sensitive protocol/CLI/runtime capability rather than conservative cleanup.

## What This Changes

The issue requests adding a native `script` cron payload kind so scheduled shell scripts can run under the OpenClaw gateway instead of external launchd jobs.

## Best Possible Solution

Design an additive gateway-owned script cron contract with explicit security policy, subprocess limits, redaction rules, schema/SDK/CLI/docs updates, and focused runtime coverage.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection does confirm the current main behavior the issue describes: cron accepts only `systemEvent` and `agentTurn` payloads.

## Solution Assessment

Unclear. A native `script` payload is a plausible direction, but the safest solution needs maintainer product and security design before implementation because it creates a privileged subprocess execution surface.

## Security Review

Status: needs_attention

Summary: The issue itself is security-sensitive because the requested feature would run arbitrary scripts under the gateway process context.

Concerns:

- **[medium] Define script execution trust boundaries:**
  - body: A script cron implementation would intentionally run local commands from the gateway and may inherit host permissions such as macOS TCC grants, so it needs an explicit approval, sandbox, path, timeout, and ownership model.
  - confidence: 0.9
- **[medium] Specify output and environment redaction:**
  - body: The proposed `env` and stdout-capture fields could expose secrets or private data through stored job definitions, run logs, delivery, or UI/API responses unless redaction and size caps are part of the contract.
  - confidence: 0.86
