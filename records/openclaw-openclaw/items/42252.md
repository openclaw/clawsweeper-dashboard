## Summary

Keep open: current main still has a source-reproducible split diagnostics path where doctor can print an RPC-level `Gateway not running.` note before the later daemon check reports supervisor runtime state. Entrypoint drift and Telegram 409 hints are partially covered, but there is still no consolidated doctor verdict or deterministic recovery ordering.

## What This Changes

The issue requests clearer `openclaw doctor` and gateway diagnostics for mixed LaunchAgent, runtime, service-drift, duplicate-poller, and recovery states.

## Best Possible Solution

Define a shared gateway diagnostics summary carrying service-loaded, runtime PID, RPC probe, service-config drift, port/listener, and recent channel-conflict facts, then render one final healthy/degraded/down verdict with deterministic next steps.

## Reproduction Assessment

Yes, at source level. Make the status RPC fail with `gateway closed` while the service remains loaded and `service.readRuntime()` reports running, and doctor can emit `Gateway not running.` before a later `Runtime: running` note.

## Solution Assessment

Yes. The maintainable path is a shared diagnostics contract and final verdict renderer; isolated message tweaks would leave split-brain facts scattered across doctor steps.
