## What This Changes

The issue requests clearer doctor/gateway diagnostics for mixed LaunchAgent, runtime, RPC, service-drift, and duplicate-poller states.

## Best Possible Solution

Define a shared gateway diagnostics summary that carries service-loaded, runtime PID, RPC probe, service-config drift, port/listener, and recent channel-conflict facts through doctor, then render one final healthy/degraded/down verdict with deterministic next steps.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make the gateway status RPC fail with `gateway closed` while the service remains loaded and `service.readRuntime()` reports running, and doctor can emit `Gateway not running.` before a later `Runtime: running` note.

## Solution Assessment

Yes, keeping this open for maintainer UX direction is the right path. The maintainable fix is a shared diagnostics contract and final verdict, not more isolated message tweaks for each mixed-state symptom.
