## Summary

Keep open. Current main still has a source-backed path for mixed doctor output: the gateway RPC health check can print “Gateway not running” before the later daemon/service pass prints a running runtime summary, and the broader consolidated verdict/action-ordering request is not fully implemented.

## What This Changes

Request clearer doctor/gateway diagnostics for mixed LaunchAgent, runtime, entrypoint-drift, and duplicate-poller states.

## Best Possible Solution

Add a single doctor health model that carries service-loaded, runtime pid, RPC probe, port, LaunchAgent, and channel conflict facts forward, then emits one final verdict with ordered next steps while leaving narrower service-config panel deduplication to the related issue.

## Reproduction Assessment

Yes, by current-source inspection: a `gateway closed` status RPC failure can emit “Gateway not running,” and a later loaded service runtime can emit `Runtime: running`. I did not execute the exact macOS LaunchAgent incident in this read-only review.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable fix should reuse existing doctor/status/diagnostics facts and avoid ad hoc output or a new JSON block that conflicts with existing `status --all` and diagnostics export surfaces.
