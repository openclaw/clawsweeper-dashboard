## Summary

Keep open: current main partially covers entrypoint drift and Telegram duplicate-poller hints, but doctor still reports gateway RPC liveness and service/runtime state through separate paths, so the mixed `Gateway not running.` plus later `Runtime: running` output remains possible. The remaining work needs maintainer UX direction for a consolidated verdict and optional structured diagnosis.

## What This Changes

Request clearer doctor/gateway diagnostics that consolidate LaunchAgent service state, runtime PID, RPC health, entrypoint drift, duplicate Telegram poller hints, deterministic recovery steps, and optional JSON diagnosis.

## Best Possible Solution

Define a shared gateway diagnostics summary that carries service-loaded, runtime PID, RPC probe, service-config drift, port/listener, and recent channel-conflict facts through doctor, then render one final healthy/degraded/down verdict with deterministic next steps.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make the gateway status RPC fail with `gateway closed` while the service remains loaded and `service.readRuntime()` reports running, and doctor can emit `Gateway not running.` before a later `Runtime: running` service note.

## Solution Assessment

Yes, keeping this open for maintainer UX direction is the right path. The maintainable fix is a shared diagnostics contract/final verdict, not more isolated messages for each mixed-state symptom.
