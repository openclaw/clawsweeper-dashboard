## Summary

Keep open: current main still has a source-reproducible mixed doctor output path, and the partial service-drift and Telegram 409 improvements do not provide the requested consolidated gateway diagnosis. The remaining work is a user-facing diagnostics contract, so it needs maintainer direction rather than cleanup closure.

## What This Changes

The issue requests clearer doctor and gateway diagnostics when LaunchAgent service state, runtime process state, RPC liveness, entrypoint drift, and duplicate channel poller signals disagree.

## Best Possible Solution

Define a shared gateway diagnostics summary that carries service-loaded, runtime PID, RPC probe, service-config drift, port/listener, and recent channel-conflict facts through doctor, then render one final healthy/degraded/down verdict with deterministic next steps.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make the gateway status RPC fail with `gateway closed` while the service remains loaded and `service.readRuntime()` reports running, and doctor can emit `Gateway not running.` before a later `Runtime: running` note.

## Solution Assessment

Yes. The requested consolidated verdict is the maintainable direction; isolated message tweaks would leave the underlying service/runtime/RPC facts split across doctor phases.
