## Summary

Keep open: current main has pieces of the requested work, including entrypoint-drift detection and Telegram duplicate-poller log hints, but the central doctor UX request is still not implemented because doctor can still report RPC failure separately from a running service/runtime state without a consolidated final verdict.

## What This Changes

The issue asks for clearer `openclaw doctor` and gateway diagnostics when LaunchAgent service state, runtime process state, entrypoint drift, and duplicate Telegram poller signals conflict.

## Best Possible Solution

Design a consolidated doctor diagnostics summary that carries service-loaded, runtime-pid, RPC-probe, service-config-drift, and channel-conflict facts forward, then emits one final state with deterministic next steps.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: make the status RPC close while the LaunchAgent service remains loaded and `readRuntime` returns running. Current doctor code can then print `Gateway not running.` from the health probe and later print a runtime summary from the daemon check.

## Solution Assessment

No, closing as implemented is not the best resolution because current main only covers parts of the request. The maintainable path is a small doctor diagnostics design pass that consolidates existing service, runtime, RPC, and channel facts instead of adding more independent notes.
