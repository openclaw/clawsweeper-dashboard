## Summary

Keep open: current main has partial improvements for entrypoint drift and Telegram poller conflicts, but the doctor flow still has a source-clear path to print `Gateway not running.` before later supervisor/runtime notes, and the requested consolidated final verdict does not exist yet.

## What This Changes

Request clearer `openclaw doctor` and gateway diagnostics for mixed macOS LaunchAgent, runtime, service-config drift, and duplicate Telegram poller states.

## Best Possible Solution

Implement a doctor-level diagnosis model that reconciles RPC health, supervisor load/runtime, service config drift, port state, and channel warnings into one final verdict with deterministic next steps; gate any JSON output behind a deliberate CLI contract decision.

## Reproduction Assessment

Yes, for the confusing mixed-status output by source inspection: doctor can print `Gateway not running.` from the RPC probe, then later print a loaded/running service runtime from daemon diagnostics. I did not run a live macOS LaunchAgent reproduction in this read-only Linux checkout.

## Solution Assessment

Unclear. The direction is reasonable, but the best fix should be a maintainer-approved doctor-level verdict/output contract rather than piecemeal suppression of individual lines.
