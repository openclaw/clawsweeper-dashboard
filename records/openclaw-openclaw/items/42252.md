## Summary

Keep open: current main and the latest release still have split doctor output where an RPC liveness failure can print `Gateway not running.` before the daemon check later reports a running supervisor runtime. The merged service-config panel cleanup partially addresses entrypoint-drift presentation, and Telegram now logs a duplicate-poller hint for 409 conflicts, but the requested consolidated final verdict, deterministic recovery order, and optional diagnosis shape are still missing.

## What This Changes

Improve `openclaw doctor` and gateway diagnostics so mixed LaunchAgent/service, runtime PID, RPC liveness, entrypoint drift, and duplicate Telegram poller states produce one actionable diagnosis.

## Best Possible Solution

Define a shared gateway diagnostics summary that carries service-loaded, runtime PID, RPC probe, service-config drift, listener/port, and recent channel-conflict facts, then render one final healthy/degraded/down verdict with deterministic next steps.

## Reproduction Assessment

Yes, at source level. Make the status RPC fail with `gateway closed` while the service remains loaded and `service.readRuntime()` reports running, and doctor can emit `Gateway not running.` before a later `Runtime: running` note.

## Solution Assessment

Yes, the requested direction is best solved as a shared diagnostics contract and final verdict renderer. Isolated wording tweaks would leave service, runtime, RPC, and channel-conflict facts split across doctor steps.
