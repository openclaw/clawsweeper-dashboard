## Summary

Keep open: current main still has separate doctor gateway-health and daemon checks that can emit the reported mixed messaging, although Telegram duplicate-poller logging and service drift detection have partially improved since the report.

## What This Changes

Request clearer doctor/gateway diagnostics that consolidate service, runtime, RPC, LaunchAgent drift, duplicate-poller, recovery-step, and optional JSON diagnosis output.

## Best Possible Solution

Centralize doctor’s gateway health facts into one final verdict that combines service loaded state, runtime PID, RPC/readiness, service-config drift, and channel conflict hints, then decide whether machine-readable diagnosis belongs in doctor or gateway status.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a closed gateway RPC call prints `Gateway not running.`, then the later daemon check can still print a running service runtime. I did not perform live macOS LaunchAgent validation in this read-only review.

## Solution Assessment

Unclear. The requested direction is sensible, but the best maintainable fix should first define a shared doctor/status diagnosis model instead of patching individual notes independently.
