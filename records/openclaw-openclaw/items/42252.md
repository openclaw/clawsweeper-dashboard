## Summary

Keep open: current main still keeps the gateway RPC health check and service/runtime diagnostics as separate doctor contributions, so the reported mixed-state output can still produce an interim `Gateway not running.` line and a later service runtime summary instead of one consolidated health verdict.

## What This Changes

The issue requests clearer `openclaw doctor` and gateway diagnostics for mixed LaunchAgent/runtime states, including one final health verdict, conflict-aware hints, and deterministic recovery steps.

## Best Possible Solution

Define a shared gateway diagnostics summary that carries RPC, service-loaded, runtime PID, service-config drift, port, and recent channel-conflict facts through doctor and emits one final healthy/degraded/down verdict with deterministic next steps.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make the gateway status RPC fail with `gateway closed` while the service remains loaded and `service.readRuntime()` returns running, and doctor can print `Gateway not running.` followed later by `Runtime: running`.

## Solution Assessment

No current implementation fully solves it. The maintainable fix is a single diagnostics contract rather than more isolated messages for each mixed-state symptom.
