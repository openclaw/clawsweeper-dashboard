## Summary

Keep open: current main has partial cleanup, including consolidated service-config panels and entrypoint-drift detection, but doctor can still emit an RPC-level `Gateway not running.` message and later service/runtime diagnostics without a single final verdict.

## What This Changes

Request clearer `openclaw doctor` and gateway diagnostics for mixed LaunchAgent/service/runtime states, Telegram duplicate-poller conflicts, entrypoint drift, ordered recovery steps, and optional machine-readable diagnosis.

## Best Possible Solution

Define and implement a doctor diagnostics summary that reuses existing service audit, runtime, RPC, port, and channel-conflict facts, then emits one final health state with deterministic next steps.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make the gateway status RPC close while the service remains loaded and `service.readRuntime` returns running; doctor can then print `Gateway not running.` and later a `Runtime: running` service summary.

## Solution Assessment

No. Closing as implemented is not the best resolution because current main covers entrypoint drift, service-config panel deduplication, and Telegram conflict hints only partially; the remaining work is a consolidated doctor diagnostics design rather than another isolated note.
