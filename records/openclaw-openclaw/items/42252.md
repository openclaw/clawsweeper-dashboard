## Summary

Keep open: current main has useful partial cleanup, but the central request for one consolidated doctor/gateway health verdict is still not implemented; the source path can still emit `Gateway not running.` from the RPC probe and later emit a running service runtime summary.

## What This Changes

The issue requests clearer `openclaw doctor` and gateway diagnostics for mixed LaunchAgent/runtime states, including a final health verdict, conflict-aware hints, action ordering, and optional machine-readable diagnosis output.

## Best Possible Solution

Define a shared gateway diagnostics summary that carries RPC, service-loaded, runtime PID, service-config drift, port, and recent channel-conflict facts through doctor and emits one final healthy/degraded/down verdict with deterministic next steps.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make the gateway status RPC fail with `gateway closed` while the service remains loaded and `service.readRuntime` returns running, and doctor can print `Gateway not running.` followed later by `Runtime: running`.

## Solution Assessment

No. The current partial fixes are useful, but the maintainable solution is a single diagnostics contract rather than additional isolated notes for each mixed-state symptom.
