## Summary

Keep open: current main still has split doctor diagnostics where an RPC failure can emit `Gateway not running.` before the later daemon/service path reports runtime state. Entrypoint drift and Telegram 409 conflict hints are partially covered, but the requested consolidated doctor verdict and deterministic recovery ordering are not implemented.

## What This Changes

The issue requests clearer `openclaw doctor` and gateway diagnostics for mixed LaunchAgent, runtime, service-config drift, and duplicate-poller states.

## Best Possible Solution

Define a shared gateway diagnostics summary carrying service-loaded, runtime PID, RPC probe, service-config drift, port/listener, and recent channel-conflict facts, then render one final healthy/degraded/down verdict with deterministic next steps.

## Reproduction Assessment

Yes, at source level. Make the status RPC fail with `gateway closed` while the service remains loaded and `service.readRuntime()` reports running, and doctor can emit `Gateway not running.` before a later `Runtime: running` note.

## Solution Assessment

Yes at the direction level. A shared diagnostics contract and final verdict renderer is the maintainable solution; isolated message tweaks would leave split service, runtime, RPC, and channel-conflict facts scattered across doctor steps.
