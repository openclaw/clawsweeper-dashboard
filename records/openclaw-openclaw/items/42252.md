## Summary

Keep open: current main and the latest release still have source-reproducible split doctor output where RPC liveness can report the gateway down before the daemon pass reports a running supervisor runtime. Adjacent entrypoint-drift and Telegram 409 hints exist, but they do not provide the requested consolidated doctor/gateway diagnosis.

## What This Changes

The issue requests clearer doctor and gateway diagnostics for mixed LaunchAgent, runtime liveness, service-config drift, and duplicate poller states.

## Best Possible Solution

Define a shared gateway diagnostics summary carrying service-loaded, runtime PID, RPC probe, service-config drift, port/listener, and recent channel-conflict facts, then render one final healthy/degraded/down verdict with deterministic next steps.

## Reproduction Assessment

Yes, at source level. Make the status RPC fail with `gateway closed` while the service remains loaded and `service.readRuntime()` reports running, and doctor can emit `Gateway not running.` before a later `Runtime: running` note.

## Solution Assessment

Yes. A shared diagnostics contract and final verdict renderer is the maintainable path; isolated wording tweaks would leave the same split-brain facts scattered across doctor steps.
