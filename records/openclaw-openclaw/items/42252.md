## Summary

Keep open: current main still has split doctor diagnostics where an RPC liveness failure can render `Gateway not running.` before the later daemon check reports a running supervisor runtime. Entrypoint drift and duplicate Telegram poller handling are partially covered, but the requested consolidated verdict and deterministic recovery ordering are not implemented.

## What This Changes

Request clearer doctor/gateway diagnostics for mixed LaunchAgent service, runtime PID, RPC health, entrypoint drift, and duplicate poller states.

## Best Possible Solution

Define a shared gateway diagnostics summary that carries service-loaded, runtime PID, RPC probe, service-config drift, listener/port, and recent channel-conflict facts, then render one final healthy/degraded/down verdict with deterministic next steps.

## Reproduction Assessment

Yes, at source level. Make the status RPC fail with `gateway closed` while the service remains loaded and `service.readRuntime()` reports running, and doctor can emit `Gateway not running.` before a later `Runtime: running` note.

## Solution Assessment

Yes, the requested direction is best solved as a shared diagnostics contract and final verdict renderer. Isolated wording tweaks would leave service, runtime, RPC, and channel-conflict facts split across doctor steps.
