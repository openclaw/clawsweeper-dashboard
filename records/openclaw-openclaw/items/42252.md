## Summary

Current main and v2026.6.5 still keep doctor’s gateway-down notes separate from the richer service/runtime/RPC status model, so the central clarity problem remains; the issue should stay open for a maintainer to define a bounded final-state UX rather than close as stale.

## What This Changes

Add clearer consolidated diagnostics and recovery guidance for mixed macOS LaunchAgent, gateway runtime, RPC health, service-drift, and duplicate-instance states.

## Best Possible Solution

Have doctor reduce existing service registration, process liveness, and RPC probe facts into one final healthy/degraded/down summary with deterministic state-specific actions, while handling Telegram log inference and any stable JSON schema as separately approved scope.

## Reproduction Assessment

Yes at source level: current main has independent doctor health and daemon branches that can report the gateway down without a shared final-state renderer, while richer service/runtime/RPC facts are presented elsewhere. The exact incident environment was not recreated, so this is source-reproducible rather than a live reproduction.

## Solution Assessment

No as one bundled implementation: consolidating existing diagnostic facts is the narrow maintainable direction, but Telegram log heuristics and a new machine-readable doctor contract require separate ownership and product decisions.
