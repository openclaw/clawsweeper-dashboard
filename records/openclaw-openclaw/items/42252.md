## Summary

Keep open: current main partially covers service-config drift and Telegram duplicate-poller hints, but doctor still reports gateway RPC liveness and service runtime through separate paths, so mixed `Gateway not running.` plus `Runtime: running` output remains source-reproducible and needs a maintainer UX decision for the final verdict shape.

## What This Changes

Request to make `openclaw doctor` and gateway diagnostics clearer for mixed LaunchAgent/runtime states with a consolidated health verdict, conflict-aware hints, deterministic next steps, and optional JSON summary.

## Best Possible Solution

Define a shared gateway diagnostics summary that carries service-loaded, runtime PID, RPC probe, service-config drift, port/listener, and recent channel-conflict facts through doctor, then render one final healthy/degraded/down verdict with deterministic next steps.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make the gateway status RPC fail with `gateway closed` while the service remains loaded and `service.readRuntime()` reports running, and doctor can emit `Gateway not running.` before a later `Runtime: running` service note.

## Solution Assessment

Yes, keeping this open for maintainer UX direction is the right path. The maintainable fix is a shared diagnostics contract and final verdict, not more isolated message tweaks for each mixed-state symptom.
