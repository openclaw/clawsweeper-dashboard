## Summary

Keep open. Current main has useful adjacent diagnostics, but doctor still emits gateway health, service config, daemon runtime, and channel hints as separate notes without the requested consolidated final verdict, probable causes, or ordered remediation.

## What This Changes

Request clearer doctor/gateway diagnostics for mixed LaunchAgent, runtime, RPC, service-drift, and duplicate-poller states.

## Best Possible Solution

Add a doctor-owned gateway diagnosis collector that records service loaded state, runtime PID/status, RPC result, port ownership, service config drift, extra services, recent Telegram 409 evidence, and recent gateway errors, then renders one final verdict with probable causes and ordered remediation.

## Reproduction Assessment

Yes. Source inspection shows `checkGatewayHealth` can print `Gateway not running.` on RPC failure, then the later daemon phase can print a running runtime summary from supervisor state.

## Solution Assessment

No. The current implementation is useful but partial; the maintainable fix is a final doctor aggregation step rather than adding more independent notes across health, daemon, service, and Telegram paths.
