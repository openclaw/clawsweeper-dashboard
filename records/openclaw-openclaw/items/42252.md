## Summary

Keep open. Current main has useful adjacent diagnostics for service drift, runtime state, port conflicts, and Telegram 409 conflicts, but it still does not provide the requested doctor-owned consolidated gateway verdict or final probable-cause/remediation summary for mixed LaunchAgent/runtime states.

## What This Changes

The issue requests clearer `openclaw doctor` and gateway diagnostics for mixed LaunchAgent/runtime states, including a final health verdict, conflict-aware hints, service drift classification, ordered next steps, and optional machine-readable diagnosis output.

## Best Possible Solution

Add a doctor-owned gateway diagnosis collector that records service loaded state, runtime PID/status, RPC probe result, port ownership, service config drift, extra gateway-like services, recent Telegram getUpdates 409 evidence, and recent gateway errors, then renders one final healthy/degraded/down verdict with probable causes and ordered remediation steps.

## Reproduction Assessment

Yes. The central mixed-message path is reproducible from current source: `checkGatewayHealth` can emit `Gateway not running.` on RPC failure, then `maybeRepairGatewayDaemon` can later emit a runtime summary when the service is loaded/running.

## Solution Assessment

No. The current implementation is useful but partial; the narrow maintainable fix is a final doctor gateway diagnosis aggregation step rather than more independent notes scattered across health, service, daemon, and Telegram paths.
