## Summary

Keep open: current main still has a source-backed Slack send ambiguity. Slack sends can return receipts when Slack returns `ts`, but the Slack adapter does not implement `reconcileUnknownSend`, and the generic tool-result middleware can produce the exact `status:error`/`middlewareError:true` result shape reported without a receipt.

## What This Changes

The issue reports ambiguous Slack `message(action="send")` results that can leave the agent without a receipt and cause duplicate visible retries.

## Best Possible Solution

Define one Slack message-tool outcome contract: confirmed sends return receipts, definite pre-accept failures return hard errors, and post-accept unknowns surface a retry-unsafe unknown state with reconciliation or idempotency support.

## Reproduction Assessment

No, not a high-confidence live reproduction. Source inspection shows the reported result shape and the missing Slack reconciliation capability, but I did not execute a real Slack send path in this read-only review.

## Solution Assessment

Unclear. The requested direction is valid, but the safest fix needs maintainer agreement on whether to add Slack `reconcileUnknownSend`, a generic retry-unsafe tool result state, idempotency metadata, or a combination of those.
