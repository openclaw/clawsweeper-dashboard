## Summary

Keep open: current main still has a source-backed Slack send ambiguity. Slack can return receipts when `chat.postMessage` yields a timestamp, but Slack does not implement unknown-send reconciliation and the generic tool-result middleware can emit the reported `status:error`/`middlewareError:true` result without `isError:true`.

## What This Changes

The issue asks OpenClaw to make Slack message-send outcomes retry-safe by distinguishing confirmed delivery, definite failure, and unknown delivery instead of returning an ambiguous middleware error without a receipt.

## Best Possible Solution

Define one Slack message-tool outcome contract: confirmed sends return receipts, definite pre-accept failures return hard errors, and post-accept unknowns surface a retry-unsafe unknown state with reconciliation or idempotency support.

## Reproduction Assessment

No, not a high-confidence live reproduction. Source inspection shows the reported result shape and the missing Slack reconciliation capability, but I did not execute a real Slack send path.

## Solution Assessment

Unclear. The requested direction is valid, but the safest fix needs maintainer agreement on whether to add Slack `reconcileUnknownSend`, a generic retry-unsafe tool result state, idempotency metadata, or a combination of those.
