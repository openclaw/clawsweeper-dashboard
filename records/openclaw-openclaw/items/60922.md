## Summary

Keep open. Current main still lacks `fallbacksFromModels`, and this PR is useful source work, but the branch is not merge-ready because it does not preserve the current shared `timeoutMs` model-config contract and does not update the embedded runner’s fallback-configured gate for catalog-derived fallbacks.

## What This Changes

The PR adds an opt-in `agents.defaults.model.fallbacksFromModels` flag with schema/docs/generated baseline updates, catalog-derived fallback candidate construction, and tests for fallback derivation plus primary-model update preservation.

## Best Possible Solution

Keep the feature direction, but update the branch so catalog-derived fallbacks participate in the shared fallback-configured decision path, preserve the existing shared `timeoutMs` model schema/type, and add the required changelog entry.

## Reproduction Assessment

Yes. Configure `agents.defaults.model.fallbacksFromModels: true` with multiple `agents.defaults.models` entries and no explicit `fallbacks`, then trigger a retry-limit, prompt, or assistant failover path that consults `fallbackConfigured`.

## Solution Assessment

No. The opt-in catalog-derived fallback design is reasonable, but the current branch is not the narrow maintainable solution until it preserves existing model-config fields and updates the embedded failover gate alongside candidate construction.
