## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Wire catalog fallbacks into embedded failover gates:** `src/agents/model-fallback.ts:393-397`
  - body: This branch derives fallback candidates when `fallbacksFromModels` is enabled, but embedded runs still decide whether model fallback is available through `hasConfiguredModelFallbacks`, which only sees explicit fallback arrays. With this new flag and no explicit `fallbacks`, prompt/assistant/retry-limit failures can surface instead of throwing `FailoverError`, so the advertised catalog fallback is never attempted in that path. Please make the embedded gate use the same effective fallback availability as candidate construction and cover it with an embedded-run regression test.
  - confidence: 0.91
- **[P1] Preserve fallbacksFromModels in auth default writes:** `src/plugins/provider-auth-choice-helpers.ts:73-78`
  - body: Provider auth/default-model flows call `applyDefaultModel`, and at the PR head it rebuilds `agents.defaults.model` with only `fallbacks` plus `primary`. After this PR, a user can enable `fallbacksFromModels: true` and then lose that setting when adding or logging into a provider that sets a default model. Please preserve the existing object config there too, and add coverage for this writer.
  - confidence: 0.84
