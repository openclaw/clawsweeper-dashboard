## Summary

Keep open: this is the canonical PR for catalog-derived model fallbacks, but it is not merge-ready because the new derived fallback source is not wired into the embedded-runner fallback-availability gate and the external PR still lacks real behavior proof.

## What This Changes

The PR adds an opt-in `agents.defaults.model.fallbacksFromModels` config option with schema/docs/tests so `agents.defaults.models` can supply fallback candidates when explicit `fallbacks` are omitted.

## Best Possible Solution

Use one shared explicit-or-derived fallback resolver/availability contract across candidate construction and embedded-runner failover, then document the option only after focused seam coverage and redacted real-run proof.

## Reproduction Assessment

Yes, source-level: with `fallbacksFromModels: true` and no explicit `fallbacks`, the PR creates outer candidates but the embedded-runner `fallbackConfigured` gate still reads only explicit fallback arrays.

## Solution Assessment

No. Candidate derivation is a plausible piece, but the best fix is a shared derived-fallback contract used by both candidate construction and embedded-runner failover decisions.
