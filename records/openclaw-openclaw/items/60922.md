- **[P1] Keep fallback availability in sync with derived candidates:** `src/agents/model-fallback.ts:393-397`
  - body: This branch derives candidates when `fallbacksFromModels` is true, but embedded runs still decide whether `fallback_model` is allowed through `hasConfiguredModelFallbacks`, which reads only explicit fallback arrays. With no explicit `fallbacks`, retry-limit/prompt/assistant failures can still surface instead of trying the newly derived catalog fallback, so the new feature is incomplete until the same effective fallback source drives both paths.
  - confidence: 0.91
- **[P2] Handle provider wildcards before deriving catalog fallbacks:** `src/agents/model-fallback.ts:370-384`
  - body: `agents.defaults.models` supports entries like `openai/*` for dynamic provider catalogs, but this loop passes every raw key through model-ref parsing and then pushes the resulting key as a fallback. Enabling the flag on a documented wildcard catalog can therefore try model `*` at runtime instead of a concrete discovered model or skipping the wildcard, so wildcard entries need explicit handling and coverage.
  - confidence: 0.86
