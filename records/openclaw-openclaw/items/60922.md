## Summary

Keep open: current main does not implement catalog-derived fallbacks, and this PR is a plausible feature direction, but the branch is not mergeable as-is because it misses current-main config contracts, leaves a failover gate explicit-array-only, and lacks contributor-run real behavior proof.

## What This Changes

The branch adds an opt-in `agents.defaults.model.fallbacksFromModels` config/docs/schema path, derives fallback candidates from `agents.defaults.models`, and adds focused unit/schema/provider-model tests.

## Best Possible Solution

Land this feature only after a rebase preserves current config contracts, wires catalog-derived candidates into every failover gate that uses configured-fallback state, and includes focused tests plus redacted real runtime proof.

## Reproduction Assessment

Yes, by source inspection: current main only reads explicit `agents.defaults.model.fallbacks`, and the PR defects are visible in the branch diff against current main. I did not run tests because this was a read-only review.

## Solution Assessment

No. The explicit opt-in design is plausible, but this implementation needs rebase fixes for `timeoutMs`, model-ref normalization, and the shared fallback-configured gate before it is a maintainable solution.
