## What This Changes

The PR adds an opt-in `agents.defaults.model.fallbacksFromModels` config flag with runtime fallback-candidate derivation, schema/type/docs/generated updates, and unit coverage.

## Best Possible Solution

Keep the feature as an explicit config opt-in, implemented through the existing fallback resolver/gate and current schema/config-write contracts, then land after a clean rebase and real runtime proof.

## Reproduction Assessment

Yes for the blocking PR behavior by source inspection: PR head derives candidates in `resolveFallbackCandidates`, but the shared fallback gate remains explicit-array-only, and the extracted model schema omits current-main `timeoutMs`. I did not run tests because this was a read-only review.

## Solution Assessment

No. The feature direction is plausible, but this implementation needs to preserve existing schema/config-write contracts, wire catalog-derived fallbacks into the common failover gate, preserve current model-ref normalization on rebase, and add real behavior proof.
