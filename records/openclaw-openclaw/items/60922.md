## Summary

Keep open. The feature direction is useful, and the earlier `applyPrimaryModel` data-loss concern appears addressed, but the latest PR diff is still not merge-ready because catalog-derived fallbacks are not wired into the embedded failover enabled gate, the extracted shared model schema drops existing `timeoutMs` support, and the user-facing config feature lacks a changelog entry.

## What This Changes

The PR adds an opt-in `agents.defaults.model.fallbacksFromModels` config/schema/docs/test path that derives default model fallbacks from `agents.defaults.models` when explicit fallbacks are omitted.

## Best Possible Solution

Keep the feature, but make the catalog-derived fallback flag participate in all fallback-enabled decisions, retain the existing shared model config fields, and record the new user-facing option in the changelog.

## Reproduction Assessment

Yes. Configure `agents.defaults.model.fallbacksFromModels: true` with multiple `agents.defaults.models` entries and no explicit `fallbacks`, then trigger a retry-limit, prompt, or assistant failover path that consults `fallbackConfigured`.

## Solution Assessment

No. The PR is close, but the narrow maintainable fix must update the shared fallback-configured decision path and preserve the existing `timeoutMs` schema contract before merge.
