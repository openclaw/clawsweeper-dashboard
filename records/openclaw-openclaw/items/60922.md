- **[P1] Sync derived fallbacks with embedded failover gates:** `src/agents/model-fallback.ts:393-397`
  - body: This branch derives candidates when `fallbacksFromModels` is true, but embedded runs still compute `fallbackConfigured` through `hasConfiguredModelFallbacks`, which only counts explicit fallback arrays. `resolveRunFailoverDecision` gates retry-limit, prompt, and assistant `fallback_model` decisions on that stale boolean, so an opted-in user can still surface the primary error instead of switching models. Please centralize the effective fallback source or update the embedded fallback gate and tests with the derived catalog path.
  - confidence: 0.87
- **[P2] Handle wildcard catalog rows before deriving fallbacks:** `src/agents/model-fallback.ts:370-385`
  - body: Current main supports `agents.defaults.models` entries such as `provider/*` for dynamic catalog visibility, but this derivation loop treats every catalog key as a concrete fallback model. After rebasing, a wildcard row could be attempted as `provider/*` or bypass dynamic expansion, so skip wildcard visibility rows or expand them through the configured catalog before adding fallback candidates.
  - confidence: 0.74
