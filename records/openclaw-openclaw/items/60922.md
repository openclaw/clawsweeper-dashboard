## Summary

Keep open: current main still lacks the requested catalog-derived fallback feature, but the PR is not mergeable as-is because the branch drops the existing model `timeoutMs` schema field, leaves the embedded-runner fallback gate explicit-only, and still lacks real behavior proof.

## What This Changes

The PR adds an opt-in `agents.defaults.model.fallbacksFromModels` config flag, docs, schema/generated metadata, and tests to derive model fallbacks from `agents.defaults.models` when explicit fallbacks are omitted.

## Best Possible Solution

Keep the feature direction, but update the branch so derived catalog fallbacks participate in the shared fallback-configured path, preserve the existing model timeout contract, refresh generated config artifacts, and add real runtime proof.

## Reproduction Assessment

Yes for the blocking PR behavior by source inspection: PR head derives candidates in `resolveFallbackCandidates`, but `hasConfiguredModelFallbacks` still only checks explicit arrays, and current main accepts `timeoutMs` that the PR schema omits. I did not run tests because this review is read-only.

## Solution Assessment

No. The requested direction is reasonable, but this implementation needs to preserve existing schema fields and wire the new derived fallback source into the common failover gate before it is the maintainable solution.
