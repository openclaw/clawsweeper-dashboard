## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep fallbackConfigured in sync with derived candidates:** `src/agents/model-fallback.ts:397`
  - body: This branch only teaches `resolveFallbackCandidates` about `fallbacksFromModels`, but embedded runs still compute `fallbackConfigured` through `hasConfiguredModelFallbacks`, which only sees explicit fallback arrays. With a config that has `fallbacksFromModels: true` and no `fallbacks`, prompt/assistant/retry-limit failures can still surface instead of throwing a model-fallback error for the outer loop to advance to the derived catalog candidate.
  - confidence: 0.91
- **[P1] Do not run provider wildcards as fallback models:** `src/agents/model-fallback.ts:371`
  - body: `agents.defaults.models` supports entries like `openai/*` for dynamic provider visibility, but this loop derives a fallback from every raw catalog key. That turns a supported wildcard entry into a runnable `openai/*` model candidate instead of skipping it or expanding it through the discovered catalog, so users who enable the new flag with existing wildcard catalogs can get invalid model attempts.
  - confidence: 0.87
