## Summary

Keep open. The PR targets a real cron duplicate-name gap, but the proposed service-level plain `Error` would currently bubble through `cron.add`/`cron.update` as `UNAVAILABLE` rather than a structured validation error for gateway, CLI, and agent callers.

## What This Changes

The PR adds a cron job name lookup, rejects duplicate names during cron add/update, covers enabled and disabled collision cases in service tests, and adds a changelog entry.

## Best Possible Solution

Land duplicate-name validation at the cron service boundary while ensuring gateway clients receive structured `INVALID_REQUEST` errors, with any `--replace` or duplicate-listing UX split into explicit follow-ups if desired.

## Reproduction Assessment

Yes. Current-main source inspection shows the duplicate add/edit path, and the linked issue gives a CLI reproduction; the PR-specific regression path is a gateway `cron.add` or `cron.update` collision that should return `INVALID_REQUEST`.

## Solution Assessment

No, not yet. Service-level validation is the right boundary, but the new rejection needs a typed or explicitly classified validation path so RPC callers do not see it as a transient gateway failure.
