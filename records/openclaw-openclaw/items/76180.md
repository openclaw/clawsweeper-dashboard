## Summary

Keep open: the PR addresses a real current-main cron duplicate-name gap, but it is not merge-ready because the new service errors would be reported through the Gateway RPC path as `UNAVAILABLE` instead of structured `INVALID_REQUEST` validation errors.

## What This Changes

The PR adds cron service duplicate-name checks for add/update, service-level regression tests, and a changelog entry for the `cron add`/`cron edit` collision fix.

## Best Possible Solution

Land service-level name uniqueness validation with Gateway/RPC error classification and focused gateway regression coverage; split `--replace` and duplicate-listing UX into separate follow-ups.

## Reproduction Assessment

Yes. Current-main source inspection shows `cron.add` and `cron.update` can persist two rows with the same normalized name, and the linked issue gives a CLI reproduction; the PR-specific regression is a Gateway `cron.add` or `cron.update` collision returning the wrong error code.

## Solution Assessment

No, not yet. Service-level validation is the right boundary, but duplicate-name conflicts need a typed or explicitly classified validation path so RPC callers receive `INVALID_REQUEST` rather than `UNAVAILABLE`.
