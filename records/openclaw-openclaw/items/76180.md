## Summary

Keep open. The PR uses the right cron-service seam and exact-head checks are green, but the proposed update collision check can miss an existing duplicate when the edited job is the first matching row in a legacy duplicate-name store.

## What This Changes

The branch adds cron job-name collision rejection to service add/update, service and Gateway regression tests, and an Unreleased changelog entry.

## Best Possible Solution

Keep the service-level duplicate-name guard, but make update scan all sibling jobs excluding the current id and add a legacy duplicate-order regression before merge.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original issue, and the PR finding is reproducible by seeding two jobs with the same name then updating the first row with that same name.

## Solution Assessment

No as currently written. The service seam and RangeError mapping are right, but using the first matching name is order-dependent; the update guard should search for any sibling collision excluding the target id.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Scan all sibling cron jobs on rename:** `src/cron/service/ops.ts:363-364`
  - body: `findJobByName` returns only the first row with the target name. In a legacy store that already has duplicate names, updating the first duplicate with `{ name: "foo" }` sees itself as the collision and allows the update even though a later sibling still has the same name, leaving this issue order-dependent. Search for any matching job whose id differs from the target id instead.
  - confidence: 0.9
