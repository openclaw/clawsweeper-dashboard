## Summary

Keep open: current main still has source-backed synchronous installed-index and manifest-registry rebuild paths, and the independent Linux reproduction makes this an active performance bug rather than cleanup noise; the fix shape overlaps broader metadata-snapshot work and needs plugin/Gateway owner judgment.

## What This Changes

The issue requests eliminating repeated synchronous installed-plugin index and manifest-registry reconstruction from hot Gateway and CLI RPC paths under concurrent load.

## Best Possible Solution

Route hot RPC, status, and model metadata reads through explicit current `PluginMetadataSnapshot` or `PluginLookUpTable` ownership, or a maintainer-approved invalidation-keyed fallback, without adding a broad wall-clock metadata cache that violates the plugin cache boundary.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. The issue supplies detailed reporter and independent runtime timings, and source inspection confirms current main still reaches the synchronous installed-index and manifest-registry fallback path.

## Solution Assessment

Unclear. The reporter's TTL cache has useful local proof, but the documented architecture points to explicit snapshot or lookup-table handoff unless plugin owners approve a bounded invalidation cache.
