## Summary

Keep open: the report is source-backed and lines up with current synchronous installed-plugin index fallback behavior, but I did not live-reproduce the concurrent RPC cascade on current main and prior maintainer comments reject the direct TTL-cache style as the default fix shape.

## What This Changes

The issue asks OpenClaw to stop `loadInstalledPluginIndex` from synchronously rebuilding plugin manifest/index metadata on hot concurrent Gateway RPC paths.

## Best Possible Solution

Keep hot Gateway/RPC metadata reads on compatible `PluginMetadataSnapshot`, `PluginLookUpTable`, or explicit manifest-registry ownership; only consider a fallback cache if it is fully keyed and invalidated by plugin inventory, config, workspace, and install-record inputs.

## Reproduction Assessment

No high-confidence current-main runtime reproduction was established. The issue body has detailed v2026.5.7 measurements and current source confirms the synchronous fallback path, but I did not prove the concurrent RPC cascade locally.

## Solution Assessment

No for a direct TTL cache as the default fix: prior maintainer comments and docs prefer explicit snapshot or registry handoff over hidden wall-clock metadata caching. The maintainable fix shape needs plugin/Gateway owner review.
