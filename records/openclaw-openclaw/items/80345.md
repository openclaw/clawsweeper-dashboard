## Summary

Keep open: the report is specific, has two live release repro data points, and current main still has source paths that synchronously reconstruct installed plugin index and manifest metadata rather than reusing a prepared snapshot in every hot fallback path.

## What This Changes

The issue asks OpenClaw to stop repeated synchronous `loadInstalledPluginIndex` and plugin manifest walks from blocking concurrent Gateway/RPC/status activity.

## Best Possible Solution

Keep hot Gateway/RPC/status metadata reads on a compatible current `PluginMetadataSnapshot`, `PluginLookUpTable`, or explicit invalidation-keyed registry product, avoiding repeated installed-index reconstruction without adding a broad wall-clock TTL cache by default.

## Reproduction Assessment

No live current-main runtime reproduction was established in this review. The issue is still high-confidence source-reproducible because current main keeps the synchronous installed-index rebuild path and the discussion includes two concrete v2026.5.7 runtime data points.

## Solution Assessment

Unclear. The local TTL cache mitigation has strong reported results, but the narrowest maintainable upstream fix likely needs plugin/Gateway owners to choose between explicit snapshot handoff, invalidation-keyed fallback reuse, or a deliberately bounded cache exception.
