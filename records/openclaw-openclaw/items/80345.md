## Summary

Keep open: the report is specific and matches current source risk around synchronous installed-plugin index reconstruction, but current main already has some snapshot fast paths and the remaining fix shape needs plugin-runtime ownership review.

## What This Changes

The issue reports that concurrent Gateway RPCs can synchronously rebuild installed plugin and manifest metadata through loadInstalledPluginIndex, causing multi-second event-loop stalls.

## Best Possible Solution

Keep hot Gateway/RPC metadata reads on explicit current PluginMetadataSnapshot or PluginLookUpTable ownership, and only add an invalidation-keyed fallback cache if the plugin owners decide it fits the documented cache boundary.

## Reproduction Assessment

No high-confidence current-main runtime reproduction was established. Source inspection confirms the synchronous fallback path, but I did not prove the reported concurrent RPC cascade on current main.

## Solution Assessment

Unclear. The reporter's TTL cache mitigation has strong local evidence, but the repository's documented design prefers explicit snapshot handoff or a bounded invalidation cache over wall-clock hidden metadata caching.
