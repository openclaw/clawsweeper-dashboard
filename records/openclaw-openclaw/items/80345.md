## Summary

Keep open: the report is specific, recently corroborated on a second Linux/status surface, and current main still has source-visible synchronous installed-index and metadata rebuild fallback paths. Current main has related mitigations, but not enough evidence that the broader concurrent-RPC cascade is fixed.

## What This Changes

The issue asks OpenClaw to stop synchronously rebuilding installed plugin manifest/index metadata on hot Gateway and status RPC paths under concurrent load.

## Best Possible Solution

Keep hot Gateway and status metadata reads on an explicit current PluginMetadataSnapshot, PluginLookUpTable, or scoped manifest registry, and only add coalescing/invalidation where every registry-affecting input is keyed or bypassed.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. Source inspection confirms the fallback rebuild path, and the issue now has two detailed runtime reports, but current-main wall-clock impact still needs measurement.

## Solution Assessment

Unclear. The reporter's TTL cache has useful local proof, but the maintainable OpenClaw direction is likely explicit snapshot/lookup-table reuse or fully keyed coalescing rather than a wall-clock cache on installed-index reconstruction.
