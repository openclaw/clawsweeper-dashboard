## Summary

Keep open: the patch is not merge-ready because it still maps the reported DeepSeek V4 short refs to invalid bare OpenRouter API ids, and its new allowlist would strip several current native `openrouter/*` model ids that OpenRouter publishes today.

## What This Changes

The PR adds an OpenRouter native-id allowlist and tests so single-segment `openrouter/...` model refs are stripped unless they match that allowlist.

## Best Possible Solution

Use the OpenRouter-owned normalizer to expand the known DeepSeek V4 short aliases to `deepseek/...` upstream slugs while preserving all current native `openrouter/*` ids, then require redacted real OpenRouter proof before merge.

## Reproduction Assessment

Yes. Source inspection plus the public OpenRouter `/models` endpoint show this PR maps `openrouter/deepseek-v4-flash` to a bare id that OpenRouter does not publish, and it strips current native `openrouter/*` ids that are published.

## Solution Assessment

No. The right layer is the OpenRouter provider normalizer, but the implementation should expand known short aliases to namespaced upstream slugs and preserve native OpenRouter routes instead of stripping every non-whitelisted single-segment ref.
