## Summary

Keep open: this PR targets a real OpenRouter provider-routing bug, but the patch currently normalizes the short DeepSeek refs to bare `deepseek-v4-*` IDs instead of the upstream OpenRouter `deepseek/deepseek-v4-*` IDs, so it likely does not fix the reported `model_not_found` failure.

## What This Changes

The PR changes OpenRouter model ID normalization so short openrouter-prefixed refs are stripped before API calls while known native OpenRouter identifiers keep their prefix, and adds focused tests.

## Best Possible Solution

Keep the OpenRouter-owned normalizer fix, but map the catalog-listed short DeepSeek refs to the actual upstream `deepseek/deepseek-v4-*` IDs and add proof for that behavior before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main keeps `openrouter/deepseek-v4-flash` as the API-facing ID, and the linked issue plus OpenRouter model list show the accepted ID is the namespaced DeepSeek slug.

## Solution Assessment

Is this the best way to solve the issue? No, not in its current form: the OpenRouter normalizer is the right layer, but the branch strips to a bare ID instead of the upstream namespaced ID.
