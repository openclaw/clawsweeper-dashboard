## Summary

Keep open: current main still lacks the requested `agents.defaults.models.<model>.contextTokens` field, and the resolver still treats `agents.defaults.contextTokens` as a global cap while the supported per-model cap lives under provider catalog config. Adding this exact field would create a second config source and overlaps with the broader token-budget policy discussion in https://github.com/openclaw/openclaw/issues/71136, so this needs maintainer product direction rather than cleanup closure.

## What This Changes

The issue requests a per-model `contextTokens` override under `agents.defaults.models` so model switches can use different context budgets without repeatedly changing the global cap.

## Best Possible Solution

Maintainers should choose one canonical owner for per-model effective context caps: either steer users to provider-catalog `contextTokens`, or add the agent-model key with explicit precedence, schema/help/docs, and focused resolver tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that the requested `agents.defaults.models.<model>.contextTokens` key is absent while the existing per-model cap is supported through provider catalog config.

## Solution Assessment

Unclear. The requested field is plausible, but current code and docs already use `models.providers.*.models[].contextTokens`, so maintainers should decide whether the agent-level duplicate is the right long-term config surface.
