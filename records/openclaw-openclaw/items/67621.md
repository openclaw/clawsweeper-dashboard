## Summary

Keep open: current main still applies a raw all-caps `$VAR` regex to Python/Node script contents without string/docstring/comment context in active script-preflight modes, so the false-positive class remains even though gateway `security=full` plus `ask=off` now skips that layer.

## What This Changes

The issue asks for exec script preflight to ignore benign `$VAR`-looking tokens inside Python literals while preserving rejection of real shell-variable leakage.

## Best Possible Solution

Implement a context-aware script preflight detector that ignores benign Python/Node string, docstring, comment, and help-text occurrences while still rejecting raw `$DM_JSON`-style leakage in executable source positions.

## Reproduction Assessment

Yes, from source inspection for active preflight modes: a direct `python bad.py` target whose file contains uppercase `$SIM` in a string literal reaches the raw regex and throws before literal context is considered. I did not run a temp-file reproduction because this review is read-only.

## Solution Assessment

Unclear which implementation variant is best without maintainer policy choice. A narrow lexical/AST or execution-context-aware detector is better than the raw regex, but the security boundary should be chosen deliberately before changing the guard.
