## Summary

Keep open: current main and the latest release bypass script preflight for gateway `security=full` plus `ask=off`, but the active approval-mode script preflight still raw-scans Python/Node source with an all-caps `$VAR` regex and has no string/docstring/comment context check. The issue discussion already identified that narrower remaining bug, and the related fix PRs are closed unmerged rather than a canonical open replacement.

## What This Changes

The issue requests that exec script preflight stop flagging benign `$VAR`-looking tokens inside Python string literals while preserving real shell-variable leakage detection.

## Best Possible Solution

Implement a context-aware script preflight detector that ignores benign Python/Node string, docstring, comment, and help-text occurrences while still rejecting raw `$DM_JSON`-style leakage in executable source positions.

## Reproduction Assessment

Yes, from source inspection for active preflight modes: a direct `python bad.py` target whose file contains uppercase `$SIM` in a string literal reaches the raw regex and throws before literal context is considered. I did not run a temp-file reproduction because this review is read-only, and the exact gateway full/off mode is now intentionally skipped.

## Solution Assessment

Unclear. The requested direction is correct, but the safest implementation boundary should be chosen deliberately because this is a security-sensitive exec guard; a lexer/AST or execution-context-aware detector with negative coverage is safer than a broad regex skip.
