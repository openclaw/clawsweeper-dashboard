## Summary

Keep open: current main still uses a raw all-caps `$VAR` regex over Python/Node script contents with no string, docstring, comment, or argparse-help context check. The exact gateway `security=full` + `ask=off` path is now documented and coded to skip this preflight, but the underlying scanner remains false-positive prone wherever script preflight still runs, and the two linked fix PRs are closed unmerged.

## What This Changes

The issue asks for exec script preflight to stop treating benign `$IDENTIFIER` text inside Python literals, docstrings, help text, and dollar-formatted output as shell variable injection.

## Best Possible Solution

End state should be a context-aware script preflight detector that ignores benign Python/Node string, docstring, comment, and help-text occurrences while still rejecting raw shell-variable leakage in executable source positions, backed by positive `$SIM` literal tests and negative `$DM_JSON` leakage tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for modes where script preflight runs: a direct `python script.py` target whose file contains an uppercase `$SIM` token in a string literal will match the raw regex and throw before literal context is considered.

## Solution Assessment

Unclear which implementation variant is best without maintainer policy choice. A narrow lexer/AST or execution-context-aware detector is better than the current raw regex, but the security boundary should be chosen deliberately before changing the guard.
