## Summary

Keep open: current main still raw-scans Python/Node script contents for all-caps `$VAR` tokens without string/comment context in approval-mode preflight. The exact no-approval `security=full` plus `ask=off` path is now skipped, but the remaining approval-mode false positive is still valid and the linked fix PRs are closed unmerged.

## What This Changes

The issue asks exec script preflight to stop treating `$IDENTIFIER` text inside Python strings, docstrings, and help text as shell variable injection.

## Best Possible Solution

Implement a language-aware or execution-context-aware script preflight detector that ignores benign string/comment/help-text occurrences while preserving rejection of executable-source `$VAR` leakage.

## Reproduction Assessment

Yes for current approval-mode preflight by source inspection: direct `python bad.py` reaches the raw scanner, and uppercase `$SIM` inside a string literal would still throw. The original no-approval `security=full` plus `ask=off` path is now intentionally bypassed, so the affected surface is narrower than first reported.

## Solution Assessment

Unclear. The requested direction is right, but the safest fix should be lexer/AST or execution-context aware with negative tests, not a broad regex skip that weakens the guard.
