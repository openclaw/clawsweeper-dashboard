## Summary

Keep open: current main and the latest release still raw-scan script contents for all-caps `$VAR` tokens without string/comment context. The original no-approval `security=full` plus `ask=off` path is now skipped, but the approval-mode false positive remains and the related fix PRs are closed unmerged.

## What This Changes

The issue asks for exec script preflight to stop treating benign `$VAR` tokens inside Python string literals, docstrings, comments, and help text as shell variable injection.

## Best Possible Solution

Implement a language-aware or execution-context-aware preflight detector that ignores benign string/comment/help-text occurrences while preserving rejection of executable-source `$VAR` leakage.

## Reproduction Assessment

Yes, by source inspection for current approval-mode preflight: direct `python bad.py` reaches the raw scanner, and uppercase `$SIM` inside a string literal would still throw. The original no-approval full/off path is now bypassed, so the current repro surface is narrower than the issue body first described.

## Solution Assessment

Unclear. The requested direction is right, but the safest fix should be lexer/AST or execution-context aware with negative tests, not a broad regex skip that weakens the guard.
