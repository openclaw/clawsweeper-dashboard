## Summary

Keep open: current main and v2026.5.18 still raw-scan identified scripts for all-caps `$VAR` tokens without string/comment context outside the gateway `security=full` plus `ask=off` skip. The prior fix PRs are closed unmerged, so the remaining approval-mode behavior still needs security-aware maintainer follow-up.

## What This Changes

Fix exec script preflight so `$ALL_CAPS` tokens inside Python string literals, docstrings, and help text are not treated as shell variable injection.

## Best Possible Solution

Implement context-aware script preflight that ignores benign Python strings, comments, and help text while still rejecting real executable-source `$VAR` leakage, with focused tests for both cases.

## Reproduction Assessment

Yes, by source inspection: direct Python script commands produce a script target, approval-mode exec still calls `validateScriptFileForShellBleed`, and the raw uppercase `$VAR` regex would match `$SIM` inside a Python string. I did not run a live exec because this review is read-only.

## Solution Assessment

Unclear without a current patch. A lexer/AST or execution-context-aware change with regression tests is safer than a broad skip that might weaken the existing `payload = $DM_JSON` rejection.
