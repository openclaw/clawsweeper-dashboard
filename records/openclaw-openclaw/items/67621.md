## Summary

Keep open: current main and `v2026.5.18` still apply the all-caps `$VAR` regex to raw script contents in approval-mode preflight, so uppercase tokens such as `$SIM` inside Python strings can still be rejected. The original gateway `security=full` plus `ask=off` path is now bypassed, but the remaining false-positive class is valid and the related fix PRs are closed unmerged.

## What This Changes

The issue asks exec script preflight to stop treating benign Python string-literal dollar tokens such as `$SIM` as shell variable injection.

## Best Possible Solution

Implement a language-aware or execution-context-aware Python preflight path that ignores benign string/comment/help-text occurrences while preserving rejection of executable-source `$VAR` leakage and the documented `security=full` plus `ask=off` bypass.

## Reproduction Assessment

Yes, by source inspection: direct Python script commands still route to `validateScriptFileForShellBleed`, which applies the `$[A-Z_]` regex to raw file text. A focused regex probe also confirms that `$SIM` inside a quoted help string matches the detector.

## Solution Assessment

Unclear. The requested direction is correct, but the safest fix should be lexer/AST or execution-context aware with both benign-literal acceptance tests and executable-source rejection tests, not a broad heuristic skip.
