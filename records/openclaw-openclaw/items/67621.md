## Summary

Keep open: current main still applies the all-caps `$VAR` detector to raw Python/Node script text without string/comment context, so the confirmed `$SIM` string-literal false positive remains. The two linked fix PRs were both closed unmerged, and current tests still cover rejection/YOLO skip cases rather than benign `$SIM` docstrings or argparse help text.

## What This Changes

The issue requests that exec script preflight stop rejecting `$IDENTIFIER` tokens that appear inside Python string literals, docstrings, help text, or benign dollar-formatted output under `security=full`.

## Best Possible Solution

Implement context-aware script preflight in `src/agents/bash-tools.exec.ts`, ideally by lexing/parsing enough Python and Node syntax to skip strings, docstrings, comments, and help text while still rejecting raw `$DM_JSON`-style code. Add focused regression coverage in `src/agents/bash-tools.exec.script-preflight.test.ts` for `$SIM` in Python docstrings/argparse help text and any intended dollar-format output, plus negative tests for real shell-variable leakage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a direct `python3 script.py` command is selected for script preflight, and any uppercase `$SIM` token in the file content is matched before literal context is considered.

## Solution Assessment

No, the current raw-regex scan is not the best solution for the reported false positive. The maintainable direction is a narrowly context-aware detector or execution-context narrowing that keeps the security guard while allowing benign literal text.
