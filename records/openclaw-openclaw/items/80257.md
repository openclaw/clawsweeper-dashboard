## Summary

Keep open: this is a plausible data-loss fix and the author appears to have addressed the earlier snapshot-overwrite/type/lint feedback, but the PR still lacks real packaged upgrade proof and the snapshot can target the wrong config path when the update doctor uses service-derived config env overrides.

## What This Changes

The PR adds a pre-update config snapshot, update-aware doctor write/strip guards, a `plugins.installs` strip exemption, and regression tests for backup rotation, config analysis, and doctor writes.

## Best Possible Solution

Snapshot the effective config path the update doctor will mutate, keep the update-time strip and write guards, and validate a packaged upgrade preserves both `openclaw.json` and `credentials/`.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available. Source inspection supports the current-main update doctor/config-loss path and the remaining custom config-path gap, but there is no real packaged upgrade artifact in the PR.

## Solution Assessment

No as written. The defense-in-depth direction is appropriate, but the snapshot must use the same env-sensitive config path as the update doctor and the PR needs real upgrade proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Snapshot the doctor config path:** `src/cli/update-cli/update-command.ts:983`
  - body: The package-update doctor env can be built from captured service env, including `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`, but this snapshot uses the updater process's module-load `CONFIG_PATH`. For a managed gateway with a service-specific config path, the doctor can still write that service config while `.pre-update` is created for the wrong or missing parent config, leaving the endangered file without the promised snapshot. Resolve the effective post-install doctor env first and snapshot the config path for that env.
