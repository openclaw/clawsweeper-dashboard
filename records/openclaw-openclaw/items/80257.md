## Summary

Keep open. The PR targets a real update-time config-loss bug, but the current head still has blocking test-type, snapshot-path, and recovery-snapshot issues, and the external contribution still lacks real after-fix upgrade proof.

## What This Changes

Adds update-time config preservation guards: a pre-update config snapshot, update-time unknown-key strip bypass, `plugins.installs` strip protection, size-drop guard changes, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the exact doctor target path, uses bounded or refreshed recovery snapshots, keeps the strip and size-drop safeguards, fixes test types, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level, but not as a live packaged macOS upgrade. Current main can run update-triggered doctor with write support, strict schema does not accept legacy `plugins.installs`, and doctor unknown-key stripping can remove it before persistence.

## Solution Assessment

No as submitted. The strip and size-drop guard direction is maintainable, but the snapshot must follow the resolved doctor config path, the recovery snapshot needs bounded refresh semantics, the test fixture must typecheck, and real upgrade proof is still required.

Full review comments:

- **[P2] Stub the full doctor prompter:** `src/flows/doctor-health-contributions.test.ts:190`
  - body: The new test context only supplies `shouldRepair`, but `DoctorHealthFlowContext` expects a full `DoctorPrompter`; the latest `check-test-types` run fails with TS2352 on this fixture. Add the missing prompter methods or build the context through a typed helper.
  - confidence: 0.95
- **[P2] Snapshot the resolved doctor config path:** `src/cli/update-cli/update-command.ts:1032`
  - body: The snapshot uses module-scope `CONFIG_PATH`, but package-update doctor can run with a managed-service `OPENCLAW_CONFIG_PATH` from `resolvePostInstallDoctorEnv`. When those differ, the recovery copy protects the caller config while doctor mutates the service config; derive the snapshot path from the same resolved doctor env.
  - confidence: 0.9
- **[P2] Refresh or bound pre-update snapshots:** `src/config/backup-rotation.ts:134-138`
  - body: Using `wx` and treating `EEXIST` as success makes the first `.pre-update` snapshot permanent. Later updates then skip capturing the actual pre-update state for that run, leaving stale recovery data; use per-update replacement, bounded timestamped snapshots, or an explicit retention policy.
  - confidence: 0.9
