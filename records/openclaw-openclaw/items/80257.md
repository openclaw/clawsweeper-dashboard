## Summary

Keep open. The PR targets a real source-reproducible updater config-loss bug, but the latest diff still snapshots the wrong config path for managed-service update doctor runs, keeps stale `.pre-update` snapshots indefinitely, and lacks required real upgrade proof.

## What This Changes

The PR adds update-time config-preservation guards, a pre-update snapshot helper, `plugins.installs` protection, update doctor size-drop protection, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the same config path the update doctor will write, refreshes or rotates recovery snapshots safely, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered doctor with write support, strip schema-unknown legacy keys such as `plugins.installs`, and bypass the size-drop guard. I did not run a live packaged upgrade in this read-only review.

## Solution Assessment

No as submitted. The direction is maintainable, but the snapshot must follow the doctor config path and must not become a stale first-write-wins file across future updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path doctor will mutate:** `src/cli/update-cli/update-command.ts:983`
  - body: The snapshot uses module-level `CONFIG_PATH`, but the package update doctor child receives `resolvePostInstallDoctorEnv(...)` and can write a managed-service `OPENCLAW_CONFIG_PATH` or `OPENCLAW_STATE_DIR`. In that path the backup is written beside the caller config while doctor mutates the service config, so derive the snapshot path from the same resolved doctor env.
  - confidence: 0.9
- **[P2] Refresh recovery snapshots between update runs:** `src/config/backup-rotation.ts:137-138`
  - body: `createPreUpdateConfigSnapshot` writes with `wx` and returns on `EEXIST`, so an older `.pre-update` file blocks all future update snapshots. A later upgrade may have no recovery copy of its actual pre-update config and can retain stale secret-bearing data indefinitely.
  - confidence: 0.9
