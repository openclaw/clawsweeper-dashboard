## Summary

Keep open: the PR targets a real source-level config-loss path, but the submitted patch still has two correctness blockers and only test-output proof rather than redacted real upgrade proof.

## What This Changes

The branch adds update-time config preservation guards, a `.pre-update` config snapshot helper, `plugins.installs` strip protection, size-drop guard changes, and focused tests.

## Best Possible Solution

Land a corrected update-preservation fix that protects the same resolved config path doctor will mutate, keeps legacy install records migratable, preserves size-drop protection during updates, and uses bounded or refreshed recovery snapshots.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor with write support, strip schema-unknown `plugins.installs` before migration, and bypass the size-drop rejection. I did not establish a live packaged macOS/npm upgrade reproduction in this read-only review.

## Solution Assessment

No as submitted. The root-cause direction is sound, but the snapshot must follow the resolved doctor config path, the recovery snapshot needs bounded refresh or retention semantics, and real upgrade proof is still missing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Snapshot the doctor config path actually used:** `src/cli/update-cli/update-command.ts:1032`
  - body: The new snapshot calls pass module-scope `CONFIG_PATH`, but package-update doctor builds its child env through `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` with the managed service profile path. In that case recovery snapshots the caller config while doctor mutates the service config; derive the snapshot path from the same resolved env used for doctor.
  - confidence: 0.91
- **[P2] Refresh recovery snapshots for later updates:** `src/config/backup-rotation.ts:134`
  - body: `createPreUpdateConfigSnapshot` writes `${configPath}.pre-update` with `flag: "wx"` and returns on `EEXIST`, so the first update snapshot blocks all future pre-update snapshots. A later update would keep stale credential-bearing data and lack recovery for its actual starting state; replace it per update or use bounded timestamped retention.
  - confidence: 0.88

## Security Review
