## Summary

Keep open. The PR targets a valid source-reproducible config-loss bug, but the latest head still snapshots `CONFIG_PATH` instead of the resolved post-install doctor config path, makes the first `.pre-update` snapshot permanent, and lacks real after-fix upgrade proof.

## What This Changes

The PR adds update-time config preservation changes across doctor unknown-key stripping, plugin install record handling, pre-update config snapshots, size-drop write safety, and focused tests.

## Best Possible Solution

Land a corrected update-preservation fix that snapshots the exact config path doctor will mutate, preserves migratable legacy install records, keeps size-drop protection during update runs, uses bounded or refreshed recovery snapshots, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor with write support, treat legacy `plugins.installs` as schema-unknown before migration, and bypass the size-drop rejection. I did not establish live packaged macOS/npm upgrade proof or the credential-directory deletion path.

## Solution Assessment

No as submitted. The root-cause direction is sound, but the PR must snapshot the same resolved config path doctor writes, use bounded or refreshed snapshot retention, and add real upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Snapshot the resolved doctor config path:** `src/cli/update-cli/update-command.ts:1031-1033`
  - body: Package-update doctor runs with `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` from the managed service profile. This snapshot uses module `CONFIG_PATH`, so it can snapshot the caller config while doctor mutates the service config; derive the snapshot path from the same resolved env passed to doctor.
  - confidence: 0.91
- **[P2] Refresh or rotate pre-update snapshots:** `src/config/backup-rotation.ts:128-135`
  - body: The fixed `.pre-update` path is created with `wx` and returns on `EEXIST`, so the first update permanently wins. Later updates cannot record their own pre-update state and stale credential-bearing config remains on disk; replace it per update or use bounded timestamped retention.
  - confidence: 0.88
