## Summary

Keep open: the reported config-loss path is source-reproducible and the PR is directionally useful, but the latest diff still snapshots the wrong config path in managed-service update flows, keeps stale `.pre-update` files first-write-wins across future updates, and lacks real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards, a `.pre-update` config snapshot helper, `plugins.installs` protection from unknown-key stripping, update doctor size-drop protection, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the active doctor config path for each update run, refreshes or rotates recovery snapshots safely, keeps the strip and size-drop guards, and includes redacted real upgrade proof preserving config and credentials.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered doctor with write support, strip schema-unknown keys such as legacy `plugins.installs`, and bypass the size-drop guard. I did not run a live upgrade in this read-only review.

## Solution Assessment

No as submitted. The main fix direction is maintainable, but the snapshot must follow the same config path the doctor will mutate and must not become a stale first-write-wins file across future updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path doctor will mutate:** `src/cli/update-cli/update-command.ts:980`
  - body: The snapshot always uses module-level `CONFIG_PATH`, but `resolvePostInstallDoctorEnv` can pass managed-service `OPENCLAW_CONFIG_PATH` or `OPENCLAW_STATE_DIR` into the child doctor. In that update path, the backup can be written beside the caller config while doctor mutates the service config, so derive the snapshot path from the same resolved doctor env.
  - confidence: 0.9
- **[P2] Refresh recovery snapshots between update runs:** `src/config/backup-rotation.ts:137-142`
  - body: `createPreUpdateConfigSnapshot` writes with `wx` and returns on `EEXIST`, so a `.pre-update` file from an older update blocks all future snapshots. A later upgrade may have no recovery copy of its actual pre-update config and can retain stale secret-bearing data indefinitely.
  - confidence: 0.9
