## Summary

Keep open. The underlying config-loss bug is source-reproducible on current main, but this PR still has blocking correctness issues, a failing type-check, a security-sensitive snapshot concern, and no redacted real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards, `.pre-update` config snapshots, `plugins.installs` strip protection, and tests for update doctor config writes.

## Best Possible Solution

Land a corrected preservation fix that snapshots the exact config file doctor will mutate, preserves migratable legacy install records through update-time doctor writes, uses bounded or refreshed recovery snapshots, and includes real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor with write support, strip schema-unknown `plugins.installs` before write-time migration, and bypass the size-drop rejection. I did not replay a packaged macOS/npm upgrade.

## Solution Assessment

No as submitted. The root-cause direction is sound, but the snapshot path, snapshot retention semantics, type-check failure, and real upgrade proof gap need to be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Snapshot the resolved doctor config path:** `src/cli/update-cli/update-command.ts:1032`
  - body: The snapshot uses imported `CONFIG_PATH`, but package-update doctor builds its child env through `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` from the managed service profile. In that case recovery snapshots the caller config while doctor mutates the service config; derive the snapshot path from the same resolved env used for doctor.
  - confidence: 0.91
- **[P2] Refresh or rotate pre-update snapshots:** `src/config/backup-rotation.ts:134-138`
  - body: The fixed `.pre-update` path is created with `wx` and `EEXIST` returns, so the first update permanently wins. Later updates can neither record their own pre-update state nor drop old credential-bearing config data; replace it per update or use bounded timestamped retention.
  - confidence: 0.88
- **[P2] Stub the full doctor prompter:** `src/flows/doctor-health-contributions.test.ts:190`
  - body: `check-test-types` fails on the new test context because the casted object only provides `prompter.shouldRepair`, but `DoctorHealthFlowContext` requires the full `DoctorPrompter` surface. Add the missing prompter methods or use an intentional narrower test helper so the test types compile.
  - confidence: 0.95

## Security Review

Status: needs_attention
