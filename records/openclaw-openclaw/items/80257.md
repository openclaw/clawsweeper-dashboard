## Summary

Keep open. The bug is source-reproducible and the branch targets the right surfaces, but the PR still has blocking issues: it snapshots the wrong path for managed-service doctor runs, keeps stale first-ever snapshots, fails `check-test-types`, and lacks real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards, `.pre-update` snapshots, a `plugins.installs` strip exemption, size-drop guard changes, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the same config path doctor will mutate, uses bounded or refreshed recovery snapshots, passes type checks, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can strip schema-unknown legacy config before write-time migration during update-triggered doctor writes, and the size-drop guard can be bypassed. I did not replay a packaged macOS/npm upgrade.

## Solution Assessment

No as submitted. The root-cause direction is sound, but the implementation still misses the managed-service config path, uses stale snapshot retention, fails the test type lane, and lacks real upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Snapshot the resolved doctor config path:** `src/cli/update-cli/update-command.ts:1031-1033`
  - body: Package-update doctor receives env from `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` from the managed service profile. This call snapshots module `CONFIG_PATH` before that env is applied, so it can back up the caller config while doctor mutates the service config; derive the snapshot path from the same resolved env.
  - confidence: 0.91
- **[P2] Refresh or bound pre-update snapshots:** `src/config/backup-rotation.ts:131-137`
  - body: The fixed `.pre-update` path is written with `wx` and `EEXIST` is ignored, so the first ever snapshot permanently wins. Later updates cannot capture their own pre-update state, and stale credential-bearing config can remain indefinitely; replace it per update or use bounded timestamped retention.
  - confidence: 0.88
- **[P2] Provide a full DoctorPrompter test stub:** `src/flows/doctor-health-contributions.test.ts:190-194`
  - body: `check-test-types` fails because this helper casts a context whose `prompter` only has `shouldRepair`, but `DoctorPrompter` requires the confirm/repair methods too. Use a complete test stub or an explicit narrow helper so the type lane passes.
  - confidence: 0.97
