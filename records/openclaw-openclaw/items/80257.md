## Summary

Keep open: this is a plausible fix for a real current-main config-loss bug, but the submitted PR still has two blocking code issues and no real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards: a pre-update config snapshot, update-time unknown-key strip bypass, `plugins.installs` strip protection, a size-drop guard change, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the exact doctor target path, uses bounded or refreshed recovery snapshots, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor with write support, strip strict-schema unknown keys such as `plugins.installs` before migration, and bypass the size-drop rejection. I did not replay a live packaged macOS npm upgrade or prove the credentials-directory deletion path.

## Solution Assessment

No as submitted. The direction is maintainable, but the snapshot must follow the same resolved config path doctor will mutate, recovery snapshots need bounded refresh or retention semantics, and real upgrade proof is still missing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Snapshot the doctor config path:** `src/cli/update-cli/update-command.ts:1032`
  - body: The snapshot is taken beside module-level `CONFIG_PATH`, but package-update doctor builds its child env with `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` from the managed service. In that case the recovery copy protects the caller config while doctor mutates the service config; derive the snapshot path from the same resolved doctor env before launching doctor.
  - confidence: 0.9
- **[P2] Refresh update recovery snapshots:** `src/config/backup-rotation.ts:132-136`
  - body: Writing `${configPath}.pre-update` with `wx` and treating `EEXIST` as success makes the first snapshot permanent, so every later update skips capturing the actual pre-update state. Use bounded timestamped snapshots, per-update replacement, or another retention policy that preserves one update run without leaving stale recovery data indefinitely.
  - confidence: 0.88
