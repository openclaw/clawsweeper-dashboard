## Summary

Keep open: the PR targets a real config-loss bug linked to https://github.com/openclaw/openclaw/issues/80077, but the current branch still has blocking correctness issues, a red type-check, a low-severity snapshot-retention security concern, and no real upgrade proof.

## What This Changes

The PR adds pre-update config snapshots, update-time unknown-key strip guards, size-drop guard changes, `plugins.installs` strip protection, and focused backup/config/doctor tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the same resolved config path doctor writes, uses bounded or refreshed recovery snapshots, keeps the strip and size-drop guards, fixes type-check CI, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor with write support, strip schema-unknown keys such as `plugins.installs` before migration, and bypass the size-drop rejection. I did not establish a live packaged macOS/npm upgrade reproduction in this read-only review.

## Solution Assessment

No as submitted. The direction is maintainable, but the snapshot must follow the same resolved config path doctor will mutate, the recovery snapshot needs bounded refresh or retention semantics, and the PR must fix the failing type-check.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Snapshot the resolved doctor config path:** `src/cli/update-cli/update-command.ts:1032`
  - body: The new snapshot uses module-scope `CONFIG_PATH`, but this package-update path runs doctor with `resolvePostInstallDoctorEnv`, which can override `OPENCLAW_CONFIG_PATH` from the managed service profile. When those differ, recovery protects the caller config while doctor mutates the service config; derive the snapshot path from the same resolved env used for doctor.
  - confidence: 0.92
- **[P2] Refresh recovery snapshots between updates:** `src/config/backup-rotation.ts:134`
  - body: `flag: "wx"` plus returning on `EEXIST` makes the first `.pre-update` copy permanent. Future updates skip their actual pre-update state and can retain old credential-bearing config, so use replacement, timestamped bounded snapshots, or another retention policy.
  - confidence: 0.9
- **[P2] Fix the doctor health test context cast:** `src/flows/doctor-health-contributions.test.ts:180`
  - body: The live `check-test-types` job fails with TS2352 on this new test context cast because the object does not sufficiently overlap `DoctorHealthFlowContext`. Convert through `unknown` or build a complete typed context so the required type gate can pass.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: The diff adds a persistent config snapshot that can retain stale credential-bearing data and block future recovery snapshots.
