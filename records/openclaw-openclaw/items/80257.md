## Summary

Keep this PR open, but it is not merge-ready: it targets a real update-time config data-loss path, yet the branch still snapshots the wrong config in managed-service update flows, leaves an indefinite stale snapshot, and lacks real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards: a pre-update snapshot, update-time unknown-key strip skip, `plugins.installs` protection, size-drop protection, and focused tests.

## Best Possible Solution

Land a corrected update-time preservation fix that snapshots the same config path doctor will write, refreshes or rotates recovery snapshots per update run without stale secret retention, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered doctor with write support, strip schema-unknown config keys including legacy `plugins.installs`, and bypass the size-drop guard on that write path. I did not run a live upgrade in this read-only review.

## Solution Assessment

No as submitted. The direction is appropriate, but the recovery snapshot must follow the active doctor config path and must not become a permanent first-write-wins stale copy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the active doctor config path:** `src/cli/update-cli/update-command.ts:983`
  - body: This snapshots module-level `CONFIG_PATH` before building the child doctor env. When managed-service `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` is supplied, `resolvePostInstallDoctorEnv` passes a different config path to the child doctor, so the recovery snapshot can be written beside the caller config while doctor mutates the service config; derive the snapshot path from the same env used for doctor.
  - confidence: 0.88
- **[P2] Refresh or rotate pre-update snapshots:** `src/config/backup-rotation.ts:134-138`
  - body: The helper writes with `wx` and treats `EEXIST` as a no-op, so the first `.pre-update` file survives forever. Later updates will neither capture the current pre-update state nor remove stale secret-bearing config; replace, rotate, or timestamp snapshots per update while preserving idempotence within one update run.
  - confidence: 0.9
