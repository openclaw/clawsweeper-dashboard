## Summary

Keep open. The PR targets a real source-reproducible config-loss bug, but it still needs real upgrade proof and has blocking correctness/security issues in the snapshot implementation plus a failing test-types check.

## What This Changes

Adds update-time config preservation guards, including pre-update snapshots, update-time unknown-key strip bypass, `plugins.installs` protection, size-drop protection during update doctor writes, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the exact doctor config path, bounds or refreshes recovery snapshots per update, keeps the strip and size-drop guards, fixes test typing, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered `doctor --fix` with write support, strict schema omits legacy `plugins.installs`, and the write path allows size drops. I did not replay a packaged macOS npm upgrade or prove the reported credentials-directory deletion.

## Solution Assessment

No as submitted. The direction is right, but the PR needs to snapshot the same config path doctor writes, avoid permanent stale snapshots, fix the test-types failure, and provide real upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Snapshot the config path passed to doctor:** `src/cli/update-cli/update-command.ts:983`
  - body: This snapshots module-level `CONFIG_PATH`, but `resolvePostInstallDoctorEnv` can pass a managed-service `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` to the child doctor. In that case recovery is written beside the caller config while doctor mutates the service config, so derive the snapshot path from the same resolved env used for doctor.
  - confidence: 0.9
- **[P2] Refresh pre-update snapshots between update runs:** `src/config/backup-rotation.ts:134-138`
  - body: The helper uses `wx` and treats `EEXIST` as a no-op, so the first `.pre-update` file wins forever. A later update will neither capture the current pre-update state nor retire stale secret-bearing config, leaving recovery ineffective after the first update run.
  - confidence: 0.9
- **[P2] Fix the doctor test context type:** `src/flows/doctor-health-contributions.test.ts:194`
  - body: The latest `check-test-types` job fails with TS2352 on this returned mock context because it does not sufficiently overlap `DoctorHealthFlowContext`. Build a complete typed context or cast through `unknown` if the partial mock is intentional so the required type gate can pass.
  - confidence: 0.92
