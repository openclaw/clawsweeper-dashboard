## Summary

Keep open. This is a focused fix for a real update-time config data-loss path, but the latest PR still snapshots the wrong file for managed-service doctor runs, leaves first-write-wins recovery snapshots stale forever, and lacks real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards, a `.pre-update` config snapshot, `plugins.installs` protection from unknown-key stripping, size-drop protection during update doctor runs, and tests.

## Best Possible Solution

Land a corrected update preservation fix that snapshots the exact config file doctor will write, refreshes recovery snapshots per update, preserves legacy config through migration, and proves a real upgrade keeps config and credential references.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered `doctor --fix` with write support, strip schema-unknown legacy keys such as `plugins.installs`, and bypass the size-drop guard. I did not replay a packaged macOS npm upgrade in this read-only review.

## Solution Assessment

No as submitted. The direction is right, but the snapshot must derive from the same resolved env/config path as doctor, stale snapshots should be refreshed or rotated per update, and real upgrade proof is still required.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path doctor will write:** `src/cli/update-cli/update-command.ts:980`
  - body: This snapshots module-level `CONFIG_PATH`, but the child doctor env is resolved through `resolvePostInstallDoctorEnv` and can carry a managed-service `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH`. When those differ, the recovery snapshot is written beside the caller config while doctor mutates the service config, leaving the changed file without a pre-update snapshot.
  - confidence: 0.9
- **[P2] Refresh pre-update snapshots for each update:** `src/config/backup-rotation.ts:134-138`
  - body: Using `flag: "wx"` and treating `EEXIST` as a no-op makes the first `.pre-update` file win forever. A later update cannot capture the current pre-update state and can leave stale secret-bearing config outside the managed backup rotation, so replace, rotate, or version the snapshot per update run.
  - confidence: 0.9
