## Summary

Keep this PR open. It targets a valid config-loss path, but the patch still has blocking recovery gaps and only supplies test-based proof, not a real upgrade run showing user config survives.

## What This Changes

The PR adds update-time safeguards to preserve OpenClaw config during doctor fixes, including a persistent pre-update snapshot, update-aware unknown-key stripping, a stricter size-drop guard, and focused tests.

## Best Possible Solution

Land a corrected fix that snapshots the active config path for each update run, preserves current config and credentials through a real upgrade, and keeps the doctor/schema guards covered by regression tests.

## Reproduction Assessment

Yes, by source inspection. Current main can run update-triggered doctor with write privileges, the strict schema treats plugins.installs as unknown, and the doctor unknown-key step runs before the config write path that migrates install records.

## Solution Assessment

No as submitted. The guard direction is appropriate, but the snapshot must be fresh per update run and must use the same config path as the doctor process; the PR also needs real upgrade proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope first-write-wins snapshots to one update run:** `src/config/backup-rotation.ts:134-137`
  - body: The snapshot is always written to `${configPath}.pre-update` and EEXIST is treated as success. After one update creates that file, later updates silently keep a stale snapshot, so a future destructive upgrade would not have a backup of the user's current config. Keep first-write-wins within a single update run, but create or rotate a fresh per-update snapshot at the start of each new update.
  - confidence: 0.9
- **[P2] Snapshot the same config path the child doctor mutates:** `src/cli/update-cli/update-command.ts:981-982`
  - body: This snapshot uses module-level `CONFIG_PATH`, but the child doctor below is launched with `resolvePostInstallDoctorEnv(...)`, which can replace `OPENCLAW_STATE_DIR` and `OPENCLAW_CONFIG_PATH` from the managed service env. In that case doctor can mutate the service config while `.pre-update` was created beside a different config, leaving the affected file without the new recovery snapshot. Resolve the snapshot path from the same env passed to `runUpdateStep`.
  - confidence: 0.86
