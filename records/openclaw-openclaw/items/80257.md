## Summary

Keep open: this PR targets a plausible active config data-loss bug, but the current head is not merge-ready because real upgrade-run proof is still missing, `check-test-types` still fails, and the new fixed `.pre-update` snapshot can become stale after the first update.

## What This Changes

The PR adds config-preservation safeguards around update-triggered doctor writes, including a pre-update snapshot, update-time unknown-key strip suppression, protected plugin install records, and focused tests.

## Best Possible Solution

Land a CI-clean patch that captures a current per-update recovery snapshot, preserves user config through a real redacted upgrade run, and separately tracks credentials-directory deletion if that remains unexplained.

## Reproduction Assessment

No. Source inspection and the linked user report make the update-triggered doctor stripping path plausible, but I did not validate a packaged v2026.5.6-to-head upgrade with populated config and credentials.

## Solution Assessment

No. The chosen update, doctor, and config-write surfaces are reasonable, but the current patch needs a non-stale snapshot strategy, a type-correct test helper, and real upgrade-run proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Avoid reusing stale pre-update snapshots:** `src/config/backup-rotation.ts:137-138`
  - body: Because `.pre-update` is a fixed path and `EEXIST` is ignored, the first successful update leaves a file that blocks every later update from capturing the current pre-upgrade config. After a user changes config, a future bad update would recover to an old snapshot instead of the latest state; use a per-update/versioned snapshot or a guard that only prevents duplicate writes within the same update run.
  - confidence: 0.88
- **[P2] Make the write-config test context type-correct:** `src/flows/doctor-health-contributions.test.ts:180-194`
  - body: `check-test-types` still fails on this new helper with TS2352. The object is forced into the private doctor context while fields such as `cfg.gateway.mode` and the partial `prompter` are not typed as that context, so CI remains red; build a complete typed stub/helper instead of relying on the cast.
  - confidence: 0.96
