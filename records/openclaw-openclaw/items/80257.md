## Summary

Keep open: the PR targets a real update-time config loss path, but the current patch still snapshots the wrong config in managed-service update flows, keeps stale .pre-update snapshots across future updates, and lacks required real upgrade proof.

## What This Changes

The PR adds update-time config preservation safeguards around doctor writes, unknown-key stripping, pre-update snapshots, and regression tests.

## Best Possible Solution

Land a corrected update-time preservation fix that snapshots the active doctor config path, refreshes or rotates snapshots per update while preserving the first snapshot within a run, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, from source inspection rather than a live upgrade run. Current main can run update-triggered doctor with write privileges, treat plugins.installs as an unknown strict-schema key, and allow a size-drop write on that path.

## Solution Assessment

No as submitted. The direction is appropriate, but the snapshot must use the same config path as doctor, avoid stale cross-update recovery files, and be backed by real upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Snapshot the active doctor config path:** `src/cli/update-cli/update-command.ts:980-981`
  - body: The child doctor env can be built from managed-service OPENCLAW_STATE_DIR or OPENCLAW_CONFIG_PATH, but this call snapshots module-level CONFIG_PATH from the parent. In managed-service updates, doctor can mutate the service config while the recovery snapshot is written beside a different config; derive the snapshot path from the env passed to doctor, and keep the restart doctor path consistent.
  - confidence: 0.87
- **[P2] Refresh pre-update snapshots across update runs:** `src/config/backup-rotation.ts:134-136`
  - body: The snapshot path is fixed and EEXIST is treated as success, so after one update creates .pre-update, later update runs keep an old config copy. A future destructive update would not have a snapshot of the user's current pre-update config, and old secret-bearing config can persist indefinitely; rotate, timestamp, or replace the snapshot at update start while keeping idempotence within one run.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds a persistent config snapshot that can retain stale secret-bearing config indefinitely unless refreshed or rotated per update.

Concerns:

- **[low] Avoid indefinitely stale config snapshots:** `src/config/backup-rotation.ts:134`
  - body: OpenClaw config can contain channel/provider secrets or secret references. Because .pre-update is first-write-wins and outside normal backup rotation, an old snapshot can persist after later updates and credential changes; replacing or rotating it per update would reduce exposure while preserving recovery.
  - confidence: 0.78
