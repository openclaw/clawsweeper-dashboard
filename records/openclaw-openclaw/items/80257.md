## Summary

Keep open: this PR addresses a real update-time config-loss bug, but the latest diff still snapshots the wrong config path for managed-service doctor runs, leaves stale pre-update snapshots outside bounded cleanup, and lacks real upgrade proof.

## What This Changes

Adds update-time config preservation guards: pre-update config snapshots, update-time unknown-key strip bypass, plugins.installs strip protection, a size-drop guard change, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the exact doctor target path, uses bounded or refreshed recovery snapshots, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor --fix with write support, strip schema-unknown keys such as plugins.installs before migration, and bypass the size-drop rejection. I did not run a live v2026.5.6 to v2026.5.7 upgrade in this read-only review.

## Solution Assessment

No as submitted. The fix direction is maintainable, but the snapshot must follow the same resolved config path doctor will mutate and the recovery snapshot needs bounded refresh or retention semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path passed to doctor:** `src/cli/update-cli/update-command.ts:1029`
  - body: The snapshot is taken beside module-level CONFIG_PATH, but package-update doctor runs with env from resolvePostInstallDoctorEnv, which can replace OPENCLAW_CONFIG_PATH and OPENCLAW_STATE_DIR from the managed service. In that case the recovery copy is for the caller config while doctor mutates service config, so derive the snapshot path from the same resolved doctor env before launching it.
  - confidence: 0.9
- **[P2] Refresh update recovery snapshots:** `src/config/backup-rotation.ts:132`
  - body: The helper writes `${configPath}.pre-update` with `wx` and treats EEXIST as success, so once any old snapshot exists every later update skips capturing the actual pre-update config. That leaves stale secret-bearing config outside backup cleanup and removes recovery for future upgrades; rotate, replace, or timestamp it with bounded retention.
  - confidence: 0.9
