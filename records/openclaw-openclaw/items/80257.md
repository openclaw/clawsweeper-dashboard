## Summary

Keep open: the PR addresses a real source-reproducible update-time config-loss bug, but the current patch still has blocking correctness/security issues around snapshot targeting and lifetime, and it lacks required real upgrade proof.

## What This Changes

The branch adds update-time config preservation defenses: a `.pre-update` snapshot, unknown-key strip guards, a protected `plugins.installs` key, update size-drop blocking, and regression tests.

## Best Possible Solution

Land a corrected update-time preservation fix that snapshots the same config path doctor writes, refreshes or bounds pre-update snapshots per upgrade, preserves credentials, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level rather than live: current main can run update-triggered doctor with write support, strip strict-schema-unknown config keys including legacy `plugins.installs`, and bypass the size-drop guard on that write path. I did not replay a packaged macOS npm upgrade or prove the credentials-directory deletion path.

## Solution Assessment

No as submitted. The preservation direction is appropriate, but the snapshot must target the same config path doctor will write and must be refreshed, rotated, or otherwise bounded across separate update runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Snapshot the config path doctor will write:** `src/cli/update-cli/update-command.ts:983`
  - body: The child doctor env is built below from managed-service `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH`, but this snapshots module-level `CONFIG_PATH` from the updater parent. Managed-service updates can still mutate one config while the recovery snapshot is written beside another, so derive the snapshot path from the same env passed to doctor.
  - confidence: 0.88
- **[P2] Refresh the pre-update snapshot per upgrade:** `src/config/backup-rotation.ts:134-138`
  - body: Using `wx` plus an `EEXIST` no-op makes `.pre-update` first-write-wins for the lifetime of the file. A later update would keep an old recovery copy instead of the user's current pre-update config, and old sensitive config can persist indefinitely; replace, rotate, or timestamp snapshots per update while preserving idempotence within one run.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds a persistent config snapshot that can retain stale sensitive config indefinitely unless refreshed, rotated, or otherwise bounded.

Concerns:

- **[low] Avoid indefinitely stale config snapshots:** `src/config/backup-rotation.ts:134`
  - body: OpenClaw config can contain channel/provider-sensitive data and credential references. Because `.pre-update` is first-write-wins and outside normal backup rotation, an old snapshot can survive later updates and credential changes while failing to capture the current pre-update state.
  - confidence: 0.8
