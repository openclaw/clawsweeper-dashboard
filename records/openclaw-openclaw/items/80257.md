## Summary

Keep open: this PR targets a source-reproducible update-time config-loss bug, but the latest diff still snapshots the wrong config path for managed-service doctor runs, can leave stale recovery snapshots across future updates, and lacks real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards, a `.pre-update` snapshot helper, `plugins.installs` strip protection, update doctor size-drop protection, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the active doctor config path for each update run, refreshes or safely rotates recovery snapshots, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered doctor with write support, strip schema-unknown keys such as legacy `plugins.installs`, and bypass the size-drop guard. I did not establish a live upgrade reproduction in this read-only review.

## Solution Assessment

No as submitted. The main fix direction is maintainable, but the snapshot must follow the same config path the doctor will mutate and must not become a stale first-write-wins file across future updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path doctor will mutate:** `src/cli/update-cli/update-command.ts:1029-1030`
  - body: The snapshot uses module-level `CONFIG_PATH`, but the child doctor env can be built from managed-service `OPENCLAW_CONFIG_PATH` or `OPENCLAW_STATE_DIR`. In that update path, the backup can be written beside the caller config while doctor mutates the service config, so derive the snapshot path from the same resolved env.
  - confidence: 0.9
- **[P2] Refresh recovery snapshots between update runs:** `src/config/backup-rotation.ts:132-136`
  - body: `createPreUpdateConfigSnapshot` writes with `wx` and ignores `EEXIST`, so a `.pre-update` file from an older update blocks all future snapshots. A later upgrade may have no recovery copy of its actual pre-update config and can retain stale secret-bearing data indefinitely.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The new persistent snapshot can preserve stale secret-bearing config indefinitely outside the managed backup rotation.

  - body: OpenClaw config can contain credential references and occasionally sensitive values; a first-write-wins `.pre-update` file outside rotation can survive credential changes and later updates while no longer representing the current pre-update state.
