## Summary

Keep open. The PR targets a real source-reproducible update-time config-loss bug, but the current patch still has blocking correctness, CI, security/retention, and missing real-behavior-proof issues before it can merge.

## What This Changes

Adds update-time config preservation guards: a pre-update config snapshot, update-time unknown-key strip bypass, plugins.installs strip protection, update doctor size-drop protection, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the resolved doctor target path, uses bounded or refreshed recovery snapshots, keeps the strip and size-drop protections, fixes typecheck, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor with write support, strip schema-unknown keys such as plugins.installs before migration, and bypass the size-drop rejection. I did not replay a live packaged macOS npm upgrade.

## Solution Assessment

No as submitted. The fix direction is right, but the snapshot must follow the resolved doctor config path, snapshot retention must be bounded or refreshed, the test type failure must be fixed, and real upgrade proof is still required.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Snapshot the resolved doctor config path:** `src/cli/update-cli/update-command.ts:1029-1030`
  - body: The snapshot uses module-scope CONFIG_PATH, but package-update doctor builds its child env with resolvePostInstallDoctorEnv, which can replace OPENCLAW_CONFIG_PATH from the managed service. In that case the recovery copy protects the caller config while doctor mutates the service config; derive the snapshot path from the same resolved doctor env before launching doctor.
  - confidence: 0.91
- **[P2] Refresh update recovery snapshots:** `src/config/backup-rotation.ts:132`
  - body: Writing `${configPath}.pre-update` with `wx` and treating EEXIST as success makes the first snapshot permanent, so every later update skips capturing the actual pre-update state. Use bounded timestamped snapshots, per-update replacement, or another retention policy that preserves one update run without leaving stale recovery data indefinitely.
  - confidence: 0.9
- **[P2] Fix the doctor test context cast:** `src/flows/doctor-health-contributions.test.ts:180`
  - body: The latest check-test-types run fails here because the test casts a partial object to DoctorHealthFlowContext while its prompter stub only has shouldRepair. Add the required DoctorPrompter methods or cast through unknown intentionally so the PR is type-clean.
  - confidence: 0.88
