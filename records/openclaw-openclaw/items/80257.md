## Summary

This PR should stay open: the fix targets a real update-time config-loss bug, but the current head still has correctness blockers, a failing test-type check, a low-severity config snapshot retention concern, and only test/CI proof rather than real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards, a pre-update config snapshot helper, unknown-key strip protections, update-time size-drop protection, and focused tests for the upgrade doctor path.

## Best Possible Solution

Land a corrected preservation fix that snapshots the exact resolved doctor config path, uses bounded refreshed recovery snapshots, fixes the test type failure, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor with config-write support, strip schema-unknown keys such as plugins.installs before write-time migration, and bypass the size-drop rejection. I did not run a live v2026.5.6 to v2026.5.7+ upgrade in this read-only review.

## Solution Assessment

No as submitted. The strip and size-drop guard direction is good, but the snapshot must follow the resolved doctor target path, the recovery file needs bounded refresh semantics, and the new test stub must typecheck.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the resolved doctor config path:** `src/cli/update-cli/update-command.ts:1031-1034`
  - body: The snapshot is taken beside module-level CONFIG_PATH, but package-update doctor runs with the env from resolvePostInstallDoctorEnv, which can replace OPENCLAW_CONFIG_PATH from the managed service. In that case the recovery copy is for the caller config while doctor mutates service config, so derive the snapshot path from the same resolved doctor env before launching doctor.
  - confidence: 0.9
- **[P2] Refresh or rotate pre-update snapshots:** `src/config/backup-rotation.ts:131-138`
  - body: The helper writes `${configPath}.pre-update` with `wx` and treats EEXIST as success, so once any old snapshot exists every later update skips capturing the actual pre-update config. That leaves stale credential-bearing config outside backup cleanup and removes recovery for future upgrades; rotate, replace, or timestamp it with bounded retention.
  - confidence: 0.9
- **[P2] Use a complete DoctorPrompter test stub:** `src/flows/doctor-health-contributions.test.ts:190-194`
  - body: The check-test-types job fails with TS2352 because the new test context casts a partial DoctorHealthFlowContext while `prompter` only defines `shouldRepair`. Fill the required DoctorPrompter methods or use an existing test helper so `pnpm check:test-types` passes.
  - confidence: 0.95
