## Summary

Keep open. This PR addresses a real source-reproducible update-time config data-loss path, but the latest head still snapshots the wrong possible config path, leaves permanent first-write-wins snapshots outside rotation, and lacks real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards: a `.pre-update` snapshot, update-time unknown-key strip bypass, `plugins.installs` protection, size-drop guard changes, and focused tests.

## Best Possible Solution

Land a corrected update-time preservation fix that snapshots the same config path doctor will write, refreshes or rotates recovery snapshots per update run, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered doctor with write support, strip schema-unknown config keys including legacy `plugins.installs`, and bypass the size-drop guard on that write path. I did not replay a packaged macOS npm upgrade in this read-only review.

## Solution Assessment

No as submitted. The overall direction is appropriate, but the snapshot must follow the active doctor config path and must not become a permanent stale secret-bearing copy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path doctor will mutate:** `src/cli/update-cli/update-command.ts:983`
  - body: This snapshots module-level `CONFIG_PATH` before constructing the child doctor env. `resolvePostInstallDoctorEnv` can pass managed-service `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH`, so the snapshot can protect a different file while doctor mutates the service config; derive the snapshot path from the same resolved env used for doctor.
  - confidence: 0.88
- **[P2] Refresh pre-update snapshots per update run:** `src/config/backup-rotation.ts:134-138`
  - body: The helper writes with `wx` and treats `EEXIST` as a no-op, so the first `.pre-update` file survives forever. Later updates will neither capture the current pre-update state nor remove stale secret-bearing config; replace, rotate, or timestamp snapshots while preserving idempotence within one update run.
  - confidence: 0.9
