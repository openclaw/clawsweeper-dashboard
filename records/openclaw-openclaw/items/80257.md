## Summary

Keep open: the PR is aimed at a real update-time config-loss bug, but the latest head still snapshots the wrong config path in managed-service updates, keeps a stale first-write-wins recovery snapshot forever, and lacks after-fix real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards, `.pre-update` config snapshots, `plugins.installs` strip protection, size-drop guard changes, and focused unit tests.

## Best Possible Solution

Land a corrected update-preservation fix that snapshots the exact config path doctor will mutate, preserves migratable legacy install records, keeps size-drop protection during update runs, and uses bounded or refreshed recovery snapshots.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor with write support, strip schema-unknown `plugins.installs` before write-time migration, and bypass the size-drop rejection. I did not establish live packaged macOS/npm upgrade proof in this read-only review.

## Solution Assessment

No as submitted. The root-cause direction is sound, but the snapshot must follow the resolved doctor config path, the recovery snapshot needs bounded refresh or retention semantics, and real upgrade proof is still missing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Snapshot the resolved doctor config path:** `src/cli/update-cli/update-command.ts:1031-1033`
  - body: The snapshot uses imported `CONFIG_PATH`, but package-update doctor builds its child env through `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` from the managed service profile. In that case recovery snapshots the caller config while doctor mutates the service config; derive the snapshot path from the same resolved env used for doctor.
  - confidence: 0.91
- **[P2] Refresh or rotate pre-update snapshots:** `src/config/backup-rotation.ts:128-135`
  - body: The fixed `.pre-update` path is created with `wx` and `EEXIST` returns, so the first update permanently wins. Later updates can neither record their own pre-update state nor drop old config data; replace it per update or use bounded timestamped retention.
  - confidence: 0.88
