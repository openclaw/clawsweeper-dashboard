## Summary

Keep open. The PR targets a real source-reproducible update-time config loss path, but the latest head still snapshots the updater parent's `CONFIG_PATH`, keeps first-write-wins `.pre-update` snapshots indefinitely, and lacks real upgrade proof.

## What This Changes

The PR adds update-time config preservation defenses: `.pre-update` snapshots, update-aware unknown-key stripping, a `plugins.installs` strip exception, size-drop blocking during update doctor runs, and focused tests.

## Best Possible Solution

Land a corrected update-time preservation fix that derives the snapshot path from the doctor environment, bounds or refreshes recovery snapshots, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level. Current main can run update-triggered `doctor --fix` with config-write support, treat legacy `plugins.installs` as schema-unknown, and persist the resulting candidate with the size-drop guard disabled; I did not replay a live packaged macOS upgrade.

## Solution Assessment

No as submitted. The direction is appropriate, but the snapshot must use the same config path as the doctor process and the recovery snapshot needs per-update freshness or bounded rotation while staying idempotent within one update run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Snapshot the active doctor config path:** `src/cli/update-cli/update-command.ts:983`
  - body: The snapshot uses module-level `CONFIG_PATH`, but the child doctor env is built below from managed-service `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH`. Managed-service updates can still mutate one config while the recovery snapshot is written beside another, so derive the snapshot path from the same env passed to doctor and apply the same fix to the in-process restart doctor path.
  - confidence: 0.88
- **[P2] Refresh snapshots between update runs:** `src/config/backup-rotation.ts:134`
  - body: Using `wx` plus an `EEXIST` no-op makes `.pre-update` first-write-wins for the lifetime of the file. A later update would keep an old recovery copy instead of the current pre-update config, and stale secret-bearing config can persist indefinitely; replace, rotate, or timestamp snapshots per update while preserving idempotence within one run.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR adds a persistent `.pre-update` config snapshot that can retain stale secret-bearing config indefinitely and miss later update states.
