## Summary

Keep open. The PR targets a real update-time config-loss bug, but it still has blocking correctness issues around which config file is snapshotted and how long the recovery snapshot lives, and it lacks required real upgrade proof.

## What This Changes

Adds update-time config preservation guards and tests: a pre-update config snapshot, update-time unknown-key strip bypass, plugins.installs strip protection, and size-drop blocking during update-triggered doctor writes.

## Best Possible Solution

Land a corrected preservation fix that snapshots the exact doctor target config path, bounds or refreshes pre-update recovery snapshots, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can strip legacy plugins.installs during update-triggered doctor repair before the write-time migration and can allow a large config size drop. I did not establish a live packaged macOS npm upgrade reproduction.

## Solution Assessment

No as submitted. The direction is maintainable, but the snapshot must follow the same resolved config path that doctor will write, and recovery snapshot retention needs bounded refresh semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Snapshot the same config path doctor will write:** `src/cli/update-cli/update-command.ts:1032`
  - body: The snapshot uses module-scope CONFIG_PATH, but package-update doctor builds its child env with resolvePostInstallDoctorEnv, which can replace OPENCLAW_CONFIG_PATH from the managed service. In that case the recovery copy protects the caller config while doctor mutates the service config; derive the snapshot path from the resolved doctor env before launching doctor.
  - confidence: 0.91
- **[P2] Bound or refresh pre-update snapshots:** `src/config/backup-rotation.ts:134-138`
  - body: Writing `${configPath}.pre-update` with `wx` and treating EEXIST as success makes the first snapshot permanent, so every later update skips capturing its actual pre-update state. Use bounded timestamped snapshots, per-update replacement, or another retention policy that preserves one update run without leaving stale recovery data indefinitely.
  - confidence: 0.88

## Security Review

Status: needs_attention
