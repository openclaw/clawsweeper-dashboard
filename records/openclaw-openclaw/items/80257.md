## Summary

Keep open. This is a plausible focused fix for the linked config-loss bug, but the latest PR head still snapshots a fixed parent-process config path instead of the active doctor config path, keeps the first pre-update snapshot across future updates, and has no real upgrade proof beyond tests and rebase comments.

## What This Changes

The PR adds update-time config preservation safeguards around doctor writes, unknown-key stripping, pre-update snapshots, and related tests.

## Best Possible Solution

Land a corrected update-time preservation fix that snapshots the active config once per update run, preserves legacy config through doctor writes, and is backed by redacted real upgrade proof.

## Reproduction Assessment

Yes, from source inspection rather than a live upgrade run. Current main can run update-triggered doctor with write privileges, treat plugins.installs as an unknown strict-schema key, and allow a size-drop write on that path.

## Solution Assessment

No as submitted. The direction is appropriate, but the snapshot needs to refresh per update and use the same config path passed to doctor, and the PR still needs real upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Refresh snapshots for each update run:** `src/config/backup-rotation.ts:134`
  - body: The snapshot path is fixed and written with flag "wx", with EEXIST treated as success. After one update creates this file, later updates keep an old config copy, so a future destructive upgrade would not have a backup of the user's current pre-update config and can retain stale secrets indefinitely. Keep idempotence within one run, but replace, rotate, or timestamp the snapshot when a new update starts.
  - confidence: 0.9
- **[P2] Snapshot the active doctor config path:** `src/cli/update-cli/update-command.ts:983`
  - body: This snapshots module-level CONFIG_PATH, but the child doctor env is built from service overrides such as OPENCLAW_STATE_DIR and OPENCLAW_CONFIG_PATH. In managed-service updates, doctor can mutate the service config while the snapshot is written beside a different parent-process config; derive the snapshot path from the env passed to doctor and keep the restart doctor path consistent.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff adds a persistent config snapshot that can retain stale secret-bearing config indefinitely unless it is refreshed or rotated per update.

Concerns:

- **[low] Avoid indefinitely stale config snapshots:** `src/config/backup-rotation.ts:134`
  - body: OpenClaw config can contain channel/provider secrets or secret references. Because .pre-update is first-write-wins and outside normal backup rotation, an old snapshot can persist after later updates and credential changes; replacing or rotating it per update would reduce exposure while preserving recovery.
  - confidence: 0.78
