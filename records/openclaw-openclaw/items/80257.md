## Summary

Keep open: this is a plausible data-loss fix, but the current PR still has blocking correctness issues, a failing type check, a low-severity snapshot retention security concern, and no after-fix real upgrade proof.

## What This Changes

The PR adds update-time config preservation defenses around doctor writes, unknown-key stripping, pre-update snapshots, size-drop protection, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the exact config file doctor will write, uses bounded or refreshed pre-update recovery snapshots, keeps legacy config through migration, fixes type coverage, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level. Current main can strip schema-unknown legacy config before write-time migration during an update-triggered doctor write, but I did not replay a packaged macOS npm upgrade.

## Solution Assessment

No as submitted. The root-cause direction is sound, but the PR still snapshots the wrong config path in managed-service updates, keeps an unbounded stale snapshot, fails the test type lane, and lacks real upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Snapshot the doctor config path after env resolution:** `src/cli/update-cli/update-command.ts:1031-1033`
  - body: `resolvePostInstallDoctorEnv` can replace `OPENCLAW_CONFIG_PATH` from the managed service profile, but this snapshots module `CONFIG_PATH` first. That can protect the caller config while doctor mutates the service config, leaving the actual update-time file without a pre-update recovery copy.
  - confidence: 0.91
- **[P2] Refresh or bound pre-update snapshots:** `src/config/backup-rotation.ts:131-138`
  - body: The fixed `.pre-update` path is written with `wx` and `EEXIST` is ignored, so the first historical snapshot permanently wins. Later updates cannot capture their own pre-update state, and stale credential-bearing config can remain indefinitely; replace it per update or use bounded timestamped retention.
  - confidence: 0.88
- **[P2] Provide a complete doctor prompter stub:** `src/flows/doctor-health-contributions.test.ts:190-194`
  - body: `check-test-types` still fails because this helper casts an object whose `prompter` only has `shouldRepair` to `DoctorHealthFlowContext`. Adding runtime methods did not satisfy `DoctorPrompter`; use a complete stub or a narrow typed test helper.
  - confidence: 0.97

## Security Review

Status: needs_attention

Summary: Needs attention: the diff adds a durable first-write-wins copy of potentially sensitive config that never refreshes or rotates.

Concerns:

- **[low] Bound or refresh pre-update config snapshots:** `src/config/backup-rotation.ts:128`
  - body: OpenClaw config can contain secret-shaped values or credential references; a permanent first-write-wins `.pre-update` file can retain stale sensitive data and fail to represent the current pre-update state on later upgrades. Mode `0o600` helps local exposure, but the retention semantics still need a bounded or refreshed design.
  - confidence: 0.82
