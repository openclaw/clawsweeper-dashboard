## Summary

Keep open: this targets a valid source-reproducible config-loss bug, but the current PR head still has two blocking correctness issues in the new snapshot layer and lacks real upgrade proof for the external PR gate.

## What This Changes

The PR adds update-time config preservation guards, a `.pre-update` config snapshot helper, `plugins.installs` protection from unknown-key stripping, update doctor size-drop protection, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the active doctor config path for each update run, refreshes or safely rotates recovery snapshots, keeps the strip and size-drop guards, and includes redacted real upgrade proof preserving config and credential references.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered doctor with write support, strip schema-unknown keys such as legacy `plugins.installs`, and bypass the size-drop guard. I did not run a live upgrade in this read-only review.

## Solution Assessment

No as submitted. The main fix direction is maintainable, but the snapshot must follow the same config path the doctor will mutate and must not become a stale first-write-wins file across future updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path doctor will mutate:** `src/cli/update-cli/update-command.ts:981`
  - body: The snapshot still uses module-level `CONFIG_PATH`, but the child doctor env is built from `resolvePostInstallDoctorEnv` and can carry managed-service `OPENCLAW_CONFIG_PATH` or `OPENCLAW_STATE_DIR`. In that update path, the backup can be written beside the caller config while doctor mutates the service config, so derive the snapshot path from the same resolved env.
  - confidence: 0.9
- **[P2] Refresh recovery snapshots between update runs:** `src/config/backup-rotation.ts:137-142`
  - body: `createPreUpdateConfigSnapshot` writes with `wx` and ignores `EEXIST`, so a `.pre-update` file from an older update blocks all future snapshots. A later upgrade may have no recovery copy of its actual pre-update config and can retain stale secret-bearing data indefinitely.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: Needs attention because the new `.pre-update` snapshot can keep stale secret-bearing config indefinitely outside rotation.

Concerns:

  - body: OpenClaw config can contain credential references and occasionally sensitive values; a first-write-wins `.pre-update` file outside rotation can survive credential changes and later updates while no longer representing the current pre-update state.
  - confidence: 0.82
