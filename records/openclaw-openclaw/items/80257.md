## Summary

Keep open. The PR targets a real source-reproducible update-time config loss path, but the current patch still backs up the wrong config path in managed-service cases, leaves a stale persistent snapshot/security issue, fails the test-types check, and lacks real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards, a `.pre-update` snapshot helper, `plugins.installs` protection from unknown-key stripping, update doctor size-drop protection, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the active doctor config path for each update run, refreshes or safely rotates recovery snapshots, keeps the strip and size-drop guards, fixes the type-check failure, and includes redacted real upgrade proof.

## Reproduction Assessment

No full end-to-end reproduction is established. Source inspection gives a clear current-main path for update-triggered doctor writes to strip legacy config before migration, but I did not replay a packaged macOS npm upgrade or explain credentials-directory deletion.

## Solution Assessment

No as submitted. The fix direction is maintainable, but the snapshot must follow the active doctor config path, old recovery snapshots must not block future update snapshots, and the new test helper must pass type checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Snapshot the config path doctor will mutate:** `src/cli/update-cli/update-command.ts:1031-1033`
  - body: The snapshot uses module-level `CONFIG_PATH`, but the child doctor env can come from managed-service `OPENCLAW_CONFIG_PATH` or `OPENCLAW_STATE_DIR`. In that path the backup is written beside the caller config while doctor mutates the service config, so derive the snapshot path from the same resolved env passed to doctor.
  - confidence: 0.91
- **[P2] Refresh recovery snapshots between update runs:** `src/config/backup-rotation.ts:131-138`
  - body: `createPreUpdateConfigSnapshot` writes with `wx` and ignores `EEXIST`, so a `.pre-update` file from an older update blocks all future snapshots. A later upgrade can then lack a recovery copy of its actual pre-update config while retaining stale secret-bearing data indefinitely.
  - confidence: 0.9
- **[P2] Fix the doctor test context type:** `src/flows/doctor-health-contributions.test.ts:180-194`
  - body: The current PR merge fails `check-test-types` with `TS2352` at this new helper because the object cast does not sufficiently overlap `DoctorHealthFlowContext`. Narrow the test fixture types or cast through `unknown` so the repository type gate passes.
  - confidence: 0.86
