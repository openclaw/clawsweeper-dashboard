## Summary

Keep open. The PR targets a real source-reproducible update config-loss path, but the latest head still has snapshot-path/snapshot-refresh blockers, a test-type CI failure, and no after-fix real upgrade proof.

## What This Changes

This PR adds update-time config-preservation safeguards: a pre-update config snapshot, update-aware unknown-key stripping, plugins.installs protection, an update size-drop guard, and regression tests.

## Best Possible Solution

Land a corrected update-time guard that snapshots the active config path once per update run, preserves config through a real upgrade, explains credentials preservation, and keeps focused regression coverage for schema, doctor write, and backup behavior.

## Reproduction Assessment

Yes by source inspection, but not by a live packaged upgrade run. Current main can run update-triggered doctor with write privileges, the strict plugins schema treats plugins.installs as unknown, and the unknown-key step runs before the final config write path.

## Solution Assessment

No as submitted. The guard direction is appropriate, but the snapshot must refresh for each update, use the same config path as the doctor process, compile cleanly, and be proven with real upgrade output before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Refresh stale pre-update snapshots for new updates:** `src/config/backup-rotation.ts:131-138`
  - body: This writes a fixed `${configPath}.pre-update` with `wx` and treats `EEXIST` as success. After one update creates that file, every later update silently keeps the old snapshot, so a future destructive upgrade would not have a backup of the user's current config. Keep idempotence within one update run, but rotate or replace the snapshot when a new update starts.
  - confidence: 0.9
- **[P2] Snapshot the config path passed to child doctor:** `src/cli/update-cli/update-command.ts:982-984`
  - body: This snapshot uses module-level `CONFIG_PATH`, but the child doctor env below is built by `resolvePostInstallDoctorEnv(...)`, which can carry `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` from the managed service. In that case doctor can mutate the service config while the snapshot was created beside a different parent-process config path; resolve the snapshot path from the same env passed to `runUpdateStep`.
  - confidence: 0.88
- **[P2] Provide a full DoctorPrompter test stub:** `src/flows/doctor-health-contributions.test.ts:190-194`
  - body: The current PR head fails `check-test-types` because this object only supplies `prompter.shouldRepair`, while `DoctorHealthFlowContext` requires the full `DoctorPrompter` interface. Use an existing helper or stub the required methods instead of relying on a direct cast.
  - confidence: 0.95
