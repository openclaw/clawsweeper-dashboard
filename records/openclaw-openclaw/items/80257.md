## Summary

Keep open. The PR targets a real source-reproducible update config-loss bug and the strip/size-drop guard direction is useful, but the current branch still snapshots a possibly wrong config path, retains stale first-write-wins snapshots indefinitely, and lacks real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards: a `.pre-update` snapshot, update-time unknown-key stripping bypass, `plugins.installs` strip protection, a size-drop guard change, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the exact doctor target path, uses bounded or refreshed recovery snapshots, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered doctor with write support, strip schema-unknown keys such as `plugins.installs` before migration, and bypass the size-drop rejection. I did not run the full packaged macOS/npm upgrade path.

## Solution Assessment

No as submitted. The fix direction is maintainable, but the snapshot must follow the same resolved config path doctor will mutate and the recovery snapshot needs bounded refresh or retention semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path doctor will mutate:** `src/cli/update-cli/update-command.ts:1029`
  - body: The snapshot uses module-level `CONFIG_PATH`, but package-update doctor runs with `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR` from the managed service. A service-profile update can still mutate the service config without a matching `.pre-update` recovery copy; derive the snapshot path from the resolved doctor env at both update-time doctor call sites.
  - confidence: 0.91
- **[P2] Refresh or bound pre-update recovery snapshots:** `src/config/backup-rotation.ts:132`
  - body: `createPreUpdateConfigSnapshot` writes with `wx` and treats `EEXIST` as success, so any older `.pre-update` file blocks later upgrades from capturing their actual pre-upgrade config while retaining stale config data outside backup cleanup. Replace, rotate, or timestamp this with bounded retention while preserving the first snapshot within a single update flow.
  - confidence: 0.9
