## Summary

Keep this PR open, but do not merge as submitted: the reported config-loss path is valid at source level, yet the current patch still snapshots the wrong config path in managed-service update flows, leaves a permanent stale recovery file, and lacks real after-fix upgrade proof.

## What This Changes

The PR adds update-time config preservation by creating pre-update snapshots, skipping unknown-key stripping during updates, protecting `plugins.installs`, disabling update-time size-drop writes, and adding focused tests.

## Best Possible Solution

Update this PR to snapshot the same resolved config path that doctor will mutate, use bounded or refreshed pre-update recovery snapshots, keep the strip and size-drop guards, and attach redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor writes while unknown-key stripping happens before the `plugins.installs` write migration, and the size-drop guard can be disabled by doctor repair writes. I did not replay a packaged upgrade in a real install.

## Solution Assessment

No as submitted. The root-cause direction is sound, but the PR should derive the snapshot from the resolved doctor env, avoid permanent stale snapshots, and include real after-fix upgrade evidence before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Snapshot the resolved doctor config path:** `src/cli/update-cli/update-command.ts:1031-1033`
  - body: Package-update doctor receives env from `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` from the managed service profile. This call snapshots module `CONFIG_PATH` before that env is applied, so it can save the caller config while doctor mutates the service config; derive the snapshot path from the same resolved env.
  - confidence: 0.91
- **[P2] Refresh or bound pre-update snapshots:** `src/config/backup-rotation.ts:131-135`
  - body: The fixed `.pre-update` path is written with `wx` and `EEXIST` is ignored, so the first ever snapshot permanently wins. Later updates cannot capture their own pre-update state and stale config data remains; replace it per update or use bounded timestamped retention.
  - confidence: 0.88
