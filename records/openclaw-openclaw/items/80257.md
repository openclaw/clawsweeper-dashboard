## Summary

Keep open. The PR targets a plausible config-loss root cause, but the current head still has blocking issues: stale fixed-name snapshots, snapshots taken from the wrong config path for managed-service update doctors, a failing test-type check, and no real after-fix upgrade proof.

## What This Changes

The PR adds pre-update config snapshots, suppresses unknown-key stripping during updates, protects `plugins.installs`, disables update-time size-drop allowance, and adds focused tests.

## Best Possible Solution

Land a CI-clean patch that snapshots the actual update-doctor config path for the current upgrade run, avoids stale snapshots across future updates, preserves migration-owned config keys, and includes redacted real upgrade proof.

## Reproduction Assessment

No high-confidence full reproduction was established. Source inspection shows a plausible strip-before-write-migration path during update-triggered doctor, but I did not verify a packaged v2026.5.6-to-head upgrade with populated config and credentials.

## Solution Assessment

No. The selected update, doctor, and config-write surfaces are reasonable, but the current patch still has stale snapshot behavior, a managed-service config-path gap, a type-check failure, and missing real upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Snapshot the config path used by post-install doctor:** `src/cli/update-cli/update-command.ts:982-984`
  - body: This snapshots module-level `CONFIG_PATH` before the doctor env is resolved. For managed service updates, `resolvePostInstallDoctorEnv` can pass `OPENCLAW_HOME`, `OPENCLAW_STATE_DIR`, or `OPENCLAW_CONFIG_PATH` from the service env, so the doctor may write a different config than the `.pre-update` snapshot protects; derive the snapshot path from the same env used for the doctor child.
  - confidence: 0.84
- **[P2] Avoid reusing stale pre-update snapshots:** `src/config/backup-rotation.ts:137-138`
  - body: Because `.pre-update` is a fixed path and `EEXIST` is ignored, the first successful update leaves a file that blocks every later update from capturing the current pre-upgrade config. After user config changes, a future bad update would recover to an old snapshot instead of the latest state; use a per-update/versioned snapshot or a guard that only prevents duplicate writes within the same update run.
  - confidence: 0.9
- **[P2] Make the write-config test context type-correct:** `src/flows/doctor-health-contributions.test.ts:180-194`
  - body: `check-test-types` still fails on this helper with TS2352 because the object is cast to `DoctorHealthFlowContext` while `prompter` only provides `shouldRepair`. Build a complete typed stub/helper instead of relying on the cast so CI can pass.
  - confidence: 0.97
