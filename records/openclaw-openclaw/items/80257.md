## Summary

Keep open. This PR targets a real source-reproducible update-time config-loss path, but the latest head still snapshots the wrong config path, leaves stale `.pre-update` snapshots across later updates, fails the test-types check, and lacks real upgrade proof.

## What This Changes

The branch adds update-time config preservation safeguards: pre-update snapshots, update-aware unknown-key stripping, `plugins.installs` protection, an update size-drop guard, and regression tests.

## Best Possible Solution

Land a corrected update-time config preservation fix that snapshots the active config path per update run, preserves `plugins.installs` until migration, keeps update size-drop protection enabled, passes test type checks, and proves a real upgrade preserves config.

## Reproduction Assessment

Yes by source inspection. Current main can run update-triggered doctor with config-write privileges, `plugins.installs` is unknown to the strict schema, and unknown-key stripping happens before the final doctor write.

## Solution Assessment

No as submitted. The guard direction is appropriate, but the snapshot layer must use the same active config path as doctor, refresh per update run, fix the failing test type check, and include real upgrade proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Refresh stale pre-update snapshots for each update:** `src/config/backup-rotation.ts:131-138`
  - body: This writes a fixed `.pre-update` file with `wx` and treats `EEXIST` as success. After one update creates that file, later updates silently keep the old snapshot, so a future destructive update would not have a backup of the user's current pre-update config. Keep idempotence within one update run, but rotate or replace the recovery point when a new update starts.
  - confidence: 0.9
- **[P2] Snapshot the doctor env config path:** `src/cli/update-cli/update-command.ts:983-984`
  - body: The snapshot uses module-level `CONFIG_PATH` before the post-install doctor env is resolved, but that env can carry service `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH`. In managed-service cases, doctor can mutate one config while the snapshot was created beside another; resolve the snapshot path from the same env passed to doctor, including the restart doctor path.
  - confidence: 0.88
- **[P2] Fix the doctor health test context type:** `src/flows/doctor-health-contributions.test.ts:194`
  - body: The PR head currently fails `check-test-types` with TS2352 on this returned object because it does not sufficiently overlap with `DoctorHealthFlowContext`. Build a structurally typed test context or cast through `unknown` so the required type check can pass.
  - confidence: 0.95
