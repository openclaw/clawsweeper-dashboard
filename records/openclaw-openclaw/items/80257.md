## Summary

Keep open: the underlying config-loss bug is source-reproducible on current main, but this PR still has two blocking snapshot correctness issues and no after-fix real upgrade proof.

## What This Changes

The PR adds update-time config preservation safeguards for doctor-driven upgrades, including a pre-update config snapshot, update-aware unknown-key stripping, a protected plugins.installs key, and size-drop protection tests.

## Best Possible Solution

Land a corrected update-time guard that snapshots the active config path once per update run, preserves config and credentials through a real upgrade, and keeps focused regression coverage for schema, doctor write, and backup behavior.

## Reproduction Assessment

Yes. Source inspection shows current main can run update-triggered doctor with write privileges, strip plugins.installs as an unknown key before final write, and allow size drops during that write path.

## Solution Assessment

No as submitted. The guard direction is appropriate, but the snapshot must be refreshed per update run and use the same config path the doctor will mutate; the PR also needs real upgrade proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Refresh stale pre-update snapshots for new updates:** `src/config/backup-rotation.ts:130-137`
  - body: This writes a fixed `${configPath}.pre-update` with `wx` and treats `EEXIST` as success. After one update creates that file, every later update silently keeps the old snapshot, so a future destructive upgrade would not have a backup of the user's current config. Keep idempotence within one update run, but replace or rotate the snapshot when a new update starts.
  - confidence: 0.9
- **[P2] Snapshot the config path passed to doctor:** `src/cli/update-cli/update-command.ts:983`
  - body: This snapshots the module-level `CONFIG_PATH`, but the child doctor env is built by `resolvePostInstallDoctorEnv(...)`, which can carry `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` from the managed service. In that case doctor can mutate the service config while the snapshot was created beside a different parent-process config path; resolve the snapshot path from the same env passed to the doctor and apply the same fix to the restart doctor call.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces a persistent secret-bearing config snapshot, and the stale-snapshot bug can retain old config secrets indefinitely even though the file mode is owner-only.

Concerns:

- **[low] Avoid indefinitely stale config snapshots:** `src/config/backup-rotation.ts:130`
  - body: OpenClaw config can contain sensitive channel/provider material or secret references. Because `.pre-update` is first-write-wins and outside the backup rotation, an old snapshot can persist after later updates and credential/config changes; replacing or rotating it per update would reduce that exposure while preserving recovery.
  - confidence: 0.78
