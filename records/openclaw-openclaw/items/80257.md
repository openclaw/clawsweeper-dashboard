## Summary

Keep open: this is a real source-reproducible config preservation bug, but the current PR still has two blocking issues in the recovery snapshot path and lacks real upgrade proof.

## What This Changes

The PR adds update-time config-preservation guards, a pre-update config snapshot helper, and focused tests for unknown-key stripping, size-drop writes, and backup rotation.

## Best Possible Solution

Land a corrected update-time preservation fix that snapshots the same config path doctor will write, rotates or refreshes recovery snapshots per update run, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered doctor with write support, strip schema-unknown config such as legacy `plugins.installs`, and bypass the size-drop rejection path. I did not replay a packaged macOS npm upgrade or prove the credential-directory deletion path.

## Solution Assessment

No as submitted. The guard direction is appropriate, but the recovery snapshot must follow the active doctor config path and must not become a permanent first-write-wins stale copy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the active doctor config path:** `src/cli/update-cli/update-command.ts:983`
  - body: This snapshots module-level `CONFIG_PATH` before building the child doctor env. When managed-service `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` is supplied, `resolvePostInstallDoctorEnv` can point doctor at a different config file, leaving the file doctor mutates without a pre-update snapshot.
  - confidence: 0.88
- **[P2] Refresh or rotate pre-update snapshots:** `src/config/backup-rotation.ts:134-138`
  - body: The helper writes with `wx` and treats `EEXIST` as a no-op, so the first `.pre-update` file survives forever. Later updates will neither capture the current pre-update state nor remove stale secret-bearing config; replace, rotate, or timestamp snapshots per update while preserving idempotence within one update run.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: Needs attention because the first-write-wins `.pre-update` snapshot can retain stale secret-bearing config indefinitely outside backup rotation.

Concerns:

- **[low] Avoid indefinitely stale config snapshots:** `src/config/backup-rotation.ts:134`
  - body: OpenClaw config can contain channel/provider-sensitive data and secret references. Because `.pre-update` is first-write-wins and outside normal backup rotation, an old snapshot can survive later updates and credential changes while also failing to capture the current pre-update state.
  - confidence: 0.82
