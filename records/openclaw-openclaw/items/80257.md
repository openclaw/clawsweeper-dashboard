## Summary

Keep open: the PR targets a valid source-reproducible update-time config data-loss path, but the latest head still snapshots the wrong config path for managed-service doctor runs, keeps stale first-write-wins recovery snapshots, and lacks real upgrade proof.

## What This Changes

Adds update-time config preservation guards: a pre-update snapshot helper, unknown-key strip bypass during update, `plugins.installs` strip protection, size-drop guard changes, and focused tests.

## Best Possible Solution

Land a corrected update-time preservation fix that snapshots the exact config path doctor will mutate, refreshes recovery snapshots under managed retention, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered `doctor --fix` with write support, strip schema-unknown keys such as legacy `plugins.installs`, and bypass the size-drop guard. I did not establish a live upgrade reproduction in this read-only review.

## Solution Assessment

No as submitted. The fix direction is maintainable, but the snapshot must follow the same resolved config path the doctor will mutate and recovery snapshots need sane refresh or retention semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path passed to doctor:** `src/cli/update-cli/update-command.ts:1029`
  - body: The snapshot uses module-level `CONFIG_PATH`, but the package-update doctor process is launched with `resolvePostInstallDoctorEnv`, which can overwrite `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR` from the managed service. In that path the recovery copy can be written beside the caller config while doctor mutates the service config, so derive the snapshot path from the same resolved doctor env before launching it.
  - confidence: 0.9
- **[P2] Refresh the update recovery snapshot:** `src/config/backup-rotation.ts:132`
  - body: Writing `.pre-update` with `wx` and ignoring `EEXIST` makes the snapshot first-write-wins across all future updates. Once an older snapshot exists, a later update gets no copy of its actual pre-update config and keeps stale credential-bearing config outside managed backup cleanup; rotate, replace, or timestamp it with bounded retention instead.
  - confidence: 0.9
