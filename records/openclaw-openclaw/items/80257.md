## Summary

Keep open. The branch targets a valid source-reproducible config-loss bug, but the current PR still has blocking correctness issues, failing test type checks, a concrete snapshot-retention security concern, and no real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards, a `.pre-update` snapshot helper, `plugins.installs` strip protection, size-drop guard changes, and focused tests.

## Best Possible Solution

Land a corrected update-preservation fix that snapshots the same resolved config path doctor will mutate, keeps legacy install records migratable, preserves size-drop protection during updates, uses bounded or refreshed recovery snapshots, and includes real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor with write support, strip schema-unknown `plugins.installs` before migration, and bypass the size-drop rejection. I did not establish a live packaged macOS/npm upgrade reproduction.

## Solution Assessment

No as submitted. The root-cause direction is sound, but the PR still snapshots the wrong config path in managed-service updates, keeps stale recovery snapshots indefinitely, fails test type checks, and lacks real upgrade proof.

  - body: The latest CI run fails `pnpm check:test-types` because this new test context casts a partial object whose `prompter` only has `shouldRepair`. Add the required `DoctorPrompter` methods or cast through a narrower helper so the test type check passes.

  - body: The new snapshot uses module-scope `CONFIG_PATH`, but the package-update doctor child env is built by `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` with the managed service profile path. In that case this snapshots the caller config while doctor mutates the service config; derive the snapshot path from the same resolved env passed to doctor.
  - confidence: 0.91
- **[P2] Refresh or rotate pre-update snapshots:** `src/config/backup-rotation.ts:134`
  - body: `createPreUpdateConfigSnapshot` writes a fixed `.pre-update` file with `flag: "wx"` and returns on `EEXIST`, so the first update snapshot prevents all future updates from recording their actual starting state. Replace it per update or use bounded timestamped retention.
  - confidence: 0.9
