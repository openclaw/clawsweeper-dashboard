## Summary

Keep open. The PR targets a real source-reproducible update-time config-loss bug, but the current branch still snapshots the wrong config path in managed-service update flows, keeps a stale first-write-wins snapshot across future updates, and lacks required after-fix real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards: a pre-update config snapshot, update-time unknown-key stripping bypass, a `plugins.installs` strip whitelist, update-doctor size-drop protection, and focused tests.

## Best Possible Solution

Land a corrected update-time preservation fix that snapshots the active doctor config once per update run, preserves legacy config through doctor writes, and is backed by redacted real upgrade proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main can run update-triggered doctor with write privileges, treat legacy `plugins.installs` as an unknown strict-schema key, and allow a size-drop write on that path. I did not run a live v2026.5.6 to v2026.5.7 upgrade in this read-only review.

## Solution Assessment

No as submitted. The direction is appropriate, but the branch still needs to snapshot the same effective config path doctor will write, refresh or rotate the pre-update snapshot per update run, and provide real upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Refresh pre-update snapshots per update:** `src/config/backup-rotation.ts:134-138`
  - body: The snapshot is a fixed `${configPath}.pre-update` written with `flag: "wx"`, and EEXIST is treated as success. After one update creates it, later updates keep the old config copy, so a future destructive update would not have a backup of the user's current pre-update config and can retain stale secret-bearing config. Keep same-run idempotence, but replace, rotate, or timestamp the snapshot when a new update starts.
  - confidence: 0.9
- **[P2] Snapshot the config path passed to doctor:** `src/cli/update-cli/update-command.ts:983`
  - body: This snapshots module-level `CONFIG_PATH`, but the post-install doctor env is built from managed service overrides such as `OPENCLAW_STATE_DIR` and `OPENCLAW_CONFIG_PATH`. In managed-service package updates, doctor can mutate the service config while the backup is written next to a different parent-process config; derive the snapshot path from the same env used to run doctor and keep the restart-doctor path consistent.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff adds a persistent config snapshot that can retain stale secret-bearing config indefinitely unless refreshed or rotated per update.

Concerns:

- **[low] Avoid indefinitely stale config snapshots:** `src/config/backup-rotation.ts:134`
  - body: OpenClaw config can contain channel/provider secrets or secret references. Because `.pre-update` is first-write-wins and outside normal backup rotation, an old snapshot can persist after later updates and credential changes; replacing, timestamping, or rotating it per update would reduce exposure while preserving recovery.
  - confidence: 0.78
