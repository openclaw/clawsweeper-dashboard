## Summary

Keep open: the PR targets a real update-time config-loss path, but current head still has two blocking issues and lacks real upgrade proof before merge.

## What This Changes

The PR branch adds a `.pre-update` config snapshot, skips unknown-key stripping during update doctor runs, protects `plugins.installs`, disables update-time size-drop writes, and adds targeted tests.

## Best Possible Solution

Land a corrected update-time preservation fix that snapshots the same config path doctor will write, refreshes or rotates recovery snapshots per update run without clobbering within one run, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level rather than live: current main can run update-triggered doctor with write support, strip strict-schema-unknown config keys including legacy `plugins.installs`, and bypass the size-drop guard on that write path.

## Solution Assessment

No as submitted. The direction is appropriate, but the snapshot path and snapshot freshness/security behavior need correction before this is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Snapshot the active doctor config path:** `src/cli/update-cli/update-command.ts:983`
  - body: The child doctor env is built below from managed-service `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH`, but this snapshot uses module-level `CONFIG_PATH` from the updater parent. Managed-service updates can still mutate one config while the recovery snapshot is written beside another, so derive the snapshot path from the same env passed to doctor and keep the restart doctor path consistent.
  - confidence: 0.88
- **[P2] Refresh snapshots between update runs:** `src/config/backup-rotation.ts:134-138`
  - body: Using `wx` plus an `EEXIST` no-op makes `.pre-update` first-write-wins for the lifetime of the file. A later update would keep an old recovery copy instead of the user's current pre-update config, and old secret-bearing config can persist indefinitely; replace, rotate, or timestamp snapshots per update while preserving idempotence within one run.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR adds a persistent config snapshot that can retain stale secret-bearing config indefinitely unless refreshed, rotated, or otherwise bounded.

Concerns:

- **[low] Avoid indefinitely stale config snapshots:** `src/config/backup-rotation.ts:134`
  - body: OpenClaw config can contain secret references and channel/provider-sensitive data. Because `.pre-update` is first-write-wins and outside normal backup rotation, an old snapshot can survive later updates and credential changes while failing to capture the current pre-update state.
  - confidence: 0.8
