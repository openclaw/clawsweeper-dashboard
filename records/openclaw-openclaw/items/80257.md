## Summary

Keep open. The PR targets a real update-time config preservation bug, but the proposed snapshot layer can overwrite the original pre-update backup later in the same update flow and the external contributor has not provided real packaged-upgrade proof.

## What This Changes

The PR adds update-time config preservation defenses: a `.pre-update` config snapshot, update-aware unknown-key stripping, `plugins.installs` protection, update-time size-drop rejection, and focused tests.

## Best Possible Solution

Make the pre-update snapshot first-write-wins, keep the update-time doctor guards, and validate the full packaged upgrade path preserves both `openclaw.json` and `credentials/`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: the current source shows the doctor/update config-loss path and the PR-introduced snapshot overwrite issue, but there is no live packaged macOS/npm upgrade reproduction in the PR.

## Solution Assessment

Is this the best way to solve the issue? No as written; the defense-in-depth direction is right, but the snapshot must not overwrite the first pre-update copy and the fix needs real upgrade proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve the first pre-update snapshot:** `src/config/backup-rotation.ts:125`
  - body: This writes `.pre-update` with `flag: "w"`, so every call overwrites the previous snapshot. The PR adds call sites before both the package doctor and the restarted in-process doctor; if an earlier update step mutates the config, the later call can replace the good pre-update copy with the bad one. Make this first-write-wins, for example by skipping an existing snapshot or using exclusive create, and add a repeated-call test.
  - confidence: 0.92
