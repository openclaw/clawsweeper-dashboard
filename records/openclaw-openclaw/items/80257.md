## Summary

Keep this PR open: the reported update-time config-loss bug is source-reproducible on current main, but the proposed patch still has two blocking recovery-path bugs and lacks after-fix proof from a real upgrade.

## What This Changes

The PR adds update-time config preservation guards, a pre-update config snapshot, `plugins.installs` protection from unknown-key stripping, size-drop protection during update doctor runs, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the exact config path doctor will write, refreshes or rotates recovery snapshots per update run, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered `doctor --fix` with write support, strip schema-unknown config keys such as legacy `plugins.installs`, and bypass the size-drop guard. I did not run a live upgrade in this read-only review.

## Solution Assessment

No as submitted. The direction is right, but the snapshot must follow the active doctor config path and must not be a permanent first-write-wins stale copy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path passed to doctor:** `src/cli/update-cli/update-command.ts:983`
  - body: This snapshots module-level `CONFIG_PATH`, but `resolvePostInstallDoctorEnv` can pass a managed-service `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` to the child doctor. In that case the recovery snapshot is written beside the caller config while doctor mutates the service config, so derive the snapshot path from the same resolved env used for doctor.
  - confidence: 0.88
- **[P2] Refresh pre-update snapshots between update runs:** `src/config/backup-rotation.ts:134-138`
  - body: The helper uses `wx` and treats `EEXIST` as a no-op, so the first `.pre-update` file wins forever. A later update will neither capture the current pre-update state nor retire stale secret-bearing config, leaving recovery ineffective after the first update run.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds a persistent config snapshot path that can retain stale secret-bearing config indefinitely outside backup rotation.

  - body: OpenClaw config can contain credential references and sometimes sensitive values; a first-write-wins `.pre-update` file outside rotation can survive credential changes and later updates while no longer representing the current pre-update state.
