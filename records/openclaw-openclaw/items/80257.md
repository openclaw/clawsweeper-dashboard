## Summary

Keep open: the PR targets a real update-time config-loss path, but the current head still snapshots the wrong config in managed-service update flows, keeps a first-write-wins snapshot outside rotation, and lacks real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards, a pre-update config snapshot, `plugins.installs` unknown-key protection, size-drop protection during update doctor runs, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the exact config path doctor will write, refreshes or rotates pre-update recovery snapshots per update run, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level: current main can invoke update-triggered `doctor --fix` with write support, strip schema-unknown legacy `plugins.installs`, and bypass the size-drop safety net. I did not run a live v2026.5.6 to v2026.5.7+ upgrade.

## Solution Assessment

No as submitted. The direction is right, but the snapshot must follow the active doctor config path and must refresh or rotate instead of permanently preserving the first `.pre-update` file.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path passed to doctor:** `src/cli/update-cli/update-command.ts:983`
  - body: This snapshots module-level `CONFIG_PATH`, but the child doctor environment is built by `resolvePostInstallDoctorEnv` and can carry a managed-service `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH`. In that case the recovery snapshot is written beside the caller config while doctor mutates the service config, so derive the snapshot path from the same resolved env used for doctor.
  - confidence: 0.88
- **[P2] Refresh pre-update snapshots between updates:** `src/config/backup-rotation.ts:135`
  - body: The helper uses `wx` and treats `EEXIST` as a no-op, so the first `.pre-update` file wins forever. A later update will neither capture the current pre-update state nor retire stale secret-bearing config, leaving recovery ineffective after the first update run.
  - confidence: 0.9
