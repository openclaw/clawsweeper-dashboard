## Summary

Keep open: the reported update-time config loss is source-reproducible and the PR heads in the right direction, but the current patch still snapshots the wrong path in managed-service updates, leaves first-write-wins stale snapshots outside rotation, and lacks real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards, a `.pre-update` config snapshot, `plugins.installs` protection from unknown-key stripping, size-drop protection during update doctor runs, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the active doctor config path for each update run, rotates or refreshes recovery snapshots, keeps the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes, at source level: current main can run update-triggered `doctor --fix` with write support, strip schema-unknown config keys such as legacy `plugins.installs`, and bypass the size-drop guard. I did not run a live upgrade in this read-only review.

## Solution Assessment

No as submitted. The direction is right, but the snapshot must follow the config path doctor will mutate and must not be a permanent first-write-wins stale copy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Snapshot the config path doctor will mutate:** `src/cli/update-cli/update-command.ts:983`
  - body: The snapshot always uses module-level `CONFIG_PATH`, but `resolvePostInstallDoctorEnv` can pass managed-service `OPENCLAW_CONFIG_PATH` or `OPENCLAW_STATE_DIR` into the child doctor. In that update path, the backup can be written beside the caller config while doctor mutates the service config, so derive the snapshot path from the same resolved doctor env.
  - confidence: 0.9
- **[P2] Refresh recovery snapshots between update runs:** `src/config/backup-rotation.ts:134-138`
  - body: `createPreUpdateConfigSnapshot` writes with `wx` and returns on `EEXIST`, so a `.pre-update` file from an older update blocks all future snapshots. A later upgrade may have no recovery copy of its actual pre-update config and can retain stale secret-bearing data indefinitely.
  - confidence: 0.9

## Security Review

  - body: OpenClaw config can contain credential references and sometimes sensitive values; a first-write-wins `.pre-update` file outside rotation can survive credential changes and later updates while no longer representing the current pre-update state.
