## Summary

Keep open. The PR targets a real, source-reproducible update config-clobber path, but the current branch still has blocking problems: stale first-write-wins snapshots, snapshotting a different config path than the child doctor may mutate, a failing test type check, and no real upgrade proof.

## What This Changes

The PR adds update-time doctor config preservation guards, a persistent pre-update config snapshot, protected unknown-key handling for `plugins.installs`, and focused regression tests.

## Best Possible Solution

Land a corrected update guard that snapshots the active config path once per update run, avoids stale secret-bearing snapshots, preserves config and credentials through a real upgrade, and keeps focused regression coverage.

## Reproduction Assessment

Yes for the config-clobber path at source level: current main can run update-triggered doctor with write privileges, strip `plugins.installs` before final write, and allow the size-drop guard to be bypassed. I did not establish a live packaged v2026.5.6 to v2026.5.7+ upgrade run.

## Solution Assessment

No as submitted. The guard direction is appropriate, but the snapshot must refresh for each update and follow the same config path the doctor will mutate; the PR also needs a complete test context and real upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Refresh snapshots for each update run:** `src/config/backup-rotation.ts:131-138`
  - body: This makes `.pre-update` first-write-wins forever: once a prior update leaves the file behind, every later update treats `EEXIST` as success and never captures the user's current config. Keep idempotence within one update run, but replace or rotate the snapshot when a new update starts.
  - confidence: 0.9
- **[P2] Snapshot the config path used by the doctor:** `src/cli/update-cli/update-command.ts:983-984`
  - body: This snapshots the module-level `CONFIG_PATH`, but the child doctor env can include `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` from the managed service. In that case the doctor may mutate a different config than the one backed up; resolve the snapshot path from the same env passed to doctor and apply the same consistency to the restart doctor path.
  - confidence: 0.87
- **[P2] Build a complete doctor test context:** `src/flows/doctor-health-contributions.test.ts:190`
  - body: `check-test-types` currently fails because this test casts a partial object to `DoctorHealthFlowContext` while `prompter` only provides `shouldRepair`. Add the missing `DoctorPrompter` methods or route the test through a typed helper so the PR typechecks.
