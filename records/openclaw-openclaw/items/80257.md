## Summary

Keep open. The PR targets a real source-reproducible update config-loss bug, but the latest branch still has blocking correctness issues, a failing `check:test-types` CI job, and no real upgrade proof.

## What This Changes

The PR adds update-time config preservation guards: a `.pre-update` snapshot helper, update-mode unknown-key strip bypass, `plugins.installs` strip protection, size-drop guard changes, and regression tests.

## Best Possible Solution

Land a corrected update-time preservation fix that snapshots the exact config path doctor mutates, keeps recovery snapshots current under bounded retention, preserves the strip and size-drop guards, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered `doctor --fix` with write support, strict-schema unknown-key stripping, and a plugin schema that omits legacy `plugins.installs`. I did not replay a packaged macOS npm upgrade.

## Solution Assessment

No as submitted. The fix direction is maintainable, but the snapshot must follow the same resolved config path the doctor will mutate, recovery snapshots need sane refresh or retention semantics, and the test type failure must be fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Snapshot the config path passed to doctor:** `src/cli/update-cli/update-command.ts:1029`
  - body: The snapshot uses module-level `CONFIG_PATH`, but package-update doctor is launched with `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` from the managed service. In that path the backup can be written beside the caller config while doctor mutates a different service config, leaving the actual pre-update config unprotected.
  - confidence: 0.9
- **[P2] Refresh the update recovery snapshot:** `src/config/backup-rotation.ts:132`
  - body: The `.pre-update` snapshot is written with `wx` and ignored on `EEXIST`, so the first update forever blocks later updates from capturing their own pre-update config. This leaves stale secret-bearing config outside backup cleanup and removes the recovery copy for future upgrade failures.
  - confidence: 0.88
- **[P2] Complete the doctor test context stub:** `src/flows/doctor-health-contributions.test.ts:180`
  - body: `check:test-types` fails because the new test casts a context whose `prompter` only has `shouldRepair`, but `DoctorHealthFlowContext` requires the full `DoctorPrompter` surface. Use a complete typed stub or a narrower helper so `pnpm check:test-types` passes again.
  - confidence: 0.95
