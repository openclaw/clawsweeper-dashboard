## Summary

Keep open. The PR targets a real update-time config preservation bug, but current head still has blocking review defects in snapshot freshness/path selection, a changed-file type-check failure, and no real upgrade proof.

## What This Changes

The PR adds update-time config preservation safeguards: a pre-update config snapshot, update-aware unknown-key stripping, a plugins.installs strip exemption, stricter size-drop handling, and focused tests.

## Best Possible Solution

Land a corrected fix that snapshots the exact config path used by update doctor, refreshes recovery snapshots per update run, preserves config and credentials through a real upgrade, and keeps focused regression coverage.

## Reproduction Assessment

Yes, by source inspection for the config-loss path: current main can run update-triggered doctor with write privileges, strict schema does not include plugins.installs, and unknown-key stripping can run before write-time install-record migration. I did not reproduce the credentials-directory deletion live.

## Solution Assessment

No as submitted. The direction is appropriate, but the snapshot recovery design can be stale or target the wrong config path, the new test code fails type checking, and the PR still needs real upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Refresh pre-update snapshots per update run:** `src/config/backup-rotation.ts:134-138`
  - body: Because `.pre-update` is opened with `wx` and EEXIST is treated as success, the first successful update leaves a permanent snapshot. Later updates with a different current config skip creating a fresh recovery point, so a destructive future update would point users at stale config.
  - confidence: 0.9
- **[P2] Snapshot the child doctor's config path:** `src/cli/update-cli/update-command.ts:982-984`
  - body: This snapshots module-level `CONFIG_PATH`, but the child doctor below is launched with `resolvePostInstallDoctorEnv(...)`, which can replace `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` from the managed service env. Resolve the snapshot path from the same env passed to the child doctor so the file being mutated has the recovery snapshot.
  - confidence: 0.87
- **[P2] Fix the new doctor context stub type:** `src/flows/doctor-health-contributions.test.ts:180`
  - body: The latest `check-test-types` run fails on this return object because it is cast directly to `DoctorHealthFlowContext` while the `prompter` stub only has `shouldRepair`. Complete the stub or cast through `unknown` intentionally so the PR type-checks.
  - confidence: 0.95
