## Summary

Keep open. The PR targets a real source-reproducible data-loss path, but it still has blocking preservation issues, a failing test-type check, a low-risk credential snapshot retention concern, and no redacted real upgrade proof.

## What This Changes

The branch adds update-time config preservation guards, a `.pre-update` snapshot helper, `plugins.installs` strip protection, a size-drop guard change, and focused tests.

## Best Possible Solution

Land a corrected preservation fix that snapshots the actual doctor target config, uses bounded or refreshed recovery snapshots, keeps the strip and size-drop guards, fixes test types, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor with write support, strip schema-unknown keys such as `plugins.installs` before migration, and bypass the size-drop rejection. I did not replay a packaged macOS/npm upgrade.

## Solution Assessment

No as submitted. The direction is maintainable, but the snapshot must follow the same resolved config path doctor will mutate, recovery snapshots need bounded refresh or retention semantics, and the test-type gate must pass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Snapshot the resolved doctor config path:** `src/cli/update-cli/update-command.ts:1032`
  - body: The snapshot uses module-scope `CONFIG_PATH`, but this package-update path builds the child doctor environment with `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` from the managed service profile. In that case recovery protects the caller config while doctor mutates the service config; compute the snapshot path from the same resolved env.
  - confidence: 0.91
- **[P2] Refresh recovery snapshots between updates:** `src/config/backup-rotation.ts:134`
  - body: Using `flag: "wx"` and returning on `EEXIST` makes the first `.pre-update` copy permanent. Future updates skip capturing their actual pre-update state and keep an old credential-bearing config; use per-update replacement, timestamped bounded snapshots, or another retention policy.
  - confidence: 0.88
- **[P2] Fix the doctor write test context type:** `src/flows/doctor-health-contributions.test.ts:180`
  - body: `check-test-types` fails because this object is cast directly to `DoctorHealthFlowContext` while its `prompter` only has `shouldRepair`. Add the required prompter methods, use an existing test helper, or cast through `unknown` after intentionally stubbing the needed surface so the test-type gate passes.
  - confidence: 0.96
