## Summary

Keep open: the root-cause direction is valid, but the current PR head still snapshots the wrong config path in managed-service update flows, makes recovery snapshots permanently first-write-wins, and lacks required real upgrade proof.

## What This Changes

The branch adds update-time unknown-key strip guards, `plugins.installs` strip protection, pre-update config snapshots, size-drop guard changes, and focused tests.

## Best Possible Solution

Land a corrected update-preservation fix that snapshots the exact config path doctor will mutate, preserves migratable legacy install records, keeps size-drop protection during update runs, uses bounded or refreshed recovery snapshots, and includes redacted real upgrade proof.

## Reproduction Assessment

Yes at source level: current main can run update-triggered doctor with write support, strip schema-unknown `plugins.installs` before write-time migration, and bypass the size-drop rejection. I did not establish live packaged macOS/npm upgrade proof in this read-only review.

## Solution Assessment

No as submitted. The root-cause direction is sound, but the snapshot must follow the resolved doctor config path, the recovery snapshot needs bounded refresh or retention semantics, and real upgrade proof is still missing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Snapshot the resolved doctor config path:** `src/cli/update-cli/update-command.ts:1031-1033`
  - body: Package-update doctor runs with `resolvePostInstallDoctorEnv`, which can replace `OPENCLAW_CONFIG_PATH` from the managed service profile. The new snapshot call uses module `CONFIG_PATH` before that env is applied, so it can snapshot the caller config while doctor mutates the service config; derive the snapshot path from the same resolved env passed to doctor.
  - confidence: 0.91
- **[P2] Refresh or rotate pre-update snapshots:** `src/config/backup-rotation.ts:128-135`
  - body: The fixed `.pre-update` path is created with `wx` and returns on `EEXIST`, so the first update permanently wins. Later updates cannot record their own pre-update state and stale credential-bearing config remains on disk; replace it per update or use bounded timestamped retention.
  - confidence: 0.88
