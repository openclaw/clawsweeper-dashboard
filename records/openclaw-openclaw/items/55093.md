## Summary

Keep this PR open: current main still lacks both the broad runtime-facade entry coverage and the dirty local version marker, but the PR does not complete the advertised `openclaw --version` behavior because it writes marker metadata without reading or rendering it.

## What This Changes

The PR adds recursive tsdown entries for `src/**/*.runtime.ts`, refactors `scripts/write-build-info.ts` into testable helpers, writes a dirty display marker, and adds focused config/build-info tests.

## Best Possible Solution

Finish the marker contract end to end while preserving the runtime-facade entry coverage: parse typed marker metadata, render it only in human-facing version output, keep commit hash semantics unchanged, add focused tests, and add an active changelog entry.

## Reproduction Assessment

Yes. The reproduction is a static focused check: with build-info containing `displayVersionMarker`, current `src/version.ts` reads only `version` and `src/entry.version-fast-path.ts` formats only version plus optional commit.

## Solution Assessment

No. The runtime entry registration is plausible, but the dirty marker is not the best complete solution until the version metadata and CLI output paths consume it.
