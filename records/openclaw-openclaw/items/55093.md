## Summary

Keep open. The runtime facade stabilization is still useful, but the dirty-version half is incomplete: the PR writes a `displayVersionMarker` into build metadata without wiring any version resolver or `openclaw --version` output path to consume it. Current main also does not already implement the requested marker behavior, so this is not obsolete or implemented on main.

## What This Changes

The PR adds recursive `src/**/*.runtime.ts` tsdown entries, refactors build-info writing into testable helpers that can record a dirty display marker, and adds focused tests for those build metadata and tsdown paths.

## Best Possible Solution

Finish the display-version contract by reading the marker from build metadata through a typed version helper, composing it only into human-facing version output, preserving existing commit hash behavior, and keeping the recursive runtime facade entry coverage with focused tests.

## Reproduction Assessment

Yes. In the PR branch, a build-info file can contain `displayVersionMarker: "dirty"`, but the version resolver still returns only `version` and the root fast path still prints only `VERSION` plus optional commit, so `openclaw --version` remains unchanged.

## Solution Assessment

No. The runtime facade half is a plausible narrow fix, but the dirty marker is only written, not displayed; the safer complete solution is to wire marker reading and formatting through the existing version output paths with tests.
