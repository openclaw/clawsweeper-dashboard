## Summary

Keep open: the Feishu-local pacing shape is useful, but this PR is not merge-ready because the documented config does not cover the default streaming-card send path and the external contributor proof gate is still missing.

## What This Changes

The PR adds opt-in Feishu per-target outbound send pacing through top-level and per-account config and applies it to text, card, image, and file send helpers.

## Best Possible Solution

Route every Feishu `im.message.create` and `im.message.reply` send surface, including streaming-card start, through one per-account/per-target limiter while keeping the default interval at `0` and documenting the exact scope.

## Reproduction Assessment

Yes, at source level: the linked issue describes Feishu throttling from bursty sends, and current/PR code shows which send paths do or do not pass through the new queue. I did not run a live Feishu burst test in this read-only review.

## Solution Assessment

No, not yet. A Feishu-plugin-local pacing helper is the right owner boundary, but the implementation must cover the default streaming-card message create/reply path or explicitly narrow the promised scope.
