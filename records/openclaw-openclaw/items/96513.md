## Summary

Keep open. The PR is a useful improvement over the earlier Feishu pacing attempts, but it is not merge-ready because proof is test-only and the new public config/docs promise outbound message pacing while the default streaming-card send path still bypasses the pacing helper.

## What This Changes

The branch adds opt-in Feishu `sendRateLimit.minIntervalMs` config at top level and per account, documents it, and queues text, card, image, and file sends by account plus Feishu receive target.

## Best Possible Solution

Route the configured Feishu pacing through every intended message create/reply delivery boundary, including streaming-card initial sends, or narrow the public docs/config wording and leave the remaining surfaces tracked by the canonical issue.

## Reproduction Assessment

Yes, at source level: current main has Feishu outbound send surfaces and reactive retry handling, but no proactive minimum-interval scheduler. The PR finding is also source-reproducible because streaming-card initial sends still call Feishu `im.message.create/reply` outside the new queue while the docs claim outbound message pacing.

## Solution Assessment

No, not yet. A Feishu-local pacing helper is the right owner boundary, but this implementation needs either complete coverage of the documented outbound message contract or narrower docs and issue scoping before it is the best fix.
