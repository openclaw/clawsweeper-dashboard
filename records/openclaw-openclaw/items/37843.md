## Summary

Keep open: current main and v2026.5.12 still treat non-empty cron `payload.model` as an explicit model ref, docs still describe strict explicit-model behavior, and the active related implementation at https://github.com/openclaw/openclaw/pull/73727 uses a different `@default` contract with review blockers and merge conflicts.

## What This Changes

Allow cron agent-turn jobs to store `payload.model` as a default/inherit sentinel resolved at run time instead of as a literal explicit model ref.

## Best Possible Solution

Define one public cron model-inheritance contract, normalize that sentinel in both model selection and fallback policy, document precedence, and land or replace the active related PR with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: set an agent-turn cron `payload.model` to `default`, and the code treats it as a non-empty explicit model ref before fallback policy can behave like an omitted model.

## Solution Assessment

No, not as the original one-line guard alone. A maintainable fix needs a settled sentinel spelling plus shared normalization in cron model selection, fallback policy, docs, and tests.
