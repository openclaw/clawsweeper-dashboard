## Summary

Keep open: current main and v2026.5.22 still treat every non-empty cron payload.model as an explicit model ref, and the active implementation path uses a different @default sentinel with product/API review still pending.

## What This Changes

Support a cron agentTurn payload.model sentinel that resolves to the runtime default model instead of storing a fixed model ref.

## Best Possible Solution

Settle one documented model-inheritance sentinel, centralize normalization across cron model selection and fallback policy, preserve strict errors for real explicit model refs, and land focused docs plus regression coverage.

## Reproduction Assessment

Yes for the current behavior: source inspection shows payload.model: "default" is treated as a non-empty explicit model ref. The requested inherited-default semantics are a new public contract rather than an already-defined expected behavior.

## Solution Assessment

No for the original one-line guard alone. A maintainable fix needs a settled sentinel spelling plus shared normalization across cron model selection, fallback policy, docs, and regression tests.
