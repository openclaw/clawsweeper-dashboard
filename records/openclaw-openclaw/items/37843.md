## Summary

Keep open because current main still treats every non-empty cron `payload.model` as a concrete model reference, and the related sentinel implementation is tracked by the still-open PR #73727 rather than shipped. This is a feature/API request, not a bug-fix automation candidate, because current docs define `--model` as an allowed model ref that fails closed when unresolved.

## What This Changes

The issue requests a stored cron job model sentinel so cron payloads can ask for runtime default-model resolution instead of pinning a concrete model.

## Best Possible Solution

Review and land or revise #73727, or an equivalent implementation, so stored cron model overrides can opt into live default resolution with aligned fallback behavior, docs, tests, and changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create or edit an agent-turn cron with `--model default` or `--model @default`, and current main persists then resolves the non-empty value as a literal model ref instead of falling through to defaults.

## Solution Assessment

No for the original one-line guard by itself. The safer solution should normalize the chosen sentinel wherever stored model overrides affect selection and fallback behavior, then document and test the public spelling.
