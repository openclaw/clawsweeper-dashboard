## Summary

Close: current main already solves the actionable stale hardcoded cron model problem through `model: null` and `cron edit --clear-model`; the literal `payload.model: "default"` sentinel was not adopted as the product contract.

## What This Changes

The issue requests support for a stored cron `payload.model` value of `default` that resolves to the runtime default model instead of being treated as a literal model id.

## Best Possible Solution

Keep the canonical clear/unset model workflow as the supported way to return existing cron jobs to runtime model resolution; any future reserved model sentinel should be a separate maintainer-sponsored product decision.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature/API request for a reserved model sentinel. Source inspection confirms the literal sentinel is still not the chosen contract, while current main provides a supported `--clear-model`/`model: null` path for the reported operational problem.

## Solution Assessment

Yes for the current product boundary: clearing the stored model override is the narrower maintainable solution because it uses the existing cron update contract and avoids reserving another model-id string. A literal `default` sentinel would need separate maintainer product approval.
