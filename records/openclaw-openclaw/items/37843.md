## Summary

Keep open: current main and v2026.5.19 still treat any non-empty cron `payload.model` as an explicit model ref, and the remaining work needs a maintained public sentinel contract rather than the original one-line guard.

## What This Changes

Request support for an explicit cron `payload.model` sentinel that resolves to the current default model at runtime instead of pinning a provider/model.

## Best Possible Solution

Define one documented cron model-inheritance sentinel, normalize it consistently in cron model selection and fallback policy, then land the active implementation path with tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: store an agent-turn cron with `payload.model: "default"`, and current main treats it as a non-empty explicit model override rather than inherited default resolution.

## Solution Assessment

No, not as the original one-line guard alone. A maintainable fix needs a settled public sentinel spelling plus shared normalization in cron model selection, fallback policy, docs, and regression tests.
