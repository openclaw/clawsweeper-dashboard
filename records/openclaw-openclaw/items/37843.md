## Summary

Current main now covers the operational problem without adding a literal `default` model alias: users can omit a cron model for new/templates, clear existing stored model pins with `cron edit --clear-model`, and repair bad persisted `default`/`null` sentinels with doctor.

## What This Changes

The issue requests treating cron `payload.model: "default"` as an explicit runtime-default sentinel instead of a literal model id.

## Best Possible Solution

Keep the current explicit model contract: omit the model or use `cron edit --clear-model` for runtime default resolution, and use doctor for legacy bad sentinel cleanup.

## Reproduction Assessment

Yes for the exact sentinel request: current model selection still treats non-empty `payload.model` strings as explicit refs, so literal `default` is not a runtime alias. The broader delete/recreate problem is no longer reproducible on current main because `--clear-model` removes stored model pins.

## Solution Assessment

Yes for current main: the supported clear/omit workflow solves the operational need without adding an ambiguous `default` alias to strict explicit-model validation.
