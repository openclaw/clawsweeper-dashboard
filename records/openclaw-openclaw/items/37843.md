## Summary

Keep open because current main still treats every non-empty cron `payload.model` as a concrete model reference, while the requested runtime-default sentinel remains an unresolved feature/product contract with an open implementation candidate.

## What This Changes

The issue requests a stored cron `payload.model` sentinel such as `default` that resolves through the current runtime default model instead of pinning a literal model reference.

## Best Possible Solution

Land or revise the active sentinel implementation so stored cron model overrides can intentionally follow runtime defaults, with one documented spelling applied consistently across model selection, fallback policy, tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: create or edit an isolated agent-turn cron with `--model default`, and the stored non-empty value is resolved as a literal model reference instead of falling through to defaults.

## Solution Assessment

No for the original one-line guard alone. The maintainable fix should normalize the accepted sentinel anywhere stored cron model overrides affect selection and fallback policy, then document and test the public spelling.
