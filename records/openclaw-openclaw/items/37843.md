## Summary

Keep open: current main and the latest release still treat literal cron `payload.model: "default"` as a non-empty explicit model reference, while the existing null/omitted path only covers clearing or not setting a model. The request remains a product/API decision about an inherit-default sentinel rather than an implemented behavior.

## What This Changes

Request support for a persisted cron job model sentinel, `default`, that defers model resolution to the agent or system default at runtime instead of pinning a provider/model at creation time.

## Best Possible Solution

Define a collision-safe canonical inherit-default contract for persisted cron model overrides, update cron model-selection and fallback policy plus CLI/RPC/docs/tests, and keep strict validation for real explicit model refs.

## Reproduction Assessment

Yes for source-level behavior: a cron `agentTurn` payload with `model: "default"` is non-empty and is resolved as an explicit model ref, not an inheritance sentinel. I did not run a live cron job in this read-only review.

## Solution Assessment

No for the exact one-line bare `default` proposal as a final decision; maintainers should first choose the permanent sentinel or clear-override contract and scope it across cron model selection and fallback handling.
