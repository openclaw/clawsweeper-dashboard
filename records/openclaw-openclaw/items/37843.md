## Summary

Current main and v2026.6.6 still interpret every non-empty cron `payload.model` as an explicit model reference, while omitted or blank values inherit the runtime default. The requested capability remains useful, but it requires a maintainer decision about the public model-routing contract rather than the proposed isolated string exception.

## What This Changes

Add an explicit cron job model value that resolves the configured default model at execution time instead of pinning a provider/model reference.

## Best Possible Solution

Define one collision-resistant, documented inherit-default representation, then apply it coherently to cron add/edit serialization, runtime model precedence, fallback policy, diagnostics, and focused tests; explicitly decide whether sibling stored overrides share the contract.

## Reproduction Assessment

Yes at source level: store an agent-turn cron with `payload.model: "default"`; current main trims the value and resolves it as an explicit allowed model rather than inheriting the configured default. No live cron was run during this read-only review.

## Solution Assessment

No for the proposed one-line bare `default` check: it does not settle collision safety, fallback-policy behavior, the CLI and documentation contract, or sibling stored overrides. Maintainers should first select a canonical sentinel or explicit inherit operation.
