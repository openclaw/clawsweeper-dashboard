## Summary

Current main and the latest release still treat every non-empty cron payload model as an explicit pin, so the requested dynamic-default capability remains missing. Keep this open for a maintainer decision on a collision-safe sentinel or clear-override contract rather than the proposed bare `default` exception.

Likely related people:
- `vincentkoc` — recent area contributor; high confidence.
- `steipete` — heavy model-selection contributor; high confidence.
- `Neerav Makwana` — adjacent model-selection contributor; medium confidence.

## What This Changes

Allow cron jobs to store an explicit model value that resolves to the live configured agent default each time the job runs.

## Best Possible Solution

Add one collision-safe, documented dynamic-default or clear-model operation to cron add/edit and resolve it centrally in `resolveCronModelSelection`, with explicit tests showing it bypasses stored session history while concrete model pins remain unchanged.

## Reproduction Assessment

Yes. Current source shows that `payload.model: "default"` enters explicit model resolution, while omitting the field can still allow a persisted session override to win.

## Solution Assessment

No, the proposed one-line bare-string guard is not the best current solution. The maintainable fix needs a collision-safe public contract in the canonical resolver and CLI, with sibling precedence and documentation covered together.

## Risks / Open Questions

- Reserving the bare string `default` would turn a previously generic model string into a persisted special value and could collide with aliases or future provider model names.
- The contract must explicitly define whether dynamic default bypasses stored session overrides, subagent defaults, agent-specific defaults, and hook-specific defaults.
- The broader https://github.com/openclaw/openclaw/issues/57638 proposes new fleet-wide cron defaults, but it does not replace the distinct need to explicitly opt one stored job back into live agent defaults.
