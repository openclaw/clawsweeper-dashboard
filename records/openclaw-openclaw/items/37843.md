## Summary

Keep open: current main and the latest release still treat non-empty cron `payload.model` values as explicit model refs, so the requested runtime-inheritance sentinel is not implemented. The remaining decision is the public contract and scope, especially `default` versus `@default`, not a stale cleanup close.

## What This Changes

Requests a cron `payload.model` sentinel that lets stored jobs explicitly inherit the runtime default model instead of pinning a provider/model string.

## Best Possible Solution

Settle one documented cron model-inheritance contract, centralize normalization in model selection and fallback policy, and preserve strict errors for real explicit model refs.

## Reproduction Assessment

Yes for the current behavior: source inspection shows `payload.model: "default"` is treated as a non-empty explicit model ref. The requested inherited-default semantics are a new public contract rather than already-defined expected behavior.

## Solution Assessment

No for the original one-line guard alone. A maintainable fix needs a settled sentinel spelling plus shared normalization across cron model selection, fallback policy, docs, and regression tests.
