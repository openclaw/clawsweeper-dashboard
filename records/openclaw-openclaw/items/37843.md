## Summary

Keep open: current main and the latest release still treat cron `payload.model: "default"` as an explicit model ref, while the active implementation path is an open conflicting PR and the sentinel spelling/scope still needs maintainer agreement.

## What This Changes

Request support for a cron `payload.model` sentinel such as `default` that resolves to the runtime default model instead of storing a pinned model ref.

## Best Possible Solution

Approve one sentinel contract, centralize cron payload-model normalization for both selection and fallback policy, update docs/tests, then land or replace the active PR with runtime proof.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.12` gives a high-confidence reproduction path: `payload.model` is trimmed and every non-empty string flows into explicit model resolution, so `default` is treated as a literal model ref.

## Solution Assessment

No for the original one-line guard alone. The maintainable solution needs a settled sentinel spelling plus aligned handling in cron model selection, fallback policy, documentation, and regression coverage.
