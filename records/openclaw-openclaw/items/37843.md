## Summary

Keep open: current main still treats every non-empty cron `payload.model` as an explicit model reference, while the active implementation path is an open conflicting PR and the public sentinel contract still needs maintainer agreement.

## What This Changes

Request support for a stored cron `payload.model` sentinel such as `default` that resolves to the configured default model at runtime.

## Best Possible Solution

Choose and document one sentinel contract, normalize it consistently in cron model selection and fallback policy, add focused regression coverage, then land or replace the active implementation PR.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: set cron `payload.model` to `default` or `@default` and the current code treats it as a non-empty explicit model ref.

## Solution Assessment

No, not for the original one-line guard alone. A maintainable fix needs a settled sentinel spelling plus aligned handling in cron model selection, fallback policy, docs, and tests.
