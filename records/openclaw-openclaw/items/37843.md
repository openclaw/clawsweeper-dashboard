## Summary

Keep open: current `main` still treats literal cron `payload.model: "default"` as an explicit model ref, so the requested sentinel is not implemented. The underlying stale-override pain is partly addressed by current `model: null` / `--clear-model` support, but adding a new stored sentinel value still needs maintainer product/API direction.

## What This Changes

The issue requests `default` as a special cron `agentTurn` `payload.model` value that resolves the agent/default model at run time.

## Best Possible Solution

Maintainers should choose one canonical cron model-inheritance contract; until then, keep the supported omit/null/`--clear-model` behavior and avoid adding a second stored sentinel by automation.

## Reproduction Assessment

Yes. Source inspection shows `payload.model: "default"` is still a non-empty string and therefore follows the explicit override validation path instead of inheriting the runtime default; no tests were run because this review is read-only.

## Solution Assessment

Unclear. The one-line sentinel guard is plausible, but current `main` already supports omit/null/`--clear-model`; adding a new persisted sentinel is a product/API decision rather than a strict bug fix.
