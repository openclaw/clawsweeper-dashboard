- `impact:auth-provider`: The issue concerns cron jobs retaining or resolving provider/model choices after default model or auth-provider changes.

## Summary

Current main solves the operational problem without accepting literal `default` as a runtime alias: users can clear a stored cron model override through `cron edit --clear-model`, the update API supports `model: null`, and runtime model selection remains intentionally strict for non-empty `payload.model` strings.

## What This Changes

The issue requests support for `payload.model: "default"` as an explicit cron sentinel that resolves to the configured default model at run time.

## Best Possible Solution

Keep the current explicit model contract: omit or clear the cron model override for default resolution, and do not add a literal `default` runtime alias unless maintainers later choose that product direction.

## Reproduction Assessment

Yes for the literal sentinel request: current source still treats non-empty `payload.model` strings as explicit model refs, so `default` is not an alias. The broader delete-and-recreate problem is solved on current main by clearing the override instead.

## Solution Assessment

Yes for the operational problem. The current clear/omit workflow is narrower and matches the repository's canonical runtime-config policy better than adding a second magic model string.
