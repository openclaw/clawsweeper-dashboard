## Summary

Keep open: current main still treats cron `payload.model: "default"` as a literal explicit model ref, the latest release has the same strict path, and the active related implementation uses a different `@default` contract with review blockers and merge conflicts.

## What This Changes

The issue requests a cron `payload.model: "default"` value that resolves to the runtime default model instead of persisting a hardcoded model ref.

## Best Possible Solution

Settle one public cron model-inheritance contract, normalize it in both model selection and fallback policy, document precedence, and land or replace the active related PR with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: set an agent-turn cron `payload.model` to `default`, and the code treats it as a non-empty explicit model ref before fallback policy can behave like an omitted model.

## Solution Assessment

No, not as the original one-line guard alone. A maintainable fix needs a settled sentinel spelling plus shared normalization in cron model selection, fallback policy, docs, and tests.
