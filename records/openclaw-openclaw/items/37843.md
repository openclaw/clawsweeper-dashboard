## Summary

Keep open: current main still treats cron `payload.model: "default"` as a literal explicit model reference, and the latest shipped tag shows the same strict path. A merged doctor-only repair removes bad persisted sentinels but intentionally keeps runtime validation strict, while the active implementation PR uses a different `@default` contract and still needs maintainer API-contract review.

## What This Changes

The issue requests a special cron `payload.model` value that resolves to the current default model at run time instead of storing a pinned model reference.

## Best Possible Solution

Settle one public cron model-inheritance contract, normalize it in both model selection and fallback policy, document the precedence, add regression coverage, and then land or replace the active related PR.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: set an agent-turn cron `payload.model` to `default`, and the code treats it as a non-empty explicit model ref before fallback policy can behave like an omitted model.

## Solution Assessment

No, not as the original one-line guard alone. A maintainable solution needs a settled sentinel spelling plus shared normalization in cron model selection, fallback policy, docs, and tests so it does not conflict with strict explicit-model validation.
