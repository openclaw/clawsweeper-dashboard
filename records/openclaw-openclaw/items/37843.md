## Summary

Keep open: current main and the latest release still treat a non-empty cron `payload.model` as an explicit model ref, while the requested runtime-default sentinel is a new public contract with an open but unresolved implementation path.

## What This Changes

The issue requests a cron `payload.model` sentinel value such as `default` so stored cron jobs resolve the active default model at runtime instead of pinning a provider/model ref.

## Best Possible Solution

Settle one documented cron model-inheritance contract, centralize normalization across model selection, fallback policy, docs, and tests, then land the related implementation or a narrow replacement after review.

## Reproduction Assessment

Yes for current behavior: source inspection shows `payload.model: "default"` is treated as a non-empty explicit model ref. The requested inherited-default semantics are a new public contract rather than already-defined expected behavior.

## Solution Assessment

No for the original one-line guard alone. A maintainable fix needs a settled sentinel spelling plus shared normalization across cron model selection, fallback policy, docs, and regression tests.
