## Label Justifications

- `P2`: The issue concerns cron model/provider selection ergonomics with limited blast radius rather than an urgent runtime outage.
- `impact:auth-provider`: The affected behavior is model choice and provider routing for scheduled agent turns.

## Summary

Current main solves the central cron-model drift problem through the supported clear/omit contract: users can remove a stored per-job model override with `cron edit --clear-model`, after which cron resolves through normal model precedence at run time. The literal `payload.model: "default"` alias is still not the chosen contract, but keeping strict explicit-model validation plus a clear operation matches current code, docs, tests, and repository configuration policy.

## What This Changes

Request support treating `payload.model: "default"` in agentTurn cron jobs as runtime default-model resolution.

## Best Possible Solution

Keep the current explicit contract: omit the job model or use `cron edit --clear-model` for runtime default resolution, and reopen only if maintainers want a literal model sentinel as a separate product decision.

## Reproduction Assessment

Yes for the exact sentinel request: source inspection shows non-empty `payload.model` strings are still explicit model refs, so literal `default` is not a runtime alias. The broader operational problem is addressed on current main by clearing or omitting the stored model override.

## Solution Assessment

Yes for closing this issue: current main uses a narrower explicit clear operation instead of adding an ambiguous sentinel to model validation. A literal `default` value would be a separate product/API decision, not a missing bug fix.
