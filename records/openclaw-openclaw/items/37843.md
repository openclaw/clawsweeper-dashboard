## Summary

Keep open: current main and the latest shipped release still treat non-empty cron `payload.model` values as explicit model refs, and the remaining work needs a settled public sentinel contract rather than another runtime-only guard.

## What This Changes

The issue requests a cron `payload.model` sentinel that lets stored cron jobs resolve the current default model at runtime instead of pinning a provider/model string.

## Best Possible Solution

Define one documented cron model-inheritance contract, normalize it consistently in cron model selection and fallback policy, and resolve the active related implementation path before landing tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: store an agent-turn cron with `payload.model: "default"`, and current main treats it as a non-empty explicit model override rather than inherited default resolution.

## Solution Assessment

No, not as the original one-line guard alone. A maintainable fix needs a settled public sentinel spelling and shared normalization in model selection, fallback policy, docs, and regression tests.
