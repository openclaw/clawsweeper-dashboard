## Summary

Current main solves the actionable problem behind this issue through the supported unset-model workflow: `cron.update` can clear `payload.model` with `model: null`, and `cron edit --clear-model` now exposes that path from the CLI. The literal `payload.model: "default"` sentinel was not added, but users no longer need to delete and recreate hardcoded cron jobs to return them to runtime model resolution.

## What This Changes

The issue requests support for `default` as a special cron `payload.model` value that resolves to the runtime default model instead of being treated as a literal model id.

## Best Possible Solution

Keep the canonical clear/unset model workflow as the supported way to return existing cron jobs to runtime model resolution; any future reserved model sentinel should be a separate maintainer-sponsored product decision.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API request for a reserved model sentinel. Source inspection confirms the literal sentinel is still not the chosen contract, while current main provides a supported `--clear-model`/`model: null` path for the reported operational problem.

## Solution Assessment

Yes for the current product boundary: clearing the stored model override is the narrower maintainable solution because it uses the existing cron update contract and avoids reserving another model-id string. A literal `default` sentinel would need separate maintainer product approval.
