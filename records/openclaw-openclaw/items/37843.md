## Summary

Keep open because current main still treats every non-empty cron `payload.model` value as a concrete model override, and the related sentinel implementation is still tracked in the open PR #73727 rather than shipped.

## What This Changes

The issue requests a cron model sentinel so stored cron jobs can explicitly resolve the model from the current default at runtime instead of pinning a concrete model.

## Best Possible Solution

Land or revise #73727, or an equivalent shared helper, so stored cron model overrides can opt into live default resolution while explicit model pins, fallback policy, docs, tests, and changelog stay aligned.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create or edit an agent-turn cron with `--model default`, and current main treats the stored non-empty value as a literal model ref instead of falling through to defaults.

## Solution Assessment

No for the original one-line guard by itself. The safer solution should normalize the sentinel wherever stored model overrides affect selection and fallback behavior, then document and test the chosen public spelling.
