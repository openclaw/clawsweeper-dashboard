## Summary

Keep open: the report matches current `main` behavior for `corpus=all`, where memory and wiki hits are fetched but then globally sorted by incomparable raw scores before applying `maxResults`, allowing wiki hits to starve memory hits.

## What This Changes

The issue asks for `memory_search corpus=all` and `wiki_search corpus=all` to return merged results from both memory and wiki when both corpora have matches.

## Best Possible Solution

Normalize or diversify the `corpus=all` merge in both shared memory and wiki search so matching memory hits are not sliced out solely because wiki and memory scores use different scales.

## Reproduction Assessment

Yes, by source inspection: any `corpus=all` call where wiki results fill `maxResults` with higher raw scores will slice out memory hits even when memory search returned matches. I did not run the reporter's private vault, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes, the best fix is in the shared merge/ranking path, not in a workaround or a new user option. Add regression coverage for bounded `maxResults` with both corpora returning hits and wiki scores higher than memory scores.
