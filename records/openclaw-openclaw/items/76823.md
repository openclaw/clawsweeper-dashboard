## Summary

Keep open: current main still preserves fresh live-PID session locks during doctor repair, and the open closing PR #76854 is the right item to review or replace before this issue closes.

## What This Changes

The issue asks for `openclaw doctor --fix` to clean session lock files when the Gateway is down even if the recorded lock PID is alive.

## Best Possible Solution

Review and land or replace #76854 with a conservative owner-check fix that cleans proven non-OpenClaw false-live locks without deleting active or unknown OpenClaw locks.

## Reproduction Assessment

Yes. Source inspection shows a recent `.jsonl.lock` with an alive PID gets no stale reason and is not removed by `cleanStaleLockFiles({ removeStale: true })`; current tests also assert a fresh live lock remains.

## Solution Assessment

No for the literal Gateway-down force cleanup because it is too broad. The narrower owner-verification approach in #76854 is the maintainable fix boundary.
