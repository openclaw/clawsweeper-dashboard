## Summary

Keep open: current main still preserves fresh live-PID session locks during doctor repair, and the open closing PR #76854 already tracks the narrow false-live owner-check fix.

## What This Changes

The issue requests doctor/session-lock cleanup for fresh `.jsonl.lock` files whose recorded PID is alive but belongs to a different non-OpenClaw process.

## Best Possible Solution

Land a conservative owner-verification fix, like #76854, that marks readable non-OpenClaw live PIDs stale while preserving active or unknown OpenClaw locks.

## Reproduction Assessment

Yes. Source inspection on current main shows a fresh live-PID lock gets no stale reason and `cleanStaleLockFiles({ removeStale: true })` removes only stale locks; current tests also assert a fresh live lock remains.

## Solution Assessment

No for the literal gateway-down force-clean proposal, because it is too broad. The narrower owner-verification approach in #76854 is the maintainable fix boundary.
