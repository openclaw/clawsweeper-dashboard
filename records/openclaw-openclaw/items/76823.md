## Summary

Keep open: current main still preserves the reported fresh live-PID lock path, and open PR #76854 is already paired with this issue as the closing implementation candidate.

## What This Changes

The issue requests that `openclaw doctor --fix` clean blocking session lock files when the gateway is down even if the recorded PID is still alive.

## Best Possible Solution

Review and land PR #76854 or an equivalent owner-aware fix that removes proven non-OpenClaw false-live locks while preserving active or unknown OpenClaw writers.

## Reproduction Assessment

Yes, from source inspection: a recent `.jsonl.lock` with an alive PID is reported but not removed because `stale` remains false and cleanup only unlinks stale locks. I did not run a mutating live `doctor --fix` command in this read-only review.

## Solution Assessment

No for the issue's literal suggested fix: deleting all live-PID locks whenever the gateway is down is too broad. The maintainable solution is an owner-aware cleanup that proves the lock belongs to a non-OpenClaw process before removal.
