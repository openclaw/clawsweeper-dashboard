## Summary

Keep this PR open: the linked bug is real, but the proposed fix maps `eu` and `us` to the global Vertex host instead of the current Google multi-region REP hostnames, so it is not merge-ready. The supplied proof is a standalone host-selection snippet, not proof that the patched OpenClaw path or a live Vertex call now works.

## What This Changes

The PR changes the Google Vertex transport and Anthropic Vertex provider catalog so `global`, `eu`, and `us` all resolve to `https://aiplatform.googleapis.com` instead of a location-prefixed Vertex host.

## Best Possible Solution

Use one documented Vertex host-selection rule for both Google and Anthropic Vertex paths: `global` stays on `aiplatform.googleapis.com`, `eu/us` use the REP hostnames, regional locations keep the location-prefixed host, and focused tests cover all four cases.

## Reproduction Assessment

Yes, source-level: current main sends `eu/us` through the location-prefixed fallback, and the linked report shows that produces Google HTML 404s. I did not run a live Vertex call, but current code plus Google's endpoint contract is enough to reproduce the URL construction fault.

## Solution Assessment

No. The PR is a plausible shape but not the best fix because current Google docs and the pinned Google SDK use REP hostnames for `eu/us`, not the global host; the narrow maintainable fix is a shared or consistently duplicated REP-host rule with tests.
