## Summary

Keep this PR open, but do not merge it as written: the reported Vertex multi-region bug is real, yet this branch routes `eu`/`us` to the global host while Google’s current docs and `@google/genai` route those locations to `.rep.googleapis.com` multi-region hosts. The proof is also mock/unit-only and the PR body says no live Vertex call was tested.

## What This Changes

The PR adds `eu`/`us` multi-region handling to the Google Vertex and Anthropic Vertex URL builders and adds mocked Google Vertex transport tests expecting those locations to use the unprefixed global host.

## Best Possible Solution

Align the provider-owned endpoint mapping with Google’s contract: `global` uses `aiplatform.googleapis.com`, `eu`/`us` use `aiplatform.<location>.rep.googleapis.com`, regional locations keep `<region>-aiplatform.googleapis.com`, and the host classifiers/manifest/tests/proof move with that change.

## Reproduction Assessment

Yes, source-backed: current main builds `eu-aiplatform.googleapis.com` and `us-aiplatform.googleapis.com` for `GOOGLE_CLOUD_LOCATION=eu/us`, matching the linked issue’s failure mode. I did not run a live Vertex request because credentials were not available in this read-only review.

## Solution Assessment

No. This PR is a plausible mitigation for the 404, but the best fix is the official REP multi-region host mapping plus sibling host-recognition and attribution updates, not treating `eu`/`us` as the global endpoint.
