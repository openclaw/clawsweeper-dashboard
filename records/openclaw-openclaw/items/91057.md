## Summary

This PR addresses a reproduced session-store growth bug and includes convincing built-CLI proof, but it needs an explicit maintainer decision before merge because it turns an already-defined ephemeral one-shot probe contract into a permanent public retention setting with enabled-by-default upgrade behavior. Likely related people: Kaspre for the model-run isolation contract, gumadeiras for session maintenance, and steipete for recent session-store work.

## What This Changes

The PR adds `session.maintenance.modelRunPruneAfter`, defaults it to 24 hours, prunes strict agent-scoped model-run UUID sessions during write, load, and cleanup maintenance, and documents and tests the new reporting.

## Best Possible Solution

Clean up future gateway model-run sessions at their terminal lifecycle boundary and keep an audited cleanup path for already-accumulated strict UUID rows; retain configurable probe history only if maintainers explicitly decide it is a supported diagnostic product contract.

## Reproduction Assessment

Yes. Current main persists a fresh explicit model-run session for each gateway probe, and the linked production evidence shows hundreds of those rows consuming nearly the entire global session cap.

## Solution Assessment

No, not without a product decision. The TTL implementation works, but lifecycle-owned deletion follows the existing ephemeral one-shot contract more directly and avoids adding a permanent gateway-specific config/default to generic maintenance.
