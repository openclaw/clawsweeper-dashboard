## Summary

Keep open. The Slack hook coverage gap is real and this PR points in the right direction, but the draft branch currently combines that fix with a bundled guard that has correctness and credential-scope blockers before merge.

## What This Changes

This PR wires Slack reply delivery through `message_sending` hooks and adds a bundled `slack-addressee-guard` plugin, manifest, and tests.

## Best Possible Solution

Land the Slack reply hook wiring as a focused change with coverage for text, media, blocks, streaming fallback, and preview/final edit paths; handle the addressee guard separately as an account-aware official plugin or a ClawHub/plugin follow-up.

## Reproduction Assessment

Yes. Current main reproduces the gap by source inspection: Slack reply delivery calls `sendMessageSlack()` directly, while the generic `message_sending` hook insertion lives in shared dispatcher wrappers that this Slack delivery path bypasses.

## Solution Assessment

No. The hook wiring is the narrow maintainable fix, but the combined PR is not the best complete solution until the bundled guard's leading-mention behavior, Slack ID support, credential scope, metadata, labeler, and changelog gaps are resolved.

## Security Review

Status: needs_attention

Summary: The diff needs security attention because the new bundled guard reads Slack thread history with a default environment token instead of the credential for the Slack delivery account.

Concerns:

- **[medium] Use account-scoped Slack credentials:** `extensions/slack-addressee-guard/index.ts:87`
  - body: The guard resolves only `SLACK_BOT_TOKEN` or `slackTokenOverride` and ignores `ctx.accountId` plus configured Slack account credentials. In multi-account or SecretRef-only installs, it can query `conversations.replies` with the wrong workspace credential or fail open even though the sending account is configured.
  - confidence: 0.84
