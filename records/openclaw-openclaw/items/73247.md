## Summary

Keep open. Current main still has the Slack agent-reply message_sending hook gap, so this PR is not obsolete; however the draft patch is not merge-ready because the bundled guard has concrete functional defects and an account-scoped Slack credential security concern.

## What This Changes

The PR wires Slack reply delivery through message_sending hooks and adds a bundled slack-addressee-guard plugin with tests.

## Best Possible Solution

Split the work: land a focused Slack hook-coverage fix that matches existing message_sending semantics and tests text, blocks, media, blank rewrites, streaming fallback, and preview finalization; handle slack-addressee-guard separately as either a ClawHub/external plugin or an official bundled plugin only after credential scope, Slack conversation id coverage, leading-mention matching, docs, config, and labeler parity are hardened.

## Reproduction Assessment

Yes. Source inspection on current main is a high-confidence reproduction for the underlying hook gap: Slack deliverNormally() calls deliverReplies(), and deliverReplies() calls sendMessageSlack() directly without runMessageSending(). The PR-introduced guard defects are reproducible with unit-level inputs such as a non-leading target mention and G/D-prefixed Slack conversation ids.

## Solution Assessment

No. Wiring message_sending into Slack reply delivery is the right narrow direction, but this PR is not the best complete solution while it combines that fix with a bundled guard that has parser, blank-rewrite, unused-config, and credential-scope problems.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Only rewrite leading target mentions:** `extensions/slack-addressee-guard/index.ts:324`
  - body: The guard enters whenever the target user is mentioned anywhere, but repairContent() prepends the prior human when the target mention is not leading. A normal reply like `I checked with <@target>` would be rewritten even though it is not the accidental leading-address case described by the plugin; gate this path on a leading target mention before fetching or rewriting.
  - confidence: 0.9
- **[P2] Accept private Slack conversation ids:** `extensions/slack-addressee-guard/index.ts:115-116`
  - body: parseChannelAndThread() rejects every Slack conversation id that does not start with `C`, so private channels, MPIMs, and DMs using `G...` or `D...` silently skip the guard even though Slack reply delivery and conversations.replies can operate on those surfaces. Accept the Slack conversation id prefixes the channel supports.
  - confidence: 0.86
- **[P2] Preserve blank-hook suppression:** `extensions/slack/src/monitor/replies.ts:149`
  - body: If a message_sending hook rewrites a text-only reply to whitespace, this path passes the blank string to sendMessageSlack(), which throws when there is no text, blocks, or media. Existing outbound delivery treats blanked hook content as no send, so normalize and skip blank hook results here as well.
  - confidence: 0.82
- **[P3] Remove the unused logPath setting:** `extensions/slack-addressee-guard/openclaw.plugin.json:20`
  - body: The manifest exposes `logPath` and promises optional JSONL decision logging, but the plugin implementation never reads the setting or writes those records. Either implement that logging behavior or remove the config field before shipping the bundled plugin.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The hook wiring adds no dependency source, but the bundled guard queries Slack with a process-wide token instead of the account-scoped Slack credential for the message being delivered.

Concerns:

- **[medium] Use account-scoped Slack credentials:** `extensions/slack-addressee-guard/index.ts:75`
  - body: The guard resolves only process.env.SLACK_BOT_TOKEN and then calls conversations.replies with that token. In multi-account or SecretRef-only installs, an official bundled plugin could query the wrong Slack workspace or fail open even though the Slack account for ctx.accountId is configured.
  - confidence: 0.86
