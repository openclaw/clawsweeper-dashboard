## Summary

Keep open. Current main still has the source-reproducible Slack reply hook gap, but this draft PR is not merge-ready because the bundled guard has correctness, metadata, changelog, and account-scoped credential blockers.

## What This Changes

The PR wires Slack `deliverReplies` through `message_sending` hooks and adds a bundled `slack-addressee-guard` plugin, manifest, and tests.

## Best Possible Solution

Land the Slack reply hook wiring as a focused, tested fix, and split or rework the addressee guard as an account-aware official plugin or ClawHub follow-up with metadata, docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows Slack agent replies call `deliverReplies()` and then `sendMessageSlack()` directly, while `message_sending` is only installed in other dispatcher/outbound paths.

## Solution Assessment

No. The hook wiring is the narrow maintainable fix, but the combined PR is not the best complete solution until the bundled guard is limited to leading mentions, supports all Slack conversation IDs, uses account-scoped credentials, and includes required metadata/changelog updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Limit the guard to leading target mentions:** `extensions/slack-addressee-guard/index.ts:351`
  - body: The guard enters when the configured target is mentioned anywhere, but `repairContent()` prepends the last human if the target mention is not leading. A normal reply such as `I checked with <@target>` would be rewritten even though the plugin is described as protecting accidental leading addresses.
  - confidence: 0.9
- **[P2] Accept Slack DM and private-channel IDs:** `extensions/slack-addressee-guard/index.ts:130`
  - body: `parseChannelAndThread()` rejects every conversation ID that does not start with `C`, so DMs, MPIMs, and private channels using Slack `D...` or `G...` IDs silently skip the guard even though Slack reply delivery can target those surfaces.
  - confidence: 0.87
- **[P3] Remove or implement the logPath setting:** `extensions/slack-addressee-guard/openclaw.plugin.json:20`
  - body: The manifest exposes `logPath` and the UI help promises JSONL decision logging, but the implementation never reads the setting or writes decision records. Either implement the logging behavior or remove the config field before shipping this bundled plugin.
  - confidence: 0.88
- **[P3] Add routing labels for the new plugin:** `extensions/slack-addressee-guard/openclaw.plugin.json:2`
  - body: This adds a new bundled plugin under `extensions/slack-addressee-guard/**`, but the labeler only maps `extensions/slack/**` to Slack changes. Add a plugin or Slack label mapping so future changes are routed consistently.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `extensions/slack/src/monitor/replies.ts:2`

## Security Review

Status: needs_attention

Summary: The new bundled guard has a concrete account-scoping credential risk before it can ship.

Concerns:

- **[medium] Use account-scoped Slack credentials:** `extensions/slack-addressee-guard/index.ts:87`
  - body: The guard resolves only `SLACK_BOT_TOKEN` or `slackTokenOverride` and ignores `ctx.accountId` plus configured Slack account credentials. In multi-account or SecretRef-only installs, it can query `conversations.replies` with the wrong workspace credential or fail open while the sending account is correctly configured.
  - confidence: 0.86
