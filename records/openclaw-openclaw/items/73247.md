## Summary

Keep this PR open for maintainer review: the Slack `message_sending` coverage gap is source-reproducible on current main and the branch is plausible, but the proposed patch has blocking behavior bugs, is draft/conflicting, and lacks required real Slack proof.

## What This Changes

The PR wires Slack agent replies through the `message_sending` hook and adds a bundled `slack-addressee-guard` plugin with tests and manifest metadata.

## Best Possible Solution

Split or revise this into a narrow Slack reply hook-coverage fix that emits stable channel/thread fields and tests the real addressee-guard event shape; handle the guard itself as a maintainer-approved bundled plugin or external ClawHub-style plugin after proof.

## Reproduction Assessment

Yes. Source inspection on current main shows Slack agent replies call `sendMessageSlack` directly while the documented `message_sending` hook can rewrite or cancel outbound content; I did not run live Slack proof in this read-only review.

## Solution Assessment

No. The current PR direction needs fixes before merge: the hook event must provide the channel id shape its guard consumes, the guard should only act on leading target mentions, and bundled-plugin scope needs maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass the Slack channel id to hooks:** `extensions/slack/src/monitor/replies.ts:81`
  - body: The addressee guard parser reads `metadata.channel` before `ctx.conversationId` and rejects non-Slack channel ids, but this hook event sets `metadata.channel` to `"slack"` and never supplies `metadata.channelId`. That makes the new guard return before fetching the thread, so it does not run on the agent-reply path this PR is trying to protect.
  - confidence: 0.93
- **[P2] Limit the guard to leading target mentions:** `extensions/slack-addressee-guard/index.ts:137`
  - body: `contentMentionsTarget` returns true for any `<@target>` occurrence. If the target is mentioned later in a normal sentence, the handler still fetches thread state and `repairContent` can prepend the last human, changing messages that are not leading addressee mistakes.
  - confidence: 0.86
