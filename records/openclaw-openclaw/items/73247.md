## Summary

Keep open for maintainer review: the PR targets a real Slack hook-coverage gap, but it is draft/conflicting, lacks inspectable real behavior proof, and the patch has blocking correctness and account-scoping issues.

## What This Changes

The PR wraps Slack `deliverReplies()` sends with `message_sending` hooks and adds a bundled `slack-addressee-guard` plugin with tests and manifest metadata.

## Best Possible Solution

Land a focused hook fix that covers Slack normal and native-streamed reply delivery without double-firing, then split the addressee guard into an account-aware plugin or ClawHub follow-up with complete metadata.

## Reproduction Assessment

Yes, from source inspection. Current Slack reply delivery still has same-channel and native-streamed paths that do not reliably pass through a `message_sending` hook-bearing seam on current `main`.

## Solution Assessment

No. The hook direction is useful, but this patch only wraps `deliverReplies()`, misses native streaming, and bundles a product-specific guard that needs account scoping and likely a split decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cover native streamed replies with hooks:** `extensions/slack/src/monitor/replies.ts:48-49`
  - body: Slack defaults to partial native streaming, and that path sends through `startSlackStream()`/stream append handling in the message handler instead of this `deliverReplies()` wrapper. Wrapping only `deliverReplies()` leaves the default streamed agent reply path invisible to `message_sending`, so guard or policy plugins can still miss visible replies.
  - confidence: 0.87
- **[P2] Resolve Slack credentials for the sending account:** `extensions/slack-addressee-guard/index.ts:294`
  - body: The guard reads only `slackTokenOverride` or `SLACK_BOT_TOKEN`, but the actual Slack send is account-scoped and can use configured account credentials or SecretRefs. Multi-account installs can query the wrong workspace, or fail open for the right one, while the message is sent with a different token.
  - confidence: 0.88
- **[P2] Require a leading target mention before rewriting:** `extensions/slack-addressee-guard/index.ts:351`
  - body: The guard starts when the configured target is mentioned anywhere, but `repairContent()` prepends the last human when that mention is not leading. A normal reply like `I checked with <@target>` would be rewritten even though the plugin is described as protecting accidental leading addresses.
  - confidence: 0.9
- **[P2] Accept Slack DM and private-channel IDs:** `extensions/slack-addressee-guard/index.ts:130-131`
  - body: `parseChannelAndThread()` rejects every conversation ID that does not start with `C`, so DMs, MPIMs, and private channels using Slack `D...` or `G...` IDs silently skip the guard even though Slack send and routing support those surfaces.
  - confidence: 0.86
- **[P3] Remove or implement the logPath setting:** `extensions/slack-addressee-guard/openclaw.plugin.json:20`
  - body: The manifest exposes `logPath` and says it controls JSONL decision logging, but the implementation never reads that setting or writes decision records. Either implement the promised logging or remove the field before shipping the bundled plugin.
  - confidence: 0.88
- **[P3] Add routing labels for the new Slack plugin:** `extensions/slack-addressee-guard/openclaw.plugin.json:2`
  - body: This adds a new bundled Slack-related plugin under `extensions/slack-addressee-guard/**`, but current labeler rules only map Slack changes under `extensions/slack/**`. Add a Slack or plugin-specific mapping so future changes route consistently.
  - confidence: 0.82
