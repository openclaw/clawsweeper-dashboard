## Summary

Keep open. Current `main` still has the Slack reply hook coverage gap and this PR targets real missing hook coverage, but the combined branch is not merge-ready because the new bundled guard introduces concrete behavior and credential-scope problems.

## What This Changes

The PR routes Slack `deliverReplies()` through `message_sending` hooks and adds a bundled `slack-addressee-guard` plugin with manifest and tests.

## Best Possible Solution

Land a focused Slack reply hook fix, then split or redesign the addressee guard as an account-aware plugin with complete metadata, changelog, routing, and tests.

## Reproduction Assessment

Yes. Source inspection on current `main` shows Slack agent replies call `deliverReplies()` and then `sendMessageSlack()` directly, while `message_sending` hook wiring exists in other outbound or shared auto-reply paths.

## Solution Assessment

No. The hook wiring is the narrow maintainable fix, but the combined PR is not the best complete solution until the bundled guard is limited to leading mentions, supports all Slack conversation IDs, uses account-scoped credentials, and includes required metadata/changelog updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Limit the guard to leading target mentions:** `extensions/slack-addressee-guard/index.ts:351`
  - body: The guard enters whenever the target mention appears anywhere, then `repairContent()` can prepend the last human before a non-leading mention. A normal reply such as `I checked with <@target>` would be rewritten even though the plugin is described as guarding accidental leading addresses.
  - confidence: 0.9
- **[P2] Accept Slack DM and private-channel IDs:** `extensions/slack-addressee-guard/index.ts:130`
  - body: `parseChannelAndThread()` rejects every conversation ID that does not start with `C`, so DMs, MPIMs, and private channels using Slack `D...` or `G...` IDs silently skip the guard even though Slack reply delivery can target those surfaces.
  - confidence: 0.87

  - body: The manifest exposes `logPath` and the UI help promises JSONL decision logging, but the implementation never reads the setting or writes decision records. Either implement the logging behavior or remove the config field before shipping this bundled plugin.
  - confidence: 0.88

  - body: This adds a new bundled plugin under `extensions/slack-addressee-guard/**`, but the labeler only maps `extensions/slack/**` to Slack changes. Add a plugin or Slack label mapping so future changes route consistently.

## Security Review

Status: needs_attention

Summary: The new bundled guard has a concrete Slack credential scoping risk before it can ship.

Concerns:

- **[medium] Use account-scoped Slack credentials:** `extensions/slack-addressee-guard/index.ts:87`
  - body: The guard resolves only `SLACK_BOT_TOKEN` or `slackTokenOverride` and ignores `ctx.accountId` plus configured Slack account credentials. In multi-account or SecretRef-only installs, it can query `conversations.replies` with the wrong workspace credential or fail open while the sending account is correctly configured.
