## Summary

Keep open. Current main still has the Slack reply `message_sending` coverage gap, but this draft PR is not merge-ready because the bundled guard can rewrite non-leading mentions, skips Slack DM/private-channel conversations, and uses unscoped Slack env credentials.

## What This Changes

The PR routes Slack agent reply delivery through `message_sending` hooks and adds a bundled `slack-addressee-guard` plugin with config, API barrel, and tests.

## Best Possible Solution

Land a focused Slack reply hook fix with regression tests, then handle the guard as an account-aware official plugin or separate ClawHub/product follow-up with docs, changelog, and label routing.

## Reproduction Assessment

Yes. Source inspection on current main shows Slack message-handler dispatch calls `deliverReplies()` for agent replies, and `deliverReplies()` calls `sendMessageSlack()` directly without `runMessageSending()`.

## Solution Assessment

No. The hook wiring is the narrow maintainable fix, but bundling the new guard in the same draft is not the best complete solution until mention scope, Slack conversation IDs, credential scoping, metadata, and release notes are resolved.

Overall confidence: 0.88

Full review comments:

- **[P2] Limit the guard to leading target mentions:** `extensions/slack-addressee-guard/index.ts:351`
  - body: The guard currently runs whenever the configured target appears anywhere in the reply. A normal reply like `I checked with <@target>` would be rewritten by prepending the last human even though the plugin is described as protecting accidental leading addresses.
  - confidence: 0.9
- **[P2] Accept Slack DM and private-channel IDs:** `extensions/slack-addressee-guard/index.ts:130`
  - body: `parseChannelAndThread()` rejects every conversation ID that does not start with `C`, so DMs and private/group conversations using Slack `D...` or `G...` IDs silently skip the guard even though Slack reply delivery supports those surfaces.
  - confidence: 0.88
- **[P2] Skip text replies blanked by hooks:** `extensions/slack/src/monitor/replies.ts:151`
  - body: If a `message_sending` hook rewrites a text-only Slack reply to an empty or whitespace string, this path still calls `sendMessageSlack()`, which throws because Slack sends require text, blocks, or media. Match the existing outbound behavior by treating blank text-only hook output as no delivery.
  - confidence: 0.82

  - body: The manifest exposes `logPath` and promises JSONL decision logging, but the implementation never reads the setting or writes decision records. Either implement that behavior or remove the config field before shipping the plugin.
  - confidence: 0.88

  - body: This adds a new bundled plugin under `extensions/slack-addressee-guard/**`, but the labeler only maps Slack changes under `extensions/slack/**`. Add label routing so future changes are classified consistently.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces an account-scoping credential risk in the new bundled Slack guard.

Concerns:

- **[medium] Use account-scoped Slack credentials:** `extensions/slack-addressee-guard/index.ts:87`
  - body: The guard resolves only `SLACK_BOT_TOKEN` or `slackTokenOverride` and ignores `ctx.accountId` plus configured Slack account credentials. In multi-account or SecretRef-backed installs it can query `conversations.replies` with the wrong workspace credential or fail open while the sending account is correctly configured.
  - confidence: 0.86
