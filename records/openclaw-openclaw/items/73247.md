## Summary

Keep open: the Slack `message_sending` coverage gap is still present on current `main`, but this draft PR is incomplete and still has correctness, security, metadata, and real-behavior-proof blockers.

## What This Changes

The branch wraps Slack `deliverReplies()` sends with `message_sending` hook calls and adds a bundled `slack-addressee-guard` plugin with tests.

## Best Possible Solution

Land a focused Slack hook fix that covers both normal and native-streamed reply delivery, then split or rework the guard as account-aware bundled plugin work or a ClawHub/plugin follow-up.

## Reproduction Assessment

Yes, by source inspection: current `main` sends normal Slack replies directly through `sendMessageSlack()` and default threaded text replies through `startSlackStream()` without `message_sending`. I did not run a live Slack workspace reproduction.

## Solution Assessment

No. The hook-wiring direction is useful, but the PR is not the best merge shape until native streaming is covered and the guard is either split or made account-aware, leading-mention-scoped, and metadata-complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cover native streaming replies with hooks:** `extensions/slack/src/monitor/replies.ts:48-49`
  - body: Slack defaults to partial native streaming for threaded replies, and that path calls `startSlackStream()` from the message handler instead of `deliverReplies()`. Wrapping only `deliverReplies()` leaves the default streamed reply path invisible to `message_sending`, so guard/thread-ownership plugins can still miss agent replies.
  - confidence: 0.9
- **[P2] Limit the guard to leading target mentions:** `extensions/slack-addressee-guard/index.ts:351`
  - body: The guard starts when the configured target is mentioned anywhere, then `repairContent()` prepends the last human if the target mention is not leading. A normal reply like `I checked with <@target>` would be rewritten even though the plugin is described as protecting accidental leading addresses.
  - confidence: 0.9
- **[P2] Accept Slack DM and private-channel IDs:** `extensions/slack-addressee-guard/index.ts:130-131`
  - body: `parseChannelAndThread()` rejects every conversation ID that does not start with `C`, so Slack DMs, MPIMs, and private channels using `D...` or `G...` IDs silently skip the guard even though Slack send/routing supports those surfaces.
  - confidence: 0.88
- **[P2] Resolve Slack credentials for the sending account:** `extensions/slack-addressee-guard/index.ts:294-296`
  - body: The guard resolves only `slackTokenOverride` or `SLACK_BOT_TOKEN` and ignores the `ctx.accountId` plus configured Slack account credentials used by the actual send. Multi-account or SecretRef-only installs can query the wrong workspace or fail open while the send uses the correct account token.
  - confidence: 0.87
- **[P2] Update the startup plugin contract:** `extensions/slack-addressee-guard/openclaw.plugin.json:3-5`
  - body: The manifest declares `activation.onStartup: true`, but the bundled startup plugin test asserts the full on-startup ID list and this PR does not add `slack-addressee-guard`. Either update the contract if this stays bundled/startup or change the activation/split the plugin.
  - confidence: 0.82
- **[P3] Remove or implement the logPath setting:** `extensions/slack-addressee-guard/openclaw.plugin.json:20`
  - body: The manifest exposes `logPath` and the UI help promises JSONL decision logging, but the implementation never reads the setting or writes decision records. Either implement the logging behavior or remove the config field before shipping this plugin.
  - confidence: 0.88
- **[P3] Add routing labels for the new plugin:** `extensions/slack-addressee-guard/openclaw.plugin.json:2`
  - body: This adds a new bundled plugin under `extensions/slack-addressee-guard/**`, but `.github/labeler.yml` only maps Slack changes under `extensions/slack/**`. Add a Slack or plugin-specific label mapping so future changes are routed consistently.
  - confidence: 0.82
