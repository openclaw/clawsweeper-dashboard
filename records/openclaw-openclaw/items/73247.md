## Summary

Keep open. Current main still has the Slack reply hook gap and the PR’s hook wiring addresses it, but the combined patch is not merge-ready because the new guard rewrites non-leading mentions, skips Slack D/G conversations, exposes unused config, misses routing/changelog metadata, and has an account-scoped credential concern.

## What This Changes

This PR adds `message_sending` hook execution to Slack agent reply delivery and adds a bundled `slack-addressee-guard` plugin with manifest and tests.

## Best Possible Solution

Land the Slack reply hook wiring as a focused, tested change; ship the addressee guard only after it is account-aware, leading-mention scoped, Slack-surface complete, and aligned with labels/changelog/docs.

## Reproduction Assessment

Yes. Current main reproduces the hook gap by source inspection: Slack reply delivery calls `sendMessageSlack()` directly, while `message_sending` runs in separate generic dispatch paths.

## Solution Assessment

No. The hook wiring is the narrow maintainable fix, but the bundled guard and metadata need hardening or a separate product decision before this PR is the best complete solution.

## Review Findings

  - body: The guard enters when the target is mentioned anywhere, but `repairContent()` prepends the last human if the mention is not leading. A normal reply like `I checked with <@target>` can be rewritten even though it is not the accidental leading-address case described by the plugin.

  - confidence: 0.88

  - body: This adds a new bundled plugin under `extensions/slack-addressee-guard/**`, but `.github/labeler.yml` only maps `extensions/slack/**` to Slack changes. Add a plugin or Slack label mapping so future changes route consistently.

## Security Review

Status: needs_attention

Summary: The new bundled guard can query Slack thread history with the wrong credential because it ignores the sending Slack account.

Concerns:

- **[medium] Use account-scoped Slack credentials:** `extensions/slack-addressee-guard/index.ts:294`
  - body: The guard resolves only `slackTokenOverride` or `SLACK_BOT_TOKEN` and ignores the `message_sending` context account. In multi-account or SecretRef-only installs, it can call `conversations.replies` against the wrong workspace credential or fail open even though the sending account is configured.
  - confidence: 0.84
