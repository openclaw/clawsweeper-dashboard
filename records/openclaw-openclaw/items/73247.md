## Summary

Keep open. Current main still has the Slack reply hook coverage gap, but this draft branch is not merge-ready because the bundled guard adds correctness and account-scoped credential blockers.

## What This Changes

The PR wires Slack reply delivery through `message_sending` hooks and adds a bundled `slack-addressee-guard` plugin with manifest and tests.

## Best Possible Solution

Land the focused Slack reply hook wiring, then split or harden the addressee guard as an account-aware official plugin or ClawHub follow-up.

## Reproduction Assessment

Yes. Current main reproduces the reported hook gap by source inspection: Slack reply delivery calls `sendMessageSlack()` directly while the documented `message_sending` hook is only wired in other delivery paths.

## Solution Assessment

No. The hook wiring is the narrow maintainable fix, but the combined PR is not the best complete solution until the guard's mention matching, Slack ID support, credential scope, metadata, labeler, and changelog gaps are resolved.

## Security Review

Status: needs_attention

Summary: The new bundled guard reads Slack thread history with a default environment token instead of the credential for the Slack delivery account.

Concerns:

- **[medium] Use account-scoped Slack credentials:** `extensions/slack-addressee-guard/index.ts:87`
  - body: The guard resolves only `SLACK_BOT_TOKEN` or `slackTokenOverride` and ignores `ctx.accountId` plus configured Slack account credentials. In multi-account or SecretRef-only installs, it can query `conversations.replies` with the wrong workspace credential or fail open even though the sending account is configured.
  - confidence: 0.84
