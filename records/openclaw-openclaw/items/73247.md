## Summary

Keep open: the reported Slack hook gap is valid, but this draft patch still misses the default native-streaming reply path and the bundled guard has account-scoping, metadata, and product-boundary blockers.

## What This Changes

The PR wires Slack `deliverReplies()` through `message_sending` hooks and adds a bundled `slack-addressee-guard` plugin with config and tests.

## Best Possible Solution

Land a focused Slack hook fix that covers both normal and native-streamed reply delivery, then split or rework the addressee guard as an account-aware bundled plugin or ClawHub follow-up with the required metadata.

## Reproduction Assessment

Yes. Source inspection on current main shows normal Slack replies call `sendMessageSlack()` directly and default threaded native streaming calls `startSlackStream()` directly, so `message_sending` coverage is incomplete without running a live Slack workspace.

## Solution Assessment

No. The hook-wiring direction is useful, but the proposed implementation is incomplete until it covers native streaming and the guard is either split out or made account-aware, scoped to leading mentions, and metadata-complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cover native streaming replies with hooks:** `extensions/slack/src/monitor/replies.ts:48-49`
  - body: Slack defaults to `streaming.mode=partial` with native streaming enabled, and threaded text replies go through `startSlackStream()` in the message handler instead of `deliverReplies()`. Wrapping only `deliverReplies()` leaves the default streamed reply path invisible to `message_sending`, so guard/thread-ownership plugins still miss those agent replies.
  - confidence: 0.86
- **[P2] Limit the guard to leading target mentions:** `extensions/slack-addressee-guard/index.ts:351`
  - body: The guard starts when the configured target is mentioned anywhere, but `repairContent()` prepends the last human when the target mention is not leading. A normal reply like `I checked with <@target>` would be rewritten even though the plugin is described as protecting accidental leading addresses.
  - confidence: 0.9
- **[P2] Accept Slack DM and private-channel IDs:** `extensions/slack-addressee-guard/index.ts:130-131`
  - body: `parseChannelAndThread()` rejects every conversation ID that does not start with `C`, so DMs, MPIMs, and private channels using Slack `D...` or `G...` IDs silently skip the guard even though Slack send/routing supports those surfaces.
  - confidence: 0.88
- **[P2] Resolve Slack credentials for the sending account:** `extensions/slack-addressee-guard/index.ts:294-296`
  - body: The guard resolves only `slackTokenOverride` or `SLACK_BOT_TOKEN` and ignores `ctx.accountId` plus configured Slack account credentials. Multi-account or SecretRef-only installs can query the wrong workspace or fail open while the actual send uses the correct account token.
  - confidence: 0.87
- **[P2] Update the startup plugin contract:** `extensions/slack-addressee-guard/openclaw.plugin.json:3-5`
  - body: The new manifest declares `activation.onStartup: true`, but the bundled startup plugin test asserts the full on-startup ID list and this PR does not add `slack-addressee-guard`. Either update the contract if this stays bundled/startup or change the activation/split the plugin.
  - confidence: 0.82
- **[P3] Remove or implement the logPath setting:** `extensions/slack-addressee-guard/openclaw.plugin.json:20`
  - body: The manifest exposes `logPath` and the UI help promises JSONL decision logging, but the implementation never reads the setting or writes decision records. Either implement the logging behavior or remove the config field before shipping this plugin.
  - confidence: 0.88
- **[P3] Add routing labels for the new plugin:** `extensions/slack-addressee-guard/openclaw.plugin.json:2`
  - body: This adds a new bundled plugin under `extensions/slack-addressee-guard/**`, but `.github/labeler.yml` only maps Slack changes under `extensions/slack/**`. Add a Slack or plugin-specific label mapping so future changes are routed consistently.
  - confidence: 0.82
