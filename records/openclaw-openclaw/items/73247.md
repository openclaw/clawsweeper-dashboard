## Summary

Keep open. Current main still has the Slack agent-reply `message_sending` coverage gap, so the PR is not obsolete, but the branch is not merge-ready because the bundled addressee guard has correctness, configuration, labeler, and account-scoped credential problems.

## What This Changes

The PR wires `message_sending` hooks into Slack reply delivery and adds a bundled `slack-addressee-guard` plugin with unit tests.

## Best Possible Solution

Land the Slack reply hook coverage as a focused change, with tests for block, text, media, streaming fallback, and preview/final edit paths. Treat the addressee guard as a separate decision: either harden it as an official Slack plugin with account-aware credentials and full Slack conversation-id support, or move it outside core through the plugin ecosystem.

## Reproduction Assessment

Yes. Current main reproduces the gap by source inspection: Slack reply delivery calls `sendMessageSlack()` directly, while the generic `message_sending` insertion only happens in wrapper helpers that Slack does not use for this prebuilt dispatcher path.

## Solution Assessment

No. The hook wiring direction is the right narrow fix, but this branch is not the best complete solution while it bundles an optional guard with leading-mention, Slack id, credential-scope, unused config, and labeler gaps.

## Review Findings

Overall confidence: 0.86

Full review comments:

- **[P2] Only run the guard for leading target mentions:** `extensions/slack-addressee-guard/index.ts:351`
  - body: The guard enters when the target is mentioned anywhere, but `repairContent()` prepends the last human if the mention is not leading. A normal reply such as `I checked with <@target>` would be rewritten even though it is not the accidental leading-address case described by the plugin.
  - confidence: 0.9
- **[P2] Accept Slack DM and private channel conversation IDs:** `extensions/slack-addressee-guard/index.ts:130`
  - body: `parseChannelAndThread()` rejects every conversation id that does not start with `C`, so DMs, MPIMs, and private channels using `D...` or `G...` silently skip the guard even though Slack reply delivery and the docs support those surfaces.
  - confidence: 0.87
- **[P3] Remove or implement the logPath setting:** `extensions/slack-addressee-guard/openclaw.plugin.json:20`
  - body: The manifest exposes `logPath` and the UI help promises JSONL decision logging, but the implementation never reads the setting or writes decision records. Either implement the logging behavior or remove the config field before shipping this bundled plugin.
  - confidence: 0.88
- **[P3] Add a labeler mapping for the new plugin:** `extensions/slack-addressee-guard/openclaw.plugin.json:2`
  - body: This adds a new bundled plugin under `extensions/slack-addressee-guard/**`, but `.github/labeler.yml` only covers `extensions/slack/**` for Slack changes. Add an appropriate plugin or Slack label mapping so future changes are routed consistently.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Security review needs attention because the new bundled guard reads Slack thread history with a default environment token instead of the Slack account credential for the delivery context.

Concerns:

- **[medium] Use account-scoped Slack credentials:** `extensions/slack-addressee-guard/index.ts:87`
  - body: The guard resolves only `SLACK_BOT_TOKEN` or `slackTokenOverride` and ignores `ctx.accountId` plus the configured Slack account credentials. In multi-account or SecretRef-only installs, it can query `conversations.replies` with the wrong workspace credential or fail open even though the sending account is configured.
  - confidence: 0.84
