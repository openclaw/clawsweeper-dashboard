## Review Findings

Overall confidence: 0.91

Full review comments:

- **[P1] Import legacy cron jobs before returning empty:** `src/cron/store.ts:1004-1010`
  - body: When an upgraded install has no `cron_jobs` rows yet, the shipped `jobs.json`/`jobs-state.json` files can still contain every schedule. This path now returns an empty store and never imports the legacy files on normal gateway, list, or timer reads, so scheduled automations disappear until an operator runs doctor. Preserve the upgrade path by importing valid legacy rows or keeping a compatibility read before returning empty.
  - confidence: 0.94
- **[P1] Read legacy run logs before returning empty history:** `src/cron/run-log.ts:679-683`
  - body: The new history readers query only `cron_run_logs`; `migrateLegacyCronRunLogsToSqlite` is only a doctor helper. Existing `cron/runs/*.jsonl` history will look empty after upgrade in `cron runs` and status surfaces until manual repair, despite the docs saying legacy files are imported on upgrade. Trigger import or retain a legacy read path before returning SQLite-only results.
  - confidence: 0.9
- **[P1] Guard stateOnly writes against stale schedule identity:** `src/cron/store.ts:718-723`
  - body: A stale scheduler can still call `saveCronStore(..., { stateOnly: true })` after another process edits the job schedule. This update writes the old `state_json` and old `schedule_identity` without checking the current row, and later SQLite loads apply that state directly, so an old `nextRunAtMs` can survive on the new schedule. Compare identities before writing or clear stale next-run state when loading rows.
  - confidence: 0.86
- **[P3] Drop the release-owned changelog entry:** `CHANGELOG.md:29`
  - body: `CHANGELOG.md` is release-owned for normal PRs in this repo, and this branch already carries release-note context in the PR body. Remove this line and leave changelog generation to the release flow.
  - confidence: 0.98
