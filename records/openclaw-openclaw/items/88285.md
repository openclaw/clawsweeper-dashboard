## Summary

Keep open: the branch is useful but still has merge-blocking cron upgrade and compatibility issues, and the PR body still provides tests rather than real legacy-upgrade proof.

## What This Changes

The branch moves cron job storage, runtime state, and run history from legacy JSON/JSONL files to the shared SQLite state database, adds doctor migration/schema repair paths, updates docs/config help, and rewrites focused cron tests.

## Best Possible Solution

Preserve existing cron behavior across upgrade by making the scheduler see legacy jobs until migration is durable, preserving schedule-state invalidation and byte/row retention semantics, and documenting any intentional doctor-gated migration only with explicit maintainer approval and proof.

## Reproduction Assessment

Yes. Source inspection gives high-confidence repro paths: upgrade with legacy cron files and an empty `cron_jobs` table, concurrent schedule edit followed by stale `stateOnly` persist, and SQLite run-log appends with a low `maxBytes` value.

## Solution Assessment

No. Moving cron persistence to SQLite is reasonable, but the current PR is not the safest implementation until it preserves upgrade visibility, schedule-state invalidation, and run-log retention compatibility or gets an explicit maintainer decision to break those behaviors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Import legacy cron rows before returning an empty SQLite store:** `src/cron/store.ts:1004-1010`
  - body: `loadCronStoreWithConfigJobs` now returns an empty store whenever SQLite has no rows, and the legacy JSON import lives under `openclaw doctor` instead of this scheduler load path. On a normal upgrade, existing `jobs.json` schedules can therefore stop running until the operator runs doctor, despite the docs saying the files are imported on upgrade.
  - confidence: 0.89
- **[P1] Preserve schedule mismatch invalidation for stateOnly saves:** `src/cron/store.ts:719-723`
  - body: A stale scheduler can call `saveCronStore(..., { stateOnly: true })` after another process edits the schedule. This update writes the old runtime state and old `schedule_identity` over the row without comparing the persisted schedule columns, so a later reload can keep the old `nextRunAtMs` for the new schedule.
  - confidence: 0.86
- **[P1] Preserve maxBytes pruning for SQLite run history:** `src/cron/run-log.ts:403-407`
  - body: Current main uses `cron.runLog.maxBytes` to trigger pruning, but the SQLite path only prunes by `keepLines` and ignores `opts.maxBytes`. Existing configs that lowered `maxBytes` to cap disk growth will silently lose that cap after this migration.
  - confidence: 0.9
- **[P3] Drop the release-owned changelog edit:** `CHANGELOG.md:26`
  - body: Normal PRs should not edit `CHANGELOG.md`; release-note context belongs in the PR body or commit message, and the release process owns changelog generation.
  - confidence: 0.93
