## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Import legacy cron jobs before returning an empty store:** `src/cron/store.ts:1004-1010`
  - body: On upgrade, cron_jobs starts empty while jobs.json/jobs-state.json can still contain every configured schedule. This path returns an empty store to gateway startup/list/timer reads instead of importing or falling back, so existing automations disappear until someone runs doctor manually.
  - confidence: 0.92
- **[P2] Preserve empty per-job failure alerts:** `src/cron/store.ts:585-595`
  - body: An empty `failureAlert: {}` is persisted as `failure_alert_disabled = 0` with the option columns null, but this reader treats that row as absent. `resolveFailureAlert` uses a present per-job object to opt into alerts when global alerts are disabled, so the SQLite round-trip can silently suppress failure notifications.
  - confidence: 0.9
- **[P2] Import legacy run logs before SQLite-only history reads:** `src/cron/run-log.ts:679-683`
  - body: Existing `cron/runs/*.jsonl` files are only migrated by doctor, while these read paths now query only `cron_run_logs`. Upgraded installs will show empty job/all run history until a manual repair even though the old files are still present.
  - confidence: 0.86
- **[P2] Drain pending writes before all-run history queries:** `src/cron/run-log.ts:731-733`
  - body: `appendCronRunLog` still queues fire-and-forget writes by log path, and the job-scoped reader drains that queue first. The all-jobs reader queries SQLite directly, so an immediate `cron.runs` with `scope: "all"` can miss the run that was just recorded.
  - confidence: 0.84
