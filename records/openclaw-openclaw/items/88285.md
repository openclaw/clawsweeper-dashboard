Full review comments:

- **[P1] Import legacy cron stores before returning empty SQLite state:** `src/cron/store.ts:1004-1010`
  - body: Current main loads `jobs.json` and `jobs-state.json` directly, but this branch returns an empty store whenever `cron_jobs` has no rows. On an upgraded install that has not run doctor yet, Gateway startup/list/status/timer paths will act as if all scheduled jobs disappeared, so preserve the legacy import/read path before returning empty or get explicit maintainer approval for a doctor-gated break.
  - confidence: 0.94
- **[P1] Clear stale runtime slots using the stored schedule identity:** `src/cron/store.ts:676`
  - body: `rowToCronJob` applies `state_json` directly, while `stateOnly` saves can overwrite `schedule_identity` from a stale in-memory job after another process edited the schedule columns. A fresh process has no previous in-memory job to compare against, so old `nextRunAtMs` can survive a schedule change; compare the row schedule identity with the current schedule before keeping runtime state.
  - confidence: 0.86
- **[P1] Import legacy run logs before SQLite-only history reads:** `src/cron/run-log.ts:679-683`
  - body: The normal history readers now query `cron_run_logs` only, while `migrateLegacyCronRunLogsToSqlite` is only a separate migration helper. Existing `cron/runs/*.jsonl` history will be invisible in CLI/UI/API history until an operator runs doctor, so import or read the legacy file before returning SQLite-only results.
  - confidence: 0.93
- **[P2] Preserve empty per-job failure alerts:** `src/cron/store.ts:585-595`
  - body: `bindFailureAlertColumns` persists `failureAlert: {}` as `failure_alert_disabled: 0`, but this reader returns `undefined` when the other alert columns are null. That drops an intentional per-job override after save/reload and can suppress failure notifications when global cron alerts are disabled.
  - confidence: 0.91
- **[P2] Drain pending writes before all-run history queries:** `src/cron/run-log.ts:731-733`
  - body: `appendCronRunLog` still queues writes by legacy log path, but the all-jobs reader queries SQLite without draining those pending promises. A cron.runs request with `scope: "all"` immediately after a fire-and-forget append can miss the just-finished run until a later request.
  - confidence: 0.84
- **[P3] Remove the release-owned changelog edit:** `CHANGELOG.md:29`
  - body: Normal PRs should carry release-note context in the PR body or commit message, not edit `CHANGELOG.md`, which is release-owned in this repo. Please drop this line from the branch and keep the migration summary in the PR body.
  - confidence: 0.87
