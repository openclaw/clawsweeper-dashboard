## Summary

Keep open. Current main still eagerly imports and constructs the Canvas startup modules, while this PR moves those implementation loads behind existing async plugin SDK route, resolver, CLI, and tool contracts with sufficient terminal benchmark proof; no cleanup close reason applies.

## What This Changes

The PR defers Canvas host, hosted media resolver, CLI, and tool implementation imports from plugin registration until first use and adds entrypoint coverage plus a shared tool schema module.

## Best Possible Solution

Keep the lazy Canvas registration approach on this PR path and let a maintainer complete normal review, CI gating, and merge judgment.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a startup performance refactor. The eager current-main imports are source-visible, and the PR body supplies after-fix benchmark output for the startup behavior being changed.

## Solution Assessment

Yes. Deferring concrete Canvas modules behind existing async plugin SDK route, resolver, CLI, and tool contracts is a narrow maintainable solution without adding API or config surface.
