## Summary

Keep open: this PR targets a real source-reproducible Canvas capability bug, but the proposed injected live-reload parser can still pick a fake path token from ordinary Canvas content paths and break WebSocket auth.

## What This Changes

The PR updates gateway capability URL normalization and Canvas live-reload injection/tests so path-scoped Canvas capability tokens override stale `oc_cap` query tokens.

## Best Possible Solution

Keep the normalizer overwrite direction, but make the injected Canvas script trust a path token only for true scoped capability URLs and add regression coverage for both stale-query and non-scoped content-path cases.

## Reproduction Assessment

Yes. Current `main` gives a high-confidence source repro: the normalizer can preserve a stale `oc_cap` in `rewrittenUrl`, and the injected Canvas script reads only `location.search`; I did not execute tests in this read-only review.

## Solution Assessment

No. The gateway normalizer change is the right narrow direction, but the injected script should mirror the gateway scoped-path prefix rule instead of scanning the whole content path with `indexOf`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Restrict path-token parsing to scoped URLs:** `extensions/canvas/src/host/a2ui-shared.ts:57`
  - body: The gateway only treats a capability as path-scoped when the request path starts with `/__openclaw__/cap/`, but this injected script scans for that segment anywhere in `location.pathname`. A normal Canvas page such as `/__openclaw__/canvas/docs/__openclaw__/cap/foo/index.html?oc_cap=<valid>` would send `foo` to the live-reload WebSocket instead of the valid query token, so live reload fails for that content path. Please only parse the path token for true scoped URLs, or share the same prefix rule as the gateway helper.
  - confidence: 0.88
