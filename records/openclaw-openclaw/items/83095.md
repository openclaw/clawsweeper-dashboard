## Summary

Keep open: the normalizer change is a plausible bug fix and the supplied proof is acceptable, but the injected Canvas live-reload parser can treat a user file path segment as the authoritative capability token. That needs a narrow patch before merge.

## What This Changes

The PR makes path-scoped Canvas capability tokens overwrite stale `oc_cap` query values and updates injected live-reload code/tests to prefer a path token.

## Best Possible Solution

Merge the path-token precedence behavior with an anchored browser-side parser that only accepts true scoped URLs, plus a regression test for normal Canvas paths containing the reserved substring.

## Reproduction Assessment

Yes. Current main has a deterministic source-level path: the normalizer preserves an existing stale `oc_cap` in `rewrittenUrl`, and the live-reload script reads only the query token; I did not execute tests because this was a read-only review.

## Solution Assessment

No, not as written. The gateway normalizer change is the right narrow direction, but the injected Canvas parser should mirror the start-anchored scoped URL semantics before this can merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Anchor the path capability parser to scoped URLs:** `extensions/canvas/src/host/a2ui-shared.ts:59`
  - body: The injected script now uses `location.pathname.indexOf(prefix)`, so a normal Canvas document path such as `/__openclaw__/canvas/foo/__openclaw__/cap/stale/index.html?oc_cap=current` will pick `stale` from the file path and ignore the real query token. The Canvas host injects this script into any served HTML under `/__openclaw__/canvas`, so this breaks live reload for valid user-created paths; require the scoped capability prefix at the start of the browser path, or otherwise prove it precedes the Canvas route, and add a regression case.
  - confidence: 0.9
