## Summary

Keep open: the gateway normalizer change matches the current-main root cause, and the supplied proof is credible, but the injected Canvas parser is broader than the gateway contract and can steal a capability-looking segment from an ordinary Canvas content path.

## What This Changes

The PR updates gateway capability URL normalization and Canvas live-reload injection/tests so path-scoped Canvas capability tokens replace stale `oc_cap` query tokens.

## Best Possible Solution

Keep the normalizer direction, but make the injected Canvas script mirror the gateway scoped-path contract so only true scoped URLs outrank `oc_cap` query fallback, with regression coverage for both stale-query and non-scoped content-path cases.

## Reproduction Assessment

Yes. Current `main` gives a high-confidence source repro: the normalizer can preserve a stale `oc_cap` in `rewrittenUrl`, and the injected Canvas script reads only the query token. I did not execute tests because this was a read-only review.

## Solution Assessment

No. The normalizer update is the right direction, but the injected script should use the same scoped-path prefix rule as the gateway instead of `indexOf` over the whole content path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Restrict path-token parsing to scoped URLs:** `extensions/canvas/src/host/a2ui-shared.ts:57`
  - body: The gateway only treats a capability as path-scoped when the request path starts with `/__openclaw__/cap/`, but this injected script scans for that segment anywhere in `location.pathname`. A normal Canvas page like `/__openclaw__/canvas/docs/__openclaw__/cap/foo/index.html?oc_cap=<valid>` would send `foo` to the live-reload WebSocket instead of the valid query token, so live reload fails for that content path. Please only parse the path token when the browser path is actually a scoped capability URL, or share the same prefix rule as the gateway helper.
  - confidence: 0.88
