## Summary

Keep open: current main has improved Codex OAuth status handling for OpenClaw auth profiles, but the source still does not use the per-agent Codex app-server account or `codex-home/auth.json` when rendering `/status` auth for `openai/*` running on the Codex harness.

## What This Changes

Report ChatGPT/Codex runtime auth in status output instead of `api-key (env: OPENAI_API_KEY)` when the Codex runtime is authenticated through ChatGPT.

## Best Possible Solution

Make status auth resolution use the effective Codex harness auth source, including the per-agent Codex app-server account, before falling back to OpenAI env/API-key labels.

## Reproduction Assessment

Yes, at source level: `/status` resolves OpenAI-on-Codex auth through OpenAI auth profiles/env and only checks Codex CLI credentials for the literal `codex` provider. I did not run a live Gateway reproduction.

## Solution Assessment

Yes: the narrow fix is to teach the existing status auth-label path about the effective Codex app-server account for `openai/*` Codex harness sessions, while keeping direct OpenAI API-key reporting intact.
