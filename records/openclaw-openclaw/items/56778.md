## Summary

Keep this issue open. Current main at 9d21df251e50ca5fbf5c0e34783cc9f66114951c still has the reported cron list isolation gap: there is no CLI `--agent` option, no protocol `agentId` list parameter, no gateway forwarding, no service predicate, and the agent cron tool does not auto-scope list calls even though add calls infer the session agent. https://github.com/openclaw/openclaw/issues/56778

## What This Changes

Add an `openclaw cron list --agent <id>` filter and make agent-session cron list calls scope to the requesting agent while preserving admin listing behavior.

## Best Possible Solution

Add a backward-compatible exact `agentId` filter to the cron list schema, service, and gateway handler; expose it as `openclaw cron list --agent <id>`; and make the agent cron tool's list action default to the caller's session agent while leaving unscoped admin list behavior available.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: current main has no accepted list `agentId` parameter or CLI option, and the service returns all enabled jobs when no query is present.

## Solution Assessment

Yes. The narrow maintainable solution is an additive list-filtering seam plus agent-tool default scoping and focused tests/docs, rather than broader agent-owned cron configuration work.

## Repair Work Prompt

Fix the cron list multi-agent isolation gap from https://github.com/openclaw/openclaw/issues/56778. Add a backward-compatible exact `agentId` filter for `cron.list`: extend `CronListParamsSchema` and types, pass the field through the gateway handler into the cron service, and have `listPage` filter jobs by exact normalized agent id. Expose `openclaw cron list --agent <id>` in the CLI. For agent-tool `action: "list"`, infer the caller's agent from `opts.agentSessionKey` and pass it by default, while preserving unfiltered list behavior for admin/CLI callers that do not pass `--agent`. Treat unset job `agentId` as visible to the configured default agent only in scoped default-agent views, without rewriting stored jobs or changing startup/load-time migration behavior. Add focused tests for CLI params, protocol validation, gateway forwarding, service filtering including unset/default-agent semantics, and agent-tool auto-scoping. Update `docs/cli/cron.md`, `docs/automation/cron-jobs.md`, and add a user-facing `CHANGELOG.md` fix entry with reporter credit if applicable. Do not broaden this into agent-owned cron config work or change unrelated cron delivery/session semantics.
