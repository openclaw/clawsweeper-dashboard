## Summary

Keep open. Current main still lacks the requested cron list isolation: the CLI has no `--agent` option, the Gateway protocol and handler do not accept or forward an exact agent filter, the cron service returns all jobs when unfiltered, and the agent cron tool list action does not infer the caller agent.

## What This Changes

Request to add an agent filter to `openclaw cron list` and auto-scope agent-session cron list calls to the requesting agent while treating unowned jobs as default-agent jobs.

## Best Possible Solution

Add a backward-compatible exact `agentId` list filter through the cron protocol, Gateway handler, service, CLI, and agent tool. Agent-session tool calls should default to the caller's agent, while admin CLI/RPC calls should keep today's global behavior unless `--agent` or `agentId` is supplied.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: unscoped `cron.list` has no accepted or forwarded `agentId` filter, and the service returns every enabled job when no query is present.

## Solution Assessment

Yes. The narrow maintainable fix is an additive exact filter plus agent-tool default scoping and focused tests/docs; broader agent-owned cron configuration from related discussions should stay separate unless maintainers intentionally combine the product work.
