## Summary

Keep open. Current main still lacks the reported cron-list isolation: the CLI has no `--agent` option for `cron list`, the Gateway schema rejects an exact agent filter, the handler never forwards an agent scope, the service returns all jobs when unfiltered, and the agent cron tool list action does not infer the caller's agent.

## What This Changes

Request to add agent-scoped filtering for `openclaw cron list` and auto-scope agent-tool cron list calls in multi-agent setups.

## Best Possible Solution

Add a backward-compatible exact `agentId` list filter through the cron protocol, Gateway handler, service page options, CLI, and agent tool; default agent-tool list calls to the caller's session agent, and document the admin versus scoped behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: there is no CLI option, accepted protocol field, Gateway forwarding, service predicate, or agent-tool inferred list scope for `agentId`, so unscoped `cron.list` still returns globally visible jobs.

## Solution Assessment

Yes. The narrow maintainable fix is an additive filtering seam plus caller-agent defaulting in the agent tool; broader agent-owned cron configuration is related but not required to solve this isolation bug.
