## Summary

Keep open: the PR targets a valid linked ACP background-task bug that is still present on current main, but the patch should not merge as submitted because it would surface non-deliverable silent tokens as terminal text and the external contributor has not supplied real after-fix behavior proof.

## What This Changes

The branch raises `ACP_BACKGROUND_TASK_PROGRESS_MAX_LENGTH` from 240 to 1200 and returns normalized progress as `terminalSummary` for non-blocked ACP background-task completions.

## Best Possible Solution

The maintainable end state is a narrow ACP manager fallback/cap fix that preserves silent-token suppression and locks the behavior with focused tests.

## Reproduction Assessment

Yes. Current-main source shows ACP progress is accumulated and stored, but ordinary successful progress resolves to `{}`, is persisted as `terminalSummary: null`, and terminal delivery formats only `terminalSummary`.

## Solution Assessment

No, not as submitted. Returning normalized progress directly is the right narrow direction, but it must strip exact/trailing non-deliverable control tokens and add regression coverage before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Strip silent tokens before terminal fallback:** `src/acp/control-plane/manager.core.ts:144`
  - body: This fallback turns any non-empty progress into `terminalSummary`. If ACP progress is exactly or ends with a non-deliverable token such as `ANNOUNCE_SKIP`, task delivery will post that marker as visible completion text because the terminal-summary sanitizer does not strip it. Clean exact/trailing silent-control tokens before returning the fallback summary and cover that case.
  - confidence: 0.87
