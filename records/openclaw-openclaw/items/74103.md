## Summary

Keep open: the linked bug is valid on current main and this PR is pointed at the right ACP manager path, but the submitted fallback would also surface silent control-token text as a visible terminal completion. The narrow fix should preserve the PR intent, add token cleanup, and cover the ACP completion path with focused tests before merge.

## What This Changes

The PR raises the ACP background-task progress accumulation cap to 1200 characters and returns non-blocked progressSummary as terminalSummary from resolveBackgroundTaskTerminalResult().

## Best Possible Solution

Land a narrow ACP manager fix that copies cleaned non-blocked progress into terminalSummary, preserves blocked-outcome detection, strips exact or trailing silent-control tokens, keeps the cap change only if maintainers accept the larger payload, and adds focused regression coverage plus the required user-facing changelog entry.

## Reproduction Assessment

Yes. A parent-owned ACP run that emits non-thought text_delta progress and completes reaches resolveBackgroundTaskTerminalResult(), current main returns {}, markBackgroundTaskTerminal() stores terminalSummary null, and formatTaskTerminalMessage() emits the bare completion message. The PR-specific regression can be reproduced with progress ending in ANNOUNCE_SKIP, which would become visible terminal text.

## Solution Assessment

No, not as submitted. The fallback-and-cap direction is the narrow maintainable fix for the linked report, but the fallback should strip silent completion tokens before terminal delivery and needs regression tests rather than a broader task-delivery refactor.
