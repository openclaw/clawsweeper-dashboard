## Summary

Keep open: current main still has a source-backed WhatsApp path where rapid media messages bypass inbound debounce, each image is dispatched as a single-message turn, WhatsApp only places one media item in the inbound context, and active-run steering accepts only text. The paired attempted fixes are closed unmerged and did not cover the production media-bypass path.

## What This Changes

The issue asks OpenClaw to preserve every image from rapid forwarded WhatsApp media bursts, either by WhatsApp-local media coalescing or media-aware active-run steering.

## Best Possible Solution

Implement a WhatsApp-owned media burst path that preserves every local media item as MediaPaths/MediaTypes in one inbound context, unless maintainers explicitly choose to extend the generic steering contract to carry media.

## Reproduction Assessment

Yes, at source level: two rapid WhatsApp image messages can be traced through media debounce rejection, single-item flushing, singular WhatsApp media context, and text-only active steering. I did not run a live WhatsApp forwarded-image session.

## Solution Assessment

Unclear: WhatsApp-local coalescing is likely the narrowest owner-boundary fix, but maintainers should decide whether the intended contract is channel-local media grouping or generic media-aware steering.
