## Summary

Keep open: current main still has a source-backed WhatsApp path where rapid media messages bypass coalescing and active-run steering carries only text, so the reported second-image loss remains credible; the related attempted PRs are closed unmerged.

## What This Changes

The issue asks OpenClaw to preserve all images from rapid multi-image WhatsApp forwards in the LLM turn, either through WhatsApp media coalescing or media-aware steering.

## Best Possible Solution

Implement a WhatsApp-owned media burst path that builds a single media-aware inbound context with every image preserved, unless maintainers explicitly choose to extend the generic steering contract for media payloads.

## Reproduction Assessment

Yes at source level: two rapid WhatsApp image messages can be traced through media debounce rejection, single-item flushing, singular WhatsApp media context, and text-only active steering. I did not run a live WhatsApp forwarded-image burst.

## Solution Assessment

Unclear: WhatsApp-local coalescing appears to be the narrowest owner-boundary fix, but the alternative media-aware steering contract would affect shared queue behavior and should be a maintainer decision.
