## Summary

Keep open: current main and the latest release still show a source-backed WhatsApp path where rapid media messages bypass coalescing, build only a single-message media context, and steer only text into active runs. The related attempted fixes are closed unmerged, so this remains the canonical open report.

## What This Changes

The issue asks OpenClaw to preserve every image from rapid WhatsApp multi-image forwards in the LLM turn, either through WhatsApp media coalescing or media-aware steering.

## Best Possible Solution

Implement a WhatsApp-owned media burst path that emits one inbound context with all local media as MediaPaths/MediaTypes, unless maintainers explicitly choose to extend the generic steering contract to carry media attachments.

## Reproduction Assessment

Yes at source level: two rapid WhatsApp image messages can be traced through media debounce rejection, single-item flushing, singular WhatsApp media context, and text-only active steering. I did not run a live WhatsApp forward-burst session.

## Solution Assessment

Unclear: WhatsApp-local coalescing looks like the narrowest owner-boundary fix, but the issue also raises generic media-aware steering, so maintainers should choose the intended contract before implementation.
