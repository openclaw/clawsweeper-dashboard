## Summary

Keep open. Current main and the latest release still have a source-visible path where WhatsApp media messages bypass inbound debounce and active `queue.mode: "steer"` injects only text into the running agent, so a rapid second image can be accepted without its image payload reaching the LLM.

## What This Changes

The issue requests preserving all images from rapid forwarded WhatsApp multi-image messages, either by WhatsApp media coalescing or by making steer-mode delivery carry media attachments.

## Best Possible Solution

Preserve every rapid WhatsApp image by adding a media-aware WhatsApp burst path and/or carrying `followupRun.images` through active-run steering, while leaving timestamp batching and per-group debounce as separate product/config work.

## Reproduction Assessment

Yes, at source level: WhatsApp media messages bypass debounce and the active steer path sends only text, while current-turn images are stored separately on `followupRun`. I did not run a live WhatsApp reproduction.

## Solution Assessment

Yes on direction, but not the closed split-dispatch PR. The maintainable fix should preserve structured image payloads through either a WhatsApp media burst coalescer or media-aware active-run steering with per-delivery tests.
