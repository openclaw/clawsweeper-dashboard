## Summary

Keep open: current main still has a source-backed path where WhatsApp media messages bypass inbound debounce and active-run steering only carries text, so rapid multi-image forwards can lose the later attachment context.

## What This Changes

The issue requests that rapid WhatsApp multi-image forwards reach one LLM turn with all images, or that queued/steered media attachments are preserved instead of dropping later images.

## Best Possible Solution

Implement a WhatsApp-owned media burst path that preserves all local media as MediaPaths/MediaTypes in one inbound context, or explicitly define media-capable queue steering if maintainers want this fixed generically.

## Reproduction Assessment

Yes, at source level: two rapid WhatsApp image messages can be traced through media debounce rejection, singular WhatsApp media context, and text-only active steering. I did not perform a live WhatsApp forward-burst run against current main.

## Solution Assessment

Unclear until maintainer review: WhatsApp-local coalescing is likely the narrowest owner-boundary fix, but changing default media timing versus extending core steer semantics is a product/runtime decision.
