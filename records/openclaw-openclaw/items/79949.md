## Summary

Keep open: current main still has a source-backed path where rapid WhatsApp image messages bypass inbound media coalescing and active-run steering only carries text, while the related fix PRs are closed and unmerged.

## What This Changes

WhatsApp should preserve multiple rapidly forwarded image attachments from the same sender so the LLM can see all images, either in one coalesced turn or via media-aware steering.

## Best Possible Solution

Implement a WhatsApp-owned media burst path that preserves every image as MediaPaths/MediaTypes in the inbound context, or deliberately extend queue steering to carry media if maintainers want the generic route.

## Reproduction Assessment

Yes, at source level: WhatsApp media is explicitly non-debounceable, non-debounced items flush singly, the WhatsApp context has singular media fields, and steer mode queues only text. I did not run a live WhatsApp forward burst in this read-only review.

## Solution Assessment

Unclear until maintainer review: WhatsApp-local media coalescing looks like the narrowest owner-boundary fix, but media-aware steering would address the alternate failure path more generally. The recent split-dispatch PR is not the best solution because it misses the production media bypass and has dedupe-finalization risk.
