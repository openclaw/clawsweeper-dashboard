## Summary

Keep open: current main still has a source-backed WhatsApp path where rapid media messages bypass inbound debounce and active-run steering carries only text, so later forwarded images can still lose attachment context. The related fix PRs were closed unmerged and the later review found that split-dispatch approach did not cover the production media path.

## What This Changes

Requests that rapid multi-image WhatsApp forwards deliver every image to the agent, either by WhatsApp media coalescing or media-aware queue steering.

## Best Possible Solution

Implement one owner-boundary fix that preserves every rapid WhatsApp image as MediaPaths/MediaTypes or explicitly extends steering to carry media, with focused regression coverage for rapid forwarded images.

## Reproduction Assessment

Yes, at source level: rapid WhatsApp images can be traced through the media debounce rejection, single-item flush, singular WhatsApp media context, and text-only steering path. I did not perform a live WhatsApp forwarded-image burst run.

## Solution Assessment

Unclear: WhatsApp-local media burst coalescing is probably the narrowest fix, but maintainers need to choose that default-timing behavior versus extending core active-run steering to carry media. The closed split-dispatch PRs did not cover the production path where WhatsApp media bypasses debounce.
