## Summary

Keep open. Current main still has a source-visible WhatsApp media batching gap: media messages are excluded from inbound debounce, and the existing combined-message shape only carries one media attachment when a batch is formed.

## What This Changes

The issue asks OpenClaw to deliver all images from rapid multi-image WhatsApp forwards to the LLM instead of only the first image.

## Best Possible Solution

Add a WhatsApp-owned rapid-media coalescing path that preserves all media facts in one inbound turn, with focused tests and no new config unless maintainers explicitly request one.

## Reproduction Assessment

Yes at source level: the WhatsApp listener excludes media from debounce, and the current batch/context shape can only expose one media attachment. I did not run a live WhatsApp reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is a WhatsApp-owned media batching/payload preservation repair. A broad shared-queue rewrite or timestamp-config feature would be a larger and less focused way to solve this issue.
