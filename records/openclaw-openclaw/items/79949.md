## Summary

Keep open: current main still has a source-reproducible WhatsApp media loss path when rapid image messages arrive while the session is in steer mode, and the related repair PR was closed unmerged after review found it did not cover the production path.

## What This Changes

The issue asks OpenClaw to preserve all images from rapid multi-image WhatsApp forwards, either by WhatsApp media coalescing or by making active-run steering carry media attachments.

## Best Possible Solution

Implement a narrow WhatsApp media-preservation fix that guarantees every rapid image reaches either the same LLM turn or a later follow-up, without adding broad config or changing other channels unless the repair proves the shared steer API must carry media.

## Reproduction Assessment

Yes at source level: current main extracts image media, bypasses WhatsApp inbound debounce for media, and then active-run steering sends only the text prompt when a run is streaming. I did not establish a live WhatsApp device reproduction in this read-only review.

## Solution Assessment

Yes, keeping this open for a narrow repair is the best path; the closed split-dispatch PR is insufficient because production media does not enter its multi-entry debounce branch. A WhatsApp-owned media coalescer or a minimal media-aware steering extension is safer than a broad new debounceMedia config surface.
