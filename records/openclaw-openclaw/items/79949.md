## Summary

Keep open: current main still has a source-backed path where rapid WhatsApp image messages bypass media coalescing and active-run steering only carries text, so later forwarded-image attachments can still be lost from the LLM turn. The related fix PRs are closed and unmerged, and the later PR was reviewed as not covering the production media-bypass path.

## What This Changes

The issue requests WhatsApp media burst coalescing, or media-aware steering, so every rapidly forwarded image reaches the LLM.

## Best Possible Solution

Define a WhatsApp-owned rapid media path that preserves every local image as MediaPaths/MediaTypes before dispatch, or explicitly extend active steering to carry media payloads, then add regression coverage and redacted live WhatsApp proof.

## Reproduction Assessment

Yes, at source level: current main rejects WhatsApp media from debounce, builds only singular WhatsApp media context, and steers active runs with text only. I did not perform a live WhatsApp forwarded-image burst against current main.

## Solution Assessment

Unclear. WhatsApp-local coalescing looks like the narrowest owner-boundary fix, but maintainers should choose it over a broader media-aware steering contract; the prior split-dispatch PR did not cover production media bypass.
