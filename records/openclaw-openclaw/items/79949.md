## Summary

This should stay open: current main still has a source-backed WhatsApp media-loss path where media messages bypass WhatsApp debounce, the text combiner would not preserve multiple attachments, and active-run steering only injects text into an existing session. The related open issues cover broader batching policy, but this issue is a narrower message-loss bug that can be fixed within the WhatsApp channel boundary.

## What This Changes

The issue requests that rapid forwarded WhatsApp messages containing multiple images be delivered to the agent as one turn with all image attachments preserved.

## Best Possible Solution

Add a WhatsApp-owned media-aware coalescing path that preserves all attachments into the existing inbound media arrays before one agent turn; leave broader timestamp-based and per-group debounce policy to the related product-decision issues.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: WhatsApp media messages bypass inbound debounce, the existing WhatsApp batch combiner keeps only one payload, and active-run steering injects only text. I did not run a live WhatsApp/Baileys repro in this read-only review.

## Solution Assessment

Yes, a WhatsApp-side attachment coalescing fix is the best narrow path: Telegram’s media-group implementation shows the channel-owned shape, while relying on steering media would require a larger embedded-agent API change.
