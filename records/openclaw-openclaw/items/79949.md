## Label Justifications

- `P2`: This is a normal-priority WhatsApp media delivery bug with limited blast radius and no evidence of a full channel outage.
- `impact:message-loss`: A second forwarded image can be omitted from the active LLM turn when it is routed through text-only steering.
- `impact:session-state`: Images intended as one user action can be split or omitted from the agent context, changing the session transcript and model state.

## Summary

Keep open. Current main still has a media-specific active-steering gap: WhatsApp media messages bypass text debounce and build image context correctly, but the active `steer` path queues only text into the embedded run, so a rapid second image can be omitted from the LLM turn.

## What This Changes

The issue requests batching/coalescing for rapid WhatsApp media messages so multiple forwarded images reach the same LLM turn.

## Best Possible Solution

Preserve media attachments when an inbound message is steered into an active embedded-agent run, and keep broader timestamp batching or per-group debounce changes in the related open WhatsApp batching issues.

## Reproduction Assessment

Yes at source level: WhatsApp media bypasses text debounce, each media message builds one current-turn image context, and active-run `steer` passes only text into the embedded queue. I did not run a live WhatsApp reproduction in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to preserve current-turn images through active steering rather than adding a WhatsApp-only media-group feature first; broader timestamp batching can stay with the related open issues.
