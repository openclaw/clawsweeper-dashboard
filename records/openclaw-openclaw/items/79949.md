## Summary

Keep open: current main and the latest release still bypass WhatsApp inbound debouncing for media, and active-run steering accepts only text, so the reported second-image loss remains a valid WhatsApp delivery bug.

## What This Changes

The issue asks OpenClaw to preserve all images from a rapid multi-image WhatsApp forward in the agent-visible turn instead of delivering only the first image.

## Best Possible Solution

Implement a WhatsApp-owned media-burst repair that preserves every image in the agent-visible current turn, while keeping broader timestamp batching and new debounce config decisions separate.

## Reproduction Assessment

Yes at source level, but not from a live WhatsApp run. Current main and v2026.6.6 both bypass inbound debounce for media, and active steering only accepts text, which matches the reported lost second image under queue steer.

## Solution Assessment

Yes, keeping this open is the right outcome. The best fix is not generic timestamp batching; it is a narrow WhatsApp media-preservation path that either coalesces rapid media before dispatch or avoids text-only steering for media-bearing turns.
