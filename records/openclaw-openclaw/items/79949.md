## Summary

Keep open: the report matches current source behavior, with WhatsApp media messages bypassing inbound debounce and steer-mode injection carrying only text into the active run, so later images can be accepted but not delivered to the LLM as media. I found no canonical open or merged duplicate for this multi-image WhatsApp case.

## What This Changes

Request WhatsApp inbound handling to coalesce rapid multi-image forwarded messages or preserve media attachments when queue steering injects a concurrent message into an active LLM turn.

## Best Possible Solution

Implement a WhatsApp-owned media burst path that delivers all rapid same-sender media attachments into one agent turn, preferably by carrying MediaPaths/MediaTypes through the existing inbound context and adding focused regression coverage.

## Reproduction Assessment

Yes, at source level: emit two accepted WhatsApp image messages from the same sender while a run is streaming under queue.mode=steer, and the second message reaches the steering path as text-only instead of as an image attachment. I did not run a live WhatsApp repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable direction is to fix the WhatsApp inbound/queued-turn media path so all images are represented as media in the agent turn. A generic core change may help steer-mode attachments, but the WhatsApp plugin still needs multi-media representation because its current WebInboundMessage and context mapping are singular.
