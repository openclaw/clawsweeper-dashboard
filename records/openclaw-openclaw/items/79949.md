## Summary

Keep open. Current main still has a concrete media-loss path: WhatsApp media messages bypass inbound text debouncing, and the active `steer` queue path accepts only the prompt text, so a second rapid image can be treated as queued while its image payload is not carried into the running agent turn.

## What This Changes

WhatsApp forwarded multi-image bursts should deliver every image to the agent instead of letting only the first reach the model.

## Best Possible Solution

Preserve media on the existing active-run queue path by carrying current-turn image payloads through internal steering, or by falling back to the existing media-preserving queued path when steering cannot carry attachments; keep broader WhatsApp timestamp coalescing tracked separately.

## Reproduction Assessment

Yes at source level: media-bearing WhatsApp events bypass inbound debouncing, and the active `steer` path queues only text while existing followup/collect paths carry images. I did not perform a live WhatsApp reproduction in this read-only sweep.

## Solution Assessment

Yes for keeping this open as a narrow bug fix: the safest repair is to preserve media in the existing active-run queue path or fall back to a media-preserving queue path. Full WhatsApp timestamp coalescing is a broader related product decision and should stay with https://github.com/openclaw/openclaw/issues/11703.
