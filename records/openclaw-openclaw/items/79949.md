## Summary

Keep open: current main still has the reported WhatsApp media path, and the related attempted fixes were closed unmerged. Rapid WhatsApp media messages bypass inbound debounce, flush as separate turns, and active-run steering still carries only text, so a later forwarded image can still be lost while an agent run is active.

## What This Changes

The issue asks for WhatsApp multi-image forwarded messages to reach the LLM together, either by WhatsApp media-burst coalescing or by media-aware active-run steering.

## Best Possible Solution

Preserve every rapid WhatsApp image in one media-aware inbound context, or explicitly make active-run steering carry media attachments if maintainers choose the generic queue contract.

## Reproduction Assessment

Yes, at source level: two rapid WhatsApp image messages can be traced through media debounce rejection, single-item flushing, singular WhatsApp media context, and text-only active steering. I did not run a live WhatsApp forward-burst session.

## Solution Assessment

Unclear: WhatsApp-local coalescing is likely the narrowest owner-boundary fix, but maintainers should decide whether the intended contract is channel-local media grouping or generic media-aware steering.
