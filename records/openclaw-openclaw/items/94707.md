## Summary

Keep this PR open because current main does not have Slack relay mode, but the branch is not merge-ready: it silently disables native Slack streaming for any custom Slack identity even though the current Slack API/SDK can carry identity fields, and it still needs dependency/security approval plus latest-head real behavior proof.

## What This Changes

The PR adds a Slack `relay` connection mode with WebSocket router ingress, relay config and SecretRef surfaces, relay-provided outbound identity, Slack docs/schema/metadata updates, tests, and a Slack plugin-local `ws` dependency.

## Best Possible Solution

Keep this as the active relay proposal, but thread Slack identity through native stream start instead of disabling streaming, then land only after dependency/security approval, maintainer product approval for relay ingress, and redacted latest-head proof.

## Reproduction Assessment

Yes for the PR finding: source inspection shows the branch sets `streamingEnabled` to false whenever `slackIdentity` is present, while the Slack SDK/API source supports identity fields on stream start. Not applicable as a current-main bug reproduction because relay mode is new feature work.

## Solution Assessment

No as currently shaped: the Slack plugin boundary is plausible for relay mode, but disabling native streaming for custom identity is not the best fix because the SDK can carry identity into `chat.startStream`. The safer path is to preserve streaming and identity together, then have maintainers approve the relay trust/config/dependency contract.
