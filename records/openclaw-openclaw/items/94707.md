## Summary

Keep open: current main and the latest release do not implement Slack relay mode, and this protected-label feature PR still needs a blocking identity/streaming fix plus maintainer/security/dependency approval before merge.

## What This Changes

The branch adds Slack `mode: "relay"` with websocket router ingress, relay auth and gateway config, relay-provided outbound identity, schema/docs/SecretRef metadata, tests, generated metadata, and a Slack plugin-local `ws` dependency.

## Best Possible Solution

Thread custom identity through native Slack streaming, then land only after maintainer/security approval for the relay trust boundary and current-head `ws` dependency, or consolidate the work with the trusted-upstream direction if maintainers want one Slack ingress primitive.

## Reproduction Assessment

Not applicable: this is a feature PR, not a current-main bug report. Current main and v2026.6.9 both only support Slack `socket` and `http` modes, while the PR proof demonstrates after-fix relay behavior.

## Solution Assessment

No: the Slack plugin boundary is plausible, but the current identity fix disables native Slack streaming even though the dependency contract supports username/icon fields on `chat.startStream`. The safer fix is to pass identity into `startSlackStream`, then require maintainer/security/dependency approval for the relay mode.
