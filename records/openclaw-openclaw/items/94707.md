## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Defer relay acks until debounced dispatch completes:** `extensions/slack/src/monitor/relay-source.ts:252`
  - body: `handleSlackMessage()` can return after enqueueing a debounced message instead of after `dispatchPreparedSlackMessage` runs. With `messages.inbound.debounceMs` or the Slack override enabled, this sends a relay ack before dispatch/reply succeeds, so the router can treat a message as delivered even if the delayed flush later fails or the process exits. Make the relay await actual flush/dispatch completion, or make relay mode bypass/complete debounce before acking.
  - confidence: 0.9
- **[P2] Cap relay websocket payloads:** `extensions/slack/src/monitor/relay-source.ts:107-112`
  - body: The new WebSocket client only sets headers and `handshakeTimeout`, so it inherits `ws` 8.21.0 defaults of 100 MiB `maxPayload` and permessage-deflate enabled. Relay frames should be small Slack envelopes; set an explicit small payload cap and disable compression unless the relay contract requires it, so a compromised router or leaked token cannot force large/decompressed messages into the gateway before parse.
  - confidence: 0.82
