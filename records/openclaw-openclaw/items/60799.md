## Summary

Keep open. Current main and the latest release still turn a valid top-level array cron store into an empty job list, so the reported first-write data-loss path remains present rather than implemented or superseded.

## What This Changes

The issue reports a data-loss regression where legacy top-level array `jobs.json` files load as zero cron jobs and the first `cron add` rewrites the store with only the new job.

## Best Possible Solution

Add shared cron-store normalization that accepts both `{ version: 1, jobs: [...] }` and legacy top-level arrays in async/sync loaders and doctor-related paths, then prove legacy jobs survive a load-add-save round trip.

## Reproduction Assessment

Yes. A valid top-level array passes JSON parsing but fails the non-array object guard, loads as an empty store, and `cron add` then persists only the newly-created in-memory job.

## Solution Assessment

Yes. Narrow loader normalization plus focused regression coverage is the maintainable fix for this report; timeout defaults, auth cooldown, and read-merge-write races should remain separate unless required by the preservation test.

## Repair Work Prompt

Fix the legacy cron store data-loss regression for https://github.com/openclaw/openclaw/issues/60799. Current `loadCronStore()` and `loadCronStoreSync()` treat a valid top-level array `jobs.json` as an empty store; the first `cron add` then persists only the new job. Add shared normalization in `src/cron/store.ts` so both `{ version: 1, jobs: [...] }` and legacy top-level `[{...}]` stores load into the same `CronStoreFile` shape, preserve inline runtime state when no sidecar exists, and migrate to the current config/state files on the next save. Make doctor/backup comparison paths see the normalized jobs instead of returning early on empty. Use PR #61113 only as source context; do not broaden this PR into the separate read-merge-write race in #53746 or the timeout/auth cooldown follow-ups. Add focused tests for async load, sync load if applicable, doctor visibility, and a load-add-save round trip that preserves existing legacy jobs. Add a user-facing `CHANGELOG.md` fix entry without inventing bot/maintainer thanks.
