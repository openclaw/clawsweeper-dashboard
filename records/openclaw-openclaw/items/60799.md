- `P0`: The issue describes and current source still supports a cron-store upgrade path that can permanently overwrite existing scheduled-job data.
- `impact:data-loss`: The affected behavior silently drops persisted cron jobs and can overwrite the original `jobs.json` with only newly-added jobs.

## Summary

Keep open: current main and v2026.5.18 still load a valid top-level array cron store as an empty job list, so the reported data-loss path remains source-provable; the focused fix at https://github.com/openclaw/openclaw/pull/61113 was closed unmerged.

## What This Changes

The issue reports a regression where legacy plain-array `jobs.json` cron stores load as zero jobs after upgrade and the first `cron add` overwrites the existing jobs.

## Best Possible Solution

Add a shared cron-store normalizer used by async, sync, and doctor-related load paths that accepts both `{ version: 1, jobs: [...] }` and legacy top-level arrays, then prove legacy jobs survive a load-add-save round trip.

## Reproduction Assessment

Yes, by source inspection. A valid top-level array parses successfully but fails both loader guards because they require a non-array object, after which `cron add` persists only the newly-created in-memory store.

## Solution Assessment

Yes. Narrow loader normalization plus regression coverage is the maintainable fix; timeout defaults, auth cooldown, and broader read-merge-write concurrency should stay separate unless the preservation test proves they share this path.
