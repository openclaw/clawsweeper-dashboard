## Summary

Keep open. Current main still treats a valid top-level array cron store as an empty store, and the latest release tag has the same loader guard, so the reported load-empty then first-write data-loss path remains live.

## What This Changes

The issue requests backward-compatible loading and migration for legacy `jobs.json` files stored as a plain top-level array of cron jobs.

## Best Possible Solution

Add shared cron-store normalization that accepts both `{ version: 1, jobs: [...] }` and legacy top-level arrays in async/sync loaders, then prove legacy jobs survive a load-add-save round trip.

## Reproduction Assessment

Yes. A top-level array passes JSON parsing but becomes `{}` at the non-array object guard, so the loader returns an empty job list and `cron add` persists only the newly-created in-memory job.

## Solution Assessment

Yes. Narrow loader normalization with focused regression coverage is the best fix for this report; timeout defaults, auth cooldown, and broader read-merge-write races should stay separate unless the preservation test exposes them.

## Repair Work Prompt

Fix the cron `jobs.json` legacy plain-array data-loss regression for this issue. Current main still loads a valid top-level array as an empty store, and the first `cron add` then rewrites only the new job. Add a shared normalization path in `src/cron/store.ts` so both `loadCronStore()` and `loadCronStoreSync()` accept legacy `[{...}]` stores as well as the current `{ version: 1, jobs: [...] }` envelope. Preserve the existing `jobs-state.json` split-store behavior, inline-state migration, serialized-store cache, and secure atomic write behavior. Add focused regression tests proving a legacy plain-array store loads its jobs and survives a load-add-save round trip without dropping existing jobs. Check whether `openclaw doctor --fix` needs a targeted test/update so it can see and normalize this shape. Do not bundle changes to cron timeout defaults, auth cooldown handling, or unrelated scheduler catch-up behavior. Add a single-line user-facing `CHANGELOG.md` fix entry.
