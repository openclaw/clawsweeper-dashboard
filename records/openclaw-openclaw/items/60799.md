## Summary

Keep open. Current main and the latest release still convert a valid top-level array cron store into an empty job list, so the reported first-write data-loss path remains present; the direct fix PR was closed unmerged.

## What This Changes

Request backward-compatible cron store loading and migration for legacy top-level array `jobs.json` so upgrades and the first `cron add` do not silently drop existing jobs.

## Best Possible Solution

Add shared cron-store normalization that accepts both `{ version: 1, jobs: [...] }` and legacy top-level arrays in async/sync loaders and doctor paths, then prove legacy jobs survive load-add-save without broadening into the separate read-merge-write race.

## Reproduction Assessment

Yes. A top-level array passes JSON parsing but fails the loader’s object-not-array guard, producing an empty store; `cron add` then appends one job and persists that one-job store.

## Solution Assessment

Yes. Loader normalization plus focused regression coverage is the narrowest maintainable fix for this report; timeout defaults, auth cooldown, and the broader partial-state write race should remain separate unless required by the preservation test.
