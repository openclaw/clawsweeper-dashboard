## Summary

Keep open. Current main and the latest release still load a valid top-level array cron store as an empty job list, and the existing closed focused fix remains useful context but was not merged.

## What This Changes

The issue requests backward-compatible cron store loading for legacy top-level JSON array `jobs.json` files so upgrades do not drop jobs or clobber them on first write.

## Best Possible Solution

Add shared cron-store normalization that accepts both `{ version: 1, jobs: [...] }` and legacy top-level arrays in async/sync loaders and doctor-related paths, then prove legacy jobs survive a load-add-save round trip.

## Reproduction Assessment

Yes by source inspection. A valid top-level array parses successfully but fails the current non-array object guard in both loaders, and the add path then persists the newly-created single-job in-memory store.

## Solution Assessment

Yes. Narrow loader normalization plus focused regression coverage is the maintainable fix; timeout defaults, auth cooldown, and broader read-merge-write concurrency should stay separate unless required by the preservation test.
