## Summary

Keep open. Current main and latest release v2026.4.29 still treat a top-level plain-array cron store as an empty store, so the reported zero-job load and first-write data-loss path remains present; the direct fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to load and preserve legacy `~/.openclaw/cron/jobs.json` files stored as a plain top-level job array instead of silently replacing them with a new empty v1 envelope on the first write.

## Best Possible Solution

Add a shared cron-store normalization path that accepts both `{ version: 1, jobs: [...] }` and legacy top-level `[...]` stores, use it from async and sync loaders while preserving split-state migration, and cover plain-array load plus load-add-save preservation with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. A valid top-level array passes JSON parsing but fails the `!Array.isArray(parsed)` store guard, producing an empty loaded store; then `cron add` appends one job and persists that one-job store.

## Solution Assessment

Yes. Loader normalization plus regression coverage is the narrowest maintainable fix for this issue; the broader read-merge-write race should stay in the related destructive-write discussion unless needed for the legacy-array preservation test.

## Repair Work Prompt

Fix the legacy cron store plain-array data-loss regression tracked by https://github.com/openclaw/openclaw/issues/60799. Current `loadCronStore()` and `loadCronStoreSync()` in `src/cron/store.ts` only accept `{ version: 1, jobs: [...] }`, so a valid legacy top-level `[{...}]` store loads as zero jobs and the first `cron add` persists only the new job. Implement a shared normalization helper that accepts both the current envelope and a legacy top-level array, preserves split-state sidecar migration behavior, and is used by async and sync loaders. Add focused regression coverage for: loading a top-level array with existing jobs; preserving those jobs through a service-level add/save round trip; and doctor seeing/repairing legacy jobs from a top-level array if the existing doctor path needs explicit coverage. Use PR #61113 only as source guidance; do not broaden into the separate read-merge-write race from #53746 unless a minimal merge is required to preserve legacy-array data. Add a user-facing `CHANGELOG.md` fix entry crediting the reporter and useful source PR context, avoiding bot/maintainer credit. Keep changes scoped to cron store/service/doctor tests and the changelog.
