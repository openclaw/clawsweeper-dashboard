## Summary

Keep open: current main still uses in-process promise lanes for embedded agent work, still records the reported prep stages, and only has a narrow accepted-ack yield plus health diagnostics rather than worker isolation, priority scheduling, caching, lazy loading, or load shedding. The open related PR at https://github.com/openclaw/openclaw/pull/78958 is a partial cooperative-yield mitigation, but it does not close the broader minimal-config WebSocket and agent-prep latency report.

## What This Changes

Request to improve Gateway and embedded-agent responsiveness when agent preparation causes 15-100s WebSocket latency and multi-minute turns under a minimal configuration.

## Best Possible Solution

Split this into maintainer-owned performance work: land measured cooperative prep-yield fixes where valid, keep worker/process isolation and queue priority under the broader Gateway architecture discussion, and prove improvements with live WebSocket and agent-turn latency measurements.

No, not for the exact reported timings: this review did not run the reporter's Windows/Node/provider setup. Source inspection does confirm the current in-process queue, agent-prep stages, and event-loop diagnostic paths that make the report credible.

## Solution Assessment
