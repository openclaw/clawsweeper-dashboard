## Summary

Keep open: current main has diagnostics and small yielding around agent dispatch, but I did not find a shipped/current implementation that fully addresses the reported gateway/agent event-loop starvation or WebSocket latency. The requested worker offload, priority scheduling, caching, lazy loading, and degradation behavior are broad performance architecture work that needs maintainer design rather than automatic cleanup.

## What This Changes

The issue reports severe Gateway event-loop saturation with slow WebSocket responses and agent preparation even after reducing channel/model load, and requests worker offload, WebSocket prioritization, agent-prep caching, lazy plugin loading, and load-shedding.

## Best Possible Solution

The desired end state is a maintainer-owned performance plan that preserves the minimal-config evidence, separates Telegram-specific throttling from agent-prep/WebSocket responsiveness, and lands measured hot-path fixes with live latency proof.
