## Summary

Keep open: current main still has a source-visible chat-continuation path where `prependContext` can be converted into hidden runtime context and appended to system-prompt space, matching the reported cache-stability regression.

## What This Changes

The issue requests confirmation and repair of a prompt-cache regression where chat-continuation dynamic context is routed into the system prompt instead of per-turn user context.

## Best Possible Solution

Keep the documented hook contract: only explicit system-context fields should enter system-prompt space, while `prependContext` and volatile reply/conversation metadata remain current-turn user context with regression coverage for auto-reply embedded runs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: auto-reply chat continuations pass distinct `prompt` and `transcriptPrompt`, `prependContext` changes `effectivePrompt`, the delta becomes `runtimeContext`, and that context is appended to the system prompt.

## Solution Assessment

Yes, the requested direction is the maintainable fix boundary. The implementation should separate transcript hiding from system-prompt placement so dynamic per-turn content stays model-visible without invalidating the stable prefix.

## Repair Work Prompt

Fix the chat-continuation prompt-cache regression described in https://github.com/openclaw/openclaw/issues/77431. Preserve the documented hook contract: `prependContext`/`appendContext` and volatile reply/conversation metadata must remain current-turn user context, while only `prependSystemContext`/`appendSystemContext` and true runtime-only events may enter system-prompt space. Keep hidden transcript behavior intact, but do not route ordinary prompt additions from `before_prompt_build` into `buildRuntimeContextSystemContext` on auto-reply embedded runs. Add regression coverage for a chat continuation where `prompt !== transcriptPrompt` and a hook returns `prependContext`, proving the model-visible per-turn context is not appended to the system prompt. Update `CHANGELOG.md` under active fixes because this is user-facing cost/performance behavior. Do not change provider-specific cache policy or add a config option.
