## Summary

Keep open: current main still has a source-level path where dynamic continuation context can be appended to the active session system prompt, even though the plugin hook contract and prompt-cache docs reserve system-prompt space for stable guidance.

## What This Changes

The issue reports that chat-continuation prompt cache hits dropped to 0% because dynamic plugin and conversation metadata now appear in the system prompt.

## Best Possible Solution

Keep this open and change the reply-continuation path so normal dynamic runtimeContext and prependContext reach the model outside system-prompt space, while preserving system-context handling for true runtime-only events.

## Reproduction Assessment

Yes, source-reproducible: a continuation with a body-only transcriptPrompt and additional prepend/runtime context flows through resolveRuntimeContextPromptParts into runtimeContext, then attempt.ts appends that runtimeContext to the active system prompt before submission. I did not run a live Anthropic cache trace in this read-only pass.

## Solution Assessment

No, the current implementation is not the best way to solve it because it preserves transcript cleanliness by mutating the system prompt with per-turn context. The narrower fix is to keep runtime-only events as system context but stop appending normal continuation runtimeContext to the system prompt, relying on hidden/custom or user-side context instead and adding regression coverage.
