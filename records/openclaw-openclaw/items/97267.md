## Summary

Keep open: current main and v2026.6.10 still use a text-block-only extractor, so structured non-image tool-result content can still disappear from the visible output path. The linked open fix candidates are implementation paths, not a reason to close before merge.

## What This Changes

The issue asks OpenClaw to keep structured non-image tool-result content visible as text instead of collapsing to an attachment placeholder.

## Best Possible Solution

Land a narrow helper-level fix that serializes sanitized structured non-image blocks, preserves image/media handling and the visible-output cap, adds regression coverage, and includes redacted clean-runtime proof before closing the issue.

## Reproduction Assessment

Yes, at source level: a result whose content array contains only structured non-image blocks reaches extractToolResultText, which currently returns undefined, and the visible-output handler only emits when that helper returns text. I did not run a live OpenClaw session in this read-only review.

## Solution Assessment

Yes, the helper boundary is the best fix location because the visible output caller already centralizes through extractToolResultText. The open candidate PRs should be reviewed or repaired and landed rather than starting a duplicate fix lane.
