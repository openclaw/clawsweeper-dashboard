## Summary

Keep open. Current main implements the core agent/assistant avatar plumbing and authenticated avatar route, but the full feature request is still incomplete: the browser favicon remains static, Quick Settings still exposes no user display-name editor, the Control UI WebSocket client still does not send a displayName, and the relevant agent/session selectors remain native text-only controls.

## What This Changes

The issue requests per-agent avatar image support, per-agent/browser-tab favicon customization, and a Control UI user display-name override.

## Best Possible Solution

Keep the shipped avatar identity model and narrow the remaining work to a focused Control UI follow-up: update the active favicon from the current agent/avatar through the existing safe avatar route or browser-local data/blob URLs, expose a browser-local display-name editor that feeds chat rendering and Gateway client displayName metadata, and decide whether agent/session pickers should gain avatar-capable custom controls.

## Reproduction Assessment

Not applicable as a runtime failure report, but yes, there is a high-confidence source-inspection path for the remaining gaps: static favicon links, fixed You identity UI with no name input, no browser client displayName in the connect payload, and text-only native agent/session selectors are all present on current main.

## Solution Assessment

Unclear for the full feature as filed because it bundles already-landed avatar plumbing with remaining favicon, display-name, and picker presentation work. The narrowest maintainable solution is a scoped Control UI enhancement that reuses the existing authenticated avatar route and preserves the same-origin/data/blob image policy.
