## Summary

Keep open. Current main still has the strict-mode unclosed reasoning-tag behavior described in the discussion, and the issue is paired with open same-author PR #49996 using closing syntax, so it should not be closed before that PR is reviewed, replaced, or closed.

## What This Changes

The issue requests that Telegram delivery stop losing response text when model output contains literal or malformed angle-bracket reasoning/control tags such as `<think>`.

## Best Possible Solution

Keep this issue open until maintainers decide the strict-mode policy, then land a narrow patch that preserves intended hidden reasoning while preventing user-visible answer text from being silently dropped in Telegram. The already-hardened raw HTML escaping and parse-error fallback paths should remain unchanged unless a separate reproduction shows unescaped tags still reach Telegram.

## Reproduction Assessment

Yes. The focused reproduction is the current strict-mode helper behavior for `Before <think>unclosed content after`, which tests expect to return `Before`; Telegram final text uses the same helper before delivery.

## Solution Assessment

Unclear. Recovering all tail text after an unclosed tag would address the data loss, but it may also reveal reasoning content; the safer path is an explicit final-message policy with targeted reasoning-tag and Telegram split coverage.
