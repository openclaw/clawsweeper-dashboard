## Summary

Keep open. Current main now covers the ordinary Telegram HTML escaping path, but the remaining source-level truncation for an unclosed reasoning tag after visible text is still pinned by tests and needs maintainer policy on strict reasoning redaction versus literal-tag recovery.

## What This Changes

The issue reports Telegram replies being truncated when agent text contains literal angle-bracket tags such as `<think>` under `parse_mode=HTML`.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose the final-message policy, then land a narrow shared-text and Telegram regression fix that preserves reasoning privacy without silently losing genuine answer text.

## Reproduction Assessment

Yes, source-level. `splitTelegramReasoningText("Before <think>unclosed content after")` reaches strict stripping and current tests pin the result to `Before`, though I did not run a live Telegram delivery probe.

## Solution Assessment

No current close is justified. Formatter-level HTML sanitization is already landed, but changing the remaining strict-mode tail behavior requires maintainer policy before a narrow implementation should proceed.
