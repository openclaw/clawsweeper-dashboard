## Summary

Close as implemented: current main and the v2026.5.12 release now escape unsupported Telegram HTML before parse_mode=HTML, while the remaining unclosed-reasoning-tail behavior is documented as a strict privacy boundary rather than the reported HTML parser truncation bug.

## What This Changes

The issue requests that Telegram delivery preserve full agent replies containing literal angle-bracket tags such as <think> by escaping or sanitizing them before parse_mode=HTML.

## Best Possible Solution

Keep the shipped formatter-level Telegram HTML escaping from https://github.com/openclaw/openclaw/pull/80977; any future change to strict reasoning redaction should be a separate product-scoped issue.

## Reproduction Assessment

No for the reported HTML parse path on current main: unsupported tags are escaped before parse_mode=HTML, and current tests plus release content cover that behavior. The separate `Before <think>...` strict-strip case still returns `Before` by documented reasoning privacy policy, not Telegram HTML parsing.

## Solution Assessment

Yes. The merged formatter-level fix is the narrow maintainable solution because it preserves Telegram-supported markup, escapes unsupported text, and keeps the strict reasoning boundary intact instead of removing parse_mode or weakening redaction globally.
