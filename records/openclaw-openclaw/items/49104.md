## Summary

Keep open: current main and the latest stable release still preserve the strict-mode behavior that drops text after an unclosed reasoning tag when visible text already exists, and the linked fix candidate is open, conflicting, and needs maintainer judgment on reasoning-exposure semantics.

## What This Changes

The issue asks OpenClaw to stop Telegram replies from being truncated or dropped when agent output contains literal angle-bracket reasoning/control tags such as `<think>` while Telegram sends HTML-formatted messages.

## Best Possible Solution

Keep this issue as the canonical tracker while maintainers either land a narrow strict-tail recovery fix or replace the conflicting linked fix candidate with a clean PR that preserves visible Telegram answers without leaking genuine leading reasoning.

## Reproduction Assessment

Yes from source inspection: Telegram final-answer splitting uses strict reasoning-tag stripping, and current main plus v2026.6.8 tests still expect text after a visible-prefix unclosed `<think>` to be dropped. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear: the linked recovery approach targets the right data-loss boundary, but maintainers still need to approve the strict-mode rule so the fix does not trade message loss for unintended reasoning exposure.
