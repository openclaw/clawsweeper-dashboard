## Summary

Keep open. Current main already escapes raw Telegram HTML and handles code-formatted literal tags, and the shipped narrow recovery covers fully wrapped unclosed reasoning-only replies, but current docs and tests still intentionally drop unclosed reasoning-tag tails after visible text; the remaining behavior needs maintainer policy before a replacement fix.

## What This Changes

The issue asks Telegram delivery to stop losing response text when assistant output contains literal angle-bracket tags such as `<think>`.

## Best Possible Solution

Keep this as the canonical Telegram literal-tag and strict-redaction tracker until maintainers choose the final-message policy, then land a narrow regression-tested fix that preserves reasoning privacy without silently losing genuine answer text.

## Reproduction Assessment

Yes. The source-level reproduction is `splitTelegramReasoningText("Before <think>unclosed content after")`, which reaches strict stripping and is pinned by current tests to return only `Before`; I did not run a live Telegram probe in this read-only review.

## Solution Assessment

No current close is justified. Raw HTML escaping and fully wrapped unclosed-reply recovery are already on current main, but broad unconditional tail recovery conflicts with the documented strict reasoning-redaction policy.
