## Summary

Keep open: the real-behavior proof gate is now satisfied, but the diff still marks every Feishu authorization-probe hit as a text command even though that probe is intentionally false-positive-prone, so the patch can broaden visible replies for ordinary chat turns.

## What This Changes

The PR adds `CommandSource: "text"` to Feishu inbound contexts when command-authorization probing runs and updates Feishu command-authorization tests.

## Best Possible Solution

Set Feishu text-command metadata only from actual explicit text command detection, keep broad authorization probing separate, and add coverage for both real slash commands and inline false positives.

## Reproduction Assessment

Yes, source-level: current main can finalize Feishu command-authorized turns without `CommandSource`, while reply delivery only forces automatic visible replies for explicit command turns. The contributor also supplied redacted live Feishu DM logs showing the after-fix `/models` path dispatching one reply.

## Solution Assessment

No, not as written: using `shouldComputeCommandAuthorized` as the `CommandSource` guard is broader than the command-turn contract. The safer fix is to derive `CommandSource: "text"` from actual whole-message text-command detection and leave inline authorization probes as normal turns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Gate CommandSource on actual text commands:** `extensions/feishu/src/bot.ts:1300`
  - body: The added spread keys off `shouldComputeCommandAuthorized`, but that predicate intentionally returns true for inline tokens such as `hey /status` so channels can compute authorization. If such a Feishu message is authorized, reply delivery will treat it as an explicit text command and bypass `message_tool_only`, changing ordinary chat visibility. Gate this marker on actual text-command detection instead.
