## Summary

Keep open: the Feishu regression is source-plausible, but this PR should not merge as-is because the new marker is broader than the text-command contract and after-fix real Feishu proof is not attached.

## What This Changes

The PR adds `CommandSource: "text"` to Feishu inbound contexts whenever command authorization probing runs and adds assertions in Feishu command-authorization tests.

## Best Possible Solution

Narrow Feishu’s command metadata so only actual explicit text-command turns set `CommandSource: "text"`, keep ordinary chat turns under existing visible-reply defaults, and attach redacted after-fix Feishu proof before merge.

## Reproduction Assessment

Yes, there is a high-confidence source-level path: current Feishu code can set `CommandAuthorized` without `CommandSource`, while reply delivery requires explicit command metadata to bypass `message_tool_only`. I did not run a live Feishu DM in this read-only review.

## Solution Assessment

No, not as written: using `shouldComputeCommandAuthorized` as the `CommandSource` guard is broader than the command-turn contract. A safer fix should derive the marker from an actual explicit text command, not from the coarse authorization-probe predicate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Mark only real text commands as CommandSource:** `extensions/feishu/src/bot.ts:1300`
  - body: `shouldComputeCommandAuthorized` is intentionally a coarse probe and returns true for inline command-token false positives, not only explicit text commands. Setting `CommandSource: "text"` from that predicate makes any authorized Feishu message with slash-like text resolve as an explicit command turn, which bypasses `message_tool_only` visible-reply suppression for ordinary group chat turns. Please gate this on an actual text command turn instead of the broad auth-probe signal.
  - confidence: 0.89
