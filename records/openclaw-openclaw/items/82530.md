## Summary

Keep open: the regression and after-fix Feishu proof are credible, but the patch still has a blocking correctness issue. It marks every coarse command-authorization probe hit as a text command, while that probe intentionally includes inline false positives, so authorized ordinary chat like `hey /status` can be treated as an explicit command turn and bypass `message_tool_only` reply suppression.

## What This Changes

The PR adds `CommandSource: "text"` to Feishu inbound contexts when command authorization probing runs and updates Feishu command-authorization tests.

## Best Possible Solution

Set Feishu text-command metadata only for actual control-command messages, keep broad authorization probing for access checks, and add a regression test for inline slash-token false positives.

## Reproduction Assessment

Yes, source-level: current main can finalize Feishu turns with `CommandAuthorized` but no `CommandSource`, while reply delivery only forces visible replies for explicit command turns. The contributor also supplied redacted after-fix Feishu DM logs, but I did not run the live Feishu setup myself.

## Solution Assessment

No, not as written: the fix uses the coarse authorization-probe predicate as the source marker. The safer solution is to derive `CommandSource: "text"` from actual text-command detection, such as `isControlCommandMessage`, while leaving coarse probing available for authorization checks.
