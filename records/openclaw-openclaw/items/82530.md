## Summary

Keep open: the reported Feishu command reply regression is source-reproducible, but this PR currently marks a coarse authorization probe as an explicit text command and still lacks after-fix real Feishu proof.

## What This Changes

The PR adds `CommandSource: "text"` to Feishu finalized inbound contexts when command authorization probing runs and updates Feishu command-authorization tests.

## Best Possible Solution

Set text-command metadata only for actual explicit slash-command turns, preserve ordinary chat delivery defaults, and add redacted after-fix Feishu proof before merge.

## Reproduction Assessment

Yes, source-level: current `main` can finalize Feishu turns with `CommandAuthorized` but no `CommandSource`, while reply delivery only forces automatic source replies for explicit command turns. I did not run a live Feishu DM.

## Solution Assessment

No, not as written: using `shouldComputeCommandAuthorized` as the `CommandSource` guard is broader than the command-turn contract. The safer fix should derive `CommandSource: "text"` from an actual explicit text command, not from the coarse probe predicate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Gate CommandSource on actual text commands:** `extensions/feishu/src/bot.ts:1300`
  - body: At this line, any message that triggers `shouldComputeCommandAuthorized` gets `CommandSource: "text"`. That helper deliberately returns true for inline command-token false positives such as `hey /status`, so an authorized Feishu group message can be treated as an explicit command turn and bypass `message_tool_only` visible-reply suppression. Please gate the marker on an actual text command turn instead of the coarse authorization-probe predicate.
  - confidence: 0.89
