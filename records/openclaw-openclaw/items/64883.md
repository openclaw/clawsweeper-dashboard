## Summary

Keep open. Current main still treats `exec`/`bash` shell activity as side-effectful by tool name, so a dummy no-op shell call plus plan-only prose does not enter the strict-agentic planning retry path; the adjacent CJK continuation PR touches the same guard but does not address this shell no-op gap.

## What This Changes

Strengthen strict-agentic planning-only detection so dummy or no-op tool calls like `sleep` do not count as concrete progress.

## Best Possible Solution

Add a narrow command/result-aware progress classifier for shell no-op calls, with regression coverage that catches dummy `exec`/`bash` planning turns while preserving fail-closed handling for real side effects.

## Reproduction Assessment

Yes, by source inspection rather than an executed test. A strict-agentic attempt with one `exec` or `bash` tool meta and plan-only text reaches the non-safe-tool and side-effect guards, so `resolvePlanningOnlyRetryInstruction` returns `null` instead of retrying.

## Solution Assessment

No, a broad tool-name whitelist that includes shell is not the safest fix. The maintainable path is a narrow no-op/result-aware shell classifier plus negative coverage proving real shell side effects remain replay-unsafe.

Fix the strict-agentic planning-only detector so a dummy/no-op shell tool call, such as a single `exec` or `bash` command running `sleep`, does not count as concrete progress when the assistant text is still planning-only. Do not add `exec`/`bash` wholesale to the retry-safe tool-name whitelist. Prefer a narrow command/result-aware classifier that treats tightly defined shell no-ops as non-progress while preserving replay-unsafe/fail-closed behavior for arbitrary shell commands and durable side effects.
