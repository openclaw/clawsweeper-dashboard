## Summary

Keep open. Current main still has the dummy shell-call gap: the strict-agentic retry path only allows the single-action loophole for read/search-style tools, while `exec`/`bash` are treated as potentially side-effectful by name and the resolver has no command/result-aware way to distinguish `sleep 0.1` from real shell progress.

## What This Changes

The issue requests strengthening the strict-agentic planning-only detector so a dummy or no-op tool call cannot make a plan-only assistant turn count as completed work.

## Best Possible Solution

Add a conservative structural progress classifier for the strict-agentic planning-only path. It should recognize only known no-op/no-output shell calls as non-progress, keep unknown shell/messaging/write/session actions replay-unsafe by default, and lock the behavior with a regression test for a dummy `sleep 0.1` turn followed by plan-only prose.

## Reproduction Assessment

Yes. A high-confidence helper-level reproduction can construct a strict-agentic attempt with one `exec` or `bash` tool meta for a no-op command, planning-only assistant text, and no real output; current main returns no retry because the tool is treated as non-plan side-effectful activity before prose classification.

## Solution Assessment

No for a plain tool-name whitelist, because adding `exec` or `bash` to a safe-name list would risk retrying real side effects. The best fix is command/result-aware classification that catches the dummy command while preserving fail-closed replay safety for shell activity.
