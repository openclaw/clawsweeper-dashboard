## Summary

Keep open. Current main still treats `bash`/`exec` as side-effectful progress by tool name, so a dummy shell call with plan-only prose stays outside the strict-agentic planning retry path; the related CJK and GPT-4o follow-ups do not supersede this shell no-op gap.

## What This Changes

The issue asks OpenClaw to treat dummy/no-op tool calls as non-progress so strict-agentic planning-only retries still fire.

## Best Possible Solution

Add a narrow command/result-aware progress classifier for shell no-op calls, with regression coverage for dummy `exec`/`bash` planning turns and negative coverage preserving fail-closed real shell side effects.

## Reproduction Assessment

Yes, by source inspection rather than a live run. A strict-agentic attempt with one `bash` or `exec` tool meta and plan-only assistant text reaches the non-safe-tool and side-effect guards, so `resolvePlanningOnlyRetryInstruction` returns `null` instead of firing the retry.

## Solution Assessment

No current implementation fully solves it. A broad tool-name whitelist would be risky for shell commands; the safer fix is a narrow no-op/result-aware shell classifier that keeps real shell side effects fail-closed.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/64883. In current main, strict-agentic planning-only retry still treats any `bash`/`exec` tool meta as side-effectful progress, so a dummy no-op shell call such as `sleep 0.1` plus plan-only prose bypasses the retry and blocked-exit path. Add a narrow classifier that can treat known no-op shell progress as non-progress for planning-only retry while preserving fail-closed handling for real shell side effects. Do not broadly add `bash` or `exec` to the retry-safe tool-name allowlist. If the existing attempt `toolMetas` lacks enough command/result detail, carry only the minimal structured fact needed from tool start/result through the existing embedded-runner metadata path. Add regression coverage for a single dummy `exec`/`bash` planning turn causing retry and eventual strict-agentic blocked exit, plus negative coverage that real shell commands remain side-effectful and are not replayed. Coordinate with the open CJK continuation PR if it has landed or conflicts. Add a changelog entry if the fix changes user-visible runtime behavior.
