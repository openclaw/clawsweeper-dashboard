## Summary

Keep open. Current main has a single-action planning retry path, but it only treats read/search-style tools as retry-safe and still classifies `exec`/`bash` as mutating without command/result awareness, so the exact dummy shell/no-op gap remains unresolved and untested.

## What This Changes

The issue asks for strict-agentic planning-only retry logic to distinguish real progress from dummy/no-op tool calls so a lone `sleep`-style tool call cannot bypass retry and blocked-exit behavior.

## Best Possible Solution

Add a conservative structural progress classifier for strict-agentic planning-only retry. It should preserve shell and messaging side effects as replay-unsafe by default, but allow well-defined no-op shell/result cases to be treated as non-progress and covered by retry-then-block regression tests.

## Reproduction Assessment

Yes. A high-confidence unit-level repro can construct a strict-agentic OpenAI GPT-5 attempt with one `exec`/`bash` tool meta, planning prose, and current replay metadata; current source returns no retry because shell is excluded from the safe bypass and marked potentially side-effectful.

## Solution Assessment

No for a broad tool-name whitelist as written in the issue. The safer fix is command/result-aware classification that catches a dummy `sleep`/empty-output shell call without weakening replay safety for real shell commands.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/64883. Current main only lets the strict-agentic single-action planning retry bypass run for read/search-style tools, while `exec`/`bash` are always treated as mutating, so a dummy/no-op shell command cannot be classified as non-progress. Add a conservative structural progress classifier for the planning-only retry path that catches well-defined no-op shell calls such as `sleep 0.1` with no meaningful output, but does not make arbitrary shell commands retry-safe by tool name. Prefer threading enough command/result metadata into `resolvePlanningOnlyRetryInstruction` or adjacent replay metadata rather than broad prompt rewrites. Add regression coverage proving a single dummy shell call plus planning prose retries and then reaches the strict-agentic blocked exit, and preserve existing tests for real side-effectful tools, unclassified tools, messaging, cron, writes, sessions, and multi-tool progress. Add a single-line user-facing `CHANGELOG.md` fix entry in the active version with `Thanks @100yenadmin`. Do not change bundled provider defaults or make `exec`/`bash` generally replay-safe.
