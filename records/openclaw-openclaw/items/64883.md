## Summary

Keep open. Current main still does not classify dummy shell/no-op tool calls as non-progress for strict-agentic planning retry; `exec`/`bash` remain side-effectful by tool name, while the single-action retry bypass only allows read/search/list-style tools.

## What This Changes

The issue asks for strict-agentic planning-only detection to reject a dummy no-op tool call such as `sleep 0.1` instead of treating any tool use as progress.

## Best Possible Solution

Add a command/result-aware progress classifier that treats tightly defined no-op shell calls as non-progress while keeping arbitrary shell and durable side effects replay-unsafe.

## Reproduction Assessment

Yes by source inspection. A focused unit repro can build a strict-agentic attempt with one `exec`/`bash` meta and plan-only prose; current source returns no planning retry because shell is excluded from the safe single-action bypass and marked side-effectful.

## Solution Assessment

No for a broad tool-name whitelist that includes shell commands. The safer fix is a narrow no-op/result-aware classifier with regression coverage for dummy shell and negative cases for real shell side effects.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/64883. In the strict-agentic embedded Pi planning-only guard, stop treating a dummy/no-op shell tool call as sufficient progress when the assistant text is still plan-only. Keep replay safety fail-closed for arbitrary `exec`/`bash`, messaging, cron, write/edit/apply_patch, sessions, and other durable side effects; do not add a broad shell whitelist. Prefer threading enough command/result metadata into the attempt replay/progress classifier to recognize tightly defined no-op shell cases such as `sleep 0.1`, and add regression coverage showing retry/exhaustion for the dummy shell case plus negative cases for real shell side effects. Do not change provider activation policy, continuation-mode config, or unrelated CJK planning detectors.
