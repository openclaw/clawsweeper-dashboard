## Summary

Keep open. Current `main` still excludes `exec`/`bash` from the single-action retry bypass and marks shell tools replay-unsafe by name, so plan-only prose after a dummy shell call still returns no retry and the requested regression test is absent.

## What This Changes

The issue asks for strict-agentic planning-only detection to catch dummy or no-op tool-call turns instead of treating any tool call as progress.

## Best Possible Solution

Add a conservative structural progress classifier for strict-agentic planning retry that recognizes well-defined no-op shell/result cases while preserving replay-unsafe handling for real shell, messaging, cron, write/edit, and session-spawn side effects.

## Reproduction Assessment

Yes. A source-level repro can call `resolvePlanningOnlyRetryInstruction` with an actionable strict-agentic GPT attempt containing one `exec`/`bash` tool meta, planning prose, and replay metadata from `buildAttemptReplayMetadata`; current source returns `null` because shell is not bypass-safe and is marked side-effectful.

## Solution Assessment

No for a broad whitelist that includes shell tools. The safer fix is command/result-aware no-op classification with tests proving `sleep`-style dummy calls retry while real side effects remain replay-unsafe.

## Repair Work Prompt

Fix the strict-agentic planning-only dummy shell evasion tracked by #64883. Add a conservative structural progress classifier so a single no-op shell tool call such as `exec`/`bash` with `sleep 0.1` followed by plan-only prose still triggers the planning-only retry and eventual strict-agentic blocked exit, while real shell side effects, messaging, cron, write/edit/apply_patch, sessions_spawn, and unclassified tools remain replay-unsafe. Prefer threading existing tool args/result or mutation metadata through the embedded runner rather than adding a broad shell whitelist. Add focused regression coverage in `src/agents/pi-embedded-runner/run.incomplete-turn.test.ts` and, if mutation classification changes, `src/agents/tool-mutation.test.ts`; include a user-facing `CHANGELOG.md` fix entry unless maintainers decide this is internal-only. Do not change provider activation, retry limits, or broad execution-contract policy.
