## Summary

Keep open. Current main and the latest release still classify `bash`/`exec` as side-effectful by tool name, so a dummy shell call with plan-only prose remains outside the strict-agentic planning retry path; the related CJK, update_plan, and GPT-4o threads do not supersede this specific shell no-op gap.

## What This Changes

The issue asks for stricter strict-agentic planning-only detection so dummy shell tool calls do not count as real progress.

## Best Possible Solution

Add a narrow command/result-aware progress classifier for shell no-op calls, with regression coverage for dummy `exec`/`bash` planning turns and negative coverage preserving fail-closed real shell side effects.

## Reproduction Assessment

Yes, by source inspection. A strict-agentic attempt with one `bash` or `exec` tool meta and plan-only assistant text reaches the non-safe-tool and side-effect guards, so `resolvePlanningOnlyRetryInstruction` returns `null` instead of firing the retry.

## Solution Assessment

No current implementation fully solves it. A broad tool-name whitelist would be risky for shell commands; the safer fix is a narrow no-op/result-aware shell classifier that keeps real shell side effects fail-closed.
