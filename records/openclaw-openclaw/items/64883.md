## Summary

Keep open. Current main still classifies `exec`/`bash` by tool name as side-effectful progress, so a dummy shell call with plan-only prose bypasses the strict-agentic planning retry instead of being caught by the advertised plan-only guard.

## What This Changes

The issue asks for strict-agentic planning-only detection to catch dummy/no-op tool calls instead of treating any shell tool call as real progress.

## Best Possible Solution

Add a narrow command/result-aware progress classifier for shell no-op calls, with regression coverage for dummy `exec`/`bash` planning turns and negative coverage preserving fail-closed real shell side effects.

## Reproduction Assessment

Yes, by source inspection. A strict-agentic attempt with one `exec` or `bash` tool meta and plan-only text reaches the non-safe-tool and side-effect guards, so the retry instruction resolves to `null` instead of firing.

## Solution Assessment

No, not as a broad tool-name whitelist that includes shell. The safer maintainable fix is a narrow no-op/result-aware shell classifier that catches dummy progress while preserving fail-closed behavior for real side effects.

Current main treats `exec`/`bash` tool metas as non-retry-safe and potentially side-effectful by tool name, so a dummy shell command such as `sleep 0.1` plus plan-only prose bypasses the strict-agentic planning-only retry.
