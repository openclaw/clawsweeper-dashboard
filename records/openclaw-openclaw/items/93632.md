## Summary

Keep this PR open: the latest revision moves the HUD timing toward the right owner and removes the new config key, but it still misses live config refresh, can stall independent Voice Wake startup on a gateway config RPC, and lacks real macOS HUD proof.

## What This Changes

The PR changes `apps/macos/Sources/OpenClaw/VoiceWakeRuntime.swift` so Voice Wake reads gateway `talk.silenceTimeoutMs` on start, stores it as a mutable capture silence window, and falls back to 2000 ms.

## Best Possible Solution

Keep Voice Wake as the config consumer, but make the HUD window update on Talk config changes without blocking recognizer startup, preserve the 2000 ms default when unset, and require focused Swift coverage plus live macOS HUD proof before merge.

## Reproduction Assessment

Yes from source, but not from a live macOS run. Current main hardcodes the Voice Wake HUD post-trigger window at 2.0 seconds, while `talk.silenceTimeoutMs` currently flows through Talk config and Talk mode only.

## Solution Assessment

No. The revised direction is close, but the start-only resolver misses config refresh and couples independent Voice Wake startup to a gateway RPC; the narrower fix is a Voice Wake-owned cached/default resolver that refreshes safely and is proven on macOS.
