## Summary

Keep open: current main and v2026.6.8 still hardcode the HUD silence window, and this PR is the candidate fix, but it is not merge-ready because running config edits are not actually wired to refresh Voice Wake, docs still describe the old scope, shared-key compatibility needs maintainer acceptance, and native macOS HUD proof is absent.

## What This Changes

The branch changes `apps/macos/Sources/OpenClaw/VoiceWakeRuntime.swift` so macOS Voice Wake asynchronously reads the HUD post-trigger silence window from existing gateway `talk.silenceTimeoutMs`, keeps a 2000 ms fallback, and cancels pending fetches on stop.

## Best Possible Solution

Preserve the 2000 ms HUD fallback, either wire a real talk-config-change refresh and update docs for the shared `talk.silenceTimeoutMs` contract or choose a separate maintainer-approved HUD setting, then require redacted native macOS HUD proof before merge.

## Reproduction Assessment

Yes from source, but not from a live macOS run. Current main and v2026.6.8 hardcode the Voice Wake HUD post-trigger silence window at 2.0 seconds while `talk.silenceTimeoutMs` is parsed only through Talk config paths.

## Solution Assessment

No for merge readiness. Moving consumption into `VoiceWakeRuntime` is a plausible owner-boundary fix, but the current branch lacks a real config-edit refresh trigger, user-facing docs for the expanded key scope, maintainer acceptance of the compatibility change, and native HUD proof.
