## Summary

Keep open: the repaired branch moved away from a new config key and now uses the right Voice Wake owner boundary, but it is still not merge-ready because live config edits do not have a proven refresh caller, the expanded `talk.silenceTimeoutMs` contract is undocumented, and the PR still lacks native macOS HUD behavior proof.

## What This Changes

The PR changes `apps/macos/Sources/OpenClaw/VoiceWakeRuntime.swift` so macOS Voice Wake asynchronously reads the HUD capture silence window from existing gateway `talk.silenceTimeoutMs`, keeps a 2000 ms fallback, and cancels pending fetches on stop.

## Best Possible Solution

Preserve the 2000 ms HUD fallback, either wire a real talk-config-change refresh and document the shared `talk.silenceTimeoutMs` contract or choose a separate maintainer-approved HUD setting, then require redacted native macOS HUD proof before merge.

## Reproduction Assessment

Yes from source inspection: current main and v2026.6.9 hardcode the HUD post-trigger silence threshold at 2.0 seconds while `talk.silenceTimeoutMs` is parsed only through Talk config paths. I did not run a live macOS HUD scenario in this read-only review.

## Solution Assessment

No for merge readiness. Moving consumption into `VoiceWakeRuntime` is a plausible owner-boundary fix, but the current branch lacks a real config-edit refresh trigger, user-facing docs for the expanded key scope, maintainer acceptance of the compatibility change, and native HUD proof.
