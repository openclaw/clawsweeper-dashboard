## Summary

Keep open: the branch is the active candidate for the linked macOS Voice Wake HUD timeout feature, but it still needs native HUD proof, a refresh path for live config edits, docs for the expanded `talk.silenceTimeoutMs` scope, and maintainer acceptance of the compatibility change.

## What This Changes

The PR changes `apps/macos/Sources/OpenClaw/VoiceWakeRuntime.swift` so macOS Voice Wake asynchronously fetches gateway `talk.silenceTimeoutMs`, uses it as a mutable HUD post-trigger silence window, and cancels pending fetches on stop.

## Best Possible Solution

Preserve the 2000 ms HUD fallback, either document and prove the shared `talk.silenceTimeoutMs` contract or choose a separate maintainer-approved HUD setting, then add a real refresh trigger and native macOS HUD proof before merge.

## Reproduction Assessment

Yes, from source inspection: current main and v2026.6.9 hardcode the Voice Wake HUD post-trigger silence threshold at 2.0 seconds while `talk.silenceTimeoutMs` is parsed only through Talk config paths. I did not establish a live macOS HUD reproduction in this read-only review.

## Solution Assessment

No for merge readiness. Moving consumption into `VoiceWakeRuntime` is a plausible owner-boundary fix, but the current branch still lacks an actual config-edit refresh trigger, docs for the expanded key scope, accepted upgrade semantics, and native HUD proof.
