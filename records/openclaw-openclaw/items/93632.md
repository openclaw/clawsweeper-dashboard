## What This Changes

The PR modifies macOS `VoiceWakeRuntime` so the wake-word HUD asynchronously reads gateway `talk.silenceTimeoutMs` for its post-trigger capture silence window and cancels the fetch on stop.

## Best Possible Solution

Use Voice Wake-owned config consumption only after preserving upgrade semantics, wiring a real refresh path or narrowing the claim, updating docs, and attaching native macOS HUD proof.

## Reproduction Assessment

Yes from source inspection, but not from a live macOS run. Current main and `v2026.6.9` hardcode the HUD post-trigger silence window at 2.0 seconds while `talk.silenceTimeoutMs` is parsed only through Talk config paths.

## Solution Assessment

No for merge readiness. Moving consumption into `VoiceWakeRuntime` is the right owner-boundary direction, but this branch still needs upgrade-safe semantics, real refresh wiring or a narrower claim, docs, and native HUD proof.
