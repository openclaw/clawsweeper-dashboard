## Summary

Keep open. Current main still validates and exposes `ui.seamColor` on backend config paths, but the Control UI bootstrap contract, HTTP payload, frontend bootstrap state, and theme application path still do not carry or apply the value; the only linked fixing PR is closed unmerged after review identified build-breaking and theme-scope problems.

## What This Changes

The issue requests that the Control UI apply the configured `ui.seamColor` as the frontend accent color instead of silently keeping hardcoded theme accents.

## Best Possible Solution

Wire `ui.seamColor` through the authenticated Control UI bootstrap payload and apply validated accent CSS variables only when configured, while preserving current theme defaults when unset.

## Reproduction Assessment

Yes, at source level. Current main accepts and exposes `ui.seamColor` through backend config paths, but the Control UI bootstrap contract, endpoint payload, frontend state, and theme-token path have no field or application logic for it.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing authenticated Control UI bootstrap and theme-token application path rather than making the browser depend on a separate `talk.config` fetch for initial theme state.
