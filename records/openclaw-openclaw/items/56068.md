## Summary

Keep this open. Current main still validates and exposes `ui.seamColor`, and native clients consume it, but the browser Control UI bootstrap contract, endpoint payload, bootstrap loader, and theme application path still do not carry or apply that value; the linked fix PR #56103 is closed unmerged.

## What This Changes

The issue asks for the browser Control UI to honor `ui.seamColor` from `openclaw.json` as the UI accent color instead of keeping hardcoded theme accents.

## Best Possible Solution

Add a narrow authenticated Control UI runtime color path: include optional `ui.seamColor` in the bootstrap config, normalize it on the frontend, derive the related accent CSS variables only when configured, and preserve current built-in/custom-theme behavior when it is absent.

## Reproduction Assessment

Yes. The issue has concrete steps, and static inspection of current main confirms the configured value is accepted and exposed by backend/native paths but never reaches or affects the browser Control UI.

## Solution Assessment

Yes. The narrowest maintainable fix is to use the existing authenticated bootstrap endpoint for this UI runtime setting and apply scoped CSS variables in the frontend; using `talk.config` from the browser would add an unnecessary WebSocket dependency for initial theme state.
