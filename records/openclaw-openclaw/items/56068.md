## Summary

Keep open. Current main still accepts and exposes `ui.seamColor` through backend/native paths, but the browser Control UI bootstrap contract, endpoint payload, frontend loader, and theme application code still do not transport or apply the configured color; the linked implementation PR #56103 is closed unmerged.

## What This Changes

The issue requests that the browser Control UI honor `ui.seamColor` from `openclaw.json` as the UI accent color instead of keeping hardcoded theme accents.

## Best Possible Solution

Add a narrow authenticated Control UI runtime color path: include optional `ui.seamColor` in the bootstrap config, normalize it on the frontend, derive related accent CSS variables only when configured, and preserve current built-in/custom-theme behavior when it is absent.

## Reproduction Assessment

Yes. The issue gives concrete steps, and static inspection of current main confirms the configured value is accepted and exposed by backend/native paths but never reaches or affects the browser Control UI.

## Solution Assessment

Yes. The best fix is the existing authenticated bootstrap endpoint plus frontend CSS-token application; using `talk.config` from the browser would add an unnecessary WebSocket dependency for initial theme state.
