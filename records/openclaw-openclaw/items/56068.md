## Summary

Keep open. Current main still accepts and exposes `ui.seamColor` on backend config paths, but the Control UI bootstrap contract, gateway bootstrap response, frontend bootstrap state, and theme application path do not carry or apply the value; the linked fixing PR is closed unmerged.

## What This Changes

The issue asks for `ui.seamColor` in `openclaw.json` to update Control UI accent, button, sidebar, selection, and focus colors instead of leaving the hardcoded red theme.

## Best Possible Solution

Wire `ui.seamColor` through the authenticated Control UI bootstrap payload, validate and normalize it in the browser, apply derived accent CSS variables only when configured, and update tests/docs/changelog around that contract.

## Reproduction Assessment

Yes, at source level. Adding `ui.seamColor` reaches validation and `talk.config`, but the authenticated Control UI bootstrap and theme application path has no field or CSS-variable update, so the browser cannot observe it.

## Solution Assessment

Yes. The narrowest maintainable fix is to use the existing authenticated Control UI bootstrap endpoint and theme-token application path; adding a browser dependency on `talk.config` is unnecessary for initial theme state.
