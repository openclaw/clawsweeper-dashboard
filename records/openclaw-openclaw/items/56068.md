## Summary

Keep open. Current main and the latest shipped release still accept and expose `ui.seamColor` on backend config paths, but the Control UI bootstrap contract, endpoint payload, frontend bootstrap state, and theme application path do not carry or apply it; the linked fix attempt at https://github.com/openclaw/openclaw/pull/56103 is closed unmerged.

## What This Changes

Make the existing `ui.seamColor` config setting affect Control UI accent styling instead of leaving web UI accents fixed by theme CSS.

## Best Possible Solution

Wire `ui.seamColor` through the authenticated Control UI bootstrap payload, validate and normalize it in the browser, apply derived accent CSS variables only when configured, preserve the unset/default path, and update tests/docs/changelog.

## Reproduction Assessment

Yes, at source level. The setting reaches schema validation and `talk.config`, but the authenticated Control UI bootstrap and theme application paths have no field or CSS-variable update, so the browser cannot observe it.

## Solution Assessment

Yes. The maintainable fix is to use the existing authenticated Control UI bootstrap endpoint and theme-token application path; making the web UI call `talk.config` for initial theme state would add an unnecessary second config transport.
