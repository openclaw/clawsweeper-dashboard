## Summary

Keep this open: current main and the latest release still accept and expose `ui.seamColor` on the backend but do not pass or apply it through the Control UI bootstrap path, and both related fix PRs are closed unmerged.

## What This Changes

The issue requests that the existing `ui.seamColor` config value affect Control UI accent CSS instead of being silently ignored by the frontend.

## Best Possible Solution

Wire the existing `ui.seamColor` through the Control UI bootstrap contract and gateway response, apply validated accent CSS variables in the frontend, and cover configured plus absent-color fallback behavior with focused tests and real UI proof.

## Reproduction Assessment

Yes, by source inspection. Current main validates and exposes `ui.seamColor` through `talk.config`, while the Control UI bootstrap payload and loader omit it, so adding `ui.seamColor` and opening the web UI cannot affect the current accent CSS.

## Solution Assessment

Yes. The best fix is to use the existing authenticated Control UI bootstrap endpoint rather than adding a separate frontend `talk.config` fetch, then apply and remove the derived CSS variables in one frontend bootstrap path.
