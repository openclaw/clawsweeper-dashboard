## Summary

Keep open. Current main still ships the WhatsApp plugin dependency path from `@openclaw/whatsapp` to Baileys and then to the pinned GPL-3.0 `@whiskeysockets/libsignal-node` tarball; the report is concrete, but the remediation needs maintainer/legal distribution judgment rather than conservative cleanup automation.

## What This Changes

The issue asks OpenClaw to mitigate the GPL-3.0 libsignal dependency risk introduced through the WhatsApp plugin's Baileys dependency.

## Best Possible Solution

Keep this as the canonical tracking issue until maintainers choose an explicit WhatsApp plugin license posture, then implement that choice narrowly across the plugin manifest/package, lockfile, release packaging, docs, and any dependency-policy checks.

Yes. Static inspection of current main reproduces the dependency path from `@openclaw/whatsapp` to Baileys and then to a pinned `@whiskeysockets/libsignal-node` package whose upstream metadata declares GPL-3.0; no live WhatsApp account is needed.

Unclear. The report identifies a real dependency and distribution concern, but it does not define the policy-approved remediation, so an automated dependency swap, removal, or disablement would be premature.
