## Summary

Keep this open: current main and the latest release still contain the reported WhatsApp dependency path from `@openclaw/whatsapp` to Baileys and then to a pinned GPL-3.0 libsignal package. This is a concrete license/supply-chain compliance concern, but the mitigation requires maintainer/legal distribution judgment rather than conservative automated cleanup.

## What This Changes

The issue reports that the WhatsApp plugin pulls in a GPL-3.0 libsignal component and asks for the license infection risk to be addressed.

## Best Possible Solution

Maintain this as the canonical issue until maintainers choose an explicit license posture for the WhatsApp plugin, then apply the chosen mitigation narrowly: replace the dependency, isolate or disable the affected distribution path, or document an approved distribution model with matching package, lockfile, and docs updates.

## Reproduction Assessment

Yes. Static inspection of current main and `v2026.4.29` reproduces the path from `@openclaw/whatsapp` to `@whiskeysockets/baileys` and then to pinned `@whiskeysockets/libsignal-node` declaring GPL-3.0; no live WhatsApp account is required.

## Solution Assessment

Unclear. The report identifies a real dependency path, but an automated dependency removal or replacement would be premature because the best mitigation depends on maintainer/legal distribution policy.
