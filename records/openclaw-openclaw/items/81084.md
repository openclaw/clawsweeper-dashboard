## Summary

Keep open: current main still routes Microsoft Teams channel-thread replies into thread-qualified sessions with no supported channel-memory opt-out, and the request is a new Teams session-routing policy/config decision rather than an already-fixed cleanup item.

## What This Changes

The issue requests an opt-in Microsoft Teams session-routing policy so channel-bound agents can keep canonical channel memory instead of always forking replies into per-thread sessions.

## Best Possible Solution

Add a Teams-owned opt-in session-routing policy that keeps thread isolation as the default, supports channel-memory mode at the maintainer-approved scope, and updates schema, docs, and focused regression tests.

## Reproduction Assessment

Yes for the current behavior: source inspection shows Teams channel replies are routed through a helper that appends a thread id when present, and no supported config branch keeps the canonical channel key. This reproduces the missing opt-out, not a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the policy contract: the requested direction is reasonable if it stays Teams-owned and default-safe, but the config name, scope, and precedence need product review before implementation.
