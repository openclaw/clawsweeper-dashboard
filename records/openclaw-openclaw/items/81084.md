## Summary

Keep open: current main and the latest release still thread-qualify Microsoft Teams channel replies and do not expose a channel-vs-thread session policy; the nested-key bug is fixed separately, but the requested opt-out remains a product/config decision.

## What This Changes

The issue requests an opt-in Microsoft Teams session-routing policy that keeps channel-bound agents on the canonical channel session key instead of always forking channel-thread replies into thread-specific sessions.

## Best Possible Solution

Add a maintainer-approved Teams-owned policy that preserves per-thread isolation by default while allowing explicit channel-session routing for channel-bound agents that intentionally share memory.

## Reproduction Assessment

Yes at source level: current Teams channel replies pass `conversationMessageId` or `replyToId` into `resolveMSTeamsRouteSessionKey`, which appends a `:thread:` suffix. I did not run a live Teams room reproduction in this read-only review.
