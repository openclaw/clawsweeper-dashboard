## Summary

Keep open: current main still routes Microsoft Teams channel replies into thread-qualified session keys and exposes no Teams/session config to preserve canonical channel memory. This is a new session-routing policy and config decision, not an already-implemented cleanup item.

## What This Changes

The issue asks for a Microsoft Teams session-routing policy that lets channel-bound agents keep canonical channel memory instead of always forking replies into per-thread sessions.

## Best Possible Solution

Add a Teams-owned session-routing policy that defaults to current per-thread isolation, supports an explicit channel-memory mode at the approved scope, and updates schema, types, docs, and focused regression tests.

## Reproduction Assessment

Yes for the current behavior: source inspection shows Teams channel replies are routed through a helper that appends a thread id when present, with no config branch to keep the canonical channel key. This proves the missing opt-out, not a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the policy contract. The requested direction is reasonable if it stays Teams-owned and default-safe, but the config name, scope, and precedence need product review before implementation.
