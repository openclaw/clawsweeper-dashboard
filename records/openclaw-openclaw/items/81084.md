## Summary

Keep open: current main still applies Microsoft Teams channel-thread session isolation without a supported channel-memory opt-out, and the requested change is a new Teams config/product policy rather than cleanup that can be closed.

## What This Changes

The issue requests a Microsoft Teams channel session-routing policy that lets channel-bound agents keep canonical channel memory instead of always using per-thread sessions.

## Best Possible Solution

Add a Teams-owned session routing policy that defaults to thread isolation, supports explicit channel-memory opt-in at the right global/team/channel or binding level, and updates schema, docs, and regression tests together.

## Reproduction Assessment

Yes for the current behavior: source inspection shows Teams channel replies derive thread-qualified session keys from conversationMessageId or replyToId. This is not a failing bug repro for an existing contract; it proves the missing opt-out described by the feature request.

## Solution Assessment

Unclear until maintainers choose the product contract: the direction is sound if it stays Teams-owned and preserves thread isolation as the default, but the config name, scope, and precedence need review first.
