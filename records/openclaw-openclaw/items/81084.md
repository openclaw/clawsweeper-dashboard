## Summary

Keep open: current main still routes Microsoft Teams channel replies through thread-qualified session keys, while the config schema and docs do not expose a channel-memory opt-out. This is a valid Teams session-routing feature/config decision, not implemented or superseded by the nested-key fix or the Microsoft tracker.

## What This Changes

Request a Teams channel/session routing policy that lets channel-bound agents keep the canonical channel session instead of always forking replies into per-thread sessions.

## Best Possible Solution

Add a Teams-owned routing policy that defaults to thread isolation, supports an explicit channel-memory mode for channel-bound agents, and updates schema, docs, and focused regression tests without weakening the existing isolation default.

## Reproduction Assessment

Yes for the current behavior: source inspection shows channel replies are routed through a helper that appends the Teams thread id when present, and no config branch exists to keep the canonical channel key. This reproduces the missing opt-out as a configuration gap, not a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the policy contract: the requested direction is reasonable if it stays Teams-owned and default-safe, but the config name, scope, and precedence need product review before implementation.
