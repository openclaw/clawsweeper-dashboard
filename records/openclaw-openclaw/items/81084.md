## Summary

Keep open: current main and v2026.6.10 still hard-code Microsoft Teams channel-thread replies onto thread-suffixed sessions and expose no channel-vs-thread session policy; related fixes solved the opposite cross-thread bleed and nested-key bugs, but this requested opt-in policy remains.

## What This Changes

The issue requests a supported Microsoft Teams channel session-routing policy that can keep channel-bound agents on the canonical channel session instead of forcing per-thread sessions.

## Best Possible Solution

Define a Teams-owned opt-in policy that preserves per-thread isolation by default while allowing explicit canonical channel-session routing for channel-bound agents that intentionally share memory.

## Reproduction Assessment

Yes at source level: current Teams channel replies pass the extracted `conversationMessageId` or `replyToId` into a helper that appends a `:thread:` suffix. I did not run a live Teams room reproduction in this read-only review.

## Solution Assessment

Yes to the opt-in policy direction, but not to the local hotfix as the final shape; maintainers need to choose the config scope, default, and compatibility expectations before implementation.
