## Summary

Keep open: current main and the latest release still thread-qualify Microsoft Teams channel replies, and the Teams config/docs expose outbound `replyStyle` but no inbound channel-vs-thread session policy. The nested-key follow-up is fixed separately, but the requested opt-out remains a maintainer product/config decision.

## What This Changes

The issue requests an opt-in Microsoft Teams channel session-routing policy so channel-bound agents can keep canonical channel memory instead of always splitting replies into per-thread sessions.

## Best Possible Solution

Add a maintainer-approved Teams-owned session-routing policy that keeps per-thread isolation as the default and allows explicit channel-session routing only for channel-bound agents that intentionally share memory.

## Reproduction Assessment

Yes at source level: current Teams channel replies pass `conversationMessageId` or `replyToId` into `resolveMSTeamsRouteSessionKey`, which appends a `:thread:` suffix. The Teams schema exposes `replyStyle` but no inbound channel-vs-thread session policy.

## Solution Assessment

Unclear as an implementation path: the issue identifies a real policy gap, but maintainers need to choose the config location, default, docs, and compatibility shape before a patch.
