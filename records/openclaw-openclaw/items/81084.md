## Summary

Keep open: current main still routes Microsoft Teams channel replies into thread-qualified session keys, and there is no supported config/docs/schema option for a channel-memory opt-out. This is a feature/config policy request, not a cleanup-close candidate.

## What This Changes

Request a supported Microsoft Teams policy/config option that lets channel-bound agents keep the canonical channel session instead of always using per-thread session keys.

## Best Possible Solution

Add an msteams-owned session routing policy that defaults to thread isolation, supports the intended channel-memory opt-out at the right global/team/channel scope, and updates schema, docs, and regression tests together.

## Reproduction Assessment

Yes for the current behavior: source inspection shows Teams channel replies derive thread-qualified session keys from conversationMessageId/replyToId. This is source-reproducible as a missing opt-out, not a failing bug against an existing documented contract.

## Solution Assessment

Unclear until maintainers choose the product contract: the requested direction is sound if it stays Teams-owned and preserves thread isolation as the default. The safer path is to settle naming and precedence before a focused implementation PR.
