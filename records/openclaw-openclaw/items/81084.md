## Summary

Keep open: current main still thread-qualifies Microsoft Teams channel replies, fixes the nested thread-key bug separately, and does not expose a channel-vs-thread session policy.

## What This Changes

The issue requests a Microsoft Teams session-routing option so channel-bound agents can keep one canonical channel session instead of always splitting channel replies by Teams thread.

## Best Possible Solution

Add a maintainer-approved Teams-owned policy that preserves thread isolation by default while allowing explicit channel-session routing for channel-bound agents that intentionally share memory.

## Reproduction Assessment

Yes, at source level: Teams channel replies pass `conversationMessageId` or `replyToId` into `resolveMSTeamsRouteSessionKey`, which appends a thread suffix. I did not run a live Teams room reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation path: the issue identifies a real policy gap, but maintainers need to choose the config location, default, and compatibility shape before a patch.
