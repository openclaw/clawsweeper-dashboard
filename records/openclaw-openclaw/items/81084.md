## Summary

Keep open: current main still routes Microsoft Teams channel replies into thread-qualified sessions and there is no supported channel-memory opt-out, so this remains a real feature/config policy request rather than cleanup-close material.

## What This Changes

The issue requests an opt-in Microsoft Teams session-routing policy so channel-bound agents can keep canonical channel memory instead of always splitting replies into per-thread sessions.

## Best Possible Solution

Add an msteams-owned session routing policy with thread isolation as the default, global/team/channel precedence, aligned schema/types/docs, and regression coverage that preserves the existing cross-thread-isolation behavior unless explicitly opted out.

## Reproduction Assessment

Yes for the current behavior: source inspection shows Teams channel replies derive thread-qualified session keys from conversationMessageId/replyToId, and the v2026.5.7 release helper had the same unconditional path. This is not a bug repro for an existing contract; it proves the missing opt-out described by the feature request.

## Solution Assessment

Unclear until maintainers choose the product contract. The direction is sound if it stays Teams-owned and keeps thread isolation as the default, but the config name, scope, and precedence need review before implementation.
