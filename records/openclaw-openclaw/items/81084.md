## Summary

Keep open: current main still routes Microsoft Teams channel replies into thread-qualified sessions without a supported channel-memory opt-out. The related cross-thread isolation and malformed-key fixes are useful context, but they do not add the requested policy/config choice.

## What This Changes

Request a supported Microsoft Teams configuration or policy to keep channel-bound agents on the canonical channel session instead of splitting channel replies into per-thread sessions.

## Best Possible Solution

Add an msteams-owned opt-in session routing policy that defaults to existing per-thread isolation, supports scoped overrides, and updates types, strict schema, docs, and regression tests without weakening the cross-thread isolation default.

## Reproduction Assessment

Yes for the current behavior: source inspection shows Teams channel replies derive thread-qualified session keys from conversationMessageId/replyToId, and config/schema/docs expose no opt-out. This is not a failing bug repro against an existing contract; it demonstrates the missing feature requested here.

## Solution Assessment

Unclear until maintainers choose the product contract. The direction is plausible if it stays Teams-owned and keeps thread isolation as the default, but config scope and precedence need review before implementation.
