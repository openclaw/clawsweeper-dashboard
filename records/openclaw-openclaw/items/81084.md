## Summary

Keep open: current main still routes Microsoft Teams channel-thread replies into thread-qualified session keys without a supported channel-memory opt-out, and the remaining request is a new policy/config surface rather than cleanup-close material.

## What This Changes

Request a supported Microsoft Teams session-routing policy so channel-bound agents can keep canonical channel memory instead of always forking thread replies into per-thread sessions.

## Best Possible Solution

Add an msteams-owned session-routing policy that defaults to thread isolation, supports global/team/channel precedence for channel memory where explicitly configured, and updates types, schema, docs, and regression tests together.

## Reproduction Assessment

Yes for the current behavior: source inspection shows channel replies derive thread-qualified session keys from `conversationMessageId` or `replyToId`. This is not a failing existing-contract bug repro; it proves the missing opt-out described by the feature request.

## Solution Assessment

Unclear until maintainers choose the product contract: the requested direction is reasonable if it stays Teams-owned and preserves thread isolation by default, but the public config name and precedence need review before implementation.
