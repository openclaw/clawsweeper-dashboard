## Summary

Keep open: current main still routes Microsoft Teams channel replies into thread-qualified sessions without a supported channel-memory opt-out, and the requested fix is a new config/product policy rather than a cleanup close.

## What This Changes

Request a Microsoft Teams channel/session policy that lets channel-bound agents keep using the canonical channel session instead of being forced into per-thread session keys.

## Best Possible Solution

Add an msteams-owned session routing policy with a thread-isolation default and global/team/channel precedence, then update schema/types/docs and regression tests without weakening the existing cross-thread isolation default.

## Reproduction Assessment

Yes for the current behavior: source inspection shows Teams channel replies derive thread-qualified session keys from conversationMessageId/replyToId, and the 2026.5.7 tag has the same unconditional helper path. This is not a bug repro for an existing contract; it proves the missing opt-out described by the feature request.

## Solution Assessment

Unclear until maintainers choose the product contract: the direction is sound if it stays Teams-owned and defaults to thread isolation, but the config name, scope, and precedence need review before implementation.
