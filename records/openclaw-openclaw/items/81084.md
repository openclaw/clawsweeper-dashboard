- `impact:session-state`: The issue is about Teams channel replies fragmenting one intended channel memory into many thread-specific session states.

## Summary

Keep open: current main still routes Microsoft Teams channel replies into thread-qualified session keys without a supported channel-memory opt-out, and the remaining work is a new Teams session-routing policy/config decision rather than an implemented cleanup item.

## What This Changes

Request a supported Microsoft Teams session-routing policy that lets channel-bound agents keep the canonical channel session instead of always splitting replies into per-thread sessions.

## Best Possible Solution

Add a Teams-owned opt-in session-memory policy with default thread isolation, clear global/team/channel or agent-binding precedence, docs/schema updates, and focused tests for both thread and channel modes.

## Reproduction Assessment

Yes for the current source behavior: the Teams handler unconditionally sends channel replies through a helper that appends a thread id when present, and the config/schema/docs expose no channel-memory opt-out. This reproduces the configuration gap rather than a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the policy contract. The requested opt-in direction is reasonable if it stays Teams-owned and default-safe, but the exact scope and precedence need product review before implementation.
