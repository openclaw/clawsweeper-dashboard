- `impact:session-state`: The issue is specifically about Teams channel memory being fragmented across multiple OpenClaw session keys.

## Summary

Keep open: current main and the latest shipped release still route Microsoft Teams channel replies into thread-qualified sessions with no supported channel-memory opt-out, and the requested behavior is a new Teams session-routing policy/config decision rather than an implemented fix.

## What This Changes

Request a Microsoft Teams session-routing option so channel-bound agents can keep canonical channel session memory instead of being split into per-thread sessions.

## Best Possible Solution

Add a Teams-owned session-routing policy that defaults to thread isolation, supports an explicit channel-memory mode at the chosen scope, and updates schema, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level: the Teams handler always rewrites channel replies through a helper that appends a thread-qualified session key when a thread id is present, and the schema has no opt-out. I did not run a live Teams tenant reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the policy contract: the requested opt-in is reasonable, but the scope and precedence should be decided before implementation so thread isolation remains the safe default.
