## Summary

Keep open: current `main` still has Microsoft Teams channel replies using thread-qualified session keys with no supported channel-memory opt-out, and the request is a new Teams session-routing policy/config decision rather than an already-fixed cleanup item.

## What This Changes

The issue requests a Microsoft Teams configuration or policy option that lets channel-bound agents keep canonical channel session memory instead of being split into per-thread sessions.

## Best Possible Solution

Add a Teams-owned session routing policy that defaults to thread isolation, supports an explicit channel-memory mode for channel-bound agents, and updates the schema, docs, and focused regression tests without weakening existing thread isolation.

## Reproduction Assessment

Yes for the current behavior: source inspection shows channel replies are routed through a helper that appends a Teams thread id when present, and no config branch exists to keep the canonical channel key. This reproduces the missing opt-out, not a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the policy contract: the requested direction is reasonable if it stays Teams-owned and default-safe, but the config name, scope, and precedence need product review before implementation.
