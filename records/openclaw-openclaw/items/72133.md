## Best Possible Solution

Reuse the existing normalized usage and `responseUsage` state: render compact native iOS/Android per-message footers, and keep channel footers on the existing `/usage` or footer-template path with no parallel configuration unless maintainers choose one.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main verification path: channel `/usage` footer support exists, but iOS shared chat does not render `message.usage` and Android chat does not parse usage fields.

## Solution Assessment
