## Summary

Keep open. Current main still has the rich Control UI metadata, but the requested mobile and channel parity is not complete: channel footers are partial, Swift chat does not render decoded usage, and Android chat drops usage metadata before rendering.

## What This Changes

The issue requests per-message token, cache, cost, model, and context-window metadata in native mobile chat and optional messaging-channel reply footers.

## Best Possible Solution

Add a compact shared usage metadata presenter and feed it from existing usage, model, and context fields across channel, Swift, and Android render paths.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence verification path: Control UI has the requested metadata while channel and native mobile paths are missing parts of it.

## Solution Assessment

Yes. Reusing existing `responseUsage`, session model/context metadata, and decoded mobile message usage is the narrow maintainable direction; adding per-surface bespoke calculations would be harder to keep consistent.
