## Summary

Keep open: current main covers the channel footer/default path through `messages.responseUsage`, but native iOS and Android chat bubbles still do not render per-message token/cost/context metadata, and the remaining mobile UX/settings behavior needs maintainer direction.

## What This Changes

Request to expose Control UI-style per-message token, cache, cost, context, and model metadata in native mobile chat and optional channel reply footers.

## Best Possible Solution

Keep `messages.responseUsage` as the channel contract and design a compact native iOS/Android assistant-message metadata component fed by existing usage/session data.

## Reproduction Assessment

Yes, source-level. iOS decodes message usage but the bubble body does not render it, while Android chat messages do not carry per-message usage into the bubble renderer; channel footers are already covered by `messages.responseUsage`.

## Solution Assessment

Unclear until maintainers choose the native mobile UX and setting behavior. The channel side should reuse the current `messages.responseUsage` path, while the remaining mobile work should render compact metadata from existing usage/session data.
