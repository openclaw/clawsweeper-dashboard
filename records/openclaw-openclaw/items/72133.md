## Summary

Keep open. Current main still has the complete per-message token/cache/cost/model/context display in the Control UI, but the channel response footer remains token/cache/cost-only, iOS decodes usage without rendering it in chat bubbles, and Android chat history still discards usage metadata entirely.

## What This Changes

Request to surface the Control UI’s per-assistant-message token, cache, cost, model, and context-window metadata in iOS/Android native chat and optional channel reply footers.

## Best Possible Solution

Add one compact usage presentation helper for assistant replies, extend the existing responseUsage channel footer with model/context percentage when reliable metadata is available, render the same compact line in Swift and Android chat bubbles, and preserve Android usage metadata during history parsing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but the gap is high-confidence by source inspection: Control UI renders the metadata, channel footers omit model/context, iOS does not render decoded usage, and Android drops usage during parsing.

## Solution Assessment

Is this the best way to solve the issue? Yes: reusing the existing /usage responseUsage setting and native chat renderers is the narrowest maintainable path, with context percentage gated on reliable model/context-window data.

Implement the feature request at https://github.com/openclaw/openclaw/issues/72133. Current main already renders compact per-assistant-message token/cache/cost/model/context metadata in the Control UI, but channel response footers and native mobile chat do not.
