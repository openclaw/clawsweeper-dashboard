## Label Justifications

- `P2`: This is a normal-priority feature with meaningful production impact for multi-tenant remote MCP users, but it is not a current regression or outage.
- `impact:security`: The requested surface handles authorization headers, tenant identity, session metadata, and redaction across remote trust boundaries.
- `impact:auth-provider`: The issue is about attaching per-request auth and routing metadata to remote MCP calls instead of relying on one static token.

## Summary

Keep open. Current main and v2026.5.18 still expose remote MCP headers as static per-server config, while the requested per-request resolver would add a security-sensitive public MCP runtime/config surface. The related open caller-context work only covers a narrower set of OpenClaw identity headers and does not replace generic dynamic auth or tenant header support.

## What This Changes

Support dynamic per-request header resolution for remote MCP servers so runtime auth, tenant, session, and routing metadata can be merged with static headers before each outbound request.

## Best Possible Solution

Design a typed request-scoped MCP header resolver or equivalent opt-in API for SSE and streamable-http with explicit context limits, merge precedence, redaction, redirect/OAuth behavior, docs, and tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, release-tag inspection, and dependency source give high confidence that current OpenClaw remote MCP headers are static.

## Solution Assessment

Unclear pending maintainer design. A hook, headersProvider, or templated-header mechanism is plausible, but the best solution needs explicit API shape, allowed context, precedence, redaction, redirects, and OAuth interaction before implementation.
