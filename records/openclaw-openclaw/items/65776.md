## Summary

Keep this open. Current main still exposes only static configured remote MCP headers: config accepts a scalar header map, launch resolution converts it once, and the SSE/streamable-http transports receive that fixed map with no request-scoped resolver or session context. The related static-header forwarding work and open OAuth/authProvider PR do not cover arbitrary per-request tenant, user, refreshed-token, or correlation headers.

## What This Changes

Request support for dynamic per-request headers on remote MCP SSE and streamable-http requests, merged with static configured headers at runtime.

## Best Possible Solution

Add a typed request-scoped header resolver or public runtime/plugin API for remote MCP transports. It should cover both streamable-http and SSE, define context exposure, merge precedence with static headers, secret redaction, and OAuth/authProvider interaction, then update tests and MCP docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives a high-confidence verification path: config, schema, transport construction, docs, and runtime callTool paths expose no per-request dynamic header entry point.

## Solution Assessment

Unclear pending maintainer design. A hook or headersProvider direction is plausible, but the maintainable solution needs explicit API, context, precedence, redaction, and OAuth coordination decisions before implementation.
