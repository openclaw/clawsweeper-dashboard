## Summary

Keep this open. Current main still supports only static remote MCP headers, while the requested per-request authorization/routing header resolver would add a new public runtime/config surface and needs maintainer API plus security design, especially alongside the open OAuth/authProvider work.

## What This Changes

Support dynamic per-request headers for remote MCP servers so outbound MCP HTTP requests can use runtime-specific authorization, routing, tenant, and metadata headers.

## Best Possible Solution

Add a typed request-scoped header resolver or plugin/runtime API for remote MCP transports, covering both SSE and streamable-http with explicit context exposure, merge precedence, redaction, OAuth interaction, and focused tests/docs.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: config, schema, transport construction, docs, tests, and runtime callTool paths expose no per-request arbitrary header entry point on current main.

## Solution Assessment

Unclear pending maintainer design. A hook or headersProvider is plausible, but the maintainable solution needs explicit API, context, precedence, redaction, and OAuth coordination decisions before implementation.
