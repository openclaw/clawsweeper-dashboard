## Summary

Keep this open. Current main still only supports static remote MCP headers: config/schema accept a scalar map, launch resolution converts it once, and SSE/streamable-http transports receive fixed requestInit headers with no request-scoped resolver, session/user context, or public runtime/plugin hook. The related static-header bug is already closed as implemented, while the open OAuth/authProvider PR is adjacent but does not cover arbitrary per-request tenant, user, refreshed-token, or correlation headers.

## What This Changes

Request support for dynamic per-request headers on remote MCP SSE and streamable-http requests, merged with static configured headers at runtime.

## Best Possible Solution

Add a typed request-scoped header resolver or public runtime/plugin API for remote MCP transports, covering both SSE and streamable-http with explicit context exposure, merge precedence, redaction, OAuth interaction, tests, and MCP docs.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: config, schema, transport construction, docs, and runtime callTool paths expose no per-request dynamic header entry point.

## Solution Assessment

Unclear pending maintainer design. A hook or headersProvider direction is plausible, but the maintainable solution needs explicit API, context, precedence, redaction, and OAuth coordination decisions before implementation.
