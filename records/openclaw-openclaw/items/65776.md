## Summary

Keep open. Current main and the latest release still model remote MCP headers as static config values, with only backend-specific environment placeholder projection in some CLI paths; a real request-scoped header resolver would need new public API and security design.

## What This Changes

The issue requests dynamic per-request headers for remote HTTP MCP servers so auth, tenant routing, and request metadata can be generated from runtime or session context before each outbound MCP request.

## Best Possible Solution

Design and add a typed request-scoped MCP header resolver or equivalent runtime API for both SSE and streamable-http, with explicit context limits, merge precedence, redaction, OAuth coordination, and docs/tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source, docs, release-tag inspection, tests, and dependency source give a high-confidence verification path that current OpenClaw MCP transport headers are static.

## Solution Assessment

Unclear pending maintainer design. A hook, headersProvider, or templated-header mechanism is plausible, but the best solution needs explicit API shape, allowed context, precedence, redaction, and OAuth interaction before implementation.
