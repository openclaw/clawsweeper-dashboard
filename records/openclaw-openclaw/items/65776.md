## Summary

Keep open: current main and v2026.6.8 still only expose static MCP headers plus the shipped OAuth path, while the requested request-scoped header resolver remains a new security-sensitive feature/API decision.

## What This Changes

Request support for dynamic per-request headers on remote MCP servers so outbound MCP calls can include runtime-specific authorization, routing, tenant, session, or correlation metadata.

## Best Possible Solution

Define an owner-approved MCP auth/header extension that reuses OAuth or narrower caller-context paths where they fit, and only adds a typed opt-in request-context header contract with explicit URL scoping, precedence, redaction, docs, and tests for the remaining use cases.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. The current-main absence is high-confidence from the MCP type, schema, transport, loopback env, docs, and latest-release checks.

## Solution Assessment

Unclear until maintainers choose the API and security boundary. A request-time resolver is technically plausible because the SDK supports custom fetch hooks, but the maintainable OpenClaw shape should be narrowed against OAuth, client_credentials, dynamic env, and caller-context alternatives first.
