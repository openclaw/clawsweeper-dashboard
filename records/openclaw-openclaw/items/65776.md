## Summary

Keep open. Current main supports static remote MCP header maps and SDK-managed OAuth, but it still lacks a request-scoped header resolver, templated header surface, or runtime/session header API; the related open issues cover narrower caller-context and client-credentials slices rather than superseding this broader product/security decision.

## What This Changes

Request support for generating and merging runtime-specific HTTP headers before each outbound remote MCP request.

## Best Possible Solution

Define an owner-approved, opt-in MCP request-header contract that reuses OAuth/client_credentials and caller-context paths where possible, with explicit origin scoping, redaction, precedence, docs, and tests.

## Reproduction Assessment

Not applicable as a bug reproduction. The limitation is source-verifiable on current main: remote MCP config and transport paths expose static headers plus OAuth, but no request-scoped resolver or templating surface.

## Solution Assessment

Unclear until maintainers choose the API and security shape. A dynamic hook/provider/template is plausible, but the safer path is to narrow it against existing OAuth, client_credentials, and caller-context work before adding a broad header surface.
