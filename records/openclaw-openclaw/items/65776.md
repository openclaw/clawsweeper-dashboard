## Summary

Keep open. Current main and v2026.6.6 now cover more remote MCP auth through OAuth, but remote MCP headers are still either static configured maps or SDK-managed OAuth; there is no OpenClaw request-scoped header resolver, templating surface, or caller/session header API, and related open items only cover narrower OAuth and caller-context slices.

## What This Changes

Support a request-scoped way for remote MCP transports to generate and merge authorization, tenant, session, and correlation headers for each outbound request.

## Best Possible Solution

Define an owner-approved MCP auth/header extension that reuses existing OAuth where possible and adds a typed, opt-in request-context header contract only if the remaining tenant/session use cases cannot be expressed by narrower OAuth or caller-context paths.

## Reproduction Assessment

Not applicable as a bug reproduction. The current limitation is source-verifiable: config and schema expose static headers plus auth: "oauth", but no request-scoped header resolver or templating surface.

## Solution Assessment

Unclear until maintainers choose the API and security shape. A hook, provider, or templated-header mechanism is plausible, but it should be narrowed against the existing OAuth and caller-context tracks first.
