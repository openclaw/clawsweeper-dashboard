## Summary

Keep open: current WebChat markdown rendering still strips custom URI schemes, and the requested behavior needs a security/product decision about which protocols should become clickable.

## What This Changes

The issue asks WebChat to render explicit Markdown links with custom URI schemes such as `obsidian://` and `things://` as clickable anchors.

## Best Possible Solution

Add a security-reviewed WebChat link-scheme policy that keeps dangerous schemes inert, renders approved custom protocols from explicit Markdown links, and includes renderer tests plus docs if a config knob is added.

## Reproduction Assessment

Yes, from source inspection: explicit Markdown links reach the shared WebChat markdown renderer, but the DOMPurify hook removes `href` for custom schemes. I did not run a live browser repro in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the scheme policy. The maintainable path is a narrow allowlist or operator-controlled setting, not enabling arbitrary protocols.
