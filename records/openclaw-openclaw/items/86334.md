Keep open: current main and v2026.6.8 still keep custom WebChat app-protocol hrefs inert, and the open linked implementation PR remains the active path but still needs maintainer security/product approval for the browser-to-OS scheme policy.

## What This Changes

The issue asks WebChat to preserve explicit markdown links using safe custom app URI schemes such as obsidian:// and things:// instead of rendering them without clickable hrefs.

## Best Possible Solution

Review and land a maintainer-approved narrow WebChat scheme policy in the linked PR, or explicitly decide that custom app-protocol links should remain inert.

## Reproduction Assessment

Yes. Source inspection shows WebChat messages flow through toSanitizedMarkdownHtml, and current main removes hrefs for parsed protocols outside http:, https:, and mailto:; I did not run a live browser repro in this read-only pass.

## Solution Assessment

Unclear. A narrow allowlist in the shared sanitizer is a plausible fix, but the exact built-in scheme set and browser-to-OS security boundary need maintainer approval before this is the best solution.
