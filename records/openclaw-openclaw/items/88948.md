## Summary

Keep open: the iMessage typing keepalive idea is useful, but the current PR bypasses the central typing policy and can show native typing during tool work even when typing has been explicitly disabled.

## What This Changes

The PR adds iMessage typing keepalive settings, forwards direct-message tool-start progress into the typing controller, and adds an iMessage monitor regression test for tool activity before reply text.

## Best Possible Solution

Keep the iMessage tool-work keepalive, but route the tool-start trigger through the existing policy-aware typing path so disabled and message-mode typing contracts remain intact.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the PR defect: configure an iMessage DM agent or session with `typingMode: "never"`, then a tool-start event still reaches the PR's direct `startTypingLoop()` callback.

## Solution Assessment

No, the current implementation is only a plausible fix because it bypasses the central typing policy. The better fix is to keep the iMessage keepalive but trigger it through policy-aware typing semantics with a disabled-mode regression test.
