## What This Changes

The PR keeps direct iMessage native typing indicators active during long tool-running gaps and adds monitor regression coverage for tool-start typing and cleanup.

## Best Possible Solution

Have the iMessage owner explicitly accept the native typing default for direct tool work before merge; if that is not the desired compatibility path, preserve persistent progress by default and make native tool typing opt-in.

## Reproduction Assessment

Yes, from source inspection: current main has no direct iMessage onToolStart typing bridge, while dispatch only forwards those callbacks under the suppress/allow progress flags that the PR adds. I did not run a local iMessage reproduction in this read-only review.

Yes, with one maintainer decision: reusing the existing iMessage typing RPC, typing keepalive, and reply dispatcher callbacks is the narrow plugin-boundary fix. The open question is whether the native-typing default should replace persistent tool-progress messages for direct iMessage turns.
