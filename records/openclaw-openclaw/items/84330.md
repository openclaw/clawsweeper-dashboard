## Summary

Keep open: current main and the latest release document the iMessage `cliPath` SSH-wrapper path, but they still do not explain the incremental stdio forwarding requirement that prevents small JSON-RPC requests from timing out.

## What This Changes

Document that custom iMessage `cliPath` wrappers must forward JSON-RPC stdin/stdout promptly instead of buffering until EOF or a large read fills.

## Best Possible Solution

Add a concise stdio contract and timeout troubleshooting note to the iMessage remote-wrapper docs, with the config reference kept aligned; no core runtime or config change is needed.

## Reproduction Assessment

Not applicable as a docs request: source inspection confirms `cliPath` wrappers sit on a long-lived, newline-delimited JSON-RPC stdio path, and the current docs omit the streaming requirement.

## Solution Assessment

Yes. The narrow maintainable fix is a docs-only update near the existing iMessage remote-wrapper example, not a new config knob or runtime fallback.
