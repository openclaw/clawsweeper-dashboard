## Summary

Keep open: current main and the latest release still document iMessage SSH/custom `cliPath` wrappers without the incremental stdio forwarding requirement, while the runtime sends newline-delimited JSON-RPC over a long-lived child process where buffering can produce the reported timeout.

## What This Changes

The issue asks for iMessage docs to state that custom `channels.imessage.cliPath` wrappers must promptly forward JSON-RPC stdin/stdout and keep stderr separate.

## Best Possible Solution

Update the public iMessage channel docs and config reference with a concise stdio contract for custom wrappers plus a short `imsg rpc timeout` troubleshooting note; no runtime or config change is needed.

## Reproduction Assessment

Not applicable as a docs request: source inspection confirms the runtime contract is newline-delimited JSON-RPC over long-lived stdio, and current docs omit the wrapper forwarding requirement.

## Solution Assessment

Yes. The narrow maintainable solution is a docs-only addition near the existing remote-wrapper examples, not a new `cliPath` mode, config knob, or runtime fallback.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/84330 with a docs-only update. Add a concise note near the iMessage Remote Mac over SSH / `channels.imessage.cliPath` wrapper examples explaining that wrappers must behave as transparent long-lived stdio pipes for JSON-RPC: forward stdin and stdout chunks or lines promptly, preserve newlines, avoid waiting for EOF or large fixed-size buffers, and keep stderr separate from JSON-RPC stdout. Add a short troubleshooting note for `imsg rpc timeout (chats.list)` caused by buffered wrappers. Keep the wording generic and avoid adding a new config option, runtime fallback, `rpcEndpoint`, or product behavior. Update both `docs/channels/imessage.md` and `docs/gateway/config-channels.md` if the config reference would otherwise continue to omit the same contract. No changelog entry is needed for this docs-only repair.
