## Label Justifications

- `P2`: This was a normal-priority cron/task-state bug with limited blast radius around long startup catch-up jobs.
- `impact:session-state`: The affected task ledger status could drift to a false lost terminal state while the cron run was still active.

## Summary

Close as implemented on current main: the startup catch-up path now marks the cron job active before execution and clears the marker through the shared outcome-finalization path, so authoritative task maintenance no longer sees a still-running catch-up cron task as unbacked solely because the marker was never set.

## What This Changes

The issue reports that startup catch-up cron command jobs did not set the active cron marker, allowing task maintenance to classify long-running catch-up runs as lost while they were still running.

## Best Possible Solution

Keep the current-main implementation from merged PR https://github.com/openclaw/openclaw/pull/89055 and handle the still-open duplicate fixing PRs as superseded separately.

## Reproduction Assessment

No current-main failing reproduction remains from source inspection: startup catch-up now sets `isCronJobActive` state before execution and clears it after outcome finalization. The original source-level repro was valid on the earlier audited main because that mark was absent.

## Solution Assessment

Yes. The current implementation fixes the producer-side owner boundary by marking the startup catch-up run active like sibling due/manual paths, without changing task-maintenance policy or adding config.
