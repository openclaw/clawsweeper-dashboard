## Summary

Keep open: current main and the latest release still leave startup catch-up cron runs unmarked in `activeJobIds`, while authoritative task maintenance relies on that marker before marking cron tasks `lost`. The report describes a real, narrow core stability bug rather than a duplicate of the already-fixed manual-run path.

## What This Changes

Startup catch-up cron command jobs should set the in-process active marker while running so task reconciliation and heartbeat busy checks treat them as live.

## Best Possible Solution

Add the startup catch-up active mark on the same producer-side contract as due and manual runs, keep the existing catch-up policy unchanged, and cover the mid-flight invariant with a focused cron regression test.

## Reproduction Assessment

Yes from source inspection, but not by running the test in this read-only review. The current startup catch-up path leaves `isCronJobActive(jobId)` false while a cron task row exists and authoritative reconciliation depends on that marker after the five-minute grace.

## Solution Assessment

Yes, the suggested repair is the narrow owner-boundary fix: mark the job active in `runStartupCatchupCandidate` and rely on the existing `applyOutcomeToStoredJob` clear path. A regression test should prove the startup catch-up mid-flight marker rather than changing catch-up scheduling or task-maintenance policy.
