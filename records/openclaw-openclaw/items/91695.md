## Label Justifications

- `P2`: Long startup catch-up command runs can receive an incorrect terminal projection and system message, but the affected scheduler path has limited blast radius.
- `impact:session-state`: The task registry can persist and surface a live cron execution as a lost terminal task.

## Summary

Current main and v2026.6.5 still omit the active marker on startup catch-up runs, so the reported false `lost` projection remains valid. Keep the issue open because the clean open closing PR https://github.com/openclaw/openclaw/pull/91713 is the current implementation candidate and must be reviewed or landed before this issue closes.

## What This Changes

Ensure startup catch-up cron runs set the in-process active marker for their full execution so authoritative task reconciliation does not mark live jobs as lost.

## Best Possible Solution

Land the existing marker insertion with a focused regression test that starts a missed command job through `CronService.start()`, blocks execution, verifies the job is active in flight, then verifies the marker is cleared after completion.

## Reproduction Assessment

Yes, at source level with high confidence: current main starts a catch-up cron task without the exact active marker consulted by authoritative reconciliation, and the issue provides a concrete blocked-run test path. This read-only review did not execute the proposed test.

## Solution Assessment

Yes in direction: restoring the established mark/clear invariant is the narrowest maintainable fix and avoids introducing a second recovery mechanism. The best landing shape also includes the focused producer-side regression test rather than relying on inspection of the one-line change.
