## Summary

Keep this issue open. Current main still has a source-proven single-shot OAuth refresh path for ordinary transient provider failures, while the direct retry PRs linked from the discussion are closed and unmerged.

## What This Changes

Request to add bounded retry behavior for transient OAuth token refresh failures so agents do not fail immediately on temporary provider or network errors.

## Best Possible Solution

Add a bounded, tested retry policy around the actual OAuth refresh attempt, using src/infra/retry.ts with a provider-safe transient classifier and preserving the existing lock, adoption, fallback, and token-reuse recovery behavior.

## Reproduction Assessment

Yes, at source level: current main calls the provider refresh path once, and HTTP 500/network timeout are not handled by the refresh_token_reused recovery branch. I did not run a live provider outage or network-blip reproduction in this read-only review.

## Solution Assessment

No for the issue body's helper verbatim: retrying every thrown refresh error would also retry permanent OAuth failures. The maintainable solution is a bounded shared retry with a transient-error classifier, placed so existing locks, fallbacks, plugin hooks, and token-reuse recovery still work.
