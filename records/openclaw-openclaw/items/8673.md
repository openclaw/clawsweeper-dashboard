## Summary

Current main still makes a single refresh call for ordinary transient OAuth failures. The issue remains necessary, but the original catch-all retry proposal is too broad because the shared path covers built-in and plugin providers and must preserve timeout, locking, fallback, and terminal-error behavior.

## What This Changes

Add bounded retry handling for transient OAuth token refresh failures without retrying terminal authentication errors.

## Best Possible Solution

Define explicit transient-versus-terminal refresh failure semantics for built-in and plugin providers, then apply one bounded manager-owned retry budget that preserves locking, timeout, credential adoption, fallback, persistence, and error redaction.

## Reproduction Assessment

Yes at source level: current main performs one provider refresh call, and ordinary network or HTTP failures are not covered by the specialized refresh-token-reused recovery. This read-only review did not execute a controlled live provider failure.

## Solution Assessment

No, the originally proposed unconditional local withRetry helper is not the best current solution. The maintainable fix is a shared manager-owned bounded retry policy with explicit provider retryability semantics and tests preserving terminal-error and fallback behavior.
