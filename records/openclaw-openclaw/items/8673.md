## Summary

Keep open: current main and v2026.5.22 still make one ordinary OAuth refresh call for provider/network failures, and the only retry-like recovery remains scoped to refresh-token-reuse/adoption. The closed retry PRs were not merged, so the remaining work needs owner-reviewed retry semantics rather than cleanup closure.

## What This Changes

The issue requests bounded retry logic for transient OAuth token refresh failures so temporary provider/network errors do not immediately fail agents.

## Best Possible Solution

Add an owner-reviewed retryAsync integration around OAuth refresh with a transient-only classifier, preserving the existing lock, timeout, adoption, fallback, plugin refresh, and refresh-token-reuse behavior with focused regression tests.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh-token-reuse recovery branch. I did not run a live transient provider outage in this read-only review.

## Solution Assessment

No for the issue body's exact local withRetry helper: retrying every thrown refresh error would also retry permanent auth failures and duplicate src/infra/retry.ts. The safer path is an owner-reviewed retryAsync integration with preserved error causes, lock safety, and a transient-only classifier.
