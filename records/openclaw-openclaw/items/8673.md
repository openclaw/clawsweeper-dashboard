## Summary

Keep open: current main still has a single-shot shared OAuth refresh path, and the two linked retry PRs are closed unmerged.

## What This Changes

Request adds bounded retry/backoff for transient OAuth token refresh failures.

## Best Possible Solution

Add provider-safe bounded retry around the shared OAuth refresh call, reuse retryAsync, preserve lock/adoption/fallback behavior, and cover transient success, retry exhaustion, and non-retryable auth errors.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: an expired OAuth profile with getOAuthApiKey throwing a network timeout or HTTP 500 would call the refresh helper once and then surface OAuthManagerRefreshError. I did not run a live provider repro in this read-only review.

## Solution Assessment

Yes, the requested direction is narrowly maintainable if implemented as a bounded retry at the shared refresh boundary with an explicit transient-error classifier. An ad hoc helper that retries all OAuth errors would be weaker than reusing retryAsync and preserving permanent-auth failure semantics.
