## Summary

Keep this issue open. Current main still has a single-shot OAuth refresh path for ordinary transient failures: locking, timeout, main-store adoption, and refresh_token_reused recovery exist, but generic bounded retry is not wired into auth profiles, and both linked retry PRs are closed unmerged.

## What This Changes

Request to add bounded retry/backoff around OAuth token refresh so transient network or provider API failures do not immediately fail agents.

## Best Possible Solution

Implement a bounded retry policy around OAuth refresh using the existing retryAsync helper, with a conservative transient-error classifier, unchanged lock/adoption/reused-token behavior, and focused auth-profile regression tests.

## Reproduction Assessment

Yes, source-reproducible. An expired OAuth profile that hits a transient getOAuthApiKey/provider refresh throw will make one refresh attempt and then flow to OAuthManagerRefreshError, because current main has no generic retry in this path.

## Solution Assessment

No, the issue's simple retry-all helper is not the best maintainable fix. The safer direction is to reuse retryAsync with a provider-safe transient classifier and tests that keep permanent auth failures non-retryable.
