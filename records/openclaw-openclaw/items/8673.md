## Summary

Keep this issue open. Current main still performs OAuth refresh as a single provider/plugin call under lock and timeout, with only a special refresh-token-reused recovery path; ordinary transient network, timeout, or HTTP 500 refresh failures are not retried. The linked retry PRs are closed and unmerged, and the maintainer discussion already points to the same remaining gap.

## What This Changes

The issue requests bounded retry logic around transient OAuth token refresh failures so one network/API blip does not immediately fail agent authentication.

## Best Possible Solution

Implement a bounded, tested retry policy in the OAuth refresh flow, preferably reusing `src/infra/retry.ts` with a provider-safe transient-error classifier. Preserve the existing file lock, timeout, main-store adoption, external-bootstrap, fallback, and refresh_token_reused recovery behavior; add focused auth-profile tests for transient success after retry, retry exhaustion preserving the final error, and non-retryable OAuth failures such as `invalid_grant`.

## Reproduction Assessment

Yes. A focused unit test can mock `adapter.refreshCredential` or `getOAuthApiKey` to throw a network timeout or HTTP 500 once and then succeed; current main calls the refresh function once under `withRefreshCallTimeout`, so the first throw reaches the final refresh error path.

## Solution Assessment

No for the issue body's simple wrap-all sketch, because retrying every OAuth refresh error would be too broad. The maintainable solution is a bounded retry around the refresh attempt with an explicit transient classifier and tests that preserve the existing lock/adoption/fallback semantics.
