## Label Justifications

- `P2`: A transient OAuth refresh failure can interrupt active agent requests, but the impact is limited and recoverable rather than emergency-level.
- `impact:auth-provider`: The issue directly concerns provider OAuth refresh and credential resolution.

## Summary

This remains necessary on current main: the shared OAuth manager still makes one timed refresh call, and both direct retry PRs are closed unmerged. The issue should stay open for maintainer approval of a provider-safe retry classifier and latency budget rather than accepting the proposed retry-all helper.

## What This Changes

Add bounded retry for transient OAuth token-refresh failures without retrying terminal OAuth errors.

## Best Possible Solution

Define one manager-owned transient-refresh predicate and use the shared retry primitive with a bounded total-time budget, preserving refresh queues, external credential adoption, refresh_token_reused recovery, terminal-failure behavior, cause propagation, and credential persistence invariants.

## Reproduction Assessment

Yes. Current-main source provides a high-confidence reproduction path: the shared manager invokes the provider refresh once under a timeout and ordinary transient failures have no retry branch; this read-only review did not induce a live provider outage.

## Solution Assessment

No. The proposed retry-every-error helper is not the best solution because terminal OAuth failures must remain single-shot; the narrow maintainable approach is a bounded shared retry policy with explicit transient classification and focused regression coverage.

## Risks / Open Questions

- An unconditional retry loop can repeat terminal OAuth failures such as invalid_grant, interact badly with rotating or single-use refresh tokens, and increase upstream throttling.
- Retry delays must fit within the existing refresh timeout and lock behavior so one credential refresh does not stall queued agent requests unexpectedly.
- The issue carries explicit maintainer-review and product-decision labels, so the retryable-error contract should not be chosen autonomously.
