## Label Justifications

- `impact:auth-provider`: The issue directly affects OAuth credential refresh and provider authentication continuity.

## Summary

Current main still performs a single provider/plugin OAuth refresh attempt for ordinary failures. The existing second attempt is narrowly reserved for refresh-token-reused recovery after credential state changes, so transient HTTP/network failures remain unhandled; both direct retry PRs are closed unmerged. The issue should remain open, but the proposed retry-all helper is not safe enough because it would also repeat terminal errors such as invalid_grant.

## What This Changes

Add bounded retries for transient OAuth token-refresh failures without retrying terminal authentication errors.

## Best Possible Solution

Implement a bounded retryAsync policy at the canonical OAuth-manager refresh call, retry only clearly transient network/timeout/5xx/429 failures, honor Retry-After when available, and preserve locking, hard timeout, credential adoption, refresh-token-reused recovery, and the final original error.

## Reproduction Assessment

Yes at source level: current main makes one ordinary provider refresh attempt, and the tests confirm HTTP 500 and network timeout do not enter the only existing retry-like recovery branch. This read-only review did not induce a live upstream outage.

## Solution Assessment

No, not exactly as proposed: retrying every exception with a new local helper is too broad and duplicates src/infra/retry.ts. The best fix is a bounded shared retry with an explicit transient-error policy and focused manager coverage.
