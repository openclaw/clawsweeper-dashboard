## Summary

Keep open: current main and v2026.5.12 still perform ordinary OAuth refresh with one provider/API call, while the only retry-like path remains limited to refresh_token_reused recovery. The direct retry PRs are closed unmerged, and the later merged Codex forced-refresh fix does not add a generic transient OAuth refresh retry policy.

## What This Changes

The issue requests bounded retry logic for transient OAuth token refresh API or network failures so agents do not fail on a single temporary refresh error.

## Best Possible Solution

Land an owner/secops-approved bounded retryAsync-based OAuth refresh policy with an explicit transient-error classifier, preserving current lock/adoption/fallback/reused-token behavior and adding focused auth-profile regression tests.

## Reproduction Assessment

Yes, by source inspection rather than a live provider run. On current main, an expired built-in OAuth credential calls getOAuthApiKey once, and HTTP 500 or network-timeout errors do not enter the refresh_token_reused recovery path.

## Solution Assessment

No for the issue's broad retry-every-error sketch. The safer path is a bounded retryAsync policy in the owner-approved refresh path, gated by a transient-error classifier and covered by auth-profile tests.

## Risks / Open Questions

- Retrying every OAuth refresh error could mask permanent failures such as invalid_grant, revoked refresh tokens, account mismatch, or provider policy denial.
- The retry placement must preserve the existing refresh lock, hard timeout, main/external credential adoption, fallback behavior, provider plugin hooks, and refresh_token_reused recovery.
- Provider plugins may already implement their own refresh behavior, so a shared retry policy needs to avoid surprising double-retry semantics.
