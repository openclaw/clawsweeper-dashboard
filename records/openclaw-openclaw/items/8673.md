- `P2`: A transient OAuth refresh failure can interrupt provider-backed agent work, but the blast radius is limited to affected auth-provider calls and has restart/re-auth workarounds.
- `impact:auth-provider`: The issue is specifically about OAuth token refresh and provider authentication behavior during agent runs.

## Summary

Keep open. Current main still single-shots the shared OAuth refresh call, the existing retry primitive is not wired into auth profiles, and the direct retry PRs for this issue are closed unmerged; the remaining work needs an auth-owner decision on retry classification and lock budget.

## What This Changes

The issue asks for bounded retry/backoff around OAuth token refresh so transient provider/API failures do not immediately fail agent work.

## Best Possible Solution

Implement bounded retry in the centralized OAuth refresh path using retryAsync, a maintainer-approved transient-error classifier, and focused auth-profile tests that preserve lock, timeout, adoption, fallback, and refresh-token-reuse behavior.

## Reproduction Assessment

Yes, at source level: an expired or forced OAuth credential whose provider refresh throws a transient error flows through one adapter.refreshCredential/getOAuthApiKey attempt and then error handling. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

Yes for a centralized bounded retry policy, but not for the sample local withRetry helper as-is. The maintainable path is retryAsync plus a transient-error classifier at the OAuth manager/shared refresh boundary.

## Risks / Open Questions

- A retry policy needs a narrow transient-error classifier so permanent OAuth failures such as invalid_grant, revoked tokens, 401, and 403 are not retried as temporary failures.

- Provider/plugin refresh hooks may already implement retries, so shared retry must avoid surprising double-retry latency or provider-side rate amplification.
