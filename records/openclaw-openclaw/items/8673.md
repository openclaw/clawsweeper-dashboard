## Summary

Keep open: current main and the latest release still perform a single ordinary OAuth refresh attempt for provider/network failures, while the only retry-like path remains scoped to refresh-token-reuse recovery.

## What This Changes

The issue asks for bounded retry/backoff around transient OAuth token refresh failures so temporary provider or network errors do not immediately fail agents.

## Best Possible Solution

Add an owner-reviewed retryAsync integration around OAuth refresh with a transient-only classifier and focused regression coverage for success-after-retry, retry exhaustion, and non-retryable OAuth errors.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh-token-reuse recovery branch. I did not run a live transient provider outage in this read-only review.

## Solution Assessment

No for the issue body's exact local withRetry helper: retrying every thrown refresh error would also retry permanent auth failures and duplicate src/infra/retry.ts. The safer path is an owner-reviewed retryAsync integration with preserved lock safety, error semantics, and a transient-only classifier.
