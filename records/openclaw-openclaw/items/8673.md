## Summary

Keep open: current main still has no bounded generic retry for transient OAuth refresh failures, and the only retry-like path remains limited to refresh_token_reused recovery. The closed retry PRs are unmerged, and the remaining change is auth-sensitive enough to need owner/secops judgment before implementation.

## What This Changes

Request adds bounded retry logic for transient OAuth token-refresh API failures so agents do not fail after one network or API blip.

## Best Possible Solution

Land a bounded retryAsync-based OAuth refresh policy with an explicit transient-error classifier, preserving lock/adoption/fallback/reused-token behavior and adding focused auth-profile regression tests.

## Reproduction Assessment

Yes, by source inspection rather than live provider repro. On current main, an expired built-in OAuth credential calls getOAuthApiKey once, and HTTP 500 or network-timeout errors do not enter the existing refresh_token_reused recovery path.

## Solution Assessment

No for the issue's broad retry-every-error sketch; the safer direction is a bounded retryAsync policy gated by a transient-error classifier in the owner-approved refresh path.
