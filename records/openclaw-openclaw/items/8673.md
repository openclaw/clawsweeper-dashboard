## Label Justifications

- `P2`: Transient OAuth refresh failures can interrupt OAuth-backed agent work, but the impact is limited to affected provider calls and has restart or later-retry workarounds.
- `impact:auth-provider`: The issue directly concerns OAuth credential refresh and continued provider authentication.

## Summary

Keep this issue open: current main and the latest release still perform ordinary OAuth refresh as one provider/plugin call, while the only retry-like second attempt is limited to refresh-token-reuse recovery. The linked retry PRs are closed unmerged, and the remaining change needs auth-owner agreement on transient classification and retry budget before automation is safe.

## What This Changes

The issue requests bounded retry handling for transient OAuth token refresh API failures so agent runs do not fail on a single temporary provider or network error.

## Best Possible Solution

Implement bounded retry in the centralized OAuth manager with retryAsync, an auth-owner-approved transient-error classifier, and focused tests preserving lock, adoption, fallback, and refresh-token-reuse semantics.

## Reproduction Assessment

Yes by source inspection, but not by a live transient-provider failure in this read-only review. An expired or forced OAuth refresh invokes one provider refresh call, and current tests show HTTP 500/network timeout do not enter the only specialized refresh-token-reuse recovery branch.

## Solution Assessment

No, the issue's proposed unconditional local withRetry helper is not the best solution. The safer fix belongs at the centralized OAuth manager boundary using retryAsync plus an explicit transient-error classifier and maintainer-approved timing budget.
