## Summary

Keep this issue open: current main and v2026.6.6 still perform ordinary OAuth refresh as one provider or plugin call, while the only retry-like second attempt is limited to refresh-token-reuse recovery. The directly linked retry PRs are closed unmerged, and the remaining retry policy needs auth-owner judgment before an automated fix is safe.

## What This Changes

Request bounded retry logic for transient OAuth token refresh failures so temporary network or provider errors do not immediately fail agents.

## Best Possible Solution

Implement bounded retry in the centralized OAuth manager with retryAsync, an auth-owner-approved transient-error classifier, and focused coverage for transient success, exhaustion, terminal no-retry behavior, and preservation of lock/adoption/fallback semantics.

## Reproduction Assessment

No live transient-provider failure was reproduced in this read-only review. The issue is source-reproducible with high confidence because ordinary refresh errors receive one provider call and tests show HTTP 500/network timeout do not enter the only specialized recovery branch.

## Solution Assessment

No, the issue's proposed unconditional local withRetry helper is not the best solution. The safer fix belongs at the centralized OAuth manager boundary using retryAsync plus an explicit transient-error classifier and timing budget.

## Risks / Open Questions

- A broad retry-on-every-error policy could repeatedly submit terminal OAuth failures such as invalid_grant, revoked credentials, malformed responses, or provider-specific permanent errors.
- Retry count, per-attempt timeout, and backoff must fit the existing 120-second refresh-call timeout plus the cross-agent refresh lock stale/wait budget.
- Provider-plugin refreshOAuth hooks share this manager boundary, so the policy must avoid surprising double-retry behavior for providers that already retry internally.
