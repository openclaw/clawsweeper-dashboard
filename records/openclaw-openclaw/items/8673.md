## Summary

Keep this issue open. Current main still lacks generic bounded retry for ordinary transient OAuth refresh failures, the direct retry PRs are closed unmerged, and the retry classifier/placement needs owner review to avoid retrying permanent auth failures or double-retrying provider hooks.

## What This Changes

The issue requests bounded retry logic for transient OAuth token refresh failures so a single network/API blip does not fail agents.

## Best Possible Solution

Add a bounded shared retry around the actual OAuth refresh attempt using retryAsync with an explicit transient-only classifier, while preserving locks, timeouts, adoption/fallback behavior, provider hooks, and token-reuse recovery.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are not covered by the existing refresh_token_reused recovery branch. I did not run a live provider outage or network-blip reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact helper: retrying every thrown refresh error would also retry permanent OAuth failures such as invalid_grant. The safer direction is a bounded shared retry using retryAsync with an explicit transient-error classifier and owner-reviewed placement.
