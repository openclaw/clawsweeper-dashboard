## Summary

Keep open. Current main still refreshes OAuth credentials through a single provider/plugin call under lock and timeout, and the only retry-like branch is scoped to refresh-token-reused recovery rather than ordinary transient network, timeout, or HTTP 500 failures.

## What This Changes

The issue requests bounded retry logic for transient OAuth token refresh API failures so agents do not fail on a single temporary refresh error.

## Best Possible Solution

Implement a bounded retry around OAuth refresh using a provider-safe transient-error classifier, preferably reusing retryAsync, with focused regression coverage for transient success, exhausted retries, and non-retryable OAuth failures.

## Reproduction Assessment

Yes, at source level. A focused test can mock adapter.refreshCredential or getOAuthApiKey to throw a network timeout or HTTP 500 once and then succeed; current main performs only one refresh call before falling into the existing error path.

## Solution Assessment

No for the issue body's wrap-all retry sketch, because OAuth has permanent auth failures that should not be retried. The maintainable direction is a bounded retry with an explicit transient classifier, placed where it preserves the existing lock and fallback semantics.
