## Summary

Keep open. Current main still performs ordinary OAuth refresh as a single provider/plugin call under locking and timeout, while the only retry-like path remains limited to refresh-token-reused recovery; both linked retry PRs are closed and unmerged.

## What This Changes

Request bounded retry logic for transient OAuth token refresh failures so agents do not fail after a single network or API blip.

## Best Possible Solution

Add a bounded retry around OAuth refresh with an explicit transient-error classifier, preserving the existing lock, timeout, adoption, fallback, and refresh-token-reused semantics, plus focused regression tests.

## Reproduction Assessment

Yes, at source level. A focused test can mock adapter.refreshCredential or getOAuthApiKey to throw a network timeout or HTTP 500 once and then succeed; current main performs only one ordinary refresh attempt before fallback/error handling.

## Solution Assessment

No for the issue body's blanket retry sketch. The maintainable fix is a bounded retry using a provider-safe transient classifier, ideally reusing retryAsync, so permanent OAuth failures are not retried as transient network blips.
