## Label Justifications

- `P2`: A transient refresh failure can interrupt active OAuth-backed agent calls, but the blast radius is limited to affected credentials and provider requests.
- `impact:auth-provider`: The issue directly concerns OAuth token refresh and continuity of provider authentication.

## Summary

Current main still has a single-shot OAuth refresh call; locking, timeout, credential adoption, and refresh-token-reused recovery do not retry ordinary transient network or server failures. The two issue-specific retry PRs are closed and unmerged, so the central reliability gap remains, but the proposed retry-all wrapper needs an auth-owner decision on which provider errors are safely retryable.

## What This Changes

Add bounded retry behavior for transient OAuth token-refresh failures without retrying terminal authentication errors.

## Best Possible Solution

Add bounded retry at the canonical OAuth manager refresh boundary using `retryAsync`, with explicit transient classification for transport/timeouts, retryable 429 responses, and retryable 5xx responses while keeping terminal OAuth and identity errors single-shot; preserve the existing lock, adoption, fallback, and reused-token flows.

## Reproduction Assessment

Yes at source level with high confidence: an injected transient throw from the refresh adapter receives one attempt on current main before existing recovery and error handling. This read-only review did not reproduce a live upstream network fault.

## Solution Assessment

No, not exactly as proposed: a local helper that retries every exception is broader than the safe contract. The best fix is manager-owned bounded retry using the existing utility and an explicit terminal-versus-transient classifier.
