## Summary

Keep open: current main and the latest release still do ordinary OAuth refresh as a single provider/plugin call. The existing second-attempt behavior is specific to refresh-token-reuse recovery, so the transient network/API retry request remains unresolved and needs owner review for retry classification.

## What This Changes

The issue requests bounded retry logic for transient failures during OAuth token refresh so agents do not fail after one network or API blip.

## Best Possible Solution

Add an owner-reviewed `retryAsync` integration around OAuth refresh with a transient-only classifier, preserving current lock, timeout, adoption, fallback, plugin refresh, and refresh-token-reuse behavior with focused regression tests.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh-token-reuse recovery branch. I did not run a live transient provider outage in this read-only review.

## Solution Assessment

No for the issue body's exact local `withRetry` helper: retrying every thrown refresh error would retry permanent auth failures and duplicate `src/infra/retry.ts`. The safer solution is an owner-reviewed `retryAsync` integration with preserved error causes, lock safety, and a transient-only classifier.
