## Label Justifications

- `P2`: Transient OAuth refresh failures can interrupt OAuth-backed agent work, but the impact is limited to affected provider calls and has restart or later-retry workarounds.
- `impact:auth-provider`: The issue directly concerns OAuth credential refresh and continued provider authentication.

## Summary

Keep open: current main and v2026.6.10 still refresh OAuth credentials with one provider/plugin refresh call; the only second-attempt path is refresh_token_reused recovery, and the related retry PRs are closed unmerged or adjacent.

## What This Changes

Request bounded retry for transient OAuth token-refresh failures so agents do not fail immediately on temporary network or provider API errors.

## Best Possible Solution

Implement bounded retry in the centralized OAuth manager using retryAsync and an auth-owner-approved transient classifier, while preserving locks, adoption/fallback, plugin hooks, and refresh-token-reuse recovery.

## Reproduction Assessment

Yes, by source inspection rather than a live transient-provider failure. An expired or forced OAuth refresh invokes one provider refresh call, and current tests show HTTP 500 and network timeout do not enter the only specialized recovery branch.

## Solution Assessment

No, the proposed local withRetry wrapper around getOAuthApiKey is not the best final shape. The safer fix belongs at the centralized OAuth manager boundary with retryAsync, an explicit transient-error classifier, and auth-owner timing semantics.
