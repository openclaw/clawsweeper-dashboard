## Summary

Keep this issue open. Current main still performs the shared OAuth refresh as a single attempt, and the only retry-like OAuth recovery path is still the refresh-token-reused branch; the direct retry PRs linked from the discussion are closed and unmerged.

## What This Changes

The issue requests bounded retry behavior for transient OAuth token refresh API failures so agents do not fail after one temporary network or provider error.

## Best Possible Solution

Add a bounded, tested retry policy around the actual OAuth refresh attempt, preferably reusing src/infra/retry.ts with a provider-safe transient classifier and regression coverage for success-after-transient-failure, retry exhaustion, and non-retryable OAuth failures.

## Reproduction Assessment

Yes, at source level: current main calls the provider refresh path once, and HTTP 500/network timeout are not handled by the refresh_token_reused recovery branch. I did not run a live provider outage or network-blip reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact helper as written: retrying every thrown refresh error would risk retrying permanent OAuth failures. The maintainable direction is a bounded shared retry with a transient-error classifier, placed so existing locks, fallbacks, and token-reuse recovery still work.

## Risks / Open Questions

- A retry fix needs a narrow transient-error classifier so invalid_grant, revoked refresh tokens, and other permanent OAuth failures are not retried as temporary failures.
- Retry placement must preserve the global refresh lock, hard refresh timeout, external/main-store credential adoption, provider-plugin hooks, and refresh_token_reused recovery.
- Provider plugins can own custom OAuth refresh behavior, so shared retry policy should avoid surprising double-retry semantics where a provider already retries internally.
