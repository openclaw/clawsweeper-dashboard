## Summary

Keep this issue open: current main and v2026.6.10 still perform ordinary OAuth refresh as one provider/plugin refresh call, and the only retry-like second attempt is limited to refresh-token-reuse recovery rather than HTTP 5xx, timeout, or network blip failures.

## What This Changes

The issue requests bounded retry behavior around OAuth token refresh so transient provider/API failures do not immediately fail agent authentication.

## Best Possible Solution

Implement bounded retry in the centralized OAuth manager using retryAsync, an auth-owner-approved transient-error classifier, and focused tests that preserve locking, adoption, fallback, plugin hook, and refresh-token-reuse semantics.

## Reproduction Assessment

Yes, by source inspection rather than a live transient-provider failure. An expired or forced OAuth refresh invokes one provider refresh call, and current tests show HTTP 500/network timeout do not enter the only specialized refresh-token-reuse recovery branch.

## Solution Assessment

No, the issue's proposed unconditional local withRetry helper is not the best solution. The safer fix belongs at the centralized OAuth manager boundary using retryAsync plus an explicit transient-error classifier and maintainer-approved timing budget.
