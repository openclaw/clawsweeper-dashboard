## Summary

Keep open: current main still lacks generic bounded retry for ordinary transient OAuth refresh failures, and source plus dependency inspection still show a single-shot shared refresh path. The two linked retry PRs are both closed and unmerged, so the requested reliability work remains unresolved.

## What This Changes

The issue requests bounded retry logic around OAuth token refresh failures so transient network/API errors do not immediately fail agents.

## Best Possible Solution

Add provider-safe bounded retry at the shared OAuth refresh boundary, preferably using retryAsync with an explicit transient-error classifier while preserving locks, adoption fallbacks, plugin hooks, and refresh_token_reused recovery.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an expired OAuth profile whose getOAuthApiKey call throws a network timeout or HTTP 500 will call the shared refresh helper once, then surface OAuthManagerRefreshError after existing fallbacks do not recover it.

## Solution Assessment

Yes, the requested direction is maintainable if implemented as bounded retry with a provider-safe transient-error classifier. Retrying every OAuth error with an ad hoc helper would be weaker because it could delay or mask permanent auth failures.
