## Summary

Keep open: current main still performs ordinary OAuth refresh as a single provider call, and the only retry-like recovery path is specific to refresh_token_reused. The two linked retry pull requests are closed and unmerged, so the requested bounded transient-refresh retry remains unresolved.

## What This Changes

The issue requests retrying transient OAuth token refresh failures instead of failing after a single refresh API call.

## Best Possible Solution

Land a bounded retryAsync-based OAuth refresh policy with a provider-safe transient-error classifier and focused auth-profile regression coverage.

## Reproduction Assessment

Yes, by source inspection. An expired OAuth credential whose built-in refresh throws a network timeout or HTTP 500 is attempted once, and the only current retry-like path is specific to refresh_token_reused recovery.

## Solution Assessment

Yes, bounded retry with an explicit transient-error classifier is the maintainable direction. Retrying every OAuth error with an ad hoc helper would be weaker because it can mask permanent auth failures and interact badly with locks or provider-owned hooks.

## Risks / Open Questions

- A retry policy must distinguish transient transport or server failures from permanent OAuth failures such as invalid_grant, revoked tokens, 401, or 403.
- Retry placement must preserve the global refresh lock, hard refresh timeout, main-store adoption, provider-plugin refresh hooks, and refresh_token_reused recovery behavior.
- Provider-owned refresh hooks may already retry internally, so shared retry should avoid surprising double-retry semantics.
