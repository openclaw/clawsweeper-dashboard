## Summary

Keep this issue open. Current main still performs the shared OAuth refresh as a single attempt, and the related direct retry pull requests are closed and unmerged; the remaining work is a bounded retry policy with a safe transient-error classifier.

## What This Changes

The issue requests bounded retry behavior for transient OAuth token refresh failures so a single network/API blip does not terminate agent auth resolution.

## Best Possible Solution

Add a bounded shared retry around the actual OAuth refresh attempt, using a provider-safe transient classifier and regression tests that preserve the current locking, adoption, fallback, and token-reuse recovery behavior.

## Reproduction Assessment

Yes, at source level: current main calls the provider refresh path once, and HTTP 500/network timeout are not handled by the existing refresh_token_reused recovery branch. I did not run a live provider outage or network-blip reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact helper as written: retrying every thrown refresh error would also retry permanent OAuth failures. The maintainable direction is a bounded shared retry using the existing retry primitive plus an explicit transient-error classifier.
