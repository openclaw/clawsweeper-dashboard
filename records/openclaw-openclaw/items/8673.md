## Summary

Keep this open: current main still performs the ordinary OAuth refresh call once and only has retry-like recovery for refresh_token_reused, so the reported transient network/API failure path remains unresolved.

## What This Changes

Request adds bounded retry/backoff around OAuth token refresh so transient provider or network failures do not immediately fail agents.

## Best Possible Solution

Implement a bounded auth-profile OAuth refresh retry policy in the current oauth-manager/refreshCredential path using retryAsync, preserving existing lock/adoption/fallback behavior and skipping permanent OAuth failures.

## Reproduction Assessment

Yes, source-reproducible: inject a transient throw from getOAuthApiKey or adapter.refreshCredential on an expired OAuth credential and current main propagates the refresh failure without a generic retry. I did not run a live provider repro in this read-only pass.

## Solution Assessment

Yes on direction, but not as the earlier local withRetry-all-errors patch. The maintainable fix should reuse retryAsync with a transient-only classifier and keep the current OAuth manager semantics intact.
