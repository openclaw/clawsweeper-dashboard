## Label Justifications

- `P2`: A transient refresh failure can interrupt OAuth-backed agent work, but the impact is limited to affected provider calls and has a restart or later-retry workaround.
- `impact:auth-provider`: The issue directly concerns OAuth credential refresh and continued provider authentication.

## Summary

Keep this issue open: current main and v2026.6.6 still perform ordinary OAuth refresh as one provider call, while the only retry-like recovery handles the separate refresh-token-reuse race. The two directly proposed retry PRs are closed unmerged, and no open canonical implementation owns the remaining work; the prior discussion correctly identifies a real gap but the retryable-error policy still needs auth-owner judgment.

## What This Changes

Add bounded retry behavior for transient OAuth token-refresh failures so temporary provider or network errors do not immediately fail agent work.

## Best Possible Solution

Implement bounded retry in the centralized OAuth manager using the shared retry primitive, with an auth-owner-approved transient-error classifier, a total budget compatible with the existing lock and timeout invariants, and focused coverage for transient success, exhaustion, and terminal no-retry behavior.

## Reproduction Assessment

No live current-main transient provider failure was reproduced during this read-only review, but the issue is source-reproducible with high confidence: ordinary refresh errors receive one provider call, and existing tests show HTTP 500 and network timeout do not enter the only specialized recovery branch.

## Solution Assessment

No, the proposed unconditional local withRetry helper is not the best solution because it retries terminal failures and duplicates the repository retry utility. The safer fix belongs at the centralized manager boundary with explicit transient classification and preservation of lock, timeout, credential-adoption, and fallback semantics.

## Risks / Open Questions

- A broad retry-on-every-error policy could repeatedly submit terminal or security-relevant OAuth failures such as invalid_grant, revoked credentials, malformed responses, or provider-specific permanent errors.
- The retry count, per-attempt timeout, and backoff must fit the existing 120-second refresh-call timeout and file-lock stale/wait budgets without holding the cross-agent lock for excessive time.
- Plugin-provided refreshOAuth implementations share this manager boundary, so a generic classifier must account for structured and wrapped errors from both built-in and plugin providers.
