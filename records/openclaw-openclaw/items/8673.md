## Summary

Keep open: current main and the latest shipped release still use a single ordinary OAuth refresh attempt for transient provider/network failures, while the only second-attempt behavior is scoped to refresh-token-reuse/adoption recovery. The existing retry PRs are closed and unmerged, and the remaining work needs an owner-reviewed retry classifier rather than cleanup closure.

## What This Changes

The issue requests bounded retry behavior for transient OAuth token refresh failures so agents do not fail after a single temporary provider or network error.

## Best Possible Solution

Add an owner-reviewed retryAsync integration around OAuth refresh with a transient-only classifier, preserving the existing lock, timeout, adoption, fallback, plugin refresh, and refresh-token-reuse behavior with focused regression tests.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh-token-reuse recovery branch. I did not run a live transient provider outage in this read-only review.

## Solution Assessment

No for the issue body's exact local withRetry helper: retrying every thrown refresh error would retry permanent auth failures and duplicate src/infra/retry.ts. The safer solution is an owner-reviewed retryAsync integration with preserved error causes, lock safety, and a transient-only classifier.
