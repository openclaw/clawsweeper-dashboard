## Summary

Keep open: current main still performs ordinary OAuth refresh with a single provider call, and the only retry-like path is the refresh_token_reused recovery branch. The two linked retry PRs are closed and unmerged, so generic transient refresh retry remains unresolved.

## What This Changes

The issue asks for bounded retry around OAuth token refresh so transient provider or network failures do not immediately fail agents.

## Best Possible Solution

Land a fresh implementation that uses the shared retryAsync primitive with a provider-safe transient-error classifier, preserves existing OAuth recovery semantics, and adds focused auth-profile regression coverage.

## Reproduction Assessment

Yes, by source inspection. An expired OAuth profile whose built-in getOAuthApiKey refresh throws a network timeout or HTTP 500 is attempted once, then existing fallback and refresh_token_reused-specific recovery do not retry the ordinary transient failure before OAuthManagerRefreshError is raised.

## Solution Assessment

Yes, bounded retry with an explicit transient-error classifier is the maintainable direction. Retrying every OAuth error with an ad hoc helper would be weaker because it can add latency, mask permanent auth failures, and interact badly with file locks or provider-owned hooks.
