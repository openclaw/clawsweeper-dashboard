- `impact:auth-provider`: The issue directly concerns OAuth credential refresh and provider authentication routing.

## Summary

Keep this issue open: current main still performs a single provider refresh for ordinary transient failures, while existing recovery only handles credential adoption and refresh-token-reuse races. The request remains useful, but retryability needs an explicit cross-provider policy rather than retrying every OAuth error.

## What This Changes

Add bounded retry behavior for transient failures during OAuth token refresh.

## Best Possible Solution

Define a narrow cross-provider transient-error contract, then use retryAsync at the canonical manager-owned refresh boundary while preserving locking, timeout, credential adoption, fallback, and refresh-token-reuse recovery; cover transient success, exhausted retry, and terminal failure behavior.

## Reproduction Assessment

Yes, at source level: current main deterministically makes one ordinary provider refresh attempt, and tests show HTTP 500 and network timeout errors do not enter its only retry-like recovery branch. This review did not induce a real upstream provider fault.

## Solution Assessment

No, the proposed retry-every-error helper is not the best solution because it would also retry terminal OAuth failures. The narrow maintainable approach is centralized bounded retry with an explicit transient classifier and preservation of the existing manager invariants.
