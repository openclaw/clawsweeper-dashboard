## Summary

Keep open: current main still performs a single generic OAuth refresh attempt for ordinary transient failures, and the only retry-like recovery path remains specific to refresh_token_reused. The linked retry PRs are closed and unmerged, and this auth-sensitive retry policy needs owner/secops review before implementation.

## What This Changes

Add bounded retry logic around OAuth token refresh so transient network or API failures do not immediately fail agents.

## Best Possible Solution

Land a bounded retryAsync-based OAuth refresh policy with an explicit transient-error classifier, preserving existing lock/adoption/fallback/reused-token behavior and adding focused auth-profile tests.

## Reproduction Assessment

Yes, by source inspection. On current main, an expired OAuth credential whose provider refresh throws a network timeout or HTTP 500 is attempted once, and those errors do not enter the existing refresh_token_reused recovery path.

## Solution Assessment

Yes, bounded retry with an explicit transient-error classifier is the maintainable direction. Retrying every OAuth error with an ad hoc helper would be weaker because it can mask permanent auth failures and interact badly with locks or provider-owned hooks.

## Risks / Open Questions

- A retry policy must distinguish transient transport, timeout, rate-limit, and 5xx failures from permanent OAuth failures such as invalid_grant, revoked refresh tokens, 401, or 403.
- Retry placement needs care around the global refresh lock, the 120s refresh call timeout, main-store and external credential adoption, provider-plugin refresh hooks, and refresh_token_reused recovery.
- Provider-owned refresh hooks may already retry internally, so a shared policy could otherwise create unexpected double-retry latency.
