## Summary

Keep this issue open. Current main still performs OAuth refresh as a single attempt for ordinary transient failures; the only retry-like branch is refresh_token_reused recovery, and the direct retry PRs are closed and unmerged.

## What This Changes

Request adding bounded retry logic for transient OAuth token refresh failures.

## Best Possible Solution

Add a bounded shared retry around the actual OAuth refresh attempt, using the existing retry primitive plus a provider-safe transient classifier and focused regression tests.

## Reproduction Assessment

Yes, at source level: current main calls the provider refresh path once, and HTTP 500/network timeout are not handled by the existing refresh_token_reused recovery branch. I did not run a live provider outage or network-blip reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact helper: retrying every thrown refresh error would also retry permanent OAuth failures such as invalid_grant. The maintainable direction is a bounded shared retry using retryAsync with an explicit transient-error classifier.
