## Label Justifications

- `P2`: Temporary OAuth refresh failures can interrupt real agent requests, but the defect is limited to OAuth-backed authentication flows.
- `impact:auth-provider`: The issue directly affects OAuth credential refresh and provider authentication.

## Summary

Current main still performs a single OAuth refresh attempt for ordinary provider or transport failures. Existing locking, timeout, credential-adoption, fallback, and reused-token recovery do not cover transient HTTP/network failures, and the two linked retry PRs are closed unmerged.

## What This Changes

Add bounded retry handling for transient OAuth token-refresh failures so temporary provider or network errors do not immediately fail agent requests.

## Best Possible Solution

Add bounded retry at the shared OAuth manager boundary using the existing retry utility and a narrow transient-error contract, while preserving lock, timeout, credential adoption, fallback, and reused-token semantics.

## Reproduction Assessment

Yes at source level: a transient exception from the shared refresh adapter ends the attempt, while HTTP 500 and network timeout errors are explicitly outside the only reused-token recovery path. No live provider fault injection was run in this read-only review.

## Solution Assessment

No, the proposed unconditional local helper is not the safest implementation. The best fix is shared bounded retry limited to transient failures, with terminal errors such as invalid_grant remaining single-attempt.
