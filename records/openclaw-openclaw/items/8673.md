## Label Justifications

- `P2`: Transient OAuth refresh failures can interrupt OAuth-backed agent work, but the impact is bounded to affected provider calls and currently has restart or later-retry workarounds.
- `impact:auth-provider`: The issue directly concerns OAuth credential refresh and continued provider authentication.

## Summary

Keep this issue open: current main and v2026.6.8 still perform ordinary OAuth refresh as a single provider/plugin refresh attempt, and the only second-attempt path is limited to refresh-token-reuse recovery. Prior discussion and the closed retry PRs are useful context, but the remaining fix needs auth-owner agreement on transient-error classification and retry timing.

## What This Changes

Add bounded retry handling for transient OAuth token-refresh failures so temporary network or API errors do not immediately fail OAuth-backed agent work.

## Best Possible Solution

Add retry at the centralized OAuth manager boundary with retryAsync, an explicit transient-error classifier, and tests preserving lock, fallback, adoption, and refresh-token-reuse semantics.

## Reproduction Assessment

Yes, by source inspection rather than a live transient-provider run. An expired or forced OAuth refresh invokes one provider/plugin refresh call, and current tests show HTTP 500/network timeout do not enter the only specialized refresh-token-reuse recovery branch.

## Solution Assessment

No, the issue's unconditional local withRetry sketch is not the best fix. The safer path is a centralized OAuth manager retry policy using retryAsync plus an auth-owner-approved transient-error classifier and timing budget.

- Provider-plugin refresh hooks share this boundary, so maintainers should avoid surprising double-retry behavior for providers that already retry internally.
