## Label Justifications

- `P2`: A transient refresh failure can interrupt active OAuth-backed agent calls, but the blast radius is limited to affected credentials and provider requests.
- `impact:auth-provider`: The issue directly concerns OAuth token refresh and continuity of provider authentication.

## Summary

Current main and the latest release still perform a single bounded-time OAuth refresh call; locking, credential adoption, fallback, and refresh-token-reused recovery do not retry ordinary transient network or server failures. The reliability gap remains, but retrying every exception would be unsafe without an auth-owner decision on terminal versus retryable provider errors.

## What This Changes

Add bounded retry behavior for transient OAuth token refresh failures without retrying terminal authentication errors.

## Best Possible Solution

Implement manager-owned bounded retry with the existing retryAsync utility and an explicit provider-safe terminal-versus-transient contract, preserving the current lock, timeout, credential-adoption, fallback, and refresh-token-reused behavior.

## Reproduction Assessment

No live current-main failure was reproduced during this read-only review, but the source path is high-confidence reproducible: one thrown HTTP 500 or network-timeout refresh error bypasses the refresh-token-reused recovery and reaches fallback or terminal error handling without retry.

## Solution Assessment

No, not exactly as proposed: a local helper that retries every exception is broader than the safe contract. The best fix is bounded retry at the shared manager boundary with an explicit classifier and focused coverage for transient success, exhaustion, and terminal failures.
