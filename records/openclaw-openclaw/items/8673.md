## Summary

Keep open: current main still has a single-shot OAuth refresh path for ordinary transient failures, and the existing retry-like recovery is limited to refresh_token_reused adoption/retry handling. The remaining work is real, but retry eligibility and error classification in auth-sensitive token code need owner/secops judgment before implementation.

## What This Changes

The issue requests bounded retry logic for transient OAuth token refresh failures so temporary network or provider errors do not immediately fail agent requests.

## Best Possible Solution

Land an owner-approved, bounded retryAsync-based OAuth refresh policy with an explicit transient-error classifier, preserving lock/adoption/fallback/reused-token behavior and adding focused auth-profile regression tests.

## Reproduction Assessment

Yes, by source inspection rather than a live provider run. On current main, an expired built-in OAuth credential calls getOAuthApiKey once, and HTTP 500 or network-timeout errors do not enter the existing refresh_token_reused recovery path.

## Solution Assessment

No for the issue's broad retry-every-error sketch. The safer solution is a bounded retryAsync policy in the owner-approved refresh path, gated by a transient-error classifier and covered by auth-profile tests.
