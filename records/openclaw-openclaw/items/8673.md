## Summary

Keep this issue open. Current main still performs the shared OAuth refresh attempt once, with only refresh-token-reuse recovery and provider-specific fixes covering narrower cases; the direct retry pull requests at https://github.com/openclaw/openclaw/pull/8677 and https://github.com/openclaw/openclaw/pull/45975 are closed and unmerged.

## What This Changes

The issue requests bounded retry behavior for transient OAuth token refresh failures so temporary network/API errors do not immediately fail agents.

## Best Possible Solution

Add a bounded shared retry around the actual OAuth refresh attempt using the existing retry primitive and a provider-safe transient classifier, while preserving current locking, adoption, fallback, and token-reuse recovery behavior.

## Reproduction Assessment

Yes, at source level: current main calls the shared provider refresh path once, and HTTP 500/network timeout are not handled by the existing refresh-token-reuse recovery branch. I did not run a live provider outage or network-blip reproduction.

## Solution Assessment

No for the exact helper proposed in the issue body: retrying every thrown refresh error would also retry permanent OAuth failures. The maintainable direction is a bounded shared retry using retryAsync plus an explicit transient-error classifier and owner-reviewed placement.
