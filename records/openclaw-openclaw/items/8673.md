## Summary

Keep open: current main and the latest release still make one ordinary OAuth refresh attempt for provider/network failures, and the only retry-like recovery remains scoped to refresh-token-reuse/adoption. The prior discussion and closed retry PRs remain relevant, but the retry semantics need owner review before implementation.

## What This Changes

The issue requests bounded retry logic for transient OAuth token refresh API failures so agents do not fail after a single network blip or timeout.

## Best Possible Solution

Add an owner-reviewed retryAsync integration around OAuth refresh with a transient-only classifier, preserved error causes, and focused regression tests for transient success, retry exhaustion, and non-retryable auth failures.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh-token-reuse recovery branch. I did not run a live transient provider outage in this read-only review.

## Solution Assessment

No for the issue body's exact local withRetry helper: retrying every thrown refresh error would also retry permanent auth failures and duplicate src/infra/retry.ts. The safer path is an owner-reviewed retryAsync integration with a transient-only classifier and preserved lock/fallback behavior.
