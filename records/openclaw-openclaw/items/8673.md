## Summary

Keep this issue open: current main still performs ordinary OAuth refresh as a single provider call, while its only retry-like recovery is narrowly limited to refresh-token reuse after another process updates stored credentials. The two directly proposed retry PRs were closed unmerged, and no open canonical implementation now owns the remaining work.

## What This Changes

Requests bounded retry handling for transient OAuth token-refresh failures.

## Best Possible Solution

Implement bounded retry at the shared OAuth manager boundary using `src/infra/retry.ts`, with an explicit provider-safe transient-error contract and regression coverage preserving locking, timeout, credential adoption, fallback, refresh-token-reuse, and terminal-error behavior.

## Reproduction Assessment

Yes at source level: on current main, an ordinary transient exception from the provider refresh call follows the single-shot failure path because it does not match the refresh-token-reuse recovery classifier. This read-only review did not induce a live upstream outage.

## Solution Assessment

No, the proposed unconditional retry helper is not the safest final solution. Retry should reuse the shared infrastructure, classify transient versus terminal failures explicitly, and respect the existing manager timeout and credential-rotation invariants.

## Risks / Open Questions

- A retry-all implementation could repeatedly submit terminal failures such as invalid_grant or revoked credentials, delaying actionable reauthentication errors.
- Retry and timeout budgets must be designed together so exponential backoff does not silently violate the manager's bounded refresh-call latency.
- Providers may expose network and HTTP failures through different error shapes, so a generic message-only classifier could retry unsafe failures or miss genuinely transient ones.
