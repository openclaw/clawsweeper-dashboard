## Summary

Keep open: current main and v2026.5.22 still make one ordinary OAuth refresh call for provider/plugin/network failures, and the only retry-like recovery is still scoped to refresh-token-reuse/adoption. The directly linked retry PRs are closed and unmerged, so the remaining work needs owner-reviewed retry semantics rather than cleanup closure.

## What This Changes

The issue requests bounded retry handling for transient OAuth token refresh failures so agents do not fail on a single temporary provider or network error.

## Best Possible Solution

Add an owner-reviewed retryAsync integration around OAuth refresh with a transient-only classifier, preserving existing lock, timeout, adoption, fallback, plugin refresh, and refresh-token-reuse behavior with focused regression tests.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh-token-reuse recovery branch. I did not run a live transient provider outage in this read-only review.

## Solution Assessment

No for the issue body's exact local withRetry helper because it would retry permanent OAuth failures and duplicate src/infra/retry.ts. The maintainable solution is a retryAsync integration with preserved error causes, lock safety, and a transient-only classifier.
