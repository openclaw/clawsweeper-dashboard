## Summary

Keep open: current main and the latest release still refresh ordinary OAuth credentials with a single provider/plugin call, while the existing second-attempt path is limited to refresh-token-reuse recovery. The related retry PRs are closed and unmerged, and the remaining work needs auth-provider owner review because the retry classifier and placement affect credential behavior.

## What This Changes

The issue requests bounded retry logic for transient OAuth token refresh failures so one network/API blip does not fail agent authentication.

## Best Possible Solution

Add an owner-reviewed bounded retryAsync integration for OAuth refresh with a transient-error classifier, preserving existing lock, timeout, fallback/adoption, plugin refresh, and refresh_token_reused behavior with focused regression tests.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh_token_reused recovery branch. I did not run a live transient network or provider-outage reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact local helper: retrying every thrown refresh error would retry permanent auth failures and duplicate src/infra/retry.ts. The safer path is an owner-reviewed retryAsync integration with preserved error causes, lock safety, and a transient-only classifier.
