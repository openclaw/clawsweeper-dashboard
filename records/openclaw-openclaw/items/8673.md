## Summary

Keep this issue open. Current main and the latest release still make ordinary OAuth refresh as a single provider/plugin refresh call; only the refresh_token_reused recovery path can call refresh again, and the prior retry PRs for this report are closed unmerged. The useful direction is valid, but retrying every OAuth refresh error would be unsafe without maintainer-owned transient-error classification and placement.

## What This Changes

Request adding bounded retry/backoff to OAuth token refresh so transient network or API failures do not immediately fail agents.

## Best Possible Solution

Add a bounded shared retry around OAuth refresh using retryAsync with a transient-only classifier, while preserving locking, timeout, fallback/adoption, provider hooks, and refresh_token_reused recovery behavior.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh_token_reused recovery branch. I did not run a live network-blip or provider-outage reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact helper: retrying every thrown refresh error would also retry permanent OAuth failures and duplicate the existing retryAsync primitive. The safer solution is an owner-reviewed retryAsync integration with preserved error causes and a transient-only classifier.
