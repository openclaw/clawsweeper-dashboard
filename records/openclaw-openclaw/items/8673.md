## Summary

Keep open: current main and the latest release still make one ordinary OAuth refresh attempt for provider/plugin refresh failures, and the only retry-like recovery remains scoped to refresh-token-reuse/adoption rather than ordinary transient HTTP, timeout, or network failures.

## What This Changes

The issue requests bounded retry logic for transient OAuth token refresh failures so temporary provider/network failures do not immediately fail agents.

## Best Possible Solution

Keep this issue open and implement an owner-reviewed bounded retry policy around OAuth refresh using `retryAsync`, a transient-only classifier, preserved error causes, and focused auth-profile regression tests.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh-token-reuse recovery branch. I did not run a live transient provider outage in this read-only review.

## Solution Assessment

No for the issue body's exact local `withRetry` helper: retrying every thrown refresh error would also retry permanent auth failures and duplicate `src/infra/retry.ts`. The safer path is an owner-reviewed `retryAsync` integration with a transient-only classifier and tests around lock/fallback behavior.
