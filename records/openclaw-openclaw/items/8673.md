## Summary

Keep this issue open: current main and v2026.5.12 still make ordinary OAuth refresh as a single provider/plugin call, and the prior retry PRs are closed unmerged. The request is valid, but retry classification and placement need auth-owner review because this path handles credentials and provider-specific permanent failures.

## What This Changes

Request a bounded retry policy for transient OAuth token refresh failures so temporary network/API errors do not immediately fail agents.

## Best Possible Solution

Add bounded retryAsync around OAuth refresh with a transient-only classifier, preserving the existing lock, timeout, fallback/adoption, provider hook, and refresh_token_reused behavior.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh_token_reused recovery branch. I did not run a live network-blip or provider-outage reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact helper: retrying every thrown refresh error would also retry permanent OAuth failures and duplicate the existing retryAsync primitive. The safer path is an owner-reviewed retryAsync integration with preserved error causes and a transient-only classifier.
