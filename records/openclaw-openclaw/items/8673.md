## Summary

Keep this issue open. Current main still performs ordinary OAuth refresh as a single provider/plugin refresh attempt, existing recovery is limited to refresh_token_reused/adoption paths, and the direct retry PRs are closed unmerged; retry classification and placement need maintainer review.

## What This Changes

The issue requests bounded retry handling for transient OAuth token-refresh failures so agents do not fail after a single temporary provider/API error.

## Best Possible Solution

Add a bounded shared retry around the actual OAuth refresh attempt using retryAsync with an explicit transient-only classifier, preserving locks, timeouts, adoption/fallback behavior, provider hooks, and token-reuse recovery.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are not handled by the existing refresh_token_reused recovery branch. I did not run a live provider outage or network-blip reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact helper: retrying every thrown refresh error would also retry permanent OAuth failures such as invalid_grant. The safer direction is a bounded shared retry using retryAsync with an explicit transient-error classifier and owner-reviewed placement.
