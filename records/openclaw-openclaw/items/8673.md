## Summary

Keep open. Current main still single-shots ordinary OAuth refresh and only has a retry-like branch for refresh_token_reused recovery; the linked retry PRs #8677 and #45975 are closed and unmerged.

## What This Changes

Add bounded retry behavior for transient OAuth token refresh failures so a single network/API blip does not terminally fail agents.

## Best Possible Solution

Add a bounded retry around OAuth refresh using the existing retryAsync primitive and an explicit transient-error classifier, while preserving lock, timeout, adoption, fallback, and refresh-token-reused semantics with focused regression tests.

## Reproduction Assessment

Yes, at source level. A focused test can mock adapter.refreshCredential or getOAuthApiKey to throw a network timeout or HTTP 500 once and then succeed; current main performs one ordinary refresh attempt before fallback/error handling.

## Solution Assessment

No, not as the issue body's blanket retry sketch. The maintainable solution is a bounded retry policy with a transient-error classifier so permanent OAuth failures are not treated as network blips.
