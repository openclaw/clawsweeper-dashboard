## Summary

Keep open: current main still performs ordinary OAuth refresh with a single provider call, and the existing retry-like path only covers refresh_token_reused recovery. The linked retry pull requests are closed and unmerged, so bounded retry for transient refresh failures remains unresolved.

## What This Changes

The issue requests bounded retry logic for transient OAuth token refresh failures so agents do not fail after a single network or API blip.

## Best Possible Solution

Land a fresh implementation that uses retryAsync with a provider-safe transient-error classifier, preserves existing OAuth recovery semantics, and adds focused auth-profile regression coverage.

## Reproduction Assessment

Yes, by source inspection. An expired OAuth profile whose built-in getOAuthApiKey refresh throws a network timeout or HTTP 500 is attempted once, then current fallback and refresh_token_reused-specific recovery do not retry the ordinary transient failure before OAuthManagerRefreshError is raised.

## Solution Assessment

Yes, bounded retry with an explicit transient-error classifier is the maintainable direction. Retrying every OAuth error with an ad hoc helper would be weaker because it can add latency, mask permanent auth failures, and interact badly with locks or provider-owned hooks.
