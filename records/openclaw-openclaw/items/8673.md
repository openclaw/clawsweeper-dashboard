## Label Justifications

- `P2`: Transient OAuth refresh failures can interrupt OAuth-backed agent work, but the impact is limited to affected provider calls and has restart or later-retry workarounds.
- `impact:auth-provider`: The issue directly concerns OAuth credential refresh and continued provider authentication.

## Summary

Keep open: current main and the latest release still perform ordinary OAuth refresh as one provider/plugin refresh call, while the only retry-like second attempt is limited to refresh-token-reuse recovery. The linked retry PRs are closed unmerged, so the remaining work needs auth-owner agreement on a transient-error classifier and retry budget.

## What This Changes

The issue requests bounded retry logic for transient OAuth token refresh failures so agents do not fail after one temporary provider/API error.

## Best Possible Solution

Implement bounded retry in the centralized OAuth manager using retryAsync, an auth-owner-approved transient-error classifier, and focused tests that preserve lock, adoption, fallback, provider-plugin, and refresh-token-reuse semantics.

## Reproduction Assessment

Yes, by source inspection rather than a live transient-provider failure: expired or forced OAuth refresh invokes one provider refresh call, and current tests show HTTP 500/network timeout do not enter the only specialized refresh-token-reuse recovery branch.

## Solution Assessment

No, the issue's proposed unconditional local withRetry helper is not the best fix. The safer solution belongs at the centralized OAuth manager boundary with retryAsync, an explicit transient-error classifier, and maintainer-approved retry timing.
