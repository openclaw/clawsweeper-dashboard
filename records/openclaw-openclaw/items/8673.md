## Summary

Keep open: current main and v2026.5.20 still make ordinary OAuth refresh as a single provider/plugin call; only refresh-token-reuse recovery gets a second attempt, and the linked retry PRs are closed without merging.

## What This Changes

The issue requests bounded retry handling for transient OAuth token refresh failures so one network or provider blip does not fail agents.

## Best Possible Solution

Add an owner-reviewed retryAsync integration for OAuth refresh with a transient-error classifier, preserving current lock, timeout, adoption, fallback, plugin refresh, and refresh-token-reuse behavior with focused regression tests.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh-token-reuse recovery branch. I did not run a live transient network or provider-outage reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact local withRetry helper: retrying every thrown refresh error would retry permanent auth failures and duplicate src/infra/retry.ts. The safer path is an owner-reviewed retryAsync integration with preserved error causes, lock safety, and a transient-only classifier.

## Risks / Open Questions

- A retry policy needs a transient-only classifier so permanent OAuth failures such as invalid_grant, revoked refresh tokens, 401/403, and sign-in-required errors are not retried as temporary failures.
- Retry placement must preserve the global refresh lock, auth-store lock, hard refresh timeout, external/main-store credential adoption, provider-plugin hook, and refresh-token-reuse recovery semantics.
- Provider plugins may already own custom refresh behavior, so a shared retry layer needs a clear contract to avoid double retries or delayed terminal auth failures.
