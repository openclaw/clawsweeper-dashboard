## Summary

Keep this issue open. Current main still performs ordinary OAuth refresh as a single provider/plugin refresh attempt; existing recovery is refresh_token_reused/adoption-specific, the direct retry PRs are closed and unmerged, and retry classification/placement needs maintainer review.

## What This Changes

Request to add bounded retry logic for transient OAuth token refresh API failures so agents do not fail on a single network blip or timeout.

## Best Possible Solution

Add a bounded shared retry around the actual OAuth refresh attempt using retryAsync with a transient-only classifier, preserving locks/timeouts/adoption/fallback behavior and token-reuse recovery with focused tests.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are not handled by the existing refresh_token_reused recovery branch. I did not run a live provider outage or network-blip reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact helper: retrying every thrown refresh error would also retry permanent OAuth failures such as invalid_grant. The safer direction is a bounded shared retry using retryAsync with an explicit transient-error classifier and owner-reviewed placement.
