## Label Justifications

- `P2`: Transient OAuth refresh failures can interrupt OAuth-backed agent work, but the impact is limited to affected provider calls and has restart or later-retry workarounds.
- `impact:auth-provider`: The issue directly concerns OAuth credential refresh and continued provider authentication.

## Summary

Keep this issue open: current main and the latest release still perform ordinary OAuth refresh as one provider/plugin call, and the only retry-like second attempt is limited to refresh-token-reuse recovery. The remaining work is real, but it needs auth-owner agreement on transient error classification and retry budget before a safe fix PR.

## What This Changes

Requests bounded retry for transient OAuth token refresh failures so a single network or API blip does not fail OAuth-backed agent calls.

## Best Possible Solution

Implement bounded retry in the centralized OAuth manager using retryAsync, an auth-owner-approved transient-error classifier, and focused tests that preserve locking, adoption, fallback, and refresh-token-reuse semantics.

## Reproduction Assessment

Yes, by source inspection rather than a live transient-provider failure. An expired or forced OAuth refresh invokes one provider refresh call, and current tests show HTTP 500/network timeout do not enter the only specialized refresh-token-reuse recovery branch.

## Solution Assessment

No, the issue's proposed unconditional local withRetry helper is not the best solution. The safer fix belongs at the centralized OAuth manager boundary using retryAsync plus an explicit transient-error classifier and maintainer-approved timing budget.
