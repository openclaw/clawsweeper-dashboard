## Label Justifications

- `P2`: Transient OAuth refresh failures can interrupt agent workflows, but the known impact is bounded and normally recoverable.
- `impact:auth-provider`: The issue directly concerns OAuth token refresh and provider credential resolution.

## Summary

Keep this issue open: current main still performs each ordinary OAuth token refresh as one provider call, and the existing retry-like recovery only handles the separate refresh-token-reuse race. Both directly proposed retry PRs were closed unmerged, so no viable canonical implementation owns the remaining work.

## What This Changes

Add bounded retry handling for transient OAuth token refresh failures without retrying terminal authorization errors.

## Best Possible Solution

Add a manager-owned policy using the shared retry primitive, retry only explicitly transient transport or server failures, cap aggregate latency within the refresh-lock invariants, and cover transient recovery, exhaustion, and terminal no-retry behavior.

## Reproduction Assessment

Yes at source level: current main makes one provider refresh call, while HTTP 500 and network-timeout errors do not enter the existing refresh-token-reuse recovery path. A controlled live provider failure was not reproduced during this read-only review.

## Solution Assessment

No, the proposed unconditional local retry helper is not the best fix. The centralized OAuth manager should own a classified policy integrated with its existing timeout, locking, credential-adoption, and fallback semantics.
