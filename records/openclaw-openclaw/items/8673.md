## Summary

Keep open: current main still lacks bounded retry for ordinary transient OAuth refresh failures, and the only retry-like path remains the refresh_token_reused recovery branch. The two linked retry PRs are closed and unmerged, while the open discussion already identifies classifier, lock-scope, and provider-plugin interactions that need owner review.

## What This Changes

The issue requests bounded retry logic around OAuth token refresh so transient network or API failures do not immediately fail agents.

## Best Possible Solution

Land a bounded retryAsync-based OAuth refresh policy with a provider-safe transient-error classifier, preserving existing lock/adoption/fallback/reused-token behavior and adding focused auth-profile regression coverage.

## Reproduction Assessment

Yes, by source inspection. On current main, an expired OAuth credential whose provider refresh throws a network timeout or HTTP 500 is attempted once, and those errors do not enter the existing refresh_token_reused recovery path.

## Solution Assessment

Yes, bounded retry with an explicit transient-error classifier is the maintainable direction. Retrying every OAuth error with an ad hoc helper would be weaker because it can mask permanent auth failures and interact badly with locks or provider-owned hooks.

## Risks / Open Questions

- A retry policy must distinguish transient transport/server failures from permanent OAuth failures such as invalid_grant, revoked refresh tokens, 401, or 403.
- Retry placement needs care around the global refresh lock, hard refresh timeout, main-store adoption, provider-plugin hooks, and refresh_token_reused recovery.
- Provider-owned refresh hooks may already retry internally, so shared retry behavior could otherwise create surprising double-retry delays.
