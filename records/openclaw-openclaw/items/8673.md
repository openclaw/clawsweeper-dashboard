## Summary

Keep open: current main and the latest shipped release still refresh ordinary OAuth credentials with one provider/API call, and the only retry-like path is limited to refresh_token_reused recovery. The work is valid, but the retry policy belongs behind owner/secops review because it must distinguish transient failures from permanent OAuth failures.

## What This Changes

Requests bounded retry behavior for transient OAuth refresh API failures so agents do not fail after a single temporary network or provider error.

## Best Possible Solution

Land an owner-approved, bounded retryAsync-based OAuth refresh policy with an explicit transient-error classifier, preserving current lock/adoption/fallback/reused-token behavior and adding focused auth-profile regression tests.

## Reproduction Assessment

Yes, by source inspection rather than a live provider run. On current main, an expired built-in OAuth credential calls getOAuthApiKey once, and HTTP 500 or network-timeout errors do not enter the refresh_token_reused recovery path.

## Solution Assessment

No for the issue's broad retry-every-error sketch. The safer solution is a bounded retryAsync policy in the owner-approved refresh path, gated by a transient-error classifier and covered by auth-profile tests.
