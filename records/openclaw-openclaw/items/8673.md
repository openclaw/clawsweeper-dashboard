## Label Justifications

- `P2`: A transient OAuth refresh failure can interrupt an agent workflow, but the reported failure is bounded and recoverable through restart or another attempt.
- `impact:auth-provider`: The issue concerns OAuth credential refresh and provider authentication behavior.

## Summary

Keep this issue open: current main still performs one OAuth refresh attempt, and both linked retry implementations were closed without merging. The remaining design question is which transport/server failures are safe to retry without delaying terminal failures that require re-authentication.

## What This Changes

Add bounded retries for transient OAuth token-refresh failures while avoiding retries for terminal authentication errors.

## Best Possible Solution

Implement bounded retry at the shared OAuth manager boundary with an explicit transient-only classifier, preserving refresh locking, per-attempt timeout, credential adoption, fallback, and refresh_token_reused recovery behavior.

## Reproduction Assessment

Yes, from source inspection: current main makes one provider refresh call, and existing tests show HTTP 500 and network-timeout errors are outside the only retry-like reused-token recovery branch. The original provider incident was not reproduced live during this read-only review.

## Solution Assessment

No, an unconditional wrapper that retries every thrown error is not the best solution. Retry should be centralized in the shared manager and limited to explicitly transient transport or server failures so terminal OAuth errors fail promptly.
