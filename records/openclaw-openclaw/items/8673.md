## Summary

Keep this issue open: current main and the latest release still perform ordinary OAuth refresh as a single provider/plugin refresh attempt, and the closed related retry PRs did not land. The request is valid, but the fix needs owner review because retry classification and lock placement affect auth-provider behavior.

## What This Changes

The issue requests bounded retry behavior for transient OAuth token refresh failures so agents do not fail after a single temporary provider or network error.

## Best Possible Solution

Add a bounded `retryAsync` integration for OAuth refresh with a transient-error classifier, preserving the existing lock, timeout, fallback/adoption, plugin refresh, and `refresh_token_reused` behavior with focused auth-profile regression tests.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing `refresh_token_reused` recovery branch. I did not run a live transient network or provider-outage reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact local helper: retrying every thrown refresh error would retry permanent auth failures and duplicate `src/infra/retry.ts`. The safer path is an owner-reviewed `retryAsync` integration with preserved error causes, lock safety, and a transient-only classifier.

## Risks / Open Questions

- A retry policy needs a transient-only classifier so permanent OAuth failures such as `invalid_grant`, revoked tokens, 401/403, and sign-in-required errors are not retried as temporary provider failures.
- Retry placement must preserve the current global refresh lock, auth-store lock, timeout, external/main-store adoption, provider plugin hook, and `refresh_token_reused` recovery semantics.
- Provider plugins may already own custom refresh behavior, so a shared retry layer needs a clear contract to avoid surprising double retries or delayed terminal auth failures.
