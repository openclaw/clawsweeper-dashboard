## Summary

Keep this issue open. Current main still performs ordinary OAuth refresh as a single provider/plugin call, and the only retry-like branch is the existing refresh_token_reused recovery path, so transient HTTP/network refresh failures remain unhandled as a generic bounded retry policy.

## What This Changes

The issue requests bounded retry behavior for transient OAuth token refresh failures so agents do not fail after one temporary provider/API error.

## Best Possible Solution

Add a bounded shared retry around OAuth refresh using retryAsync with an explicit transient-only classifier, while preserving the existing lock, timeout, fallback/adoption, provider-hook, and token-reuse recovery behavior.

## Reproduction Assessment

Yes, at source level: current main calls the provider/plugin OAuth refresh path once, and HTTP 500/network timeout are outside the existing refresh_token_reused recovery branch. I did not run a live provider outage or network-blip reproduction in this read-only review.

## Solution Assessment

No for the issue body's exact helper: retrying every thrown refresh error would also retry permanent OAuth failures and duplicate the existing retryAsync primitive. The safer direction is an owner-reviewed retryAsync integration with a transient-error classifier and careful placement around the current lock/fallback flow.
