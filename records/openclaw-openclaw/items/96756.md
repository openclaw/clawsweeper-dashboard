## Summary

Keep open: this targets a real Feishu multi-bot NO_REPLY bug, but the patch preserves the self-mention in the same content string later used as Feishu command text, so mention-prefixed slash commands can stop being recognized; the PR body also provides Vitest output only, not real Feishu behavior proof.

## What This Changes

The PR changes Feishu mention normalization so the receiving bot's own mention becomes `@name` instead of being removed, and updates Feishu mention parsing tests to expect that preserved text.

## Best Possible Solution

Preserve the bot self-mention in agent-visible Feishu context while deriving command bodies from mention-stripped, slash-clean text for both p2p and group paths, then provide redacted real Feishu or runtime-log proof for the multi-bot and slash-command cases.

## Reproduction Assessment

Yes, at source level: current main strips the receiving Feishu bot's own mention before building ctx.content, and the PR head preserves it in the same field that command parsing later reads. I did not run a live Feishu tenant reproduction.

## Solution Assessment

No in its current shape: preserving self-mention context is the right direction, but using one normalized string for both agent context and command bodies is not the narrowest maintainable fix. The safer repair splits agent-visible mention context from slash-clean command text.
