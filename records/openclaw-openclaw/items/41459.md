## Label Justifications

- `P2`: This is a meaningful multi-agent Discord usability improvement, but routing and message delivery continue to function.
- `impact:other`: Users sharing one Discord bot across agents cannot visually attribute successful replies to the routed agent.

## Summary

Keep open. Current main still restricts Discord webhook persona delivery to credentialed thread bindings; ordinary route bindings have no supported identity override. The request fits the roadmap, but the public configuration, secret ownership, delivery consistency, and fallback behavior require a maintainer product decision.

## What This Changes

Add a configurable Discord identity override so agents routed through one bot token can display distinct names by binding or channel.

## Best Possible Solution

Expose an opt-in Discord-owned outbound persona contract for ordinary routed replies by reusing the existing webhook sender and `OutboundIdentity` plumbing, with explicit credential storage, target precedence, edit/thread behavior, and fallback semantics; do not mutate the bot’s global nickname per response.

## Reproduction Assessment

Not applicable as a bug reproduction: this requests a new Discord presentation capability. Current source and documentation nevertheless confirm that ordinary route bindings cannot configure webhook credentials or visible identity.

## Solution Assessment

No. Changing a nickname whenever an agent responds would mutate shared Discord state and race across channels; extending the existing per-message webhook identity path inside the Discord plugin is safer, subject to maintainer approval of its public configuration and fallback contract.

## Risks / Open Questions

- Adding Discord presentation fields to generic route bindings would couple transport-neutral routing configuration to one channel and create an unclear ownership boundary.
- A webhook design needs an explicit secret-storage and precedence contract, including whether identity is selected per agent, account, channel, or binding.
- Fallback behavior must be intentional: webhook failures could silently change visible identity, duplicate delivery, or stop replies unless send, retry, and fallback semantics are designed together.
- The implementation must keep replies, edits, attachments, threads, proactive sends, and ordinary routed responses behaviorally consistent.
