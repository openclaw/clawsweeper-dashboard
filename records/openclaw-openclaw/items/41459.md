## Summary

Keep open: current main still lacks a per-binding Discord visible identity override for ordinary route-bound replies, and the remaining work is a new public Discord config/API decision rather than conservative cleanup.

## What This Changes

The issue requests an optional Discord binding identity override so one bot token can present different visible names per routed channel.

## Best Possible Solution

Define a documented Discord plugin-local route-binding persona config that uses webhook username/avatar delivery for normal channel replies, with secure credential handling and clear bot-send fallback behavior.

## Reproduction Assessment

Not applicable as a bug report. Source inspection gives a high-confidence absence path: current route binding config rejects nickname/persona fields, and ordinary Discord replies do not receive a route-binding identity.

## Solution Assessment

No for the literal nickname mutation approach; webhook persona delivery is the narrower maintainable direction because Discord exposes per-message webhook username/avatar overrides, while nicknames are guild-member attributes. The exact OpenClaw config surface still needs maintainer product/API judgment. ([discord.com](https://discord.com/developers/docs/resources/webhook?source=post_page---------------------------))
