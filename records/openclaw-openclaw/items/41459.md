## Summary

Keep open: current main still lacks a per-binding Discord visible identity override for ordinary route-bound replies. Webhook persona delivery exists only for stored thread bindings, and the remaining change adds public config/API behavior plus Discord credential and fallback semantics.

## What This Changes

Add an optional Discord binding nickname/persona override so one bot token can present different agent identities in different bound channels.

## Best Possible Solution

Define a documented Discord plugin-local route-binding persona config that uses webhook username/avatar delivery for configured normal channel replies, with secure credential handling and clear bot-send fallback behavior.

## Reproduction Assessment

Not applicable as a bug report, but source inspection gives a high-confidence absence path: current route binding config rejects nickname/persona fields and ordinary Discord replies receive no route-binding identity. The only implemented persona path is for stored thread bindings with webhook credentials.

## Solution Assessment

No for the literal nickname mutation approach; a documented webhook persona delivery path is the narrower maintainable direction because Discord webhooks support per-message identity while nicknames are guild-member updates. The exact config surface still needs maintainer product/API judgment.
