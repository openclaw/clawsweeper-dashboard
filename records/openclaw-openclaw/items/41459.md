## Summary

Keep open: current main and v2026.6.8 still do not expose a Discord binding-level nickname or ordinary routed-reply identity override. The remaining work is a valid Discord product/config/security decision, not an automatic fix candidate.

## What This Changes

Requests an optional Discord binding-level nickname override so one bot token can appear with different identities across bound channels.

## Best Possible Solution

Have maintainers choose one Discord visible-identity model, then implement it in the Discord plugin with schema, docs, tests, and explicit credential, permission, fallback, and upgrade behavior.

## Reproduction Assessment

Not applicable; this is a requested new Discord identity/config capability, not a broken documented behavior. Source inspection gives high-confidence absence of a binding-level nickname override on current main.

## Solution Assessment

Unclear as proposed. A binding-level nickname is plausible, but the best solution must first choose between Discord webhook delivery, agent identity, route binding config, and self-profile mutation so OpenClaw does not create multiple identity sources.
