## Summary

Keep open. Current main still lacks ordinary Discord per-binding or per-channel visible identity overrides; the existing webhook username/avatar path only applies to thread-bound replies, and the related webhook request is already closed back to this as the canonical remaining feature. The fix would add public config plus Discord credential and fallback semantics, so it needs maintainer product review rather than an automated repair.

## What This Changes

The issue requests an optional Discord binding identity override so one bot token can display different names per channel.

## Best Possible Solution

Define a Discord-plugin-owned visible identity contract for routed Discord replies, likely reusing OutboundIdentity plus webhook username/avatar delivery, with schema, docs, tests, and explicit credential/fallback rules.

## Reproduction Assessment

Not applicable; this is a feature request rather than a bug. Source inspection shows the proposed nickname field is not in the current binding schema and ordinary Discord sends still use the bot account unless the reply is thread-bound with webhook metadata.

## Solution Assessment

Unclear. The user need is concrete and current webhook username/avatar primitives exist, but maintainers should choose the public config and credential contract before implementation.
