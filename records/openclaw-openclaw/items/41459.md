## Summary

Keep open: current main still lacks a per-binding Discord visible identity override for ordinary route-bound replies. The only implemented persona path is stored thread-binding webhook delivery, and extending this to normal bindings adds public config, secret handling, and fallback semantics that need maintainer product/API direction.

## What This Changes

Add an optional Discord binding identity override so one bot token can show different agent names per routed channel.

## Best Possible Solution

Define a documented Discord plugin-local route-binding persona config that uses webhook username/avatar delivery for normal channel replies, stores credentials securely, and has explicit bot-send fallback behavior.

## Reproduction Assessment

Not applicable as a bug report. Source inspection on current main gives a high-confidence absence path: strict route binding config has no nickname/persona field, and normal Discord replies only receive identity through stored thread bindings.

## Solution Assessment

No for the literal nickname mutation approach; Discord webhook execute supports `username` and `avatar_url` per message, while nickname is a guild member field. A documented webhook persona path is narrower, but the exact config/API still needs maintainer direction. ([docs.discord.com](https://docs.discord.com/developers/resources/webhook?utm_source=openai))
