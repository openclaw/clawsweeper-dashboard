## Summary

Keep open. Current main still lacks a Discord route-binding identity override: normal bindings only route conversations, while Discord webhook persona delivery is limited to thread-bound sessions with stored webhook credentials. The related webhook request at https://github.com/openclaw/openclaw/issues/46351 is closed back to this issue, and the earlier implementation PR at https://github.com/openclaw/openclaw/pull/6835 was closed with maintainer direction to solve it through deeper integration.

## What This Changes

Request an optional Discord per-channel or per-binding identity override so one Discord bot token can show different agent display names in different routed channels.

## Best Possible Solution

Define a Discord-plugin-owned visible identity contract for routed Discord replies, likely reusing OutboundIdentity plus webhook username/avatar delivery, with schema, docs, tests, and explicit credential/fallback rules.

## Reproduction Assessment

Not applicable; this is a feature request rather than a bug. Source inspection shows route bindings do not expose nickname/identity fields and Discord webhook persona delivery is only used for thread-bound sessions with webhook credentials.

## Solution Assessment

Unclear. The user need is valid, but a raw binding nickname is likely too narrow because Discord visible identity also needs webhook credentials, avatar handling, fallback behavior, and plugin/core ownership rules.
