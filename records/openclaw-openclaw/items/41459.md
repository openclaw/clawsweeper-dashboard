## Summary

Keep open. Current main still lacks a Discord route-binding identity override: ordinary route bindings are strict routing/session records, while Discord webhook persona delivery is only wired for bound thread sessions with webhook credentials; the related webhook request is already closed back to this issue as the canonical remaining feature.

## What This Changes

The issue requests an optional per-binding Discord visible identity override so multiple agents using one bot token can appear with different names per channel.

## Best Possible Solution

Define a Discord-plugin-owned visible identity contract for routed Discord replies, likely reusing OutboundIdentity plus webhook username/avatar delivery, with schema, docs, tests, and explicit credential/fallback rules.

## Reproduction Assessment

Not applicable; this is a feature request rather than a bug. The current behavior is source-proven: route bindings do not expose a Discord nickname or outbound identity field, and ordinary Discord replies only get visible identity through bound thread webhook plumbing.

## Solution Assessment

Unclear. The user need is valid, but a raw binding nickname alone is probably too narrow because Discord visible identity also needs webhook credential, avatar, fallback, and ownership rules.
