## Summary

Keep open: current main and v2026.6.10 still lack a per-binding Discord nickname or ordinary routed-reply identity override. The closest open related item, https://github.com/openclaw/openclaw/issues/34004, is adjacent self-profile work rather than this one-token, multi-agent visible-identity decision.

## What This Changes

The issue requests an optional Discord binding-level nickname or identity override so one bot token can present different agent names in different Discord channels.

## Best Possible Solution

Have maintainers choose one Discord plugin-owned visible-identity model, then implement it with explicit schema, docs, focused tests, and security handling for webhook credentials or bot-profile mutation.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Source and release inspection give high confidence that current main and v2026.6.10 do not expose the requested per-binding Discord identity override.

## Solution Assessment

Unclear as proposed: setting a bot nickname on each response is likely the wrong Discord shape because nicknames are guild-scoped. A webhook persona or explicit self-profile model may be safer, but maintainers need to choose the public contract and security boundary first.
