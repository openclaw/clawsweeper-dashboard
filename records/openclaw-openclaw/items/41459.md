## Summary

Keep open: current main and v2026.6.6 still do not expose a Discord per-binding nickname or ordinary routed-reply identity override, and the closest duplicate was already consolidated into this issue as the canonical product/config decision.

## What This Changes

Request an optional Discord binding identity or nickname override so one bot token can display different agent names per routed channel.

## Best Possible Solution

Have maintainers choose the Discord visible-identity model first, then implement it inside the Discord plugin with schema/docs/tests and explicit security handling for webhook credentials and fallback behavior.

## Reproduction Assessment

Not applicable as a feature request, not a broken existing behavior. Source inspection gives high-confidence absence: the strict binding schema has no nickname field and Discord webhook identity is only wired through thread bindings.

## Solution Assessment

Unclear as proposed. A binding-level nickname is plausible, but the best fix should first compare that shape against existing agent identity and Discord plugin-owned webhook delivery to avoid duplicate config sources.
