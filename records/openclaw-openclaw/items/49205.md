## Summary

Keep open. Current main only updates the OpenClaw transcript/WebChat live path for `chat.send`; I found no Open WebUI visible-history/SQLite mirror, and the related Open WebUI UUID PR explicitly leaves this visual-sync behavior as a separate follow-up.

## What This Changes

The issue asks OpenClaw to mirror Control UI-originated turns into Open WebUI’s visible chat storage when both surfaces share a WebUI-backed UUID chat.

## Best Possible Solution

Define a supported Open WebUI visible-history sync contract, then add focused gateway integration and Docker E2E coverage for Control UI writes into a WebUI-backed UUID chat without relying on an unreviewed ad hoc DB patch.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the current `chat.send` path updates OpenClaw transcript/live WebChat state without any Open WebUI visible-history mirror, matching the reported split between model context and Open WebUI bubbles.

## Solution Assessment

Unclear. The proposed SQLite mirroring sketch may address the symptom, but maintainers should first decide whether OpenClaw should own direct Open WebUI DB writes, an Open WebUI API integration, or a documented unsupported boundary.
