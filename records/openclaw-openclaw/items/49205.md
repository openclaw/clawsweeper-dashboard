## Summary

Keep open. Current main and the latest release still persist Control UI `chat.send` turns into OpenClaw transcript/session state and broadcast to OpenClaw clients, but I found no supported path that mirrors those turns into Open WebUI's own `chat` / `chat_message` visible-history store; the related UUID-session PR explicitly leaves this DB sync out of scope.

## What This Changes

Control UI turns in an Open WebUI-backed chat should appear in Open WebUI visible chat history, not only in OpenClaw transcript/model context.

## Best Possible Solution

Define a maintainer-approved, opt-in Open WebUI visible-history sync and backfill contract, then implement it with focused gateway tests plus Docker Open WebUI proof, or explicitly document that Control UI co-editing Open WebUI history is unsupported.

## Reproduction Assessment

Yes, at source level. Current `chat.send` persists to OpenClaw session/transcript state and broadcasts OpenClaw chat events, while source search found no path that writes the corresponding Open WebUI `chat` / `chat_message` records.

## Solution Assessment

Unclear as proposed. A direct DB mirror is plausible, but the best maintainable fix needs a maintainer-approved external-store ownership boundary, opt-in configuration, schema-version handling, and historical backfill policy.
