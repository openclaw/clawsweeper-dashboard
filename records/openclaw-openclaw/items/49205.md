## Summary

Keep open. Current main still has no supported path that mirrors Control UI-originated `chat.send` turns into Open WebUI's visible chat store, and the related UUID-session PR is open and explicitly leaves this visual-sync behavior out of scope.

## What This Changes

The issue requests that Control UI messages sent into an Open WebUI-backed chat appear both in OpenClaw transcript/model context and in Open WebUI's visible chat history.

## Best Possible Solution

Define an owner-approved, opt-in Open WebUI visual-sync contract for new external turns, preferably through a supported Open WebUI API boundary when available, plus a documented backfill path for already-affected chats.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection plus the reporter's local validation gives a credible path: `chat.send` updates OpenClaw transcript/Gateway subscribers, while no Open WebUI visible-history write path exists.

## Solution Assessment

No, the direct Docker exec/SQLite mirroring diff is useful proof but not the best maintainable solution by itself. A safe implementation needs a supported sync boundary, opt-in configuration, schema/version handling, live proof, and a backfill policy.
