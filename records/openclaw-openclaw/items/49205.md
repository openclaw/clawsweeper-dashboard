## Summary

Keep open. Current main and v2026.6.10 still persist Control UI `chat.send` turns through OpenClaw transcript/session paths and live delivery, but I found no supported runtime path that mirrors those turns into Open WebUI's `chat` or `chat_message` visible-history store; the related open PRs address adjacent session identity/header issues, not this DB sync.

## What This Changes

The issue requests that Control UI turns sent into an Open WebUI-backed chat appear in Open WebUI visible chat history as normal user/assistant messages, not only in OpenClaw transcript/model context.

## Best Possible Solution

Define a maintainer-approved, opt-in Open WebUI visible-history sync and backfill contract, then implement it with focused gateway tests and Docker Open WebUI proof, or explicitly document that Control UI co-editing of Open WebUI history is unsupported.

## Reproduction Assessment

Yes, at source level: current `chat.send` persists to OpenClaw session/transcript state and broadcasts OpenClaw events, while targeted source search found no current gateway path that writes Open WebUI `chat` or `chat_message` rows for Control UI turns. I did not run the live Open WebUI plus Control UI scenario in this read-only pass.

## Solution Assessment

Unclear as proposed. Direct DB mirroring is plausible, but the best maintainable fix needs maintainer agreement on external Open WebUI schema ownership, operator opt-in/configuration, schema-version handling, and historical backfill behavior.
