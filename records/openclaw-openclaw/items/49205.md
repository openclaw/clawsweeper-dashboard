## Summary

Keep this issue open. Current main still supports Open WebUI as an OpenAI-compatible frontend and writes Control UI sends into OpenClaw transcript/broadcast paths, but I found no implementation that mirrors external Control UI turns into Open WebUI's own visible-history store.

## What This Changes

The issue requests that external Control UI turns in Open WebUI-backed chats be mirrored into Open WebUI visible chat history, not only into OpenClaw transcript/model context.

## Best Possible Solution

Add a supported post-turn sync path for successful external `chat.send` into WebUI-backed UUID sessions, using client id to skip only native `webchat-ui` traffic while allowing `openclaw-control-ui`. If maintainers do not want core writing Open WebUI storage directly, define a small plugin/API hook for external visible-history sync and cover both transcript context and Open WebUI visible history in regression tests.

## Reproduction Assessment

Yes. The issue provides concrete self-hosted Open WebUI + Control UI steps, and source inspection confirms the current path updates OpenClaw transcript/history without any Open WebUI visible-history mirror.

## Solution Assessment

Unclear. The reporter's DB-mirror direction matches the observed gap, but the best maintainable solution needs maintainer judgment on direct Open WebUI DB writes versus a plugin/API seam with a narrower ownership boundary.
