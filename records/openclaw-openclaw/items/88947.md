## Summary

This PR is a useful partial mitigation, but it is not merge-ready because it only adds fallback delivery in `runAgentTurnWithFallback` while the linked async/queued follow-up compaction path still does not route `notifyUser` notices. It also has only mocked test proof for a user-visible message-delivery change.

## What This Changes

The PR adds a `routeReply` fallback for compaction notices when `agents.defaults.compaction.notifyUser` is enabled and no block-reply callback is available.

## Best Possible Solution

Use one compaction-notice delivery contract for both active turns and queued/proactive follow-up runs, then prove it with focused tests plus redacted live channel evidence before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main returns from `sendCompactionNotice` when `onBlockReply` is absent, while docs say `notifyUser` sends notices. The queued follow-up path is also visible in source and does not apply the same notice contract.

## Solution Assessment

No, not yet. The proposed fallback is plausible for direct `runAgentTurnWithFallback` calls, but it is too narrow for the linked async/queued follow-up path that already has origin routing for final payloads.
