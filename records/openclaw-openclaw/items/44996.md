Full review comments:

- **[P1] Seed the resolved NetBird origin:** `src/config/gateway-control-ui-origins.ts:35-38`
  - body: For `gateway.bind="netbird"`, this only changes seeded localhost origins to HTTPS. A browser opened at `https://<netbird-ip>:<port>` sends that exact origin, and the Gateway origin check requires an exact allowlist match, so the advertised default Control UI path can still fail with `origin not allowed`. Seed the detected NetBird origin or make explicit `allowedOrigins` a documented requirement for this mode.
  - confidence: 0.89
- **[P2] Load the auto-TLS fingerprint for status probes:** `src/cli/daemon-cli/status.gather.ts:199-200`
  - body: The PR makes default NetBird probes use `wss://`, but the later status-probe TLS runtime path still only loads a fingerprint for explicit `gateway.tls.enabled`. A default NetBird setup uses a generated self-signed cert, so `openclaw daemon status --probe` can fail certificate verification unless it loads the same effective TLS config as the server.
  - confidence: 0.87
- **[P2] Move NetBird links into the shared resolver:** `src/commands/onboard-helpers.ts:459-480`
  - body: Current main resolves dashboard/status/setup Control UI links through `src/gateway/control-ui-links.ts`, while `onboard-helpers.ts` only re-exports that helper. Adding NetBird handling to the older local helper body leaves current link surfaces without NetBird host/TLS behavior after refresh.
  - confidence: 0.86

  - confidence: 0.8
