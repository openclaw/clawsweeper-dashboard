- **[P2] Support NetBird macOS interfaces:** `src/infra/netbird.ts:8-16`
  - body: The detector only accepts interface names starting with `wt`, but NetBird's own docs show macOS using `utun100`/`utun*` while Linux uses `wt0`. Since the PR claims macOS scope, macOS users would silently fall back to loopback instead of binding the VPN IP. ([docs.netbird.io](https://docs.netbird.io/get-started/install/macos?utm_source=openai))
  - confidence: 0.91
- **[P2] Move NetBird TLS into the active connection helper:** `src/gateway/call.ts:152-153`
  - body: The branch changes the older inline `buildGatewayConnectionDetails` body in `call.ts`, but current main delegates local URL construction to `src/gateway/connection-details.ts`. After rebase, local clients would keep using `ws://` for NetBird unless the active helper gets the auto-TLS logic.
  - confidence: 0.9
- **[P2] Pin NetBird auto-TLS for daemon probes:** `src/cli/daemon-cli/status.gather.ts:200-201`
  - body: This switches daemon probe URLs to `wss://` for NetBird, but current main only loads the local TLS runtime/fingerprint for probes when `gateway.tls.enabled` is explicit. Default NetBird probes can reject the generated self-signed certificate unless the active probe fingerprint path is updated too.
  - confidence: 0.86
- **[P2] Update the active gateway bind validator:** `src/cli/gateway-cli/run.ts:218-224`
  - body: The branch adds `netbird` to the old inline bind validation shape, while current main validates through `VALID_BIND_MODES`. A main-native version must update the active set and help text or `openclaw gateway run --bind netbird` and config-driven startup still fail validation.
  - confidence: 0.9
- **[P2] Move the wizard option into active setup:** `src/wizard/onboarding.gateway-config.ts:81`
  - body: Current main presents gateway choices from `src/wizard/setup.gateway-config.ts` with localized labels. Adding the option only to the older onboarding file will not show NetBird in the active setup flow after rebase.
  - confidence: 0.88
- **[P2] Seed the resolved NetBird HTTPS origin:** `src/config/gateway-control-ui-origins.ts:35-39`
  - body: The PR advertises Control UI access at `https://<netbird-ip>:<port>`, but the seeded origins only switch localhost and 127.0.0.1 to HTTPS. Browser Origin matching includes the host, so default NetBird setups can still reject websocket auth unless the resolved NetBird host origin is added.
  - confidence: 0.87
- **[P2] Move Control UI links to the active helper:** `src/commands/onboard-helpers.ts:460-493`
  - body: Current main exports Control UI link resolution from `src/gateway/control-ui-links.ts`; `src/commands/onboard-helpers.ts` only re-exports it. Updating the removed inline helper will not make dashboard, status, or setup links use the NetBird host or HTTPS after rebase.
  - confidence: 0.88
