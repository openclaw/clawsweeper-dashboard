## Summary

Keep this PR open: the NetBird support is not present on current main and the discussion shows real user demand, but the patch still has merge-blocking integration gaps around Control UI origins and current gateway URL helpers.

## What This Changes

The PR adds `gateway.bind: "netbird"` to detect NetBird `wt*` interfaces, auto-enable self-signed TLS, and update gateway, daemon/status, onboarding/config help, tests, and the changelog.

## Best Possible Solution

Port the feature onto current main's gateway connection, Control UI link/origin, pairing/setup, docs, and tests, then have the gateway owners decide whether a NetBird-specific core bind mode is the right public API.

## Reproduction Assessment

Not applicable: this is a feature PR rather than a bug report. Source inspection confirms current main does not include a `netbird` bind value, and the PR is the implementation candidate rather than a current-main regression reproduction.

## Solution Assessment

No. The direction is plausible, but the current patch still fails the default VPN Control UI origin path and needs to be ported to the current shared gateway URL/pairing helpers before it is a maintainable merge candidate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P1] Seed the NetBird origin, not only HTTPS localhost:** `src/config/gateway-control-ui-origins.ts:35-37`
  - body: The netbird mode opens the Control UI at `https://<netbird-ip>:<port>`, but this seeding path only switches the default localhost origins to `https://`. `checkBrowserOrigin` matches the full origin, so the browser's `Origin: https://<netbird-ip>:<port>` will still be rejected in the default setup unless users manually add it. Include the detected NetBird origin or keep the configuration explicit instead of claiming zero-manual-origin setup.
  - confidence: 0.86
- **[P2] Update the current shared URL helpers for netbird:** `src/config/types.gateway.ts:3`
  - body: Adding `netbird` to `GatewayBindMode` makes current setup/pairing code able to receive that value, but main's `resolveGatewayBindUrl` and `resolveScheme` still only know custom/tailnet/lan and explicit TLS. NetBird pair/setup URLs would fall through to the loopback-only error or plaintext scheme instead of `wss://<netbird-ip>`. Add the NetBird picker/scheme path and regression coverage when rebasing this PR.
  - confidence: 0.82
