Full review comments:

- **[P2] Support NetBird macOS interfaces:** `src/infra/netbird.ts:8-16`
  - body: The detector only accepts interface names starting with `wt`, but NetBird's official macOS docs use `utun100`/`utun*`. Since the PR claims macOS scope, macOS users would silently fall back to loopback instead of binding the VPN IP. ([docs.netbird.io](https://docs.netbird.io/get-started/install?utm_source=openai))
  - confidence: 0.91
- **[P2] Move NetBird TLS into the active connection helper:** `src/gateway/call.ts:152-153`
  - body: Current main builds local gateway URLs through `src/gateway/connection-details.ts`, where TLS still depends only on explicit `gateway.tls.enabled`. Updating the older inline helper shape leaves rebased local clients using `ws://` instead of NetBird auto-TLS.
  - confidence: 0.9
- **[P2] Pin NetBird auto-TLS for daemon probes:** `src/cli/daemon-cli/status.gather.ts:198-199`
  - body: The PR switches daemon probes to `wss://` for NetBird, but current main only loads the local TLS fingerprint when `gateway.tls.enabled` is explicit. `openclaw daemon status --probe` can reject the generated self-signed cert unless users manually enable TLS.
  - confidence: 0.86
- **[P2] Update the active gateway bind validator:** `src/cli/gateway-cli/run.ts:217-219`
  - body: Current main validates explicit binds through the active `VALID_BIND_MODES` set around `gateway run`. Updating the older inline validator shape still leaves `openclaw gateway run --bind netbird` and config-driven startup rejected after rebase.
  - confidence: 0.9
- **[P2] Move the wizard option into active setup:** `src/wizard/onboarding.gateway-config.ts:81`
  - body: Current main uses `src/wizard/setup.gateway-config.ts`, `src/wizard/setup.ts`, and localized setup labels for gateway choices. Adding the option only to the older onboarding files will not show NetBird in the active setup flow after rebase.
  - confidence: 0.88
- **[P2] Seed the resolved NetBird HTTPS origin:** `src/config/gateway-control-ui-origins.ts:35-38`
  - body: The PR advertises Control UI access at `https://<netbird-ip>:<port>`, but the origin builder only switches localhost and 127.0.0.1 to HTTPS and never adds the resolved NetBird host. Browser origin checks match scheme and host, so default NetBird setups can still reject websocket auth.
  - confidence: 0.87
- **[P2] Move Control UI links to the active helper:** `src/commands/onboard-helpers.ts:459-493`
  - body: Current main exports Control UI link resolution from `src/gateway/control-ui-links.ts`; `src/commands/onboard-helpers.ts` only re-exports it. Updating the removed inline helper will not make dashboard/status/setup links use the NetBird host or HTTPS after rebase.
  - confidence: 0.88
