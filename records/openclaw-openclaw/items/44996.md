  - body: NetBird's own source uses `wt0` only on non-Darwin builds and `utun100` on Darwin, with the CLI requiring `utun*` names there. Scanning only `wt*` makes `gateway.bind: "netbird"` fall back to loopback on macOS, including the macOS environment listed in the PR body.
  - confidence: 0.93
- **[P2] Port auto-TLS URL selection to the active connection helper:** `src/gateway/call.ts:150`
  - body: Current main builds local gateway URLs in `src/gateway/connection-details.ts`, not in this moved `call.ts` block. After rebase, this leaves local CLI/RPC connections using `ws://127.0.0.1` while the NetBird server path auto-enables TLS.
  - confidence: 0.9
- **[P2] Port the bind validator to the current gateway runner:** `src/cli/gateway-cli/run.ts:216-222`
  - body: Current main validates bind modes through `VALID_BIND_MODES` in the later runner path. Updating the old validator shape does not keep `openclaw gateway run --bind netbird` valid after rebase unless the current Set and help text are updated too.
  - confidence: 0.87
- **[P2] Port the wizard option to the active setup files:** `src/wizard/onboarding.gateway-config.ts:81`
  - body: Current main uses `src/wizard/setup.gateway-config.ts`, `setup.ts`, and `setup.types.ts`; the `onboarding.*` files this branch edits are no longer the active setup surface. After rebase, the NetBird option and types would not be present in the wizard users actually run.
  - confidence: 0.94
- **[P2] Move link resolution into the active helper:** `src/commands/onboard-helpers.ts:459-465`
  - body: Current main re-exports `resolveControlUiLinks` from `src/gateway/control-ui-links.ts`, whose API also carries `tlsEnabled`. Updating the old implementation in `onboard-helpers.ts` will not make setup/dashboard/status links use the NetBird host or HTTPS after rebase.
  - confidence: 0.92
- **[P2] Preserve auto origin seeding:** `src/config/gateway-control-ui-origins.ts:4-7`
  - body: Current main includes `auto` in non-loopback origin seeding for container-default binds. Replacing that union while adding NetBird can stop seeding Control UI origins for container installs, causing startup/origin failures unrelated to NetBird.
  - confidence: 0.88
- **[P2] Seed the advertised NetBird origin:** `src/config/gateway-control-ui-origins.ts:35-38`
  - body: The PR points users at `https://<netbird-ip>:port`, but the default origins only include localhost and 127.0.0.1 with HTTPS. Because `checkBrowserOrigin` exact-matches the browser `Origin`, the default NetBird UI path can still reject unless users manually add `allowedOrigins`.
  - confidence: 0.9
