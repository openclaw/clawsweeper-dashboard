## Summary

Keep open. The linked NetBird support request and follow-up comments show real user demand, and the PR addressed earlier bot feedback, but it is not merge-ready because the default Control UI origin seeding still does not allow the actual NetBird HTTPS URL that the feature is meant to make usable.

## What This Changes

The PR adds a `gateway.bind: "netbird"` mode that detects NetBird `wt*` interfaces, auto-enables self-signed TLS, updates CLI/onboarding/status surfaces, and adds focused tests plus a changelog entry.

## Best Possible Solution

If maintainers want core NetBird support, rebase it onto current main’s centralized gateway helpers, seed or derive the actual HTTPS NetBird origin, cover origin/auth/fingerprint paths with tests, update docs, and attach real behavior proof.

## Reproduction Assessment

Yes for the review finding. Source inspection shows the PR seeds only HTTPS localhost/127 origins for NetBird while current origin checking requires the exact `https://<netbird-ip>:<port>` origin, so the intended browser URL would be rejected without manual config.

## Solution Assessment

No. The direction is plausible, but this implementation needs the actual NetBird HTTPS origin path fixed and a rebase onto current main before it is the narrow maintainable solution.
