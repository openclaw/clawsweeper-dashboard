## Summary

Keep open: current main still routes active native Codex context-engine assemblies through a fixed 24,000-character projection, while the context-engine contract says assembled messages are the model-visible context and Codex runtime docs advertise lifecycle support. The report is source-reproducible and narrow enough for a focused fix rather than cleanup closure.

## What This Changes

The issue asks native Codex context-engine projection to stop silently truncating full-fit context-engine assemblies to a fixed 24,000-character rendered prompt block and to expose projection diagnostics.

## Best Possible Solution

Size Codex context-engine projection from the active context/model budget, preserve the quoted-reference and tool-payload safety behavior, and record projection stats so diagnostics separate assembled frontier, projected context, and provider usage.

## Reproduction Assessment

Yes, source-level: pass a rendered assembly over 24,000 characters through projectContextEngineAssemblyForCodex and the prompt is truncated before Codex sees it. I did not perform the full live Lossless Claw runtime reproduction, but the active Codex call path and current unit test make the cap behavior high-confidence.

## Solution Assessment

Yes, the best fix is a narrow Codex app-server projection change that derives the cap from the active budget and reports truncation metadata. Changing context-engine plugin semantics or the Pi embedded runner would be the wrong boundary.
