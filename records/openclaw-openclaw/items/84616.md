Full review comments:

- **[P1] Guard recovery against stale same-status outcomes:** `src/agents/subagent-registry.ts:364-365`
  - body: The recovery timer only checks `entry.outcome.status` before replaying the captured completion closure. If an earlier failed error completion schedules recovery and a later error terminal state lands before the timer fires, the status still matches, so the old closure can call `completeSubagentRun` with stale `endedAt`, error text, and reason and overwrite the newer entry. Compare the full terminal state or drive cleanup from the current entry instead of replaying captured params.
  - confidence: 0.88
- **[P2] Exercise the actual completion rejection path:** `src/agents/subagent-registry.test.ts:928`
  - body: Mocking `runSubagentAnnounceFlow` does not make `completeSubagentRun` reject, because announce cleanup is started asynchronously and catches that rejection internally. This test can pass without exercising the new `.catch()` or `scheduleLifecycleCatchRecovery` path, so inject a rejection from a path that `completeSubagentRun` actually awaits before cleanup.
  - confidence: 0.82
