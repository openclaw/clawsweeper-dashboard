## Summary

The PR targets a real source-level lifecycle hole, but the current recovery path is still not merge-ready because it can resume announce cleanup directly after a failed completion and skip earlier completion-tail cleanup work. The submitted live proof demonstrates the catch handler in a running gateway, but it does not show registered-run recovery, cleanup completion, or orphan reconciliation after the failure state this PR is meant to handle.

## What This Changes

The PR adds `.catch()` handlers and delayed recovery logic around three fire-and-forget `completeSubagentRun` calls in `src/agents/subagent-registry.ts`, plus regression tests for warning logs and recovery scheduling.

## Best Possible Solution

Keep the catch handlers, but factor and reuse the full post-completion cleanup tail so catch recovery preserves browser cleanup, run-mode bundle MCP retirement, and announce cleanup ordering, then prove it on a registered run.

Yes, at source level: current main has three fire-and-forget `completeSubagentRun` paths without `.catch()`, and `completeSubagentRun` has awaited work after persisting ended state that can reject before cleanup finishes. I did not run a live current-main repro in this read-only review.

## Solution Assessment
