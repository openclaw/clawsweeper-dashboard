## Summary

Keep this PR open: the unhandled lifecycle promise problem is real and the catch-handler direction is useful, but the proposed non-orphan recovery resumes at announce cleanup and can skip the normal completion cleanup tail. The supplied terminal proof shows catch logging for a fake unregistered run, not recovery of a registered run through cleanup or orphan reconciliation.

## What This Changes

The PR adds warning `.catch()` handlers and delayed recovery for three fire-and-forget subagent lifecycle completion paths, plus subagent registry tests for the warning logs.

## Best Possible Solution

Keep the catch handlers, but recover through a retry of `completeSubagentRun` with the original completion params or a shared post-completion cleanup helper that preserves the existing cleanup order.

## Reproduction Assessment

Yes, at source level: current main has two timer-triggered `completeSubagentRun` calls and one lifecycle listener async IIFE without catch handlers. I did not run runtime fault injection because this review is read-only.

## Solution Assessment

No. Catch logging is the right direction, but the recovery path should preserve the normal completion cleanup tail instead of jumping directly to announce cleanup.
