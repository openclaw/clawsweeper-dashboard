Full review comments:

- **[P1] Guard catch recovery against stale terminal state:** `src/agents/subagent-registry.ts:350-355`
  - body: `scheduleLifecycleCatchRecovery` retries the captured completion params whenever the same run remains ended with cleanup pending. If an older failed blocked/timeout completion is followed by a newer terminal event before the 1s timer fires, this retry can overwrite the current outcome/endedReason and send the wrong cleanup message. Match the current terminal state before retrying, or resume cleanup from the current entry instead.
  - confidence: 0.84
- **[P1] Add recovery for aborted lifecycle completions:** `src/agents/subagent-registry.ts:1127-1129`
  - body: The current aborted stop-reason branch still awaits `completeSubagentRun` without branch-aware recovery; the new outer listener catch will only log that rejection. If the killed completion fails after persisting `endedAt`/`endedReason`, the run can remain half-ended with no cleanup flow until a later resume, so this branch needs the same catch recovery as blocked and ok completions.
  - confidence: 0.79
- **[P2] Make the regression reject completeSubagentRun:** `src/agents/subagent-registry.test.ts:925`
  - body: This mock does not make `completeSubagentRun` reject: `startSubagentAnnounceCleanupFlow` launches `runSubagentAnnounceFlow` asynchronously and catches announce failures internally. The test can pass without exercising the new `.catch()` or delayed recovery path, so it should fault-inject an awaited completion dependency and advance through the recovery timer.
  - confidence: 0.91
