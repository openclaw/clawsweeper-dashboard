## Summary

## What This Changes

Adds `.catch()` warning handlers and delayed catch recovery to three subagent lifecycle completion fire-and-forget paths, with subagent-registry tests.

## Best Possible Solution

Keep the catch handlers, but recover by retrying the original completion parameters or sharing a post-completion cleanup helper that preserves browser cleanup, MCP runtime retirement, task/session updates, and announce ordering.

## Reproduction Assessment

Yes, at source level: current main has three fire-and-forget completion paths without rejection handlers, and `completeSubagentRun` can reject from completion hook/runtime setup after persisting ended state. I did not run fault injection in this read-only review.

## Solution Assessment

No. The catch handlers are the right direction, but direct catch recovery through `startSubagentAnnounceCleanupFlow` is not the best fix because it bypasses cleanup work that normal completion performs first.
