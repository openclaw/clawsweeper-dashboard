## Summary

Keep this PR open: it targets a real fire-and-forget rejection path, but the new non-orphan recovery path bypasses part of the normal completion cleanup sequence and the supplied proof does not show a registered run recovering through cleanup or orphan reconciliation.

## What This Changes

The PR adds `.catch()` logging and delayed recovery for three fire-and-forget subagent lifecycle completion paths, plus focused Vitest coverage in `src/agents/subagent-registry.test.ts`.

## Best Possible Solution

Keep the catch handlers, but route recovery through a shared or retried post-completion cleanup tail that preserves browser cleanup, run-mode MCP retirement, and announce cleanup ordering, then prove it on a registered run.

## Reproduction Assessment

Yes, at source level: current main has three fire-and-forget lifecycle completion paths without rejection handlers, and `completeSubagentRun` can do throwing or awaited work after persisting terminal run state. I did not run a live current-main repro because this review is read-only.

## Solution Assessment

No. The `.catch()` handlers are the right direction, but the recovery path should reuse or retry the existing completion cleanup tail instead of starting announce cleanup directly.
