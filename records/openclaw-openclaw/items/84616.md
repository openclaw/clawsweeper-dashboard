## Summary

Keep open: the PR targets a real unhandled-rejection path, but the new catch recovery can bypass normal subagent cleanup ordering and the supplied proof does not show recovery of a registered run.

## What This Changes

The PR adds `.catch()` handlers and delayed recovery around three fire-and-forget subagent lifecycle completion promises, plus warning-log regression tests.

## Best Possible Solution

Keep the catch handlers, but route lifecycle catch recovery through a shared post-completion cleanup tail that preserves browser cleanup, run-mode MCP retirement, and announce cleanup ordering.

## Reproduction Assessment

Yes, at source level: current main has three fire-and-forget lifecycle completion paths without catch handlers, and `completeSubagentRun` performs awaited work after mutating run state. I did not run a live current-main repro because this review is read-only.

## Solution Assessment

No. The catch handlers are the right direction, but the recovery path should reuse the existing completion cleanup tail instead of starting announce cleanup directly.
