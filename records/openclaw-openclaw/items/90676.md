## Summary

Keep open for normal PR review and landing: the patch is a focused cron bug fix, and current source shows the no-channel policy currently blocks the final-assistant-text recovery path for `--no-deliver` runs. I found no blocking correctness or security findings; the main residual gap is that the PR body reports targeted unit proof, not a live no-deliver cron run.

## What This Changes

The PR changes cron channel output policy so unresolved/no-channel cron delivery prefers final assistant text and adds a regression test for the undefined-channel case.

## Best Possible Solution

Land the focused policy change with the new regression test, then close the linked bug after CI or maintainer verification confirms the cron status classification path.

## Reproduction Assessment

Yes, from source inspection: a bare `delivery.mode: "none"` cron run produces `resolvedDelivery.channel: undefined`, current main returns `preferFinalAssistantVisibleText: false`, and the recovery gate requires that flag to be true. I did not run the live cron scenario because this review is read-only.

## Solution Assessment

Yes, this is the best narrow fix: the incorrect default lives in `resolveCronChannelOutputPolicy`, and changing it there lets both final-text selection and tool-warning recovery use the same policy. Plumbed `deliveryPlan.mode` would be narrower but would duplicate state already represented by the absence of a channel and would not improve the downstream fatal-signal and structured-payload guards.
