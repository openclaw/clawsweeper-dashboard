## Summary

Keep open: current main still has the Telegram/Feishu channel fallback gap and this PR is useful, but the current head should not merge as-is because verbose reasoning action updates bypass the serialized draft lane and the branch now includes broader user-facing verbose/retry scope that needs maintainer review.

## What This Changes

The PR adds Telegram/Feishu `reasoningDefault` fallback via a shared SDK helper, expands reasoning with a new `verbose` mode, updates docs/schema/tests/changelog, and changes Telegram draft-stream retry handling.

## Best Possible Solution

Preserve the channel fallback through the shared normalized helper, serialize verbose action updates through the existing draft lane queue, align the thinking docs, and have channel owners decide whether the new verbose/retry behavior belongs in this PR or a follow-up.

## Reproduction Assessment

Yes, source-reproducible. Current main’s Telegram and Feishu helpers ignore configured reasoning defaults, while the core config/docs expose per-agent and global `reasoningDefault` behavior.

## Solution Assessment

No. The shared fallback helper is a good direction, but the current PR also adds a new verbose mode whose Telegram action updates bypass the serialized draft queue and whose docs/scope need maintainer alignment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Serialize verbose reasoning updates through the draft queue:** `extensions/telegram/src/bot-message-dispatch.ts:562-571`
  - body: `onPartialReply` and `onReasoningStream` updates are serialized with `enqueueDraftLaneEvent`, but verbose tool/action lines mutate `reasoningLane.lastPartialText` and call `stream.update` directly. If a tool event arrives while a queued reasoning delta or boundary rotation is pending, the action line can be reordered or overwritten; route these updates through the same draft lane queue.
  - confidence: 0.82
- **[P3] Key verbose tool actions by a stable tool id:** `extensions/telegram/src/bot-message-dispatch.ts:1210-1212`
  - body: The start path stores active actions under `payload.phase` (`start`/`update`), while the end path looks them up by `itemId` or name. That means durations usually never match and concurrent tools overwrite the same `start` entry; thread a stable tool call/item id through the callback or key both sides consistently.
  - confidence: 0.8
- **[P3] Document the global reasoning default fallback:** `docs/tools/thinking.md:108`
  - body: This page still says resolution goes from per-agent default directly to `off`, but the PR helper and config docs resolve per-agent, then `agents.defaults.reasoningDefault`, then `off`. Update the thinking docs so users see the same order the channel fallback now implements.
  - confidence: 0.9
