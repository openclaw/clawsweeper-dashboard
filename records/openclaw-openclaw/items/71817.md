## Summary

Keep this PR open: the current-main Telegram/Feishu gap is real and the shared fallback helper is a useful direction, but the branch now also exposes a new `verbose` reasoning mode without wiring the rest of the public reasoning surfaces consistently.

## What This Changes

The PR adds a plugin SDK helper for resolving per-agent/global `reasoningDefault`, wires Telegram and Feishu preview resolution through it, adds tests/docs/changelog/API-baseline updates, and extends reasoning with a new `verbose` mode.

## Best Possible Solution

Land the shared `reasoningDefault` fallback for Telegram and Feishu, while either splitting `verbose` into a separate fully wired feature or updating every public command, docs, UI, ACP, prompt, and test surface in this PR.

## Reproduction Assessment

Yes. Source inspection on current main shows both Telegram and Feishu reasoning preview resolution ignore configured `reasoningDefault` fallbacks, matching the PR's reported bug.

## Solution Assessment

No, not yet. The shared fallback helper is the maintainable direction, but the branch now exposes `verbose` as a new reasoning mode without finishing or explicitly splitting the public contract work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Finish the verbose public contract:** `src/config/zod-schema.agent-runtime.ts:841`
  - body: Adding `verbose` to the config schema makes it a public `reasoningDefault` value, but the command status/error text, runtime prompt, docs, ACP config option, and Web sessions picker still advertise or expose only `on|off|stream`. Users can configure a mode that the rest of the product cannot consistently discover or select, so please either drop `verbose` from this PR or wire the full public surface.
  - confidence: 0.87
- **[P3] Track verbose tool actions by item id:** `extensions/telegram/src/bot-message-dispatch.ts:1127-1129`
  - body: `onToolStart` stores active actions under `payload.phase`, which is normally `start` or `update`, but the end path looks up `payload.itemId ?? payload.name`. That means tool durations for the new verbose preview never resolve reliably and the active-action map keeps stale entries; key the start/end lifecycle from the item events or pass a stable tool id through both callbacks.
  - confidence: 0.82
