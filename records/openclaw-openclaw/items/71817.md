## Summary

Keep this PR open: it addresses a real current-main Telegram/Feishu reasoningDefault gap, but the current head still bypasses the existing configured-default authorization contract and adds broader verbose/retry behavior that needs owner review before merge.

## What This Changes

The PR adds Telegram and Feishu config fallback for reasoningDefault, introduces a plugin SDK helper, expands reasoning with a new verbose mode, updates docs/schema/tests, and changes Telegram draft-stream retry behavior.

## Best Possible Solution

Preserve the channel fallback fix, but gate configured defaults with the same owner/authorized/operator decision as core, serialize verbose reasoning draft updates, and split or explicitly approve the new verbose/retry scope before merge.

## Reproduction Assessment

Yes, from source inspection. Current main’s Telegram and Feishu helpers fall back to off without reading configured reasoning defaults, while the PR’s new fallback path can be reviewed directly against the existing authorization-gated core resolver.

## Solution Assessment

No. The shared normalized fallback helper is a good direction, but the channel callers need the same authorization gate as core and the new verbose Telegram draft updates need to stay on the existing serialized draft lane.
