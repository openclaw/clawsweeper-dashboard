## Summary

Keep open. Current main still has the Telegram/Feishu reasoning-default gap, but this PR is not ready to merge because the new shared fallback path bypasses the existing authorization contract for configured reasoning visibility, and the new public Plugin SDK subpath is not fully registered in the docs/API metadata surface.

## What This Changes

The PR updates Telegram and Feishu reasoning fallback logic, adds a shared Plugin SDK helper/export, updates focused tests, changelog and API hash metadata, and includes small type/test cleanups.

## Best Possible Solution

Keep the channel behavior fix, but resolve the effective reasoning default through a helper that preserves the core access gate, keeps Feishu stream-only behavior, and either fully documents the new public SDK subpath or keeps the helper on an existing/local surface.

## Reproduction Assessment

Yes. On current main, configure `reasoningDefault`, leave the Telegram or Feishu session without a stored level, and source inspection shows the channels still fall back to off/false; for the PR regression, use a Telegram/Feishu message with `CommandAuthorized: false` and a configured default to show the channel preview path enabling behavior core tests keep off.

## Solution Assessment

No, not yet. The fallback strategy is the right narrow direction, but the current helper needs sender-access gating and the new SDK subpath needs the documented metadata/API contract if it remains public.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate config reasoning defaults by sender access:** `src/plugin-sdk/agent-config-helpers.ts:13-15`
  - body: The helper returns configured `reasoningDefault` without knowing whether the sender may use reasoning state. Core already forces configured reasoning defaults back to `off` for untrusted senders, but these channel previews call the helper before draft/preview suppression, so Telegram/Feishu can enable reasoning preview behavior for messages that core would keep off.
  - confidence: 0.87
- **[P2] Register SDK docs for the new helper subpath:** `scripts/lib/plugin-sdk-entrypoints.json:132`
  - body: Adding `agent-config-helpers` here makes it a public SDK entrypoint, but the branch does not add it to `pluginSdkDocMetadata` or `docs/plugins/sdk-subpaths.md`. The API baseline renderer enumerates `pluginSdkDocMetadata`, so the helper is omitted from the generated SDK API/docs contract unless those artifacts are added or the helper stays on an existing surface.
  - confidence: 0.86

## Security Review

Status: needs_attention
