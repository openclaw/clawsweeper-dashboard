## Summary

Keep open. The PR fixes a real current-main gap, but the latest head still changes Telegram and Feishu store-read failures from fail-closed behavior to config fallback behavior, which can bypass persisted session reasoning overrides despite green exact-head CI.

## What This Changes

The PR adds a Plugin SDK helper for resolving effective `reasoningDefault`, wires Telegram and Feishu to use per-agent/global config defaults, and updates extension tests, SDK docs/exports, API baseline, changelog, plus two adjacent test/type cleanups.

## Best Possible Solution

Preserve the new normalized config fallback only after a successful session-store read with no valid stored level; store-read failures should remain fail-closed for Telegram and Feishu with focused regression tests.

## Reproduction Assessment

Yes. Current main can be checked with Telegram or Feishu configured for `reasoningDefault: "stream"` and no stored reasoning level; the PR regression can be reproduced by mocking `loadSessionStore` to throw while the config default is `stream`.

## Solution Assessment

No, not yet. The shared helper and precedence direction are maintainable, but falling back to config after session-store errors is not the safest implementation of the documented session-override priority.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Fail closed when Feishu session state is unreadable:** `extensions/feishu/src/reasoning-preview.ts:26-30`
  - body: With `reasoningDefault: "stream"`, this catch now falls through to the final `configDefault === "stream"` return. That regresses the stream-only preview guard from #61271 because an unreadable session store can expose reasoning previews even when the persisted session may have `reasoningLevel: "off"`; keep this path returning `false` and use config fallback only after a successful read with no stored level.
  - confidence: 0.88
- **[P2] Preserve Telegram session overrides on store errors:** `extensions/telegram/src/bot-message-dispatch.ts:221`
  - body: After `resolveStorePath` or `loadSessionStore` throws, returning `configDefault` means a session that previously used `/reasoning off` can be treated as `on` or `stream`. The PR promises session-store overrides still take priority, so the error path should remain fail-closed while empty entries can still use the configured fallback.
  - confidence: 0.8
