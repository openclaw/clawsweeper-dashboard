## Summary

Keep open. Current main and the latest release still have the Signal daemon shutdown race, and an open closing PR already owns the focused implementation path.

## What This Changes

The issue reports that Signal channel hot-restart stops signal-cli with fire-and-forget SIGTERM, causing orphaned daemon processes and restart send failures.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal plugin fix that makes daemon stop async and idempotent, waits after SIGTERM, escalates after a bounded timeout, and keeps monitor teardown pending until shutdown completes.

## Reproduction Assessment

Yes at source level: current main still sends SIGTERM and returns from Signal daemon stop, while monitor teardown does not await the daemon exit promise. I did not run a live Signal SIGUSR1 restart in this read-only review.

## Solution Assessment

Yes in direction: a Signal-owned async, idempotent stop awaited through monitor teardown is the narrow maintainable fix. Because https://github.com/openclaw/openclaw/pull/71863 is open with closing syntax, the issue should stay open until that PR or an equivalent fix lands.
