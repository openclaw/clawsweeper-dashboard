## Label Justifications

- `P1`: The issue describes a repeatable bundled Signal restart regression that continues to affect real users and remains present in the latest release.
- `impact:message-loss`: A stale signal-cli process can retain the RPC resources and cause outbound Signal sends to fail.
- `impact:crash-loop`: Orphaned signal-cli processes can block subsequent startup and degrade process-level availability across repeated restarts.

## Summary

Current main and the latest release still return from Signal daemon shutdown immediately after SIGTERM, so the restart race remains necessary to fix; the open closing PR is the canonical implementation candidate, though it now conflicts with main and needs refresh or replacement.

## What This Changes

Make Signal daemon shutdown wait for signal-cli to exit during gateway restarts so stale processes cannot retain the RPC port or config lock.

## Best Possible Solution

Refresh or replace the linked PR while preserving an async, idempotent Signal stop operation, bounded SIGTERM-to-SIGKILL escalation, awaited monitor teardown, and focused daemon/abort regression coverage.

## Reproduction Assessment

Yes, at source level with high confidence: current main sends SIGTERM and returns immediately, while monitor teardown does not await the daemon’s existing exit promise. This review did not run a live linked Signal account reproduction.

## Solution Assessment

Yes in direction: the narrowest maintainable fix is Signal-owned asynchronous shutdown awaited through monitor teardown, with bounded escalation and regression tests. The existing linked PR should be refreshed against current main rather than creating a competing implementation.
