## Summary

Keep open: current main and the latest release still use synchronous fire-and-forget Signal daemon shutdown, while the open closing PR for this issue remains unmerged and currently conflicting.

## What This Changes

Issue reports that Signal daemon restarts can orphan signal-cli processes because stop() sends SIGTERM without waiting for exit before the gateway starts a replacement daemon.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent Signal-owned lifecycle fix with async idempotent stop, bounded SIGKILL escalation, awaited monitor teardown, and regression coverage, then close this issue with commit and release provenance.

## Reproduction Assessment

Yes, at source level. With Signal auto-start enabled, a channels.signal hot reload stops the channel and starts it again while Signal cleanup only sends SIGTERM and returns before signal-cli exits; no live signal-cli run was performed during this read-only review.

## Solution Assessment

Yes. Making Signal daemon stop async and idempotent, waiting on the existing exit promise with bounded SIGKILL escalation, and awaiting monitor teardown is the narrow maintainable fix for this race; the linked PR is the current implementation candidate.
