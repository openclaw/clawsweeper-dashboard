## Summary

Keep open: current main and the latest release still use fire-and-forget Signal daemon shutdown, while the focused closing fix at https://github.com/openclaw/openclaw/pull/71863 is open and unmerged.

## What This Changes

Make Signal daemon shutdown wait for signal-cli to exit during hot-reload restarts so old processes do not hold ports or config locks and break sends.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent Signal-owned lifecycle fix with async idempotent stop, bounded SIGKILL escalation, awaited monitor teardown, and regression coverage, then close this issue with commit and release provenance.

## Reproduction Assessment

Yes, at source level. With Signal autoStart enabled, a channels.signal hot reload aborts the channel and immediately restarts it while Signal cleanup only sends SIGTERM and returns before signal-cli exits.

## Solution Assessment

Yes. Making Signal daemon stop async and idempotent, waiting on the existing exit promise with bounded SIGKILL escalation, and awaiting monitor teardown is the narrow maintainable fix.
