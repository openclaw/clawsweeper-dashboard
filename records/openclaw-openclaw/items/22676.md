## Summary

Current main still has the reported Signal daemon restart race, and the focused closing PR remains open and unmerged, so this issue is still needed as the canonical tracker.

## What This Changes

The issue asks OpenClaw to make Signal daemon hot-restart shutdown await `signal-cli` process exit with bounded SIGTERM/SIGKILL handling so reloads do not leave orphaned daemons or break sends.

## Best Possible Solution

Keep this issue open and review or land https://github.com/openclaw/openclaw/pull/71863, or land an equivalent focused Signal-owned fix that makes daemon stop async/idempotent, waits for exit after SIGTERM, escalates after a bounded timeout, and keeps monitor teardown pending until the child exits.

## Reproduction Assessment

Yes for source reproduction: current main still sends SIGTERM and returns immediately from Signal daemon stop, while hot reload stops then starts the Signal channel. I did not run a live Signal account restart, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes. The best fix belongs in the Signal daemon/monitor lifecycle: make stop awaitable and bounded, then let the existing gateway stop/start sequencing wait through monitor cleanup.
