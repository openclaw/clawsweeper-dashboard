## Summary

Keep open. Current main and the latest release still have the reported Signal daemon shutdown race, and the open linked implementation candidate at https://github.com/openclaw/openclaw/pull/71863 has not merged yet.

## What This Changes

The issue reports that Signal gateway hot restarts stop signal-cli without waiting for process exit, leaving orphaned daemons that can block the next daemon and break sends.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or an equivalent focused Signal-owned fix that makes daemon stop async/idempotent, waits for process exit after SIGTERM, escalates after a bounded timeout, and awaits shutdown through monitor teardown before channel restart.

## Reproduction Assessment

Yes at source level. Current main still sends SIGTERM and returns immediately from Signal daemon stop, and the channel reload path can stop then start Signal without waiting for signal-cli exit; I did not run a live linked Signal account reproduction.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs in the Signal daemon/monitor lifecycle, and the open linked PR already targets that boundary instead of a separate core workaround.
