## Summary

Keep open: current main and the latest release still use the synchronous Signal daemon shutdown path, and the focused closing fix at https://github.com/openclaw/openclaw/pull/71863 is open but unmerged.

## What This Changes

The issue requests that Signal daemon shutdown during config-triggered restarts wait for signal-cli process exit, with bounded termination escalation, before starting a replacement daemon.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent Signal-owned lifecycle fix with regression coverage, then close this issue with fixing commit and release provenance.

## Reproduction Assessment

Yes, at source level. With Signal autoStart enabled, a channels.signal hot reload aborts the channel and immediately restarts it while Signal cleanup only sends SIGTERM and returns before signal-cli exits.

## Solution Assessment

Yes. Making Signal daemon stop async and idempotent, waiting on the existing exit promise with bounded SIGKILL escalation, and awaiting monitor teardown is the narrow maintainable fix.
