## Summary

Keep open: current main and v2026.5.12 still have the synchronous Signal daemon stop race, and the open closing PR is the active fix path rather than a reason to close this issue before merge.

## What This Changes

The issue reports that Signal hot reload sends fire-and-forget SIGTERM to signal-cli and immediately restarts, leaving old daemons holding the HTTP port or config lock.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal plugin lifecycle fix, then close this issue with the fixing commit and release provenance.

## Reproduction Assessment

Yes, source-level. With Signal auto-start enabled, a channels.signal config hot reload stops then starts the channel while Signal daemon cleanup only sends SIGTERM and returns before the child exits.

## Solution Assessment

Yes. Making Signal daemon stop async/idempotent, waiting on the existing exit promise, adding bounded SIGKILL escalation, and awaiting monitor teardown is the narrow maintainable fix for this race.
