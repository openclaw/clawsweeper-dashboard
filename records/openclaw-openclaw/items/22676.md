## Summary

Keep open: current main and v2026.5.19 still have the synchronous Signal daemon shutdown path, and the direct closing fix at https://github.com/openclaw/openclaw/pull/71863 is still open and unmerged.

## What This Changes

The issue asks OpenClaw to wait for signal-cli daemon shutdown during Signal hot reload/restart so old processes do not hold the HTTP port or config lock and cause send failures.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal lifecycle fix with regression coverage, then close this issue with fixing commit and release provenance.

## Reproduction Assessment

Yes, at source level. With Signal autoStart enabled, a channels.signal hot reload aborts the channel and immediately restarts it while Signal cleanup only sends SIGTERM and returns before signal-cli exits.

## Solution Assessment

Yes. Making Signal daemon stop async and idempotent, waiting on process exit with bounded SIGKILL escalation, and awaiting monitor teardown is the narrow maintainable fix; the linked closing PR is the active implementation candidate.
