## Summary

Keep open: current main and the latest release still have synchronous fire-and-forget Signal daemon shutdown, and https://github.com/openclaw/openclaw/pull/71863 remains the open, unmerged closing fix for this exact lifecycle race.

## What This Changes

The issue requests a Signal-owned daemon lifecycle fix so hot reload waits for signal-cli to exit after SIGTERM and escalates if shutdown hangs.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent Signal-owned lifecycle fix with async idempotent stop, bounded SIGKILL escalation, awaited monitor teardown, and regression coverage, then close this issue with commit and release provenance.

## Reproduction Assessment

Yes, at source level. With Signal autoStart enabled, a channels.signal hot reload stops the channel and starts it again while Signal cleanup only sends SIGTERM and returns before signal-cli exits.

## Solution Assessment

Yes. Making Signal daemon stop async and idempotent, waiting on the existing exit promise with bounded SIGKILL escalation, and awaiting monitor teardown is the narrow maintainable fix.
