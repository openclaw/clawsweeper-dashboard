## Summary

Keep open: current main still has the Signal hot-reload race, and https://github.com/openclaw/openclaw/pull/71863 is an open, unmerged closing PR for the same bug.

## What This Changes

The issue requests fixing Signal hot-reload restarts so the auto-started signal-cli daemon waits for shutdown, with bounded escalation, before a new daemon starts.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent Signal-owned lifecycle fix with awaited daemon exit, bounded SIGKILL escalation, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable `channels.signal.autoStart`, trigger a `channels.signal` hot reload, and current main stops then starts the channel while Signal daemon stop only sends SIGTERM and returns.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Signal daemon lifecycle and monitor teardown; broader iMessage/Gmail child-process cleanup should stay separate from this Signal restart bug.
