## Summary

Keep open: current main still has the synchronous Signal daemon shutdown race, and https://github.com/openclaw/openclaw/pull/71863 is the open closing implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Report that Signal daemon shutdown during SIGUSR1/config hot reload sends SIGTERM without waiting, leaving old signal-cli processes that can block restart sends.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal plugin lifecycle fix, then close this issue with fixing commit and release provenance.

## Reproduction Assessment

Yes, source-level. With Signal auto-start enabled, a channels.signal config hot reload stops then starts the channel while Signal daemon cleanup only sends SIGTERM and returns before the child exits.

## Solution Assessment

Yes. Making Signal daemon stop async/idempotent, waiting on the existing exit promise, adding bounded SIGKILL escalation, and awaiting monitor teardown is the narrow maintainable fix for this race.
