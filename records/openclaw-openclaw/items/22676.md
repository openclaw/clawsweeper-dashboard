## Label Justifications

- `P1`: The restart race can repeatedly disable a real Signal channel until the operator manually kills orphaned signal-cli processes.
- `impact:message-loss`: The stale or failed Signal RPC daemon causes outbound send failures after restart.
- `impact:crash-loop`: Orphaned daemon processes retain the HTTP port or config lock and can prevent clean replacement startup.

## Summary

Current main and the latest release still return from Signal shutdown after sending SIGTERM rather than after signal-cli exits, so the reported restart race remains; the explicitly linked fix PR is still open and must be reviewed or refreshed before this issue can close.

## What This Changes

Make Signal daemon shutdown asynchronous, bounded, idempotent, and awaited during restart teardown so old signal-cli processes release their port and config lock before replacement startup.

## Best Possible Solution

Refresh and land the focused Signal-owned repair: reuse one in-flight stop promise, send SIGTERM, await exit, escalate to SIGKILL after a bounded timeout, await cleanup before monitor resolution, and retain focused daemon and abort-path regression coverage.

## Reproduction Assessment

Yes, at source level with high confidence: current main sends SIGTERM and allows monitor teardown to finish without awaiting the daemon’s existing exit promise. This read-only review did not run a live linked-account Signal restart.

## Solution Assessment

Yes. The linked PR’s plugin-local async/idempotent shutdown with bounded escalation and awaited monitor cleanup is narrower and safer than adding gateway-wide delays or generic process policy.
