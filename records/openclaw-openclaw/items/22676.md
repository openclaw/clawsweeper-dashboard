- `impact:message-loss`: The user-visible symptom includes failed Signal sends when the restarted gateway races a stale signal-cli daemon or occupied RPC port.

## Summary

Keep this issue open because current main and the latest release still have the synchronous Signal daemon shutdown path, while the direct closing PR is open and unmerged.

## What This Changes

The issue requests an awaited Signal daemon shutdown with bounded kill escalation so SIGUSR1/config reload restarts do not orphan signal-cli processes or cause sends to fail.

## Best Possible Solution

Keep this issue open and land https://github.com/openclaw/openclaw/pull/71863, or an equivalent focused Signal-owned fix, after maintainer review of the bounded SIGKILL shutdown policy.

## Reproduction Assessment

Yes, source-level: current main still sends SIGTERM and returns immediately from Signal daemon stop, while monitor teardown does not wait for signal-cli exit before the gateway can restart the channel. I did not run a live Signal account restart in this read-only review.

## Solution Assessment

Yes in direction: a Signal-owned async, idempotent stop awaited through monitor teardown is the narrow maintainable fix. The open linked PR is the active candidate rather than a reason to close the issue before merge.
