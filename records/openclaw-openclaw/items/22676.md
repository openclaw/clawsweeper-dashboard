## Label Justifications

- `P1`: The report describes a restart race that can break a real Signal channel workflow and leave daemon processes running.
- `impact:message-loss`: The stale daemon/RPC race can make Signal sends fail after a gateway restart.
- `impact:crash-loop`: The orphaned `signal-cli` process and restart race affect process lifecycle availability.

## Summary

Keep open. Current main and the latest release still have synchronous Signal daemon shutdown, and the linked closing PR is open and unmerged, so this issue remains the active tracker for the restart race.

## What This Changes

The issue reports that Signal daemon restart teardown sends SIGTERM without waiting for signal-cli to exit, allowing orphaned processes and send failures after SIGUSR1/config hot reloads.

## Best Possible Solution

Keep this issue open and review or land https://github.com/openclaw/openclaw/pull/71863, or an equivalent focused Signal fix that waits for daemon exit after SIGTERM, escalates after a bounded timeout, and keeps monitor teardown pending before restart.

## Reproduction Assessment

Yes from source inspection, but not from a live process repro in this pass: current main returns from Signal `stop()` immediately after SIGTERM and monitor cleanup does not await `exited`. Signal config reloads still restart the channel by stopping and then starting it, so the race is source-reproducible.

## Solution Assessment

Yes. The narrow maintainable solution is to fix Signal daemon lifecycle ownership by making stop async/idempotent, waiting for child exit, escalating after a bounded timeout, and awaiting that shutdown through monitor teardown.
