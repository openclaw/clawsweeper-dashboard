## Summary

Keep open. Current main and v2026.5.3 still stop the auto-started Signal daemon by sending SIGTERM without awaiting process exit, and #71863 is an open unmerged closing PR for this same race.

## What This Changes

The issue asks OpenClaw to make Signal daemon shutdown awaited and bounded during config-triggered channel restarts so old signal-cli processes release their port and config lock before a new daemon starts.

## Best Possible Solution

The Signal plugin should own an async, idempotent daemon shutdown contract with awaited SIGTERM exit, bounded SIGKILL escalation, and regression coverage so channel hot reload cannot start a new daemon before the old process exits.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable Signal auto-start, trigger a `channels.signal` hot reload, and current main stops then starts the channel while daemon stop only sends SIGTERM and returns.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Signal daemon lifecycle and monitor teardown; reviewing/landing #71863 or an equivalent Signal-owned patch is the right path.
