## Label Justifications

- `P1`: The issue describes a repeatable Signal restart race that can break a real channel workflow and leave daemon processes orphaned during normal gateway hot reloads.
- `impact:message-loss`: The stale or unavailable Signal daemon can cause outbound sends to fail after restart.
- `impact:crash-loop`: The report is about process-level availability: orphaned signal-cli daemons, held ports/config locks, and restart recovery requiring manual process cleanup.

## Summary

Keep open. Current main and the latest release still use fire-and-forget Signal daemon shutdown, while the direct closing PR is open and unmerged.

## What This Changes

The issue reports that Signal gateway hot reload can restart signal-cli before the old daemon exits, causing orphaned processes, port/config-lock races, and failed sends.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or an equivalent focused Signal-owned fix that makes daemon shutdown async/idempotent, awaits child exit with bounded SIGKILL escalation, and keeps monitor teardown pending until shutdown finishes.

## Reproduction Assessment

Yes, source-level: current main sends SIGTERM from Signal daemon stop without waiting, and Signal config hot reload stops then starts the channel. I did not run a live linked Signal daemon reproduction in this read-only review.

## Solution Assessment

Yes, the best fix belongs in the Signal-owned daemon lifecycle and monitor teardown, not as a gateway delay. Broader iMessage/Gmail child-process cleanup from the older report should remain separate unless maintainers choose a wider follow-up.
