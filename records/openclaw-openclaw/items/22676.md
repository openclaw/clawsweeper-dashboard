## Label Justifications

- `P1`: The issue describes a broken Signal channel restart workflow that can leave orphaned daemon processes and cause send failures for real users.
- `impact:message-loss`: The reported race can make Signal sends fail after restart because the new daemon collides with the old process or stale RPC endpoint.
- `impact:crash-loop`: The issue is about process-level availability: orphaned signal-cli daemons, held ports/locks, and recovery requiring manual process cleanup.

## Summary

Keep open. Current main and the latest release still have the reported Signal daemon restart race, while the linked closing PR remains open and unmerged.

## What This Changes

The issue requests async Signal daemon shutdown with bounded SIGKILL escalation so hot reloads do not leave orphaned signal-cli processes or race message sends.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal plugin fix that makes daemon stop async/idempotent, waits for exit after SIGTERM, escalates after a bounded timeout, and keeps monitor teardown pending until shutdown completes.

## Reproduction Assessment

Yes from source inspection: current Signal daemon stop still sends SIGTERM and returns synchronously, while monitor teardown and Gateway channel restart can proceed without waiting for signal-cli exit. I did not run a live Signal daemon repro in this read-only review.

## Solution Assessment

Yes. The narrow Signal-owned lifecycle repair in the linked open PR is the best path: make stop await process exit with bounded escalation and add regression coverage, without adding config or broader restart policy.
