## Label Justifications

- `P1`: The issue describes a user-facing Signal restart failure that can leave orphaned daemon processes and break message sends for configured users.
- `impact:message-loss`: The reported race can make Signal sends fail after hot restart because RPC may hit a stale or non-started daemon.
- `impact:crash-loop`: The failure mode leaves unmanaged signal-cli processes and can make gateway restart/recovery paths repeatedly fail until manual cleanup.

## Summary

Keep open. Current main and the latest release still have the Signal daemon stop race, and the open closing pull request is the active implementation path rather than a reason to close this issue early.

## What This Changes

The issue requests fixing Signal hot-restart shutdown so OpenClaw waits for signal-cli to exit, with bounded escalation, before starting a replacement daemon.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or an equivalent focused Signal-owned fix that makes daemon stop async/idempotent, waits for exit after SIGTERM, escalates after a bounded timeout, and awaits shutdown through monitor teardown.

## Reproduction Assessment

Yes at source level, but not from a live linked Signal account in this review. Current main still returns from Signal daemon stop immediately after SIGTERM, while hot reload awaits channel stop and starts the channel again without Signal proving the child process exited.

## Solution Assessment

Yes, the linked fix direction is the narrow maintainable solution. The right boundary is the Signal daemon and monitor lifecycle, not a new gateway delay or a separate workaround, and the existing open PR already implements that path.
