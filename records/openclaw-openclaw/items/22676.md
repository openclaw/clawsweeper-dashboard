## Summary

Keep open: current main and the latest release still use fire-and-forget Signal daemon shutdown, and https://github.com/openclaw/openclaw/pull/71863 remains an open, unmerged closing PR for this exact race.

## What This Changes

The issue requests making Signal daemon shutdown async so SIGUSR1/config hot reload waits for signal-cli to exit, with bounded SIGKILL escalation, before starting a replacement daemon.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal plugin fix, then close this issue with the fixing commit and release provenance.

## Reproduction Assessment

Yes, source-level. Enable Signal auto-start and trigger a channels.signal hot reload/SIGUSR1 restart; current main stops then starts the channel while Signal daemon cleanup only sends SIGTERM and returns before signal-cli exits.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Signal daemon lifecycle and monitor teardown; broader iMessage or Gmail child-process cleanup should stay separate unless independently reproduced.
