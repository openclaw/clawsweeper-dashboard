## Summary

Keep open: current main and the latest release still have the synchronous Signal daemon stop path, and the open closing PR is the active implementation candidate rather than a reason to close early.

## What This Changes

The issue asks OpenClaw to make Signal daemon shutdown on restart wait for signal-cli exit so restarts do not leave orphaned processes or break sends.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or an equivalent Signal-owned fix that makes daemon stop async/idempotent, waits after SIGTERM, escalates after a bounded timeout, and awaits shutdown through monitor teardown before restart.

## Reproduction Assessment

Yes at source level: current main still returns from Signal daemon stop immediately after SIGTERM, while hot reload restarts the channel by stopping and then starting it. I did not run a live linked Signal account restart in this read-only review.

## Solution Assessment

Yes in direction: a Signal-owned async, idempotent daemon stop awaited through monitor teardown is the narrow maintainable fix. The active closing PR implements that shape and should be reviewed rather than replaced by a new ClawSweeper fix PR.
