## Summary

Keep open: current main still has the synchronous Signal daemon stop path, v2026.5.12 still ships that behavior, and the direct closing PR is open but unmerged.

## What This Changes

The issue reports that Signal hot reload can restart signal-cli before the previous daemon exits, leaving orphaned processes and failed sends.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal lifecycle fix with regression coverage, then close this issue with the fixing commit and release provenance.

## Reproduction Assessment

Yes, at source level. With Signal autoStart enabled, a channels.signal hot reload aborts the channel and immediately restarts it while Signal cleanup only sends SIGTERM and returns before signal-cli exits.

## Solution Assessment

Yes. Making Signal daemon stop async/idempotent, waiting on the existing exit promise with bounded SIGKILL escalation, and awaiting that shutdown through monitor teardown is the narrow maintainable fix.
