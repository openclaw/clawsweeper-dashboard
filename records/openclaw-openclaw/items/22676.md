## Label Justifications

- `P1`: The source-visible race repeatedly breaks an active Signal channel during configuration-driven restarts and has multiple user confirmations.
- `impact:message-loss`: The replacement daemon can fail to bind or send while the prior signal-cli process still owns its port or config lock.
- `impact:crash-loop`: Repeated restart attempts can accumulate orphaned daemon processes and leave the Signal channel unavailable until manual recovery.

## Summary

Keep open: current main and v2026.6.5 still allow Signal monitor teardown to finish after only sending SIGTERM, while the open linked fix PR remains unmerged and needs conflict resolution before landing.

## What This Changes

Fix Signal hot-reload shutdown so channel restart waits for signal-cli to exit and uses bounded forced termination when necessary.

## Best Possible Solution

Refresh and land the linked Signal-owned fix with async idempotent shutdown, bounded SIGKILL escalation, and regression coverage proving monitor teardown cannot resolve before daemon exit.

## Reproduction Assessment

Yes at source level: hot reload awaits the Signal channel task, but that task can finish after merely requesting SIGTERM because daemon exit is not awaited. The issue discussion also contains repeated real-user confirmations, though this review did not run a live linked Signal account reproduction.

## Solution Assessment

Yes in direction: fixing shutdown inside the Signal plugin and propagating the promise through monitor teardown is the narrowest owner-correct solution. The linked PR follows that design but must be refreshed against current main before landing.
