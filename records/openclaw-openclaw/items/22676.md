## Summary

Keep open. Current main still has the synchronous Signal daemon shutdown path that can race hot restart, and open PR #71863 is the active unmerged implementation candidate that explicitly closes this issue.

## What This Changes

The issue requests async Signal daemon shutdown during hot restart so the old signal-cli process exits or is killed before a replacement instance starts.

## Best Possible Solution

The desired end state is a focused Signal lifecycle change that awaits process exit after SIGTERM, escalates after a bounded timeout, and keeps channel teardown pending until daemon shutdown completes.

## Reproduction Assessment

Yes, source-level. With Signal autoStart enabled, a `channels.signal` hot reload can stop the channel and then start it again after Signal cleanup returns from a SIGTERM-only daemon stop, before signal-cli has necessarily exited.

## Solution Assessment

Yes. A focused Signal-owned lifecycle fix is the narrowest maintainable solution; broader iMessage or Gmail process hardening should stay separate unless those paths have their own concrete restart reproductions.
