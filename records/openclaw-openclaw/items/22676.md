## Label Justifications

- `P1`: The issue reports a reproducible Signal restart race that can orphan signal-cli processes and break a real channel workflow.
- `impact:message-loss`: The reported failure mode includes Signal send failures when a restarted gateway races a stale daemon process.
- `impact:crash-loop`: The reported SIGUSR1/config hot-reload path can leave orphaned daemon processes and require manual process cleanup to recover.

## Summary

This issue should stay open because current main and the latest release still use fire-and-forget Signal daemon shutdown, while the linked closing PR is open and unmerged.

## What This Changes

The issue requests fixing Signal daemon restart shutdown so SIGUSR1/config hot reload waits for signal-cli to exit and avoids orphaned processes and send failures.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal fix that makes daemon stop async/idempotent, waits for exit after SIGTERM, escalates after a bounded timeout, and awaits cleanup through monitor teardown before restart.

## Reproduction Assessment

Yes from source inspection: current main still sends SIGTERM without waiting in Signal daemon stop, while Signal config hot reload stops and starts the channel. I did not run a live Signal daemon reproduction in this read-only review.

## Solution Assessment

Yes at the issue level: a Signal-owned async/idempotent daemon shutdown that is awaited through monitor teardown is the narrow maintainable fix. The open linked PR is the active implementation candidate to review rather than opening another fix branch.
