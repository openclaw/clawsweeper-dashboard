## Label Justifications

- `P1`: The issue affects a real Signal channel restart workflow and can leave orphaned daemons that break message sends after hot reload.
- `impact:message-loss`: The reported race can make Signal sends fail when the replacement daemon races the old process still holding the HTTP port or config lock.
- `impact:crash-loop`: The issue is about orphaned `signal-cli` processes and restart recovery failure after gateway reload.

## Summary

Keep open. Current `main` and the latest release still have the reported Signal daemon restart race, and https://github.com/openclaw/openclaw/pull/71863 is an open, unmerged implementation candidate that explicitly closes this issue.

## What This Changes

The issue asks OpenClaw to make Signal daemon shutdown during hot restart wait for `signal-cli` process exit, with bounded SIGKILL escalation, before starting a replacement daemon.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or land an equivalent focused Signal-owned fix that makes daemon stop async/idempotent, waits for process exit after SIGTERM, escalates after a bounded timeout, and awaits monitor teardown before restart.

## Reproduction Assessment

Yes, at source level: current main still sends SIGTERM and returns before `signal-cli` exits, while Signal config hot reload restarts through gateway stop/start. I did not run a live linked Signal account restart in this read-only review.

## Solution Assessment

Yes in direction. A Signal-owned async, idempotent daemon stop awaited through monitor teardown is the narrow maintainable fix; the linked PR is the active path rather than a reason to close the issue early.
