## Summary

Keep open. Current main still has the reported Signal daemon restart race: Signal shutdown sends SIGTERM without awaiting process exit or SIGKILL escalation, and an open unmerged PR explicitly closes this issue, so the issue should remain open until that PR or an equivalent focused fix lands.

## What This Changes

Signal daemon shutdown during hot-reload restarts should wait for signal-cli to exit and escalate from SIGTERM to SIGKILL before starting a replacement daemon.

## Best Possible Solution

Land PR #71863 or an equivalent Signal-owned fix that makes daemon stop async and idempotent, waits for signal-cli exit after SIGTERM, escalates with a bounded SIGKILL fallback, and keeps monitor teardown open until shutdown completes.

## Reproduction Assessment

Yes. The source-level path is high confidence: enable Signal autoStart, trigger a `channels.signal` config hot reload, and the gateway can start a replacement channel after monitor cleanup calls a synchronous SIGTERM-only daemon stop rather than waiting for signal-cli exit.

## Solution Assessment

Yes. A focused Signal lifecycle fix is the narrowest maintainable solution for this issue; broader iMessage/Gmail process-hardening can remain separate unless those paths have their own concrete restart reproduction.
