## Summary

Keep open: current main and the latest release still use fire-and-forget Signal daemon shutdown, while the linked closing fix remains open and unmerged.

## What This Changes

The issue requests awaiting Signal daemon process exit during SIGUSR1/config hot reload, with bounded SIGKILL escalation, to avoid orphaned signal-cli processes and send failures.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal plugin fix that makes daemon shutdown async/idempotent, awaits exit, escalates after a bounded timeout, and adds daemon/monitor regression coverage.

## Reproduction Assessment

Yes, source-level. Enable Signal auto-start and trigger a channels.signal hot reload; current main stops then starts the channel while Signal daemon cleanup only sends SIGTERM and returns.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Signal daemon lifecycle and monitor teardown; broader child-process cleanup for iMessage or Gmail should remain separate unless independently reproduced.
