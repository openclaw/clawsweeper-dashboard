## Label Justifications

- `P1`: The issue affects a real Signal channel restart workflow by orphaning signal-cli processes and causing send failures after config/SIGUSR1 restarts.
- `impact:message-loss`: The reported race can make Signal sends fail after restart because the replacement daemon races a stale process still holding the port or config lock.
- `impact:crash-loop`: The issue is about process-level availability: orphaned signal-cli daemons accumulate and gateway restart/recovery can remain unhealthy until manual cleanup.

## Summary

Keep open. Current main and the latest release still use fire-and-forget Signal daemon shutdown, while https://github.com/openclaw/openclaw/pull/71863 is an open, mergeable, proof-supplied implementation candidate that explicitly closes this issue.

## What This Changes

The issue asks for Signal daemon shutdown to wait for signal-cli exit, with bounded SIGKILL escalation, before config/SIGUSR1 restarts start a replacement daemon.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or an equivalent Signal-owned fix that makes daemon shutdown async, idempotent, bounded, and awaited through monitor teardown before channel restart.

## Reproduction Assessment

Yes, source-level: current main sends SIGTERM from Signal daemon stop without waiting for child exit, and hot reload stops then starts the Signal channel. I did not run a live linked Signal account reproduction in this read-only review.

## Solution Assessment

Yes. The best fix is the narrow Signal-owned lifecycle change already proposed in https://github.com/openclaw/openclaw/pull/71863, not a gateway-wide workaround or a new configuration option.
