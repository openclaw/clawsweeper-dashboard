## Summary

Keep open. Current main still has the Signal daemon restart race: Signal shutdown sends SIGTERM without waiting for process exit or SIGKILL escalation, and open PR #71863 explicitly closes this issue, so the issue should remain open until that PR or an equivalent focused fix lands.

## What This Changes

The issue reports that Signal hot-reload restarts can spawn a new signal-cli before the old daemon exits because daemon stop sends SIGTERM without waiting.

## Best Possible Solution

Land PR #71863 or an equivalent Signal-owned fix that makes daemon stop async and idempotent, waits for signal-cli exit after SIGTERM, escalates with a bounded SIGKILL fallback, and keeps monitor teardown open until shutdown completes.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: enable Signal autoStart, change `channels.signal` through config reload, and the gateway can start a replacement Signal channel after monitor cleanup calls a synchronous SIGTERM-only daemon stop rather than waiting for signal-cli exit.

## Solution Assessment

Yes. The focused Signal lifecycle fix proposed by the paired PR is the narrowest maintainable solution for this issue; broader iMessage/Gmail child-process hardening should remain separate unless maintainers intentionally widen the scope.
