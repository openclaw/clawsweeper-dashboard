## Label Justifications

- `P1`: The issue affects a real Signal channel restart workflow by orphaning `signal-cli` processes and causing send failures after config or SIGUSR1 restarts.
- `impact:message-loss`: The reported race can make Signal sends fail after restart because the replacement daemon races a stale process still holding the port or config lock.
- `impact:crash-loop`: The issue is about process-level availability: orphaned `signal-cli` daemons can accumulate and require manual cleanup before recovery.

## Summary

Keep open. Current main and the latest release still use fire-and-forget Signal daemon shutdown, and https://github.com/openclaw/openclaw/pull/71863 is the open, clean, proof-supplied implementation candidate that explicitly closes this issue.

## What This Changes

The issue requests fixing Signal daemon restart shutdown so `stop()` waits for `signal-cli` exit and escalates before the channel is restarted.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or an equivalent Signal-owned fix that makes daemon shutdown async, idempotent, bounded, and awaited through monitor teardown before channel restart.

## Reproduction Assessment

Yes, at source level. Current main sends `SIGTERM` from Signal daemon stop without waiting for child exit, and Signal hot reload stops then immediately starts the channel; I did not run a live linked Signal account restart.

## Solution Assessment

Yes. The narrow Signal-owned lifecycle change in https://github.com/openclaw/openclaw/pull/71863 is the best path; a gateway-wide delay or new config option would be a weaker workaround.
