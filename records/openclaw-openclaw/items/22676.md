## Label Justifications

- `P1`: The issue reports a restart-path lifecycle race that can leave Signal unusable until manual process cleanup.
- `impact:message-loss`: The issue reports Signal send failures after the stale daemon holds the RPC port or config lock.
- `impact:crash-loop`: The issue reports orphaned daemon processes and restart failures around gateway reloads.

## Summary

Keep open. Current main and the latest release still use fire-and-forget Signal daemon shutdown, and https://github.com/openclaw/openclaw/pull/71863 is the open, clean, closing implementation candidate for the same Signal lifecycle bug.

## What This Changes

The issue asks OpenClaw to wait for Signal daemon shutdown during config/SIGUSR1 restarts so old `signal-cli` processes cannot race the replacement process.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863 or an equivalent Signal-owned fix that makes daemon stop async/idempotent and keeps monitor teardown open until the child exits or is killed.

## Reproduction Assessment

Yes at source level: current main still returns from Signal daemon stop immediately after SIGTERM instead of awaiting the child exit promise. I did not run a live Signal daemon reproduction in this read-only review.

## Solution Assessment

Yes, the best fix belongs in the Signal plugin daemon lifecycle and monitor teardown path; generic gateway reload already awaits channel stop, so the Signal task must represent real process shutdown.
