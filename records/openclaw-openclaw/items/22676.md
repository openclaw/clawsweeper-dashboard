## Label Justifications

- `P1`: The issue reports a repeatable Signal channel restart failure that can orphan daemon processes and break message sending for affected users.
- `impact:message-loss`: The reported race causes Signal send failures when the restarted gateway hits a stale daemon or an unready replacement.

## Summary

Keep open: current main and the latest release still have the Signal daemon shutdown race, and the live linked closing PR is open, mergeable, and owns the focused fix path.

## What This Changes

Report that Signal hot reload can orphan the old `signal-cli` daemon because shutdown sends SIGTERM without awaiting process exit before restart.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or an equivalent Signal-owned fix that makes daemon stop async/idempotent, awaits exit through monitor teardown, and keeps focused regression coverage.

## Reproduction Assessment

Yes at source level: current main still sends SIGTERM without awaiting `signal-cli` process exit while the gateway hot-reload path restarts the Signal channel after `stopChannel` resolves. I did not run a live linked-Signal SIGUSR1 restart in this read-only review.

## Solution Assessment

Yes: a Signal-owned async/idempotent daemon shutdown with bounded SIGKILL escalation is the narrowest maintainable fix because the gateway already awaits channel task shutdown and the race is inside Signal monitor cleanup.
