## Summary

Keep open. Current main and the latest release still have the Signal daemon fire-and-forget stop path, and the live linked closing PR is open but unmerged, so the user-visible restart race is still tracked here.

## What This Changes

Report and fix the Signal hot-reload race where OpenClaw can restart before the old signal-cli daemon exits, leaving orphaned processes and send failures.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or an equivalent Signal-owned async/idempotent daemon shutdown fix with bounded SIGKILL escalation and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still sends SIGTERM and returns before signal-cli process exit, while Signal config reload still stops and starts the channel back-to-back; I did not run a live linked Signal account restart.

## Solution Assessment

Yes in direction. A Signal-owned async, idempotent stop awaited through monitor teardown is the narrow maintainable fix, and the linked PR is already the focused implementation candidate.
