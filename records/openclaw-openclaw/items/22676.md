## Label Justifications

## Summary

Keep open: current main and the latest release still return from Signal daemon stop before signal-cli exits, and the open linked pull request is the active fix path.

## What This Changes

The issue reports a Signal hot-reload restart race where a new signal-cli daemon can start before the old daemon releases its port or config lock, causing orphaned processes and failed sends.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or land an equivalent Signal-owned async, idempotent shutdown fix that waits for daemon exit, applies bounded SIGKILL escalation, and keeps focused regression coverage.

## Reproduction Assessment

Yes at source level: current main still sends SIGTERM and returns immediately from Signal daemon stop, while monitor teardown does not await daemon exit. I did not run a live linked Signal restart in this read-only review.

## Solution Assessment

Yes in direction: the narrow maintainable fix is to make Signal daemon stop async/idempotent and await it through monitor teardown. The open linked PR is the active implementation candidate rather than a reason to queue duplicate work.
