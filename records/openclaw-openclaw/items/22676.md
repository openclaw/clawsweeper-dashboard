## Summary

Keep open. Current main still has the Signal restart race in source, and open PR #71863 is the active unmerged closing fix, so this issue should remain open until that PR or an equivalent Signal-owned fix lands.

## What This Changes

The issue reports that Signal hot-restart sends SIGTERM to signal-cli without awaiting process exit, causing orphaned processes and restart send failures.

## Best Possible Solution

Review and land PR #71863, or land an equivalent focused Signal fix that makes daemon shutdown async/idempotent, waits for exit after SIGTERM, escalates with a bounded SIGKILL timeout, and awaits cleanup through monitor teardown before restart.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure Signal with auto-start, trigger a `channels.signal` hot reload, and the gateway will stop then start the channel while Signal's daemon stop only sends SIGTERM and returns. I did not execute the live Signal/macOS process test in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Signal daemon lifecycle and monitor teardown; the existing open closing PR is the right path to review rather than closing the issue early.
