## Summary

Keep open. Current main still has the reported Signal hot-restart race, and open PR #71863 is the active unmerged implementation candidate that explicitly closes this issue.

## What This Changes

The issue asks for Signal daemon shutdown to wait for signal-cli exit during hot restart so stale processes do not hold the HTTP port or config lock.

## Best Possible Solution

Land PR #71863 or an equivalent focused Signal-owned fix that makes daemon stop async and idempotent, waits for exit after SIGTERM, escalates with bounded SIGKILL, and keeps monitor teardown open until shutdown completes.

## Reproduction Assessment

Yes, source-level. Enable Signal autoStart, trigger a hot reload for `channels.signal`, and the gateway can start the replacement channel after Signal cleanup returns from a SIGTERM-only daemon stop rather than waiting for `signal-cli` exit.

## Solution Assessment

Yes. A focused Signal lifecycle fix is the narrowest maintainable solution here; broader iMessage/Gmail process hardening should stay separate unless those paths have their own concrete restart reproductions.
