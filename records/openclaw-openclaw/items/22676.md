## Label Justifications

- `P1`: The issue affects a real Signal channel restart workflow and can leave orphaned signal-cli processes plus send failures after config/SIGUSR1 restarts.
- `impact:message-loss`: The reported race causes Signal sends to fail when the restarted gateway talks to a stale or blocked daemon.
- `impact:crash-loop`: The issue is about process-level availability: orphaned signal-cli daemons, held ports/locks, and restart recovery requiring manual process cleanup.

## Summary

Keep open: current main and the latest release still return from Signal daemon shutdown immediately after SIGTERM, and the open linked PR is the active fix path rather than a reason to close this issue early.

## What This Changes

The issue reports that Signal daemon shutdown returns during config/SIGUSR1 restarts before signal-cli exits, leaving old processes holding ports or config locks and causing send failures.

## Best Possible Solution

Make the Signal-owned daemon lifecycle async and idempotent, wait for signal-cli exit with bounded SIGKILL escalation, and await that shutdown through monitor teardown before channel restart; https://github.com/openclaw/openclaw/pull/71863 is the current focused path.

## Reproduction Assessment

Yes, source-level: current main has a Signal stop() that sends SIGTERM and returns without waiting, while Signal config hot reload stops and immediately starts the channel. I did not run a live linked Signal account restart in this read-only review.

## Solution Assessment

Yes, the best fix belongs in the Signal plugin daemon lifecycle and monitor teardown rather than a gateway delay workaround. The focused open PR matches that boundary; maintainers still need to review and land it or an equivalent Signal-owned fix.
