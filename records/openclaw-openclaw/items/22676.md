## Label Justifications

- `P1`: The issue describes a broken Signal restart workflow that can leave orphaned signal-cli processes and cause message send failures for real users.
- `impact:message-loss`: The reported race can make Signal sends fail or target a stale daemon after channel restart.
- `impact:crash-loop`: The report centers on orphaned processes and restart/shutdown availability failures around the Signal daemon.

## Summary

Keep open: current main and the latest release still have the Signal daemon stop race, and https://github.com/openclaw/openclaw/pull/71863 is an open, unmerged closing PR for the same bug.

## What This Changes

The issue asks OpenClaw to make Signal daemon shutdown await process exit, escalate after timeout, and complete before restart paths spawn a new signal-cli process.

## Best Possible Solution

Review and land, or deliberately replace, https://github.com/openclaw/openclaw/pull/71863 so Signal daemon shutdown is async, idempotent, bounded, and awaited before channel restart.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main still sends SIGTERM and returns from Signal daemon stop without waiting, while channel reload stops and starts the channel in sequence. I did not run a live linked Signal account restart in this read-only review.

## Solution Assessment

Yes, the best fix shape is a focused Signal-owned lifecycle change that makes daemon stop async/idempotent, waits for exit after SIGTERM, escalates after a bounded timeout, and awaits it during monitor teardown. The broader iMessage/Gmail process-cleanup concern should remain separate unless maintainers intentionally broaden the work.
