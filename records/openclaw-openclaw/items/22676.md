## Summary

Keep open. Current main and the latest release still return from Signal daemon shutdown immediately after SIGTERM, and the open linked pull request https://github.com/openclaw/openclaw/pull/71863 is the active fix path for this issue.

## What This Changes

The issue requests fixing Signal hot-reload shutdown so signal-cli is awaited after SIGTERM, escalated if needed, and not restarted while the old daemon still owns ports or locks.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent Signal-owned async daemon shutdown fix, then close this issue from the merged commit.

## Reproduction Assessment

Yes at source level: current main still returns from Signal daemon stop immediately after SIGTERM while gateway hot reload stops and starts the channel. I did not run a live Signal account SIGUSR1 restart in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is Signal-owned async daemon shutdown with bounded SIGKILL escalation, idempotence, and monitor teardown awaiting it before restart. The open linked pull request is the current implementation candidate for that shape.
