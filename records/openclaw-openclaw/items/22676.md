## Summary

Keep open: current main still has the fire-and-forget Signal daemon shutdown path, and https://github.com/openclaw/openclaw/pull/71863 is an open closing PR rather than a reason to close this issue before merge.

## What This Changes

The issue asks for Signal daemon shutdown during config/SIGUSR1 restarts to wait for signal-cli exit and escalate after a timeout to prevent orphaned processes and send failures.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal plugin lifecycle fix, add regression coverage, then close this issue with fixing commit and release provenance.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: with Signal autoStart enabled, a channels.signal config hot reload stops then starts the channel while Signal daemon cleanup only sends SIGTERM and returns before the child exits.

## Solution Assessment

Yes. Making Signal daemon stop async/idempotent, waiting on the existing exit promise, adding bounded SIGKILL escalation, and awaiting monitor teardown is the narrow maintainable fix for this race.
