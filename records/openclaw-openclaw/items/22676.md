## Label Justifications

- `P1`: The issue describes a source-supported Signal restart race that can leave orphaned daemon processes and break active channel sends.
- `impact:message-loss`: The reported failure mode includes Signal send failures when restart races leave the daemon or RPC target in a bad state.

## Summary

Keep open: current `main` and the latest release still have synchronous fire-and-forget Signal daemon shutdown, and the linked closing fix remains open and unmerged.

## What This Changes

The issue asks for Signal daemon shutdown to wait for `signal-cli` process exit during gateway/channel restarts instead of sending fire-and-forget `SIGTERM`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent Signal-owned lifecycle fix with async idempotent stop, bounded SIGKILL escalation, awaited monitor teardown, and regression coverage, then close this issue with commit and release provenance.

## Reproduction Assessment

Yes, at source level. With Signal auto-start enabled, a `channels.signal` hot reload stops the channel and starts it again while Signal cleanup only sends `SIGTERM` and returns before `signal-cli` exits.

## Solution Assessment

Yes. Making Signal daemon stop async and idempotent, waiting on the existing exit promise with bounded SIGKILL escalation, and awaiting monitor teardown is the narrow maintainable fix for this race.
