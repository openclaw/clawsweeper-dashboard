## Label Justifications

- `P1`: The issue tracks a user-facing Signal restart failure that can leave orphaned daemon processes and break sends after hot reload.
- `impact:message-loss`: The reported race can make Signal RPC/send paths fail after a gateway restart.
- `impact:crash-loop`: The reported race leaves orphaned signal-cli processes and can require manual process cleanup or a full gateway stop/start to recover.

## Summary

Keep open: current main and the latest release still have the synchronous Signal daemon stop path, and https://github.com/openclaw/openclaw/pull/71863 is the open closing PR that should land or be replaced before this issue closes.

## What This Changes

The issue reports that Signal hot restarts send SIGTERM to signal-cli without awaiting process exit, allowing the replacement daemon to race with the old process for its HTTP port and config lock.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or an equivalent focused Signal-owned fix that makes daemon stop async/idempotent, waits for exit after SIGTERM, escalates to SIGKILL after a bounded timeout, and awaits shutdown through monitor teardown.

## Reproduction Assessment

Yes, at source level: current main still sends SIGTERM and returns immediately from Signal daemon stop while gateway reload restarts Signal through stop/start. I did not run a live linked signal-cli restart in this read-only review.

## Solution Assessment

Yes in direction: a Signal-owned async, idempotent stop awaited through monitor teardown is the narrow maintainable fix. The active linked PR is the right implementation path, so this issue should stay open until that lands or is replaced.
