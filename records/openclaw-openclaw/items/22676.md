## Summary

Keep open: current main still has the Signal daemon shutdown race, and the open linked fix PR is the active implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests awaiting Signal daemon shutdown during gateway restarts so old signal-cli processes release their port/config lock before a new daemon starts.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal plugin lifecycle fix, then close this issue with the fixing commit and release provenance.

## Reproduction Assessment

Yes, source-level. Enable Signal auto-start and trigger a channels.signal hot reload; current main stops then starts the channel while Signal daemon cleanup only sends SIGTERM and returns before the child process exits.

## Solution Assessment

Yes. Making Signal daemon stop async/idempotent, waiting on the existing exit promise, bounding SIGKILL escalation, and awaiting monitor teardown is the narrow maintainable fix for this race.
