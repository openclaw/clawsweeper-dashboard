## Label Justifications

- `P1`: The issue reports a real restart race that can break Signal channel delivery and leave orphaned processes during routine config-triggered restarts.
- `impact:message-loss`: The reported failure mode includes Signal send failures after the restart races against the old daemon.
- `impact:crash-loop`: The report is about process lifecycle availability: orphaned signal-cli daemons, held ports/config locks, and failed restarts.

## Summary

Keep open. Current main and the latest release still use synchronous SIGTERM-only Signal daemon shutdown, and https://github.com/openclaw/openclaw/pull/71863 is the open, mergeable closing PR for the same lifecycle bug.

## What This Changes

Fix the Signal auto-start daemon shutdown race where config-triggered restarts can start a new signal-cli process before the old one exits.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or an equivalent focused Signal-owned fix that makes daemon stop async/idempotent, waits for exit after SIGTERM, escalates after a bounded timeout, and awaits monitor shutdown before restart.

## Reproduction Assessment

Yes. The current main source shows the Signal daemon stop path sends SIGTERM without awaiting child exit, while hot reload can stop and immediately restart the Signal channel; I did not run a live Signal account restart in this read-only sweep.

## Solution Assessment

Yes. The active linked PR is the narrowest maintainable fix shape because it changes the Signal-owned daemon lifecycle and monitor cleanup, with focused tests, instead of adding gateway-level delay or a broad unrelated child-process rewrite.
