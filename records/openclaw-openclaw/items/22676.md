## Label Justifications

- `P1`: The issue affects a real Signal channel restart workflow and can leave users unable to send messages until manual process cleanup.
- `impact:message-loss`: The reported race causes Signal send failures when the new daemon races a stale daemon on the HTTP port or config lock.
- `impact:crash-loop`: The report describes orphaned signal-cli processes and restart recovery that requires manually killing stale processes.

## Summary

Keep open: current main and the latest release still have the reported fire-and-forget Signal daemon shutdown path, and the direct closing PR is open but unmerged. This issue should keep tracking the user-facing Signal restart race until https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused fix lands.

## What This Changes

The issue reports that Signal hot-reload restarts send SIGTERM to signal-cli without awaiting process exit, allowing orphaned daemons, port/config-lock races, and send failures.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/71863, or land an equivalent focused Signal-owned fix that makes daemon shutdown async, idempotent, bounded by SIGKILL escalation, and awaited through monitor teardown before channel restart.

## Reproduction Assessment

Yes at source level: current main still aborts the Signal monitor, calls a synchronous daemon stop, and lets gateway reload proceed without waiting for the signal-cli process to exit. I did not run a live Signal account restart reproduction in this read-only review.

## Solution Assessment

Yes: making the Signal daemon stop async/idempotent, waiting for exit after SIGTERM, escalating to SIGKILL after a bounded timeout, and awaiting that through monitor teardown is the narrow maintainable fix. The open linked PR targets that shape, so a second automated fix PR should not be queued while it remains active.
