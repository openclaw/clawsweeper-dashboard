## Summary

Keep open: current main and v2026.6.5 still have the Signal shutdown race, and the open linked fix PR has not merged yet.

## What This Changes

The issue reports that Signal hot-reload/restart can orphan signal-cli and cause sends to fail because daemon stop is fire-and-forget during restart.

## Best Possible Solution

Refresh and land the linked Signal-owned fix, or an equivalent focused fix, then close this issue with the fixing commit and release provenance.

## Reproduction Assessment

Yes, at source level: hot reload awaits stopChannel before startChannel, but Signal monitor cleanup can finish after only requesting SIGTERM because daemon exit is not awaited. The discussion also has repeated real-user confirmations, but this review did not run a live Signal account reproduction.

## Solution Assessment

Yes in direction: fixing shutdown inside the Signal plugin by making stop async/idempotent, waiting for exit, adding bounded SIGKILL escalation, and awaiting monitor teardown is the narrow owner-correct solution. The existing linked PR should be refreshed and reviewed rather than opening a duplicate repair lane.
