## Summary

Keep open. Current main and the latest release still send SIGTERM to the Signal daemon without awaiting process exit, and PR #71863 is an open unmerged closing fix, so this issue should keep tracking the bug until that PR or an equivalent Signal-owned fix lands.

## What This Changes

Make Signal daemon restart shutdown wait for signal-cli exit, with bounded SIGKILL escalation, before a hot reload starts a new instance.

## Best Possible Solution

The Signal plugin should own an async, idempotent daemon shutdown contract and regression tests so channel hot reload cannot start a new daemon until the old signal-cli process has exited or been killed after a bounded timeout.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable Signal auto-start, trigger a `channels.signal` hot reload, and current main stops then starts the channel while daemon stop only sends SIGTERM and returns.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Signal daemon lifecycle and monitor teardown with awaited SIGTERM exit plus bounded SIGKILL escalation; reviewing/landing PR #71863 or an equivalent patch is the right path.
