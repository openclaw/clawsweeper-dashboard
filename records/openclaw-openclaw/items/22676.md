## Label Justifications

- `P1`: The race can break an active Signal restart path and cause failed sends for real users.
- `impact:message-loss`: The report is about Signal send failures when restart races leave the daemon or RPC target in a bad state.
- `impact:crash-loop`: The report is about orphaned `signal-cli` processes and restart failures that affect channel process availability.

## Summary

Keep open: current main and v2026.5.19 still have the synchronous Signal daemon stop path, and the direct closing fix at https://github.com/openclaw/openclaw/pull/71863 is open but unmerged.

## What This Changes

The issue asks OpenClaw to wait for `signal-cli` daemon exit, with bounded escalation, before restarting Signal after SIGUSR1 or hot-reload teardown.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal lifecycle fix with regression coverage, then close this issue with fixing commit and release provenance.

## Reproduction Assessment

Yes, at source level. With Signal autoStart enabled, a `channels.signal` hot reload aborts the channel and immediately restarts it while Signal cleanup only sends SIGTERM and returns before `signal-cli` exits.

## Solution Assessment

Yes. Making Signal daemon stop async/idempotent, waiting on the existing exit promise with bounded SIGKILL escalation, and awaiting that shutdown through monitor teardown is the narrow maintainable fix.
