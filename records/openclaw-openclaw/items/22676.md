## Label Justifications

- `P1`: The issue affects real Signal restart workflows by leaving daemon processes behind and breaking message sends after hot reloads.
- `impact:message-loss`: The reported failure mode includes Signal send failures when restart races against the previous daemon process.

## Summary

The issue is still necessary: current main and the latest release still use a synchronous SIGTERM-only Signal daemon stop path, and the direct closing pull request remains open and unmerged.

## What This Changes

The issue requests awaited Signal daemon shutdown during gateway restarts so old signal-cli processes exit before a replacement daemon starts.

## Best Possible Solution

Keep this issue open and review or land https://github.com/openclaw/openclaw/pull/71863, or land an equivalent focused Signal plugin fix that makes daemon stop async, idempotent, bounded by SIGKILL escalation, and awaited through monitor teardown before restart.

## Reproduction Assessment

Yes at source level. Current main still returns immediately after SIGTERM in Signal daemon stop, while Signal config reload can stop then start the channel; I did not run a live linked-Signal restart in this read-only review.

## Solution Assessment

Yes in direction. A Signal-owned async, idempotent stop awaited through monitor teardown is the narrow maintainable fix; the existing linked pull request is the right landing path rather than opening another repair branch.
