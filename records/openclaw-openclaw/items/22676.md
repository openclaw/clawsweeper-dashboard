## Summary

Keep open: current main still has the reported Signal daemon stop race, the latest release still has the same synchronous cleanup, and https://github.com/openclaw/openclaw/pull/71863 is an open closing PR rather than a reason to close this issue before merge.

## What This Changes

The issue requests that Signal daemon shutdown wait for signal-cli to exit, with bounded SIGKILL escalation, before hot restart starts a new daemon.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal plugin lifecycle fix with regression coverage, then close this issue with fixing commit and release provenance.

## Reproduction Assessment

Yes, at source level. With Signal autoStart enabled, a channels.signal hot reload aborts the channel and immediately restarts it, while Signal cleanup only sends SIGTERM and returns before signal-cli exits.

## Solution Assessment

Yes. Making Signal daemon stop async/idempotent, waiting on the existing exit promise with bounded SIGKILL escalation, and awaiting that shutdown through monitor teardown is the narrow maintainable fix.
