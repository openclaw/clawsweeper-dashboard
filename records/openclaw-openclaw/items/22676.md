## Summary

Keep open: current main still matches the reported Signal daemon stop race, latest release v2026.5.12 still has the same synchronous cleanup, and https://github.com/openclaw/openclaw/pull/71863 is the open closing PR rather than a reason to close this issue before merge.

## What This Changes

The issue reports that Signal auto-start hot restarts stop signal-cli with fire-and-forget SIGTERM, causing orphaned daemon processes and failed sends when the channel restarts before the old daemon exits.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/71863 or an equivalent focused Signal plugin lifecycle fix with regression coverage, then close this issue with fixing commit and release provenance.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: with Signal autoStart enabled, a channels.signal hot reload aborts the channel, but Signal cleanup only sends SIGTERM and returns before signal-cli exits.

## Solution Assessment

Yes. Making Signal daemon stop async/idempotent, waiting on the existing exit promise with bounded SIGKILL escalation, and awaiting that through monitor teardown is the narrow maintainable fix.
