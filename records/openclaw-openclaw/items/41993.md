## Summary

Keep open: current main and the latest release still reject the reported mixed DNS-answer shape before the IPv4-first pinned address list is built. The direct fix PRs for this issue are closed unmerged, and the remaining change is an SSRF policy decision that should stay with maintainers/security review.

## What This Changes

The issue reports that `web_fetch` fails in dual-stack networks when DNS returns an allowed public IPv4 address alongside a blocked IPv6 special-use address.

## Best Possible Solution

Have maintainers approve the mixed-DNS SSRF policy, then land one focused helper that rejects all-blocked DNS responses and feeds only allowed records into initial pinned DNS and pinned dispatcher override paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: mock DNS to return a public IPv4 plus blocked IPv6 such as `::`, then call `resolvePinnedHostnameWithPolicy` or `fetchWithSsrFGuard`; current main rejects before IPv4-first pinning.

## Solution Assessment

No current fix is present. The narrow maintainable fix should return a filtered allowed-record set and use it consistently for both initial pinned DNS and pinned dispatcher override paths, after maintainer security approval.
