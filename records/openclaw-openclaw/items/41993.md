## Summary

Keep open. Current main and the latest release still reject a DNS result set as soon as any resolved address is blocked, before IPv4-first pinning can choose a safe public answer; the related fix PRs are closed unmerged, and the remaining change is SSRF-sensitive.

## What This Changes

Allow `web_fetch` to use a safe public DNS answer when the same hostname also resolves to blocked special-use IPv6 addresses, while keeping blocked addresses out of pinned lookups.

## Best Possible Solution

Implement a filter-then-pin change in `src/infra/net/ssrf.ts`: drop blocked resolved records, throw only when none remain, apply the filtered set to dispatcher overrides too, and add focused SSRF regression coverage.

## Reproduction Assessment

Yes, at source level. A lookup returning a public IPv4 plus `::`, `::1`, or another blocked special-use IPv6 reaches the strict resolved-address assertion before IPv4-first pinning.

## Solution Assessment

Yes in direction, with a security constraint. The maintainable fix is filter-then-pin across both normal pinned lookup and dispatcher override paths, not merely relaxing the throw condition.
