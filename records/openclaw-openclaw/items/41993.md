## Summary

Keep open. Current main still rejects a hostname when any DNS answer is blocked, so the reported public IPv4 plus blocked IPv6 special-use case remains source-reproducible; the related fix PRs were closed unmerged.

## What This Changes

`web_fetch` should allow a DNS result set with at least one safe public address while still blocking result sets where every address is private, internal, or special-use.

## Best Possible Solution

Add a narrow helper that filters resolved DNS answers to allowed addresses, throws only when none remain, passes only the filtered set into dedupe and pinned dispatcher construction, and covers mixed allow/block cases in SSRF and web_fetch tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a mocked lookup returning `[{ address: "93.184.216.34", family: 4 }, { address: "::", family: 6 }]` reaches `assertAllowedResolvedAddressesOrThrow()` and throws before IPv4-first pinning can use the public address.

## Solution Assessment

Yes. The issue's filter-then-pin direction is the narrow maintainable fix, provided blocked answers are removed before pinned lookup and dispatcher override construction; broad private-network or fake-IP config opt-ins are not equivalent.
