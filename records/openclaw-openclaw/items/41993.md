- `impact:security`: The remaining decision is in the SSRF DNS guard for outbound fetches.

## Summary

Keep open: current `main` and latest release still reject mixed DNS answers when any resolved address is blocked, before IPv4-first pinning can use the public IPv4 address. The direct fix PRs are all closed unmerged, and the later merged proxy/fake-IP fixes cover adjacent cases rather than this default strict-path behavior.

## What This Changes

The issue reports that `web_fetch` fails on dual-stack DNS results when a blocked IPv6 special-use address is returned alongside an otherwise usable public IPv4 address.

## Best Possible Solution

Land a maintainer-approved SSRF fix that filters blocked DNS candidates before pinning while preserving fail-closed behavior when no allowed address remains.

## Reproduction Assessment

Yes from source inspection: a lookup result containing `::` plus a public IPv4 address still reaches an any-blocked-address assertion before IPv4-first pinning. I did not run a live DNS-pollution repro because this review is read-only.

## Solution Assessment

Unclear: filtering blocked resolved addresses before pinning is a plausible narrow fix, but the SSRF policy change needs maintainer/security confirmation because it relaxes the current all-answers-must-be-safe rule.
