## Summary

Keep open. Current main still rejects a hostname when any resolved DNS answer is blocked, so the reported public-IPv4 plus blocked IPv6 case remains valid and unmerged fix attempts have not superseded it.

## What This Changes

The issue requests that `web_fetch` SSRF DNS pinning allow public targets when DNS returns at least one allowed address alongside blocked IPv6 special-use answers, while excluding blocked answers from pinned candidates.

## Best Possible Solution

Keep this issue open and implement a narrow SSRF helper that returns the allowed resolved-address set, throws only when no allowed DNS answer remains, and feeds only that filtered set into pinned lookup and dispatcher construction. Cover mixed public IPv4 plus blocked IPv6, all-blocked DNS, and IPv4-first ordering after filtering.

## Reproduction Assessment

Yes. A mocked lookup returning a public IPv4 plus `::` or `::1` would hit `assertAllowedResolvedAddressesOrThrow()` and throw before `dedupeAndPreferIpv4()`; `web_fetch` reaches that guarded path and rethrows the SSRF block.

## Solution Assessment

Yes, the issue’s filter-then-pin direction is the narrow maintainable fix. Fake-IP/SSRF policy opt-ins are not equivalent because they widen specific ranges and do not address mixed blocked IPv6 answers with an otherwise allowed public address.
