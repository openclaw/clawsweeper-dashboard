## Summary

Keep open. Current main still rejects a DNS answer set as soon as any resolved address is blocked, so a public IPv4 answer mixed with `::`, `::1`, or another blocked IPv6 answer cannot reach the later IPv4-first pinning path; the related fix PRs are closed unmerged.

## What This Changes

The issue asks for `web_fetch` SSRF DNS handling to allow mixed DNS results when at least one resolved address is allowed, while excluding blocked addresses from the pinned connection pool.

## Best Possible Solution

Implement a narrow filter-then-pin helper that removes blocked DNS answers, fails only when no allowed answer remains, and passes only allowed addresses into normal pinned lookups and dispatcher overrides with focused SSRF tests.

## Reproduction Assessment

Yes, at source level. A lookup returning an allowed public IPv4 plus `::` or `::1` reaches `assertAllowedResolvedAddressesOrThrow()` and throws before `dedupeAndPreferIpv4()` can choose the IPv4 answer.

## Solution Assessment

Yes in direction, with a security constraint. The maintainable fix is filter-then-pin; merely allowing mixed result sets while leaving blocked answers in the pinned pool would be unsafe.
