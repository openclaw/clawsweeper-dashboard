## Label Justifications

- `P1`: The issue can break the user-facing `web_fetch` tool for real users on dual-stack or DNS-polluted networks.
- `impact:security`: The affected behavior is in the SSRF DNS-resolution guard for outbound fetches.

## Summary

Keep open: current main and the latest release still validate every DNS answer before IPv4-first pinning, so one blocked special-use IPv6 answer still blocks an otherwise usable public IPv4 result. The direct fix attempts for this behavior are closed unmerged, while merged fake-IP and proxy work only covers adjacent opt-in paths.

## What This Changes

The issue asks for `web_fetch` DNS pinning to allow mixed DNS responses when at least one resolved address is public and safe instead of failing on any blocked special-use IPv6 answer.

## Best Possible Solution

Keep this open for a maintainer-approved SSRF fix that filters blocked DNS candidates before pinning, fails closed when no allowed address remains, and adds regression coverage for mixed allowed-plus-blocked DNS answers.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a lookup result containing `::` plus a public IPv4 address still reaches an any-blocked-address assertion before IPv4-first pinning; I did not run a live DNS-pollution repro in this read-only review.

## Solution Assessment

Unclear until maintainer security review. Filtering blocked resolved addresses before pinning is the narrow candidate fix, but maintainers need to approve the mixed-answer SSRF policy change and verify blocked candidates cannot be used later.
