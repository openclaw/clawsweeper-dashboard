## Summary

Keep open. Current main and the latest release still reject a mixed DNS answer as soon as any resolved address is blocked, before IPv4-first pinning can choose the public address; the prior focused fix PRs are closed unmerged and the remaining change is SSRF-sensitive.

## What This Changes

The issue requests that `web_fetch` tolerate mixed DNS answers by filtering blocked special-use IPv6 records when at least one public address remains.

## Best Possible Solution

Implement a filter-then-pin change in `src/infra/net/ssrf.ts`: build an allowed resolved-address set, throw only when none remain, use the filtered set for normal pinning and dispatcher overrides, and add focused SSRF regression coverage.

## Reproduction Assessment

Yes, at source level. A lookup returning a public IPv4 plus `::`, `::1`, or another blocked special-use IPv6 reaches the strict resolved-address assertion before IPv4-first pinning can choose the safe public answer.

## Solution Assessment

Yes in direction, with a security constraint. The maintainable fix is filter-then-pin across both normal pinned lookup and dispatcher override paths, not a broad SSRF relaxation or a config-only workaround.
