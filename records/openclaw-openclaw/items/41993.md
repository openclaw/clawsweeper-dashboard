## Summary

Keep open. Current main still rejects the whole hostname resolution when any DNS answer is blocked, so the reported public IPv4 plus blocked IPv6 special-use case remains valid; the related fix PRs were closed unmerged.

## What This Changes

`web_fetch` should allow public fetches when DNS returns at least one allowed address alongside blocked IPv6 special-use answers, while keeping blocked answers out of pinned candidates.

## Best Possible Solution

Implement a narrow helper that filters DNS results to allowed addresses, throws only when no allowed answer remains, and feeds only the filtered set into pinned lookup and dispatcher construction.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a mocked lookup returning `[{ address: "93.184.216.34", family: 4 }, { address: "::", family: 6 }]` reaches `assertAllowedResolvedAddressesOrThrow()` and throws before IPv4-first pinning can use the public address.

## Solution Assessment

Yes, the issue’s filter-then-pin direction is the narrow maintainable fix, provided blocked answers are removed before dispatcher construction. Fake-IP policy opt-ins are not equivalent because they widen specific synthetic ranges instead of handling mixed DNS answers safely by default.
