## Summary

Current `main` still rejects a DNS result set when any answer is private, internal, or special-use, so the reported dual-stack failure remains source-reproducible. Three focused fix PRs closed without merge because of branch conflicts, excessive unrelated churn, or contributor queue management—not because the behavior was resolved—but changing this rule requires security-owner approval and proof across every pinned dispatcher path.

## What This Changes

Allow `web_fetch` to use public answers from a mixed DNS response while removing blocked answers from the pinned connection pool.

## Best Possible Solution

Use one canonical helper that derives a non-empty allowed DNS record set, rejects all-blocked responses, and passes only that filtered set into every pinned dispatcher mode, with explicit security-owner approval and focused direct, proxy, retry, and family-selection regression coverage.

## Reproduction Assessment

Yes, from source with high confidence: return one public IPv4 answer plus one blocked IPv6 answer from the injected lookup, and `resolvePinnedHostnameWithPolicy` throws before creating its pinned lookup. This read-only review did not exercise a real polluted dual-stack network.

## Solution Assessment

Unclear without maintainer approval: filtering blocked records before pinning is the narrowest proposed fix, but it is the best solution only if every dispatcher and proxy path is proven unable to connect to a removed record and maintainers accept the mixed-answer threat model.
