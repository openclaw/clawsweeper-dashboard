## Summary

Keep open. Current main and the latest shipped release still throw on any blocked DNS answer before IPv4-first pinning, so the reported mixed public IPv4 plus special-use IPv6 `web_fetch` failure remains source-reproducible; the focused fix attempts were closed without merge.

## What This Changes

The issue reports that `web_fetch` rejects public sites on dual-stack DNS responses when any resolved IPv6 special-use address is present alongside a usable public IPv4 address.

## Best Possible Solution

Implement a security-reviewed filter-then-pin change in `src/infra/net/ssrf.ts`: keep strict hostname checks, discard blocked resolved addresses, throw only when none remain, and cover normal DNS plus dispatcher overrides with focused tests.

## Reproduction Assessment

Yes, at source level. A lookup returning a public IPv4 plus `::`, `::1`, Teredo, or another blocked special-use IPv6 reaches `assertAllowedResolvedAddressesOrThrow()` before IPv4-first pinning can select the safe address.

## Solution Assessment

Yes in direction, but not as an automatic repair. The maintainable fix is a narrow filter-then-pin change that preserves all-blocked denial and receives explicit security review.
