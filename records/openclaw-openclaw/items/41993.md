## Summary

Keep open. Current `main` still fails closed when any DNS answer is blocked, so a public IPv4 answer mixed with `::`, `::1`, or Teredo-style IPv6 never reaches the IPv4-first pinned lookup; the focused fix PRs are all closed unmerged and the remaining change is SSRF-sensitive.

## What This Changes

The issue requests that guarded `web_fetch` tolerate mixed DNS answer sets by using allowed public addresses while filtering blocked IPv6 special-use addresses.

## Best Possible Solution

Filter blocked DNS answers once, throw only when no allowed answer remains, pass only the filtered set into pinned lookups and dispatcher overrides, and add focused SSRF regression coverage.

## Reproduction Assessment

Yes, at source level. A mocked lookup returning an allowed public IPv4 plus `::` or `::1` reaches `assertAllowedResolvedAddressesOrThrow()` and throws before `dedupeAndPreferIpv4()` can choose the IPv4 answer.

## Solution Assessment

Yes in direction, with a security constraint. The maintainable fix is filter-then-pin; relaxing the throw while leaving blocked answers in any pinned pool would be unsafe.
