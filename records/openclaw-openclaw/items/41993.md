## Summary

Keep open. Current main and the latest release still reject a DNS result set when any resolved address is blocked before IPv4-first pinning can select the safe public address, and the repair changes a security-sensitive SSRF boundary.

## What This Changes

The issue requests that `web_fetch` succeed when DNS returns at least one allowed public address alongside blocked IPv6 special-use answers.

## Best Possible Solution

Approve and implement a narrow filter-then-pin SSRF change: keep strict hostname checks, discard blocked DNS answers from normal and override candidate pools, deny all-blocked result sets, and add regression coverage for both paths.

## Reproduction Assessment

Yes, at source level. A lookup result containing a public IPv4 plus `::`, `::1`, Teredo, or another blocked IPv6 answer reaches `assertAllowedResolvedAddressesOrThrow()` before IPv4-first pinning can select the safe address.

## Solution Assessment

Yes in direction, but not as a broad allow-any-mixed-result change. The maintainable fix is to filter blocked resolved addresses before pinning while preserving all-blocked denial and dispatcher override safety.
