## Summary

Keep open. Current main still rejects a hostname resolution as soon as any DNS answer is blocked, so the reported public IPv4 plus blocked IPv6 special-use case remains valid; the related fix PRs were closed unmerged, and the later FakeIP opt-ins do not cover `::`, `::1`, or Teredo-style mixed answers by default.

## What This Changes

The issue asks `web_fetch` to allow mixed DNS results when at least one resolved address is public, while continuing to block all-private or all-special-use resolutions.

## Best Possible Solution

Implement a narrow filtered-DNS helper that returns only allowed resolved addresses, throws when no allowed answer remains, and feeds the filtered set into pinned lookup and dispatcher construction after maintainer security review.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a lookup returning `[{ address: "93.184.216.34", family: 4 }, { address: "::", family: 6 }]` reaches `assertAllowedResolvedAddressesOrThrow()` and throws before IPv4-first pinning can use the public address.

## Solution Assessment

Yes, with one important constraint: the maintainable fix is filter-then-pin, not merely relaxing the assertion. All-blocked resolutions must remain fail-closed and blocked answers must not remain in the pinned candidate pool.
