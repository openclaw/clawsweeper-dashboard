## Summary

Keep open. Current main still rejects a DNS result set as soon as any resolved address is blocked, before IPv4-first pinning can use an allowed public address, and the related fix PRs were closed unmerged.

## What This Changes

The issue reports that `web_fetch` fails on dual-stack DNS responses that mix an allowed public IPv4 answer with blocked IPv6 special-use answers.

## Best Possible Solution

Implement a narrow filter-then-pin helper that removes blocked resolved addresses, fails only when no allowed answer remains, and feeds only allowed addresses into pinned lookup and dispatcher overrides.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a lookup returning an allowed public IPv4 plus `::` or `::1` reaches `assertAllowedResolvedAddressesOrThrow()` and throws before `dedupeAndPreferIpv4()` can choose IPv4.

## Solution Assessment

Yes in direction, with a security constraint. The maintainable fix is filter-then-pin; merely allowing mixed result sets while leaving blocked answers in the pinned pool would be unsafe.
