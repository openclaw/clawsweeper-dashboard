## Summary

Keep open. Current main still rejects a DNS answer set as soon as any resolved address is blocked, so an allowed public IPv4 answer mixed with blocked IPv6 special-use records cannot reach the later IPv4-first pinned lookup; the focused fix PRs are closed unmerged and the remaining change is SSRF-sensitive.

## What This Changes

The issue requests that `web_fetch` tolerate mixed DNS answers by filtering blocked IPv6 special-use records before pinned fetch resolution when at least one public address is available.

## Best Possible Solution

Filter blocked DNS answers once, throw only when no allowed answer remains, and pass only the filtered set into pinned lookups and dispatcher overrides with focused SSRF regression coverage.

## Reproduction Assessment

Yes, at source level. A lookup returning an allowed public IPv4 plus `::` or `::1` reaches `assertAllowedResolvedAddressesOrThrow()` and throws before `dedupeAndPreferIpv4()` can choose the IPv4 answer.

## Solution Assessment

Yes in direction, with a security constraint. The maintainable fix is filter-then-pin; relaxing the throw while leaving blocked answers in the pinned pool would be unsafe.
