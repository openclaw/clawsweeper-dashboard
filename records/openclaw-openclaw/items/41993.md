## Summary

Keep open: current `main` and latest release still validate every DNS answer before IPv4-first pinning, so a blocked IPv6 special-use answer still rejects the whole `web_fetch` request. The direct fix PRs are closed unmerged, and adjacent trusted-proxy/fake-IP work does not cover the default mixed-answer case.

## What This Changes

The issue asks OpenClaw to let `web_fetch` proceed when DNS returns at least one allowed public address alongside blocked IPv6 special-use addresses.

## Best Possible Solution

Land a maintainer-approved SSRF fix that filters blocked DNS candidates before pinning while preserving fail-closed behavior when no allowed address remains.

## Reproduction Assessment

Yes from source inspection: a lookup result containing `::` plus a public IPv4 address still reaches an any-blocked-address assertion before IPv4-first pinning. I did not run a live DNS-pollution repro in this read-only pass.

## Solution Assessment

Unclear: filtering blocked resolved addresses before pinning is a plausible narrow fix, but the SSRF policy change needs maintainer/security confirmation because it relaxes the current all-answers-must-be-safe rule.
