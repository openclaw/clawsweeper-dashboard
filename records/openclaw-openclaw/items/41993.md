## Summary

Keep open. Current main and the latest release still reject a mixed DNS answer set as soon as any resolved address is blocked, before IPv4-first pinning can select a safe public address; the related fix PRs are closed unmerged and the remaining repair changes an SSRF boundary.

## What This Changes

The issue asks for `web_fetch` to succeed when DNS returns at least one safe public address alongside blocked IPv6 special-use answers.

## Best Possible Solution

Have a maintainer/security owner approve and land a narrow filter-then-pin SSRF fix that filters blocked records from both normal and override pools, preserves all-blocked denial, and adds mixed-answer regression coverage.

## Reproduction Assessment

Yes, at source level. A lookup result containing a public IPv4 plus `::`, `::1`, Teredo, or another blocked IPv6 answer reaches `assertAllowedResolvedAddressesOrThrow()` before `dedupeAndPreferIpv4()` can select the safe address.

## Solution Assessment

Yes in direction, but not as a bare allow-any-mixed-result assertion change. The maintainable fix is to filter blocked resolved addresses before pinning while preserving all-blocked denial and dispatcher override safety.
