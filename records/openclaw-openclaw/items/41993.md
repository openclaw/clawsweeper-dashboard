## Summary

Keep open. Current `main` and `v2026.5.22` still throw on the first blocked DNS answer before IPv4-first pinning, so the reported `web_fetch` regression remains; the repair is narrow but changes an SSRF boundary.

## What This Changes

The issue reports that `web_fetch` rejects public hosts on dual-stack DNS when a blocked IPv6 special-use answer is returned alongside an allowed public IPv4 answer.

## Best Possible Solution

Have a maintainer/security owner approve a narrow filter-then-pin SSRF change that removes blocked records from normal and override pinned pools, preserves all-blocked denial, and adds mixed-answer regression coverage.

## Reproduction Assessment

Yes, at source level. A lookup result containing a public IPv4 plus `::`, `::1`, Teredo, or another blocked IPv6 answer reaches `assertAllowedResolvedAddressesOrThrow()` before `dedupeAndPreferIpv4()` can select the safe address.

## Solution Assessment

No for the bare assertion-only proposed patch; the maintainable fix should filter blocked resolved addresses before pinning, apply the same rule to override candidates, and keep all-blocked DNS answers denied.
