## Summary

Keep open: current `main` and the latest release still reject the reported mixed DNS result shape before the IPv4-first pinned lookup is built. The exact fix attempts are closed unmerged, and the remaining decision is security/product-sensitive rather than stale cleanup.

## What This Changes

The issue asks for `web_fetch` to allow a hostname when DNS returns at least one allowed public address, even if another returned IPv6 address is special-use.

## Best Possible Solution

Have maintainers approve the mixed-DNS SSRF policy, then implement one canonical filter-and-assert helper that rejects all-blocked responses and feeds only allowed records into every pinned dispatcher path with focused regression coverage.

## Reproduction Assessment

Yes from source: `web_fetch` reaches `resolvePinnedHostnameWithPolicy`, where any blocked DNS answer throws before the IPv4-first pinned list is created. A mixed DNS response with public IPv4 plus blocked IPv6 therefore still fails on current `main`.

## Solution Assessment

No current fix is present. The proposed direction is plausible, but the best fix should return a filtered allowed-record set and use it consistently for both initial pinned DNS and pinned dispatcher override paths.
