## Summary

Keep open. Current `main` and the latest release still reject a mixed DNS answer as soon as any resolved address is blocked, before IPv4-first pinning can select the public answer; the prior focused fix PRs are closed unmerged and the remaining change is SSRF-sensitive.

## What This Changes

The issue asks `web_fetch` to allow DNS result sets with at least one public resolved address even when blocked IPv6 special-use addresses are also returned.

## Best Possible Solution

Implement a filter-then-pin change in `src/infra/net/ssrf.ts`: drop blocked resolved records, throw only when none remain, apply the filtered set to dispatcher overrides, and add focused SSRF regression coverage.

## Reproduction Assessment

Yes, at source level. A lookup returning a public IPv4 plus `::`, `::1`, or another blocked special-use IPv6 reaches the strict resolved-address assertion before IPv4-first pinning can choose the safe public answer.

## Solution Assessment

Yes in direction, with a security constraint. The maintainable fix is filter-then-pin across both normal pinned lookup and dispatcher override paths, not a simple relaxation that leaves blocked addresses in the candidate pool.
