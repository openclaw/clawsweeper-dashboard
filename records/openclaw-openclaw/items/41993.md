## Summary

Keep open. Current main still rejects a mixed DNS answer set as soon as any resolved address is blocked, so a public IPv4 answer cannot reach the later IPv4-first pinned lookup; the known focused fix PRs are closed unmerged and the remaining change is SSRF-sensitive.

## What This Changes

The issue asks for `web_fetch` DNS pinning to allow mixed public-plus-blocked DNS answers when at least one resolved address is safe, while filtering blocked answers out of the pinned connection pool.

## Best Possible Solution

Filter blocked DNS answers once, throw only when no allowed answer remains, and pass only the filtered set into pinned lookups and dispatcher overrides with focused SSRF regression coverage and owner/security review.

## Reproduction Assessment

Yes, at source level. A lookup returning a public IPv4 plus `::`, `::1`, or another blocked IPv6 special-use address reaches `assertAllowedResolvedAddressesOrThrow()` and throws before `dedupeAndPreferIpv4()` can choose the IPv4 answer.

## Solution Assessment

Yes in direction, with a security constraint. The maintainable fix is filter-then-pin; simply changing the assertion while leaving blocked answers available to pinned lookup or dispatcher override paths would not be safe.
