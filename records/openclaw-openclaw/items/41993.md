## Summary

Keep this open: current main and the latest release still reject a mixed DNS result set as soon as any answer is blocked, and the linked fix PRs are closed unmerged. The repair shape is clear, but it changes an SSRF boundary and needs maintainer/security approval before a new implementation lane.

## What This Changes

The issue requests that web_fetch allow mixed DNS answer sets when at least one resolved address is public-safe instead of failing because another answer is special-use IPv6.

## Best Possible Solution

Have maintainers approve the mixed-DNS SSRF policy, then implement one canonical helper that filters to a non-empty allowed DNS record set, rejects all-blocked responses, and feeds only filtered records into every pinned dispatcher path with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence repro path: return one public IPv4 answer plus one blocked IPv6 answer from the injected lookup, and `resolvePinnedHostnameWithPolicy` throws before creating its pinned lookup; I did not run a live polluted dual-stack network test in this read-only review.

## Solution Assessment

Unclear without maintainer/security approval. Filtering blocked records before pinning is the narrow maintainable fix shape, but it is only the best solution if maintainers accept the mixed-answer threat model and every dispatcher path is proven to use only the filtered set.
