## Summary

Keep open: current main and v2026.6.10 still reject a DNS result set as soon as any resolved address is blocked, before IPv4-first pinning can use a valid public IPv4 answer. The direct fix PRs are all closed unmerged, while the merged proxy and fake-IP work covers adjacent paths rather than this strict default mixed-answer behavior.

## What This Changes

The issue asks for `web_fetch` to succeed when DNS returns at least one allowed public address alongside blocked IPv6 special-use answers, without ever pinning blocked addresses.

## Best Possible Solution

Land a maintainer-approved SSRF fix that filters blocked DNS candidates before pinning, fails closed when no allowed address remains, and adds regression coverage for mixed allowed-plus-blocked DNS answers.

## Reproduction Assessment

Yes, source-level: a lookup result containing `::` plus a public IPv4 address still reaches an any-blocked-address assertion before IPv4-first pinning. I did not run a live DNS-pollution repro in this read-only review.

## Solution Assessment

Unclear until maintainer security review: filtering blocked resolved addresses before pinning is the narrow candidate fix, but maintainers need to approve the mixed-answer SSRF policy change.
