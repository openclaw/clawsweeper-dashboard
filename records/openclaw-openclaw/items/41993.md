## Summary

Keep open. Current main still rejects a DNS answer set as soon as any resolved address is blocked, so a public IPv4 answer mixed with `::`, `::1`, or similar special-use IPv6 never reaches the IPv4-first pinned lookup; the focused fix attempts linked from the issue are closed unmerged and the remaining change is SSRF-sensitive.

## What This Changes

The issue asks OpenClaw to let `web_fetch` proceed when DNS returns at least one allowed public address alongside blocked IPv6 special-use addresses.

## Best Possible Solution

Filter blocked DNS answers once, throw only when no allowed answer remains, and pass only the filtered set into pinned lookups and dispatcher overrides with focused SSRF regression coverage and owner/security review.

## Reproduction Assessment

Yes, at source level. A lookup returning a public IPv4 plus `::`, `::1`, or another blocked IPv6 special-use address reaches `assertAllowedResolvedAddressesOrThrow()` and throws before `dedupeAndPreferIpv4()` can choose the IPv4 answer.

## Solution Assessment

Yes in direction, with a security constraint. The maintainable fix is filter-then-pin; simply changing the assertion while leaving blocked answers available to pinned lookup or dispatcher override paths is not the best solution.
