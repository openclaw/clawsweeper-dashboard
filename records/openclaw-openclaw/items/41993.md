## Summary

Keep open. Current main still rejects a hostname resolution as soon as any DNS answer is blocked, before IPv4-first pinning can use an allowed public address; the related fix PRs were closed unmerged, and later fake-IP opt-ins do not cover `::`, `::1`, or Teredo-style mixed answers by default.

## What This Changes

The issue asks `web_fetch` to allow mixed DNS results when at least one resolved address is public and to filter blocked IPv6 special-use answers out of the pinned pool.

## Best Possible Solution

Add a narrow helper that filters blocked DNS answers, throws when no allowed answer remains, and feeds only the filtered set into IPv4-first ordering and pinned dispatcher lookup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a lookup returning a public IPv4 plus `::` or another blocked IPv6 answer reaches `assertAllowedResolvedAddressesOrThrow()` and throws before `dedupeAndPreferIpv4()` can select the public address.

## Solution Assessment

Yes, with the security constraint that the fix should be filter-then-pin rather than merely relaxing the throw condition; all-blocked result sets must still fail closed.
