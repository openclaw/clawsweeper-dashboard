## Summary

Keep open. Current `main` and the latest shipped release still reject a DNS result set when any resolved address is blocked, so the reported mixed public IPv4 plus special-use IPv6 `web_fetch` failure remains source-reproducible; the linked fix PRs were all closed without merge.

## What This Changes

The issue requests changing `web_fetch` SSRF DNS validation so mixed public IPv4 plus blocked IPv6 answers are allowed only after blocked addresses are removed from pinned candidates.

## Best Possible Solution

Approve and implement a narrow filter-then-pin SSRF change: keep strict hostname checks, remove blocked DNS answers from every candidate pool, deny all-blocked results, and cover normal DNS plus dispatcher override paths with regression tests.

## Reproduction Assessment

Yes, at source level. A lookup result containing a public IPv4 plus `::`, `::1`, Teredo, or another blocked IPv6 answer reaches `assertAllowedResolvedAddressesOrThrow()` before IPv4-first pinning can select the safe public address.

## Solution Assessment

Yes in direction, but it needs maintainer and security approval. The maintainable fix is not a broad SSRF bypass or config workaround; it is filtering blocked resolved addresses before pinning while preserving all-blocked denial.
