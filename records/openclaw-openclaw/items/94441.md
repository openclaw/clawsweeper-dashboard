## Summary

Keep open: the PR fixes a valid source-reproducible exec bug and I found no blocking patch defect, but it intentionally changes a shipped explicit-workdir fallback into a failed tool result, so maintainers should accept that compatibility tradeoff before merge.

## What This Changes

The PR changes local gateway exec so an explicit unavailable `workdir` returns a failed result before command execution, while implicit/default workdir fallback stays in place.

## Best Possible Solution

Land the narrow fail-fast behavior if maintainers accept the explicit-workdir compatibility change; otherwise ask for a tilde-expansion or other narrower compatibility-preserving variant.

## Reproduction Assessment

Yes from source: current main and `v2026.6.8` call `resolveWorkdir` for local gateway exec, which falls back to cwd/home when the explicit workdir is unavailable. I did not run a live repro because this review is read-only.

## Solution Assessment

Yes, with a compatibility caveat: failing an explicit invalid local workdir before execution is the narrowest way to prevent wrong-directory commands while preserving implicit/default fallback. Leading-home expansion alone would not fix other invalid explicit paths that can still execute from the wrong directory.
