## Summary

Keep open: current main still conflates queued/steered announce acceptance with delivery. The lifecycle code writes `completionAnnouncedAt` and marks delivery status once `runSubagentAnnounceFlow()` returns true, while the steered path returns true from queue acceptance rather than transcript-commit evidence; current main also lacks the requested enqueued/delivered timestamp split and drop-reason field.

## What This Changes

The issue requests that subagent completion announce state distinguish queue acceptance from actual parent-transcript delivery, keeping `completionAnnouncedAt` as a delivered-time signal.

## Best Possible Solution

Keep this issue open and fix the registry contract so enqueue, transcript delivery, delivery failure, and drop/dedupe outcomes are recorded as separate durable states.

## Reproduction Assessment

Yes, from source inspection: a steered completion announce can be marked delivered when the active parent session accepts the queued steering message, before the upstream session emits the queued message into the transcript. I did not execute the full live timing scenario.

## Solution Assessment

Yes, the proposed direction is the right maintainable shape: preserve `completionAnnouncedAt` as a delivery-time/back-compat field and add explicit enqueue, delivered, error, and drop/dedupe state instead of treating queue acceptance as delivery.
