## Summary

Keep open. Current main still treats active requester queue acceptance as a successful completion announce and then writes `completionAnnouncedAt`; that matches the reported premature “announced” status before the parent-visible transcript update has actually drained.

## What This Changes

Request to record subagent completion announce timing at transcript delivery rather than active-run queue acceptance, with separate enqueue/delivery/error/drop metadata.

## Best Possible Solution

Keep this open and fix the announce lifecycle contract so enqueue, transcript delivery, and terminal failure/drop are recorded as distinct phases while preserving `completionAnnouncedAt` as the delivered-time compatibility field.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: active requester completion delivery returns delivered on queue acceptance, and the registry writes `completionAnnouncedAt` from that result. I did not run the live parent/child timing scenario in this read-only review.

## Solution Assessment

Yes, the proposed direction is the right shape: split enqueue and delivered phases and only keep `completionAnnouncedAt` as delivered-time evidence. The implementation should add durable in-flight/dedupe handling so waiting for transcript commit does not create duplicate announces.
