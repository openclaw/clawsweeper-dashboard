## Summary

Keep open: current main and v2026.6.8 still allow normal `chat_type: "group"` topic-scoped Feishu events to choose an `om_*` `rootId` before an `omt_*` `threadId`, and current parsing does not carry the reporter's `topic_id` metadata. Several open Feishu PRs overlap this area, but none is a merged, safe replacement for closing this issue yet.

## What This Changes

The issue reports Feishu `group_topic` session splitting when reply-context messages expose an `om_*` root or reply id instead of the canonical `omt_*` topic id.

## Best Possible Solution

Normalize Feishu topic-message-format reply events to a canonical topic identity only when that mode is reliably identified, preserve existing normal-group root/message-id continuity, and land one reviewed Feishu plugin PR with focused regressions plus redacted live Feishu proof.

## Reproduction Assessment

Yes at source level: current main can still choose `rootId` before `threadId` for normal `chat_type: "group"` topic-scoped events, and it does not parse the reporter's `topic_id` field. I did not run a live Feishu tenant repro in this read-only review.
