## Summary

Keep open: current main and the latest release cover the earlier native `topic_group` fixes, but the reported `chat_type: "group"` reply-context shape can still route by an `om_*` root/reply id instead of the canonical `omt_*` topic id. The closest active fix path is https://github.com/openclaw/openclaw/pull/78927, but that PR has unresolved compatibility and access-order review blockers, so this should stay in maintainer review.

## What This Changes

The issue requests stable Feishu `group_topic` sessions when reply-context messages expose an `om_*` reply/root id while the real topic id is `omt_*`.

## Best Possible Solution

Normalize Feishu topic-message-format reply events to one canonical `omt_*` topic id before session routing, preserve existing OpenClaw-created root-id thread behavior, and land it through the active canonical PR or a narrower replacement with regression coverage.

## Reproduction Assessment

Yes at source level: current main can still choose `root_id` for `chat_type: "group"` topic-scoped events, and it does not parse the reporter's `topic_id` field. I did not run a live Feishu tenant repro in this read-only review.

## Solution Assessment

No, current main is not a complete solution for this shape. The best path is Feishu-plugin-owned normalization with maintainer review for compatibility and hydration ordering, most likely through https://github.com/openclaw/openclaw/pull/78927 or a narrower replacement.
