## Summary

Keep open: current main only gives `thread_id` precedence for `chat_type: "topic_group"`, while the reported reply-context shape says `chat_type: "group"`, so an `om_*` root/reply anchor can still win before the `omt_*` topic key.

## What This Changes

The issue reports that Feishu/Lark `group_topic` sessions still split when reply-context messages carry an `om_*` root or reply anchor instead of the `omt_*` topic id.

## Best Possible Solution

Keep the fix in the Feishu plugin: normalize native topic metadata to one canonical `omt_*` session key for reply-context turns, with regression coverage that preserves normal-group `om_*` reply behavior.

## Reproduction Assessment

Yes, at source level for the reported field class: a `chat_type: "group"` event with both an `om_*` root/reply anchor and an `omt_*` thread id still routes through root-before-thread logic. A live Feishu replay or redacted raw event pair is still needed for high-confidence platform confirmation.

## Solution Assessment

Yes for the direction, but not as a blind thread-id-first swap. The maintainable repair needs a verified topic-group discriminator and focused tests so native topics use `omt_*` while normal group replies keep the existing `om_*` behavior.
