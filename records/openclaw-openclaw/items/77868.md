## Summary

Keep open: current main still lacks a generic plugin API compatibility gate for npm/local installs and non-bundled runtime loading, so a newer official plugin can pass the older `minHostVersion` check even when `openclaw.compat.pluginApi` requires a newer core. The Discord helper mismatch is source-backed, the invalid `visibleReplies` part is already covered by schema validation, and the follow-up msteams comment points to the same remaining cross-plugin gap.

## What This Changes

The issue asks OpenClaw to prevent newer Discord or official plugin packages from loading on older cores and to handle invalid `visibleReplies` config clearly.

## Best Possible Solution

Add a generic plugin API compatibility gate for npm/local install and non-bundled manifest registry loading, reusing the ClawHub range semantics, and avoid a Discord-only fallback that would leave other plugins exposed.

## Reproduction Assessment

Yes, as a source-level reproduction: use an older host version with a plugin package whose `openclaw.compat.pluginApi` requires a newer SDK while `minHostVersion` remains satisfiable, and current install/registry code still admits it. I did not run a live Discord gateway reproduction in this read-only review.

## Solution Assessment

Yes. A generic plugin API compatibility gate is the narrow maintainable fix; a Discord-specific fallback would mask the version-contract failure and would not address the msteams follow-up.
