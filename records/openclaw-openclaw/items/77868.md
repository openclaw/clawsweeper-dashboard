## Summary

Keep open: current main still has the generic non-ClawHub plugin/core compatibility gap. The specific Discord helper now exists and invalid `visibleReplies` values are rejected, but npm/local install and non-bundled registry discovery still gate only `openclaw.install.minHostVersion`, while the issue thread includes an MSTeams follow-up showing this is not Discord-only.

## What This Changes

The issue requests a guard or graceful failure when a plugin requiring a newer OpenClaw plugin API is installed or loaded on an older core.

## Best Possible Solution

Add a generic plugin API compatibility gate for npm/local install and non-bundled runtime registry loading, reusing the existing package metadata and range helper while keeping ClawHub and `visibleReplies` behavior intact.

## Reproduction Assessment

Yes, by source inspection: install or load a non-bundled package whose `openclaw.install.minHostVersion` is compatible but whose `openclaw.compat.pluginApi` exceeds the host. Current npm/local install and registry discovery do not reject that package before runtime load.

## Solution Assessment

Yes. The narrow maintainable solution is a shared plugin API range gate in the generic install and runtime discovery paths, not a Discord-specific fallback that would leave other plugins exposed.
