## Summary

Keep open: current main still has the non-ClawHub compatibility gap described in the prior review and reinforced by the MSTeams follow-up. Matching core/plugin versions now have the Discord progress helper and invalid `visibleReplies` values are rejected, but npm/local install and non-bundled runtime discovery still enforce only `openclaw.install.minHostVersion`, not `openclaw.compat.pluginApi`.

## What This Changes

The issue asks OpenClaw to reject or skip newer plugins when their declared plugin API range is newer than the running core, instead of loading them until delivery crashes.

## Best Possible Solution

Add a generic plugin API compatibility gate for npm/local install and non-bundled runtime loading, using the existing package metadata and range helper, while leaving ClawHub and `visibleReplies` behavior intact.

## Reproduction Assessment

No live install was run, but there is a high-confidence source reproduction: install or load a non-bundled package whose `minHostVersion` is compatible and whose `openclaw.compat.pluginApi` exceeds the host. Current npm/local install and registry discovery do not reject that package before runtime load.

## Solution Assessment

Yes. The narrow maintainable fix is a generic plugin API range gate shared with the existing ClawHub contract, not a Discord-specific fallback that would leave other plugins exposed.
