## Summary

Keep open: current main still has the same non-ClawHub compatibility gap the earlier review identified, and a follow-up comment reports the same pattern on another plugin. The Discord-specific helper now exists when core and plugin versions match, and invalid `visibleReplies` values are already rejected, but npm/local install and non-bundled load paths still do not enforce `openclaw.compat.pluginApi`.

## What This Changes

The issue asks OpenClaw to prevent newer Discord and other plugins from loading on older cores that do not expose the plugin SDK API they require.

## Best Possible Solution

Add a generic plugin API compatibility gate for npm/local install and non-bundled runtime loading, and align official plugin package/catalog metadata so older cores fail closed before loading newer SDK-dependent plugins.

## Reproduction Assessment

No live current-main reproduction was run. The mismatch is source-reproducible: older core lacks the helper, the newer Discord plugin calls it, and current non-ClawHub install/load code still ignores the package's plugin API range.

## Solution Assessment

Yes. A generic compatibility gate is the narrow maintainable solution; a Discord-only fallback would hide the version contract failure and would not protect other plugins called out in the discussion.
