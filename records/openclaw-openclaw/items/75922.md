## Summary

Keep open: the PR addresses real latency regressions, but the current head still has a blocking tool-allowlist correctness/security regression and is missing the required user-facing changelog entry.

## What This Changes

The PR adds a plugin-only tool-construction fast path, separates full-workspace plugin-registry cache entries from scoped loads, and adds latency regression coverage.

## Best Possible Solution

Keep the registry-cache separation, but rework the allowlist fast path around canonical constructed core-tool names, carry core-name conflict protection into plugin-only resolution, and ship focused regressions plus a changelog entry.

## Reproduction Assessment

Yes. Diff inspection on the PR head shows `toolsAllow: ["process"]` or `toolsAllow: ["heartbeat_respond"]` takes the plugin-only path, while current main proves those are core tools and the final exact filter runs after construction.

## Solution Assessment

No. The optimization direction is maintainable, but the hardcoded classifier and missing core-name conflict set are not the safest implementation; deriving/preserving the canonical core tool identity is the narrower fix.
