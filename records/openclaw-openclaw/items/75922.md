## Summary

Keep open. The registry-cache half is directionally reasonable, but the plugin-only shortcut bypasses the current tool allowlist, owner-policy, hook, and abort-wrapper path, so this PR should not merge as-is.

## What This Changes

The PR skips core coding tool construction for plugin-only allowlists, separates full-workspace plugin registry cache entries from scoped loads, and adds latency regression tests.

## Best Possible Solution

Keep the latency optimization only if plugin-only construction still applies the same explicit allowlist expansion, owner-only policy, tool-policy pipeline, schema normalization, before-call hook, and abort wrapping as the normal tool path.

## Reproduction Assessment

Yes. A focused reproduction is to register two non-optional plugin tools, run an embedded attempt with toolsAllow containing only one plugin tool, and assert the new shortcut does not return the other tool; the current diff would return it because the helper only filters optional plugin tools by allowlist.

## Solution Assessment

No. The cache split can remain if tests pass, but the plugin-only branch should reuse or extract the existing policy/wrapper pipeline instead of returning raw plugin tools directly.

## Review Findings

- **[P1] Preserve tool policy on the plugin-only path:** `src/agents/pi-embedded-runner/run/attempt.ts:840`
