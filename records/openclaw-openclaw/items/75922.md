## Summary

Keep open. The PR targets real latency regressions, but exact head still has a blocking tool-identity regression: the new plugin-only fast path uses an incomplete hardcoded core-tool classifier and can skip requested built-in tools or lose core-name conflict context.

## What This Changes

The PR skips core coding-tool construction for plugin-only allowlists, separates full-workspace plugin registry cache entries from scoped loads, and adds latency regression coverage.

## Best Possible Solution

Keep the registry-cache separation, but derive core-tool detection from the canonical tool catalog or constructed core inventory, preserve core-name collision context when resolving plugin-only tools, and add the required changelog entry.

## Reproduction Assessment

Yes. On exact PR head, `toolsAllow: ["process"]` or `toolsAllow: ["heartbeat_respond"]` makes `includeCoreTools` false because those normalized names are missing from the new classifier, and the later allow filter cannot recover skipped core tools.

## Solution Assessment

No. The optimization direction is sound, but the hardcoded classifier and plugin-only collision handling are not the safest maintainable implementation; the fix should reuse canonical core-tool identity and existing conflict guards.
