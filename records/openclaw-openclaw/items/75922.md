## Summary

Keep open. The PR addresses real latency regressions, but the latest head still has a blocking correctness/security issue in the plugin-only fast path and is missing the required changelog entry.

## What This Changes

The PR adds a plugin-only `toolsAllow` fast path for embedded runs, separates full-workspace plugin registry cache entries from scoped loads, and adds latency regression tests.

## Best Possible Solution

Keep the registry-cache separation, derive core-tool detection from the canonical tool catalog or constructed core inventory, preserve existing core-name collision context on the plugin-only path, and add focused regressions plus a changelog entry.

## Reproduction Assessment

Yes. On the PR head, `toolsAllow: ["process"]` or `toolsAllow: ["heartbeat_respond"]` makes `includeCoreTools` false because those normalized names are absent from the new set, and the final allow filter runs only after core construction has been skipped.

## Solution Assessment

No. The optimization direction is reasonable, but the hardcoded classifier is not the narrowest maintainable fix; it should reuse canonical core-tool identity and the existing plugin conflict guard.
