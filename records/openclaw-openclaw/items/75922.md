## Summary

Keep open. The PR is useful and targets real latency paths, but the latest reviewed patch still misclassifies existing core tools and needs a changelog entry before automerge can proceed.

## What This Changes

The PR adds an embedded-run plugin-only tool construction fast path, separates full-workspace plugin registry cache entries from scoped plugin loads, and adds latency regression tests.

## Best Possible Solution

Keep the cache separation and plugin-only optimization, but derive core-tool detection from a canonical built-in tool source, preserve core-name conflict checks, add regressions for `process` and `heartbeat_respond`, and document the user-facing latency fix.

## Reproduction Assessment

Yes. On the reviewed PR head, `toolsAllow: ["process"]` or `toolsAllow: ["heartbeat_respond"]` takes the plugin-only path, skips core construction, and the later exact allow filter cannot recover the requested core tool.

## Solution Assessment

No. The direction is maintainable, but the hardcoded classifier is incomplete; a catalog-derived core-name source or shared helper is the safer fix.
