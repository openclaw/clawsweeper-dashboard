## Summary

Keep open. The registry-cache split is directionally sound, but the PR head still routes allowlists for current core tools such as `process` and `heartbeat_respond` down the plugin-only branch, so exact allowlists can drop the requested core tool; the user-facing changelog entry is also missing.

## What This Changes

This PR adds an allowlist-based plugin-only tool-construction fast path, separates scoped and full-workspace plugin registry caches, and adds latency regression tests.

## Best Possible Solution

Keep the registry-cache separation, derive core-tool detection from canonical constructed tool names or a shared catalog, preserve plugin conflict context on the plugin-only path, and add focused regressions plus the required changelog entry.

## Reproduction Assessment

Yes. On the PR head, `toolsAllow: ["process"]` and `toolsAllow: ["heartbeat_respond"]` both make `includeCoreTools` false because those normalized names are absent from the new set, and the final allow filter runs only after the core tools have been skipped.

## Solution Assessment

No. The optimization direction is reasonable, but a manually maintained classifier is not the safest fix; deriving from the canonical core tool inventory and carrying the existing-name guard into plugin-only resolution is narrower and more maintainable.
