## Summary

Keep open. The latest head addresses the earlier raw plugin-tool bypass by routing plugin-only allowlists through the shared tool policy path, but it now misclassifies the core `process` tool as plugin-only and needs a changelog entry before merge.

## What This Changes

The PR adds a plugin-only tool-construction path, separates full-workspace plugin registry cache entries from scoped loads, and adds latency regression tests.

## Best Possible Solution

Keep the optimization, but make the core-tool classifier complete or derive it from the canonical tool catalog, then add a regression for `toolsAllow: ["process"]` and the required changelog entry.

## Reproduction Assessment

Yes. On PR head, an embedded attempt with `toolsAllow: ["process"]` should produce the core `process` tool, but the new classifier treats that allowlist as plugin-only and the final allow filter removes it.

## Solution Assessment

No. The direction is maintainable, but the hardcoded classifier is incomplete; deriving or testing against the canonical core tool list would be safer than hand-maintaining a partial set.
