## Summary

Keep open. The cache split is directionally aligned with the linked registry-latency report, but the latest PR head still misclassifies current core tools, so exact allowlists such as `process` or `heartbeat_respond` can lose the requested tool before the final allow filter runs.

## What This Changes

The PR adds a plugin-only fast path for embedded attempt tool construction, separates full-workspace plugin loader cache entries from scoped registry loads, and adds latency regression tests.

## Best Possible Solution

Keep the registry-cache separation and plugin-only optimization, but derive the core-tool decision from a canonical built-in tool catalog or shared helper, preserve core-name conflict checks, add `process` and `heartbeat_respond` regressions, and add one Unreleased/Fixes changelog line.

## Reproduction Assessment

Yes. On the latest PR patch, `toolsAllow: ["process"]` or `toolsAllow: ["heartbeat_respond"]` makes the new classifier return `includeCoreTools: false`; the requested core tool is never constructed and the final exact allow filter cannot recover it.

## Solution Assessment

No. The proposed direction is maintainable, but the hand-maintained classifier is incomplete and should be replaced with a catalog-derived or otherwise centrally tested core-tool name source.
