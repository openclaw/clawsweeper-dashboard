## Summary

Keep open: current main still has the linked memory-status bug, but this PR head is conflicting, lacks real behavior proof, and still returns `off` for light-only/REM-only setups while dropping existing deep-promotion diagnostics.

## What This Changes

The PR changes memory-core `memory status` dreaming output to use full dreaming config phase state, adds phase-oriented CLI tests, exposes `status --force`, and trims blank timezone labels in `/dreaming status`.

## Best Possible Solution

Rebase and narrow this PR, or replace it with a focused memory-core status fix that reports effective light/deep/REM phase state while preserving deep-promotion limit/threshold diagnostics and covering light-only, REM-only, global-off, and deep `limit=0` cases.

## Reproduction Assessment

Yes. Source inspection on current main shows `memory status` uses the deep-derived short-term resolver while light and REM phases resolve and run independently, so the light-enabled/deep-disabled false-negative path is high-confidence without running the CLI.

## Solution Assessment

No. Reading the full dreaming config is the right direction, but this PR still gates the summary on deep promotion enablement and removes existing deep-promotion diagnostics, so it is not the narrowest maintainable fix as written.
