## Summary

Keep open. Current main still reports light-only dreaming as off, and the latest PR diff keeps that false negative by gating the new phase summary on deep-promotion enablement while also dropping existing deep limit/threshold troubleshooting details.

## What This Changes

This PR changes memory-core status rendering to use the full dreaming config, adds phase-oriented CLI tests, exposes `memory status --force`, and trims blank timezone labels in `/dreaming status`.

## Best Possible Solution

Revise the memory-core status output to show global dreaming state, effective light/deep/REM phase state, and managed deep-promotion details including disabled reasons such as deep off or `limit=0`, with focused tests and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure global dreaming on with light enabled and deep disabled; current main uses the deep-only resolver and prints `Dreaming: off`, and the PR head still returns `off` through the added `shortTermDreaming.enabled` gate.

## Solution Assessment

No. Consulting the full dreaming config is the right direction, but this PR is not the best solution unchanged because it still hides light-only/REM-only phase state and removes deep limit/threshold context that operators use to debug disabled managed dreaming.
