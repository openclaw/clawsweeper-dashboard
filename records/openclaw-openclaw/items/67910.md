## Summary

Keep open. Current main still has the CLI status behavior this PR is trying to change, but the latest PR diff is not merge-ready: it still returns `Dreaming: off` for light-only/REM-only configurations through the added deep-promotion gate, drops visible deep limit/threshold details, and lacks the required user-facing changelog entry.

## What This Changes

This PR changes memory-core CLI status output to derive dreaming status from the full dreaming config and render light/deep/REM phase toggles, adds `memory status --force`, trims blank timezone suffixes in `/dreaming status`, and adds related tests.

## Best Possible Solution

Revise this PR or replace it with a narrow memory-core status fix that mirrors the gateway Dreams status model: show global dreaming state, light/deep/REM effective phase state and limits, and a separate managed-sweep/deep-promotion status including disabled reasons such as deep off and `limit=0`. Cover light-only, REM-only, global-disabled, deep-disabled, and deep `limit=0` cases, and add the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with global dreaming enabled, light enabled, and deep disabled, current main derives CLI status from the deep-only short-term config and prints `Dreaming: off`; the PR head still returns `off` through the added `shortTermDreaming.enabled` gate.

## Solution Assessment

No. Consulting the full dreaming config is the right direction, but this PR is not the best solution unchanged because it gates phase visibility on deep promotion and removes deep limit/threshold context that current status uses for troubleshooting.
