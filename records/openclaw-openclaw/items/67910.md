## Summary

Keep this PR open: current main still has the linked memory status bug, but this branch still reports light-only and REM-only configs as off, drops useful deep-promotion diagnostics, and lacks required real behavior proof.

## What This Changes

This PR changes memory-core `memory status` dreaming output to use the full dreaming config, adds phase-oriented CLI tests, adds `memory status --force`, and trims blank `/dreaming status` timezone labels.

## Best Possible Solution

Replace or narrow this branch to a focused memory-core status fix that reports effective light/deep/REM phase state while preserving deep-promotion limit and threshold context, with coverage for light-only, REM-only, global-off, deep-off, and deep `limit=0` cases.

## Reproduction Assessment

Yes. Source inspection on current main shows `memory status` uses the deep-derived short-term resolver while light and REM phase configs resolve and run independently, so the reported false-negative path is clear without running the CLI.

## Solution Assessment

No. Reading the full dreaming config is the right direction, but this PR still gates the summary on deep-promotion enablement and removes deep limit/threshold diagnostics needed for accurate status output.
