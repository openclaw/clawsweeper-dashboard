## Summary

Keep this PR open. It addresses a real Telegram formatting bug and is paired with the linked open issue, but the proposed pre-parser normalizer can still mutate indented code blocks and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR adds Telegram-local markdown normalization before Telegram HTML and chunk rendering, with regression tests for bullet-to-numbered-section spacing and fenced code blocks.

## Best Possible Solution

Keep the Telegram-local spacing repair, but make it syntax-aware for fenced and indented code blocks, include regression coverage for both, and add the release-note entry before merge.

## Reproduction Assessment

Yes. The linked issue gives exact input and expected output, and current main routes Telegram markdown sends through markdownToTelegramHtml and markdownToIR without a Telegram boundary normalizer; I did not execute the focused Vitest in this read-only review.

## Solution Assessment

No. A Telegram-local pre-IR normalization is a reasonable boundary, but this implementation is not yet syntax-aware enough because it can insert blank lines into indented code blocks before Markdown parsing.
