## Summary

Keep this PR open: it targets a real Telegram formatting bug, but the patch currently inserts spacing before Markdown parsing without excluding indented code blocks, so it can alter literal code content; it also needs the required user-facing changelog entry.

## What This Changes

The PR adds a Telegram-specific pre-IR normalization that inserts blank lines between bullet-like lines and following numbered or heading lines, plus regression tests.

## Best Possible Solution

Keep the Telegram-local fix, but make the boundary normalizer leave fenced and indented code blocks unchanged, add regression coverage for that case, and add a single-line changelog entry.

## Reproduction Assessment

Yes. The linked issue provides exact input and observed Telegram output, and current main routes outbound Telegram markdown through `markdownToTelegramHtml`; the focused repro is `markdownToTelegramHtml(input, { wrapFileRefs: false })`.

## Solution Assessment

No. A Telegram-local pre-IR normalization is a reasonable boundary, but this implementation should be syntax-aware enough to avoid mutating indented code blocks before Markdown parsing.

## Review Findings
