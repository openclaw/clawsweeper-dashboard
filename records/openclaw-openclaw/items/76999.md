## Summary

Keep this PR open because it targets a valid Telegram formatting bug, but the current patch has a blocking formatter regression: its pre-IR normalizer can mutate indented code blocks before Markdown parsing, and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR adds Telegram-local markdown normalization before HTML and chunk rendering, with regression tests for bullet-to-numbered-section spacing and fenced code blocks.

## Best Possible Solution

Keep the Telegram-local spacing fix, but make the normalizer syntax-aware for both fenced and indented code blocks, add regression coverage for the indented-code case, and add a single-line Unreleased/Fixes changelog entry.

## Reproduction Assessment

Yes. The linked issue provides exact input and expected output, and source inspection shows current main routes outbound Telegram markdown through `markdownToTelegramHtml`; I did not execute the focused Vitest in this read-only review.

## Solution Assessment

No. A Telegram-local pre-IR normalization is a reasonable boundary, but this implementation is not yet syntax-aware enough because it can alter indented code blocks before Markdown parsing.
