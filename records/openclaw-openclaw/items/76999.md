## Summary

Keep open: the PR targets a real Telegram formatting bug and now has real Telegram proof, but the proposed pre-IR normalizer still mutates literal indented code blocks because it only tracks fenced code.

## What This Changes

Adds Telegram-local pre-IR normalization to preserve a blank line between bullet blocks and following numbered or heading lines, with regression tests for the reported spacing and fenced code.

## Best Possible Solution

Keep the Telegram-local spacing fix, but make the normalizer skip both fenced and CommonMark indented code blocks and add regression coverage for the indented-code case before merge.

## Reproduction Assessment

Yes. The linked issue gives an exact input and expected output, and the Crabbox before/after proof shows current main dropping the blank line while this PR head preserves it in a real Telegram session.

## Solution Assessment

No. Telegram-local pre-IR normalization is a reasonable boundary, but this implementation is not yet syntax-aware enough because it can alter indented code blocks before Markdown parsing.
