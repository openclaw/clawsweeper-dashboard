## Summary

Keep open: this PR targets a valid Telegram formatting bug from the linked issue, but the proposed pre-parser normalizer can still mutate literal indented code blocks before Markdown parsing, and the user-facing fix needs a changelog entry.

## What This Changes

The PR adds Telegram-local pre-IR spacing normalization for bullet-to-numbered or heading boundaries in HTML and chunk rendering, plus formatter regression tests.

## Best Possible Solution

Keep the Telegram-local spacing fix, but make the normalizer syntax-aware for fenced and indented code blocks, add regression coverage for the indented-code case, and add one Unreleased/Fixes changelog line.

## Reproduction Assessment

Yes. The linked issue gives exact input and expected output, and current main routes Telegram send and chunk rendering through the formatter under review; the PR regression is source-reproducible with a four-space or tab-indented code block containing bullet-like then numbered-list-like lines.

## Solution Assessment

No. A Telegram-local pre-IR normalization boundary is reasonable, but this implementation is not yet syntax-aware enough because it skips fenced code only and can alter indented code blocks before markdownToIR.
