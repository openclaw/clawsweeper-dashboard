## Summary

Keep open: current main and v2026.6.8 still do not expose per-run token usage as a passive Gateway WebSocket run/lifecycle event. The issue has real third-party client demand and an open overlapping PR, but the remaining work is a public Gateway event-contract decision rather than a conservative cleanup close.

## What This Changes

Expose per-run token usage on the Gateway WebSocket lifecycle event stream so external observers can consume run cost data without polling session APIs.

## Best Possible Solution

Define one canonical Gateway run-usage event contract, then land it through either the existing lifecycle/SDK path or the overlapping agent.run.status PR after identity, correlation, docs, tests, and real Gateway-client proof are complete.

## Reproduction Assessment

Not applicable as a bug reproduction; yes for the product gap by source inspection. Current lifecycle events carry terminal timing/status data, while usage remains exposed through RPC/documented usage APIs rather than a passive run-usage stream event.

## Solution Assessment

Unclear as a final design: a lifecycle usage frame is plausible, but maintainers need to choose whether the canonical contract should extend current agent lifecycle frames, SDK-normalized run events, agent.wait, or the open agent.run.status proposal.
