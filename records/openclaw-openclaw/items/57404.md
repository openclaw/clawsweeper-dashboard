## Summary

Keep open. Current main still has usage data on the run/session side and related session broadcasts/RPC surfaces, but it does not expose the requested passive per-run usage event on the WebSocket lifecycle stream; the closed unmerged implementation and Paperclip incident remain useful input for a Gateway/SDK protocol decision.

## What This Changes

Expose per-run token usage, provider/model, duration, and last-call usage as a WebSocket lifecycle event after agent runs complete.

## Best Possible Solution

Keep this as the focused tracker until maintainers choose an additive Gateway/SDK usage-event contract, then implement it across every run path that produces `agentMeta.usage` with docs and regression coverage.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken existing behavior. Source inspection confirms current main emits lifecycle terminal events and persists usage metadata, but does not emit the requested lifecycle usage event.

## Solution Assessment

Unclear until maintainer protocol review. A passive usage event is plausible, but maintainers should first choose whether usage belongs in a new lifecycle phase, terminal lifecycle metadata, SDK mapping, or a separate event family.
