## Summary

Keep open: current main and the latest release still do not expose per-run token usage as a passive Gateway WebSocket lifecycle/run event, and the remaining work is a public Gateway event-contract decision with real third-party client demand.

## What This Changes

The issue requests a structured per-run token and cost usage event on the Gateway WebSocket lifecycle stream so external observers do not need to poll session usage RPCs after each run.

## Best Possible Solution

Define one canonical Gateway run-usage event contract with run correlation, provider/model, aggregate and last-call usage, terminal-state parity, docs, focused tests, and real external-client proof.

## Reproduction Assessment

Not applicable as a bug reproduction; yes for the product gap by source inspection. Current main emits lifecycle timing/status events and exposes usage through sessions.usage RPCs, but it does not emit a passive per-run usage lifecycle event.

## Solution Assessment

Unclear as a final design. A lifecycle usage frame is plausible, but maintainers need to choose whether the canonical contract extends raw lifecycle frames, SDK-normalized run events, agent.wait, or the overlapping agent.run.status proposal.
