## Summary

Keep open. Current main still does not expose per-run token usage as a passive WebSocket agent lifecycle event; usage exists in session persistence/RPC and session transcript events, but not on the requested lifecycle stream.

## What This Changes

The issue requests a post-run WebSocket lifecycle usage event with accumulated token counts, provider/model, wall-clock duration, and last-call usage.

## Best Possible Solution

Keep this open and land a fresh additive protocol change, likely by reviving or superseding PR #51317 after maintainer sign-off on the usage event shape and coverage plan.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection provide a high-confidence verification path: current main has lifecycle start/end/error and adjacent session usage surfaces, but no lifecycle usage phase.

## Solution Assessment

No, current main is not the best solution because polling session RPCs or listening to session.message events does not satisfy passive agent lifecycle observers. The maintainable path is an additive, documented post-run lifecycle usage event with focused tests across each emitter path.
