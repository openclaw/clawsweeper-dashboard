## Summary

Keep open. Current main still has no passive WebSocket lifecycle usage event: lifecycle docs and emitters cover start/end/error, usage is persisted or exposed through session RPC/OpenAI-compatible surfaces, and the directly related implementation PR #51317 was closed unmerged. Adjacent #48355 may help plugin/channel consumers, but it does not implement this requested Gateway WebSocket lifecycle usage event.

## What This Changes

Expose a post-run WebSocket lifecycle usage event carrying token counts, model/provider identifiers, duration, and last-call usage so observers can track per-run resource consumption without polling session state.

## Best Possible Solution

Land an additive, normalized Gateway event contract for post-run usage after usage metadata is known, preferably through a shared helper used by each agent run path, with protocol docs and focused WebSocket broadcast tests covering accumulated usage, lastCallUsage, provider/model, duration, and session/run identity.

## Reproduction Assessment

Not applicable as a feature request. The absence is high-confidence from source/docs verification: current lifecycle emitters and docs expose start/end/error only, while usage is stored or exposed through other surfaces.

## Solution Assessment

Unclear until maintainer API review. A supplemental lifecycle usage event is a plausible narrow solution, but the best implementation should centralize event construction and settle timing/shape alongside the existing session and plugin event surfaces.
