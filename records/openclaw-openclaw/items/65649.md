## Summary

Keep open. Current main still writes `.clobbered.*` recovery snapshots without a retention cap, so the PR addresses a real remaining disk-fill failure mode, but the proposed count-before-write cap is not merge-ready because overlapping async recoveries can all pass the count check before writing. The current head appears to address the earlier repeated-warning review, but it still needs a serialized or atomic cap plus the required changelog entry.

## What This Changes

The PR adds a 32-file per-config-path cap for config recovery `.clobbered.*` snapshots in both config persistence helpers, extends the observe-recovery fs dependency shape, and adds async/sync regression tests.

## Best Possible Solution

Land a hard per-path clobber snapshot cap in both config persistence paths using a per-config-path critical section or atomic bounded slot claim, keep one bounded warning per path, preserve audit records after snapshots are skipped, add concurrent async regression coverage, and include a single-line changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. Current main can be verified source-level by the direct `.clobbered.*` writes, and the PR’s remaining defect can be reproduced with parallel async suspicious reads against one config path after seeding a last-good backup.

## Solution Assessment

No. The direction is right, but counting existing siblings before a separate async write is not the narrowest reliable solution for a hard cap; the cap check and snapshot claim need to be serialized or made atomic per config path.
