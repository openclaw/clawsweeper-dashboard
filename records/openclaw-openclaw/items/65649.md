## Summary

Keep open. Current main is still uncapped, so this PR is not obsolete, but the latest PR head is not merge-ready because the async cap check and snapshot write are separate operations that can overshoot under concurrent suspicious reads. The user-facing fix also needs changelog coverage before landing.

## What This Changes

The PR adds a 32-file per-config-path cap and per-path warning for `.clobbered.*` config recovery snapshots in `src/config/io.ts` and `src/config/io.observe-recovery.ts`, with async and sync regression tests for repeated suspicious config reads.

## Best Possible Solution

Land a hard per-path clobber snapshot cap in both config persistence paths using a per-config-path critical section or atomic bounded slot strategy, keep one bounded warning per path, preserve audit records after snapshots are skipped, add concurrent async regression coverage, and include the required changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. Current main can be verified source-level by the direct `.clobbered.*` writes, and the PR’s remaining defect can be reproduced with parallel async suspicious reads against one config path after seeding a last-good backup.

## Solution Assessment

No. Counting existing siblings before a separate async write is not a reliable hard cap under concurrency; the cap check and snapshot claim need to be serialized or made atomic per config path.
