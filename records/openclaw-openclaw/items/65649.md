## Summary

Keep this PR open. Current main still writes uncapped `.clobbered.*` snapshots, and the PR targets a valid disk-fill failure mode, but the latest patch still uses a count-then-async-write sequence that can overshoot the advertised per-path cap under overlapping suspicious reads.

## What This Changes

The PR adds a 32-file per-config-path clobber snapshot cap, per-path warning dedupe, `readdir` dependency typing, and async/sync regression coverage in the config recovery paths.

## Best Possible Solution

Land a hard per-path forensic snapshot cap in both config persistence helpers using a per-config-path critical section or atomic bounded slot strategy, keep one bounded warning per path, preserve audit records after snapshots are skipped, and add concurrent async regression coverage.

## Reproduction Assessment

Yes. Current main can be checked source-level by observing the direct `.clobbered.*` writes, and the PR's remaining defect can be reproduced with parallel async suspicious reads against one config path after seeding a last-good backup.

## Solution Assessment

No. Counting existing siblings before a separate async write is not the narrowest reliable way to enforce a hard cap; the count/claim/write path needs to be serialized or made atomic per config path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Serialize the async cap check with snapshot creation:** `src/config/io.observe-recovery.ts:535-542`
  - body: `persistClobberedConfigSnapshot` counts existing siblings and then performs a separate async `writeFile`. If many suspicious reads for the same config path overlap, each call can observe `existing < 32` before any write lands and then create another timestamped snapshot, so the advertised cap can be exceeded. Put the count/claim/write path behind a per-path lock or use atomic bounded slots, and mirror the fix in the parallel `src/config/io.ts` helper.
  - confidence: 0.88
