## Summary

Keep open. The PR targets a real Telegram duplicate-intake area, but the patch drops distinct user messages solely because their body repeats within 60 seconds, while current main already dedupes repeated Telegram deliveries by update/message identity; it also lacks required after-fix real behavior proof.

## What This Changes

Adds a per-processor Telegram inbound cache that drops repeated `RawBody` values for the same chat/topic within 60 seconds and adds Telegram bot-message regression tests.

## Best Possible Solution

Keep the existing update/message-id dedupe; if a real loop remains, add a narrowly keyed guard that distinguishes intentional new messages and uses the resolved Telegram `threadSpec`, backed by live Telegram proof.

## Reproduction Assessment

Yes for the PR-introduced behavior from source: the added test sends two different Telegram message ids with the same text and expects only one dispatch. No live Telegram reproduction or after-fix proof is present.

## Solution Assessment

No. The safer existing solution is update/message-id dedupe in `bot-core`/`bot-updates`; payload-level debouncing needs maintainer product judgment and, if accepted, should use resolved Telegram thread semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not suppress distinct Telegram messages by body:** `extensions/telegram/src/bot-message.ts:140-141`
  - body: This cache ignores `message_id`/`update_id` and only compares `RawBody` within 60 seconds, so two real Telegram messages in the same chat/topic with different message ids but the same text will drop the second agent turn. Repeating a command or short answer is valid user input, and current main already dedupes duplicate deliveries by update/message id.
  - confidence: 0.91
- **[P2] Key topic dedupe from the resolved thread spec:** `extensions/telegram/src/bot-message.ts:137`
  - body: The new key reads raw `message_thread_id` from `primaryCtx`. Existing Telegram routing intentionally ignores that field for non-forum groups and resolves omitted forum thread ids to the General topic, so this can split or miss dedupe for the same OpenClaw conversation if this guard stays.
  - confidence: 0.83
