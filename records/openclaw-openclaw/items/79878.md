## Summary

Keep open: this is a plausible Telegram intake guard, but the current implementation can suppress legitimate group inputs because the duplicate identity is too coarse, and the external PR lacks real behavior proof.

## What This Changes

The branch adds a 60-second in-memory Telegram inbound dedupe cache keyed by chat/topic and RawBody, plus bot-message regression tests.

## Best Possible Solution

Implement dedupe around Telegram retry/update identity or a richer sender-and-media payload key using resolved thread semantics, then add focused tests and real Telegram runtime proof.

## Reproduction Assessment

Yes for the review findings from source inspection: the proposed cache key/value omit sender and media identity, and the topic key reads raw message_thread_id instead of the resolved thread scope. I did not run tests because this review is read-only.

## Solution Assessment

No. The current RawBody-only chat/topic cache is not the narrowest safe solution; retry/update identity or a sender/media-aware key based on resolved thread scope would avoid dropping legitimate inputs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Include sender and media identity in Telegram dedupe:** `extensions/telegram/src/bot-message.ts:137-143`
  - body: The cache key is only chat/topic and the cached value is RawBody, so two different users in the same group/topic saying the same text, or two distinct media messages that both become a placeholder like <media:image>, will drop the second legitimate turn within the window. Scope the duplicate identity to the sender and actual media/file/update identity, or limit this to true Telegram retry identity.
  - confidence: 0.86
- **[P3] Use resolved thread scope for Telegram dedupe keys:** `extensions/telegram/src/bot-message.ts:137`
  - body: The key reads raw message_thread_id from the Telegram update. Existing Telegram thread resolution intentionally ignores that field for non-forum groups because it can be a reply thread rather than a topic, so exact repeats in a regular group can bypass the guard depending on whether one copy was a reply. Build the dedupe key from the resolved context thread scope instead.
  - confidence: 0.79
