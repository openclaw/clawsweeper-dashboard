## Summary

Keep open. Current main already fixed the broad all-lane serial drain from the released code, but the remaining same-lane hung-handler case is still source-reproducible and needs a maintainer decision before adding timeout or retry semantics.

## What This Changes

The issue asks for isolated Telegram polling to avoid letting one slow or hung `handleUpdate()` call freeze spooled update processing.

## Best Possible Solution

Define a bounded same-lane recovery policy for isolated ingress that preserves Telegram sequential semantics, avoids duplicate side effects, and either cancels, quarantines, or explicitly marks wedged updates before later same-lane work proceeds.

## Reproduction Assessment

Yes, from current-main source for the remaining same-lane hang: ordinary messages in one chat share a lane, and that lane is held until the active `handleUpdate()` promise settles. The broader all-lane serial drain from v2026.5.12 is no longer present on current main.

## Solution Assessment

Unclear. The requested timeout direction addresses the symptom, but the simple proposed `Promise.race` plus `continue` is not clearly safe because the underlying handler may keep running and the retained update may be retried.
