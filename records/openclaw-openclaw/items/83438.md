## Summary

Keep open: the PR fixes a real config-validation regression for unrelated discovered plugins, but the patch also makes error diagnostics from explicitly configured `plugins.load.paths` warning-only when the diagnostic has a plugin id.

## What This Changes

The PR changes config plugin validation so registry error diagnostics for discovered plugins not referenced by active plugin config become warnings, keeps explicit plugin references fatal, and adds regression tests plus a changelog entry.

## Best Possible Solution

Keep warning-only behavior for truly unrelated discovered plugins, but keep error diagnostics fatal for configured `plugins.load.paths` as well as entries, allow, deny, and slots, with regression coverage.

## Reproduction Assessment

Yes. Current-main source unconditionally promotes registry error diagnostics to config issues, and the PR body provides before/after terminal output for the unrelated-plugin repro; I did not run the CLI in this read-only review.

## Solution Assessment

No. The direction is good, but the implementation misses `plugins.load.paths`, which is an active config reference and should not be downgraded as unrelated discovery noise.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep load-path plugin errors fatal:** `src/config/validation.ts:957`
  - body: When a plugin is discovered from `plugins.load.paths`, the config itself asked OpenClaw to load that filesystem source. This predicate only treats entries/allow/deny/slots as explicit, so an error diagnostic with a plugin id from a configured load path, such as a missing runtime entry, now becomes a warning and `config validate --json` can return `valid:true`. Please include configured load-path/source matching in the explicit-reference check and add a regression for that case.
