## Summary

Keep open. Current main still has the core denial-loop behavior: denied exec approvals in normal sessions can create a fresh agent followup, node `exec.denied` events enqueue system work and wake the session, and the exact reported Chinese stop phrases are still not recognized. Existing `/stop` cleanup is useful once an abort is detected, but it does not fully cover the reported denial-triggered retry loop.

## What This Changes

The issue requests reliable cancellation and queue handling when users deny exec approvals or send stop commands, including denial circuit breaking, stop-priority handling, visible/cancellable delayed commands, and queue overflow protection.

## Best Possible Solution

Keep this as the denial-loop tracker and land a focused production fix that treats repeated denied exec approvals and explicit stop phrases as high-priority cancellation signals, suppresses or circuit-breaks denied completion followups for the affected session, and clears pending followup/command queues on stop. Broader approved-completion batching and approval-pending pause semantics should remain coordinated with #69570 and #70953.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: deny an exec approval in a normal session and the existing test path still expects a normal agent followup; emit a node `exec.denied` event with a session key and the current server-node-events test expects a queued system event plus heartbeat wake.

## Solution Assessment

No current implementation fully solves this issue. The closed unmerged PR #69387 is not the best remaining solution because it targeted task-flow creation, while current main still has normal-session denied followups and node denied-event wakeups; the safer fix is a narrow denial/abort/queue circuit breaker with regression tests.
