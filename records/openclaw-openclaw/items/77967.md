## Summary

Keep open: current main still has a shared accounting gap where committed message-tool/side-path delivery can be tracked separately from the channel turn result that Feishu logs as `replies=0`, and the related Feishu no-reply issue does not supersede this side-path-satisfied case.

## What This Changes

Request to distinguish Feishu/Lark side-path-satisfied inbound turns from true no-reply turns when an outbound send succeeds but dispatcher final-reply count is zero.

## Best Possible Solution

Introduce a versioned shared turn-result delivery signal that Feishu/Lark completion logging can consume while preserving dispatcher final-reply counts.

## Reproduction Assessment

Yes, at source level: current main can complete a message-tool-only group/channel turn with `queuedFinal=false` and `counts.final=0` while committed messaging-tool delivery evidence is tracked in the agent runner. I did not run a live Feishu WebSocket scenario.

## Solution Assessment

Unclear: distinguishing the states is the right direction, but a Feishu-only log tweak would be too narrow because the missing signal belongs in shared channel-turn accounting semantics.
