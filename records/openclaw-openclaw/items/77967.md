## Summary

Keep open: current main still reports Feishu completion using dispatcher-managed final reply counts, while same-turn message-tool delivery evidence is tracked lower in the agent runtime and is not surfaced as a distinct channel-turn outcome.

## What This Changes

The issue asks OpenClaw to distinguish true no-reply Feishu turns from same-turn side-path/message-tool sends that already succeeded.

## Best Possible Solution

Add a small, versioned channel-turn delivery/accounting signal that carries committed same-turn message-tool delivery evidence into completion diagnostics without redefining dispatcher final-reply counts.

## Reproduction Assessment

Yes from source inspection, not a live Feishu run: current main has a path where message-tool-only delivery suppresses automatic final replies and can return `queuedFinal=false` with `counts.final=0`, while committed message-tool delivery evidence is tracked separately.

## Solution Assessment

Unclear: distinguishing the states is the right direction, but the best API/log shape needs maintainer judgment because dispatch results are shared channel and plugin contract surface.
