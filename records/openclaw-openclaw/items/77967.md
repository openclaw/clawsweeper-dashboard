## Summary

Keep open: current main still has a real accounting gap where successful message-tool side-path delivery is tracked in agent telemetry but not exposed through the inbound dispatch result that Feishu uses for turn closure logging.

## What This Changes

The issue asks OpenClaw to distinguish true no-reply Feishu/Lark turns from same-turn side-path-satisfied turns where `message(action=send)` succeeded.

## Best Possible Solution

Add a generic, additive delivery/accounting signal for successful message-tool side-path sends, then have Feishu and shared turn diagnostics report a side-path-satisfied outcome separately from final-reply delivery.

## Reproduction Assessment

No high-confidence live Feishu reproduction was established in this read-only review. Source inspection does show the gap: message-tool delivery evidence exists in agent telemetry, while Feishu closure logging only reports `counts.final` from a dispatch result that lacks a side-path field.

## Solution Assessment

Unclear until maintainers choose the contract shape. The safest direction is a generic additive dispatch/turn result signal, not a Feishu-only log workaround.
