## Summary

Keep open. Current main still constructs broad plugin metadata and manifest registries before it derives the Gateway startup plugin-id scope, so the reported discovery-time manifest fanout remains source-confirmed; newer related reports add per-request evidence but do not supersede this boot/discovery issue.

## What This Changes

Request discovery-time plugin scoping so allowlist/disabled plugin config prevents reading and validating manifests for unused bundled plugins.

## Best Possible Solution

Thread a conservative plugin-id scope into metadata snapshot and installed-index registry reconstruction before manifest reads, preserving validation, auto-enable, channel/provider/setup, slot, and tool-contract behavior.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main builds the metadata snapshot and manifest registry before `startup.pluginIds` can narrow runtime loading. A live heap profile is only needed to quantify the savings on a real install.

## Solution Assessment

Yes, discovery-time scoping is the right direction, but the implementation should derive a conservative plugin/control-plane scope rather than blindly applying `tools.allow` alone.
