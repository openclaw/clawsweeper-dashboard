## Summary

Keep open. Current main still constructs broad plugin metadata and manifest registries before startup and tool allowlists can narrow the plugin set, and the newer per-chat strace report fits the same source path rather than a resolved or duplicate issue.

## What This Changes

Request discovery-time filtering so disabled or out-of-scope bundled plugin manifests are not read and validated when tool/plugin configuration narrows the active set.

## Best Possible Solution

Thread a conservative startup and tool-contract plugin scope into metadata snapshot and installed-index registry reconstruction so out-of-scope bundled candidates are skipped before package or manifest reads while preserving manifest-first activation semantics.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main builds full metadata/manifest snapshots before deriving startup or tool allowlist scopes, and the latest issue comment adds request-time filesystem evidence for the same path.

## Solution Assessment

Yes, discovery-time scoping is the right direction, but the maintainable version should derive a conservative plugin scope from plugin config, channel/provider activation, and manifest contracts rather than blindly applying only `tools.allow`.

## Repair Work Prompt

Fix openclaw/openclaw#70533 by preventing broad bundled plugin manifest/package reads when the startup or tool-control-plane scope is already narrow. Derive a conservative plugin scope for metadata snapshot / installed-index manifest registry reconstruction and thread it through the existing `pluginIds` seam in `loadPluginManifestRegistryForInstalledIndex`; also cover the per-request plugin tool path so `tools.allow` / plugin groups do not force a full metadata scan before `onlyPluginIds` is known.
