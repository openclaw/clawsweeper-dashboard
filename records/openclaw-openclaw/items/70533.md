## Summary

Keep open. Current main still constructs the plugin metadata snapshot and installed-index manifest registry before a startup/control-plane plugin scope is available, so unallowed or disabled bundled plugin manifests can still be materialized on the path reported here.

## What This Changes

The issue asks OpenClaw to apply a configured plugin/tool scope before reading and validating bundled plugin manifests, reducing boot and request-time discovery overhead.

## Best Possible Solution

Thread a conservative plugin-id scope into metadata snapshot and installed-index registry reconstruction before manifest reads, while preserving existing activation and validation semantics.

## Reproduction Assessment

Yes, at source level: loadPluginLookUpTable builds the metadata snapshot before startup.pluginIds exists, and the snapshot calls the installed-index manifest registry without pluginIds. A live profile would quantify current RSS and latency, but the broad read path is still present on current main.

## Solution Assessment

Yes, discovery-time scoping is the right direction, but the implementation should derive a conservative startup/control-plane plugin scope from current activation rules rather than applying tools.allow alone.
