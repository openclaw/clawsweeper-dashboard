## Summary

Keep open. Current main still materializes the installed-index manifest registry without a plugin-id scope before startup/plugin allowlist scope is available, and the latest issue comments add current-release evidence that pruning the bundled tree still reduces the reported cost.

## What This Changes

The issue asks OpenClaw to apply plugin allowlist scope before bundled plugin manifest reads and validation so disabled or unallowed bundled plugins do not dominate startup and request-time discovery cost.

## Best Possible Solution

Thread a conservative plugin-id scope derived from existing plugin activation and startup policy into metadata snapshot and installed-index registry reconstruction before manifest reads, while preserving manifest-first validation for plugins that remain in scope.

## Reproduction Assessment

Yes at source level: current main still builds the manifest registry for the full installed index before startup/plugin allowlist scope is available. I did not run a live heap profile in this read-only review.

## Solution Assessment

Yes, discovery-time scoping is the right fix direction, but it should use the existing plugin activation and startup policy rather than a literal `tools.allow`-only filter.

## Repair Work Prompt

Fix the plugin discovery performance bug tracked at https://github.com/openclaw/openclaw/issues/70533. Current main builds plugin metadata snapshots by calling `loadPluginManifestRegistryForInstalledIndex` with `includeDisabled: true` and no `pluginIds` scope, so unallowed or disabled bundled plugin manifests can still be materialized before startup scope is known. Thread a conservative plugin-id scope derived from existing plugin activation/startup policy into metadata snapshot and installed-index registry reconstruction before manifest reads. Do not add a new config option, persistent metadata cache, public API break, or runtime-load shortcut. Preserve manifest-first behavior and plugin allow/deny/entry semantics. Add focused regression coverage showing a restrictive allowlist/startup scope passes `pluginIds` into the installed-index registry and does not read/load manifests outside that scope while retaining required startup/control-plane plugin ids. Add a changelog entry unless maintainers classify the final patch as purely internal.
