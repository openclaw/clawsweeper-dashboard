## Summary

Keep open. Current main still builds the broad metadata snapshot and manifest registry before deriving the Gateway startup plugin-id scope, while the issue discussion includes concrete heap and per-chat syscall evidence; related open reports cover request-time rescans but do not supersede this boot/discovery scoping bug.

## What This Changes

Apply plugin/config scoping before discovery-time manifest reads so Gateway startup does not load every bundled plugin manifest when only a small plugin/tool set is enabled.

## Best Possible Solution

Thread a conservative plugin-id scope into metadata snapshot and installed-index registry reconstruction before manifest reads, while preserving activation, channel/provider/setup, slot, tool-contract, and validation semantics.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main builds the metadata snapshot and manifest registry before startup.pluginIds can narrow runtime loading; a live heap profile would only quantify the current savings.

## Solution Assessment

Yes. Discovery-time scoping is the right fix direction, but it should derive a conservative plugin/control-plane scope rather than blindly applying tools.allow alone.
