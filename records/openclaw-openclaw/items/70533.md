## Summary

Keep open. Current main still builds broad plugin metadata and installed-index manifest registries before deriving the Gateway startup plugin-id scope; the related open tracker https://github.com/openclaw/openclaw/issues/78461 covers broader hot-path rescans, but it does not replace this boot/discovery allowlist-scoping bug.

## What This Changes

Apply configured plugin scoping during plugin metadata discovery so omitted bundled plugins are not read and validated at Gateway boot.

## Best Possible Solution

Thread a conservative startup/control-plane plugin-id scope into metadata snapshot and installed-index registry reconstruction before manifest reads while preserving plugin activation and validation semantics.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main loads the metadata snapshot and unscoped installed-index manifest registry before startup.pluginIds exists; a live profile would only quantify RSS and latency.

## Solution Assessment

Yes. Discovery-time scoping is the right direction, but the implementation should derive a conservative plugin/control-plane scope rather than blindly applying tools.allow alone.

Current main builds loadPluginMetadataSnapshot/loadPluginManifestRegistryForInstalledIndex with includeDisabled: true and no pluginIds, then derives startup.pluginIds afterward. Derive a conservative plugin/control-plane scope early enough to pass into the installed-index manifest registry reconstruction before manifest reads. Preserve plugin auto-enable, configured channel/provider/setup ownership, slots, plugin config validation, diagnostics, workspaceDir/plugins.load.paths compatibility, and existing manifest freshness rules. Check overlap with https://github.com/openclaw/openclaw/issues/78461 and keep that broader rescan issue coordinated rather than duplicating cache policy.
