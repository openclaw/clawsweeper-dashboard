## Summary

Keep open. Current main still builds broad plugin metadata and manifest registries before it derives the startup/runtime plugin id scope, so the reported discovery-time cost is still source-confirmed.

## What This Changes

Request discovery-time filtering so restrictive plugin/tool allowlists do not force all bundled plugin manifests to be read and validated at gateway boot.

## Best Possible Solution

Thread a conservative startup/control-plane plugin scope into metadata snapshot and installed-index registry reconstruction so out-of-scope bundled candidates are skipped before package or manifest reads, while preserving channel, provider, slot, config-path, and tool-contract activation behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main constructs the metadata snapshot and manifest registry before `startup.pluginIds`/`onlyPluginIds` can narrow runtime loading; a live heap profile is only needed to quantify the final savings.

## Solution Assessment

Yes, discovery-time scoping is the right fix direction, but it should derive a conservative startup/plugin-contract scope instead of blindly applying `tools.allow` alone.
