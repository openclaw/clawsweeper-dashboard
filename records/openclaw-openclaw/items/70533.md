## Summary

Keep open. Current main still builds the plugin metadata snapshot and installed-index manifest registry before a startup/plugin allowlist scope is available, so the related memoization fixes reduce repeated work but do not solve the first unscoped manifest materialization reported here.

## What This Changes

Request discovery-time scoping so restrictive plugin allowlists avoid reading and validating manifests for unneeded bundled plugins.

## Best Possible Solution

Thread a conservative early plugin-id scope into metadata snapshot and installed-index manifest reconstruction before manifest reads, while keeping full-catalog paths unscoped where the UI/CLI genuinely needs discovery.

## Reproduction Assessment

Yes, at source level: current main still calls loadPluginMetadataSnapshot before startup pluginIds exist, and that snapshot calls the installed-index manifest registry with includeDisabled: true but no pluginIds. A live profile would quantify the current RSS/latency impact, but the broad read path remains visible in source.

## Solution Assessment

Yes, discovery-time scoping is the right fix direction, but it should derive a conservative startup/control-plane scope from existing activation rules rather than applying tools.allow or plugins.allow alone.
