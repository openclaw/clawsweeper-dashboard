## Summary

Keep open. Current main still builds the plugin metadata snapshot and installed-index manifest registry before a narrowed startup plugin scope is available, so bundled plugin manifests outside `plugins.allow` can still be materialized on the reported path.

## What This Changes

The issue asks OpenClaw to apply plugin allowlist scope before bundled plugin manifest read and validation so unallowed plugins do not dominate gateway startup or request-time memory and CPU cost.

## Best Possible Solution

Carry a conservative plugin-id scope into plugin metadata snapshot and installed-index registry reconstruction before manifest materialization, while preserving activation, validation, setup, and registry freshness semantics.

## Reproduction Assessment

Yes, at source level: current main derives startup plugin ids only after `loadPluginMetadataSnapshot` has already built an unscoped manifest registry. The reporter comments provide live heap, strace, and pruning evidence, but I did not run a local heap profile in this read-only review.

## Solution Assessment

Yes, discovery-time scoping is the right direction, but the scope should come from current plugin activation/startup planning rather than applying `tools.allow` literally. The existing `pluginIds` filter in `loadPluginManifestRegistryForInstalledIndex` is the maintainable seam to reuse.
