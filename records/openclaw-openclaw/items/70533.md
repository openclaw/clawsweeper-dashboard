## Summary

Keep open. Current main still builds the plugin metadata snapshot and installed-index manifest registry before it knows the gateway startup plugin scope, so disabled or unallowed bundled plugin manifests can still be materialized during the control-plane path.

## What This Changes

The issue asks OpenClaw to apply plugin allow/enable scoping before manifest discovery so disabled bundled plugin manifests are not read and validated at gateway boot.

## Best Possible Solution

Thread a conservative startup/control-plane plugin-id scope into metadata snapshot and installed-index registry reconstruction before manifest reads, while preserving existing plugin activation and validation semantics.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main builds the metadata snapshot and unscoped installed-index manifest registry before startup.pluginIds exists. A live profile would quantify RSS and latency, but the call chain still matches the report.

## Solution Assessment

Yes, discovery-time scoping is the right direction, but it should derive a conservative plugin/control-plane scope from the current activation rules rather than blindly applying tools.allow alone.
