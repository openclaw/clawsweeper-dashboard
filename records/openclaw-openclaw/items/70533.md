## Summary

Keep open. Current main still builds a broad plugin metadata snapshot and manifest registry before applying the gateway startup plugin scope, so the reported discovery-time manifest/materialization cost is still present; the latest runtime-deps optimization only avoids a different manifest scan in the dependency staging path.

## What This Changes

The issue asks OpenClaw to apply restrictive plugin startup scoping before bundled plugin manifest/package reads and validation during gateway startup.

## Best Possible Solution

Add a conservative startup plugin scope before installed-index manifest registry reconstruction and bundled directory discovery, thread it through metadata snapshot/manifest-registry APIs, and skip out-of-scope bundled candidates before package or plugin manifest reads while preserving explicit channel, provider, slot, and config-activation behavior.

## Reproduction Assessment

Yes for the control-flow bug: current main shows gateway startup materializing the metadata snapshot and manifest registry before `onlyPluginIds` limits runtime loading. The exact heap magnitude still needs a live profile to quantify after a fix.

## Solution Assessment

Yes, discovery-time scoping is the right direction, but the maintainable fix should derive a conservative plugin startup scope rather than blindly using `tools.allow`; OpenClaw must still include plugins activated by channel config, providers, slots, and auto-enable rules.
