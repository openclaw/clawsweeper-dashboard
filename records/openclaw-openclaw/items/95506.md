## Summary

Keep open: current main and the latest release still key bundled MCP runtimes by session id, so the identical workspace/config fan-out remains source-reproducible. ## What This Changes

Request to avoid duplicate bundled MCP runtimes, catalogs, stdio server sessions, and `tools/list` calls when multiple sessions use the same workspace and MCP config.

## Best Possible Solution

Keep this issue open until maintainers choose and land one shared-runtime implementation, preserving `session` as the default and requiring explicit coverage for fan-out, leases, idle sweep, config invalidation, disposal, and shared-server failure behavior.

Yes at source level: current main still keys bundled MCP runtime ownership and in-flight creation by session id, while each runtime owns its own catalog and MCP server sessions. I did not run the temporary stdio fan-out test because this review is read-only.
