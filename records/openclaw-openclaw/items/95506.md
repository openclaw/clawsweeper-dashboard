## Summary

Keep open: current main still keys bundled MCP runtimes by session id, so the reported duplicate workspace/config fan-out remains real, and the protected `maintainer` label plus cross-session runtime-boundary tradeoff require explicit maintainer handling.

## What This Changes

Requests sharing MCP runtime, catalog, and transport ownership for identical workspace/config sessions instead of duplicating that work per session.

## Best Possible Solution

Adopt a maintainer-approved shared-runtime or pooling contract with tests for fan-out, active leases, idle sweep, config invalidation, server crash/disposal, and redacted diagnostics before changing the default runtime boundary.

## Reproduction Assessment

Yes at source level: current main keys the manager by session id and stores catalog/stdio sessions inside each runtime, so identical workspace/config but different session ids create distinct runtime/catalog state. I did not run the reporter's temporary stdio fan-out test in this read-only review.

## Solution Assessment

Unclear as a final product direction. Workspace/config sharing solves the duplicate fan-out, but it changes the session isolation boundary, so the best path is an explicit maintainer-approved shared mode or pooling contract rather than an unreviewed default change.
