## Summary

Keep open: current main and the latest release still key bundled MCP runtimes by session id, so identical workspace/config sessions can fan out into duplicate runtime/catalog/stdio work; the protected maintainer label, open candidate PR, and cross-session isolation tradeoff require maintainer handling.

## What This Changes

Request to collapse duplicate bundled MCP runtimes, catalogs, and stdio sessions when multiple sessions use the same workspace and MCP config.

## Best Possible Solution

Use a maintainer-approved shared-runtime or pooling contract, likely by rebasing and reviewing https://github.com/openclaw/openclaw/pull/79882 or a replacement, while preserving per-session cleanup, active leases, idle sweep, config invalidation, disposal, diagnostics redaction, and shared-server failure behavior.

## Reproduction Assessment

Yes at source level: current main keys MCP runtimes and in-flight creation by session id, and each runtime owns its own catalog plus MCP server sessions. I did not run the temporary stdio fan-out test because this review is read-only.

## Solution Assessment

Unclear as final product direction. Workspace/config sharing addresses the duplication, but it changes session isolation and lifecycle semantics, so maintainers need to choose the approved pooling boundary and whether it is default or opt-in.
