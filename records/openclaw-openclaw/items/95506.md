## Summary

Keep open: current main and v2026.6.9 still key bundled MCP runtimes by session id, while the direct shared-runtime candidate at https://github.com/openclaw/openclaw/pull/79882 is open but conflicting and still needs maintainer acceptance of the cross-session isolation tradeoff.

## What This Changes

Request sharing or pooling bundled MCP runtime, catalog, and stdio transport ownership for identical workspace/config sessions instead of creating one session-scoped runtime per session.

## Best Possible Solution

Use a maintainer-approved shared-runtime or pooling contract, likely by rebasing and reviewing https://github.com/openclaw/openclaw/pull/79882 or a replacement, with explicit coverage for fan-out, leases, idle sweep, config invalidation, disposal, and shared-server failure behavior.

## Reproduction Assessment

Yes at source level: current main keys MCP runtimes and in-flight creation by session id, and each runtime owns its own catalog plus MCP server sessions. I did not run the temporary stdio fan-out test because this review is read-only.

## Solution Assessment

Unclear as final product direction. Workspace/config sharing addresses the duplication, but it changes session isolation and lifecycle semantics, so maintainers need to choose default pooling versus an explicit opt-in shared mode.
