## Summary

Keep open: current main and the latest release still key bundled MCP runtimes by session id, so the identical workspace/config fan-out remains source-reproducible; the protected maintainer label, open candidate PR, and cross-session runtime-isolation tradeoff require maintainer handling.

## What This Changes

The issue requests reducing duplicate MCP runtime, catalog, and stdio server ownership when multiple sessions use the same workspace and identical MCP config.

## Best Possible Solution

Use a maintainer-approved shared-runtime or pooling contract, likely by rebasing and reviewing https://github.com/openclaw/openclaw/pull/79882 or a replacement, with explicit tests for fan-out, active leases, idle sweep, config invalidation, disposal, and shared-server failure behavior.

## Reproduction Assessment

Yes at source level: current main keys MCP runtimes and in-flight creation by session id, and each runtime owns its own catalog plus MCP server sessions. I did not run the temporary stdio fan-out test because this review is read-only.

## Solution Assessment

Unclear as final product direction. Workspace/config sharing addresses the duplication, but it changes session isolation and lifecycle semantics, so maintainers need to choose default pooling versus an explicit opt-in shared mode.
