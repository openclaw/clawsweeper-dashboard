## Summary

Keep open: current main and v2026.6.9 still key bundled MCP runtimes and in-flight creation by session id, so identical workspace/config sessions still fan out into separate runtime/catalog/stdio ownership; the protected maintainer label and open shared-runtime PR make this a maintainer decision, not cleanup-closeable.

## What This Changes

The issue requests collapsing duplicate bundled MCP runtime, catalog, and stdio session ownership for sessions that share the same workspace and MCP config fingerprint while preserving session lifecycle semantics.

## Best Possible Solution

Use a maintainer-approved shared-runtime or pooling contract, likely by rebasing and reviewing https://github.com/openclaw/openclaw/pull/79882 or a replacement, with explicit tests for fan-out, leases, idle sweep, config invalidation, disposal, and shared-server failure behavior.

## Reproduction Assessment

Yes at source level: current main keys MCP runtime objects and in-flight creation by session id, while each runtime owns its own catalog and MCP server sessions. I did not run the temporary stdio fan-out test in this read-only review.

## Solution Assessment

Unclear as final product direction. Workspace/config sharing directly addresses the duplication, but the safe product shape needs maintainer approval because it changes session isolation and likely needs an explicit opt-in config contract.
