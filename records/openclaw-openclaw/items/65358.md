## Summary

Keep open. Current main has useful mitigations around manual backups, cron-store repair, memory diagnostics, and Gateway session mutation RPCs, but the central upgrade-safety request is still unresolved and the related upgrade-amnesia report remains open and corroborated.

## What This Changes

The issue requests stronger upgrade safety for OpenClaw state, including automatic pre-update backups, rollback/restore support, stale session override repair, memory continuity diagnostics, cron validation, and per-session reset tooling.

## Best Possible Solution

Add an opt-out pre-update state snapshot using the existing backup planner, provide a supported restore/rollback path for state and workspace data, expose operator-facing session reset/delete/clear-override tooling, and extend doctor/status to flag stale session overrides plus missing canonical `MEMORY.md` when daily memory exists. Cron/model/auth migration validation should be an explicit post-update health check with actionable errors.

## Reproduction Assessment

Partly: the original live 2026.4.10 two-host failure was not replayed, but the remaining gaps are reproducible by source inspection on current main: update has no automatic state backup, sessions CLI lacks targeted repair, and reset still preserves legacy model overrides. The user-visible cron failure needs a narrower fixture or logs to reproduce with full confidence.

## Solution Assessment

No: current main is useful mitigation, not the best complete solution for this issue. The maintainable path is a small set of owner-reviewed upgrade-safety and repair flows that preserve deliberate user selections while giving operators a supported way to diagnose or clear stale state.
