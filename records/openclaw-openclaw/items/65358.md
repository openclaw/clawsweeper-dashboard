## Summary

Keep open. Current main has partial mitigations for backups, startup memory, cron payload migration, and stale auto-created session route state, but it still does not implement the reported upgrade-safety outcome across automatic state backup/restore, rollback, legacy session override repair, targeted CLI session reset, and post-update memory/auth/cron diagnostics.

## What This Changes

The issue asks for upgrade-safety hardening around memory continuity, session override repair, rollback and backup, cron validation, and per-session reset tooling after a 2026.4.10/2026.4.11 regression.

## Best Possible Solution

Define a maintainer-owned upgrade-safety plan, then ship a cohesive flow for verified state backups, retention/restore, rollback, stale-session repair, per-session reset, and post-update memory/auth/cron diagnostics.

## Solution Assessment

No, not as a one-off patch. The narrowest maintainable direction is a scoped maintainer design across updater, backup/restore, sessions, doctor, cron, auth, and Control UI recovery rather than manual JSON surgery or a single stale-override special case.
