## Summary

Keep open. Current main has partial mitigations, but it still does not fully solve the reported upgrade-safety failure across automatic state backup/restore, rollback, stale legacy session overrides, targeted CLI repair, and post-update diagnostics; related safe-update and sibling upgrade reports remain open rather than superseding this issue.

## What This Changes

Request to harden OpenClaw upgrades so memory, workspace, session, cron, and auth state survive updates, with automatic backup/rollback, session override validation, clearer diagnostics, and targeted session reset tooling.

## Best Possible Solution

Ship a maintainer-scoped upgrade-safety design that combines update-tied verified backups, state restore/rollback, stale session override repair policy, targeted session CLI repair, and post-update doctor checks for workspace memory, cron, and model/auth health.

## Reproduction Assessment

No high-confidence end-to-end replay is available for the original multi-host 2026.4.10/2026.4.11 upgrade. Source inspection does reproduce important remaining gaps: update paths lack automatic backup/restore, reset preserves legacy override records, and the CLI has no targeted stale-session repair command.

## Solution Assessment

No as a one-off repair. The maintainable fix is a coordinated upgrade-safety and repair flow, not manual JSON surgery or a narrow patch that only clears one reported stale override shape.
