## Summary

Keep open. Current main has useful partial mitigations for manual backups, startup memory preload, cron payload normalization, and auto-created route-state cleanup, but it still does not provide the reported upgrade-safety outcome for automatic state backup/restore, rollback, legacy session override repair, targeted CLI session reset, and post-update memory/auth/cron diagnostics.

## What This Changes

The issue requests upgrade-safety hardening for memory/session continuity, stale session/provider/auth override repair, backup/rollback, cron validation, and per-session reset tooling after a 2026.4.10 update.

## Best Possible Solution

Define a maintainer-owned upgrade-safety plan, then ship a cohesive flow for verified state backups, retention/restore, rollback, stale-session repair, per-session reset, and post-update memory/auth/cron diagnostics.

## Reproduction Assessment

No high-confidence end-to-end reproduction exists for the original multi-host upgrade. Source inspection does confirm current-main gaps around safe-update flags, restore/retention, legacy override handling, targeted CLI session repair, and update-coupled diagnostics.

## Solution Assessment

No, not as a one-off patch. The maintainable direction is a scoped maintainer design across updater, backup/restore, sessions, doctor, cron, auth, and Control UI recovery rather than manual JSON surgery or a single stale-override special case.
