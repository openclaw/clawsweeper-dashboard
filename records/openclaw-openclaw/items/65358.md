## Summary

Keep open. Current main has partial mitigations, but the central upgrade-safety problem remains unresolved: update paths still do not create a state/workspace snapshot, rollback remains code-version pinning rather than state restore, the CLI lacks a first-class single-session reset/delete/clear-overrides command, and legacy session model overrides are still deliberately preserved.

## What This Changes

Harden upgrades so memory, session stores, gateway config, auth/provider state, cron jobs, and dashboard/session recovery survive version updates without manual session-store edits.

## Best Possible Solution

Ship an opt-out pre-update backup using the existing backup planner, add a supported restore/rollback path for state and workspace data, expose CLI session reset/delete/clear-overrides tooling, and extend doctor/status to flag stale overrides plus missing root MEMORY.md when daily memory exists. Cron/model/auth validation should run as explicit post-update health checks with actionable errors.

## Reproduction Assessment

Partly. The original multi-host upgrade cannot be replayed from the report alone, but the remaining gaps are reproducible by inspecting current main: update paths lack backup hooks, sessions CLI lacks targeted repair commands, reset preserves legacy overrides, and diagnostics remain partial.

## Solution Assessment

No. Current main provides useful mitigations, but the best solution is a set of targeted upgrade-safety and repair flows rather than relying on manual JSON edits or preserving all legacy overrides indefinitely.
