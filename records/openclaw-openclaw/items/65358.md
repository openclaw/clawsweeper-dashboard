## Summary

Keep open. Current main has useful partial mitigations, but the central upgrade-safety problem is still unresolved across automatic state backup, state rollback, targeted session repair, legacy override semantics, and post-update diagnostics.

## What This Changes

Requests hardening for 2026.4.10 upgrade regressions affecting workspace memory continuity, stale session model/provider/auth overrides, cron agentTurn behavior, rollback, and per-session repair tooling.

## Best Possible Solution

Ship an explicit upgrade-safety design: automatic verified pre-update backup, supported state restore/rollback, targeted session reset/delete/clear-overrides tooling, stale override validation, and post-update doctor checks for workspace memory plus cron/model health.

## Reproduction Assessment

Partly. The original multi-host 2026.4.10/2026.4.11 upgrade cannot be replayed from the report alone, but current source inspection reproduces key remaining gaps: update paths lack backup hooks, the sessions CLI lacks targeted repair commands, and reset logic preserves legacy overrides without a source marker.

## Solution Assessment

No. Current main mitigates some related symptoms, but the best solution is a coordinated upgrade-safety and repair flow rather than manual JSON edits or a narrow one-off change to one symptom.
