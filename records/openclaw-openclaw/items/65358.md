## Summary

Keep open. Current main has useful partial mitigations for update verification, manual backup, stale auto session route repair, and cron payload migration, but it still does not fully address the reported upgrade-continuity failure across automatic state backup/restore, rollback, explicit or legacy session override recovery, targeted session reset tooling, and post-update diagnostics.

## What This Changes

The issue requests hardening for a 2026.4.10/2026.4.11 upgrade regression involving memory continuity, Telegram session overrides, auth/provider recovery, cron jobs, diagnostics, backups, rollback, and targeted session reset tooling.

## Best Possible Solution

Design and ship a maintainer-scoped upgrade-continuity flow that combines verified pre-update backups, restore or rollback semantics, stale session override repair policy, targeted per-session repair/reset UX, and post-update doctor checks for workspace memory, cron, and model/auth health.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available for the original multi-host upgrade. Source inspection does show important current-main gaps in update-coupled backup/restore, rollback, explicit or legacy session override recovery, and targeted session repair tooling.

## Solution Assessment

No as a one-off repair. The maintainable solution is a coordinated upgrade-safety and recovery design, not manual JSON surgery or a narrow patch that only clears one stale override shape.
