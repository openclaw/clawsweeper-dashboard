## Summary

Keep open: the PR is a plausible focused implementation for the external wake/sessionKey gap, but it should not merge yet because the supplied proof only exercises helper functions, exact-head CI has failing checks, and the new `--session-key` timing behavior is not reflected in the user docs.

## What This Changes

The PR adds an optional typed `sessionKey` to wake params, threads it through cron wake routing, adds `openclaw system event --session-key`, and updates docs, tests, changelog, and generated Swift protocol models.

## Best Possible Solution

Ship a session-targeted wake/CLI contract after maintainer acceptance of the public timing semantics, docs that match the immediate targeted behavior, green exact-head checks, and redacted runtime proof from a running Gateway.

## Reproduction Assessment

Yes for the source-level gap: current main has no typed wake `sessionKey`, no `openclaw system event --session-key`, and cron wake does not pass a session target. I did not establish a live Telegram/WhatsApp delivery failure path on current main.

## Solution Assessment

Mostly yes for the narrow external sessionKey slice, but not ready as-is. The code direction is maintainable, while proof, CI, and user-facing timing docs still need to be resolved; https://github.com/openclaw/openclaw/pull/50818 remains the separate internal enqueue remap slice.
