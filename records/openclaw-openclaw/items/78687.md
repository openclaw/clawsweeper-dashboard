## Summary

Keep open for https://github.com/openclaw/openclaw/pull/78687: the additive wake `sessionKey` direction is useful, but the latest head still introduces a P2 cron wake-target regression.

## What This Changes

The PR adds typed wake `sessionKey` support, an `openclaw system event --session-key` flag, cron wake plumbing, docs, tests, a changelog entry, and generated Swift protocol model updates.

## Best Possible Solution

Keep the additive protocol and CLI surface, but repair the cron adapter so omitted wake targets remain untargeted fanout or so call sites with a known job agent pass that agent explicitly.

## Reproduction Assessment

Yes for source-level reproduction: PR head routes omitted wake targets through `resolveCronAgent(undefined)`, while current heartbeat-runner behavior fans out only when no target is supplied. The existing failure-alert fallback still creates exactly that omitted-target wake after enqueueing for a specific job agent.

## Solution Assessment

No as written: adding typed `sessionKey` to wake and the CLI is a maintainable direction, but converting omitted cron wakes into default-agent targeted wakes changes existing multi-agent behavior. The safer fix is to preserve the no-target branch or pass a known `agentId` from the specific cron call site that owns it.

## Review Findings
