## Summary

Keep this PR open: the additive wake `sessionKey` and CLI surface are useful and have real behavior proof, but the patch still has a drain-safety bug for targeted wakes under `session.scope: "global"`.

## What This Changes

The PR adds optional `sessionKey` support to the wake protocol, `openclaw system event`, cron wake routing, Swift protocol models, docs, changelog, and focused tests.

## Best Possible Solution

Land the additive wake protocol and CLI targeting surface after fixing global-scope routing and coordinating the broader issue closure with the companion internal remap PR.

## Reproduction Assessment

Yes. Current main source shows no typed wake `sessionKey` or CLI flag, and the PR bug is source-reproducible because global-scope heartbeat resolution drains `global` while the new targeted path can enqueue under an agent-prefixed key.

## Solution Assessment

No, not yet. The additive field and flag are the right shape, but the cron adapter needs to reject or canonicalize targeted session keys under global scope so enqueue and wake inspect the same queue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep global-scope wake queues drainable:** `src/gateway/server-cron.ts:287-290`
  - body: When `session.scope` is `"global"`, the heartbeat runner ignores forced session keys and always peeks the literal `global` queue. This new targeted wake path can still enqueue the event under a requested agent-prefixed session key, then request a heartbeat that drains `global`, so `openclaw system event --session-key ...` can return `{ ok: true }` while leaving the event stranded. Please reject non-global targets in global scope or canonicalize both enqueue and wake to the same `global` key, with a regression test.
  - confidence: 0.89
