## Summary

Keep open: current main still has the source-level session-targeting gap, but this PR still turns omitted cron wake targets into default-agent wakes and the supplied proof only exercises helper functions, not the changed CLI/RPC/Gateway path.

## What This Changes

The PR adds optional `sessionKey` support to the wake schema and `openclaw system event`, threads it through cron wake routing, updates docs/changelog/tests, and regenerates the Swift Gateway model.

## Best Possible Solution

Land a focused wake protocol/CLI session-key path that preserves untargeted heartbeat broadcast semantics, adds coverage for targeted and omitted wake targets, and includes redacted real Gateway proof.

## Reproduction Assessment

Yes for the source-level gap: current main's CLI sends only `{ mode, text }`, `WakeParamsSchema` lacks `sessionKey`, and the wake handler forwards only `mode`/`text`. I did not live-reproduce the original Telegram or external async-completion loss.

## Solution Assessment

No as written: the protocol/CLI direction is plausible, but PR head changes omitted cron wake targets into default-agent targeted wakes and can strand non-default cron failure alerts. The safer fix is to keep no-target wakes untargeted or pass the owning agent at existing call sites, then prove the actual Gateway path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve untargeted cron wake broadcasts:** `src/gateway/server-cron.ts:202-203`
  - body: `resolveCronWakeTarget` now calls `resolveCronAgent` even when neither `agentId` nor `sessionKey` is supplied, so an omitted wake target becomes the configured default agent. Keep the no-target case untargeted, or pass the job agent through that call site and cover it with a regression test.
