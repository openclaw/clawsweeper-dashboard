## Summary

Keep open: the session-key wake direction is plausible, but the PR head introduces a non-default-agent cron wake regression and the supplied proof still does not exercise the changed CLI/RPC/Gateway path.

## What This Changes

The PR adds optional `sessionKey` support to the wake protocol and `openclaw system event`, threads it through cron wake routing, updates docs/changelog/tests, and regenerates the shared Swift Gateway model.

## Best Possible Solution

Keep the focused wake protocol/CLI change, but preserve untargeted cron wake semantics or explicitly pass the owning agent at no-target call sites, then merge only with redacted real Gateway proof; leave the internal cron-run remap to https://github.com/openclaw/openclaw/pull/50818.

## Reproduction Assessment

Yes for the source-level gap: current main's CLI sends only `{ mode, text }`, `WakeParamsSchema` lacks `sessionKey`, and cron wake does not pass a session target. I did not establish the full Telegram/WhatsApp async-completion loss live against current main.

## Solution Assessment

No as written: the session-key plumbing is plausible, but PR head now turns omitted cron wake targets into default-agent targeted wakes, which can strand non-default cron failure alerts. It also still needs contributor-supplied real CLI/RPC/Gateway proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve untargeted cron wake broadcasts:** `src/gateway/server-cron.ts:202-203`
  - body: This now resolves an omitted wake target to the configured default agent. Existing cron failure alerts enqueue under `params.job.agentId` and then call `requestHeartbeat` without an agentId; for non-default jobs the event is written to that agent's queue but only the default agent is woken, so the alert can remain undelivered. Keep the no-target case untargeted, or pass the job agent through that call site.
  - confidence: 0.88
