## Summary

Keep open: the PR addresses a real source-level wake/session targeting gap and now supplies real behavior proof, but PR head still turns omitted cron wake targets into default-agent targeted wakes, which can strand non-default cron failure alerts.

## What This Changes

The branch adds an optional wake `sessionKey` protocol field, an `openclaw system event --session-key` flag, cron wake plumbing, docs/changelog, tests, and a regenerated Swift protocol model.

## Best Possible Solution

Keep the typed wake `sessionKey` and CLI direction, but repair the cron adapter so omitted heartbeat targets remain broadcasts or so owning agents are passed explicitly at call sites that know them, with regression coverage.

## Reproduction Assessment

Yes for the source-level gap: current main's CLI sends only `{ mode, text }`, the wake schema lacks `sessionKey`, and the wake handler drops it. The PR regression is also source-reproducible from the no-target `resolveCronWakeTarget` branch and the existing failure-alert call site.

## Solution Assessment

No as written: the additive protocol/CLI path is maintainable, but `resolveCronWakeTarget` should not turn omitted targets into default-agent targeted wakes. The safer fix is to preserve the no-target broadcast path or pass the known job agent explicitly where available.
