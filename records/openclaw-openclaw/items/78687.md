## Summary

Keep open. Current main still lacks the typed wake/session-key CLI surface, and this active PR is a plausible implementation with sufficient real behavior proof; I found one low-priority docs mismatch around the fallback behavior that should be cleaned up before landing.

## What This Changes

This PR adds an optional wake `sessionKey`, exposes `openclaw system event --session-key`, threads it through cron wake/enqueue routing, updates generated Swift models/docs/changelog, and adds focused tests.

## Best Possible Solution

Land the additive wake protocol and CLI targeting surface after the fallback docs are corrected and maintainer review confirms the session-key timing semantics; keep the broader issue open until the companion internal routing work also lands.

## Reproduction Assessment

Yes, from source inspection: current main has no typed wake `sessionKey`, no CLI flag, and `wake()` enqueues/wakes without a session target. I did not run live reproduction in this read-only review.

## Solution Assessment

Mostly yes: an additive protocol field plus CLI flag threaded through the existing cron/session-key routing is the narrow maintainable fix for this slice. The fallback docs should be aligned with the code before merge.
