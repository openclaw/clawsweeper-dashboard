## Summary

Keep open: the additive sessionKey surface is useful and real proof is sufficient, but PR head still changes omitted cron wakes into default-agent targeted wakes, which regresses existing multi-agent fanout behavior.

## What This Changes

The PR adds optional wake sessionKey protocol and CLI support, threads it through cron wake plumbing, updates docs/changelog and generated Swift models, and adds focused validation and cron tests.

## Best Possible Solution

Keep the additive protocol and CLI surface, but preserve omitted wake targets as untargeted fanout or pass a known job agent explicitly where the call site owns one.

## Reproduction Assessment

Yes for source-level reproduction: PR head resolves omitted cron wake targets through resolveCronAgent(undefined), while current main uses an omitted target to reach the heartbeat runner's broadcast branch. The failure-alert fallback still creates that omitted-target wake after enqueueing for a specific job agent.

## Solution Assessment

No as written: the sessionKey protocol and CLI direction is maintainable, but changing omitted wakes to default-agent targeted wakes is not the narrowest safe fix. Preserve the no-target branch, or pass the concrete job agent at the failure-alert call site.
