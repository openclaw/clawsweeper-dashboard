## Summary

Keep this PR open for maintainer review. The proposed seam is a plausible core SDK/Gateway feature for the linked Smarter-Claw use case, but the patch currently misses documented priority ordering, can throw on malformed `activeWhen` payloads, lacks the required public SDK API baseline update, and still needs real behavior proof beyond fixture tests.

## What This Changes

The PR adds chat-message, chat-input-bar, and chat-header-chip Control UI descriptor surfaces plus priority/activeWhen metadata, SDK exports, Gateway schema/Swift bindings, docs, and contract tests for plugin-owned inline chat UI.

## Best Possible Solution

Land this as a reviewed core plugin SDK/Gateway seam only after the ordering contract, malformed predicate validation, generated API baseline, and real behavior proof are corrected.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Implement the documented priority ordering:** `src/plugins/registry.ts:2103-2104`
  - body: The PR documents `priority` as the stable render-order key and says the host falls back by `pluginId` and descriptor id, but this code only stores the field while `plugins.uiDescriptors` returns the registry array in registration/load order. With multiple chat-stream contributors, lower priority will not reliably render first unless every client invents its own sorting, so the host projection should sort or the documented contract should be changed.
  - confidence: 0.88
- **[P2] Reject malformed activeWhen objects without throwing:** `src/plugins/registry.ts:2044-2045`
  - body: This validation assumes `descriptor.activeWhen` is an object whenever it is not `undefined`. A JS plugin can pass `null` or another non-object value and the `.sessionExtensionNamespace` read will throw during registration, turning a malformed descriptor into a plugin load failure instead of a diagnostic drop. Add an object-shape guard before reading the predicate fields.
  - confidence: 0.82
- **[P2] Update the Plugin SDK API baseline:** `src/plugin-sdk/core.ts:56`
  - body: Adding `PluginControlUiActiveWhen` to the public SDK barrels changes the exported Plugin SDK surface, but the PR does not update `docs/.generated/plugin-sdk-api-baseline.sha256`. The repo's `plugin-sdk:api:check` is designed to fail on this drift, so regenerate and commit the baseline hash for the intentional API addition.
  - confidence: 0.78
