## What This Changes

## Best Possible Solution

Settle the public descriptor contract, reject or explicitly document unregistered activeWhen namespaces, and land only after real plugin/Gateway proof and the normal CI gates are green.

## Solution Assessment

No, not yet: extending the descriptor seam may be the right maintainable direction, but the patch still projects activeWhen namespaces without proving they belong to a registered session extension and still lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Validate activeWhen namespace ownership:** `src/plugins/registry.ts:2064`
  - body: The new predicate is documented as being keyed by a plugin-owned session-extension namespace, but this validation only checks that the namespace is non-empty. Because session rows only project values for registered extensions, a typo or missing `registerSessionExtension` call will still be exposed as a valid chat-stream descriptor and then never mount. Reject descriptors whose `activeWhen.sessionExtensionNamespace` is not registered by the same plugin, or explicitly change the contract if forward references are intended.
  - confidence: 0.82
- **[P3] Update the SDK overview surface list:** `docs/plugins/sdk-overview.md:150`
  - body: The PR adds three new Control UI surfaces, but the SDK overview still says `registerControlUiDescriptor` only owns `session`, `tool`, `run`, or `settings` surfaces. That leaves the public docs inconsistent for plugin authors discovering this API outside the new hooks subsection.
  - confidence: 0.77
