## Summary

Keep open: current main and the latest release still show both reported failure paths, and the linked closing PR is open but conflicting and unmerged, so the issue remains necessary.

## What This Changes

The issue reports that the unscheduled-reminder guard appends a contradictory note for isolated-session cron reminders because compaction resets `successfulCronAdds` and existing-cron lookup only matches `sessionKey`.

## Best Possible Solution

Land a focused fix that preserves same-turn cron-add evidence across compaction retries and treats enabled same-agent isolated cron jobs as reminder coverage, preferably by rebasing or adopting the linked PR after review.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: a successful cron add can be reset to zero during compaction, then an enabled isolated same-agent cron without `sessionKey` cannot satisfy the current session-key-only lookup.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow maintainable fix is to preserve the same-turn counter through compaction and extend the existing guard lookup to same-agent isolated jobs without changing reminder detection broadly.
