## Summary

Current main and release v2026.6.5 still contain both reported gaps, while the linked implementation remains open and conflicting; this issue is therefore still necessary and should remain paired with that work.

## What This Changes

Prevent false unscheduled-reminder disclaimers when isolated-session cron coverage or a same-turn successful cron add is lost across compaction.

## Best Possible Solution

Preserve successfulCronAdds across compaction retries and extend existing-cron coverage to enabled isolated jobs owned by the current agent, with tests proving disabled, different-agent, unrelated-session, and retry cases remain correctly guarded.

## Reproduction Assessment

Yes at source level: current main resets successfulCronAdds during compaction and then uses a session-key-only store lookup that cannot recognize isolated jobs without a session key. A live current-main Telegram run was not performed during this read-only review.

## Solution Assessment

Yes, the proposed two-part repair is the narrowest maintainable direction, provided isolated coverage is limited to enabled jobs for the current agent and tests prevent unrelated jobs from suppressing the warning.
