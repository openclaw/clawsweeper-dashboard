Keep open: current main and v2026.6.8 still have the reported reminder-guard gaps, and the only linked closing PR is open, conflicting, and not merged. No safe close reason applies because the user-visible false note remains source-reproducible.

## What This Changes

The issue reports that the reply reminder guard can append a false unscheduled-reminder note after isolated-session cron creation because compaction clears `successfulCronAdds` and existing isolated crons do not match the session-key-only lookup.

## Best Possible Solution

Land a refreshed narrow core fix that preserves same-turn cron-add evidence through compaction and treats enabled same-agent isolated crons as reminder coverage in the existing reply guard.

## Reproduction Assessment

Yes. Source inspection shows a successful cron add can be counted, then reset to zero during compaction, and an enabled same-agent isolated cron cannot satisfy the current sessionKey-only lookup.

## Solution Assessment

Yes on direction. The best fix is a narrow repair in the existing reply guard and compaction state path, ideally by refreshing the linked PR or landing an equivalent current-main patch.
