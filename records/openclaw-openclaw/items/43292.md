## What This Changes

The issue reports a false unscheduled-reminder note when isolated-session cron creation is followed by compaction because cron-add state is cleared and existing-cron lookup only matches session keys.

## Best Possible Solution

Keep this issue open until maintainers land, refresh, or replace the focused fix that preserves same-turn cron-add evidence through compaction and treats enabled same-agent isolated crons as reminder coverage.

## Reproduction Assessment

Yes, at source level. Current main resets state.successfulCronAdds during compaction retry and only matches existing crons by sessionKey, so an enabled isolated cron without a sessionKey cannot suppress the note after the counter is lost.

## Solution Assessment

Yes on direction. The narrowest maintainable fix is in the existing core reply guard and compaction state path, not in channel-specific filtering or a broad reminder-policy rewrite.
