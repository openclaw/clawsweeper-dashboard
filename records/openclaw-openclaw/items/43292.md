## Summary

Keep open: current main still has both reported source gaps, and the linked implementation PR remains open rather than merged. The issue is source-reproducible today because compaction still clears same-turn cron-add evidence, then the fallback cron-store check can only match enabled jobs by `sessionKey`.

## What This Changes

The issue requests that the unscheduled-reminder guard stop appending a false no-reminder note when an isolated-session cron was created or already exists for the same agent.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/43404 with a narrow fix that preserves cron-add evidence across compaction and matches enabled isolated crons only from trusted same-agent/session evidence, with regression tests and runtime proof.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: compaction resets `successfulCronAdds` to zero, then an enabled isolated cron without `sessionKey` cannot satisfy the session-key-only existing-cron check.

## Solution Assessment

Yes on the reported fix direction. The maintainable solution is to preserve same-turn cron evidence across compaction and add isolated same-agent cron coverage without broadening suppression to unrelated jobs.
