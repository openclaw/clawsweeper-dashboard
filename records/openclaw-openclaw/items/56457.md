## Summary

Keep this PR open. Current main still applies the Discord line-count default when `maxLines` is omitted, so the user-visible problem is real, but the PR head is not merge-ready because it regresses Unicode/CJK split safety, mishandles returned split indexes and fence context in the new shared helper, can emit unbalanced fence-opener chunks, and expands the public SDK surface without the required baseline/docs/changelog alignment.

## What This Changes

The PR changes Discord reply chunking to use character-first splitting when `maxLines` is omitted, adds a shared balanced-fence Markdown chunker, re-exports it through plugin SDK reply subpaths, and adds Discord/shared regression tests.

## Best Possible Solution

Land a repaired version of the character-first Discord default that preserves current Unicode/CJK behavior, keeps all fenced chunks balanced without byte or context loss, updates the public SDK contract artifacts and user-facing docs/changelog, then validates the focused chunk tests, SDK API check, and changed gate.

Yes. ## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Unicode-safe hard splits:** `src/markdown/chunk.ts:180`
  - body: The new default helper falls back to `Math.min(maxChars, text.length)`, and the Discord line-mode copy now slices at raw UTF-16 indexes. That drops current main's code-point clamp and CJK punctuation preference, so emoji/CJK replies can split inside surrogate pairs or at worse boundaries; carry the existing Unicode/CJK split logic into both paths.
  - confidence: 0.88
- **[P2] Use the adjusted split index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can return an incremented `breakIdx`, but the caller ignores it and slices the emitted chunk with the original value. When that branch fires, the consumed character is omitted from both the emitted chunk and `nextRemaining`; destructure the returned index and use it for `remaining.slice`.
  - confidence: 0.9
- **[P2] Keep fence context on fallback tails:** `src/markdown/chunk.ts:253-257`
  - body: This fallback assigns `nextRemaining = tail`, dropping the active fence opener when carry-over cannot shrink. Later chunks then render bytes that were inside the fenced block as plain text, so the fallback should preserve or reconstruct the fence context instead of switching to raw tail content.
  - confidence: 0.82
- **[P2] Avoid splitting inside fence opener lines:** `src/markdown/chunk.ts:217-223`
  - body: After reserving space for a synthetic fence close, the loop can shrink `effectiveMaxChars` until the next break lands inside the fence opener. For small valid limits this emits an unbalanced opener fragment even when an opener-plus-close chunk would fit; keep the adjusted break outside opener lines or handle opener chunks explicitly.
  - confidence: 0.79
- **[P2] Regenerate and document the SDK export:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This adds `chunkMarkdownWithBalancedFences` to a public plugin SDK subpath, but the PR does not update the SDK API baseline or public SDK docs, and the Discord default change lacks the required user-facing docs/changelog update. `plugin-sdk:api:check` and users will see drift until those contract surfaces are aligned.
  - confidence: 0.86
