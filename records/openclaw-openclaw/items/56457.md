## Summary

Keep open: current main still has the reported Discord line-count default, but the PR head introduces source-visible chunking regressions and public SDK contract drift that need repair before merge.

## What This Changes

This PR changes Discord reply chunking to use character-first defaults when `maxLines` is omitted, extracts balanced-fence Markdown chunking into shared Plugin SDK exports, and adds Discord/shared chunking regression tests.

## Best Possible Solution

Land a repaired character-first Discord default that preserves existing Unicode, CJK, and fence invariants, keeps the SDK contract documented/generated, and records the user-facing behavior change in the changelog.

## Reproduction Assessment

Yes, from source inspection. Current main routes omitted Discord `maxLines` to `DEFAULT_MAX_LINES = 17`, and the PR-head blockers are visible in the new shared Markdown chunker logic.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Unicode-safe hard splits:** `src/markdown/chunk.ts:180`
  - body: The new shared helper falls back to a raw UTF-16 index, while current Discord chunking clamps surrogate pairs and has CJK punctuation split coverage. The new default path can split emoji or regress CJK boundaries, so carry those protections into this helper.
  - confidence: 0.88
- **[P2] Use the adjusted split index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can increment and return `safeBreakIdx`, but the caller slices the emitted chunk with the original `breakIdx`. When that branch fires, one consumed character is omitted from both the emitted chunk and `nextRemaining`; destructure and slice with the returned index.
  - confidence: 0.9
- **[P2] Keep fence context on fallback tails:** `src/markdown/chunk.ts:253-257`
  - body: When reopening cannot shrink but a trailing fence is active, this fallback assigns `nextRemaining = tail`, dropping the opener for following chunks. Content that was inside the fenced block can then render as plain text, so preserve or reconstruct the active fence context.
  - confidence: 0.84
- **[P2] Avoid splitting inside fence opener lines:** `src/markdown/chunk.ts:217-223`
  - body: The reserve adjustment can shrink the effective limit until the next break lands inside the opener line. For small valid limits this emits an unbalanced opener fragment even when a balanced opener chunk would fit, so keep adjusted breaks outside opener lines or handle opener chunks explicitly.
  - confidence: 0.79
- **[P2] Align the new SDK export contract:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This adds `chunkMarkdownWithBalancedFences` to public Plugin SDK reply subpaths, but the matching generated API baseline, docs, and changelog updates are missing. Keep the public contract artifacts aligned before merging the new export.
  - confidence: 0.86
