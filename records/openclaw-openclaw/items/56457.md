## Summary

Keep open. Current main still has the Discord line-count default that this PR is trying to change, but the proposed branch has concrete chunking correctness blockers and public SDK/docs contract drift before it can merge.

## What This Changes

This PR changes Discord chunking to use character-first splitting when `maxLines` is omitted, adds a shared balanced-fence Markdown chunker through plugin SDK reply subpaths, and adds Discord/shared regression tests.

## Best Possible Solution

Land a repaired character-first Discord default that preserves Unicode/CJK and fence invariants, keeps the SDK contract documented and generated, and records the user-facing Discord behavior change in the changelog.

## Reproduction Assessment

Yes. The PR blockers have concrete source-level examples in existing review comments.

## Solution Assessment

No, not in the current PR form. The direction is maintainable, but the implementation needs to preserve existing chunking invariants and align public SDK/docs artifacts before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Unicode-safe hard splits:** `src/markdown/chunk.ts:180`
  - body: The new helper falls back to raw UTF-16 indexes and the Discord line-mode copy no longer carries current main's code-point clamp and CJK punctuation preference. This can split emoji surrogate pairs or regress CJK boundary selection, so carry the existing safe split logic into the new paths.
  - confidence: 0.88
- **[P2] Use the adjusted split index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can return an incremented `breakIdx`, but the caller slices the emitted chunk with the original value. When that branch fires, one consumed character is omitted from both the emitted chunk and `nextRemaining`; destructure and use the returned index.
  - confidence: 0.9
- **[P2] Keep fence context on fallback tails:** `src/markdown/chunk.ts:253-257`
  - body: The fallback assigns `nextRemaining = tail`, dropping the active fence opener when carry-over cannot shrink. Later chunks can render bytes that were inside the fenced block as plain text, so preserve or reconstruct the fence context instead of switching to raw tail content.
  - confidence: 0.82
- **[P2] Avoid splitting inside fence opener lines:** `src/markdown/chunk.ts:217-223`
  - body: After reserving space for a synthetic close fence, the loop can shrink the effective limit until the next break lands inside the opener line. For small valid limits this emits an unbalanced opener fragment even when a balanced opener chunk would fit; keep adjusted breaks outside opener lines or handle opener chunks explicitly.
  - confidence: 0.79
- **[P2] Regenerate and document the SDK export:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This adds `chunkMarkdownWithBalancedFences` to a public plugin SDK subpath without the matching API baseline/docs/changelog updates required for a new contract surface. Align the generated SDK artifacts and user-facing docs before merge.
  - confidence: 0.86
