## Summary

Keep open: current main still has the Discord default line-count split that this PR targets, but the PR head should not merge as-is because the shared chunker introduces source-visible Unicode/fence regressions and an unaligned public SDK export.

## What This Changes

The PR changes Discord reply chunking to use character-first defaults, factors fence-aware Markdown splitting into a shared helper exported through Plugin SDK reply subpaths, and adds Discord/shared regression tests.

## Best Possible Solution

Land a repaired Discord character-first chunking change that preserves current Markdown, CJK, surrogate-pair, whitespace, and fence invariants, and either fully aligns the SDK contract artifacts or keeps the helper private.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted `maxLines` still resolves to `DEFAULT_MAX_LINES = 17`, so a many-line under-character-limit Discord reply will take the splitting path; the PR-head blockers are also directly visible in the added `src/markdown/chunk.ts` logic.

## Solution Assessment

No, not in the current PR form. The direction can be maintainable, but the branch must preserve current chunking invariants and align or avoid the new public SDK surface before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Unicode-safe hard splits:** `src/markdown/chunk.ts:180`
  - body: The new helper falls back to a raw UTF-16 index, while current Discord chunking clamps surrogate pairs and prefers CJK punctuation before hard splitting. This can split emoji or regress CJK boundary selection on the new default path, so carry those protections into the shared helper.
  - confidence: 0.88
- **[P2] Use the adjusted split index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can increment and return `safeBreakIdx`, but the caller slices the emitted chunk with the original `breakIdx`. When that branch fires, one consumed character is omitted from both the emitted chunk and `nextRemaining`; destructure and slice with the returned index.
  - confidence: 0.9
- **[P2] Keep fence context on fallback tails:** `src/markdown/chunk.ts:253-257`
  - body: When reopening cannot shrink but a trailing fence is active, this fallback assigns `nextRemaining = tail`, dropping the opener for following chunks. Content that was inside the fenced block can then render as plain text, so preserve or reconstruct the active fence context instead of switching to raw tail content.
  - confidence: 0.84
- **[P2] Avoid splitting inside fence opener lines:** `src/markdown/chunk.ts:217-223`
  - body: The reserve adjustment can shrink the effective limit until the next break lands inside the opener line. For small valid limits this emits an unbalanced opener fragment even when a balanced opener chunk would fit, so keep adjusted breaks outside opener lines or handle opener chunks explicitly.
  - confidence: 0.79
- **[P2] Align the new SDK export contract:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This adds `chunkMarkdownWithBalancedFences` to public Plugin SDK reply subpaths, but the matching generated API baseline/docs/changelog updates are missing. Keep the public contract artifacts aligned before merging the new export, or keep the helper private to Discord.
  - confidence: 0.86
