## Summary

Keep open. Current main still has the Discord line-count default this PR targets, so the work is not obsolete, but the branch is not merge-ready because the new shared chunker regresses current Unicode/CJK splitting, still has fence-boundary correctness bugs, adds a public SDK export without the required baseline update, and leaves public Discord default docs/changelog out of sync.

## What This Changes

The PR changes Discord reply chunking to use character-first splitting when `maxLines` is omitted, adds a shared balanced Markdown fence-aware chunker, re-exports it through plugin SDK reply subpaths, and adds Discord/shared regression tests.

## Best Possible Solution

Land the character-first Discord default only after the shared helper preserves current Unicode/CJK behavior, keeps fenced content balanced without byte loss or context loss, updates the SDK baseline and public docs/changelog, rebases onto current main, and passes focused chunking/API checks plus the changed gate.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Unicode-safe hard splits:** `src/markdown/chunk.ts:180`
  - body: The new shared helper falls back to `Math.min(maxChars, text.length)`, which can split UTF-16 surrogate pairs and ignores the CJK punctuation break logic already present on current main. Because omitted `maxLines` now routes Discord replies through this helper, long CJK or emoji-heavy replies would regress the current chunker behavior; carry over the current clamp/CJK break logic here and keep the explicit line-mode path aligned.
  - confidence: 0.88
- **[P2] Use the adjusted split index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can increment and return `safeBreakIdx`, but the caller ignores the returned `breakIdx` and slices the emitted chunk with the original value. When that branch fires, the consumed character is omitted from both the emitted chunk and `nextRemaining`; destructure the returned index and use it for `remaining.slice`.
  - confidence: 0.9
- **[P2] Preserve fence context on fallback tails:** `src/markdown/chunk.ts:253-258`
  - body: When reopening the tail would not shrink the remaining input, this fallback assigns `nextRemaining = tail`, dropping the active fence context for later chunks. For malformed long-opener cases, bytes that were inside a fenced block render as plain text in the next chunk; keep or reconstruct a fence marker instead of switching to raw tail content.
  - confidence: 0.82
- **[P2] Keep splits out of fence opener lines:** `src/markdown/chunk.ts:217-223`
  - body: Reserve adjustment can shrink `effectiveMaxChars` so the next preferred break lands inside a fence opener. That lets the helper emit an unbalanced opener fragment even when a balanced opener-plus-close chunk would fit, violating the balanced-fence contract for small but valid limits.
  - confidence: 0.79
- **[P2] Update the SDK API baseline hash:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This adds `chunkMarkdownWithBalancedFences` to the public `openclaw/plugin-sdk/reply-chunking` surface, but the PR does not update the generated SDK API baseline hash. `pnpm plugin-sdk:api:check` will report drift until `pnpm plugin-sdk:api:gen` is run and the updated hash is committed.
  - confidence: 0.93
- **[P2] Document and changelog the default change:** `extensions/discord/src/chunk.ts:39-40`
  - body: The PR changes omitted `maxLines` from the documented 17-line default to character-first chunking, and also changes the default character budget, but the public docs/UI hints still describe the old default and the branch has no required user-facing changelog entry. Update those surfaces so users and release notes match the shipped behavior.
  - confidence: 0.84
