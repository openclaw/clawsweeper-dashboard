## Summary

Keep open, but do not merge this head as-is. Current main still has the Discord default line-cap behavior the PR targets, while the PR head has concrete chunking correctness blockers and SDK contract drift that need repair first.

## What This Changes

The PR changes Discord reply chunking to use character-first splitting when `maxLines` is omitted, adds a shared balanced-fence Markdown chunker through reply SDK subpaths, and adds Discord/shared regression tests.

## Best Possible Solution

Land a repaired current-main change that keeps Discord's default length mode character-first while preserving Unicode, CJK, max-character, line-mode, and fence-balancing invariants, and update changelog plus SDK docs/API artifacts only if a public SDK surface is still needed.

## Reproduction Assessment

Yes, source-level. Current main resolves omitted `maxLines` to `17` and checks line count before returning a single chunk, so a reply with more than 17 lines under the Discord character cap will split without needing a live Discord run.

## Solution Assessment

No, not in the current PR form. The direction is maintainable, but the implementation needs to preserve existing Unicode/CJK and fence invariants and align public SDK artifacts before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Unicode-safe hard splits:** `src/markdown/chunk.ts:180`
  - body: The new shared helper falls back to raw UTF-16 indexes instead of carrying current Discord's code-point clamp and CJK punctuation preference. Long no-whitespace content can split emoji surrogate pairs or regress CJK boundary selection, so reuse the existing safe split logic in the new path.
  - confidence: 0.88
- **[P2] Use the adjusted split index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can return an incremented `breakIdx`, but the caller slices the emitted chunk with the original value. When that branch fires, one consumed character is omitted from both the emitted chunk and `nextRemaining`; destructure and use the returned index.
  - confidence: 0.9
- **[P2] Keep fence context on fallback tails:** `src/markdown/chunk.ts:253-257`
  - body: The fallback assigns `nextRemaining = tail`, dropping the active fence opener when carry-over cannot shrink. Later chunks can render bytes that were inside the fenced block as plain text, so preserve or reconstruct the fence context instead of switching to raw tail content.
  - confidence: 0.82
- **[P2] Avoid splitting inside fence opener lines:** `src/markdown/chunk.ts:217-223`
  - body: After reserving space for a synthetic close fence, the loop can shrink the effective limit until the next break lands inside the opener line. For small valid limits this emits an unbalanced opener fragment even when a balanced opener chunk would fit; keep adjusted breaks outside opener lines or handle opener chunks explicitly.
  - confidence: 0.79
- **[P2] Regenerate and document the SDK export:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This adds `chunkMarkdownWithBalancedFences` to a public plugin SDK subpath without the matching API baseline, docs, and changelog updates required for a new contract surface. Align the generated SDK artifacts and user-facing docs before merge.
  - confidence: 0.86
