## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P1] Slice with the returned split index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can increment `safeBreakIdx` to consume one more character, but the caller still builds `chunk` with the original `breakIdx`. In that branch the emitted chunk excludes the consumed character and `nextRemaining` starts after it, so fenced output silently loses content; slice with the returned index.
  - confidence: 0.91
- **[P1] Preserve CJK and code-point safe Discord splits:** `src/markdown/chunk.ts:180`
  - body: Current main's Discord splitter has CJK punctuation and surrogate-pair guards, but the new default path calls this helper and its hard fallback is only `Math.min(maxChars, text.length)`. Long CJK or emoji-heavy Discord replies can again split at bad boundaries, regressing the behavior merged in https://github.com/openclaw/openclaw/pull/73960.
  - confidence: 0.86
- **[P1] Keep the documented line cap compatible by default:** `extensions/discord/src/chunk.ts:40`
  - body: Setting the omitted `maxLines` path to `Number.MAX_SAFE_INTEGER` changes existing Discord installs away from the documented `maxLinesPerMessage` default of 17. Without maintainer approval, migration docs, or an opt-in setting, users who relied on shorter Discord messages will get much taller replies after upgrade.
  - confidence: 0.82
- **[P1] Do not publish the unsettled chunker as SDK API:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This exposes `chunkMarkdownWithBalancedFences` through a plugin SDK subpath while the helper still has blocking correctness and default-behavior questions. Public SDK exports are compatibility surface, so keep this internal or Discord-local until the contract is approved, documented, and tested as API.
  - confidence: 0.78
