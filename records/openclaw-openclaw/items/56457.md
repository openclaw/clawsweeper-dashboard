## Summary

Keep open: the change is useful, but the current PR is not merge-ready because it lacks real Discord/runtime proof, changes a documented Discord default plus a public SDK surface, and still has source-backed Unicode/fence chunking regressions.

## What This Changes

The PR changes Discord reply chunking to character-first when `maxLines` is omitted, extracts a balanced-fence Markdown helper through reply SDK subpaths, and adds regression tests.

## Best Possible Solution

Land a repaired Discord chunking change only after maintainers approve the default policy and SDK surface, with Unicode/CJK/fence invariants preserved and real runtime proof attached.

## Reproduction Assessment

Yes, from source: current main sets `DEFAULT_MAX_LINES` to 17 and the default send path leaves `maxLines` optional, so tall under-character-limit replies still hit the line cap. I did not run live Discord delivery in this read-only review.

## Solution Assessment

No, not in the current PR form. The direction may be useful, but the patch regresses existing chunking invariants and expands a public SDK surface without the expected contract alignment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Unicode-safe hard splits:** `src/markdown/chunk.ts:180`
  - body: The new helper falls back to a raw UTF-16 index, while current Discord chunking clamps surrogate pairs and prefers CJK punctuation before hard splitting. Since default Discord replies now route through this helper, emoji or CJK text can split worse than on main; carry those protections into the shared helper before using it here.
  - confidence: 0.89
- **[P2] Use the adjusted split index:** `src/markdown/chunk.ts:303`
  - body: `splitRemainingChunk` can increment and return `safeBreakIdx`, but the caller slices the emitted chunk with the original `breakIdx`. When that branch fires, one consumed character is omitted from both the emitted chunk and `nextRemaining`; slice with the returned index.
  - confidence: 0.9
- **[P2] Keep fence context on fallback tails:** `src/markdown/chunk.ts:253-257`
  - body: When reopening cannot shrink but a trailing fence is active, this fallback assigns `nextRemaining = tail`, dropping the opener for following chunks. Content that was inside the fenced block can then render as plain text, so preserve or reconstruct the active fence context instead of switching to raw tail content.
  - confidence: 0.86
- **[P2] Avoid splitting inside fence opener lines:** `src/markdown/chunk.ts:217-223`
  - body: The reserve adjustment can shrink the effective limit until the next break lands inside the opener line. For small valid limits this emits an unbalanced opener fragment even when a balanced opener chunk would fit, so keep adjusted breaks outside opener lines or handle opener chunks explicitly.
  - confidence: 0.8
- **[P2] Align the public SDK export contract:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This adds `chunkMarkdownWithBalancedFences` to public Plugin SDK reply subpaths, but the branch only updates the source-mention test. Either keep the helper plugin-local or align the docs, entrypoint/API baseline, and export contract artifacts expected for a new public SDK helper.
  - confidence: 0.84
