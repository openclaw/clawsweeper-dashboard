## Summary

Keep this PR open. Current main still has the reported Discord default line-splitting behavior, so the PR is not obsolete, but the current patch is not merge-ready because it regresses existing Unicode/CJK split guarantees, has fence-carry correctness bugs in the new shared helper, and changes a public SDK surface without the required baseline/docs/changelog alignment.

## What This Changes

The PR changes Discord reply chunking to use a character-first default, extracts a balanced-fence Markdown chunker into `src/markdown/chunk.ts`, re-exports it through plugin SDK reply subpaths, and adds Discord/shared chunking regression tests.

## Best Possible Solution

Land a repaired, rebased version of the character-first Discord default that preserves existing Unicode/CJK and fence-balancing invariants, keeps explicit `maxLines` and `chunkMode: "newline"` compatibility, and aligns SDK baseline, docs, and changelog before the changed gate runs.

## Reproduction Assessment

Yes. Current main can be reproduced by calling `chunkDiscordText` with many newline-separated lines under Discord's character limit and no explicit `maxLines`; source inspection shows the default still applies the 17-line cap. The PR blockers are reproducible by inspecting the PR head split-index, hard-fallback, fence-context, SDK export, and missing generated/public contract updates.

## Solution Assessment

No, not yet. The direction is plausible, but this patch is not the best mergeable solution until it preserves current split invariants and updates the public SDK/docs/changelog contracts it changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Unicode-safe line-mode splits:** `extensions/discord/src/chunk.ts:173-186`
  - body: The PR replaces the current line splitter's code-point/CJK fallback with raw `slice(0, limit)` and `breakIdx = limit`. Explicit `maxLines` callers still use this path, so emoji surrogate pairs can be split and CJK punctuation no longer anchors readable breaks; carry over the current clamp/CJK logic here.
  - confidence: 0.86
- **[P2] Preserve Unicode-safe hard splits:** `src/markdown/chunk.ts:180`
  - body: The new shared helper falls back to `Math.min(maxChars, text.length)`, which can split UTF-16 surrogate pairs and ignores the CJK punctuation break logic already present on current main. Since omitted `maxLines` now routes Discord replies through this helper, long CJK or emoji-heavy replies would regress current behavior.
  - confidence: 0.88
- **[P2] Use the adjusted split index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can increment and return `safeBreakIdx`, but the caller ignores the returned `breakIdx` and slices the emitted chunk with the original value. When that branch fires, the consumed character is omitted from both the emitted chunk and `nextRemaining`; destructure the returned index and use it for `remaining.slice`.
  - confidence: 0.9
- **[P2] Preserve fence context on fallback tails:** `src/markdown/chunk.ts:253-258`
  - body: When reopening the tail would not shrink the remaining input, this fallback assigns `nextRemaining = tail`, dropping the active fence context for later chunks. For malformed long-opener cases, bytes that were inside a fenced block can render as plain text in the next chunk; keep or reconstruct a fence marker instead of switching to raw tail content.
  - confidence: 0.82
- **[P2] Keep splits out of fence opener lines:** `src/markdown/chunk.ts:217-223`
  - body: Reserve adjustment can shrink `effectiveMaxChars` so the next preferred break lands inside a fence opener. That lets the helper emit an unbalanced opener fragment even when a balanced opener-plus-close chunk would fit, violating the balanced-fence contract for small but valid limits.
  - confidence: 0.79
- **[P2] Update the SDK API baseline:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This adds `chunkMarkdownWithBalancedFences` to the public `openclaw/plugin-sdk/reply-chunking` surface, but the generated SDK API baseline is not part of the PR. `pnpm plugin-sdk:api:check` should fail until the baseline is regenerated or the helper is kept out of the public SDK.
  - confidence: 0.93
- **[P3] Document and changelog the default change:** `extensions/discord/src/chunk.ts:39-40`
  - body: The PR changes omitted `maxLines` from the documented 17-line default to character-first chunking and lowers the default character budget, but it does not update user-facing docs or the required changelog. Add the release note and public contract docs so shipped behavior matches operator guidance.
  - confidence: 0.84
