## Summary

Keep this PR open. Current main still has the Discord line-count default the PR is trying to improve, so the work is not obsolete, but the current head is not merge-ready because it regresses current Unicode/CJK chunking behavior, has fence carry-over edge cases, and changes public SDK/docs-visible behavior without the required baseline, docs, and changelog updates.

## What This Changes

The PR changes Discord reply chunking to use character-first splitting when `maxLines` is omitted, adds a shared balanced Markdown fence chunker exported through plugin SDK reply subpaths, and adds Discord/shared regression tests.

## Best Possible Solution

Land a rebased, repaired version that keeps Discord's character-first default for omitted `maxLines`, preserves the current Unicode/CJK and explicit line-mode invariants, keeps balanced-fence splitting within `maxChars`, and updates the SDK baseline, public docs/config help, and changelog in the same PR.

## Reproduction Assessment

Yes. The current-main behavior is reproducible by calling `chunkDiscordText` with many lines under the character limit and no explicit `maxLines`; source shows the 17-line default path. The PR blockers are reproducible by source inspection of the PR head's shared chunker and SDK/docs surfaces.

## Solution Assessment

No, not yet. The product direction is reasonable, but this patch should not merge until the shared helper preserves existing Unicode/CJK behavior, fixes the fence carry-over/index bugs, and updates the public contract artifacts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Unicode-safe hard splits:** `src/markdown/chunk.ts:180`
  - body: The shared helper falls back to a raw UTF-16 index, so long emoji or CJK replies can split inside surrogate pairs and lose the CJK punctuation break behavior already on current main. Carry the current Discord clamp/CJK logic into the shared helper and keep the explicit line-mode path aligned.
  - confidence: 0.9
- **[P2] Use the adjusted split index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can increment and return `safeBreakIdx`, but the caller ignores that returned index and slices with the original `breakIdx`. When that branch fires, the consumed character is omitted from both the emitted chunk and `nextRemaining`; destructure the returned index and use it for the slice.
  - confidence: 0.91
- **[P2] Preserve fence context on fallback tails:** `src/markdown/chunk.ts:253-258`
  - body: When reopening the tail would not shrink the remaining input, this fallback assigns `nextRemaining = tail`, dropping the active fence context for later chunks. Content that was inside a fenced block can then render as plain text; keep or reconstruct a fence marker instead of switching to raw tail content.
  - confidence: 0.84
- **[P2] Keep splits out of fence opener lines:** `src/markdown/chunk.ts:217-223`
  - body: Reserve adjustment can shrink `effectiveMaxChars` so the next preferred break lands inside the fence opener. That lets the helper emit an unbalanced opener fragment even when a balanced opener-plus-close chunk would fit, violating the balanced-fence contract for small but valid limits.
  - confidence: 0.8
- **[P2] Update the SDK API baseline:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This adds `chunkMarkdownWithBalancedFences` to the public `openclaw/plugin-sdk/reply-chunking` surface, but the PR does not update the generated SDK API baseline hash. `pnpm plugin-sdk:api:check` will report drift until the baseline is regenerated and committed.
  - confidence: 0.93
- **[P2] Document and changelog the default change:** `extensions/discord/src/chunk.ts:39-40`
  - body: The PR changes omitted `maxLines` from the documented 17-line default to character-first chunking and changes the default character budget, but the public docs/config help still describe the old default and the branch has no required user-facing changelog entry. Update those surfaces so shipped behavior and release notes match.
  - confidence: 0.86
