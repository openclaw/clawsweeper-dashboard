## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the existing Discord line cap unless approved:** `extensions/discord/src/chunk.ts:40`
  - body: Changing `DEFAULT_MAX_LINES` to `Number.MAX_SAFE_INTEGER` silently replaces the current 17-line default for every Discord user who omits `maxLines`. That is an upgrade-visible behavior change, so this needs explicit maintainer approval and docs/proof, or the existing default should remain with an opt-in path for character-first splitting.
  - confidence: 0.88
- **[P1] Use the returned safe break index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can increment and return `breakIdx`, but the caller ignores that returned index and slices with the original value. When the increment path fires, the consumed character is in neither the emitted chunk nor `nextRemaining`, so fenced content can lose one byte.
  - confidence: 0.93
- **[P2] Keep fence context when falling back to the tail:** `src/markdown/chunk.ts:257`
  - body: This fallback sets `nextRemaining` to raw `tail`, dropping the active fence opener for subsequent chunks. Content that was inside the fence can then render as plain text, which violates the balanced-fence contract for both Discord and any SDK caller of the new helper.
  - confidence: 0.89
- **[P2] Align the new SDK export with the public contract:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: Exporting `chunkMarkdownWithBalancedFences` from `plugin-sdk/reply-chunking` makes it public API, but this branch only updates a source-mention test. The scoped SDK policy requires docs, entrypoint metadata, package exports, API baselines, and export checks to stay aligned when adding public surface.
  - confidence: 0.76
