Full review comments:

- **[P1] Use the adjusted split index when slicing chunks:** `src/markdown/chunk.ts:303`
  - body: `splitRemainingChunk` can increment `safeBreakIdx` to make progress, but the caller discards the returned `breakIdx` and slices with the original index. When that branch fires, one character is omitted from both the emitted chunk and `nextRemaining`, so Discord/plugin SDK callers can silently lose fenced content.
  - confidence: 0.9
- **[P1] Keep the existing Discord default until approved:** `extensions/discord/src/chunk.ts:40`
  - body: Current main documents and applies a 17-line Discord soft cap to avoid very tall messages; this changes omitted `maxLines` to unlimited character-only chunking. Existing users will get different message counts and taller Discord posts, so this needs an explicit compatibility decision or opt-in path before merge.
  - confidence: 0.86
- **[P2] Preserve current CJK and codepoint-safe splits:** `extensions/discord/src/chunk.ts:173-174`
  - body: Current main has a merged repair that clamps hard splits to code-point boundaries and uses CJK punctuation before hard cuts. This branch's line splitter still slices raw UTF-16 limits and whitespace only, so rebasing or merging as written would regress those Discord message-splitting cases.
  - confidence: 0.82
- **[P1] Hold the SDK export until the contract is stable:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: Adding `chunkMarkdownWithBalancedFences` to `openclaw/plugin-sdk/reply-chunking` makes it a third-party plugin API, but the helper still has correctness issues and the SDK docs/entrypoint contract are not settled. Keep it Discord-local or add the documented SDK contract and compatibility coverage after the algorithm is stable.
  - confidence: 0.8
