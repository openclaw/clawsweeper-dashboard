## Summary

Keep open: current main still has the Discord line-count default, and this PR's current head is useful but not mergeable because the new shared chunker regresses existing Unicode/CJK and fenced-code invariants while also expanding the SDK contract without the required artifact alignment.

## What This Changes

This PR changes Discord reply chunking to use a character-first default, extracts a balanced-fence Markdown chunker into shared/Plugin SDK reply paths, and adds Discord/shared regression tests.

## Best Possible Solution

Land a repaired character-first Discord default that preserves existing Unicode/CJK/fence invariants and either aligns the SDK export artifacts or keeps the helper private.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Unicode-safe hard splits:** `src/markdown/chunk.ts:180`
  - body: The new helper falls back to a raw UTF-16 index, while current Discord chunking clamps surrogate pairs and prefers CJK punctuation before hard splitting. Since the default Discord path now routes through this helper, emoji or CJK text can regress on ordinary length splits.
  - confidence: 0.88
- **[P2] Use the adjusted split index:** `src/markdown/chunk.ts:299-303`
  - body: `splitRemainingChunk` can increment and return `safeBreakIdx`, but the caller slices the emitted chunk with the original `breakIdx`. When that branch fires, one consumed character is omitted from both the emitted chunk and `nextRemaining`; destructure and slice with the returned index.
  - confidence: 0.9
- **[P2] Keep fence context on fallback tails:** `src/markdown/chunk.ts:253-257`
  - body: When reopening cannot shrink but a trailing fence is active, this fallback assigns `nextRemaining = tail`, dropping the opener for following chunks. Content that was inside the fenced block can then render as plain text, so preserve or reconstruct the active fence context instead of switching to raw tail content.
  - confidence: 0.84
- **[P2] Avoid splitting inside fence opener lines:** `src/markdown/chunk.ts:217-223`
  - body: The reserve adjustment can shrink the effective limit until the next break lands inside the opener line. For small valid limits this emits an unbalanced opener fragment even when a balanced opener chunk would fit, so keep adjusted breaks outside opener lines or handle opener chunks explicitly.
  - confidence: 0.79
- **[P2] Align the new SDK export contract:** `src/plugin-sdk/reply-chunking.ts:11`
  - body: This adds `chunkMarkdownWithBalancedFences` to public Plugin SDK reply subpaths, but the matching generated API baseline/docs/contract artifacts are not updated. Keep the public contract artifacts aligned before merging the new export.
  - confidence: 0.86
