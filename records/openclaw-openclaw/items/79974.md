## Summary

Keep open. This PR adds a plausible feature, but it is not merge-ready: the new request surfaces are secret-sensitive, cron patch/update wiring is incomplete, default-agent cron jobs can miss job overrides, and no after-fix real behavior proof is present.

## What This Changes

The branch adds per-agent and per-cron-job provider request overrides for headers/auth/proxy/TLS/private-network settings, wires model resolution to merge them, updates config/cron types and schemas, adds tests, and changes a generated canvas bundle hash.

## Best Possible Solution

Keep the feature on a reviewed config/runtime boundary: add secret collection/redaction, wire cron create/update/clear/default-agent behavior through tests, avoid raw secret-bearing cron payload storage unless explicitly designed, and remove unrelated generated churn.

## Reproduction Assessment

Yes for the review blockers: source inspection shows the new cron patch field is not consumed by merge/build helpers, the default-agent merge path can skip job providers, and SecretInput-bearing fields lack matching collectors. I did not run tests because this was a read-only review and pnpm was unavailable in the checkout.

## Solution Assessment

No. The implementation should carry a sanitized runtime request override or complete canonical config plumbing, with explicit secret/redaction handling and cron patch semantics, instead of partially mutating agents.list during cron preparation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep cron request secrets out of raw job payloads:** `src/cron/types.ts:168-169`
  - body: `payload.providers` reuses `ConfiguredModelProviderRequest`, whose headers/auth/TLS fields can carry SecretInput material. Cron jobs are persisted and listed as ordinary job objects, and this patch adds no redaction or secret-resolution path for that new payload field, so literal tokens can be exposed through job state/listing and SecretRefs can reach the unresolved-secret guard later. Please move this onto a redacted config/runtime path or use an explicit non-secret job override contract.
  - confidence: 0.87
- **[P2] Wire providers through cron payload patch merging:** `src/cron/types.ts:179`
  - body: The patch type now accepts `providers` and `null`, but `mergeCronPayload` and `buildPayloadFromPatch` only copy the existing known fields. Updating an existing agentTurn cron job with `payload.providers` will silently drop the override, and `null` cannot clear it, so this new update surface does not work.
  - confidence: 0.93
- **[P2] Apply job overrides for default-agent cron jobs:** `src/cron/isolated-agent/run.ts:494-497`
  - body: This merge only runs when `agents.list` exists and contains the selected agent. Cron agentTurn jobs can run with the configured default agent when no agent list entry exists, so `payload.providers` is ignored for that common path; carry the job request override directly to model resolution or synthesize the selected agent override before use.
  - confidence: 0.88
- **[P2] Resolve per-agent request SecretRefs before use:** `src/config/types.agents.ts:15`
  - body: `AgentProviderConfig.request` reuses the model provider request type, so headers/auth/TLS may be SecretInput refs. The current secret collectors resolve `models.providers.*.request` but not `agents.list[].providers.*.request`, while `sanitizeConfiguredRequestString` throws on unresolved SecretRefs, so env/file/exec refs in the new per-agent config will fail at runtime.
  - confidence: 0.9
