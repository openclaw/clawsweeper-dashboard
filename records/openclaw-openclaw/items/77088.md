## Summary

Keep open: current main still has the monotonic Telegram transport fallback described in the report, and the active fix path is the open linked PR that closes this issue after merge.

## What This Changes

The issue requests recovery from Telegram sticky IPv4/pinned-IP fallback after transient network failures instead of staying degraded until gateway restart.

## Best Possible Solution

Land #77157 or an equivalent Telegram plugin fix that probes or demotes back to the primary transport after bounded successful fallback use, keeping config knobs and pool-size tuning separate unless maintainers choose that policy.

## Reproduction Assessment

Yes at source level: a default-dispatcher timeout followed by fallback success leaves `stickyAttemptIndex` promoted, and the next request starts from that promoted dispatcher. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes for the repair direction: the narrowest maintainable solution belongs in `extensions/telegram/src/fetch.ts` with regression coverage replacing the permanent-sticky assertions. Broader pool-size, fallback-IP, or config-knob changes should remain separate unless maintainers explicitly want them.
