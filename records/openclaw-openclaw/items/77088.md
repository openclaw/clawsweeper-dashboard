## Summary

Keep open: current main still has the monotonic Telegram sticky fallback described here, and the linked fix PR is open but unmerged, so this issue should stay open until that PR lands or is closed.

## What This Changes

The issue asks for Telegram fetch fallback to recover from transient network failures instead of staying pinned to IPv4 or a single fallback IP until gateway restart.

## Best Possible Solution

Review and land or close PR #77157; the desired end state is an internal Telegram fetch recovery path with focused regression tests, while dispatcher pool tuning and config knobs remain separate follow-up decisions.

## Reproduction Assessment

Yes at source level: a default dispatcher failure followed by fallback success leaves `stickyAttemptIndex` promoted, and the next request starts from that promoted dispatcher. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes for the narrow repair direction: recovery should live in `extensions/telegram/src/fetch.ts` with tests replacing the permanent-sticky assertions. Pool-size, fallback-IP, or config-knob changes should be separate maintainer decisions.
