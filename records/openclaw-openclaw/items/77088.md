## Summary

Keep open: current main still has the monotonic Telegram fetch fallback state described in the report, and open PR #77157 already targets this issue with closing syntax, so the issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks for Telegram fetch fallback to recover back to the primary transport after transient network failures instead of staying permanently promoted to IPv4 or pinned-IP fallback.

## Best Possible Solution

Telegram transport should regain a bounded recovery path to primary after sticky fallback succeeds, with broader dispatcher-pool tuning or config knobs kept separate unless maintainers choose that direction.

## Reproduction Assessment

Yes, at source level: a primary transport failure can promote Telegram fetch to a sticky fallback, and current main has no recovery path back to primary. I did not live-reproduce the overnight flaky-network saturation behavior.

## Solution Assessment

Yes: the narrow maintainable repair belongs in `extensions/telegram/src/fetch.ts` with regression coverage that replaces the permanent-sticky assertions. PR #77157 is already the focused implementation candidate for that path.
