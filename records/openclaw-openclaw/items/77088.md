## Summary

Keep open: current main still implements the monotonic Telegram transport fallback described in the report, and the existing tests explicitly assert that the fallback dispatcher remains sticky after recovery.

## What This Changes

The issue asks for Telegram Bot API transport fallback to recover back toward the default dispatcher after transient network failures instead of staying on IPv4 or a pinned fallback IP until process restart.

## Best Possible Solution

Add an internal recovery path in the Telegram plugin transport, such as demoting after a small number of successful sticky requests or probing the primary dispatcher, with focused regression coverage and no new config surface unless maintainers explicitly want tunable knobs.

## Reproduction Assessment

Yes at source level: a default-dispatcher failure followed by fallback success leaves `stickyAttemptIndex` promoted, and the next request starts from that promoted dispatcher. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes for a repair direction: the narrowest maintainable fix belongs in `extensions/telegram/src/fetch.ts` with tests that replace the current permanent-sticky assertions. Pool-size or config-knob changes should stay separate unless maintainers choose that policy.
