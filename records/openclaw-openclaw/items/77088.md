## Summary

Keep open: current main still has the monotonic Telegram transport fallback described here, and an open fix PR (#77157) already references this issue with closing syntax, so the issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks for Telegram fetch fallback recovery so sticky IPv4 or pinned-IP transport can return to the primary path after transient network failures.

## Best Possible Solution

Review and land or close #77157; the desired end state is a narrow internal Telegram transport recovery path with regression tests, leaving config knobs and dispatcher-pool tuning as separate maintainer decisions if needed.

## Reproduction Assessment

Yes at source level: a recoverable fetch error promotes `stickyAttemptIndex`, a fallback success returns without demotion, and the next request starts from the promoted dispatcher. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes for the linked repair direction: a bounded primary recovery probe or demotion path in `extensions/telegram/src/fetch.ts` is the narrowest maintainable fix. Pool-size changes, fallback-IP changes, and new config knobs should stay separate unless maintainers explicitly choose them.
