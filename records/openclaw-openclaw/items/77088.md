## Summary

Keep open: current main still has the monotonic Telegram fetch fallback described in the report, and the linked closing PR is open rather than merged.

## What This Changes

The issue asks for Telegram fetch fallback to recover from sticky IPv4/pinned-IP transports after transient network failures instead of staying promoted until gateway restart.

## Best Possible Solution

Land a focused Telegram plugin transport recovery fix with regression coverage, while keeping dispatcher pool tuning and new config knobs separate unless maintainers choose that product surface.

## Reproduction Assessment

Yes at source level: a default-dispatcher failure followed by fallback success promotes `stickyAttemptIndex`, and later success starts from that promoted dispatcher without resetting. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes for the repair direction: the narrowest maintainable solution is an internal recovery/probe path in `extensions/telegram/src/fetch.ts` with tests replacing the permanent-sticky assertions. Pool-size and config-knob changes should stay separate.
