## Summary

Keep open: current main still has the monotonic Telegram sticky fallback described in the report, and open PR #77157 already targets the repair with closing syntax, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests Telegram fetch transport recovery after transient fallback promotion so the gateway can return to the primary transport without a process restart.

## Best Possible Solution

Land a narrow Telegram plugin fix that probes or demotes sticky fallback after successful sticky requests and updates the fetch tests; leave pool tuning, extra fallback IPs, and config knobs for separate maintainer policy if needed.

## Reproduction Assessment

Yes, at source level: a default-dispatcher timeout followed by fallback success leaves the sticky index promoted, and the next request starts from that promoted dispatcher. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes: the linked repair direction is the narrowest maintainable fix, centered in `extensions/telegram/src/fetch.ts` with regression coverage. Pool-size changes and new config knobs should stay separate unless maintainers explicitly choose that policy.
