## Summary

Keep open: current main still has the monotonic Telegram sticky fallback, and this issue is now paired with open PR #77157 using closing syntax, so it should remain open until that PR is reviewed and either merged or closed.

## What This Changes

The issue reports that Telegram fetch fallback promotion is sticky and does not recover to the primary transport after transient network failures.

## Best Possible Solution

The desired end state is a bounded recovery path in the Telegram plugin transport that can return from sticky IPv4/pinned-IP fallback to the primary transport, with regression tests replacing the permanent-sticky assertions.

## Reproduction Assessment

Yes at source level: a fallback-triggering fetch failure promotes `stickyAttemptIndex`, and later successful requests start from that promoted dispatcher because there is no demotion path. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes for the repair direction: the narrowest maintainable fix belongs in `extensions/telegram/src/fetch.ts` and `extensions/telegram/src/fetch.test.ts`. Pool-size changes, extra fallback IPs, or new config knobs should remain separate unless maintainers explicitly choose that policy.
