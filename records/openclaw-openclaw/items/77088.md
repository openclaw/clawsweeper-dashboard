## Summary

Keep open: current main still has the monotonic Telegram fetch fallback state, and an open linked PR explicitly closes this issue but has not merged yet.

## What This Changes

The issue requests recovery behavior for Telegram fetch fallback so transient transport failures do not permanently pin the gateway to degraded IPv4 or fallback-IP dispatchers.

## Best Possible Solution

Review and land or close PR #77157; the end state should be a bounded Telegram plugin fallback recovery path with regression tests, leaving config knobs or dispatcher pool tuning for separate maintainer decisions.

## Reproduction Assessment

Yes at source level: a fallback-triggering failure promotes `stickyAttemptIndex`, a later success returns without resetting it, and current tests assert that the next request reuses fallback dispatchers. I did not live-reproduce the 12-24 hour flaky-network saturation path.

## Solution Assessment

Yes for the repair direction: the narrowest maintainable fix belongs in `extensions/telegram/src/fetch.ts` with tests replacing the permanent-sticky expectations. PR #77157 is the current implementation candidate; pool-size and config-knob changes should stay separate unless maintainers choose that policy.
