## Summary

Keep open: current main still latches Telegram transport fallback monotonically, and the existing tests assert sticky IPv4 and pinned-IP reuse. The behavior also conflicts with a previously shipped sticky-fallback fix, so this needs maintainer direction rather than cleanup closure.

## What This Changes

Requests that Telegram transport fallback recover from transient network failures by demoting or probing back to the primary transport and reconsidering dispatcher pool defaults.

## Best Possible Solution

Define a Telegram-owned recovery policy that keeps immediate sticky fallback benefits but periodically or success-count demotes/probes the primary transport, with regression coverage for both transient recovery and #48282.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the core latch: current source and tests show `stickyAttemptIndex` only increases and subsequent requests reuse the fallback. The long-running gateway saturation was not live-reproduced here.

## Solution Assessment

Is this the best way to solve the issue? Unclear: success-count demotion is plausible, but maintainers should choose the recovery policy so it does not undo the intentional sticky-across-restarts behavior from #48282.
