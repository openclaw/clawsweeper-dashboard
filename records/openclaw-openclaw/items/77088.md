## Summary

Keep open: current main still has the monotonic Telegram sticky fallback behavior, and an open linked PR explicitly targets this issue but has not merged yet.

## What This Changes

The issue requests that Telegram fetch recover from sticky IPv4/pinned-IP fallback after transient network failures instead of staying degraded until gateway restart.

## Best Possible Solution

Review and land PR #77157, or an equivalent narrow Telegram plugin fix, so sticky fallback probes or demotes back to the primary transport after recovery with focused regression coverage and no new config knobs in the first pass.

## Reproduction Assessment

Yes at source level: simulate a default dispatcher failure followed by fallback success, then a second request starts from the promoted dispatcher because `stickyAttemptIndex` is never reset. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes for the narrow repair direction: the fix belongs in `extensions/telegram/src/fetch.ts` with `extensions/telegram/src/fetch.test.ts` updated to assert recovery instead of permanent stickiness. Because open PR #77157 already closes this issue, the issue should remain open until that PR merges or is replaced.
