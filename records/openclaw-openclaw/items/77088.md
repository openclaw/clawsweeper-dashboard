## Summary

Keep open: current main still has monotonic Telegram transport fallback state, and open PR #77157 uses closing syntax for this report, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The report asks for Telegram fetch fallback to recover from transient failures instead of staying pinned to IPv4 or one fallback IP until gateway restart.

## Best Possible Solution

Review and land #77157, or an equivalent narrow Telegram plugin change, to add bounded recovery probing or demotion with regression tests while leaving pool-size and config-knob tuning separate.

## Reproduction Assessment

Yes at source level: a default-dispatcher failure followed by fallback success promotes `stickyAttemptIndex`, and the next request starts from that fallback attempt. I did not live-reproduce the long-running flaky-network saturation path.

## Solution Assessment

Yes for the repair direction: the narrowest maintainable fix is inside `extensions/telegram/src/fetch.ts` with focused tests replacing the current permanent-sticky assertions. Adding config knobs or changing pool defaults should remain a separate maintainer decision.
