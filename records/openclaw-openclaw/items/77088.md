## Summary

Keep open: current main still has the monotonic Telegram sticky fallback behavior, and #77157 is an open closing PR for the same issue, so this should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue requests Telegram fetch fallback recovery after transient network failures, with the central fix in the sticky transport state rather than new config knobs.

## Best Possible Solution

Review and land or replace #77157 with a narrow Telegram plugin transport recovery fix and regression tests; keep dispatcher pool/config knob changes separate unless maintainers choose that policy.

## Reproduction Assessment

Yes at source level: a default-dispatcher failure followed by fallback success leaves `stickyAttemptIndex` promoted, and the next request starts from that promoted dispatcher. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes: the best repair direction is a focused recovery path in `extensions/telegram/src/fetch.ts` with tests replacing the current permanent-sticky assertions. The open PR #77157 is the active implementation candidate, while pool sizing and config knobs should remain separate decisions.
