## Summary

Keep open: current main still latches Telegram fetch fallback state, and the linked repair PR is open but not merged.

## What This Changes

The issue asks Telegram fetch transport to recover from sticky IPv4 or pinned-IP fallback after transient network failures without requiring a gateway restart.

## Best Possible Solution

Review and land #77157, or an equivalent narrow Telegram plugin change that probes primary transport or demotes sticky fallback after successful fallback requests with regression coverage.

## Reproduction Assessment

Yes at source level: a default-dispatcher failure followed by fallback success leaves `stickyAttemptIndex` promoted, and later requests start from that fallback dispatcher. I did not reproduce the overnight flaky-network saturation path live.

## Solution Assessment

Yes for the current repair direction: the narrowest maintainable fix is in `extensions/telegram/src/fetch.ts` with updated transport-state tests, while pool-size and config-knob changes should stay separate unless maintainers choose them.
