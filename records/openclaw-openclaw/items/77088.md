## Summary

Keep open: current main and v2026.5.3 still retain the monotonic Telegram sticky fallback behavior, and #77157 is an open closing PR rather than a merged fix.

## What This Changes

Make Telegram fetch fallback recover to the default transport after transient failures instead of keeping IPv4 or pinned-IP fallback sticky until gateway restart.

## Best Possible Solution

Review and land #77157 or an equivalent focused Telegram transport recovery patch, keeping broader network tuning policy separate.

## Reproduction Assessment

Yes at source level: a fallback-triggering fetch error promotes `stickyAttemptIndex`, and later requests start from the promoted dispatcher. I did not live-reproduce the reporter’s overnight flaky-network saturation path.

## Solution Assessment

Yes for the narrow repair direction: a Telegram-owned recovery path with regression tests is the maintainable fix. Adding config knobs, changing pool limits, or expanding fallback IPs should stay separate unless maintainers explicitly choose that policy.
