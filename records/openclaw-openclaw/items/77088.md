## Summary

Keep open: current main still permanently promotes Telegram fetch fallback state, and the focused regression tests currently encode that sticky behavior instead of recovery after upstream health returns.

## What This Changes

Request to make Telegram transport fallback recover from transient failures instead of remaining pinned to IPv4 or a single fallback IP for the process lifetime.

## Best Possible Solution

Add an internal recovery path in `extensions/telegram/src/fetch.ts`, backed by focused regression tests, and leave pool-size or new config knobs for a separate maintainer decision.

## Reproduction Assessment

Yes at source level: a default-dispatcher failure followed by fallback success promotes `stickyAttemptIndex`, and the next request starts from that promoted dispatcher. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes for the repair direction: the narrowest maintainable fix is a Telegram fetch transport demotion or primary-probe path with updated sticky fallback tests. Pool tuning and user-facing config knobs should stay separate unless maintainers explicitly request them.

## Repair Work Prompt

Fix the Telegram transport fallback recovery bug reported at https://github.com/openclaw/openclaw/issues/77088. In `extensions/telegram/src/fetch.ts`, add an internal recovery path so a transient default-dispatcher failure can promote to IPv4 or pinned-IP fallback, but successful sticky fallback traffic eventually lets the transport try the primary/default dispatcher again. Prefer a small deterministic mechanism such as demoting after a bounded number of successful sticky requests or a low-risk primary probe; do not add `channels.telegram.network.*` config knobs, new env vars, or change unrelated Telegram retry policy unless maintainers explicitly ask. Update the focused tests in `extensions/telegram/src/fetch.test.ts` that currently assert permanent sticky reuse, and preserve existing behavior for explicit proxy/caller-provided dispatcher cases. Review related network reports #76852, #45759, and #56096 for context only; do not broaden this PR into typing-loop circuit breakers, gateway self-healing, or general pool tuning. Add a user-facing changelog entry because this changes Telegram network recovery behavior.
