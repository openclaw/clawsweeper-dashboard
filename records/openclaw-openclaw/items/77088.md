## Summary

Keep open: current main still has the monotonic Telegram transport fallback described in the report, and the linked closing PR #77157 is open and unmerged, so this issue should remain open until that fix lands or is replaced.

## What This Changes

The request is to let Telegram fetch recover from sticky IPv4/pinned-IP fallback after transient network failures instead of requiring a gateway restart.

## Best Possible Solution

Land or replace #77157 with a narrow Telegram plugin patch that recovers from sticky fallback and updates fetch regression tests, leaving config knobs and pool tuning separate unless maintainers choose them.

## Reproduction Assessment

Yes at source level: a default-dispatcher failure followed by fallback success promotes `stickyAttemptIndex`, and subsequent requests start from the promoted dispatcher. I did not live-reproduce the overnight flaky-network saturation path.

## Solution Assessment

Yes for the repair direction: the narrowest maintainable fix belongs in `extensions/telegram/src/fetch.ts` with tests replacing the current permanent-sticky assertions. The open closing PR should be reviewed or superseded before closing this issue.
