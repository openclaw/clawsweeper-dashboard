## Summary

Keep open: current main still has the monotonic Telegram transport fallback described here, and an open closing PR already targets the issue. The issue should stay open until that PR merges or is explicitly replaced.

## What This Changes

The issue asks for Telegram fetch fallback to recover after transient network failures instead of staying pinned to IPv4 or a fallback IP until gateway restart.

## Best Possible Solution

Add an internal Telegram transport recovery path, with focused regression coverage replacing the permanent-sticky assertions; keep pool tuning and new config knobs separate unless maintainers choose that policy.

## Reproduction Assessment

Yes at source level: a fallback promotion leaves `stickyAttemptIndex` promoted, and later successful requests return without resetting it. I did not run tests or live flaky-network reproduction because this was a read-only review and `pnpm` is unavailable in this environment.

## Solution Assessment

Yes for the repair direction: the narrowest maintainable fix belongs in `extensions/telegram/src/fetch.ts` with regression tests in `extensions/telegram/src/fetch.test.ts`. The already-open PR is the right implementation lane to review before opening duplicate work.
