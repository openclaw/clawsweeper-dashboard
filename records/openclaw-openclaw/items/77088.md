## Summary

Keep open: current main still implements the monotonic Telegram transport fallback described in the report, and the regression tests currently assert permanent sticky fallback behavior after recovery.

## What This Changes

Requests a Telegram fetch recovery path so transient network failures can return from sticky fallback dispatchers to the default transport without restarting the gateway.

## Best Possible Solution

Add an internal Telegram transport recovery path that demotes or probes back toward the primary dispatcher after successful sticky fallback use, with regression coverage replacing the permanent-sticky assertions; keep pool-size and new config knobs separate unless maintainers explicitly choose that policy.

## Reproduction Assessment

Yes at source level: a default-dispatcher failure followed by fallback success promotes `stickyAttemptIndex`, and the next request starts from the promoted dispatcher. The long-running network saturation was not live-reproduced here.

## Solution Assessment

Yes for a narrow repair direction: this belongs in `extensions/telegram/src/fetch.ts` with focused tests in `extensions/telegram/src/fetch.test.ts`. Pool tuning and operator config knobs should be separate follow-up choices unless needed for the minimal fix.
