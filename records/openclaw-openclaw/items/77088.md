## Summary

Keep open: current main still has the one-way Telegram sticky fallback described here, and the linked fix PR #77157 is open and unmerged, so this issue should remain open until that PR lands or is closed.

## What This Changes

Requests Telegram fetch transport to recover from sticky IPv4 or pinned-IP fallback after transient network failures, with optional follow-up tuning for dispatcher pool and fallback IP settings.

## Best Possible Solution

Review and land or close PR #77157; if it is not accepted, implement an equivalent narrow recovery path in the Telegram transport with regression tests.

## Reproduction Assessment

Yes at source level: simulated fallback failures in `extensions/telegram/src/fetch.test.ts` show the next request starts from the promoted dispatcher, and current source has no demotion path. I did not live-reproduce the long-running flaky-network saturation path.

## Solution Assessment

Yes for the repair direction: a small recovery probe or demotion path in `extensions/telegram/src/fetch.ts` with focused tests is the narrow maintainable fix. Pool sizing, fallback IP expansion, and config knobs should stay separate unless maintainers choose that policy.
