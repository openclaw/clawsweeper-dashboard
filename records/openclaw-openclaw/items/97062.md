## Summary

Keep open: current main still does not support host-local `.vcf`/`text/vcard` sends, and the alternative configurable allowlist is a host-read security/product boundary that needs maintainer approval before automation.

## What This Changes

Request to allow host-local `.vcf`/`text/vcard` attachments through `MEDIA:` or `message(media=...)`, or add an operator config knob to extend the host-read MIME allowlist.

## Best Possible Solution

Have a maintainer decide the host-read boundary, then prefer a narrow vCard implementation with `.vcf` MIME mapping, validated-text gating, docs, and focused media tests over a broad global MIME override.

## Reproduction Assessment

Yes, from source inspection: current main omits `text/vcard` from the host-read document/text allowlists and has no `.vcf` MIME mapping, while host-read local media must pass `assertHostReadMediaAllowed` before delivery. I did not run a live Telegram send in this read-only review.

## Solution Assessment

Unclear until a maintainer decides the host-read policy. A narrow built-in vCard allowance with validation looks more maintainable than the requested broad config knob, because repository guidance keeps config/env surface high-bar and prior review rejected a global MIME override for a narrower case.
