## Summary

Keep open. The PR is not mergeable as-is: it adds a useful read-tool seam, but still preserves unrestricted default host reads while using closing syntax for the privacy-isolation security report, and it bundles unrelated security-sensitive config-write behavior plus an embedded-runner payload regression.

## What This Changes

The PR adds workspaceOnly-aware host read operations and read-tool tests, while also changing config writes, web-chat optimistic message merging, embedded-runner payload fallback, Telegram session locking, media URI hydration, gateway locality, Discord mention fallback, and Anthropic tool schema conversion.

## Best Possible Solution

Split the PR. Land a focused read-tool fix for #70573 only after deciding the default read boundary, with explicit tests for cross-agent memory reads, absolute paths, parent traversal, and symlink escapes; move the config symlink write behavior and other unrelated fixes into separate reviewed PRs.

## Reproduction Assessment

Yes. The reproduction is high-confidence from source and tests: with `tools.fs.workspaceOnly` unset or false, current main and the PR's new read operations take an unrestricted host read path, and #70573 provides the concrete cross-agent `MEMORY.md` read scenario.

## Solution Assessment

No. A host read operations seam is a plausible implementation piece, but this PR keeps the unsafe default, claims to close the security report, and bundles unrelated security-sensitive changes; the safer solution is a narrow PR plus explicit maintainer/security policy review.
