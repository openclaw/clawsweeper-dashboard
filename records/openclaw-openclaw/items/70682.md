## Summary

Keep open. Current main has not implemented the PR’s read-tool seam, and the PR is not mergeable as-is because it still leaves the linked default privacy bypass unresolved while bundling unrelated security-sensitive config-write changes and an embedded-runner payload regression.

## What This Changes

The PR adds a host read-tool operations wrapper keyed by `tools.fs.workspaceOnly` and also changes config writes, web-chat optimistic message merging, embedded-runner payload fallback, Telegram session locking, media URI hydration, Discord mention fallback, gateway locality, and Anthropic tool schema conversion.

## Best Possible Solution

Split the PR. Land a focused agent read-tool fix that enforces the chosen privacy boundary for #70573, with explicit tests for cross-agent memory reads, absolute paths, parent traversal, symlink/hardlink escapes, and the intended `workspaceOnly=false` behavior; keep config symlink writes and the unrelated UI/gateway/channel/runtime fixes in separate reviewed PRs.

## Reproduction Assessment

Yes. The default-read privacy gap is reproducible statically from current main and the PR diff: `tools.fs.workspaceOnly` defaults false, the PR’s new read operations also default false, and #70573 provides a concrete `read path="~/.openclaw/agents/therapist/MEMORY.md"` scenario.

## Solution Assessment

No. A read-tool operations seam is useful directionally, but this patch is not the best fix because it preserves the unsafe default while claiming to close the security report and mixes unrelated security-sensitive changes into the same PR.

  - body: `workspaceOnly` still defaults to `false`, so callers using the documented default take the unrestricted host `fs.readFile` path.   - confidence: 0.86

  - body: The new fallback injects `finalAssistantVisibleText` whenever tool-media payloads are empty.
