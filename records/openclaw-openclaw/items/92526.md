## Summary

The linked issue establishes a real gateway attachment regression, but this PR's fallback is broader than OpenClaw's canonical Gemini model-family logic and would override an explicit text-only catalog row for any `gemini-*` identifier. The branch also lacks real post-patch Gateway-to-Gemini-CLI proof, so it is not merge-ready.

## What This Changes

The PR makes the gateway image-capability resolver treat every `google-gemini-cli` model whose identifier or catalog name starts with `gemini-` as image-capable when metadata is stale or missing.

## Best Possible Solution

Use a shared, contract-backed predicate limited to the Gemini generative families that OpenClaw can prove are image-capable, preserve rejection for explicit text-only or unrelated `gemini-*` rows, and cover stale, missing, and negative catalog cases before live verification.

## Solution Assessment

No, not as written. A provider-specific fallback matches existing architecture, but the predicate should follow OpenClaw's narrower canonical Gemini families and retain negative handling for explicit text-only or unrelated model rows.
