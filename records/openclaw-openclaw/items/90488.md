## Label Justifications

- `P1`: The linked behavior can leave a managed gateway unable to start because an env SecretRef-backed provider key is missing from the service environment.
- `merge-risk: 🚨 compatibility`: The PR changes which existing config env values are collected for runtime/service environments during upgrade.

## Summary

Keep this PR open: the change targets the right service-env masking problem, but the patch appears incomplete because the real gateway-install path can resolve the `${VAR}` placeholder from state-dir `.env` before service-env collection, leaving the concrete config env value to mask the `state-dotenv` entry on macOS.

## What This Changes

The PR filters `${VAR}`-style values out of shared config env collection and adds config/gateway install-plan tests for preserving a state-dir `.env` SecretRef value.

## Best Possible Solution

Fix service-env planning or render policy so macOS LaunchAgent inlines state-dir `.env` values for managed SecretRef keys after normal config read/substitution, while preserving concrete config-env precedence and adding a full config-read regression.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source path: current main can let a later config-env entry replace a state-dotenv entry before macOS render policy re-inlines only final state-dotenv sources, though I did not run a live LaunchAgent crash-loop reproduction.

## Solution Assessment

Is this the best way to solve the issue? No. Filtering only unresolved placeholders is too late for the normal gateway-install config snapshot path, where state-dir `.env` can resolve the placeholder to a concrete config value before service-env collection.
