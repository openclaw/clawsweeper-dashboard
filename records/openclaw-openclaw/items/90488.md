## Summary

Keep open for maintainer review. The PR addresses a real gateway install/service-env failure path and I did not find a line-level correctness defect, but it intentionally changes shared config-env placeholder collection and the protected maintainer label requires human acceptance before merge.

## What This Changes

The PR skips unresolved `${VAR}` placeholders while collecting config env vars and adds config/service-env tests for state-dir `.env` SecretRefs in gateway install planning.

## Best Possible Solution

Land this focused fix after maintainer acceptance of the placeholder-filtering policy, or narrow the filter to service-env collection if runtime env collection must preserve placeholders.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the install-plan source path: current main can let a config placeholder mask a concrete state-dir `.env` value before LaunchAgent inlining, and the PR adds focused after-fix coverage; I did not run a live current-main macOS crash-loop reproduction in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes, conditionally: filtering unresolved placeholders at the shared collector prevents literal config env refs from masking state-dir dotenv values without provider special-casing, but maintainer acceptance is needed because the collector is shared by runtime and service env paths.
