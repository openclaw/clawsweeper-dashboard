## Summary

Keep open. Current main still has no plugin trust-tier model, and this PR targets a real core/docs/CLI surface, but the proposed patch has a blocking correctness issue in security-facing metadata: omitted-format native plugins would be shown as content-tier. The public taxonomy also needs maintainer product/security review before merge.

## What This Changes

The PR adds plugin trust-tier types and resolution, stores the tier on plugin records, prints it in plugin CLI output, and adds trust-tier documentation and tests.

## Best Possible Solution

If maintainers want this taxonomy, land a rebased version that computes tiers from the same normalized effective plugin format used in `PluginRecord`, documents current capabilities separately from reserved future behavior, and covers native, bundle, missing-format fallback, loader record paths, and CLI list/info output in tests.

## Reproduction Assessment

Yes for the blocking PR bug. A manifest without an explicit format flows into `createPluginRecord`, which normalizes the record format to `openclaw`, while the PR's raw `resolveTrustTier` call would classify that same record as `content`.

## Solution Assessment

No, the patch is not the best way to solve the issue as written. The safer narrow fix is to resolve trust from the effective normalized format at both loader call sites and align the undefined-format docs/tests with that contract before landing security-facing metadata.

## Review Findings
