## Summary

Keep open: current main does not yet have this trust-tier model, and this PR is useful but not mergeable as-is because its security-facing metadata can under-report native plugins and it is missing required release-note coverage.

## What This Changes

This PR adds plugin trust-tier types and resolution, stores the tier on plugin records, prints it in plugin list/inspect output, and adds trust-tier docs and tests.

## Best Possible Solution

Land a rebased version only if maintainers want this taxonomy, with trust tiers derived from the normalized effective plugin format, current-capability docs, CLI/list/inspect coverage, and a changelog entry.

## Reproduction Assessment

Yes for the patch defect by source inspection: on the PR branch, an undefined manifest format reaches resolveTrustTier as content while PluginRecord format defaults to openclaw/native. I did not run a live reproduction because this review was read-only.

## Solution Assessment

No. The direction is plausible, but the branch should compute trust from the normalized effective format, not raw manifest input, and should not publish user-facing security metadata without changelog and maintainer taxonomy review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve trust tiers from the effective plugin format:** `src/plugins/loader.ts:849-853`
  - body: The loader stores missing formats as openclaw through createPluginRecord, but this new call passes the raw manifestRecord.format to resolveTrustTier.
