## Summary

Keep open: current main still lacks a trust-tier surface, but this PR is not merge-ready because it introduces security-facing plugin metadata that can misclassify missing-format native plugins and lacks real CLI behavior proof.

## What This Changes

This PR adds plugin trust-tier types and resolution, stores the tier on plugin records, prints it in plugin CLI output, and documents/tests content, sandboxed, and native tiers.

## Best Possible Solution

If maintainers want this taxonomy, rebase it onto the current split loader/registry layout, compute tiers from the normalized effective plugin format, document only current capabilities plus clearly reserved future behavior, and test missing-format, native, bundle, list, info, and inspect/status output.

## Reproduction Assessment

Yes for the PR-introduced issue: source inspection shows current records normalize missing format to `openclaw`, while the PR computes the new trust tier from raw `manifestRecord.format`. That is enough to reproduce the metadata mismatch from source without running the branch.

## Solution Assessment

No. The taxonomy may be useful, but this branch is not the best merge path until security-facing metadata is computed from the same effective format the loader stores and the contributor supplies real CLI proof.
