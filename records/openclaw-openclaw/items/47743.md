## Summary

Keep open. Current main does not implement plugin trust tiers, and the PR is still potentially useful, but the patch has a blocking security-metadata mismatch and lacks real behavior proof for the new CLI output.

## What This Changes

This PR adds plugin trust-tier types and resolution, stores the tier on plugin records, prints it in plugin CLI output, and adds trust-tier docs and tests.

## Best Possible Solution

Land a rebased version only if maintainers want this taxonomy, with trust tiers derived from the normalized effective plugin format and documented as a supported CLI/status contract.

## Reproduction Assessment

Yes for the PR-introduced issue: source inspection shows missing formats are normalized to `openclaw` in current records, while the PR computes trust from the raw manifest format and would label that case `content`. This is source-reproducible without running the branch.

## Solution Assessment

No. The taxonomy may be useful, but the current implementation is not the best merge path because security-facing metadata must be computed from the same effective format that the loader and CLI display.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve trust tiers from the effective plugin format:** `src/plugins/loader.ts:849`
  - body: The loader stores missing formats as `openclaw`, but this new call passes raw `manifestRecord.format` into `resolveTrustTier`. A manifest without an explicit format would display `format: openclaw` with `trustTier: content`, under-reporting a native-capable plugin in CLI/security metadata; use the normalized effective format at both call sites and align the docs/tests with that behavior.
  - confidence: 0.92
