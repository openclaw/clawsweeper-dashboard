## Summary

Keep open. Current main still lacks an explicit trust-tier model, but this PR's implementation can mislabel omitted-format native plugins as `content`, and the branch needs maintainer product/security review before landing security-facing CLI and docs metadata.

## What This Changes

This PR adds a `PluginTrustTier` model, resolver, plugin-record field, CLI display, documentation, and tests for plugin trust classification.

## Best Possible Solution

Land a rebased, maintainer-approved version that derives tier from the normalized effective plugin format, documents current capabilities separately from reserved future behavior, and tests native, bundle, missing-format fallback, loader records, and CLI list/info output.

## Reproduction Assessment

Yes for the blocking PR bug. A manifest without `format` would pass `undefined` to the PR's resolver and get `content`, while current record creation normalizes that same plugin to `openclaw` and current runtime treats non-bundle loaded plugins as native-capable.

## Solution Assessment

No, the patch is not the best way to solve the issue as written. The safer narrow fix is to compute trust from the same effective normalized format stored on `PluginRecord`, then align docs and tests with that contract.

## Review Findings

Overall correctness: patch is incorrect
