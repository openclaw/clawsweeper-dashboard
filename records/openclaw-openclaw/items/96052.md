## Summary

Keep this PR open: it is the active, useful repair for the memory_search filename/date-token bug, but the current head still has a stored-index compatibility gap for users who indexed with hybrid search disabled and later enable it.

## What This Changes

The PR prefixes memory-core FTS indexed text with memory paths, adds FTS text-format metadata and self-heal logic, strips the prefix from snippets/scoring, and adds filename-token regression tests.

## Best Possible Solution

Make every FTS row creation/backfill path use the same path-prefixed payload, or keep `ftsTextFormat` dirty until FTS rows are rebuilt, then land this as the canonical repair for the linked memory_search bug.

## Reproduction Assessment

Yes. Current main writes only chunk body text into the indexed FTS column while keeping path unindexed, and the remaining PR defect is source-reproducible through the documented hybrid-disabled then hybrid-enabled setting path.

## Solution Assessment

No, not yet. The path-prefixed FTS approach is a reasonable memory-core fix, but the implementation must also handle schema/bootstrap paths that create FTS rows from existing chunks after hybrid search is enabled.
