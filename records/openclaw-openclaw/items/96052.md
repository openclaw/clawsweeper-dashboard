## Summary

Keep this PR open: current main and v2026.6.10 still write body-only FTS text with path stored as UNINDEXED metadata, while this PR is the active candidate repair with sufficient terminal proof and no blocking code findings. ## What This Changes

The PR changes memory-core FTS indexing to include the memory file path in indexed text, strips that prefix from snippets/scoring, versions and self-heals the FTS text format, fixes schema backfill, and adds regression coverage.

## Best Possible Solution

Land this PR as the canonical repair if maintainers accept path-prefixed FTS payloads; otherwise replace it with a separate path-index strategy before merge.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 store paths as UNINDEXED FTS metadata and write only chunk body text into the indexed FTS column, matching the linked filename/date-token miss.

## Solution Assessment

Yes, if maintainers accept the stored-format and ranking tradeoff. The PR stays inside memory-core, uses the existing FTS table, keeps snippets body-only, and covers backfill/search self-heal; the main alternative is a heavier separate path-index design.
