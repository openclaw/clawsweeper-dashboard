## Summary

Keep this PR open: it fixes the central filename-token FTS path, but it still has a concrete upgrade gap when an index is built with hybrid search disabled and later reopened with hybrid enabled.

## What This Changes

The PR prefixes memory-core FTS indexed text with the memory path, strips that prefix from snippets/scoring, versions FTS text-format metadata, self-heals legacy indexes, and adds regression tests.

## Best Possible Solution

Make every path that creates or backfills FTS rows write the same path-prefixed payload, or keep metadata dirty until those rows are rebuilt, then land with regression coverage for the hybrid toggle upgrade path.

## Reproduction Assessment

Yes, at source level. Current main stores `path` as UNINDEXED FTS metadata and writes body-only text into the indexed column, while the PR's remaining gap is traceable through the documented hybrid setting and unchanged FTS bootstrap SQL.

## Solution Assessment

No, not yet. The path-prefixed FTS approach is a reasonable owner-boundary fix, but the implementation must also handle schema/backfill paths that create FTS rows from existing chunks after hybrid search is enabled.
