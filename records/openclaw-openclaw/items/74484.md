## Summary

Keep open. Current main has improved approve recovery, including the newest admin-retry commit, but it still does not provide a deterministic supported way for a read-scoped paired CLI to reject, downgrade, or replace only its own broader repair request.

## What This Changes

The issue requests a supported recovery path for a read-scoped CLI stuck behind an auto-reissued, broader gateway pairing repair request.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes, source-backed. A read-scoped paired-device session cannot authorize `device.pair.reject` because the global method-scope gate requires `operator.pairing` before the same-device handler logic can run, and the CLI reject command has no local fallback.

## Solution Assessment

Unclear. Current main's approve fallback and admin retry are useful adjacent fixes, but they still approve the broad request; maintainers need to choose whether same-device self-reject, local reject fallback, narrower repair-scope requests, or another bootstrap path is the right contract.
