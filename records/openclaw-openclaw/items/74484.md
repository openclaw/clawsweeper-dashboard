## Summary

Keep open. Current main has partial loopback list/approve recovery, but the read-scoped CLI still lacks a deterministic supported way to reject, downgrade, or replace its own over-scoped repair request, and the reporter explicitly confirmed the local recovery was timing-sensitive rather than a close signal.

## What This Changes

The issue asks for a deterministic recovery path when a CLI paired with only `operator.read` is stuck behind an auto-reissued broader device-pairing repair request.

## Best Possible Solution

Define a bounded recovery contract for a limited paired CLI to clear or replace only its own pending repair request, then align gateway authorization, CLI behavior, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. An `operator.read` device-token session calling `device.pair.reject` will fail the global method-scope gate before the same-device reject guard can run, and the current fallback path is only list/approve.

## Solution Assessment

Unclear. The partial approve fallback is not the complete fix; maintainers need to choose between same-device self-reject, local reject fallback, narrower repair-scope requests, or another bootstrap recovery flow.
