## Summary

Keep open. Current main still has a source-backed recovery gap: a read-only paired CLI can hit a scope-upgrade repair loop where explicit approve/reject paths either require pairing scope or approve a now-stale request id after reconnect creates a replacement request.

## What This Changes

The issue asks for a deterministic CLI/Gateway recovery path when a paired CLI with only `operator.read` is blocked by auto-reissued broader repair pairing requests.

## Best Possible Solution

Define a bounded recovery contract for a limited paired CLI, then align gateway authorization, CLI fallback behavior, replacement-request handling, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. A read-only paired CLI that triggers a broader repair request cannot call gateway approve/reject, and the local approve fallback can approve the old explicit id after connect has replaced it, producing `unknown requestId`.

## Solution Assessment

Unclear. Fixing the existing local approve fallback may be narrow, but the larger requested recovery choice between broad approval, same-device self-reject, downgrade, or narrower repair scopes needs maintainer/security direction.
