## Summary

Keep open. Current main has partial recovery help for list/approve and newer request-id handling, but it still lacks a deterministic supported path for a read-only paired CLI to reject, downgrade, or replace only its own over-scoped repair request.

## What This Changes

The issue requests a deterministic recovery path for a read-only paired CLI stuck behind rotating over-scoped gateway pairing repair requests.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` and a broader pending request for the same device will fail `device.pair.reject` at the global method-scope gate before the handler's same-device rejection guard can run.

## Solution Assessment

Unclear. The current approve fallback helps one recovery path, but the best final behavior needs maintainer/security choice between same-device self-reject, local fallback, narrower repair-scope requests, or another bootstrap recovery flow.
