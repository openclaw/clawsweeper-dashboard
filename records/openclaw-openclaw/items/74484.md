## Summary

Keep open. Current main has a loopback local approval fallback and better exact-request guidance, but it still does not provide a deterministic way for a read-only paired CLI to reject, downgrade, or replace its own over-scoped repair request without approving the broad scope bundle.

## What This Changes

The issue requests a deterministic recovery path for a CLI paired only with `operator.read` when auto-reissued repair requests ask for broader operator scopes.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-backed: a device-token session with only `operator.read` and its own broader pending request will fail `device.pair.reject` at the global method-scope gate before the handler's same-device rejection guard can run.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The current approve fallback is useful but not a complete recovery contract; maintainers still need to choose whether the supported path is same-device self-reject, local fallback, narrower repair-scope requests, or another bootstrap flow.
