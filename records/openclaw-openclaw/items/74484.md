## Summary

Keep open. Current main still has the reported recovery deadlock: pairing RPC authorization runs before same-device rejection checks, `device.pair.reject` remains gated by `operator.pairing`, and the CLI still has local fallback only for list/approve. Related #75820 overlaps gateway/device CLI fallback work but is open and does not supersede this narrower reject-or-downgrade recovery problem.

## What This Changes

The issue requests a supported recovery path for a CLI paired with only `operator.read` when a broader same-device repair request keeps reappearing and approve/reject require higher pairing scope.

## Best Possible Solution

Define a bounded pairing recovery contract and implement it consistently across gateway authorization, CLI local fallback, docs, and regression tests. The desired end state is that a read-only paired CLI can clear or replace only its own pending repair request, or request a narrower approved scope bundle, without approving broader authority as the recovery mechanism.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI session with only `operator.read`, a broader same-device pending repair request, and `openclaw devices reject <requestId>` failing at method authorization before the same-device reject handler can run.

## Solution Assessment

Unclear. The report identifies a real deadlock, but the best fix needs maintainer/security choice between same-device self-reject, a local reject fallback, narrower repair-scope requests, or a separate bootstrap recovery flow.
