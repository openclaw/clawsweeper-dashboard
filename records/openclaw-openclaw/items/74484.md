## Summary

Keep open. Current main still leaves the reported recovery gap for rejecting or replacing a broad same-device repair request: `device.pair.reject` is globally gated by `operator.pairing` before the same-device handler checks can run, and the CLI only has local loopback fallback for list/approve, not reject. The reporter's follow-up confirms this is a practical recovery deadlock when they do not want to approve the broad upgrade.

## What This Changes

The issue asks for a supported recovery path when a read-only paired CLI receives an auto-reissued broader repair request that it cannot approve, reject, downgrade, or replace with narrower scopes.

## Best Possible Solution

Define a bounded same-device recovery model for local repair deadlocks, then implement it consistently in gateway authorization, CLI fallback behavior, docs, and regression tests. The end state should let a limited CLI clear or replace only its own pending repair request, or request a narrower approved scope bundle, without silently approving broader authority.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a paired CLI device session with only `operator.read`, a broader pending repair request for the same device, and `openclaw devices reject <requestId>` failing method authorization because `device.pair.reject` requires `operator.pairing` before the handler can apply same-device checks.

## Solution Assessment

Unclear. The current approve fallback can unblock by approving the broad request, but that is not the same as a least-privilege reject, downgrade, or narrower repair flow; the best fix needs maintainer/security choice between those recovery models.
