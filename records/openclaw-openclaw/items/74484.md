## Label Justifications

- `P1`: The issue blocks real users from recovering CLI control-plane access after device-pairing scope drift or requestId churn.
- `impact:auth-provider`: The affected behavior is the device-pairing, operator-scope, and token authorization path that decides whether the CLI may access the Gateway.

## Summary

Keep open. Current main has partial fixes for replacement request IDs, but the issue still covers an unresolved CLI recovery contract for under-scoped paired devices, and a newer report confirms the deadlock class on a June 2026 release.

## What This Changes

The issue asks for a deterministic recovery path when a paired CLI has only narrow operator scopes and cannot approve, reject, or downgrade a reissued broader device-pairing request.

## Best Possible Solution

Define one deterministic, documented local recovery contract for under-scoped CLI devices, including approve and reject behavior, then cover it with focused CLI/Gateway tests and release-proofed user steps.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: start from a paired operator CLI with only narrow scopes and a broader pending repair request, then approval/rejection depends on Gateway pairing/admin scope or the partial local fallback; I did not run a live packaged repro in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the security contract; the current same-device replacement approval fix is a good slice, but the best end state still needs a deterministic approve/reject or bootstrap recovery path.
