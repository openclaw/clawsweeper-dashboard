## Summary

Keep open. Current main still has a source-backed recovery deadlock: the CLI has partial list/approve fallback, but reject remains pairing-scoped and approve can still operate on a request ID that was replaced during the failing reconnect path; the related PR is still open/unmerged in the provided context.

## What This Changes

The issue requests a deterministic recovery path for a CLI paired with only `operator.read` when a broad auto-reissued repair request blocks approve/reject and other gateway control-plane calls.

## Best Possible Solution

Define a bounded recovery contract that lets a limited local CLI deterministically resolve only its own pending repair request, then align gateway authorization, CLI fallback behavior, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` cannot reach `device.pair.reject`, and approve fallback can miss after the connect attempt supersedes the visible pending request ID.

## Solution Assessment

Unclear. Current main's list/approve fallback and live request-id re-resolution are useful partial fixes, but the durable fix needs a maintainer pairing-security choice before broadening any self-recovery behavior.
