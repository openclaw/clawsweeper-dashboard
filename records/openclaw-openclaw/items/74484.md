## Summary

Keep open. Current main has partial local approve/list recovery, but the remaining read-scoped CLI recovery path is still not deterministic and `devices reject` still cannot clear the caller's own pending repair request without `operator.pairing`.

## What This Changes

The issue asks for a supported way for a read-only paired CLI to recover from recurring over-scoped gateway repair requests without manually editing pairing state or approving the full broad scope bundle.

## Best Possible Solution

Define a bounded recovery contract for read-scoped paired CLIs, then align gateway authorization, CLI behavior, docs, and regression tests around that exact contract.

## Reproduction Assessment

Yes, source-backed. A read-only paired device session calling `device.pair.reject` is blocked by the global method-scope gate before the same-device guard runs, and the current CLI has no local reject fallback.

## Solution Assessment

Unclear. The existing local approve fallback is a useful partial recovery path, but the best final fix needs a maintainer security choice between same-device self-reject, narrower repair requests, deterministic approve re-resolution, or documented broad-scope approval recovery.
