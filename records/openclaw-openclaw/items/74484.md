## Summary

Keep open. Current main still leaves the reported recovery deadlock in place for rejection: `device.pair.reject` is globally gated by `operator.pairing` before the device handler can apply its same-device ownership check, and the CLI has local loopback fallback paths for list/approve but not reject. The referenced and related context does not show a merged current-main recovery path for a read-only paired CLI to clear or replace its own broad repair request.

## What This Changes

This issue requests a supported recovery path for a CLI paired with only `operator.read` when broad auto-reissued CLI repair requests cannot be approved or rejected without `operator.pairing`.

## Best Possible Solution

Define a bounded same-device recovery model for local repair deadlocks, then implement it consistently in gateway authorization, the devices CLI, docs, and regression tests. The end state should let a limited CLI clear or replace only its own pending repair request without silently approving broader scopes.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired device session with only `operator.read`, a broader pending repair request for that same device, and `openclaw devices reject <requestId>` failing at gateway method authorization because `device.pair.reject` requires `operator.pairing` before the same-device handler guard runs.

## Solution Assessment

Unclear. The issue identifies a real gap, but the best fix needs maintainer/security choice between same-device self-reject fallback, narrower repair-scope requests, or another explicit bootstrap/recovery flow.
