## Summary

Keep open. Current main still matches the reported recovery deadlock: a read-only paired CLI cannot reach `device.pair.reject`, the same-device reject checks remain behind the global method-scope gate, and the CLI/docs still only provide local fallback for list/approve. The reporter's follow-up confirms this is a real recovery block, while related #74650 overlaps pairing approval state but does not supersede this narrower CLI recovery issue.

## What This Changes

The issue requests a supported recovery path for a CLI paired with only `operator.read` when an over-scoped same-device repair request keeps being reissued and cannot be approved or rejected.

## Best Possible Solution

Define a bounded recovery contract for limited paired devices, then implement it consistently in gateway authorization, CLI fallback behavior, docs, and regression tests. The desired end state is that a read-only CLI can clear or replace only its own pending repair request without approving broader scopes.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI session with only `operator.read`, a broader pending repair request for the same device, and `openclaw devices reject <requestId>` failing at global gateway method authorization before the same-device reject handler can run.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the best fix needs maintainer/security choice between same-device self-reject, narrower repair-scope requests, or a separate bootstrap recovery flow.
