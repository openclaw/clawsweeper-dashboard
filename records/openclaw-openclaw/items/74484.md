## Summary

Keep open. Current main still has a real pairing recovery deadlock: `openclaw devices reject` requires a pairing-scoped CLI connection, but a device approved only for `operator.read` cannot establish that higher-scope session, and there is no local reject fallback or documented downgrade/narrow-repair path. The related closed reports cover default scopes or approve/list fallback, not clearing or replacing an over-scoped same-device repair request.

## What This Changes

A read-scoped paired CLI should have a supported way to reject, downgrade, or replace auto-reissued broader gateway pairing repair requests without already holding `operator.pairing`.

## Best Possible Solution

Define and implement a bounded same-device recovery path for local repair deadlocks. The end state should let a limited CLI clear or replace only its own pending repair request, preserve explicit approval for scope upgrades, document the recovery flow, and add regression coverage for read-only CLI plus over-scoped pending repair state.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI/device approved only for `operator.read`, plus a pending broader repair request for the same device; `openclaw devices reject <requestId>` attempts `device.pair.reject` with `operator.pairing`, which the current approval cannot use to connect, and there is no local reject fallback.

## Solution Assessment

Unclear. The report identifies a valid bug, but the best fix is a maintainer/security decision between self-reject fallback, narrower repair-scope requests, downgrade semantics, or another bootstrap flow.
