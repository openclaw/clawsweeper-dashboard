## Summary

Keep open. Current main still has the reported recovery deadlock for `devices reject`: `device.pair.reject` is still globally authorized as an `operator.pairing` method before the same-device reject handler can run, and the CLI still has local fallback paths for list/approve but not reject. The reporter's follow-up confirms this is a live recovery blocker, and open #74650 may be related but does not supersede this narrower CLI pairing issue.

## What This Changes

The issue requests a supported recovery path for a CLI paired with only `operator.read` to reject, downgrade, or narrowly reissue its own broader pending repair request instead of being blocked by pairing-scope authorization.

## Best Possible Solution

Define a bounded recovery contract for limited paired devices, then implement it consistently in gateway authorization, CLI fallback behavior, docs, and regression tests. The end state should let a read-only CLI clear or replace only its own pending repair request without silently approving broader scopes or affecting other devices.

## Reproduction Assessment

Yes. A high-confidence source-backed path is a paired CLI session with only `operator.read`, a broader pending repair request for the same device, and `openclaw devices reject <requestId>` failing at gateway method authorization because `device.pair.reject` requires `operator.pairing`.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the best fix needs a maintainer/security choice between same-device self-reject, narrower repair-scope requests, or another bootstrap recovery flow.
