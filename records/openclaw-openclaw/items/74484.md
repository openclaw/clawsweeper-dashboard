## Summary

Keep open. Current main partially recovers this through local loopback list/approve fallback, but the source still leaves `device.pair.reject` behind the global `operator.pairing` gate and does not provide a deterministic same-device reject, downgrade, or narrow repair-scope recovery path for the reported read-only CLI state.

## What This Changes

The issue requests a deterministic recovery path for a CLI paired with only `operator.read` when an auto-reissued broader device repair request blocks higher-scope gateway work.

## Best Possible Solution

Define a bounded, documented recovery contract for a limited paired CLI to resolve only its own pending repair request without racing rotating request IDs or silently granting broader access.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: connect as a device-token CLI session with only `operator.read`, create a broader pending request for the same device, then call `device.pair.reject` and fail at global method-scope authorization before the same-device handler check runs. The reporter's latest comments also confirm the practical approve path was timing-sensitive rather than deterministic.

## Solution Assessment

Unclear. Current local approve fallback is a partial recovery path, but maintainers still need to choose the safest contract among same-device self-reject, stable approve retry behavior, narrower repair-scope requests, or a documented bootstrap flow.
