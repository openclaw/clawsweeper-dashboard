## Summary

Keep open. Current main has partial recovery help for `devices list` and `devices approve`, plus newer replacement-request-id handling, but a read-only paired CLI still has no deterministic supported way to reject, downgrade, or replace its own over-scoped repair request without approving the broad bundle.

## What This Changes

The issue requests a deterministic recovery path for a CLI paired with only `operator.read` when an auto-reissued repair request asks for broader operator scopes.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI fallback behavior, docs, and regression tests.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` and a broader pending request will fail `device.pair.reject` at the global method-scope gate before the handler's same-device rejection guard can run.

## Solution Assessment

Unclear. Current main's approve fallback and live request-id re-resolution are useful partial fixes, but the best final behavior still needs a maintainer/security choice between same-device self-reject, local reject fallback, narrower repair-scope requests, or another bootstrap recovery flow.
