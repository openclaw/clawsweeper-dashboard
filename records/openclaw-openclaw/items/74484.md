## Summary

Keep open. Current main still has a source-backed recovery gap: approving can fall back locally on loopback, but rejecting or replacing the same read-only CLI's broader repair request still requires pairing/admin authority or a broader approval, and the reporter explicitly confirmed the eventual recovery was flaky rather than a close request.

## What This Changes

The issue requests a deterministic recovery path for a read-only paired CLI stuck behind an auto-reissued broader gateway repair request.

## Best Possible Solution

Define a bounded recovery contract for limited paired CLI devices, then align gateway authorization, CLI behavior, docs, and regression tests so users can deterministically clear or replace only their own pending repair request without approving unintended broader access.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a paired device-token session with only `operator.read`, a broader pending request for that same device, and `device.pair.reject` failing at the global method-scope authorization gate before the handler's same-device guard can run.

## Solution Assessment

Unclear. The current approve fallback helps one recovery path, but deciding whether to add same-device reject, narrower repair-scope requests, or another bootstrap flow is a pairing-security product decision.
