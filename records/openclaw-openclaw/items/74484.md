## Summary

Keep open. Current main still gates `device.pair.reject` behind `operator.pairing` before the same-device reject handler can run, and the CLI only has local fallback for list/approve, so the reported read-only repair deadlock remains plausible and source-backed. The related referenced commit is not an upstream-main fix, and the open ACP approval-loop report does not supersede this narrower recovery issue.

## What This Changes

The issue requests a supported recovery path for a CLI paired only with `operator.read` when broader auto-reissued repair requests cannot be approved or rejected without pairing scope.

## Best Possible Solution

Define a bounded recovery model for same-device repair deadlocks, then implement it consistently across gateway authorization, the device pairing handlers, CLI behavior, docs, and regression tests. The desired end state should let a limited CLI clear or replace only its own pending repair request without silently approving broader scopes or widening bootstrap/default CLI authority.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired device-token session with only `operator.read`, a broader same-device pending repair request, and `openclaw devices reject <requestId>` failing before the handler-level same-device check because `device.pair.reject` requires `operator.pairing`.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the best fix needs maintainer/security choice between a same-device self-reject fallback, narrower repair-scope requests, or another bootstrap flow; the safest direction is one that only clears or replaces the caller's own pending request and never approves broader scopes implicitly.
