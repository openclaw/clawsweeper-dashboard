## Summary

Keep open. Current main has partial recovery improvements, but a read-only paired CLI still lacks a clear supported way to reject or downgrade its own over-scoped repair request, and stale request IDs can still fall through to a weak approve-fallback error path.

## What This Changes

The issue requests a deterministic recovery path for a CLI paired with only `operator.read` when an over-scoped repair request blocks higher-scope gateway operations.

## Best Possible Solution

Define a bounded pairing recovery contract for limited paired CLIs, then align gateway authorization, CLI fallback behavior, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. A device-token session with only `operator.read` cannot reach `device.pair.reject`, because global method authorization requires `operator.pairing` before the same-device reject guard runs.

## Solution Assessment

Unclear. The current approve/list fallback and request-id re-resolution are useful partial fixes, but the remaining reject, downgrade, and narrower-repair behavior needs a maintainer security decision before implementation.
