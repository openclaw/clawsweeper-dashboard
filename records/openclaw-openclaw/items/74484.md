## Summary

Keep open. Current main still has only partial recovery: local list/approve fallback exists, but a read-only paired CLI cannot deterministically reject or downgrade its own over-scoped repair request, and the reporter explicitly confirmed recovery was timing-sensitive rather than a close request.

## What This Changes

Provide deterministic CLI/gateway recovery for a read-only paired CLI stuck behind auto-reissued, over-scoped repair pairing requests.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes, source-backed. A device-token session with only `operator.read` and a broader pending request will fail `device.pair.reject` at the global method-scope gate before the handler's same-device rejection guard can run.

## Solution Assessment

Unclear. Current main's approve fallback and live request-id handling are useful partial fixes, but the final recovery path requires a maintainer/security choice among same-device self-reject, local reject fallback, narrower repair scopes, or documented bootstrap recovery.
