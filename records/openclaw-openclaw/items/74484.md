## Summary

Keep open. Current main still has a pairing-scope recovery gap: list/approve have partial local fallback, but reject remains pairing-scoped and the exact-ID approve fallback can still lose to replacement request churn; #75820 is open and unmerged.

## What This Changes

Requests a deterministic recovery path for a CLI paired with only `operator.read` when broader repair requests are auto-reissued and pairing commands require broader scopes.

## Best Possible Solution

Define a bounded same-device recovery contract that lets a limited paired CLI clear or replace only its own pending repair request, then align gateway authorization, CLI fallback, docs, and regression coverage.

## Reproduction Assessment

Yes, source-backed. A read-only device-token session cannot pass the global `operator.pairing` method gate for `device.pair.reject`, and the approve fallback still depends on an exact request ID while the handshake can replace that request before fallback reads it.

## Solution Assessment

Unclear. The narrow maintainable fix should address same-device recovery without widening pairing authority, but maintainers need to choose between self-reject, replacement-ID approve fallback, narrower repair scopes, or a different bootstrap path.
