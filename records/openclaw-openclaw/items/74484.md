## Summary

Keep open. Current main still leaves an `operator.read` paired CLI unable to reject its own broader pending repair request through the gateway, and the related open PR context only covers adjacent CLI fallback/timeout behavior rather than the same-device reject or narrower repair-scope recovery path.

## What This Changes

A read-only paired CLI cannot clear or narrow its own broader auto-reissued repair pairing request because approve/reject require pairing scope.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes. A source-backed reproduction is a paired device-token session with only `operator.read`, a broader pending request for the same device, and `device.pair.reject` failing at the global method-scope authorization before the handler's same-device check can run.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the safest fix needs maintainer choice between same-device self-reject, narrower repair-scope requests, or another bootstrap recovery flow.
