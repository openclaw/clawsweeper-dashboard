## Summary

Keep open. Current main still leaves a read-only paired CLI unable to clear or replace its own broader pending repair request through the gateway, and the related open PR context does not cover same-device reject or narrower repair-scope recovery.

## What This Changes

The issue reports a read-only paired CLI stuck behind a rotating broader repair pairing request that it cannot approve or reject through the gateway.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: connect with a paired device token carrying only `operator.read`, create a broader pending repair request for the same device, then call `device.pair.reject` and hit the global `operator.pairing` gate before the same-device reject handler runs.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the narrowest maintainable fix needs maintainer/security selection among same-device self-reject, local loopback fallback, narrower repair-scope requests, or another bootstrap recovery flow.
