## Summary

Keep open. Current main still only has partial CLI approve/list fallback, while the source path confirms a read-only paired CLI cannot reject its own pending repair request and the remaining recovery behavior needs an explicit pairing-security decision.

## What This Changes

The issue reports that a CLI paired with only `operator.read` can get stuck behind an auto-reissued full-scope repair request that cannot be approved or rejected deterministically.

## Best Possible Solution

Define a bounded same-device recovery contract that makes read-only CLI repair recovery deterministic, then align gateway authorization, CLI fallback behavior, docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` cannot call `device.pair.reject` because the global method-scope gate requires `operator.pairing` before the handler's same-device guard can run.

## Solution Assessment

Unclear. The existing approve fallback is useful but not sufficient for the reported rotating-request failure; choosing same-device reject, replacement, narrower repair scopes, or bootstrap behavior is a maintainer security/product decision.
