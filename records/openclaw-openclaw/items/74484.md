## Summary

Keep open. Current main still leaves the canonical pairing-recovery contract unresolved: a read-only paired device-token session cannot reach gateway approve/reject methods, local approve fallback is exact-request and local-state dependent, and a member is actively gathering fresh recovery data from the reporter today.

## What This Changes

The issue reports a read-only paired CLI getting stuck on rotating over-scoped gateway pairing repair requests that it cannot approve or reject reliably.

## Best Possible Solution

Define a maintainer-approved deterministic recovery contract for same-device repair requests, then align gateway authorization, CLI local fallback behavior, public docs, and regression tests around that contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-backed: seed a paired CLI device-token session with only `operator.read`, then attempt `device.pair.approve` or `device.pair.reject`; both are `operator.pairing` methods and fail before the same-device handler checks run.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The current approve fallback and scope-preservation work are useful partial fixes, but maintainers still need to choose the supported recovery contract before a narrow implementation is safe.
