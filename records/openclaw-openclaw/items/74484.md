## Summary

Keep open. Current main has partial approve/list recovery, but the read-scoped CLI still has no deterministic supported path to reject, downgrade, or replace its own over-scoped repair request without widening to the broad operator bundle.

## What This Changes

The issue asks for a deterministic recovery path when a CLI paired with only `operator.read` is blocked by an auto-reissued broad gateway pairing repair request.

## Best Possible Solution

Define a bounded pairing recovery contract for limited paired CLIs, then align gateway authorization, CLI fallback behavior, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` cannot reach `device.pair.reject` for its own pending request because the method requires `operator.pairing` and the global gateway method gate runs before the same-device handler guard.

## Solution Assessment

Unclear. The current approve fallback helps one recovery path, but the best final behavior needs maintainer/security choice between same-device self-reject, local reject fallback, narrower repair-scope requests, or another bootstrap recovery flow.
