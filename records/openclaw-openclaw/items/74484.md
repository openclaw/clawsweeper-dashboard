## Summary

Keep open. Current main still leaves a CLI paired only with `operator.read` unable to reject its own broader pending repair request through the gateway, and the linked open PR context covers adjacent approve/list fallback work rather than this recovery deadlock.

## What This Changes

The issue requests a supported recovery path for a read-only paired CLI stuck behind an auto-reissued broader device repair request it cannot approve or reject.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes. A paired device-token session with only `operator.read` can reproduce the deadlock by calling `device.pair.reject` for its own pending repair request; global method-scope authorization rejects it before handler-local ownership checks run.

## Solution Assessment

Unclear. The issue identifies a real recovery gap, but the narrowest maintainable fix needs maintainer/security choice between self-reject, narrower requested scopes, or a separate bootstrap recovery path.
