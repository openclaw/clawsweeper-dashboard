## Summary

Keep open. Current main still has only partial CLI pairing recovery: approve/list have a local fallback, but reject remains pairing-scoped and the approve fallback still depends on an exact request id that can be superseded during reconnect.

## What This Changes

The issue requests a deterministic recovery path for a read-only paired CLI whose broader repair pairing request rotates and cannot be approved or rejected through normal gateway methods.

## Best Possible Solution

Define a bounded recovery contract for same-device repair requests, then align gateway authorization, CLI fallback behavior, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` still fails the global `operator.pairing` gate for approve/reject, and the local approve fallback can only approve the exact request id it was given.

## Solution Assessment

Unclear. The current partial fallback is not enough; maintainers should choose whether the supported fix is same-device self-reject, latest-id local approve, narrower repair requests, or another bootstrap flow.
