## Summary

Keep open. Current main and v2026.5.19 include partial pairing recovery fixes, but the CLI-specific recovery contract is still unresolved for a paired device stuck with only `operator.read`, especially rejection and deterministic approval of its own over-scoped repair request.

## What This Changes

A read-only paired CLI cannot deterministically approve or reject its own broad scope-upgrade repair request, leaving gateway control-plane actions without a supported recovery path.

## Best Possible Solution

Define a bounded same-device pairing recovery contract for limited paired CLIs, then align gateway authz, CLI fallback behavior, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. A read-only operator session cannot reach `device.pair.reject` because the method requires `operator.pairing`, and approve fallback still depends on matching live/local request state.

## Solution Assessment

Unclear. Existing fallback and reconnect fixes reduce adjacent churn, but maintainers still need to choose the supported recovery contract before this can be safely fixed.
