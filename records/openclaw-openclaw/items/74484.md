## Summary

Keep open. Current main has a partial approve/list fallback and the reporter later recovered by accepting the broad repair request, but the read-only CLI still lacks a deterministic supported way to reject, downgrade, or replace its own over-scoped pending repair request without a pairing/security policy decision.

## What This Changes

The issue asks for a supported recovery path when a paired CLI has only `operator.read` while an auto-reissued repair request asks for the full default operator scope bundle.

## Best Possible Solution

Define and implement one bounded recovery contract for limited paired CLIs, then align gateway authorization, CLI behavior, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` and an own-device pending repair request will fail `device.pair.reject` at the global method-scope gate before the handler can apply its same-device guard.

## Solution Assessment

Unclear. Current main's broad local approve fallback can recover some installations, but the durable fix needs maintainer/security choice on whether limited CLIs may self-reject, locally reject, request narrower repair scopes, or follow a documented approve-only bootstrap path.
