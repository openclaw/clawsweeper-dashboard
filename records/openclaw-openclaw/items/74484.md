## Summary

Keep open. Current main still gates `device.pair.reject` behind `operator.pairing` before the same-device ownership check can run, and the CLI/docs still only provide local fallback for list/approve; related #75820 is adjacent but does not resolve same-device reject or narrower repair-scope recovery.

## What This Changes

The issue requests a supported recovery path for a read-only paired CLI stuck behind an auto-reissued, over-scoped device repair request.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI session with only `operator.read`, a broader pending repair request for the same device, and `openclaw devices reject <requestId>` failing at global gateway method authorization before the same-device reject handler can run.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the best fix requires maintainer/security choice between same-device self-reject, narrower repair-scope requests, or another bootstrap recovery flow.
