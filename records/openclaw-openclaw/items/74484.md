## Summary

Keep open: current main still blocks a read-only paired CLI from reaching `device.pair.reject`, and the existing approve/list fallback is only a partial recovery path for the timing-sensitive request-id churn described by the reporter.

## What This Changes

The issue asks for a deterministic recovery path when a read-only paired CLI is blocked by auto-reissued broader pairing repair requests.

## Best Possible Solution

Define a bounded recovery contract for a limited paired CLI to clear, replace, or approve only its own pending repair request, then align gateway authorization, CLI retry behavior, docs, and regression tests.

## Reproduction Assessment

Yes, source-backed. A device-token session with only `operator.read` calling `device.pair.reject` fails the global `operator.pairing` method-scope check before the same-device rejection guard can inspect the pending request.

## Solution Assessment

No. Current main's local approve fallback is useful but incomplete; the best fix still needs an explicit maintainer/security choice for reject, downgrade, narrower repair scopes, or documented approve-only recovery.
