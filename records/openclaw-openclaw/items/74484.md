## Summary

Keep open. Current main has useful recovery hardening, but the deterministic recovery contract is still unresolved for a paired CLI that only has `operator.read`: gateway reject is still gated before handler-local self checks, approve remains exact-request/local-state dependent, and fresh discussion shows a maintainer is still gathering state from the reporter.

## What This Changes

The issue reports a gateway pairing recovery deadlock where a read-only paired CLI cannot reliably approve or reject rotating over-scoped repair requests.

## Best Possible Solution

Define a bounded pairing recovery contract for read-only or stale paired CLIs, then align gateway authorization, CLI local fallback behavior, docs, and regression coverage so users can deterministically approve or reject their own current repair request without widening unrelated authority.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` cannot call `device.pair.reject` because the method is gated at `operator.pairing` before the handler can apply same-device checks; approve can still fail if the exact pending request is replaced before local fallback acts.

## Solution Assessment

Unclear. Current main's approve fallback and diagnostics are useful partial fixes, but maintainers still need to choose the supported recovery contract for self-reject, request pinning, narrower repair requests, or another bootstrap path.
