## Summary

Keep open. Current main still leaves a read-only paired CLI without a deterministic supported way to reject or downgrade its own broad repair request; the reporter recovered once through approve fallback, but confirmed the recovery path is still timing-sensitive and unclear.

## What This Changes

Allow a read-only paired CLI caught in a broad repair pairing loop to deterministically clear, replace, or request a narrower scope upgrade without needing `operator.pairing` first.

## Best Possible Solution

Define a bounded recovery contract for own-device repair requests, then align gateway authorization, CLI behavior, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` calling `device.pair.reject` for its own pending repair request will fail at the global `operator.pairing` method gate before handler ownership checks can run.

## Solution Assessment

Unclear. The current approve fallback is useful, but the final fix needs a maintainer/security choice between same-device self-reject, local fallback, narrower repair-scope requests, or another documented bootstrap flow.
