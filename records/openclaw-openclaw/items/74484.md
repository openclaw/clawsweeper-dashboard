## Summary

Keep open. Current main partially improves recovery with local list/approve fallback and request-id guidance, but the read-only paired CLI still lacks a deterministic way to reject, downgrade, or replace its own over-scoped repair request without approving broader access.

## What This Changes

The issue requests deterministic recovery for a CLI paired with only `operator.read` when auto-reissued repair requests ask for broader operator scopes.

## Best Possible Solution

Define a bounded recovery contract that lets a limited local CLI clear or replace only its own pending repair request without approving broader access, then document and test the request-id retry behavior.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` cannot authorize `device.pair.reject`, and the global scope gate runs before the handler's same-device rejection guard.

## Solution Assessment

Unclear. Current main's approve fallback is useful but not a complete fix; the safest path needs a maintainer-chosen recovery contract before code changes.
