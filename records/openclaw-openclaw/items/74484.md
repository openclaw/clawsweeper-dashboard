## Summary

Keep open. Current main has useful partial recovery for compatible same-device request replacement, but the canonical under-scoped CLI recovery problem remains live: a latest-release user still reports rotating request IDs and impossible approval, and the remaining fix crosses the pairing authority boundary.

## What This Changes

The issue requests a deterministic recovery path when an already paired CLI lacks enough scopes to approve or reject auto-reissued repair requests.

## Best Possible Solution

Keep this as the canonical issue until maintainers define and implement a bounded CLI recovery contract for under-scoped paired devices, then cover gateway auth, CLI fallback, docs, and regression tests together.

## Reproduction Assessment

Yes, source-reproducible, though not live-reproduced in this read-only pass. Current tests and code preserve the reported `unknown requestId` fail-closed path, and a latest-release user reports the same rotating-request deadlock on v2026.6.6.

## Solution Assessment

Unclear. The compatible replacement fallback is a useful partial fix, but the best remaining solution requires a maintainer choice between bootstrap, downgrade, reject, or further bounded same-device approval semantics.
