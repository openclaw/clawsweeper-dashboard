## Summary

Keep open. Current main still matches the reported recovery deadlock: `device.pair.reject` is pairing-scoped, gateway method authorization runs before the same-device reject handler, and the CLI only has local fallback paths for list/approve, not reject. Related #74650 overlaps pairing approval loops but does not supersede this narrower CLI recovery gap.

## What This Changes

The issue requests a supported recovery path when a paired CLI with only `operator.read` receives repeated broader repair requests and cannot approve, reject, or downgrade them without `operator.pairing`.

## Best Possible Solution

Define a bounded recovery contract for limited paired devices, then implement it consistently across gateway authorization, CLI fallback behavior, docs, and regression tests. The desired end state is that a read-only CLI can clear or replace only its own pending repair request without approving broader scopes.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI session with only `operator.read`, a broader pending repair request for the same device, and `openclaw devices reject <requestId>` failing at the global pairing-scope gate before the same-device reject handler can run.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the narrowest maintainable fix needs maintainer/security choice between same-device self-reject, narrower repair-scope requests, or a separate bootstrap recovery flow.
