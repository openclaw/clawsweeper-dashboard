## Summary

Keep open. Current main still leaves a read-only paired CLI unable to clear its own broader pending repair request through the gateway, and the adjacent open PR context does not cover same-device reject or narrower repair-scope recovery.

## What This Changes

The issue asks for a supported recovery path when a CLI paired only with `operator.read` is stuck behind an auto-reissued broader gateway pairing repair request.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes. A source-backed reproduction is a paired device-token session with only `operator.read`, a broader pending request for the same device, and `device.pair.reject` failing at the global method-scope authorization before the handler's same-device check can run.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the safest fix needs maintainer/security choice between same-device self-reject, local loopback fallback, narrower repair-scope requests, or another bootstrap recovery flow.
