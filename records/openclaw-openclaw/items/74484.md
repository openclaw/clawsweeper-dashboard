## Summary

Keep open. Current main still leaves the reported recovery deadlock in place for rejection: `device.pair.reject` is globally gated by `operator.pairing` before the device handler can apply its same-device ownership checks, and the CLI has local fallback paths for list/approve but not reject. The related closed reports cover approve/list or read-scope behavior, while this issue still needs a maintainer decision on reject, downgrade, or narrower repair-scope recovery.

## What This Changes

The issue requests a supported recovery path for a CLI paired with only `operator.read` when it receives an auto-reissued broader repair request that it cannot approve, reject, downgrade, or replace with narrower scopes.

## Best Possible Solution

Define a bounded same-device recovery model for local repair deadlocks, then implement it consistently in gateway authorization, device pairing handlers, CLI behavior, docs, and regression tests. The end state should let a limited CLI clear or replace only its own pending repair request without silently approving broader scopes.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired device session with only `operator.read`, an existing broader pending repair request for that same device, and `openclaw devices reject <requestId>` failing at gateway method authorization because `device.pair.reject` requires `operator.pairing`.

## Solution Assessment

Unclear. The report identifies a real issue, but the best fix is a maintainer/security choice between same-device self-reject fallback, narrower repair-scope requests, or another documented bootstrap flow.
