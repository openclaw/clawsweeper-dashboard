## Summary

Keep open. Current main still blocks the read-scoped CLI at the global method-scope gate before same-device reject can run; the local fallback covers list/approve only, and related #75820 is adjacent but does not resolve same-device reject or narrower repair-scope recovery.

## What This Changes

Request a supported gateway/CLI recovery path for a paired CLI with only `operator.read` that is stuck behind auto-reissued broader repair requests it cannot approve or reject.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without granting broader scopes, then align gateway authorization, CLI fallback behavior, docs, and regression tests.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI session with only `operator.read`, a broader pending repair request for the same device, and `openclaw devices reject <requestId>` failing at global gateway authorization before the same-device reject handler runs.

## Solution Assessment

Unclear. The report identifies a real deadlock, but the best fix needs maintainer/security choice between same-device self-reject, narrower repair-scope requests, or another bootstrap recovery flow.
