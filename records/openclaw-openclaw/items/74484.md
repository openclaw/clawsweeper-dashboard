## Summary

Keep open. Current main still has only a partial recovery path: local loopback approve/list fallback exists, but read-only paired CLI sessions still cannot reject their own pending repair request through the gateway, and the reporter’s latest comments say the eventual approve recovery was timing-sensitive rather than deterministic.

## What This Changes

Request a deterministic gateway/CLI recovery path for a CLI paired with only `operator.read` when a broader auto-reissued repair pairing request blocks control-plane operations.

## Best Possible Solution

Define and test a bounded recovery contract that lets a limited local CLI clear, approve, or replace only its own pending repair request without broadening unrelated pairing authority, then document the supported flow.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` and a broader same-device pending request will fail `device.pair.reject` at the global method-scope gate before the handler's same-device guard can run; approve has only the existing local loopback fallback path.

## Solution Assessment

Unclear. The current local approve fallback can recover some installations, but the maintainable final fix needs a security/product decision between deterministic self-reject, narrower repair-scope requests, a stronger approve retry contract, or another bootstrap path.
