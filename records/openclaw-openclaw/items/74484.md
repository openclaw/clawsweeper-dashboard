## Summary

Keep open. Current main still has the reported recovery deadlock: `device.pair.reject` is globally gated on `operator.pairing`, the gateway authorizes before reaching the same-device reject checks, and the CLI has local fallback paths for list/approve but not reject.

## What This Changes

Allow a CLI paired with only `operator.read` to recover from its own auto-reissued broader device repair request without already having `operator.pairing`.

## Best Possible Solution

Define a bounded recovery contract for limited paired devices, then implement it consistently in gateway authorization, CLI behavior, docs, and regression tests. The end state should let a read-only CLI clear or replace only its own pending repair request without approving broader scopes or affecting other devices.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI session with only `operator.read`, a broader pending repair request for the same device, and `openclaw devices reject <requestId>` failing before the reject handler because the RPC requires `operator.pairing`.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the best fix needs maintainer/security direction on whether to add same-device self-reject, support narrower repair-scope requests, or provide a different bootstrap recovery flow.
