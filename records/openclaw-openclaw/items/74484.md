## Summary

Keep open. Current main still leaves a read-only paired CLI unable to clear its own broader pending repair request because `device.pair.reject` remains pairing-scoped and the CLI reject command has no local fallback; the recovery authority model needs maintainer/security selection.

## What This Changes

The issue asks for a supported recovery path when a read-only paired CLI cannot reject, downgrade, or narrowly reissue an auto-reappearing broader pairing repair request.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: connect as a device-token CLI with only `operator.read`, create a broader pending repair request for the same device, then `device.pair.reject` is blocked by global method-scope authorization before the handler's same-device guard can run.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the safest solution needs maintainer/security choice between same-device self-reject, local loopback fallback, narrower repair-scope requests, or another bootstrap recovery flow.
