## Summary

Keep open. Current main has useful pairing recovery work, but it still does not fully settle this report: a read-only paired CLI has no reject fallback, approve remains exact-request dependent, and a maintainer is actively asking for fresh redacted device state from the reporter rather than treating the issue as resolved.

## What This Changes

The issue asks for a deterministic recovery path when a CLI paired with only `operator.read` cannot approve or reject rotating over-scoped gateway pairing repair requests.

## Best Possible Solution

Define a bounded pairing recovery contract for limited paired CLIs, then align gateway authorization, CLI local recovery behavior, public docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, source-backed. A device-token session with only `operator.read` cannot reach `device.pair.reject`, and approve fallback can still fail when the exact pending request is superseded or absent before the fallback acts.

## Solution Assessment

Unclear. The current approve fallback and request-id guards are useful, but maintainers still need to choose the supported recovery contract, such as same-device self-reject, a local reject fallback, narrower repair requests, or documented bootstrap recovery.
