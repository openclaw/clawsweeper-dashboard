## Summary

Keep open. Current main still leaves a read-only paired CLI unable to reject its own over-scoped repair request through the gateway, and the adjacent open PR does not resolve same-device reject or narrower repair-scope recovery.

## What This Changes

The issue asks for a recovery path when a CLI paired with only `operator.read` gets stuck behind auto-reissued broader device repair requests it cannot approve or reject.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request, or request a narrower approved scope set, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI device with only `operator.read`, a same-device pending repair request for broader operator scopes, and `openclaw devices reject <requestId>` failing at the global `device.pair.reject` method-scope gate before the same-device handler guard can run.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the safest solution needs maintainer/security choice between bounded same-device self-reject, narrower repair-scope requests, or a separate bootstrap recovery path.
