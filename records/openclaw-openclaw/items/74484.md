## Summary

Keep open. Current main partially narrows classified CLI gateway requests, but a read-only paired CLI still cannot clear its own pending repair request because reject is globally pairing-scoped and has no local fallback.

## What This Changes

Allow a CLI paired only with `operator.read` to recover from broader auto-reissued repair pairing requests by rejecting, downgrading, or requesting narrower scopes without already having `operator.pairing`.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes. A source-backed reproduction is a paired device-token session with only `operator.read`, a broader pending request for the same device, and `openclaw devices reject <requestId>` failing before the reject handler because the gateway requires `operator.pairing`.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the safest fix needs maintainer/security choice between same-device self-reject, a local loopback fallback, narrower repair-scope requests, or another bootstrap flow.
