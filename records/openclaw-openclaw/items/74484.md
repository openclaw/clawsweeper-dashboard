## Summary

Keep open. Current main still lacks a deterministic recovery path for this state: list/approve have a local loopback fallback, but reject still goes through the pairing-scoped RPC, gateway method authorization runs before the same-device handler guard, and the docs only promise local fallback for list/approve. The reporter's later recovery confirms the immediate machine was unblocked, but also confirms the path remains timing-sensitive around rotating request IDs.

## What This Changes

A read-only paired CLI can get stuck behind broader auto-reissued device repair or scope-upgrade requests, leaving approve/reject recovery fragile and control-plane operations blocked.

## Best Possible Solution

Define a deterministic, bounded pairing recovery contract for a limited paired CLI to clear or replace only its own pending repair request, then align gateway authorization, CLI behavior, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. A paired CLI with only `operator.read` cannot call `device.pair.reject` because the method requires `operator.pairing` and gateway authorization runs before the handler's same-device guard; the rotating approve retry race was not live-reproduced.

## Solution Assessment

Unclear. The issue identifies a real recovery gap, but the safest fix requires a maintainer/security choice between same-device self-reject, stable approve-by-current-request behavior, narrower repair-scope requests, or another bootstrap path.
