## Summary

Keep open. Current main still leaves a read-only paired CLI unable to reject its own over-scoped repair request through the gateway, and the documented fallbacks only cover list/approve while the adjacent open PR does not address same-device reject or narrower repair-scope recovery.

## What This Changes

Request a supported recovery path for a CLI paired with only `operator.read` when an auto-reissued repair request asks for broader operator scopes.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI session with only `operator.read`, a broader pending repair request for the same device, and `openclaw devices reject <requestId>` failing at the global gateway method-scope check before the same-device reject handler can run.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the safest fix needs maintainer/security choice between same-device self-reject, narrower repair-scope requests, or another bootstrap recovery flow.
