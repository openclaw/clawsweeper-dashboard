## Summary

Keep open. Current main has a partial same-device approval fallback, but it does not implement a deterministic recovery contract for this deadlock: stale approve paths can still end in `unknown requestId`, `devices reject` remains Gateway-only, and newer duplicate reports confirm the CLI scope-upgrade loop remained user-visible after the partial fixes.

## What This Changes

The issue asks for a deterministic CLI recovery path when an already-paired CLI has insufficient operator scopes and its pending scope-upgrade repair request churns or cannot be approved/rejected.

## Best Possible Solution

Define and implement one maintainer-approved CLI pairing recovery contract that keeps same-device security checks intact, handles request-id churn deterministically, covers reject or a documented downgrade/recovery path, and aligns docs/tests with that contract.

## Reproduction Assessment

Yes, at source level with medium confidence: current main still has a paired-device scope gate, broad CLI default upgrade requests, a partial approve fallback, and Gateway-only reject. I did not run a fresh live reproduction in this read-only review.

## Solution Assessment

No. The merged same-device approval fallback is a useful partial mitigation, but the best fix still needs a maintainer-approved recovery contract for stale request IDs, reject/downgrade behavior, and local authority boundaries.
