## Summary

Keep open. Current main still has the reported pairing recovery deadlock: `device.pair.reject` remains behind the global `operator.pairing` method gate, so the same-device rejection guard in the device handler is unreachable for a CLI paired with only `operator.read`; the related open ACP approval-loop report does not supersede this narrower CLI recovery issue.

## What This Changes

The issue asks for a supported recovery path when a CLI paired with only `operator.read` cannot reject, downgrade, or narrowly repair an auto-reissued over-scoped device pairing request.

## Best Possible Solution

Define an explicit bounded recovery contract for limited paired devices, then implement it consistently in gateway authorization, CLI behavior, docs, and regression tests. The target state should let a read-only CLI clear or replace only its own pending repair request without approving broader scopes by accident.

## Reproduction Assessment

Yes. A source-backed reproduction is a paired CLI session with only `operator.read`, a broader pending repair request for the same device, and `openclaw devices reject <requestId>` failing before the handler because `device.pair.reject` requires `operator.pairing`.

## Solution Assessment

Unclear. The report identifies a real deadlock, but the best fix needs maintainer/security direction on whether to add same-device self-reject, support narrower repair-scope requests, or provide another bootstrap recovery path.
