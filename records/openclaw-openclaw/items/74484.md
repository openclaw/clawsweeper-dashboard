## Summary

Keep open. Current main still has the deadlock shape for a CLI that can connect only with `operator.read`: reject is globally pairing-scoped, local approve still approves the exact request ID without replacement re-resolution, and the linked PR that may address this surface is still an open implementation candidate rather than shipped main behavior.

## What This Changes

The issue requests a deterministic recovery path for a read-only paired CLI stuck behind auto-reissued, broader device-pairing repair requests.

## Best Possible Solution

Define and land a bounded recovery contract that makes local loopback CLI recovery deterministic for the caller's own repair request, then update gateway auth, CLI fallback behavior, docs, and regression coverage together.

## Reproduction Assessment

Yes, source-backed. A paired CLI session limited to `operator.read` cannot call `device.pair.reject`, and the current local approve fallback can still fail with `unknown requestId` when the repair request is replaced before fallback approval.

## Solution Assessment

Unclear. Current main's list/approve fallback and replacement-ID reporting are useful partial fixes, but the final path should be chosen deliberately between replacement-aware approve, same-device self-reject, narrower repair scopes, or another bootstrap recovery flow.
