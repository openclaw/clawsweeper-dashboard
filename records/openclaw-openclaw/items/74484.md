## Summary

Keep open. Current main and v2026.6.10 partially mitigate compatible same-device approve churn, but the broader CLI recovery contract is still unresolved: `devices reject` remains Gateway-only, pairing RPCs still require `operator.pairing`, and related open work only covers narrower slices.

## What This Changes

The issue requests a deterministic, supported recovery path for under-scoped paired CLI devices stuck behind auto-reissued over-scoped repair requests.

## Best Possible Solution

Define the supported under-scoped CLI recovery contract, keep Gateway scope-upgrade protections intact, and land narrow tests/docs for approve, reject, requestId churn, and local-vs-remote fallback boundaries.

## Reproduction Assessment

Yes, at source level with medium confidence: the remaining reject path is Gateway-only while the method table requires `operator.pairing`, and current tests still preserve some stale `unknown requestId` cases. I did not run a live stale-device repro in this read-only review.

## Solution Assessment

No, the current implementation is only a partial solution. The merged approve fallback is useful, but the best fix still needs a maintainer security/product decision on deterministic recovery semantics for reject and requestId churn.
