## Summary

Keep open. Current main has useful partial recovery for `devices list` and `devices approve`, but a read-only paired CLI still cannot reject its own rotating over-scoped repair request through the gateway, and the deterministic supported recovery contract remains unresolved.

## What This Changes

The issue requests a deterministic recovery path for a read-only paired CLI stuck with rotating over-scoped gateway pairing repair requests.

## Best Possible Solution

Define a bounded pairing recovery contract for the local CLI, then align gateway authorization, CLI fallback behavior, docs, and regression tests so users can recover without racing request churn or approving unclear scopes.

## Reproduction Assessment

Yes, source-backed. A device-token session with only `operator.read` cannot pass the global `operator.pairing` gate for `device.pair.reject`, and the CLI reject command has no local fallback to bypass that dead end on loopback.

## Solution Assessment

Unclear. Current main's approve/list local fallback is useful, but the final fix needs a maintainer security choice between same-device self-reject, stale request-id re-resolution, narrower repair-scope requests, clearer documentation, or another bootstrap flow.
