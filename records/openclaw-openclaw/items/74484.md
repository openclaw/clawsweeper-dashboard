## Summary

Keep open. Current main has useful partial recovery for local `devices list` and `devices approve`, and the reporter later recovered one machine through that path, but source and docs still show no deterministic supported way for a read-only paired CLI to reject, downgrade, or replace its own over-scoped repair request without approving the broad bundle.

## What This Changes

The issue requests a deterministic recovery path for a CLI paired with only `operator.read` when the gateway keeps reissuing a broader pending repair request.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without granting broader scopes, then align gateway authorization, CLI fallback behavior, docs, and regression tests.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` calling `device.pair.reject` will fail the global method-scope check before the same-device reject guard can inspect ownership.

## Solution Assessment

Unclear. Current main's local list/approve fallback and live request-id re-resolution are useful partial fixes, but the best final path needs a maintainer/security decision between self-reject, local reject fallback, narrower repair-scope requests, or another bootstrap flow.
