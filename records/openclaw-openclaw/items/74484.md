## Summary

Keep open. Current main and the latest release still have only partial recovery coverage: list/approve can use local fallback, but reject is still gateway-only and pending request replacement can still leave a read-only paired CLI with no deterministic supported recovery path.

## What This Changes

The issue asks for a reliable recovery path when a read-only paired CLI keeps receiving broad auto-reissued repair requests that it cannot approve or reject through the gateway.

## Best Possible Solution

Define and document a bounded pairing recovery contract for limited paired CLIs, then align gateway authorization, CLI fallback behavior, request-id handling, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed rather than live. A read-only paired CLI cannot use gateway approve/reject without operator.pairing, and current pending replacement logic can still invalidate a visible requestId before approval resolves.

## Solution Assessment

Unclear. The current approve fallback and token-scope fixes are useful partial repairs, but the best remaining fix needs a maintainer security/product choice between same-device self-reject, local reject fallback, narrower repair requests, or another bootstrap recovery flow.
