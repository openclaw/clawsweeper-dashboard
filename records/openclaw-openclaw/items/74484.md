## Summary

Keep open. Current main still has only partial recovery: list/approve can use local fallback on loopback, but a read-only paired CLI still cannot reject, downgrade, or replace its own over-scoped repair request through a deterministic supported path.

## What This Changes

The issue asks for a supported gateway/CLI recovery path when a read-only paired CLI is stuck behind an auto-reissued over-scoped repair request.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway auth, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes, source-backed. A device-token session with only `operator.read` will fail `device.pair.reject` at the global method-scope gate before the same-device reject guard can run.

## Solution Assessment

Unclear. Current main's approve fallback is useful partial recovery, but the best final path needs maintainer/security choice between same-device self-reject, local reject fallback, narrower repair-scope requests, or another bootstrap flow.
