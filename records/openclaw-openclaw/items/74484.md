## Summary

Keep open. Current main has useful partial fixes for pairing approval fallback and token-scope preservation, but the canonical recovery contract is still unresolved: a read-only paired CLI cannot reject its own pending repair request through the gateway, and only list/approve have local fallback coverage.

## What This Changes

The issue asks for a deterministic recovery path when a read-only paired CLI is blocked by auto-reissued, over-scoped device repair requests.

## Best Possible Solution

Define a bounded pairing recovery contract for limited paired CLIs, then align gateway authorization, CLI local recovery behavior, public docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, source-backed rather than live. A device-token session with only operator.read cannot call device.pair.reject because the global method-scope gate requires operator.pairing before the same-device handler guard runs, and related reports document the same request-id churn during repair approval.

## Solution Assessment

Unclear. The approve fallback and token-scope fixes are useful partial solutions, but the remaining best fix needs a maintainer security/product choice between same-device self-reject, local reject fallback, narrower repair requests, or another bootstrap recovery flow.
