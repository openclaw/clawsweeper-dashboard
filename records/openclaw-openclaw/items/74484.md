## Summary

Keep open. Current main still has a source-backed pairing recovery gap: a read-only paired CLI cannot reject or downgrade its own broader repair request, and the documented local fallback only covers list/approve rather than a deterministic recovery path.

## What This Changes

The issue asks for a deterministic CLI recovery path when a read-only paired device gets an auto-reissued broader operator-scope repair request.

## Best Possible Solution

Define a bounded recovery contract for over-scoped repair requests, then align gateway authorization, CLI fallback/error handling, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` cannot pass the global `operator.pairing` gate for `device.pair.reject`, so the handler's same-device rejection guard is unreachable.

## Solution Assessment

Unclear. Current main's approve fallback is useful, but the best final fix needs maintainer/security choice between same-device self-reject, local reject fallback, narrower repair requests, or another documented bootstrap path.
