## Summary

Keep open. Current main still leaves a read-only paired CLI unable to reach the gateway reject path for its own broader pending repair request, and the related open PR context covers nearby list/approve fallback behavior rather than this same-device reject or narrower repair-scope recovery deadlock.

## What This Changes

The issue requests a supported recovery path for a read-only paired CLI stuck behind an auto-reissued broader device repair pairing request.

## Best Possible Solution

Define a bounded recovery contract that lets a limited paired CLI clear or replace only its own pending repair request without approving broader scopes, then align gateway authorization, CLI behavior, docs, and regression tests.

## Reproduction Assessment

Yes, source-backed rather than live-run. A paired device-token session with only `operator.read` cannot authorize `device.pair.reject`, and current dispatch checks global method scope before the handler's same-device reject guard.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the safest fix needs maintainer/security choice between a same-device auth exception, local reject fallback, narrower repair-scope requests, or another bootstrap recovery path.
