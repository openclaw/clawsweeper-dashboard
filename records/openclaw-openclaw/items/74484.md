## Label Justifications

- `impact:security`: The requested recovery path changes who may approve, reject, or downgrade broader device/operator scopes and must preserve the pairing authorization boundary.

## Summary

Keep open. Current main and v2026.6.10 still leave the broader CLI recovery contract unresolved: `devices approve` has a bounded same-device local fallback, but `devices reject` remains Gateway-only, pairing methods still require `operator.pairing`, and recent related reports confirm the under-scoped CLI scope-upgrade loop remains a real class of failure.

## What This Changes

The issue requests a deterministic recovery path for a paired CLI that has insufficient operator scope and cannot approve or reject an auto-reissued broader repair request.

## Best Possible Solution

Define one deterministic, documented recovery contract for under-scoped local CLI devices, covering approve, reject, requestId churn, and safe bootstrap authority without weakening remote or cross-device pairing checks.

## Reproduction Assessment

Yes at source level, with medium confidence: start from a paired CLI whose stored operator token lacks the scopes needed by a broader repair request, then Gateway approval/rejection depends on `operator.pairing`/admin while only approve has a bounded local fallback. I did not run the live packaged repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security contract. The merged same-device replacement approval fix is a good partial repair, but it is not the complete supported recovery path for reject, downgrade, or under-scoped bootstrap behavior.
