## Summary

Keep open. Current main has a bounded same-device approval fallback, but the broader CLI recovery contract is still unresolved: reject remains Gateway-only, the method gate still requires pairing scope, and a newer user report confirms the deadlock class after the partial fixes.

## What This Changes

The issue reports a CLI/Gateway pairing recovery deadlock where an under-scoped paired CLI cannot deterministically approve or reject auto-reissued scope-upgrade repair requests.

## Best Possible Solution

Define one deterministic, documented recovery contract for under-scoped local CLI devices, including approve and reject behavior, requestId churn handling, and the security limits for any local fallback.

## Reproduction Assessment

Yes at source level, with medium confidence: start from a paired CLI whose stored operator token lacks the scopes needed by a broader repair request, then Gateway approval/rejection depends on operator.pairing/admin while only approve has a bounded local fallback. I did not run a live packaged repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security contract. The current same-device replacement approval fix is a good partial repair, but it is not the complete supported recovery path for reject, downgrade, or under-scoped bootstrap behavior.
