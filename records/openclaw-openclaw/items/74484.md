## Summary

Keep open. Current main has useful approve/list fallback work, but the reported recovery gap remains: a read-only paired CLI still cannot reject or downgrade its own over-scoped repair request, and the deterministic recovery contract is still a pairing-security/product decision.

## What This Changes

The issue requests a deterministic recovery path for a CLI paired with only `operator.read` when auto-reissued repair requests ask for broader operator scopes.

## Best Possible Solution

Define a bounded same-device repair recovery contract, then align gateway authorization, CLI behavior, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, source-backed. A paired device-token session with only `operator.read` calling `device.pair.reject` will fail at the global `operator.pairing` method-scope gate before the handler can check that the pending request belongs to the same device.

## Solution Assessment

Unclear. Current main's local approve fallback is a useful recovery path, but the best final fix still needs maintainer choice between constrained same-device self-reject, narrower repair-scope requests, clearer approve retry behavior, or documented bootstrap guidance.
