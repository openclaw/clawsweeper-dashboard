## Summary

Keep open. Current main has partial approve/list recovery and request-id supersede handling, but a read-only paired CLI still has no deterministic way to reject or downgrade its own over-scoped repair request without widening approval.

## What This Changes

The issue requests a deterministic recovery path for a read-only paired CLI stuck behind an auto-reissued, over-scoped device repair request.

## Best Possible Solution

Define a bounded pairing recovery contract for limited paired CLIs, then align gateway authorization, CLI fallback behavior, public docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, source-backed. A device-token session with only operator.read cannot call device.pair.reject because global method authorization requires operator.pairing before the handler can enforce same-device ownership.

## Solution Assessment

Unclear. Current approve fallback and request-id re-resolution are useful partial fixes, but the remaining fix needs a maintainer/security choice between same-device self-reject, local reject fallback, narrower repair requests, or another bootstrap recovery flow.
