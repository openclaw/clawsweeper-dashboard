## Summary

Keep open. Current main still matches the reported recovery deadlock: a read-only paired CLI cannot reach `device.pair.reject`, the reject handler's same-device checks sit behind the global method-scope gate, and the CLI/local docs only cover list/approve fallback. The reporter's follow-up confirms this is blocking real recovery, and the adjacent ACP approval-loop report is related but does not supersede this narrower pairing recovery issue.

## What This Changes

The issue requests a supported recovery path for a CLI paired with only `operator.read` when an over-scoped same-device repair request keeps being reissued and cannot be approved or rejected.

## Best Possible Solution

Define a bounded recovery contract for limited paired devices, then implement it consistently in gateway authorization, CLI fallback behavior, docs, and regression tests. The end state should let a read-only CLI clear or replace only its own pending repair request without approving broader scopes.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI session with only `operator.read`, a broader pending repair request for the same device, and `openclaw devices reject <requestId>` failing at global gateway method authorization before the same-device reject handler can run.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the best fix needs maintainer/security choice between same-device self-reject, narrower repair-scope requests, or a separate bootstrap recovery flow.
