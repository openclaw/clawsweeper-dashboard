## Summary

Keep open: this PR targets a real Claude live-session stuck-session bug, but the current patch is too broad because it makes every active-tool heartbeat passive and can regress the shipped protection for valid quiet Claude-native tools.

## What This Changes

The PR changes diagnostic run-activity bookkeeping so `cli_live:tool_running` progress events are emitted but no longer create or refresh session activity, and updates focused tests around Claude live active-tool heartbeats.

## Best Possible Solution

Rework the diagnostic boundary so synthetic heartbeats do not hide truly wedged missing-result tools while preserving the shipped protection for valid quiet Claude-native tools, with real-runtime proof for both cases.

## Reproduction Assessment

Yes at source level: current main emits `cli_live:tool_running` heartbeats while active tools remain in the Claude live map, and diagnostic activity currently records every `run.progress` reason as fresh progress. I did not run a live Claude CLI reproduction in this read-only review.

## Solution Assessment

No. The branch is a plausible diagnostic-boundary fix, but it is too broad because it demotes all active-tool heartbeats instead of distinguishing a wedged missing-result tool from a valid quiet foreground tool.
