## Summary

Keep open: current main and the latest release still only show ordinary chat-visible tool warnings as concise failures without safe failure reasons, and retry success only clears internal state rather than producing a recovery-aware visible outcome. The fresh Telegram/Dialpad production comment adds a concrete external side-effect risk, so this needs maintainer product and security policy rather than cleanup closure.

## What This Changes

The issue requests chat-visible tool failure messages that include a safe failure reason and confirm when a retry or recovery resolves the failure.

## Best Possible Solution

Define a channel-safe tool-failure recovery policy that redacts or categorizes visible reasons, suppresses stale recovered warnings when appropriate, and emits recovery confirmations only for matched failures.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.9 tests preserve ordinary warnings without failure details and show retry success clearing internal `lastToolError` state without a visible recovery confirmation.

## Solution Assessment

No, not as a raw always-show-reason patch. The maintainable solution is a channel-aware diagnostics and recovery-state policy with redaction, matching, and tests across recovered versus terminal failures.
