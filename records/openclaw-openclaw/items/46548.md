## Summary

Keep open. Current main still omits ordinary non-verbose failure reasons from chat-visible tool warnings and only clears same-action retry failures in state; the requested behavior is a valid UX enhancement but needs maintainer policy before exposing more diagnostics.

## What This Changes

Request chat-visible tool failure warnings to include safe failure reasons and emit a confirmation when a retry succeeds.

## Best Possible Solution

Define sanitized, allowlisted failure summaries by tool/channel, then emit retry-success confirmation only when a previously surfaced mutating failure is resolved by a matching successful action.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: build a verbose-off payload with a mutating `lastToolError` and the reason is omitted, then follow a failed edit through a matching successful edit where state clears without any chat payload.

## Solution Assessment

No current implementation fully solves it. The direct raw-error approach from the closed PR was not the best endpoint because it omitted retry confirmation and did not settle redaction/channel policy; the narrower maintainable path is sanitized summaries plus explicit retry-success rules.
