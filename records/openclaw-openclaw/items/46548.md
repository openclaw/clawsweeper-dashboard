## Summary

Keep open: current main and v2026.6.10 still omit ordinary chat-visible tool failure reasons outside full verbose or selected exec cases, and same-action retry success only clears internal state without a visible recovered/success confirmation. The fresh Telegram/Dialpad production datapoint makes this a real operator-safety issue, but the fix needs maintainer product/security policy before implementation.

## What This Changes

The issue asks for chat-visible tool failure messages to include a safe failure reason and to indicate when a retry or same-turn recovery later succeeded.

## Best Possible Solution

Define a channel-safe tool-failure diagnostics policy with redacted or allowlisted reason categories, then emit a recovery/confirmation only when a visible failure is resolved by a matching successful action.

## Reproduction Assessment

Yes, at source level: current main stores the tool error reason, but ordinary chat-visible warnings omit it and matching retry success only clears internal state. The June 23 production comment provides a concrete Telegram/Dialpad occurrence with the same stale failure-warning symptom after success.

## Solution Assessment

No for the old always-append-reason PR; that would expose raw tool errors into chat without a redaction or channel-trust policy. The better solution is a safe diagnostics/recovery policy at the embedded-run payload and tool-state boundary.
