## Summary

Keep open: current main still omits ordinary chat-visible failure reasons by default and only clears retry state internally after a matching success. The remaining work is valid, but it needs maintainer policy for redaction, channel visibility, and retry-confirmation behavior before implementation.

## What This Changes

The issue requests richer chat-visible tool failure messages that include the failure reason and a confirmation when a retry later succeeds.

## Best Possible Solution

Define a safe chat-visible diagnostics policy for tool failures, including redacted or allowlisted reasons by tool and channel, then add retry-success confirmation only when a surfaced failure is resolved by a matching successful action.

## Reproduction Assessment

Yes, by source inspection. Current main records the failure reason but default warning policy omits it from ordinary chat-visible warnings, and matching retry success only clears internal `lastToolError` state.

## Solution Assessment

No, not as raw error passthrough. The maintainable solution is a sanitized or allowlisted reason policy plus explicit retry-confirmation rules for matching surfaced mutating failures.
