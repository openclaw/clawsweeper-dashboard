## Summary

Keep open: current main still intentionally hides ordinary tool failure detail outside full verbose and clears retry state internally without emitting the requested retry-success confirmation. The linked failure-reason PR was closed unmerged as needing a different approach, while the open channel-leak issues overlap on safety policy rather than replacing this operator-diagnostics request.

## What This Changes

The issue asks OpenClaw to include a safe failure reason in chat-visible tool failure notices and confirm when a later retry succeeds.

## Best Possible Solution

Define a channel-safe diagnostics policy with redacted or allowlisted visible reasons, then emit retry-success confirmations only when a previously visible failure is resolved by a matching successful action.

## Reproduction Assessment

Yes, at source level. Current code and tests show ordinary non-verbose tool warnings omit the stored failure reason, and matching retry success only clears internal `lastToolError` state.

## Solution Assessment

No, the direct closed PR shape of always appending raw failure reasons is not the best fix. A maintainable solution needs a redacted/channel-aware diagnostics policy plus narrow retry-success semantics.
