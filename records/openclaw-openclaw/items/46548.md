## Summary

Keep open: current main still only partially satisfies the request because ordinary chat-visible tool warnings stay compact by default and matching retry success only clears internal state, while any broader reason surfacing needs a safe redaction/channel policy.

## What This Changes

The issue asks for chat-visible tool failure messages to include a safe failure reason and for matching retry successes to be confirmed to the user.

## Best Possible Solution

Define redacted or allowlisted tool-failure summaries by tool and channel, then emit retry-success confirmation only when a previously surfaced mutating failure is resolved by a matching successful action.

## Reproduction Assessment

Yes, by source inspection. Current main gates ordinary tool-error details behind `/verbose full` or narrow cron-timeout handling, and retry success only clears `lastToolError` internally.

## Solution Assessment

No, not as a raw-error passthrough. The maintainable path is sanitized failure summaries plus explicit retry-confirmation rules for matching surfaced mutating failures.
