## Summary

Keep open: current main still omits failure reasons from ordinary chat-visible tool warnings and clears retry state internally without a user-visible retry-success confirmation. The closed unmerged fix attempt acknowledged this needed a different approach, and the newer Telegram/privacy issue overlaps but does not replace this diagnostics-and-confirmation request.

## What This Changes

Request safe failure reasons in chat-visible tool failure warnings and confirmation when a matching retry later succeeds.

## Best Possible Solution

Define a channel-safe diagnostics policy with redacted or allowlisted visible reasons, then add retry-success confirmation only when a previously visible failure is resolved by a matching successful action.

## Reproduction Assessment

Yes, at source level. Current code and tests show ordinary non-verbose tool warnings omit failure details, and successful matching retries clear internal state without producing a visible confirmation.

## Solution Assessment

No, the closed direct approach of always appending raw failure reasons is not the best fix because it conflicts with channel privacy and security boundaries. A maintainable solution needs a redacted/channel-aware diagnostics policy plus narrow retry-success semantics.
