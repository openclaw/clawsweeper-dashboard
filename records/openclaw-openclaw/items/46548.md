## Summary

Keep open: current main and the latest release still omit ordinary chat-visible failure reasons by default and only clear retry state internally after a matching success. The remaining work is valid, but it needs maintainer policy for redaction, channel visibility, and retry-confirmation behavior before implementation.

## What This Changes

Request chat-visible tool failure warnings to include a safe failure reason and to confirm when a matching retry succeeds.

## Best Possible Solution

Define a safe chat-visible diagnostics policy for tool failures, including redacted or allowlisted reasons by tool and channel, then add retry-success confirmation only when a surfaced failure is resolved by a matching successful action.

## Reproduction Assessment

Yes, by source inspection. Current main records the failure reason, but default warning policy omits it from ordinary chat-visible warnings, and matching retry success only clears internal `lastToolError` state.

## Solution Assessment

No, raw error passthrough is not the best solution. The maintainable path is a sanitized or allowlisted reason policy plus explicit retry-confirmation rules for matching surfaced mutating failures.
