## Summary

Keep open: current main still does not provide the requested recovery-aware chat status, and the fresh Telegram/Dialpad production data makes this a real operator-safety and privacy-policy issue rather than stale cleanup.

## What This Changes

The issue requests safer chat-visible tool failure messages that include an appropriate reason and confirm or clear a retry/recovery when the same turn later succeeds.

## Best Possible Solution

Define a channel-safe diagnostics policy that redacts or allowlists visible failure reasons, then emit recovery-aware status only when a previously visible failure is resolved by a matching successful action in the same turn.

## Reproduction Assessment

Yes, at source level. Current code and tests show ordinary non-verbose warnings omit most stored failure reasons, and matching retry success clears only internal `lastToolError` state.

## Solution Assessment

No, the closed direct PR shape of always appending raw reasons is not the best fix. The safer path is a redacted, channel-aware policy plus narrow same-action recovery confirmation.
