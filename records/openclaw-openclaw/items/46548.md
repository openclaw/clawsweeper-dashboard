## Summary

Keep open. Current main still omits ordinary non-verbose tool failure reasons from chat-visible warnings, and a matching retry success clears internal state without a user-visible confirmation; a safe implementation needs maintainer policy for redacted, channel-safe diagnostics.

## What This Changes

The issue requests chat-visible tool failure messages to show a safe failure reason and confirm when a matching retry succeeds.

## Best Possible Solution

Define redacted or allowlisted tool-failure summaries by tool and channel, then emit retry-success confirmation only when a previously surfaced mutating failure is resolved by a matching successful action.

## Reproduction Assessment

Yes, by source inspection. Current main gates ordinary tool-error details behind full verbose mode or narrow cron-timeout handling, and retry success only clears `lastToolError` internally.

## Solution Assessment

No, not as a raw-error passthrough. The maintainable path is sanitized failure summaries plus explicit retry-confirmation rules for matching surfaced mutating failures.
