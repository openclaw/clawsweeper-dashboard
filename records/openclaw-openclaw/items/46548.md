## Summary

Keep open. Current main still emits ordinary non-verbose chat warnings without the failure reason, and matching retry success clears runner state without a chat-visible confirmation; the remaining fix needs maintainer policy for redacted, channel-safe diagnostics.

## What This Changes

The issue requests chat-visible tool failure reasons plus a visible confirmation when a later retry succeeds.

## Best Possible Solution

Define a redacted or allowlisted tool-failure summary contract by tool and channel, then emit retry-success confirmation only when a previously surfaced mutating failure is resolved by a matching success.

## Reproduction Assessment

Yes. Source inspection on current main shows non-verbose payloads omit ordinary tool failure reasons, and the retry path only clears `lastToolError` after a matching success.

## Solution Assessment

No current implementation fully solves it. The requested UX is useful, but the maintainable fix needs sanitized failure summaries and explicit retry-confirmation rules before changing every chat surface.
