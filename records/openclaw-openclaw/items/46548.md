## Summary

Keep open. Current main still omits ordinary non-verbose tool failure reasons from chat-visible warnings and only clears internal retry state on a matching successful mutating retry; a safe fix needs maintainer product/security policy for redacted channel-safe diagnostics.

## What This Changes

Request chat-visible tool failure warnings include the failure reason and show confirmation when a retry resolves a prior failure.

## Best Possible Solution

Define redacted or allowlisted tool-failure summaries by tool and channel, then emit retry-success confirmation only when a previously surfaced mutating failure is resolved by a matching successful action.

## Reproduction Assessment

Yes, by source inspection. Current main gates ordinary tool-error details behind verbose/full or narrow cron timeout cases, and retry success only clears `lastToolError` internally.

## Solution Assessment

No. The UX goal is valid, but surfacing raw `lastToolError.error` broadly is not the safest solution; the maintainable path needs sanitized summaries and explicit retry-confirmation rules.
