## Summary

Keep open: current main and v2026.6.9 still intentionally emit compact tool-error warnings without ordinary failure reasons, and retry/recovery matching can clear internal error state without a user-visible recovery confirmation. The fresh Telegram/Dialpad data point makes the remaining policy security-sensitive rather than safe for automatic cleanup.

## What This Changes

The issue requests clearer chat-visible tool failure warnings that include a safe failure reason and confirm when a retry or recovery later succeeds.

## Best Possible Solution

Define a central recovery-aware warning policy for embedded-run payloads: safe redacted reason categories, same-fingerprint recovery status, preserved side-effect receipts, and tests for compact, verbose, recovered, and mutating external-action cases.

## Reproduction Assessment

Yes, source-reproducible: current payload code and tests show ordinary warnings omit the failure reason, and tool success can clear `lastToolError` internally without a recovery confirmation. I did not run a live Telegram/Dialpad reproduction in this read-only review.

## Solution Assessment

No, a blanket “always show the raw error” change is not the best fix because it would create a chat security leak. The maintainable solution is a product/security-approved policy for safe reason text and recovery-aware status transitions.
