## Summary

Keep open. Current main still omits ordinary non-verbose tool failure reasons from chat-visible warnings and clears matching retry failures only in runner state, while the remaining change needs a safe policy for what diagnostic text can be exposed to chat surfaces.

## What This Changes

Request tool failure chat messages include failure reasons and a visible confirmation when a matching retry later succeeds.

## Best Possible Solution

Define a channel-aware, redacted or allowlisted failure-summary contract, then emit retry-success confirmation only when a matching mutating retry resolves a previously surfaced failure.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: build a non-verbose payload with lastToolError and the warning omits the reason, then follow a failed edit through a matching successful edit where state clears without a reply payload.

## Solution Assessment

No current implementation fully solves it. The safest path is sanitized failure summaries plus explicit retry-success rules, not raw error passthrough to every chat surface.
