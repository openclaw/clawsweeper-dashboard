## Summary

Keep open. Current main still only partially addresses the request: normal chat-visible tool warnings omit the underlying reason unless the detail gate allows it, and a successful matching retry clears internal error state without emitting a retry-success confirmation. The remaining work is valid but needs maintainer policy because expanding chat-visible diagnostics can expose sensitive path, command, provider, or external-channel context.

## What This Changes

Request that chat-visible tool failure warnings include a useful failure reason and that successful retries provide a visible confirmation when they resolve a previously surfaced failure.

## Best Possible Solution

Define sanitized, allowlisted failure summaries by tool and channel, then add retry-success confirmation only when a previously surfaced failure is resolved by a matching successful mutating action. Keep the policy aligned with the broader rich-error work in #60946, the privacy constraints surfaced by #54757, and the narrower edit-mismatch direction in #67276.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists by building a verbose-off payload with a mutating `lastToolError` and observing that the warning omits the detail, then simulating a failed edit followed by a matching successful edit and observing that `lastToolError` is cleared without a user-visible retry-success payload.

## Solution Assessment

No current implementation fully solves the issue. The narrow maintainable direction is sanitized reason summaries plus explicit retry-success semantics, not raw error surfacing in every chat context.
