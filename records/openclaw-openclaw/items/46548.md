## Summary

Keep open. Current main still omits ordinary non-verbose tool failure reasons from chat-visible warnings, and successful same-action retries only clear internal error state without a chat-visible retry confirmation; the remaining fix needs a product and security decision for sanitized channel-safe diagnostics.

## What This Changes

The issue asks for chat-visible tool failure messages to include a failure reason and for successful retries to confirm that the failed operation was resolved.

## Best Possible Solution

Define a redacted or allowlisted tool-failure summary contract by tool and channel, then emit retry-success confirmation only when a previously surfaced mutating failure is resolved by a matching successful action.

## Reproduction Assessment

Yes. Source inspection on current main shows non-verbose payloads omit ordinary tool failure reasons, and same-action retry success only clears `lastToolError` internally.

## Solution Assessment

No. The UX goal is valid, but the safe implementation should not expose raw tool errors broadly; it needs sanitized summaries and explicit retry-confirmation rules before changing chat surfaces.
