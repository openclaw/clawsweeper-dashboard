## Summary

Keep open. Current main still omits ordinary non-verbose tool failure reasons from chat-visible warnings and only clears retry state internally after a matching successful mutating retry; a safe fix needs product and security policy for redacted channel-safe diagnostics.

## What This Changes

The issue asks for chat-visible tool failure messages to include a failure reason and to confirm when a retry later succeeds.

## Best Possible Solution

Define redacted or allowlisted tool-failure summaries by tool and channel, then emit retry-success confirmation only when a previously surfaced mutating failure is resolved by a matching successful action.

## Reproduction Assessment

Yes, by source inspection. Current main gates ordinary tool-error details behind full verbose mode or narrow cron timeout cases, and retry success only clears `lastToolError` internally.

## Solution Assessment

No, not as a raw-error passthrough. The maintainable path is sanitized failure summaries plus explicit retry-confirmation rules for matching surfaced mutating failures.
