## Summary

Keep open. Current main still omits ordinary non-verbose tool failure reasons from chat-visible warnings, and same-action retry success only clears internal state; the remaining fix needs maintainer-owned redaction and channel policy before broadening diagnostic text.

## What This Changes

The issue requests chat-visible tool failure messages that include a failure reason and a confirmation when a retry succeeds.

## Best Possible Solution

Define a redacted or allowlisted tool-failure summary contract by tool and channel, then emit retry-success confirmation only when a previously surfaced mutating failure is resolved by a matching successful action.

## Reproduction Assessment

Yes. Source inspection on current main shows non-verbose payloads omit ordinary tool failure reasons, and same-action retry success only clears `lastToolError` internally.

## Solution Assessment

No. The UX goal is valid, but exposing raw `lastToolError.error` broadly is not the safest solution; the maintainable path needs sanitized summaries and explicit retry-confirmation rules.
