## Summary

Keep open. Current main still only partially addresses the request: ordinary non-verbose chat-visible tool warnings can omit the underlying failure reason, and a matching retry success clears internal error state without emitting a user-visible retry-success confirmation.

## What This Changes

The issue requests chat-visible tool failure messages that include a safe failure reason and a later confirmation when a retry resolves the surfaced failure.

## Best Possible Solution

Define sanitized, allowlisted failure summaries by tool and channel, then add retry-success confirmation only when a previously surfaced failure is resolved by a matching successful mutating action.

## Reproduction Assessment

Yes. A high-confidence unit-level path exists by building a verbose-off payload with a mutating `lastToolError` and observing the omitted reason, then simulating an edit failure followed by a matching successful edit and observing state clearing without a chat payload.

## Solution Assessment

No current implementation fully solves the issue. The maintainable direction is sanitized reason summaries plus explicit retry-success semantics, coordinated with #60946, #67276, and the privacy constraints from #54757.
