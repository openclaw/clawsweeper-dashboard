## Summary

Keep open. Current main still omits ordinary non-verbose failure reasons from chat-visible tool warnings and clears retry failures only in state; exposing more diagnostics to chat needs maintainer policy around redaction and channel scope.

## What This Changes

The issue asks for chat-visible tool failure messages to include a failure reason and for successful retries to be confirmed.

## Best Possible Solution

Define sanitized, allowlisted failure summaries by tool and channel, then emit retry-success confirmation only when a previously surfaced mutating failure is resolved by the matching successful action.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: build a normal non-verbose payload with a mutating `lastToolError` and the reason is omitted, then follow a failed edit through a matching successful edit where state clears without a chat payload.

## Solution Assessment

No current implementation fully solves it. The closed raw-reason PR did not settle redaction/channel policy or retry confirmation, so the safer direction is sanitized summaries plus explicit retry-success rules.
