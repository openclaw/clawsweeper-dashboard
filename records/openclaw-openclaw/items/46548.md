## Summary

Keep open. Current main still emits ordinary non-verbose tool warnings without the underlying reason and clears same-action retry failures as internal state only; the remaining fix needs a privacy-safe channel policy before exposing more diagnostics.

## What This Changes

The issue asks for chat-visible tool failure messages to include a useful failure reason and for retry success to be confirmed after a previously surfaced failure is resolved.

## Best Possible Solution

Define sanitized, allowlisted failure summaries by tool and channel, then emit retry-success confirmation only when a previously surfaced mutating failure is resolved by a matching successful action.

## Reproduction Assessment

Yes. A high-confidence source-level path exists by building a verbose-off payload with a mutating `lastToolError` and observing the omitted reason, then following a failed edit through a matching successful edit where state clears without a chat payload.

## Solution Assessment

No current implementation fully solves this. The narrow maintainable direction is sanitized reason summaries plus explicit retry-success semantics, coordinated with #60946, #67276, and the privacy constraints highlighted by #54757.
