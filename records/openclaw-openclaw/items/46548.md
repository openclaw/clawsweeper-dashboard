## Summary

Keep open. Current main still omits ordinary non-verbose failure reasons from chat-visible tool warnings and only clears retry failures in state; the requested UX is valid, but exposing diagnostics to chat needs maintainer policy around redaction and channel scope.

## What This Changes

Request chat-visible tool failure warnings to include a safe failure reason and confirm when a retry succeeds.

## Best Possible Solution

Define sanitized, allowlisted failure summaries by tool and channel, then emit retry-success confirmation only when a previously surfaced mutating failure is resolved by the matching successful action.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: build a non-verbose payload with a mutating `lastToolError` and the reason is omitted, then follow a failed edit/write through a matching successful action where state clears without any chat payload.

## Solution Assessment

No current implementation fully solves it. The closed raw-reason PR did not settle redaction/channel policy or retry confirmation, so the safer endpoint is sanitized summaries plus explicit retry-success rules.
