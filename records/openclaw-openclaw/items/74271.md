## Summary

Keep open. Current main still has the reported local gateway credential precedence split and state-dir dotenv loading, but I did not find the requested redacted warning for a stale `OPENCLAW_GATEWAY_TOKEN` in `~/.openclaw/.env`. The provided discussion also shows open PR #74433 uses closing syntax for this issue, so this issue should remain open until that PR lands or is replaced.

## What This Changes

Warn users in doctor/status/gateway diagnostics when a non-empty state-dir `OPENCLAW_GATEWAY_TOKEN` shadows or differs from configured local `gateway.auth.token`.

## Best Possible Solution

Keep this as the canonical report and resolve it through PR #74433 or a replacement that adds one shared, redacted diagnostic for local mode: parse the state-dir `.env`, compare a non-empty `OPENCLAW_GATEWAY_TOKEN` against configured `gateway.auth.token`, explain env/config precedence, and suggest removing or syncing the env key plus restarting the gateway without printing secrets.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is local gateway mode with `gateway.auth.token` set in config and a different non-empty `OPENCLAW_GATEWAY_TOKEN` loaded from state-dir `.env`; current parity tests document the resulting env/config credential disagreement.

## Solution Assessment

Yes. A redacted warning is the narrowest maintainable fix because automatically deleting or rewriting `OPENCLAW_GATEWAY_TOKEN` could break intentional env-based auth; the open PR or replacement should focus on detection, wording, and surface coverage.
