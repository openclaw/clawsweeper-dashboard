## Summary

Keep open. Current main still has the env/config gateway-token precedence split and still lacks a redacted state-dir `.env` mismatch diagnostic; open PR #74433 is the unmerged closing implementation candidate, so this issue should remain open until that PR lands or is replaced.

## What This Changes

Warn users when a non-empty state-dir `OPENCLAW_GATEWAY_TOKEN` can shadow a configured local `gateway.auth.token`, without printing either secret.

## Best Possible Solution

Keep this as the canonical report and resolve it through #74433 or a replacement that adds one shared redacted mismatch diagnostic for differing non-empty state-dir `OPENCLAW_GATEWAY_TOKEN` and configured `gateway.auth.token`.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is local gateway mode with `gateway.auth.token` in config and a different non-empty `OPENCLAW_GATEWAY_TOKEN` loaded from state-dir `.env`, which exercises the documented env-first client/tool path against config-first gateway auth.

## Solution Assessment

Yes. A warning is the narrowest maintainable solution because it preserves intentional env-based auth, avoids deleting credentials, and can be shared across diagnostic surfaces without printing secrets.
