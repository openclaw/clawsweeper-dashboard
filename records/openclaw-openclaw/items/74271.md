## What This Changes

Add a redacted diagnostic warning when local gateway mode has `gateway.auth.token` configured while state-dir `.env` contains a non-empty `OPENCLAW_GATEWAY_TOKEN` that can shadow or disagree with the config token.

## Best Possible Solution

Keep this as the canonical report and resolve it through PR #74433 or a replacement that adds one shared, redacted diagnostic: read the state-dir `.env`, detect a non-empty `OPENCLAW_GATEWAY_TOKEN` that differs from configured local `gateway.auth.token`, explain env/config precedence, and suggest removing or syncing the env key plus restarting the gateway without printing secrets.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is local gateway mode with `gateway.auth.token` set in config and a different non-empty `OPENCLAW_GATEWAY_TOKEN` loaded from state-dir `.env`; current parity tests document the resulting env/config credential disagreement.

## Solution Assessment

Yes for the requested direction: a warning is safer than automatically deleting or rewriting credentials that may be intentional. Current main does not implement it yet, and the open PR should be reviewed for redaction and requested surface coverage.
