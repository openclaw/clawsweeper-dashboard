## Summary

Keep open. Current main still has the gateway-token env/config precedence split and does not contain the requested redacted state-dir `.env` mismatch diagnostic; open PR #74433 is the linked implementation candidate, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Request a redacted doctor/status warning when local gateway mode has `gateway.auth.token` configured and state-dir `.env` contains a different non-empty `OPENCLAW_GATEWAY_TOKEN`.

## Best Possible Solution

Keep this issue as the canonical report and resolve it by landing #74433 or a replacement that reads state-dir `.env`, compares a differing non-empty token against `gateway.auth.token` in local mode, and emits one shared secret-safe diagnostic on the agreed surfaces.

## Reproduction Assessment

Yes, source-backed. Local gateway mode with `gateway.auth.token` in config and a different `OPENCLAW_GATEWAY_TOKEN` in state-dir `.env` exercises env-first command/probe credential selection against config-first gateway auth; I did not run a live gateway in this read-only review.

## Solution Assessment

Yes. A warning is the narrowest maintainable direction because it preserves intentional env-based auth, avoids deleting secrets, and can be shared across diagnostics without printing token values.
