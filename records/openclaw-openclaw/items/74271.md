## What This Changes

The issue requests a redacted doctor/status diagnostic when a state-dir OPENCLAW_GATEWAY_TOKEN shadows gateway.auth.token in local gateway mode.

## Best Possible Solution

Keep this as the canonical report and resolve it through PR #74433 or a replacement that adds one shared redacted diagnostic for local mode: read the state-dir .env, detect a differing non-empty OPENCLAW_GATEWAY_TOKEN plus configured gateway.auth.token, explain env/config precedence, suggest removing or syncing the env key and restarting the gateway, and reuse that helper across the agreed doctor/status surfaces.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is local gateway mode with gateway.auth.token set in config and a different non-empty OPENCLAW_GATEWAY_TOKEN loaded from the state-dir .env; current credential-precedence tests document the resulting env-first command credential path versus config-first gateway auth path.

## Solution Assessment

Yes for the requested direction. A redacted warning is safer than automatically deleting or rewriting credentials that may be intentional; the implementation should compare values from the state-dir file, not only ambient process env, and avoid printing either secret.
