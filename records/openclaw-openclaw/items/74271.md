## Summary

Keep open. Current main still has the local gateway env/config credential precedence split and no redacted diagnostic for a non-empty state-dir `OPENCLAW_GATEWAY_TOKEN` shadowing configured `gateway.auth.token`; open PR #74433 is the unmerged implementation candidate, so this issue should remain open until that PR lands or is replaced.

## What This Changes

Request a doctor/status warning when local gateway mode has configured `gateway.auth.token` but `~/.openclaw/.env` also contains a non-empty `OPENCLAW_GATEWAY_TOKEN` that may override client/probe credentials.

## Best Possible Solution

Keep this issue as the canonical report and resolve it by landing #74433 or a replacement that adds one shared redacted mismatch diagnostic for differing non-empty state-dir `OPENCLAW_GATEWAY_TOKEN` and configured `gateway.auth.token`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: local gateway config with `gateway.auth.token` plus a different non-empty `OPENCLAW_GATEWAY_TOKEN` loaded into command/probe env exercises env-first client credentials against config-first gateway auth.

## Solution Assessment

Yes, with one surface caveat. A redacted warning is the narrow maintainable fix because it preserves intentional env-based auth and avoids deleting secrets, but maintainers still need to settle exactly which status/install surfaces should emit it.
