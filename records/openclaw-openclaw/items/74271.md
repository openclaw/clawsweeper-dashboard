## Summary

Keep open. Current main still has the local gateway env/config credential split and the state-dir dotenv reader, but it does not yet emit the requested redacted warning in doctor/status/gateway-status. PR #74433 is an open, unmerged implementation candidate that explicitly targets this issue, so the issue should remain open until that PR lands or is replaced.

## What This Changes

Warn users when a non-empty `OPENCLAW_GATEWAY_TOKEN` in the state-dir `.env` shadows a configured local `gateway.auth.token`, without printing secrets.

## Best Possible Solution

Keep this as the canonical report and resolve it through PR #74433 or a replacement that adds one shared, redacted diagnostic for local mode: read the state-dir `.env`, compare a non-empty `OPENCLAW_GATEWAY_TOKEN` with configured `gateway.auth.token`, explain env precedence, and suggest removing or syncing the env key plus restarting the gateway.

## Reproduction Assessment

Yes. A high-confidence reproduction path is local gateway mode with `gateway.auth.token` set in `openclaw.json` and a different non-empty `OPENCLAW_GATEWAY_TOKEN` in the state-dir `.env`; current precedence tests and CLI dotenv loading show why command paths can disagree with gateway auth.

## Solution Assessment

Yes for the requested direction: a warning is safer than deleting or rewriting a credential that may be intentional. Current main does not yet implement it, and the maintainable fix should avoid printing token values while sharing the diagnostic across the relevant command surfaces.
