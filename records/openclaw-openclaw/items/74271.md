## Summary

Keep open. Current main still has the reported local gateway credential-precedence split and state-dir dotenv loading, but it does not yet emit the requested redacted warning for a mismatched state-dir OPENCLAW_GATEWAY_TOKEN versus configured gateway.auth.token. The issue is already paired with open PR #74433, so it should remain open until that implementation or a replacement lands.

## What This Changes

Request a doctor/status diagnostic that warns when local gateway mode has gateway.auth.token configured while ~/.openclaw/.env contains a different non-empty OPENCLAW_GATEWAY_TOKEN that can override client/probe credentials.

## Best Possible Solution

Resolve this through PR #74433 or a replacement that adds one shared, redacted local-mode diagnostic: compare the parsed state-dir .env OPENCLAW_GATEWAY_TOKEN with configured gateway.auth.token, explain env/config precedence and the source file, suggest removing or syncing the env key, and remind the user to restart the gateway without printing either token.

## Reproduction Assessment

Yes. A high-confidence reproduction path is local gateway mode with gateway.auth.token set in config and a different non-empty OPENCLAW_GATEWAY_TOKEN loaded from the state-dir .env; current source and parity tests establish the env-first client/probe versus config-first gateway auth split.

## Solution Assessment

Yes for the requested direction. A warning is the narrow maintainable fix because automatically deleting or rewriting env credentials could break intentional env-based auth; current main has the underlying behavior but not the diagnostic yet.
