## Summary

Keep open. Current main still has the reported local gateway credential precedence split and state-dir dotenv loading, but it does not yet emit the requested redacted warning in doctor/status/gateway status. PR #74433 is an open closing implementation candidate, so this issue should remain open until that PR lands or is replaced.

## What This Changes

Add redacted diagnostics when local gateway mode has `gateway.auth.token` configured but state-dir `.env` contains a non-empty `OPENCLAW_GATEWAY_TOKEN` that can shadow the config token.

## Best Possible Solution

Keep this as the canonical report and resolve it through PR #74433 or a replacement that adds one shared redacted diagnostic: read the state-dir `.env`, compare a non-empty `OPENCLAW_GATEWAY_TOKEN` with configured local `gateway.auth.token`, explain env precedence, and suggest removing or syncing the env key plus restarting the gateway without printing secrets.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is local gateway mode with `gateway.auth.token` set in `openclaw.json` and a different non-empty `OPENCLAW_GATEWAY_TOKEN` in state-dir `.env`; current parity tests document the resulting env/config credential disagreement.

## Solution Assessment

Yes for the requested direction: a warning is safer than deleting or rewriting credentials that may be intentional. Current main does not implement it yet, and the implementation should be reviewed for shared redaction and requested surface coverage rather than separate ad hoc checks.
