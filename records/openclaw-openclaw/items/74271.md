## Summary

Keep open. Current main still has the reported local gateway credential split and state-dir dotenv loading, but it does not yet expose the requested redacted warning in doctor/status/gateway status; the linked PR #74433 is an open, unmerged closing implementation candidate, so the issue should stay open until that PR lands or is replaced.

## What This Changes

Add a redacted diagnostic warning when a non-empty `OPENCLAW_GATEWAY_TOKEN` in the state-dir `.env` can disagree with or shadow configured local `gateway.auth.token`.

## Best Possible Solution

Keep this as the canonical report and resolve it through PR #74433 or a replacement that adds one shared redacted diagnostic: inspect the state-dir `.env`, compare a non-empty `OPENCLAW_GATEWAY_TOKEN` with configured local `gateway.auth.token`, explain env/config precedence, and suggest removing or syncing the env key plus restarting the gateway without printing secrets.

## Reproduction Assessment

Yes. A high-confidence reproduction is local gateway mode with `gateway.auth.token` configured and a different non-empty `OPENCLAW_GATEWAY_TOKEN` loaded from state-dir `.env`; current credential precedence tests and source show the resulting env/config disagreement across command/probe and gateway-auth paths.

## Solution Assessment

Yes. A warning is the narrowest maintainable fix because deleting or rewriting env credentials could break intentional env-based setups; the implementation should be shared enough to keep doctor, status, and gateway-status copy consistent and secret-redacted.
