## Summary

Keep open. Current `main` still has the reported local gateway credential precedence split and state-dir dotenv loading, but it does not yet emit the requested redacted warning in doctor/status/gateway status. The linked PR #74433 is an open closing implementation candidate, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests redacted doctor/status diagnostics when a non-empty state-dir `.env` `OPENCLAW_GATEWAY_TOKEN` shadows a configured local `gateway.auth.token`.

## Best Possible Solution

Keep this as the canonical report and resolve it through PR #74433 or a replacement that adds one shared redacted diagnostic for local gateway mode, configured `gateway.auth.token`, and a differing non-empty state-dir `.env` `OPENCLAW_GATEWAY_TOKEN`, with tests for doctor and any status surfaces maintainers choose to support.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is local gateway mode with `gateway.auth.token` configured and a different non-empty `OPENCLAW_GATEWAY_TOKEN` loaded from the state-dir `.env`; current parity coverage documents the resulting credential disagreement across local paths.

## Solution Assessment

Yes, the requested warning is the best direction. It preserves supported env-based auth instead of deleting credentials, but current main still needs the warning implemented and redacted consistently.
