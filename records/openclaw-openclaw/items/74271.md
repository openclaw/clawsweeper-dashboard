## Summary

Keep open. Current main still has the env/config gateway-token precedence split and lacks a redacted state-dir `.env` mismatch diagnostic; open PR #74433 is the unmerged implementation candidate, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Add a redacted doctor/status warning when local gateway config token is shadowed by a non-empty state-dir `OPENCLAW_GATEWAY_TOKEN`.

## Best Possible Solution

Resolve through #74433 or a replacement that reads state-dir `.env`, compares differing non-empty `OPENCLAW_GATEWAY_TOKEN` and configured `gateway.auth.token` in local mode, and emits redacted guidance on the chosen diagnostic surfaces.

## Reproduction Assessment

Yes. Local gateway mode with `gateway.auth.token` in config and a different non-empty `OPENCLAW_GATEWAY_TOKEN` loaded from state-dir `.env` reproduces the source-backed env-first client/probe versus config-first gateway auth split.

## Solution Assessment

Yes. A redacted warning is the narrowest maintainable fix because env-based auth can be intentional; the implementation should inspect state-dir `.env` rather than only `process.env`.
