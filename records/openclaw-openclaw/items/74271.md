## Summary

Keep open. Current `main` still has the reported local gateway credential precedence split and state-dir dotenv loading, but it does not yet emit the requested redacted doctor/status warning for a mismatched `OPENCLAW_GATEWAY_TOKEN` in the state-dir `.env`. The reporter’s follow-up and martingarramon’s comments narrow this to likely setup-flow drift, and open PR #74433 is an unmerged closing implementation candidate, so this issue should remain open until that PR lands or is replaced.

## What This Changes

Warn users when local `gateway.auth.token` is configured but a non-empty state-dir `OPENCLAW_GATEWAY_TOKEN` overrides or conflicts with it, without printing secret values.

## Best Possible Solution

Keep this as the canonical report and resolve it through PR #74433 or a replacement that adds one shared, redacted diagnostic for local mode: detect differing non-empty state-dir `OPENCLAW_GATEWAY_TOKEN` and configured `gateway.auth.token`, explain env/config precedence, and suggest removing or syncing the env key plus restarting the gateway without printing secrets.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is local gateway mode with `gateway.auth.token` set in config and a different non-empty `OPENCLAW_GATEWAY_TOKEN` loaded from the state-dir `.env`; current parity tests and docs show the env/config credential disagreement behind the report.

## Solution Assessment

Yes for the requested direction: a warning is safer than automatically deleting or rewriting credentials that may be intentional. Current `main` does not implement it yet, and the open PR should be reviewed for redaction and requested surface coverage.
