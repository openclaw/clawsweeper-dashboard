## Summary

Keep open. Current main and v2026.6.8 still report unresolved isolated cron shell-prompt warnings with a blanket `doctor --fix` repair hint even when that issue has no automatic mutation path, and the displayed store path is still the logical legacy `jobs.json` key rather than a clear SQLite-backed location.

## What This Changes

The issue reports that `openclaw doctor` keeps warning about isolated cron shell/process jobs after `doctor --fix` and points users at a legacy `jobs.json` path even though cron state is SQLite-backed.

## Best Possible Solution

Make the cron doctor output distinguish automatically repairable cron store issues from manual-only command-conversion warnings, and describe `cron.store` as a logical SQLite store key or CLI-managed store instead of implying a live JSON file must exist.

## Reproduction Assessment

Yes, at source level. An SQLite-backed cron job whose agent-turn prompt matches the shell/process heuristic will be reported as needing manual conversion, but the repair path has no automatic mutation for that issue alone and the note still tells the user to run `doctor --fix`.

## Solution Assessment

Yes in direction. The narrow maintainable fix is to split manual-only warnings from repairable migrations and update the path wording; changing cron runtime execution or adding a config option would be the wrong layer.
