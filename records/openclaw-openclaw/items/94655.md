## Summary

This issue should stay open: current main and v2026.6.8 still route an unresolved manual cron command-conversion warning through the generic repairable cron-store note, so a pure manual-conversion case can keep promising `openclaw doctor --fix` even when the repair path has no mutation to apply.

## What This Changes

The issue asks for `openclaw doctor` to stop giving a persistent, stale `--fix` and `jobs.json` cron-store diagnostic when SQLite-backed isolated cron jobs only need manual command conversion.

## Best Possible Solution

Separate manual command-conversion warnings from repairable cron-store normalizations, only promise `doctor --fix` when a fixable mutation exists, and make the store wording explain the SQLite-backed cron store/key without sending users to a non-live JSON file.

## Reproduction Assessment

Yes at source level: a SQLite-backed cron job with an unresolved shell-style `agentTurn` prompt reaches the warning branch, but the repair path has no mutation to apply. I did not execute the CLI because this read-only review cannot run state-writing doctor/test flows.

## Solution Assessment

Yes, the best fix is in the doctor cron diagnostic/repair-plan layer, not in cron runtime execution. Keep command jobs and manual-conversion safety intact while making the doctor note truthful and test-covered.
