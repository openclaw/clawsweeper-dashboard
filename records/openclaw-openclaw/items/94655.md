## Summary

Keep open. Current `main` and latest release still have the reported doctor cron UX bug: manual-only command-conversion warnings are shown with an automatic `doctor --fix` repair hint, while the displayed store path is still the logical legacy `cron/jobs.json` key even though live cron rows are SQLite-backed.

## What This Changes

The issue requests clearer cron doctor output when SQLite-backed isolated cron jobs trigger manual command-conversion warnings that `doctor --fix` cannot clear.

## Best Possible Solution

Make cron doctor output separate automatically repairable store migrations from manual-only command-conversion warnings, and describe `cron.store` as a logical SQLite-backed store key/import path rather than a live JSON file.

## Reproduction Assessment

Yes, at source level. An SQLite-backed cron job whose isolated agent-turn prompt matches the shell/process heuristic reaches the manual-conversion warning while the same doctor note still promises `openclaw doctor --fix`; I did not run the mutating command in this read-only review.

## Solution Assessment

Yes in direction. The narrow maintainable fix is to split manual-only warnings from repairable migrations and update the store wording; changing cron runtime execution or adding a config option would be the wrong layer.
