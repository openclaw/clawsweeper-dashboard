## Summary

Keep open. Current main still uses default `logs.tail` redaction patterns that cover Bearer-style credentials but do not match the reported Basic auth, generic JWT, `x-pomerium-jwt-assertion`, or hyphenated `X-OpenClaw-Token` header examples; an open contributor PR already targets the issue and should be reviewed before closing.

## What This Changes

The issue requests stronger `logs.tail` credential redaction for Basic auth, JWTs, Pomerium assertions, and OpenClaw token-style headers before log lines are returned to `operator.read` clients.

## Best Possible Solution

Keep this issue open while maintainers review and either land PR #67041 or apply an equivalent bounded patch on main that extends the default logging redaction patterns and adds focused regression coverage for the reported formats.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path: place any reported sample in the configured gateway log file and call `logs.tail`; the current default regex set does not match those samples, while the handler returns only the `redactSensitiveLines` output.

## Solution Assessment

Yes, the best direction is a narrow change in `src/logging/redact.ts` with tests in `src/logging/redact.test.ts`, plus optional `logs.tail` handler coverage. Reusing or aligning with the existing support-export Basic/JWT redaction helpers would reduce drift, but the fix belongs in the default log-tail redactor.
