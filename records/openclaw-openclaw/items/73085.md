## Summary

Keep this PR open: it targets a real Slack dmScope routing bug on current main, but the proposed dispatch change still writes threaded non-main DM route metadata to the base peer session and the user-visible fix is missing a changelog entry.

## What This Changes

The PR changes Slack prepare/dispatch last-route persistence so non-main DM scopes avoid the main-session route and adds focused dmScope regression tests.

## Best Possible Solution

Land a corrected Slack plugin patch that keeps main-DM owner pinning only for main mode, writes non-main DM route metadata to the prepared turn session, and includes a threaded regression plus changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence seam reproduction: a non-main Slack DM thread where route.sessionKey is the base peer session and prepared.ctxPayload.SessionKey is thread-suffixed will make the PR dispatch path update the wrong session key.

## Solution Assessment

No. The prepare-side direction is maintainable, but dispatch should use prepared.ctxPayload.SessionKey ?? route.sessionKey for non-main DMs and pass ctx only when updating that prepared turn session.
