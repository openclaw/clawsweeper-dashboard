## Summary

Keep open: this PR addresses a real current-main Slack DM route contamination bug, but the current PR head still has a dispatch-side threaded DM regression. The earlier Greptile per-account dispatch test gap appears addressed in the latest diff; the remaining blocker is that non-main dispatch updates use the base route session instead of the prepared turn session, plus the required user-facing changelog entry is missing.

## What This Changes

The PR changes Slack DM last-route persistence in prepare and dispatch so non-main `session.dmScope` values avoid writing the global main-session route, and adds focused dmScope regression tests.

## Best Possible Solution

Merge this after dispatch targets the prepared turn session for non-main DM route updates, main-DM owner pinning remains limited to true main-session routes, and the Slack routing fix is captured in release notes.

## Reproduction Assessment

Yes. A high-confidence seam reproduction is a prepared Slack DM with `lastRoutePolicy: "session"`, `route.sessionKey` set to the base peer key, and `ctxPayload.SessionKey` set to the same key with a `:thread:<threadTs>` suffix; the PR dispatch code writes `updateLastRoute.sessionKey` to the base key.

## Solution Assessment

No. The prepare-side policy is the right direction, but dispatch should target `prepared.ctxPayload.SessionKey ?? route.sessionKey` for non-main routes and include regression coverage for a threaded non-main DM.
