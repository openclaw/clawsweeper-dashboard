## What This Changes

The PR gates Slack DM last-route updates for non-main dmScope values, adds Slack dmScope tests, and also changes ACP session identity/retry behavior and tests.

## Best Possible Solution

Land a scoped fix that targets Slack non-main DM route metadata to the prepared turn session, preserves main-DM owner-pin behavior, adds threaded regression and changelog coverage, and handles the ACP changes in a separate reviewed path or fully documented scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence seam path: current main handles a Slack DM with non-main dmScope and writes last-route metadata to route.mainSessionKey; the PR-specific regression appears with a threaded prepared DM where ctxPayload.SessionKey differs from route.sessionKey.

## Solution Assessment

No. The prepare-side direction is right, but dispatch should target prepared.ctxPayload.SessionKey ?? route.sessionKey for non-main DMs, add threaded coverage and changelog entries, and split or explicitly justify the unrelated ACP changes.
