## Summary

Keep open: the updated PR preserves package-scoped cache behavior for an already-scoped NODE_COMPILE_CACHE path, but it still does not solve the documented/root cache path that the linked issue and PR body describe.

## What This Changes

The PR changes the packaged launcher compile-cache directory comparison, avoids double-appending the OpenClaw cache scope when NODE_COMPILE_CACHE already points at it, and adds an E2E regression plus a live repro script.

## Best Possible Solution

Decide the cache contract first, then either implement and test the documented root NODE_COMPILE_CACHE path or explicitly narrow the fix/docs to an already-scoped OpenClaw cache directory.

## Reproduction Assessment

Yes. The linked issue includes a current-main packaged-launcher repro, and a focused Node probe confirms that a root NODE_COMPILE_CACHE starts as a versioned sibling of OpenClaw's scoped desired directory.

## Solution Assessment

No. The current patch is maintainable for already-scoped cache paths, but it does not fix the documented root-cache setup without an explicit maintainer decision about package-scoped cache behavior.
