## Summary

Keep open: the PR addresses a real launcher bug, but the shared-root compatibility check appears to bypass OpenClaw's existing package-scoped compile-cache contract.

## What This Changes

The PR broadens the packaged launcher's compile-cache compatibility check, adds a packaged launcher E2E assertion, and adds a live repro script for repeated `NODE_COMPILE_CACHE` launches.

## Best Possible Solution

Revise the launcher fix so avoiding redundant respawn does not bypass the package-version/install-marker cache scope, or explicitly change that contract with matching tests and docs.

## Reproduction Assessment

Yes. The linked issue has a current-main packaged-fixture reproduction, and the source path plus Node probe explain why `NODE_COMPILE_CACHE` produces a versioned active directory that differs from OpenClaw's desired scoped directory.

## Solution Assessment

No. The PR stops the respawn marker by accepting any shared cache base, but that also lets the process continue with Node's generic versioned cache instead of OpenClaw's package-scoped cache.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve the package-scoped cache before skipping respawn:** `openclaw.mjs:88-90`
  - body: When `NODE_COMPILE_CACHE` points at a base directory, `currentDirectory` is Node's `base/v...` directory while `desiredDirectory` is `base/openclaw/<version>/<marker>`. This new shared-base check returns true in that first process, so the launcher skips the respawn and the later `enableCompileCache(desiredDirectory)` call leaves Node on the already-enabled generic cache. That breaks the existing package-scoped cache behavior and the current scoping E2E assertion; keep the shortcut limited to an active cache under the OpenClaw desired root, or intentionally update the scoping contract.
  - confidence: 0.9
