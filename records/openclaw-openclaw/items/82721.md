## Summary

Keep open with a blocking review finding: the branch proves scoped cache-path idempotence, but the linked launcher issue and public operator docs still describe a raw `NODE_COMPILE_CACHE` root. Merging this as-is would leave that documented setup taking the packaged respawn path on each fresh CLI invocation unless maintainers explicitly narrow the contract first.

## What This Changes

The PR changes `openclaw.mjs` to accept active compile-cache directories nested under the desired packaged cache root, avoids double-suffixing already scoped `NODE_COMPILE_CACHE` values, and adds launcher E2E plus live repro coverage.

## Best Possible Solution

Preserve package-version/install-marker cache invalidation while either making documented raw-root `NODE_COMPILE_CACHE` idempotent or explicitly narrowing the PR, linked issue, and docs to scoped cache roots before merge.

## Reproduction Assessment

Yes. The linked issue includes a current-main packaged-fixture reproduction of repeated packaged respawn, and source inspection shows why raw-root `NODE_COMPILE_CACHE` still misses the PR head's nested-under-desired guard.

## Solution Assessment

No as written. The scoped-path comparison is a reasonable partial fix, but the PR still advertises and closes a raw-root launcher bug that the current diff intentionally does not solve.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle raw cache roots before closing the launcher bug:** `openclaw.mjs:232`
  - body: With the documented `NODE_COMPILE_CACHE=/var/tmp/openclaw-compile-cache`, Node's active cache is under the raw root while OpenClaw's desired directory is `raw/openclaw/<version>/<marker>`. Either support that path without losing package-scoped invalidation, or narrow the PR/body/issue relationship before merge.
