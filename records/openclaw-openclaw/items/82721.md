## Summary

Keep open: the latest PR narrows the behavior to an already scoped `NODE_COMPILE_CACHE`, but the linked bug and public docs still cover a raw cache root, where the PR head continues to respawn on every fresh CLI invocation. The remaining blocker is a maintainer contract decision around raw-root zero-respawn versus preserving package-scoped compile-cache invalidation.

## What This Changes

The PR updates the packaged launcher compile-cache guard and resolver, adds a scoped-cache launcher E2E case, and adds a live repro script.

## Best Possible Solution

Decide the raw-root cache contract, then either implement and document that behavior end to end or narrow this PR so it does not close the linked bug.

## Reproduction Assessment

Yes. A temporary packaged fixture against current main reproduces repeated raw-root respawn, and the same fixture against the latest PR head still prints the packaged respawn marker on both raw-root launches.

## Solution Assessment

No, not as written. The code is reasonable for an already scoped `NODE_COMPILE_CACHE`, but it does not solve the documented raw-root path that the PR body and linked issue claim to fix.
