## Summary

Keep this open: current main and shipped v2026.5.18 still delegate Signal typing to signal-cli `sendTyping`, while the upstream 1:1 delivery path still uses `online=true`; the reporter's added live proof supports that changing the upstream flag to `online=false` makes official clients show typing again.

## What This Changes

Investigate or fix Signal native typing indicators that signal-cli accepts successfully but official Signal clients do not display during long OpenClaw turns.

## Best Possible Solution

Patch or upstream the 1:1 typing delivery flag in signal-service-java/signal-cli, then update OpenClaw's Signal runtime or install guidance to a released fixed version, with separate live proof for group typing before changing that path.

## Reproduction Assessment

Yes for a high-confidence source and reporter-live path: current main and v2026.5.18 call signal-cli `sendTyping`, and the inspected upstream 1:1 path still passes `online=true`. I did not independently run an official Signal client in this read-only review.

## Solution Assessment

Yes for the proposed upstream direction, but unclear as an OpenClaw-only fix: the narrow durable repair belongs in signal-cli or Turasa signal-service-java, while carrying a patched runtime in OpenClaw needs maintainer approval.
