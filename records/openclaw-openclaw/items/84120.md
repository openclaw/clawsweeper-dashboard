- `P2`: This is a real Signal channel liveness bug with limited blast radius: replies still deliver, but official clients do not show the expected typing indicator during long turns.

## Summary

Keep open: current main and the latest shipped OpenClaw release still delegate Signal typing to signal-cli, while the latest signal-cli dependency path I inspected still sends 1:1 typing with the reported `online=true` flag. The reporter’s added live validation strengthens the root-cause case, but the durable fix is an upstream signal-cli/Turasa runtime decision rather than a narrow OpenClaw caller patch.

## What This Changes

The issue requests restoring visible Signal typing indicators in official clients when OpenClaw calls signal-cli `sendTyping` during long Signal agent turns.

## Best Possible Solution

Get the 1:1 typing delivery change fixed upstream in Turasa/signal-service-java or signal-cli, then update OpenClaw's Signal runtime guidance or installer once a released runtime is available and verify 1:1 plus group behavior separately.

## Reproduction Assessment

Yes, with high confidence for the source/dependency path: current main calls signal-cli `sendTyping`, latest signal-cli/Turasa source still uses `online=true` for 1:1 typing, and the reporter supplied live validation that an `online=false` patched runtime restores official-client bubbles. I did not independently run the official-client live test in this review.

## Solution Assessment

Yes for keeping this open and pursuing an upstream-first fix; no for an OpenClaw-only JSON-RPC caller patch, because signal-cli does not expose the needed delivery flag. A temporary patched runtime would need explicit maintainer approval and live native/group regression proof.
