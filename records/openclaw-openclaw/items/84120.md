## Summary

This should stay open: current main and the latest release still delegate Signal typing to signal-cli, and signal-cli v0.14.3 still routes 1:1 typing through Turasa signal-service-java with the reported `online=true` delivery flag. The related open PR is about starting group typing earlier, not the upstream 1:1 delivery flag that the reporter validated with a patched runtime.

## What This Changes

The issue asks maintainers to address missing visible Signal typing indicators despite successful `sendTyping` calls, likely by upstreaming or adopting a signal-cli/Turasa runtime fix.

## Best Possible Solution

Patch the upstream signal-service-java or signal-cli 1:1 typing delivery path, then update OpenClaw's Signal runtime or install guidance once a fixed runtime is available, with separate live proof for 1:1 and group typing.

## Reproduction Assessment

Yes for source-level reproduction: current main calls `sendTypingSignal`, and signal-cli v0.14.3 routes 1:1 typing to Turasa `sendTyping(... online=true)`. I did not establish a reviewer-owned live official-client reproduction here, but the reporter supplied patched-runtime live validation.

## Solution Assessment

Unclear as an OpenClaw-only fix: the narrow durable repair appears to belong upstream in signal-cli or signal-service-java. Carrying a patched runtime could work temporarily but needs explicit maintainer approval and live upgrade proof.
