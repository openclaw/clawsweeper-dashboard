## Summary

Keep open: current main and the latest shipped release still delegate Signal typing to signal-cli, while the reported failure is in the upstream 1:1 typing delivery path. The reporter’s added live validation strengthens the root-cause case, but fixing it requires an upstream signal-service-java/signal-cli change or an explicit maintainer decision to carry a patched Signal runtime.

## What This Changes

Report asks for Signal native typing indicators to become visible in official clients when OpenClaw sends `sendTyping` during long agent turns.

## Best Possible Solution

Patch or upstream the signal-service-java 1:1 typing delivery flag, then update OpenClaw’s Signal runtime/install guidance once a vetted signal-cli build is available and separately validate group typing.

## Reproduction Assessment

Yes for source-level and reporter-backed reproduction: current main calls signal-cli `sendTyping`, and the upstream dependency source uses `online=true` for 1:1 typing. I did not run a live official-client repro in this read-only review, so this is `source_reproducible` rather than directly reproduced.

## Solution Assessment

Unclear as an OpenClaw-only fix: the narrow durable fix appears to belong upstream in signal-service-java or signal-cli. Carrying a patched runtime could work but needs maintainer approval because it changes dependency and release ownership.
