## Summary

This should stay open because current main and the latest shipped release still send Signal typing through signal-cli’s supported `sendTyping` path, while the visible-client failure is traced to signal-cli/Turasa’s 1:1 delivery path rather than an OpenClaw caller bug. The reporter’s added live validation is strong useful context, but choosing whether to wait for upstream, carry a patched runtime, or document a temporary limitation needs maintainer/product ownership.

## What This Changes

The issue reports that OpenClaw’s Signal channel successfully calls `sendTyping`, but official Signal clients do not show the native typing indicator unless the upstream 1:1 typing delivery path uses `online=false`.

## Best Possible Solution

Upstream or adopt a narrowly tested signal-cli/Turasa 1:1 typing fix, then update OpenClaw’s Signal runtime guidance and verify official-client 1:1 typing plus unchanged message delivery, with group typing tested separately before any group-path change.

## Reproduction Assessment

Yes for source-level reproduction and reporter-side live proof: current main calls signal-cli `sendTyping`, and the reporter verified official clients show typing after changing only the upstream 1:1 delivery flag. I did not independently run a live official Signal client plus patched signal-cli pair in this review, so this is not marked as directly reproduced here.

## Solution Assessment

Unclear as an OpenClaw-only source fix: the durable narrow fix appears to belong in signal-cli/Turasa, while carrying a patched runtime or documenting a temporary workaround needs maintainer approval. The current OpenClaw caller is already using the supported `sendTyping` contract.
