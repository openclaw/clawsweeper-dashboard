## Summary

Keep open: current main still calls Signal `sendTyping` correctly, but the live-visible failure appears to be in signal-cli's signal-service-java delivery path, where 1:1 typing is sent with `online=true`. The reporter added useful patched-runtime validation, and the related open PR is about earlier typing timing, not this upstream delivery-flag root cause.

## What This Changes

The issue reports that Signal typing requests succeed through signal-cli but official Signal clients do not display the typing indicator during long OpenClaw turns.

## Best Possible Solution

Patch or consume an upstream signal-service-java/signal-cli fix for 1:1 typing delivery, then update OpenClaw's Signal runtime or install guidance once that fix is available while testing group typing separately.

## Reproduction Assessment

Yes for source-level reproduction: current main calls signal-cli `sendTyping`, and the inspected signal-service-java path uses the reported `online=true` delivery flag. Full live reproduction is supported by the reporter's patched-runtime validation, but I did not independently run official Signal clients in this review.

## Solution Assessment

No as a pure OpenClaw caller change: the narrow durable fix appears to belong upstream in signal-cli or signal-service-java, or in an explicitly approved temporary patched runtime. OpenClaw should avoid inventing a local workaround that cannot control the dependency's delivery flag through the existing JSON-RPC contract.
