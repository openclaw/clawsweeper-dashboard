## Summary

Keep this PR open: it targets a real source-reproducible scheduler bug, but the current branch still regresses the newer unresolved-cron-next-run protection and is missing the required user-facing changelog entry.

## What This Changes

The PR applies a 2-second refire floor to successful `every` cron schedules when their computed next run is at or before `endedAt`, and adds regression tests for those interval edge cases.

## Best Possible Solution

Update the branch so `every` jobs clamp only due or past next runs, cron schedules keep current-main unresolved-next-run clearing, and the user-visible fix is recorded in the changelog.

## Reproduction Assessment

Yes. Source inspection shows current main can persist an `every` job's `naturalNext` at `endedAt`, and the PR body adds live Docker reproduction of the same tight-refire symptom.

## Solution Assessment

No, not as currently proposed. The direction is narrow, but the implementation should reuse or preserve the current cron lower-bound resolver so unresolved cron completions remain unscheduled instead of becoming a 2-second retry.
