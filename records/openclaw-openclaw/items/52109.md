## Summary

Keep open: current main still has the reported `every`-schedule refire gap, but the PR branch would regress the current unresolved-cron clearing behavior from #66019 and is missing the required changelog entry.

## What This Changes

The PR changes cron `applyJobResult` so successful `every` schedules can be clamped by `MIN_REFIRE_GAP_MS`, and adds regression coverage for overrun and exact-boundary interval jobs.

## Best Possible Solution

Update the branch so successful `every` jobs apply the 2-second floor only for a defined next run at or before `endedAt`, while preserving current cron unresolved-next-run clearing and adding the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an `every` job whose execution reaches an anchored boundary can compute `naturalNext === endedAt`, and current main writes that due timestamp directly for non-cron completions; the PR body also includes live Docker reproduction of the same symptom.

## Solution Assessment

No, not as currently proposed. The direction is narrow, but the branch should route cron schedules through the current lower-bound resolver so unresolved next-run results remain unscheduled instead of becoming a 2-second retry.
