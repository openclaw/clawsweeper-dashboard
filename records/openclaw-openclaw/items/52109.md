## Summary

## What This Changes

The PR updates cron completion scheduling so `every` jobs get the 2-second refire floor only when their computed next run is at or before the completed run, and adds regression tests for overrun cases.

## Best Possible Solution

Rebase the PR onto current main, keep the narrow `every` completion floor only for `naturalNext <= endedAt`, preserve the existing cron/unresolved-schedule lower-bound helper, and tighten the weak regression test.

## Reproduction Assessment

Yes from source inspection: current main's successful `every` branch stores `naturalNext` directly, and the schedule code can compute a timestamp at the completed run boundary after an interval overrun. I did not execute tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes, with a refresh. The intended fix belongs in `applyJobResult` next to the existing refire guard, but the branch must be rebased onto current main and keep the newer unresolved-schedule helper semantics.
