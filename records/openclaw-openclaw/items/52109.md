## Summary

Keep open: current main still lacks the useful `every` overrun floor, but the PR head also changes unresolved cron completions back to a short refire fallback, conflicting with the current no-refire contract.

## What This Changes

The PR adds focused cron scheduler regression tests and changes `applyJobResult` so overrun `every` jobs receive a `MIN_REFIRE_GAP_MS` floor after successful completion.

## Best Possible Solution

Apply an `every`-only overrun floor on top of current main while leaving cron completions routed through `resolveCronNextRunWithLowerBound` and keeping unresolved-next-run coverage green.

## Reproduction Assessment

Yes, source-level. Current main stores `naturalNext` directly for successful non-cron completions, `every` schedule computation can return the current boundary after an overrun, and the PR body includes live Docker before/after proof of the refire loop and the proposed fix.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right narrow fix, but the PR must preserve current main's unresolved-cron helper instead of synthesizing `endedAt + MIN_REFIRE_GAP_MS` for unresolved cron schedules.
