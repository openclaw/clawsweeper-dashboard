## Summary

Keep this PR open because the central `every`-schedule refire-gap bug is still present on current `main`, but the branch is not mergeable as-is because it regresses current `main`'s unresolved cron next-run contract.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs get a minimum refire gap and adds focused regression tests for that behavior.

## Best Possible Solution

Apply the `every` overrun floor on top of current `main` while preserving unresolved cron completions as unscheduled and keeping the new focused regression coverage.

## Reproduction Assessment

Yes, at source level: current `main` stores unfloored `naturalNext` for successful `every` jobs, and the next-run calculation can return the just-finished timestamp after an overrun. The PR body also contains copied live Docker before/after output for the tight refire loop.

## Solution Assessment

No, not as currently proposed. The `every`-schedule clamp is the right direction, but the branch must compose with current `main`'s unresolved-cron helper instead of scheduling unresolved cron completions two seconds later.
