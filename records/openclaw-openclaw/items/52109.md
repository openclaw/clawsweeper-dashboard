## Summary

Keep open: this is a useful scheduler bug fix, but the current PR head conflicts with newer cron completion behavior on main and would reintroduce unresolved-next-run refires for cron schedules.

## What This Changes

The PR changes cron completion handling so every-schedule jobs whose computed next run is at or before completion time are clamped behind the scheduler refire gap, with focused regression tests.

## Best Possible Solution

Repair or replace the branch so cron schedules keep `resolveCronNextRunWithLowerBound`, while defined every-schedule next-runs at or before `endedAt` are clamped behind `MIN_REFIRE_GAP_MS` with regression coverage.

## Reproduction Assessment

Yes, at source level. Current main can store a due-now every next run after completion, and the PR body includes copied live Docker before/after output for the overrun case.

## Solution Assessment

No, not at the current PR head. The intended every-schedule floor is the right shape, but the patch must preserve current main's unresolved-cron behavior instead of synthesizing a cron retry when next-run resolution returns undefined.
