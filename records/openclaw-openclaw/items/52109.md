## Summary

This PR should stay open: the `every`-schedule refire bug is valid, but the current patch still regresses the existing unresolved-cron safeguard by scheduling unresolved cron jobs for a two-second retry instead of leaving them unscheduled.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs receive a two-second refire floor and adds focused `applyJobResult` regression tests.

## Best Possible Solution

Merge the narrow `every` overrun clamp while keeping cron schedules on `resolveCronNextRunWithLowerBound`, with regression coverage for both the new `every` case and the existing unresolved-cron no-refire contract.

## Reproduction Assessment

Yes. Source inspection on current main shows successful `every` completions still store `naturalNext`, and the `every` next-run computation can return the completion boundary after an overrun; the PR body also includes copied live Docker before/after output.

## Solution Assessment

No, not as currently proposed. The `every` floor is the right narrow fix, but the patch should preserve the current cron unresolved-next-run helper instead of scheduling unresolved cron jobs at `endedAt + MIN_REFIRE_GAP_MS`.

## Review Findings
