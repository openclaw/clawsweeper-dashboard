## Summary

Keep open: the `every` refire-loop bug is still source-reproducible on current main, but this PR head is stale/conflicting and would bypass current main's unresolved-cron protection if rebased naively.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs get a two-second minimum refire gap and adds focused `applyJobResult` regression tests.

## Best Possible Solution

Apply an `every`-only overrun floor on current main while leaving cron completions on `resolveCronNextRunWithLowerBound` and preserving the unresolved-cron regression contract.

## Reproduction Assessment

Yes. Current main still stores `naturalNext` directly for successful `every` completions, and source inspection shows the overrun path can compute a next run at the completion boundary; the PR body also includes copied live Docker before/after output.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right narrow fix, but the branch must preserve current main's unresolved-cron helper instead of synthesizing `endedAt + MIN_REFIRE_GAP_MS` for unresolved cron schedules.
