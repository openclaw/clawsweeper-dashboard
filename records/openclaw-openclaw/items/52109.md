## Summary

Keep this PR open: it targets a real cron scheduler bug, but the current branch still conflicts with newer current-main unresolved-cron protection and should not merge as-is.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs get a minimum refire gap and adds focused regression tests for that path.

## Best Possible Solution

Apply an `every`-only overrun floor on top of current main while keeping cron completions routed through `resolveCronNextRunWithLowerBound` and preserving the unresolved-cron regression contract.

## Reproduction Assessment

Yes. Source inspection on current main shows successful `every` completions still store `naturalNext` directly, and the PR body includes copied live Docker before/after output showing the immediate refire and the fixed `endedAt + 2000` result.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right narrow fix, but the branch must preserve current main's unresolved-cron helper instead of synthesizing `endedAt + MIN_REFIRE_GAP_MS` for unresolved cron schedules.
