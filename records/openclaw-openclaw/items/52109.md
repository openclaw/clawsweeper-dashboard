## Summary

Keep open: the reported scheduler bug is valid and current main still stores successful `every` completions as the computed natural next run, but this PR is not mergeable as-is because it would reintroduce the unresolved-cron refire behavior fixed for https://github.com/openclaw/openclaw/issues/66019.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs get a two-second minimum refire gap and adds focused `applyJobResult` regression coverage.

## Best Possible Solution

Apply a current-main patch that clamps only overrun `every` completions while preserving unresolved cron schedules as unscheduled and keeping focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows successful `every` completions persist `naturalNext` directly, while `computeJobNextRunAtMs` can return the completion boundary after an overrun; the PR body also includes copied live Docker before/after output for an `every 5s` job.

## Solution Assessment

No, not as currently proposed. The `every` overrun clamp is the right narrow fix, but the branch must preserve `resolveCronNextRunWithLowerBound` instead of synthesizing `endedAt + MIN_REFIRE_GAP_MS` for unresolved cron schedules.
