## Summary

Keep open: the PR targets a valid cron `every` refire bug and includes useful live Docker proof, but the current diff still regresses current main's unresolved cron-completion behavior by scheduling a 2-second refire instead of leaving the job unscheduled.

## What This Changes

The branch changes cron completion scheduling to apply a 2-second refire floor to overrun `every` schedules and adds focused `applyJobResult` regression tests.

## Best Possible Solution

Apply an `every`-only overrun floor in `applyJobResult` while keeping cron `naturalNext === undefined` routed through the current unresolved-cron helper, with regression coverage for both behaviors.

## Reproduction Assessment

Yes. Current-main source shows overrun `every` completions can store `naturalNext` directly, and the PR body includes live Docker before/after output showing the immediate refire and the fixed `endedAt + 2000` result.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right narrow fix, but the branch must compose with current main's unresolved-cron helper instead of synthesizing a 2-second cron refire when cron next-run resolution returns `undefined`.
