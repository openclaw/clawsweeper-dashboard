## Summary

Keep open: current main still leaves successful non-cron recurring jobs on `naturalNext`, so the overrun `every` refire bug is still a real source-level issue. The PR is useful, but it is not merge-ready because it restores a cron `naturalNext === undefined` fallback that current main deliberately replaced with unscheduled behavior.

## What This Changes

The PR changes cron result handling so overrun `every` schedules get a 2s refire floor and adds regression tests for that path.

## Best Possible Solution

Layer the overrun `every` floor onto current main while preserving `resolveCronNextRunWithLowerBound` for cron schedules and keeping focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main can compute an overrun `every` schedule at or before `endedAt` and then persist it without a lower bound. The PR body also includes live Docker before/after output for the same tight-refire symptom.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right fix direction, but cron schedules whose next run is unresolved must remain unscheduled instead of falling back to `endedAt + 2s`.
