## Summary

Keep open: the PR addresses a real `every`-schedule refire bug with useful live proof, but the branch is stale and reintroduces the older unresolved-cron fallback that current `main` now prevents.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs receive a minimum 2-second refire gap and adds focused Vitest regression coverage.

## Best Possible Solution

Apply the `every` overrun floor on top of current `main` while preserving `resolveCronNextRunWithLowerBound` for cron schedules, leaving malformed `every` schedules unscheduled, and retaining focused regression coverage.

## Reproduction Assessment

Yes, at source level: current `main` stores unfloored `naturalNext` for successful `every` jobs, and the next-run calculation can return the just-finished timestamp after an overrun. The PR body also includes copied live Docker before/after output for the tight refire loop.

## Solution Assessment

No, not as currently proposed. The `every`-schedule clamp is the right direction, but the branch must compose with current `main`'s unresolved-cron helper instead of scheduling unresolved cron completions two seconds later.
