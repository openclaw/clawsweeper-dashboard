## Summary

Keep open: the reported `every` overrun bug is valid and the PR has useful tests plus live proof, but the current branch is stale against main and would reintroduce the unresolved-cron 2-second refire behavior that main now explicitly prevents.

## What This Changes

The PR changes cron completion scheduling so overrun `every` schedules receive the `MIN_REFIRE_GAP_MS` floor and adds focused regression tests.

## Best Possible Solution

Apply an `every`-only overrun floor on current main while preserving `resolveCronNextRunWithLowerBound` for cron completions and keeping focused regression coverage.

## Reproduction Assessment

Yes. Current main can store an unfloored `every` next run at the just-finished timestamp after an overrun, and the PR body includes copied live Docker before/after output for that behavior.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right direction, but the branch must compose with current main's unresolved-cron helper instead of falling back to `endedAt + MIN_REFIRE_GAP_MS` for unresolved cron schedules.
