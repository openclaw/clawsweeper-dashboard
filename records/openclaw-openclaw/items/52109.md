## Summary

Keep this PR open, but it still needs changes before merge: the reported `every` refire bug is valid and the PR has real runtime proof, but the current branch would reintroduce the unresolved cron next-run fallback fixed on main by scheduling cron jobs at `endedAt + 2000` when next-run resolution returns `undefined`.

## What This Changes

The PR changes `applyJobResult` so due-now `every` schedule completions receive a 2-second refire floor and adds focused regression tests.

## Best Possible Solution

Repair or replace the branch so current-main unresolved cron schedules remain unscheduled while defined `every` next-runs at or before `endedAt` are clamped behind `MIN_REFIRE_GAP_MS`, with regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main can store a due-now `every` next run after completion, and the PR body includes copied live Docker before/after output for the `every 5s` overrun case.

## Solution Assessment

No, not at the current branch head. The narrow `every` floor is the right shape, but it must preserve `resolveCronNextRunWithLowerBound` for unresolved cron schedules instead of falling back to `endedAt + MIN_REFIRE_GAP_MS`.
