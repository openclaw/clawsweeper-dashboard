## Summary

This PR still targets a valid current-main cron scheduler bug, but it should not be closed because the branch has a blocking regression for unresolved cron completions that current main explicitly keeps unscheduled. The useful `every` overrun floor should be reapplied in a way that preserves current main's cron next-run resolution contract.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs get a minimum refire gap and adds focused regression coverage for that path.

## Best Possible Solution

Apply an `every`-only overrun floor on top of current main while continuing to route cron completions through the unresolved-cron helper and keeping both regression-test families green.

## Reproduction Assessment

Yes. Source inspection on current main shows successful `every` completions still store `naturalNext` directly, and the PR body includes copied live Docker before/after output showing the immediate refire and the fixed `endedAt + 2000` result.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right narrow fix, but the branch must compose with current main's unresolved-cron helper instead of synthesizing `endedAt + MIN_REFIRE_GAP_MS` for unresolved cron schedules.
