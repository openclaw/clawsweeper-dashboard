## Summary

Keep open: the underlying `every`-schedule refire bug is valid and the PR has useful tests/proof, but the current branch still conflicts with current `main` by rescheduling unresolved cron completions instead of leaving them unscheduled.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs receive the two-second `MIN_REFIRE_GAP_MS` floor and adds focused Vitest regression coverage.

## Best Possible Solution

Apply an `every`-only overrun floor on top of current `main` while preserving unresolved cron jobs as unscheduled and keeping focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current `main` can store an unfloored `every` next run at the just-finished time after an overrun, and the PR body includes live Docker before/after output for that tight refire loop.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right direction, but the branch must compose with current `main`'s unresolved-cron helper instead of synthesizing a two-second retry for unresolved cron schedules.
