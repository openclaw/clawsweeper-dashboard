## Summary

Keep open. The central `every`-schedule bug fix is still useful and is not on current main, but the PR head is stale and would reintroduce a cron unresolved-next-run fallback that current main intentionally removed.

## What This Changes

The PR changes `applyJobResult` to apply the 2-second refire floor to `every` schedules when their computed next run is at or before completion, and adds focused regression tests.

## Best Possible Solution

Repair or rebase the branch so the `every` refire floor is layered onto current main’s `resolveCronNextRunWithLowerBound` behavior, preserving unresolved cron schedules as unscheduled.

## Reproduction Assessment

Yes. Source inspection shows current main can persist an `every` job's computed next run at the completion boundary, and the PR body includes live Docker before/after output for the tight-refire symptom.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right narrow direction, but the stale branch must preserve current main’s unresolved cron next-run handling instead of scheduling `endedAt + 2000`.
