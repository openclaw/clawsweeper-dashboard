## Summary

Keep this PR open: the every-schedule refire bug is still source-reproducible on current main and the PR has useful tests and live proof, but the branch is stale and would regress current unresolved cron next-run handling if merged as-is.

## What This Changes

The branch applies a 2-second post-run floor to overrun `every` schedules and adds two focused regression tests for the boundary and short-interval cases.

## Best Possible Solution

Repair the branch so the `every` refire floor is layered onto current main’s cron helper contract, leaving unresolved cron schedules unscheduled while keeping the targeted `every` regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main can persist an `every` job's computed next run at the completion boundary, and the PR body includes live Docker before/after output for the tight-refire symptom.

## Solution Assessment

No, not as currently proposed. The narrow `every` clamp is the right direction, but the branch must preserve current main’s unresolved cron next-run behavior instead of falling back to `endedAt + MIN_REFIRE_GAP_MS` for unresolved cron completions.
