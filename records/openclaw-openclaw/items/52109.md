## Summary

Keep open: the every-schedule refire bug is still valid, but the current PR head is not merge-ready because it would undo current main's unresolved-cron guard and can reintroduce two-second refire loops.

## What This Changes

The PR changes cron completion scheduling so successful `every` jobs clamp immediate or overdue next runs behind `MIN_REFIRE_GAP_MS`, with focused regression tests for that behavior.

## Best Possible Solution

Keep the every-schedule refire floor, but apply it on top of current main so cron schedules still use `resolveCronNextRunWithLowerBound` and every schedules only clamp defined next-runs that land at or before `endedAt`.

## Reproduction Assessment

Yes. Source inspection shows current main can store an every-schedule completion result at the completion boundary, and the PR body includes copied live Docker before/after output for the overrun case.

## Solution Assessment

No, not at the current head. The intended every-schedule floor is the right fix shape, but the patch must preserve current main's unresolved cron behavior instead of synthesizing a retry when cron next-run resolution returns `undefined`.
