## Summary

Keep this PR open: the reported scheduler bug still exists on current main and the PR has useful proof and tests, but the current branch head is still not merge-safe because it bypasses current main's unresolved-cron guard.

## What This Changes

This PR changes cron completion scheduling so overrun `every` jobs are clamped behind `MIN_REFIRE_GAP_MS` and adds focused regression tests for that behavior.

## Best Possible Solution

Rebase and repair the scheduler hunk so only successful `every` completions with `naturalNext <= endedAt` are clamped, while cron completions continue using `resolveCronNextRunWithLowerBound`.

## Reproduction Assessment

Yes. Source inspection shows current main stores `naturalNext` directly for successful `every` completions, and the PR body includes copied live Docker output reproducing the immediate refire and the after-fix `endedAt + 2000` result.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right narrow fix, but the branch must preserve current main's unresolved-cron helper instead of synthesizing `minNext` when cron next-run resolution is undefined.
