## Summary

Keep this PR open. The every-schedule refire bug still appears valid on current main, but the branch still conflicts with the unresolved cron next-run clearing behavior and would reintroduce a short fallback retry for cron schedules whose next run cannot be resolved.

## What This Changes

The PR changes cron `applyJobResult` to clamp successful `every` schedules to a 2-second refire gap when the computed next run is at or before `endedAt`, and adds focused regression tests.

## Best Possible Solution

Repair the narrow scheduler change so successful `every` jobs only apply the 2-second floor when a defined next run is at or before `endedAt`, while preserving unresolved cron clearing, malformed every unscheduling, manual preserve-schedule behavior, focused regression coverage, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows an `every` job whose execution reaches or exceeds its interval can compute `naturalNext === endedAt`, and current main then persists that due timestamp directly; the PR body also includes live Docker reproduction for the same symptom.

## Solution Assessment

No, not as currently proposed. The direction is narrow, but the branch must preserve current main's unresolved-next-run clearing contract instead of falling back to `minNext` for cron schedules.
