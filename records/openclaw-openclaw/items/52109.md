## Summary

Keep this PR open: the every-schedule refire bug is still source-reproducible on current `main`, but the current PR branch regresses the newer unresolved cron next-run guard and is missing the required changelog entry.

## What This Changes

The PR changes cron `applyJobResult` so successful `every` schedules can be clamped by `MIN_REFIRE_GAP_MS`, and adds regression tests for overrun and exact-boundary interval jobs.

## Best Possible Solution

Update the branch so successful `every` jobs apply the 2-second floor only when a defined next run lands at or before `endedAt`, keep current cron unresolved-next-run clearing intact, and add the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an `every` job whose execution reaches an anchored boundary can compute `naturalNext === endedAt`, and current `main` writes that due timestamp directly for non-cron completions.

## Solution Assessment

No, not as currently proposed. The direction is narrow, but the PR should reuse or preserve the current cron lower-bound resolver so unresolved cron completions remain unscheduled instead of becoming a 2-second retry.
