## Summary

Keep open: the `every`-schedule refire bug is still source-reproducible on current main, but this PR head still conflicts with current unresolved-cron protection by synthesizing a two-second retry when cron next-run resolution returns `undefined`.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs receive the `MIN_REFIRE_GAP_MS` floor and adds focused regression tests.

## Best Possible Solution

Apply an `every`-only overrun floor on current main while preserving `resolveCronNextRunWithLowerBound` for cron completions and keeping focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible. Current main can store an unfloored `every` next run at the just-finished timestamp after an overrun, and the PR body includes live Docker before/after output for that behavior.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right direction, but the branch must compose with current main's unresolved-cron helper instead of falling back to `endedAt + MIN_REFIRE_GAP_MS` for unresolved cron schedules.
