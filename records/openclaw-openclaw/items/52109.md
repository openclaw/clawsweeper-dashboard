## Summary

Keep this PR open. The underlying every-schedule refire bug still appears valid on current main, but the proposed branch still conflicts with current main's unresolved cron next-run clearing behavior and would reintroduce a 2-second refire fallback for cron schedules whose next run cannot be resolved.

## What This Changes

The PR changes cron `applyJobResult` to clamp successful `every` schedules to a 2-second refire gap when the computed next run is at or before `endedAt`, and adds regression tests for overrun and exact-boundary every jobs.

## Best Possible Solution

Keep the narrow cron scheduler fix, but rebase or repair it so successful `every` jobs only get the 2-second floor when a defined computed next run is at or before `endedAt`, while unresolved cron next-runs and malformed every schedules remain unscheduled. Add regression tests for the every overrun case, exact-boundary behavior, malformed `everyMs`, preserved manual schedules, and current #66019 unresolved cron behavior, plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is an `every` job whose execution reaches or exceeds its interval: current main can compute `naturalNext === endedAt`, then the non-cron completion branch persists that value directly. The PR body also includes live Docker reproduction and post-fix verification for the same symptom.

## Solution Assessment

No, not as currently proposed. The direction is narrow and maintainable, but the patch must preserve current main's unresolved-next-run clearing contract instead of falling back to `minNext` for cron schedules.
