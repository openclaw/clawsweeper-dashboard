## Summary

Keep open: the bug is valid and the branch targets the right scheduler path, but the current PR head still regresses the current-main unresolved cron next-run contract by scheduling unresolved cron completions two seconds later.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs receive a minimum 2-second refire gap and adds focused Vitest regression coverage.

## Best Possible Solution

The scheduler should floor overrun `every` completions while retaining current-main unresolved-cron clearing semantics and regression coverage.

## Reproduction Assessment

Yes. At source level, current main stores unfloored `naturalNext` for successful `every` jobs and the `every` calculation can return the just-finished timestamp after an overrun; the PR body also includes copied live Docker before/after output.

## Solution Assessment

No, not as currently proposed. The `every`-schedule clamp is the right direction, but the branch must compose with current main's unresolved-cron helper instead of scheduling unresolved cron completions two seconds later.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve unresolved cron completions as unscheduled:** `src/cron/service/timer.ts:470`
