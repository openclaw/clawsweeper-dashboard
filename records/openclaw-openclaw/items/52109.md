## Best Possible Solution

Rebase and repair the branch so overdue `every` completions get the refire floor while cron schedules with unresolved next-runs continue to clear through `resolveCronNextRunWithLowerBound`, then land with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main stores `naturalNext` directly for successful `every` completions, and the anchor calculation can return the completion boundary; the PR body also includes copied live Docker before/after output.

## Solution Assessment

No, not at the current branch head. The narrow `every` refire floor is the right direction, but the branch must preserve the current-main unresolved cron guard instead of falling back to `endedAt + MIN_REFIRE_GAP_MS` for cron schedules.
