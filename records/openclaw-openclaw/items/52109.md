## Summary

Keep this PR open. It targets a real cron scheduler bug that current main still appears to have for successful `every` schedules, but the branch is not merge-ready because it reintroduces the unresolved cron next-run fallback that current main already fixed in #66083/#66113.

## What This Changes

The PR changes cron completion scheduling so successful every-schedule jobs get a minimum refire gap when their computed next run is not after the just-finished run, and adds regression tests for exact-boundary and overrun every schedules.

## Best Possible Solution

Land a small rebased cron scheduler fix that applies a refire floor only to valid `every` completion next-runs that would otherwise be due immediately, while preserving current main's behavior for unresolved cron schedules, malformed `everyMs`, error backoff, one-shot jobs, and preserved manual-run schedules.

## Reproduction Assessment

Yes. A source-level reproduction is an enabled `every` job whose `lastRunAtMs + everyMs` is at or before `endedAt`; current main can compute `naturalNext === endedAt` and then persist it directly for the non-cron completion branch. The PR body also includes live Docker reproduction details for the same refire symptom.

## Solution Assessment

No. The intended direction is narrow and useful, but the current branch is not the best fix because it routes `naturalNext === undefined` cron completions back to `endedAt + MIN_REFIRE_GAP_MS`. The safer implementation is to reuse the current cron lower-bound resolver for cron schedules and add the `every` floor only when `naturalNext` is defined and not after `endedAt`.
