## Best Possible Solution

## Reproduction Assessment

Yes. Current main stores successful `every` completions as `naturalNext`, and source inspection shows `computeJobNextRunAtMs` can return the completion boundary after an overrun; the PR body also includes copied live Docker before/after output.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right narrow fix, but the branch must preserve `resolveCronNextRunWithLowerBound` instead of synthesizing `endedAt + MIN_REFIRE_GAP_MS` for unresolved cron schedules.
