## Summary

Keep this PR open: the reported `every` overrun bug is still present on current main, but this PR head reintroduces a newer unresolved-cron refire path and should not merge as-is.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs get a minimum refire gap and adds focused regression tests.

## Best Possible Solution

Apply an `every`-only overrun floor on current main while keeping `resolveCronNextRunWithLowerBound` for cron completions and retaining focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main can compute and store an unfloored `every` next run at the just-finished timestamp after an overrun, and the PR body includes live Docker before/after output for that behavior.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right direction, but the patch must compose with current main's unresolved-cron helper instead of falling back to `endedAt + MIN_REFIRE_GAP_MS` for unresolved cron schedules.
