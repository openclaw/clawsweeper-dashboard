## Summary

Keep open: the reported scheduler bug is real and current main still leaves successful overrun `every` jobs on the raw computed next run, but this PR head should not merge because it bypasses current main's unresolved-cron protection.

## What This Changes

The PR changes cron completion scheduling so overrun `every` jobs receive a two-second refire floor and adds focused `applyJobResult` regression tests.

## Best Possible Solution

Keep the `every` overrun floor, but implement it on top of current main so cron completions still use `resolveCronNextRunWithLowerBound` and malformed `every` schedules stay unscheduled.

## Reproduction Assessment

Yes, from source inspection. Current main stores `naturalNext` for successful `every` completions, and the `every` computation can return the completion boundary after an overrun; the PR body also includes copied live Docker before/after output.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right narrow fix, but the branch must preserve current main's unresolved-cron helper instead of synthesizing `endedAt + MIN_REFIRE_GAP_MS` for unresolved cron schedules.
