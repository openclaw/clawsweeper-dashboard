## Summary

Keep this PR open: the reported `every`-schedule refire bug is still source-reproducible on current `main`, and the PR contains useful tests and live before/after proof, but the branch still needs a focused repair against current `main` because it would reintroduce the unresolved-cron refire behavior that main now explicitly prevents.

## What This Changes

The PR changes cron timer completion handling to apply a minimum refire gap to overrun `every` schedules and adds two regression tests for that scheduler edge case.

## Best Possible Solution

Layer the `every` overrun floor onto current main's `applyJobResult` while keeping cron schedules routed through `resolveCronNextRunWithLowerBound`, leaving malformed `every` schedules unscheduled, and retaining focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main stores `naturalNext` directly for successful enabled `every` jobs, and the `every` next-run calculation can return a boundary timestamp at or before `endedAt` after an overrun. The PR body also includes copied live Docker before/after output showing immediate refire before the fix and `endedAt + 2000` after the fix.

## Solution Assessment

No, not as currently proposed. The `every`-schedule floor is the right direction, but the branch needs to compose with current main's unresolved-cron helper instead of scheduling unresolved cron completions at the refire gap.
