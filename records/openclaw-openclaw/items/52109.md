## Summary

Keep this PR open because the reported cron `every` refire problem is real, but the current branch is not merge-ready: it conflicts with current `main`, can reintroduce the unresolved cron next-run refire loop fixed later on `main`, and shifts exact-boundary `every` jobs off their anchored cadence.

## What This Changes

The PR changes cron completion scheduling so successful `every`-schedule jobs apply a 2s refire floor when the computed next run is at or before `endedAt`, with regression tests.

## Best Possible Solution

Refresh the branch on current `main`, preserve the cron unresolved-next-run helper, and implement a separate `every` catch-up guard that advances to the next valid anchored future tick before falling back to a 2s safety floor.
