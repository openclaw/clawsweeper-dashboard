## Summary

Keep open: the reported `every`-schedule refire bug is source-reproducible and the PR has sufficient real Docker proof, but the branch is not mergeable because it revives the cron unresolved-next-run fallback that current main removed.

## What This Changes

The PR clamps successful `every` schedule next runs behind `MIN_REFIRE_GAP_MS` when the computed next run lands at or before completion, and adds two focused regression tests.

## Best Possible Solution

Keep the useful `every` refire clamp, but route cron schedules through the current `resolveCronNextRunWithLowerBound` contract so unresolved cron next runs remain unscheduled.

## Reproduction Assessment

Yes. Source inspection shows current main can persist an `every` job's computed next run at `endedAt`, and the PR body includes live Docker before/after evidence for the same tight-refire symptom.

## Solution Assessment

No, not as currently proposed. The intended `every` clamp is narrow, but the PR also restores a cron undefined-next-run fallback that current main removed in https://github.com/openclaw/openclaw/pull/66083.
