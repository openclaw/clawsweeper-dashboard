## Summary

Keep open: the PR addresses a real, source-reproducible `every`-schedule refire bug, but the branch is stale/conflicting and still reintroduces a cron fallback that current main deliberately removed.

## What This Changes

The PR changes cron timer completion handling so overrun `every` schedules get a minimum refire gap and adds two focused regression tests.

## Best Possible Solution

Layer the `every` overrun floor onto current main's `applyJobResult` while keeping cron schedules routed through `resolveCronNextRunWithLowerBound`, leaving malformed `every` schedules unscheduled, and retaining focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main stores `naturalNext` directly for successful enabled `every` jobs, and `computeJobNextRunAtMs` can return a boundary time at or before `endedAt` after an overrun. The PR body also includes copied live Docker before/after output showing immediate refire before the fix and `endedAt + 2000` after the fix.

## Solution Assessment

No, not as currently proposed. The `every`-schedule floor is the right direction, but this branch needs to compose with current main's unresolved-cron helper instead of scheduling unresolved cron completions at the refire gap.
