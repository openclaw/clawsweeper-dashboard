## Summary

Keep this PR open because it targets a real `every`-schedule refire bug that current main still has, but the branch is stale and would undo the newer unresolved-cron-next-run clearing contract before it can merge.

## What This Changes

The PR applies a minimum refire gap to successful `every` cron jobs whose computed next run lands at or before completion, and adds regression tests for those edge cases.

## Best Possible Solution

Apply the `every` refire floor on top of current main’s cron lower-bound helper, preserving unresolved cron schedules as unscheduled and keeping the regression coverage focused on the `every` overrun path.

## Reproduction Assessment

Yes. Source inspection shows current main can persist an `every` job's computed next run at `endedAt`, and the PR body includes live Docker before/after output for that tight-refire symptom.

## Solution Assessment

No, not as currently proposed. The intended `every` clamp is narrow and maintainable, but the stale branch must be adapted so cron `naturalNext === undefined` still clears instead of scheduling `endedAt + 2000`.
