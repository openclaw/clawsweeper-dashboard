## Summary

Keep open: the reported tight-refire bug for overrun `every` schedules is valid and the PR is close, but the current head is stale/dirty against `main` and would reintroduce current `main`'s unresolved-cron refire behavior.

## What This Changes

The PR clamps overrun `every` schedule completions behind `MIN_REFIRE_GAP_MS` and adds regression tests for exact-boundary and short-interval `every` jobs.

## Best Possible Solution

Layer the `every` overrun floor onto current `main` while keeping cron schedules routed through `resolveCronNextRunWithLowerBound`, leaving malformed `every` schedules unscheduled, and retaining focused regression coverage.

## Reproduction Assessment

Yes, at source level: current `main` stores `naturalNext` directly for successful enabled `every` jobs, and the `every` next-run calculation can return the completion timestamp after an overrun. The PR body also includes copied live Docker before/after output showing immediate refire before the fix and `endedAt + 2000` after the fix.

## Solution Assessment

No, not as currently proposed. The `every`-schedule clamp is the right direction, but the branch needs to compose with current `main`'s unresolved-cron helper instead of scheduling unresolved cron completions at the refire gap.
