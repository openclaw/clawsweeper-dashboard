## Summary

Keep open: the reported scheduler bug is credible and the PR is narrowly aimed, but the current branch still regresses current-main cron unresolved-next-run clearing and lacks the required changelog entry.

## What This Changes

The PR changes cron timer result handling so successful `every` schedules get a 2-second minimum refire gap when the computed next run is at or before completion, and adds regression tests for those interval edge cases.

## Best Possible Solution

Update the branch so `every` schedules clamp only due or past successful next runs while cron schedules keep current unresolved-next-run clearing, then add the user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main can persist an `every` job's computed next run at `endedAt`, and the PR body includes a live Docker reproduction of the same tight-refire symptom.

## Solution Assessment

No, not as currently proposed. The direction is narrow, but the implementation should preserve `resolveCronNextRunWithLowerBound` semantics for cron schedules and add the required changelog entry.
