## Summary

Keep open: the underlying every-schedule refire bug is valid and the branch has useful proof, but the current head is not mergeable as-is because its completion hunk would bypass current-main unresolved cron protection.

## What This Changes

The PR adds an every-schedule completion refire floor in `applyJobResult` and regression tests for overrun `every` jobs.

## Best Possible Solution

Rebase and repair the PR so successful `every` completions clamp only when `naturalNext <= endedAt`, while preserving current-main cron unresolved-next-run handling and regression coverage.

## Reproduction Assessment

Yes, from source inspection and the PR body’s copied live Docker output. Current main stores `naturalNext` for successful `every` completions, and the `every` computation can return the completion boundary after an overrun.

## Solution Assessment

No, not as currently proposed. The `every` clamp is the right narrow fix, but the branch must preserve current main’s `resolveCronNextRunWithLowerBound` behavior for unresolved cron schedules.
