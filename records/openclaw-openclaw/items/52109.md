## Summary

Keep open: the every-schedule refire fix is useful and source-reproducible, but the current PR head still conflicts with current main's unresolved-cron guard by synthesizing `endedAt + MIN_REFIRE_GAP_MS` when cron next-run resolution returns `undefined`.

## What This Changes

The PR changes cron completion scheduling so successful every-schedule jobs whose computed next run is at or before completion are delayed by `MIN_REFIRE_GAP_MS`, with focused regression tests.

## Best Possible Solution

Repair the branch so cron schedules continue using `resolveCronNextRunWithLowerBound`, while every schedules only clamp defined next-runs that land at or before `endedAt`.

## Reproduction Assessment

Yes, at source level: current main stores every-schedule completion results directly from `computeJobNextRunAtMs`, and that computation can return the completion boundary. The PR body also contains copied live Docker before/after output for the overrun case.

## Solution Assessment

No, not at the current PR head. The intended every-schedule floor is the right fix shape, but the patch must preserve current main's unresolved-cron behavior instead of synthesizing a cron retry when next-run resolution returns `undefined`.
