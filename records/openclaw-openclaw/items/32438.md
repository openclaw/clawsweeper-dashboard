## Summary

Keep open. Current main and v2026.5.20 still persist Gateway file logs as one JSONL stream, while the CLI and Control UI only format that JSONL at read/display time; the related sidecar PRs are closed unmerged and the merged date-boundary log fix is adjacent rather than a replacement.

## What This Changes

The issue requests a derived human-readable formatted log file beside the existing JSONL log file for operators who cannot use the Control UI when the Gateway is down.

## Best Possible Solution

Add the companion file in the logger layer from the resolved active `logging.file`, preserve the JSONL contract and all existing consumers, and cover redaction plus date/rotation lifecycle in focused docs and tests.

## Reproduction Assessment

Yes, at source level: current main writes one redacted JSONL payload to one configured file, while CLI and Control UI parse that JSONL at read/display time. I did not run a live Gateway because this review is read-only.

## Solution Assessment

Yes for the requested direction: a logger-layer sidecar is the narrowest maintainable path because it preserves the existing JSONL contract and still works when Gateway RPC/UI paths are unavailable. The implementation should deliberately define sidecar rotation/pruning and reuse the existing redaction path.
