## Summary

Keep open. Current main and latest stable still persist Gateway file logs as one JSONL stream; the shipped CLI fallback helps read that JSONL when local RPC is unavailable, but it does not create the requested persistent formatted companion file. The earlier duplicate is closed, and the related implementation PRs are closed unmerged rather than viable canonical fixes.

## What This Changes

The issue requests a derived human-readable formatted log file next to the existing JSONL Gateway file log, without changing current JSONL consumers or adding a new config key.

## Best Possible Solution

Land a focused logger-side implementation that derives a readable companion path from the active logging.file, writes redacted plain-text lines beside the unchanged JSONL file, and documents a bounded lifecycle for rotation and pruning.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main resolves and writes only one JSONL file path, and docs/UI still describe the file-log surface as JSONL. I did not run a live gateway because this review is read-only and tests would create artifacts, but the inspected code path is direct.

## Solution Assessment

Yes, the best direction is a narrow logger-owned companion file rather than changing CLI or Control UI consumers. The existing CLI local fallback is useful adjacent behavior, but it does not satisfy the persistent tail/grep file requested here.
