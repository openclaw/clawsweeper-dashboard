## Summary

Keep open. Current main and latest release v2026.5.12 still persist Gateway file logs as a single JSONL stream, while CLI and Control UI readability is produced by parsing that file at read time; the two same-author formatted-sidecar PRs were closed unmerged, and the merged date-boundary log discovery work is adjacent rather than a replacement.

## What This Changes

The issue requests a derived plain-text formatted log file alongside the existing JSONL file log so operators can inspect logs when the Gateway or Control UI is unavailable.

## Best Possible Solution

Land a focused logger-layer enhancement that derives a human-readable companion path from the active JSONL log file, writes redacted plain-text lines without changing existing JSONL consumers, and documents the lifecycle guarantees.

## Reproduction Assessment

Yes for the current behavior: source and docs inspection show current main and v2026.5.12 write only JSONL file logs and format them later for CLI/UI display. No live Gateway run was performed because this review is read-only.

## Solution Assessment

Yes. A logger-layer sidecar is the narrowest maintainable direction if it preserves the JSONL contract, shares redaction, and handles rolling/rotation deliberately; the closed PRs are useful context but should not be cherry-picked blindly.

## Risks / Open Questions

- The sidecar lifecycle needs an explicit rule for default rolling dates, JSONL rotation, and pruning so the formatted file does not grow or drift unexpectedly.
- The formatted file must reuse the same redaction path as JSONL file logs so it does not become a less-protected duplicate of sensitive content.
- Always writing a second file adds disk I/O and a new operator-visible artifact, so docs and tests should make the no-new-config behavior and cleanup behavior clear.
