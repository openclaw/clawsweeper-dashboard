## Summary

Keep open. Current main still writes Gateway file logs as a single JSONL stream and only formats logs at CLI/UI read time; the same-author implementation PRs are closed unmerged, while the merged date-boundary log discovery work is adjacent and does not create the requested durable formatted sidecar.

## What This Changes

Add a derived plain-text formatted log file beside the existing JSONL log so operators can tail or grep readable logs when the Gateway or Control UI is unavailable.

## Best Possible Solution

Land a focused logger-layer enhancement that derives the sidecar path from `logging.file`, writes redacted plain-text lines beside the unchanged JSONL file, and documents rotation/pruning behavior with regression coverage.

## Reproduction Assessment

Yes for the current behavior: source inspection shows file logging writes only one JSONL payload to `getResolvedLoggerSettings().file`, while CLI and UI readability is produced by parsing that file at read time. No live run was performed because this review was read-only.

## Solution Assessment

Yes. A logger-layer sidecar is the narrowest maintainable direction if it preserves the JSONL consumer contract, shares redaction, and handles rotation/pruning deliberately; the closed older PRs are useful context but should be replaced rather than revived blindly.
