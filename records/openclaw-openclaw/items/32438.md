## Summary

Keep open. Current main still writes Gateway file logs as a single redacted JSONL stream, while the CLI fallback and Control UI only parse that JSONL at display time. The earlier duplicate is closed, the same-author implementation PR was closed unmerged, and the merged date-boundary log discovery work is adjacent but does not provide the requested durable formatted companion file.

## What This Changes

The issue requests a derived human-readable formatted log file beside the unchanged JSONL Gateway log so operators can inspect logs with `tail` or `grep` when the Control UI cannot use `logs.tail`.

## Best Possible Solution

Add a focused logger-layer enhancement that derives the readable companion path from the resolved `logging.file`, writes redacted plain-text lines beside the unchanged JSONL stream, follows current-day rolling file changes deliberately, and updates operator docs plus regression coverage for redaction, rotation, and unchanged JSONL consumers.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: enable file logging and emit a log event; current `buildLogger` appends one JSONL payload to `activeFile`, and there is no formatted sidecar path in settings, schema, docs, CLI, or UI. I did not run a live write because this review is read-only.

## Solution Assessment

Yes. The requested logger-layer sidecar is the narrowest maintainable solution for offline host inspection when the Gateway is down; CLI local fallback and Control UI parsing help display-time access but do not create a durable plain-text file.
