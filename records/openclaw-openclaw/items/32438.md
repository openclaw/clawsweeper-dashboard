## Summary

Keep open. Current main still persists Gateway file logs as JSONL only, with readable formatting done by the CLI/Control UI at read time rather than by a durable formatted sidecar; the related implementation PRs are closed unmerged, and the merged rolling-log fallback work is adjacent but does not satisfy this request.

## What This Changes

Request adding a derived human-readable formatted log sidecar beside the unchanged Gateway JSONL file for operators who need disk logs when the Control UI is unavailable.

## Best Possible Solution

Land a focused logger-layer enhancement that derives a redacted human-readable companion path from logging.file, preserves the JSONL consumer contract, defines sidecar retention behavior, and updates operator docs plus regression coverage.

## Reproduction Assessment

Yes for the current behavior. Source inspection shows enabling file logging writes only the configured JSONL file, while CLI/UI readability comes from parsing and formatting that JSONL at read time.

## Solution Assessment

Yes, the requested logger-layer sidecar is the narrowest maintainable direction if it shares redaction and handles rotation/pruning deliberately. The older PRs are useful context but are closed unmerged and should be replaced by a current-main implementation.
