## Summary

Keep open. Current main and the latest release still persist Gateway file logs as a single JSONL stream, while CLI and Control UI readability is produced by parsing that JSONL at read time; the formatted-sidecar PRs are closed unmerged, and nearby log timestamp/UI fixes do not supersede this request.

## What This Changes

Request a derived human-readable companion log file beside the existing JSONL Gateway file log without changing JSONL consumers.

## Best Possible Solution

Land a focused logger-layer enhancement that derives a human-readable companion path from logging.file, writes redacted plain-text lines beside the unchanged JSONL file, and documents rotation/pruning behavior.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.12 write one JSONL file and format logs only when CLI/UI read that file. I did not run a live Gateway because this was a read-only review.

## Solution Assessment

Yes, the requested logger-layer sidecar is the narrow maintainable direction if it preserves the JSONL contract, shares redaction, and defines rolling/rotation behavior. The closed PRs are useful context but should not be cherry-picked blindly.
