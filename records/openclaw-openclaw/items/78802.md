## Summary

Close as cannot reproduce: the exact exec-backed plan should fail closed without `--allow-exec`, and current-main source writes the sibling `keyRef` into the same auth-store object or throws before any plaintext deletion can create the reported empty-stub credentialless state.

## What This Changes

The issue asks to harden `secrets apply` so auth-profile sibling-ref migrations cannot remove plaintext `key` without writing the replacement `keyRef`.

## Best Possible Solution

Keep the current idempotent write-before-delete behavior and reopen only with a deterministic current-main repro that produces a missing `keyRef` after a successful apply.

## Reproduction Assessment

No. The exact provided exec-backed command is rejected without `--allow-exec`, and the non-exec sibling-ref source path writes `keyRef` into the same mutable auth store for empty stubs; no failing current-main path was found.

## Solution Assessment

No. The suggested `!wroteRef` throw is not the best fix because idempotent re-apply legitimately returns false when `keyRef` already matches; if new evidence appears, a postcondition guard plus regression test would be safer.
