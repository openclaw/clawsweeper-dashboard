## Summary

Keep open: current `main` still has a plausible no-arg `session_status` path that can read the policy/sandbox session entry instead of the live run session, so it can report a stale `Think` value even though the UI/session row was patched to `high`.

## What This Changes

The issue reports that after setting `/think high`, the `session_status` tool reports `Think: off` while the UI shows the current thinking level as `high`.

## Best Possible Solution

Make a no-arg `session_status` call resolve to the live run session when `runSessionKey` is available, while preserving explicit cross-session visibility checks and existing `sessionKey="current"` behavior.

## Reproduction Assessment

Yes, source-level: construct `session_status` with a stale `agentSessionKey` plus a live `runSessionKey` whose entry has `thinkingLevel: high`, then call it with no args; current code resolves the stale key and passes that entry's `thinkingLevel` to the status card. This was not live-run reproduced in the read-only checkout.

## Solution Assessment

Yes: the narrow fix is in `session-status-tool` session resolution, with a regression test for implicit no-arg status calls under a sandbox/policy key. Changing UI thinking persistence or provider thinking defaults would miss the stale-session lookup root cause.
