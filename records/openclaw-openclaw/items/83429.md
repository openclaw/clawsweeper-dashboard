## Summary

Keep open, but this PR should not merge as-is: it strips shell commands from every `openclaw-macos` node even though current macOS app code declares and handles `system.which`/`system.run`, and the linked issue's latest human evidence points to connect-time probe readiness rather than missing command implementation.

## What This Changes

The PR adds app-node detection to gateway node command policy, forwards client metadata into command allowlist checks, and adds a gateway E2E test for macOS app nodes avoiding remote bin probing.

## Best Possible Solution

Revise away from blanket app-node shell-command removal; fix the initial remote-bin probe readiness/defer path while preserving declared and approved remote macOS app commands, then prove it with a regression that declares `system.which`/`system.run` plus redacted real macOS app and gateway evidence.

## Reproduction Assessment

Yes, as a source-level reproduction: current main can start the automatic remote-bin probe during node-role connect before the Swift client starts normal event listening, and the linked issue confirms automatic timeouts while manual `system.which` succeeds. The PR's new test does not reproduce that path because it connects with no declared shell commands.

## Solution Assessment

No. The proposed blanket command removal is broader than the bug and would break valid remote macOS app node exec/remote-skill behavior; a readiness or deferred initial-probe fix is the narrower maintainable direction.
