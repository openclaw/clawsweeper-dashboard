## Summary

Keep open. Current main has partial Discord slash-command deploy mitigations, but it still lacks the requested deploy-mode config and persisted cross-restart command hash, so the reported restart-driven deploy pressure remains actionable.

## What This Changes

Expose Discord slash-command deployment control so unchanged command sets can be skipped across restarts and deployment can be disabled when needed.

## Best Possible Solution

Add a Discord-owned deploy-mode config with persisted successful command-set hashes per account/application/scope while keeping reconcile, rate-limit warning, and explicit native-command clearing semantics intact.

Fix the Discord slash-command deploy pressure reported in https://github.com/openclaw/openclaw/issues/75888. Add a Discord-owned config surface for deploy mode, preferably `channels.discord.slashCommandDeploy` with account-level inheritance if consistent with existing Discord account config patterns, supporting `always`, `changed-only`, and `disabled`; default should avoid unnecessary cross-restart redeploys. Persist successful command-set hashes per account/application/scope under the OpenClaw cache/config runtime location used by this codebase, not in core startup migrations, and skip deploy when the persisted hash matches. Update config types, zod schema, generated config docs/metadata as required by repo scripts, Discord docs, and CHANGELOG.md. Add focused tests for schema acceptance/rejection, persisted changed-only skip across fresh deployer/client instances, disabled mode, and clearing semantics.
