## Summary

Keep open. Current main still lacks the requested Discord slash-command deploy policy, durable cross-restart fingerprints, and per-process deploy serialization, and #75961 is an open closing PR that should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests a Discord `slashCommandDeploy` config mode with changed-only persistence, a disabled option, and safer multi-account deploy behavior around Discord 429s.

## Best Possible Solution

Review and land #75961 if correct, or replace it with a narrow Discord plugin-local config, persistence, docs, and test change that preserves native-command clearing semantics.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: configure multiple native-enabled Discord accounts and restart the gateway; current main schedules a background reconcile deploy for each account, while the exact 429 response depends on live Discord state.

## Solution Assessment

Yes. A Discord-owned deploy policy with non-secret persisted command fingerprints and serialized background deploy coordination is the narrow maintainable direction; the open PR is the active implementation candidate to validate.
