## Summary

Keep open. Current main has in-process deploy dedupe, `retry_after` backoff, and account startup staggering, but it still lacks a deploy-mode config surface and persisted cross-restart command hash, so restart-driven Discord slash-command deploy pressure remains actionable.

## What This Changes

Expose a Discord slash-command deploy mode such as `always`, `changed-only`, or `disabled`, with unchanged-deploy skipping and better rate-limit behavior across multiple accounts.

## Best Possible Solution

Add a Discord-owned deploy-mode config with persisted successful command-set hashes per account/application/scope, while keeping current reconcile, retry, and explicit native-command clearing semantics intact.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: restart constructs a fresh `Client`/`DiscordCommandDeployer` with empty hash state, and each native-enabled Discord account schedules a deploy on startup; only the exact 429 rate depends on Discord's live API state.

## Solution Assessment

Yes. A provider-local config/schema/runtime change is the narrow maintainable fix; it should not add bundled Discord behavior to core or change other channels' native command defaults.
