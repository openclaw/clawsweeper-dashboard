## Summary

Keep this open: current main partially reduces the reported Discord restart churn with persisted command-set hashes, but it still does not expose the requested deploy-policy config, and the linked implementation PR remains open as the active path for the remaining work.

## What This Changes

The issue requests configurable Discord slash-command deploy behavior so operators can use changed-only or disabled startup deploy modes and avoid repeated Discord 429s on restarts.

## Best Possible Solution

End state should be either a documented, config-safe Discord deploy-policy implementation or an explicit maintainer decision that the persisted hash cache plus commands.native controls are the supported model.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live high-confidence repro was run; source confirms startup still schedules reconcile when native commands are enabled, but reproducing the 429 cluster requires a live multi-account Discord setup.

## Solution Assessment
