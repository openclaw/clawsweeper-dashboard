## Summary

Keep open. Current main has in-process deploy dedupe, `retry_after` backoff, and account startup staggering, but it still lacks a Discord deploy-mode config surface and persisted cross-restart command hashes, so restart-driven multi-account slash-command deploy pressure remains unresolved.

## What This Changes

Expose a Discord slash-command deploy mode that can skip unchanged startup deploys or disable deploys, with persisted hashes and safer 429 handling.

## Best Possible Solution

Add a Discord-owned deploy-mode config with persisted successful command-set hashes per account/application/scope while retaining current reconcile, retry, and explicit native-command clearing behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: restart creates a fresh `Client` and `DiscordCommandDeployer` with empty process-local hash state, then each native-enabled Discord account schedules a background reconcile deploy; only the exact live 429 response depends on Discord API state.

## Solution Assessment

Yes. A provider-local config/schema/runtime/docs change is the narrow maintainable fix; it should not add Discord-specific defaults to core or change Slack/Telegram native-command behavior.
