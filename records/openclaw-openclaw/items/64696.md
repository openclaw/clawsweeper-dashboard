## Summary

Keep open. Current main still shows the requested memory-wiki bridge and lint durability gaps in source: raw link extraction is markdown-blind, broken-link lint does not resolve page-relative Markdown targets, report links are emitted as vault-root paths from `reports/`, bridge render fingerprints lack a renderer-contract token, raw bridge content is fenced without reply-tag immunity, and source-sync skip logic still trusts matching state plus file existence. The earlier full fix PR #65197 is closed unmerged, and open PR #73722 only covers part of the lint target-normalization surface, leaving the bridge render/cache criteria unhandled.

## What This Changes

The issue requests memory-wiki bridge import and lint hardening for relative report links, reply tags in bridged source pages, renderer invalidation, and malformed cached source-page repair.

## Best Possible Solution

Land an extension-owned memory-wiki fix with regression coverage that makes link extraction or resolution safe for fenced bridge content, resolves native Markdown links relative to the page containing them, emits report-local native links from generated reports, includes a bridge renderer-contract token in source render fingerprints, and validates existing cached source pages before import skips.

## Reproduction Assessment

Yes. The current source gives a high-confidence reproduction path without mutating the checkout: a native link from `reports/stale-pages.md` is extracted as a raw target and compared directly, and a bridged markdown page containing `[[reply_to_current]]` inside the raw fenced content is still scanned by `extractWikiLinks`.

## Solution Assessment

Yes, the memory-wiki plugin is the right fix boundary. The safest implementation should coordinate with #73722 for lint target normalization and still cover the bridge renderer/cache and report-link criteria that the open partial PR does not address.
