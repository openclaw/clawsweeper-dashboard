## What This Changes

The PR adds `::add-mask::${auth_header}` after derived GitHub checkout auth headers in `.github/workflows/ci.yml` and the two Testbox CI workflows.

## Best Possible Solution

Merge the focused mask registrations after workflow-security review and once required checks are green or explicitly unblocked; handle unrelated zizmor findings separately.

## Reproduction Assessment

Yes, at source level. Current main has 13 custom checkout blocks that derive `auth_header` from `github.token` and pass it to Git as an authorization extraheader before any matching mask registration; I did not run a live leaking-token workflow against current main.

## Solution Assessment

Yes. Registering the already-computed derived header immediately after construction and before the first Git invocation is the narrowest maintainable fix under the checked GitHub Actions `add-mask` contract. ([docs.github.com](https://docs.github.com/en/actions/writing-workflows/choosing-what-your-workflow-does/workflow-commands-for-github-actions?utm_source=openai))
