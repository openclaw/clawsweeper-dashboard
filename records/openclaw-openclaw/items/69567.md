## Summary

Keep open for maintainer handling: current main still has the unmasked derived checkout auth header pattern, and this PR is a focused security hardening with no blocking code-review findings.

## What This Changes

The PR adds `echo "::add-mask::${auth_header}"` after each derived custom checkout Basic-auth header in the CI and Testbox workflows.

## Best Possible Solution

Merge the focused mask registrations after workflow-security review and either green checks or maintainer acknowledgement that the remaining failures are unrelated.

## Reproduction Assessment

Yes, at source level. Current main has 13 custom checkout blocks that derive a base64 auth header and use it in Git before any `add-mask`, and the attached Actions screenshot demonstrates the same value class being redacted after registration.

## Solution Assessment

Yes. Registering the already-computed derived header immediately after construction and before the first Git invocation is the narrowest maintainable fix under the GitHub Actions masking contract.
