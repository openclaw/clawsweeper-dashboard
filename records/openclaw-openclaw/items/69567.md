## Summary

This PR remains useful: current main still derives checkout authorization headers in the affected workflow checkout blocks without registering the derived value with the runner masker, and the PR addresses that exact logging exposure with a narrow workflow change.

## What This Changes

The PR adds `echo "::add-mask::${auth_header}"` after derived checkout auth header construction in 13 custom checkout blocks across `.github/workflows/ci.yml` and the two Testbox workflows.

## Best Possible Solution

Rebase or otherwise resolve the stale workflow conflict, then merge the focused mask registrations after workflow-security review and required-check handling.

## Reproduction Assessment

Yes, at source level. Current main has 12 custom checkout blocks that derive `auth_header` from `CHECKOUT_TOKEN` and pass it to Git before registering any matching mask; I did not run a live leaking-token workflow against main.

## Solution Assessment

Yes. Registering the derived header with `::add-mask::` immediately after construction and before the first Git invocation is the narrowest maintainable fix under the GitHub Actions workflow-command contract.
