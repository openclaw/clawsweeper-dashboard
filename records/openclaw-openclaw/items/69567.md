## Summary

Keep open: this is a focused, useful security hardening that current main still lacks; the patch looks correct, but it is a security-sensitive workflow change with failing gates that need maintainer handling before merge.

## What This Changes

The PR adds `::add-mask::${auth_header}` immediately after derived checkout auth header construction in `.github/workflows/ci.yml` and both Testbox checkout workflows.

## Best Possible Solution

Merge the focused mask registrations after workflow-security owner review and accepted gate state; track the unrelated workflow audit failures separately.

## Reproduction Assessment

Yes. Current main shows derived Basic-auth headers passed to Git without `add-mask`, and the inspected Actions demo shows that the same class of derived value is redacted after registration.

## Solution Assessment

Yes. Registering the derived header with `add-mask` immediately after construction and before any Git command is the narrowest maintainable fix for this logging surface.
