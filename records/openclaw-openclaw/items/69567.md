## Summary

Keep open. The PR is a valid CI token log-redaction hardening, but the current head still changes only `.github/workflows/ci.yml` while current `main` has the same derived checkout auth header pattern in two Testbox workflows. Merging this as-is would leave the same exposure in workflow PR paths.

## What This Changes

The PR adds `echo "::add-mask::${auth_header}"` after each derived checkout auth header construction in `.github/workflows/ci.yml`.

## Best Possible Solution

Extend the same focused log-redaction hardening to every current custom checkout block that derives `auth_header`, covering `.github/workflows/ci.yml`, `.github/workflows/ci-check-testbox.yml`, and `.github/workflows/ci-build-artifacts-testbox.yml` without changing job graph, permissions, runner selection, action refs, or checkout semantics.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to inspect current `main` and the PR diff: the PR masks 11 `ci.yml` blocks, while the two Testbox workflows still build `auth_header` and pass it to Git without `::add-mask::` before use.

## Solution Assessment

No. The approach is correct for the `ci.yml` blocks it touches, but it is incomplete; the narrow maintainable fix is to add the same mask line immediately after `auth_header` construction in both Testbox workflows and then run or inspect the workflow audit and normal PR CI.
