## Summary

Keep open: current main still derives checkout Basic-auth headers from `github.token` and passes them to Git without registering the derived value with the Actions masker, while this PR applies the narrow `add-mask` mitigation and has real screenshot proof. The remaining gate is maintainer workflow-security review plus required-check handling, not a code repair finding.

## What This Changes

Adds `echo "::add-mask::${auth_header}"` after each custom checkout `auth_header` derivation in the CI and Testbox workflow files.

## Best Possible Solution

Merge the focused mask registrations after workflow-security review and check resolution, while keeping unrelated zizmor findings in other workflows separate.

## Reproduction Assessment

Yes, at source level. Current main has 13 custom checkout blocks that derive a base64 Basic-auth header from `github.token` and pass it to Git before any mask registration, and the attached Actions screenshot demonstrates the redaction behavior after registration.

## Solution Assessment

Yes. Registering the already-computed derived header immediately after construction and before the first Git invocation is the narrowest maintainable fix under GitHub Actions' masking contract.
