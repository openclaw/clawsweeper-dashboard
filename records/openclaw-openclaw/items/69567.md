## Summary

Keep open. Current main still has unmasked derived checkout auth headers, and the PR is a narrow hardening patch, but the exact head still has failing workflow security checks that need maintainer resolution before merge.

## What This Changes

The PR adds `echo "::add-mask::${auth_header}"` after derived checkout Basic-auth header construction in 11 `ci.yml` checkout blocks and two Testbox workflow checkout blocks.

## Best Possible Solution

Land this focused masking hardening across all custom checkout blocks once the exact-head workflow security audit is green or explicitly accepted as unrelated.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the gap: each custom checkout builds a base64 `auth_header` and uses it in Git before any mask registration, while the PR head adds the mask before first use.

## Solution Assessment

Yes, with the CI gate caveat. `::add-mask::` is the narrow runner-native fix for this derived-secret logging surface, and the PR avoids changing permissions, action refs, job graph, or checkout semantics.
