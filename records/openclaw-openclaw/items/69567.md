Adds `::add-mask::${auth_header}` after each custom checkout script derives the base64 GitHub token header in `ci.yml` and the two Testbox workflow files.

## Best Possible Solution

Refresh the PR against current main, keep the mask immediately after every `auth_header` assignment in all affected workflow files, and merge after workflow-security owner review and required checks are unblocked.

## Reproduction Assessment

Yes, at source level. Current main derives a base64 checkout auth header and uses it in Git `http.extraheader` before registering any mask, and the attached Actions screenshot demonstrates that `::add-mask::` redacts the derived value after registration.

## Solution Assessment

Yes. Registering the already-computed derived header immediately after construction and before the first Git command is the narrowest maintainable fix under the GitHub Actions masking contract, with the caveat that the branch needs a current-main refresh.
