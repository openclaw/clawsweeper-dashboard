## Summary

Keep open. Current main still has the unmasked derived checkout headers, and this PR’s patch matches the affected blocks, but exact-head security checks are still failing and need maintainer CI/security-gate judgment before merge.

## What This Changes

The PR adds `echo "::add-mask::${auth_header}"` after each custom checkout Basic-auth header in the main CI and Testbox workflows.

## Best Possible Solution

Merge the focused masking hardening once the exact-head security gates are green or the unrelated workflow audit failures are resolved or explicitly accepted separately.

## Reproduction Assessment

Yes. Current main at `9880b7c914c0e768c4c57920f041b6abb7796b69` has a high-confidence static reproduction: 13 custom checkout blocks compute a base64 `auth_header` and use it in Git without first registering it with `::add-mask::`.

## Solution Assessment

Yes. `add-mask` is the documented GitHub Actions mechanism for this generated log-sensitive value, and placing it immediately after construction and before `git fetch` is the narrowest maintainable hardening path.
