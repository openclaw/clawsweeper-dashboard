## Summary

Keep open: current main still has unmasked derived Basic-auth headers in the touched custom checkout blocks, and this PR is a focused security hardening with sufficient real-run proof. It still needs maintainer handling because it changes CI token logging and current gates are failing for unrelated workflow/proof-check reasons.

## What This Changes

The PR adds `echo "::add-mask::${auth_header}"` after derived checkout auth-header construction in `.github/workflows/ci.yml` and both Testbox checkout workflows.

## Best Possible Solution

Merge the focused mask registrations after workflow-security review and green or explicitly accepted gates, then track the performance-workflow header separately if repo-wide coverage is desired.

## Reproduction Assessment

Yes. Current main source shows derived checkout auth headers used as Git extraheaders without `add-mask`, and the inspected GitHub Actions proof shows the same class of derived value being redacted after registration.

## Solution Assessment

Yes. Adding `add-mask` immediately after header construction and before the first Git use is the narrowest maintainable fix and does not change runners, permissions, job graph, or checkout semantics.
