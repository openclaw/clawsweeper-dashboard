## Summary

Keep open. The PR targets a real Control UI reverse-proxy bug, but the current head still preserves unflagged legacy gateway records, so upgraded browsers can keep connecting to the stale sibling path that the PR is meant to fix.

## What This Changes

The PR adds a persisted `gatewayUrlAutoDerived` flag, a gateway URL resolution helper, page-derived scoped settings writes, and storage test updates for path-prefixed Control UI deployments.

## Best Possible Solution

Update this PR before merge. The final fix should resolve the legacy fallback at read time or otherwise seed the current page scope during startup, preserve explicit scoped/user gateway overrides and token isolation, add focused tests for `/a` to `/b` legacy fallback plus custom override cases, preserve current-main storage fields, and add the required user-facing changelog entry.

## Reproduction Assessment

Yes. Seed `openclaw.control.settings.v1` with `gatewayUrl: "wss://gateway.example/a"`, set the page location to `/b`, and call `loadSettings()`; current main and this PR's undefined-flag path both select the stored `/a` URL instead of the page-derived `/b` URL.

## Solution Assessment

No. The flag and page-scoped key direction is useful, but this head is not yet the best fix because unflagged legacy records are preserved and the page-scoped write does not run during ordinary `loadSettings()` startup.
