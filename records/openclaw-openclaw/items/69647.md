## Summary

Keep open: the reported reverse-proxy/basePath bug is valid, but the latest PR still preserves unflagged legacy gateway URLs on the first upgraded load and lacks contributor-provided real behavior proof.

## What This Changes

The PR adds `gatewayUrlAutoDerived` metadata, a `resolveGatewayUrl()` helper, page-derived scoped localStorage writes, and storage test updates for Control UI gateway URL persistence.

## Best Possible Solution

Resolve legacy fallback during `loadSettings()` so the first upgraded visit selects the page-derived gateway for sibling base paths while preserving explicit same-host custom paths, then add regression coverage and real reverse-proxy proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed the shared legacy settings key with `gatewayUrl: "wss://gateway.example/a"`, set the page under `/b`, and call `loadSettings()`; current main and PR head both preserve `/a` on the first upgraded load.

## Solution Assessment

No. The flag and page-scoped-key direction is useful, but the undefined-flag branch returns the stored legacy URL, so the first upgraded load still connects to the wrong backend.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Migrate unflagged legacy gateway URLs on load:** `ui/src/ui/storage.ts:128`

Because `loadSettings()` reads that legacy record before any page-scoped save can run, the first upgraded load still connects to the wrong backend.
