## Summary

Keep open. The underlying current-main bug is valid, but the latest PR head still preserves unflagged legacy records, so upgraded browsers that only have the shared legacy settings key can continue using the old sibling-path gateway URL on first load.

## What This Changes

The PR adds a persisted `gatewayUrlAutoDerived` flag, a gateway URL resolver, page-derived scoped localStorage writes, and storage test expectation updates for path-prefixed Control UI gateway URLs.

## Best Possible Solution

Update this PR before merge. The final fix should resolve the legacy fallback at read time or otherwise seed the current page scope during startup, preserve explicit scoped/user gateway overrides and token isolation, add focused `/a` to `/b` legacy-fallback and custom-override tests, keep current-main storage fields intact, and add the required user-facing changelog entry.

## Reproduction Assessment

Yes. Seed `openclaw.control.settings.v1` with `gatewayUrl: "wss://gateway.example/a"`, set the page location to `/b`, and call `loadSettings()`; current main and this PR's undefined-flag path both select the stored `/a` URL instead of the page-derived `/b` URL.

## Solution Assessment

No. The flag and page-scoped key direction is useful, but this head is not yet the best fix because unflagged legacy records are preserved and the page-scoped write does not run during ordinary `loadSettings()` startup.
