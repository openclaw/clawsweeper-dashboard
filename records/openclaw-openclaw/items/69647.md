## Summary

Keep this PR open: it targets a real Control UI reverse-proxy storage bug, but the current patch still leaves upgraded browsers with unflagged legacy settings pinned to the old gateway URL on the first load and the external PR has no real after-fix behavior proof.

## What This Changes

The PR adds persisted gateway URL derivation metadata, load-time gateway URL resolution, page-derived scoped localStorage writes, and focused Control UI storage test updates.

## Best Possible Solution

Resolve legacy fallback during loadSettings so the first upgraded visit selects the page-derived gateway for sibling base paths while preserving explicit same-host custom gateway paths, then add regression coverage and real reverse-proxy proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed the shared legacy settings key with gatewayUrl "wss://gateway.example/a", set the page under /b, and call loadSettings; current main and PR head both preserve the stored /a URL on the first upgraded load.

## Solution Assessment

No. The flag and page-scoped-key direction is useful, but returning stored for an undefined derivation flag defers the fix until after a later save and leaves the first upgraded load broken.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Migrate unflagged legacy gateway URLs on load:** `ui/src/ui/storage.ts:128`
  - body: For upgraded browsers that only have the shared legacy settings key, gatewayUrlAutoDerived is absent, so this branch preserves a stale sibling URL such as wss://gateway.example/a while the page is under /b. Because loadSettings reads that legacy record before any page-scoped save can run, the first upgraded load still connects to the wrong backend.
