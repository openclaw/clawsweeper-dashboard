## Summary

Keep open. Current main still has the stale gateway fallback path, and the PR head does not fix the first upgraded load for browsers that only have an unflagged shared legacy settings record; that is the central reverse-proxy multi-instance case the PR is trying to repair.

## What This Changes

The PR adds a persisted `gatewayUrlAutoDerived` flag, a gateway URL resolver, page-derived scoped settings writes, and focused storage test updates for path-prefixed Control UI deployments.

## Best Possible Solution

Update this PR before merge so `loadSettings()` resolves unflagged legacy fallback data correctly on first startup, preserves explicit scoped/user gateway overrides and token isolation, adds regression coverage for `/a` to `/b` legacy fallback plus custom override cases, keeps #73987 separate, and adds the required `CHANGELOG.md` entry.

## Reproduction Assessment

Yes. Seed `openclaw.control.settings.v1` with `gatewayUrl: "wss://gateway.example/a"`, set the page location under `/b`, and call `loadSettings()`; current main and the PR head's undefined-flag path both select the stored `/a` URL instead of the page-derived `/b` URL before any later save happens.

## Solution Assessment

No. The flag and page-scoped key direction is useful, but this PR is not yet the best fix because unflagged legacy records are preserved at read time and the page-scoped write only happens during `persistSettings()`. A safer fix should resolve the legacy fallback during `loadSettings()` or track which storage key supplied the record while preserving explicit user overrides.
