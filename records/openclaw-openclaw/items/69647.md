## Summary

Keep open. The PR is aimed at a real current-main Control UI storage bug, but the latest head still preserves unflagged legacy records on first upgraded load, so a browser that only has the shared legacy settings key can still carry a `/a` gateway URL while visiting `/b`. The existing review discussion correctly identified this remaining defect; the branch should be repaired before merge.

## What This Changes

The PR adds a persisted `gatewayUrlAutoDerived` flag, resolves stored gateway URLs through that flag, writes an additional page-derived scoped settings key, and updates Control UI storage tests.

## Best Possible Solution

Repair the PR so legacy fallback is resolved before the WebSocket URL is used, or otherwise seed the page-derived scope during startup, while preserving explicit user gateway overrides and existing token/session isolation. Add focused tests for `/a` to `/b` legacy fallback, same-host custom override preservation, page-scoped cache updates, and the required changelog entry.

## Reproduction Assessment

Yes. Seed `openclaw.control.settings.v1` with `gatewayUrl: "wss://gateway.example/a"` and no `gatewayUrlAutoDerived`, set the page location under `/b`, and call `loadSettings()`; current main and the PR head’s undefined-flag path both keep the stored `/a` URL.

## Solution Assessment

No. The explicit flag is a safer direction than the original same-host path heuristic, but this implementation does not fix upgraded clients on first load because unflagged legacy records are preserved and the page-scoped write happens only during `persistSettings()`.
