## Summary

Keep open. The reported Control UI reverse-proxy storage bug is valid, but PR head still preserves unflagged legacy gateway URLs on the first upgraded load, so existing browsers can still connect to the sibling backend before any page-scoped save occurs; the external PR also still lacks real nginx/browser proof.

## What This Changes

The PR adds gatewayUrlAutoDerived metadata, load-time gateway URL resolution, page-derived scoped settings writes, and focused Control UI storage test updates.

## Best Possible Solution

Resolve legacy fallback during loadSettings so the first upgraded sibling base-path visit selects the page-derived gateway while preserving explicit same-host custom gateway paths, then add regression coverage and real reverse-proxy proof.

## Reproduction Assessment

Yes. By source inspection, seed the shared legacy settings key with gatewayUrl "wss://gateway.example/a", set the page under /b, and call loadSettings; current main and PR head both preserve /a on the first upgraded load.

## Solution Assessment

No. The flag and page-scoped-key direction is useful, but returning stored for an undefined derivation flag defers the fix until after a later save and leaves the first upgraded load broken.

## Review Findings
