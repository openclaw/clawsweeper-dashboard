## Summary

Keep this PR open: it targets a real Control UI base-path persistence bug, but the current PR head still preserves an unflagged legacy gateway URL on the first upgraded load and the external contributor has not supplied real nginx/browser behavior proof.

## What This Changes

The PR adds gateway URL derivation metadata, load-time gateway URL resolution, page-derived scoped localStorage writes, and Control UI storage test updates for path-prefixed reverse-proxy instances.

## Best Possible Solution

Resolve legacy fallback during loadSettings so the first upgraded visit selects the page-derived gateway for sibling base paths while preserving explicit same-host custom paths, then add regression coverage and real reverse-proxy proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed the shared legacy settings key with gatewayUrl "wss://gateway.example/a", set the page under /b, and call loadSettings; current main and PR head both preserve /a on the first upgraded load.

## Solution Assessment

No. The flag and page-scoped-key direction is useful, but the undefined-flag branch returns the stored legacy URL, so the first upgraded load still connects to the wrong backend.
