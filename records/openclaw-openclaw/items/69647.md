## Summary

Keep open: the reported Control UI bug is valid, but the latest PR head still preserves unflagged legacy gateway URLs on the first upgraded load and lacks real behavior proof.

## What This Changes

The PR adds persisted gateway URL derivation metadata, load-time gateway URL resolution, page-derived scoped localStorage writes, and Control UI storage test updates for path-prefixed reverse-proxy instances.

## Best Possible Solution

Resolve legacy fallback during loadSettings so the first upgraded visit selects the page-derived gateway for sibling base paths while preserving explicit custom gateway paths, then add focused regression coverage and redacted real reverse-proxy proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed the shared legacy settings key with gatewayUrl `wss://gateway.example/a`, set the page under `/b`, and call `loadSettings`; current main and PR head both preserve `/a` on the first upgraded load.

## Solution Assessment

No. The metadata and page-scoped-key direction is useful, but the undefined-flag branch returns the stored legacy URL, so the first upgraded load can still connect to the wrong backend.
