## Summary

Keep open. Current main still has the sibling-path legacy settings fallback, so this PR is not obsolete; however, the PR head still preserves unflagged legacy gateway URLs on first upgraded load and needs reconciliation with newer Control UI storage fields before it is merge-ready.

## What This Changes

The PR adds a `gatewayUrlAutoDerived` persisted flag, gateway URL resolution helper, page-derived scoped settings writes, and storage test updates for path-prefixed reverse-proxy Control UI deployments.

## Best Possible Solution

Keep this PR open as the active implementation candidate, but update it before merge. The final fix should live in `ui/src/ui/storage.ts` with focused coverage in `ui/src/ui/storage.node.test.ts`, distinguish page-derived defaults from explicit gateway overrides, handle or explicitly document unflagged legacy records, preserve same-host custom gateway paths and cross-host overrides, retain current browser-local storage fields, and include a changelog entry for the user-visible Control UI fix.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: seed the shared legacy settings key with a `/a` gateway URL, set the page location to sibling path `/b`, and call `loadSettings()`; current main and the PR head's undefined-flag path both preserve the stored `/a` URL instead of selecting the page-derived `/b` URL.

## Solution Assessment

No. The flag-and-page-key approach is directionally useful, but this PR head is not yet the best fix because it preserves unflagged legacy records on first upgraded load and needs current-main storage reconciliation; the safer solution is a targeted migration/read-time resolution that does not discard explicit user overrides.
