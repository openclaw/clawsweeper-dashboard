## Summary

Keep open. Current `main` still lacks a writer-level no-op return after `writeConfigFile` computes the new and previous hashes, and an open closing PR already targets the issue, so this should stay open until that PR or an equivalent fix lands.

## What This Changes

Request a `writeConfigFile` no-op short-circuit that skips disk I/O, overwrite/audit noise, and file-watcher reload cascades when a config write is not a real change.

## Best Possible Solution

Land or replace PR #75543 with a narrow writer-boundary guard that skips disk I/O for true stable no-ops while preserving the existing validation, destructive-write protection, audit, backup, runtime snapshot, and reload behavior for real config changes.

## Reproduction Assessment

Yes. A focused regression can write a config, call `writeConfigFile` again with an equivalent stable payload or volatile-meta-only change, and assert that current `main` still reaches temp-write, backup, overwrite-log, audit, or mtime-changing side effects instead of returning early.

## Solution Assessment

Yes for keeping this open behind the linked PR. The maintainable fix belongs at the config writer boundary; a pure `config.patch` guard is too narrow, and a hash-only guard may not cover the reported `lastTouchedAt` semantic no-op case.
