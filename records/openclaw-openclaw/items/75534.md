## Summary

Keep open. Current main still has the low-level writer gap: `writeConfigFile` computes `nextHash` and `previousHash`, but then continues into stat, anomaly/audit setup, temp-file write, backup rotation, rename/copy fallback, overwrite logging, and audit append without an equal-hash short-circuit. Higher-level `config.patch` and the gateway reloader mitigate some restart cases, but they do not remove unnecessary byte-identical writer side effects. The prior related PR was closed unmerged, so there is no canonical merged fix to close this against.

## What This Changes

Add a no-op short-circuit in `writeConfigFile` so byte-identical persisted config output returns before disk, audit, log, backup, and watcher-triggering side effects.

## Best Possible Solution

Add a writer-level no-op guard after validation, env-ref restoration, stamping, and final persisted JSON/hash calculation, but before stat/audit/backup/temp-write work. Cover both sides with tests: byte-identical writer calls should not touch the file, backups, audit, overwrite logs, or write notifications, while hash-different writes should keep the existing safety, logging, audit, and reload-notification behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence focused reproduction: arrange an existing valid config whose snapshot hash matches the final serialized writer output, call `createConfigIO().writeConfigFile`, and current main will still proceed past hash calculation into temp write/rename/audit work. A broader production-style semantic no-op fixture should account for `stampConfigVersion()` updating `meta.lastTouchedAt`.

## Solution Assessment

Mostly yes. A writer-boundary guard is the narrowest maintainable place to stop byte-identical rewrite side effects, but the implementation should explicitly decide whether exact hash equality is sufficient or whether the reported semantic no-op churn requires a normalized comparison that ignores only intentional volatile metadata.
