## Summary

Keep open. Current main still lacks a writer-level no-op branch, and the provided context shows open PR #75543 with closing syntax for this exact config writer behavior, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Add a no-op short-circuit in `writeConfigFile` so unchanged or semantically unchanged config updates do not touch disk and trigger gateway reload cascades.

## Best Possible Solution

Land or replace PR #75543 with a narrow writer-boundary guard that skips disk I/O for true stable no-ops while preserving audit, backup, suspicious-write rejection, and reload behavior for real config changes.

## Reproduction Assessment

Yes. A focused repo reproduction can write a config, call `writeConfigFile` again with an equivalent stable payload, and assert that current main still reaches backup/temp-write/log/audit side effects instead of returning before disk I/O.

## Solution Assessment

Yes for keeping the issue open behind the linked PR. The best code fix is a narrow `writeConfigFile` short-circuit, preferably covering both byte-identical writes and the volatile-meta-only semantic no-op described in the report.
