## Summary

Keep open. Current main still lacks a writer-level no-op short-circuit in `writeConfigFile`, and the linked open PR #75543 already targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent replacement lands.

## What This Changes

Request to make `writeConfigFile` skip disk I/O, backup rotation, audit logging, overwrite logging, and watcher-triggering side effects when the next config write is a no-op or only semantically volatile metadata changed.

## Best Possible Solution

Land or replace PR #75543 with a narrow writer-boundary guard that suppresses true stable no-op disk writes while preserving validation, destructive-write protection, audit/backup behavior for real changes, runtime snapshot consistency, and reload notifications when the file actually changes.

## Reproduction Assessment

Yes. A focused repo reproduction can write a config, call `writeConfigFile` again with an equivalent stable payload or volatile-meta-only output, and assert that current main still reaches temp-write, backup, logging, audit, or mtime-changing side effects instead of returning before disk I/O.

## Solution Assessment

Yes. Solving this at the writer boundary is the narrowest maintainable direction because it covers all callers; the existing `config.patch` no-op and reloader hash dedupe are useful but do not prevent generic writer side effects.
