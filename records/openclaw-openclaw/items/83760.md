- **[P1] Keep provider capabilities buildable:** `extensions/comfy/image-generation-provider.ts:31-32`
  - body: These fields call `isComfyDimensionsConfigured(cfg)`, but `cfg` is not in scope when `buildComfyImageGenerationProvider()` constructs the static provider object, so the plugin will fail TypeScript before the feature can run. The capability surface also needs to match the existing registration contract or an approved config-aware pattern.
  - confidence: 0.98
- **[P1] Narrow dimensions config before injection:** `extensions/comfy/workflow-runtime.ts:750-753`
  - body: `capabilityConfig` is a `Record<string, unknown>`, so `capabilityConfig.dimensions` is not a `ComfyDimensionConfig`; strict typecheck should reject both the `baseConfig` assignment and the injection call. Parse or validate the dimensions object before using its node IDs.
  - confidence: 0.95
- **[P1] Read dimensions from the configured image section:** `extensions/comfy/workflow-runtime.ts:712-715`
  - body: The schema and examples add `image.dimensions`, but this helper checks `config.widthNodeId` and `config.heightNodeId` at the plugin root. With the documented config shape, Comfy would keep `supportsSize` and `supportsAspectRatio` false and normalization would drop the requested geometry before `generateImage()` sees it.
  - confidence: 0.93
- **[P2] Document dimensions as nested fields:** `docs/providers/comfy.md:236-240`
  - body: The examples and manifest put these keys under `dimensions`, but the table lists `widthNodeId`, `heightNodeId`, and related fields as direct image/video keys. Users following the table would write config that the schema rejects or the runtime ignores.
  - confidence: 0.91
