## Summary

Keep open. Current main makes memory_search failures non-fatal, but it still allows auto memory embeddings to select the OpenAI adapter and OpenAI-scoped credentials independently of a DeepSeek chat model.

## What This Changes

The issue asks memory-core to avoid using OpenAI embedding credentials when the active agent model is DeepSeek, after a v2026.4.25 regression caused embedding 401s that blocked replies.

## Best Possible Solution

Keep the non-fatal memory_search behavior, then make embedding selection deterministic through clearer memorySearch.provider/remote diagnostics or a DeepSeek-owned embedding adapter if maintainers choose that contract.

## Reproduction Assessment

Yes for the source-level path: with memorySearch.provider unset or auto and a resolvable OpenAI key, current main can select OpenAI memory embeddings without consulting the DeepSeek chat model. The exact 401 needs the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Automatically following the active chat provider is not clearly the best fix because memory embeddings are a separate provider-owned adapter contract; explicit pinning diagnostics or a DeepSeek embedding adapter is safer to evaluate first.
