## Summary

Keep open: current main and the latest release still default unset or legacy-auto memory embeddings to OpenAI, while DeepSeek remains a chat provider without a memory embedding adapter. The issue is still necessary because the remaining fix is a maintainer product/auth decision, not an already-implemented cleanup.

## What This Changes

The issue asks memory-core to avoid using OpenAI embedding credentials for non-OpenAI agent setups and instead use the configured provider or clearly disclose the separate embedding credential requirement.

## Best Possible Solution

Keep this open for a maintainer-owned provider/auth decision: either make non-OpenAI onboarding and doctor/status clearly disclose the separate memory embedding requirement, or route memory embeddings only through providers with explicit embedding-provider support.

## Reproduction Assessment

Yes at source level: unset or legacy-auto memorySearch still resolves to OpenAI, the OpenAI adapter uses OpenAI auth, and DeepSeek does not declare a memory embedding provider. I did not establish a live current-main DeepSeek credential reproduction, so the original total chat-blocking impact remains unproven.

## Solution Assessment

Unclear as an implementation direction because the source supports the complaint but current docs and tests also define OpenAI as the memory default and an explicit memorySearch provider path exists. The best solution needs maintainer choice on automatic chat-provider reuse versus explicit setup, doctor, and onboarding disclosure.
