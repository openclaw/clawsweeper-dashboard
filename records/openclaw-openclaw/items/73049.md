## Summary

Keep open: current main and v2026.6.10 still default unset or legacy-auto memory embeddings to OpenAI, DeepSeek registers chat/provider auth but no memory embedding adapter, and the related open items cover adjacent OAuth, onboarding, or custom-baseURL cases rather than this full non-OpenAI memory/auth contract.

## What This Changes

The issue requests fixing a regression where memory-core uses OpenAI embedding credentials in a DeepSeek chat setup, blocking or degrading agent message processing.

## Best Possible Solution

Keep this issue open for a maintainer-owned memory embedding contract: prefer explicit memorySearch provider setup and doctor/onboarding disclosure, and add provider-owned embedding adapters only where the provider declares embedding capability.

## Reproduction Assessment

Yes at source level: current main still defaults unset or legacy-auto memorySearch to OpenAI, the OpenAI adapter resolves OpenAI credentials, and DeepSeek does not declare an embedding provider. I did not run a live DeepSeek or QQ bot repro, and current tests/source do not prove the original total message-blocking impact still occurs.

## Solution Assessment

Unclear as an implementation direction: automatically reusing chat-provider credentials is not the best fix unless the provider advertises embedding support. The maintainable path is a maintainer decision around explicit embedding-provider setup, doctor/onboarding disclosure, and provider-owned adapters for embedding-capable providers.
