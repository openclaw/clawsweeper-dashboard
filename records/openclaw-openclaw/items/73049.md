## Summary

Keep open. Current `main` mitigates the original agent-blocking symptom by making memory search/bootstrap failures non-fatal, but it still does not implement the requested credential-routing behavior: memory-core auto-selects from registered memory embedding adapters and OpenAI-scoped auth, while the bundled DeepSeek plugin only registers a model provider and no memory embedding adapter.

## What This Changes

The issue asks memory-core to avoid selecting OpenAI-scoped embedding credentials when the active agent model is DeepSeek, because the `v2026.4.25` upgrade caused embedding 401s and blocked message processing.

## Best Possible Solution

Keep memory failures non-fatal, then make memory embedding provider selection deterministic and documented. The durable fix should either require explicit `memorySearch.provider` configuration with doctor guidance for mismatched credentials, or add a DeepSeek/OpenAI-compatible memory embedding adapter only if maintainers confirm that endpoint and credential contract is supported.

## Reproduction Assessment

Yes for the credential-selection path: current source shows `provider: "auto"` chooses memory embedding adapters and OpenAI-scoped auth without consulting the active DeepSeek chat provider. Reproducing the exact 401 would need the reporter's invalid key/proxy setup, but the remaining behavior is source-verifiable.

## Solution Assessment

Unclear. The non-fatal error handling is the right narrow mitigation for agent replies, but it is not the full fix for deterministic embedding credential routing; the safer next step is maintainer policy on supported memory embedding providers and explicit configuration behavior.
