## Summary

Keep open. Current main still allows `memorySearch.provider` auto-selection to choose an OpenAI embedding adapter and OpenAI-scoped credentials independently of a DeepSeek chat model; the tool failure path is now non-fatal, but the credential-selection behavior remains unresolved and security-sensitive.

## What This Changes

The issue asks memory-core to avoid selecting OpenAI embedding credentials when the agent chat model is DeepSeek and to stop memory sync failures from blocking message processing.

## Best Possible Solution

Keep `memory_search` failures non-fatal, then make embedding credential selection deterministic through explicit `memorySearch` provider diagnostics/config or a provider-owned DeepSeek embedding adapter before changing credential reuse semantics.

## Reproduction Assessment

Yes for the source-level path: with `memorySearch.provider` unset or `auto` and a resolvable OpenAI key, current main can select OpenAI memory embeddings without consulting the DeepSeek chat model. The exact 401 requires the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Automatically reusing the active chat provider is not proven to be the best fix because memory embeddings are a separate provider adapter contract; explicit memorySearch pinning diagnostics or a DeepSeek-owned embedding adapter is safer to evaluate first.
