## Summary

Keep open. Current main still lets auto memory embeddings select an OpenAI-scoped adapter/key independently of a DeepSeek chat model; the empty-index bootstrap path is non-fatal now, but the provider credential contract and runtime embedding-failure behavior remain unresolved and security-sensitive.

## What This Changes

The issue reports a v2026.4.25 regression where memory-core selects OpenAI embedding credentials while the agent chat model uses DeepSeek, causing embedding 401s and blocked message processing.

## Best Possible Solution

Keep memory failures from blocking agent replies, then make embedding provider selection explicit and credential-safe; if DeepSeek embeddings are intended, add a DeepSeek-owned adapter instead of implicitly reusing chat credentials.

## Reproduction Assessment

Yes for a source-level path: with `memorySearch.provider` unset/auto and a resolvable OpenAI key, current main can select OpenAI embeddings without consulting a DeepSeek chat model. The exact upstream 401 requires the reporter's invalid OpenAI/proxy credential setup.

## Solution Assessment

Unclear. Automatically following the active chat provider is not clearly safest because memory embeddings use a separate provider-owned adapter and credential contract; explicit pinning diagnostics or a DeepSeek embedding adapter needs maintainer direction.
