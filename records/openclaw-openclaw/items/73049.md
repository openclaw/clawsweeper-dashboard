## Summary

Keep open. Current main mitigates the fatal reply-path failure by converting memory search/setup failures into warnings or unavailable results, but the reported credential-routing behavior remains because automatic memory embeddings can still select the OpenAI adapter and OpenAI-scoped credentials independently of a DeepSeek chat model.

## What This Changes

The issue asks memory-core to stop selecting OpenAI embedding credentials when the active agent model is DeepSeek-backed.

## Best Possible Solution

Keep the nonfatal failure handling, then make the credential contract explicit: either improve diagnostics/docs around pinned memorySearch.provider and remote credentials, or add a DeepSeek-owned embedding adapter if maintainers confirm DeepSeek should satisfy the memory embedding contract.

## Reproduction Assessment

Yes for the source-level path: with memorySearch.provider unset and a resolvable OpenAI key, auto mode can select the OpenAI memory embedding adapter without consulting the DeepSeek chat model. The exact 401 needs the reporter's external proxy/key setup.

## Solution Assessment

Unclear. Making memory embeddings automatically follow the active chat provider is not clearly the best fix because embeddings use a separate adapter contract; explicit config/doctor guidance or a provider-owned DeepSeek adapter is safer pending maintainer direction.
