## Summary

Keep open. Current main prevents memory search failures from taking down the tool path, but the reported credential-routing behavior remains: auto memory embeddings can still select the OpenAI adapter and resolve OpenAI-scoped credentials independently of a DeepSeek chat model.

## What This Changes

The issue requests deterministic memory-core embedding credential routing so a DeepSeek-backed agent does not auto-use an unrelated OpenAI API key for memory embeddings.

## Best Possible Solution

Keep the non-fatal memory failure behavior, then make embedding credential selection deterministic and documented. The maintainable end state should either guide operators to explicit memorySearch.provider/remote settings with doctor diagnostics for mismatched keys, or add a DeepSeek-owned embedding adapter if maintainers confirm DeepSeek should satisfy the memory embedding contract.

## Reproduction Assessment

Yes for the source-level credential-selection path: with memorySearch.provider unset or auto and a resolvable OpenAI key, current main can auto-select the OpenAI memory embedding adapter without consulting a DeepSeek chat model. The exact 401 requires the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Automatically making embeddings follow the active chat provider is not obviously the best fix because memory embeddings are a separate adapter contract; the safer path is for maintainers to choose between explicit config/doctor guidance and a provider-owned DeepSeek embedding adapter.
