## Summary

Keep open. Current main mitigates the worst agent-blocking symptom by converting memory search failures into warnings or unavailable results, but it still does not implement the reported credential-routing behavior: automatic memory embeddings can select the OpenAI adapter and resolve OpenAI-scoped credentials without consulting the active DeepSeek chat provider.

## What This Changes

The issue requests deterministic memory-core embedding credential routing so a DeepSeek-backed agent does not auto-use an unrelated OpenAI API key and block message handling.

## Best Possible Solution

Keep the non-fatal memory-search behavior, then make embedding credential selection deterministic and documented. The maintainable end state is either explicit memorySearch.provider/remote configuration with doctor guidance for mismatched OpenAI keys, or a DeepSeek-owned memory embedding adapter if maintainers confirm DeepSeek should support the embedding contract.

## Reproduction Assessment

Yes for the credential-selection path: with memorySearch.provider unset/auto and a resolvable OpenAI key, current source can auto-select the OpenAI memory embedding adapter and resolve OpenAI credentials regardless of a DeepSeek chat model. The exact 401 requires the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Making embeddings automatically follow the active chat provider is not obviously the best fix because memory embeddings use an explicit adapter contract; the safer path is for maintainers to choose between explicit config/doctor guidance and a provider-owned DeepSeek embedding adapter.
