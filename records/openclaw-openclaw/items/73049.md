## Summary

Keep open. Current main still leaves the reported credential-routing behavior unresolved: auto memory embeddings can select the OpenAI adapter and resolve OpenAI-scoped credentials without consulting the active DeepSeek chat provider. The worst message-blocking symptom is mitigated by non-fatal memory-search handling, but that is not the requested fix.

## What This Changes

The issue asks memory-core to stop using OpenAI embedding credentials for a DeepSeek-backed agent and to avoid breaking agent message processing after the v2026.4.25 upgrade.

## Best Possible Solution

Keep the non-fatal memory-search behavior, then make embedding credential selection deterministic and documented. The maintainable end state is either explicit memorySearch.provider/remote configuration with doctor guidance for mismatched OpenAI keys, or a DeepSeek-owned embedding adapter if maintainers confirm DeepSeek should support the memory embedding contract.

## Reproduction Assessment

Yes for the credential-selection path: with memorySearch.provider unset/auto and an available OpenAI credential, current main can auto-select the OpenAI embedding adapter and resolve provider "openai" credentials regardless of a DeepSeek chat model. The exact 401 requires the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Automatically making embeddings follow the active chat provider is not obviously the best fix because memory embeddings use an explicit adapter contract and DeepSeek does not currently declare one; explicit config/doctor guidance or a provider-owned DeepSeek adapter is safer.
