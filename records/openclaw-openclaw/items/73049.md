## Summary

Keep open. Current main mitigates the worst reported symptom by turning memory-search embedding failures into structured unavailable results instead of throwing through the tool path, but it still does not implement the requested DeepSeek credential routing: automatic memory embeddings select registered embedding adapters such as OpenAI and resolve that adapter's credentials, while the DeepSeek plugin only declares chat/model auth and no memory embedding adapter.

## What This Changes

The issue requests fixing a v2026.4.25 memory-core regression where a DeepSeek-backed agent's memory embeddings resolve OpenAI credentials and can block message processing with embedding 401 errors.

## Best Possible Solution

Keep the non-fatal memory-search behavior, then make embedding credential selection deterministic and documented. The maintainable end state is either explicit memorySearch.provider/remote configuration with doctor guidance for mismatched OpenAI keys, or a DeepSeek-owned memory embedding adapter if maintainers confirm DeepSeek should support the embedding contract.

## Reproduction Assessment

Yes for the source-level credential path: current main shows auto memory embeddings can select the OpenAI adapter and resolve OpenAI credentials without consulting the active DeepSeek chat provider. The exact 401 needs the reporter's invalid OpenAI key/proxy setup, but the routing that produces it is visible in source.

## Solution Assessment

Unclear. The current non-fatal mitigation is the right narrow protection for agent replies, but it is not the complete credential-routing fix; maintainers should choose between explicit memorySearch configuration guidance and adding a provider-owned DeepSeek/OpenAI-compatible embedding adapter.
