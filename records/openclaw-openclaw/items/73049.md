## Summary

Keep open: current main and the latest inspected release still default unset or legacy-auto memory embeddings to OpenAI, while DeepSeek remains a chat provider without a memory embedding adapter. Existing explicit memorySearch provider config and related follow-up issues help, but they do not settle the requested automatic provider/auth routing or onboarding disclosure policy.

## What This Changes

The issue reports that memory-core uses OpenAI embedding credentials instead of the selected non-OpenAI chat provider credentials after upgrade.

## Best Possible Solution

Keep this issue open for a maintainer-owned provider/auth decision: either make non-OpenAI onboarding and doctor/status clearly disclose the separate memory embedding requirement, or add a narrow tested routing rule only for providers with explicit embedding-provider support.

## Reproduction Assessment

Yes at source level: unset or legacy-auto memorySearch still resolves to OpenAI, OpenAI embeddings require OpenAI credentials, and DeepSeek does not declare a memory embedding provider. I did not perform a live DeepSeek credential run, and the original total chat-blocking impact is not proven on current main.

## Solution Assessment

Unclear as an implementation direction because the source supports the complaint but also shows an explicit provider configuration path already exists. The best solution needs maintainer choice on whether automatic chat-provider reuse is intended or whether setup/doctor disclosure is the supported contract.
