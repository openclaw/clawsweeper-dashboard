## Summary

Keep open. Current main mitigates the original agent-blocking symptom by turning memory search/bootstrap failures into logged or structured unavailable results, but it still does not satisfy the credential-routing request: memory embedding selection is based on memory embedding adapters and OpenAI-scoped credentials, while the DeepSeek plugin does not declare a memory embedding adapter.

## What This Changes

The issue asks memory-core to avoid selecting OpenAI-scoped embedding credentials when the active agent model is DeepSeek, because the v2026.4.25 upgrade caused embedding 401s and blocked message processing.

## Best Possible Solution

Keep the non-fatal memory-search behavior, then make embedding credential selection deterministic and documented. The maintainable end state is either explicit `memorySearch.provider` plus doctor guidance for mismatched credentials, or a supported DeepSeek/OpenAI-compatible memory embedding adapter if maintainers confirm that endpoint and credential contract is valid.

## Reproduction Assessment

Yes for the credential-selection path: current source shows `provider: "auto"` chooses memory embedding adapters and OpenAI-scoped auth without consulting the active DeepSeek chat provider. The exact live 401 would require the reporter's invalid key/proxy setup.

## Solution Assessment

Unclear. The current non-fatal mitigation is the right narrow protection for agent replies, but it is not the complete fix for deterministic embedding credential routing; a maintainer should decide the supported provider and configuration policy first.
