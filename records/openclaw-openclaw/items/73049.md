## Summary

Keep open. Current main still lets `memorySearch.provider: "auto"` select an embedding adapter independently from the active DeepSeek chat provider, so OpenAI embeddings can be selected from `OPENAI_API_KEY` while DeepSeek has no memory embedding contract in its plugin manifest. The search-bootstrap sync path is non-fatal now, but the credential-selection contract remains unresolved and security-sensitive.

## What This Changes

The issue asks memory-core to stop selecting OpenAI embedding credentials for a DeepSeek-backed agent after v2026.4.25 and to avoid blocking message processing.

## Best Possible Solution

Make memory embedding selection deterministic without crossing credentials: either keep users on explicit `memorySearch.provider` diagnostics/pinning or add a provider-owned DeepSeek embedding adapter if maintainers decide that contract is supported, while keeping memory failures non-fatal to chat turns.

## Reproduction Assessment

Yes for the source-level path: with `memorySearch.provider` unset and a resolvable OpenAI key, current main can select the OpenAI embedding adapter without consulting the DeepSeek chat model. The exact 401 requires the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Automatically following the active chat provider is not clearly the narrowest fix because memory embeddings use separate provider-owned adapters and credential scopes; explicit diagnostics or a DeepSeek adapter decision is safer first.
