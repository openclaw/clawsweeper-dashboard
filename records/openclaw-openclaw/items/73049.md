## Summary

Keep open. Current main still permits `memorySearch.provider: "auto"` to select the OpenAI memory embedding adapter and OpenAI-scoped auth independently of a DeepSeek chat model, while DeepSeek has no memory-embedding provider contract; the remaining fix needs product/security judgment rather than cleanup closure.

## What This Changes

The issue asks memory-core to stop using OpenAI embedding credentials for a DeepSeek-backed agent after the v2026.4.25 regression.

## Best Possible Solution

Keep memory failures non-fatal, then define the embedding-provider contract explicitly through diagnostics, explicit provider pinning, or a provider-owned DeepSeek embedding adapter if maintainers choose to support it.

## Reproduction Assessment

Yes, for the source-level path: with memorySearch.provider unset/auto and a resolvable OpenAI key, current main can select OpenAI memory embeddings without consulting the DeepSeek chat model. The exact 401 needs the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Reusing the active chat provider's credentials for embeddings is not obviously the safest fix because memory embeddings are provider-adapter contracts; explicit diagnostics or a DeepSeek-owned adapter needs maintainer judgment.
