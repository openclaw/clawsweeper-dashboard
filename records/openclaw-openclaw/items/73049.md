## Summary

Keep open. Current main and the latest release still allow memory embedding auto-selection to choose OpenAI credentials independently of a DeepSeek chat model; current code has better unavailable-payload handling for memory_search failures, but the central provider credential-routing question remains unresolved.

## What This Changes

The issue asks memory-core to stop selecting OpenAI embedding credentials for a DeepSeek agent and to avoid blocking message processing after embedding auth failures.

## Best Possible Solution

Define memory embedding provider ownership explicitly, keep auth provider-scoped, make embedding failures non-fatal to chat, and add a DeepSeek-owned embedding adapter/docs only if DeepSeek supports that API contract.

## Reproduction Assessment

Yes for the source-level path: with memorySearch.provider unset or auto, current main can select OpenAI memory embeddings without consulting a DeepSeek chat model. The exact 401 still requires the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Reusing the chat provider's credentials for embeddings is not necessarily the safest fix; maintainers should choose between explicit provider pinning/diagnostics, non-fatal fallback behavior, or a provider-owned DeepSeek embedding adapter.
