## Summary

Keep open. Current main mitigates the fatal search-bootstrap path, but the reported credential-routing behavior remains: automatic memory embeddings can still select the OpenAI adapter and resolve OpenAI-scoped credentials independently of a DeepSeek chat model.

## What This Changes

The issue reports a regression where memory-core selects OpenAI-scoped embedding credentials even when the agent chat model is DeepSeek, causing embedding 401s and message processing failure.

## Best Possible Solution

Keep the non-fatal sync behavior, then make embedding selection deterministic and documented: either guide operators to explicit memorySearch.provider/remote settings with doctor diagnostics, or add a DeepSeek-owned embedding adapter if maintainers confirm DeepSeek should satisfy the memory embedding contract.

## Reproduction Assessment

Yes for the source-level credential-selection path: with memorySearch.provider unset or auto and a resolvable OpenAI key, current main can auto-select the OpenAI memory embedding adapter without consulting a DeepSeek chat model. The exact 401 needs the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Making embeddings automatically follow the active chat provider is not clearly the safest fix because memory embeddings are a separate adapter contract; maintainer judgment is needed between explicit config/doctor guidance and a provider-owned DeepSeek embedding adapter.
