- `impact:auth-provider`: The reported failure is about embedding provider selection and API-key resolution using OpenAI credentials for a non-OpenAI chat setup.

## Summary

Keep open: current main and the latest release still default unset or legacy-auto memory embeddings to OpenAI, so non-OpenAI setups can still hit OpenAI credential failures. Current source suggests memory now degrades instead of fully blocking chat, and https://github.com/openclaw/openclaw/issues/16670 covers onboarding disclosure, but the provider-routing policy behind this issue is still unresolved.

## What This Changes

The issue requests a fix for a v2026.4.25 memory-core embedding-auth regression where non-OpenAI chat providers, especially DeepSeek, end up using OpenAI embedding credentials.

## Best Possible Solution

Keep this open for a maintainer decision on memory embedding provider policy: either require explicit memorySearch configuration with clear onboarding/doctor disclosure, or define a supported auto-routing rule for compatible non-OpenAI providers.

## Reproduction Assessment

Yes, at source level for the default OpenAI embedding-auth path; I did not live-reproduce the original full chat block on current main. The later QA comment gives current real-environment evidence that non-OpenAI onboarding still produces memory auth failures or degradation.

## Solution Assessment

Unclear. Reusing DeepSeek chat credentials is not currently documented and the DeepSeek plugin has no memory embedding adapter, so the narrowest maintainable fix needs maintainer policy on explicit memory config versus supported provider auto-routing.
