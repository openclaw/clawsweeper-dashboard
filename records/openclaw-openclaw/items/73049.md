## Label Justifications

- `P2`: The issue affects semantic memory for non-OpenAI setups, but current evidence points to degraded memory/search rather than a current whole-runtime outage.
- `impact:session-state`: Broken embeddings reduce semantic memory recall, which affects retrieval of memory and session context.
- `impact:auth-provider`: The central failure is provider credential selection for memory embeddings in a non-OpenAI chat setup.

## Summary

Keep open: current main and v2026.6.8 still default unset or legacy-auto memory embeddings to OpenAI, while DeepSeek remains a chat provider without a memory embedding adapter. The merged OpenAI-compatible provider work gives users an explicit configuration path, but it does not settle the requested automatic provider/auth routing or onboarding disclosure policy.

## What This Changes

The issue asks memory-core to stop using OpenAI embedding credentials for non-OpenAI agent setups and instead use the configured provider credentials or clearly disclose the separate embedding credential requirement.

## Best Possible Solution

Maintain the explicit memory embedding provider path already on main, then have maintainers choose whether non-OpenAI onboarding should warn about OpenAI defaults or whether a narrow, tested auto-routing rule should be added for providers with embedding-capable adapters.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: unset or legacy-auto memorySearch config still selects OpenAI, and DeepSeek does not register a memory embedding provider. I did not perform a live DeepSeek credential run in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the product contract. Explicit memorySearch provider config is already supported; automatic chat-provider reuse would be a new provider/auth behavior that needs policy and regression proof.
