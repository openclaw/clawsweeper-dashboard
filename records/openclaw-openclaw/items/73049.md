## Summary

Keep open. Current main still lets auto memory embeddings select an OpenAI-scoped provider/key independently of a DeepSeek chat model; the exact blocking behavior is partly mitigated by non-fatal search-bootstrap sync handling, but the credential-selection contract remains unresolved and security-sensitive.

## What This Changes

The issue asks memory-core to stop selecting OpenAI embedding credentials for a DeepSeek-backed agent after the v2026.4.25 upgrade.

## Best Possible Solution

Keep memory failures non-fatal, then make embedding selection deterministic through explicit memorySearch.provider diagnostics or a provider-owned DeepSeek embedding adapter if maintainers choose that contract.

## Reproduction Assessment

Yes for the source-level path: with memorySearch.provider unset/auto and a resolvable OpenAI key, current main can select OpenAI memory embeddings without consulting the DeepSeek chat model. The exact 401 requires the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Automatically following the active chat provider is not clearly the best fix because memory embeddings use a separate provider-owned adapter contract; explicit pinning diagnostics or a DeepSeek embedding adapter is safer to decide first.
