## Summary

Keep open. Current `main` still has the reported credential-routing shape: memory embeddings default to `auto`, auto-selection can choose the OpenAI embedding adapter, and that adapter resolves OpenAI-scoped credentials independently of a DeepSeek chat model. The memory-search tool now catches provider/search failures and returns an unavailable result, which mitigates message-processing breakage, but the underlying wrong-provider/key selection complaint remains unresolved.

## What This Changes

The issue requests that `memory-core` embeddings stop using an OpenAI API key when the active agent model is DeepSeek, after a v2026.4.25 upgrade caused OpenAI 401 errors during memory sync.

## Best Possible Solution

Keep the non-fatal memory-search failure handling, then make embedding credential selection deterministic and documented. The maintainable end state should either steer operators to explicit `memorySearch.provider` or `memorySearch.remote` settings with doctor diagnostics for mismatched keys, or add a DeepSeek-owned embedding adapter if maintainers decide DeepSeek should satisfy the memory embedding contract.

## Reproduction Assessment

Yes for the source-level routing path: with memory search left on `auto` and an OpenAI key resolvable, current `main` can select the OpenAI embedding adapter without consulting a DeepSeek chat model. The exact 401 needs the reporter's external OpenAI/proxy credential setup.

## Solution Assessment

Unclear. Automatically making embeddings follow the active chat provider is not obviously the safest fix because memory embeddings are a separate adapter contract; explicit config/doctor guidance or a provider-owned DeepSeek embedding adapter is a safer maintainer decision point.
