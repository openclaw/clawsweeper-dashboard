## Summary

Keep open. Current main mitigates the fatal memory_search failure path, but the reported credential-selection behavior remains: memory embeddings in auto mode can select the OpenAI adapter and resolve OpenAI-scoped credentials independently of a DeepSeek chat model.

## What This Changes

The issue asks memory-core to avoid choosing OpenAI-scoped embedding credentials when a DeepSeek agent model is active, because the 2026.4.25 upgrade produced OpenAI 401s during memory sync.

## Best Possible Solution

Keep the non-fatal error handling, then make embedding selection deterministic and documented: either improve explicit memorySearch.provider/remote diagnostics or add a DeepSeek-owned embedding adapter if maintainers choose that contract.

## Reproduction Assessment

Yes for the source-level path: with memorySearch.provider unset or auto and a resolvable OpenAI key, current main can select the OpenAI memory embedding adapter without consulting the DeepSeek chat model. The exact 401 requires the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Making embeddings automatically follow the active chat provider is not clearly the narrowest fix because memory embeddings are a separate adapter contract; explicit config/doctor guidance or a DeepSeek-owned adapter is safer to evaluate first.
