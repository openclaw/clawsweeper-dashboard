## Summary

Keep open. Current main mitigates the hard failure path by returning unavailable memory results instead of throwing out of the tool, but the reported credential-selection behavior remains: automatic memory embeddings are selected independently of the active DeepSeek chat provider and can still use OpenAI-scoped credentials when an OpenAI key is resolvable.

## What This Changes

The report asks for memory-core embedding credential selection to stop using OpenAI credentials when the active agent model is DeepSeek, after a 2026.4.25 regression caused embedding 401s and blocked message processing.

## Best Possible Solution

Keep the non-fatal memory-search failure handling, then make embedding provider selection deterministic and documented. The maintainable end state is either stronger explicit memorySearch provider diagnostics and doctor guidance for mismatched keys, or a DeepSeek-owned memory embedding adapter if maintainers confirm DeepSeek should satisfy the embedding contract.

## Reproduction Assessment

Yes for the source-level reproduction path: with memorySearch.provider unset or auto and an OpenAI key resolvable, current main can select the OpenAI embedding adapter without consulting a DeepSeek chat model. The exact 401 requires the reporter's external credential/proxy setup.

## Solution Assessment

Unclear. The requested behavior of using DeepSeek chat credentials for embeddings is not obviously the best fix because memory embeddings are a separate provider contract and DeepSeek does not currently declare an embedding provider; explicit config/doctor guidance or a DeepSeek embedding adapter is safer than silently reusing chat auth.
