## Summary

Keep open: current main and v2026.6.8 still default unset or legacy-auto memory embeddings to OpenAI, and the DeepSeek plugin does not register a memory embedding adapter or automatic embedding-auth route. Current source now appears to degrade memory/search rather than fully blocking chat, while the remaining provider-routing and onboarding-disclosure policy still needs maintainer direction.

## What This Changes

The issue reports a v2026.4.25 regression where memory-core uses OpenAI embedding credentials instead of the configured non-OpenAI agent provider, breaking or degrading memory for DeepSeek-style setups.

## Best Possible Solution

Keep this open until maintainers choose the memory embedding policy: either require explicit memorySearch setup with visible onboarding/doctor/status disclosure, or define a supported auto-routing rule only for providers with embedding adapters and regression proof.

## Reproduction Assessment

Yes, at source level for the remaining issue: unset or legacy-auto memory embeddings still select OpenAI credentials, and DeepSeek does not register a memory embedding adapter. I did not live-reproduce the original full chat-blocking behavior on current main; the later QA comment supports current memory/auth degradation instead.

## Solution Assessment

Unclear. Reusing DeepSeek chat credentials is not currently a documented or implemented memory embedding path, so the maintainable solution needs a maintainer choice between explicit memorySearch setup/disclosure and a narrowly supported provider auto-routing contract.
