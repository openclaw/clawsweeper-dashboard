## Summary

Keep open. Current main and v2026.4.29 mitigate the worst symptom by making memory search/bootstrap embedding failures non-fatal, but the requested credential-routing behavior is still not implemented: memory auto-selection still chooses registered embedding adapters such as OpenAI and resolves credentials for that adapter, while the DeepSeek plugin only declares model/provider auth and no memory embedding provider contract.

## What This Changes

The issue reports that after upgrading to v2026.4.25, memory-core selects OpenAI embedding credentials instead of the active DeepSeek provider credentials, causing embedding 401s that block agent replies.

## Best Possible Solution

Keep the shipped non-fatal memory-search behavior, then make embedding credential selection deterministic and documented. The maintainable end state is either explicit memorySearch.provider/remote configuration with doctor guidance for mismatched OpenAI keys, or a DeepSeek-owned memory embedding adapter if maintainers confirm DeepSeek should support the embedding contract.

## Reproduction Assessment

Yes for the source-level reproduction path: current main shows auto memory embeddings choose registered memory adapters and OpenAI-scoped auth without consulting the active DeepSeek chat provider. The exact 401 requires the reporter's invalid OPENAI_API_KEY/proxy setup, which is not present in this checkout.

## Solution Assessment

Unclear. The current non-fatal mitigation is the right narrow protection for agent replies, but it is not the complete fix for deterministic embedding credential routing; maintainers should decide whether the supported path is explicit memorySearch configuration or a provider-owned DeepSeek/OpenAI-compatible embedding adapter.
