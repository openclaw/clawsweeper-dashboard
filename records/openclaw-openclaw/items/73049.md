## Summary

Keep open. Current main still lets `memorySearch.provider: "auto"` select OpenAI memory embeddings and OpenAI-scoped auth independently of a DeepSeek chat model, while DeepSeek still has no memory-embedding provider contract; the remaining fix is a credential-routing and product/security decision, not cleanup closure.

## What This Changes

The issue reports a v2026.4.25 regression where memory-core uses OpenAI embedding credentials during DeepSeek-backed agent processing and can fail with OpenAI 401s that block responses.

## Best Possible Solution

Define memory embedding credential ownership explicitly and make embedding failures degrade without blocking chat; if DeepSeek embeddings are supported, add that as a DeepSeek-owned adapter with docs and regression coverage.

## Reproduction Assessment

Yes for the source-level path: with `memorySearch.provider` unset or `auto`, current main can select OpenAI memory embeddings without consulting a DeepSeek chat model. The exact 401 still needs the reporter's external key/proxy setup.

## Solution Assessment

Unclear. Reusing the chat provider's credentials for embeddings is not necessarily the safest fix; maintainers should choose between explicit provider pinning/diagnostics, non-fatal fallback behavior, or a provider-owned DeepSeek embedding adapter.
