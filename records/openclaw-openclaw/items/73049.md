## Summary

Keep open. Current main mitigates the most severe symptom by making memory search/bootstrap embedding failures non-fatal, but it still does not implement the requested credential routing: automatic memory embeddings choose registered embedding adapters such as OpenAI and resolve that adapter's credentials, while the DeepSeek plugin only declares chat/model auth and no memory embedding adapter.

## What This Changes

Issue requests that memory-core stop selecting OpenAI-scoped embedding credentials when the agent chat model is DeepSeek, because the 2026.4.25 behavior produced OpenAI embedding 401s and blocked QQ Bot replies.

## Best Possible Solution

Keep the non-fatal memory-search behavior, then make embedding credential selection deterministic and documented. The maintainable end state is either explicit memorySearch.provider/remote configuration with doctor guidance for mismatched OpenAI keys, or a DeepSeek-owned memory embedding adapter if maintainers confirm DeepSeek should support the embedding contract.

## Reproduction Assessment

Yes for the credential-selection path: current source shows provider "auto" chooses memory embedding adapters and OpenAI-scoped auth without consulting the active DeepSeek chat provider. The exact live 401 requires the reporter's invalid OpenAI key/proxy setup, which is not available in this checkout environment.

## Solution Assessment

Unclear. The current non-fatal mitigation is the right narrow protection for agent replies, but it is not the complete fix for deterministic embedding credential routing; a maintainer should decide whether the supported path is explicit memorySearch configuration or a new DeepSeek/OpenAI-compatible embedding adapter.
