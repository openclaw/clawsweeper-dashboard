## Summary

Keep this PR open. Current main still omits cache read/write token counts from isolated cron run-log telemetry, while the proposed patch is a narrow additive forwarding change with no blocking functional or security issue found.

## What This Changes

The PR updates `src/cron/isolated-agent/run.ts` to copy positive `usage.cacheRead` and `usage.cacheWrite` values into cron telemetry as `cache_read_tokens` and `cache_write_tokens`.

## Best Possible Solution

Land this PR or an equivalent one-file fix, ideally with focused regression coverage on the isolated cron runner or run-log write path proving positive cache read/write counts appear in emitted telemetry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is available by providing `finalRunResult.meta.agentMeta.usage` with positive `cacheRead` or `cacheWrite`; current `finalizeCronRun` enters the usage block but returns telemetry without the corresponding JSONL cache fields.

## Solution Assessment

Yes. The proposed forwarding is the narrowest maintainable fix because the normalized usage, cron telemetry type, protocol schema, run-log reader, and cost estimator already support these fields.
