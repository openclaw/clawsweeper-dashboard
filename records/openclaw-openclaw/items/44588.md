## What This Changes

The PR copies positive cron `usage.cacheRead` and `usage.cacheWrite` values into `telemetryUsage.cache_read_tokens` and `telemetryUsage.cache_write_tokens` before run-log emission.

## Best Possible Solution

Run-log telemetry should carry the already-normalized cache counters into the existing `CronUsageSummary` fields, with the branch refreshed and redacted cron JSONL/log proof added before merge.

## Reproduction Assessment

Yes, source-reproducible: current main accepts cache fields in the run-log type/reader and recognizes cache counters as usage, but the isolated runner only copies input/output/total into the telemetry usage object. I did not run a live cron job in this read-only review.

## Solution Assessment

Yes: forwarding already-normalized counters into already-declared run-log fields is the narrowest maintainable fix. The implementation direction is sound, but merge should wait for branch refresh and real behavior proof.
