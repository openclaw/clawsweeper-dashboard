## Summary

Keep open: this is a narrow, valid bug-fix PR. Current main still stores cron cache read/write counters on the session entry but omits them from run-log telemetry, while the run-log schema and reader already support those fields; the remaining blocker is contributor-supplied real behavior proof before merge.

## What This Changes

The PR forwards positive isolated cron `cacheRead` and `cacheWrite` usage counters into run-log telemetry as `cache_read_tokens` and `cache_write_tokens`.

## Best Possible Solution

Land this hunk or an equivalent isolated-run telemetry fix after the contributor adds redacted real JSONL or log proof and normal cron validation passes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: positive `usage.cacheRead` or `usage.cacheWrite` reaches the isolated cron telemetry block, but current main only stores those values on the session entry and not in the run-log `usage` object.

## Solution Assessment

Is this the best way to solve the issue? Yes: forwarding the existing normalized counters into already-supported telemetry fields is the narrowest maintainable fix and avoids adding a new data path.
