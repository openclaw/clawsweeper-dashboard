## Summary

Keep this PR open: current main still omits cache read/write counters from isolated cron run-log telemetry, and the proposed hunk is a narrow fit, but the PR still lacks contributor-provided real cron/run-log proof before merge.

## What This Changes

The PR forwards positive normalized cron `cacheRead` and `cacheWrite` counters into run-log telemetry as `cache_read_tokens` and `cache_write_tokens`.

## Best Possible Solution

Land this hunk or an equivalent isolated cron telemetry producer fix after redacted real JSONL/log proof and current validation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-reproducible: an isolated cron run with positive normalized `cacheRead` or `cacheWrite` enters the telemetry block on current main, but those counters are only saved to the session entry and not written into `evt.usage`.

## Solution Assessment

Is this the best way to solve the issue? Yes: forwarding the existing normalized counters into already-supported telemetry fields is the narrowest maintainable fix and avoids a new data path.
