## Summary

Keep open: current main still omits cache token fields from cron run telemetry, and this PR is a narrow, correct fix for that gap. The remaining blocker is process proof, not a code finding: the external PR does not include after-fix real cron/run-log evidence.

## What This Changes

The PR adds positive `cacheRead` and `cacheWrite` counts from isolated cron usage to `telemetryUsage` as `cache_read_tokens` and `cache_write_tokens` before run-log emission.

## Best Possible Solution

Land this hunk or an equivalent isolated-run telemetry fix after redacted real run-log proof is added and normal rebase/CI validation passes; maintainer landing can handle release-note bookkeeping if needed.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: source inspection shows a run with positive `finalRunResult.meta.agentMeta.usage.cacheRead` or `cacheWrite` enters the telemetry block, but current main only persists those counters on the session entry and omits them from `evt.usage` written to the run log.

## Solution Assessment

Is this the best way to solve the issue? Yes: the PR forwards existing normalized counters into already-supported telemetry fields without adding a new data path or changing unrelated cron behavior.
