## Summary

Keep open: the restore guidance is useful and now verifies before extraction, but one security-boundary wording issue remains because it implies `openclaw backup verify` can make an archive trusted even though current verification is structural only.

## What This Changes

The PR adds a `docs/cli/backup.md` section explaining the current manual restore flow for backup archives: verify, extract, inspect `manifest.json`, stop OpenClaw, restore selected paths, and run doctor.

## Best Possible Solution

Land the docs after they say restore only archives from trusted or self-created sources, and that `backup verify` is structural validation before extraction, not an authenticity guarantee.

## Reproduction Assessment

Yes. Current-main source shows `openclaw backup` only registers create and verify subcommands, and current `docs/cli/backup.md` has no restore section.

## Solution Assessment

No, not quite. A manual restore docs section is the right narrow solution until a restore command exists, but the wording should not imply structural verification makes an archive trustworthy.
