## Summary

Keep this PR open: it is a useful docs fix for the linked backup restore gap, but the new restore flow verifies the archive only after extraction and the restore/copy step, which is the wrong safety order for a destructive recovery procedure.

## What This Changes

The PR adds a `Restore from an archive` section to `docs/cli/backup.md` describing manual restore steps because the CLI currently has create and verify commands but no restore command.

## Best Possible Solution

Keep the manual restore documentation, but make archive verification the first operational step before extraction or any filesystem overwrite, then continue with manifest inspection and minimal-path restore guidance.

## Reproduction Assessment

Yes. Reading PR head lines 85-98 shows the docs extract and restore before verification, while current source shows `openclaw backup verify` is the available archive validation path.

## Solution Assessment

No as written. The requested docs direction is right, but the maintainable fix is to reorder verification before extraction and restore while leaving runtime behavior unchanged.
