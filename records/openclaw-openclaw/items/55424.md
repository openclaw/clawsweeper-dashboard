## Summary

Keep open. Current main and v2026.5.22 only solve the opt-in `messages.suppressToolErrors` path; the default path can still emit a visible `Write failed`/`Edit failed` warning from a stale mutating `lastToolError` after success-looking output, and the direct timeout-suppression PR was closed unmerged.

## What This Changes

The issue reports false visible write/edit failure warnings when the filesystem write actually succeeds.

## Best Possible Solution

Add a narrow payload warning-policy repair that suppresses response/ack-timeout false warnings only for file `write`/`edit` cases that likely succeeded, while preserving visible warnings for confirmed file failures and non-file mutating failures.

## Reproduction Assessment

Yes, at source level. Current main still has a path where success-looking assistant text plus a mutating `write` `lastToolError` emits a visible failed-warning payload, and the checked-in tests preserve that default behavior.

## Solution Assessment

Yes. The narrow maintainable fix is to use existing timeout and file-mutation metadata in the payload warning policy, not to add broad mutating-tool suppression, a new config mode, or post-write filesystem verification.
