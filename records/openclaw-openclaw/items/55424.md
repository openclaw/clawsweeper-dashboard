## Summary

Keep open. Current main still surfaces mutating write/edit tool errors as user-facing warnings without using the available timeout marker, and the current regression test still expects a `write` connection-timeout case to show a `Write` warning. The closed related PR proposed the right narrow direction but was not merged, while the later related issue is duplicate context rather than a shipped fix.

## What This Changes

The issue requests that OpenClaw stop showing false Write/Edit failed warnings when file write/edit operations actually succeed but the tool response or acknowledgement path times out.

## Best Possible Solution

Keep this as the canonical open bug and land a narrow agent payload-policy fix that suppresses write/edit warnings only for structured response or acknowledgement timeouts, while preserving visible warnings for real write/edit failures and adding regression coverage for both paths.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists in the current payload tests: a `write` tool with `connection timeout` still produces a `Write` warning, and the runtime already records timeout state before payload rendering.

## Solution Assessment

Yes. The narrowest maintainable fix is to use existing `lastToolError.timedOut` state in `resolveToolErrorWarningPolicy` for write/edit response timeouts, not to add broad filesystem verification, retry behavior, or channel-specific logic.
