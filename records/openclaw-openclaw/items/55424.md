## Summary

Keep open. Current main still emits default visible `write`/`edit` mutating-tool failure warnings when `lastToolError` remains set, while the merged related fix only honors the opt-in `messages.suppressToolErrors` path and the earlier direct timeout-suppression PR was closed unmerged.

## What This Changes

The issue reports false `Write failed` or `Edit failed` warnings in Feishu even though file write/edit operations completed successfully on disk.

## Best Possible Solution

Add a narrow payload warning-policy repair that suppresses response/ack-timeout false warnings only for file `write`/`edit` cases that likely succeeded, while preserving visible warnings for confirmed file failures and non-file mutating failures.

Yes, at source level. Current main still has a path where a success-looking assistant reply plus a `write` `lastToolError` emits a visible `Write failed` payload, and the checked-in regression test encodes that behavior.

## Solution Assessment

Yes. The maintainable fix is a narrow warning-policy change using existing timeout/file-mutation metadata, not broad suppression across all mutating tools or a new filesystem verification wrapper.
