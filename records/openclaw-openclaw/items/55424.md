## Summary

Keep open. Current main still shows default mutating `write`/`edit` failure warnings when the tool-error state says a file mutation failed, and merged https://github.com/openclaw/openclaw/pull/81561 only fixed the opt-in `messages.suppressToolErrors` path rather than the reported default false-warning case.

## What This Changes

The issue reports OpenClaw showing `Write failed` or `Edit failed` for file operations even though the file was saved successfully after a tool response timeout.

## Best Possible Solution

Add a narrow payload warning-policy fix that suppresses response or acknowledgement-timeout false warnings only for file `write`/`edit` cases while preserving warnings for confirmed file failures and non-file mutating failures.

## Reproduction Assessment

Yes, at source level. Current main still has a payload-policy path and regression expectations where a `write`/`edit` mutating error can produce a visible failed warning even when assistant output claims success; I did not run the live Windows/Feishu/Bailian scenario.

## Solution Assessment

Yes. The best fix is a narrow warning-policy change using existing timeout metadata or tightly scoped response-timeout classification for file `write`/`edit`, not broad all-mutating timeout suppression like the closed https://github.com/openclaw/openclaw/pull/55444 attempt.
