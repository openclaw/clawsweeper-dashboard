## Summary

Keep open. Current main still has the same Linux user-systemd direct restart shape reported here: `gateway restart` awaits `systemctl --user restart` and reports `completed`, while adjacent launchd and update-restart paths already use detached/scheduled handoffs for caller-termination risk; the related PR is closed unmerged and was not a safe canonical fix.

## What This Changes

The issue reports that `openclaw gateway restart` on Linux user-systemd can report a successful restart but leave the CLI process with `exit=null` instead of `0`.

## Best Possible Solution

Add a narrow Linux/systemd restart handoff for at-risk managed-service invocations, returning the existing scheduled restart result while preserving synchronous error reporting for normal systemctl failures.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. The issue includes a concrete 2026.4.15 log, and current source still gives a clear reproduction path: invoke `openclaw gateway restart` from the managed Linux user-systemd service context and observe whether the spawned command is killed before returning code 0.

## Solution Assessment

Yes. The narrow maintainable repair is a systemd-specific detached or scheduled handoff using the existing service restart result contract, not a broad lifecycle rewrite or a late `process.exitCode = 0` after a synchronous restart that may kill the caller first.
