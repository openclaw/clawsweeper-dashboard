## Summary

Keep open. Current main still runs the Linux systemd restart path synchronously through `systemctl --user restart` and reports completion only if the caller survives, while adjacent launchd and update-restart code already use detached handoffs for this process-tree termination class. The related fix PR is closed unmerged, and the newer `Restart=always` packaged-unit work helps service recovery but does not prove the invoking `openclaw gateway restart` child exits with code 0.

## What This Changes

The issue asks for successful `openclaw gateway restart` invocations, especially on Linux systemd user services, to return process exit code 0 instead of a null child exit result.

## Best Possible Solution

Add a narrow Linux/systemd restart handoff for direct gateway service restarts when the caller is inside, or otherwise at risk from, the managed service process tree. Preserve synchronous error reporting for ordinary external `systemctl` failures, reuse the existing scheduled restart result shape, and add regression coverage that observes the CLI child process result rather than only mocked `systemctl` arguments.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The reported log plus current source give a strong plausible path: invoke `openclaw gateway restart` from the managed systemd user-service context or equivalent cgroup and observe the spawned child close result.

## Solution Assessment

Yes, the best direction is a narrow systemd handoff modeled on the existing scheduled-restart contract and launchd/update restart precedent. A broad platform rewrite is unnecessary because launchd already has this shape and Windows uses separate scheduled-task fallback logic.
