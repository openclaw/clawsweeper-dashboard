## Summary

Keep open. Current main still routes Linux `openclaw gateway restart` through a synchronous `systemctl --user restart` call, so the source still matches the reported process-tree termination risk; adjacent launchd and update-restart paths already use detached handoffs, and the related PR #68302 is closed unmerged rather than a canonical fix.

## What This Changes

The issue asks for successful `openclaw gateway restart` operations to return exit code 0 on Linux systemd, instead of leaving automation with a null child-process exit code.

## Best Possible Solution

Linux systemd gateway restarts should use an exit-safe handoff when the caller is inside, or otherwise at risk from, the managed service process tree, while preserving synchronous `systemctl` error reporting for ordinary external restart failures.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The issue includes a concrete 2026.4.15 log, and current source gives a strong plausible reproduction path: invoke `openclaw gateway restart` from the managed systemd user-service context or equivalent cgroup and observe the spawned child close result.

## Solution Assessment

Yes. The narrow maintainable fix is to add a systemd-specific handoff that reuses the existing scheduled restart result shape, rather than rewriting the whole service lifecycle or relying on `Restart=always` unit recovery to fix the CLI child exit code.
