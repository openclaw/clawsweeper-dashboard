## Summary

Keep open. Current main and the latest release still use the synchronous Linux `systemctl --user restart` path for direct gateway restarts, while the only focused handoff PR is closed unmerged.

## What This Changes

The issue asks for successful `openclaw gateway restart` on Ubuntu/user-systemd to exit with code 0 instead of producing a null process exit.

## Best Possible Solution

Add a focused Linux/user-systemd restart handoff for service-internal gateway restarts using the existing scheduled-restart contract, while preserving synchronous failure reporting for external `systemctl` failures.

No high-confidence live current-main reproduction was run in this read-only review. The reporter supplied a concrete 2026.4.15 Ubuntu/user-systemd log, and current source still follows the synchronous restart path that explains the failure mode.

## Solution Assessment

Yes. The narrow maintainable direction is a systemd-specific detached handoff that reuses the existing `{ outcome: "scheduled" }` contract, not a broad CLI policy change or a late `process.exitCode` workaround.
