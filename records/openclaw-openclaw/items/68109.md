## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/68109. Current main still runs direct Linux `openclaw gateway restart` through synchronous `systemctl --user restart` in `src/daemon/systemd.ts`, while `runServiceRestart()` awaits the service restart in `src/cli/daemon-cli/lifecycle-core.ts`; when invoked from the managed systemd service context this can terminate the caller and leave automation seeing `exit=null`. Add a narrow Linux/systemd detached or scheduled restart handoff only for at-risk managed-service invocations, return the existing `{ outcome: "scheduled" }` result where appropriate, and preserve normal synchronous `systemctl` failure reporting for ordinary external invocations. Do not implement a `process.exitCode = 0`-only workaround like the closed unmerged PR; do not change unrelated gateway HTTP/OpenAI paths. Add focused regression coverage around systemd restart scheduling and lifecycle scheduled-result handling, plus a changelog fix entry. Validate with the listed unit tests, formatting check, `pnpm check:changed` in Testbox, and live Linux user-systemd proof that a successful direct restart exits with code 0.
