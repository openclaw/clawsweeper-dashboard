## Summary

Keep open. Current main still routes Linux user-systemd `gateway restart` through a synchronous `systemctl --user restart` call while the CLI lifecycle awaits the service restart result, so the reported `exit=null` process-tree termination risk remains; the related PR #68302 is closed unmerged.

## What This Changes

The issue asks for successful `openclaw gateway restart` calls, especially under Linux user systemd, to return exit code 0 instead of a null child-process exit result.

## Best Possible Solution

Add a narrow Linux/systemd restart handoff for at-risk managed-service invocations, return the existing scheduled restart result shape, and preserve synchronous `systemctl` error reporting for ordinary external failures.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. The issue includes a concrete 2026.4.15 log, and current source gives a clear reproduction path: invoke `openclaw gateway restart` from the managed user-systemd service context or equivalent cgroup and observe the spawned command close result.

## Solution Assessment

Yes. The narrow maintainable fix is a systemd-specific detached or scheduled handoff that reuses the existing service restart result contract, rather than rewriting all service lifecycle handling or relying on unit `Restart=always` to make the caller's child exit code reliable.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/68109. Current main still makes Linux `openclaw gateway restart` call `systemctl --user restart` synchronously from `src/daemon/systemd.ts`, which can terminate the invoking CLI process tree under systemd and leave automation seeing a null child exit code. Implement a narrow systemd restart handoff when the caller is inside, or otherwise at risk from, the managed service process tree; return the existing `{ outcome: "scheduled" }` result shape and emit a clear scheduled restart line. Preserve synchronous `systemctl` error reporting for ordinary external restart failures. Do not rewrite launchd/schtasks behavior or broaden the service contract beyond the existing scheduled/completed result. Add focused tests for the systemd restart behavior and lifecycle scheduled-result path, and add a user-facing `CHANGELOG.md` entry without guessed attribution. Before handoff, run the targeted tests/formatter and get live Linux user-systemd proof that the spawned `openclaw gateway restart` command exits with code 0 on a successful restart.
