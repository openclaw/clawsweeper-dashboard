## Summary

Keep open: current `main` still has no supported path to pass custom HTTP headers from gateway-facing CLI commands into the WebSocket upgrade, and the only directly linked fix PR was closed unmerged. The adjacent open node PR is useful context, but it does not cover this issue’s `OPENCLAW_GATEWAY_HEADERS` / `gateway.remote.headers` / gateway CLI surface.

## What This Changes

Request custom HTTP header support for CLI WebSocket connections to remote gateways via a repeatable flag, environment variable, and config.

## Best Possible Solution

Define a maintainer-approved gateway header contract, then add validated parsing plus redaction-aware schema/docs and thread the resulting header map through `callGateway` into `GatewayClient` and the `ws` constructor with focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new capability request, not a broken documented contract. Source inspection gives high confidence the requested header path is absent on current `main`.

## Solution Assessment

Unclear until maintainers choose the public contract. The proposed flag/env/config layering is plausible, but because it adds a secret-bearing config and CLI surface, the safest solution is a deliberate gateway header design with validation and redaction before implementation.
