## Summary

Keep open: current main still lacks a gateway CLI custom-header path, and the related direct implementation PR was closed unmerged. The adjacent open node PR may cover part of the lower-level client plumbing, but it does not close this remote gateway CLI request.

## What This Changes

The issue requests custom HTTP header support for CLI WebSocket connections to remote gateways via a repeatable CLI flag, environment variable, and `gateway.remote.headers` config.

## Best Possible Solution

Add a gateway-wide header contract that threads validated, redacted custom headers through the CLI gateway option layer, config/env resolution, `callGateway()`, and `GatewayClient`, while coordinating with the adjacent node header work.

## Reproduction Assessment

Yes from source inspection: current main has no CLI, env, config, call-layer, or WebSocket-client path for custom gateway headers. I did not run a live proxy reproduction because this was a read-only review and the missing option is visible in the code.

## Solution Assessment

Unclear: the lower-level `ws` threading is likely narrow, but the public flag/env/config shape and secret-handling policy need maintainer product/security review. The implementation should align with existing gateway credential precedence and any shared pieces from #42841.
