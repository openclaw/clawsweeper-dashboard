## Summary

Keep open: current main and the latest release still lack custom-header input and forwarding for Gateway CLI WebSocket connections. The request is plausible and aligned with remote Gateway operations, but it adds credential-bearing CLI/env/config surface that needs maintainer product and security review before implementation.

## What This Changes

Request to let Gateway CLI WebSocket clients pass custom HTTP headers via a repeatable CLI flag, environment variable, and `gateway.remote` config.

## Best Possible Solution

Define one secret-safe Gateway WebSocket header contract in `packages/gateway-client` and `callGateway`, then expose only approved CLI/env/config surfaces with redaction, SecretInput/env handling, docs, and focused tests.

## Reproduction Assessment

Yes, at source level: current main exposes no shared Gateway CLI `--header` option, no headers on `GatewayRpcOpts`, `CallGatewayBaseOptions`, or `GatewayClientOptions`, and no `gateway.remote.headers` schema. I did not run a live Cloudflare Access repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product and security contract. The proposed owner boundary is plausible, but the exact flag/env/config precedence and secret-handling model should be approved before implementation.
