## Summary

Keep open: current `main` and latest release `v2026.6.8` still lack a Gateway CLI custom-header path, while the exact and node-specific implementation PRs are both closed unmerged. The remaining work is real, but it adds secret-bearing CLI/env/config surface and should stay with maintainer product/security review before implementation.

## What This Changes

The issue requests custom HTTP headers for CLI WebSocket connections to remote gateways via a repeatable flag, environment variable, and `gateway.remote` config.

## Best Possible Solution

Define one secret-safe Gateway WebSocket header contract in the active `packages/gateway-client` and `callGateway` path, then expose it consistently through approved CLI/env/config surfaces with docs, redaction, and focused tests.

## Reproduction Assessment

Yes, at source level: current `main` exposes no shared Gateway CLI `--header` option, no `callGateway` or `GatewayClientOptions` header field, and no `gateway.remote.headers` config field. I did not run a live Cloudflare Access repro because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the product/security contract. The proposed shape is plausible, but the maintainable fix should thread through the active GatewayClient/callGateway boundary with explicit secret handling rather than only one CLI wrapper.
