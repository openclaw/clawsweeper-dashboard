## Summary

Keep open: current main still lacks a supported custom-header path for remote Gateway CLI WebSocket connections. The direct implementation PR was closed unmerged, and the open node header PR covers a different command surface, so the remaining work needs maintainer product and security approval rather than cleanup closure.

## What This Changes

The issue requests custom HTTP header support for remote Gateway CLI WebSocket connections via a repeatable CLI flag, environment variable, and `gateway.remote.headers` config.

## Best Possible Solution

Define one secret-safe custom-header contract for remote Gateway clients, resolve headers centrally alongside existing remote URL/auth/TLS settings, and document/test CLI, env, and config precedence.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows no `--header`, no `OPENCLAW_GATEWAY_HEADERS`, no `gateway.remote.headers`, and no header forwarding into `GatewayClient`. This is a missing feature path, not a failing documented behavior bug.

## Solution Assessment

Unclear until maintainers approve the public secret-bearing surface. If accepted, the maintainable path is to resolve headers in the shared Gateway call path and forward them once into `GatewayClient`, with schema/help/docs and focused tests.
