## What This Changes

Request support for custom HTTP headers on CLI WebSocket connections through `--header`, `OPENCLAW_GATEWAY_HEADERS`, and `gateway.remote.headers`.

## Best Possible Solution

Define one secret-safe custom-header contract for remote Gateway clients, resolve it centrally beside remote URL/auth/TLS settings, and document/test CLI, env, and config precedence.

## Reproduction Assessment

Yes, for the current limitation: source inspection on target main shows no `--header`, no `OPENCLAW_GATEWAY_HEADERS`, no `gateway.remote.headers`, and no header forwarding into `GatewayClient`. This is a missing feature path, not a failing documented behavior bug.

## Solution Assessment

Unclear until maintainers approve the public secret-bearing surface. If accepted, the maintainable path is to resolve headers in the shared Gateway call path and forward them once into `GatewayClient`, with schema/help/docs and focused tests.

## Risks / Open Questions

- Header values can contain service tokens, cookies, or Authorization values, so CLI arguments, environment variables, persisted config, diagnostics, and logs need an explicit redaction and storage policy.
- A comma-separated `OPENCLAW_GATEWAY_HEADERS` format can mishandle valid header values containing commas unless maintainers choose a safer representation or document the limitation.
- The open node-header PR overlaps at the `GatewayClient` forwarding seam; Gateway CLI and node header semantics should be coordinated to avoid divergent precedence or secret-at-rest behavior.
