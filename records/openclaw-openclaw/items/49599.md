## What This Changes

The issue requests custom HTTP header support for remote Gateway CLI WebSocket connections via a repeatable CLI flag, an environment variable, and `gateway.remote.headers` config.

## Best Possible Solution

Define one secret-safe custom-header contract for remote Gateway WebSocket clients, resolve headers centrally in the Gateway call path, and document/test CLI, env, and config precedence.

## Reproduction Assessment

Yes, for the current limitation: source inspection on current main shows no `--header`, no `OPENCLAW_GATEWAY_HEADERS`, no `gateway.remote.headers`, and no forwarding into `GatewayClient`. This is a missing feature path, not a failing documented behavior bug.

## Solution Assessment

Unclear until maintainers approve the public secret-bearing header surface. If accepted, the maintainable path is to resolve headers once in the shared Gateway call path and forward them into `GatewayClient`, with schema/help/docs and focused tests.
