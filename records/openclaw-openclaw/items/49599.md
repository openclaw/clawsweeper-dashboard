## Summary

Keep open: current main still has no custom-header path for shared Gateway CLI RPC WebSocket connections, and the exact earlier PR for this issue was closed unmerged. The open related node PR covers a different node run/install surface, so it should not close this broader CLI remote-gateway request.

## What This Changes

Add repeatable Gateway CLI header support plus env/config handling for custom HTTP headers on remote Gateway WebSocket upgrade requests.

## Best Possible Solution

Define one supported, secret-safe custom-header surface for remote Gateway clients and implement it centrally in the shared Gateway call path with schema/help/docs and focused tests.

## Reproduction Assessment

Yes for the current limitation: source inspection shows no `--header` option, no `OPENCLAW_GATEWAY_HEADERS` handling, no `gateway.remote.headers` schema field, and no header forwarding into `GatewayClient`. I did not run a live proxy because this is a missing feature rather than a broken documented contract.

## Solution Assessment

Unclear until maintainers choose the public surface. If accepted, the narrow maintainable approach is to resolve headers once with the other remote Gateway connection settings and pass them into `GatewayClient` without logging or duplicating secret values.
