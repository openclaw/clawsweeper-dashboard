## Summary

Keep open: current main still lacks a supported custom-header path for remote Gateway CLI WebSocket connections, the direct implementation PR was closed unmerged, and the open node header PR covers a different command surface. The recent user comment reinforces that this remains useful for Cloudflare Access-style deployments, but the secret-bearing header surface needs maintainer product and security approval before implementation.

## What This Changes

Request adding repeatable Gateway CLI custom headers via `--header`, `OPENCLAW_GATEWAY_HEADERS`, and `gateway.remote.headers` for remote WebSocket connections behind auth proxies.

## Best Possible Solution

Define one secret-safe custom-header contract for remote Gateway clients, resolve headers centrally alongside existing remote URL/auth/TLS settings, and document/test CLI, env, and config precedence.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows no `--header`, no `OPENCLAW_GATEWAY_HEADERS`, no `gateway.remote.headers`, and no header forwarding into `GatewayClient`. This is a missing feature path, not a failing documented behavior bug.

## Solution Assessment

Unclear until maintainers approve the public secret-bearing surface. If accepted, the maintainable path is to resolve headers in the shared Gateway call path and forward them once into `GatewayClient`, with schema/help/docs and focused tests.
