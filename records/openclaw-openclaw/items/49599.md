## Summary

Keep open: current main and the latest release still lack a way for Gateway CLI WebSocket callers to pass custom HTTP headers, and the remaining work is a credential-bearing config/env/CLI product decision rather than conservative cleanup.

## What This Changes

The issue requests custom HTTP headers for remote Gateway WebSocket CLI connections via a repeatable CLI flag, environment variable, and/or `gateway.remote.headers` config.

## Best Possible Solution

Define one secret-safe Gateway WebSocket header contract in the core GatewayClient/callGateway path, then expose it consistently through CLI, env/config schema, docs, redaction, and focused tests.

## Reproduction Assessment

Yes at source level: current main exposes no gateway CLI `--header` option, no `GatewayRpcOpts.headers`, no `CallGatewayBaseOptions.headers`, no `GatewayClientOptions.headers`, and no `gateway.remote.headers` schema field. I did not run a live Cloudflare Access reproduction.

## Solution Assessment

Unclear until maintainers choose the product/security contract. The proposed three-source API is plausible, but the best fix should live in the active `packages/gateway-client` and core `callGateway` boundary with explicit secret handling rather than only in one CLI wrapper.
