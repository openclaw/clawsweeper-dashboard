## Summary

Keep open. Current main still does not expose custom HTTP headers for CLI Gateway WebSocket connections through shared Gateway CLI options, the low-level gateway call path, GatewayClient construction, or the strict gateway.remote config contract. The exact prior implementation PR #49646 was closed unmerged, and the open #42841 is adjacent node run/install work rather than the requested CLI Gateway path.

## What This Changes

Request support for custom HTTP headers on CLI Gateway WebSocket connections via repeatable `--header`, `OPENCLAW_GATEWAY_HEADERS`, and `gateway.remote.headers`.

## Best Possible Solution

Implement the feature end-to-end in the core Gateway CLI/client path: parse and validate custom headers from CLI, env, and config with documented precedence, treat values as sensitive, pass them into `ws` `ClientOptions.headers`, update config docs/help, and add focused tests for parsing, precedence, redaction, and upgrade delivery.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a remote gateway behind a proxy that rejects WebSocket upgrades unless Cloudflare Access-style headers are present; current main has no CLI/env/config route to send those headers, as confirmed by the option, config, and WebSocket construction code.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution for CLI Gateway connections. The safest implementation should reuse or align with any generic `GatewayClient` header seam from #42841 while keeping `OPENCLAW_GATEWAY_HEADERS` and `gateway.remote.headers` scoped to Gateway CLI remote connections.

## Repair Work Prompt

Implement custom HTTP headers for CLI Gateway WebSocket connections tracked by https://github.com/openclaw/openclaw/issues/49599. Add a repeatable `--header "Name: Value"` option to the shared Gateway CLI call surfaces, add `OPENCLAW_GATEWAY_HEADERS` parsing, and add `gateway.remote.headers` to the config contract. Precedence must be CLI > env > config. Validate header names and parse values by splitting on the first colon; preserve colons in values; decide and test empty-value behavior explicitly. Treat header values as sensitive: do not log them, do not leak them in errors/status/config snapshots, and wire redaction/schema sensitivity where config values can hold secrets. Thread a `Record<string, string>` through `GatewayRpcOpts`, `CallGatewayBaseOptions`, and `GatewayClientOptions`, then merge into `ws` `ClientOptions.headers` before `new WebSocket(url, wsOptions)`. Update config types, zod schema, generated/help/labels as needed, and user-facing docs for Gateway remote CLI usage. Coordinate with open PR #42841 if it has landed or still touches `GatewayClient` headers; do not duplicate node run/install-specific `OPENCLAW_NODE_HEADERS` behavior, and keep this PR scoped to CLI Gateway remote connections. Use closed PR #49646 only as implementation context, not as shipped behavior. Add focused tests for CLI/env/config parsing and precedence, `GatewayClient` WebSocket header delivery, strict config acceptance, and redaction. Add a changelog entry because this is user-facing CLI behavior.
