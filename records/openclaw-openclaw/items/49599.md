## Summary

Keep open: current main still lacks custom WebSocket header plumbing in the CLI options, gateway call layer, GatewayClient, and `gateway.remote` config. The earlier implementation PR was closed unmerged, and the open node PR covers a narrower node run/install surface rather than general CLI gateway calls.

## What This Changes

Allow remote CLI WebSocket calls to pass custom HTTP headers via a repeatable CLI flag, environment variable, and `gateway.remote.headers` config.

## Best Possible Solution

Implement a narrow core CLI/gateway-client feature that centralizes header parsing/resolution, threads headers through `callGateway` into `GatewayClient`, updates config schema/docs/changelog, and keeps node-specific header work aligned.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main gap: CLI gateway options, `callGateway`, `GatewayClient`, and `gateway.remote` config do not expose or forward custom WebSocket headers.

## Solution Assessment

Yes in direction, but the implementation should be refined. `ws` supports request headers, while OpenClaw should resolve them centrally, avoid fragile env parsing, and protect secret-bearing values from logs/status output.

## Risks / Open Questions

- Header values may contain secrets, so any implementation needs the same redaction and persistence discipline as gateway tokens/passwords.
- The proposed comma-delimited `OPENCLAW_GATEWAY_HEADERS` format can lose valid header values containing commas unless the env format is tightened or documented.
- PR #42841 is solving a node-specific header path in parallel, so naming, parsing, and tests should be coordinated to avoid two incompatible header mechanisms.
