## Summary

Keep open: current main and the latest release still lack a supported custom-header path for remote Gateway CLI WebSocket connections, and both linked implementation PRs are closed unmerged. The request is valid, but it needs maintainer product/security approval because it adds credential-bearing CLI, environment, and config surface.

## What This Changes

Add support for custom HTTP headers on remote Gateway CLI WebSocket connections through a repeatable CLI flag, environment variable, and `gateway.remote` config.

## Best Possible Solution

Define one secret-safe Gateway header contract resolved beside remote URL/auth/TLS, forward it once through `packages/gateway-client`, and document/test CLI, env, and config precedence before implementation.

## Reproduction Assessment

Yes for the current limitation: source inspection shows no `--header`, no `OPENCLAW_GATEWAY_HEADERS`, no `gateway.remote.headers`, and no GatewayClient header forwarding. This is a missing feature path rather than a failing documented bug.

## Risks / Open Questions

- Header values may contain service tokens, cookies, or Authorization values, so CLI argument exposure, environment parsing, config persistence, diagnostics, and log redaction need an explicit maintainer policy.
- A comma-separated `OPENCLAW_GATEWAY_HEADERS` format would mishandle valid header values containing commas unless maintainers choose a safer representation or document the limitation.
- Gateway CLI and node-host header semantics should be coordinated so precedence, storage, and redaction do not diverge across remote management paths.
