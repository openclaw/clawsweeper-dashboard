## Summary

Keep open: current main and v2026.6.6 still lack a supported custom-header path for remote Gateway CLI WebSocket connections, and the linked implementation attempts are closed unmerged. The request is valid, but it needs maintainer product/security approval because it adds secret-bearing CLI, env, and config surface.

## What This Changes

Request support for custom HTTP headers on CLI Gateway WebSocket connections via a repeatable flag, environment variable, and remote Gateway config.

## Best Possible Solution

Define one secret-safe Gateway header contract resolved beside remote URL/auth/TLS, forward it once through `packages/gateway-client`, and document/test CLI, env, and config precedence before implementation.

## Reproduction Assessment

Yes for the current limitation: source inspection shows no `--header`, no `OPENCLAW_GATEWAY_HEADERS`, no `gateway.remote.headers`, and no GatewayClient header forwarding. This is a missing feature path rather than a failing documented bug.

## Solution Assessment

Unclear until maintainers approve the public secret-bearing surface. If accepted, central Gateway call resolution plus `packages/gateway-client` forwarding is the maintainable implementation boundary.

## Risks / Open Questions

- Header values can contain service tokens, cookies, or Authorization values, so CLI argument exposure, environment parsing, config persistence, diagnostics, and log redaction need an explicit policy.
- A comma-separated env format would mishandle valid header values containing commas unless maintainers choose a safer representation or document the limitation.
- Gateway CLI, node-host, macOS remote mode, and non-CLI call paths should share one contract so precedence, storage, and redaction do not diverge.
