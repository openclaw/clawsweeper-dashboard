## Summary

Keep open: current main and v2026.6.6 still lack a way for Gateway CLI WebSocket callers to pass custom HTTP headers, and the remaining work is a secret-bearing CLI/env/config product and security decision.

## What This Changes

The issue requests CLI, environment, and config support for custom HTTP headers on Gateway CLI WebSocket connections to authenticate through reverse proxies.

## Best Possible Solution

Define one secret-safe Gateway WebSocket header contract in the core GatewayClient/callGateway path, then expose it consistently through CLI, env/config schema, docs, redaction, and focused tests.

## Reproduction Assessment

Yes, at source level: current main exposes no Gateway CLI `--header` option, no Gateway RPC/call/client header field, and no `gateway.remote.headers` config field. I did not run a live Cloudflare Access reproduction.

## Solution Assessment

Unclear until maintainers choose the product/security contract. The requested shape is plausible, but the best fix should thread through the active `packages/gateway-client` and core `callGateway` boundary with explicit secret handling rather than only one CLI wrapper.
