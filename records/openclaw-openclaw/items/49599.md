## Summary

Keep open: current main and v2026.6.6 still lack a supported custom-header path for remote Gateway CLI WebSocket connections, and the two directly linked implementation PRs are closed unmerged. The request is valid, but it needs maintainer product/security approval because it adds credential-bearing CLI, environment, config, redaction, and transport surface.

## What This Changes

The issue requests custom HTTP headers on remote Gateway CLI WebSocket connections via a repeatable CLI flag, environment variable, and `gateway.remote.headers` config.

## Best Possible Solution

Define one secret-safe Gateway remote-header contract resolved beside remote URL/auth/TLS, forward it through `packages/gateway-client`, and document/test CLI, env, and config precedence before implementation.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.6 shows no `--header`, no `OPENCLAW_GATEWAY_HEADERS`, no `gateway.remote.headers`, and no `GatewayClient` header forwarding path; this is a missing feature rather than a broken documented behavior.

## Solution Assessment

Unclear until maintainers approve the public secret-bearing surface. If accepted, the maintainable boundary is central Gateway call/config resolution plus `packages/gateway-client` forwarding, not a one-off CLI-only parser.
