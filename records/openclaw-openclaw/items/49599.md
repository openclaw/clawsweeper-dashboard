## Summary

Keep open: current main still lacks a custom-header path for shared remote Gateway CLI WebSocket connections, and the directly related implementation pull request was closed unmerged. The remaining request is a new secret-bearing CLI/env/config surface, so it needs maintainer product and security review before implementation.

## What This Changes

The issue requests custom HTTP headers for remote Gateway CLI WebSocket connections via a repeatable CLI flag, environment variable, and `gateway.remote.headers` config.

## Best Possible Solution

Define one supported, secret-safe custom-header contract for remote Gateway clients and implement header resolution centrally in the shared Gateway call path with schema/help/docs and focused tests.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows no `--header` option, no `OPENCLAW_GATEWAY_HEADERS` handling, no `gateway.remote.headers` schema field, and no header forwarding into `GatewayClient`. This is not a broken documented behavior repro; it is a missing feature path.

## Solution Assessment

Unclear until maintainers approve the public secret-bearing surface. If accepted, the narrow maintainable approach is to resolve headers alongside the other remote Gateway connection settings and pass them once into `GatewayClient`.
