## Best Possible Solution

Define a single secret-safe header contract resolved beside Gateway remote URL/auth/TLS, forward it once through `GatewayClient`, and document/test CLI, env, and config precedence.

## Risks / Open Questions

- Header values can contain service tokens, cookies, or Authorization values, so CLI arguments, environment variables, persisted config, diagnostics, and logs need an explicit redaction and storage policy.
- A comma-separated `OPENCLAW_GATEWAY_HEADERS` format would mishandle valid header values containing commas unless maintainers choose a safer representation or document the limitation.
- The open node-header PR overlaps at the `GatewayClient` forwarding seam, so Gateway CLI and node header semantics should be coordinated to avoid divergent precedence or secret-at-rest behavior.
