## Summary

Keep open. The status bug is still source-reproducible on current main, but this PR does not fix the full status path, lacks real behavior proof, and bundles unrelated stale UI/script/rate-limit changes that should not ride with the model-status fix.

## What This Changes

The branch validates model status default labels, expands rate-limit matching, treats empty target fields as absent, moves Podman `.env` sourcing earlier, and adds WebChat slash-command autocomplete.

## Best Possible Solution

Replace this with a focused status-model resolver change that feeds one provider-valid effective default into JSON output, rich output, auth probes, and the status summary, with regression coverage and no unrelated UI/script changes.

## Reproduction Assessment

Yes, by source inspection. Current main still routes provider-qualified raw defaults into model-status JSON, auth probe candidates, rich output, and the separate status summary resolver.

## Solution Assessment

No. The PR validates only `defaultLabel` after `resolvedLabel` can already be stale, leaves other consumers untouched, and bundles unrelated changes; the safer fix is a narrow shared status resolver plus tests.
