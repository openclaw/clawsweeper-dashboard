## Summary

Keep open: the stale-model status bug is still present on current main, but this PR is incomplete and now bundles unrelated or superseded changes that should not merge as-is.

## What This Changes

The PR validates the raw default model provider for models status, broadens several rate-limit matchers, treats empty legacy channel target fields as absent, moves Podman env-file sourcing before bind resolution, and adds WebChat slash-command autocomplete UI.

## Best Possible Solution

Replace this branch with a focused status-model resolver fix reused by JSON output, rich output, auth probes, missing-provider summaries, and the `openclaw status` summary, with regression tests and a changelog entry.

## Reproduction Assessment

Yes. By source inspection on current main, a provider-qualified raw default from a removed provider still feeds `defaultLabel`, auth probe candidates, rich output, and the status summary resolver.

## Solution Assessment

No. The PR only partially changes one status label, falls back through a resolver that can still be based on the stale raw value, and bundles unrelated UI/script/runtime changes; a shared provider-valid effective-default helper is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use a provider-valid default for every status consumer:** `src/commands/models/list.status-command.ts:109`
  - body: The new validation only changes `defaultLabel`, but `resolvedLabel` can already be derived from the stale raw default, and `rawCandidates`, rich output, and the separate status summary path still consume raw values. The stale provider can therefore still be shown or probed after this patch.
  - confidence: 0.93
- **[P2] Keep bare token-count errors out of rate-limit matching:** `src/agents/pi-embedded-helpers/errors.ts:122`
  - body: Adding bare `too many tokens` to the rate-limit matchers also catches context-window errors such as `too many tokens per request`. Current main explicitly preserves those as context overflow instead of rate limit, so this broad matcher would regress failover classification.
  - confidence: 0.86
- **[P3] Hide the autocomplete when there are no matches:** `ui/src/ui/views/chat.ts:491`
  - body: The dropdown is rendered whenever the draft starts with `/`, even when the filtered command list is empty. Typing an unknown command such as `/xyz` leaves an empty styled menu above the composer.
  - confidence: 0.88
- **[P3] Wire the swallowed navigation keys to the menu:** `ui/src/ui/views/chat.ts:461-463`
  - body: The key handler intercepts ArrowUp, ArrowDown, and Tab while the dropdown has no selected state or keyboard handler. Keyboard navigation and completion therefore do not work despite those keys being reserved.
  - confidence: 0.84
- **[P3] Return focus to the composer after command selection:** `ui/src/ui/views/chat.ts:511-514`
  - body: Clicking a command suggestion updates the draft but leaves focus on the clicked button, so users cannot continue typing arguments until they manually focus the textarea again.
  - confidence: 0.86
