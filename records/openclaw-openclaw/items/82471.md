## Summary

Keep open: this is a useful focused Gateway protocol PR with supplied real behavior proof, but the new `chat.inject` idempotency path is not race-safe and can still duplicate concurrent retries before merge.

## What This Changes

The PR adds optional `idempotencyKey`, `command`, `interactive`, and `channelData` fields to `chat.inject`, persists and broadcasts them, updates Swift protocol models, docs, and focused tests.

## Best Possible Solution

Land the additive metadata contract after moving idempotency dedupe into the locked transcript append path and after API review plus green checks.

## Reproduction Assessment

Not applicable as a bug reproduction for the requested API feature. For the PR defect, source inspection gives a high-confidence reproduction path: concurrent `chat.inject` calls with the same `idempotencyKey` can both pass the pre-lock scan before either write lands.

## Solution Assessment

No, not yet: the metadata plumbing is the right narrow direction, but idempotency must be enforced inside the locked append path so duplicate retries are at-most-once under concurrency.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Make chat.inject idempotency atomic:** `src/gateway/server-methods/chat.ts:2916`
  - body: This passes the caller-supplied `idempotencyKey` into `appendAssistantTranscriptMessage`, but that helper checks `transcriptHasIdempotencyKey` before the per-transcript append queue/write lock. Two concurrent retries with the same key can both finish the precheck before either append lands, so both will persist and broadcast, violating the advertised at-most-once `chat.inject` contract. Move the scan into the locked append path, or reuse the shared race-safe approach from https://github.com/openclaw/openclaw/pull/81513, and add a concurrent replay test.
  - confidence: 0.91
