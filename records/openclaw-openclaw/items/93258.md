## Summary

Keep open: current main has partial SDK retry, session auto-retry, and model-fallback behavior, but I did not find a merged/current implementation of the requested configurable provider-classified retry-before-fallback policy, and related canonical work is still open rather than resolved.

## What This Changes

Request to add configurable retry-with-backoff for transient upstream LLM provider failures before a turn becomes terminal or falls through to model fallback.

## Best Possible Solution

Define a narrow provider request retry policy that retries only replay-safe pre-output model-call failures, honors Retry-After and a bounded budget, composes with SDK retries, model fallback, session suspension, and terminal-turn UX, and documents any new config surface before implementation.

## Reproduction Assessment

No high-confidence live repro is available. The report says the failures are transient and offers sanitized transcripts on request; source inspection shows partial retry/fallback paths and a remaining retry-before-fallback design gap, but I did not reproduce a live Anthropic failure.

## Solution Assessment

Unclear. A provider-classified retry-before-fallback policy is plausible, but the best fix needs maintainer design for streaming partial output, tool side effects, SDK retry interaction, and fallback/suspension ordering.
