## Summary

Keep open: current main and the latest release still keep a module-level diffs viewer controller list and append to it during hydration without clearing it. The remaining uncertainty is the exact current-main trigger for running the same hydration path twice, so this should stay as low-priority lifecycle follow-up rather than close as fixed.

## What This Changes

The issue reports that the diffs viewer can retain stale controller entries if the viewer hydrates more than once in the same page lifecycle.

## Best Possible Solution

Make the viewer hydration path idempotent only after proving the repeated-hydration trigger, then reset or replace the controller collection at the hydration boundary with focused DOM regression coverage and a regenerated runtime asset.

## Reproduction Assessment

No high-confidence executed current-main repro is available. Source inspection shows that a second call to the same module's hydration path would append duplicate controllers, but current main's loader path appears to import the runtime once and does not export `hydrateViewer`.

## Solution Assessment

Unclear until the repeated-hydration path is proven. If rehydration is possible or becomes supported by the adjacent test seam, clearing or replacing `controllers` at the start of hydration is the narrow maintainable fix.
