## Summary

Keep open. This is a real security-hardening proposal linked to #7604, but the PR head has merge-blocking build/runtime problems, incomplete transcript coverage, and unresolved at-rest encryption threat-model issues that need maintainer/security-owner design review before implementation can continue.

## What This Changes

The PR adds AES-GCM session encryption helpers, transcript readers and migration utilities, sessions.json encryption hooks, and gateway/session-reader changes intended to transparently encrypt session JSONL transcripts and session metadata.

## Best Possible Solution

Design this at the central session persistence boundary with explicit key-source and recovery semantics, fail-closed migration behavior, full reader/writer coverage, focused tests, docs/changelog updates, and secops maintainer approval before landing.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists from the PR head and current main: the diff adds async/CommonJS constructs to synchronous ESM paths, while current main still writes ordinary transcripts through direct SessionManager.open calls outside the new wrapper.

## Solution Assessment

No. The proposed approach patches selected readers and one delivery-mirror helper instead of the central persistence seam, and the local key storage/fallback model does not satisfy the stated filesystem-compromise defense boundary.

## Security Review

Status: needs_attention

Summary: The diff introduces at-rest encryption and migration behavior with concrete key co-location and plaintext-preservation risks.

Concerns:

- **[high] Session keys are stored beside encrypted state:** `src/config/sessions/encryption.ts:38`
  - body: The new key wrapper stores passphrase/key material under the OpenClaw state directory. A same-user filesystem reader who can copy session files can likely copy these files too, so this does not meet the stated filesystem-compromise defense boundary without a different key source such as a user secret or platform keystore.
  - confidence: 0.88
- **[medium] Transcript migration can preserve plaintext lines:** `src/config/sessions/migrate-transcripts.ts:36`
  - body: migrateTranscriptToEncrypted catches per-line encryption failures, appends the original plaintext line, and proceeds to write the migrated file. That can leave sensitive transcript content unencrypted while the migration appears successful.
  - confidence: 0.9

## Risks / Open Questions

- The proposed key material is stored under the same OpenClaw state directory as the encrypted sessions, so a same-user filesystem compromise can likely copy both ciphertext and keys.
- The patch does not cover all primary SessionManager read/write paths, so normal runs, compaction, and repair flows can still create plaintext or mixed-format transcripts.
- The migration helper can write original plaintext lines after an encryption failure while still reporting migration success.
- The PR introduces ESM/runtime and synchronous API blockers before the security design can be validated.
- The user-facing feature lacks a complete maintainer-approved docs, changelog, rollout, and recovery story for encrypted session state.
