## Summary

Keep this PR open for maintainer/security review, but do not merge it as-is. Current main does not implement transparent session transcript encryption, yet the PR branch has build/runtime blockers, incomplete coverage of the central session persistence path, and security-sensitive key-management and migration concerns that need product/security design before repair automation.

## What This Changes

The PR adds session transcript and sessions.json encryption helpers, migration utilities, gateway/session reader changes, and a status document intended to transparently encrypt local session JSONL transcripts and metadata.

## Best Possible Solution

Design encryption at the central session persistence boundary with explicit key sourcing, recovery and rollback semantics, fail-closed migration, complete reader/writer coverage, docs, changelog, focused tests, and maintainer security approval before landing.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists from the PR head: the patch contains an `await` in a synchronous function, CommonJS `require()` calls in ESM code, and current main's normal embedded run still writes through `SessionManager.open(params.sessionFile)` directly.

## Solution Assessment

No. The proposed patch is not the best way to solve this because it patches selected readers and a secondary append helper instead of the central persistence seam, while storing local key material beside the encrypted state it is meant to protect.

## Review Findings

Full review comments:

- **[P1] Keep async imports out of the synchronous preview reader:** `src/gateway/session-utils.fs.ts:474`
  - body: `readLastMessagePreviewFromOpenTranscript` is still a synchronous function, but this patch adds `await import(...)` inside it. That is a build/syntax blocker before runtime; use a static import or change the exported call chain consistently.
  - confidence: 0.97

  - body: This package is loaded as ESM, but `EncryptedSessionManager.open()` and `forkFrom()` call `require(...)`. Any path that reaches these helpers can fail with `require` undefined; use static ESM imports or an existing lazy runtime boundary.
  - confidence: 0.92
- **[P2] Cover the primary transcript writer:** `src/config/sessions/transcript.ts:281`
  - body: This routes only the delivery-mirror append helper through the encrypted wrapper, while normal embedded runs still write via direct Pi `SessionManager.open(...)` paths. Ordinary session JSONL writes can remain plaintext, so the claimed transparent encryption is incomplete.
  - confidence: 0.91
- **[P2] Fail migration instead of writing plaintext lines:** `src/config/sessions/migrate-transcripts.ts:36-37`
  - body: When a line fails to encrypt, migration appends the original plaintext line and can still write the file as a successful migration. That leaves sensitive content unencrypted while callers believe migration completed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces at-rest encryption and migration behavior with concrete key co-location and plaintext fallback concerns.

Concerns:

- **[high] Session keys are stored beside encrypted state:** `src/config/sessions/encryption.ts:45`
  - body: The new key wrapper stores both the passphrase file and wrapped master key under the OpenClaw state directory. A same-user filesystem reader who can copy session files can likely copy these key files too, so this does not satisfy the stated filesystem-compromise defense boundary without a different key source such as a user secret or platform keystore.
  - confidence: 0.88
- **[medium] Transcript migration can preserve plaintext lines:** `src/config/sessions/migrate-transcripts.ts:37`
  - body: `migrateTranscriptToEncrypted` catches per-line encryption failures, appends the original plaintext line, and proceeds to write the migrated file. That can leave sensitive transcript content unencrypted while the migration appears successful.
  - confidence: 0.9
