## Summary

Keep open: the PR targets a real current-main Codex transcript cap drift, but the patch treats a user-controlled truncation marker as trusted state, which can bypass the configured cap; the supplied proof also does not directly show an oversized tool result being projected and capped after the change.

## What This Changes

The PR updates Codex app-server transcript projection and tests so streamed and snapshot native tool-result messages use configured default or per-agent `toolResultMaxChars` instead of the projector's local 12,000-character cap.

## Best Possible Solution

Resolve the projector cap through existing `contextLimits`, but track generated truncation state separately from tool text, add regression coverage for marker-containing output, and provide direct redacted proof of an oversized Codex tool result being capped in the native transcript.

## Reproduction Assessment

Yes, by source inspection: configure `agents.defaults.contextLimits.toolResultMaxChars` below 12,000 and send larger Codex app-server command output; current main still uses the projector's hardcoded 12,000-character cap for streamed and snapshot transcript text. I did not run a live WebChat turn in this read-only review.

## Solution Assessment

No, not as submitted. The PR targets the right owner path, but the sentinel-string shortcut must be replaced with explicit truncation state so arbitrary tool output cannot bypass the cap.
