## Best Possible Solution

Land the projector change so Codex app-server transcripts follow the existing context-limit contract, while keeping truncation metadata accurate for both snapshot and streamed fallback paths.

## Reproduction Assessment

Yes, source-reproducible: current `main` uses a fixed 12K projector cap while the existing config/docs expose `toolResultMaxChars` for persisted tool-result limits. The PR body also supplies a real local terminal proof of the after-fix app-server transcript path.

## Solution Assessment

Yes, with a minor correction. Resolving the existing context-limit setting inside the Codex app-server projector is the right owner-boundary fix; the streamed fallback notice should either track total seen characters or avoid claiming an exact original size.

- **[P3] Track full streamed output size before reporting it:** `extensions/codex/src/app-server/event-projector.ts:1959`
