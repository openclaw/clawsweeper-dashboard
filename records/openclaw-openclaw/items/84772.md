## Summary

Keep open: the PR addresses a real Codex transcript cap drift, but the proposed truncation-marker guard can let raw oversized tool output bypass the configured cap, and the supplied real-environment proof is indirect rather than showing the capped app-server/native transcript behavior.

## What This Changes

The PR updates Codex app-server transcript projection to use configured default or per-agent `toolResultMaxChars` for streamed and snapshot tool-result transcript text instead of a fixed 12K cap.

## Best Possible Solution

Land a corrected projector change that resolves the existing context-limit setting for Codex app-server transcripts while keeping truncation state separate from untrusted tool text, with focused regression coverage and direct redacted runtime proof of a capped transcript entry.

## Reproduction Assessment

Yes, source-reproducible rather than locally executed: current main uses a fixed 12K projector cap while the config schema already defines `toolResultMaxChars`, and the PR defect is visible from the new early marker check before length enforcement.

## Solution Assessment

No as written. Resolving the configured cap in the projector is the right owner-boundary direction, but raw tool output must not be able to spoof the projector's truncation notice and disable the cap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Enforce the cap before trusting the truncation marker:** `extensions/codex/src/app-server/event-projector.ts:2016-2017`
  - body: `params.text` can be raw command or tool output from a snapshot, so the output itself may contain `\n...(truncated:`. Because this guard returns before checking `text.length`, any oversized result containing that substring bypasses `toolResultMaxChars` and is mirrored into the app-server transcript unbounded, recreating the context-bloat path this PR is trying to close. Keep the streamed-output double-truncation state separate from raw text, or only skip the second notice in a way that still enforces the configured cap.
  - confidence: 0.89
