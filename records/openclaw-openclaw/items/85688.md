## Summary

Keep open: current main and v2026.6.8 still expose only partial session usage attribution, while the requested cron identity, session-kind, runtime, and explicit channel-target labeling remain a product/API contract decision.

## What This Changes

The issue requests additive stable JSON attribution fields on official usage/status surfaces for agent, session kind, cron job, channel, model, and runtime labels.

## Best Possible Solution

Define one additive, privacy-safe usage attribution contract that joins existing session, cron, channel, model, and runtime metadata into the official JSON surface without requiring dashboards to parse internal stores.

## Reproduction Assessment

Not applicable: this is a new public JSON attribution contract request, not a broken documented behavior. Source inspection shows the current surfaces are partial and lack the requested cron/session-kind/runtime fields.

## Solution Assessment

Unclear until maintainers choose the public JSON and privacy contract. The maintainable direction is additive core usage/status attribution backed by existing session and cron metadata, not downstream adapters over internal files.
