## Summary

Keep open: current `main` and the latest release already expose partial per-session attribution, but the central requested contract still remains missing for cron identity, session kind, runtime, and explicit channel-target labeling.

## What This Changes

The issue requests stable attribution fields in official usage/status JSON for agent, cron job, session kind, channel, model, and runtime labels.

## Best Possible Solution

Define an additive, privacy-safe usage attribution contract that joins existing session and cron metadata into `sessions.usage` and, if desired, exposes it through status JSON without requiring dashboards to parse cron/session stores directly.

## Reproduction Assessment

Not applicable: this is a new JSON contract request, not a report of broken documented behavior. Source inspection shows the current row output is only partially attributed and lacks the requested cron/session-kind/runtime fields.

## Solution Assessment

Unclear until maintainers choose the public JSON/privacy contract. The likely best path is additive core usage/status fields backed by existing session and cron stores, not downstream local adapters.
