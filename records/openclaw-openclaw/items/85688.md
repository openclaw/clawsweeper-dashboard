## Summary

Keep open. Current main partially exposes `agentId`, `channel`, `origin`, and model fields on `sessions.usage`, while `openclaw status --usage --json` is still a provider-usage snapshot; the requested stable session-kind, cron job, channel target, and runtime attribution fields are not implemented and need a small schema/privacy decision. The broader usage-ledger thread at https://github.com/openclaw/openclaw/issues/13219 is related but does not fully replace this narrower additive JSON-surface request.

## What This Changes

Request stable optional attribution labels in official usage/status JSON for agent, session kind, cron job identity, channel, model, and runtime.

## Best Possible Solution

Define an additive, privacy-safe attribution contract, likely starting with `sessions.usage`, for stable agent, session kind, cron job, channel, model, and runtime labels, then decide whether `status --usage --json` should link to or summarize that surface.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source and docs provide a high-confidence current-state check showing the requested stable cron/session-kind attribution fields are not present.

## Solution Assessment

Unclear for the exact API shape. An additive `sessions.usage` attribution contract looks like the narrowest maintainable first step, but maintainers should decide how it relates to `status --usage --json`, the open ledger issue, and privacy-safe channel target handling.
