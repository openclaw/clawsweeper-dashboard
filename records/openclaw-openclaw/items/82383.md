## Summary

Keep open: current main removes managed dreaming cron jobs when disabled only if a cron service is available, but the startup retry path is gated off for disabled configs, so a stale enabled job can survive a restart where cron is not yet exposed during `gateway_start`.

## What This Changes

The issue reports that changing `plugins.entries.memory-core.config.dreaming.enabled` to `false` leaves the managed `Memory Dreaming Promotion` cron job enabled after gateway restart.

## Best Possible Solution

Make memory-core reconcile disabled dreaming cleanup once the gateway cron service becomes available, preserving live-config reads and avoiding a broad new plugin API unless existing gateway lifecycle hooks are insufficient.

## Reproduction Assessment

Yes, at source level. Current main can reach `gateway_start` with `dreaming.enabled=false` and no cron service, return without removing the managed job, and skip the bounded startup retry because disabled configs clear the retry timer.

## Solution Assessment

Yes, the narrowest maintainable fix is in memory-core’s cron reconciliation lifecycle: retry or observe cron availability for disabled cleanup and add a regression test, rather than adding a broad config listener first.
