## Summary

Keep open: current main and the latest tagged release still have a source-reproducible cleanup gap when dreaming is disabled and the cron service is unavailable during `gateway_start`. The direct disabled reconciler removes managed jobs, but the startup retry path is skipped for disabled config, so an existing managed job can remain until a later heartbeat/cron runtime reconciliation happens.

## What This Changes

Disable or remove the memory-core managed dreaming cron job when `dreaming.enabled` is turned off across gateway restarts.

## Best Possible Solution

Make memory-core retry or defer one cleanup reconciliation until cron is available even when dreaming is disabled, without recreating jobs or broadening ordinary chat-triggered reconciliation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, as a source-level unit repro: start with an existing managed dreaming cron job, use disabled dreaming config, make `gateway_start` return no cron service, then make cron available and observe that no startup retry removes the job.

## Solution Assessment

Is this the best way to solve the issue? Yes: a narrow retry/cleanup fix in memory-core startup reconciliation is safer than adding a broader config-change listener or changing cron service behavior.
