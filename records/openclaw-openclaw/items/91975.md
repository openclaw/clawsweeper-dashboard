## Summary

Current `main` still drops the already-resolved model API before thinking-policy dispatch, so renamed `anthropic-messages` providers cannot select the Anthropic profile and may silently lose extended thinking levels; no linked or canonical fix PR currently owns the repair.

## What This Changes

Fix custom Anthropic-compatible provider ids losing Claude-specific thinking levels because provider-policy routing uses literal provider ownership instead of the configured wire protocol.

## Best Possible Solution

Carry the resolved model API into thinking-policy dispatch, try exact provider ownership before API-owner policy, and cover both explicit and inherited thinking levels for renamed Anthropic providers.

## Solution Assessment

Yes—the narrowest maintainable repair is ordered exact-provider then API-owner policy routing, reusing the existing hook-ref contract rather than matching provider-name substrings.
