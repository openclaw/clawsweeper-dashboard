## Label Justifications

- `P2`: This is a confirmed, bounded provider-routing bug that silently disables a supported model capability without making the core runtime unusable.
- `impact:auth-provider`: Provider identity and model capability routing incorrectly determine whether the requested Claude thinking level is available.

## Summary

Current `main` still routes Claude thinking profiles through exact provider ownership, so custom provider ids fall back to the generic profile and can lose `xhigh`; the report is valid, bounded, and not covered by a linked fix.

## What This Changes

Preserve Claude extended-thinking profiles for custom provider ids using the native Anthropic adapter instead of silently reducing unsupported requested levels to `off`.

## Best Possible Solution

Resolve the Claude profile when both the configured transport is `anthropic-messages` and the canonical model identity is Claude, while retaining provider-specific exceptions such as Claude CLI Fable handling and leaving non-Claude Anthropic-wire providers unchanged.

## Reproduction Assessment

Yes, at high confidence from current source: an arbitrary custom provider id cannot select the Anthropic policy, and the resulting generic profile lacks Claude’s extended thinking levels. The review did not execute the reporter’s external provider endpoint, so this is source-reproducible rather than a live reproduced run.

## Solution Assessment

The exact-id diagnosis is correct, but provider-name substring matching is not the best solution. The narrow maintainable repair is to combine the existing adapter fact with canonical Claude model identification so other `anthropic-messages` providers are unaffected.
