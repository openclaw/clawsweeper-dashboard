## Summary

Keep open. Current main still has the same timer-only deferred lifecycle-error path, so a later non-error event for the same run can cancel the pending terminal cleanup before the session state is finalized, matching the reported stuck-`running` timeout failure.

## What This Changes

The issue asks OpenClaw to recover sessions that remain persisted as `running` after request timeouts in the Telegram plus local llama.cpp path despite the prior abort cleanup fix.

## Best Possible Solution

Preserve deferred terminal lifecycle-error state through the retry grace window and finalize it when a later non-retry event would otherwise clear it, with focused regression coverage for both fallback success and no-terminal-end failure.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: lifecycle `start` persists `running`, lifecycle `error` schedules deferred cleanup, then a later non-error event with the same runId clears the only pending terminal cleanup before the grace timer fires.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the generic gateway lifecycle-error handler with regression coverage, not in Telegram, llama.cpp, provider policy, or broad session reset behavior.
