## Summary

Keep open. Current main still has the timer-only deferred lifecycle-error path: a later non-error event for the same run can clear the pending terminal cleanup without retaining or finalizing the lifecycle error, so the reported stuck-running session state remains source-reproducible.

## What This Changes

Timeout-driven lifecycle errors can leave gateway sessions stuck in running status after terminal cleanup is discarded.

## Best Possible Solution

Preserve deferred terminal lifecycle-error state through the grace window and finalize it when a non-retry cleanup path would otherwise drop it, while keeping retry/fallback cancellation behavior intact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: lifecycle start persists `running`, lifecycle error schedules terminal cleanup, then a later non-error event for the same run clears the only pending cleanup before the grace timer fires and no lifecycle end is required.

## Solution Assessment

Yes, with a narrower version than the reporter's dist patch. The maintainable fix belongs in the generic gateway lifecycle-error handler with regression coverage, not in Telegram, llama.cpp, provider-specific policy, or broad session reset behavior.

## Repair Work Prompt

Fix the stuck-running lifecycle cleanup bug reported in https://github.com/openclaw/openclaw/issues/63819. In `src/gateway/server-chat.ts`, `pendingTerminalLifecycleErrors` currently stores only a timer; when a lifecycle error is deferred and a later non-error event for the same run arrives before the grace timer, `clearPendingTerminalLifecycleError` drops the only terminal cleanup path. Retain enough pending lifecycle error state to finalize terminal cleanup when a non-retry event would otherwise discard it, while preserving existing retry/fallback behavior and `chat.send` `skipChatErrorFinal` semantics. Add focused coverage in `src/gateway/server-chat.agent-events.test.ts` for lifecycle start -> lifecycle error scheduled -> later assistant/tool/item event before grace -> no lifecycle end, proving context/sequence cleanup and persisted terminal lifecycle handling. Do not add Telegram, llama.cpp, provider-specific, sessions.abort, restart-reconciliation, or configurable watchdog behavior unless the narrow fix requires it. Add a `CHANGELOG.md` fix entry because this is user-visible.
