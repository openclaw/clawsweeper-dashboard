## Summary

Keep open. Current main still matches the reporter's timer-only deferred lifecycle-error path: a later non-error event can clear the pending terminal lifecycle cleanup without retaining or finalizing the event, and v2026.4.29 has the same code. This is still a narrow gateway lifecycle bug rather than a stale support report or a duplicate of the broader stuck-session tracker.

## What This Changes

The issue reports that after request timeouts in v2026.4.9, gateway sessions remain in `running` because the prior phase-before-abort cleanup no longer releases the session state.

## Best Possible Solution

Keep this issue open and land a focused gateway fix that preserves deferred terminal lifecycle-error state long enough to either intentionally discard it for a retry/fallback replacement or finalize it when a later event would otherwise drop the only cleanup path.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a gateway event sequence: lifecycle start, lifecycle error scheduled under the grace window, then a later non-error event for the same run before the timer fires and no lifecycle end; current main clears the pending timer and loses terminal finalization.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `server-chat` lifecycle-error handling plus a regression test; it should stay generic and avoid Telegram, llama.cpp, or provider-specific policy.
