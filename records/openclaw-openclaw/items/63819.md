## Summary

Keep open. The checked-out main still has the timer-only pending terminal lifecycle-error path described by the reporter's workaround, the latest local release tag has the same behavior, and no direct open PR was found for this issue. This is narrow enough for a focused repair lane.

## What This Changes

The issue reports that Telegram sessions remain persisted as `running` after local llama.cpp request timeouts in v2026.4.9, blocking later messages until watchdog recovery.

## Best Possible Solution

Add a focused gateway regression test for a deferred lifecycle error followed by a later non-error event with no lifecycle end, then update `server-chat` so pending terminal lifecycle-error handling retains the event/options and finalizes it when clearing is not part of an intentional retry or fallback replacement.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is available in `src/gateway/server-chat.agent-events.test.ts`: emit lifecycle start/error, then a later non-error agent event before the grace timer and assert terminal lifecycle persistence/cleanup instead of leaving the session running.

## Solution Assessment

Yes. The narrowest maintainable fix is to keep the existing gateway lifecycle grace contract, but store enough deferred terminal event state to finalize cleanup when a non-retry event would otherwise drop it; no Telegram, provider, or llama.cpp special case is needed.
