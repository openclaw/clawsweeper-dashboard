## Summary

Keep open. Current main still has the timer-only deferred lifecycle-error path, so a later non-error event for the same run can clear the pending terminal cleanup before the session state is finalized, matching the reported stuck-`running` failure mode.

## What This Changes

The issue reports a regression where gateway sessions remain `running` after request timeouts in v2026.4.9 even after the earlier phase-before-abort fix.

## Best Possible Solution

Preserve deferred terminal lifecycle-error state through the retry grace window and finalize it when a later non-retry event would otherwise drop it, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: lifecycle `start` persists `running`, lifecycle `error` schedules deferred cleanup, then a later non-error event for the same run clears the only pending terminal cleanup before the grace timer fires.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the generic gateway lifecycle-error handler with regression coverage, not in Telegram, llama.cpp, provider policy, or broad session reset behavior.
