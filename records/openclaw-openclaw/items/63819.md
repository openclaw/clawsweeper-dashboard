## Summary

Keep open. Current main still has the timer-only deferred lifecycle-error path described by the reporter: a later non-error event for the same run can clear the pending terminal cleanup without retaining or finalizing the lifecycle error, leaving persisted session state vulnerable to staying `running`.

## What This Changes

The issue reports gateway sessions staying in `running` after request timeouts in v2026.4.9, despite the earlier phase-before-abort cleanup fix.

## Best Possible Solution

Retain pending terminal lifecycle-error event/options through the grace window and finalize them when a later non-retry event would otherwise drop the only cleanup path, with focused gateway regression coverage.

## Reproduction Assessment

Yes. The high-confidence reproduction path is lifecycle start, lifecycle error scheduled under retry grace, then a later non-error event for the same run before the timer fires and no lifecycle end; current main clears the pending timer and loses terminal finalization.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `server-chat` lifecycle-error handling plus a regression test, and should stay generic rather than adding Telegram, llama.cpp, or provider-specific policy.
