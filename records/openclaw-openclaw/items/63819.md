## Summary

Keep open. Current main and the latest shipped tag still have the source-level lifecycle timer gap: terminal lifecycle errors are deferred as a timer-only entry, and a later non-error event for the same run can clear that timer before terminal session state is persisted. The reporter's workaround matches that path, while the open broader stale-row PR targets a different recovery seam and does not close this issue.

## What This Changes

The issue requests a fix for timeout/error cleanup leaving gateway sessions persisted as running and blocking later channel messages.

## Best Possible Solution

Store the pending terminal lifecycle error as structured timer, event, and option state, then finalize or retain terminal cleanup when a later non-retry event would otherwise delete it, with regression coverage for the exact event ordering.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: lifecycle start persists running, lifecycle error schedules deferred terminal cleanup, and a later non-error event for the same run clears the only pending timer before terminal state is persisted.

## Solution Assessment

Yes. The narrow maintainable fix belongs in src/gateway/server-chat.ts lifecycle-error cleanup with focused tests, not in Telegram, llama.cpp, provider policy, cron reaping, or broad stale-row recovery.
