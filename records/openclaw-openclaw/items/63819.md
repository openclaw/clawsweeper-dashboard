## Summary

Keep open. Current main still has the timer-only deferred lifecycle-error cleanup path, so a later non-error event for the same run can cancel terminal cleanup before the session row is finalized, matching the reported stuck-running timeout behavior.

## What This Changes

The issue reports OpenClaw 2026.4.9 leaving sessions in `running` after request timeouts because the earlier abort cleanup fix no longer covers the gateway lifecycle-error path.

## Best Possible Solution

Preserve deferred terminal lifecycle-error state through the retry grace window and finalize it when a later non-retry event would otherwise clear it, with regression coverage for both fallback success and terminal error followed by non-error events.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: lifecycle `start` persists `running`, lifecycle `error` schedules deferred terminal cleanup, and a later non-error event for the same run clears the only pending timer before terminal state is persisted.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/gateway/server-chat.ts` lifecycle-error cleanup with focused tests, not in Telegram, llama.cpp, provider policy, or broad diagnostic recovery.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/63819. On current main, `src/gateway/server-chat.ts` stores pending terminal lifecycle errors only as timers and clears them on any later non-error event for the same run, which can drop terminal cleanup before the session row leaves `running`. Preserve the pending terminal lifecycle event and options through the retry grace window, and finalize it when a non-retry event would otherwise clear that pending terminal cleanup. Keep the fallback behavior from https://github.com/openclaw/openclaw/pull/60043: a later lifecycle `end` or fallback success must still supersede a per-attempt lifecycle `error`. Do not add config, Telegram-specific handling, provider policy, or broad stale-session recovery for https://github.com/openclaw/openclaw/issues/59878 or https://github.com/openclaw/openclaw/issues/71127. Add focused regression coverage in `src/gateway/server-chat.agent-events.test.ts` for lifecycle error followed by a later non-error assistant/tool/item event with no lifecycle end, and keep the existing fallback-success tests passing. Add a user-facing changelog entry if repository policy requires it for this fix.
