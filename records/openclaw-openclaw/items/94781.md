## Summary

Keep open. The code change is a reasonable Copilot registration fallback and current main/release still have the failing dereference, but the supplied proof is still mock-only and the session-persistence downgrade needs maintainer acceptance before merge.

## What This Changes

The PR changes `extensions/copilot/index.ts` to guard `api.runtime.state?.openSyncKeyedStore` and omit `sessionStore` when the state API is unavailable.

## Best Possible Solution

Land one canonical Copilot fix that preserves the durable state-backed path, adds missing-state regression coverage, and includes redacted installed or reporter-equivalent runtime proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: the linked issue reports a shipped `v2026.6.8` failure and source shows the dereference will throw if `runtime.state` is undefined, but I did not reproduce the installed host path that omits `state`.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the plugin-local fallback is a reasonable mitigation because the harness already treats `sessionStore` as optional, but maintainers should confirm fail-open registration is preferred over making the host always provide `runtime.state`.
