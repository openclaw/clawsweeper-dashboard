## Summary

Keep open: current main still lacks the proposed `injectAgentHeader` feature, but the PR has a concrete correctness bug, missing real behavior proof, and an unresolved product/security decision about this narrow flag versus the broader provider-header design.

## What This Changes

The PR adds an opt-in `models.providers.*.injectAgentHeader` config field, schema/help metadata, a stream wrapper, and tests to send `X-OpenClaw-Agent: <agentId>` on matching LLM provider requests.

## Best Possible Solution

Land a narrow opt-in implementation only after it reuses normalized provider config resolution, documents the trusted-endpoint disclosure boundary, and includes real request-path proof; alternatively fold this into the broader provider-header design before closing this PR.

## Reproduction Assessment

Yes for the PR defect by source inspection: current model resolution accepts normalized provider aliases, while the PR's direct provider-key lookup would skip `injectAgentHeader` for an alias-matched provider. This is a proposed-patch failure path, not a current-main bug reproduction.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation should share normalized provider lookup and document the trusted-endpoint disclosure boundary before maintainers decide whether to land this narrow flag or the broader provider-header surface.
