## Summary

Keep open. Current main does not implement the requested provider config flag, and the PR is a useful narrow implementation, but it still needs normalized provider-config lookup, disclosure wording, changelog coverage, and generated config-surface updates before merge.

## What This Changes

The PR adds an opt-in `injectAgentHeader` model-provider config flag, schema/help/label entries, request-stream header injection, test support, and focused tests for `X-OpenClaw-Agent` forwarding.

## Best Possible Solution

Land the feature as a small opt-in provider setting that reuses normalized provider lookup, documents trusted-provider disclosure, updates generated config artifacts and changelog, and keeps the broader OpenRouter `X-Title` request tracked separately.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure the provider as `z-ai` with `injectAgentHeader: true` while runtime uses `z.ai`; current model resolution can find that provider config through normalization, but the PR's direct lookup would skip the header.

## Solution Assessment

No. The feature shape is close, but the mergeable path should reuse the existing normalized provider-config contract and include the required config-surface, disclosure, test, and changelog follow-through.
