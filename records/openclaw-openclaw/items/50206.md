## Summary

Keep open. Current main still does not implement dynamic per-agent LLM request headers, and this PR remains a plausible small implementation, but the branch should not merge as-is because the proposed provider-config lookup bypasses existing normalized provider matching and the user-facing disclosure/changelog work is incomplete.

## What This Changes

The PR adds an `injectAgentHeader` model-provider config flag, schema/help/label entries, a runner stream wrapper, test-support plumbing, and tests to send `X-OpenClaw-Agent` with the active agent ID on outgoing model requests.

## Best Possible Solution

Land a narrow opt-in implementation that resolves provider config through the same normalized provider path as model resolution, adds alias/case regression coverage, documents the trusted-provider disclosure clearly, and records the user-facing feature in the changelog. Do not broaden this into default header injection or the separate OpenRouter `X-Title` request tracked by #48379.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure the provider under a normalized alias such as `z-ai` while runtime uses `z.ai`; current model resolution can find that config, but the PR's direct `providers[provider]` lookup would not enable the header.

## Solution Assessment

No. The feature shape is close, but the proposed lookup is not the narrowest reliable solution because it diverges from existing normalized provider resolution; the safer mergeable path is to reuse that lookup and add explicit trusted-provider documentation plus changelog coverage.
