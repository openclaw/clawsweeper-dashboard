## Summary

Keep open. Current main still lacks the opt-in per-agent LLM header feature, and the PR is a plausible small implementation, but it should not merge as-is because the provider-config lookup bypasses existing normalized provider matching and the disclosure/changelog work is incomplete.

## What This Changes

The PR adds an `injectAgentHeader` provider config flag, schema/help/label entries, runner stream wrapping, test support, and tests for sending `X-OpenClaw-Agent` with the active agent ID.

## Best Possible Solution

Land a narrow opt-in implementation that reuses normalized provider config lookup, adds alias regression coverage, documents trusted-provider disclosure, and records the user-facing feature in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure the provider under a normalized alias such as `z-ai` while runtime uses `z.ai`; current model resolution can find that config, but the PR's direct lookup would not enable the header.

## Solution Assessment

No. The feature shape is close, but the proposed lookup diverges from existing normalized provider resolution; the safer mergeable path is to reuse that lookup and add explicit trusted-provider documentation plus changelog coverage.
