## Label Justifications

- `P1`: The PR targets a broken agent/channel workflow where generated Discord channel replies can be silently dropped before reaching users.
- `merge-risk: 🚨 message-delivery`: The diff changes whether channel replies are suppressed by the announce decider or delivered directly, which can affect visible message delivery.

## Summary

Keep open: the patch appears focused and supported by live Discord proof, but merging it changes shared channel-delivery semantics by bypassing the announce decider for same-session same-channel replies, so this should remain a maintainer merge decision.

## What This Changes

The PR extracts announce reply sending into a helper and sends same-session same-channel channel replies directly while keeping different-channel and cross-session flows on the announce-decider path.

## Best Possible Solution

Land the shared direct-delivery branch only after maintainers accept same-session same-channel source replies as a direct user-facing reply boundary, while preserving the decider for different-channel and cross-session announcements.

## Reproduction Assessment

Yes from source and report evidence, though I did not rerun the live scenario: current main and v2026.5.27 still run the announce decider before delivery and suppress ANNOUNCE_SKIP, matching the linked Discord message-loss report at https://github.com/openclaw/openclaw/issues/87157.

## Solution Assessment

Yes if maintainers accept the generic boundary: the PR uses the existing announce target, limits the bypass to same-session same-channel sends, and covers direct delivery plus negative cases. If maintainers want a Discord-only policy, the guard should be narrowed before merge.
