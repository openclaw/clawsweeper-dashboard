## Summary

Keep open. Current main still lacks the session-action SDK/Gateway protocol, so this draft PR is the active review surface rather than obsolete cleanup; however, the latest head still exposes a security-sensitive public event-emission path that lets non-bundled plugins emit host-owned run streams that Gateway clients treat as trusted agent output.

## What This Changes

The branch adds public Plugin SDK session-action and plugin event-emission APIs, registry storage and validation, Gateway `plugins.sessionAction` dispatch with scoped authorization, protocol and Swift model updates, focused tests, and a changelog entry.

## Best Possible Solution

Keep the generic `plugins.sessionAction` Gateway seam, but tighten `emitAgentEvent` before merge so external plugins can only emit plugin-namespaced/custom workflow streams while built-in host streams stay bundled/trusted; add regression coverage for blocked host streams and then run the API, protocol, lint, and targeted contract gates on the exact head.

## Reproduction Assessment

Not applicable as a feature PR, but yes for the blocking review finding: on PR head, a workspace plugin can call the new public `emitAgentEvent` with `stream: "assistant"` and `data.text`, and current Gateway code consumes assistant-stream text as live assistant output.

## Solution Assessment

No as-is. The session-action design is a reasonable narrow seam, but the event-emission addition should reserve all built-in host streams for bundled/trusted emitters or require external plugins to use plugin-owned stream names.
