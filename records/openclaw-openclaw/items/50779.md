## Summary

Keep open. Current main still loses the direct read-tool image result shape before Control UI can show a preview: tool results can carry `{ type: "image", data, mimeType }`, but the UI image extractor only handles `source`/`url` image blocks and live tool events flatten sanitized results to text. Open PR #52025 targets this bug, but it remains open and is an implementation candidate rather than proof that the issue is fixed.

## What This Changes

The issue requests inline rendering or an obvious preview for images returned by tools in Control UI chat.

## Best Possible Solution

Control UI should preserve structured image data from stored history and live tool-result updates, render direct `{ type: "image", data, mimeType }` blocks as safe inline previews or obvious thumbnails, and keep managed-media/local-media security checks covered by focused regression tests.

## Reproduction Assessment

Yes, by source-level reproduction. A tool-result message whose content includes `{ type: "image", data: "...", mimeType: "image/png" }` follows the current read-tool contract, while the current Control UI extraction and live stream paths do not surface that image as a visible preview.

## Solution Assessment

Unclear whether the open PR is the best implementation until it is reviewed. The maintainable direction is a narrow Control UI/tool-stream fix with regression coverage, not a broad remote-media or all-media rendering expansion.
