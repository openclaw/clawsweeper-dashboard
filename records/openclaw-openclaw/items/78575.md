## Summary

Keep open: this addresses a valid Telegram outbound bug, but the PR still has a compatibility blocker around the Telegram topic-field contract plus a missing changelog entry and insufficient topic-placement proof.

## What This Changes

The PR changes Telegram topic-scoped DM sends, replies, media, and draft previews to use `direct_messages_topic_id` instead of `message_thread_id`, stops the DM threadless retry, updates config help copy, and adjusts Telegram tests.

## Best Possible Solution

Land a Telegram plugin fix that distinguishes direct-message chats from private bot topic mode or uses a guarded compatibility retry between the two topic fields, with focused regression coverage, a changelog entry, and redacted live proof that the message lands in the requested topic.

## Reproduction Assessment

Yes, from source inspection and linked live evidence: current main builds DM topic sends with `message_thread_id` and falls back without a topic after `message thread not found`, matching https://github.com/openclaw/openclaw/issues/79455. I did not run a tokened Telegram repro from this checkout.

## Solution Assessment

No for the PR as written: the direction is plausible for affected direct-message chats, but the unconditional `scope: "dm"` field swap is wider than the Telegram and grammY contract supports. A compatibility-aware guard or retry path plus clearer proof and changelog is safer.
