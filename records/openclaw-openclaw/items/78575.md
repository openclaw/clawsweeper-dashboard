## Summary

Keep open: the PR targets a real Telegram outbound bug, but it is not merge-ready because the field swap is broader than the Telegram type contract supports, after-fix real behavior proof is missing, and the required changelog entry is absent.

## What This Changes

The PR changes Telegram outbound topic handling so private/direct DM topic sends use `direct_messages_topic_id`, preserves forum/group sends on `message_thread_id`, and updates related retry behavior and tests.

## Best Possible Solution

Land a compatibility-aware Telegram plugin fix that uses `direct_messages_topic_id` for direct-message topics without regressing private bot topic sends that still require `message_thread_id`, with focused regression coverage, a changelog entry, and redacted live after-fix proof.

## Reproduction Assessment

Yes from source inspection and linked live evidence: current main builds DM topic outbound params with `message_thread_id` and then falls back to a bare DM after `message thread not found`. I did not run a tokened live Telegram repro from this checkout.

## Solution Assessment

No for the PR as written: the direction is right for affected direct-message topics, but the unconditional `scope: "dm"` replacement is wider than the Telegram type contract supports. A guarded compatibility path plus proof and changelog would be safer.
