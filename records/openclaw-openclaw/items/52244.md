## Summary

Keep open. The dynamic OpenRouter reasoning bug class is still plausible, but this branch is not currently correct: it can infer thinking while leaving `reasoningMode` off, and fresh-cache catalog augmentation can still be empty before dynamic model preparation. The PR also needs real behavior proof before merge.

## What This Changes

This PR adds OpenRouter catalog augmentation and default-thinking fallback from cached OpenRouter capabilities, exports an all-cached OpenRouter capability accessor, and reformats one daemon lifecycle test mock.

## Best Possible Solution

Rework the PR around the current provider catalog/profile architecture so dynamic OpenRouter capability metadata is available before default resolution, native reasoning visibility is preserved when thinking is inferred, and any SDK export is avoided or explicitly approved and documented.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the dynamic-model class: current defaults can still fall through for reasoning-capable OpenRouter models absent from the catalog/cache, but I did not run a live OpenRouter session and the exact Kimi K2.5 example is now static on main.

## Solution Assessment

Is this the best way to solve the issue? No. The PR uses nonblocking cache reads and infers thinking without enabling reasoning visibility; a safer solution should use the current catalog/provider-profile surfaces and preserve `reasoningMode` when native thinking must be shown.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Keep native reasoning enabled when inferring thinking:** `extensions/openrouter/index.ts:169-177`
  - body: Returning `"low"` here makes `thinkingActive` true, and the reply path only calls the model reasoning default while thinking is inactive. Since the embedded subscriber extracts OpenRouter native thinking only when `reasoningMode` is `on` or `stream`, dynamic reasoning models can still drop `reasoning_content` instead of displaying it.
  - confidence: 0.86
- **[P2] Wait for the OpenRouter cache before catalog augmentation:** `src/agents/pi-embedded-runner/openrouter-model-capabilities.ts:307-309`
  - body: `getAllCachedOpenRouterModels()` only kicks the nonblocking cache initializer and immediately returns an empty map when there is no disk cache yet. On a fresh setup, default resolution can run before `prepareDynamicModel()` awaits the OpenRouter model fetch, so the supplemental catalog stays empty and the first dynamic reasoning model still defaults off.
  - confidence: 0.79
