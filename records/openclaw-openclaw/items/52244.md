## Summary

Keep open: current main only supersedes the PR body's Kimi K2.5 example by adding it to the static OpenRouter catalog, but the broader dynamic-model reasoning default bug is not implemented on main and this PR still has concrete correctness blockers plus missing real behavior proof.

## What This Changes

The PR adds OpenRouter catalog augmentation and default-thinking fallback from cached dynamic model capabilities, exports an all-cached OpenRouter capability accessor, and reformats one daemon lifecycle test mock.

## Best Possible Solution

Revise and rebase the OpenRouter-owned fix so dynamic capability metadata is available before thinking and reasoning defaults resolve, native reasoning remains visible by default, and any public SDK/cache seam is intentionally documented and covered.

## Reproduction Assessment

Yes, from source inspection. A reasoning-capable OpenRouter model absent from the static catalog with no explicit `/think` or `/reasoning` can reach default resolution before awaited dynamic capability loading, and native thinking is only surfaced when `reasoningMode` is `on` or `stream`.

## Solution Assessment

No. The provider-owned direction is plausible, but this patch can still keep `reasoningMode` off after inferring thinking and can read an empty supplemental catalog on a fresh cache before the OpenRouter fetch is awaited.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Preserve reasoning visibility when thinking is inferred:** `extensions/openrouter/index.ts:175-178`
  - body: Returning `"low"` for reasoning-capable OpenRouter models makes reply defaults treat thinking as active before reasoning defaults are resolved. `get-reply-directives.ts` then skips `resolveDefaultReasoningLevel()`, while the embedded subscriber only extracts native thinking when `reasoningMode` is `on` or `stream`, so `reasoning_content` can still be dropped unless the user explicitly sets `/reasoning`.
  - confidence: 0.83
- **[P2] Await the OpenRouter catalog before using it:** `src/agents/pi-embedded-runner/openrouter-model-capabilities.ts:310-311`
  - body: `getAllCachedOpenRouterModels()` calls the nonblocking `ensureOpenRouterModelCache()` path and immediately returns `cache ?? new Map()`. On a fresh cache this starts the OpenRouter fetch but returns an empty map, so the supplemental catalog can still be missing when reply defaults run before `prepareDynamicModel()` awaits the per-model load.
  - confidence: 0.8
