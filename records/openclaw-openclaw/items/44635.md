## Summary

Keep this issue open. Current main at 1b25dcf has named browser profiles, `browser.defaultProfile`, per-profile CDP/executable/headless settings, and shipped `browser.profiles.<name>.userDataDir` for Chrome MCP `existing-session` attachment, but it still does not implement the requested agent-scoped browser configuration or per-agent default browser-profile binding. The proposed `agents.list[].browser.userDataDir` shape remains outside the strict agent schema, managed browser storage is still derived from browser profile name, and the browser tool only uses an explicit tool-call `profile` or the global browser default.

## Best Possible Solution

Keep this open for a maintainer-reviewed browser/core configuration design. The best path is an agent-scoped browser policy contract, likely under `agents.defaults.browser` and `agents.list[].browser`, that can bind an agent to a named browser profile or policy. The browser plugin should receive the active agent id and derive the default profile from that context when the tool call omits `profile`; any managed custom `userDataDir` support should be schema-backed, documented, tested, and safe around profile reset/delete behavior. Coordinate with broader per-agent browser isolation/proxy work in #37487 rather than treating existing-session `userDataDir` as complete coverage.

## Risks / Open Questions

- Closing this now would hide a real privacy/security boundary: agents can still share the same browser profile unless callers manually choose different profiles or change the global default.
- The existing `browser.profiles.<name>.userDataDir` support may look like a fix, but it is limited to Chrome MCP existing-session attachment and does not provide managed per-agent storage isolation.
- A future implementation needs careful reset/delete semantics if custom managed `userDataDir` paths are allowed, so OpenClaw does not trash user-owned browser data unexpectedly.
