## Summary

Keep open. Current main has global named browser profiles and shipped `browser.profiles.<name>.userDataDir` for Chrome MCP existing-session attachment, but it still lacks agent-scoped browser configuration or an agent-to-default-browser-profile binding. The reported `agents.list[].browser.userDataDir` shape remains outside the strict agent schema, managed browser storage is still derived from browser profile name, and the browser tool falls back to the global browser default unless the tool call explicitly supplies `profile`.

## What This Changes

The issue requests agent-scoped browser profile isolation so different agents can use independent browser storage such as cookies, localStorage, CDP endpoints, and related browser identity state.

## Best Possible Solution

Keep this open as a browser/core configuration feature. The maintainable end state is a schema-backed agent browser policy, likely under `agents.defaults.browser` and `agents.list[].browser`, that binds an agent to a named browser profile or browser policy; the browser plugin should receive trusted agent context and choose that default when the tool call omits `profile`, while managed `userDataDir`, CDP URL, reset/delete, docs, and tests stay aligned with the profile trust model.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: adding `browser` under an `agents.list[]` entry still does not match `AgentConfig` or `AgentEntrySchema`, and the strict schema rejects unrecognized per-agent keys; separately, omitted browser-tool `profile` still resolves through the global browser default.

## Solution Assessment

Unclear for the exact requested shape. Directly supporting `agents.list[].browser.userDataDir` would answer the report, but the safer design is probably an agent-to-named-profile or agent-to-browser-policy binding that reuses the existing browser profile model and handles managed storage, existing-session, CDP URLs, and reset semantics deliberately.
