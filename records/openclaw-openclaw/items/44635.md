## Summary

Keep open. Current main still supports named browser profiles and `browser.profiles.<name>.userDataDir` for existing-session Chrome MCP attachment, but it does not support per-agent browser config or an agent-to-default-browser-profile binding; #37487 overlaps, but it is broader and not clearly a canonical replacement for this focused userDataDir report and its useful discussion.

## What This Changes

The issue requests first-class per-agent browser profile isolation so different agents can use independent cookies, storage, CDP endpoints, or browser identity state.

## Best Possible Solution

Keep this open as a browser/core configuration feature. The maintainable end state is a schema-backed agent browser policy, likely under `agents.defaults.browser` and `agents.list[].browser`, that binds an agent to a named browser profile or browser policy while keeping managed storage, existing-session `userDataDir`, CDP URL isolation, scoped flags, reset/delete behavior, docs, and tests aligned with the browser profile trust model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to add `browser` under an `agents.list[]` entry and validate against `AgentEntrySchema`, or to omit `profile` from a browser tool call and observe fallback to the global `browser.defaultProfile`.

## Solution Assessment

Unclear for the exact requested shape. Direct `agents.list[].browser.userDataDir` support would answer the example, but an agent-to-named-profile or agent-to-browser-policy binding is likely the safer maintainable solution because it reuses existing profile semantics and can account for managed storage, existing-session attachment, CDP URLs, flags, and reset behavior.
