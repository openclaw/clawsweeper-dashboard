## Summary

Keep open. Current main supports global named browser profiles and `browser.profiles.<name>.userDataDir` for existing-session attachment, but it still lacks an agent-scoped browser config surface or an agent-to-default-browser-profile binding, so the reported per-agent isolation request remains unresolved.

## What This Changes

The issue requests per-agent browser profile isolation so different agents can use separate browser storage, identity, CDP endpoints, and related launch policy.

## Best Possible Solution

Keep this open as a browser/core configuration feature. The maintainable end state is a schema-backed agent browser policy, likely under `agents.defaults.browser` and `agents.list[].browser`, that binds an agent to a named browser profile or browser policy; the browser tool should receive trusted agent context and choose that default when the tool call omits `profile`, with docs/tests covering managed `userDataDir`, CDP URL isolation, scoped flags, reset/delete behavior, and parallel-launch safety.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the requested `agents.list[].browser.userDataDir` shape is absent from `AgentConfig` and rejected by the strict agent schema, while omitted browser-tool `profile` still resolves through the global browser default.

## Solution Assessment

Unclear for the exact requested config shape. Directly supporting `agents.list[].browser.userDataDir` would answer the initial report, but a safer maintainable solution is probably an agent-to-named-profile or agent-to-browser-policy binding that reuses the existing browser profile model and handles managed storage, existing-session, CDP URLs, flags, and reset semantics deliberately.
