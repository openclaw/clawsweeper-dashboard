## Summary

Close as implemented on current main: the released 2026.5.2 CLI approve path used the pairing method scope, while current main now inspects the pending request and requests `operator.admin` when approval would mint or inherit admin-scoped operator access; the gateway ownership-mismatch guard only applies to non-admin device-token sessions.

## What This Changes

Allow CLI device-pairing approval after upgrade when a pending operator/admin browser request must be approved from existing admin credentials.

## Best Possible Solution

Keep the current least-privilege CLI scope selection, ship it in the next release, and track any remaining upgrade-time config mutation separately with before/after `openclaw.json` proof.

## Reproduction Assessment

No for current main: the failing 2026.5.2 source path is clear, but current main now requests `operator.admin` for admin approvals and has regression tests for that behavior.

## Solution Assessment

Yes: the main-branch fix is narrow because it requests admin scope only when the pending approval requires or inherits admin access, while lower-risk approvals remain pairing-scoped.
