## Summary

Keep open. Current main still lacks the requested config surface and still widens bundled provider/runtime activation outside a restrictive plugin allowlist; the open closing PR #76085 now targets this exact gap, so the issue should stay open until that PR lands or is closed.

## What This Changes

Add a default-preserving `plugins.bundledMode: "respect-allow"` option so bundled runtime provider, web-search, and web-fetch paths can honor `plugins.allow` and `plugins.entries.<id>.enabled`.

## Best Possible Solution

Review and land #76085 or an equivalent core patch that adds the config contract, preserves default compat behavior, and applies respect-allow semantics consistently across provider setup/runtime and web-provider paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists with a restrictive `plugins.allow`: current main still discovers bundled provider-surface plugins during setup and widens activation through `withActivatedPluginIds`, matching the K8s runtime report.

## Solution Assessment

Yes for the near-term direction. A default-preserving opt-in is narrower than changing bundled plugin semantics globally, while larger manifest-first runtime-loader cleanup can remain separate follow-up work.
