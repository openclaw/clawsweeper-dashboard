## Summary

Keep open. Current main still lacks the requested `plugins.bundledMode` config surface and the named runtime provider paths still apply bundled compatibility overrides, while open PR #75644 is already the active closing implementation candidate.

## What This Changes

The issue requests an opt-in config mode that makes bundled runtime provider loading respect `plugins.allow` and `plugins.entries.<id>.enabled` without changing the default compatibility behavior.

## Best Possible Solution

Land a reviewed implementation through PR #75644 or an equivalent narrow PR that adds the config/schema/help contract, wires `respect-allow` through the web-search, web-fetch, and LLM provider runtime paths, enables the intended runtime loader cache behavior, and covers both default compatibility and opt-in allowlist behavior with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a restrictive `plugins.allow` and trace the current runtime provider paths, which still inject bundled compatibility IDs while the strict config schema has no `plugins.bundledMode` opt-in.

## Solution Assessment

Yes for the near-term direction. A default-preserving opt-in is narrower than changing bundled allowlist semantics globally, while the larger manifest-only runtime-loader cleanup can remain separate follow-up work.
