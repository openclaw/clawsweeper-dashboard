## Summary

Keep open. Current main still lacks the requested opt-in config surface and the setup/runtime provider paths can still widen a restrictive `plugins.allow`; the linked implementation PR remains open with closing syntax, so this issue should stay open until that or an equivalent fix lands.

## What This Changes

Request adds an opt-in `plugins.bundledMode: "respect-allow"` mode so bundled runtime provider, web-search, and web-fetch paths honor `plugins.allow` and per-plugin disabled entries.

## Best Possible Solution

Review and land #76085 or an equivalent default-preserving change that adds the config/schema/help/tests and gates bundled runtime provider discovery plus activation by allowlist in the new opt-in mode.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: restrictive `plugins.allow` can still be widened by setup provider discovery plus `withActivatedPluginIds`, and the requested opt-in config key is absent from the strict schema.

## Solution Assessment

Yes for the near-term direction. A default-preserving opt-in mode is narrower than globally changing bundled allowlist semantics, while the broader manifest-first runtime-loader cleanup can remain follow-up work.
