## Summary

Keep open. Current main still lacks the requested `plugins.bundledMode` config surface and still has runtime/setup provider paths that can expand bundled plugin IDs outside `plugins.allow`; the open closing-syntax PR #76085 is the right implementation candidate to review before this issue closes.

## What This Changes

Add an opt-in `plugins.bundledMode: "respect-allow"` so bundled runtime provider, web-search, and web-fetch paths honor `plugins.allow` and `plugins.entries.<id>.enabled` while preserving current compat behavior by default.

## Best Possible Solution

Review and land a focused PR, currently #76085 if it covers the full surface, that adds the documented opt-in config contract and gates all cited runtime compatibility paths without changing the default compat mode.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a restrictive `plugins.allow`, then follow `resolvePluginProviders({ mode: "setup" })` through `resolveDiscoveredProviderPluginIds` and `withActivatedPluginIds`, while the strict schema still rejects the proposed `plugins.bundledMode` key.

## Solution Assessment

Yes for the near-term direction. A default-preserving `respect-allow` opt-in is narrower than globally changing bundled plugin semantics, but the final patch should also cover the web-search and web-fetch compat paths named in the report.
