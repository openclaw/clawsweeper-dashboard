## Summary

Keep open. Current main still has the bundled-provider compatibility paths and no `plugins.bundledMode` config key, and open PR #75644 is already the linked implementation candidate, so this issue should stay open until that PR is reviewed and either merged or closed.

## What This Changes

The issue requests a default-preserving `plugins.bundledMode: "respect-allow"` option so bundled runtime provider paths honor `plugins.allow` and `plugins.entries.<id>.enabled` instead of force-loading bundled web-search, web-fetch, and LLM provider plugins.

## Best Possible Solution

Land a reviewed fix through PR #75644 or an equivalent narrow PR that adds the default-preserving config contract, wires `respect-allow` through the three runtime provider paths, enables the intended runtime loader cache only where safe, and covers the behavior with focused config and plugin runtime tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a restrictive `plugins.allow`, then follow the current web-search, web-fetch, and LLM provider runtime resolution paths, which still inject bundled compatibility IDs while the strict schema has no `plugins.bundledMode` escape hatch.

## Solution Assessment

Yes for the near-term path. A default-preserving opt-in is narrower and safer than changing all bundled allowlist semantics globally, while a larger manifest-only runtime loader cleanup can remain a separate follow-up.
