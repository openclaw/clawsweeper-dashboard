## Summary

Keep open. Current main still has the reported split between documented plugin allowlist semantics and runtime provider compatibility paths: `plugins.allow` is documented as exclusive, but web-search, web-fetch/content extractor, and LLM provider runtime paths still opt into bundled compatibility overrides, and the requested `plugins.bundledMode` config key is absent from the strict config schema.

## What This Changes

Request an opt-in `plugins.bundledMode: "respect-allow"` mode so bundled web-search, web-fetch, and LLM provider runtime paths honor `plugins.allow` and `plugins.entries.<id>.enabled` while preserving current bundled compat behavior by default.

## Best Possible Solution

Add a backward-compatible config schema/type/help field for bundled runtime allowlist mode, thread it through the LLM, web-search, web-fetch, and extractor runtime paths, keep current compat behavior as the default, and cover both modes with focused runtime/config tests plus a changelog entry.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction: current main has no `plugins.bundledMode` schema support, while the named runtime paths still pass bundled compatibility flags that expand restrictive allowlists. Live latency reproduction is not necessary to confirm the policy mismatch, though it should be profiled before claiming exact performance wins.

## Solution Assessment

Yes, the opt-in default-compat direction is a narrow maintainable fix for the documented mismatch, provided maintainers accept the public config key. A broader manifest-only runtime loader remains the long-term architecture but is not required to unblock this issue.
