## Summary

Keep open. Current main still lacks the requested `plugins.bundledMode` config surface and retains bundled-provider compatibility activation on the runtime paths, while the earlier closing-syntax PR is closed unmerged.

## What This Changes

The issue requests a default-compatible `plugins.bundledMode: "respect-allow"` opt-in so bundled runtime provider and web-tool paths honor `plugins.allow` and per-entry enablement.

## Best Possible Solution

Add the opt-in mode as a documented config contract, preserve the default compatibility behavior, and route web-search, web-fetch, and provider runtime compatibility through the same allowlist semantics when the mode is enabled.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a restrictive `plugins.allow`, then inspect the current runtime paths that still inject bundled compatibility activation while the strict schema rejects the proposed `plugins.bundledMode` key.

## Solution Assessment

Yes for a near-term fix. The default-preserving opt-in is narrower than changing bundled allowlist semantics globally, while deeper manifest-first runtime-loader cleanup can remain separate follow-up work.
