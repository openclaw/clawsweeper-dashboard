## Summary

Keep open. Current main still lacks the requested `plugins.bundledMode` surface and still has runtime/provider paths that can expand bundled plugins outside a restrictive `plugins.allow`; the linked fix PR #76085 remains open with closing syntax, so this issue should stay open until that or an equivalent fix lands.

## What This Changes

Add an opt-in `plugins.bundledMode: "respect-allow"` mode so bundled runtime provider discovery and activation honor `plugins.allow` and `plugins.entries.<id>.enabled` while preserving the current default behavior.

## Best Possible Solution

Land #76085 or an equivalent default-preserving change that adds the config/schema/help/tests and gates bundled runtime provider discovery plus activation by allowlist in the new opt-in mode.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a restrictive `plugins.allow` can still be widened by setup provider discovery plus `withActivatedPluginIds`, and the requested opt-in config key is absent from the strict schema.

## Solution Assessment

Yes for the near-term direction. A default-preserving opt-in mode is narrower than globally changing bundled allowlist semantics, while the larger manifest-first runtime-loader cleanup can remain follow-up work.
