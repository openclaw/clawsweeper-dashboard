## Summary

Keep open. Current main still lacks the requested config surface and still has bundled provider discovery paths that can widen a restrictive `plugins.allow`; the linked canonical PR #77194 remains open and a maintainer explicitly said not to close this issue until its remaining fixes land.

## What This Changes

Request to add a default-preserving plugin config mode so bundled runtime provider and web-provider paths can honor `plugins.allow` and per-plugin disablement instead of force-loading bundled providers.

## Best Possible Solution

Land #77194 or an equivalent default-preserving policy change after the two maintainer-identified edge cases are fixed, then close this issue through the merge.

## Reproduction Assessment

Yes. Source inspection on current main shows no implemented bundled-provider policy config field, setup discovery still accepts bundled provider plugins before allowlist activation checks, and `withActivatedPluginIds` can append those IDs into `plugins.allow`.

## Solution Assessment

Yes for the near-term direction. The maintainer-selected `plugins.bundledDiscovery` approach in #77194 is the narrow default-preserving version of the requested opt-in behavior, while a larger manifest-first runtime-loader cleanup can remain separate follow-up work.
