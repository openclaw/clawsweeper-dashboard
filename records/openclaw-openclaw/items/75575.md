## Summary

Keep open. Current main still has the reported bundled-provider compatibility paths and strict config schema lacks the requested opt-in key, while open PR #75644 now specifically references this issue as its implementation candidate, so the issue should remain open until that PR is reviewed and either merged or closed.

## What This Changes

The issue requests a backwards-compatible `plugins.bundledMode: "respect-allow"` setting so web-search, web-fetch, and LLM provider runtime paths honor `plugins.allow` and per-plugin enablement instead of force-loading bundled provider plugins.

## Best Possible Solution

Land a reviewed fix through PR #75644 or an equivalent narrow PR that adds the default-preserving config contract, wires `respect-allow` through the three runtime provider paths, safely enables the existing runtime loader cache where intended, and covers the behavior with focused config and plugin runtime tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a restrictive `plugins.allow` and follow the current runtime resolution paths, which still inject bundled provider compatibility IDs while the schema has no `plugins.bundledMode` escape hatch.

## Solution Assessment

Yes for the near-term product direction. A default-preserving opt-in is narrower than changing all bundled plugin allowlist semantics globally, while the larger manifest-only runtime cleanup can remain separate follow-up work.
