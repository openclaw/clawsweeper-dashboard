## Summary

Keep open. Current main still lacks the requested `plugins.bundledMode` config surface and still has runtime/setup paths that can widen a restrictive `plugins.allow`; the open closing PR is the right implementation candidate to review rather than closing this issue early.

## What This Changes

Request an opt-in `plugins.bundledMode: "respect-allow"` mode so bundled runtime provider paths honor `plugins.allow` and per-plugin enablement while preserving current compatibility defaults.

## Best Possible Solution

Review and land a default-preserving fix through the open closing PR, ensuring the config schema, generated help, provider discovery, activation context, and web-search/web-fetch runtime paths all share the same `respect-allow` semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a narrow `plugins.allow`, then follow provider setup discovery into `withActivatedPluginIds`, which adds discovered bundled provider IDs back into the allowlist while the strict schema still rejects `plugins.bundledMode`.

## Solution Assessment

Yes for the near-term direction. A default-preserving opt-in is narrower than changing bundled plugin semantics globally, but the implementation should be checked against every runtime surface listed in the issue before merge.
