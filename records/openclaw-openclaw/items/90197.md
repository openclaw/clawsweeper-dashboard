Full review comments:

- **[P1] Avoid rewalking rejected tool lists:** `src/agents/tool-schema-projection.ts:69-84`
  - body: These new length-failure branches return a diagnostic, but runtime callers still pass the original tools object into logRuntimeToolSchemaQuarantine(), whose recovery path iterates params.tools. A proxy with an unsafe or huge length can therefore still stall the runtime immediately after projection rejects it; return a list-level result that downstream logging can handle without scanning the rejected source list, or sanitize the list before logging.
  - confidence: 0.86
- **[P1] Preserve or approve over-cap tool-list upgrades:** `src/agents/tool-schema-projection.ts:79-84`
  - body: This branch quarantines the whole active tool list once length exceeds 10,000. Current main and v2026.6.10 continue scanning and projecting those entries, so an existing high-tool-count setup could upgrade into an empty active tool surface unless maintainers explicitly accept this cap or the patch preserves a compatible default.
  - confidence: 0.79
- **[P2] Use aggregate diagnostics for list-length failures:** `src/agents/tool-schema-projection.ts:69-84`
  - body: Length metadata failures are list-level failures, but this reports invalid lengths as tool[0] and oversized lists as tool[10000]. Existing consumers use diagnostic.toolIndex for owner attribution and doctor/inventory notices, so this can blame a real or nonsensical tool instead of the malformed list boundary.
  - confidence: 0.88
