## Summary

Keep open: current main still remaps backup tar entry paths without handling hardlink targets, verification still checks only entry paths, and https://github.com/openclaw/openclaw/pull/83938 is the open closing implementation candidate rather than a merged fix.

## What This Changes

The issue reports that `openclaw backup create --verify` can produce and accept tar archives whose hardlink targets contain `..`, causing broad extraction with macOS `bsdtar` to fail after verification passed.

## Best Possible Solution

Review and land or explicitly close https://github.com/openclaw/openclaw/pull/83938, or an equivalent patch that dereferences hardlinked backup payloads and verifies unsafe tar link targets.

## Reproduction Assessment

Yes, at source level. Include hardlinked files in a backup source so node-tar can emit a `Link` entry while OpenClaw remaps only `entry.path`; current verification then reads only entry paths and misses the unsafe `linkpath`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in backup archive creation and verification; configurable excludes and broader backup redesign are related but do not close this restore-safety gap.
