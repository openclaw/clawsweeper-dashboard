## Summary

Keep open. Current main and the latest release still validate only tar entry paths, and the linked fix PR is open rather than merged, so the issue should stay open until that fix or a replacement lands.

## What This Changes

The issue reports that `openclaw backup create --verify` can produce archives whose hardlink targets contain `..`, causing broad extraction with macOS tar to fail despite verification passing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83938 or a replacement that makes created archives restore-safe and makes verification reject unsafe hardlink targets.

## Reproduction Assessment

Yes. A high-confidence source path exists: create a backup from a tree with hardlinked files where `tar` emits a `Link` entry with a traversal-bearing `linkpath`; current main rewrites `entry.path` and verify ignores `linkpath`.

## Solution Assessment

Yes. The narrow maintainable solution belongs in backup archive emission and verification; configurable excludes and a broader restore command are related but separate tracks.
