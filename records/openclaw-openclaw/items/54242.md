## Summary

Keep open: current main and v2026.5.18 still only remap and verify tar entry paths, not hardlink `linkpath`, and the linked open PR is an implementation candidate rather than a reason to close before merge.

## What This Changes

The issue asks for backup creation and verification to handle tar hardlink targets containing `..` so verified archives can be broadly extracted by macOS tar.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/83938, or an equivalent patch that makes backup creation emit restore-safe hardlink entries and makes verification reject unsafe link targets with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create a backup of a tree where `tar` 7.5.15 emits a hardlink `Link` entry with `linkpath` containing `..`; current create and verify code inspect only `entry.path`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in backup archive emission and verification; configurable excludes and broader backup redesign items are related but do not close this restore-safety gap.
