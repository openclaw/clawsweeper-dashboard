## Summary

Keep open. Current main still remaps backup tar entry names without handling hardlink targets, and `backup verify` still validates only entry paths even though the pinned tar dependency stores hardlink targets in a separate `linkpath` field.

## What This Changes

Make `openclaw backup create --verify` produce and validate restore-safe tar hardlink targets so broad macOS `bsdtar` extraction does not fail on `..` link paths.

## Best Possible Solution

Update backup creation to emit restore-safe hardlink entries, or dereference unsafe hardlinks, and update backup verification to reject unsafe `ReadEntry.linkpath` values with focused regression coverage.

## Reproduction Assessment

Yes. The high-confidence source reproduction is a backup source tree with two hardlinked files where `tar@7.5.13` emits the second file as a `Link` entry whose `linkpath` can contain `..`, while current create and verify code handle only `entry.path`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in backup archive creation and verification, while broader configurable exclusions remain separate feature work tracked by related backup-design issues.

## Repair Work Prompt

Fix the backup hardlink target restore-safety bug reported at https://github.com/openclaw/openclaw/issues/54242. Keep the change narrow: in `src/infra/backup-create.ts`, make `openclaw backup create` avoid emitting hardlink `linkpath` values that contain traversal or otherwise fail broad extraction, either by rewriting hardlink targets to archive-relative safe targets or by dereferencing unsafe hardlinks. In `src/commands/backup-verify.ts`, inspect `tar` `ReadEntry.linkpath` for `Link` and symlink entries as appropriate and reject unsafe relative, absolute, backslash, or traversal targets that common extractors reject. Add focused regression coverage with a fixture containing hardlinked files that currently produces an unsafe link target, and keep configurable exclude support out of scope because #40786/#67990 track that separately. Add a user-facing changelog entry if behavior changes are committed.
