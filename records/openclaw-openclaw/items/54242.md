- `impact:data-loss`: A backup archive that verifies successfully but fails broad restore extraction can block recovery of user state after catastrophic loss.

## Summary

Keep open: current main and v2026.5.18 still remap and verify tar entry paths, not hardlink `linkpath`, and the linked fix PR is open rather than merged.

## What This Changes

The issue reports that backup archives can pass OpenClaw verification while containing hardlink targets with `..` that macOS bsdtar rejects during broad extraction.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/83938, or an equivalent patch that makes backup creation emit restore-safe hardlink entries and makes verification reject unsafe link targets with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create a backup of a tree where tar 7.5.15 emits a hardlink `Link` entry whose `linkpath` contains `..`; current create and verify code inspect only `entry.path`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in backup archive emission and verification; configurable excludes and broader backup redesign issues are related but do not close this restore-safety gap.
