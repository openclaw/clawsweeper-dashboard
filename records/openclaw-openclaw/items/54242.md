## Summary

Keep open: current main and v2026.5.19 still remap and verify tar entry paths without handling hardlink `linkpath`, and the focused fix is an open linked PR rather than merged.

## What This Changes

The issue asks for backup creation or verification to prevent tar hardlink targets with traversal components from producing archives that verify but fail broad extraction on macOS tar.

## Best Possible Solution

Review and land or explicitly close https://github.com/openclaw/openclaw/pull/83938, or an equivalent patch that makes backup archives restore-safe for hardlinked files and verifies hardlink targets.

## Reproduction Assessment

Yes. Source inspection plus tar 7.5.15 source gives a high-confidence reproduction path: include hardlinked files in a backup source so node-tar can emit a `Link` entry while OpenClaw remaps only `entry.path`, leaving `linkpath` outside the verified path set.

## Solution Assessment

Yes. The narrow maintainable fix belongs in backup archive creation and verification; configurable excludes and broader backup redesign are related but do not close this restore-safety gap.
