## Summary

Keep open: current main has adjacent transcript replay hygiene, narrow transcript repair, memory readiness/status audits, and session cleanup, but no first-class configurable dry-run scan/sanitize workflow across transcripts, summaries, active memory, and memory stores; related PR #77445 is narrower and remains open.

## What This Changes

Add a configurable memory/transcript hygiene doctor and sanitizer for unsafe persisted artifacts such as raw local paths, file URLs, delivery directives, and internal tool-output leakage.

## Best Possible Solution

Design a core-owned read-only JSON scan/report contract first, then add an opt-in sanitizer with reviewed rule IDs, redaction, allowlists, locking/backups, and docs.

## Reproduction Assessment

Not applicable: this is a feature request for a new operator hygiene surface, not broken existing behavior. Source and docs inspection show adjacent repair/readiness tools but no current command that implements the requested cross-surface scan/sanitize workflow.

## Solution Assessment

Unclear until maintainer design review: a read-only scanner with machine-readable evidence appears to be the safest first step, while sanitizer/apply semantics, allowlists, locking, and backups need product/ownership decisions.
