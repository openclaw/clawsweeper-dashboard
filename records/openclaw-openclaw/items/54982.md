## Summary

Keep open. Current main still lacks the opt-in TUI flag requested by #11669, and this PR is useful source work, but the patch is not merge-ready because preserve mode leaves phantom partial replies when final output is suppressed, can duplicate unchanged final snapshots, and lacks the required changelog entry.

## What This Changes

The PR adds an opt-in `--preserve-intermediate` TUI flag, threads it into `ChatLog`, appends streamed assistant updates as separate log entries, and adds focused CLI docs and tests.

## Best Possible Solution

Land a rebased, default-off TUI option that records intermediate chunks without breaking no-output suppression or duplicating final snapshots, with focused tests plus user-facing docs and changelog coverage.

## Reproduction Assessment

Yes. A focused PR-head repro can construct `ChatLog(40, { preserveIntermediate: true })`, call two `updateAssistant` calls for one run, then exercise `dropAssistant` through the existing final-with-no-message path and observe the earlier partial reply remains; calling `finalizeAssistant` with the unchanged latest text also reproduces the duplicate final snapshot.

## Solution Assessment

No. The TUI/`ChatLog` boundary is the right place for the option, but the implementation should track all preserved components for a run, remove the whole run when output is suppressed, skip unchanged final snapshots, and preserve default rendering semantics.

## Review Findings

  - body: With preserve mode enabled, each delta appends a component but `streamingRuns` only tracks the newest one.
