## Best Possible Solution

Land a rebased implementation that tracks all preserved assistant chunks per run, removes them for dropped or suppressed finals, documents the scrollback-budget behavior, and keeps default TUI in-place streaming unchanged.

## Reproduction Assessment

Yes. From source inspection, PR-head preserve mode appends multiple assistant components while current event handling still calls `dropAssistant` for no-message and suppressed-output finals, so only the newest component would be removed.

## Solution Assessment

No. The opt-in flag is a plausible user-facing shape, but the best fix needs a current-main rebase, per-run preserved-chunk ownership, cleanup coverage, and docs for the scrollback cost.
