## Summary

Keep open: current main still lacks the requested TUI preserve mode, and this PR is the canonical implementation path, but the PR head has source-reproducible preserve-mode bugs and is conflicting with newer TUI/docs changes.

## What This Changes

Adds an opt-in `--preserve-intermediate` TUI flag, threads it through CLI/runTui/ChatLog, appends streamed assistant snapshots, and adds a CLI doc example plus focused tests.

## Best Possible Solution

Land a rebased, default-off implementation that preserves intermediate snapshots, removes all preserved chunks for suppressed runs, skips duplicate final snapshots, and documents the fixed log-budget trade-off in CLI and web TUI references.

## Reproduction Assessment

Yes, for the PR-head defects by source inspection: enable preserve mode, append multiple deltas for one run, then either drop a suppressed final or finalize with text equal to the latest delta. I did not run tests because this review is read-only.

## Solution Assessment

No. The CLI/TUI option boundary is reasonable, but the branch should track all preserved chunks for a run, drop them together when output is suppressed, skip unchanged final snapshots, and update both CLI and web references before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Drop every preserved chunk for suppressed runs:** `src/tui/components/chat-log.ts:81-84`
  - body: Preserve mode overwrites the run's tracked component on every delta, so `dropAssistant` only knows about the newest chunk. Existing final handlers call `dropAssistant` for missing finals and suppressed `(no output)`, leaving earlier chunks visible even though that run was intentionally discarded.
  - confidence: 0.91
- **[P3] Skip unchanged final snapshots:** `src/tui/components/chat-log.ts:115-118`
  - body: `finalizeAssistant` appends a fresh component in preserve mode even when the final text matches the latest streamed text. Normal streams can repeat the terminal payload, producing duplicate trailing entries and consuming the fixed chat-log budget sooner.
  - confidence: 0.84
- **[P3] Document the flag in reference sections:** `docs/cli/tui.md:28-29`
  - body: The PR adds the flag only to an example. Current main has a CLI options table and a web TUI options section that should include the supported option and the per-delta log-budget trade-off so users scanning references can find it.
  - confidence: 0.8
