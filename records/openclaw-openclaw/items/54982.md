## Summary

Keep open: current main still lacks the requested opt-in TUI preservation mode, and this PR is the canonical implementation candidate, but its head is merge-conflicting and preserve-mode ChatLog state handling still has a blocking defect.

## What This Changes

The PR adds a `--preserve-intermediate` TUI flag, threads it through CLI options into `ChatLog`, appends streamed assistant updates as separate entries, and adds docs plus tests.

## Best Possible Solution

Keep the opt-in flag direction only if maintainers want the feature, but rebase over current TUI, track every preserved component for a run, remove all preserved chunks on discarded finals, skip unchanged final snapshots, and document the bounded log-budget tradeoff.

## Reproduction Assessment

Yes for the PR-head defects by source inspection: preserve mode appends multiple chunks while tracking only the latest component, and current final-event handling relies on `dropAssistant` to discard no-output runs. Current-main bug reproduction is not applicable because this is new feature work.

## Solution Assessment

No. The opt-in flag boundary is plausible, but the branch should track all preserved components for a run, skip unchanged final snapshots, document the fixed log budget, and rebase cleanly before maintainer feature review.
