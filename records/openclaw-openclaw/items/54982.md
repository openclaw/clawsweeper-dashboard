## Summary

Keep open: current main still lacks the requested preserve-intermediate TUI mode and this PR is the canonical implementation path for the closed feature request, but the branch still has source-proven preserve-mode defects that can leave phantom partial output and duplicate final entries.

## What This Changes

This PR adds an opt-in `--preserve-intermediate` TUI flag, threads it into `ChatLog`, adds happy-path tests, and documents a CLI example.

## Best Possible Solution

Keep the opt-in flag direction, but repair preserve-mode run tracking so discarded runs remove all preserved chunks, unchanged finals do not duplicate the last snapshot, and public docs describe the bounded log budget before maintainer feature review.

## Reproduction Assessment

Yes for the PR-head defects by source inspection: preserve mode appends multiple chunks while tracking only the latest component, and current final-event handling relies on `dropAssistant` to discard no-output runs. Current-main bug reproduction is not applicable because this is new feature work and main still has the documented in-place streaming behavior.

## Solution Assessment

No. The opt-in CLI/TUI boundary is reasonable, but this branch should track all preserved chunks for a run, skip unchanged final snapshots, and document the fixed log budget before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Drop every preserved chunk for discarded runs:** `src/tui/components/chat-log.ts:83`
  - body: In preserve mode this assignment keeps only the newest appended component for the run. When a final event is intentionally discarded, `dropAssistant` removes only that latest component and leaves earlier partial assistant text visible as a phantom reply.
  - confidence: 0.92
- **[P3] Skip unchanged final snapshots in preserve mode:** `src/tui/components/chat-log.ts:116`
  - body: `finalizeAssistant` always appends another preserved component before clearing the run. Normal streamed finals can repeat the latest displayed text, so the TUI shows a duplicate trailing assistant entry and consumes the bounded log budget sooner.
  - confidence: 0.86
- **[P3] Document the preserve-mode log budget:** `docs/cli/tui.md:28-29`
  - body: The PR adds the flag only as an example. The CLI reference should say that each preserved stream update creates a log entry and can displace older history during long responses.
  - confidence: 0.8
