## Summary

Keeping this PR open: the feature request is still valid and not implemented on current main, but the branch is not merge-ready because preserved assistant chunks are not owned per run, so dropped/no-output finals can leave stale partial output visible.

## What This Changes

The PR adds an opt-in `openclaw tui --preserve-intermediate` flag, threads it into `runTui`, changes `ChatLog` to append streamed assistant updates as separate entries, and adds focused tests plus CLI docs.

## Best Possible Solution

Land a refreshed opt-in implementation that tracks preserved assistant chunks by run, removes all chunks on dropped output, documents or bounds scrollback cost, and keeps default in-place streaming unchanged.

## Reproduction Assessment

Yes for the review findings: source inspection shows preserve mode appends multiple assistant components while silent and suppressed finals call `dropAssistant`, which only removes the newest tracked component. I did not run tests because this review is read-only.

## Solution Assessment

No. The opt-in flag is a plausible feature shape and screenshots show the happy path, but the branch needs current-main rebase plus per-run cleanup and scrollback-budget handling before it is the best implementation.
