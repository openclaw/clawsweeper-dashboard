## Summary

Keep open: current main still lacks the requested TUI preservation mode, and this PR is the canonical implementation path, but the branch needs fixes before merge because preserved chunks are not tracked as a full run, unchanged finals duplicate the last snapshot, and the new flag is under-documented.

## What This Changes

Adds a default-off TUI `--preserve-intermediate` flag wired through CLI options, `runTui`, and `ChatLog`, with docs and tests for preserved streaming assistant entries.

## Best Possible Solution

Land a rebased default-off TUI flag that preserves visible intermediate assistant chunks while preserving existing no-output suppression/final dedupe semantics and documenting the fixed log-budget tradeoff.

## Reproduction Assessment

Yes, for the PR-head defects by source inspection: enable preserve mode, stream multiple deltas, then receive either an absent/suppressed final or a final payload equal to the latest delta. Current main lacks the requested feature, so this is PR-behavior reproduction rather than a current-main bug reproduction.

## Solution Assessment

No. The opt-in CLI/TUI boundary is reasonable, but this branch should track preserved chunks per run, drop the whole run on discarded finals, skip unchanged final snapshots, and document the log-budget tradeoff before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Track all preserved chunks per run:** `src/tui/components/chat-log.ts:83-84`
  - body: In preserve mode, every delta appends a component, but this assignment overwrites the run map with only the newest one. When a final event has no message or `(no output)` is suppressed, `dropAssistant` removes only that latest component and leaves earlier partial assistant text visible for a run that was explicitly discarded.
  - confidence: 0.91
- **[P3] Skip unchanged final snapshots:** `src/tui/components/chat-log.ts:116`
  - body: `finalizeAssistant` always appends another preserved component before clearing the run. When the final text repeats the latest streamed snapshot, the TUI shows a duplicate trailing assistant entry and consumes the fixed chat-log budget sooner.
  - confidence: 0.84
- **[P3] Document the preserve-mode tradeoff:** `docs/cli/tui.md:28-29`
  - body: The PR adds the flag only as an example. The current CLI reference has an Options table, and neither that reference nor the TUI guide tells users that each preserved stream update creates a log entry that can displace older history.
  - confidence: 0.8
