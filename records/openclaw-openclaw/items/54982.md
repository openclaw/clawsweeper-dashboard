## Summary

Keep this PR open. Current main still lacks the requested `--preserve-intermediate` TUI flag, so the PR is not obsolete, but the proposed preserve-mode state handling can leave suppressed partial assistant output in the log and can duplicate an unchanged final snapshot.

## What This Changes

The PR branch adds an opt-in `--preserve-intermediate` TUI flag, threads it into `ChatLog`, appends streamed assistant updates as separate entries, and adds focused docs and tests.

## Best Possible Solution

Land the feature after preserve mode records the full set of assistant components created for a run, removes all of them when that run is explicitly suppressed, avoids adding a duplicate final entry when the final text is unchanged, and documents the log-budget tradeoff while keeping default TUI rendering unchanged.

## Reproduction Assessment

Yes. A focused `ChatLog` unit repro can enable `preserveIntermediate`, call `updateAssistant` twice for one run, then exercise `dropAssistant` or `finalizeAssistant` with unchanged final text; current event-handler tests show the real final-suppression paths that call `dropAssistant`. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The option belongs in the TUI/`ChatLog` boundary, but the current PR implementation is not the best merge shape until preserved chunks are tracked per run, suppression removes the whole run, duplicate final snapshots are skipped, and the required changelog/docs coverage is completed.

  - body: With preserve mode enabled, every delta appends a new assistant component while `streamingRuns` points only at the newest one. Existing final-event suppression paths call `dropAssistant`, so this removes only the last chunk and leaves earlier partial text visible as a phantom reply.
