## Summary

Keep this PR open but do not merge it yet. Current main still lacks the requested TUI preservation option, and the branch is a plausible implementation, but preserve mode leaves source-reproducible edge cases around discarded finals, duplicate final snapshots, and incomplete docs.

## What This Changes

The PR adds a default-off `openclaw tui --preserve-intermediate` option wired through CLI options, TUI construction, ChatLog rendering, docs, and focused tests.

## Best Possible Solution

Land a rebased default-off TUI flag that preserves visible intermediate chunks while preserving existing no-output suppression, avoiding duplicate final snapshots, and documenting the fixed log-budget tradeoff.

## Reproduction Assessment

Yes, for the PR-head defects by source inspection: enable preserve mode, stream multiple deltas, then receive either an absent/suppressed final or a final payload equal to the latest delta. Current-main bug reproduction is not applicable because this is a new feature; current main only shows the existing in-place behavior.

## Solution Assessment

No. The opt-in CLI/TUI boundary is reasonable, but this branch should track all preserved components for a run, drop the whole preserved run on discarded finals, skip unchanged final snapshots, and document the log-budget tradeoff before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Track all preserved chunks per run:** `src/tui/components/chat-log.ts:83-84`
  - body: In preserve mode, every delta appends a component, but this assignment overwrites the run map with only the newest one. When a final event has no message or `(no output)` is suppressed, `dropAssistant` removes only that latest component and leaves earlier partial assistant text visible for a run that was explicitly discarded.
  - confidence: 0.91
- **[P3] Skip unchanged final snapshots:** `src/tui/components/chat-log.ts:116`
  - body: `finalizeAssistant` always appends another preserved component before clearing the run. When the final text repeats the latest streamed snapshot, the TUI shows a duplicate trailing assistant entry and consumes the fixed chat-log budget sooner.
  - confidence: 0.84
- **[P3] Document the preserve-mode tradeoff:** `docs/cli/tui.md:28-29`
  - body: The PR adds the flag only as an example. The current CLI reference has an Options table, and neither that reference nor the TUI guide tells users that each preserved stream update creates a log entry that can displace older history.
  - confidence: 0.8
