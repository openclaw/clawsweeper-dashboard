## Summary

Keep open: current main still lacks the requested TUI preservation feature, and this PR is the canonical implementation path for the linked feature request, but the branch still has concrete preserve-mode behavior defects and is currently merge-conflicting.

## What This Changes

The branch adds a default-off `openclaw tui --preserve-intermediate` flag, threads it through TUI options, changes `ChatLog` to append streamed assistant updates, and adds focused docs/tests.

## Best Possible Solution

Land a rebased default-off TUI flag that preserves intermediate chunks while preserving no-output suppression, avoiding duplicate final snapshots, and documenting the fixed log-budget tradeoff.

## Reproduction Assessment

Yes, for the PR-head defects by source inspection: enable preserve mode, stream multiple deltas, then receive either an absent/suppressed final or a final payload equal to the latest streamed text. Current-main bug reproduction is not applicable because this is a new feature and current main only has the existing in-place behavior.

## Solution Assessment

No. The opt-in CLI/TUI boundary is reasonable, but the branch should track all preserved chunks for a run, drop the whole preserved run on discarded finals, skip unchanged final snapshots, document the tradeoff, and rebase before merge.

- **[P2] Track every preserved chunk for a run:** `src/tui/components/chat-log.ts:83-84`
  - body: In preserve mode, every delta appends a component, but this map assignment keeps only the newest one.   - body: The PR adds the flag only as an example. The CLI options table and TUI guide should explain that each preserved stream update creates a log entry and can displace older history in long responses.
  - confidence: 0.8
