## Summary

The watchdog is useful and well tested for pending asynchronous startup, but its documented contract incorrectly includes event-loop starvation, where the timer callback cannot execute.

## What This Changes

Adds a configurable gateway startup watchdog that emits the active diagnostic phase stack to stderr if HTTP binding does not complete before the threshold.

## Best Possible Solution

Retain the cold-start watchdog for pending asynchronous startup, but state clearly that its timer must be able to run and that fully blocked event-loop stalls remain outside its coverage.

## Reproduction Assessment

Yes for the review concern: the diagnostic is emitted by `setTimeout`, so a synchronous event-loop block prevents it from firing until the block ends.

## Solution Assessment

No as currently described; the implementation is appropriate for pending asynchronous startup and bind hangs, but the stated event-loop-starvation contract is broader than the mechanism can satisfy.
