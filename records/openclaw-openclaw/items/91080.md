## Summary

Keep this PR open, but do not merge it yet: the diagnostic mechanism is coherent and the earlier multi-bind cancellation bug is fixed, yet the watchdog is armed only once outside a loop that can restart the gateway in-process, so later startup attempts can silently lose the promised hang diagnostic.

## What This Changes

Adds a default-on, environment-configurable gateway startup watchdog that writes the active diagnostic phase stack to stderr when HTTP binding does not complete within the threshold.

## Best Possible Solution

Own the watchdog per gateway start attempt: arm immediately before each `startGatewayServer` invocation, cancel after all bind hosts complete or that attempt fails, add restart regression coverage, and explicitly approve or simplify the public threshold contract.

## Reproduction Assessment

Yes, from source inspection: `runGatewayLoop` can invoke its `start` callback again, while this PR arms before entering the loop and the first `http.bound` cancels the only timer. A focused restart-loop test should turn this into an executable regression.

## Solution Assessment

No. Active-phase snapshots are a suitable diagnostic mechanism, but the watchdog must be owned per startup attempt, and maintainers must affirm that a default-on timer plus a new environment variable is the intended product contract.
