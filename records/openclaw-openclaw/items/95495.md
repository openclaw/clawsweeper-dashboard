## Label Justifications

- `P1`: The latest stable upgrade can leave memory_search paused for users with existing corpora, breaking active agent recall until manual re-embedding.
- `impact:data-loss`: The existing indexed memory dataset remains on disk but is silently dropped from the active runtime path after upgrade.
- `impact:session-state`: The active agent memory index state drifts from the user's persisted corpus and reports missing identity in the new per-agent store.

## Summary

Keep open: current main and v2026.6.9 still route built-in memory to the per-agent SQLite database, while the visible migration paths only remove legacy config keys or migrate same-database table shapes, not an existing separate legacy memory index file.

## What This Changes

The issue reports that upgrading to v2026.6.9 moved the memory vector store from the legacy global SQLite path to the per-agent SQLite database without migrating existing embeddings or warning users, forcing a full re-embed.

## Best Possible Solution

Add a doctor/update-owned migration or explicit upgrade warning for existing legacy memory DBs before or during the per-agent DB cutover, with regression coverage for an already-indexed corpus and no steady-state runtime fallback reader.

## Reproduction Assessment

Yes. The source path is high-confidence: v2026.6.9 and current main open the per-agent database, while the inspected migration paths do not import a separate legacy memory/main.sqlite store and the reporter supplied concrete before/after upgrade status, file sizes, and recovery output.

## Solution Assessment

No. Current main is not the best solution yet because it documents and uses the new per-agent database but does not migrate or warn about an existing separate legacy memory index before the old corpus disappears from active search.
