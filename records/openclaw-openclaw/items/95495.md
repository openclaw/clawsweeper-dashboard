## Summary

Keep open: current main and v2026.6.9 still route built-in memory to the per-agent SQLite database, while the visible legacy handling removes old store-path config and only migrates legacy tables when they are already in the opened database. I found no current import or upgrade warning for an existing separate legacy `memory/main.sqlite` corpus.

## What This Changes

The issue reports a v2026.6.8 to v2026.6.9 memory storage relocation that leaves an existing legacy vector corpus unused and forces a full re-embed without upgrade-time warning.

## Best Possible Solution

Add a maintainer-approved upgrade path for legacy global memory stores: either a tested doctor/update migration into the canonical per-agent SQLite store or an explicit upgrade warning/rebuild policy that preserves data and cost expectations.

## Reproduction Assessment

Yes from source inspection: current main and v2026.6.9 derive built-in memory storage from the per-agent database and only remove legacy store-path config, while the reporter supplied concrete upgrade output showing the old populated DB untouched and the new store empty. I did not execute a live upgrade in this read-only review.

## Solution Assessment

Unclear until maintainers choose the upgrade policy; a silent steady-state fallback reader would conflict with canonical-storage policy, so the safer direction is a bounded doctor/update migration or explicit warning/rebuild path.
