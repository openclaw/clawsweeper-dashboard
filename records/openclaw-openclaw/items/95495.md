## Label Justifications

- `P1`: The issue reports a shipped upgrade regression that disables semantic memory search for existing self-hosted users until a potentially expensive full reindex completes.
- `impact:data-loss`: Existing embedding/index data is left behind and effectively lost to the upgraded runtime even though the old database remains on disk.
- `impact:session-state`: The failure is about durable memory/search state moving stores and drifting from the agent's usable recall state.

## Summary

Keep open: current main still routes builtin memory to the per-agent SQLite database, while the visible doctor/config migration removes legacy memory store path config and does not import an existing legacy memory index database.

## What This Changes

The issue asks OpenClaw to avoid silently moving builtin memory indexing from the legacy global store to the per-agent SQLite database without migrating existing embeddings or warning during upgrade.

## Best Possible Solution

Add an explicit upgrade path for legacy builtin memory indexes: either a tested doctor/update import into the per-agent database or a clear pre-upgrade/post-upgrade warning that the index must be rebuilt.

## Reproduction Assessment

Yes, at source level: current main resolves builtin memory to `agents/<agentId>/agent/openclaw-agent.sqlite` and only removes legacy memory store path config, so an existing legacy `memory/main.sqlite` corpus is no longer the active store. I did not run a live upgrade with a 1499-file corpus in this read-only review.

## Solution Assessment

Unclear: a code fix is needed, but maintainers need to choose whether to import legacy embeddings, preserve a one-time migration, or intentionally require rebuild with prominent upgrade warning.
