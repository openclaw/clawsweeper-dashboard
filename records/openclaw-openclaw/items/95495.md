## Label Justifications

- `P1`: The shipped upgrade path can leave existing semantic memory search paused until a manual full re-embed or migration restores the corpus.
- `impact:data-loss`: A populated legacy memory index remains on disk but is silently dropped from the active runtime search path after upgrade.
- `impact:session-state`: The affected state is persistent agent memory/search context stored in SQLite and used for future recall.

## Summary

Keep open: current main and shipped v2026.6.9 still use the per-agent memory database while the schema migration only handles legacy tables inside the already-opened database. The open closing PR is the active fix candidate, so this issue should remain open until that PR lands or maintainers choose a different migration policy.

## What This Changes

The issue requests that upgrading from v2026.6.8 to v2026.6.9 preserve or explicitly migrate the existing built-in memory SQLite index instead of silently switching to an empty per-agent store.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/95631 with a one-shot migration or explicit upgrade warning that preserves existing indexed corpora without steady-state fallback drift.

## Reproduction Assessment

Yes. Current main and v2026.6.9 route built-in memory to the per-agent SQLite database, while current schema initialization only migrates legacy tables inside the opened database and the issue/comments provide field evidence of populated sidecars being left behind.

## Solution Assessment

No, current main is not the best solution because it switches the active store without importing or warning about an existing sidecar corpus. The best path is to resolve the linked PR or a maintainer-approved alternative migration policy before closing this issue.
