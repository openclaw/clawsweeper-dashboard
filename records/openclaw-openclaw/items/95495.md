## Summary

Keep open: current main and shipped v2026.6.9 still route built-in memory to the per-agent SQLite database, while the legacy sidecar import only handles tables already present in the opened database. The open linked fix PR is a candidate, but it is unmerged and already flagged for maintainer migration-policy review.

## What This Changes

The issue reports a shipped upgrade regression where the built-in memory index moved from the legacy global SQLite sidecar to the per-agent SQLite database without migration or upgrade-time warning.

## Best Possible Solution

Keep the issue open while maintainers decide and land a one-shot migration or explicit upgrade-warning path for legacy memory sidecars, with regression coverage for legacy sidecar, empty canonical DB, and already-rebuilt canonical DB cases.

## Reproduction Assessment

Yes. The source path is clear on current main and v2026.6.9: memory opens the per-agent SQLite database, while the legacy migration only copies old tables inside that same opened database; the reporter and two commenters provide field evidence of the legacy sidecar remaining populated or active.

## Solution Assessment

No. Current main is not the best solution because it documents and uses the per-agent database without importing or warning about an existing separate memory sidecar; the linked PR is the active candidate but still needs maintainer migration-policy resolution.
