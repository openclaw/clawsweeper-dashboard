- `P1`: The latest stable upgrade can leave memory_search paused for users with existing corpora, breaking an active agent recall workflow until manual re-embedding.
- `impact:data-loss`: The existing indexed memory dataset remains on disk but is silently dropped from the active runtime path after upgrade.

## Summary

Keep open: the latest release and current main still route built-in memory to the per-agent SQLite database, while the visible migration path only removes legacy config keys and does not import an existing separate memory index database.

## What This Changes

The issue reports that upgrading from v2026.6.8 to v2026.6.9 moves built-in memory indexes from the legacy global SQLite store to the per-agent SQLite database without migration or warning, forcing a full re-embed.

## Best Possible Solution

Add a doctor/update-owned migration or explicit upgrade warning for existing legacy memory DBs before or during the per-agent DB cutover, with regression coverage for an already-indexed corpus and no steady-state runtime fallback reader.

## Reproduction Assessment

Yes. The source path is high-confidence: v2026.6.9 and current main open the per-agent DB, while the only visible migration removes legacy config paths and the reporter supplied concrete before/after upgrade status, file sizes, and recovery output.

## Solution Assessment

No. Current main is not the best solution yet because it documents and uses the new per-agent DB but does not migrate or warn about an existing separate legacy memory index before the old corpus disappears from active search.
