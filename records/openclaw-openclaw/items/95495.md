## Label Justifications

- `P1`: The latest stable upgrade can leave semantic memory search paused for users with existing corpora until they notice and pay for a full manual re-embed.
- `impact:data-loss`: The indexed memory dataset remains on disk but is silently dropped from the active runtime path after upgrade.
- `impact:session-state`: The active agent memory index state drifts from the user's persisted corpus and reports missing identity in the new per-agent store.

## Summary

Keep open: v2026.6.9 and current main still route built-in memory to the per-agent SQLite database, while the visible migration path removes legacy memory store config paths and same-database legacy table shapes without importing an existing separate legacy memory index database.

## What This Changes

The issue reports that upgrading from 2026.6.8 to 2026.6.9 silently moves built-in memory indexing from the legacy global SQLite store to the per-agent SQLite store without migrating existing embeddings or warning during update.

## Best Possible Solution

Add an update/doctor-owned migration or explicit upgrade warning for existing legacy memory DBs before the per-agent DB cutover leaves them inactive, with regression coverage for an already-indexed corpus and no permanent runtime dual-read fallback.

## Reproduction Assessment

Yes at source level, but not as an independent live upgrade replay. Current main and v2026.6.9 open the per-agent DB, while the visible migration removes legacy store path config and the reporter supplied concrete before/after upgrade status, file sizes, and recovery output.

## Solution Assessment

No current close-ready solution exists. The narrow maintainable fix should live in doctor/update migration policy for persistent memory state, not as a steady-state runtime fallback to old sidecar databases.
