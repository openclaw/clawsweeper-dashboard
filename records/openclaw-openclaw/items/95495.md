## Summary

Keep open: current main and v2026.6.9 still route built-in memory to the per-agent SQLite database, while the visible upgrade/doctor path removes legacy store-path config rather than importing an existing separate memory index database.

## What This Changes

The issue reports that upgrading to v2026.6.9 silently moved the active built-in memory index from the legacy memory database into the per-agent database, leaving an existing embedded corpus unused and requiring a full re-embed.

## Best Possible Solution

Add a doctor/update-owned legacy memory database import or explicit warning/rebuild policy for existing indexed corpora before the per-agent database cutover leaves them inactive, with upgrade regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.9 route built-in memory to the per-agent database while the visible doctor migration only removes legacy store-path config. I did not run the full live upgrade, but the reporter supplied concrete before/after upgrade status, file-size, and recovery output.

## Solution Assessment

No, current main is not the best solution yet because it documents and uses the new per-agent database without migrating or warning about an existing separate legacy memory index. The safer end state is a doctor/update-owned import or explicit upgrade warning rather than a steady-state runtime fallback reader.
