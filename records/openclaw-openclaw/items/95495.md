## Summary

Keep open: current source and the latest release still route built-in memory indexes to the per-agent SQLite database while only removing legacy memory store path config. The existing migration coverage moves old table names inside an already-opened database; it does not prove an upgrade-time import from the legacy global memory store, so the reported v2026.6.9 regression remains actionable.

## What This Changes

The issue requests an upgrade-safe migration or explicit warning for the v2026.6.9 built-in memory store relocation from the legacy global store to the per-agent SQLite database.

## Best Possible Solution

Keep this issue open while maintainers choose and implement the upgrade contract: doctor/update should detect a legacy populated memory store and either import it into the per-agent database with backup/verification or explicitly warn before requiring a full rebuild.

## Reproduction Assessment

Yes, from source at high confidence: the reported upgrade path matches current/latest-release code that resolves memory to the per-agent database while only migrating same-database legacy table names. I did not run a live packaged v2026.6.8 to v2026.6.9 upgrade in this read-only review.
