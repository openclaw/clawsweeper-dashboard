## Summary

Keep open: current main and the latest stable release still route built-in memory to the per-agent SQLite database, while the visible migration path removes legacy store-path config and only migrates legacy tables inside the already-opened database, not an existing separate legacy memory index.

## What This Changes

The issue reports that the 2026.6.9 upgrade moves the built-in memory index to the per-agent database without migration or warning, forcing a full re-embed of an existing corpus.

## Best Possible Solution

Add a doctor/update-owned migration or explicit upgrade warning for existing legacy memory databases before the per-agent database cutover, with regression coverage and no steady-state runtime fallback reader.

## Reproduction Assessment

Yes, source-reproducible. Current main and v2026.6.9 open the per-agent memory database, and the inspected migration code does not import an existing separate legacy memory database.

## Solution Assessment

No. Current main documents and uses the new per-agent store, but it does not yet provide the upgrade-safe import, warning, or rebuild policy for existing legacy indexes.
