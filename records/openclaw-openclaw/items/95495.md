## Label Justifications

- `P1`: The latest stable upgrade can leave memory_search paused for users with existing corpora, breaking an active agent recall workflow until manual re-embedding.
- `impact:data-loss`: The existing indexed memory dataset remains on disk but is silently dropped from the active runtime path after upgrade.
- `impact:session-state`: The active agent memory index state drifts from the user's persisted corpus and reports missing identity in the new per-agent store.

## Summary

Keep open: latest release v2026.6.9 and current main still use the per-agent SQLite database for built-in memory while the visible migration path removes legacy store-path config and normalizes tables only inside the opened database, so the reported upgrade-time loss of the active legacy index remains unresolved.

## What This Changes

The issue reports that upgrading from 2026.6.8 to 2026.6.9 moves built-in memory from the legacy global SQLite index to the per-agent SQLite DB without importing or warning, forcing a full re-embed.

## Best Possible Solution

Add a doctor/update-owned migration or explicit upgrade warning for existing legacy memory DBs before or during the per-agent DB cutover, with regression coverage for an already-indexed corpus and no steady-state runtime fallback reader.

## Reproduction Assessment

Yes, source-reproducible. The reporter supplied concrete upgrade/status/file-size evidence, and source plus release inspection shows v2026.6.9/current main opens the per-agent DB while visible migrations do not import the separate legacy memory DB.

## Solution Assessment

No. The DB-first runtime target matches current architecture, but without a doctor/update migration or explicit upgrade warning it is not the best solution for existing indexed corpora.
