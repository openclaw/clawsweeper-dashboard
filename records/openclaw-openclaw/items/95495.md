## Label Justifications

- `P1`: The latest stable upgrade can leave memory_search paused for users with existing corpora, breaking an active agent recall workflow until manual re-embedding.
- `impact:data-loss`: The existing indexed memory dataset remains on disk but is silently dropped from the active runtime path after upgrade.
- `impact:session-state`: The active agent memory index state drifts from the user's persisted corpus and reports missing identity in the new per-agent store.

## Summary

Keep open: current main still uses the per-agent SQLite database as the built-in memory index store, while the inspected migration paths remove legacy config keys and migrate in-database legacy tables but do not import or warn about an existing separate legacy memory database.

## What This Changes

The issue reports that upgrading to v2026.6.9 moves the built-in memory vector store from the legacy global SQLite file to the per-agent SQLite database without migrating existing embeddings or warning operators.

## Best Possible Solution

Add a doctor/update-owned migration or explicit upgrade warning for existing legacy memory index databases before the per-agent DB cutover, with regression coverage for an already-indexed corpus and no permanent runtime fallback reader.

## Reproduction Assessment

Yes, source-reproducible with reporter evidence: current main resolves built-in memory to the per-agent DB while the inspected migrations do not import the separate legacy global DB, matching the reported 0/1499 post-upgrade status. I did not run a live upgrade in this read-only review.

## Solution Assessment

No. Current main is not the best solution yet because it documents and uses the new per-agent DB but does not migrate or warn about an existing separate legacy memory index before the old corpus disappears from active search.
