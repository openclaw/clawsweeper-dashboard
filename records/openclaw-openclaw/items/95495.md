## Summary

Keep open: current main and v2026.6.9 still use the per-agent SQLite database for built-in memory, while the visible legacy handling removes `memorySearch.store.path` and migrates legacy tables only when they are already in the opened database. I found no current import or upgrade warning for an existing separate `memory/main.sqlite` corpus.

## What This Changes

The issue reports a 2026.6.9 upgrade regression where the built-in memory index moves from the legacy global SQLite store to the per-agent database without migrating or warning, forcing a full re-embed.

## Best Possible Solution

Add a doctor/update-owned migration or explicit upgrade warning for existing legacy memory DBs before or during the per-agent DB cutover, with regression coverage for an already-indexed corpus and no steady-state runtime fallback reader.

## Reproduction Assessment

Yes, source-reproducible. Current main and v2026.6.9 route built-in memory to the per-agent database, while the reporter provided concrete upgrade output showing the old populated DB untouched and the new store empty.

## Solution Assessment

No. Current main is not the best solution yet because it documents and uses the new per-agent DB but does not migrate or warn about an existing separate legacy memory index before the old corpus disappears from active search.
