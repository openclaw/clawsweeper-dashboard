## Label Justifications

- `P1`: The latest stable upgrade can leave memory_search paused for users with existing corpora, breaking a real agent recall workflow until manual re-embedding.
- `impact:data-loss`: The prior indexed memory dataset remains on disk but is silently dropped from the active runtime path after upgrade.
- `impact:session-state`: The active agent memory index state drifts from the user's persisted corpus and reports missing identity in the new per-agent store.

## Summary

Keep open: this is a well-evidenced latest-release regression, and current main still routes built-in memory indexes to the per-agent SQLite database without a visible migration/import path for an existing legacy memory DB.

## What This Changes

The issue requests an upgrade-safe migration or warning when built-in memory moves from the legacy global memory SQLite store to the per-agent OpenClaw agent database.

## Best Possible Solution

Add a doctor/update-owned legacy memory DB migration or explicit upgrade warning before switching users to the per-agent DB, with regression coverage for an existing indexed corpus and no permanent runtime fallback reader.

## Reproduction Assessment

Yes, source-reproducible with strong reporter evidence: upgrading an existing legacy memory DB to current code opens the per-agent DB path, while the old global DB is neither imported nor referenced by runtime.

## Solution Assessment

No, current main is not the best solution yet: it documents and uses the new per-agent DB but only removes the retired config key, leaving existing indexed corpora without migration or upgrade-time warning.
