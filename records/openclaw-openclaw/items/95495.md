## Summary

Keep open: current main and shipped v2026.6.9 still route built-in memory to the per-agent SQLite database, while current migration code only handles legacy memory tables already inside that opened database. The central upgrade problem is still owned by open candidate PRs, so the issue should remain open until one lands or is replaced.

## What This Changes

The issue reports that upgrading to v2026.6.9 moves memory indexing from the legacy global SQLite store to the per-agent database without migration or warning, forcing a full re-embed.

## Best Possible Solution

Land one canonical doctor-owned Memory Core migration that imports legacy sidecar indexes into the existing per-agent database without steady-state runtime fallback, then close this issue when the fix is merged and release provenance is known.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.9 route built-in memory to the per-agent SQLite DB while the reporter and commenters provide field evidence that populated legacy sidecars remain outside the active store.

## Solution Assessment

No. Current main is not the best solution because it changes the canonical store without importing or warning about existing sidecar data; the best path is a maintainer-approved one-shot doctor/Memory Core migration, likely through https://github.com/openclaw/openclaw/pull/95631 or a replacement.
