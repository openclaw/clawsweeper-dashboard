## Label Justifications

- `P1`: The shipped upgrade path can leave semantic memory search paused for existing corpora until a manual full re-embed completes.
- `impact:data-loss`: The existing indexed memory dataset remains on disk but is silently dropped from the active runtime path after upgrade.
- `impact:session-state`: The active agent memory index state can drift from the user's persisted corpus and report missing identity in the new per-agent store.

## Summary

Keep open: current main and v2026.6.9 still route built-in memory to the per-agent SQLite database while the only visible legacy handling removes `memorySearch.store.path` and migrates legacy tables only when they are already inside the opened database. I found no current import or upgrade warning for an existing separate legacy memory index, and the issue discussion adds a second user report that downgrade restored the old index path.

## What This Changes

The issue reports that upgrading from v2026.6.8 to v2026.6.9 moves memory search from the legacy global SQLite store to the per-agent SQLite store without importing or warning about the existing embeddings.

## Best Possible Solution

Add a doctor/update-owned migration or explicit upgrade warning for existing legacy memory DBs before or during the per-agent DB cutover, with regression coverage and no permanent runtime fallback reader.

## Reproduction Assessment

Yes. Current main and v2026.6.9 route built-in memory to the per-agent database, while the report and comments show the old populated DB left untouched and the new store empty after upgrade.

## Solution Assessment

No. Current main documents and uses the new per-agent DB, but it is not the best solution until the upgrade path either preserves the separate legacy index or clearly warns before forcing a rebuild.
