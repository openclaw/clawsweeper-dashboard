## Label Justifications

- `P1`: The shipped upgrade path can leave semantic memory search paused for existing corpora until a manual full re-embed completes.
- `impact:data-loss`: An existing indexed memory corpus remains on disk but is silently dropped from the active runtime path after upgrade.

## Summary

Keep open: current main and shipped v2026.6.9 still route built-in memory to the per-agent SQLite database, while doctor removes legacy `memorySearch.store.path` and the schema migration only handles legacy tables inside the database that is already opened. I found no current import or upgrade warning for an existing separate `memory/main.sqlite` corpus.

## What This Changes

The issue reports that upgrading to v2026.6.9 silently moves built-in memory from the legacy global SQLite store to the per-agent database, leaving the old embedded corpus inactive and requiring a full reindex.

## Best Possible Solution

Add a doctor/update-owned migration or explicit upgrade warning for existing legacy memory databases before or during the per-agent database cutover, with regression coverage for an already-indexed corpus and no permanent runtime fallback reader.

## Reproduction Assessment

Yes, source-reproducible. Current main and v2026.6.9 route built-in memory to the per-agent database, while the reporter and a second commenter provide upgrade evidence that the old populated DB remains untouched and the new active store starts empty.

## Solution Assessment

No. Current main is not the best solution yet because it documents and uses the new per-agent DB but does not migrate or warn about an existing separate legacy memory index before that corpus disappears from active search.
