## Summary

Keep open: current main still lacks the reported hex-suffix recovery, but this PR should not merge as-is because the tail-only dropped-leading-character recovery can dispatch the wrong allowlisted tool, and the PR still lacks real-provider proof plus focused regression tests.

## What This Changes

The PR adds a reverse-lookup fallback in embedded-runner tool-call normalization to map malformed streamed tool names back to allowlisted names, including underscore-stripped, dot-prefixed, dropped-leading-character, and hex-suffixed variants.

## Best Possible Solution

Land a narrow embedded-runner normalization fix that recovers strict provider-prefixed or hash-suffixed allowlisted names, removes or structurally gates tail-only exact recovery, and adds regression tests for both recovery and fail-closed collisions before requiring redacted provider proof.

## Reproduction Assessment

Yes, from source inspection: functionsread5f3787893b0d4c49a with read allowlisted strips to read5f3787893b0d4c49a and remains unresolved on current main. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

No. Strict prefix plus hash-suffix recovery is a reasonable direction, but tail-only exact recovery should be removed or gated on structural evidence before this dispatch-sensitive path lands.
