## Summary

Keep open: the current-main bug is still present and the PR is focused, but the patch should not merge as-is because tail-only exact recovery can dispatch the wrong allowlisted tool and the PR still lacks real behavior proof.

## What This Changes

The PR adds reverse lookup in the embedded runner tool-call normalizer for malformed streamed tool names, including underscore-stripped, dot-prefixed, dropped-leading-character, and hex-suffixed variants.

## Best Possible Solution

Land a narrow embedded-runner normalizer fix that only recovers structurally signaled malformed names, fails closed on ambiguous or tail-only collisions, and includes colocated regression tests plus redacted real-provider proof.

## Reproduction Assessment

Yes. Source inspection shows current main will not map functionsread5f3787893b0d4c49a to read, and the PR head can map raw search to msearch when only msearch is allowlisted; I did not run tests because this review is read-only.

## Solution Assessment

No. Prefix-plus-strict-hash recovery is a reasonable direction, but exact dropped-leading-character recovery is too broad for dispatch-sensitive code; remove or structurally gate it and add focused tests/proof.
