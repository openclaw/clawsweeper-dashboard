## Summary

Keep open: the PR is still addressing a real gap on current main, but the latest patch is not merge-ready because it lacks real provider proof, lacks regression tests, and still permits tail-only exact recovery that can dispatch the wrong allowlisted tool.

## What This Changes

The PR adds reverse lookup in embedded-runner tool-call normalization for underscore-stripped, dot-prefixed, dropped-leading-character, and hex-suffixed malformed tool names.

## Best Possible Solution

Land a narrower embedded-runner normalization fix that recovers strict provider-prefixed or hex-suffixed allowlisted names, fails closed on tail-only or ambiguous variants, and includes focused tests plus redacted provider proof.

## Reproduction Assessment

Yes, from source inspection: `functionsread5f3787893b0d4c49a` with `read` allowlisted strips to `read5f3787893b0d4c49a` and remains unresolved on current main. I did not run a live provider reproduction.

## Solution Assessment

No. Strict prefix plus hash-suffix recovery is a reasonable direction, but dropped-leading-character exact remapping should be removed or gated on independent structural evidence before this dispatch-sensitive path lands.
