## Summary

Keep this PR open: it targets a real embedded-runner normalization gap that current main still has, but the branch is not merge-ready because tail-only exact recovery can turn an unknown tool name into an allowlisted dispatch and the external PR still lacks after-fix real provider proof.

## What This Changes

The PR adds reverse lookup to embedded-runner tool-call normalization so malformed allowlisted tool names can be recovered from underscore-stripped, dot-prefixed, leading-character-dropped, and hex-suffixed model output.

## Best Possible Solution

Land a narrow resolver change that recovers only structurally evidenced malformed allowlisted names, removes or tightly gates tail-only exact recovery, and adds regression tests for both recovery and fail-closed collisions before real provider proof is accepted.

## Reproduction Assessment

Yes, by source inspection: current main leaves functionsread5f3787893b0d4c49a unresolved against an allowlisted read tool after prefix stripping. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

No. Strict prefix plus hash-suffix recovery is a reasonable fix direction, but tail-only exact recovery should be removed or gated on stronger structural evidence before this dispatch-sensitive path lands.
