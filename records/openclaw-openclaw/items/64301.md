## Summary

Keep open: current main still lacks the proposed reverse lookup, but this PR changes dispatch-sensitive normalization and still accepts dropped-leading-character exact recovery without an independent malformed-name signal; it also lacks attached real behavior proof and focused regression coverage.

## What This Changes

The PR modifies `src/agents/pi-embedded-runner/run/attempt.tool-call-normalization.ts` to reverse-match malformed streamed tool names against allowlisted variants, including underscore-stripped, dot-prefixed, dropped-leading-character, and digit-bearing hex-suffixed forms.

## Best Possible Solution

Land a narrow embedded-runner normalizer fix that recovers only structurally signaled allowlisted malformed names, fails closed on ambiguity, and adds colocated tests for recovery and non-recovery cases.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main will not map `functionsread5f3787893b0d4c49a` to `read`, and the PR diff can map `search` to `msearch` when only `msearch` is allowlisted. I did not run tests because this review is read-only.

## Solution Assessment

No: strict hash-suffix recovery is a reasonable direction, but tail-only exact recovery is too broad for dispatch-sensitive code and the branch lacks focused regression tests and real behavior proof. The safer fix is to remove or structurally gate dropped-leading-character recovery.
