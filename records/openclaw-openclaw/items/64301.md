## Summary

Keep open: the PR targets a real current-main normalization gap, but the branch still broadens dispatch-sensitive tool-name recovery with an under-evidenced tail-only exact-match path, has no regression coverage for the new collision cases, and lacks contributor-provided real behavior proof.

## What This Changes

The PR adds reverse lookup in embedded-runner tool-call normalization to recover malformed streamed tool names with underscore stripping, dot prefixes, leading-character drops, and hex suffixes.

## Best Possible Solution

Land a narrow resolver that recovers only structurally evidenced malformed allowlisted names, fails closed on tail-only or ambiguous collisions, adds focused regression tests, and then includes redacted provider proof.

## Reproduction Assessment

Yes, by source inspection: current main leaves functionsread5f3787893b0d4c49a unresolved against an allowlisted read tool after stripping the functions prefix because only exact/case-insensitive candidates and digit suffix stripping are attempted. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

No. Strict prefix plus strong hash-suffix recovery is a reasonable direction, but exact dropped-leading-character recovery should be removed or gated on stronger structural evidence before this dispatch-sensitive path lands.
