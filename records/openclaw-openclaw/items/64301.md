## Summary

Keep open. Current main still lacks reverse lookup for the reported malformed hex/underscore tool names, so the PR addresses a live dispatch gap, but the proposed tail-only exact recovery can coerce an unrelated raw tool name into an allowlisted tool and is not merge-ready.

## What This Changes

The PR adds allowlist-backed reverse lookup in the embedded runner tool-call normalizer for underscore-stripped, dot-prefixed, dropped-leading-character, and digit-bearing hex-suffixed malformed tool names.

## Best Possible Solution

Land a narrow embedded-runner normalization fix that keeps strict allowlist-backed hash recovery, preserves existing delimiter and alias behavior, and fails closed for exact, prefix, underscore-stripped, dropped-leading-character, and cross-plugin collisions with focused regression tests.

## Reproduction Assessment

Yes: the proposed patch can be reasoned through directly by using allowedToolNames = {"msearch"} and rawName = "search"; the new substring(1) variant enters exactMatches and returns msearch. Current main also provides a high-confidence negative baseline because it has no reverse lookup path for the PR's reported read5f-style names.

## Solution Assessment

No: the prefix-plus-strict-hash portion is a reasonable direction, but accepting dropped-leading-character exact matches without another structural signal is too broad for a dispatch-sensitive normalizer. A safer fix should remove that exact tail recovery or gate it behind provider prefix/id/hash evidence and ambiguity tests.
