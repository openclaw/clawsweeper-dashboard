## Summary

Keep open: current main does not implement the requested hash/underscore recovery, but this PR still introduces a wrong-tool dispatch risk through tail-only exact recovery in a dispatch-sensitive normalizer.

## What This Changes

The PR adds a reverse-lookup fallback in the embedded runner tool-call normalizer for underscore-stripped, dot-prefixed, dropped-leading-character, and digit-bearing hex-suffixed malformed tool names.

## Best Possible Solution

Land a narrow normalization fix that keeps strict allowlist-backed hash recovery, fails closed on collisions, removes or gates tail-only exact recovery, and adds focused regression tests for both recovery and non-recovery cases.

## Reproduction Assessment

Yes: the PR diff can be reasoned through with allowedToolNames = new Set(["msearch"]) and rawName = "search"; the substring(1) variant is accepted as a single exact match and resolves to msearch. Current main also gives a clear baseline for the original gap because read5f-style suffixes are not recovered.

## Solution Assessment

No: the strict digit-bearing hash-suffix recovery is a reasonable direction, but tail-only exact recovery is too broad for dispatch-sensitive code. A safer solution should remove that heuristic or require a provider/id/hash/prefix signal, then lock the behavior with regression tests.
