## Summary

## What This Changes

The branch emits latest-wins `progress` steps from wizard session progress calls, rewrites `next()` to wake and reclassify events, and adds `WizardSession` tests for delivery, cancellation, non-answerability, and concurrency.

## Best Possible Solution

Emit progress in-band while preserving or explicitly gating current RPC clients, then document/test the non-interactive progress contract and add redacted real RPC-client proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main drops `WizardSessionPrompter.progress()` updates, and the PR returns progress ids that current bundled clients answer while `session.answer()` rejects unregistered step ids. I did not run a live RPC client in this read-only review.

## Solution Assessment

No. Emitting progress from `WizardSession` is the right owner boundary, but the best fix must preserve or explicitly gate current RPC clients and include real-client proof.
