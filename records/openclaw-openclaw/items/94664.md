## Summary

Keep open: the progress-delivery idea is valid, but the patch currently breaks bundled RPC wizard clients that already treat `progress` as answerable, and the submitted proof is only tests/type/lint rather than a real client run.

## What This Changes

The PR changes `WizardSessionPrompter.progress()` to emit latest-wins non-interactive `progress` wizard steps and adds session tests for delivery, wakeup, non-answerability, and cancel cleanup.

## Best Possible Solution

Keep the core progress-delivery fix, but make progress steps backwards-compatible for existing RPC clients, update bundled client/docs/tests around the non-interactive contract, and add real RPC-client proof before merge.

## Reproduction Assessment

Yes, source-reproducible: bundled macOS UI and CLI code submit answers for `progress`, while the patch deliberately leaves progress IDs unanswerable. I did not run a live app, but the current source path is direct.

## Solution Assessment

No. Emitting progress from `WizardSession` is the right owner boundary, but the PR needs compatibility handling for existing RPC clients before this is the best fix.
