## Summary

Keep open: the progress-delivery direction addresses a real RPC wizard gap, but the current patch would regress existing clients that answer progress steps and it still lacks real RPC-client proof.

## What This Changes

This PR makes `WizardSessionPrompter.progress()` emit latest-wins non-interactive progress steps over `wizard.next` and adds `WizardSession` tests for delivery, wakeup, non-answerability, and cancel cleanup.

## Best Possible Solution

Emit progress in-band while preserving existing RPC clients, then document the non-interactive progress contract and add redacted real-client proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main drops progress, and the patch returns progress IDs that existing bundled clients answer while `session.answer()` rejects IDs not registered in `answerDeferred`. I did not run a live client in this read-only review.

## Solution Assessment

No. Emitting progress from `WizardSession` is the right owner boundary, but the patch needs compatibility handling for existing clients before this is the best fix.
