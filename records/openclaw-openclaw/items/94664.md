## Summary

Keep this PR open: it fixes a real source-level RPC wizard progress gap, but the current head still breaks existing progress-answering wizard clients and still has only test-based proof.

## What This Changes

The branch changes `WizardSessionPrompter.progress()` to emit latest-wins non-interactive `progress` steps over `wizard.next` and adds `WizardSession` tests for progress delivery, cancellation, non-answerability, and concurrent `next()` callers.

## Best Possible Solution

Emit progress in-band while preserving or gating current RPC clients, then document/test the non-interactive contract and add redacted real RPC-client proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main drops `WizardSessionPrompter.progress()` updates, and the PR returns progress ids that current bundled clients answer while `session.answer()` rejects unregistered progress step ids. I did not run a live RPC client in this read-only review.

## Solution Assessment

No. Emitting progress from `WizardSession` is the right owner boundary, but the best fix must preserve or explicitly gate current RPC clients and include real-client proof.
