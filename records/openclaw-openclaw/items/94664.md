## Summary

Keep open: the PR addresses a real RPC wizard progress gap, but it is not merge-ready because it emits unanswerable progress steps to all clients while current bundled wizard clients still answer progress steps, and the submitted proof is test-only.

## What This Changes

The branch changes `WizardSessionPrompter.progress()` to emit latest-wins non-interactive `progress` steps over `wizard.next` and adds `WizardSession` tests for delivery, cancellation, non-answerability, and concurrent `next()` callers.

## Best Possible Solution

Emit progress in-band while preserving or gating current RPC clients, then document/test the non-interactive progress contract and add redacted real RPC-client proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main drops `WizardSessionPrompter.progress()` updates, and the PR head returns unanswerable progress ids that current bundled clients still answer; I did not run a live RPC client in this read-only review.

## Solution Assessment

No. Emitting progress from `WizardSession` is the right owner boundary, but the best fix must preserve or explicitly gate current RPC clients and include real-client proof.
