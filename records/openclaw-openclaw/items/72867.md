## Summary

Keep open. Current main has adjacent Telegram startup, polling liveness, retry, and generic stuck-session hardening, but I did not find source, tests, release notes, maintainer confirmation, or reporter retest proving the session-specific Telegram conversational reply delivery failure is fixed. The normal reply path is still distinct from the direct message tool path, and the likely polling-restart reply-loss path remains represented by the open targeted PR.

## What This Changes

The issue asks for Telegram conversational replies to keep delivering reliably in a session even when direct `message` tool sends still work and `openclaw doctor` reports a healthy Telegram connection.

## Best Possible Solution

Keep this as the Telegram/session-delivery tracking issue until maintainers land or supersede the targeted polling-restart fix, add regression or live proof that conversational Telegram replies survive polling restart and session recovery, and keep the broader non-Telegram latency work tracked separately, such as #74911.
