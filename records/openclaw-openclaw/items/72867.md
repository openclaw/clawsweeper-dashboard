## Summary

Keep open. Current main has adjacent Telegram startup, timeout, and watchdog hardening, but the session-specific conversational reply stall is not proven fixed: normal replies and direct message-tool sends still use different paths, and main still aborts the shared Telegram fetch signal during polling restart paths while #73050 remains the targeted open fix.

## What This Changes

Telegram conversational replies should consistently deliver in a session when the connection is healthy and direct `message` tool sends work.

## Best Possible Solution

Keep this as the Telegram session-delivery tracker until maintainers land or supersede #73050 with equivalent scoped restart-vs-shutdown abort behavior and retest the original symptom.
