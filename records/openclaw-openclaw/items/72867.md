## Summary

Keep open. Current main has adjacent Telegram polling, startup, and liveness hardening, but it does not prove the reported session-specific conversational reply delivery failure is fixed: normal Telegram replies still travel through the session turn pipeline, direct `message` tool sends still use a separate action path, and polling restart still aborts the shared Telegram fetch controller rather than only `getUpdates`.

## What This Changes

Ensure standard conversational replies from Telegram sessions are delivered consistently when direct Telegram message tool sends still work and `/reset` only temporarily clears the failure.

## Best Possible Solution

Keep this as the Telegram session-delivery tracker until maintainers land or supersede #73050 with equivalent scoped polling-restart behavior, add regression or live proof that conversational Telegram replies survive polling restart/session recovery, and keep broader non-Telegram latency reports tracked separately.

## Reproduction Assessment

Unclear. The report gives plausible manual steps and useful diagnostics, but the failure is session-specific and eventually recurring; I found code evidence for distinct direct-send and conversational-reply paths, not a deterministic current-main reproduction.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable direction is to review and land #73050 or an equivalent scoped restart-abort fix with coverage, then retest the original session-delivery symptom.
