## Summary

Keep open: current main and v2026.6.1 already include a Feishu per-chat queue timeout, but the central self-message feedback-loop guard is still missing from the Feishu message receive path.

## What This Changes

The issue reports a Feishu DM failure mode where slow LLM runs serialize same-chat messages, lack effective timeout/backpressure, echo bot-sent messages as inbound input, and lose observability while session writes are locked.

## Best Possible Solution

Split the work: add a connector-owned Feishu bot/app self-message filter with regression tests, then route true task abortion, queue-depth backpressure, and collect-mode behavior through the existing Feishu queue/session follow-ups.

## Reproduction Assessment

Yes for the source-level self-message gap: Feishu message receive events are parsed, claimed, debounced, and dispatched without an early sender_type app or bot-open-id filter, unlike sibling channel guards. I did not run a live Feishu WebSocket or slow-LLM incident reproduction in this read-only review.

## Solution Assessment

A Feishu ingress self-message filter is the best narrow first fix because it matches sibling channel patterns and avoids adding config. The requested timeout configurability, queue-depth backpressure, run abortion, and per-chat concurrency are broader policy choices and overlap with https://github.com/openclaw/openclaw/issues/85692 and https://github.com/openclaw/openclaw/issues/54409.
