## Summary

The issue remains necessary: current main still admits Feishu receive events without a self-sender check, and its five-minute queue cap releases later same-chat work without aborting the original run. A focused, clean PR with sufficient real-environment proof covers only the self-echo portion, while live v2026.6.5 evidence and current source support separate unresolved queue/session-lock behavior. The umbrella should stay open until maintainers land or reject the narrow PR and route the remaining concurrency and lock work to canonical issues.

## What This Changes

Prevent Feishu slow-run incidents from starving later messages, overlapping session work, losing transcript observability, or recursively processing the bot’s own receive echoes.

## Best Possible Solution

Land the narrow bot-open-id receive-boundary guard if maintainers accept its disclosed transport-proof gap, route active-run follow-up handling to https://github.com/openclaw/openclaw/issues/54409, resolve non-aborting timeout/session overlap under https://github.com/openclaw/openclaw/issues/85692, and keep transcript-lock repair such as https://github.com/openclaw/openclaw/pull/88919 independently scoped unless an end-to-end reproduction proves one shared cause.

## Reproduction Assessment

Partly yes: released v2026.6.5 logs reproduce the Feishu queue delay and same-process session-lock symptoms, and current-main source preserves the implicated non-aborting queue path. The exact natural self-authored WebSocket echo was not reproduced by the follow-up reporter, so the complete three-defect cascade is not yet a single high-confidence reproduction.

## Solution Assessment

No as one bundled repair: the bot-open-id receive filter is the narrowest maintainable mitigation for proven matching events, but cancellation, backpressure, active-run follow-ups, and transcript locking cross different ownership and require separate semantics and validation.
