## Summary

Keep open. Current main still isolates each agent's QMD index under its own agent state directory and has no OpenClaw-owned WAL/checkpoint/vacuum, shared-cache topology, reduced-write mode, or cache-size controls; related merged QMD work reduced embed stampedes and model duplication but did not resolve the reported SQLite write amplification.

## What This Changes

The issue asks OpenClaw to reduce QMD disk write amplification from per-agent SQLite caches through shared-cache or maintenance/bounding controls.

## Best Possible Solution

Keep this as a QMD storage/backend design issue and choose a supported contract: either a safe shared agent-namespaced index or bounded per-agent maintenance with explicit cache limits, docs, and multi-agent validation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the full macOS disk-write volume: source inspection reproduces the per-agent WAL-backed QMD index topology and missing maintenance controls, but no current-main 19-agent I/O run was performed.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the requested mitigations are plausible, but the narrowest maintainable fix depends on a maintainer decision about shared storage versus automatic per-agent maintenance and new config surface.
