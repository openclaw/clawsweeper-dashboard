## Summary

Keep open. Current main still creates per-agent QMD homes and runs QMD update/embed/search without an OpenClaw-owned cleanup, compaction, cache-size, shared-cache, or reduced-write policy; upstream QMD exposes cleanup/vacuum machinery, but OpenClaw has not integrated it for the reported multi-agent disk-I/O problem.

## What This Changes

Request QMD storage maintenance or redesign so multi-agent deployments do not multiply per-agent SQLite cache growth and WAL write amplification.

## Best Possible Solution

Define an owner-reviewed QMD cache maintenance plan, likely starting with measured integration of upstream `qmd cleanup`/vacuum where safe, with docs, focused tests, and multi-agent disk-I/O proof.

## Reproduction Assessment

No. Source inspection confirms the per-agent QMD home and missing OpenClaw cleanup/size-policy path, but I did not run a 19-agent macOS measurement to reproduce the reported 34.36GB writes.

## Solution Assessment

Unclear. A narrow integration of upstream `qmd cleanup` may be promising, but shared caches, WAL tuning, and reduced-write modes need QMD/OpenClaw owner judgment and before/after disk-I/O proof.
