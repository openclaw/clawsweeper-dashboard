## Summary

Keep open. Current main and the latest release still route QMD through a per-agent `XDG_CACHE_HOME` and `qmd/index.sqlite`; related merged and open work reduces adjacent contention or session-export churn, but it does not settle the multi-agent cache layout, compaction, eviction, or write-amplification problem.

## What This Changes

The issue requests a fix for extreme disk I/O and availability impact from per-agent QMD SQLite caches in multi-agent deployments.

## Best Possible Solution

Have the memory/QMD owner choose one canonical cache-layout or maintenance policy, then validate it with a live multi-agent workload measuring cache size, WAL sidecars, disk writes, and gateway recovery.

## Reproduction Assessment

Yes for the core layout claim from source: current main and v2026.6.8 still resolve QMD to `agents/<agentId>/qmd/xdg-cache/qmd/index.sqlite`. No for the exact reported 34 GB over 20 hours disk-write magnitude because this read-only review did not run a live 19-agent macOS workload.

## Solution Assessment

No current solution is complete. Existing main has partial QMD startup, same-store writer-lock, and per-agent cache-store work, but no maintainer-chosen policy for shared QMD cache layout, compaction, eviction, reduced-write mode, or measured multi-agent disk-write control.
