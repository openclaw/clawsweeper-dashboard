## Summary

Keep open. Current main and the latest release still put QMD’s own `index.sqlite` under each agent’s QMD cache home, while the merged per-agent SQLite cache scaffold is not wired into production QMD and the merged SQLite-lock fix only addresses writer contention, not cache size or write amplification.

## What This Changes

The issue requests reducing QMD per-agent SQLite cache write amplification in multi-agent deployments through shared cache, compaction, reduced-write mode, or cache limits.

## Best Possible Solution

Keep this issue open for the memory/QMD owner to choose a cache-layout or maintenance policy, then validate it with a live multi-agent QMD workload that measures disk writes, WAL growth, cache size, and agent recovery behavior.

## Reproduction Assessment

Yes for the core layout claim from source: QMD still resolves a per-agent XDG cache at `agents/<agentId>/qmd/xdg-cache/qmd/index.sqlite`. No for the reported 34 GB/day disk-write magnitude because I did not run a live 19-agent macOS workload.

## Solution Assessment

No. Current main has partial mitigations for QMD startup and SQLite writer contention, but it still preserves one QMD cache/index per agent and has no visible compaction, eviction, reduced-write, or shared-cache policy for this report.
