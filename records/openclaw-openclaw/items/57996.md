## Summary

Keep open. Current main still gives each QMD-enabled agent its own XDG cache and SQLite index, and the requested shared-cache, WAL/checkpoint/vacuum, reduced-write, and cache-size controls are not present; the existing startup/embed mitigations only reduce some concurrency pressure.

## What This Changes

The issue requests reducing multi-agent QMD disk write amplification from per-agent SQLite caches, with possible shared cache, WAL/checkpoint tuning, reduced-write mode, and cache-size/LRU controls.

## Best Possible Solution

Keep this as a memory-core/QMD backend design issue and choose either a shared agent-namespaced QMD home/index or bounded per-agent maintenance and cache-size controls, with docs and multi-agent tests for the selected contract.

## Reproduction Assessment

Partial. Current source deterministically reproduces the per-agent QMD cache path for each agent, but the extreme macOS write-volume claim needs a multi-agent live measurement that was not run in this read-only review.

## Solution Assessment

Unclear. The suggested fixes are plausible, but the narrowest maintainable solution depends on whether OpenClaw can safely control QMD SQLite maintenance or needs upstream/shared-index support first.
