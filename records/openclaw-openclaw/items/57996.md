## Summary

Keep open. Current main still uses per-agent QMD SQLite homes and has no OpenClaw-owned QMD cache compaction, WAL checkpoint, shared-cache, or cache-size policy; existing mitigations cover startup/embed stampedes and busy timeouts but not the reported disk-write/cache-growth problem.

## What This Changes

Request to reduce disk-write amplification from per-agent QMD SQLite caches in multi-agent gateways via shared cache, WAL/compaction tuning, reduced-write mode, or cache limits.

## Best Possible Solution

Define an owner-reviewed QMD cache maintenance plan, then implement the narrowest measured fix with docs and multi-agent disk-I/O proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms the per-agent QMD index path and missing cache-maintenance controls, but the reported 34GB write volume needs a multi-agent runtime measurement to reproduce fully.

## Solution Assessment

Unclear. The suggested shared cache, WAL tuning, reduced-write mode, and eviction controls are plausible but require QMD/OpenClaw owner judgment; a measured maintenance/limits design is safer than a drive-by SQLite pragma change.
