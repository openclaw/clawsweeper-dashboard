## Summary

Keep open. Current main and v2026.6.10 still route QMD through an agent-local `XDG_CACHE_HOME` and `qmd/index.sqlite`; merged related work reduces same-store writer contention or adds adjacent per-agent cache infrastructure, but it does not settle the multi-agent cache layout, compaction, eviction, or write-amplification problem.

## What This Changes

The issue reports heavy per-agent QMD SQLite cache I/O on multi-agent deployments and asks for shared cache, checkpointing, reduced-write mode, or cache bounds.

## Best Possible Solution

Have the memory/QMD owner choose one canonical cache-layout or maintenance policy, then validate it with a live multi-agent workload measuring cache size, WAL sidecars, disk writes, and gateway recovery.

## Reproduction Assessment

Yes for the core layout claim from source: current main still gives each agent its own QMD `xdg-cache/qmd/index.sqlite`. No for the exact reported disk-write magnitude because this read-only review did not run a live 19-agent macOS workload.

## Solution Assessment

No current solution is complete. Existing main has adjacent same-store writer locking and per-agent cache-store infrastructure, but no maintainer-chosen shared-cache, compaction, eviction, or reduced-write policy for QMD.
