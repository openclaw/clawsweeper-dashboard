## Summary

Keep open. Current main and the latest release still use a per-agent QMD home and SQLite index path, with no shared-cache, WAL checkpoint, vacuum, reduced-write, or QMD cache-size policy; related merged work reduces embed stampedes and watcher FD pressure but does not address QMD SQLite cache growth or write amplification.

## What This Changes

The issue asks OpenClaw to reduce QMD per-agent SQLite cache disk I/O for large multi-agent deployments through cache sharing, maintenance, reduced-write mode, or eviction controls.

## Best Possible Solution

Define an owner-reviewed QMD cache maintenance and storage policy, then implement the smallest measured change with docs, config/schema alignment if needed, and multi-agent disk-I/O proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms the per-agent QMD SQLite path and missing OpenClaw cache-maintenance controls, but the reported macOS multi-agent write volume still needs measured reproduction.

## Solution Assessment

Unclear. Shared cache, scheduled QMD cleanup, WAL/checkpoint tuning, reduced-write mode, and eviction caps are plausible, but the best path depends on QMD ownership boundaries and before/after I/O measurements.
