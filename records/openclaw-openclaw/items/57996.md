## Summary

Keep open. Current main and v2026.6.10 still route QMD through an agent-local `XDG_CACHE_HOME` and `qmd/index.sqlite`; related work only addresses same-store writer contention or adjacent per-agent cache infrastructure, not the multi-agent cache layout, compaction, eviction, or write-amplification problem.

## What This Changes

The issue asks OpenClaw to reduce QMD per-agent SQLite cache write amplification in multi-agent deployments, potentially via shared cache, checkpoint tuning, reduced-write mode, or eviction.

## Best Possible Solution

Have the memory/QMD owner choose a canonical cache-layout or maintenance policy, prefer existing QMD cleanup hooks if they are sufficient, and validate it with a live multi-agent workload measuring DB size, WAL sidecars, disk writes, and gateway recovery.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the core layout claim from source: current main still gives each agent its own QMD `xdg-cache/qmd/index.sqlite`; no for the exact 19-agent macOS disk-write magnitude because this review did not run that live workload.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the suggested fixes are plausible, but upstream QMD already has cleanup/vacuum hooks and OpenClaw needs a maintainer-owned choice between automatic maintenance, shared storage, eviction, or explicit config.
