## Summary

Keep open. Current main and the latest shipped tag still use per-agent QMD homes with per-agent `index.sqlite` files, and neither config/schema/docs nor the QMD update path exposes shared-cache, checkpoint/vacuum, reduced-write, or cache-size policy; the related shared-memory, lock-contention, and active-memory reliability items are adjacent but do not resolve this disk-I/O report.

## What This Changes

The issue requests a QMD cache/storage maintenance policy for multi-agent deployments to reduce per-agent SQLite WAL write amplification and unbounded cache growth.

## Best Possible Solution

Define an owner-approved QMD cache storage and maintenance policy, then implement the smallest measured change with config/schema/docs alignment and before/after multi-agent disk-I/O proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms the per-agent QMD SQLite topology and missing cache-maintenance controls, but the reported macOS multi-agent write volume still needs measured validation.

## Solution Assessment

Unclear. Shared cache, scheduled maintenance, WAL/checkpoint tuning, reduced-write mode, and eviction caps are plausible, but the best fix depends on QMD ownership boundaries, compatibility, and before/after I/O measurements.
