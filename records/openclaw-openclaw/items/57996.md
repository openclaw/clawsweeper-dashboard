## Summary

Keep open. Current main and v2026.6.8 still route QMD to a per-agent `XDG_CACHE_HOME` and `index.sqlite`; related merged work adds an agent cache scaffold or serializes same-store writers, but does not address multi-agent QMD cache size, compaction, eviction, or write amplification.

## What This Changes

Reduce extreme disk I/O from QMD’s per-agent SQLite caches in multi-agent deployments, potentially through shared cache layout, maintenance, reduced-write mode, or cache limits.

## Best Possible Solution

Have the memory/QMD owner choose one canonical cache-layout or maintenance policy, then validate it with a live multi-agent QMD workload measuring cache size, WAL sidecars, disk writes, and gateway recovery.

## Reproduction Assessment

Yes for the core layout claim from source: current main and v2026.6.8 still resolve QMD to `agents/<agentId>/qmd/xdg-cache/qmd/index.sqlite`. No for the exact reported 34 GB/20h disk-write magnitude because this read-only review did not run a live 19-agent macOS workload.

## Solution Assessment

No. Current main has partial QMD startup, writer-lock, and per-agent cache-store work, but no maintainer-chosen policy for shared QMD cache layout, compaction, eviction, reduced-write mode, or measured multi-agent disk-write control.
