## Summary

Keep open. Current main still creates one QMD SQLite home per agent and does not expose or run cache compaction, WAL checkpointing, shared-cache layout, reduced-write mode, or QMD cache-size policy; the related merged mitigation only addressed embed stampedes, not the reported cache growth and disk-write amplification.

## What This Changes

The issue requests reducing QMD SQLite disk-write amplification for multi-agent deployments through shared caches, maintenance/checkpointing, reduced-write mode, or size/eviction limits.

## Best Possible Solution

Define an owner-reviewed QMD cache maintenance and layout policy, then implement the narrowest measured fix with docs and multi-agent disk-I/O proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms the per-agent QMD SQLite path and missing OpenClaw maintenance controls, but the reported multi-agent disk-write volume needs a real or Crabbox measurement to reproduce fully.

## Solution Assessment

Unclear. Shared cache, QMD cleanup/vacuum scheduling, WAL tuning, reduced-write mode, and eviction controls are plausible, but the best fix should be chosen by the Memory/QMD owners after measuring write volume and cache growth.
