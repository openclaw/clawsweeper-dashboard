## Summary

Keep open. Current main and the latest release still use per-agent QMD homes with per-agent `index.sqlite` files, and OpenClaw does not expose or run QMD shared-cache, WAL checkpoint, vacuum, reduced-write, or cache-size maintenance for this reported multi-agent disk-I/O problem.

## What This Changes

The issue asks OpenClaw to reduce extreme QMD SQLite cache disk writes in large multi-agent deployments, likely through shared cache storage, maintenance, reduced-write modes, or cache limits.

## Best Possible Solution

Define an owner-reviewed QMD cache storage and maintenance policy, then implement the smallest measured change with schema/docs alignment and before/after multi-agent disk-I/O proof.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms the per-agent QMD SQLite topology and missing OpenClaw maintenance controls, but the reported macOS multi-agent write volume still needs measured reproduction.

## Solution Assessment

Unclear. Shared cache, scheduled QMD cleanup, WAL/checkpoint tuning, reduced-write mode, and eviction caps are plausible, but the best path depends on QMD ownership boundaries, compatibility, and before/after I/O measurements.
