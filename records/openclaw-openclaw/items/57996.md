## Summary

Keep this issue open. Current main still gives each agent its own QMD XDG cache and `index.sqlite`, and the recent lazy-startup/QMD contention mitigations do not add shared-cache support, WAL/checkpoint/vacuum controls, read-only/reduced-write mode, or cache-size/LRU bounds for multi-agent deployments.

## What This Changes

The issue requests multi-agent QMD cache changes to reduce per-agent SQLite WAL write amplification, add maintenance controls, and bound QMD cache growth.

## Best Possible Solution

Treat this as a memory-core/QMD backend design issue: choose either a shared agent-namespaced QMD home/index or bounded maintenance and cache-size controls around isolated per-agent QMD homes, then document the chosen contract and cover it with multi-agent QMD lifecycle tests.

## Reproduction Assessment

No, not for the full production failure with high confidence. Source and tests prove the structural precondition on current main, but the reported macOS write rate and forced logout need a live multi-agent QMD workload to reproduce quantitatively.

## Solution Assessment

Unclear; current main's lazy-startup and serialization changes are useful mitigations but not the complete solution to the reported disk-I/O problem. The best fix needs maintainer design on shared cache versus per-agent maintenance/limits, likely with upstream QMD contract review.
