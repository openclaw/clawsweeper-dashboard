## Summary

Keep open. Current main still scopes QMD's XDG cache under each agent and runs QMD update/embed subprocesses without a shared-cache topology or QMD-specific checkpoint/vacuum/cache-size contract, so the reported multi-agent write-amplification problem remains unresolved.

## What This Changes

Reduce extreme disk I/O from per-agent QMD SQLite caches in multi-agent deployments by sharing or maintaining/bounding QMD storage.

## Best Possible Solution

Keep this as a memory-core/QMD backend design issue and choose either a supported shared agent-namespaced QMD home/index or bounded per-agent QMD maintenance and cache-size controls, with docs and multi-agent disk-I/O validation for the selected contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the exact 34GB macOS write volume, but source inspection does reproduce the per-agent QMD index path and upstream QMD WAL use that make the report plausible.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the suggested mitigations are plausible, but the narrowest maintainable fix depends on whether OpenClaw should change QMD storage topology, call QMD cleanup hooks, add config controls, or require upstream QMD support first.
