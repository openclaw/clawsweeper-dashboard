## Summary

Keep open. Current main still gives each QMD-enabled agent its own QMD cache/index path and lacks shared-cache, QMD WAL/checkpoint/vacuum, reduced-write, or cache-size controls; the newer QMD mitigations reduce startup/embed contention and model duplication but do not resolve the reported per-agent SQLite write amplification.

## What This Changes

The issue requests reducing QMD per-agent SQLite cache write amplification for multi-agent deployments through shared cache topology, WAL/checkpoint maintenance, reduced-write mode, or cache limits.

## Best Possible Solution

Keep this as a memory-core/QMD backend design issue and choose either a supported shared agent-namespaced QMD home/index or bounded per-agent maintenance and cache-size controls, with docs and multi-agent validation for the selected contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No, not for the full reported macOS write volume: source inspection clearly reproduces the per-agent QMD index path, but no live 19-agent disk-I/O measurement was run.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the suggested mitigations are plausible, but the narrowest maintainable fix depends on whether OpenClaw can safely control QMD SQLite maintenance or needs upstream/shared-index support first.
