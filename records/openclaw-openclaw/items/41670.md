## Summary

Keep open. The compatibility problem is real, but this PR is stale and incomplete against current main: it adds `encoding_format: "float"` to the mirrored `src/memory-host-sdk` helper, while the active package-backed helper and OpenAI inline/batch embedding paths still omit the field. A global hardcode also conflicts with current memory-lancedb docs/code noting that some OpenAI-compatible providers reject `encoding_format`, so maintainer review should choose a configurable or provider-scoped policy, likely in relation to the still-open #43910.

## What This Changes

The PR adds `encoding_format: "float"` to the remote embedding request body in `src/memory-host-sdk/host/embeddings-remote-provider.ts`.

## Best Possible Solution

Adopt an owner-approved encoding-format policy in the active package-backed memory host SDK and OpenAI embedding adapter paths, with config or provider-scoped behavior plus focused tests and config/docs updates. If #43910 becomes the accepted implementation and merges, close this PR as superseded by that merged work.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main still constructs embedding request bodies without `encoding_format` in the active remote helper and OpenAI inline/batch paths, so a proxy requiring that field would still see the reported request shape.

## Solution Assessment

No. The PR is too narrow and too global: it changes a stale mirror path and hardcodes the field instead of updating the active helper/adapters with a provider-scoped or configurable policy.
