## Summary

Keep this PR open for maintainer judgment. The compatibility gap is source-reproducible, but the current patch edits a stale path and hardcodes an OpenAI-specific request field in a shared helper even though current docs and shipped LanceDB behavior say some compatible providers reject that field; #43910 is the related broader configurable approach.

## What This Changes

The PR adds `encoding_format: "float"` to the remote embedding request body in `src/memory-host-sdk/host/embeddings-remote-provider.ts`.

## Best Possible Solution

Implement encoding format as a provider-scoped or configurable option in the active memory-host SDK and OpenAI embedding paths, preserving omit-by-default behavior for providers that reject the field and covering direct, batch, docs, and changelog surfaces.

## Reproduction Assessment

Yes for the request shape: current main's active helper and OpenAI adapter still send embedding requests without `encoding_format`. I did not live-test the reporter's internal proxy, so the external service failure is source-reproducible rather than live-reproduced.

## Solution Assessment

No. Hardcoding `encoding_format: "float"` in a stale shared helper is not the narrowest maintainable fix; a provider-scoped or configurable policy is safer because current OpenAI-compatible providers disagree on whether this field is accepted.
