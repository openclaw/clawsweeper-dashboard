## Summary

Close as superseded: #43910 is the canonical open PR for configurable `encoding_format`, while this PR's hardcoded shared-helper change is stale against current main and conflicts with documented provider behavior.

## What This Changes

This PR adds `encoding_format: "float"` to the remote embedding request body.

## Best Possible Solution

Keep one provider-scoped implementation path: review #43910 or a similar configurable approach that preserves omit-by-default behavior for providers that reject this field.

## Reproduction Assessment

Yes for the request shape: source inspection shows current main sends embedding requests without `encoding_format`. The reporter's internal proxy failure was not live-tested, so the service-specific failure remains source-supported rather than live-reproduced.

## Solution Assessment

No. Hardcoding `encoding_format: "float"` in a stale shared helper is not the narrowest maintainable fix; a provider-scoped or configurable option is safer because current docs and code account for providers that reject the field.
