## Summary

Keep open. The reported request-shape problem still exists on current main, but this PR is not ready to merge because it edits the stale mirror path and hardcodes an OpenAI-style field in a shared helper used by multiple embedding providers. The related #43910 tracks a broader configurable direction, but it is still open and also needs current-layout rework before it can supersede this safely.

## What This Changes

The PR adds `encoding_format: "float"` to the remote embedding request body in `src/memory-host-sdk/host/embeddings-remote-provider.ts`.

## Best Possible Solution

Adopt a provider-scoped or configurable encoding-format policy in the active memory-host SDK and OpenAI embedding adapter paths, with focused tests, docs, and changelog coverage. Once a canonical implementation is selected and merged, close or supersede the narrower hardcoded PR.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main still sends embedding request bodies without `encoding_format` in the active package helper and OpenAI direct/batch paths, so a proxy requiring that field would still see the reported request shape.

## Solution Assessment

No. This PR is not the best fix because it changes a stale mirror path and applies an OpenAI-specific format globally instead of making the behavior provider-scoped or configurable in the active implementation.
