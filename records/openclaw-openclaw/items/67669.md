Rebase onto the current lazy model CLI, then either extend the existing `models status --probe` runtime-backed path or make `custom-test` reuse the same resolved model/auth/request transport semantics, with pure JSON stdout, nonzero failure exits, redacted output, focused tests, and redacted real terminal proof.

- **[P1] Rebase this command onto the current lazy model CLI:** `src/cli/models-cli.ts:305-312`
  - body: Current main moved model command helpers into `models-cli.runtime.ts`; `git merge-tree` reports a conflict in this file, and the PR registration is still written against the old eager-import shape. Rebase and wire `custom-test` through the current lazy loader so the command does not reintroduce stale model CLI behavior.
  - confidence: 0.9
- **[P1] Collect targets using each model's effective API:** `src/commands/models/custom-test.ts:123-124`
  - body: This filter only accepts provider-level `openai-completions`, `anthropic-messages`, and `azure-openai-responses`, but current custom config supports `openai-responses` and the runtime resolves `model.api ?? provider.api`. Responses-compatible custom providers, and mixed-api providers with per-model APIs, will be silently skipped or probed with the wrong protocol.
  - confidence: 0.94
- **[P1] Run probes through the runtime model path:** `src/commands/models/custom-test.ts:52-56`
  - body: Calling the onboarding probe helper directly ignores the resolved model request path that applies provider/model headers, authHeader, request overrides, proxy/TLS/private-network policy, and per-model runtime semantics. That can make this diagnostic report false failures or false passes for valid configured providers.
  - confidence: 0.88
- **[P1] Return failure status in JSON mode:** `src/commands/models/custom-test.ts:216-218`
  - body: The JSON branch writes results and returns before computing `failCount` or calling `runtime.exit(1)`, so `openclaw models custom-test --json` exits successfully even when one or more verifications failed. Automation should not have to parse the payload to recover the command's failure semantics.
  - confidence: 0.96
- **[P1] Keep JSON stdout parseable:** `src/commands/models/custom-test.ts:151-154`
  - body: The command always writes the human progress line before checking `opts.json`, so `--json` output starts with non-JSON text. Current docs for model commands reserve stdout for the JSON payload, and scripts piping to `jq` will fail on this output.
  - confidence: 0.93
- **[P2] Redact provider base URLs from JSON results:** `src/commands/models/custom-test.ts:174-177`
  - body: Each result includes the raw provider `baseUrl`, and those URLs are treated as sensitive config because they can contain environment-specific hosts or embedded credentials. JSON/log output should omit or redact this field before users put it in CI artifacts or support threads.
  - confidence: 0.9
- **[P2] Require integer concurrency values:** `src/commands/models/custom-test.ts:100-102`
  - body: `Number('0.5')` passes this validation, but `Array.from({ length: 0.5 })` creates no workers, so the command can skip every target and return an empty success result. Parse this like the existing model probe options and reject non-integer concurrency values.
  - confidence: 0.86
- **[P2] Fail when an explicit provider filter tests nothing:** `src/commands/models/custom-test.ts:142-148`
  - body: When `--provider` is misspelled or matches a provider with no collected targets, this branch reports no providers and exits successfully. For a verification command, an explicit provider filter that verifies zero models should be a nonzero result so CI cannot silently pass a typo.
  - confidence: 0.84
