## Summary

Keep open. Current main does not implement the new command, and the PR head still has blocking CLI semantics plus a concrete JSON data-exposure concern that should be fixed before merge.

## What This Changes

This PR adds an `openclaw models custom-test` subcommand that probes configured custom-provider models with provider filtering, concurrency, table output, and JSON output.

## Best Possible Solution

Keep the PR open and rework the feature around the maintained runtime probe path or a shared probe helper, with parseable JSON, correct failure exits, effective API resolution, URL redaction, docs, and focused command tests.

## Reproduction Assessment

Yes. The blocking PR issues are reproducible by static inspection of the PR head: JSON mode mixes human logs with JSON and exits success on failures, target collection skips supported API shapes, and raw base URLs are emitted in JSON.

## Solution Assessment

No. A custom-provider verification command is useful, but this implementation bypasses the runtime probe semantics that already handle config, auth, transport, failure classification, and redaction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Suppress human logs in JSON mode:** `src/commands/models/custom-test.ts:152-154`
  - body: `--json` still emits the progress line before the JSON payload, so callers piping stdout to `jq` or CI parsers receive mixed human text and JSON. Suppress human logs or route them away from stdout when JSON output is requested.
  - confidence: 0.96
- **[P2] Exit non-zero for JSON failures:** `src/commands/models/custom-test.ts:216-218`
  - body: The JSON branch writes results and returns before `failCount` is computed or `runtime.exit(1)` can run. Failed probes therefore exit successfully in JSON mode, making automation report a false pass.
  - confidence: 0.96
- **[P2] Build targets from effective model APIs:** `src/commands/models/custom-test.ts:123-138`
  - body: Target collection only checks `providerConfig.api` and omits supported Responses-family APIs, while model entries can override `api`. Custom providers using `openai-responses`, `openai-codex-responses`, or per-model overrides are skipped or probed with the wrong protocol.
  - confidence: 0.94
- **[P2] Probe through the runtime transport:** `src/commands/models/custom-test.ts:168-173`
  - body: The probe calls onboarding helpers with only `baseUrl`, `apiKey`, and `modelId`, bypassing configured headers, auth overrides, transport settings, local-service setup, and runtime network policy. That can report false health for providers that work only through the normal runtime path.
  - confidence: 0.9
- **[P2] Reject fractional concurrency values:** `src/commands/models/custom-test.ts:100-102`
  - body: Values such as `--concurrency 0.5` pass validation, then worker creation uses that fractional length and can start no workers. Require a positive integer so a typo cannot skip all probes and return success.
  - confidence: 0.89
- **[P2] Fail empty explicit provider filters:** `src/commands/models/custom-test.ts:142-148`
  - body: When `--provider` is misspelled or selects a provider with no collected targets, the command logs an empty result and exits success. An explicit filter that verifies nothing should fail so scripts do not get a false pass.
  - confidence: 0.91
- **[P1] Redact provider base URLs in JSON:** `src/commands/models/custom-test.ts:176`
  - body: Each result includes raw `baseUrl`, and JSON mode writes the raw results. Provider base URLs are treated as sensitive URL data in this repo because they can include embedded credentials, token query parameters, or private endpoint details.
  - confidence: 0.92
- **[P2] Normalize Azure Foundry URLs before raw probes:** `src/commands/models/custom-test.ts:168-172`
  - body: Azure Foundry providers can be persisted as `.../openai/v1`, but this path passes that stored base URL directly into the onboarding probe helper, which appends another `/openai/deployments/...` path and can falsely fail working Azure custom providers.
  - confidence: 0.84
