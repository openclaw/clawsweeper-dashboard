## Summary

Keep open. Current main still does not implement the requested command, and the PR is a useful feature contribution, but the latest head is not mergeable as-is because it bypasses runtime probe semantics, skips supported custom-provider API shapes, breaks JSON/exit behavior, and exposes raw provider URLs in JSON output.

## What This Changes

This PR adds an `openclaw models custom-test` CLI subcommand, exports onboarding verification helpers, and probes configured custom provider models with provider filtering, concurrency, and JSON output options.

## Best Possible Solution

Keep the PR open and rework the feature around the maintained runtime probe path or a shared runtime probe helper, then enumerate effective custom-provider models with `model.api ?? provider.api`, preserve script-safe JSON and failure exits, redact sensitive endpoint fields, and add focused command tests plus docs/changelog coverage.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: on the PR head, `--json` logs before writing JSON and returns before applying failure exits, while an `openai-responses` custom provider is skipped by the allowlist before any live network call is needed.

## Solution Assessment

No. A dedicated custom-provider verification command may be a reasonable product direction, but this implementation should not ship until it uses the same runtime semantics users actually run with and fixes the JSON, target enumeration, redaction, and validation gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep JSON output parseable:** `src/commands/models/custom-test.ts:152-154`
  - body: `--json` still emits the progress line before the JSON payload, so callers piping stdout to `jq` or CI parsers receive non-JSON text. Suppress human logs or route them away from stdout when JSON output is requested.
  - confidence: 0.96
- **[P2] Return non-zero from JSON failures:** `src/commands/models/custom-test.ts:216-218`
  - body: The JSON branch writes results and returns before `failCount` is computed or `runtime.exit(1)` runs. Failed probes therefore exit successfully in JSON mode, which makes automation report a false pass.
  - confidence: 0.96
- **[P2] Build targets from effective model APIs:** `src/commands/models/custom-test.ts:123-138`
  - body: Target collection only checks `providerConfig.api` and the allowlist omits supported Responses-family APIs. Custom providers using `openai-responses`, `openai-codex-responses`, or per-model `model.api` overrides are skipped or probed with the wrong protocol.
  - confidence: 0.93
- **[P2] Use runtime probe transport:** `src/commands/models/custom-test.ts:168-173`
  - body: The probe calls onboarding helpers with only `baseUrl`, `apiKey`, and `modelId`, bypassing configured headers, auth overrides, request transport settings, and runtime network policy. That can report false health for providers that work through the normal runtime path.
  - confidence: 0.88
- **[P2] Fail empty explicit provider filters:** `src/commands/models/custom-test.ts:142-148`
  - body: When `--provider` is misspelled or selects a provider with no collected targets, the command logs an empty result and exits success. An explicit filter that verifies nothing should fail so scripts do not get a false pass.
  - confidence: 0.9
- **[P2] Reject fractional concurrency values:** `src/commands/models/custom-test.ts:100-103`
  - body: Values such as `--concurrency 0.5` pass validation, then `Array.from({ length: 0.5 })` starts no workers and the command can return success without probing any target. Require a positive integer.
  - confidence: 0.87
- **[P1] Redact provider URLs from JSON results:** `src/commands/models/custom-test.ts:174-177`
  - body: Each result includes raw `baseUrl`, and `--json` writes those results directly. Provider URLs are treated as sensitive in this repo because they can contain credentials or private endpoint details, so JSON output should omit or redact them.
  - confidence: 0.91
