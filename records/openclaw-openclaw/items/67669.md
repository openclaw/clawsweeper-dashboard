## Summary

Keep open. This is a useful CLI feature request, but the current branch still has merge-blocking correctness issues, a concrete URL exposure issue, and no external real-behavior proof for the changed command behavior.

## What This Changes

The PR adds an `openclaw models custom-test` command that probes configured custom providers, with provider filtering, concurrency, table/JSON output, and exported onboarding probe helpers.

## Best Possible Solution

Rework the feature around the maintained runtime probe path or a shared probe helper, preserving runtime config/auth/transport semantics while adding parseable JSON, correct failure exits, URL redaction, API-resolution parity, docs, and focused command tests.

## Reproduction Assessment

Yes. Static inspection of the current PR head gives a high-confidence reproduction path for the blockers: run the command with `--json`, failed probes, fractional concurrency, missing provider filters, Responses-backed or per-model API configs, or sensitive `baseUrl` values.

## Solution Assessment

No. A custom-provider verification command is useful, but this implementation bypasses existing runtime model transport and config semantics that `models status --probe` already exercises.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Redact provider URLs before JSON output:** `src/commands/models/custom-test.ts:176`
  - body: Each result includes raw `baseUrl`, and `--json` writes the raw results. Provider URLs can contain embedded credentials, token query parameters, or private endpoints, so emit a redacted display URL or omit this field.
  - confidence: 0.92
- **[P2] Keep JSON output parseable:** `src/commands/models/custom-test.ts:152-154`
  - body: `--json` still logs the progress line before writing the JSON payload, so callers piping stdout to `jq` or CI parsers receive mixed human text and JSON. Suppress human logs or route them away from stdout in JSON mode.
  - confidence: 0.96
- **[P2] Exit non-zero for failed JSON probes:** `src/commands/models/custom-test.ts:216-218`
  - body: The JSON branch returns before `failCount` is computed and before `runtime.exit(1)` can run. Failed probes therefore exit successfully in JSON mode, which makes automation report a false pass.
  - confidence: 0.96
- **[P2] Use effective model APIs when collecting targets:** `src/commands/models/custom-test.ts:123-138`
  - body: Target collection only checks provider-level `api`, but model entries can override `api` and current config supports Responses-family APIs. Mixed or Responses-backed custom providers are skipped or probed with the wrong protocol.
  - confidence: 0.94
- **[P2] Probe through the runtime transport:** `src/commands/models/custom-test.ts:168-173`
  - body: The command calls onboarding helpers with only `baseUrl`, `apiKey`, and `modelId`, bypassing configured headers, auth overrides, local service setup, request transport, and runtime network policy. That can produce false health results for providers that work only through normal runtime resolution.
  - confidence: 0.9
- **[P2] Reject fractional concurrency values:** `src/commands/models/custom-test.ts:100-102`
  - body: Values such as `--concurrency 0.5` pass validation, then worker creation uses a fractional length and can start no workers. Require a positive integer so a typo cannot skip all probes and return success.
  - confidence: 0.89
- **[P2] Fail empty explicit provider filters:** `src/commands/models/custom-test.ts:142-148`
  - body: When `--provider` is misspelled or selects a provider with no collected targets, the command reports no providers and exits success. An explicit filter that verifies nothing should fail so scripts do not get a false pass.
  - confidence: 0.91
- **[P2] Normalize Azure Foundry URLs before raw probes:** `src/commands/models/custom-test.ts:168-172`
  - body: Azure Foundry providers can be stored as `.../openai/v1`, but this path passes that URL directly into the onboarding helper that appends `/openai/deployments/...`. Working Foundry providers can therefore be reported as broken.
  - confidence: 0.84
