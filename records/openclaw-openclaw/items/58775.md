## Summary

Keep open. Current main still has only partial Google Vertex support: explicit `api: "google-vertex"` models can use the Vertex REST URL and bearer auth path, but ordinary forward-compatible `google-vertex/*` resolution is still covered by a test that resolves to `api: "google-gemini-cli"`, which misses the native Vertex stream gate. The issue also has open PR #60860 using closing syntax, so the issue should stay open until that PR or a replacement lands.

## What This Changes

The issue asks to keep `google-vertex` on native Vertex AI transport and ADC auth while retaining Google model ID normalization.

## Best Possible Solution

Finish native Google Vertex support in the bundled Google plugin so `google-vertex/*` model refs resolve to runtime models with `api: "google-vertex"`, build Vertex project/location endpoints, and use ADC/Bearer auth separately from AI Studio and Gemini CLI paths. PR #60860 or a maintainer replacement should also document and test the accepted ADC credential types.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the current forward-compatible model test resolves `google-vertex/gemini-3.1-pro-preview` to `api: "google-gemini-cli"`, while provider registration only selects the Vertex stream for `api: "google-vertex"`.

## Solution Assessment

No. Current main is partial; the narrow maintainable solution is to connect implicit/configured `google-vertex` model resolution to the native Vertex transport and review the ADC credential behavior in PR #60860 or an equivalent replacement.
