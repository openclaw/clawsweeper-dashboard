## Summary

Keep open. Current main still has the reported gap: isolated cron announce delivery sends with the cron-run session context and queues only main-session awareness, while later `sessions_history` reads the destination channel session transcript. The issue is also paired with the open closing PR https://github.com/openclaw/openclaw/pull/80786, so it should stay open until that PR lands or is replaced/closed.

## What This Changes

The issue requests that cron announce deliveries into a channel be recorded in the destination channel session transcript so later channel turns and `sessions_history` can see them.

## Best Possible Solution

Land one reviewed implementation that mirrors only successfully delivered cron announce output into the resolved destination channel session transcript with idempotency, while keeping the isolated cron worker transcript separate.

## Reproduction Assessment

Yes. The live repro comment shows a real delivered WhatsApp cron token missing from the destination session, and current source inspection shows direct cron delivery still writes awareness only to main/global while `sessions_history` reads the channel session transcript.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse outbound destination-session resolution and append the successfully delivered assistant payload into that transcript with an idempotency key; the linked PR follows that direction.
