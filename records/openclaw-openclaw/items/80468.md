## Summary

Keep open. Current main still sends isolated cron direct deliveries with the cron run session and only queues main-session awareness, so a later Discord channel session read through `sessions_history` will not see the delivered cron report; the adjacent open PR only covers stored custom-session mirroring.

## What This Changes

The issue asks for cron announce deliveries to be written or queryable in the target channel session context so later Discord channel turns can see the cron report that was posted.

## Best Possible Solution

Mirror only the successfully delivered final payload into the resolved destination channel session transcript with an idempotency key, while keeping the isolated worker transcript separate and coordinating with the open custom-session mirror PR.

## Reproduction Assessment

Yes, by source inspection rather than a live Discord run. The direct cron path uses the cron session context and omits destination-session transcript mirroring, while `sessions_history` reads `chat.history` for the requested channel session.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the delivered-payload mirror/session append path after successful delivery and resolve the same durable channel session that normal outbound sends use; expanding the broader cron routing model is not required.
