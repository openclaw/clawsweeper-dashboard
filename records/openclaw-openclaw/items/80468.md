## Summary

Keep open. Current main still sends isolated cron direct announce delivery with the cron run session context and only queues main-session awareness, so the destination channel session transcript can still miss the delivered report; https://github.com/openclaw/openclaw/pull/80786 explicitly targets this issue and remains open.

## What This Changes

Cron announce deliveries should be mirrored into the target channel session transcript so later `sessions_history` calls can see the delivered report.

## Best Possible Solution

Land one focused fix that resolves the same outbound destination session normal sends use, ensures the session entry exists, and mirrors only successfully delivered cron output idempotently into that transcript.

## Reproduction Assessment

Yes. Current main is source-reproducible because direct cron delivery uses the cron session key and only queues main-session awareness, and the issue comments add live before-fix proof where a delivered WhatsApp cron token had zero hits in the destination session transcript.

## Solution Assessment

Yes. The narrow maintainable path is the one in the open fixing PR: resolve the outbound destination session, send with that context, and append the delivered assistant payload after successful delivery using an idempotency key.
