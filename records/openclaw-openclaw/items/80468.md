## Summary

Keep open. The bug is still present on current main by source inspection, and an open fixing PR already references this issue with closing syntax, so the issue should remain open until that PR is reviewed, landed, or explicitly closed.

## What This Changes

Cron announce delivery should make successfully delivered channel messages visible in the destination channel session transcript used by later `sessions_history` calls.

## Best Possible Solution

Review and land one focused fix, likely the linked open PR or a rebased equivalent, that resolves the outbound destination session and idempotently mirrors only successfully delivered cron output into that transcript.

## Reproduction Assessment

Yes, by source inspection on current main plus the live before-fix comment: isolated cron delivery uses the cron session and only queues main-session awareness, while `sessions_history` reads the destination session transcript. I did not run a new live Discord reproduction in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse outbound destination-session resolution and the existing transcript mirror append path after a successful delivery, while keeping the isolated worker transcript separate and avoiding best-effort partial-send mirrors.
