## Summary

Keep open. Current main still delivers isolated cron announce output to the external channel without mirroring it into the target channel session transcript that `sessions_history` reads; the related merged fix only queues ephemeral main-session awareness and explicitly avoided transcript or routed-session changes.

## What This Changes

Cron announce deliveries should be visible or queryable in the target channel session history so later channel turns can reference prior cron reports.

## Best Possible Solution

Reuse the existing outbound-session route and delivery-mirror transcript path after successful cron announce delivery, while preserving isolated cron run semantics and the current main-session awareness event.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: run an isolated `agentTurn` cron with `delivery.mode="announce"` to a Discord channel, then inspect the derived channel session through `sessions_history`; current code sends externally without writing a mirror to that session.

## Solution Assessment

Yes, for the queryable-history part: the narrow maintainable fix is to align cron announce delivery with Gateway send mirroring rather than inventing a separate delivery log. If maintainers want cron reports included automatically in provider replay, that should be a separate product decision.
