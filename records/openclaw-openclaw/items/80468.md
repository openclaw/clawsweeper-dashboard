## Summary

Keep open. Current `main` still has a source-proven bug: isolated cron direct delivery sends to the chat surface using the cron run session and only queues main-session awareness, while `sessions_history` reads the destination channel session transcript. The open fix PR https://github.com/openclaw/openclaw/pull/80786 closes this issue and has supplied real behavior proof, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests that cron `announce` deliveries be recorded in the destination channel session context so later `sessions_history` reads can see what was delivered.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/80786 with a narrow fix that resolves the outbound destination session, ensures its session entry exists, and mirrors only successfully delivered cron assistant output with an idempotency key.

## Reproduction Assessment

Yes by source inspection, with recent live release evidence in the issue comments. I did not run a live current-main Discord/WhatsApp job in this read-only review, so the structured status is source-reproducible rather than directly reproduced on current `main`.

## Solution Assessment

Yes, the best direction is the narrow destination-session mirror already represented by the open closing PR. It should preserve the isolated worker transcript boundary and avoid mirroring partial best-effort failures or unrelated worker history.
