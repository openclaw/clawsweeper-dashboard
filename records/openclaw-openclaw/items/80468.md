## Summary

Keep open. Current main still has a source-proven gap: isolated cron direct delivery sends with the cron session context and only queues main-session awareness, while `sessions_history` reads the requested session transcript via `chat.history`, so a later Discord channel session will not see the delivered cron report. The related open PR https://github.com/openclaw/openclaw/pull/74913 is useful adjacent work, but its current diff covers `session:<id>` custom-session mirroring rather than deriving and mirroring into the resolved channel session for isolated announce delivery.

## What This Changes

The issue asks for cron-delivered Discord channel messages to be recorded or queryable in the target channel session context.

## Best Possible Solution

Mirror only the successfully delivered cron final payload into the resolved destination channel session transcript, using the session append API with an idempotency key and keeping the isolated worker transcript separate.

## Reproduction Assessment

Yes by source inspection, not by a live Discord run. The direct cron path uses the cron session context and only queues main-session awareness, while `sessions_history` reads `chat.history` from the requested durable channel session.

## Solution Assessment

Yes, the narrow maintainable direction is to reuse the delivered-payload mirror approach and resolve the same destination channel session that normal outbound sends use. The broader cron output-routing feature request is not required for this bug.
