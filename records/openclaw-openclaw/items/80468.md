## Summary

Keep open. Current main still sends isolated cron announce delivery from the cron worker session and only queues main-session awareness, while the destination channel session read by `sessions_history` is never mirrored; the later live WhatsApp repro strengthens the source-backed report. The adjacent open PR is useful for `session:<id>` custom-session mirroring, but its current diff still uses the stored cron session key and does not resolve the channel destination session for isolated announce delivery.

## What This Changes

Cron announce deliveries should be mirrored into the destination channel session so later `sessions_history` calls can find what the agent posted.

## Best Possible Solution

Extend direct cron announce delivery to resolve the same destination channel session route used by normal outbound sends, then append the successfully delivered payload projection to that session with idempotency while keeping the isolated worker transcript separate.

## Reproduction Assessment

Yes. Current-main source inspection shows the cron delivery path never mirrors into the destination channel session, and the issue discussion includes a live OpenClaw 2026.5.7 cross-surface repro with a unique token absent from the destination transcript.

## Solution Assessment

Yes. Reusing outbound session route resolution plus the existing idempotent transcript append behavior is the narrow maintainable direction; the adjacent custom-session PR is not sufficient unless expanded to derive channel-session destinations for isolated announce delivery.
