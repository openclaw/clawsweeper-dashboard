## Summary

Keep open. Current main still has the cron announce split-brain path, and https://github.com/openclaw/openclaw/pull/80786 is the open implementation candidate linked with closing syntax, so this issue should close only after that or an equivalent fix lands.

## What This Changes

Cron announce deliveries should be visible in the target channel session transcript so later `sessions_history` calls can see what the job posted.

## Best Possible Solution

Review and land one focused fix, likely https://github.com/openclaw/openclaw/pull/80786 coordinated with https://github.com/openclaw/openclaw/pull/74913, that mirrors only successfully delivered cron output into the resolved destination session with idempotency while preserving the isolated worker transcript boundary.

## Reproduction Assessment

Yes by source inspection, with live supporting evidence from the issue comments. Current main sends direct cron delivery with the cron session context, queues only main-session awareness, and `sessions_history` reads the destination session transcript that never receives the delivered cron output.

## Solution Assessment

Yes. The narrow maintainable solution is to resolve the same outbound destination session normal sends use, append the successfully delivered assistant payload there after delivery success, and avoid copying the whole isolated worker transcript.
