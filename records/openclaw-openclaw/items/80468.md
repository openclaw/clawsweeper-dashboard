## Summary

Keep open. Current main still sends isolated cron direct delivery using the cron run session and only queues main-session awareness, while `sessions_history` reads the requested channel session transcript; the open fixing PR https://github.com/openclaw/openclaw/pull/80786 uses closing syntax and should be reviewed or closed before this issue is resolved.

## What This Changes

Cron direct announce deliveries should be recorded in the target channel session transcript so later `sessions_history` calls can see what was posted.

## Best Possible Solution

Land one focused fix that resolves the normal outbound destination session, ensures its session entry when needed, and idempotently mirrors only successfully delivered cron output into that transcript.

## Reproduction Assessment

Yes. The source path on current main still uses the cron session for direct delivery and only queues main awareness, and the issue discussion includes a recent live per-channel-peer repro showing zero hits in the destination session transcript.

## Solution Assessment

Yes. Resolving the outbound destination session and appending the successful delivered assistant payload there is the narrow maintainable fix; the current action should be review and merge sequencing for the open fixing PR, not closing this issue early.
