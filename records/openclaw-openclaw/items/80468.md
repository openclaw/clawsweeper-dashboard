## Summary

Keep open. Current main still sends isolated cron announce output with the cron run session as delivery context and only queues main-session awareness; it does not append the delivered payload into the resolved Discord channel session that `sessions_history` later reads. The active related PR https://github.com/openclaw/openclaw/pull/74913 is useful adjacent work, but its current diff mirrors `session:<id>` custom-session delivery rather than deriving the destination channel session for isolated announce delivery.

## What This Changes

The issue asks for cron announce messages delivered to a Discord channel to be visible in the target channel session transcript for later agent context.

## Best Possible Solution

Mirror only the successfully delivered cron final payload into the resolved destination channel session transcript with an idempotency key, while keeping the isolated worker transcript separate and coordinating with the existing cron mirroring PR.

## Reproduction Assessment

Yes, by source inspection. The direct cron path sends with the cron session key and no mirror, while `sessions_history` reads the requested channel session through `chat.history`.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the delivered-payload mirror approach but resolve the same destination channel session that users later query, rather than broadening cron context sharing.
