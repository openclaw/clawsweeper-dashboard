## Summary

Keep open. The PR targets a real npm audit regression, but the proposed local Vertex client introduces a blocking header-merging regression and the PR is still draft/protected.

## What This Changes

The PR removes `@anthropic-ai/vertex-sdk`, adds `google-auth-library@10.6.2`, and replaces the dependency-backed Anthropic Vertex client with a local `BaseAnthropic` subclass.

## Best Possible Solution

Keep the dependency cleanup, but preserve Anthropic SDK header-container semantics in the local client and add focused regression coverage before merge.

## Reproduction Assessment

Yes. The review finding is reproducible by passing an Anthropic SDK `buildHeaders(...)` result through the PR's `prepareOptions`; `mergeHeaders` treats the branded container as a plain object and drops or garbles the contained headers.

## Solution Assessment

No. Removing the vulnerable dependency chain is the right direction, but the safest implementation should reuse or faithfully copy the SDK header merge semantics instead of hand-rolling a lossy record merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve SDK header containers when adding auth headers:** `extensions/anthropic-vertex/stream-runtime.ts:140`
  - body: Anthropic resources pass `options.headers` as the SDK's branded nullable-header container from `buildHeaders(...)`, not always as a plain object or `Headers`. This merge treats that container as a record, producing `values`/`nulls` string headers and dropping headers such as `anthropic-beta` and helper headers before `BaseAnthropic.buildHeaders` runs. Please preserve the SDK container semantics when prepending Google auth headers.
  - confidence: 0.93
