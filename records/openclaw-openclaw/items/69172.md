## Summary

Keep open. The requested Google/Gemini proxy override behavior is still not implemented on current main, but this PR is not merge-ready because it mixes the Google fix with unrelated runtime/script changes and introduces concrete functional and security regressions.

## What This Changes

This PR attempts to restore Gemini custom endpoint and OpenAI-compatible routing across web search, embeddings, image generation, and media understanding, while also touching scripts, memory SDK runtime exports, plugin SDK loading, and other provider files.

## Best Possible Solution

Keep the work focused inside the Google plugin: centralize endpoint and apiType resolution, preserve current SSRF/private-network defaults, support the intended proxy formats per capability, and add targeted regression tests without changing unrelated scripts or SDK/runtime surfaces.

## Reproduction Assessment

Yes. Current-main source inspection is enough to reproduce the missing behavior: web search uses the default Gemini URL, embeddings ignore the endpoint env aliases, and the current API test explicitly rejects non-Google Gemini base URLs.

## Solution Assessment

No. The requested direction is valid, but this PR is not the best implementation because it introduces unrelated changes and weakens network/auth behavior; the safer path is a cleaned Google-only patch with explicit policy handling and capability-specific payloads.

Full review comments:

- **[P1] Launch pnpm JS entrypoints through Node:** `scripts/pnpm-runner.mjs:55-56`
  - body: This path classifies JS/CJS/MJS pnpm entrypoints as Node-runnable, but then tries to spawn the script path directly. Corepack setups commonly expose pnpm as a .cjs file, so this can fail with EACCES/ENOEXEC and also drops nodeArgs/nodeExecPath.
  - confidence: 0.94
- **[P1] Keep private-network access behind explicit opt-in:** `extensions/google/image-generation-provider.ts:160`
  - body: The image path now passes allowPrivateNetwork: true for every resolved Google base URL. A custom GEMINI_BASE_URL can therefore target private/internal hosts without the existing explicit dangerous opt-in, which is a security regression from current main's guarded Google helper.
  - confidence: 0.93
- **[P1] Keep media private-network access behind explicit opt-in:** `extensions/google/media-understanding-provider.ts:70`
  - body: The shared media-understanding path also forces allowPrivateNetwork: true for custom Google endpoints. Audio/video requests should preserve the existing request-policy opt-in instead of granting private-network routing whenever a base URL is configured.
  - confidence: 0.92
- **[P1] Restore runtime-core host helpers instead of stubs:** `packages/memory-host-sdk/src/runtime-core.ts:28-42`
  - body: This package entrypoint used to expose real host config/state helpers, but the PR replaces loadConfig and state-directory helpers with empty values. Node consumers of @openclaw/memory-host-sdk/runtime-core would silently lose persisted config and filesystem paths.
  - confidence: 0.9
- **[P1] Strip compat suffixes before native Gemini requests:** `extensions/google/env-utils.ts:47-48`
  - body: resolveGoogleApiType classifies every googleapis.com URL as native Gemini, while resolveGoogleBaseUrl preserves paths such as /v1beta/openai. Native callers will then build /v1beta/openai/models/...:generateContent, regressing the compat-suffix normalization that current main carries.
  - confidence: 0.88

  - body: The OpenAI-compatible image branch sends only model, prompt, n, and size. Requests with inputImages are silently downgraded from edits to plain generation, so proxy users get outputs that ignore their reference images.
  - confidence: 0.91
- **[P2] Send media-specific payloads for audio and video:** `extensions/google/media-understanding-provider.ts:93-109`
  - body: generateGeminiInlineDataText is shared by audio and video, but the OpenAI-compatible branch always sends the binary as image_url to /chat/completions. Standards-compatible APIs expect audio/video-specific fields or endpoints, so these media requests will fail or be misinterpreted.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR introduces concrete network-policy regressions by enabling private-network routing for custom Google endpoints without the existing explicit opt-in.

Concerns:

- **[high] Unconditional private-network opt-in for image requests:** `extensions/google/image-generation-provider.ts:160`
  - body: The image provider passes allowPrivateNetwork: true for all resolved Google base URLs, so a configured or env-provided custom endpoint can route to private/internal hosts without the existing dangerous opt-in.
  - confidence: 0.93

  - body: The media-understanding provider makes the same unconditional private-network allowance for audio/video requests, widening the SSRF surface for custom endpoint configuration.
  - confidence: 0.92
