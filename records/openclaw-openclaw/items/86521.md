## Summary

Keep open: current main still appears to drop `reasoning_content` for the reported OpenCode Zen DeepSeek aliases because `opencode-native` is not detected as DeepSeek-style compat and the model-id replay allowlist does not include `deepseek-v4-flash-free` or `big-pickle`. Prior DeepSeek proxy fixes cover nearby cases, but not this exact OpenCode model surface.

## What This Changes

The issue requests preserving DeepSeek-style `reasoning_content` replay for DeepSeek V4 models routed through `opencode-native` proxy providers.

## Best Possible Solution

Add a narrow, tested OpenCode-hosted DeepSeek V4 replay policy for confirmed DeepSeek aliases while keeping non-DeepSeek OpenCode refs on the minimal OpenAI-compatible path.

## Reproduction Assessment

No live repro was run, but the source path is clear: `opencode-native` does not get DeepSeek replay compat and the reported model ids are not in the replay allowlist, so the sanitizer will strip assistant `reasoning_content`. A live OpenCode Zen credentialed test would upgrade this to a full reproduction.

## Solution Assessment

Yes, the narrow fix is to extend provider/model replay eligibility for confirmed DeepSeek V4 aliases, not to mark every `opencode-native` model as DeepSeek-style because OpenCode also serves non-DeepSeek models.
