## Label Justifications

- `P2`: The bug can mislead agents with incomplete command results, but it is limited to async exec approval completion delivery.
- `impact:message-loss`: The current follow-up formatter silently drops the beginning of command output before the agent receives it.
- `impact:session-state`: Incomplete `System:` follow-ups can make the conversation state diverge from the actual command result and drive wrong later decisions.

## Summary

Keep open: current main still builds async gateway and node exec approval follow-ups from a whitespace-normalized 400-character tail, so the agent can still receive incomplete single-line `System:` completion details. The direct fix PR is closed unmerged, and the adjacent open command-output-event PR does not replace this follow-up path.

## What This Changes

The issue asks OpenClaw to preserve multiline async exec approval output in the agent follow-up instead of silently sending only a flattened tail.

## Best Possible Solution

Add a dedicated async exec approval completion formatter for gateway and node results that preserves captured multiline output up to the existing exec capture cap, marks real capture-cap truncation, and leaves generic background notifications compact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: async gateway or node exec approval completions with multiline output over 400 characters are reduced to `DEFAULT_NOTIFY_TAIL_CHARS` and whitespace-normalized before follow-up injection.

## Solution Assessment

Yes. The maintainable fix is to split approval-completion result formatting from the compact background notification formatter and cover both gateway and node approval paths with focused regressions.
