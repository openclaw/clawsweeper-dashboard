## Label Justifications

- `P2`: The bug can mislead agents with incomplete exec results, but the affected path is limited to async approval completion delivery.
- `impact:message-loss`: The reported path silently drops the beginning of command output before the agent receives the follow-up message.
- `impact:session-state`: Incomplete System follow-ups can cause the conversation state and later agent decisions to diverge from the actual command result.

## Summary

Keep open: current main still applies the compact notify formatter to async exec approval completions, so multiline output over 400 characters can be tailed and whitespace-normalized before the follow-up reaches the agent.

## What This Changes

The issue reports that async exec approval completion messages delivered back to the agent keep only the tail of command output and collapse multiline text.

## Best Possible Solution

Add a dedicated async exec approval result formatter for gateway and node completions that preserves captured multiline output up to the existing exec capture cap, marks real cap truncation explicitly, and leaves generic background notifications compact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: async gateway or node exec approval completions with multiline output over 400 characters are tailed to DEFAULT_NOTIFY_TAIL_CHARS and whitespace-normalized before follow-up injection.

## Solution Assessment

Yes. The narrow maintainable fix is to split approval-completion result formatting from the compact notifyOnExit summary formatter and cover both gateway and node approval paths with regressions.
