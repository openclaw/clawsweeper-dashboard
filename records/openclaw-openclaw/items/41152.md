## Summary

Keep open. Current main and the latest release still route async exec approval completion text through the compact notification formatter, so multiline output over 400 characters is tailed and whitespace-collapsed before the follow-up agent turn; the linked fix PR #41170 is closed and unmerged.

## What This Changes

The issue reports that async exec approval completions deliver only the final ~400 characters of command output and collapse newlines in the injected follow-up System message.

## Best Possible Solution

Split approval-completion formatting from compact background notification formatting, preserve captured newlines up to the existing exec capture cap, and add regression coverage for gateway and node async approval completions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main reproduction path: any approved gateway or node async exec completion with multiline output over 400 characters reaches tail plus normalizeNotifyOutput before the follow-up agent turn.

## Solution Assessment

Yes. The narrow maintainable repair is to reimplement the useful direction from closed PR #41170 on current main, keeping compact notifyOnExit behavior unchanged and adding focused regression tests.

Current main still sends gateway and node async approval completions through the compact notify formatter, which tails to 400 characters and collapses newlines before the follow-up agent turn. Implement a narrow replacement for the useful direction from closed PR #41170: preserve captured multiline stdout/stderr/error in approval-completion follow-ups up to the existing exec capture cap, add an explicit note when that capture cap was hit, and do not change ordinary background notifyOnExit compact summaries.
