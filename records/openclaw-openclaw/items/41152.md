## Label Justifications

- `P2`: The bug can mislead agents with incomplete exec results, but the affected path is limited to async approval completion delivery.
- `impact:message-loss`: The current formatter silently drops the beginning of command output before the agent receives the follow-up message.
- `impact:session-state`: Incomplete System follow-ups can cause the conversation state and later agent decisions to diverge from the actual command result.

## Summary

Keep open: current main still compacts async exec approval completion output to a whitespace-normalized 400-character tail before the agent follow-up is built. The focused fix PR is closed unmerged, and the adjacent command-output event PR does not replace this System follow-up path.

## What This Changes

The issue requests preserving full multiline async exec approval output in the agent System follow-up, with explicit marking only when the real exec capture cap is hit.

## Best Possible Solution

Add a dedicated async exec approval completion formatter for gateway and node results that preserves captured multiline output up to the existing exec capture cap, explicitly marks capture-cap truncation, and leaves generic background notifications compact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: async gateway or node exec approval completions with multiline output over 400 characters are tailed to DEFAULT_NOTIFY_TAIL_CHARS and whitespace-normalized before follow-up injection.

## Solution Assessment

Yes. The maintainable fix is to split approval-completion result formatting from the compact notifyOnExit summary formatter and cover both gateway and node approval paths with regression tests.
