## Label Justifications

- `P2`: The bug can mislead agents with incomplete exec results, but the affected path is limited to async approval completion delivery.
- `impact:message-loss`: The reported path silently drops the beginning of command output before the agent receives the follow-up message.
- `impact:session-state`: Incomplete System follow-ups can cause the conversation state and later agent decisions to diverge from the actual command result.

## Summary

Keep open: current main and the latest release still compact async exec approval completion output before the follow-up is injected, so the reported 400-character tail and newline-loss behavior remains source-reproducible. The earlier focused fix at https://github.com/openclaw/openclaw/pull/41170 closed unmerged, and the newer related event-stream PR does not replace the agent follow-up result text path.

## What This Changes

Async exec approval completion should deliver full captured multiline command output to the agent instead of a whitespace-normalized tail.

## Best Possible Solution

Add a dedicated async exec approval completion formatter for gateway and node results that preserves captured multiline output up to the existing exec capture cap, marks real capture-cap truncation explicitly, and leaves generic background notifications compact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: async gateway or node exec approval completions with multiline output over 400 characters are tailed to DEFAULT_NOTIFY_TAIL_CHARS and whitespace-normalized before follow-up injection.

## Solution Assessment

Yes. The narrow maintainable fix is to split approval-completion result formatting from the compact notifyOnExit summary formatter and cover both gateway and node approval paths with regressions.
