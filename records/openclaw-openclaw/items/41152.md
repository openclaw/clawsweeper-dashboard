## Label Justifications

- `P2`: The bug can mislead agents with incomplete exec results, but the affected path is limited to async approval completion delivery.
- `impact:message-loss`: The current formatter silently drops the beginning of command output before the agent receives the follow-up message.
- `impact:session-state`: Incomplete System follow-ups can cause the conversation state and later agent decisions to diverge from the actual command result.

## Summary

Keep open: current main still formats async exec approval completion text as a whitespace-normalized 400-character tail before the System follow-up is built, and the direct fix attempt at https://github.com/openclaw/openclaw/pull/41170 is closed unmerged. An adjacent open PR at https://github.com/openclaw/openclaw/pull/85522 adds command-output events, but it does not replace the truncated agent follow-up path reported here.

## What This Changes

The issue reports that async exec approval follow-up messages deliver only a flattened 400-character tail of command output instead of preserving full multiline output.

## Best Possible Solution

Add a dedicated async exec approval completion formatter for gateway and node results that preserves captured multiline output up to the existing exec capture cap, explicitly marks real capture-cap truncation, and leaves generic background notifications compact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: async gateway or node exec approval completions with multiline output over 400 characters are tailed to DEFAULT_NOTIFY_TAIL_CHARS and whitespace-normalized before follow-up injection.

## Solution Assessment

Yes. The maintainable fix is to split approval-completion result formatting from the compact notifyOnExit summary formatter and cover both gateway and node approval paths with regression tests.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/41152. Current main still formats async gateway and node exec approval follow-up text with DEFAULT_NOTIFY_TAIL_CHARS plus normalizeNotifyOutput, which drops the front of long output and strips newlines before buildExecApprovalFollowupPrompt sees it. Add a dedicated approval-completion result formatter that preserves captured multiline output up to the existing exec capture cap, includes an explicit truncation marker only when the capture cap was actually hit, and keeps generic background notifyOnExit summaries compact. Cover both gateway and node async approval follow-up paths with focused regressions, including multiline output over 400 chars, no-output completion behavior, and a capture-cap truncation note. Coordinate with https://github.com/openclaw/openclaw/pull/85522 if it lands, but do not add a new command_output event as the fix for this issue unless maintainers explicitly choose that product direction. Add a user-facing CHANGELOG.md fix entry during landing because this changes agent-visible exec output behavior.
