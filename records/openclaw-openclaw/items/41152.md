## Label Justifications

- `P2`: The bug can mislead agents with incomplete exec results, but the affected path is limited to async approval completion delivery.
- `impact:message-loss`: The current formatter silently drops the beginning of command output before the agent receives the follow-up message.
- `impact:session-state`: Incomplete `System:` follow-ups can cause the conversation transcript and later agent decisions to diverge from the actual command result.

## Summary

Keep open: current `main` and the latest release still build async exec approval follow-ups from a whitespace-normalized 400-character tail before injecting the result into the follow-up prompt. The focused fix at https://github.com/openclaw/openclaw/pull/41170 was closed unmerged, and the nearby open event PR at https://github.com/openclaw/openclaw/pull/85522 does not replace the `System:` follow-up result text path.

## What This Changes

The issue requests preserving full multiline async exec approval output in the agent-delivered `System:` follow-up instead of sending a newline-stripped 400-character tail.

## Best Possible Solution

Land a narrow exec approval completion formatter that preserves captured multiline gateway and node output up to the existing exec capture cap, marks capture-cap truncation explicitly, and leaves background `notifyOnExit` summaries compact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: async gateway or node exec approval completions with multiline output over 400 characters are tailed to `DEFAULT_NOTIFY_TAIL_CHARS` and whitespace-normalized before follow-up injection.

## Solution Assessment

Yes. The maintainable fix is to split approval-completion result formatting from the compact `notifyOnExit` summary formatter and cover both gateway and node approval paths with regression tests.
