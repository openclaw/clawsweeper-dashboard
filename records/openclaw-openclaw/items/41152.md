## Label Justifications

- `P2`: The bug can mislead agents with incomplete exec results, but the affected path is limited to async approval completion delivery.
- `impact:message-loss`: The reported path silently drops the beginning of command output before the agent receives the follow-up message.
- `impact:session-state`: The incomplete System follow-up can cause the agent's conversation state and subsequent decisions to diverge from the actual command result.

## Summary

Current main still matches the reported failure mode: async exec approval completion output is compacted through the 400-character notify tail formatter and whitespace normalization before the follow-up is injected, so this remains a valid source-reproducible bug rather than cleanup to close.

## What This Changes

The issue requests preserving full multiline async exec approval output in the agent follow-up message instead of silently tailing it to about 400 characters and flattening newlines.

## Best Possible Solution

Add a dedicated async exec approval result formatter for gateway and node completions that preserves captured multiline output up to the existing exec capture cap, marks real cap truncation explicitly, and keeps generic background notifications compact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: async gateway or node exec approval completions with multiline output over 400 characters are tailed to DEFAULT_NOTIFY_TAIL_CHARS and whitespace-normalized before follow-up injection.

## Solution Assessment

Yes. The narrow maintainable fix is to split approval-completion result formatting from the compact notifyOnExit summary formatter and cover both gateway and node approval paths with regressions.
