## Label Justifications

- `P1`: The PR targets a real session-state regression where cross-channel direct DMs can repeatedly lose CLI context for users relying on `dmScope: "main"` continuity.
- `merge-risk: 🚨 compatibility`: The new static hash shape changes persisted direct-chat CLI binding identity and can reset existing sessions once after upgrade.
- `merge-risk: 🚨 session-state`: Both the bug and the proposed fix affect whether a direct-message CLI session is reused or reset across turns and channels.

## Summary

Keep open: the patch targets the right root cause, but it still needs real gateway plus CLI reuse proof and a maintainer decision on the one-time persisted-hash reset before merge.

## What This Changes

This PR factors direct-chat context rendering, adds a channel-neutral direct-chat context for CLI reuse hashing, wires it into `extraSystemPromptStaticParts`, and updates focused tests/mocks.

## Best Possible Solution

Keep the provider label in the live prompt and use a channel-neutral direct-chat hash input, but merge only after real dual-channel CLI proof and an explicit migration or acceptance decision for the one-time stored-hash reset.

## Reproduction Assessment

Yes, source-reproducible: current main feeds a provider-labeled direct-chat context into the static CLI reuse hash while `dmScope: "main"` can route multiple channels to one session key, and the reuse helper invalidates on hash mismatch. I did not run the full gateway plus `claude-cli` Telegram/WebChat flow in this read-only review.

## Solution Assessment

Mostly yes: separating the live provider-labeled prompt from a channel-neutral hash input is the narrow fix for the root cause. The merge path is incomplete until maintainers either accept or mitigate the one-time persisted-hash invalidation and real runtime proof is supplied.
