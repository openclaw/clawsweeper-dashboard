## Best Possible Solution

Keep the live provider label in the actual prompt, make only the direct-chat CLI reuse identity channel-neutral, handle or explicitly accept the one-time persisted-hash upgrade effect, and prove the real dual-channel CLI continuity path before merge.

## Reproduction Assessment

Yes, source-reproducible: current main hashes a provider-labeled direct-chat context while `session.dmScope: "main"` is documented to share direct-message sessions, and the CLI reuse code invalidates on any stored/current hash mismatch. I did not run the full gateway plus claude-cli Telegram/WebChat reproduction in this read-only review.

## Solution Assessment
