## Summary

Keep open: this maintainer-labeled draft is a focused provider/plugin hardening PR, and current main still calls bundled provider-policy hooks directly. The patch looks mechanically sound, but it changes auth/config provider-policy failures from hard errors to logged fail-open behavior, so it needs maintainer/provider-owner confirmation plus the missing build and changed-gate proof before merge.

## What This Changes

This PR wraps bundled provider-policy public-artifact hooks with logged fail-open handling and updates provider-runtime tests so runtime hooks can run after a current bundled-hook failure.

## Best Possible Solution

Land the centralized wrapper only after provider owners confirm fail-open semantics for all four hooks and build/changed-gate validation is complete.

## Reproduction Assessment

Yes, source-reproducible: current main returns raw bundled provider-policy surfaces and provider-runtime invokes those hooks directly, so a throwing bundled hook propagates before runtime fallback. I did not run tests because this review is read-only.

## Solution Assessment

Yes, conditionally: central wrapping at the public-artifact seam is the narrowest maintainable repair if fail-open is the intended policy. If API-key or config-default hooks must remain fail-closed, the PR should split those hooks before merge.
