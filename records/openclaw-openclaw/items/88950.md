## Summary

Keep open: current main and the latest release still use hard-failing bundled provider-policy hooks, while this maintainer-labeled draft changes provider/auth fallback semantics and needs owner confirmation plus broader validation before merge.

## What This Changes

The PR wraps bundled provider-policy public-artifact hooks so throws are logged and ignored, then lets provider-runtime fall through to runtime hooks after a bundled-hook failure.

## Best Possible Solution

Land the centralized wrapper only after provider owners accept fail-open semantics for all four hooks and the build/changed-check validation gap is closed.

## Reproduction Assessment

Yes, source-reproducible: current main directly calls bundled provider-policy hooks, so a throwing hook propagates through provider/config resolution. I did not run tests because this review is read-only.

## Solution Assessment

Yes, conditionally: central wrapping at the public-artifact seam is the narrowest maintainable repair if fail-open is the intended policy. If API-key or config-default hooks must remain fail-closed, the PR should split those hooks before merge.
