- `merge-risk: 🚨 auth-provider`: The changed hooks participate in provider config, API-key, defaults, and thinking-profile resolution, which can affect auth/provider behavior when a hook fails.

## Summary

Keep open for maintainer review: the patch appears to fix a source-reproducible provider-policy crash path, but it intentionally changes bundled provider-policy hook failures from hard errors to logged fail-open behavior across provider/config/auth-related hooks.

## What This Changes

The PR wraps bundled provider-policy public-artifact hooks with logged fail-open handling and lets provider-runtime fall back to runtime hooks only after a current bundled-hook failure.

## Best Possible Solution

Land the centralized wrapper after provider owners confirm fail-open semantics for all four hooks and build/changed-gate validation is complete; split stricter auth/default hooks if those must remain fail-closed.

## Reproduction Assessment

Yes. Current main returns raw bundled provider-policy surfaces and provider-runtime invokes those hooks directly, so a throwing hook propagates before runtime fallback.

## Solution Assessment

Yes, conditionally. Central wrapping at the public-artifact seam is the narrowest maintainable repair if fail-open is the intended policy; if API-key or config-default hooks must remain fail-closed, split those hooks before merge.
