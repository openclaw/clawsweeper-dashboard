## Summary

Keep this PR open: the patch is a focused provider-policy hardening change and I did not find a concrete correctness defect, but it is maintainer-labeled/draft and changes bundled provider-policy hook errors from hard failure to logged fail-open behavior across provider/config and auth-adjacent paths.

## What This Changes

The PR wraps bundled provider-policy public-artifact hooks, logs ignored failures, and lets provider-runtime fall back to runtime hooks only after a bundled hook failure.

## Best Possible Solution

Land the centralized wrapper after provider owners confirm fail-open semantics for all four hooks and build/changed-gate validation is complete.

## Reproduction Assessment

Yes, source-reproducible: current main returns raw bundled provider-policy surfaces and provider-runtime invokes those hooks directly, so a throwing hook propagates before runtime fallback. I did not run tests because this review is read-only.

## Solution Assessment

Yes, conditionally: central wrapping at the public-artifact seam is the narrowest maintainable repair if fail-open is the intended policy. If API-key or config-default hooks must remain fail-closed, the PR should split those hooks before merge.
