## Label Justifications

- `P2`: This is a normal-priority provider/plugin hardening PR with limited blast radius but real provider config and auth-resolution impact.
- `merge-risk: 🚨 compatibility`: The patch changes the runtime failure mode for bundled provider-policy hook errors from throwing to logged fail-open behavior.
- `merge-risk: 🚨 auth-provider`: The changed hook set includes provider config, API-key, defaults, and thinking-profile resolution, which can affect auth/provider behavior when a hook fails.

## Summary

Keep open for maintainer review: current main still has the throwing-hook path, and the PR has no concrete line-level defect from this pass, but it intentionally changes provider-policy errors from hard failures to logged fail-open behavior across compatibility- and auth-sensitive hooks.

## What This Changes

This PR wraps bundled provider-policy public-artifact hooks to log and ignore throws, tracks per-hook failures, falls through to runtime provider hooks after those failures, and adds focused regressions.

## Best Possible Solution

Keep the centralized wrapper and failure-marker shape only if provider owners accept fail-open semantics for all four hooks, then land after the scoped build and changed-check validation pass.

## Reproduction Assessment

Yes. Current main returns raw bundled provider-policy surfaces and provider-runtime invokes those hooks directly, so a throwing hook propagates through provider/config resolution; this review did not run tests because the checkout is read-only.

## Solution Assessment

Yes, conditionally. Central wrapping at the public-artifact seam is the narrowest maintainable repair if fail-open is the intended policy; if API-key or config-default hooks must remain fail-closed, those hooks should be split out before merge.
