## Summary

Keep open: this is a maintainer-labeled MEMBER draft with a focused provider-policy hardening patch, but the fail-open behavior changes provider/config/auth fallback semantics and still needs maintainer/provider-owner confirmation plus build and changed-gate proof before merge.

## What This Changes

The PR wraps bundled provider-policy public-artifact hooks with logged fail-open handling and lets provider-runtime fall through to runtime provider hooks after current bundled-hook failures.

## Best Possible Solution

Land the centralized wrapper only after provider/plugin owners explicitly accept fail-open semantics for all four hook classes and build/changed-gate validation is complete; split auth/default handling if those should remain fail-closed.

## Reproduction Assessment

Yes, from source inspection. Current main returns raw bundled provider-policy surfaces and provider-runtime invokes those hooks directly before runtime fallback, so a throwing hook propagates.

## Solution Assessment

Yes, conditionally. Central wrapping at the public-artifact seam is the narrowest maintainable repair if fail-open is the intended policy; if auth/default hooks must remain fail-closed, the PR should split those before merge.
