## Label Justifications

- `P2`: This is a normal-priority provider/plugin hardening fix with limited blast radius but real config and provider-resolution impact.
- `merge-risk: 🚨 compatibility`: The patch changes bundled provider-policy hook errors from hard failures into logged fail-open behavior that existing setups may observe during upgrades.
- `merge-risk: 🚨 auth-provider`: The changed hooks participate in provider config, API-key, defaults, and thinking-profile resolution, which can affect auth/provider behavior when a hook fails.

## Summary

Keep open: current main still has the direct crashable bundled-hook path, and the PR is a focused hardening fix, but it intentionally changes provider-policy failures from fail-closed to logged fail-open behavior across config and auth-provider surfaces, so it needs maintainer/provider-owner confirmation and the remaining build/changed-gate validation before merge.

## What This Changes

The PR wraps bundled provider-policy public-artifact hooks so thrown hooks are logged and ignored, while provider-runtime can fall through to runtime provider hooks after a bundled hook failure.

## Best Possible Solution

Land the centralized wrapper only after provider owners accept fail-open semantics for all four hooks and the build/changed-check validation gap is closed.

## Reproduction Assessment

Yes, source-reproducible: current main returns raw bundled provider-policy hooks and provider-runtime invokes them directly, so a throwing hook propagates through provider/config resolution. I did not run tests because this review is read-only.

## Solution Assessment

Yes, conditionally: central wrapping at the public-artifact seam is the narrowest maintainable repair if fail-open is the intended policy. If API-key or config-default hooks must remain fail-closed, the PR should split those hooks before merge.
