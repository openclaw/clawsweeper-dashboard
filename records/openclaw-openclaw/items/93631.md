## Summary

Keep open. The PR targets a real web UI clipboard bug, but the patch currently fails TypeScript because the fallback path can return a boolean before `.then()` is called, and the supplied proof is a described procedure plus diff rather than observed browser output.

## What This Changes

The PR adds an `execCommand` clipboard fallback to web UI code-block copy and copy-as-markdown buttons when `navigator.clipboard.writeText` fails.

## Best Possible Solution

Land one narrow Control UI clipboard fix that uses a consistently async fallback path for both affected buttons, passes type checks, and includes redacted HTTP browser proof.

## Reproduction Assessment

Yes, source-level: current `main` directly uses the async Clipboard API for code-block copy, and the linked issue gives a concrete HTTP LAN reproduction path. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

No. The fallback direction is plausible, but this implementation is not the best fix until the helper consistently returns `Promise<boolean>`, type checks pass, and real HTTP browser proof is supplied.
