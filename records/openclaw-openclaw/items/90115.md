## Summary

Keep this PR open with changes requested: the fallback is directionally right for managed inbound PDFs, but it is reached only after the sandbox-size rejection, so default PDFs between the 5 MB staging cap and the 20 MB chat attachment cap still fail before the new pass-through path. The supplied proof also checks the trust predicate rather than the patched `chat.send` behavior in a real gateway run.

## What This Changes

The PR adds a `chat.send` fallback that returns absolute managed inbound PDF paths when sandbox staging throws or is incomplete, plus regression tests for fallback and mixed-batch rejection.

## Best Possible Solution

Move the trusted managed-inbound PDF pass-through ahead of the sandbox staging cap, or explicitly exempt those refs from that cap, while keeping non-PDF and untrusted paths on the existing fail-closed path and adding a 5-20 MB managed-PDF regression test.

## Reproduction Assessment

Yes. Source inspection shows current main saves non-image PDF offloads, then `prestageMediaPathOffloads` can fail before dispatch when sandbox staging is incomplete or size-gated; I did not run a live gateway repro in this read-only review.

## Solution Assessment

No. The owner boundary and managed-inbound predicate are good, but the current PR is not the best fix because the fallback is placed after the sandbox-size rejection and does not cover the large managed-PDF path from the linked issue.
