## Summary

Keep open. Current main still lacks outbound iMessage rich style spans, text-size attributes, underline/strikethrough, and the requested Shake/Nod/Explode/Ripple/Bloom/Jitter effects; existing maintainer-side comments keeping this open for product/API follow-up still match the source.

## What This Changes

The issue requests rich iMessage formatting and animated text/effect support for outbound iMessage replies.

## Best Possible Solution

Define a typed outbound iMessage capability model, implement only upstream-backed formatting/effect subsets per transport, preserve deterministic plain-text fallbacks, and document legacy imsg limitations beside BlueBubbles capabilities.

## Reproduction Assessment

Yes for the observable capability gap from source inspection: legacy imsg sends plain JSON-RPC text/media/reply params, while BlueBubbles strips Markdown and only carries an optional classic effectId. Live transport proof is still needed before selecting the exact rich-attribute serialization.

## Solution Assessment

Unclear as an implementation question because there is no patch. A one-off imsg flag layer would be brittle; the maintainable direction is a capability-gated transport model with provider-specific fallbacks and docs.
