## Summary

Keep open. Current main still lacks rich iMessage style spans, text-size attributes, underline/strikethrough, and the requested Shake/Nod/Explode/Ripple/Bloom/Jitter effects; the legacy imsg path is plain-text RPC delivery, while BlueBubbles only has a narrower private-API `effectId` action and strips Markdown on normal sends. The existing maintainer-side discussion asking for product/API follow-up still matches the current code.

## What This Changes

The issue requests rich formatting and iMessage effects support for outbound iMessage messages.

## Best Possible Solution

Define a typed outbound iMessage capability model, implement only the upstream-backed rich formatting/effect subset per transport, preserve plain-text fallback where unsupported, and document legacy imsg limitations.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an enhancement request, not broken documented behavior. Source inspection does give high confidence that current main lacks the requested rich formatting/effect support.

## Solution Assessment

Unclear for the proposed options as written: adding imsg flags alone would not cover BlueBubbles, Markdown stripping, or shared outbound capability semantics. A maintainer-defined capability contract is the narrowest maintainable direction before code work.
