## Summary

Keep open. Current main still lacks outbound rich iMessage style spans, text-size attributes, underline/strikethrough, and the requested Shake/Nod/Explode/Ripple/Bloom/Jitter effects; BlueBubbles only exposes a narrower effectId action and strips Markdown on normal sends, so the prior maintainer-side keep-open comments remain accurate.

## What This Changes

The issue requests rich outbound iMessage formatting and newer iMessage text effects for the imsg and/or BlueBubbles channel paths.

## Best Possible Solution

Define a typed outbound iMessage capability model, implement only the upstream-backed formatting/effect subset per transport, keep deterministic plain-text fallbacks, and document legacy imsg limitations next to BlueBubbles capabilities.

## Reproduction Assessment

Yes for the observable current gap from source inspection: legacy imsg sends plain JSON-RPC text/media/reply params, while BlueBubbles strips Markdown and only carries an optional effectId string. Live transport proof is still needed before choosing exact rich-attribute serialization.

## Solution Assessment

Unclear as an implementation question because there is no proposed patch. A one-off imsg flag layer would be brittle; the maintainable direction is a capability-gated transport model with provider-specific fallbacks and docs.
