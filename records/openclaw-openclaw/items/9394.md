## Summary

Keep open. Current main still lacks outbound rich iMessage style spans, text-size attributes, underline/strikethrough, and the requested newer text effects; legacy imsg remains plain-text oriented while BlueBubbles only exposes a narrower effectId action and strips Markdown on normal sends.

## What This Changes

The issue requests rich text formatting and iMessage text effects for outbound iMessage messages.

## Best Possible Solution

Define a typed outbound iMessage capability model, implement only the upstream-backed formatting/effect subset per transport, keep deterministic plain-text fallback where unsupported, and document legacy imsg limitations next to BlueBubbles capabilities.

## Reproduction Assessment

Yes for the observable current gap: source and focused tests show legacy imsg sends plain text/media/reply params, while BlueBubbles strips Markdown and only carries an optional effectId string. Live transport proof is still needed before choosing exact rich-attribute serialization.

## Solution Assessment

Unclear as an implementation question because there is no proposed patch. A one-off imsg flag layer would be brittle; the maintainable path is a capability-gated transport model with provider-specific fallbacks and docs.
