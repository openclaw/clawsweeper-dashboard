## Summary

Keep open. Current main still lacks outbound rich iMessage style spans, text-size attributes, underline/strikethrough, and the requested Shake/Nod/Explode/Ripple/Bloom/Jitter text animations; the April maintainer-side keep-open comments remain consistent with the current tree.

## What This Changes

Request support for iMessage outbound rich text formatting and newer message/text effects in the legacy imsg and/or BlueBubbles iMessage paths.

## Best Possible Solution

Define a typed outbound iMessage capability model, implement only the upstream-backed formatting/effect subset per transport, keep deterministic plain-text fallback where unsupported, and document legacy imsg limitations alongside BlueBubbles capabilities.

## Reproduction Assessment

Yes for the observable current gap: source and focused tests show legacy imsg sends plain text/media/reply params, while BlueBubbles strips Markdown and only carries an optional effectId string. A live transport smoke is still needed before choosing exact rich-attribute serialization.

## Solution Assessment

Unclear as an implementation question because there is no proposed patch. A one-off imsg flag layer would be brittle; the maintainable direction is a capability-gated transport model with provider-specific fallbacks and docs.
