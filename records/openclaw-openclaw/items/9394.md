## Summary

Keep open. Current main still lacks outbound rich iMessage style spans, text-size attributes, underline/strikethrough, and the requested Shake/Nod/Explode/Ripple/Bloom/Jitter effects; BlueBubbles only has a narrower classic `effectId` action and normal sends strip Markdown.

## What This Changes

Request rich iMessage/BlueBubbles outbound formatting, text-size styling, and Shake/Nod/Explode/Ripple/Bloom/Jitter effects.

## Best Possible Solution

Define a typed outbound iMessage capability model, implement only upstream-backed formatting/effect subsets per transport, preserve deterministic plain-text fallbacks, and document legacy imsg limitations beside BlueBubbles capabilities.

## Reproduction Assessment

Yes for the observable capability gap from source inspection: legacy imsg sends plain JSON-RPC text/media/reply params, while BlueBubbles strips Markdown and only carries an optional classic `effectId`. Live transport proof is still needed before choosing exact rich-attribute serialization.

## Solution Assessment

Unclear as an implementation question because there is no patch. A one-off imsg flag layer would be brittle; the maintainable direction is a capability-gated transport model with provider-specific fallbacks and docs.
