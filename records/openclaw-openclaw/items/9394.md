## Summary

Keep open. Current main still lacks outbound rich iMessage style spans, text-size attributes, underline/strikethrough, and the requested Shake/Nod/Explode/Ripple/Bloom/Jitter effects; the prior maintainer-side reviews keeping this open for product/API direction remain accurate.

## What This Changes

Request support for rich iMessage formatting, text-size variations, and newer iMessage text effects in the iMessage/BlueBubbles outbound paths.

## Best Possible Solution

Define a typed outbound iMessage capability model, implement only the upstream-backed formatting/effect subset per transport, preserve deterministic plain-text fallback where unsupported, and document legacy imsg limitations beside BlueBubbles capabilities.

## Reproduction Assessment

Yes for the observable current gap: source and focused tests show legacy imsg sends plain text/media/reply params, while BlueBubbles strips Markdown and only carries an optional effectId string. Live transport proof is still needed before choosing exact rich-attribute serialization.

## Solution Assessment

Unclear as an implementation question because there is no proposed patch. A one-off imsg flag layer would be brittle; the maintainable path is capability-gated transport behavior with provider-specific fallbacks and docs.
