## Summary

Keep open: the fix is close and proof is adequate, but the branch still lets Gateway reset-command parsing run before direct delivery is selected, so literal `/new` or `/reset` notifications can be rejected or mutate the session instead of being sent.

## What This Changes

The PR adds an explicit `directDelivery` Gateway/agent path so CLI `--deliver` sends the original message payload through outbound delivery, with docs, changelog, protocol, Swift model, and regression-test updates.

## Best Possible Solution

Land a trusted direct-notification path once eligible direct sends bypass reset-command parsing, while keeping internal runtime-event delivery on the composed agent path and preserving protocol/docs/client alignment.

## Reproduction Assessment

Yes from source inspection: current main sends CLI `--deliver` through the Gateway agent-turn path and returns before outbound delivery when the agent/model yields no payloads. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No, not quite. The explicit `directDelivery` signal is the right shape, but the Gateway handler must skip `/new` and `/reset` command parsing for eligible direct sends before the patch is correct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Skip reset parsing for direct sends:** `src/gateway/server-methods/agent.ts:1263-1269`
  - body: Because `directDeliveryText` is computed only after the `/new`/`/reset` block, a direct notification like `--message "/reset deploy done"` still enters reset handling before the direct send is selected. CLI `agent` RPCs use the `agent` method's least-privilege write scope, so this returns `missing scope: operator.admin`; an admin caller would reset the session instead of sending the literal text. Gate reset-command parsing off for eligible direct-delivery requests.
