## Summary

Keep open: the PR addresses a real current-main delivery bug, but the latest head is dirty against base and the new direct-delivery path still lets Gateway slash-command reset parsing run before the direct payload is selected.

## What This Changes

The PR adds a direct-delivery path for `openclaw agent --deliver`, adds a `directDelivery` Gateway agent parameter, preserves silent-token text in outbound planning, and updates docs, changelog, tests, and Swift protocol models.

## Best Possible Solution

Land an explicit direct-notification path for trusted CLI/direct RPC sends after it bypasses agent command parsing and keeps internal-event delivery on the composed agent path.

## Reproduction Assessment

Yes from source inspection: current main sends CLI `--deliver` through the Gateway agent turn path and returns before outbound delivery when the model yields no payloads. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No, not quite. The explicit `directDelivery` signal is the right shape, but the Gateway handler still parses `/new` and `/reset` before selecting the direct payload, so literal direct notifications can be rejected or mutate session state instead of being sent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Skip reset parsing for direct sends:** `src/gateway/server-methods/agent.ts:1263-1269`
  - body: Because the direct-delivery text is only selected after the `/new`/`/reset` handler above, a direct notification such as `openclaw agent --deliver --message "/reset deploy done"` still hits reset handling before this branch. CLI agent RPCs use the `agent` method's write scope, so that path returns `missing scope: operator.admin` instead of sending the literal message; with admin it would reset the session. Gate reset command parsing off when this is an eligible direct-delivery request.
