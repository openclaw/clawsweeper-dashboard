## Summary

Keep open. The PR targets a valid direct-delivery bug, but the patch currently infers direct notification mode from `deliver` itself, which breaks existing agent-composed delivery callers, and the external real behavior proof is only tests/typechecks.

## What This Changes

The PR changes `openclaw agent --deliver` to bypass agent execution for direct notifications, adds `directDeliveryText` plumbing and silent-token preservation, and updates tests plus CLI docs.

## Best Possible Solution

Use an explicit direct-notification signal for user-authored CLI/Gateway sends, preserve agent-composed delivery for runtime follow-ups, cover both paths with regression tests, and add redacted real delivery proof before merge.

## Reproduction Assessment

Yes, from source inspection. Current main routes `openclaw agent --deliver` through an agent turn and skips outbound delivery when the model returns no payloads; the PR regression is also source-reproducible through `sendExecApprovalFollowup`.

## Solution Assessment

No. Direct delivery is the right direction for user-authored notifications, but this implementation uses `deliver` as the whole discriminator and must instead distinguish direct notifications from existing agent-composed delivery workflows.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Gate direct delivery behind an explicit signal:** `src/agents/agent-command.ts:512`
  - body: This bypass fires for any `deliver: true` run with no `internalEvents` or images. Existing `sendExecApprovalFollowup` calls the Gateway agent with `deliver: true` and an instruction prompt from `buildExecApprovalFollowupPrompt`; after this change that prompt is sent verbatim instead of an agent-composed follow-up. Please require an explicit direct-notification signal from the CLI/direct RPC path and leave composed follow-ups on the normal agent path.
