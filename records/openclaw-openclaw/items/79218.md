## Summary

Keep open: current main still exhibits the reported `openclaw agent --deliver` failure mode and the PR moves in the right direction, but the patch is not mergeable because the direct-delivery bypass is inferred too broadly and can expose existing agent follow-up prompts; it also needs the required changelog entry.

## What This Changes

The branch adds direct-delivery plumbing for `openclaw agent --deliver`, preserves silent-token text through outbound payload planning, updates focused tests, and revises CLI docs/help.

## Best Possible Solution

Use an explicit direct-notification signal for user-authored CLI/Gateway sends, preserve agent-composed delivery for exec/subagent/runtime followups, and land the final behavior with regression tests plus a changelog entry.

## Reproduction Assessment

Yes, from source inspection. Current main routes `openclaw agent --deliver` through `runAgentAttempt` and returns before outbound delivery when the model produces no payloads; the PR regression is also source-reproducible through exec approval followup delivery.

## Solution Assessment

No. Direct delivery is the right fix direction for user-authored notifications, but inferring it from `deliver` plus absence of internal events/images is not the narrowest maintainable discriminator.
