## Label Justifications

- `P1`: The issue reports a source-confirmed session-memory poisoning loop that can progressively degrade active agents into malformed output or silence.
- `impact:session-state`: The affected data is persisted session memory that is reused across `/new` and `/reset`, corrupting later context.
- `impact:security`: Raw model and transcript artifacts are persisted and reintroduced as future prompt context, crossing a trust boundary.
- `impact:message-loss`: The reported end state includes agents emitting role tokens or NO_REPLY instead of useful replies across later sessions.

## Summary

Keep open: current main and v2026.6.8 still extract and persist session-memory transcript text without artifact sanitization. Current startup-context framing mitigates raw instruction replay, but it does not prevent poisoned memory artifacts from being written or reused by other memory paths.

## What This Changes

Fix the bundled session-memory hook so raw chat-template tokens, leaked tool-call XML, orphaned role markers, and NO_REPLY-only turns are sanitized or skipped before memory persistence and later context reuse.

## Best Possible Solution

Add a maintainer-approved persistence-boundary sanitizer or validator that reuses the shared assistant-visible sanitizer where appropriate, elides malformed-only turns, keeps untrusted startup framing, and adds regression tests plus redacted real hook proof.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 still extract raw user/assistant text and write it under `## Conversation Summary`, so a poisoned transcript containing `<tool_call>...</tool_call><|im_end|>` follows the reported persistence path.

## Solution Assessment

Unclear without maintainer/security direction. Sanitizing before persistence is the right boundary, but the best implementation should align with the shared sanitizer and decide how to quarantine or elide malformed memory safely.
