## Label Justifications

- `P1`: The report describes a source-reproducible session-memory poisoning loop that can degrade real agents into malformed output or NO_REPLY-only behavior.
- `impact:session-state`: The affected data is persisted session memory that can be reintroduced into later session context.

## Summary

Keep open: current main and v2026.6.8 still persist raw session-memory transcript text before any artifact sanitizer or validation gate, while the newer startup-context wrapper only mitigates one re-injection path.

## What This Changes

The issue requests that the bundled session-memory hook stop saving raw chat-template tokens, orphan role markers, raw tool-call XML, and NO_REPLY-only artifacts into reusable memory files.

## Best Possible Solution

Keep this issue open until the session-memory persistence boundary uses an owner-approved shared sanitizer/validator for model scaffolding, raw tool-call XML, orphan role markers, and silent-token artifacts, with tests and an explicit recovery story for existing poisoned memory files.

## Reproduction Assessment

Yes, at source level. A JSONL message containing raw `<tool_call>`, chat-template tokens, role markers, or `NO_REPLY` is extracted as text and written into `## Conversation Summary` without sanitization on current main.

## Solution Assessment

Unclear as an implementation choice. Sanitizing or validating before persistence is the right boundary, but the exact shared sanitizer contract, elision/quarantine behavior, and existing-file recovery path need maintainer/security direction.
