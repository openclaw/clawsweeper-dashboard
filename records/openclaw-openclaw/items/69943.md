## Label Justifications

- `P1`: The issue describes a progressive poisoned-memory loop that can break active agent workflows after `/new` or `/reset`.
- `impact:session-state`: The affected data is cross-session memory that can carry corrupted context into later sessions.
- `impact:security`: Untrusted model/tool-call scaffolding can be stored as durable memory and later returned to prompt context.
- `impact:message-loss`: The reported failure mode includes agents emitting malformed role-token output or `NO_REPLY` instead of useful replies.

## Summary

Keep open: current main and the latest release still persist raw session transcript text into session-memory Markdown files without stripping chat-template tokens, role markers, raw tool-call XML, or `NO_REPLY`; newer startup-context wrapping reduces one injection path but does not fix the durable poisoned-memory write path.

## What This Changes

The issue asks OpenClaw to stop the bundled session-memory hook from saving malformed model output and re-injecting it as future session context.

## Best Possible Solution

Add a bounded validator/sanitizer at the bundled session-memory transcript/write boundary, reuse existing model-token and assistant-visible text helpers where possible, and keep startup-context untrusted wrapping as defense in depth rather than the only guard.

## Reproduction Assessment

Yes, from source inspection: a transcript containing `<|im_end|>`, raw `<tool_call>`, role markers, or `NO_REPLY` in message text would be returned unchanged by `getRecentSessionContent` and written unchanged by the handler. I did not live-reproduce the model self-poisoning loop.

## Solution Assessment

No, current main is not the best solved state: startup-context untrusted wrappers help the first-turn injection path, but they do not prevent raw poisoned artifacts from being persisted or indexed. The best fix is at the session-memory store boundary before writing.
