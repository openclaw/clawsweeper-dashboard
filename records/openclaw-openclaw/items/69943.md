## Label Justifications

- `P1`: The issue describes a high-confidence session-memory poisoning loop that can degrade real agents into malformed or NO_REPLY-only behavior.
- `impact:session-state`: The affected surface is persisted session memory that is later reintroduced into new session context.
- `impact:security`: Saved memory becomes future prompt context, so model scaffolding and role-marker artifacts can cross a persistent prompt-injection boundary.
- `impact:message-loss`: The observed failure mode includes agents becoming non-responsive or emitting only NO_REPLY across subsequent sessions.

## Summary

Keep open: https://github.com/openclaw/openclaw/pull/95791 partially mitigated current main, but the current sanitizer still does not remove the plain orphan role-marker lines reported in this issue. The latest release also still has the original raw persistence path, so this remains the canonical issue for the remaining persistence-boundary and security decision.

## What This Changes

The issue asks OpenClaw to stop session-memory from persisting and re-injecting raw model artifacts such as chat-template tokens, tool-call XML, role markers, and NO_REPLY completions.

## Best Possible Solution

Keep this canonical issue open until session-memory has a maintainer-approved persistence-boundary validator that also removes or fences orphan role-marker lines and defines recovery for already-poisoned memory files.

## Reproduction Assessment

Yes, at source level. An assistant text containing newline-delimited user/assistant role-marker lines survives the current sanitizer and is still written under Conversation Summary.

## Solution Assessment

No, not fully. Current main is a useful partial fix, but it is not yet the best complete solution because orphan role-marker lines and existing poisoned-memory recovery remain unresolved policy and security decisions.
