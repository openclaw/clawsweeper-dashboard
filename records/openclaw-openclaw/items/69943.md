## Label Justifications

- `P1`: The report describes deployed agents becoming progressively non-responsive across new sessions because persisted memory reintroduces malformed context.
- `impact:session-state`: The affected data is persisted per-agent session memory and is reloaded into later session context.
- `impact:security`: Model/tool scaffolding and prior transcript artifacts can cross a trust boundary into future prompts as if they were usable context.
- `impact:message-loss`: The observed loop can suppress future replies by causing agents to emit only NO_REPLY or malformed role-token output.

## Summary

Keep open. Current main and the latest release still pass session-memory transcript text through verbatim and write it under Conversation Summary, while the two linked sanitizer PRs were closed unmerged; the related broader memory-trust issue does not supersede this narrower artifact-sanitization bug.

## What This Changes

The issue reports that the bundled session-memory hook persists raw chat-template tokens, role markers, raw tool-call XML, and NO_REPLY artifacts into memory files that are re-injected into later sessions.

## Best Possible Solution

Add a focused session-memory sanitizer at the transcript/write boundary, with regression tests for model control tokens, raw tool-call XML, orphan role-marker lines, and silent-token-only turns, while leaving broader historical-context wrapping to the related security/product discussion.

## Reproduction Assessment

Yes, source-reproducible: a JSONL assistant message containing `<|im_end|>`, raw `<tool_call>` XML, role-marker lines, or `NO_REPLY` will be returned by getRecentSessionContent and written by the handler without sanitization. I did not run a live provider reproduction because this review is read-only.

## Solution Assessment

Yes, the best narrow fix is to sanitize the session-memory persistence path before memory files are written or reused. A broader redesign of historical session memory trust boundaries should stay with https://github.com/openclaw/openclaw/issues/68751 rather than blocking this artifact-filtering repair.
