## Label Justifications

- `P1`: The report describes a reproducible progressive memory-poisoning loop that can leave real chat agents non-responsive across subsequent sessions.
- `impact:session-state`: The affected data is persisted session memory that is reloaded into future session context.
- `impact:security`: Model-supplied control scaffolding can be persisted and later reintroduced as prompt context.
- `impact:message-loss`: The reported failure mode includes agents degrading into NO_REPLY-only or otherwise suppressed visible responses.

## Summary

Keep open. Current main and the latest release still persist raw session-memory transcript text, and the linked implementation PR with closing syntax remains open rather than merged.

## What This Changes

The issue asks OpenClaw to sanitize or elide raw chat-template tokens, role markers, tool-call XML, and NO_REPLY-only turns before session-memory persists and re-injects prior conversation summaries.

## Best Possible Solution

Review, land, or replace https://github.com/openclaw/openclaw/pull/69961 with a focused sanitizer and regression coverage at the session-memory transcript and write boundaries.

## Reproduction Assessment

Yes, at source level. A session JSONL assistant message containing `<tool_call>...</tool_call><|im_end|>` is returned unchanged by `extractTextMessageContent`, appended by `getRecentSessionContent`, and written under `## Conversation Summary` by the handler.

## Solution Assessment

Yes for the fix direction. Sanitizing or eliding malformed transcript text before persistence is the narrowest maintainable solution, but this issue should stay open until the linked PR lands, is replaced, or is explicitly closed.
