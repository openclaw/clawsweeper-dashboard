## Summary

Keep open: current `main` still has the unbounded skill download response path, and the open closing PR should be reviewed or landed before this issue closes.

## What This Changes

The issue asks for skill `download` installers to reject responses over the installer artifact limit by checking `Content-Length` and enforcing the same cap while streaming.

## Best Possible Solution

Land or replace the linked focused PR with a bounded `installDownloadSpec` implementation, regression coverage, and the docs note, then close this issue once the fix is on `main`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `installDownloadSpec` with a successful `download` response advertising `Content-Length: 268435457` and `extract: false`; current `main` has no preflight or streaming byte cap before success.

## Solution Assessment

Yes. The narrow maintainable fix is to follow the adjacent marketplace pattern: reject oversized advertised lengths and count streamed bytes before the staging file can exceed the shared 256 MiB artifact cap.
