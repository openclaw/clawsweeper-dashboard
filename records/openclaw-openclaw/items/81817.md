## Summary

Keep open: current main still pipes skill download responses into staging without checking Content-Length or counting streamed bytes, and https://github.com/openclaw/openclaw/pull/81818 is an open linked fix candidate that should land or be replaced before this issue closes.

## What This Changes

The issue asks for skill download installers to reject responses over the accepted 256 MiB artifact limit before opening the staging write and while streaming.

## Best Possible Solution

Merge or replace https://github.com/openclaw/openclaw/pull/81818 with a narrow installDownloadSpec cap that rejects oversized advertised lengths, enforces the same 256 MiB limit while streaming, and keeps focused regression/docs coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock installDownloadSpec with a successful response advertising Content-Length 268435457, extract false, and a small body; current main has no preflight or streaming byte cap before success.

## Solution Assessment

Yes. The narrow maintainable solution is to mirror the adjacent marketplace installer pattern by checking advertised size before writing and counting streamed bytes before staging can exceed the shared 256 MiB cap.
