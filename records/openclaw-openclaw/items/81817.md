## Summary

Keep open: current main still writes skill download installer responses to staging without an advertised or streaming byte cap, and the linked implementation PR is still open, so closing this issue before merge would be premature.

## What This Changes

The issue requests bounding skill download installer responses by advertised and streamed byte count before `.openclaw-download-staging` can fill disk.

## Best Possible Solution

Land a narrow hardening fix that reuses the 256 MiB installer artifact limit, rejects oversized `Content-Length` before writing, enforces the same limit while streaming, and documents/tests the behavior.

## Reproduction Assessment

Yes, at source level: current main ignores response `Content-Length`, streams the body to a temp file, and then returns success for `extract: false`. I did not execute the mocked regression because this review was read-only.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix: enforce the existing 256 MiB download-artifact cap before and during streaming. The issue itself should stay open until that PR merges or another equivalent fix lands.
