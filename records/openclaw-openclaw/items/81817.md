## Summary

Keep this open: current `main` and latest `v2026.6.8` still stream successful skill download responses directly into staging without an advertised-size preflight or streamed-byte cap. The linked fixing PR is the right candidate path, but it is open, conflicting, and targets the removed `src/agents` installer files rather than the current lifecycle installer.

## What This Changes

The issue asks for skill `download` installers to reject oversized remote responses before and during streaming.

## Best Possible Solution

Port the advertised-size and streaming-byte cap to the current lifecycle installer path, update the focused regression test and skills docs, then land the linked or replacement PR.

## Reproduction Assessment

Yes, source-level: current `main` routes download install specs to `src/skills/lifecycle/install-download.ts`, where `downloadFile` pipelines the response body to disk without a size preflight or streaming cap. The issue also gives a focused oversized `Content-Length` mock case that maps directly to that path.

## Solution Assessment

Yes on direction, no on the stale linked branch as-is. A fail-closed advertised and streamed response cap is the narrow maintainable repair, but it needs to be applied to `src/skills/lifecycle/install-download.ts` rather than the removed `src/agents` path.
