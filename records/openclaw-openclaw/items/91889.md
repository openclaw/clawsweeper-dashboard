## What This Changes

The branch adds optional `accountId` to public plugin agent-hook context and propagates it through agent harness, tool, reply, focused tests, and hook documentation.

## Best Possible Solution

Expose one stable, turn-bound `accountId` field from authoritative runtime data across every agent-hook path, migrate internal callers to that single name, and prove it with redacted output from an installed plugin hook.

## Reproduction Assessment

Yes at source level: current main omits the turn account from `PluginHookAgentContext`, and group or cron session keys cannot reliably reconstruct it. The branch’s actual plugin-boundary result has not been demonstrated in a real run.

## Solution Assessment

No, not in its current final shape. Adding the field to the existing hook context is the right narrow boundary, but internal callers should use one canonical name and the real plugin boundary needs direct proof.
