## Summary

Keep open: this is useful additive plugin SDK work and it is not already on current main or the latest release, with sufficient terminal proof for the non-visual SDK path. The remaining blocker is compatibility/API ownership: `ctx.accountId` needs maintainer acceptance as a public hook-context field, and the exported helper should not publish the internal `agentAccountId` alias unless maintainers intentionally accept that compatibility debt.

## What This Changes

The PR adds optional `accountId` propagation to plugin agent hook context, exec-env and before-reply hook contexts, harness context rebuilding, tests, and hook documentation.

## Best Possible Solution

Accept canonical `ctx.accountId` if maintainers approve it, but keep the exported helper contract canonical by mapping internal `agentAccountId` to `accountId` at internal call sites instead of publishing the alias.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a public SDK feature PR. Source inspection confirms the current gap because main and `v2026.6.6` have `event.accountId` for `before_agent_run` but no `ctx.accountId` on `PluginHookAgentContext` or the shared context builder.

## Solution Assessment

No, not as submitted. The canonical `ctx.accountId` seam is a good narrow direction, but the best fix should avoid publishing the internal `agentAccountId` alias unless maintainers explicitly want that as public SDK compatibility.
