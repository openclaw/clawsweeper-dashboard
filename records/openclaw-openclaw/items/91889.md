## What This Changes

The PR adds optional `accountId` propagation to plugin agent hook context, exec-env hook context, before-agent-reply paths, harness context rebuilding, tests, and hook docs.

## Best Possible Solution

Accept canonical `ctx.accountId` if maintainers approve it, but keep the exported helper contract canonical by mapping internal `agentAccountId` to `accountId` at internal call sites instead of publishing the alias.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a public SDK feature PR. Source inspection does confirm the current gap because main has `event.accountId` for `before_agent_run` but no `ctx.accountId` on `PluginHookAgentContext` or the shared context builder.

## Solution Assessment

No, not as submitted. The canonical `ctx.accountId` seam is a good narrow direction, but the best fix should avoid publishing the internal `agentAccountId` alias unless maintainers explicitly want that as public SDK compatibility.
