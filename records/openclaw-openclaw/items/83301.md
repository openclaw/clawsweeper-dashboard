The PR overlaps gateway startup logging, plugin service startup, channel startup, and sidecar readiness work to reduce restart-ready latency while preserving readiness waits.

## Best Possible Solution

Preserve the startup overlap while keeping the plugin-services handle authoritative in deferred startup, add regression coverage for the overwrite case, and provide split-branch runtime or benchmark proof.

## Reproduction Assessment

Yes. Source inspection on the PR head shows the deferred branch can return pluginServices: null after the new callback has already reported a real handle, and the current caller assigns that returned null into runtimeState.pluginServices.

## Solution Assessment

No. The scheduling direction is maintainable, but the patch should not introduce a second source of truth that can overwrite the early plugin-services handle with null.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve deferred plugin service handles:** `src/gateway/server-startup-post-attach.ts:887`
  - body: Passing this callback lets deferred sidecars publish a plugin-services handle before post-attach returns, but the defer branch still returns `pluginServices: null`; `startGatewayServer` assigns that return value back to `runtimeState.pluginServices`, so a slow update-check/tailscale/startup-log path can overwrite the real handle and skip plugin-service cleanup on shutdown or reload. Keep the reported handle, or avoid assigning null over an already-reported handle, and cover that ordering.
