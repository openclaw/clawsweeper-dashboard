## Summary

Keep this PR open: it carries the active zh-CN setup localization work that is not on current main, but the current head still exposes zh-CN users to English-only setup paths and the supplied real behavior proof does not cover the current head or an actual setup flow.

## What This Changes

The PR adds a CLI setup locale catalog, zh-CN onboarding copy, locale selection and persistence through `cli.locale`/`OPENCLAW_LOCALE`, plus config schema/help/docs/changelog updates.

## Best Possible Solution

Finish wiring the CLI catalog through the risk acknowledgement and delegated setup helper paths, add focused reset/manual/remote coverage, and require current-head real behavior proof for a zh-CN setup run before merge.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR head selects and persists zh-CN, then immediately calls English-only risk acknowledgement and delegates manual/remote/custom/hook setup to helpers with literal English prompts. I did not run the wizard because this review is read-only.

## Solution Assessment

No. The catalog/config direction is reasonable, but the implementation is not the best complete solution until all exposed setup stages use the same locale contract and current-head proof covers a real setup path.
