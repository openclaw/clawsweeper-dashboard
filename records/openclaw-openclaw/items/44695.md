Keep open: current main still has no CLI setup locale surface, and the latest PR head remains non-mergeable because source inspection shows locale reset, docs/config, changelog, remaining localization, and token-storage copy blockers.

## What This Changes

This PR adds an English/zh-CN CLI setup message catalog, setup locale selection/resolution, `cli.locale` schema support, and localized setup/finalization copy.

## Best Possible Solution

Keep this as the consolidated zh-CN setup localization candidate, but make locale resolution return/persist the effective locale, finish the remaining setup copy localization, document the new config/env contract, add the required changelog entry, and align security copy with current Control UI token storage before maintainer product acceptance.

Yes. Source inspection of PR head gives a high-confidence path: set `OPENCLAW_LOCALE=zh-CN` or start with `baseConfig.cli.locale`, choose existing config Reset, and `applySelectedLocale({})` cannot preserve `cli.locale` because `selectedLocale` is `undefined`.

## Solution Assessment

No. The feature direction is plausible and current main does not replace it, but this patch is not the best mergeable solution until the reset persistence, docs/help, changelog, token-storage copy, and remaining localization gaps are fixed.
