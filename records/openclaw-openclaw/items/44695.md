## Summary

Keep this PR open. Current main still lacks the requested CLI setup localization, but the latest PR head is not merge-ready because reset can still drop the effective locale, the new config/env contract is undocumented, token-storage copy contradicts current Control UI behavior, and some zh-CN setup branches still fall back to English.

## What This Changes

This PR adds an English/zh-CN setup message catalog, resolves setup locale from env/config/LANG/prompt, persists `cli.locale`, updates config schemas, and routes much of setup/finalization copy through the new catalog.

## Best Possible Solution

Keep this as the consolidated zh-CN setup localization candidate, but make locale resolution return and persist the effective locale for env, config, LANG, and prompt paths; document `cli.locale` and `OPENCLAW_LOCALE`; align token-storage copy with sessionStorage; route remaining setup/finalization copy through the catalog; and add focused setup tests before changed-gate validation.

## Reproduction Assessment

Yes. The reset bug is reproducible by inspection: with `OPENCLAW_LOCALE=zh-CN` or an existing `baseConfig.cli.locale`, choose existing config handling `reset`; `selectedLocale` is undefined, so `baseConfig = applySelectedLocale({})` writes no `cli.locale`.

## Solution Assessment

No. The direction is a reasonable consolidated implementation, but the current patch is not the best mergeable solution until the effective locale, docs contract, security-sensitive token copy, and remaining localization gaps are corrected.
