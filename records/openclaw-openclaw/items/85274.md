## Summary

Keep open: current main still seeds ingress agent runs from the agent default even though the documented channel contract says `channels.modelByChannel` applies when there is no session model override. The PR is a focused fix with regression coverage and terminal real-behavior proof; I found no blocking code-review findings, but the auth-profile routing edge remains a maintainer merge-risk decision.

## What This Changes

The PR updates ingress agent model selection to apply `channels.modelByChannel` before stored or explicit overrides, and adds agent-command regression tests/mocks for aliases, precedence, and auto-fallback primary probes.

## Best Possible Solution

Land the focused model-routing fix with its regression tests once maintainers decide whether the auth-profile persistence edge can be accepted as a tracked follow-up or must be folded in first.

## Reproduction Assessment

Yes, this is source-reproducible: current main starts ingress runs from `resolveDefaultModelForAgent()` and never consults `resolveChannelModelOverride`, while docs say channel mappings should apply when no session model override exists. The PR body also supplies before/after terminal proof from a local gateway run.

## Solution Assessment

Yes for the model-routing bug: reusing the existing channel override resolver and existing alias-aware model resolver is the narrow maintainable path, and the tests cover the precedence chain. The only unsettled solution question is whether the auth-profile persistence edge should be merged as a follow-up or included before this PR lands.
