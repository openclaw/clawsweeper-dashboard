## Summary

Keep this PR open, but it is not merge-ready: the branch targets a real current-main diagnostics gap, yet it reads and preserves an undeclared plugin-tool `availability` field, which turns the fix into an implicit plugin SDK/API surface without the typed contract, docs, or maintainer-visible API decision.

## What This Changes

The PR preserves plugin-authored tool availability in cached descriptors and logs empty `allOf`/`anyOf` availability diagnostics during plugin tool descriptor capture.

## Best Possible Solution

Surface empty availability-group diagnostics once at plugin descriptor registration or planner integration while either documenting and typing plugin-tool `availability` as a real SDK contract or avoiding a new plugin-tool field entirely.

## Reproduction Assessment

Yes. Source inspection on current main shows empty availability groups return `unsupported-signal` diagnostics, while plugin descriptor capture omits `availability` and the planner only records hidden diagnostics without logging.

## Solution Assessment

No, not yet. The registration-boundary warning is the right general direction, but the current patch should not merge until the plugin-tool availability field is either made an explicit SDK contract or removed from the fix shape.
