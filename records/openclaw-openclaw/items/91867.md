## Label Justifications

- `P2`: This is a normal-priority channel delivery design issue with a real duplicate-message symptom but a bounded affected surface.
- `impact:message-loss`: The motivating behavior is duplicated user-visible channel text when one logical final reply contains both text and media.

## Summary

Keep open. Current main still excludes ordinary final text-plus-media payloads from the plugin `sendPayload` path and plans media captions in core, while the related open bug only tracks the narrower duplicate-text reproduction.

## What This Changes

The issue requests a channel.message design invariant where one logical text-plus-media final reply stays intact until the channel plugin/backend chooses the physical send shape.

## Best Possible Solution

Maintainers should first define the durable `channel.message` logical-payload contract, then update final delivery and channel adapter tests so eligible text-plus-media replies either delegate through the plugin payload path or preserve equivalent plugin-owned physical-send semantics.

## Reproduction Assessment

Yes, at source level: current main still has block replies that do not extract `MEDIA:` directives, final replies that do, and ordinary media final delivery that plans core media sends with remaining text as caption. I did not run a live Bncr transport reproduction during this read-only review.

## Solution Assessment

Unclear until maintainer design review. The requested plugin/backend ownership is policy-aligned, but choosing `sendPayload` or equivalent adapter semantics for ordinary final delivery changes a durable channel contract and needs an explicit compatibility path.
