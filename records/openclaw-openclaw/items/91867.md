## Summary

Keep open. Current main and v2026.6.6 still send ordinary final text+media payloads through core media planning instead of the plugin `sendPayload` path; the linked duplicate-text bug was narrowed, but this broader channel/plugin contract question still needs maintainer design review.

## What This Changes

Request to preserve logical text-plus-media final reply payloads through `channel.message` so channel plugins can choose the physical split or merge behavior.

## Best Possible Solution

Define a durable final text+media contract that lets capable channel plugins receive one logical payload through `sendPayload` or an explicit new capability, with compatibility tests for existing core delivery behavior.

## Reproduction Assessment

Yes for source-level reproduction: current main still routes ordinary final text+media through core media-unit planning rather than plugin `sendPayload`. I did not run a live Bncr transport reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the channel/plugin contract. The narrow duplicate-text dedupe fix is not the full design solution, and an unconditional sendPayload switch would be compatibility-sensitive across channels.
