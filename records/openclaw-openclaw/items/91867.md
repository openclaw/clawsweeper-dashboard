## Summary

Keep open. Current main fixes the narrower duplicate-text symptom, but ordinary final text-plus-media payloads still fall back to core media-unit planning instead of consistently handing one logical payload to the channel/plugin send backend, so the broader invariant remains a maintainer product/API decision.

## What This Changes

The issue requests that `channel.message` preserve one logical text-plus-media payload until the channel plugin/backend chooses the physical send shape.

## Best Possible Solution

Have maintainers choose the durable `channel.message` contract, then implement a typed channel/plugin path that preserves logical text-plus-media payloads while retaining receipts, hooks, pinning, and fallback behavior under focused tests.

## Reproduction Assessment

Yes, for a source-level reproduction of the design gap: current main excludes ordinary text-plus-media payloads from `sendPayload` and then core-plans media units with leading caption text. I did not run live Bncr or Telegram transport proof in this read-only review.

## Solution Assessment

Unclear until maintainers choose the durable contract. The narrow duplicate-text fix is present, but a broad solution likely needs an explicit typed seam or contract update rather than an ad hoc predicate expansion.
