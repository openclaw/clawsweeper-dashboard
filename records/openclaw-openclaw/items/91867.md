## Summary

Current main still performs physical text/media planning in core before primitive adapter sends, while the existing rich-payload adapter seam is not used for ordinary final media delivery. The related bug at https://github.com/openclaw/openclaw/issues/91861 covers one concrete duplicate-text failure, but it does not replace this broader compatibility-sensitive contract decision, so this design issue remains necessary.

## What This Changes

Define a channel delivery invariant that preserves logical text-plus-media payloads until the channel plugin selects the physical send representation.

## Best Possible Solution

Standardize one capability-aware logical payload handoff for final delivery, preserving core durability, hooks, receipts, and reply policy while explicitly falling back to the existing primitive planner for adapters that do not support rich payload sends.

## Reproduction Assessment

Yes for the source-level premise: current main converts one logical text-plus-media payload into core-planned media units before adapter delivery. The narrower user-visible duplicate has live evidence in https://github.com/openclaw/openclaw/issues/91861.

## Solution Assessment

The proposed invariant is directionally the best maintainable approach, but the exact API is not yet settled because capability negotiation, primitive-only adapter fallback, and durable-final lifecycle behavior must be specified together.
