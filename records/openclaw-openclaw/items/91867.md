## Label Justifications

- `P2`: This is a normal-priority channel delivery design issue with a real duplicate-message symptom but bounded blast radius.
- `impact:message-loss`: The motivating behavior is duplicated visible text when one logical final reply contains text plus media.

## Summary

Keep open. Current main and v2026.6.5 still route ordinary final text+media payloads through core text/media planning instead of the plugin payload path, while https://github.com/openclaw/openclaw/issues/91861 only tracks the narrower duplicate-text bug.

## What This Changes

Request that channel.message preserve one logical text+media final-reply payload until the channel plugin/backend chooses the physical send shape.

## Best Possible Solution

Define the channel.message logical payload contract first, then update durable final delivery to delegate eligible text+media payloads through the channel message/payload adapter with focused compatibility tests for existing text, media, reply, receipt, and queue behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection shows the current design path: block replies do not parse MEDIA directives, final replies do, and ordinary media final delivery is split by core rather than delegated as one payload.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer design review; plugin/backend ownership is plausible and policy-aligned, but maintainers need to choose the durable-delivery contract and compatibility path before implementation.
