## Summary

Keep open: the linked duplicate-text bug is fixed, but the broader channel.message contract remains unresolved because ordinary text+media final payloads are still planned as core text/media sends before a plugin payload adapter can choose the physical send shape.

## What This Changes

Request to preserve channel.message final replies as one logical text+media payload until the channel plugin or backend chooses the physical send shape.

## Best Possible Solution

Choose an explicit durable final-delivery contract, then route eligible logical text+media replies through the message payload adapter with regression coverage for receipts, reply-to, media sequencing, and fallback channels.

## Reproduction Assessment

Yes for source-level reproduction: current main and v2026.6.6 still route ordinary text+media final replies through core media planning rather than the payload adapter. I did not run a live Bncr transport reproduction in this read-only review.

## Solution Assessment

Unclear. Plugin-owned physical send shape is policy-aligned, but this is a durable-delivery and plugin API contract decision now that the narrower duplicate-text bug is fixed.
