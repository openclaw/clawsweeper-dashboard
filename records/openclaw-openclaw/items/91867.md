## Summary

Keep open. Current main fixes the narrower duplicate-text bug, but ordinary final text-plus-media payload delivery still falls back to core media-unit planning instead of consistently handing one logical payload to the channel/plugin send backend; this remains a maintainer product/API boundary question.

## What This Changes

The issue requests that runtime `channel.message` preserve one logical text-plus-media payload until the channel plugin/backend chooses the physical send shape.

## Best Possible Solution

Have maintainers choose the intended durable `channel.message` contract, then implement a typed channel/plugin path that preserves logical payload shape for ordinary text-plus-media replies while keeping queueing, receipts, hooks, and upgrade behavior covered by focused tests.

## Reproduction Assessment

Yes for source-level reproduction: current main still sends ordinary text-plus-media payloads through core media-unit planning rather than the plugin `sendPayload` path. I did not run live Bncr or Telegram transport proof during this read-only review.

## Solution Assessment

Unclear until maintainers decide the contract. The narrow duplicate-text bug fix is already present, but the broader best fix likely needs an explicit durable channel/plugin payload seam rather than extending ad hoc media routing.
