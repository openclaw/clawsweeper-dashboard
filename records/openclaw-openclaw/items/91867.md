## Summary

Keep open. The narrow duplicate-text bug is fixed on current main, but current main and v2026.6.6 still route ordinary final text+media payloads through core media-unit planning instead of preserving the logical payload for plugin-owned physical send semantics.

## What This Changes

Request to keep `channel.message` as the final-reply runtime contract while preserving one logical text+media payload until the channel plugin chooses the physical send shape.

## Best Possible Solution

Define durable `channel.message` logical-payload semantics, then route eligible final text+media through message/payload adapters with conformance tests for receipts, reply threading, queue recovery, and media fallback behavior.

## Reproduction Assessment

Yes for source-level reproduction: current main still sends ordinary text+media final payloads through core media-unit planning rather than the plugin payload path. I did not run a live Bncr transport repro in this read-only review.

## Solution Assessment

Unclear until maintainer design review. The requested plugin-owned send shape is policy-aligned, but choosing it safely requires a durable-delivery compatibility decision rather than extending the narrow duplicate-text fix.
