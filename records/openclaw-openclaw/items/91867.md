## Label Justifications

- `P2`: This is a normal-priority channel delivery design issue tied to visible duplicate message delivery but not a release emergency.
- `impact:message-loss`: The issue concerns duplicated or incorrectly shaped channel message delivery for mixed text-plus-media replies.

## Summary

Keep open: current main still has a real channel-message contract gap where plain text-plus-media payloads can fall through core's physical media planning instead of always reaching a plugin payload sender, and the narrower duplicate-text bug is still open separately.

## What This Changes

Request to preserve one logical text-plus-media channel.message payload until the channel plugin or backend chooses the physical send shape.

## Best Possible Solution

Define the channel.message final-delivery invariant so core normalizes one logical reply payload and plugin adapters with payload support choose text/media splitting, while the concrete duplicate-text behavior remains tracked by https://github.com/openclaw/openclaw/issues/91861.

## Reproduction Assessment

Yes for source-level reproduction: current main still routes ordinary text-plus-media payloads through core media planning instead of sendPayload, and the block/final MEDIA directive paths remain asymmetric. I did not run a live Bncr transport repro in this read-only review.

## Solution Assessment

Unclear as an implementation plan. The requested invariant is a plausible clean endpoint, but maintainers should decide the channel/plugin contract before an automated repair changes payload routing across adapters.
