## What This Changes

The issue requests that `openclaw status` show live channel state when the gateway is already reachable instead of reporting fast-mode placeholders.

## Best Possible Solution

Resolve this through a maintainer-reviewed fix that either uses bounded live gateway channel data to render these rows, or changes the fast-mode row wording so default status is no longer self-contradictory.

Yes, source-reproducible: default text status skips live `channels.status`, and the table builder still emits the reported fast-mode placeholder for configured channels when fallback plugins are skipped. I did not run a live configured gateway in this read-only review.

Unclear until maintainers choose the fast-status contract. The open PR is a narrow candidate, but the table code still has a placeholder path that does not consult live status, so the final fix should be verified against a real configured gateway.
