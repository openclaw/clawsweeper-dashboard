## Summary

## What This Changes

The issue asks for `openclaw status` to show live channel health when the same command has already proven the local gateway is reachable.

## Best Possible Solution

Make fast status honest and non-contradictory: either query `channels.status` after a successful gateway probe, or keep fast mode but render channel state as skipped/unknown with a direct `openclaw status --deep` or `openclaw channels status` hint.

## Reproduction Assessment

Yes, source-reproducible: default text status intentionally skips `channels.status`, and the table code renders configured skipped channels with the reported fast-mode detail. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

Unclear: fetching live channel status by default may be useful, but it conflicts with docs that reserve live probes for `openclaw status --deep`. Maintainers should pick the latency/UX direction before implementation.
