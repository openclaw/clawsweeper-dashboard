## What This Changes

`openclaw status` is reported to show placeholder channel rows even when the same run has already confirmed the gateway is reachable.

## Best Possible Solution

Resolve this through the active linked PR or an equivalent maintainer-approved fix: either fetch bounded non-probe channel status after a reachable gateway, or make the fast-path row wording/state non-contradictory while preserving the documented fast path.

Yes, source-reproducible: current `scanStatus` passes `includeLiveChannelStatus: isFullScan`, so plain text status skips `channels.status` even after a reachable gateway probe. The adjacent test also asserts no `channels.status` call in default text status.

## Solution Assessment
