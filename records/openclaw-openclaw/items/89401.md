## Summary

Keep open: current `main` and the latest release still keep plain text status off live channel status, so the reported fast-mode placeholder remains possible; the direct fix path is the open, unmerged linked PR.

## What This Changes

`openclaw status` should avoid reporting configured channels as fast-mode-unavailable when the same run has already confirmed the gateway is reachable.

## Best Possible Solution

Resolve this through the linked PR or an equivalent maintainer-approved fix that either fetches bounded live channel status after a reachable gateway probe or changes the fast-path channel wording so it is not self-contradictory.

## Reproduction Assessment

Yes, source-reproducible: current `main` passes `includeLiveChannelStatus: isFullScan`, and the adjacent test asserts default text status does not call `channels.status` after a reachable gateway probe. I did not run a live configured gateway reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the fast-status contract. The linked PR is the narrow code path for live status, but preserving strict fast status with clearer row wording is the safer alternative if the default RPC is not accepted.
