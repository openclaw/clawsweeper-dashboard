## Summary

Keep open. Current main and the latest release still omit `errorBody` from the Anthropic thinking-recovery detector even though provider HTTP failures and assistant error classification carry raw provider detail there, so the reported carrier gap is not implemented or fully covered by the existing genericized-error fixes.

## What This Changes

The issue requests that Anthropic thinking-block recovery inspect `ProviderHttpError.errorBody` so invalid-signature replay errors trigger the existing strip-and-retry repair path.

## Best Possible Solution

Add focused `errorBody` and provider body/detail carrier coverage to the existing recovery detector with regression tests, while preserving the current one-retry and false-positive boundaries and coordinating with the broader Anthropic replay cluster.

## Reproduction Assessment

Yes at source level: a ProviderHttpError-like object with generic `message` and matching provider JSON in `errorBody` is not traversed by the current detector. I did not run a live Anthropic repro in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to extend the existing recovery detector and tests for provider body/detail carriers, not add a config knob or new recovery system. The implementation should avoid scanning arbitrary assistant content or tool arguments.
