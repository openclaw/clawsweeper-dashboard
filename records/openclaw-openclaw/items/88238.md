## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Align the source-wide chunk cap with provider batch capacity:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:900-904`
  - body: The source-wide path still flushes a provider batch whenever the accumulated chunks exceed 4096. The reporter's latest log shows this exact path: 1328 files are considered together, but group 1 stops at 3871 chunks with `reason=max-chunks`, so the remaining files will be submitted as another provider batch job even though OpenAI-compatible batching is capped later at 50000 requests. That leaves the linked large memory+session case only partially fixed; the source-wide grouping should stay together until the provider batch runner's request limit or another documented provider limit is reached.
  - confidence: 0.91
- **[P2] Expose the generic batch runtime contract instead of casting:** `extensions/memory-core/src/memory/embeddings.ts:74-85`
  - body: `adaptGenericRuntime` reads `sourceWideBatchEmbed` and `batchEmbed` from a generic embedding provider runtime via a local cast, but the public `EmbeddingProviderRuntime` type still only exposes cache and timeout fields. External generic providers therefore cannot implement the new source-wide batch behavior without hidden properties and casts, and the ordering/wait/concurrency/fallback semantics are not part of the SDK contract. Add the typed runtime surface and contract coverage, or keep this behavior on the memory-specific provider runtime only.
  - confidence: 0.84
