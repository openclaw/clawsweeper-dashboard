## Summary

## What This Changes

The PR changes memory-core sync so batch-enabled provider runtimes can embed dirty memory and session chunks across multiple files while preserving per-file index writes.

## Best Possible Solution

Land this after maintainers accept cross-file batchEmbed semantics and the current proof level, or request provider-cap segmentation or a live redacted CLI batch proof before merge.

## Reproduction Assessment

Yes. Current main is source-reproducible because memory and session sync call indexFile per dirty file, and the reporter's forced indexing log shows repeated batch submissions that match that path.

## Solution Assessment

Mostly yes. Moving batching to the dirty source-sync boundary while preserving per-file writes is the narrow maintainable fix; the remaining decision is whether maintainers accept the changed provider-runtime batch contract and proof level.
