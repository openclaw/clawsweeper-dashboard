## Summary

Keep open for maintainer review: current main still submits runtime memory embedding batches from the per-file indexing path, while this PR directly addresses the linked performance issue with focused memory-core changes, regression coverage, and sufficient local production-path proof. The remaining decision is whether maintainers accept the provider-runtime contract change for cross-file chunk arrays.

## What This Changes

The PR batches dirty memory-source chunks across files through the memory provider runtime batch API while preserving per-file index writes and adds a regression test for that cross-file batch path.

## Best Possible Solution

Land the cross-file memory-source runtime batch path if maintainers accept aggregated dirty-file chunks as the batchEmbed contract, while keeping session, FTS-only, and multimodal handling on their current paths.

## Reproduction Assessment

Yes, from source inspection and the linked issue evidence: current main builds one dirty-file task per memory file and each task calls indexFile, so provider runtime batch submission happens at file boundaries. I did not rerun a live remote provider reproduction.

## Solution Assessment

Mostly yes: the patch moves batching to the memory-source dirty-file boundary and preserves per-file database writes, which is the narrow code path implicated by the issue. The unresolved solution-fit point is maintainer acceptance of cross-file batchEmbed semantics and whether live remote-provider proof is required before merge.
