## Summary

Keep this PR open: current main still batches provider runtime memory embeddings one file at a time, while this branch adds a focused cross-file dirty-memory batching path with a regression test and terminal proof. I found no concrete blocking code or security finding; the only material review caveat is that the after-fix proof uses the production runtime seam with a local provider adapter rather than a live remote batch service.

## What This Changes

This PR adds a memory-core multi-file indexing path that prepares dirty memory-file chunks together for provider runtime batch embedding, while preserving per-file index writes and adding a regression test.

## Best Possible Solution

Land a focused cross-file batching implementation for memory-source runtime batch providers, with the existing regression test and either maintainer acceptance of the non-live proof gap or an optional live batch smoke before merge.

## Reproduction Assessment

Yes. Current main source shows memory files are indexed through per-file tasks, and each dirty file calls the batch embedding path with only that file's chunks; I did not run the repro because this review is read-only.

## Solution Assessment

Yes. The PR moves aggregation to the memory-source sync boundary while keeping sessions, FTS/no-runtime providers, configured cache behavior, stale pruning, and multimodal entries on the existing per-file paths.
