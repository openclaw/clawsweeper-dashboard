## Summary

Keep open: the report still describes a valid session-bricking provider replay bug on current main, and three open pull requests already reference it as the implementation path. Current source still has the raw Anthropic wire path that can pass a nonconforming composite tool-call id through, so the issue should stay open until one focused fix PR lands.

## What This Changes

The issue asks OpenClaw to sanitize OpenAI Responses composite tool-call ids before replaying them to Anthropic so failover history cannot permanently brick a session.

## Best Possible Solution

Land one focused fix that guards Anthropic wire ids and keeps assistant/tool-result pairing stable, then close this issue from the merged PR; treat raw-shape sanitizer broadening as separate hardening unless included cleanly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: construct an Anthropic transport replay containing a `toolCall`/`toolResult` pair with a `call_...|fc_...` id on a path the transform treats as same-model, and current main serializes the raw id to `tool_use.id`.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is a deterministic Anthropic wire-boundary id guard with pairing-preserving regression coverage; sanitizer raw-shape widening is useful but not sufficient for the reported camelCase persisted path.
