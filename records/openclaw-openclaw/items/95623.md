## Summary

Keep open: current main still has a source-backed provider replay gap where Anthropic-target replay is configured to sanitize foreign IDs, but the canonical sanitizer only rewrites normalized camel-case tool-call blocks and can miss raw replay shapes carrying OpenAI Responses composite IDs.

## What This Changes

The issue reports that OpenAI Responses composite tool-call IDs containing `|` can survive cross-provider failover replay into Anthropic, causing repeated Anthropic `tool_use.id` validation failures.

## Best Possible Solution

Provider replay should sanitize every foreign assistant/tool-result ID shape before Anthropic transport while preserving native Anthropic IDs and same-target OpenAI Responses replay semantics.

## Reproduction Assessment

Yes at source level, not live: construct an Anthropic-target replay history containing an OpenAI Responses `call_abc|fc_prior` assistant/tool-result pair and verify the current sanitizer only rewrites normalized camel-case assistant blocks.

## Solution Assessment

Yes: the narrow maintainable fix is to broaden the canonical sanitizer's recognized replay ID and result ID shapes, not add a config knob or flip Anthropic policy globally.
