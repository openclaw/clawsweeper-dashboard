## Label Justifications

- `P1`: The report describes a real provider-failover workflow that can permanently poison a session transcript and prevent the recovered Anthropic primary from running.
- `impact:session-state`: The invalid tool-call ID is persisted in session history and replayed on every later Anthropic turn.

## Summary

Keep open: current main still has a credible provider-failover replay gap where the strict tool-call-id sanitizer is wired for Anthropic replay but only rewrites the normalized camel-case tool-call shapes, while adjacent transcript-repair code already recognizes raw `tool_use`/`function_call` shapes that can appear in replay data.

## What This Changes

The issue requests that OpenAI Responses composite tool-call IDs introduced during fallback replay be sanitized before later Anthropic replay, while preserving native Anthropic `toolu_*` IDs.

## Best Possible Solution

Extend the canonical replay ID sanitizer to cover the raw replay block and ID field shapes that already appear in transcript repair, then add a focused Anthropic-target regression for an OpenAI Responses `call_...|fc_...` failover transcript.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live-run no: construct an Anthropic-target replay history containing an OpenAI Responses `call_abc|fc_prior` assistant/tool-result pair and verify the current sanitizer only rewrites normalized camel-case tool-call blocks.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to broaden the canonical sanitizer's recognized replay ID shapes and preserve existing OpenAI Responses behavior, not to add a new config knob or flip Anthropic policy globally.
