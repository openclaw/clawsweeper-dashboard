## Summary

Keep open. Current main and the latest release still derive titles from display name, subject, the first transcript user message, then a session-id/date fallback; the linked closing PR remains open and only partially addresses the requested behavior.

## What This Changes

Improve session derived-title generation so session lists prefer conversation-core signals such as compaction summaries or meaningful later user messages over low-information first prompts.

## Best Possible Solution

Land or replace the linked implementation with a bounded Gateway title-source path that preserves displayName/subject precedence, can use stored compaction summaries when appropriate, and supplies a meaningful transcript candidate instead of low-information first prompts.

## Reproduction Assessment

Yes, from source inspection. A session whose first user message is a greeting still flows through readSessionTitleFieldsFromTranscript into deriveSessionTitle, which returns that first message before any session-id fallback.

## Solution Assessment

Unclear. The issue's direction is plausible, but maintainers need to decide whether the acceptable fix is the full compaction-summary and meaningful-message source path or a narrower greeting-filter mitigation.
