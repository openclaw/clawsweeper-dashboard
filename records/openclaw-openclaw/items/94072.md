## What This Changes

The PR changes `buildEmbeddedRunPayloads` so `message_tool_only` source replies delivered by the message tool count as a user-facing assistant reply when deciding whether to append tool-error warnings.

## Best Possible Solution

Land this maintainer-opted PR or an identical canonical fix, then close the linked bug and duplicate PRs; add the exact regression test only if maintainers want stronger guard coverage before merge.

## Reproduction Assessment

Yes, source inspection of current main shows the reported path: `message_tool_only` delivery can set `deliveredSourceReplyViaMessageTool` while `hasUserFacingAssistantReply` remains false, so a non-recoverable non-mutating `lastToolError` can append a stale warning. The PR body also includes before/after terminal output from a script calling the production payload builder with those flags.

## Solution Assessment

Yes, this is the narrowest maintainable fix: the payload builder should count the existing runner-owned delivery fact as a user-facing reply. Moving this into Discord-specific delivery or globally weakening the tool-error policy would duplicate ownership or risk suppressing unrelated warnings.
