## Summary

Keep open: current main still returns an unavailable builtin memory manager when `node:sqlite` cannot load, while this PR is a viable candidate implementation for the open keyword-fallback issue. I found no concrete line-level correctness defect, but the degraded response shape and per-search markdown scan are compatibility and availability decisions that need maintainer acceptance before merge.

## What This Changes

Adds a memory-core keyword fallback manager for explicit missing `node:sqlite` failures and a regression test that searches markdown memory files through the fallback path.

## Best Possible Solution

Land the fallback only if maintainers accept degraded keyword recall as the supported missing-`node:sqlite` behavior; otherwise keep the narrower unavailable-message path as the supported response.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source/proof level: current main returns an unavailable manager when `MemoryIndexManager.get()` throws the missing-`node:sqlite` error, and the PR body includes after-fix CLI output with `node:sqlite` blocked at the loader seam.

## Solution Assessment

Is this the best way to solve the issue? Yes as a technical mitigation at the manager-resolution seam, because it keeps SQLite/vector search canonical when available and reuses the existing memory-file boundary; the remaining question is maintainer acceptance of the degraded mode.
