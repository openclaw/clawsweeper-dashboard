## Summary

Keep open: the PR is a useful candidate for the linked missing-`node:sqlite` fallback issue, but it introduces a no-op sync path that can make CLI indexing report success even though no SQLite index exists.

## What This Changes

The PR adds a memory-core keyword fallback manager for explicit missing `node:sqlite` failures and a regression test for markdown memory search through that fallback path.

## Best Possible Solution

Fix the fallback manager's index/status contract, then have maintainers decide whether degraded keyword fallback is the supported missing-node:sqlite behavior.

## Reproduction Assessment

Yes at source/proof level: current main returns an unavailable manager when MemoryIndexManager.get throws the missing-node:sqlite error, and the PR body includes after-fix CLI output with node:sqlite blocked at the loader seam. I did not run a live downlevel Node build in this read-only review.

## Solution Assessment

No, not as-is: the manager-resolution seam and memory-file helper reuse are the right technical shape, but the fallback should not advertise successful manual indexing through a no-op sync method. After that repair, the remaining decision is maintainer acceptance of degraded keyword fallback semantics.
