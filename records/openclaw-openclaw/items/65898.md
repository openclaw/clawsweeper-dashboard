## Summary

Keep this PR open. Current main still does not implement a separate first-token watchdog, so the contribution is not obsolete, but the branch is stale against the current provider-scoped timeout direction and has concrete merge-blocking review issues around provider timeout preservation, doctor/config consistency, and QA timeout leak detection.

## What This Changes

The PR adds a two-phase LLM stream timeout with `agents.defaults.llm.firstTokenTimeoutSeconds`, propagates `llmTimeoutPhase` through embedded-run attempt results, updates timeout error text and config schema/types, and expands related tests/fixtures.

## Best Possible Solution

Rebase/adapt the feature onto current main, preserve provider-scoped request timeout behavior, decide explicitly whether first-token tuning belongs under provider config or a reintroduced agent-level LLM config, update doctor/schema/docs together, and add QA detection coverage for both first-token and idle timeout messages before merge.

## Reproduction Assessment

Yes for review purposes: comparing current main with the PR merge candidate shows the missing `modelRequestTimeoutMs` propagation, the legacy-config conflict, and the QA regex gap without needing a live LLM run. The runtime motivation remains plausible and is supported by related timeout reports such as #53540, but live provider reproduction was not run in this read-only review.

## Solution Assessment

No. Splitting first-token and mid-stream idle phases is a reasonable direction, but this PR is not yet the best implementation because it conflicts with current provider-scoped timeout behavior and leaves adjacent QA/config migration surfaces inconsistent.
