## Summary

Keep this PR open. The feature direction is useful and matches real timeout pressure, and the author already addressed the early first-token-disable and parameter-shadowing review comments, but the current branch is stale against `main`: it drops provider request-timeout preservation and adds a new key under `agents.defaults.llm`, which current `main` treats as legacy doctor-migration state.

## What This Changes

The PR splits embedded-agent LLM stream watchdogs into first-token and mid-stream idle phases, adds `agents.defaults.llm.firstTokenTimeoutSeconds`, propagates timeout phase metadata through attempt results, and updates config schema/types plus related tests and fixtures.

## Best Possible Solution

Add two-phase timeout handling only after aligning it with the current provider-scoped timeout contract: preserve `modelRequestTimeoutMs` for both phases, decide whether first-token tuning belongs under provider config or another supported surface, update doctor/schema/docs together, and extend QA timeout leak detection for the new user-facing phase text.

## Reproduction Assessment

Yes for review purposes. Source inspection of current `main` plus the supplied PR diff directly reproduces the patch-introduced regressions: provider timeout is dropped from resolver parameters, the new config key sits under a doctor-removed block, and the QA matcher does not recognize the new first-token timeout text.

## Solution Assessment

No. Splitting first-token and idle windows is a reasonable direction, but this branch is not the best implementation because it conflicts with the current provider timeout ownership and leaves migration and QA surfaces inconsistent.
