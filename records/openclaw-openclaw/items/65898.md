## Summary

Keep open: splitting first-token and mid-stream idle timeouts is a plausible feature direction, but this branch is stale against current main and would regress provider-scoped timeout behavior, legacy-config handling, stream abort plumbing, and QA timeout detection.

## What This Changes

The PR adds a first-token LLM streaming timeout phase, a new `agents.defaults.llm.firstTokenTimeoutSeconds` config key, timeout-phase result metadata, schema updates, and focused tests.

## Best Possible Solution

Define first-token timeout semantics on a supported provider/model timeout surface, preserve current provider request and local-provider watchdog behavior, update doctor/schema/docs/QA together, and prove the result in a real slow or cold model setup.

## Reproduction Assessment

Yes for review purposes: source inspection of current main and the PR head gives a high-confidence path to the review defects without running a live slow-model setup. The user-facing feature itself still needs real behavior proof after the branch is corrected.

## Solution Assessment

No. The two-phase timeout idea is reasonable, but this branch is not the best implementation until config ownership is resolved and current provider-scoped timeout, local-provider, abort, and QA contracts are preserved.
