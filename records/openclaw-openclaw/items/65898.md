## Summary

Keep open: current main does not implement the requested first-token timeout split, and this PR still has correctness blockers around current timeout inputs, abort behavior, config ownership, QA detection, and missing real behavior proof.

## What This Changes

The PR adds a two-phase LLM stream watchdog with separate first-token and idle timeout handling, a new `agents.defaults.llm.firstTokenTimeoutSeconds` config field, timeout-phase result metadata, schema updates, and focused tests.

## Best Possible Solution

Define two-phase timeout semantics on a supported provider/model-scoped surface, preserve current provider timeout/local-provider/abort contracts, update doctor/schema/docs/QA together, and prove it in a real slow or cold model setup.

## Reproduction Assessment

Yes for the review defects: current-main source and the PR diff show the omitted provider/model timeout inputs, lost abort plumbing, legacy config surface, and QA matcher gap. The user-facing slow-model behavior still lacks real runtime proof.

## Solution Assessment

No. The two-phase timeout idea is reasonable, but this branch is not the best implementation until it targets the current supported timeout surface and preserves provider timeout, local-provider, abort, doctor, and QA contracts.

- **[P2] Preserve provider timeout inputs:** `src/agents/pi-embedded-runner/run/attempt.ts:1296-1300`
  - body: Current main passes the selected model's request timeout and base URL into `resolveLlmIdleTimeoutMs`, preserving provider `timeoutSeconds` and local-provider watchdog opt-out behavior.   - confidence: 0.93

  - body: Current main wraps the provider stream with a fresh abort signal and clamps `model.requestTimeoutMs` to the watchdog. The phased wrapper calls `baseFn` with the original model/options and, on timeout, only calls the hook and rejects the local race, so the timed-out provider request can keep running and request-timeout bounds are lost.
  - confidence: 0.9

  - body: Current main removes `agents.defaults.llm` via doctor and documents slow provider/model tuning under `models.providers.<id>.timeoutSeconds`; the agent-defaults `llm` schema is gone. Adding `firstTokenTimeoutSeconds` under that legacy block advertises a key that current repair flow can delete.
  - confidence: 0.91
- **[P2] Classify first-token errors in QA leak detection:** `src/agents/pi-embedded-runner/run.ts:1536-1541`
  - body: The PR adds a new first-token timeout message, but QA transcript failure detection only matches the existing idle-timeout wording. A transcript containing the new first-token failure could be judged clean unless the QA matcher and its tests classify both timeout messages.
  - confidence: 0.86
