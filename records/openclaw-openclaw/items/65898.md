## Summary

Keep this PR open. The feature direction is useful and matches real timeout reports, but the current branch regresses provider-scoped timeout handling and reintroduces a config key under a block current main treats as legacy and removes via doctor. The previous ClawSweeper review comments remain substantively valid after checking current main.

## What This Changes

The PR changes embedded-agent LLM streaming from one idle watchdog to separate first-token and mid-stream idle timeout phases, adds `agents.defaults.llm.firstTokenTimeoutSeconds`, propagates timeout phase metadata through attempt results, updates schema/types, and expands related tests/fixtures.

## Best Possible Solution

Split first-token and mid-stream idle handling only after the timeout contract is reconciled with current provider-scoped configuration: preserve `modelRequestTimeoutMs`, decide where first-token tuning belongs, update doctor/schema/docs together, and keep QA transcript-failure detection phase-aware.

## Reproduction Assessment

Yes for review purposes: current main plus the supplied PR diff show the provider-timeout parameter drop, the doctor migration conflict, and the QA regex gap directly. A live LLM reproduction was not needed to confirm those patch-introduced regressions.

## Solution Assessment

No. The two-phase timeout model is a reasonable product direction, but this PR is not the best implementation yet because it conflicts with current provider timeout ownership and leaves migration and QA surfaces inconsistent.
