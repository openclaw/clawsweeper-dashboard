## Summary

Keep open. The first-token versus idle split is a useful direction, but this PR should not merge as-is because current main moved slow model timeout ownership to provider-scoped request timeouts while the branch adds a new agent-level legacy key, drops provider timeout propagation in the stream watchdog path, and leaves the new first-token timeout text outside QA transcript failure detection.

## What This Changes

This PR changes embedded-agent LLM stream timeout handling to support separate first-token and mid-stream idle phases, adds `agents.defaults.llm.firstTokenTimeoutSeconds`, propagates timeout phase metadata through attempt results, and updates schema/types plus tests and fixtures.

## Best Possible Solution

Implement two-phase timeout handling only after aligning it with the current provider-scoped timeout contract: preserve `modelRequestTimeoutMs` for both phases, decide whether first-token tuning belongs under provider config or another supported surface, update doctor/schema/help/docs together, and extend QA timeout leak detection for the new user-facing phase text.

## Reproduction Assessment

Yes for review purposes. Source inspection of current main plus the supplied PR diff reproduces the patch regressions: provider timeout is omitted from resolver params, the new setting is placed under a doctor-removed legacy block, and first-token timeout text is not classified by QA transcript failure detection.

## Solution Assessment

No. Splitting first-token and idle windows is reasonable, but this branch is not the best implementation because it conflicts with current timeout ownership and leaves migration and QA surfaces inconsistent.
