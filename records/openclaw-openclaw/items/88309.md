## Summary

Keep open. Current main has several launchd restart protections, but the report is fresh, matches a real availability failure, and I could not prove that every gateway-initiated LaunchAgent restart path avoids the SIGTERM-as-shutdown/KeepAlive race described here.

## What This Changes

The issue reports a macOS LaunchAgent KeepAlive race where a gateway restart can leave the managed gateway stopped instead of restarted.

## Best Possible Solution

Keep the issue open and make the macOS LaunchAgent restart flow use one canonical launchd-aware restart path, with focused proof that gateway-tool/config, CLI restart, and update recovery cannot leave the service unloaded or the port held by a competing process.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection does show the reported failure mode is plausible: SIGTERM without a restart intent becomes shutdown, while at least one launchctl restart helper remains separate from the guarded restart-intent/handoff paths.

## Solution Assessment

Unclear whether the reporter's proposed retry strategy is the best fix. The safer direction is to remove the split restart semantics and route gateway-initiated macOS restarts through the existing launchd handoff or restart-intent contract, then prove it with a real LaunchAgent scenario.
