## Summary

## What This Changes

The PR makes Android `GatewaySession` track pending RPC ids per `Connection`, cancel that connection's waiters from `closeQuietly()`, and adds a regression test for disconnecting while a slow RPC is pending.

## Best Possible Solution

Land the per-connection pending-RPC cleanup after maintainer review and redacted real Android/gateway proof shows a pending RPC resolves promptly on disconnect/reconnect without canceling a newer connection's waiters.

## Reproduction Assessment

Yes, from source inspection rather than an executed current-main run: current main keeps pending RPC waiters in a session-wide map and `closeQuietly()` does not cancel them, while the PR's test encodes the slow-RPC disconnect path.

## Solution Assessment

Yes. Per-connection pending-id ownership is the maintainable fix because simply calling the existing session-wide `failPending()` from `closeQuietly()` could let an old socket close cancel waiters owned by a replacement connection.
