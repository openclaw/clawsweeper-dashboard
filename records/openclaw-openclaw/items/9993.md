## Summary

Keep open. Current main has stronger built-in config validation, SecretRef preflight, base-hash guarded writes, and agent-tool guardrails, but it still lacks a mandatory synchronous plugin/API-level pre-persistence veto; the recent maintainer/RFC context also says this API gap remains.

## What This Changes

Request to add a synchronous `config:pre-apply` hook that receives current and pending config and can reject `config.set`, `config.patch`, or `config.apply` before persistence.

## Best Possible Solution

Add a documented, versioned config-write interception API in the shared mutation path after pending config computation and built-in validation/preflight, before persistence, with tests proving rejection leaves disk and runtime config unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the plugin hook catalog lacks a config mutation event, and the config write paths validate/preflight then persist through shared helpers without a rejectable plugin interception point.

## Solution Assessment

Unclear pending maintainer design review. A shared pre-persistence config-write hook is the right class of solution, but the exact public API and security semantics should be coordinated with the config validation/mutation RFC.
