## Summary

Keep open: current main has config schema validation, SecretRef preflight, base-hash guarded writes, and agent-facing config guardrails, but it still lacks the requested mandatory synchronous plugin/API veto before config persistence. The issue is a feature/API gap that needs product and security semantics, not stale cleanup.

## What This Changes

The issue requests a synchronous config pre-apply hook that can validate and veto gateway config mutations before they are written.

## Best Possible Solution

Add a documented shared config-write interception contract after pending config computation and built-in validation/preflight but before persistence, with typed rejection and regression tests proving rejected writes leave disk and active runtime config unchanged.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API gap. Source inspection gives high confidence that current main still lacks a mandatory plugin-level config pre-apply veto.

## Solution Assessment

Unclear until maintainers settle API and security semantics. A shared pre-persistence config mutation contract is the right shape; a one-off hook around only config.patch/config.apply would be too narrow.
