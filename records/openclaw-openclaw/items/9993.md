## Summary

Keep open: current main has schema validation, SecretRef preflight, base-hash guarded writes, and no-op handling, but it still lacks the requested mandatory synchronous plugin/API veto before config persistence. The linked config-guard project is useful external validation context, yet the reported gap is that built-in config writes can bypass it unless OpenClaw exposes a pre-apply interception contract.

## What This Changes

Request a synchronous `config:pre-apply` hook that can inspect current and pending config and reject config writes before they persist.

## Best Possible Solution

Define a documented shared pre-persistence config mutation contract after pending config computation and built-in validation/preflight, covering `config.set`, `config.patch`, `config.apply`, and shared mutation helpers with typed rejection and regression tests proving rejected writes leave disk and runtime config unchanged.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new plugin/API interception request. Source inspection of the hook catalog and config write paths gives high confidence the requested mandatory pre-persistence veto is absent on current main.

## Solution Assessment

Unclear until maintainers settle the API and security semantics. A shared pre-persistence config mutation contract is the right shape; a hook wrapped only around `config.patch` and `config.apply` would be too narrow.
