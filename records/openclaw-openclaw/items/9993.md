## Summary

Keep open: current main has config schema validation, SecretRef preflight, base-hash guarded writes, no-op handling, and an agent-tool allowlist, but it still lacks the requested mandatory synchronous plugin/API veto before config persistence. The recent issue discussion also explicitly kept this API gap open, so this is feature/security design work rather than conservative cleanup.

## What This Changes

The issue requests a synchronous config pre-apply hook so plugins or skills can validate and reject pending config mutations before they are persisted.

## Best Possible Solution

Add a documented shared config-write interception contract after pending config computation and built-in validation/preflight but before persistence, covering config.set, config.patch, config.apply, and shared mutation helpers with typed rejection and regression tests proving rejected writes leave disk and runtime config unchanged.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API gap. Source inspection of the hook catalog and config write paths gives high confidence that current main still lacks a mandatory plugin-level config pre-apply veto.

## Solution Assessment

Unclear until maintainers settle the API and security semantics. A shared pre-persistence config mutation contract is the right shape; a one-off hook around only config.patch/config.apply would be too narrow.
