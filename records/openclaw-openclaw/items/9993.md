## Summary

Keep open: current main and v2026.6.5 have adjacent config validation, SecretRef preflight, base-hash guards, destructive-array checks, and protected config tool controls, but they still do not expose the requested synchronous plugin/API-level veto before config writes persist. The linked config-guard workaround is useful external validation, but the reported bypass remains because Gateway config mutations can still proceed without a mandatory pre-persistence plugin hook.

## What This Changes

The issue requests a synchronous config pre-apply hook that can validate and reject Gateway config mutations before they are persisted.

## Best Possible Solution

Define one shared documented pre-persistence config mutation contract invoked by config.set, config.patch, config.apply, and shared config mutation helpers after pending config computation plus built-in validation/preflight, with typed reject results and tests proving rejected writes leave disk and runtime config unchanged.

## Reproduction Assessment

Yes, at source level: current main lacks a config pre-apply hook in the plugin hook catalog, and config.set/config.patch/config.apply commit through shared helpers without a plugin veto. I did not run a live crash reproduction because the missing API is directly visible from source, docs, and tests.

## Solution Assessment

Unclear until maintainers settle the API and security semantics. A config pre-apply hook is the right class of seam, but the maintainable fix is a shared pre-persistence config mutation contract rather than wrapping only Gateway RPC methods.
