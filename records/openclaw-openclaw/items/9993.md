## Summary

Keep open. Current main and v2026.6.10 have adjacent config validation, SecretRef preflight, destructive-array guards, base-hash writes, and agent-tool config guardrails, but still do not expose the requested mandatory synchronous plugin/API-level config pre-apply veto before persistence.

## What This Changes

The issue requests a synchronous config pre-apply hook that can inspect and reject pending config.set/config.patch/config.apply changes before they are written.

## Best Possible Solution

Define one maintainer-approved synchronous config-write interception contract at the shared mutation boundary, with typed reject results, security semantics, docs, and tests proving rejected writes leave disk and runtime state unchanged.

## Reproduction Assessment

Yes, at source level: current main exposes config.set/config.patch/config.apply and shared config mutation helpers with built-in validation/preflight, but no mandatory plugin/API-level pre-persistence veto. I did not rerun the historical provider-model incident because the remaining issue is the missing API contract.

## Solution Assessment

Yes in direction, but the exact API needs maintainer design. A shared write-boundary hook is more maintainable than relying on the config-guard skill or wrapping only Gateway RPC handlers.
