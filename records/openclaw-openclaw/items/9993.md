## Summary

Keep open: current main still lacks a mandatory synchronous plugin/API veto before config writes persist, and the issue discussion treats this as a security-sensitive API design gap rather than completed cleanup.

## What This Changes

Request a `config:pre-apply` hook that can synchronously validate and reject pending `config.set`, `config.patch`, and `config.apply` changes before they are written.

## Best Possible Solution

Define one documented shared pre-persistence config mutation contract that covers all config writers with typed rejection and tests proving rejected writes leave disk and runtime config unchanged.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new plugin/API interception request. Source inspection gives high confidence that current main has no mandatory pre-persistence veto; config.patch/config.apply validate and preflight, then write through commitGatewayConfigWrite.

## Solution Assessment

Unclear until maintainers settle API and security semantics. A shared pre-persistence config mutation contract is the maintainable shape; wrapping only config.patch and config.apply would be too narrow.
