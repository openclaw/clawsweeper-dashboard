## Summary

Keep this open. Current main has built-in config validation, SecretRef preflight, base-hash guarded writes, and an agent-tool config write allowlist, but it still lacks the requested synchronous plugin/API-level pre-persistence veto; the discussion also ties this to the config validation/mutation RFC direction.

## What This Changes

Request to add a synchronous `config:pre-apply` hook that receives current and pending config and can reject `config.set`, `config.patch`, and `config.apply` before disk writes.

## Best Possible Solution

Add a documented, versioned pre-persistence config write interception API in the shared mutation path after pending config computation and built-in validation/preflight, with rejection tests proving disk and runtime config stay unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows the hook catalogs have no config mutation event, and the config write paths validate/preflight pending config before persisting without a rejectable plugin interception point.

## Solution Assessment

Unclear pending maintainer design review. A shared pre-persistence config write hook is the right class of solution, but the exact API contract should be coordinated with the config validation/mutation RFC because it crosses public plugin and security boundaries.

## Risks / Open Questions

- The hook contract still needs ordering decisions relative to schema validation, SecretRef preflight, base-hash checks, post-write listeners, and runtime reload.
- Fail-open/fail-closed behavior, recursion from hook handlers, and plugin trust boundaries could create availability or security regressions if designed narrowly around only one writer.
- The related config validation/mutation RFC should settle actor metadata, changed-path shape, and which shared mutation helpers must be covered before implementation.
