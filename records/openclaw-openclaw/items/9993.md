## Summary

Keep open. Current main has adjacent config validation, SecretRef preflight, base-hash guarded writes, destructive-array guards, and admin-scoped config RPCs, but it still does not provide the requested mandatory synchronous plugin/API-level pre-apply veto before config persistence.

## What This Changes

Request a synchronous config pre-apply hook that lets plugins or policy validate and reject pending config writes before they are persisted.

## Best Possible Solution

Define one maintainer-approved synchronous config-write interception contract at the shared config mutation boundary, with typed reject results, security semantics, docs, and tests proving rejected writes leave disk and runtime state unchanged.

## Reproduction Assessment

Yes at source level: current main exposes config mutation RPCs and shared config mutation helpers with built-in validation/preflight, but no mandatory plugin/API-level pre-persistence veto. I did not rerun the historical provider-model incident because the remaining issue is the missing API contract.

## Solution Assessment

Yes in direction, but the exact API needs maintainer design. A shared write-boundary hook is more maintainable than wrapping only agent tools or Gateway RPC handlers, while existing SecretRef preflight and agent allowlists solve only adjacent cases.
