## Summary

Keep open. Current main and the latest release have adjacent config validation, SecretRef preflight, guarded writes, and an agent-tool allowlist, but they still do not expose the requested mandatory plugin/API-level pre-persistence config veto.

## What This Changes

Request a synchronous `config:pre-apply` hook that can inspect current and pending config and reject `config.patch`/`config.apply` before persistence.

## Best Possible Solution

Add one typed synchronous interception contract at the shared config-write owner boundary after pending-config computation and built-in validation/preflight but before persistence, with docs and regression tests proving rejection leaves disk and active runtime state unchanged.

## Reproduction Assessment

Yes at source level: current main exposes config mutation RPCs and shared mutation helpers with built-in validation and SecretRef preflight, but no mandatory plugin/API-level pre-persistence veto. The historical provider-model incident was not rerun because the remaining gap is the missing API contract.

## Solution Assessment

Yes in direction, but the exact API needs maintainer design. A shared owner-boundary hook is more maintainable than wrapping only agent tools or gateway RPC handlers, while existing startup preflight and SecretRef rollback work solve adjacent lifecycle problems.
