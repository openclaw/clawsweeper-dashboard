## Summary

Keep open. Current main and v2026.6.6 have adjacent config validation, SecretRef preflight, base-hash guarded writes, and an agent Gateway-tool allowlist, but they still do not expose the requested mandatory synchronous plugin/API-level config pre-apply veto before persistence.

## What This Changes

Request a synchronous config pre-apply plugin hook that receives current and pending config for config.patch/config.apply and can reject the write before it is persisted.

## Best Possible Solution

Define one maintainer-approved synchronous config-write interception contract at the shared config mutation boundary, with typed reject results, security semantics, docs, and regression tests proving rejected writes leave disk and runtime state unchanged.

## Reproduction Assessment

Yes at source level: current main exposes config mutation RPCs and shared config mutation helpers with built-in validation/preflight, but no mandatory plugin/API-level pre-persistence veto. I did not rerun the historical provider-model incident because the remaining issue is the missing API contract.

## Solution Assessment

Yes in direction, but the exact API needs maintainer design. A shared write-boundary hook is more maintainable than wrapping only agent tools or Gateway RPC handlers, while existing SecretRef preflight and agent allowlists solve only adjacent cases.
