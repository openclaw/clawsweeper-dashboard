## Summary

Keep this open. Current main has stronger built-in config validation, SecretRef preflight, base-hash guarded writes, and agent-tool config write allowlisting, but it still lacks the requested synchronous plugin/API-level pre-persistence veto; recent issue discussion also keeps this API gap tied to the config validation/mutation RFC direction.

## What This Changes

Request a synchronous `config:pre-apply` hook that receives current and pending config and can reject `config.set`, `config.patch`, and `config.apply` before persistence.

## Best Possible Solution

Add a documented synchronous config-write interception point in the shared mutation path after pending config computation and built-in validation/preflight, but before persistence, with regression coverage proving rejection leaves disk and runtime config unchanged.

## Reproduction Assessment

Yes. A source-level reproduction is available on current main: the hook catalog has no config write hook, and the config write paths validate/preflight pending config before calling persistence without a rejectable plugin interception point.

## Solution Assessment

Unclear pending maintainer design review. A shared pre-persistence config write hook is the right class of solution, but the exact API contract should be coordinated with the config validation/mutation RFC because this crosses plugin and security boundaries.
