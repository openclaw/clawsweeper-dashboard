- `impact:auth-provider`: The reported failure involved selecting a nonexistent provider model through a Claude CLI OAuth-backed configuration.

## Summary

Keep this issue open. Current main validates pending configuration, resolves SecretRefs, guards writes with base hashes, and detects destructive array replacements, but plugins still have no mandatory pre-persistence veto covering all configuration writers; the related preflight issues address startup validation rather than this mutation-time API gap.

## What This Changes

Requests a synchronous plugin-level pre-apply hook that can inspect and veto gateway configuration mutations before persistence.

## Best Possible Solution

Define one documented public plugin contract at the canonical configuration-write boundary, after pending-config validation and SecretRef preflight but before persistence, with typed rejection, actor/action metadata, changed paths, bounded execution semantics, and tests proving every supported mutation entry point leaves disk and runtime state unchanged when rejected.

## Reproduction Assessment

Yes at the source level. Current main's config.set, config.patch, config.apply, shared mutation helpers, and public hook catalog can be traced directly and show no mandatory plugin-level pre-persistence veto; this read-only review did not execute a live destructive mutation.

## Solution Assessment

Yes in direction, but the example event alone is not sufficient. The maintainable solution is a public contract at the canonical shared write boundary rather than an RPC-only hook or a second validation implementation that can drift.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because a plugin veto would become an authorization and availability boundary around persisted configuration.

Concerns:

- **[medium] Specify trusted veto authority and failure policy:**
  - body: The API design must define which installed plugins may reject writes, whether errors and timeouts fail closed, and how untrusted or faulty plugins are prevented from permanently denying configuration recovery.
  - confidence: 0.94
- **[medium] Prevent recursive configuration mutation:**
  - body: A pre-apply handler that can call configuration mutation APIs could recurse, deadlock, or bypass the original decision unless reentrancy and nested-write behavior are explicitly constrained.
  - confidence: 0.9
