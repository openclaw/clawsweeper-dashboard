## Summary

Keep open: current main and v2026.6.10 have adjacent validation, SecretRef preflight, base-hash guarded writes, and agent-tool guards, but still lack the requested mandatory synchronous plugin/API-level veto before config writes are persisted.

## What This Changes

Add a synchronous `config:pre-apply` hook/API so validators can inspect and reject pending config mutations before `config.patch` or `config.apply` writes are committed.

## Best Possible Solution

Define one maintainer-approved synchronous config-write interception contract at the shared mutation boundary, with typed reject results, security semantics, docs, and tests proving rejected writes leave disk and runtime state unchanged.

## Reproduction Assessment

Yes, at source level: current main exposes `config.set`, `config.patch`, `config.apply`, and shared config mutation helpers with validation/preflight, but no mandatory plugin/API-level pre-persistence veto. I did not rerun the historical provider-model incident because the remaining issue is the missing API contract.

## Solution Assessment

Yes in direction, but the exact API needs maintainer design. A shared write-boundary hook is more maintainable than wrapping only agent tools or Gateway RPC handlers, while existing SecretRef preflight and agent allowlists solve only adjacent cases.
