## Summary

Keep open. Current main has built-in config validation, SecretRef preflight, base-hash guarded writes, and an agent gateway-tool guard, but it still lacks the requested synchronous public plugin/API veto before config persistence; maintainer discussion also ties this to the config validation/mutation RFC direction.

## What This Changes

The issue requests a synchronous `config:pre-apply` hook that receives current and pending config and can reject `config.set`, `config.patch`, and `config.apply` before disk writes.

## Best Possible Solution

Add a documented, versioned config-write interception API in the shared mutation path after pending config computation and built-in validation/preflight, before persistence, with regression coverage proving rejection leaves disk and runtime config unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the public hook catalogs have no config mutation event, and config write paths validate/preflight pending config before persisting without a rejectable plugin interception point.

## Solution Assessment

Unclear pending maintainer design review. A shared pre-persistence config write hook is the right class of solution, but the exact API contract should be coordinated with the config validation/mutation RFC because it crosses public plugin and security boundaries.

## Risks / Open Questions

- The hook ordering contract is security-sensitive: maintainers need to decide whether custom validation runs before or after built-in schema validation and SecretRef preflight.
- A config write hook must define fail-closed/fail-open behavior, timeout handling, recursion avoidance, and what actor/client metadata plugins can see.
- The implementation must cover shared mutation helpers as well as the RPC methods, otherwise non-RPC config writers could still bypass the veto.
