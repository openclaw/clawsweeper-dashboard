## Summary

Keep this open. Current main has built-in config validation, SecretRef preflight, base-hash guarded writes, and agent-tool config write allowlisting, but it still lacks the requested synchronous plugin/API-level pre-persistence veto; recent maintainer discussion also keeps this gap tied to the config validation/mutation RFC direction.

## What This Changes

Request a synchronous `config:pre-apply` hook that can inspect and reject pending gateway config changes before `config.patch` or `config.apply` persists them.

## Best Possible Solution

Add a documented, versioned config-write interception API in the shared mutation path after pending config computation and built-in validation/preflight, before persistence, with regression coverage proving rejection leaves disk and runtime config unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available on current main: the hook catalogs have no config mutation event, and config write paths validate/preflight pending config before persisting without a rejectable plugin interception point.

## Solution Assessment

Unclear pending maintainer design review. A shared pre-persistence config write hook is the right class of solution, but the exact API contract should be coordinated with the config validation/mutation RFC because it crosses public plugin and security boundaries.
