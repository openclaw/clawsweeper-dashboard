## Summary

Keep open: current main has stronger built-in config validation and agent-tool guardrails, but it still lacks the requested mandatory synchronous plugin/API veto before config writes are persisted. Recent maintainer discussion also explicitly kept this API gap open and tied it to the config validation/mutation design direction.

## What This Changes

Request a synchronous config pre-apply plugin/API hook that can validate and reject gateway config changes before persistence.

## Best Possible Solution

Define a documented shared pre-persistence config mutation contract after pending config computation and built-in validation/preflight, covering config.set, config.patch, config.apply, and shared config mutation helpers with typed rejection and regression tests proving rejected writes leave disk and active runtime config unchanged.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new plugin/API interception request. Source inspection of the hook catalog and config write paths gives high confidence the requested mandatory pre-persistence veto is absent on current main.

## Solution Assessment

Unclear until maintainers settle the API and security semantics. A shared pre-persistence config mutation contract is the right shape; wrapping only config.patch/config.apply would be too narrow because shared config mutation helpers remain bypass paths.
