## Summary

Keep open: current main has built-in config validation, SecretRef preflight, base-hash guarded writes, and model-facing config-write guards, but it still lacks the requested mandatory synchronous plugin/API-level veto before config mutations are persisted. Recent review comments and related config-mutation discussions point to an intentional API/security design item rather than an already-implemented request.

## What This Changes

The issue requests a `config:pre-apply` hook that can validate and reject pending config changes before `config.patch` or `config.apply` writes them.

## Best Possible Solution

Add a documented additive config-write interception contract in the shared mutation flow after pending config computation and built-in validation/preflight but before persistence, with typed reject semantics and regression coverage proving rejected writes leave disk and active runtime config unchanged.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API gap. Source inspection gives high confidence that current main still lacks a mandatory plugin-level config pre-apply veto despite existing validation and model-facing guardrails.

## Solution Assessment

Unclear until the RFC/security direction is settled. The maintainable path is a shared config mutation seam covering RPC and runtime/plugin write helpers, not a hook attached only to one gateway method.
