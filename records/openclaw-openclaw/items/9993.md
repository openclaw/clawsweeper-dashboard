## Summary

Current main has strong built-in config validation, SecretRef preflight, base-hash concurrency guards, and atomic writes, but it still lacks a mandatory plugin/API veto between pending-config validation and persistence. The linked startup-preflight proposals are adjacent rather than canonical replacements, and the recent issue discussion explicitly preserves this plugin API gap for maintainer design.

## What This Changes

Requests a synchronous plugin-level veto hook before gateway config mutations are persisted.

## Best Possible Solution

Add one typed synchronous interception contract at the shared config-write owner boundary after pending-config computation and built-in validation but before persistence, covering all canonical writers and proving rejection leaves both disk and active runtime state unchanged.

## Reproduction Assessment

Yes at source level: current main exposes config mutation RPCs and shared persistence helpers with built-in validation, but no mandatory plugin-level pre-persistence veto, so an external validator remains bypassable. The historical incident itself was not rerun because the missing API contract is directly verifiable from current source.

## Solution Assessment

Yes in direction, but the exact API requires maintainer design. A shared owner-boundary hook is narrower and more reliable than wrapping only the agent tool or gateway RPC methods, while startup-preflight proposals solve a different lifecycle problem.
