## Summary

Keep open: current main has config validation, SecretRef preflight, base-hash guarded writes, and agent-facing config write guardrails, but it still lacks the requested mandatory synchronous plugin/API veto before config persistence.

## What This Changes

Request a synchronous `config:pre-apply` hook that lets plugins validate and reject pending `config.patch`/`config.apply` writes before persistence.

## Best Possible Solution

Add a documented shared config-write interception contract after pending config computation and built-in validation/preflight but before persistence, with typed rejection and regression tests proving rejected writes leave disk and active runtime config unchanged.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API gap. Source inspection of current main gives high confidence that there is still no mandatory plugin-level config pre-apply veto before persistence.

## Solution Assessment

Unclear until maintainers settle API and security semantics. A pre-persistence interception point is the right shape, but it should be a shared config mutation contract rather than a hook attached only to one gateway method.

## Security Review

Status: needs_attention

Summary: Needs attention: the requested hook sits on the config mutation trust boundary and needs fail-closed, timeout, recursion, and coverage semantics before implementation.
