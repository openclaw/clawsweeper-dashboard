## Summary

Keep open: current main has stronger built-in config validation, SecretRef preflight, base-hash guarded writes, and an agent-facing config mutation allowlist, but it still lacks the requested synchronous plugin/API-level pre-persistence veto for config writes. A maintainer comment in the issue also explicitly kept this open and tied it to the config validation/mutation RFC direction.

## What This Changes

Request a synchronous `config:pre-apply` hook that lets external validators inspect and reject `config.set`, `config.patch`, and `config.apply` changes before they are persisted.

## Best Possible Solution

Add a documented, additive config-write interception contract that runs after pending config computation and built-in validation/preflight but before persistence, covers RPC and shared mutation helpers, and proves rejection leaves disk and runtime config unchanged.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/API gap. Source inspection gives high confidence that current main has no mandatory plugin-level config pre-apply veto despite existing validation and tool-level guardrails.

## Solution Assessment

Unclear until the RFC direction is settled. A narrow hook added only to `config.patch` would not be the best solution; the maintainable path needs a shared config mutation seam covering `config.set`, `config.patch`, `config.apply`, and plugin/runtime config mutation helpers.
