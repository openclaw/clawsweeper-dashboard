## Summary

Keep open: current main has adjacent config validation, SecretRef preflight, base-hash guarded writes, and agent-tool guardrails, but it still lacks the requested mandatory synchronous plugin/API veto before config persistence. The linked config-guard skill is useful context, but it cannot reliably protect built-in config writes without a core plugin/API interception point.

## What This Changes

The issue requests a synchronous config pre-apply hook that can validate and reject pending config mutations before they are persisted.

## Best Possible Solution

Define a documented shared pre-persistence config mutation contract after pending config computation and built-in validation/preflight, covering config.set, config.patch, config.apply, and shared mutation helpers with typed rejection and regression tests proving rejected writes leave disk and runtime config unchanged.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new plugin/API interception request. Source inspection gives high confidence the requested mandatory pre-persistence veto is absent on current main.

## Solution Assessment

Unclear until maintainers settle the API and security semantics. A shared pre-persistence config mutation contract is the right shape; wrapping only config.patch and config.apply would be too narrow.

Status: needs_attention
