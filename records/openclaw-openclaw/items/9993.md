## Summary

Keep open: current main and v2026.6.5 have adjacent config validation, SecretRef preflight, base-hash guards, destructive-array checks, and agent-tool allowlist hardening, but still no synchronous plugin/API-level config pre-apply veto before persistence.

## What This Changes

The issue requests a synchronous config pre-apply hook that can validate and veto gateway config mutations before they are persisted.

## Best Possible Solution

Keep this open and design one shared documented pre-persistence config mutation contract invoked by config.set, config.patch, config.apply, and shared mutation helpers after pending config computation plus built-in validation/preflight, with typed reject results and tests proving rejected writes leave disk and runtime config unchanged.

## Reproduction Assessment

Yes, at source level: current main lacks a config pre-apply hook in the plugin hook catalog, and config.set/config.patch/config.apply commit through shared helpers without a plugin veto. I did not run a live crash reproduction because this is a read-only issue review and the missing API is directly visible from source and docs.

## Solution Assessment

Unclear until maintainers settle the API and security semantics. A config pre-apply hook is the right class of seam, but the maintainable fix is a shared pre-persistence config mutation contract rather than wrapping only Gateway RPC methods.
