## Summary

Keep this open. Current main has built-in config validation, active SecretRef preflight, base-hash guarded writes, and atomic mutation helpers, but it still does not provide the requested mandatory plugin/API-level pre-persistence config veto; the merged SecretRef work and related preflight issues are adjacent rather than replacements.

## What This Changes

Request a synchronous config pre-apply hook/API that lets a validator inspect current and pending config and reject config.set, config.patch, config.apply, or shared config mutations before persistence.

## Best Possible Solution

Add one typed synchronous interception contract at the shared config-write owner boundary after pending-config computation and built-in validation/preflight but before persistence, with docs and regression tests proving rejection leaves disk and active runtime state unchanged.

## Reproduction Assessment

Yes at source level: current main exposes config mutation RPCs and shared config mutation helpers with built-in validation and SecretRef preflight, but no mandatory plugin-level pre-persistence veto. The historical provider-model incident was not rerun because the remaining gap is the missing API contract, which is directly visible in source.

## Solution Assessment

Yes in direction, but the exact API still needs maintainer design. A shared owner-boundary hook is narrower and more reliable than wrapping only agent tools or gateway RPC handlers, while startup-preflight and rollback issues solve different lifecycle problems.
