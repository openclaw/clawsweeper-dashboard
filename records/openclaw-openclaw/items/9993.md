## Summary

Keep this issue open. Current main has stronger built-in config validation, SecretRef preflight, base-hash guarded writes, and an agent-tool allowlist, but it still does not expose the requested synchronous plugin/API-level pre-apply veto before config mutations are persisted; the recent maintainer discussion also explicitly kept this API gap open for the config validation/mutation RFC direction.

## What This Changes

Request a synchronous `config:pre-apply` hook that receives current and pending config for `config.patch`/`config.apply` writes and can reject the change before it is written.

## Best Possible Solution

Add a documented synchronous config-write interception point in the shared config mutation path after pending config computation and built-in validation/preflight, but before persistence. It should cover `config.set`, `config.patch`, `config.apply`, and plugin runtime config writes, expose typed current/pending config context plus actor/base-hash/change metadata, and prove rejection leaves disk and runtime config unchanged.

## Reproduction Assessment

Yes. A source-level reproduction is available on current main: inspecting the hook catalog and config write path shows no rejectable plugin hook between validated pending config and `replaceConfigFile` persistence.

## Solution Assessment

Unclear pending maintainer design review. A pre-persistence shared config-write hook is the right class of solution, but the exact API contract should be coordinated with the config validation/mutation RFC because this is a security and plugin-platform boundary.
