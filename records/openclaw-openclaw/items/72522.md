## Summary

Keep open. The latest PR head addresses the transient reconnect flash directionally, but it still misses a nonrecoverable `AUTH_SCOPE_MISMATCH` reset path and has only unit/mocked proof, so it should not be merged or cleaned up yet.

## What This Changes

The PR adds sticky Control UI `hasEverConnected` state across gateway, view state, renderer, tests, and changelog so transient reconnects keep the chat dashboard mounted while selected auth/revocation closes return to the credentials gate.

## Best Possible Solution

Keep the sticky prior-success design, add `AUTH_SCOPE_MISMATCH` to the reset path with regression coverage, and require redacted real Control UI proof showing transient reconnect keeps chat mounted while auth/scope failure returns to the gate.

## Reproduction Assessment

Yes, source-reproducible. Current main renders the credentials gate whenever `connected` is false and clears `connected` on active gateway close; the linked issue gives concrete tab-refocus steps, though I did not run a browser session in this read-only review.

## Solution Assessment

No, not as-is. The sticky prior-hello state is the right narrow shape, but the patch should also reset on `AUTH_SCOPE_MISMATCH` and provide real after-fix Control UI proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reset sticky state on scope mismatch:** `ui/src/ui/app-gateway.ts:81-104`
  - body: The new allowlist drops `hasEverConnected` only for the listed detail codes, but `AUTH_SCOPE_MISMATCH` is a nonrecoverable Control UI auth failure and the browser client stops reconnecting after it. After a tab has connected once, a reconnect failure with this detail code leaves stale dashboard content mounted instead of returning to the credentials gate; add this code and cover it in `app-gateway.node.test.ts`.
  - confidence: 0.9
