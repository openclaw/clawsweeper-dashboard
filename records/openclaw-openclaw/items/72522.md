## Summary

Keep open. The PR fixes the visible transient reconnect flash directionally, but the inspected head still misses explicit auth/device revocation closes that arrive as WebSocket code/reason frames without a structured error, leaving the sticky Control UI state set after revocation.

## What This Changes

The branch adds a sticky Control UI `hasEverConnected` state across the gateway host, Lit/view state, render gate, focused tests, and changelog so transient reconnects keep webchat mounted while structured auth failures restore the credentials gate.

## Best Possible Solution

Keep the sticky prior-hello state for transient reconnect UX, but clear it immediately on explicit auth/session/device revocation closes, either by mapping known `4001`/pairing close reasons in the UI client path or by propagating structured close details from the gateway, with no-error close tests.

## Reproduction Assessment

Yes. On the PR head, after a successful hello, invoke the close path with `{ code: 4001, reason: "gateway auth changed", error: undefined }`; `lastErrorCode` stays null, `hasEverConnected` remains true, and the render gate suppresses the credentials view.

## Solution Assessment

No, not as written. The sticky flag is the right narrow shape for #72500, but the implementation must also treat current server auth/device revocation close frames as re-auth failures before it is safe to merge.
