## Summary

Keep open. This is a focused fix for a valid current-main bug, but the branch is not merge-ready because the sticky reset misses a nonrecoverable auth failure, real behavior proof is still mock-only, and live GitHub reports the PR as conflicting.

## What This Changes

The PR adds sticky Control UI `hasEverConnected` state across the app, view state, gateway host, renderer, tests, and changelog so transient disconnects keep chat mounted while selected auth failures restore the credentials gate.

## Best Possible Solution

Preserve the sticky prior-hello approach, add `AUTH_SCOPE_MISMATCH` to the reset path with regression coverage, rebase the branch, and require redacted real Control UI proof before merge.

## Reproduction Assessment

Yes, at source level. Current main returns the credentials gate whenever `connected` is false, and the gateway close path clears `connected` on transient WebSocket closes before reconnect can restore the chat UI.

## Solution Assessment

No, not as-is. The sticky prior-hello state is the right narrow shape, but the PR should also reset on `AUTH_SCOPE_MISMATCH`, which current gateway code treats as nonrecoverable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reset sticky state on scope mismatch:** `ui/src/ui/app-gateway.ts:81-104`
  - body: `AUTH_SCOPE_MISMATCH` is a nonrecoverable auth failure in the browser gateway client, but it is omitted from this reset allowlist. After a tab has connected once, a later reconnect that fails with this structured detail code leaves `hasEverConnected` true, so `renderApp` keeps stale Control UI content mounted even though the client will not recover; add this detail code to the reset set and cover it in `app-gateway.node.test.ts`.
  - confidence: 0.87
