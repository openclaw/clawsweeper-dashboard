## Summary

Keep open: the PR is a plausible fix for the Control UI reconnect flash, but latest head still misses the current-main `AUTH_SCOPE_MISMATCH` terminal-auth path and lacks required real behavior proof.

## What This Changes

This PR adds a sticky per-tab `hasEverConnected` flag to Control UI gateway/render state, tests transient and auth close behavior, and updates the changelog for the webchat reconnect flash.

## Best Possible Solution

Keep the sticky prior-hello design, add `AUTH_SCOPE_MISMATCH` to the reset path with regression coverage, rebase onto current main, and require redacted real Control UI proof before merge.

## Reproduction Assessment

Yes, at source level. Current main flips `connected=false` on reconnect/close and renders the credentials gate for any disconnected state; the PR-head blocker is also source-reproducible because `AUTH_SCOPE_MISMATCH` is nonrecoverable on main but absent from the sticky reset allowlist.

## Solution Assessment

No, not as-is. The sticky prior-hello state is the narrow maintainable shape, but it must align with the current gateway auth contract and include real after-fix behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reset sticky state on scope mismatch:** `ui/src/ui/app-gateway.ts:81-104`
  - body: The new reset allowlist omits `ConnectErrorDetailCodes.AUTH_SCOPE_MISMATCH`, while current main treats that detail as nonrecoverable and stops reconnecting. After a tab has connected once, a later scope-mismatch connect failure leaves `hasEverConnected` true, so `renderApp` keeps cached Control UI content mounted instead of returning to credentials or pairing guidance; add the detail code and coverage.
  - confidence: 0.91
