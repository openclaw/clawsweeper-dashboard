## Summary

Keep open. Current main still has the disconnected-state credentials-gate flash, while the latest PR revision addresses the earlier auth-revocation security blocker and I found no remaining discrete blocking review finding.

## What This Changes

The PR adds sticky Control UI `hasEverConnected` state across gateway/view/rendering, code-only revocation-close handling, targeted tests, and a changelog entry so transient reconnects keep chat mounted.

## Best Possible Solution

Land this PR or an equivalent maintainer revision after it is mergeable and targeted UI/gateway validation passes, keeping the sticky prior-hello state while resetting it on explicit auth, pairing, and revocation failures.

## Reproduction Assessment

Yes. Current main can be reproduced by opening an authenticated Control UI session and forcing a transient WebSocket close/tab refocus; `connected=false` immediately renders the credentials gate. The PR's unit path also simulates transient `1006` closes and code-only `4001` revocation closes.

## Solution Assessment

Yes. A per-tab flag latched on successful hello and cleared on explicit structured or code-only auth revocation is narrower and less timing-sensitive than a debounce, while preserving the real re-auth gate.
