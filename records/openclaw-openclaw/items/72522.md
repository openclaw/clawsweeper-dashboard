## Summary

Keep open: the PR is a plausible fix for the linked Control UI reconnect flash, but latest head still misses a nonrecoverable scope-mismatch reset path and only provides unit/mocked proof rather than real Control UI behavior proof.

## What This Changes

The PR adds sticky Control UI `hasEverConnected` state so the chat dashboard stays mounted during transient WebSocket disconnects and returns to the credentials gate on selected auth, pairing, and revocation failures, with tests and a changelog entry.

## Best Possible Solution

Keep the sticky prior-hello design, add `AUTH_SCOPE_MISMATCH` to the reset path with regression coverage, and require redacted real Control UI proof for both transient reconnect and auth-failure gate behavior.

## Reproduction Assessment

Yes, at source level. Current main returns the login gate whenever `connected` is false, and the gateway close path clears `connected` immediately on WebSocket close before a later hello can restore the dashboard.

## Solution Assessment

No, not as-is. The sticky prior-hello state is the right narrow shape, but the PR should also reset on `AUTH_SCOPE_MISMATCH` and provide real after-fix behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reset sticky state on scope mismatch:** `ui/src/ui/app-gateway.ts:81-104`
  - body: `AUTH_SCOPE_MISMATCH` is treated by the browser gateway client as a nonrecoverable auth failure, but this allowlist omits it. After a tab has connected once, a later connect failure with that structured detail code leaves `hasEverConnected` true, so `renderApp` keeps stale Control UI content mounted instead of returning to the credentials gate; add this code and cover it in `app-gateway.node.test.ts`.
  - confidence: 0.89
