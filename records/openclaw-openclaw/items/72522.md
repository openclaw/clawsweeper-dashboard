## What This Changes

The PR adds a sticky per-tab `hasEverConnected` Control UI state, uses it to keep the dashboard mounted after transient disconnects, resets it for selected auth/revocation closes, adds gateway tests, and updates the changelog.

## Best Possible Solution

Keep the sticky prior-hello approach only after it resets for every current nonrecoverable auth/scope failure, has regression coverage, is rebased, and has redacted browser proof.

## Reproduction Assessment

Yes, at source level. Current main clears `connected` into a login-gate render path on reconnect, and PR head still keeps cached UI mounted after structured `AUTH_SCOPE_MISMATCH` because that detail is missing from the reset allowlist.

## Solution Assessment

No, not as-is. The sticky prior-hello state is a narrow maintainable direction, but the auth reset contract, branch conflicts, and real behavior proof must be fixed before merge.
