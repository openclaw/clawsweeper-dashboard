## Summary

Keep open. The PR addresses the main reconnect flash directionally, but the latest head still misses the structured `AUTH_SCOPE_MISMATCH` reset path and lacks required real Control UI proof before merge.

## What This Changes

This PR adds sticky Control UI `hasEverConnected` state across gateway, view state, rendering, tests, and changelog so transient disconnects keep chat mounted while selected auth/revocation closes return to the credentials gate.

## Best Possible Solution

Keep the sticky prior-hello design, add `AUTH_SCOPE_MISMATCH` to the reset path with regression coverage, and require redacted real Control UI proof for transient reconnect and auth-failure gate behavior.

## Reproduction Assessment

Yes, at source level. Current main sets `connected=false` on connect/close and `renderApp` maps any disconnected state to the credentials gate; the PR-head blocker is source-reproducible with an `AUTH_SCOPE_MISMATCH` pending connect error after a prior hello.

## Solution Assessment

No, not as-is. The sticky prior-hello state is the right narrow shape, but it must align with the gateway nonrecoverable auth contract and include real after-fix proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

- **[P2] Reset sticky state on scope mismatch:** `ui/src/ui/app-gateway.ts:81-104`
