## Summary

Keep open: the patch is not merge-ready because it changes existing Discord reaction notification settings into agent-turn triggers without the approved default-off reaction-wake contract, and the external PR only provides unit/mock proof rather than a live Discord run.

## What This Changes

The PR updates Discord reaction handling to add stable reaction keys and request an immediate heartbeat after a reaction system event is enqueued, with docs, changelog, and monitor tests.

## Best Possible Solution

Get an explicit maintainer decision on the reaction-wake contract, then wake only after accepted, deduplicated reaction events with focused tests, docs, and redacted live Discord proof.

## Reproduction Assessment

Yes, source-reproducible: current main's Discord reaction listener reaches enqueueSystemEvent for accepted reactions and does not call requestHeartbeat on that path. I did not run a live Discord repro because this is a read-only review.

## Solution Assessment

No: this patch wires wakes directly to existing reactionNotifications modes, while the safer path is an explicit default-off reaction-trigger contract or maintainer approval that those modes now imply wake.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate reaction wakes behind an explicit opt-in:** `extensions/discord/src/monitor/listeners.reactions.ts:514-515`
  - body: This turns every accepted `reactionNotifications` event into a heartbeat wake for existing `own`/`all`/`allowlist` configs. Current docs only promise passive system events, and https://github.com/openclaw/openclaw/issues/17840 is still open for choosing a default-off reaction-trigger contract, so merging this would silently change stored Discord configs from notification-only to agent-turn triggers. Add an approved opt-in wake contract or wait for that product decision before calling `requestHeartbeat` here.
  - confidence: 0.9
