## Best Possible Solution

Land one canonical notification wake policy path that preserves queue-by-default compatibility, makes override precedence explicit in code/docs/tests, and reconciles the Discord-only branch with the broader shared-policy branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Let routed agent overrides win:** `src/infra/notification-system-events.ts:158-162`
  - body: The new help text presents `agents.list[].notificationWake` as a per-agent override for routed notification events, but this resolver checks `channelConfig?.notificationWake` before `resolveAgentNotificationWake`. With `channels.discord.notificationWake.reactions = "wake"` and a routed agent set to `"queue"` or `"off"`, the channel value wins, so that agent cannot keep noisy reactions passive or suppressed as documented. Please either move routed-agent policy ahead of channel-level policy, or explicitly document/test that channel policy is intended to dominate agent policy.
  - confidence: 0.84
