## Summary

Keep open: the Discord reaction wake problem is real and the PR has useful proof, but the patch is not merge-ready because it changes the default wake policy for existing reaction notifications and its heartbeat request does not use the wake classification that current heartbeat preflight relies on.

## What This Changes

The PR adds stable Discord reaction keys, requests a heartbeat after accepted reaction events enqueue, and updates Discord reaction tests, docs, and changelog.

## Best Possible Solution

Route Discord reactions through the shared notificationWake.reactions policy from https://github.com/openclaw/openclaw/pull/83611, preserving queued-only default behavior while adding explicit opt-in wake coverage and the heartbeat reason shape needed by preflight.

## Reproduction Assessment

Yes for source-level reproduction: current main enqueues accepted Discord reactions as system events without a heartbeat wake, and the PR proof exercises the listener path after the patch. I did not run a live long-running Discord Gateway reproduction in this read-only review.

## Solution Assessment

No: the direct wake call is not the safest solution because it changes default behavior and does not match heartbeat preflight's wake classification. The maintainable path is the shared notificationWake.reactions policy with queue as the default and opt-in wake behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate reaction wakes behind the notification policy:** `extensions/discord/src/monitor/listeners.reactions.ts:514-520`
  - body: This unconditionally wakes the routed session whenever an accepted reaction event enqueues. Current Discord docs/main behavior only attach reactions as queued system context, and the related shared policy keeps `queue` as the default with explicit opt-in `wake`; merging this as-is can make existing reaction-enabled installations start agent turns or send replies after upgrade. Please preserve the queued default through `notificationWake.reactions` or get explicit maintainer approval for the compatibility break.
  - confidence: 0.9
- **[P2] Use a wake reason that heartbeat preflight consumes:** `extensions/discord/src/monitor/listeners.reactions.ts:515-519`
  - body: `source: "notifications-event"` with `reason: "discord-reaction"` is not classified as a wake payload by current heartbeat preflight, which only bypasses empty `HEARTBEAT.md` gates and inspects wake pending events for hook/acp/wake-style reasons. In those setups the requested heartbeat can skip before the reaction event is consumed, so reaction-only input can still wait for a later turn.
  - confidence: 0.86
