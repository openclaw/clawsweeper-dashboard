- `merge-risk: 🚨 compatibility`: Existing users with reactionNotifications enabled or default-own behavior would see reactions start immediate agent turns after upgrade.

## Summary

Keep open: the PR targets the right Discord reaction listener path and includes convincing redacted live proof, but as written it changes existing reaction notifications from passive context into immediate agent turns without the default-off opt-in/product contract tracked by https://github.com/openclaw/openclaw/issues/17840.

## What This Changes

The PR adds stable Discord reaction keys, requests a heartbeat wake after successful reaction event enqueue, and updates docs, changelog, and focused monitor tests.

## Best Possible Solution

Preserve passive reaction notifications on upgrade; if maintainers want reaction-triggered turns, add or explicitly approve a default-off wake contract and wire requestHeartbeat only after existing Discord access filters and enqueue success.

## Reproduction Assessment

Yes. Current main is source-reproducible: accepted Discord reactions reach enqueueSystemEvent without requestHeartbeat, and the compatibility risk is visible from the notification-only config/docs contract; no tests were run in this read-only review.

## Solution Assessment

No as-is. The wake-after-success placement is technically clean, but reusing reactionNotifications as an immediate-turn trigger changes upgrade behavior without the explicit opt-in/product contract tracked by https://github.com/openclaw/openclaw/issues/17840.
