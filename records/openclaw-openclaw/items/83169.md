## Summary

Keep open. The reaction-key part is useful, but the PR currently makes Discord reaction wakes unconditional and uses a heartbeat reason that current main does not classify as a wake payload; the shared-policy work in https://github.com/openclaw/openclaw/pull/83611 is the safer path to reconcile before merge.

## What This Changes

The PR adds Discord reaction_key text/context keys, calls requestHeartbeat after accepted reaction system events enqueue, and updates focused Discord tests, docs, and changelog.

## Best Possible Solution

Route Discord reaction system events through the shared notificationWake.reactions policy with queue as the default, explicit wake/off options, stable reaction keys, and heartbeat preflight coverage, then reconcile with https://github.com/openclaw/openclaw/pull/83611.

## Reproduction Assessment

Yes for source-level behavior: current main enqueues accepted Discord reactions as system events without a heartbeat request, and the PR body supplies PR-head listener proof. I did not run a live long-running Discord Gateway reproduction in this read-only review.

## Solution Assessment

No. The direct wake call addresses part of the symptom, but it changes default behavior and misses current heartbeat preflight semantics; the shared notificationWake policy is the narrower maintainable solution.

  - body: This unconditionally wakes every accepted Discord reaction after enqueue. Empty `HEARTBEAT.md` and pending-event inspection paths can still skip or run without consuming the queued reaction, so use the shared `notification-wake:*` path or extend preflight with coverage.
  - confidence: 0.88
