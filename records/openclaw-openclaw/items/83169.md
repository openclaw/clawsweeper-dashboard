## Summary

Keep open for maintainer review: the patch targets a real Discord reaction wake gap, but it changes existing reaction notification behavior from queued-only context into immediate agent wakeups without a compatibility policy.

## What This Changes

The PR adds stable Discord reaction keys, requests a heartbeat after successfully enqueued reaction events, and updates Discord docs, tests, and changelog.

## Best Possible Solution

Route Discord reaction wakes through the shared notification wake policy so queued-only remains the upgrade default, explicit wake behavior is documented and tested, and this small Discord patch can either adopt that helper or close in favor of the canonical policy PR.

## Reproduction Assessment

Yes for source-level reproduction: current main enqueues accepted Discord reaction system events without any requestHeartbeat call. I did not run a live Discord Gateway repro in this read-only review.

## Solution Assessment

No. The enqueue-success guard and reaction keys are reasonable, but unconditional wake is not the safest merge shape while the related shared notification wake policy preserves queued-only behavior by default.
