## Summary

Keep open: current main and v2026.6.8 still serialize the full Feishu message handler behind the same-key sequential queue, so active-run collect/steer followups can arrive after the session is inactive. The linked implementation candidate at https://github.com/openclaw/openclaw/pull/92962 is still open and conflicting, so this issue should remain the tracking point until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Feishu's per-chat sequential queue prevents `messages.queue.mode = "collect"` followups from reaching the shared active-run queue while the first run is still active.

## Best Possible Solution

Review, prove, and land https://github.com/openclaw/openclaw/pull/92962 or an equivalent Feishu plugin dispatch-boundary fix that preserves control and `/btw` ordering while ordinary active followups reach shared collect/steer policy.

## Reproduction Assessment

Yes. The issue is source-reproducible because current main still awaits a same-key Feishu sequential task that wraps the full `handleMessage` path, while collect only batches if the followup reaches the shared queue during an active run; I did not run a live Feishu tenant reproduction.

## Solution Assessment

Yes, the best current path is a Feishu plugin dispatch-boundary fix rather than a global queue-policy rewrite. The linked PR's running-key bypass is the active candidate, with remaining review focused on live proof and preserving control/`/btw` lanes.
