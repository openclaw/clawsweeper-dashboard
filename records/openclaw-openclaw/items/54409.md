## Summary

Keep open: current main and the latest release still route ordinary Feishu same-chat messages through the full sequential queue before they can reach the shared active-run queue, and the linked fix PR is still open and unmerged.

## What This Changes

The issue asks for Feishu same-chat follow-up messages to reach `messages.queue.mode = "collect"` while the prior run is active instead of being serialized until that run finishes.

## Best Possible Solution

Review, rebase, prove, and land https://github.com/openclaw/openclaw/pull/92962 or an equivalent Feishu plugin fix, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, source-reproducible: current main awaits the same-key Feishu sequential task around the full handler, while the shared collect/follow-up queue only triggers if the message reaches core during an active run. I did not run a live Feishu tenant repro in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow Feishu plugin dispatch-boundary change that lets ordinary active follow-ups reach the shared queue while preserving first-message ordering, dedupe, `:control`, and `/btw` lanes. The linked PR appears to target that boundary, but it must land or be replaced before this issue closes.
