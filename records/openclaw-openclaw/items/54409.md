## Label Justifications

- `impact:session-state`: The failure comes from Feishu hiding later same-session messages from active-run queue state until the original run finishes.

## Summary

Keep open. Current main still keeps ordinary Feishu same-chat messages behind the per-key sequential queue through the full `handleMessage` turn, so later collect-mode messages can reach the gateway only after the active run is no longer active; the related timeout and out-of-band command fixes do not resolve this batching path.

## What This Changes

The issue asks Feishu to let follow-up messages reach `messages.queue.mode = "collect"` batching instead of serializing each same-chat message as a separate turn.

## Best Possible Solution

Refactor the Feishu inbound scheduling boundary so normal same-chat follow-up messages can be claimed/deduped in order but reach the shared active-run follow-up queue while the original turn is still active.

## Reproduction Assessment

Yes, at source level: keep the first Feishu same-chat `handleMessage` task pending, enqueue later ordinary messages, and current main leaves them behind the sequential queue until the first task finishes. I did not run a live Feishu E2E reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the Feishu ingress scheduling boundary, not as a new global queue mode or a broad gateway rewrite.
