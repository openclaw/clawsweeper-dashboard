## Summary

Keep open: current main and v2026.6.10 still run Feishu same-key messages through the full sequential task before the shared active-run queue can see follow-ups, and the explicit candidate fix is still open and conflicting.

## What This Changes

The issue asks for Feishu ordinary same-chat follow-up messages to reach `messages.queue.mode = "collect"` while an earlier run is active instead of being serialized into separate turns.

## Best Possible Solution

Land or supersede the linked Feishu fix with a narrow change that lets ordinary active follow-ups reach the shared queue while preserving same-tick ordering, dedupe, and `:control`/`:btw` lane serialization.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: send a second ordinary Feishu message while the first same-key handler task is still awaited, and it cannot reach the shared queue until `isActive` is likely false. I did not run a live Feishu tenant repro in this read-only review.

## Solution Assessment

Yes, the best fix belongs at the Feishu receive/sequential-queue boundary because the shared queue already enqueues active follow-ups when they arrive while a run is active. The open linked PR is the current candidate, but it still needs maintainer review, proof, and conflict resolution before this issue can close.
