## Label Justifications

- `P2`: This is a documented Feishu channel behavior bug with medium user impact and a bounded plugin-owned repair surface.
- `impact:session-state`: The symptom fragments same-chat follow-up messages into separate agent turns instead of preserving them in the active session's collect queue.

## Summary

Keep open. Current main and the latest release still run ordinary Feishu same-chat messages through a full-message sequential queue before they reach the shared active-run queue policy, so collect-mode follow-ups can still arrive after the prior run is inactive and become separate turns.

## What This Changes

Fix Feishu so `messages.queue.mode = "collect"` can batch same-chat messages sent while an agent run is active instead of processing them one by one after Feishu serialization.

## Best Possible Solution

Refactor the Feishu dispatch boundary so ordinary same-chat follow-ups reach the shared active-run queue while the prior run is still active, while keeping Feishu-owned claim, dedupe, ordering, control, `/btw`, and timeout guarantees intact.

## Reproduction Assessment

Yes, source-reproducible. Feishu awaits a same-chat sequential task that wraps full `handleFeishuMessage`, while the shared queue policy can only collect follow-ups when it sees `isActive === true`.

## Solution Assessment

Yes, the best direction is a targeted Feishu dispatch-boundary refactor rather than changing the global queue policy. The fix should shorten or split the queued Feishu work so shared active-run queue semantics can operate without losing Feishu-specific ordering and dedupe protections.
