## Summary

Keep open. Current main and the latest shipped release still route ordinary same-chat Feishu messages through the sequential queue before the gateway can observe active-run state, so collect-mode follow-ups can still be seen as separate `run-now` turns; the merged Feishu timeout fix explicitly scoped this behavior out.

## What This Changes

The issue reports that Feishu's same-chat serial queue prevents `messages.queue.mode = "collect"` from batching messages sent while an agent run is active.

## Best Possible Solution

Refactor Feishu ingress scheduling so ordinary same-chat active-run follow-ups can reach the shared collect queue while preserving Feishu FIFO, dedupe, inbound debounce, out-of-band lanes, and the timeout guard.

## Reproduction Assessment

Yes, source-level: hold the first same-key Feishu `handleMessage` task pending, enqueue later ordinary messages, and current main leaves them behind the sequential queue until the first task advances, so the gateway can see `isActive=false`. I did not run a live Feishu E2E repro in this read-only pass.

## Solution Assessment

Yes on direction, but not necessarily the exact workaround in the issue body. The narrow maintainable fix belongs at the Feishu ingress/dispatch boundary so active-run follow-ups reach the existing collect queue without changing global queue policy.
