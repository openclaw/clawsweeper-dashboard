## Summary

Keep open. Current main and the latest shipped release still route ordinary Feishu same-chat messages through the sequential queue before the gateway can observe active-run state, so collect-mode follow-ups can still become separate `run-now` turns; the related Feishu timeout work explicitly did not address this behavior.

## What This Changes

The issue reports that Feishu serializes same-chat messages before they reach the gateway, so `messages.queue.mode = "collect"` cannot batch follow-up messages during an active run.

## Best Possible Solution

Refactor Feishu ingress scheduling so ordinary same-chat active-run follow-ups can reach the shared follow-up collect queue while preserving Feishu FIFO, dedupe, inbound debounce, out-of-band lanes, and the timeout guard.

## Reproduction Assessment

Yes, source-level: keep the first Feishu same-chat `handleMessage` task pending, enqueue later ordinary messages, and current main leaves them behind the sequential queue until the first task advances, so the gateway sees `isActive=false`. I did not run a live Feishu E2E repro in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-owned ingress scheduling that lets active-run follow-ups reach the existing collect queue without adding a new config mode or rewriting global queue policy.
