## Summary

Keep open: current main and the latest release still serialize the full Feishu message handler per same-chat key before the shared active-run queue can see followups, and the matching fix is still an open, unmerged PR rather than landed behavior.

## What This Changes

Fix Feishu same-chat dispatch so `messages.queue.mode = "collect"` and active-run queue modes can batch or steer followup messages instead of processing each one as a separate turn.

## Best Possible Solution

Update, review, and land or explicitly replace https://github.com/openclaw/openclaw/pull/92962; close this issue only after an equivalent fix merges.

## Reproduction Assessment

Yes. Current main still awaits a same-key Feishu sequential task that wraps the full handler, while the shared queue only collects or steers if the followup reaches it during an active run.

## Solution Assessment

Yes. The best fix is a bounded Feishu dispatch-boundary repair that preserves control and `/btw` serialization; the linked open PR is the current candidate, but it must be updated, reviewed, and landed or replaced.
