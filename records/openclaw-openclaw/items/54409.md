## Summary

Keep open. Current main and the latest release still await Feishu ordinary same-chat messages behind the sequential queue around the full agent/reply dispatch path, so later messages can miss active-run state and `messages.queue.mode = "collect"` still cannot reliably batch them.

## What This Changes

The issue asks for Feishu ordinary same-chat messages sent during an active run to reach the shared collect follow-up queue instead of being serialized until the prior run finishes.

## Best Possible Solution

Refactor Feishu ingress scheduling so same-chat active-run follow-ups reach the existing shared collect queue while preserving Feishu ordering, dedupe, inbound debounce, out-of-band lanes, and the timeout guard.

## Reproduction Assessment

Yes, source-level. Hold the first same-key Feishu `handleMessage` task pending, enqueue later ordinary same-chat text, and current main leaves the later task behind the sequential promise chain until the first task advances or times out; I did not run a live Feishu E2E repro.

## Solution Assessment

No, not the broad bypass sketch as written. The maintainable fix should split or shorten Feishu ingress work so active-run follow-ups reach the shared collect queue without weakening Feishu queue invariants.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/54409. Current Feishu receive handling serializes ordinary same-chat messages through `createSequentialQueue` until the full `handleFeishuMessage` agent/reply dispatch completes, so active-run `messages.queue.mode = "collect"` follow-ups reach the shared queue only after `isActive` is false. Add a focused regression that holds the first same-key Feishu dispatch active, sends later ordinary same-chat text with queue mode collect, and proves the later messages are admitted to the shared follow-up/collect path instead of starting separate run-now turns. Implement the narrow Feishu-owned fix by shortening or splitting the queued work so claim/dedupe/ordering invariants remain protected while the agent turn/reply dispatch no longer prevents active-run follow-ups from reaching gateway queue state. Preserve inbound debounce merging, message dedupe, `:control` and `/btw` lanes, same-chat FIFO where required, and the task-timeout guard from https://github.com/openclaw/openclaw/pull/76687. Do not change global queue semantics unless the new regression proves a shared queue bug. Coordinate with https://github.com/openclaw/openclaw/issues/67793 but do not fold the idle inbound-debounce investigation into this fix. Add a `CHANGELOG.md` Fixes entry if preparing a landing PR.
