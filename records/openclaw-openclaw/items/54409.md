## Summary

Keep open: current main and v2026.6.10 still await the full Feishu same-key sequential task before the shared active-run queue can see ordinary follow-ups, and the explicit closing PR is open, conflicting, and still needs proof.

## What This Changes

Feishu ordinary same-chat follow-ups should reach the active-run collect/steer queue instead of being held until the previous full handler turn completes.

## Best Possible Solution

A narrow Feishu receive/sequential-queue fix should let only ordinary active follow-ups reach the shared queue while preserving same-tick ordering, dedupe, and :control/:btw lane serialization.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: send a second ordinary Feishu message while the first same-key handler task is still awaited, and it cannot reach the shared active-run queue until the prior task completes. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes, the best fix belongs at the Feishu receive/sequential-queue boundary because the shared queue already enqueues active follow-ups when they arrive while a run is active. The linked PR is the current candidate, but it still needs maintainer review, proof, and conflict resolution before this issue can close.
