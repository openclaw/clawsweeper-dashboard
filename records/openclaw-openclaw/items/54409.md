## Summary

Keep open: current main still queues the full Feishu message handler behind a same-chat sequential key, so follow-up messages can miss the active-run collect/steer window. The linked fix PR is still open and unmerged, so this issue should close only after that PR or an equivalent repair lands.

## What This Changes

The issue asks for Feishu same-chat follow-up messages to reach the shared active-run queue so `messages.queue.mode = "collect"` can batch them instead of processing each as a separate turn.

## Best Possible Solution

Land or replace the focused Feishu running-key bypass after review confirms live delivery timing and control/`/btw` serialization, then close this issue as implemented.

## Reproduction Assessment

Yes, source-reproducible: current main awaits a same-key Feishu sequential task around the full `handleMessage` path, while the shared queue can only collect if the follow-up reaches it during an active run. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes, keeping this issue open behind the linked repair PR is the best current path. The narrow fix belongs in the Feishu plugin dispatch boundary rather than a new global queue mode or config surface.
