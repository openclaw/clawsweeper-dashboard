## Summary

Keep open: current main still awaits same-chat Feishu dispatch through the per-chat promise chain before the message reaches the gateway, while `collect` only batches when the gateway sees the follow-up during an active run. The merged #76687 timeout fix and the closed #43512 control-command bypass do not resolve ordinary Feishu collect-mode messages.

## What This Changes

Feishu collect mode should batch same-chat messages sent during an active run instead of processing them as separate turns after the per-chat queue drains.

## Best Possible Solution

Fix the Feishu inbound dispatch boundary so normal same-chat follow-up messages can reach the gateway while the first run is active, preserving Feishu claim/dedupe/order lanes and relying on the existing gateway collect queue to batch them.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: hold the first Feishu same-key dispatch task, enqueue later normal same-chat messages, and current main keeps them behind `previous.then(...)` until the first dispatch settles, so the gateway sees `isActive=false`.

## Solution Assessment

Yes. The narrow maintainable fix should be in the Feishu inbound dispatch boundary unless implementation proves a small generic gateway seam is needed; #76687's timeout cap and #43512's control-command bypass leave ordinary collect-mode messages serialized.

Current main serializes ordinary same-chat Feishu messages through `createSequentialQueue` until the first full `handleMessage`/agent dispatch completes, so `messages.queue.mode = "collect"` does not see later messages during an active run.
