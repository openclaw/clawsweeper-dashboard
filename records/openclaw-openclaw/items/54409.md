## Summary

Keep open. Current main still serializes normal same-chat Feishu messages through the sequential queue until the full agent dispatch settles, so collect-mode followups can miss the active-run window; the merged timeout work for #70133 only caps pathological hangs and does not supersede this focused collect-mode bug.

## What This Changes

The issue asks Feishu to let `messages.queue.mode = "collect"` batch normal same-chat follow-up messages while an earlier Feishu agent turn is still active.

## Best Possible Solution

Choose a Feishu-owned dispatch boundary that lets normal same-chat followups reach `dispatchReplyFromConfig` while the first run is active, while preserving the existing dedupe, ordering, `:control`, `:btw`, reaction, and thread routing contracts.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: hold the first normal same-chat Feishu task in `createSequentialQueue`, enqueue later normal messages, and they remain behind `previous.then(...)` until the first task settles, so the gateway sees `isActive=false` and returns `run-now`.

## Solution Assessment

Unclear. The requested direction is correct, but the latest discussion identifies multiple plausible seams for releasing the Feishu queue without breaking ordering or lane behavior, so the maintainable fix should start with maintainer agreement on that boundary.
