## Label Justifications

- `P2`: This is a bounded Feishu channel behavior bug that fragments active-run follow-ups but does not crash the runtime or lose persisted data.
- `impact:session-state`: The bug causes follow-up messages to become separate turns instead of one collected session follow-up, fragmenting conversation context.

## Summary

Keep open: current main still serializes Feishu same-chat dispatch around the full handler call, so active-run collect or steer follow-ups can miss the gateway active-run window; the linked repair PR is open and conflicting, so this is not implemented yet.

## What This Changes

The issue requests that ordinary Feishu same-chat follow-up messages reach the shared active-run queue while a prior Feishu run is still active so `messages.queue.mode = "collect"` can batch them.

## Best Possible Solution

Refresh, review, and land https://github.com/openclaw/openclaw/pull/92962 or an equivalent Feishu plugin dispatch-boundary fix, then close this issue after the fix reaches current main.

## Reproduction Assessment

Yes, this is source-reproducible: current main awaits a same-key Feishu sequential task that wraps the full `handleMessage` path, while the shared queue can only collect or steer when the follow-up reaches it during an active run. I did not run a live Feishu tenant reproduction.

## Solution Assessment

Yes, the best path appears to be a Feishu plugin dispatch-boundary fix rather than a global queue-policy change, because the shared queue behaves correctly once a follow-up arrives while `isActive` is true. The linked PR still needs refresh/review/proof before this issue can close.
