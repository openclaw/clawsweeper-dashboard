## Summary

Keep open. Current main and the latest release still run normal Feishu same-chat messages through a per-chat sequential queue before they reach the active-run followup queue, so collect-mode batching can still be missed; the related timeout work fixed starvation but explicitly did not resolve this active-run collect behavior.

## What This Changes

The issue asks Feishu to let same-chat messages sent during an active run reach collect-mode followup batching instead of being serialized into separate later turns.

## Best Possible Solution

Refactor the Feishu inbound dispatch boundary so normal same-chat follow-up messages can reach the shared active-run followup queue while the first run is still active, preserving dedupe, inbound debounce, first-message FIFO, and out-of-band lanes.

## Reproduction Assessment

Yes, source-level. Keep the first same-key Feishu `handleMessage` task pending, enqueue later normal same-chat messages, and current main leaves them behind the sequential queue until the first task advances, so the gateway no longer sees the original run as active.

## Solution Assessment

Yes. The narrow maintainable direction is a Feishu-owned dispatch-boundary fix with regression coverage; avoid a global queue-policy change unless the Feishu repair exposes a missing shared contract.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/54409. Current main serializes normal Feishu same-chat messages through `createSequentialQueue` and awaits the full `handleMessage`/agent/reply lifecycle, so later messages reach `dispatchReplyFromConfig` only after the active run is no longer active and `messages.queue.mode: "collect"` cannot batch them. Implement a Feishu-owned dispatch-boundary fix so same-chat normal follow-up messages can reach the shared active-run followup queue while the first run is active. Preserve inbound debounce merging, message dedupe, first normal-message ordering, `/stop`/abort `:control`, `/btw` `:btw`, and the timeout behavior from https://github.com/openclaw/openclaw/pull/76687. Add focused regression coverage that holds the first same-key Feishu dispatch pending, sends later normal same-chat messages with collect mode, and proves they are enqueued/collected rather than processed as separate `run-now` turns. Do not add a config option or broad gateway policy change; stop and ask for maintainer review if a shared gateway/plugin SDK contract change is required.
