## Summary

Keep open: current main and the latest release still await the full Feishu same-key handler task before the shared active-run queue can see follow-ups, and the linked closing PR is open but unmerged.

## What This Changes

The issue reports that Feishu serializes same-chat messages before they reach OpenClaw's active-run queue, so collect-mode follow-ups are processed as separate turns instead of one batched follow-up.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/92962 with a channel-boundary fix that lets ordinary active follow-ups reach the shared queue while preserving Feishu ordering, control, and /btw lanes.

## Reproduction Assessment

Yes, with high-confidence source reproduction: current main still awaits the same-key Feishu sequential task around handleMessage, while the shared queue only collects follow-ups that arrive during an active run. I did not run a live Feishu tenant repro in this read-only review.

## Solution Assessment
