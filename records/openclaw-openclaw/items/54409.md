## Summary

Keep open: current main and the latest release still run the full Feishu same-key message handler behind the per-chat sequential queue, while the shared collect/follow-up path only batches messages that arrive while the session is still active. The open linked implementation candidate has closing syntax for this issue but is not merged yet.

## What This Changes

The issue asks Feishu to let ordinary same-chat messages sent during an active run reach shared `messages.queue.mode = "collect"` or `"steer"` handling instead of becoming separate later turns.

## Best Possible Solution

Preserve Feishu ordering, dedupe, `:control`, and `/btw` lane behavior while allowing ordinary same-chat active follow-ups to reach the shared collect/steer queue path during an active run, with focused Feishu and shared queue regression proof.

## Reproduction Assessment

Yes, source-reproducible: current main still awaits the same-key Feishu sequential task around the full `handleMessage` path, while the shared queue only collects follow-ups that arrive while `isActive` is true. I did not run a live Feishu tenant repro in this read-only review.

## Solution Assessment

Yes, the issue identifies the right boundary: ordinary Feishu same-chat follow-ups need to reach the shared queue while a run is active without removing the channel ordering lanes. The open linked PR appears to be the current candidate, but it still needs mergeability and maintainer review before this issue can close.
