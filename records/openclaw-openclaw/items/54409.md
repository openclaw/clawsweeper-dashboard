## Label Justifications

- `P2`: This is a documented queue-mode behavior bug in one channel with medium user impact and a bounded repair surface.
- `impact:session-state`: Feishu hides later same-session messages from active-run queue state until the prior run advances, fragmenting the intended collected turn.

## Summary

Keep open. Current main and the latest release still route ordinary same-chat Feishu messages through the Feishu sequential queue before the shared active-run queue can observe them, so `messages.queue.mode: "collect"` can still miss mid-run followups and process them as separate turns.

## What This Changes

Feishu collect-mode followups should batch queued same-chat messages instead of being hidden behind per-chat serial dispatch.

## Best Possible Solution

Refactor Feishu ingress scheduling so ordinary same-chat active-run followups can reach the shared collect queue while preserving Feishu FIFO, dedupe, inbound debounce, out-of-band lanes, and the timeout guard.

## Reproduction Assessment

Yes, source-level: hold the first same-key Feishu `handleMessage` task pending, enqueue later ordinary same-chat text, and current main leaves the later task behind the sequential promise chain until the first task advances or times out. I did not run a live Feishu E2E repro in this read-only pass.

## Solution Assessment

Yes on direction, but not necessarily the exact bypass sketch in the issue body. The narrow maintainable fix belongs at the Feishu ingress or dispatch boundary so active-run followups reach the existing collect queue without changing global queue policy.
