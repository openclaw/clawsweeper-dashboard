## Summary

Keep open. Current main and the latest release still await Feishu's same-chat sequential queue around the full message handling and reply dispatch path, so later ordinary same-chat messages can enter the shared queue only after the prior run is no longer active and `collect` can still miss batching them.

## What This Changes

The issue reports that Feishu same-chat message serialization prevents `messages.queue.mode = "collect"` from batching mid-run follow-up messages into one later turn.

## Best Possible Solution

Refactor Feishu ingress scheduling so ordinary same-chat active-run follow-ups can reach the shared collect queue while preserving Feishu FIFO, dedupe, inbound debounce, out-of-band lanes, and the timeout guard.

## Reproduction Assessment

Yes, source-level: hold the first same-key Feishu `handleMessage` task pending, enqueue later ordinary same-chat text, and current main leaves the later task behind the sequential promise chain until the first task advances or times out. I did not run a live Feishu E2E repro in this read-only pass.

## Solution Assessment

Yes on direction, but not necessarily the broad bypass sketch in the issue body. The narrow maintainable fix should split or shorten Feishu ingress work so active-run follow-ups reach the existing collect queue without weakening the Feishu queue invariants.
