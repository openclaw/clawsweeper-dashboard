## Summary

Keep open: current main and v2026.6.10 still have the Feishu same-key queue boundary that blocks active follow-ups, and the linked fixing PR is still open and conflicting.

## What This Changes

The issue reports that Feishu same-chat serialization prevents `messages.queue.mode = "collect"` follow-up messages from reaching the shared active-run queue while a prior turn is still active.

## Best Possible Solution

Land or replace the linked Feishu follow-up delivery fix so ordinary same-chat follow-ups can reach the shared queue while preserving Feishu control and `/btw` lane ordering.

## Reproduction Assessment

Yes from source inspection: the Feishu handler still awaits the same-key sequential enqueue around the full message task, so later ordinary messages can be held until the prior run completes instead of arriving while the session is active.

## Solution Assessment

Yes, the best repair belongs at the Feishu receive/sequential-queue boundary because shared queue policy already queues active followups when they arrive during an active run; the open linked PR is the candidate but has not landed.
