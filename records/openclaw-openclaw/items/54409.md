## Summary

Keep this issue open. Current main still serializes normal same-chat Feishu messages through a promise chain before they reach the gateway reply pipeline, while the gateway `collect` behavior still depends on seeing the message during an active run; the related open reports are useful context but do not supersede this focused Feishu collect-mode bug.

## What This Changes

Feishu should allow same-chat follow-up messages to reach the gateway while an agent run is active so `messages.queue.mode = "collect"` can batch them into one follow-up turn.

## Best Possible Solution

Fix the Feishu inbound dispatch path so normal same-chat follow-up messages can reach `dispatchReplyFromConfig` while the first run is still active, letting the existing gateway `collect` queue batch them. Preserve Feishu dedupe, inbound debounce, first-message ordering, abort/control lanes, `/btw`, and comment/reaction behavior, with a focused regression test for the active-run collect scenario.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: hold the first same-key Feishu dispatch task, enqueue later normal same-chat messages, and current main keeps them behind `previous.then(task, task)` until the first task settles, so the gateway sees `isActive=false` and returns `run-now`.

## Solution Assessment

Yes. The narrow maintainable fix should stay in the Feishu plugin dispatch boundary unless evidence shows multiple channels need a shared core seam; PR #43512 is not sufficient because it targets control-command bypasses and leaves ordinary messages on the serialized path.

Current main serializes normal same-chat Feishu messages through `createSequentialQueue` before `handleFeishuMessage` reaches `dispatchReplyFromConfig`, so follow-up messages arrive after the first run is no longer active and gateway `messages.queue.mode = "collect"` treats them as separate `run-now` turns.
