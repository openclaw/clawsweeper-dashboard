## Summary

Keep open. Current main and the latest shipped release still run normal Feishu same-chat messages through the per-chat sequential queue before they reach the gateway active-run queue, so `messages.queue.mode = "collect"` can still miss batching; the merged timeout fix at https://github.com/openclaw/openclaw/pull/76687 addressed starvation for https://github.com/openclaw/openclaw/issues/70133 and was explicitly scoped away from this collect-mode behavior.

## What This Changes

The issue asks Feishu to let same-chat follow-up messages reach the gateway collect queue while an agent run is active instead of serializing each normal message through the full agent turn.

## Best Possible Solution

Refactor the Feishu inbound dispatch boundary so normal same-chat follow-up messages can reach `dispatchReplyFromConfig` while the first run is still active, preserving dedupe, inbound debounce, first-message FIFO, `:control`, and `:btw`, with focused regression coverage for active-run collect batching.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: keep the first same-key Feishu `handleMessage` task pending, enqueue later normal same-chat messages, and current main leaves them behind `previous.then(...)` until the first task or timeout advances, so the gateway no longer sees the original run as active.

## Solution Assessment

Yes. The best direction is a Feishu-owned dispatch-boundary fix with regression tests; the merged timeout fix at https://github.com/openclaw/openclaw/pull/76687 is useful adjacent work but intentionally does not solve collect-mode batching.
