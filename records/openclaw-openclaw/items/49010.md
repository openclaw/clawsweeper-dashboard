## Summary

Keep open: current main has a stdio MCP transport, but it does not implement MCPS/message-level signing. The linked standalone integration is useful context, but this would add security-sensitive transport semantics, config, key-management, and dependency choices, so it needs maintainer product/security design rather than cleanup closure.

## What This Changes

The issue requests opt-in MCPS signing for MCP stdio JSON-RPC messages, including nonce/timestamp replay protection and agent identity metadata.

## Best Possible Solution

Define the MCPS trust/config boundary first, then either add a documented transport-signing seam that plugins can use or implement a narrow opt-in core stdio mode with regression tests and MCP docs.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new security capability request. Source and tests do give high-confidence proof of current behavior: stdio MCP messages are serialized and written as plain JSON-RPC lines.

## Solution Assessment

Unclear/no. The proposed package integration may be useful, but the best solution first needs maintainer decisions on protocol compatibility, key management, config shape, and whether this belongs in core or behind a plugin/API seam.
