## Summary

Keep open. Current main still sends MCP stdio traffic as plain JSON-RPC, has no supported `mcps` signing configuration or transport-decorator seam, and the related MCPS PR was closed unmerged after security/API concerns; the linked `razashariff/mcps-openclaw` package is useful prior art, but the remaining decision belongs to maintainer security/API design rather than cleanup closure.

## What This Changes

Add an opt-in MCPS signing layer for OpenClaw's MCP stdio transport so JSON-RPC messages can carry signatures, nonce/timestamp replay protection, and agent identity while unsigned operation remains compatible.

## Best Possible Solution

Keep the issue open for a maintainer security/API decision: either add a generic opt-in MCP stdio signing/decorator seam with documented config and regression tests for replay, forgery, stripping, downgrade, key handling, and strict-mode failures, or explicitly document that MCPS remains external and expose any missing plugin API needed to wrap OpenClaw-managed frames safely.

## Reproduction Assessment

Yes. A source/test inspection reproduces the current behavior: `OpenClawStdioClientTransport.send()` writes the raw serialized JSON-RPC line, and the transport test asserts that exact unsigned frame.

## Solution Assessment

Unclear. The requested direction is plausible, but a direct dependency import plus permissive wrapper is not established as the best fix; the safer path is a maintainer-approved generic signing/decorator contract and threat model before code changes.
