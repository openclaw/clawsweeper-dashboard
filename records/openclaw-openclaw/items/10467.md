## Summary

Keep open. Current main still routes `sessions_spawn` through one fixed `subagent` queue lane, and the related open items cover adjacent per-parent/CLI lifecycle concerns rather than this caller-selectable lane feature.

## What This Changes

Add caller-selectable multi-lane concurrency for `sessions_spawn`, including a tool `lane` parameter, native/ACP forwarding, and per-lane subagent concurrency config.

## Best Possible Solution

Add an additive `sessions_spawn.lane` field that defaults to `subagent`, validates lane names, forwards through native and ACP spawns, applies compatible per-lane subagent concurrency, and updates docs, schema, changelog, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives a high-confidence verification path: no tool `lane` field, no spawn forwarding, hard-coded native/ACP subagent lane dispatch, and only scalar concurrency config.

## Solution Assessment

Yes. The existing command queue already supports named lanes, so a backward-compatible tool parameter plus scoped per-lane config is the narrowest maintainable solution while preserving current defaults.
