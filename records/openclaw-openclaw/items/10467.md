## Summary

Keep open: current main and v2026.6.8 still expose `sessions_spawn` without a caller-selected lane, and both native and ACP child runs still dispatch on the fixed `subagent` queue lane with one scalar subagent concurrency cap.

## What This Changes

Request to add caller-selected `sessions_spawn` lanes with per-lane concurrency configuration while preserving the existing `subagent` fallback.

## Best Possible Solution

Keep the current `subagent` default, then add a maintainer-approved additive lane parameter and per-lane cap semantics across schema, native/ACP dispatch, gateway lane configuration, docs, and regression tests.

## Reproduction Assessment

Yes for the source-level feature gap: current main exposes no `sessions_spawn.lane` schema field and hard-codes native and ACP dispatch to `subagent`. A live multi-lane workload was not run because the requested input surface does not exist.

## Solution Assessment

Unclear until maintainers choose the product contract. The likely best path is additive and default-preserving, but it must compose with adjacent per-agent cap and dispatch-lane proposals.
