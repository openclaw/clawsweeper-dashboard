## Summary

Keep open: current main and the latest release still expose one fixed `subagent` lane for `sessions_spawn`, while the requested caller-selected lane API and per-lane config remain an unresolved core scheduling decision.

## What This Changes

The issue requests an optional `sessions_spawn.lane` parameter plus per-lane sub-agent concurrency configuration so independent sub-agent groups do not share one global queue lane.

## Best Possible Solution

Define one bounded subagent scheduling model with a shared aggregate ceiling, then add an additive validated lane input and config shape with native/ACP parity, docs, reload behavior, and focused tests.

## Reproduction Assessment

Yes, at source level: current main exposes no `lane` argument and hard-codes both native and ACP `sessions_spawn` runs to the global `subagent` lane. This proves the missing capability without needing a live overload run.

## Solution Assessment

No, not as currently specified. Workload isolation is reasonable, but arbitrary per-call lanes need a maintainer-selected aggregate budget, validation rule, and precedence model before they are the narrowest maintainable solution.
