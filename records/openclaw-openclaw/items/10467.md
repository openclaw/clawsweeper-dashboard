## Summary

Keep open. Current main still implements sessions_spawn with one fixed subagent lane, and the April maintainer-side comment remains accurate against the checked-out source.

## What This Changes

The issue requests optional named lanes for sessions_spawn sub-agents, including per-lane concurrency configuration and fallback to the existing subagent lane.

## Best Possible Solution

Add an additive sessions_spawn.lane surface with validation, default it to subagent, forward it through native and ACP spawn paths, and add compatible per-lane subagent concurrency config plus docs/tests.

## Reproduction Assessment

Yes. The feature gap is source-reproducible: current main has no lane parameter and both native and ACP spawn paths hard-code the subagent lane; I did not run a live saturation scenario because this is a read-only review.

## Solution Assessment

Yes. An additive lane parameter backed by the existing string-based command queue is the narrowest maintainable direction, provided existing callers continue to default to subagent.
