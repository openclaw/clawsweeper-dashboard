## Summary

Keep open: current main and v2026.6.10 still expose `sessions_spawn` without a caller-selected lane, and native plus ACP sub-agent spawns still dispatch through the fixed `subagent` gateway lane with one scalar concurrency cap. The remaining work is real, but it needs maintainer product/API direction because it adds model-visible tool input and new config semantics that overlap adjacent scheduler proposals.

## What This Changes

The issue requests an optional `sessions_spawn.lane` parameter plus per-lane subagent concurrency configuration so different subagent task classes can run in independent queue lanes.

## Best Possible Solution

Add a maintainer-approved, default-preserving subagent lane contract that keeps existing callers on `subagent`, validates custom lane names, forwards the selected lane through native and ACP spawn paths, and documents/tests compatible per-lane subagent caps.

## Reproduction Assessment

Yes, as a source-level feature gap: current main exposes no `sessions_spawn.lane` schema field and hard-codes native and ACP dispatch to the `subagent` lane. I did not run a live multi-lane workload because the requested input surface does not exist.

## Solution Assessment

Unclear until maintainers choose the product/API contract. The likely direction is an additive, default-preserving lane parameter plus per-lane caps, but it must compose with adjacent subagent concurrency and queue-visibility proposals.
