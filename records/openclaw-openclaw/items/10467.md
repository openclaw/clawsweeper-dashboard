- `P2`: The confirmed limitation affects advanced multi-agent throughput and isolation, but it is not an emergency or broad core-runtime outage.
- `impact:session-state`: The request changes queue isolation and contention for spawned sub-agent sessions.

## Summary

Current main still has the exact single-lane limitation, and adjacent open items address different controls: per-agent caps and per-parent caps do not replace caller-selected task lanes. The issue should stay open for a maintainer API/config decision rather than automatic implementation.

## What This Changes

Add caller-selectable queue lanes and per-lane concurrency for sub-agents spawned through `sessions_spawn`.

## Best Possible Solution

Define a bounded lane identity model and one concurrency-precedence contract, then add an optional lane selector through both native and ACP paths while preserving `subagent` and the existing scalar cap as the default behavior.

## Reproduction Assessment

Yes at source level: current main has no lane input, both native and ACP spawns hard-code the same lane, and docs define one global sub-agent lane cap.

## Solution Assessment

No, not fully as proposed; workload isolation is useful, but arbitrary names and per-lane configuration should not be implemented until maintainers choose bounded lane ownership and precedence with per-agent and per-parent caps.
