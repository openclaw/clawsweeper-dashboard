## Summary

Keep open: current main and v2026.6.10 still expose `sessions_spawn` without a caller-selected lane, while native and ACP child runs both dispatch through the fixed `subagent` lane with one scalar subagent concurrency cap. The gap is real, but it needs maintainer product/API direction because it adds model-visible tool input and config semantics that overlap adjacent scheduler proposals.

## What This Changes

The issue requests a `sessions_spawn.lane` parameter and per-lane subagent concurrency configuration so independent subagent workloads do not all share the global `subagent` lane.

## Best Possible Solution

Add a default-preserving, maintainer-approved lane contract: validate lane names, default omitted calls to `subagent`, forward the resolved lane through native and ACP spawn paths, configure compatible per-lane subagent caps, and update schema, docs, and focused tests.

## Reproduction Assessment

Yes. Current main has no `sessions_spawn.lane` schema field and hard-codes native plus ACP dispatch to the `subagent` lane; a live multi-lane workload cannot be run because the requested input surface does not exist.

## Solution Assessment

Unclear until maintainers choose the public contract. The narrowest maintainable path appears to be an additive, default-preserving lane parameter plus compatible per-lane caps, but it must compose with adjacent scheduler work.
