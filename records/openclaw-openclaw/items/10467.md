## Summary

Keep open. Current main still exposes `sessions_spawn` with one fixed `subagent` lane and one scalar subagent concurrency limit, while adjacent open issues cover capacity visibility and per-parent throttling rather than this named-lane feature.

## What This Changes

Add an optional `sessions_spawn.lane` parameter with default `subagent` behavior and compatible per-lane subagent concurrency configuration.

## Best Possible Solution

Implement additive named lanes for `sessions_spawn` while preserving the existing default `subagent` lane and keeping capacity/status and per-parent throttling as separate follow-ups.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested lane input, forwarding, and per-lane config surface.

## Solution Assessment

Yes. An additive `lane` parameter plus compatible per-lane concurrency wiring is the narrowest maintainable path because the underlying queue already supports arbitrary lane ids and existing callers can keep defaulting to `subagent`.
