## Summary

Keep open: current main and v2026.6.6 still expose one fixed `subagent` lane for `sessions_spawn`, but the requested free-form lane API and per-lane config need maintainer direction on validation, aggregate budget, and composition with nearby concurrency controls.

## What This Changes

Request adds an optional `sessions_spawn.lane` input plus per-lane sub-agent concurrency configuration for workload isolation.

## Best Possible Solution

Define one bounded subagent scheduling model with a shared aggregate ceiling, then add an additive validated lane input and config shape with native/ACP parity, docs, reload behavior, and focused tests.

## Reproduction Assessment

Yes, at source level: current main exposes no `lane` argument and hard-codes both native and ACP `sessions_spawn` runs to the global `subagent` lane. This proves the missing capability without needing a live overload run.

## Solution Assessment

No, not as currently specified. Workload isolation is reasonable, but arbitrary per-call lanes need a maintainer-selected aggregate budget and precedence model before they are the narrowest maintainable solution.
