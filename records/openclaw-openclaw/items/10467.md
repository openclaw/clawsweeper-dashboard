## Summary

Keep open. Current main still has no `sessions_spawn.lane` input, both native and ACP spawn paths still dispatch on the fixed `subagent` lane, and the adjacent capacity, per-parent, and per-agent concurrency items do not supersede this named-lane request.

## What This Changes

Request adding an optional `sessions_spawn` lane parameter and per-lane subagent concurrency config so independent sub-agent work can use separate queue lanes.

## Best Possible Solution

Add a validated `sessions_spawn.lane` field defaulting to `subagent`, forward it through native and ACP dispatch, and support compatible per-lane subagent concurrency config with docs and regression tests.

## Reproduction Assessment

Yes. The issue is source-reproducible: current main has no `lane` input, native and ACP dispatch hard-code `subagent`, and config/docs still describe one scalar lane cap; I did not run a live saturation scenario.

## Solution Assessment

Yes. An additive lane parameter backed by the existing string-lane queue is the narrow maintainable direction, provided lane validation, default fallback, and per-lane config compatibility are handled carefully.
