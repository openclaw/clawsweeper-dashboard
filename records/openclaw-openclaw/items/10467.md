## Summary

Keep open: current main and v2026.6.8 still expose `sessions_spawn` without a caller-selected lane, while native and ACP child runs still dispatch on the fixed `subagent` queue lane with one scalar subagent concurrency cap.

## What This Changes

Request to add an optional `sessions_spawn.lane` parameter plus per-lane subagent concurrency configuration while preserving the existing `subagent` default.

## Best Possible Solution

Define a maintainer-approved additive `sessions_spawn.lane` contract that defaults to `subagent`, validates lane names, applies compatible per-lane caps, and updates native/ACP dispatch, gateway setup, docs, and tests together.

## Reproduction Assessment

Yes, at source level: current main exposes no `sessions_spawn.lane` schema field and hard-codes native and ACP child dispatch to `subagent`. I did not run a live multi-lane workload because the requested input surface does not exist.

## Solution Assessment

Unclear until maintainers choose the product contract. The likely best path is additive and default-preserving, but it must compose with adjacent per-agent cap and dispatch-lane proposals.
