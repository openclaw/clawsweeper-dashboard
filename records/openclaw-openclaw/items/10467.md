## Summary

Keep open. Current main still exposes one fixed `subagent` lane for `sessions_spawn`, while the related capacity, per-parent, and per-agent queue items cover adjacent scheduler scopes rather than superseding this named subagent-lane request.

## What This Changes

Request an additive `sessions_spawn.lane` parameter with independent named subagent queue lanes and compatible per-lane concurrency configuration.

## Best Possible Solution

Add a validated `sessions_spawn.lane` field defaulting to `subagent`, forward it through native and ACP dispatch, and support compatible per-lane subagent concurrency config with docs and regression tests.

## Reproduction Assessment

Yes. The issue is source-reproducible: current main has no `lane` input, native and ACP dispatch hard-code `subagent`, and config/docs/tests still describe one scalar lane cap; I did not run a live saturation scenario.

## Solution Assessment

Yes. An additive lane parameter backed by the existing string-lane queue is the narrow maintainable direction, provided lane validation, default fallback, and per-lane config compatibility are handled carefully.
