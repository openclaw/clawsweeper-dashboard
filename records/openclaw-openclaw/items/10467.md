## Summary

Keep open. Current main still lacks caller-selectable `sessions_spawn` lanes: the tool schema has no `lane` input, native and ACP spawns still dispatch through the fixed `subagent` lane, and config/docs/tests still model one scalar sub-agent concurrency cap.

## What This Changes

The issue requests an optional `sessions_spawn.lane` parameter plus compatible per-lane sub-agent concurrency configuration.

## Best Possible Solution

Add a validated `sessions_spawn.lane` field defaulting to `subagent`, forward it through native and ACP dispatch, and support compatible per-lane subagent concurrency config with docs and regression tests.

## Reproduction Assessment

Yes. This is source-reproducible on current main: `sessions_spawn` has no lane field, both dispatch paths hard-code `subagent`, and config/docs/tests still describe one scalar lane cap; no live saturation scenario was attempted in this read-only review.

## Solution Assessment

Yes. An additive lane parameter backed by the existing string-lane queue is the narrow maintainable direction, provided lane validation, default fallback, and per-lane config compatibility are handled carefully.
