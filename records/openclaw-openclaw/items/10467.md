## Summary

Keep open. Current main still exposes `sessions_spawn` through one fixed `subagent` queue lane: there is no caller-selected lane in the tool schema, no native or ACP lane forwarding, and only scalar subagent concurrency config. The closed per-agent-lane PR addressed inbound agent runs, not this subagent-lane feature.

## What This Changes

The issue requests an additive `sessions_spawn.lane` parameter with independent per-lane subagent queue concurrency and compatible configuration defaults.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` field that defaults to `subagent`, validates lane names, forwards through native and ACP spawn paths, applies scoped per-lane subagent concurrency, and updates schema, docs, changelog, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: schema, spawn params, dispatch calls, config schema, docs, and related tests still expose one fixed `subagent` lane.

## Solution Assessment

Yes. The existing command queue already supports named lanes, so an additive `sessions_spawn.lane` parameter plus compatible per-lane subagent config is the narrow maintainable direction while preserving existing defaults.
