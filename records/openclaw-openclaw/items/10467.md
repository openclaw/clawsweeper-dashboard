## Summary

Keep open. Current main and the latest release still implement `sessions_spawn` with a fixed `subagent` queue lane, no caller-provided lane parameter, and only one scalar subagent concurrency cap.

## What This Changes

The issue requests an optional `sessions_spawn.lane` input with independent sub-agent queue lanes and per-lane concurrency configuration while preserving the default `subagent` lane.

## Best Possible Solution

Add a validated additive `sessions_spawn.lane` field that defaults to `subagent`, thread it through native and ACP spawns, and support compatible per-lane subagent concurrency config with docs and regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main and `v2026.5.22` lack a `sessions_spawn.lane` schema field, both native and ACP dispatch are hard-coded to `subagent`, and config/docs expose one scalar subagent cap.

## Solution Assessment

Yes. The narrow maintainable path is an additive lane parameter backed by the existing named-lane queue, preserving current defaults while adding bounded validation, config, docs, and tests.
