## Summary

Keep open. Current main still exposes `sessions_spawn` through one fixed `subagent` lane with no caller `lane` field, no native/ACP lane forwarding, and only scalar subagent concurrency config; the recent maintainer-side comments remain accurate.

## What This Changes

Add optional multi-lane support to `sessions_spawn`, including per-lane subagent concurrency configuration while preserving the default `subagent` behavior.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` field that defaults to `subagent`, validates lane names, forwards through native and ACP spawns, applies scoped per-lane subagent concurrency, and updates schema, docs, changelog, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection on current main gives a high-confidence verification path: the schema, spawn params, dispatch calls, config schema, docs, and tests still expose one fixed `subagent` lane.

## Solution Assessment

Yes. The existing command queue already supports arbitrary named lanes, so an additive tool parameter plus compatible per-lane config is the narrowest maintainable direction while preserving current defaults.
