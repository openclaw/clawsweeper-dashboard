## Summary

Keep open. Current main still exposes `sessions_spawn` through one fixed `subagent` queue lane, with no caller-selected lane, no native/ACP lane forwarding, and only scalar subagent concurrency config; the recent issue comments and the closed per-agent-lane PR do not supersede this requested subagent-lane feature.

## What This Changes

Add an optional `lane` parameter to `sessions_spawn` plus per-lane subagent concurrency config so independent spawned-agent workloads can avoid sharing one global lane.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` field that defaults to `subagent`, validates lane names, forwards through native and ACP spawns, applies scoped per-lane subagent concurrency, and updates schema, docs, changelog, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: the schema, spawn params, dispatch calls, config schema, docs, and tests still expose one fixed `subagent` lane.

## Solution Assessment

Yes. The existing command queue already supports arbitrary named lanes, so an additive `sessions_spawn.lane` parameter plus compatible per-lane config is the narrow maintainable direction while preserving existing defaults.
