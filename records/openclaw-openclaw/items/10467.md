## Summary

Current main still has the requested feature gap, no merged or viable canonical replacement owns it, and maintainer history says queue-lane semantics need an owner-backed design. This should stay open for a product/API decision rather than autonomous implementation.

## What This Changes

Add an optional `sessions_spawn` lane selector with compatible per-lane concurrency configuration while retaining the existing default sub-agent lane.

## Best Possible Solution

Approve one additive, default-preserving scheduling contract that validates lane names and defines precedence across global, lane, agent, and parent limits, then implement it consistently for native and ACP spawns with schema, reload, docs, and regression coverage.

## Reproduction Assessment

Yes at source level: current main has no `sessions_spawn` lane input, both runtime paths use the fixed sub-agent lane, and configuration exposes only one scalar sub-agent cap. This establishes the feature gap without needing a failing runtime test.

## Solution Assessment

The additive fallback-to-`subagent` direction is sensible, but the exact proposed API and config are not yet the best final solution because maintainers must first settle lane namespaces, lifecycle, and concurrency precedence.
