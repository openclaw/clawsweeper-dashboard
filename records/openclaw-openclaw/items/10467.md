## Summary

Keep open. Current main still exposes `sessions_spawn` without a `lane` input, both native and ACP child dispatches hard-code the `subagent` lane, and configuration/docs/tests still describe one global sub-agent concurrency cap. The underlying Gateway queue already accepts arbitrary string lanes, so this remains a valid bounded feature gap rather than stale, implemented, or out of scope.

## What This Changes

Add optional named lane support to `sessions_spawn`, with per-lane sub-agent concurrency configuration and default fallback to the existing `subagent` lane.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` API that defaults to `subagent`, validates lane names, forwards the resolved lane through native and ACP spawn paths, applies per-lane sub-agent concurrency during Gateway startup and config reload, and updates schema/help/docs/changelog plus regression tests without changing existing callers.

## Reproduction Assessment

Yes. The feature gap is reproducible by static inspection on current main: no `lane` schema field exists, no execution path reads or forwards a lane, and both native and ACP child dispatches send the fixed `subagent` lane.

## Solution Assessment

Yes. An additive lane parameter is the narrowest maintainable direction because the Gateway `agent` protocol and command queue already support named lanes; the repair should reuse that infrastructure instead of adding a separate scheduler.
