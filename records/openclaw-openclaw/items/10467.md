## Summary

Keep open. Current main still routes `sessions_spawn` through one fixed `subagent` lane, and source/docs/tests still expose only scalar sub-agent concurrency; #52655 was a separate per-agent inbound-lane PR and was closed unmerged, so it does not supersede this work.

## What This Changes

Request an additive `sessions_spawn.lane` parameter and per-lane sub-agent concurrency configuration so independent child-agent classes do not all share the single `subagent` queue.

## Best Possible Solution

Keep the issue open until OpenClaw exposes a backward-compatible lane field, configures per-lane sub-agent concurrency, and covers both native and ACP spawns with docs and regression tests.

## Reproduction Assessment

Yes. Static source inspection on current main shows no `lane` field in the tool schema, no execute-path forwarding, and both native/ACP child dispatches still send the fixed `subagent` lane.

## Solution Assessment

Yes. An additive `sessions_spawn.lane` parameter that defaults to `subagent` is the narrowest maintainable direction because the command queue already supports arbitrary named lanes and existing spawn paths only need to forward a resolved lane.
