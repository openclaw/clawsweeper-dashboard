## Summary

Keep open. Current main still lacks a caller-selectable `sessions_spawn` lane: the tool schema has no `lane` input, both native and ACP spawn paths dispatch through the fixed `subagent` lane, and gateway/config concurrency remains a single scalar subagent cap.

## What This Changes

The issue requests a `sessions_spawn.lane` parameter with per-lane subagent concurrency configuration while preserving the existing default `subagent` lane.

## Best Possible Solution

Add a validated `sessions_spawn.lane` field defaulting to `subagent`, forward it through native and ACP dispatch, and support compatible per-lane subagent concurrency config with docs and regression tests.

## Reproduction Assessment

Yes. This is source-reproducible on current main: `sessions_spawn` has no lane field, both dispatch paths hard-code the `subagent` lane, and config/docs/tests still describe one scalar lane cap; I did not run a live saturation scenario because this was a read-only review.

## Solution Assessment

Yes. An additive lane parameter backed by the existing string-lane queue is the narrow maintainable direction, provided lane validation, default fallback, and per-lane config compatibility are handled carefully.
