## Summary

Keep open: current main and v2026.6.6 still expose `sessions_spawn` without a caller-selected lane and dispatch both native and ACP children onto the fixed `subagent` lane, so the requested lane API and per-lane concurrency semantics remain unresolved.

## What This Changes

Request adding a caller-selected `lane` parameter to `sessions_spawn` plus per-lane concurrency configuration for independent native and ACP sub-agent queues.

## Best Possible Solution

Keep the existing `subagent` default, then add a maintainer-approved additive lane parameter and per-lane cap semantics across schema, native/ACP dispatch, gateway lane configuration, docs, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for the source-level feature gap: current main exposes no `sessions_spawn.lane` schema field and hard-codes native and ACP dispatch to `subagent`. I did not run a live multi-lane workload because the requested input surface does not exist.

## Solution Assessment

Unclear until maintainers choose the product contract. The narrow implementation should be additive and default-preserving, but the best public shape needs to account for adjacent per-agent concurrency and lane-isolation proposals.
