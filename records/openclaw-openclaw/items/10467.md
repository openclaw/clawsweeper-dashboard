## Summary

Keep open. Current main still routes every `sessions_spawn` child run through one fixed `subagent` lane, and the related open items cover adjacent capacity/per-agent/per-parent scheduling rather than this caller-selectable lane request.

## What This Changes

Request add caller-selectable `sessions_spawn` queue lanes with per-lane subagent concurrency configuration.

## Best Possible Solution

Add an additive `lane` field that defaults to `subagent`, validates lane names, forwards it through native and ACP spawns, applies compatible per-lane subagent concurrency config, and updates schema, docs, changelog, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives a high-confidence way to verify the gap: no tool `lane` field, no spawn forwarding, hard-coded native/ACP subagent lane dispatch, and only scalar concurrency config.

## Solution Assessment

Yes. The existing command queue already supports named lanes, so a backward-compatible `sessions_spawn.lane` plus per-lane config is the narrowest maintainable path while preserving existing callers.
