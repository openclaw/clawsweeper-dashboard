## Summary

Keep open. Current main still routes `sessions_spawn` through one fixed `subagent` queue lane: the schema has no caller-selected lane, native and ACP dispatch both hard-code `AGENT_LANE_SUBAGENT`, and config/docs/tests still describe scalar global subagent concurrency.

## What This Changes

Request adds optional `sessions_spawn.lane` support plus per-lane subagent concurrency so independent sub-agent workloads can avoid one shared queue bottleneck.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` field that defaults to `subagent`, validates lane names, forwards through native and ACP spawn paths, applies scoped per-lane subagent concurrency, and updates schema, docs, changelog, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives a high-confidence verification path: schema, spawn params, dispatch calls, config schema, docs, and related tests still expose one fixed `subagent` lane.

## Solution Assessment

Yes. The existing command queue already supports named lanes, so adding an additive `sessions_spawn.lane` parameter plus compatible per-lane subagent config is the narrow maintainable direction while preserving existing defaults.
