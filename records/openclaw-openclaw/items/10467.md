## Summary

Keep open. Current main still exposes `sessions_spawn` through one fixed `subagent` lane, with no caller-selectable lane field, no native/ACP forwarding, and no per-lane subagent concurrency config; the related open items cover adjacent scheduling scopes rather than superseding this exact request.

## What This Changes

Add an optional `sessions_spawn.lane` parameter plus compatible per-lane subagent concurrency configuration while preserving the default `subagent` lane.

## Best Possible Solution

Implement a backward-compatible `lane` field that defaults to `subagent`, validates lane names, forwards the resolved lane through native and ACP spawns, applies per-lane subagent concurrency config, and updates schema, docs, changelog, and regression tests.

## Reproduction Assessment

Yes for the feature gap. Static source inspection on current main shows no `lane` field in the tool schema, no execute-path forwarding, and hard-coded `subagent` dispatch for both native and ACP child runs.

## Solution Assessment

Yes. The existing command queue already supports named lanes, so an additive `sessions_spawn.lane` plus compatible per-lane config is the narrowest maintainable direction while preserving current callers.
