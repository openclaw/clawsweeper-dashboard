## Summary

Keep open: current main and v2026.6.8 still expose `sessions_spawn` without a caller-selected lane, and both native and ACP child runs still dispatch on the fixed `subagent` queue lane with one scalar subagent concurrency cap. The remaining work is real but needs maintainer product/API direction because it adds model-visible tool input and config semantics that overlap adjacent scheduler proposals.

## What This Changes

The issue requests optional lane selection for `sessions_spawn` plus per-lane subagent concurrency configuration.

## Best Possible Solution

Add a default-preserving, maintainer-approved lane contract that keeps existing callers on `subagent` while wiring validated custom lanes through native and ACP spawn paths, gateway configuration, docs, and tests.

## Reproduction Assessment

Yes for the source-level feature gap: current main exposes no `sessions_spawn.lane` schema field and hard-codes native and ACP dispatch to `subagent`. I did not run a live multi-lane workload because the requested input surface does not exist.

## Solution Assessment

Unclear until maintainers choose the product/API contract. The likely direction is an additive, default-preserving lane parameter plus per-lane caps, but it must compose with adjacent subagent concurrency proposals.
