## Summary

Keep open. Current main still exposes `sessions_spawn` through one fixed `subagent` lane for both native and ACP child runs, and the related open/closed items do not supersede this named-lane request.

## What This Changes

Request an optional `sessions_spawn.lane` parameter with compatible per-lane subagent concurrency configuration while preserving the default `subagent` behavior.

## Best Possible Solution

Add a validated `sessions_spawn.lane` surface that defaults to `subagent`, forward it through native and ACP dispatch, and support compatible per-lane subagent concurrency config with focused tests and docs.

## Reproduction Assessment

Yes. The issue is source-reproducible: current main has no `lane` input in `sessions_spawn`, both native and ACP dispatch paths hard-code `subagent`, and docs/config still describe one scalar lane cap; I did not run a live saturation scenario.

## Solution Assessment

Yes. An additive lane parameter backed by the existing string-lane queue is the narrow maintainable direction, provided default behavior stays `subagent` and lane names/config are bounded.
