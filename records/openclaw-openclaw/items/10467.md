## Summary

Keep open: current main and the latest v2026.6.8 release still expose `sessions_spawn` without a caller-selected lane, and native plus ACP child runs still dispatch on the fixed `subagent` queue lane with one scalar subagent concurrency cap. The discussion includes another user confirming the single-lane bottleneck, and the existing maintainer-side review remains accurate against current source.

## What This Changes

Request to add a caller-selected `lane` parameter to `sessions_spawn` plus compatible per-lane subagent concurrency configuration.

## Best Possible Solution

Add a maintainer-approved, additive `sessions_spawn.lane` contract that defaults to `subagent`, validates bounded lane names, forwards through native and ACP paths, preserves the current scalar default, and documents/tests per-lane cap semantics.

## Reproduction Assessment

Yes, at source level: current main exposes no `sessions_spawn.lane` schema field and hard-codes both native and ACP dispatch to `subagent`. A live multi-lane workload was not run because the requested input surface does not exist.

## Solution Assessment

Unclear until maintainers choose the product contract. The maintainable direction is likely additive and default-preserving, but it must compose with per-target-agent cap and queue-visibility proposals instead of creating competing scheduler knobs.
