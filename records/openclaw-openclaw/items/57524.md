## Summary

Keep open. Current main still has the sudo/systemd mismatch this PR targets, so the work is not obsolete; however, the latest patch is not ready to merge because it leaves first-party install paths uncovered, breaks existing daemon install tests, emits an incorrect custom-unit diagnostic, and includes an unrelated build-output change.

## What This Changes

The PR adds a Linux sudo guard to daemon install, adds post-install linger/runtime warnings and tests, and enables node build source maps.

## Best Possible Solution

Update this as a narrow Linux user-systemd install fix: reject sudo consistently at a shared install boundary or every first-party daemon install entrypoint, keep `loginctl enable-linger` as the explicit privileged step, fix the affected CLI/test mocks and custom-unit diagnostics, remove unrelated build config changes, and add focused tests for sudo rejection and non-sudo install behavior.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main resolves the unit path from `HOME` while sudo systemctl operations target `SUDO_USER`, and the PR’s latest patch has directly inspectable test failures from the message mismatch and missing mocked export.

## Solution Assessment

No. Rejecting sudo before writing a user systemd unit is a maintainable direction, but this patch is not the best current implementation because the guard is not applied consistently, tests are stale, diagnostics can point at the wrong unit, and an unrelated build artifact change is mixed in.
