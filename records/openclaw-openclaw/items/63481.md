## Summary

Keep open. The paired Feishu startup bug still exists on current main, and this PR is a targeted Feishu-owned fix, but the branch is currently dirty against main, lacks the required user-facing changelog entry, and should be reconciled with the open Feishu startup-latency PR before merge.

## What This Changes

The PR adds a module-level Feishu startup bot-identity probe queue around `fetchBotIdentityForMonitor` plus regression tests for concurrent callers, failed probes, and aborted queued callers.

## Best Possible Solution

Keep the fix in the Feishu plugin, not the generic gateway: serialize or otherwise rate-limit startup bot-identity probes across concurrent account starts, preserve prompt abort behavior, add a current changelog entry, then validate the rebased branch alongside the latency work in #67466.

## Reproduction Assessment

Yes. Current main shows a high-confidence code-path reproduction: gateway account startup runs accounts through `Promise.all`, Feishu `startAccount` uses the account-specific monitor path, and that path directly awaits `fetchBotIdentityForMonitor` without any shared queue.

## Solution Assessment

Yes, with maintainer validation. Queuing at `fetchBotIdentityForMonitor` is the narrowest Feishu-owned place to cover both gateway-driven concurrent account starts and direct callers, but the final merge should reconcile the startup-latency tradeoff raised by #67466.
