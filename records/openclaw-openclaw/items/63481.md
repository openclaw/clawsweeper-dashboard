## Summary

Keep this PR open. Current main still has the Feishu multi-account startup path that can call the startup identity probe concurrently, and this PR remains a focused implementation candidate for the linked timeout report. I did not find a concrete functional or security regression in the proposed queue/abort handling, but maintainers should reconcile it with the separate Feishu startup-latency PR and require current validation before merge.

## What This Changes

The PR adds a shared Feishu startup probe queue around `fetchBotIdentityForMonitor` and regression tests for concurrent callers, failure isolation, and aborted callers waiting behind an in-flight probe.

## Best Possible Solution

Keep the fix in the Feishu plugin: throttle or serialize startup bot-identity probing across concurrent gateway account startups, preserve prompt abort behavior and background recovery, then merge with focused Feishu regression coverage and a current changed gate.

## Reproduction Assessment

Yes. The current-main reproduction path is visible in code: gateway account startup uses `Promise.all`, Feishu `startAccount` enters `monitorFeishuProvider` with an `accountId`, and the single-account monitor directly calls `fetchBotIdentityForMonitor` with no shared queue.

## Solution Assessment

Yes, with maintainer validation. Queuing at `fetchBotIdentityForMonitor` is a narrow Feishu-owned fix that covers concurrent per-account callers without changing generic gateway startup semantics; the main alternative to review is the separate latency PR's parallelize-and-cap approach.
