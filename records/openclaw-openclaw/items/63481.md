## Summary

Keep open. The PR still targets a valid source-reproducible Feishu multi-account startup bug on current main, but it lacks real after-fix Feishu proof and is currently conflicting/dirty, so it needs maintainer review rather than cleanup closure.

## What This Changes

This PR adds a module-level Feishu startup bot-identity probe queue in `monitor.startup.ts` and regression tests for concurrent, failed, pre-aborted, and aborted-while-waiting callers.

## Best Possible Solution

Land a Feishu plugin fix that shares startup bot-identity probe throttling across gateway account fan-out while preserving abort behavior and gateway monitor lifecycle, then reconcile it with the related Feishu startup-timeout work before merge.

## Reproduction Assessment

Yes, at source level: gateway startup can run up to four Feishu account starts concurrently, and the accountId path fetches bot identity directly without a shared queue. Live Feishu credentials would be needed to re-exercise the external vendor timeout itself.

## Solution Assessment

Yes, with maintainer validation. Queuing at `fetchBotIdentityForMonitor` is a narrow Feishu-owned fix that covers gateway fan-out and direct callers without serializing all channel startup in core, but it must be rebased and reconciled with related Feishu latency/timeout PRs.
