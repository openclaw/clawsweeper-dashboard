## What This Changes

The PR adds a module-level Feishu startup bot-identity probe queue in `monitor.startup.ts` plus regression tests for concurrent, failed, and aborted queued probes.

## Best Possible Solution

Keep the fix in the Feishu plugin: share startup bot-identity probe throttling across gateway concurrent account starts, preserve abort behavior, and reconcile the queue with the latency work before landing.

## Reproduction Assessment

Yes. Current main gives a high-confidence source-level path: gateway account startup can run up to four accounts concurrently, Feishu passes `accountId` into the per-account monitor path, and that path fetches bot identity directly without a shared queue.

## Solution Assessment

Yes, with maintainer validation. Queuing at `fetchBotIdentityForMonitor` is a narrow Feishu-owned fix that covers gateway fan-out and direct callers, but it should be rebased and reconciled with the competing startup-latency approach before merge.
