## Summary

Keep open. Current main still lacks the internal-probe log downgrade, but this PR’s current predicate depends on an Origin header that the current probe client does not send, and the external contributor proof gate is still missing.

## What This Changes

This PR adds a server-side `isInternalProbeClose()` check in `src/gateway/server/ws-connection.ts` to log loopback code-1000 pre-handshake closes at debug instead of warn when Origin matches the gateway host.

## Best Possible Solution

Use a deterministic internal-probe signal the server can observe, downgrade only that clean probe close, and add regression coverage for debug-vs-warn routing while preserving WARN for unrelated pre-connect failures.

## Reproduction Assessment

Yes, at source level. Current main routes non-SwiftPM pre-client closes to WARN, `probeGateway()` uses a probe-mode client and stops it after success, and the PR defect is visible because the client does not configure the Origin header the new predicate requires.

## Solution Assessment

No. The server-only Origin predicate is not the best fix because the current probe client does not send that signal; a deterministic probe header/client option or server-side probe close cause is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Match a signal the probe actually sends:** `src/gateway/server/ws-connection.ts:246`
  - body: `isInternalProbeClose()` requires `requestOrigin`, but `GatewayClient` builds `new WebSocket(url, wsOptions)` without `origin` or custom headers, and `ws` only sends Origin when `opts.origin` is set. Internal probe closes therefore will not match this predicate, so the WARN flood can remain; add or use a deterministic probe signal and cover the log-level branch.
