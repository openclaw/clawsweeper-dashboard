## Summary

Keep open. The PR targets a real gateway WARN-noise bug, but the current patch still depends on a WebSocket Origin value that the internal `GatewayClient` does not send, so the intended health-probe close will not reliably match; the PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds an `isInternalProbeClose()` predicate in `src/gateway/server/ws-connection.ts` to downgrade loopback code-1000 pre-handshake health-probe closes from warn to debug.

## Best Possible Solution

Use a deterministic probe signal the client actually sends, such as an explicit probe Origin/header or existing server-side probe metadata, and add focused regression coverage for debug-vs-warn routing while preserving WARN for unrelated pre-connect closes.

## Reproduction Assessment

Yes, at source level. Current main warns non-SwiftPM pre-connect closes, `probeGateway` stops a probe-mode `GatewayClient` after success, and that client does not send the Origin header required by the PR predicate.

## Solution Assessment

No. The current server-only Origin predicate is not the best fix because it depends on a signal absent from the probe client; the safer path is to add or use a deterministic probe signal and test the log-level branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Match a signal the probe actually sends:** `src/gateway/server/ws-connection.ts:246`
  - body: `isInternalProbeClose` requires `requestOrigin`, but `GatewayClient` creates `new WebSocket(url, wsOptions)` without `origin` or custom headers, and `ws` only sends Origin when `opts.origin` is set. Internal probe closes therefore will not match this predicate, so the WARN flood remains; set a deterministic probe Origin/header or key this check off a signal the probe actually sends, then test it.
  - confidence: 0.9
