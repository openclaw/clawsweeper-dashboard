## Summary

Keep open. Current main still sends non-SwiftPM pre-connect closes through `warn`, but this PR's server-only predicate depends on an Origin header the current probe client does not set, so it likely leaves the reported WARN flood unchanged.

## What This Changes

The PR adds a gateway WebSocket helper that downgrades loopback, origin-matching, code-1000 pre-handshake closes from `warn` to `debug`.

## Best Possible Solution

Use an explicit, tested probe signal: either set a deterministic probe Origin/header in `GatewayClient` for probe mode or key the server downgrade off a signal the probe actually sends, while preserving warnings for external, non-loopback, and non-1000 pre-auth closes.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: `probeGateway` stops a probe-mode `GatewayClient`, current `GatewayClient` does not set an Origin header, and current `ws-connection.ts` warns every pre-connect close except the SwiftPM helper case.

## Solution Assessment

No. The submitted server-only Origin predicate is not the narrowest reliable fix unless the probe client also sends that Origin; the safer approach is to make the probe signal explicit and cover the exact log-level routing in gateway tests.
