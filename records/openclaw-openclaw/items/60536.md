## Summary

Keep open, but the patch is not correct yet. Current main still warns the reported pre-connect close path, and this PR's new predicate depends on an Origin header that the internal probe client does not send, so the WARN flood can remain unchanged.

## What This Changes

The PR adds an internal health-probe predicate in `src/gateway/server/ws-connection.ts` to route loopback code-1000 pre-handshake closes to debug instead of warn.

## Best Possible Solution

Use a deterministic probe signal the client actually sends, then add regression coverage for debug-vs-warn routing while preserving WARN for unrelated pre-connect closes.

## Reproduction Assessment

Yes, at source level. Current main routes non-SwiftPM pre-client closes to WARN, `probeGateway` stops a probe-mode client after success, and the client constructs the WebSocket without an Origin signal; I did not run a live gateway because this review is read-only.

## Solution Assessment

No. The server-only Origin predicate is not the best fix because the probe does not send that signal; a deterministic probe header/origin or server-side probe metadata is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
