## Summary

This PR targets a real gateway log-noise bug, but it should stay open because the current patch relies on an `Origin` header the probe client does not send and the external contributor proof gate is still missing.

## What This Changes

The PR adds an `isInternalProbeClose()` check in `src/gateway/server/ws-connection.ts` to log loopback, code-1000, pre-handshake health-probe closes at debug instead of warn.

## Best Possible Solution

Use a deterministic internal-probe signal the server can actually observe, downgrade only clean internal probe closes, and add focused regression coverage for debug-vs-warn routing while preserving warnings for unrelated pre-handshake failures.

## Reproduction Assessment

Yes, at source level. Current main still routes ordinary pre-client closes to WARN, the probe path stops a probe-mode `GatewayClient` after success, and the PR's replacement predicate depends on an Origin value the current client does not set.

## Solution Assessment

No. The server-only Origin check is not the best fix because the current probe client does not provide that signal; a deterministic probe marker or another reliable server-visible probe cause would be narrower and more maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Match a signal the probe actually sends:** `src/gateway/server/ws-connection.ts:246`
  - body: `isInternalProbeClose()` requires `requestOrigin`, but `GatewayClient` constructs the `ws` connection without `origin` or custom headers, and `ws` only sends `Origin` when `opts.origin` is configured. Internal probe closes can therefore miss this predicate and keep logging at WARN; use a deterministic probe signal and cover the log-level branch.
  - confidence: 0.91
