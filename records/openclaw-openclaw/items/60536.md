## Summary

Keep open. The PR targets the right gateway WARN-noise path and already addressed Greptile's wildcard-host/comment concerns, but the remaining predicate depends on an Origin header that current probe clients do not send, so it likely leaves the reported WARN flood unchanged.

## What This Changes

Adds an internal health-probe close predicate in gateway WebSocket close handling to downgrade loopback code-1000 pre-handshake closes from warn to debug.

## Best Possible Solution

Set an explicit, tested probe signal, such as a deterministic probe Origin/header in GatewayClient, or key the downgrade on an existing server-side signal the probe actually sends while preserving WARN for external, non-loopback, and non-1000 pre-auth closes.

## Reproduction Assessment

Yes, at source level. Current main routes non-SwiftPM pre-connect closes to WARN, probeGateway stops a probe-mode GatewayClient after success, and GatewayClient does not set the Origin header required by the PR predicate.

## Solution Assessment

No. The submitted server-only Origin predicate is not reliable unless the probe client also sends that Origin; the narrower maintainable fix is to add or use an explicit probe signal and cover the debug-vs-warn routing in gateway tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Match a signal the probe actually sends:** `src/gateway/server/ws-connection.ts:246`
  - body: `isInternalProbeClose` requires `requestOrigin`, but the current `GatewayClient` creates `new WebSocket(url, wsOptions)` without `origin` or custom headers, and ws only sends Origin when `opts.origin` is set. Internal probe closes therefore will not match this predicate, so the WARN flood remains; set a deterministic probe Origin/header or key this check off a signal the probe actually sends, then test it.
