## Summary

Keep open. The PR targets a valid gateway log-noise bug, but the current head predicates the downgrade on an Origin header that the internal probe client does not set, so it likely leaves the reported WARN flood in place and still lacks after-fix real behavior proof.

## What This Changes

The PR adds an internal health-probe close predicate in `src/gateway/server/ws-connection.ts` to downgrade loopback code-1000 pre-handshake closes from warn to debug.

## Best Possible Solution

Set an explicit, tested probe signal, such as a deterministic probe Origin/header in `GatewayClient`, or key the downgrade on an existing server-side signal the probe actually sends while preserving WARN for unrelated pre-auth closes.

## Reproduction Assessment

Yes, at source level. Current main warns non-SwiftPM pre-connect closes, `probeGateway` stops a probe-mode `GatewayClient` after success, and that client does not set the Origin header the PR predicate requires.

## Solution Assessment

No. The server-only Origin predicate is not reliable unless the probe client also sends that Origin; the maintainable fix should add or use a deterministic probe signal and cover debug-vs-warn routing in gateway tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Match a signal the probe actually sends:** `src/gateway/server/ws-connection.ts:246`
  - body: `isInternalProbeClose` requires `requestOrigin`, but `GatewayClient` creates `new WebSocket(url, wsOptions)` without `origin` or custom headers, and ws only sends Origin when `opts.origin` is set. Internal probe closes therefore will not match this predicate, so the WARN flood remains; set a deterministic probe Origin/header or key this check off a signal the probe actually sends, then test it.
