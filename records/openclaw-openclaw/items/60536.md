## Summary

Keep this PR open. It targets the right gateway log-noise surface, but current main still only downgrades SwiftPM helper pre-connect closes, and this branch is stale plus likely ineffective as submitted because the server-side predicate requires an Origin header that the current `GatewayClient` construction does not set.

## What This Changes

The PR adds a gateway WebSocket helper that downgrades loopback, origin-matching, code-1000 pre-handshake closes from `warn` to `debug`.

## Best Possible Solution

Update or replace the PR with a current-main patch that uses a signal the probe actually sends, or sets a deterministic probe Origin/header in the client, then add focused gateway WebSocket logging tests and a changelog entry for the user-visible log-noise fix.

## Reproduction Assessment

Yes. A live gateway restart with log tailing should reproduce the reported WARN noise, and the static path is high-confidence: `probeGateway` stops a probe-mode `GatewayClient`, while the current close handler warns every pre-connect close except the SwiftPM helper case.

## Solution Assessment

No. The submitted server-only Origin predicate is not the best fix unless the probe client also sends that Origin; the safer repair is to make the client/server signal explicit and test the exact debug-vs-warn routing.
