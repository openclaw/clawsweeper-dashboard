## Summary

Current main still contains a Discord IDENTIFY wait path that can hit `this.ws === null` before IDENTIFY is sent, while the focused fix PR for this issue was closed unmerged. The report has concrete user logs, an independent confirmation on another Linux setup, and a source-level reproduction path, so this should stay open and be handled as a focused Discord gateway bug.

## What This Changes

The issue reports a 2026.5.7 regression where Discord guild inbound messages never arrive because startup stays at `awaiting gateway readiness` and never logs `logged in to discord`.

## Best Possible Solution

Land a focused Discord gateway fix that handles `this.ws === null` during the IDENTIFY concurrency wait, keeps the existing non-null reconnect behavior, and adds regression coverage for the null-socket path; then validate against a real Discord bot startup/guild inbound flow.

## Reproduction Assessment

Yes at source level: hold the shared IDENTIFY limiter, let HELLO start `identifyWithConcurrency()`, clear `this.ws` via disconnect/reconnect before the limiter resumes, and current main enters the null-socket branch without scheduling from that branch. I did not establish a live Discord reproduction in this read-only review.

## Solution Assessment

Yes, the narrowest repair is in `extensions/discord/src/internal/gateway.ts`: split the null-socket and non-open-socket cases and cover both in `gateway.test.ts`. A broader startup-lifecycle refactor is not required for the concrete branch shown here, but live proof should verify it resolves the reported guild inbound outage.

## Repair Work Prompt

Fix the Discord gateway READY stall reported in https://github.com/openclaw/openclaw/issues/79794. On current main, `extensions/discord/src/internal/gateway.ts` `identifyWithConcurrency()` waits on `sharedGatewayIdentifyLimiter`, rereads `this.ws`, and when the socket is null it emits `Discord gateway socket closed before IDENTIFY could be sent` and returns without scheduling from that branch. Make the fix narrowly in the Discord gateway owner module: handle `socket === null` as already disconnected/reconnecting or intentionally disconnected, and handle a non-null non-open socket by scheduling a clean reconnect without duplicate scheduling. Preserve existing reconnect/session semantics and do not change Discord config, gateway timeout defaults, docs wording, or unrelated startup lifecycle code. Add/update colocated regression tests in `extensions/discord/src/internal/gateway.test.ts` for both the non-null closed socket and null `this.ws` while the identify limiter is waiting. Treat https://github.com/openclaw/openclaw/pull/79799 as useful source context, but re-evaluate the exact code against current main and coordinate with adjacent https://github.com/openclaw/openclaw/pull/78227 so the fixes do not conflict. Add a single user-facing changelog fix entry if the repair changes shipped Discord behavior. Validation should include `pnpm test extensions/discord/src/internal/gateway.test.ts`; if a live Discord bot credential is available, also run `openclaw gateway run` and confirm `logged in to discord`, `/healthz` connected state, and a guild `MESSAGE_CREATE` reaches the agent, with private IDs/tokens redacted.
