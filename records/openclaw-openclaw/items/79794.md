## Summary

Keep open. Current main and the latest 2026.5.7 release still have a Discord IDENTIFY wait path that can leave READY unset when `this.ws` is null, and the only focused fix PR for this issue was closed unmerged.

## What This Changes

Fix a Discord gateway regression where the bot reaches the WebSocket/REST-visible online state but never reaches READY, so guild inbound messages are not received.

## Best Possible Solution

Port or recreate the focused Discord gateway fix from the closed PR, add null-socket regression coverage, and require redacted live proof showing READY, health status, and guild message receipt before closing this issue.

## Reproduction Assessment

Yes at source level: hold `sharedGatewayIdentifyLimiter`, emit Gateway HELLO, clear `this.ws` before the limiter resumes, and current main enters the null-socket branch without scheduling reconnect from that branch. I did not establish a live Discord reproduction in this read-only review.

## Solution Assessment

Yes for the narrow source-level race: splitting null and non-open socket handling in `GatewayPlugin.identifyWithConcurrency()` is the smallest owner-local fix. Because reporters also saw REST probe timeouts, live proof should confirm the fix covers the observed READY outage before this closes.
