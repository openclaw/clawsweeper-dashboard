## What This Changes

The issue requests a fix for a Discord startup regression where the gateway never reaches READY and guild messages are not received after upgrading to 2026.5.7.

## Best Possible Solution

Land a focused Discord gateway fix that splits null-socket and non-open-socket IDENTIFY handling, adds regression coverage for the null-socket limiter window, and validates READY/guild inbound recovery with redacted runtime proof when credentials are available.

## Reproduction Assessment

Yes at source level, but not from a live Discord bot in this read-only review. Hold `sharedGatewayIdentifyLimiter`, deliver HELLO, clear `this.ws` before the wait resumes, and current main enters the null-socket path that emits an error and skips reconnect from that branch.

## Solution Assessment

Yes. The narrow maintainable fix is to split `socket === null` from `socket.readyState !== READY_STATE_OPEN` in `identifyWithConcurrency()` and cover both branches with focused gateway tests; broader READY/reconnect reports should stay linked but not block this repair.
