## Summary

The central parent-routing and review-first behavior is now on main, but the issue still contains a distinct thread-bound session case where completion wake is disabled. Because the requested config/API shape is no longer clearly the best fix and an open focused PR addresses adjacent wake durability rather than this exact mode, this needs maintainer narrowing rather than closure.

## What This Changes

Add configurable native routing so sub-agent completion wakes the parent session without directly announcing to the channel.

## Best Possible Solution

Narrow this issue to require exactly one parent completion event after a directly delivered thread-bound session finishes, while keeping child-thread run delivery intact; only add a public routing option if maintainers explicitly want channel-directed completion as a separate supported mode.

## Reproduction Assessment

Yes. Current source shows ordinary native completions are parent-mediated, while a thread-bound session with direct child delivery deliberately sets `expectsCompletionMessage` false and omits the terminal parent wake.

## Solution Assessment

No, not in the original broad form. The narrowest maintainable solution is to close the remaining thread-bound completion-event gap on the canonical parent handoff path, then add a routing knob only if maintainers decide multiple permanent delivery modes are desirable.
