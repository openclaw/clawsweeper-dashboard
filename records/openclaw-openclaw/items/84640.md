## Summary

Keep open: current main still has a source-visible observability gap between embedded `attempt-dispatch` and Codex `session.started`, and the related searches did not find a canonical merged fix or duplicate that owns this exact lifecycle timing work.

## What This Changes

The issue requests Codex app-server thread lifecycle timing so latency between embedded attempt dispatch and `session.started` is visible.

## Best Possible Solution

Add a Codex app-server lifecycle stage tracker around `startOrResumeThread`, reuse the embedded startup summary style, cover both resume and start paths, and keep any native hook relay logging cleanup small and separately testable.

## Reproduction Assessment

Yes for the observability gap: current main records `session.started` only after `startOrResumeThread` completes and source search found no comparable Codex lifecycle stage summary. I did not live-reproduce the Docker latency sample itself.

## Solution Assessment

Yes, adding focused lifecycle timing is the narrowest maintainable way to make future latency reports actionable; it should not try to optimize the thread path until the new timings identify the expensive stage.
