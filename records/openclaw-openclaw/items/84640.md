Keep open: current main still has an observability gap between embedded `attempt-dispatch` and Codex `session.started`; the code emits coarse lifecycle events, but not the requested per-stage timing for `startOrResumeThread`.

## What This Changes

Add Codex app-server thread lifecycle stage timing around `startOrResumeThread` and reduce native hook relay debug noise that can confuse latency investigations.

## Best Possible Solution

Add a focused Codex app-server lifecycle stage tracker around `startOrResumeThread`, mirror the embedded startup summary format, cover both resume and start paths, and keep any native hook relay cleanup narrow.

## Solution Assessment

Yes, focused lifecycle timing is the narrowest maintainable way to make future latency reports actionable before attempting any optimization of the thread path.
