## Summary

Keep open: current main still emits the embedded startup summary before awaiting Codex thread startup, then records `session.started` only after `startOrResumeThread` returns, with no comparable Codex app-server lifecycle timing in that gap.

## What This Changes

The issue asks for trace/warn-stage instrumentation for the Codex app-server thread lifecycle so operators can see where time is spent before `session.started`.

## Best Possible Solution

Add bounded trace-level Codex app-server lifecycle stage timing around `startOrResumeThread`, with focused tests, while leaving broader warm-path caching and performance work to the related latency issues.

## Reproduction Assessment

Yes for the missing instrumentation: source inspection shows the uninstrumented span between embedded `attempt-dispatch` and Codex `session.started`. I did not live-reproduce the reported Docker latency sample or prove the six-second gap on current main.

## Solution Assessment

Yes. Focused lifecycle timing is the narrowest maintainable next step; optimizing thread startup or warm-path setup should wait until the new timings identify the expensive stage.
