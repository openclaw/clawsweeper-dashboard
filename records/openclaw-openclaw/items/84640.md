## Summary

Keep open: current main and the latest shipped tag still lack per-stage Codex app-server thread lifecycle timing between embedded attempt dispatch and `session.started`, and the source path matches the reported observability gap.

## What This Changes

The issue asks for trace-level Codex app-server thread lifecycle timing around `startOrResumeThread`, plus a small native hook relay debug-race cleanup if kept in scope.

## Best Possible Solution

Add a Codex app-server lifecycle stage tracker around `startOrResumeThread`, reuse the existing embedded startup summary style where practical, cover resume and start paths, and keep any native hook relay change limited to reducing misleading debug noise.

## Reproduction Assessment

Yes for the observability gap: current main records `session.started` only after `startOrResumeThread` completes and has no comparable Codex lifecycle stage summary. The exact Docker latency sample was not live-reproduced in this read-only review.

## Solution Assessment

Yes. Focused lifecycle timing is the narrowest maintainable solution because it makes future latency reports attributable without changing thread lifecycle behavior or prematurely optimizing an unmeasured stage.
