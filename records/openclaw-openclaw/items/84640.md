## Summary

Keep open: current main still has a real observability gap between the embedded `attempt-dispatch` trace and Codex `session.started`. A newer related report refines the likely expensive phases, so the fix should time the broader Codex active-run startup gap as well as the thread lifecycle.

## What This Changes

Request adds trace-level Codex app-server lifecycle timing between embedded `attempt-dispatch` and `session.started`, with optional cleanup for a noisy native hook relay debug race.

## Best Possible Solution

Add built-in timing for the full Codex active-run startup gap from embedded dispatch through `session.started`, with thread-lifecycle substages and a quieter/native-hook bridge-ready signal, while leaving actual latency optimization to follow-up measurements.

## Reproduction Assessment

Yes for the missing instrumentation: source inspection shows `session.started` is recorded only after Codex client/thread startup and no comparable built-in Codex lifecycle or active-run stage summary exists. The exact production latency sample was not live-reproduced on current main.

## Solution Assessment

Partly: lifecycle timing is useful and narrow, but the best fix should also time the surrounding Codex active-run phases because newer trace evidence shows the thread lifecycle itself may not be the dominant cost.
