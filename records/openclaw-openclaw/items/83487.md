## Summary

Keep this PR open: the source path confirms a real gateway security-boundary bug and the patch is a narrow, plausible fix, but the external PR still lacks after-fix proof from a real two-gateway OpenClaw setup.

## What This Changes

The PR stores a server-local `GatewayRequestContext` for plugin registries, injects it into plugin HTTP request/upgrade runtime scopes, reapplies it on plugin runtime replacement, and adds a regression test for active-route versus server-context mismatch.

## Best Possible Solution

Land the server-local context binding after a real two-gateway proof shows `admin-http-rpc` dispatch stays on the receiving gateway across startup and plugin reload boundaries.

## Reproduction Assessment

Yes, at source level: current main plugin HTTP scopes carry a client but no `GatewayRequestContext`, while dispatch falls back to a process-global context resolver. I did not run a live two-gateway repro in this read-only review.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: carry the server-local gateway context into plugin HTTP request/upgrade scopes while preserving existing route registry resolution. The remaining blocker is proof from a real multi-gateway setup, not an obvious code defect in the patch.
