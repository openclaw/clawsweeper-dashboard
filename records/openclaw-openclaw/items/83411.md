## Summary

Keep open: the bug is real, but this PR does not map the actual QuickJS interrupt-timeout path; the worker reports `internal_error` before the parent watchdog timer fires, so the new timeout assertion should fail.

## What This Changes

The branch widens file-local code-mode worker failure codes, classifies worker load errors as `runtime_unavailable`, maps the parent watchdog timeout to `timeout`, and adds regression assertions.

## Best Possible Solution

Classify timeout failures at the worker result boundary as well as the parent watchdog, keep `runtime_unavailable` mapping for worker-load failures, and prove both documented codes with focused regression tests plus redacted terminal or log output from patched code mode.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `while (true) {}` sets `didTimeout()` in `code-mode.worker.ts`, then the worker catch serializes the failed result with `code: "internal_error"` before the parent watchdog branch runs.

## Solution Assessment

No. The direction is right, but the patch maps only the parent watchdog timeout; the maintainable fix needs the worker-side timeout result to carry `timeout` too, with the existing code-mode execution test proving that path.
