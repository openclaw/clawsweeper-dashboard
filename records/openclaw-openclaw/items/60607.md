## Summary

Keep open. Current `main` has narrowed several no-reply cases, but it still deliberately allows no visible final reply in silent, deduped, already-delivered, and group/internal paths, so the requested always-notify behavior needs maintainer policy rather than cleanup closure.

## What This Changes

The issue requests a user-visible fallback, and possibly a `dispatch.requireReply`-style setting, when an agent run completes without replyable content.

## Best Possible Solution

Define an explicit require-reply contract per surface or conversation type, with documented exceptions for intentional silence, message-tool delivery, block-stream delivery, media-only payloads, and pending subagent completion.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: an agent result with `payloads: []`, or payloads filtered to `replyPayloads.length === 0`, finalizes without a final reply and can produce the Feishu `queuedFinal=false, replies=0` log shape.

## Solution Assessment

No. An unconditional fallback is not the best fix because current docs and tests intentionally preserve silent and already-delivered paths; a scoped opt-in policy is safer.
