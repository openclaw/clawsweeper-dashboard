## Summary

Keep open. Current main still has source-visible paths that can finish with `queuedFinal=false` and zero final replies, and the reported Feishu log shape still exists; however current docs, tests, and related PR review history show that a blanket fallback would change intentional silence, delivery suppression, and already-delivered side-effect behavior.

## What This Changes

The issue requests a cross-channel fallback or error reply, and possibly a new `dispatch.requireReply` option, when an agent run completes with zero renderable reply payloads.

## Best Possible Solution

Keep this issue as the broad policy tracker while maintainers review the scoped fallback work in https://github.com/openclaw/openclaw/pull/63025 and the silentReply failure-path work in https://github.com/openclaw/openclaw/issues/82060 / https://github.com/openclaw/openclaw/pull/82086.

## Reproduction Assessment

Yes, at source level: current main still has empty payload and post-filter zero-reply paths that return without a final payload, and Feishu still logs `queuedFinal` plus final reply count. I did not establish a live Feishu websocket reproduction in this read-only review.

## Solution Assessment

No, not as a blanket fallback or new universal `requireReply` knob yet. The narrower maintainable direction is to add scoped fallbacks that honor existing delivery suppression, silentReply policy, block/media delivery, and already-sent messaging-tool evidence.
