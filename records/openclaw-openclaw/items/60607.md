## Summary

Keep open. Current main and latest release still allow the central zero-sendable-output path to complete with `queuedFinal=false` and no fallback, while related merged work only covers a narrower failure-fallback silentReply subset and the active fallback PR still needs maintainer policy review.

## What This Changes

The issue requests a user-visible fallback when an agent run completes with no sendable reply payload, plus clearer timeout/failure handling.

## Best Possible Solution

Keep this issue as the broad tracker until maintainers land or supersede https://github.com/openclaw/openclaw/pull/63025 with a scoped fallback that preserves intentional silence and explicit delivery-suppression policy.

## Reproduction Assessment

Yes, at source level: current main can return `undefined` after empty or filtered payloads, and `dispatchReplyFromConfig` can complete with zero queued final replies. I did not run a live Feishu websocket reproduction in this read-only review.

## Solution Assessment

No, not as a blanket fallback or universal `dispatch.requireReply` option. The maintainable path is a scoped dispatch-level fallback that respects intentional silence, `sendPolicy`, message-tool delivery, routed-origin delivery, and side-effect-only turns.
