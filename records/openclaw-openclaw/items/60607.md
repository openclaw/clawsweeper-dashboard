## Summary

Keep open. Current main still does not implement the requested always-visible fallback or a `dispatch.requireReply` policy; it has partial empty/timeout safeguards, but documented `NO_REPLY`, group/channel silence, and message-tool-only delivery still permit intentional no-reply completions.

## What This Changes

The issue requests a guaranteed user-visible fallback when an agent run completes with no reply payloads, plus timeout error surfacing and optional dispatch-level reply enforcement.

## Best Possible Solution

Define an explicit opt-in require-reply policy at the dispatch or surface level, with clear exceptions for intentional silence, already-delivered message-tool output, media-only delivery, pending subagents, cron/internal runs, and group/channel defaults. Keep the narrower Anthropic tool-result fallback work tracked through #46080 unless maintainers choose to fold that case into the broader policy.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path remains: produce an agent turn whose runner result reduces to `payloadArray.length === 0` or `replyPayloads.length === 0`; the auto-reply runner finalizes without a final payload, and Feishu can then log the reported `queuedFinal=false, replies=0` completion.

## Solution Assessment

Unclear. The requested always-reply fallback is understandable, but applying it unconditionally is not the best maintainable fix because current docs and code intentionally support silence; an opt-in require-reply policy with scoped exceptions is safer.
