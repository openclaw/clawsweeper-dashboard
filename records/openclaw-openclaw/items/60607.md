## Summary

Keep open. Current main still has source-level no-final-reply paths and Feishu can still emit the reported zero-reply dispatch log, but existing silent-reply docs and tests preserve intentional silence and already-delivered cases, so the remaining change needs maintainer policy/config judgment.

## What This Changes

Request a user-visible fallback or require-reply mode when an agent run completes with no replyable payloads.

## Best Possible Solution

Define an explicit require-reply contract by surface or conversation type, with documented exceptions for intentional silence, already-delivered side effects, block-stream delivery, media-only payloads, and pending child runs; keep narrower provider/channel bugs in focused issues such as https://github.com/openclaw/openclaw/issues/46080.

## Reproduction Assessment

Yes, source-level: an empty run payload array or post-filter zero replyPayloads path returns undefined, and Feishu logs queuedFinal plus final reply counts. I did not run a live Feishu websocket reproduction in this read-only review.

## Solution Assessment

No, an unconditional fallback is not the best maintainable fix because current docs and tests preserve intentional silence and already-delivered paths. A scoped require-reply policy/config contract is safer than changing every zero-payload turn into a visible message.
