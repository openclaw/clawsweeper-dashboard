## Summary

Keep open. Current main fixed two related subsets, but the central zero-sendable-output dispatch path can still complete with `queuedFinal=false` and zero final replies, and the active dispatch-level fallback work still needs maintainer policy review.

## What This Changes

The issue requests a user-visible fallback or error when an agent run completes with no sendable reply instead of silently logging `queuedFinal=false, replies=0`.

## Best Possible Solution

Keep this as the broad policy tracker while maintainers land or supersede https://github.com/openclaw/openclaw/pull/63025 with a scoped fallback that respects delivery suppression, message-tool mode, side-effect delivery, and silent-reply policy.

## Reproduction Assessment

Yes, at source level: current main can return `undefined` from the runner after empty or post-filtered payloads, and dispatch then completes with dispatcher counts instead of a synthesized fallback. I did not run a live Feishu websocket reproduction.

## Solution Assessment

No, not as a blanket fallback or universal `dispatch.requireReply` knob yet. The maintainable direction is a scoped dispatch fallback that preserves intentional silence, delivery suppression, already-sent message-tool evidence, and group/channel visible-reply policy.
