## Summary

Keep open. Current main still has a source-level path where an agent run can finish with no reply payload queued, and the repo intentionally supports several no-visible-reply behaviors, so the requested always-visible fallback needs maintainer policy rather than cleanup closure.

## What This Changes

The issue requests a user-visible fallback when agent dispatch completes with no replyable payloads, plus an optional `dispatch.requireReply` policy.

## Best Possible Solution

Define an explicit opt-in require-reply contract at the dispatch or surface level, with scoped exceptions for intentional silence and already-delivered output, then cover the selected behavior in runner, dispatch, channel, and docs tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: produce a runner result that reduces to `payloadArray.length === 0` or `replyPayloads.length === 0`; the auto-reply runner finalizes without a final payload and Feishu can log `queuedFinal=false, replies=0`.

## Solution Assessment

Unclear. The requested always-reply fallback is understandable, but applying it globally is not the safest fix because current main deliberately supports silent completions; an opt-in policy with explicit exceptions is more maintainable.
