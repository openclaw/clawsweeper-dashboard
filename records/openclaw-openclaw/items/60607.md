## Summary

Keep open. Current main has narrowed some empty-response cases, including Kimi Anthropic-compatible recovery, but it still intentionally permits no visible final reply in empty-payload, deduped, already-delivered, group/internal, and silent-token paths, so the broad always-notify behavior remains a product/config decision.

## What This Changes

The issue requests a user-visible fallback or error whenever an agent run completes with no replyable payload, potentially via a new require-reply dispatch policy.

## Best Possible Solution

Define an explicit require-reply contract per surface or conversation type, with documented exceptions for intentional silence, message-tool delivery, block-stream delivery, media-only payloads, and pending subagent completion.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: an empty run payload array or a filtered replyPayloads.length === 0 path finalizes without a final reply, and Feishu still logs queuedFinal=false with replies=0 for such dispatches.

## Solution Assessment

No. An unconditional fallback is not the best fix because current docs and tests preserve intentional silence and already-delivered paths; a scoped opt-in policy is the safer maintainable direction.
