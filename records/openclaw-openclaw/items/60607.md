## Summary

Keep open. Current main still has zero-final-reply source paths that can produce the reported Feishu `queuedFinal=false, replies=0` log, but existing silent-reply policy, already-delivered dedupe, block-stream handling, and timeout synthesis make an unconditional fallback a product/config decision rather than conservative cleanup.

## What This Changes

Request a cross-channel require-reply/fallback policy so completed agent turns with no replyable payload notify the user instead of silently ending.

## Best Possible Solution

Define a scoped require-reply contract by surface or conversation type, with documented exceptions for intentional silence, already-delivered side effects, block streams, media-only payloads, and pending child runs; keep provider/channel-specific empty-final bugs in focused linked issues.

## Reproduction Assessment

Yes, at source level: an empty run payload array or post-filter zero `replyPayloads` path returns undefined, and Feishu still logs queued final state plus final reply count. I did not run a live Feishu websocket reproduction in this read-only review.

## Solution Assessment

No, an unconditional fallback is not the best maintainable fix because current docs and tests preserve intentional silence and already-delivered paths. A scoped require-reply policy/config contract is safer than changing every zero-payload turn into a visible message.
