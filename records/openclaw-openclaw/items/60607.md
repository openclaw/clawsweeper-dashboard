## Summary

Keep open. Current main still has a source-reproducible no-sendable-output path that can complete with `queuedFinal=false` and zero final replies, and the newest comment reports a fresh 2026.5.19 cross-channel variant rather than evidence that the broad no-reply class is fixed.

## What This Changes

The issue asks OpenClaw to avoid silently completing agent turns that produce no replyable output and instead send a controlled user-visible fallback or failure notice.

## Best Possible Solution

Keep this issue as the broad tracker while maintainers review or supersede https://github.com/openclaw/openclaw/pull/63025 with a scoped dispatch fallback that preserves intentional silence, delivery-suppression policy, routed-origin delivery, and already-delivered side effects.

## Reproduction Assessment

Yes, at source level: current main can reduce empty or post-filtered runner output to no replies, and dispatch can complete with zero queued final replies. I did not run a live Feishu, WeCom, or Telegram reproduction.

## Solution Assessment

No, not as a blanket fallback or universal `dispatch.requireReply` switch. The maintainable fix is a scoped dispatch-level fallback that preserves intentional silence and documented delivery-suppression policy.
