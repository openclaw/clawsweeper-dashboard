## Summary

Keep open for review, but this draft should not merge as-is: the recovery path depends on `replyTurn*` session fields that current main neither types nor writes, so the new branch is not grounded in the real session-store contract. The PR also lacks real Telegram behavior proof for the visible fallback message.

## What This Changes

The PR adds stale reply-turn recovery to diagnostic stuck-session cleanup by scanning session stores, sending a fallback through `message.action`, marking the turn failed, and adding regression coverage.

## Best Possible Solution

Base stale-turn recovery on a real persisted current-main turn-state contract, or add that typed contract and its writer before sending the fallback through the current `message.action` schema.

## Reproduction Assessment

No, not for the reported live Telegram stale-turn silence: the PR describes the scenario, but no forced stale-turn Telegram recovery proof is provided. Source inspection does show a high-confidence patch defect because the proposed `replyTurn*` trigger fields are not present or written on current main.

## Solution Assessment

No. The nested `message.action` request is the right schema direction, but the PR must first connect recovery to persisted runtime state instead of hand-seeded test-only fields.
