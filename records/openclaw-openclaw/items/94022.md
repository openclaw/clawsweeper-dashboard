## What This Changes

The PR moves startup catch-up deferral IDs into cron service state, uses that state in maintenance recompute, records and clears IDs in timer/startup paths, and adds regression tests plus a standalone runtime proof script.

## Best Possible Solution

## Solution Assessment

Yes. Moving pending deferral IDs into `CronServiceState` is the best current boundary because startup, read, manual, and timer maintenance callers share the recompute invariant; threading one-off skip sets through every caller would keep the policy duplicated.
