## Summary

Keep open: current main still has the cron startup catch-up deferral bug, and this PR is the current clean candidate fix with maintainer-updated history, sufficient terminal proof, and no concrete blocking review finding.

## What This Changes

The branch adds `pendingCatchupDeferralJobIds` to cron service state, uses it during maintenance recompute, records and clears IDs in startup/timer paths, and adds regression tests plus a standalone runtime proof script.

## Best Possible Solution

Land this branch as the canonical cron scheduler fix if maintainers accept the remaining upgrade risk, then close the linked issue and duplicate PRs after merge.

## Reproduction Assessment

Yes. Current main is source-reproducible because startup deferral IDs are only passed through `start()`'s local option, while read and timer maintenance recompute callers can run before the deferred slot without that exemption.

## Solution Assessment

Yes. Moving pending deferral IDs into `CronServiceState` is the best current boundary because all maintenance recompute callers share it; threading local skip sets through every caller would keep the policy duplicated.
