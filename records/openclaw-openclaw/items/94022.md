## Summary

Keep open: this PR is a viable canonical fix for the cron startup catch-up deferral bug, and I found no blocking code findings. Current main still keeps the exemption as a startup-only local skip set, while this branch moves it into shared cron service state; the remaining decision is maintainer selection/landing because the scheduler path is upgrade-sensitive and sibling PRs exist.

## What This Changes

The PR adds `pendingCatchupDeferralJobIds` to cron service state, uses it in maintenance recompute, records and clears deferral IDs in timer/startup paths, and adds cron regression coverage plus a standalone runtime proof script.

## Best Possible Solution

Land this branch as the canonical cron scheduler fix if maintainers accept the remaining upgrade risk, then close the linked issue and duplicate PRs after merge.

## Reproduction Assessment

Yes. Current main is source-reproducible because startup deferral IDs are only passed through `start()`'s local option, while read, manual, and timer maintenance recompute callers can run before the deferred slot without that exemption.

## Solution Assessment

Yes. Moving pending deferral IDs into `CronServiceState` is the best current boundary because all maintenance recompute callers share it; threading local skip sets through every caller would keep the policy duplicated and easier to miss.
