## What This Changes

The PR adds an in-memory pending startup catch-up deferral set to cron service state, uses it during maintenance recompute, cleans it up after firing or stale jobs, and adds cron regression coverage plus a runtime proof script.

## Best Possible Solution

Land the state-owned cron deferral repair after exact-head validation stays green, then close the linked cron issue and sweep duplicate fix PRs as superseded.

## Reproduction Assessment

Yes. The linked issue and current source show startup overflow deferrals are protected only in start() on main, while read/finalize/empty-due maintenance recomputes can still run future-slot repair without the skip set; I did not execute tests because this review is read-only.

## Solution Assessment

Yes, with normal merge caution. Moving the exemption into CronServiceState is the narrow owner-boundary fix because every maintenance recompute shares the same future-slot repair invariant, and the PR preserves ordinary stale-future repair for non-deferred jobs.
