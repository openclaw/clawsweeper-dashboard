## Summary

Keep open: current main still has only the startup-local deferral skip, while this PR is the canonical open candidate that moves startup catch-up deferral tracking into cron service state. I found no blocking patch defect; the remaining choice is whether maintainers accept the in-process CronService proof or want live gateway restart proof before merge.

## What This Changes

The branch stores startup catch-up deferral ids in `CronServiceState`, uses that set during maintenance recompute, cleans it up in timer paths, and adds cron regression tests plus a standalone runtime proof script.

## Best Possible Solution

Land this branch as the canonical state-owned cron deferral fix if maintainers accept the remaining proof scope, then close the linked issue and duplicate unmerged attempts after merge.

## Reproduction Assessment

Yes. Current main is source-reproducible because read/status, finalization, reservation release, and timer maintenance recompute paths can call `recomputeNextRunsForMaintenance` without the startup-local deferral skip set before the deferred slot fires.

## Solution Assessment

Yes. Moving pending deferral ids into `CronServiceState` is the best current boundary because all affected maintenance recompute callers share the same future-slot repair invariant; continuing to thread one-off skip sets through selected callers would keep the policy duplicated.
