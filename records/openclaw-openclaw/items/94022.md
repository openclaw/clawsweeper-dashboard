Keep open: this remains a useful, mergeable candidate fix for the cron startup deferral clobber, with no blocking code findings found. The remaining maintainer work is choosing this canonical lane over the duplicate PRs and accepting the scheduler-risk tradeoff before landing.

## What This Changes

The PR moves startup catch-up deferral exemptions into `CronServiceState`, has maintenance recompute honor that shared state, cleans deferral IDs after execution or stale-job cleanup, and adds cron regression coverage plus a standalone runtime proof script.

## Best Possible Solution

Land this branch as the canonical fix if maintainers accept the scheduler-risk tradeoff, then close the duplicate branches and linked issue after merge.

## Reproduction Assessment

Yes. Current `origin/main` keeps pending overflow deferral IDs only in a one-shot `start()` option, while read, manual, and timer maintenance callers run without that skip before the deferred slot fires.

## Solution Assessment

Yes. Moving pending deferral IDs into `CronServiceState` is the shared owner boundary for all maintenance recompute callers and avoids the two weaker alternatives: globally disabling future repair or threading another local skip through every caller.
