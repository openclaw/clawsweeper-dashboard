## Summary

Keep this issue open. Current main still has the reported config-contract mismatch: strict per-agent validation omits `verboseDefault` and `elevatedDefault`, while defaults and runtime resolution still expose those values; open PR #74643 is the active unmerged closing implementation.

## What This Changes

The issue requests aligning per-agent `agents.list[]` config validation, types, help, docs, and tests for `verboseDefault` and `elevatedDefault`, or explicitly removing the unsupported per-agent surface.

## Best Possible Solution

Align the config contract by accepting both fields on `agents.list[]` with the same enums as `agents.defaults`, updating public types/help/generated schema/docs, and adding parser-level regression coverage for accepted and rejected values.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `agents.list[]` is validated through strict `AgentEntrySchema`, and that schema omits both reported keys while global defaults accept them; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes. Adding the missing schema/type/help/docs/test coverage is the narrowest maintainable fix because runtime/status code already consumes these resolved defaults; removing the per-agent runtime surface would be broader and more disruptive.
