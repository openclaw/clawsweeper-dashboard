## Summary

Keep this issue open. Current main still rejects the requested per-agent keys: the strict per-agent schema omits `verboseDefault` and `elevatedDefault`, while defaults, generated docs metadata, and runtime resolution still expose or consume those values; open PR #74643 is the active unmerged closing implementation.

## What This Changes

The issue requests that per-agent `verboseDefault` and `elevatedDefault` be accepted by the config schema or else removed from the per-agent runtime/API surface.

## Best Possible Solution

Align the config contract by accepting both fields on `agents.list[]` with the same enums as `agents.defaults`, updating public types/help/generated schema/docs, and keeping parser-level regression coverage for accepted and rejected values; PR #74643 appears to be the active implementation path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `agents.list[]` is validated by strict `AgentEntrySchema`, and that schema omits both reported keys while defaults and runtime code still support them.

## Solution Assessment

Yes. Adding the missing per-agent schema/type/help/docs/test coverage is the narrowest maintainable fix because runtime and status code already consume the resolved defaults; removing that surface would be broader and more disruptive.
