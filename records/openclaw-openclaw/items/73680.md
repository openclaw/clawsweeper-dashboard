## Summary

Keep this issue open. Current main still validates `agents.list[]` through a strict `AgentEntrySchema` that omits `verboseDefault` and `elevatedDefault`, while related defaults/runtime surfaces still reference those concepts; the active closing implementation is open PR #74643, not merged code.

## What This Changes

The issue asks OpenClaw to align per-agent agent config validation and docs so `agents.list[].verboseDefault` and `agents.list[].elevatedDefault` are either accepted like adjacent per-agent defaults or removed from the exposed runtime/type surface.

## Best Possible Solution

Align the config contract by accepting the two per-agent fields with the same enums as `agents.defaults`, updating generated schema/help/docs/type surfaces, and adding parser-level regression tests for both accepted and rejected values.

Yes. A high-confidence static reproduction exists: `agents.list[]` is validated by strict `AgentEntrySchema`, and that schema omits both reported keys, so config validation rejects them as unrecognized; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes. Adding the missing schema/type/help/docs/test coverage is the narrowest maintainable path because current runtime and status surfaces already reference these defaults; removing those paths would be broader and more disruptive.
