## Summary

Keep this issue open. Current main still has the reported config-contract mismatch: strict per-agent validation omits `verboseDefault` and `elevatedDefault`, while defaults/runtime surfaces still recognize those values, and open PR #74643 is the active unmerged closing implementation.

## What This Changes

Allow or explicitly reconcile per-agent `verboseDefault` and `elevatedDefault` so `agents.list[]` entries match resolver behavior, public config types, generated metadata, and docs.

## Best Possible Solution

Review and land PR #74643, or an equivalent narrow fix, that aligns `AgentEntrySchema`, public config types, labels/help/generated schema, docs, and focused validation tests before closing this issue.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `agents.list` validates through strict `AgentEntrySchema`, and that schema still omits the two reported keys, so config validation rejects them as unrecognized. I did not run the CLI because this review is read-only.

## Solution Assessment

Yes. Accepting the two fields in the per-agent schema and documenting their precedence is the narrowest maintainable fix because resolver, directive, and status paths already treat them as per-agent-capable values.
