## Summary

Keep this issue open. Current main still rejects both reported per-agent keys through the strict `AgentEntrySchema`, and the remaining fix path should also verify `elevatedDefault` end to end because current main carries `verboseDefault` through the per-agent resolver but not `elevatedDefault`.

## What This Changes

Allow `agents.list[]` entries to define per-agent `verboseDefault` and `elevatedDefault` overrides, with schema, type, docs, and regression coverage aligned.

## Best Possible Solution

Land #74643 or an equivalent fix after ensuring schema, generated config metadata/help/docs, `AgentConfig`, `ResolvedAgentConfig`, resolver/runtime paths, and validation tests cover both fields.

## Reproduction Assessment

Yes. Static reproduction on current main is high-confidence: `agents.list[]` is validated by strict `AgentEntrySchema`, which omits both reported keys while defaults and runtime-adjacent code still expose at least part of the surface.

## Solution Assessment

Yes, with one caveat. Allowing the per-agent fields and aligning the schema, types, resolver, docs, generated metadata, and enum tests is narrower and more maintainable than removing the existing public/runtime surface.
