## Summary

Keep this issue open. Current `origin/main` still has the reported config-contract mismatch: `agents.list[]` is validated by a strict `AgentEntrySchema` that omits `verboseDefault` and `elevatedDefault`, while global defaults and parts of the runtime/default-resolution surface still expose those values; PR #74643 is the active unmerged implementation candidate.

## What This Changes

Allow per-agent `agents.list[]` entries to set `verboseDefault` and `elevatedDefault` instead of only supporting those keys under `agents.defaults`.

## Best Possible Solution

Land or replace PR #74643 with a narrow contract-alignment fix that accepts both keys on `agents.list[]`, updates generated schema/help/docs/types, preserves strict rejection for unknown keys, and adds parser-level regression coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `agents.list[]` uses strict `AgentEntrySchema`, and that schema lacks both reported keys while global defaults accept them.

## Solution Assessment

Yes. Adding the missing per-agent schema, metadata, docs, types, and regression tests is the narrowest maintainable direction; the elevated path should also be checked so accepted config is actually resolved per agent.
