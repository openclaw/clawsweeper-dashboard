## Summary

Keep this issue open. Current main still has the reported config-contract mismatch: `agents.list[]` is validated through strict `AgentEntrySchema`, which still omits `verboseDefault` and `elevatedDefault`, while defaults/runtime paths continue to expose those values. The linked open PR #74643 is the active closing implementation and should be reviewed, refreshed, landed, or replaced rather than closing the issue early.

## What This Changes

The issue requests accepting `verboseDefault` and `elevatedDefault` as per-agent `agents.list[]` config fields, or otherwise removing the existing per-agent runtime/type surface.

## Best Possible Solution

Align the config contract by accepting both fields on `agents.list[]` with the same enums as `agents.defaults`, update public type/help/generated schema/docs metadata, and keep parser-level regression coverage for accepted and rejected values. The existing PR #74643 is the natural vehicle if it can be refreshed and landed.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `agents.list[]` is validated through strict `AgentEntrySchema`, and that schema omits both reported keys while global defaults accept them. I did not run the CLI because this review was constrained to read-only inspection.

## Solution Assessment

Yes. Adding the missing schema/type/help/docs/test coverage is the narrowest maintainable fix because runtime/status code already consumes these resolved defaults; removing the per-agent runtime surface would be broader and more disruptive.
