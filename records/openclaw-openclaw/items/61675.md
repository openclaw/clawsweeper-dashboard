## Summary

Keep open. The automatic rollover memory gap is real on current main, but this PR is not merge-ready because contributor real behavior proof is missing and the patch still misses a valid daily rollover case where the freshness logic resets the session but the new hook blocks never fire.

## What This Changes

The PR adds daily/idle lazy reset hook dispatch, plugin before_reset calls, session-memory event filtering, docs, changelog text, and regression tests.

## Best Possible Solution

Land one maintainer-approved implementation that persists session-memory exactly once for automatic daily and idle rollovers, aligns reset reason calculation with freshness evaluation, preserves manual reset behavior, and documents the chosen hook contract.

## Reproduction Assessment

Yes, by source inspection. Configure an existing session to become stale by daily or idle policy and send the next inbound message; current main creates a new session and lifecycle hooks while bundled session-memory only accepts command:new and command:reset.

## Solution Assessment

No, not as-is. The PR is a plausible hook-contract direction, but it should first align lazy reset reason derivation with freshness evaluation and maintainers need to choose it over the overlapping command:reset/commandSource approach in https://github.com/openclaw/openclaw/pull/71792.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Align lazy reset reason with freshness evaluation:** `src/auto-reply/reply/session.ts:879-882`
  - body: The new hook blocks only run when `previousSessionEndReason` is truthy, but that helper still decides daily staleness with `entry.updatedAt`. Current freshness evaluation rolls daily sessions based on `sessionStartedAt`, so a session whose start is before the daily boundary but whose metadata `updatedAt` moved after it will be rolled over while this PR skips both `triggerInternalHook` and `before_reset`; session-memory remains unsaved for that valid daily rollover. Carry the stale reason from `evaluateSessionFreshness` or update `resolveStaleSessionEndReason` to use the same timestamps.
  - confidence: 0.88
