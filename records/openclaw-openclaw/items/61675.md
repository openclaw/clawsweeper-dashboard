## Summary

Keep open. Current main still rotates stale daily/idle sessions without emitting the internal command hook or before_reset hook that bundled session-memory needs, while this PR addresses that missing path with focused code, docs, changelog, and regression coverage. I found no blocking review findings; the remaining action is maintainer review and canonical choice against overlapping open PRs before landing.

## What This Changes

The PR adds lazy daily/idle reset hook dispatch in session initialization, exports reset transcript helpers, widens bundled session-memory to accept command:daily and command:idle, updates hooks docs/changelog, and adds focused regression tests.

## Best Possible Solution

Land one canonical implementation that makes lazy daily/idle rollovers preserve memory with the same hook semantics as manual resets, documents the supported events, and then close or supersede duplicate auto-reset memory reports and competing PRs against the shipped fix.

## Reproduction Assessment

Yes. Configure an existing session to become stale by daily or idle policy and send the next message; current main rotates the session in initSessionState, but the only reset hook fallback in get-reply.ts is limited to explicit /new and /reset commands.

## Solution Assessment

Yes. The PR is a narrow maintainable fix if maintainers accept command:daily and command:idle as the compatibility path: it reuses existing hook infrastructure, avoids adding a new SDK event, keeps dispatch fire-and-forget, and updates bundled hook metadata/docs/tests.
