## Summary

Keep open. The PR targets a real automatic daily/idle rollover persistence gap, but it still needs real behavior proof, conflict resolution, and a maintainer decision on the public hook contract before merge.

## What This Changes

The PR adds lazy daily/idle rollover dispatch for internal command hooks and before_reset, widens bundled session-memory docs/metadata/tests to accept daily/idle actions, and adds a changelog entry.

## Best Possible Solution

Land one canonical implementation that persists session-memory exactly once for automatic daily and idle rollovers, derives the reset reason from the same freshness inputs, preserves manual reset behavior, and documents the approved hook contract.

## Reproduction Assessment

Yes, by source inspection. Current main captures and archives a stale previous session, then fires session_end/session_start only, while bundled session-memory only persists command:new and command:reset.

## Solution Assessment

No, not as-is. The direction is plausible, but the hook reason must be aligned with the freshness evaluator and maintainers need to choose this public command:daily/idle contract over the overlapping alternative.
