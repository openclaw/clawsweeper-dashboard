## Summary

Keep open: current main and the latest shipped release still route `~/...` memory `extraPaths` through workspace-relative resolution, so the reported invisible-vault behavior remains source-reproducible. The focused fix PR was closed unmerged, and the remaining related open PR adds doctor diagnostics rather than fixing the runtime normalizer.

## What This Changes

The issue requests that `agents.defaults.memorySearch.extraPaths` entries beginning with `~/` resolve against the user home directory instead of each agent workspace.

## Best Possible Solution

Land a single shared runtime normalizer fix that expands a leading home prefix before absolute/workspace-relative resolution, preserves the existing safety checks, and updates focused tests plus the memory config docs.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main routes `~/OneDrive/MyVault` through `path.resolve(workspaceDir, value)`, and Node confirms `path.isAbsolute('~/...')` is false. I did not run a live memory indexing command in this read-only checkout.

## Solution Assessment

Yes. The narrow maintainable fix is to expand the home prefix in the shared runtime normalizer before `path.isAbsolute()`, while preserving absolute paths, workspace-relative paths, dedupe, and existing symlink/containment protections.
