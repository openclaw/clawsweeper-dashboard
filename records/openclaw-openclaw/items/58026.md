## Summary

Keep open: current main and the latest release still resolve `~/...` builtin memory `extraPaths` under the workspace, while the focused fix PRs for this root cause were closed unmerged and the remaining open related PR only adds a doctor warning.

## What This Changes

The issue reports that builtin memory search `extraPaths` with `~/` are resolved under the agent workspace instead of the user's home directory.

## Best Possible Solution

Expand a leading home prefix in the canonical builtin `normalizeExtraMemoryPaths()` path before the absolute/workspace-relative branch, then add focused regression coverage and keep doctor diagnostics aligned with that shared runtime behavior.

## Reproduction Assessment

Yes, by source inspection and the direct Node path contract: current main routes `~/OneDrive/MyVault` through the workspace-relative branch used by builtin list/read paths. I did not run a live memory index command in this read-only checkout.

## Solution Assessment

Yes. The narrow maintainable fix is to expand a leading home prefix in `normalizeExtraMemoryPaths()` before checking `path.isAbsolute()`, while preserving existing absolute, workspace-relative, dedupe, and symlink-safety behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/58026 on current main. Builtin memory `agents.defaults.memorySearch.extraPaths` currently treats `~/...` as workspace-relative because `normalizeExtraMemoryPaths()` only checks `path.isAbsolute()` before resolving against `workspaceDir`; QMD already expands `~` through `resolveUserPath()`. Update the canonical builtin normalizer in `packages/memory-host-sdk/src/host/internal.ts` to expand a leading home prefix before the absolute/workspace-relative branch, preserving trimming, blank filtering, dedupe, absolute paths, workspace-relative paths, and read-file symlink containment. Add focused regression coverage in `packages/memory-host-sdk/src/host/internal.test.ts` and, if needed, `packages/memory-host-sdk/src/host/read-file.test.ts` so `~/...` extra paths work for normalization and read/list behavior. Do not add a doctor-only resolver that diverges from runtime behavior; if docs are updated, use public `memorySearch.extraPaths` wording and call it plugin-facing/user-facing. Include a changelog entry if the repair lane is preparing a user-facing fix PR under repo policy. Related refs: closed unmerged fix PRs https://github.com/openclaw/openclaw/pull/58047 and https://github.com/openclaw/openclaw/pull/58050; related doctor warning PR https://github.com/openclaw/openclaw/pull/46837 should stay aligned but is not the runtime fix.
