Keep open: current main still sends `~/...` memory extra paths through the workspace-relative branch, so the reported vault invisibility remains source-reproducible. The earlier focused fix PR is closed unmerged, and the remaining open related PR only covers doctor diagnostics rather than the runtime normalizer.

## What This Changes

The issue asks for `agents.defaults.memorySearch.extraPaths` entries beginning with `~/` to resolve against the user's home directory instead of each agent workspace.

## Best Possible Solution

Expand a leading home prefix in the canonical builtin `normalizeExtraMemoryPaths()` path before the absolute/workspace-relative branch, add regression coverage for list/read/status behavior, and keep doctor diagnostics aligned with the shared runtime normalizer.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main routes `~/OneDrive/MyVault` through `path.resolve(workspaceDir, value)`, and Node confirms `path.isAbsolute('~/...')` is false. I did not run a live memory index command in this read-only checkout.

## Solution Assessment

Yes. The narrow maintainable fix is to expand the home prefix in the shared runtime normalizer before `path.isAbsolute()`, while preserving absolute paths, workspace-relative paths, dedupe, and the existing symlink/containment protections.
