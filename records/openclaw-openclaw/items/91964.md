## Label Justifications

- `P1`: The canonical bug can silently truncate an active external-channel response when a user sends a same-session follow-up.
- `impact:message-loss`: Later visible reply payloads can be suppressed before reaching the originating plugin or channel.

## Summary

This follow-up restates the same source-delivery loss, root cause, invariant, and repair already tracked by the original open bug; the focused open PR is attached to that canonical issue, so a second issue would split one implementation discussion.

## What This Changes

Require queued same-session inbound work to preserve the active turn’s source-delivery freshness until the newer turn begins visible delivery.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/91914 as the single canonical report and review the focused implementation in https://github.com/openclaw/openclaw/pull/91963, preserving this issue’s serial-versus-interrupt invariant in the regression coverage and PR rationale.

## Reproduction Assessment

Yes at source level with high confidence: current main starts a shared freshness generation at buffered-dispatch entry, while later delivery checks can reject the older turn after a newer dispatch arrives. The linked PR also reports real plugin-backed gateway validation, though this read-only review did not independently run it.

## Solution Assessment

Yes: lazily creating the fence at first visible-delivery attempt is the narrowest coherent repair because it preserves freshness cancellation for genuinely delivering newer turns without letting merely queued work supersede the active turn.
