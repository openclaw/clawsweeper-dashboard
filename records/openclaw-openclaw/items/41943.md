## Summary

Keep open: the Feishu HTTPS proxy bug still appears valid on current main, but this PR is not merge-ready because it forces proxy routing without target-aware NO_PROXY handling and writes Axios-only options through the narrower Lark SDK request type.

## What This Changes

The PR changes the Feishu client wrapper to attach an HTTPS proxy agent and disable Axios built-in proxy handling for Lark SDK HTTP requests when proxy environment variables are set.

## Best Possible Solution

Port a target-aware Feishu HTTP proxy fix on current main that preserves NO_PROXY, keeps the Lark SDK boundary typed, uses plugin-safe dependency ownership, and adds focused regression coverage plus changelog.

## Reproduction Assessment

Yes, source-backed rather than live-run: the Feishu probe path calls `createFeishuClient().request(...)`, and current main's HTTP wrapper only injects timeouts. The linked bug's proxy failure path remains plausible because HTTP API calls still lack explicit HTTPS CONNECT proxy handling.

## Solution Assessment

No. The PR targets the right Feishu-owned surface, but the maintainable fix should resolve the effective Feishu/Lark request URL, honor NO_PROXY, bridge Axios-specific options with a typed adapter, and align package metadata if a direct proxy dependency is used.
