## Summary

Keep open: current main still lacks Feishu HTTP HTTPS_PROXY handling, but this branch needs rework before merge because it forces proxy routing without NO_PROXY targeting, crosses the Lark/Axios type boundary, needs dependency alignment after current-main changes, and has no real after-fix proxy proof.

## What This Changes

The PR modifies the Feishu client to create an `HttpsProxyAgent` from proxy environment variables and pass it to Lark SDK HTTP requests while disabling Axios proxy handling.

## Best Possible Solution

Rebase or replace the branch with target-aware Feishu HTTP proxy handling at the Lark SDK `httpInstance` seam, honoring `NO_PROXY`, preserving timeout/User-Agent behavior, using the current shared dependency surface, and adding focused tests plus real proxy proof.

## Reproduction Assessment

Yes, source-backed rather than live-run: current main sends Feishu probe/token/API HTTP calls through the Lark Axios-backed default instance without an HTTP proxy agent policy, and the linked reports provide concrete proxy env steps plus the `400 The plain HTTP request was sent to HTTPS port` failure mode.

## Solution Assessment

No. The PR targets the right Feishu-owned seam, but the maintainable fix should be target-aware, honor `NO_PROXY`, keep Axios-only options behind a typed adapter, and align with current plugin dependency ownership.

  - confidence: 0.82
