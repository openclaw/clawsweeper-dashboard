## Summary

Keep open. The PR targets a real Feishu HTTP proxy bug, but the branch is not mergeable as-is because it unconditionally forces proxy routing, bypasses NO_PROXY, crosses the Lark SDK request type contract, is stale against current proxy/dependency packaging, and lacks after-fix real behavior proof.

## What This Changes

The branch modifies Feishu's Lark SDK HTTP wrapper to build an HttpsProxyAgent from proxy environment variables and attach `httpsAgent` plus `proxy: false` to every wrapped request.

## Best Possible Solution

Rebase or replace the branch with target-aware Feishu HTTP proxy handling at the Lark SDK `httpInstance` seam, preserving timeout/User-Agent behavior, honoring NO_PROXY, using the current proxy dependency surface, and adding focused tests plus real proxy proof.

## Reproduction Assessment

Yes, source-backed rather than live-run: current main sends Feishu probe/token/API HTTP calls through the Lark Axios-backed SDK path without an HTTP proxy policy, and linked reports provide concrete proxy env steps plus the `400 The plain HTTP request was sent to HTTPS port` failure mode.

## Solution Assessment

No. The PR targets the right Feishu-owned seam, but the maintainable fix should be target-aware, honor NO_PROXY, keep Axios-only options behind a typed adapter or valid dependency contract, and align with current plugin dependency ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Honor NO_PROXY before forcing the proxy agent:** `extensions/feishu/src/client.ts:57-60`
  - body: When `proxyAgent` exists, every Lark HTTP request gets `httpsAgent` and `proxy: false`, bypassing Axios' normal `proxy-from-env` and NO_PROXY selection. Resolve the effective Feishu/Lark target URL and only attach the agent when that request should actually use the proxy.
  - confidence: 0.93
- **[P2] Keep Axios options out of Lark request types:** `extensions/feishu/src/client.ts:58-60`
  - body: `Lark.HttpRequestOptions` does not declare `httpsAgent` or `proxy`, so assigning those fields directly to `result` crosses the SDK type contract and should fail strict TypeScript. Keep the Lark options typed and add a narrow Axios-backed adapter or explicit handoff type.
  - confidence: 0.9
- **[P2] Use the current proxy dependency surface:** `extensions/feishu/src/client.ts:10-17`
  - body: This helper constructs `HttpsProxyAgent` directly, but current main moved Feishu proxy construction to the shared `proxy-agent` helper and no longer declares `https-proxy-agent` in `@openclaw/feishu`. A rebased fix should use the current helper seam or add plugin-owned runtime dependency metadata.
  - confidence: 0.82
- **[P3] Refresh proxy routing when the cached client is reused:** `extensions/feishu/src/client.ts:52`
  - body: `proxyAgent` is captured when the HTTP instance is constructed, while the cached client key tracks credentials, domain, and timeout only. If proxy env changes during the process lifetime, cached clients keep stale routing; resolve eligibility per request or include proxy state in the cache comparison.
  - confidence: 0.76
