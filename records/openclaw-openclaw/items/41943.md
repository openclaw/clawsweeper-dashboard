## Summary

Keep open: the PR targets a real Feishu HTTP proxy bug, but the branch is not safe to merge as-is because it unconditionally forces Axios proxy-agent routing, bypasses target/NO_PROXY and managed Proxyline policy, writes Axios-only fields outside the Lark SDK request-options contract, and still lacks after-fix real behavior proof.

## What This Changes

The PR changes the Feishu client HTTP wrapper to create an HTTPS proxy agent from proxy environment variables and attach `httpsAgent` plus `proxy: false` to every Lark SDK HTTP call.

## Best Possible Solution

Implement a Feishu-owned Lark HTTP adapter that prevents Axios proxy-env misrouting while preserving managed Proxyline egress, target/NO_PROXY eligibility, timeout/User-Agent behavior, focused tests, docs, and redacted real proxy proof.

## Reproduction Assessment

Yes, source-backed: Feishu probe/startup reaches `probeFeishu` -> `createFeishuClient` -> the Lark Axios-backed HTTP instance while proxy env or managed proxy routing is active, and linked reports provide 400/plain-HTTP-to-HTTPS-port plus ECONNRESET logs. I did not run live Feishu credentials through a local proxy in this read-only review.

## Solution Assessment

No. The PR hits the right Feishu-owned seam, but the submitted approach unconditionally forces proxy routing, bypasses Axios NO_PROXY/Proxyline policy, crosses the Lark SDK type contract, and needs a current-main rebase plus real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Honor target proxy eligibility before forcing routing:** `extensions/feishu/src/client.ts:57-60`
  - body: When `proxyAgent` exists, every Lark HTTP request gets `httpsAgent` and `proxy: false`, so Axios no longer applies env-derived target exclusions such as NO_PROXY. Resolve the effective Feishu/Lark target and only attach proxy routing when that request is proxy-eligible under the chosen policy.
  - confidence: 0.93
- **[P2] Keep Axios-only fields behind a typed adapter:** `extensions/feishu/src/client.ts:58-60`
  - body: `Lark.HttpRequestOptions` in `@larksuiteoapi/node-sdk@1.64.0` does not declare `httpsAgent` or `proxy`, so assigning those fields directly crosses the SDK type contract. Keep the Lark request options typed and add a narrow Axios handoff type or adapter where those fields are valid.
  - confidence: 0.9
- **[P2] Use the current proxy helper surface:** `extensions/feishu/src/client.ts:10-17`
  - body: The branch constructs proxy behavior directly from HTTP(S)_PROXY env vars, but current main routes plugin proxy construction through the shared Proxyline-backed helper and Feishu no longer declares its own `https-proxy-agent` dependency. Rebase onto that helper seam or add valid plugin-owned dependency metadata.
  - confidence: 0.84
- **[P3] Refresh proxy routing when reusing cached clients:** `extensions/feishu/src/client.ts:52`
  - body: `proxyAgent` is captured when the HTTP instance is constructed, while cached clients are reused based on credentials, domain, and timeout only. If proxy state changes during the process lifetime, cached clients keep stale routing; resolve eligibility per request or include proxy state in the cache comparison.
  - confidence: 0.76
