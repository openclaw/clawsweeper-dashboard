## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks Feishu HTTP proxy handling for the Lark SDK request path, while this patch forces proxy routing whenever proxy env exists and can bypass operator NO_PROXY exclusions.

## What This Changes

The PR modifies `extensions/feishu/src/client.ts` so Feishu/Lark SDK HTTP calls add an `HttpsProxyAgent` and `proxy: false` when proxy environment variables are set.

## Best Possible Solution

Revise the Feishu-owned client wrapper on current main so each HTTP request resolves its effective target URL, injects Axios proxy options only when the target is proxy-eligible, preserves NO_PROXY behavior, keeps cache/proxy state deliberate, and adds focused regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. The linked report gives a concrete path: set `HTTP_PROXY`/`HTTPS_PROXY` and run `openclaw channels status --probe`; current main routes that probe through `probeFeishu` and the timeout-only Feishu HTTP wrapper.

## Solution Assessment

No. The PR targets the right owner file, but the safest fix should use the existing target-aware proxy helpers and explicitly bridge Axios-specific options instead of forcing proxy options whenever an env var exists.

  - body: `Lark.HttpRequestOptions` is used as the type for `result`, but the patch assigns Axios-specific fields such as `httpsAgent` and `proxy`. Use a narrow local adapter/intersection type for the Axios-backed options and cast only at the Lark boundary.
  - confidence: 0.82
- **[P3] Include proxy state in the cached client decision:** `extensions/feishu/src/client.ts:52`
  - body: The proxy agent is captured when the cached Feishu client is created, while the cache decision still compares only credentials, domain, and timeout. If proxy env changes during process lifetime, reused clients keep stale routing; include resolved proxy state in the cache key or resolve per request.
  - confidence: 0.78
