## Summary

Keep open: current main still lacks a Feishu REST/token HTTP proxy policy, but this PR is not merge-ready because it unconditionally forces Axios proxy options, conflicts with current main, and has no after-fix real Feishu proxy proof.

## What This Changes

The PR changes `extensions/feishu/src/client.ts` to attach an `HttpsProxyAgent` and `proxy: false` to the Lark SDK HTTP wrapper when proxy environment variables are set.

## Best Possible Solution

Replace or rebase the branch with a target-aware Feishu HTTP adapter at the Lark `httpInstance` seam that preserves timeout/User-Agent behavior, honors proxy eligibility, uses the current helper/dependency surface, adds focused tests, and includes redacted real proxy proof.

## Reproduction Assessment

Yes, source-backed: current main sends Feishu REST/token/probe calls through the Lark Axios-backed SDK path without a Feishu HTTP proxy policy, and linked reports provide concrete proxy-env failures. I did not run a live Feishu/proxy setup in this read-only review.

## Solution Assessment

No. The PR targets the right Feishu-owned seam, but the submitted patch unconditionally forces proxy routing, bypasses target/NO_PROXY decisions, crosses the SDK type contract, and needs a current-main rebase plus real behavior proof.

  - body: When `proxyAgent` exists, every Lark HTTP request gets `httpsAgent` and `proxy: false`, so Axios no longer applies target-based `NO_PROXY` exclusions. Resolve the effective Feishu/Lark target URL and only attach proxy routing when that request should use it.

  - confidence: 0.82
