## Summary

Keep open: the Feishu HTTPS proxy bug still appears valid on current main and this PR targets the right owner-owned HTTP seam, but the submitted patch is not mergeable without rebase, target-aware proxy policy, typed Axios handoff, tests, and contributor real behavior proof.

## What This Changes

The PR adds `https-proxy-agent` use inside Feishu's Lark SDK HTTP wrapper so Feishu HTTP requests use an HTTP CONNECT proxy when proxy environment variables are set.

## Best Possible Solution

Replace or rebase the branch with a target-aware Feishu HTTP adapter at the Lark `httpInstance` seam that preserves timeout and User-Agent behavior, honors proxy eligibility, uses the current proxy helper/dependency surface, adds focused tests, and includes redacted real proxy proof.

## Reproduction Assessment

Yes, source-backed rather than live reproduced here: `probeFeishu` reaches `createFeishuClient`, current main's HTTP wrapper only injects timeout, and the linked reports provide concrete proxy-env failures such as `400 The plain HTTP request was sent to HTTPS port` and ECONNRESET.

## Solution Assessment

No. The PR targets the right Feishu-owned seam, but the patch unconditionally forces proxy routing, crosses the Lark SDK option type contract, uses stale helper/dependency state, and lacks real proxy proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Honor target proxy eligibility before injecting the agent:** `extensions/feishu/src/client.ts:57-60`
  - body: When any proxy env var exists, this wraps every Lark HTTP request with `httpsAgent` and `proxy = false`. That disables Axios/proxy-from-env target handling, including `NO_PROXY`, so hosts configured to bypass the proxy can still be tunneled; resolve the effective Feishu/Lark target URL and only attach proxy routing when that request is eligible.
  - confidence: 0.93
- **[P2] Keep Axios-only options behind a typed adapter:** `extensions/feishu/src/client.ts:58-60`
  - body: `Lark.HttpRequestOptions` in the current SDK does not declare `httpsAgent` or `proxy`, so assigning those fields directly to `result` violates the SDK type surface under strict TypeScript. Keep the Lark request options typed and cast only at a narrow Axios-backed handoff or adapter boundary.
  - confidence: 0.9
- **[P2] Use the current proxy helper surface:** `extensions/feishu/src/client.ts:10-17`
  - body: The branch repurposes the old direct `https-proxy-agent` helper, but current main moved Feishu proxy construction to `resolveAmbientNodeProxyAgent` from `openclaw/plugin-sdk/extension-shared` and no longer declares `https-proxy-agent` in `@openclaw/feishu`. Rebase onto that helper seam or restore valid plugin-owned runtime dependency metadata.
  - confidence: 0.84
- **[P3] Refresh proxy routing when cached clients are reused:** `extensions/feishu/src/client.ts:52`
  - body: `proxyAgent` is captured when the HTTP instance is created, while cached clients are reused by credentials, domain, and timeout only. If the proxy environment changes in-process, Feishu keeps stale routing; resolve eligibility per request or include proxy state in the cache comparison.
  - confidence: 0.76
