## Summary

Keep open: current main still lacks the Feishu HTTP proxy fix, but this stale/conflicting patch is not merge-ready because it bypasses `NO_PROXY`, crosses the Lark SDK request type boundary, is not aligned with current plugin dependency ownership, and has no after-fix real behavior proof.

## What This Changes

The PR changes `extensions/feishu/src/client.ts` so Feishu Lark SDK HTTP calls attach an HTTPS proxy agent and set Axios `proxy: false` when proxy environment variables are present.

## Best Possible Solution

Port or replace the branch with target-aware Feishu HTTP proxy handling at the Lark SDK `httpInstance` seam, honoring `NO_PROXY`, keeping the Axios-only options behind a typed adapter, preserving plugin dependency ownership, and adding focused client/probe coverage plus real proxy proof.

## Reproduction Assessment

Yes, source-backed rather than live-run: Feishu probe and token/API paths go through `createFeishuClient`, and current main's wrapper does not add an HTTP proxy agent or disable Axios proxy handling. The linked reports provide concrete proxy env steps and the `400 The plain HTTP request was sent to HTTPS port` failure mode.

## Solution Assessment

No. The PR targets the right Feishu-owned surface, but the safe fix should resolve the effective request URL/domain, honor `NO_PROXY`, keep the Lark/Axios type boundary explicit, and align dependency metadata with current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor NO_PROXY before injecting the proxy agent:** `extensions/feishu/src/client.ts:57-60`
  - body: When `proxyAgent` exists, every Lark HTTP request gets `httpsAgent` and `proxy: false`, which bypasses `NO_PROXY`/`no_proxy` exclusions. Resolve the effective target URL or domain and only attach the agent when the current env proxy policy says that request should be proxied.
  - confidence: 0.93
- **[P2] Keep Axios-only options out of Lark request types:** `extensions/feishu/src/client.ts:58-60`
  - body: The PR assigns `httpsAgent` and `proxy` to a `Lark.HttpRequestOptions` object, but the Lark SDK type does not declare those Axios fields. Keep the Lark options typed and use a narrow local Axios-backed adapter or intersection at the handoff.
  - confidence: 0.9
- **[P2] Keep the Feishu proxy dependency explicit:** `extensions/feishu/src/client.ts:2`
  - body: Current main's Feishu package no longer declares `https-proxy-agent` after WS proxy handling moved through the shared helper. A current-main port should reuse that helper seam or add plugin-local runtime dependency metadata so standalone installs do not rely on incidental host resolution.
  - confidence: 0.82
- **[P3] Refresh proxy choice with the cached client:** `extensions/feishu/src/client.ts:52`
  - body: `proxyAgent` is captured when the HTTP instance is constructed, while the cached client key only tracks credentials, domain, and timeout. If proxy env changes during the process lifetime, cached clients keep stale routing; resolve eligibility per request or include proxy state in the cache comparison.
  - confidence: 0.76
