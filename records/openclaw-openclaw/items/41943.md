## Summary

Keep open for human review: the branch targets a real Feishu proxy bug, but the proposed patch is stale against current main and still has blocking correctness, dependency, security-policy, and real-behavior-proof gaps.

## What This Changes

The PR modifies the Feishu Lark SDK HTTP wrapper to create an `HttpsProxyAgent` from proxy environment variables and attach it to wrapped Feishu HTTP requests.

## Best Possible Solution

Replace or rebase the branch with target-aware Feishu HTTP proxy handling at the Lark SDK `httpInstance` seam, preserving timeout/User-Agent behavior, honoring NO_PROXY, using the current proxy helper surface, and adding focused tests plus real proxy proof.

## Reproduction Assessment

Yes, source-backed: current main sends Feishu probe/token/API HTTP calls through the Lark Axios-backed SDK path without an HTTP proxy policy, and linked reports provide concrete proxy env steps plus `400 The plain HTTP request was sent to HTTPS port` and ECONNRESET failure modes.

## Solution Assessment

No. The PR targets the right Feishu-owned seam, but the maintainable fix should be target-aware, honor NO_PROXY, keep Axios-only options behind a typed adapter or valid dependency contract, and align with current plugin dependency ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

  - body: When `proxyAgent` exists, every Lark HTTP request gets `httpsAgent` and `proxy: false`, so Axios no longer applies target-based `NO_PROXY` exclusions. Resolve the effective Feishu/Lark target URL and only attach the proxy agent when that request should use the proxy.
  - confidence: 0.93

  - body: `Lark.HttpRequestOptions` does not declare `httpsAgent` or `proxy`, so assigning those fields directly crosses the SDK type contract and should fail strict TypeScript. Keep the Lark request options typed and add a narrow Axios-backed adapter or explicit handoff type.
  - confidence: 0.9
- **[P2] Use the current proxy dependency surface:** `extensions/feishu/src/client.ts:10-17`
  - body: The branch constructs `HttpsProxyAgent` directly, but current main moved Feishu proxy construction to the shared proxyline-backed helper and `@openclaw/feishu` no longer declares `https-proxy-agent`. A rebased fix should use the current helper seam or add valid plugin-owned runtime dependency metadata.
  - confidence: 0.82
- **[P3] Refresh proxy routing when reusing cached clients:** `extensions/feishu/src/client.ts:52`
  - body: `proxyAgent` is captured when the HTTP instance is constructed, while the cached client key tracks credentials, domain, and timeout only. If proxy env changes during the process lifetime, cached clients keep stale routing; resolve eligibility per request or include proxy state in the cache comparison.
  - confidence: 0.76

## Security Review

Status: needs_attention
