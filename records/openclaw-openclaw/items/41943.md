## Summary

Keep this PR open. Current main still lacks Feishu/Lark Axios HTTP proxy tunneling for ambient proxy env vars, but this patch is not merge-ready because it forces proxy use without NO_PROXY eligibility checks, writes Axios-only options through the narrower Lark request type, captures proxy state in cached clients, and adds no focused HTTP proxy regression coverage.

## What This Changes

The PR modifies `extensions/feishu/src/client.ts` to create an `HttpsProxyAgent` from ambient proxy env vars and inject it plus `proxy: false` into Feishu/Lark Axios HTTP requests.

## Best Possible Solution

Revise the Feishu-owned client wrapper on current main so HTTP requests resolve the effective Feishu/Lark target URL, apply proxy agents only when `shouldUseEnvHttpProxyForUrl` says the target is proxy-eligible, bridge Axios-specific options through a narrow local adapter type, handle cached proxy state deliberately, declare any direct runtime dependency in the Feishu plugin package, and add targeted regression tests.

## Reproduction Assessment

Yes. The linked report gives a concrete path: set `HTTP_PROXY`/`HTTPS_PROXY` and run `openclaw channels status --probe`; current main still sends that probe through the Feishu `client.request()` path backed by a timeout-only Lark HTTP wrapper.

## Solution Assessment

No. The PR targets the right owner file, but the safest maintainable fix should reuse OpenClaw's target-aware proxy helpers, preserve NO_PROXY behavior, and type the Axios option bridge explicitly instead of forcing proxy options whenever an env var exists.
