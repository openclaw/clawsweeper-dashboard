## Label Justifications

- `P2`: The issue is a concrete Feishu CLI media reliability problem with limited channel-specific blast radius.
- `impact:message-loss`: The Feishu path can return a normal send result after the requested media upload fails, leaving the attachment undelivered.
- `impact:security`: The requested behavior touches local file-read allowlists and whether a channel-specific config may widen media roots.

## Summary

Keep open: current main and the latest release have addressed part of the local-root propagation story, but Feishu still converts media upload failures into a success-shaped text fallback and the reported `channels.feishu.mediaLocalRoots` key is not a current Feishu config surface.

## What This Changes

The issue asks for Feishu CLI media sends to honor the intended local media roots or clearly fail/report when media upload cannot happen.

## Best Possible Solution

Keep the generic `tools.fs` local-media policy as the source of truth unless maintainers explicitly approve a Feishu-specific config surface, and make local media upload failure produce either a failed command or an explicit partial-success result instead of a plain success.

## Reproduction Assessment

Yes, source-level. Current Feishu outbound catches media-send errors and returns a fallback text message, while the CLI formats plugin send results as success; the exact live Feishu upload path was not run in this read-only review.

## Solution Assessment

No, not as an automatic repair. The narrow code path is clear, but the durable fix depends on maintainer direction for fail-hard versus partial-success semantics and whether to add or reject a Feishu-specific media roots config key.
