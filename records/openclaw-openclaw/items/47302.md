## Summary

Keep open: current main does not include `doctor --check-config`, and the latest PR head still lets important broken configs report as warning or OK; the supplied real behavior proof is also too narrow for the configured-validation behavior being added.

## What This Changes

The PR adds an `openclaw doctor --check-config` CLI flag, a new deep config diagnostic module, and formatter tests for schema, model, fallback, TTS, and channel validation.

## Best Possible Solution

Land a revised `--check-config` that fails malformed or unavailable env/model/fallback inputs, adds subsystem and CLI coverage, and includes redacted real CLI proof for passing and failing configs.

## Reproduction Assessment

Yes for the review blockers: configs with an unset `${OPENAI_API_KEY}`, `agents.defaults.model: "anthropic/"`, and a fallback like `missing-provider/k2p5` follow clear source paths to warning/OK output. This read-only review did not execute those configs.

## Solution Assessment

No. The feature direction is useful, but the current implementation is not the best mergeable form because it does not fail closed for several broken configured states that the command is meant to catch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Fail unresolved env substitution warnings:** `src/commands/doctor-check-config.ts:68-73`
  - body: `readConfigFileSnapshot` preserves missing `${VAR}` placeholders as warnings, but this path emits them as `warn` while `hasFailures` only checks `fail`. With an unset TTS API key env var, the placeholder remains a non-empty string, so the provider can still look configured and the command can exit 0.
  - confidence: 0.9
- **[P2] Fail malformed configured primary refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: For a configured primary such as `anthropic/`, `parseModelRef(rawConfigured, DEFAULT_PROVIDER)` returns null, so the silent-fallback warning is skipped after `resolveConfiguredModelRef` falls back to the default. The command can report the default as OK even though the configured value is invalid.
  - confidence: 0.9
- **[P2] Validate fallback refs against availability:** `src/commands/doctor-check-config.ts:279-280`
  - body: `resolveModelRefFromString` accepts syntactic `provider/model` refs, so a fallback like `missing-provider/k2p5` is counted as validated without proving the provider/model is available under the current config. That misses the undefined-fallback-provider case this PR says it covers.
  - confidence: 0.84
