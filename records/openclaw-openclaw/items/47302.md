## Summary

Keep open: current main still lacks this feature, and the latest PR head still has source-provable false-success paths for broken config values plus insufficient real behavior proof for the configured validation flows.

## What This Changes

The PR adds an `openclaw doctor --check-config` CLI option, a deep config diagnostic module, and formatter tests for schema/model/fallback/TTS/channel output.

## Best Possible Solution

Land a revised flag that fails closed for broken configured inputs, validates fallback refs through runtime/catalog status without requiring explicit `models.providers`, adds behavior coverage, updates CLI docs, and includes redacted CLI proof for both passing and failing configs.

## Reproduction Assessment

Yes, for the PR blockers by source inspection: missing env placeholders, malformed primary refs, and parser-only fallback refs follow clear paths to warning or OK output. I did not execute those configs in this read-only review.

## Solution Assessment

No. The feature direction is useful, but this implementation is not the best mergeable form because it can still report success for broken configured values that the command is intended to catch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Fail unresolved env substitutions:** `src/commands/doctor-check-config.ts:72`
  - body: `readConfigFileSnapshot` preserves missing `${VAR}` placeholders as warnings, but this branch records them as `warn` and `hasFailures` only checks `fail`. For a configured TTS key such as `${OPENAI_API_KEY}`, provider config can still see a non-empty placeholder and the command can exit 0 while the feature is unavailable.
  - confidence: 0.9
- **[P2] Fail malformed primary model refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: For a configured primary like `anthropic/`, `parseModelRef` returns null, this condition skips the warning, and the command reports the fallback default as `Primary model` OK. That hides an invalid configured model instead of failing the deep config check.
  - confidence: 0.9
- **[P2] Validate fallback refs against runtime status:** `src/commands/doctor-check-config.ts:279-280`
  - body: `resolveModelRefFromString` only proves that a fallback string parses or aliases, so an unknown or unusable `provider/model` ref is counted in the validated total without checking catalog/runtime status. This misses broken fallback chains the command is supposed to catch.
  - confidence: 0.84
