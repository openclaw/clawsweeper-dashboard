## Summary

Keep open. The PR is aimed at a useful doctor/config diagnostic gap and the author already addressed several early review comments, but the current head is not mergeable against current main: it imports a TTS symbol that no longer exists and still has correctness issues around unresolved config inputs, catalog side effects, malformed primary refs, and fallback availability validation.

## What This Changes

The PR adds an `openclaw doctor --check-config` option, a new deep config validation module, and formatter tests for schema, model, fallback, TTS, and channel diagnostics.

## Best Possible Solution

Land a rebased deep-config doctor mode that uses current registry-backed TTS APIs, keeps catalog/config inspection read-only, treats active unresolved config inputs as failures, validates primary and fallback model refs against runtime-equivalent availability, documents the new flag, adds behavior tests, and includes the required changelog entry.

## Reproduction Assessment

Yes. The highest-confidence reproduction is static: applying the PR diff to current main introduces an import of `TTS_PROVIDERS`, which current `src/tts/tts.ts` does not export. The remaining diagnostic issues are traceable through current `loadModelCatalog`, `resolveConfiguredModelRef`, and `resolveModelRefFromString` behavior.

## Solution Assessment

No. The feature direction is useful for #45552 and #20909, but this patch is not the best current implementation until it uses the current TTS/provider contracts, avoids write-capable catalog loading, and validates the failure modes the runtime actually hits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Use the registry-backed TTS provider order:** `src/commands/doctor-check-config.ts:18`
  - body: `TTS_PROVIDERS` is not exported by current `../tts/tts.js`, so this new module fails typecheck/build once included. Use the registry-backed TTS helpers, such as `resolveTtsProviderOrder`, instead of importing a removed static list.
  - confidence: 0.98
- **[P2] Fail unresolved config inputs that affect active checks:** `src/commands/doctor-check-config.ts:72`
  - body: Snapshot warnings are emitted as `warn`, and `hasFailures` only considers `fail`. An active credential left as an unresolved `${ENV_VAR}` placeholder can still look configured to later checks and exit 0, missing the broken config this mode is meant to catch.
  - confidence: 0.9
- **[P2] Load the model catalog in read-only mode:** `src/commands/doctor-check-config.ts:183`
  - body: `loadModelCatalog({ config: cfg })` uses the default writable path and can call `ensureOpenClawModelsJson`. A doctor validation mode intended for CI gating should pass `readOnly: true` so checking config does not mutate user state.
  - confidence: 0.91
- **[P2] Reject malformed primary model refs before fallback:** `src/commands/doctor-check-config.ts:152-153`
  - body: When `agents.defaults.model` is a malformed qualified ref such as `anthropic/`, `parseModelRef` returns null and this branch falls through to reporting the resolver's default fallback as healthy. That misses a broken explicitly configured primary model.
  - confidence: 0.86
- **[P2] Validate fallback refs against catalog availability:** `src/commands/doctor-check-config.ts:279`
  - body: `resolveModelRefFromString` only proves that a fallback string can be parsed. A fallback like `missing-provider/model` can still be counted as validated even though it is not present in the runtime catalog/provider set, leaving the requested fallback-validation gap uncaught.
  - confidence: 0.88
