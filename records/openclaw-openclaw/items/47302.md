## Summary

Keep open: current main does not yet have this flag, and the PR still has source-backed false-success paths in the new diagnostic plus insufficient configured-config behavior proof.

## What This Changes

Adds an `openclaw doctor --check-config` CLI flag with a new deep config validation module and formatter tests for schema, model, fallback, TTS, and channel output.

## Best Possible Solution

Land a revised flag that fails closed for broken configured inputs, validates fallbacks against runtime/catalog availability, adds behavior coverage and CLI docs, and includes redacted CLI proof for valid and failing configs.

## Reproduction Assessment

Yes, by source inspection for the PR blockers: unresolved env placeholders are preserved, malformed `anthropic/` parses to null while the resolver falls back, and fallback validation only calls the parser. I did not execute those configs in this read-only review.

## Solution Assessment

No. The feature direction is useful, but this implementation is not yet the best mergeable form because it can still report OK or exit 0 for broken configured values the command is meant to catch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Fail unresolved env substitutions:** `src/commands/doctor-check-config.ts:72`
  - body: `readConfigFileSnapshot` preserves missing `${VAR}` placeholders and records a warning, but this branch keeps that as `warn` while `hasFailures` ignores warnings. For a configured OpenAI TTS key like `${OPENAI_API_KEY}`, the provider sees a non-empty apiKey, `TTS provider` can report OK, and the command exits 0 even though the feature is unavailable.
  - confidence: 0.9
- **[P2] Fail malformed configured primary refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: For a primary model such as `anthropic/`, `parseModelRef` returns null, so this mismatch check is skipped after `resolveConfiguredModelRef` has already fallen back to the default. The command then reports `Primary model` OK for the fallback instead of failing the malformed configured value.
  - confidence: 0.9
- **[P2] Validate fallback refs against runtime status:** `src/commands/doctor-check-config.ts:279`
  - body: `resolveModelRefFromString` only proves the fallback string parses or aliases; an unknown provider/model ref can still be counted in the validated total without checking catalog or provider availability. That misses broken fallback chains the new command is expected to surface.
  - confidence: 0.83
