## Summary

Keep this PR open, but it still has merge-blocking correctness issues in the new diagnostic path and the supplied proof only covers typecheck plus the no-config CLI path, not the configured validation cases the feature is meant to catch.

## What This Changes

The PR adds an `openclaw doctor --check-config` flag with a new deep config validation module and formatter tests for schema, model, fallback, TTS, and channel checks.

## Best Possible Solution

Land a revised flag that fails closed for broken configured inputs, validates fallbacks through the same runtime/catalog status contract users rely on, adds behavior coverage, documents the CLI surface, and includes redacted real CLI proof for passing and failing configs.

## Reproduction Assessment

Yes, by source inspection: unresolved env placeholders remain warnings, malformed primary refs can fall through to a resolver fallback, and fallback validation only checks parsing. I did not execute those configs in this read-only review.

## Solution Assessment

No. The feature direction is useful, but this implementation is not yet the best mergeable solution because it can still report success for broken configured values the command is intended to catch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Fail unresolved env substitutions:** `src/commands/doctor-check-config.ts:72`
  - body: `readConfigFileSnapshot` preserves missing env placeholders, but this branch records them as `warn` while `hasFailures` ignores warnings. With a configured key like `${OPENAI_API_KEY}`, the OpenAI TTS provider sees a non-empty apiKey, so the command can exit 0 while TTS is unavailable.
  - confidence: 0.9
- **[P2] Fail malformed configured primary refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: For a primary model such as `anthropic/`, `parseModelRef` returns null after `resolveConfiguredModelRef` has already fallen back to the default. The mismatch branch is skipped and the command can print `Primary model` OK for the fallback instead of failing the malformed configured value.
  - confidence: 0.9
- **[P2] Validate fallback refs against runtime status:** `src/commands/doctor-check-config.ts:279`
  - body: `resolveModelRefFromString` proves only that the fallback string parses or aliases; an unknown provider/model can still be counted in the validated total. This misses broken fallback chains unless the check also uses the runtime/catalog status contract before reporting success.
  - confidence: 0.83
