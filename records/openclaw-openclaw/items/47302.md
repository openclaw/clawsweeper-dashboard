Full review comments:

- **[P1] Use the existing doctor lint health-check path:** `src/cli/program/register.maintenance.ts:28-32`
  - body: Current main already exposes `openclaw doctor --lint` as the read-only CI/preflight surface with JSON output, severity thresholds, and `--only`/`--skip`. Adding `--check-config` here creates a competing automation contract, so these diagnostics should be registered as health checks under `doctor --lint` unless maintainers explicitly approve a second surface.
  - confidence: 0.93
- **[P1] Fail missing env substitutions in check-config:** `src/commands/doctor-check-config.ts:72`
  - body: `readConfigFileSnapshot` reports missing `${ENV}` placeholders as warnings that make the referenced feature unavailable, but this code records them as `warn` and `hasFailures` ignores warnings. A config with an unresolved credential placeholder can still exit 0, missing the silent degradation this command is meant to catch for CI gates.
  - confidence: 0.88
- **[P2] Reject unparseable configured primary refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: When a configured primary model contains `/` but cannot be parsed, such as `openai/`, `resolveConfiguredModelRef` can fall back before this check runs. Because this branch only warns when `parseModelRef` returns a parsed value, the malformed configured ref falls through to the final OK result.
  - confidence: 0.86
- **[P2] Validate fallback refs against runtime availability:** `src/commands/doctor-check-config.ts:279`
  - body: `resolveModelRefFromString` only proves the fallback string can be parsed or resolved as an alias; it does not prove the provider/model is available in the configured runtime or catalog set. An unknown but syntactically valid fallback can be reported as validated, missing the broken fallback-chain condition this command is intended to catch.
  - confidence: 0.82
