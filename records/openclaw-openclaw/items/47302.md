## Summary

Keep open. Current main does not implement `doctor --check-config`, and the PR still has blocking diagnostic gaps that can let broken config report healthy output despite the feature's CI-gating goal.

## What This Changes

The PR adds `openclaw doctor --check-config`, a config-validation module, CLI registration, and formatter tests for schema/plugin/model/fallback/TTS/channel diagnostics.

## Best Possible Solution

Land this only after the diagnostic checks fail closed for unresolved required inputs, validate primary and fallback model refs against runtime availability, add focused subsystem/CLI tests, and provide redacted real CLI proof for passing and failing configs.

## Reproduction Assessment

Yes. The blocking PR findings are source-reproducible with configs using an unset `${OPENAI_API_KEY}`, a malformed primary model such as `anthropic/`, and a fallback like `missing-provider/k2p5`; I did not execute them because this review is read-only.

## Solution Assessment

No. The feature direction is useful, but this implementation is not yet the best solution because it can still miss the silent degradation cases it is meant to detect and lacks subsystem coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Fail unresolved env-var config warnings:** `src/commands/doctor-check-config.ts:72`
  - body: `readConfigFileSnapshot` preserves missing `${VAR}` placeholders as warnings, but this PR emits those warnings as `status: "warn"` and `hasFailures` ignores them. Providers such as OpenAI TTS treat a nonempty placeholder API key as configured, so `messages.tts.providers.openai.apiKey: "${OPENAI_API_KEY}"` with the env var unset can still report TTS OK and exit 0 instead of catching the broken setup.
  - confidence: 0.9
- **[P2] Reject parse-null primary refs before fallback:** `src/commands/doctor-check-config.ts:153-154`
  - body: For slash-qualified malformed primaries like `anthropic/`, `parseModelRef` returns null while `resolveConfiguredModelRef` falls back to the default model. Because this branch only warns when `parsed` is truthy, the command can still print an OK primary model for a broken configured value.
  - confidence: 0.9
- **[P2] Validate fallback refs against available models:** `src/commands/doctor-check-config.ts:279`
  - body: `resolveModelRefFromString` only parses aliases/refs; it does not prove the provider or model is available. A fallback such as `missing-provider/k2p5` is accepted as a ref and then counted as validated, so the fallback-provider validation requested by the linked issue still does not happen.
  - confidence: 0.88
