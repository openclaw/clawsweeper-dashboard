## Summary

Keep open: the PR is a useful implementation candidate for the deep config doctor work tracked from https://github.com/openclaw/openclaw/issues/45552, but latest head still has source-reproducible false-green cases and the supplied proof only covers the no-config CLI path, not configured success/failure validation.

## What This Changes

Adds an `openclaw doctor --check-config` CLI flag with schema/plugin validation, model/TTS/channel dry-run checks, result formatting, and formatter tests.

## Best Possible Solution

Land a revised implementation that fails closed for unresolved required inputs, distinguishes invalid configured primary refs before fallback, validates fallback refs against current runtime/catalog/provider availability, adds subsystem and CLI coverage, and includes redacted configured CLI proof.

## Reproduction Assessment

Yes, source-reproducible. Configs with an unset `${OPENAI_API_KEY}`, `agents.defaults.model: "anthropic/"`, and a fallback such as `missing-provider/k2p5` follow deterministic false-green paths in the latest PR head; I did not execute them because this pass is read-only.

## Solution Assessment

No. The flag is the right surface, but the current implementation still reports some broken configured states as warning or OK; the safer path is to reuse current runtime validation contracts and fail non-zero for broken required inputs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Fail unresolved config env warnings:** `src/commands/doctor-check-config.ts:68-73`
  - body: `readConfigFileSnapshot` preserves missing `${VAR}` placeholders as warnings, but this command emits them as `warn` and only exits non-zero for `fail`. With a configured TTS key like `${OPENAI_API_KEY}` and the env var unset, the placeholder remains a nonempty string, so provider configuration can look healthy while `--check-config` exits 0.
  - confidence: 0.9
- **[P2] Reject unparseable primary refs before fallback:** `src/commands/doctor-check-config.ts:152-154`
  - body: For a malformed configured primary such as `anthropic/`, `resolveConfiguredModelRef` falls back to the default model first. `parseModelRef` then returns null, so this branch does not report the bad configured value and the command can print the fallback as the OK primary model.
  - confidence: 0.9
- **[P2] Validate fallback refs against runtime availability:** `src/commands/doctor-check-config.ts:279-299`
  - body: The fallback loop treats any syntactically resolved `provider/model` as validated. A value such as `missing-provider/k2p5` can be counted in `fallbacks validated` even though the command has not shown that the provider/model is available under the current runtime contract.
  - confidence: 0.84
