## Summary

Keep open: this PR is still the canonical implementation path for the requested config-check doctor mode, but it is not mergeable against current main because the branch is stale against the current TTS facade, can write during catalog validation, treats unresolved env-backed config as warn-only, and lacks real behavior proof.

## What This Changes

Adds an `openclaw doctor --check-config` flag that dynamically loads a new deep config validation module with formatter tests for schema, model, fallback, TTS, and channel diagnostics.

## Best Possible Solution

Land a rebased, side-effect-free doctor config-check mode that uses current registry-backed TTS and model contracts, fails active unresolved inputs and malformed refs, documents the CLI flag, and includes behavior tests plus redacted live command output.

## Reproduction Assessment

Yes. The build and behavior blockers are source-reproducible: the PR imports a TTS symbol absent from current main, uses write-capable catalog loading, and keeps missing env substitutions warn-only while failures are the only non-zero exit path.

## Solution Assessment

No. The feature direction is useful, but this patch needs to be rebased onto current TTS/provider contracts, made read-only for diagnostics, tightened around unresolved config inputs and malformed model refs, and proven in a real `openclaw doctor --check-config` run before merge.

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Use the current TTS provider-order facade:** `src/commands/doctor-check-config.ts:18`
  - body: `src/tts/tts.ts` on current main no longer exports `TTS_PROVIDERS`, so this new module will fail after rebase. Use the exported `resolveTtsProviderOrder` facade and the current `isTtsProviderConfigured(..., cfg)` contract instead of a fixed provider list.
  - confidence: 0.98
- **[P2] Fail active unresolved config inputs:** `src/commands/doctor-check-config.ts:72`
  - body: Snapshot warnings for missing `${ENV_VAR}` substitutions are emitted as `warn`, and `hasFailures` only counts `fail`. An active provider secret left as an unresolved placeholder can therefore still produce exit 0, which misses the CI-gating purpose of this mode.
  - confidence: 0.9
- **[P2] Keep catalog loading read-only:** `src/commands/doctor-check-config.ts:183`
  - body: `loadModelCatalog({ config: cfg })` takes the write-capable path on current main and can call `ensureOpenClawModelsJson`. A doctor validation mode should pass `readOnly: true` so checking config does not mutate user state.
  - confidence: 0.93
- **[P2] Reject malformed explicit primary refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: When a qualified configured model such as `anthropic/` cannot be parsed, `parseModelRef` returns null and this branch does not warn, while `resolveConfiguredModelRef` can fall back to defaults. The command can then report the primary model as healthy instead of flagging the broken explicit config.
  - confidence: 0.86
- **[P3] Document the new doctor flag:** `src/cli/program/register.maintenance.ts:28-32`
  - body: This adds a user-facing CLI flag, but the PR does not update the doctor CLI docs. Add the `--check-config` option and an example so the released CLI surface matches the public reference.
  - confidence: 0.82
