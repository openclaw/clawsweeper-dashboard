## Summary

Keep open: current main does not implement `doctor --check-config`, and the PR still has blocking API and diagnostic-correctness issues that should be fixed before merge.

## What This Changes

The PR adds an `openclaw doctor --check-config` CLI path, a new deep config validation module, and formatter-focused tests for schema, model, fallback, TTS, and channel diagnostics.

## Best Possible Solution

Land a rebased deep-config doctor mode that uses current registry-backed TTS APIs, keeps catalog/config inspection read-only, validates primary and fallback model refs against runtime-equivalent availability, documents the flag, adds behavior tests, and includes the required changelog entry.

## Reproduction Assessment

Yes. The highest-confidence reproduction is static: the PR branch imports `TTS_PROVIDERS` from `../tts/tts.js`, but current main does not export that symbol, and the remaining diagnostic gaps are traceable through current config, catalog, and model-selection code paths.

## Solution Assessment

No. The feature direction is useful for the linked config-validation requests, but this patch is not the best current implementation until it aligns with current TTS/provider contracts, avoids write-capable catalog loading, and validates the runtime failure modes it advertises.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Use the exported TTS provider order helper:** `src/commands/doctor-check-config.ts:18`
  - body: `../tts/tts.js` does not export `TTS_PROVIDERS` on current main, so this new module will fail typecheck/build as soon as it is included. Use the current registry-backed `resolveTtsProviderOrder` API to enumerate fallback providers.
  - confidence: 0.98
- **[P2] Fail unresolved active config inputs:** `src/commands/doctor-check-config.ts:72`
  - body: Snapshot warnings for missing `${ENV_VAR}` substitutions are emitted as `warn`, and `hasFailures` only counts `fail`. That lets an unresolved API key used by an active provider produce exit 0, which misses the broken config this CI-gating mode is meant to catch.
  - confidence: 0.9
- **[P2] Load the model catalog in read-only mode:** `src/commands/doctor-check-config.ts:183`
  - body: `loadModelCatalog({ config: cfg })` takes the writable path on current main and can call `ensureOpenClawModelsJson`. A doctor validation mode should pass `readOnly: true` so checking config does not mutate user state.
  - confidence: 0.93
- **[P2] Reject malformed primary refs before fallback:** `src/commands/doctor-check-config.ts:153-154`
  - body: When `agents.defaults.model` is a malformed qualified ref such as `anthropic/`, `parseModelRef` returns null and this branch skips the warning, while `resolveConfiguredModelRef` can fall back to the default model. The check can then report the fallback as healthy instead of flagging the broken explicit primary model.
  - confidence: 0.88
- **[P2] Validate fallback refs against catalog availability:** `src/commands/doctor-check-config.ts:279`
  - body: `resolveModelRefFromString` only proves that a fallback string can be parsed. A fallback like `missing-provider/model` can still be counted as validated even when it is absent from the runtime catalog/provider set, leaving the linked fallback-validation request unhandled.
  - confidence: 0.88
- **[P3] Document the new doctor flag and changelog entry:** `src/cli/program/register.maintenance.ts:28-32`
  - body: This user-facing `feat` adds a new CLI option, but the PR does not update the doctor docs or `CHANGELOG.md`. Add both so the released CLI surface and release notes match the new behavior.
  - confidence: 0.84
