## Summary

Keep open: the idea is useful, but this branch should not merge a second doctor automation surface while current `doctor --lint` already owns that contract and the new checks still have false-clean paths.

## What This Changes

The branch adds an `openclaw doctor --check-config` flag, a new deep config validation module for schema/model/fallback/TTS/channel checks, and formatter-only tests.

## Best Possible Solution

Port the deep config diagnostics into structured `doctor --lint` health checks, share the existing JSON/severity/skip/only/exit behavior, and prove valid plus intentionally broken configured cases.

## Reproduction Assessment

Yes, for the review findings by source inspection: unresolved `${ENV}` warnings, malformed `agents.defaults.model` values such as `openai/`, and parseable unknown fallbacks each have clear paths through the new code. I did not execute the CLI in this read-only review.

## Solution Assessment

No. The maintainable path is to port these diagnostics into `doctor --lint` and fix the false-clean validation cases before merge, rather than adding a parallel flag.

Full review comments:

- **[P1] Use the existing doctor lint health-check path:** `src/cli/program/register.maintenance.ts:29-32`
  - body: Current main already exposes `openclaw doctor --lint` as the read-only CI/preflight surface with JSON output, severity thresholds, and `--only`/`--skip`. Adding `--check-config` here creates a competing automation contract, so the deep checks should be registered as health checks under `doctor --lint` instead.
  - confidence: 0.91
- **[P1] Fail unresolved env substitutions:** `src/commands/doctor-check-config.ts:72`
  - body: `readConfigFileSnapshot` preserves missing `${ENV}` placeholders and reports them in `snapshot.warnings`, but this code records those as `warn` and `hasFailures` ignores warnings. A config with an unresolved credential placeholder can still exit 0, which misses the silent degradation this command is meant to catch.
  - confidence: 0.86
- **[P2] Reject malformed configured primary refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: When a configured primary model includes `/` but cannot be parsed, such as `openai/`, `resolveConfiguredModelRef` can fall back before this check runs. Because the mismatch branch only fires when `parseModelRef` returns a parsed value, the malformed configured ref falls through to the final OK result.
  - confidence: 0.84
- **[P2] Validate fallback refs against runtime status:** `src/commands/doctor-check-config.ts:279`
  - body: `resolveModelRefFromString` only proves the fallback string can be parsed or resolved as an alias; it does not prove the provider/model is in the configured/catalog runtime set. An unknown but syntactically valid fallback can be reported as validated, missing the provider-definition failure this PR says it closes.
  - confidence: 0.88
