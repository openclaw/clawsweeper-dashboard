## Summary

Keep open: the feature is not on current main, and the latest PR head still has source-reproducible validation gaps that can let broken model config report healthy.

## What This Changes

The PR adds `openclaw doctor --check-config` plus a new config-check module and formatter tests for schema, model, fallback, TTS, and channel diagnostics.

## Best Possible Solution

Land this only after the diagnostic uses the runtime model/catalog contracts, has focused subsystem tests, and includes redacted real CLI proof for configured passing and failing cases.

## Reproduction Assessment

Yes for the blocking PR findings: source inspection shows malformed slash primary refs and unknown fallback refs can still pass through the new checks. I did not run a live CLI repro because this review is read-only.

## Solution Assessment

No. The feature direction is useful, but the implementation should reject parse-null configured primaries and validate fallback refs against catalog/allowlist status before reporting success.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Reject parse-null primary refs before reporting OK:** `src/commands/doctor-check-config.ts:152-154`
  - body: For a malformed configured primary like `anthropic/`, `parseModelRef` returns null, but this branch only warns when `parsed` is truthy. Since `resolveConfiguredModelRef` can already fall back to the default, the command can still print an OK primary model for a broken configured value.
  - confidence: 0.91
- **[P2] Check fallback refs against catalog and allowlist status:** `src/commands/doctor-check-config.ts:279-285`
  - body: `resolveModelRefFromString` only parses and resolves aliases, so a syntactically valid unknown ref such as `missing-provider/k2p5` is counted as validated. This misses the undefined fallback-provider problem the new command is meant to catch; check each resolved fallback with the model status/catalog contract before reporting success.
  - confidence: 0.86
- **[P3] Remove the no-op channel config assertion:** `src/commands/doctor-check-config.ts:398`
  - body: Latest CI fails lint with a no-effect type-conversion error, and this changed assertion is the matching pattern: after the object guard, `channelConfig` is already the value being asserted. Remove the assertion or replace it with a real typed helper so the lint shard can pass.
  - confidence: 0.72
