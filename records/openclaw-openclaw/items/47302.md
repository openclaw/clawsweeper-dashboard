## Summary

Keep open: this is still the canonical active PR for the deep config-validation feature, but the latest head still lets several broken config states report as warnings or OK, and the supplied proof only covers typecheck plus the no-config CLI path.

## What This Changes

The PR adds `openclaw doctor --check-config`, a new diagnostic module, CLI registration, and formatter tests for schema, model, fallback, TTS, and channel config checks.

## Best Possible Solution

Land a revised version that fails closed for unresolved required inputs, validates primary and fallback refs against the current runtime contract, adds subsystem/CLI coverage, and includes redacted real CLI proof for configured success and failure cases.

## Reproduction Assessment

Yes. The blocking findings are source-reproducible with configs using an unset `${OPENAI_API_KEY}`, `agents.defaults.model: "anthropic/"`, and a fallback like `missing-provider/k2p5`; I did not execute them because this review is read-only.

## Solution Assessment

No. The feature direction is useful, but this implementation still reports some broken config states as warning or OK, so it is not yet the narrowest maintainable solution for CI-gated deep validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Fail unresolved env-var warnings in check-config:** `src/commands/doctor-check-config.ts:68-73`
  - body: `readConfigFileSnapshot` preserves missing `${VAR}` placeholders as warnings, but this path emits them as `warn` and only exits non-zero for `fail`. With `messages.tts.providers.openai.apiKey: "${OPENAI_API_KEY}"` and the env var unset, the placeholder survives as a nonempty string, so TTS can still look configured while `--check-config` exits 0.
  - confidence: 0.9
- **[P2] Reject unparseable primary refs before fallback:** `src/commands/doctor-check-config.ts:153-154`
  - body: For a malformed configured primary such as `anthropic/`, `resolveConfiguredModelRef` falls back to the default model after parsing fails. This check then only warns when `parsed` is truthy, so the unparseable configured value falls through and reports the fallback as the OK primary model.
  - confidence: 0.9
- **[P2] Validate fallback refs beyond syntax:** `src/commands/doctor-check-config.ts:279-285`
  - body: The fallback loop treats any syntactically parsed ref as validated. A value like `missing-provider/k2p5` can return a ref from `resolveModelRefFromString`, then get counted in `fallbacks validated` even though the command has not shown that the provider/model is actually available under the current runtime contract.
  - confidence: 0.82
