## Best Possible Solution

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Fail missing env substitutions in check-config:** `src/commands/doctor-check-config.ts:72`
  - body: `readConfigFileSnapshot` preserves missing `${VAR}` placeholders as warnings, and this PR emits those as `status: "warn"` while `hasFailures` ignores warnings. With `messages.tts.provider: "openai"` and `messages.tts.providers.openai.apiKey: "${OPENAI_API_KEY}"` unset, OpenAI TTS still sees a nonempty placeholder as configured, so `doctor --check-config` can exit 0 even though the feature is unavailable.
  - confidence: 0.91
- **[P2] Reject parse-null primary model refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: For malformed slash-qualified primaries like `anthropic/`, `resolveConfiguredModelRef` falls back to the default model and `parseModelRef` returns null, but this branch only warns when `parsed` is truthy. That lets a broken configured primary print an OK primary model instead of failing or warning about the invalid value.
  - confidence: 0.9
- **[P2] Check fallback refs against available models:** `src/commands/doctor-check-config.ts:279`
  - body: `resolveModelRefFromString` only parses aliases and provider/model syntax; it does not prove the provider or model is available. A fallback like `missing-provider/k2p5` is accepted and counted as validated, so the new check still misses the undefined fallback-provider case it is intended to catch.
  - confidence: 0.89
