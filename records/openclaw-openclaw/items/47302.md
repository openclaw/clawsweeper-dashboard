## Summary

Keep open: the feature direction is useful, but the current PR is not merge-ready because it duplicates the current `doctor --lint` health-check surface and still has source-reproducible false-OK paths for broken configured values.

## What This Changes

The PR adds an `openclaw doctor --check-config` CLI path, a new deep config validation module for schema/plugin/model/TTS/channel checks, and formatter tests.

## Best Possible Solution

Port the deep config checks into the existing doctor health-check/`--lint` framework, fail closed for broken configured values, add behavior coverage and docs, and include redacted CLI proof for valid and failing configs.

## Reproduction Assessment

Yes, by source inspection: unresolved env substitutions are warnings ignored by `hasFailures`, malformed primary refs can fall through after resolver fallback, and fallback refs are only parsed/resolved rather than checked against runtime status. I did not execute those configs in this read-only review.

## Solution Assessment

No. The feature direction is useful, but the best mergeable form should extend the existing `doctor --lint` health-check contract and fix the false-OK validation paths before adding a new user-facing surface.

Full review comments:

- **[P1] Use the existing doctor lint health-check path:** `src/cli/program/register.maintenance.ts:29-30`
  - body: Current main already has `openclaw doctor --lint` as the documented read-only CI/preflight surface with JSON, severity thresholds, and skip/only controls. Adding `--check-config` as a separate runner creates a second source of truth for doctor automation and the branch is currently conflicting with that command path; these checks should be registered in the health-check framework or explicitly designed as a maintainer-approved alias.
  - confidence: 0.88
- **[P1] Fail unresolved env substitutions:** `src/commands/doctor-check-config.ts:72`
  - body: Snapshot warnings such as missing `${OPENAI_API_KEY}` are emitted as `warn`, but `hasFailures` only considers `fail`. Because config loading preserves the unresolved placeholder and OpenAI TTS treats a non-empty apiKey as configured, this can still print a healthy TTS provider and exit 0 for a broken configured secret.
  - confidence: 0.91
- **[P2] Fail malformed configured primary refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: For a configured primary like `anthropic/`, `parseModelRef` returns null, so the mismatch warning is skipped after `resolveConfiguredModelRef` falls back to the default. The command can then report `Primary model` OK for the fallback instead of failing the malformed configured value.
  - confidence: 0.9
- **[P2] Validate fallback refs against runtime status:** `src/commands/doctor-check-config.ts:279`
  - body: `resolveModelRefFromString` only proves the fallback string can resolve to a ref or alias; it does not prove that the provider/model is in the catalog or allowed runtime set. Unknown fallback providers can therefore be counted as validated, missing the broken fallback chain this feature is meant to catch.
  - confidence: 0.84
