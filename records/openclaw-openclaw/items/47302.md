## Summary

Keep open: current main still lacks `doctor --check-config`, but this branch is not mergeable as-is because the latest head still imports a non-exported TTS symbol, can miss malformed primary refs and missing fallback providers, and lacks real CLI proof. It also overlaps the active doctor diagnostics design discussion in https://github.com/openclaw/openclaw/pull/80055.

## What This Changes

The branch adds `openclaw doctor --check-config`, a config validation module for schema/model/fallback/TTS/channel checks, command registration, and formatter tests.

## Best Possible Solution

Land this only after rebasing onto the current TTS/model contracts, validating fallback availability rather than syntax alone, aligning the CLI surface with the doctor diagnostics direction, adding focused subsystem tests, and providing redacted real CLI proof.

## Reproduction Assessment

Yes for the blocking PR findings: source inspection of the latest head shows a non-exported TTS import, parse-null primary refs that can fall through after fallback resolution, and fallback checks that only parse refs. I did not run commands because this review is read-only.

## Solution Assessment

No. The feature direction is useful, but this implementation should first use the current provider-order TTS API, validate configured model/fallback refs against runtime availability, and coordinate with the active doctor diagnostics contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use the current TTS provider-order API:** `src/commands/doctor-check-config.ts:18`
  - body: The new module imports `TTS_PROVIDERS` from `../tts/tts.js`, but current main's TTS facade no longer exports that symbol. This will fail typecheck/build before `doctor --check-config` can run; use `resolveTtsProviderOrder(ttsConfig.provider, cfg)` with `isTtsProviderConfigured(..., cfg)` instead.
  - confidence: 0.97
- **[P2] Fail parse-null primary refs before fallback masks them:** `src/commands/doctor-check-config.ts:152-154`
  - body: For a slash-qualified but malformed primary like `anthropic/`, `parseModelRef` returns null while `resolveConfiguredModelRef` can already have fallen back to the default. Because this branch only warns when `parsed` is truthy, the command can still print an OK primary model for a broken configured value.
  - confidence: 0.9
- **[P2] Validate fallback refs against available models:** `src/commands/doctor-check-config.ts:279-285`
  - body: `resolveModelRefFromString` accepts syntactically valid unknown refs, so a fallback such as `missing-provider/k2p5` is counted as validated. That misses the fallback-provider failure this feature claims to catch; check each resolved fallback with the catalog/provider status before reporting success.
  - confidence: 0.86
