## Label Justifications

- `P1`: The issue reports a user-facing embedded-agent/webchat failure where overflow recovery can stall for many minutes and reject new messages with session lock errors.
- `impact:session-state`: The failure is rooted in session JSONL write-lock ownership during transcript compaction.
- `impact:message-loss`: New webchat messages during the recovery window can fail before any agent reply is delivered.
- `maturity:stable`: The affected embedded gateway/webchat/session runtime falls under the M4 Stable `gateway-runtime` feature area from the maturity shortlist.

## Summary

Keep open: current main and v2026.6.10 still acquire the direct embedded compaction session lock without `allowReentrant`, while an open proof-positive PR already targets the exact narrow fix path.

## What This Changes

The issue requests that overflow-recovery compaction treat same-process nested session write-lock acquisition as reentrant instead of timing out on its own lock.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/88919 or an equivalent narrow patch that opts direct runtime compaction into reentrant session-lock acquisition, with focused regression coverage for the intended same-writer boundary.

## Reproduction Assessment

Yes at source level: current main still acquires the direct compaction lock without reentrancy while the embedded attempt controller can already own the same session lock. I did not live-run the long overflow/fallback scenario in this read-only review.

## Solution Assessment

Yes, conditionally: a narrow same-logical-writer reentrant opt-in for direct compaction matches the existing lock primitive and documented transcript-lock policy, but maintainers still need to land the open candidate PR or an equivalent focused patch.
