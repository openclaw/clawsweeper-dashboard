## Label Justifications

- `P1`: The issue reports a user-facing embedded-agent/webchat workflow failure where overflow recovery can stall for many minutes and reject new messages with session lock errors.
- `impact:session-state`: The root cause is session JSONL write-lock ownership during transcript compaction and same-process nested mutation.
- `impact:message-loss`: New webchat messages during the recovery window can fail before any agent reply is delivered.
- `maturity:stable`: The WebChat/Gateway session API impact maps to the M4 Stable `gateway-runtime` surface, even though the underlying session/context implementation is M3.

## Summary

Keep open: current main and v2026.6.10 still acquire the direct embedded compaction session lock without `allowReentrant`, while https://github.com/openclaw/openclaw/pull/88919 is an open proof-positive candidate for the exact fix path.

## What This Changes

The issue requests fixing embedded overflow-recovery compaction so an already-held same-process session write lock is re-entered instead of timing out and blocking webchat replies.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/88919 or an equivalent narrow patch that opts direct runtime compaction into reentrant session-lock acquisition, then decide separately whether broader inner-writer sites from https://github.com/openclaw/openclaw/pull/96000 belong in a follow-up.

## Reproduction Assessment

Yes, at source level. Current main still sends overflow recovery through direct compaction, direct compaction acquires the session lock without `allowReentrant`, and the lock primitive only re-enters when that option is true; I did not live-run the long overflow/fallback scenario in this read-only review.

## Solution Assessment

Yes, conditionally. The narrow direct-compaction `allowReentrant: true` opt-in is the best immediate fix for this report because it matches the documented same-logical-writer lock contract; broader inner-writer reentry should be reviewed separately.
