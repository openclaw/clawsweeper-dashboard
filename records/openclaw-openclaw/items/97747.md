## Label Justifications

- `P1`: The issue reports a user-facing embedded-agent/WebChat workflow failure where overflow recovery can stall for many minutes and reject new messages.
- `impact:session-state`: The root cause is session JSONL write-lock ownership during transcript compaction and same-process nested mutation.
- `impact:message-loss`: New webchat messages during the recovery window can fail before any agent reply is delivered.
- `maturity:stable`: The user-visible WebChat/Gateway message failure maps to the M4 Stable `gateway-runtime` surface, even though the lower-level session/context implementation is M3.

## Summary

Keep open: current `main` and the latest release still acquire the direct embedded compaction session lock without `allowReentrant`, while the older open PR at https://github.com/openclaw/openclaw/pull/88919 is the proof-positive candidate for the exact fix path.

## What This Changes

Issue reports that overflow-recovery auto-compaction can self-contend on the session write lock and block embedded agent replies during fallback recovery.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/88919 or an equivalent narrow direct-compaction reentrancy patch, then review broader inner-writer reentry separately.

## Reproduction Assessment

Yes, at source level. Current main sends overflow recovery through direct compaction, direct compaction acquires the session lock without `allowReentrant`, and the lock primitive only re-enters when that option is true; I did not live-run the long overflow/fallback scenario.

## Solution Assessment

Yes, conditionally. The narrow direct-compaction `allowReentrant: true` opt-in matches the documented same-logical-writer lock contract, while broader inner-writer reentry should be evaluated as a separate follow-up.
