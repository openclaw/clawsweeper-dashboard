## Label Justifications

- `P1`: The issue describes a real channel workflow where a final assistant reply can be generated locally but never become visible to the Signal user after gateway restart recovery.
- `impact:message-loss`: The affected behavior is a stranded or missing user-visible final reply after an interrupted outbound send.
- `impact:session-state`: The session can appear complete while the durable outbound delivery state remains ambiguous or failed outside the session-visible flow.

## Summary

Keep open: current main and the latest release still refuse no-reconciliation `send_attempt_started` recovery without giving the originating Signal/session a durable recovery signal, and the related open PR is only a draft partial mitigation.

## What This Changes

The issue reports that Signal final replies can be stranded when a gateway restart interrupts outbound delivery after `send_attempt_started`.

## Best Possible Solution

Choose one owner-approved ambiguous-send recovery contract, then implement it through the shared outbound queue with Signal-specific reconciliation or durable session/operator-visible failed-delivery signaling.

## Reproduction Assessment

Yes, at source level: create or inspect a queued Signal outbound entry with `recoveryState: "send_attempt_started"`; because Signal lacks `reconcileUnknownSend`, current recovery refuses replay and moves/fails the entry without a session-visible recovery signal. I did not run a live Signal restart reproduction.

## Solution Assessment

Unclear as a final product choice. The current fail-closed behavior prevents duplicate sends but does not meet the issue's visible recovery requirement; maintainers need to choose the recovery contract before implementation.
