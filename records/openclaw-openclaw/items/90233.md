## Summary

Keep open: current main and v2026.6.9 still route Signal send_attempt_started recovery through the no-reconciliation fail-closed path, and current source shows failed queue diagnostics rather than session-visible recovery of the missing reply. Related delivery-reliability issues are adjacent but do not replace this specific Signal restart failure.

## What This Changes

The issue reports that Signal final replies can be stranded when a gateway restart interrupts delivery after send_attempt_started.

## Best Possible Solution

Choose a maintainer-approved ambiguous-send recovery contract, then implement it through shared outbound recovery plus Signal reconciliation or durable session/operator-visible failed-delivery signaling.

## Reproduction Assessment

Yes, at source level: persist a queued Signal outbound entry with recoveryState send_attempt_started, and current recovery refuses replay because Signal does not declare reconcileUnknownSend. I did not run a live Signal restart reproduction in this read-only review.

## Solution Assessment

Unclear as a final implementation choice. The current fail-closed behavior prevents duplicate sends but does not meet the issue's visible recovery requirement, so maintainers need to choose the recovery contract before implementation.
