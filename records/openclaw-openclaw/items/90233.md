## Label Justifications

- `P1`: A final assistant reply can be generated locally but never become visible to the Signal user after gateway restart recovery.
- `impact:message-loss`: The affected behavior can strand or lose a user-visible channel reply after an interrupted outbound send.
- `impact:session-state`: The session can appear complete while durable outbound delivery remains ambiguous or failed outside the session-visible flow.

## Summary

Keep open: current main and v2026.6.8 still refuse no-reconciliation `send_attempt_started` recovery, Signal still lacks `reconcileUnknownSend`, and no merged related item resolves the session-visible stranded-reply failure.

## What This Changes

The issue asks OpenClaw to prevent or surface Signal final replies that are interrupted after `send_attempt_started` during gateway restart.

## Best Possible Solution

Choose one maintainer-approved ambiguous-send recovery contract, then implement it through shared outbound recovery plus Signal reconciliation or durable session/operator-visible failed-delivery signaling.

## Reproduction Assessment

Yes, at source level: persist a queued Signal outbound entry with `recoveryState: "send_attempt_started"`; current recovery refuses replay because Signal lacks `reconcileUnknownSend`. I did not run a live Signal restart reproduction.

## Solution Assessment

Unclear as a final implementation choice. The current fail-closed behavior prevents duplicate sends but does not meet the issue's visible recovery requirement, so maintainers need to choose the recovery contract before implementation.
