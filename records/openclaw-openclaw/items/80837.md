## Summary

Keep open. The PR is aimed at the right plugin-local PDF fallback path and addresses the linked canvas-missing bug, but the Poppler path can violate the existing pixel budget and the external PR still needs real after-fix behavior proof before merge.

## What This Changes

The PR adds a document-extract `pdftoppm` fallback renderer, focused tests, PDF docs, and a changelog entry for PDF image extraction when `@napi-rs/canvas` cannot load.

## Best Possible Solution

Keep the fallback in `extensions/document-extract`, but enforce actual Poppler output bounds with scaling or dimension checks, keep the docs/tests/changelog coverage, and require real headless-Linux proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows the low-text PDF path catches a missing `@napi-rs/canvas` import and returns empty images, and the linked bug report gives matching Ubuntu/poppler setup evidence.

## Solution Assessment

No as written. The plugin-local `pdftoppm` fallback is the right direction, but the PR must preserve the image pixel budget and be proven in a real setup before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor the render plan when spawning pdftoppm:** `extensions/document-extract/document-extractor.ts:266`
  - body: `resolveRenderPlan` can choose a scale below 72 DPI to keep the page within `request.maxPixels`, but this line clamps the fallback back up to at least 72 DPI before rendering. For large pages or small budgets, `pdftoppm` can produce a much larger PNG while `remainingPixels` is decremented by the smaller planned size, so the Poppler path no longer enforces the hard image budget. Use Poppler scaling options or inspect/account for the actual output dimensions before accepting the image.
  - confidence: 0.88
