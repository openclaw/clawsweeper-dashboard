## Summary

Keep open. The plugin-local Poppler fallback is the right direction for the linked PDF extraction bug, but the current PR still weakens the image pixel budget on the external renderer path and lacks required real behavior proof from a setup where canvas is absent.

## What This Changes

The PR adds a document-extract `pdftoppm` fallback for canvas-missing PDF image rendering, plus docs, changelog, and mocked regression tests.

## Best Possible Solution

Keep the fallback plugin-local, enforce Poppler output bounds with scaling or actual-dimension accounting, preserve the docs/tests/changelog coverage, and require real headless-Linux proof before merge.

## Reproduction Assessment

Yes, by source inspection. On current main, a low-text PDF reaches `loadCanvasModule()`, catches a missing `@napi-rs/canvas` import, and returns empty images; the linked issue provides matching Ubuntu/poppler setup evidence.

## Solution Assessment

No as written. A plugin-local `pdftoppm` fallback is the narrow owner-module direction, but the patch must preserve the hard image budget and include real behavior proof before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor the render plan when spawning pdftoppm:** `extensions/document-extract/document-extractor.ts:266`
  - body: `resolveRenderPlan` can choose a scale below 72 DPI to keep the page within `request.maxPixels`, but this line clamps the fallback back up to at least 72 DPI before rendering. For large pages or small budgets, `pdftoppm` can produce a much larger PNG while `remainingPixels` is decremented by the smaller planned size, so the Poppler path no longer enforces the hard image budget. Use Poppler scaling options or inspect and account for the actual output dimensions before accepting the image.
  - confidence: 0.88
