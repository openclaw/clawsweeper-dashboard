## Summary

## What This Changes

The PR adds a document-extract pdftoppm fallback, regression tests, PDF docs, and a changelog entry for PDF image rendering when @napi-rs/canvas cannot load.

## Best Possible Solution

Keep the fallback plugin-local, but render Poppler output to the planned dimensions under the OpenClaw temp root, fix the new test type/lint failures, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes, at source level. Current main reaches the low-text PDF image path, catches the missing @napi-rs/canvas import, and returns no images; the linked issue provides matching Ubuntu headless setup evidence.

## Solution Assessment

No, not in this exact form. A plugin-local pdftoppm fallback is the right direction, but this implementation should use target dimensions rather than clamped DPI, use the plugin-sdk/OpenClaw temp helper, and pass CI plus real setup proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Render Poppler pages at the planned dimensions:** `extensions/document-extract/document-extractor.ts:266-267`
  - body: The fallback computes a render plan but then passes only a clamped DPI to pdftoppm. For any page that needs downscaling below 1x, the 72 DPI floor can render near the original page size while `remainingPixels` subtracts only `plan.pixels`, so this path can exceed the PDF image budget it claims to honor. Pass `plan.width`/`plan.height` via pdftoppm scaling options or verify the actual output before accepting it.
  - confidence: 0.91
- **[P2] Use the OpenClaw temp root for PDF scratch files:** `extensions/document-extract/document-extractor.ts:108`
  - body: This writes user PDF bytes under `os.tmpdir()`, and the exact-head boundary check rejects that in messaging/plugin runtime sources. Use `resolvePreferredOpenClawTmpDir()` or `withTempWorkspace` from `openclaw/plugin-sdk/temp-path` so scratch files land under the repo-approved private temp root.
  - confidence: 0.9
- **[P2] Make the new fallback tests type-check:** `extensions/document-extract/document-extractor.test.ts:200-201`
  - body: `extract()` can return `DocumentExtractionResult | null`, but the new tests read `result.images` without narrowing. The exact-head `check-test-types` job fails on these assertions, so add the same non-null guard used by the existing test before inspecting the result.
  - confidence: 0.96
- **[P3] Drop the redundant dynamic import assertions:** `extensions/document-extract/document-extractor.test.ts:156-157`
  - body: Oxlint fails the branch on the `as typeof import(...)` assertions around the two new dynamic imports because they do not change the expression type. Removing those assertions should clear the exact-head lint failure.
  - confidence: 0.82
