## Summary

This PR should stay open because it is the active implementation for the linked PDF extraction bug, but the proposed Poppler fallback still weakens the existing image budget and lacks real behavior proof from a setup where canvas is absent.

## What This Changes

The branch adds a `pdftoppm` fallback renderer to the bundled `document-extract` plugin, updates PDF docs, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the plugin-local Poppler fallback direction, but bound actual renderer output with Poppler scaling or post-render dimension accounting and require redacted real setup proof before merge.

## Reproduction Assessment

Yes for the underlying issue from source inspection: current main returns empty images when the canvas module cannot load, and the linked report gives a matching Ubuntu/Poppler setup. I did not run a live PDF extraction in this read-only review, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

No as written. A plugin-local `pdftoppm` fallback is the right narrow solution, but the PR needs to preserve the image pixel budget in the actual Poppler output and show real behavior proof before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor the render plan when spawning pdftoppm:** `extensions/document-extract/document-extractor.ts:266`
  - body: `resolveRenderPlan` can choose dimensions below what 72 DPI produces, but this line clamps the fallback back up to at least 72 DPI before rendering. For large pages or small `request.maxPixels` budgets, Poppler can emit a PNG with more pixels than the plan while `remainingPixels` is decremented by the smaller planned size, so the fallback no longer enforces the image budget. Use Poppler scaling options or inspect/account for the actual output dimensions before accepting the image.
  - confidence: 0.88
