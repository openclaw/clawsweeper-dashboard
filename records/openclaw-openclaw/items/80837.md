## Summary

Keep open. The PR targets a real document-extract bug and is paired with the open closing issue, but the current head still weakens the Poppler renderer pixel/resource budget and lacks required after-fix proof from a real setup.

## What This Changes

This PR adds a document-extract `pdftoppm` fallback with tests, docs, and a changelog entry for PDF image extraction when `@napi-rs/canvas` cannot load.

## Best Possible Solution

Keep the fallback plugin-local, but enforce Poppler output bounds with scaling or actual dimension checks, retain the docs/tests/changelog coverage, and require real headless-Linux proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows low-text PDFs return empty images when `@napi-rs/canvas` cannot load, and the linked Ubuntu report provides matching setup evidence.

## Solution Assessment

No as written. A plugin-local `pdftoppm` fallback is the right boundary, but this implementation must preserve the hard image budget and be proven in a real setup before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor the render plan when spawning pdftoppm:** `extensions/document-extract/document-extractor.ts:266`
  - body: `resolveRenderPlan()` can choose a scale below 72 DPI to keep the page within `request.maxPixels`, but this line clamps the fallback back up to at least 72 DPI. For large pages or small budgets, `pdftoppm` can produce a much larger PNG while `remainingPixels` is decremented by the smaller planned size, so the Poppler path no longer enforces the hard image budget. Use Poppler scaling options or inspect/account for the actual output dimensions before accepting the image.
  - confidence: 0.88
