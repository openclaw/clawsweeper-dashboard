## Summary

Keep open. The PR targets a real current-main PDF fallback gap, but it is not merge-ready because the real-behavior proof gate is still failing, several relevant CI jobs are queued, and the new Poppler path can still throw from temp setup before its fallback guard runs.

## What This Changes

The PR adds a document-extract `pdftoppm` fallback when `@napi-rs/canvas` cannot load, plus docs, changelog, regression tests, and a small formatting-only change outside the plugin.

## Best Possible Solution

Land a plugin-local Poppler fallback after guarding temp setup failures, adding real headless/Linux OpenClaw proof for the canvas-absent path, and getting the relevant PR checks green.

## Reproduction Assessment

Yes from source inspection: current main reaches `loadCanvasModule()` for low-text PDFs and returns `{ text, images: [] }` when the optional canvas import fails. The linked issue supplies a matching Ubuntu/Poppler setup, but I did not run a live reproduction in this read-only review.

## Solution Assessment

Not as-is. The plugin-local `pdftoppm` fallback is the right owner boundary, but the branch still needs the temp setup guard fixed and real runtime proof before it is the safest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Guard pdftoppm temp setup failures:** `extensions/document-extract/document-extractor.ts:109`
  - body: `fs.mkdtemp` runs before this function's `try`, so failures from `resolvePreferredOpenClawTmpDir()` or temp-directory creation escape the Poppler fallback and reject PDF extraction after text was already extracted. On a host with `pdftoppm` on PATH but an unavailable or unsafe temp root, this regresses the current canvas-missing behavior from returning text with empty images to a thrown extraction error. Move temp allocation into the guarded path and return `null` or report the image error so text extraction is preserved.
  - confidence: 0.83
