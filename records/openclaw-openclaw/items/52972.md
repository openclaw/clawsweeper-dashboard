## Summary

Keep open. Current main still has the reported cron-success telemetry gap: the reply guard suppresses the false note only for native cron-tool success or same-session cron rows, while assistant shell/CLI `openclaw cron add --at` creates jobs through the Gateway RPC path without incrementing that same-turn signal.

## What This Changes

The issue reports that successful Telegram reminder creation via `openclaw cron add --at` still gets a contradictory unscheduled-reminder note appended to the reply.

## Best Possible Solution

Add a narrow, verified scheduling signal so successful current-turn cron creation through assistant shell/CLI paths suppresses the false note, with focused regression tests and without broad note stripping, regex policy changes, or a new config option.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw cron add --at` can create the cron through Gateway `cron.add`, but current-main telemetry increments `successfulCronAdds` only for native `cron` tool adds, leaving the reply guard able to append the false note.

## Solution Assessment

No. Current main is not the best solution because it recognizes only the native cron-tool success path; the maintainable fix is to record verified successful CLI-through-exec scheduling without broad note stripping or a new opt-out setting.
