## Summary

Keep open. Current main still has the reported cron-success telemetry gap: the reply guard suppresses the note only when `successfulCronAdds` is positive or an existing same-session cron is found, while the embedded and Codex telemetry still count only successful native `cron` tool adds, not assistant shell/CLI `openclaw cron add --at` paths.

## What This Changes

The issue reports that Telegram replies can receive the unscheduled-reminder note even after `openclaw cron add --at` successfully creates a reminder.

## Best Possible Solution

Add a narrow, verified scheduling signal so successful current-turn cron creation through assistant shell/CLI paths suppresses the false note, with focused regression tests and without broad note stripping, regex policy changes, or a new config option.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw cron add --at` can create the cron through Gateway `cron.add`, but current-main telemetry only increments `successfulCronAdds` for native `cron` tool adds, leaving the reply guard free to append the false note.

## Solution Assessment

No. Current main is not the best solution because it recognizes only the native cron-tool success path; the maintainable fix is to record verified successful CLI-through-exec scheduling without broad note stripping or a new opt-out setting.
