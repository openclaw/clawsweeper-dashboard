## Summary

Keep open: current main and the latest release still only count native `cron` tool adds as `successfulCronAdds`; a successful `openclaw cron add --at` through CLI/shell scheduling can create the job without setting the same-turn signal that suppresses the contradictory reminder note.

## What This Changes

The issue reports a false unscheduled-reminder note being appended after a successful one-time cron reminder is created from a Telegram request.

## Best Possible Solution

Add a narrow verified same-turn scheduling signal so successful cron creation through assistant shell/CLI paths suppresses the false note, with regression coverage and without broad note stripping, regex-policy changes, or a new config option.

## Reproduction Assessment

Yes, at source level. `openclaw cron add --at` creates a job through Gateway `cron.add`, while current-main telemetry increments `successfulCronAdds` only for native cron-tool adds, leaving the reply guard able to append the false note.

## Solution Assessment

No. Current main is not the best solution because it recognizes only native cron-tool success; the maintainable fix is to record verified successful CLI-through-exec scheduling before the reply guard runs.
