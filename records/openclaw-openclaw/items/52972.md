## Summary

Keep open: current main still counts only structured cron add tool success, while the reply guard still appends the unscheduled-reminder note when that per-turn counter is zero. The linked fix path is https://github.com/openclaw/openclaw/pull/86064, but it remains open and conflicting, so this issue is still necessary.

## What This Changes

The issue asks OpenClaw to stop appending the contradictory unscheduled-reminder note after a reminder was successfully scheduled through cron.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/86064 with a narrow fix that counts successful shell-driven cron adds before the reply guard runs, with regression coverage and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current main increments successfulCronAdds only for structured cron tool adds, while the reply guard appends the note when reminder text has zero successful cron adds. I did not run a live Telegram or cron scenario in this read-only review.

## Solution Assessment

Yes, the best fix shape is to count successful shell-driven cron adds at the telemetry producer before the reply guard runs, rather than hiding the note only in Telegram delivery. The current linked PR is the right ownership area but needs repair or replacement before merge.
