## Summary

Keep open: current main and the latest release still only count native `cron` tool adds as `successfulCronAdds`, while `openclaw cron add --at` creates the job through the CLI/Gateway path without the same-turn scheduling signal. The exact prior fix at https://github.com/openclaw/openclaw/pull/53024 is closed and unmerged, and the open reminder-guard PRs are adjacent UX fixes rather than a verified CLI/exec telemetry fix.

## What This Changes

The issue reports a contradictory unscheduled-reminder note being appended after a successful cron reminder is created from a Telegram agent turn.

## Best Possible Solution

Add a narrow verified same-turn scheduling signal so successful cron creation through assistant shell/CLI paths suppresses the false note, with regression coverage and without broad note stripping, regex-policy changes, or a new config option.

## Reproduction Assessment

Yes, at source level. `openclaw cron add --at` creates a cron job through Gateway `cron.add`, while current-main telemetry increments `successfulCronAdds` only for native `cron` tool adds, leaving the reply guard able to append the false note.

## Solution Assessment

No. Current main is not the best solution because it recognizes only native cron-tool success; the maintainable fix is to record verified successful CLI-through-exec scheduling before the reply guard runs.
