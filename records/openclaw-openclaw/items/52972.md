## Summary

Keep open. Current main still has the reported cron-success signal gap: the reply guard suppresses the note only when `successfulCronAdds` is nonzero or an enabled same-session cron already exists, while the embedded telemetry still counts only native `cron` tool adds. A successful `openclaw cron add --at ...` issued through `exec` reaches the Gateway `cron.add` RPC but does not update that current-turn counter, and the prior exact fix PR #53024 is closed unmerged.

## What This Changes

The issue reports that successful reminder scheduling through `openclaw cron add --at` can still result in the user-visible unscheduled-reminder note.

## Best Possible Solution

Implement a narrow scheduling-signal fix so successful current-turn cron reminders are recognized consistently across the native cron tool and the assistant `exec`/CLI path, with regression coverage for successful, failed, and unrelated shell commands plus the final reply guard. Keep regex tightening, Telegram outbound sanitization, plugin-managed scheduler APIs, and broader commitment-gating policy in their separate related threads.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is visible: an embedded run can execute a successful `exec` command that invokes `openclaw cron add --at ...`, the CLI creates the Gateway cron job, but `successfulCronAdds` stays zero because only native `cron` tool adds are counted; a reminder-like final reply can then still hit the note-append guard.

## Solution Assessment

Yes, the narrow telemetry direction from the closed-unmerged #53024 is the best repair shape if implemented carefully. It should record successful current-turn cron creation for the CLI-through-exec path without changing unrelated reminder regex, Telegram delivery, or the broader guard policy.
