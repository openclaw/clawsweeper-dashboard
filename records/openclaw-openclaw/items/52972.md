## Summary

Keep open. Current main still has the reported source-level gap: the reply guard suppresses the false reminder note for native cron-tool success or same-session cron rows, but assistant shell/CLI `openclaw cron add --at` creates jobs through the Gateway RPC path without incrementing the same-turn `successfulCronAdds` signal.

## What This Changes

The issue reports that successful cron reminder creation through assistant-triggered CLI or tool paths can still produce the contradictory unscheduled-reminder note.

## Best Possible Solution

Add a narrow verified scheduling signal so successful current-turn cron creation through assistant shell/CLI paths suppresses the false note, with regression tests and without broad note stripping, regex policy changes, or a new config option.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw cron add --at` can create a cron through Gateway `cron.add`, while current-main telemetry increments `successfulCronAdds` only for native `cron` tool adds.

## Solution Assessment

No. Current main is not the best solution because it recognizes only the native cron-tool success path; the maintainable fix is to record verified successful CLI-through-exec scheduling without stripping notes broadly.
