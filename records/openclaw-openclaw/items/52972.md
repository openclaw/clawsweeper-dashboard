## Summary

Keep open. Current main still has the cron-success telemetry gap: the reply guard trusts `successfulCronAdds`, but the embedded and Codex tool telemetry still count only successful native `cron` tool adds, while `openclaw cron add --at` creates the job through the CLI/Gateway path without updating that current-turn signal.

## What This Changes

The issue reports that Telegram replies can include the unscheduled-reminder note even after a cron reminder was successfully scheduled through `openclaw cron add --at`.

## Best Possible Solution

Add a narrow, verified scheduling signal so successful current-turn cron creation through assistant shell/CLI paths suppresses the false note, with focused regression tests and without changing broader reminder-guard UX policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw cron add --at` can create the cron through `cron.add`, but current-main telemetry only increments `successfulCronAdds` for native `cron` tool adds, leaving the reply guard free to append the false note.

## Solution Assessment

No. Current main is not the best solution because it recognizes only the native cron-tool success path; the maintainable fix is to record verified successful CLI-through-exec scheduling without broad note stripping, regex policy changes, or a new config option.
