## Summary

## What This Changes

The issue reports that successful shell-driven cron reminder scheduling can still receive the misleading unscheduled-reminder note.

## Best Possible Solution

Land or replace the linked fix with a narrow counter-path change that counts successful shell-driven `openclaw cron add` results before the reply guard runs, with focused regression coverage and real behavior proof.

## Reproduction Assessment

Yes, at source level: current main increments `successfulCronAdds` only for structured cron tool adds, while the reply guard appends the note when reminder text has zero successful cron adds. I did not run a live Telegram or cron scenario in this read-only review.

## Solution Assessment

Yes, the best fix shape is to count successful shell-driven cron adds at the telemetry producer before the reply guard runs, rather than hiding the note only in Telegram delivery. The current linked PR is in the right ownership area but still needs review, repair, or replacement before merge.
