## Summary

Keep open. Current main still has the reported cron-success telemetry gap: the reminder guard suppresses the note only when `successfulCronAdds` is nonzero or an enabled same-session cron already exists, while the embedded tool handler still increments that counter only for the native `cron` tool path. A successful `openclaw cron add --at ...` run through `exec`/`bash` can reach Gateway `cron.add` through the CLI without updating the current-turn counter, and the prior exact fix PR #53024 is closed unmerged while #47589 and #64486 cover adjacent reminder-guard policy rather than this shell/CLI success path.

## What This Changes

Fix the bug where successful cron reminder scheduling through the assistant shell/exec CLI path can still be followed by the misleading unscheduled-reminder note.

## Best Possible Solution

Implement a narrow scheduling-signal fix so successful current-turn reminders are recognized consistently across the native cron tool and the assistant shell/exec CLI path, with regression tests for successful, failed, and unrelated shell commands plus final reply behavior. Keep broader regex narrowing, Telegram sanitization, config opt-outs, plugin-created scheduler signals, and commitment-gating policy separate.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is visible: an embedded run can execute a successful `exec`/`bash` command that invokes `openclaw cron add --at ...`, the CLI creates the Gateway cron job, but `successfulCronAdds` stays zero because only the native `cron` tool path increments it; a reminder-like final reply then still satisfies the note-append guard when no matching same-session cron covers it.

## Solution Assessment

No. Current main is not the best solution because it recognizes only native `cron` tool success; the narrow maintainable fix is to record successful current-turn cron scheduling for the CLI-through-exec path without changing unrelated reminder regex, Telegram delivery, or broader commitment-gating behavior.
