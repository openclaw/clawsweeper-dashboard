## Summary

Keep this open: current main and v2026.6.8 still have the cron run-history session-key mismatch, and the only direct fix PR was closed unmerged without becoming a safe canonical fix.

## What This Changes

The issue reports that cron run history's Open Chat button redirects to the wrong session for cron jobs in 2026.5.6 through 2026.5.27.

## Best Possible Solution

Resolve cron run-log sessionKey values to the matching transcript, or change Open Chat to a stored canonical session key that still displays the selected run's history, with focused regression coverage and real Control UI proof.

## Reproduction Assessment

Yes at source level: run history opens entry.sessionKey while cron session persistence stores only the stable agentSessionKey. A live click-through reproduction was not run in this read-only review.

## Solution Assessment

Unclear as a final code path: the closed alias-persistence PR is the narrowest known repair, but maintainers should confirm it is the intended session-state model because current tests intentionally avoid storing visible duplicate cron run aliases.
