## Summary

Keep open: current main still uses unbounded success-path JSON reads in the two Discord paths named by this PR, and the patch uses the existing bounded SDK helper without introducing a concrete correctness or security defect. The remaining review concern is coverage and scope: the PR body proves the behavior with a terminal run, but the success-path overflow regression is not committed as a test, and a sibling webhook success JSON reader remains outside this PR.

## What This Changes

The PR replaces `res.json()` with `readProviderJsonResponse` for Discord PluralKit lookups and voice upload URL responses, and adjusts the PluralKit test mock for the bounded reader.

## Best Possible Solution

Land the narrow bounded-read fix after required checks, ideally with focused success-overflow regression coverage or an explicit follow-up for the remaining Discord webhook success JSON reader.

## Reproduction Assessment

Yes, from source: current main still calls unbounded `res.json()` in the PluralKit lookup and voice upload URL success paths. I did not run a local repro because this review was read-only, but the PR body includes after-fix terminal output showing both streams canceled at the cap.

## Solution Assessment

Yes for the two named paths: using `readProviderJsonResponse` is the existing SDK helper for bounded JSON responses and stays within the extension boundary. It is not a complete sweep of every Discord success JSON reader, so the webhook reader should be a follow-up or an intentional scope decision.
