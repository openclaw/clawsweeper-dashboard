## Summary

Keep open: current main does not implement the proposed tool-temperature policy, but this PR is not mergeable as-is because it suppresses the new default based on reasoning visibility, misses compaction tool sessions, carries unrelated secrets changes, and lacks real behavior proof.

## What This Changes

The PR adds embedded-runner heuristics that inject temperature 0.3 for tool-enabled turns and 0.1 for execution-like tool turns, plus tests, changelog, lockfile, and stale file-secret allowInsecurePath plumbing.

## Best Possible Solution

If maintainers accept the sampling policy, land a focused agents-only patch that uses actual provider thinking state, covers normal and compaction tool-capable runs, preserves cache semantics, drops unrelated secrets/lockfile churn, and includes real provider proof.

## Reproduction Assessment

Yes for the PR blockers by source inspection: PR head gates on reasoningLevel, only wires the normal attempt path, and carries unrelated stale secrets changes. The requested sampling default itself is a new policy behavior, not a broken current-main contract.

## Solution Assessment

No. The extra-params layer is plausible, but this branch is not the best solution until maintainers accept the default-sampling policy, the thinking and compaction gaps are fixed, stale secrets changes are removed, and real behavior proof is supplied.
