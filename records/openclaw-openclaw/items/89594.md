## Summary

Keep open: current main still has a source-visible gap for Microsoft Teams inbound channel file attachments, because the plugin only does Graph media lookup when Teams sends attachment markup in the activity text and there is no evidence that main now fetches full Graph message context for ordinary channel file uploads or threaded replies.

## What This Changes

The issue requests Microsoft Teams inbound channel attachments be discoverable to OpenClaw, including Graph lookup for top-level channel messages and threaded replies where Teams omits attachment references from the Bot Framework payload.

## Best Possible Solution

Add a maintainer-approved Teams plugin policy for inbound Graph message lookup that preserves the current default unless explicitly enabled, handles top-level and reply Graph URLs, and includes live Teams proof plus focused regression tests.

## Reproduction Assessment

Yes from source inspection: the current Teams handler only attempts Graph attachment resolution when activity text contains '<attachment ', so the reported Teams channel upload shape with no Bot Framework attachment marker will not be recognized. I did not run a live Microsoft Teams tenant repro, and the existing labels correctly call for live repro before repair.

## Solution Assessment

Unclear as an implementation path: the issue identifies the right missing Graph lookup, but adding per-message Graph queries or a per-channel opt-in is a product/config decision because it changes external API calls, permissions, and runtime cost.
