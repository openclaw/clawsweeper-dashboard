## Summary

## What This Changes

The PR changes three OpenAI plugin success-response reads to use shared bounded response helpers for Batch status, ChatGPT OAuth token responses, and Realtime client-secret responses.

## Best Possible Solution

Land this cap-based OpenAI hardening only after maintainers explicitly accept the shared 16 MiB success-response policy and the PR body includes redacted changed-path proof for Batch status, OAuth exchange/refresh, and Realtime client-secret responses.

## Reproduction Assessment

Yes, at source level. Current main reaches raw response.json() and arrayBuffer() in the targeted OpenAI paths, so an oversized streaming success body would exercise the unbounded buffering path; I did not run live OpenAI endpoints.

## Solution Assessment

Yes for the helper boundary, not yet for merge readiness. Reusing the shared bounded readers with the shared 16 MiB cap is the narrow maintainable fix, while the remaining gap is changed-path real behavior proof and maintainer acceptance of the compatibility tradeoff.
