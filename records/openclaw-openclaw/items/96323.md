## Summary

Keep open: the hardening goal is useful, but this branch is not merge-ready because 1 MiB caps can reject valid image and batch payloads, OAuth is capped after the first unbounded read, lint is failing, and the posted proof does not exercise the changed OpenAI paths.

## What This Changes

The PR replaces several OpenAI plugin JSON/text response reads with `readResponseWithLimit` at a 1 MiB cap and adds local HTTP/unit proof coverage.

## Best Possible Solution

Use bounded reads at the first body-read boundary with payload-specific caps, repair the added tests, and add redacted changed-path proof for oversized rejection plus normal success.

Yes at source level: current main has raw JSON/text/arrayBuffer reads on the changed OpenAI paths, and the PR's cap and OAuth-placement failures are visible from source inspection. I did not run a live OpenAI endpoint in this read-only review.

## Solution Assessment

No. `readResponseWithLimit` is the right primitive, but this patch uses one-size 1 MiB caps for payload bodies and applies the OAuth limit after the first untrusted body read.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Raise the image response cap for real payloads:** `extensions/openai/image-generation-provider.ts:82`
  - body: This adds a 1 MiB cap to direct OpenAI image success JSON, but that response can carry base64 image data and the provider supports large sizes and multiple results. Use an image-payload-sized cap or another bounded strategy that preserves valid outputs.
  - confidence: 0.94
- **[P1] Do not cap batch output files at one MiB:** `extensions/openai/embedding-batch.ts:117-121`
  - body: `fetchOpenAiFileContent` reads completed batch output and error files, not only small metadata. Existing batch settings allow large request groups, so a 1 MiB file-content cap can reject valid embedding batch results.
  - confidence: 0.93
- **[P2] Bound OAuth bodies before postTokenForm buffers them:** `extensions/openai/openai-chatgpt-oauth-flow.runtime.ts:226-233`
  - body: `postTokenForm` already reads the token endpoint body with `arrayBuffer()` before this bounded parse runs, so the untrusted body is fully buffered first. Move the byte cap into the first read and cover exchange, refresh, and non-OK token responses consistently.
  - confidence: 0.91
- **[P3] Fix the added test lint failures:** `extensions/openai/openai-chatgpt-oauth-flow.runtime.test.ts:179-183`
  - body: The latest CI lint run fails on the new tests because the OAuth test leaves `canceled` unused and the new stream helpers use braceless `if` statements. Clean up those additions so the extension lint lanes can pass.
  - confidence: 0.96
