## Label Justifications

- `P2`: This is a bounded OpenAI provider hardening PR with important but localized plugin impact.
- `merge-risk: 🚨 compatibility`: The PR can make existing valid OpenAI image generation and embedding batch outputs fail by capping payload-bearing success bodies at 1 MiB.
- `merge-risk: 🚨 availability`: Merging as-is can turn normal large image/batch provider responses into runtime failures while leaving the OAuth OOM path unbounded.

## Summary

Keep open: the bounded-read direction is useful, but this patch is not mergeable because it applies 1 MiB caps to valid large image and batch payloads, leaves the OAuth body prebuffered before the new cap runs, and lacks after-fix real behavior proof.

## What This Changes

The PR changes OpenAI plugin response parsing to use `readResponseWithLimit` with 1 MiB caps in realtime secrets, OAuth token exchange, image generation, and embedding batch status/file reads.

## Best Possible Solution

Move bounded reads to the first body read in each path and use payload-specific caps that preserve valid image and batch outputs while still capping metadata-sized responses; add focused under-cap and overflow proof.

## Reproduction Assessment

Yes from source. The PR head applies a 1 MiB cap to image and batch file-content success bodies, while current source/docs show those paths carry large base64 image and batch output payloads; OAuth also still prebuffers in `postTokenForm` before the changed parse.

## Solution Assessment

No. `readResponseWithLimit` is the right primitive, but the current patch applies it at the wrong OAuth layer and with one-size-fits-all caps that break valid payload classes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Raise the image response cap for real payloads:** `extensions/openai/image-generation-provider.ts:85`
  - body: This 1 MiB cap is applied to `/images/generations` and `/images/edits` success bodies before parsing, but this path returns `b64_json` image payloads and OpenClaw documents large OpenAI image requests. Valid generated images can exceed 1 MiB, so this would reject normal image generation before decoding.
  - confidence: 0.94
- **[P1] Do not cap batch output files at one MiB:** `extensions/openai/embedding-batch.ts:117-121`
  - body: `fetchOpenAiFileContent` reads completed batch output files, not just small metadata. Existing code allows 50k requests and 190 MiB JSONL input groups, so capping the output file read at 1 MiB will break valid embedding batches; use a separate larger file-output cap from the small status JSON cap.
  - confidence: 0.93
- **[P2] Bound OAuth bodies before postTokenForm buffers them:** `extensions/openai/openai-chatgpt-oauth-flow.runtime.ts:229-233`
  - body: `postTokenForm` already reads the network response with `response.arrayBuffer()` before this bounded parse runs, so the untrusted token body is fully buffered first. Move the byte cap into the first read in `postTokenForm` and make sure refresh/error token responses share the same bound.
  - confidence: 0.91
