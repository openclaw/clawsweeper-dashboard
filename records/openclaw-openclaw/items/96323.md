## Label Justifications

- `P2`: This is a localized OpenAI plugin hardening PR with normal-priority security/availability value but merge-blocking compatibility issues.
- `merge-risk: 🚨 compatibility`: The PR can make existing valid OpenAI image generation and embedding batch outputs fail by capping payload-bearing success bodies at 1 MiB.

## Summary

Keep open: the hardening direction is useful, but this head is not merge-ready because the uniform 1 MiB caps can reject valid image and batch payloads, the OAuth path still buffers before the cap, and the new proof does not exercise the changed provider paths.

## What This Changes

The PR replaces selected OpenAI plugin JSON/text response reads with `readResponseWithLimit` and adds OAuth, realtime, image mock, and local HTTP proof coverage.

## Best Possible Solution

Land bounded reads at the first body-read boundary, with small metadata caps for OAuth/realtime/status JSON and payload-specific caps for image and batch output bodies.

## Reproduction Assessment

Yes from source inspection: current main has unbounded reads on the changed OpenAI paths, and the PR head would cap payload-bearing responses at 1 MiB while leaving OAuth first-read buffering in place. I did not run a live OpenAI request in this read-only review.

## Solution Assessment

No. `readResponseWithLimit` is the right primitive, but this patch applies it at the wrong OAuth layer and uses one-size-fits-all caps that are too small for valid payload classes.
