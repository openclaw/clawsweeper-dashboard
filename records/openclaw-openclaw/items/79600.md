## Summary

Keep this PR open: it targets a real Discord default-account token fallback bug that is still present on current main and the latest release, but the patch is not merge-ready because it changes only runtime token resolution while sibling inspection/setup/doctor/action-discovery paths still report the same config as unconfigured, and the external PR still lacks real behavior proof.

## What This Changes

This PR changes Discord token resolution so a blank `channels.discord.accounts.default.token` can fall through to `channels.discord.token` or `DISCORD_BOT_TOKEN`, with focused resolver tests.

## Best Possible Solution

Use one tested Discord default-account fallback rule across runtime token resolution, account inspection, setup inspection, doctor/status/action discovery, and require redacted real gateway or REST-path proof before merge.

## Reproduction Assessment

Yes, from source: current `resolveDiscordToken` returns `missing` before env fallback whenever `accounts.default.token` exists but resolves missing. I did not run tests in this read-only review, but the current source and the PR's added cases give a high-confidence reproduction path.

## Solution Assessment

No, not yet; changing only `resolveDiscordToken` is too narrow because sibling inspection and setup paths still encode the old blank-token rule. The best fix is a single tested fallback invariant across runtime and user-visible inspection surfaces.
