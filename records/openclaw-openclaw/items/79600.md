## Best Possible Solution

Use one tested Discord default-account fallback rule across runtime token resolution, account inspection, setup inspection, doctor/status/action discovery, then require redacted real gateway or REST-path proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main returns `missing` before env fallback whenever `accounts.default.token` exists but resolves missing. I did not run tests in this read-only review, but the current source and the PR's added cases give a high-confidence path.

## Solution Assessment

No, not yet; changing only `resolveDiscordToken` is too narrow because sibling inspection and setup paths still encode the old blank-token rule. The best fix is a single tested fallback invariant across runtime and user-visible inspection surfaces.
