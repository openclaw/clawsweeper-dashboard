## Summary

## What This Changes

The PR changes Discord default-account token resolution so a blank `channels.discord.accounts.default.token` falls through to `channels.discord.token` or `DISCORD_BOT_TOKEN`, and adds resolver tests.

## Best Possible Solution

Use one tested Discord default-account fallback rule across runtime token resolution, account inspection, setup inspection, doctor/status/action discovery, then require redacted real gateway or REST-path proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main returns `missing` before top-level or env fallback whenever `accounts.default.token` exists but resolves missing. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No, not yet; changing only `resolveDiscordToken` is too narrow because sibling inspection/setup paths still encode the old blank-token rule. The best fix is a single tested fallback invariant across runtime and user-visible inspection surfaces.
