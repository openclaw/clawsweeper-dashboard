## What This Changes

The PR changes Discord default-account token resolution so a blank `accounts.default.token` can fall through to top-level `channels.discord.token` or `DISCORD_BOT_TOKEN`, with resolver tests added.

## Best Possible Solution

Land one Discord default-account credential fallback invariant across runtime, setup/status inspection, doctor/action discovery, and tests, with an explicit maintainer-approved blank-token compatibility decision and redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main returns `missing` when `accounts.default.token` exists but is blank, before checking top-level config or `DISCORD_BOT_TOKEN`. I did not run a live gateway in this read-only review.

## Solution Assessment

No, not yet. The resolver change is a plausible mitigation, but the best fix needs the same fallback invariant across sibling inspection/status paths and an explicit compatibility decision for blank default-account tokens.
