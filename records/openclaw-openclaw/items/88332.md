## Summary

Keep open. Current main still has the reported shared Codex/OpenClaw path: failed Codex-native command items become `lastToolError` and can be rendered as channel-visible `⚠️ ... failed` warnings, while related channel reports remain open as repro anchors and the merged Discord heartbeat PR only fixed one delivery-policy path.

## What This Changes

The issue asks OpenClaw to distinguish real Codex-native tool failures from expected negative checks and internal probes before producing channel-visible failed-tool warnings.

## Best Possible Solution

Define a shared Codex-native status/mutation mapping that suppresses expected negative checks and internal probes from `lastToolError` fallback warnings while preserving visible warnings for real mutating failures, then prove it with Codex projector, payload-policy, and one external-channel regression path.

## Reproduction Assessment

Yes, source-level rather than live: a failed Codex `commandExecution` item with exit code 1 flows into `lastNativeToolError` as a mutating `bash` error, then `buildEmbeddedRunPayloads` can render `⚠️ ... failed`. I did not run a live Telegram or Discord replay in this read-only review.

## Solution Assessment

No, a channel-only sanitizer or grep-string special case would not be the best fix. The maintainable path is a shared Codex-native status and fallback-warning policy with sibling channel proof.
