## What This Changes

The PR updates `src/acp/event-mapper.ts` to emit image `ToolCallContent` from tool result content blocks and adds helper-level tests for text, image, mixed, malformed, and fallback cases.

## Best Possible Solution

Land a narrow ACP mapper/test update after redacted real-client proof shows a tool-produced PNG/JPG result rendering through the ACP bridge, then refresh mergeability and validation against current main.

## Reproduction Assessment

Yes, source-level: current main routes ACP partial and final tool outputs through `extractToolCallContent()`, and that helper only emits text blocks while read/image tool results can contain `{ type: "image", data, mimeType }`. I did not run a live ACP client reproduction in this read-only review.

## Solution Assessment

Yes for the code shape: the ACP event mapper is the central conversion point, and the pinned ACP SDK already accepts image content in tool-call updates. The remaining blocker is proof of real ACP client behavior, not a different implementation layer.
