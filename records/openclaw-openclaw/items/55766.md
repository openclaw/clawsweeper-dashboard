## Summary

This PR is still useful directionally but is not merge-ready: the mapper-level image branch does not cover the normal sanitized ACP event path, which currently removes inline image data before the mapper can forward it, and the PR still lacks real ACP client proof.

## What This Changes

The PR extends ACP tool-call content mapping to emit image content blocks from tool results and adds unit tests for text, image, mixed, malformed, empty, whitespace-only, and string fallback cases.

## Best Possible Solution

Thread an ACP-approved image payload or media reference through the sanitized tool-result pipeline, cover the handler-to-mapper path, and merge only with redacted real ACP client rendering proof.

## Reproduction Assessment

Yes, from source inspection: the ACP tool-call update/result path receives sanitized tool results, and current sanitizer tests assert image blocks lose their data field. That gives a high-confidence production-path reproduction without running a live ACP client.

## Solution Assessment

No. The mapper change matches the protocol type, but the maintainable fix needs an explicit ACP media/redaction contract that survives sanitization before the mapper converts content for clients.
