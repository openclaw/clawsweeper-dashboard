## Summary

Keep open: this is a useful ACP feature, but the current PR only maps raw image blocks while the production ACP tool-event path receives sanitized blocks without `data`, so it still does not prove image rendering for real clients.

## What This Changes

The PR adds `image` block extraction to `extractToolCallContent()` and unit tests for raw text, image, malformed, empty, whitespace-only, and string inputs.

## Best Possible Solution

Carry an explicitly ACP-safe image payload or media reference through the sanitized tool-event pipeline, add coverage for that real path, and land only with redacted ACP client proof that a tool-result image renders.

## Reproduction Assessment

Yes. Source inspection shows current main routes ACP tool updates through `extractToolCallContent()`, and the real tool-event path strips image `data` before ACP mapping.

## Solution Assessment

No. Mapping raw image blocks in the helper is only sufficient when the helper receives raw bytes; production ACP events receive sanitized blocks, so the fix needs to bridge that boundary safely.
