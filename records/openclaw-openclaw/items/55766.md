## Summary

Keep open: current main still drops ACP tool-result image blocks, and this focused PR is at the right shared mapper boundary with earlier validation feedback addressed, but the external contributor has not supplied real ACP client behavior proof.

## What This Changes

The PR updates `extractToolCallContent()` to include valid ACP image blocks from tool results and adds focused unit coverage for text, image, mixed, malformed, empty, and string-fallback cases.

## Best Possible Solution

Land the mapper-level image forwarding only after the contributor adds redacted real ACP client proof and maintainers refresh the review against the current branch head.

## Reproduction Assessment

Yes at source level: current main sends ACP tool update and result payloads through `extractToolCallContent()`, whose content loop only emits text blocks, so image-only tool results are dropped from ACP `content`. I did not establish a live ACP client reproduction in this read-only review.

## Solution Assessment

Yes, this appears to be the best fix layer: the ACP event mapper is the shared conversion point for both partial and final tool outputs, and the pinned ACP SDK already models image content inside tool-call content. Client-specific handling would duplicate the invariant.
