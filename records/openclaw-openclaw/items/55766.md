## Summary

Keep open. Current main still drops image blocks from ACP tool-call content, so the PR is useful and not obsolete; however, the PR head still accepts whitespace-only image `data` or `mimeType`, which can suppress text fallback and send broken image content to ACP clients.

## What This Changes

This PR adds ACP `extractToolCallContent()` support for image content blocks in tool results and expands mapper unit coverage for text, image, mixed, malformed, empty, and string fallback cases.

## Best Possible Solution

Finish the PR or land an equivalent maintainer patch that forwards valid ACP image blocks from tool results, validates both `data` and `mimeType` after trimming, keeps current text/fallback behavior, adds whitespace-only regression coverage, and updates the ACP compatibility docs if maintainers want the newly supported image output advertised.

## Reproduction Assessment

Yes. On current main, `extractToolCallContent({ content: [{ type: "image", data: "iVBORw0KGgo=", mimeType: "image/png" }] })` would produce no content because the helper only emits text blocks; on the PR head, a whitespace-only image field passes the raw `length > 0` guard and returns before fallback text is considered.

## Solution Assessment

No, not yet. The PR targets the right narrow helper and test file, but the current validation should use trimmed non-empty strings before emitting ACP image content.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Reject whitespace-only image fields:** `src/acp/event-mapper.ts:380-383`
  - body: The image branch accepts any non-empty string, so `{ data: "   ", mimeType: "image/png" }` or a whitespace-only `mimeType` is emitted as ACP image content. Because `contents.length > 0` then returns early, fallback `text`/`message`/`error` is skipped and clients can receive a broken image with no readable fallback. Trim both fields before pushing the image block and cover this case in the mapper test.
  - confidence: 0.91
