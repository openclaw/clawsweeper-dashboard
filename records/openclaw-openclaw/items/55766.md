## Summary

Keep open: the PR is still aimed at a useful ACP capability, but it does not fix the production path because tool results are sanitized before ACP translation and image data is removed there. It also still lacks real ACP client behavior proof.

## What This Changes

The PR adds image block handling to ACP `extractToolCallContent()` plus unit tests for text, image, malformed, empty, whitespace-only, and string fallback tool results.

## Best Possible Solution

Thread an ACP-safe image payload or media reference through sanitized tool events, add handler-to-translator coverage, and verify rendering in a real ACP client.

## Reproduction Assessment

Yes, from source inspection. Tool update/result events pass through `sanitizeToolResult()` before ACP translation, and that sanitizer deletes image `data`, so the PR's raw-data branch will not fire for the real production path it is meant to fix.

## Solution Assessment

No. Mapping raw image blocks inside `extractToolCallContent()` is not enough; the maintainable fix needs an ACP-safe image payload or media reference that survives the sanitized event pipeline.
