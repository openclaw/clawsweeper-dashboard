## Summary

Keep open: the PR addresses a real ACP image-output gap, but the current diff still maps only raw image blocks while the production ACP event path receives sanitized tool results with image data removed. It also still lacks after-fix real behavior proof from an ACP client.

## What This Changes

This PR adds image block extraction to ACP `extractToolCallContent()` with validation and unit coverage for text, image, malformed, empty, whitespace-only, and string inputs.

## Best Possible Solution

Carry an explicitly ACP-safe image payload or media reference through the sanitized tool-event pipeline, cover that real path in tests, and land only after redacted ACP client proof shows a tool-result image renders.

## Reproduction Assessment

Yes, at source level: current main routes ACP tool updates through `extractToolCallContent()`, while the real tool-event handlers sanitize results first and strip image `data`. I did not run a live ACP client, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. Mapping raw image blocks in the helper is insufficient for production because ACP receives sanitized blocks; the fix needs to bridge the sanitizer/media boundary safely.
