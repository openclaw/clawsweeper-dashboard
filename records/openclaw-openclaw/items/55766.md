## Summary

Keep open: current main still drops image blocks from ACP tool-call update content, and this focused PR appears to be the right implementation layer, but it cannot be merge-ready until the external contributor supplies real ACP client behavior proof.

## What This Changes

The PR adds ACP tool-call image block extraction in `src/acp/event-mapper.ts` and expands mapper tests for text, image, mixed, missing, empty, and whitespace-only content blocks.

## Best Possible Solution

Keep this focused branch as the ACP mapper implementation candidate, require redacted real ACP client proof, then land it if maintainers still want ACP clients to render tool-result image blocks.

## Reproduction Assessment

Yes, at source level: current main routes ACP partial and final tool outputs through `extractToolCallContent()`, whose content loop only emits text blocks. I did not establish a live ACP client reproduction in this read-only review.

## Solution Assessment

Yes, this appears to be the best implementation layer: the event mapper is the shared conversion point for ACP tool-call update content, and the ACP SDK already supports image content there. Translator-level or client-specific handling would duplicate the invariant.
