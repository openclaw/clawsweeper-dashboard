## Summary

Keep open: the ACP image-forwarding direction is useful, but this PR is not merge-ready because real tool-event image data is stripped before the new mapper branch can see it, and the contributor still has only tests/CI rather than real ACP client proof.

## What This Changes

The PR adds ACP image block forwarding in `extractToolCallContent()` plus mapper unit tests for text, image, malformed, empty, whitespace-only, and string fallback cases.

## Best Possible Solution

Carry an explicitly ACP-safe image payload or media reference through the tool-event pipeline, add tests that exercise the real sanitized event path, and land only with redacted real ACP client proof that a tool-result image renders.

## Reproduction Assessment

Yes. Source inspection shows current main routes ACP tool updates through `extractToolCallContent()`, which only emits text, and also shows the real tool-event path strips image `data` before ACP mapping.

## Solution Assessment

No. The mapper helper is the right narrow surface only if it receives image bytes, but this PR maps only raw image blocks while production ACP events receive sanitized image blocks; the fix needs to bridge that boundary safely.
