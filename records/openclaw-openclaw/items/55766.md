## Summary

Keep open: the PR’s helper-level image extraction does not make the claimed ACP client behavior work on the normal production path, because tool update/result payloads are sanitized before the ACP mapper receives them and current sanitization strips image bytes.

## What This Changes

This PR extends ACP tool-call result mapping so image content blocks can be forwarded alongside text content blocks.

## Best Possible Solution

Thread an ACP-approved image payload or safe media reference through the sanitized tool-result pipeline, cover the handler-to-mapper path, and require redacted real ACP client rendering proof before merge.

## Reproduction Assessment

Yes for the PR correctness issue: source inspection shows production ACP tool-call updates receive sanitized tool results, and current sanitizer tests assert image blocks lose `data`. I did not run a live ACP client reproduction.

## Solution Assessment

No. The mapper change matches the ACP protocol type, but the maintainable fix needs an explicit ACP media/redaction contract that survives sanitization before the mapper converts content for clients.
