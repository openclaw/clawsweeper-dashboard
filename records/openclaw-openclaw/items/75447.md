## Summary

Keep this issue open. Current main still has an `openclaw update` path that refreshes recorded ClawHub plugin installs after the core update, and the bundled-plugin sync step does not protect a current bundled plugin from a same-id ClawHub install record before that refresh runs.

## What This Changes

The issue asks for `openclaw update` to avoid automatically installing a ClawHub WhatsApp plugin that is older than the bundled WhatsApp plugin.

## Best Possible Solution

Add a generic bundled-plugin freshness guard in the plugin update flow so a current bundled plugin is not automatically replaced by an older same-id external ClawHub or npm package; default noninteractive updates should skip or restore bundled with a clear warning, while explicit manual installs can still opt into external packages.

## Reproduction Assessment

Yes. A focused unit reproduction can seed `plugins.installs.whatsapp` as a ClawHub record, mock a ClawHub update result for version `2026.2.9`, provide a bundled WhatsApp source at `2026.4.25`, and show current `updateNpmInstalledPlugins` updates the external record because no bundled-version guard runs.

## Solution Assessment

Yes, a generic bundled-plugin precedence/freshness guard is the right direction. It should not hardcode WhatsApp; the safe fix belongs in `src/plugins/update.ts` with tests for ClawHub and, if practical, npm/external records that share a bundled plugin id.
