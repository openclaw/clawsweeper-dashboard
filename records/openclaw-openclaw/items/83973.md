## Label Justifications

- `P1`: The PR touches Teams owner-trust enforcement for agent/tool execution and has a likely blocking regression for active owner Teams turns.
- `merge-risk: 🚨 compatibility`: Merging this as-is can make existing Teams owner or allowlisted workflows lose owner-only tool access during normal message dispatch.

## Summary

Keep this PR open: the security-hardening direction is valid, but the patch appears to over-apply the downgrade by marking the message preview before the same Teams turn dispatches, which can make authorized/owner Teams messages lose owner-only tool access.

## What This Changes

The PR adds non-owner/untrusted metadata to Microsoft Teams message-preview, thread-parent, and reaction `enqueueSystemEvent` calls and adds focused regression assertions.

## Best Possible Solution

Keep the non-owner metadata for deferred/skipped external Teams context, but avoid letting the current Teams message preview downgrade the same turn; add regression coverage for an owner or allowlisted active Teams dispatch plus the skipped/thread/reaction cases.

## Reproduction Assessment

Yes by source inspection: enqueueing the current Teams message preview before `core.channel.turn.run`, then draining `forceSenderIsOwnerFalse` system events in `get-reply-run`, is enough to reproduce the owner downgrade path. I did not execute tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The intended skipped/deferred-context hardening is appropriate, but applying the downgrade to every message preview before the same turn dispatches is broader than the safe fix.

- **[P1] Avoid downgrading the active Teams turn from its own preview:** `extensions/msteams/src/monitor-handler/message-handler.ts:504-505`
