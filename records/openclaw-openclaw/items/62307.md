## Summary

Keep open: current main still lacks WhatsApp edit/revocation handling and the Baileys contract supports the intended event, but this PR is not merge-ready because the head still conflates buffered and in-flight edits and has no real WhatsApp runtime proof.

## What This Changes

The PR adds WhatsApp `messages.update` handling in `extensions/whatsapp/src/inbound/monitor.ts` to update buffered edited messages, ignore revocations, and notify users about edits after a message was processed.

## Best Possible Solution

Implement explicit buffered, in-flight, and processed message states in the WhatsApp monitor, add focused edit/revocation/access-control tests, rebase onto current main, and provide redacted real WhatsApp proof before merge.

## Reproduction Assessment

Yes. A focused source-level test can enqueue a WhatsApp message, hold `onMessage` on a deferred promise, emit `messages.update` before resolving it, and observe that the PR still treats the ID as pending after the payload was handed off.

## Solution Assessment

No. The feature direction is reasonable, but the branch should distinguish buffered messages from in-flight processing instead of leaving delivered entries in the buffered map until `onMessage` resolves.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Separate in-flight edits from buffered edits:** `extensions/whatsapp/src/inbound/monitor.ts:162-168`
