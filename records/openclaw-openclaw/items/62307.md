## Summary

Keep open: current main still lacks inbound WhatsApp edit/revocation handling, and the PR targets a real Baileys event surface, but the branch is conflicting, lacks real WhatsApp proof, and still misclassifies edits that arrive while the agent is already processing.

## What This Changes

The PR adds WhatsApp `messages.update` handling in `extensions/whatsapp/src/inbound/monitor.ts` for inbound user message edits and revocations, including buffered edit replacement, processed-message tracking, and a late-edit notice.

## Best Possible Solution

Implement inbound WhatsApp edit/revocation handling with explicit buffered, in-flight, and processed states, focused timing and access-control coverage, a current-main rebase, and redacted live WhatsApp proof.

## Reproduction Assessment

Yes, for the PR defect by source inspection: enqueue a debounced WhatsApp batch, let `onFlush` compute and pass `messageToDeliver`, keep `onMessage` unresolved, then emit `messages.update`; the handler still sees the original message ID in `pendingMessageIds`. I did not run a test in this read-only review.

## Solution Assessment

No. The feature direction is reasonable, but the branch should separate buffered, in-flight, and processed states instead of leaving delivered entries in the buffered map until `onMessage` resolves.
