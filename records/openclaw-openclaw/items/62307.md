## Summary

Keep open: current main does not implement WhatsApp inbound edit/revocation handling, but this branch is not merge-ready because it is conflicting, lacks real WhatsApp proof, and leaves in-flight edits in the buffered path where stale text can be processed without notice.

## What This Changes

The PR modifies WhatsApp inbound monitoring to listen for Baileys `messages.update`, mutate buffered edited messages, notify on late edits, and ignore revocations.

## Best Possible Solution

Land a current-main WhatsApp plugin implementation that models buffered, in-flight, and processed message IDs separately, preserves current debounce/dedupe/close-drain behavior, adds focused timing/access-control tests, and includes redacted live WhatsApp proof for buffered edits, late edits, and revocations.

## Reproduction Assessment

Yes for the blocking PR defect by source inspection: enqueue a debounced WhatsApp message, let `onFlush` pass `messageToDeliver` to an unresolved `onMessage`, then emit `messages.update`; the handler still sees the ID as pending even though the agent may already be processing stale text. I did not run a live WhatsApp test in this read-only review.

## Solution Assessment

No, not as currently implemented. The feature belongs in the WhatsApp plugin, but the maintainable path needs a current-main rebase, distinct buffered/in-flight/processed edit states, and maintainer agreement on the late-edit UX.

Full review comments:

- **[P1] Track in-flight edits separately:** `extensions/whatsapp/src/inbound/monitor.ts:619`
  - body: When `onFlush` is already awaiting `options.onMessage(messageToDeliver)`, the payload has been handed to the agent, but this branch still leaves the ID in `pendingMessageIds`. Move flushed IDs out of the buffered map before delivery and track an explicit in-flight state for the late-edit path.
