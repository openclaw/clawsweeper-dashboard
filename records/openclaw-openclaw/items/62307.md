## Summary

Keep open: current main still does not handle inbound WhatsApp edit/revoke updates, and the feature is not superseded by the separate outbound-action PR; this branch still has a source-reproducible in-flight edit bug and lacks required real WhatsApp proof.

## What This Changes

The PR adds a WhatsApp `messages.update` handler that swaps edited debounced inbound text, ignores revocations, tracks processed message IDs, and sends a late-edit notice.

## Best Possible Solution

Implement WhatsApp inbound edit/revocation handling in the plugin with explicit buffered, in-flight, and processed states, a current-main rebase, focused timing/access-control tests, and redacted live WhatsApp proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: enqueue a debounced WhatsApp message, keep `onMessage` unresolved, emit `messages.update`, and the handler still treats the ID as pending after the payload has been handed to the agent. I did not run that test in this read-only review.

## Solution Assessment

No. The feature direction is reasonable, but the branch should separate buffered, in-flight, and processed states instead of leaving delivered entries in the buffered map until `onMessage` resolves.
