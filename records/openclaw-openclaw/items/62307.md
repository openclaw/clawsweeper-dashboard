## Summary

Keep open. The WhatsApp edit/revocation feature is still absent from current main and the PR is useful source work, but the latest PR head still conflates buffered and in-flight edits and lacks the required user-facing changelog entry.

## What This Changes

This PR updates WhatsApp inbound monitoring to listen for Baileys `messages.update`, replace buffered edited message bodies, ignore revocations, and notify users when edits arrive after a processed message.

## Best Possible Solution

Implement the WhatsApp edit feature on current main with explicit buffered, in-flight, and processed states, focused monitor tests for the timing and access-control cases, and an Unreleased changelog entry.

## Reproduction Assessment

Yes. A focused test can enqueue a WhatsApp message, hold `onMessage` on a deferred promise, emit a `messages.update` edit before resolving it, and observe that the PR still treats the ID as pending after the agent payload has already been handed off.

## Solution Assessment

No. The approach is close, but the state model should move IDs out of buffered state before awaiting delivery and track in-flight handling separately from fully processed messages.
