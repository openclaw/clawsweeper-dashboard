## Summary

Keep this PR open: current main still lacks WhatsApp edit/revocation handling, but the branch has a real in-flight edit state bug and no after-fix real WhatsApp behavior proof. Prior review comments already drove several fixes, and the remaining blocker is narrow but still affects user-visible correctness.

## What This Changes

This PR modifies the WhatsApp inbound monitor to listen for Baileys message update events, track buffered and processed message IDs, apply pending edits before delivery, ignore revocations, and notify users about edits after processing.

## Best Possible Solution

Implement WhatsApp edit support with explicit buffered, in-flight, and processed states, focused monitor tests for edit timing and revocation handling, and maintainer-owned release-note handling at landing.

## Reproduction Assessment

Yes. A focused source-level test can enqueue a WhatsApp message, hold `onMessage` on a deferred promise, emit `messages.update` before resolving it, and observe that the PR still treats the ID as pending after the payload was handed off.

## Solution Assessment

No. The feature direction is reasonable, but the branch conflates buffered and in-flight messages; entries should leave the buffered map before awaiting delivery and use an explicit in-flight state for edits that arrive while the agent is processing.
