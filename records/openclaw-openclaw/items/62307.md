## Summary

Keep open: current main still lacks WhatsApp edit/revocation handling, but this PR is not merge-ready because the head still misclassifies edits that arrive while `onMessage` is in flight and has no real WhatsApp proof.

## What This Changes

The PR adds WhatsApp `messages.update` handling in `extensions/whatsapp/src/inbound/monitor.ts` to swap buffered edited text, ignore revocations, track processed IDs, and send a late-edit notice.

## Best Possible Solution

Implement explicit buffered, in-flight, and processed message states in the WhatsApp monitor, rebase onto current main, add focused edit/revocation/access-control timing tests, and provide redacted real WhatsApp proof.

## Reproduction Assessment

Yes for the PR bug. A focused source-level check can enqueue a WhatsApp message, hold `onMessage` unresolved, emit `messages.update`, and observe that the PR still treats the ID as pending after the payload has been handed to `onMessage`.

## Solution Assessment

No. The feature direction is reasonable, but the branch should distinguish buffered, in-flight, and processed messages instead of leaving delivered entries in the buffered map until `onMessage` resolves.
