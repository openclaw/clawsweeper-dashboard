## What This Changes

The PR adds a WhatsApp `messages.update` listener to mutate buffered edited messages, send a late-edit notice, and ignore revocations in `extensions/whatsapp/src/inbound/monitor.ts`.

## Best Possible Solution

Land a current-main WhatsApp plugin implementation that separates buffered, in-flight, and processed edit states, preserves existing debounce/dedupe/close-drain behavior, adds focused timing and access-control coverage, and includes redacted live WhatsApp proof.

## Reproduction Assessment

Yes, for the blocking PR defect by source inspection: enqueue a debounced WhatsApp message, let `onFlush` pass `messageToDeliver` to an unresolved `onMessage`, then emit `messages.update`; the handler still sees the ID as pending even though stale text may already be processing. I did not run a live WhatsApp test in this read-only review.

## Solution Assessment

No, not as currently implemented. The feature belongs in the WhatsApp plugin, but the maintainable path needs a current-main rebase, distinct buffered/in-flight/processed edit states, and maintainer agreement on the late-edit UX.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Track in-flight edits separately:** `extensions/whatsapp/src/inbound/monitor.ts:619`
  - body: When `onFlush` is already awaiting `options.onMessage(messageToDeliver)`, the payload has been handed to the agent, but this branch still leaves the ID in `pendingMessageIds`. A later `messages.update` event mutates `pending.body`; for batched messages the combined body was already computed, and for single messages the agent may already have read the original, so stale text can be processed with no late-edit notice. Move flushed IDs out of the buffered map before delivery and track an explicit in-flight state for the late-edit path.
