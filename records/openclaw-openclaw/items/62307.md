## Summary

Keep open: the feature is useful and current main still lacks WhatsApp edit/revocation handling, but the PR head has a real in-flight edit state bug and still lacks real WhatsApp after-fix proof.

## What This Changes

The PR adds WhatsApp `messages.update` handling in `extensions/whatsapp/src/inbound/monitor.ts` to update buffered edited messages, ignore revocations, and notify users when an edit arrives after processing.

## Best Possible Solution

Implement explicit buffered, in-flight, and processed message states in the WhatsApp monitor, add focused edit/revocation/access-control tests, rebase onto current main, and provide redacted real WhatsApp proof.

## Reproduction Assessment

Yes, source-level. A focused test can enqueue a debounced WhatsApp message, hold `onMessage`, emit `messages.update` before resolving it, and observe that the PR still treats the ID as pending after handoff.

## Solution Assessment

No. The feature direction is reasonable, but the PR should distinguish buffered messages from in-flight processing instead of keeping delivered entries in the buffered map until `onMessage` resolves.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Separate in-flight edits from buffered edits:** `extensions/whatsapp/src/inbound/monitor.ts:162-168`
  - body: `pendingMessageIds` is not cleared until after `await options.onMessage(messageToDeliver)`, but the update handler treats any pending ID as still buffered.
