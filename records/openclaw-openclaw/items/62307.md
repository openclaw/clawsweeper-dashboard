## Summary

Keep open. The WhatsApp edit/revocation feature is still missing on current main and is directionally valid, but this PR still has a concrete in-flight edit state bug, lacks the required user-facing changelog entry, and is dirty against current main.

## What This Changes

This PR changes the WhatsApp inbound monitor to track buffered and processed message IDs, listen for Baileys `messages.update` edits/revocations, replace pending edited text, and send a late-edit notice after processed messages.

## Best Possible Solution

Implement the feature on current main with explicit buffered, in-flight, and processed edit states, focused WhatsApp monitor tests for edits/revocations/access-control/self-chat timing, and an Unreleased changelog entry.

## Reproduction Assessment

Yes. A focused test can enqueue a WhatsApp message, hold `onMessage` on a deferred promise, emit a `messages.update` edit before resolving it, and observe that the PR still treats the message ID as pending instead of using the late/in-flight path.

## Solution Assessment

No. The PR is close, but keeping IDs in `pendingMessageIds` during `onMessage` conflates buffered and in-flight states; the safer fix is to move entries out of buffered state before awaiting delivery and track in-flight handling explicitly.
