## Summary

Keep this PR open: current main still lacks WhatsApp edit/revocation handling, but the branch is not ready because the edit state model still misclassifies edits that arrive while `onMessage` is running and the external PR has no real WhatsApp after-fix proof.

## What This Changes

The PR modifies `extensions/whatsapp/src/inbound/monitor.ts` to listen for Baileys `messages.update`, swap edited buffered text, ignore revocations, track processed IDs, and send a late-edit notice.

## Best Possible Solution

Implement explicit buffered, in-flight, and processed message states in the WhatsApp monitor, add focused edit/revocation/access-control timing tests, rebase onto current main, and provide redacted real WhatsApp proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: enqueue a debounced WhatsApp message, hold `onMessage` unresolved, emit `messages.update`, and the PR still sees the ID in `pendingMessageIds` after the payload has already been handed to the agent. I did not execute that test in this read-only review.

## Solution Assessment

No. The feature direction is reasonable, but the current branch should separate buffered, in-flight, and processed states instead of leaving delivered entries in the buffered map until `onMessage` resolves.
