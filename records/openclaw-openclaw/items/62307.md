## Summary

Keep open. Current main still has no WhatsApp `messages.update` listener, so the requested edit/revocation behavior is not implemented; the PR is useful source work, but its current head is dirty against main and still has a correctness bug for edits that arrive while `onMessage` is in flight.

## What This Changes

This PR changes the WhatsApp inbound monitor to track buffered and processed message IDs, listen for Baileys `messages.update`, replace pending edited text, notify after already-processed edits, and ignore revocations.

## Best Possible Solution

Implement the feature on current main inside the WhatsApp plugin’s inbound monitor with explicit state transitions for buffered, in-flight, and processed messages, add focused monitor tests for pending edits, in-flight edits, processed edits, revocations, self-chat, and access-control suppression, and document the user-facing behavior in the changelog.

## Reproduction Assessment

Yes for the PR defect: a unit test can enqueue a WhatsApp message, hold `onMessage` on a deferred promise, emit a `messages.update` edit before resolving it, and observe that the PR still treats the ID as pending instead of sending the late-edit path. Current main also has a direct code reproduction for the missing feature: emitting `messages.update` has no listener in the WhatsApp monitor.

## Solution Assessment

No. The PR is directionally close, but the current approach keeps IDs in the pending map during `onMessage`; a safer implementation should move entries out of buffered state before awaiting delivery and use an explicit in-flight state for late-edit notification or suppression.
