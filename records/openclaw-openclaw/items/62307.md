## Summary

Keep open. The PR addresses a real WhatsApp inbound feature gap, but it is conflicting, lacks real WhatsApp behavior proof, and still has a source-reproducible in-flight edit bug that can let the agent process stale pre-edit text without notifying the user.

## What This Changes

The branch adds a WhatsApp `messages.update` listener that mutates buffered inbound message bodies for edits, sends a late-edit notice after successful processing, and ignores revocation protocol messages.

## Best Possible Solution

Land a current-main WhatsApp plugin implementation that models buffered, in-flight, and processed message IDs separately, preserves current debounce/dedupe/close-drain behavior, adds focused timing tests, and includes redacted live WhatsApp proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: enqueue a debounced WhatsApp message, let `onFlush` choose and pass `messageToDeliver`, keep `onMessage` unresolved, then emit `messages.update`; the handler still sees the ID in `pendingMessageIds` even though mutating it no longer reliably changes what the agent processes.

## Solution Assessment

No, not as currently implemented. The feature belongs in the WhatsApp plugin, but the maintainable fix needs a current-main rebase plus distinct buffered, in-flight, and processed edit states rather than treating every pending ID as still safely mutable.
