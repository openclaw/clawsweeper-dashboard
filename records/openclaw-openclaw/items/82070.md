## Summary

Keep open: current main and v2026.6.10 only partially mitigate the report. Direct `openclaw health` is routed through a no-plugin fast path now, but doctor-triggered completion cache generation still uses the plugin-capable `completion --write-state` path, and the Codex first-turn stall plus Chrome/helper-process leak remain unproven on current main.

## What This Changes

The issue reports a post-2026.5.12 regression where CLI commands, completion cache generation, and Codex-backed gateway startup paths spend ~14s or more in cold CPU work and can accumulate helper processes.

## Best Possible Solution

Land or replace the focused doctor skip-env fix, then live-profile the Codex first-turn and helper-process symptoms and split any remaining app-server CPU work to the canonical related issue before closing this report.

## Reproduction Assessment

Yes for one remaining subpath by source inspection: doctor cache generation still spawns `completion --write-state` without the skip env while completion loads plugin commands unless that env is set. No live current-main reproduction was established for the full Linux Codex first-turn stall or Chrome helper-process accumulation.

## Solution Assessment

Keeping this open is the best current outcome. The narrow doctor fix should reuse the existing skip-env contract, while the broader Codex runtime symptoms need live profiling before maintainers can safely call the regression fixed.
