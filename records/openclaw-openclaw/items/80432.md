## Summary

Keep open. Current main still renders full Matrix MXIDs into `m.mentions` and `matrix.to` anchors, but bare sibling account aliases remain plain text and need a Matrix product/owner decision before changing notification semantics.

## What This Changes

The issue requests Matrix outbound auto-replies to resolve bare sibling account mentions into `matrix.to` pills and `m.mentions.user_ids` metadata.

## Best Possible Solution

Add a Matrix-plugin-owned outbound sibling mention roster with documented alias precedence, collision handling, and regression coverage across auto-reply, send, media caption, edit, and poll paths; otherwise document that only full MXIDs are supported.

## Reproduction Assessment

Yes, at source level. Current formatter and send tests show that full MXIDs get `m.mentions` and `matrix.to` anchors while bare names like `@alice` remain plain text with empty mention metadata.

## Solution Assessment

Unclear. Auto-deriving a sibling roster from configured Matrix accounts is plausible, but maintainers should first decide alias precedence, collision behavior, and whether non-OpenClaw aliases require explicit configuration.
