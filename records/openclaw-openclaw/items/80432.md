## Summary

Keep open. Current main still renders bare Matrix sibling aliases like `@alice` as plain text while only qualified Matrix IDs become `matrix.to` pills and `m.mentions`; the requested behavior is useful, but it needs a Matrix plugin decision on roster source, alias precedence, and collisions before implementation.

## What This Changes

Request Matrix auto-replies to render bare sibling account aliases as `matrix.to` mentions with `m.mentions.user_ids` metadata.

## Best Possible Solution

Define a Matrix-plugin-owned outbound alias/roster contract with documented source, precedence, and collision behavior, or explicitly document that only qualified Matrix IDs are supported.

## Reproduction Assessment

Yes, at source level. Current formatter/send tests show qualified MXIDs become mention metadata while bare `@name` remains plain text with empty `m.mentions`, and the Matrix outbound adapter routes auto-reply text through that same send path.

## Solution Assessment

Unclear. Auto-deriving aliases from configured sibling Matrix accounts is plausible, but maintainers should first choose implicit roster versus explicit alias config and define collision handling.
