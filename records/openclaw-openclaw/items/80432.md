## Summary

Keep open. Current main still renders qualified Matrix user IDs as `matrix.to` anchors with `m.mentions`, but bare sibling aliases such as `@alice` are intentionally left as plain text; supporting sibling aliases is useful but needs a Matrix roster and collision semantics decision before implementation.

## What This Changes

The issue requests Matrix auto-replies to render bare sibling account mentions as Matrix pills with `m.mentions` metadata.

## Best Possible Solution

Define a Matrix-plugin-owned sibling mention roster contract with alias precedence and collision handling, then add outbound formatter/send regression coverage or document that only full MXIDs are supported.

## Reproduction Assessment

Yes, at source level. Current main routes auto-replies through `sendMessageMatrix`, and formatter/send tests prove qualified MXIDs get mention metadata while bare `@name` tokens remain plain text with empty `m.mentions`.

## Solution Assessment

Unclear. Auto-deriving sibling aliases from configured Matrix accounts is plausible, but maintainers should first decide roster source, alias precedence, collision behavior, and whether explicit non-OpenClaw aliases are in scope.
