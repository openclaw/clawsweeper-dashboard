## Summary

Keep open. Current main still treats bare Matrix `@name` text as plain text, while only qualified Matrix user IDs become `matrix.to` pills and `m.mentions`; the requested sibling-account alias behavior is real and useful, but it needs a Matrix product decision on roster source, alias precedence, and collisions before implementation.

## What This Changes

Request Matrix auto-replies to render bare sibling account mentions like `@b` as Matrix pills with populated `m.mentions.user_ids`.

## Best Possible Solution

Add a Matrix-plugin-owned sibling mention roster with documented alias source, precedence, and collision behavior, or document that only fully qualified Matrix user IDs are supported.

## Reproduction Assessment

Yes, at source level. The auto-reply path reaches `sendMessageMatrix`, and current formatter/send tests show qualified MXIDs become mention metadata while bare `@name` tokens remain plain text with empty `m.mentions`.

## Solution Assessment

Unclear. Auto-deriving aliases from configured sibling Matrix accounts is plausible, but the maintainable solution needs a decision on roster source, alias precedence, collisions, and whether explicit non-OpenClaw aliases are in scope.
