## Summary

Keep open. Current main and the latest release still render outbound Matrix mention metadata only for qualified MXIDs and `@room`, while bare sibling aliases such as `@b` remain plain text; resolving those aliases needs a Matrix roster and collision policy decision rather than cleanup closure.

## What This Changes

The issue requests Matrix auto-replies to resolve sibling account aliases such as `@b` into Matrix mention pills and `m.mentions.user_ids` metadata.

## Best Possible Solution

Define a Matrix-plugin-owned outbound mention roster contract with documented source, precedence, and collision behavior, then add focused formatter/send/outbound tests; if maintainers reject aliases, document that only qualified MXIDs are supported.

## Reproduction Assessment

Yes, at source level. Current formatter and tests show qualified MXIDs become `m.mentions` plus `matrix.to` links while bare `@name` stays plain text, and Matrix auto-replies call the same `sendMessageMatrix` path.

## Solution Assessment

Unclear. Auto-deriving aliases from configured sibling Matrix accounts is plausible, but maintainers should first choose implicit roster versus explicit alias config and define precedence for collisions.
