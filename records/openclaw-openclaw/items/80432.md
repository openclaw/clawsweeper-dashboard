## Summary

Keep open. Current main and the latest tagged release still render qualified Matrix MXIDs as Matrix mentions but intentionally keep bare localpart text like `@alice` as plain text, so the sibling-account outbound mention problem remains and needs a Matrix roster/alias product decision rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to render sibling Matrix account aliases in agent auto-replies as Matrix mention pills with `m.mentions.user_ids`.

## Best Possible Solution

Define a Matrix-owned sibling mention roster contract and collision policy, then route outbound auto-reply text through that resolver with regression coverage for qualified MXIDs, bare sibling aliases, and non-sibling text.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current tests assert `sendMessageMatrix` leaves `@alice` with empty `m.mentions`, and auto-replies route through the same Matrix outbound adapter. I did not run a live Matrix QA lane in this read-only review.

## Solution Assessment

Unclear until maintainers choose the roster contract. Auto-deriving sibling accounts may be a good default, but alias precedence, collision handling, and spoofing boundaries should be decided before a fix PR.
