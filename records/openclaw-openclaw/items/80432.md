## Summary

Keep open. Current main still converts outbound Matrix mentions only when the outgoing text contains a qualified MXID or `@room`; bare sibling aliases like `@b` remain plain text, and the Matrix auto-reply adapter routes through that same formatter, so the reported sibling-account auto-reply gap is still present and needs a roster/alias product decision.

## What This Changes

Request Matrix auto-replies to resolve sibling account aliases like `@b` into `matrix.to` mention pills and `m.mentions.user_ids` metadata.

## Best Possible Solution

Define a Matrix-plugin-owned outbound sibling alias or roster contract with documented source, precedence, collision behavior, and regression coverage, or explicitly document that only qualified Matrix IDs are supported.

## Reproduction Assessment

Yes, at source level. Current formatter and send tests show qualified MXIDs produce `m.mentions` while bare aliases remain plain text, and Matrix auto-replies use that same send path.

## Solution Assessment

Unclear. Auto-deriving sibling aliases from configured Matrix accounts is plausible, but maintainers should first choose implicit roster versus explicit alias config and define precedence and collision handling.
