## Summary

Keep open. Current main still converts qualified Matrix user IDs into `matrix.to` anchors and `m.mentions`, but bare sibling account names such as `@alice` remain plain text; supporting those aliases is a useful Matrix feature that needs owner/product decisions on roster source, alias precedence, and collision handling.

## What This Changes

Request Matrix outbound auto-replies to resolve bare sibling account mentions into Matrix mention pills and `m.mentions.user_ids`.

## Best Possible Solution

Add a Matrix-plugin-owned outbound sibling mention roster with documented alias precedence, collision handling, and regression coverage, or explicitly document that only full MXIDs are supported.

## Reproduction Assessment

Yes, at source level. The current auto-reply send path reaches `sendMessageMatrix`, and current formatter/send tests prove qualified MXIDs get mention metadata while bare `@name` tokens remain plain text with empty `m.mentions`.

## Solution Assessment

Unclear. Auto-deriving sibling aliases from configured Matrix accounts is plausible, but maintainers should first decide roster source, alias precedence, collision behavior, and whether explicit non-OpenClaw aliases are in scope.
