## Summary

Keep open. Current main and the latest release still render full Matrix user IDs into `m.mentions` and `matrix.to` anchors, but bare sibling account aliases remain plain text; changing that needs Matrix owner/product judgment on roster source, alias precedence, and collision behavior.

## What This Changes

Request Matrix outbound auto-replies to resolve bare sibling account mentions like `@b` into `matrix.to` anchors and `m.mentions.user_ids`.

## Best Possible Solution

Add a Matrix-plugin-owned sibling mention roster/resolver with documented precedence and collision handling, plus regression coverage for auto-reply text, media captions, edits, and polls; otherwise document that only full MXIDs are supported.

## Reproduction Assessment

Yes, at source level. The shared Matrix send path reaches the formatter, and current source/tests show qualified MXIDs get metadata while bare sibling-style aliases stay plain text with empty `m.mentions`.

## Solution Assessment

Unclear. Auto-deriving a roster from configured Matrix accounts is plausible, but maintainers should first decide alias precedence, collision behavior, and whether non-OpenClaw aliases need explicit configuration.
