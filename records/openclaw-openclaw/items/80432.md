## Summary

Keep open. Current main still emits outbound Matrix mention metadata only for qualified MXIDs and `@room`, keeps bare aliases like `@alice` as plain text, and routes Matrix auto-replies through that same send formatter; adding sibling alias resolution needs a Matrix roster/alias decision first.

## What This Changes

Render bare Matrix sibling account aliases in agent auto-replies as `matrix.to` links with populated `m.mentions` so sibling agents and Matrix clients treat them as real mentions.

## Best Possible Solution

Define a Matrix-plugin-owned outbound alias/roster contract with documented source, precedence, and collision behavior, or explicitly document that only qualified Matrix IDs are supported.

## Reproduction Assessment

Yes, at source level. Current formatter and send tests show qualified MXIDs become mention metadata while bare `@name` remains plain text, and Matrix auto-replies use that same send path.

## Solution Assessment

Unclear. Auto-deriving aliases from configured sibling Matrix accounts is plausible, but maintainers should first choose implicit roster versus explicit alias config and define collision handling.
