## Summary

Keep open. Current main still renders bare Matrix @names as plain text while bot-to-bot mention gating depends on visible or metadata-backed mentions; resolving sibling account aliases is a real Matrix behavior gap, but it needs maintainer judgment on roster, alias, and collision semantics before a safe code change.

## What This Changes

Request Matrix agent auto-replies that mention sibling accounts by bare @name to emit matrix.to formatted_body anchors and m.mentions user metadata.

## Best Possible Solution

Add a Matrix-plugin-owned outbound sibling mention roster with documented alias precedence, collision handling, and regression coverage for auto-reply, send, and edit paths; otherwise document that only full MXIDs are supported.

## Reproduction Assessment

Yes, at source level. Current main's formatter and tests show qualified MXIDs get m.mentions and matrix.to anchors while bare @names stay plain text, matching the reported symptom.

## Solution Assessment

Unclear. Auto-deriving sibling aliases from configured Matrix accounts is plausible, but maintainers should decide alias precedence and collision behavior before changing notification semantics.
