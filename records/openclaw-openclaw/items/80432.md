## Summary

Keep open. Current main still renders only qualified Matrix user IDs into outbound `m.mentions`/`matrix.to` links, while the requested sibling-account alias behavior would change Matrix notification semantics and needs Matrix owner/product judgment.

## What This Changes

Add Matrix outbound mention rendering for bare sibling account aliases in agent auto-replies so `@name` can become a `matrix.to` pill with `m.mentions.user_ids`.

## Best Possible Solution

Define a Matrix-plugin-owned sibling mention roster/alias contract, then update outbound send/reply/edit formatting, docs, and regression tests; otherwise document that only full MXIDs are supported.

## Reproduction Assessment

Yes, at source level. Current formatter and tests show qualified MXIDs become `m.mentions` and `matrix.to` anchors, while bare `@alice` remains plain text with empty mention metadata.

## Solution Assessment

Unclear. Auto-deriving sibling aliases is plausible, but maintainers should decide alias precedence, collision behavior, and whether explicit configuration is required before changing notification semantics.
