## Summary

Keep open. Current main still renders qualified Matrix user IDs into outbound m.mentions and matrix.to anchors, but bare sibling account names remain plain text; resolving those aliases is useful but needs Matrix owner/product judgment on roster source, alias precedence, and collision behavior.

## What This Changes

The issue requests Matrix outbound mention rendering for bare sibling account names so auto-replies produce matrix.to pills and m.mentions metadata.

## Best Possible Solution

Add a Matrix-plugin-owned outbound sibling mention roster with documented alias precedence, collision handling, and regression coverage for auto-reply, send, media caption, edit, and poll paths; otherwise document that only full MXIDs are supported.

## Reproduction Assessment

Yes, at source level. The auto-reply/send path reaches sendMessageMatrix, and current formatter/tests show qualified MXIDs get m.mentions and matrix.to anchors while bare @names stay plain text with empty m.mentions.

## Solution Assessment

Unclear. Auto-deriving a sibling roster from configured Matrix accounts is plausible, but maintainers should first decide alias precedence, collision behavior, and whether non-OpenClaw aliases need explicit configuration.
