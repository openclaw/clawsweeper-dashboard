## Summary

Keep open: current main still renders bare Matrix sibling aliases such as `@alice` as plain text with empty `m.mentions`, while the existing outbound mention pipeline only converts qualified MXIDs. The report is useful, but it needs a Matrix product decision on roster source, alias precedence, and collision behavior before implementation.

## What This Changes

Request Matrix auto-replies to render sibling-account bare `@name` aliases as `matrix.to` pills with `m.mentions.user_ids` metadata.

## Best Possible Solution

Define a Matrix-plugin-owned sibling mention roster contract for outbound rendering, including alias source and collision handling, or explicitly document that only fully qualified MXIDs are supported for Matrix mention metadata.

## Reproduction Assessment

Yes, at source level. Current formatter and send tests show qualified MXIDs become mention metadata while bare `@name` aliases remain plain text, and the Matrix auto-reply adapter reaches that same send path.

## Solution Assessment

Unclear. Auto-deriving aliases from configured sibling Matrix accounts is plausible, but maintainers need to choose roster source, precedence, collisions, and whether non-OpenClaw aliases belong in scope.
