## Summary

Keep open. Current main and the latest release still only render qualified Matrix IDs, while bare sibling aliases such as `@alice` stay plain text with empty `m.mentions`; the requested sibling roster behavior is useful but needs a Matrix plugin product decision on alias source, precedence, and collisions.

## What This Changes

The issue asks Matrix outbound auto-replies to render sibling-account bare aliases like `@b` into `matrix.to` mention pills and `m.mentions.user_ids`.

## Best Possible Solution

Define a Matrix-plugin-owned outbound alias/roster contract with documented source, precedence, collision behavior, and tests, or explicitly document that only qualified Matrix IDs are supported.

## Reproduction Assessment

Yes, at source level. Current formatter and send tests show qualified MXIDs produce mention metadata while bare `@name` stays plain text, and the Matrix outbound adapter routes auto-reply text through that send path.

## Solution Assessment

Unclear. Auto-deriving aliases from configured sibling Matrix accounts is plausible, but maintainers should first choose implicit roster versus explicit alias config and define precedence/collision semantics.
