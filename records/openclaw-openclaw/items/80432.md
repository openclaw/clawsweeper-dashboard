## Summary

Keep open. Current main and the latest tagged release still render only qualified Matrix user IDs as `matrix.to` links and `m.mentions`, while bare sibling aliases remain plain text; the remaining work needs a Matrix roster/alias policy decision rather than conservative cleanup closure.

## What This Changes

Request Matrix agent auto-replies to render sibling account aliases such as `@alice` as Matrix mention pills with `m.mentions.user_ids`.

## Best Possible Solution

Define a Matrix-plugin-owned outbound alias/roster contract with documented source, precedence, and collision behavior, or explicitly document that only qualified Matrix IDs are supported.

## Reproduction Assessment

Yes, at source level. Current formatter/send tests show qualified MXIDs become mention metadata while bare aliases stay plain, and the Matrix outbound adapter routes auto-reply text through that same send path.

## Solution Assessment

Unclear. Auto-deriving aliases from configured sibling Matrix accounts is plausible, but maintainers should first choose implicit roster versus explicit alias config and define collision handling.
