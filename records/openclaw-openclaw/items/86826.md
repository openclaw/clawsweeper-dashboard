## Summary

Keep open: current main does not implement raw external session alias preservation, but this PR is incomplete because it only syncs aliases during migration while later session mutation paths still write or delete a single key; it also lacks real behavior proof.

## What This Changes

The PR adds a gateway alias helper and calls it from session-store key migration to keep selected raw external keys alongside the canonical session key.

## Best Possible Solution

Use one shared alias-group contract that chooses the freshest raw/canonical entry and syncs or deletes all aliases across migration, patch/delete/reset/compact/list/search, and plugin host-hook writers.

## Reproduction Assessment

Yes, source inspection and the linked issue establish a repro shape: create a raw external key entry, let gateway canonicalization promote/prune it, then use the raw key again or mutate via the canonical key. I did not execute a live external-channel replay in this read-only review.

## Solution Assessment

No: preserving aliases at migration time is only a partial fix unless all subsequent read/write/delete/list/plugin paths treat the alias group as one logical session.
