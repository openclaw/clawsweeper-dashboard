## Summary

Keep open. The reported Slack auth crash/restart problem is plausible, but this PR is not merge-safe as written: resolving `monitorSlackProvider` maps the auth failure into the gateway supervisor's clean-exit path, which records the wrong status and schedules auto-restart instead of creating a terminal Slack failure.

## What This Changes

This PR changes Slack socket startup auth and retry-exhaustion failures from thrown errors to clean returns, and adds a direct `monitorSlackProvider` unit test for an `account_inactive` start failure.

## Best Possible Solution

Preserve non-recoverable Slack auth failures as terminal channel failures that keep the gateway and other channels alive, record a clear Slack `lastError`, and suppress supervisor restart for that account through a documented lifecycle path with gateway-level regression coverage.

## Reproduction Assessment

Yes. The source path is high-confidence: make Slack `app.start()` reject with `account_inactive`, let this PR return from `monitorSlackProvider`, and the gateway manager's resolved-task branch records a clean exit and schedules restart.

## Solution Assessment

No. The return-only approach is not the best fix because it loses the failure signal and enters the clean-exit restart path; the safer fix is terminal/no-restart channel failure handling plus a gateway-manager test.
