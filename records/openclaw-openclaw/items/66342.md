## Label Justifications

- `P1`: The PR changes Slack/gateway lifecycle handling for a real channel availability bug and currently has a blocking restart-suppression regression.
- `merge-risk: 🚨 message-delivery`: Suppressing gateway restart after recoverable Slack reconnect exhaustion can leave Slack disconnected and drop later Slack messages until manual intervention.
- `merge-risk: 🚨 availability`: The gateway lifecycle change can turn a transient Slack outage into a permanently stopped channel instead of using the current restart fallback.

## Summary

Keep open. The Slack/gateway failure is source-reproducible and a typed terminal signal is a plausible direction, but this branch over-suppresses restarts for recoverable Slack reconnect exhaustion, still has current type/lint/boundary blockers, and lacks real behavior proof.

## What This Changes

The PR adds a TerminalChannelError signal exported through runtime-env, converts Slack auth and reconnect-exhaustion failures to that signal, suppresses gateway auto-restarts for terminal channel errors, and adds Slack/gateway regression tests.

## Best Possible Solution

Keep a terminal failure signal only for permanent Slack auth failures, leave recoverable reconnect exhaustion restartable or coordinate it with the separate reconnect-cap work, move supervisor tests to an allowed core/public surface, and align any public SDK export before merge.

## Reproduction Assessment

Yes, at source level. Current main still throws on non-recoverable Slack auth errors and the gateway restart path is visible in server-channels; the PR's over-broad terminal classification is also visible in the head diff.

## Solution Assessment

No, not as submitted. A typed terminal channel signal is maintainable for permanent auth failures, but applying it to recoverable max-attempt exhaustion and exporting it without SDK contract alignment is not the safest fix.
