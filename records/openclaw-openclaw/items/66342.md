## Summary

Keep open. The PR addresses a real Slack crash-loop report, but the branch changes terminal Slack startup failures into normal resolution while the current gateway supervisor treats normal channel exits as restartable clean exits, so the patch can still churn and lose the auth-failure diagnostic.

## What This Changes

This PR changes Slack socket startup auth and retry-exhaustion failures from thrown errors to clean returns and adds a direct monitor unit test for an `account_inactive` start failure.

## Best Possible Solution

Record non-recoverable Slack auth failures as terminal channel failures with a clear `lastError`, keep the gateway and other channels alive, and suppress automatic Slack restart through a documented lifecycle path with Slack and gateway supervisor regression coverage.

## Reproduction Assessment

Yes. Make Slack `app.start()` reject with `account_inactive` under this PR: `monitorSlackProvider` resolves, and `src/gateway/server-channels.ts` then records a clean exit and schedules auto-restart, a path already asserted in `src/gateway/server-channels.test.ts`.

## Solution Assessment

No. The return-only approach avoids a thrown rejection but conflicts with the gateway supervisor contract; the safer fix is to preserve the failure signal while adding terminal/no-restart handling for this account.
