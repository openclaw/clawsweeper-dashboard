## Summary

Close as implemented on current main. The Slack startup auth rejection leak and gateway-wide crash class are already fixed in shipped code, and the proposed clean-return path would now be worse than the shipped behavior because the gateway treats clean channel exits as restartable and less diagnostic.

## What This Changes

The PR changes Slack socket-mode startup auth and max-retry failures to return from `monitorSlackProvider` and adds a regression test for an `app.start()` `account_inactive` rejection.

## Best Possible Solution

Keep the shipped Slack Bolt `tokenVerificationEnabled: false` fix plus gateway channel supervision. If maintainers still want permanent Slack auth failures to pause without retry, that should be an explicit terminal failure/no-restart state rather than a clean monitor return.

## Reproduction Assessment

Yes. The source-level path is clear: invalid Slack auth can reject during Slack startup, and current main consumes channel task failures through `server-channels.ts` while tests cover later channels continuing after one startup failure.

## Solution Assessment

Yes for the shipped implementation, not for this PR diff. Disabling Bolt's eager token verification and keeping gateway-level failure supervision is narrower and preserves diagnostics; returning cleanly from the Slack monitor loses the terminal error signal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve terminal Slack failure status:** `extensions/slack/src/monitor/provider.ts:502-509`
  - body: These returns make `monitorSlackProvider` resolve for permanent auth failures and exhausted startup retries. The gateway treats a non-manual resolved channel task as a clean exit, records `channel exited without an error`, and schedules auto-restart, so invalid Slack tokens lose the concrete auth error and still restart. Keep a handled failure signal or explicit terminal no-restart state before exiting.
  - confidence: 0.88
