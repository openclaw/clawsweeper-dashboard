## Summary

Keep open. Current main has useful mitigations for stale session bookkeeping, transcript-size compaction, Feishu probe timeouts, trajectory file caps, and service PATH false positives, but the central outage contract is still incomplete because active embedded runs, active reply work, and active lane tasks are classified or handled as non-recoverable by diagnostics.

## What This Changes

The issue requests automatic recovery and operational guardrails for Feishu/gateway stuck sessions, startup compaction, trajectory growth, memory limits, ping timeouts, and service PATH repair.

## Best Possible Solution

Keep this as the outage umbrella until #73243 or an equivalent configurable, race-safe active stuck-session abort/reset path lands, then split any still-current Feishu, startup-compaction, trajectory, memory-limit, or PATH repair gaps into narrower follow-ups.

## Reproduction Assessment

Yes for the remaining central code-path gap: current source and tests show active processing sessions are detected but active embedded runs, active reply work, and active lane tasks are not aborted/reset by the heartbeat path. The exact Feishu production incident was not live-reproduced.

## Solution Assessment

No, current main is only a partial mitigation. The maintainable path is to review and land #73243 or an equivalent diagnostics-owned recovery policy, then separately validate storage growth and Feishu-specific requests.
