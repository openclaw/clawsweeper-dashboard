## Summary

Keep open. Current main still has useful mitigations for stale lane cleanup, Feishu probe timeouts, and trajectory growth, but the central outage request is not implemented because diagnostics still defaults to warning/conservative cleanup and does not enable a configurable active stuck-session abort/reset policy.

## What This Changes

The issue requests gateway self-recovery for stuck sessions plus startup compaction, trajectory limits, memory-limit guidance, Feishu probe isolation, and service PATH repair around a Feishu WebSocket outage on v2026.4.26.

## Best Possible Solution

Land a diagnostics-owned, configurable, race-safe active stuck-session abort/reset path first. After that, keep or create narrower follow-ups only for any still-current startup compaction, trajectory, Feishu probe, service PATH, or memory-limit behavior that remains reproducible on main.

## Reproduction Assessment

Yes for the central remaining code-path gap: current source and tests show stuck processing sessions are detected, but active embedded runs, active reply work, and active lane tasks are not aborted or reset by default. The complete Feishu outage was not live-reproduced here.

## Solution Assessment

No, current main is only a partial solution. The maintainable fix is #73243 or an equivalent diagnostics-owned abort threshold and recovery policy, with separate follow-up only for remaining storage, channel, or service hardening gaps.
