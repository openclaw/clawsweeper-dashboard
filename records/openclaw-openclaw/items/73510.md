## Summary

Keep open. Current main has several mitigations, but the central outage contract is still not implemented: diagnostics can observe and release stale bookkeeping, while active embedded runs, active reply work, and active lane tasks remain protected by default and there is no public abort threshold.

## What This Changes

The issue requests automatic recovery for stuck Feishu gateway sessions plus startup compaction, trajectory size control, memory-limit guidance, and Feishu heartbeat hardening.

## Best Possible Solution

Keep this as the outage umbrella until #73243 or an equivalent configurable, race-safe active stuck-session abort/reset path lands; then close it as implemented or split any remaining current-main gaps into narrower follow-ups.

## Reproduction Assessment

Yes for the remaining central code-path gap: current source and tests show stuck processing sessions are detected, but active embedded runs, active reply work, and active lane tasks are not aborted/reset by default. The complete Feishu incident was not live-reproduced.

## Solution Assessment

No, current main is only a partial solution. The maintainable path is to review and land #73243 or equivalent diagnostics-owned abort/reset policy, then separately track any still-reproducible storage, Feishu, service PATH, or memory-limit gaps.
