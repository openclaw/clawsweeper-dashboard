## Summary

Keep open. Current main has several mitigations for this outage family, but it still does not implement the central requested active stuck-session recovery policy; diagnostics warn and can release stale lanes only when no active work is present, while active embedded runs, active reply work, and active lane tasks remain protected by default. Open PR #73243 is the relevant implementation candidate, not a reason to close this issue before maintainer review and merge.

## What This Changes

The issue requests automatic recovery for stuck gateway sessions plus related Feishu probe, startup compaction, trajectory growth, memory-limit guidance, and service PATH self-healing improvements.

## Best Possible Solution

Keep this as the outage umbrella until maintainers land a configurable, race-safe active stuck-session abort/reset path, likely through review of #73243 or a narrower replacement. After that, split or close the already-mitigated Feishu probe, trajectory, service PATH, and memory guidance pieces according to what current main still lacks.

## Reproduction Assessment

Yes for the central code-path gap: current source and tests show diagnostics detect stuck processing sessions but do not abort active embedded runs or reset active reply/lane work by default. The full Feishu incident remains log-backed rather than live-reproduced here.

## Solution Assessment

No, current main is not yet the best complete solution because it only performs conservative stale-lane recovery. The narrow maintainable direction is a diagnostics-owned, configurable active abort/reset policy with tests for active run cancellation, lane cleanup, and safe no-op cases.
