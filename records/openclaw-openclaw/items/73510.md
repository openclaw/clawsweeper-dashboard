## Summary

Keep open. Current main has useful partial mitigations, but the central outage contract is still incomplete: active stuck sessions are classified and logged, while active embedded runs, active reply work, and active lane tasks are not aborted or reset by the diagnostics heartbeat.

## What This Changes

The issue requests auto-recovery for stuck gateway sessions plus related startup compaction, trajectory size, Feishu heartbeat, memory-limit, and PATH repair hardening after a v2026.4.26 Feishu outage.

## Best Possible Solution

Keep this as the outage umbrella until #73243 or an equivalent configurable, race-safe active stuck-session abort/reset path lands, then split any still-current Feishu, startup-compaction, trajectory, memory-limit, or PATH repair gaps into narrower follow-ups.

## Reproduction Assessment

Yes for the remaining central code-path gap: current source and tests show active processing sessions are detected but active embedded runs, active reply work, and active lane tasks are not aborted or reset by the heartbeat path. The exact Feishu production incident was not live-reproduced.

## Solution Assessment

No, current main is only a partial mitigation. The maintainable path is to review and land #73243 or an equivalent diagnostics-owned recovery policy, then separately validate storage growth and Feishu-specific requests.
