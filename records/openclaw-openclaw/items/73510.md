## Summary

Keep open. Current main has several mitigations around Feishu probing, trajectory size, transcript byte-triggered pre-run compaction, service PATH auditing, and Linux memory guidance, but the central outage behavior remains: stuck active sessions are detected and warned on while active embedded runs, active reply work, and active lane tasks are protected from default abort/reset.

## What This Changes

The issue requests automatic recovery for stuck gateway sessions plus startup compaction, trajectory file limits, gateway memory-limit guidance, Feishu heartbeat isolation, and service PATH repair improvements.

## Best Possible Solution

Keep this as the outage umbrella until maintainers land a configurable, race-safe stuck-session abort/reset policy for active work. After that, close it as implemented or split any remaining startup compaction, Feishu, trajectory, PATH, and memory-limit concerns into narrower current-main reports.

## Reproduction Assessment

Yes for the remaining central code-path gap: current source and tests show stuck processing sessions are detected, but active embedded runs, active reply work, and active lane tasks are not aborted or reset by default. The exact Feishu outage remains log-backed rather than live-reproduced in this read-only review.

## Solution Assessment

No, current main is not yet the complete solution because the default recovery path remains conservative and lacks a public abort threshold. The maintainable path is to review #73243 or a narrower equivalent diagnostics-owned policy before closing this umbrella.
