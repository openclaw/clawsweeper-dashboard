## Summary

Keep open. Current main has partial mitigations around transcript byte compaction, trajectory file caps, Feishu probe timeouts, and service PATH diagnostics, but the central outage path is still incomplete because diagnostics still does not abort or reset active embedded runs, active reply work, or active lane tasks.

## What This Changes

The issue requests automatic recovery for stuck gateway sessions plus startup compaction, trajectory caps, memory-limit guidance, Feishu heartbeat isolation, and service PATH repair improvements.

## Best Possible Solution

Keep this as the outage umbrella until #73243 or an equivalent configurable, race-safe active stuck-session recovery policy lands, then split any remaining storage, Feishu, memory-limit, and PATH concerns into narrower follow-ups.

## Reproduction Assessment

Yes for the remaining central code-path gap: current source and tests show active processing sessions are detected but active embedded runs, active reply work, and active lane tasks are not aborted or reset by diagnostics. The exact Feishu incident was not live-reproduced.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix is to review and land #73243 or an equivalent diagnostics-owned active recovery policy, then handle the remaining umbrella asks separately.
