## Summary

Keep open. Current main has partial mitigations, but the central outage contract is still missing: the diagnostic heartbeat can warn and release stale bookkeeping, while active embedded runs, active reply work, and active lane tasks are still not aborted or reset by default.

## What This Changes

The issue requests automatic recovery for stuck gateway sessions plus startup compaction, trajectory-size, memory-limit, and Feishu heartbeat hardening after a v2026.4.26 Feishu outage.

## Best Possible Solution

Keep this as the outage umbrella until #73243 or an equivalent configurable, race-safe active stuck-session abort/reset path lands, then split any still-current Feishu, startup-compaction, trajectory, or service-limit gaps into narrower follow-ups.

## Reproduction Assessment

Yes for the remaining central code-path gap: current source and tests show active processing sessions are detected, but active embedded runs, active reply work, and active lane tasks are not aborted/reset by the heartbeat path. The exact Feishu production incident was not live-reproduced.

## Solution Assessment

No, current main is only a partial solution. The maintainable path is to review and land #73243 or equivalent diagnostics-owned recovery policy, then separately validate the storage and Feishu-specific requests.
