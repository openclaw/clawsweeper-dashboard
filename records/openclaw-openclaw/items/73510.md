## Summary

Keep open. Current main contains several mitigations around stale lanes, Feishu probes, trajectory cleanup, Linux PATH handling, and compaction triggers, but the central requested behavior is still incomplete: diagnostics warn and invoke conservative recovery, while active embedded runs, active reply work, and active lane tasks remain warning-only by default, with no public abort threshold config on main. The related open PR #73243 is still the active implementation candidate for the core stuck-session abort policy.

## What This Changes

The issue asks for automatic recovery of stuck gateway sessions plus startup compaction, trajectory-size controls, memory guidance, and Feishu probe hardening after a v2026.4.26 Feishu WebSocket outage.

## Best Possible Solution

Keep this as the production-outage umbrella until maintainers land or revise a configurable, race-safe active stuck-session abort/reset policy, then split any remaining startup compaction, trajectory retention, Feishu probe, memory guidance, and PATH cleanup gaps into narrower follow-ups where current main does not already cover them.

## Reproduction Assessment

Yes for the central code-path gap: current source and tests show diagnostics can detect stuck processing sessions, but active embedded runs, active reply work, and active lane tasks are not aborted or reset by the default heartbeat path. The full Feishu outage sequence remains a production-log reproduction rather than a live deterministic test here.

## Solution Assessment

No, current main is not the complete solution because it only releases stale lanes when no active work is present and leaves active stuck work warning-only. The safest path is maintainer review of #73243 or equivalent diagnostics-owned active abort/reset behavior, with the other reported surfaces split after that policy is settled.
