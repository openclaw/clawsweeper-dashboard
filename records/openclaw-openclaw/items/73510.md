## Summary

Keep open. Current main contains several mitigations, but the central outage request is not fully implemented: diagnostics still warn and run conservative cleanup, while active embedded runs, active reply work, and active lane tasks are protected from default abort/reset.

## What This Changes

The issue asks OpenClaw to automatically recover stuck gateway sessions and harden Feishu probing, compaction, trajectory, service PATH, and memory-limit behavior after a v2026.4.26 production outage.

## Best Possible Solution

Keep this as the outage umbrella until maintainers land a configurable, race-safe active stuck-session abort/reset path. After that, close this as implemented or split any remaining Feishu, startup compaction, trajectory, PATH, and memory-limit work into narrower issues with current-main evidence.

## Reproduction Assessment

Yes for the remaining central code-path gap: current source and tests show stuck processing sessions are detected, but active embedded runs, active reply work, and active lane tasks are not aborted or reset by default. The complete Feishu incident was not live-reproduced here.

## Solution Assessment

No, current main is only a partial solution. The maintainable path is to review and land #73243 or an equivalent diagnostics-owned abort/reset policy, then separately track any still-reproducible storage, Feishu, or service hardening gaps.
