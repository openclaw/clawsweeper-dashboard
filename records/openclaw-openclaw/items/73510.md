## Summary

Keep open. Current main has useful mitigations around stale-lane diagnostics, Feishu probe timeouts, trajectory sidecar limits/cleanup, preflight transcript compaction, service PATH handling, and Linux memory guidance, but it still does not implement the central requested active stuck-session recovery policy: the default diagnostics heartbeat warns and invokes conservative recovery, while active embedded runs, active reply work, and active lane tasks remain protected from abort/reset.

## What This Changes

The issue requests automatic recovery for stuck gateway sessions plus related startup compaction, trajectory file limits, gateway memory guidance, Feishu heartbeat/probe isolation, and service PATH repair.

## Best Possible Solution

Keep this as the outage umbrella until maintainers settle and land a configurable, race-safe active stuck-session abort/reset path. After that, close or split the remaining startup compaction, Feishu probe, trajectory, PATH, and memory-limit concerns according to what current main still lacks.

## Reproduction Assessment

Yes for the central code-path gap: current source and tests show diagnostics detect stuck processing sessions but do not abort active embedded runs or reset active reply/lane work by default. The full Feishu incident remains log-backed rather than live-reproduced here.

## Solution Assessment

No, current main is not yet the complete solution because it warns and does conservative stale-lane cleanup but does not satisfy the requested kill/reset/restart behavior for confirmed active stuck sessions. The narrow maintainable direction is a diagnostics-owned, configurable active abort/reset policy, with the broader Feishu and compaction concerns split after that lands.
