## Summary

Keep open. Current main has partial mitigations for Feishu probe timeouts, trajectory sidecar growth, transcript byte-triggered preflight compaction, service PATH handling, and Linux memory guidance, but it still does not implement the central requested policy: diagnostics can warn and run conservative cleanup, while active embedded runs, active reply work, and active lane tasks remain protected from default abort/reset.

## What This Changes

Request active recovery for stuck gateway sessions plus related safeguards for startup compaction, trajectory growth, memory limits, Feishu probe timeout behavior, and service PATH repair.

## Best Possible Solution

Keep this as the outage umbrella until maintainers land a configurable, race-safe active stuck-session abort/reset path. After that, close this as implemented or split any remaining startup compaction, Feishu, trajectory, PATH, and memory-limit follow-ups into narrower issues with current-main evidence.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the remaining central code-path gap: current source and tests show stuck processing sessions are detected, but active embedded runs, active reply work, and active lane tasks are not aborted or reset by default. The full Feishu incident remains log-backed rather than live-reproduced here.

## Solution Assessment

Is this the best way to solve the issue? No, current main is not yet the complete solution because the default recovery path is conservative and lacks a public abort threshold. The maintainable fix is a diagnostics-owned, configurable active abort/reset policy, likely by reviewing #73243 or a narrower superseding PR.
