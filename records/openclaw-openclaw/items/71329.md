## Summary

Keep open. Current main still has the Slack status-reaction activation gap, and the PR’s core approach is narrow and aligned with the existing controller/docs, but the PR head is currently unmergeable and the changelog/test follow-up is only present in a fork PR, not on this PR head.

## What This Changes

The PR updates Slack message preparation so `messages.statusReactions.enabled === true` can create the lifecycle reaction handle independently of ack-reaction gating, with colocated prepare tests for explicit enable and disable cases.

## Best Possible Solution

Land a rebased current-main version of the Slack prepare change that preserves the message-tool-only guard, includes the Slack changelog entry, and asserts the explicit-disable fallback ack promise; the cleanest path is to merge the existing fork follow-up into the source branch, rebase/adapt, then rerun the targeted Slack tests and changed gate.

## Reproduction Assessment

Yes. On current main, a timestamped Slack DM with `messages.statusReactions.enabled: true` and the default `group-mentions` ack scope leaves `shouldSendAckReaction` false, so `statusReactionsWillHandle` is false and `ackReactionPromise` remains null; dispatch then keeps lifecycle reactions disabled.

## Solution Assessment

Yes, with current-main adaptation. Decoupling explicit status-reaction enablement inside Slack prepare is the narrowest maintainable fix, provided the rebase preserves `sourceRepliesAreToolOnly` and the explicit-disable ack fallback remains covered.
