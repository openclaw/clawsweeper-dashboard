## Summary

Keep open: current main still gates Slack lifecycle status reactions through the ack-reaction path, while the PR is a narrow fix for the documented explicit opt-in. It is not merge-ready because the external PR still lacks real Slack behavior proof, is conflicting, and has a small test assertion gap already addressed only in a fork follow-up.

## What This Changes

The PR changes Slack message preparation so `messages.statusReactions.enabled: true` can activate lifecycle status reactions without a configured `messages.ackReaction`, and adds two prepare tests.

## Best Possible Solution

Rebase the Slack prepare change onto current main, let explicit status-reaction opt-in prime the controller without requiring a plain ack emoji, preserve the tool-only mention/bypass guard, fold in the fallback ack assertion, and attach redacted Slack runtime proof.

## Reproduction Assessment

Yes, from source inspection. With `messages.statusReactions.enabled: true` but no ack reaction, current `prepareSlackMessage` leaves `ackReactionPromise` null, and dispatch requires that promise before enabling the Slack lifecycle status-reaction controller.

## Solution Assessment

Yes in direction, but not merge-ready as-is. Decoupling explicit `messages.statusReactions.enabled` from the plain ack gate is the narrow maintainable fix, provided the rebase preserves current-main tool-only privacy handling and strengthens the explicit-disable test.
