## Summary

Keep open. Current main still gates Slack lifecycle status reactions on the ack-reaction path, so the documented explicit `messages.statusReactions.enabled` opt-in remains unimplemented; this PR is the right narrow direction, but it needs current-main rebase/adaptation plus a required changelog entry and one tighter regression assertion before merge.

## What This Changes

The PR changes Slack message preparation so `messages.statusReactions.enabled === true` can create the lifecycle status-reaction handle without requiring the ack-reaction gate, and adds prepare tests for explicit enable and explicit disable cases.

## Best Possible Solution

Adapt this narrow Slack change on current main so an explicit `messages.statusReactions.enabled === true` activates lifecycle reactions independently of ack scope, while `enabled: false`, message-tool-only source replies, and normal ack fallback behavior remain covered by focused tests and changelog documentation.

## Reproduction Assessment

Yes. On current main, a timestamped Slack message with `messages.statusReactions.enabled: true` still gets no lifecycle handle when the ack gate is false, because `statusReactionsWillHandle` is tied to `shouldSendAckReaction` and dispatch requires `prepared.ackReactionPromise`.

## Solution Assessment

Yes, with rebase adjustments. Decoupling explicit status-reaction enablement in Slack prepare is the narrowest maintainable fix, but the final patch should be applied to current main's `shouldSendAckReaction` logic and include the missing changelog/test assertion.
