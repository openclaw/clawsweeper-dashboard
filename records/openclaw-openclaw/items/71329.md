## Summary

Keep open. Current main still gates Slack lifecycle status reactions through the ack-reaction path, while the PR targets a real Slack UX gap; the branch needs current-main adaptation plus the small changelog/test follow-ups before merge.

## What This Changes

The PR changes Slack message preparation so `messages.statusReactions.enabled === true` can create the lifecycle reaction handle independently of ack-reaction gating, with prepare tests for explicit enable and explicit disable cases.

## Best Possible Solution

Land a rebased current-main version of the Slack prepare change that decouples explicit status-reaction enablement from ack-reaction gating, preserves message-tool-only source reply behavior, includes the Slack changelog entry, and strengthens the explicit-disable fallback ack assertion.

## Reproduction Assessment

Yes. On current main, a timestamped Slack message with `messages.statusReactions.enabled: true` but no effective ack reaction leaves `statusReactionsWillHandle` false in prepare, so `ackReactionPromise` remains null and dispatch keeps lifecycle reactions disabled.

## Solution Assessment

Yes, with current-main adaptation. Decoupling explicit enablement in Slack prepare is the narrowest maintainable fix because the lifecycle controller and dispatch wiring already exist; the safer adaptation is to keep the current `sourceRepliesAreToolOnly` guard while adding the explicit opt-in branch.
