The PR changes Slack message preparation and colocated tests so `messages.statusReactions.enabled === true` can activate lifecycle status reactions without requiring the ack-reaction gate to pass.

## Best Possible Solution

## Solution Assessment

Yes, with the already-requested adjustments. Decoupling explicit Slack status-reaction enablement in `prepareSlackMessage` is the narrowest maintainable fix, but the branch should include the changelog/test follow-up and preserve the current-main message-tool-only guard before merge.
