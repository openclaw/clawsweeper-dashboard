## Summary

Keep open. Current main still ties Slack lifecycle status reactions to the ack-reaction gate, so this PR is not obsolete; the proposed direction is narrow and compatible, but it still needs the required changelog entry, a tighter explicit-disable assertion, and a rebase/repair that preserves current main's message-tool-only guard.

## What This Changes

The PR changes Slack message preparation and colocated tests so `messages.statusReactions.enabled === true` can activate lifecycle status reactions without requiring the ack-reaction gate to pass.

## Best Possible Solution

Adapt the narrow Slack prepare change on current main so explicit `messages.statusReactions.enabled === true` creates the lifecycle reaction handle independently of ack scope, while `enabled: false`, message-tool-only source replies, and plain ack fallback behavior stay covered by focused tests and an Unreleased changelog entry.

## Reproduction Assessment

Yes. On current main, a timestamped Slack DM with `messages.statusReactions.enabled: true` and the default group-mentions ack scope leaves `shouldSendAckReaction` false, so `statusReactionsWillHandle` is false, `ackReactionPromise` remains null, and dispatch keeps lifecycle reactions disabled.

## Solution Assessment

Yes, with rebase adjustments. Decoupling explicit status-reaction enablement inside Slack prepare is the narrowest maintainable fix, as long as the current-main message-tool-only guard and the explicit-disable fallback ack behavior are preserved.
