## Summary

Keep open: current main still blocks explicit Slack status reactions when the ack gate is false, and this PR remains a useful narrow fix, but it is not merge-ready because the external PR lacks real Slack proof and needs a current-main update.

## What This Changes

The PR changes Slack message preparation and colocated tests so `messages.statusReactions.enabled: true` can prime lifecycle reactions without requiring plain ack-reaction eligibility.

## Best Possible Solution

Land a rebased Slack prepare change that treats explicit `statusReactions.enabled` as an independent opt-in, preserves the tool-only mention/bypass guard, covers enable/disable fallback behavior, and includes redacted real Slack proof.

## Reproduction Assessment

Yes, from source inspection. A Slack DM with `messages.statusReactions.enabled: true` still leaves `prepared.ackReactionPromise` null on current main when the default `group-mentions` ack gate is false, and dispatch then cannot start the status controller.

## Solution Assessment

Yes, with current-main adaptation. Decoupling explicit Slack status-reaction enablement from the plain ack gate is the narrow fix, but the branch must preserve the newer tool-only guard and include the explicit-disable fallback assertion.
