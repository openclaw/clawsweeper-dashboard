## Summary

Keep open. Current main still does not let Slack `messages.statusReactions.enabled === true` activate lifecycle reactions independently of the ack-reaction gate, so the PR is still useful; however the current PR head still needs the required changelog entry and the stronger explicit-disable test assertion. The later follow-up PR in the contributor fork reportedly addresses both points, but those changes have not landed on this PR head yet.

## What This Changes

The PR changes Slack message preparation and colocated tests so `messages.statusReactions.enabled === true` can create the lifecycle status-reaction handle without requiring the ack-reaction gate to pass.

## Best Possible Solution

Land a current-main-adapted Slack prepare change that treats explicit `messages.statusReactions.enabled === true` as lifecycle opt-in while preserving the message-tool-only guard, explicit-disable fallback ack behavior, timestamp guard, focused prepare tests, and an Unreleased changelog entry with valid contributor credit.

## Reproduction Assessment

Yes. A source-level reproduction is a timestamped Slack DM with `messages.statusReactions.enabled: true` and the default `ackReactionScope: group-mentions`; current main leaves `shouldSendAckReaction` false, so `statusReactionsWillHandle` is false and dispatch never enables lifecycle reactions.

## Solution Assessment

Yes, with integration of the follow-up adjustments. Decoupling explicit status-reaction enablement in Slack prepare is the narrowest maintainable fix, provided the current-main tool-only reply guard and the explicit-disable ack fallback test remain covered.
