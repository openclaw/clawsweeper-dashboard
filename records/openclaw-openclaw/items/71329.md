## Summary

Keep open. The PR still addresses a real Slack status-reaction gap on current main, but the external PR lacks after-fix Slack proof and its head is conflicting/stale relative to newer tool-only privacy handling.

## What This Changes

The PR changes Slack message preparation so `messages.statusReactions.enabled: true` can activate lifecycle reactions without a configured plain ack reaction, and adds focused prepare tests.

## Best Possible Solution

Rebase the Slack prepare change onto current main, decouple explicit status-reaction opt-in from the plain ack gate, preserve the tool-only mentioned/bypass guard, add the fallback ack assertion, and attach redacted real Slack proof.

## Reproduction Assessment

Yes, from source inspection. Current main still routes Slack lifecycle status reactions through `shouldSendAckReaction`, while dispatch requires `prepared.ackReactionPromise` before the status controller can run.

## Solution Assessment

Yes in direction, but not merge-ready as-is. Decoupling explicit `messages.statusReactions.enabled` from the plain ack gate is the narrow fix, provided the rebase preserves newer tool-only privacy handling and strengthens the explicit-disable test.
