## Summary

Keep open. Current main still requires an ack-reaction-backed prepare handle before Slack lifecycle status reactions can run, so this PR still addresses a real documented gap, but the head needs current-main adaptation and real behavior proof before merge.

## What This Changes

The PR changes Slack message preparation so `messages.statusReactions.enabled: true` can activate lifecycle status reactions without requiring a configured `ackReaction`.

## Best Possible Solution

Land a rebased current-main Slack prepare change that lets explicit status-reaction opt-in work without `ackReaction`, preserves the message-tool-only privacy guard, and keeps focused prepare/dispatch coverage around both explicit-enable and explicit-disable paths.

## Reproduction Assessment

Yes. Source inspection on current main shows a timestamped Slack message with `messages.statusReactions.enabled: true` but no resolved `ackReaction` leaves `shouldSendAckReaction` false, so `ackReactionPromise` stays null and dispatch disables the lifecycle controller.

## Solution Assessment

Yes, with current-main adaptation. Decoupling explicit status-reaction enablement in Slack prepare is the narrow fix, but the rebase must preserve the newer tool-only privacy gate and the explicit-disable fallback ack behavior.
