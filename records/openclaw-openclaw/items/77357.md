## Risks / Open Questions

- Forwarding customer audio, images, documents, or chat content between WhatsApp conversations needs an explicit authorization and privacy policy before implementation.
- The proposed `message_id` plus `target` API is not sufficient by itself because Baileys forwarding wants a retained full message object, while current OpenClaw inbound surfaces expose normalized message IDs, media paths, and context rather than an arbitrary full-message lookup contract.
- Adding `forward` would expand the shared message-action contract and likely the WhatsApp action gate/config surface, so maintainers should settle default enablement and compatibility expectations first.
- Open adjacent WhatsApp action PRs add edit/unsend and location actions; a forwarding design should coordinate with those action-gating and target-authorization patterns rather than adding a one-off path.
