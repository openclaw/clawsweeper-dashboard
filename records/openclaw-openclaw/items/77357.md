## Summary

Keep open: current main still lacks an outbound WhatsApp `forward` action, and adding one would be a new cross-chat text/media capability with privacy, target-authorization, config-gating, and Baileys message-retention decisions.

## What This Changes

The issue requests a WhatsApp channel `forward` action that forwards an existing message by message id to another chat target.

## Best Possible Solution

Design a WhatsApp-owned, config-gated forward action that safely resolves authorized targets, retains or looks up the required Baileys message object, updates discovery/runtime/docs/tests, and proves text plus media forwarding in a real WhatsApp setup with private data redacted.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Source inspection on current main verifies that WhatsApp discovery, dispatch, runtime, config, and the shared action-name contract do not currently expose `forward`.

## Solution Assessment

Unclear: the requested `message_id` plus `target` shape captures the user goal, but it is not yet the best implementation contract because Baileys forwarding needs a `WAMessage` and OpenClaw needs maintainer decisions around privacy and authorization.
