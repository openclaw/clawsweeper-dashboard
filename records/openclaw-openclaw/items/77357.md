## Summary

Keep open: current main still exposes only WhatsApp react/poll action handling and no forward path, and adding forwarding is a new privacy-sensitive plugin capability needing maintainer action-contract and authorization decisions.

## What This Changes

The issue requests a WhatsApp message-tool `forward` action that takes an original message id and target chat so agents can forward audio, images, documents, and other messages between conversations.

## Best Possible Solution

Design a gated WhatsApp-owned forward action that resolves message IDs safely, enforces target authorization, documents privacy semantics, and promotes a shared action only if multiple channels need it.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing contract. Source inspection of current main shows forwarding is absent from discovery, dispatch, runtime handling, config schema, docs, and the active listener surface.

## Solution Assessment

Unclear: `action=forward` with `message_id` and `target` is plausible, but maintainers need to decide whether forwarding is a shared message action or a WhatsApp-specific capability and what config/authorization gates are required.
