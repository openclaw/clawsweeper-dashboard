## Summary

Keep open: the PR addresses a source-confirmed `/pair qr` rendering bug, but it still needs maintainer judgment because the fix publishes pairing-specific reply metadata through a public plugin SDK subpath and changes the public surface budget.

## What This Changes

The branch moves device-pair `/pair qr` webchat fallback from a sensitive data-URL `mediaUrl` to typed pairing QR `channelData`, adds live-only gateway display blocks, renders them in TUI/Control UI, and updates SDK surface artifacts and tests.

## Best Possible Solution

Land the live-only QR rendering fix after maintainers either internalize the pairing QR metadata or explicitly accept and document it as a supported public SDK contract with aligned generated artifacts.

## Reproduction Assessment

Yes. Source inspection on current main shows `/pair qr` webchat fallback returns sensitive QR data through `mediaUrl`, while the gateway display path excludes sensitive media; the PR body also reports a parent-commit live regression repro.

## Solution Assessment

Partly. The live-only typed QR display is a good runtime shape for gateway/TUI/Control UI rendering and non-persistence, but publishing pairing-specific helpers through the public SDK is not clearly the narrowest maintainable boundary.
