## Summary

Keep open: this is a focused bug-fix PR for broken `/pair qr` display in gateway-backed chat surfaces, and I did not find a blocking functional defect in the patch. It should still get maintainer API-contract review because it adds pairing-specific symbols to the public plugin SDK reply-payload surface.

## What This Changes

The PR moves `/pair qr` webchat fallback from sensitive `mediaUrl` data URLs to typed pairing QR channel data, has the gateway build live-only QR image and terminal blocks, and teaches TUI and Control UI renderers to display them without persisting QR content.

## Best Possible Solution

Land the live-only QR rendering fix after maintainers either accept the new public SDK reply-payload symbols or internalize the pairing-specific metadata helper while keeping the non-persistence tests.

## Reproduction Assessment

Yes. Source inspection shows current main returns a sensitive `mediaUrl` data URL for `/pair qr` webchat fallback while TUI/assistant display paths do not have a typed terminal QR block; the PR body also reports a parent-commit live regression repro.

## Solution Assessment

Mostly yes. A live-only typed QR block is the right shape for gateway/TUI/Control UI rendering and transcript non-persistence, but the public plugin SDK export choice needs maintainer acceptance or internalization before merge.
