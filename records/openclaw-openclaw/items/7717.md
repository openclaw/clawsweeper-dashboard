## Summary

Keep open: current main and v2026.6.8 still lack a Discord selected-role trigger surface, and preflight still records role mentions only as generic mention metadata instead of turning configured role IDs into `wasMentioned`. The closed duplicate reports point back to this issue, while the open broader Discord routing/access-control issues only partially overlap.

## What This Changes

The issue requests Discord role mentions, such as a configured `@AI` role, to wake matching bot instances instead of requiring direct user mentions.

## Best Possible Solution

Add a Discord-owned, opt-in selected-role trigger at guild and channel scope, then wire it through config types, strict schema, metadata/docs, preflight, and focused tests while preserving current direct mention, @everyone/@here, sender allowlist, and `ignoreOtherMentions` semantics.

## Reproduction Assessment

Yes, at source level: with `requireMention` true, a guild message containing only a role mention leaves `wasMentioned` false because `mentionedRoles` is never compared with configured trigger role IDs. I did not run a live Discord replay in this read-only review.

## Solution Assessment

Unclear as a finalized contract: a selected-role opt-in integrated into the existing mention-state path appears to be the narrowest maintainable shape, but maintainers still need to choose naming, inheritance, and fan-out safety before implementation.
