## Summary

Keep this open: current main still lacks a Discord role-mention trigger surface, and the existing @everyone/@here work is adjacent but does not satisfy configured role-ID activation.

## What This Changes

Add support for configured Discord role mentions to wake mention-gated agents.

## Best Possible Solution

Implement an additive Discord guild/channel role-trigger config, compare mentioned role IDs against it in preflight, treat matches as explicit wake mentions, and update schema, generated metadata, docs, changelog, and focused tests.

## Reproduction Assessment

Yes from source inspection. Current main records role mentions as generic metadata, but the mention gate only sees explicit bot mentions, text patterns, @everyone/@here, and reply-to-bot state.

## Solution Assessment

Yes, an additive role-ID trigger config is the narrow maintainable direction. Reusing the existing roles key would be unsafe because it already controls sender access.

## Repair Work Prompt

Implement #7717 as an additive Discord role-mention trigger feature. Add a guild/channel config field such as triggerOnRoles?: string[] using Discord role IDs only; do not reuse the existing roles field because it is sender allowlist access control. Wire the field through src/config/types.discord.ts, src/config/zod-schema.providers-core.ts, Discord config metadata/help generation as needed, extensions/discord/src/monitor/allow-list.ts or the resolved channel config shape, and extensions/discord/src/monitor/message-handler.preflight.ts so a message whose mentionedRoles/mention_roles intersects the configured trigger list is treated as an explicit wake mention. Preserve existing requireMention, ignoreOtherMentions, @everyone/@here, reply-to-bot, allowBots, sender users/roles allowlist, routing, and thread behavior. Add focused tests for matched role mention, unmatched role mention, role-only message with requireMention true, and role-only message with ignoreOtherMentions true. Update docs/channels/discord.md, docs/gateway/config-channels.md if relevant, and CHANGELOG.md because this is user-facing. Stop if the implementation requires changing the meaning of existing roles or broad core/plugin API behavior.
