## Summary

Keep open. Current main still lacks a Discord role-mention trigger config or runtime path, the existing `roles` fields are sender allowlists, and the closed related reports point back to this issue as the canonical tracker rather than resolving it.

## What This Changes

Add Discord role-mention wake triggers, likely via a guild/channel `triggerOnRoles`-style config that treats configured role mentions as explicit mentions.

## Best Possible Solution

Add an opt-in Discord plugin config surface for role-trigger IDs and wire it through config types, strict schema, generated metadata/docs, preflight, and focused tests while preserving existing sender `roles` allowlist semantics.

## Reproduction Assessment

Yes. A guild message with `requireMention: true`, no bot user mention, and `mention_roles` populated follows the current preflight path where role IDs count as generic mentions but never become `wasMentioned`, so it reaches the no-mention drop path.

## Solution Assessment

Unclear until maintainers decide the public config semantics. The narrowest maintainable direction appears to be an additive `triggerOnRoles`-style Discord plugin option rather than overloading the existing sender `roles` allowlist.
