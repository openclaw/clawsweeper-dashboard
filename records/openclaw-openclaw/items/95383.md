## Summary

Keep open. This PR fixes a real current-main SDK normalization bug: terminal item events can carry `phase:"end"` with `status:"failed"` or `status:"blocked"`, but the current classifier returns `tool.call.completed` before it checks those failure statuses. The patch is narrow and has no blocking review findings; maintainers should still notice the public SDK event behavior change before merge.

## What This Changes

This PR reorders the SDK agent-event normalizer so failed/blocked terminal tool/item events map to `tool.call.failed` and adds focused regression coverage.

## Best Possible Solution

Land the narrow SDK classifier fix with regression coverage and clear release-note context that failed/blocked terminal tool events now emit `tool.call.failed`.

## Reproduction Assessment

Yes from source inspection: current main checks `phase:"end"` before failed/blocked status while the producer emits terminal item events with both fields. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. Moving the failure-status branch inside the SDK normalizer is the narrowest maintainable fix because producers already preserve the real status and the SDK client centralizes normalized event typing there.
