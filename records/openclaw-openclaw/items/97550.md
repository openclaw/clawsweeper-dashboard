## Summary

Keep open: the APNs relay hardening is at the right boundary and the updated PR body now has real loopback proof, but the patch introduces a small JSON-decoding regression that can turn a BOM-prefixed relay failure body into HTTP-derived success.

## What This Changes

The PR replaces APNs relay `response.json()` with a 16 MiB bounded read that returns `RelayResponseTooLarge` on overflow and updates relay response tests for real `Response` bodies and size-boundary cases.

## Best Possible Solution

Land the bounded APNs relay read after preserving `Response.json`/`TextDecoder` UTF-8 decoding semantics and after maintainers accept the intentional over-cap fail-closed behavior.

## Reproduction Assessment

Yes. Current main and the latest release call `response.json()` in `sendApnsRelayRequest`, so an oversized relay body is source-reproducible as an unbounded response-buffering path; the PR body also provides a real loopback relay proof.

## Solution Assessment

No, not yet. Bounding the response at the relay read boundary with the existing helper is the right fix shape, but the implementation should preserve `Response.json` UTF-8/BOM decoding semantics before merge.

- **[P2] Preserve Response.json BOM handling:** `src/infra/push-apns.relay.ts:287`
  - body: This swaps `response.json()` for `JSON.parse(buffer.toString("utf8"))`, but `Buffer.toString` keeps a leading UTF-8 BOM that `Response.json()` accepts. Decode with `TextDecoder` or strip the BOM before parsing.
