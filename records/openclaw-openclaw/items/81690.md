## Summary

Keep open: the documented contract is assistant-first result selection, and current main still lets a non-empty raw tool/toolResult result short-circuit the retry path before a late final assistant reply can be observed. The report is narrow and source-reproducible enough for a focused fix.

## What This Changes

The issue reports that `subagent_announce` can send a large truncated toolResult to the parent instead of the subagent's final assistant completion report.

## Best Possible Solution

Adjust completion capture so raw tool/toolResult fallback is provisional when a final assistant reply is expected, then add regression coverage for the large toolResult-before-final-assistant case.

## Reproduction Assessment

Yes, at source level: if `chat.history` momentarily ends at a toolResult, `readSubagentOutput` returns `latestRawText` and the retry helper stops before a later assistant reply can be observed. I did not run a live subagent timing repro in this read-only sweep.

## Solution Assessment

Yes, the best fix is a narrow change in the capture/extraction retry path, not a new config knob or broad delivery rewrite. Treat raw tool/toolResult fallback as provisional when completion text may still arrive, while preserving timeout and tool-only fallback behavior.
