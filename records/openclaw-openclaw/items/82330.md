## Summary

Keep open: current main still has source-visible paths where a completed sub-agent can fail to produce a Telegram-visible completion notification when requester wake/steer fails or the requester-agent handoff produces no visible payload. The adjacent 5.12-era fixes removed unsafe raw-child fallback and transcript leakage, but they do not fully solve this reported no-notification regression.

## What This Changes

The issue reports a regression in OpenClaw 2026.5.12 where completed sub-agents no longer send Telegram completion notifications after announce delivery moved through requester-agent handoff and transcript suppression.

## Best Possible Solution

Consolidate the active-requester and no-visible-output failure paths into one current-main-compatible completion delivery fix that preserves requester-agent mediation, avoids raw child-result sends, and proves visible Telegram delivery.

## Reproduction Assessment

Yes for a source-level reproduction: current code and tests show Telegram active-requester wake failure returning `delivered: false` with no gateway agent call and no send call. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No for the current behavior: returning `delivered: false` without a durable user-visible notification does not satisfy the documented completion-update contract. The best fix is a consolidated queue/mediated fallback that preserves the raw-output safety boundary, not a raw direct-send fallback or a new config switch as the first step.
