## Summary

Keep open: current main still has source-visible sub-agent completion announce paths that can end with `delivered: false` and no Telegram send when the requester wake or mediated visible-payload handoff fails. Adjacent fixes removed unsafe raw fallback and transcript leakage, but they do not fully cover this reported no-notification regression.

## What This Changes

The issue reports that after upgrading to OpenClaw 2026.5.12, successful sub-agent runs no longer post completion notifications to a Telegram direct message.

## Best Possible Solution

Keep the requester-agent safety boundary, but make completion announce failure paths produce either a sanitized visible notification or a durable queued retry with focused tests and Telegram proof.

## Reproduction Assessment

Yes at source level: the current Telegram announce-delivery test drives an active requester wake failure to `delivered: false` with no gateway call and no send call. I did not establish a live Telegram account reproduction in this read-only review.

## Solution Assessment

No for the current behavior: ending at `delivered: false` without a durable or visible notification conflicts with the documented push-completion contract. The safer direction is a mediated or sanitized queued fallback, not a raw child-result fallback or a new config switch as the first step.
