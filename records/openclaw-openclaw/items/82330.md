## Summary

Keep open: current main still has a source-visible path where sub-agent completion announces can end with `delivered: false` and no Telegram send when the requester-agent handoff produces no visible payload or cannot wake the active requester. The related fixed items removed unsafe raw-child fallback and hid internal prompt rows, but they do not fully cover this reported Telegram no-notification regression.

## What This Changes

The issue asks OpenClaw to restore Telegram delivery of sub-agent completion notifications after upgrading to 2026.5.12.

## Best Possible Solution

Keep the requester-agent safety boundary, but make completion announce failure paths produce a safe visible notification or durable queued wake instead of silently ending with no Telegram delivery.

## Reproduction Assessment

Yes for a source-level reproduction: the current Telegram announce-delivery test helper drives the active-requester wake failure to `delivered: false` with no send call. I did not run a live Telegram account in this read-only review.

## Solution Assessment

No for the current behavior: returning `delivered: false` without a durable notification conflicts with the push-based completion contract. The safer fix is a sanitized queued or mediated fallback, not a raw child-result fallback or a new config switch as the first step.
