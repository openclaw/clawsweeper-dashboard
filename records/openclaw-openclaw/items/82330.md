## Summary

Keep open: current main still has source-visible Telegram completion announce paths that can return `delivered: false` with no user-visible send when active requester wake fails or the mediated announce produces no visible payload. The adjacent fixes removed unsafe raw-child fallback and fixed some transient retry handling, but they do not fully satisfy this reported no-notification regression.

## What This Changes

The issue reports that after upgrading to OpenClaw 2026.5.12, completed sub-agent announcements no longer reach Telegram direct-message requesters.

## Best Possible Solution

Keep the requester-agent safety boundary, but make completion announce failure paths produce a sanitized visible notification or durable queued retry/failure signal instead of ending silently with no Telegram delivery.

## Reproduction Assessment

Yes, at source level: the current Telegram announce-delivery test helper drives active requester wake failure to `delivered: false` with no send call, and the no-visible-payload branch has the same terminal shape. I did not run a live Telegram account in this read-only review.

## Solution Assessment

No for the current behavior and the raw/config fallback suggestion: returning `delivered: false` without a durable user-visible outcome violates the push-completion contract. The safer fix is a sanitized mediated fallback or retryable delivery state, not raw child-result delivery or a new mode switch as the first step.
