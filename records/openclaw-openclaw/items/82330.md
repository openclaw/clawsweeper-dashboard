## Summary

Keep open: current main still has a source-visible path where Telegram sub-agent completion announces can end as `delivered: false` with no chat-visible notification when the requester-agent handoff produces no visible payload or the active requester cannot be woken. Adjacent fixes removed unsafe raw fallback and transcript leakage, but they do not fully cover this no-notification regression.

## What This Changes

The issue requests restoring Telegram delivery for sub-agent completion notifications after upgrading to OpenClaw 2026.5.12.

## Best Possible Solution

Review and land, or replace, a narrow queue/mediated-delivery fix that guarantees completion announces become visible or durably retried without raw-sending child output.

## Reproduction Assessment

Yes, at source level: current tests and code show an active Telegram requester wake failure can return `delivered: false` with no send call. I did not run a live Telegram account in this read-only review.

## Solution Assessment

No for the raw fallback/config-switch suggestion; the safer solution is a mediated or queue-backed delivery path that preserves the existing child-output safety boundary. The open queue-first PR looks aligned with that direction and should be reviewed against this report.
