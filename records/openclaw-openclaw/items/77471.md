## Summary

Keep open: current main already has a QMD backend and a mcporter-backed QMD MCP search path, but it does not implement the requested direct `memory.provider: "qmd-mcp"` or `memory.qmd.endpoint` routing to an existing streamable-HTTP QMD service. The remaining work is a feature and ownership decision, not a conservative close candidate.

## What This Changes

The issue requests a new QMD MCP-backed memory provider/endpoint so `memory_search` can use an already-warm local QMD service instead of cold-spawning QMD processes.

## Best Possible Solution

Decide whether this belongs as a documented streamable-HTTP QMD MCP runtime in `memory-core` or as a ClawHub memory plugin, then add focused config, docs, and tests for that chosen boundary.

## Reproduction Assessment

Not applicable for the main feature request. The bug note is not high-confidence reproducible from the issue alone; source inspection shows an existing mcporter-backed QMD MCP path and tests, but no live QMD HTTP service was exercised.

## Solution Assessment

Unclear: adding `memory.provider: "qmd-mcp"` directly to core may not be the best boundary because active memory plugins can register their own runtime and OpenClaw already has generic streamable-HTTP MCP config. If maintainers confirm the shipped `memory.qmd.mcporter` path is broken, that narrower bug should be fixed in `memory-core`.
