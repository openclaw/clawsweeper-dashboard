## Summary

Close as implemented: current main and shipped releases already route QMD memory searches through mcporter keep-alive using the current QMD MCP query shape, with config, docs, and regression coverage; the exact proposed `memory.provider: "qmd-mcp"` spelling is not needed.

## What This Changes

The issue requests a configurable QMD MCP-backed memory search provider to avoid cold-starting the local QMD memory search process.

## Best Possible Solution

Keep the documented `memory.backend: "qmd"` plus `memory.qmd.mcporter.*` and `searchTool` path; open a new focused issue only if a direct non-mcporter HTTP endpoint is still required after testing that shipped path.

## Reproduction Assessment

Not applicable for the requested feature: this review verified the current source, tests, docs, and release history instead of reproducing a failing current-main path. The reported mcporter compatibility concern is covered by the shipped QMD 1.1+ mcporter tests and implementation.

## Solution Assessment

Yes: the existing mcporter-backed QMD backend is the narrower maintainable solution because it reuses the QMD memory backend, existing MCP client surface, daemon keep-alive, custom tool override, and legacy fallback rather than adding a second memory-provider config namespace.
