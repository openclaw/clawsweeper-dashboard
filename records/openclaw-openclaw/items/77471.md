## Summary

Keep open: current main has a QMD backend and an mcporter bridge, but it does not implement the requested direct QMD MCP HTTP provider, and the report asks for new config/product surface rather than a conservative cleanup close.

## What This Changes

Request to route memory search through a warm QMD MCP HTTP service instead of cold process invocation and to address a claimed `memory.qmd.mcporter` transport mismatch.

## Best Possible Solution

Maintainers should decide the QMD integration boundary first, then either fix/document the existing mcporter Streamable HTTP setup or add a plugin-owned/direct Streamable HTTP QMD backend with tests and docs.

## Reproduction Assessment

Partially. Source inspection confirms current main lacks the requested direct `qmd-mcp` memory backend and still routes QMD search through mcporter or `qmd`, but the claimed transport mismatch was not reproduced live.

## Solution Assessment

Unclear. A new `memory.provider` key is not obviously the best fit because current public config uses `memory.backend` and `memory.qmd.mcporter`; the safer path is a maintainer decision on direct Streamable HTTP versus the existing mcporter/plugin surface.
