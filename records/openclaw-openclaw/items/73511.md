## Summary

Keep open. Current main still rejects explicit custom-scheme Control UI origins because `parseOrigin` compares the allowlist against `URL.origin === "null"`, and this PR is a narrow implementation candidate for the linked Gateway bug with no concrete review finding found.

## What This Changes

The PR changes Gateway origin normalization so custom-scheme origins such as `tauri://localhost` can match explicit `gateway.controlUi.allowedOrigins` entries, adds origin-check regression cases, and adds a changelog fix entry.

## Best Possible Solution

Land this narrow Gateway normalization change after maintainer review and green validation, preserving explicit allowlist enforcement for custom-scheme origins while keeping literal `Origin: null`, malformed origins, unlisted schemes, wildcard, Host-header fallback, and local-loopback behavior bounded by the existing rules.

## Reproduction Assessment

Yes. Current main plus the local Node URL check gives a high-confidence reproduction: `tauri://localhost` parses with `URL.origin === "null"`, so `checkBrowserOrigin` compares `"null"` to an explicit `tauri://localhost` allowlist entry and rejects it.

## Solution Assessment

Yes. Reconstructing `scheme://host` only for opaque URL origins with a host is the narrowest maintainable fix because it feeds the existing allowlist check instead of adding a Tauri/Electron-specific bypass.
