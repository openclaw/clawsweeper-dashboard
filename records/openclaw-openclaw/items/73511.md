## Summary

Keep open. Current main still rejects explicitly allowlisted custom-scheme Control UI origins because it compares the allowlist against URL.origin, which is the opaque string "null" for tauri/electron schemes; this PR is the active, narrow implementation candidate for #46520 and I found no discrete correctness issue in the patch.

## What This Changes

This PR changes Gateway origin normalization to preserve scheme://host when URL.origin is opaque null, adds custom-scheme allowlist regression cases, and adds a changelog fix entry.

## Best Possible Solution

Land this narrow Gateway normalization change after maintainer review and green CI: custom desktop-wrapper origins should match only when explicitly listed, while literal Origin:null, malformed origins, unlisted schemes, wildcard, Host-header fallback, and local-loopback behavior remain unchanged.

## Reproduction Assessment

Yes. Current main plus the Node URL check gives a high-confidence reproduction: tauri://localhost parses with URL.origin === "null", so checkBrowserOrigin compares "null" to an explicit tauri://localhost allowlist entry and rejects it.

## Solution Assessment

Yes. Reconstructing scheme://host only for opaque URL origins with a host is the narrowest maintainable fix because it feeds the existing allowlist check instead of adding a Tauri/Electron-specific bypass.
