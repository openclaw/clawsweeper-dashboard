## Summary

Keep open. Current main still rejects explicit custom-scheme Control UI origins because `parseOrigin` compares the allowlist against `URL.origin`, which is `"null"` for `tauri://`/`electron://`; the PR is a narrow active fix with regression coverage and a changelog entry, so it should go through maintainer auth-boundary review and normal CI rather than cleanup closure.

## What This Changes

The PR reconstructs `scheme://host` for opaque URL origins with a host in Gateway origin parsing, adds custom-scheme allowlist regression tests, and adds a changelog fix entry.

## Best Possible Solution

Land the narrow Gateway normalization change after maintainer auth-boundary review and green CI, preserving the existing requirement that custom desktop-wrapper origins only pass when explicitly listed while literal `Origin: null`, malformed origins, unlisted schemes, wildcard behavior, Host-header fallback, and local-loopback behavior remain unchanged.

## Reproduction Assessment

Yes. Current main plus the Node URL check gives a high-confidence reproduction: `tauri://localhost` parses with `URL.origin === "null"`, and `checkBrowserOrigin` compares that normalized origin to the explicit allowlist entry, so the match fails.

## Solution Assessment

Yes. Reconstructing `scheme://host` only for opaque URL origins with a non-empty host is the narrowest maintainable fix because it feeds the existing allowlist gate instead of adding a Tauri/Electron-specific bypass or requiring wildcard origins.
