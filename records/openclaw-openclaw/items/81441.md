## Summary

Keep open: current main still has a source-reproducible heartbeat scaffolding leak, and this PR is a focused implementation candidate with no blocking correctness findings. ## What This Changes

This PR routes legacy heartbeat reply text through `sanitizeAssistantVisibleText`, strips `<file_contents>` blocks and standalone plural tool-call marker lines, blocks ordinary non-tool delivery in heartbeat-tool mode except explicit internal notices, and adds focused heartbeat/sanitizer regressions.

## Best Possible Solution

Land exactly one canonical heartbeat/sanitizer fix for https://github.com/openclaw/openclaw/issues/81369, then close the issue with merge proof and avoid merging the overlapping branch separately.

## Reproduction Assessment

Yes. Current main shows heartbeat delivery normalizes raw `payload.text` without the shared assistant-visible sanitizer, so a heartbeat payload containing only `<tool_calls>` or `<file_contents ...>` is a high-confidence source reproduction path.

## Solution Assessment

Yes, with a branch-selection caveat. Routing heartbeat text through the shared sanitizer, adding focused stripping stages, and requiring `heartbeat_respond` in heartbeat-tool mode is the narrow maintainable fix; maintainers should choose this PR or https://github.com/openclaw/openclaw/pull/81434, not both.
