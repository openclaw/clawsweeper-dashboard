## Summary

Keep open: the performance direction is sound and real terminal proof is supplied, but the latest head still changes explicit-workspace image availability when no current metadata snapshot is registered.

## What This Changes

The PR reuses media availability planning in agent tool prep, passes an `availabilityResolved` fast path into image/video/music generation factories, and adds focused tests plus a changelog entry.

## Best Possible Solution

Keep the plan reuse, but compute image availability from the already loaded workspace-scoped snapshot/fallback so tool prep stays fast without changing workspace-specific plugin availability.

## Reproduction Assessment

Yes. The intended performance issue is source-reproducible from current main's plan-plus-factory duplicate availability checks, and the new blocking regression is source-reproducible from the PR head's current-only image snapshot lookup.

## Solution Assessment

No. Reusing the plan is the right direction, but image planning should reuse the loaded workspace snapshot or otherwise preserve `loadCapabilityMetadataSnapshot` fallback semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reuse the loaded workspace snapshot for image planning:** `src/agents/openclaw-tools.media-factory-plan.ts:230-235`
  - body: `snapshot` has already been loaded with `params.workspaceDir`, but this new branch switches explicit-workspace image planning to `getCurrentCapabilityMetadataSnapshot`. That helper returns undefined when no current metadata handoff is installed, while current main falls back to loading the requested workspace's manifest snapshot; in that case the `image` tool is hidden even though the workspace manifest/auth would make it available. Reuse the loaded `snapshot` for image planning, or otherwise preserve the fallback, and cover the no-current-snapshot workspace case.
  - confidence: 0.88
