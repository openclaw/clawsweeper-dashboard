## Summary

Keep open: the performance direction and supplied proof are useful, but the current head still bypasses part of the existing generation-tool auth fallback, so merging can hide media generation tools for existing config-backed provider setups.

## What This Changes

The PR centralizes optional media tool availability planning and passes `availabilityResolved` to image, video, and music generation tool factories to avoid repeated registration-time scans.

## Best Possible Solution

Keep the central availability-plan performance improvement, but preserve exact generation availability parity with the existing factory contract and add regressions for config-only provider API keys before merge.

## Reproduction Assessment

Yes from source inspection: the PR's central plan only checks explicit generation models and snapshot capability availability, while current main's factory helper also falls back through manifest provider IDs to `hasProviderAuthForTool`. Passing `availabilityResolved: true` then prevents that fallback from running.

## Solution Assessment

No; centralizing registration-time availability is the right layer for the performance goal, but this implementation is not yet the best fix because it drops part of the existing auth contract. The safer shape is to reuse or extract the current factory availability helper against the already-loaded snapshot, or add an equivalent fallback in the central plan.
