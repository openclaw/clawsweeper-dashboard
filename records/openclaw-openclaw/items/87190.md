## Summary

The PR is still needed because current main reproduces the reported Control UI gap: `sessions.usage` supports and defaults by `agentId`, while the current usage controller never sends an agent selection. The branch fixes the central request and includes useful live WebSocket proof, but it still leaves some agent-filtered usage data unscoped and has stale locale output, so it is not ready to merge as-is.

## What This Changes

This PR adds Control UI usage-page agent selection state, sends `agentId` in `sessions.usage` requests, renders agent filter options from the configured agent list, adds an older-gateway retry path, and extends usage tests/i18n strings.

## Best Possible Solution

Make the selected-agent filter scope every usage-page data source or derive the daily/totals view from the scoped `sessions.usage` result, keep the compatibility retry path, and regenerate the locale bundles for the new Default label before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main's usage controller omits `agentId`, while the backend defaults omitted usage requests to the default agent; the contributor also supplied Playwright WebSocket proof for the fixed request payload.

## Solution Assessment

No, not yet. Passing the selected agent through `sessions.usage` is the right core fix, but the PR also needs to keep daily cost data, selected-day totals, fallback totals, and generated locale output consistent with that selected agent.
