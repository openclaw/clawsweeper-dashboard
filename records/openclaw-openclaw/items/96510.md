## Summary

Keep open: the PR targets a real unbounded successful MCP response read, but the proposed success path uses a diagnostic prefix reader, so oversized 2xx bodies are truncated without an explicit overflow error and can produce ambiguous or partial parsing behavior.

## What This Changes

The PR adds a 16 MiB cap to successful Parallel MCP response body reads by replacing the success-path `response.text()` call with `readResponseTextLimited`.

## Best Possible Solution

Use a throwing bounded reader for successful MCP text, keep the 16 MiB cap aligned with provider JSON, and cover oversized success responses plus normal small responses in `parallel-mcp-search.runtime.test.ts`.

## Reproduction Assessment

Yes, at source level: current `postMcp` consumes any successful MCP body with `response.text()`, so a 2xx streaming response can be buffered without a byte cap. I did not run a live OOM repro in this read-only review.

## Solution Assessment

No, not yet: bounding the success read is the right layer, but `readResponseTextLimited` is a diagnostic prefix helper. The better fix is to use a throwing bounded reader for success bodies so over-limit data cannot be silently processed as a partial response.
