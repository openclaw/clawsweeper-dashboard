## Summary

Keep open for maintainer review: the patch appears correct and well proven for the managed npm plugin-update slice, but it intentionally changes upgrade and cleanup semantics by retaining old package generations until maintainers accept or bound that policy.

## What This Changes

The PR installs managed npm plugin updates into generation-specific project roots, marks replaced generations as retained, and skips retained generations during registry recovery and doctor cleanup.

## Best Possible Solution

Land the managed npm generation-retention fix only after maintainers explicitly accept or bound retained-generation lifetime, while keeping broader stale core-dist lifecycle work on the canonical issue.

## Reproduction Assessment

Yes for source-level review: current main can replace the active managed npm package tree while a running gateway still holds old module URLs, and the canonical issue includes recent Windows stale-hash proof. I did not run a fresh live reproduction in this read-only pass.

## Solution Assessment

Mostly yes for the managed npm slice: generation-root retention sits in the plugin install and registry boundary and avoids deleting live import targets. The unresolved part is whether indefinite retained package trees are the maintainer-approved long-term policy.
