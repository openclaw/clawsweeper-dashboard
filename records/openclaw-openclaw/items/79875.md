## Summary

Keep open: the PR targets a real `/btw` Copilot wrapper gap, but the patch still misses the default fallback stream path and the external real-behavior proof gate remains unsatisfied.

## What This Changes

The PR imports `wrapProviderStreamFn` into the `/btw` side-question runner, wraps the registered provider stream, and adds a Copilot-focused regression test.

## Best Possible Solution

Compose the existing provider wrapper around the actual `/btw` stream function, including the `streamSimple` fallback, add regression coverage for the no registered provider-stream case, then land after redacted live proof and green checks.

## Reproduction Assessment

Yes from source inspection: current main's `/btw` path can fall back to `streamSimple` without `wrapProviderStreamFn`, while GitHub Copilot IDE header injection lives in the provider `wrapStreamFn`. I did not run a live Discord/Copilot scenario.

## Solution Assessment

No. The direction is right, but the implementation must wrap the stream that will actually be invoked, not only `providerStreamBase`; it also needs coverage where `registerProviderStreamForModel` returns `undefined`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Wrap the actual fallback stream:** `src/agents/btw.ts:425-428`
  - body: Default Copilot models do not provide a `createStreamFn`, and `createTransportAwareStreamFnForModel` returns `undefined` unless proxy/TLS request overrides exist. Passing `providerStreamBase` into the wrapper means Copilot receives `undefined`, `wrapCopilotProviderStream` returns `undefined`, and `/btw` still falls back to unwrapped `streamSimple` without the IDE headers. Wrap `providerStreamBase ?? streamSimple` or otherwise pass the actual stream that will be invoked, and cover that fallback path.
  - confidence: 0.92
