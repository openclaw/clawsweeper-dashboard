## Summary

Current main already fixes the reported lane-deadlock failure mode with caller-owned command-lane task timeouts for embedded runs, and that fix shipped in v2026.5.2; the suggested Discord listener await conflicts with the tested non-blocking listener design.

## What This Changes

Report asks to prevent Discord and other inbound message handlers from permanently blocking session lanes when a handler or embedded-run promise never settles.

## Best Possible Solution

Keep the shipped command-lane timeout and embedded-run cleanup path, while leaving the Discord gateway listener non-blocking as the tested dispatch behavior.

## Reproduction Assessment

No. I did not live-run Discord, but the reported current-main failure path is covered by source and regression tests showing a never-settling queued task now times out and drains the next task.

## Solution Assessment

Yes. Fixing the command-lane/runtime owner path is the narrower maintainable solution; awaiting the Discord listener would contradict tests that intentionally keep gateway dispatch non-blocking.
