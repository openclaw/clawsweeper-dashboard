## Summary

Current main already addresses the reported permanent lane blockage with a queue-level task timeout and regression coverage; the Discord listener still hands off immediately by design, with ordering and recovery owned by the shared run/command queue.

## What This Changes

The issue reports Discord and other inbound message handlers hanging indefinitely when lane-queued work never settles, leaving session lanes blocked until gateway restart.

## Best Possible Solution

Keep the current shared queue timeout approach on main and let the next release carry it; separate lane-routing or browser-lane product work should stay tracked by narrower follow-up issues such as #41120.

## Reproduction Assessment

No high-confidence failing reproduction remains for current main from source inspection: the affected active-lane path now has a timeout and a regression test for a never-settling task draining queued work. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

Yes, this is the best current fix direction: bounding the shared command queue protects Discord and other inbound channels without making the Discord gateway listener block event dispatch. Awaiting the listener would fight the current channel architecture and would not address non-Discord lane hangs.
