## Summary

Keep open. Current main handles the Bonjour/ciao unhandled-rejection shape from the report, but the issue is still paired with open closing PR #71950 and current LAN/tailnet interface discovery paths can still propagate `os.networkInterfaces()` failures.

## What This Changes

The issue asks for the gateway to survive `uv_interface_addresses`/`os.networkInterfaces()` failures in restricted Nemoclaw/OpenShell sandboxes instead of disconnecting the TUI.

## Best Possible Solution

Review and land, revise, or close #71950 so gateway-critical LAN and tailnet discovery degrade through the existing safe interface-enumeration boundary while Bonjour-specific ciao handling remains plugin-owned.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by mocking `os.networkInterfaces()` to throw and exercising `pickPrimaryLanIPv4()` or `listTailnetAddresses()`; current tests still assert those paths throw, and ciao tests cover the exact reported SystemError text.

## Solution Assessment

Yes for the narrow hardening direction. The maintainable fix is to use safe enumeration only at best-effort LAN/tailnet discovery boundaries, while keeping dependency-specific ciao suppression inside the Bonjour plugin.
