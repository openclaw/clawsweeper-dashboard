## Summary

Keep open. Current main still leaves the reported `os.networkInterfaces()` failure reachable through raw LAN and tailnet discovery helpers, and the discussion already has PR #71950 open with closing syntax as the focused implementation candidate.

## What This Changes

The issue requests that Gateway startup survive `uv_interface_addresses` / `os.networkInterfaces()` failures in restricted sandbox environments instead of crashing and disconnecting the TUI.

## Best Possible Solution

Keep the issue open until PR #71950 or an equivalent narrow fix lands, hardening startup-critical LAN and tailnet discovery callers while preserving the raw `readNetworkInterfaces()` contract for code that intentionally wants propagation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by mocking `os.networkInterfaces()` to throw and calling `pickPrimaryLanIPv4()` or `listTailnetAddresses()`; current tests still assert both paths throw.

## Solution Assessment

Yes. The linked-PR direction is the narrowest maintainable solution: use the existing safe reader at best-effort gateway/tailnet discovery boundaries instead of globally changing raw helper semantics.
