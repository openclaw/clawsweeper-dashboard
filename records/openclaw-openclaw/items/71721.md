## Summary

Keep open. Current main handles the Bonjour/ciao unhandled-rejection shape, but raw LAN and tailnet interface enumeration still propagates `os.networkInterfaces()` failures, and the linked fix PR is still open with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports a sandbox `uv_interface_addresses` crash during Gateway startup and asks for network interface discovery to degrade instead of disconnecting TUI clients.

## Best Possible Solution

Land PR #71950 or an equivalent narrow change that uses the existing safe interface reader at best-effort Gateway LAN, tailnet, bind/discovery call sites while preserving the raw `readNetworkInterfaces()` contract for callers that intentionally need hard failures.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by mocking `os.networkInterfaces()` to throw and calling `pickPrimaryLanIPv4()` or `listTailnetAddresses()`; current tests explicitly assert both paths throw.

## Solution Assessment

Yes. The best direction is the focused fix represented by PR #71950: harden the best-effort Gateway/tailnet discovery boundaries rather than globally changing raw helper semantics.
