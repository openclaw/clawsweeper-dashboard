## Summary

Keep open. Current main has Bonjour-specific handling for the original `@homebridge/ciao`/NetworkManager `uv_interface_addresses` crash signature, but this issue is still paired with open unmerged PR #71950 via closing syntax, and current main still leaves related raw LAN/tailnet `os.networkInterfaces()` failures reachable in source and tests. Repository cleanup policy says the issue should stay open until the linked PR lands, is replaced, or is closed.

## What This Changes

The issue requests that Gateway startup degrade gracefully instead of crashing or disconnecting when `os.networkInterfaces()` throws a `uv_interface_addresses` SystemError inside a Nemoclaw/OpenShell sandbox.

## Best Possible Solution

Keep the issue open while maintainers resolve PR #71950 or an equivalent narrow follow-up: preserve the Bonjour/ciao-specific suppression already on main, and harden best-effort LAN/tailnet discovery callers with the existing safe network-interface reader without changing the raw helper contract globally. The eventual merged fix should include focused regression tests and a user-facing changelog entry if behavior changes ship.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by mocking `os.networkInterfaces()` to throw and exercising the current Bonjour ciao classifier plus the raw `pickPrimaryLanIPv4()` and `listTailnetAddresses()` paths; current tests already encode these behaviors.

## Solution Assessment

Yes. The maintainable direction is narrow: keep ciao-specific process-error handling inside the Bonjour plugin, and use `safeNetworkInterfaces()` only at best-effort Gateway/tailnet discovery boundaries that should degrade instead of crashing.
