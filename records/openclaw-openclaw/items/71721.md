## Summary

Keep open. Current main handles the Bonjour/ciao unhandled-rejection class from the reported stack, but the issue is paired with open closing PR #71950 and current main still has raw LAN and tailnet interface-discovery paths that can propagate the same `os.networkInterfaces()` failure.

## What This Changes

Make gateway startup tolerate sandbox `os.networkInterfaces()` / `uv_interface_addresses` failures so the TUI can still connect.

## Best Possible Solution

Resolve the linked PR path: land #71950 after its blockers are fixed, or explicitly close/retarget it if the Bonjour-only handling is deemed sufficient.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by mocking `os.networkInterfaces()` to throw: current main covers the exact ciao SystemError text and still has LAN/tailnet helper tests that assert the same failure propagates.

## Solution Assessment

Yes for the narrow hardening direction. Keeping ciao failure handling in the Bonjour plugin and using the existing safe network-interface seam at best-effort LAN/tailnet boundaries is the maintainable path, but the issue should not close before the linked PR is resolved.
