## Summary

Keep open. Current main handles the reported Bonjour/ciao `uv_interface_addresses` unhandled-rejection class, but this issue is paired with open closing PR #71950 and current main still has raw LAN and tailnet interface-enumeration paths that can throw when `os.networkInterfaces()` is unavailable.

## What This Changes

The issue asks OpenClaw to keep the gateway running in restricted Nemoclaw/OpenShell sandboxes where `os.networkInterfaces()` throws `uv_interface_addresses returned Unknown system error 1`.

## Best Possible Solution

Land or deliberately retarget #71950 after review so gateway-critical LAN/tailnet discovery uses the existing safe interface-enumeration boundary while the Bonjour-specific ciao suppression remains in the plugin.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by mocking `os.networkInterfaces()` to throw and exercising the current LAN/tailnet helpers; the ciao tests also cover the exact reported SystemError text.

## Solution Assessment

Yes for the narrow hardening direction. The maintainable fix is to keep dependency-specific ciao handling inside the Bonjour plugin and use `safeNetworkInterfaces()` only at best-effort LAN/tailnet discovery boundaries, rather than changing the raw helper contract globally.
