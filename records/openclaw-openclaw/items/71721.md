## Summary

Keep open. Current main now handles the exact Bonjour/ciao `uv_interface_addresses` SystemError path as non-fatal, but the broader `os.networkInterfaces()` failure is still reachable through raw LAN and tailnet helpers, and PR #71950 remains open with closing syntax as the focused fix candidate.

## What This Changes

The issue requests that gateway startup and discovery tolerate sandboxed `os.networkInterfaces()` / `uv_interface_addresses` failures so the gateway does not disconnect the TUI.

## Best Possible Solution

Keep the issue open until PR #71950 or an equivalent narrow fix lands, preserving the raw `readNetworkInterfaces()` contract while hardening best-effort gateway LAN, tailnet, and discovery boundaries that should degrade when sandbox interface enumeration is unavailable.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by mocking `os.networkInterfaces()` to throw and calling `pickPrimaryLanIPv4()` or `listTailnetAddresses()`; current tests still assert both paths throw.

## Solution Assessment

Yes. The best direction is the narrow boundary fix already represented by PR #71950: use the existing safe reader at best-effort gateway/tailnet discovery call sites rather than globally changing raw helper semantics.
