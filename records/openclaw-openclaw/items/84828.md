## Summary

This PR should stay open, but it is not merge-ready: the session-context helper can inflate configured context caps, and the Telegram progress change conflicts with the documented one-draft progress contract while lacking real Telegram proof.

## What This Changes

The branch adds a shared session context-token resolver and changes Telegram progress mode from editable drafts to standalone durable progress messages, with updated tests.

## Best Possible Solution

Preserve larger same-model session context without overriding explicit caps, keep Telegram progress compatible by default, and make any durable standalone progress an explicit documented maintainer-approved path with live proof.

## Reproduction Assessment

Yes, by source inspection: the PR head's progress-mode tool event reaches a standalone deliverReplies call that spreads transcriptMirror, and the new resolver returns the maximum of cap candidates including DEFAULT_CONTEXT_TOKENS. I did not run tests because this review is read-only.

## Solution Assessment

No. The session-context direction may be useful, but the implementation must preserve configured caps and Telegram progress should not replace the documented draft semantics without a maintainer decision, migration story, and real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve configured context caps:** `src/agents/session-context-tokens.ts:78`
  - body: The new resolver returns `Math.max(...candidates)` after always adding `DEFAULT_CONTEXT_TOKENS`, so an explicit lower cap such as `agents.defaults.contextTokens: 20000` or a model-level `contextTokens: 160000` can be raised to 200k or more. That breaks the documented cap contract and changes upgrade behavior for existing sessions; preserve larger same-model session values only when they are known to be the intended runtime cap, not by maximizing every fallback.
  - confidence: 0.92
- **[P1] Prevent progress messages from entering transcripts:** `extensions/telegram/src/bot-message-dispatch.ts:1197-1199`
  - body: The standalone progress sender spreads `deliveryBaseOptions` into `deliverReplies`, and that base object includes `transcriptMirror` whenever the inbound message has a session key. Since `deliverReplies` mirrors delivered content when that callback is present, progress labels and command summaries can be appended as assistant transcript content; pass `transcriptMirror: undefined` for non-final progress and add progress-mode coverage.
  - confidence: 0.93
- **[P1] Preserve label-only progress mode:** `extensions/telegram/src/bot-message-dispatch.ts:680-682`
  - body: With draft lanes disabled for progress mode, this branch returns without sending anything when `streaming.progress.toolProgress` is false. Current docs say that setting hides tool lines but keeps the progress label behavior, so existing users with label-only progress lose their visible in-progress cue unless this is made an explicit documented migration.
  - confidence: 0.88
- **[P2] Restore the post-final progress guard:** `extensions/telegram/src/bot-message-dispatch.ts:671-678`
  - body: The patched `pushStreamToolProgress` no longer checks `answerLane.finalized`, `finalAnswerDeliveryStarted`, or `finalAnswerDelivered` before accepting a progress line. A late tool or command-output callback after final delivery can now create extra progress in draft-streaming paths; keep that guard for every progress path, including standalone progress.
  - confidence: 0.78
