## Summary

Keep open: the session-context part may be useful, but the Telegram progress change conflicts with the documented progress-mode contract, can mirror progress into transcripts, removes the final-delivery progress guard, and lacks required real Telegram proof.

## What This Changes

The branch adds a session context-token preservation helper and switches Telegram progress mode from editable drafts to standalone progress messages with updated tests.

## Best Possible Solution

Keep the session-context preservation if validated, but make Telegram progress compatible with the existing contract unless maintainers explicitly approve and document a migration to durable standalone progress.

## Reproduction Assessment

Yes, by source inspection: a progress-mode tool event reaches the new standalone sender, and deliverReplies mirrors delivered content whenever transcriptMirror is present. I did not run tests because this review is read-only.

## Solution Assessment

No. The session-context direction may be useful, but the Telegram solution should not persist progress into transcripts or silently replace documented progress-mode semantics without an explicit maintainer decision and upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Prevent progress messages from entering transcripts:** `extensions/telegram/src/bot-message-dispatch.ts:1198-1199`
  - body: The new standalone progress sender spreads deliveryBaseOptions into deliverReplies, which includes transcriptMirror whenever the inbound message has a session key. deliverReplies mirrors any delivered content when that callback is present, so progress text like tool labels or command summaries can become assistant transcript content and feed later session context; clear transcriptMirror for non-final progress and add coverage.
  - confidence: 0.93
- **[P1] Preserve label-only progress mode:** `extensions/telegram/src/bot-message-dispatch.ts:680-682`
  - body: With draft lanes disabled for progress mode, this branch returns without sending anything when streaming.progress.toolProgress is false. Current docs and config hints say that setting hides tool lines but keeps the progress label until final delivery, so existing users with label-only progress lose their visible in-progress cue.
  - confidence: 0.88
- **[P2] Restore the final-delivery progress guard:** `extensions/telegram/src/bot-message-dispatch.ts:672-678`
  - body: The patched pushStreamToolProgress no longer checks answerLane.finalized or final-delivery state before accepting progress. Because final delivery is async, a late tool or command-output callback while the final answer is pending can send extra progress around the final response, which current main explicitly suppresses.
  - confidence: 0.78
