## Summary

Keep open: the goal is plausible, but the patch currently risks transcript/session contamination, breaks documented Telegram progress-mode behavior, and has only mock/unit-test proof for a user-visible Telegram change.

## What This Changes

The PR adds a shared session context-token resolver and changes Telegram progress mode to send tool progress as standalone messages with updated tests.

## Best Possible Solution

Keep the session-context preservation if validated, but make Telegram progress explicit and compatible: non-final progress must not mirror to transcripts, existing progress settings must keep their documented behavior unless maintainers approve a migration, and live Telegram proof should verify the final user flow.

## Reproduction Assessment

Yes by source inspection: a progress-mode tool event with a session key reaches the new standalone sender, and current `deliverReplies` mirrors delivered content whenever `transcriptMirror` is present. I did not execute tests because this review is read-only.

## Solution Assessment

No. The session-context direction may be useful, but the Telegram solution should not persist progress into transcripts or silently replace documented progress-mode semantics without an explicit maintainer product decision and upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Prevent progress messages from entering transcripts:** `extensions/telegram/src/bot-message-dispatch.ts:1198-1199`
  - body: The new standalone progress sender spreads `deliveryBaseOptions` into `deliverReplies`, which includes `transcriptMirror` whenever the inbound message has a session key. `deliverReplies` mirrors any delivered content when that callback is present, while the existing test contract keeps non-final tool progress out of transcripts. This would persist `Exec`/command progress as assistant transcript content and feed it back into later session context; pass `transcriptMirror: undefined` for standalone progress and add progress-mode coverage.
  - confidence: 0.93
- **[P1] Preserve label-only progress mode:** `extensions/telegram/src/bot-message-dispatch.ts:680-682`
  - body: With draft lanes disabled for progress mode, this branch returns without sending anything when `streaming.progress.toolProgress` is false. Current docs and config hints say that setting hides tool lines but keeps the progress label until final delivery, so existing users with label-only progress lose their visible in-progress cue. Preserve that behavior or make the migration an explicit documented maintainer choice.
  - confidence: 0.88
- **[P2] Restore the post-final progress guard:** `extensions/telegram/src/bot-message-dispatch.ts:671-678`
  - body: The patched `pushStreamToolProgress` no longer checks `answerLane.finalized`, `finalAnswerDeliveryStarted`, or `finalAnswerDelivered` before accepting a progress line. In non-progress draft modes, a late tool or command-output callback after the final answer can now start another progress update after final delivery, which current main explicitly suppresses. Keep the guard for every progress path, including the new standalone progress branch.
  - confidence: 0.78
