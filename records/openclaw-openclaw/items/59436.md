## Summary

Keep this PR open. Current main already has the lower-risk search/filter/activity/limit-before-row-build behavior, but it still resolves child sessions by scanning the store inside each returned row; the child-index idea remains useful, but this branch predates current child-link liveness semantics and should not be merged as-is.

## What This Changes

The PR changes `src/gateway/session-utils.ts` to prefilter, sort, and limit session entries before row enrichment and to add a per-request child-session reverse index for child lookups.

## Best Possible Solution

Keep the shipped prefilter/search/limit behavior and land a small current-main patch that indexes child candidates while reusing the existing latest-run, lifecycle, moved-child, and store-only visibility predicates. Broader summary indexing, pagination, direct sessionId lookup, and TTL caching should stay in the canonical performance trackers unless benchmarks show this narrow fix is insufficient.

## Reproduction Assessment

Yes. The discussion includes a stage reproduction where row builds collapsed from hundreds to only surviving recent rows, and current source gives a deterministic remaining reproduction path: many returned sessions still call a child resolver that scans the full store per row.

## Solution Assessment

No, not as-is. The optimization direction is good, but the proposed branch should be adapted to current main so the child index is an implementation detail behind the existing child-link filters rather than a raw `spawnedBy`/`parentSessionKey` relationship map.
