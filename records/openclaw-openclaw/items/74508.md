## Summary

Keep this PR open: the branch addresses real Telegram streaming/throttling and fork-resume bugs, but the latest head still introduces two blocking draft-stream regressions around the new chat send gate and 429 retry lifecycle.

## What This Changes

The PR shares Telegram apiThrottler state by bot token across polling and CLI fallback delivery, chains long reasoning draft previews, restores fork-resume system prompts, adds QQBot SSRF guarding, updates changelog/config baseline files, and adds regression coverage.

## Best Possible Solution

Keep the owner-local fixes, but move the chat gate to wrap only actual Telegram API sends, add a self-scheduled 429 retry path, and rebase or replace the dirty branch before merge consideration.

## Reproduction Assessment

Yes. Current main has source-visible paths for the original overflow, throttler, CLI fallback, and fork-resume bugs, and the PR regressions are source-reproducible from the final draft-stream and draft-loop control flow.

## Solution Assessment

No, not as patched. The overall direction is maintainable, but the per-chat gate must only reserve around real API calls and the 429 backoff needs a retry mechanism that does not depend on later token updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reserve the chat gate only for real API calls:** `extensions/telegram/src/draft-stream.ts:278-280`
  - body: The new gate runs before trimming/rendering, duplicate checks, overflow splitting, and the minInitialChars guard. A normal first token under the initial preview threshold consumes the 3-second chat slot without sending anything, delaying the first real preview and making Telegram streaming choppy again. Move the gate to immediately wrap the actual sendMessage/editMessageText call after no-op checks.
  - confidence: 0.9
- **[P2] Schedule the 429 retry when no later update arrives:** `extensions/telegram/src/draft-stream.ts:373-386`
  - body: On 429 this path records rateLimitedUntilMs and returns false, but the draft loop only restores pendingText and returns without scheduling another flush. If the last edit hits a rate limit and no newer update arrives, the backoff is never waited out and the preview is not retried. Either schedule the retry here or teach the loop to retry after the backoff.
  - confidence: 0.86
