## Summary

Keep open: current main still lacks the requested fixes, and the PR head is useful but not merge-ready because the draft stream can throttle no-op updates and can stall after a Telegram 429 when no later update arrives.

## What This Changes

The PR adds per-token Telegram apiThrottler sharing for polling and CLI fallback, chains Telegram reasoning overflow previews, restores fork-resume system prompts, hardens QQBot API fetches with SSRF guarding, and updates tests/changelog/config baseline.

## Best Possible Solution

Keep the useful owner-local fixes, but repair the draft-stream gate and 429 retry lifecycle, then rebase or replace the dirty branch before merge consideration.

## Reproduction Assessment

Yes. Current main has source-visible overflow, throttler, CLI fallback, and fork-resume prompt paths, and the PR regressions are source-reproducible from the PR head draft-stream and draft-loop control flow.

## Solution Assessment

No, not as patched. The general direction is maintainable, but the chat gate should reserve only around real Telegram API calls and the 429 path needs a retry scheduled even when no newer update arrives.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move the chat send gate to wrap actual API calls:** `extensions/telegram/src/draft-stream.ts:278-280`
  - body: The new gate is acquired before trimming, rendering, duplicate/min-initial checks, and overflow splitting. A first token below `minInitialChars` or a duplicate/no-op update can reserve the 3-second chat slot without any Telegram API call, delaying the first real preview and making streaming choppy again. Acquire the gate immediately before `sendMessage`/`editMessageText` after no-op checks.
  - confidence: 0.9
- **[P2] Schedule a retry after Telegram 429 backoff:** `extensions/telegram/src/draft-stream.ts:376-389`
  - body: On 429 this records `rateLimitedUntilMs` and returns `false`, but the loop only restores `pendingText` and returns when no newer update arrived. If the last edit hits a rate limit, the preview remains pending forever until another token/update happens. Schedule a flush after `retry_after` or make the loop schedule one when `sent === false`.
  - confidence: 0.87
