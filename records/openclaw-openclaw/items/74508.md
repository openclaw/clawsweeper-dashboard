## Summary

Keep open for maintainer review. The PR addresses real current-main gaps and I did not find a discrete blocking bug in the diff, but it is a broad contributor change across Telegram streaming, CLI runner resume behavior, shared draft-loop retry semantics, and QQBot SSRF hardening.

## What This Changes

The PR shares Telegram API throttling by bot token, applies it to CLI fallback sends, chains reasoning overflow drafts, preserves fork-resume system prompts, keeps newer pending draft text after failed sends, and guards QQBot API fetches with SSRF protection.

## Best Possible Solution

Land a reviewed, exact-head-green version if maintainers accept the combined Telegram, CLI-runner, draft-loop, and QQBot hardening scope.

## Reproduction Assessment

Yes. Current main has source-level reproduction paths for independent Telegram throttlers, an unthrottled CLI fallback Bot, stop-on-overflow draft previews, stale pending text restore after false sends, and fork-resume prompt skipping; the PR adds focused regression tests for those paths.

## Solution Assessment

Yes, with scope review. Sharing the grammY apiThrottler transformer by token matches the dependency queue boundary, the overflow/retry changes stay in Telegram draft streaming and shared draft-loop seams, and the fork-resume fix is localized; the safer alternative is only to split QQBot hardening if maintainers want narrower review.
