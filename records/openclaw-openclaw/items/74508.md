## Summary

Keep open. Current main still has the Telegram throttling, overflow, pending-text, and fork-resume prompt gaps this PR targets, but the PR head needs changes because its new shared limiter does not cover preview final edits and exact-head CI is failing.

## What This Changes

The PR shares Telegram API throttling by bot token, extends Telegram draft/reasoning overflow and rate-limit handling, preserves fork-resume system prompts, guards QQBot API fetches with SSRF checks, and updates tests/changelog.

## Best Possible Solution

Route the same Telegram send limiter through preview finalization/cleanup paths, resolve exact-head checks, then land a reviewed version if maintainers accept the combined scope.

## Reproduction Assessment

Yes, source-level. Current main has clear paths for independent Telegram throttlers, an unthrottled CLI fallback Bot, stop-on-overflow draft previews, stale pending text restore after false sends, and fork-resume prompt skipping; I did not run live Telegram reproduction in this read-only review.

## Solution Assessment

No. The shared throttler and overflow direction are maintainable, but the proposed per-dispatch limiter reaches draft streams and deliverReplies while preview final edits/deletes still bypass it.
