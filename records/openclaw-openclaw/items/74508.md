## Summary

Keep open: the Telegram overflow problem is valid and the PR implements useful streaming changes, but the latest head still misses the execute-level fork-resume prompt eligibility gate, so the fork-session system-prompt fix described in the PR body is not actually complete.

## What This Changes

This PR updates Telegram draft streaming to chain overflowing reasoning previews across messages, add shared edit throttling and 429 backoff, suppress partial reasoning-tag answer leakage, pass system prompts on CLI fork resumes, and add tests plus a changelog entry.

## Best Possible Solution

Keep the Telegram overflow/rate-limit work, but compute fork-resume status before resolving system-prompt usage, treat fork resumes as prompt-eligible for that special path, and add executePreparedCliRun coverage proving the prompt file or prompt arg reaches the spawned CLI command.

## Reproduction Assessment

Yes. Call executePreparedCliRun with an existing cliSessionId, backend resumeArgs containing --fork-session, default systemPromptWhen: "first", and a configured system prompt; resolveSystemPromptUsage receives isNewSession: false and returns null before the fork-resume branch runs.

## Solution Assessment

No. The PR's helper-level buildCliArgs change is necessary but not sufficient because executePreparedCliRun can still discard the prompt before buildCliArgs receives it; the safer fix is to move fork-resume detection ahead of prompt resolution.
