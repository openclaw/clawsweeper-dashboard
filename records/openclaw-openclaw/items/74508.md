## Summary

Keep open: this PR addresses real current-main Telegram streaming/throttling gaps, but PR head still has a pending-text loss in the overflow continuation path that should block merge.

## What This Changes

The PR shares Telegram throttlers per bot token, changes Telegram draft/reasoning overflow and fork-resume behavior, hardens QQBot API fetches with the SSRF guard, and updates tests plus changelog entries.

## Best Possible Solution

Keep the shared throttler, fork-resume prompt handling, and QQBot SSRF guard if maintainers accept the scope, but preserve pending draft updates during internal overflow rotations before merging.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: trigger an oversized first Telegram draft update, let the first split chunk's send be in flight, call update() with newer text, and the continuation rotation clears that pending value.

## Solution Assessment

No, not yet. The main direction is maintainable, but the overflow split should rotate messages without clearing or losing newer pending stream text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve pending text during overflow rotations:** `extensions/telegram/src/draft-stream.ts:404`
  - body: When a first oversized update is split, `sendOrEditStreamMessage(fitText)` can be in flight while `update()` records newer text. The continuation then reaches this `resetStreamToNewMessage()`, whose helper calls `loop.resetPending()`, erasing that newer value; after the old continuation succeeds, the loop sees no pending update and the latest stream text is lost. Use an internal rotation path that preserves or merges pending updates for overflow splits.
  - confidence: 0.87
