Keep open. The PR is still addressing real current-main Telegram streaming/rate-limit and CLI fork-resume gaps, but the latest head is not merge-ready: static review found one limiter coverage defect and exact-head CI is not green.

## What This Changes

The PR shares Telegram throttling by bot token, adds draft overflow/backoff handling and per-dispatch send limiting, preserves fork-resume system prompts, guards QQBot API fetches with the SSRF guard, and updates tests/changelog.

## Best Possible Solution

Route the same Telegram send limiter through preview finalization/cleanup paths, then land a reviewed exact-head-green version if maintainers accept the combined scope.

## Reproduction Assessment

Yes, source-level. Current main has clear paths for independent Telegram throttlers, an unthrottled CLI fallback Bot, stop-on-overflow draft previews, stale pending text restore after false sends, and fork-resume prompt skipping; I did not run live Telegram reproduction in this read-only review.

## Solution Assessment

No. The shared throttler and overflow direction are maintainable, but the proposed per-dispatch limiter currently reaches draft streams and deliverReplies while preview final edits/deletes still bypass it, leaving a rate-limit path in the surface the PR is trying to fix.
