## Summary

Keep open. The latest head improves the main Telegram throttling path, fork-resume prompt handling, draft overflow behavior, and QQBot fetch hardening, but it still leaves preview finalization/cleanup and voice fallback text sends outside the new per-chat limiter.

## What This Changes

The PR adds a shared per-token Telegram throttler, a per-dispatch Telegram send limiter, reasoning overflow chaining, CLI fork-resume system prompt handling, QQBot SSRF-guarded fetches, and related tests/changelog entries.

## Best Possible Solution

Finish the limiter plumbing so every mutating Telegram send in a streaming dispatch, including preview finalization/cleanup and voice fallback text chunks, uses the same per-chat limiter while keeping the per-token throttler, fork-resume, overflow, and QQBot SSRF fixes.

## Reproduction Assessment

Yes, source-level. Current main clearly has independent Telegram throttlers, an unthrottled CLI fallback Bot, stop-on-overflow draft previews, fork-resume prompt skipping, and stale pending-text restore after false sends; I did not run live Telegram traffic.

## Solution Assessment

No. The overall direction is maintainable, but the current PR head still misses two mutating Telegram send paths that can collide with active draft-stream edits.

## Review Findings

- **[P2] Route preview callbacks through the limiter:** `extensions/telegram/src/bot-message-dispatch.ts:754`

  - confidence: 0.86

Pass the limiter into that helper and acquire it before each fallback `sendTelegramText` call so fallback chunks cannot collide with active streaming edits.
  - confidence: 0.77
