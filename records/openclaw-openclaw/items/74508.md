## Summary

Keep open. The PR is now draft with an active scope pivot to per-account Telegram API throttling, and the latest head still has a concrete retry-loop bug in the new 429 handling: a rate-limited draft edit can remain stale unless another update or final flush happens later.

## What This Changes

The branch changes Telegram draft streaming to chain reasoning overflow into continuation messages, adds Telegram throttling/backoff changes, suppresses partial reasoning-tag answer flicker, passes system prompts on fork-session resumes, and updates tests plus changelog.

## Best Possible Solution

Finish the draft by making rate-limit handling align with the draft-loop contract, keep the account-scoped Telegram throttler at the Telegram plugin boundary, preserve the overflow/fork-resume regression coverage, then run the Telegram and changed-lane checks before normal maintainer review.

## Reproduction Assessment

Yes. Current main has a source-level reproduction for the original overflow stop at `extensions/telegram/src/draft-stream.ts:207`; the PR-head retry bug is also reproducible from source by making `editMessageText` return 429 and observing that `flush()` restores pending text without scheduling another attempt.

## Solution Assessment

No, not yet. The overflow chaining and fork-resume prompt fixes are in the right owner areas, but the new 429 path should either schedule a retry through the draft loop or defer that behavior to the shared account throttler instead of returning false without a follow-up trigger.
