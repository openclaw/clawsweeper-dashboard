## Summary

Keep open: current main still has the Telegram overflow, throttler-sharing, CLI fallback, and fork-resume prompt gaps this PR targets, but PR head still introduces two source-visible Telegram draft-stream regressions that should block merge.

## What This Changes

The PR adds a per-token Telegram apiThrottler singleton for polling and CLI fallback Bot instances, chains Telegram reasoning draft overflow, preserves fork-resume system prompts, hardens QQBot API fetches with the SSRF guard, and updates tests, changelog, and generated config baseline hashes.

## Best Possible Solution

Keep the owner-local fixes, but move the chat gate to wrap only actual Telegram API sends/edits, add a self-scheduled retry after 429 backoff, and refresh the dirty branch before merge consideration.

## Reproduction Assessment

Yes. Source inspection of current main shows the original overflow, throttler, CLI fallback, and fork-resume prompt paths, and PR head makes the two review findings reproducible from draft-stream control flow.

## Solution Assessment

No, not as patched. The broad direction is maintainable, but the per-chat gate must not reserve time for no-op attempts and the 429 path needs an automatic retry that does not depend on a later token update.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move chat gating to real API sends:** `extensions/telegram/src/draft-stream.ts:278-280`
  - body: The new gate is acquired before trimming, duplicate checks, overflow splitting, and the minInitialChars guard. A normal first token under the initial preview threshold can consume the 3-second chat slot without any sendMessage/editMessageText call, delaying the first real preview and making streaming choppy. Move this gate to wrap only the actual Telegram API send/edit path after no-op checks.
  - confidence: 0.9
- **[P2] Schedule retry after 429 backoff:** `extensions/telegram/src/draft-stream.ts:386-389`
  - body: On 429 this records rateLimitedUntilMs and returns false, but the draft loop only restores pendingText and returns. If the last edit hits the rate limit and no newer update arrives, nothing calls back into sendOrEditStreamMessage after retry_after, so the preview is never retried. Schedule the retry from this path or teach the loop to re-flush after the backoff.
  - confidence: 0.88
