## Summary

Keep this PR open. The Telegram overflow direction looks useful and is covered by focused tests, but the fork-session resume fix is still incomplete in the real execute path because the system prompt is filtered out before the new fork-resume condition can use it.

## What This Changes

The PR updates Telegram reasoning draft streaming overflow, Telegram draft throttling/rate-limit handling, partial reasoning-tag splitting, cli-runner fork-resume prompt argument handling, tests, and the changelog.

## Best Possible Solution

Preserve the Telegram overflow work, but compute fork-resume status before resolving system prompt usage and treat fork resumes as prompt-eligible in executePreparedCliRun, then cover that full execute path with a regression test.

## Reproduction Assessment

Yes. A source-level reproduction is clear: use a backend with resumeArgs containing --fork-session and default systemPromptWhen='first', then call executePreparedCliRun with an existing session id; systemPromptArg resolves to null before the PR's fork-resume condition is applied.

## Solution Assessment

No. The proposed direction is narrow, but the current patch only proves buildCliArgs and misses the earlier executePreparedCliRun prompt eligibility gate; the safer fix is to pass an effective isNewSession of isNew || isForkResume into resolveSystemPromptUsage.
