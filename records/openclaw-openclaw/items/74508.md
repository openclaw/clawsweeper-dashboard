## Summary

Keep open. The PR addresses real current-main Telegram, CLI runner, and QQBot gaps, but the new 429 retry path can clobber newer draft text queued while the failed edit request is still in flight, so it should be fixed before merge. The dedicated security pass is otherwise clear.

## What This Changes

This PR shares Telegram grammY throttling by bot token across polling and CLI fallback sends, changes Telegram draft/reasoning overflow and rate-limit handling, reapplies system prompts for fork-session CLI resumes, wraps QQBot API fetches in the SSRF guard, and updates tests plus changelog entries.

## Best Possible Solution

Keep the PR’s shared throttler, fork-resume prompt, overflow chaining, and QQBot SSRF directions, but fix the draft retry path so a 429 failure cannot overwrite newer queued draft text, add a regression where the newer update arrives before the rejected edit promise resolves, then complete the targeted Telegram/CLI checks and changed gate.

## Reproduction Assessment

Yes. A source-level reproduction is clear: start with an existing draft message, call update("stale") so editMessageText is in flight, call update("newest") before that promise rejects with 429, and the shared draft loop restores the older failed snapshot when the PR’s new catch returns false. The existing PR test only updates after flush returns, so it misses this in-flight case.

## Solution Assessment

No, not yet. Most of the implementation is narrow and aligned with the current-main root causes, but the rate-limit retry fix must preserve newer pending text already queued in the draft loop before this is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve queued draft text after 429 failures:** `extensions/telegram/src/draft-stream.ts:361`
  - body: When the new 429 catch returns `false`, `createDraftStreamLoop.flush()` restores the failed snapshot into `pendingText`. If another `update()` arrived while the edit request was still in flight, that newer text is already in `pendingText` and gets overwritten by the stale snapshot, so the retry can send old reasoning/answer text. Add a regression where the newer update happens before the rejected edit promise resolves and preserve the newer pending value on false-return retries.
  - confidence: 0.87

## Repair Work Prompt

Repair PR https://github.com/openclaw/openclaw/pull/74508 without broadening its scope. The observable bug is in the new Telegram draft-stream 429 retry path: when editMessageText/sendMessage rejects with a 429 after another stream.update() has already queued newer text, draft-stream.ts returns false and src/channels/draft-stream-loop.ts restores the older failed snapshot over the newer pending text. Fix the retry semantics so newer pending text is preserved while still retrying the failed snapshot when nothing newer arrived. Add a regression where the edit promise is held, stream.update("newest") is called before that promise rejects with 429, timers advance past retry_after, and the retry sends "newest" rather than "stale". Keep the existing shared per-token throttler, fork-session system-prompt, overflow chaining, and QQBot default SSRF guard direction. Do not add dependencies, workflow changes, broad channel refactors, or new public API. Changelog is already present; update it only if the wording changes materially.
