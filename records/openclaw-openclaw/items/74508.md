## Summary

Keep this PR open. The Telegram throttler, draft overflow, and fork-session prompt changes address real current-main gaps and the latest PR check-runs are green, but the QQBot SSRF hardening is not safe to merge as written because it uses an SSRF policy helper that exempts the configured hostname from private-network checks rather than strictly blocking private-network targets.

## What This Changes

The PR adds a Telegram token-shared grammY throttler, installs it on CLI fallback sends, extends Telegram draft/reasoning overflow and rate-limit handling, reapplies CLI system prompts on fork-session resumes, adds QQBot guarded fetch use, tests, and changelog entries.

## Best Possible Solution

Keep the Telegram fixes together if maintainers agree on the product behavior, but repair or split the QQBot SSRF change so it either uses a strict hostname allowlist that preserves private-network checks or is handled in a separate security-focused PR with explicit threat-model review.

## Reproduction Assessment

Yes. The current-main issues are reproducible from source: polling Bot instances create independent apiThrottler transformers, the CLI fallback Bot has no throttler, draft streaming stops when rendered text exceeds 4096 chars, and fork-session resumes are not treated as prompt-bearing sessions; the QQBot security finding is also source-reproducible from the allowedHostnames skip path.

## Solution Assessment

No, not as written. The Telegram and CLI changes are mostly narrow maintainable fixes, but the QQBot SSRF policy uses an exemption helper rather than a strict allowlist, so the security hardening should be corrected or split before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Use a strict QQBot SSRF hostname policy:** `extensions/qqbot/src/engine/api/api-client.ts:61`
  - body: This derives the guard policy with `ssrfPolicyFromHttpBaseUrlAllowedHostname()`, but that helper sets `allowedHostnames`, which `resolveHostnamePolicyChecks()` treats as an exemption from private-network DNS/IP checks. A configured `channels.qqbot.baseUrl` on `127.0.0.1` or another private-resolving host would still be fetched, so the advertised private-network SSRF hardening is not achieved. Use a hostname allowlist that keeps private-network checks active, or explicitly split/describe private operator endpoints as allowed.
  - confidence: 0.88
