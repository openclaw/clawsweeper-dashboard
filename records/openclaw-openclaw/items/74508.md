## Summary

Keep this PR open. The Telegram overflow work addresses a real current-main behavior, but the current PR head still does not fix the fork-session resume system-prompt path end to end because the execute path resolves prompt eligibility before treating fork resumes as prompt-eligible.

## What This Changes

This PR changes Telegram reasoning draft overflow handling, Telegram draft stream throttling/rate-limit backoff, partial reasoning-tag splitting, CLI fork-resume system-prompt argument handling, tests, and the changelog.

## Best Possible Solution

Preserve the Telegram overflow and rate-limit improvements, but compute fork-resume status before resolving system-prompt usage in `executePreparedCliRun`, treat fork resumes as prompt-eligible only for that special path, and add execute-level regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `executePreparedCliRun` with `resumeArgs` containing `--fork-session`, default `systemPromptWhen: "first"`, a configured system prompt, and an existing session id; `resolveSystemPromptUsage` receives `isNewSession: false` and returns null.

## Solution Assessment

No. The proposed direction is narrow, but the current patch only makes `buildCliArgs` accept a prompt during fork resumes and misses the earlier execute-level eligibility gate that decides whether a prompt exists at all.
