## Summary

Keep open. The PR is directionally useful and addresses source-visible current-main defects, but the latest head still leaves Telegram preview finalization and a voice fallback text path outside the new shared per-chat limiter, so the rate-limit fix is incomplete.

## What This Changes

The PR adds a per-token Telegram API throttler singleton, applies it to polling and CLI fallback bots, changes Telegram draft streaming overflow and 429 handling, preserves system prompts on CLI fork resumes, fixes draft-loop retry state, adds QQBot SSRF guarding, and updates tests/changelog.

## Best Possible Solution

Thread the shared per-chat limiter through preview edit/delete and voice fallback text sends, keep the per-token throttler and overflow/fork-resume fixes, then land only after focused tests and changed-gate validation pass.

## Reproduction Assessment

Yes, source-level. Current main has independent Telegram throttlers, an unthrottled CLI fallback Bot, stop-on-overflow draft previews, fork-resume system prompt skipping, and stale pending-text restore after false sends; I did not run live Telegram reproduction in this read-only review.

## Solution Assessment

No. The main design is maintainable, but the proposed in-dispatch limiter still misses preview finalization/cleanup and voice fallback text sends, so a narrower follow-up repair should route those calls through the same limiter before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Queue preview callbacks through the limiter:** `extensions/telegram/src/bot-message-dispatch.ts:478-480`
  - body: The new `sharedEditRateLimiter` is passed into draft streams and `deliverReplies`, but it never reaches the `createLaneTextDeliverer` callbacks: final preview edits still call `editMessageTelegram` directly and preview cleanup still calls `bot.api.deleteMessage` directly. After a throttled draft flush, finalization can therefore issue an immediate edit/delete outside the bucket and still hit the Telegram per-chat rate limit this PR is trying to avoid. Thread the limiter into these callbacks before making the API call.
  - confidence: 0.84
- **[P3] Apply the limiter to voice fallback text sends:** `extensions/telegram/src/bot/delivery.replies.ts:340`
  - body: `deliverMediaReply` now accepts a `rateLimiter` and acquires it for normal media sends, but the voice privacy and caption-too-long fallback branches call `sendTelegramVoiceFallbackText`, whose `sendTelegramText` loop never acquires the limiter. When a voice reply falls back while a draft stream is active, those text chunks bypass the queue and can collide with streaming edits. Pass the limiter into that helper and acquire it before each fallback text send.
  - confidence: 0.73
