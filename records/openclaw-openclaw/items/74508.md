## Reproduction Assessment

## Solution Assessment

No, not as patched. The direction is maintainable, but the per-chat gate must only reserve around real API calls and 429 backoff needs a self-scheduled retry path that does not depend on later token updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reserve the chat gate only for real API calls:** `extensions/telegram/src/draft-stream.ts:278-280`
  - body: The new gate runs before trimming/rendering, duplicate checks, overflow splitting, and the minInitialChars guard. A normal first token under the initial preview threshold consumes the 3-second chat slot without sending anything, so the first real preview is delayed and the PR reintroduces choppy Telegram streaming. Move the gate to immediately wrap the actual sendMessage/editMessageText call after no-op checks.
  - confidence: 0.9
- **[P2] Schedule the 429 retry when no later update arrives:** `extensions/telegram/src/draft-stream.ts:376-389`
  - body: On 429 this path records rateLimitedUntilMs and returns false, but createDraftStreamLoop only restores pendingText and returns; it does not schedule another flush. If the last edit hits a rate limit and no newer update arrives, the backoff is never waited out and the preview is not retried. Either wait/schedule the retry here or teach the loop to retry after the backoff.
  - confidence: 0.84
