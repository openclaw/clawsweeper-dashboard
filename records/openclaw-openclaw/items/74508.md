## Summary

Keep open. The PR addresses real current-main Telegram streaming/throttling, CLI fork-resume, and QQBot SSRF gaps, but the latest head still misses two Telegram mutating send paths that can bypass the new limiter.

## What This Changes

The PR adds a per-token Telegram `apiThrottler` singleton, Telegram reasoning overflow/backoff/dispatch limiter changes, CLI fork-resume system-prompt handling, QQBot SSRF-guarded fetches, tests, and changelog entries.

## Best Possible Solution

Finish the Telegram limiter plumbing so every mutating send in a streaming dispatch shares the same per-chat limiter, then rerun targeted Telegram/agent/channel tests and the changed gate before merge.

## Reproduction Assessment

Yes, source-level. Current main clearly has independent Telegram polling throttlers, an unthrottled CLI fallback Bot, stop-on-overflow draft previews, and fork-resume prompt skipping; I did not run live Telegram traffic.

## Solution Assessment

No. The overall approach is maintainable, but the current PR head still leaves preview finalization/cleanup and voice fallback text outside the limiter it introduces.
