## Summary

This PR should stay open because current main still shows the Telegram throttling/overflow and fork-resume prompt gaps the branch targets, but the PR head introduces a blocking per-chat gate timer in the draft-stream flush path that can hang or time out the new regression tests.

## What This Changes

The PR adds Telegram per-token throttler sharing, CLI fallback throttling, fork-session system-prompt handling, reasoning overflow/backoff changes, QQBot SSRF guarding, changelog/config-baseline updates, and regression coverage.

## Best Possible Solution

Keep the owner-local Telegram, CLI-runner, and QQBot fixes, but rework the per-chat gate so overflow/backoff paths remain testable and do not add recursive real-time waits before maintainers consider merging.

## Reproduction Assessment

Yes. Current main has source-visible paths for draft-stream overflow stopping, independent Telegram throttlers, CLI fallback throttler bypass, and fork-resume prompt omission; the PR blocker is source-reproducible from the newly inserted 3-second gate before recursive flush work.

## Solution Assessment

No, not as currently patched. The per-token throttler sharing and owner-local fixes are maintainable, but the per-chat gate needs to be moved, injected, reset, or explicitly advanced in tests so it does not hang or slow the new regression coverage.
