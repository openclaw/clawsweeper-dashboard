## Summary

Keep this PR open: current `main` still has the Telegram streaming/throttling and CLI fork-resume gaps the PR targets, but the latest PR head introduces a remaining P2 draft-stream overflow race that can drop newer pending text during internal message rotation.

## What This Changes

The PR updates Telegram draft streaming overflow/rate-limit handling, shares Telegram API throttlers between polling and CLI fallback delivery, re-applies CLI system prompts on fork-session resumes, adds QQBot SSRF guarding, and updates tests/changelog/baseline hashes.

## Best Possible Solution

Keep the shared per-token throttler, fork-resume prompt fix, overflow chaining, and QQBot SSRF guard if maintainers accept the scope, but repair overflow rotations so pending stream text is preserved before merge.

## Reproduction Assessment

Yes. Source inspection on current `main` shows the overflow stop, split/bypassed Telegram throttler paths, and fork-resume prompt omission; PR-head source inspection shows a deterministic pending-text drop in the overflow split path.

## Solution Assessment

No, not yet. The overall direction is maintainable, but internal overflow message rotation should not clear the draft loop's pending text while an in-flight send can be racing with newer updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve pending text while rotating overflow messages:** `extensions/telegram/src/draft-stream.ts:404`
  - body: When an oversized preview is split, this internal rotation calls `resetStreamToNewMessage()`, which clears the draft loop's pending text. If `update()` receives newer text while the first chunk's `sendMessage` is in flight, that newer value is erased before the continuation is sent, so a final flush can miss the latest stream content. Use an internal rotation path that preserves or merges pending updates instead of clearing them.
  - confidence: 0.86
