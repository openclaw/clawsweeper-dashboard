Keep open, but this PR is not merge-ready: the Mattermost bug is source-reproducible on current main, while the branch now conflicts with the refactored ingress flow and changes the denied-sender context boundary without an explicit policy gate or regression coverage.

## What This Changes

The PR modifies the Mattermost monitor so a non-allowlisted group sender sets a history-only flag, skips the command gate, records pending history, then returns without sending a reply.

## Best Possible Solution

Make denied-sender context a maintainer-approved Mattermost context policy, port it to the current ingress/commandAccess flow, and add focused tests for history-only/no-reply/no-command behavior.

## Reproduction Assessment

Yes, at source level. Current main returns on `group_policy_not_allowlisted` before `recordPendingHistory()` is defined or called, while later onchar/missing-mention drops record pending history before returning.

## Solution Assessment

No. The submitted branch is a plausible implementation idea, but the best fix needs an explicit Mattermost context policy or maintainer-approved default plus a port to the current ingress flow and focused regression coverage.
