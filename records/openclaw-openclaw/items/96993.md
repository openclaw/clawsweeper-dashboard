## Label Justifications

- `P2`: This is normal-priority runtime memory hardening with limited blast radius and a focused correctness blocker.
- `merge-risk: 🚨 compatibility`: The guard can reject existing valid proxy SSE streams based on transport chunk coalescing rather than actual SSE line size.
- `merge-risk: 🚨 availability`: A valid proxied streaming LLM request can fail before complete SSE lines are processed.

## Summary

Keep open: the hardening target is valid, but the guard currently caps raw transport chunks before draining complete SSE lines, so it can reject valid proxied streams based on fetch chunking rather than an oversized unterminated line.

## What This Changes

The PR adds a 64 KiB internal line-buffer cap to the runtime proxy SSE parser and a regression test for a delimiter-free oversized proxy response.

## Best Possible Solution

Keep the cap, but drain complete newline-delimited proxy SSE lines first, enforce the 64 KiB limit only on the remaining unterminated tail, and add a coalesced-valid-chunk regression test.

## Reproduction Assessment

Yes, by source inspection. Current main leaves an unterminated proxy SSE line buffered indefinitely, and the PR-introduced valid coalesced-chunk failure follows directly from checking buffer.length before buffer.split("\n").

## Solution Assessment

No. The cap is the right mitigation, but the best narrow fix is to process complete lines first and cap only the remaining unfinished tail.
