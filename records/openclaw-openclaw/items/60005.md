## What This Changes

Adds MCP resources to the channel and plugin-tools MCP servers, including version/status/conversation/tool/config resources and plugin-tools list/read tests.

## Best Possible Solution

Rebase onto current main's shared MCP tools server/handlers, expose only allowlisted or centrally redacted resource data, and align the resource API boundary with the open gateway MCP Apps work before merge.

## Reproduction Assessment

Yes for the review defects: PR-head source shows openclaw://config/mcp returns arbitrary server fields after local masking, and plugin-tool calls use a direct tool.execute path. No real MCP client behavior proof is available for the feature itself.

## Solution Assessment

No. The feature direction is plausible, but this patch duplicates sensitive config scrubbing and plugin-tool handling instead of extending current shared helpers and settling the overlapping MCP resource boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route config resources through shared redaction:** `src/mcp/plugin-resources.ts:105-143`
  - body: `openclaw://config/mcp` clones each server entry and masks only selected fields before returning every other key. Since `McpServerConfig` permits arbitrary additional keys, credential-like custom fields such as `apiKey`, `authToken`, or `clientSecret` can leak to any MCP client that reads the resource; build this payload through shared redaction or a strict allowlist.
  - confidence: 0.92
- **[P1] Preserve the shared plugin-tool MCP handler:** `src/mcp/plugin-tools-serve.ts:54-87`
  - body: This branch rebuilds the plugin tool map and calls `tool.execute` directly, while current main routes plugin MCP tools through shared handlers that filter `ownerOnly` tools and wrap `before_tool_call` approval enforcement. Add resources on top of the shared server/handlers instead of replacing that path.
  - confidence: 0.88
