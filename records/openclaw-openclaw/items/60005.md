## Summary

Keep open: current main does not implement this MCP resource feature, and the branch still has blocking security/correctness issues plus missing real behavior proof. The resource API surface also overlaps the open MCP Apps gateway work, so this needs maintainer security/product review rather than cleanup closure.

## What This Changes

The PR adds MCP resource list/read support to channel and plugin-tools MCP servers, including version, status, conversation, tool, and config resources plus plugin-tools tests.

## Best Possible Solution

Rebase the resource feature onto current shared MCP server helpers and shared config redaction, then align the exposed resource API with the MCP Apps gateway contract before merge.

## Reproduction Assessment

Yes for the review defects: PR-head source inspection shows arbitrary MCP server fields are serialized by openclaw://config/mcp and plugin tools are executed directly. Current-main tests and handlers show the expected ownerOnly and before_tool_call enforcement path.

## Solution Assessment

No. The feature direction is plausible, but this patch duplicates sensitive config scrubbing and plugin-tool dispatch instead of extending the current shared helpers and the related gateway MCP Apps resource boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route MCP config resources through shared redaction:** `src/mcp/plugin-resources.ts:105-143`
  - body: `openclaw://config/mcp` returns every property from each MCP server entry after masking only headers, env, and selected URL pieces. Because `McpServerConfig` allows arbitrary keys, auth-like fields such as `apiKey`, `authToken`, or `clientSecret` would be exposed to any MCP client reading the resource; build this payload from the shared config redaction path instead of a local allowlist.
  - confidence: 0.92
- **[P2] Preserve the shared plugin-tool MCP handler:** `src/mcp/plugin-tools-serve.ts:54-86`
  - body: The branch reconstructs a local tool map and calls `tool.execute` directly, while current main routes plugin MCP calls through shared handlers that filter `ownerOnly` tools and enforce `before_tool_call` approval hooks. Rebase the resource additions onto that helper instead of reintroducing direct execution.
  - confidence: 0.87
