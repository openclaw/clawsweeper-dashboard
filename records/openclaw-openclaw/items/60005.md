## Summary

Keep open, but this PR should not merge as-is: current main still lacks MCP resource support, while the branch exposes config over MCP with incomplete redaction and predates the current shared plugin-tools MCP handler.

## What This Changes

The PR adds channel and plugin-tools MCP resource list/read support, including OpenClaw version, bridge status, conversations, tool inventory, and redacted plugin/MCP config resources with plugin-tools tests.

## Best Possible Solution

Ship MCP resources through the current shared MCP server/helper structure, reuse shared config redaction for every exposed config payload, and coordinate the external resource contract with the gateway MCP Apps work.

## Reproduction Assessment

Yes. Static review is enough: the PR's readResource path for openclaw://config/mcp serializes arbitrary MCP server fields outside its allowlisted masks, and its plugin-tools server shape predates current main's policy-enforcing helper.

## Solution Assessment

No. The feature direction is plausible, but the maintainable path is to extend the current shared handlers and redaction utilities instead of adding parallel MCP server and config-scrubbing logic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use shared redaction for MCP config resources:** `src/mcp/plugin-resources.ts:105-143`
  - body: `openclaw://config/mcp` clones each server entry and masks only headers, env, and selected URL pieces before returning the rest of the object. MCP server entries allow arbitrary keys, so fields such as `apiKey`, `authToken`, or `clientSecret` would be exposed to any MCP client reading this resource; route this payload through the existing config redaction helper instead.
  - confidence: 0.92
- **[P2] Preserve the shared plugin-tool MCP handler:** `src/mcp/plugin-tools-serve.ts:53-86`
  - body: The branch keeps a local tool map and direct `tool.execute` path, while current main routes plugin MCP tools through `createPluginToolsMcpHandlers`/`createToolsMcpServer` to filter `ownerOnly` tools and enforce `before_tool_call` approvals. Rebase the resource support around that helper instead of restoring the older direct execution path.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff exposes config over MCP and uses a stale plugin-tool execution path, so security review is required before merge.

Concerns:

- **[high] MCP config resource can leak custom secret fields:** `src/mcp/plugin-resources.ts:105`
  - body: The proposed `openclaw://config/mcp` resource returns arbitrary MCP server properties after masking only known fields. Because server entries allow additional properties, auth-like custom fields can be exposed to MCP clients.
  - confidence: 0.92
- **[medium] Stale tool handler can bypass approval hooks:** `src/mcp/plugin-tools-serve.ts:53`
  - body: The PR branch predates the current plugin-tools handler split and contains direct plugin tool execution. If rebased in that shape, plugin-tool owner filtering and `before_tool_call` approval enforcement would be bypassed.
  - confidence: 0.86
