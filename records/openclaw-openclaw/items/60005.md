## Summary

Keep this PR open for maintainer review: the MCP resource feature is not implemented on current main, but the branch still has concrete security and stale-handler regressions and overlaps the open MCP Apps resource boundary in https://github.com/openclaw/openclaw/pull/66168.

## What This Changes

The branch adds MCP resource list/read support to channel and plugin-tools MCP servers, including version, bridge status, conversation, tool, plugin config, and MCP config resources plus plugin-tools resource tests.

## Best Possible Solution

Add MCP resource support after maintainers choose the channel/plugin/gateway boundary, reusing shared MCP tool handlers and shared config redaction for every config-derived payload.

## Reproduction Assessment

Yes for the review defects: source inspection of the PR head shows arbitrary MCP server properties are returned by openclaw://config/mcp, and the branch directly executes plugin tools instead of using current main's shared MCP handler. Current main also has a test proving before_tool_call approval must block MCP plugin-tool execution.

## Solution Assessment

No. The feature direction is plausible, but this patch should be rebased onto the shared MCP server/helper structure and aligned with the related gateway MCP Apps resource contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Route MCP config resources through shared redaction:** `src/mcp/plugin-resources.ts:105-143`
  - body: `openclaw://config/mcp` returns every MCP server property after masking only headers, env, and selected URL pieces. Since `McpServerConfig` allows arbitrary keys, auth-like fields such as `apiKey`, `authToken`, or `clientSecret` would be exposed to any MCP client reading the resource; build this payload from the shared config redaction path instead.
  - confidence: 0.92
- **[P2] Preserve the shared plugin-tool MCP handler:** `src/mcp/plugin-tools-serve.ts:54-86`
  - body: The branch reconstructs a local tool map and calls `tool.execute` directly, while current main routes plugin MCP calls through `createToolsMcpServer`/`createPluginToolsMcpHandlers` to filter `ownerOnly` tools and run `before_tool_call` approval hooks. Rebase the resource support onto that helper instead of reintroducing direct execution.
  - confidence: 0.86
