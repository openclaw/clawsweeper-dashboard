## Summary

Keep open: current main still does not implement this resource surface, but the PR has security/correctness blockers, missing real behavior proof, and overlaps an open gateway MCP Apps resource design that has not superseded it yet.

## What This Changes

Adds MCP resource list/read support for channel bridge metadata and plugin-tool/config inspection, plus plugin-tools resource tests.

## Best Possible Solution

Rebase the feature onto current main's shared MCP tool server/helpers, use shared config redaction for any config-derived resource payload, and align the resource API with the chosen gateway MCP Apps boundary.

## Reproduction Assessment

Yes for the review defects: source inspection of the PR head shows `openclaw://config/mcp` serializes arbitrary server fields and plugin MCP calls use a local direct-execute handler. Current main tests define that `before_tool_call` approval must block MCP plugin-tool execution.

## Solution Assessment

No. The feature direction is plausible, but this patch duplicates config scrubbing and plugin-tool handling instead of extending current shared helpers and resolving the overlapping gateway MCP Apps resource boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Route MCP config resources through shared redaction:** `src/mcp/plugin-resources.ts:105-143`
  - body: `openclaw://config/mcp` clones each server entry and masks only headers, env, and selected URL pieces before returning every other key. Since `McpServerConfig` accepts additional keys, auth-like fields such as `apiKey`, `authToken`, or `clientSecret` would be exposed to any MCP client reading the resource; build this payload from the shared config redaction path instead of a local allowlist.
  - confidence: 0.92
- **[P2] Preserve the shared plugin-tool MCP handler:** `src/mcp/plugin-tools-serve.ts:54-86`
  - body: The branch builds a local tool map and calls `tool.execute` directly, while current main routes plugin MCP calls through `createToolsMcpServer`/`createPluginToolsMcpHandlers` to filter `ownerOnly` tools and enforce `before_tool_call` approvals. Rebase the resource support onto that helper instead of reintroducing direct execution.
  - confidence: 0.86
