## Summary

This PR should stay open for human review rather than cleanup-close: current main does not implement the requested channel/plugin-tools MCP resource surface, but PR head still has blocking security and behavior regressions and lacks real behavior proof.

## What This Changes

The branch adds MCP resource list/read support to the channel and standalone plugin-tools MCP servers, exposing version, status, conversation, tool, and config resources with tests.

## Best Possible Solution

Rework resource support around shared MCP handlers and central or allowlist-based config redaction, coordinate the API boundary with https://github.com/openclaw/openclaw/pull/66168, and require redacted real MCP client proof before merge.

## Reproduction Assessment

Yes for the review defects: PR-head source inspection shows the config resource returns arbitrary server fields after local masking and the plugin-tools path calls tool.execute directly. The requested MCP resource surface is a new feature, so there is no broken current-main behavior to reproduce.

## Solution Assessment

No. The feature direction may be useful, but this patch should extend current shared handlers and central redaction instead of adding duplicate sensitive config and tool-execution paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route config resources through shared redaction:** `src/mcp/plugin-resources.ts:105-143`
  - body: `openclaw://config/mcp` clones each server entry, masks only selected known fields, and returns every other key. `McpServerConfig` permits arbitrary extra keys, so `apiKey`, `authToken`, or `clientSecret`-style fields can leak to any MCP client that reads the resource; build this payload from a strict allowlist or the shared config redaction path.
  - confidence: 0.92
- **[P1] Preserve shared plugin-tool handling:** `src/mcp/plugin-tools-serve.ts:77-87`
  - body: This branch replaces current main's shared plugin-tool MCP server with a local CallTool handler that calls `tool.execute` directly. That drops ownerOnly filtering, before_tool_call wrapping, and abort-signal forwarding from `createPluginToolsMcpHandlers`; add resources on top of the shared server/handlers instead.
  - confidence: 0.9
