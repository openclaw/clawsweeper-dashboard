## Summary

Keep open: current main still lacks this MCP resource support, but this PR is not safe to merge as-is because it exposes MCP config with local field-by-field redaction and predates the current shared plugin-tools MCP handler. Related #66168 is still open and overlaps the broader resource protocol direction.

## What This Changes

The PR adds channel and plugin-tools MCP resources for version, bridge status, conversations, tools, and config, wires resource handlers into the servers, and adds plugin-tools resource tests.

## Best Possible Solution

Ship a rebased resource implementation through the current shared MCP helpers and reviewed resource contract, with shared config redaction, docs, changelog, and targeted tests for secret redaction and approval enforcement.

## Reproduction Assessment

Yes. Static review is enough: a config entry like mcp.servers.demo.apiKey would pass through the PR's openclaw://config/mcp resource unchanged, and the PR branch's plugin-tools server directly executes tools instead of using current main's shared policy-enforcing handler.

## Solution Assessment

No. The feature direction is plausible, but the maintainable solution should extend current shared MCP handlers/resource protocol work and reuse the existing config redaction utilities instead of adding parallel redaction and tool execution paths.
