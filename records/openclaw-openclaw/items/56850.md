## Summary

Keep open: current main and the latest release still expose static cron schedule kinds only, while the June 17 field report reinforces the same unmet state-aware cadence need. This is useful, but it needs maintainer product/API design because it would add a new persisted scheduling condition and state-evaluation contract.

## What This Changes

Request to let cron or heartbeat jobs choose a shorter or longer next interval based on runtime state conditions such as approaching deadlines, quiet periods, or urgent alerts.

## Best Possible Solution

Design a narrow core scheduler contract for state-aware cadence first, including condition evaluation ownership, persistence, docs, migration behavior, and focused cron/heartbeat regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection confirms the requested behavior is absent because current cron schedules are limited to at/every/cron and heartbeat scheduling has no state-condition interval contract.

## Solution Assessment

Unclear: the requested `conditional_interval` shape is plausible, but not yet proven to be the best product/API boundary. A safer path is a maintainer-designed scheduler condition contract or explicit plugin/hook seam before implementation.
