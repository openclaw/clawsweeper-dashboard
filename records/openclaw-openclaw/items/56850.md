## Summary

Keep open: current main and v2026.6.10 still expose only static `at`, `every`, and `cron` schedules, while the June 17 field report reinforces the same unmet state-aware cadence need. The request is useful, but it needs maintainer product/API design before implementation because it would add a new persisted scheduler condition language and state-evaluation boundary.

## What This Changes

The issue requests deadline-aware cron scheduling where a job can choose a different next heartbeat interval based on runtime state conditions.

## Best Possible Solution

Maintainers should choose a typed, auditable scheduler contract or plugin next-run API for state-aware cadence, then update cron persistence, protocol/schema, CLI/docs, and scheduler tests together.
