## Summary

Keep open. Current main and v2026.5.18 still expose a closed static cron schedule contract, while this request needs a new scheduler/API design for conditional cadence rather than stale cleanup or an already-available plugin-only path.

## What This Changes

The issue requests deadline-aware cron scheduling so a job can use a shorter interval when a configured state condition is true.

## Best Possible Solution

Define a backward-compatible conditional scheduling contract with restricted predicates or an approved scheduler hook, then align cron types, gateway schema, CLI/docs, persistence, and focused scheduler coverage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main and v2026.5.18 shows only static `at`, `every`, and `cron` schedules plus retry/backoff behavior, with no existing conditional cadence contract to reproduce as broken.

## Solution Assessment

No for the literal free-form YAML expression as written. The maintainable path is a maintainer-approved predicate contract or scheduler hook with clear security boundaries, not arbitrary state-expression evaluation in cron config.
