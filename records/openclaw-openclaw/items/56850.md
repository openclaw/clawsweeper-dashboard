## Summary

Keep open. Current main and v2026.5.12 still expose a closed static cron schedule contract, while this request needs a new scheduler/API design for conditional cadence rather than a stale bug cleanup or a ClawHub-only extension.

## What This Changes

Request to let cron jobs specify a shorter conditional interval based on state predicates instead of always using a fixed base cadence.

## Best Possible Solution

Define a backward-compatible conditional scheduling contract with restricted predicates or an approved scheduler hook, then align cron types, protocol schema, store identity, CLI/UI/docs, and focused scheduler tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main and shipped v2026.5.12 shows only static `at`, `every`, and `cron` schedules plus retry/backoff behavior, with no existing conditional cadence contract to reproduce as broken.

## Solution Assessment

No for the literal free-form YAML expression as written. The maintainable path is a maintainer-approved predicate contract or scheduler hook with clear security boundaries, not arbitrary state-expression evaluation in cron config.
