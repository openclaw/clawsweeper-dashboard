## Summary

Keep open: current main still exposes fixed `at`, `every`, and `cron` schedules plus retry/backoff, but no deadline-aware or state-conditioned cadence contract, and the requested arbitrary condition DSL needs maintainer product design before implementation.

## What This Changes

Allow cron job definitions to switch from a base cadence to a shorter conditional cadence when a deadline or state condition is true.

## Best Possible Solution

Have maintainers choose the product boundary first: either add a small typed core conditional schedule API with migration/docs/tests, or document a plugin/hook pattern instead of adding an arbitrary expression DSL to cron.
