## Summary

Keep open. Current main still exposes only static cron schedules and retry/backoff recovery, while this request needs a new core scheduler/API contract plus product and security decisions around scheduler-visible state predicates.

## What This Changes

Request adding deadline-aware conditional cron intervals that shorten the next run cadence when a state predicate is true.

## Best Possible Solution

Define a backward-compatible cron conditional scheduling contract with a restricted predicate or approved scheduler hook, then align schema, store identity, CLI/UI/docs, and focused scheduler tests.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection on current main shows only static `at`, `every`, and `cron` schedules plus retry backoff, with no conditional cadence path.

## Solution Assessment

No for the literal freeform YAML expression as written. The maintainable path is a maintainer-approved predicate contract or scheduler hook, not arbitrary `state...` expression evaluation in cron config.
