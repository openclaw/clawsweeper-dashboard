## Summary

Keep open: current main and the latest release still expose fixed `at`, `every`, and `cron` schedules plus a single heartbeat cadence. The newer field report reinforces the same unmet need, but the condition language, trusted state source, and core/plugin boundary need maintainer product/API design before implementation.

## What This Changes

The issue requests condition-dependent cron/heartbeat intervals so scheduled jobs can run sooner or later based on runtime state such as deadlines, queues, or user inactivity.

## Best Possible Solution

Maintainers should choose a narrow, typed state-aware scheduling contract or plugin hook, then update cron persistence, protocol/tool/CLI/docs, and focused scheduler tests together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source and docs inspection give high confidence that current main lacks conditional cron or heartbeat intervals.

## Solution Assessment

Unclear as proposed. The user need is real, but an arbitrary string condition DSL over scheduler state is not yet proven to be the safest or narrowest contract compared with a typed condition model or plugin-owned next-run API.
