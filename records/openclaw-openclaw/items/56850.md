## Summary

Keep open. Current main still has only static cron scheduling (`at`, fixed `every`, and cron expressions) plus retry backoff, with no public schedule field or scheduler hook for deadline-aware/state-conditional interval selection. The request is valid core scheduling/API design work and should not be closed as stale, implemented, or moved to a plugin without a core predicate contract.

## What This Changes

Request to let cron jobs use a shorter interval when a scheduler-visible state condition is true instead of always waiting for the fixed base interval.

## Best Possible Solution

Define a backward-compatible core cron scheduling contract for conditional intervals, including a restricted declarative predicate or approved scheduler hook, schema/type/store identity changes, next-run recomputation semantics, CLI/UI/docs coverage, and focused cron/protocol tests. The implementation should avoid unrestricted expression evaluation and preserve existing static schedules.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection: current main accepts only `at`, `every`, and `cron` schedules and computes one static `nextRunAtMs` plus retry backoff.

## Solution Assessment

No for the literal freeform YAML shape as written. The desired capability belongs in core cron scheduling, but the maintainable path is a maintainer-approved predicate contract or scheduler hook rather than accepting arbitrary `state...` expression strings.
