## Summary

Keep open: current main and latest release v2026.6.8 still expose only fixed `at`, `every`, and `cron` schedules. Existing plugin hooks can claim or silence a cron-triggered agent run after the scheduler fires, but they do not provide a scheduler-owned conditional interval or deadline-aware next-run calculation.

## What This Changes

Request to let cron job definitions use a shorter conditional interval when state conditions such as deadline proximity are true.

## Best Possible Solution

Have maintainers choose a narrow schedule API or plugin extension that can safely compute deadline-aware `nextRunAtMs`, then update cron persistence, Gateway protocol, agent tool schema, CLI/docs, and focused scheduler tests together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives a high-confidence current-main check that conditional interval schedules are absent.

## Solution Assessment

Unclear as proposed. The user need is real, but an arbitrary string condition DSL is not yet proven to be the safest or narrowest core contract compared with a typed condition model or plugin-owned next-run API.
