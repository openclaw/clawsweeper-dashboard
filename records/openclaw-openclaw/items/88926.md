Keep open for normal PR review: current main still has the streamed tool-argument path corruption, the PR targets the shared parser used by the relevant provider streams, and I found no blocking correctness or security finding.

## What This Changes

The PR updates `src/llm/utils/json-parse.ts` and its tests to preserve Windows path backslashes when streamed tool-call arguments contain control-letter path segments.

## Best Possible Solution

Land a focused parser-level fix with the added regression coverage after required checks pass, keeping the behavior scoped to streamed tool arguments.

## Reproduction Assessment

Yes. Source inspection shows current main first lets `JSON.parse` decode valid JSON control escapes in streamed arguments, and the linked issue gives a deterministic parser-level repro; I did not execute it because this review was read-only.

## Solution Assessment

Yes. A shared `parseStreamingJson` fix is the narrow maintainable layer because all inspected streaming tool-call providers funnel argument deltas through that parser; per-provider repair would duplicate the same invariant.
