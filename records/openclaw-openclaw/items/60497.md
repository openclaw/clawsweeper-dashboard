## Summary

Keep open. Current main still suppresses default-verbosity, non-timeout exec/bash failures before the mutating-tool warning branch runs, and existing tests still assert that a command-not-found exec error produces no payload, leaving assistant success-looking text able to stand alone.

## What This Changes

The issue requests that OpenClaw report failed exec/bash tool calls, such as command-not-found, instead of letting fabricated success-looking assistant text stand alone.

## Best Possible Solution

Update the embedded-run payload policy so failed mutating exec/bash actions produce a default-visible failure payload when the assistant has not explicitly acknowledged the failed action, while keeping raw error details gated by the existing verbosity/suppression policy and preserving special cases such as `sessions_send` and timed-out cron behavior.

## Reproduction Assessment

Yes. A deterministic source-level reproduction exists through `buildEmbeddedRunPayloads` with default verbosity, assistant success text, and `lastToolError` for `exec`/`Command not found`; current tests also assert no payload for the non-timeout command-not-found case.

## Solution Assessment

Yes. A narrow payload-policy fix is the best path because the exec runtime already classifies command-not-found and the subscribe handler retains failed mutating tool state; changing NemoClaw sandbox setup or adding the destructive-command catalog from #60507 would not fix this post-failure reporting path.
