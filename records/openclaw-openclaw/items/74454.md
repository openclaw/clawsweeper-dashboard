## Summary

Keep open. Current main already has the workspace dotenv blocklist portion from related merged work, but the registry helper still prefers `SystemRoot`/`WINDIR`-derived `reg.exe` candidates, and this PR’s latest diff leaves that behavior intact while adding a test that expects the env-derived executable to win.

## What This Changes

The PR currently adds a changelog entry, extends dotenv test coverage, changes `defaultQueryRegistryValue` to call `locateWindowsRegExe()` through its default argument, and adds a `locateWindowsRegExe` test for a readable non-C system root.

## Best Possible Solution

Land a maintainer-approved security fix that keeps the already-merged workspace dotenv blocklist, changes `src/infra/windows-install-roots.ts` so registry probing cannot execute an env-selected `reg.exe`, and updates the tests to assert env-derived registry executables are not chosen.

## Reproduction Assessment

Yes. A unit-level reproduction is available: current main’s `_private.getWindowsRegExeCandidates({ SystemRoot, WINDIR })` returns env-derived paths first, and the PR adds a `locateWindowsRegExe({ SystemRoot: "D:\\Windows" })` test expecting the env-derived executable to win.

## Solution Assessment

No. Calling `locateWindowsRegExe()` without an argument is equivalent to using its default `process.env`, so the narrow maintainable fix is to remove env-derived `reg.exe` selection or bind it to a trusted non-workspace source and update the tests accordingly.
