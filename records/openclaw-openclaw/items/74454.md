## Summary

Keep open. Current main still has the reported Windows dotenv and registry-helper trust-boundary gap, and the latest PR head only completes the dotenv half while preserving env-derived `reg.exe` selection, so it needs maintainer/security correction before merge.

## What This Changes

The PR adds `SYSTEMROOT` and `WINDIR` workspace `.env` blocklist coverage and updates Windows registry-probing tests/call sites, with the latest head retaining env-derived `reg.exe` candidates.

## Best Possible Solution

Land a maintainer-approved security fix that blocks workspace `.env` from setting Windows system-root variables and makes registry probing execute only a trusted, non-workspace-controlled `reg.exe` path, with regression tests asserting env-derived registry executable paths are not selected.

## Reproduction Assessment

Yes. The unit seam is enough to reproduce the remaining patch bug: the latest PR adds a test where `_private.locateWindowsRegExe({ SystemRoot: "D:\\Windows" })` returns the env-derived executable when readable. A live Windows run is not needed to verify that code path.

## Solution Assessment

No. Blocking `SystemRoot` and `WINDIR` in workspace dotenv is necessary, but the latest PR is not the best fix because it preserves the environment-controlled `reg.exe` resolver; the safer path is to remove env-derived candidates for the registry executable while keeping install-root env fallbacks separate if maintainers still want that compatibility.
