## Summary

Keep open: current main already has the workspace `.env` blocklist half, but the registry helper still selects env-derived `reg.exe` candidates and executes the selected path, so the PR still contains relevant security hardening that is not implemented on main.

## What This Changes

The PR pins Windows registry probing to `C:\Windows\System32\reg.exe`, removes env-derived registry-client candidates, updates focused infra tests, and adds a changelog entry.

## Best Possible Solution

Land a maintainer/security-approved, rebased version that prevents registry probing from selecting env-derived `reg.exe` paths while preserving the intended Windows install-root fallback behavior and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Current main no longer lets workspace `.env` set `SystemRoot`/`WINDIR`, but `_private.getWindowsRegExeCandidates({ SystemRoot: "D:\\Windows", WINDIR: "E:\\Windows" })` still returns env-derived candidates and `defaultQueryRegistryValue` executes the selected path.

## Solution Assessment

Yes, with a maintainer compatibility check. The PR uses the narrow registry-client resolver seam and leaves `buildWindowsInstallRoots` env fallback intact, but the `C:\Windows` pin should get security/Windows compatibility approval before landing.
