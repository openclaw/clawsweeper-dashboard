## Summary

Keep open. The PR is still trying to address a real Windows command-resolution issue, but the latest head does not implement the advertised `reg.exe` hardening: current `main` already has the workspace dotenv blocklist from #74456, while both current `main` and this PR still allow env-derived `SystemRoot`/`WINDIR` to select `reg.exe`, and the PR adds a test asserting that unsafe preference.

## What This Changes

The branch adds a changelog and dotenv test coverage around blocked Windows system-root workspace env keys, changes `defaultQueryRegistryValue` to call `locateWindowsRegExe()` without an explicit env argument, and adds a `locateWindowsRegExe` test that prefers a readable non-C `SystemRoot` candidate.

## Best Possible Solution

Land a maintainer-approved security fix that keeps the current workspace dotenv blocklist, changes registry probing so `reg.exe` cannot be selected from workspace-controlled or ambient env roots, and updates `windows-install-roots` tests to assert that readable env-derived registry executables are not chosen.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists: current `main` returns env-derived `SystemRoot`/`WINDIR` candidates before `C:\Windows`, and the PR's added `locateWindowsRegExe({ SystemRoot: "D:\\Windows" })` test demonstrates a readable env-derived binary still wins.

## Solution Assessment

No. The latest patch is not the best way to solve the issue because changing `locateWindowsRegExe(process.env)` to `locateWindowsRegExe()` preserves the same default `process.env` behavior; the safer narrow fix is to remove env-derived `reg.exe` candidates or bind them to a trusted non-workspace source.
