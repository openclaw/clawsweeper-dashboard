## Summary

Keep open. Current main already has the workspace dotenv blocklist portion from merged #74456, but it still resolves `reg.exe` from `SystemRoot`/`WINDIR`, and this PR's latest head preserves that env-derived registry executable selection while claiming to remove it.

## What This Changes

The PR adds `SystemRoot`/`WINDIR` workspace dotenv coverage, changes `defaultQueryRegistryValue()` to call `locateWindowsRegExe()` without an explicit env argument, and adds Windows registry-helper tests that still prefer env-derived `reg.exe` candidates.

## Best Possible Solution

Land a maintainer-approved security fix that keeps the current dotenv blocklist, changes registry probing so `reg.exe` cannot be selected from workspace-controlled environment roots, and updates the Windows install-root tests to assert env-derived registry executables are not chosen.

## Reproduction Assessment

Yes. A unit-level reproduction is available on current main and this PR head: `_private.getWindowsRegExeCandidates` returns env-derived candidates, and the PR head adds a test expecting `locateWindowsRegExe({ SystemRoot: "D:\\Windows" })` to choose that env-derived executable.

## Solution Assessment

No. The current PR head is not the best solution because it changes the call site to the same default `process.env` behavior and adds a regression test for the unsafe candidate preference; the narrower fix is to remove env-derived `reg.exe` selection or bind it to a trusted non-workspace source.
