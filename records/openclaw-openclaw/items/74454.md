## Summary

Keep open. Current main still lets the Windows registry helper choose `reg.exe` from env-derived `SystemRoot`/`WINDIR` before executing it, while the related merged dotenv/taskkill/shell hardening does not cover this registry-probe helper.

## What This Changes

This PR adds a changelog entry, extends dotenv and Windows install-root tests, and changes registry probing so `reg.exe` resolution ignores `SystemRoot`/`WINDIR` and uses `C:\Windows\System32\reg.exe`.

## Best Possible Solution

Land a maintainer/security-approved version that prevents registry probing from selecting env-derived `reg.exe` paths while preserving the intended Windows install-root fallback contract.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists on current main: `_private.getWindowsRegExeCandidates({ SystemRoot: "D:\\Windows", WINDIR: "E:\\Windows" })` returns env-derived candidates, and `defaultQueryRegistryValue` passes `process.env` into that resolver before `execFileSync`.

## Solution Assessment

Yes for the stated registry command-resolution issue. The PR uses a narrow registry-helper change and matching tests; the remaining question is maintainer/security approval of the Windows compatibility tradeoff, not a discrete patch bug.
