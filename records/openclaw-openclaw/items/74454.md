## Summary

Keep open. Current main still builds env-derived `reg.exe` candidates from `SystemRoot`/`WINDIR` and executes the first readable match, while this PR removes that remaining registry-client path and adds focused tests; because it is security-sensitive command-execution hardening, it needs maintainer/security review and CI rather than cleanup closure.

## What This Changes

This PR pins Windows registry probing to `C:\Windows\System32\reg.exe`, removes env-derived registry-client candidates, updates focused infra tests, and adds a changelog entry.

## Best Possible Solution

Land a maintainer/security-approved, rebased version that prevents registry probing from selecting env-derived `reg.exe` paths while preserving the existing install-root fallback contract and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main exposes `_private.getWindowsRegExeCandidates({ SystemRoot: "D:\\Windows", WINDIR: "E:\\Windows" })` as env-derived candidates, and `defaultQueryRegistryValue` executes the selected readable path via `execFileSync`.

## Solution Assessment

Yes. The PR uses the narrowest relevant seam by pinning only the registry-client resolver while leaving `buildWindowsInstallRoots` env fallback behavior intact for install-root discovery.
