## Summary

Keep open. Current main already contains the workspace dotenv blocklist portion through merged #74456, but it still resolves `reg.exe` candidates from `SystemRoot`/`WINDIR`, and this PR's latest head preserves that behavior while adding a test that expects the env-derived path to win.

## What This Changes

The PR adds `SYSTEMROOT`/`WINDIR` workspace dotenv blocking coverage and changes the registry lookup call site to use `locateWindowsRegExe()`'s default, while adding a test that still prefers an env-derived non-C `reg.exe` path.

## Best Possible Solution

Land a maintainer-approved security fix that keeps the current dotenv blocklist, changes registry probing so `reg.exe` cannot be selected from workspace-controlled environment roots, and updates the Windows install-root tests to assert env-derived registry executables are not chosen.

## Reproduction Assessment

Yes. A unit-level reproduction is available: on current main and this PR head, `_private.getWindowsRegExeCandidates` and the added `locateWindowsRegExe({ SystemRoot: "D:\\Windows" })` test demonstrate that an env-derived readable `reg.exe` path remains preferred.

## Solution Assessment

No. The narrow maintainable fix is to remove env-derived `reg.exe` selection or otherwise bind it to a trusted non-workspace source; this patch changes a call to the same default `process.env` behavior and adds a regression test for the unsafe candidate preference.
