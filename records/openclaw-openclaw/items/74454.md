## Summary

Keep open. Current main already blocks `SystemRoot` and `WINDIR` from workspace `.env`, but it still resolves `reg.exe` from env-derived Windows roots before executing registry queries, so this PR's unique registry-helper hardening is not implemented on main. The latest PR head is focused, has no concrete correctness finding from this review, and should proceed through maintainer/security review rather than conservative cleanup closure.

## What This Changes

The PR adds a changelog entry, extends dotenv and Windows install-root tests, and changes Windows registry probing so `locateWindowsRegExe` ignores env-derived roots and only selects `C:\Windows\System32\reg.exe`.

## Best Possible Solution

Land the focused registry-helper hardening after maintainer/security approval of the Windows compatibility tradeoff, keep the regression tests and changelog entry, and coordinate any remaining Windows command-resolution work through the adjacent open hardening PRs rather than duplicating blocklist-only changes.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists on current main: `_private.getWindowsRegExeCandidates({ SystemRoot: "D:\\Windows", WINDIR: "E:\\Windows" })` returns env-derived candidates before `C:\Windows`, and `defaultQueryRegistryValue` passes `process.env` into that resolver before `execFileSync`; the workspace `.env` injection portion is already blocked on current main.

## Solution Assessment

Yes, for the registry-helper issue if maintainers accept pinning `reg.exe` to the default Windows root. A compatibility-preserving alternative would need a trusted system-root source that cannot be influenced by workspace `.env`, but this PR's current diff is the narrowest concrete hardening present here.
