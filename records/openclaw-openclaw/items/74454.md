## Summary

Keep open. Current main already blocks `SystemRoot`/`WINDIR` from workspace dotenv, but it still lets the registry helper prefer env-derived `reg.exe` candidates before executing the selected binary, so the PR is not implemented or superseded.

## What This Changes

This PR pins Windows registry probing to `C:\Windows\System32\reg.exe`, removes env-derived registry-client candidates, updates focused tests, and adds a changelog entry.

## Best Possible Solution

Land a maintainer/security-approved, rebased version that prevents registry probing from selecting env-derived `reg.exe` paths while preserving the intended Windows install-root fallback contract.

## Reproduction Assessment

Yes. Source inspection on current main shows `_private.getWindowsRegExeCandidates({ SystemRoot: "D:\\Windows", WINDIR: "E:\\Windows" })` still returns env-derived candidates, and `defaultQueryRegistryValue` executes the selected path via `execFileSync`.

## Solution Assessment

Yes. The PR uses the narrowest relevant seam by pinning only the registry client resolver while leaving `buildWindowsInstallRoots` env fallback behavior intact for install-root discovery.
