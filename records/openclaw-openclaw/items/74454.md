## Summary

Keep open: current main already blocks the workspace `.env` half, but the registry helper still builds `reg.exe` candidates from `SystemRoot`/`WINDIR` and executes the selected path, so the PR still covers a real remaining hardening gap.

## What This Changes

The PR pins Windows registry probing to `C:\Windows\System32\reg.exe`, updates focused dotenv/install-root tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased, maintainer/security-approved version that prevents registry probing from selecting env-derived `reg.exe` paths while preserving the intended install-root fallback behavior and regression coverage.

## Reproduction Assessment

Yes, source-level. On current main, `_private.getWindowsRegExeCandidates({ SystemRoot: "D:\\Windows", WINDIR: "E:\\Windows" })` still yields env-derived candidates, and `defaultQueryRegistryValue` executes the path selected by `locateWindowsRegExe`.

## Solution Assessment

Yes, with maintainer compatibility review. The PR targets the narrow registry-client resolver seam and leaves `buildWindowsInstallRoots` fallback behavior intact, but the fixed `C:\Windows` policy should be approved before landing.
