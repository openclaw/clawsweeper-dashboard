## Summary

Keep this PR open: current main already has the workspace `.env` blocklist half, but the registry helper still prefers env-derived `SystemRoot`/`WINDIR` `reg.exe` candidates before executing the selected path. The PR addresses the remaining command-execution hardening, but it is still an open security-sensitive change with `mergeable: false`, so it needs maintainer/security review and rebase or conflict resolution rather than cleanup closure.

## What This Changes

The PR pins Windows registry-probe `reg.exe` resolution to `C:\Windows\System32\reg.exe`, adds focused regression coverage, and records the security fix in the changelog.

## Best Possible Solution

Land a maintainer/security-approved, rebased version that prevents registry probing from selecting env-derived `reg.exe` paths while preserving the intended Windows install-root fallback behavior and focused regression coverage.

## Reproduction Assessment

Yes, source-level. Current main still lets `_private.getWindowsRegExeCandidates({ SystemRoot: "D:\\Windows", WINDIR: "E:\\Windows" })` produce env-derived candidates, and `defaultQueryRegistryValue` executes the selected readable candidate.

## Solution Assessment

Yes, with maintainer/security compatibility review. The PR uses the narrow registry-client resolver seam and leaves `buildWindowsInstallRoots` env fallback intact, but the fixed `C:\Windows` registry-client path is a Windows compatibility policy choice.
