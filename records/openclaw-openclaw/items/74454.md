## Summary

Keep open. Current main still builds `reg.exe` candidates from `SystemRoot`/`WINDIR` before executing the selected registry client, so this PR addresses a real remaining Windows command-resolution hardening gap even though the workspace-dotenv blocklist half is already present on main.

## What This Changes

The PR pins Windows registry probing to `C:\Windows\System32\reg.exe`, adds `SystemRoot`/`WINDIR` workspace-dotenv test coverage, updates install-root tests, and adds a changelog entry.

## Best Possible Solution

Land a maintainer/security-approved version that prevents registry probing from selecting env-derived `reg.exe` paths while keeping the Windows install-root fallback contract covered by focused tests.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists on current main: `_private.getWindowsRegExeCandidates({ SystemRoot: "D:\\Windows", WINDIR: "E:\\Windows" })` returns env-derived candidates, and `defaultQueryRegistryValue` passes `process.env` into that resolver before `execFileSync`.

## Solution Assessment

Yes for the stated registry command-resolution issue. The patch is narrow and directly updates the resolver and tests; the remaining question is maintainer/security acceptance of the Windows compatibility tradeoff, not a discrete patch bug.
