## Summary

Keep open: the PR addresses a real current-main cron diagnostics gap and includes real behavior proof, but the patch currently bypasses the documented `diagnostics.enabled` master toggle for isolated-cron event emissions.

## What This Changes

The PR adds diagnostic queued/session-state/processed lifecycle events around isolated cron agent turns, a focused regression test, and an Unreleased changelog entry.

## Best Possible Solution

Land the owner-path cron lifecycle fix after gating it with the diagnostics master toggle and covering both enabled and disabled configurations.

## Reproduction Assessment

Yes, source-reproducible: current main lacks diagnostic lifecycle calls in `runCronIsolatedAgentTurn`, while the established dispatch path emits them. A diagnostic subscriber around an isolated cron turn is the focused reproduction path.

## Solution Assessment

No, not yet: adding lifecycle events in the cron owner path is the right shape, but the implementation should preserve `diagnostics.enabled: false` before merge. The safer fix is to guard the queued/state/processed lifecycle with `isDiagnosticsEnabled(params.cfg)` and add a disabled-config regression case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard cron lifecycle events behind diagnostics.enabled:** `src/cron/isolated-agent/run.ts:1094-1104`
  - body: The new lifecycle calls run unconditionally once `prepared.ok` succeeds. `diagnostics.enabled: false` is the documented master toggle, and the dispatch reference path only calls these helpers when `isDiagnosticsEnabled(cfg)` is true. As written, disabled deployments with a diagnostic subscriber still receive cron `message.queued`, `session.state`, and `message.processed` events carrying the run session identifiers. Please gate this lifecycle on `isDiagnosticsEnabled(params.cfg)` and add a disabled-config regression check.
  - confidence: 0.89
