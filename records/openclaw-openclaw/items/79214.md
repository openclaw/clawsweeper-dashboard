## Summary

Keep open: the missing cron diagnostic lifecycle is a real source-level gap, but the latest PR head still emits the new cron events without the existing config-level diagnostics gate, so it can bypass `diagnostics.enabled: false`.

## What This Changes

The PR adds queued, session-state, and processed diagnostic lifecycle events around isolated cron agent turns, plus a regression test and changelog entry.

## Best Possible Solution

Land the cron lifecycle emission in the cron owner path after gating it with `isDiagnosticsEnabled(params.cfg)` and adding enabled/disabled regression coverage for the new lifecycle.

## Reproduction Assessment

Yes. Source inspection shows current main lacks queued/state/processed emissions in `runCronIsolatedAgentTurn`, while the dispatch path emits the same lifecycle through a diagnostics config gate.

## Solution Assessment

No, not yet. Emitting the lifecycle in the cron path is the right shape, but the implementation must preserve the `diagnostics.enabled: false` contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard cron lifecycle events behind diagnostics.enabled:** `src/cron/isolated-agent/run.ts:1120-1127`
  - body: The new cron lifecycle calls run once preparation succeeds even when `params.cfg.diagnostics.enabled` is false. `diagnostics.enabled` is the documented master toggle, and the dispatch path gates the same queued/state/processed helpers with `isDiagnosticsEnabled(cfg)`, so this should skip all new cron diagnostic events when diagnostics are disabled and cover that with a regression test.
  - confidence: 0.91

## Security Review
