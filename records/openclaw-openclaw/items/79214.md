## Summary

Keep open: the PR addresses a real isolated-cron diagnostics gap and the supplied live proof is convincing, but the current head still emits cron lifecycle diagnostic events without honoring the documented `diagnostics.enabled` master toggle.

## What This Changes

The PR adds queued, session-state, and processed diagnostic events around isolated-cron agent turns, plus a focused regression test and changelog entry.

## Best Possible Solution

Land the cron lifecycle emission after wrapping it in the same `isDiagnosticsEnabled(params.cfg)` gate as dispatch, adding disabled-config regression coverage, and preserving current-main timeout handling during rebase.

## Reproduction Assessment

Yes. Source inspection shows current main's isolated-cron path lacks the queued/state/processed lifecycle calls while the dispatch path emits them behind `isDiagnosticsEnabled`, so a diagnostic subscriber around an isolated cron turn would miss those events.

## Solution Assessment

No, not yet. Emitting from the cron owner path is the right shape, but the implementation must honor `diagnostics.enabled: false` and cover that disabled path before merge.
