## Summary

Keep open: the missing cron diagnostic seam is valid on source inspection, but this PR currently has blocking correctness issues and still needs contributor-supplied real behavior proof.

## What This Changes

The PR adds `logMessageQueued` and `logMessageProcessed` calls around `runCronIsolatedAgentTurn` for isolated cron runs.

## Best Possible Solution

Emit the existing cron diagnostic lifecycle with supported event fields, pair queue-depth changes with the proper session-state cleanup or avoid queue mutation, add regression coverage, and include real subscriber proof.

## Reproduction Assessment

Yes, source-reproducible: current main has no cron diagnostic event emission in `runCronIsolatedAgentTurn`, while dispatch and embedded-runner paths do emit the existing events. A focused diagnostic-event subscriber test around an isolated cron run would exercise the gap directly.

## Solution Assessment

No. Adding events in this path is the right direction, but this diff violates the existing helper type and leaves diagnostic queue state permanently queued unless it also uses the established session-state lifecycle or a non-mutating event path.

- **[P1] Remove the unsupported taskLabel field:** `src/cron/isolated-agent/run.ts:1101`
  - body: The object literal passed to `logMessageQueued` now includes `taskLabel`, but that helper and `DiagnosticMessageQueuedEvent` only accept the existing session/channel/source fields.   - confidence: 0.98
- **[P2] Pair the queued event with session-state cleanup:** `src/cron/isolated-agent/run.ts:1096-1102`
  - body: `logMessageQueued` increments diagnostic `queueDepth`, and the only decrement path is `logSessionStateChange(... state: "idle")`; `logMessageProcessed` does not clear it. This cron path adds queued/processed without any state transition, so every isolated cron run would leave a permanent queued diagnostic entry and can trigger stale queued or stuck-session warnings.
  - confidence: 0.91
