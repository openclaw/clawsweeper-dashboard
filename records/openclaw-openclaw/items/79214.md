## Summary

Keep open: the cron diagnostic lifecycle fix is useful and source-reproducible, but the current PR head still emits the new events without respecting the `diagnostics.enabled` config gate used by the reference dispatch path.

## What This Changes

This PR adds queued/session-state/processed diagnostic lifecycle events around isolated cron agent turns, plus a focused regression test and changelog entry.

## Best Possible Solution

Land the cron lifecycle events after wrapping them in the same `isDiagnosticsEnabled(params.cfg)` gate as dispatch and adding enabled/disabled regression coverage.

## Reproduction Assessment

Yes. Current main has no diagnostic lifecycle calls in `runCronIsolatedAgentTurn`, while the established dispatch path emits them; a diagnostic subscriber around an isolated cron turn is the focused reproduction path.

## Solution Assessment

No, not yet. Adding lifecycle events in the cron owner path is the right shape, but the implementation should preserve the `diagnostics.enabled: false` contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard cron lifecycle events behind diagnostics.enabled:** `src/cron/isolated-agent/run.ts:1096-1106`
  - body: The new lifecycle calls run unconditionally once preparation succeeds. `diagnostics.enabled: false` is the documented master toggle, and the dispatch path gates these helpers with `isDiagnosticsEnabled(cfg)`. Please skip the queued/state/processed lifecycle when diagnostics are disabled and add a disabled-config regression test.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a narrow privacy/observability concern by emitting session identifiers without the config-level diagnostics gate.

Concerns:

- **[medium] Diagnostics master toggle bypass:** `src/cron/isolated-agent/run.ts:1096`
