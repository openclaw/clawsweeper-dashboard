## Label Justifications

- `P1`: A real user reports Docker sandbox sessions failing immediately after upgrade, which breaks an agent execution workflow and involves a security-sensitive sandbox default.
- `impact:security`: The issue centers on Docker sandbox hardening with `no-new-privileges` and any fix may change the sandbox security boundary.
- `impact:crash-loop`: The sandbox container exits immediately with code 255, leaving session commands unable to run.

## Summary

Keep open: current main and the latest release still create Docker sandbox containers with `--security-opt no-new-privileges`, there is no supported config or fallback path to disable it, and the only linked fix PR is closed unmerged. The report remains actionable, but the fix needs maintainer security/product judgment because the obvious opt-out weakens sandbox hardening.

## What This Changes

The issue requests a way for Docker sandbox sessions to survive hosts where `--security-opt no-new-privileges` makes `/usr/bin/sleep` fail, via an opt-out, fallback retry, or clearer diagnostics.

## Best Possible Solution

Keep the secure default, then add a maintainer-approved audited path for incompatible Docker hosts, such as an explicit dangerous opt-out with warnings, an image-level fix, or diagnostics that explain the host failure without silently weakening isolation.

## Reproduction Assessment

Yes. The reporter provided a focused Docker matrix showing `--security-opt no-new-privileges` is the trigger on the affected host, and current source still always passes that flag during sandbox container creation.

## Solution Assessment

Unclear. A simple config opt-out is plausible, but it weakens sandbox hardening; the best fix needs maintainer choice between an image/runtime compatibility fix, an explicit dangerous override, fallback behavior, or diagnostic-only handling.
