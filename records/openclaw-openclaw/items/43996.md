## Summary

Keep open: current main and the latest release still unconditionally pass Docker no-new-privileges for sandbox containers, while the only linked fix PR is closed unmerged and the remaining fix needs a sandbox-security decision.

## What This Changes

The issue reports Docker sandbox startup failure on hosts where no-new-privileges makes the sandbox container exit with exec /usr/bin/sleep: operation not permitted.

## Best Possible Solution

Keep the secure default, then add a maintainer-approved dangerous opt-out or equivalent diagnostic path with audit/status/docs coverage rather than silently weakening sandbox hardening.

## Reproduction Assessment

Yes, at source/report level: current main always emits the reported Docker security option, tests assert it, and the issue includes a host-level Docker matrix where only that flag fails. I did not rerun the exact failing Ubuntu/Docker host, so this is source-reproducible rather than live reproduced here.

## Solution Assessment

Unclear until maintainers choose the sandbox-security tradeoff. The best narrow direction appears to be an explicit dangerous opt-out with secure default and visible diagnostics, not a silent fallback that weakens isolation without operator intent.
