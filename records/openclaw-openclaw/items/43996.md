## Summary

Keep this open: current `main` still unconditionally adds Docker `no-new-privileges`, and an open PR already uses closing syntax to implement the requested opt-out, so this issue should not be closed before that PR is resolved.

## What This Changes

The issue asks for a safe way to handle Docker hosts where sandbox startup fails because `--security-opt no-new-privileges` makes `/usr/bin/sleep` return EPERM.

## Best Possible Solution

Review and land a narrow fix like #70229 that keeps `no-new-privileges` enabled by default while offering an explicit, documented, warning-surfaced escape hatch for incompatible hosts.

## Reproduction Assessment

Yes. Current `main` always includes `--security-opt no-new-privileges` and starts the sandbox image with `sleep infinity`, while the reporter supplied a focused Docker command that reproduces EPERM on the affected host.

## Solution Assessment

Yes, the linked PR direction is a narrow maintainable fix: keep the hardening default, add an explicit opt-out, and document/report the weaker setting. The issue should stay open until that implementation is reviewed and merged or intentionally rejected.
