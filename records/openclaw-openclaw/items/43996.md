## Summary

Keep open: current main and the latest release still unconditionally add Docker `--security-opt no-new-privileges`, the strict sandbox Docker config has no opt-out, and the only directly linked fix PR closed unmerged. The report remains actionable, but the recovery path changes sandbox hardening posture and needs maintainer security/product approval before implementation.

## What This Changes

The issue asks OpenClaw to recover from or clearly diagnose Docker sandbox startup failures when `--security-opt no-new-privileges` makes the sandbox container exit with EPERM.

## Best Possible Solution

Keep `no-new-privileges` as the secure default, then add a maintainer-approved break-glass opt-out and clear diagnostics, or explicitly decide that affected Docker hosts are unsupported without local patching.

## Reproduction Assessment

Yes at source/report level: current main still unconditionally passes Docker `no-new-privileges`, and the reporter supplied a host-level Docker matrix where that flag alone makes `/usr/bin/sleep` fail. I did not reproduce the exact affected host locally.

## Solution Assessment

Unclear until maintainers choose the security posture: an explicit secure-default opt-out plus diagnostics looks like the narrowest maintainable path, while a silent fallback would need explicit acceptance because it weakens sandbox hardening.
