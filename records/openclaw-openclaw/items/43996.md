## Summary

Keep open: current main and the latest release still always create Docker sandbox containers with `--security-opt no-new-privileges`, the reporter supplied a focused host-level repro showing that flag alone triggers the EPERM, and the only linked fix PR was closed unmerged. The remaining repair is security-sensitive because it either weakens sandbox hardening, adds a dangerous config opt-out, or changes startup/fallback diagnostics.

## What This Changes

The issue asks OpenClaw to recover or clearly diagnose Docker sandbox startup failures when `--security-opt no-new-privileges` prevents the sandbox `sleep infinity` process from execing.

## Best Possible Solution

Preserve the secure default, then add a maintainer-approved explicit dangerous opt-out or similarly visible diagnostic path with tests and docs instead of silently weakening sandbox creation.

## Reproduction Assessment

Yes at source/report level, but I did not rerun the failing Docker host locally. Current main always emits `--security-opt no-new-privileges`, and the reporter's minimal Docker matrix isolates that flag as the EPERM trigger.

## Solution Assessment

Unclear until maintainers choose the security posture. The narrowest maintainable path appears to be a secure default plus an explicit dangerous opt-out and clear diagnostics, while silent fallback is riskier because it can hide weakened sandboxing.
