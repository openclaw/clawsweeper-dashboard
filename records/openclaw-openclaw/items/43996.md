## Summary

Keep open: current main and v2026.5.20 still unconditionally add Docker no-new-privileges before starting the sandbox sleep process, and the linked fix PR remains open. The report is well-supported, but the fix path changes sandbox hardening and needs maintainer security/product review.

## What This Changes

The issue asks for a safe recovery path when Docker-backed sandbox sessions fail on hosts where --security-opt no-new-privileges makes /usr/bin/sleep exit with EPERM.

## Best Possible Solution

Land the linked PR or a refreshed replacement that keeps no-new-privileges enabled by default, adds a clearly dangerous opt-out with diagnostics/docs/tests, and includes redacted proof from an affected Docker host.

## Reproduction Assessment

Yes for source-level review: the reporter supplied a focused Docker matrix where only --security-opt no-new-privileges fails, and current main still applies that flag before sleep infinity. I did not rerun the host-specific Docker/runc failure locally.

## Solution Assessment

Yes in direction, but it is not landed. A secure-default explicit opt-out with dangerous-config reporting is the narrowest maintainable path; a silent fallback that weakens containment would need stronger security/product approval.
