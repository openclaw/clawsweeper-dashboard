Keep open: current main and the latest release still unconditionally pass Docker `--security-opt no-new-privileges` to sandbox containers, the reporter provided a focused host-level EPERM reproduction, and https://github.com/openclaw/openclaw/pull/70229 remains open with closing syntax for this issue.

## What This Changes

The issue asks for sandbox Docker sessions to recover from hosts where Docker `no-new-privileges` prevents `/usr/bin/sleep` from starting, via an opt-out, fallback, or clearer diagnostics.

## Best Possible Solution

Land a refreshed secure-default fix, likely via https://github.com/openclaw/openclaw/pull/70229 or an equivalent replacement, that omits `no-new-privileges` only through an explicit break-glass config path and reports the downgrade in dangerous-config diagnostics.

Yes for source-level review. The issue provides a focused Docker command that fails only with `--security-opt no-new-privileges`, and current main unconditionally applies that flag before running `sleep infinity`; I did not rerun the host-specific Docker/runc failure locally.

## Solution Assessment

Yes in direction, but not landed. A secure-default explicit opt-out with dangerous-config reporting is the narrowest maintainable fix; a silent fallback that weakens sandbox hardening would need stronger security/product review.
