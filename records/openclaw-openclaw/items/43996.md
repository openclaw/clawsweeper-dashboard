## Label Justifications

- `P1`: Affected users cannot start Docker-backed maker sandbox sessions after the hardening flag is applied.
- `impact:security`: The compatibility fix would change Docker sandbox hardening around `no-new-privileges`.
- `impact:crash-loop`: The reported sandbox container exits immediately before session commands can run.

## Summary

Keep open: current main and the latest release still unconditionally pass Docker `--security-opt no-new-privileges` when creating sandbox containers, while the implementation PR that would add a secure-default opt-out is still open and unmerged.

## What This Changes

The issue asks for sandbox startup compatibility when Docker `--security-opt no-new-privileges` makes the sandbox container fail to exec `/usr/bin/sleep` on an affected host.

## Best Possible Solution

Land the linked PR or a refreshed replacement that keeps `no-new-privileges` enabled by default, adds a deliberate dangerous opt-out with docs/tests/diagnostics, and includes redacted proof from an affected Docker host.

## Reproduction Assessment

Yes for source-level review: the reporter supplied a focused Docker matrix where only `--security-opt no-new-privileges` fails, and current main unconditionally applies that flag before `sleep infinity`. I did not rerun the host-specific Docker/runc failure locally.

## Solution Assessment

Yes in direction, but it is not landed. A secure-default explicit opt-out with dangerous-config reporting is the narrowest maintainable fix; a silent fallback that weakens containment would need stronger security/product approval.
