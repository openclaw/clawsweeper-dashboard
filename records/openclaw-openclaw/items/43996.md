## Summary

Keep open. Current main and the latest release still apply Docker `no-new-privileges` unconditionally, there is no config/schema opt-out or startup fallback, and open PR #70229 is the linked implementation candidate rather than a merged fix.

## What This Changes

The issue asks for Docker sandboxes to keep working on hosts where `--security-opt no-new-privileges` makes even `/usr/bin/sleep` fail with EPERM.

## Best Possible Solution

Review, land, or replace PR #70229 with a secure default-on Docker sandbox setting wired through config types, strict schema, config resolution, Docker create args, dangerous-config reporting, docs, and regression tests.

## Reproduction Assessment

Yes. The reporter supplied a minimal Docker reproduction for the host failure, and current main still sends the exact `--security-opt no-new-privileges` flag through the Docker sandbox create path.

## Solution Assessment

Yes. An explicit default-on break-glass configuration knob with audit/docs coverage is narrower and safer than silently retrying without a sandbox hardening flag.
