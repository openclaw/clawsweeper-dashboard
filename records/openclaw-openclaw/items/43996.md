## Summary

Keep open: current main and the latest release still unconditionally pass Docker `--security-opt no-new-privileges` to sandbox containers, while the reporter provided a focused host-level EPERM reproduction and the linked fix PR remains open/unmerged.

## What This Changes

Sandbox startup should recover or provide an explicit escape hatch when Docker hosts reject `--security-opt no-new-privileges` and make `/usr/bin/sleep` fail with EPERM.

## Best Possible Solution

Land a refreshed secure-default fix, likely via https://github.com/openclaw/openclaw/pull/70229 or an equivalent replacement, that omits `no-new-privileges` only through an explicit break-glass config path and reports that downgrade in dangerous-config diagnostics.

## Reproduction Assessment

Yes. The issue provides a focused Docker command that fails only with `--security-opt no-new-privileges`, and current main unconditionally applies that flag before running `sleep infinity`; I did not rerun the host-specific Docker failure in this read-only pass.

## Solution Assessment

Yes in direction, but not landed. A default-on explicit opt-out with dangerous-config reporting is the narrowest maintainable fix; a silent fallback that weakens sandbox hardening would need stronger security/product review.
