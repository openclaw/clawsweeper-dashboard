## Summary

Keep open: current main and latest v2026.5.7 still always pass Docker `no-new-privileges` for sandbox containers, while https://github.com/openclaw/openclaw/pull/70229 is an open, unmerged fix path that explicitly targets this issue.

## What This Changes

The issue asks for a compatibility path when Docker sandbox startup fails because `--security-opt no-new-privileges` makes even `/usr/bin/sleep` return EPERM on the reporter's host.

## Best Possible Solution

Review, refresh, and either land or deliberately replace the linked default-on compatibility fix with equivalent fallback/diagnostic behavior that preserves secure defaults and reports any break-glass downgrade.

## Reproduction Assessment

Yes. The issue gives a minimal Docker command that fails only when `--security-opt no-new-privileges` is present on the affected host, and current main/latest release unconditionally emit that flag for sandbox creation; I did not run the host-specific Docker matrix locally.

## Solution Assessment

Yes in direction, but not complete in current main. A default-on opt-out with dangerous-config diagnostics or an automatic fallback would address the compatibility failure while preserving the secure default, and https://github.com/openclaw/openclaw/pull/70229 is the current review path.
