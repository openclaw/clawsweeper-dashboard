## Summary

Keep open: current main and the latest release still unconditionally add Docker `--security-opt no-new-privileges`, the strict sandbox Docker config schema has no opt-out, and the only directly linked fix PR closed unmerged. The issue is actionable, but the repair changes sandbox hardening behavior and needs maintainer security/product approval before implementation.

## What This Changes

The issue reports that maker sandbox Docker containers fail to start on some hosts because OpenClaw always applies Docker no-new-privileges, and asks for an opt-out, fallback, or clearer diagnostic.

## Best Possible Solution

Keep no-new-privileges as the secure default, then add a maintainer-approved break-glass opt-out or diagnostic path that makes affected Docker hosts recoverable without silently weakening sandboxing.

## Reproduction Assessment

Yes at source/report level: current main still unconditionally adds Docker no-new-privileges, and the reporter supplied a Docker matrix where only that flag makes `/usr/bin/sleep` fail. I did not run Docker on the affected host, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes for an explicit secure-default opt-out or diagnostic path; no for a silent fallback unless maintainers explicitly accept that security tradeoff. The current repository does not yet contain any of those fixes.
