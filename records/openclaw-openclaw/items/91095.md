## Summary

Keep open: the report points at a real current-main ambiguity in interactive TUI tool exposure. Source inspection supports the local embedded path omitting `sessions_spawn`, while the non-local gateway half needs maintainer/live repro confirmation because current gateway code does not set embedded mode.

## What This Changes

The issue requests that interactive TUI sessions expose `sessions_spawn` when subagent target allowlists and effective tool policy allow it.

## Best Possible Solution

Define the intended TUI boundary, then make gateway-connected TUI expose and execute `sessions_spawn` when policy allows; for local embedded TUI, either implement a supported gateway-bound spawn bridge with regression coverage or document that subagent spawning requires a gateway session.

## Reproduction Assessment

Yes for the local embedded path from source: embedded mode excludes `sessions_spawn` unless `allowGatewaySubagentBinding` is true, and local TUI reaches the generic agent ingress path without that option. I did not establish a live Windows/npm reproduction, and current source does not prove the same issue for non-local gateway TUI.

## Solution Assessment

Unclear: flipping the flag everywhere may be too broad because `sessions_spawn` calls the Gateway. The maintainable fix should separate gateway-connected TUI behavior from local embedded behavior and add focused coverage for the effective tool list or `context.compiled`.
