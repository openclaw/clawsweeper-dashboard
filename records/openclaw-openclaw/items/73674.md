## Summary

Keep open: current main still has the Windows QMD `.cmd` shim gap, and this PR is aimed at the right surface, but the latest diff is not merge-ready because the scoped QMD fallback can resolve a POSIX shell wrapper as a Node entrypoint and the required changelog entry is missing.

## What This Changes

The PR adds quoted absolute Windows `.cmd` shim parsing for QMD, switches memory runtime and host probes to the `@tobilu/qmd` package fallback, copies the resolver into the memory-host SDK package, and updates focused tests.

## Best Possible Solution

Land the absolute-entrypoint parser after the scoped fallback safely unwraps or rejects the real `@tobilu/qmd` package shape, keeps SDK and memory-host behavior aligned, and records the user-facing Windows memory fix in the changelog.

## Reproduction Assessment

Yes. Source inspection on current main shows absolute quoted QMD `.cmd` shims without `%~dp0` still reach unresolved-wrapper handling, and the proposed fallback regression is reproducible from the `@tobilu/qmd@2.1.0` package metadata and tarball shape.

## Solution Assessment

No, not as currently proposed. The quoted absolute shim parsing is the right narrow direction, but the scoped package fallback must match the real upstream package wrapper instead of treating `bin/qmd` as JavaScript.
