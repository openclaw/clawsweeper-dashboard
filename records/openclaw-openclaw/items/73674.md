## Summary

Keep open: current main still has the reported QMD absolute `.cmd` shim gap, but this PR should not merge as-is because the scoped fallback resolves the published POSIX `bin/qmd` wrapper as a Node entrypoint and the user-facing fix lacks a changelog entry.

## What This Changes

The PR adds Windows absolute `.cmd` shim entrypoint parsing, copies the resolver into the memory-host SDK path, switches QMD fallback lookup to `@tobilu/qmd`, and updates focused tests.

## Best Possible Solution

Land a repaired Windows-only resolver that unwraps absolute QMD shim targets and handles the published `@tobilu/qmd` package wrapper safely, with aligned tests and changelog coverage.

## Reproduction Assessment

Yes, by source and dependency inspection. Current main only resolves `%~dp0`-relative quoted `.cmd` targets, and the published `@tobilu/qmd@2.1.0` package shape makes the PR's scoped-fallback regression clear without a live Windows run.

## Solution Assessment

No, not as currently proposed. The absolute `.cmd` shim parsing is the right narrow direction, but the QMD package fallback must unwrap or reject the real POSIX wrapper before handing an entrypoint to Node, and the changelog entry is required.
