## Summary

Keep open: the PR addresses a real current-main Windows QMD `.cmd` shim gap, but it is not merge-ready because the scoped QMD fallback now resolves the real package's shell wrapper as a Node entrypoint and the required user-facing changelog entry is still missing.

## What This Changes

The PR updates Windows QMD `.cmd` shim resolution to accept quoted absolute entrypoints, switches QMD package fallback lookup to `@tobilu/qmd`, and updates related tests in plugin SDK and memory host code.

## Best Possible Solution

Land the absolute-entrypoint parser, but make the `@tobilu/qmd` fallback unwrap the real package shell bin to `dist/cli/qmd.js` or fail closed, keep plugin SDK and memory-host behavior aligned, and add the changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows absolute quoted QMD `.cmd` shims without `%~dp0` still reach the unresolved-wrapper path; the PR regression is also source-reproducible from the current `@tobilu/qmd@2.1.0` package metadata and tarball.

## Solution Assessment

No, not as currently proposed. The quoted absolute shim parsing is the right narrow direction, but the scoped package fallback must match the upstream QMD package shape instead of treating its shell wrapper as JavaScript.
