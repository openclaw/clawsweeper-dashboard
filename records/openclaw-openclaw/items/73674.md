## Summary

Keep open: the QMD Windows resolver fix appears valid, but the current PR head still includes a blocking security/correctness issue in unrelated OpenAI-compatible image URL fetching and lacks inspectable real behavior proof from the contributor's Windows setup.

## What This Changes

The PR updates Windows QMD `.cmd` shim and scoped package fallback handling, copies Windows spawn resolution into memory-host SDK, adds tests, and adds async OpenAI-compatible image URL response parsing/fetching.

## Best Possible Solution

Keep the narrow QMD no-shell absolute shim and scoped fallback fix, split or guard the image URL support through provider HTTP policy, then land only after redacted Windows runtime proof is added.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main skips quoted absolute `.cmd` entrypoints without `%~dp0` and uses `packageName: "qmd"` while the documented package is `@tobilu/qmd`; I did not run live Windows validation in this read-only review.

## Solution Assessment

No as-is. The QMD resolver change is the narrow maintainable direction, but the unrelated image URL path must be split or routed through guarded provider HTTP before merge.

## Review Findings

Overall correctness: patch is incorrect
