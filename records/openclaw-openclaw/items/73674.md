## Summary

Keep open: current main still has the Windows absolute `.cmd` QMD shim gap, and this PR targets the right surface, but the latest diff is not merge-ready because the scoped package fallback resolves the real `@tobilu/qmd` shell wrapper as a Node entrypoint and the required changelog entry is missing.

## What This Changes

The PR updates plugin SDK and memory-host Windows `.cmd` shim resolution for quoted absolute entrypoints, copies the resolver into memory-host SDK, switches QMD fallback package lookup to `@tobilu/qmd`, and adjusts resolver/QMD tests.

## Best Possible Solution

Land the absolute-entrypoint parser, but make the `@tobilu/qmd` fallback unwrap the real package shell bin to `dist/cli/qmd.js` or fail closed, keep plugin SDK and memory-host behavior aligned, and add the changelog entry.

## Reproduction Assessment

Yes. The reported bug is statically reproducible on current main with a Windows `qmd.cmd` containing `node "E:\\...\\dist\\cli\\qmd.js" %*`, which lacks `%~dp0` and reaches the unresolved-wrapper error path. The PR regression is reproducible from the current `@tobilu/qmd@2.1.0` package metadata, whose `qmd` bin points to a shell wrapper rather than the JavaScript entrypoint.

## Solution Assessment

No, not as currently proposed. The quoted absolute shim parsing is the right narrow direction, but the scoped fallback must match the upstream QMD package shape and tests should use that real metadata before merge.
