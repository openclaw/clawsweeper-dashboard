## Summary

Keep open. Current main still lacks the reported Windows QMD `.cmd` absolute-entrypoint handling and still uses the stale `qmd` package fallback, while the PR is a narrow fix for that behavior. The patch looks functionally sound, but it still needs the required user-facing changelog entry plus final CI/maintainer validation before merge.

## What This Changes

The PR extends Windows `.cmd` shim parsing for QMD to accept quoted absolute JS/EXE entrypoints, switches QMD fallback resolution to `@tobilu/qmd`, adds focused tests, and replaces the memory-host Windows spawn re-export with a local implementation.

## Best Possible Solution

Land the narrow Windows QMD fix after adding the required changelog entry and getting green targeted validation, while preserving fail-closed wrapper policy and keeping the central plugin SDK and memory-host resolver behavior aligned.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: use a Windows `qmd.cmd` wrapper containing `node "E:\\...\\@tobilu\\qmd\\dist\\cli\\qmd.js" %*`; the parser has no `%~dp0` match and the QMD paths still request package fallback `qmd`, so resolution fails closed.

## Solution Assessment

Mostly yes. Directly parsing existing quoted absolute wrapper targets and using the scoped package name is the narrow maintainable fix; the safer merge shape is to add the changelog and either accept or guard the memory-host copy so it does not drift from the shared resolver.
