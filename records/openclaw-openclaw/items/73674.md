## Summary

Keep open. The PR targets a real current-main Windows QMD gap and preserves the no-shell fallback direction from the earlier Windows wrapper hardening work, but the latest head is not merge-ready because the scoped QMD package fallback does not match the current `@tobilu/qmd` package shape and the user-facing fix still needs a changelog entry.

## What This Changes

The PR updates Windows QMD `.cmd` shim parsing in the plugin SDK and memory-host SDK, switches QMD fallback package lookup to `@tobilu/qmd`, and adds unit coverage for quoted absolute JavaScript entrypoints.

## Best Possible Solution

Land a narrow Windows QMD resolver fix that accepts quoted absolute JS/EXE wrapper targets without shell execution, keeps unresolved wrappers fail-closed, and makes the scoped `@tobilu/qmd` fallback match the real package contract or decline unsafe shell-wrapper bin targets. Add regression coverage using the real `bin/qmd` shape and include the required single-line changelog entry before merge.

## Reproduction Assessment

Yes. Current main can be reproduced statically with a Windows `.cmd` shim containing `node "E:\\...\\@tobilu\\qmd\\dist\\cli\\qmd.js" %*`, because the parser only handles `%~dp0`-relative quoted tokens; the PR-specific package fallback bug is reproduced by the current `@tobilu/qmd` metadata where `bin.qmd` points to `bin/qmd`.

## Solution Assessment

No, not as currently proposed. The quoted-absolute-entrypoint parsing is the right narrow direction, but changing fallback lookup to `@tobilu/qmd` without handling its shell-wrapper `bin/qmd` target can produce a false available result and later run the wrong file through Node.
