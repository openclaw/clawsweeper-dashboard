## Summary

Keep this PR open: the Windows QMD `.cmd` shim bug is source-reproducible and the proposed core resolver change is useful, but the branch now forks the memory-host SDK Windows resolver away from current main and still lacks contributor-supplied real Windows behavior proof.

## What This Changes

The PR updates QMD Windows `.cmd` shim resolution to accept quoted absolute JS/EXE entrypoints and changes QMD package fallback resolution from `qmd` to `@tobilu/qmd`, with focused tests.

## Best Possible Solution

Keep the shared Windows spawn resolver as the single implementation, add quoted absolute shim support there, update QMD fallback to `@tobilu/qmd`, and merge only after redacted Windows proof shows QMD-backed memory search working.

## Reproduction Assessment

Yes, at source level: current main ignores quoted absolute `.cmd` shim entrypoints and still uses the unscoped QMD package fallback, which matches the reported failure path. I did not run a native Windows QMD install during this read-only review.

## Solution Assessment

No as currently shaped against main: the shared resolver and QMD package-name changes are the right narrow fix, but the memory-host SDK bridge should be preserved instead of replaced with a copied resolver.
