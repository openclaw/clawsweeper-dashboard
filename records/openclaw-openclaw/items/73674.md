## Summary

This is a plausible focused bug fix for native Windows QMD `.cmd` shims, but it should stay open because the external PR still lacks inspectable after-fix real behavior proof. I did not find a definite blocking code bug in the diff.

## What This Changes

This PR extends Windows QMD `.cmd` shim resolution for quoted absolute JS/EXE entrypoints, updates the QMD fallback package name to `@tobilu/qmd`, and adds unit coverage for the plugin SDK path.

## Best Possible Solution

Keep the no-shell resolver approach, land the scoped QMD fallback and absolute-entrypoint support after inspectable Windows proof is added, and prefer a shared resolver shape unless there is a package-boundary reason to fork it.

## Reproduction Assessment

Yes from source inspection: a `.cmd` containing `node "C:\...\qmd.js" %*` has no `%~dp0` relative token, so current main skips it and then fails closed. I did not run a live Windows reproduction in this read-only review.

## Solution Assessment

Mostly yes: extending the existing no-shell Windows resolver and correcting the package fallback is the narrow fix. The main improvement I would consider is keeping the memory-host SDK on the shared resolver instead of copying the implementation.
