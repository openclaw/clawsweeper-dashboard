## Summary

Keep open: this is a plausible focused bug fix for native Windows QMD `.cmd` shims, and current `main` still has the failing source path. The remaining blocker is not a code finding; it is missing inspectable real Windows behavior proof from the contributor's setup before merge.

## What This Changes

The branch extends Windows `.cmd` wrapper resolution for QMD to quoted absolute JS/EXE entrypoints, changes QMD package fallback to `@tobilu/qmd`, and adds unit coverage while replacing the memory-host SDK resolver re-export with a local copy.

## Best Possible Solution

Land the narrow no-shell resolver and scoped QMD package fallback once redacted, inspectable Windows proof is added, preferably keeping the memory-host SDK aligned with the shared resolver unless maintainers want a package-local copy.

## Reproduction Assessment

Yes, from source inspection: a `.cmd` containing `node "C:\...\qmd.js" %*` has no `%~dp0` token, so current `main` skips the quoted absolute entrypoint and then fails closed. I did not run a live Windows reproduction in this read-only review.

## Solution Assessment

Mostly yes: extending the existing no-shell Windows resolver and correcting the QMD package fallback is the narrow maintainable fix. The safer refinement is to avoid duplicating the resolver in the memory-host SDK unless the package boundary requires it.
