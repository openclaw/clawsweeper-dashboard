## Summary

Keep open: current main still has the reported QMD Windows shim failure path, and the PR is a plausible narrow fix, but it needs inspectable Windows behavior proof before merge and should avoid reintroducing a copied memory-host resolver.

## What This Changes

The PR teaches Windows QMD `.cmd` shim resolution to accept quoted absolute JS/EXE entrypoints, switches QMD fallback lookup to `@tobilu/qmd`, and adds focused unit coverage.

## Best Possible Solution

Land a minimal fix that updates the canonical Windows spawn resolver and QMD package fallback, keeps the memory-host SDK sharing that resolver, and includes inspectable Windows runtime proof.

## Reproduction Assessment

Yes, from source inspection: current main skips quoted absolute shim targets and still falls back to package name `qmd`, while the dependency contract is `@tobilu/qmd`. I did not run a live Windows reproduction in this read-only review.

## Solution Assessment

Mostly yes: resolving existing quoted absolute `.cmd` entrypoints without shell execution and using the scoped QMD package is the narrow fix. The safer implementation detail is to keep the memory-host SDK on the shared resolver instead of merging a second copied implementation.
