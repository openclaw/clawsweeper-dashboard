## Summary

Keep open: the PR is a focused fix for a real Windows QMD shim gap, but the latest head still lacks inspectable Windows behavior proof and introduces a QMD package-fallback path that can execute the upstream POSIX wrapper with Node.

## What This Changes

The PR parses quoted absolute QMD `.cmd` entrypoints on Windows, switches QMD fallback lookup to `@tobilu/qmd` in both memory call paths, copies the resolver into `memory-host-sdk`, and adds focused tests.

## Best Possible Solution

Land a rebased Windows-only fix that parses quoted absolute `.cmd` entrypoints, safely unwraps or rejects the real `@tobilu/qmd` package wrapper in both call paths, adds upstream-shaped tests and changelog, and includes redacted Windows proof.

## Reproduction Assessment

Yes, by source inspection. Current main only unwraps quoted `%~dp0`-relative `.cmd` shim targets, so a `qmd.cmd` containing `node "E:\...\qmd.js" %*` reaches unresolved-wrapper handling without shell fallback.

## Solution Assessment

No, not as currently proposed. The absolute shim parser is the right narrow direction, but the scoped fallback must handle the real `@tobilu/qmd` `bin/qmd` wrapper safely and the PR still needs changelog plus real Windows proof.
