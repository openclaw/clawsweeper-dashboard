Keep open. Current main still has the reported Windows QMD `.cmd` shim gap, and the PR targets the right behavior without enabling shell fallback by default. ## What This Changes

The PR updates Windows `.cmd` shim resolution for QMD to accept quoted absolute JS/EXE entrypoints, switches QMD package fallback lookup to `@tobilu/qmd`, adds focused resolver coverage, and mirrors the resolver change into the memory-host SDK path.

## Best Possible Solution

Land the narrow Windows QMD fix after adding the required changelog entry, resolving the lint/check failure, and validating the current-main merge result with targeted Windows-spawn/QMD tests plus the changed gate. The implementation should keep unresolved wrappers fail-closed by default and keep QMD package lookup aligned with the documented `@tobilu/qmd` install path.

## Reproduction Assessment

Yes. A static reproduction on current main is a Windows `qmd.cmd` containing `node "E:\\...\\@tobilu\\qmd\\...\\qmd.js" %*`; the current parser ignores that quoted absolute token, finds no package fallback under unscoped `qmd`, and throws the unresolved-wrapper error.

## Solution Assessment

Yes, with follow-up. Resolving an existing quoted absolute entrypoint without shell execution is the narrow maintainable fix; the safer merge shape is to keep the central and memory-host resolver implementations aligned and prove the final branch with the focused tests.
