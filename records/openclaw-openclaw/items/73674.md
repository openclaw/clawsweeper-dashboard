## Summary

Keep open: the reported Windows QMD shim bug is real and the PR targets the right resolver surface, but the latest diff is not merge-ready because the scoped QMD fallback resolves the real package's POSIX shell wrapper as a Node entrypoint and the required changelog entry is still missing.

## What This Changes

The PR adds Windows `.cmd` absolute-entrypoint parsing to the plugin SDK and memory-host spawn helpers, changes QMD fallback package lookup to `@tobilu/qmd`, and updates related tests.

## Best Possible Solution

Land the absolute-entrypoint parser, but make the `@tobilu/qmd` fallback unwrap the real `bin/qmd` wrapper to `dist/cli/qmd.js` or fail closed, keep plugin SDK and memory-host behavior aligned, and add the changelog entry.

## Reproduction Assessment

Yes. Current main reaches the unresolved-wrapper error for a Windows `qmd.cmd` containing `node "E:\\...\\dist\\cli\\qmd.js" %*`, and the PR regression follows from the current `@tobilu/qmd@2.1.0` package metadata and tarball contents.

## Solution Assessment

No, not as currently proposed. The quoted absolute shim parsing is the right narrow fix, but the scoped package fallback must match the real QMD package shape before merge.
