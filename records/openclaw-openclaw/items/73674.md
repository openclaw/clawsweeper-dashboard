## Summary

Keep open: the QMD Windows shim bug is source-reproducible and the QMD portion of the patch is directionally right, but current head lacks inspectable real behavior proof and adds an unrelated unguarded provider URL fetch path.

## What This Changes

The PR updates Windows QMD command resolution and scoped fallback while also adding OpenAI-compatible image URL response fetching/parsing.

## Best Possible Solution

Land the narrow QMD Windows no-shell shim and @tobilu/qmd fallback after proof, with any image URL response feature split out or routed through guarded provider HTTP.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main skips quoted absolute .cmd entrypoints without %~dp0 and uses packageName "qmd" while the documented/registry package is @tobilu/qmd. I did not run a live Windows reproduction in this read-only review.

## Solution Assessment

No as-is: the QMD resolver and scoped package changes are the narrow maintainable fix, but the image URL response support should be split or guarded and the external PR still needs inspectable Windows proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Route image URL downloads through guarded provider HTTP:** `src/image-generation/image-assets.ts:186`
  - body: This new response path downloads provider-returned URLs with raw `fetchFn(url)`, bypassing the existing provider HTTP SSRF/private-network/dispatcher/timeout controls. A malicious or compromised OpenAI-compatible endpoint could make OpenClaw fetch internal URLs or hang outside the guard.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces provider-supplied URL downloads that bypass existing guarded provider HTTP controls.

Concerns:

- **[medium] Unguarded provider URL fetch:** `src/image-generation/image-assets.ts:186`
  - body: `generatedImageAssetFromOpenAiCompatibleUrlEntry` calls `fetchFn(url)` directly for a URL supplied by the provider response instead of using the existing guarded provider HTTP helpers, so SSRF/private-network policy, dispatcher/proxy policy, and timeouts are not enforced on that follow-up request.
  - confidence: 0.86
