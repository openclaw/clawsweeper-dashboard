## Summary

Keep open: current main still has the Windows QMD shim gap and this PR targets the right resolver path, but it is not merge-ready because the scoped package fallback can resolve `@tobilu/qmd`'s POSIX shell bin as a Node entrypoint and the required changelog entry is missing.

## What This Changes

The PR adds quoted absolute Windows `.cmd` entrypoint parsing, switches QMD fallback lookup to `@tobilu/qmd`, updates memory-host resolver code, and adds/updates focused tests.

## Best Possible Solution

Land the absolute-entrypoint parser, make the `@tobilu/qmd` fallback unwrap or reject shell-wrapper bins safely, keep plugin SDK and memory-host behavior aligned, and add the user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows absolute quoted QMD `.cmd` shims without `%~dp0` still reach unresolved-wrapper handling, and both QMD callers still use the old `qmd` package fallback.

## Solution Assessment

No, not as currently proposed. The quoted absolute shim parsing is the right narrow direction, but the scoped package fallback must match the real upstream package shape instead of treating its shell wrapper as JavaScript.

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Resolve the real QMD package entrypoint:** `extensions/memory-core/src/memory/qmd-manager.ts:1856`
  - body: Changing the fallback to `@tobilu/qmd` makes the resolver read the real package metadata, where `qmd` points to `bin/qmd`.
