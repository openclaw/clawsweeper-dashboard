## Summary

Keep open. The PR targets a real Windows QMD wrapper gap and the quoted absolute entrypoint parser is the right direction, but the latest head still makes the scoped QMD package fallback resolve the real package shell wrapper as a Node entrypoint and is missing the required user-facing changelog entry.

## What This Changes

The PR adds quoted absolute entrypoint parsing to Windows `.cmd` resolution, copies the resolver into the memory-host SDK, switches QMD fallback lookup to `@tobilu/qmd`, and updates resolver/QMD tests.

## Best Possible Solution

Land a narrow Windows QMD resolver fix that accepts quoted absolute `.js`/`.exe` shim targets without shell execution, but make the package fallback either unwrap the real `@tobilu/qmd/bin/qmd` wrapper to `dist/cli/qmd.js` safely or fail closed instead of running shell wrappers under Node. Add fixtures based on the upstream package shape and a single-line changelog entry before merge.

## Reproduction Assessment

Yes. The current-main bug can be reproduced statically with a Windows `.cmd` wrapper containing `node "E:\\...\\@tobilu\\qmd\\dist\\cli\\qmd.js" %*`, which has no `%~dp0` token; the PR blocker is reproducible from the current `@tobilu/qmd` npm metadata and tarball.

## Solution Assessment

No, not as currently proposed. The absolute-entrypoint parser is the right narrow fix, but the scoped package fallback must be corrected against the actual QMD package contract and covered by a realistic fixture before merge.
