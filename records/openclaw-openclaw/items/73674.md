## Summary

Keep open. The PR targets a real current-main QMD Windows shim gap, and the final head appears to address the earlier Greptile non-Windows test concern by accepting POSIX absolute temp paths in the test resolver. It is not merge-ready because the new scoped package fallback and fixture do not match the current `@tobilu/qmd` package bin contract, and the user-facing fix still needs the required changelog entry.

## What This Changes

The PR updates Windows `.cmd` shim parsing for quoted absolute QMD entrypoints, changes QMD fallback package lookup to `@tobilu/qmd`, adds resolver coverage, and replaces the memory-host SDK resolver re-export with a local copy.

## Best Possible Solution

Land a narrow Windows QMD resolver fix that accepts quoted absolute JS/EXE targets without shell execution, but first make the scoped package fallback match the real `@tobilu/qmd` package shape or keep that fallback fail-closed when the bin target is a shell wrapper. Keep the plugin SDK and memory-host behavior aligned with tests that use the real package metadata shape.

## Reproduction Assessment

Yes. For the reported bug, current main can be reproduced statically with a Windows `.cmd` wrapper containing `node "E:\\...\\@tobilu\\qmd\\dist\\cli\\qmd.js" %*`, which lacks a `%~dp0` token and reaches the unresolved-wrapper error path. For the PR finding, the current `@tobilu/qmd` package metadata resolves fallback to `bin/qmd`, which is a shell wrapper rather than the JS entrypoint the resolver will run with Node.

## Solution Assessment

No, not as currently proposed. The absolute-entrypoint parsing is the right narrow direction, but the package fallback should be corrected against the upstream QMD package contract and covered with a fixture that matches the real `bin/qmd` wrapper before merge.
