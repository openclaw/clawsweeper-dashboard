## What This Changes

The branch passes active reply-model metadata into image-tool resolution and lets the image tool prefer a vision-capable active model when `agents.defaults.imageModel` is unset.

## Best Possible Solution

Land this fallback after redacted real-provider proof shows an unset image model using the active vision-capable reply model while existing explicit image-model and override precedence remain intact.

## Reproduction Assessment

Yes at source level: current main carries the active model and vision bit into tool construction but does not pass the active provider/model into image-model resolution, so an unset image model can fall through to other candidates or fail. I did not run a live provider repro in this read-only review.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: carry the already-known active model into image-tool resolution and use it only when it supports images. The remaining blocker is real provider proof, not a different implementation direction.
