## Summary

Keep this PR open: it fixes the missing-MediaTypes path and now has useful terminal proof, but the generic-MIME case described in the PR can still hydrate an image with `application/octet-stream` instead of an image MIME.

## What This Changes

The branch updates current-turn media hydration to classify image attachments by normalized attachment kind, derive a media type from MIME or file extension, and add regression coverage for `MediaPaths` without `MediaTypes`.

## Best Possible Solution

Land the current-turn hydration fix after missing and generic MIME inputs both resolve to a concrete image MIME by extension or sniffed buffer, without widening prompt image scanning again.

## Reproduction Assessment

Yes at source level: current main skips `MediaPaths` images when `MediaTypes` are absent or non-image, and the PR body shows a direct after-fix terminal run for the missing-type path. I did not execute a live Feishu message in this read-only review.

## Solution Assessment

No, not yet: using the shared attachment classifier is the right direction, but the media type resolver should not let generic declared MIME override a concrete image extension or sniffed MIME.
