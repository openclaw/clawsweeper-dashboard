## Summary

Keep open: current main and the latest release still drop runtime settings overrides after save/reload, while this PR moves the override layer into SettingsManager and adds focused regression coverage. I found no blocking patch findings, and the supplied after-fix terminal proof covers the settings and compaction path.

## What This Changes

The PR stores SettingsManager.applyOverrides() as non-persisted runtime overrides, reapplies them after save/reload, and adds compaction regression tests.

## Best Possible Solution

Land the narrow SettingsManager runtime-overrides fix after normal merge gates so overrides remain non-persisted but highest precedence for the manager lifetime.

## Reproduction Assessment

Yes, from source inspection rather than executing it locally: current main applies reserve overrides through applyOverrides(), then setCompactionEnabled() or setRetryEnabled() calls save(), which rebuilds effective settings from global/project settings and drops those overrides.

## Solution Assessment

Yes. Keeping the runtime override layer inside SettingsManager is the cleanest owner-boundary fix because save(), project writes, and reload all share the same effective-settings recompute point.
