## Summary

Keep open for maintainer review: current main still drops SettingsManager runtime overrides after save/reload paths, and this PR is a focused fix with regression coverage and sufficient terminal proof. I found no blocking correctness or security findings.

## What This Changes

This PR makes SettingsManager retain non-persisted runtime overrides across save/reload and adds regression tests for embedded compaction reserve settings.

## Best Possible Solution

Land the narrow SettingsManager runtime override layer after ordinary maintainer review and merge gates, keeping overrides non-persisted but highest precedence for that manager lifetime.

## Reproduction Assessment

Yes. From source inspection, current main applies compaction reserve overrides through applyOverrides(), then setRetryEnabled() or setCompactionEnabled() calls save(), which recomputes from global/project settings and drops those overrides.

## Solution Assessment

Yes. Keeping runtime overrides inside SettingsManager is the best owner-boundary fix because save(), project writes, and reload all share the same effective-settings recompute point.
