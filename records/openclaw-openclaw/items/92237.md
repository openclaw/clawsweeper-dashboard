## Summary

Keep open. Current main and the latest release still drop `applyOverrides()` during `save()`/`reload()` recomputation, while this PR is focused, sufficiently proven, and has no blocking review findings.

## What This Changes

The PR changes `SettingsManager` so non-persisted runtime overrides are retained across saves and reloads, with regression tests for embedded compaction reserve settings.

## Best Possible Solution

Land the narrow `SettingsManager` runtime-override layer after ordinary maintainer review and merge gates, keeping overrides non-persisted and covered by the added regression tests.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.9` apply a compaction override and then can drop it when retry or compaction setters recompute effective settings without runtime overrides. I did not execute tests because this review was read-only.

## Solution Assessment

Yes. Keeping runtime overrides inside `SettingsManager` is the best owner-boundary fix because save, project writes, and reload all share that effective-settings recomputation point; reapplying overrides at individual callers would miss sibling setter paths.
