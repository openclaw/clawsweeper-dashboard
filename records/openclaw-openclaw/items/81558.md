## Summary

Keep open: current main still has the shared context-engine post-turn lifecycle for embedded/Codex harnesses, but the CLI runner path has no contextEngine parameter, no finalizer call, and no turn maintenance scheduling. The linked fix PR is open and uses closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Route successful claude-cli/CLI-backed turns through context-engine afterTurn or ingest fallback and turn maintenance so LCM/transcript persistence advances like embedded runs.

## Best Possible Solution

Land a focused CLI lifecycle fix that reuses the shared context-engine finalizer after successful CLI turns, preserves single ownership of transcript writes, and disposes resolved engine resources on every prepare/run exit path.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI runner returns successful turns without a contextEngine field, finalizer call, or maintenance scheduling, while embedded and Codex paths call the shared finalizer; the issue body also provides a live A/B showing LCM frozen on claude-cli and advancing on embedded Codex.

## Solution Assessment

Yes in direction. The narrow maintainable path is to review and land https://github.com/openclaw/openclaw/pull/81869 or an equivalent focused fix after it addresses cleanup and provides real claude-cli plus non-legacy context-engine proof.
