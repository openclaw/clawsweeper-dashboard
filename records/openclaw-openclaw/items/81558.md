## Summary

Keep open: current main still has the shared context-engine turn lifecycle for embedded/Codex paths, but the CLI runner returns successful turns without a context-engine parameter, finalizer call, or turn-maintenance scheduling; the open closing PR should be reviewed and landed or closed before this issue is resolved.

## What This Changes

The issue requests that successful claude-cli routed turns run the same context-engine after-turn ingestion and maintenance lifecycle as embedded/Codex harness turns.

## Best Possible Solution

Land a reviewed fix that wires successful CLI turns through the shared context-engine finalizer and turn-maintenance scheduling while keeping transcript ownership single-owned and avoiding Telegram-specific behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a non-legacy context engine, route a successful turn through runCliAgent, and current main has no way for that CLI run to invoke afterTurn, ingestBatch, or turn maintenance.

## Solution Assessment
