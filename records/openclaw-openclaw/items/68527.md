## Summary

Keep open. Current main only partially mitigates MCP process accumulation with runtime reuse, idle cleanup, process-tree teardown, and child OOM bias; it still lacks per-MCP failed-start backoff/circuit state, `openclaw mcp set` preflight, MCP child/RSS status accounting, and generated systemd memory/task/OOM ceilings.

## What This Changes

The issue requests MCP failed-start backoff/circuit breaking, MCP config preflight, process/RSS status diagnostics, and default Linux service guardrails to prevent misconfigured MCP servers from exhausting a VM.

## Best Possible Solution

Keep this as the canonical hardening issue and split the work into runtime cooldown/circuit state keyed by MCP config, explicit MCP preflight, focused status diagnostics, and a reviewed Linux service guardrail policy with docs.

## Reproduction Assessment

Yes for the missing guardrail class: configure a fail-fast stdio or HTTP MCP server and trigger new bundled-MCP session runtime catalog builds; current source shows each new runtime can attempt startup without persisted per-server cooldown. The exact VM incident was not reproduced live.

## Solution Assessment

Partly. Backoff, circuit breaking, cleanup, and preflight are the right maintainable direction, but systemd/swap/watchdog defaults should be split and reviewed as operator policy rather than bundled into one automatic change.
