## Summary

Keep open: current main and v2026.6.10 contain meaningful MCP hardening, but the central resource-exhaustion request is still only partially implemented, especially failed-start backoff/circuiting, raw `openclaw mcp set` probe semantics, and default Linux service resource caps.

## What This Changes

Request gateway MCP failed-start backoff, per-server circuit breakers, save-time validation, process accounting, and default Linux service guardrails to prevent MCP retry storms from exhausting a VM.

## Best Possible Solution

Land a maintainer-approved hardening plan that separates shared failed-start backoff/circuiting, explicit raw `mcp set` probe semantics, diagnostics/process accounting, and Linux service resource-cap defaults with upgrade/operator proof.

## Reproduction Assessment

Yes for the remaining source-level gaps: current main shows no shared failed-start backoff/circuit before future MCP startup attempts, raw `openclaw mcp set` saves without probing, and generated systemd units lack hard resource caps. I did not live-reproduce the full VM exhaustion incident.

## Solution Assessment

No, this should not be treated as a one-shot automated fix. The right solution needs maintainer and security choices for retry policy, probe behavior, and operator-facing Linux service defaults.
