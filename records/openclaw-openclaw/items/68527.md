## Summary

Keep open. Current main has useful cleanup mitigations for bundled-MCP child lifetimes, but it still lacks shared per-server failed-start backoff/circuit state, `openclaw mcp set` preflight/status accounting, and default Linux service memory/task guardrails; the related open PR is observe-first and does not close the resource-exhaustion tracker.

## What This Changes

The issue requests MCP retry-storm hardening through failed-start backoff, per-server circuit breaking, CLI preflight/status diagnostics, default systemd resource limits, swap/watchdog guidance, and MCP runbooks.

## Best Possible Solution

Keep this as the canonical resource-exhaustion hardening tracker and split the implementation into shared per-server failed-start cooldown/circuit state, explicit MCP preflight/status diagnostics, and maintainer-approved Linux service guardrail defaults.

## Reproduction Assessment

Yes, at source level: configure a fail-fast stdio or HTTP MCP server and trigger separate bundled-MCP session runtime catalog builds; current main disposes the failed local session without shared per-server cooldown before later sessions can retry. I did not replay the full VM exhaustion incident.

## Solution Assessment

Unclear on the exact implementation shape. The requested hardening direction is valid, but the best fix needs maintainer decisions for circuit reset semantics, CLI/status UX, and Linux service resource-limit policy.
