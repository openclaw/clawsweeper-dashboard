## Summary

Keep open. Current main still lacks shared per-server MCP failed-start backoff/circuit state, `openclaw mcp set` preflight validation, MCP child-process status accounting, and default systemd memory/task guardrails, while related open PRs only cover narrower slices.

## What This Changes

The issue requests MCP failed-start backoff/circuit breaking, MCP config preflight/status diagnostics, and default Linux service resource guardrails to prevent VM-level resource exhaustion.

## Best Possible Solution

Keep this as the canonical resource-exhaustion hardening tracker and split the fix into runtime failed-start cooldown/circuit state, explicit MCP preflight/status diagnostics, and maintainer-approved Linux service guardrail policy.

## Reproduction Assessment

Yes, at source level: configure a fail-fast stdio or HTTP MCP server and trigger separate bundled-MCP session runtime catalog builds; current main disposes the failed session without shared per-server cooldown before later sessions can retry. I did not replay the full VM exhaustion incident.

## Solution Assessment

Yes for keeping this open as the canonical tracker. The requested directions are valid, but default systemd limits, circuit reset semantics, and status/preflight UX need maintainer policy review rather than a single automatic repair.
