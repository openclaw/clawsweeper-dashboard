## Summary

Keep open. Current main has partial MCP cleanup mitigations, but it still lacks shared per-server failed-start backoff/circuit state, `openclaw mcp set` preflight, MCP child/RSS status accounting, and generated systemd memory/task guardrails.

## What This Changes

The issue asks for MCP failed-start backoff/circuit breaking, CLI preflight, process/status accounting, and default Linux service guardrails to prevent MCP retry storms from exhausting host resources.

## Best Possible Solution

Keep this as the canonical hardening issue, then split it into runtime failed-start cooldown/circuit state, explicit MCP preflight, focused status diagnostics, and reviewed Linux service guardrail policy with docs.

## Reproduction Assessment

Yes, at source level: configure a fail-fast stdio or HTTP MCP server and trigger separate bundled-MCP session runtime catalog builds; current main disposes the failed server without any shared per-server cooldown before later runtimes can retry. I did not live-replay the dev01 VM exhaustion incident.

## Solution Assessment

Partly. Backoff, circuit breaking, preflight, status diagnostics, and cleanup are the right direction, but service limits, swap/watchdog defaults, and reset semantics should be decomposed and reviewed as maintainer policy.
