## Summary

Keep open. Current main still only partially mitigates the MCP retry-storm class: runtime reuse, idle cleanup, stdio process-tree teardown, and child OOM bias exist, but there is still no per-MCP failed-start cooldown/circuit state, no `openclaw mcp set` preflight, no MCP-specific child/RSS status accounting, and no generated systemd memory/task/OOM ceilings.

## What This Changes

The issue asks OpenClaw to harden MCP startup failures with backoff, circuit breaking, preflight validation, process diagnostics, systemd guardrails, swap/watchdog guidance, and docs.

## Best Possible Solution

Keep this as the canonical resource-exhaustion hardening issue and split implementation into MCP runtime cooldown/circuit state, MCP config preflight, MCP-specific diagnostics, and reviewed Linux service guardrail policy/docs.

## Reproduction Assessment

Yes for the missing guardrail class: configure a fail-fast stdio or HTTP MCP server and trigger new bundled-MCP session runtime catalog builds. Current source shows later runtimes can attempt startup again without persisted per-server cooldown; the original VM incident was not live-reproduced.

## Solution Assessment

Partly. Backoff, circuit breaking, cleanup, preflight, and diagnostics are the right maintainable direction, but systemd/swap/watchdog defaults should be reviewed and likely split as operator policy rather than merged as one automatic change.
