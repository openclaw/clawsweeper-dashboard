## Summary

Keep open. Current main has useful partial mitigations for MCP process cleanup and runtime reuse, but it still does not implement the broader hardening requested here: failed MCP starts have no cross-session backoff/circuit state, `openclaw mcp set` remains config-only, status output does not account for MCP child count/RSS, and generated systemd units still lack memory/task/OOM ceilings.

## What This Changes

The issue requests MCP failed-start backoff/circuit breaking, MCP config preflight, process/RSS status diagnostics, and default Linux service guardrails to prevent resource exhaustion from misconfigured MCP servers.

## Best Possible Solution

Keep this as the canonical hardening issue and split the implementation: add per-MCP failed-start state keyed by normalized config fingerprint, clear it on config change or operator action, add an explicit preflight path for saved MCP definitions, expose focused MCP child/RSS diagnostics, and make a maintainer decision on Linux service cgroup/swap/watchdog defaults with docs.

## Reproduction Assessment

Yes for the missing guardrail class: configure a fail-fast stdio or HTTP MCP server and trigger new bundled-MCP session runtime catalog builds; current source shows each new runtime can attempt startup without persisted per-server cooldown. The exact VM exhaustion incident was not reproduced live.

## Solution Assessment

Partly. The requested backoff/circuit/preflight direction is the right maintainable path, but the systemd/swap/watchdog defaults should be split and reviewed as operator policy before changing generated service behavior.
