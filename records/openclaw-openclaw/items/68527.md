## Summary

Keep open. Current main has useful partial mitigations for MCP process accumulation, but it still lacks per-server failed-start backoff/circuit state, `openclaw mcp set` preflight, MCP child/RSS status accounting, and generated Linux service memory/task/OOM guardrails.

## What This Changes

Request MCP retry-storm hardening with backoff, circuit breaking, config preflight, status accounting, systemd resource guardrails, swap/watchdog guidance, and docs.

## Best Possible Solution

Keep this as the canonical hardening issue and split it into maintainer-reviewed work for MCP runtime cooldown/circuit state, MCP config preflight, status diagnostics, and Linux service guardrail policy/docs.

## Reproduction Assessment

Yes for the missing guardrail class: configure a fail-fast stdio or HTTP MCP server and trigger new bundled-MCP session runtime catalog builds. The source path shows later runtimes can attempt startup again without persisted per-server cooldown; the original VM incident was not live-reproduced.

## Solution Assessment

Partly. Backoff, circuit breaking, cleanup, preflight, and diagnostics are the right maintainable direction, but systemd/swap/watchdog defaults should be reviewed and likely split as operator policy rather than merged as one automatic change.
