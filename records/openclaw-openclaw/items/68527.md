## Summary

Keep open. Current main has session-scoped MCP runtime reuse, child cleanup, and Linux child OOM-score mitigation, but it still lacks shared per-server failed-start backoff/circuit state, `openclaw mcp set` preflight, MCP child/RSS status accounting, and generated systemd memory/task guardrails.

## What This Changes

The issue requests MCP failed-start backoff/circuit breaking, preflight validation, status diagnostics, and default Linux service guardrails to prevent resource exhaustion.

## Best Possible Solution

Keep this as the canonical hardening issue, then split implementation into runtime per-server failed-start cooldown/circuit state, explicit MCP preflight UX, focused status diagnostics, and reviewed Linux service guardrail policy with docs.

## Reproduction Assessment

Yes, at source level: configure a fail-fast stdio or HTTP MCP server and trigger separate bundled-MCP session runtime catalog builds; current main disposes the failed server without any shared per-server cooldown before later sessions can retry. I did not live-replay the full dev01 VM exhaustion incident.

## Solution Assessment

Partly. Backoff, circuit breaking, preflight, status diagnostics, and cleanup are the right direction, but service limits, swap/watchdog defaults, and reset semantics should be decomposed and reviewed as maintainer policy rather than handled as one automatic bug fix.
