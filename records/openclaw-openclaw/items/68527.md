## Summary

Keep open. Current main has useful cleanup and Linux child OOM-score mitigations, but the remaining MCP failed-start backoff/circuit, `openclaw mcp set` preflight, MCP child/RSS status accounting, and default Linux service guardrail policy are not implemented on main.

## What This Changes

Request backoff/circuit breakers, MCP config preflight, process/RSS diagnostics, and default Linux service guardrails so misconfigured MCP servers cannot exhaust VM resources.

## Best Possible Solution

Keep this as the canonical resource-exhaustion hardening tracker and split the fix into reviewed runtime failed-start cooldown/circuit state, explicit MCP preflight/status diagnostics, and maintainer-approved Linux service guardrail policy.

## Reproduction Assessment

Yes, at source level: configure a fail-fast stdio or HTTP MCP server and trigger separate bundled-MCP session runtime catalog builds; current main disposes the failed session without a shared per-server cooldown before later sessions can retry. I did not replay the full VM exhaustion incident.

## Solution Assessment

Partly. Backoff, circuit breaking, preflight, diagnostics, and cleanup are the right direction, but service limits, swap/watchdog defaults, and circuit reset semantics need maintainer policy review rather than a single automatic bug-fix PR.
