## Summary

Keep open. Current main has important partial mitigations for MCP child cleanup and Linux child OOM scoring, but it still lacks shared per-server failed-start backoff/circuit state, `openclaw mcp set` preflight, MCP child/RSS status accounting, and generated systemd memory/task ceilings.

## What This Changes

The issue requests MCP retry-storm hardening across runtime backoff/circuit breaking, config preflight, diagnostics, and default Linux service guardrails.

## Best Possible Solution

Keep this as the canonical hardening issue and decompose it into reviewed runtime failed-start cooldown/circuit state, explicit MCP preflight UX, focused status diagnostics, and Linux service guardrail policy with docs.

## Reproduction Assessment

Yes, at source level: configure a fail-fast stdio or HTTP MCP server and trigger separate bundled-MCP session runtime catalog builds; current main disposes the failed server without a shared per-server cooldown before later sessions can retry. I did not live-replay the full dev01 VM exhaustion incident.

## Solution Assessment

Partly. Backoff, circuit breaking, preflight, diagnostics, and cleanup are the right direction, but service limits, swap/watchdog defaults, and reset semantics should be decomposed and reviewed as maintainer policy rather than handled as one automatic bug fix.
