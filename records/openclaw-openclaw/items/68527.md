## Summary

Keep open. Current main has useful partial mitigations for the related bundled-MCP process leak, but the central resource-exhaustion request is still unresolved: failed MCP starts are not circuit-broken across new session runtimes, `openclaw mcp set` still does not preflight server reachability, status does not expose MCP child/RSS accounting, and generated systemd units do not include the requested memory/task/OOM guardrails. The closed #68660 companion covers duplicate/orphaned runtime cleanup, not this broader hardening plan.

## What This Changes

The issue requests MCP failed-start backoff and per-server circuit breaking, `openclaw mcp set` preflight validation, MCP child/RSS diagnostics, Linux systemd resource guardrails, swap/watchdog defaults, and runbook documentation.

## Best Possible Solution

Split the hardening into maintainable pieces: add per-server failed-start state keyed by MCP config fingerprint with exponential backoff and reset-on-config-change behavior, add an explicit saved-MCP preflight path, expose MCP child/RSS health in status, and separately decide Linux service cgroup/swap/watchdog defaults with docs.

## Reproduction Assessment

Yes for the source-level bug class: configure a fail-fast stdio MCP server under `mcp.servers` and trigger bundled MCP catalog creation from new session runtimes; current code attempts startup and disposes the failed session without persisted cooldown or circuit state. The original VM incident was not replayed live.

## Solution Assessment

No as a single automated cleanup item. The requested direction is the right class of fix, but the maintainable solution should be split between runtime retry semantics, CLI/API preflight, status diagnostics, and Linux service policy.
