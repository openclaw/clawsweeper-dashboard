## Summary

Keep open. Current main has partial blast-radius work for MCP child cleanup and Linux child OOM scoring, but it still lacks shared per-server MCP failed-start backoff/circuit state, `openclaw mcp set` preflight validation, MCP child/RSS status accounting, and default systemd memory/task ceilings; the related open PRs cover narrower observe-only or opt-in runtime-sharing slices.

## What This Changes

Request gateway-side MCP failed-start backoff/circuit/preflight/status accounting plus deployment-level resource guardrails for the generated Linux service.

## Best Possible Solution

Keep this as the canonical resource-exhaustion hardening tracker and split the fix into runtime failed-start cooldown/circuit state, explicit MCP preflight/status diagnostics, and maintainer-approved Linux service guardrail policy.

## Reproduction Assessment

Yes, at source level: configure a fail-fast stdio or HTTP MCP server and trigger separate bundled-MCP session runtime catalog builds; current main disposes the failed session without shared per-server cooldown before later sessions can retry. I did not replay the full VM exhaustion incident.

## Solution Assessment

Yes for keeping this open as the canonical tracker. The requested directions are valid, but default systemd limits, circuit reset semantics, preflight UX, and status diagnostics need maintainer policy review rather than a single automatic repair.
