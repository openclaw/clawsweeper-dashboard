## Summary

Keep this PR open for normal review, but do not merge it as-is: the one-line formula change does not fix the reported long-running cron repro on current main because the 60s pre-execution cap still dominates, and current execution milestones already clear the pre-execution watchdog once CLI process or tool execution starts.

## What This Changes

The PR changes `resolveCronAgentPreExecutionWatchdogMs()` to use the full job timeout instead of `Math.floor(jobTimeoutMs / 2)`.

## Best Possible Solution

Preserve the fast pre-execution stall detector; if a real rclone/rsync abort remains, localize the last recorded phase and either add the missing execution milestone or consolidate the policy/config change with regression coverage and PR-head real proof.

## Reproduction Assessment

No high-confidence live reproduction of the PR's silent-tool claim is present. Source inspection shows the formula/cap behavior exactly, but current CLI process and embedded tool milestones should clear the pre-execution watchdog before a long silent command runs.

## Solution Assessment

No. Removing `/2` alone does not address the current 60s cap or the existing execution-milestone boundary, so the safer fix is phase-specific evidence plus a targeted milestone/config change if a real failure remains.
