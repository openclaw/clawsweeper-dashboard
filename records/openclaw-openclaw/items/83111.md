## Summary

Keep open: the bug report is plausible, but this PR is not ready because the current 60s cap still dominates the reported 600s/1800s repros, and the best fix likely needs to coordinate the watchdog cap/config or execution-phase behavior with regression coverage.

## What This Changes

The PR changes `resolveCronAgentPreExecutionWatchdogMs()` to use the full job timeout instead of `Math.floor(jobTimeoutMs / 2)` when computing the pre-execution watchdog ceiling.

## Best Possible Solution

Land a focused cron fix that preserves fast pre-execution stall detection, clears the watchdog once CLI or embedded execution actually starts, and includes regression coverage for silent long-running commands.

## Reproduction Assessment

Yes, by source inspection: an isolated CLI-backed cron run can call `onExecutionStarted()` and then emit no execution-stage phase, leaving `createCronAgentWatchdog()` in `waiting_for_execution` until the pre-execution timer fires. I did not run a live cron job in this read-only review.

## Solution Assessment

No. Removing `/2` may be one component, but current main still caps the reported 600s job at 60s, and the maintainable fix likely needs regression coverage plus either a CLI execution milestone or the linked cap/config change.
