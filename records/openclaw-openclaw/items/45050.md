## Summary

Current main still has a source-reproducible Feishu/provider-error leak path: the shared raw error formatter converts the JSON wrapper to `LLM error server_error: <provider message>`, but the provider message can still contain help-center text and request IDs, and the Feishu sender forwards that text as normal post content. The related implementation PR remains open and still needs repair, so this issue should stay open.

## What This Changes

The issue requests channel-safe Feishu handling for Codex/provider `server_error` failures, with raw payloads and request IDs kept out of user-visible replies and pre-output transient retry confirmed.

## Best Possible Solution

Fix the shared assistant error formatter/sanitizer path so structured and plain-text provider internal errors produce generic chat-safe copy without raw request IDs, while preserving the existing pre-output transient retry behavior.

## Reproduction Assessment

Yes, by source inspection. A reported Codex `server_error` payload flows through the shared formatter into `LLM error server_error: <provider message>`, then through the final reply payload and Feishu send path without a channel-safe fallback.

## Solution Assessment

Yes in direction, but the narrow maintainable fix is shared formatter/sanitizer coverage rather than a Feishu-only filter. The related PR should be repaired or replaced so provider request IDs stay in logs/hashed diagnostics, not user copy.
