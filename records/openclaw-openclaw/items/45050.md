## Summary

Keep open: current main still has a source-visible path that can turn the reported provider-shaped `server_error` into user-facing Feishu text, and the related open fix PR is useful but not yet mergeable as reviewed.

## What This Changes

Feishu channel replies should sanitize generic provider `server_error` payloads and only retry transient provider failures before visible output is delivered.

## Best Possible Solution

Land a narrow rebased fix, or a replacement for https://github.com/openclaw/openclaw/pull/49401, that rewrites generic provider internal errors in the shared formatter/sanitizer path, omits raw request IDs from chat copy, preserves the existing pre-output retry boundary, and adds focused regression tests.

## Reproduction Assessment

Yes, by source inspection rather than live Feishu execution. A stopped assistant error with the reported Codex `server_error` shape flows through `buildEmbeddedRunPayloads` to `formatAssistantErrorText`, then Feishu sends the resulting text without a channel-safe fallback.

## Solution Assessment

No, current main is not the best solution because it still formats provider-shaped error text for users. The related PR has the right narrow direction, but it should be rebased into the shared formatter/sanitizer path and must not display raw provider request IDs.
