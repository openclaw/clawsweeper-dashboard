## Summary

Current main still has a source-reproducible Feishu/provider-error leak path: structured Codex `server_error` payloads are converted into `LLM error server_error: <provider message>`, and that provider message can still include help-center text and request IDs before Feishu sends it as normal post text. A related open PR targets the same area but still needs repair, so this issue should remain open.

## What This Changes

The issue requests Feishu-safe handling for provider `server_error` failures so raw provider payloads and request IDs are not sent to users, plus a pre-output transient retry when safe.

## Best Possible Solution

Fix the shared assistant error formatter/sanitizer path so structured and plain-text provider internal errors produce generic chat-safe copy without raw request IDs, while preserving the existing pre-output transient retry behavior.

## Reproduction Assessment

Yes, by source inspection. A reported Codex `server_error` payload flows through the shared formatter into `LLM error server_error: <provider message>`, then through final reply payload construction and Feishu send without a channel-safe fallback.

## Solution Assessment

Yes in direction, but the narrow maintainable fix is shared formatter/sanitizer coverage rather than a Feishu-only filter. The related PR should be repaired or replaced so provider request IDs stay in logs or bounded diagnostics, not user copy.
