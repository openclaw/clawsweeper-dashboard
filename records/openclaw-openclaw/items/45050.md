## Summary

Keep open. Current main already sanitizes raw JSON in some Codex/server_error paths and has the pre-output transient retry, but Feishu can still receive provider-shaped `LLM error server_error` text with request IDs embedded in the provider message; the open related PR overlaps but does not settle the request-ID and post-visible-output policy.

## What This Changes

Request that Feishu sanitize provider/internal `server_error` output, keep raw provider details in logs, and retry transient failures only before visible output is delivered.

## Best Possible Solution

Define one shared external-chat-safe provider-internal-error message, omit raw request IDs from chat copy, keep raw payloads in diagnostics, and decide whether late final assistant errors after visible output should be suppressed or genericized while retaining the existing pre-output retry.

## Reproduction Assessment

Yes, by source inspection. A Codex `server_error` can flow through `formatAssistantErrorText` into an `isError` payload, be preserved after streaming, and then be delivered by Feishu final handling; current tests still assert the provider-shaped copy.

## Solution Assessment

No complete fix is present yet. The narrow maintainable direction is a shared formatter/sanitizer and external-chat delivery policy; https://github.com/openclaw/openclaw/pull/49401 is related but still displays request IDs and does not resolve the Feishu post-visible-output behavior.
