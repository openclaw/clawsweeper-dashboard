## Summary

Keep open. Current main partially sanitizes raw JSON and already has the pre-output transient retry path, but the Feishu path can still deliver provider-shaped server_error copy with a provider request ID embedded in the message, and the related open PR does not resolve the request-ID or post-visible-output policy.

## What This Changes

The issue asks OpenClaw to keep raw provider server_error payloads and request IDs out of Feishu chat output and to retry transient provider failures only before visible output when safe.

## Best Possible Solution

Define one shared external-chat-safe provider-internal-error message, omit raw request IDs from chat copy, keep raw payloads in diagnostics, and decide whether late final assistant errors after visible output should be suppressed or genericized while retaining the existing pre-output retry.

## Reproduction Assessment

Yes, by source inspection. A Codex server_error payload can flow through `formatAssistantErrorText` into an `isError` reply payload, survive post-stream filtering, and then be delivered by Feishu final handling; tests still assert provider-shaped `LLM error server_error` copy.

## Solution Assessment

No complete solution is present yet. The narrow maintainable fix is a shared formatter/sanitizer and external-chat delivery policy; https://github.com/openclaw/openclaw/pull/49401 is related but still displays request IDs and does not resolve Feishu post-visible-output behavior.
