## Summary

Keep open. Current main mitigates raw JSON and some late Feishu final duplication, but it can still surface provider-shaped `LLM error server_error` text and request IDs embedded in provider messages through external chat delivery; #49401 overlaps but does not settle the Feishu/post-visible-output policy.

## What This Changes

Feishu should convert provider/internal `server_error` failures into channel-safe user copy, keep raw payloads and request IDs in diagnostics only, and retry transient failures only before visible output is delivered.

## Best Possible Solution

Define one shared external-chat-safe provider-internal-error message, redact request IDs from chat copy, keep raw payloads in diagnostics, and suppress or genericize final assistant error payloads after visible assistant content while retaining the existing pre-output retry.

## Reproduction Assessment

Yes. A high-confidence source-level path exists by constructing a Codex `server_error` assistant error, passing it through `formatAssistantErrorText`/`buildEmbeddedRunPayloads`, and observing Feishu final delivery use the resulting `isError` text; current tests also assert the provider-shaped copy.

## Solution Assessment

No complete fix is present yet. The maintainable fix should be a shared external-chat error formatting and post-visible-output policy rather than a Feishu-only string replacement, and #49401 still needs maintainer alignment because it preserves request IDs in displayed copy.
