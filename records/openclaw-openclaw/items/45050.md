## Summary

Keep open. Current main still has a source-level Feishu path where a Codex/provider `server_error` becomes provider-shaped `LLM error ...` chat text; existing retry and late-final suppression work only partially mitigates the report, and open #49401 overlaps without resolving the external-chat request-ID/post-visible-output policy.

## What This Changes

The issue requests Feishu-safe sanitization for provider/internal errors and an optional one-time transient retry only before any visible output is delivered.

## Best Possible Solution

Define one shared external-chat-safe provider-internal-error message, redact request IDs from chat copy, keep raw payloads in diagnostics, and genericize or suppress final assistant error payloads after visible assistant content while preserving the existing pre-output retry.

## Reproduction Assessment

Yes. A high-confidence source-level path exists by constructing a Codex `server_error`, passing it through `formatAssistantErrorText`/`buildEmbeddedRunPayloads`, and following the resulting final `isError` payload into Feishu delivery.

## Solution Assessment

No complete fix is present on current main. The narrow maintainable direction is shared external-chat error formatting plus a post-visible-output policy, not a Feishu-only string replacement; #49401 should be reviewed against that policy before this issue closes.
