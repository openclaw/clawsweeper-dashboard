## Summary

Keep open. Current main has partial mitigation for raw JSON and some Feishu late-final duplication, but it still formats provider failures as user-visible provider-shaped copy and Feishu can deliver that resolved text without a generic channel-safe rewrite. The existing pre-output retry is present, so the remaining work is the chat-safe error surface and post-visible-output behavior, not a broad retry rewrite.

## What This Changes

The issue asks OpenClaw to stop exposing Feishu users to raw or provider-shaped `server_error` failures, keep raw payloads/request IDs in diagnostics, and retry transient failures only before visible output is delivered.

## Best Possible Solution

Define a shared channel-safe provider-internal-error message for external chat delivery, keep raw payloads and provider request IDs in diagnostics, suppress or genericize final assistant error payloads after visible assistant content, and preserve the existing pre-output retry behavior with focused Feishu and shared-helper regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists through `formatAssistantErrorText`/`sanitizeUserFacingText`, `buildEmbeddedRunPayloads`, the final payload filter, and Feishu reply dispatch; no live Feishu outage is needed to prove the current delivery path.

## Solution Assessment

No current complete fix is present. #42892 and #72294 address adjacent pieces, while #49401 is an open partial formatter change that preserves request IDs in displayed copy and does not cover Feishu dispatch or post-visible-output behavior.
