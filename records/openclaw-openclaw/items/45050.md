## Summary

Current main still has a source-reproducible user-facing provider-error leak: structured Codex `server_error` payloads are formatted into `LLM error server_error: <provider message>`, and that provider message can include the help-center/request-ID text reported by Feishu users. The related open PR is the right implementation candidate, but it remains open and has review concerns, so this issue should stay open.

## What This Changes

The issue asks OpenClaw to prevent Feishu users from seeing raw provider `server_error` payloads and to retry transient provider failures only before visible output is delivered.

## Best Possible Solution

Fix the shared assistant error formatter/sanitizer path so generic provider internal errors become channel-safe copy without raw request IDs, preserve the existing pre-output retry/failover behavior, and land or replace the related open PR with focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. The reported Codex `server_error` payload flows through the shared raw formatter into chat-visible `LLM error server_error: <provider message>` text, then the Feishu send path posts the supplied text without a channel-specific fallback.

## Solution Assessment

Yes in direction, but the maintainable fix belongs in the shared formatter/sanitizer and failover path rather than a Feishu-only filter. The related PR should be repaired or replaced so user copy omits raw provider request IDs while logs retain diagnostics.
