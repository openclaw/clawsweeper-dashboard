## Summary

Current main still has a source-reproducible provider-error leak path: structured Codex `server_error` payloads are formatted into user-facing `LLM error server_error: ...` text, and the reported message body can still carry the provider help-center/request-ID text. The related open PR is the right implementation candidate, but it is not merged and has outstanding review concerns, so this issue should stay open.

## What This Changes

The issue asks OpenClaw to keep raw Feishu provider `server_error` payloads and request IDs out of chat-visible replies, with retry limited to the pre-output path.

## Best Possible Solution

Fix the shared assistant error formatter/sanitizer path so generic provider internal errors become channel-safe copy without raw request IDs, then land or replace the related open PR with focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. The reported `Codex error: { ... server_error ... }` payload flows through the shared formatter into chat-visible `LLM error server_error: <provider message>` text, and current tests pin that behavior rather than a generic fallback.

## Solution Assessment

Yes in direction, but the best implementation is shared formatter/sanitizer coverage rather than a Feishu-only filter. The related PR should be rebased into the current shared formatter path and omit raw provider request IDs from user copy.
