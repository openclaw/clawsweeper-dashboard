## Summary

Keep open. Current main mitigates raw JSON and some Feishu late-final duplication, but the Feishu-visible path can still emit provider-shaped assistant error text instead of a shared channel-safe fallback, and the remaining request-ID/post-visible-output policy is not fully resolved by the related open PR.

## What This Changes

Sanitize Feishu-visible provider failures into generic user-facing copy, keep raw payloads and request IDs in diagnostics, and preserve the existing safe retry only before visible output.

## Best Possible Solution

Define one shared channel-safe provider-internal-error message for external chat delivery, keep raw payloads and request IDs in diagnostics, suppress or genericize final assistant error payloads after visible assistant content, and retain the existing pre-output transient retry behavior with focused Feishu and shared-helper regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists through `formatAssistantErrorText`, `buildEmbeddedRunPayloads`, final payload filtering, and Feishu reply dispatch; current tests also lock in provider-shaped `LLM error server_error` copy.

## Solution Assessment

No complete fix is present yet. #42892 and #72294 address adjacent pieces, while #49401 is still open and partial because it preserves displayed request IDs and does not cover Feishu dispatch or post-visible-output behavior.
