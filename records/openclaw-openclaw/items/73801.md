## Summary

Keep open. Current main still supports and recommends the reported Active Memory plus Cerebras `gpt-oss-120b` setup, and although it has generic hard-timeout, abort-signal, circuit-breaker, and recent OpenAI transport signal wiring, I did not find a source/test proof that the Cerebras reasoning stream is cancelled cleanly or that gateway health stays responsive after timeout. The related setup-grace PR is useful but only covers a timeout-budget mismatch, not this provider-specific CPU/health report.

## What This Changes

The issue reports that Active Memory configured with `cerebras/gpt-oss-120b` times out far beyond its configured budget and can leave the local gateway CPU-bound/unresponsive despite direct Cerebras chat and tool calls succeeding.

## Best Possible Solution

Keep this as a focused Active Memory/Cerebras reliability regression until maintainers can reproduce it with either a live Cerebras run or a simulated OpenAI-compatible reasoning/tool stream, then fix cancellation/transport cleanup and gateway responsiveness or qualify the docs recommendation for Cerebras Active Memory recall.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path: exact config, environment, logs, direct Cerebras API controls, and recovery steps are included; the remaining proof should measure timeout, CPU, `/health`, and websocket/admin responsiveness during and after the recall.

## Solution Assessment

Unclear. Current main's generic abort and timeout mitigations are necessary but not enough to call this solved; the best fix needs provider-specific or simulated proof that the embedded Cerebras gpt-oss path releases work and preserves gateway health.
