## Summary

Keep open: this is a focused fix for a source-reproducible overflow-rewrite bug, but the current patch should not merge as-is because its compaction replay identity can collapse distinct compaction boundaries.

## What This Changes

The PR adds replay-identity deduplication to transcript rewrite suffix replay, adds regression tests for duplicate message/bootstrap/compaction replay, and adds a changelog entry.

## Best Possible Solution

Land a current-base transcript rewrite idempotency fix that dedupes replay clones while preserving distinct compaction kept-entry boundaries, then close the concrete Track A issue and leave the broader umbrella open.

## Reproduction Assessment

Yes. The linked issue provides persisted JSONL/log evidence, and current-main source tracing shows overflow recovery reaches unguarded suffix replay; I did not run a live overflow in this read-only review.

## Solution Assessment

No, not as written. The central rewrite replay boundary is the right place to fix this, but compaction identity should include the remapped firstKeptEntryId so distinct compaction boundaries are not collapsed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Include the kept boundary in compaction identity:** `src/agents/pi-embedded-runner/transcript-rewrite.ts:42-46`
  - body: The new compaction replay key ignores `firstKeptEntryId`. `buildSessionContext()` uses the latest compaction's kept-entry boundary to decide which pre-compaction messages remain in context, so two legitimate compactions with identical summaries/details/tokens but different kept boundaries would collapse to the earlier entry during rewrite and lose the newer boundary. Include the remapped kept id in the identity so cloned entries still dedupe without merging distinct compactions.
  - confidence: 0.83
