## Summary

Keep open: this PR targets a valid Mattermost thread-context recovery bug, but the patch treats an empty pending-history map as recovery even though that map is normally cleared after successful turns, and the updated proof still does not exercise the live Mattermost fetch and monitor path.

## What This Changes

The PR adds a Mattermost thread fetch helper, a server-thread-to-history mapper, monitor wiring, and tests to seed empty thread history before inbound context assembly.

## Best Possible Solution

Keep the Mattermost-owned server backfill, but gate it on a real recovery, new-session, or missing-context signal and add both live Mattermost proof and regression coverage for active-thread steady state.

## Reproduction Assessment

Yes, at source level. Current main keeps Mattermost thread context in a process-local map and has no server-thread fetch, so restart or session clear can leave an inbound thread turn with no prior thread messages.

## Solution Assessment

No, not as written. The Mattermost-owned fetch/backfill boundary is plausible, but the trigger must be recovery-aware because an empty pending-history window is also normal after successful active thread replies.
