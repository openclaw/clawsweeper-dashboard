## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Track the effective thread root:** `extensions/mattermost/src/mattermost/monitor.ts:1582-1585`
  - body: This passes only `post.root_id` into the serviced-root gate. For a top-level message that starts a Mattermost thread session, `post.root_id` is still empty while `effectiveReplyToId` and `sessionKey` are already thread-scoped; after that turn succeeds, the kernel clears the window, so the first real thread reply is treated as recovery and fetches the full server thread. Mark or gate on the effective thread root so normal active follow-ups do not rehydrate.
  - confidence: 0.9
- **[P2] Bound the recovery thread fetch:** `extensions/mattermost/src/mattermost/client.ts:192`
  - body: The helper requests `/posts/{root}/thread` without pagination even though the monitor only keeps `historyLimit` entries. Mattermost's API supports thread paging parameters, so a long thread can be fully transferred and parsed on the awaited inbound path before the agent sees the mention. Pass a bounded request shape or otherwise cap server work before dispatch.
  - confidence: 0.78
