## What This Changes

## Best Possible Solution

Keep the Mattermost-owned server backfill, but gate it on actual recovery, new-session, or missing-context state and require live Mattermost proof of recovered thread context.

## Reproduction Assessment

Yes, at source level. Current main keeps Mattermost thread history in a process-local map and has no server-thread fetch, so restart or session clear can leave an inbound thread turn with no prior thread messages.

## Solution Assessment

No, not as written. The Mattermost-owned fetch/backfill boundary is plausible, but the trigger must be recovery-aware because an empty pending-history window is also normal after successful active thread replies.
