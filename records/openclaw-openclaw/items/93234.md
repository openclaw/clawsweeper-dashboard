## Summary

Keep open: this PR is a useful candidate for the Mattermost restart/session-clear thread-context bug, but the current branch still gates recovery on `post.root_id` instead of the effective Mattermost thread root, so an ordinary follow-up after a top-level thread-starting turn can be misclassified as recovery.

## What This Changes

The PR adds a Mattermost thread-fetch helper, a first-sighting thread backfill gate, monitor wiring, and unit tests for rebuilding thread context after restart or session clear.

## Best Possible Solution

Land one canonical Mattermost-owned fix for https://github.com/openclaw/openclaw/issues/93204 that gates cold backfill on the effective thread root, avoids normal follow-up REST fetches, and includes redacted live Mattermost recovery proof.

## Reproduction Assessment

Yes, at source level. Current main initializes Mattermost thread history in a process-local map and builds context from that map, while the shared turn kernel clears group pending history after dispatch.

## Solution Assessment

No, not yet. The Mattermost-owned backfill layer is appropriate, but the serviced-root gate should use the effective thread root and the PR still needs real Mattermost recovery proof.
