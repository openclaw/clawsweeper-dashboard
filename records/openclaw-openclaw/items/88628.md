## Summary

Keep open: the PR addresses a real plugin callback scoping bug, but the current wrapper can break valid class/prototype-backed plugin tools by cloning only own enumerable fields. The maintainer label also makes this ineligible for cleanup close, and the related broader implementation at https://github.com/openclaw/openclaw/pull/87829 is still open rather than merged.

## What This Changes

The PR wraps plugin tool factory results at materialization so factories, `prepareArguments`, and `execute` run in the owning plugin runtime scope, with regression tests in `src/plugins/tools.optional.test.ts`.

## Best Possible Solution

Keep the materialization-boundary scoping fix, but implement it as a delegating wrapper that preserves the original tool surface and add regression coverage for inherited or non-enumerable tool properties before choosing between this PR and the related open alternative.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: return a plugin tool object whose `name` or `parameters` are inherited or non-enumerable, then the PR wrapper's object spread drops them before validation. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. Scoping callbacks at the shared materialization boundary is the right direction, but the wrapper should delegate to the original tool surface instead of cloning enumerable fields.
