## Label Justifications

## Summary

This PR is still necessary: current main and the latest release still allow same-channel fallback route fields to cross account ids, while the branch restores the earlier account-bound guard at the shared helper. No blocking patch defect was found after checking callers, sibling media routing, tests, docs, and the regression provenance.

## What This Changes

The PR restores an account-mismatch guard in `mergeDeliveryContext` and adds a regression test for same-channel fallback across different account ids.

## Best Possible Solution

Land the shared account-bound route guard with the restored regression test after CI and maintainer acceptance of the fail-closed account-conflict behavior.

## Reproduction Assessment

Yes, at source level: current main's `mergeDeliveryContext` only checks channel mismatch, so same-channel contexts with different account ids can inherit fallback `to` and `threadId`. I did not run a live Telegram two-account repro in this read-only review.

## Solution Assessment

Yes. The shared helper is the right boundary because subagent, cron, media, sentinel, outbound, and session-store callers already centralize route-field crossing there; per-caller fixes would leave drift.
