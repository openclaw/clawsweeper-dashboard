## Summary

Keep open: the PR targets a real Codex dynamic-tool timeout bug, but the fixed 240s watchdog still ignores existing per-call/configured tool timeout budgets and overlaps with more complete open timeout work.

## What This Changes

The PR raises the Codex app-server dynamic tool execution and server-request watchdogs from 30s to a shared 240s constant, adds focused tests, and updates the changelog.

## Best Possible Solution

Resolve Codex dynamic-tool deadlines from existing per-call/configured tool budgets with a bounded cap, and keep the run-attempt and client watchdogs aligned.

## Reproduction Assessment

Yes, from source. Current main has fixed 30000ms Codex dynamic-tool watchdogs while `image_generate` exposes and forwards longer per-call/configured `timeoutMs` values, so the bridge can preempt a valid long-running tool call.

## Solution Assessment

No. Raising the shared deadline to 240s addresses the immediate 30s symptom, but the maintainable fix should derive the bridge deadline from the existing tool timeout budget and apply a bounded cap instead of hard-coding another global limit.
