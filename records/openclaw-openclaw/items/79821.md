## Summary

Keep open: the patch addresses the immediate 30s Codex watchdog by moving both watchdogs to 240s, but it still uses a fixed bridge budget instead of honoring existing per-call/configured tool timeout contracts and it needs real behavior proof before merge. This also overlaps active timeout work in https://github.com/openclaw/openclaw/pull/74528 and https://github.com/openclaw/openclaw/pull/79591.

## What This Changes

The PR adds a shared 240000 ms Codex dynamic-tool timeout constant, uses it in the run-attempt and app-server client watchdogs, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a tool-budget-aware Codex dynamic-tool timeout fix that keeps run-attempt and client watchdogs aligned, honors existing media/tool timeout contracts, updates the Codex harness docs, and is backed by redacted real behavior proof.

## Reproduction Assessment

Yes from source, but not from a live run in this review. Current main uses 30000 ms run-attempt and client watchdogs while image_generate exposes and forwards configured/per-call timeoutMs such as 180000 ms.

## Solution Assessment

No, not yet. Raising a shared constant to 240s fixes the immediate workaround shape, but the maintainable fix should resolve and cap the existing tool timeout budget instead of replacing one fixed bridge deadline with another.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Honor the tool timeout budget:** `extensions/codex/src/app-server/dynamic-tool-timeout.ts:1`
