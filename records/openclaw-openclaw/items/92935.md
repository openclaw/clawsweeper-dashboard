## Summary

Current main and the shipped v2026.5.22 release already implement the supported Control UI/WebChat canvas embed path: bare unauthenticated canvas URLs still 401 by design, while WebChat iframe sources are rewritten through scoped canvas capability URLs.

## What This Changes

The issue reports that generated local PNGs do not reliably render through Control UI/WebChat `[embed ...]` canvas embeds on OpenClaw 2026.5.6, while the same files render through the `MEDIA:` outbound route workaround.

## Best Possible Solution

Keep the shipped scoped canvas capability URL path and the ticketed assistant-media route; if a newer build still fails, collect the actual iframe `src` and `hello-ok.pluginSurfaceUrls.canvas` value as a new report.

## Reproduction Assessment

No current-main failing reproduction is present. The reporter's 2026.5.6 evidence explains the old bare canvas URL 401, while current source and tests show WebChat uses a scoped capability URL and intentionally keeps bare unauthenticated canvas paths unauthorized.

## Solution Assessment

Yes. The scoped capability URL is the best current fix because it lets the connected WebChat session load canvas iframe documents and subresources without exposing reusable gateway credentials in iframe markup.
