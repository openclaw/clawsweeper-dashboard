## Summary

Current main and shipped releases already implement the supported behavior: Control UI embeds render through scoped canvas capability URLs, while generated image/file attachments are delivered through the media path; the reported bare canvas 401 is expected auth behavior rather than evidence that WebChat is using the wrong iframe URL.

## What This Changes

The issue asks for generated local images to render consistently in Control UI/WebChat via documented canvas `[embed ref="..."]` embeds instead of requiring the `MEDIA:/data/.openclaw/media/outbound/...` workaround.

## Best Possible Solution

Keep the current contract: use structured media or `MEDIA:` for generated image/file attachments, and use `[embed ...]` only for hosted Control UI documents rendered through scoped capability URLs.

## Reproduction Assessment

No high-confidence current-main failing reproduction remains. The provided live curl evidence exercises the bare canvas URL, and current source/tests show that path is supposed to return 401 while WebChat should render through a scoped capability URL.

## Solution Assessment

Yes, the current implementation is the best supported shape because it keeps attachments on the media delivery path and keeps hosted canvas embeds behind scoped plugin capability URLs instead of weakening gateway auth for iframe content.
