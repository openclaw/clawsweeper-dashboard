## Summary

Close as implemented. Current main and the latest release route Control UI `[embed ...]` canvas iframes through scoped plugin capability URLs, while `MEDIA:` local attachments use the separate assistant-media ticket path; the reported bare canvas 401 is expected for an unauthenticated direct route and the report is from an older 2026.5.6 install.

## What This Changes

The issue asks for generated local images to render reliably through Control UI canvas/`[embed ...]` instead of requiring the `MEDIA:` outbound workaround.

## Best Possible Solution

Keep the scoped capability iframe path as the supported Control UI embed mechanism and ask for a fresh report only if `v2026.6.6` or current main still fails with the actual iframe `src` and hello `pluginSurfaceUrls.canvas` payload.

## Reproduction Assessment

No high-confidence current-main failure was reproduced. The reporter provided useful live evidence on OpenClaw 2026.5.6, but current main and `v2026.6.6` source/tests show the supported WebChat path uses scoped capability URLs and the bare canvas 401 remains expected.

## Solution Assessment

Yes. The current scoped capability design is the best maintained fix because it authorizes the connected WebChat session without placing the shared Gateway token in iframe markup, while `MEDIA:` remains a separate attachment path.
