## Summary

Keep open: current main has scoped canvas capability URL handling and tests/docs show a bare canvas 401 is expected, but the reporter supplied live WebChat failure evidence from v2026.5.6 and we do not have the browser-side scoped iframe or hello payload proof needed to close this as implemented.

## What This Changes

Reports that Control UI/WebChat can fail to render generated PNGs through `[embed ref="..."]` canvas documents while the same files render through `MEDIA:` outbound delivery.

## Best Possible Solution

Verify the current-main WebChat iframe path with a browser/network trace, then either fix the scoped canvas capability flow or clarify that generated image files should use media delivery while `[embed ...]` remains for hosted documents.

## Reproduction Assessment

Unclear: the reporter gave a live v2026.5.6 setup where `[embed ref="..."]` did not render and bare canvas URLs returned 401, but current source/tests say those bare URLs should 401 while WebChat should load a scoped capability URL. A current-main browser network trace or hello payload would make the failure high-confidence.

## Solution Assessment

Unclear: the current design is maintainable if generated files use `MEDIA:` and hosted documents use scoped `[embed ...]`, but a real scoped iframe failure would need a gateway/UI fix rather than loosening the canvas auth boundary.
