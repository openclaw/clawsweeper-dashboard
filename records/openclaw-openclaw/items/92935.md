## Summary

Close as implemented: current main and shipped releases already define `[embed ref]` as a hosted canvas document rendered through scoped capability URLs, while `MEDIA:` remains the supported generated-image attachment path and the reported bare 401 is expected auth behavior.

## What This Changes

Issue requests reliable Control UI/WebChat rendering for generated images via `[embed ref="..."]` without requiring the `MEDIA:` outbound workaround.

## Best Possible Solution

Keep the current contract: use structured media or `MEDIA:` for generated image/file attachments, and use `[embed ref]` for hosted Control UI canvas documents rendered through scoped capability URLs.

## Reproduction Assessment

No. I do not have a high-confidence current-main failing repro; the provided curl evidence hits the unauthenticated canvas URL that current tests expect to return 401, while current and shipped source show WebChat should render through a scoped capability URL.

## Solution Assessment

Yes. The current implementation is the best core solution because it keeps the auth boundary intact: generated files use media delivery, and rich embeds use hosted canvas documents through scoped capability URLs instead of putting reusable gateway credentials into iframe URLs.
