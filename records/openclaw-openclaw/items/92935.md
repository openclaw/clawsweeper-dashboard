## Summary

Close as implemented. Current main and shipped releases already support the two intended paths: generated images/files are attachment media, while `[embed ...]` is a Control UI hosted-document path that is rendered through scoped capability URLs; the reported bare canvas 401 is expected auth behavior, not proof that WebChat is using the wrong iframe URL.

## What This Changes

The issue reports unreliable Control UI/WebChat rendering for generated images via `[embed ref]` while the same PNG renders through the `MEDIA:` route.

## Best Possible Solution

Keep the current contract: use structured media or `MEDIA:` for generated image/file attachments, and use `[embed ...]` only for hosted Control UI documents rendered through scoped capability URLs.

## Reproduction Assessment

No high-confidence current-main failing repro remains. The provided curl evidence exercises the bare canvas URL, and current source/tests show that path is supposed to return 401 while WebChat should render through a scoped capability URL.

## Solution Assessment

Yes for the supported behavior. The existing solution keeps the security boundary intact: attachments use media delivery, and hosted embeds use scoped canvas capability URLs instead of putting Gateway auth into iframe markup.
