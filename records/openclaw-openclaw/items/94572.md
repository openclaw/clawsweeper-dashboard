## Summary

Keep open: current main still shows the deferred runtime-plugin prewarm path loading through `ensureRuntimePluginsLoaded` without handing the resulting registry back to the gateway attachment owner or starting plugin services. The open closing PR at https://github.com/openclaw/openclaw/pull/94822 is unmerged and already appears too narrow for the service-lifecycle half of this issue, so this issue remains necessary.

## What This Changes

The issue reports that workspace plugins loaded via `plugins.load.paths` without `activation.onStartup` can register HTTP routes and services, but the gateway returns 404 for those routes and never calls `service.start()`.

## Best Possible Solution

Wire deferred runtime-loaded plugin registries into the gateway-owned attached route registry and service lifecycle, add focused regression coverage, and verify with redacted curl output plus a service-start log before closing this issue.

## Reproduction Assessment

Yes at source level: current main still has a deferred prewarm path that loads runtime plugins without the gateway attachment and service-start handoff. I did not run the live curl/service repro during this read-only review.

## Solution Assessment

Yes: the best fix is in the gateway-owned deferred-load attachment path, not an undocumented `activation.onStartup` workaround or a bootstrap-only pin. The current open PR should be repaired or replaced to cover both HTTP route dispatch and service lifecycle.
