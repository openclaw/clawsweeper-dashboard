## Label Justifications

- `P2`: The issue blocks documented workspace-plugin HTTP routes and services with real but bounded plugin-author impact.
- `impact:message-loss`: Webhook-style plugin HTTP routes returning 404 can silently prevent inbound plugin/channel messages from reaching OpenClaw.
- `impact:other`: Plugin services registered through the SDK may never receive their host-managed lifecycle start callback.

## Summary

Current `main` and the latest release still leave this issue actionable: the post-ready runtime plugin prewarm loads workspace plugins after gateway startup without handing that registry to the gateway’s attached HTTP-route/service lifecycle, and the linked open PRs are partial or unsafe rather than merged fixes.

## What This Changes

The issue reports that workspace plugins loaded through `plugins.load.paths` without `activation.onStartup` register HTTP routes and services, but cold boot leaves their HTTP routes returning 404 and their services unstarted.

## Best Possible Solution

Keep this issue open until one complete fix lands that attaches deferred runtime-loaded plugin registries through the gateway-owned HTTP-route and service lifecycle, with focused regression tests and redacted runtime proof.

## Reproduction Assessment

Yes at source level: current main still loads runtime plugins through a post-ready prewarm path that does not attach the resulting registry to gateway-owned HTTP routes or services. I did not run the live curl/service repro in this read-only review.

## Solution Assessment

No for the current open PR candidates: route-only registry pinning is not the best full fix because the issue also covers service lifecycle and gateway-owned attached registry semantics. The maintainable path is one deferred attachment owner for routes, methods, and services.
