## Label Justifications

- `P2`: This is a normal-priority core/plugin runtime bug with clear impact on workspace plugin HTTP routes and services but no evidence of a core-wide outage.
- `impact:message-loss`: Plugin HTTP route 404s can drop inbound webhook/channel-style messages before a plugin handler sees them.
- `impact:other`: The report also covers plugin background services registering successfully but never receiving host-managed `start()` callbacks.

## Summary

Keep open: current main still shows the runtime-loaded plugin path warming an active registry after post-attach sidecars without replacing the gateway-attached HTTP-route registry or starting services for that loaded registry, and the linked fix branches are still open and unmerged.

## What This Changes

The issue reports that workspace plugins loaded from `plugins.load.paths` without `activation.onStartup` can register HTTP routes and services, but gateway HTTP dispatch returns 404 and service `start()` callbacks never run.

## Best Possible Solution

Land one reviewed fix that wires deferred runtime plugin loading into the gateway-attached HTTP-route registry and plugin-service lifecycle, with focused regression coverage and real gateway proof, then close or supersede the overlapping open branches.

## Reproduction Assessment

Yes, source-reproducible: the issue provides a concrete workspace-plugin repro, and current main shows the post-ready runtime load path does not update the gateway-attached route registry or start services for that later registry. I did not run the live gateway curl scenario in this read-only pass.

## Solution Assessment

No, current main is not the best end state because it still separates deferred runtime loading from gateway route/service attachment. The right shape is additive wiring for the deferred load path rather than requiring plugin authors to set `activation.onStartup` for documented HTTP-route or service APIs.
