## Label Justifications

- `P2`: The bug blocks documented workspace-plugin HTTP routes and services with real plugin-author impact but a bounded gateway/plugin surface.
- `impact:message-loss`: Webhook-style plugin HTTP routes returning 404 can silently prevent inbound plugin or channel messages from reaching OpenClaw.

## Summary

Keep open: current main still has a source-visible gap in the deferred runtime-loaded workspace plugin path, and the open linked PRs have not yet landed a complete route plus service lifecycle fix.

## What This Changes

The issue asks OpenClaw to make workspace plugins loaded through `plugins.load.paths` wire their registered HTTP routes and background services into the running gateway without requiring `activation.onStartup`.

## Best Possible Solution

Land one gateway-owned deferred runtime attachment fix that updates the server-local route/method registries and starts/stops plugin services for runtime-loaded workspace plugins, with focused route and service lifecycle proof.

## Reproduction Assessment

Yes at source level: the deferred prewarm path loads runtime plugins without attaching the loaded registry to the gateway route/service lifecycle. A live packaged curl and service-start repro is still missing.

## Solution Assessment

Yes: the best fix belongs in a gateway-owned deferred-load attachment path, not in making plugin authors opt into startup loading or landing a bootstrap-only route pin.
