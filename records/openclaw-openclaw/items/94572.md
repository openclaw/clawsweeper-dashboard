## Label Justifications

- `P2`: Runtime-loaded workspace plugin HTTP routes and services can be unusable, but the impact is bounded to plugin authors and has a known `activation.onStartup` workaround.
- `impact:message-loss`: Webhook-style plugin HTTP routes returning 404 can silently prevent inbound plugin or channel messages from reaching OpenClaw.
- `impact:other`: Plugin services registered through the SDK may never receive their host-managed lifecycle start callback.

## Summary

Keep open: current main still leaves the post-ready runtime plugin prewarm path detached from the gateway-owned HTTP route and plugin service lifecycle, and the open linked PRs are unmerged candidate fixes rather than completed implementation.

## What This Changes

Issue requests fixing runtime-loaded workspace plugins so HTTP routes and `registerService` callbacks are wired during cold-boot deferred plugin loading.

## Best Possible Solution

Add one gateway-owned deferred-load attachment path that updates the server-local route/method registry and starts/stops services for runtime-loaded plugins, with focused regression tests and redacted live proof.

## Reproduction Assessment

Yes, at source level: the post-ready runtime plugin prewarm path loads plugins without attaching that registry to the gateway route/service lifecycle. I did not run the live curl/systemd reproduction in this read-only review.

## Solution Assessment

Yes for the issue direction: the narrow maintainable solution is a gateway-owned deferred attachment path for routes, methods, and services. The current open route-only PR shapes are not the best full fix for the reported route-plus-service failure.
