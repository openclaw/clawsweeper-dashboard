## Label Justifications

- `P2`: The issue blocks documented workspace-plugin HTTP routes and background services for plugin authors, with real but bounded blast radius.
- `impact:message-loss`: Webhook-style plugin HTTP routes returning 404 can silently prevent inbound plugin or channel messages from reaching OpenClaw.
- `impact:other`: Plugin services registered through the SDK may never receive their host-managed lifecycle start callback.

## Summary

Keep open: current main and v2026.6.8 still show the deferred runtime plugin path loading registries without attaching HTTP routes or starting plugin services, and the linked open PR is only a candidate rather than a merged fix.

## What This Changes

Runtime-loaded workspace plugins should have `api.registerHttpRoute(...)` routes attached to the gateway and `api.registerService(...)` callbacks started without requiring `activation.onStartup: true`.

## Best Possible Solution

Add one gateway-owned deferred-load attachment path that updates the server-local route registry and starts/stops services for runtime-loaded plugins, with focused HTTP dispatch and service lifecycle regression coverage.

## Reproduction Assessment

Yes at source level: the reported cold-boot deferred path loads runtime plugins through `scheduleAgentRuntimePluginPrewarm` and `ensureRuntimePluginsLoaded` without gateway route attachment or service startup. I did not run the live curl/service reproduction.

## Solution Assessment

Yes for the issue's proposed direction: the maintainable fix is to wire deferred runtime-loaded registries into the same gateway-owned route and service lifecycle used by startup and reload paths. The current linked PR's bootstrap-only pin is too narrow to be the full solution.
