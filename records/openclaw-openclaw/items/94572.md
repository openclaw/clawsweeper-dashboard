## Label Justifications

- `P2`: This is a bounded gateway/plugin runtime bug with real workspace-plugin webhook and background-service impact.
- `impact:message-loss`: Runtime-loaded plugin webhooks can return 404, which can drop inbound messages or webhook events before plugin handlers run.
- `impact:other`: The same deferred-load gap leaves plugin background service `start()` callbacks unrun, which is meaningful plugin lifecycle impact outside the specific owned labels.

## Summary

Keep open: current main still leaves the post-ready runtime plugin load detached from the gateway-attached HTTP route registry and plugin service lifecycle, and the linked fix PRs are still open/unmerged.

## What This Changes

The issue reports that workspace plugins loaded through `plugins.load.paths` without `activation.onStartup` register HTTP routes and services, but cold-boot deferred loading leaves routes returning 404 and service `start()` callbacks unrun.

## Best Possible Solution

Land or replace a gateway-owned deferred runtime attachment fix that updates the attached route/method registries and starts/stops plugin services for runtime-loaded workspace plugins, with focused regression coverage and redacted curl/log proof.

## Reproduction Assessment

Yes. Source inspection shows the cold-boot deferred prewarm path loads runtime plugins through `ensureRuntimePluginsLoaded` without replacing the gateway-attached registry or starting services; I did not run the live VPS curl/service reproduction.

## Solution Assessment

Yes, the maintainable fix direction is gateway-owned deferred runtime attachment that reuses startup/reload registry and service lifecycle behavior. Route-only bootstrap pinning is too narrow because it does not cover the reported service-start failure.
