## Label Justifications

- `P2`: The issue blocks documented workspace-plugin HTTP routes and background services with real but bounded plugin-author impact.
- `impact:message-loss`: Webhook-style plugin HTTP routes returning 404 can silently prevent inbound plugin or channel messages from reaching OpenClaw.
- `impact:other`: Plugin services registered through the SDK may never receive their host-managed lifecycle start callback.

## Summary

Keep open: current main and v2026.6.8 still leave the deferred runtime plugin prewarm path detached from the gateway-owned HTTP route and service lifecycle, and the open candidate PRs have not merged.

## What This Changes

The issue asks for runtime-loaded workspace plugin HTTP routes and registered services to work on cold boot without requiring `activation.onStartup`.

## Best Possible Solution

Add one gateway-owned deferred-load attachment path that updates the server-local route/method registry and starts/stops services for runtime-loaded plugins, with focused regression tests and redacted live proof.

## Reproduction Assessment

Yes, at source level: the cold-boot deferred prewarm path loads runtime plugins without attaching their registry to the gateway route/service lifecycle. I did not run the live VPS curl reproduction in this read-only review.

## Solution Assessment

Yes for the requested direction. The maintainable fix is gateway-owned deferred attachment for routes, methods, and services, not an undocumented `activation.onStartup` requirement or a process-global route fallback.
