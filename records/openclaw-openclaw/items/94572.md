## Summary

Keep open: current main still has a source-visible gap between the cold runtime plugin load path and the gateway-owned HTTP route/service attachment path, so the reported workspace-plugin route 404 and missing service start remain actionable.

## What This Changes

The issue reports that workspace plugins loaded through `plugins.load.paths` can register HTTP routes and services, but those surfaces are not attached to the running gateway unless the plugin is loaded as a startup plugin.

## Best Possible Solution

Add a gateway-owned deferred plugin attachment path that updates the server-local plugin registry/HTTP route registry and starts/stops services for the same loaded registry, with focused regression coverage and docs alignment for activation expectations.

## Reproduction Assessment

Yes for a source-level reproduction path: current main shows cold runtime prewarm can load a registry without attaching it to the server-local HTTP route registry or starting services. I did not run the reporter's packaged gateway/curl repro in this read-only review.

## Solution Assessment

Yes, the reporter's additive wiring direction is better than removing the startup HTTP-route pin because it preserves the earlier registry-churn fix while attaching the deferred registry through the gateway-owned lifecycle.

## Repair Work Prompt

Trace the gateway post-ready runtime plugin prewarm and standalone runtime registry loader paths, then make a loaded runtime registry that owns HTTP routes/services attach through the gateway-owned server-local plugin runtime path: route dispatch should see routes registered by a deferred workspace plugin, and registered services should receive exactly one host-managed `start()` plus a clean `stop()` on gateway shutdown/reload. Preserve the existing startup HTTP route pin and server-local registry isolation from prior fixes; do not remove `pinActivePluginHttpRouteRegistry`, do not start services from generic tool-discovery/CLI-only standalone loads, do not fix the separate gateway-method issue tracked by https://github.com/openclaw/openclaw/issues/94127, and do not edit `CHANGELOG.md`.
