## Label Justifications

- `P2`: The issue blocks documented workspace-plugin HTTP routes and background services with limited but real plugin-author impact.
- `impact:message-loss`: Webhook-style plugin HTTP routes returning 404 can silently prevent inbound plugin/channel messages from reaching OpenClaw.
- `impact:other`: Plugin services registered through the SDK may never receive their host-managed lifecycle start callback.

## Summary

Keep open: current main and the latest release still show a source-visible gap between deferred runtime plugin loading and the gateway-owned HTTP route/service attachment lifecycle, so the reported workspace-plugin 404 and missing `service.start()` remain actionable.

## What This Changes

Runtime-loaded workspace plugins can register HTTP routes and services that do not attach to the running gateway on cold boot unless the plugin is forced into startup activation.

## Best Possible Solution

Add one gateway-owned deferred-load attachment path that updates the server-local route registry and starts/stops services for runtime-loaded plugins, with regression coverage for HTTP dispatch and service lifecycle.

## Reproduction Assessment

Yes at source level: the deferred runtime prewarm path loads plugins without attaching HTTP route and service surfaces to the gateway-owned runtime state. I did not run the packaged curl/systemd reproduction in this read-only review.

## Solution Assessment

Yes, the best direction is to wire deferred runtime-loaded plugin registries into the same gateway-owned route and service lifecycle used by startup and reload paths, rather than requiring `activation.onStartup` as an undocumented workaround.
