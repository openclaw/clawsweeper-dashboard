## Label Justifications

- `P2`: The issue blocks documented workspace-plugin HTTP routes and background services for plugin authors, but the blast radius is bounded to plugin lifecycle wiring.
- `impact:message-loss`: Plugin HTTP routes used for webhook-style inbound messages can return 404 after registration, silently dropping plugin/channel input.
- `impact:other`: Plugin services registered through the SDK may never receive their host-managed lifecycle start callback.

## Summary

Keep open: current main and v2026.6.8 still show the deferred runtime plugin load path separate from the gateway-owned HTTP route and service attachment lifecycle, and the open closing PR has not landed and has already been reviewed as only partial.

## What This Changes

The issue reports that runtime-loaded workspace plugins can register HTTP routes and services successfully, but the gateway returns 404 for those routes and never starts registered services unless the plugin opts into startup activation.

## Best Possible Solution

Wire deferred runtime-loaded plugin registries into one gateway-owned attachment path that updates server-local route state and starts/stops services, with regression coverage for both route dispatch and service lifecycle.

## Reproduction Assessment

Yes at source level: the deferred runtime prewarm path loads runtime plugins without handing the resulting registry to the gateway attachment/service lifecycle. I did not run the live curl/systemd reproduction in this read-only review.

## Solution Assessment

Yes for the requested direction: the maintainable fix is to attach deferred runtime-loaded registries through the same gateway-owned route and service lifecycle used by startup and reload paths, not to require undocumented startup activation.
