## Summary

Keep open: current main still leaves a meaningful unresolved gateway/plugin lifecycle question for runtime-loaded workspace plugin HTTP routes and services, and two open PRs already claim this issue but are not merged.

## What This Changes

The issue reports that workspace plugins loaded via `plugins.load.paths` can register HTTP routes and services during `register(api)`, but cold-boot deferred loading leaves routes returning 404 and service callbacks unstarted.

## Best Possible Solution

Review, land, or replace the existing linked PRs with one gateway-owned lifecycle fix that attaches deferred runtime registries and starts/stops plugin services, then close this issue with live route and service proof.

## Reproduction Assessment

Yes, at source level: the prewarm path still loads runtime plugins without the gateway-attached route/service lifecycle. I did not run the live VPS curl and service-log reproduction in this read-only review.

## Solution Assessment

No, current main is not enough for the reported runtime-loaded surface, and the open route-only PRs are not a complete fix. The best fix is the gateway-owned deferred attachment path that covers routes and services together.
