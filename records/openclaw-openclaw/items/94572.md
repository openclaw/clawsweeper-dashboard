## Summary

Keep open: current main and latest release evidence still show the reported deferred runtime-loaded workspace plugin path is not fully wired into the gateway route and service lifecycle, and the linked open PR is only an unmerged partial candidate.

## What This Changes

The issue requests fixing runtime-loaded workspace plugins so registered HTTP routes dispatch through the gateway and registered services start without requiring `activation.onStartup: true`.

## Best Possible Solution

Broaden or replace https://github.com/openclaw/openclaw/pull/94822 with a focused gateway/plugin lifecycle fix that attaches deferred runtime-loaded registries to the running gateway and starts/stops their services, with route and service regression proof.

Yes at source level: the deferred runtime prewarm path loads plugins through `ensureRuntimePluginsLoaded` but does not replace the gateway-attached runtime or start plugin services. I did not run the live curl/log reproduction in this read-only review.

## Solution Assessment

No for the current linked PR as written: adding only the bootstrap HTTP-route pin is a partial mitigation and does not cover the service-start half of the reported cold-boot deferred-load path. The best fix should wire the deferred runtime registry into the gateway-owned route and service lifecycle.
