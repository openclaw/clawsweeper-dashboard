## Summary

Keep open. Current main still lacks script-backed CLI taglines, but this is a new feature/config surface that needs maintainer product and security acceptance, and the patch has concrete blockers against current startup and config-schema contracts.

## What This Changes

The PR adds a `script` CLI banner tagline mode, a `cli.banner.taglineScriptFile` config key, dynamic JS default-export loading for banner text, tests, schema/help metadata, and docs.

## Best Possible Solution

If maintainers want this feature, rebase it onto the current startup wrapper, update generated schema/docs/changelog, and run script taglines in a killable worker or child-process boundary; otherwise prefer a safer static custom-tagline config.

## Reproduction Assessment

Yes. Source inspection is enough: current main only accepts `random`, `default`, and `off` in types, parser, generated schema, and docs, while the PR adds a new `script` mode.

## Solution Assessment

No. The direction may be acceptable after maintainer approval, but the current patch is not the narrowest safe implementation because it executes JavaScript on the main startup path and misses current generated-schema/startup contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Await the async banner path from current startup:** `src/cli/banner.ts:170`
  - body: This changes `emitCliBanner` to async, but current main's `applyCliExecutionStartupPresentation` still calls it without `await`. After a rebase, script resolution can continue in the background and write or populate the help cache after command startup has already moved on, so the current startup wrapper and callers need to await this path.
  - confidence: 0.84
- **[P2] Regenerate the public config schema:** `src/config/zod-schema.ts:334-337`
  - body: The PR updates the handwritten Zod schema for `script` and `taglineScriptFile`, but it does not update the generated base schema served to config-schema consumers. `openclaw config schema` would still expose only `random/default/off` and omit `taglineScriptFile`, and the schema generation check should fail on a current-main rebase.
  - confidence: 0.88
- **[P2] Run tagline scripts in an abortable boundary:** `src/cli/tagline.ts:299-304`
  - body: The timeout races code still running on the main Node event loop, so it cannot stop top-level module work or a synchronous default export such as the documented `execSync` example. A blocking script can hang normal CLI startup indefinitely despite `TIMEOUT_MS`; use a worker/child process with a killable timeout or avoid executing JS here.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds config-selected JavaScript execution during CLI startup and needs explicit security review plus real containment before merge.

Concerns:

- **[medium] Config-selected JavaScript runs at startup:** `src/cli/tagline.ts:287`
  - body: `taglineScriptFile` is dynamically imported and its default export is invoked while preparing banner/help output. Even as an opt-in local setting, a compromised config or script path can run code before ordinary CLI actions.
  - confidence: 0.84
- **[medium] Timeout cannot interrupt synchronous script work:** `src/cli/tagline.ts:299`
  - body: The `Promise.race` timeout cannot interrupt module evaluation or a synchronous default export on the main event loop, leaving a local denial-of-service path for every CLI startup that resolves the tagline.
  - confidence: 0.86
