## Summary

Keep open: current main still does not implement script taglines, but this PR is not ready because it adds config-driven startup JavaScript, lacks captured real behavior proof, and is stale against the current shared startup/help contracts.

## What This Changes

This PR adds opt-in `cli.banner.taglineMode: "script"` and `taglineScriptFile` support, dynamic JS tagline resolution, startup/help cache priming, docs, config schema/help updates, and banner/tagline tests.

## Best Possible Solution

If maintainers accept core support for script taglines, rebase it through the current startup presentation and help/precomputed-help contracts with bounded execution semantics, focused tests, docs, and redacted real CLI proof.

## Reproduction Assessment

Yes, by source inspection: current main lacks the `script` enum/config key, and the PR's branch adds it in a way that conflicts with current startup/help integration. I did not run a live CLI repro because this is a read-only review.

## Solution Assessment

No, not as currently proposed. The safer path is a rebased implementation through `applyCliExecutionStartupPresentation` and the help/precomputed-help contracts after explicit approval for startup JavaScript execution.
