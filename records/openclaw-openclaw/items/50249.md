## Summary

Keep open. Current main still lacks script tagline mode, the earlier related PR #41147 was closed unmerged, and this PR remains the live implementation candidate rather than cleanup-closeable. The branch is also not ready as-is because it is not mergeable against current main, adds config-selected JavaScript execution during CLI startup/help rendering, and has concrete generated-schema and timeout-boundary issues to resolve.

## What This Changes

The PR adds an opt-in `script` value for `cli.banner.taglineMode`, a `taglineScriptFile` config key, dynamic JS default-export loading for banner taglines, async banner startup plumbing, tests, and docs.

## Best Possible Solution

Keep the PR open for maintainer review. If maintainers want script-backed taglines, rebase it onto the current `applyCliExecutionStartupPresentation` and help/startup policy surfaces, update generated config schema/help/docs/changelog together, and run the script only in paths that intentionally render banner/help output with an abortable execution boundary. If arbitrary startup code execution is not acceptable, close with that product rationale and consider a safer static custom-tagline config.

## Reproduction Assessment

Yes. The current-main source path is enough to reproduce the missing feature: `taglineMode` accepts only `random/default/off`, `cli.banner` is strict with no `taglineScriptFile`, and banner formatting has no script-resolution path.

## Solution Assessment

No as currently proposed. The direction may be acceptable after maintainer approval, but the present patch is stale, leaves generated schema drift, and relies on main-event-loop execution for arbitrary user JavaScript where the timeout cannot stop synchronous blocking work.

## Review Findings

Full review comments:

  - body: This adds `taglineScriptFile` to the Zod schema and docs, but the generated base schema used by `openclaw config schema` and UI metadata is not updated. Those consumers would still expose only `random/default/off` and omit the new field, and the schema generation check should fail.
  - confidence: 0.86
- **[P2] Run tagline scripts in an abortable boundary:** `src/cli/tagline.ts:276-305`
  - body: The advertised timeout races code that still runs on the main event loop. A script with blocking top-level work or a hung sync export, including the documented `execSync` style, prevents the timer from firing and can hang every CLI startup/help path that resolves the tagline.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR adds a new opt-in startup code-execution path from user config and needs explicit maintainer security/product acceptance before merge.

Concerns:

- **[medium] Config-selected JavaScript runs at CLI startup:** `src/cli/tagline.ts:277`
  - body: `taglineScriptFile` is dynamically imported and invoked for banner/help rendering. Even if intended for local operator control, this broadens startup code execution to any process that reads the config, so a compromised or unexpected config edit could execute code before ordinary CLI actions.
  - confidence: 0.82
- **[medium] Timeout does not stop synchronous script work:** `src/cli/tagline.ts:276`
  - body: The `Promise.race` timeout cannot interrupt synchronous top-level module evaluation or a blocking default-export function on the main event loop, leaving a straightforward local denial-of-service path for normal CLI startup.
  - confidence: 0.84
