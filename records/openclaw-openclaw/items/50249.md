## Summary

Keep open because current main has not implemented script-backed CLI taglines, but the PR should not merge as-is: it adds a new config-controlled startup code execution surface and its async banner contract needs to be rebased onto the current startup path.

## What This Changes

The PR adds a `script` CLI banner tagline mode, a `taglineScriptFile` config field, async JS-file tagline loading, docs, schema/type updates, and banner/tagline tests.

## Best Possible Solution

If maintainers want this capability in core, rebase it onto the current `applyCliExecutionStartupPresentation` flow, await script resolution consistently, and document the security boundary for config-controlled startup JS.

## Reproduction Assessment

Yes for the blocking integration issue by source inspection: the PR makes `emitCliBanner` async, while current main has a central caller that still invokes it synchronously. The requested script-tagline capability itself is a feature, not a bug reproduction.

## Solution Assessment

No for the current diff. The direction may be acceptable after maintainer approval, but the branch needs a rebase onto the current startup presentation helper and a clear security decision before it is the maintainable merge shape.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0.8

Full review comments:

- **[P2] Await async banner emission in the current startup path:** `src/cli/banner.ts:170`
  - body: This PR makes `emitCliBanner` async so script taglines resolve before rendering, but current main's shared startup presentation helper still calls `emitCliBanner(...)` without `await`. After rebase, normal commands can continue before the script tagline resolves, causing missing or interleaved banner output for the new mode; update that central caller when changing the function contract.
  - confidence: 0.82
- **[P3] Split the script examples into valid modules:** `docs/gateway/configuration-reference.md:2969-2973`
  - body: The configuration reference puts a string default export and a function default export in the same `tagline.js` block. Copying that example as-is creates two default exports and fails before the tagline can load; split the examples into separate blocks or mark them as alternatives.
  - confidence: 0.88
