## Summary

## What This Changes

The PR adds opt-in `cli.banner.taglineMode: "script"` and `taglineScriptFile` config to dynamically import a JS default export for CLI banner taglines, with schema/help/docs updates and tests.

## Best Possible Solution

Define a maintainer-approved bounded script-execution contract, wire it through shared startup/help rendering, then land with focused config/banner tests and redacted real CLI proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Await async banner emission from shared startup:** `src/cli/banner.ts:170`
  - body: This makes `emitCliBanner` async, but current main's shared startup helper still calls it without `await`. After rebase, script-mode startup can continue before `resolveScriptTagline` finishes, so banner/help output can race command output instead of reliably using the configured tagline.
  - confidence: 0.87
- **[P2] Route script taglines through help rendering:** `src/cli/banner.ts:85-88`
  - body: The cache is only read when the resolved mode is `script`, but current help rendering calls `formatCliBannerLine(..., { mode: "default" })`. Non-TTY or subcommand help paths that rely on priming still render the default tagline instead of the configured script tagline.
  - confidence: 0.84
