## Summary

Keep open: current main still lacks script tagline support, but this PR is not merge-ready because it introduces startup JavaScript execution, is stale against current startup/help paths, and still lacks captured real behavior proof.

## What This Changes

The PR adds opt-in `cli.banner.taglineMode: "script"` and `taglineScriptFile` support, dynamic JS tagline resolution, config schema/help/docs updates, startup integration, and banner/tagline tests.

## Best Possible Solution

If maintainers accept core support for script taglines, rebase it through the current shared startup and help/precomputed-help contracts with an explicitly approved bounded-execution model, focused tests, docs, and redacted real CLI proof.

## Reproduction Assessment

Yes, by source inspection: current main lacks the `script` enum/config key, and the PR head’s async banner path conflicts with current startup/help integration. I did not run a live CLI repro because this is a read-only review.

## Solution Assessment

No, not as currently proposed. The safer path is a rebased implementation through `applyCliExecutionStartupPresentation`, gateway fast-path startup, root/subcommand help rendering, and precomputed-help behavior after explicit approval for startup JavaScript execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Await banner emission from current startup paths:** `src/cli/banner.ts:170`
  - body: This makes `emitCliBanner` async, but current main routes startup through helpers that still call banner emission without `await`. After rebase, script tagline resolution can race command startup/help rendering, so configured taglines may be omitted or written late.
  - confidence: 0.87
- **[P2] Resolve script taglines through the help contract:** `src/cli/banner.ts:84-89`
  - body: The formatter only consumes `options.resolvedTagline` or a module cache, while current help paths can force the default tagline or print precomputed root help. Rebased help output can therefore ignore `taglineMode: "script"` instead of rendering the configured script tagline.
  - confidence: 0.84
