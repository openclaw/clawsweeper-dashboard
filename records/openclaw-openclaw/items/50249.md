## Summary

Keep open. Current main still only supports `random`, `default`, and `off`, but this PR is not merge-ready because it is conflicting with the current CLI startup/help architecture, has blocking integration issues, lacks convincing after-fix real behavior proof, and adds config-driven JS execution that needs explicit maintainer review.

## What This Changes

The PR adds a CLI banner `script` tagline mode, a `taglineScriptFile` config key, docs, and tests for resolving a user-provided JS module as the banner tagline.

## Best Possible Solution

Rework the feature on current main, integrate it through the shared CLI startup and help-rendering paths, document the code-execution trust boundary and timeout semantics, and require convincing redacted real CLI output before merge.

## Reproduction Assessment

Yes, by source inspection. Current main lacks the `script` enum/config key, and the PR's rebased behavior would currently miss the shared startup await path and the help renderer's hardcoded default tagline mode.

## Solution Assessment

No, not as currently proposed. The useful idea needs a current-main rebase plus a design decision for arbitrary startup JS execution, and the implementation should flow through the central startup/help contracts rather than patching older entry paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Await banner emission from the shared startup path:** `src/cli/banner.ts:170`
  - body: On current main, `applyCliExecutionStartupPresentation` calls `emitCliBanner` without awaiting it. After this line makes banner emission async, script-mode startup can continue into command bootstrap/output before the tagline script resolves, so the banner/cache may race or interleave unless the shared startup helper is updated during rebase.
  - confidence: 0.88

The PR's documented verification path will keep showing the default tagline unless the help renderer is updated and covered.
  - confidence: 0.9
