## Summary

Keep open. Current main still lacks script tagline support, but this PR is not safe to merge as-is because it adds a startup JavaScript execution surface, lacks after-fix real behavior proof, and is stale against current startup/help presentation paths.

## What This Changes

The PR adds a CLI banner `script` tagline mode that loads a user-configured JavaScript file, plus config schema/help/docs/tests and startup/help integration changes.

## Best Possible Solution

Approve or reject the startup script personalization surface explicitly; an accepted version should rebase onto current startup/help helpers, bound script execution semantics, update schema/help/docs/tests, and include redacted real CLI proof.

## Reproduction Assessment

Yes, by source inspection. Current main lacks the `script` enum/config key, while the PR head's async script-resolution path needs rework for current startup and help contracts.

## Solution Assessment

No, not as currently proposed. The narrow maintainable path is a rebased implementation through `applyCliExecutionStartupPresentation` and help rendering, with explicit approval for startup JavaScript execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Await async banner emission after rebase:** `src/cli/banner.ts:170`
  - body: This changes `emitCliBanner` to return a Promise, but current main's shared startup helper and gateway fast path still call it synchronously. After rebasing, script-mode startup can race command bootstrap/output or miss cache priming unless those callers await the new async boundary.
  - confidence: 0.88
- **[P2] Resolve script taglines for all help renderers:** `src/cli/banner.ts:85-88`
  - body: `formatCliBannerLine` only reads `cachedScriptTagline` when script mode is configured. Help paths that render without first calling `emitCliBanner`, such as subcommand help, can still show an empty tagline despite `taglineScriptFile` being set.
  - confidence: 0.86
