## Summary

Keep open: current main still does not implement script-backed CLI taglines, but this PR is not ready to merge because it is stale against the current startup/help paths, lacks convincing real behavior proof, and introduces an arbitrary startup JS execution surface that needs maintainer approval.

## What This Changes

The PR adds a `script` value for `cli.banner.taglineMode`, a `taglineScriptFile` config field, dynamic JS module resolution for banner taglines, related startup/help wiring, docs, and tests.

## Best Possible Solution

Rework the feature on current main, if maintainers approve the startup JS surface, so script taglines run only through the intended banner paths with clear timeout/suppression semantics, aligned schema/docs/tests, and redacted real CLI output proof.

## Reproduction Assessment

Yes, by source inspection for the review findings. Current main lacks the `script` enum/config field, and the PR's async banner/help changes miss current shared startup and configless help contracts.

## Solution Assessment

No, not as currently proposed. The useful idea needs a current-main rebase plus maintainer approval for arbitrary startup JS execution, and the implementation should go through the central startup/help contracts rather than older entry paths.
