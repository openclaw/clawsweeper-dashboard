## Summary

Keep this PR open for maintainer judgment, but it is not merge-ready: the feature is still absent from current main, the branch’s async banner/cache approach does not fit current shared startup and help rendering paths, and startup JavaScript execution needs explicit product/security approval.

## What This Changes

The PR adds an opt-in `cli.banner.taglineMode: "script"` mode, `taglineScriptFile` config, docs, schema/help updates, and banner/tagline tests for loading a JS default export as the CLI banner tagline.

## Best Possible Solution

If maintainers accept core script taglines, rebase through the current shared startup and help/precomputed-help contracts with an approved bounded-execution model, focused tests/docs, and redacted real CLI proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is an opt-in feature/config request. Source inspection confirms current main only supports `random`, `default`, and `off`, and the PR’s current approach needs integration with newer startup/help paths.

## Solution Assessment

No, not as currently proposed. The safer path is a maintainer-approved startup-JavaScript contract implemented through `applyCliExecutionStartupPresentation`, route-first startup, root/subcommand help, and precomputed-help behavior rather than a cache-only banner path.
