## Summary

Keep open. Current main still does not implement script-backed CLI taglines, but this PR is not ready to merge: it is dirty against current main, leaves generated config-schema consumers stale, and introduces config-selected JavaScript execution at CLI startup that needs explicit maintainer security/product acceptance.

## What This Changes

The PR adds an opt-in `script` mode for `cli.banner.taglineMode`, a `taglineScriptFile` config key, dynamic JS default-export loading for banner taglines, async banner/help startup plumbing, tests, and docs/schema metadata.

## Best Possible Solution

Keep the PR open for maintainer review. If maintainers want script-backed taglines, rebase it onto the current startup-presentation policy, regenerate the public config schema, add a changelog entry, and run user scripts only in an abortable worker or child-process boundary; if arbitrary startup code is not acceptable, prefer a safer static custom-tagline config.

## Reproduction Assessment

Yes. Current main source is enough to reproduce the missing feature: the parser, TypeScript type, strict Zod schema, generated schema, and docs all expose only `random`, `default`, and `off`.

## Solution Assessment

No as currently proposed. The direction may be viable after maintainer approval, but the present patch is stale and uses same-event-loop execution for arbitrary user JavaScript where the documented timeout cannot stop synchronous blocking work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Regenerate the public config schema:** `src/config/zod-schema.ts:334-340`
  - body: This updates the handwritten Zod schema/help for `taglineScriptFile`, but the generated base schema served by config-schema consumers is not updated. `openclaw config schema` and UI metadata would still expose only `random/default/off` and omit the new field, and the schema generation check should fail.
  - confidence: 0.86
- **[P2] Run tagline scripts in an abortable boundary:** `src/cli/tagline.ts:276-305`
  - body: The timeout races code that still runs on the main Node event loop. A script with blocking top-level work or a hung sync export, including the documented `execSync` style, prevents the timer from firing and can hang every CLI startup/help path that awaits banner priming.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces a new config-selected JavaScript execution path during CLI startup and needs explicit security/product acceptance before merge.

Concerns:

- **[medium] Config-selected JavaScript runs at CLI startup:** `src/cli/tagline.ts:284`
  - body: `taglineScriptFile` is dynamically imported and its default export is invoked for banner/help rendering. Even if intended for local operator control, a compromised or unexpected config edit could execute code before ordinary CLI actions.
  - confidence: 0.82
- **[medium] Timeout cannot stop synchronous script work:** `src/cli/tagline.ts:299`
  - body: The `Promise.race` timeout cannot interrupt synchronous module evaluation or a blocking default-export function on the main event loop, leaving a straightforward local denial-of-service path for normal CLI startup.
  - confidence: 0.84
