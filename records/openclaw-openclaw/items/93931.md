## Summary

Keep open: current main and v2026.6.8 still leave the exact bare `LLM request failed.` wrapper outside the shared transient failover classifier, and open linked PRs already claim the fix path.

## What This Changes

The issue asks OpenClaw to classify bare generic `LLM request failed.` cron/model failures as fallback-worthy so configured `payload.fallbacks` engage when a local primary model is unavailable.

## Best Possible Solution

Land one canonical exact-match shared failover-classifier fix with focused regression coverage and, if feasible, live cron proof; avoid adding a user-configured error-string allowlist for this existing fallback contract.

## Reproduction Assessment

Yes, at source level: current main still defines the exact generic wrapper, the shared failover matcher does not classify it, and cron retry/fallback depends on that classification. I did not run a live LM Studio unloaded-model cron job in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to classify the exact bare wrapper in the shared failover classifier with a regression test while preserving the schema-rejection variant as non-transient. A new error-string allowlist would add avoidable config surface for an existing fallback contract.
