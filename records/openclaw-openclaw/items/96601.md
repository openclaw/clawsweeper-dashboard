The PR updates cron failure-alert SQLite decoding and adds codec round-trip tests so `failureAlert: {}` survives reload as an enabled-with-defaults alert config.

## Best Possible Solution

Land the narrow codec fix with the focused round-trip coverage, then close https://github.com/openclaw/openclaw/issues/96589 after merge if CI stays green.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: encode `failureAlert: {}` through `bindFailureAlertColumns`, decode it through current-main `failureAlertFromRow`, and observe it becomes `undefined` instead of `{}`.

## Solution Assessment

Is this the best way to solve the issue? Yes: the write path already stores `failure_alert_disabled === 0` as the enabled sentinel, so preserving that sentinel in the decoder is the narrowest maintainable fix without adding runtime fallback logic or config surface.
