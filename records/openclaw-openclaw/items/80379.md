## Summary

Keep open. Current main still redacts text-bearing tool results before session persistence and still forces Control UI tool-payload redaction, so this is a real source-level conflict between local-provider cache stability and OpenClaw's current secret-redaction security contract rather than a cleanup-close candidate.

## What This Changes

The issue asks OpenClaw to avoid or gate secret-redaction mutation of model-facing tool history for trusted local LLM providers so prefix/KV cache reuse remains stable.

## Best Possible Solution

Define a provider-aware redaction contract that preserves stable model-facing bytes for trusted local cache providers while keeping persisted, UI, diagnostic, and support secret-protection boundaries explicit and tested.

## Reproduction Assessment

Yes, at source level: toolResult writes pass through transcript redaction before session persistence, and the default patterns match the reported ENV-style TOKEN/KEY assignments. I did not run a live local LLM KV-cache trace.

## Solution Assessment

Unclear. A blanket local-provider bypass or new toggle would likely preserve cache bytes, but the safer solution needs maintainer-owned security/product policy or a separate model-facing copy so UI, diagnostics, support, and persisted transcripts do not silently lose redaction.
