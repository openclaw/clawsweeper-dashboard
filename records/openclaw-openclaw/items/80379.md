## Summary

Keep open. Current main still redacts persisted transcript text before session writes and still forces tool-payload redaction for UI/safety surfaces, so the reported local-provider cache impact remains valid and the fix needs maintainer security/product judgment rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop secret-redaction from changing model-facing persisted tool-result bytes for trusted local LLM providers because that breaks KV-cache prefix reuse.

## Best Possible Solution

Define a provider-aware redaction/trust-boundary contract, or maintain a separate model-facing prompt copy, so trusted local-provider bytes stay cache-stable while persisted transcripts, Control UI, diagnostics, support exports, approval prompts, and logs remain redacted.

## Reproduction Assessment

Yes at source level: append a toolResult containing an ENV-style TOKEN/KEY assignment through guardSessionManager and the before-write hook will redact the stored text before the next prompt is built from sessionManager state. I did not run a live local LLM KV-cache trace in this read-only review.

## Solution Assessment

Unclear. The suggested provider/default toggle would preserve local prompt bytes, but the safer maintainable direction is a maintainer-owned trust-boundary design or separate model-facing copy that does not weaken UI, transcript, diagnostic, support, approval, or log redaction.
