## Summary

Keep open: current main still has a source-reproducible LM Studio auth-resolution gap for arbitrary `${ENV}` API-key templates, and the linked auth-fix PR remains open. The native `/api/v1/chat` or endpoint-fallback request is separate provider/product scope, not something already implemented on main.

## What This Changes

The issue asks OpenClaw to resolve LM Studio env-backed API keys and avoid failing on LM Studio endpoint differences during inference.

## Best Possible Solution

Land the narrow SecretInput auth fix after real LM Studio proof, then split native `/api/v1/chat` fallback into a separate provider/API decision if maintainers want that behavior.

## Reproduction Assessment

Partly: source inspection gives a high-confidence reproduction path for the documented `${ENV}` auth-template bug in `models.providers.lmstudio.apiKey`. I did not reproduce the endpoint-mismatch half against a live LM Studio server, and bare `$VAR` syntax is not the current documented config contract.

## Solution Assessment

Unclear as one combined issue. Reusing SecretInput resolution in the LM Studio plugin is the narrow fix for the auth bug, while native endpoint fallback should be a separate provider/API scope decision rather than bundled into the same repair.
