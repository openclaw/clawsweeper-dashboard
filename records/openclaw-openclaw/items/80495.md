## Summary

Keep open. Source inspection supports part of the report: LM Studio config auth only resolves documented/known env markers, so the reported bare `$LMSTUDIO_API_KEY` value would not be resolved as an environment reference; the endpoint request also overlaps a previously closed native-API scope decision, so this needs maintainer triage rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to expand LM Studio API-key environment variables and avoid failing on LM Studio endpoint compatibility by using native or fallback endpoints.

## Best Possible Solution

Split the response into a narrow LM Studio auth/config validation decision and a provider-boundary decision: improve handling or diagnostics for unsupported env placeholders, and use the existing `openai-responses` path or a ClawHub/native plugin unless maintainers explicitly want bundled native fallback.

## Reproduction Assessment

No high-confidence full reproduction path was established. Source inspection shows the bare `$LMSTUDIO_API_KEY` auth value would not be expanded, but the LM Studio endpoint failure still needs a live server or captured request/response proof against current main.

## Solution Assessment

Unclear. Supporting native `/api/v1/chat` or endpoint fallback in bundled OpenClaw is not clearly the narrowest fix; a safer first step is to validate/error on unsupported env placeholder syntax and guide users toward documented `${LM_API_TOKEN}` or existing `api: "openai-responses"` configuration.
