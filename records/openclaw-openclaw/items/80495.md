## Summary

Keep open: current `main` still has source evidence for the LM Studio env-template API-key bug, and the open follow-up PR addresses only that auth-resolution half while the native endpoint/fallback request is separate provider-scope work.

## What This Changes

The issue asks OpenClaw to expand LM Studio API-key environment templates and avoid LM Studio inference failures caused by the current OpenAI-compatible chat-completions endpoint choice.

## Best Possible Solution

Land or revise the focused auth-resolution fix for arbitrary SecretInput env templates, then keep native LM Studio inference or endpoint fallback in a separate provider-plugin/API discussion if maintainers want that behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the auth half by source inspection: current `main` treats non-marker configured API-key strings as literal token values, and tests only cover the default `LM_API_TOKEN` marker. No for the endpoint half, which was not live-tested against LM Studio.

## Solution Assessment

Is this the best way to solve the issue? No as one bundled change; the env-template auth bug has a narrow fix path, while `/api/v1/chat` inference or endpoint fallback is a separate provider/API scope decision.
