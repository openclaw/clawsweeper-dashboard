## Summary

Keep open for maintainer triage. Source inspection supports the literal `$LMSTUDIO_API_KEY` auth failure path, but the documented OpenClaw syntax is `LM_API_TOKEN` / `${LM_API_TOKEN}` or SecretRef, and the native endpoint/fallback request is broader provider behavior rather than a narrow bug fix.

## What This Changes

Request support for LM Studio API key environment-variable expansion and alternative/native endpoint fallback when LM Studio rejects `/v1/chat/completions`.

## Best Possible Solution

A maintainer should decide whether to improve diagnostics/docs for unsupported `$VAR` config strings, add an explicit `LMSTUDIO_API_KEY` compatibility alias, or keep native `/api/v1/chat` work in the external provider-plugin path.

## Reproduction Assessment

Yes for the auth half by source inspection: a bare `$LMSTUDIO_API_KEY` string is not normalized as an env reference and would be used literally. No live reproduction was established for the LM Studio endpoint mismatch because the report includes no request/response logs and upstream docs still list `/v1/chat/completions` as supported.

## Solution Assessment

Unclear. Supporting documented `LM_API_TOKEN`/SecretRef syntax is already the narrow path, while automatic endpoint fallback or native `/api/v1/chat` inference is a new provider behavior that should be decided deliberately.
