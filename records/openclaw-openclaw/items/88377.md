- `impact:security`: The issue is about preventing sensitive or high-risk memory corpora from being exposed to live agents at the tool-policy boundary.

## Summary

Keep open. Current main has per-agent memory source config and filters unsupported session results to empty, but `memory_search` still advertises and accepts `corpus=sessions` at the schema/tool surface, so the requested policy-level hiding/rejection is not implemented; the linked liveness issue covers a related but separate fail-open timeout problem.

## What This Changes

Request an official per-agent/runtime policy that limits which `memory_search` corpora are exposed and accepted, especially allowing `memory` while hiding or rejecting `sessions`.

## Best Possible Solution

Define a clear corpus exposure contract for `memory_search`: build the per-agent tool schema and execute-time validation from resolved runtime policy, default to no session corpus unless explicitly enabled, and keep the liveness timeout/fail-open work tracked separately.

## Reproduction Assessment

Yes for the schema/policy gap: current source shows `memory_search` advertises and accepts `corpus=sessions` even though resolved memory sources default to `memory`. No live Telegram reproduction was run for the reported stall.

## Solution Assessment

Unclear until maintainers choose the policy shape. Reusing `memorySearch.sources` as the exposure policy may be the narrowest path, but a separate runtime corpus allowlist could be safer if indexing scope and tool exposure must remain independent.
