## Summary

Keep open. Current main still falls back from failed implicit local `logs.tail` calls to the configured file log, and neither the CLI output nor docs show the requested Linux/systemd journal preference or service provenance.

## What This Changes

Make `openclaw logs --follow` prefer the active Linux systemd gateway journal after side-by-side cutover and print CLI/service/log-source provenance before tailing.

## Best Possible Solution

Add a Linux-only implicit-local fallback that detects the active managed gateway service, tails its journal by PID when RPC cannot provide logs, prints concise redacted provenance, warns on mismatches, and keeps remote RPC plus Control UI file-log behavior unchanged.

## Reproduction Assessment

Yes. Current source and tests reproduce the root path: implicit local pairing/close/timeout failures call `readConfiguredLogTail`, while a full live reproduction requires the reporter's Linux/systemd side-by-side cutover setup.

## Solution Assessment

Yes. The requested service-aware Linux fallback is the narrowest maintainable direction if it is limited to implicit local failures and leaves the `logs.tail` RPC contract unchanged.
