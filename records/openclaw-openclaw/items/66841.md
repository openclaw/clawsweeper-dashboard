## Summary

Keep open. Current main still routes `openclaw logs` through `logs.tail` and, for implicit local loopback failures, falls back to the configured file log; the requested Linux/systemd journal preference and provenance output are not implemented.

## What This Changes

The issue requests that Linux `openclaw logs --follow` prefer the active `openclaw-gateway.service` journal after side-by-side cutover and print CLI, service, and log-source provenance before tailing.

## Best Possible Solution

Add a Linux-only, implicit-local fallback path that detects the active managed `openclaw-gateway.service`, tails the service journal by PID when RPC cannot provide logs, prints concise redacted provenance before following, warns on mismatched CLI/service/source state, and leaves remote RPC plus Control UI file-log behavior unchanged.

## Reproduction Assessment

Yes. Source and tests give a high-confidence reproduction of the root behavior: local loopback pairing/close/timeout paths call `readConfiguredLogTail` and tests assert the file-log fallback; a full live repro requires the reporter's side-by-side systemd cutover setup.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix if it stays Linux-only and service-aware for implicit local failures, with file tailing kept as an explicit fallback rather than changing the `logs.tail` RPC contract.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/66841. Observable bug: after a Linux side-by-side service cutover, `openclaw logs --follow` can fall back to the configured rolling file log from the CLI/install context and show stale version-mismatch warnings instead of the active gateway service logs. Implement a Linux-only implicit-local fallback: when `logs.tail` cannot be used because local loopback RPC is pairing-required, closed, or timed out, detect the active managed `openclaw-gateway.service`, read redacted service provenance (CLI version, service ExecStart, service PID, service/version when safely available, exact log source), and prefer `journalctl --user _PID=<pid>` for `--follow` when the active PID is available. Keep explicit `--url` remote behavior and Control UI/`logs.tail` file-log behavior unchanged. If systemd service-aware access is unavailable, keep the configured file-log fallback but print it as an explicit fallback and warn when the source may not represent the live service. Reuse existing daemon systemd helpers where possible; do not execute arbitrary service commands or print environment secrets. Update focused CLI/systemd tests and docs. Add a user-facing `CHANGELOG.md` fix entry crediting the reporter, not bot/maintainer handles.
