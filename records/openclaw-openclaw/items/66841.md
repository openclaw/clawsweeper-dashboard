## Summary

Keep open. Current main and the latest stable release still fall back from implicit local `logs.tail` failures to the configured file log, and the requested Linux/systemd journal source plus CLI/service/log-source provenance is not implemented or owned by an existing merged fix.

## What This Changes

The issue requests that `openclaw logs --follow` prefer the active Linux systemd gateway journal and print CLI, service, PID, source, and mismatch provenance before tailing logs.

## Best Possible Solution

Add a Linux-only implicit-local fallback that detects the active managed gateway service, tails its user journal by PID when RPC cannot provide logs, prints concise redacted provenance and mismatch warnings, and leaves explicit remote RPC plus Control UI file-log behavior unchanged.

## Reproduction Assessment

Yes at source level: current main and its tests show implicit local pairing/close/timeout failures still call `readConfiguredLogTail`. A full live reproduction needs a Linux/systemd side-by-side cutover host like the reporter used.

## Solution Assessment

Yes. The narrow maintainable fix is a Linux-only, service-aware implicit-local fallback with redacted provenance while preserving the `logs.tail` RPC contract, explicit `--url` behavior, and current follow retry handling.
