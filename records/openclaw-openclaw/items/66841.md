## Summary

Keep this open. Current main still routes `openclaw logs` through `logs.tail` and a configured file-log fallback for implicit local loopback failures, so the side-by-side Linux/systemd cutover problem remains reproducible from source and tests; the reporter's prototype is useful implementation evidence, but it is not present on main and no open PR with a direct reference to this issue was found.

## What This Changes

The issue requests that `openclaw logs --follow` prefer the active Linux systemd gateway journal and print CLI/service/log-source provenance instead of silently tailing stale configured file logs after side-by-side cutover.

## Best Possible Solution

Add a Linux-only local CLI path that detects the active managed `openclaw-gateway.service`, uses the active service journal by PID for `--follow` when implicit local RPC cannot provide logs, prints concise provenance before tailing, warns on version/source mismatches, and keeps remote RPC plus Control UI file-log behavior unchanged.

## Reproduction Assessment

Yes. Source and tests give a high-confidence reproduction of the root behavior: local loopback pairing/close/timeout paths call `readConfiguredLogTail` and print file-log fallback output rather than active systemd journal provenance; a full live repro would require the reporter's Linux side-by-side service cutover setup.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix if it stays Linux-only and implicit-local: prefer the active systemd journal when available, then make any remaining file fallback explicit. The safer implementation detail is to reuse existing daemon service helpers and avoid executing arbitrary service commands to discover metadata.

## Repair Work Prompt

Fix the Linux/systemd `openclaw logs --follow` stale file-tail problem tracked by https://github.com/openclaw/openclaw/issues/66841. Implement a narrow CLI-side change: for implicit local Gateway access on Linux, when `logs.tail` cannot be used because local loopback RPC is unavailable/pairing/scope/close/timeout, inspect the active managed systemd gateway service using existing daemon helpers, prefer `journalctl --user _PID=<active-pid>` for follow mode when the service is running, and print provenance before tailing. Provenance should include CLI version, active service ExecStart, active service PID, service version when safely available, exact log source, and a mismatch/fallback warning when source or versions may not represent the active service. Do not execute arbitrary service commands solely to obtain version metadata, and do not print service environment values or secrets. Preserve explicit `--url` behavior, remote RPC behavior, Gateway `logs.tail`, and Control UI file-log behavior. If journal access is unavailable, keep the configured file-tail fallback but make that fallback explicit. Add focused tests around pairing/scope/close fallback choosing journal when systemd runtime is active, falling back to file logs when not available, preserving explicit `--url`, and preserving JSON output deliberately. Update `docs/cli/logs.md`, `docs/logging.md`, and add a user-facing `CHANGELOG.md` fix entry crediting `Thanks @UGDev-Chane`; do not credit bot or maintainer handles.
