## Summary

Keep open. Current main still uses the configured file-log fallback for failed implicit local `logs.tail` calls and does not show the requested Linux/systemd journal preference or service provenance; the reporter's prototype remains outside main and no linked implementation PR closes this issue.

## What This Changes

Request that `openclaw logs --follow` on Linux prefer the active systemd gateway journal and print CLI/service/log-source provenance instead of silently using a potentially stale file-log fallback.

## Best Possible Solution

Add a Linux-only implicit-local fallback that detects the active managed gateway service, tails its user journal by PID when RPC cannot provide logs, prints concise redacted provenance and mismatch warnings, and leaves explicit remote RPC plus Control UI file-log behavior unchanged.

## Reproduction Assessment

Yes, at source level: current main and its tests show implicit local pairing/close/timeout failures call `readConfiguredLogTail`. A full live reproduction still requires a Linux/systemd side-by-side cutover host like the reporter used.

## Solution Assessment

Yes. The narrowest maintainable fix is a Linux-only, service-aware implicit-local fallback with redacted provenance, while preserving the `logs.tail` RPC contract and explicit remote URL behavior.

## Repair Work Prompt

Fix the stale/misleading `openclaw logs --follow` behavior after Linux systemd side-by-side cutover. Limit the change to implicit local loopback failures where `logs.tail` cannot be used because pairing, close, timeout, or scope-upgrade handling would currently fall back to `readConfiguredLogTail`; do not change explicit `--url` targets, the Control UI `logs.tail` RPC contract, or unrelated logs-follow streaming work. On Linux, use existing systemd helpers or a small shared helper to detect the active managed gateway unit, read redacted service provenance, and when a running MainPID is available tail `journalctl --user _PID=<pid>` as the authoritative live source for `--follow`. Before tailing, print concise provenance including CLI version, redacted service ExecStart, service PID, service version when safely available, exact log source, and mismatch warnings; if journal access is unavailable, fall back to the configured file log with an explicit warning that it may not represent the active service. Add focused tests for pairing/close/timeout fallback choosing journal vs file fallback, provenance redaction, and docs updates. Add a user-facing changelog entry. Stop and ask for maintainer direction if the fix would require a new config option, a new gateway protocol, or broad streaming refactor.
