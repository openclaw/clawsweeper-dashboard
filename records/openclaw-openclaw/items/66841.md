## Summary

Keep open. Current main still uses the configured file-log fallback after implicit local `logs.tail` failures, and the CLI/docs do not implement the requested Linux systemd journal preference or service provenance output.

## What This Changes

The issue requests that `openclaw logs --follow` prefer the active Linux systemd gateway journal after side-by-side cutover and print clear CLI/service/log-source provenance before tailing.

## Best Possible Solution

Add a Linux-only implicit-local fallback that detects the active managed gateway service, tails its journal by PID or unit when RPC cannot provide logs, prints concise redacted provenance and mismatch warnings, and keeps remote RPC plus Control UI file-log behavior unchanged.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: implicit local pairing/close/timeout failures in `fetchLogs` call `readConfiguredLogTail`, and tests assert that behavior; the full operational replay requires a Linux/systemd side-by-side cutover host.

## Solution Assessment

Yes. The requested service-aware Linux fallback is the narrowest maintainable fix when scoped to implicit local failures and implemented with existing systemd runtime/ExecStart seams instead of changing `logs.tail`.
