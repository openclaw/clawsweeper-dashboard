## Summary

Keep open. Current main still handles implicit local `logs.tail` failures by tailing the configured file log and printing only the fallback notice plus `Log file:`, while docs and tests still describe that behavior; the reporter’s Linux/systemd journal prototype is useful context but is not on main and no closing PR implements it.

## What This Changes

The issue asks for `openclaw logs --follow` to prefer the active Linux systemd gateway journal after side-by-side cutover and print clear CLI/service/log-source provenance.

## Best Possible Solution

Add a Linux-only implicit-local fallback that detects the active managed gateway service, tails its user journal by PID when RPC cannot provide logs, prints concise redacted provenance and mismatch warnings, and leaves explicit remote RPC plus Control UI file-log behavior unchanged.

## Reproduction Assessment

Yes at source level: current main and its tests show implicit local pairing/close/timeout failures still call `readConfiguredLogTail`. A full live reproduction requires a Linux/systemd side-by-side cutover host like the reporter used.

## Solution Assessment

Yes. The narrow maintainable fix is a Linux-only, service-aware implicit-local fallback with redacted provenance, while preserving the `logs.tail` RPC contract, explicit `--url` behavior, and current follow retry handling.

## Repair Work Prompt

Fix the stale `openclaw logs --follow` implicit-local fallback described at https://github.com/openclaw/openclaw/issues/66841. Keep the boundary narrow: on Linux/systemd, when implicit local `logs.tail` RPC fails due to pairing/close/timeout and no explicit `--url` was provided, detect the active managed gateway service, prefer `journalctl --user _PID=<pid>` for the active service when available, and print redacted provenance before tailing: CLI version, service ExecStart, service PID, service/gateway version when available, exact log source, and a mismatch warning when the CLI/service/log source do not agree. Preserve explicit remote RPC behavior, Control UI `logs.tail` file behavior, and existing `--follow` transient reconnect semantics. Keep file-log fallback explicit when service/journal access is unavailable. Add focused tests for pairing/close/timeout fallback selection and provenance output, update CLI/logging docs, and add a user-facing changelog fix entry if the repair lands. Avoid broad websocket streaming changes; coordinate with https://github.com/openclaw/openclaw/pull/56475 rather than reimplementing it.
