## Summary

Keep open: current main still rejects the legacy string-shaped `dreaming.storage` config and the available doctor repair path would quarantine the `memory-core` plugin config instead of migrating and preserving the selected storage mode.

## What This Changes

The issue requests an upgrade-safe migration or actionable recovery path for legacy `plugins.entries.memory-core.config.dreaming.storage` string values such as `"both"`.

## Best Possible Solution

Add a focused doctor compatibility migration for `memory-core` that rewrites legacy `dreaming.storage` string values (`inline`, `separate`, `both`) to the canonical object shape before invalid-plugin quarantine runs, while keeping the manifest strict and preserving the user's selected mode.

## Reproduction Assessment

Yes. A config containing `plugins.entries.memory-core.config.dreaming.storage: "both"` has a high-confidence source-level reproduction path: the manifest requires an object, plugin validation reports an invalid config, and no current migration converts the string before quarantine.

## Solution Assessment

Yes. The narrowest maintainable solution is a targeted memory-core doctor/legacy migration plus regression tests; globally relaxing plugin schema validation would be broader and more likely to hide real plugin config errors.
