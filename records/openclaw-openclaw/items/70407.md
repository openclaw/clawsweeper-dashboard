## Summary

Keep open: current main and v2026.5.18 still have the object-only `memory-core` `dreaming.storage` schema with no doctor migration for legacy string values, while https://github.com/openclaw/openclaw/pull/82552 is now the focused open fix path.

## What This Changes

Migrate legacy `plugins.entries.memory-core.config.dreaming.storage` string values or provide usable recovery so upgrades from v2026.4.15 do not break CLI commands.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82552 or an equivalent targeted doctor migration that rewrites known legacy storage strings before invalid-plugin quarantine while keeping the manifest and runtime schema strict.

## Reproduction Assessment

Yes, at source level. A config with `plugins.entries.memory-core.config.dreaming.storage: "both"` violates the current object-only manifest schema, and current doctor sequencing has no memory-core migration before generic invalid-plugin quarantine.

## Solution Assessment

Yes. A targeted `doctor --fix` migration before invalid-plugin quarantine is the narrow maintainable fix; the open PR implements that direction, so the issue should stay open until that PR or an equivalent fix lands.
