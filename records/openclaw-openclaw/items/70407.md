## Summary

Keep open: current main still has the object-only `memory-core` dreaming storage schema and no legacy migration that rewrites string values before invalid plugin config repair can quarantine the plugin config.

## What This Changes

The issue asks OpenClaw to migrate or clearly recover from legacy string-valued `plugins.entries.memory-core.config.dreaming.storage` after upgrading from v2026.4.15 to v2026.4.21.

## Best Possible Solution

Add a focused doctor legacy migration that rewrites valid legacy storage strings to `{ mode, separateReports: false }` before invalid-plugin quarantine, while keeping the manifest schema strict and adding regression coverage.

## Reproduction Assessment

Yes, at source level. A config with `plugins.entries.memory-core.config.dreaming.storage: "both"` is outside the current object-only manifest schema, and no registered legacy migration rewrites it before generic invalid-plugin repair.

## Solution Assessment

Yes. A targeted memory-core doctor migration plus sequencing/regression tests is the narrowest maintainable fix; broadly relaxing plugin schema validation would hide real plugin config errors.
