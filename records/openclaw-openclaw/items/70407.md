## Summary

Keep open: current main still rejects legacy string-valued `dreaming.storage` and has no memory-core migration that preserves the user's old storage mode; `doctor --fix` would fall through to generic invalid-plugin quarantine instead of rewriting the field.

## What This Changes

The issue requests upgrade-safe handling for legacy `plugins.entries.memory-core.config.dreaming.storage` string values after that config changed to an object shape.

## Best Possible Solution

Add a focused doctor legacy migration that rewrites valid legacy storage strings to `{ mode, separateReports: false }` before invalid-plugin quarantine, while keeping the manifest schema strict and adding regression coverage.

## Reproduction Assessment

Yes, at source level. A config with `plugins.entries.memory-core.config.dreaming.storage: "both"` is outside the current object-only manifest schema, and no registered legacy migration rewrites it before generic invalid-plugin repair.

## Solution Assessment

Yes. A targeted memory-core doctor migration plus regression tests is the narrowest maintainable fix; relaxing plugin schema validation broadly would hide real plugin config errors.
