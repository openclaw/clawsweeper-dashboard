## Summary

Keep open: current main still has the same upgrade regression. The `memory-core` manifest requires object-shaped `dreaming.storage`, no registered legacy migration rewrites legacy string values, and the reachable doctor fallback would quarantine the plugin config instead of preserving the user's storage mode.

## What This Changes

The issue requests migration or targeted recovery for legacy string values at `plugins.entries.memory-core.config.dreaming.storage` after upgrading to v2026.4.21.

## Best Possible Solution

Add a focused doctor legacy migration that rewrites valid legacy storage strings to `{ mode, separateReports: false }` before invalid-plugin quarantine, while keeping the manifest schema strict.

## Reproduction Assessment

Yes, at source level. A config with `plugins.entries.memory-core.config.dreaming.storage: "both"` fails the current manifest schema, and no current migration rewrites it before the generic quarantine path.

## Solution Assessment

Yes. A targeted memory-core doctor migration plus regression coverage is the narrowest maintainable fix; relaxing plugin schema validation globally would be broader and could hide real plugin config errors.
