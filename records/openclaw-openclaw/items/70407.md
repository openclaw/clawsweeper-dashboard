## Summary

Keep open: current main still has no legacy migration for string-valued `dreaming.storage`; the newer config guard reduces the total-CLI-outage blast radius, but `doctor --fix` would still quarantine the invalid memory-core config instead of preserving the user's storage mode.

## What This Changes

The issue reports an upgrade regression where legacy string-valued `memory-core` `dreaming.storage` config blocks CLI startup after the schema changed to an object.

## Best Possible Solution

Add a focused doctor legacy migration that rewrites valid legacy storage strings to `{ mode, separateReports: false }` before invalid-plugin quarantine, while keeping the manifest schema strict and adding regression coverage.

## Reproduction Assessment

Yes, at source level. A config with `plugins.entries.memory-core.config.dreaming.storage: "both"` fails the current manifest schema, and no registered legacy migration rewrites it before generic invalid-plugin repair.

## Solution Assessment

Yes. A targeted memory-core doctor migration plus regression tests is the narrowest maintainable fix; relaxing plugin schema validation broadly would hide real plugin config errors.
