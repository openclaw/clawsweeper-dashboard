## Summary

Keep open: current main and the latest release still keep `memory-core` dreaming storage object-only, and no legacy doctor migration preserves old string values before the invalid-plugin quarantine path.

## What This Changes

The issue requests an upgrade-safe migration or actionable recovery path for legacy string-shaped `plugins.entries.memory-core.config.dreaming.storage` configs.

## Best Possible Solution

Add a focused doctor legacy migration that rewrites valid legacy storage strings to `{ mode, separateReports: false }` before invalid-plugin quarantine, while keeping the manifest schema strict.

## Reproduction Assessment

Yes, at source level. A config with `plugins.entries.memory-core.config.dreaming.storage: "both"` fails the current manifest schema, and no registered migration rewrites it before generic invalid-plugin repair.

## Solution Assessment

Yes. A targeted memory-core doctor migration plus regression coverage is the narrowest maintainable fix; broadly relaxing plugin schema validation would hide real plugin config errors.
