## Summary

Keep open: current main has atomic session-store writes, but it still treats an empty or invalid `sessions.json` as an empty store, so the recovery gap remains.

## What This Changes

Request defensive handling for a zero-byte or corrupted `sessions.json` so session listing and management recover instead of silently aggregating an empty store.

## Best Possible Solution

Add central session-store corruption detection with a safe recovery/repair path, keep the existing atomic writer, and cover the Gateway list path plus doctor diagnostics with regression tests.

## Reproduction Assessment

Yes, source-reproducible: a zero-byte store makes `loadSessionStore` parse empty text, catch the error, and return `{}`, after which `sessions.list` aggregates an empty store. I did not run a mutating temp-file repro because this review is read-only.

## Solution Assessment

No, current main is not sufficient: atomic writes reduce future corruption risk, but they do not detect or recover an already empty/invalid store. The maintainable fix belongs in the shared session-store/doctor layer, not in each caller.
