## Summary

Keep open: current main already uses atomic session-store writes, but empty or invalid `sessions.json` is still decoded as `{}`, so session listing and doctor paths can operate on an empty store with no recovery signal.

## What This Changes

The issue asks for atomic writes, corruption detection, automatic recovery or rebuild, and doctor diagnostics for empty or invalid session stores.

## Best Possible Solution

Add central session-store corruption detection with a safe repair path, keep the existing atomic writer, and prevent doctor from treating all transcripts as archiveable orphans when the index is corrupt.

## Reproduction Assessment

Yes, source-reproducible: a zero-byte store makes `loadSessionStore` parse empty text, catch the error, keep `{}`, and downstream list paths report no sessions. I did not run a mutating temp-file repro because this review is read-only.

## Solution Assessment

No, current main is not sufficient: atomic writes reduce future corruption risk, but empty/invalid reads still collapse to an empty store. The narrow maintainable fix belongs in the shared session-store and doctor layers, not in each caller.
