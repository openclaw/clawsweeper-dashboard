## Summary

Keep open. Current main and the latest release still have a source-proven false invalidation path: transient group/direct/inter-session prompt context can change `extraSystemPromptHash`, and the CLI session resolver still treats that drift as a fresh-session reset; recent comments add live v2026.5.7 evidence across Discord, subagent announce, and cross-channel DM triggers.

## What This Changes

The issue reports Claude CLI session resets when transient prompt context changes `extraSystemPromptHash`, causing group/channel and related chat flows to lose conversation memory.

## Best Possible Solution

Move transient prompt overlays out of durable CLI session identity while preserving strict invalidation for auth, MCP, model, workspace, skills, and genuinely stable prompt changes, with regression coverage for group turn 1-to-2, subagent announce, and cross-channel direct DM reuse.

## Reproduction Assessment

Yes at source level. A first group turn can include `groupIntro` in `extraSystemPromptStatic`, a follow-up can omit it, and the current resolver invalidates the stored CLI session when that hash changes; I did not run the live Discord/Telegram repro.

## Solution Assessment

Yes, the requested bug boundary is right. The maintainable fix should define a stable CLI reuse-hash input that excludes transient group/direct/inter-session overlays rather than special-casing one channel or dropping all durable identity checks.
