## Summary

Keep open. Current main still has the reported false-invalidation path: first-turn group intro text is included in the static CLI prompt hash, and the session resolver still resets on any stored/current prompt-hash drift; the related wake-up metadata PR explicitly covered a different path.

## What This Changes

The issue asks for Claude CLI session reuse to survive group-channel follow-up turns where first-turn-only group intro text changes `extraSystemPromptHash`.

## Best Possible Solution

Define a stable CLI prompt identity that excludes turn-scoped groupIntro and other per-invocation prompt context while still sending those prompts to the CLI, then add regression coverage for group follow-up and empty-static announcement paths.

## Reproduction Assessment

Yes. Source inspection on current main shows a first group turn includes `groupIntro` in `extraSystemPromptStaticParts`, later group turns omit it, and the strict hash comparator returns `system-prompt` on that drift.

## Solution Assessment

Yes. A narrow fix at the CLI reuse identity boundary is the maintainable path: preserve actual prompt delivery, but hash only durable context and tolerate missing prompt hashes where no stable identity was asserted.

## Repair Work Prompt

Fix the CLI session continuity bug tracked by https://github.com/openclaw/openclaw/issues/69118. On current main, `groupIntro` is first-turn/re-intro only but is included in `extraSystemPromptStaticParts` and hashed for CLI session reuse, so follow-up group turns can invalidate with `reason=system-prompt`. Implement a narrow fix in the prompt assembly/preparation/session reuse path so session reuse identity excludes turn-scoped prompt pieces at minimum `groupIntro`, and avoid false positives for legacy or under-populated static prompt hashes if the resolver is touched. Keep those prompt pieces in the actual CLI system prompt. Do not remove auth invalidation, and do not bundle broad MCP hash policy changes unless a maintainer expands scope. Add regression tests proving first group turn to second group follow-up reuses the same CLI session identity, and cover the empty-static announcement/direct caller case from the May 4 comment if the resolver changes. Add a user-facing `CHANGELOG.md` fix entry.
