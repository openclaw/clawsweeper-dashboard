## Summary

Keep open. Current main still has the reported false-invalidation path: groupIntro is first-turn/re-intro only, but it is still included in extraSystemPromptStatic and hashed for CLI session reuse; the resolver still resets on any extraSystemPromptHash mismatch.

## What This Changes

The issue requests fixing Claude CLI session reuse in group-style channels where first-turn-only group intro text changes the static extra system prompt hash and resets the CLI session on follow-up turns.

## Best Possible Solution

Keep the first-turn/re-intro group instructions in the actual CLI system prompt, but exclude turn-scoped groupIntro from the stable prompt identity used for CLI session reuse. Add a focused regression test that compares a first group-channel turn with the next group-channel turn and verifies the existing CLI binding is reused without a system-prompt reset.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a first group turn includes groupIntro in extraSystemPromptStatic, a follow-up group turn omits it, and resolveCliSessionReuse treats the resulting hash mismatch as invalidatedReason: "system-prompt".

## Solution Assessment

Yes. A narrow fix at the stable prompt identity boundary is the best maintainable path; dropping all prompt/MCP reuse keys would be broader policy work, while excluding first-turn-only groupIntro addresses the reported false invalidation directly.
