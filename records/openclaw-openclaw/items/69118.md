## Summary

Keep open. Current main still has the reported false-invalidation path: groupIntro is first-turn/re-intro only, but it remains in the static prompt text that is hashed for CLI session reuse, and the resolver still resets on any prompt-hash mismatch.

## What This Changes

The issue reports that Claude CLI group-channel sessions reset between consecutive turns because first-turn-only groupIntro text changes the extraSystemPromptHash used for CLI session reuse.

## Best Possible Solution

Keep the first-turn/re-intro group instructions in the actual CLI system prompt, but exclude turn-scoped groupIntro from the stable prompt identity used for CLI session reuse and add a focused group turn-transition regression test.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: first group turns include `groupIntro` in `extraSystemPromptStatic`, follow-up group turns omit it, and `resolveCliSessionReuse` returns `invalidatedReason: "system-prompt"` when the stored hash differs.

## Solution Assessment

Yes. A narrow fix at the stable prompt identity boundary is the best maintainable path for this issue; dropping all prompt/MCP reuse keys would be broader policy work beyond the reported groupIntro drift.
