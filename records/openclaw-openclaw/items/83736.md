## Summary

Keeping this open: current main and the latest release still reject ordinary protocol-3 node clients before role parsing, node registration, or maintenance RPC can come online. The issue has concrete user reports for Windows subordinate nodes and Android nodes, but the repair needs a maintainer protocol-compatibility and security-boundary decision rather than an automatic code change.

## What This Changes

The issue requests gateway tolerance or a stable maintenance path for slightly older subordinate nodes after a gateway protocol upgrade.

## Best Possible Solution

Define the supported gateway/node skew contract, then either implement a narrow authenticated degraded maintenance mode for older nodes or document a safe upgrade order with regression coverage for protocol negotiation and allowed maintenance RPCs.

## Reproduction Assessment

Yes at source level: a node/client advertising only protocol 3 against current protocol 4 reaches the PROTOCOL_MISMATCH branch before role parsing or node RPC registration. I did not run the reported Windows two-node setup, but multiple issue comments include matching Android logs.

## Solution Assessment

Unclear as an immediate patch: broadly accepting stale protocol traffic would be the wrong boundary. The maintainable solution is an explicit skew policy with a narrow degraded maintenance contract, or a documented safe upgrade order if maintainers reject compatibility mode.
