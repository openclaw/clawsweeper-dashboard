## Summary

Keeping this open: current main and the latest release still reject ordinary node clients whose protocol range does not include Gateway protocol v4 before node pairing or registration can proceed, and the related closed issues point back here as the canonical compatibility thread. The remaining work needs a maintainer decision on supported gateway/node skew and the safe degraded maintenance RPC/auth boundary.

## What This Changes

The issue asks OpenClaw Gateway to tolerate, degrade, or provide a stable maintenance path for slightly older subordinate node clients during gateway/node protocol skew.

## Best Possible Solution

Define the supported gateway/node version-skew contract, then either implement a narrow degraded mode for already-paired older nodes with explicitly limited update/restart/log/status RPCs, or document a safe upgrade order if maintainers decide not to support skew.

## Reproduction Assessment

Yes for a source-level reproduction: a non-probe node advertising only protocol 3 against current protocol 4 reaches the `PROTOCOL_MISMATCH` close path before node registration. I did not run the reported Windows gateway plus subordinate-node setup in this read-only review.

## Solution Assessment

Unclear as an implementation direction: accepting old protocol traffic wholesale is not the best fix. The maintainable solution needs an explicit degraded maintenance contract with tests for protocol negotiation, node status/listing, and the limited RPCs older nodes may use.
