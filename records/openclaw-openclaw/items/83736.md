## Summary

Keep open: current main and v2026.6.8 still require normal Gateway clients to advertise protocol 4, and the only merged adjacent fix covers same-install local node-host restarts while explicitly leaving remote/subordinate node skew unresolved.

## What This Changes

The issue requests a safe subordinate-node upgrade path when a managed node is one Gateway protocol version behind the upgraded Gateway.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a narrow authenticated degraded maintenance RPC surface for older paired nodes or document an intentional safe upgrade order if compatibility is rejected.

## Reproduction Assessment

Yes. A protocol-3 node/client connecting to the current protocol-4 Gateway reaches the PROTOCOL_MISMATCH branch before node registration or maintenance RPCs.

## Solution Assessment

No as an automatic code change. Broadly accepting older wire protocols is not the best fix; the safer path is a maintainer-approved degraded maintenance contract with explicit limits.
