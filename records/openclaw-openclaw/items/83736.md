## Summary

Keep this open: current main and the latest release still enforce strict Gateway protocol matching for ordinary node clients, and this issue is now the canonical thread for deciding whether older authenticated nodes get a degraded maintenance path.

## What This Changes

Request that the Gateway tolerate minor gateway/node protocol skew, keep older nodes manageable in a degraded state, and document safe upgrade order.

## Best Possible Solution

Define the supported protocol-skew contract, then implement only a narrow authenticated degraded maintenance surface for older paired nodes or document a safe upgrade order with native-client release guidance.

## Reproduction Assessment

Yes for a source-level reproduction: send a node connect frame with minProtocol=3 and maxProtocol=3 to the current protocol-4 Gateway and it reaches PROTOCOL_MISMATCH before node registration. I did not run the reported Windows two-node setup.

## Solution Assessment

Unclear as a patch direction: accepting older protocol traffic broadly is not the best solution. The maintainable path is an explicit skew policy with a limited degraded maintenance RPC surface and tests for allowed versus rejected older-node behavior.
