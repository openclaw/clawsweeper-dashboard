- `impact:crash-loop`: The protocol mismatch can leave subordinate or mobile nodes unavailable for normal management/RPC after a gateway upgrade.

## Summary

Keep open: current main and the latest checked release still reject protocol-3 node/non-probe clients before node RPC registration, and the new Android report suggests the same version-skew problem is not limited to the original Windows node case.

## What This Changes

Request gateway/node protocol negotiation or a degraded maintenance path so older subordinate nodes remain manageable during staggered upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then add a narrow degraded maintenance mode for authenticated older nodes or document a safe upgrade order with tests around protocol negotiation, node listing, and maintenance RPC limits.

## Reproduction Assessment

Yes for a source-level reproduction: a node advertising only protocol 3 against the current protocol-4 gateway reaches PROTOCOL_MISMATCH before node registration or RPC. I did not run the reported Windows or Android setup.

## Solution Assessment

Unclear as a patch direction: accepting older protocol traffic broadly would be too risky. The maintainable fix is an explicit degraded maintenance contract or documented safe upgrade order, with the allowed RPC surface and protocol range chosen by maintainers.
