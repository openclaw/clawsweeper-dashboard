## Summary

Keeping this open: current main and the latest release still hard-reject older non-probe node/client protocol ranges before node registration, and related Android/version-skew reports now point here as the canonical compatibility thread. The remaining fix needs an explicit gateway/node skew policy and a safe degraded maintenance surface, not an automatic narrow bug patch.

## What This Changes

The issue requests that the gateway tolerate minor subordinate node protocol skew during upgrades or provide a stable maintenance path instead of marooning older nodes.

## Best Possible Solution

Define the supported gateway/node skew contract, then either implement a narrow authenticated degraded maintenance channel for approved older nodes or document a safe upgrade order and compatibility matrix with focused protocol, node-listing, and maintenance-RPC tests.

## Reproduction Assessment

Yes for a source-level reproduction: a node or Android client advertising only protocol 3 against the current protocol 4 gateway reaches the PROTOCOL_MISMATCH close path before node registration. I did not run the reported Windows two-node setup or a live Android pairing attempt.

## Solution Assessment

Unclear as requested: blanket N-1 protocol tolerance would be too broad for a wire/security boundary. The maintainable path is an explicit degraded maintenance contract or a documented upgrade matrix, with tests for what older authenticated nodes may and may not do.
