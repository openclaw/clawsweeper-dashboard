## Summary

Keep open: current main and the latest release still reject protocol-3 subordinate or Android clients before degraded node management can come online, and the merged local node-host restart fix is explicitly adjacent rather than a fix for remote/subordinate skew.

## What This Changes

The issue requests Gateway tolerance or a stable maintenance path for subordinate nodes during minor protocol/version skew after staged upgrades.

## Best Possible Solution

Define a secure N-1 maintenance handshake for authenticated subordinate nodes, keep incompatible full RPC rejected, and document supported Gateway/node upgrade order plus recovery behavior.

## Reproduction Assessment

Yes. A client advertising only protocol v3 cannot satisfy the current v4 Gateway handshake, so it fails before node management/RPC registration; I did not run a live Windows or Android setup in this read-only pass.

## Solution Assessment

No current solution is sufficient. The maintainable path is a designed degraded-maintenance protocol or documented upgrade contract, not simply lowering the global protocol floor.
