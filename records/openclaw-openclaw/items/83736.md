## Summary

Keep open: current main and the latest release still hard-reject non-probe clients whose protocol range does not include protocol 4 before node registration or normal RPC. The newer Android Play Store issue is a related release-artifact problem, but it does not replace this broader gateway/node degraded-maintenance compatibility decision.

## What This Changes

The issue asks the Gateway to tolerate minor protocol skew for subordinate nodes during upgrades, preferably through capability negotiation and a degraded maintenance RPC path.

## Best Possible Solution

Define the supported gateway/node skew contract, then either implement an explicitly scoped degraded maintenance mode for authenticated older nodes or document and enforce a safe upgrade order with clear Android and node release guidance.

## Reproduction Assessment

Yes for source-level reproduction: a node or Android client advertising minProtocol=3/maxProtocol=3 against current protocol 4 reaches the PROTOCOL_MISMATCH branch before role registration and normal RPC. I did not run the reported Windows two-node setup.

## Solution Assessment

Unclear as a patch direction: accepting stale protocol clients broadly is not the best fix. The maintainable path is a maintainer-approved degraded-maintenance contract with a deliberately limited RPC surface and tests for both rejection and allowed maintenance behavior.
