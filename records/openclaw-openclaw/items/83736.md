## Label Justifications

- `P2`: This is a clear user-impacting gateway/node upgrade compatibility problem, but the fix needs protocol and product policy before a narrow implementation can proceed.
- `impact:crash-loop`: The reported protocol mismatch makes subordinate nodes and current Play Store Android clients unavailable for normal gateway management or pairing after upgrade.

## Summary

Keep open: current main and v2026.5.20 still enforce protocol v4 for non-probe clients, recent comments add matching Android/node reports, and the remaining change needs a maintainer-defined compatibility or degraded-maintenance policy rather than cleanup closure.

## What This Changes

Request a gateway/node protocol-skew policy that lets older subordinate nodes remain manageable instead of hard-failing the handshake during upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then either implement a narrowly scoped degraded maintenance mode for approved older nodes or document a safe upgrade order with explicit Android release compatibility.

## Reproduction Assessment

Yes for a source-level reproduction: a protocol-3 non-probe client against the current protocol-4 gateway reaches the PROTOCOL_MISMATCH close path before node management RPC can register. I did not run the reported two-node Windows setup or Play Store Android app.

## Solution Assessment

Unclear as an implementation until maintainers choose the skew contract. Simply accepting older protocol traffic is too broad; the maintainable path is an explicit degraded maintenance surface or documented release/upgrade policy.
