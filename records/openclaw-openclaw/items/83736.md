## Label Justifications

- `P2`: This is a real gateway/node upgrade compatibility problem with a clear source path, but it needs protocol and product policy before implementation.
- `impact:crash-loop`: The mismatch makes subordinate nodes unavailable for normal management/RPC after a gateway upgrade, which is an availability failure for managed-node workflows.

## Summary

Keep open: current main still rejects ordinary node/client protocol ranges that do not include protocol v4 before node registration or maintenance RPC can run, and the added Android report shows this is not limited to the original Windows node case.

## What This Changes

Request gateway/node protocol negotiation or degraded maintenance support so slightly older subordinate nodes remain manageable during staggered upgrades.

## Best Possible Solution

Define the supported gateway/node version-skew contract, then add either a narrow authenticated degraded maintenance mode for older paired nodes or clear upgrade-order documentation with tests and regenerated protocol artifacts.

## Reproduction Assessment

Yes. A source-level reproduction is a non-probe node/client advertising only protocol 3 to the current protocol-4 gateway; the handler returns PROTOCOL_MISMATCH before node registration or maintenance RPC can run.

## Solution Assessment

Unclear as a direct patch direction. The maintainable fix is not broad protocol bypass, but an explicit negotiated degraded maintenance contract or a documented safe upgrade order.
