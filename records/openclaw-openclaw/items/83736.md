## Summary

Keep open. Current main still rejects node-mode clients whose protocol range does not include protocol v4, so a 2026.5.7/protocol-v3 subordinate node can be blocked before node registration or maintenance RPCs come online; the related broad skew issue remains open but does not cover this node-specific recovery path.

## What This Changes

Request gateway/node protocol skew tolerance or a minimal maintenance channel so older subordinate nodes can be updated after a gateway upgrade.

## Best Possible Solution

Define a supported gateway/node skew policy, negotiate safe N-1 node capabilities, and add a locked-down maintenance-only path for update, restart, and log collection with documented upgrade order.

## Reproduction Assessment

Yes, at source level: a node-mode client whose maxProtocol is 3 cannot satisfy current PROTOCOL_VERSION 4, so the handler returns PROTOCOL_MISMATCH before node registration or node.invoke maintenance paths can run. I did not run the exact two-version Windows repro in this read-only review.

## Solution Assessment

Partly. The requested direction is valid, but the maintainable fix should be explicit negotiated maintenance compatibility rather than blindly accepting old protocol ranges.
