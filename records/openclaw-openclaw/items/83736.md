## Summary

Keep open: current main and the latest shipped release still reject ordinary node/UI clients whose protocol range does not include Gateway protocol v4, and new comments confirm the same v3-to-v4 failure for Android users. This remains the canonical compatibility thread because related broader and Android-specific reports point back here, while the fix needs a maintainer decision on supported skew and degraded maintenance scope.

## What This Changes

Request that the Gateway tolerate slightly older subordinate node or Android clients after a gateway protocol bump, or provide a stable maintenance and upgrade path.

## Best Possible Solution

Define a supported gateway/node compatibility matrix and a narrow authenticated degraded-maintenance contract, then update protocol docs, tests, generated client constants, and Android release guidance accordingly.

## Reproduction Assessment

Yes for a source-level reproduction: a client advertising protocol range 3..3 against the current protocol-4 Gateway reaches the PROTOCOL_MISMATCH branch before node registration, pairing, or maintenance RPC. I did not run the reported Windows two-node or Play Store Android setup.

## Solution Assessment

Unclear as an implementation path: blanket N-1 acceptance would be too broad for a wire/auth boundary. The maintainable direction is an explicit compatibility policy plus a limited degraded maintenance mode or a documented release/upgrade order.
