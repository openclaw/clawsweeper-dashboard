## Summary

Keep open: current main and the latest stable release still reject ordinary node clients whose protocol range does not include the gateway protocol before node registration or management RPC can run. The Android follow-up suggests the same compatibility gap reaches mobile nodes, and the fix needs an explicit gateway/node skew and degraded-maintenance policy rather than an automated narrow patch.

## What This Changes

The issue asks the gateway to tolerate slightly older subordinate node protocol versions, or provide a degraded maintenance channel during staged node upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a narrow degraded maintenance mode for approved older nodes or document a safe upgrade order with protocol and node-management tests.

## Reproduction Assessment

Yes for a source-level reproduction: a non-probe node advertising only protocol 3 against current protocol 4 reaches the PROTOCOL_MISMATCH close path before node registration. I did not run the reported Windows two-node setup or the Android follow-up live.

## Solution Assessment

Unclear as an implementation path: accepting older node protocols wholesale would be too broad. The safer solution is an explicit degraded maintenance contract with a limited RPC surface and tests for both accepted and rejected protocol ranges.
