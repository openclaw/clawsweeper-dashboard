## Summary

Keep open: current main and the latest stable release still reject older non-probe node clients before node registration or management RPC can come online. The broader version-lock discussion overlaps policy, but this report adds a distinct subordinate-node maintenance-path failure that still needs a maintainer protocol/security decision.

## What This Changes

The issue requests gateway tolerance or a stable maintenance path for subordinate nodes whose node protocol is one minor version behind the upgraded gateway.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a narrow degraded maintenance mode for authenticated older nodes or document a safe upgrade order with explicit tests for protocol negotiation, node listing, and allowed maintenance RPCs.

## Reproduction Assessment

Yes for a source-level reproduction: a non-probe node advertising protocol 3 against current protocol 4 reaches the PROTOCOL_MISMATCH close path before node registration. I did not run the reported Windows two-node or Android setup.

## Solution Assessment

Unclear as a patch direction: accepting older protocol traffic broadly would be too wide for a gateway auth/RPC boundary. The maintainable solution should define an explicit degraded maintenance contract and limit the safe RPC surface for outdated nodes.
