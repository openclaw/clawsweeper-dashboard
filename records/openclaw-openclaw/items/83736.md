## Summary

Keep open: current main and the latest release still reject ordinary node/client protocol ranges that do not include Gateway protocol v4, and recent comments add more Android v3-client/v4-gateway confirmations. The remaining fix needs a maintainer decision on supported skew and the safe degraded maintenance surface rather than an automatic patch.

## What This Changes

The issue requests Gateway support for minor protocol skew so subordinate nodes can still be updated, restarted, or diagnosed after a gateway upgrade.

## Best Possible Solution

Define the supported Gateway/node skew contract, then either implement a narrowly scoped degraded maintenance mode for already trusted older nodes or document a safe upgrade order with focused protocol, node-listing, and maintenance-RPC tests.

## Reproduction Assessment

Yes for a source-level reproduction: a client advertising only protocol 3 against the current protocol-4 Gateway reaches the PROTOCOL_MISMATCH close path before node registration or ordinary RPC. I did not run the two-node Windows setup live.

## Solution Assessment

Unclear as a patch direction: blanket N-1 acceptance or skip-version behavior is not the best safe fix. The maintainable path is an explicit degraded maintenance contract with a limited authenticated RPC surface and tests for both accepted and rejected stale clients.
