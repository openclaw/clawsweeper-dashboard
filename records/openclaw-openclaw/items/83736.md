## Summary

Keeping this open: current main and the latest shipped release still reject protocol-3 node/client handshakes before node registration or maintenance RPC can come online, and the linked broader version-lock item points back here as the canonical node-skew thread.

## What This Changes

The issue asks the gateway to tolerate limited subordinate-node protocol skew or provide a stable degraded maintenance path during upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a constrained degraded maintenance mode for approved older nodes or document a safe upgrade order with tests around handshake, node listing, and allowed maintenance RPCs.

## Reproduction Assessment

Yes for a source-level reproduction: a protocol-3 node or client reaches the PROTOCOL_MISMATCH branch against the current protocol-4 gateway before node registration or RPC can come online. I did not run the reported Windows or Android setup.

## Solution Assessment

Unclear as an implementation path: the requested tolerance is valid, but broad acceptance of older protocol traffic would be too loose. The maintainable solution is an explicit degraded maintenance contract with a limited safe RPC surface.
