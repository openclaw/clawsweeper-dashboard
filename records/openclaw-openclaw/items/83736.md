## Label Justifications

- `P2`: This is a real user-impacting gateway/node upgrade compatibility problem with clear source evidence, but it needs product/protocol design rather than an urgent one-line bug fix.
- `impact:crash-loop`: The protocol mismatch makes subordinate node and Android client workflows unavailable for normal management/RPC after a Gateway upgrade.

## Summary

Keeping this open: current main and the latest stable release still reject ordinary protocol-3 node/mobile clients before node registration or maintenance RPC can run, and the new Android comments confirm this is not only the original Windows node case. The remaining fix needs a maintainer decision on supported gateway/node skew and the safe degraded maintenance surface.

## What This Changes

Request to let the Gateway tolerate slightly older subordinate node/mobile protocol versions, or provide a stable maintenance path during staggered upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement either a narrow degraded maintenance channel for authenticated stale nodes or document a safe upgrade order with tests around protocol negotiation, node listing, and allowed maintenance RPCs.

## Reproduction Assessment

Yes for a source-level path: a protocol-3 ordinary node/mobile client against current protocol 4 reaches the PROTOCOL_MISMATCH close branch before node registration or RPC. I did not run the reported Windows two-node or Android setup.

## Solution Assessment

Unclear as a direct patch: simply accepting stale protocol clients would be too broad for this security and wire-contract boundary. The maintainable direction is an explicit degraded maintenance contract or a documented upgrade order, chosen by maintainers.
