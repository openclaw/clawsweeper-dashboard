## Label Justifications

- `P2`: This is a user-impacting Gateway/node upgrade compatibility problem with a clear source path, but it requires protocol and product design before implementation.
- `impact:crash-loop`: The protocol mismatch leaves subordinate nodes unavailable for normal management/RPC after a Gateway upgrade, which is a process-level availability failure.

## Summary

Keep open: current main and the latest release still reject ordinary node/client protocol ranges that do not include the Gateway protocol, and the user reports show real v3-client/v4-gateway failures. The fix needs a maintainer decision on the supported skew and degraded maintenance boundary.

## What This Changes

Request support for minor Gateway/node protocol skew so older subordinate nodes remain available for degraded maintenance after Gateway upgrades.

## Best Possible Solution

Define the supported Gateway/node skew contract, then either implement a narrow authenticated degraded maintenance channel for older approved nodes or document a safe upgrade order if compatibility is intentionally rejected.

## Reproduction Assessment

Yes for a source-level reproduction: a protocol-3 ordinary node/client against the current protocol-4 Gateway reaches the PROTOCOL_MISMATCH close path before node RPC can register. I did not run the reported Windows two-node setup.

## Solution Assessment

Unclear as requested: blanket N-1 tolerance would be too broad for a protocol/auth boundary. The best path is an explicit degraded maintenance contract with tests for allowed RPCs, denied RPCs, and current v4 clients.
