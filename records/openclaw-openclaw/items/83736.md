## Summary

Keep open: current main and the latest release still reject non-probe clients whose protocol range does not include the gateway protocol before node registration or RPC can come online. The related Android comments add live user evidence for the same v3-client/v4-gateway mismatch, and no merged fix now owns the degraded maintenance path.

## What This Changes

The issue asks OpenClaw Gateway to tolerate minor gateway/subordinate-node protocol skew or provide a stable maintenance path for older nodes during upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then either document a safe strict upgrade order or implement a narrow degraded maintenance mode for already-paired older nodes with explicit RPC limits, authz tests, and protocol docs.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a node or mobile client advertising only protocol 3 against current protocol 4 reaches the PROTOCOL_MISMATCH branch before registration, pairing, or RPC; I did not live-run the reported Windows two-node setup.

## Solution Assessment

Unclear as a concrete implementation direction. Accepting stale protocol traffic broadly is not the best fix; maintainers should first define the safe degraded maintenance contract and supported skew window.
