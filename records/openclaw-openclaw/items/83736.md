## Summary

Keeping this open: current main still rejects protocol-3 node clients before node registration or maintenance RPC can attach, and the broader version-lock issue was closed in favor of this node-specific protocol-skew thread.

## What This Changes

Request to let upgraded gateways keep slightly older subordinate node clients reachable for degraded maintenance instead of hard-rejecting minor protocol skew.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a narrow degraded maintenance mode for approved older nodes or document a safe upgrade order with protocol, node-listing, and maintenance-RPC tests.

## Reproduction Assessment

Yes for a source-level reproduction: a node advertising minProtocol=3 and maxProtocol=3 against the current protocol-4 Gateway reaches PROTOCOL_MISMATCH before node registration or RPC. I did not run the reported Windows two-node setup.

## Solution Assessment

Unclear as an implementation path: accepting older protocol traffic broadly is not the best fix. The safer solution is an explicit degraded maintenance contract with a tightly limited RPC and auth surface.
