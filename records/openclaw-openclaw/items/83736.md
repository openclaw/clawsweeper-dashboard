- `impact:crash-loop`: Older subordinate nodes can be rejected before normal management RPC comes online, leaving them unavailable to OpenClaw for update/restart/log collection.

## Summary

Keep open: current main and the latest shipped release still reject ordinary node/client protocol ranges that do not include Gateway protocol v4, while the related broad and Android-specific reports now point back here as the canonical compatibility decision.

## What This Changes

Gateway should tolerate minor protocol skew for subordinate node/client upgrades or provide a stable degraded maintenance path.

## Best Possible Solution

Define the supported gateway/node skew contract, then either add a narrow degraded maintenance channel for already-paired older nodes with explicit RPC limits and tests, or document a safe upgrade order and compatibility matrix.

## Reproduction Assessment

Yes for a source-level reproduction: a non-probe node/client advertising only protocol 3 against the current protocol-4 Gateway reaches the PROTOCOL_MISMATCH close path before node registration or RPC. I did not run the reported Windows two-node setup.

## Solution Assessment

Unclear as an implementation direction: simply accepting older protocol traffic would be too broad. The maintainable path is an explicit degraded maintenance contract with scoped methods, auth rules, docs, and regression tests.
