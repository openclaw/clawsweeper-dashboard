## Summary

Keep open: current main and v2026.5.19 still reject non-probe clients whose advertised protocol range excludes v4, including node clients before registration, and the remaining fix needs a maintainer decision on safe degraded maintenance RPC rather than cleanup closure.

## What This Changes

The issue requests gateway/node protocol skew tolerance or a degraded maintenance path so older subordinate nodes can be updated after a gateway upgrade.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a narrow authenticated degraded maintenance mode for older nodes or document a safe upgrade order with regression coverage.

## Reproduction Assessment

Yes for a source-level reproduction: a node advertising only protocol 3 against the current protocol 4 Gateway reaches the PROTOCOL_MISMATCH branch before node registration or RPC. I did not run the reported Windows two-node setup.

## Solution Assessment

Unclear as a patch direction: accepting stale protocol traffic broadly would be unsafe. The maintainable solution should define an explicit degraded maintenance contract and restrict which RPCs older authenticated nodes may use.
