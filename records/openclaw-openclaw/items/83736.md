## Summary

Keep open: current main and the latest v2026.5.20 release still hard-reject protocol-3 non-probe node/UI clients before node registration or maintenance RPC can run, and recent Android comments add live logs for the same min=3 max=3 expected=4 failure class. This should stay with maintainers because the fix needs an explicit gateway/node skew and degraded-maintenance security contract.

## What This Changes

The issue asks the Gateway to tolerate N-1 subordinate node protocol skew or preserve a minimal maintenance path instead of marooning older nodes after a gateway upgrade.

## Best Possible Solution

Define the supported gateway/node protocol-skew contract, then add a narrow degraded maintenance mode for approved older nodes or document a safe upgrade order with tests around negotiation, node listing, and allowed maintenance RPCs.

## Reproduction Assessment

Yes for a source-level reproduction: a client advertising minProtocol=3 and maxProtocol=3 against current protocol 4 reaches the PROTOCOL_MISMATCH close path before node role registration or maintenance RPC. I did not run the reported Windows fleet setup.

## Solution Assessment

Unclear as an implementation direction: accepting stale protocol traffic wholesale is not the best fix. The safer path is an explicit degraded maintenance contract with a limited RPC surface and matching protocol/docs/tests.
