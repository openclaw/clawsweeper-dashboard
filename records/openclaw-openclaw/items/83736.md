## Summary

Keeping this open: current main and the latest release still reject non-probe clients whose protocol range does not include the gateway protocol. The merged local stale-node restart fix is related, but it explicitly leaves remote/subordinate nodes out of scope, so the degraded maintenance/version-skew contract remains unresolved.

## What This Changes

The issue asks for gateway/subordinate node protocol skew tolerance or a stable degraded maintenance path so older nodes are not stranded during staged upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a narrow authenticated degraded maintenance path for older paired nodes or document a safe upgrade order if compatibility is intentionally rejected.

## Reproduction Assessment

Yes. A source-level repro is clear: a protocol-3 node/client connecting to the current protocol-4 gateway reaches the PROTOCOL_MISMATCH branch before node registration or maintenance RPCs.

## Solution Assessment

No, not as a direct automation fix. Broadly accepting older wire protocols would be too risky; the best path is a maintainer-approved degraded maintenance contract with explicit limits.
