## Summary

Keep open: current main still rejects ordinary node clients whose protocol range does not include gateway protocol v4 before node registration or maintenance RPC can come online. The broader version-lock issue at https://github.com/openclaw/openclaw/issues/65095 overlaps policy but does not own this subordinate-node maintenance-channel detail, and the merged Control UI fix at https://github.com/openclaw/openclaw/pull/82883 explicitly did not change Gateway server compatibility behavior.

## What This Changes

The issue requests gateway/node protocol compatibility so older subordinate node hosts can still expose a minimal maintenance path during gateway upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a narrow degraded maintenance mode for authenticated older nodes or document a safe upgrade order with regression tests and protocol docs.

## Reproduction Assessment

Yes for a source-level reproduction: an ordinary node client advertising only protocol 3 against the current protocol 4 gateway reaches the PROTOCOL_MISMATCH branch before node registration or RPC. I did not run the reported two-node Windows setup.

## Solution Assessment

Unclear as a patch direction: blindly accepting older protocol traffic would be too broad. The maintainable solution should define an explicit degraded maintenance contract and limit the safe RPC surface for outdated nodes.
