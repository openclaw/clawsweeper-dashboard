## Summary

Keep this open: current main still rejects node clients whose protocol range does not include the gateway protocol before node registration or maintenance RPC can come online, and newer comments confirm the same v3-to-v4 failure for Android nodes. This needs a maintainer decision on supported gateway/node skew and the safe degraded maintenance surface before implementation.

## What This Changes

The issue asks the Gateway to tolerate minor node protocol/version skew, or provide a degraded maintenance path, so subordinate nodes are not stranded during upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then add a restricted degraded maintenance channel for authenticated older nodes or document a safe upgrade order if maintainers choose strict matching.

## Reproduction Assessment

Yes for a source-level reproduction: a node advertising protocol 3 against current protocol 4 reaches the PROTOCOL_MISMATCH branch before node registration or RPC. I did not run the reported Windows two-node setup or the Android app live.

## Solution Assessment

Unclear as an implementation direction: blanket N-1 acceptance or a skip-version-check mode would be too broad. The maintainable solution should define a narrow degraded maintenance contract with explicit RPC and auth limits.
