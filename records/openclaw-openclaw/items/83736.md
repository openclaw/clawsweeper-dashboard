## Summary

Keeping this open: current main and the latest shipped tag still reject non-probe node clients whose advertised protocol range does not include the gateway protocol, before node registration or maintenance RPCs can come online. The related broad version-lock issue was closed in favor of this issue, and the Android reports add confirming protocol-3 evidence rather than replacing the node maintenance-channel decision.

## What This Changes

Request gateway/node protocol compatibility or a degraded maintenance path so subordinate nodes are not marooned during staggered upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then either implement a constrained degraded maintenance channel for authenticated older nodes or document a safe upgrade order if protocol skew remains unsupported.

## Reproduction Assessment

Yes for a source-level reproduction: a normal node client advertising only protocol 3 reaches the PROTOCOL_MISMATCH branch before node role registration or maintenance RPC can run. I did not run the reported live Windows two-node setup.

## Solution Assessment

Unclear as a patch direction: accepting all older protocol traffic would be too broad. The maintainable path is an explicit degraded maintenance contract with scoped methods, authz rules, tests, and docs.
