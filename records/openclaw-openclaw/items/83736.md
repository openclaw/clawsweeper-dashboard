## Summary

Keeping this open: current main and the latest release still reject protocol-3 non-probe node clients before node registration/RPC, and the newer Android-specific thread does not replace this broader gateway/node skew policy decision.

## What This Changes

The issue asks for gateway/subordinate node protocol skew tolerance or a stable degraded maintenance path during staggered node upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a narrow degraded maintenance mode for approved older nodes or document a safe upgrade order with tests for protocol negotiation, node listing, and maintenance RPC limits.

## Reproduction Assessment

Yes for a source-level reproduction: a client advertising protocol range 3..3 against current protocol 4 reaches the PROTOCOL_MISMATCH branch before node registration or RPC. I did not run the reported Windows two-node setup.

## Solution Assessment

Unclear as an implementation direction: accepting older protocol traffic broadly would be too risky. The maintainable path is an explicit degraded maintenance contract with limited RPCs, scopes, docs, and regression tests.
