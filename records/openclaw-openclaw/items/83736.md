## Summary

Keeping this open: current main and the latest shipped release still reject ordinary protocol-3 nodes/clients before node registration, and the related broad version-lock issue was closed in favor of this more concrete protocol-skew thread.

## What This Changes

The issue requests a supported gateway/node version-skew path so older subordinate nodes can still connect in a degraded maintenance mode after the gateway is upgraded.

## Best Possible Solution

Define an explicit supported skew contract, then implement either a narrow authenticated degraded maintenance channel for approved older nodes/clients or a documented upgrade/release-order policy that prevents marooned nodes.

## Reproduction Assessment

Yes for source-level reproduction: a node or client advertising only protocol 3 against the current protocol-4 gateway reaches the PROTOCOL_MISMATCH close path before node registration or RPC setup. I did not run the reported Windows or Android live setup.

## Solution Assessment

Unclear as a patch direction: broadly accepting stale protocol traffic would be too risky. The best solution is a maintainer-approved degraded maintenance contract with explicit RPC and authorization limits, or a clear documented upgrade/release policy.
