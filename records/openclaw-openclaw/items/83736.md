## Label Justifications

- `P2`: The issue affects real gateway/node upgrade availability but needs protocol policy before a safe implementation.
- `impact:crash-loop`: The protocol mismatch can leave subordinate nodes unable to connect for normal management/RPC after a gateway upgrade.
- `impact:security`: The requested degraded maintenance path would change which older node clients can authenticate and which management RPCs they can invoke.

## Summary

Keep this open: current main still rejects node/native clients whose protocol range does not include the gateway protocol before node registration or maintenance RPC can come online, and the recent Android comments confirm the same protocol-3-to-4 failure mode. This is not ready for an automated fix because the safe gateway/node skew and degraded maintenance surface need maintainer protocol and security-boundary decisions first.

## What This Changes

The issue asks the gateway to tolerate minor gateway/node protocol skew so subordinate nodes remain reachable for maintenance during staged upgrades.

## Best Possible Solution

Define a supported gateway/node skew contract, then add a tightly scoped degraded maintenance mode for approved older nodes or document a safe upgrade order if older node RPC must remain rejected.

## Reproduction Assessment

Yes for a source-level reproduction: a node or native client advertising only protocol 3 against current protocol 4 reaches the PROTOCOL_MISMATCH branch before node registration or RPC. I did not run the reported Windows two-node or Android setup.

## Solution Assessment

Unclear as an implementation direction: accepting all older protocol traffic would be too broad. The maintainable solution needs an explicit degraded maintenance contract and tests for the allowed older-node RPC surface.
