## Summary

Keeping this open: current main still rejects node clients whose protocol range does not include the gateway protocol before normal node RPC can register, and the broader version-lock discussion at https://github.com/openclaw/openclaw/issues/65095 does not cover this node maintenance-channel detail. This should not be auto-fixed until maintainers choose the supported gateway/node skew and the safe degraded maintenance surface.

## What This Changes

The issue asks for gateway/node protocol skew tolerance or a stable maintenance path so older subordinate nodes can still be upgraded after the gateway moves ahead.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a narrow degraded maintenance mode for approved older nodes or document a safe upgrade order with tests around protocol negotiation, node listing, and maintenance RPC limits.

## Reproduction Assessment

Yes for a source-level reproduction: a node advertising only protocol 3 against current protocol 4 reaches the PROTOCOL_MISMATCH branch before node registration or RPC. I did not run the reported Windows two-node setup.

## Solution Assessment

Unclear as a patch direction: simply accepting older protocol traffic would be too broad. The maintainable solution should define an explicit degraded maintenance contract and limit the safe RPC surface for outdated nodes.
