## Summary

Keeping this open: current main still rejects ordinary node clients whose protocol range does not include the gateway protocol before node registration or node RPC can come online. The broader version-lock thread at https://github.com/openclaw/openclaw/issues/65095 overlaps policy, but this issue adds a distinct subordinate-node maintenance-channel failure, and the Android follow-up suggests the same skew problem affects native clients too.

## What This Changes

The issue requests gateway support for older subordinate node clients during rolling upgrades, preferably through capability negotiation and a degraded maintenance RPC path.

## Best Possible Solution

Define the supported gateway/node skew contract, then either implement a narrow degraded maintenance mode for approved older nodes or document a safe upgrade order with tests around protocol negotiation and maintenance RPC limits.

## Reproduction Assessment

Yes for a source-level reproduction: a node advertising only protocol 3 against the current protocol 4 gateway reaches the PROTOCOL_MISMATCH branch before node registration or RPC. I did not run the reported Windows two-node setup or Android client.

## Solution Assessment

Unclear as a patch direction: accepting older node traffic broadly is not the best solution. The maintainable path is an explicit degraded maintenance contract with a limited safe RPC surface and protocol/version documentation.
