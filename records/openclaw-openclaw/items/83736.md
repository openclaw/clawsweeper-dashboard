## Summary

Keeping this open: current main still hard-rejects protocol-3 node clients before node registration or maintenance RPC, and the related broader issue explicitly points here as the canonical remaining gateway/node skew thread. This needs a maintainer protocol and security-boundary decision, not cleanup closure.

## What This Changes

The issue requests gateway/node handshake compatibility or a degraded maintenance path for older subordinate nodes during rolling upgrades.

## Best Possible Solution

Define the supported gateway/node version-skew contract, then implement a narrowly scoped degraded maintenance path for authenticated older nodes or document a safe upgrade order with tests around negotiation, node status, and allowed maintenance RPCs.

## Reproduction Assessment

Yes for a source-level reproduction: a protocol-3 node against the current protocol-4 gateway reaches PROTOCOL_MISMATCH before node registration or maintenance RPC. I did not run the reported Windows two-node setup or Android Play Store setup.

## Solution Assessment

Unclear as a patch direction: broad protocol acceptance would be too risky. The maintainable solution is an explicit degraded maintenance contract with capability limits and regression coverage.
