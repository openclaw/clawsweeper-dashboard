## Summary

Keep open: current main and the latest release still reject protocol-3 non-probe nodes against a protocol-4 Gateway before node RPC can come online, and the related merged fix only covers same-install local node-host restarts.

## What This Changes

The issue requests Gateway/node protocol skew tolerance or a degraded maintenance path so subordinate nodes can still be upgraded, restarted, or inspected after a Gateway protocol bump.

## Best Possible Solution

Define the supported Gateway/node skew contract, then either implement a narrow authenticated degraded maintenance RPC path for older paired nodes or document a safe upgrade order if compatibility is intentionally rejected.

## Reproduction Assessment

Yes, at source level: a protocol-3 non-probe node/client connecting to the current protocol-4 Gateway reaches the PROTOCOL_MISMATCH branch before node registration or maintenance RPCs.

## Solution Assessment

No as a direct automation fix. Broadly accepting older wire protocols would be risky; the maintainable path is a maintainer-approved degraded maintenance contract with explicit limits.
