## Label Justifications

- `P2`: This is a user-impacting gateway/node upgrade compatibility problem with a clear source path, but it requires maintainer protocol policy before implementation.
- `impact:crash-loop`: The protocol mismatch leaves subordinate nodes unavailable for normal gateway management after an upgrade.
- `impact:security`: Any fix would intentionally relax connect-time protocol rejection for outdated node clients, so authz and maintenance RPC boundaries must be designed explicitly.

## Summary

Keep open: current main and the latest stable release still hard-reject node clients whose protocol range does not include gateway protocol v4 before node registration, and the newer Android comments match that source path. The fix needs a maintainer decision on supported gateway/node skew and a safe degraded maintenance surface, so this is not automation-safe yet.

## What This Changes

The issue asks the gateway to tolerate N-1 subordinate node protocol skew or provide a stable degraded maintenance path during node upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement an authenticated degraded node maintenance mode or document a safe upgrade order with protocol, authz, node listing, and maintenance RPC tests.

## Reproduction Assessment

Yes for a source-level reproduction: a node or Android client advertising only protocol 3 against current protocol 4 reaches the PROTOCOL_MISMATCH branch before node registration or RPC. I did not run the reported Windows or Android live setup.

## Solution Assessment

Unclear as an implementation path: accepting older protocol traffic broadly would be too risky. The maintainable solution is an explicit degraded maintenance contract with tightly limited RPCs and documented skew support.
