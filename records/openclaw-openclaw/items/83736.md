## Summary

Keep open: current main still hard-rejects protocol-3 node clients before node registration or RPC, and newer comments add Android v2026.4.5 confirmations. The remaining work needs a maintainer decision on supported gateway/node skew and degraded maintenance scope.

## What This Changes

Requests a gateway/node compatibility policy that lets slightly older subordinate nodes stay reachable for degraded maintenance during gateway upgrades.

## Best Possible Solution

Define supported gateway/node skew, then either publish/update client compatibility guidance or implement a narrow authenticated degraded maintenance RPC surface with protocol docs and regression tests.

## Reproduction Assessment

Yes for a source-level reproduction: a node client advertising protocol range 3..3 reaches the protocol-mismatch close path before node registration or RPC on current main. I did not run the reported Windows two-node setup.

## Solution Assessment

Unclear as an implementation path: accepting stale protocol traffic wholesale is not the best fix. The maintainable direction is an explicit degraded maintenance contract with a limited safe RPC surface, or a documented upgrade order if maintainers choose not to support skew.
