## Label Justifications

- `P2`: This is a user-impacting gateway/node upgrade compatibility problem with a clear source path, but it needs protocol and product design before implementation.
- `impact:crash-loop`: The protocol mismatch makes subordinate nodes unavailable for normal management/RPC after a gateway upgrade, which is a process-level availability failure.

## Summary

Keep open: current main and the latest stable release still hard-reject non-probe node clients whose protocol range does not include the gateway protocol before node registration or maintenance RPC can come online. The newer Android comment is consistent with the native-client constants, and the broader version-lock discussion at https://github.com/openclaw/openclaw/issues/65095 does not fully cover this node-specific degraded maintenance path.

## What This Changes

Requests gateway/node protocol compatibility or a degraded maintenance path so older subordinate nodes can still be upgraded, restarted, or inspected after a gateway upgrade.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a narrow degraded maintenance mode for authenticated older nodes or document a safe upgrade order with tests for protocol negotiation, node listing, and maintenance RPC limits.

## Reproduction Assessment

Yes, at source level: a node advertising only protocol 3 against the current protocol 4 gateway reaches the PROTOCOL_MISMATCH branch before node registration or RPC. I did not run the reported Windows or Android multi-node setup.

## Solution Assessment

Unclear as an implementation direction: broadly accepting older protocol traffic would be too risky. The maintainable path is an explicit degraded maintenance contract with a limited safe RPC surface for outdated authenticated nodes.
