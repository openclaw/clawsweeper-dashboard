## Summary

Keep open: current main and the latest release still reject protocol-3 subordinate or Android clients before any degraded node-management channel can come online, and the merged local-node restart fix is only adjacent.

## What This Changes

Request to let Gateway tolerate or safely degrade slightly older subordinate node clients during staggered upgrades instead of marooning them at handshake.

## Best Possible Solution

Define a secure N-1 maintenance handshake for authenticated subordinate nodes, keep incompatible full RPC rejected, and add focused gateway/node/native docs plus tests for upgrade and rollback behavior.

## Reproduction Assessment

Yes, source-reproducible: a client advertising only protocol v3 against the current v4 Gateway fails the handshake before node management/RPC registration. I did not run a live Windows or Android setup in this read-only pass.

## Solution Assessment

No current solution is sufficient; the merged local node-host restart fix is adjacent and intentionally leaves remote/subordinate nodes outside its scope. The maintainable path is a designed degraded-maintenance protocol rather than simply lowering the global protocol floor.
