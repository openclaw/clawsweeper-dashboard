- `impact:crash-loop`: The protocol mismatch makes subordinate nodes unavailable for normal management/RPC after a gateway upgrade.

## Summary

Keep this open: current main and the latest v2026.5.18 tag still reject protocol-3 non-probe node clients against protocol 4 before node registration or RPC, and the open broader version-lock issue does not cover the node-specific degraded maintenance channel. The new Android comment is useful supporting evidence that this is not limited to the original Windows node report.

## What This Changes

The issue requests N-1 or degraded-maintenance compatibility for subordinate nodes whose gateway protocol is behind during gateway upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then implement a narrow degraded maintenance mode for approved older nodes or document a safe upgrade order with tests around protocol negotiation, node listing, and maintenance RPC limits.

## Reproduction Assessment

Yes for source-level reproduction: a node advertising only protocol 3 against current protocol 4 reaches the PROTOCOL_MISMATCH branch before node registration or RPC. I did not run the reported Windows or Android setup.

## Solution Assessment

Unclear as an implementation path: accepting all older protocol traffic would be too broad. The maintainable solution should define an explicit degraded maintenance contract and limit which RPCs/scopes are safe for outdated nodes.
