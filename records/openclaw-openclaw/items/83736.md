## Label Justifications

- `P2`: This is a real gateway/node upgrade compatibility failure with clear source proof, but the fix needs protocol and product policy before implementation.
- `impact:crash-loop`: The protocol mismatch makes a subordinate node unavailable for normal management/RPC after the gateway upgrade.

## Summary

Keep open: current main still rejects older non-probe node clients before they can register for node RPC, and the related broad version-lock issue does not preserve this node-maintenance evidence. This needs a maintainer decision on the supported gateway/node skew and safe degraded maintenance surface before implementation.

## What This Changes

Request a gateway/node version-skew contract so subordinate nodes one protocol behind can remain degraded or maintainable instead of being hard-rejected during staged upgrades.

## Best Possible Solution

Define the supported gateway/node skew contract, then either implement a narrow authenticated degraded maintenance channel for approved older nodes or document a safe upgrade order with protocol, node-listing, and maintenance-RPC tests.

## Reproduction Assessment

Yes, at source level: a node advertising only protocol 3 against the current protocol 4 gateway reaches the `PROTOCOL_MISMATCH` branch before role parsing, node registration, or node RPC can come online. I did not run the reported Windows two-node setup or the later Android client case.

## Solution Assessment

Unclear as a patch direction: broad acceptance of older protocol traffic would be unsafe. The maintainable solution is an explicit degraded maintenance contract with a limited safe RPC surface, or a documented upgrade order if compatibility is not supported.
