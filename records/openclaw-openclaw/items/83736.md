## Summary

Keep open: current main and the latest shipped release still hard-reject ordinary node/mobile clients whose protocol range does not include Gateway protocol v4, while related broad and Android-specific reports now point to this issue as the canonical compatibility decision. This needs maintainer product and security-boundary review before implementation, because a degraded maintenance channel would intentionally accept older authenticated clients with a reduced RPC surface.

## What This Changes

Request that the Gateway tolerate minor protocol skew for subordinate node and mobile clients during upgrades, or provide a degraded maintenance path instead of hard-rejecting them.

## Best Possible Solution

Define the supported gateway/node protocol-skew contract, then either implement a narrow degraded maintenance mode for authenticated older nodes or document a safe upgrade order; Android release distribution should remain tracked separately where needed.

## Reproduction Assessment

Yes for a source-level reproduction: an ordinary node or Android client advertising protocol range 3..3 against current Gateway protocol 4 reaches the PROTOCOL_MISMATCH close path before node registration or maintenance RPC can run. I did not run the reported Windows two-node setup or a live Android device in this read-only review.

## Solution Assessment

Unclear as an implementation path: simply accepting stale protocol traffic would be too broad. The maintainable solution is a maintainer-approved protocol-skew policy with a minimal, authenticated degraded maintenance surface or an explicit documented upgrade order.
