## Summary

Keep open: the PR still does not deliver the claimed Paperclip env propagation, and the remaining question is a maintainer boundary decision rather than a cleanup-close case.

## What This Changes

The PR adds Paperclip runtime auth env helper plumbing from embedded runner into coding tool defaults, regression tests for `PAPERCLIP_*` env propagation, and shared fetch-cast test helper cleanup.

## Best Possible Solution

Decide the product boundary first; if this capability is still needed, rebuild it as a typed end-to-end exec env contract with attempt forwarding, exec merging, sanitizer review, docs, and regression plus real runtime proof.

## Reproduction Assessment

Yes, for the PR defects. Static inspection shows the branch uses fields missing from current main types, `createExecTool` never consumes the proposed default, and the embedded attempt path still omits the env when constructing tools.

## Solution Assessment

No. The current PR is partial Paperclip-specific core plumbing; the maintainable path is either the existing provider runtime-auth mechanism or a fully typed and reviewed generic exec-env contract.
