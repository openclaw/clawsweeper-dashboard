## Summary

Keep open: current main still sends pre-auth bootstrap-token handshakes into the mutex-backed verifier without a bootstrap-token limiter, and the linked same-author fix PR is still open. This issue should stay paired with that PR until the security/owner decision on mandatory versus configured limiter behavior is resolved and merged.

## What This Changes

The issue requests rate limiting or lockout before pre-auth bootstrap-token verification can enter the serialized bootstrap verifier.

## Best Possible Solution

Review and update https://github.com/openclaw/openclaw/pull/77527 so bootstrap-token verification is throttled before `verifyDeviceBootstrapToken` with the limiter policy explicitly settled, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection shows a signed device handshake with `auth.bootstrapToken` reaches `verifyBootstrapToken` before any bootstrap-token limiter, and the verifier serializes through `withLock`.

## Solution Assessment

Yes on direction, but not fully settled. Gating before `verifyDeviceBootstrapToken` is the right fix shape; the open PR still needs security/owner agreement on mandatory versus configured limiter behavior.
