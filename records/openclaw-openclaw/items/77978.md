## Summary

Keep open: current main still sends signed pre-auth bootstrap-token handshakes into the mutex-backed verifier without a bootstrap-token limiter, and the linked same-author fix PR is still open and explicitly paired with this issue.

## What This Changes

The issue reports that invalid pre-auth bootstrap-token handshakes can reach `verifyDeviceBootstrapToken` without a limiter, stalling the shared bootstrap mutex.

## Best Possible Solution

Review and update https://github.com/openclaw/openclaw/pull/77527 so bootstrap-token verification is throttled before `verifyDeviceBootstrapToken` with the limiter policy explicitly settled, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection shows a signed device handshake with `auth.bootstrapToken` reaches `verifyBootstrapToken` before any bootstrap-token limiter, and the verifier serializes through `withLock`.

## Solution Assessment

Yes on direction, but not fully settled. Gating before `verifyDeviceBootstrapToken` is the right fix shape; the open PR still needs security/owner agreement on mandatory versus configured and loopback-exempt limiter behavior.
