## Summary

Keep open: current main still sends pre-auth bootstrap-token handshakes into the mutex-protected verifier without a bootstrap-token limiter, and #77527 is the active open fix PR for this issue.

## What This Changes

Requests rate limiting and lockout before pre-auth bootstrap-token verification so bogus tokens cannot serialize the gateway bootstrap mutex.

## Best Possible Solution

Land a reviewed fix that gates bootstrap-token verification before `verifyDeviceBootstrapToken`, includes regression coverage for lockout before mutex entry, and resolves the always-on versus configured-limiter policy question.

## Reproduction Assessment

Yes, from source inspection rather than a live run: a signed device handshake with `auth.bootstrapToken` reaches `verifyBootstrapToken` before any bootstrap limiter, and that verifier serializes under `withLock`. #77527 also documents an in-process WebSocket reproduction path and expected `rate_limited` behavior after the fix.

## Solution Assessment

Yes on direction, with one maintainer decision still needed: #77527 is the narrow fix candidate because it adds a bootstrap-token limiter before verification. The open question is whether this scope should mirror existing configured rate limits or be mandatory for all bootstrap-token attempts.
