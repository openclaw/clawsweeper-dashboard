## Summary

Keep open: this PR still contains useful channel reply-routing work, but the current head lacks real behavior proof, is merge-conflicting, and introduces Telegram idempotency behavior that can silently drop later chunks or media sends.

## What This Changes

The PR threads inbound idempotency keys through routeReply and outbound delivery, adds Telegram send dedupe, normalizes Discord send/poll targets, and adds focused tests.

## Best Possible Solution

Scope dedupe per outbound send unit or move it above provider sends, encode Telegram cache keys structurally, preserve the useful Discord target normalization, add regression coverage, and require redacted real Telegram/Discord proof before merge.

## Reproduction Assessment

Yes, for the PR-introduced regression by source inspection: the branch passes one idempotency key into a delivery context that current main uses for multiple text chunks or media sends. I did not run a live Telegram reproduction because this is a read-only review.

## Solution Assessment

No. Threading stable routing keys is reasonable, but provider-level dedupe keyed only by account/chat/idempotencyKey needs a send-unit discriminator, structural key encoding, or a higher-level delivery-attempt boundary.
