## Summary

Keep open: the branch still has a blocking Telegram idempotency bug and lacks after-fix real behavior proof, while current main has not superseded the useful parts of the PR.

## What This Changes

The PR threads idempotency keys through reply routing and Telegram outbound sends, adds Telegram send dedupe, normalizes Discord send/poll targets, and adds focused tests.

## Best Possible Solution

Scope dedupe per outbound send unit or move it above provider sends, keep the Discord target normalization if still needed, add regression coverage, and require redacted real Telegram/Discord behavior proof before merge.

## Reproduction Assessment

Yes, by source inspection for the PR-introduced bug: the branch adds one idempotency key to a delivery context that current main uses for multiple text or media sends. I did not run a live Telegram reproduction.

## Solution Assessment

No. Threading a stable key through routing is directionally reasonable, but provider-level dedupe needs a chunk/media discriminator or a higher-level delivery-attempt boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Scope idempotency per outbound send unit:** `src/infra/outbound/deliver.ts:582`
  - body: This passes one `idempotencyKey` into the shared outbound context for a whole delivery attempt. Long text and media are sent as separate provider calls, but Telegram's new cache is keyed only by account/chat/idempotency, so later chunks or attachments can reuse the first result instead of being sent.
  - confidence: 0.94
- **[P2] Encode Telegram idempotency key components:** `extensions/telegram/src/send.ts:208`
  - body: The cache key joins free-form `accountId` and `idempotencyKey` values with `:`. Since those values can contain separators, distinct sends can collide and return the wrong cached result; encode the tuple or use a structured key.
  - confidence: 0.78
