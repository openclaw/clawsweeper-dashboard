## Summary

Keep open. The PR targets real Telegram/Discord reply-routing hardening, but it should not merge as-is because the new Telegram idempotency cache can collapse distinct chunks or media sends under one delivery key; current main has not otherwise implemented the intended routed-send dedupe.

## What This Changes

The PR threads reply-routing idempotency keys through routeReply, outbound delivery, and Telegram sends; adds Telegram send dedupe; normalizes Discord send/poll targets; and adds focused tests.

## Best Possible Solution

Repair the patch so retries dedupe duplicate delivery attempts without collapsing distinct text chunks, media items, or payload sends; keep the Discord target normalization; add regression coverage and the required changelog entry.

## Reproduction Assessment

Yes, at source level. On the PR head, a routed Telegram reply longer than 4000 characters or a multi-media payload with a stable idempotency key reaches Telegram as multiple provider sends using the same account/chat/key cache identity.

## Solution Assessment

No. Threading idempotency through routed replies is the right direction, but this implementation scopes the key too broadly; the safer fix is to add a per-send-unit discriminator or enforce dedupe above the provider-send loop.

## Review Findings
