## Summary

Keep open: the Discord target normalization is useful, but the Telegram dedupe design is unsafe because the PR forwards one idempotency key through a delivery attempt that can contain multiple provider sends. Real behavior proof is also still insufficient for this external Telegram/Discord routing change.

## What This Changes

The PR threads reply idempotency keys through routed outbound delivery and Telegram sends, adds an in-memory Telegram send dedupe cache, normalizes Discord send/poll targets, and adds focused tests.

## Best Possible Solution

Keep the Discord target normalization, redesign Telegram dedupe to scope idempotency per outbound send unit or above provider sends, and require focused tests plus real Telegram/Discord proof before merge.

## Reproduction Assessment

Yes, by source inspection for the PR-introduced regression: a routed Telegram reply with an idempotencyKey and multiple text chunks or media units reaches the same PR-added Telegram cache key for distinct sends. I did not run live Telegram proof in this read-only review.

## Solution Assessment

No. Threading a stable key is directionally right, but caching sendMessageTelegram by account/chat/idempotency is too coarse; the fix should add a send-unit discriminator or dedupe the outer delivery attempt before provider sends.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

- **[P1] Scope idempotency per outbound send unit:** `src/infra/outbound/deliver.ts:579`
  - body: This adds one idempotencyKey to the shared outbound context for the whole delivery attempt. Telegram's new cache is keyed only by account/chat/idempotency, but delivery can split a reply into text chunks or multiple media sends, so later sends can reuse the first cached result and never reach Telegram; add a chunk/media discriminator or dedupe above provider sends.
  - confidence: 0.94

  - body: Joining accountId, chatId, and idempotencyKey with ':' can make different triples share one cache entry when account IDs or idempotency keys contain ':'. Encode the components or use a structured key so unrelated sends cannot dedupe each other.
  - confidence: 0.78
