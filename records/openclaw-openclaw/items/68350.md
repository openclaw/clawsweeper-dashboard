## Summary

Keep this PR open. Current main still lacks the PR's routed idempotency-key plumbing and Discord send/poll target normalization, so it is not obsolete; however, the proposed Telegram dedupe layer has a blocking correctness regression for chunked text and multi-send media paths.

## What This Changes

The PR threads idempotency keys through reply routing and Telegram outbound delivery, adds Telegram send deduplication, and normalizes Discord send/poll targets to channel targets.

## Best Possible Solution

Revise the PR so idempotency dedupes repeated whole delivery attempts without collapsing individual text chunks or media units, then add focused tests for long Telegram replies, media sequences, routed transcript mirroring, and Discord send/poll target normalization before maintainer review.

## Reproduction Assessment

Yes. A routed Telegram reply that provides an idempotency key and is split into multiple outbound send units, such as text over the Telegram chunk limit or multiple media sends, exercises the cache collision path introduced by the PR.

## Solution Assessment

No, not as currently proposed. The direction is useful, but the dedupe key needs per-delivery-unit scoping or an outer-attempt design that preserves every chunk/media send while still suppressing true repeated attempts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope Telegram idempotency per delivery unit:** `src/infra/outbound/deliver.ts:253`
  - body: Adding `idempotencyKey` to the shared outbound context means every chunk or media item produced by one delivery attempt reuses the same key. Telegram then caches by account/chat/key, so long text chunks or later media sends return the first send result instead of being delivered. Scope the dedupe key per text/media unit, or move dedupe to a layer that does not collapse multi-send deliveries.
  - confidence: 0.93
