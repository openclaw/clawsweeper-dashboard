## Summary

Keep this PR open, but do not merge it as-is: the Discord target normalization is useful, while the Telegram idempotency cache can collapse distinct chunks/media sends and the posted proof does not show the changed routing behavior in a real setup.

## What This Changes

The PR threads inbound idempotency keys through reply routing and Telegram sends, adds an in-memory Telegram send dedupe cache, normalizes Discord send/poll targets, and adds focused tests.

## Best Possible Solution

Scope dedupe per outbound send unit or move it above provider sends, keep the Discord target normalization, add focused regression coverage, and require real Telegram/Discord behavior proof before merge.

## Reproduction Assessment

Yes, by source inspection for the PR-introduced bug: a routed Telegram reply that splits into multiple text chunks or media sends would reuse the same PR-added cache key across distinct provider sends. I did not run a live Telegram reproduction.

## Solution Assessment

No. Threading a stable key through routing is directionally reasonable, but the proposed provider-level cache needs a chunk/media discriminator or a higher-level delivery-attempt dedupe boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Scope idempotency per outbound send unit:** `src/infra/outbound/deliver.ts:582`
  - body: This passes one `idempotencyKey` into the shared outbound context for the whole delivery attempt. Telegram's new cache is keyed only by account/chat/idempotency, but delivery splits long text and media into multiple provider sends, so later chunks or attachments can reuse the first cached result instead of being sent.
  - confidence: 0.94
- **[P2] Encode Telegram idempotency key components:** `extensions/telegram/src/send.ts:208`
  - body: The cache key joins free-form `accountId` and `idempotencyKey` values with `:`, while idempotency keys already use colon-like structured strings in nearby outbound paths. Encode the tuple or use an unambiguous structured key so unrelated sends cannot collide and reuse the wrong result.
  - confidence: 0.78
