## Summary

Keep open. The PR addresses a real channel-routing gap that current main does not fully solve, but the proposed Telegram idempotency cache is not safe to merge because it reuses one inbound idempotency key across every chunk/media send in a delivery attempt, so long or multi-part Telegram replies can collapse to the first send result.

## What This Changes

The PR threads reply-routing idempotency keys through core outbound delivery and Telegram sends, adds Telegram send deduplication, normalizes Discord send/poll targets to channel targets, and adds focused tests for those paths.

## Best Possible Solution

Keep the useful routing direction, but make outbound idempotency scoped to the logical delivery attempt without collapsing distinct provider send units. A repaired patch should preserve duplicate-attempt dedupe, distinguish text chunks and media sends, keep the Discord channel-target normalization, add regression coverage for long Telegram replies and multi-media payloads, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. On the PR branch, route a Telegram reply longer than 4000 characters, or a payload with multiple Telegram media sends, with a stable idempotency key; static inspection shows every send unit reaches Telegram with the same cache key, so only the first provider send is executed.

## Solution Assessment

No, not as proposed. Threading an idempotency key through routing is the right class of fix, but the key must be salted per provider send unit or enforced at a higher delivery-attempt layer so separate chunks and attachments are not deduped against each other.
