## Summary

Keep this PR open, but do not merge it as-is. The routing hardening direction is useful and current main has not implemented the provider-send idempotency or Discord action normalization pieces, but the proposed Telegram dedupe cache would collapse distinct text chunks and media sends under the same idempotency key.

## What This Changes

The PR threads reply-routing idempotency keys through outbound delivery and Telegram sends, adds Telegram in-memory send deduplication, normalizes Discord send/poll targets, and adds focused tests.

## Best Possible Solution

Keep the routing hardening direction, but scope idempotency to the logical delivery attempt without deduping distinct provider send units against each other. A repaired patch should preserve duplicate-attempt protection, distinguish text chunks and media sends, keep the Discord target normalization, add regressions for long Telegram replies and multi-media payloads, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. On the PR branch, route a Telegram reply longer than 4000 characters or a payload with multiple Telegram media sends using a stable idempotency key; static inspection shows each provider send reaches Telegram with the same account/chat/idempotency cache key.

## Solution Assessment

No, not as proposed. Threading an idempotency key through routing is the right class of fix, but the key must be salted per provider send unit or enforced at a higher delivery-attempt layer so chunks and attachments remain distinct.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope idempotency per outbound send unit:** `src/infra/outbound/deliver.ts:579`
  - body: This passes one `idempotencyKey` through the shared outbound context used for every chunk and media send. Telegram caches by account/chat/idempotency key, so a long reply split by the Telegram chunker, or a multi-media payload, will return the first send result for later units instead of sending them. Include a chunk/media/payload discriminator or move dedupe to a layer that distinguishes send units.
  - confidence: 0.94
- **[P3] Use a collision-safe cache key:** `extensions/telegram/src/send.ts:311-317`
  - body: The Telegram dedupe key joins account, chat, and idempotency values with `:`. Account IDs are configuration-derived and idempotency keys are caller-supplied, so distinct tuples can collide if those values contain separator-shaped content. Encode the tuple or use a structured key before storing it in the in-memory cache.
  - confidence: 0.78
