## Summary

Keep this PR open, but it is not mergeable as-is: the idempotency threading is directionally useful, yet the current branch can collapse multiple Telegram send units under one dedupe key and lacks real behavior proof for the changed Telegram/Discord routing.

## What This Changes

The PR threads channel reply idempotency keys through outbound delivery, adds Telegram send dedupe, normalizes Discord outbound targets, and adds focused tests.

## Best Possible Solution

Scope dedupe per outbound send unit or move it above provider sends, preserve any still-needed Discord target normalization, add regression coverage, and require redacted real Telegram/Discord behavior proof before merge.

## Reproduction Assessment

Yes, for the PR-introduced regression by source inspection: the branch passes one idempotency key into a delivery context that current main uses for multiple text chunks or media sends. I did not run a live Telegram reproduction.

## Solution Assessment

No. Threading a stable key through routing is reasonable, but provider-level dedupe keyed only by account/chat/idempotencyKey needs a send-unit discriminator or a higher-level delivery-attempt boundary.
