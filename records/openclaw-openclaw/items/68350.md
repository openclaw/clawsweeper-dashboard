## Summary

Keep open: this PR addresses plausible Telegram/Discord reply-routing gaps, but it is not mergeable as-is because the proposed Telegram idempotency cache can collapse distinct text chunks or media sends, and the external PR still lacks real behavior proof for the changed routing path.

## What This Changes

The PR threads reply-routing idempotency keys into outbound delivery and Telegram sends, adds an in-memory Telegram send dedupe cache, normalizes Discord send/poll targets, and adds focused tests.

## Best Possible Solution

Repair the idempotency design so duplicate delivery attempts dedupe without collapsing distinct text chunks or media sends, preserve the Discord target normalization, and require real Telegram/Discord proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Telegram routed reply that splits into multiple text chunks, or a payload with multiple media sends, would reuse the same PR-added Telegram cache key for distinct provider sends.

## Solution Assessment

No. Threading idempotency through routing is the right direction, but dedupe must be scoped per outbound send unit or moved above provider sends so retries dedupe without collapsing chunks or media.
