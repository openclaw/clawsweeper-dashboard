## Summary

Keep open. The routing-hardening direction is useful, but the proposed patch is not safe to merge because Telegram dedupe reuses one idempotency key for every provider send unit, which can drop later chunks or media sends.

## What This Changes

The PR threads reply-routing idempotency keys through routeReply, outbound delivery, and Telegram sends; adds Telegram send dedupe; normalizes Discord send/poll targets; and adds focused tests.

## Best Possible Solution

Repair the patch so duplicate delivery attempts dedupe without collapsing distinct provider send units, keep the Discord channel-target normalization, add regression coverage for long Telegram replies and multi-media payloads, and add the required changelog entry.

## Reproduction Assessment

Yes. On the PR head, route a Telegram reply longer than 4000 characters or a payload with multiple media sends using a stable idempotency key; static inspection shows each send unit reaches Telegram with the same cache key.

## Solution Assessment

No. Threading idempotency through routing is the right class of fix, but the dedupe key must include a chunk/media/payload discriminator or be enforced at a higher delivery-attempt layer that distinguishes provider sends.

## Review Findings
