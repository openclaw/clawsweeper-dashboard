## Summary

Keep open: current main has not implemented this routing hardening, and the branch still has a blocking Telegram regression where one idempotency key can collapse multiple outbound send units.

## What This Changes

The PR threads reply idempotency keys through routeReply, outbound delivery, and Telegram sends, adds Telegram send dedupe, normalizes Discord send/poll targets, and adds focused tests.

## Best Possible Solution

Implement duplicate-attempt protection without collapsing distinct chunks, media items, or payload sends; keep the Discord target normalization and ship the change with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. On the PR head, route a Telegram reply longer than 4000 characters or a payload with multiple Telegram media sends using a stable idempotencyKey; static inspection shows each provider send receives the same cache key.

## Solution Assessment

No. Threading idempotency through routing is the right direction, but the dedupe must be scoped per outbound send unit or enforced at a higher delivery-attempt layer that still distinguishes provider sends.
