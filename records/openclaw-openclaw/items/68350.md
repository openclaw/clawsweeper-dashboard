## Summary

Keep open: this PR still has a high-confidence message-delivery regression in the proposed Telegram idempotency path and lacks real Telegram/Discord behavior proof for the changed routing/dedupe behavior.

## What This Changes

The branch threads reply-routing idempotency keys into outbound delivery and Telegram sends, adds Telegram in-memory send dedupe, normalizes Discord send/poll targets, and adds focused tests.

## Best Possible Solution

Scope dedupe per outbound send unit or move it above provider sends, encode Telegram cache-key components, preserve only still-needed Discord target normalization, and require redacted real transport proof before merge.

## Reproduction Assessment

Yes, for the PR-introduced regression by source inspection: the branch passes one idempotency key into a delivery context that current main uses for multiple text chunks and media sends. I did not establish a live Telegram reproduction.

## Solution Assessment

No. Threading a stable key through routing is reasonable, but provider-level dedupe keyed only by account/chat/idempotencyKey needs a send-unit discriminator or a higher-level delivery-attempt boundary.
