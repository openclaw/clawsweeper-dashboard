## Summary

Keep open: the patch has a concrete Telegram correctness regression, and the available live proof only shows gateway readiness rather than the changed reply-routing behavior.

## What This Changes

The PR threads idempotency keys through reply routing and Telegram sends, adds Telegram send deduplication, normalizes Discord send/poll targets, and adds focused tests.

## Best Possible Solution

Repair the patch so duplicate delivery attempts dedupe without collapsing distinct text chunks or media sends, while preserving the Discord target normalization.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: route a Telegram reply longer than 4000 characters, or multiple media sends, with one idempotency key and later send units reuse the same Telegram cache key.

## Solution Assessment

No. Threading idempotency through routing is the right direction, but the dedupe must include a send-unit discriminator or move above the per-provider send layer without collapsing chunks/media.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope idempotency per outbound send unit:** `src/infra/outbound/deliver.ts:579`
  - body: This passes one `idempotencyKey` into the shared channel context, so every text chunk and media send in a delivery attempt reaches Telegram with the same account/chat/idempotency cache key. Long replies split by Telegram's chunker, or payloads with multiple media items, can return the first send result for later units instead of sending them; add a chunk/media/payload discriminator or move dedupe above the per-send layer.
  - confidence: 0.94
