## Summary

Keep open. Current main bounds the OpenClaw file logger, but the Linux systemd/journald storage-pressure path reported here remains unresolved: generated user units do not set output routing or journald policy, runtime hints still point users to journalctl, and the searched docs do not mention journald retention or vacuum guidance.

## What This Changes

The issue requests that Linux/systemd logging account for journald growth instead of only rotating an OpenClaw file log.

## Best Possible Solution

Document Linux service logging policy clearly: explain that systemd services also write to journald, add retention options such as SystemMaxUse/RuntimeMaxUse and journalctl vacuum commands, and keep automatic unit-output or vacuum behavior out of the runtime until maintainers choose a default policy.

## Reproduction Assessment

Yes for the source/docs gap: current main can be inspected to show Linux services use journald, generated units do not set journald policy, and docs lack retention guidance. A full disk-growth reproduction would require running the Gateway as a Linux systemd user service under sustained logging and checking journalctl disk usage.

## Solution Assessment

Yes for a narrow first repair: docs/setup guidance is the lowest-risk maintainable fix for operators today. Automatic vacuuming or redirecting stdout/stderr from generated units may be useful later but needs maintainer ops-policy review.
