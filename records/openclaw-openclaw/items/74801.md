## Summary

Keep open. Current main has in-process rotation for OpenClaw file logs and no longer appears to contain the old rotate-script/cron mechanics from the report, but the Linux systemd journald pressure concern remains: generated user units do not bound or redirect journald output, runtime hints still rely on journalctl, and docs do not give journald retention guidance.

## What This Changes

The issue asks OpenClaw to address unbounded journald growth for Linux systemd user services because the active service log stream is not covered by the old file-log rotation check.

## Best Possible Solution

Define the Linux service logging policy explicitly, likely starting with docs and/or doctor/setup guidance for journald retention such as SystemMaxUse/RuntimeMaxUse or journalctl vacuum guidance, and only change generated systemd stream behavior if maintainers choose that as the product default.

## Reproduction Assessment

Yes for the configuration/docs gap: current main can be checked by rendering or inspecting buildSystemdUnit, runtime hints, and Linux docs, which show journalctl usage without retention policy. A full disk-growth reproduction would still require installing the current gateway as a Linux systemd user unit, generating logs, and measuring journalctl disk usage.

## Solution Assessment

Unclear. The report's docs/setup-retention option is the narrowest maintainable first fix, while automatic journal vacuuming or changing generated StandardOutput/StandardError defaults needs maintainer ops-policy review.
