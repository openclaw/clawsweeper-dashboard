## Summary

Keep open. Current main still treats Linux systemd service logs as journald output while OpenClaw file-log rotation only covers the Gateway JSONL file, and the inspected docs do not yet describe journald disk usage or retention controls.

## What This Changes

The issue requests journald retention guidance or service-log behavior changes for systemd installs where real gateway output is stored in journald rather than OpenClaw file logs.

## Best Possible Solution

Document journald as a separate Linux/systemd log-storage surface, including disk-usage checks, manual vacuum commands, and persistent host journal caps, while leaving automatic vacuuming or output rerouting for a separate maintainer decision.

## Reproduction Assessment

Yes for the actionable gap: current main points Linux service log inspection at `journalctl`, file-log rotation applies only to OpenClaw file logs, and the inspected docs lack journald retention guidance. A full storage-growth reproduction would require running the gateway as a systemd user service under sustained logging and checking journal disk usage.

## Solution Assessment

Yes for a docs-first repair: the issue lists setup documentation as an acceptable outcome, and it is the narrowest maintainable fix for the current gap. Automatic journal vacuuming or service output rerouting should be handled only after explicit product/ops direction.
