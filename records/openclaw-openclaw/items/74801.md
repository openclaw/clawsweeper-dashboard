## Summary

Keep open. Current main has moved past the old cron/script rotation mechanics and now rotates the Gateway file log in-process, but the reported Linux systemd journald storage-pressure path is still not covered: generated user units do not set a journald/output policy, runtime hints still direct Linux users to journalctl, and docs do not mention journald retention settings.

## What This Changes

This issue requests bounding or documenting Linux systemd service log storage because OpenClaw file-log rotation does not cap journald output from the Gateway service.

## Best Possible Solution

Add an explicit Linux service logging policy that distinguishes bounded OpenClaw file logs from systemd journal storage, likely starting with public docs and/or doctor/runtime guidance for journald retention; avoid automatic journal vacuuming or generated unit stream changes unless maintainers choose that policy.

## Reproduction Assessment

No high-confidence live reproduction was run. Static inspection shows the current systemd unit and docs still leave journald retention unaddressed; a full reproduction would install the current Gateway as a Linux systemd user service, generate logs, and compare `journalctl` disk usage against the rotating file log.

## Solution Assessment

Unclear. The safest narrow first step is docs or diagnostic guidance for journald retention; automatic `journalctl --vacuum-*` or changing generated `StandardOutput`/`StandardError` would need explicit maintainer ops policy.
