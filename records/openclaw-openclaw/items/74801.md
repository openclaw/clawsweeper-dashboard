## Summary

Keep open. Current main bounds OpenClaw-managed file logs, but the reported Linux systemd journald storage pressure is still not addressed in code or docs, and the related open disk-growth report tracks different JSONL files rather than the systemd journal.

## What This Changes

Request to make Linux systemd logging storage bounded by vacuuming/configuring journald, documenting journald retention, or routing gateway output into the OpenClaw file logger.

## Best Possible Solution

Document Linux systemd journald retention as a separate log-storage surface from OpenClaw file-log rotation, then leave automatic journal vacuuming or service output rerouting for explicit maintainer product/ops review.

## Reproduction Assessment

Yes for the source/docs gap: current main can be inspected to show no systemd journald policy, runtime hints use `journalctl`, and docs lack retention guidance. A full storage-growth reproduction would require running the Gateway as a Linux systemd user service under sustained logging and checking `journalctl --disk-usage`.

## Solution Assessment

Unclear for automatic runtime changes, because vacuuming journald or rerouting service output changes host operations policy. The narrowest maintainable first fix is docs/setup guidance that explains journald retention controls and manual vacuum options without changing generated service behavior.
