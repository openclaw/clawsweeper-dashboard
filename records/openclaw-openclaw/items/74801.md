## Summary

Keep open. Current main still treats Linux systemd service output as a journald surface, while OpenClaw's documented and implemented rotation only bounds the Gateway file log; the inspected docs still do not describe journal disk usage, vacuuming, or host retention caps.

## What This Changes

Document or otherwise bound journald storage for Linux systemd Gateway services instead of relying only on OpenClaw file-log rotation.

## Best Possible Solution

Document journald as a separate Linux/systemd service log surface, including disk-usage checks, safe manual vacuum commands, and persistent host journal caps, while leaving automatic vacuuming or output rerouting for a separate product decision.

## Reproduction Assessment

Yes for the actionable gap: current main points Linux service log inspection at `journalctl`, file-log rotation applies only to OpenClaw file logs, and the inspected docs lack journald retention guidance. A full storage-growth reproduction would require running the Gateway as a systemd user service under sustained logging and checking journal disk usage.

## Solution Assessment

Yes for a docs-first repair. The issue lists setup documentation as an acceptable outcome, and that is narrower than adding automatic journal vacuuming or changing service output behavior without maintainer ops direction.
