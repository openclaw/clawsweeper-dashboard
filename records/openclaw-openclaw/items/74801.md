## Summary

Keep open. Current main still bounds OpenClaw-managed file logs only; Linux systemd service output is still surfaced through journald, and the logging/Linux service docs do not document journald retention or vacuum guidance.

## What This Changes

Request to address Linux systemd journald storage pressure that is not covered by OpenClaw file-log rotation.

## Best Possible Solution

Document journald as a separate Linux systemd log-storage surface from OpenClaw file-log rotation, including `journalctl --disk-usage`, manual vacuum commands, and systemd/journald retention settings; leave automatic vacuuming or service output rerouting for explicit maintainer ops review.

## Reproduction Assessment

Yes for the source/docs gap: current main shows systemd log hints use `journalctl`, file-log rotation is separate, and docs lack journald retention guidance. A full storage-growth reproduction would require running the Gateway as a Linux systemd user service under sustained logging and checking `journalctl --disk-usage`.

## Solution Assessment

Yes for a narrow first repair: docs/setup guidance is the least invasive way to address the reported storage-pressure blind spot. Automatic journal vacuuming or output rerouting may be useful later, but those are broader service-default decisions.
