## Summary

Keep open. Current main has fixed the obsolete file-log rotation side by rotating Gateway file logs in-process, but the Linux systemd/journald storage-pressure concern remains: service output is still forwarded to service streams, the generated unit has no journal/stream retention policy, and the user-facing Linux docs point to journalctl without retention guidance.

## What This Changes

The issue asks OpenClaw to address unbounded journald growth for Linux systemd Gateway services and reconcile the old file-log rotation behavior with the current logging surface.

## Best Possible Solution

Define the intended Linux service logging policy, likely starting with docs or doctor/setup guidance for journald retention, then only change generated unit stream behavior or add automatic vacuuming if maintainers explicitly want OpenClaw to own that host-level policy.

## Reproduction Assessment

No high-confidence live reproduction was run. A full reproduction would install current main as a Linux systemd user service, generate Gateway output, and compare journalctl disk usage with the rotating Gateway file log; static inspection shows the path remains plausible.

## Solution Assessment

Unclear. The issue lists plausible fixes, but the narrowest maintainable direction appears to be documented or diagnostic guidance for journald retention before changing generated units or vacuuming host journals by default.
