## Summary

Keep open. Current main still treats Linux systemd service logs as journald-facing while OpenClaw's documented rotation only covers its structured file logger; no current docs explain journal disk usage, vacuuming, or host journald retention caps.

## What This Changes

Request documentation or behavior changes so Linux systemd installs do not leave real gateway service logs growing only in journald while OpenClaw file-log rotation watches a separate file surface.

## Best Possible Solution

Document Linux/systemd journald retention as separate from OpenClaw file-log rotation, including disk-usage checks, manual vacuum commands, and persistent journald cap guidance, while leaving automatic vacuuming or output rerouting to a separate maintainer decision if desired.

## Reproduction Assessment

Yes for the source-level gap: current main points Linux service log inspection at journalctl, the generated systemd unit does not cap or redirect journald output, and the inspected docs lack journal retention guidance. A full storage-growth reproduction would require running the gateway as a systemd user service under sustained logging.

## Solution Assessment

Yes for a docs-first repair: setup documentation is one of the requested outcomes and is the narrowest maintainable fix for the current gap. Automatic journal vacuuming or changing service output should remain separate unless maintainers choose that policy.
