## Summary

Keep open. Current main still treats Linux systemd service output as journald while OpenClaw file-log rotation only covers the Gateway file log, and the inspected docs do not yet explain journald disk usage, vacuuming, or retention caps.

## What This Changes

The issue asks OpenClaw to address Linux systemd service log growth by vacuuming journald, documenting journald retention, or routing service output into the rotating OpenClaw file log.

## Best Possible Solution

Document journald as a separate Linux/systemd log-retention surface with disk-usage checks, manual vacuum examples, and host-level journald cap guidance while leaving automatic journal management or output rerouting for a separate decision.

## Reproduction Assessment

Yes for the actionable docs/source gap: current main points Linux service log inspection at `journalctl`, file-log rotation only bounds OpenClaw file logs, and the inspected docs lack journald retention guidance. A full storage-growth reproduction would require running the gateway as a systemd user service under sustained logging and checking journal disk usage.

## Solution Assessment

Yes for a docs-first repair: the issue lists setup documentation as an acceptable outcome, and it is the narrowest maintainable fix for the current gap. Automatic journal vacuuming or service output rerouting should stay out of scope unless a maintainer asks for that behavior change.
