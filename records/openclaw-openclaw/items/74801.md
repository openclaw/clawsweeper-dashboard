## Summary

Keep open. Current main has in-process rotation for OpenClaw file logs and the old cron/rotate-script path appears obsolete, but the Linux systemd journald pressure described by the report still remains: generated user units do not cap or redirect journal output, Linux diagnostics still point users at journalctl, and docs do not provide journald retention guidance.

## What This Changes

The issue requests fixing Linux systemd service log retention because OpenClaw file-log rotation does not bound journald output when service stdout/stderr are captured by systemd.

## Best Possible Solution

Define the Linux service logging policy explicitly, likely starting with user-facing journald retention guidance for systemd installs and adding generated unit or doctor/setup behavior only if maintainers choose that as the product default.

## Reproduction Assessment

Yes for the source/docs gap: current main can be inspected at buildSystemdUnit, runtime hints, console forwarding, and logging docs to see that journald is still used without retention guidance. A full disk-growth reproduction would require installing the current gateway as a Linux systemd user service and measuring journalctl disk usage over generated log volume.

## Solution Assessment

Unclear. The report's docs/setup-retention option is the narrowest maintainable first fix, while automatic journal vacuuming or changing generated StandardOutput/StandardError defaults should wait for maintainer ops-policy review.
