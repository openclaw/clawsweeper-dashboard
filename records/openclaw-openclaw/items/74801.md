## Summary

Keep open. Current main still treats Linux systemd service output as a journald surface, while OpenClaw’s documented rotation only covers its own file logger and the docs do not yet explain journal disk usage or retention controls.

## What This Changes

The issue asks OpenClaw to address Linux systemd logging pressure where journald grows while file-log rotation covers a different log file.

## Best Possible Solution

Document Linux/systemd journald retention as separate from OpenClaw file logs, including disk-usage checks, manual vacuum commands, and persistent journald cap guidance; defer automatic vacuuming or output rerouting unless maintainers explicitly choose that behavior.

## Reproduction Assessment

Yes for the source-level gap: current main points Linux service log inspection at journalctl, the generated systemd unit does not cap or redirect journald output, and the inspected docs lack journal retention guidance. A full storage-growth reproduction would require running the gateway as a systemd user service under sustained logging.

## Solution Assessment

Yes for a docs-first repair: setup documentation is one of the requested outcomes and is the narrowest maintainable fix for the current gap. Automatic journal vacuuming or changing service output should remain separate unless maintainers choose that policy.
