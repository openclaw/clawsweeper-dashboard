## Summary

Keep open. Current main has bounded OpenClaw file-log rotation, but the Linux systemd path still relies on journald for service stdout/stderr and the docs do not tell users how to cap or vacuum journald storage, so the reported storage-pressure gap remains actionable.

## What This Changes

The issue requests that OpenClaw address Linux systemd logging storage pressure because journald receives service output while the existing file-log rotation path does not bound journald growth.

## Best Possible Solution

Document Linux service logging policy explicitly: file logs rotate via `logging.maxFileBytes`, while systemd service stdout/stderr is governed by journald, with guidance for checking disk usage and setting host-level retention or running manual vacuum commands. Runtime output routing or automatic vacuuming should remain a separate maintainer decision.

## Reproduction Assessment

Yes for the source/docs gap: current main shows systemd units have no journald policy, Linux diagnostics use `journalctl`, and docs lack retention guidance. A full storage-growth reproduction would require sustained Gateway logging under a Linux systemd user service and `journalctl --disk-usage` proof.

## Solution Assessment

Yes for a narrow docs-first repair because the issue explicitly accepts docs/setup guidance and it avoids changing Linux service defaults. Automatic vacuuming or service output redirection is not clearly the best first fix without maintainer policy review.
