## Summary

Keep open. Current main still separates Linux service logs in journald from OpenClaw-managed file-log rotation, and the docs do not yet tell systemd operators how to bound or vacuum journal storage.

## What This Changes

Request to bound or document journald retention for Linux systemd gateway logs because the existing file-log rotation does not constrain journal storage.

## Best Possible Solution

Document journald as a separate Linux/systemd log-storage surface, including `journalctl --disk-usage`, manual `journalctl --vacuum-*`, and persistent journald cap settings, while leaving automatic vacuuming or stdout/stderr rerouting for explicit maintainer design.

## Reproduction Assessment

Yes for the actionable gap: current main points Linux service log inspection at `journalctl`, file-log rotation only applies to OpenClaw JSONL files, and docs lack journald retention guidance. A full storage-growth reproduction would require running the gateway as a systemd user service under sustained logging and checking `journalctl --disk-usage`.

## Solution Assessment

Yes for a docs-first repair: it satisfies one of the issue's stated expected outcomes with the narrowest maintainable change. Automatic journal vacuuming or service output routing should be a separate maintainer decision.
