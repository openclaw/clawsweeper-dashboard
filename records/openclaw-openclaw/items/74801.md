## Summary

Keep open. Current main still treats Linux systemd service logs as a journald surface while OpenClaw rotation only bounds its own file logger, and I found no journald retention or vacuum guidance in the relevant docs/source.

## What This Changes

Request that Linux/systemd log handling cover journald storage pressure instead of only OpenClaw file-log rotation.

## Best Possible Solution

Document journald as a separate Linux/systemd log-storage surface with retention and manual vacuum guidance, while keeping automatic vacuuming or service output rerouting as a separate maintainer decision.

## Reproduction Assessment

Yes for the actionable gap: current main shows Linux service logs are inspected through `journalctl`, file-log rotation only covers OpenClaw-managed log files, and docs do not cover journald retention. A full storage-growth reproduction would require running the systemd user service long enough to measure `journalctl --disk-usage`.

## Solution Assessment

Yes for a docs-first repair: adding Linux/systemd journald retention guidance is the narrowest maintainable fix that satisfies one expected outcome without changing host-wide journal policy. Automatic `journalctl --vacuum-*` behavior or unit output rerouting should be reviewed separately.
