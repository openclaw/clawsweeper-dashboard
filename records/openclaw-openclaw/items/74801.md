## Summary

Keep open. Current main still treats Linux systemd service logs as journald output while OpenClaw-managed rotation only bounds the Gateway file log, and the docs do not yet explain journal disk usage, vacuuming, or persistent journald caps.

## What This Changes

Request that Linux/systemd logging guidance address journald storage growth instead of only relying on OpenClaw file-log rotation.

## Best Possible Solution

Document journald as a separate Linux/systemd log-storage surface, including disk-usage checks, manual vacuum commands, and persistent journald cap settings, while leaving automatic vacuuming or service-output rerouting for an explicit maintainer design.

## Reproduction Assessment

Yes for the actionable gap: current main points Linux service log inspection at `journalctl`, file-log rotation only applies to OpenClaw JSONL files, and docs lack journald retention guidance. A full storage-growth reproduction would require running the Gateway as a systemd user service under sustained logging and checking `journalctl --disk-usage`.

## Solution Assessment

Yes for a docs-first repair: it satisfies one of the issue's stated expected outcomes with the narrowest maintainable change. Automatic journal vacuuming or service output routing should be handled separately if maintainers want that behavior.
