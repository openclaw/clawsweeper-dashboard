## Summary

Keep open. Current main still rotates only the OpenClaw file logger, while Linux service hints direct operators to journald and I found no generated systemd unit policy or docs guidance that bounds journal storage.

## What This Changes

The issue asks OpenClaw to address Linux systemd journal growth when file-log rotation watches a different OpenClaw-managed log surface.

## Best Possible Solution

Document journald retention as a separate Linux/systemd log-storage surface, including `SystemMaxUse`/`RuntimeMaxUse` and manual `journalctl --vacuum-*` options, while leaving automatic vacuuming or output rerouting for a separate maintainer decision.

## Reproduction Assessment

Yes for the actionable gap: current main shows Linux service logs are surfaced through `journalctl` and file-log rotation/docs do not cover journald retention. A full disk-growth reproduction would require a sustained Linux systemd user-service run plus `journalctl --disk-usage`.

## Solution Assessment

Yes for a docs-first repair: it is the narrowest maintainable fix that satisfies one of the issue's expected outcomes without editing host-wide journal policy. Automatic vacuuming or service output changes should not be bundled into that repair.
