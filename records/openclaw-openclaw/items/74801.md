## Summary

Keep open. Current main has bounded OpenClaw file-log rotation and the old rotate-script/cron mechanism appears obsolete, but the Linux systemd journald pressure reported here remains unaddressed: generated user units do not set output routing or journald policy, runtime/docs still direct users to `journalctl`, and docs do not mention journald retention controls.

## What This Changes

The issue asks OpenClaw to address Linux systemd installs where real Gateway output accumulates in journald while the file-log rotation path does not bound journald disk use.

## Best Possible Solution

Document Linux service logging policy explicitly and add journald retention guidance for systemd installs, then consider generated unit output routing or automated vacuuming only if maintainers choose that as the default product behavior.

## Reproduction Assessment

Yes for the source/docs gap: current main can be inspected to show systemd units have no journald policy, runtime hints use `journalctl`, and docs lack retention guidance. A full storage-growth reproduction would require running the Gateway as a Linux systemd user service under sustained logging and checking `journalctl --disk-usage`.

## Solution Assessment

Unclear for the broader policy choice. The narrowest maintainable first fix is docs/setup guidance for `SystemMaxUse`/`RuntimeMaxUse` or manual `journalctl --vacuum-*`; automatic vacuuming or stdout/stderr rerouting needs maintainer ops-policy review.
