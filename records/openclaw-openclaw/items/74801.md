## Summary

Keep open. Current main still treats Linux systemd logs as a journald surface while OpenClaw's documented rotation only covers its own JSONL file logger, so the reported storage-retention gap remains valid and narrow enough for a docs-first repair.

## What This Changes

The issue asks OpenClaw to address journald storage growth for Linux systemd service logs, either through rotation/vacuum guidance or file logging behavior.

## Best Possible Solution

Document journald retention as a separate Linux/systemd log-storage surface, including `journalctl --disk-usage`, `SystemMaxUse`/`RuntimeMaxUse`, and manual `journalctl --vacuum-*` options, while leaving automatic vacuuming or output rerouting for a separate design decision.

## Reproduction Assessment

Yes for the actionable gap: current main shows Linux service logs are surfaced through `journalctl`, while OpenClaw rotation/docs cover only the file logger. A full disk-growth reproduction would require a sustained Linux systemd user-service run plus `journalctl --disk-usage`.

## Solution Assessment

Yes for a docs-first repair: it is the narrowest maintainable fix that satisfies one of the issue's expected outcomes without editing host-wide journal policy. Automatic vacuuming or service output changes should not be bundled into that repair.
