## Summary

Keep open. Current main still bounds OpenClaw file logs, but Linux systemd service output is an expected journald surface and the generated unit/docs do not give operators a journald retention cap or vacuum guidance.

## What This Changes

Request to bound or document Linux systemd journald retention because OpenClaw file-log rotation does not constrain journal storage.

## Best Possible Solution

Document journald as a separate Linux/systemd retention surface with SystemMaxUse/RuntimeMaxUse and journalctl --vacuum-* guidance, while leaving automatic vacuuming or output redirection for explicit maintainer ops review.

## Reproduction Assessment

Yes for the actionable gap: current main can be inspected to show Linux systemd logs are surfaced through journalctl while file rotation only applies to OpenClaw file logs and docs lack journald retention. A full disk-growth reproduction would require a sustained systemd run and journalctl --disk-usage.

## Solution Assessment

Yes for a docs-first repair; it is the narrowest maintainable fix that addresses the misleading storage expectation without changing host-wide journal policy. Automatic journald vacuuming or service output rerouting should be a separate maintainer decision.
