## Summary

Keep open. Current main still caps OpenClaw file logs, but Linux systemd service output is surfaced through journald and there is no generated unit policy or docs guidance that bounds journal storage.

## What This Changes

Request to update OpenClaw's Linux/systemd logging behavior or documentation so journald growth is bounded or explicitly managed.

## Best Possible Solution

Document journald retention as a separate Linux/systemd log-storage surface with `SystemMaxUse`/`RuntimeMaxUse` and manual `journalctl --vacuum-*` options, while leaving automatic vacuuming or output rerouting for explicit maintainer ops review.

## Reproduction Assessment

Yes for the actionable gap: current main can be inspected to show systemd logs are surfaced through `journalctl` while file-log rotation and docs do not cover journald retention. A full disk-growth reproduction would require a sustained Linux systemd user-service run and `journalctl --disk-usage`.

## Solution Assessment

Yes for a docs-first repair; it is the narrowest maintainable fix that addresses the misleading storage expectation without changing host-wide journal policy. Automatic vacuuming or changing service output should be a separate maintainer decision.
