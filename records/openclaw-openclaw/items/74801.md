## Summary

Keep open. Current main still routes Linux service log inspection through journald while OpenClaw-managed rotation only bounds the Gateway file log, and the docs do not yet tell systemd operators how to inspect, vacuum, or cap journal storage.

## What This Changes

Request to correct Linux service log retention behavior or documentation because file-log rotation does not bound journald storage for systemd Gateway services.

## Best Possible Solution

Document journald as a separate Linux/systemd log-storage surface, including disk-usage checks, manual vacuum commands, and persistent retention caps, while leaving automatic vacuuming or stdout/stderr rerouting for explicit maintainer design.

## Reproduction Assessment

Yes for the actionable gap: current main points Linux service log inspection at `journalctl`, file-log rotation only applies to OpenClaw file logs, and docs lack journald retention guidance. A full growth repro would require running the Gateway as a systemd user service under sustained logging and checking `journalctl --disk-usage`.

## Solution Assessment

Yes for a docs-first repair: it satisfies one of the issue's stated expected outcomes with the narrowest maintainable change. Automatic journal vacuuming or service output routing would be broader operational behavior and should be a separate maintainer decision.
