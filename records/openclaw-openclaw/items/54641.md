## Summary

Keep open: the branch targets a real configure-auth gap, but the current patch can change an existing user's primary model during provider reauth and still lacks after-fix real behavior proof.

## What This Changes

The PR refactors configure auth to run the default-model prompt after provider-backed auth, loads provider catalog context, and adds an OpenRouter regression test for preserving `openrouter/free`.

## Best Possible Solution

Preserve existing `agents.defaults.model.primary` as the default choice during provider reauth, still offer a provider-scoped model picker for fresh/no-primary OpenRouter setup, and cover both paths with regression tests plus redacted real configure proof.

## Reproduction Assessment

Yes from source inspection, not a live run: current main's fresh configure provider-auth path can apply OpenRouter's `openrouter/auto` default and exits before a default-model prompt. The PR's mocked regression test exercises that intended repair path.

## Solution Assessment

No as written: adding the provider-scoped prompt is the right direction for fresh OpenRouter setup, but it must not default existing configs away from their current primary model during reauth.
