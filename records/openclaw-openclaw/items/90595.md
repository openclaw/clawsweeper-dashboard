## Summary

Keep open: current main and v2026.6.8 still route restart-interrupted and retryable cron errors through ordinary `error` state and failed notification delivery, while the merged auth-cooldown fix only addresses one adjacent failure mode.

## What This Changes

The issue asks OpenClaw to stop sending misleading cron `failed` notifications for retrying or gateway-restart-interrupted work that may still complete later.

## Best Possible Solution

Define one canonical cron outcome model where retrying or restart-interrupted work is not rendered as a final failed notification, while true terminal failures remain visible to operators.

## Reproduction Assessment

Yes. A current-main cron job interrupted by restart is source-reproducible through `start()` and `markInterruptedStartupRun`, and any cron `error` event is enough to drive the current failed notification path; I did not run a live Feishu restart scenario.

## Solution Assessment

Unclear. Distinguishing retrying/interrupted work from final failure is the right direction, but maintainers need to choose the durable status, counter, retry, and alert semantics before implementation.
