## Summary

Keep open: current main still records restart-interrupted and transient retry cron attempts as ordinary error outcomes, and the notification layer still emits `failed` messages for any cron `error` event. The related auth-profile contamination report was fixed by a merged PR, but that does not settle this issue's remaining alert semantics.

## What This Changes

The issue asks for cron notifications to distinguish or suppress retrying/reload-interrupted work instead of sending contradictory failed/completed alerts for the same scheduling window.

## Best Possible Solution

Define one canonical cron terminal-state model where retrying or restart-interrupted work is not rendered as a final failed notification, while preserving explicit operator alerts for true final failures.

## Reproduction Assessment

Yes, source-reproducible: current main records restart interruptions and retryable transient failures as `error` states, then the gateway notification layer sends failed notifications for those `error` events. I did not run a live Feishu or gateway-restart scenario in this read-only review.

## Solution Assessment

Unclear: the requested direction is valid, but the best fix needs maintainer choice between final-only alerts, a distinct interrupted/retrying state, or preserving per-attempt alerts with clearer wording. The current implementation is not the best end state because it conflates terminal failure with retryable or restart-interrupted work.
