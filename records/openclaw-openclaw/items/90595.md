## Label Justifications

- `P2`: The issue causes misleading operational cron alerts for real users, but the blast radius is bounded to cron status/notification semantics rather than a core outage.
- `impact:other`: The concrete impact is alert fatigue and ambiguous cron failure semantics, which do not fit the narrower owned impact labels cleanly.

## Summary

Keep open: current main and v2026.6.8 still classify restart-interrupted cron work as ordinary `error`/`failed` outcomes, and retryable cron errors still increment the same failure counters before retry scheduling. The merged auth-cooldown work in https://github.com/openclaw/openclaw/pull/91187 fixes an adjacent provider-state leak, but it does not resolve this issue's cron alert-semantics problem.

## What This Changes

The issue asks OpenClaw to distinguish interrupted or retrying cron work from terminal failures so hot reload and transient retry paths do not produce misleading failed notifications.

## Best Possible Solution

Define one maintainer-approved cron interruption and retry outcome policy, then align cron state, counters, retry scheduling, run history, protocol/status schema, and notification copy to that policy.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a persisted in-flight cron run after gateway restart flows through `markInterruptedStartupRun`, becomes `error`, increments `consecutiveErrors`, and is eligible for ordinary failed notification text; I did not run a live Feishu hot-reload scenario.

## Solution Assessment

Unclear until maintainers choose the cron semantics. A hot-reload opt-out alone would reduce one trigger but would not fix misleading terminal-failure alerts; the maintainable path is a first-class interrupted/retrying policy coordinated across state and notifications.
