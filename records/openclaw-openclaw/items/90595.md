## Summary

Keep open: current main and v2026.6.8 still classify restart-interrupted and retryable cron outcomes as ordinary `error` runs, and the Gateway notification path still renders any cron error as `failed`. The merged auth-cooldown PR fixed an adjacent provider-state leak but does not solve the alert-semantics problem reported here.

## What This Changes

The issue asks OpenClaw to stop sending ambiguous cron `failed` notifications for hot-reload interruptions and retryable runs that may still resume or later complete.

## Best Possible Solution

Define one maintainer-approved cron interruption/retry outcome policy, then align status, counters, backoff, run history, Gateway protocol, docs, and notification text to that policy.

## Reproduction Assessment

Yes for a source-level reproduction: current main records restart-interrupted cron runs as `error` and sends failed notification text for `error` finished events. I did not execute a live Feishu/Gateway hot-reload scenario in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the cron semantics. A hot-reload opt-out alone would not fix misleading terminal-failure alerts; the maintainable path is a first-class interrupted/retrying policy across cron state and notifications.
