## Label Justifications

- `P2`: The issue causes misleading operational cron alerts for real users, but the blast radius is bounded to cron status/notification semantics rather than a core outage.
- `impact:other`: The concrete impact is alert fatigue and misleading cron failure semantics, which do not fit the narrower owned impact labels cleanly.

## Summary

Keep open: current main and v2026.6.8 still model gateway-restart interruptions and retryable cron failures as ordinary cron errors, then send “failed” notifications. The merged auth-cooldown fix at https://github.com/openclaw/openclaw/pull/91187 resolves an adjacent shared-provider-state problem, but it does not resolve the cron alert/status semantics reported here.

## What This Changes

The issue asks OpenClaw to distinguish interrupted or still-retrying cron work from terminal failures so hot reload and retry paths do not produce contradictory failed/completed notifications.

## Best Possible Solution

Define one canonical cron interruption/retry outcome policy across status, counters, backoff, run history, and notification text; use the existing `gateway.reload.mode="off"` for manual-restart deployments unless maintainers choose a new config surface.

## Reproduction Assessment

Yes for a source-level reproduction: current main converts gateway-restart-interrupted cron runs to `error`, increments error accounting, and formats any cron error notification as failed. I did not run a live Feishu/Gateway hot-reload scenario in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the cron semantics. A hot-reload opt-out already exists, so the maintainable fix is a first-class interrupted/retrying policy rather than another config flag alone.
