- `impact:other`: The concrete impact is misleading operational alerting for cron automation, which does not fit the more specific owned impact labels.

## Summary

Keep open: current main and v2026.6.1 still treat interrupted or retrying cron attempts as error events and route them through failed notifications, so the reported alert-fatigue behavior remains unresolved.

## What This Changes

The issue asks OpenClaw to stop sending terminal-looking cron failed notifications for hot-reload interruptions and transient retry attempts that later complete successfully.

## Best Possible Solution

Define terminal-versus-transient cron notification semantics so retrying or reload-interrupted work is reported as retrying/interrupted, or stays silent until final failure, while documenting the existing `gateway.reload.mode = "off"` workaround.

## Reproduction Assessment

Yes, from source for cron config changes/restarts and transient retry attempts: current main records those attempts as error events and the notification layer sends failed alerts. I did not prove the exact whitespace-only mtime path because parsed no-op config changes appear to be ignored.

## Solution Assessment

No, current main is not the best end state because it conflates terminal failure, transient retry, and restart interruption in the same failed notification path. The safer fix needs a maintainer decision on final-only alerts versus a distinct retrying/interrupted state.
