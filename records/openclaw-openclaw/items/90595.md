## Label Justifications

- `impact:other`: The concrete impact is operational alert fatigue and misleading cron failure semantics, which does not fit the narrower owned impact labels cleanly.

## Summary

Keep open: current main and v2026.6.8 still treat cron restart interruptions and retryable cron errors as ordinary `error` outcomes that can produce `failed` notifications; the merged auth-cooldown isolation work fixes an adjacent provider-state problem but not the alert-semantics problem reported here.

## What This Changes

The issue requests cron notifications that distinguish reload interruptions, retrying/resuming work, and terminal failures so operators do not receive misleading failed alerts for work that may still complete.

## Best Possible Solution

Define one maintainer-approved cron outcome and alert policy that separates interrupted/retrying/resuming work from terminal execution failure, then coordinate that policy with the open restart-counter and execution-vs-delivery status issues.

## Reproduction Assessment

Yes for a source-level reproduction: current main emits `failed` notifications for any cron `error`, and restart-interrupted runs are still converted into `error` state with the gateway-restart message. I did not run a live Feishu/hot-reload scenario in this read-only pass.

## Solution Assessment

No final solution is selected yet. The report's config opt-out suggestion is only one option; a first-class cron interrupted/retrying/terminal-failure policy is the safer maintainer decision point before any code change.
