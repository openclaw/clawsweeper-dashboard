- `impact:message-loss`: The central user impact is duplicated and misleading channel notifications for cron outcomes.

## Summary

Keep open: current main and v2026.6.9 still record gateway-restart-interrupted cron runs as `error` and send `failed` notifications for any cron `error`; the merged auth-cooldown fix resolved an adjacent provider-state leak but not the alert semantics reported here.

## What This Changes

The issue asks OpenClaw to distinguish cron restart interruptions and retryable/transient states from real terminal failures in cron notifications and state.

## Best Possible Solution

Define one maintainer-approved cron interruption/retry outcome policy, then align persisted state, counters, retry/backoff, run logs, Gateway schema, docs, and notification rendering to that policy.

## Reproduction Assessment

Yes for source-level reproduction: current main records restart-interrupted cron runs as `error` and the Gateway notification path renders cron `error` events as failed notifications. I did not run a live Feishu/Gateway hot-reload scenario in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the cron semantics. A hot-reload opt-out or message text tweak alone is too narrow; the maintainable solution is a first-class interrupted/retrying policy across cron state and notifications.
