- `impact:other`: The concrete impact is alert fatigue and misleading cron failure semantics, which do not fit the narrower owned impact labels cleanly.

## Summary

Keep open: current main and the latest release still turn restart-interrupted and retryable cron outcomes into ordinary cron `error`/`failed` notifications. The merged auth-cooldown isolation PR fixes an adjacent provider-state leak, but it does not solve the alert-semantics problem reported here.

## What This Changes

Request to make cron notifications distinguish terminal failures from retrying or gateway-restart-interrupted work so operators do not receive false failed alerts followed by successful completion alerts.

## Best Possible Solution

Define one maintainer-approved cron interruption/retry outcome policy that separates terminal execution failures from interrupted or still-recoverable work, then align status, counters, backoff, run history, and notification text to that policy.

## Reproduction Assessment

Yes for a source-level reproduction: current main converts gateway-restart-interrupted cron runs to `error`, increments the error counter, and sends `failed` notification text for any cron error event. I did not run a live Feishu/Gateway hot-reload scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the cron semantics. A config opt-out alone would not fix misleading terminal-failure alerts; the more maintainable direction is a first-class interrupted/retrying policy coordinated across cron state and notifications.
