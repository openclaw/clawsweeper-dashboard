## Summary

Keep open: current main and v2026.6.8 still route retryable cron errors and restart-interrupted runs through ordinary `error` state, and the gateway still renders any cron error event as `Cron job ... failed`. The related auth-profile cooldown contamination was fixed separately, but the alert semantics remain unresolved.

## What This Changes

The issue reports noisy cron failure notifications when retry, cooldown, and hot reload or transient provider failures make a run appear failed even though work may later continue or complete.

## Best Possible Solution

Define cron alert semantics so restart-interrupted or retrying work is not rendered as a final failed notification, while preserving true final failure alerts.

## Reproduction Assessment

Yes, source-reproducible: current main still records retryable cron errors and restart interruptions as `error`, then the gateway sends `Cron job ... failed` notifications for any cron error event. I did not run a live Feishu or hot-reload scenario in this read-only review.

## Solution Assessment

Unclear: the requested distinction is valid, but the best fix needs maintainer choice between final-only alerts, a distinct interrupted/retrying state, or explicit preservation of per-attempt alerts with clearer wording.
