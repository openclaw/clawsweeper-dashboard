## Summary

Keep open. Current main still builds Anthropic-compatible payloads from `convertAnthropicMessages(...)` without a final non-empty guard, MiniMax uses that transport, and open PR #74731 is already the focused closing implementation candidate rather than a reason to close this issue before merge.

## What This Changes

MiniMax M2.7 can intermittently receive an empty Anthropic-compatible `messages` payload after chained tool calls, causing provider error 400/2013 and leaving the session state flaky until repaired or switched.

## Best Possible Solution

Review and land PR #74731 or an equivalent final non-empty guard in `src/agents/anthropic-transport-stream.ts`, with regression coverage in `src/agents/anthropic-transport-stream.test.ts`; keep the broader `/new` empty-prompt race tracked separately under #73926.

## Reproduction Assessment

Yes for the core failure path, but no for a deterministic live MiniMax session. Current main can build an Anthropic-compatible request whose converted `messages` array is empty when replay content is empty or fully filtered, matching MiniMax's reported 2013 failure condition.

## Solution Assessment

Yes. A final guard at the Anthropic-compatible transport payload boundary is the narrowest maintainable fix because it covers MiniMax and any other strict Anthropic-compatible endpoint without adding MiniMax-specific retry logic or session-store repair.
