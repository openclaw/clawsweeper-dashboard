## Summary

Keep open. Current main still forwards the Anthropic-compatible converted message array without a final non-empty guard, MiniMax uses that transport, and open PR #74731 is the focused closing implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports intermittent MiniMax M2.7 400/2013 empty-chat-content errors after tool-call chains and asks for a provider-layer guard against empty Anthropic-compatible message payloads.

## Best Possible Solution

The desired end state is a final non-empty Anthropic Messages payload invariant in `src/agents/anthropic-transport-stream.ts`, covered by regression tests in `src/agents/anthropic-transport-stream.test.ts`, while keeping the broader `/new` empty-prompt race tracked separately under #73926.

## Reproduction Assessment

Yes for the core failure path, but not for a deterministic live MiniMax session. Current main can build an Anthropic-compatible request with an empty converted `messages` array when replay content is empty or fully filtered, matching the reported MiniMax 2013 condition.

## Solution Assessment

Yes for the linked PR direction. A guard at the Anthropic-compatible transport payload boundary is the narrowest maintainable fix because it covers MiniMax and other strict Anthropic-compatible endpoints without adding provider-specific retry or transcript repair policy.
