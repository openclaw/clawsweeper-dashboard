## Summary

Keep open because current main still forwards `convertAnthropicMessages(...)` directly into the Anthropic-compatible request body without a final non-empty guard, and open PR #74731 is linked with closing intent for this issue.

## What This Changes

The issue reports intermittent MiniMax M2.7 400/2013 errors after tool-call chains when Anthropic-compatible request conversion can leave `messages: []`.

## Best Possible Solution

Review and land PR #74731, or an equivalent final non-empty guard at `buildAnthropicParams` with focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes for the source-level failure path, unclear for a deterministic live MiniMax session. Current main can still convert empty or fully filtered replay history into `messages: []` and submit it through the Anthropic-compatible transport.

## Solution Assessment

Yes. A final guard at the Anthropic-compatible payload boundary is the narrowest maintainable fix because it covers MiniMax and other strict compatible endpoints without MiniMax-specific retries or transcript repair.
