## Summary

Keep open. Current main still has the reported Anthropic-compatible empty-payload risk for MiniMax: conversion can return an empty message array, the payload builder forwards it unchanged, and MiniMax is registered on that transport. The linked closing PR is still open and unmerged, so this issue should remain open until that guard lands or is replaced.

## What This Changes

The issue reports intermittent MiniMax M2.7 400/2013 errors after multi-tool-call turns and requests a provider-layer guard that prevents empty Anthropic-compatible `messages: []` payloads from being sent.

## Best Possible Solution

Land PR #74731 or an equivalent transport-level invariant in `src/agents/anthropic-transport-stream.ts` that guarantees Anthropic-compatible provider payloads are never submitted with an empty `messages` array, with focused regression coverage in `src/agents/anthropic-transport-stream.test.ts` and a changelog entry when the user-facing fix ships.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for a deterministic live MiniMax failure because the report is intermittent and lacks a minimal transcript; yes for the core failure path because current main can still construct and forward an empty converted Anthropic `messages` array to the provider boundary.

## Solution Assessment

Is this the best way to solve the issue? Yes, a final guard at the Anthropic-compatible transport payload boundary is the narrowest maintainable solution because it protects MiniMax and other compatible providers after conversion without MiniMax-specific retry policy or transcript repair behavior.
