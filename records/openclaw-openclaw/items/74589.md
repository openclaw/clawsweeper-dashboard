## Summary

Keep open. Current main still allows the Anthropic-compatible transport to submit `messages: []` after conversion/filtering, MiniMax M2.7 is routed through that transport, and open PR #74731 is already the focused closing implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Guard MiniMax/Anthropic-compatible provider submissions so tool-call-chain transcript filtering cannot send an empty `messages` array and leave the session in a flaky failed state.

## Best Possible Solution

Land PR #74731 or an equivalent final non-empty guard in `src/agents/anthropic-transport-stream.ts`, with focused regression coverage in `src/agents/anthropic-transport-stream.test.ts` and a user-facing changelog entry; keep the broader `/new` empty-prompt race tracked separately under #73926.

## Reproduction Assessment

Yes for the core failure path, but not for a deterministic live MiniMax session. Current main can still build an Anthropic-compatible request whose converted `messages` array is empty when the input history is empty or fully filtered, matching MiniMax's reported 2013 condition.

## Solution Assessment

Yes. A final guard at the Anthropic-compatible transport payload boundary is the narrowest maintainable fix because it covers MiniMax and other strict Anthropic-compatible endpoints without adding MiniMax-specific retry logic or session-store repair.
