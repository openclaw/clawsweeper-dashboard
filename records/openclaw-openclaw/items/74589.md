## Summary

Keep open because current main still forwards the Anthropic-compatible converted message array without a final non-empty guard, while open PR #74731 is the focused implementation candidate and has not merged.

## What This Changes

The issue requests a provider-boundary guard so MiniMax M2.7 cannot receive an empty Anthropic-compatible `messages` payload after tool-call replay filtering.

## Best Possible Solution

Land PR #74731 or an equivalent final non-empty guard at the Anthropic-compatible payload boundary, while keeping the broader `/new` empty-prompt race under #73926.

## Reproduction Assessment

Yes for the core failure path, but not for a deterministic live MiniMax session. Current main can still build an Anthropic-compatible request whose converted `messages` array is empty when replay history is empty or fully filtered.

## Solution Assessment

Yes. A final guard in `buildAnthropicParams` or an immediately adjacent payload helper is the narrowest maintainable fix because it covers MiniMax and other strict Anthropic-compatible endpoints without provider-specific retry or session-store repair.
