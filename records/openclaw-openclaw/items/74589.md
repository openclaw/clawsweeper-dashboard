## Summary

Keep open. Current main still forwards the Anthropic-compatible converted message array directly into the provider request without a final non-empty invariant, MiniMax is configured to use that transport, and open PR #74731 is the linked closing implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue asks OpenClaw to prevent MiniMax Anthropic-compatible requests from sending `messages: []` after tool-call chains, ideally with a provider-boundary fallback guard.

## Best Possible Solution

Review and land PR #74731 or an equivalent final non-empty guard in `src/agents/anthropic-transport-stream.ts`, with regression coverage in `src/agents/anthropic-transport-stream.test.ts`; keep the broader `/new` empty-prompt race tracked separately under #73926.

## Reproduction Assessment

Yes for the core provider failure condition: constructing an Anthropic-compatible transport context whose replay history is empty or fully filtered can produce `messages: []` on current main. The exact live MiniMax tool-chain sequence is still intermittent and was not deterministically reproduced here.

## Solution Assessment

Yes. A final guard at the Anthropic-compatible transport payload boundary is the narrowest maintainable fix because it covers MiniMax and any other strict Anthropic-compatible endpoint without MiniMax-specific retry policy or transcript-store mutation.
