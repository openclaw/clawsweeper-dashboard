## Summary

Keep open. Current main still can forward an empty Anthropic-compatible `messages` array to the provider, MiniMax uses that transport, and open PR #74731 is already the focused closing implementation candidate for this issue.

## What This Changes

Prevent MiniMax M2.7 on the Anthropic-compatible transport from sending an empty `messages` array after tool-call chains, avoiding 400/2013 provider errors and session transcript pollution.

## Best Possible Solution

Land PR #74731 or an equivalent final non-empty guard in `src/agents/anthropic-transport-stream.ts`, with regression coverage in `src/agents/anthropic-transport-stream.test.ts`; keep the broader `/new` empty-prompt race tracked separately under #73926.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for a deterministic live MiniMax session because the report is intermittent and lacks a minimal transcript; yes for the core code path because current main can build and submit an empty converted Anthropic `messages` array.

## Solution Assessment

Is this the best way to solve the issue? Yes: a final guard at the Anthropic-compatible transport payload boundary is narrower and more maintainable than MiniMax-specific retry logic or session-store repair.
