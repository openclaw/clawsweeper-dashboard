## Summary

Keep open. Current main still has a provider-boundary path where Anthropic-compatible message conversion can produce an empty payload, MiniMax is routed through that transport, and the open closing PR is the implementation candidate rather than a reason to close before merge.

## What This Changes

MiniMax M2.7 via minimax/minimax-portal can intermittently return 400/2013 after tool-call chains when the Anthropic-compatible transport submits an empty converted `messages` payload.

## Best Possible Solution

Land PR #74731 or an equivalent final non-empty guard in `src/agents/anthropic-transport-stream.ts`, with focused regression coverage and a changelog entry; keep the broader `/new` empty-prompt race tracked separately under #73926.

## Reproduction Assessment

Yes for the core failure path: current main can build an Anthropic-compatible request whose converted `messages` array is empty when the input history is empty or fully filtered. The exact live MiniMax tool-chain trigger remains intermittent rather than a deterministic end-to-end reproduction.

## Solution Assessment

Yes. A final guard at the Anthropic-compatible transport payload boundary is the narrowest maintainable fix because it covers MiniMax and other strict compatible endpoints without adding MiniMax-specific retries or session-store repair.
