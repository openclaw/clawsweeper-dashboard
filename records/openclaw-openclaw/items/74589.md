## Summary

Keep open. Current main still forwards the Anthropic-compatible converted message list directly, MiniMax M2.7 uses that transport, and open PR #74731 specifically carries the closing implementation rather than making this issue closeable before merge.

## What This Changes

Prevent MiniMax M2.7 Anthropic-compatible requests from sending an empty `messages` payload after tool-call chains, avoiding 400/2013 provider errors and session pollution.

## Best Possible Solution

Land PR #74731 or an equivalent transport-boundary non-empty message guard in `src/agents/anthropic-transport-stream.ts`, with focused regression coverage and a user-facing changelog entry; keep the broader `/new` empty-prompt race tracked separately under #73926.

## Reproduction Assessment

Yes for the core failure path, but not for a deterministic live MiniMax session. Current main can construct an Anthropic-compatible request with `messages: []` when the original history is empty or every replay item is filtered away, matching the reported MiniMax 2013 condition.

## Solution Assessment

Yes. A final guard at the Anthropic-compatible payload boundary is the narrowest maintainable fix because it covers MiniMax and other strict Anthropic-compatible endpoints without adding MiniMax-specific retry logic or session-store repair.
