## Summary

Keep open. The reported Signal/Codex silent-reply shape is still source-reproducible on current main and overlaps broader origin-routing work, but a later assigned maintainer comment explicitly says the direct-chat `message_tool` policy is being revisited, so this is not safe duplicate cleanup.

## What This Changes

Signal direct-session final replies generated through a Codex/internal runner can remain on an internal surface unless the model explicitly sends through the message tool.

## Best Possible Solution

Make a maintainer-approved direct-chat visible-reply policy that preserves the Codex message-tool benefits while ensuring one-to-one Signal-originated turns do not silently lose ordinary final replies, with stale-route safeguards and regression coverage.

## Reproduction Assessment

Yes, at source level: current main has a Codex direct-chat `message_tool` default plus an internal-WebChat/no-`ExplicitDeliverRoute` gate that can suppress ordinary final delivery. I did not run a live Signal gateway reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation path. The desired user outcome is valid, but the safest fix needs maintainer policy on direct-chat defaults, explicit delivery, and stale cross-surface route safeguards.
