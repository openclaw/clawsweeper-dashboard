## Summary

Keep open: the issue is recent, contributor-authored, unprotected, and current main still has an intentional internal/webchat routing guard that can keep final replies on the internal surface unless a captured external route is explicitly marked deliverable. The related closed Signal issue and merged message-tool fix cover adjacent paths, but they do not prove this normal-final-reply path is fixed.

## What This Changes

The issue reports that Signal-originated final assistant replies can remain on an internal/Codex surface instead of being delivered back to the originating Signal recipient.

## Best Possible Solution

Define a narrow turn-source delivery invariant for internal/Codex resumes that marks the original Signal route as explicit for that turn while preserving the existing no-external-route safety rule for ordinary webchat or UI-originated sends.

## Reproduction Assessment

No, not as a high-confidence end-to-end reproduction. Source inspection shows a credible current-main path where Signal captures a delivery route but an internal/webchat final reply will not route externally unless ExplicitDeliverRoute is true.

## Solution Assessment

Unclear. The requested captured-origin invariant is the right target, but the implementation needs maintainer judgment because existing tests intentionally prevent internal/webchat turns from inheriting external routes without an explicit delivery decision.
