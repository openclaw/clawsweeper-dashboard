## Summary

Keep open for maintainer review: the reported silent Signal reply shape is plausible, but current main also intentionally keeps Codex-harness direct final text private in message-tool mode and intentionally blocks external-origin routing from internal webchat unless an explicit delivery route is set. Changing that invariant is a product/routing decision, not a conservative cleanup close or safe automatic bug fix.

## What This Changes

The issue requests that Signal-originated turns always deliver normal final assistant replies back to the original Signal route, even when processed by Codex/vscode/internal surfaces without an explicit message-tool send.

## Best Possible Solution

Decide whether Signal/Codex direct sessions should bypass the Codex message-tool-only default or mark these internal resumes with an explicit captured delivery route, then add focused routing coverage for the chosen invariant.

## Reproduction Assessment

Yes at source level for the policy class: current main has tests proving internal-webchat external-origin replies do not route without ExplicitDeliverRoute and Codex direct sessions can suppress final text in message-tool-only mode. No live Signal deployment reproduction was performed.

## Solution Assessment

No for an automatic fix: the requested behavior conflicts with documented and tested Codex visible-reply defaults, so the maintainable solution needs a maintainer decision on whether to change the default, add a narrower Signal/internal-route exception, or keep the message-tool contract.
