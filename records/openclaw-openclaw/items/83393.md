## Summary

Keep open: the report maps to a real current-main delivery-policy edge, but current docs and tests intentionally make Codex direct-chat replies message-tool-only unless `messages.visibleReplies: "automatic"` is configured, so changing Signal final fallback behavior needs maintainer product review rather than an automated bug fix.

## What This Changes

Request that final assistant text from a Signal-originated turn be delivered back to Signal even when the turn runs through Codex/internal surfaces and the model does not call the message tool.

## Best Possible Solution

Maintainers should choose one direct-Codex delivery contract, then align Signal docs, Codex harness docs, dispatch tests, and runtime behavior around either automatic Signal fallback or explicit message-tool-only delivery with clear diagnostics.

## Reproduction Assessment

Yes, at source level: Codex direct-chat defaults select `message_tool_only`, current dispatch tests assert no automatic final reply is queued, and the final loop skips ordinary final payloads under that policy. I did not run a live Signal repro, so the exact production incident remains unverified end to end.

## Solution Assessment

Unclear. The suggested invariant is user-friendly for Signal DMs, but it conflicts with the documented Codex message-tool-only direct-chat contract and needs a maintainer decision before changing dispatch policy.
