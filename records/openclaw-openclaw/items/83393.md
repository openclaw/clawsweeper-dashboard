## Summary

Keep open: current main still has a source-level path where a Signal-originated route is available, but final-reply routing is explicitly suppressed when the active surface is internal WebChat/Codex without `ExplicitDeliverRoute`. That matches the reported silent non-delivery shape closely enough for maintainer follow-up rather than cleanup closure.

## What This Changes

Ensure a Signal-originated direct turn that runs on a Codex/internal surface returns normal final assistant replies to the original Signal route without requiring an explicit message tool call.

## Best Possible Solution

Define a narrow route contract for live inbound turns so the captured channel/account/target/thread survives internal Codex/WebChat execution and final replies auto-deliver to Signal, while preserving stale-route protections and the current WebChat/TUI message-tool sink.

## Reproduction Assessment

Yes at source level. Signal inbound contexts carry `OriginatingChannel`/`OriginatingTo`, and current routing code plus tests show that an internal WebChat/Codex surface suppresses route-to-origin unless `ExplicitDeliverRoute` is true.

## Solution Assessment

Unclear as an automated fix. The suggested invariant is directionally right, but the safe implementation should not simply remove the internal-webchat guard; it should distinguish live captured inbound routes from stale persisted fallback routes and add focused Signal/internal-surface regression coverage.
