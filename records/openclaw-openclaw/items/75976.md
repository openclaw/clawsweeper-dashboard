## Summary

Keep open: the PR targets a real current-main plugin reload gap, but this head should not merge because source-changing plugin install/update/uninstall writes only force restart for in-process config notifications and can still hot-reload or miss restarts when run from a separate CLI process.

## What This Changes

The PR adds Gateway plugin runtime hot reload for plugin enable/disable management paths, keeps source-changing plugin operations restart-backed in the CLI write intent, clears stale plugin runtime state, and updates docs, changelog, and regression tests.

## Best Possible Solution

Keep the enable/disable hot reload path, but make source-changing plugin install/update/uninstall operations queue a Gateway restart through a durable cross-process signal before merging.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run `openclaw plugins install` or `openclaw plugins update` in a separate process while a Gateway is watching config: the Gateway watcher receives no `afterWrite` restart intent and evaluates only changed config paths.

## Solution Assessment

No. The reload-versus-restart boundary is the right direction, but the current implementation relies on process-local write intent for source changes and needs a cross-process restart path or equivalent durable marker.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep source-changing plugin writes restart-backed:** `src/gateway/config-reload-plan.ts:122`
  - body: The restart intent added to install/update/uninstall is delivered through `ConfigWriteNotification`, which only exists for in-process writes. When `openclaw plugins install/update/uninstall` runs as a separate CLI process, the Gateway file watcher rereads the config without `afterWrite`; with the new broad `plugins` hot rule, `plugins.allow` or `plugins.entries.*` changes take the hot-reload path instead of queuing a restart, and update-only managed-index changes may have no restartable config diff at all. Please persist or otherwise signal the restart across process boundaries before making `plugins` hot by default.
  - confidence: 0.9
