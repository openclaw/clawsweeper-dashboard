## Label Justifications

- `P1`: A documented Telegram delivery workflow is failing in a released version and blocks proactive messages for affected users.
- `impact:message-loss`: Requested Telegram messages are rejected before the outbound adapter can deliver them.

## Summary

Current `main` and v2026.6.5 still use active-plugin-registry-only normalization before outbound resolution, so the documented proactive Telegram send can fail exactly as reported. The issue should remain open because two unmerged PRs claim it, and the runtime-focused candidate still needs review against unsupported-channel semantics and a focused regression test.

## What This Changes

Restore proactive Telegram delivery through `message(action=send)` and `openclaw message send` when the Telegram outbound plugin is available.

## Best Possible Solution

Use the canonical plugin-aware message-channel path at the gateway boundary, preserve the special webchat and genuinely unsupported-channel errors, and add focused gateway coverage proving a bundled Telegram ID reaches outbound resolution even when the active plugin registry is not yet populated.

## Reproduction Assessment

Yes, at source level with high confidence: an explicit `telegram` request is rejected whenever the active registry lookup returns null, before the existing plugin-aware outbound resolver runs. This read-only review did not perform a credentialed live Telegram send.

## Solution Assessment

No, the test-reset-only proposal is not the best direct solution to the reported production path. The gateway-normalization direction is narrower and better aligned with existing message routing, but it should retain clear unsupported-channel behavior and gain a focused regression test before merge.
