## Summary

The report remains necessary: current source has a complete Telegram outbound adapter, but explicit gateway sends still reject any channel absent from the process registry before adapter resolution, and the existing tests only pass after manually installing a Telegram registry fixture. The same code is present in v2026.6.1 and v2026.6.5, so there is no source or release provenance for an intervening fix.

## What This Changes

Fix proactive Telegram sends through the message tool and CLI being rejected as an unsupported channel.

## Best Possible Solution

Make explicit gateway channel validation consume the already-prepared process registry reliably, then add a regression test that starts from a realistic configured Telegram runtime and reaches the existing Telegram outbound adapter through both CLI send and message-tool action paths.

## Reproduction Assessment

Yes, at source level: when the active/pinned registry does not expose Telegram, either reported command reaches the exact `unsupported channel: telegram` branch before outbound dispatch. A real current-main Telegram environment has not yet been run, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

No, adding another Telegram message adapter is not the best fix because the canonical adapter already exists. The narrow maintainable fix is to correct the shared registry lifecycle or lookup used by explicit gateway sends and prove that path without request-time discovery or a Telegram-specific fallback.
