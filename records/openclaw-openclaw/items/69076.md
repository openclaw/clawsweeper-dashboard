## Summary

Keep open: current main still has the manual-cancel failure/notify behavior this PR targets, but the current patch has correctness blockers around killed-status derivation, the now config-scoped Telegram sent-message cache API, and the required changelog entry.

## What This Changes

The PR adds a `killed` exec/process outcome for manual cancellations, suppresses notify-on-exit for killed sessions, returns completed kill/remove control results, and adds Telegram sent-message metadata.

## Best Possible Solution

Update the PR so `killed` is derived only from explicit manual cancellation or the known fallback kill path, preserve config-scoped Telegram cache storage while adding metadata, and add focused tests plus a single-line changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows manual-cancel exits flow through `buildExecExitOutcome` as failed, notify-on-exit still wakes for manual-cancel with output, and successful process kill/remove paths return failed tool details.

## Solution Assessment

No, not as currently proposed. The direction is right, but the patch should gate killed status on explicit cancellation and merge Telegram metadata into the current config-scoped cache API instead of replacing that argument.
