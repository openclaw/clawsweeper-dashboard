## Summary

Keep open: current main still has the reported cancellation behavior, and the PR is useful, but the branch is not correct as-is because it can misclassify ordinary signal exits as killed and conflicts with current Telegram sent-message store scoping.

## What This Changes

This PR adds a killed exec/process outcome for manual cancellations, suppresses notify-on-exit for killed sessions, returns completed kill/remove control results, and records Telegram sent-message metadata.

## Best Possible Solution

Rebase and refine the patch so killed status is produced only from explicit manual cancellation or the process tool's fallback kill path, preserve config-scoped Telegram cache buckets while adding metadata and legacy timestamp compatibility, and add the required changelog entry before normal validation.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: a RunExit with reason manual-cancel flows through buildExecExitOutcome to failed, and the existing notify-on-exit test still expects manual-cancel with output to wake the agent.

## Solution Assessment

No, not as currently proposed. The direction is right, but the implementation should gate killed status on the explicit termination reason and merge Telegram metadata into the existing config-scoped API instead of replacing that argument.
