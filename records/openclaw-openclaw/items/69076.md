## Summary

This PR addresses a real current-main bug, but it still needs changes before merge because the Telegram cache metadata change conflicts with the current config-scoped cache contract and the process status mapping is too broad.

## What This Changes

The PR adds a killed exec/process outcome for manual cancellations, suppresses notify-on-exit for killed sessions, returns completed kill/remove control results, adds regression tests, and stores Telegram sent-message metadata.

## Best Possible Solution

Keep the cancellation semantics change, but rebase it onto current main with explicit manual-cancel/fallback-kill gating, a metadata API that preserves config-scoped Telegram stores, regression coverage, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows manual-cancel exits flow through buildExecExitOutcome as failed, manual-cancel with output still triggers notifyOnExit, and process kill/remove returns failed details after a termination request.

## Solution Assessment

No, not as currently proposed. The direction is right, but the patch should preserve Telegram cache cfg scoping while adding metadata and keep killed status tied to explicit cancellation or the known fallback kill path.
