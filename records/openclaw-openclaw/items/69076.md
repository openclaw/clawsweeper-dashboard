## Summary

Keep this PR open: current main still has the reported cancellation behavior, so the work is not obsolete. The branch is not merge-ready as-is because the process status helper can turn non-manual SIGTERM/SIGKILL exits into completed killed results, and the Telegram metadata change needs to be reconciled with current main's config-scoped sent-message cache.

## What This Changes

The PR classifies manual exec/process cancellations as killed, suppresses notify-on-exit for killed sessions, returns completed control-action results for successful kill/remove requests, and adds Telegram sent-message metadata diagnostics.

## Best Possible Solution

Rebase and refine the PR so explicit manual-cancel exits become killed and skip notify-on-exit, while ordinary signal, timeout, spawn-error, and shell failures remain failed. Add Telegram metadata as an additive record field without losing the current config/session-store parameter or its isolation tests.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: a RunExit with reason manual-cancel flows through buildExecExitOutcome to failed, and the existing notify-on-exit test still expects manual-cancel with output to wake the agent.

## Solution Assessment

No, not as currently proposed. The direction is right, but the maintainable fix should gate killed status on explicit manual-cancel or an explicit process-tool fallback kill, and should merge Telegram metadata with the current config-scoped cache API rather than replacing that argument.
