## Summary

Keep open. Current main still matches the reported mismatch: the headless node host declares and implements exec approval commands, but the gateway node command allowlist does not include them, so connect reconciliation filters them out before node describe and the Control UI see the advertised command list.

## What This Changes

The issue requests that Windows headless node hosts advertise `system.execApprovals.get/set` through node describe and the Control UI when exec approvals are already invokable through the dedicated approvals CLI path.

## Best Possible Solution

Update the gateway node command policy so desktop headless node hosts can advertise the exec approval commands they already declare and implement, with regression coverage for filtering, describe propagation, and the Control UI target selector.

## Reproduction Assessment

Yes, via source inspection on current main: declared `system.execApprovals.get/set` commands are filtered out by the gateway allowlist before describe/UI consume the live command list. I did not run the Windows live handshake, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is to include the exec approval command constants in the appropriate desktop node allowlist path while preserving the dedicated approvals RPC and direct `node.invoke` block.
