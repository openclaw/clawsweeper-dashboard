## Summary

Keep open: current main and the latest release still drop the exec-approval command IDs from the gateway node command policy, and the direct fix remains an open linked PR.

## What This Changes

The issue requests that Windows headless node hosts advertise `system.execApprovals.get/set` in `nodes describe` and Control UI when scoped approvals CLI calls already work.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/88296 with the same narrow policy shape, preserving scoped `exec.approvals.node.*` enforcement while retaining the advertised command IDs for approved desktop nodes.

## Reproduction Assessment

Yes. Current main source shows the node host declares `system.execApprovals.get/set`, the gateway policy filters those IDs out of the desktop node command surface, and the UI filters node targets by those advertised commands; I did not run a live Windows node in this read-only review.

## Solution Assessment

Yes. The narrow linked PR is the best current fix shape because it corrects the gateway advertisement policy without broadening raw `node.invoke` access to exec-approval commands.
