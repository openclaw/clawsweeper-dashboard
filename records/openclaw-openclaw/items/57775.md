## Summary

Keep this issue open: current main and the latest release still omit `system.execApprovals.get/set` from the gateway desktop node command policy, and the linked open PR is the canonical fix candidate rather than a reason to close before merge.

## What This Changes

The issue reports that Windows headless node hosts can service exec-approval RPCs, but `nodes describe` and the Control UI do not see `system.execApprovals.get/set` in the advertised node command list.

## Best Possible Solution

Land or intentionally close https://github.com/openclaw/openclaw/pull/88296; after it merges, close this issue as implemented while keeping broader Windows node command-surface reports separate.

## Reproduction Assessment

Yes from source: the headless node host advertises and handles `system.execApprovals.get/set`, while current main filters approved desktop node commands through `DESKTOP_HOST_COMMANDS`, which omits those IDs. I did not run a live Windows node in this read-only review.

## Solution Assessment

Yes. The linked PR is the narrow maintainable fix path because it aligns the gateway desktop command policy with existing node-host command constants while preserving the scoped `exec.approvals.node.*` RPC boundary.
