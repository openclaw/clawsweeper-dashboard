## Summary

Keep open. Current main still matches the reported mismatch: the headless node host declares and implements exec approval commands, while the gateway node command allowlist filters them out before node describe and the Control UI selector see the node command list.

## What This Changes

Request that Windows headless node hosts advertise `system.execApprovals.get/set` through node describe and the Control UI when those commands are already implemented and CLI-routable.

## Best Possible Solution

Update the gateway node command policy so implemented headless node hosts advertise exec approval commands, then add regression coverage for allowlist filtering, node describe propagation, and the Control UI selector.

## Reproduction Assessment

Yes. Source inspection on current main shows the deterministic path: Windows defaults flow through `SYSTEM_COMMANDS`, which omits `system.execApprovals.get/set`, and node connect filters declared commands through that allowlist before describe/UI consume them.

## Solution Assessment

Yes. The narrow maintainable fix is to add the exec approval command constants to the appropriate node-host allowlist; teaching the Control UI to ignore advertised commands would bypass the capability contract.

Fix https://github.com/openclaw/openclaw/issues/57775 on current main. The bug is that headless node hosts declare and implement `system.execApprovals.get/set`, but the gateway node command policy filters those commands out, so `openclaw nodes describe` and the Control UI do not treat Windows headless nodes as exec-approval-capable targets. Update the node command allowlist for the platforms whose node hosts implement the approval handlers, add focused regression coverage for allowlist normalization/connect propagation/describe-visible commands and the Control UI exec approvals node selector, and add a single-line `CHANGELOG.md` fix entry because this is user-visible. Do not add a config option or broaden unknown/mobile node defaults beyond implemented handlers.
