## Summary

Keep open: current main and the latest release still have the documented and implemented node-host exec approval commands, but the gateway node command policy does not allow them through pairing/reconnect advertisement, so `nodes describe` and the Control UI target picker can still miss valid exec-approvals-capable Windows headless nodes.

## What This Changes

The issue asks for Windows headless node hosts to advertise `system.execApprovals.get/set` consistently in `nodes describe` and Control UI discovery.

## Best Possible Solution

Add `NODE_EXEC_APPROVALS_COMMANDS` to the desktop node command policy used during pairing/reconnect advertisement, keep direct generic `node.invoke` blocked for those commands, and add regression coverage for describe/UI discovery.

## Reproduction Assessment

Yes. Source inspection on current main shows the node host declares and implements `system.execApprovals.get/set`, while gateway pairing policy omits those command IDs and filters them before describe/UI consume the command list.

## Solution Assessment

Yes. The maintainable fix is a small gateway node-command policy alignment with regression tests, not a UI workaround, new config option, or broad node-host rewrite.
