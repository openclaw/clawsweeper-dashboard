## Summary

This issue should stay open because current main still has the advertised-command mismatch: the node host declares exec-approval commands, but the gateway desktop-node command policy does not preserve `system.execApprovals.get/set` for paired/live desktop nodes. The linked fix PR is the right narrow implementation path, but it is still open, so this issue should remain open until that PR lands or maintainers choose an equivalent security-boundary-safe fix.

## What This Changes

Windows headless node hosts can serve exec approval get/set through the CLI, but `nodes describe` and the Control UI do not see `system.execApprovals.get/set` as advertised node commands.

## Best Possible Solution

Land or replace the narrow gateway policy fix in https://github.com/openclaw/openclaw/pull/88296, preserving scoped `exec.approvals.node.*` enforcement while allowing approved desktop node descriptors to retain exec-approval command names.

## Reproduction Assessment

Yes for a source-level reproduction: current main declares `system.execApprovals.get/set` from the node host, then filters them out of the gateway command policy used for pairing/reconnect and describe output. I did not run a live Windows node in this read-only review.

## Solution Assessment
