## Summary

Keep open. Current main still matches the reported mismatch: the headless node host declares and implements exec approval commands, but the gateway node command allowlist omits them before node describe and the Control UI selector read the node command list.

## What This Changes

Report says Windows headless node hosts can service exec approval CLI requests but do not advertise `system.execApprovals.get/set` through `nodes describe` or the Control UI.

## Best Possible Solution

Add the exec approval command constants to the supported node-host allowlist for the platforms whose node hosts implement them, then cover the allowlist, connect/describe propagation, and Control UI selector behavior with focused regression tests.

## Reproduction Assessment

Yes. The current-main source path deterministically reproduces the report: Windows defaults flow through `SYSTEM_COMMANDS`, which omits `system.execApprovals.get/set`, and the connect path filters declared commands through that allowlist before describe/UI see them.

## Solution Assessment

Yes. Fixing the gateway command policy is the narrowest maintainable solution; changing the Control UI to ignore advertised commands would bypass the capability contract instead of repairing it.
