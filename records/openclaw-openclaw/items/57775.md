## Summary

Keep open: current main and v2026.5.19 still show the same source-level mismatch. The headless node host declares and implements exec approval commands, but the gateway command policy does not admit those IDs before `node.describe` and the Control UI consume the advertised command list.

## What This Changes

The issue asks for Windows headless node hosts to advertise `system.execApprovals.get/set` so `nodes describe` and the Control UI recognize exec-approvals-capable nodes.

## Best Possible Solution

Add the exec approval command IDs to the gateway node command policy used for pairing/reconnect advertisement, preserve the existing approvals API path, and cover CLI describe plus Control UI discovery with focused regression tests.

## Reproduction Assessment

Yes, by source inspection rather than live Windows transport. Current main declares and implements `system.execApprovals.get/set` on the node host, while the gateway command policy omits those IDs before describe/UI consume the advertised command list.

## Solution Assessment

Yes. The narrow maintainable fix is to align the gateway node-command allowlist with the existing node-host command contract and add regression coverage, not to add a UI workaround or a new config option.
