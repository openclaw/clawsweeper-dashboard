## Summary

Keep open. Current main still has the reported mismatch: the headless node host declares and implements exec-approval commands, but the gateway's default node command allowlist for Windows/system-capable nodes does not include them, so the commands are filtered before `node.describe` and the Control UI selector see them.

## What This Changes

Windows headless node hosts should advertise `system.execApprovals.get/set` through `nodes describe` and the Control UI when node-targeted approvals work through the CLI.

## Best Possible Solution

Add the exec-approval command constants to the default allowlist for system-capable node platforms that already expose node-host approvals, then cover the allowlist, connect/describe path, and Control UI target selector so future node command policy changes cannot hide working approval targets.

## Reproduction Assessment

Yes. A Windows node that declares `NODE_EXEC_APPROVALS_COMMANDS` is filtered through `resolveNodeCommandAllowlist`; current Windows defaults omit those commands, so `node.describe` and the Control UI lose them while dedicated CLI approvals can still invoke the node handler directly.

## Solution Assessment

Yes. The narrowest maintainable fix is in the gateway node command policy plus regression tests, preserving the dedicated approvals RPC and the `node.invoke` guard instead of routing approval edits through generic node invocation.
