## Summary

Keep open: current main still omits the exec approval command IDs from the gateway node command allowlist, so connect reconciliation filters them before `nodes describe` and the Control UI target picker can see them.

## What This Changes

The issue asks for Windows headless node hosts to advertise `system.execApprovals.get/set` in node describe and Control UI target discovery, matching the CLI-backed approvals handlers.

## Best Possible Solution

Add the exec approval command constants to the desktop node command policy used during pairing/reconnect so advertised commands include them, while keeping direct `node.invoke` blocked and routing edits through `exec.approvals.node.*`.

## Reproduction Assessment

Yes, via source inspection on current main: the node host declares `system.execApprovals.get/set`, but the gateway allowlist omits those IDs and connect reconciliation filters them before describe/UI consume the command list. I did not run the live Windows handshake.

## Solution Assessment

Yes. The narrow maintainable fix is policy alignment in the gateway node command allowlist plus regression coverage, not a new UI workaround or a new config option.
