## Summary

Keep open. This is a fresh, specific Telegram forum-topic message-loss report against the latest shipped release, and current main still has the code path that treats a group/channel subagent completion as failed when the requester announce agent returns normal final text without message-tool delivery evidence. The broader open reports at https://github.com/openclaw/openclaw/issues/44925 and https://github.com/openclaw/openclaw/issues/83863 are related, but this issue has narrower Telegram/forum-topic evidence and should remain actionable rather than being swept into those umbrellas.

## What This Changes

The issue asks OpenClaw to reliably deliver Telegram forum-topic subagent completion announcements when the requester announce agent must use `message(action="send")`.

## Best Possible Solution

Preserve the mediated message-tool policy, but make Telegram group/forum-topic subagent completion handoffs deterministically prompt or retry the requester agent to call `message(action="send")`, with a focused regression for normal-final-text failure and successful message-tool delivery.

## Reproduction Assessment

Yes at source/log level: the issue provides a concrete 2026.5.19 live run and current main has the same failure branch when a message-tool-only completion lacks messaging-tool evidence. I did not run a live Telegram reproduction during this read-only review.

## Solution Assessment

Unclear for the reporter's raw-conversion fallback, because that would change the documented group/channel delivery policy. The maintainable fix is to keep mediated delivery and make the announce path reliably produce or retry a `message(action="send")` call.
