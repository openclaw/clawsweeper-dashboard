## Summary

Keep open: the packaging hardening is a plausible fix direction, but this external PR lacks after-fix real behavior proof and the new scanner misses escaped Windows paths in packaged JS/JSON text.

## What This Changes

The PR adds a shared scanner for package-local absolute paths in dist artifacts, wires it into release-check and tarball validation, and adds release-check regression coverage.

## Best Possible Solution

Keep the shared scanner approach, but cover escaped Windows path representations and require redacted real package-install output before merge.

## Reproduction Assessment

Yes, source-level: current main only scans private QA-prefixed content in collectForbiddenPackContentPaths, so a packaged dist JS file with a /Users or /home build path is not rejected by that check. I did not run package E2E because this review was required to keep the checkout read-only.

## Solution Assessment

No, not yet: a shared scanner is the right seam for release and tarball validation, but this implementation should also detect escaped Windows paths and include inspectable after-fix package proof before merge.
