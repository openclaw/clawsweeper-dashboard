## Summary

Keep open. The PR addresses a real package-portability guardrail gap, but the proposed scanner still misses common Windows path representations and the external contributor has not provided inspectable after-fix real behavior proof.

## What This Changes

The PR adds shared scanning for package-local absolute paths in packaged dist text artifacts and wires it into release-check, tarball validation, and regression tests.

## Best Possible Solution

Keep the shared scanner direction, add coverage for escaped and forward-slash Windows path forms, then require redacted packed-tarball install output before merge.

## Reproduction Assessment

Yes, source-level: current main only scans private-QA-prefixed text paths in collectForbiddenPackContentPaths, so a dist JS file with /Users or /home content outside those prefixes would not be rejected. I did not run package E2E because this review had to keep the checkout read-only.

## Solution Assessment

No, not yet: the shared scanner is the right seam for release and tarball validation, but this implementation should catch escaped and forward-slash Windows paths and include inspectable after-fix package proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Detect escaped Windows paths:** `scripts/lib/package-local-path-scan.mjs:5`
  - body: The scanner's C:\Users\ pattern only matches single-backslash text. Packaged JS/JSON commonly stores Windows paths as escaped text like C:\\Users\\builder\\openclaw\\dist\\entry.js, and the probe shows those are not matched, so a Windows build-root leak can pass the new release-check and tarball validation hooks. Add escaped-backslash coverage, and preferably C:/Users/ coverage, with a regression case.
  - confidence: 0.84
