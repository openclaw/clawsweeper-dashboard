## Label Justifications

- `P2`: This is meaningful memory durability work with multi-agent impact, but it is a product/API design request rather than an urgent reproduced regression.
- `impact:data-loss`: The reported failure mode is durable session knowledge not being flushed or preserved before compaction.
- `impact:session-state`: The requested behavior changes memory flush timing, compaction retention, and memory-search state around long-running sessions.

## Summary

Keep open: current main and v2026.6.6 partially cover forced flushing with `forceFlushTranscriptBytes`, and the embedding cache is content-hash/provider keyed, but the requested hard token-floor threshold and priority-preserving memory/compaction contract remain unimplemented product/API work.

## What This Changes

The issue requests three new memory durability configuration capabilities: a hard memory-flush threshold, priority-preserving compaction, and an embedding cache TTL.

## Best Possible Solution

Split the design by owner boundary: keep the shipped transcript-size forced flush, design priority-preserving durable memory through the memory/context-engine plugin contracts, and add cache TTL only if a reproducible stale-cache gap remains after content-hash invalidation.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main only partially covers the flush safety goal and does not implement priority-aware compaction or cache TTL.

## Solution Assessment

No, not as the bundled three-key proposal. The safer path is to narrow the flush, priority-retention, and cache-freshness contracts separately because current code already has different forced-flush and cache-invalidation behavior.
