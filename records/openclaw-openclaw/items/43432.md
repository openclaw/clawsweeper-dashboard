## Summary

Keep open. Current main has meaningful memory-durability work, including soft pre-compaction flushing, transcript-size forced flushes, safeguard compaction, and embedding cache caps, but it does not implement the three requested config surfaces or the priority-tag preservation semantics. The request is concrete and current, but it bundles token-threshold behavior, compaction strategy design, and cache freshness policy, so it needs maintainer design review rather than conservative cleanup closure.

## What This Changes

Add memory durability configuration for a hard token-based memory flush threshold, priority-aware compaction, and an embedding-cache TTL under agent defaults.

## Best Possible Solution

Split the request into a maintainer-approved memory durability design: decide whether a token hard flush can be guaranteed before emergency compaction, define a durable priority metadata contract or plugin/provider strategy for compaction preservation, and add cache expiration only if existing content-hash and sync invalidation cannot cover the stale-embedding case. Update config schema, docs, and focused tests with each accepted surface.

## Reproduction Assessment

Partly: the schema rejection is high-confidence because current strict schemas do not accept the proposed keys. The production-scale memory-loss scenario itself is not reproduced here; it needs workload logs or a targeted stress harness to prove the bad path on current main.

## Solution Assessment

Unclear: the requested direction is plausible, but bundling three separate config/API changes is not the narrowest maintainable fix. A safer path is to design each durability control against current memory-core, compaction-provider, and cache invalidation contracts before coding.
