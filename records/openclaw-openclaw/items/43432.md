## Summary

Keep open for maintainer design, but narrow the scope: current main partially addresses forced flushing with `forceFlushTranscriptBytes`, content-hash invalidation undercuts the proposed embedding TTL rationale, and priority-preserving durable memory remains a substantial unresolved product/API capability.

## What This Changes

Add a hard memory-flush threshold, priority-aware compaction, and an embedding-cache TTL under agent defaults.

## Best Possible Solution

Split the design by contract: prove and strengthen the existing forced transcript flush if it still permits unflushed compaction, define durable priority metadata and retention in the active memory-plugin API, and keep content-hash cache invalidation unless a separate reproducible indexing defect is found.

## Reproduction Assessment

No, there is not a high-confidence reproduction for the full issue. The unsupported-key validation is expected, current source contradicts the stated embedding-TTL failure mode, and the flush and priority portions request new behavior rather than repair a documented contract.

## Solution Assessment

No, the proposed three-key bundle is not the best solution. Current main already has different forced-flush and cache contracts, while priority preservation needs a separately designed memory-owner data model rather than a generic compaction switch.
