## Summary

Keep open: current main and v2026.6.6 only partially cover the durability concern with transcript-size forced flushing and content-hash/provider-keyed embedding cache reuse; the requested hard token-floor threshold, priority-preserving compaction contract, and cache TTL remain product/API work.

## What This Changes

The issue requests three new memory durability config surfaces: a hard memory-flush token threshold, priority-aware compaction, and an embedding-cache TTL.

## Best Possible Solution

Split the design into focused decisions: keep the shipped transcript-size forced flush, design priority preservation through memory/context-engine plugin contracts, and add cache TTL only if a concrete stale-cache gap remains after hash-based invalidation.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main only partially covers the flush safety goal and does not implement priority-aware compaction or cache TTL.

## Solution Assessment

No, not as one bundled three-key patch. The safer solution is to narrow the flush, priority-retention, and cache-freshness contracts separately because current code already has different forced-flush and cache-invalidation behavior.
