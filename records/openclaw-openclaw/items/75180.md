## Summary

Keep open. The PR targets a valid media-dedupe bug and the delivery-aware callback contract is the right direction, but the latest patch still confirms queued block media from payload content before queued delivery can actually succeed, so final dedupe can still strip an attachment that never reached the channel.

## What This Changes

This PR adds a block-reply media-confirmation return path, updates reply option and delivery types, wires production dispatch, adds block-pipeline tests, and adds a changelog fix entry for the Telegram mixed text-plus-MEDIA dedupe bug.

## Best Possible Solution

Keep the delivery-aware callback result shape, but make production confirmation exact: the direct branch may confirm media after the awaited send succeeds, while the queued dispatcher branch should return no media confirmation unless the dispatcher exposes a real accepted or delivered media result. Add regression coverage for sendBlockReply returning false and for queued cancellation or delivery failure before relying on sentMediaUrls.

## Reproduction Assessment

Yes. The linked bug has concrete Telegram mixed text-plus-MEDIA steps, and the remaining PR bug has a source-level reproduction path: make dispatcher.sendBlockReply return false, or cancel/fail queued block delivery, then observe the returned media confirmation let final dedupe strip unsent media.

## Solution Assessment

No. The API direction is maintainable, but deriving sent-media confirmation from normalizedPayload content is not delivery proof for the queued dispatcher path; direct-only confirmation or a dispatcher-level delivery signal is safer.
