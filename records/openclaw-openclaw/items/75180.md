## Summary

Keep open. The PR is aimed at a valid mixed text-plus-media reply bug, but the latest head still confirms media for the queued dispatch path from payload content rather than from a real delivery signal, so final dedupe can still strip media that never reached the channel.

## What This Changes

This PR adds delivery-aware block-reply media tracking, widens the `onBlockReply` return type, threads media confirmation through reply dispatch/delivery, adds focused block-pipeline tests, and updates the changelog.

## Best Possible Solution

Keep the delivery-aware callback result contract, but make production confirmation semantically exact: direct delivery may confirm after the awaited send succeeds, while queued dispatcher delivery should return no sent-media confirmation unless the dispatcher exposes a tested acceptance or delivery signal.

## Reproduction Assessment

Yes. The linked bug has concrete Telegram mixed text-plus-MEDIA steps, and the PR regression has a source-level reproduction path: have queued dispatch reject, cancel, or fail after enqueue, then observe the callback still reporting media as sent and final dedupe stripping an unsent attachment.

## Solution Assessment

No. The API shape is maintainable, but deriving delivery confirmation from `normalizedPayload` content is not the narrowest correct implementation for queued dispatch because content presence is not delivery proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Confirm media only after queued delivery succeeds:** `src/auto-reply/reply/dispatch-from-config.ts:1341-1350`
  - body: This returns `sentMediaUrls` for the queued path based only on `normalizedPayload`. `dispatcher.sendBlockReply()` can return `false`, and accepted queue entries can later be cancelled by `beforeDeliver` or fail in `deliver`, so the pipeline may record media as sent and the final dedupe can strip an attachment that never reached the channel. Gate this on a real delivery/acceptance signal, or return no confirmation for queued delivery.
