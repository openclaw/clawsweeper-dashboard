## Summary

Keep open: this is a focused fix for a valid reply-media bug, but the current patch can still record media as delivered before the channel or dispatcher has actually succeeded, and the contributor-supplied proof is unit-test output rather than real Telegram or equivalent channel behavior.

## What This Changes

The PR adds callback-returned sent-media tracking to the block reply pipeline, threads the return type through reply delivery and dispatch, updates focused tests, and adds a changelog entry.

## Best Possible Solution

Keep the delivery-aware callback direction, but return sentMediaUrls only from a real successful channel or dispatcher delivery signal, add failure-path coverage, and attach redacted Telegram or equivalent live proof.

## Reproduction Assessment

Yes, as a high-confidence source-level path. The linked Telegram report gives concrete DM steps, and current main records mixed block media before final payload dedupe can strip matching attachments.

## Solution Assessment

No. The callback-result contract is maintainable, but this patch derives confirmation from normalized payload contents instead of actual successful delivery, so the same premature dedupe class can still occur on failure paths.
