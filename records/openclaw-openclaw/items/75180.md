## Summary

Keep open. The PR is useful and targets a real current-main bug, but the latest head still has a blocking delivery-confirmation bug: `dispatch-from-config` derives `sentMediaUrls` from payload content before queued block delivery is accepted or completed, so final media dedupe can still strip an attachment that never reached the channel.

## What This Changes

The PR widens block-reply callbacks to return media delivery confirmations, changes block reply media tracking, threads the callback return type through dispatch/reply delivery, adds focused pipeline tests, and adds a changelog fix for mixed text-plus-media replies.

## Best Possible Solution

Keep the delivery-aware callback contract, but make sent-media confirmation semantically exact: direct delivery should confirm URLs only after a successful send result, and queued dispatcher delivery should either expose real accepted/delivered media metadata or return no confirmation so final dedupe preserves unsent attachments. Add dispatch coverage for rejected queueing and later cancellation/failure.

## Reproduction Assessment

Yes. The original issue has a high-confidence source-level path through mixed text-plus-MEDIA block streaming, block sent-media recording, and final media dedupe; the remaining PR bug can be reproduced with a dispatch-from-config test where `sendBlockReply` returns false or `beforeDeliver` cancels after the callback has returned sentMediaUrls.

## Solution Assessment

No. The proposed API shape is reasonable, but deriving confirmation from `normalizedPayload` content is not delivery-aware; the safer fix is to confirm only after successful direct send or real dispatcher acceptance/delivery, or leave confirmation undefined on the queued path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Return sent media only after accepted delivery:** `src/auto-reply/reply/dispatch-from-config.ts:1340-1349`
  - body: This returns `confirmedMedia` even after `dispatcher.sendBlockReply(normalizedPayload)` merely queues work. That call can return `false`, and `beforeDeliver` or `deliver` can later cancel/fail, so `createBlockReplyPipeline` still records the media as delivered and final dedupe can strip an attachment that never reached the channel. Gate `sentMediaUrls` on actual acceptance/delivery, or return no confirmation for the queued path.
