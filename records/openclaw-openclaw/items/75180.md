## Summary

Keep open: this PR addresses the valid mixed text+media reply bug, but the current head still marks media as confirmed from payload content before delivery success is proven, so final dedupe can still strip attachments after a failed or cancelled block delivery.

## What This Changes

The PR adds a block-reply callback result for confirmed sent media URLs, threads it through reply delivery types, updates block-pipeline tests, and adds a changelog fix for mixed text+MEDIA replies.

## Best Possible Solution

Keep the callback-result direction, but only mark media as sent when the selected delivery path can prove success; otherwise leave the media unconfirmed so final dedupe preserves it.

## Reproduction Assessment

Yes. #75156 gives concrete Telegram DM steps, and current-main source shows block sent-media URLs flowing into final media dedupe; I did not run live Telegram, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. The API direction is maintainable, but this PR derives confirmation from payload content before direct route success or queued dispatcher delivery, so it is not yet the safest fix.
