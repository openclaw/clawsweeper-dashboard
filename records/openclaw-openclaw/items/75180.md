## Summary

Keep open. The PR is directionally useful and current main still has the mixed text-plus-media dedupe problem, but the latest head is not merge-ready because the queued dispatch path still reports media as confirmed before actual delivery can succeed or fail.

## What This Changes

The PR adds a block-reply media confirmation result, updates reply option and delivery types, wires dispatch-from-config to return confirmed media URLs, adds block-pipeline regression tests, and adds a changelog fix entry for #75156.

## Best Possible Solution

Keep the delivery-aware callback shape, but make production confirmation exact: return media confirmation after the awaited direct-send path succeeds, and return no confirmation on the queued dispatcher path unless the dispatcher exposes a real delivered-media result. Add focused regression coverage for sendBlockReply returning false and queued cancellation or delivery failure.

## Reproduction Assessment

Yes. The linked bug has concrete Telegram mixed text-plus-MEDIA steps, and the remaining PR bug has a source-level path: make dispatcher.sendBlockReply return false or have queued delivery cancel/fail, then observe the returned confirmation feed final media dedupe.

## Solution Assessment

No. The API direction is maintainable, but deriving confirmation from normalizedPayload content is not proof of delivery for the queued dispatcher path; direct-only confirmation or a dispatcher-level delivery result is the safer fix.
