## Summary

Keep this PR open: the linked bug is valid, but the latest patch still confirms media from payload content before delivery is proven, and the supplied proof is only a test run rather than after-fix Telegram/live behavior.

## What This Changes

This PR adds a sent-media callback result through the block reply pipeline, reply delivery, and dispatch config, updates block-pipeline tests, and adds a changelog fix for mixed text+media reply dedupe.

## Best Possible Solution

Keep the delivery-aware accounting direction, but derive sentMediaUrls only from an actual delivery-success signal, add failure-path regression coverage, and require redacted live Telegram or equivalent channel proof before merge.

## Reproduction Assessment

Yes. The linked Telegram direct-chat report gives concrete steps, and current main source shows mixed text+media block media is recorded as sent before final dedupe filters against that set; I did not run a live Telegram gateway test in this read-only review.

## Solution Assessment

No. The callback-result design is maintainable, but this patch derives confirmation from payload content before the selected delivery path proves success; the narrow fix should only confirm URLs after route send success or a real dispatcher delivery result.
