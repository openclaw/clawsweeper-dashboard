## Summary

Keep open. The PR is the active fix for the Telegram mixed text-plus-MEDIA dedupe bug, but the latest head still confirms media from the queued dispatcher path before the dispatcher proves delivery, so it can preserve the original attachment-drop failure in skipped or failed queued sends.

## What This Changes

This PR adds a delivery-aware block-reply media confirmation result, updates reply callback types and delivery plumbing, adds block-pipeline tests, and adds a changelog fix for #75156.

## Best Possible Solution

Keep the delivery-aware callback shape, but make production confirmation exact: direct sends may confirm media only after awaited delivery succeeds, while queued sends should return an empty confirmation unless the dispatcher exposes a real delivery result.

## Reproduction Assessment

Yes. Use the linked Telegram text-plus-MEDIA scenario for the user bug, and for the remaining PR regression make dispatcher.sendBlockReply return false or have queued delivery cancel/fail before observing that sentMediaUrls still drives final media dedupe.

## Solution Assessment

No. The API direction is maintainable, but deriving sent-media confirmation from normalizedPayload content is not delivery proof for the queued dispatcher path; direct-only confirmation or a dispatcher-level delivered-media signal is safer.
