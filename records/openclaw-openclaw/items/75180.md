## Summary

Keep open: the PR targets the right bug, but the production callback can still mark media as sent when delivery failed, aborted, or only queued work, so final media dedupe can preserve #75156.

## What This Changes

The PR adds a delivery-confirmation return value for block replies, threads it through reply delivery types, updates block-pipeline tests, and adds a changelog fix for #75156.

## Best Possible Solution

Keep the delivery-aware callback shape, but only return sent-media confirmations after the selected delivery path reports success; queued dispatcher paths need either no confirmation or a real delivery-result signal.

## Reproduction Assessment

Yes. #75156 provides concrete Telegram DM steps, and source inspection shows a current-main path where mixed text+media block media is marked sent and later stripped by final dedupe.

## Solution Assessment

No. The API direction is maintainable, but deriving confirmation from payload content before verifying delivery success is not the safest fix for either direct route failures or queued dispatcher failures.
