## Summary

Keep open. The PR is directionally useful, but the latest head still confirms media from the queued block-reply path before delivery has actually succeeded, so it can preserve the dropped-attachment failure from the linked bug.

## What This Changes

This PR adds delivery-aware block-reply media confirmation plumbing, callback type updates, block-pipeline tests, and a changelog fix for mixed text-plus-media replies.

## Best Possible Solution

Keep the delivery-aware callback shape, but make production confirmation exact: direct awaited sends may return delivered media, while queued sends should report no sent media unless the dispatcher exposes a real delivery acknowledgement.

## Reproduction Assessment

Yes. The linked Telegram direct-chat steps reproduce the user bug, and source inspection gives a focused PR regression path: make queued block delivery return false or later cancel/fail, then observe final media dedupe using the premature sentMediaUrls confirmation.

## Solution Assessment

No. The API direction is maintainable, but deriving sent-media confirmation from normalizedPayload content is not enough for the queued dispatcher path; direct-only confirmation or a dispatcher-level delivered-media signal is safer.
