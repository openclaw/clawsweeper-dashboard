## Summary

Keep this PR open; the latest head still has a blocking correctness issue. The new callback result contract is directionally useful, but the generic dispatch path still confirms media from payload contents before queued delivery is known to have been accepted or completed, so final dedupe can still strip an attachment that never reached the channel.

## What This Changes

This PR adds delivery-aware block-reply media tracking, updates the `onBlockReply` return type, threads it through dispatch and reply delivery, adds focused block-pipeline tests, and adds a changelog fix for #75156.

## Best Possible Solution

Keep the delivery-aware callback result contract, but make production confirmation semantically exact: direct delivery may confirm after a successful awaited send, while queued dispatcher delivery should only confirm media if the dispatcher exposes a real acceptance or delivery signal; otherwise it should return no sent-media confirmation.

## Reproduction Assessment

Yes. The linked bug has live Telegram mixed text-plus-MEDIA steps, and the remaining PR bug has a source-level reproduction path: make `dispatcher.sendBlockReply` return `false` or have queued delivery cancel/fail, then observe the callback still return media confirmation and final dedupe strip unsent media.

## Solution Assessment

No. The API shape is maintainable, but deriving delivery confirmation from `normalizedPayload` content is not the best implementation for the queued dispatcher path because content presence is not delivery proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Confirm media only after queued delivery succeeds:** `src/auto-reply/reply/dispatch-from-config.ts:1341-1350`
  - body: This returns `sentMediaUrls` for the queued path based only on `normalizedPayload`. `dispatcher.sendBlockReply()` can return `false`, and accepted queue entries can later be cancelled by `beforeDeliver` or fail in `deliver`, so the pipeline may record media as sent and the final dedupe can strip an attachment that never reached the channel. Gate this on real direct delivery/accepted metadata, or return no confirmation for queued delivery.
