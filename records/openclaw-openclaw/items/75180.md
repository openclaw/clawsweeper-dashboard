## Summary

Keep this PR open. It targets the right mixed text-plus-media block-reply bug from #75156, but the latest diff still confirms media for the queued dispatcher path from payload shape alone, so final dedupe can strip media that was never actually delivered.

## What This Changes

This PR adds a delivery-aware `onBlockReply` media-confirmation result, updates block-reply media tracking and callback types, threads the type through reply delivery, adds focused block-pipeline tests, and adds a changelog fix for mixed text-plus-media replies.

## Best Possible Solution

Keep the delivery-aware callback result contract, but make production confirmation exact: direct delivery may confirm after the awaited send succeeds, while queued dispatcher delivery should either return no `sentMediaUrls` or use a real accepted/delivered media signal backed by regression tests for enqueue rejection, cancellation, and delivery failure.

## Reproduction Assessment

Yes. #75156 provides live Telegram mixed text-plus-`MEDIA:` steps; the remaining PR bug is reproducible at source level by making `dispatcher.sendBlockReply` return `false` or by cancelling/failing queued block delivery and observing that the PR callback still confirms media for final dedupe.

## Solution Assessment

No. The API shape is maintainable, but deriving delivery confirmation from `normalizedPayload` content is not the narrowest correct production fix for the queued dispatcher path because queued content is not delivery proof.
