## Summary

Keep open: this PR targets a valid media-dedupe bug, but the current patch still confirms media before delivery is actually proven and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds a sent-media confirmation return for block replies, threads it through reply delivery/config types, updates block-pipeline tests, and adds a changelog fix for mixed text+media media dedupe.

## Best Possible Solution

Keep the delivery-aware callback direction, but only record media after a real delivery-success signal, add dispatcher/direct-failure regression coverage, and require redacted after-fix Telegram/live proof before merge.

## Reproduction Assessment

Yes. The linked Telegram direct-chat report gives concrete steps, and current main source shows mixed text+media block media is recorded as sent before final dedupe filters against that set.

## Solution Assessment

No. The callback-result design is maintainable, but this patch derives confirmation from payload content before the selected delivery path proves success; confirmation should come only after route send success or a real dispatcher delivery result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Confirm media only after delivery succeeds:** `src/auto-reply/reply/dispatch-from-config.ts:1500`
