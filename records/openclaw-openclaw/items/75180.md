## Summary

Keep this PR open, but it still needs changes before merge. The delivery-aware callback contract is a reasonable direction, yet the latest head still reports media as sent in the queued dispatcher path before the dispatcher has actually accepted or delivered the block, so final dedupe can still strip media that never reached the channel.

## What This Changes

This PR adds a block-reply result type for confirmed sent media URLs, updates block-reply media tracking and callback types, threads the return type through dispatch/reply delivery, adds focused block pipeline tests, and adds a changelog fix for mixed text-plus-media replies.

## Best Possible Solution

Keep the delivery-aware media confirmation contract, but make the production confirmation semantically exact: direct delivery may confirm after a successful awaited send, while queued dispatcher delivery should either expose real accepted/delivered media metadata or return no sentMediaUrls so final dedupe preserves attachments unless delivery is known to have happened.

## Reproduction Assessment

Yes. The linked bug has a live Telegram mixed text-plus-MEDIA reproduction, and the remaining PR defect has a focused source-level path: make dispatcher.sendBlockReply return false or have beforeDeliver cancel after enqueue, then the callback still returns sentMediaUrls and final dedupe can remove unsent media.

## Solution Assessment

No. The API shape is maintainable, but deriving delivery confirmation from normalizedPayload content is not actually delivery-aware for queued dispatch; the safer solution is to confirm only after direct send success or real dispatcher acceptance/delivery metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Confirm media only after queued delivery can be accepted:** `src/auto-reply/reply/dispatch-from-config.ts:1342-1350`
  - body: This returns `sentMediaUrls` based on `normalizedPayload` even when `dispatcher.sendBlockReply()` only queues work. That enqueue can return `false`, and `beforeDeliver` or `deliver` can cancel/fail later, so the block pipeline records media as sent and final dedupe may strip an attachment that never reached the channel. Gate the confirmation on actual direct send/accepted delivery, or return no confirmation for the queued path.
  - confidence: 0.9
