## Summary

Keep open: current main still has the reported diagnostic gap, and the linked PR #76151 is open rather than merged. The ownership block itself is intentional, but blocked-present plugins can still fall through to stale/missing config warnings and duplicate diagnostic surfaces.

## What This Changes

The issue asks OpenClaw to avoid misleading stale-plugin warnings and duplicate suspicious-ownership logs when a present plugin directory is blocked by ownership, and to document the remediation path.

## Best Possible Solution

Preserve the ownership safety gate, propagate a blocked-present plugin state into config diagnostics, deduplicate ownership warnings per boot, and add focused user-facing plugin ownership guidance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-bundled plugin owned by neither the gateway uid nor root is blocked and omitted from known ids, so explicit config references still follow the missing-plugin warning branch.

## Solution Assessment

Yes for the diagnostic/docs portion: distinguishing blocked-present plugins from absent plugins is the narrow maintainable fix. The container ownership-reset portion is not yet the best merged path without separate confirmation and maintainer policy review.
