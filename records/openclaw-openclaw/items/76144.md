## Summary

Keep open: current main still drops ownership-blocked plugin candidates before registration, so configured references can still fall through to the misleading stale missing-plugin warning path, and discovery diagnostics are still appended without dedupe.

## What This Changes

The issue asks for clearer and deduplicated plugin ownership-block diagnostics, docs for plugin directory ownership, and a separately considered Docker ownership-preservation change.

## Best Possible Solution

Keep the ownership safety gate, carry blocked-present plugin id/path state into config validation, dedupe ownership diagnostics per boot, document plugin directory ownership, and leave Docker chown policy for a separately confirmed follow-up.

## Reproduction Assessment

Yes by source inspection. A non-bundled uid-mismatch candidate is blocked before registration, its diagnostic lacks a plugin id, and configured entries or allowlist references still reach the stale missing-plugin warning branch.

## Solution Assessment

Yes for the diagnostics and docs portion: propagating blocked-present state from discovery into validation and deduping diagnostics is the narrow maintainable fix. Changing Docker chown behavior is a separate policy decision, not part of the safe first patch.
