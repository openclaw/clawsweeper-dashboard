## Summary

Keep open: current main still has the reported diagnostic bug, and the linked repair PR #76151 is open rather than merged. The ownership block is intentional, but blocked-present plugins still fall through to misleading stale/missing config warnings and can produce duplicate discovery diagnostics.

## What This Changes

The issue reports misleading and duplicated plugin-loader diagnostics for ownership-blocked plugin directories plus non-durable container ownership after restart.

## Best Possible Solution

Land or replace PR #76151 for the diagnostics/docs path, then separately decide whether Docker setup should exempt plugin install roots from node ownership repair without weakening plugin trust checks.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an ownership-blocked non-bundled plugin is omitted from discovery/knownIds, so explicit plugins.entries or plugins.allow references still take the stale missing-plugin warning branch.

## Solution Assessment

Yes for the diagnostic part: carry blocked plugin id/path through discovery, dedupe diagnostics, and replace stale-config guidance for blocked-present plugins. The Docker ownership-reset part should be handled as a separate maintainer policy decision because it touches the plugin trust boundary.
