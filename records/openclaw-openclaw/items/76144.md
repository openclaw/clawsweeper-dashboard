## Summary

Keep open: current main still has the reported diagnostic gap. The ownership block is intentional, but ownership-blocked plugins are still omitted from known plugin ids, so configured references can still fall through to the misleading stale-config warning path.

## What This Changes

The issue requests clearer plugin-loader diagnostics and documentation when a present plugin is blocked by suspicious directory ownership, plus deduped ownership warnings and a Docker ownership-policy check.

## Best Possible Solution

Keep the ownership safety gate, carry blocked-present plugin id/path state into config validation, dedupe ownership diagnostics per boot, document the constraint, and split any Docker ownership-policy change into a separately confirmed follow-up.

## Reproduction Assessment

Yes by source inspection. A non-bundled uid-mismatch candidate is blocked before registration, lacks a diagnostic plugin id, and configured entries or allowlist references still flow to the stale missing-plugin warning branch.

## Solution Assessment

Yes for the diagnostics and docs portion: propagating blocked-present state from discovery into validation and deduping discovery diagnostics is the narrow maintainable fix. Changing Docker chown behavior should be treated as a separately confirmed policy decision.
