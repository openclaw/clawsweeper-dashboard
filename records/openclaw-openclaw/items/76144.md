## Summary

Keep open: current main still has the reported diagnostic gap, and the related repair PR is closed without being merged. The ownership block itself is intentional, but ownership-blocked plugins are still dropped before config validation can distinguish them from truly missing plugins.

## What This Changes

The issue asks OpenClaw to replace misleading stale-plugin warnings for ownership-blocked plugin directories, deduplicate ownership diagnostics, document the ownership constraint, and revisit Docker ownership reset behavior.

## Best Possible Solution

Preserve the ownership safety gate, propagate a blocked-present plugin id/path into validation, dedupe ownership diagnostics per boot, add focused docs, and treat any Docker ownership policy change as a separately confirmed follow-up.

## Reproduction Assessment

Yes by source inspection. A non-bundled uid-mismatch candidate is blocked and omitted from known plugin ids, so explicit `plugins.entries` or `plugins.allow` references still flow to the stale missing-plugin warning branch.

## Solution Assessment

Yes for the diagnostics/docs portion: carrying blocked-present state from discovery into config validation is the narrow maintainable fix. The Docker ownership-reset portion should not be changed until the restart-time behavior is independently confirmed.
