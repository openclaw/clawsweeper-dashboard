## Summary

Keep open: current main still has the ownership-blocked plugin diagnostic gap, and the related PR #76151 is closed without merging. The ownership safety gate appears intentional; the unresolved bug is that a present-but-blocked plugin can still be reported as stale or missing config and can emit duplicate ownership diagnostics.

## What This Changes

The issue asks for clearer diagnostics for ownership-blocked plugin directories, deduplicated suspicious-ownership warnings, documentation of the ownership constraint, and durable container ownership behavior.

## Best Possible Solution

Preserve the plugin ownership safety gate, propagate blocked-present plugin id/path state into config validation, deduplicate ownership diagnostics per boot, add focused docs and regression tests, and handle Docker ownership policy only after separate validation.

## Reproduction Assessment

Yes for the diagnostics portion by source inspection: a mismatched-ownership non-bundled plugin is blocked before registration, so explicit config refs still follow the missing-plugin warning branch. The Docker restart portion was not live-reproduced in this read-only review.

## Solution Assessment

Yes for the diagnostic/docs portion: carrying blocked-present state into validation and deduplicating diagnostics is the narrow maintainable fix. Docker ownership changes should be treated as a separately verified follow-up.
