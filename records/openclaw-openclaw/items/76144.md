## Summary

Keep open: current main still has a real diagnostic gap where an ownership-blocked plugin disappears from the discovered id set, so explicit `plugins.entries` and `plugins.allow` references can be reported as stale/missing instead of present-but-blocked. The current official Docker files do not clearly match the reported `/data` restart behavior, but the core warning and dedupe problem is actionable.

## What This Changes

The issue asks OpenClaw to make ownership-blocked plugin diagnostics accurate and non-duplicative, document the ownership constraint, and avoid Docker ownership resets that make the remediation non-durable.

## Best Possible Solution

Preserve the ownership block, but propagate a present-but-blocked plugin state into config and gateway diagnostics, dedupe the warning per boot, and document the supported ownership/remediation path.

## Reproduction Assessment

Yes. A non-bundled plugin path owned by a uid that is neither the gateway process uid nor root is deterministically blocked in discovery, and an explicit config reference to that id then follows the current missing-plugin warning path.

## Solution Assessment

Yes for the diagnostic/docs portion: the narrow maintainable fix is to distinguish blocked-present plugins from absent plugins and avoid duplicate logging. The container ownership-reset portion should be handled conservatively after confirming the current official image path.
