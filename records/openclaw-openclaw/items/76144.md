## Summary

Keep open: current main still has the reported diagnostic gap for ownership-blocked plugins, and the related repair PR #76151 is closed but unmerged. The ownership gate itself is intentional; the remaining bug is that blocked-present plugins can still be reported as stale/missing config and can produce duplicate diagnostics.

## What This Changes

The issue asks for clearer ownership-block diagnostics, deduplicated suspicious-ownership warnings, plugin ownership docs, and durable container ownership behavior for plugin directories.

## Best Possible Solution

Keep the ownership safety gate, carry a blocked-present plugin id/path into config validation, deduplicate ownership diagnostics per boot, add focused docs/tests, and treat Docker ownership policy as a separately verified follow-up if it still reproduces.

## Reproduction Assessment

Yes for the diagnostic bug, by source inspection: a non-bundled plugin with mismatched ownership is blocked and omitted from known ids, so explicit config references still follow the missing-plugin warning branch. I did not live-run the Docker restart path in this read-only review.

## Solution Assessment

Yes for the diagnostic/docs portion: distinguishing blocked-present plugins from absent plugins is the narrow maintainable fix. The container ownership-reset claim should be separately validated before changing image behavior.
