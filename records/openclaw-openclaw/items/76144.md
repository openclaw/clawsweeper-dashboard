## Summary

Keep open: current main still has the reported diagnostic bug. Ownership-blocked plugin candidates are omitted from known plugin ids, the block diagnostic is not tied to the configured id, and configured entries still fall through to stale-config guidance; the related repair PR is closed unmerged.

## What This Changes

The issue asks for clearer and deduplicated diagnostics for ownership-blocked plugin directories, documentation of the ownership requirement, and durable container ownership behavior.

## Best Possible Solution

Keep the ownership safety gate, propagate blocked-present plugin id/path state into config validation, dedupe ownership diagnostics per boot, add focused docs, and split Docker ownership-policy changes into a confirmed follow-up.

## Reproduction Assessment

Yes, by source inspection. A uid-mismatched non-bundled candidate is blocked before registration, the diagnostic lacks pluginId, and configured entries or allowlist references still flow to the stale missing-plugin warning branch.

## Solution Assessment

Yes for the diagnostics and docs portion: carrying blocked-present state from discovery into config validation and deduping diagnostics is the narrow maintainable fix. The container chown behavior should remain out of the automated patch unless maintainers confirm the intended image policy.

## Repair Work Prompt

Fix the ownership-blocked plugin diagnostics for https://github.com/openclaw/openclaw/issues/76144. Keep the existing ownership safety gate. Carry enough blocked-present state from plugin discovery into config validation so `plugins.entries.<id>` and `plugins.allow` do not say `plugin not found ... remove it from plugins config` when the plugin directory exists but is blocked for suspicious ownership; emit an actionable ownership-blocked warning instead. Deduplicate suspicious-ownership diagnostics once per plugin id/source path per gateway boot, including across scoped/shared discovery result merges. Add regression coverage for the blocked-present config warning and duplicate discovery diagnostics. Add a short public docs note for plugin directory ownership/remediation. Add a user-facing CHANGELOG.md fix entry. Do not change Dockerfile, docker-compose.yml, entrypoint behavior, UID policy, or broad container ownership semantics unless a maintainer explicitly confirms that follow-up.
