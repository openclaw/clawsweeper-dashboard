## Summary

Keep open: current main still drops ownership-blocked plugin candidates from the known plugin id set, so explicit config references can still be reported as missing/stale, and the duplicate diagnostic path is still present. The already-linked PR #76151 is the right active implementation track.

## What This Changes

The issue asks for ownership-blocked plugin directories to produce accurate, deduplicated diagnostics, user-facing documentation, and a separately considered durable container ownership story.

## Best Possible Solution

Preserve the ownership safety block, propagate a present-but-blocked plugin state into config diagnostics, deduplicate the ownership warning per boot, and document the supported remediation; handle container ownership policy only after separate current-image proof.

## Reproduction Assessment

Yes. A non-bundled plugin path owned by a uid that is neither the gateway process uid nor root is blocked during discovery, and an explicit `plugins.entries` or `plugins.allow` reference to that id then follows the current missing-plugin warning path.

## Solution Assessment

Yes for the diagnostic/docs portion: PR #76151 or an equivalent narrow patch is the maintainable fix. The container ownership-reset change is not yet proven against current official image code and should be split unless maintainers confirm it.
