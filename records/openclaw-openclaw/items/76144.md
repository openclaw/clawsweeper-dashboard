## Summary

Keep open: current main still drops ownership-blocked plugin candidates from the known plugin id set, so explicit config references can still be reported as missing/stale and duplicate ownership diagnostics can still be emitted. The linked PR #76151 is the active fix candidate for the narrow diagnostic and dedupe path.

## What This Changes

The issue requests accurate, deduplicated diagnostics for ownership-blocked plugin directories, user-facing remediation docs, and separate consideration of durable container ownership behavior.

## Best Possible Solution

Preserve the ownership safety block, propagate a present-but-blocked plugin state into config and gateway diagnostics, dedupe the warning per boot, and document the supported remediation; split container ownership policy unless maintainers confirm current-image behavior.

## Reproduction Assessment

Yes. A non-bundled plugin path owned by a uid that is neither the gateway process uid nor root is deterministically blocked during discovery, and an explicit `plugins.entries` or `plugins.allow` reference then follows the current missing-plugin warning path.

## Solution Assessment

Yes for the diagnostic/docs portion: PR #76151 or an equivalent narrow patch is the maintainable direction. The container ownership-reset change is unclear and should not be bundled without current-image proof.
