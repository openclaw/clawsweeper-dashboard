## Summary

Keep open: current main still treats ownership-blocked plugin directories as absent during config validation, so configured `plugins.entries` and `plugins.allow` references can still receive stale "plugin not found" guidance. The related repair PR #76151 is closed unmerged, and the current source lacks its blocked-present diagnostic and dedupe changes.

## What This Changes

The issue asks for ownership-blocked plugins to get accurate present-but-blocked diagnostics, duplicate ownership warnings to be deduped, the ownership constraint to be documented, and Docker chown durability to be considered.

## Best Possible Solution

Keep the ownership safety gate, propagate blocked-present plugin id/path state from discovery into config validation, dedupe per boot, and document the durable remediation while leaving Docker ownership policy as a separate confirmed follow-up.

## Reproduction Assessment

Yes, by source inspection. A non-bundled uid-mismatch candidate is blocked before registration and therefore omitted from known plugin ids, so explicit `plugins.entries` or `plugins.allow` references still reach the stale missing-plugin warning branch.

## Solution Assessment

Yes for the diagnostics and docs portion: carrying blocked-present state from discovery into config validation is the narrow maintainable fix. A Docker chown-policy change is a separate security/operations decision and should not be bundled into the first repair.
