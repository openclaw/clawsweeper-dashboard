## Summary

Keep open: current main still has the misleading diagnostic path for ownership-blocked plugins, and the linked repair PR was closed unmerged. The exact node-owned Docker repro may have shifted because the current image runs as `node`, but the blocked-present/stale-config warning bug is still source-reproducible.

## What This Changes

The issue asks to improve plugin loader diagnostics for ownership-blocked plugin directories, deduplicate warnings, document the ownership constraint, and revisit Docker chown behavior that can undo manual ownership fixes.

## Best Possible Solution

Keep the ownership safety gate, carry blocked-present plugin id/path state into config validation, emit actionable present-but-blocked guidance once per plugin/path, add focused docs, and split any Docker ownership-policy change for maintainer review.

## Reproduction Assessment

Yes for the diagnostic path by source inspection: a non-bundled plugin owned by neither the gateway uid nor root is blocked before registration, then explicit config references miss `knownIds` and take the stale warning branch. I did not establish a live current-image reproduction of the exact node-owned Docker case.

## Solution Assessment

Unclear for the whole issue. The diagnostics/docs fix is the narrow maintainable path, but changing Docker chown behavior needs separate maintainer policy review because the current image and setup script intentionally run/chown as `node`.
