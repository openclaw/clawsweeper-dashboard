## Summary

Keep this PR open for maintainer handling. Current main does not implement the heartbeat/session activity UI or progress-prompt changes, and the proposed patch still has blocking correctness and security-sensitive design issues around broadening normal Stop/session abort behavior.

## What This Changes

The PR adds WebChat heartbeat/session stop controls, a new session activity RPC, broader session abort cleanup, Control UI/OpenClawKit stream/progress handling, and system-prompt guidance for milestone/progress updates.

## Best Possible Solution

Split and rebase the work: keep ordinary generation Stop on `chat.abort`, design session activity and any autonomy-stop control against #39127 with protocol docs and authorization tests, and land progress-update prompt guidance separately with focused prompt or behavior coverage plus a changelog entry if user-facing behavior changes.

## Reproduction Assessment

Yes for the review defects. The PR diff and current-main code give a high-confidence static reproduction path for the abort routing, stream merge fallback, queue count, heartbeat drift, and timeout response issues; live UI testing is still needed before accepting a revised UX.

## Solution Assessment

No. The proposed patch is not the best way to solve the issue because it combines several independent surfaces and routes ordinary Stop through broader destructive cleanup; a split design with a distinct session/autonomy-stop API is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep normal Stop on chat.abort:** `ui/src/ui/controllers/chat.ts:337`
  - body: The regular WebChat Stop path now sends `sessions.abort`, whose handler also clears queues, stops subagents, cancels ACP sessions, and disables session-bound cron jobs. Stopping one answer can pause unrelated automation, so normal generation Stop should stay on `chat.abort` or use a distinct explicit autonomy-stop action.
  - confidence: 0.9
- **[P2] Replace snapshots when no delta is present:** `ui/src/ui/controllers/chat.ts:208`
  - body: When a non-prefix assistant snapshot arrives without `delta`, the fallback appends it to the current stream. Gateway snapshots are authoritative in that case, so rewritten or replayed text can be duplicated or corrupted in the UI and abort partials.
  - confidence: 0.86
- **[P2] Exclude dropped followups from pending size:** `src/auto-reply/reply/queue/state.ts:43`
  - body: `droppedCount` tracks discarded entries, not pending work. Including it makes `followupQueued` positive and `canStop: true` after the queue is actually empty, causing the stop control to appear for work that cannot be stopped.
  - confidence: 0.91
- **[P2] Return cleanup state after abort timeout errors:** `src/gateway/server-methods/sessions.ts:1030-1040`
  - body: By this point queues, subagents, ACP sessions, and cron jobs may already have been cleaned up. If the embedded-run wait times out, returning only `UNAVAILABLE` leaves clients unable to reconcile stopped work and can keep stale UI state or trigger redundant retries.
  - confidence: 0.75
- **[P3] Refresh heartbeat state when another client disables it:** `ui/src/ui/app-gateway.ts:413-419`
  - body: Incoming `heartbeat` events only set `heartbeatsEnabled` to true, while no connected event path clears it when another client disables heartbeats. The heart-pulse button can remain visible until reconnect even though heartbeats are off.
  - confidence: 0.78
