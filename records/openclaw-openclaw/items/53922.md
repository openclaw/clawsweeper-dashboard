## Summary

Keep this PR open, but it is not ready to merge: current main has not implemented the heartbeat/session stop or progress-prompt work, and the PR still carries blocking design and correctness issues around Stop semantics, session activity, stream merging, and queue accounting.

## What This Changes

This PR adds WebChat heartbeat/session stop controls, a new session activity RPC, broader session abort cleanup, Control UI/OpenClawKit stream and progress handling, and system-prompt guidance for milestone updates.

## Best Possible Solution

Split and rebase the work: keep ordinary generation Stop on chat.abort, design session activity/autonomy-stop against #39127 with protocol docs and authorization coverage, and land progress-prompt guidance separately with focused tests or prompt snapshots.

## Reproduction Assessment

Yes for the review defects: the PR diff and current-main source show normal Stop being rerouted through broader sessions.abort cleanup, dropped followups counted as pending work, and non-delta snapshots appended. I did not run a live UI reproduction in this read-only pass.

## Solution Assessment

No. The proposed patch is not the best way to solve the issue because it combines independent surfaces and broadens an existing abort RPC; a dedicated activity/autonomy-stop design plus a separate prompt-guidance change is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep normal chat Stop on chat.abort:** `ui/src/ui/controllers/chat.ts:337`
  - body: The regular WebChat Stop path now calls sessions.abort, whose handler also clears queues, stops subagents, cancels ACP sessions, and disables session-bound cron jobs. Stopping one active answer can pause unrelated automation, so ordinary generation Stop should stay on chat.abort or use a distinct explicit autonomy-stop action.
  - confidence: 0.9
- **[P2] Use snapshots as replacements when delta is absent:** `ui/src/ui/controllers/chat.ts:208`
  - body: When a non-prefix assistant snapshot arrives without delta, this fallback appends it to the existing stream. Gateway snapshots are authoritative in that case, so rewritten or replayed text can be duplicated or corrupted in the UI and abort partials.
  - confidence: 0.86
- **[P2] Exclude dropped followups from pending size:** `src/auto-reply/reply/queue/state.ts:43`
  - body: droppedCount tracks discarded queue entries, not pending work. Including it makes followupQueued positive and canStop true after the queue is actually empty, causing the stop control to appear for work that cannot be stopped.
  - confidence: 0.91
- **[P2] Return cleanup state after abort timeout errors:** `src/gateway/server-methods/sessions.ts:1030-1040`
  - body: By this point queues, subagents, ACP sessions, and cron jobs may already have been cleaned up. If the embedded-run wait times out, returning only UNAVAILABLE leaves clients unable to reconcile stopped work and can keep stale UI state or trigger redundant retries.
  - confidence: 0.75
- **[P3] Refresh heartbeat state when another client disables it:** `ui/src/ui/app-gateway.ts:413-419`
  - body: Incoming heartbeat events only set heartbeatsEnabled to true, while no connected event path clears it when another client disables heartbeats. The heart-pulse button can remain visible until reconnect even though heartbeats are off.
  - confidence: 0.78
