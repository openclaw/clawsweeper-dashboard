## Summary

The PR still contains meaningful unmerged feature work, but it is not merge-ready: current main keeps ordinary WebChat Stop scoped to `chat.abort`, the branch broadens session abort authority, omits real behavior proof, conflicts with current main, and overlaps the open session-activity API design.

## What This Changes

This PR adds WebChat heartbeat/session stop controls, a `sessions.activity` RPC, broader `sessions.abort` cleanup, stream/progress handling changes, OpenClawKit synthetic progress, and system-prompt progress-update guidance.

## Best Possible Solution

Split and rebase the work: keep ordinary generation Stop on `chat.abort`, design activity/autonomy stop against the open activity API issue with docs and authorization coverage, and land progress-prompt guidance separately with focused proof.

Yes for the PR defects by source inspection: the branch routes ordinary Stop through broader `sessions.abort`, counts dropped followups as pending, and appends no-delta snapshots. I did not run live UI tests in this read-only review.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep normal chat Stop on chat.abort:** `ui/src/ui/controllers/chat.ts:337`
  - body: The WebChat stop path now calls `sessions.abort`, whose server handler also clears queues, stops subagents, cancels ACP sessions, and disables bound cron jobs. Stopping one active answer can therefore pause unrelated session automation; ordinary generation Stop should stay on `chat.abort` or use a separate explicit autonomy-stop action.
  - confidence: 0.9
- **[P2] Use snapshots as replacements when no delta is present:** `ui/src/ui/controllers/chat.ts:208`
  - body: When a non-prefix assistant snapshot arrives without `delta`, this fallback appends the snapshot to the existing stream. Gateway snapshots are authoritative in that case, so rewritten or replayed text can be duplicated or corrupted in the UI and in locally committed abort partials.
  - confidence: 0.86
- **[P2] Exclude dropped followups from pending size:** `src/auto-reply/reply/queue/state.ts:43`
  - body: `droppedCount` tracks discarded queue entries, not pending work. Including it makes `followupQueued > 0` and `canStop: true` after the queue is actually empty, causing the stop control to appear for work that cannot be stopped.
  - confidence: 0.91
- **[P2] Return cleanup state after timeout errors:** `src/gateway/server-methods/sessions.ts:1030-1040`
  - body: By this point queues, subagents, ACP sessions, and cron jobs may already have been cleaned up. If the embedded-run wait times out, returning only `UNAVAILABLE` leaves clients unable to reconcile stopped work and can keep stale UI state or retry unnecessarily.
  - confidence: 0.75
- **[P3] Refresh heartbeat state when disabled elsewhere:** `ui/src/ui/app-gateway.ts:413-419`
  - body: Incoming heartbeat events only set `heartbeatsEnabled` to true, and no connected event path clears it when another client disables heartbeats. The heart-pulse button can remain visible until reconnect even though heartbeats are off.
  - confidence: 0.78
