## Summary

Keep open: the PR still contains blocking correctness issues, broadens Gateway session-control behavior in a security-sensitive way, overlaps the open session-activity API design, and lacks required real behavior proof.

## What This Changes

The PR adds WebChat heartbeat/session stop controls, a new `sessions.activity` RPC, broader `sessions.abort` cleanup, stream/progress handling changes, and system-prompt guidance for milestone updates.

## Best Possible Solution

Split and rebase the work: keep ordinary chat generation Stop on `chat.abort`, design session/autonomy activity APIs against the open canonical activity issue with docs and authorization coverage, and land progress-prompt guidance separately with focused tests and real behavior proof.

## Reproduction Assessment

Yes for the review defects from source inspection of the PR diff and current-main contracts: normal Stop would route through broader `sessions.abort` cleanup, dropped queue entries are counted as pending, no-delta snapshots append, and heartbeat state can drift. I did not run live UI tests in this read-only review.

## Solution Assessment

No. The proposed patch mixes independent prompt policy, UI, streaming, protocol, and session-control changes; the safer solution is a split sequence aligned with the canonical session activity design and existing chat abort contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep normal chat Stop on chat.abort:** `ui/src/ui/controllers/chat.ts:337`
  - body: The regular WebChat Stop path is routed through `sessions.abort`, whose new handler also clears queues, stops subagents, cancels ACP sessions, and disables session-bound cron jobs. Stopping one active answer can pause unrelated automation, so ordinary generation Stop should stay on `chat.abort` or use a separate explicit autonomy-stop action.
  - confidence: 0.9
- **[P2] Treat no-delta snapshots as replacements:** `ui/src/ui/controllers/chat.ts:208`
  - body: When a non-prefix assistant snapshot arrives without `delta`, the fallback appends it to the existing stream. Gateway snapshots are authoritative in that case, so rewritten or replayed text can be duplicated or corrupted in the UI and abort partials.
  - confidence: 0.86
- **[P2] Exclude dropped followups from pending size:** `src/auto-reply/reply/queue/state.ts:43`
  - body: `droppedCount` tracks discarded entries, not pending work. Including it makes `followupQueued > 0` and `canStop: true` after the queue is empty, causing the stop control to appear for work that cannot be stopped.
  - confidence: 0.91
- **[P2] Return cleanup state after timeout errors:** `src/gateway/server-methods/sessions.ts:1022-1031`
  - body: By this point queues, subagents, ACP sessions, and cron jobs may already have been cleaned up. If the embedded-run wait times out, returning only `UNAVAILABLE` leaves clients unable to reconcile stopped work and can keep stale UI state or trigger redundant retries.
  - confidence: 0.75
- **[P3] Refresh heartbeat state when another client disables it:** `ui/src/ui/app-gateway.ts:413-419`
  - body: Incoming heartbeat events only set `heartbeatsEnabled` to true, while no connected event path clears it when another client disables heartbeats. The heart-pulse button can remain visible until reconnect even though heartbeats are off.
  - confidence: 0.78
