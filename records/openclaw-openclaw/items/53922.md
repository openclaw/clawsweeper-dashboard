## Summary

Keep open: current main does not implement the PR, and the patch still has blocking source-level regressions around Stop semantics, stream merging, queue counts, and broadened session-abort authority; the broader session-activity design is already tracked by #39127 and #39141.

## What This Changes

The PR adds a WebChat heartbeat/session stop control, a new `sessions.activity` RPC, broader `sessions.abort` cleanup, Control UI/OpenClawKit stream and progress handling changes, and system-prompt guidance for milestone updates.

## Best Possible Solution

Split and rebase the work: keep ordinary chat generation Stop on `chat.abort`, design session activity/autonomy stop against #39127 with protocol docs and authorization coverage, and land progress-prompt guidance separately with focused tests.

## Reproduction Assessment

Yes for the review defects from source inspection of the PR diff and current-main contracts: normal Stop would route through broader `sessions.abort` cleanup, dropped queue entries are counted as pending, and no-delta snapshots append. I did not run live UI tests in this read-only review.

## Solution Assessment

No. The proposed patch combines independent UI, protocol, session-abort, OpenClawKit, and prompt-policy changes; the safer path is a split PR sequence aligned with the canonical session-activity design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep normal chat Stop on chat.abort:** `ui/src/ui/controllers/chat.ts:337`
  - body: The regular WebChat Stop path is routed through `sessions.abort`, whose new handler also clears queues, stops subagents, cancels ACP sessions, and disables session-bound cron jobs. Stopping one active answer can pause unrelated automation, so ordinary generation Stop should stay on `chat.abort` or use a separate explicit autonomy-stop action.
  - confidence: 0.9
- **[P2] Use snapshots as replacements when no delta is present:** `ui/src/ui/controllers/chat.ts:208`
  - body: When a non-prefix assistant snapshot arrives without `delta`, the fallback appends it to the current stream. Gateway snapshots are authoritative in that case, so replayed or rewritten text can be duplicated or corrupted in the UI and in abort partials.
  - confidence: 0.86
- **[P2] Exclude dropped followups from pending size:** `src/auto-reply/reply/queue/state.ts:43`
  - body: `droppedCount` tracks discarded entries, not pending work. Including it makes `followupQueued > 0` and `canStop: true` after the queue is empty, causing the stop control to appear for work that cannot be stopped.
  - confidence: 0.91
- **[P2] Return cleanup state after abort timeout errors:** `src/gateway/server-methods/sessions.ts:1030-1040`
  - body: By this point queues, subagents, ACP sessions, and cron jobs may already have been cleaned up. If the embedded-run wait times out, returning only `UNAVAILABLE` leaves clients unable to reconcile stopped work and can keep stale UI state or trigger redundant retries.
  - confidence: 0.75
- **[P3] Refresh heartbeat state when another client disables it:** `ui/src/ui/app-gateway.ts:413-419`
  - body: Incoming heartbeat events only set `heartbeatsEnabled` to true, while no connected event path clears it when another client disables heartbeats. The heart-pulse button can remain visible until reconnect even though heartbeats are off.
  - confidence: 0.78
