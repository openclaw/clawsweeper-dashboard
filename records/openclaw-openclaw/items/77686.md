## Summary

Keep open: the PR targets a real current-main Slack recovery bug, but the current diff introduces a blocking lifecycle regression by making the recovery restart backoff uncancellable by later manual stops.

## What This Changes

The PR adds a non-manual stop mode for health-monitor channel restarts, tracks recovery stop timeouts for auto-reconnect, adds gateway regression coverage and a changelog entry, and removes one unused WhatsApp import.

## Best Possible Solution

Keep the recovery/manual stop split, but preserve a cancellable pending-restart path so health-monitor recovery can reconnect while a later manual stop reliably cancels the retry.

## Reproduction Assessment

Yes, source-level: current main routes health-monitor restarts through the same manual stop path and the PR adds a focused timeout regression test for that failure. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No. The PR fixes the original poison-pill state, but dropping the abort signal for recovery backoff is not the safest fix because it makes later manual stops wait on an uncancellable retry sleep.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep recovery restart backoff cancellable:** `src/gateway/server-channels.ts:577-578`
  - body: When `recoveryStopTimedOut` is set, this drops the abort signal before sleeping for the auto-restart backoff. A user stop during that sleep can no longer cancel the tracked promise; with the real first retry delay of about 5-5.5s and the 5s stop timeout, the manual stop can falsely time out and leave the account marked running. Keep this pending restart cancellable while still avoiding the already-aborted recovery signal.
  - confidence: 0.88
