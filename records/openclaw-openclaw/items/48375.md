## Summary

Keep open: the feature is useful and current main still lacks it, but the PR has blocking runtime/security issues and does not include after-fix real behavior proof for the Headscale/Tailscale path.

## What This Changes

Adds a `gateway.tailscale.controlUrl` config field, `--tailscale-control-url` CLI flag, validation, docs, and tests for custom Tailscale-compatible control servers.

## Best Possible Solution

Keep the PR open and revise it so official Tailscale control URLs pass runtime checks, custom control URLs are enforced or clearly made non-enforcing, and all control-server URLs in errors are sanitized.

## Reproduction Assessment

Yes for the review findings: source inspection shows validation/runtime disagreement, an unenforced supported path, and raw daemon URL error output. No high-confidence live reproduction is available for the intended Headscale flow because the PR lacks runtime proof.

## Solution Assessment

No. The config/CLI/docs surface is a reasonable direction, but the current runtime handling is not the narrowest maintainable solution until it shares validation predicates, enforces or narrows `controlUrl` semantics, and redacts URL secrets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor official control URLs at runtime:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation accepts official Tailscale control URLs with `serve`/`funnel`, but this runtime guard throws for any non-empty `controlUrl` before checking `isOfficialTailscaleControlServer`. A config that validation accepts will still fail at startup, so use the same official-host predicate here before rejecting.
  - confidence: 0.9
- **[P2] Verify controlUrl against the daemon in tailnet mode:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The advertised Headscale path is `mode=off` with `bind=tailnet`, but the daemon ControlURL is only read when no `controlUrl` is set and mode is `serve`/`funnel`. A typo or stale `--tailscale-control-url` is accepted while the gateway binds to whichever tailnet the daemon is already joined to, making the new option non-enforcing in its supported path.
  - confidence: 0.86
- **[P2] Redact daemon control URLs before reporting startup failures:** `src/gateway/server-runtime-config.ts:165-169`
  - body: This error embeds `daemonControlUrl` verbatim, and `runGatewayCommand()` prints startup exceptions with `String(err)`. If the daemon login URL contains userinfo or query tokens, a rejected serve/funnel start leaks those values into ordinary terminal logs; strip credentials and query data before including it.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete log-disclosure risk by printing daemon control-server URLs without sanitization.

Concerns:

- **[medium] Daemon control URL can leak secrets:** `src/gateway/server-runtime-config.ts:167`
  - body: `daemonControlUrl` is interpolated directly into the thrown startup error, and the CLI prints that error. Control-server URLs should be sanitized before display because userinfo or query tokens would otherwise land in terminal or gateway logs.
  - confidence: 0.87
- **[medium] Configured control URL is not enforced in the supported path:** `src/gateway/server-runtime-config.ts:161`
  - body: The new `controlUrl` is accepted for the documented tailnet/off Headscale path without comparing it to the daemon's actual control server, so a stale value can leave the gateway exposed on an unintended tailnet while appearing configured for another.
  - confidence: 0.78
