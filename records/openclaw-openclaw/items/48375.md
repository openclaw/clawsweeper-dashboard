## Summary

Keep open: current main still lacks the custom Tailscale control-server feature, and this PR remains the canonical implementation thread, but the branch has blocking correctness/security findings plus missing real Headscale/Tailscale behavior proof.

## What This Changes

The branch adds `gateway.tailscale.controlUrl` and `--tailscale-control-url` with URL/mode validation, runtime checks, tests, schema help/labels, and Tailscale docs for Headscale/custom control servers.

## Best Possible Solution

Land a revised Gateway feature that uses one official/custom predicate, reads daemon control-server state from a supported Tailscale source, redacts URLs in errors, and either enforces the configured control server or clearly removes record-only semantics.

## Reproduction Assessment

Yes for the PR review findings: source inspection shows the config/runtime predicate mismatch, unsupported `tailscale status --json` `ControlURL` lookup, skipped `controlUrl` enforcement path, and unredacted daemon URL error. No live Headscale/Tailscale reproduction proof is attached.

## Solution Assessment

No. The feature direction is useful, but the current branch is not the narrowest maintainable solution until runtime validation, daemon-state discovery, URL redaction, and `controlUrl` semantics are aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the official-control predicate at runtime:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation accepts an explicit official Tailscale control URL with `serve` or `funnel`, but this guard rejects any non-empty `controlUrl`. That makes a supported config fail later at startup, so reuse the same official/custom predicate here.
  - confidence: 0.9
- **[P2] Read daemon control URLs from a supported source:** `src/gateway/server-runtime-config.ts:41-45`
  - body: This helper checks `ControlURL` fields in `tailscale status --json`, but upstream status JSON is `ipnstate.Status` and does not expose that field; `ControlURL` lives on prefs. Headscale daemons without `controlUrl` will miss the intended fast-fail path.
  - confidence: 0.86
- **[P2] Make controlUrl enforce the advertised tailnet setup:** `src/gateway/server-runtime-config.ts:161-164`
  - body: When the documented Headscale path uses `mode=off` with a tailnet bind, this branch skips daemon inspection whenever `controlUrl` is set. A typo or stale URL is silently ignored while OpenClaw binds to whatever tailnet the local daemon already joined.
  - confidence: 0.84
- **[P2] Redact daemon control URLs before throwing:** `src/gateway/server-runtime-config.ts:167`
  - body: This startup error includes `daemonControlUrl` verbatim, and gateway startup errors are printed to the terminal/logs. Strip userinfo and query data before including a daemon login server URL in the message.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds Gateway networking configuration with concrete URL-disclosure and misconfiguration risks that should be fixed before merge.

Concerns:

- **[medium] Daemon control URL can leak secrets:** `src/gateway/server-runtime-config.ts:167`
  - body: `daemonControlUrl` is interpolated directly into a startup error that the CLI prints; userinfo or query tokens in a login-server URL would be written to ordinary terminal/log output.
  - confidence: 0.82
- **[medium] Configured control URL is not enforced:** `src/gateway/server-runtime-config.ts:161`
  - body: The documented off/tailnet Headscale path accepts `controlUrl` without comparing it to the daemon's actual control server, so a stale value can leave the gateway exposed on an unintended tailnet while appearing configured for another.
  - confidence: 0.78
