## Summary

Keep this PR open because current main still lacks the Headscale/custom-control-server feature and the linked feature issue was closed as superseded by this PR, but the proposed patch still has blocking runtime semantics, security, and real-proof gaps.

## What This Changes

The PR adds Gateway `gateway.tailscale.controlUrl` and `--tailscale-control-url`, compatibility validation, schema UI hints, tests, and docs for Headscale/custom Tailscale control servers.

## Best Possible Solution

Land a revised Gateway feature with one shared official/custom control-server predicate, a supported daemon-state source or clearly documentation-only semantics, redacted URL display, and real Headscale/Tailscale proof.

## Reproduction Assessment

Yes for the PR review findings: the PR-head source and Tailscale upstream status contract show the runtime predicate mismatch, unsupported daemon lookup, skipped enforcement path, and URL redaction risk. No live Headscale reproduction artifact is attached.

## Solution Assessment

No. The feature direction is useful, but the current implementation is not the best maintainable solution until runtime semantics, supported daemon inspection, URL redaction, and real behavior proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor official control URLs at runtime:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation accepts official Tailscale control URLs with `serve` and `funnel`, but this runtime guard rejects any non-empty `controlUrl`. A deployment that explicitly pins `https://controlplane.tailscale.com` can pass config validation and then fail startup, so the runtime path should use the same official-control predicate.
  - confidence: 0.9
- **[P2] Read the daemon control URL from a supported source:** `src/gateway/server-runtime-config.ts:41-45`
  - body: This helper reads `ControlURL` from `tailscale status --json`, but Tailscale documents that output as `ipnstate.Status`, whose status and tailnet fields do not include `ControlURL`. Existing Headscale daemons without `gateway.tailscale.controlUrl` can still miss the intended fast-fail and fall through to the old Serve/Funnel failure path.
  - confidence: 0.88
- **[P2] Enforce controlUrl in the supported tailnet mode:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The documented Headscale path is `mode=off` with `bind=tailnet`, but this branch skips daemon inspection whenever `controlUrl` is set. A typo or stale URL is accepted while the gateway binds to whichever tailnet the local daemon is already joined to, making the new option non-enforcing in its advertised mode.
  - confidence: 0.85
- **[P2] Redact daemon control URLs before throwing:** `src/gateway/server-runtime-config.ts:167`
  - body: This startup error interpolates `daemonControlUrl` verbatim and the CLI prints startup exceptions directly. Strip userinfo and query data before including a daemon control URL in terminal output or diagnostics.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR adds networking configuration and leaves concrete URL-disclosure and unintended-tailnet exposure risks in the runtime path.

Concerns:

- **[medium] Daemon control URL can leak secrets:** `src/gateway/server-runtime-config.ts:167`
  - body: `daemonControlUrl` is interpolated directly into the startup error; if a daemon/login-server URL contains userinfo or query tokens, normal CLI or diagnostic logs can disclose it.
  - confidence: 0.82
- **[medium] Configured control URL is not enforced:** `src/gateway/server-runtime-config.ts:161`
  - body: The documented off/tailnet Headscale path accepts `controlUrl` without comparing it to the local daemon's actual control server, so a stale value can leave the gateway exposed on an unintended tailnet while appearing configured for another.
  - confidence: 0.78
