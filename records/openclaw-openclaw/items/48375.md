## Summary

Keep this PR open: it is the canonical implementation for the closed Headscale/custom-control-server request, but the current patch still has blocking runtime consistency and credential-redaction issues, plus missing real behavior proof for an external feature PR.

## What This Changes

The PR adds `gateway.tailscale.controlUrl`, `--tailscale-control-url`, Headscale documentation, validation/runtime guards, and focused tests for custom Tailscale control servers.

## Best Possible Solution

Merge a rebased version only after the control URL semantics are consistent across schema/runtime/CLI/docs, daemon URLs are sanitized before display, and redacted real Headscale/Tailscale behavior proof is added.

## Reproduction Assessment

Yes for the review findings: PR-head source shows config validation accepts an official control URL that runtime rejects, and the new daemon-control error prints the raw URL. I did not run tests because this review is constrained to read-only inspection.

## Solution Assessment

No. The feature direction is plausible, but the current implementation should share the official/custom URL classification in runtime, sanitize daemon URLs, and either enforce or clearly narrow what `controlUrl` means in the supported tailnet mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Allow official control URLs at runtime:** `src/gateway/server-runtime-config.ts:155`
  - body: `validateGatewayTailscaleControlUrl()` now explicitly allows `serve`/`funnel` when `controlUrl` points at the official Tailscale control plane, but this runtime guard still throws for any non-empty `controlUrl`. A config or CLI invocation using `https://controlplane.tailscale.com` therefore passes config validation and then fails startup, so runtime needs the same `isOfficialTailscaleControlServer()` exception.
  - confidence: 0.93
- **[P2] Redact daemon control URLs before throwing:** `src/gateway/server-runtime-config.ts:167`
  - body: `daemonControlUrl` comes from `tailscale status --json` and is interpolated directly into a startup error. If the daemon was started with a login-server URL containing userinfo or query tokens, the CLI will print those credentials in normal error output; sanitize the URL before including it in the message.
  - confidence: 0.9
- **[P2] Check the daemon when controlUrl is configured:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The documented Headscale path is `--bind tailnet --tailscale-control-url ...`, but this code only reads the daemon control URL when `controlUrl` is unset and mode is `serve`/`funnel`. In the supported `mode=off` path, a typo or stale control URL is accepted while the gateway binds to whatever tailnet the local daemon already joined; compare the configured URL to the daemon or make the option explicitly metadata-only.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete credential-disclosure risk by printing raw daemon control-server URLs in startup errors.

Concerns:

- **[medium] Raw control-server URL can leak credentials:** `src/gateway/server-runtime-config.ts:167`
  - body: The runtime error includes `daemonControlUrl` directly from Tailscale daemon status. URLs can carry userinfo or query tokens, and startup exceptions are printed to normal CLI/log output, so the error should redact userinfo, query, and fragment before display.
  - confidence: 0.9
