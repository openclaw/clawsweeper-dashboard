## Summary

Keep open: current main still lacks the requested custom-control-server Gateway surface, but this PR should not merge as-is because the branch has source-proven runtime/security issues and no real Headscale/Tailscale proof.

## What This Changes

The PR adds `gateway.tailscale.controlUrl` and `--tailscale-control-url`, URL/mode validation, tests, schema UI hints, and Tailscale docs for Headscale/custom control servers.

## Best Possible Solution

Land a revised Gateway feature that shares one official/custom predicate across config and runtime, uses a supported Tailscale source for daemon control URL checks, redacts URLs in errors, and either enforces the configured control server in off/tailnet mode or removes the record-only config semantics.

## Reproduction Assessment

Yes for the PR review findings: source inspection shows the config/runtime predicate mismatch, unsupported `tailscale status --json` `ControlURL` lookup, and skipped `controlUrl` verification path. No live end-to-end Headscale proof is attached.

## Solution Assessment

No. The feature direction is useful, but the proposed implementation is not the narrowest maintainable solution until runtime validation, daemon state discovery, URL redaction, and `controlUrl` semantics are aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor official control URLs at runtime:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation allows official Tailscale control URLs with `serve`/`funnel`, but this runtime guard rejects any non-empty `controlUrl`. A deployment that explicitly pins the official control plane passes config validation and then fails startup, so use the same official-control-server predicate here.
  - confidence: 0.9
- **[P2] Read the daemon control URL from a supported source:** `src/gateway/server-runtime-config.ts:41-45`
  - body: This helper reads `ControlURL` from `tailscale status --json`, but upstream Tailscale exposes `ControlURL` on prefs, not in the status JSON fields checked here. Existing Headscale daemons without `gateway.tailscale.controlUrl` still miss the intended fast-fail and fall through to the old Serve/Funnel failure path.
  - confidence: 0.86
- **[P2] Enforce controlUrl in the supported tailnet mode:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The documented Headscale path is `mode=off` with a tailnet bind, but this branch skips daemon inspection whenever `controlUrl` is set. A typo or stale URL is accepted while the gateway binds to whichever tailnet the local daemon is already joined to, making the new option non-enforcing in its advertised mode.
  - confidence: 0.84
- **[P2] Redact daemon login URLs before throwing:** `src/gateway/server-runtime-config.ts:167`
  - body: This startup error interpolates `daemonControlUrl` verbatim and the CLI prints startup exceptions directly. Strip userinfo and query data before including a daemon control URL in terminal or gateway logs.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds Gateway networking configuration and leaves concrete URL-disclosure and misconfiguration risks in the runtime path.

Concerns:

- **[medium] Daemon control URL can leak secrets:** `src/gateway/server-runtime-config.ts:167`
  - body: `daemonControlUrl` is interpolated directly into the startup error and printed by the CLI; userinfo or query tokens in a login server URL would be written to normal logs.
  - confidence: 0.82
- **[medium] Configured control URL is not enforced:** `src/gateway/server-runtime-config.ts:161`
  - body: The new `controlUrl` is accepted in the documented off/tailnet Headscale path without comparing it to the daemon's actual control server, so a stale value can leave the gateway exposed on an unintended tailnet while appearing configured for another.
  - confidence: 0.78
