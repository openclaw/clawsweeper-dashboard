## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Share the official-control check in runtime validation:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation and tests allow `https://controlplane.tailscale.com` with serve/funnel, but this runtime guard throws for any non-empty `controlUrl`. A config or CLI invocation that pins the official Tailscale control plane can pass validation and then fail startup, so this should use the same official/custom predicate.
  - confidence: 0.9
- **[P2] Read daemon control state from a supported source:** `src/gateway/server-runtime-config.ts:42-45`
  - body: This probes `Self.ControlURL`, `CurrentTailnet.ControlURL`, `Prefs.ControlURL`, and root `ControlURL` from `tailscale status --json`, but upstream `ipnstate.Status` does not expose those fields. Existing Headscale daemons without `gateway.tailscale.controlUrl` will miss the intended fast-fail and fall through to the old Serve/Funnel failure path.
  - confidence: 0.89
- **[P2] Enforce or remove controlUrl in tailnet mode:** `src/gateway/server-runtime-config.ts:161-164`
  - body: In the documented `mode=off` plus `bind=tailnet` path, setting `controlUrl` prevents daemon inspection and the value is never compared to the logged-in control server. A typo or stale URL is accepted while the gateway binds to whatever tailnet the daemon is already using, so enforce it or document it as non-authoritative metadata.
  - confidence: 0.84
- **[P2] Redact daemon login URLs before throwing:** `src/gateway/server-runtime-config.ts:167`
  - body: This error interpolates `daemonControlUrl` verbatim, and the CLI prints startup exceptions directly. If the daemon login-server URL contains userinfo or query tokens, a failed gateway start writes them to ordinary terminal/log output.
  - confidence: 0.83
