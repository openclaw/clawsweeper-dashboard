Overall confidence: 0.94

Full review comments:

- **[P1] Export the daemon status reader before importing it:** `src/gateway/server-runtime-config.ts:7`
  - body: `resolveGatewayRuntimeConfig()` now imports `readTailscaleStatusJson`, and the test mock expects the same export, but `src/infra/tailscale.ts` does not define or export that symbol in current main or in this diff. This will fail typecheck/module resolution before the new runtime checks can run.
  - confidence: 0.96
- **[P2] Allow official control URLs in serve/funnel:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation now permits official Tailscale control URLs, but runtime still throws for any non-empty `controlUrl` before checking the host. A deployment that explicitly pins `https://controlplane.tailscale.com` will fail startup even though it is still using supported Tailscale infrastructure.
  - confidence: 0.91
- **[P2] Redact daemon control URLs in startup errors:** `src/gateway/server-runtime-config.ts:167`
  - body: `daemonControlUrl` comes from the local Tailscale daemon and may include userinfo or query tokens from `tailscale up --login-server`. Gateway startup prints this thrown error, so sanitize it to an origin/host before including it in the message.
  - confidence: 0.9
- **[P2] Enforce controlUrl in the supported tailnet path:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The documented Headscale path is `mode=off` with `bind=tailnet`, but the new daemon check only runs for serve/funnel and skips checks when `controlUrl` is set. A typo or stale daemon login is silently ignored, so the gateway can bind whichever tailnet the daemon is already joined to.
  - confidence: 0.84
- **[P2] Regenerate the config schema artifacts:** `src/config/zod-schema.ts:729`
  - body: Adding `gateway.tailscale.controlUrl` to the Zod schema without refreshing `src/config/schema.base.generated.ts` leaves the live `config.schema`/Control UI schema without the new field and should fail the generated config schema check.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `src/cli/gateway-cli/run.ts:502-503`
  - body: This adds a user-facing config key and CLI flag, but the PR file list does not include `CHANGELOG.md`. Add a single-line entry under the active `### Changes` section with contributor credit before merge.
  - confidence: 0.96

## Security Review

Status: needs_attention

Summary: The diff has no dependency or workflow changes, but it adds Gateway startup paths that can leak daemon control-server URLs and silently ignore the configured Headscale control server in the supported mode.

Concerns:

- **[medium] Daemon control URL is not redacted:** `src/gateway/server-runtime-config.ts:167`
  - body: The new runtime error embeds `daemonControlUrl` verbatim. If the login-server URL contains userinfo or query tokens, ordinary CLI/log output can disclose those secrets during startup failure.
  - confidence: 0.9
- **[medium] Configured control URL is not enforced:** `src/gateway/server-runtime-config.ts:161`
  - body: In the documented `mode=off` + `bind=tailnet` path, a stale or mistyped `controlUrl` is not compared against the local daemon, so the Gateway can expose itself on a different tailnet than the operator intended.
  - confidence: 0.78
