## Summary

Keep open. Current main still lacks first-class custom Tailscale control-server support, so this PR is not obsolete; however the proposed patch is not merge-ready because its runtime validation contradicts its config validation and it can expose daemon control-server URLs in startup errors.

## What This Changes

The PR adds `gateway.tailscale.controlUrl` and `--tailscale-control-url`, custom-control-server validation, Headscale/Tailscale documentation, and focused config/CLI/runtime tests.

## Best Possible Solution

Keep this work open and finish it as a rebased Gateway feature: add the config, CLI, schema help, docs, and tests, but share one official/custom control-server predicate across validation and runtime, sanitize all URL output, and make `controlUrl` meaningful in the supported tailnet/off path by checking the daemon or clearly narrowing the contract.

## Reproduction Assessment

Yes for the PR findings: a config with `mode: "serve"` and `controlUrl: "https://controlplane.tailscale.com"` passes the PR’s config validation test but fails runtime at `server-runtime-config.ts:155`, and a daemon `ControlURL` containing userinfo/query is interpolated into the thrown error. Live Headscale end-to-end verification was not provided.

## Solution Assessment

No. The feature direction is plausible, but this patch is not yet the best implementation because runtime and config validation disagree, sensitive URLs are not redacted, and the only supported custom-server mode does not verify that the daemon is actually on the requested control server.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Skip the custom-server guard for official URLs:** `src/gateway/server-runtime-config.ts:155`
  - body: Config validation now accepts explicit official Tailscale control URLs in serve/funnel mode, but runtime throws for any non-empty `controlUrl` before checking the host. Deployments that pin the default control plane, such as `https://controlplane.tailscale.com`, would fail startup even though they are still using supported Tailscale infrastructure.
  - confidence: 0.91
- **[P2] Redact daemon control URLs in startup errors:** `src/gateway/server-runtime-config.ts:167`
  - body: `daemonControlUrl` comes from the local Tailscale daemon and may include userinfo or query tokens from `tailscale up --login-server`. This error is printed by the gateway CLI on startup failure, so include only a sanitized origin/host in the message.
  - confidence: 0.9
- **[P2] Verify controlUrl in tailnet mode:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The documented Headscale path is `mode=off` with `bind=tailnet`, but the new daemon check only runs for serve/funnel and the configured `controlUrl` is otherwise just carried through. A typo or stale daemon login is silently ignored, so the gateway can bind whichever tailnet the daemon is already joined to.
  - confidence: 0.82
