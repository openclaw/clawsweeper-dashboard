## Summary

This PR should stay open because current main does not provide the requested Headscale/custom-control-server Gateway surface, and the PR still has blocking runtime semantics and proof gaps before it is safe to merge.

## What This Changes

The PR adds `gateway.tailscale.controlUrl`, a `--tailscale-control-url` CLI flag, config/runtime validation, schema UI hints, tests, and Tailscale Gateway docs for Headscale/custom control servers.

## Best Possible Solution

Land a revised Gateway feature that has one shared official/custom control-server predicate, uses a supported daemon state source, redacts displayed URLs, and clearly defines whether `controlUrl` is documentation-only or enforced against the local Tailscale daemon.

## Reproduction Assessment

Yes for the PR review findings: the source diff and review comments provide a high-confidence path for the runtime predicate mismatch, daemon lookup concern, skipped off/tailnet enforcement path, and redaction risk. No live Headscale reproduction artifact is attached.

## Solution Assessment

No. The feature direction is useful, but this implementation still needs aligned runtime semantics, supported daemon inspection, URL redaction, and real Headscale/Tailscale proof before it is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor official control URLs at runtime:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation allows official Tailscale control URLs with `serve` and `funnel`, but this runtime guard rejects any non-empty `controlUrl`. A deployment that explicitly pins the official control plane can pass config validation and then fail startup, so the runtime path should use the same official-control-server predicate.
  - confidence: 0.9
- **[P2] Read the daemon control URL from a supported source:** `src/gateway/server-runtime-config.ts:41-45`
  - body: This helper reads `ControlURL` from `tailscale status --json`, but the reviewed path does not establish that field as a supported status JSON value. Existing Headscale daemons without `gateway.tailscale.controlUrl` can still miss the intended fast-fail and fall through to the old Serve/Funnel failure path.
  - confidence: 0.86
- **[P2] Enforce controlUrl in the supported tailnet mode:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The documented Headscale path is `mode=off` with a tailnet bind, but this branch skips daemon inspection whenever `controlUrl` is set. A typo or stale URL is accepted while the gateway binds to whichever tailnet the local daemon is already joined to, making the new option non-enforcing in its advertised mode.
  - confidence: 0.84
- **[P2] Redact daemon login URLs before throwing:** `src/gateway/server-runtime-config.ts:167`
  - body: This startup error interpolates `daemonControlUrl` verbatim and the CLI prints startup exceptions directly. Strip userinfo and query data before including a daemon control URL in terminal or gateway logs.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds Gateway networking configuration and leaves concrete URL-disclosure and misconfiguration risks in the runtime path.

Concerns:

- **[medium] Daemon control URL can leak secrets:** `src/gateway/server-runtime-config.ts:167`
  - body: `daemonControlUrl` is interpolated directly into the startup error and printed by the CLI; userinfo or query tokens in a login-server URL would be written to normal logs.
  - confidence: 0.82
- **[medium] Configured control URL is not enforced:** `src/gateway/server-runtime-config.ts:161`
  - body: The new `controlUrl` is accepted in the documented off/tailnet Headscale path without comparing it to the daemon's actual control server, so a stale value can leave the gateway exposed on an unintended tailnet while appearing configured for another.
  - confidence: 0.78
