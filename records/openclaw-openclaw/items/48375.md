## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Enforce the control URL before using tailnet bind:** `src/gateway/server-runtime-config.ts:161-164`
  - body: `--tailscale-control-url` is the advertised Headscale path, but in the only supported mode this resolver never applies or compares it with the daemon. A typo or stale login server is silently accepted and the gateway binds on whatever tailnet the daemon already uses, which can expose the dashboard on the wrong network; either enforce this with a supported source or make the option explicitly record-only metadata.
  - confidence: 0.82
- **[P2] Allow official control URLs through runtime validation:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation accepts `https://controlplane.tailscale.com` with `serve`/`funnel`, but runtime validation throws for any non-empty `controlUrl`. That makes a supported explicit official-control-plane config fail only at startup, so the runtime guard should use the same official-host predicate before rejecting.
  - confidence: 0.89
- **[P2] Read the daemon control URL from a supported source:** `src/gateway/server-runtime-config.ts:41-45`
  - body: This helper probes `tailscale status --json` for `ControlURL` fields that upstream `ipnstate.Status` and `TailnetStatus` do not expose, so real Headscale daemons can skip the intended fast-fail. Use a supported prefs/profile source or remove the daemon check instead of testing against mocked fields only.
  - confidence: 0.86
- **[P2] Redact daemon URLs before error output:** `src/gateway/server-runtime-config.ts:165-168`
  - body: The startup error embeds `daemonControlUrl` verbatim, and the CLI prints startup exceptions directly. If the login server URL contains userinfo or query tokens, this leaks credentials into normal terminal/log output; sanitize the URL before including it in diagnostics.
  - confidence: 0.78
