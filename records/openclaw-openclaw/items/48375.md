## Summary

Keep open. Current main still lacks the requested Headscale/custom Tailscale control-server surface, and the PR remains useful source work, but the proposed patch is not merge-ready because its runtime validation disagrees with config validation, one startup error can expose a daemon control URL, and the supported tailnet/off path does not verify the configured control server.

## What This Changes

The PR adds `gateway.tailscale.controlUrl`, `--tailscale-control-url`, config/schema UI metadata, validation/runtime tests, runtime daemon checks, and Headscale-oriented Tailscale Gateway documentation.

## Best Possible Solution

Finish this as a rebased Gateway feature: keep the explicit config/CLI/docs/tests, share one official-vs-custom control-server predicate across config and runtime, sanitize all URL output, verify the daemon control server in the documented tailnet/off path or narrow the documented contract, and add the required changelog entry.

## Reproduction Assessment

Yes. Code review of the PR diff gives high-confidence reproduction paths: an official `controlUrl` with `mode="serve"` is accepted by config validation but rejected by runtime, and a daemon `ControlURL` containing userinfo/query would be printed in the new startup error; no live Headscale E2E proof is provided.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the best implementation until runtime/config validation agree, daemon/control URLs are redacted before errors, and the supported tailnet/off path either verifies the daemon control server or documents that `controlUrl` is advisory only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Skip the custom-server guard for official URLs:** `src/gateway/server-runtime-config.ts:155-158`
  - body: Config validation now accepts explicit official Tailscale control URLs in serve/funnel mode, but runtime throws for any non-empty `controlUrl` before checking the host. Deployments that pin `https://controlplane.tailscale.com` would fail startup even though they are still using supported Tailscale infrastructure.
  - confidence: 0.91
- **[P2] Redact daemon control URLs in startup errors:** `src/gateway/server-runtime-config.ts:167`
  - body: `daemonControlUrl` comes from the local Tailscale daemon and may include userinfo or query tokens from `tailscale up --login-server`. This error is printed by the gateway CLI on startup failure, so include only a sanitized origin or host in the message.
  - confidence: 0.9
- **[P2] Verify controlUrl in tailnet mode:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The documented Headscale path is `mode=off` with `bind=tailnet`, but the new daemon check only runs for serve/funnel and the configured `controlUrl` is otherwise just carried through. A typo or stale daemon login is silently ignored, so the gateway can bind whichever tailnet the daemon is already joined to.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/cli/gateway-cli/run.ts:498-501`
  - body: This adds a user-facing config and CLI feature, but the PR file list does not include `CHANGELOG.md`. Add a single-line entry under the active `### Changes` section with contributor credit before merge.
  - confidence: 0.96
