## Summary

Keep this PR open: current main still lacks the custom Tailscale control-server surface, but the branch still has blocking runtime/security issues and no real Headscale/Tailscale runtime proof after the fix.

## What This Changes

The PR adds `gateway.tailscale.controlUrl`, `--tailscale-control-url`, validation/tests, and Tailscale Gateway docs for Headscale or other custom control servers.

## Best Possible Solution

Keep the feature direction, but revise the runtime so config validation and startup use the same official/custom predicate, custom control URLs are either enforced against the daemon or documented as non-enforcing metadata, and displayed URLs are sanitized.

## Reproduction Assessment

Yes for the review findings: source inspection shows validation/runtime disagreement, an unsupported `tailscale status --json` ControlURL assumption, and raw daemon URL error output. No high-confidence live reproduction exists for the intended Headscale flow because the PR does not include runtime proof.

## Solution Assessment

No. The config and docs direction is plausible, but the current implementation is not the best mergeable form until it shares validation predicates, enforces or narrows `controlUrl` semantics, and redacts control-server URLs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor official control URLs at runtime:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation allows official Tailscale control URLs with `serve`/`funnel`, but this runtime guard throws for any non-empty `controlUrl`. A config or CLI override that explicitly pins `https://controlplane.tailscale.com` will pass validation and then fail at startup, so use the same official-control-server predicate here before rejecting.
  - confidence: 0.9
- **[P2] Read the daemon control URL from a supported source:** `src/gateway/server-runtime-config.ts:41-45`
  - body: This helper looks for `ControlURL` under `tailscale status --json`, but upstream Tailscale's status JSON does not expose those fields; `ControlURL` lives in prefs instead. Existing Headscale installs without `gateway.tailscale.controlUrl` will therefore still skip the intended fast-fail and hit the old opaque serve/funnel failure path.
  - confidence: 0.86
- **[P2] Enforce controlUrl in the supported tailnet mode:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The documented Headscale path is `mode=off` with a tailnet bind, but this branch skips daemon inspection whenever `controlUrl` is set. A typo or stale `--tailscale-control-url` is accepted while the gateway binds to whichever tailnet the local daemon is already joined to, making the new option non-enforcing in its only supported mode.
  - confidence: 0.84
- **[P2] Redact daemon control URLs before reporting startup failures:** `src/gateway/server-runtime-config.ts:165-169`
  - body: This error embeds `daemonControlUrl` verbatim, and the CLI prints startup exceptions directly. If the daemon login URL includes userinfo or query tokens, a failed serve/funnel start leaks those values into normal terminal logs; strip credentials and query data before including it.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive Gateway networking behavior and currently has URL-disclosure and misconfiguration risks.

Concerns:

- **[medium] Daemon control URL can leak secrets:** `src/gateway/server-runtime-config.ts:167`
  - body: `daemonControlUrl` is interpolated directly into the thrown startup error, and `runGatewayCommand()` prints that error. Control-server URLs should be sanitized before display because userinfo or query tokens would otherwise land in terminal or gateway logs.
  - confidence: 0.82
- **[medium] Configured control URL is not enforced:** `src/gateway/server-runtime-config.ts:161`
  - body: The new `controlUrl` is accepted in the documented tailnet/off Headscale path without comparing it to the daemon's actual control server, so a stale value can leave the gateway exposed on an unintended tailnet while appearing configured for another.
  - confidence: 0.78
