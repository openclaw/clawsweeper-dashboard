## Summary

Keep open: current main still lacks this Gateway Headscale/custom-control-server feature, and the linked feature issue was closed in favor of this PR, but the branch is not merge-ready because runtime validation contradicts config validation, startup errors can expose daemon control-server URLs, the supported tailnet path does not enforce the configured control server, and real behavior proof is missing.

## What This Changes

The PR adds Gateway Tailscale custom control server configuration and a CLI flag, plus Headscale-oriented validation, docs, UI metadata, and tests.

## Best Possible Solution

Finish this as explicit Gateway Headscale support with consistent official/custom detection, redacted diagnostics, enforced or clearly documented `controlUrl` semantics in tailnet/off mode, updated config baselines, focused tests, and redacted real Gateway proof.

## Reproduction Assessment

Yes for the PR review findings via source inspection: official `controlUrl` plus serve/funnel is accepted by config validation but rejected by runtime, daemon URLs are interpolated into startup errors, and the mode=off path never compares daemon `ControlURL`. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The feature direction is plausible, but the patch needs a rebase plus consistent validation, URL redaction, explicit tailnet/off enforcement semantics, config baseline updates, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Make runtime validation match config validation:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation now allows an explicit official Tailscale control URL with `serve` or `funnel`, but `resolveGatewayRuntimeConfig()` throws for any non-empty `controlUrl` before checking `isOfficialTailscaleControlServer`. A config that pins `https://controlplane.tailscale.com` will pass validation and then fail at startup.
  - confidence: 0.92
- **[P2] Redact daemon control URLs before throwing:** `src/gateway/server-runtime-config.ts:167`
  - body: This error includes `daemonControlUrl` verbatim, and the CLI prints startup exceptions directly. If the local daemon was enrolled with a login-server URL containing userinfo or query tokens, a failed Gateway start can disclose those values in normal logs.
  - confidence: 0.9
- **[P2] Enforce the configured control URL in tailnet mode:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The documented Headscale path is `mode=off` with `bind=tailnet`, but this check only inspects the daemon for serve/funnel when `controlUrl` is absent. With `controlUrl` set, a typo or stale daemon login is ignored and the Gateway still binds whichever tailnet the daemon is already joined to.
  - confidence: 0.84
- **[P3] Regenerate the config baseline hashes:** `src/config/zod-schema.ts:729`
  - body: Adding `gateway.tailscale.controlUrl` to the public config schema changes the generated config-doc baseline, but the PR does not update `docs/.generated/config-baseline.sha256`. The config docs baseline check should be updated with the intended schema output.
  - confidence: 0.82
