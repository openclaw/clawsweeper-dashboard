## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor official control URLs at runtime:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation now accepts `https://controlplane.tailscale.com` with `serve`/`funnel`, but this runtime guard still throws for any non-empty `controlUrl` before checking `isOfficialTailscaleControlServer`. A user who pins Tailscale's official control plane gets a startup failure from a config the schema accepted.
  - confidence: 0.91
- **[P2] Use a supported daemon control-server source:** `src/gateway/server-runtime-config.ts:42-45`
  - body: The new daemon check reads `Self.ControlURL`, `CurrentTailnet.ControlURL`, `Prefs.ControlURL`, and top-level `ControlURL` from `tailscale status --json`, but upstream `ipnstate.Status` does not expose the control URL there. On a real Headscale daemon this likely returns `undefined`, skipping the intended fast-fail and falling through to the old late Serve/Funnel failure.
  - confidence: 0.88
- **[P2] Redact daemon URLs before throwing:** `src/gateway/server-runtime-config.ts:167`
  - body: This startup error includes `daemonControlUrl` verbatim. Login-server URLs can carry userinfo or query tokens, and gateway startup errors are printed to terminal/log output, so sanitize the URL before including it in the message.
  - confidence: 0.9
- **[P2] Enforce `controlUrl` in the supported mode:** `src/gateway/server-runtime-config.ts:154-155`
  - body: The advertised Headscale path is `mode=off` with `bind=tailnet`, but that path never compares `gateway.tailscale.controlUrl` to the daemon's actual control server. A typo or stale value is silently ignored and the gateway binds to whichever tailnet the daemon is already on, so either enforce it with a supported source or clearly make the field record-only metadata.
  - confidence: 0.82
