## Summary

Keep open. Current main still lacks custom Tailscale control-server support, and this PR remains the canonical implementation path for that feature, but the current branch still has blocking runtime-contract/security issues and lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds `gateway.tailscale.controlUrl` and `--tailscale-control-url`, validation/runtime guards, tests, schema UI hints, and Tailscale docs for Headscale/custom control servers.

## Best Possible Solution

Land a revised Gateway feature with one shared official/custom control-server predicate, a supported daemon-state source or explicit record-only contract, redacted URL display, focused tests/docs, and real Headscale/Tailscale proof.

## Reproduction Assessment

Yes for the PR findings: the PR head source plus the upstream Tailscale type contract show the runtime mismatch, unsupported daemon lookup, skipped enforcement, and URL disclosure paths. No live Headscale artifact is attached.

## Solution Assessment

No. The feature direction is useful, but the current patch is not the best maintainable solution until the runtime contract, daemon lookup, URL redaction, and real behavior proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the official-control predicate in runtime validation:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation and the new test accept `https://controlplane.tailscale.com` with `serve`/`funnel`, but this runtime guard throws for any non-empty `controlUrl`. A config or CLI invocation that pins official Tailscale can pass earlier validation and then fail startup, so this guard should share the same official/custom predicate.
  - confidence: 0.9
- **[P2] Read daemon control state from a supported status field:** `src/gateway/server-runtime-config.ts:42-45`
  - body: This probes `Self.ControlURL`, `CurrentTailnet.ControlURL`, `Prefs.ControlURL`, and root `ControlURL` in `tailscale status --json`, but upstream `ipnstate.Status` does not expose those paths. Existing Headscale daemons without `gateway.tailscale.controlUrl` will miss the intended fast-fail and fall through to the old Serve/Funnel failure path.
  - confidence: 0.89
- **[P2] Enforce controlUrl in the tailnet bind path:** `src/gateway/server-runtime-config.ts:161-164`
  - body: In the documented `mode=off` plus `bind=tailnet` path, setting `controlUrl` prevents daemon inspection and the value is never compared to the logged-in control server. A typo or stale URL is accepted while the gateway binds to whatever tailnet the daemon is already using; enforce it or document it as non-authoritative metadata.
  - confidence: 0.82
- **[P2] Redact daemon login URLs before reporting startup errors:** `src/gateway/server-runtime-config.ts:167`
  - body: This error interpolates `daemonControlUrl` verbatim, and the CLI prints startup exceptions directly. If the daemon login server URL contains userinfo or query tokens, the failed start writes them to normal terminal/log output; strip credentials and query before including it.
  - confidence: 0.82

## Security Review

  - body: `daemonControlUrl` is included verbatim in a startup error that the CLI prints, so userinfo or query tokens in a login-server URL could be written to ordinary logs.
