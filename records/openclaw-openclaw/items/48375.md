## Summary

Keep this PR open. Current main still lacks the Headscale/custom-control-server surface, but the PR has blocking runtime/security issues and is missing real behavior proof for the external feature path.

## What This Changes

Adds `gateway.tailscale.controlUrl` and `--tailscale-control-url` with Headscale-oriented validation, docs, schema help/labels, and focused tests.

## Best Possible Solution

Finish this as explicit Gateway Headscale support with consistent official/custom detection, redacted diagnostics, enforced or clearly documented `controlUrl` semantics, current-main rebase, generated config baseline updates, focused tests, and redacted real Gateway proof.

## Reproduction Assessment

Yes for the PR review findings by source inspection: the PR head accepts official control URLs in config but rejects them at runtime, interpolates daemon URLs into startup errors, and never compares the configured control URL in the documented mode=off/tailnet path. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The feature direction is plausible, but this patch is not yet the narrow maintainable solution because it needs consistent validation, redaction, explicit runtime semantics, a rebase, baseline updates, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Make runtime validation match config validation:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation allows official Tailscale control URLs with `serve` or `funnel`, but `resolveGatewayRuntimeConfig()` throws for any non-empty `controlUrl`. A deployment that pins `https://controlplane.tailscale.com` passes config validation and then fails at startup.
  - confidence: 0.92
- **[P2] Redact daemon control URLs before throwing:** `src/gateway/server-runtime-config.ts:167`
  - body: This error includes `daemonControlUrl` verbatim, and the CLI prints startup exceptions directly. If the local daemon was enrolled with a login-server URL containing userinfo or query tokens, a failed Gateway start can disclose those values in normal logs.
  - confidence: 0.9
- **[P2] Enforce controlUrl in tailnet mode:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The documented Headscale path is `mode=off` with `bind=tailnet`, but this check only inspects the daemon for serve/funnel when `controlUrl` is absent. With `controlUrl` set, a typo or stale daemon login is ignored and the Gateway still binds whichever tailnet the daemon is already joined to.
  - confidence: 0.84
- **[P3] Regenerate the config docs baseline:** `src/config/zod-schema.ts:732`
  - body: Adding `gateway.tailscale.controlUrl` changes the generated config docs surface, but the PR does not include the current-main `docs/.generated/config-baseline.sha256` update. After rebasing, the config docs baseline check should be updated with the intended schema output.
  - confidence: 0.78
