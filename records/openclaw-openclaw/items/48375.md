## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Make controlUrl enforce the tailnet target:** `src/gateway/server-runtime-config.ts:161-164`
  - body: The documented Headscale path is `bind=tailnet` with `mode=off`, but this code only inspects the daemon URL for `serve`/`funnel` when no `controlUrl` is set. As a result, `--tailscale-control-url` is ignored in its only supported mode, so a typo or stale daemon login can still expose the gateway on the wrong tailnet.
  - confidence: 0.91
- **[P2] Allow official control URLs at runtime:** `src/gateway/server-runtime-config.ts:155-159`
  - body: Config validation accepts official Tailscale control URLs with `serve`/`funnel`, but this runtime guard throws for any non-empty `controlUrl`. A config using `https://controlplane.tailscale.com` therefore passes validation and then fails at startup.
  - confidence: 0.93
- **[P2] Use a supported daemon control-server source:** `src/gateway/server-runtime-config.ts:41-45`
  - body: The Tailscale `status --json` contract points to `ipnstate.Status`, which does not expose `ControlURL` on the paths read here. These mocked fields are unlikely to detect a real Headscale daemon, so the intended fast-fail for existing Headscale logins will silently fall back to the old late failure.
  - confidence: 0.86
- **[P2] Redact daemon control URLs before errors:** `src/gateway/server-runtime-config.ts:166-168`
  - body: This error embeds the daemon URL verbatim, and the CLI prints startup exceptions directly. If a login-server URL contains userinfo or query tokens, a failed gateway start can leak those secrets into normal terminal or service logs.
  - confidence: 0.84
