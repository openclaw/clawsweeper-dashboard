## Summary

Keep open: current main matches the reported device-token media behavior at source/test level, but existing docs and tests also treat `chat.history`-scoped access as an operator read surface, so this needs maintainer/secops judgment rather than cleanup closure.

## What This Changes

The issue asks for `/api/chat/media/outgoing/:sessionKey/:attachmentId/full` to require requester-session ownership for device-token callers instead of treating device-token auth as privileged chat-history access.

## Best Possible Solution

Decide the intended authorization contract for managed outgoing media, then either bind media downloads to an authenticated requester/session or explicitly document and test chat-history-scoped media access as the supported behavior.

## Reproduction Assessment

Yes, at source level. I did not execute tests because this was a read-only review, but current code and an existing test show a device-token request without requester-session ownership returns image bytes.

## Solution Assessment

Unclear. The requested direction is plausible hardening, but it conflicts with current tests/docs and the existing requester-session header is not identity-bound, so the best fix needs a maintainer/secops authorization decision first.
