## Summary

Keep open: current main has a merged partial hardening path, but the media gate still depends on the generic OpenAI-compatible owner helper, whose device-token/trusted-identity path can inherit default `operator.admin` when `x-openclaw-scopes` is absent. That leaves the authorization contract too security-sensitive and unresolved for cleanup closure.

## What This Changes

Device-token-authenticated clients with chat-history access should not be able to download managed outgoing media from sessions they do not own unless an explicit elevated admin path is intended.

## Best Possible Solution

Define a media-specific authorization predicate that rejects client-declared requester-session ownership and requires explicit authenticated owner/admin proof for managed outgoing media, then cover production helper semantics with regression tests.

## Reproduction Assessment

Yes, at source level. On current main, a device-token-style non-shared HTTP auth result can enter the trusted-scope path, omit `x-openclaw-scopes`, inherit default `operator.admin`, pass `resolveOpenAiCompatibleHttpSenderIsOwner`, and reach byte serving if the media record matches.

## Solution Assessment

No, not fully. The merged helper-based hardening removes the old `privilegedAccess` branch, but the safer solution is a media-specific strict owner/admin check whose tests use production default-scope semantics.
