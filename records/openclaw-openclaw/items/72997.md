## Summary

Keep open. Current main still exposes channel status/start/stop/logout plus automatic config/secrets-driven channel restarts, but it does not expose the requested one-step manual channel reload CLI command or Gateway RPC; the related watcher issue is adjacent automatic-reload work, not a replacement for this operator feature.

## What This Changes

Add a targeted channel reload operator command/RPC that restarts only a named channel plugin without restarting the Gateway or disturbing other channels.

## Best Possible Solution

Expose an additive admin-scoped `channels.reload` RPC and a matching `openclaw channels reload --channel <name> [--account <id>]` CLI wrapper that reuses the existing channel lifecycle, with docs, protocol types, tests, and a changelog entry.

## Reproduction Assessment

Yes. A current-main source/docs check reproduces the gap: the CLI, Gateway method list, admin scopes, protocol schemas, handlers, tests, and docs expose start/stop/logout/status paths but no one-step manual reload surface.

## Solution Assessment

Yes. A generic reload method layered over the existing stop/start lifecycle is the narrow maintainable fix; provider-specific recovery policy or a full Gateway restart would be broader than this request.

## Repair Work Prompt

Implement the feature requested by https://github.com/openclaw/openclaw/issues/72997: add a targeted channel reload operator surface without restarting the Gateway. Add an additive admin-scoped Gateway RPC, preferably `channels.reload`, that accepts `channel` and optional `accountId`, validates like the existing channel lifecycle methods, and reloads only the selected channel/account by reusing the existing stop/start lifecycle and active-work deferral semantics where appropriate. Add a CLI wrapper using the existing plural command style, e.g. `openclaw channels reload --channel <name> [--account <id>]`, and document it in the channels CLI and Gateway protocol docs. Keep the fix generic in core; do not add provider-specific Telegram/WhatsApp policy, do not solve the broader watcher debounce/scoped automatic reload work from #74122, and do not change plugin install/update behavior. Add focused tests for the RPC handler, method registry/scope/protocol exports, CLI parsing/callGateway behavior, and docs/changelog coverage. Because this is user-facing, add a single-line `CHANGELOG.md` entry under the active version fixes/changes section with appropriate non-bot attribution if known.
