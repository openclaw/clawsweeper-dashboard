## Summary

Keep this open: current main and the latest release already support internal per-channel hot reload for config/secrets changes, but they do not expose the requested manual `channels reload` CLI, gateway RPC, or gateway-tool `channel.reload` action. The adjacent issues cover reload granularity, deferral races, and gateway load, not this operator command surface.

## What This Changes

The issue requests a targeted command or gateway action to reload one channel provider without restarting the full gateway.

## Best Possible Solution

Design an operator-admin channel reload RPC and CLI that reuse the existing ChannelManager stop/start and active-work deferral semantics, then optionally expose a bounded gateway-tool action after the authorization and lifecycle contract is agreed.

## Reproduction Assessment

Not applicable. This is a feature request, and source plus latest-release inspection shows the requested manual command/RPC/tool action is absent rather than an existing behavior failing.

## Solution Assessment

Unclear. A targeted reload command is a plausible direction, but the safest solution is an explicit operator-admin lifecycle API that preserves existing reload deferral and rollback semantics rather than an ad hoc gateway-tool-only action.
