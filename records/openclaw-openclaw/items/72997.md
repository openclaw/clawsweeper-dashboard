## Summary

Keep open. Current main still has per-channel lifecycle primitives and config/secrets-triggered channel restarts, but it does not expose the requested manual single-channel reload CLI command or gateway action; `channels.start` also cannot force a reconnect for an already-running channel.

## What This Changes

The issue requests a targeted channel reload command or gateway action that reconnects one named channel without restarting the gateway or interrupting other channels.

## Best Possible Solution

Add an additive, admin-gated `channels.reload` gateway RPC plus an `openclaw channels reload --channel <name> [--account <id>] [--json]` wrapper that validates the channel/account, stop-starts only that channel runtime through the existing channel manager, returns refreshed runtime status, and documents the CLI/RPC behavior without folding in watcher debounce work.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the gap: CLI/docs/method inventory expose status/start/logout but no reload, and the existing start path returns early for already-running channel tasks.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse the existing `stopChannel` and `startChannel` lifecycle seam behind an additive admin-scoped RPC and CLI wrapper, rather than restarting the gateway or adding Telegram/WhatsApp-specific recovery logic.

## Repair Work Prompt

Implement a targeted channel reload surface for https://github.com/openclaw/openclaw/issues/72997. Add an additive admin-gated gateway RPC, preferably `channels.reload` to match existing `channels.*` methods, with params `{ channel, accountId? }`. Validate the channel, reject unknown or unsupported channel runtimes, call the existing channel manager `stopChannel(channel, accountId)` then `startChannel(channel, accountId)`, and return refreshed channel/account status similar to `channels.start`; if `accountId` is omitted, use a clearly documented whole-channel reload or default-account behavior and cover it in tests. Add `openclaw channels reload --channel <name> [--account <id>] [--json]` using existing gateway RPC CLI patterns. Update protocol schemas/validators, method list/scopes, CLI help/docs, gateway protocol docs, and CHANGELOG if the final change is user-facing. Keep the fix generic: do not add Telegram/WhatsApp-specific code, do not implement watcher debounce from #74122, do not change plugin registry reload behavior, and do not restart the gateway process. Deliberately handle or document active-work behavior for manual reloads so it does not accidentally bypass the existing config hot-reload deferral contract.
