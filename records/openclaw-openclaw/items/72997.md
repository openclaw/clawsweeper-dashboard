## Summary

Keep this issue open. Current main has per-channel lifecycle primitives and config-triggered hot reload, but it does not expose a manual single-channel reload command or gateway action, so the requested operator workflow remains missing.

## What This Changes

The issue requests a targeted `openclaw channel reload <provider>` or equivalent gateway action that reloads one channel without restarting the gateway or disrupting other channels.

## Best Possible Solution

Add an additive, admin-gated channel reload surface that validates the channel/account, stop-starts only that runtime through the existing channel manager, reports the resulting runtime status, and documents the CLI/RPC behavior while keeping watcher debounce work separate.

## Reproduction Assessment

Yes. A source/docs check against current main reproduces the gap: the CLI and gateway method registry have status/start/logout but no reload action, and `channels.start` is not a forced reconnect for an already-running channel.

## Solution Assessment

Yes. The narrowest maintainable direction is to reuse the existing `stopChannel` and `startChannel` lifecycle seam behind a new `channels.reload` RPC plus CLI wrapper, rather than restarting the gateway or adding channel-specific behavior.

## Repair Work Prompt

Implement a targeted channel reload surface for https://github.com/openclaw/openclaw/issues/72997. Add an additive admin-gated gateway RPC, preferably `channels.reload` to match existing `channels.*` methods, that validates `channel` and optional `accountId`, rejects unknown or unsupported channel runtimes, calls the existing channel manager `stopChannel(channel, accountId)` then `startChannel(channel, accountId)`, and returns channel/account status similar to `channels.start`. Add `openclaw channels reload --channel <name> [--account <id>] [--json]` using existing gateway RPC CLI patterns. Update protocol schemas/validators, method listing/scopes, CLI help/docs, and changelog with non-bot credit to the reporter. Keep the fix generic; do not add Telegram/WhatsApp-specific code, do not implement watcher debounce from the related watcher issue in this PR, and do not restart the gateway process. Consider whether to reuse or document the existing config hot-reload active-work deferral behavior for manual reloads.
