## Summary

Keep open. Current main still exposes per-channel lifecycle building blocks, but not the requested one-step operator command or Gateway RPC to reload a single channel without restarting the Gateway.

## What This Changes

Add a targeted channel reload command/RPC that disconnects and reconnects only the named channel while preserving the Gateway process and other sessions.

## Best Possible Solution

Add an additive, admin-scoped `channels.reload` Gateway RPC plus `openclaw channels reload --channel <name> [--account <id>]` CLI wrapper that reuses the existing generic stop/start lifecycle and documents the operator behavior.

## Reproduction Assessment

Yes. A source and docs check against current main reproduces the gap: the CLI, method registry, admin scopes, protocol schema, tests, and docs expose status/start/stop/logout paths but no single manual reload surface.

## Solution Assessment

Yes. A generic reload method built on the existing stop/start channel manager seam is the narrow maintainable path; provider-specific recovery policy or a full Gateway restart would be broader than this request.
