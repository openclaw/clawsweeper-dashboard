## Summary

Keep open. Current main has per-channel start/stop/logout RPCs and config/health restart building blocks, but it still lacks the requested one-step manual channel reload command or Gateway RPC.

## What This Changes

The issue requests a targeted CLI or Gateway action to reload one named channel provider without restarting the Gateway.

## Best Possible Solution

Add an additive, admin-scoped `channels.reload` Gateway RPC plus an `openclaw channels reload --channel <name> [--account <id>]` CLI wrapper that reuses the existing channel stop/start lifecycle and documents the operator behavior.

## Reproduction Assessment

Yes. A source and docs inspection against current main reproduces the gap: the channel CLI, Gateway registry, method scopes, protocol schema, handlers, tests, and docs expose start/stop/logout/status paths but no manual reload surface.

## Solution Assessment

Yes. A generic RPC and CLI wrapper over the existing channel lifecycle seam is the narrow maintainable fix; provider-specific recovery policy or a full Gateway restart would be broader than the request.

## Repair Work Prompt

Implement the targeted channel reload feature requested at https://github.com/openclaw/openclaw/issues/72997. Add an additive admin-scoped `channels.reload` Gateway RPC with typed protocol params/result, method registry entry, scope mapping, and server handler that resolves channel/account like existing start/stop, stops only the selected channel account, then starts it again without restarting the Gateway. Add `openclaw channels reload --channel <name> [--account <id>]` with human and JSON output, update route/preload tests as needed, document the CLI and Gateway protocol, and add a user-facing changelog entry. Reuse existing generic channel lifecycle helpers; do not add provider-specific recovery policy, do not broaden watcher debounce/scoped reload work from #74122, and preserve active session/active work behavior as closely as the current config channel reload path does.
