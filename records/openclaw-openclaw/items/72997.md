## Summary

Keep open. Current main has lower-level per-channel lifecycle seams, including a new admin-scoped `channels.stop` RPC and existing hot-reload stop/start logic, but it still does not provide the requested single manual `openclaw channels reload <provider>` command or `channels.reload` Gateway RPC.

## What This Changes

Request a targeted operator command/RPC to disconnect and reconnect one channel provider/account without restarting the Gateway or interrupting other channels/sessions.

## Best Possible Solution

Add an additive, admin-scoped `channels.reload` Gateway RPC and matching `openclaw channels reload` CLI command that validates channel/account through existing channel plugin helpers, stops and restarts only the selected account/channel via the generic lifecycle manager, returns runtime status, and updates protocol/CLI docs, tests, and changelog without embedding Telegram or WhatsApp-specific policy in core.

## Reproduction Assessment

Yes. A source and docs check against current main reproduces the gap: the CLI, method registry, admin scopes, protocol schema, tests, and docs expose status/start/stop/logout building blocks but no single manual reload surface.

## Solution Assessment

Yes. The narrow maintainable fix is a generic admin-gated reload method built on the existing stop/start lifecycle seam; provider-specific recovery policy or a full Gateway restart would be broader than the request.
