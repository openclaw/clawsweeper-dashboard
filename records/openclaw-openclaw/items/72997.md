## Summary

Keep open. Current main still lacks a one-step manual channel reload CLI command or Gateway RPC; existing config/secrets hot reload only covers automatic restart triggers.

## What This Changes

Request to add a targeted channel reload command or RPC that restarts one named channel without restarting the Gateway or disrupting other channels.

## Best Possible Solution

Expose an additive admin-scoped `channels.reload` RPC and a matching `openclaw channels reload --channel <name> [--account <id>]` wrapper that reuses existing channel stop/start lifecycle behavior.

## Reproduction Assessment

Not applicable as a bug reproduction, but the requested surface is source-verifiable. CLI, Gateway method registry/admin scopes, protocol schema, handlers, and docs expose status/start/stop/logout but no manual reload.

## Solution Assessment

Yes. A generic admin-scoped reload method layered over existing stop/start lifecycle is the narrow maintainable direction; provider-specific recovery policy or a full Gateway restart would be broader than this request.

## Repair Work Prompt

Implement the feature requested by this issue: add a targeted channel reload operator surface without restarting the Gateway. Add an additive admin-scoped Gateway RPC, preferably `channels.reload`, that accepts `channel` and optional `accountId`, validates like the existing channel lifecycle methods, and reloads only the selected channel/account by reusing the existing stop/start lifecycle and active-work deferral semantics where appropriate. Add a CLI wrapper using the existing plural command style, e.g. `openclaw channels reload --channel <name> [--account <id>]`, and document it in the channels CLI and Gateway protocol docs. Keep the fix generic in core; do not add provider-specific Telegram/WhatsApp recovery policy, do not solve the broader watcher debounce/scoped automatic reload work from #74122, and do not change plugin install/update behavior. Add focused tests for the RPC handler, method registry/scope/protocol exports, CLI parsing/Gateway call behavior, and docs/changelog coverage. Because this is user-facing, add a single-line `CHANGELOG.md` entry under the active version changes/fixes section with appropriate non-bot attribution if known.
