## Summary

Keep this open. Current main has channel lifecycle primitives, config-triggered channel hot reload, and `secrets.reload`-triggered restarts, but it still does not expose the requested manual single-channel reload command or `channels.reload` gateway RPC; the related watcher debounce issue is adjacent rather than a canonical replacement.

## What This Changes

Request a targeted operator command or gateway action to reload one named channel without restarting the whole Gateway or interrupting unrelated channels and sessions.

## Best Possible Solution

Add an additive `channels.reload` admin RPC plus `openclaw channels reload` CLI wrapper that validates channel/account input, stop-starts only the selected runtime through the existing channel manager, returns the resulting channel/account status, and documents the behavior without folding in the separate watcher debounce work.

## Reproduction Assessment

Yes. A source/docs check against current main reproduces the gap: the CLI, protocol schema, method list, and admin scope table expose status/start/logout but no reload action, while `channels.start` is not a forced reconnect for a running account.

## Solution Assessment

Yes. Reusing the existing channel manager `stopChannel`/`startChannel` lifecycle seam behind a new admin-gated RPC is the narrowest maintainable fix; adding provider-specific recovery paths or restarting the whole Gateway would be broader and less consistent with current architecture.
