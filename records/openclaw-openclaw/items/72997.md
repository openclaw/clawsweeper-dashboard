## Summary

Keep open. Current main has automatic channel-scoped config hot reload, but it still lacks the requested operator-facing reload command or gateway RPC that deliberately stop/starts a named channel after runtime errors without restarting the whole gateway.

## What This Changes

Request an `openclaw channel reload <provider>` or equivalent gateway action that reconnects only the selected channel while preserving the gateway process, other channels, and active sessions.

## Best Possible Solution

Add a narrow, admin-scoped channel reload surface that reuses the existing channel manager stop/start lifecycle for one channel or channel account, documents it under the existing plural `openclaw channels` namespace, and leaves watcher debounce and native plugin hot reload to their separate issues.

## Reproduction Assessment

Yes. Static source and docs inspection is enough here: the CLI and gateway method registries have no reload command/RPC, and the existing start path is a no-op for already-running channel tasks rather than a forced stop/start reload.

## Solution Assessment

Yes. The narrowest maintainable fix is an additive `channels.reload` gateway method plus `openclaw channels reload --channel <provider> [--account <id>]`, guarded as admin-only and implemented by stopping then starting only the requested channel/account.

## Repair Work Prompt

Implement https://github.com/openclaw/openclaw/issues/72997 as a narrow operator channel reload feature. Add an admin-scoped `channels.reload` gateway RPC and an `openclaw channels reload --channel <provider> [--account <id>]` CLI command in the existing plural channels namespace. The RPC should validate/normalize the channel like `channels.start`, require a plugin runtime start capability, call the existing channel manager `stopChannel` then `startChannel` for only the requested channel/account, and return a small payload with channel, account scope, and reloaded/started status. When `--account` is omitted, prefer provider-level behavior that reloads all configured accounts for that channel, matching the issue's provider-level request; document this clearly. Do not change config watcher debounce, native plugin code hot reload, full gateway restart policy, or unrelated channel plugin behavior. Add focused unit tests for the gateway handler, CLI registration/command policy, and channel manager interaction, update `docs/cli/channels.md`, and add a user-facing `CHANGELOG.md` entry crediting `Thanks @axonrelaybot` unless maintainers choose a different reporter credit.
