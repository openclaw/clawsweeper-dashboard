## Summary

Keep open. Current main still has automatic/config-triggered channel restart paths and admin channel start/logout methods, but it does not expose the requested manual single-channel reload command or Gateway RPC; the related watcher issue #74122 tracks debounce/scoped automatic reload behavior, not this operator action.

## What This Changes

The issue requests a targeted channel reload command or Gateway action that reconnects one named channel without restarting the Gateway or interrupting other channels/sessions.

## Best Possible Solution

Add an additive, admin-scoped `channels.reload` RPC plus `openclaw channels reload --channel <provider> [--account <id>]` wrapper that validates the channel/account, reuses existing generic stop/start lifecycle seams for only that target, returns runtime status, and updates protocol/CLI docs and changelog without adding Telegram/WhatsApp-specific core policy.

## Reproduction Assessment

Yes. A source and docs check against current main reproduces the gap: the CLI, Gateway method list, method scopes, protocol schema, and docs expose status/start/logout/secrets.reload but no manual channel reload surface.

## Solution Assessment

Yes. The narrow maintainable solution is a generic admin-gated lifecycle method that reuses the current channel manager stop/start behavior; using a full Gateway restart or adding provider-specific reconnect logic would be broader than the reported operator need.
