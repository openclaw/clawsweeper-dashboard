## Summary

Keep open. Current main still lacks the requested one-step per-channel reload CLI/RPC surface; it has low-level channel start/stop RPCs and config/secrets hot-reload restart paths, but no operator command or Gateway method that reloads exactly one channel in one action.

## What This Changes

Add a targeted command or Gateway RPC to reload one named channel without restarting the Gateway or interrupting other channels.

## Best Possible Solution

Add an additive, admin-scoped `channels.reload` Gateway RPC and `openclaw channels reload --channel <name> [--account <id>]` wrapper that reuse the existing stop/start lifecycle and document the operator behavior.

## Reproduction Assessment

Yes. A current-main source and docs check reproduces the gap: the CLI, method registry, admin scopes, protocol schema, tests, and docs expose status/start/stop/logout paths but no single manual reload surface.

## Solution Assessment

Yes. A server-side reload RPC layered over the existing channel manager lifecycle is the narrow maintainable fix; a full Gateway restart is too broad, and client-side stop/start is less atomic.
