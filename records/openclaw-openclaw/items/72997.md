## Summary

Keep open. Current main still has automatic channel restart seams for config/secrets reloads, but it does not expose the requested manual single-channel reload command or Gateway RPC, and the related watcher report tracks a different automatic debounce/scoping problem.

## What This Changes

Request a CLI and/or Gateway operator surface to reload one named channel provider without restarting the full Gateway process or interrupting other channels.

## Best Possible Solution

Add an additive, admin-scoped `channels.reload` Gateway RPC plus a matching `openclaw channels reload --channel <name> [--account <id>]` CLI wrapper. It should validate the target through existing channel plugin config helpers, stop and restart only that channel/account via the generic lifecycle manager, return the resulting runtime status, and update protocol docs, CLI docs, tests, and changelog without adding provider-specific core behavior.

## Reproduction Assessment

Yes. A read-only source/docs check against current main reproduces the feature gap: there is no CLI subcommand, Gateway method, protocol schema, scope entry, or documentation for manual channel reload, and `channels.start` is not a forced reconnect for an already-running account.

## Solution Assessment

Yes. The narrow maintainable solution is a generic additive admin RPC that reuses existing stop/start lifecycle seams; provider-specific recovery policy or a full Gateway restart would be broader than the reported operator need.
