## Summary

Keep open. Current main still lacks the requested one-step manual channel reload CLI command or Gateway RPC; existing automatic config/secrets-driven channel restarts and the adjacent watcher debounce issue do not satisfy this operator surface.

## What This Changes

Request to add a targeted channel reload CLI/Gateway RPC that restarts one named channel without restarting the Gateway or disturbing other channels or sessions.

## Best Possible Solution

Expose an additive admin-scoped `channels.reload` RPC plus `openclaw channels reload --channel <name> [--account <id>]`, reusing the existing channel lifecycle and documenting the behavior.

## Reproduction Assessment

Yes. Source and docs inspection against current main reproduces the gap: CLI, method registry, admin scopes, protocol schema, handlers, tests, and docs expose status/start/stop/logout but no one-step manual reload surface.

## Solution Assessment

Yes. A generic reload method layered over the existing stop/start lifecycle is the narrow maintainable solution; provider-specific recovery policy or a full Gateway restart would be broader than the request.

## Repair Work Prompt

Implement the feature requested by this issue: add a targeted channel reload operator surface without restarting the Gateway. Add an additive admin-scoped Gateway RPC, preferably `channels.reload`, that accepts `channel` and optional `accountId`, validates like the existing channel lifecycle methods, and reloads only the selected channel/account by reusing the existing stop/start lifecycle and active-work deferral semantics where appropriate. Add a CLI wrapper using the existing plural command style, e.g. `openclaw channels reload --channel <name> [--account <id>]`, and document it in the channels CLI and Gateway protocol docs. Add focused tests for the RPC handler, method registry/scope/protocol exports, CLI parsing/Gateway call behavior, and docs/changelog coverage.
