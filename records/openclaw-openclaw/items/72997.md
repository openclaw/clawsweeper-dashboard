## Summary

Keep open. Current main has automatic per-channel restart seams for config/secrets reloads, but it still does not expose the requested manual single-channel reload command or Gateway RPC, and the related watcher issue covers a different automatic debounce/scoping problem.

## What This Changes

Request an operator command or Gateway RPC to reload one named channel plugin without restarting the Gateway or interrupting other channels and sessions.

## Best Possible Solution

Add an additive, admin-scoped `channels.reload` Gateway RPC plus a matching `openclaw channels reload` CLI wrapper that validates the channel/account through the existing plugin config helpers, stops and restarts only that target via the generic ChannelManager lifecycle, returns the resulting runtime status, and updates protocol docs, CLI docs, tests, and changelog without adding provider-specific core behavior.

## Reproduction Assessment

Yes. A source/docs check against current main reproduces the gap: the CLI, Gateway method registry, protocol schema, method scopes, tests, and docs expose status/start/logout but no manual reload surface, and `channels.start` is not a forced reconnect for an already-running channel.

## Solution Assessment

Yes. The narrow maintainable direction is an additive admin-gated generic `channels.reload` method that reuses existing stop/start lifecycle seams, preferably following the current plural `channels` namespace rather than introducing Telegram/WhatsApp-specific policy or requiring a full Gateway restart.
