## Summary

Keep open. Current main still has per-channel lifecycle primitives and config hot-reload restart logic, but it does not expose the requested one-step channel reload CLI command or Gateway RPC.

## What This Changes

Request a targeted operator command/RPC to reload one channel provider or account without restarting the Gateway or interrupting other channels and sessions.

## Best Possible Solution

Add an additive, admin-scoped `channels.reload` Gateway RPC plus an `openclaw channels reload --channel <name> [--account <id>]` CLI wrapper that reuses the existing stop/start lifecycle and documents the operator behavior.

## Reproduction Assessment

Yes. A current-main source and docs check reproduces the gap: the CLI, method registry, admin scopes, protocol schema, tests, and docs expose status/start/stop/logout paths but no single manual reload surface.

## Solution Assessment

Yes. A generic reload method layered over the existing channel manager lifecycle is the narrow maintainable fix; provider-specific recovery policy or a full Gateway restart would be broader than this request.

## Repair Work Prompt

Implement issue #72997 as a focused OpenClaw operator feature. Add an additive admin-scoped `channels.reload` Gateway RPC that validates `{ channel, accountId? }`, resolves the configured account like `channels.start`/`channels.stop`, stops then starts only that selected channel/account via the existing channel manager lifecycle, and returns a small typed result. Add an `openclaw channels reload --channel <name> [--account <id>] [--json]` CLI wrapper and docs. Keep this generic in core; do not add Telegram/WhatsApp-specific recovery policy, do not bundle watcher debounce/logging work from #74122, and do not change plugin/channel ownership boundaries. Add a user-facing `CHANGELOG.md` entry unless maintainers decide this is internal-only.
