## Summary

Keep open. Current main supports config-driven per-channel hot reloads, but it does not expose the requested manual channel reload command or gateway/tool action for operator-triggered stop-and-start recovery of a named channel.

## What This Changes

Request to add a targeted channel/provider reload command or gateway action that reconnects one channel without restarting the whole gateway.

## Best Possible Solution

Add an additive, admin-scoped `channels.reload` gateway RPC plus `openclaw channels reload --channel <id> [--account <id>] [--json]`, implemented by reusing the existing channel stop/start lifecycle rather than restarting the gateway. If the gateway tool surface is included, keep it owner/admin gated and document the operational behavior.

## Reproduction Assessment

Yes for the missing-feature symptom. Current-main source inspection shows no reload subcommand, no `channels.reload` RPC, and no gateway-tool `channel.reload` action, while the issue’s requested operation is concrete.

## Solution Assessment

Yes, the narrow maintainable direction is an additive RPC/CLI wrapper around the existing `stopChannel`/`startChannel` lifecycle. Closing as implemented would be incorrect because current config hot reload and `channels.start` do not provide a manual per-channel reload surface.

Implement a targeted channel reload surface for https://github.com/openclaw/openclaw/issues/72997. Add an additive admin-scoped gateway RPC, preferably `channels.reload`, that accepts `channel` and optional `accountId`, validates/normalizes the channel like `channels.start`, stops that channel/account, then starts it again using the existing `GatewayRequestContext.stopChannel` and `startChannel` lifecycle. Add `openclaw channels reload --channel <id> [--account <id>] [--json]` using `callGatewayFromCli`; keep the repo’s plural `channels` namespace unless maintainers explicitly ask for a singular alias.
