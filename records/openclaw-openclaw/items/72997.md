## Summary

Keep this open. Current main has several adjacent recovery paths, including channel-scoped config hot reloads, `secrets.reload`, automatic health-monitor restarts, and `channels.start`, but it still does not expose the requested manual per-channel reload command or `channel(s).reload` gateway method. The prior related cleanup in #71613 covered automatic config reload behavior, not this remaining operator-triggered control surface.

## What This Changes

Request a targeted command or gateway RPC that disconnects and reconnects only a named channel plugin or account without restarting the gateway.

## Best Possible Solution

Add a small admin-gated channel reload surface that reuses existing channel/account resolution and channel-manager stop/start primitives, documents it next to the existing channels CLI and gateway protocol methods, and leaves automatic hot reload, health monitoring, and watcher debounce behavior unchanged.

## Reproduction Assessment

Yes. A source-level reproduction is enough here: the current command and gateway method registries do not contain any channel reload/restart entry, while adjacent start/logout and automatic reload paths are present.

## Solution Assessment

Yes for the remaining gap, with naming aligned to the existing plural `channels` CLI/RPC namespace. Automatic config hot reload and the health monitor are useful but do not give operators the explicit per-channel reload trigger requested here.

## Repair Work Prompt

Implement the remaining manual channel reload control for https://github.com/openclaw/openclaw/issues/72997. Add a narrow admin-scoped gateway RPC, preferably `channels.reload` to match existing plural channel methods, that accepts `channel` and optional `accountId`, resolves the default account consistently with `channels.start`/`channels.logout`, stops then starts only that channel/account, and returns a small payload such as `{ channel, accountId, reloaded, running }`. Add `openclaw channels reload --channel <provider> [--account <id>]`; keep naming consistent with existing `openclaw channels` rather than creating a new singular top-level command. Update the advertised gateway method list, operator scopes, protocol schema/types, docs, and focused tests.
