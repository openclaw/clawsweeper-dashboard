## Summary

Keep open. Current main still lacks the requested one-step manual channel reload CLI command or Gateway RPC; the existing automatic config/secrets-driven channel restarts and the earlier closed config-hot-reload report do not provide this operator surface.

## What This Changes

Request a command or Gateway action that reloads one named channel provider without restarting the Gateway process or interrupting other channels.

## Best Possible Solution

Expose an additive admin-scoped `channels.reload` RPC plus `openclaw channels reload --channel <name> [--account <id>]`, reusing the existing channel lifecycle and documenting the behavior.

## Reproduction Assessment

Yes. Source and docs inspection against current main reproduces the gap: the CLI, method registry, admin scopes, protocol schema, handlers, and docs expose status/start/stop/logout but no one-step manual reload surface.

## Solution Assessment

Yes. A generic reload method layered over the existing stop/start lifecycle is the narrow maintainable direction; provider-specific recovery policy or a full Gateway restart would be broader than the request.
