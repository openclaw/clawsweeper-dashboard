## Summary

Keep this issue open. Current main has channel lifecycle primitives and config-triggered per-channel hot reload, but it still does not expose the requested manual single-channel reload command or Gateway RPC. The related watcher issue #74122 covers debounce/scoped automatic reload behavior, not this operator command.

## What This Changes

Request a targeted channel reload command or Gateway action that reconnects one named channel/account without restarting the Gateway or interrupting other active channels and sessions.

## Best Possible Solution

Add an additive, admin-scoped `channels.reload` RPC plus CLI wrapper that validates the channel/account, reuses the existing channel manager stop/start lifecycle for only that target, returns the resulting runtime status, and updates protocol/CLI docs and changelog without adding provider-specific core policy.

## Reproduction Assessment

Yes. A source/docs check against current main reproduces the gap: the CLI, Gateway method registry, protocol schema, and docs have status/start/logout but no reload surface, and `channels.start` is not a forced reconnect for an already-running channel.

## Solution Assessment

Yes. The narrow maintainable path is to reuse the existing generic channel lifecycle seams behind an additive admin-gated method, preferably using the current plural `channels` naming, instead of restarting the Gateway or adding Telegram/WhatsApp-specific behavior.
