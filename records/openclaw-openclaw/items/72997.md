## Summary

Keep open. Current main still lacks the requested one-step manual channel reload CLI command or Gateway RPC; the existing config-driven hot reload and health-monitor restart paths are automatic lifecycle behavior, not an operator surface.

## What This Changes

Request an operator-facing command or Gateway RPC to disconnect and reconnect one named channel without restarting the Gateway.

## Best Possible Solution

Add an admin-scoped `channels.reload` RPC plus `openclaw channels reload --channel <name> [--account <id>]`, implemented by the existing stop/start lifecycle with docs and regression coverage.

## Reproduction Assessment

Yes. Source and docs inspection against current main reproduces the gap: status/start/stop/logout exist in CLI, Gateway registry, scopes, protocol schema, handlers, and docs, but there is no one-step manual reload surface.

## Solution Assessment

Yes. A generic reload method layered over the existing channel stop/start lifecycle is the narrow maintainable direction; provider-specific recovery policy or full Gateway restart would be broader than the request.
