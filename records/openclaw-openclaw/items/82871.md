## Summary

Keep open: current main still has Copilot request paths that do not explicitly opt out of compressed responses or decode compressed JSON before parsing, which matches the reported gzip-magic JSON parse failure.

## What This Changes

The issue requests fixing github-copilot model calls that fail after auth when Copilot returns gzip-encoded JSON responses.

## Best Possible Solution

Add a provider-owned Copilot HTTP encoding policy, likely `Accept-Encoding: identity` in the shared Copilot headers or equivalent decoding, and lock it with focused regression tests across token exchange, catalog, embeddings, and stream wrappers.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: authenticate, target a `github-copilot/*` model in an environment where Copilot returns gzip JSON, and the current direct `json()` parsing can hit the same gzip magic failure. I did not run the live provider call because no Copilot test credential was available in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to make the Copilot provider own its response encoding contract rather than adding a global transport workaround. The safest patch should update shared Copilot headers or decoding once and add endpoint-specific tests.
