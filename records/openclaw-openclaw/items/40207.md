## Summary

Keep open. Current main still generates macOS LaunchAgent stdout/stderr paths from the OpenClaw state directory, so a symlinked or external `~/.openclaw` can still feed those paths into launchd; the related launchd hang report is distinct and does not supersede this EX_CONFIG log-path crash.

## What This Changes

The issue requests that macOS Gateway LaunchAgent stdout/stderr log paths avoid symlinked or external OpenClaw state-directory paths so launchd can start the Gateway reliably.

## Best Possible Solution

OpenClaw should generate macOS LaunchAgent `StandardOutPath` and `StandardErrorPath` values from a launchd-safe user log directory, update the status/diagnostic log readers and hints to match, and leave the distinct launchd module-evaluation hang tracked separately in #46153.

## Reproduction Assessment

Yes. The issue gives concrete macOS reproduction steps and two independent confirmations, and current source inspection shows the generated plist still routes stdout/stderr through the state-dir log resolver.

## Solution Assessment

Yes. A focused macOS LaunchAgent log-path change is the narrowest maintainable fix; broad Gateway runtime changes or a general state-dir migration are not required for this report.
