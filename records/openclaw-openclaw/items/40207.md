## Summary

Keep open. Current main still derives macOS LaunchAgent stdout/stderr paths from the OpenClaw state directory and rewrites restart plists with those values, so the confirmed symlinked/external-volume launchd EX_CONFIG failure remains actionable.

## What This Changes

The issue asks OpenClaw to stop generating launchd stdout/stderr paths under symlinked or external-volume `~/.openclaw/logs` so the Gateway starts reliably under macOS LaunchAgent.

## Best Possible Solution

Generate LaunchAgent stdout/stderr paths from a launchd-safe user log directory outside the OpenClaw state directory, while keeping diagnostics and user-facing log guidance consistent.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40207. Current main generates macOS LaunchAgent `StandardOutPath`/`StandardErrorPath` from `resolveGatewayLogPaths`, which uses `OPENCLAW_STATE_DIR` / `~/.openclaw*/logs`; launchd can reject those paths with EX_CONFIG when `.openclaw` is symlinked or on an external volume. Change the macOS LaunchAgent stdout/stderr destination to a launchd-safe user log directory outside the OpenClaw state directory, keep profile/log-prefix behavior intentional, and update install, stage, restart-rewrite, runtime hints, status/diagnosis log readers, tests, docs, and changelog consistently. Do not combine this with the separate launchd stdin hang in https://github.com/openclaw/openclaw/issues/46153, the external-home plist bootstrap failure in https://github.com/openclaw/openclaw/issues/60398, or generic log-retention work such as https://github.com/openclaw/openclaw/issues/41676 / https://github.com/openclaw/openclaw/pull/79892. Add regression coverage showing generated launchd stdout/stderr no longer live under a symlinkable state dir while restart logs and diagnostics still point to the documented locations. Validate with focused daemon/status tests and a macOS launchd smoke when available; include a user-facing changelog entry.
