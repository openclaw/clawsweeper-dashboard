## Summary

Current main still writes macOS LaunchAgent stdout/stderr under the OpenClaw state directory, so this reported launchd `EX_CONFIG` failure is still plausible and not superseded by the related open launchd reports.

## What This Changes

The issue asks OpenClaw to stop generating LaunchAgent `StandardOutPath` and `StandardErrorPath` values that point through symlinked or external `~/.openclaw/logs` paths.

## Best Possible Solution

Update macOS LaunchAgent plist generation to use a launchd-safe local log destination, then align status hints, tests, docs, and changelog while keeping the broader launchd hang and external-home bootstrap reports separate.

## Repair Work Prompt

Fix the macOS LaunchAgent log-path failure tracked by this issue. Current main writes LaunchAgent `StandardOutPath`/`StandardErrorPath` through `resolveGatewayLogPaths()`, which defaults to `HOME/.openclaw/logs`; when `.openclaw` is symlinked or external, launchd can exit with `EX_CONFIG` before the gateway starts. Implement the narrowest maintainable fix so generated macOS LaunchAgent plists use a launchd-safe local user log location, preferably consistent with existing macOS logging conventions, or otherwise validate/fallback before bootstrap. Keep systemd and Windows service log behavior unchanged, preserve `OPENCLAW_LOG_PREFIX`/profile behavior where it still makes sense, and do not move service env files or secrets. Update `src/daemon/launchd.ts`, `src/daemon/restart-logs.ts` or a LaunchAgent-specific resolver as appropriate, adjust status/runtime log hints, add regression tests for generated plist stdout/stderr paths with a default `HOME`, and update docs/changelog for the user-facing behavior. Treat #46153 and #60398 as related but distinct; do not broaden this PR into launchd module-hang or external-home plist bootstrap policy.
