## Summary

Keep open. Current main still generates macOS LaunchAgent stdout/stderr paths under the OpenClaw state directory and can rewrite manual plist workarounds during restart repair, so the confirmed symlinked/external-volume launchd EX_CONFIG failure remains source-reproducible.

## What This Changes

The issue asks OpenClaw to stop generating LaunchAgent log paths under symlinked or external-volume `~/.openclaw/logs` because launchd rejects those stdout/stderr paths with exit 78.

## Best Possible Solution

Generate LaunchAgent stdout/stderr paths from a launchd-safe user log directory outside the OpenClaw state directory, while keeping restart logs, diagnostics, status hints, tests, and docs consistent.

## Repair Work Prompt

Fix the macOS Gateway LaunchAgent stdout/stderr log-path bug reported in https://github.com/openclaw/openclaw/issues/40207. Current main still derives LaunchAgent `StandardOutPath` and `StandardErrorPath` from `resolveGatewayStateDir(env)/logs`, which fails under launchd when `.openclaw` or `OPENCLAW_STATE_DIR` is symlinked or on an external volume. Move the LaunchAgent stdout/stderr paths to a launchd-safe user log directory outside the OpenClaw state directory, keep `OPENCLAW_LOG_PREFIX` behavior where appropriate, and update install/stage/restart-rewrite, diagnostics/status/runtime hints, focused tests, user-visible docs, and CHANGELOG.md. Do not mix in the distinct launchd startup hang from https://github.com/openclaw/openclaw/issues/46153 or the external-home plist bootstrap error from https://github.com/openclaw/openclaw/issues/60398. Add regression coverage for the generated log paths and restart-rewrite behavior, then validate with the focused daemon/status tests plus a macOS launchd smoke when available.
