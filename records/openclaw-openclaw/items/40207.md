## Summary

Keep open. Current main still generates macOS Gateway LaunchAgent stdout/stderr paths from the OpenClaw state directory, passes those paths directly into the plist, and rewrites the same values during restart repair; the linked launchd hang and external-home bootstrap reports are related but distinct remaining issues.

## What This Changes

The issue asks OpenClaw to stop generating launchd `StandardOutPath`/`StandardErrorPath` values under symlinked or external `~/.openclaw/logs` paths so the Gateway LaunchAgent can start reliably on macOS.

## Best Possible Solution

Generate macOS LaunchAgent stdout/stderr paths from a launchd-safe user log directory, update the status/diagnostic readers and hints to match, and leave #46153 and #60398 to track their distinct launchd failure modes.

## Reproduction Assessment

Yes. The issue has concrete macOS reproduction steps plus an independent confirmation, and current source inspection shows the generated plist still routes stdout/stderr through the state-dir log resolver.

## Solution Assessment

Yes. A focused macOS LaunchAgent log-path change is the narrowest maintainable fix; a broad Gateway runtime rewrite or general state-dir migration is not needed for this report.
