## Summary

Keep open. Current main has solid run-cancellation primitives for `/stop`, `chat.abort`, and `sessions.abort`, but I found no inbound channel event path that maps platform message delete/recall/edit events to those primitives. The related Feishu work was a channel message-management tool proposal, not this cross-channel recall-to-abort behavior, and it was routed for maintainer review rather than shipped.

## What This Changes

The issue requests platform message recall/delete events to cancel running channel agent tasks, with edited messages optionally updating context or canceling and rerunning.

## Best Possible Solution

Add a generic, typed inbound channel control event for message deleted/recalled/edited with original message id, sender, conversation, account, and session correlation, then map authorized recall/delete events to the existing `chat.abort`/`sessions.abort` path. Implement one or two channel slices behind that contract first, document edit semantics, and keep destructive or cross-user cancellation policy explicit.

## Reproduction Assessment

Yes, at source level. Current main can reproduce the gap by tracing an active channel run to `chat.abort`/`sessions.abort` and then confirming Telegram, Feishu, and Discord inbound handlers do not register delete/recall/edit events that call that abort path.

## Solution Assessment

Unclear until maintainers choose the API and product policy. The narrowest maintainable direction is not a one-off per-channel workaround, but a shared channel-event contract plus per-channel adapters that reuse the existing abort primitives.
