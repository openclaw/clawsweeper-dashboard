## Summary

Keep open: current main has run-abort primitives and some channel-specific edit/delete observation, but it still does not connect message recall/delete or edit events to cancelling or re-running active session work. The related Discord trackers cover scoped pieces, not the broader cross-channel product/API question.

## What This Changes

The issue requests messaging-platform recall/delete events to cancel active tasks and edit events to update or re-run the task context.

## Best Possible Solution

Add an authorized generic channel-event contract for recall/delete/edit events, then let capable channel plugins map those events to precise session/run cancellation or edit reprocessing with focused per-channel tests.

## Reproduction Assessment

Not applicable. This is a new cross-channel feature request, and source inspection shows existing abort paths plus isolated event observation but no current recall/edit-to-abort behavior to reproduce as a broken contract.

## Solution Assessment

Unclear as an implementation plan. The requested behavior is useful, but the maintainable solution should define authorization, source-message mapping, and per-channel capability semantics instead of adding one-off cancellation hooks.

## Risks / Open Questions

- Platform support differs across Feishu, Telegram, Discord, WhatsApp, Signal, Slack, and other plugins, so a generic promise likely needs per-channel capability reporting and policy defaults.
- The related Discord issues split edit reprocessing from delete-to-cancel, so maintainers should decide whether to keep them separate or define one shared channel-event contract first.
