## Summary

Keep open. Current main still has run/session-scoped cancellation and some platform-specific edit handling, but it does not provide the requested cross-channel source-message recall/edit-to-cancel contract; the open Discord PR covers only one channel and is still blocked, while the Feishu PR explicitly complements rather than replaces inbound recall-event work.

## What This Changes

The issue requests support for channel-native message recall/delete/edit events that cancel or correct the active agent run triggered by the original source message.

## Best Possible Solution

Keep this as the broad design tracker, review the Discord PR as a channel-specific candidate, and define a generic source-message lifecycle seam only where plugin-local handlers cannot safely map native edit/delete/recall events to exact run cancellation or reprocessing.

## Reproduction Assessment

Yes, source-level. Current main has source message ids in inbound context and active run cancellation by run/session, but the inspected contracts do not expose source-message edit/delete/recall events or a native message-id to active-run cancellation map.

## Solution Assessment

No current implementation fully solves the issue. The maintainable solution is a shared contract where needed plus channel-specific handlers and tests, not closing the umbrella because one Discord PR or adjacent Feishu message-management PR exists.
