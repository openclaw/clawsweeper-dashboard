## Summary

Keep open: current main still chooses the `openclaw agent` session key before using Discord delivery context, so a live channel/thread session can be bypassed and the reported source-level bug remains.

## What This Changes

The issue requests source-level CLI routing for `openclaw agent --channel discord --to ...` to reuse the live Discord channel/thread session instead of falling back to `agent:*:main`.

## Best Possible Solution

Add generic delivery-context session lookup in the source resolver for channel-targeted agent CLI calls, using channel target parsing for Discord normalization while preserving explicit `--session-key` and `--session-id` precedence.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: with a stored `agent:main:discord:channel:<id>` session, `openclaw agent --channel discord --to <id>` resolves from raw `to` before channel normalization and never scans stored delivery context.

## Solution Assessment

Yes. The best fix is a narrow source-level resolver and CLI wiring change, not compiled `dist` patching; #42009 is adjacent but does not solve delivery-context lookup for existing live Discord sessions.
