## Summary

Keep open. Current main still selects the `openclaw agent` session before Discord delivery context is normalized or compared with stored session metadata, so the reported channel/thread routing gap remains. The open #42009 PR is adjacent `--agent --to` work, but it does not cover delivery-context lookup for an existing live Discord channel/thread session.

## What This Changes

The issue asks for `openclaw agent --channel discord --to ...` to resolve an existing live Discord channel/thread session by delivery context instead of falling back to an agent main session.

## Best Possible Solution

Add source-level, generic delivery-context session lookup for `openclaw agent` when a channel target is provided, with Discord target normalization through the channel parsing contract, explicit selector precedence preserved, and docs updated to describe the supported routing behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with an existing `agent:main:discord:channel:<id>` session and a CLI call using `--channel discord --to <id>`, current main resolves from `opts.to` alone and never scans the stored delivery context before falling back to the main session key.

## Solution Assessment

Yes, the narrow maintainable path is to extend the existing session resolver and CLI wiring rather than patch compiled `dist` bundles. #42009 may help adjacent `--agent --to` precedence, but it is not sufficient for this delivery-context bug.
