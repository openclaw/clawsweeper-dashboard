## Summary

Close as superseded: the same reporter's broader root report was already reviewed and closed with the maintainer-facing conclusion that `--channel` is delivery-only, while current docs and source still define session routing through `--to`, `--session-id`, `--session-key`, or agent selection rather than Discord delivery context.

## What This Changes

The issue asks for `openclaw agent --channel discord --to` to resolve an existing Discord channel/thread session by delivery context instead of deriving the main agent session.

## Best Possible Solution

Keep the documented delivery/session split; if maintainers want delivery-context session selection, track it as a separate CLI product proposal with explicit contract, docs, and regression tests.

## Reproduction Assessment

Yes for the observed routing shape: current source selects the session from raw `--to` before channel/account delivery metadata is applied. That is not a high-confidence bug reproduction because current docs define `--channel` as delivery-only.

## Solution Assessment

No as a bug fix. The maintainable path would be a separate product decision for a new delivery-context session selector or explicit session-key workflow, not a silent change to `--channel` semantics.
