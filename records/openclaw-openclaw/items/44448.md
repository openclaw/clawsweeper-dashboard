## Summary

Keep open: current main still selects the agent session from raw `--to` before Discord channel/account delivery context is normalized, so existing live Discord channel/thread sessions can still be bypassed. The related open items are broader or adjacent rather than a merged replacement for this narrow routing bug.

## What This Changes

Route `openclaw agent --channel discord --to ...` by the normalized Discord delivery context and existing live session instead of falling back to the agent main session.

## Best Possible Solution

Add a source-level delivery-context lookup for channel-targeted agent CLI calls that normalizes the target through the channel parser, reuses matching stored session entries, and preserves explicit session selectors and default-agent compatibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with an existing `agent:main:discord:channel:<id>` session, `openclaw agent --channel discord --to <id>` computes the session key from raw `to` before Discord target parsing or stored delivery-context lookup can run.

## Solution Assessment

Yes. The maintainable fix is a narrow source change in the CLI/session resolver and tests; patching compiled `dist` bundles or relying on the adjacent explicit-agent PR would leave the live Discord delivery-context case unresolved.
