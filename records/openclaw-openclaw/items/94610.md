## Summary

Keep this PR open: the updated branch resolves the prior token-validation and SDK follow-up save findings, but the new lock wait budget can still fail concurrent callers before the in-flight refresh request is allowed to finish.

## What This Changes

The PR serializes MCP OAuth refresh-token requests, persists validated rotated tokens through a locked store path, and wires the wrapped fetch into streamable HTTP and SSE MCP transports with focused tests.

## Best Possible Solution

Keep the provider/transport serialization approach, but make the lock wait budget outlast the refresh request deadline and add a slow-concurrent-follower regression before merge.

## Reproduction Assessment

Yes, by source inspection rather than a live third-party OAuth provider. Current main sends SDK MCP OAuth refresh-token POSTs through unwrapped transport fetches and plain store writes, while the SDK refresh flow validates the response and then calls provider.saveTokens(newTokens).

## Solution Assessment

No, not quite yet. The provider/transport wrapper is the right layer and the prior validation/save findings are addressed, but the lock timeout must be longer than the refresh request deadline so concurrent callers can actually wait for the serialized refresh result.

## Repair Work Prompt

Fix the remaining MCP OAuth refresh serialization timeout bug in https://github.com/openclaw/openclaw/pull/94610. The PR currently sets REFRESH_LOCK_TIMEOUT_MS to 60_000 while REFRESH_REQUEST_TIMEOUT_MS is 90_000, so a concurrent caller can time out waiting for the lock before the active refresh request is allowed to finish. Adjust the lock wait budget so followers can outwait an in-flight refresh request, keep the existing validated token persistence and merge-aware saveTokens behavior, and add a focused regression in src/agents/mcp-oauth.test.ts where a second concurrent refresh waits longer than the old 60s limit and receives the first refresh's rotated tokens without a second token endpoint call. Do not change config surfaces, dependency versions, redirect fallback behavior, or unrelated MCP transport behavior. No CHANGELOG.md edit.
