Land one validated, serialized MCP OAuth refresh persistence path shared by the wrapper and the SDK token-save path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate refresh tokens before persisting the clone:** `src/agents/mcp-oauth.ts:368`
  - body: The wrapper writes tokens parsed from `response.clone().json()` before the MCP SDK validates the original response with `OAuthTokensSchema`. A 200 response with malformed token JSON that current main would reject without saving can now be persisted into the MCP OAuth store, so validate the clone with the SDK schema before `writeStore`.
  - confidence: 0.91
- **[P2] Keep SDK saveTokens under the refresh write policy:** `src/agents/mcp-oauth.ts:369`
  - body: After the wrapper returns the original response, the MCP SDK still calls `provider.saveTokens(newTokens)` outside this lock. That second plain read/write can stale-overwrite MCP OAuth metadata saved after the wrapper write, so make the follow-up save share the serialized merge path or skip it when the wrapper already persisted the validated tokens.
  - confidence: 0.86
