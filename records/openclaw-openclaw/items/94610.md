## Summary

Keep open: the updated branch fixes the earlier stale-lock and pre-request store-write concerns, but the SDK still performs a second `saveTokens` write after the wrapper releases its lock, leaving a narrow stale-overwrite race for MCP OAuth metadata.

## What This Changes

The PR adds serialized MCP OAuth refresh-token handling, timeout-bounded refresh fetches, atomic store writes, streamable HTTP/SSE wiring, and focused tests.

## Best Possible Solution

Route refresh-token persistence through one serialized, merge-aware path so the SDK follow-up `saveTokens` call cannot stale-overwrite OAuth metadata, then land with focused MCP OAuth validation.

## Reproduction Assessment

Yes, by source inspection. Current main sends SDK MCP OAuth refresh-token POSTs through unwrapped transport fetches and persists MCP OAuth credentials through plain JSON read/modify/write helpers, so concurrent rotating-token refreshes can replay stale tokens.

## Solution Assessment

No, not yet. The provider/transport wrapper is the right layer, but the implementation should also cover the SDK's later `saveTokens` write before this is the narrow maintainable fix.
