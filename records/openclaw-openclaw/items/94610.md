## Summary

Keep open: the MCP OAuth refresh serialization fix is still needed, but the current PR head is not merge-ready because refreshed tokens are written over a store snapshot captured before the network request, which can drop concurrent OAuth metadata writes.

## What This Changes

The PR serializes MCP OAuth refresh-token fetches, writes refresh responses atomically, and routes streamable HTTP/SSE transports through the wrapped fetch.

## Best Possible Solution

Land the narrow serialization fix after the refreshed-token write re-reads and merges the latest store state, with focused regression coverage for concurrent non-token OAuth metadata writes.

## Reproduction Assessment

Yes by source inspection. Current main sends SDK refresh-token POSTs through an unwrapped fetch and stores OAuth credentials through direct JSON read/modify/write, so concurrent rotating-token refreshes can replay a stale refresh token.

## Solution Assessment

No, not as written. Wrapping at the provider/transport boundary is the right fix shape, but the refresh persistence must merge into the latest store state before the patch is safe to merge.
