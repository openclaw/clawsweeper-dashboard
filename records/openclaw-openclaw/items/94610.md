## Summary

Keep open: current main and v2026.6.8 still refresh MCP OAuth tokens without a per-store serialization wrapper, while this PR is a narrow replacement for the earlier broad closed attempt and I found no blocking correctness issue in the patch. The remaining maintainer-visible concern is auth-provider proof depth: the PR has good local HTTP smoke proof, but no live third-party rotating-token provider run.

## What This Changes

The PR adds per-store serialization around MCP OAuth refresh-token fetches, atomic token-store writes, and wires the wrapper through streamable HTTP and SSE transports with regression tests.

## Best Possible Solution

Land the narrow MCP OAuth refresh serialization after required checks and maintainer acceptance of the local proof or a live-provider follow-up.

## Reproduction Assessment

Yes, by source inspection: current main sends SDK refresh_token POSTs through an unwrapped fetch and saves the same JSON credential store without a lock, so concurrent refreshes can replay the same rotating token. I did not run a failing current-main repro because this review is read-only.

## Solution Assessment

Yes, this is the best fix shape I found: wrapping the SDK fetch at the OpenClaw OAuth provider/transport boundary covers CLI login, streamable HTTP, and SSE without patching the dependency or duplicating transport-specific logic.
