## Summary

Keep open. Current main still only partially overlaps with the PR: it has internal agent-event streaming and host-owned plugin event subscription seams, but it does not expose the PR's public `api.onAgentEvent` shape, typed `thinking_start`/`thinking_end` plugin hooks, or `agent_end` token/tool metadata. The PR remains relevant, but the proposed branch is not merge-ready because the thinking lifecycle path still has concrete edge-case bugs and the public event surface needs maintainer review.

## What This Changes

The PR adds plugin-facing thinking lifecycle hooks, a session-filtered agent event subscription API, thinking stream delivery, and `agent_end` token/tool-count metadata for richer channel/plugin UIs.

## Best Possible Solution

Keep the PR available for maintainer review, but require a rebase onto current main and a clear decision on whether public plugins should use the existing `registerAgentEventSubscription`/runtime-events model or a new `api.onAgentEvent` method. Any landed version should fix thinking lifecycle dedupe, use code-span-aware tagged reasoning extraction, preserve session scoping, update SDK docs/baselines, and add focused tests for native, tagged, mixed, duplicate, orphaned, and message-end reasoning flows.

## Reproduction Assessment

Yes for the PR findings: the diff supports focused subscribe-harness tests that send duplicate native `thinking_end` events after a normal start/end, and tests that close an open native thinking lifecycle at `message_end` with a literal `<think>...</think>` code example in the final answer. I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No, the PR is not the best mergeable solution as-is. The direction is plausible, but the safest solution is to align the public contract with current host-hook architecture and fix the edge-case thinking lifecycle behavior before exposing it as SDK surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Deduplicate repeated thinking_end events:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:222-241`
  - body: This branch treats every native `thinking_end` received while `reasoningStreamOpen` is false as an orphan and synthesizes a new `thinking_start` before emitting `thinking_end`. A duplicate terminal event after a normal close follows the same path, so plugins can receive a second start/end pair for one thought stream. Track whether the current message's thinking block was already closed, or ignore duplicate terminal events after a matched end.
  - confidence: 0.86
- **[P2] Respect code spans in message_end fallback reasoning:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:463-465`
  - body: The `message_end` safety net falls back to `extractThinkingFromTaggedText(extractAssistantTextWithThinkingTags(msg))`, but that extractor is regex-based and does not honor the code-span filtering used by streamed tag parsing. If native thinking is open and the final answer contains a literal code example like `<think>...</think>`, the hook can emit bogus reasoning text to plugins. Reuse a code-span-aware extractor or avoid parsing user-visible code as fallback reasoning.
  - confidence: 0.78
