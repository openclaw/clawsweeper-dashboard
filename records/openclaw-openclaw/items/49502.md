## Summary

Keep open: the feature is useful and current main still lacks the requested `agent.wait` metadata, but the PR head has correctness blockers in the wait race, optional cost path, and stale terminal response shape, plus it still needs real behavior proof before merge.

## What This Changes

The branch adds optional `meta` usage/cost/session metadata to Gateway agent wait terminal snapshots and forwards it through the dedupe and `agent.wait` response paths.

## Best Possible Solution

Add the metadata as an additive Gateway contract while preserving current terminal fields and cancellation semantics, keeping cost calculation best-effort, adding focused wait-path coverage/docs, and requiring redacted real Gateway output before merge.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection of PR head shows the aborted dedupe fallback, fatal optional cost path, and stale terminal response shape. I did not run a live Gateway flow in this read-only review.

## Solution Assessment

No. The API direction is plausible, but this branch is not yet the best maintainable implementation until the race, best-effort metadata handling, current-main field preservation, rebase, tests, and real behavior proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Keep the dedupe wait alive for metadata fallback:** `src/gateway/server-methods/agent.ts:856`
  - body: When lifecycle wins, this aborts `dedupePromise` before the later metadata fallback awaits it. `waitForTerminalGatewayDedupe` resolves `null` on abort, so the common lifecycle-first `agent.wait` path can still return without the usage/cost `meta` that becomes available in the dedupe payload moments later.
  - confidence: 0.91
- **[P2] Keep cost metadata best-effort:** `src/gateway/server-methods/agent.ts:166-170`
  - body: `buildUsageMetaFromRunResult` runs after `agentCommandFromIngress` succeeds, but it calls `loadConfig()` while building optional metadata. If config reading or validation throws, the surrounding promise catch converts a completed run into an error response instead of omitting `costUsd` and preserving the successful result.
  - confidence: 0.84
- **[P2] Preserve current wait terminal fields:** `src/gateway/server-methods/agent.ts:885-891`
  - body: Current main returns `stopReason`, `livenessState`, and `yielded` from `agent.wait`, and the SDK uses those fields to classify cancelled and timed-out runs. This stale response shape only adds `meta` and would drop those fields if carried forward during rebase.
  - confidence: 0.86
