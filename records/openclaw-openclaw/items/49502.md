## Summary

Keep this PR open. Current main still lacks the requested `agent.wait` usage/cost metadata, and the PR remains relevant, but the patch is not merge-ready because lifecycle-first waits can still drop the new metadata and optional cost estimation can fail an otherwise successful run.

## What This Changes

The PR adds optional usage/cost metadata to gateway wait snapshots, derives it from completed agent run metadata, stores it in gateway dedupe payloads, and forwards it through `agent.wait` terminal responses.

## Best Possible Solution

Refresh the PR against current main, add a typed optional metadata shape through the gateway wait contract, make cost lookup non-fatal, make lifecycle-first waits deterministically obtain metadata, and add focused tests plus protocol documentation for the chosen response surface.

## Reproduction Assessment

Not applicable as a normal user-facing bug reproduction, but there is a high-confidence source inspection path. Current main lacks the metadata while `EmbeddedPiAgentMeta` has the source data, and the PR's lifecycle-first race is visible in the wait/dedupe control flow.

## Solution Assessment

No. The additive direction is reasonable, but this patch is not the best implementation until the lifecycle/dedupe race and non-fatal cost estimation issues are fixed and the new wire shape is documented/tested.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep dedupe alive before reading metadata fallback:** `src/gateway/server-methods/agent.ts:868`
  - body: When lifecycle wins the wait race, this patch aborts the dedupe wait before awaiting it for `meta`. That makes the new metadata nondeterministic for the common in-flight `agent.wait` flow because the aborted dedupe promise can resolve `null` before the terminal payload with usage data is written.
  - confidence: 0.89
- **[P2] Make cost metadata calculation best-effort:** `src/gateway/server-methods/agent.ts:160-164`
  - body: `buildUsageMetaFromRunResult` calls `loadConfig()` while constructing optional response metadata in the successful completion path. If config loading or validation throws, the outer `.catch` reports the already-completed run as an error; cost lookup should be wrapped or use already-loaded config so metadata cannot fail the run.
  - confidence: 0.86
