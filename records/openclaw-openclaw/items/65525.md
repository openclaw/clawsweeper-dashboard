## Summary

Keep open. Current `main` still resolves stdio MCP server env from the configured server block only, then spawns with the MCP SDK default environment plus that explicit env, so ambient gateway `HTTP_PROXY`/`HTTPS_PROXY`/`NODE_OPTIONS` are not propagated. The issue is also paired with open PR #65574 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests MCP stdio child processes to inherit gateway proxy env vars and `NODE_OPTIONS`, or to document that these values must be set explicitly per server.

## Best Possible Solution

Land or replace the linked MCP stdio fix with a current-main patch that propagates safe proxy env defaults with precedence tests, preserves dangerous-env filtering, and documents the deliberate `NODE_OPTIONS` policy.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: start the gateway with proxy env or `NODE_OPTIONS`, configure a stdio MCP server without explicit `env`, and current `main` resolves only the raw server env before the transport spawns with SDK defaults that exclude those ambient keys.

## Solution Assessment

Unclear for the full requested scope. Propagating safe proxy env vars from the MCP stdio launch path is narrow, but copying `NODE_OPTIONS` blindly would bypass the existing startup-safety model.
