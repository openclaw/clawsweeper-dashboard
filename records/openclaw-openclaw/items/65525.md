## Summary

Keep open. Current main still launches MCP stdio children with only the MCP SDK default environment plus explicit server env, so gateway-level proxy variables and NODE_OPTIONS are not propagated. The issue is also paired with open PR #65574 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

MCP stdio child processes should inherit or clearly document gateway proxy-related environment and NODE_OPTIONS behavior so proxy-only sandboxes can run external MCP servers.

## Best Possible Solution

Land or replace the linked MCP stdio fix with a current-main patch that propagates safe ambient proxy-related env vars with explicit precedence tests, preserves the existing dangerous-env filtering model, and makes a deliberate documented decision for NODE_OPTIONS.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: start the gateway with HTTP_PROXY, HTTPS_PROXY, or NODE_OPTIONS and configure a stdio MCP server without explicit env; current main resolves only raw server env and the transport spawns with SDK defaults that exclude those ambient keys.

## Solution Assessment

Unclear for the full NODE_OPTIONS part. Propagating safe proxy env from the MCP stdio launch path is narrow and maintainable, but NODE_OPTIONS should not be copied blindly because current code and docs treat it as interpreter startup control.
