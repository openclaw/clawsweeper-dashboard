## Summary

Keep open. Current main still does not propagate ambient proxy or NODE_OPTIONS state into MCP stdio child processes, and the linked implementation PR remains open with a real NODE_OPTIONS startup-policy question.

## What This Changes

The issue asks OpenClaw to propagate gateway proxy-related environment variables and NODE_OPTIONS to MCP stdio child processes, or clearly document that each server must configure them explicitly.

## Best Possible Solution

Land or replace the linked MCP stdio fix with safe proxy env propagation, explicit user-env precedence, preserved startup-env filtering, and a documented NODE_OPTIONS policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, from source inspection: start the gateway with HTTPS_PROXY or NODE_OPTIONS, resolve a stdio MCP server without explicit env, and current main still passes no ambient proxy/default startup env before the transport adds only the MCP SDK allowlist.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the full requested scope. Propagating safe proxy env defaults at the stdio launch boundary is the right direction, but NODE_OPTIONS needs an explicit security decision and tests before merge.
