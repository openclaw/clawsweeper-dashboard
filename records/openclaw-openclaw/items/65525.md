## Summary

Keep open. Current main still does not propagate ambient proxy variables or NODE_OPTIONS into MCP stdio children, and the open closing PR remains the active but conflicting implementation path with a NODE_OPTIONS policy/security question.

## What This Changes

MCP stdio server child processes should inherit gateway proxy environment, and possibly NODE_OPTIONS, so proxy-only sandbox MCP servers can reach external APIs.

## Best Possible Solution

Land or replace the linked PR with a refreshed stdio launch-boundary fix that propagates safe ambient proxy defaults, preserves user-env precedence and the startup-env denylist, and documents/tests the NODE_OPTIONS policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start the gateway with HTTPS_PROXY or NODE_OPTIONS, configure a stdio MCP server without explicit env, and current main resolves no ambient proxy/startup env before spawning with only the MCP SDK allowlist plus configured env.

## Solution Assessment

Unclear for the full requested scope. Propagating safe proxy env defaults at the stdio resolver boundary is the maintainable direction, but NODE_OPTIONS inheritance needs a deliberate security/product policy and the open PR must preserve main's env filter.
