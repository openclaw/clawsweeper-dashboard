## Summary

Keep open. Current main still passes only configured stdio MCP env into child processes, the linked implementation PR is still open/conflicting, and the NODE_OPTIONS part needs explicit security/product policy before merge.

## What This Changes

The issue requests MCP stdio child processes to receive gateway proxy env vars and NODE_OPTIONS, or clearer documentation for explicit per-server env configuration.

## Best Possible Solution

Land or replace the linked fix with safe ambient proxy env propagation at the stdio launch boundary, user-env precedence, preserved toMcpEnvRecord filtering, tests/docs, and an explicit NODE_OPTIONS host-inheritance policy.

## Reproduction Assessment

Yes. From source inspection, start the gateway with HTTPS_PROXY or NODE_OPTIONS, configure a stdio MCP server without explicit env, and current main resolves no ambient proxy/default startup env before the transport adds only the MCP SDK allowlist.

## Solution Assessment

Unclear for the full requested scope. Propagating safe proxy env defaults at the resolver boundary is the right direction, but NODE_OPTIONS inheritance must be scoped, documented, and tested against the current startup-env safety policy.
