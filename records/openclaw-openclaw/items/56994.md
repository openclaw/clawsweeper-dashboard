## Summary

Keep open. Current main has implemented the restart-sentinel and `saveJsonFile` parts, but shell completion installation still writes user shell profiles with direct `fs.writeFile`, and exec approvals still use a bespoke temp-and-rename path without the temp-file fsync and parent-directory sync guarantees requested here.

## What This Changes

The issue requests durable atomic writes for exec approvals, restart sentinels, shell profile updates, and the sync JSON save helper to reduce crash-corruption risk.

## Best Possible Solution

Finish the two remaining paths by reusing or extracting a durable sync writer for exec approvals and using the existing async text atomic writer for shell profile creation/update.

## Reproduction Assessment

Yes, at source level: current main still reaches direct shell-profile `fs.writeFile` calls and an exec-approvals temp write without fsync or parent-directory sync. I did not run a crash harness, so this is not marked as live reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is to leave the already-fixed restart sentinel and `saveJsonFile` behavior alone, then update only shell profile writes and exec approvals with focused regression coverage.

## Repair Work Prompt

Fix the remaining crash-safety gaps from openclaw/openclaw#56994 on current main. Keep the already-implemented restart sentinel and `saveJsonFile` behavior intact unless extracting a shared helper is necessary. Update shell completion profile creation/update in `src/cli/completion-runtime.ts` to use `writeTextAtomic` with mode 0o644. Replace or refactor `writeExecApprovalsRaw` in `src/infra/exec-approvals.ts` so it preserves the existing trusted-root/symlink guards, hard-link replacement behavior, JSON formatting, and 0o600 final mode while adding temp-file fsync, atomic replace, chmod, cleanup, and best-effort parent-directory fsync. Prefer a small shared sync atomic writer under `src/infra` if it avoids duplicating `saveJsonFile` durability logic. Add focused tests for completion profile atomic writes and exec approval durability behavior, and add a one-line user-facing changelog fix entry. Do not broaden the PR to unrelated direct writes such as device identity files.
