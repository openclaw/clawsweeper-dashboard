- `impact:data-loss`: The issue is about crash-time truncation or corruption of persisted user state and shell profile files.

## Summary

Keep open: current main and latest `v2026.6.10` still rewrite shell completion profiles with direct `fs.writeFile`, so the crash-safety report is only partially fixed; shared JSON, exec approval persistence, and restart sentinel persistence have already moved away from the originally reported direct target writes.

## What This Changes

The issue asks OpenClaw to use atomic writes for exec approvals, restart sentinel state, shell completion profiles, and shared JSON persistence so crashes cannot corrupt user state.

## Best Possible Solution

Keep the fixed persistence paths as-is and land a narrow completion-profile patch that uses the existing atomic text replacement helper for profile creation and updates without changing profile resolution behavior in the same change.

## Reproduction Assessment

Yes. Source inspection shows `installCompletion` reads the selected profile and writes the full updated profile back with direct `fs.writeFile`; I did not run crash injection because this review is read-only.

## Solution Assessment

Yes for the remaining gap: the narrow maintainable fix is to route profile creation and update through the existing atomic text writer, with focused tests proving the profile content and permissions stay compatible.
