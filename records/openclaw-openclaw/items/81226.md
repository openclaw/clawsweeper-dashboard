## Summary

Keep this PR open, but it is not merge-ready: the bounded Discord reconnect recovery is useful, yet the new persisted inbound dedupe can mark messages as processed before the queued Discord turn has actually committed, creating a message-loss window after restart.

## What This Changes

The PR adds Discord READY/RESUMED reconnect backfill from recent outbound anchors, reply-reference hydration hardening, persisted recent inbound/outbound dedupe state, and focused Discord regression coverage.

## Best Possible Solution

Keep the bounded Discord recovery only if maintainers accept that scope, move recent inbound persistence to the successful queue/replay commit path, and require current-head live Discord proof or an explicit proof override before merge.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection shows current main lacks reconnect backfill, and the blocking PR timing bug is source-reproducible from the handler enqueue and run-queue commit contracts.

## Solution Assessment

No. The direction is plausible, but the proposed dedupe persistence happens before successful processing commits; the safer path is post-commit persistence plus live proof or a maintainer override.
