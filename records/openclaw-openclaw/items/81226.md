## Summary

Keep open; this PR is not merge-ready because the newest head still persists recent inbound ids before Discord queued processing/replay commit, which can make reconnect backfill skip an unprocessed message, and the supplied proof is synthetic rather than live current-head Discord reconnect proof.

## What This Changes

The PR adds Discord READY/RESUMED reconnect backfill from recent outbound anchors, persisted recent inbound/outbound message-id caches, reply-reference hydration hardening, and focused Discord monitor tests.

## Best Possible Solution

Move recent inbound persistence to the successful queued-processing/replay-commit boundary for normal and backfilled messages, add a restart-window regression, then require current-head live Discord reconnect proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes, at source level: current main registers live Discord MESSAGE_CREATE handling without a READY/RESUMED history replay hook, and related open issues describe the missed-message window. I did not run a live credentialed Discord restart in this read-only review.

## Solution Assessment

No, not yet. The bounded Discord backfill direction is plausible, but this implementation persists inbound dedupe before successful queued processing and lacks current-head live behavior proof.
