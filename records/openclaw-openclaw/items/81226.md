## Summary

Keep open. This is a focused Discord reliability fix, but the patch can let the recent-outbound recovery anchor expire based on an older bot message, and the supplied behavior proof is still simulated rather than a real Discord Gateway/runtime reconnect.

## What This Changes

The PR adds Discord in-memory outbound anchors, READY/RESUMED REST backfill through the existing message handler, reply-reference hydration hardening, and regression tests.

## Best Possible Solution

Land the bounded reconnect backfill only after refreshing anchor freshness correctly and obtaining live Gateway/runtime proof or an explicit maintainer proof override.

## Reproduction Assessment

No for a high-confidence live reproduction. Source shows current main lacks READY/RESUMED backfill and the PR harness models the dropped-event path, but no live Discord reconnect proof demonstrates recovery.

## Solution Assessment

No, not as written. The bounded handler/dedupe design is the right direction, but the recent-outbound anchor freshness bug should be fixed and the runtime proof gap closed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Refresh anchors for newer outbound messages:** `extensions/discord/src/recent-outbound.ts:54-56`
  - body: When a channel already has an anchor, a newer outbound inside the 15-minute window returns without refreshing `at`. A long or multi-message bot response near the end of that window can then expire the only anchor before reconnect, so a user reply shortly after the newer bot message is not backfilled. Keep the earliest message id if needed for overlap, but refresh the window timestamp or store a separate last-seen time.
  - confidence: 0.87
