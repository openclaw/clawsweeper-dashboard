## Summary

Keep open: the PR addresses a real Discord message-loss gap, but latest head still has a blocking delivery bug and only synthetic, stale proof. The new persisted inbound guard can mark a Gateway message as handled before the queued Discord turn actually finishes, so a restart in that window can make reconnect backfill skip the message and lose it.

## What This Changes

The branch adds Discord READY/RESUMED reconnect backfill from recent outbound anchors, persists recent inbound/outbound message ids, hardens reply hydration, and adds focused Discord monitor tests.

## Best Possible Solution

Land the Discord-plugin-local backfill only after persisted inbound dedupe is tied to successful processing/replay commit and current-head live Discord reconnect proof or an explicit proof override is supplied; keep broader all-channel catch-up under https://github.com/openclaw/openclaw/issues/55792.

## Review Findings

Full review comments:

- **[P1] Mark inbound messages only after processing succeeds:** `extensions/discord/src/monitor/listeners.ts:80-84`
  - body: This records the message id in the persisted recent-inbound guard as soon as `this.handler` resolves, but the Discord handler has only enqueued/debounced the turn at that point; actual processing and replay commit happen later in the run queue. If the process restarts after enqueue but before preflight/process succeeds, reconnect backfill will see the REST candidate as already processed and skip it, silently losing the exact message this PR is meant to recover. Move this persistence to the successful processing/commit path and cover that restart window.
