## Summary

Keep open: the feature direction is concrete, but the proposed WhatsApp adapter can clear the terminal reaction it is trying to show, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a WhatsApp status-reaction controller path, wires it into WhatsApp inbound dispatch, and extends shared status-reaction emoji categories/config keys.

## Best Possible Solution

Use a WhatsApp-safe lifecycle path that treats reactions as replace-only until explicit final cleanup, then add focused tests and real WhatsApp plus Telegram proof for the visible reaction sequence.

## Reproduction Assessment

Yes for the blocking review finding from source: with WhatsApp status reactions enabled, terminal setDone/setError flows into per-emoji cleanup, while the proposed WhatsApp adapter removes the single current reaction for each cleanup call. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

No: the shared remove-capable controller adapter is not the narrowest safe solution for WhatsApp's single-reaction semantics. A replace-only WhatsApp adapter or controller mode should preserve terminal reactions and perform cleanup only when explicitly requested.
