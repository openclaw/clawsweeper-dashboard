## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the current WhatsApp status reaction:** `extensions/whatsapp/src/auto-reply/monitor/status-reaction.ts:97-99`
  - body: The shared controller removes tracked old emojis after terminal or restore transitions, but this adapter maps every named removal to `sendReactionWhatsApp(..., "")`. On WhatsApp that clears the sender's single reaction slot, so `setDone()`/`restoreInitial()` can immediately erase the reaction that was just set.
  - confidence: 0.92
- **[P2] Keep audio preflight acknowledgements immediate:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:141-143`
  - body: When status reactions are enabled, this skips the preflight ack before audio transcription, but the status controller is only created later in `processMessage` after `runAudioPreflightOnce()` completes. Audio messages therefore lose the current immediate receipt reaction before slow STT work.
  - confidence: 0.86
- **[P2] Use distinct Telegram fallbacks for new categories:** `extensions/telegram/src/status-reaction-variants.ts:97-98`
  - body: The new deploy/build defaults are not in Telegram's supported reaction list, and both fallback lists start with the generic tool emoji. In allowed-all chats, `fastlane` and `xcodebuild` still render as the generic tool reaction instead of a distinct category signal.
  - confidence: 0.8
- **[P3] Update the public status reaction docs:** `src/config/schema.help.ts:1823-1824`
  - body: This adds WhatsApp status reactions and new emoji override keys, but the public config guide still says lifecycle status reactions are Slack/Discord/Telegram-only. Update the user-facing docs so operators can discover the WhatsApp support and new overrides.
  - confidence: 0.78
