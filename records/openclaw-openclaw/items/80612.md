## What This Changes

The PR adds WhatsApp lifecycle status reactions, deploy/build/concierge status-reaction emoji categories, clearer default emoji, and related Slack, Telegram, config, and changelog updates.

## Best Possible Solution

Repair the branch with WhatsApp-aware single-reaction semantics, preserve pre-STT voice-note feedback, update public docs, and provide redacted live WhatsApp lifecycle plus Telegram-visible proof.

## Reproduction Assessment

Yes for source-level review findings: the PR diff maps cleanup removals to empty WhatsApp reactions and skips the preflight ack, while current main and tests show the existing ack-before-STT contract. I did not run a live WhatsApp or Telegram session in this read-only review.

## Solution Assessment

No. The current PR reuses a per-emoji cleanup contract on WhatsApp's single reaction slot and delays voice-note receipt feedback; the safer solution is a WhatsApp-specific reaction lifecycle path plus docs and redacted live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve WhatsApp's final status reaction:** `extensions/whatsapp/src/auto-reply/monitor/status-reaction.ts:97-99`
  - body: The adapter maps every shared-controller `removeReaction` call to `sendReactionWhatsApp(..., "")`, but Baileys treats an empty reaction as removing the current reaction. When `setDone`, `setError`, or `restoreInitial` applies the final/restored emoji and then removes prior active emojis, these callbacks can clear the single WhatsApp reaction slot instead of preserving the intended status.
  - confidence: 0.91
- **[P2] Keep voice-note acknowledgements before STT:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:153-166`
  - body: When status reactions are enabled, this skips the preflight ack, but the status controller is only created later in `processMessage` after `runAudioPreflightOnce` completes. Voice notes can therefore lose the immediate receipt reaction that current code and tests preserve before slow transcription work.
  - confidence: 0.86
- **[P3] Update the public status reaction docs:** `src/config/schema.help.ts:1850-1853`
  - body: This adds WhatsApp status reactions and new emoji override keys, but the public config guide still says lifecycle status reactions are Slack/Discord/Telegram-only. Update the user-facing docs so operators can discover WhatsApp support and the deploy/build/concierge overrides.
  - confidence: 0.78
