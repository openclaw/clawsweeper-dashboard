## Summary

Keep this PR open, but do not merge it yet: the WhatsApp status-reaction adapter uses the shared controller with the wrong removal semantics, the audio preflight path loses early receipt feedback, dispatcher errors can leave lifecycle timers/reactions active, and the external PR has no after-fix real behavior proof.

## What This Changes

The PR adds WhatsApp status reaction lifecycle wiring, extends status reaction emoji categories/defaults across shared channel feedback and Telegram variants, and updates config schema/types/help for the new emoji keys.

## Best Possible Solution

Fix the WhatsApp controller integration so single-slot reaction replacement and clearing are modeled correctly, preserve early audio preflight feedback, finalize/clear on dispatch errors, add focused regression coverage, and require redacted real WhatsApp/Telegram proof before merge.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR adapter maps per-emoji removal to empty WhatsApp reactions, suppresses preflight ack before the replacement controller exists, and awaits dispatch without error finalization. No live WhatsApp or Telegram proof was provided.

## Solution Assessment

No. The feature direction is plausible, but this implementation needs a WhatsApp-safe reaction clear/replace contract, preflight lifecycle wiring, and dispatch-error finalization before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Do not clear the WhatsApp terminal reaction:** `extensions/whatsapp/src/auto-reply/monitor/status-reaction.ts:97-99`
  - body: The shared controller treats `removeReaction` as per-emoji cleanup for platforms with multiple active reactions. Here it calls `sendReactionWhatsApp(..., "")`, so after `setDone`/`setError` the cleanup of older tracked emojis clears WhatsApp's single reaction slot and removes the terminal reaction that was just set. Use a WhatsApp-safe clear/replace path instead of exposing empty-string removal as per-emoji removal.
  - confidence: 0.94
- **[P2] Keep audio preflight receipt feedback:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:144-157`
  - body: When `statusReactions.enabled` is true this skips the early ack before `transcribeFirstAudio`, but the new controller is only created later in `processMessage`. Slow voice-note transcription therefore has no queued/ack reaction until STT finishes, regressing the existing preflight feedback path; create the status controller before preflight or keep the plain ack until it can take over.
  - confidence: 0.88
- **[P2] Finalize status reactions when dispatch throws:** `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:495-500`
  - body: After starting the status controller, the dispatcher is awaited without a catch/finally path. If `dispatchReplyWithBufferedBlockDispatcher` rejects, `finalizeWhatsAppStatusReaction` is never called and the controller's reaction/timers can remain active instead of moving to error or clearing.
  - confidence: 0.84
