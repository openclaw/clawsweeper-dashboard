## What This Changes

The PR wires shared lifecycle status reactions into WhatsApp, adds deploy/build/concierge status-reaction categories/defaults, and extends config schema/types/help plus Telegram reaction variants.

## Best Possible Solution

Preserve WhatsApp single-reaction semantics with a WhatsApp-safe replace/cleanup path, keep audio preflight receipt feedback immediate, update public docs, and add redacted live WhatsApp lifecycle plus Telegram-visible proof.

## Reproduction Assessment

Yes for the review findings from source: the shared controller’s terminal cleanup, the proposed WhatsApp clear adapter, and the existing audio-preflight ack ordering make the regressions traceable without a live account. I did not run a live WhatsApp or Telegram session in this read-only review.

## Solution Assessment

No: as written, the patch reuses a multi-reaction cleanup contract on a single-current-reaction surface and delays audio receipt feedback. A safer solution should make WhatsApp status reactions replace-only or otherwise avoid clearing the final/restored emoji while preserving the preflight ack path.

  - body: The shared controller calls removeReaction for tracked prior emojis after setDone/setError.
