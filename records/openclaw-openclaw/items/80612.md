## Summary

Keep open: the PR still has source-backed correctness blockers, and the refreshed real-behavior proof only demonstrates the first WhatsApp queued reaction rather than the full lifecycle, cleanup behavior, or Telegram-visible category changes.

## What This Changes

The PR adds WhatsApp lifecycle status reactions and extends shared status-reaction emoji categories/config keys for deploy, build, and concierge tool activity.

## Best Possible Solution

Use a WhatsApp-safe status-reaction path that preserves single-reaction replacement semantics, keeps audio receipt feedback immediate, updates user-facing config docs, and then adds redacted live WhatsApp plus Telegram-visible proof.

## Reproduction Assessment

Yes for the review findings from source: the shared controller’s terminal cleanup calls per-emoji removal, the proposed WhatsApp adapter clears the single current reaction for each removal, and current tests require ack before audio STT. I did not run a live WhatsApp or Telegram account in this read-only review.

## Solution Assessment

No: wiring WhatsApp into the shared multi-reaction removal adapter is not the narrowest safe solution for a single-current-reaction surface. A WhatsApp-specific replace-only mode or adapter contract should preserve terminal reactions and perform explicit cleanup only when configured.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the final WhatsApp status reaction:** `extensions/whatsapp/src/auto-reply/monitor/status-reaction.ts:97-99`
  - body: The shared controller calls `removeReaction` for tracked prior emojis after `setDone`/`setError`. This adapter maps each removal to `sendReactionWhatsApp(..., "")`, which clears WhatsApp's single current reaction slot, so the final done/error reaction can disappear immediately instead of being held or restored.
  - confidence: 0.91
- **[P2] Keep audio preflight acknowledgements immediate:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:141-143`
  - body: When status reactions are enabled, this skips the preflight ack before audio transcription, but the status controller is only created later in `processMessage` after `runAudioPreflightOnce()` completes. Audio messages can therefore lose the immediate receipt reaction that current code sends before STT.
  - confidence: 0.84
- **[P3] Update the public status reaction docs:** `src/config/schema.help.ts:1823-1824`
  - body: This adds WhatsApp status reactions and new emoji override keys, but the public config guide still says status reactions are Slack/Discord/Telegram-only. Update the user-facing docs so operators can discover the WhatsApp support and the new `deploy`, `build`, and `concierge` overrides.
  - confidence: 0.78
