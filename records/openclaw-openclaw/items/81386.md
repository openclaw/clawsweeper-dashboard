## Summary

Keep open: the hardening direction is useful, but the patch still leaves no-agent `sessions.usage` requests on broad all-agent discovery, lacks external real behavior proof, and carries the protected `maintainer` label.

## What This Changes

The PR threads optional `agentId` through Gateway session/artifact/usage APIs and memory-core/memory-wiki visibility paths, with Swift/SDK schema updates and focused regression tests.

## Best Possible Solution

Finish the agent-scoping hardening by making no-agent `sessions.usage` follow the agreed effective/default agent, or keep all-agent usage behind an explicit maintainer-approved mode, then prove it with a redacted multi-agent Gateway run.

## Reproduction Assessment

Yes from source inspection: the unchanged UI sends `sessions.usage` without `agentId`, and the PR keeps that path unscoped while tests assert both agents are discovered. I did not run a live Gateway because this was a read-only review.

## Solution Assessment

No. Most of the PR follows the right scoping direction, but the no-agent usage path still performs broad discovery; the narrower maintainable fix is to resolve an effective/default agent before the store load and discovery, unless maintainers explicitly choose an all-agent mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope default usage queries before loading sessions:** `src/gateway/server-methods/usage.ts:869-880`
  - body: For a list-style `sessions.usage` call with no `agentId` and no `key`, `effectiveAgentId` stays undefined, so the handler still loads the combined store with `{}` and discovers every configured agent. The Control UI sends exactly that shape, while the new schema/PR text says omitted usage queries should use the effective/default agent, so this leaves the data-scope hardening incomplete.
  - confidence: 0.9
