## Summary

Keep open: current main and the latest release still lack the requested opt-in `agent` RPC image-offload path, while the source-backed divergence between `agent` and `chat.send` remains. This is a plausible core gateway protocol enhancement, but it needs maintainer API/product approval before implementation.

## What This Changes

Request an opt-in `agent` RPC parameter that offloads image attachments to `agents.defaults.imageModel` when the resolved primary model is text-only.

## Best Possible Solution

Approve an additive `agent` RPC option only after maintainer API/product review, keep strict behavior as the default, and route opt-in text-only image offloads through the existing dispatch/media-understanding path with focused protocol, gateway, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: current `agent` passes `supportsInlineImages` without `supportsImages: false`, and the parser throws `UnsupportedAttachmentError` for text-only image input. I did not run a live gateway repro because this review is read-only and the source/test path is clear.

## Solution Assessment

Unclear until maintainer direction: the opt-in flag is a plausible narrow design, but the final public `agent` RPC parameter shape and media-staging contract need API/product approval.
