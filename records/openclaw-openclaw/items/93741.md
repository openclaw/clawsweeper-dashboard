## Summary

Keep open: current main still lacks an opt-in `agent` RPC image-offload path, while source and tests show `chat.send` has the text-only image offload behavior and `agent` remains strict. This is a small, aligned feature/API request with no open duplicate implementation, but the public parameter shape needs maintainer product approval.

## What This Changes

The issue requests an opt-in `agent` RPC parameter so text-only primary models can offload image understanding to `agents.defaults.imageModel` while preserving strict default behavior.

## Best Possible Solution

Add a narrow opt-in gateway protocol field, keep the current strict default, and cover both the `agent` RPC success path and existing rejection paths in focused tests/docs after maintainers approve the API shape.

## Reproduction Assessment

Yes from source inspection: a current-main `agent` request with an image attachment and a resolved text-only model reaches `parseMessageWithAttachments` without `supportsImages: false`, which is the condition that throws `UnsupportedAttachmentError`. I did not run a live gateway repro in this read-only review.

## Solution Assessment

Likely yes if maintainers want the capability: an additive opt-in field is narrower than changing the default or widening `chat.send`, but the public field name and exact behavior need product/API approval.
