## Summary

Keep open: current main and v2026.6.9 still lack the requested opt-in `agent` RPC image-offload behavior, while source and tests confirm `chat.send` has the staged media-understanding route and `agent` still keeps the strict text-only image rejection path. The request is well-supported but adds public Gateway protocol/provider-routing surface, so it needs maintainer API/product approval before implementation.

## What This Changes

Requests an additive opt-in `agent` RPC parameter that lets text-only primary models offload image attachments to configured `agents.defaults.imageModel` while preserving strict default behavior.

## Best Possible Solution

If maintainers accept the API direction, add a narrow additive `agent` RPC opt-in that preserves the strict default and routes text-only image offloads through the existing staged media-understanding path with focused protocol, gateway, test, and docs updates.

## Reproduction Assessment

Yes, source-reproducible: current `agent` passes only `supportsInlineImages`, while the parser only offloads text-only images when `supportsImages` is explicitly false. I did not run a live gateway reproduction because this review is read-only and the source/test path is clear.

## Solution Assessment

Unclear until maintainer direction: the opt-in flag is a plausible narrow solution, but the public `agent` RPC surface and media-understanding staging contract need API/product approval first.
