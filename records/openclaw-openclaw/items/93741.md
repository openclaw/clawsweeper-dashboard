## Summary

Keep open: current `main` and the latest release still do not provide the requested opt-in `agent` RPC image-offload behavior, while the source confirms `chat.send` and `agent` still diverge for text-only image attachments. The request is well-supported, but it adds public gateway protocol surface, so it needs maintainer API/product approval before implementation.

## What This Changes

The issue requests an opt-in `agent` RPC parameter that lets image attachments to text-only models use the configured `agents.defaults.imageModel` path instead of failing fast.

## Best Possible Solution

Add a maintainer-approved additive gateway protocol option only if the strict `agent` default remains unchanged and the shared dispatch/media-understanding path owns the offload behavior.

## Reproduction Assessment

Yes from source: `agent` still reaches `parseMessageWithAttachments` with `supportsInlineImages` but without the `supportsImages: false` offload trigger, and the adjacent test expects text-only image input to be rejected. I did not run a live gateway reproduction because this review is read-only and the source/test path is clear.

## Solution Assessment

Unclear until maintainer direction: the opt-in flag is a plausible narrow design, but public `agent` RPC surface and media-understanding staging need API/product acceptance first.
