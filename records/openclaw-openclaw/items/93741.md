## Summary

Keep open: current main and the latest release still lack the requested opt-in `agent` RPC image offload path, while source shows `agent` remains strict for text-only image attachments and `chat.send` already routes those images into media understanding. The request is plausible, but it adds public gateway protocol behavior and needs maintainer API/product approval before implementation.

## What This Changes

Request an additive `agent` RPC option that lets text-only primary models offload image attachments through `agents.defaults.imageModel` while preserving the strict default.

## Best Possible Solution

Approve a narrow additive `agent` RPC option if maintainers want this behavior, keep the default strict, and wire the opt-in through the same media path plus media-understanding staging used by `chat.send`.

## Reproduction Assessment

Yes, at source level: current `agent` handling reaches `parseMessageWithAttachments` without `supportsImages: false`, and the adjacent test expects text-only image input to be rejected. I did not run a live gateway reproduction because this review is read-only and the source/test path is clear.

## Solution Assessment

Unclear until maintainer direction: the opt-in flag is a plausible narrow solution, but it adds public gateway protocol behavior and needs API/product approval. The safer implementation shape is to reuse the existing `chat.send` media-understanding staging path and preserve the strict default.
