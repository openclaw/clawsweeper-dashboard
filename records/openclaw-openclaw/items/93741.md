## Summary

Keep open: current main and v2026.6.9 still keep `agent` image attachments strict for text-only models, while `chat.send` has the staged media-understanding path. The request is source-backed but adds public gateway protocol behavior, so it needs maintainer product/API approval before implementation.

## What This Changes

Requested an opt-in `agent` RPC parameter that lets text-only primary models offload image attachments through the configured image model while preserving the current fail-fast default.

## Best Possible Solution

Approve and implement an additive `agent` RPC offload option only after maintainer API direction, keep the strict default unchanged, and route the opt-in path through the existing staged media-understanding flow with regression docs/tests.

## Reproduction Assessment

Yes. Source and adjacent tests show current `agent` handling reaches `parseMessageWithAttachments` without `supportsImages: false`, so a text-only image request remains the strict `UnsupportedAttachmentError` path; I did not run a live gateway repro in this read-only review.

## Solution Assessment

Unclear until maintainer direction. The opt-in flag is a plausible narrow API shape, but the implementation must also wire staged media-understanding context; simply passing `supportsImages: false` would not by itself mirror `chat.send`.
