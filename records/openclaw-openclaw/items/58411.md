## Summary

Keep open: current main and the latest release still expose only `thread` for ACP `sessions_spawn`, while Discord resolves that path to child-thread placement; the existing `/acp spawn --bind here` current-conversation path is not available to agents. The related Telegram issue was fixed for Telegram placement selection, but it does not cover the requested Discord current-thread bind API.

## What This Changes

The issue requests a programmatic `sessions_spawn` option equivalent to `/acp spawn --bind here` so an agent can bind an ACP session to the current Discord thread.

## Best Possible Solution

Add an additive ACP `sessions_spawn` current-conversation binding mode that reuses the existing `/acp spawn --bind here` validation and binding service, while preserving existing `thread: true` child-thread behavior.

## Reproduction Assessment

Yes. The current source path is clear: Discord `sessions_spawn({ runtime: "acp", mode: "session", thread: true })` forwards only `thread`, resolves Discord's default placement as `child`, and the existing ACP spawn test asserts that child placement even from a requester thread.

## Solution Assessment

Yes. An additive `bindTo: "current"` or equivalent ACP-only placement option is the narrow maintainable solution because it exposes existing slash-command semantics without changing `thread: true` behavior.
