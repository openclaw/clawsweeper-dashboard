## Summary

Keep open: current main and the latest release still lack an agent-callable current-conversation bind option for `sessions_spawn`; the tool exposes and forwards only `thread`, while Discord keeps `thread: true` on the child-thread path and `/acp spawn --bind here` remains a slash-command-only flow.

## What This Changes

The issue requests an ACP `sessions_spawn` option that binds the spawned session to the current Discord thread in place, matching `/acp spawn --bind here` semantics.

## Best Possible Solution

Add an ACP-only current-conversation binding mode to `sessions_spawn` that reuses `/acp spawn --bind here` validation and binding behavior while preserving `thread: true` child-thread placement and leaving the broader unthreaded-session contract to its separate issue.

## Reproduction Assessment

Yes, source-level. Current main and `v2026.5.20` expose and forward only `thread` for ACP `sessions_spawn`, while Discord `thread: true` resolves to child placement and the current-bind flow exists only in `/acp spawn --bind here`; no live Discord smoke was run.

## Solution Assessment

Yes. An additive ACP-only `bindTo: "current"` or equivalent option is the narrow maintainable path; changing `thread: true` would risk existing child-thread semantics, and decoupling session mode from thread binding is tracked separately.
