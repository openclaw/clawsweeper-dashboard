## Label Justifications

- `P2`: The issue is a clear, narrow agent/channel workflow gap with limited blast radius and a focused implementation path.
- `impact:session-state`: The requested behavior controls which Discord conversation remains bound to the spawned ACP session for follow-up turns.
- `impact:message-loss`: Without a current-bind tool option, agent-initiated ACP work can move follow-up traffic into a new child thread instead of the active user thread.

## Summary

Keep open: current main still has no agent-callable current-conversation bind option for `sessions_spawn`; the tool exposes and forwards only `thread`, while Discord `thread: true` still follows the child-thread default and `/acp spawn --bind here` remains slash-command-only.

## What This Changes

Request an ACP `sessions_spawn` option that binds the spawned session to the current Discord conversation/thread instead of creating a child thread.

## Best Possible Solution

Add an ACP-only current-conversation binding mode to `sessions_spawn` that reuses `/acp spawn --bind here` validation and binding behavior while preserving `thread: true` child-thread placement.

## Reproduction Assessment

Yes. Source inspection shows the tool schema and execution path expose only `thread`, ACP session mode still requires thread binding, and Discord `thread: true` resolves to child placement; no live Discord smoke was run.

## Solution Assessment

Yes. An additive `bindTo: "current"` or equivalent ACP-only option is the narrow maintainable fix; changing `thread: true` would risk existing child-thread semantics, and the broader unthreaded-session contract is tracked separately.
