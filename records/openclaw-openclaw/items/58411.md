## Summary

Keep open. Current main still has the reported ACP spawn asymmetry: `/acp spawn --bind here` has a current-conversation binding path, but the `sessions_spawn` tool exposes only `thread: true` for ACP binding and the Discord ACP spawn path still resolves that to child-thread placement.

## What This Changes

Add a programmatic `sessions_spawn` option equivalent to `/acp spawn <agent> --bind here` so an agent can bind an ACP session to the current Discord conversation or thread instead of creating a child thread.

## Best Possible Solution

Add an additive current-conversation binding mode to the ACP `sessions_spawn` path, reusing the existing `/acp spawn --bind here` semantics and session binding service. Keep `thread: true` behavior for child thread creation, update docs and system/tool guidance, and add regression coverage for Discord current-thread binding without requiring `spawnAcpSessions`.

## Reproduction Assessment

Yes. The current source and tests give a high-confidence reproduction path: `sessions_spawn({ runtime: "acp", thread: true, mode: "session" })` from Discord goes through `prepareAcpThreadBinding`, resolves Discord placement as `child`, and the existing test asserts that child placement even with a requester thread id.

## Solution Assessment

Yes. An additive `bindTo: "current"` or equivalent ACP-only placement option is the narrow maintainable fix because it preserves existing child-thread semantics while exposing the already-implemented current bind path to agents.
