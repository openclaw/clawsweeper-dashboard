## Summary

Keep open: current main still exposes only `thread` for ACP `sessions_spawn`, and the Discord path still maps that to child-thread placement, while `/acp spawn --bind here` has current-conversation binding semantics that agents cannot request. The related Telegram issue at https://github.com/openclaw/openclaw/issues/41004 was fixed for `thread:true` channel placement, but it does not cover this Discord current-bind API gap.

## What This Changes

Add a programmatic ACP `sessions_spawn` option to bind a spawned session to the current Discord conversation instead of creating or binding a child thread.

## Best Possible Solution

Add an additive ACP-only current-conversation binding mode to `sessions_spawn` that reuses existing `/acp spawn --bind here` validation and binding service while preserving `thread: true` child-thread behavior.

## Reproduction Assessment

Yes, source-backed. On current main, the tool schema and ACP params expose no current-bind field, the execution path forwards only `thread`, Discord defaults that path to child placement, and existing coverage asserts child placement from a requester thread.

## Solution Assessment

Yes. An additive `bindTo: "current"` or equivalent ACP-only option is the narrow maintainable fix because changing `thread: true` would regress existing child-thread semantics.
