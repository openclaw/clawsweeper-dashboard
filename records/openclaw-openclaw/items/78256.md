## Summary

Keep open: current main still has a plausible approved-exec followup routing path where successful subagent completions are resumed through another agent turn instead of being delivered as a strictly output-bound result. The macOS SSH case was not live-reproduced here, but the source path and existing tests leave the reported association bug unresolved.

## What This Changes

Ensure approved subagent exec completions return the stdout/stderr from the approved command rather than unrelated session or run context.

## Best Possible Solution

Bind approved exec completion delivery to the approval id and child session/run, and add a regression proving successful subagent approvals surface only the captured command output even when unrelated prior tool output exists.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the risky current-main path remains: approved exec completions for sessions are fed through a followup agent turn, but the exact intermittent macOS SSH mismatch still needs live trace proof.

## Solution Assessment

Unclear until a regression pins the failing layer. The likely maintainable fix is narrow result/session/approval-id binding in the exec approval followup path, not SSH/auth handling or a new configuration option.

Stay within exec approval and subagent result delivery: successful approved gateway/node exec completions from subagent sessions must be correlated by approval id plus child session/run and must deliver or record the exact captured stdout/stderr/error for the approved command, without reusing unrelated prior tool output or another session's context.
