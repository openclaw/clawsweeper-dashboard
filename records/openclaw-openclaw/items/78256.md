## Summary

The report should stay open: current main still routes approved exec completion follow-ups by session key without a stronger approval/run/session binding, which matches the reported class of wrong-output association in subagent contexts.

## What This Changes

The issue asks for approved subagent exec results to return the stdout/stderr from the approved command rather than unrelated output from another context.

## Best Possible Solution

Bind exec approval completions to the original approval and requester context strongly enough that a late or stale follow-up cannot resume an unrelated subagent/session, while preserving the existing async approval flow.

## Reproduction Assessment

Yes from source inspection, but not from a live run: the approved exec follow-up path is sessionKey-based and lacks the stronger requester/run binding needed to rule out the reported wrong-context result. The exact intermittent SSH scenario was not reproduced on current main.

## Solution Assessment

Unclear until the implementation is designed, but the likely narrow fix is to add correlation/provenance checks around exec approval follow-up delivery rather than changing SSH handling, approval policy, or adding a config option.

Inspect src/agents/bash-tools.exec-host-gateway.ts, src/agents/bash-tools.exec-host-node.ts, src/agents/bash-tools.exec-host-shared.ts, src/agents/bash-tools.exec-approval-followup.ts, src/gateway/server-methods/agent.ts, and subagent session setup in src/agents/subagent-spawn.ts.
