## Summary

Keep open. Current main and the latest release still route successful async exec approval follow-ups without carrying the approved command into the follow-up target or success prompt, and the directly linked fix PR is closed unmerged; the open formatter PR is adjacent but does not solve this output-association bug.

## What This Changes

The issue requests that approved subagent exec results be bound to the exact approved command/run so unrelated stdout or stderr cannot be returned as that command's result.

## Best Possible Solution

Bind the approved command plus approval/run/session identity through gateway and node follow-up targets and the successful follow-up prompt, then add concurrent/out-of-order approval completion coverage and live approval proof.

## Reproduction Assessment

No high-confidence live reproduction was established for the intermittent SSH/subagent case. Source inspection does show the successful async approval follow-up path still lacks approved-command binding on current main and in v2026.6.10.

## Solution Assessment

Yes, the best known fix shape is narrow: carry the approved command through the existing follow-up target/result path and harden the success prompt with out-of-order completion tests. The broader stale-subagent and output-truncation items should remain separate unless maintainers choose to combine the repairs.
