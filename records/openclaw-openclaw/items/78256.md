## Summary

Keep open. Current main and v2026.6.6 still do not carry the approved command into successful exec approval followups, while OpenClaw docs and Codex protocol contracts treat approval ids, command text, and completed output as correlated; the directly linked fix PR was closed unmerged.

## What This Changes

The issue reports approved subagent exec commands sometimes returning output from an unrelated run/context instead of the approved command output.

## Best Possible Solution

Bind the approved command and approval/run/session identity through gateway/node followup targets and the successful followup prompt, make exact completion details authoritative, and add concurrent out-of-order completion coverage.

## Reproduction Assessment

No high-confidence live reproduction was established for the intermittent SSH case. Source inspection shows the successful approval followup still lacks approved-command binding and a successful-path stale-output guard, matching the closed direct fix's diagnosis.

## Solution Assessment

Yes, the best narrow fix is to bind the approved command through the existing followup target/prompt path and cover out-of-order completions. The linked direct fix PR sketched that path but closed unmerged, so current main still needs maintainer follow-up.
