## Summary

Keep this PR open. It addresses a real CLI context-loss path and the branch has already handled the earlier prompt/MCP-priority review concern, but it is not merge-ready because current auth epochs are identity-oriented: OAuth refreshes stay stable, while account/profile identity changes alter the epoch. Returning the old CLI session id for every auth-profile/auth-epoch mismatch can carry conversation context across an identity switch.

## What This Changes

The PR changes CLI session reuse so auth-profile/auth-epoch invalidations return the existing CLI session id, prompt/MCP invalidations force fresh sessions first, reset logging distinguishes reauth from reset, and helper tests cover the new behavior.

## Best Possible Solution

Refine the PR so native CLI resume is preserved only for refresh-safe or same-principal auth metadata changes, while prompt/MCP changes and account/profile identity switches continue to start fresh sessions. The focused coverage should include token refresh, explicit profile switch, auth-epoch identity change, and simultaneous prompt/MCP changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to combine the PR diff with a binding whose prompt/MCP hashes match but whose auth epoch changes due to account identity; current tests prove that identity changes alter the epoch, and the PR would return the old session id for that mismatch.

## Solution Assessment

No. The PR fixes the context-loss mechanism, but the current approach is broader than the maintained auth contract supports; the safer solution needs a same-principal signal or must keep identity-changing auth mismatches on the fresh-session path.
