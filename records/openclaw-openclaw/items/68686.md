## Summary

Keep this PR open. Current main already makes OAuth token refresh stable for CLI auth epochs, but the PR's remaining auth-profile/auth-epoch reuse is broader than the current identity contract and can resume a CLI conversation across profile or account identity changes.

## What This Changes

The PR changes CLI session reuse so auth-profile/auth-epoch invalidations return the existing CLI session id, prompt/MCP invalidations still force fresh sessions first, reset logging distinguishes reauth from reset, and helper tests cover the new behavior.

## Best Possible Solution

Preserve CLI resume only for refresh-safe credential changes that are proven not to change the account principal, keep prompt/MCP changes on fresh sessions, and keep profile or identity switches on fresh sessions unless a narrower identity-continuity contract is added and tested.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: call resolveCliSessionReuse with a binding that has a sessionId and matching prompt/MCP hashes but a changed authProfileId or same-version changed authEpoch; current main returns only invalidatedReason, while this PR would resume the existing CLI session.

## Solution Assessment

No. The PR fixes the context-loss mechanism, but current main's auth epoch contract is identity-oriented and already stable across OAuth token refreshes, so resuming every auth-profile/auth-epoch mismatch is broader than the safest maintainable fix.
