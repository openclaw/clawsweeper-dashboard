## Summary

Keep this PR open, but do not merge it as-is. The branch addresses the prompt/MCP short-circuit from earlier review, but it still resumes an existing CLI session across broad auth-profile/auth-epoch mismatches even though current auth epochs are identity-oriented.

## What This Changes

The PR changes CLI session reuse so auth-profile/auth-epoch invalidations can keep the existing CLI session while prompt and MCP invalidations still force a fresh session, and it adjusts reset logging/tests for that behavior.

## Best Possible Solution

Preserve CLI resume only for credential changes proven to be the same principal, while keeping prompt/MCP changes and identity/profile switches on fresh backend sessions.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call resolveCliSessionReuse with a stored sessionId, matching prompt/MCP hashes, and a changed authProfileId or same-version changed authEpoch; the PR returns sessionId where current main resets.

## Solution Assessment

No. The proposed branch fixes the context-loss mechanism too broadly; the safer fix is to resume only when the auth mismatch is proven refresh-safe or same-principal.

## Review Findings
