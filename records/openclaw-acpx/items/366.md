## Summary

Keep open. Current main still lacks a general caller-provided response-schema path: the CLI has no such option and ACP prompts carry only the session ID and normalized prompt. The request fits acpx’s backend and structured-output direction, but it introduces durable CLI/API conventions and unresolved cross-adapter enforcement behavior that require a maintainer product decision.

## What This Changes

Add a caller-provided JSON Schema contract for the agent’s final response, separate from ACPX transport formatting.

## Best Possible Solution

Define one protocol-aligned response-contract surface that keeps `--format` solely for transport, negotiates native enforcement where supported, and returns typed unsupported-capability or validation errors under an explicit fallback policy.

## Reproduction Assessment

Not applicable; this requests a new response-contract capability rather than reporting broken behavior already promised by documentation or tests. Source inspection does provide high-confidence confirmation that current main lacks the requested option and prompt metadata.

## Solution Assessment

Unclear until maintainers settle the public contract. Separating response constraints from transport formatting is the right direction, but flag naming, schema input form, capability negotiation, validation ownership, and fallback behavior should be designed before implementation.
