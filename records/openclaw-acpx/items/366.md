## Summary

Keep open. Current main only supports transport/output modes such as `json`, `quiet`, and `json-strict`, while the prompt path still forwards only the normalized prompt to ACP; the requested response-schema contract is useful and aligned, but it is new public API surface that needs maintainer design.

## What This Changes

Add an ACPX-level way to provide a JSON Schema for the agent's final response, with native adapter passthrough or clear fallback and validation errors.

## Best Possible Solution

Define a narrow response-schema contract that preserves `--format` as transport selection, uses native adapter support where available, and returns typed validation or unsupported-capability errors when enforcement cannot be provided.

## Reproduction Assessment

Not applicable; this is a new capability request rather than a broken documented behavior. Source inspection confirms current main has transport JSON modes but no response-schema option or prompt schema pass-through.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is aligned, but the safest solution should first settle flag naming, adapter capability detection, fallback behavior, and machine-readable validation errors.
