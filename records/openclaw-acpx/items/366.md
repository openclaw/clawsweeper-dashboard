## Summary

Keep open. Current main still has transport JSON modes and prompt input handling, but no caller-provided response schema surface for `exec`, `prompt`, or the ACP prompt request; this is useful backend work, but it adds durable CLI/API and adapter fallback semantics that need maintainer design first.

## What This Changes

The issue requests an ACPX CLI/API capability for callers to provide a JSON Schema that constrains the agent's final response separately from ACPX transport formatting.

## Best Possible Solution

Define one protocol-aligned response-contract surface that keeps `--format` for transport output, negotiates native schema enforcement when available, and returns typed unsupported-capability or validation errors under an explicit fallback policy.

## Reproduction Assessment

Not applicable; this is a new response-contract capability request rather than a bug in documented behavior. Source inspection gives high-confidence confirmation that current main lacks the requested CLI/API path.

## Solution Assessment

Unclear until maintainers settle the public contract. Separating response constraints from `--format` is the right direction, but flag naming, schema input form, adapter capability handling, and fallback behavior need design first.
