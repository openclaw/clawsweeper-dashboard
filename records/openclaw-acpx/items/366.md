## Summary

Keep open. Current main has no general response-schema request surface: ACP prompts carry only the session and content blocks, while existing JSON options control transport rendering. The nearby `decision()` helper is deliberately prompt-based and narrow, and its merged PR explicitly leaves first-class structured output for future design.

## What This Changes

Requests a caller-provided JSON Schema to constrain an agent’s final response independently of acpx transport formatting.

## Best Possible Solution

Define one transport-independent response-constraint contract with adapter capability detection, explicit native-versus-validation semantics, typed unsupported/invalid-response errors, and focused supported/unsupported adapter coverage.

## Reproduction Assessment

Not applicable. Current source and documentation establish that general response-schema enforcement is absent, but the issue requests a new capability rather than reporting broken documented behavior.

## Solution Assessment

Unclear. The outcome fits acpx, but maintainers must choose whether the contract guarantees native provider enforcement, post-generation validation, or both before selecting a permanent CLI and API shape.
