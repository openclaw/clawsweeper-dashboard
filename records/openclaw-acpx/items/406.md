## Label Justifications

- `P2`: This is a normal-priority persistence/status bug with a clear workaround for visible output but incorrect stored usage for affected Claude sessions.
- `impact:data-loss`: The reported token breakdown is silently dropped from persisted session records even though the agent provides it.
- `impact:session-state`: The affected fields are part of persisted session state and feed `getStatus().usage` for downstream runtime consumers.

## Summary

Keep open: current main still only persists token breakdowns from `usage_update` payloads, while successful `session/prompt` response usage is displayed by output code but not folded into the session conversation or `getStatus()` state.

## What This Changes

Persist token breakdowns from successful `session/prompt` response `result.usage` into `cumulative_token_usage` and `request_token_usage` when `usage_update` lacks `_meta.usage`.

## Best Possible Solution

Keep the existing persisted/status usage fields and teach the prompt success path to normalize `PromptResponse.usage` into them, with regression coverage for the Claude response shape and no new config or output contract.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the reported payload shape: a `usage_update` carrying only `used` and `size` produces no token usage, and `runPromptTurn()` drops successful `PromptResponse.usage`. I did not run a live Claude adapter session, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes, the narrow fix is to reuse the existing token-usage normalization and persistence surfaces for prompt response usage while preserving the current `usage_update._meta.usage` path. Adding a new setting or output convention would be unnecessary.
