## Label Justifications

- `P2`: The issue describes a real but scoped workflow break for users combining acpx-spawned Claude agents with a live Telegram-connected Claude Code session.
- `impact:message-loss`: The reported one-poller eviction can stop Telegram message delivery for the parent session.
- `impact:session-state`: The spawned adapter can interfere with a separate live parent session's channel state.

## Summary

Keep open: current `main` still launches the built-in Claude ACP adapter with the inherited environment, and the existing docs only cover manual command overrides rather than default isolation or a warning for channel-plugin singleton resources.

## What This Changes

The issue asks acpx to prevent built-in Claude adapter spawns from loading global channel or daemon plugins that can evict an existing Telegram poller, or at least document the hazard.

## Best Possible Solution

Define a maintainer-approved Claude launch isolation behavior that prevents channel or daemon singleton side effects while preserving explicit command overrides and documenting any compatibility tradeoff.

## Reproduction Assessment

No high-confidence runtime reproduction was established in this review. The issue provides concrete live steps, and current source inspection matches the missing-isolation root cause, but I did not run Claude Code with the Telegram plugin and bot token.

## Solution Assessment

Unclear: default isolation is likely the right direction for robustness, but maintainers need to choose the compatibility policy because some users may rely on global Claude plugins in spawned agents.
