## Summary

Keep open: current main still has no Cursor-specific or generic alias mapping for shorthand model labels, and the remaining question is a product decision about shared-label semantics versus native advertised Cursor model IDs.

## What This Changes

The issue asks ACPX to handle Cursor model selectors such as `composer-2.5` more consistently with shared/global model labels accepted by other built-in agents.

## Best Possible Solution

Maintainers should decide the Cursor model-selector contract, then either add a narrow unambiguous alias resolver with docs and regression coverage or document/enforce native Cursor advertised IDs with clearer errors.

## Reproduction Assessment

No, not from this review. The issue includes live Cursor output and current source clearly forwards or validates model IDs without Cursor alias mapping, but I did not run the live Cursor adapter here.

## Solution Assessment

Unclear. Alias normalization would improve portability, but native-ID-only behavior matches current docs and avoids ambiguous shorthand mapping until maintainers choose the contract.
