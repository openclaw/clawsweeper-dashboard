## Summary

Keep open: current main still requires exact advertised model ids for non-Claude adapters and sends Cursor set-model values unchanged, so the shared-label Cursor workflow remains unresolved and needs maintainer direction.

## What This Changes

The issue asks ACPX to handle or clearly define Cursor behavior when shared/global model labels such as `composer-2.5` do not match Cursor's native advertised model ids.

## Best Possible Solution

Have maintainers choose the model contract first, then either add a narrow Cursor-compatible advertised-id alias resolver or improve Cursor docs and errors around native advertised model ids.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No end-to-end live reproduction here, but source inspection matches the reported failure shape: non-Claude `--model` requires exact advertised ids and `set model` sends the requested value unchanged.

## Solution Assessment

Is this the best way to solve the issue? Unclear; alias normalization would improve interoperability, while native-id-only docs/errors would preserve the existing strict contract, so maintainers should choose the direction before implementation.
