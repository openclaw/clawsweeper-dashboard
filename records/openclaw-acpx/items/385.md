## Summary

Keep open: current main still does not implement Cursor shorthand model aliases, and the existing strict non-Claude model contract plus Cursor-specific native-ID behavior leaves a real product/docs decision unresolved.

## What This Changes

The issue asks whether Cursor should accept provider-agnostic global model labels such as `composer-2.5` or document/handle its native model-id requirement.

## Best Possible Solution

Maintainers should choose the Cursor model contract first, then either normalize unambiguous shorthand to advertised Cursor IDs in both `--model` and `set model` paths or document native advertised IDs with clearer errors.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection gives a medium-confidence path because non-Claude `--model` requires exact advertised IDs and `set model` forwards the literal value, matching the reporter's live Cursor output.

## Solution Assessment

Unclear; shorthand alias normalization would improve portability, but native-ID-only docs and errors would preserve the current strict adapter contract. The best path depends on a maintainer decision about shared model-label semantics.
