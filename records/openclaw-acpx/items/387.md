## Summary

Keep open: current main supports `mcpServers` only through global/project config and forwards them to ACP sessions, but it still lacks the requested per-invocation MCP/config-location entry point.

## What This Changes

Request a per-invocation way to provide MCP server definitions without writing global config or `<cwd>/.acpxrc.json`, with documentation of the supported shape as a fallback.

## Best Possible Solution

Add a redaction-aware per-invocation MCP/config source, likely file-path based, with explicit precedence, session-scope semantics, docs, and regression tests for fresh and reused sessions.

## Reproduction Assessment

Not applicable as a feature/config-surface request. Source inspection does confirm the reported gap: current main reads fixed global/project config files and has no per-invocation MCP or config-path flag.

## Solution Assessment

Unclear until maintainers choose the API. A file-path override or MCP-specific config file looks safer than raw inline JSON because `mcpServers` may contain credentials and should not be echoed by default.
