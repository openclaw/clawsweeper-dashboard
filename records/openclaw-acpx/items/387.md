## Summary

Keep open: current main and v0.11.0 still load `mcpServers` only from fixed global/project config files, and a per-invocation MCP or config-path surface needs maintainer product and secret-display decisions.

## What This Changes

The issue requests a per-invocation way to provide MCP servers or override the config path without writing `.acpxrc.json` into the target working tree.

## Best Possible Solution

Choose one stable per-invocation MCP input or config-path override, define its precedence, keep routine config display secret-safe, and cover the new behavior in docs and tests.

## Reproduction Assessment

Yes. Source inspection on current main and v0.11.0 shows MCP servers are forwarded only from fixed config files, with no CLI/env/config-path entry point for session-scoped MCP input.

## Solution Assessment

Unclear until maintainers choose the public surface. A config-path override or MCP file flag could be narrow, but the API name, precedence, and secret-display behavior are durable product choices.
