## Label Justifications

- `impact:data-loss`: The reported token breakdown is silently dropped from persisted session records even though adapters provide it on the prompt response.
- `impact:session-state`: The affected fields are persisted session state and feed `getStatus().usage` for multi-turn/backend consumers.

## Summary

Keep open: current main still persists token breakdowns only from `usage_update` payloads, while prompt response `result.usage` is consumed by output code but dropped before session persistence and `getStatus()`.

## What This Changes

The issue asks for final prompt response `result.usage` to be folded into persisted session token usage so cumulative and per-request status usage are populated for adapters that do not send `_meta.usage` on `usage_update`.

## Best Possible Solution

Normalize successful prompt response usage into the existing `SessionTokenUsage` fields during the prompt success path, preserving the current `usage_update._meta.usage` path and avoiding new config or public field names.

## Reproduction Assessment

Yes, source-level: a `usage_update` carrying only `used` and `size` produces no `SessionTokenUsage`, and successful prompt response usage is reduced to `stopReason` before session persistence. I did not run a live adapter session, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes: reusing the existing token normalization and persisted/status surfaces for prompt response usage is the narrowest maintainable fix. A new flag, config option, or output convention would add unnecessary product surface.
