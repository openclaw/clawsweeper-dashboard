## Summary

Keep open: current main still appears to drop the parent-agent `-s` value for the explicit `prompt` subcommand, while docs and neighboring subcommands treat named-session selection as supported CLI surface.

## What This Changes

This issue reports that `acpx <agent> -s <name> prompt ...` misses the named session and falls back to cwd-default lookup with a misleading no-session error.

## Best Possible Solution

Reuse the existing parent-option fallback for `handlePrompt`, add a focused regression test for `acpx <agent> -s <name> prompt ...`, and leave broader CLI conventions unchanged.

## Reproduction Assessment

Yes, there is a high-confidence reproduction path from the reporter transcript and source: create a named session, then run `acpx <agent> -s <name> prompt ...`. I did not run it locally, but current `handlePrompt` reads only `flags.session` while the parent-option fallback exists for sibling subcommands.

## Solution Assessment

Yes, the narrowest maintainable fix is to make prompt use the existing `resolveSessionNameFromFlags(flags, command)` helper and cover the explicit prompt form with a regression test.
