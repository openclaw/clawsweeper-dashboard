## Summary

Keep open: current main still forwards `mcpServers` only from the fixed global/project config files, and the requested per-invocation MCP or config-location entry point remains a product-surface decision.

## What This Changes

The issue requests a per-invocation MCP-server or config-path mechanism so headless callers can scope MCP servers to one acpx session without touching global config or the working tree.

## Best Possible Solution

Choose one minimal stable per-invocation entry point, define precedence against global/project config, preserve secret-safe display behavior, and cover the forwarding path in config, CLI, docs, and integration tests.

## Reproduction Assessment

Yes. Source inspection on current main shows MCP servers can be forwarded from fixed config files, but there is no CLI, env, or config-path entry point for session-scoped MCP input.

## Solution Assessment

Unclear. A config-path override or MCP-config file flag could be narrow, but maintainers need to choose the durable API and secret-display behavior first.
