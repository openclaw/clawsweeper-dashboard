## Label Justifications

- `P1`: The report describes a live security boundary issue where repository config can influence local credential file handling.
- `impact:security`: The affected behavior involves credentials, local SSH key paths, and lower-trust repository configuration.

## Summary

Keep open: current main still loads repo-local config as untrusted but applies KubeVirt `sshKey` and `sshPublicKey` without checking that trust bit, and the backend can read path-like public-key values before VM manifest creation. I found no merged or open PR that already owns this KubeVirt key-path boundary.

## What This Changes

Repository-local Crabbox config can set KubeVirt SSH key path fields that should be restricted to trusted user configuration.

## Best Possible Solution

Gate KubeVirt operator-owned SSH key paths to trusted config sources while preserving safe repo-level KubeVirt routing fields and adding regression tests for both trusted and untrusted sources.

## Reproduction Assessment

Yes. The source-level path is high confidence: untrusted repo config is loaded after trusted user config, the KubeVirt merge block accepts SSH key path fields without checking `trusted`, and the backend can read file-form `sshPublicKey` values before manifest creation.

## Solution Assessment

Mostly yes. Restricting KubeVirt SSH key path fields to trusted config is the narrow fix, but maintainers should confirm whether untrusted repo config may still carry validated inline public keys or should ignore `sshPublicKey` entirely.
