## Summary

Keep open: current main does not implement Azure Windows snapshot forks, and this useful feature PR still has a blocking direct-snapshot guard gap that can create misleading checkpoints for Azure ephemeral OS-disk leases.

## What This Changes

The PR adds direct Azure Windows OS-disk checkpoint creation and snapshot-backed forks with source deallocate/restart, credential rehydration, docs, and Go tests.

## Best Possible Solution

Mirror the existing Worker ephemeral-disk rejection in the direct Go snapshot path with regression coverage, then land only after maintainer acceptance of the deallocate/restart and credential-rehydration contract.

## Reproduction Assessment

Yes for the review finding by source inspection: the new direct Windows Azure snapshot path can accept an ephemeral OS disk because capability routing ignores OS-disk mode and CreateOSDiskSnapshot only checks managedDisk.id. I did not run live Azure in this read-only review.

## Solution Assessment

No as-is. The maintainable fix is to mirror the Worker Azure guard for diffDiskSettings.option == Local before deallocation/snapshot creation, plus focused Go tests for ephemeral rejection and managed-disk acceptance.
