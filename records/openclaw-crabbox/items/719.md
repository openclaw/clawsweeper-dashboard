## Label Justifications

- `P1`: The issue affects a documented bootstrap trust path and can lead to root package installation in managed Linux leases or images if the key delivery path is compromised.
- `impact:security`: The report is about supply-chain integrity for trusted APT key material used before installing privileged packages into Crabbox-managed Linux environments.

## Summary

Keep this open: current main and the latest release still trust live Google Chrome APT signing-key bytes in the Linux browser bootstrap paths, and the related open items do not fully own the Worker cloud-init path covered here.

## What This Changes

The issue reports that Linux browser bootstrap paths trust Google Chrome APT signing-key bytes without reviewed digest verification.

## Best Possible Solution

Add maintainer-reviewed digest or fingerprint verification for the Google key in both Linux browser bootstrap paths, scope the Worker Chrome repo with signed-by, add mismatch-refusal tests, and coordinate any shared helper expansion with https://github.com/openclaw/crabbox/issues/687.

## Reproduction Assessment

Yes for source-level reproduction: current main still downloads Google's key into the Worker bootstrap and developer-image keyring path without any digest or fingerprint check. I did not run the privileged bootstrap itself.

## Solution Assessment

Yes in shape: pin-and-verify plus signed-by scoping is the narrow maintainable repair. The maintainer-owned part is choosing reviewed key material and a rotation process.
