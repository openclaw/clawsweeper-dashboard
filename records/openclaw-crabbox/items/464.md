## Label Justifications

- `P0`: The report describes a current security-boundary bypass that can expose an inherited Azure Dynamic Sessions bearer token to a repository-selected endpoint.
- `impact:security`: The issue concerns a trusted credential being sent to a lower-trust destination selected by repository configuration.
- `impact:auth-provider`: The affected path is Azure Dynamic Sessions provider authentication and endpoint routing.

## Summary

Keep open: current main still appears to allow Azure Dynamic Sessions to combine a repository-configured endpoint with inherited env or Azure CLI bearer-token auth, and no merged fix or shipped release currently covers this exact source-binding gap.

## What This Changes

The issue asks Crabbox to source-bind Azure Dynamic Sessions endpoint configuration before inherited Azure Dynamic Sessions bearer tokens are sent to that endpoint.

## Best Possible Solution

Extend the existing credential-destination provenance guard to Azure Dynamic Sessions, preserve explicit endpoint approval through env or flag, and add focused regression coverage for repo endpoint plus inherited token.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: repository config can set `azureDynamicSessions.endpoint`, Azure Dynamic Sessions is absent from the provenance guard, and env/Azure CLI bearer tokens are sent to the configured endpoint.

## Solution Assessment

Unclear until maintainer security review confirms the compatibility policy. The narrow likely fix is to reuse the existing source-binding provenance pattern for Azure Dynamic Sessions rather than adding a parallel mechanism.
