## Summary

Keep this PR open for review: it is a maintainer-authored draft with useful Azure Windows ARM64 work, but the worker-side explicit-image guard does not match the CLI/Azure default-image logic and can accept a legacy Linux default as if it were an explicit ARM64 Windows image.

## What This Changes

The PR adds Azure native Windows ARM64 lease support across CLI and worker validation, rejects Azure Windows ARM64 WSL2, updates Azure VM candidate selection, repairs WSL2 interop bootstrap, threads broker Azure image defaults, and updates docs/tests.

## Best Possible Solution

Land the Azure Windows ARM64 support after the worker image validation uses the same known-default image set as the CLI/Azure provisioning path and keeps focused CLI and worker coverage for the rejected default-image cases.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear path: pass Azure Windows ARM64 through worker leaseConfig with a legacy Azure Linux default image and the new helper returns it as explicit, while Azure provisioning still treats that same image as a default Linux image.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes, but the worker explicit-image check should share or mirror the full Azure default-image set before this support lands; otherwise the CLI and broker enforce different contracts.
