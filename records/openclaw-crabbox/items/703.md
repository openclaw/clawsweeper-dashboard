## Summary

Keep open: current main and v0.33.0 still have the raw AWS instance-ID stop path, empty expected identity validation can no-op, and release can reach EC2 TerminateInstances without an AWS ownership gate. This is a source-reproducible security boundary issue, not an implemented or duplicate cleanup item.

## What This Changes

The issue reports that AWS direct `crabbox stop --provider aws --id i-*` can terminate a describable EC2 instance without verifying Crabbox ownership tags.

## Best Possible Solution

Add AWS-owned-resource validation before raw-ID release and again before `DeleteServer`, preserving canonical Crabbox-tagged lease release while rejecting or explicitly adopting unowned EC2 IDs.

## Reproduction Assessment

Yes. Source inspection shows `crabbox stop --provider aws --id i-*` can resolve a raw EC2 instance through `GetServer`, skip identity validation when no expected identity is present, and reach `TerminateInstances`; I did not run live AWS.

## Solution Assessment

Yes on the remediation direction: fail closed before destructive AWS direct release unless the resolved server has canonical Crabbox ownership evidence. The exact legacy raw-ID or adoption behavior needs maintainer approval because it changes a destructive compatibility path.
