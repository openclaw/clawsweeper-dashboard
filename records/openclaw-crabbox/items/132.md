Full review comments:

- **[P1] Read the advertised W&B login credential:** `internal/providers/wandb/client.go:149-157`
  - body: The provider docs and error text tell users that `wandb login` is enough, but `resolveAuth` only checks environment/config values and never reads the netrc credential that login writes. Users who follow the zero-new-credential path fail before the first API call, so either read the W&B credential source or remove that claim consistently.
  - confidence: 0.94
- **[P2] Wire the documented YAML credential:** `internal/cli/config.go:1017-1019`
  - body: `resolveAuth` and the docs list `cfg.wandb.apiKey` as a config-file credential source, but `fileWandbConfig` has no `apiKey` field and `applyFileConfig` only copies image/lifetime settings. A `wandb.apiKey` YAML value is ignored, so wire it through or remove the config-file credential claim.
  - confidence: 0.9
- **[P2] Forward or reject allowed run environment:** `internal/providers/wandb/backend.go:248-250`
  - body: This only rejects `EnvSummary`, but repo-config `env.allow` can populate `req.Env` with `SummaryRequested=false`; `Run` then calls `Acquire` without `EnvironmentVars`, silently dropping variables the run pipeline selected for the remote command. Forward `req.Env` to W&B start or reject any non-empty env map with a clear unsupported message.
  - confidence: 0.87
- **[P2] Return mapped API errors as CLI exit errors:** `internal/providers/wandb/client.go:385-389`
  - body: `mapRPCError` records 77/69/124 in `wandbAPIError`, but `cmd/crabbox` only preserves process exit codes for `cli.ExitError`; these W&B API failures otherwise exit as 1. Convert mapped API errors at backend boundaries or make them visible through `cli.AsExitError` so the documented sysexits behavior is real.
  - confidence: 0.86
