## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Normalize proto status before setting Ready:** `internal/providers/wandb/backend.go:132-133`
  - body: `client.Status` returns the generated enum string (`SANDBOX_STATUS_RUNNING`), but this code lowercases it to `sandbox_status_running` and then compares it with `running`. A live running sandbox will therefore render `Ready: false`; normalize the enum name or compare against the proto running value, and add a test that uses `SANDBOX_STATUS_RUNNING`.
  - confidence: 0.93
- **[P2] Honor keep-on-failure before stopping sandboxes:** `internal/providers/wandb/backend.go:78-82`
  - body: This unconditional stop runs before the code checks `execErr` or non-zero exit status, so `--keep-on-failure` cannot preserve a newly acquired W&B sandbox for debugging. Match the other delegated providers by deferring stop behind a `shouldStop` flag and calling the shared failure handler before returning the error.
  - confidence: 0.91
- **[P2] Make the documented config API key path real:** `internal/cli/config.go:1016-1019`
  - body: The docs and `resolveAuth` say `cfg.wandb.apiKey` is a config-file credential source, but `fileWandbConfig` has no `apiKey` field and `applyFileConfig` only copies image and lifetime settings. Users who put `wandb.apiKey` in `crabbox.yaml` will still fail auth when the env vars are absent; either parse that field or remove the documented config-file credential path.
  - confidence: 0.89
- **[P2] Reject or forward allowed run environment values:** `internal/providers/wandb/backend.go:230-232`
  - body: `RunRequest.Env` can be populated by `--allow-env` without `EnvSummary`, but this provider neither forwards it to `Acquire`/`Exec` nor rejects it unless summary output was requested. That silently drops selected environment values for W&B runs; either pass them through where the API supports it or reject any non-empty `req.Env` with a clear message.
  - confidence: 0.84
