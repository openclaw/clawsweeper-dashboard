## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Read the advertised W&B login credential:** `internal/providers/wandb/client.go:149-157`
  - body: The provider's main value proposition is that users who already ran `wandb login` can use the key in `~/.netrc`, but `resolveAuth` only checks environment/config fields and then tells users to run `wandb login`. A normal W&B user with only the login-created credential still fails before any API call, so either read that credential source or remove the claim consistently.
  - confidence: 0.92
- **[P2] Wire the documented config credential:** `internal/cli/config.go:1016-1019`
  - body: The docs and auth comments list `cfg.wandb.apiKey` as a config-file credential source, but the YAML shape only accepts `defaultImage` and `maxLifetimeSeconds`, and `applyFileConfig` never assigns `Wandb.APIKey`. Users who follow that documented config path get an ignored key and a missing-credential error.
  - confidence: 0.9
- **[P2] Forward or reject allowed environment variables:** `internal/providers/wandb/backend.go:248-250`
  - body: `RunRequest.Env` is populated by the existing `--allow-env`/profile machinery, but this backend only rejects the summary flag and otherwise runs without forwarding those values. That silently drops secrets or settings users intentionally allowed, so pass them to the sandbox when supported or fail clearly when `len(req.Env) > 0`.
  - confidence: 0.88
- **[P2] Return mapped API failures as CLI exit errors:** `internal/providers/wandb/client.go:385-389`
  - body: `mapRPCError` computes documented exit codes like 77, 69, and 124, but it returns `*wandbAPIError`; the CLI process only honors `cli.ExitError`, so these failures still exit 1. Convert mapped API failures to the shared exit-error surface or wrap them so the documented exit codes are actually observable.
  - confidence: 0.86
