## Summary

This PR should stay open: current main does not already contain a W&B/CoreWeave provider, and the latest head still has source-visible blockers in the advertised credential path, config loading, environment forwarding, and failure-retention behavior.

## What This Changes

The PR adds a native Go W&B/CoreWeave delegated-run provider with vendored v1beta2 gRPC protos, provider registration, config/env plumbing, docs, unit tests, and an opt-in live smoke test.

## Best Possible Solution

Align W&B credential loading, config parsing, env handling, and failure-retention semantics with existing delegated-run providers, then merge only if maintainers accept W&B/CoreWeave as a core provider despite the beta API surface.

## Reproduction Assessment

Yes for the review findings: the latest PR head source shows resolveAuth omitting the advertised ~/.netrc path, auto-stop ignoring KeepOnFailure, RunRequest.Env accepted but ignored, and YAML config omitting apiKey. I did not run live service tests in this read-only review.

## Solution Assessment

No, not yet: the native gRPC direction is plausible and responds to the shim review, but the implementation needs to match advertised credentials and existing delegated-run semantics before merge. The final decision on whether W&B/CoreWeave belongs in core remains a maintainer product call.

Overall confidence: 0.86

Full review comments:

- **[P1] Read the advertised W&B login credential:** `internal/providers/wandb/client.go:148-157`
  - body: The README/docs sell this as working with the API key already in `~/.netrc` from `wandb login`, but `resolveAuth` only checks `CRABBOX_WANDB_API_KEY`, `cfg.Wandb.APIKey`, and `WANDB_API_KEY`. A user who has only run `wandb login` still gets the missing-key error, so either read the supported W&B credential source or remove that claim consistently.
  - confidence: 0.88
- **[P2] Honor keep-on-failure before stopping sandboxes:** `internal/providers/wandb/backend.go:78-82`
  - body: `Run` stops any newly acquired sandbox whenever `--keep` is false, before checking whether `Exec` failed and without consulting `KeepOnFailure`. That makes `crabbox run --provider wandb --keep-on-failure ...` discard the sandbox users asked to keep for debugging; use the delegated-run failure helper or equivalent before deciding to stop.
  - confidence: 0.95
- **[P2] Forward or reject per-run environment variables:** `internal/providers/wandb/backend.go:230-232`
  - body: The CLI can populate `RunRequest.Env`, but this backend only rejects `EnvSummary` and otherwise accepts non-empty env while never passing it into `Acquire` or `Exec`. That silently drops allowed env/profile variables for W&B runs; either forward them to the sandbox or fail fast with a clear unsupported-env error.
  - confidence: 0.9
- **[P2] Make the documented config API key path real:** `internal/cli/config.go:1016-1019`
  - body: The docs list `cfg.wandb.apiKey` as a credential source and `resolveAuth` checks `cfg.Wandb.APIKey`, but `fileWandbConfig` has no `APIKey` YAML field and `applyFileConfig` cannot load it. Users following the documented config-file path will still get a missing-key error; add the field and test it, or remove that path from docs and auth precedence.
  - confidence: 0.9
- **[P3] Cache the gRPC client after dialing:** `internal/providers/wandb/backend.go:178-182`
  - body: `api()` returns a newly created client without assigning it to `b.client`, so calls like Doctor's probe plus List create multiple `grpc.ClientConn`s despite the one-connection lifecycle described by the client. Cache the successful client or change the design comment to avoid repeated dials and idle keepalive sockets.
  - confidence: 0.84
