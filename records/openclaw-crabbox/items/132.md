Full review comments:

- **[P1] Read the advertised W&B login credential:** `internal/providers/wandb/client.go:149-157`
  - body: The PR and docs say users can rely on the API key written by `wandb login` in `~/.netrc`, but `resolveAuth` only checks env/config values and then fails. A normal W&B user without an exported `WANDB_API_KEY` still cannot use the provider; either read the W&B netrc credential or remove that claim consistently.
  - confidence: 0.93
- **[P2] Wire the documented config credential:** `internal/cli/config.go:1016-1019`
  - body: `resolveAuth` treats `cfg.Wandb.APIKey` as a credential source, and the docs call out `cfg.wandb.apiKey`, but the YAML file config struct has no `apiKey` field and `applyFileConfig` never assigns it. Users who put the documented key in config will still get a missing-key error.
  - confidence: 0.94
- **[P2] Forward or reject allowed environment variables:** `internal/providers/wandb/backend.go:57-61`
  - body: `RunRequest.Env` is populated from `--allow-env` and env profiles, and the W&B acquire request already has `EnvironmentVars`, but `Run` never passes `req.Env` when starting a sandbox and only rejects the summary flag. Commands that depend on allowed env vars will run without them, so forward them or fail clearly when the map is non-empty.
  - confidence: 0.9
- **[P2] Return CLI exit errors for mapped API failures:** `internal/providers/wandb/client.go:385-389`
  - body: The docs and comments promise sysexits-style W&B API exit codes, but `mapRPCError` returns `*wandbAPIError`; the top-level CLI only preserves custom process codes for `cli.ExitError`, so these failures will print and exit as 1. Convert the mapped API failures into the shared exit-error path before returning them to the CLI.
  - confidence: 0.88
