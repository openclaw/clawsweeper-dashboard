## What This Changes

The branch adds a `wandb` delegated-run provider using an embedded Python `cwsandbox` shim, with config, docs, registration, changelog, and provider tests.

## Best Possible Solution

Land the provider only after credential precedence is enforced through backend checks and shim env, exec preserves provider failures, conflicts are resolved, and redacted Crabbox CLI live proof is attached.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: set only `CRABBOX_WANDB_API_KEY`; config fills `cfg.Wandb.APIKey`, but Run/Doctor check only `WANDB_API_KEY` and the shim child receives the environment unchanged.

## Solution Assessment

No. The provider shape is plausible, but this implementation should route `cfg.Wandb.APIKey` consistently into checks and the child env, then preserve shim/API errors separately from remote command exits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Route configured W&B key through auth checks:** `internal/providers/wandb/backend.go:42-43`
  - body: `Run` rejects execution unless `WANDB_API_KEY` is present in the process environment, and `Doctor` repeats that direct check. The config layer supports `CRABBOX_WANDB_API_KEY` as the higher-precedence source, so users who set only the documented prefixed variable fail before any API call despite having a valid configured key.
  - confidence: 0.96
- **[P1] Pass the configured key to the shim environment:** `internal/providers/wandb/client.go:160`
  - body: The shim subprocess is launched with `os.Environ()` only, while the shim itself reads `WANDB_API_KEY`. When credentials come from `CRABBOX_WANDB_API_KEY` or any config path that populates `cfg.Wandb.APIKey` without exporting `WANDB_API_KEY`, list/status/stop and the fixed run path will still fail in the child process.
  - confidence: 0.95
- **[P2] Preserve shim failures during exec:** `internal/providers/wandb/client.go:255-258`
  - body: `Exec` treats every non-zero shim exit as the remote command's exit code, but the same path also covers auth failures, import errors, bad sandbox IDs, and SDK exceptions before the command runs. Those should remain provider/shim errors so users do not see infrastructure failures reported as ordinary sandbox exits.
  - confidence: 0.88
