## Summary

Keep open: current main and the latest release still preserve non-admin GCP selector fields and use them in brokered GCP provider calls, and related search did not find a replacement PR. The fix should get maintainer security review because fail-closed gating or allowlists can change existing GCP workflows.

## What This Changes

The issue reports that non-admin brokered GCP lease creation can override GCP project, image, network, subnet, tags, and service-account selectors used under coordinator credentials.

## Best Possible Solution

Keep this open for maintainer security review, define the GCP selector authorization policy, then add provider-boundary regression tests for non-admin requests and any approved admin or allowlist path.

## Reproduction Assessment

Yes. A source reproduction path is clear: a non-admin lease request can carry GCP selector fields through leaseConfig into GCP provider creation and firewall code; no live GCP exploitation was attempted.

## Solution Assessment

Unclear. Admin-gating the fields is directionally narrow, but maintainers need to choose the compatibility policy for existing per-request GCP project/network/image/tag/service-account workflows before implementation.
