- `P0`: The issue describes a plausible non-admin-to-provider-authority security boundary bypass in brokered GCP lease creation.
- `impact:security`: The issue is about coordinator authorization boundaries and privileged provider credential use.

## Summary

Keep open: current main and latest release v0.33.0 still preserve non-admin GCP selector fields and use them in brokered GCP provider calls, with no replacement fix PR found.

## What This Changes

The issue reports that non-admin brokered GCP lease requests can set privileged GCP project, network, image, tags, and service-account selectors that the coordinator later uses with broker credentials.

## Best Possible Solution

Keep this open for maintainer security review, define the GCP selector authorization policy, then gate non-admin selectors or add explicit allowlists with regression tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: a non-admin lease request can carry GCP selector fields through leaseConfig into GCP provider creation and firewall code; no live GCP exploitation was attempted.

## Solution Assessment

Unclear. Admin-gating the fields is directionally narrow, but maintainers need to choose the compatibility policy for existing per-request GCP project, network, image, tag, and service-account workflows before implementation.
