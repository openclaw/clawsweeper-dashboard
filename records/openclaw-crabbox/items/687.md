## Label Justifications

- `P1`: The issue affects a documented default image-construction trust path and can lead to root package installation in managed developer images if the key and repository origin are compromised.
- `impact:security`: The report is about supply-chain integrity for trusted key material and packages installed into Crabbox-managed images.

## Summary

Keep this issue open: current main and the latest release still use the documented Linux image bootstrap path without pinned digest or fingerprint verification for third-party APT key material, and no canonical fixing PR is open.

## What This Changes

The issue requests pinning or fingerprint verification for third-party APT signing keys used by the Linux developer image bootstrap.

## Best Possible Solution

Add reviewed SHA-256 and/or fingerprint verification for each third-party APT key before dearmoring, fail closed without replacing existing keyrings on mismatch, cover mismatch tests, and document key rotation.

## Reproduction Assessment

Yes for source-level reproduction: current main downloads repository keys and dearmors them without a digest or fingerprint check, and the existing test harness shows arbitrary fake key bytes are accepted. I did not run the privileged image bootstrap itself.

## Solution Assessment

Yes in shape: the recommended pin-and-verify approach is the narrow maintainable fix. A maintainer-owned patch should choose the reviewed key material and rotation process before implementation proceeds.
