## Summary

Keep this open: current main still lets untrusted repo-local XCP-Ng config override `username` and `password`, while adjacent fields already use the trusted-config guard.

## What This Changes

The issue requests trusted-only handling for XCP-Ng username/password values so repository-local `.crabbox.yaml` cannot replace operator credentials.

## Best Possible Solution

Guard XCP-Ng username/password with the existing trusted-config check and add a focused regression test that attempts repo-local credential overrides.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: load trusted user XCP-Ng credentials, then load repo-local `.crabbox.yaml` with `xcpNg.username` and `xcpNg.password`; those assignments are currently ungated.

## Solution Assessment

Yes. Applying the existing `trusted &&` guard to XCP-Ng username/password and expanding the current XCP-Ng repo-config regression test is the narrowest maintainable repair.
