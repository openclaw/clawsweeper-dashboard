## What This Changes

The branch adds a built-in `vercel-sandbox` delegated-run provider with config/env/flags, a Node SDK bridge, lifecycle/sync/list/status/stop/cleanup code, docs, provider metadata, and guarded live-smoke coverage.

## Best Possible Solution

Land the provider after fixing retained resume and command streaming, preserving env-only credential handling, and attaching redacted live Vercel proof for create, sync, reuse, nonzero exit, status/list, and stop.

## Reproduction Assessment

Yes for the review findings by source inspection: the bridge hard-codes `resume: false` for sandbox retrieval and buffers stdout/stderr until `runCommand` completes. I did not run a live Vercel Sandbox because this review is read-only and the PR lacks live credentials/proof.

## Solution Assessment

No: the provider direction is plausible, but the current implementation is not the best mergeable form until retained reuse, output streaming, and live proof are addressed. A safer path is to keep status reads non-mutating while allowing run/sync paths to resume retained sandboxes and stream command output through the bridge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Allow retained sandboxes to resume on reuse:** `internal/providers/vercelsandbox/bridge_cli.go:147`
  - body: `getSandbox` always passes `resume: false`, and the run/sync/upload/exec paths reuse this helper. A kept Vercel sandbox that has idled or been stopped will not be resumed for `run --id`, so retained lease reuse can fail instead of doing the documented implicit resume; split status/no-mutation reads from run paths or allow resume where execution is intended.
  - confidence: 0.86
- **[P2] Stream bridge command output before completion:** `internal/providers/vercelsandbox/bridge_cli.go:227`
  - body: The bridge sends stdout/stderr into capture buffers and only writes them back in the final JSON response after `sandbox.runCommand` returns. That makes long-running delegated commands appear silent until they finish and does not satisfy Crabbox's normal streamed run output contract; the bridge needs a streaming frame/side-channel protocol or another way to forward chunks as they arrive.
  - confidence: 0.82
