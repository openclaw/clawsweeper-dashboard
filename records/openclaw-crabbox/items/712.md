## What This Changes

The issue reports that credential-bearing user config rewrites use a non-atomic write that can truncate config.yaml if interrupted.

## Best Possible Solution

Add an atomic same-directory temp-file, chmod 0600, write, sync, close, rename, and cleanup path for writeUserFileConfig while preserving the existing config schema and happy-path behavior.

## Reproduction Assessment

Yes, as a source-reproducible issue rather than a captured live interruption: current main uses os.WriteFile for the authoritative config and readFileConfig hard-fails on malformed non-empty YAML. The exact torn-write timing was not executed in this read-only review.

## Solution Assessment

Yes, routing writeUserFileConfig through the existing atomic-replace pattern is the narrowest maintainable fix. A backup/recovery UX can be considered later, but it is not required to remove the primary truncation failure mode.
