Full review comments:

- **[P1] Strip pond command paths before dispatch:** `internal/cli/cli_kong.go:516-520`
  - body: `pond peers` strips the Kong command path before parsing, but `pond connect` and `pond release` still pass raw passthrough args through. If Kong includes `pond connect` or `pond release` in `c.Args`, connect treats `pond` as the name and release normalizes the whole path, so the commands miss or target the wrong pond.
  - confidence: 0.9
- **[P1] Start export forwards before returning:** `internal/cli/pond_mesh.go:313-314`
  - body: Starting `runPondMeshForwards` in a goroutine and then returning is not daemonization: when the Crabbox process exits, the goroutine can be torn down before it starts or supervises any ssh process. That lets `eval $(crabbox pond connect demo --export)` receive endpoints that are not actually listening.
  - confidence: 0.88
- **[P2] Persist Railway pond membership before advertising bridge support:** `internal/providers/railway/provider.go:23`
  - body: Railway is now advertised as `FeatureURLBridge`, but the Railway backend never writes a local pond claim and its list/status labels only include the project id. Since `resolvePondPeers` discovers members from pond claims, Railway services will not appear in `pond peers` or `pond release` despite being documented as live bridge peers.
  - confidence: 0.82
