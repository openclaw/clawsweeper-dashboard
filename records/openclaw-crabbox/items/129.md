Full review comments:

- **[P1] Strip pond command paths before dispatch:** `internal/cli/cli_kong.go:516-520`
  - body: `pond peers` strips the Kong command path before calling the handler, but `pond connect` and `pond release` still pass raw passthrough args. When Kong includes `pond connect ...` or `pond release ...` in `c.Args`, the handlers parse the command words as the pond name, so the new commands can target the wrong pond or report empty results.
  - confidence: 0.91
- **[P1] Start export forwards before returning:** `internal/cli/pond_mesh.go:327-329`
  - body: The `--export` path launches `runPondMeshForwards` in a goroutine and then returns, which exits the Go process and can kill that goroutine before it starts or supervises the SSH children. This leaves `eval $(crabbox pond connect ... --export)` with exported loopback endpoints that may already be dead.
  - confidence: 0.9
- **[P2] Persist Railway pond membership before advertising bridge support:** `internal/providers/railway/provider.go:23`
  - body: Railway now advertises `FeatureURLBridge`, but the backend rejects warmup and has no claim-writing path for existing service IDs. Since `pond peers` and `pond release` discover members only from local pond claims, Railway peers will be documented as supported while remaining undiscoverable in normal use.
  - confidence: 0.84
- **[P2] Use the real HuJSON standardizer for ACL row checks:** `internal/cli/doctor_pond.go:82-94`
  - body: `normalizeHuJSON` is not string-aware and only removes trailing commas when the comma is immediately adjacent to `]` or `}`. Real HuJSON policies with comments, URLs, or whitespace before the closing bracket can still fail the row-present check, causing false doctor failures or duplicate ACL writes even though the policy already contains the pond rule.
  - confidence: 0.8
