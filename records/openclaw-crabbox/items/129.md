## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Strip the pond command path before dispatch:** `internal/cli/cli_kong.go:517-520`
  - body: The new nested `pond connect` and `pond release` handlers pass raw Kong passthrough args into the subcommand implementations. Other nested passthrough handlers strip their command path; without doing the same here, `crabbox pond connect demo` can be parsed as pond name `pond`, and release can normalize the whole command path instead of the requested pond.
  - confidence: 0.89
- **[P1] Reuse the robust HuJSON parser for ACL checks:** `internal/cli/doctor_pond.go:99`
  - body: `normalizeHuJSON` only removes literal `,]`/`,}` trailing commas, but real policy files commonly format them as `,\n]` or `,\n}`. That makes `pondACLRowPresent` return false for an existing rule, so doctor can fail and `pondACLEnsure` can unnecessarily rewrite or duplicate tailnet policy rows.
  - confidence: 0.9
- **[P2] Render pending peers in the doctor matrix:** `internal/cli/pond_bridge_doctor.go:151-156`
  - body: `buildPondReachabilityMatrix` counts peers under `pending`, but `observedTransports` and the renderer only know `tailnet`, `url`, `ssh`, and `none`. Pending members therefore disappear from the displayed breakdown and rows, which under-reports provisioning or missing-endpoint peers during rollout.
  - confidence: 0.87
- **[P3] Canonicalize provider aliases before filtering claims:** `internal/cli/pond_bridge.go:521-522`
  - body: Claims are stored with canonical provider names, but this filter compares the raw `--provider` value against `claim.Provider`. Alias inputs can silently exclude valid pond members, making `pond peers` or `pond connect` look empty for an otherwise valid pond.
  - confidence: 0.82

## Security Review

Status: needs_attention
