Full review comments:

- **[P1] Strip pond command paths before dispatch:** `internal/cli/cli_kong.go:517-520`
  - body: `pond peers` strips the Kong command path, but `pond connect` and `pond release` pass raw `c.Args`. With passthrough subcommands this can make the handlers parse `pond` or `pond release ...` as the pond name instead of the user-provided name, so the new commands operate on the wrong label or report no leases.
  - confidence: 0.9
- **[P1] Persist pond on all SSH provider claims:** `internal/cli/pond_bridge.go:519-523`
  - body: `filterClaimsForPond` requires `claim.Pond`, but several SSH-mesh providers advertised by this PR still create claims through pond-unaware helpers. Those leases are then filtered out before backend listing, so `pond peers`, `pond connect`, and `pond release` silently miss providers such as RunPod, exe.dev, namespace, and sprites.
  - confidence: 0.88
- **[P2] Use a real HuJSON normalizer for ACL checks:** `internal/cli/doctor_pond.go:112-124`
  - body: The comment-stripping loop mutates the `range` index, which does not skip the rest of the line, and the trailing-comma replacer only handles adjacent `,]`/`,}`. Real HuJSON policy sections with comments or whitespace-formatted trailing commas can still fail parsing, causing false missing-policy reports or duplicate ACL rewrites.
  - confidence: 0.86
- **[P2] Render pending peers in the doctor matrix:** `internal/cli/pond_bridge_doctor.go:151-156`
  - body: The matrix counts `pending` peers in `Breakdown`, but `observedTransports` only emits tailnet, url, ssh, and none. Pending members therefore disappear from the rendered transport breakdown and reachability rows, which misleads operators while endpoints are still provisioning.
  - confidence: 0.87
