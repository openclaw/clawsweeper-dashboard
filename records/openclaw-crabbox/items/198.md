## Summary

Keep open: current main still has the old Islo status/delete behavior, and this PR is a focused provider repair with tests and live proof. I found no blocking code-review findings, but maintainers should be aware that dropping legacy ready aliases is a small compatibility tradeoff for any custom Islo-compatible base URL.

## What This Changes

The PR updates the Islo provider to treat only `running` as ready, fail fast on terminal statuses, surface SSE error events, delete sandboxes via raw HTTP, and align Islo docs/tests with the live API contract.

## Best Possible Solution

Land the focused Islo provider fix once maintainers accept the current `api.islo.dev` contract as authoritative; preserve the legacy aliases only if custom Islo-compatible base URL compatibility is a supported promise.

## Solution Assessment

Yes, the proposed fix is narrow and provider-local: raw DELETE matches the existing direct HTTP pattern for other Islo endpoints, and the status/SSE changes are covered by focused tests. The only open judgment is whether legacy status aliases should remain for custom base URLs.
