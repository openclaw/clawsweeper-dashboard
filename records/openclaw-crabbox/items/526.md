## Summary

Keep this PR open: the body redaction is useful and has live TLS proof, but the new helper still stores the API error path unchanged. Semaphore pagination can source that path from a response-controlled `Link` header, so a reflected token can still reach user-visible diagnostics if the follow-up request fails.

## What This Changes

The PR routes Semaphore non-2xx GET, GET-with-headers, and POST errors through a token-redacting helper, adds provider tests, updates Semaphore docs, and adds a changelog line.

## Best Possible Solution

Redact the configured Semaphore token from both stored API error fields, then add a regression test for a token-bearing pagination Link followed by a failing request.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a 2xx project-list response can provide a token-bearing next Link, and a later non-2xx getWithHeaders call formats that unchanged path through semaphoreAPIError.Error. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. The provider-local helper is the right narrow shape, but it needs to redact the stored Path as well as Body before this is a complete fix for response-controlled Semaphore diagnostics.
