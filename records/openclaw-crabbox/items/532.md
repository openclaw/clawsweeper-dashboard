## Summary

Keep open: current main and the latest release still send Azure Dynamic Sessions exec/upload/JSON requests through an unguarded HTTP client, and no merged or open PR appears to own this Azure-specific redirect replay gap.

## What This Changes

The issue reports that Azure Dynamic Sessions command JSON bodies can be replayed to a cross-origin 307/308 redirect target from the configured pool endpoint.

## Best Possible Solution

Add an Azure Dynamic Sessions same-origin redirect guard, modeled on the accepted Cloudflare/RunPod/coordinator helpers, and cover ExecStream 307/308 non-replay plus same-origin redirect behavior in focused tests.

## Reproduction Assessment

Yes. Current source builds the ExecStream JSON body in a bytes.Buffer and sends it through c.httpClient.Do, while the issue body supplies a bounded Go redirect proof; I did not rerun that harness in the read-only checkout.

## Solution Assessment

Yes. The narrow maintainable fix is a cloned/wrapped Azure Dynamic Sessions HTTP client with same-origin CheckRedirect, preserving same-origin redirects and caller policy like the current Cloudflare, RunPod, and coordinator fixes.
