## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Record the actual Docker context for checkpoints:** `internal/cli/checkpoint.go:210`
  - body: This stores `DockerContext` from `server.Labels["runtime_context"]`, but local-container lease creation does not set that label. With a named Docker context and no `DOCKER_HOST`, docker-commit checkpoints record an empty context, so verify/delete can later run against whichever daemon is ambient instead of the daemon that owns the image. Capture the active context/host at creation time or persist it on the lease labels, and add a changed-context regression.
  - confidence: 0.9
- **[P2] Replay the recorded daemon scope when forking:** `internal/providers/localcontainer/provider.go:87-91`
  - body: The fork hook only points `LocalContainer.Image` at the checkpoint image. Even after the checkpoint has recorded a runtime/context/host for verify/delete, `checkpoint fork` loads a normal local-container backend and runs `docker run` against the current ambient daemon, so switching Docker context after creation can make fork fail or use a different image. Carry the recorded scope through the native fork record/provider hook and use it during acquisition.
  - confidence: 0.89
