## Label Justifications

- `P2`: This is a normal-priority security boundary bug with authenticated prerequisites and limited blast radius to deployments using the artifact broker.
- `impact:security`: The issue concerns authorization-scope isolation and signed artifact upload/read URLs across owner/org identities.

## Summary

Keep open: current main still signs brokered artifact object keys under config prefix, owner, and caller prefix only, while the authenticated coordinator identity and docs treat owner/org as the normal authorization scope.

## What This Changes

The issue requests an org-aware namespace for coordinator-brokered artifact upload/read grants so the same owner cannot collide across Crabbox org scopes.

## Best Possible Solution

Add a normalized org segment to newly signed brokered artifact keys, preserve or document the read behavior for existing owner-only URLs through their expected lifetime, and cover same owner/prefix/name across different orgs with Worker tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to send two authenticated `POST /v1/artifacts/uploads` requests with the same owner, prefix, and file name but different orgs; current main passes only owner into the key builder, so both requests target the same key.

## Solution Assessment

Yes, the narrow fix is to make the server-side artifact signing namespace include org as part of the authenticated identity. The remaining maintainer decision is the compatibility path for already-published owner-only keys and URLs.
