## Label Justifications

- `P1`: This is an urgent authorization-boundary issue that lets a lower-share user take interactive control of another operator's WebVNC desktop.
- `impact:security`: The issue is about a `use` share crossing into manage-level interactive desktop control.

## Summary

Keep open: current `main` still has the reported WebVNC control-takeover authorization gap, and I did not find an open or merged PR that already owns this exact fix.

## What This Changes

Require manage-level access for portal WebVNC take-control requests while preserving `use` viewers' ability to observe shared desktops.

## Best Possible Solution

Add a server-side `leaseManageableByRequest` guard in `webVNCTakeControl` and regression coverage for `use` denied, owner/manage allowed, and `use` observation preserved.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a `use`-shared portal viewer with a connected WebVNC viewer id can post to `/portal/leases/<id>/vnc/control`, after which the handler sets that viewer as controller. I did not run Vitest in this read-only review, but the route, role helper, and sink are direct.

## Solution Assessment

Yes, the recommended server-side manage-access guard is the narrowest maintainable fix and matches WebVNC ticket/reset authorization. The observer/viewer path should remain available for `use` shares.
