## Summary

Keep open: current main still lets a GitHub OAuth `redirect_uri` callback origin replace the operator-selected broker during CLI login, and the prior browser-target validation fix does not bind that embedded origin to the selected broker.

## What This Changes

The issue asks Crabbox to reject or explicitly trust GitHub OAuth callback origins that differ from the broker selected for `crabbox login --url`.

## Best Possible Solution

Bind callback-origin canonicalization to the selected broker and reject cross-origin callback origins before browser open, polling, or config write unless maintainers add an explicit trusted-alias mechanism.

## Reproduction Assessment

Yes. Source inspection and the existing auth test show the CLI restarts login against the `redirect_uri` origin and writes config for that origin; I did not run a live OAuth flow.

## Solution Assessment

Yes, with a policy choice. The narrow fix is to reject mismatched callback origins before browser/poll/write, while any trusted alias support should be explicit and maintainer-approved.
