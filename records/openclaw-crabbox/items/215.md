## Summary

This should stay open: current main still has the reported Azure Linux coordinator timeout path, and the Worker only persists the lease record after provider provisioning returns, so late provisioning can still leave users without reliable status or cleanup during the failure window.

## What This Changes

The issue asks Crabbox to make brokered Azure lease acquisition either succeed, report precise provisioning delay, or reliably expose and clean up late-ready leases after a caller timeout.

## Best Possible Solution

Track Azure brokered leases through a pending/provisioning lifecycle before VM creation finishes, then expose clear active/failed/capacity states and make stop/cleanup reliable for any late-created resources.

## Reproduction Assessment

No high-confidence live reproduction was run here. The report includes detailed Azure logs, and current source confirms the 10-minute Azure Linux coordinator timeout plus post-provision lease persistence path, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Yes, the best repair direction is lifecycle tracking rather than only increasing the timeout. Publishing a pending/provisioning lease and guaranteeing cleanup covers both slow Azure capacity and late-ready resources.

## Repair Work Prompt

Fix brokered Azure Linux lease acquisition so a lease ID generated for POST /v1/leases is reliably visible and manageable even when Azure provisioning outlives the CLI acquire window. Keep core provider-neutral: put Azure-specific cleanup/provisioning behavior in the Worker provider adapter and generic lease-state handling in the coordinator. Prefer publishing a pending/provisioning lease record before provider.createServerWithFallback starts, updating it to active when host details are ready, recording failed/capacity/provisioning-delay state on provider failure, and making status/inspect/list/stop by lease ID work for pending or late-created Azure leases. Preserve existing successful brokered leases and cost-limit behavior. Add regression coverage for a slow provider/create timeout path and for cleanup/status visibility; update Azure docs only if user-facing timeout/status behavior changes. Do not add project-specific OpenClaw examples or secrets.
