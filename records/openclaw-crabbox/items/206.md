## Summary

Keep open: the PR is a useful opt-in docker-commit checkpoint path with real Docker proof, but the latest diff still records only the runtime executable, so verify/delete can hit the wrong Docker daemon after a Docker context or DOCKER_HOST change.

## What This Changes

Adds opt-in local-container native checkpoint create/verify/delete via `docker commit`, including a new checkpoint kind, provider capability, docs, unit tests, and a Docker-backed proof test.

## Best Possible Solution

Land this as opt-in create/verify/delete only after checkpoint metadata stores and replays the full Docker daemon scope; keep local-container fork in the separate follow-up PR.

## Reproduction Assessment

Yes from source inspection: the PR stores only `Native.Runtime`, while current local-container code treats runtime, Docker context, and host as the provider scope. A create under one Docker context followed by verify/delete under another will run the recorded executable against the current daemon.

## Solution Assessment

No, not yet. The maintainable path is to persist and replay the full daemon scope for docker-commit checkpoints, or explicitly document a narrower current-context-only contract if maintainers choose that limitation.
