## Summary

Keep this PR open. It adds meaningful local-container checkpoint functionality that is not on current main, but it still has a merge-blocking daemon-scope bug: docker-commit checkpoint records replay only the runtime executable, not the Docker context or effective DOCKER_HOST used to create the image.

## What This Changes

The PR adds opt-in local-container native checkpoint create, verify, and delete via docker commit, with docs, unit coverage, and a Docker proof test.

## Best Possible Solution

Keep the opt-in docker-commit create/verify/delete feature, but persist and replay or strictly validate the full Docker daemon scope before merge, with docs matching the implemented guarantee.

## Reproduction Assessment

Yes for the blocking review finding from source inspection: create persists only the runtime, while verify/delete run that executable under the current ambient Docker context and DOCKER_HOST. A checkpoint created under one daemon scope and verified after switching scope follows the wrong-daemon path.

## Solution Assessment

No. The opt-in native create/verify/delete feature is a reasonable shape, but the implementation should carry the same daemon scope that local-container already uses for lease isolation, or explicitly fail when the recorded scope cannot be honored.
