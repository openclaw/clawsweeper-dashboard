## Summary

Keep open: current main still lacks native local-container checkpoints, and this PR is useful, but the latest head still fails to capture the actual Docker context because it reads an unset `runtime_context` label, so verify/delete can target the wrong daemon after an ambient context change.

## What This Changes

This PR adds an opt-in local-container `docker-commit` native checkpoint kind for create/verify/delete, with checkpoint metadata, provider capability wiring, docs, unit tests, and a dockerproof lifecycle test.

## Best Possible Solution

Land the opt-in create/verify/delete path after recording and replaying the effective Docker context/host from the actual runtime at create time, with a regression that changes ambient daemon scope before verify/delete.

## Reproduction Assessment

Yes. Source inspection shows create reads `server.Labels["runtime_context"]`, while the local-container create path does not set that label, so a non-default Docker context without `DOCKER_HOST` is not preserved for later verify/delete.

## Solution Assessment

No. The feature shape is reasonable, but the daemon-scope fix should capture the actual create-time Docker context or endpoint instead of relying on an unset lease label.
