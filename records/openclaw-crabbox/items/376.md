## Label Justifications

- `P1`: A documented opt-in cleanup mode can let a tag-writing AWS principal induce destructive termination with coordinator credentials.
- `impact:security`: The report concerns a confused-deputy boundary where mutable AWS tags can authorize a higher-privilege destructive action.
- `impact:data-loss`: The concrete outcome is unauthorized EC2 instance termination, which can destroy workloads or state.

## Summary

Keep open: current main still allows AWS orphan sweep delete mode to terminate a stale tag-discovered EC2 candidate without non-forgeable Crabbox ownership proof, and no canonical fix or duplicate report was found.

## What This Changes

The issue asks Crabbox to require stronger ownership evidence than mutable AWS provider tags before AWS orphan sweep delete mode terminates EC2 instances.

## Best Possible Solution

Require provider/coordinator-owned AWS ownership proof before delete mode can call TerminateInstances; keep tag-only stale resources in report or quarantine, document IAM tag protections, and add focused regression coverage.

## Reproduction Assessment

Yes. Source inspection and existing focused tests show a stale crabbox=true AWS machine with no active lease is reported, and delete mode passes its cloud ID to the delete hook; I did not run Vitest because this review must keep the checkout read-only.

## Solution Assessment

Yes on the remediation direction, but the exact durable ownership-proof mechanism needs maintainer design. The narrow maintainable fix should live in AWS/provider cleanup semantics rather than relying on mutable discovery tags as delete authorization.
