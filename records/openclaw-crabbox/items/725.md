## Label Justifications

- `P2`: This is a concrete security-boundary issue in the Code portal, but the affected path is a documented trusted-team compatibility mode with an existing opt-in mitigation.
- `impact:security`: The issue concerns coordinator-origin script execution, portal session authority, and isolation of lease-controlled content.
- `impact:auth-provider`: The exploit path depends on `crabbox_session` being converted into authenticated portal authority for same-origin `/portal` requests.

## Summary

Keep open: current main still allows the documented same-origin Code fallback when `CRABBOX_CODE_ORIGIN_TEMPLATE` is missing or invalid, while the merged isolation work only added an opt-in isolated-origin path and explicitly preserved fallback compatibility.

## What This Changes

The issue reports that Code bridge HTML can fall back to the coordinator portal origin when per-lease Code origin isolation is missing or invalid, allowing lease-controlled JavaScript to act with the viewer’s portal session.

## Best Possible Solution

Adopt a maintainer-approved untrusted-origin boundary for script-capable Code bridge content, either by failing closed without a valid isolated origin or by requiring an explicit compatibility exception, with Worker regression tests and live portal/Code bridge proof.

## Reproduction Assessment

Yes, by source trace: missing or invalid `CRABBOX_CODE_ORIGIN_TEMPLATE` returns no isolated origin, `codePortalProxy` then serves Code responses under `/portal/leases/.../code/...`, and `/portal` requests use `crabbox_session` as internal bearer auth. I did not run a live coordinator plus malicious bridge exploit.

## Solution Assessment

Yes on direction, but the exact compatibility path needs maintainer approval. Origin isolation or fail-closed behavior is the durable fix; cookie filtering alone does not prevent same-origin JavaScript from calling portal routes.
