## What This Changes

The PR rejects ASCII control characters in portal OAuth `returnTo` values before storing OAuth state and adds a Worker regression test for CRLF-tainted fallback to `/portal`.

## Best Possible Solution

Land a narrow OAuth hardening change that rejects decoded header-invalid `returnTo` values before storage, with focused regression coverage and redacted real-behavior proof of the callback fallback.

## Reproduction Assessment

Yes, at source level. Current main shows decoded `returnTo` reaching a `Location` header, and a local Node sanity check confirms the decoded CRLF value is rejected by `Headers`; I did not run the full Worker callback test in this read-only review.

## Solution Assessment

Yes. Validating decoded control characters in `safePortalReturnTo` before persisting OAuth state is the narrow maintainable fix, and the added callback regression test covers the intended fallback behavior.
