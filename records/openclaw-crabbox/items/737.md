## Summary

Keep open: this is a useful, narrow candidate fix for the linked OAuth `returnTo` security report, and current `main` still has the vulnerable source path. The patch looks correct, but the external-PR proof is still mock-only Vitest output and should not clear the real-behavior proof gate before maintainer/security review.

## What This Changes

The branch rejects header control characters in portal OAuth return targets and adds a Worker regression test for CRLF-tainted `returnTo` fallback to `/portal`.

## Best Possible Solution

Land the narrow OAuth hardening after redacted real-behavior proof shows the deployed callback falls back to `/portal` and still issues the portal session cookie for decoded header-invalid `returnTo` values.

## Reproduction Assessment

Yes, at source level. Current `main` decodes `returnTo`, stores it in pending OAuth state, and later emits it as a `Location` header; a read-only Node sanity check shows a decoded CRLF value is rejected by `Headers`.

## Solution Assessment

Yes. Rejecting decoded ASCII control characters in `safePortalReturnTo` before persisting OAuth state is the narrow maintainable fix, and the added callback regression test covers the intended fallback path.
