- `P2`: This is a concrete security-hardening issue with clear source evidence, but SSH key authentication bounds the immediate impact.
- `impact:security`: The issue concerns AWS security-group SSH exposure and provider firewall scoping for runtime-adapter workspaces.

## Summary

Keep open: current main still creates runtime-adapter AWS workspace leases with world-open SSH ingress, while the docs contain both general source-scoped AWS guidance and an explicit workspace bridge exception, so this needs maintainer security/product judgment rather than cleanup closure.

## What This Changes

The issue asks Crabbox to stop runtime-adapter AWS workspace leases from defaulting SSH ingress to `0.0.0.0/0` and add focused regression coverage.

## Best Possible Solution

Maintainers should decide the intended AWS workspace ingress policy, then either keep the documented world-open exception explicitly or remove the hard-coded CIDR and require `CRABBOX_AWS_SSH_CIDRS` or another trusted source before AWS workspace provisioning.

## Reproduction Assessment

Yes, by source trace: current main's `/v1/workspaces` provisioning path synthesizes a lease request with AWS `awsSSHCIDRs: ["0.0.0.0/0"]`, and the AWS provider authorizes configured CIDRs. I did not run live AWS provisioning in this read-only review.

## Solution Assessment

Unclear: the proposed source-scoped/fail-closed direction is a plausible security fix, but `docs/operations.md` explicitly documents world-open AWS workspace bridges as an operational workaround, so maintainers need to choose the intended policy first.
