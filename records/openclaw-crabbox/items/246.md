## Summary

Keep open. The feature is useful and not already on main, but the PR still has merge-blocking reliability and security issues: Screen Sharing startup can silently succeed, the guest account password is exposed through a new CLI flag, and the host-local VNC boundary plus vendored browser code need explicit maintainer acceptance.

## What This Changes

The PR adds tart desktop support by enabling macOS Screen Sharing, routing tart WebVNC through a host-side noVNC/WebSocket bridge, adding viewer password config, tests, embedded assets, and tart docs.

## Best Possible Solution

Land tart desktop only after Screen Sharing startup is verified or fails closed, account password input avoids argv exposure, the embedded noVNC provenance is auditable, and maintainers explicitly accept the host-local Screen Sharing boundary.

## Reproduction Assessment

Yes, from source inspection: all three launchctl operations are ignored with `|| true`, so a fake tart runner or failing service command can still return a ready desktop lease. I did not run live tart tests because this review is read-only and hardware-dependent.

## Solution Assessment

No. The host-side bridge direction is reasonable, but the implementation is not merge-ready until startup verification, non-argv password handling, and vendored asset provenance are addressed.

## Review Findings

Overall confidence: 0.92

Full review comments:

- **[P1] Fail when Screen Sharing does not start:** `internal/providers/tart/backend.go:544-546`
  - body: `enableScreenSharing` currently ignores every `launchctl` failure, so `crabbox warmup --provider tart --desktop` can return a ready lease even though the VNC service never started and `webvnc` later cannot reach `127.0.0.1:5900`. Make this path verify service startup or the loopback VNC probe and return an acquire error when it fails.
  - confidence: 0.96
- **[P1] Do not accept the account password as a CLI flag:** `internal/providers/tart/flags.go:25`
  - body: `--tart-password` is the guest account password used for ARD auth, and adding it as a flag exposes it via shell history and process listings. This conflicts with the repo's secret-handling guidance and existing provider precedent; keep the override in env/config or use an interactive/non-argv path instead.
  - confidence: 0.91
