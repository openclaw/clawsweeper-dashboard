## Summary

Keep open: current main and v0.33.0 still use owner-only brokered artifact keys, and the earlier open report at https://github.com/openclaw/crabbox/issues/702 already owns the same remaining security work. Under this repository's conservative cleanup policy, this should not be auto-closed unless the fix is implemented on main, so maintainers can manually consolidate the duplicate when appropriate.

## What This Changes

The issue reports that brokered artifact upload grants omit authenticated org from the object namespace, allowing same-owner artifact key collisions across org scopes.

## Best Possible Solution

Use https://github.com/openclaw/crabbox/issues/702 as the canonical tracker, then land an org-aware brokered artifact namespace with compatibility guidance and regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: send two authenticated artifact upload requests with the same owner, prefix, and file name but different orgs, and current main signs the same owner-only key. I did not run a live object-store overwrite/read test in this read-only review.

## Solution Assessment

Yes on direction, but not yet complete as a maintainer-ready change: the narrow fix is to include normalized org in newly signed brokered artifact keys, while the compatibility behavior for existing owner-only keys needs a deliberate maintainer decision.
