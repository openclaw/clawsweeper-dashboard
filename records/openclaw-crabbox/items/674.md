Full review comments:

- **[P1] Pin the Lambda MicroVM runner base images:** `runtimes/aws-lambda-microvm/Dockerfile:1-6`
  - body: This new runtime Dockerfile uses mutable `golang:1.26-alpine` and `ubuntu:24.04` tags while current runtime Dockerfiles are checked for SHA-256-pinned `FROM` images. Pin both images and add this file to `scripts/check-docker-base-images.mjs` so the reviewed execution environment cannot drift after merge.
  - confidence: 0.93
- **[P2] Reject broad Lambda MicroVM workdirs:** `internal/providers/awslambdamicrovm/flags.go:91-93`
  - body: `validateConfig` rejects only empty, relative, unclean paths and `/`, so broad paths like `/tmp`, `/usr`, `/etc`, and `/workspace` still pass. With default delegated archive sync delete mode, Crabbox moves and replaces that directory; reject broad system/runtime paths and require a dedicated subdirectory.
  - confidence: 0.9
- **[P2] Bound each runner health probe:** `internal/providers/awslambdamicrovm/backend.go:405`
  - body: When the MicroVM is `RUNNING`, `waitReady` calls `runner.Health(ctx, vm)` using the top-level CLI context. If the endpoint accepts the request and stalls, the five-minute readiness deadline is never checked and rollback can hang; use a short per-probe context or bounded HTTP client.
  - confidence: 0.85
