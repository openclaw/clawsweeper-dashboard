Overall confidence: 0.91

Full review comments:

- **[P1] Pin the Lambda MicroVM runner base images:** `runtimes/aws-lambda-microvm/Dockerfile:1-6`
  - body: This new runtime Dockerfile uses mutable `golang:1.26-alpine` and `ubuntu:24.04` tags while the repository's runtime Dockerfiles are checked for SHA-256-pinned `FROM` images. Pin both base images and add this Dockerfile to `scripts/check-docker-base-images.mjs` so the reviewed runner cannot drift after merge.
  - confidence: 0.93
- **[P2] Reject broad Lambda MicroVM workdirs:** `internal/providers/awslambdamicrovm/flags.go:91-93`
  - body: `validateConfig` currently allows broad absolute paths such as `/tmp`, `/usr`, or `/workspace`; with default archive-sync delete mode, `RunDelegatedArchiveSync` moves the configured workdir aside and replaces it with the checkout. Reject the same broad system/runtime paths other delegated providers reject and require a dedicated subdirectory such as `/workspace/crabbox`.
  - confidence: 0.9
- **[P2] Bound each runner health probe:** `internal/providers/awslambdamicrovm/backend.go:405`
  - body: When a MicroVM is `RUNNING`, `waitReady` calls `runner.Health(ctx, vm)` with the top-level CLI context, and the default runner HTTP client has no timeout. If the endpoint accepts the request and stalls, the five-minute readiness deadline is never checked and rollback can hang; use a short per-probe context or a bounded HTTP client.
  - confidence: 0.85
