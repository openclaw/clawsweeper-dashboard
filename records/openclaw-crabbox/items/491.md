## Summary

Keep open: current main still sends the Freestyle API key as a bearer header and formats non-2xx response bodies into user-visible errors without redacting the configured key.

## What This Changes

The issue requests Freestyle diagnostic error bodies to redact the configured API key before Crabbox prints provider failures.

## Best Possible Solution

Add a Freestyle-specific redaction helper for the exact API key and `Bearer <key>` form, apply it to every non-2xx response snippet, and add focused regression coverage modeled on the OVH/Superserve redaction tests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a Freestyle test server that returns a non-2xx body containing the configured key would be wrapped directly into the returned error. I did not run tests because this review is read-only and source inspection was sufficient.

## Solution Assessment

Yes, the requested repair is the narrow maintainable fix: redact the configured key and bearer-token form before formatting Freestyle API error snippets while preserving status and useful provider context.
