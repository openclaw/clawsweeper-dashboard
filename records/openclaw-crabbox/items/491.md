## Summary

Keep open: current main still sends the Freestyle API key as a bearer header and formats non-2xx Freestyle response bodies into user-visible errors without redacting the configured key.

## What This Changes

Request redaction of the configured Freestyle API key and Bearer form from Freestyle API error snippets before they appear in diagnostics.

## Best Possible Solution

Add a shared Freestyle response-error redactor for `c.apiKey` and `Bearer <key>`, apply it to every non-2xx branch, and cover it with focused provider tests modeled on OVH/Superserve.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a Freestyle test server returning a non-2xx body containing the configured key would have that body included directly in the returned error. I did not run a test because this review is read-only and the source path is clear.

## Solution Assessment

Yes, the narrow maintainable fix is to redact the exact configured key and `Bearer <key>` form before formatting Freestyle API error snippets while preserving status and useful provider context.
