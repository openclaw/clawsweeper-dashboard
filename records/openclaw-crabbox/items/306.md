## Label Justifications

- `P2`: This is a concrete security hardening issue with limited blast radius in managed Linux Tailscale bootstrap rather than an active core outage.
- `impact:security`: The issue concerns avoidable credential exposure through process argv during runner bootstrap.

## Summary

Keep open: current main and the latest release still render the Tailscale auth key into the `tailscale up` argv for both direct-provider and Worker bootstrap paths, while the Islo path already demonstrates the safer stdin pattern.

## What This Changes

The issue asks Crabbox to harden managed Linux Tailscale bootstrap so auth keys are piped through `file:/dev/stdin` instead of expanded into the `tailscale up` command line.

## Best Possible Solution

Update both bootstrap generators to pipe the auth key into `tailscale up --auth-key=file:/dev/stdin`, keep `set +x` and key cleanup, and add Go and Worker regression coverage for the generated cloud-init.

## Reproduction Assessment

Yes. Source inspection of current main shows both generated bootstrap paths expand `TS_AUTHKEY` into the `tailscale up` command line, and existing tests assert that generated string.

## Solution Assessment

Yes. Matching the existing Islo stdin delivery pattern in the direct and Worker bootstrap generators is the narrowest maintainable fix, provided the existing Tailscale flags remain covered by tests.
