## Label Justifications

- `P2`: This is a bounded but real security bug exposing coordinator auth metadata to a lease-controlled runtime.
- `impact:security`: The issue concerns a supported coordinator trust boundary and sensitive viewer identity/role/session metadata.
- `impact:auth-provider`: The leaked headers are broker auth-context markers, including auth type, admin state, owner/org, GitHub login, and token expiry.

## Summary

Keep open: current main still forwards broker `x-crabbox-*` auth-context headers through the Code portal proxy to the lease-side Code bridge, and no merged PR or canonical issue currently owns this exact security fix.

## What This Changes

Strip Crabbox-internal authentication context headers from Code portal HTTP and WebSocket forwarding before requests reach the lease-controlled Code bridge.

## Best Possible Solution

Filter `x-crabbox-*` before the broad Code header forwarding rule, or replace that rule with a narrow allowlist, then add Worker and bridge regression coverage for HTTP and WebSocket forwarding.

## Reproduction Assessment

Yes for a source-level reproduction: current main injects the auth-context headers, forwards all `x-*` Code headers, and applies them on the lease-side upstream request. I did not run a live bridge harness.

## Solution Assessment

Yes, the narrowest maintainable fix is to stop forwarding broker-internal `x-crabbox-*` headers while preserving the existing Code header and cookie behavior. A stricter explicit allowlist would be stronger defense-in-depth if maintainers want to take that broader step now.
