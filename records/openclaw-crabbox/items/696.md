## Label Justifications

- `P0`: The report describes a plausible cross-owner SSH access path through a coordinator/provider security boundary.
- `impact:security`: The affected behavior can attach one authenticated caller's EC2 SSH key to another owner's AWS lease when providerKey is omitted.

## Summary

Keep open: current main still resolves an omitted direct lease providerKey to the shared name crabbox-steipete, and the AWS provider still accepts any existing EC2 key-pair name without checking the requested public key. The latest release v0.33.0 has the same code shape, and related search did not find a merged fix or canonical replacement.

## What This Changes

Fix a security bug where direct AWS coordinator leases that omit providerKey can reuse a shared EC2 key-pair name across owners.

## Best Possible Solution

Remove the shared fallback for coordinator-created AWS leases by deriving a unique lease-scoped provider key before provisioning or failing closed, then add regression coverage for omitted providerKey and AWS key-pair collisions.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: an authenticated direct POST /v1/leases can omit providerKey, leaseConfig supplies the shared name, AWS reuses an existing key-pair name, and RunInstances uses that name. I did not run a live AWS reproduction.

## Solution Assessment

Yes, the narrow maintainable direction is to stop using a shared fallback and ensure AWS key-pair names cannot silently bind to mismatched key material. The standard CLI path should remain compatible because it already sends a per-lease providerKey.
