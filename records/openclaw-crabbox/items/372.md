## Summary

Keep open: current main still constructs, accepts, documents, and tests WebSocket query-string bridge tickets for Code, WebVNC, and egress, while the latest maintainer comment reclassifies this as compatibility-sensitive hardening rather than closing it.

## What This Changes

The issue asks Crabbox to keep short-lived bridge bearer tickets out of WebSocket request URLs while preserving bridge compatibility.

## Best Possible Solution

Add a compatibility-preserving non-URL bridge credential path, such as a one-time HTTPS exchange for an upgrade nonce, then update the CLI, Worker parsing, tests, and docs so bearer tickets no longer appear in WebSocket request targets.

## Reproduction Assessment

Yes. Source inspection on current main shows Code/WebVNC query fallback, egress query-ticket URLs, Worker query-ticket acceptance, and tests that pin the behavior; no live replay was needed to establish the reported path exists.

## Solution Assessment

No for simply deleting query fallback. The best solution is a compatibility-preserving non-URL exchange because maintainer discussion explicitly requires preserving WebSocket operation through Authorization-stripping intermediaries and preserving egress.
