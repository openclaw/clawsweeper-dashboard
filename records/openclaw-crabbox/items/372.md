## Label Justifications

- `P0`: The issue reports a bridge credential exposure path that can let a log reader replay a short-lived bearer ticket into active bridge traffic.
- `impact:security`: The affected behavior is bridge-ticket credential transport and authorization boundary handling.

## Summary

Keep open: current main still constructs and accepts bridge tickets in WebSocket query strings, and no open or merged PR currently owns this fix.

## What This Changes

The issue reports that code, WebVNC, and egress bridge tickets can be sent and accepted through WebSocket URL query strings.

## Best Possible Solution

Move bridge authentication to a non-URL credential path or short-lived upgrade nonce, update Worker parsing to reject or explicitly gate query tickets, and document any temporary compatibility mode without logging secrets.

## Reproduction Assessment

Yes. Current-main source and tests directly show code/WebVNC query fallback, egress query-ticket URLs, and Worker query-ticket acceptance; I did not need a live replay to establish the reported behavior exists.

## Solution Assessment

Unclear. Removing query tickets is the right security end state, but maintainers need to choose the compatibility strategy for older CLIs, intermediaries that reject Authorization on WebSocket upgrades, and the egress API shape.
