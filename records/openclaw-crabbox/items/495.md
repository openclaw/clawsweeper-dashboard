- `impact:security`: The issue concerns accidental disclosure of files outside the selected artifact bundle through trusted artifact publishing credentials and URLs.

## Summary

Keep open: current main still accepts symlinked artifact publish bundle entries and then follows those paths during upload and manifest generation, so the reported security boundary issue remains necessary.

## What This Changes

The issue requests rejecting symlinked files in artifact publish bundles so Crabbox does not upload bytes outside the selected artifact directory.

## Best Possible Solution

Add publish-side symlink rejection before upload or manifest side effects, with regression coverage around listArtifactBundleFiles or artifacts publish --dir.

## Reproduction Assessment

Yes from source: a symlink entry under a publish directory reaches listArtifactBundleFiles, then later file APIs and upload commands operate on that symlink path. I did not run a live upload because this review is read-only and should avoid backend side effects.

## Solution Assessment

Yes: rejecting symlinked entries during bundle enumeration is the narrowest maintainable fix because it stops broker, S3/R2, Cloudflare, Markdown, and manifest paths before side effects.
