## Review Findings

Full review comments:

- **[P2] Emit `--json` before the harness path:** `index.js:321-322`
  - body: `crabbox_harness_validate` currently builds `crabbox harness validate HARNESS.md --json`, but `harnessValidate` uses the normal Go flag parser, which stops parsing at the first positional argument. With the real CLI this leaves `--json` as an extra arg and returns the usage error instead of JSON, while the test only checks a fake binary argv echo. Emit `--json` before the path or switch this command to interspersed flag parsing, and add a test through the real CLI parser.
  - confidence: 0.93
- **[P2] Match each required artifact instead of any artifact:** `internal/cli/harness.go:493-496`
  - body: When `required_artifacts` is configured, this code only reports missing artifacts if the run produced zero artifacts. A harness that requires `junit` will pass if any unrelated artifact exists, so the compliance gate can approve a run missing the evidence it required. Compare each requested name or kind against the collected artifacts and report the specific missing entries.
  - confidence: 0.94
- **[P3] Document a job flag order the parser accepts:** `README.md:97`
  - body: `jobRun` uses ordinary `parseFlags`, so examples that put `--harness` after the job name are not accepted by the real parser. Either make `job run` parse interspersed flags or update the added examples to the supported `crabbox job run --harness HARNESS.md <name>` form everywhere.
  - confidence: 0.88
- **[P3] Remove project-specific names from the roadmap page:** `docs/plan/crabbox-harness-station-reasoning-roadmap.html:379-396`
  - body: AGENTS.md asks new Crabbox code and docs to stay generic unless the file is explicitly about legacy compatibility or release history. This new roadmap page names OpenClaw and Peter in product-positioning text, so it should be made generic or moved to a clearly scoped plugin/release-history document before merge.
  - confidence: 0.9
