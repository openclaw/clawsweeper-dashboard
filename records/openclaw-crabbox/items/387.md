## Label Justifications

- `P1`: The issue concerns a credible release-channel and publication-credential supply-chain risk in workflows that publish Crabbox artifacts and docs.
- `impact:security`: The report is about workflow action trust, publication credentials, release integrity, and Pages deployment permissions.

## Summary

Keep open: current main still uses mutable major GitHub Action tags in the release, CI, hydrate, and Pages workflows, and the release job passes publish credentials to `goreleaser/goreleaser-action@v7`. The local Blacksmith workflow shows the requested pinned-SHA pattern already exists in this repo, and no canonical merged or open PR appears to own this workflow hardening yet.

## What This Changes

The issue asks Crabbox to pin release, CI, hydrate, and Pages workflow `uses:` actions to immutable SHAs and add enforcement around mutable action tags in publication paths.

## Best Possible Solution

Pin every workflow `uses:` reference in the affected workflows to reviewed full SHAs with version comments, add a repository check for mutable action refs, and review release-token scope plus Homebrew formula integrity verification as maintainer-owned release hardening.

## Reproduction Assessment

Yes: source inspection on current main reproduces the report with a direct `rg` scan for mutable action refs and confirms `release.yml` passes `GITHUB_TOKEN` plus `HOMEBREW_TAP_GITHUB_TOKEN` to the mutable GoReleaser action.

## Solution Assessment

Yes for the core pinning and enforcement direction; the broader token-scope and Homebrew checksum items are security policy choices that should be reviewed by maintainers rather than auto-applied blindly.
