- `impact:auth-provider`: The issue is about OAuth credential/tag configuration and how the coordinator requests Tailscale auth-key permissions.
- `impact:security`: Tailscale tag ownership controls authorization scope, and unclear guidance can push operators toward over-broad tag permissions.

## Summary

Keep open: current main still uses `CRABBOX_TAILSCALE_TAGS` as both the local allowlist/default and the OAuth request tags, while the docs and error mapping do not yet explain or translate Tailscale's tag-owner subset failure mode.

## What This Changes

The issue requests clearer brokered Tailscale OAuth tag-ownership docs and actionable Worker errors for related Tailscale API failures while preserving the existing tag allowlist guardrail.

## Best Possible Solution

Update the Tailscale docs/troubleshooting pages for exact-match, self-owner, and deployment-owner tag patterns, then add focused Worker error classification and regression tests for tag-ownership API failures without adding a new env var unless maintainers choose that direction.

## Reproduction Assessment

Yes for a focused source-level or Worker-test reproduction: stub the Tailscale OAuth/auth-key call to reject a requested subset tag and current main will return generic `tailscale_unavailable`. I did not run a live Tailscale tenant reproduction.

## Solution Assessment

Yes, the narrowest maintainable fix is documentation plus actionable error translation and tests. A split OAuth-tag env var is optional product scope and should be deferred unless maintainers explicitly want it.
