Full review comments:

- **[P1] Use one Theme type so the iOS app compiles:** `mobile/App/Assistant/AssistantView.swift:17`
  - body: The app target already has `App/Theme.swift` declaring `Theme`, so this second `private enum Theme` in the same module triggers the current CI error `invalid redeclaration of 'Theme'` and prevents the iOS app from building. Please reuse or rename the shared theme tokens before merge.
  - confidence: 0.99
- **[P1] Validate provider URLs before using stored secrets:** `mobile/App/Sandboxes/ProviderSettingsView.swift:132-139`
  - body: `save()` persists raw `coordinatorURL` and `isloBaseURL`, but `makeProvisioner()` later creates clients that send a crabbox bearer token or islo API key to those bases. This bypasses the Portal HTTPS/loopback policy and can leak credentials to `http://` or malformed configured origins; reject non-HTTPS production URLs before a provisioner can be built.
  - confidence: 0.91
- **[P2] Fail the launch when the bootstrap exec fails:** `mobile/Sources/CrabboxKit/SandboxProvisioner.swift:63-64`
  - body: The direct islo launch path discards the result of the Ollama bootstrap command and proceeds to create a share, so a failed install or model pull can still mark the sandbox ready and register a dead chat engine. Check the exec exit code and surface stdout/stderr in the thrown error before exposing the endpoint.
  - confidence: 0.88
- **[P2] Require an exit event in the Swift islo parser:** `mobile/Sources/CrabboxKit/IsloClient.swift:222-237`
  - body: `parseExecStream` initializes `exit` to zero and returns success even when the SSE stream ends without an exit event. The existing Go islo client treats a missing exit event as an error, and the Swift path should do the same so truncated or errored streams cannot look successful.
  - confidence: 0.9
