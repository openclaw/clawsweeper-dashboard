## Label Justifications

- `P2`: This is a bounded but concrete credential-transport security concern on live bridge agent routes.
- `impact:security`: The issue concerns generated bridge credentials being accepted from URL request targets where logs or intermediaries may expose them.

## Summary

Keep open: current main still accepts valid WebVNC, code, and egress bridge tickets from `?ticket=` when header credentials are absent, and the latest release retains the same legacy fallback. This is concrete and source-reproducible, but it needs maintainer/security direction because removing the fallback changes compatibility-sensitive bridge authentication behavior.

## What This Changes

The issue asks Crabbox to stop accepting live bridge agent tickets from URL query strings on WebVNC, code, and egress agent upgrade routes.

## Best Possible Solution

Reject query-string bridge tickets by default on agent upgrade routes, preserve `X-Crabbox-Bridge-Ticket` and bearer-header auth, and use a maintainer-approved compatibility gate or replacement nonce exchange only if legacy deployments still need it.

## Reproduction Assessment

Yes. Source inspection on current main shows `bridgeTicketFromRequest` still returns a valid `ticket` query parameter when stronger header forms are absent, and Worker tests explicitly assert legacy query acceptance.

## Solution Assessment

Unclear. The narrow security fix is to reject query-string tickets, but the best rollout needs maintainer direction because existing code and prior discussion frame the query path as compatibility-sensitive legacy behavior.
