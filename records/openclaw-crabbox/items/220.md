## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Validate the API origin before attaching the token:** `internal/providers/smolvm/client.go:151-152`
  - body: `smolvm.baseUrl` can come from repo-local config, but `newAPI` accepts any trimmed string and `doJSON` later adds the SmolVM bearer token plus archive payloads to that origin. Please restrict this to the hosted Smol Machines API, or add an explicit maintainer-approved custom-origin boundary with tests before credentials are sent.
  - confidence: 0.91
- **[P1] Honor sync.include before building SmolVM archives:** `internal/providers/smolvm/sync.go:21`
  - body: This path calls `syncManifest(req.Repo.Root, excludes)`, and the SmolVM wrapper maps that to the unfiltered core manifest builder. Existing delegated providers pass `cfg.Sync.Includes`, so a user with `sync.include` set can upload files outside their whitelist when switching to SmolVM.
  - confidence: 0.97
- **[P1] Decode snake_case exec exit codes explicitly:** `internal/providers/smolvm/client.go:59`
  - body: The duplicate struct tag on `ExitCode` does not make `encoding/json` accept both `exitCode` and `exit_code`; only the first key is used. If SmolVM returns `exit_code`, failed exec/archive/write responses decode as zero and Crabbox can report a failed remote command as successful.
  - confidence: 0.88
- **[P2] Route remote stderr to the stderr writer:** `internal/providers/smolvm/client.go:255-257`
  - body: `ExecStream` receives `stderr` separately in the API response but writes it to the same `stdout` writer as normal output. That breaks shell redirection and Crabbox output semantics for commands that write diagnostics to stderr; the backend/client seam should carry a stderr writer like the other providers that can distinguish streams.
  - confidence: 0.82
