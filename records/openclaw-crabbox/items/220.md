Full review comments:

- **[P1] Honor sync.include when building SmolVM archives:** `internal/providers/smolvm/sync.go:21`
  - body: `syncWorkspace` builds the archive manifest without passing `cfg.Sync.Includes`, unlike the existing delegated providers. A user who configured `sync.include` to limit uploads would still send the whole non-excluded working tree to SmolVM, so this needs to use the filtered manifest path and have regression coverage.
  - confidence: 0.96
- **[P2] Decode snake_case exec exit codes:** `internal/providers/smolvm/client.go:59`
  - body: This struct tag lists two JSON names, but `encoding/json` only uses the effective `exitCode` tag. If the SmolVM API returns `exit_code` for a failed exec, `ExitCode` stays zero and `run` can report success for a failed command; add an explicit fallback field or custom unmarshal plus a regression test.
  - confidence: 0.9
- **[P1] Validate SmolVM base URLs before sending secrets:** `internal/providers/smolvm/client.go:151`
  - body: `newAPI` accepts the configured base URL as-is and later attaches the bearer token plus archive/exec payloads. Token-bearing provider clients in current main reject non-HTTPS non-loopback URLs and unsafe URL components; SmolVM needs the same guardrail or an explicit maintainer-approved trust model before merge.
  - confidence: 0.92
- **[P1] Refresh the provider docs conflict:** `docs/providers/README.md:38`
  - body: A merge-tree against current main shows this docs hunk conflicts with the newly added Docker Sandbox entry, so GitHub reports the branch dirty and the PR cannot merge cleanly until the provider list is rebased/resolved.
