## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Validate SmolVM base URLs before sending secrets:** `internal/providers/smolvm/client.go:151-152`
  - body: `newAPI` copies `cfg.Smolvm.BaseURL` into the client without parsing or enforcing a trusted scheme/host, and `doJSON` later adds `Authorization` to requests built from that value. Because `smolvm.baseUrl` can come from repo config, a checked-out project can redirect a user's SmolVM API key and archive payloads to another endpoint; validate/normalize the URL or make custom endpoints explicit and safe before constructing the client.
  - confidence: 0.92
- **[P1] Honor sync.include when building SmolVM archives:** `internal/providers/smolvm/sync.go:21`
  - body: This calls the unfiltered manifest wrapper, so a config with `sync.include` still archives every tracked/untracked file except excludes. Other delegated providers pass `cfg.Sync.Includes` to the filtered manifest builder; SmolVM should do the same and add a regression test because users may rely on includes to avoid uploading unrelated content.
  - confidence: 0.96
- **[P2] Decode snake_case exec exit codes:** `internal/providers/smolvm/client.go:59`
  - body: A struct field cannot have two effective `json` tags, so responses using `exit_code` leave `ExitCode` at zero. If the API returns snake_case for a failed command, Crabbox reports success; add an alias/custom unmarshal path and cover it with a test.
  - confidence: 0.86
