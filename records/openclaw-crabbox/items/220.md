## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor sync.include when building SmolVM archives:** `internal/providers/smolvm/sync.go:21`
  - body: This builds the SmolVM archive manifest without `cfg.Sync.Includes`; the wrapper also calls `BuildSyncManifest`, not the include-aware helper used by existing delegated providers. Users who configure `sync.include` to upload only selected paths can still send the full tracked set to SmolVM, including files they expected to keep out of the remote archive.
  - confidence: 0.95
- **[P1] Validate SmolVM base URLs before sending secrets:** `internal/providers/smolvm/client.go:151-152`
  - body: `newAPI` accepts the configured base URL verbatim and later `doJSON` adds the `Authorization: Bearer` header and archive payloads to requests built from it. A value such as a remote `http://` URL, userinfo URL, or URL with query/fragment components can receive the SmolVM key and repo tarball; parse and normalize the URL and require HTTPS except loopback before constructing the client.
  - confidence: 0.92
- **[P2] Decode snake_case exec exit codes:** `internal/providers/smolvm/client.go:59`
  - body: The repeated struct tag only registers the first `json` key with `encoding/json`, so responses using `exit_code` leave `ExitCode` at zero. That can make a failed SmolVM command look successful; add an explicit alias/custom unmarshal path and a regression test for snake_case failures.
  - confidence: 0.9
