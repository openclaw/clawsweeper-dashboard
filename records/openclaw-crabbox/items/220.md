## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor sync.include when building SmolVM archives:** `internal/providers/smolvm/sync.go:21`
  - body: SmolVM currently builds its archive manifest with only excludes, unlike the current archive-sync providers that pass `cfg.Sync.Includes` into the filtered manifest helper. With `sync.include` configured, this provider would upload the whole repo minus excludes instead of the user's whitelist, which can expose files the user intentionally left out of sync.
  - confidence: 0.95
- **[P1] Decode snake_case exit_code before reporting success:** `internal/providers/smolvm/client.go:59`
  - body: The duplicate struct tag on `ExitCode` does not make `encoding/json` accept both `exitCode` and `exit_code`; the snake_case response leaves `ExitCode` at zero. If SmolVM returns a failed exec as `exit_code`, `ExecStream` reports success and the run can exit 0 despite a failed command.
  - confidence: 0.9
- **[P1] Guard custom API origins before sending SmolVM tokens:** `internal/providers/smolvm/client.go:151-152`
  - body: `newAPI` accepts a configured base URL while `doJSON` always attaches the `smk_*` bearer token. Unless maintainers explicitly want arbitrary SmolVM API origins, this should be restricted to the hosted origin or an approved allowlist/test-only path so config or environment changes cannot redirect provider credentials to an unintended host.
  - confidence: 0.82
