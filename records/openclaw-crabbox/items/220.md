Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Honor sync.include before archiving SmolVM workspaces:** `internal/providers/smolvm/sync.go:21`
  - body: `syncWorkspace` builds the manifest with only excludes, while the current sync contract has include whitelists via `cfg.Sync.Includes`. With `sync.include` configured, this still archives and uploads the rest of the repo to hosted SmolVM, which breaks the documented whitelist and can expose files users opted out of syncing.
  - confidence: 0.93
- **[P1] Validate SmolVM API URLs before sending bearer auth:** `internal/providers/smolvm/client.go:151-152`
  - body: `newAPI` accepts the configured base URL verbatim and `doJSON` sends `Authorization: Bearer ...` plus archive exec payloads to `c.base`. Since Crabbox can load config from a checkout, require a parsed HTTPS host or narrow localhost/test escape before this provider sends SmolVM credentials and synced source to a configurable endpoint.
  - confidence: 0.88
- **[P2] Decode snake_case SmolVM exit codes:** `internal/providers/smolvm/client.go:59`
  - body: A Go struct tag cannot map one field to both `exitCode` and `exit_code`; `encoding/json` will use the first `json` tag. If the API returns `exit_code`, failed commands decode as zero and `Run` can report success, so add a separate field or custom unmarshal with regression coverage.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete credential/source exfiltration risk because the SmolVM base URL is configurable and used for bearer-auth archive/exec requests without validation.

Concerns:

- **[high] Validate bearer-auth SmolVM base URLs:** `internal/providers/smolvm/client.go:151`
