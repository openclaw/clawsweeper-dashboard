## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route remote stderr to stderr:** `internal/providers/smolvm/client.go:255-257`
  - body: `ExecStream` now forwards stderr, but it forwards it through the same writer used for stdout. That means a command that writes diagnostics to stderr will pollute stdout, so shell redirection, captured stdout, and machine-readable output differ from other delegated providers. Pass a stderr writer through this API and call it with `b.rt.Stderr`, then add a regression that stderr does not appear in stdout.
  - confidence: 0.95
- **[P1] Validate the API base URL before sending tokens:** `internal/providers/smolvm/client.go:151-152`
  - body: `smolvm.baseUrl` is configurable, but `newAPI` stores it without parsing or enforcing HTTPS/loopback before `addHeaders` attaches the bearer token. A typo or config change can send `smk_*` credentials to an arbitrary cleartext or hostile origin; mirror the HTTPS-or-localhost validation used by nearby bearer-token providers before constructing the client.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The diff introduces a token-bearing HTTP client whose configurable base URL is not validated before attaching `Authorization: Bearer ...`.

  - body: `newAPI` trims `cfg.Smolvm.BaseURL` into `client.base`, and `addHeaders` later attaches the bearer token to every request built from that base. Nearby providers validate scheme and loopback exceptions before sending provider tokens.
