## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate the SmolVM API URL before sending auth:** `internal/providers/smolvm/client.go:151-152`
  - body: `smolvm.baseUrl` can come from repo-local config while the API key comes from env/user config, but this accepts any configured base and `doJSON` always attaches `Authorization`. A checked-out repo can therefore route the SmolVM bearer token plus archive and exec payloads to an arbitrary origin. Parse and validate the origin before constructing the client, and cover the rejection path.
  - confidence: 0.9
- **[P1] Honor sync.include before uploading SmolVM archives:** `internal/providers/smolvm/sync.go:21`
  - body: Current Crabbox supports `sync.include` as a whitelist, but SmolVM calls the unfiltered manifest builder here. Users who restrict sync to selected paths would still upload the full tracked/untracked manifest to SmolVM. Pass `b.cfg.Sync.Includes` through the filtered builder and add a regression test for excluded files not reaching the injected archive.
  - confidence: 0.95
- **[P1] Decode snake_case exec exit codes:** `internal/providers/smolvm/client.go:59`
  - body: A Go struct field cannot decode both `exitCode` and `exit_code` by repeating the same tag key, so a SmolVM response using `exit_code` leaves `ExitCode` at zero. That can turn failed exec/archive/write responses into apparent success. Add a separate snake-case field or custom unmarshal and assert non-zero failures propagate.
  - confidence: 0.92
