## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor sync.include when building SmolVM archives:** `internal/providers/smolvm/sync.go:21`
  - body: SmolVM builds its archive with `syncManifest(req.Repo.Root, excludes)`, whose wrapper calls `core.BuildSyncManifest` without `cfg.Sync.Includes`. Other delegated archive-sync providers pass includes into the filtered manifest helper, so a user with `sync.include` configured would upload the whole repo minus excludes to SmolVM. Pass the includes through and add a regression proving a non-included path stays out of the archive.
  - confidence: 0.9
- **[P1] Decode snake_case exit_code before reporting success:** `internal/providers/smolvm/client.go:59`
  - body: The duplicated struct tag only registers `exitCode`; `encoding/json` will not fill this field from `exit_code`. If SmolVM returns or a compatibility response uses snake_case, a failed command decodes as exit 0 and `Run` reports success. Use a separate alias field or custom unmarshal and cover snake_case failures.
  - confidence: 0.88
- **[P1] Guard custom API origins before sending SmolVM tokens:** `internal/providers/smolvm/client.go:151`
  - body: `newAPI` accepts the configured base URL unchanged, and every request is signed with `Authorization: Bearer ...`. Because the base URL is exposed through flag/env/config, a typo or malicious config can send `smk_*` credentials to an arbitrary host; remove the override or require an explicit allowlisted/dev-only trust path before adding this provider.
  - confidence: 0.86
