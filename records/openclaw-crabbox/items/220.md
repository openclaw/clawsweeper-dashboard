## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Honor sync.include before archiving SmolVM workspaces:** `internal/providers/smolvm/sync.go:21`
  - body: `syncWorkspace` builds the SmolVM archive with `syncManifest(req.Repo.Root, excludes)`, so a configured `sync.include` whitelist is ignored. Other delegated providers pass `b.cfg.Sync.Includes`; as written, users who whitelist files to avoid uploading secrets or large directories can still send the whole working tree to the hosted SmolVM API.
  - confidence: 0.96
- **[P1] Validate the SmolVM API URL before adding auth:** `internal/providers/smolvm/client.go:151-152`
  - body: `newAPI` accepts the configured base URL verbatim and `doJSON` later sends the bearer token plus archive/env `/exec` payloads to that host. A mistaken or malicious `smolvm.baseUrl` / `CRABBOX_SMOLVM_BASE_URL` can therefore receive credentials and workspace data; validate the URL scheme and host before constructing the client.
  - confidence: 0.9
- **[P1] Decode snake_case SmolVM exit codes:** `internal/providers/smolvm/client.go:59`
  - body: This tag only registers `exitCode`; Go ignores the second `json:"exit_code"` key in the same struct tag. If the API returns `exit_code: 1`, Crabbox decodes zero and treats failed exec, sync, or write steps as success.
  - confidence: 0.97
