## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use the shared safe sync archive writer:** `internal/providers/smolvm/sync.go:113`
  - body: SmolVM sync reintroduces a raw `tar --null -T -` archive path fed by repository filenames. Current main has a shared `CreateSyncArchive` path with option-like filename coverage; using raw tar here can let a malicious repo filename affect local archive creation before the hosted run starts.
  - confidence: 0.94
- **[P1] Honor sync.include before uploading SmolVM archives:** `internal/providers/smolvm/sync.go:21`
  - body: This builds the manifest with only excludes, so a user-configured `sync.include` whitelist is ignored for SmolVM. Users relying on that whitelist can upload more workspace content to the hosted API than they intended; mirror the other delegated archive providers and pass `cfg.Sync.Includes`.
  - confidence: 0.96
- **[P1] Validate the SmolVM API URL before sending auth:** `internal/providers/smolvm/client.go:151`
  - body: The configured BaseURL is accepted as-is and later receives `Authorization: Bearer ...` plus exec/archive payloads. A repo or environment config can redirect the SmolVM token and workspace contents to an arbitrary origin unless the client rejects unsafe or unexpected API bases before constructing requests.
  - confidence: 0.91
- **[P1] Decode snake_case exec exit codes:** `internal/providers/smolvm/client.go:59`
  - body: The duplicate struct tag does not make Go decode both `exitCode` and `exit_code`; an `exit_code` response leaves `ExitCode` at zero. That can report failed hosted commands or archive extraction as successful, so add explicit decoding and a non-zero snake_case regression test.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR introduces concrete security-boundary concerns around raw local archive creation, unvalidated bearer-auth destinations, and ignored upload whitelists.

Concerns:

- **[high] Unvalidated API base can receive bearer auth:** `internal/providers/smolvm/client.go:151`
  - body: SmolVM BaseURL can come from config/env/flags and is used for authenticated requests without scheme or origin validation, so a malicious config can exfiltrate the SmolVM token and exec/archive payloads.
  - confidence: 0.91
- **[high] Raw tar archive path bypasses current hardening:** `internal/providers/smolvm/sync.go:113`
  - body: The PR uses GNU tar with a repo-controlled file list instead of the shared safe archive writer, reintroducing option-like filename risk during local archive creation.
  - confidence: 0.94

  - body: SmolVM archive sync ignores the configured include whitelist, so users can send more workspace content to the hosted API than their Crabbox config allows for other sync providers.
  - confidence: 0.96
