## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate the SmolVM base URL before sending secrets:** `internal/providers/smolvm/client.go:151-152`
  - body: `newAPI` accepts `cfg.Smolvm.BaseURL` as a trimmed string and `doJSON` then sends every request with `Authorization: Bearer ...` plus command/env/archive bodies to that URL. Nearby authenticated providers parse their endpoint and reject non-HTTPS except loopback, and Azure also rejects userinfo/query/fragment components; SmolVM needs the same validation and regression coverage before exposing `--smolvm-base-url`/env/config overrides.
  - confidence: 0.93
- **[P2] Decode snake_case exec exit codes explicitly:** `internal/providers/smolvm/client.go:59`
  - body: The struct tag `json:"exitCode" json:"exit_code"` still only maps the field to `exitCode`; `encoding/json` will not also populate it from `exit_code`. If the API or a compatibility response returns `exit_code`, failed commands, archive extraction, or file writes can look successful because `ExitCode` remains zero, so add a separate field/custom unmarshal and a regression test for the snake_case response.
  - confidence: 0.86
- **[P2] Reject unsupported delegated options in the backend:** `internal/providers/smolvm/backend.go:59`
  - body: The SmolVM docs say delegated-run restrictions apply, but `Run` starts provisioning without calling the common delegated option rejection helper or rejecting desktop/browser/code/Tailscale options that this provider cannot honor. Direct backend callers and focused tests can therefore silently ignore unsupported requested behavior; add the backend guardrail and coverage before merge.
  - confidence: 0.78
