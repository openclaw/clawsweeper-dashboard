## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject unsupported delegated-run flags before provisioning:** `internal/providers/smolvm/backend.go:59-60`
  - body: `Run` never calls the delegated option rejection helper, so flags such as `--fresh-pr`, `--script`, `--download`, `--artifact-glob`, captures, `--stop-after`, and `--emit-proof` can be accepted and then ignored after a SmolVM machine is created. That can run the wrong checkout or omit requested artifacts while reporting a normal delegated run; call `RejectDelegatedSyncOptionsForSpec` at the start and cover representative rejected flags.
  - confidence: 0.9
- **[P1] Honor sync.include when building SmolVM archives:** `internal/providers/smolvm/sync.go:21`
  - body: This builds the archive manifest with only excludes, while current archive-sync providers pass `cfg.Sync.Includes` into `BuildSyncManifestFiltered`. A user who configures `sync.include` to send only selected paths will still upload the full tracked/untracked tree to the hosted SmolVM API, defeating both the documented whitelist and the expected source-minimization boundary.
  - confidence: 0.94
- **[P1] Route remote stderr away from stdout:** `internal/providers/smolvm/client.go:255-257`
  - body: `ExecStream` appends `result.Stderr` to the stdout writer, so `crabbox run --provider smolvm -- ... >out` will mix diagnostics into machine-readable stdout and corrupt normal shell redirection expectations. Thread a stderr writer through the SmolVM exec path, or otherwise emit stderr on `Runtime.Stderr`, and update the test that currently expects `hi\nwarn\n` on stdout.
  - confidence: 0.92
