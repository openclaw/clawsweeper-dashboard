## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Validate the SmolVM API origin before sending secrets:** `internal/providers/smolvm/client.go:151`
  - body: `smolvm.baseUrl` can come from repo config, env, or flags, but `newAPI` accepts it after trimming and every request adds `Authorization: Bearer ...`. A malicious or mistaken `.crabbox.yaml` can point this provider at plain HTTP or an arbitrary host and receive both the API key and archive payloads; require HTTPS for non-loopback origins before constructing the client.
  - confidence: 0.92
- **[P1] Honor sync.include before building SmolVM archives:** `internal/providers/smolvm/sync.go:21`
  - body: SmolVM archive sync calls the unfiltered manifest builder, so configured `sync.include` whitelists are ignored and the provider can upload the whole nonignored tree. Other delegated providers pass `b.cfg.Sync.Includes` through `BuildSyncManifestFiltered`; SmolVM should do the same with regression coverage.
  - confidence: 0.95
- **[P1] Decode snake_case exec exit codes:** `internal/providers/smolvm/client.go:59`
  - body: This struct tag only registers `exitCode` with `encoding/json`; it does not also decode `exit_code`. If the SmolVM API returns snake_case, a failed command decodes with exit code 0 and Crabbox reports success, so add an explicit fallback field or custom unmarshal and cover it in tests.
  - confidence: 0.9
- **[P2] Keep remote stderr on stderr:** `internal/providers/smolvm/client.go:255-256`
  - body: `ExecStream` appends `result.Stderr` to the stdout writer, so a command that writes machine-readable stdout plus diagnostics on stderr will have its stdout corrupted. The backend should route remote stderr to `b.rt.Stderr`, matching the delegated providers that preserve stream separation.
  - confidence: 0.84
