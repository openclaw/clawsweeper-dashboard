## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate the API origin before attaching the token:** `internal/providers/smolvm/client.go:151-152`
  - body: `cfg.Smolvm.BaseURL` can be supplied from config/env/flags, but `newAPI` accepts it as-is and the client later sends `Authorization: Bearer ...` plus archive payloads there. Require HTTPS and a trusted-host or explicit opt-in boundary before this provider can safely use `smk_*` credentials.
  - confidence: 0.86
- **[P1] Honor sync.include before building SmolVM archives:** `internal/providers/smolvm/sync.go:21`
  - body: `syncWorkspace` builds the archive manifest with excludes only, so a repo using `sync.include` will still upload files outside the whitelist. Thread `cfg.Sync.Include` through to `BuildSyncManifestFiltered`, matching the existing delegated providers, before injecting the archive.
  - confidence: 0.94
- **[P1] Decode snake_case exec exit codes:** `internal/providers/smolvm/client.go:59`
  - body: A struct tag cannot define two `json` keys, so this field only decodes `exitCode`; a response with `exit_code` leaves `ExitCode` at zero and can turn a failed remote command into success. Add a real fallback field or custom unmarshal with regression coverage.
  - confidence: 0.82
- **[P2] Route remote stderr to stderr:** `internal/providers/smolvm/client.go:255-257`
  - body: `ExecStream` writes `result.Stderr` to the stdout writer, and the backend passes `Runtime.Stdout`, so commands that keep stdout machine-readable get stderr mixed into it. Extend the stream path to accept `Runtime.Stderr` and test that stdout and stderr stay separate.
  - confidence: 0.95
