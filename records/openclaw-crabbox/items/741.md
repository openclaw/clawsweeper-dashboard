## Label Justifications

- `P1`: This is security hardening for a documented privileged image-prep workflow whose compromise can persist into reusable Windows developer images.
- `merge-risk: 🚨 compatibility`: The PR changes the default Windows image prep to a hash-pinned fail-closed bootstrap, which can break existing bakes when upstream installer bytes change unless maintainers own the update path.

## Summary

Keep open: this PR is a plausible narrow security fix, but it is not merge-ready because the proof is still only a static script regression and the new fail-closed Chocolatey pin is compatibility-sensitive. Current main still has the unverified Chocolatey bootstrap, so the linked security issue should remain open until a reviewed fix lands.

## What This Changes

The PR changes Windows developer-tools prep to download Chocolatey's installer to a temporary file, verify a SHA-256 digest before execution, add URL/digest environment overrides, and add a Node static regression test.

## Best Possible Solution

Land a maintainer-reviewed version that verifies Chocolatey's installer before execution, owns the digest refresh or override policy, includes redacted real Windows prep proof, and leaves the Node/Docker hardening to https://github.com/openclaw/crabbox/issues/686.

## Reproduction Assessment

Yes. Source inspection on current main shows the default Windows image bake selects the affected prep script, and that script executes Chocolatey's downloaded installer through DownloadString and Invoke-Expression without integrity verification; no live Windows bake was run.

## Solution Assessment

Yes, mostly. The PR follows the narrow download, hash, fail-closed, then execute flow, but merge readiness still depends on real Windows prep proof and maintainer acceptance of the fail-closed digest/update policy.
