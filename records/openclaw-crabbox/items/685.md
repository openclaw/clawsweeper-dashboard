## Label Justifications

- `P1`: The issue is an urgent current-main supply-chain integrity gap in a documented privileged image-prep workflow, but the affected script is not proven shipped in the latest release.
- `impact:security`: The report concerns arbitrary privileged code execution through an unverified downloaded installer in a supported image-build path.

## Summary

Keep open: current main still executes Chocolatey's remote install script through `Invoke-Expression` in the default Windows developer-image prep path, while the repository's security policy and Windows bootstrap docs expect managed downloaded artifacts to be integrity-verified. No open or merged PR currently owns the Chocolatey fix; https://github.com/openclaw/crabbox/issues/686 is a related sibling for Node and Docker artifacts.

## What This Changes

The issue asks Crabbox to stop the documented Windows developer-image prep from executing the Chocolatey installer without pinned integrity verification.

## Best Possible Solution

Add a reusable PowerShell download-and-SHA-256 verification helper to the Windows developer-tools prep script, pin the Chocolatey installer to a reviewed source and digest before execution, fail closed on mismatch, and coordinate the helper with https://github.com/openclaw/crabbox/issues/686.

## Reproduction Assessment

Yes. Source inspection on current main shows the default Windows image bake selects the affected prep script and runs it as `SYSTEM`; that script executes Chocolatey's downloaded `install.ps1` without an integrity check. A live Windows bake was not run.

## Solution Assessment

Yes. The narrow maintainable fix is to download the Chocolatey installer to a file, verify a reviewed SHA-256 or equivalent pinned identity before execution, fail closed on mismatch, and add regression coverage; the same helper can support the sibling Node/Docker artifact issue.
