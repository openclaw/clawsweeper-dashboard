Full review comments:

- **[P1] Validate bootstrap_dir before deleting it:** `internal/providers/localcontainer/backend.go:264-267`
  - body: `bootstrap_dir` comes back from Docker labels, and cleanup scans any container carrying the Crabbox local-container labels. A stray or user-created labeled container with `bootstrap_dir` pointed at an unrelated host directory would make cleanup call `os.RemoveAll` on that path. The existing `host_work_root` cleanup path validates trusted roots before deleting; this new path needs the same kind of ownership guard.
  - confidence: 0.88
- **[P2] Clean up bootstrap dirs on acquire failure:** `internal/providers/localcontainer/backend.go:464`
  - body: After `createContainer` returns, failures in inspect, SSH readiness, or claim creation remove the container directly from `Acquire`, but no caller knows the `bootstrapDir` path to remove. The old temp file was removed by a defer in `createContainer`; failed acquire attempts now leak `/tmp/crabbox-bootstrap-*` directories unless the path is handed back or cleanup is otherwise wired through.
  - confidence: 0.82
