## Summary

Keep open: the status fields are useful, but the patch currently misses timing-record-only storage and delegated outcome preservation, so it can emit incomplete or misclassified status metadata.

## What This Changes

The PR adds `runStatus` and optional `errorKind` to CLI timing reports, updates run/provider docs, and adds Go tests for default and delegated timing JSON behavior.

## Best Possible Solution

Normalize status fields before every timing consumer, and populate delegated reports from `FinalizeRunResult(result, runErr)` with focused coverage for `--timing-json`, `--timing-record`, and delegated timeout/provider-error cases.

## Reproduction Assessment

Yes for the review findings: source inspection shows `--timing-record` appends before `writeTimingJSON`, and delegated reports do not copy available `RunResult` outcome metadata.

## Solution Assessment

No, not yet. Adding the fields is the right direction, but the implementation needs to normalize at the report/storage boundary and preserve provider-specific delegated outcomes.

- **[P2] Normalize timing records before appending them:** `internal/cli/timing.go:55`
  - body: The new status fields are finalized only inside `writeTimingJSON`, but `runCommand` appends the benchmark record before that call and returns early when only `--timing-record` is set.   - confidence: 0.93
- **[P2] Preserve delegated run outcomes:** `internal/cli/timing.go:63-68`
  - body: Defaulting from only `RunResult{ExitCode: report.ExitCode}` collapses delegated timeouts, cancellations, and provider errors into exit-code-derived statuses unless the provider separately writes captured timing JSON. `timingReportFromDelegatedRunResult` already has `result` and `runErr`; populate the new fields from `FinalizeRunResult(result, runErr)` there before serialization.
  - confidence: 0.9
