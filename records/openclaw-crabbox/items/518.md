- `impact:security`: The issue concerns possible disclosure of a configured Semaphore API token in terminal, CI, or copied diagnostics.

## Summary

Keep this issue open: current main still sends the configured Semaphore token and returns non-2xx Semaphore response bodies through user-visible errors without redaction, and no Semaphore-specific fix PR or duplicate canonical item is present.

## What This Changes

The issue asks Crabbox to redact the configured Semaphore provider token from Semaphore API error bodies before surfacing diagnostics.

## Best Possible Solution

Add provider-local Semaphore error redaction in `getWithHeaders`, `get`, and `post`, with regression tests for reflected-token non-2xx bodies while preserving status and path context.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a fake Semaphore response returning a non-2xx body containing the configured token would be stored in `semaphoreAPIError.Body` and formatted unchanged; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to redact the configured token before constructing Semaphore API errors, matching existing provider-local redaction helpers in adjacent clients.
