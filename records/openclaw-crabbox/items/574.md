## Label Justifications

- `P1`: The issue describes a concrete security boundary problem in an admin route that can perform destructive operations against unowned cloud resources.
- `impact:security`: The report is about enforcing Crabbox's supported provider ownership boundary before destructive actions.
- `impact:data-loss`: The affected operation can delete AMIs, snapshots, managed images, or machine images that may be recovery or golden-image assets outside Crabbox ownership.

## Summary

The issue is still necessary: current main still lets the admin image deletion route forward caller-supplied AWS, Azure, and GCP image or snapshot identifiers to provider delete calls without first proving Crabbox ownership, except for the narrow AWS promoted-image block.

## What This Changes

Require Crabbox image deletion to fail closed unless stored metadata or verified provider ownership markers prove the target image or snapshot belongs to Crabbox.

## Best Possible Solution

Implement a fail-closed ownership gate for every default image and snapshot deletion path, keep the AWS promoted-image protection, and make any emergency force-delete flow explicit, audited, and separate from normal image deletion.

## Reproduction Assessment

Yes from source inspection, but not from a live provider run. The current route and tests show admin deletion reaches provider delete hooks unless the narrow AWS promoted-image check blocks it.

## Solution Assessment

Yes in direction: fail-closed ownership validation is the narrowest maintainable fix for the supported security boundary. The remaining maintainer decision is whether to add an explicit force-delete escape hatch and how to document upgrade behavior.
