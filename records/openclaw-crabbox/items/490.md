## Summary

Current main still matches the reported Cloudflare delegated-run redirect exposure: the Cloudflare client validates the configured runner URL but stores an unguarded HTTP client, while adjacent credentialed provider clients reject cross-origin redirects. This should stay open for maintainer security handling rather than cleanup closure.

## What This Changes

The issue reports that Cloudflare delegated-run requests can replay command, environment, and upload bodies to a cross-origin redirect target.

## Best Possible Solution

Add a Cloudflare-specific secure HTTP client wrapper that preserves same-origin redirects and existing caller redirect policy while rejecting cross-origin redirects for exec, upload, and JSON runner requests.

## Reproduction Assessment

Yes, from source inspection and the issue's proof description: Cloudflare exec requests use replayable buffers through an unguarded client, and no current test covers cross-origin redirect refusal. I did not rerun the dynamic proof because the review is read-only.

## Solution Assessment

Yes, the recommended provider-local CheckRedirect wrapper is the narrow maintainable fix and matches the existing pattern in adjacent provider clients. It should include regression tests for exec, upload, same-origin redirects, and caller-supplied redirect policy.
