## Label Justifications

- `P0`: The issue reports a current coordinator auth and authorization bypass where a signed GitHub token can carry an unverified owner with cross-owner and admin-owner consequences.
- `impact:security`: The affected behavior concerns trusted identity binding, signed credentials, owner-scoped authorization, and optional admin promotion.
- `impact:auth-provider`: The affected flow is GitHub OAuth identity resolution and the signed user-token auth path.

## Summary

Keep open: current main and the latest release still initialize the GitHub token owner from the public profile email before looking for verified emails, then trust the signed owner for normal scoping and optional admin-owner promotion.

## What This Changes

The issue requests hardening GitHub OAuth identity resolution so signed user tokens cannot use an unverified public profile email as the trusted owner.

## Best Possible Solution

Bind `owner` only to a verified GitHub email or a documented stable non-email fallback, then add no-verified-email and admin-owner regression tests.

## Reproduction Assessment

Yes, source-reproducible. Current main initializes the GitHub owner from the public profile email, only replaces it with verified `/user/emails` data, signs it, and later trusts it for owner/admin decisions.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. The narrow maintainable fix is to remove the public-email fallback and either require a verified email or use a documented stable login-derived fallback.
