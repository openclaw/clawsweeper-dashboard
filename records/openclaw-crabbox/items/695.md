- `impact:security`: The reported failure crosses a security boundary by creating unintended command execution from forwarded environment variable names.

## Summary

Keep open: current main still lets invalid local environment names selected by a wildcard allowlist reach SSH/PowerShell command-prefix rendering, despite docs saying invalid names are dropped. The prior env-forwarding security fix addressed a different precedence bug, not this invalid-name injection path.

## What This Changes

The issue requests validation of inline forwarded environment variable names before SSH-backed command construction so invalid names cannot inject shell commands.

## Best Possible Solution

Filter invalid names at env-selection time so summaries and behavior match, defensively skip invalid keys in POSIX and Windows command-prefix writers, and add focused regression tests for wildcard-allowed invalid inline env names.

## Reproduction Assessment

Yes from source inspection: a wildcard allowlist can select an invalid local environment name and the remote command prefix writes that name directly into shell syntax. I did not provision a live lease in this read-only review.

## Solution Assessment

Yes: validating or dropping invalid inline env names before any command-prefix writer sees them is the narrowest maintainable fix. The writers should also defensively skip invalid keys so future call sites cannot reintroduce the class.
