## Summary

Keep open: current main still allows an unauthenticated request with a JWT-shaped Cloudflare Access header to trigger Access cert lookup before Crabbox bearer-token authentication rejects the request.

## What This Changes

The issue requests hardening Cloudflare Access JWT verification so unknown attacker-controlled `kid` values cannot force repeated pre-auth cert fetches.

## Best Possible Solution

Harden the existing Access auth path by avoiding network key lookup until a Crabbox credential is otherwise acceptable, bounding header and `kid` input, and caching/throttling Access JWK misses with focused regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: send a protected coordinator request with no valid Crabbox bearer token and a JWT-shaped `cf-access-jwt-assertion` header whose decoded header has `alg: "RS256"` and an unknown `kid`; current main reaches `fetch("https://<teamDomain>/cdn-cgi/access/certs")` before returning unauthorized.

## Solution Assessment

Yes, the requested hardening direction is the narrow maintainable fix. The safer path is to keep Cloudflare Access as an optional identity source for already-authenticated Crabbox requests while bounding and caching cert lookup work, not to broaden route behavior or add a new auth mode.
