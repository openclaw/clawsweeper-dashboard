## Label Justifications

- `P2`: The report describes a normal-priority security integrity bug with limited audit-history impact and authenticated preconditions.
- `impact:security`: The issue concerns authorization boundaries around shared leases and cross-owner run audit visibility.

## Summary

Keep open: current main still permits a run owner to post a visible use-shared lease ID in run event metadata, which can add that lease owner to the run audit attribution. The older shared-lease audit fix is related but does not cover this post-hoc re-tagging path.

## What This Changes

Security report requests preventing client-supplied run event lease IDs from re-attributing an existing run into another lease owner's audit history.

## Best Possible Solution

Keep the issue open and land a focused security fix that rejects or ignores event-supplied lease IDs which would add a new backing lease owner without validated server-side lease replacement.

## Reproduction Assessment

Yes. Source inspection on current main shows an owned run plus a visible use-shared victim lease can pass the event endpoint guard, store the event lease ID, and append the victim owner to run.leaseOwners; I did not run the Worker test harness in this read-only review.

## Solution Assessment

Unclear as an implementation direction. Rejecting or ignoring event-only cross-owner lease IDs is the likely narrow repair, but maintainers should decide how to preserve the legitimate replacement-flow contract.
