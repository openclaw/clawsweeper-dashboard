## Label Justifications

- `P1`: The issue reports a live coordinator authorization failure where a revoked shared manager can retain an active network proxy path through another owner's lease.
- `impact:security`: The affected behavior crosses Crabbox's documented coordinator ownership and sharing authorization boundary.
- `impact:session-state`: The stale authorization persists in active WebSocket bridge/session state after the stored lease share changes.

## Summary

Keep open: current main still accepts and forwards egress bridges without a principal-bearing attachment, while share-shrink revocation only closes WebVNC and Code viewers.

## What This Changes

A revoked shared lease manager can keep active egress host/client bridge sockets after the owner shrinks or removes lease sharing.

## Best Possible Solution

Persist the accepted egress bridge principal, re-authorize and close egress host/client sockets on share shrink and restored-socket reconciliation, and add Worker regression tests proving revoked egress bridges close while authorized bridges survive.

## Reproduction Assessment

Yes. Source tracing on current main shows share shrink revokes WebVNC and Code viewers only, while egress bridge attachments lack principal metadata and continue forwarding by leaseID/sessionID.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing share-shrink and restored-bridge revocation model to egress, rather than changing the CLI relay or adding a new policy mode.
