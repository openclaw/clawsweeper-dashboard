## Label Justifications

- `P1`: The report describes a high-impact supply-chain integrity failure in a documented managed lease workflow, though exploitation depends on upstream installer compromise and operator use of `--code`.
- `impact:security`: The affected behavior executes externally downloaded installer bytes as root and can compromise the managed lease and code-server bridge environment.

## Summary

Keep open: current main still executes the unpinned code-server installer as root in both bootstrap generators, and the repo security policy treats downloaded artifact integrity failures in documented workflows as in scope. The same code-server-specific work is also tracked in https://github.com/openclaw/crabbox/issues/724, with broader overlap in https://github.com/openclaw/crabbox/issues/715 and https://github.com/openclaw/crabbox/issues/711, so maintainers may want to consolidate manually rather than close this security report automatically.

## What This Changes

Request to replace the unpinned root-executed code-server install script in managed Linux `--code` bootstrap with a pinned and verified install path.

## Best Possible Solution

Implement one pinned, versioned, digest-verified code-server bootstrap path in both Go and Worker generators, add matching regression tests, and consolidate the duplicate public reports under a single security-tracked issue.

## Reproduction Assessment

Yes for source-level reproduction: enabling the Code capability still causes generated bootstrap to include the root `curl ... | sh` code-server install pipeline, and focused search found no digest or signature verification. I did not attempt a live upstream compromise path.

## Solution Assessment

Yes: replacing the mutable pipe-to-shell installer with a pinned, digest-verified release artifact mirrors the repository's documented Windows bootstrap integrity pattern. Maintainers should still approve the code-server version, digest source, and update process before landing it.
