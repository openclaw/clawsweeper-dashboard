## Summary

Keep this PR open: the feature is not on current main and the branch is a useful implementation candidate, but it still needs an early Apple `container` CLI capability/version gate before merge because the provider unconditionally invokes `container run` while older released Apple CLI sources do not expose that subcommand.

## What This Changes

Adds a built-in `apple-container` local SSH-lease provider with config/env/flags, docs, registration, lifecycle code, and provider regression tests.

## Best Possible Solution

Keep the provider-owned adapter, but add an early capability/version check and documentation for the minimum Apple `container` CLI release, or switch the implementation to a compatible create/start lifecycle before landing.

## Reproduction Assessment

Yes. Source inspection shows the PR invokes `container run`, while Apple container 0.3.0 and 0.4.0 source trees do not include the `ContainerRun` command, so those installs can pass the current doctor path and fail when creating a lease.

## Solution Assessment

No. The provider direction is reasonable, but the mergeable solution needs an early CLI capability/version gate or an older create/start lifecycle so users do not discover unsupported runtime versions only after warmup/run fails.
