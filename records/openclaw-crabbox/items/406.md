## Label Justifications

- `P1`: The issue describes repository-controlled local command execution and credential exposure through an existing provider path, which needs urgent maintainer security review.
- `impact:security`: The affected path crosses Crabbox's local trust boundary and can run attacker-selected commands under the operator account.
- `impact:auth-provider`: The report centers on External provider configuration and routing fields loaded from repository config.

## Summary

Keep open: current main still lets untrusted repository config populate External provider command, lifecycle, connection, routing, and SSH proxy-command fields that later reach local execution sinks, and I found no canonical merged fix or duplicate that owns this security boundary.

## What This Changes

The issue asks Crabbox to prevent repository-local `.crabbox.yaml` from configuring External provider local command, lifecycle, connection, routing, and proxy-command execution paths.

## Best Possible Solution

Gate External fields that can influence local command execution, SSH proxying, routing state, or local work roots behind trusted user or explicit config, then deliberately reopen only safe project-level fields with regression coverage.

## Reproduction Assessment

Yes, the source-level reproduction path is high confidence: load untrusted project config with `provider: external`, observe `file.External` copied without a trusted guard, then follow the configured command or proxy field to local execution sinks. I did not run a live command because this was a read-only review.

## Solution Assessment

Yes with maintainer security review: trusted-gating dangerous External fields is the narrow maintainable fix, but maintainers should decide whether any non-executable External settings remain safe in project config.
