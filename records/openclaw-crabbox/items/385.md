## Label Justifications

- `P1`: The issue is a source-supported authorization-boundary failure that can let a lower-privileged shared user affect another user's active Code workspace.
- `impact:security`: The affected behavior is a share-role authorization bypass across the trusted Code portal and bridge boundary.

## Summary

Keep open: current main and the latest release still let any visible Code portal caller reach the Code proxy, while the nearby ticket route already requires manage access and the docs make the portal plus bridge ticket the trusted boundary.

## What This Changes

The issue asks Crabbox to stop `use` share recipients from sending write-capable Code portal proxy traffic to another user's code-server bridge.

## Best Possible Solution

Add role-aware authorization before any Code HTTP or WebSocket proxy forwarding, with focused Worker regressions for owner, admin, manage-share, and use-share callers.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: a `use` share can satisfy lease visibility, reach `codePortalProxy`, and have write-capable HTTP methods forwarded through the connected Code bridge.

## Solution Assessment

The direction is right, but the final policy needs maintainer security judgment. The narrowest safe default is to require manage access for all proxied Code traffic unless maintainers design and test real read-only Code semantics.
