## Summary

Keep this issue open: current main has no OVHcloud provider, and no other issue or pull request owns the request. The shipped external-provider protocol offers a viable experimentation path, but it does not provide the requested named, first-party OVH configuration and UX.

## What This Changes

Add a first-party direct OVHcloud Public Cloud provider supporting Linux SSH-lease provisioning, lifecycle commands, configuration, cleanup, diagnostics, tests, and documentation.

## Best Possible Solution

Establish an OVH adapter against the external-provider protocol first, then promote it to a bundled `ovh` provider only if maintainers explicitly accept the first-party UX and long-term support boundary.

## Reproduction Assessment

Not applicable; this requests a new provider rather than reporting broken existing behavior. Current-main inspection confirms that a named OVH provider is not presently offered.

## Solution Assessment

No—not without a product decision. A provider adapter is the correct architecture, but the shipped external-provider protocol should be evaluated before Crabbox assumes first-party OVH configuration, credential, and API-compatibility ownership.
