## Label Justifications

- `P0`: Affected deployments can turn one leaked shared automation credential into forged owner/org identity and possible admin identity.
- `impact:security`: The issue is about credential-role separation and trusted identity forgery at the coordinator boundary.
- `impact:auth-provider`: The affected logic is the coordinator authentication provider path for shared bearer, signed user, and admin identity.

## Summary

This issue remains necessary: current main and the latest release still use the shared automation token as the fallback HMAC key for signed `cbxu_` user tokens, while the surrounding tests and docs preserve that behavior. Prior related auth hardening fixed different shared-token and admin-claim paths, but did not address this credential-role coupling.

## What This Changes

Security report that a leaked shared automation token can forge signed `cbxu_` owner/org/login identity when `CRABBOX_SESSION_SECRET` is absent.

## Best Possible Solution

Require independent signing material for `cbxu_` user tokens and update docs/tests so shared-token automation remains non-admin while fallback-signed user tokens are rejected or intentionally migrated.

## Reproduction Assessment

Yes from source inspection: current main and `v0.31.0` both sign and verify `cbxu_` tokens through `sessionSecret(env)`, which falls back to `CRABBOX_SHARED_TOKEN`. I did not run a live exploit request in this read-only review.

## Solution Assessment

Unclear; the secure endpoint is independent user-token signing material, but maintainers should decide whether the repository should fail closed immediately or provide a warning/migration window for existing fallback deployments.
