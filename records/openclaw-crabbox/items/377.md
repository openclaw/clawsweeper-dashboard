## Summary

This issue should stay open: current main and the latest release still use `CRABBOX_SHARED_TOKEN` as the fallback HMAC key for signed `cbxu_` user tokens, and the merged related PR only changed documentation/policy while explicitly saying runtime behavior was unchanged.

## What This Changes

Request to separate shared automation bearer-token credentials from signed user-token session credentials while preserving shared-token-only automation deployments.

## Best Possible Solution

Implement compatibility-aware auth hardening that requires independent session-signing material for browser/user tokens, keeps shared-token-only automation working, emits precise startup/configuration feedback, documents token rotation, and updates Worker regression tests plus docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with only `CRABBOX_SHARED_TOKEN` set, a `cbxu_` payload signed with that token is verified through `sessionSecret(env)` and then trusted as GitHub user identity.

## Solution Assessment

Yes, the accepted direction is the right boundary: signed user tokens should use independent session-signing material while shared-token automation remains non-admin. The implementation should avoid a broad auth rewrite and focus on auth, startup/config diagnostics, tests, and docs.
