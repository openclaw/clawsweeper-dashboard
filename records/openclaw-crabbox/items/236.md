## Summary

Keep this PR open: it adds a useful provider, but the latest env-forwarding change introduces a source-reproducible secret-retention bug when `--keep-on-failure` preserves a failed sandbox.

## What This Changes

Adds an OpenComputer delegated-run provider using the `oc` CLI and a direct file API path for env forwarding, with config, docs, provider registration, archive sync, bridge handling, and tests.

## Best Possible Solution

Clean env profiles regardless of final sandbox retention state, align security guidance with the header/body API path, then let maintainers decide whether to accept the new provider boundary with the supplied e2e proof.

## Reproduction Assessment

Yes. Source inspection shows `--allow-env` uploads a profile, skips cleanup while `shouldStop` is initially true, then `--keep-on-failure` can flip `shouldStop` false and preserve the sandbox after a failed command.

## Solution Assessment

No. The off-argv file API approach can be maintainable, but this PR should first scrub env profiles before any failure-retained sandbox is returned and align the security guidance with the implementation.

- **[P1] Clean env profiles before retaining failed sandboxes:** `internal/providers/opencomputer/backend.go:144-145`
  - body: When env forwarding is used on a newly created run, `shouldStop` is true here so the cleanup defer is skipped.
