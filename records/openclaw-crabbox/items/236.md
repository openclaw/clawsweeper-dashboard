## Summary

## What This Changes

## Best Possible Solution

Keep the provider PR open, fix env rejection to distinguish explicit forwarding from implicit defaults, add regression coverage for both paths, and leave release notes to the release-owned changelog flow.

## Reproduction Assessment

Yes. Source inspection shows `selectRunEnv` can populate `RunRequest.Env` with default `CI`/`NODE_OPTIONS` while `EnvSummary` remains false, and the OpenComputer backend rejects any non-empty env map before sandbox creation.

## Solution Assessment

No. Rejecting explicit env forwarding is the right security direction, but keying the rejection only on `len(req.Env)` is too broad; the narrow fix is to reject explicit env requests while ignoring or dropping implicit defaults.
