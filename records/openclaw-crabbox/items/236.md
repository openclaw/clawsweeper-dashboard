## Summary

Keep open: the implementation is well-tested and has credible live validation, but the exact head contains an unbounded cleanup path that can prevent a completed run from returning; the new external execution boundary also needs explicit maintainer acceptance.

## What This Changes

Adds an OpenComputer delegated-run provider backed by its REST API, including configuration, lifecycle, archive sync, command execution, documentation, and tests.

## Best Possible Solution

Keep the provider-specific REST implementation inside its adapter, add short timeout contexts around all best-effort cleanup requests with stalled-transport coverage, and merge only after maintainers explicitly accept the opt-in OpenComputer data boundary.

## Reproduction Assessment

Yes for the review defect: both cleanup sites use `context.Background()` and the selected HTTP client has no guaranteed global timeout, providing a direct source-level path where a response that never arrives blocks completion.

## Solution Assessment

Mostly yes: the direct REST adapter is narrower and better aligned with provider isolation than a CLI hybrid, but best-effort cleanup must be deadline-bounded before the implementation is safe to merge.
