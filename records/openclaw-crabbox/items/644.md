## Summary

Keep open: current main and the latest release still build the Tart SSH target without `SSHConfigProxy`, so Tart readiness can still take the raw Go TCP probe path that the reporter showed failing while OpenSSH succeeds. No canonical duplicate or linked fix PR was found.

## What This Changes

The issue asks for Tart SSH readiness to use the OpenSSH/SSH-config path, or an equivalent fallback, when OpenSSH can connect even though Go's raw TCP dial reports the port closed.

## Best Possible Solution

Make the Tart adapter use OpenSSH readiness for its guest-IP target, or add an equivalent provider-local fallback, with a focused regression test proving Tart lease targets take the OpenSSH path while preserving the current ready command and no fallback ports.

## Reproduction Assessment

No live current-main reproduction was run in this review because the failure requires an Apple Silicon Tart host with the reported networking behavior. Source inspection gives a clear path: Tart leaves `SSHConfigProxy` false, so core readiness uses the raw Go TCP dial branch that the reporter showed failing.

## Solution Assessment

Yes, a Tart-provider-local readiness change is the narrowest maintainable fix. It aligns with the provider boundary because Tart-specific connection semantics belong in `internal/providers/tart`, not in a core `provider == tart` branch.
