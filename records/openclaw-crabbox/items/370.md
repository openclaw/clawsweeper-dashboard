## Summary

Keep open: current main and the latest release still implement the runtime adapter ID as a durable first-ticket claim, and the reporter’s later comment narrows this from a security-boundary report to product/availability hardening that needs a maintainer-chosen ownership or recovery contract.

## What This Changes

The issue requests an explicit ownership and recovery contract for runtime adapter IDs instead of allowing the first ticket request to globally claim an arbitrary DNS-style adapter ID.

## Best Possible Solution

Define and document a durable runtime adapter ID ownership/recovery model that requires ownership proof before persistence or provides an authorized reclaim path, while preserving the existing owner/org relay rejection.

## Reproduction Assessment

Yes. Source inspection on current main shows the first ticket request can persist the adapter identity and later registration by another owner/org follows the conflict path; existing Worker tests encode that flow, but no runtime test was run in this read-only review.

## Solution Assessment

Unclear. The report identifies a real source-visible first-claim behavior, but maintainers need to choose the intended claim model before implementation: lease-bound proof, generated IDs, reserved IDs, or an explicit reclaim flow.
