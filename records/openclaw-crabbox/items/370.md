- `impact:other`: The concrete supported impact is availability denial for a contested adapter ID, which does not fit the more specific owned impact labels.

## Summary

Keep open: current main still persists runtime adapter identity records as a first ticket side effect and later rejects another owner/org for the same adapter ID, so the reported namespace-squatting concern is not obsolete or superseded.

## What This Changes

The issue asks Crabbox to prevent arbitrary authenticated users from first-claiming global runtime adapter IDs without proving ownership of the lease or deployment that will use them.

## Best Possible Solution

Define a durable runtime adapter identity contract that requires ownership proof before an adapter ID is persisted for an owner/org, while preserving the existing cross-owner relay rejection and adding regression tests for both boundaries.

## Reproduction Assessment

Yes. Current main still lets the first ticket request persist an identity for an adapter ID, and later registration by another owner/org follows the conflict path; this was verified by source and existing Worker tests rather than by running Vitest.

## Solution Assessment

Unclear. The report names plausible repairs, but maintainers need to choose the intended claim model: lease-bound proof, server-generated IDs, reserved IDs, or an explicit claim/reclaim flow.
