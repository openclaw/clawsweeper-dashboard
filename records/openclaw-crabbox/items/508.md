## Summary

Keep open: current main still leaves RunPod's bearer-auth REST client on the caller/default redirect policy while same-origin redirect guards exist in sibling provider clients, so the credential-exposure report remains actionable.

## What This Changes

The issue requests hardening the RunPod REST client so provider bearer credentials and pod-create bodies are not sent across cross-origin redirects.

## Best Possible Solution

Clone the RunPod HTTP client in newRunpodClient, add a same-origin CheckRedirect guard that preserves caller redirect behavior and the default limit, and cover cross-origin refusal plus same-origin behavior in focused tests.

## Reproduction Assessment

Yes. Source inspection shows RunPod stores an unwrapped HTTP client, sets Authorization: Bearer on REST requests, and sends POST /pods through that client; I did not run a live redirect proof in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to mirror the established same-origin redirect wrapper used by sibling credentialed provider clients and add RunPod-specific regression coverage.
