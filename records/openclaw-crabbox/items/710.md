- `impact:security`: The report concerns a lower-privileged share recipient crossing the lease owner/run history authorization boundary.

## Summary

Keep open: current main still allows a use-share recipient to create an attacker-owned run against a visible victim lease and immediately add the victim owner to readable audit attribution. The open related issue covers event-time re-tagging, but this creation-time path is distinct and unresolved.

## What This Changes

The issue asks Crabbox to prevent use-only lease-share recipients from binding attacker-owned runs, events, and logs into a lease owner’s run history.

## Best Possible Solution

Decide and document the intended share-role semantics, then land a focused security change so use-only visibility cannot create or expand owner-visible run attribution unless that is explicitly approved by the lease owner/manage policy; keep the event re-tagging half coordinated with https://github.com/openclaw/crabbox/issues/684.

## Reproduction Assessment

Yes. Current source and tests show Bob with a use share can POST /v1/runs using Alice's leaseID, receive 201, and cause Alice to appear in run.leaseOwners; I did not run the Worker tests because this review is read-only.

## Solution Assessment

Unclear as an implementation direction. Requiring owner/manage access for createRun lease attribution is the narrow obvious repair, but maintainers need to confirm the intended use-share and replacement-flow audit contract before changing shipped behavior.
