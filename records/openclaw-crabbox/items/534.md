## Label Justifications

- `P2`: This is a concrete coordinator credential-scope bug with meaningful but constrained security impact.
- `impact:security`: The issue is about a service bearer token crossing a documented authorization boundary into interactive terminal access.

## Summary

Keep open: current main still authenticates the runtime-adapter bearer token for the whole `/v1/workspaces/*` tree, which includes the terminal WebSocket route, while docs describe the token as lifecycle API access only.

## What This Changes

The issue requests tightening `CRABBOX_RUNTIME_ADAPTER_TOKEN` authorization so the service token cannot attach to workspace terminal WebSockets outside the documented runtime-adapter surface.

## Best Possible Solution

Restrict runtime-adapter service auth to the intended lifecycle and desktop relay allowlist, reject terminal attach for that token, and add regression tests that preserve normal user terminal access.

## Reproduction Assessment

Yes, source-reproducible: current main accepts the runtime-adapter token for `/v1/workspaces/:id/terminal`, and an existing Worker test pins that authentication result. I did not run an end-to-end WebSocket harness in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is to share or mirror the runtime-adapter relay route allowlist in `runtimeAdapterServiceAuth()` and flip the terminal-token test to rejection. If maintainers intentionally want service terminal access, that should be a separate explicit capability rather than the current broad prefix match.
