## Label Justifications

- `P1`: An authenticated non-admin caller can cross a broker/provider authorization boundary, but the proven impact is limited to provider key identity and cleanup rather than full credential disclosure.
- `impact:security`: The issue concerns non-admin control over provider resource identifiers and destructive cleanup under coordinator/provider credentials.

## Summary

The issue is still necessary: current main still lets a non-admin lease request carry `providerKey`, stores that value on the lease, and later uses the stored name for AWS/Hetzner provider-key cleanup when it matches the Crabbox key pattern. The source path is direct enough to keep open for security review, but remediation needs maintainer judgment because it changes broker authorization and provider cleanup semantics.

## What This Changes

The issue asks Crabbox to stop non-admin broker lease requests from choosing provider SSH key names that later drive provider key reuse, labeling, and cleanup.

## Best Possible Solution

Derive non-admin broker provider keys server-side from the final lease identity, preserve any manual override only behind an explicit admin/internal path, and bind provider key cleanup to lease ownership rather than a caller-controlled name pattern.

## Reproduction Assessment

Yes. A crafted non-admin `POST /v1/leases` can carry `providerKey`, and source inspection shows that value reaches lease storage, AWS/Hetzner key identity, provider labels, and cleanup predicates without server-side enforcement of the CLI-derived per-lease invariant.

## Solution Assessment

Mostly yes, but maintainer policy is needed for overrides. The narrow technical direction is to derive non-admin provider keys from the lease server-side and require admin-owned ownership metadata or a deliberate override path for any manual provider key.
