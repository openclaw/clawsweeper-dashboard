## Summary

Keep open: current main still lets `workflow_dispatch` check out an operator-supplied ref and, when the helper directory is absent, replace the stashed reviewed GoReleaser config with that ref's `.goreleaser.yaml` before the credentialed GoReleaser step runs. The closed mutable-action-tag issue at https://github.com/openclaw/crabbox/issues/387 and its merged fix at https://github.com/openclaw/crabbox/pull/409 are adjacent release hardening, but they do not address this config-selection path.

## What This Changes

The issue reports that the manual release workflow can run GoReleaser with release credentials using configuration taken from the dispatcher-selected ref.

## Best Possible Solution

Make the credentialed release path use only reviewed current-branch or explicitly allowlisted GoReleaser configs, validate `inputs.tag` as an allowed protected release tag, and update the release workflow test so selected refs cannot provide the publish config by default.

## Reproduction Assessment

Yes at source level: current main shows the manual release checkout, fallback config replacement, and credentialed GoReleaser step without needing a live workflow dispatch. A full live exploit proof would depend on repository workflow-dispatch and ref-protection settings.

## Solution Assessment

Yes for the narrow direction: decouple publish config from the dispatcher-selected ref and cover it with the existing release workflow test. The exact historical-tag compatibility policy should be decided by maintainers before implementation.
