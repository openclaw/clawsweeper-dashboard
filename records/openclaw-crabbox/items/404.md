## Label Justifications

- `P2`: The issue is a concrete authorization-boundary bug with limited direct impact: a lower-role shared viewer can change a desktop preference but this route does not itself grant broader desktop input control.
- `impact:security`: The report concerns a `use` share crossing an intended owner/manage authorization boundary for a state-changing WebVNC bridge message.

## Summary

The report remains necessary: current `main` still lets an authenticated `use` share viewer send a `desktop_theme` frame when that viewer is the active WebVNC controller, while the same share role is intentionally visible but not manageable. The issue should stay open for maintainer/security review because the final fix needs to preserve intended WebVNC sharing semantics while closing the role-boundary write path.

## What This Changes

Require desktop theme writes for shared WebVNC sessions to respect owner/admin/`manage` authorization instead of controller status alone.

## Best Possible Solution

Harden `webVNCTheme` and the portal theme sync so only owner/admin/`manage` callers can send `desktop_theme` bridge frames, with regression coverage for `use`, `manage`, and owner paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `use`-shared portal user can be an active WebVNC controller, and `webVNCTheme` allows the controller branch without requiring `leaseManageableByRequest`; I did not run Vitest in this read-only review.

## Solution Assessment

Unclear until maintainer security review. Requiring manage access for theme writes is the narrowest conservative fix, but maintainers should confirm whether controller-only desktop appearance sync was intended to carry write authority.
