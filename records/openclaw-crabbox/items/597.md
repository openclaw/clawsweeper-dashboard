## Label Justifications

- `P2`: This is a normal-priority provider credential exposure issue with a bounded redirect precondition and limited blast radius.
- `impact:security`: The issue is about a provider bearer token being replayed outside the documented credential-destination boundary.
- `impact:auth-provider`: The affected behavior is provider authentication material and HTTP destination handling for the SmolVM provider.

## Summary

Keep open: current main still stores the SmolVM API client without a provider redirect guard, while SmolVM bearer-token requests are sent through `c.http.Do` after `Authorization` is attached. The report is security-sensitive and already has source-level evidence, so it should remain for maintainer/security review rather than be closed or automatically repaired.

## What This Changes

The issue asks Crabbox to add SmolVM provider redirect handling that refuses credential-bearing redirects when scheme, hostname, or effective port changes.

## Best Possible Solution

Add a SmolVM-local secure HTTP client wrapper matching the RunPod/Freestyle origin check, then add focused SmolVM redirect tests for same-origin allow, scheme/port/host refusal, and caller `CheckRedirect` preservation.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: SmolVM attaches `Authorization`, uses an unwrapped `http.Client`, and current tests lack a redirect guard while sibling providers implement one. I did not run a live redirect server in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to wrap the SmolVM HTTP client with the existing provider same-origin redirect pattern and add provider-local regression tests. No new config or product surface appears necessary unless maintainers choose a broader redirect-helper refactor.
