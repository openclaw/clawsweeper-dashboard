## Label Justifications

- `P2`: This is a bounded provider credential leak/hardening bug with clear code shape but a redirect precondition, not a core-runtime emergency.
- `impact:security`: The issue concerns replay of a provider bearer token across a redirect destination outside the documented credential boundary.
- `impact:auth-provider`: The affected credential is the SmolVM provider API key used for authenticated provider operations.

## Summary

Keep open: current main still stores the raw/default SmolVM HTTP client, adds the bearer token before `Do`, and has no `CheckRedirect` guard matching the documented scheme/host/effective-port credential boundary.

## What This Changes

The issue reports that the SmolVM provider can replay its bearer API key across same-host redirects that change scheme or effective port.

## Best Possible Solution

Add a SmolVM-specific secure HTTP-client wrapper that rejects redirects unless scheme, hostname, and effective port match the configured base URL, preserves any caller `CheckRedirect`, and adds focused redirect regression tests.

## Reproduction Assessment

Yes at source level: current main builds SmolVM requests with `Authorization`, uses the raw/default HTTP client without `CheckRedirect`, and Go forwards sensitive headers on same-domain redirects. I did not run a live endpoint PoC.

## Solution Assessment

Yes: mirroring the existing RunPod/Azure redirect wrapper inside the SmolVM client is the narrowest maintainable fix and does not require new config or product surface.
