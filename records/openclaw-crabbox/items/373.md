## Summary

This issue should stay open: current main still lets the configured broker response provide the URL passed to the local browser opener, and I found no merged duplicate or implemented allowlist on the default branch.

## What This Changes

The issue requests CLI-side validation of the GitHub OAuth browser URL returned by the configured broker before `crabbox login` dispatches it to the OS URL handler.

## Best Possible Solution

Add a CLI-side allowlist for the broker-returned OAuth browser target before dispatching it, with regression coverage for rejected schemes/hosts and the repo-local broker reachability path.

## Reproduction Assessment

Yes, source-reproducible: a broker response controls `start.URL`, and default browser mode passes it to `openBrowser` without validating scheme, host, or path. I did not execute the OS opener because that would intentionally dispatch a local URL handler.

## Solution Assessment

Yes. A narrow CLI allowlist for `https://github.com/login/oauth/authorize` before browser dispatch is the best immediate fix; repo-local broker policy can be evaluated separately.
