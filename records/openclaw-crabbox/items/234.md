## Summary

Keep open: the PR addresses a real Tenki status-wait gap, but as submitted it can make `status --wait` resume paused Tenki sessions and can still block past the requested wait timeout; the live proof is also not inspectable.

## What This Changes

The PR adds a `ReadyProbe` resolve flag so SSH-backed `crabbox status --wait` can request provider readiness material while plain status remains status-only.

## Best Possible Solution

Keep the readiness-probe idea, but implement it as a non-mutating, timeout-bounded Tenki readiness-material path with regression tests for paused sessions and short timeouts plus redacted live Tenki output.

## Reproduction Assessment

Yes. Source inspection shows `status --wait` sets `ReadyProbe`, and Tenki then falls through to `prepareLease`, which can resume paused sessions and wait under `bootstrapWaitTimeout`; I did not run a live Tenki session.

## Solution Assessment

No. The PR’s direction is narrower than full generic resolve, but the implementation should use a non-mutating timeout-bounded readiness path and include inspectable live Tenki proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep ReadyProbe status from resuming Tenki sessions:** `internal/providers/tenki/backend.go:174`
  - body: When `ReadyProbe` is set, this condition no longer returns for `StatusOnly`, so `Resolve` falls through to `prepareLease`; that calls `ensureSessionReadyForSSH`, whose paused and pausing branches run `tenki sandbox resume`. Since `crabbox status --wait` now sets `ReadyProbe`, a status command can resume a paused sandbox instead of only observing it. Please split a non-mutating readiness-material path or skip probing non-running states.
  - confidence: 0.9
- **[P2] Honor wait-timeout while preparing readiness data:** `internal/cli/status.go:55`
  - body: `ReadyProbe: *wait` lets Tenki enter `prepareLease`, where session readiness, `ssh-command`, and SSH readiness waits use `bootstrapWaitTimeout(cfg)` rather than the status wait budget. The status loop checks `--wait-timeout` only after `Resolve` returns, so a short wait can still block inside the provider. Please pass the remaining wait budget through this probe path and add a blocking-resolve regression test.
  - confidence: 0.88
