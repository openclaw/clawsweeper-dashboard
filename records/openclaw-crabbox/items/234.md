## Summary

Keep open: this is a useful follow-up to the merged Tenki provider, but the patch is not merge-ready because the full SSH resolve path can bypass the command's `--wait-timeout`, and the external PR has no after-fix real behavior proof.

## What This Changes

The PR changes SSH-backed `crabbox status --wait` to call full lease resolution instead of status-only resolution and adds a CLI regression test for plain status versus wait behavior.

## Best Possible Solution

Keep plain status on the read-only status-only path, but make `status --wait` obtain SSH material under the remaining wait deadline and cover both successful Tenki readiness and timeout behavior.

## Reproduction Assessment

Yes from source inspection, without executing the provider: current main uses `StatusOnly: true` for SSH-backed status, while Tenki status-only resolve returns no SSH target for readiness probing. The PR's timeout regression is also source-reproducible through Tenki full `Resolve` and `prepareLease`.

## Solution Assessment

No as submitted. Full SSH resolution is a plausible direction for `--wait`, but it needs to be bounded by the status wait deadline and proven against Tenki's real SSH readiness path.

- **[P1] Honor wait-timeout during full resolve:** `internal/cli/status.go:55`
  - body: When `--wait` is true, this now calls provider full `Resolve` before the loop checks the status deadline. Tenki's full resolve goes through `prepareLease`, which waits for SSH command material and SSH readiness using the bootstrap timeout, so `crabbox status --wait --wait-timeout 1s` can stall far beyond the documented timeout before returning. Please bound this resolve with the remaining wait deadline, or use a status-wait resolve path that can return SSH material without unbounded bootstrap waits.
