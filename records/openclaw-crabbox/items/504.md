## Label Justifications

- `P2`: This is a concrete credential-confidentiality security issue with limited provider scope and a clear focused fix.
- `impact:security`: The affected behavior can send trusted provider credentials to an unsafe configured destination.
- `impact:auth-provider`: The affected credential is the E2B provider API key used for authenticated provider routing.

## Summary

Keep open: current main and the latest release still accept E2B API URL overrides without the HTTPS-or-loopback validation used by neighboring credentialed providers, and the closed related hardening work addressed mixed-source credential destinations rather than this plaintext endpoint gap.

## What This Changes

The issue asks Crabbox to reject unsafe E2B API URL overrides so provider credentials are not sent to plaintext non-loopback endpoints.

## Best Possible Solution

Add a focused E2B API URL validator that rejects non-loopback HTTP, userinfo, query strings, and fragments before any credentialed request is built, with tests and provider docs covering accepted HTTPS and loopback development URLs.

## Reproduction Assessment

Yes. Source inspection shows newE2BClient accepts the configured API base without validation and doJSONWithHeaders attaches X-API-Key to requests made against that base.

## Solution Assessment

Yes. The narrow maintainable fix is to mirror the existing credentialed-provider URL validators in the E2B client and add regression coverage plus docs for the HTTPS-or-loopback rule.
