## Label Justifications

- `P1`: The issue reports a security-boundary bug where Crabbox can route destructive provider operations to resources identified only by name shape.
- `impact:security`: The report concerns provider resource ownership checks before destructive actions.
- `impact:data-loss`: The affected stop/delete paths can remove provider VMs or sandboxes whose only Crabbox evidence is a canonical-looking name.

## Summary

The report is still necessary: current main still lets Freestyle and Islo treat claimless canonical-looking provider resource names as enough evidence before provider mutation, and the merged related hardening only covers adjacent GCP or malformed Freestyle-name cases.

## What This Changes

The issue requests hardening Freestyle and Islo lifecycle resolution so destructive operations require stronger Crabbox ownership evidence than a canonical-looking provider resource name.

## Best Possible Solution

Add provider-adapter ownership checks or an explicit reclaim flow so Freestyle and Islo can recover lost leases without allowing stop/delete/pause/resume/SSH to act on claimless canonical names by default.

## Reproduction Assessment

Yes for source reproduction: current main still has resolver paths and tests showing canonical-looking Freestyle and Islo resources can be accepted without a retained claim before provider mutation. I did not run live provider operations because the report is security-sensitive and source evidence is sufficient for triage.

## Solution Assessment

Unclear until maintainers choose the recovery model: requiring exact retained claims is the narrowest safety fix, but preserving documented lost-claim recovery may need an explicit reclaim flow or provider-side immutable ownership marker.
