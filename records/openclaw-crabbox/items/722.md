## Summary

Keep open: current main and v0.33.0 still pass Daytona upload and API error bodies into diagnostics without a Daytona redaction pass, while the security docs require secret-looking diagnostics redaction and sibling providers already redact similar error bodies.

## What This Changes

The issue requests Daytona-specific redaction for provider credentials in failed upload and API error diagnostics.

## Best Possible Solution

Add a Daytona-local diagnostic redaction helper, apply it to upload and GenericOpenAPIError bodies, and add focused regression tests for configured credentials, bearer echoes, and token/access_token JSON fields.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a fake Daytona upload or API response returning a 4xx/5xx body containing the configured token would flow into the returned error string without redaction.

## Solution Assessment

Yes. The requested provider-local redaction is the narrow maintainable fix and matches the existing Cloudflare and Superserve provider patterns, without adding core provider-specific branching.
