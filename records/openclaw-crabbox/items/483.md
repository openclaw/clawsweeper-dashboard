## Label Justifications

- `P2`: This is a normal-priority CLI reliability bug that can remove durable run diagnostics for one coordinator-backed run after a transient create failure.
- `impact:data-loss`: A failed initial run-record create can silently lose the run handle, run events, retained logs, and run-based recovery path for the entire invocation.

## Summary

Keep open: current main and v0.32.0 still drop run-history creation after a non-`invalid_lease_id` `CreateRun` failure, leaving later `AttachLease` calls without the retry flag needed to recover a run ID.

## What This Changes

The issue requests retrying coordinator-backed run history creation after transient `CreateRun` failures and surfacing a clear history-unavailable fallback when no run ID exists.

## Best Possible Solution

Keep the issue open and fix the recorder to remember initial create failure, retry `CreateRun` once when a resolved lease ID is available, and print an explicit history-unavailable notice if no run ID can be recovered.

## Reproduction Assessment

Yes, by source inspection. `newRunRecorder` leaves `runID` empty and does not set `deferUntilLease` for non-`invalid_lease_id` errors, and `AttachLease` then returns without retrying because that flag is false.

## Solution Assessment

Yes, the suggested direction is the narrow maintainable fix: broaden the existing deferred-create retry path and add focused failure-digest feedback if the retry still cannot produce a run ID.
