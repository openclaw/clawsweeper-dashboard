## Summary

Keep open: current main and the latest release still allow decoded control characters through the OAuth portal returnTo filter, and the stored value later reaches a Location header in the successful GitHub callback path. No duplicate issue or fixing PR was found, so this remains the canonical actionable report.

## What This Changes

The issue reports that a crafted OAuth portal login returnTo value containing decoded control characters can be persisted and later crash the callback redirect instead of completing login.

## Best Possible Solution

Harden safePortalReturnTo so decoded header-invalid control characters fall back to /portal before OAuth state is stored, and add focused Vitest coverage for rejected malicious values plus valid portal paths.

## Reproduction Assessment

Yes. Source inspection shows decoded returnTo reaches a Location header, and a local runtime check confirms the decoded CR/LF value is rejected by Headers; I did not run the full Worker OAuth callback test because this was a read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to validate or fall back in safePortalReturnTo before persisting pending OAuth state, with regression coverage in the existing portal login test area.
