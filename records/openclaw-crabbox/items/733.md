Keep open: current main and latest v0.33.0 still generate Windows Sandbox cleanup and liveness code with only the legacy process names, so the reported early-exit/orphaned-sandbox failure remains unresolved.

## What This Changes

The issue asks Crabbox to fix Windows Sandbox provider liveness and cleanup on Windows 11 24H2+ where the running sandbox process names changed.

## Best Possible Solution

Land a provider-local compatibility fix that recognizes both legacy and Windows 11 24H2+ sandbox processes for liveness and teardown, with regression tests for the generated host runner and cleanup path.

## Reproduction Assessment

Yes for a Windows 11 24H2+ host, based on the reporter's command/logs and source inspection; I did not execute a live Windows Sandbox run locally. Current main's generated host runner and cancellation cleanup only match the legacy process names, which is enough to source-reproduce the reported false early-exit path when the OS uses renamed sandbox processes.

## Solution Assessment

Yes, the narrow provider-local repair is the best first path: update cleanup/liveness detection and tests without changing core provider routing. Optional `wsb list`/`wsb stop` use is acceptable only if kept backward-compatible with the existing process-name fallback.
