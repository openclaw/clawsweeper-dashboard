## Label Justifications

- `P1`: The issue reports a plausible provider-resource ownership bug on a destructive Freestyle stop path, but it requires local privileged operator action rather than unauthenticated remote access.
- `impact:security`: The report concerns the trust boundary Crabbox uses before treating a provider VM as Crabbox-owned.
- `impact:data-loss`: If the weak predicate accepts a malformed provider name, the stop path can delete a non-Crabbox Freestyle VM.

## Summary

The issue is still necessary: current main and the latest release still normalize Freestyle provider VM names before the Crabbox ownership prefix check, and that resolver feeds the destructive stop path. No canonical open fix PR or merged fix was found for the malformed-name case.

## What This Changes

The issue asks Crabbox to reject malformed Freestyle provider VM names that only become `crabbox-*` after slug normalization during raw `fsb_` recovery.

## Best Possible Solution

Harden the Freestyle adapter so raw-ID recovery accepts only provider names already in Crabbox's generated slug form, then add focused regression tests for malformed names and the stop path.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence reproduction path with a fake Freestyle VM named like `crabbox repo abc123`; I did not run tests because this review must keep the checkout read-only.

## Solution Assessment

Yes. Requiring the provider-returned name to already be canonical slug form before accepting raw-ID recovery is the narrowest maintainable fix and stays within the Freestyle adapter boundary.
