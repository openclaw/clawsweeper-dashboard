- `impact:security`: The affected behavior crosses a generated-artifact boundary by preserving untrusted remote tar link metadata in a local diagnostic archive.

## Summary

Keep open: current main still copies remote tar link metadata into local failure bundles after only sanitizing member names, so the reported security issue remains actionable.

## What This Changes

The issue reports that SSH-backed failure bundle merging rewrites remote tar member names but preserves symlink and hardlink targets that can escape `crabbox-artifacts/remote/`.

## Best Possible Solution

Validate remote tar symlink and hardlink targets during `appendRemoteFailureTar`, skip absolute, empty, or escaping targets, preserve safe regular files, and add a crafted-tar regression test.

## Reproduction Assessment

Yes. Source inspection shows `appendRemoteFailureTar` copies `header.Linkname` unchanged after only sanitizing `header.Name`, so a crafted remote tar with an escaping symlink or hardlink target would be written into the local bundle.

## Solution Assessment

Yes. The narrow maintainable fix is to validate or normalize link metadata at the local bundle merge point and cover it with a crafted remote-tar regression test.
