- `impact:security`: The issue concerns a coordinator authorization boundary and exposure of access-control metadata to a lower share role.

## Summary

Keep open: current main still returns the normalized share roster from the dedicated API share route to any visible lease recipient before enforcing manage access, and no merged PR or canonical follow-up owns this exact leak.

## What This Changes

The issue asks Crabbox to prevent `use`-only shared lease recipients from reading manager-facing lease share roster metadata through the coordinator API.

## Best Possible Solution

Require manage-level access before any `/v1/leases/{id-or-slug}/share` response returns the roster, add a `use`-recipient regression test, and separately decide whether visible lease JSON should redact `share`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `use`-shared caller can resolve the visible lease, call `GET /v1/leases/{id-or-slug}/share`, and receive the normalized share roster before any manage-access check runs.

## Solution Assessment

Mostly yes. Moving the manage guard before the `GET` return is the narrow route fix, but maintainers should also decide whether the existing visible lease JSON exposure of `share` is intentional.
