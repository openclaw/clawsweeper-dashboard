## Summary

Keep this PR open for maintainer review: the AWS rule-limit recovery is a useful bug fix with focused tests, but the branch changes compatibility-sensitive security-group cleanup and can revoke legacy unlabeled SSH rules on groups classified as Crabbox-managed, including configured groups whose docs say ingress policy is user-owned.

## What This Changes

The branch retries Worker AWS SSH ingress authorization after compacting stale Crabbox-managed SSH rules on AWS rule-limit errors, adds AWS Worker regression tests, and edits CHANGELOG.md.

## Best Possible Solution

Ship a constrained AWS retry that preserves configured security-group ownership, reclaims legacy unlabeled rules only when Crabbox ownership is unambiguous or explicitly approved, keeps the regression tests, and leaves release notes to the release process.

## Reproduction Assessment

Yes. Source inspection shows current main throws non-duplicate AuthorizeSecurityGroupIngress errors, and the PR adds a mocked EC2 path where RulesPerSecurityGroupLimitExceeded currently needs recovery.

## Solution Assessment

No, not as-is. The retry-after-compaction approach is narrow, but deleting legacy unlabeled rules via a group name/tag heuristic needs a safer configured-security-group boundary or explicit maintainer approval.
