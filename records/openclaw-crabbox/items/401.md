- `impact:security`: The report concerns authentication identity trust and caller-controlled owner/org attribution across coordinator state.

## Summary

This issue remains necessary: current `main` and `v0.31.0` still let the admin-token branch derive owner/org from caller-supplied headers, and no related open or merged item found in the canonical search owns this exact admin-token fallback.

## What This Changes

The issue reports that admin-token authentication can derive the authenticated owner/org from caller-supplied `X-Crabbox-Owner` and `X-Crabbox-Org` headers.

## Best Possible Solution

Admin-token requests should derive actor identity only from verified Access, trusted proxy identity, or deployment-controlled admin defaults, then keep downstream header stamping as an internal-only operation.

## Reproduction Assessment

Yes from source inspection: `authenticateRequest` reads raw owner/org headers for admin-token requests, and the existing Worker auth test asserts that behavior. I did not run Vitest because this was a read-only review.

## Solution Assessment

Yes in direction: stop trusting raw identity headers for admin-token requests and pin admin identity to verified or deployment-controlled values. The exact admin owner/org configuration names, defaults, and upgrade behavior need maintainer security review before implementation.
