## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Honor generic SSH user and work root:** `internal/providers/tart/backend.go:49-68`
  - body: `baseConfig` gives Tart non-empty `User` and `WorkRoot` defaults, and this path then writes those values back over `cfg.SSHUser` and `cfg.WorkRoot`. A user who only sets the existing generic `ssh.user` or `workRoot` will connect and sync with the wrong Tart account/root; carry non-default generic values into Tart before falling back to provider defaults or track Tart-specific explicitness.
  - confidence: 0.87
- **[P1] Honor cancellation before detached Tart starts:** `internal/providers/tart/backend.go:409-410`
  - body: When `keep` is true this uses `exec.Command` without first checking `ctx`, so an already-canceled acquire can still launch a detached local VM and only try to clean it up later. Check the context before `Start`, or use a context-aware startup path that detaches only after the start is still valid.
  - confidence: 0.86
- **[P1] Keep started VMs available after readiness failures:** `internal/providers/tart/backend.go:131-140`
  - body: After `startVM` succeeds, failures in IP polling or SSH key injection always stop and delete the VM even for `req.Keep`. Adjacent local VM providers preserve kept resources after post-create failures, and this path removes the exact VM an operator would need to inspect; guard post-start cleanup for kept acquisitions or make the different policy explicit and tested.
  - confidence: 0.82
- **[P2] Align the disk flag help with validation:** `internal/providers/tart/flags.go:24`
  - body: The help text promises `0 = use clone default`, but `applyFlags` rejects `--tart-disk 0`. Either accept explicit zero as the documented clone-default value or remove that promise so users do not follow the CLI help into an error.
  - confidence: 0.95
