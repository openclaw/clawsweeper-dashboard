## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-140`
  - body: After `startVM` succeeds, the IP, SSH-key, and lease-claim failure paths still stop and delete the Tart VM unconditionally. That breaks `--keep` exactly when an operator needs to inspect a partially provisioned local VM; guard these post-start cleanup branches with `if !req.Keep`, matching the adjacent local-provider keep semantics.
  - confidence: 0.95
- **[P1] Honor cancellation before starting kept VMs:** `internal/providers/tart/backend.go:410-414`
  - body: The keep=true branch uses `exec.Command` and never checks `ctx` before launching `tart run`, so a canceled acquire can still start a detached VM and then return nil after the observation timer. Check `ctx.Err()` before launch and make the keep path return cancellation instead of creating background VMs after the caller has stopped waiting.
  - confidence: 0.94
- **[P1] Inherit generic Tart SSH user and work root safely:** `internal/cli/config.go:777-785`
  - body: Because `baseConfig` pre-populates `Tart.User` and `Tart.WorkRoot`, this branch copies `admin` and `/Users/admin/crabbox` back over generic `ssh.user` and top-level `workRoot` settings. Track explicit `tart.user` and `tart.workRoot` values and let generic settings flow into Tart unless provider-specific fields were configured.
  - confidence: 0.9
- **[P2] Use Tart state for resolve instead of Running:** `internal/providers/tart/backend.go:185-186`
  - body: `Running` is an optional JSON boolean, so when Tart omits it a `State: "running"` instance decodes as `Running=false` and non-status resolve rejects an otherwise usable claimed VM. Use the normalized state helper here so inventory shape differences do not break lease reuse.
  - confidence: 0.93
- **[P2] Prune stale Tart claims when the VM is gone:** `internal/providers/tart/backend.go:252-254`
  - body: When stop resolves by lease or slug and the claim points at a VM no longer present in `tart list`, `resolveInstance` returns an error before `ReleaseLease` can remove the local claim and stored key. Return a stale target or prune the claim directly so `crabbox stop --provider tart <lease>` can clean up Crabbox-owned state.
  - confidence: 0.86
- **[P3] Align explicit zero disk validation:** `internal/providers/tart/flags.go:49-52`
  - body: The flag help advertises `--tart-disk 0` as clone-default, but the flag validation rejects zero before a run can start. Either allow zero as the documented sentinel or change the help/docs so users are not given a command that fails immediately.
  - confidence: 0.91
