## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-134`
  - body: After `startVM` succeeds, these failure branches still stop and delete the VM even when `req.Keep` is true. That defeats `--keep` exactly when an operator needs to inspect a VM that failed during IP discovery, key injection, SSH readiness, or claim writing.
  - confidence: 0.94
- **[P2] Honor cancellation before detached Tart launch:** `internal/providers/tart/backend.go:409-410`
  - body: The keep path uses plain `exec.Command`, so an already-canceled context can still launch `tart run`, wait two seconds, and return success. Check `ctx.Err()` before starting and add a `ctx.Done()` case during the startup observation window so canceled acquisitions do not create detached VMs.
  - confidence: 0.92
- **[P2] Use normalized state when resolving running VMs:** `internal/providers/tart/backend.go:185-186`
  - body: `Running` is an optional decoded bool, so records with `State: "running"` but no `Running` field are treated as stopped here. Use the existing state normalizer for this boundary or reuse of claimed running Tart VMs can fail for valid `tart list` output.
  - confidence: 0.9
- **[P2] Preserve shared SSH and workRoot overrides:** `internal/cli/config.go:777-785`
  - body: Because `baseConfig` pre-populates Tart defaults, this branch overwrites shared `ssh.user` and top-level `workRoot` with `admin` and `/Users/admin/crabbox` even when the user set generic Crabbox values. Track explicit Tart-specific user/workRoot settings so shared config remains the fallback.
  - confidence: 0.88
- **[P2] Let stop prune stale Tart claims:** `internal/providers/tart/backend.go:173-175`
  - body: `Resolve` always requires the claimed Tart VM to still exist before the release-only path can run. If the VM was already removed outside Crabbox, `crabbox stop --provider tart <lease>` cannot remove the stale local claim/key, so release-only resolution needs a missing-instance cleanup path.
  - confidence: 0.82
- **[P3] Align tart-disk zero semantics:** `internal/providers/tart/flags.go:24`
  - body: The flag help advertises `0 = use clone default`, but the validation rejects `--tart-disk 0`. Either allow zero as the documented clone-default sentinel or remove that promise from the CLI help and docs so users do not get a contradictory error.
  - confidence: 0.95
