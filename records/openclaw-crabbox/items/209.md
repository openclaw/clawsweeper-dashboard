## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve generic Tart SSH settings:** `internal/cli/config.go:777-785`
  - body: When a user selects `provider: tart` and sets the normal top-level `ssh.user` or `workRoot`, this branch copies the pre-filled Tart defaults back over those generic settings. That silently routes the lease to `admin` and `/Users/admin/crabbox` unless the user knows to duplicate the values under `tart:`, so Tart should only override the generic settings when the Tart-specific fields were explicitly configured.
  - confidence: 0.92
- **[P2] Allow the documented clone-default disk value:** `internal/providers/tart/flags.go:49-70`
  - body: The flag help and provider docs describe `0` as the clone-default disk sentinel, but `--tart-disk 0` is rejected here and again in provider validation. This prevents users from clearing a configured resize back to clone default and makes the shipped docs/config contract false.
  - confidence: 0.9
- **[P1] Preserve kept Tart VMs after post-start failures:** `internal/providers/tart/backend.go:131-166`
  - body: After `startVM` succeeds, every later acquisition failure still stops and deletes the VM even when `req.Keep` is true. That breaks the operator's ability to inspect a failed kept VM after IP readiness, SSH injection, SSH readiness, or claim persistence fails; mirror the adjacent local-provider pattern and guard post-start cleanup with `!req.Keep`.
  - confidence: 0.9
- **[P1] Honor cancellation before keep-mode launch:** `internal/providers/tart/backend.go:409-434`
  - body: The keep path uses `exec.Command` and the startup observe select never checks `ctx.Done()`, so a canceled acquire can still launch and detach a Tart VM, then return success after the observe timeout. Check `ctx.Err()` before starting and handle cancellation during the observe window so canceled runs do not leave surprise local VMs behind.
  - confidence: 0.88
- **[P2] Treat running state as active without Running:** `internal/providers/tart/backend.go:185-186`
  - body: `Running` is an optional JSON bool and defaults to false when omitted, while Tart's canonical state string can still be `running`. Using `!inst.Running` here rejects claimed running VMs from run/ssh/cache reuse; use the normalized state helper already used elsewhere for this boundary.
  - confidence: 0.91
- **[P2] Let stop prune stale claimed Tart leases:** `internal/providers/tart/backend.go:516-539`
  - body: For `ReleaseOnly`, a claim whose referenced VM is already gone should still produce a release target so `ReleaseLease` can remove the local claim and stored key. The current `resolveInstance` path errors before release can run, leaving stale Tart lease metadata that `crabbox stop` cannot clean up.
  - confidence: 0.86
