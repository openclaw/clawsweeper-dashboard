## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve generic Tart SSH and workRoot overrides:** `internal/cli/config.go:777-785`
  - body: For `provider=tart`, this branch copies `cfg.Tart.User` and `cfg.Tart.WorkRoot` into the generic SSH/workroot fields even when those Tart values are only base defaults. A user who already sets top-level `ssh.user` or `workRoot` is silently reset to `admin` and `/Users/admin/crabbox`, so key injection and sync target the wrong account/root. Only override generic settings when Tart-specific values were explicitly configured, and add regression coverage for both YAML/env generic settings.
  - confidence: 0.9
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-140`
  - body: Once `startVM(..., keep=true)` has launched the VM, later readiness, key injection, prepareLease, claim, or endpoint errors still stop and delete the VM. That breaks the expected `--keep` behavior and removes the VM users need to inspect or reuse after a failed acquisition. Gate the cleanup after a successful kept start, or leave manual cleanup instructions instead.
  - confidence: 0.92
- **[P2] Honor cancellation before detached Tart launch:** `internal/providers/tart/backend.go:409-414`
  - body: The keep path uses `exec.Command` and never checks `ctx` before `Start`, so an already-canceled or timed-out acquisition can still launch a detached Tart VM. Check the context before starting the process so cancellation cannot create unexpected local resources or stale lease state.
  - confidence: 0.87
- **[P2] Use normalized state when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: `Running` defaults to false if Tart omits that JSON field, but `State` can still be `running`; this check then rejects healthy claimed VMs for every non-status command. Use the normalized state helper, or normalize `Running` from `State`, before deciding the VM is stopped.
  - confidence: 0.88
- **[P2] Let release prune stale Tart claims:** `internal/providers/tart/backend.go:252-254`
  - body: When a lease claim points at a VM that no longer exists, `resolveInstance` returns an error and `ReleaseLease` exits before removing the local claim/key. `crabbox stop --provider tart <lease>` should be able to clear that stale state even when there is no VM left to stop/delete.
  - confidence: 0.84
- **[P3] Align tart-disk help with validation:** `internal/providers/tart/flags.go:24`
  - body: The flag help documents `0 = use clone default`, but the validation rejects `--tart-disk 0`. Either accept 0 as the explicit clone-default value or remove that promise from the flag/docs so users do not get an error after following the help text.
  - confidence: 0.96
