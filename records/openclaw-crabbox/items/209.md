## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve generic Tart user and work-root overrides:** `internal/cli/config.go:777-785`
  - body: `baseConfig()` pre-populates `cfg.Tart.User` and `cfg.Tart.WorkRoot`, so this branch always copies the Tart defaults back over top-level `ssh.user` and `workRoot`. A config with `provider: tart`, `ssh.user: root`, or a custom `workRoot` will still run as `admin` in `/Users/admin/crabbox` unless `tart.user`/`tart.workRoot` are set, which breaks the existing generic config contract; track explicit Tart values and only let provider-specific settings win when the user actually set them.
  - confidence: 0.97
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:131-135`
  - body: After `startVM` succeeds, every later acquisition failure path still calls `stopVM` and `deleteVM` unconditionally. That makes `crabbox run --provider tart --keep` delete the VM when IP polling, SSH key injection, SSH readiness, claim writing, or endpoint update fails, which removes the exact VM a user asked to keep for inspection; guard post-start cleanup with `if !req.Keep` and cover both keep/non-keep behavior.
  - confidence: 0.98
- **[P1] Honor cancellation before detached Tart starts:** `internal/providers/tart/backend.go:409-414`
  - body: The `keep` path uses plain `exec.Command` and the startup select only waits for process exit or a timer. If the acquire context is already canceled, this can still launch a detached Tart VM and return nil after the observe window; check `ctx.Err()` before starting and handle `ctx.Done()` while observing startup.
  - confidence: 0.96
- **[P1] Use Tart state when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: `tart list --format json` records can report `State:"running"` without the optional `Running` bool. This check treats the zero-value bool as stopped and rejects claimed running leases for run/ssh/cache/code paths; use the existing normalized `instanceRunning(inst.State)` helper for the non-status resolve boundary.
  - confidence: 0.98
- **[P2] Let stop prune claim-backed missing VMs:** `internal/providers/tart/backend.go:538`
  - body: When a local claim exists but the referenced Tart VM is already gone, `resolveInstance` errors before `ReleaseOnly` can return a target to `ReleaseLease`. That leaves `crabbox stop --provider tart <lease-or-slug>` unable to remove the stale local claim/key; add a release-only path that returns a missing-instance target and prunes the claim without calling `tart stop/delete`.
  - confidence: 0.95
- **[P3] Align tart disk zero validation with help text:** `internal/providers/tart/flags.go:49-51`
  - body: The flag help advertises `--tart-disk 0` as clone-default behavior, but the validation rejects `0` before the provider can skip `--disk-size`. This also prevents a CLI override from resetting an env/YAML disk size back to clone default; either accept zero as the documented sentinel or change the help/docs/tests to say only unset means clone default.
  - confidence: 0.94
