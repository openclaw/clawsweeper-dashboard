## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Use Tart state when `Running` is absent:** `internal/providers/tart/backend.go:185-186`
  - body: This rejects any non-status resolve when `inst.Running` is false, but that bool defaults to false when Tart omits the optional field even if `State` is `running`. Claimed running VMs can then be rejected as stopped, breaking `run --id` and other SSH-target reuse paths; use the existing normalized state helper for this boundary.
  - confidence: 0.97
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:131-140`
  - body: After `startVM` succeeds, the IP, key injection, lease preparation, claim, and endpoint failure paths still stop and delete the VM unconditionally. That defeats `--keep` exactly when an operator needs to inspect a partially booted Tart VM, so these post-start cleanup blocks should be guarded by `!req.Keep`.
  - confidence: 0.96
- **[P1] Honor cancellation before launching kept Tart runs:** `internal/providers/tart/backend.go:409-435`
  - body: The keep path uses `exec.Command` and then waits only for process exit or the observation timer, so an already-canceled context can still launch a detached Tart VM and return nil. Check `ctx.Err()` before starting and handle `ctx.Done()` during the observation window, killing the just-started process if cancellation arrives.
  - confidence: 0.94
- **[P3] Align `--tart-disk` help with validation:** `internal/providers/tart/flags.go:24`
  - body: The flag help says `0 = use clone default`, but `applyFlags` rejects zero before the provider can treat it as the clone-default sentinel. Either allow explicit zero and skip `--disk-size`, or change the help/docs so users do not get a documented reset path that always fails.
  - confidence: 0.93
