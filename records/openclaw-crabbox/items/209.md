## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve generic Tart account and work-root overrides:** `internal/cli/config.go:1093-1096`
  - body: Because baseConfig pre-populates `Tart.User`/`Tart.WorkRoot` with provider defaults, the Tart branch later treats those defaults as configured values and overwrites top-level `ssh.user` and `workRoot`. A user selecting `provider: tart` with generic Crabbox config gets `admin` and `/Users/admin/crabbox` instead of their configured account/path unless they also duplicate the values under `tart:`.
  - confidence: 0.92
- **[P1] Use Tart state when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: This check rejects any non-status resolve when `Running` is false, but that field defaults to false if Tart omits it from JSON even when `State` is `running`. Reusing a claimed running VM can therefore fail as "stopped"; use the normalized state helper for the boundary instead.
  - confidence: 0.94
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:132-135`
  - body: After `startVM` succeeds, failures in IP polling, key injection, SSH readiness, claim creation, or endpoint update still stop and delete the VM unconditionally. That defeats `--keep` exactly when an operator needs to inspect a partially started local VM after a provisioning failure.
  - confidence: 0.91
- **[P1] Honor cancellation before detached Tart starts:** `internal/providers/tart/backend.go:409-414`
  - body: The keep path uses `exec.Command` and does not check `ctx.Err()` before starting `tart run` or select on `ctx.Done()` during the observation window. A canceled acquire can still launch a kept VM and return success after the timer, leaving a background VM the caller abandoned.
  - confidence: 0.9
- **[P2] Let release-only stop prune missing Tart claims:** `internal/providers/tart/backend.go:532-538`
  - body: When a local claim points at a Tart VM that no longer appears in `tart list`, `resolveInstance` returns an error before `ReleaseOnly` can return a target for `ReleaseLease` to remove the stale claim/key. Stop-by-lease should be able to prune Crabbox-owned local state when the backing VM is already gone.
  - confidence: 0.88
- **[P3] Align the tart disk zero contract:** `internal/providers/tart/flags.go:24`
  - body: The flag help says `0 = use clone default`, but the validation rejects `--tart-disk 0` as non-positive. Either accept zero as the documented reset-to-clone-default sentinel or change the help/docs so users are not given an unusable value.
  - confidence: 0.9
