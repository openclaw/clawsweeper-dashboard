## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the generic SSH user for Tart:** `internal/providers/tart/backend.go:49-65`
  - body: When `tart.user` is unset, this defaulting path sets `Tart.User` to `admin` and then overwrites `cfg.SSHUser`, so a top-level `ssh.user` from config is ignored for Tart even though generic workRoot is preserved. Please carry a non-default generic SSH user into `cfg.Tart.User` before falling back to `admin`, and add a regression test for that config path.
  - confidence: 0.91
- **[P1] Honor cancellation before starting kept Tart VMs:** `internal/providers/tart/backend.go:410`
  - body: The kept path uses `exec.Command`, so if the acquire context is already canceled it can still launch a detached `tart run` that Crabbox no longer owns. Check the context before starting, or otherwise use a context-aware start path that still detaches only after a valid start.
  - confidence: 0.88
- **[P1] Do not delete kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-140`
  - body: After `startVM` succeeds, failures in IP polling or SSH key injection always stop and delete the VM even when the user requested `--keep`. That defeats the kept-lease debugging path and diverges from the adjacent local VM behavior; guard these cleanup calls with `!req.Keep` once the VM has started.
  - confidence: 0.9
- **[P2] Align the tart disk flag help with validation:** `internal/providers/tart/flags.go:24`
  - body: The flag help says `0 = use clone default`, but `applyFlags` rejects `--tart-disk 0`. Either accept explicit zero as the documented clone-default value or remove that promise from the flag text so users do not get an error for following the CLI help.
  - confidence: 0.96
