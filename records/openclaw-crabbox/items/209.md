## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use Tart State when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: This rejects every non-status Resolve call when the optional `Running` JSON bool is false. If Tart omits that field while reporting `State:"running"`, a claimed VM with a current IP is treated as stopped, so reuse paths like run/ssh by lease can fail. Use the normalized state helper for this boundary and cover the omitted-Running case.
  - confidence: 0.94
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-135`
  - body: After `startVM` succeeds, this failure path still stops and deletes the VM even when the user requested `--keep`; the same cleanup pattern repeats for SSH injection, readiness, claim creation, and endpoint update failures. That removes the local VM users need to inspect after a failed kept acquisition.
  - confidence: 0.95
- **[P1] Honor cancellation before launching kept Tart runs:** `internal/providers/tart/backend.go:410`
  - body: The keep-mode path starts `tart run` with plain `exec.Command` and then waits only for process exit or the observation timeout. If the acquire context is already canceled, this can still launch a background VM and return success after the timeout; check `ctx.Err()` before start and handle `ctx.Done()` during observation.
  - confidence: 0.94
- **[P2] Respect generic SSH user and workRoot for Tart:** `internal/cli/config.go:777-785`
  - body: Because Tart defaults are preloaded in config, this branch copies `tart.user` and `tart.workRoot` back over top-level `ssh.user` and `workRoot` even when the user only set the generic values. Track explicit Tart-specific user/root settings so provider-specific config wins, but generic settings still flow into Tart when no Tart override exists.
  - confidence: 0.9
- **[P3] Align the tart disk zero contract:** `internal/providers/tart/flags.go:49-51`
  - body: The flag help says `0 = use clone default`, but explicit zero is rejected here and again in provider validation. That prevents users from clearing a configured disk resize back to clone-default behavior; accept zero as the sentinel or change the docs/help to match strict positive validation.
  - confidence: 0.9
