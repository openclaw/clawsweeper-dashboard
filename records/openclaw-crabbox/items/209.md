## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve generic Tart user and work-root overrides:** `internal/cli/config.go:777-785`
  - body: Because baseConfig already fills Tart.User and Tart.WorkRoot, this branch treats those defaults as explicit and copies them over the generic ssh.user/workRoot values. A user with provider: tart plus top-level ssh.user or workRoot will silently run as admin in /Users/admin/crabbox instead of their configured account/path; track tart.user and tart.workRoot explicitness, then let generic settings flow through when Tart-specific values are not set.
  - confidence: 0.93
- **[P1] Use Tart state when Running is absent:** `internal/providers/tart/backend.go:185-186`
  - body: This check depends on the optional JSON Running bool, so tart list output with State: running but no Running field defaults to false and every non-status Resolve path rejects a usable VM as stopped. Use instanceRunning(inst.State) or otherwise derive the bool from State when Running is omitted.
  - confidence: 0.91
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-135`
  - body: After tart run has started, failures while waiting for IP, injecting the SSH key, preparing the lease, or writing the claim always stop and delete the VM, even for --keep. That removes the exact VM the user asked to keep for inspection/reuse; for keep=true, leave the started VM in place and clean only the local claim/key state that was actually created.
  - confidence: 0.9
- **[P1] Honor cancellation before detached Tart starts:** `internal/providers/tart/backend.go:409-410`
  - body: The keep=true path launches tart with exec.Command and does not check ctx before Start, so a canceled acquire can still create a detached VM that Crabbox no longer owns. Check ctx before launching the detached process, and keep the non-keep path tied to CommandContext.
  - confidence: 0.87
- **[P3] Align the tart disk zero validation:** `internal/providers/tart/flags.go:24`
  - body: The flag help says --tart-disk 0 means use the clone default, but ApplyFlags rejects zero as an error. Either accept explicit zero without marking a resize, or change the help/docs so users do not get contradictory CLI guidance.
  - confidence: 0.86
