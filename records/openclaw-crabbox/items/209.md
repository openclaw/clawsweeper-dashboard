## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve generic Tart SSH settings:** `internal/cli/config.go:777-785`
  - body: `baseConfig` pre-populates `cfg.Tart.User` and `cfg.Tart.WorkRoot`, then this branch copies those provider defaults over `cfg.SSHUser` and `cfg.WorkRoot`. A user who sets top-level `ssh.user` or `workRoot` therefore gets silently switched to `admin` and `/Users/admin/crabbox` unless they duplicate the values under `tart`, so Tart should inherit generic SSH lease settings when provider-specific fields are absent.
  - confidence: 0.91
- **[P1] Preserve kept Tart VMs after post-start failures:** `internal/providers/tart/backend.go:131-140`
  - body: After `startVM` succeeds, later provisioning failures still unconditionally stop and delete the VM, including `--keep` acquisitions. That defeats the kept-lease debugging path for DHCP/IP readiness, key injection, SSH readiness, or claim-recording failures; post-start cleanup should preserve a user-requested kept VM.
  - confidence: 0.88
- **[P1] Honor cancellation before keep-mode Tart launch:** `internal/providers/tart/backend.go:409-420`
  - body: The keep path uses `exec.Command` instead of `exec.CommandContext` and does not check `ctx.Err()` before starting the detached `tart run`, so a canceled warmup can still leave a persistent VM running. Check cancellation before launching keep-mode Tart and cover it with a focused regression test.
  - confidence: 0.84
- **[P2] Treat State=running as active without Running:** `internal/providers/tart/backend.go:185-186`
  - body: `Resolve` rejects non-status callers with `!inst.Running`, but Tart JSON can report `State: "running"` while omitting the optional `Running` field, which defaults the bool to false. Use the normalized state helper here so claimed running VMs remain reusable when Tart omits the boolean.
  - confidence: 0.86
- **[P2] Allow documented --tart-disk 0 clone default:** `internal/providers/tart/flags.go:49-54`
  - body: The flag help says `--tart-disk` accepts `0 = use clone default`, but `applyFlags` rejects `<= 0`, so users cannot explicitly reset a configured disk size to the documented clone-default behavior. Either accept zero as the sentinel and skip `tart set --disk-size`, or remove that documented contract everywhere.
  - confidence: 0.89
- **[P2] Let release-only stop prune stale Tart claims:** `internal/providers/tart/backend.go:532-538`
  - body: When a Crabbox Tart claim points at a VM that no longer exists, `resolveInstance` returns an error before `ReleaseOnly` can produce a target for `ReleaseLease`. That leaves `crabbox stop --provider tart <lease>` unable to prune stale claim/key state after the local VM has already been removed.
  - confidence: 0.84
