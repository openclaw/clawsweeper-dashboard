## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve kept VMs after late acquire failures:** `internal/providers/tart/backend.go:162-163`
  - body: This cleanup path runs after the VM has started and key injection has completed, but it still stops/deletes the VM even when the user passed `--keep`. If SSH readiness, claim creation, or endpoint persistence fails, a kept Tart VM is destroyed despite the keep contract; guard these cleanup calls the same way the earlier waitForIP/injectSSHKey failures are guarded.
  - confidence: 0.89
- **[P1] Return real delete failures before removing lease state:** `internal/providers/tart/backend.go:273`
  - body: `ReleaseLease` discards `tart delete` failures and then removes the lease claim and stored key. That is only safe for a proven stale/missing VM; for any real delete failure, Crabbox will report success while the VM remains and future cleanup loses the lease/key state needed to manage it.
  - confidence: 0.91
- **[P2] Accept clone-default disk consistently across env config:** `internal/providers/tart/flags.go:80`
  - body: The docs and CLI support disk `0` as “use clone default,” and `configureVM` skips `--disk-size` when the value is zero, but `CRABBOX_TART_DISK=0` still fails validation here. That makes the documented clone-default setting impossible through the env surface, unlike the CLI/YAML behavior.
  - confidence: 0.84
