## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve generic Tart SSH settings:** `internal/providers/tart/backend.go:65-67`
  - body: Because baseConfig pre-populates Tart.User and Tart.WorkRoot, applyDefaults always copies the Tart defaults back over cfg.SSHUser and cfg.WorkRoot. A user who sets top-level ssh.user or workRoot for provider=tart will silently get admin and /Users/admin/crabbox instead unless the Tart-specific keys are also set.
  - confidence: 0.93
- **[P1] Preserve kept Tart VMs after post-start failures:** `internal/providers/tart/backend.go:133-134`
  - body: After tart run succeeds, the waitForIP, injectSSHKey, prepareLease, claim, and endpoint-update error paths still stop and delete the VM even when req.Keep is true. That makes --keep fail exactly when the operator needs the partially provisioned local VM for inspection.
  - confidence: 0.94
- **[P1] Honor cancellation before keep-mode launch:** `internal/providers/tart/backend.go:410`
  - body: The keep path uses exec.Command and the startup select only waits for process exit or the fixed timer, so a canceled context can still launch a detached Tart VM and return nil. Check ctx before starting and handle ctx.Done during the observation window.
  - confidence: 0.91
- **[P2] Treat running state as active without Running:** `internal/providers/tart/backend.go:185`
  - body: Resolve rejects non-status callers by checking inst.Running, but that bool is false when Tart omits the JSON field even if State is "running". Use the normalized state helper here so valid running leases are not reported as stopped.
  - confidence: 0.92
- **[P2] Let stop prune stale claimed Tart leases:** `internal/providers/tart/backend.go:175`
  - body: ReleaseOnly resolution calls resolveInstance first, and resolveInstance errors when a local claim points at a VM that is already gone. That prevents crabbox stop from removing Crabbox's stale claim/key state for a lease it can still identify locally.
  - confidence: 0.88
- **[P2] Align --tart-disk 0 with the documented sentinel:** `internal/providers/tart/flags.go:50-51`
  - body: The flag help says 0 means use the clone default, but ApplyFlags rejects --tart-disk 0 as invalid. Either accept zero as the clone-default sentinel or remove that advertised behavior from help/docs so users can rely on the CLI contract.
  - confidence: 0.9
