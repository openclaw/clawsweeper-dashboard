## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve generic Tart SSH settings:** `internal/cli/config.go:777-785`
  - body: Because baseConfig now always seeds `Tart.User` and `Tart.WorkRoot`, this provider-default branch overwrites a user's top-level `ssh.user` and `workRoot` even when `tart.user`/`tart.workRoot` were never configured. That makes `provider: tart` ignore shared Crabbox config unless users duplicate the same values under `tart:`, which is a compatibility break for the generic config contract.
  - confidence: 0.89
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:132-140`
  - body: These cleanup branches run even when `req.Keep` is true, so a `crabbox run --keep --provider tart` that starts the VM but then fails IP polling, key injection, SSH readiness, claim write, or endpoint update immediately stops and deletes the VM. Adjacent local lease providers preserve post-start failures for kept leases so operators can inspect them.
  - confidence: 0.9
- **[P1] Honor cancellation before detached Tart starts:** `internal/providers/tart/backend.go:409-410`
  - body: The keep path uses `exec.Command` instead of `exec.CommandContext` and does not check `ctx` before `Start`, so a canceled acquire can still launch a detached `tart run` process. If later cleanup is interrupted or fails, the user can be left with an unexpected local VM after cancellation.
  - confidence: 0.84
- **[P2] Use Tart state when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: Resolve rejects non-status callers whenever `Running` is false before consulting the parsed `State`. Tart inventory already carries `State`, and output variants that omit `Running` will decode it as false, causing a claimed VM with `State: "running"` to be treated as stopped for run/ssh/code/cache/egress reuse.
  - confidence: 0.88
- **[P2] Allow stale claimed leases to be pruned:** `internal/providers/tart/backend.go:252-254`
  - body: `ReleaseLease` resolves a lease ID through `resolveInstance`, but `resolveInstance` returns an error when the claim points at a VM that has already disappeared. That means `crabbox stop --provider tart <lease>` cannot remove the stale Tart claim/key; the release path should prune the claim while still refusing to delete unclaimed VMs.
  - confidence: 0.83
- **[P3] Align the Tart disk zero contract:** `internal/providers/tart/flags.go:24`
  - body: The flag help advertises `0 = use clone default`, but the flag application rejects an explicit zero as invalid. Either accept zero without marking disk resize explicit, or change the help/docs so users do not get a contradictory CLI contract.
  - confidence: 0.91
