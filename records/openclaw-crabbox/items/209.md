## Best Possible Solution

Land the macOS-only Tart provider after ReleaseLease handles both fallback-resolved and pre-resolved missing-VM leases before tart stop/delete, with Linux guest support tracked separately if maintainers want it.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: crabbox stop resolves with ReleaseOnly, Resolve returns a missing instance with CloudID set, and ReleaseLease only prunes missing instances in its fallback resolve branch.

## Solution Assessment

No. The current patch is close, but the durable fix is to handle missing-state leases before stop/delete regardless of whether the instance name came from fallback resolution or a pre-resolved ReleaseOnly LeaseTarget.

Full review comments:

- **[P1] Prune release-only missing Tart leases before stop/delete:** `internal/providers/tart/backend.go:283-284`
  - body: `crabbox stop` first resolves with `ReleaseOnly`, which returns a `LeaseTarget` whose `Server.CloudID` is already set even when the claim points at a missing VM. Because this method only prunes `State:"missing"` inside the fallback block where `name == ""`, the real stop path falls through to `tart stop`/`tart delete`; delete then fails for an already-deleted VM and the stale claim/key remain. Please prune missing-state leases before stop/delete and add a regression that calls `Resolve(... ReleaseOnly:true)` followed by `ReleaseLease`.
  - confidence: 0.96
