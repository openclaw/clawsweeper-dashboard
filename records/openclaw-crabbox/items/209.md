Full review comments:

- **[P1] Preserve generic Tart SSH and work-root overrides:** `internal/cli/config.go:777-785`
  - body: Because baseConfig pre-populates Tart.User and Tart.WorkRoot, this new branch copies those provider defaults back over top-level ssh.user/workRoot. A config with provider=tart, ssh.user=root, and workRoot=/custom still loads as admin and /Users/admin/crabbox unless the Tart-specific fields are also set, so the provider needs explicit Tart markers or deferred defaults so generic settings flow through when Tart-specific values are unset.
  - confidence: 0.9
- **[P1] Use Tart state when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: This check treats a missing JSON Running field as false even when Tart reports State="running". Some Tart list output may omit Running, so claimed running VMs can be rejected as stopped for run/ssh/cache reuse; use the normalized State helper for the active/stopped boundary.
  - confidence: 0.92
- **[P1] Do not delete kept VMs after post-start failures:** `internal/providers/tart/backend.go:132-135`
  - body: After tart run has succeeded, the waitForIP and later provisioning failure paths still stop and delete the VM unconditionally. With --keep, those are exactly the failures where the operator expects the partially started local VM to remain available for inspection, so guard post-start cleanup with !req.Keep.
  - confidence: 0.9
- **[P1] Honor cancellation before starting kept VMs:** `internal/providers/tart/backend.go:409-414`
  - body: The keep=true path uses exec.Command and the startup select only waits for process exit or a timer, so a canceled context can still launch a detached Tart VM and return success after the observation window. Check ctx.Err before Start and handle ctx.Done during the startup window, killing the just-started process if cancellation arrives.
  - confidence: 0.88
- **[P2] Let stop prune claim-backed missing VMs:** `internal/providers/tart/backend.go:528-538`
  - body: When a stored Tart claim points to an instance that no longer appears in tart list, resolveInstance errors before ReleaseOnly can return a target for ReleaseLease. That leaves crabbox stop --provider tart <lease-or-slug> unable to remove the stale local claim/key, so ReleaseOnly needs a missing-instance path that prunes local state without calling tart stop/delete.
  - confidence: 0.86
- **[P3] Align disk zero with clone-default behavior:** `internal/providers/tart/flags.go:49-51`
  - body: The flag help and provider docs describe zero as the clone-default disk sentinel, and configureVM already skips --disk-size when Disk is 0, but validation rejects --tart-disk 0 and explicit config/env zero. Accept zero as clone-default or change every documented contract consistently.
  - confidence: 0.88
