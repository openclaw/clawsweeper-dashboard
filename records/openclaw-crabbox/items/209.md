## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve generic Tart SSH settings:** `internal/cli/config.go:777-785`
  - body: Because `baseConfig` pre-populates `Tart.User` and `Tart.WorkRoot`, this branch copies provider defaults over top-level `ssh.user` and `workRoot`. A config with `provider: tart` plus generic SSH/workroot settings silently becomes `admin` and `/Users/admin/crabbox`; track explicit Tart fields or let generic settings flow through unless Tart-specific values were set.
  - confidence: 0.91
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-135`
  - body: After `startVM` succeeds, the IP, SSH injection, prepare, claim, and endpoint failure branches still stop and delete the VM even when `req.Keep` is true. That makes `crabbox run --provider tart --keep` unable to leave a partially provisioned VM for inspection after readiness failures; guard post-start cleanup with `!req.Keep`.
  - confidence: 0.9
- **[P1] Honor cancellation for kept Tart starts:** `internal/providers/tart/backend.go:410-414`
  - body: The keep path uses `exec.Command` and the observation select only waits for process exit or the fixed timer, so an already-canceled acquire can still launch a detached Tart VM and return success. Check `ctx.Err()` before starting and select on `ctx.Done()` during the startup window.
  - confidence: 0.88
- **[P2] Use Tart state when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: `Running` defaults to false when Tart JSON omits that field, even if `State` is `running`, so this rejects a claimed active VM for non-status resolve paths. Use the existing normalized `instanceRunning(inst.State)` helper for this boundary.
  - confidence: 0.89
- **[P2] Return a release target for missing claimed VMs:** `internal/providers/tart/backend.go:538`
  - body: When a local Tart claim exists but the VM is already gone, `resolveInstance` errors before `ReleaseLease` can prune the stale claim and key. For `ReleaseOnly`, return a missing-instance target and remove local state without calling `tart stop` or `tart delete`.
  - confidence: 0.82
- **[P3] Align the tart disk zero contract:** `internal/providers/tart/flags.go:50-51`
  - body: The help text says `0 = use clone default`, and `configureVM` skips disk resize unless `Disk > 0`, but the flag validator rejects `--tart-disk 0`. Accept zero as the documented sentinel or remove that contract from help and docs.
  - confidence: 0.93
