## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve kept VMs after SSH readiness failures:** `internal/providers/tart/backend.go:161-163`
  - body: For `--keep`, `prepareLease(..., wait=true)` performs the final SSH readiness wait after the VM is already started and the key has been injected. If that wait fails, this block always stops and deletes the VM, so a user who requested a kept Tart VM loses the instance they likely need for reuse or debugging; gate this cleanup on `!req.Keep` or otherwise preserve the kept VM and cover it with a regression test.
  - confidence: 0.93
- **[P1] Keep claims when live Tart deletion fails:** `internal/providers/tart/backend.go:272-275`
  - body: `ReleaseLease` ignores `deleteVM` errors and then removes the lease claim and stored SSH key. That is safe for confirmed stale claims where the VM is already gone, but if `tart delete` fails for a live or locked VM, Crabbox forgets the lease while the VM remains, leaving an orphan that can no longer be managed with the stored key; only prune state for known-missing VMs or propagate unexpected delete failures.
  - confidence: 0.91
- **[P1] Stop started kept VMs when startup is canceled:** `internal/providers/tart/backend.go:433-437`
  - body: In the keep path, if the context is canceled after `cmd.Start` but before the startup observation window completes, this branch returns without killing or stopping the started Tart process. The caller then only tries `deleteVM` without a stop, which can fail for a running VM and leave an unclaimed kept VM behind; acquisition cancellation before a claim exists should stop and delete the started VM even when the requested lease was `--keep`.
  - confidence: 0.86
