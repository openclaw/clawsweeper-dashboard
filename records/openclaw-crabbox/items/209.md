Overall confidence: 0.89

Full review comments:

- **[P1] Preserve ownership state when Tart delete fails:** `internal/providers/tart/backend.go:273-276`
  - body: `ReleaseLease` ignores `deleteVM` errors and then removes the lease claim and stored key. If `tart delete` fails for a real VM, Crabbox reports release success while the VM can remain present or running without the normal claim/key path for later status or cleanup; only confirmed missing-VM stale-claim pruning should tolerate the delete failure.
  - confidence: 0.94
- **[P2] Reject running Tart VMs that still have no IP:** `internal/providers/tart/backend.go:595-599`
  - body: Final head lets `Resolve` accept Tart JSON where `Running` is omitted but `State` is `running`, yet `prepareLease` still treats `Running=false` plus an empty IP as a non-error stopped lease. That can hand SSH-target commands a lease with no SSH endpoint instead of a clear unavailable error when Tart reports a running VM without an IP.
  - confidence: 0.9
- **[P2] Honor keep semantics through SSH readiness failure:** `internal/providers/tart/backend.go:160-164`
  - body: For `--keep` acquisitions, failures in `waitForIP` and key injection preserve the started VM, but an SSH readiness failure from `prepareLease(..., wait=true)` still triggers unconditional stop/delete. That leaves the keep=true post-start failure behavior inconsistent and can destroy exactly the VM the user asked to retain for debugging or reuse.
  - confidence: 0.82
