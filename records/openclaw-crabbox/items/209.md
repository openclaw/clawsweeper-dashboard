## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-135`
  - body: Once `startVM(..., keep=true)` succeeds, later failures in IP discovery, SSH key injection, SSH readiness, claim creation, or endpoint persistence still run `stopVM` and `deleteVM` unconditionally. That makes `--keep` lose the VM state users kept for reuse/debugging; gate post-start cleanup on `!req.Keep` or add an explicit tested policy.
  - confidence: 0.92
- **[P1] Honor cancellation before launching kept VMs:** `internal/providers/tart/backend.go:410`
  - body: The keep path uses `exec.Command` and never checks `ctx` before `Start`, so a canceled or timed-out acquire can still launch a detached Tart VM after the caller has abandoned the operation. Check the context before starting and cover the canceled keep path in tests.
  - confidence: 0.86
- **[P1] Inherit generic Tart SSH user and work root safely:** `internal/cli/config.go:777-785`
  - body: Because `baseConfig` always fills `Tart.User` and `Tart.WorkRoot`, this branch overwrites top-level `ssh.user` and `workRoot` with `admin` and `/Users/admin/crabbox` unless users also set the Tart-specific fields. That silently ignores existing generic config and can inject keys or sync into the wrong account/path; track Tart-specific explicitness or only default after generic config has been honored.
  - confidence: 0.9
- **[P2] Use Tart state instead of the optional Running bool:** `internal/providers/tart/backend.go:185-186`
  - body: `tart list` compatibility depends on `State`; if a Tart version omits `Running`, Go decodes it as false and this branch rejects a running VM for non-status resolves. Use `instanceRunning(inst.State)` or derive `Running` from `State` so claimed running VMs are not rejected.
  - confidence: 0.88
- **[P2] Prune stale claims when stop resolves a missing VM:** `internal/providers/tart/backend.go:252-254`
  - body: `ReleaseLease` resolves a lease ID before removing the claim, but `resolveInstance` returns an error when the claim's VM is already gone. That means `crabbox stop --provider tart <lease>` cannot clean up the stale local claim; handle the missing-instance release path by removing the claim/key without touching unrelated VMs.
  - confidence: 0.82
- **[P3] Align the explicit zero disk contract:** `internal/providers/tart/flags.go:24`
  - body: The flag help says `--tart-disk` uses `0 = use clone default`, but `applyFlags` rejects `<= 0` and the config/env validation rejects explicit zero. Users following the help will get an error, so either allow explicit zero consistently or change the help/docs to say the flag must be omitted for clone default.
  - confidence: 0.95
