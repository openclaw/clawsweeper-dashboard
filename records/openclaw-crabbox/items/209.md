## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve kept Tart VMs after post-start failures:** `internal/providers/tart/backend.go:133-139`
  - body: After `startVM(..., keep=true)` succeeds, any IP wait, key injection, SSH readiness, claim, or endpoint error still runs `tart stop` and `tart delete`. That makes a `--keep` VM disappear during the exact failure window where users need it preserved for inspection; follow the local-VM pattern and only remove post-start VMs on these paths when `!req.Keep`, with focused regression tests.
  - confidence: 0.91
- **[P1] Honor cancellation before launching kept VMs:** `internal/providers/tart/backend.go:409-414`
  - body: The keep path switches to `exec.Command`, but it never checks `ctx` before `cmd.Start()` or while waiting for the startup observe window. A canceled acquire can therefore still launch a detached Tart VM and return nil after two seconds; check cancellation before start and include `ctx.Done()` in the observe select.
  - confidence: 0.88
- **[P2] Treat State as running when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: `Running` is an optional JSON bool, so a Tart inventory item like `{State:"running"}` decodes as `Running=false` and is rejected for every non-status resolve. The provider already normalizes `State` through `instanceRunning`; use that state path here so older or variant Tart output does not make valid leases unavailable.
  - confidence: 0.9
- **[P2] Prune stale Tart claims during stop:** `internal/providers/tart/backend.go:252-254`
  - body: When `crabbox stop --provider tart <lease>` resolves a claim whose VM has already disappeared, `resolveInstance` returns an error and `ReleaseLease` exits before removing the local claim/key. That leaves a stale lease that the user's explicit stop command cannot clear; handle the missing-instance case in release-only cleanup and add a regression test.
  - confidence: 0.84
- **[P3] Align the tart disk zero contract:** `internal/providers/tart/flags.go:24`
  - body: The flag help advertises `0 = use clone default`, but the flag parser rejects `--tart-disk 0`. Either accept explicit zero as the documented clone-default sentinel or change the help/docs/tests to say the flag must be omitted for clone default.
  - confidence: 0.96
