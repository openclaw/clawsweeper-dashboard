## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve lease state when Tart delete fails:** `internal/providers/tart/backend.go:272-275`
  - body: `ReleaseLease` ignores `deleteVM` for every release and then removes the claim and stored key. If `tart delete` fails because the VM is busy or the host rejects deletion, Crabbox loses the only ownership state while the VM remains; only tolerate missing/stale VM cases or keep the claim on a real delete failure.
  - confidence: 0.9
- **[P1] Honor keep on every post-start acquire failure:** `internal/providers/tart/backend.go:161-174`
  - body: After the VM has started, failures from SSH readiness, claim creation, or endpoint update still hit this unconditional stop/delete cleanup. That can destroy a VM the user asked to keep; gate every post-start cleanup on `!req.Keep` or add an explicit tested policy for deleting kept acquisitions.
  - confidence: 0.86
- **[P2] Check State before returning stopped no-IP leases:** `internal/providers/tart/backend.go:595-600`
  - body: `Resolve` treats `State=running` as live when the `Running` JSON field is omitted, but `prepareLease` still uses only `!inst.Running` when IP is empty. A running Tart VM with omitted `Running` and no resolved IP can return a LeaseTarget without SSH details instead of an actionable error for non-status commands.
  - confidence: 0.84
