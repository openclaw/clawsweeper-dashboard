## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve kept Tart VMs after post-start failures:** `internal/providers/tart/backend.go:133-139`
  - body: After `startVM` succeeds, this path still calls `tart stop` and `tart delete` on IP/key/readiness failures even when `req.Keep` is true. That defeats `--keep` for the exact cases where a user needs the VM left around for reuse or debugging, so the cleanup needs to be conditional once the kept VM has started.
  - confidence: 0.9
- **[P1] Honor cancellation before starting kept Tart VMs:** `internal/providers/tart/backend.go:410`
  - body: The keep path builds `exec.Command` and starts `tart run` without first checking `ctx`, so a request that has already timed out or been canceled can still launch a detached kept VM. Check `ctx.Err()` before starting or otherwise wire cancellation so canceled acquisitions do not create local VMs.
  - confidence: 0.87
- **[P1] Treat Tart State=running as running in Resolve:** `internal/providers/tart/backend.go:185`
  - body: `Resolve` rejects non-status callers when `inst.Running` is false, but the provider already has `instanceRunning(inst.State)` because Tart state comes from the string field too. If Tart omits `Running` while reporting `State: "running"`, SSH-target commands will reject a live VM as stopped.
  - confidence: 0.89
- **[P1] Preserve generic Tart SSH user and work root overrides:** `internal/cli/config.go:778-785`
  - body: Because `baseConfig` pre-populates `cfg.Tart.User` and `cfg.Tart.WorkRoot`, this Tart default block overwrites top-level `ssh.user` and `workRoot` even when the user did not set Tart-specific values. That breaks provider-neutral config expectations; only explicit Tart-specific user/workRoot should win over the generic settings.
  - confidence: 0.84
- **[P2] Align tart disk zero semantics with the help text:** `internal/providers/tart/flags.go:50-51`
  - body: The flag help says `0 = use clone default`, but explicitly passing `--tart-disk 0` reaches this branch and exits with an error. Either allow zero as the documented clone-default sentinel or change the help/docs so users are not told to pass a rejected value.
  - confidence: 0.82
