## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve generic Tart SSH and work-root overrides:** `internal/cli/config.go:777-785`
  - body: Because `baseConfig()` pre-populates `cfg.Tart.User` and `cfg.Tart.WorkRoot`, this branch copies the Tart defaults back over top-level `ssh.user` and `workRoot`. A config with `provider: tart` plus generic SSH/path settings will still run as `admin` in `/Users/admin/crabbox` unless the user duplicates those values under `tart:`, which silently ignores existing generic settings.
  - confidence: 0.88
- **[P1] Honor cancellation before kept Tart starts:** `internal/providers/tart/backend.go:410-414`
  - body: The `keep` path starts `tart run` with plain `exec.Command` and the startup wait does not select on `ctx.Done()`. If the acquire context is already canceled or is canceled during the observation window, Crabbox can still launch a kept local VM after the caller has abandoned the operation.
  - confidence: 0.9
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:132-166`
  - body: After `startVM` succeeds, each later acquisition failure still calls `stopVM` and `deleteVM` unconditionally. That means `--keep` cannot preserve a VM for inspection after IP, SSH injection, readiness, claim, or endpoint-update failure, unlike the adjacent Multipass/local-container keep behavior.
  - confidence: 0.86
- **[P2] Use Tart state, not optional Running, for Resolve:** `internal/providers/tart/backend.go:185-186`
  - body: `tart list --format json` can omit the `Running` boolean while still reporting `State: "running"`; the zero value then makes this check reject a claimed running VM as stopped. Use the normalized state helper here so existing Tart leases can be reused by run/ssh/cache commands.
  - confidence: 0.9
- **[P2] Let release-only stop prune stale Tart claims:** `internal/providers/tart/backend.go:528-538`
  - body: When a local Tart claim exists but the VM no longer appears in `tart list`, `resolveInstance` errors before the stop path can remove the stale claim and stored key. Release-only resolution should return a stale target or otherwise prune claim-backed local state without requiring the missing VM to exist.
  - confidence: 0.82
- **[P3] Align the Tart disk zero contract:** `internal/providers/tart/flags.go:49-52`
  - body: The flag help and provider docs say `0` means clone default, but validation rejects `--tart-disk 0` and `CRABBOX_TART_DISK=0`. Either accept zero as the documented sentinel or change the help/docs so users are not told to use a value the CLI rejects.
  - confidence: 0.9
