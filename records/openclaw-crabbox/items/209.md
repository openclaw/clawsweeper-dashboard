## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Honor cancellation before launching kept VMs:** `internal/providers/tart/backend.go:410`
  - body: In keep mode this path builds `exec.Command` without the caller context and does not check `ctx.Done()` before `Start`, so a canceled or timed-out acquire can still launch a detached `tart run` process and then rely on later best-effort cleanup. Please fail before starting the VM when the context is already canceled and add a regression for canceled keep-mode startup.
  - confidence: 0.92
- **[P2] Use state-aware running checks in Resolve:** `internal/providers/tart/backend.go:185-186`
  - body: `Resolve` rejects non-status callers solely from the decoded `Running` bool. If Tart output omits that field while still reporting `State: "running"`, the zero bool makes live claimed VMs look stopped and breaks ssh/run/code/cache/egress resolution; use the normalized state helper or a combined state/bool check instead.
  - confidence: 0.86
- **[P2] Let stop prune stale Tart claims:** `internal/providers/tart/backend.go:538`
  - body: When a lease claim exists but the VM has already been deleted, this error aborts `crabbox stop --id <lease>` before `ReleaseLease` can remove the stale claim and stored key. The explicit stop path should be able to release claimed Tart state even when the backing VM is gone, while still refusing unclaimed VM deletion.
  - confidence: 0.82
- **[P3] Align disk help with validation:** `internal/providers/tart/flags.go:24`
  - body: The help text says `--tart-disk 0` means use the clone default, but `applyFlags` rejects zero as invalid. Either remove the zero claim from help/docs or make zero a valid explicit clone-default value so users do not get a documented-but-failing flag.
  - confidence: 0.95
