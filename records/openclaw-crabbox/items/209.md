## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-135`
  - body: After `startVM` succeeds, these failure paths still call `stopVM` and `deleteVM` without checking `req.Keep`. That means `crabbox run --provider tart --keep` can delete the partially booted VM exactly when the operator needs it left around for inspection; guard the post-start cleanup branches the same way the adjacent local providers do.
  - confidence: 0.95
- **[P2] Honor cancellation before starting kept VMs:** `internal/providers/tart/backend.go:410`
  - body: The keep path drops the context by using plain `exec.Command`, and the startup observation select never listens to `ctx.Done()`. If the acquire context is already canceled or times out during the observe window, this can still launch a background Tart VM and return success; check `ctx.Err()` before start and kill/return on cancellation during observation.
  - confidence: 0.93
- **[P2] Preserve generic SSH user and work root:** `internal/cli/config.go:777-785`
  - body: Because `baseConfig` pre-populates `Tart.User` and `Tart.WorkRoot`, this branch copies the provider defaults back over top-level `ssh.user` and `workRoot` for `provider: tart`. Track whether `tart.user`/`tart.workRoot` were explicitly configured so generic settings still work unless the Tart-specific values are set.
  - confidence: 0.9
- **[P3] Align tart-disk help with validation:** `internal/providers/tart/flags.go:24`
  - body: The flag help says `0 = use clone default`, but `applyFlags` rejects `--tart-disk 0`. Either allow the documented sentinel or update the help text so users do not follow a value the CLI refuses.
  - confidence: 0.98
