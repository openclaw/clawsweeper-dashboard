## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor cancellation before kept Tart starts:** `internal/providers/tart/backend.go:410`
  - body: When `keep` is true, this path uses `exec.Command` and never checks `ctx` before starting `tart run`. A canceled warmup/run can still launch a detached local VM before the caller observes cancellation, which is exactly the kind of local resource surprise keep-mode should avoid. Check the context before launch and make the startup observation path cancellation-aware.
  - confidence: 0.91
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-139`
  - body: After `tart run` succeeds, readiness and key-injection failures always stop and delete the VM, even when the user requested `--keep`. That removes the VM a user expected to inspect or reuse after a failed kept acquisition; match the other local SSH providers by preserving post-start kept resources and adding a regression test.
  - confidence: 0.88
- **[P1] Use Tart state when Running is omitted:** `internal/providers/tart/backend.go:185`
  - body: Resolve treats the optional decoded `Running` bool as authoritative. If Tart returns `State: "running"` without a `Running` field, Go decodes `Running=false` and every non-status command rejects the valid lease as stopped. Use `instanceRunning(inst.State)` or synthesize `Running` from `State` before the `StatusOnly` check.
  - confidence: 0.9
- **[P3] Align the Tart disk zero contract:** `internal/providers/tart/flags.go:50-51`
  - body: The flag help advertises `--tart-disk 0` as the clone-default sentinel, but validation rejects zero. Either accept zero as the documented no-resize value or remove that promise from the help/docs so users do not get a contradictory CLI contract.
  - confidence: 0.86
