## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:131-135`
  - body: After `tart run` succeeds, later acquisition failures always stop and delete the VM, even when `req.Keep` is true. Existing local SSH-lease providers preserve kept resources after post-start failures so operators can inspect readiness, SSH, or claim-writing problems.
  - confidence: 0.94
- **[P1] Honor cancellation before detached Tart starts:** `internal/providers/tart/backend.go:409-414`
  - body: The `keep=true` branch creates `exec.Command` without checking `ctx`, so a canceled acquire can still launch `tart run` and wait out the observation window. Check cancellation before starting or otherwise wire cancellation into this path before detaching.
  - confidence: 0.9
- **[P2] Derive running state from Tart state too:** `internal/providers/tart/backend.go:185-186`
  - body: Resolve rejects non-status callers when `inst.Running` is false, but Go leaves that bool false if Tart omits the field while still reporting `State: "running"`. Use the normalized state predicate, or combine both fields, so live claimed VMs are not rejected.
  - confidence: 0.89
- **[P1] Preserve explicit generic SSH settings:** `internal/providers/tart/backend.go:65-68`
  - body: `applyDefaults` always copies Tart defaults into the effective SSH user/work root, so a user who set generic `ssh.user` or `workRoot` but not Tart-specific settings gets reset to `admin` and `/Users/admin/crabbox`. Preserve explicit generic settings unless a Tart-specific value was provided.
  - confidence: 0.87
- **[P2] Make disk-zero help match validation:** `internal/providers/tart/flags.go:49-52`
  - body: The flag help documents `--tart-disk 0` as clone-default behavior, but ApplyFlags rejects zero before the run can start. Either accept explicit zero as clearing the disk override or remove the documented zero semantics.
  - confidence: 0.91
- **[P2] Prune stale claims when the VM is gone:** `internal/providers/tart/backend.go:538`
  - body: `ReleaseLease` resolves lease IDs through `resolveInstance`, which errors when the claim points to a VM that no longer exists, so `crabbox stop --provider tart <lease>` never reaches local claim/key removal. Let release-only cleanup prune stale claims even when the runtime object is already gone.
  - confidence: 0.84
