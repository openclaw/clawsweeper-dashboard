## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use Tart state when Running is absent:** `internal/providers/tart/backend.go:185-186`
  - body: `tartInstance.Running` decodes to `false` when the optional JSON field is omitted, so a record with `State:"running"` is rejected as stopped for non-status resolves. Use the existing normalized state helper at this boundary so valid running leases remain reusable.
  - confidence: 0.93
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:132-139`
  - body: After `startVM` succeeds, the later IP, key injection, prepare, claim, and endpoint-update failure paths still stop and delete the VM even when `req.Keep` is true. That loses the local VM state users asked to keep for inspection; guard post-start cleanup with `!req.Keep` and cover the failure path.
  - confidence: 0.94
- **[P1] Honor cancellation before detached Tart starts:** `internal/providers/tart/backend.go:409-414`
  - body: The keep path uses plain `exec.Command` and the startup observation select has no `ctx.Done()` branch, so an already-canceled acquire can still launch a detached Tart VM and return success after the timer. Check cancellation before starting and stop the child if cancellation arrives during the observation window.
  - confidence: 0.91
- **[P1] Preserve generic Tart user and work-root overrides:** `internal/cli/config.go:1093-1097`
  - body: These base Tart defaults make the loader unable to tell defaults from explicit `tart:` values, and the Tart defaulting branch later copies them over top-level `ssh.user` and `workRoot`. Users selecting Tart with generic config can silently run as `admin` in `/Users/admin/crabbox`; track explicit Tart-specific settings and inherit generic values when they are the only user-provided settings.
  - confidence: 0.86
- **[P3] Align the tart disk zero validation:** `internal/providers/tart/flags.go:24`
  - body: The flag help documents `0` as the clone-default sentinel, but the CLI rejects `--tart-disk 0` before the backend can skip resizing. Either accept zero as documented or change the help/docs so users are not given an impossible reset value.
  - confidence: 0.9
