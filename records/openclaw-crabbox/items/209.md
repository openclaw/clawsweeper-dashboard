## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use Tart state when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: This rejects a usable VM whenever `tart list` omits the optional `Running` boolean, because Go leaves it false even if `State` is `running`. The rest of this file already normalizes state via `instanceRunning`, so non-status Resolve should use that state check instead of the raw boolean or commands like `run`, `ssh`, `cache`, and `egress` can fail against live claimed VMs.
  - confidence: 0.92
- **[P1] Honor cancellation before starting kept VMs:** `internal/providers/tart/backend.go:409-414`
  - body: For `keep=true`, `startVM` uses `exec.Command` and never checks `ctx` before `cmd.Start`, so a request canceled before this point can still launch a detached Tart VM that Crabbox then no longer owns through the canceled operation. Check `ctx.Err()` before starting and add coverage so cancellation cannot create a background VM.
  - confidence: 0.91
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-139`
  - body: `--keep` is the escape hatch for inspecting a newly created local runner after readiness/setup fails. The existing Multipass and local-container acquire paths only remove the resource on later failures when `Keep` is false, but Tart stops and deletes the VM after IP readiness or SSH-key injection fails regardless of `req.Keep`, so users lose the VM they asked to retain for debugging.
  - confidence: 0.88
- **[P3] Align the tart-disk zero contract:** `internal/providers/tart/flags.go:24`
  - body: The flag help says `0 = use clone default`, but the parser rejects `--tart-disk 0` before the provider can use clone-default behavior. Either accept explicit zero as clone default or change the help text so users are not told to pass a value that always fails.
  - confidence: 0.86
