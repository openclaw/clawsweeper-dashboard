## Summary

The PR is useful but still needs changes: source inspection shows tart leases return an SSH target host without copying that IP into the returned `Server`, which breaks the standard status/inspect host and readiness path; the new tart config surface also remains absent from `config show`.

## What This Changes

The PR adds a built-in `tart` local SSH-lease provider for Apple Silicon macOS VMs, with tart lifecycle code, CLI/YAML/env config, provider registration, tests, and provider docs.

## Best Possible Solution

Land the tart provider after the lease/status host metadata and config diagnostics match existing SSH-provider patterns, with maintainer sign-off on the new built-in provider contract.

## Reproduction Assessment

Yes for the PR defect by source inspection: `prepareLease` has the tart IP and `statusViewFromLeaseTarget` requires it on `Server.PublicNet.IPv4.IP`, but the PR never copies it there. I did not reproduce the full tart lifecycle locally because it requires Apple Silicon with tart installed, and the PR body supplies real terminal proof for warmup, run, and stop.

## Solution Assessment

No; the provider is close, but it should follow the existing SSH-provider contract by returning a server with the resolved IP and exposing its effective config through `config show`. The safer path is a narrow repair before maintainers decide whether to accept the built-in provider surface.
