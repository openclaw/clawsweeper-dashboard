## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve generic Tart SSH and work-root overrides:** `internal/cli/config.go:777-785`
  - body: Because `baseConfig()` pre-populates `Tart.User` and `Tart.WorkRoot`, this branch always copies the Tart defaults back over top-level `ssh.user` and `workRoot`. A config that sets only the generic SSH user or work root still runs as `admin` under `/Users/admin/crabbox`; add explicit Tart user/work-root markers or equivalent precedence so generic settings win unless `tart.user`/`tart.workRoot` were explicitly set.
  - confidence: 0.92
- **[P1] Use Tart state when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: `Running` is an optional JSON boolean, so a `tart list` record with `State:"running"` but no `Running` field decodes as `false` and this path rejects a live claimed VM as stopped. Use the normalized state helper for this boundary so existing running leases remain usable.
  - confidence: 0.91
- **[P1] Keep kept VMs after post-start failures:** `internal/providers/tart/backend.go:131-140`
  - body: After `startVM` succeeds, every later failure path still stops and deletes the VM even when the request used `Keep`. That removes exactly the partially provisioned local VM the operator asked to keep for inspection; guard the post-start cleanup branches with `!req.Keep` while preserving cleanup for non-keep runs.
  - confidence: 0.91
- **[P1] Honor cancellation before starting kept VMs:** `internal/providers/tart/backend.go:409-414`
  - body: The keep path uses plain `exec.Command` and the startup wait only selects on process exit or a timer, so an already-canceled acquire can still launch a background Tart VM and return success after the observation window. Check `ctx.Err()` before `Start` and handle `ctx.Done()` during startup observation.
  - confidence: 0.9
- **[P2] Let release-only stop prune stale Tart claims:** `internal/providers/tart/backend.go:175-178`
  - body: `Resolve` calls `resolveInstance` before it considers `ReleaseOnly`, so a claim whose Tart VM has already been deleted fails before `ReleaseLease` can remove the local claim and key. Add a release-only claim path that returns a missing-instance target and prunes local state without calling `tart stop` or `tart delete`.
  - confidence: 0.86
- **[P3] Align tart disk zero with clone-default help:** `internal/providers/tart/flags.go:49-51`
  - body: The flag help says `0 = use clone default`, but this validation rejects `--tart-disk 0`; users also cannot override an env/config disk size back to clone default from the CLI. Either accept explicit zero as the clone-default sentinel or change the help/docs to remove that contract.
  - confidence: 0.88
