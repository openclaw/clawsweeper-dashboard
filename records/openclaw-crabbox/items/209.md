- **[P1] Refresh the branch against current main:** `internal/cli/config.go:17`
  - body: `git merge-tree` against current main reports conflicts in `internal/cli/config.go` and `internal/cli/config_cmd.go`, so maintainers cannot review or land the final integrated config behavior from this head. Please rebase or merge current main and preserve both the recent config changes and the Tart additions.
  - confidence: 0.91
- **[P1] Preserve generic SSH user and workRoot overrides:** `internal/cli/config.go:777-785`
  - body: Because `baseConfig()` pre-populates `cfg.Tart.User` and `cfg.Tart.WorkRoot`, this branch always copies those provider defaults back over top-level `ssh.user` and `workRoot`. A user selecting `provider: tart` with generic Crabbox SSH/work-root settings will silently get `admin` and `/Users/admin/crabbox` unless they know to duplicate the values under `tart:`.
  - confidence: 0.9
- **[P3] Align tart-disk zero with the documented semantics:** `internal/providers/tart/flags.go:49-51`
  - body: The flag help says `0 = use clone default`, but the flag path rejects `--tart-disk 0` before the provider can treat it as clone default. Either accept explicit zero as the documented reset value or change the help/docs/tests so users do not get contradictory behavior.
  - confidence: 0.95
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:131-135`
  - body: After `startVM` succeeds, every later failure branch still stops and deletes the VM even when `req.Keep` is true. That breaks the inspection workflow for `--keep` exactly when IP polling, SSH injection, SSH readiness, or local claim writing fails after the VM has started.
  - confidence: 0.92
- **[P2] Honor cancellation before detached Tart launch:** `internal/providers/tart/backend.go:409-417`
  - body: The keep path uses plain `exec.Command` and the startup select only waits for process exit or a timer, so an already-canceled context can still launch a detached Tart VM and return success after the observation window. Check `ctx.Err()` before starting and handle `ctx.Done()` during the observation window.
  - confidence: 0.88
- **[P2] Use normalized Tart state when resolving leases:** `internal/providers/tart/backend.go:185-186`
  - body: `tart list` records can report `State:"running"` while omitting the optional `Running` boolean, which leaves this Go field false. The non-status resolve gate should use the existing `instanceRunning(inst.State)` helper so running VMs are not rejected as stopped.
  - confidence: 0.94
- **[P2] Let release-only stop remove unclaimed Tart VMs:** `internal/providers/tart/backend.go:179-183`
  - body: Release-only resolution checks for a Crabbox claim before returning a target, so `crabbox stop --provider tart <instance>` cannot remove a Crabbox-prefixed VM if the local claim was lost. The stop path already has the instance name, so release-only resolution should be able to return an unclaimed Crabbox-owned Tart VM for deletion.
  - confidence: 0.86
- **[P2] Let stop prune stale claims for missing Tart VMs:** `internal/providers/tart/backend.go:532-538`
  - body: When a claim exists but the referenced VM no longer appears in `tart list`, `resolveInstance` errors before `ReleaseLease` can remove the local claim and key. For `ReleaseOnly`, return a stale target or equivalent marker so stop can clean local Crabbox state without calling Tart for a missing VM.
  - confidence: 0.86
