## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use Tart state when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: `Running` defaults to false when Tart omits that optional JSON field, so this rejects a claimed VM whose `State` is still `running`. Use the existing `instanceRunning(inst.State)` helper for this boundary and keep the stopped-VM StatusOnly behavior covered by tests.
  - confidence: 0.93
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-135`
  - body: After `startVM` succeeds, the later IP, key injection, SSH readiness, claim, and endpoint failures still stop and delete the VM even when the request used `--keep`. That removes the inspection path for exactly the post-start failures operators need to debug; guard those cleanup branches with `!req.Keep` and add a regression test.
  - confidence: 0.9
- **[P1] Honor cancellation before launching kept Tart runs:** `internal/providers/tart/backend.go:409-414`
  - body: The keep path uses `exec.Command` and never checks `ctx.Err()` or selects on `ctx.Done()`, so an already-canceled acquire can still start a background `tart run` and return nil after the observation timeout. Check cancellation before starting and kill/reap the process if cancellation arrives during the startup window.
  - confidence: 0.91
- **[P2] Respect generic ssh.user and workRoot for Tart:** `internal/cli/config.go:777-785`
  - body: Because `baseConfig` pre-populates `Tart.User` and `Tart.WorkRoot`, this branch overwrites top-level `ssh.user` and `workRoot` with `admin` and `/Users/admin/crabbox` unless users know to set Tart-specific keys. Track explicit Tart user/workRoot settings so generic config remains effective unless provider-specific config wins.
  - confidence: 0.86
- **[P2] Let stop prune stale Tart claims:** `internal/providers/tart/backend.go:528-538`
  - body: When a local Tart claim points at a VM that no longer appears in `tart list`, `resolveInstance` errors before `ReleaseLease` can remove the stale claim and stored key. Add a release-only path for missing claimed instances so `crabbox stop --provider tart <lease-or-slug>` can clean local state without calling `tart stop/delete` on a missing VM.
  - confidence: 0.82
- **[P3] Align the tart disk zero contract:** `internal/providers/tart/flags.go:24`
  - body: The flag help advertises `0 = use clone default`, but `ApplyFlags` rejects `--tart-disk 0`. Either allow zero as the documented clone-default reset or update the help/docs/tests so users do not get contradictory behavior.
  - confidence: 0.88
