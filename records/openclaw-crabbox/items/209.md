## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve generic Tart SSH and work-root overrides:** `internal/providers/tart/backend.go:49-57`
  - body: `baseConfig()` pre-populates `cfg.Tart.User` and `cfg.Tart.WorkRoot`, so this fallback never inherits a user-provided generic `ssh.user` or `workRoot`; `applyProviderConfigDefaults` then copies the Tart defaults back over the generic settings. A Tart user who configures the normal Crabbox account/path can silently run as `admin` in `/Users/admin/crabbox` instead.
  - confidence: 0.93
- **[P1] Use Tart state when Running is omitted:** `internal/providers/tart/backend.go:185-186`
  - body: This check treats a missing JSON `Running` field as `false`, even when `tart list` reports `State:"running"`. Tart inventory that omits the optional boolean will make reuse paths reject a live claimed VM as stopped; use the normalized state helper for this boundary.
  - confidence: 0.94
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:132-135`
  - body: After `startVM` succeeds, the IP, SSH injection, lease preparation, claim, and endpoint-update failure branches still call `stop` and `delete` unconditionally. With `--keep`, those are the exact partially started VMs an operator expects to inspect after a provisioning failure.
  - confidence: 0.92
- **[P1] Honor cancellation before detached Tart starts:** `internal/providers/tart/backend.go:410-414`
  - body: The keep path uses `exec.Command` and the observation select only waits for process exit or a timer, so an already canceled context can still launch a detached Tart VM and return success. Check `ctx.Err()` before start and handle `ctx.Done()` during the observation window.
  - confidence: 0.9
- **[P2] Let stop prune claim-backed missing Tart VMs:** `internal/providers/tart/backend.go:528-538`
  - body: Release-only resolution still goes through `resolveInstance`, which errors when the local claim references a VM no longer present in `tart list`. That prevents `crabbox stop --provider tart <lease-or-slug>` from removing Crabbox's stale local claim/key state after the VM is already gone.
  - confidence: 0.88
- **[P3] Align the Tart disk zero contract:** `internal/providers/tart/flags.go:49-52`
  - body: The flag help says `0 = use clone default`, but this validation rejects `--tart-disk 0`, so users cannot override an env/YAML disk setting back to clone default from the CLI. Accept zero as the documented no-resize sentinel or change the docs/help consistently.
  - confidence: 0.87
