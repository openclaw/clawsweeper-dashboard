## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve kept Tart VMs after startup failures:** `internal/providers/tart/backend.go:132-140`
  - body: After `tart run` succeeds, both the IP-wait and SSH-key injection error paths unconditionally stop and delete the VM. That defeats `--keep` inspection semantics for a VM the user asked Crabbox to preserve, so these cleanup calls need to be conditional on `!req.Keep` or replaced with a kept-VM diagnostic path.
  - confidence: 0.91
- **[P1] Honor cancellation before keep-mode launch:** `internal/providers/tart/backend.go:409-410`
  - body: For kept leases this path uses `exec.Command` and starts the detached `tart run` process without checking `ctx`, so a canceled acquire can still launch a long-lived VM. Check the context before starting and avoid detaching anything after cancellation.
  - confidence: 0.88
- **[P2] Treat Tart State as active when Running is absent:** `internal/providers/tart/backend.go:185`
  - body: `Running` defaults to false when `tart list` JSON omits that field, but the provider already has `State` and an `instanceRunning` helper. As written, a valid `State:"running"` VM can be rejected as stopped or returned without SSH, so the checks should use the state-derived predicate or synthesize `Running` during parse.
  - confidence: 0.9
- **[P3] Align disk zero help with validation:** `internal/providers/tart/flags.go:24`
  - body: The flag help says `--tart-disk 0` means use the clone default, but `applyFlags` rejects zero. Either allow zero as the documented sentinel or change the help/docs/tests to say users must omit the flag for clone-default behavior.
  - confidence: 0.96
