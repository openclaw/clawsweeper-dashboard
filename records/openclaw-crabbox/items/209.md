## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve explicit generic SSH settings:** `internal/cli/config.go:777-785`
  - body: When provider=tart and the user sets top-level `ssh.user` or `workRoot` without tart-specific overrides, `cfg.Tart.User` and `cfg.Tart.WorkRoot` already contain defaults, so this block replaces the generic settings with `admin` and `/Users/admin/crabbox`. Preserve non-default generic SSH settings unless `tart.user` or `tart.workRoot` was explicitly configured, and add config tests for that upgrade path.
  - confidence: 0.92
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:131-166`
  - body: After `startVM` succeeds, the acquire path still stops and deletes the VM on IP polling, key injection, readiness, claim, or endpoint failures even when the lease was requested with `--keep`. That defeats the main debugging/survival reason for keep-mode local VMs; condition the cleanup on non-keep or leave a clear manual cleanup path, with regression coverage.
  - confidence: 0.9
- **[P1] Honor cancellation before detached Tart starts:** `internal/providers/tart/backend.go:409-410`
  - body: The keep path uses `exec.Command` and never checks `ctx` before `cmd.Start`, so a canceled acquire can still launch a local VM before later cleanup notices cancellation. Check `ctx.Err()` before starting the detached process, and cover the canceled keep path so Crabbox does not create local VM state after cancellation.
  - confidence: 0.88
- **[P2] Derive running state from Tart State too:** `internal/providers/tart/backend.go:185-186`
  - body: This rejects any non-status resolve when the JSON `Running` field is omitted, even if Tart reports `State:"running"`; Go leaves the bool false. Use the existing state helper as the source of truth, or combine both fields, so running VMs from older or variant Tart JSON remain usable.
  - confidence: 0.89
- **[P2] Prune stale claims when the VM is gone:** `internal/providers/tart/backend.go:538`
  - body: Resolving by a stored claim returns an error as soon as the referenced VM is missing, before the release-only stop path can remove the stale claim. `crabbox stop --provider tart <lease>` should be able to clean up a claim after the VM was deleted out of band, matching the cleanup behavior documented in the PR discussion.
  - confidence: 0.86
- **[P3] Make disk-zero help match validation:** `internal/providers/tart/flags.go:24-51`
  - body: The flag help says `--tart-disk 0` uses the clone default, but the parser rejects zero whenever the flag is set. Either accept zero as the documented clone-default sentinel or change the help/docs to say the clone default is used only when the option is omitted.
  - confidence: 0.93
