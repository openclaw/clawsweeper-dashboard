## Review Findings

Overall confidence: 0.95

Full review comments:

- **[P1] Preserve generic Tart SSH user and work root overrides:** `internal/cli/config.go:777-785`
  - body: Because `baseConfig` already gives `cfg.Tart.User` and `cfg.Tart.WorkRoot` non-empty defaults, this branch always copies those defaults over `cfg.SSHUser` and `cfg.WorkRoot`. A config with provider-neutral `ssh.user` or top-level `workRoot` but no Tart-specific override will silently fall back to `admin` and `/Users/admin/crabbox`, which breaks the shared config contract.
  - confidence: 0.95
- **[P1] Honor cancellation before starting kept Tart VMs:** `internal/providers/tart/backend.go:410`
  - body: For `keep=true` this path uses `exec.Command` and never checks `ctx` before `cmd.Start()`, so a canceled acquire can still launch a detached VM that Crabbox no longer owns. Check the context before starting the keep process and cover the pre-canceled case.
  - confidence: 0.94
- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:133-139`
  - body: After `startVM` succeeds, these error paths always stop and delete the VM even when the user requested `--keep`. That defeats the keep contract for readiness/key-injection failures and can remove the exact VM the user expected to inspect or reuse.
  - confidence: 0.93
- **[P2] Align the tart disk flag help with validation:** `internal/providers/tart/flags.go:24`
  - body: The flag help says `--tart-disk 0` uses the clone default, but the parser rejects any value `<= 0`. Either make explicit zero valid everywhere or update this help text so users know the clone default is selected by leaving the flag unset.
  - confidence: 0.97
