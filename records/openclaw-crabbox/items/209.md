## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve generic Tart SSH settings:** `internal/cli/config.go:1095-1096`
  - body: Because baseConfig pre-populates Tart.User and Tart.WorkRoot with provider defaults, a config that sets top-level ssh.user or workRoot but not tart.user/workRoot is overwritten to admin and /Users/admin/crabbox. That breaks the generic Crabbox settings users already rely on for remote account and checkout path, so Tart-specific defaults need explicitness tracking or late defaulting that preserves non-default generic values.
  - confidence: 0.93
- **[P1] Preserve kept Tart VMs after post-start failures:** `internal/providers/tart/backend.go:133-135`
  - body: After tart run has succeeded, the waitForIP, injectSSHKey, prepareLease, claim, and endpoint-error paths still stop and delete the VM even when req.Keep is true. That defeats --keep for exactly the failures users need to inspect, and it diverges from the adjacent Multipass provider's post-create cleanup behavior.
  - confidence: 0.92
- **[P1] Honor cancellation before keep-mode launch:** `internal/providers/tart/backend.go:410`
  - body: The keep path starts tart with exec.Command and never checks ctx before launching the detached process. If the acquire context has already been canceled or times out at this boundary, Crabbox can still start a kept VM that the caller believes was aborted.
  - confidence: 0.88
- **[P2] Treat running state as active without Running:** `internal/providers/tart/backend.go:185-186`
  - body: Resolve rejects non-status callers when inst.Running is false, but that bool is false when Tart omits the JSON field even if State is "running". Use the normalized State path already present in instanceRunning so existing claimed running VMs are not rejected as stopped.
  - confidence: 0.9
- **[P2] Let stop prune stale claimed Tart leases:** `internal/providers/tart/backend.go:538`
  - body: When a lease claim points at a VM that no longer exists, resolveInstance returns an error before the release path can remove the local claim. That leaves users with a stale Crabbox lease record that crabbox stop cannot clean up.
  - confidence: 0.86
- **[P2] Align --tart-disk 0 with the documented sentinel:** `internal/providers/tart/flags.go:50-51`
  - body: The flag help says 0 means use the clone default, but applyFlags rejects --tart-disk 0 as invalid. Either accept explicit zero as an unset/reset-to-clone-default value or remove the sentinel claim from the user-facing help and docs.
  - confidence: 0.91
