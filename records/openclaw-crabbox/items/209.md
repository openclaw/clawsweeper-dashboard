## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve kept VMs after post-start failures:** `internal/providers/tart/backend.go:132-166`
  - body: After `startVM` succeeds, every later provisioning failure still calls `stopVM` and `deleteVM` unconditionally. That means `crabbox run --provider tart --keep` can delete the VM exactly when the operator needs to inspect a DHCP, SSH injection, readiness, or claim failure; match the adjacent local VM behavior by skipping those post-start cleanup calls when `req.Keep` is true.
  - confidence: 0.94
- **[P1] Honor cancellation before detached Tart starts:** `internal/providers/tart/backend.go:409-434`
  - body: The keep path uses `exec.Command` instead of `exec.CommandContext` and the startup select never watches `ctx.Done()`, so a canceled acquire can still launch a detached Tart VM and return success after the observation timeout. Check `ctx.Err()` before starting and handle cancellation during the observation window so abandoned runs do not leave new VMs behind.
  - confidence: 0.91
- **[P2] Use Tart state when Running is absent:** `internal/providers/tart/backend.go:185-186`
  - body: `tart list --format json` records with `State:"running"` but no `Running` field decode with `Running=false`, so this branch rejects a usable claimed VM as stopped for non-status callers. Use the existing normalized state helper for the resolve boundary instead of the optional boolean.
  - confidence: 0.9
- **[P2] Make disk-zero help match validation:** `internal/providers/tart/flags.go:24`
  - body: The flag help says `--tart-disk 0` means use the clone default, but `applyFlags` rejects zero before the run can start and the later config validation also rejects explicit zero. Either accept zero as the documented clone-default sentinel or change the help/docs and tests so users do not get a contradictory CLI contract.
  - confidence: 0.88
- **[P1] Preserve explicit generic SSH users:** `internal/providers/tart/backend.go:49-64`
  - body: When `provider: tart` is selected without `tart.user`, `applyDefaults` fills `cfg.Tart.User` with `admin` and then overwrites `cfg.SSHUser`. A config that sets top-level `ssh.user` or `CRABBOX_SSH_USER` will still run as `admin` unless the user duplicates that value under `tart:`, so the provider-specific default should yield to an explicit generic SSH user.
  - confidence: 0.84
