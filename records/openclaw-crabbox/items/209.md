## Best Possible Solution

Teach Tart release to prune a claim when resolveInstance already proved the VM is missing, while still preserving delete-error protection for VMs that may still exist; keep Linux/Ubuntu Tart support as a separate follow-up scope decision.

## Reproduction Assessment

Yes for the blocking bug by source inspection: keep a Tart lease claim, remove the VM outside Crabbox so it is absent from tart list, then stop by lease ID; resolveInstance returns State=missing and ReleaseLease still fails at deleteVM before pruning state.

## Solution Assessment

No; the provider direction is reasonable, but the current release implementation is not the safest way to handle an already-missing VM because it treats the expected stale-prune condition like a real delete failure.

- **[P2] Prune missing Tart claims before deleting:** `internal/providers/tart/backend.go:279-280`
  - body: When a lease claim resolves but the VM is absent from `tart list`, `resolveInstance` returns `State: "missing"`. Treat the already-missing instance as a successful provider-side deletion and prune local state, while keeping delete errors fatal for VMs that still appear to exist.
