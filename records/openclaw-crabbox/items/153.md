## Label Justifications

- `P1`: The PR addresses an AWS SSH connectivity regression where one active lease can remove another active lease's ingress during normal launch or heartbeat workflow.
- `merge-risk: 🚨 compatibility`: The new persisted awsSSHCIDRs field has no migration path for existing active lease records, so rollout behavior differs for old and new leases.
- `merge-risk: 🚨 availability`: If the rollout gap is hit, existing AWS leases can lose SSH reachability until another refresh or manual recovery restores ingress.

## Summary

Keep open: the patch targets a real AWS SSH ingress churn bug, but it still needs real behavior proof and an upgrade-safe path for active AWS leases that predate the new stored CIDR field.

## What This Changes

The branch records AWS SSH CIDRs on lease records and uses active AWS lease CIDRs when creating or refreshing shared security-group ingress, with focused tests and docs updates.

## Best Possible Solution

Merge a version that preserves active AWS SSH ingress for both new records and rollout-era records without awsSSHCIDRs, backed by a regression test and redacted real AWS proof.

## Reproduction Assessment

Yes from source inspection: current main sends only the current request CIDR to AWS ingress refresh, and the AWS provider prunes managed SSH rules outside that desired set. I did not run live AWS reproduction in this read-only review.

## Solution Assessment

No, not yet: storing per-lease CIDRs is the right shape for new leases, but the patch needs upgrade-safe behavior for active records without the new field and real AWS proof before merge.
