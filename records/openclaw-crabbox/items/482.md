## Summary

Keep this open: current main still has the false-success paths described, and no merged or open related item appears to own the fix. Source inspection shows the paste helper can swallow clipboard-helper failure, and proof/terminal launch can proceed after a detached POSIX launch without a liveness or visibility gate.

## What This Changes

Request that desktop paste/type and POSIX desktop launch/terminal/proof fail instead of reporting success when clipboard delivery or detached GUI/terminal launch was not actually verified.

## Best Possible Solution

Make helper success mean the requested clipboard delivery or visible launch was verified, with focused regression coverage and docs adjusted only if the user-facing contract changes.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: the generated paste script ignores clipboard-helper failure, and the proof flow continues after detached launch success. I did not execute a live Linux desktop lease during this read-only review.

## Solution Assessment

Yes, the narrow fix is to propagate clipboard-helper failure and add a targeted terminal/proof launch liveness or visibility gate before success output and proof artifacts. A broader generic launch check should reuse the existing browser-check pattern without adding new provider or config surface.
