## Label Justifications

- `P2`: This is a medium-scope provider security and integrity issue bounded to Hetzner credentials, admin pool visibility, and known-lease recovery.
- `impact:security`: The issue is about trusting forgeable provider labels across a cloud-provider ownership boundary.
- `impact:session-state`: A forged recovery match can persist the wrong cloud ID and host into coordinator lease/workspace state.

## Summary

Keep open: current main still lets the Hetzner Worker path discover servers by forgeable provider labels and then persist a recovered match into coordinator lease state, and no canonical fix or superseding item exists.

## What This Changes

Harden Hetzner Worker server discovery and workspace recovery so provider labels alone are not treated as Crabbox ownership proof.

## Best Possible Solution

Add a Hetzner adapter-owned, coordinator-state-backed ownership proof and regression tests that reject unverified label matches in pool listing and workspace recovery.

## Reproduction Assessment

Yes. A source-level reproduction can mock Hetzner `/servers` to return a forged `crabbox=true,lease=...` server and drive workspace recovery; the current code then adopts that server without a live Hetzner run.

## Solution Assessment

Yes in direction, but the exact ownership-proof and migration design needs maintainer review. The maintainable fix should stay provider-specific and avoid adding Hetzner special cases to core routing.
