## Label Justifications

- `P1`: The issue describes a coordinator authorization-boundary failure that can disrupt another user's live WebVNC bridge.
- `impact:security`: The affected behavior is a share-role authorization bypass between use and manage access.

## Summary

Keep open: current main and the latest release still let WebVNC reset resolve a lease through visibility without the manage-access check used by adjacent bridge-ticket routes, so the reported share-role authorization boundary remains unfixed.

## What This Changes

The issue reports that a use-role share recipient can reset another user's active WebVNC bridge.

## Best Possible Solution

Add a manage-access authorization check to WebVNC reset before any bridge/viewer state mutation, with regression coverage for direct use shares, org use shares, owners, admins, and manage shares.

## Reproduction Assessment

Yes for source-level reproduction: a use-share request can pass resolveLease through lease visibility and reach WebVNC reset state changes without a manage check. I did not run a live Worker test in this read-only review.

## Solution Assessment

Yes, the narrow fix is to apply the same leaseManageableByRequest gate used by adjacent WebVNC ticket creation before reset mutates bridge state.
