## Label Justifications

- `P2`: This is a valid medium-severity security/auditability bug with limited blast radius to shared-lease run history rather than an immediate execution or availability emergency.
- `impact:security`: The issue concerns coordinator authorization and owner-side audit visibility for commands, logs, events, telemetry, and completion records on shared leases.

## Summary

Keep open. Current main still allows a `use`-shared caller to create a run on a visible lease, stores that run under the caller's owner/org, and then filters API and portal run reads by the stored run owner/org, so the lease owner auditability gap remains.

## What This Changes

The issue requests fixing shared-lease run attribution or authorization so lease owners can see and audit runs created on their lease by share recipients.

## Best Possible Solution

Define the lease-backed run owner/actor model, then make run list/detail/log/event/telemetry and portal authorization expose owner-visible audit records while keeping strangers blocked and preserving share-role semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Alice owns a lease shared to Bob as `use`, Bob posts `/v1/runs` with Alice's `leaseID`, and Alice's owner-scoped run list/detail/log/portal reads omit or reject the caller-owned run.

## Solution Assessment

Unclear until maintainers choose the durable attribution model. The maintainable fix should either attribute lease-backed runs to the lease owner while storing the actor separately, or make run authorization lease-aware with explicit tests for owner, share recipient, revocation, and stranger access.
