## Summary

Keep open: current main still defaults human-facing WebVNC output to unredacted, and source, tests, and docs confirm credential-bearing URLs and `password:` lines. The closed related reports are useful context, but they do not conclusively resolve this separate stdout/transcript disclosure path.

## What This Changes

The issue asks Crabbox to stop printing reusable WebVNC credentials and credential-bearing viewer URLs by default in foreground, status, reset, and direct-SSH output.

## Best Possible Solution

Maintainers should decide the security boundary for stdout/transcript exposure; if accepted, make human-facing WebVNC output redact credentials by default while preserving a deliberate opt-in reveal or compatible one-click handoff, with focused CLI/docs tests.

## Reproduction Assessment

Yes. Source inspection on current main shows default redaction is off and the foreground, status, reset, and direct-SSH WebVNC paths still print credential-bearing URLs or raw password lines; I did not run a live WebVNC session because the source and tests already establish the data flow.

## Solution Assessment

Unclear. Redacting default stdout is a plausible narrow fix, but maintainers need to confirm the compatibility and security-boundary decision because related URL-fragment behavior was intentionally preserved.
