- `impact:security`: The report concerns integrity verification for external artifacts that become privileged code or trusted rootfs contents inside Crabbox-managed Windows leases.

## Summary

Keep open: current main and v0.32.0 still download and install/import Windows bootstrap artifacts without pinned SHA-256 verification in both the Go CLI and Worker generators, while the repository security policy treats default artifact-integrity failures as in scope.

## What This Changes

The issue asks Crabbox to verify TightVNC, Git for Windows, OpenSSH-Win64, and Ubuntu WSL rootfs downloads with pinned SHA-256 checks before Windows bootstrap installs or imports them.

## Best Possible Solution

Add pinned artifact identity next to each Windows bootstrap URL, verify it in generated PowerShell before extraction/import/execution in both generators, replace the moving WSL rootfs with a versioned pinned artifact, and cover normal desktop, core, and WSL2 paths in Go and Worker tests.

## Reproduction Assessment

Yes, at source level. Current main shows each Windows artifact download flowing directly into extraction, installation, import, or execution without a pinned hash check; I did not run a live substitution exploit in this read-only review.

## Solution Assessment

Yes for the remediation direction: mirroring the existing MinGit and Tailscale pinned-verification patterns is the narrow maintainable fix. Maintainers still need to choose the durable digest update policy, especially for the current WSL rootfs URL.
