## Summary

Keep this PR open: the CodeSandbox lifecycle fixes are directionally useful, but the new mount replacement rollback can delete existing workspace entries if backing up a child fails, and the PR still needs real redacted live-output proof rather than only a verification summary.

## What This Changes

The PR hardens the CodeSandbox provider lifecycle around ownership tags, resume validation, SDK bridge shutdown, archive replacement, preview URLs, smoke tests, and related docs/tests.

## Best Possible Solution

Keep the provider-specific replacement hook, but make the mount replacement abort without deleting entries that were not successfully backed up, remove the release-owned CHANGELOG edit, and add redacted live CodeSandbox output before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: make one existing workdir child fail the `mv` into the backup directory and the rollback loop then removes non-backup workdir entries before restoring only backed-up entries.

## Solution Assessment

No. The provider-specific hook is the right boundary, but this rollback sequence is not safe until backup-stage failures preserve unmoved existing content and the PR includes inspectable real-service proof.
