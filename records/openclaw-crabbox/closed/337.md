## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use Lambda's launch request schema:** `internal/providers/lambda/types.go:66-71`
  - body: Lambda's launch schema accepts `image: {id|family}`, `firewall_rulesets: [{id}]`, and filesystem mounts with `file_system_id`/`mount_point`, but this struct emits top-level `image_id`, `image_family`, `firewall_ruleset_name`, and `name`/`mount_path`. The provider sets an image family by default and exposes firewall/filesystem config, so real launches can fail or ignore user-specified resources.
  - confidence: 0.95
- **[P1] Decode Lambda regions as objects:** `internal/providers/lambda/types.go:21-28`
  - body: The live API returns `regions_with_capacity_available` and image `region` values as Region objects, not strings. Because `doctor --provider lambda` lists instance types and images for the default image family, these fields can make doctor fail before it validates a real account.
  - confidence: 0.92
- **[P2] Decode Lambda tags as entries:** `internal/providers/lambda/types.go:40`
  - body: Instance responses expose `tags` as an array of key/value entries, but this model decodes them as `map[string]string`. Any tagged instance in inventory can make list/doctor decoding fail and prevents the promised compatibility path for provider-tagged Lambda instances.
  - confidence: 0.89
