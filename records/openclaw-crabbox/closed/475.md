## Summary

Keep open: current main still uses a symlink-accepting required-artifact predicate, the latest release has the same `run_artifacts.go` implementation, and the canonical search found no merged fix or duplicate report owning the dangling-symlink case.

## What This Changes

The issue reports that `crabbox run --require-artifact` can be satisfied by a dangling symlink instead of a real proof artifact.

## Best Possible Solution

Harden the generated required-artifact and delegated artifact scripts so dangling symlinks never satisfy the proof gate, then add focused regression tests for regular files, valid symlinks if retained, dangling symlinks, symlinked directories, and recursive glob paths.

## Reproduction Assessment

Yes at source level: current main still generates `[ -f "$f" ] || [ -L "$f" ]` in the required-artifact predicate, so a dangling symlink can be counted as a required artifact. The nearby tests cover present and missing files but not dangling symlinks.

## Solution Assessment

Yes: the narrow maintainable fix is to reject dangling symlinks in the generated required-artifact and delegated artifact scripts while preserving the existing safe-relative path and protected-directory checks.
