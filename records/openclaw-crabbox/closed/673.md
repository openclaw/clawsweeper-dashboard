## Summary

Keep open: the patch is useful safety hardening, but the new fail-closed delete gate can break deletion of pre-existing Azure/GCP images created before ownership metadata existed, so maintainers should decide or patch the upgrade path before merge.

## What This Changes

The PR makes worker image deletion require stored Crabbox-created ownership metadata before deleting AWS, Azure, or GCP images or snapshots, with related worker tests, command docs, feature docs, and a changelog entry.

## Best Possible Solution

Keep the stored-ownership delete requirement, but either add a safe ownership import/backfill path for pre-existing Crabbox-created provider images or explicitly document provider-console cleanup as the accepted upgrade contract.

## Reproduction Assessment

Yes from source inspection: current main lets admin DELETE reach Azure/GCP provider-native deletion because those providers return no stored metadata and allow deletion validation. I did not run tests because this review is required to keep the checkout read-only.

## Solution Assessment

Mostly yes: requiring stored ownership metadata is a narrow safety fix, but the PR needs an explicit maintainer choice for old images created before the new metadata existed.
