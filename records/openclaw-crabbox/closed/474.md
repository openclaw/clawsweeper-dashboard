- `P1`: A documented narrowing control can fail in a way that forwards credential-bearing environment variables to a run destination.
- `impact:security`: The report concerns trusted credential/environment forwarding across a documented Crabbox security boundary.

## Summary

Keep open: current main still applies selected profile env allow entries after CRABBOX_ENV_ALLOW, so the documented narrowing control remains bypassable by source inspection. No merged or open canonical fix PR was found for this precedence bug.

## What This Changes

The issue reports that CRABBOX_ENV_ALLOW replaces global env allowlists but is later widened again by selected profile env allow entries.

## Best Possible Solution

Resolve profile env allow entries below CRABBOX_ENV_ALLOW, keep --allow-env as the final explicit append layer, and add regression coverage for both env.allow and legacy envAllow profile shapes.

## Reproduction Assessment

Yes: source inspection gives a high-confidence reproduction path because CRABBOX_ENV_ALLOW is applied in applyEnv, selected profile allowlists are appended afterward, and selectRunEnv receives the widened list. I did not execute a live run, but the dataflow is direct in current main.

## Solution Assessment

Yes: the reporter's proposed direction is the narrow maintainable fix because it preserves the documented precedence and leaves --allow-env as the explicit post-override escape hatch.
