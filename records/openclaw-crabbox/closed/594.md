## Summary

Keep open: this maintainer-authored PR is a useful bounded follow-up to the timing-status work, but the new helper can misclassify delegated timing failures when the effective exit code was inferred outside the RunResult.

## What This Changes

The branch adds a shared timing-report helper, routes delegated run timing records through it, and updates Blacksmith artifact-failure timing JSON coverage.

## Best Possible Solution

Keep the shared helper, but finalize outcome fields against the effective report exit code and cover the ExitError-with-empty-RunResult case.

## Reproduction Assessment

Yes for the patch defect from source inspection: an ExitError can populate report.ExitCode while TimingReportWithRunResult finalizes status from the unchanged RunResult. I did not run tests in this read-only review.

## Solution Assessment

No: finalizing from the original RunResult before reconciling the inferred exit code is not the safest implementation. The helper should use the effective exit code or explicitly preserve only provider-set outcome fields.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Finalize timing from the effective exit code:** `internal/cli/timing.go:73-82`
  - body: When `timingReportFromDelegatedRunResult` receives `RunResult{ExitCode: 0}` with an `ExitError`, it copies the inferred code only into `report.ExitCode`. This helper then calls `FinalizeRunResult(result, err)` using the original zero-valued result, so `errorKind` becomes `provider-error` even though the report carries a nonzero command exit; current timing JSON derives `command-exit` from that effective exit code. Copy the report exit code into the temporary result before finalizing, or handle `ExitError` explicitly.
  - confidence: 0.87
