## Summary

Keep open: current main still lacks a `docker-sandbox` provider or sbx-specific provider package, and the existing Docker surface is the separate `local-container` SSH-lease provider. The request is plausible but needs maintainer product/API choices before implementation.

## What This Changes

The issue requests a built-in Docker Sandboxes provider backed by Docker's standalone `sbx` CLI, separate from the existing `local-container` Docker-compatible provider.

## Best Possible Solution

Define the v1 sbx lifecycle and config contract first, then add a provider-specific adapter package that preserves `local-container` as the Docker-compatible container path and keeps Docker Sandboxes behind a separate delegated-run provider.
