## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce hard overrides as non-code jobs:** `src/repair/issue-implementation-intake.ts:330`
  - body: For hard blockers, this returns `shouldRepair: true`, so `repair-issue-implementation-intake.yml` dispatches the repair worker. The job rendered below still has `allowed_actions: fix/raise_pr`, `allow_fix_pr: true`, and `security_sensitive: false`; the non-code boundary is only prompt text. A security/protected/locked/PR-attached issue can therefore be overridden into a code-capable implementation lane, which violates the hard-blocker contract.
  - confidence: 0.92
- **[P2] Update existing jobs when override is requested:** `src/repair/comment-router.ts:1846-1854`
  - body: The override metadata is only supplied while rendering a brand-new job. The expected use case is continuing after a refused build, but that issue commonly already has a durable issue implementation job; the early return for `command.target?.job_path` reuses the old job without `operator_override`, blocker class, or override action, so the worker sees the same non-override prompt again.
  - confidence: 0.86
- **[P2] Classify PR-reference blockers as hard:** `src/repair/comment-router-core.ts:1603`
  - body: The hard-blocker regex misses blocker strings emitted by intake such as `open PR already mentions this issue` and `work cluster references a PR`. Those are existing linked-PR cases, which the docs added here define as hard blockers, but the current classifier will treat at least some of them as soft and allow a bounded PR attempt.
  - confidence: 0.84
