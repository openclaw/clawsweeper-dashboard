- `impact:other`: The stale mutually exclusive workflow label sends incorrect ownership and next-action signals on reviewed pull requests.

## Summary

Current main still treats every `securityReview.status === "needs_attention"` result as contributor work, so a verdict whose only remaining action is maintainer privacy/security approval selects `waiting_on_author`; the linked CodexBar case demonstrates the resulting stale routing label.

## What This Changes

Make PR status labels mutually exclusive and replace `status: ⏳ waiting on author` with `status: 👀 ready for maintainer look` when the latest durable verdict leaves only maintainer action.

## Best Possible Solution

Derive contributor work from the structured verdict’s actual assignee and actionable findings, then atomically synchronize one status label from that latest durable result, with regression coverage for a security-sensitive maintainer-only approval case.

## Reproduction Assessment

Yes. On current main, a PR report with sufficient proof, no blocking finding, a correct patch, and `securityReview.status: needs_attention` deterministically selects `waiting_on_author`, matching the linked live case where only maintainer approval remained.

## Solution Assessment

Yes, with one refinement: preserve the existing single-label synchronization and fix the structured contributor-versus-maintainer work classification rather than adding a special-case label cleanup pass.
