## Summary

Keep open: the PR improves the previous unsupported-field approach, but it still turns hint-only `likely_files` into hard checkpoint contracts and leaves runtime validation looser than the schema, so it can reject valid repair automation or accept invalid artifacts.

## What This Changes

The PR adds a canonical `fix_artifact.repair_contract` field, validates it, emits it from deterministic repair producers, and enforces it before repair checkpoint commits.

## Best Possible Solution

Keep the canonical repair contract schema and checkpoint enforcement, but make runtime validation match the schema and avoid emitting hard contracts from hint-only `likely_files`.

## Reproduction Assessment

Yes from source: deterministic producers create `repair_contract` from `likelyFiles`, and the executor enforces any present contract before checkpoint commits. Runtime/schema divergence is also source-reproducible because missing `match`/`scope` are not rejected but later default to `any` and `every_checkpoint`.

## Solution Assessment

No; the canonical contract approach is plausible, but this PR still needs a narrower activation policy and schema-equivalent runtime validation before merge.
