## Summary

Keep this PR open: the live proof supports the GitHub Search path, but the gitcrawl enrichment is not merge-ready because the new SQL selects `threads.body_excerpt` for any `cluster_groups` database while the current gitcrawl default schema uses `threads.body`; maintainers also need to accept the default-on exact-event Search rollout risk.

## What This Changes

The PR expands related item context for ClawSweeper reviews with gitcrawl cluster siblings, exact-event GitHub issue search, prompt/docs updates, workflow env wiring, and focused tests.

## Best Possible Solution

Land the related-context enrichment after the gitcrawl query is schema-aware for current and pruned portable gitcrawl databases, and after maintainers choose whether exact-event Search stays default-on or starts opt-in.

## Reproduction Assessment

Yes for the blocking patch defect: a minimal current gitcrawl-style SQLite schema with `cluster_groups` and `threads.body` reproduces the `no such column: t.body_excerpt` failure. This is a source-reproducible PR defect rather than a current-main end-user bug.

## Solution Assessment

No: the feature direction is reasonable, but the implementation should select the available gitcrawl body column and include fixture coverage before merge. The default-on exact-event Search behavior also needs an explicit maintainer rollout choice.
