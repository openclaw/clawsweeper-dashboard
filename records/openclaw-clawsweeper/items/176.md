## What This Changes

## Best Possible Solution

Land the related-context enrichment after the gitcrawl SQL is schema-aware with fixture coverage, and after maintainers choose whether exact-event Search should remain default-on or start opt-in.

## Reproduction Assessment

Yes for the blocking patch defect: source inspection shows the PR selecting `t.body_excerpt`, while the current gitcrawl schema defines `threads.body`. This is a source-reproducible feature-path defect rather than an end-user bug reproduction.

## Solution Assessment

No, not yet: the related-context design is reasonable, but the gitcrawl SQL should support the actual current schema before merge. The Search default can still be acceptable if maintainers intentionally own the quota and match-quality tradeoff.
