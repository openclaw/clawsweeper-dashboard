Land the related-context enrichment after the gitcrawl lookup is schema-aware with SQLite fixture coverage, and after maintainers choose the default-on or opt-in Search rollout.

## Reproduction Assessment

Yes for the blocking patch defect: source inspection shows the PR query selecting `t.body_excerpt` while the current default gitcrawl schema exposes `threads.body`. This is not an end-user bug report; it is a feature PR with a source-reproducible schema mismatch.

## Solution Assessment

No: the direction is useful, but the gitcrawl query should use a schema-aware body expression like gitcrawl itself does and should have fixture coverage for full/default and portable schemas before rollout.
