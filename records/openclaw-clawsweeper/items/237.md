## Summary

Keep this PR open: the protected-branch auto-merge direction is useful, but the patch is not merge-ready because it updates only one automerge readiness path and changes every approved merge to depend on GitHub auto-merge.

## What This Changes

The PR allows `BLOCKED` merge-state PRs through the comment-router automerge gate, adds `--auto` to the final `gh pr merge` command, fetches `autoMergeRequest`, and reports still-open PRs as waiting.

## Best Possible Solution

Land a narrower fix that shares the protected-branch merge-state allowance across router and shepherd readiness, uses GitHub auto-merge only when needed or with a direct-merge fallback, and includes redacted live proof of the BLOCKED auto-merge request path.

- **[P1] Update the shepherd readiness gate:** `src/repair/comment-router.ts:2502-2503`
  - body: This only lets `BLOCKED` through the comment-router readiness check.   - confidence: 0.9
- **[P1] Preserve the direct merge path:** `src/repair/comment-router-core.ts:717`
  - body: Putting `--auto` in the shared merge-args builder changes every approved automerge, including already-clean PRs that currently squash-merge directly after ClawSweeper's gates. Repositories without GitHub auto-merge enabled, or clean PRs that should merge immediately, can now fail or be left waiting. Limit `--auto` to the protected-branch case or fall back to the existing direct squash merge when auto-merge setup is not needed or unavailable.
  - confidence: 0.78
