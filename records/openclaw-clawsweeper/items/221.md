## Summary

This PR is useful but not merge-ready: the new coalescer only deduplicates fast acks inside one running process or worker isolate, while the linked production race can still happen when duplicate deliveries run in separate handlers. It also still lacks redacted after-fix real behavior proof.

## What This Changes

The PR adds in-memory fast-ack coalescing in the repair webhook and hosted worker, routes trusted ClawSweeper markers before copyable commands, carries status metadata for autofix/automerge dispatches, and adds same-process duplicate webhook regression tests.

## Best Possible Solution

Use a shared ack identity or deterministic cleanup path that works across processes and worker isolates, keep the marker-routing/status-metadata changes if needed, and require redacted after-fix proof showing duplicate deliveries converge to one visible ack/status comment.

## Reproduction Assessment

Yes for the source-level race: current main still uses a non-atomic list/post/list prune flow, and https://github.com/openclaw/clawsweeper/issues/220 gives live duplicate ack comment examples. I did not run a live multi-isolate reproduction during this read-only review.

## Solution Assessment

No: the PR is a useful partial mitigation, but an in-memory map is not the narrowest complete fix for a duplicate-delivery race that can cross process or worker-isolate boundaries.
