## Summary

Keep open: current main still has a reproduced exact-review capacity failure where the run is recognized, waits outside the four configured slots, times out before Codex review starts, and only reports retry-after-capacity rather than durably requeueing or backfilling the item.

## What This Changes

The issue reports that exact issue/PR reviews can fail while waiting for the exact-review capacity semaphore instead of being durably retried or visibly requeued.

## Best Possible Solution

Add a bounded, observable recovery path for exact-review capacity overflow so a timed-out waiter is requeued or backfilled exactly once and status clearly separates waiting, reviewing, retry scheduled, and permanent failure.

## Reproduction Assessment

Yes. The named workflow run on current main waited in `repair:codex-capacity` for about twenty minutes, reported ranks outside the four exact-review slots, and failed with `timed out waiting for Codex capacity` before the review step ran.

## Solution Assessment

Yes on the direction, but the exact mechanism should stay narrow: recover capacity timeouts through durable retry/backfill or scheduler backpressure without loosening the semaphore into unbounded concurrent Codex runs.
