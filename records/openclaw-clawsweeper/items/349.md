## Summary

Current main implements the central fix through https://github.com/openclaw/clawsweeper/pull/354: production now lets one target use all four exact-review slots, and the queue/test/docs paths cover that behavior. The broader worker-budget tuning in https://github.com/openclaw/clawsweeper/pull/350 is adjacent and not required to resolve the per-target cap starvation reported here.

## What This Changes

The issue reports that a per-target exact-review concurrency cap of 1 serialized openclaw/openclaw review dispatches and delayed fresh pull request reviews behind older re-review work.

## Best Possible Solution

Keep the merged current-main queue-capacity implementation from https://github.com/openclaw/clawsweeper/pull/354 and treat https://github.com/openclaw/clawsweeper/pull/350 as optional broader throughput tuning.

## Reproduction Assessment

Yes from source: with the prior production target cap of 1, same-target pending reviews were skipped while a single openclaw/openclaw lease was active. I did not run the provided repro because this is a read-only review, but current main now has focused regression coverage for the fixed path.

## Solution Assessment

Yes: making the per-target exact-review cap configurable and setting production to the existing global cap is the narrowest maintainable fix for the reported idle-slot starvation without increasing global API pressure.
