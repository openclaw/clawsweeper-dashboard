## Summary

Keep open: current main still enforces the per-target exact-review cap of 1 with oldest-first admission and long leases, and no newer issue or PR appears to own the fresh-PR starvation fix. The cap was an intentional load throttle, so the remaining work needs a maintainer policy choice rather than cleanup closure.

## What This Changes

Issue requests preventing fresh `openclaw/openclaw` pull requests from starving behind older exact-review work after the per-target exact-review cap added by https://github.com/openclaw/clawsweeper/pull/348.

## Best Possible Solution

Keep exact-review load bounded but guarantee timely first review for newly opened pull requests, either through a fresh-PR reservation/exemption or a target-aware cap that is explicit and tested.

## Reproduction Assessment

Yes from source inspection: current main still counts active exact-review leases per target and skips pending items once that target has one active lease, with pending work ordered oldest first. I did not run the supplied repro because this review is read-only and the test is not already present in the checkout.

## Solution Assessment

Unclear as a final policy choice: the report identifies the right starvation point, but maintainers should choose the load-safe shape, such as a fresh-PR reservation or target-specific cap, rather than only reverting the throttle.
