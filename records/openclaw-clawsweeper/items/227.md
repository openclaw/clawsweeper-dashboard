## Label Justifications

- `P1`: The issue shows a user-facing ClawSweeper command workflow accepting commands while silently failing to run or surface progress.
- `impact:other`: The concrete impact is broken maintainer automation observability: acknowledged rerun commands do not produce durable review or dashboard state.

## Summary

Keep open: current main still has a source-level trigger mismatch where the webhook posts a fast ack and dispatches `clawsweeper_comment`, but the sweep workflow only listens for `clawsweeper_item` and `clawsweeper_target_sweep`, so an acknowledged comment command can fail to start the review workflow or produce dashboard-visible progress.

## What This Changes

The issue reports that issue-author `@clawsweeper review`/`re-review` commands are acknowledged as queued but do not produce a fresh durable review or visible dashboard rerun.

## Best Possible Solution

Make the fast-ack comment path and sweep workflow share one supported event path, then ensure every acknowledged command reaches an in-progress, complete, superseded, or failed status that is visible from the issue and dashboard.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `src/repair/comment-webhook.ts` dispatches `clawsweeper_comment`, while `.github/workflows/sweep.yml` does not subscribe to that repository_dispatch type. The linked OpenClaw comments also show acked commands without a later durable review update.

## Solution Assessment

Yes: the narrow maintainable fix is to align the dispatch event type and workflow trigger, then add regression coverage that a fast-acked issue-author re-review command can start the exact-item workflow and update command status. Unsupported freeform author commands should fail visibly rather than being silently acknowledged.
