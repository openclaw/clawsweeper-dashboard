## Best Possible Solution

Full review comments:

- **[P2] Preserve repair-pr jobs during state publishes:** `src/repair/pr-repair-intake.ts:75-76`
  - body: The intake writes job files named `repair-pr-...` under `jobs/<owner>/inbox`, but current state publishing only preserves state-only inbox jobs whose names start `automerge-`, `issue-`, or `self-heal-`. After a normal `--path jobs` publish from a checkout that does not contain the new file, these queued PR-repair jobs can be dropped before dispatch. Please either use a preserved prefix or extend the publish allowlist and tests for this job type.
  - confidence: 0.87
- **[P2] Do not enqueue repairs for transient UNKNOWN merge state:** `src/repair/pr-repair-intake.ts:167-168`
  - body: `UNKNOWN` is not an objective dirty/blocked PR state; GitHub can return it while mergeability is still being computed, including for this freshly opened PR with only a successful notify check. With this condition, intake can generate repair jobs for otherwise clean or pending PRs. Please restrict this signal to states that prove a repair is needed, or re-query/wait before treating unknown as actionable.
  - confidence: 0.82
