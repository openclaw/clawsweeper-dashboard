## Summary

Keep this open: current main still routes `securityReview.status === "needs_attention"` through `status: ⏳ waiting on author`, even though that label says the blocker is contributor-facing and the reported live PR is proof-sufficient with only security-owner acceptance left.

## What This Changes

The issue asks ClawSweeper to stop labeling PRs as waiting on the author when the only remaining blocker is human sandbox/security owner review.

## Best Possible Solution

Represent human security/sandbox-owner review separately from author work, then route proof-sufficient, patch-correct PRs with only owner acceptance to a maintainer/security-review status instead of `waiting_on_author`.

## Reproduction Assessment

Yes. Source inspection on current main shows `securityReview.needs_attention` becomes unresolved work and is labeled `waiting_on_author`; the live referenced PR also still carries that label while proof is sufficient.

## Solution Assessment

Unclear until maintainers choose the status semantics. The safest direction is to distinguish owner/security acceptance from author-remediable security defects instead of treating every security-review concern as either author work or maintainer-ready.
