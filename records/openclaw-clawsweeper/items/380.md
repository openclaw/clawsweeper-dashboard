## Summary

Keep open: the PR targets a real duplicate-dispatch gap in the comment router, but as submitted it can permanently suppress a maintainer command after a pre-dispatch claim failure and still lacks real behavior proof from a live or controlled router run.

## What This Changes

The PR adds a `claimed` command-ledger status for dispatch-bearing comment-router commands, writes that claim before acknowledgements and execution, and adds focused ledger/order tests.

## Best Possible Solution

Land this only after maintainers either accept explicit at-most-once command semantics or the PR adds stale-claim recovery that prevents duplicate dispatches without silently dropping failed claims.

## Reproduction Assessment

Yes, source inspection gives a high-confidence current-main path: dispatch-bearing commands execute before `appendLedger` and `writeLedger`, so a failure after dispatch but before the ledger write can replay the command version. I did not run a live dispatch because that would mutate GitHub.

## Solution Assessment

No, not as submitted. The implementation is narrow, but making `claimed` terminal before dispatch succeeds introduces a command-loss window unless maintainers accept that tradeoff or the PR adds stale-claim recovery.
