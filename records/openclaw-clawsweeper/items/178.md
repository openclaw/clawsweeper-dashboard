## Summary

This is a valid feature proposal, but it is not already implemented as a dedicated freshness recheck lane and the ClawSweeper repository profile keeps issues open for maintainer triage. Current main has adjacent machinery for scheduled/exact reviews, re-review commands, and guarded apply of existing close proposals, so the right next step is a maintainer product decision on the lane shape rather than auto-closing or queueing a repair PR.

## What This Changes

The issue proposes a manual-first freshness recheck lane that selects stale open issues, verifies release and current-main status, and reuses the existing implemented-on-main close machinery when a fix is already present.

## Best Possible Solution

Add a report/manual-dispatch freshness lane only after maintainers choose candidate thresholds and rollout defaults, keeping review read-only and routing any close through the existing apply gate.

Not applicable: this is a feature proposal for a new review lane, not a report of broken existing behavior. Source inspection confirms adjacent mechanisms exist, but there is no failing current-main behavior to reproduce.

## Solution Assessment
