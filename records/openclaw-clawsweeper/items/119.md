Keep open: current main still lacks an explicit merge-result versus behind-branch drift guard for PR review, and the reported false merge-risk review is concrete enough to treat as a valid automation bug.

## What This Changes

The issue asks ClawSweeper to stop treating current-main-only changes on a clean, behind PR branch as if merging the PR would delete them.

## Best Possible Solution

Add deterministic PR review context and prompt/test coverage so clean behind branches are treated as stale or needing rebase, while merge-risk labels are reserved for real merge-result regressions.

## Repair Work Prompt

Implement a narrow fix for https://github.com/openclaw/clawsweeper/issues/119. In ClawSweeper PR reviews, distinguish true merge-result regressions from normal behind-main drift: a clean mergeable behind branch must not receive `patch is incorrect` or merge-risk labels solely because raw current-main-vs-head inspection appears to delete main-only files. Prefer adding explicit PR merge-state context such as REST `mergeable_state` or GraphQL `mergeStateStatus`, and update `prompts/review-item.md` so reviewers use merge-tree/GitHub merge-result/equivalent three-way evidence before writing deletion-style findings. Add focused regression coverage in `test/clawsweeper.test.ts` for the prompt/context behavior. Do not change apply/close behavior, live GitHub mutation, or unrelated label semantics.
