## Summary

Keep open. Current main has model-level config-option metadata, but the requested deterministic guard is still missing: there is no stored `config_surface_change` flag and the automerge pass path does not check config-surface additions before emitting a pass verdict.

## What This Changes

Request ClawSweeper to require maintainer review when OpenClaw PRs add public config values, schema fields, CLI/env knobs, or plugin manifest contracts.

## Best Possible Solution

Add a conservative config-surface detector over hydrated PR files/patches, persist the detected flag and keys in reports, and force needs-human before automerge/pass markers while covering the requested positive and negative cases.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection does show the requested deterministic config-surface guard is absent from the current automerge pass path.

## Solution Assessment

Yes. Combining prompt guidance with a deterministic report/marker guard is the narrowest maintainable way to avoid relying only on model classification for public config-surface additions.

## Repair Work Prompt

Implement the config-surface human-review guard requested in https://github.com/openclaw/clawsweeper/issues/171. Update `prompts/review-item.md` so PRs adding public OpenClaw config values, CLI flags, env vars, plugin manifest contracts, schema fields, or policy knobs are classified with `requiresNewConfigOption: true` and routed to maintainer review rather than pass/repair automation. Add deterministic detection in `src/clawsweeper.ts` using hydrated PR file paths and patches for the listed OpenClaw config/manifest/docs surfaces; persist `config_surface_change: true` and, where practical, `config_surface_keys: [...]` in report frontmatter. Make `reviewAutomationMarkersFromReport(...)`/`isRepairLoopPassReport(...)` force `clawsweeper-verdict:needs-human` when the flag is true, while preserving existing needs-changes findings behavior for non-config PRs. Add focused tests for root/user config schema additions, plugin manifest contract additions such as `contracts.embeddingProviders`, and a negative non-semantic docs wording case. Do not broaden this into a general OpenClaw schema parser or change unrelated automerge/comment-router behavior unless needed to reuse the existing needs-human pause path.
