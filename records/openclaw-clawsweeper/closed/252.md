## Label Justifications

- `P1`: A valid re-review command can fail before checkout for target repositories whose default branch is not `main`, blocking real ClawSweeper review automation.
- `impact:other`: The issue affects ClawSweeper command routing and targeted review workflow execution, which is maintainer-visible automation behavior outside the more specific impact labels.

## Summary

This issue is still necessary: current `main` preserves the target default branch in the webhook intake, but the comment-router re-review path drops it before dispatching the actual item review, so the sweep workflow falls back to `main` and can fail before checkout on `master`-default repositories.

## What This Changes

The issue asks ClawSweeper re-review dispatches to use the target repository's actual default branch instead of assuming `main`.

## Best Possible Solution

Thread the known target default branch through `repair-comment-router.yml`, comment-router config, and the `clawsweeper_item` re-review dispatch, with regression coverage for a non-`main` branch target.

## Reproduction Assessment

Yes. It is source-reproducible without running the live workflow: the branch is present in webhook intake, omitted from the comment-router re-review dispatch, and then `sweep.yml` defaults the missing branch to `main` before cloning.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve `target_branch` across the existing re-review dispatch chain and add tests for a `master` or `trunk` default-branch target.
