## Summary

Keep this PR open: the direction is narrow and useful, but the hosted webhook suppression misses slash-form hatch commands, so non-OpenClaw repositories can still get ClawSweeper acknowledgement/comment-routing activity from `/clawsweeper hatch`; the PR also has only test-command validation, not real behavior proof.

## What This Changes

The PR limits PR-egg rendering, image generation, hatch-only apply work, webhook intake, and router handling to `openclaw/*` repositories, with regressions for non-OpenClaw rendering and hatch suppression.

## Best Possible Solution

Make hatch-command detection consistent across hosted webhook and router intake, cover both mention and slash forms for non-OpenClaw repositories, then land with redacted live proof of the suppressed path.

## Reproduction Assessment

Yes, from source inspection: `dashboard/worker.ts` accepts author read-only slash hatch commands, while the new non-OpenClaw hatch detector only matches mention syntax. I did not run the webhook test suite because this review is read-only.

## Solution Assessment

No, not yet: the core apply/render/router direction is maintainable, but the hosted webhook should use equivalent hatch parsing for slash and mention forms before the suppression is complete.
