## Label Justifications

- `P1`: The issue describes repeated Codex review timeouts on a live external PR workflow, which blocks ClawSweeper from reviewing large media-heavy contributor PRs.
- `impact:crash-loop`: The affected behavior is a review worker timeout/stall before a durable review result is produced.

## Summary

Keep open; current main still uses the fixed 600000 ms default unless a dispatch or manual workflow input supplies `codex_timeout_ms`, media-proof preprocessing still runs before Codex, and PR surface stats are rendered for review context rather than used for timeout sizing.

## What This Changes

Request adaptive per-item Codex timeouts for large PRs and video-proof reviews instead of relying on the 600000 ms default for webhook-dispatched reviews.

## Best Possible Solution

Add a conservative adaptive timeout calculation for PR item dispatches and failed-review retries that uses changed-file/addition counts plus detected video-proof URLs, caps total review time within sweep job limits, and preserves the 600000 ms default for small or non-media items.

## Reproduction Assessment

Yes from source inspection, though not from a live timeout run. Webhook-dispatched item reviews do not include `codex_timeout_ms`, sweep falls back to 600000 ms, and media-proof preprocessing runs before Codex while sharing the outer shard budget.

## Solution Assessment

Yes; the narrowest maintainable fix is to calculate and pass a bounded timeout where the PR payload is accepted, including media-proof budget, rather than increasing every review's global default.
