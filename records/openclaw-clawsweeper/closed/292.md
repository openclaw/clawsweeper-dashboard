## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Honor the read-only repair sandbox:** `src/repair/openai-compatible-tools-runner.ts:80-112`
  - body: run-worker still invokes the model backend with `--sandbox read-only`, and the repair docs say planning runs are read-only. This adapter ignores that flag and always exposes `write_file`, `replace_in_file`, `apply_patch`, plus arbitrary `run_command`, so attacker-controlled PR/job text can mutate the checkout during planning before the deterministic execute gates run. Parse/enforce the sandbox and disable write-capable tools and side-effecting commands in read-only mode.
  - confidence: 0.91
- **[P1] Keep GitHub tokens out of model-controlled tools:** `src/repair/model-backend.ts:38`
  - body: The current repair boundary strips `GH_TOKEN`/`GITHUB_TOKEN` before Codex sees attacker-controlled issue or PR text, but this restores the token for the OpenAI-compatible runner. Because that runner lets the model issue shell commands, the model can call `gh` directly with the app token instead of consuming only deterministic prehydrated evidence. Keep GitHub API calls in the wrapper/preflight layer or pass only a narrowly controlled read helper rather than restoring the token to the tool process.
  - confidence: 0.88
- **[P2] Route steerable sessions through the selected backend:** `src/repair/run-worker.ts:294-296`
  - body: The new backend command is only used after the app-server early return, so setting `CLAWSWEEPER_MODEL_BACKEND_CONFIG` has no effect when `CLAWSWEEPER_STEERABLE_CODEX=1`. That workflow mode is documented and supported by the repair worker, so operators enabling both would silently get the Codex app-server path instead of the requested OpenAI-compatible backend. Either make the combination explicit/unsupported or route the app-server branch through the backend selection.
  - confidence: 0.78
