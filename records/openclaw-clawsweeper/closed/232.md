## Summary

Keep this issue open: current main still has a source-visible fast-ack/router mismatch, because the fast-ack gates can match inline prose mentions while the router parser only accepts command-shaped lines.

## What This Changes

The issue requests aligning ClawSweeper fast-ack command detection with the repair command router so inline prose mentions are not acknowledged as queued commands when the router will ignore them.

## Best Possible Solution

Use one parser-safe command predicate, or equivalent shared logic, so every fast-ack path only posts a queued ack for comments the repair router will actually parse.

## Reproduction Assessment

Yes from current source inspection: the reported inline prose body matches the fast-ack regex but not the router's whole-line command grammar. I did not run the live webhook server, so this is source-reproducible rather than a full live reproduction.

## Solution Assessment

Yes; rejecting null `parseCommand()` results before posting a queued ack is the narrowest fix and avoids broadening inline command support accidentally.
