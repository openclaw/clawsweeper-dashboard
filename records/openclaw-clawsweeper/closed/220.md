## Summary

Keep open: current main still uses a best-effort list/prune, create, list/prune flow for fast acknowledgements in both the local webhook and hosted Worker, while the docs promise one queued status comment and existing tests do not cover the stale-list concurrent creation race.

## What This Changes

Ensure concurrent duplicate webhook deliveries for the same source command converge to one marker-backed fast-ack/status comment.

## Best Possible Solution

Make the fast-ack/status identity deterministic or add a final router-side convergence cleanup so one source command id leaves at most one visible ack/status comment across webhook redeliveries.

## Reproduction Assessment

Yes at source level: two concurrent webhook handlers can both observe no matching ack before posting, and the current tests only prove cleanup after duplicates become visible to listing. I did not run a live GitHub repro from this read-only review.

## Solution Assessment

Yes, keeping this open for a narrow idempotency/convergence fix is the best path; another best-effort prune-only pass would not address the stale-list race described in the report.
