## What This Changes

Adds a default cap of three ClawSweeper event hook notifications per notifier run, configurable via `--max-sent` or `CLAWSWEEPER_EVENT_NOTIFY_MAX_SENT`, plus cap coverage and a changelog entry.

## Best Possible Solution

Land the narrow capped notifier once live proof shows overflow events remain unledgered and retryable, and maintainers accept or tune the default cap for the shadow update thread.

## Reproduction Assessment

Yes from source inspection: current `main` posts every item in `collected.events`, so a configured notifier run with more than three unledgered events will attempt all of them. I did not execute the notifier because this review is read-only.

## Solution Assessment

Mostly yes: the patch is narrow, keeps overflow events out of the ledger, and adds focused coverage. The remaining question is whether the default cap of three is the desired product/operations tradeoff, plus live proof that the drain path works outside the unit harness.
