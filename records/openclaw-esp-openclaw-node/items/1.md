## Summary

Keep this PR open: the OIDC direction is supported and current main still uses the registry API token, but the patch likely breaks the existing checkout step by granting only `id-token: write`, and the attached screenshot does not prove a successful after-change upload.

## What This Changes

The PR changes `.github/workflows/upload_component.yml` to publish with GitHub OIDC by adding job `id-token: write` permission and removing `secrets.IDF_COMPONENT_API_TOKEN`.

## Best Possible Solution

Land the OIDC upload workflow with explicit least-privilege permissions, including both `contents: read` for checkout and `id-token: write` for the registry upload, after maintainers have redacted successful workflow evidence.

## Reproduction Assessment

Yes for source-level review: GitHub Actions sets unspecified permissions to `none` once a job permissions block is present, and this workflow still uses actions/checkout without an alternate token or SSH key. I did not run the workflow, but the failure path is directly supported by the workflow YAML and upstream docs.

## Solution Assessment

No as written: switching the upload action to OIDC is the right narrow direction, but the permissions block should retain `contents: read` and the PR should include a redacted successful workflow run or upload log.
