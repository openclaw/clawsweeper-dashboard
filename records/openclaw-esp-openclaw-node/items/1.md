## Summary

## What This Changes

## Best Possible Solution

Keep the OIDC migration direction, add `contents: read` beside `id-token: write`, and require redacted upload or dry-run evidence before merge.

## Reproduction Assessment

Yes for the merge blocker from source: the PR grants only `id-token: write` at job scope while the job still uses `actions/checkout`. The actual registry upload success is not reproduced because it requires the main-branch publish workflow and registry trust configuration.

## Solution Assessment

No as currently patched: moving to OIDC is reasonable, but the workflow should preserve checkout access and include real publish or dry-run evidence before becoming the maintained release path.
