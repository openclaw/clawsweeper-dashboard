## What This Changes

## Best Possible Solution

Implement scope support with one canonical bounded scope representation reused for signing and JSON, add focused registry/connect tests, and require redacted real ESP/gateway proof before merge.

## Reproduction Assessment

Yes for the PR-introduced defect from source inspection: register enough scopes for the comma-separated form to exceed 127 bytes, and the PR signs only the prefix while still sending every registered scope in JSON. ## Solution Assessment

No in its current form. Scope support belongs in the registry/connect path, but this branch needs one canonical representation for signing and JSON, explicit validation/limits, tests, real behavior proof, and maintainer confirmation of the public API contract.
