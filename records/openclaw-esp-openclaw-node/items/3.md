## Summary

Keep this PR open: it is a useful scope-support feature, but it still needs real behavior proof and has a blocking authorization-scope serialization defect before it is merge-ready.

## What This Changes

The PR adds scope registration limits/storage/API/docs and sends registered scopes in the connect request and signed auth payload.

## Best Possible Solution

Land scope support only after the signed payload and advertised JSON are derived from the same complete representation, with explicit rejection or a documented tested limit for unsupported scope sizes.

## Reproduction Assessment

Yes for the blocking PR defect by source inspection: registering scopes whose comma-joined value exceeds the 128-byte buffer makes the signed payload omit data while the JSON still advertises all registered scopes. I did not run an ESP-IDF target test.

## Solution Assessment

No as submitted; the maintainable solution is a scope registry whose advertised JSON and signed payload are derived from the same complete representation, with explicit failure for unsupported size or syntax.
