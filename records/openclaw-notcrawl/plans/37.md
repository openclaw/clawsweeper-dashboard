# Coding Plan for openclaw/notcrawl#37: API sync aborts on copied synced_block children returning object_not_found

## Summary

Current main still has the reported API traversal behavior: copied synced blocks are not distinguished from ordinary child-bearing blocks, and child-fetch errors propagate out of API sync. This is a narrow valid bug with a clear source-level reproduction path, not a cleanup close.

## Plan

Fix the Notion API sync abort for copied synced blocks. In `internal/notionapi/api.go`, keep storing every block, but replace the raw `truthy(block["has_children"])` recursion check with a helper that returns false for copied synced blocks where `block.string("type") == "synced_block"` and `block.mapObj("synced_block").mapObj("synced_from")` is non-empty. Continue recursing for original/source synced blocks where `synced_from` is absent or empty, and do not crawl `synced_from.block_id` or synthesize copied child rows in this narrow fix because `blocks.id` is the primary key. Add regression tests in `internal/notionapi/api_test.go` for a copied synced block that is stored but never requests `/blocks/{copyID}/children`, and an original synced block that still fetches its children. Add a concise `CHANGELOG.md` fix entry if repository policy expects one; do not add unrelated refactors or dependency changes.
