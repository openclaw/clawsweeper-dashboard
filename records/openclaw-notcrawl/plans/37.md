## Summary

Keep open: current main still recurses into every API block with `has_children` and propagates a `/blocks/{id}/children` error, so the copied synced-block failure described here remains a valid source-level bug.

## Plan

Fix API block traversal for copied Notion synced blocks. In `internal/notionapi/api.go`, keep storing every block, but change `walkBlocks` so it does not recurse into children when `block.type == "synced_block"` and `block.synced_block.synced_from` is non-empty; preserve existing recursion for original/source synced blocks where `synced_from` is absent or empty, and for all other child-bearing blocks. Add focused httptest coverage in `internal/notionapi/api_test.go`: one test should prove a copied synced block is upserted but `/blocks/{copyID}/children` is not requested, and another should prove an original synced block still fetches children. Do not crawl `synced_from.block_id` under the copy and do not change `blocks.id` primary-key semantics. Keep the patch scoped to API traversal/tests, avoid unrelated refactors, and add release-note/changelog text only if maintainers explicitly want it.
