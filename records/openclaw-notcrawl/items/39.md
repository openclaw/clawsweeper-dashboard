## Label Justifications

- `P2`: This is a normal-priority API sync resilience bug with a clear failure mode and limited blast radius.
- `impact:crash-loop`: The reported `/users` 403 currently aborts API sync before page and database discovery can run.
- `impact:other`: The zero-discovery case affects freshness diagnostics for wrappers even when an existing local mirror remains usable.

## Summary

This should stay open: current main and the latest release still treat `/users?page_size=100` as a hard API-sync prerequisite, and a zero-result API discovery can still be recorded as a finished sync with only raw counts printed.

## What This Changes

The issue requests tolerating Notion 403 `restricted_resource` from user listing and surfacing clearer diagnostics when API discovery finds zero fresh pages, databases, blocks, and comments.

## Best Possible Solution

Make only `/users` 403 `restricted_resource` nonfatal, continue API discovery with missing user labels, and emit a clear zero-discovery warning while preserving existing hard failures for unrelated Notion API errors.

## Reproduction Assessment

Yes from source inspection: `Client.Sync` calls `listUsers` first, and `listUsers` returns the `/users?page_size=100` error directly. I did not run a live Notion-token reproduction in this read-only review.

## Solution Assessment

Yes for the core fix: user labels should remain best-effort metadata instead of blocking page/database mirroring. A warning is the narrowest maintainable diagnostic; persisted wrapper-readable freshness fields can be added only if maintainers want that broader status surface.
