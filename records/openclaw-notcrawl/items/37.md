## Summary

Current main still recurses into every Notion API block with `has_children` and propagates `/children` errors back through page ingestion, so this remains a narrow valid API traversal bug rather than a cleanup close.

## What This Changes

The issue asks for API block traversal to skip fetching children for copied Notion `synced_block` blocks while still storing the copy and crawling source synced blocks.

## Best Possible Solution

Add a guarded traversal check in `internal/notionapi/api.go` that stores copied synced blocks but skips their child fetch when `synced_block.synced_from` is populated, with regression tests for copied versus source synced blocks.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a copied `synced_block` with `has_children: true` makes `walkBlocks` request `/blocks/{copyID}/children`, and a Notion 404 from that request is returned up through `ingestPage` and `Sync`. I did not run a live Notion-token reproduction in this read-only review.

## Solution Assessment

Yes, the proposed skip for synced-block copies is the narrowest maintainable fix because it avoids the known bad child request while preserving normal recursion for original/source synced blocks and avoiding unsafe source-child ID reuse.
