## Summary

This should stay open: current main still recurses into every block with `has_children` and returns the copied synced-block child request error, so the reported API sync abort is a valid narrow bug rather than stale cleanup.

## What This Changes

The issue asks API block traversal to skip fetching children for copied Notion `synced_block` blocks while still storing the copy and crawling source synced blocks.

## Best Possible Solution

Add a narrow child-fetch predicate for API blocks that skips copied synced blocks, preserves source synced-block recursion, and covers both cases with handler-level regression tests.

## Reproduction Assessment

Yes, source-level: a handler test can return a copied `synced_block` with `has_children: true` and then return a 404 for `/blocks/{copy}/children`, which current `walkBlocks` will propagate. I did not run a live Notion reproduction in this read-only review.

## Solution Assessment

Yes; skipping child traversal only for copied `synced_block` values with non-empty `synced_from` is the narrowest maintainable fix and avoids unsafe source-child ID reuse.

## Repair Work Prompt

Fix API block traversal for copied Notion synced blocks. In `internal/notionapi/api.go`, keep storing every block returned by `/blocks/{parent}/children`, but only recurse into child blocks when `has_children` is true and the block is not a copied `synced_block` with a non-empty `synced_block.synced_from`. Do not follow `synced_from.block_id` from the copy or synthesize copied child IDs in this PR. Add focused tests in `internal/notionapi/api_test.go` using an httptest server: one test must verify a copied synced block is stored and `/blocks/{copyID}/children` is not requested even if `has_children` is true; another must verify an original/source synced block with absent or null `synced_from` still fetches children. Keep the change limited to API traversal and tests. Add a concise changelog fix entry only if maintainers expect one for repair PRs.
