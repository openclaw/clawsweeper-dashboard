## Label Justifications

- `P2`: The bug is a narrow CLI parsing defect with clear source reproduction and limited blast radius, but it can inspect the wrong file and write output to an unintended path.
- `impact:data-loss`: The reported missing-positional case can overwrite a plugin source path because the same token may be treated as both entrypoint and output.

## Summary

Keep open: current main still selects the first non-flag token as the capture entrypoint, so a value passed to `--output`, `--plugin-root`, or `--sdk` can be treated as plugin code before the intended positional path is considered.

## What This Changes

The issue asks for capture CLI positional entrypoint detection to ignore values consumed by value-taking flags.

## Best Possible Solution

Add a dependency-free value-aware entrypoint parser for capture paths, keep existing positional-first behavior, and cover flag-first plus missing-entrypoint cases with focused regression tests.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction: current main's predicate returns `./out.json` for `--output ./out.json ./my-plugin.js`; I did not run full capture because it imports plugin code and may write artifacts.

## Solution Assessment

Yes. The narrowest maintainable solution is to exclude tokens consumed by known value-taking flags before selecting the positional entrypoint, without adding a runtime CLI parsing dependency.
