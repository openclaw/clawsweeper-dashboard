## Summary

This should stay open: current main still uses a flag-unaware first non-hyphen token search for capture entrypoints, the flag-first case is valid, and I found no duplicate or fixing PR.

## What This Changes

The issue asks for capture CLI positional parsing to ignore values consumed by flags before selecting the plugin entrypoint.

## Best Possible Solution

Introduce a small dependency-free capture argument helper that skips tokens consumed by value flags, apply it to both capture entrypoints, and add regression tests for flag-first and missing-positional cases.

## Reproduction Assessment

Yes. Source inspection plus a focused parser check show the current predicate selects `./out.json` from the flag-first argument list before it can reach the real plugin path.

## Solution Assessment

Yes. The suggested skip-set approach is directionally right, but the maintainable fix should cover all value-taking capture flags and the matching `plugin-inspector capture` parser too.
