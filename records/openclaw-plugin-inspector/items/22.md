## Label Justifications

- `P2`: The bug is a narrow CLI parsing defect with clear reproduction and limited blast radius, but it can inspect the wrong file and write JSON to an unintended path.
- `impact:data-loss`: The reported missing-positional case can overwrite a plugin source path because the same token is treated as both entrypoint and output.

## Summary

This is a valid current-main bug: the capture helper still chooses the first non-flag token as the entrypoint, so a value for `--output`, `--plugin-root`, or `--sdk` can be misrouted as plugin code and the intended positional entrypoint can be ignored.

## What This Changes

The issue requests fixing `capture-cli` so values consumed by flags before the positional entrypoint are not mistaken for the plugin entrypoint.

## Best Possible Solution

Add dependency-free value-flag-aware positional parsing for `capture-cli`, preserve the documented positional-first flow, and cover flags-before-entrypoint plus missing-entrypoint overwrite cases.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: the current predicate returns `./out.json` for `--output ./out.json ./my-plugin.js`. I did not run a full capture because that path imports plugin code and may write artifacts.

## Solution Assessment

Yes, the narrowest maintainable fix is to exclude values consumed by known value-taking flags before selecting the positional entrypoint, with regression coverage and no new runtime dependency.
