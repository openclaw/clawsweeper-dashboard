# Coding Plan for openclaw/plugin-inspector#22: [Bug]: capture-cli positional-argument detection misidentifies flag values as entrypoint path

Source report: [records/openclaw-plugin-inspector/items/22.md](records/openclaw-plugin-inspector/items/22.md)

## Summary

This is a valid current-main bug: the capture helper still chooses the first non-flag token as the entrypoint, so a value for `--output`, `--plugin-root`, or `--sdk` can be misrouted as plugin code and the intended positional entrypoint can be ignored.

## Plan

Fix the current `capture-cli` argument parser so values consumed by `--output`, `--plugin-root`, and `--sdk` are never selected as the positional entrypoint. Preserve existing positional-first usage, `--mock-sdk`/`--real-sdk`, and the offline/credential-free default; do not add runtime dependencies. Add focused regression coverage in `test/helper-clis.test.js` for `--output` before the entrypoint and for a missing positional argument where `--output` points at a plugin source path, proving the source file is not overwritten. Check whether `src/synthetic-probes-cli.js` should share the same parser helper for consistency, but do not broaden behavior beyond the helper CLI argument parsing defect. Add an appropriate `CHANGELOG.md` fix entry if maintainers expect release-note coverage. Because this changes plugin-inspector CLI behavior, run the helper CLI test and the crabpot plugin-inspector smoke path required by `AGENTS.md` before calling the repair done.
