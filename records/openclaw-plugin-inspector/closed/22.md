## Label Justifications

- `P2`: The bug is a narrow CLI parsing defect with clear source reproduction and limited blast radius, but it can inspect the wrong file and write JSON to an unintended path.
- `impact:data-loss`: The reported missing-positional case can treat the same token as both entrypoint and output path, which can overwrite a plugin source file.

## Summary

This is a valid current-main bug with a narrow fix shape: the helper capture CLI and the packaged `plugin-inspector capture` path still select the first non-flag token before excluding values consumed by flags, and no canonical fix PR appears to exist.

## What This Changes

Make capture CLI positional entrypoint parsing ignore values consumed by flags such as `--output`, `--plugin-root`, and `--sdk`.

## Best Possible Solution

Add dependency-free, value-flag-aware positional parsing for both capture entrypoints and cover flags-before-entrypoint plus missing-positional overwrite scenarios.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: the current predicate returns `./out.json` for `--output ./out.json ./my-plugin.js`, and the missing-positional case computes the same token for entrypoint and output. I did not run a full capture because that path imports plugin code and may write artifacts.

## Solution Assessment

Yes, the narrowest maintainable fix is to exclude values consumed by known value-taking flags before selecting the positional entrypoint, shared across the helper and packaged capture command, with no new runtime dependency.
