## Label Justifications

- `P2`: This is a bounded security-hardening PR for the autoreview helper, with a merge-blocking compatibility issue but no current production outage.
- `merge-risk: 🚨 compatibility`: Unconditional Claude `--bare` changes the default runtime contract for existing `--engine claude` users.
- `merge-risk: 🚨 auth-provider`: Claude docs state bare mode skips OAuth/keychain reads, so the PR can force users onto a different authentication path.

## Summary

Keep open: the hardening direction is useful, but the Claude path is not merge-ready because the new unconditional bare mode can break existing subscription/OAuth-based Claude autoreview users, and the PR proof does not show the changed runtime path clearly enough.

## What This Changes

The PR adds Codex and Claude isolation flags to the autoreview helper, expands accepted Codex thinking levels, adds a hidden isolation self-test, and documents the review-boundary contract.

## Best Possible Solution

Keep the Codex isolation flags, but change the Claude isolation path to block project-controlled hooks, skills, plugins, MCP, CLAUDE.md, and project/local settings while preserving the existing Claude Code auth path, then add clear terminal proof of the resulting behavior.

## Reproduction Assessment

Yes. Source inspection shows the PR adds `--bare` to every Claude engine invocation, and official Claude docs state bare mode skips OAuth/keychain reads, which is enough to source-reproduce the compatibility regression. ([code.claude.com](https://code.claude.com/docs/en/headless))

## Solution Assessment

No. The Codex hardening is direct, but the Claude solution should use an auth-preserving isolation path or explicitly document a maintainer-approved breaking auth change; Claude's own CLI reference points to safe mode as preserving auth while disabling customizations. ([code.claude.com](https://code.claude.com/docs/en/cli-reference?utm_source=openai))
