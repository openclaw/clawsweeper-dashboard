# Coding Plan for openclaw/openclaw-windows-node#458: Surface visible diagnostics for keyless chat events

Source report: [records/openclaw-openclaw-windows-node/items/458.md](records/openclaw-openclaw-windows-node/items/458.md)

## Summary

Keep open: current main still drops keyless chat and agent events with warning logs only, while the requested user-visible one-shot diagnostic is not implemented. The related incompatible-handshake work fixed a different path and does not cover mixed-version keyless events after connection.

## Plan

Implement the visible diagnostic requested by https://github.com/openclaw/openclaw-windows-node/issues/458. In `OpenClawChatDataProvider`, when `OnChatMessageReceived` or `OnAgentEventReceived` drops an event because `SessionKey` is null/empty, raise one user-visible `ChatProviderNotification` or equivalent chat diagnostic per connection/reconnect that tells the user some events were discarded and suggests updating or restarting the gateway. Do not route unknown events to `main`. Do not include `message.Text`, `evt.Data`, raw JSON, prompts, tool output, URLs, or message content in the diagnostic or logs; only include event kind/count-style context. Reset the one-shot guard when the provider leaves/re-enters the connected state. Add focused tests in `OpenClawChatDataProviderTests` for first chat drop emits once, repeated drops do not duplicate in the same connection, reconnect resets the guard, agent drops are covered, and message content is not present in the notification. Leave changelog handling to maintainer landing policy; mention the user-facing diagnostic in the PR summary. Run `./build.ps1`, `dotnet test ./tests/OpenClaw.Shared.Tests/OpenClaw.Shared.Tests.csproj --no-restore`, and `dotnet test ./tests/OpenClaw.Tray.Tests/OpenClaw.Tray.Tests.csproj --no-restore` before completion.
