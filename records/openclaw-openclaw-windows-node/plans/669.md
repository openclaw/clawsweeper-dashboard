## Summary

The request remains necessary on current main: both web Canvas and native A2UI Canvas use ordinary WinUI windows without a fullscreen presenter transition or F11/Escape fullscreen handling. It is a bounded, low-priority UX feature with no merged or open canonical replacement, but implementation should cover both Canvas surfaces consistently.

## Plan

Implement borderless fullscreen support for both Windows Canvas surfaces tracked by https://github.com/openclaw/openclaw-windows-node/issues/669. Add a shared or consistently applied reversible fullscreen mechanism for CanvasWindow and A2UICanvasWindow: F11 enters or exits fullscreen, Escape exits only, window chrome is hidden, and the previous presenter, bounds, monitor placement, and topmost state are restored. Ensure shortcuts work while WebView2 or native Canvas content has focus. Keep the patch scoped to window presentation and keyboard handling; do not change gateway connection, credential precedence, Canvas URL/security filtering, navigation, or NodeService routing. Add focused unit coverage for extracted state-transition logic where practical. Put user-visible release-note context and manual proof in the PR body; do not edit CHANGELOG.md. Run the required repository validation and provide a redacted Windows recording or screenshots plus diagnostic/live output showing F11 enter, F11 exit, and Escape exit on both Canvas types.
