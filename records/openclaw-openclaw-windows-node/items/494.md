## Label Justifications

- `P1`: The report describes an upgrade-time regression that blocks remote `system.run` command execution on Windows 10 nodes.
- `impact:security`: The requested behavior changes the sandbox boundary for agent-started local command execution.

## Summary

Keep open: current main still intentionally blocks `system.run` on hosts where MXC is unavailable, including Windows build 19044, and this is not already implemented or duplicated. The remaining decision is a security/product choice about whether unsupported Windows builds may run agent-started commands on the host without MXC containment.

## What This Changes

Requests an opt-out or unsandboxed fallback for `system.run` on Windows 10 builds where MXC sandboxing is unavailable.

## Best Possible Solution

Resolve the compatibility gap with an explicit maintainer policy: either restore a clearly warned high-risk host-execution opt-in for unsupported MXC hosts, or document that v0.6 node exec requires Windows build 26100+ and make the Windows 10 limitation visible before upgrade.

## Reproduction Assessment

Yes, by source inspection: build 19044 is below the MXC minimum, so availability is false, the Sandbox page disables the controls, and the runner returns a blocked result before checking the sandbox-off setting. I did not run a live Windows 10 smoke in this read-only checkout.

## Solution Assessment

Unclear until maintainers decide the security policy. The safest maintainable path is fail-closed by default, with either a deliberate high-risk opt-in host mode or explicit Windows 10 unsupported messaging for sandboxed node exec.
