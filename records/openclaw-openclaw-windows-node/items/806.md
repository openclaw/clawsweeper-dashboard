## Summary

Keep this PR open for maintainer review: it adds useful TTS resilience and a readiness command, but the diff also changes explicit and unknown provider requests from strict failures into Windows playback, which is a compatibility-sensitive provider contract change.

## What This Changes

The PR adds Windows TTS fallback behavior for unavailable configured/requested TTS providers, adds a `tts.status` readiness command, and wires that command through node registration, MCP/CLI docs, setup allowlists, and tests.

## Best Possible Solution

Keep fallback for configured/default provider degradation, preserve strict behavior for explicit and unknown provider requests unless maintainers approve that contract change, and land `tts.status` with focused API and allowlist coverage.

## Reproduction Assessment

Yes from source inspection: at PR head, `tts.speak` with an explicit unavailable or unknown provider reaches `ResolveEffectiveProvider`, which returns Windows whenever Windows is ready. I did not run the Windows tray, but the code path is direct.

## Solution Assessment

No. The maintainable path is to make configured/default fallback resilient while preserving strict explicit provider and unsupported-provider behavior, unless maintainers explicitly approve the compatibility break.
