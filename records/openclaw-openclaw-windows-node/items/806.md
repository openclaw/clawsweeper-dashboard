## Summary

Keep open: the PR adds useful TTS status and fallback plumbing that is not on current main, but it silently changes explicit provider requests to Windows fallback and lacks real behavior proof for the runtime TTS path.

## What This Changes

Adds a `tts.status` command, TTS provider readiness/fallback resolution, fallback-aware speak results, setup/MCP/skill docs, and shared/setup regression tests.

## Best Possible Solution

Land the additive status command and default-provider fallback only after preserving explicit provider semantics or explicitly approving the break, with real tray/MCP proof of fallback and status output.

## Reproduction Assessment

Yes. Source inspection shows `ResolveEffectiveProvider("piper", ..., readyProviders: {"windows"})` returns effective `windows` with `FellBack=true`, so an explicit provider override no longer reaches the existing Piper not-downloaded error path.

## Solution Assessment

No. The status command is a reasonable direction, but the fallback implementation is not the safest shape because it changes explicit provider overrides; default/configured fallback should be separated from caller-requested strict provider behavior unless maintainers approve the break.

- **[P1] Preserve explicit provider failures:** `src/OpenClaw.Shared/Capabilities/TtsCapability.cs:100`
  - body: `ResolveEffectiveProvider` does not distinguish an explicit per-call `provider` from the configured/default provider.
