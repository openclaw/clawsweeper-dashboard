## Summary

Keep open: current main still has a source-reproducible path where the legacy Tray/Hub chat readiness flow sends the first-run bootstrap message based only on a one-shot settings flag, with no durable existing-workspace or configured-gateway guard.

## What This Changes

The issue reports that Tray chat can inject the first-run bootstrap prompt for a workspace that already has OpenClaw state and configuration.

## Best Possible Solution

Keep the post-onboarding hatching flow, but gate prompt injection on a durable fresh-setup signal or explicit confirmation and skip it when existing gateway/workspace state shows the user is already configured.

## Reproduction Assessment

Yes by source inspection: with a connected operator client and `HasInjectedFirstRunBootstrap == false`, `ChatPage` calls `BootstrapAsync`, which sends the first-run prompt without checking existing workspace files or configured gateway state. I did not run a live Windows tray repro in this read-only review.

## Solution Assessment

Yes, the requested guard is the right direction; the narrow maintainable fix is to move the bootstrap decision from a bare one-shot flag to a true fresh-setup/configured-workspace check while preserving the intended post-setup launch path.
