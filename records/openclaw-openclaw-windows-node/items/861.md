## Summary

Keep open: current main has meaningful Markdown rendering and selectable TextBlocks, but the central full-fidelity/select-all request is not definitely solved. The renderer still intentionally flattens or disables active Markdown surfaces and renders multi-block messages as multiple controls, while the parent tracking work at https://github.com/openclaw/openclaw-windows-node/issues/839 remains open.

## What This Changes

The issue requests full-fidelity Markdown rendering in chat with selectable text across the message content.

## Best Possible Solution

Define a safe chat Markdown viewer contract, then implement a single selectable message surface or equivalent select-all behavior while preserving the current no-fetch/no-click policy unless maintainers explicitly change it.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a plausible reproduction path for the remaining select-all gap: multi-block Markdown is rendered as multiple selection-enabled controls rather than one selectable document surface.

## Solution Assessment

No: the current renderer is a useful partial implementation, but it is not enough to treat the full-fidelity/select-all request as done. The narrow maintainable path is to first settle the safe fidelity contract, then implement the viewer/selection behavior with focused tests and visual proof.
