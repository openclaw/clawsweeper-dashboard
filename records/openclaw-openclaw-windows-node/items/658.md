## Summary

Keep open: current main still returns screenshots as base64/data URLs and has no visible media-ID handoff for chat attachments in this repository, so the requested one-line screenshot-to-channel workflow is not implemented.

## What This Changes

The issue asks for `screen.snapshot` to return a gateway media ID so agents can attach screenshots to Discord or Telegram messages without decoding and writing files.

## Best Possible Solution

Define one canonical media handoff contract between node capability results and `chat.send`, then update `screen.snapshot` with compatibility for existing base64 consumers and focused regression tests.

## Reproduction Assessment

Yes from source inspection: current main returns `base64` and a data URL from `screen.snapshot`, while chat attachments are modeled as base64 `content` objects rather than a media ID handoff. I did not execute a live gateway/channel flow because this is a read-only review.

## Solution Assessment

Unclear. Returning a media ID may be the right end state, but it crosses the node, gateway, and chat attachment contract and should preserve existing agents that consume `base64` or `image`.
