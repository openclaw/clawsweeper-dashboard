## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Use a supported deep link for Advanced setup:** `src/OpenClaw.SetupEngine.UI/Pages/WelcomePage.xaml.cs:74-77`
  - body: The Advanced setup button starts the tray with `--page connection`, but the tray only forwards command-line arguments to an existing instance when the argument starts with `openclaw://`. In the common case where the tray is already running, this launches a second instance that exits without navigating, then closes SetupEngine.UI and leaves the user outside Connection settings. Route this through an existing supported link such as `openclaw://hub/connection`.
  - confidence: 0.93
- **[P2] Adopt the existing keepalive marker before spawning:** `src/OpenClaw.Tray.WinUI/App.xaml.cs:1764-1777`
  - body: SetupEngine writes a keepalive marker after starting the WSL keepalive, but the tray startup path now ignores that marker and starts a new `wsl.exe ... sleep infinity` every launch. That regresses the current marker/adoption contract and can leave duplicate or orphaned keepalive processes that interfere with restart and uninstall. Reuse a shared marker-aware helper or port the existing adoption logic.
  - confidence: 0.9
- **[P2] Resolve the tray executable from the SetupEngine folder:** `src/OpenClaw.SetupEngine.UI/Pages/CompletePage.xaml.cs:107-108`
  - body: In the release layout SetupEngine.UI runs from `<tray output>\SetupEngine`, but these candidates check `<tray output>\SetupEngine\OpenClaw.Tray.WinUI.exe` and `<tray output>\OpenClaw.Tray.WinUI\OpenClaw.Tray.WinUI.exe`, not the actual parent tray exe. When users enable the startup toggle on the completion page, `trayPath` is null and the opt-in silently does nothing. Include the parent directory tray executable path.
  - confidence: 0.88
- **[P2] Keep the replacement setup UI localizable:** `src/OpenClaw.SetupEngine.UI/Pages/WelcomePage.xaml:15-21`
  - body: The old onboarding contract uses localized resources for all user-visible setup strings, but the new SetupEngine.UI pages hard-code English text and the project has no resource/localization hook. Because this PR deletes the localized V2 surface, non-English users regress to English during first-run setup. Wire the new setup UI into the existing resource pipeline before removing the old path.
  - confidence: 0.86
