## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Package SetupEngine.UI with the tray artifacts:** `src/OpenClaw.Tray.WinUI/App.xaml.cs:2720-2724`
  - body: `ShowOnboardingAsync` now returns if `ResolveSetupEngineUiPath()` cannot find `OpenClaw.SetupEngine.UI.exe`, but the release workflow still publishes only `src/OpenClaw.Tray.WinUI` and the tray csproj has no SetupEngine content/project reference. Installed first-run setup and `--uninstall` will fail unless the delegated executable is included in every publish/MSIX/installer layout.
  - confidence: 0.92
- **[P1] Preserve gateway identities unless they are app-owned local WSL:** `src/OpenClaw.SetupEngine/SetupSteps.cs:166-176`
  - body: `CleanupStaleGatewayStep` deletes the identity directory and registry record for any record whose URL matches `ctx.GatewayUrl`. With the bundled config setting `CleanBeforeRun=true`, a user with an existing loopback, SSH, or forwarded gateway at the default URL can lose paired device credentials before the new setup succeeds; gate this on app-owned local WSL evidence instead of URL alone.
  - confidence: 0.9
- **[P1] Quote and restrict the installer URL before running bash:** `src/OpenClaw.SetupEngine/SetupSteps.cs:524-527`
  - body: `InstallCliStep` interpolates `Gateway.InstallUrl` directly into `curl -fsSL {installUrl} | bash`, so shell metacharacters in config are executed and the previous `--proto '=https' --tlsv1.2` restriction is lost. Parse and allowlist HTTPS URLs, then shell-quote the value before composing the WSL script.
  - confidence: 0.88
- **[P2] Reject unsafe ExtraConfig keys before composing the shell script:** `src/OpenClaw.SetupEngine/SetupSteps.cs:588-593`
  - body: The value is single-quoted, but the config key is still inserted as shell syntax after removing only a few characters. Keys containing whitespace, `$()`, or backticks can change the command that runs; validate keys against the supported OpenClaw config-key grammar or pass them through an argv-safe path.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces concrete setup execution and credential-boundary concerns in the new config-driven engine.

Concerns:

- **[high] Installer URL is executed through bash without safe parsing:** `src/OpenClaw.SetupEngine/SetupSteps.cs:526`
  - body: `Gateway.InstallUrl` is interpolated into a `bash -c` command and no longer uses the current HTTPS/TLS and shell-quoting pattern, so a malicious or malformed config can execute unintended shell syntax during setup.
  - confidence: 0.88
- **[medium] ExtraConfig keys remain shell syntax:** `src/OpenClaw.SetupEngine/SetupSteps.cs:592`
  - body: Only values are shell-quoted; keys are inserted into the generated script after a partial character strip, leaving command substitution and whitespace parsing risks.
  - confidence: 0.82
- **[medium] Gateway identity cleanup lacks ownership proof:** `src/OpenClaw.SetupEngine/SetupSteps.cs:166`
  - body: Deleting a registry record and identity directory by URL can remove existing operator/node credentials for a gateway that is not the setup engine's app-owned WSL instance.
  - confidence: 0.9
