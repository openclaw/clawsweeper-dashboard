## Summary

Keep open: the underlying Usage-page disconnected-client bug is source-backed, but the patch still misses the operator-client swap window during reconnect and the external PR has no after-fix real behavior proof.

## What This Changes

The branch updates the Usage page layout/localized title strings and adds disconnected-gateway handling around usage refresh requests.

## Best Possible Solution

Handle both disconnected existing clients and operator-client replacement during reconnect, then show redacted runtime/UI proof that the Usage page recovers and reloads values after the gateway reconnects.

## Reproduction Assessment

Yes by source inspection: current main can start Usage loading with a non-null but disconnected gateway client, while the request path returns early when disconnected. I did not run the WinUI app, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

No. Checking IsConnectedToGateway and listening for StatusChanged is the right direction, but the implementation should also reattach when the connection manager replaces the operator client during reconnect.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P2] Handle operator-client swaps during reconnect:** `src/OpenClaw.Tray.WinUI/Pages/UsagePage.xaml.cs:45`
  - body: `ReconnectAsync` disposes the active client and raises `OperatorClientChanged` with `NewClient=null` before creating the replacement. If this page initializes or remains open while `CurrentApp.GatewayClient` is null or old, `AttachClient` subscribes to that null/old instance only, so the new client's `Connected` event never reloads the Usage data. Reattach through the existing client-changed/connection-manager signal, or otherwise refresh the tracked client when the app's operator client changes.
  - confidence: 0.83
