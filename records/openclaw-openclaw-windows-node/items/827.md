## Label Justifications

- `P1`: The PR touches MCP/node connection startup and currently would leave local MCP unavailable for active-gateway MCP-only users after restart.
- `merge-risk: 🚨 compatibility`: Existing users with saved active gateways and MCP-only mode can lose working local MCP behavior after upgrading to this patch.

## Summary

Keep open: the direction is useful, but the patch is not merge-ready because source inspection shows MCP-only startup can be skipped after a successful operator connection, and the PR lacks real after-change behavior proof.

## What This Changes

The branch separates gateway-node connection gating from NodeService/MCP initialization, adds MCP-only and connecting node UI states with tests/localization, and adds proof-validation docs, a repo-local skill, and a PR template.

## Best Possible Solution

Keep the separate gateway-node gate, but explicitly start local-only NodeService/MCP whenever MCP is enabled and gateway node mode is off, preserve truthful per-capability UI gating, and then add real UI/MCP proof.

## Reproduction Assessment

Yes from source inspection: with an active gateway record, an operator credential, `EnableMcpServer=true`, and `EnableNodeMode=false`, `TryConnectGatewayIfCredentialAvailable()` returns true and skips `TryStartLocalMcpOnlyNode()`. Since the NodeService constructor alone does not register capabilities or start MCP, the local MCP server remains unavailable.

## Solution Assessment

No. Separating gateway-node connection from NodeService initialization is the right shape, but the PR must preserve local-only MCP startup after successful operator connections and avoid enabling controls for capabilities that MCP-only cannot serve.
