## Summary

Keep open: the current PR still leaves a supported MCP-only startup path broken for users with an active credentialed gateway, exposes a gateway-only toggle in MCP-only UI, and has no real behavior proof yet.

## What This Changes

The branch separates gateway-node gating from NodeService initialization, adds MCP-only and connecting node states in Connection and Permissions UI with localization, and adds source-level tray tests for those states.

## Best Possible Solution

Keep the separate gateway-node gate, explicitly start local-only NodeService/MCP whenever MCP is enabled and node mode is off, keep browser.proxy UI scoped to gateway-node mode, and require redacted live MCP/UI proof.

## Reproduction Assessment

Yes from source inspection: with an active gateway record, an operator credential, EnableMcpServer=true, and EnableNodeMode=false, the successful operator-connect path returns true and skips TryStartLocalMcpOnlyNode(). Since NodeService construction alone does not register capabilities or start MCP, local MCP remains down after restart.

## Solution Assessment

No. Separating gateway-node connection from NodeService initialization is the right direction, but the PR must preserve local-only MCP startup after successful operator connections and avoid enabling controls for capabilities MCP-only cannot serve.

Keep Browser proxy disabled or clearly scoped unless Node Mode is on.
