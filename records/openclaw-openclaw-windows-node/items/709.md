## What This Changes

The issue reports that selecting the Teams SDK channel during companion app setup can stall the wizard, then retry/skip can hit a lingering “wizard already running” session state.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw-windows-node/pull/718 or an equivalent cancel-before-disconnect fix with real setup recovery proof, then close the duplicate https://github.com/openclaw/openclaw-windows-node/pull/717 and track any remaining Teams timeout separately.

## Reproduction Assessment

Yes at source level: a `wizard.next` timeout reaches `EnterWizardErrorAsync`, which disconnects before the existing cancel helper can send `wizard.cancel`. I did not run a live setup reproduction in this read-only review.

## Solution Assessment

Likely yes: reusing the existing best-effort `CancelCurrentSessionAsync()` before disconnect is the narrowest maintainable fix. The remaining merge decision belongs on the linked PR, where live recovery proof and required validation still matter.
