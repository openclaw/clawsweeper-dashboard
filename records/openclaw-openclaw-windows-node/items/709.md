Keep open: current main still has a plausible setup-wizard recovery bug where a `wizard.next` timeout can leave the gateway session active and make retry/skip hit the reported “wizard already running” state.

## What This Changes

The issue reports that selecting the MS Teams SDK channel during fresh onboarding leaves Continue disabled, times out, and then blocks retry or skip because a wizard session is already running.

## Best Possible Solution

Fix the setup UI so timeout/error recovery reliably sends or preserves a `wizard.cancel` for the active session before disconnecting, then prove the Teams SDK channel flow can retry or skip without a stuck gateway session.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here: a `wizard.next` timeout throws, the UI disconnects before canceling the active session, and later recovery actions have no client left to send `wizard.cancel`. The report supplies the concrete Teams channel path and screenshot.

## Solution Assessment

Yes, the narrowest maintainable fix is in the Windows setup wizard recovery path: preserve or reacquire enough connection state to cancel the active session before retry/skip. If Teams SDK still hangs after that, gateway-side logs should drive a separate gateway/plugin fix.
