## Summary

Keep this issue open: current main and v0.6.0 strip NUL/BOM characters, but the parser still only accepts `WSL version:` with whitespace, while the reported sample includes the hyphenated `WSL-Version:` form that remains unparsable.

## What This Changes

The issue reports that the v0.6.0 installer still fails `preflight-wsl` version parsing on UTF-16-style `wsl.exe --version` output.

## Best Possible Solution

Land a narrow setup-engine parser fix with regression coverage for NUL-separated and hyphenated WSL version output while preserving the existing minimum-version and environment-diagnostic behavior.

## Reproduction Assessment

Yes from source inspection, though not by running the Windows installer here. After current normalization, the reported `WSL-Version:` label still cannot match the parser's `WSL\s+version:` regex, so `preflight-wsl` can still reach the reported terminal failure.

## Solution Assessment

Yes, the best path is a narrow parser/test repair. It should accept the observed hyphenated/NUL-separated WSL label without changing setup policy, version thresholds, or unrelated WSL diagnostics.
