## Summary

Keep this PR open: the direct-argv direction is useful and has real terminal proof, but the patch currently builds an execution payload that can diverge from the command the resolver evaluated for transparent env wrappers.

## What This Changes

The PR adds an approved execution payload to V2 allow results, adds a CommandRequest.Argv direct execution mode to LocalCommandRunner, and adds shared tests for payload and direct-argv behavior.

## Best Possible Solution

Carry the effective argv used for executable resolution into the approved execution payload, preserve fail-closed behavior for env modifiers, and add focused regression coverage before merging or wiring this path into production.

## Reproduction Assessment

Yes for the patch defect: source inspection shows the resolver unwraps transparent env prefixes while the new payload builder preserves the original argv tail, so the two paths can diverge.

## Solution Assessment

No. The maintainable fix is to build the payload from the same effective argv that resolution evaluated, not by mixing the resolved executable with the original argv tail.
