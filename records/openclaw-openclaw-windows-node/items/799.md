## Summary

Keep open: the latest head fixes the prior approval-store ordering issue and has credible local argv proof, but it still allows unrepresentable batch payloads past approval and leaves the active MXC runner unable to honor or reject the new Argv contract.

## What This Changes

The PR adds an approved execution payload to exec-approvals V2, a CommandRequest.Argv direct launch mode in LocalCommandRunner, and shared tests for argv fidelity and fail-closed behavior.

## Best Possible Solution

Land the shell-free approved argv path only after all unrepresentable payloads fail closed before approval writes and every active ICommandRunner either honors Argv or rejects it explicitly.

## Reproduction Assessment

Yes. Source inspection of PR head shows batch payloads are built before the runner rejects them, and current main shows MxcCommandRunner serializes only legacy Command/Args fields while NodeService uses it for system.run.

## Solution Assessment

No. The direct-argv approach is appropriate, but this branch should first fail closed before store writes for all unrepresentable payloads and keep every active ICommandRunner from silently ignoring Argv.
