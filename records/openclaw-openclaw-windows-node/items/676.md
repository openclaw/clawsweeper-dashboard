## Summary

Keep this PR open: it contains meaningful native chat fixes, but the new exec approval RPC path still clears the pending approval banner after the frame is sent rather than after the gateway confirms success, so gateway `ok:false` responses can leave the user without a retry path.

## What This Changes

The branch changes native chat timeline reconciliation, reasoning boundaries, top-level exec approval event translation, approval response routing, trace logging, and related shared/tray tests.

## Best Possible Solution

Keep the PR open until the approval RPC awaits and validates the gateway response before clearing UI, has regression coverage for rejected resolve responses, and includes redacted live proof of native bubble rendering plus approval click behavior.

## Reproduction Assessment

Yes from source inspection: the new approval method returns after sending the request, while generic gateway `ok:false` responses are handled later outside the awaited provider path. I did not run the Windows tray live because this review is read-only.

## Solution Assessment

No. The overall direction is right, but the approval resolve path must be request/response-gated so UI clearing reflects a confirmed gateway success rather than a sent frame.
