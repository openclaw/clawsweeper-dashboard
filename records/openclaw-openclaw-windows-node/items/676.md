## Summary

Keep open: current main does not have this native chat RPC/timeline work, but PR head still clears the approval banner after a send-only exec.approval.resolve call, so an ok:false response can leave native chat without retry UI; the posted screenshots are useful partial proof only.

## What This Changes

The branch changes native chat timeline reconciliation, reasoning boundaries, top-level exec approval event translation/response routing, trace logging, and related shared/tray tests.

## Best Possible Solution

Keep the native timeline and approval direction, but make exec.approval.resolve await and interpret the gateway response, preserve the banner on ok:false, timeout, or disconnect, and require redacted proof for both success and failure/retry approval flows.

## Reproduction Assessment

Yes for the review blocker from source inspection: ResolveExecApprovalAsync returns after SendTrackedRequestAsync sends the frame, while ok:false responses are handled later without completing that task. A focused regression test can drive an exec.approval.resolve ok:false response and assert the pending banner remains.

## Solution Assessment

No. The PR is changing the right native chat and gateway surfaces, but exec.approval.resolve needs a response-awaiting path rather than the fire-and-log tracked request helper.

- **[P1] Wait for approval resolve responses before clearing UI:** `src/OpenClaw.Shared/OpenClawGatewayClient.cs:427`
  - body: `ResolveExecApprovalAsync` calls `SendTrackedRequestAsync`, but that helper only sends the frame and returns; an `ok:false` response is handled later by `HandleResponse`/`HandleRequestError` without completing this task. Please send this RPC through a response-awaiting path and add a regression test for `ok:false`.
