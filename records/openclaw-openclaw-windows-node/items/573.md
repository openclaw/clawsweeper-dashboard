## Summary

Keep open: the wrapper work is useful, but this branch changes a recently documented manifest-activation default, adds an update-channel option that current code does not consume, and lacks real launch proof beyond builds, tests, and dry runs.

## What This Changes

This PR changes `run-app-local.ps1` to default to direct unpackaged WinUI EXE launch, adds isolated data/update/dry-run/wait options, and updates README/build output launch guidance.

## Best Possible Solution

Land a wrapper that either preserves manifest activation as the default or records an explicit maintainer decision to change it, wires or removes update-channel behavior, and includes redacted live launch proof for the intended modes.

## Reproduction Assessment

Yes. Source inspection shows the PR sets `OPENCLAW_UPDATE_CHANNEL`, while current main has no reader for that variable and uses a fixed `UpdatumManager` update path.

## Solution Assessment

No, not as submitted. The isolated/direct wrapper idea is maintainable, but the PR mixes a default validation-policy change with an unimplemented update-channel flag and no live launch proof.
