## Summary

Keep open: the PR is useful and current main does not implement native chat approval responses, but the branch still has a P1 approval-lifecycle bug where the UI records Allowed/Denied after `chat.send` succeeds rather than after the matched gateway terminal approval event.

## What This Changes

## Best Possible Solution

Keep the inline timeline approval UX, but stamp the decision only after a matched terminal gateway approval event and preserve a retry or failure state when resolution is missing or rejected.

## Reproduction Assessment

## Solution Assessment

No, not yet: the in-bubble approval direction is maintainable, but the decision state should wait for the matched terminal gateway event instead of relying on optimistic local send success.

Full review comments:

- **[P1] Wait for gateway resolution before stamping approvals:** `src/OpenClaw.Tray.WinUI/Chat/OpenClawChatDataProvider.cs:898-899`
