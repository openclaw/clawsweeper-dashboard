## Summary

This PR should stay open: the inline approval UI is useful and now has real screenshots/logs, but the branch still treats a local Allow/Deny send acknowledgement as the final displayed decision before the gateway emits the matching terminal approval event. That is the same command-execution security boundary called out in the prior review, and it is not safe to merge until the timeline state reflects gateway-confirmed resolution or an explicit failure/timeout state.

## What This Changes

The PR moves exec approval prompts into native chat timeline permission entries, routes Allow/Deny through approval slash commands, adds seam repair/bootstrap hardening, updates setup badge contrast and localized strings, and extends tray tests.

## Best Possible Solution

Keep the inline timeline approval UI, but make the displayed terminal decision come from a matched gateway terminal event or a distinct failed/expired state, with regression coverage for missing, mismatched, rejected, and timed-out approval resolutions.

## Reproduction Assessment

Yes for the review concern: source inspection shows RespondToPermissionAsync stamps Allowed/Denied after the slash-command send returns but before a matched gateway terminal event. ResolvePermission then refuses to revise an already-decided entry on later gateway backstop events.

## Solution Assessment

No: rendering approval requests in the timeline is a good direction, but the terminal decision should be gateway-confirmed or explicitly failed rather than locally final. The narrower safe fix is to preserve the inline UI while separating clicked/sent, confirmed, denied, expired, and failed states.
