## Summary

Keep open: current `master` only partially covers this by rendering URLs/codes that are already present in wizard step text, while hidden WSL command output is still captured off-screen and the restart escape hatch appears only after an error.

## What This Changes

The issue asks for the configuration wizard to surface browser-login URLs and required device codes from the hidden terminal auth flow, plus a visible way to start over when the flow gets stuck.

## Best Possible Solution

Expose provider-auth URLs and device codes through wizard data that WinUI can render, and add a user-visible cancel/restart path during long auth waits.

## Reproduction Assessment

Yes, from source inspection rather than a live WSL/Codex login run: hidden WSL command output is captured off-screen, and the WinUI wizard only renders links/codes already present in step messages. The always-available start-over safety valve requested in the report is also absent during normal auth waits.

## Solution Assessment

Unclear. A UI-only start-over button is straightforward, but the auth-link fix should probably come from gateway wizard output/metadata rather than ad hoc scraping unless maintainers explicitly choose that boundary.
