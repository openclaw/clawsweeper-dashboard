## Summary

Keep open: the branch is now retargeted to main, clean, and includes the requested shutdown nulling plus video proof, but the refactor still queues a delegate bound to the coordinator instance, so already queued work can outlive App's nulled field and touch the disposed tray icon.

## What This Changes

The PR extracts tray icon update and tooltip logic from App.xaml.cs into a new TrayIconCoordinator and adds a source-level delegation contract test.

## Best Possible Solution

Keep the narrow extraction, but make marshalled tray updates re-enter an App-owned nullable/liveness-checked path or add an equivalent disposed guard before any TrayIcon access.

## Reproduction Assessment

Yes from source inspection: an off-UI-thread tray update queues TrayIconCoordinator.UpdateTrayIcon, and if shutdown disposes the tray icon before the dispatcher runs it, the queued delegate bypasses App's now-null coordinator field. I did not run a live WinUI shutdown race in this read-only review.

## Solution Assessment

No; clearing App._trayIconCoordinator is useful but not sufficient because already queued coordinator-bound delegates survive that assignment. The narrow repair is to queue an App-owned/liveness-checked callback or add an equivalent guard before using the TrayIcon.
