## Summary

Current main and v0.6.3 still encode the reported bootstrap-to-node role mapping, so the issue remains necessary. The open collaborator draft https://github.com/openclaw/openclaw-windows-node/pull/741 now has local setup-code proof and appears to own the repair, but it is unmerged and explicitly leaves remote/external validation outstanding.

## What This Changes

Fix QR/setup-code bootstrap pairing so the Windows tray operator connection uses operator role, durably stores returned role tokens, and reconnects without reusing bootstrap credentials.

## Best Possible Solution

Land the canonical connection fix after proving fresh local and remote QR/setup-code pairing plus upgrade/reconnect behavior, while preserving device-token-first precedence and separate operator/node lifecycles.

## Reproduction Assessment

Yes at source level with high confidence: current main deterministically creates a bootstrap-backed tray operator client with node pairing enabled, and its focused test asserts the resulting node role. The reporter supplies a concrete live Windows reproduction, but this read-only review did not independently run it.

## Solution Assessment

Yes in the direction taken by https://github.com/openclaw/openclaw-windows-node/pull/741: correcting operator role selection together with durable multi-role token persistence and safe bootstrap cleanup is more complete than the closed factory-only patch. Merge should wait for the remaining remote and upgrade proof.
