## Label Justifications

- `P1`: The documented fresh QR/setup-code pairing path can leave real users unable to use the tray's primary chat workflow.
- `impact:auth-provider`: Bootstrap credential role selection and durable device-token handoff determine whether the tray is authorized as operator or node.
- `impact:message-loss`: The incorrect node role causes user chat sends to be rejected rather than delivered.

## Summary

Current `main` still routes bootstrap-backed operator connections through node-role pairing, and the focused factory test still asserts `node`, so the reported chat failure remains necessary to track. The open collaborator draft https://github.com/openclaw/openclaw-windows-node/pull/741 appears to own the repair, but it has not merged and still needs maintainer handling.

## What This Changes

Fix QR/setup-code pairing so the tray operator connection authenticates as `operator` and transitions safely from bootstrap credentials to durable per-role device tokens.

## Best Possible Solution

Land a reviewed, upgrade-safe connection fix that keeps operator and node WebSocket lifecycles separate, persists every returned role device token before clearing bootstrap credentials, and covers both fresh QR pairing and subsequent reconnect recovery.

## Reproduction Assessment

Yes, from source with high confidence: current `main` constructs a bootstrap-backed operator lifecycle with node pairing enabled, and its focused test asserts the resulting role is `node`. The reporter also provides concrete fresh-install QR pairing steps and the resulting authorization error.

## Solution Assessment

Yes in direction, but a factory-only flag flip is not sufficient evidence for an upgrade-safe solution. The best repair should also persist returned operator and node tokens durably and clear the bootstrap credential only after the required tokens can be recovered.
