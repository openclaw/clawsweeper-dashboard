## Label Justifications

- `P1`: The reported behavior blocked the primary chat workflow after QR/setup-code pairing.
- `impact:auth-provider`: The bug was caused by bootstrap credential role selection and durable device-token handoff.

## Summary

Current main already fixes the reported QR/setup-code role lock: the operator client now pairs as `operator`, role handoff tokens are persisted, and post-bootstrap reconnect uses the durable operator credential. The fix is proven on main by the merged connection-hardening PR and real setup proof, but it is not in latest release v0.6.3.

## What This Changes

The issue reports that QR-code/bootstrap pairing caused the tray operator connection to remain in node role, breaking chat authorization after pairing.

## Best Possible Solution

Keep the #741 current-main implementation for this specific node-role loop; broader local/remote pairing reliability should remain tracked by the separate reliability issues.

## Reproduction Assessment

Yes, source/history provide a high-confidence reproduction of the old behavior: v0.6.3 passed bootstrap credentials into `bootstrapPairAsNode` and its focused test expected `node`. I did not run the Windows QR flow during this read-only review, but the discussion includes post-fix real setup proof.

## Solution Assessment

Yes; the current fix is the narrow maintainable direction for this bug because it separates the operator and node lifecycles, keeps setup-code operator connects as operator, persists role tokens, and reconnects with durable credentials.
