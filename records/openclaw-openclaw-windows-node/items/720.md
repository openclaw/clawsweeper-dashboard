## Label Justifications

- `P1`: QR/setup-code pairing can leave real users with a connected tray whose chat sends are rejected immediately after setup.
- `impact:auth-provider`: The issue is about bootstrap credentials, gateway role authorization, and durable operator device-token persistence.
- `impact:message-loss`: The user-visible failure is `chat.send` being rejected as unauthorized after pairing, so chat messages cannot be delivered.

## Summary

Keep open: current main still routes setup-code bootstrap credentials through the operator factory as node-pair handoff, and current tests still assert that bootstrap operator-factory clients connect as `node`. The open draft https://github.com/openclaw/openclaw-windows-node/pull/741 appears to cover this, but it has not merged, so the issue still owns real user-facing work.

## What This Changes

The issue requests that QR/setup-code pairing leave the tray operator client paired with a durable operator credential instead of permanently connecting as gateway role `node`.

## Best Possible Solution

Land a reviewed fix that makes setup-code operator connects use `role=operator`, persists returned operator and node role tokens durably, and preserves the separate explicit node-mode handoff path.

## Reproduction Assessment

Yes, from source inspection rather than a live Windows/gateway run. The setup-code path stores a bootstrap token and clears device tokens, the factory maps bootstrap credentials to node handoff, and the current factory test asserts `node` for that path.

## Solution Assessment

Yes, the narrow maintainable fix is to decouple the operator factory role from bootstrap credential type and persist returned role tokens without disturbing the explicit node-client path. The broad draft PR may be acceptable after review, but a smaller extraction would also solve this issue.
