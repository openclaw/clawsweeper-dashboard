## Label Justifications

- `P1`: QR pairing can leave the tray unable to send chat through the operator workflow, which breaks a core agent/channel path for affected users.
- `impact:auth-provider`: The reported failure is caused by credential source, role selection, and device-token persistence across gateway pairing.
- `impact:message-loss`: Chat messages are rejected by `chat.send` after QR pairing because the active connection is authenticated as the wrong role.

## Summary

This still looks necessary: current main still maps any setup-code bootstrap credential into a node-role operator client, and the existing handoff tests explicitly cover the no-operator-token path that matches the reported chat authorization failure.

## What This Changes

The issue asks for QR/setup-code pairing to produce a usable operator-role tray connection and durable operator device token instead of leaving chat authenticated as role node.

## Best Possible Solution

Make setup-code bootstrap credentials establish the tray operator connection first, persist the operator device token for chat, then let the separate node connection consume the same bootstrap flow when Node Mode is enabled.

## Reproduction Assessment

Yes for source-level reproduction, not for a live Windows run: current main constructs bootstrap setup-code clients with `bootstrapPairAsNode` and has tests asserting the resulting connect role is `node` with no operator token after a node-role hello-ok.

## Solution Assessment

Yes: the narrow repair is to decouple first-contact bootstrap credentials from the operator client's role while preserving the separate node bootstrap path and adding regression coverage for both operator chat and Node Mode.

## Repair Work Prompt

Fix QR/setup-code pairing so a fresh tray bootstrap credential creates a usable operator connection for chat instead of authenticating the `OpenClawGatewayClient` as role `node` solely because the credential is a bootstrap token. Preserve the separate Windows node flow: setup codes should still be stored as `GatewayRecord.BootstrapToken`, operator device tokens must win for chat reconnects, Node Mode should still connect through `WindowsNodeClient`/`NodeConnector`, and bootstrap tokens should not be cleared before the roles that still need them have durable device tokens. Update the tests that currently assert bootstrap credentials pair as node, add regression coverage for setup-code -> operator role/device token, and keep connection ownership in `GatewayConnectionManager`; do not reintroduce legacy `SettingsData.Token`/`BootstrapToken` writes.
