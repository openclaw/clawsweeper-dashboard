## Summary

Keep open: current main and the latest release only model SSH user, host, remote gateway port, and local forward port, and the generated `ssh` arguments have no way to include `-p` for a non-default SSH daemon port. The related search found this issue as the only matching tracker, so there is no implemented or canonical replacement yet.

## What This Changes

Request separate persisted SSH server port support for managed SSH tunnels, defaulting to 22 and emitted in generated `ssh` commands as `-p <port>`.

## Best Possible Solution

Add a distinct SSH server port field defaulting to 22 across the tunnel model, settings/registry persistence, Connection UI, command builder, diagnostics/previews, and regression tests while preserving `RemotePort` as the forwarded gateway port.

## Reproduction Assessment

Yes. Source inspection of current main shows no SSH server-port field in the tunnel config/settings and no `-p` argument in `SshTunnelCommandLine.BuildArguments`, so a managed tunnel can only use the SSH client default port.

## Solution Assessment

Yes. A separate default-22 SSH server port is the narrow maintainable fix; overloading `RemotePort` would conflict with its existing meaning as the remote gateway port forwarded by `-L`.
