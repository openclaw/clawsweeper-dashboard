## Summary

This should stay open: current main still has a source-level UI-thread blocking path immediately after the install button, and the reporter's screenshot matches the observable symptom. The actual setup pipeline is already dispatched off the UI thread, so the likely narrow bug is the pre-install replacement detection before the progress screen appears.

## What This Changes

The issue reports that clicking Install new WSL Gateway can freeze the setup window and asks for background work with visible progress.

## Best Possible Solution

Make the welcome/install click path non-blocking and timeout-safe before the replacement dialog or progress transition, while preserving the existing registry and distro replacement semantics.

## Reproduction Assessment

Yes for source-level reproduction: current main calls ExistingConfigDetector.Detect on the install button path, and that detector synchronously waits on wsl.exe output before progress UI is shown. I did not run a live Windows WinUI/WSL repro in this read-only environment.

## Solution Assessment

Yes: the narrow maintainable fix is to make the pre-install detection async, cancellable, and timeout-safe with a visible busy state. The setup pipeline itself already runs off the UI thread, so a broad pipeline rewrite is not indicated.
