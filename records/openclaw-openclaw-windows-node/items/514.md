## Label Justifications

- `P2`: This is a normal-priority user-facing Sessions/Chat improvement with a bounded but merge-blocking deep-link correctness issue.
- `merge-risk: 🚨 message-delivery`: If Chat remains on the wrong selected thread after Open chat, a user can send the next message to a different session than the one they clicked.
- `merge-risk: 🚨 session-state`: The PR adds cross-page selected-session state and can leave that state stale relative to OpenClawChatRoot's internal selection.

## Summary

Keep open: the PR is a useful Sessions-page direction, but the new Chat deep-link can reuse an already-mounted chat root while leaving the internally selected thread unchanged, and the external PR still lacks real behavior proof for the visual flow.

## What This Changes

The PR redesigns the WinUI Sessions page, hides cron sessions, moves session actions into a flyout, and adds a pending-session handoff so Open chat navigates to Chat with an initial thread.

## Best Possible Solution

Honor every Sessions Open chat request with a deterministic thread switch or remount, then require redacted UI proof before merging the redesign.

## Reproduction Assessment

Yes, from source inspection: mount Chat with session A, switch to B using the composer dropdown, return to Sessions, and click Open chat for A. ChatPage compares the pending key to _mountedThreadId, reuses the host, and OpenClawChatRoot's internal selectedIdState remains B.

## Solution Assessment

No, not yet: the Sessions handoff should always force the requested thread to become selected, either by remounting on every pending key or by adding a real switch-thread API/current-selection callback. The design also needs real UI proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Remount when Sessions requests a thread:** `src/OpenClaw.Tray.WinUI/Pages/ChatPage.xaml.cs:225-230`
  - body: `_mountedThreadId` only records the thread id used when the chat root was mounted; it is not updated when the user changes sessions inside the composer dropdown. If the host was mounted for session A, the user switches to B, then returns to Sessions and clicks Open chat for A, `threadChanged` is false and this branch reuses the existing host, leaving B selected. Treat any pending session key as requiring a switch/remount, or keep ChatPage synchronized with the root's live selection.
