## Label Justifications

- `merge-risk: 🚨 message-delivery`: The new Open chat path can leave Chat selected on a different thread than the clicked Sessions row, so a user may send to the wrong session.
- `merge-risk: 🚨 session-state`: The PR adds a cross-page pending-session state handoff whose sentinel can drift from OpenClawChatRoot's internal selected state.

## Summary

Keep open: the Sessions redesign has useful UI work and a screenshot of the list, but the new Open chat handoff has a source-reproducible wrong-thread edge case and the proof does not show the Chat selected-thread behavior after clicking through.

## What This Changes

The branch redesigns SessionsPage with Fluent-style layout/actions, filters cron sessions, and adds a pending session key handoff so an Open chat button navigates to Chat with an initial thread.

## Best Possible Solution

Make every Sessions Open chat request deterministically select the requested thread, either by remounting on every pending handoff or by adding a live thread-selection API, then provide redacted UI proof of the handoff.

## Reproduction Assessment

Yes. From source inspection: mount Chat for session A, switch to B inside the composer dropdown, return to Sessions, and click Open chat for A; `_mountedThreadId` still equals A, so the PR skips remount and leaves the chat root selected on B.

## Solution Assessment

No. The current `_mountedThreadId` check is not the best fix because it cannot observe internal Chat dropdown changes; a pending Sessions handoff should force selection/remount or use an explicit live thread-switch API.
