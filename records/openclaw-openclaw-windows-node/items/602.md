## Summary

Keep this PR open: the shared-token direction is useful, but the current patch still removes the documented and tested token-absent browser.proxy path and needs maintainer review before that compatibility/auth-provider contract changes.

## What This Changes

The PR changes browser.proxy registration to use SharedGatewayToken and suppresses the capability when it is missing, while also adding chat session-key routing through notifications and toasts.

## Best Possible Solution

Use SharedGatewayToken for browser-control auth when available, preserve the existing token-absent browser.proxy path with visible Settings guidance, and validate both fresh and upgrade behaviors unless maintainers explicitly approve a fail-closed contract.

## Reproduction Assessment

Yes for the shared-token problem, by source inspection and contributor terminal proof: current main passes the node token into BrowserProxyCapability and the PR proof shows the auth error changing after the shared-token path is used. The no-shared-token upgrade path is not proven.

## Solution Assessment

No, not as-is. Passing SharedGatewayToken when present is the right direction, but suppressing browser.proxy when the shared token is missing is broader than the tested and documented current behavior.
