## Summary

Keep this PR open: the shared-token direction is valid, but the current patch removes tested token-absent browser.proxy behavior, hides the existing Settings guidance path, and still lacks no-shared-token live proof plus required tray validation.

## What This Changes

The PR adds a shared gateway token resolver for browser.proxy registration and also threads chat session keys through notification/toast chat routing.

## Best Possible Solution

Use the shared gateway token for browser-control auth when present, preserve the tested token-absent/no-auth-compatible browser.proxy path with visible Settings guidance, and validate both upgrade paths.

## Reproduction Assessment

Yes, with source and contributor proof rather than a local live run. Current main passes the node bearer token into BrowserProxyCapability, and the contributor's terminal output shows the auth rejection changing once the PR uses a shared token.

## Solution Assessment

No, not as-is. Passing the shared token when present is the right direction, but suppressing browser.proxy when the shared token is missing is broader than the current tested/docs-backed behavior; preserving the token-absent path with guidance is safer.
