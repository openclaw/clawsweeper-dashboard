## Summary

Keep open: the PR targets a real source-level browser.proxy auth mismatch, but the branch still changes no-shared-token compatibility and diagnostics without live behavior proof or the repo-required tray test validation.

## What This Changes

The PR wires a shared gateway token resolver into NodeService, registers browser.proxy only when that token is present, and updates Command Center browser auth guidance.

## Best Possible Solution

Land a narrow fix that uses SharedGatewayToken for browser-control auth while preserving or explicitly documenting and diagnosing no-shared-token behavior, with tests and live proof for both token-present and token-absent paths.

## Reproduction Assessment

Yes at source level: current main passes the node bearer token into BrowserProxyCapability while the credential architecture separates node device credentials from shared HTTP tokens. I did not see live after-fix output proving the browser.proxy route in a real setup.

## Solution Assessment

No, not yet: passing the shared token when present matches the credential contract, but suppressing the command entirely when the token is absent needs compatibility handling, diagnostics, and proof before it is the narrowest maintainable fix.
