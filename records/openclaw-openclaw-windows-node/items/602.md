## What This Changes

The branch changes Windows Node capability registration so `browser.proxy` is registered only when `GatewayRegistry.GetActive().SharedGatewayToken` is available, and passes that shared token into `BrowserProxyCapability`.

## Best Possible Solution

Use the shared gateway token for browser-control auth while keeping no-shared-token behavior and Command Center diagnostics explicit, then merge after live browser.proxy proof and the repo-required validation set.

## Reproduction Assessment

Yes from source, but not live-reproduced in this review. Current master passes the node/client bearer token into BrowserProxyCapability while the docs say browser-control HTTP auth expects matching shared-token-style auth.

## Solution Assessment

Unclear as submitted. Passing the shared token is the narrow fix, but hiding the capability whenever the shared token is absent needs diagnostic updates, upgrade proof, and maintainer confirmation of the intended fallback behavior.
