## Label Justifications

- `P1`: This PR targets a broken browser.proxy auth workflow and currently introduces a blocking compatibility regression for token-absent setups.
- `merge-risk: 🚨 compatibility`: Merging as-is can stop existing QR/bootstrap or no-secret-compatible setups from advertising browser.proxy at all.
- `merge-risk: 🚨 auth-provider`: The diff changes which credential source is used for browser-control auth and when the capability is exposed.

## Summary

Keep open: the PR fixes the credential source for the shared-token path, but it still fail-closes browser.proxy whenever no shared token is saved, which current code, tests, and docs treat as a supported and diagnosable path. The contributor added useful live terminal proof for the token-present path, but token-absent behavior and AGENTS-required tray validation remain unresolved.

## What This Changes

The PR wires a shared gateway token resolver into NodeService, registers browser.proxy only when that token is present, passes that token to BrowserProxyCapability, and updates Command Center browser auth copy.

## Best Possible Solution

Use the shared gateway token for browser-control auth when present, preserve an advertised token-absent path with actionable Settings guidance or no-auth-compatible host support, and validate both token-present and token-absent upgrade behavior.

## Reproduction Assessment

Yes. Source inspection shows current main passes the node bearer token from AttachClient into BrowserProxyCapability, and the contributor's terminal output shows the before-fix auth rejection changing after using a shared token.

## Solution Assessment

No, not as-is. Passing the shared token is the right fix direction, but suppressing browser.proxy when the shared token is missing is broader than the current tested/docs-backed behavior and needs a preserved fallback or explicit maintainer-approved fail-closed path.
