## Summary

Keep this PR open. The MSIX hardening direction is useful, but the branch is not merge-ready because MSIX-only releases still allow missing MSIX artifacts, the documented recovery CLI is not actually shipped through MSIX yet, and the branch now conflicts with current main in core update/uninstall paths.

## What This Changes

The PR moves Windows distribution toward MSIX-only by changing manifests, startup/update permission handling, release/AppInstaller workflows, uninstall recovery CLI/docs, and related tests/runbooks while removing Inno, Updatum, and portable paths.

## Best Possible Solution

Land the MSIX-only transition only after the branch is rebased, MSIX artifacts are mandatory release outputs, the recovery path is actually shipped or documented as manual PowerShell, and release/feed proof covers clean install plus upgrade for both architectures.

## Reproduction Assessment

Yes for the review findings: source inspection shows the release job can publish only MSIX/AppInstaller assets while build-msix remains optional, and the docs promote a CLI command before the MSIX packaging note ships the alias. I did not run build/tests because this is a read-only review.

## Solution Assessment

No. The MSIX-only direction may be right, but this PR is not the safest implementation until release gating, recovery distribution, current-main conflicts, and hosting proof are resolved.
