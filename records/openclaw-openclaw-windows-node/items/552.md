## Label Justifications

- `P2`: The issue identifies a real process-availability hazard across user-facing WinUI code, but it does not include a current crash loop or urgent live regression trace.
- `impact:crash-loop`: The reported failure mode is unhandled async handler exceptions terminating the tray process.

## Summary

This issue is still necessary: current main still has a broad unguarded `async void` surface in WinUI and connection code, and the repo already contains a local comment acknowledging that this exact pattern can tear down the process.

## What This Changes

The issue asks for a centralized guarded fire-and-forget helper and refactors so required `async void` event handlers delegate to guarded async work.

## Best Possible Solution

Add a shared guarded async-dispatch helper accessible from both Tray.WinUI and OpenClaw.Connection, refactor required `async void` entry points into minimal delegating wrappers, and add a focused regression guard so new unguarded handlers are caught in tests.

## Reproduction Assessment

Source-level yes, runtime no: current main still has dozens of `async void` handlers, including connection handlers that await before any catch, and PermissionsPage documents the same process-crash risk. I did not execute a Windows crash harness in this read-only review.

## Solution Assessment

Yes: a centralized guarded helper plus one-line event-handler wrappers is the narrowest maintainable direction, provided the refactor preserves UI-thread behavior and connection-manager ownership.
