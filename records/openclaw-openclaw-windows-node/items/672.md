## Summary

Keep open: current main still has a plausible native Companion chat rendering/reconciliation gap where a failed tool row can remain the terminal visible row even after the assistant response exists, and I found no linked fix or canonical replacement in the provided context.

## What This Changes

The issue asks the Windows Companion native chat to update automatically from a failed tool event to the latest assistant response, matching the web UI without an app restart.

## Best Possible Solution

Keep the failed tool event as context, but make the native chat's final visible state and scroll target prioritize the completed assistant response, with regression coverage for failed-tool-plus-final-assistant live ordering and restart/history parity.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection gives a plausible current-main path: failed ToolCall rows render after assistant rows within a turn, so the bottom visible row can remain the failed tool event even after the assistant response exists.

## Solution Assessment

Yes, a narrow native chat rendering/reconciliation fix is the right direction. The transcript appears to exist in backend/history, so this should not require a backend migration or new configuration option.
