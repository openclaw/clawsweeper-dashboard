## Summary

Keep open: the PR is useful and the new body proof shows a real setup pass, but the patch is not merge-ready because its Progress/NonInteractive path sends `wizard.next` with only `sessionId`, while the gateway keeps returning the same current step until an answer envelope clears it.

## What This Changes

The PR adds shared gateway wizard step classification, URL/code formatting, auth-aware timeouts, progress/noninteractive handling, restart replay hardening, and focused setup-engine tests.

## Best Possible Solution

Keep the shared helper direction, but make drive-through steps acknowledge the current step with the gateway-compatible answer envelope or use a documented `wizard.status` polling path, then cover both callers with focused regression tests before maintainer review.

## Reproduction Assessment

Yes, source-level: upstream wizard.next only clears a current step when `params.answer` is present, while this PR sends only `sessionId` for Progress/NonInteractive paths. I did not run a live Windows setup in this read-only review.

## Solution Assessment

No. The shared-helper approach is maintainable, but the no-answer drive-through contract is not the narrowest safe implementation against the current gateway and macOS behavior.
