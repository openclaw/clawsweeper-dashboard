## Summary

Keep open: current main does not yet handle richer gateway wizard steps, but this PR is not merge-ready because its drive-through progress/action path sends `wizard.next` without an answer, which the upstream gateway treats as a status read and returns the same current step. Real behavior proof is still only helper-harness output.

## What This Changes

The branch adds shared wizard step classification, message formatting, auth timeouts, progress/noninteractive handling, restart replay hardening, and setup-engine tests for richer gateway wizard steps.

## Best Possible Solution

Acknowledge drive-through progress/action steps with a `wizard.next` answer envelope for the current step, or use `wizard.status` for true polling, then add regression coverage and redacted real setup proof before merge.

## Reproduction Assessment

Yes, source-level: the upstream gateway repeats the current step when `wizard.next` lacks an answer, while this PR sends only `sessionId` for Progress/NonInteractive paths. I did not run a live Windows gateway setup.

## Solution Assessment

No. The shared-helper direction is maintainable, but the implementation must follow the gateway answer contract and include real setup proof before it is the best merge path.
