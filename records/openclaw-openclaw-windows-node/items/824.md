## Summary

Keep open: the current PR head still has a source-confirmed wizard progression bug, because both setup callers can send `wizard.next` without an answer for drive-through steps even though the gateway repeats the current step until an answer clears it.

## What This Changes

The PR adds shared wizard classification, auth URL/code formatting, auth-aware timeouts, progress/action handling, raw option-value preservation, restart replay hardening, and setup-engine tests.

## Best Possible Solution

Keep the shared helper direction, but make drive-through steps acknowledge the current step with a gateway-compatible answer envelope or use a documented `wizard.status` polling contract, then cover both callers with focused regression tests.

## Reproduction Assessment

Yes, source-level: the PR sends sessionId-only `wizard.next` for drive-through steps, while the gateway only clears `currentStep` when `answer` is present. I did not run live Windows setup in this read-only review.

## Solution Assessment

No. The shared-helper approach is maintainable, but the no-answer drive-through contract is not safe against the current gateway behavior.
