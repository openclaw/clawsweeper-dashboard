## Summary

Keep open: the PR is useful setup wizard resilience work, but it still advances Progress/NonInteractive steps with a sessionId-only `wizard.next` call even though the gateway only clears a pending step when an answer envelope is provided.

## What This Changes

The PR adds shared wizard step classification, URL/code formatting, auth-aware timeouts, progress/action handling, raw option-value preservation, restart replay hardening, and focused setup-engine tests.

## Best Possible Solution

Keep the shared helper direction, but make drive-through steps acknowledge the current step with the gateway-compatible answer envelope or use a documented status-polling contract, then cover both callers with focused regression tests.

## Reproduction Assessment

Yes, source-level: upstream wizard.next only clears a current step when params.answer is present, while this PR sends only sessionId for Progress/NonInteractive paths. I did not run a live Windows setup in this read-only review.

## Solution Assessment

No. The shared-helper approach is maintainable, but the no-answer drive-through contract is not the narrowest safe implementation against the current gateway behavior.
