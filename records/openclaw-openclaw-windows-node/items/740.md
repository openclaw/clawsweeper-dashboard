## Label Justifications

- `P1`: The PR fixes a primary session workflow but currently introduces a path that can silently suppress valid post-reset chat traffic.
- `merge-risk: 🚨 message-delivery`: The reset gate intentionally drops chat and agent events and can misclassify a valid timestamp-less remote turn as stale.
- `merge-risk: 🚨 session-state`: The patch clears and reseeds timeline, history, run, session-ID, abort, attachment, and tool-metadata state.

## Summary

The PR fixes a real stale-timeline bug with strong visual proof, but its reset gate can suppress a valid first remote post-reset turn when chat events omit timestamps, a shape the existing provider already supports.

## What This Changes

The branch clears native Chat timelines after successful session resets and adds generation/run/timestamp fencing plus regression tests for stale history, events, sends, aborts, metadata, and session IDs.

## Best Possible Solution

Keep provider-owned generation fencing, but correlate fresh remote turns without requiring a positive timestamp—prefer a run/session generation signal or a narrowly bounded event-order rule—and add a regression test for a timestamp-less remote first turn.

## Reproduction Assessment

Yes. Current main has no chat-provider reset-completion handling, and the supplied screenshots directly show the populated timeline before reset and cleared state after the proposed fix; the blocking Ts == 0 path is also reproducible from the PR source and fake bridge.

## Solution Assessment

No, not yet. Provider-owned reset invalidation is the right boundary, but requiring a positive timestamp to recognize a fresh remote user turn is narrower than the existing best-effort timestamp contract and can lose messages.
