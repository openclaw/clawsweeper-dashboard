## Label Justifications

- `impact:data-loss`: An interrupted write can corrupt the only stored Ed25519 private key and device tokens for a paired gateway.

- `impact:auth-provider`: The bug affects device-token credential precedence and gateway pairing/re-enrollment behavior.

## Summary

Keep open: current main and v0.6.3 still rewrite the per-gateway device identity file non-atomically in `ClearStoredTokens`, while the shared identity code already documents and uses an atomic writer for this exact torn-write hazard.

## What This Changes

The issue asks for token-clearing and rollback writes of `device-key-ed25519.json` to use the existing atomic key-file write path so clearing tokens cannot corrupt or rotate device identity.

## Best Possible Solution

Reuse or expose the shared `DeviceIdentity` atomic key-file path for all token-clearing and rollback identity rewrites, with focused regression coverage that preserves identity fields, tokens cleared as intended, ACL behavior, and no leftover temp files.

## Reproduction Assessment

Yes for the source-level path, but not with a live crash harness: a paired identity can enter `ClearStoredTokens`, an interrupted `File.WriteAllBytes` can leave invalid JSON, and `LoadExisting()` then regenerates the keypair.

## Solution Assessment

Yes: routing the duplicate token-clear and restore writes through the existing atomic identity writer is the narrowest maintainable fix; fail-loud corrupt-file handling would be useful defense-in-depth but can be separate.
