## Summary

Keep open: the PR still addresses a real current-main MXC false-negative, but it is not merge-ready because live Windows behavior proof is missing and the SDK restore path remains version-blind after the 0.7.0/config-shape bump.

## What This Changes

The PR replaces the MXC hardcoded Windows build/UBR gate with `wxc-exec --probe`, bumps `@microsoft/mxc-sdk` to 0.7.0, renames the emitted config key to `processContainer`, and updates sandbox UI/cache/tests for probe outcomes.

## Best Possible Solution

Land the probe-based MXC fix after making the SDK restore/copy path version-aware and adding redacted live Windows proof for the probe plus contained execution or SandboxPage behavior.

## Reproduction Assessment

Yes from source inspection: current main rejects any Windows build other than 26300 before asking `wxc-exec`, so a supported newer build such as 26632 can be falsely treated as unsupported. I did not run a real Windows host in this read-only review.

## Solution Assessment

No as submitted: using `wxc-exec --probe` is the right narrow implementation direction, but the SDK restore path must be version-aware and the PR needs inspectable real behavior proof before merge.
