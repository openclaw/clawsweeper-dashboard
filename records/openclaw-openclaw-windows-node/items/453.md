## Label Justifications

- `P2`: This is a concrete user-facing chat attachment persistence bug with a clear but bounded chat surface impact.
- `impact:data-loss`: The reported behavior loses attachment metadata/previews across reload and restart even when chat history remains available.
- `impact:session-state`: The bug appears when rebuilding chat timelines from history after session switch, history reload, or app restart.

## Summary

Keep open. Current main still reconstructs attachments from marker text and a process-local image byte cache, while PR review and the issue discussion identify structured metadata persistence as the remaining blocker.

## What This Changes

Persist and rehydrate chat attachment metadata so attachment chips/previews survive history reloads, session switches, and app restarts without relying on in-band marker text.

## Best Possible Solution

Add structured attachment metadata persistence/rehydration in the chat model/provider path, render from that metadata, and keep marker text only as a migration/display fallback backed by stored metadata.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send an attachment, rebuild from chat.history after ImagePreviewCache is empty, and the timeline has only marker text with no structured attachment metadata to rehydrate from.

## Solution Assessment

Yes. A focused provider/model metadata sidecar, with tests for reload/restart and pasted-marker spoofing, is the narrowest maintainable fix until the gateway returns attachment metadata directly.
