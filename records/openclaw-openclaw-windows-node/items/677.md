- `P1`: Setup can still remove a successfully created local gateway after a late pairing/auth failure, blocking real users from completing onboarding.
- `impact:data-loss`: The issue is about rollback deleting the app-owned OpenClawGateway distro and generated local gateway state after setup has already created them.
- `impact:auth-provider`: The observed trigger is the pairing/auth path around device-pair and bootstrap/device credentials during setup.

## Summary

Keep open: current main has merged the narrow old-gateway device-pair diagnostic and now pins a newer gateway CLI, but the rollback/deletion behavior still exists whenever UI setup fails with rollback enabled.

## What This Changes

Issue reports that setup creates and starts the OpenClawGateway WSL distro/service, then after pair-node fails because device-pair is missing, rollback deletes the gateway and leaves its ports closed.

## Best Possible Solution

Maintain the newer gateway CLI and device-pair diagnostic, then make an explicit maintainer decision for UI rollback policy after late setup failures, with tests proving both cleanup and preserve/retry paths.

## Reproduction Assessment

Yes at source level: the provided logs show rollback after pair-node failure, and current main still executes failed-step and completed-step rollback whenever UI setup has RollbackOnFailure enabled. I did not run a live Windows/WSL setup reproduction in this read-only review.

## Solution Assessment

No for closing this as fixed: the merged diagnostic PR is a good narrow mitigation for old gateway CLIs, but it deliberately leaves the broader rollback/deletion policy tracked here.
