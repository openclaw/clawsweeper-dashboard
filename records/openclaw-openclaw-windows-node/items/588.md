## Summary

Keep open. Current main and v0.6.3 still lack the requested `pluginSurfaceUrls.canvas` plumbing, but this PR is not merge-ready because it trusts an unvalidated cap URL, does not fully propagate cap revocation to open canvas windows, and has no inspectable real behavior proof attached.

## What This Changes

The PR parses `pluginSurfaceUrls.canvas`, caches it on `WindowsNodeClient`, passes it through `NodeService`, and rewrites relative canvas WebView2 URLs through the cap-scoped surface base.

## Best Possible Solution

Validate the advertised canvas surface URL as an absolute http(s) cap route, preserve explicit removal signals, clear any open canvas window on revocation, add focused regression coverage, and require redacted runtime proof before merge.

## Reproduction Assessment

Source-reproducible, but not live-reproduced in this read-only review. Current main rewrites relative `/__openclaw__/canvas/*` paths to the bare gateway origin and ignores `pluginSurfaceUrls`, which matches the reported 401 when the gateway requires cap-scoped surface URLs.

## Solution Assessment

No for the current branch. Routing through the gateway-advertised cap URL is likely the right narrow fix, but this implementation needs URL validation and complete stale-cap clearing before it is safe to merge.
