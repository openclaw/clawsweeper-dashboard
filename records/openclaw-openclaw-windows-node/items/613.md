## Label Justifications

- `P1`: The PR can unintentionally expose gateway node capabilities for MCP-only users who have node mode disabled, which is an urgent pre-merge settings/security regression.
- `merge-risk: 🚨 compatibility`: Existing MCP-only settings semantics can change on upgrade when a persisted node token exists.
- `merge-risk: 🚨 security-boundary`: The diff can open a remote gateway node capability path even though the user selected local MCP-only mode.
- `merge-risk: 🚨 auth-provider`: The new startup path changes credential routing by using a persisted node device token when operator credentials are absent.

## Summary

Keep open: the PR fixes real restart/version/approval paths and now reports useful patched runtime proof, but the new startup branch can connect a gateway node in MCP-only mode because it gates node-only reconnect on a predicate that also includes `EnableMcpServer`. That would bypass the documented `EnableNodeMode=false` local-only behavior before merge.

## What This Changes

The branch replaces hardcoded handshake versions, adds persisted node-token startup reconnect, rewrites exec approval to a native Win32 prompt, updates the manifest, and adds regression tests.

## Best Possible Solution

Keep the node-only reconnect for `EnableNodeMode=true` paired nodes, leave `EnableMcpServer=true`/`EnableNodeMode=false` on the local MCP-only startup path, and add a regression test for that settings/token combination.

## Reproduction Assessment

Yes from source inspection: configure `EnableMcpServer=true`, `EnableNodeMode=false`, an active gateway record with a persisted node token, and no operator credential; the new branch will call `ConnectNodeOnlyAsync` and skip `TryStartLocalMcpOnlyNode()`. I did not run a Windows tray instance in this read-only review.

## Solution Assessment

No: the PR is close, but the startup fix should use a gateway-node-mode predicate rather than `ShouldInitializeNodeService()`. That keeps the intended node reconnect while preserving the documented MCP-only mode.
