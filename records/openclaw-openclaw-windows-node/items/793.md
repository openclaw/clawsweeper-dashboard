## Summary

Keep open: the PR is useful and now has live-output proof for the port override, but it is not merge-ready because the override is global while the token it sends is active-gateway scoped, and diagnostics still inspect the old gateway/tunnel-plus-two endpoint.

## What This Changes

The PR adds a nullable BrowserControlPort settings value, passes it into BrowserProxyCapability, and adds capability tests for override routing, invalid ports, and a real loopback HTTP round trip.

## Best Possible Solution

Keep the shared-token auth path, but bind the effective browser-control endpoint to the active GatewayRecord/tunnel contract, align diagnostics with that endpoint, and document/test gateway-switch behavior if a global override is intentional.

## Reproduction Assessment

Yes for the review findings: source inspection shows the override is global while browser.proxy sends the active shared gateway token to the configured endpoint. The contributor also provided live output showing the intended split-topology port override path over real loopback TCP.

## Solution Assessment

No. A port override may be useful, but the safer solution is to bind the effective endpoint to the active gateway/tunnel contract and keep diagnostics aligned with the same endpoint.
