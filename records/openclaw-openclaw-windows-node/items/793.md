## Summary

Keep open: current main and v0.6.3 still hard-derive browser-control as gateway port + 2, but this PR is not merge-ready because the final head still mixes an active-record port override with legacy SettingsManager tunnel state, diagnostics can probe a different port than browser.proxy dials, and live proof predates the final scoping commit.

## What This Changes

The PR adds a shared browser-control endpoint resolver, per-gateway BrowserControlPort storage, NodeService and Command Center wiring, and tests for override, tunnel, and live loopback browser.proxy traffic.

## Best Possible Solution

Resolve BrowserControlPort, tunnel local/remote ports, and diagnostics from the active GatewayRecord or live SshTunnelSnapshot through BrowserControlEndpoint, then require final-head split/remote proof before merge.

## Reproduction Assessment

Yes, from source inspection. Switch from a saved SSH-tunnel gateway to a direct gateway while SettingsManager.UseSshTunnel remains true and this head can still choose the stale tunnel local + 2 port; separately, an override-only listener is not what PortDiagnosticsService probes.

## Solution Assessment

No. A per-gateway port override is a plausible direction, but this implementation should use active GatewayRecord or live tunnel state as the single endpoint source and make diagnostics consume the same resolved endpoint.
