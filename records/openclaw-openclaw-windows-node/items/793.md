## Summary

Keep open for maintainer review: the latest head appears to solve the browser-control port scoping problem and closes the earlier stale-tunnel diagnostics gaps, but it adds persisted per-gateway endpoint configuration that determines which loopback service receives the shared gateway token, so the remaining blocker is an explicit maintainer product/security decision rather than a narrow code repair.

## What This Changes

The branch adds a per-gateway BrowserControlPort override and shared browser-control endpoint resolver, wires active gateway tunnel state into browser.proxy and Command Center diagnostics, preserves the field across gateway edits, and adds shared/connection/tray regression tests.

## Best Possible Solution

Land the scoped per-gateway override if maintainers accept it as an advanced endpoint configuration surface; otherwise require a documented/operator-facing configuration path or a different browser-control endpoint policy before merging.

## Reproduction Assessment

Yes from source, but not from a live WSL2/Windows split-gateway run in this read-only review. Current main always derives the control endpoint as gateway port + 2, so a split/remote setup whose browser-control listener is forwarded to a different local port cannot be targeted.

## Solution Assessment

Unclear pending maintainer product/security acceptance. The implementation is narrow and now shares endpoint resolution across dial path and diagnostics, but the advanced per-gateway config surface and shared-token recipient contract need maintainer sign-off.
