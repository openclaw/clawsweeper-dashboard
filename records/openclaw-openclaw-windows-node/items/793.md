## Summary

Keep this PR open; the requested override is not on current main or v0.6.3, and the contributor provided strong live proof for the intended socket path, but final head still has endpoint-scoping defects that can misroute browser.proxy and its shared token after gateway switches.

## What This Changes

The branch adds a per-gateway browser-control port override, a shared endpoint resolver, NodeService and Command Center wiring, and tests for override, tunnel, and real loopback browser.proxy traffic.

## Best Possible Solution

Use BrowserControlEndpoint as the single source for BrowserProxyCapability and Command Center diagnostics, deriving tunnel state from the active GatewayRecord or live tunnel snapshot and preserving legacy SettingsManager fallback only when no active resolver exists.

## Reproduction Assessment

Yes for the PR review findings from source inspection: App.xaml.cs supplies an active gateway tunnel resolver, while NodeService still falls back to SettingsManager.UseSshTunnel when that resolver returns null. The contributor's terminal proof is sufficient for the core override socket path, but it does not cover this gateway-switch derivation bug or the diagnostic probe mismatch.

## Solution Assessment

No as currently implemented. A per-gateway BrowserControlPort is a plausible direction, but the maintainable fix is to have NodeService and Command Center consume the same active BrowserControlEndpoint result instead of mixing active records with stale global settings.
