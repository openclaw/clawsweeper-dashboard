## Summary

Keep open: the port-override idea is useful and has credible live proof, but the patch still mis-scopes the override relative to active-gateway credentials and leaves Command Center probing a different port than browser.proxy will dial.

## What This Changes

Adds a nullable BrowserControlPort setting, a shared browser-control endpoint resolver, browser.proxy and Command Center wiring, and endpoint-routing tests for split or tunneled gateway topologies.

## Best Possible Solution

Resolve the browser-control endpoint from active GatewayRecord/tunnel state and use the same resolver for browser.proxy, diagnostics, and copied guidance, with tests for gateway switching and override listener probing.

## Reproduction Assessment

Yes from source inspection: with BrowserControlPort set, NodeService passes the same global port override alongside whichever sharedGatewayToken is active, and diagnostics still build their listener probe from gateway/tunnel + 2. I did not run tests because this is a read-only cleanup review.

## Solution Assessment

No. A port override is a plausible solution, but the maintainable version needs active-gateway scoping and one shared endpoint resolver for proxy execution, diagnostics, and guidance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Scope the browser control override to the active gateway:** `src/OpenClaw.Tray.WinUI/Services/NodeService.cs:389`
  - body: This passes a persisted global BrowserControlPort into BrowserProxyCapability while the bearer token next to it is resolved from the active gateway. After the user switches gateways, the override can still point at the previous gateway's forwarded browser-control listener, so browser.proxy can send the new active gateway's shared token to the stale local endpoint. Bind the endpoint override to the active gateway/tunnel record, or otherwise clear/re-resolve it per active connection before constructing the capability.
  - confidence: 0.89
- **[P2] Probe the resolved browser-control endpoint in diagnostics:** `src/OpenClaw.Tray.WinUI/Services/CommandCenterStateBuilder.cs:50`
  - body: The warning builder now receives BrowserControlPort, but portDiagnostics was already built by PortDiagnosticsService from gateway/tunnel + 2. With BrowserControlPort=19000 and only that listener running, Command Center still marks the gateway+2 port as not listening and emits a browser warning even though browser.proxy will dial 19000. Build the diagnostic port list from the same BrowserControlEndpoint result and add a regression test for the override-listener case.
  - confidence: 0.86
