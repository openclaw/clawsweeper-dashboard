## Summary

Keep open: current main still lacks split browser-control port support, and this PR is useful, but the current head can still route browser.proxy through stale global SSH tunnel settings and its diagnostics do not probe the same endpoint the proxy dials.

## What This Changes

The PR adds a browser-control port resolver and per-gateway override for browser.proxy, wires Command Center guidance to it, and adds shared/connection tests for override and tunnel endpoint selection.

## Best Possible Solution

Resolve browser-control endpoint state entirely from the active GatewayRecord or active tunnel snapshot, then feed that same resolved endpoint into browser.proxy, diagnostics, guidance, and regression tests.

## Reproduction Assessment

Yes, from source inspection. Switch from a saved SSH-tunnel gateway to a direct gateway without updating SettingsManager and the PR's new useSshTunnel path can still choose the stale tunnel local + 2 port; separately, an override-only listener is not what PortDiagnosticsService probes.

## Solution Assessment

No. A port override is a plausible direction, but this implementation should use the active GatewayRecord/tunnel state as the single source of truth and make diagnostics consume the same resolved endpoint.
