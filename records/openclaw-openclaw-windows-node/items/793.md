## Label Justifications

- `P2`: This is a normal-priority browser.proxy improvement with limited blast radius, but the remaining endpoint and token-routing defects should block merge.
- `merge-risk: 🚨 compatibility`: The new endpoint resolver can change browser.proxy behavior for multi-gateway users by letting stale global SSH tunnel settings affect the active gateway.
- `merge-risk: 🚨 auth-provider`: browser.proxy sends the active gateway shared token to the resolved browser-control endpoint, so wrong endpoint selection affects credential routing.
- `merge-risk: 🚨 security-boundary`: Misrouting the local browser-control request can expose an active gateway shared token to a stale or unintended loopback listener.

## Summary

Keep open: current main and v0.6.3 still hard-derive browser.proxy from gateway port + 2, so the PR is not obsolete, but final head still has endpoint-scoping defects that can misroute browser.proxy and shared-token traffic after gateway switches.

## What This Changes

The PR adds a per-gateway browser-control port override, a shared endpoint resolver, NodeService and Command Center wiring, and tests for override, tunnel, and real loopback browser.proxy traffic.

## Best Possible Solution

Use BrowserControlEndpoint as the single source for BrowserProxyCapability and Command Center diagnostics, deriving tunnel state from the active GatewayRecord or live tunnel snapshot and preserving an explicit legacy-settings fallback only where maintainers want it.

## Reproduction Assessment

Yes for the PR review findings from source inspection: switch from a saved SSH-tunnel gateway to a direct gateway while SettingsManager.UseSshTunnel remains true, and final head can still choose the stale tunnel local + 2 port. The contributor’s terminal proof is sufficient for the core override socket path, but it does not cover this gateway-switch derivation bug or the diagnostic probe mismatch.

## Solution Assessment

No as currently implemented. A per-gateway BrowserControlPort is a plausible direction, but the maintainable fix is to have NodeService and Command Center consume the same active BrowserControlEndpoint result instead of mixing active records with stale global settings.
