## Summary

## What This Changes

The PR adds a nullable BrowserControlPort setting, passes it into BrowserProxyCapability, and adds two shared capability tests for configured-port routing and invalid-port handling.

## Best Possible Solution

Keep the shared-token auth path, but scope the effective browser-control endpoint to the active GatewayRecord/tunnel model, align diagnostics/tests with that endpoint, and require redacted live split-gateway proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection shows the override is global while browser.proxy sends the active shared gateway token to the configured endpoint. The underlying split/remote success path is not live-proven by the PR artifacts.

## Solution Assessment

No. A port override may be useful, but the safer solution is to bind the effective endpoint to the active gateway/tunnel contract and keep diagnostics aligned with the same endpoint.
