## Summary

Keep open: the newer head adds useful tunnel-local routing and sufficient live proof, but the global override still crosses the per-gateway credential boundary and Command Center diagnostics still do not probe the same effective endpoint as browser.proxy.

## What This Changes

The branch adds a nullable BrowserControlPort setting, a shared BrowserControlEndpoint resolver, browser.proxy and Command Center wiring, and endpoint routing tests for split/remote gateway topologies.

## Best Possible Solution

Resolve the browser-control endpoint from the active GatewayRecord/tunnel contract, or explicitly document and test a global override, then make browser.proxy, diagnostics, setup guidance, and reload behavior consume that same contract.

## Reproduction Assessment

Yes for the review findings: source inspection shows the override is global while browser.proxy sends the active shared gateway token, and diagnostics still build ports from topology/tunnel only. The contributor also supplied live loopback output for the intended override and tunnel-local success paths.

## Solution Assessment

No. A port override is useful, but the safer solution is to bind the effective endpoint to the active gateway/tunnel contract and keep diagnostics aligned with the same endpoint.
