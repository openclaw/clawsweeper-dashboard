## Summary

Keep open: current main does not show a NO_REPLY suppression guard in the tray live/history chat ingestion path, so a narrow bug fix remains valid.

## What This Changes

The issue reports that the Windows tray chat UI renders the internal NO_REPLY directive as visible chat text instead of suppressing it.

## Best Possible Solution

Add a narrow suppression guard for exact NO_REPLY assistant/control-directive payloads in the shared chat ingestion path used by the tray, with regression tests for live events and any history path that can carry the directive.

## Reproduction Assessment

Yes from source inspection, but not from a live tray run: a non-empty assistant ChatMessageInfo with text NO_REPLY reaches the assistant render path because current guards only drop empty text, missing session keys, aborted threads, and late non-final frames.

## Solution Assessment

Yes, a narrow source-level fix is appropriate; the safest place is the shared chat ingestion/normalization boundary before the tray timeline renders assistant text, not a visual-only special case.
