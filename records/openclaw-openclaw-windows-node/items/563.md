## Summary

The issue remains necessary: current main only implements Updatum-based update checks, while the AppInstaller feed path and its failure handling remain in an open, conflicted draft PR. The existing manual-check dialog is a useful UX pattern, but it does not exercise or solve missing, invalid, or unregistered AppInstaller feeds.

## What This Changes

Show a user-visible diagnostic when a manual AppInstaller update check cannot retrieve, parse, or resolve its feed.

## Best Possible Solution

Add localized visible failure feedback and detailed redacted logging directly to the final AppInstaller feed-check path, covering unavailable HTTP feeds, invalid XML, and missing registered sources while preserving Updatum for existing distribution channels.

## Reproduction Assessment

No on current main: the described AppInstaller feed-check path has not landed, so its 404, parse, and registration failures cannot be exercised; the existing Updatum path already shows manual failure dialogs.

## Solution Assessment

Yes in direction, but only as part of the chosen AppInstaller update architecture; reusing the current updater's Failed-status dialog pattern is narrower and more maintainable than adding a parallel standalone notification path.
