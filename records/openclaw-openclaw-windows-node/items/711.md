## Summary

Keep this PR open, but do not merge as-is: the updated direction is useful and the external WebView proof is now present, but the tunnel rewrite fix reads the configured gateway from `SettingsManager`, which can be stale when users switch saved SSH-tunnel gateways backed by `GatewayRegistry`.

## What This Changes

The PR routes `canvas.navigate` into the Windows canvas WebView, stops rewriting external canvas URLs to the gateway origin, and adds a helper plus tests for external, relative, and tunnel gateway URL rewriting.

## Best Possible Solution

Thread the active configured gateway URL from the active `GatewayRecord` into the canvas origin setup, keep external URLs unchanged, and add regression coverage plus redacted proof for a saved SSH-tunnel gateway canvas URL.

## Reproduction Assessment

Yes from source inspection: saved gateway switching uses `GatewayRegistry.SetActive` and `SwitchGatewayAsync`, while tunnel node connections use the active record's localhost effective URL and do not update `SettingsManager` in that path. I did not run the Windows/WebView runtime because this was a read-only review.

## Solution Assessment

No, not as submitted: separating external URLs from gateway rewrites is the right shape, but the configured gateway origin must come from the active connection record rather than legacy/settings state.
