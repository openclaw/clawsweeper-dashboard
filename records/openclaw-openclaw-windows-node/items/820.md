## Summary

Keep open: current main still routes most tray-visible notifications through Windows toasts only, while the in-app Notifications page renders only AppNotificationService items.

## What This Changes

The issue requests that Windows toast notifications also populate the in-app Notifications tab instead of only exec-approval app notifications appearing there.

## Best Possible Solution

Add a shared tray notification dispatch path that records eligible AppNotification entries alongside toasts while preserving notification settings, toast dedupe, and chat toast session arguments.

## Reproduction Assessment

Yes, from source: trigger a gateway, node, pairing, or system.notify toast and the code calls ToastService without adding an AppNotification, so the Notifications tab has nothing to render. I did not run the Windows UI smoke.

## Solution Assessment

Yes, the suggested shared dispatch helper is the right general direction, but the implementation should classify persistence and dedupe rather than blindly persisting every transient toast.
