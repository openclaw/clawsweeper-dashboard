## Summary

Keep open: current main still drops keyless chat and agent events with warning logs only, while the requested user-visible one-shot diagnostic is not implemented. The related incompatible-handshake work fixed a different path and does not cover mixed-version keyless events after connection.

## What This Changes

Request a user-visible diagnostic when chat or agent events without `sessionKey` are dropped, without routing them into `main` or exposing message content.

## Best Possible Solution

Add a rate-limited visible provider notification or chat diagnostic for empty-sessionKey chat and agent event drops, reset it on reconnect, and preserve the current fail-closed drop behavior.

## Reproduction Assessment

Yes, at source level: the existing fake-bridge test already raises an empty-sessionKey chat event and current code returns after a warning with no notification path. I did not run the app or tests because this cleanup review is read-only.

## Solution Assessment

Yes, reusing `ChatProviderNotification` from `OpenClawChatDataProvider` is the narrowest maintainable path because it avoids new transport plumbing and preserves the no-`main`-fallback safety property.
