## Summary

Current main already fixes the likely Spanish-locale setup failure by parsing WSL version output without depending on English label text, and regression tests now cover Spanish `Versión de WSL` output. The fix is not in the latest v0.6.0 tag, so this is current-main-only cleanup rather than shipped-release cleanup.

## What This Changes

The issue reports that OpenClaw Gateway setup/startup fails on Windows when the system language is Spanish but works after switching to English.

## Best Possible Solution

Keep the locale-independent WSL version parser and Spanish regression coverage on main, then ship it in the next release so v0.6.0 users no longer need the English-language workaround.

## Reproduction Assessment

Yes for the likely source path: current tests model Spanish `wsl.exe --version` output that would have failed the previous English-label matcher. I did not run a live Spanish Windows VM, and the issue did not include logs tying the failure to a specific setup step.

## Solution Assessment

Yes: parsing the stable WSL product token plus numeric version is narrower and more maintainable than branching on individual localized labels. The remaining release work is to ship the current-main fix, not to add a new product mode.
