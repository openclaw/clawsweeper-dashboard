## Summary

This issue is still necessary: current main still contains the reported unguarded `async void` connection handlers and a broad remaining WinUI surface, while the linked automation PR only covers two connection-layer handlers.

## What This Changes

The issue asks for a guarded fire-and-forget pattern across WinUI and connection async-void event handlers to prevent unhandled exceptions from terminating the tray process.

## Best Possible Solution

Keep this issue open as the umbrella, review or land the connection-layer guard from the linked draft PR or equivalent, then add a shared guarded fire-and-forget helper and refactor the remaining WinUI/page/window/chat handlers with focused regression coverage and Windows smoke validation.

## Reproduction Assessment

Yes from source inspection: current main awaits inside `async void` connection handlers before any catch block, and several WinUI/chat handlers remain unguarded. I did not run a live Windows crash reproduction because this is a read-only review.

## Solution Assessment

Yes in direction: a centralized guarded fire-and-forget helper with one-line event-handler delegation is the narrow maintainable fix. The best sequencing is to handle the open connection-layer partial PR first, then do the broader WinUI pass.
