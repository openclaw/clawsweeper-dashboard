- `P2`: The report describes a bounded CanvasWindow lifecycle and credential-injection bug with limited blast radius.
- `impact:security`: The issue is about stale gateway bearer tokens being retained and injected after token refresh or gateway switch.

## Summary

Keep open: current main still wires an anonymous WebResourceRequested handler that captures the trusted origin and gateway token, while the close path only unsubscribes WebMessageReceived and NavigationCompleted.

## What This Changes

The issue requests CanvasWindow cleanup for WebView2 event subscriptions and avoidance of stale gateway-token capture in WebResourceRequested handling.

## Best Possible Solution

Store the WebResourceRequested handler in a field, remove the matching filter and unsubscribe all CoreWebView2 handlers on close, clear gateway auth fields on disposal, and add focused regression coverage plus a tray/WebView2 smoke.

## Reproduction Assessment

Yes. Source inspection on current main shows the WebResourceRequested lambda captures trustedOrigin/token locals and has no corresponding unsubscribe in OnWindowClosed; I did not run a live WebView2 repro.

## Solution Assessment

Yes. The suggested fix is the narrow maintainable direction: give the WebResourceRequested handler a field identity, unsubscribe it during close/disposal, avoid captured stale token state, and cover the lifecycle behavior.
