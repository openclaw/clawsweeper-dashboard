## Summary

Keep open. Current main still routes object-valued config schema fields without fixed child properties through the scalar editor path, so provider headers can be staged and saved as a string instead of a JSON object.

## What This Changes

The issue requests support for setting object-valued embedding provider headers, including an Azure Foundry `api-key` header, from the OpenClaw Config UI instead of the CLI.

## Best Possible Solution

Add generic object-map editing for schema fields like `headers`, preserving JSON object values through validation and save while keeping existing scalar and array behavior intact.

## Reproduction Assessment

Yes, from source inspection: an object-valued schema field without fixed `properties` falls through to the string editor and stages text for save. I did not run the WinUI app or a live Azure Foundry gateway, so this is source-reproducible rather than an executed UI repro.

## Solution Assessment

Yes, the best fix is in the existing schema config editor: support object/additionalProperties fields as JSON objects or key-value maps instead of adding provider-specific Azure-only UI or relying on the CLI workaround.
