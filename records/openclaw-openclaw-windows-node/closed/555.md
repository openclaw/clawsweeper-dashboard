## Summary

Keep open: current main still has the manual SettingsManager mapping and mutable SettingsData shape described here, but the report is a maintainability refactor rather than a reproduced user-facing P1 bug.

## What This Changes

Refactor SettingsManager persistence so SettingsData is the single source of truth, reducing manual load/save mapping and making secret protection transparent.

## Best Possible Solution

Preserve the existing settings file schema and migration semantics while collapsing the duplicate SettingsManager/SettingsData mapping into one tested persistence model, with focused upgrade tests for legacy credentials and protected secrets.

## Reproduction Assessment

Yes for the structural issue: current source shows manual load/save mapping and a mutable SettingsData class. No for a concrete user-visible bug: the report does not identify a specific setting that currently fails to persist.

## Solution Assessment

Unclear: using SettingsData as the single source of truth is directionally sensible, but the exact immutable-record and custom-converter design needs care around legacy gateway credential migration, defaults, and DPAPI behavior.
