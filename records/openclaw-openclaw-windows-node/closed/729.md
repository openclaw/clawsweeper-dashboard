## Label Justifications

- `P2`: This is a localized onboarding regression with meaningful but limited blast radius for established Tray users.
- `impact:data-loss`: The unwanted ritual can cause existing workspace instruction or personality files to be overwritten or drifted.
- `impact:session-state`: The bootstrap gate incorrectly classifies an established workspace and gateway session as first-run state.

## Summary

Current main still contains the reported bootstrap path, and two open closing PRs now own the fix; the issue should remain open until one is reviewed and merged.

## What This Changes

Prevent Tray chat from automatically sending the first-run bootstrap ritual when durable existing gateway or workspace state shows the user is already configured.

## Best Possible Solution

Review and land one consolidated PR that uses the existing configured-gateway classifier with correct registry and data-directory inputs, covers shared/bootstrap/device-token and true-first-run cases, and then close the duplicate PR.

## Reproduction Assessment

Yes. On current main, set or load `HasInjectedFirstRunBootstrap` as false with an already configured registry and connected operator client; the Tray path reaches `SendChatMessageForRunAsync` because no configured-workspace guard exists.

## Solution Assessment

Yes directionally: reusing `SetupExistingGatewayClassifier` is the narrow ownership-aligned fix, but one PR should be consolidated and validated for device-token-only and true-first-run behavior before merge.
