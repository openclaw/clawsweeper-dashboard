## Summary

Keep open: current main still carries a source-reproducible path where the chat model selector can display models returned by `models.list` even when they are explicitly not configured, then selecting one calls the session model patch path that can be rejected by the gateway.

## What This Changes

The issue reports that the chat model ComboBox shows non-whitelisted models and snaps back with a model-not-allowed error when switching.

## Best Possible Solution

Filter the chat model selector to configured/allowed model ids before they reach `AvailableModels`, while preserving a non-selectable display fallback for the current session model if needed.

## Reproduction Assessment

Yes from source inspection, though not from a live UI run: a `models.list` response containing both configured and explicitly unconfigured models is parsed into `AvailableModels`, rendered in the ComboBox, and selection is sent to the gateway patch path.

## Solution Assessment

Yes: the narrow maintainable fix is to filter explicitly unconfigured model entries in the existing model-list pipeline and add regression coverage for mixed configured true/false payloads.
