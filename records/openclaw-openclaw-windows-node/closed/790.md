## Summary

Current main already fixes the reported Ask/default-action behavior: the UI now stores Ask as the engine's canonical prompt value, legacy ask values are migrated/accepted, and prompt execution remains wired through the approval handler.

## What This Changes

The issue reports that the permission execution-policy default action offers Deny, Allow, and Ask, but Ask does not work and changes to Deny after restart.

## Best Possible Solution

Keep the current main fix and let the next release carry the Ask/Prompt compatibility behavior to users still on v0.6.3 or older.

## Reproduction Assessment

Yes from source inspection: legacy files written with defaultAction ask would fail the enum parse path and fall back to Deny, matching the restart symptom. I did not run the Windows tray UI because this was a read-only cleanup review.

## Solution Assessment

Yes; the current implementation is the narrow maintainable fix because it writes the canonical prompt value, accepts legacy ask, and leaves deny-by-default behavior intact for true failures.
