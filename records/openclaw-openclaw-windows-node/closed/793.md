## Summary

Keep open for maintainer review: current main still cannot target a split/remote browser-control listener on a non-derived port, and the PR appears technically coherent with sufficient live proof, but it adds persisted endpoint configuration that changes which loopback listener receives the shared gateway token.

## What This Changes

The PR adds a per-gateway BrowserControlPort override/resolver for browser.proxy, scopes tunnel and Command Center diagnostics to the active GatewayRecord, preserves the field across gateway edits, documents manual configuration, and adds shared/connection/tray tests.

## Best Possible Solution

Land the scoped per-gateway override if maintainers accept it as the advanced endpoint configuration surface; otherwise require an operator-facing configuration path or a different browser-control endpoint policy before merge.

## Reproduction Assessment

Yes, from source and contributor live output. Current main always derives the control endpoint as gateway port + 2, so a split/remote setup whose browser-control listener is forwarded to a different local port cannot be targeted without this new setting.

## Solution Assessment

Unclear pending maintainer product/security acceptance. The patch is the narrowest implementation I found for a per-gateway port override and aligns diagnostics with the dial path, but the persisted endpoint setting and shared-token recipient contract need maintainer sign-off.
