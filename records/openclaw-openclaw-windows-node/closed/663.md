## Summary

Current main already has the requested remote Windows node path: a configured gateway record can connect the tray as a node to a remote gateway, pair with device credentials/bootstrap credentials, advertise node capabilities, and handle remote `system.run` invocations. The optional direct file read/write/edit tools are not a separate capability, but the request explicitly allowed filesystem access via `exec` plus PowerShell, which the shipped implementation supports.

## What This Changes

Request a lightweight Windows remote node mode that pairs to a configurable remote gateway and exposes exec plus optional local filesystem operations without requiring a local gateway.

## Best Possible Solution

Keep the shipped tray Node Mode as the supported remote Windows node path, using `system.run` plus PowerShell for filesystem access unless maintainers later design a separate first-class file capability.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing-behavior report. The current-main source/docs/tests verify the requested remote node workflow instead of a failing reproduction path.

## Solution Assessment

Yes: the existing manager-owned tray Node Mode is the narrowest maintainable solution because it reuses GatewayConnectionManager, GatewayRegistry, NodeConnector, pairing, allowCommands, and exec approval policy instead of adding a parallel client path.
