## Summary

Keep open; this PR is still useful and current main does not contain the WebSocket generation guard, but the latest head still leaves one reconnect entry point unscoped. A failed current `ConnectAsync` can start backoff without socket/generation ownership, so a later successful connect can still be replaced by stale reconnect work.

## What This Changes

The branch adds connection-generation ownership checks to shared WebSocket connect/listen/reconnect handling and adds loopback regression tests for overlapping connects and reconnect takeover.

## Best Possible Solution

Land the focused generation guard only after every reconnect kickoff is generation-aware and targeted WebSocketClientBase tests cover listener, failed-connect, and closing-socket takeover cases.

## Reproduction Assessment

Yes, source-level. A failed current `ConnectAsync` can start `ReconnectWithBackoffAsync()` without owner context, then a later successful `ConnectAsync` can be disposed when the stale backoff resumes.

## Solution Assessment

No, not yet. Generation guarding is the right ownership boundary, but the fix needs to carry that ownership through every reconnect kickoff, including failed-connect backoff.
