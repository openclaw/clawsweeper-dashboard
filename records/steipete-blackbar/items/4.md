## Summary

Keep open: the reported maintenance-only inconsistency is real on current main, but this PR should not merge as written because it treats every maintenance row as non-operational and the added proof is still synthetic model/test output rather than a real BlackBar behavior run.

## What This Changes

The branch adds `maintenances.isEmpty` to `BlacksmithStatus.isOperational`, adds a maintenance-only model test, and adds a 0.2.5 changelog entry.

## Best Possible Solution

Make operational state maintenance-status-aware: keep maintenance notices visible, treat active maintenance as non-operational, and keep planned or not-started maintenance operational until it begins.

## Reproduction Assessment

Yes at source level. Current main ignores maintenances in `isOperational`, and the PR regression for scheduled maintenance is source-reproducible because the new predicate ignores `StatusEvent.status`; I did not run tests in this read-only review.

## Solution Assessment

No, not as written. Adding `maintenances.isEmpty` fixes the in-progress case but the narrower maintainable fix is to base operational impact on maintenance status.
