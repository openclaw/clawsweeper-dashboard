# Coding Plan for openclaw/gitcrawl#83: Add bounded PR-detail fill

Source report: [records/openclaw-gitcrawl/items/83.md](records/openclaw-gitcrawl/items/83.md)

## Summary

Keep open: current main and v0.6.2 provide targeted `sync --numbers --with pr-details`, but not a first-class command that selects locally missing PR-detail rows, bounds the run, and emits quota-aware progress.

## Plan

Implement a focused bounded PR-detail fill command for openclaw/gitcrawl. Add a CLI command such as `gitcrawl fill-pr-details owner/repo --limit N --json-progress` that selects local pull-request threads for the repository that are missing `pull_request_details`, orders them predictably for a first version, batches by the requested limit, reuses existing PR-detail hydration/persistence behavior, emits structured JSON progress including counts and observed GitHub rate-limit remaining/reset data, and stops in a state that can be resumed by rerunning the command. Keep this separate from archive coverage reporting in https://github.com/openclaw/gitcrawl/issues/82 and failed-attempt persistence in https://github.com/openclaw/gitcrawl/issues/84. Do not change default `gitcrawl sync` semantics. Add focused unit tests for selection, limit handling, progress output, rate-limit reporting, and resumability, plus docs/help updates; no changelog edit is required.
