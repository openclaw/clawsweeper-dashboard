## What This Changes

## Best Possible Solution

Carry the selected thread ID through refresh orchestration, add a regression for `applyClusterRefresh` with reordered clusters, and provide redacted terminal or recording proof of a real TUI reload retaining the highlight.

## Reproduction Assessment

Yes from source inspection: current main clears the selected member rows before the restore logic can read them, and the PR body includes a matching before-fix state log. I did not run the interactive TUI in this read-only review.

## Solution Assessment

No as written: restoring inside `loadSelectedCluster` fixes direct reload, but refresh orchestration must preserve the original member ID across `applyClusterRefresh` reselects. The safer fix is to snapshot before refresh mutates rows and add a reordered-refresh regression test.
