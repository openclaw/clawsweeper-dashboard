## Summary

Keep open: the reported TUI selection reset is valid, but this PR is not merge-ready because the snapshot is taken too locally and can still be lost during refresh reselects; it also lacks after-fix real TUI proof.

## What This Changes

The PR snapshots the selected TUI cluster member ID in `loadSelectedCluster`, restores it after detail reload, and adds a regression test plus changelog entry.

## Best Possible Solution

Preserve the selected thread ID before refresh/filter orchestration mutates rows, carry it through the final `currentKey` reload, add reordered-refresh coverage, and then provide redacted real TUI proof.

## Reproduction Assessment

Yes, from source inspection: current main clears the selected-member rows before `sortMembers` can read them, and the PR body includes a matching before-fix state log. I did not run the interactive TUI because this review is read-only.

## Solution Assessment

No as written. Restoring inside `loadSelectedCluster` fixes a direct reload, but the refresh orchestration needs to carry the original selected thread ID across `applyClusterRefresh` reselects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Carry the member snapshot through refresh reselects:** `internal/cli/tui.go:3348-3350`
  - body: `applyClusterRefresh` calls `applyClusterFilters()` before it reselects `currentKey`; if a refresh reorders clusters, that first load can replace the original selected-member snapshot with a temporary cluster's first member, so the final reload still falls back to the first row. Capture the selected thread ID before refresh mutates the rows or pass it into the final reload, and add reordered-refresh coverage.
