## Best Possible Solution

Carry the selected thread ID through refresh orchestration, add a regression for `applyClusterRefresh` with reordered clusters, and then provide redacted terminal or recording proof of the TUI retaining the member selection after reload.

## Reproduction Assessment

Yes from source inspection: current main clears the selected member state before the existing restore logic can read it, and the PR body includes a matching before-fix state log. I did not run the interactive TUI in this read-only review.

## Solution Assessment

No as written: restoring inside `loadSelectedCluster` fixes the direct reload case, but the auto-refresh/manual-refresh orchestration must preserve the original member ID across `applyClusterRefresh` reselects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Carry the member snapshot through refresh reselects:** `internal/cli/tui.go:3348-3350`
  - body: `applyClusterRefresh` can call `applyClusterFilters()` and load a temporary cluster before it reselects `currentKey`. Capture the selected member before refresh mutates the rows, or pass it through the final reselect path, and cover the reordered-refresh case.
