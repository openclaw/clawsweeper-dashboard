## Label Justifications

- `P2`: This is a normal-priority sync bug with a clear failure mode that can abort PR detail hydration for affected pull requests.
- `impact:other`: The issue affects Gitcrawl's local PR metadata hydration workflow, which is maintainer-visible but outside the more specific impact categories.

## Summary

Keep open: current main and v0.6.1 still key PR files by `(thread_id, path)`, while the reported GitHub API response can contain two distinct file rows with the same filename, so `--include-pr-details` can still abort during hydration.

## What This Changes

Make PR detail hydration tolerate duplicate GitHub file filenames within one PR without aborting sync, ideally while preserving the full fetched file list.

## Best Possible Solution

Store PR files as a replaceable fetched snapshot keyed by a per-fetch position or synthetic row id, keep path indexed but non-unique, and add regression coverage for duplicate filenames plus idempotent resync.

## Reproduction Assessment

Yes at source level: GitHub returns duplicate filename rows for the cited PR, and current main inserts those rows into a `(thread_id, path)` primary key. I did not run the mutating sync command during this read-only review.

## Solution Assessment

Yes: treating PR files as a fetched snapshot with a non-path identity is the narrowest durable fix because it preserves GitHub's file-list shape instead of losing detail through coalescing.

## Repair Work Prompt

Fix `gitcrawl sync --include-pr-details` so GitHub pull file rows with duplicate `filename` values for one PR do not violate `pull_request_files` constraints. Treat PR files as a replaceable fetched snapshot: add a per-thread file position or synthetic row id, keep `path` indexed but non-unique, preserve `status`, `previous_path`, `patch`, `raw_json`, and fetched timestamps, and update schema/sqlc/store models plus any migration/open compatibility needed for existing databases. Add regression coverage using duplicate removed/added rows for the same path and verify repeated upsert/sync is idempotent. Do not coalesce duplicate paths unless maintainers explicitly choose detail loss, and do not edit `CHANGELOG.md`.
