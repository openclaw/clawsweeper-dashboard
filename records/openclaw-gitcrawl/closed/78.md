## Summary

Keep open: current main still has `init --db` as a database-only override, with `vector_dir` following that DB path while `cache_dir` and `log_dir` stay on platform defaults. No duplicate or existing supported fully isolated init mode surfaced in code, docs, or GitHub search.

## What This Changes

The issue requests a documented fully isolated `gitcrawl init` mode so temporary configs keep database, cache, vector, and log paths under one runtime directory.

## Best Possible Solution

Add a default-compatible isolated init path, probably `init --runtime-dir <dir>`, that writes DB, cache, vector, and log paths under that directory while documenting that `--db` remains database-only.

## Reproduction Assessment

Yes at source level: `runInit` only changes `DBPath` for `--db`, and `Config.Normalize` leaves cache/log defaults while retargeting only `vector_dir`. I did not run `gitcrawl init` because this review must avoid commands that write config/runtime files.

## Solution Assessment

Unclear until maintainers choose the public CLI surface. A single `--runtime-dir` looks like the narrowest user-facing option, but a docs-only clarification or explicit per-path flags are also possible product choices.
