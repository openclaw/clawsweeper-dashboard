## Summary

Keep open: this is a useful SQLite lock retry fix with credible real-smoke proof, but the patch changes the generic transaction helper so existing callbacks can be replayed after they mutate caller-owned state.

## What This Changes

This PR adds SQLite busy/locked transaction retries, quiet gh-shim auto-hydration, retry-safe sync stats, focused tests, and a changelog entry.

## Best Possible Solution

Keep the transient SQLite retry behavior, but place it behind a replay-safe boundary or make every retried transaction callback use attempt-local result/progress state with regression coverage.

## Reproduction Assessment

No high-confidence local reproduction from this read-only review. The PR body provides credible before/after concurrent gh-shim smoke evidence, and source inspection confirms main currently has single-attempt transactions and noisy auto-hydration paths.

## Solution Assessment

Not yet: retrying transient SQLite lock failures is the right behavior, but putting replay at Store.WithTx broadens the contract and needs either idempotent callers or a narrower sync-cache retry boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep transaction retries from replaying result mutations:** `internal/store/store.go:115-117`
  - body: The new retry wraps every `Store.WithTx` callback, but existing callbacks mutate caller-owned state while the transaction is still in progress. For example, `SaveDurableClusters` increments `result.MemberCount` inside the callback; if a later write or commit hits `SQLITE_BUSY` and the callback is retried, the database rollback is fine but the returned result is counted again. Scope the retry to side-effect-safe sync persistence or make all `WithTx` callbacks accumulate attempt-local state before exposing results.
  - confidence: 0.88
