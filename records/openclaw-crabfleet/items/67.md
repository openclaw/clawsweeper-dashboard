## Summary

Keep open: current main still lacks ordinary browser-origin validation for the terminal WebSocket, and this PR targets a real security boundary, but the proposed comparator can reject supported direct incoming origins and the PR still needs real behavior proof before merge.

## What This Changes

The PR adds a terminal WebSocket origin validator, calls it before opening `/api/terminal/ws`, and adds unit coverage for same-origin, cross-origin, origin-less, and service-authenticated cases.

## Best Possible Solution

Land a narrow security fix that checks `Origin` against the browser-visible request origin, preserves trusted-proxy public-origin handling and service-authenticated terminal paths, and includes redacted runtime proof for rejected and allowed WebSocket cases.

## Reproduction Assessment

Yes, source-level: current main routes `GET /api/terminal/ws` to `TerminalHub.open` after user resolution without an ordinary browser `Origin` comparison. I did not run a live authenticated WebSocket setup in this read-only review.

## Solution Assessment

No, not yet: enforcing Origin before the terminal hub is the right shape, but comparing only to `browserAppOrigin(env)` is too strict for supported direct incoming origins. Use `browserRequestOrigin(request, env)` or equivalent trusted-proxy-public/request-origin logic and keep the service-authenticated bypass covered.
