## Summary

Keep this PR open, but it is not merge-ready: the native VNC field is exposed for stale/inactive sessions and the PR has no after-fix real behavior proof for the macOS/Crabbox handoff.

## What This Changes

The PR adds a native Crabbox VNC handoff path that carries a non-secret native lease id through Fleet, launches the installed Crabbox CLI from the macOS app, and documents/tests the new capability.

## Best Possible Solution

Expose nativeVncLeaseId only for controllable sessions whose lifecycle is connectable, keep browser VNC disabled for native-only leases, and land after redacted real handoff proof is added.

## Reproduction Assessment

Yes at source level: a runtime-v1 session with providerResourceId, canControl, and capabilities.nativeVnc still receives nativeVncLeaseId even when its status is inactive. I did not run a live macOS/Crabbox handoff because this is a read-only review and no proof artifact was provided.

## Solution Assessment

No; the implementation needs a lifecycle gate before exposing nativeVncLeaseId so the native client cannot offer stale helper launches. The CLI handoff design is otherwise coherent but still needs real behavior proof after the dependency is available.
