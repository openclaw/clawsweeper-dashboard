## Summary

Keep open: current main does not already contain this branch, and the PR still has a concrete filesystem-boundary blocker plus missing real behavior proof before it should merge.

## What This Changes

The PR centralizes filesystem boundary guard primitives and adds hardening, tests, docs, changelog notes, and a static lint guard for several fs-safe fallback paths.

## Best Possible Solution

Land the guard centralization only after the fallback code guards the actual created parent directory, adds regression coverage for that race, and includes real behavior proof from the contributor's setup.

## Reproduction Assessment

Yes, from source inspection: force pinned Python mode off, write to a nested missing parent with mkdir enabled, and swap the newly created parent to a symlink after fallback mkdir but before the path-based open. This read-only review did not execute the PR branch.

## Solution Assessment

No; shared guard primitives are the right direction, but this implementation should recreate/assert a guard for parentPath after mkdir before opening, renaming, or cleaning up fallback paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Guard the created parent before fallback writes:** `src/pinned-write.ts:202-205`
  - body: When mkdir creates a missing parent, parentGuard still refers to the nearest existing ancestor. The overwrite=false and temp-file fallback paths then open names under the newly created parent while only checking that ancestor, so a symlink swap of the new parent can redirect fallback writes outside the root; recreate and use a guard for parentPath after mkdir.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The security hardening leaves a concrete path-escape gap in the pinned-write fallback when mkdir creates a new parent.

Concerns:

- **[high] Fallback writes can escape a newly created parent:** `src/pinned-write.ts:202`
  - body: runPinnedWriteFallback records parentGuard before mkdir, so when parentPath did not exist the later fs.open(targetPath/tempPath) calls are guarded only by an ancestor. A swapped new parent symlink can redirect writes or cleanup outside the intended root.
  - confidence: 0.82
