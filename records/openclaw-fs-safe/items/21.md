## Label Justifications

- `P1`: A resolved append can still leave the canonical path missing for a later reader in a downstream production memory workflow, which is an urgent data-durability failure mode.
- `impact:data-loss`: The report is about successful append calls whose data may become unreachable or lost after concurrent rotation.

## Summary

Keep open: current main still implements `append` as a direct `O_APPEND` write with no fd sync, parent fsync, or atomic replacement, while the repo’s durable write paths already use temp-file replacement and fsync. The report is credible, but the exact append contract under concurrent rotators needs maintainer judgment before an automated repair.

## What This Changes

The issue requests making root append durable and atomic against concurrent rename/unlink rotators, with related fsync fixes for the pinned write fallback.

## Best Possible Solution

Define `root.append`’s durability contract explicitly, likely by making it serialized read-concat-atomic-replace through the existing pinned write path while leaving streaming append behavior to `openWritable({ writeMode: "append" })`, then add a focused rotator regression and fix the JS fallback fsync gap.

## Reproduction Assessment

Yes, there is a plausible high-confidence reproduction path: the issue includes a minimal concurrent writer/rotator script, and source inspection shows current main follows the direct append path that script targets. I did not execute it because this review is read-only.

## Solution Assessment

Unclear. Reusing the atomic write path is the most direct match for the documented atomic-write contract, but the append-vs-streaming behavior and guarantees against external rotators need a maintainer decision before treating the proposed sketch as the final fix.
