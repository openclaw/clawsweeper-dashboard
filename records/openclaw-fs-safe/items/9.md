## Summary

Keep this PR open: current main does not implement the `durable: false` async atomic write option, and the linked OpenClaw context makes this a useful fs-safe API change, but it still needs maintainer API/durability review and real fs-safe branch proof before merge.

## What This Changes

The PR adds a default-durable `durable?: boolean` option to async `writeTextAtomic` and `writeJson`, plus focused tests and an Unreleased changelog entry.

## Best Possible Solution

Land a narrow fs-safe API update that preserves durable defaults, exposes the opt-out only on async atomic text/JSON helpers, documents the tradeoff, and then lets OpenClaw update its pin before opting reconstructible session metadata out.

## Reproduction Assessment

Not applicable as a fs-safe bug reproduction; this is an API feature. Source inspection does verify that current main has no `durable` option and always requests both fsync steps for async text/JSON atomic writes.

## Solution Assessment

Mostly yes: defaulting `durable` to true and mapping `durable: false` to the existing `replaceFileAtomic` fsync flags is a narrow implementation. The remaining questions are maintainer API/durability acceptance and branch-specific proof, not a clearly better code path.
