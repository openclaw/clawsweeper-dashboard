## Label Justifications

- `P2`: This is a real write-path compatibility problem with limited filesystem scope and a security-sensitive design choice, not an urgent global regression.
- `impact:security`: The disputed behavior is the post-rename identity check that protects fs-safe writes from TOCTOU target swaps.

## Summary

Keep open: current main and the latest release still use dev/ino identity after the overwrite rename in both the Node fallback and Python helper paths, and the proposed hash-based workaround changes a security boundary.

## What This Changes

The issue asks fs-safe to handle or document rclone FUSE mounts where post-rename path lookups do not preserve dev/ino identity, causing overwrite writes to fail.

## Best Possible Solution

Keep the dev/ino identity check as the secure default, then have maintainers choose between documenting unstable-inode mounts as unsupported or adding an explicit compatibility mode with weaker, well-tested hash-based guarantees.

## Reproduction Assessment

Yes for a source-level reproduction: on a filesystem that changes inode identity across rename, the current fallback and helper paths deterministically compare the old temp identity with a fresh target lstat and throw. I did not establish a live rclone runtime reproduction in this checkout.

## Solution Assessment

Unclear: the report is valid, but making hashing the default would trade away object-identity proof. The safer path is a maintainer security decision on unsupported-mount documentation versus an explicit opt-in compatibility mode.
