## Summary

Keep open: current main and the latest release still rely on dev/inode identity after overwrite rename, and the proposed hash workaround changes a filesystem safety boundary that needs maintainer judgment.

## What This Changes

The issue reports that overwrite writes fail on rclone FUSE mounts because post-rename file identity verification depends on stable dev/inode values.

## Best Possible Solution

Keep dev/inode identity as the secure default, then have maintainers choose either explicit unsupported-mount documentation or an opt-in compatibility mode with clearly documented weaker guarantees.

## Reproduction Assessment

Yes for a source-level reproduction: on any filesystem that changes inode identity across rename, current main compares the pre-rename temp identity with a fresh target lstat and deterministically throws. I did not establish a live rclone mount reproduction in this checkout.

## Solution Assessment

Unclear: the report is valid, but making hashing the default would trade object-identity proof for content equality. The safer path is a maintainer security decision on unsupported-mount documentation versus an explicit opt-in compatibility mode.
