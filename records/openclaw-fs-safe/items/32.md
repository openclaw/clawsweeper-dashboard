- `impact:security`: The issue centers on fs-safe's post-write TOCTOU identity check and any fix changes a filesystem safety boundary.

## Summary

Keep open: current main still enforces dev/ino identity after the overwrite rename path, the Python helper has the same post-commit identity assumption, and docs/tests do not cover unstable-inode FUSE mounts. The suggested hash-based direction is security-sensitive rather than a narrow automatic repair.

## What This Changes

The issue reports that overwrite writes through `runPinnedWriteFallback` fail on rclone FUSE mounts because the post-rename destination lookup gets a different inode than the temp file written by fs-safe.

## Best Possible Solution

Document the unstable-inode FUSE limitation now, then only add hash/content verification behind an explicit security-reviewed compatibility path if maintainers accept the weaker identity guarantee.

## Reproduction Assessment

Yes. The report gives a concrete rclone mount procedure and current source shows any post-rename inode change will deterministically trip the identity check; I did not run rclone locally.

## Solution Assessment

Unclear. Documentation is the narrow safe first step, while replacing inode identity with hashing is not obviously equivalent to the current security guarantee and needs maintainer direction.
