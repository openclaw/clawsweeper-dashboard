## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route the opt-in around the Python helper:** `src/root-impl.ts:1138-1140`
  - body: With `renameIdentity: "verify-content-with-lock"`, this still calls `runPinnedWriteHelper`, which uses the Python worker whenever Python mode is not `off`. That worker never receives the new policy and still rejects the rclone case by comparing post-rename dev/ino, so the advertised fix fails on the default POSIX setup; either teach the helper the policy or bypass it for this opt-in and add helper-enabled coverage.
  - confidence: 0.92
- **[P2] Fail closed for unproven lock owners:** `src/root-impl.ts:1132-1134`
  - body: This reclaim callback treats any PID that is not alive in the local process table as safe to remove immediately, ignoring age and host/namespace identity. On a shared FUSE mount or another PID namespace, an active lock can look dead locally, letting two writers enter the supposedly exclusive section; keep the existing fail-closed contract unless the payload proves the owner cannot still be writing.
  - confidence: 0.78
- **[P3] Match the docs to the actual lock timing:** `docs/writing.md:249`
  - body: This says the write first runs strict and then retries under the lock, but the implementation acquires the sidecar lock before the write whenever the option is set. That mismatch matters for reviewers and users reasoning about double writes and lock cost, so the docs should describe the actual flow.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: Needs attention: the PR adds a weaker opt-in write verification mode and a PID-only stale-lock recovery path that can affect fs-safe's write safety boundary.

Concerns:

- **[medium] PID-only stale reclaim can break mutual exclusion:** `src/root-impl.ts:1132`
  - body: The new `shouldReclaim` and `shouldRemoveStaleLock` callbacks remove locks based only on local PID liveness, which is not a reliable owner proof across namespaces or shared/network mounts and can undermine the advisory lock used to justify content verification.
  - confidence: 0.78

  - body: The fallback accepts matching content instead of proving the destination path names the temp file object; that may be acceptable as an explicit compatibility mode, but it needs maintainer-approved documentation and tests for the weaker same-UID race boundary.
  - confidence: 0.82
