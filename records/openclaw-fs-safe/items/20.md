## Summary

Keep open: this member-authored PR adds a security-sensitive root mutation policy, but the proposed implementation has a blocking path-normalization issue that can make configured deny entries miss valid filesystem paths.

## What This Changes

The PR adds a root mutation deny policy API across root mutation methods, exports the new types/helpers, documents `denied-path`, and adds mutation-policy tests.

## Best Possible Solution

Land the root mutation policy only after preserving exact configured path strings, adding regression coverage for whitespace-suffixed paths, and completing maintainer review of the deny semantics.

## Reproduction Assessment

Yes for the PR defect: source inspection shows `policyPathEntries` trims configured path strings before comparison. A focused regression would configure `denyExact` for an absolute path ending in a space and then mutate that same relative path.

## Solution Assessment

No. The API direction may be reasonable, but the implementation should preserve configured path bytes instead of trimming them before applying a security-sensitive deny decision.
