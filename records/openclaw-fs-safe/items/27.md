## Summary

Keep open: current main still rethrows EPERM in this move helper, but this PR is not merge-ready because it treats every EPERM as a copy/remove fallback in a filesystem-safety primitive, rewrites the whole file to CRLF, and lacks after-fix Windows proof.

## What This Changes

The PR modifies `src/move-path.ts` so `movePathWithCopyFallback` falls back to copying when `guardedRename` throws `EPERM` as well as `EXDEV`.

## Best Possible Solution

Implement a narrow EPERM fallback for the intended Windows file-lock rename case, or add an explicit opt-in, while preserving fail-closed behavior for unrelated permission failures, restoring LF endings, adding regression coverage, and posting redacted Windows proof.

## Reproduction Assessment

Source-reproducible: current main catches only EXDEV, so a simulated EPERM from `fs.rename` through `guardedRename` will rethrow. The real Windows watcher scenario was not live-reproduced in this review.

## Solution Assessment

No: the fallback idea is plausible, but this implementation is too broad because EPERM can mean more than the Windows file-lock case. The safer path is a scoped EPERM fallback plus tests and real Windows proof.
