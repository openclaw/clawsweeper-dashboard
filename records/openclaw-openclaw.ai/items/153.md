## Summary

Keep this PR open, but it is not merge-ready: the CSP is useful security hardening, yet the current allowlist would block existing X-hosted images that the site renders today, and the PR lacks deployed runtime proof for a global browser security policy.

## What This Changes

The PR adds a site-wide `headers` block to `vercel.json` with a Content-Security-Policy plus `nosniff`, frame, referrer, and permissions hardening headers.

## Best Possible Solution

Ship the hardening after the CSP inventory matches all rendered assets, or those assets are self-hosted, and after an authorized Vercel preview or live deployment proves the headers and CSP behavior.

## Reproduction Assessment

Yes by source inspection: the proposed `img-src` omits `pbs.twimg.com` while current `/showcase` and `/shoutouts` render that origin. I did not run a Vercel preview because the discussion shows no authorized deployment proof.

## Solution Assessment

No. Adding headers is a reasonable direction, but this implementation needs a complete asset allowlist or local asset migration plus deployed CSP proof before it is the maintainable fix.
