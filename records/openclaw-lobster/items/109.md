## Label Justifications

- `P2`: The report describes a bounded reliability bug in persisted state rather than an immediate core-runtime outage.
- `impact:data-loss`: Interrupted writes can leave persisted state truncated or empty, losing the last committed checkpoint value.
- `impact:session-state`: The affected files store workflow/user state used to resume or continue later processing.

## Summary

Keep open: current main still has the reported direct `fsp.writeFile` paths for general-purpose CLI and SDK state, and the related open report only covers the separate pipeline resume-state path.

## What This Changes

The issue requests atomic persistence and clearer corrupt-state handling for `state.set`, `stateGet`, `stateSet`, `readState`, and `writeState`.

## Best Possible Solution

Add a shared atomic JSON state write helper, use it from the CLI and SDK general-purpose state writers, and surface invalid JSON as a clear corrupted-state error while preserving `null` for missing keys.

## Reproduction Assessment

Yes from source inspection: create a truncated JSON file for a state key and run `state.get` or `readState`; the current paths call `JSON.parse` and rethrow the parse error. I did not execute that harness because this review was read-only.

## Solution Assessment

Yes. Atomic temp-file rename plus explicit invalid-JSON handling is the narrowest maintainable repair, especially if shared with the related resume-state fix instead of duplicating logic.
