## Label Justifications

- `P2`: A corrupt approval index can break short-ID resume, but the issue is bounded to resume reliability and full-token paths appear separate.
- `impact:data-loss`: The report concerns corruption of a persisted approval-ID index in Lobster's state directory.
- `impact:session-state`: The affected index maps short approval IDs to resume state keys, so failures can strand session resume by short ID.

## Summary

Keep open: current main still writes the approval-ID index with a direct file write and parses corrupt index JSON without a fallback, so the short-ID resume reliability gap remains despite the earlier atomic state-write repair.

## What This Changes

The issue asks Lobster to make short approval-ID index persistence crash-safe and avoid raw SyntaxError failures when that index is corrupt.

## Best Possible Solution

Add an atomic create-if-absent approval-index write that preserves EEXIST/no-overwrite semantics, plus corrupt-index handling that returns a clear not-found path, with focused regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: corrupt the current approval_<id>.json file and call findStateKeyByApprovalId or resume --id for that ID. The read path calls JSON.parse and only catches ENOENT, so invalid JSON will escape instead of becoming a clean not-found result.

## Solution Assessment

The issue is valid, but the proposed one-line switch to writeFileAtomic is not the safest repair because that helper overwrites existing targets. The maintainable fix should preserve exclusive approval-ID creation while adding atomicity and a defensive corrupt-index fallback.
