## Summary

Keep open: current main still has a source-confirmed retry/timeout interaction where abort-shaped timeout failures can bypass the workflow retry policy, and no linked fix or canonical replacement is present in the provided context.

## What This Changes

The issue reports that workflow steps configured with both `timeout_ms` and `retry.max` do not retry after per-attempt timeout failures.

## Best Possible Solution

Preserve external workflow cancellation as non-retriable, make per-attempt `timeout_ms` failures retryable, and add regression coverage proving retry attempts and `[RETRY]` output for a timed-out step.

## Reproduction Assessment

Yes, from source with high confidence: `timeout_ms` uses abort signaling and `withRetry` currently short-circuits abort-shaped errors before workflow retry policy can run. I did not execute the workflow, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is to distinguish per-attempt timeout from external cancellation and add a focused regression test. A broad change that retries all `AbortError`s would be riskier than preserving the cancellation boundary.
