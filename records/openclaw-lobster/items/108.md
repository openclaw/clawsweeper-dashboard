## Label Justifications

- `P2`: The report is a normal-priority reliability bug in persisted resume state with a bounded implementation path.
- `impact:data-loss`: A process termination during persistence can leave the saved resume payload truncated or unusable.
- `impact:session-state`: The affected data is Lobster's persisted approval/input resume state.

## Summary

Keep open: current main and the latest release still write state payload JSON directly and rethrow invalid JSON parse errors, so the reported resume-state corruption remains actionable.

## What This Changes

The issue asks Lobster to persist resume state atomically and surface corrupted state JSON with a coherent resume error.

## Best Possible Solution

Implement atomic same-directory temp-file write plus rename for shared state JSON, keep ENOENT semantics, surface invalid JSON as a clear corrupted-state error, and add regression coverage for resume by token and approval id.

## Reproduction Assessment

Yes as a source-level path: corrupt or truncate a pipeline resume payload under LOBSTER_STATE_DIR and resume by token or approval id. The live process-kill timing window was not executed in this read-only review.

## Solution Assessment

Yes. Fixing the shared state store with atomic replacement and clearer invalid-JSON handling is narrower than adding a new resume mode or policy knob.
