## Summary

Keep open. Current main still only supports `needs_input` at stage and workflow boundaries, while this issue asks for command-internal `ctx.requestInput()` suspension, same-command replay, and workflow `pipeline:` halt propagation; the recent maintainer comment also explicitly asks for a dedicated implementation PR.

## What This Changes

The issue requests a new command-level `ctx.requestInput()` API that can suspend inside `run()`, persist suspended state, and resume the same command in tool/CLI workflows.

## Best Possible Solution

Add a dedicated, documented `ctx.requestInput()` command API with same-command replay, persisted `suspendedState`, CLI/tool/SDK coverage, and explicit workflow `pipeline:` halt propagation semantics.

## Reproduction Assessment

Yes, by source inspection: current command contexts do not expose `requestInput`, pipeline input resumes advance past the halting stage, and workflow `pipeline:` halts throw instead of bubbling `needs_input`. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the proposed replay-based direction is the maintainable fit, but it should be a dedicated maintainer-reviewed feature PR with compatibility coverage rather than cleanup automation.
