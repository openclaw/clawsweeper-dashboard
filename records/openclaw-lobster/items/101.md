## Summary

Keep open. Current main still only supports `needs_input` at stage/workflow boundaries, while the requested command-level `ctx.requestInput()` API with same-command replay and persisted suspended state is absent; the recent maintainer comment also explicitly keeps this for a dedicated implementation PR.

## What This Changes

Add a command-context API that can suspend for structured user input from inside a command `run()` and resume that same command with saved state.

## Best Possible Solution

Land a maintainer-reviewed implementation that adds `ctx.requestInput()` with replay/suspended-state semantics, preserves existing boundary pauses, and covers tool, CLI, SDK, workflow, and docs behavior.

## Reproduction Assessment

Yes. By source inspection, current command contexts do not expose `requestInput`, input resumes advance after the halted stage, and workflow `pipeline:` halts throw instead of bubbling `needs_input`.

## Solution Assessment

Yes, directionally. Same-command replay with persisted `suspendedState` is the maintainable cross-process approach, but it should land through a dedicated maintainer-reviewed design and implementation PR.
