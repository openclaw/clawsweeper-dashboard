## Summary

Keep open: current main supports `needs_input` at pipeline/workflow stage boundaries, but it does not expose a command-context `requestInput` API or resume a suspended command in place, and workflow-embedded pipeline halts still error instead of bubbling through.

## What This Changes

The issue requests a new `ctx.requestInput()` capability so a command can ask for structured input during `run()` and resume the same command with persisted suspended state.

## Best Possible Solution

Design a core command-context input API that reuses the existing `needs_input` envelope while adding explicit in-place resume state and regression coverage across CLI, tool runtime, workflow pipeline steps, and the SDK.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection does give high confidence that current main only supports stage-boundary `needs_input` and lacks the requested command-local suspension API.

## Solution Assessment

Unclear; the requested `ctx.requestInput()` direction is coherent, but the exact replay, persistence, and SDK contract should be settled by maintainers before implementation.
