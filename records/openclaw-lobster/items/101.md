## Summary

Keep open: current main supports `needs_input` at pipeline/workflow stage boundaries, but it does not expose a command-level `ctx.requestInput()` API or same-command resume semantics.

## What This Changes

Request adding a command-level `ctx.requestInput()` API that can suspend a running command with `needs_input` and resume it in place with persisted state.

## Best Possible Solution

Settle the command re-entry contract, then implement it consistently across runtime context creation, pipeline resume state, CLI/core resume paths, SDK behavior, workflow pipeline propagation, docs, and focused regression tests while preserving existing `ask` and workflow `input` behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection does confirm the current limitation: no `ctx.requestInput()`/in-place resume path exists and workflow pipeline halts still throw.

## Solution Assessment

Unclear pending maintainer direction: the proposed replay plus `suspendedState` design is coherent for cross-process resume, but it changes public command-runtime semantics and should be approved before implementation.
