## Summary

Keep open. Current main still supports `needs_input` at stage and workflow boundaries only; there is no command-internal `ctx.requestInput()` API, same-command replay state, or workflow `pipeline:` halt bubbling, and the maintainer discussion asks for a dedicated implementation PR.

## What This Changes

The issue requests command-internal structured input suspension with a `ctx.requestInput()`-style API, persisted suspended state, and same-command replay on resume.

## Best Possible Solution

Implement a dedicated command-level input API that reuses the existing `needs_input` protocol, preserves stage-boundary behavior, and documents the command replay/idempotency contract.

## Reproduction Assessment

Yes. Source inspection shows command contexts do not expose `requestInput` or `resumeInput`, pipeline input resumes advance past the halting stage, and workflow `pipeline:` halts throw instead of returning `needs_input`.

## Solution Assessment

Yes, the requested direction is a plausible maintainable solution if it lands as a dedicated maintainer-reviewed API and resume-semantics change with focused compatibility tests.
