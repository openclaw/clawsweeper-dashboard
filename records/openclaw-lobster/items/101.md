## Summary

Keep open. Current main and the latest release still support `needs_input` only at stage/workflow boundaries, not from inside a command `run()` with same-command replay, and the recent maintainer comment explicitly asks for a dedicated implementation PR.

## What This Changes

The issue requests a command-level `ctx.requestInput()` API that can suspend structured input from inside `run()` and resume the same command with persisted state.

## Best Possible Solution

Implement a dedicated, tested command-level input API that reuses the existing halt/resume protocol, preserves stage-boundary behavior, and documents the command replay/idempotency contract.

## Reproduction Assessment

Yes, by source inspection: current command contexts do not expose `requestInput`, input resumes advance after the halted stage, and workflow `pipeline:` halts throw instead of bubbling `needs_input`. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the proposed direction is a maintainable fit if implemented as a dedicated maintainer-reviewed PR with same-command replay, persisted `suspendedState`, CLI/tool/SDK coverage, and workflow propagation tests.
