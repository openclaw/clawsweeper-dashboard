## Summary

Keep open. Current main supports `needs_input` at stage/workflow boundaries through `ask` and workflow `input:` steps, but it does not implement command-internal `ctx.requestInput()` suspension or same-command replay; the recent maintainer comment also explicitly keeps this open for a dedicated implementation PR.

## What This Changes

Request adds a command-level `ctx.requestInput()` API that can suspend inside `run()`, persist `suspendedState`, and resume the same command across CLI/tool/workflow execution.

## Best Possible Solution

Implement a dedicated, tested command-level input API that reuses the existing halt/resume protocol while preserving current stage-boundary behavior and documenting the command replay/idempotency contract.

## Reproduction Assessment

Yes, by source inspection: current command contexts do not expose `requestInput`, pipeline input resumes advance past the halting stage, and workflow `pipeline:` halts throw instead of bubbling `needs_input`. I did not run artifact-producing tests because this review is read-only.

## Solution Assessment

Yes, the proposed direction is a maintainable fit if implemented as a dedicated PR with same-command replay, persisted `suspendedState`, CLI/tool/SDK coverage, and workflow propagation tests; it is too cross-cutting for cleanup automation.
