## Summary

Keep open: current main still records zero cost for any model missing from the pricing map, and the existing test suite explicitly asserts that behavior for unknown models. The safest first repair is a focused diagnostic fix so `cost_limit` users can see unknown-model or invalid-pricing configuration instead of getting silent `$0` accounting.

## What This Changes

The issue asks Lobster to stop silently recording `$0` cost for LLM usage when `CostTracker.DEFAULT_PRICING` does not contain the runtime model ID.

## Best Possible Solution

Add non-breaking diagnostics for unknown non-null model IDs and invalid `LOBSTER_LLM_PRICING_JSON`, with focused regression tests; defer dynamic pricing or fail-closed policy to a separate maintainer decision.

## Reproduction Assessment

Yes, from source inspection rather than executing a workflow: any `recordUsage` call with a model key absent from `this.pricing` records token counts but computes zero cost, and `checkLimit` compares against that zero estimate. The existing test suite also asserts unknown models cost `$0`.

## Solution Assessment

Yes for the first repair: warning on unknown model IDs and invalid pricing JSON is the narrowest maintainable fix that makes the failure mode visible without changing pricing policy. A dynamic pricing URL or unconditional fail-closed behavior should be considered separately because it changes configuration and upgrade behavior.
