## Label Justifications

- `P2`: The report identifies a normal-priority reliability bug with a clear source path and limited persistence-surface blast radius.
- `impact:data-loss`: A killed write can leave the stored diff snapshot truncated or unusable, corrupting Lobster's remembered state for that key.
- `impact:crash-loop`: Once the snapshot is malformed, every subsequent use of the same SDK diff key can fail with the same unhandled `SyntaxError`.

## Summary

This issue should stay open: current main still has the vulnerable SDK diff-state read/write path, while the merged atomic-state work only covers adjacent state helpers and the CLI command path partially.

## What This Changes

The issue asks for diff-state persistence to survive interrupted writes and recover from malformed snapshots in `diffLast` and `diffAndStoreValue`.

## Best Possible Solution

Add a focused diff-state persistence fix: atomic same-directory writes, malformed snapshot recovery as a cache miss for diff comparisons, and regression coverage without changing diff output semantics.

## Reproduction Assessment

Yes at source level: a malformed `<key>.json` makes `readFile` succeed and `JSON.parse` throw before the snapshot can be rewritten. I did not run a live SIGKILL harness in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is the proposed dual defense: use the existing atomic-write pattern and treat corrupt diff snapshots as missing. The command/shared-state behavior should be kept deliberate so workflow resume semantics are not broadened accidentally.
