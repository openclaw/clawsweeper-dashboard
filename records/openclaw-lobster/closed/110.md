## Summary

The PR is still useful: current main still direct-writes state files, while the latest head addresses the earlier mode-preservation, temp-cleanup, branch-refresh, and real-proof blockers. I found no discrete blocking patch defect; the remaining maintainer call is the deliberate compatibility tradeoff around private new state files and atomic replacement semantics.

## What This Changes

The PR routes pipeline, CLI, and SDK state writes through same-directory atomic temp-file replacement, preserves existing file modes, creates new state files as 0600, cleans failed temp files, and adds focused regression tests plus a changelog entry.

## Best Possible Solution

Land the atomic-write fix after maintainers accept the private-default/new-inode compatibility behavior, keeping the mode-preservation and cleanup coverage in both helper copies.

## Reproduction Assessment

Yes, source-reproducible: current main writes pipeline, CLI, and SDK state directly with fsp.writeFile and later reads with JSON.parse, so a truncated state file propagates as a parse failure. I did not run a destructive termination harness during this read-only review.

## Solution Assessment

Yes: same-directory temp-file creation with exclusive open, fsync, mode preservation, cleanup, and atomic rename is the narrow maintainable fix for preventing new truncation. Friendly handling of already-corrupt legacy state can remain a separate follow-up if maintainers want it.
