## Summary

Keep open: current main still has the per-call PDFium init/destroy lifecycle described in the report, with no visible replacement PR or shipped fix, but a runtime concurrency harness has not been established.

## What This Changes

The issue asks for a safer PDFium lifecycle so concurrent `extractPdfContent` calls cannot tear down library state while another extraction is active.

## Best Possible Solution

Choose and document one library lifecycle model, then add implementation and regression coverage that prevents active PDF extraction from racing `_FPDF_DestroyLibrary()` while preserving custom WASM loading behavior.

## Reproduction Assessment

No high-confidence runtime reproduction is present yet. Source inspection does show the reported lifecycle path: the helper loads a library per call and then calls a destroy method that reaches `_FPDF_DestroyLibrary()`.

## Solution Assessment

Unclear until maintainers pick the lifecycle contract. Singleton/refcounting is plausible, but the implementation needs an explicit decision for `destroy()`, custom WASM options, and whether `extractPdfContent` should accept an existing library handle.
