## Summary

Keep open: current main still has the reported form-fill allocation path, and PDFium 7623's public header shows a version-2 `FPDF_FORMFILLINFO` interface extends well past the small buffer being allocated.

## What This Changes

The issue asks to fix the `FPDF_FORMFILLINFO` buffer/version contract by either using a smaller supported version or allocating enough zeroed storage for the pinned PDFium v2 interface.

## Best Possible Solution

Fix `#ensureFormHandle()` so the allocated and zeroed buffer matches the advertised PDFium form-fill interface, then add focused coverage for `renderForms`/fallback rendering against the pinned PDFium release.

## Reproduction Assessment

No runtime reproduction was established; this is source-reproducible from current main because `renderForms` reaches the 256-byte allocation and passes it as a version-2 PDFium form-fill struct. The PDFium 7623 header supports the reported mismatch risk.

## Solution Assessment

Yes on direction, but the exact variant needs ABI review: either lower the advertised interface version when headless callbacks are enough, or allocate a documented version-2-sized buffer for the pinned PDFium release.
