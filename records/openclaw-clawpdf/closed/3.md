## Label Justifications

- `P2`: The report is a bounded API bug that can silently omit requested extracted text but does not make the package unusable.
- `impact:data-loss`: The affected API can silently drop text from explicitly requested pages beyond the default cap.

## Summary

This should stay open: current `main` still slices explicit `pageNumbers` through the implicit 20-page default, and the docs describe `pageNumbers` as selected pages without documenting that hidden cap.

## What This Changes

The issue reports that explicit `pageNumbers` passed to `extractText` are capped by the default `maxPages` limit, silently omitting pages beyond 20.

## Best Possible Solution

Make page selection semantics explicit in the shared helper: keep the default page cap for implicit page ranges, honor all valid explicit `pageNumbers` when `maxPages` is omitted, and preserve caller-supplied `maxPages` as an explicit cap.

## Reproduction Assessment

Yes from source inspection: explicit valid `pageNumbers` are filtered and then sliced to `maxPages`, which defaults to 20 in the direct `extractText` path. I did not run a live PDF fixture, so this is source-reproducible rather than executed reproduction.

## Solution Assessment

Yes, the narrow repair is to distinguish an omitted page cap from an explicit one in the shared page-selection path and add regression tests for explicit page lists beyond 20.
