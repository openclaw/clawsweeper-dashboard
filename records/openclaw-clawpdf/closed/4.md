## Label Justifications

- `P2`: This is a real security/availability hardening issue in a bounded library API, but it is not an active outage or proven emergency.
- `impact:security`: Caller-controlled render dimensions can reach WASM allocation and PDFium rendering without validation.
- `impact:crash-loop`: Huge render options can plausibly trigger OOM, process termination, or hangs during bitmap allocation.

## Summary

Keep open: current main still derives PDF bitmap dimensions and allocation sizes directly from caller-provided render options, and the extraction fallback accepts an unbounded pixel budget before rendering. This is actionable security and availability hardening, not conservative cleanup.

## What This Changes

Validate `scale`, `width`, `height`, `maxPixels`, and `renderScale` and add hard render pixel limits before WASM bitmap allocation.

## Best Possible Solution

Add shared finite/positive render-option validation plus a hard maximum pixel or byte budget before PDFium allocation, with regression tests for `NaN`, `Infinity`, negative, and huge values.

## Reproduction Assessment

Yes, from source inspection: loading any valid PDF and calling `renderPage` with a huge or non-finite scale follows the allocation path before validation. I did not execute an OOM repro in the checkout.

## Solution Assessment

Yes, validation and a hard render budget are the narrow maintainable fix. The only open product choice is the exact cap values and error contract.
