Keep open: the PR fixes the chat/unread `0@status` sentinel path and includes useful real terminal proof, but it still leaves `Store.Status` using direct `time.Unix` conversions for message bounds despite claiming `status` coverage.

## What This Changes

The PR clamps out-of-range WhatsApp Apple-epoch timestamps to zero during import and on stored timestamp reads, with regression coverage for the `0@status` chat sentinel and already-populated chat archives.

## Best Possible Solution

Use one JSON-safe timestamp conversion for every persisted Unix timestamp surfaced through JSON, and keep the PR body and tests aligned with the commands that are actually covered.

## Reproduction Assessment

Yes, at source level: current main converts impossible positive timestamps into year 11001 values that `time.Time.MarshalJSON` cannot encode, and the PR body includes real terminal proof for the chat/unread failure. I did not run tests because this review must keep the checkout read-only.

## Solution Assessment

No: the import clamp and read-side clamp are maintainable, but the proposed fix should also route `Status` message bounds through the same clamp or narrow the `status` claim if no message-bound sentinel exists.
