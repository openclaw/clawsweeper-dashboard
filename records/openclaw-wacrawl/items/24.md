## Summary

Keep open: current main still saves the authoritative backup config with a direct `os.WriteFile`, so the reported atomicity/data-integrity problem remains real and narrow enough to fix.

## What This Changes

The issue requests making backup configuration saves atomic so an interrupted `backup init` cannot corrupt `backup.json` or lose age recipients.

## Best Possible Solution

Make SaveConfig use a same-directory temporary file plus close/sync and rename, preserving 0600 permissions and the existing LoadConfig/CLI contract.

## Reproduction Assessment

No live high-confidence torn-write reproduction is available. Source inspection does show the vulnerable path clearly: backup init calls SaveConfig, and SaveConfig rewrites backup.json directly with os.WriteFile.

## Solution Assessment

Yes: an atomic same-directory temp-file write with cleanup and rename is the narrowest maintainable fix, and it matches the package's staged rename approach for media restore.
