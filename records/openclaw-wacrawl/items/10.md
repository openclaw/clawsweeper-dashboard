## Summary

Current main and v0.2.7 still provide a local CLI, JSON commands, and crawlkit control metadata, not a web dashboard or scheduling/configuration UI. The request remains useful, and maintainer discussion is explicitly positive but unresolved around Birdclaw, wacli/wacrawl unification, and the privacy model for browser access to private WhatsApp data.

## What This Changes

Add a web interface for configuring crawls and viewing crawl status, history, search results, and related archive data.

## Best Possible Solution

Define one local-first companion/control-plane architecture that reuses crawlkit JSON commands, defaults to authenticated loopback-only access, preserves read-only WhatsApp source handling, and clearly assigns UI and scheduling ownership between wacrawl, Birdclaw, and any unified wacli experience.

## Reproduction Assessment

Not applicable: this issue requests a new web management and visualization capability rather than reporting broken existing behavior. Current-main source inspection confirms the capability is absent without requiring a runtime failure reproduction.

## Solution Assessment

Unclear: a web UI could solve the usability goal, but embedding it directly in wacrawl is not yet established as the narrowest maintainable approach. The safer direction is to settle the Birdclaw/wacli/wacrawl ownership and security model, then build on the existing crawlkit/JSON surface.
