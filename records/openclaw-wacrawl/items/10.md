## Summary

Keep open: current main and v0.2.7 still provide a local CLI and machine-readable control contract rather than a built-in web experience. The linked companion dashboard already demonstrates substantial user value, but the positive maintainer discussion leaves ownership across wacrawl, Birdclaw, and wacli unresolved, so this needs an explicit product and security decision rather than cleanup closure.

## What This Changes

Add a web interface for configuring crawls and viewing crawl status, history, archived conversations, and related data.

## Best Possible Solution

Adopt a local-first companion architecture that consumes a stable, versioned wacrawl control/query contract, binds to loopback with secure defaults, and leaves archive import and synchronization logic inside wacrawl while maintainers decide whether the UI ships through Birdclaw, wacli, or a separately maintained dashboard.

## Reproduction Assessment

Not applicable: this issue requests a new web management and viewing capability rather than reporting broken existing behavior. Source and release inspection clearly establish that the capability is currently absent.

## Solution Assessment

Unclear: the linked dashboard validates the idea, but directly incorporating it into wacrawl is not yet demonstrably the best ownership model. A stable local control/query boundary plus an explicitly chosen companion product would avoid duplicating crawler logic and prematurely coupling core to a web stack.
