## Summary

Keep open: current main still has a source-reproducible path where an out-of-range chat timestamp can reach `encoding/json` as a `time.Time`, so the structured `chats` and `unread` commands can still fail for affected archives.

## What This Changes

The issue reports that `wacrawl --json` can fail for `chats` and `unread` when a WhatsApp status pseudo-chat has a sentinel timestamp outside Go JSON's supported year range.

## Best Possible Solution

Normalize, omit, or encode unsupported chat timestamps as null before JSON output for `chats` and `unread`, while preserving human table behavior and the archive-only read workflow.

## Reproduction Assessment

Not fully from a live archive in this review; yes from source inspection. A regression test can insert or construct a chat whose stored timestamp decodes beyond year 9999, then verify `--json chats`, `--json chats --unread`, and `--json unread` do not fail.

## Solution Assessment

Yes: the narrow maintainable fix is timestamp sanitization at the JSON boundary or store conversion boundary with focused regression coverage, not a new flag or product mode.
