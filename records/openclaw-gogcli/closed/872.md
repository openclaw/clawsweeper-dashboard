## Summary

Keep open: current main still returns a generic corrupt-token decode error from the token store and the Google API auth path only converts missing keyring entries into re-auth, so the central auth-recovery bug remains unsolved and has no same-repo canonical fix PR.

## What This Changes

Treat present-but-undecodable stored OAuth tokens as unusable credentials that trigger re-authentication instead of causing every command for that account to fail fatally.

## Best Possible Solution

Add a secrets-level corrupt-token classifier for stored token JSON decode failures and map only that classifier plus `keyring.ErrKeyNotFound` to `AuthRequiredError`, with regression tests preserving fatal handling for operational keyring errors.

## Reproduction Assessment

Yes. The issue is source-reproducible on current main: corrupt token JSON returns a generic decode error and the caller only treats `ErrKeyNotFound` as re-auth; I did not execute tests because this review is read-only.

## Solution Assessment

Yes. The sentinel/classifier approach is the narrowest maintainable fix because it handles unusable stored credentials without masking genuine keyring availability or configuration failures.
