## What This Changes

The issue asks gog to avoid a misleading macOS Keychain refresh warning when the primary refreshed token persists but a later legacy or subject alias write returns duplicate-item.

## Best Possible Solution

Distinguish primary-token persistence from alias persistence, repairing duplicate alias writes or downgrading alias-only failures only after read-back confirms the primary refreshed token was stored.

## Reproduction Assessment

Yes. The reporter provided a live macOS Keychain path: let a stored access token expire, run an auth-using command, then compare the warning with `gog auth tokens export`; source inspection confirms the same broad warning and partial-write shape on current main.

## Solution Assessment

Yes. The narrow maintainable fix is in the auth/secrets persistence path, preserving real primary-token persist failures while making legacy/subject alias duplicate failures precise, repaired, or safely downgraded.

Fix https://github.com/openclaw/gogcli/issues/718. Current main logs `persist refreshed token metadata failed` for any `SetToken` error during refresh, while `SetToken` writes the primary token before default-client legacy and subject aliases; on macOS Keychain the primary refreshed access token/expiry can persist but a later alias write can return duplicate-item -25299, producing a misleading broad warning. Implement a narrow auth/secrets fix that preserves real primary-token persistence failures, but repairs, retries, or precisely downgrades alias-only duplicate failures after read-back proves the primary token was updated. Add focused unit coverage with a fake keyring/store for primary success followed by legacy or subject alias failure, and verify refresh state/logging no longer claims the whole metadata persist failed.
