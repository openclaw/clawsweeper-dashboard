## Summary

Keep open: current main and v0.27.0 still have no draft `--reply-all` or recipient inference. Source shows draft reply metadata is prepared, but `To` comes only from explicit `--to` while missing recipients are intentionally allowed for drafts, so the remaining fix needs a maintainer compatibility choice.

## What This Changes

Add recipient inference, draft `--reply-all`, or warning behavior for Gmail draft replies created with `--reply-to-message-id` or `--thread-id` and no `--to`.

## Best Possible Solution

Add an explicit draft reply recipient mode, likely `--reply-all`, and preserve no-recipient drafts unless maintainers intentionally approve scoped Gmail-UI-style default inference for reply targets.

## Reproduction Assessment

Yes from source inspection: draft reply metadata is prepared, but recipient lists are built only from explicit `--to`/`--cc`/`--bcc`, and missing `To` is allowed for drafts. I did not run a live Gmail reproduction in this read-only review.

## Solution Assessment

Unclear: adding draft `--reply-all` is the safest parity path, while implicit `To` inference or warning-only behavior needs maintainer judgment because no-recipient drafts are intentionally supported.
