- `impact:data-loss`: The affected behavior could delete or duplicate nearby document characters during mutation.

## Summary

Current main already routes the affected Docs find/replace and format match paths through the shared per-run text range engine, and the fix is contained in shipped release v0.26.0 and newer.

## What This Changes

The issue reports Google Docs find-replace and format match corruption when client-side matching skips non-text inline elements before the matched text.

## Best Possible Solution

Keep the shared `docsedit.FindTextRanges` implementation and direct affected users to upgrade to v0.26.0 or newer.

## Reproduction Assessment

Yes for the affected v0.25-era build: the issue includes a concrete live Docs reproduction, and source history shows that build predates the fix. No current-main failing path remains because the affected commands now use per-run document indices.

## Solution Assessment

Yes. Reusing the shared Docs text range engine is the narrow maintainable fix because it anchors matches on each text run's own Google Docs UTF-16 indices instead of recomputing from the first run.
