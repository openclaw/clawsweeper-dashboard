## Label Justifications

- `P1`: The PR changes auth and state path resolution for existing users, and a migration fallback bug can break credential discovery after upgrade.
- `merge-risk: 🚨 compatibility`: The path migration affects legacy config, data, state, and cache locations that existing installations rely on.
- `merge-risk: 🚨 auth-provider`: Credential, keyring, and service-account lookup behavior moves between config/data roots and can change auth provider resolution at runtime.

## Summary

This collaborator-authored PR should stay open for maintainer handling, and the diff still has a concrete upgrade-safety defect: the new explicit override checks key off environment variable presence instead of a successfully parsed override, so an invalid per-kind env var can suppress legacy fallback and break existing reads even when the override is rejected or not usable.

## What This Changes

The PR adds kind-aware config/data/state/cache path resolution with XDG defaults, `GOG_HOME`, per-kind `GOG_*_DIR` overrides, a global `--home` flag, docs, changelog, and migration-oriented tests across config, auth, backup, Gmail watch, and tracking state.

## Best Possible Solution

Keep the path model, but make legacy-fallback suppression depend on the same successfully resolved override state as `kindDir` and cover invalid override cases with focused regression tests before merge.

## Reproduction Assessment

Yes, by source inspection: `HasExplicitDataOverride`, `HasExplicitStateOverride`, and `HasExplicitConfigOverride` return true for any non-empty per-kind override env var, while `kindDir` rejects relative override paths later, so legacy fallback can be skipped based on an invalid override string. I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently patched: the requested XDG/GOG path model is the right direction, but the compatibility gate should be tied to valid active overrides rather than raw environment presence.
