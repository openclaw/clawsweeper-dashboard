The issue is still necessary: current `main` still normalizes relative dates for `calendar events`, while `calendar freebusy` sends `--from` and `--to` directly to the Google FreeBusy API.

## What This Changes

`gog calendar freebusy --from/--to` should accept the same RFC3339, date, and relative expressions already accepted by `gog calendar events --from/--to`.

## Best Possible Solution

Reuse the existing calendar time-range resolution path for freebusy, preserve the same timezone semantics as related calendar commands, update generated help/docs, and cover the request payload with focused tests.

## Reproduction Assessment

Yes from source inspection: `calendar events` normalizes the range before API use, while `calendar freebusy` forwards raw flag strings into `calendar.FreeBusyRequest`. I did not run the live command because that would require Google credentials and external API traffic.

## Solution Assessment

Yes: reusing `ResolveTimeRange` or the same underlying parser is the narrowest maintainable fix and avoids a second date-expression implementation.
