## Summary

Keep open: current main still omits paragraph-style `headingId` from `docs headings list` and `docs paragraphs list` JSON, although related named-range listing and run-link target metadata already exist.

## What This Changes

The issue requests exposing Google Docs heading, bookmark, and named-range anchor identifiers from existing Docs listing output so scripts can build in-document deep links.

## Best Possible Solution

Add optional `headingId` fields to headings and paragraphs JSON output, preserve current plain TSV stability, and point named-range deep-linking at the existing named-range list command unless maintainers choose a broader output contract.

## Reproduction Assessment

Yes. Source inspection shows current main does not carry `ParagraphStyle.HeadingId` into either headings or paragraphs JSON output; I did not run live Docs commands because that requires Google credentials.

## Solution Assessment

Yes for the central headingId gap: adding optional JSON fields to the existing enumerator models is the narrowest maintainable fix. The named-range part should reuse the existing `docs named-range list` surface unless maintainers explicitly want duplicated paragraph output.
