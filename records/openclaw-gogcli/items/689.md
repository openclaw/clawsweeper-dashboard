## Summary

Keep open: current main has Drive changes start/list/watch/stop, but no Drive-side HTTPS receiver, callback runner, page-token state, or auto-renew flow. The request fits supported Drive automation, but it needs maintainer design review because it adds long-running server behavior and externally triggered command execution.

## What This Changes

Request a new `gog drive changes serve` command that receives Drive push notifications, validates the channel token, lists changes, persists page tokens, optionally filters by file, invokes a callback, and renews the watch channel.

## Best Possible Solution

Settle a narrow Drive receiver design first, then implement it with explicit auth/TLS behavior, atomic page-token state, callback safety, focused httptest coverage, generated command docs, and live Google Drive proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and generated docs confirm the current Drive changes surface has watch/list/stop primitives but no receiver command.

## Solution Assessment

Unclear until maintainers choose the product and security shape. Mirroring Gmail watch serve is plausible, but Drive's direct webhook headers, HTTPS/TLS expectations, page-token state, callback execution, and auto-renew behavior need a deliberate design.
