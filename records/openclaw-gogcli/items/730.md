## Summary

Current main still preserves `<br>` literally in table-cell text, while adjacent merged work establishes inline table-cell Markdown as supported behavior, so this is a valid narrow bug that remains necessary.

## What This Changes

Convert `<br>` and, where unambiguous, Markdown hard breaks inside Docs Markdown table cells into in-cell line breaks.

## Best Possible Solution

Normalize recognized `<br>`, `<br/>`, and `<br />` forms within table cells to newline characters before insertion, preserve existing inline formatting, and add parser plus request-generation tests; support Markdown hard breaks only if the existing table grammar can express them unambiguously.

## Reproduction Assessment

Yes at high confidence from current-main source: `parseTableRow` returns the exact trimmed cell string, so `Alice<br>Bob` remains literal through the converter path. The issue also provides a concrete live CLI reproduction for v0.23.0 and current main.

## Solution Assessment

Yes for narrowly converting recognized `<br>` variants in table cells because that extends the existing supported inline-cell conversion rather than creating a parallel implementation. The optional physical-newline hard-break request should wait unless a clear table syntax and regression test can define it.
