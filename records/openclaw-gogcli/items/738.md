## Summary

Current main still has the reported gap: calendar create supports attachments, but calendar update exposes no attachment mutation flags and its PATCH helper does not enable attachment support. The request is a small, vision-aligned Calendar capability, but maintainers should choose replace-versus-append semantics before implementation.

## What This Changes

Add attachment replacement and append support to calendar event updates, including the required attachment-aware Calendar API PATCH parameter.

## Best Possible Solution

Add explicit --attachment replacement and --add-attachment append operations, preserve Google-provided metadata during append, and enable SupportsAttachments only for attachment-mutating PATCH requests.

## Reproduction Assessment

Yes at source level. Current main defines --attachment only for create, so passing it to calendar update necessarily produces an unknown-flag error; no live Google account is needed to establish that CLI gap.

## Solution Assessment

Mostly yes. Mirroring create is maintainable, but update should use separate replace and append flags so existing attachments are never discarded through ambiguous behavior.
