## Summary

Keep this PR open: current main still has the per-cell Docs `batchUpdate` loop, and the latest branch is a useful narrower fix, but it still has a merge-blocking 500-request cap gap for large table-cell batches plus insufficient inspectable live proof.

## What This Changes

The PR batches Docs markdown body requests with a 500-request splitter, batches each native table's cell-content requests into one request list, removes the old per-cell index bookkeeping, and updates table append tests.

## Best Possible Solution

Use the safer server-readback-per-table approach, but split oversized cell-content batches with focused tests and require inspectable redacted live proof before merge.

## Reproduction Assessment

Yes from source, but not by executing a live Docs repro in this review. Current main clearly sends one `batchUpdate` inside the cell loop, and the PR head clearly batches those cell requests into one per-table request list.

## Solution Assessment

No, not yet. The revised server-readback direction is safer than the reverted index-prediction approach, but the cell-content batch needs the same 500-request cap handling and the proof/test gaps should be closed before merge.
