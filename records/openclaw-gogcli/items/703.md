## Summary

Keep open: current main still treats trailing `{#slug}` as ordinary heading text in the local Docs markdown path, and the existing heading-link resolver only handles generated slugs after whole-document Drive markdown import.

## What This Changes

Support Pandoc-style explicit heading IDs in `docs write --markdown` so trailing `{#slug}` is stripped from headings and same-document `#slug` links target that heading on both tab and whole-document paths.

## Best Possible Solution

Add shared explicit-heading-anchor parsing metadata, strip the attribute from rendered heading text, prefer explicit slugs when rewriting same-document links, and cover both tab and whole-document markdown paths with focused unit tests plus live Docs proof.

## Reproduction Assessment

Yes from source inspection: `parseHeading` keeps `{#slug}` in the heading content and the formatter inserts that text into the Docs request. I did not run the live Google Docs sequence during this read-only review.

## Solution Assessment

Yes, supporting explicit heading anchors inside the existing markdown conversion flow is the narrow maintainable direction. The safer implementation is shared parsing/link-rewrite logic across both paths, not a new flag or separate converter mode.
