## Summary

The latest branch has strong real-account proof and correctly isolates read commands from the new write service, but it is not ready to merge: the write client reuses the same stored YouTube service credential without validating that the token contains the full mutation scope, and the branch still includes a 594-line agent execution plan. This PR remains necessary because current main has only read-oriented YouTube commands, but the authorization upgrade path and repository scope need repair first.

## What This Changes

The PR adds YouTube subscription list/subscribe/unsubscribe and playlist create/add/remove/delete commands, routes mutations through a full-scope OAuth client, adds service-isolation and command tests, and includes an implementation-plan document.

## Best Possible Solution

Keep read operations on the existing read-only service, give mutation commands an explicit full-scope authorization path that detects and upgrades insufficient stored consent with focused tests, and remove the transient implementation plan before merge.

## Reproduction Assessment

Yes for the merge blocker: authorize an account with only the existing `youtube.readonly` scope, then invoke a new mutation command; source inspection shows no preflight scope check or consent-upgrade path before the API request. The contributor’s proof covers a successfully write-authorized account, not this upgrade case.

## Solution Assessment

No, not yet. The separate read/write service routing is the right foundation, but it must validate or upgrade stored consent before mutation and should not ship the transient agent plan.
