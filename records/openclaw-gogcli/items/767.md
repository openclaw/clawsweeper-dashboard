## Label Justifications

- `merge-risk: 🚨 compatibility`: Existing accounts authorized only for YouTube reads can fail when first using the new mutations unless the PR provides an explicit scope-upgrade path.

## Summary

The final design preserves read-only routing and the contributor supplied credible real-account proof, but existing stored read-only credentials are reused without checking whether they grant the new write scope. That upgrade failure path, a formatting failure, and transient implementation-plan content should be fixed before merge.

## What This Changes

The PR adds YouTube subscription list, subscribe, and unsubscribe commands plus playlist create and add commands, routing mutations through a separate full-scope OAuth client while retaining read-only clients for existing commands.

## Best Possible Solution

Keep the separate least-privilege write client, reject stored tokens that lack the required write scope with actionable reauthorization guidance, cover fresh and upgraded account paths, and remove transient planning material.

## Reproduction Assessment

Yes for the upgrade defect: authorize an account with the existing `youtube.readonly` scope, then invoke a new mutation; source inspection shows the stored token is reused without a required-scope comparison. The new feature itself is not a bug reproduction.

## Solution Assessment

No, not yet: separating read and write clients is the correct foundation, but the implementation must add an explicit existing-token authorization upgrade path instead of relying on a downstream API error.
