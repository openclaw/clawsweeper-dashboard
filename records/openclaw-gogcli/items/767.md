## Summary

The latest head fixes the original least-privilege regression, but the write constructor cannot obtain broader consent from an existing read-only token, so standard users can reach YouTube mutations with insufficient authorization; real-account proof is also still missing.

## What This Changes

Adds YouTube subscription list, subscribe, and unsubscribe commands plus playlist create and add commands, routing mutations through a new full-scope YouTube client while retaining read-only routing for existing commands.

## Best Possible Solution

Add an explicit authorizable write-scope path that detects insufficient stored grants and guides or performs reauthorization while preserving read-only authorization for users who never invoke mutations, with focused fresh-token and upgrade-token tests plus redacted live proof.

## Reproduction Assessment

Yes for the review finding: authorize YouTube through the current read-only service, then invoke any new mutation command; the token loader reuses that refresh token without verifying the full scope, so no broader consent is obtained. A real-account run is still needed to capture Google's resulting API response.

## Solution Assessment

No. The final split preserves least privilege for reads but provides no supported way for mutation commands to acquire write grants; the narrow solution is an explicit scope-upgrade or separately authorizable write-service flow.
