## Label Justifications

- `P2`: The command can falsely report a successful document write, but the known blast radius is limited to Markdown list input for one Docs editing command.
- `impact:other`: The issue concerns a document mutation reported as successful even though the requested content is not applied.

## Summary

Current main still has a credible false-success path for Markdown lists in `docs cell-update`: cell-specific handling removes the paragraph terminator needed by the shared list formatter, while an ineffective update can still return success and print `updated=true`. This is a focused, valid bug in an existing command and should remain open for repair.

## What This Changes

Make `gog docs cell-update` avoid silently succeeding when Markdown list content produces no effective cell update.

## Best Possible Solution

Preserve the paragraph boundary required for native list formatting while keeping every mutation within the selected cell, and return a clear error whenever nonempty Markdown would otherwise produce no effective update.

## Reproduction Assessment

Yes at the source level: list formatting depends on newline-terminated paragraph ranges, while cell-update trims that boundary and can still return success after an ineffective request set. The reporter also provides a concrete live command, though this review did not independently call Google Docs.

## Solution Assessment

Yes. A focused correction to cell-scoped request ranges, an explicit no-effective-update guard, and regression tests are narrower and more maintainable than adding a new command, flag, or parallel Markdown implementation.
