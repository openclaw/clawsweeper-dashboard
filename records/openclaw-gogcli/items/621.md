## Summary

Keep open: current main still routes most persisted credentials, auth state, downloads, and backup config through config-style paths, and the linked discussion plus related open feature request show this is live design work rather than stale cleanup.

## What This Changes

Request kind-aware XDG path resolution for gogcli credentials, keyring data, mutable state, cache, downloads, and backup config with legacy read compatibility.

## Best Possible Solution

Add centralized kind-aware path resolvers plus tested legacy read/write-new migration for credentials, keyring, service-account files, tracking, Gmail watch state, cache, downloads, backup config, and docs.

## Reproduction Assessment

Yes for the current source behavior: inspection shows the relevant paths either derive from config.Dir(), which only reads XDG_CONFIG_HOME, or use ~/.gog/os.UserCacheDir directly. I did not run a mutating path test because this review is read-only.

## Solution Assessment

Unclear as an implementation plan until maintainers decide the migration, downloads, and GOG_HOME precedence semantics. The direction is maintainable if it lands as one centralized resolver with focused compatibility tests rather than scattered call-site rewrites.
