## Summary

Keep open: current main has Docs comment CRUD and literal Docs text matching internals, but it does not provide a `docs comments locate` or `docs find-range` command, and the related range-finding primitive remains separate work.

## What This Changes

Request a new `gog docs comments locate` subcommand that maps a comment's `quotedFileContent.value` to current Google Docs body ranges with tab, whitespace, and HTML-entity handling.

## Best Possible Solution

Co-design this with the open find-range primitive, then add a focused locate command that reuses shared range resolution, Drive comment fetching, tab resolution, JSON/TSV output, orphan handling, tests, and live Google proof.

## Reproduction Assessment

Not applicable as a feature request. Source and generated-doc inspection against current main show no `docs comments locate` or `docs find-range` command, so the proposed workflow still requires custom scripting.

## Solution Assessment

Unclear until maintainers settle the command and exit-code contract. The maintainable direction is likely a shared find-range implementation that `docs comments locate` calls rather than a separate one-off resolver.
