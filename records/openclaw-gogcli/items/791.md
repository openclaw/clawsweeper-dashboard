## Summary

Keep open: the read-only backup repository split is the right direction, but the PR currently removes released `backup status` and `backup verify` flags in a way that can break existing scripts before they reach the safer read path.

## What This Changes

This PR separates backup read repository preparation from write initialization, wires status/verify to read flags, adds regression tests, and updates backup docs, generated command docs, and the changelog.

## Best Possible Solution

Land the read-only backup repository preparation while preserving the released no-op status/verify flags as hidden/deprecated compatibility aliases and proving both the compatibility and non-mutating paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main read commands call `ensureRepo`, and that helper can fall back to `git init` after a missing or failed clone. I did not execute it because this review is read-only.

## Solution Assessment

No, not as written: the repository-preparation split is maintainable, but dropping released status/verify flags is not the narrowest compatible fix. The safer path is to keep those flags accepted and ignored while adding `--no-pull`.
