## Summary

The PKCE implementation is coherent and covers every production installed-app OAuth path, but this external PR still lacks the required after-fix live Google authorization and token-exchange proof. The documented restart requirement is intentional and reinforced by a current area owner’s migration-doc commit, making it a visible compatibility risk rather than a patch defect.

## What This Changes

The PR adds S256 PKCE to browser, manual, remote, and account-manager OAuth authorization and token exchange, persists short-lived manual verifiers, documents the upgrade restart requirement, and adds focused tests.

## Best Possible Solution

Keep the consistent fail-closed PKCE binding and documented restart requirement, then merge only after redacted live Google proof shows a fresh installed-app authorization callback and successful refresh-token exchange.

## Reproduction Assessment

Yes from source: current main constructs installed-app authorization requests and exchanges codes without PKCE, and the PR’s focused tests exercise the missing challenge/verifier binding. The actual Google provider path has not been reproduced in the submitted evidence.

## Solution Assessment

Yes. Applying `golang.org/x/oauth2` S256 challenge and verifier options consistently across every installed-app flow is the narrow standard fix, and the migration behavior is explicitly documented on the maintainer-updated head.
