## Summary

Keep open: current main still has the HTML-fragment signature bug, and the updated PR head now addresses the prior regex boundary concern. I found no definite line-level correctness blocker, but the shared classifier change still needs maintainer acceptance because it affects several Gmail content paths.

## What This Changes

The PR broadens Gmail HTML detection to fragments, reuses that classifier in backup export, preserves HTML-looking text/plain content for quote replies, and adds `LooksLikeHTML` regression tests.

## Best Possible Solution

Keep the fragment-aware classifier at the current PR head if maintainers accept the shared behavior; otherwise scope fragment detection to signature/backup paths while leaving display and reply classification narrower.

## Reproduction Assessment

Yes. Source inspection shows current main accepts plain text or HTML signature files, but a `<style>` or `<table>` fragment falls through the narrow document-level detector and gets escaped; the PR body also includes live raw RFC822 before/after proof.

## Solution Assessment

Yes, with maintainer acceptance of the broader shared-helper behavior. The current head fixes the earlier regex boundary flaw and adds focused regression coverage; the safer alternative is to limit fragment detection to the signature and backup paths.
