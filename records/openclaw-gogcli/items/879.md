Keep open. Current main and v0.30.0 still use the narrow document-level detector, while this PR now includes the boundary fix, reply quote handling, regression tests, and before/after raw RFC822 proof needed for normal maintainer review.

## What This Changes

The PR broadens `gmailcontent.LooksLikeHTML` to detect HTML fragments, removes duplicate Gmail backup fragment detection, carries HTML-looking text/plain replies into `BodyHTML`, and adds regression tests.

## Best Possible Solution

Land the tight shared fragment detector with regression tests and real Gmail proof, or scope fragment detection to signature/backup paths if maintainers do not want display and reply behavior to change.

## Reproduction Assessment

Yes from source inspection: docs allow HTML `--signature-file`, but current main does not classify `<style>` or `<table>` fragments as HTML before escaping. The PR body also includes before/after raw RFC822 proof for the real Gmail send path.

## Solution Assessment

Mostly yes. Consolidating fragment detection in `LooksLikeHTML` fixes the signature bug and removes duplicate backup logic, with the remaining maintainer choice being whether the shared classifier behavior is acceptable across all callers.
