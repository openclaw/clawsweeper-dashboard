## Label Justifications

- `P2`: The issue is a bounded safety bug in existing CLI commands, with potential local file loss but limited blast radius.
- `impact:data-loss`: The reported behavior can silently truncate or replace user-selected local output files.

## Summary

Keep open: current main still routes Drive download, Drive tab export, and Slides thumbnail output through a helper that truncates existing files without an `--overwrite` opt-in, while Photos already uses the no-clobber helper path. The fix is narrow, but maintainers should confirm the default behavior change because existing scripts may rely on overwriting.

## What This Changes

The issue asks Drive download, Drive tab export, and Slides thumbnail outputs to refuse existing local files by default unless the user explicitly opts into overwrite behavior.

## Best Possible Solution

Adopt one consistent local-output policy for these commands, ideally no-clobber by default with an explicit `--overwrite` opt-in, covering normal Drive downloads, Drive tab export, and Slides thumbnails with focused tests and generated command docs.

## Reproduction Assessment

Yes, source-reproducible: call `downloadDriveFile`, `runDocsTabExport`, or `downloadSlidesThumbnail` with an existing destination and a successful mocked response; current main reaches `createUserOutputFile`, which opens with truncation. I did not execute a runtime reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers confirm the default behavior change; routing these commands through `openUserOutputFile` with an explicit overwrite flag is the narrow maintainable fix once that policy is accepted.
