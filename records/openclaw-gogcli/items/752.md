## Summary

Keep open: current main still has the reported command-surface mismatch, with `gmail search` emitting thread IDs while `gmail archive` treats positional IDs as message IDs and calls message batch modify. I found no canonical related issue or PR that already owns the remaining work.

## What This Changes

The issue requests that `gog gmail archive` handle thread IDs or warn when it only archives one message from a multi-message thread.

## Best Possible Solution

Maintainers should choose a compatibility-safe UX for thread-aware archive, then implement it narrowly by reusing Gmail thread modify semantics, updating generated command docs/tests, and collecting live Gmail proof on a multi-message inbox thread.

## Reproduction Assessment

Yes for the command mismatch: current source shows `gmail search` emits thread IDs and `gmail archive` uses message batch modify for positional IDs. The reporter also gave a concrete v0.24.0 multi-message thread workflow, but I did not run a live Gmail account check here.

## Solution Assessment

Unclear until maintainers choose the CLI contract. The narrow code path is clear, but the best UX could be default thread detection, an explicit thread mode, or a warning-only compatibility path.
