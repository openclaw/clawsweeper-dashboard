## Summary

Current main still matches the reported failure shape: the Docs markdown/table path sends a separate documents.batchUpdate from inside the table-cell loop, so the issue remains necessary and should stay open for a focused batching fix.

## What This Changes

The issue requests batching Google Docs tab markdown table-cell writes so table-heavy markdown does not consume one Docs API write call per cell.

## Best Possible Solution

Refactor the tab markdown table path to stop per-cell API submissions, batching table-cell text/style requests into bounded chunks while preserving existing tab IDs, inline formatting, table placement, and image cleanup behavior.

## Reproduction Assessment

Yes for the central per-cell API-call pattern: current main still calls Documents.BatchUpdate inside the table-cell loop reached by --replace --markdown --tab. The downstream 429 quota failure was not live-reproduced in this read-only review.

## Solution Assessment

Mostly yes: batching request submission is the right fix, but the implementation should be conservative about table cell index calculation and chunking rather than assuming the entire renderer can be made single-call without proof.
