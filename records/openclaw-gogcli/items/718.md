## Summary

Keep this issue open: current main still has the broad refresh-persist warning path, and the token store still writes the primary token before legacy/subject aliases, so the reported partial-success/misleading-warning case remains plausible and unsolved.

## What This Changes

The issue reports that macOS Keychain can persist refreshed OAuth token metadata while `gog` still logs a misleading `persist refreshed token metadata failed` warning for duplicate-item alias errors.

## Best Possible Solution

Fix the token persistence path so primary-token refresh failures stay visible, while verified primary success plus benign legacy/subject alias duplicate failures get repaired, downgraded, or logged with precise alias context.

## Reproduction Assessment

Yes, maintainers with macOS Keychain have a high-confidence reproduction path from the report: let a stored access token expire, run an auth-using command, then compare the warning with `gog auth tokens export`. This review verified the source path but did not execute the live Keychain repro.

## Solution Assessment

Yes, the right solution is a narrow auth/secrets fix rather than a new option or broad logging change. The fix should distinguish primary token persistence from legacy/subject alias persistence and add regression coverage for partial alias failure.
