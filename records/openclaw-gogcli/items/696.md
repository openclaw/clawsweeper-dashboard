## Summary

Keep open. Current `main` still sends tabbed markdown replacement through the local parser/formatter path, and that path does not parse or emit nested list levels; I did not find a merged fix or canonical open item that supersedes this nested-list gap.

## What This Changes

The issue reports that `gog docs write --replace --markdown --tab=<id>` flattens nested markdown child list items into plain paragraph text instead of preserving Google Docs bullet nesting levels.

## Best Possible Solution

Teach the local markdown writer to preserve nested list levels for tab-scoped replacement, preferably reusing the existing Docs API leading-tab bullet behavior, with regression tests and live Google Docs proof before merge.

## Reproduction Assessment

Yes. The reporter supplied a public repro and current source inspection corroborates it: indented list markers are not parsed as list items, and the formatter has no nesting-level handling for parsed list items.

## Solution Assessment

Yes. The narrow maintainable fix is to make the existing local tab markdown converter match the non-tab markdown semantics for nested lists, not to add a new command surface or config flag.
