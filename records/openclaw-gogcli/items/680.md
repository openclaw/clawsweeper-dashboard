## Label Justifications

- `P2`: This is a normal-priority Gmail draft update bug with silent user-visible attachment loss but a bounded command surface.
- `impact:data-loss`: The reported behavior silently removes draft attachments when a user updates body or headers without passing `--attach` again.

## Summary

This issue is still necessary: current main still rebuilds `gmail drafts update` from the flags provided on that invocation, so omitting `--attach` produces an updated draft with no carried-forward attachments.

## What This Changes

The issue asks for `gmail drafts update` to preserve an existing draft's attachments when `--attach` is omitted instead of silently replacing them with an empty attachment set.

## Best Possible Solution

Implement attachment preservation for omitted `--attach` by carrying existing draft attachment bytes into the rebuilt MIME message, while making explicit attach replacement and any attachment-clearing behavior clear in tests and docs.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, and source inspection shows omitted `--attach` flows to an empty attachment slice while the existing draft fetch does not collect or reattach attachment parts.

## Solution Assessment

Unclear until maintainers choose final semantics. Preserve-by-default is the narrowest data-loss fix, but the update command should also define how callers intentionally remove or replace existing attachments.
