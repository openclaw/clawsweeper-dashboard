## Summary

Keep open. Current main still lacks `--thread-id` on Gmail draft create/update while `gmail send` supports it; the existing compose helper already has thread-aware reply resolution, so this is a narrow, actionable parity gap.

## What This Changes

Add `--thread-id` to `gog gmail drafts create` and `gog gmail drafts update`, mutually exclusive with `--reply-to-message-id`, using the latest message in the thread for reply headers.

## Best Possible Solution

Implement the flag by reusing the existing thread-aware reply helper, add mutual-exclusion validation and tests, update generated command docs, and require redacted live Gmail proof before merging.

## Reproduction Assessment

Yes, at source level: current main defines `--thread-id` for `gmail send` but not for either draft command, and the generated docs match that gap. The Gmail UI threading effect was not live-tested in this review.

## Solution Assessment

Yes: adding the flag and routing it into the existing `prepareComposeReply` thread path is the narrowest maintainable fix. It should keep `--reply-to-message-id` mutually exclusive and avoid changing unrelated draft update fallback behavior.
