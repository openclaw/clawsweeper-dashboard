## Summary

Keep open: current main still requires a local file upload and an `anyone` Drive permission before inline insertion; restricted Workspace orgs only get the existing error or link fallback, so the inline embed gap remains.

## What This Changes

Request `gog docs insert-image` support an inline-image path that does not require public Drive sharing, such as a direct URL, domain-scoped permission, or reusable uploaded file ID.

## Best Possible Solution

Add a maintainer-approved inline mode, preferably a `--url` option mutually exclusive with `--file`, while preserving the existing local-upload flow and link fallback, with unit coverage and live Google proof for the chosen restricted-sharing path.

## Reproduction Assessment

Yes from source inspection, but not from a live enterprise Workspace run here. Current main creates an `anyone` Drive permission before inline insertion, so a tenant that rejects that permission follows the reported `publishOutNotPermitted` error path unless link fallback is selected.

## Solution Assessment

Unclear until maintainers choose the supported contract. `--url` looks like the narrowest maintainable option because the internal request builder already accepts an image URI, while domain sharing and fileId reuse need stronger API and UX decisions.
