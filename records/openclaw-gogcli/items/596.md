## Summary

Keep open: current main still persists the Google OAuth `client_secret` in `credentials*.json`, while the Zoom path already stores its client secret in the keyring. The request is valid security hardening, but it changes the credential storage contract and likely needs maintainer judgment on the default flip, legacy opt-out, diagnostics, and sequencing with https://github.com/openclaw/gogcli/issues/597.

## What This Changes

Request to make `gog auth credentials set` store Google OAuth `client_secret` in the keyring by default, mirroring Zoom, with legacy plaintext fallback or opt-out and diagnostic updates.

## Best Possible Solution

Adopt a single Google OAuth credential contract where new writes keep only non-sensitive client metadata on disk, store `client_secret` in the keyring, preserve legacy plaintext reads, remove both layers together, and expose diagnostics for missing or keyring-backed secrets after maintainers choose the default/opt-out policy.

## Reproduction Assessment

Yes for the reported storage shape: source inspection shows `auth credentials set` parses Google OAuth JSON and writes a `ClientCredentials` struct containing `client_secret` to `credentials*.json`. I did not execute the CLI because this review is required to keep the checkout read-only.

## Solution Assessment

Unclear as a final product decision: keyring-backed new writes with legacy plaintext fallback is the narrow maintainable direction, but the default flip, opt-out flag, diagnostics, and sequencing with the keyring-locking issue need maintainer approval before implementation.
