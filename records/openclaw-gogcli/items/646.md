## Summary

Keep open. Current main and v0.19.0 have a `docs sed` table-cell replacement workaround, but they do not have the requested first-class `gog docs cell-update` command, and VISION.md says new API surfaces should be discussed first.

## What This Changes

The issue requests a first-class `gog docs cell-update` command that replaces or appends content inside one Google Docs table cell without disturbing surrounding document content.

## Best Possible Solution

Decide the product shape first: either add a tab-aware `docs cell-update` command with `--content-file`, append/replace semantics, tests, and live Docs proof, or document `docs sed` as the supported cell-edit path and track the metadata-loss incident separately.

## Reproduction Assessment

No, not for the destructive `find-replace` incident; this review only source-verified that `cell-update` is absent on main and v0.19.0. A live Google Docs reproduction is still needed before treating the metadata loss as a strict bug-fix lane.

## Solution Assessment

Unclear. The requested command is plausible and fits the Docs area, but `docs sed` already has cell targeting, so maintainers should first choose whether a new first-class wrapper is the best user-facing surface.
