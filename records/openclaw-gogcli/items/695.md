## What This Changes

The PR adds a `gog slides insert-image` subcommand, implementation, and mocked unit tests for placing a sized local PNG/JPG/GIF on an existing slide via Drive upload and Slides `createImage`.

## Best Possible Solution

Land a narrow `slides insert-image` command only after it validates the target slide before Drive setup, treats temporary public-file cleanup as a checked security boundary, and includes inspectable live Google Slides proof.

## Reproduction Assessment

## Solution Assessment

No. A small Slides primitive is a good fit, but the implementation should delay Drive setup until after slide validation and handle temporary-public cleanup failures before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Delay Drive setup until after slide lookup:** `internal/cmd/slides_insert_image.go:105-108`
  - body: The command constructs the Drive service before it loads the presentation and checks whether `slideId` exists. That makes the new missing-slide test's `newDriveService` fatal path fire, and in real use it can report a Drive auth/setup error before the advertised slide-not-found validation; move Drive service creation below the slide existence check.
  - confidence: 0.93
- **[P1] Surface failures when deleting the public temp image:** `internal/cmd/slides_insert_image.go:135-137`
  - body: After granting an anyone-reader permission, cleanup is a deferred Drive delete using the command context and the error is ignored. If the context is canceled or Drive deletion fails, the command can still report success while the uploaded local image remains public; use a cleanup context that survives cancellation and return or report cleanup failure, with coverage for the failure path.
  - confidence: 0.86
