## Summary

Keep open: current main still routes scoped OAuth clients through the generic retry transport, and that transport returns all non-429 4xx responses without a refresh/retry, so the reported transient insufficient-scope 403 path is not already handled.

## What This Changes

The issue requests an internal retry or clearer error handling for intermittent Google API 403 insufficient-scope responses that succeed on retry with the same refresh token.

## Best Possible Solution

Add a narrow auth-aware retry path that refreshes or remints once for replayable Google API 403 insufficient-scope responses, then surfaces a clearer true-scope-missing error if the refreshed attempt also fails.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show that if Google returns a 403 insufficient-scope response on first use, current main returns it without a refresh/retry.

## Solution Assessment

Yes, a narrowly scoped refresh/remint retry with targeted insufficient-scope detection is the maintainable direction. It should not broaden into retrying all 403s or changing stored scope metadata based only on requested scopes.

Fix the intermittent insufficient-scope 403 handling for stored OAuth account clients. In internal/googleapi, add a narrow once-only retry path for Google API 403 responses whose structured/body text indicates insufficient authentication scopes: force a fresh access-token mint or otherwise bypass the cached access token, retry only when the request body is replayable, and preserve normal behavior for true permission denials, Workspace policy failures, direct access tokens, service accounts, ADC, and non-replayable requests. Add regression tests with a fake token source/transport showing first insufficient-scope 403 then success after refresh, plus a test proving ordinary 403s are not retried.
