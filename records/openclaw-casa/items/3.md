## Summary

Current main already provides the central health-check surface: `GET /health` returns JSON server status, is documented, and is wired into the CLI. The richer tunnel/node heartbeat fields do not map to an existing Casa tunnel subsystem in the current source, so there is no separate concrete core implementation to keep this issue open for.

## What This Changes

The issue requests a `GET /health` or `/status` endpoint returning JSON for Casa health, optionally including tunnel status, connected nodes, and heartbeat data.

## Best Possible Solution

Keep the existing `/health` endpoint as Casa's health-check surface; add richer tunnel or node metrics only if Casa later gains a real tunnel/heartbeat subsystem to report.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing behavior report. Source inspection confirms current main already has the requested endpoint path and JSON response.

## Solution Assessment

Yes for the central request: the existing local API route, docs, and CLI command are the narrow maintainable solution. The proposed tunnel/node heartbeat fields should wait for an actual Casa tunnel state model before becoming a separate request.
