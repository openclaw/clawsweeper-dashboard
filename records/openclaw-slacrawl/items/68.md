## Summary

Keep open: current main and latest release v0.7.3 still append a `message_events` row for every successful message upsert, and purge still only removes event rows for messages selected for deletion. The report is actionable, but the dedupe scope and retained-history compaction semantics need maintainer policy direction before a safe fix PR.

## What This Changes

The issue requests suppressing duplicate `message_events` snapshots from desktop watch refreshes and optionally adding purge support to compact retained event history.

## Best Possible Solution

Maintainers should choose the event-history policy, then implement latest-snapshot dedupe and/or an explicit purge compaction flag with preview/reporting, docs, and focused store plus CLI regression tests.

## Reproduction Assessment

No live high-confidence reproduction was performed. Source inspection does show the growth mechanism: desktop watch repeatedly syncs desktop state and each successful message upsert appends a new event row.

## Solution Assessment

Unclear until maintainers choose the retention semantics. Latest-event dedupe is a narrow candidate to prevent new duplicates, but retained-history compaction is destructive and should not be selected automatically.
