## Summary

Keep open: current main and v0.7.3 still append a `message_events` row for every message upsert, and purge only removes event history for messages selected for deletion. The request needs maintainer direction on event-history dedupe and retained-history compaction before implementation.

## What This Changes

Request to suppress identical message event snapshots from repeated desktop watch upserts and add purge support for compacting retained message event history.

## Best Possible Solution

Choose the event-retention policy, then implement latest-event dedupe and/or explicit purge compaction with preview/reporting and focused store plus CLI regression tests.

## Reproduction Assessment

Partly: source inspection gives a high-confidence path for duplicate event insertion because every message upsert appends an event and watch repeatedly upserts desktop state. The several-GB live growth itself was not reproduced during this read-only review.

## Solution Assessment

Unclear until maintainers choose the retention semantics. Latest-event dedupe is a narrow candidate for preventing new duplicates, but retained-history compaction is destructive and needs an explicit product decision.
