- `P2`: The report shows large local database growth from desktop watch and retention behavior without data loss, security impact, or core runtime outage.

## Summary

Keep open: current main and v0.7.3 still append a `message_events` row for every message upsert, and purge still only removes event history for messages selected for deletion. The remaining work needs maintainer direction because deduping snapshots and compacting retained history change the event-retention policy.

## What This Changes

The issue asks Slacrawl to avoid accumulating duplicate `message_events` snapshots from desktop watch and optionally add purge compaction for retained event history.

## Best Possible Solution

Define the event-history retention policy, then implement latest-snapshot dedupe and any retained-history compaction with preview/reporting plus focused store, CLI, and docs coverage.

## Reproduction Assessment

Yes from source for repeated identical event rows: `watch` repeatedly runs desktop sync, desktop ingest calls `UpsertMessage`, and `upsertMessage` always appends `message_events`. I did not reproduce the several-GB live database in this read-only review.

## Solution Assessment

Unclear until maintainers choose retention semantics. Latest-event dedupe is a narrow candidate for preventing new duplicate snapshots, but retained-history compaction is destructive and should be an explicit policy choice.
