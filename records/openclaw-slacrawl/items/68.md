## Summary

Keep open: current main still appends a `message_events` row for every successful message upsert, and purge only removes event rows for messages selected by the cutoff. The request is not implemented, and because docs describe watch as appending event history, dedupe and retained-history compaction need maintainer policy direction.

## What This Changes

The issue asks Slacrawl to avoid unbounded duplicate `message_events` growth from desktop/watch ingestion and optionally add purge compaction for retained event history.

## Best Possible Solution

Maintainers should decide the intended event-history policy, then implement duplicate snapshot suppression and retained-history compaction in the store and purge surfaces if that policy is accepted.

## Reproduction Assessment

Yes at source level: current main appends an event row on each successful upsert, and purge only removes events for purged messages. I did not run a live desktop watch loop against a real Slack Desktop cache.

## Solution Assessment

Unclear: the proposed SQL shapes are plausible, but the best policy depends on whether maintainers want `message_events` to be a full ingestion audit log or compacted change history.
