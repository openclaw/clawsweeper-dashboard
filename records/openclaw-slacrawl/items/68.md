## Summary

Keep open: current main and v0.7.3 still append a `message_events` row for every message upsert, and purge still only deletes event rows for messages selected for deletion. The requested duplicate-snapshot suppression and retained-history compaction remain real, but they need maintainer policy direction because they change audit/retention semantics.

## What This Changes

The issue requests reducing repeated `message_events` growth during desktop watch and adding purge support for compacting retained message event history.

## Best Possible Solution

Choose the event-retention policy, then implement latest-event dedupe and/or explicit purge compaction with preview/reporting plus focused store, CLI, and docs coverage.

## Reproduction Assessment

Partly: source inspection gives a high-confidence path for duplicate event insertion because every successful message upsert appends an event and desktop watch repeatedly upserts local desktop state. The several-GB live growth itself was not reproduced during this read-only review.

## Solution Assessment

Unclear until maintainers choose the retention semantics. Latest-event dedupe is a narrow candidate for preventing new duplicate snapshots, but retained-history compaction is destructive and needs an explicit product decision.
