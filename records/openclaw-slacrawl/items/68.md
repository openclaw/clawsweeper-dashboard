## Summary

Keep open: current main still appends a `message_events` row on every message upsert and purge only removes events for messages selected for deletion. The request is not already implemented, and the docs currently describe desktop watch as appending event history, so dedupe and retained-history compaction need maintainer direction.

## What This Changes

The issue requests deduplicating identical message event snapshots from desktop watch ingestion and optionally adding purge compaction for retained message event history.

## Best Possible Solution

Decide the event-history contract first, then implement latest-identical event dedupe with focused store coverage and consider a separate purge-history compaction design with docs and preview counts.

## Reproduction Assessment

Yes at source level for the current behavior: repeated message upserts flow to an unconditional `InsertMessageEvent`. I did not run a live desktop watch against a large database, and the purge compaction half is a new capability rather than a current failing command.

## Solution Assessment

Unclear: latest-identical snapshot dedupe is likely the narrowest implementation if maintainers agree those rows are not meaningful history. The purge compaction flag should be separated or explicitly designed because it changes retained-history cleanup behavior.
