## Summary

Keep open. Current nix-openclaw main packages OpenClaw v2026.5.28, where the Gmail preset renders a templated `hook:gmail` session key and mapped hook dispatch returns 400 unless `hooks.allowRequestSessionKey=true`; the reporter's Nix config has prefixes but not that gate, and the upstream policy repair is still unresolved.

## What This Changes

The issue reports that a gogcli Gmail Pub/Sub watcher starts successfully but calls the OpenClaw hook endpoint and receives hook status 400.

## Best Possible Solution

Coordinate with the upstream Gmail hook policy fix, then update nix-openclaw's pin and README/Nix setup guidance so Gmail Pub/Sub uses one bounded, documented session policy end to end.

## Reproduction Assessment

Yes, at source level: the reporter's config enables the Gmail preset and constrained prefixes but omits `hooks.allowRequestSessionKey`, while the pinned upstream handler returns 400 for templated mapped session keys in that state. I did not run a live Gmail Pub/Sub setup in this read-only review.

## Solution Assessment

Unclear as an immediate target-repo fix: adding `allowRequestSessionKey=true` with bounded prefixes explains the reporter's 400, but maintainers should align with the upstream decision on per-message Gmail sessions versus a static-session mapping before changing nix-openclaw defaults or docs.
