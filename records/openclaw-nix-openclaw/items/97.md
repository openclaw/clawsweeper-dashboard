## Summary

Keep open. Current main still ships a push-oriented Gmail watcher and a pinned `gogcli` package that predates the merged pull consumer, while the maintainer-linked RFC/OpenClaw/nix-openclaw follow-up path has not landed in this repository yet.

## What This Changes

The issue asks for nix-openclaw to make Gog/Gmail hook setup work declaratively without the hook returning 400 or requiring undiscoverable session-policy config.

## Best Possible Solution

Keep this open until maintainers land or reject the cross-repo Gmail pull/default path, then update nix-openclaw pins, tooling, README/templates, and any first-class module surface so Gmail setup is declarative and has a bounded session policy.

## Reproduction Assessment

Yes, at source level: the reporter's config omits `hooks.allowRequestSessionKey`, while current pinned OpenClaw still renders the Gmail preset with a templated `hook:gmail` session key and returns 400 when that gate is disabled. I did not run a live Gmail/Pub/Sub setup in this read-only review.

## Solution Assessment

Unclear as a target-repo-only fix: adding `hooks.allowRequestSessionKey=true` explains the current 400, but the maintainer-linked direction is a safer first-class pull/default setup that depends on upstream OpenClaw and packaged Gog support.
