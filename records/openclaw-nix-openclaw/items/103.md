## Summary

Current main still creates the Codex native profile during activation but does not solve Codex command/exec rebuilding PATH around inherited HOME, so the PR remains necessary. The branch proves the intended command-path fix, but its current head still changes HOME for existing users and aborts startup on non-owned profile state; because the author is a repository maintainer/collaborator and these are compatibility-policy choices, it should remain open for explicit maintainer resolution.

## What This Changes

The PR separates generic OpenClaw runtime tool path generation from Codex-specific handling, adds a conditional Nix-managed Codex app-server launcher, and expands runtime, activation, documentation, and migration coverage.

## Best Possible Solution

Keep the generic runtime-tools/Codex-adapter split, but preserve or narrowly migrate inherited HOME-dependent state and handle non-owned profile conflicts without app-server downtime; if maintainers intentionally accept the pre-1.0 break, document recovery and prove both fresh-install and representative upgrade behavior.

## Reproduction Assessment

Yes. Current main's source path and the PR's exact-head before/after JSON-RPC command/exec output provide a high-confidence reproduction: the tool remains missing with inherited HOME and resolves after the managed launcher selects the CODEX_HOME-native profile.

## Solution Assessment

No, not yet. Moving runtime injection into the selected Nix-managed Codex launcher is a plausible and well-proven boundary, but changing HOME and failing closed on existing profile content needs a safer migration or explicit maintainer acceptance.
