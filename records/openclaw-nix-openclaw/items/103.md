## Label Justifications

- `P2`: This is a real but bounded managed-Codex runtime integration bug with limited blast radius.
- `merge-risk: 🚨 compatibility`: The patch changes HOME semantics and ownership of an existing native-profile path for upgraded installations.
- `merge-risk: 🚨 auth-provider`: Changing HOME can hide existing Codex credentials, provider choices, configuration, and cached state.
- `merge-risk: 🚨 availability`: Profile-path conflicts cause the wrapper to exit before the Codex app-server starts.

## Summary

Current main still has the reported managed-Codex runtime-path problem, and the PR provides convincing real-path proof, but two P1 upgrade regressions remain: it relocates Codex HOME without preserving existing state and can refuse to start Codex when the new managed profile path already contains operator-owned content.

## What This Changes

The PR separates generic OpenClaw runtime tool PATH generation from Codex-specific handling, adds a managed Codex app-server wrapper that uses `$CODEX_HOME/home`, and expands runtime-path checks and documentation.

## Best Possible Solution

Keep the narrow launcher selection and runtime-tool injection, but preserve or explicitly migrate existing Codex state and handle pre-existing native-profile content without preventing app-server startup; then add fresh-install and upgrade proof for both paths.

## Reproduction Assessment

Yes. The current-head proof exercises the actual Codex app-server JSON-RPC command path and shows the runtime tool missing before the wrapper and resolving afterward; source inspection confirms that path.

## Solution Assessment

No, not yet. Moving runtime injection into the selected managed Codex launcher is the narrowest plausible boundary, but the HOME-state migration and fail-closed profile behavior must be repaired or explicitly accepted by maintainers.
