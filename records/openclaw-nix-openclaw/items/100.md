## What This Changes

This PR switches the default stable `openclaw-gateway` package to a lockfile-backed npm shrinkwrap build, adds wrapper locks for `openclaw` and `@openclaw/acpx`, stages the npm dist into the existing runtime layout, and extends pin automation to refresh npm hashes.

## Best Possible Solution

Keep the npm shrinkwrap direction, but land it only after release promotion treats the wrapper package.json and package-lock files as first-class pin artifacts and preserves the source fallback checks.

## Reproduction Assessment

Yes for the review finding by source inspection: the PR-head updater edits wrapper locks, while the current promotion workflow still digests and commits only the old three files. A future version bump then hits the new lock-version assertions with stale committed wrapper locks.

## Solution Assessment

No, not as submitted; the shrinkwrap packaging path is plausible, but the release promotion write set must include wrapper locks before the default builder can safely track future upstream versions.
