## Summary

Keep open for maintainer judgment: current main intentionally rejects the mixed raw-load-path setup, and ByteRover 4.0.0 does not satisfy the documented Nix packageability rule for dependencyful runtime plugins.

## What This Changes

The issue asks how nix-openclaw should declaratively install ByteRover as a context-engine plugin without conflicting with configured runtime plugin paths.

## Best Possible Solution

Keep the Nix runtime-plugin contract fail-closed unless maintainers intentionally expand it; ask ByteRover to publish bundled dependencies or `npm-shrinkwrap.json`, then use `runtimePluginSources` plus `plugins.slots.contextEngine` config instead of raw load paths.

## Reproduction Assessment

Yes, at source level: ByteRover 4.0.0 declares a runtime dependency, the tarball lacks node_modules and npm-shrinkwrap.json, and current installer code fails that artifact shape. I did not run a Nix build because this was a read-only review.

## Solution Assessment

No automatic repo fix is clearly best: the current fail-closed behavior matches README and module policy. Installing ByteRover as published needs an upstream artifact change or a maintainer decision to expand nix-openclaw's dependency materialization model.
