## Summary

Keep open for maintainer judgment: current main documents and enforces a deterministic runtime-plugin contract, and ByteRover 4.0.0 as published does not satisfy that packageability rule.

## What This Changes

The issue asks how to declaratively install the ByteRover context-engine plugin while also using nix-openclaw `runtimePlugins`.

## Best Possible Solution

Keep the deterministic Nix plugin contract unless maintainers choose otherwise; ByteRover should publish bundled dependencies or `npm-shrinkwrap.json`, then users can use `runtimePluginSources` plus `plugins.slots.contextEngine = "byterover"`.

## Reproduction Assessment

Yes, at source level: ByteRover 4.0.0 declares a runtime dependency, the tarball lacks `node_modules` and `npm-shrinkwrap.json`, and current installer code fails that artifact shape. I did not run a Nix build because this was a read-only review.

## Solution Assessment

No automatic repo fix is clearly best: the current fail-closed behavior matches README and module policy. Installing ByteRover as published needs an upstream artifact change or a maintainer decision to expand nix-openclaw's dependency materialization model.
