## Summary

Keep open for maintainer judgment: current main and v2026.6.10 still cannot package ByteRover 4.0.0 through `runtimePluginSources`, and changing that would be a product/packageability decision rather than a narrow bug fix.

## What This Changes

The issue asks how to declaratively install ByteRover as a context-engine plugin alongside packaged runtime plugins without conflicting plugin load configuration.

## Best Possible Solution

Keep the Nix runtime-plugin contract deterministic, and either have ByteRover publish bundled dependencies or npm-shrinkwrap.json so runtimePluginSources works, or make an explicit maintainer-approved policy change for unshrinkwrapped packages.

## Reproduction Assessment

Yes, at source level: ByteRover 4.0.0 declares a runtime dependency, its tarball lacks node_modules and npm-shrinkwrap.json, and current installer code fails that artifact shape. The raw-load-path conflict is also directly enforced by current module assertions.

## Solution Assessment

No automatic repo fix is clearly best: the current fail-closed behavior matches README and module policy. Installing ByteRover as published needs an upstream artifact change or a maintainer decision to expand nix-openclaw's dependency materialization model.
