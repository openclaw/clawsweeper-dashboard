## Label Justifications

- `P3`: The issue is a low-risk third-party plugin setup/packageability problem rather than a broken core OpenClaw runtime path.
- `impact:other`: The concrete impact is inability to declaratively package one third-party context-engine plugin under the current Nix runtime-plugin contract.

## Summary

Keep open for maintainer judgment: current main documents and enforces the raw-load-path conflict the reporter hit, and ByteRover 4.0.0 is not packageable by the current Nix runtime-plugin contract because it has a runtime dependency without shrinkwrap or bundled dependencies.

## What This Changes

The issue asks for a workable declarative way to install the ByteRover context-engine plugin alongside packaged runtime plugins.

## Best Possible Solution

Keep the Nix runtime-plugin contract fail-closed unless a maintainer decides to expand it; for ByteRover specifically, the clean end state is a shrinkwrapped or bundled ByteRover artifact that works through runtimePluginSources plus plugins.slots.contextEngine.

## Reproduction Assessment

Yes. Source and package metadata show ByteRover 4.0.0 declares a runtime dependency but its tarball lacks npm-shrinkwrap.json and bundled node_modules, while the module intentionally rejects mixing raw plugins.load.paths with runtimePlugins or runtimePluginSources.

## Solution Assessment

No automatic repository fix is clearly best. The current behavior matches README and AGENTS.md policy, so the remaining choice is whether maintainers want to keep this as third-party artifact support or deliberately broaden nix-openclaw's runtime-plugin model.
