## Summary

This can close because nix-openclaw already documents and enforces a fail-closed boundary for arbitrary npm runtime plugins, and the ByteRover 4.0.0 artifact currently lacks the bundled dependency roots or npm-shrinkwrap needed for reproducible Nix packaging.

## What This Changes

The issue asks how to declaratively install the third-party ByteRover context-engine plugin while still using nix-openclaw runtime plugins.

## Best Possible Solution

Keep nix-openclaw's fail-closed packageability contract; ByteRover should publish a dependency-free, bundled, or shrinkwrapped runtime-plugin artifact, after which users can combine runtimePluginSources with plugins.slots.contextEngine config instead of raw load paths.

## Reproduction Assessment

Yes, at source level: ByteRover 4.0.0 declares a runtime dependency but its published tarball lacks node_modules and npm-shrinkwrap.json, and the current installer fails that artifact shape by design. I did not run a Nix build because this was a read-only review.

## Solution Assessment

Yes for cleanup: the current fail-closed behavior is the maintainable solution because nix-openclaw intentionally avoids mutable npm dependency solving during Home Manager activation. The fix belongs in the ByteRover package artifact, not in this repository.
