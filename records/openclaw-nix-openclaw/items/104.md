## Summary

This can close because current nix-openclaw behavior matches the documented runtime-plugin boundary: ByteRover's published npm artifact is not Nix-packageable as-is, and the raw load-path conflict is intentional.

## What This Changes

The issue asks how to install the ByteRover context-engine runtime plugin while also using nix-openclaw managed runtime plugins.

## Best Possible Solution

Keep the deterministic runtime-plugin boundary, and have ByteRover publish bundled runtime dependencies or npm-shrinkwrap.json so users can install it through runtimePluginSources.

## Reproduction Assessment

Yes. Source inspection shows ByteRover 4.0.0 declares a runtime dependency but ships neither bundled node_modules nor npm-shrinkwrap.json, and current main rejects that shape while also rejecting mixed managed and raw plugin load paths.

## Solution Assessment

Yes. The narrow maintainable answer is to keep nix-openclaw's reproducible package boundary and fix the upstream ByteRover artifact, rather than allowing mutable dependency installs or mixed sources of truth in this repo.
