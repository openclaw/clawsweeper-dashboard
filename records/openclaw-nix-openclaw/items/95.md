## Label Justifications

- `P2`: This is a valid, narrow configuration bug affecting users who wire model provider API keys through non-env SecretRefs.
- `impact:auth-provider`: The affected path is provider-backed API-key SecretRef resolution for model providers and related credentials.

## Summary

Keep open. Current main still generates `models.providers.<name>.apiKey` as a nested `t.oneOf` of `env`/`file`/`exec` SecretRef submodules, while the earlier fix and regression check only cover `secrets.providers`.

## What This Changes

The issue asks for non-env SecretRef sources such as `exec` and `file` to work when model provider API keys reference configured secret providers.

## Best Possible Solution

Generalize the config-option generator so same-shaped `source`-discriminated SecretRef unions are emitted as one evaluable submodule, regenerate the options, and add targeted `apiKey` checks for `file` and `exec` providers.

## Reproduction Assessment

Yes from source inspection: configuring `models.providers.groq.apiKey` with `source = "exec"` still maps to a generated nested `t.oneOf` of same-key SecretRef submodules on current main. I did not run Nix evaluation during this read-only review, so this is source-reproducible rather than reproduced.

## Solution Assessment

Yes. The maintainable fix is to extend the existing generator flattening for same-shaped SecretRef unions and add focused regression coverage, rather than patching individual generated `apiKey` locations by hand.

## Repair Work Prompt

Fix the generated Nix config option handling for SecretRef-style `apiKey` unions. Extend `nix/scripts/generate-config-options.ts` so same-shaped `source`-discriminated object unions such as `{ id, provider, source = env|file|exec }` no longer fall back to nested `t.oneOf` submodules, while preserving the upstream JSON config shape. Regenerate `nix/generated/openclaw-config-options.nix` and add focused coverage in `nix/checks/openclaw-default-instance.nix` for `models.providers.<name>.apiKey` referencing both a file-backed and an exec-backed `secrets.providers` entry. Do not introduce a new config syntax, do not change runtime SecretRef semantics, and avoid broad unrelated schema reshaping.
