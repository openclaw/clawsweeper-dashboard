## Summary

Keep open. Current main still emits the reported SecretRef `apiKey` shape as nested `t.oneOf` submodules, while the earlier fix only made `secrets.providers` evaluable.

## What This Changes

The issue asks for SecretRef consumers such as `models.providers.<name>.apiKey` to accept non-env providers like `file` and `exec`.

## Best Possible Solution

Extend the config-option generator to safely flatten same-key `source`-discriminated SecretRef unions, regenerate the generated options, and add regression coverage for `apiKey` with `file` and `exec` providers.

## Reproduction Assessment

Yes from source inspection: current main still generates `models.providers.<name>.apiKey` as nested `t.oneOf` SecretRef submodules for `env`, `file`, and `exec`. I did not run Nix evaluation during this read-only pass, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Yes. A narrow generator fix plus targeted default-instance regression coverage is the maintainable solution; hand-editing individual generated `apiKey` blocks would drift from the source schema pipeline.
