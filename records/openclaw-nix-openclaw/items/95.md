## Label Justifications

- `P2`: This is a valid, narrow configuration bug affecting users who wire model provider API keys through non-env SecretRefs.
- `impact:auth-provider`: The broken path is API-key SecretRef resolution for model providers and other provider-backed credentials.

## Summary

Keep open. Current main still emits the reported `apiKey` SecretRef as a nested `t.oneOf` of `env`/`file`/`exec` submodules, while the earlier provider-only fix and regression check only cover `secrets.providers`; likely routing is `joshp123` based on the generator/check history.

## What This Changes

The issue asks for SecretRef consumers such as model provider `apiKey` to accept `file` and `exec` providers instead of collapsing to the first `env` variant.

## Best Possible Solution

Extend the config-option generator to flatten same-shaped `source`-discriminated SecretRef unions, regenerate the options, and add regression coverage for `apiKey` using `file` and `exec` providers.

## Reproduction Assessment

Yes from source inspection: configuring `models.providers.groq.apiKey` with `source = "exec"` still hits a generated nested `t.oneOf` of SecretRef submodules on current main. I did not run Nix evaluation during this read-only review, so the status is source-reproducible rather than reproduced.

## Solution Assessment

Yes, a narrow generator fix plus a targeted default-instance regression check is the maintainable path. The existing `secrets.providers` fix should be generalized to same-key SecretRef unions rather than patched one `apiKey` location at a time.
