## Summary

Keep open: current main still runs Claude with a temporary HOME/XDG environment and a replaced allowlisted env, so the configured binary's local OAuth/keychain context is not used by default. The remaining work is a provider auth-boundary/product decision, especially because a prior opt-in host-environment PR was closed after maintainer discussion.

## What This Changes

The issue requests that the Claude provider use the configured `claude` binary's normal auth context by default, align `doctor` with the runtime path, and improve Claude auth failure reporting.

## Best Possible Solution

Maintainers should decide the Claude auth boundary first; if accepted, implement a narrow documented mode or default policy, align doctor with the actual runtime environment, and improve redacted failure classification for Claude auth envelopes.

## Reproduction Assessment

Yes for source-level reproduction: current main visibly runs Claude with a temp HOME/XDG env and `replaceEnv: true`, and docs confirm OAuth/keychain state is not passed. I did not run a live local Claude OAuth login path in this read-only review.

## Solution Assessment

Unclear: using host auth by default may solve the reported operator path, but it would overturn the documented hardened boundary and prior maintainer direction. A safer path needs an explicit maintainer-approved policy for host auth exposure before code changes.
