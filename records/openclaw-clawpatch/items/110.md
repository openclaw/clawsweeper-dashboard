## Label Justifications

- `P1`: The issue describes a command-injection path in validation commands where repo-controlled names or paths can become shell syntax.
- `impact:security`: The affected behavior crosses a command-execution security boundary by passing unquoted repo-derived arguments to `shell: true`.

## Summary

Keep open: current main still builds mapper validation command strings with unquoted repo-derived arguments and executes them through `shell: true`; the linked fixing PR is open but not merged yet.

## What This Changes

The issue requests quoting or rejecting repo-derived mapper validation command arguments before they are executed through the shell.

## Best Possible Solution

Land https://github.com/openclaw/clawpatch/pull/111 after maintainer/security review, or merge an equivalent focused fix that quotes all repo-derived mapper validation arguments and adds regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a package root such as `apps/web; touch INJECTED` can be inserted unquoted into a generated validation command, and the validation executor runs that string with `shell: true`; I did not execute it because this review is read-only.

## Solution Assessment

Yes. Quoting dynamic mapper command arguments with the existing helper, or moving the validation path to args-based execution, is the narrow maintainable fix; the linked PR already follows the focused quoting direction.
