## Summary

## What This Changes

The PR changes `Queue.Enqueue` to return `false` instead of blocking when full, updates `LineCapture` path mode to scan without closing the shared channel, attempts to skip unchanged regular files while preserving FIFO reopening, and adds queue/line-capture tests.

## Best Possible Solution

Land a corrected narrow fix that scans path input without closing the shared output channel, records regular-file state from a valid stat before or without closing the handle, preserves FIFO reopening, and includes regression coverage plus redacted runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main’s path loop calls a reader that closes the shared output channel after one scan, and the PR’s unchanged-file test describes a focused failing path. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. Splitting scanning from channel closing is the right shape, but the replay gate must record file state from a valid stat before close or from the pre-open info so unchanged regular files are actually skipped.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Record file state before closing the handle:** `modules/audio/line_capture.go:137`
