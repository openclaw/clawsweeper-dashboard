## Summary

Current main still contains the LineCapture path-loop bug this PR targets, so it is not safe to close as already implemented; however, the submitted fix is not merge-ready because the unchanged-file gate records file state after closing the file and therefore never activates.

## What This Changes

The PR changes LineCapture path polling to avoid replaying unchanged regular files, makes queue enqueue non-blocking when full, and adds focused Go regression tests.

## Best Possible Solution

Keep the PR open for a narrow repair that snapshots regular-file state before close, preserves FIFO behavior, and includes redacted real behavior proof for the path-mode capture scenario.

## Reproduction Assessment

Yes, from source inspection: current main calls `readReader` inside the path loop, and `readReader` closes the shared output channel after the first scan. The PR head also has a source-reproducible defect because `f.Stat()` runs after `f.Close()`.

## Solution Assessment

No, not as written. Splitting scanning from channel closure is the right direction, but the file-state snapshot must be captured before the file is closed or reused from the pre-open stat result.
