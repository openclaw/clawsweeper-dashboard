## Summary

Keep this PR open because it targets a real, narrow bug, but the proposed line-capture gate is currently ineffective: it stats the file after closing the handle, so the unchanged-file replay prevention does not reliably record state.

## What This Changes

The PR makes Queue.Enqueue non-blocking when full and changes LineCapture path mode to skip unchanged regular files, preserve FIFO reopening, and add regression tests.

## Best Possible Solution

Land a corrected narrow fix that separates scanning from channel closing, records regular-file state from a valid stat before closing or from the pre-open info after a successful scan, preserves FIFO reopening, and includes regression coverage plus real runtime proof.

## Reproduction Assessment

Yes, from source inspection rather than executing tests: current main closes the shared output channel after the first path scan, and the PR's own unchanged-file test describes a focused reproduction. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. Splitting scanReader is the right shape, but the file-state gate must stat before close or reuse the pre-open info so unchanged files are actually skipped.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move stat before closing the file:** `modules/audio/line_capture.go:137`
  - body: After scanning, this closes `f` and then calls `f.Stat()` to remember the last mod time and size. `(*os.File).Stat` on a closed file returns an error and no `FileInfo`, so `lastModTime` stays zero and the unchanged-file gate above never becomes active; the path loop can emit the same line again after the 200 ms sleep. Capture the stat before closing, or reuse the pre-open `info`, after a successful scan.
  - confidence: 0.92
