## Summary

Keep this PR open: it is a member-authored draft with useful feature work, but the current diff still has a blocking source-level attribution bug for shared PTY input and is conflicting with current main.

## What This Changes

This PR adds a persisted opt-in multiplayer mode for interactive Codex sessions, UI/API toggle actions, actor-prefixed terminal prompt submission, docs, migration, and changelog text.

## Best Possible Solution

Keep the opt-in feature direction only after pending terminal input is tracked per shared session/upstream PTY, or after maintainers intentionally enforce one active writer at a time.

## Reproduction Assessment

Yes at source level: current main allows owners/maintainers and delegated controllers to be write-capable, while the PR stores pending text per TerminalHubSubscription before submitting into one shared PTY. I did not run a live browser/PTY reproduction in this read-only review.

## Solution Assessment

No: the opt-in feature direction is workable, but per-subscription pending input is not the right implementation for a shared PTY. The narrower maintainable fix is session/upstream-scoped tracking or an explicit single-writer rule.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track pending input per shared PTY:** `src/index.ts:2634-2635`
  - body: This buffers the typed line on each TerminalHubSubscription, but multiple write-capable clients can target the same interactive session. If one actor types a partial line and another actor submits, the Enter path sends Ctrl-U plus only that second subscription's buffer, clearing the shared PTY line and dropping or misattributing the other actor's pending text. Track the line per shared session/upstream PTY, or explicitly serialize one active writer.
  - confidence: 0.89
