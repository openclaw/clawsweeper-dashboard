## Summary

Keep this PR open: it is meaningful feature work that is not on current main, and it is member-authored, but the current branch still has a source-level multiplayer correctness blocker around shared PTY input attribution.

## What This Changes

The PR adds a D1-backed opt-in multiplayer flag, owner/maintainer session actions and UI controls, docs/changelog text, and terminal-hub prompt prefixing for enabled Codex sessions.

## Best Possible Solution

Keep the opt-in feature direction, but make pending input attribution session/upstream-scoped or explicitly serialize one active editor before merge.

## Reproduction Assessment

Yes at source level: two write-capable subscriptions can each hold a separate inputLine while writing into the same interactive session PTY, then Enter triggers attribution from only one buffer. I did not run a live browser/PTY reproduction in this read-only pass.

## Solution Assessment

No: the proposed per-subscription line buffer is not the best way to solve the issue because the PTY line is shared across writable actors. A session/upstream-scoped buffer or an explicit single-editor guard is the safer implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track pending input per shared PTY:** `src/index.ts:2634-2635`
  - body: `terminalSubmittedLine` reads and clears `subscription.inputLine`, but each browser subscription gets its own buffer while bytes are sent into the same interactive session PTY. Store pending input per session/upstream, or gate to one active editor, before merge.
