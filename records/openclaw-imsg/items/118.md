## Label Justifications

- `P1`: A live iMessage group event can be exposed as a direct-message-shaped payload, creating wrong-recipient routing in real channel workflows.
- `impact:security`: Wrong-recipient iMessage delivery can expose group-context agent output to an unintended direct recipient.
- `impact:message-loss`: The reported failure can misroute a reply away from the originating group conversation to a sender DM.

## Summary

Keep open. Current main still has a source-reproducible path where an all-chat watch row without a resolved `chat_message_join` record becomes `chat_id: 0`, then the RPC payload builder emits empty chat identifiers and `is_group: false`; no merged PR or current code path appears to guard this case yet.

## What This Changes

The issue asks `watch.subscribe` to preserve or fail closed on group conversation metadata for link-preview messages instead of emitting `chat_id=0` direct-message-shaped payloads.

## Best Possible Solution

Make watch payload construction refuse normal message notifications when chat metadata is unresolved: retry or requery by row/GUID until the chat join resolves, then emit stable metadata, or otherwise emit an explicit error/partial state or drop fail-closed.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only pass. Source inspection does show a current-main path from an unjoined all-chat watch row to `chat_id: 0`, empty chat metadata, and `is_group: false`, matching the reporter's redacted payload.

## Solution Assessment

Partly. The invariant is clear, but maintainers still need to choose the exact live-watch behavior for unresolved metadata because retry/requery, explicit partial/error, and fail-closed drop have different delivery semantics.
