## Summary

Keep open. Current main still has a source-reproducible watch path where an all-chat message row without a visible chat join decodes as `chatID=0`, then the RPC payload builder emits empty chat identifiers and `is_group: false`; the linked OpenClaw report at https://github.com/openclaw/openclaw/issues/84470 confirms the downstream wrong-recipient risk.

## What This Changes

Report that `watch.subscribe` can emit malformed group link-preview notifications with `chat_id=0` and empty chat metadata while `messages.history` resolves the same GUID to the correct group.

## Best Possible Solution

Make watch notifications require stable conversation metadata: retry or re-query by message GUID/chat join after debounce, emit the same chat metadata as history when resolved, and otherwise return an explicit partial/error path instead of a DM-shaped payload.

## Reproduction Assessment

Yes from source inspection: `watch.subscribe` streams `messagesAfter`, whose LEFT JOIN can produce a nil chat id that `decodeMessageRow` converts to `0`, and the payload builder then emits empty chat metadata. I did not run a live Messages DB reproduction in this read-only review.

## Solution Assessment

Yes, the requested direction is the right fix boundary: imsg should not emit a DM-shaped payload when the message GUID can resolve to a group through history. The safest implementation is a source-side retry/recovery path with a clear unresolved state rather than leaving every consumer to infer the missing group anchor.
