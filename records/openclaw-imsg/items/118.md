## Label Justifications

- `P1`: A live iMessage group event can be exposed to downstream agents as a direct-message-shaped payload, creating wrong-recipient routing in real channel workflows.
- `impact:security`: Wrong-recipient iMessage delivery can expose group-context agent output to an unintended direct recipient.
- `impact:message-loss`: The reported failure can misroute a reply away from the originating group conversation to the sender DM.

## Summary

Keep open. Current main still has a source-reproducible path where all-chat watch can decode an unresolved chat join as `chat_id: 0`, then build a payload with empty chat identifiers and `is_group: false`, matching the reported wrong-recipient routing risk.

## What This Changes

Requests that `watch.subscribe` preserve group metadata for group link-preview messages instead of emitting `chat_id=0` direct-message-shaped payloads.

## Best Possible Solution

Make watch payload construction treat unresolved chat metadata as non-routable: recover or requery by row/GUID until chat metadata resolves, and otherwise emit an explicit partial/error state or drop fail-closed instead of producing a DM-shaped payload.

## Reproduction Assessment

No, not as a live end-to-end reproduction in this read-only pass. Source inspection shows a clear current-main path from an unresolved `chat_message_join` row to `chat_id: 0`, empty chat metadata, and `is_group: false`, matching the reporter's live payload shape.

## Solution Assessment

Partly. The invariant is clear, but maintainers should decide the precise live-stream semantics for unresolved metadata because retrying, emitting partial errors, or dropping fail-closed have different delivery tradeoffs.
