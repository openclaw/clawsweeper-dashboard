## Label Justifications

- `P1`: A live iMessage group event can be exposed to downstream agents as a direct-message-shaped payload, creating wrong-recipient routing in real channel workflows.
- `impact:message-loss`: The reported failure can misroute a reply away from the originating group conversation to a sender DM.
- `impact:security`: Wrong-recipient iMessage delivery can expose group-context agent output to an unintended direct recipient.

## Summary

Keep open. Current main still has a source-reproducible path where a live watch row without a resolved chat join becomes `chat_id: 0`, then the RPC payload builder emits empty chat identifiers and `is_group: false`; that matches the reported DM-shaped group-link-preview payload and is not covered by an existing imsg PR.

## What This Changes

The issue asks for `watch.subscribe` to preserve group conversation metadata for link-preview messages, or fail closed clearly, instead of emitting `chat_id=0` DM-shaped payloads.

## Best Possible Solution

Make watch payload construction refuse normal message notifications when chat metadata is unresolved: retry or requery by row/GUID until the chat join resolves, and if it cannot, emit an explicit error/partial state or drop fail-closed instead of producing empty chat identifiers.

## Reproduction Assessment

No full live reproduction was run in this read-only pass. Source inspection shows a clear current-main path from an unjoined watch row to `chat_id: 0`, empty chat metadata, and `is_group: false`, matching the reporter's redacted live payload.

## Solution Assessment

Partly. The invariant is clear, but maintainers should choose the exact watch behavior for unresolved metadata, such as short retry/requery versus explicit error or fail-closed drop, because it affects live stream delivery semantics.
