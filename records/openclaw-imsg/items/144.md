## Summary

Keep open. Current main still drops the requested reply target on the plain JSON-RPC `send` path: `handleSend` never reads `reply_to`, `sendViaBridge` forwards only chat/text/file data, while `send.rich` and the injected bridge already use `selectedMessageGuid` to build threaded replies.

## What This Changes

The issue requests that JSON-RPC `send` honor `reply_to` by forwarding it to the IMCore bridge so outgoing messages thread to the specified parent message.

## Best Possible Solution

Forward existing reply target aliases from RPC `send` into the bridge `selectedMessageGuid` contract, add focused regression coverage, and fail clearly rather than falling back to an unthreaded AppleScript send when a reply target is requested.

## Reproduction Assessment

Yes, from source inspection: current main never reads `reply_to` on RPC `send` and never forwards `selectedMessageGuid` to the bridge. I did not perform a live macOS Messages database reproduction.

## Solution Assessment

Yes, forwarding the existing reply aliases to the existing bridge reply contract is the narrow maintainable fix. The implementation should explicitly avoid unthreaded AppleScript fallback when a reply target is supplied.

## Repair Work Prompt

Fix JSON-RPC `send` reply target forwarding on the bridge path. In `Sources/imsg/RPCServer+Handlers.swift`, parse the same reply aliases used by `send.rich` (`reply_to`, `replyTo`, `reply_to_guid`, `message_guid`) and pass the value through `sendViaBridge` as `selectedMessageGuid` for text sends, and for attachment sends if the existing bridge action supports it. Do not make AppleScript pretend to support threaded replies; when a reply target is supplied and bridge transport is unavailable or explicitly bypassed, return a clear unsupported/invalid error rather than silently sending unthreaded. Add regression tests in `Tests/imsgTests/RPCServerBridgeTests.swift` proving RPC `send` forwards `selectedMessageGuid` to `.sendMessage`, omits it when absent, and does not fall back to AppleScript with `reply_to`. Keep the change focused, avoid release-owned `CHANGELOG.md`, and update `docs/rpc.md` or `docs/send.md` only if the new RPC `send` parameter is intentionally documented.
