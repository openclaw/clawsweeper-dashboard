## Summary

Current main still drops the plain RPC `reply_to` before the bridge call, and the open linked PR is the active candidate fix, so this issue should remain open until that PR lands or is replaced.

## What This Changes

Plain JSON-RPC `send` should thread replies by forwarding the requested `reply_to` GUID through the IMCore bridge, matching the existing rich send path.

## Best Possible Solution

Land or replace https://github.com/openclaw/imsg/pull/145 with a narrow fix that forwards `reply_to` aliases into `selectedMessageGuid` and covers bridge-unavailable behavior, then close this issue once the fix reaches main.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: current main’s plain RPC send path never reads reply aliases and its bridge helper does not include `selectedMessageGuid`, while `send.rich` and `send-rich` already do. I did not run a live Messages bridge reproduction because this review stayed read-only and avoided live `chat.db` or Messages side effects.

## Solution Assessment

Yes, the narrow maintainable solution is to reuse the existing `selectedMessageGuid` bridge contract for plain RPC `send` and add focused regression coverage. The linked PR is the active candidate, but it should be reviewed and proven before this issue is closed.
