## Label Justifications

- `P2`: This is a bounded RPC messaging bug with user-visible reply-threading impact and an open candidate fix.
- `impact:message-loss`: The outgoing message can lose or misroute the intended reply/thread context when the requested parent GUID is ignored.

## Summary

Current main still drops the plain JSON-RPC `reply_to` value before bridge dispatch, and https://github.com/openclaw/imsg/pull/145 is the open closing candidate, so this issue should stay open until that fix lands or is replaced.

## What This Changes

Plain JSON-RPC `send` should honor `reply_to` by threading the outgoing message to the requested parent GUID like the existing bridge-backed rich send path.

## Best Possible Solution

Review, prove, and land or replace https://github.com/openclaw/imsg/pull/145 with a narrow fix that forwards reply aliases into `selectedMessageGuid`, then close this issue once the fix reaches main.

## Reproduction Assessment

Yes. Source inspection shows current main's plain RPC `send` path never reads reply aliases and its bridge helper does not pass `selectedMessageGuid`, while the rich RPC and CLI paths already do.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing bridge `selectedMessageGuid` contract for plain RPC `send` and cover it with focused regression tests; the open linked PR is the active candidate but still needs proof/review.
