## Summary

Keep open: current main only has watch-side URL balloon duplicate suppression and still builds one Message per decoded row for history and search, so the requested source-level text+preview coalescing is not implemented. The related metadata PR is still open and is a prerequisite rather than the requested coalescer.

## What This Changes

Request to coalesce Apple URL-preview balloon split-send rows into one logical message across watch, history, and search outputs.

## Best Possible Solution

After the balloon metadata prerequisite is resolved, add one shared MessageStore coalescing layer used by messages(), messagesAfter(), and searchMessages(), preserving the text row GUID/timestamp and leaving non-URL balloons and genuinely separate sends untouched.

## Reproduction Assessment

Yes, source-level: a deterministic fixture with adjacent same-chat/same-sender text and com.apple.messages.URLBalloonProvider rows would exercise the current gap. I did not run a live Messages DB reproduction, but the current history/search/watch source paths show why the requested coalescing is absent.

## Solution Assessment

No, the existing watch-only dedupe is not the complete solution requested here. The maintainable path is a shared coalescer with explicit output semantics, not another client-side debounce or a watch-only special case.
