## Summary

Keep open: current main and v0.11.0 still use the 10s bridge response timeout for all actions, so the PR addresses a real remaining send-timeout problem, but it should not merge yet because tapback sends remain on the short timeout and the external PR only includes test output, not live bridge proof.

## What This Changes

The PR changes the bridge client to choose an action-specific default timeout, adds a 150s default for message, multipart, attachment, and poll sends, and adds protocol tests for the new timeout helper.

## Best Possible Solution

Land a scoped timeout helper that covers every intended private send path, preserves short probes and explicit overrides, and is backed by unit tests plus redacted live bridge-send proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show current main gives all bridge invokes the same 10s default, and the PR body reports macOS private sends can exceed that window.

## Solution Assessment

No, not quite as submitted. The action-specific default is a narrow maintainable direction, but it should cover or explicitly exclude .sendReaction and needs real bridge proof before merge.
