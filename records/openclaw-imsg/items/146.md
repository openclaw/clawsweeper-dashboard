## Summary

Keep open: current main and v0.11.1 still expose `chat-delete`/`chats.delete`, and the injected helper still uses only `IMChatRegistry.deleteChat:` with the exact reported error when that selector is absent; no related fix or canonical replacement was found.

## What This Changes

The issue reports that the documented `imsg chat-delete` CLI command and `chats.delete` RPC method return `Dylib error: deleteChat: not available` on macOS 26.5.1.

## Best Possible Solution

Add a macOS 26-safe chat deletion path or capability-gate `chat-delete`/`chats.delete` when deletion is unsupported, with regression coverage and live macOS 26.5.1 proof.

## Reproduction Assessment

Yes for source and reporter-provided reproduction: the issue body gives v0.11.1/macOS 26.5.1 commands, and current main still returns the reported error whenever `IMChatRegistry` lacks `deleteChat:`. I did not run the live macOS runtime in this Linux checkout.

## Solution Assessment

Unclear: the current implementation is not the best end state because it advertises deletion while only calling one private selector. The maintainable fix is either a verified macOS 26-compatible deletion path or clear capability gating when deletion is unavailable.
