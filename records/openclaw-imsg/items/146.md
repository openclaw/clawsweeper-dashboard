## Summary

Keep open: current main still exposes `chat-delete` and `chats.delete`, and both still route to an injected helper path that returns the exact reported `deleteChat: not available` error whenever `IMChatRegistry` does not implement `deleteChat:`. ## What This Changes

The issue asks for the exposed CLI and RPC chat deletion paths to work, or be accurately capability-gated, on macOS 26 when `IMChatRegistry.deleteChat:` is unavailable.

## Best Possible Solution

Add a verified macOS 26-compatible chat deletion path, or capability-gate/remove advertised deletion when the runtime does not support it, with focused regression coverage and live proof on a disposable chat.

## Reproduction Assessment

Yes for source and reporter-provided reproduction: current main still returns the exact error whenever `IMChatRegistry` lacks `deleteChat:`, and the issue body gives v0.11.1/macOS 26.5.1 CLI/RPC commands. I did not run the live macOS runtime in this Linux checkout.

## Solution Assessment

Unclear: the current implementation is not the best end state because it advertises deletion while only calling one private selector. The maintainable fix is either a verified macOS 26-compatible deletion path or clear capability gating when deletion is unavailable.
