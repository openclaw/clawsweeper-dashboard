## Summary

Keep open: current main still has the runtime AX path generating legacy `elem_*` IDs while CLI and MCP help still advertise role-coded examples. The merged docs/skill update mitigates some guidance by treating IDs as opaque, but it explicitly left the runtime ID-contract issue open.

## What This Changes

The issue asks Peekaboo to make runtime element IDs and CLI/MCP documentation agree on either legacy `elem_*` IDs or role-coded `B1`/`T2` IDs.

## Best Possible Solution

Maintainers should choose the public ID contract, then update runtime output, CLI/MCP help, docs, skills, and regression coverage together; if role-coded IDs win, preserve compatibility with aliases or explicit migration behavior.

## Reproduction Assessment

Yes. Source inspection shows `AXTreeCollector` still creates `elem_*` IDs while CLI/MCP help advertises `B1`/`T2`, and the merged docs PR includes live Calculator proof showing only legacy IDs.

## Solution Assessment

Unclear. Rewriting runtime IDs to role-coded values may be right, but the maintainable fix needs a maintainer decision on compatibility versus documenting legacy IDs as the contract.
