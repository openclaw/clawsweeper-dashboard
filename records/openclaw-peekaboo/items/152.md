## Best Possible Solution

Land GameBridge as a core detection fallback only after it derives trusted window bounds, rejects stale or mismatched manifests, isolates tests, and has redacted Firestaff see/click proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: inspect_ui builds a WindowContext without windowBounds, and the PR scales GameBridge elements with nil bounds to framebuffer coordinates. I did not run a live Firestaff session.

## Solution Assessment

No; the core idea is useful, but the current implementation needs trusted window bounds, stale-manifest handling, isolated tests, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Require window bounds before scaling GameBridge elements:** `Core/PeekabooAutomationKit/Sources/PeekabooAutomationKit/Services/GameBridge/GameBridgeDetectionIntegration.swift:30-32`
  - body: `tryDetect` passes `windowContext?.windowBounds` into scaling, but callers like `inspect_ui` build a WindowContext with app/window identifiers and no bounds. In that path GameBridge returns framebuffer coordinates, so a later click by ID lands at the wrong screen point; derive the resolved window frame or fall back until bounds are known.
  - confidence: 0.9
- **[P2] Reject stale manifests before bypassing AX:** `Core/PeekabooAutomationKit/Sources/PeekabooAutomationKit/Services/GameBridge/GameBridgeDetectionIntegration.swift:26-27`
  - body: As soon as `readManifest` decodes a file, this returns GameBridge elements and skips AX traversal. A crashed or old Firestaff session can leave `~/.firestaff/accessibility.json` behind with stale IDs and coordinates, so add freshness/session/app validation and fall back when it does not match the current window.
  - confidence: 0.87
- **[P3] Keep GameBridge tests out of the real home directory:** `Apps/CLI/Tests/CoreCLITests/GameBridgeDetectionTests.swift:188-195`
  - body: The test helper writes `~/.firestaff/accessibility.json` in the user’s actual home directory. Running tests while Firestaff is active or after a concurrent update can clobber the manifest and leaves the dotdir behind, so inject a manifest root/path and use a temp directory.
  - confidence: 0.84
