## Summary

Keep open: this PR is a targeted follow-up to the recorder ID-parity feedback on https://github.com/openclaw/crabline/pull/20, but the patch now emits the in-process WhatsApp inbound event before the recorder append succeeds, so a recorder write failure can leave sockets with a delivered message while the admin request returns 500.

## What This Changes

The branch records the generated WhatsApp admin inbound Baileys message in recorder events, replays it from runtime sockets with fallback for older lines, adds ID-parity assertions, and bumps release metadata to 0.1.3.

## Best Possible Solution

Keep the recorded-message replay approach, but build the canonical admin inbound message first, append the recorder event containing that message successfully, then emit to sockets and return success.

## Reproduction Assessment

Yes, at source level. In PR head, `/crabline/whatsapp/inbound` calls `handleAdminInbound`, that emits the Baileys inbound message, and only afterward does `appendEvent` run; I did not run tests because this review is read-only and the relevant tests create artifacts.

## Solution Assessment

No, not yet. The ID-parity fix is the right direction, but recorder persistence should remain before socket emission so failed writes do not create delivered-but-unrecorded messages.
