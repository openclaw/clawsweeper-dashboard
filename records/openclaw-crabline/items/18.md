## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Wire inbound messages into the WhatsApp listener path:** `src/fake-servers/whatsapp.ts:458`
  - body: The admin endpoint creates a Baileys-shaped inbound `message` and `webhook`, but then only returns them to the HTTP caller. There is no shared state with `createWhatsAppBaileysMockSocket()`, and the socket only emits `messages.upsert` from outbound `sendMessage()`, so a smoke run can inject a user message and get `ok: true` while OpenClaw never receives it.
  - confidence: 0.86
- **[P2] Require admin auth for WhatsApp inbound injection:** `src/fake-servers/whatsapp.ts:477-483`
  - body: `/crabline/whatsapp/inbound` returns before the bearer-token check, and the WhatsApp server has no admin-token check equivalent to Telegram's. If a user binds this fake server beyond loopback with `--host`, anyone who can reach it can forge inbound test messages.
  - confidence: 0.9
- **[P3] Avoid regex trimming for mock API URLs:** `src/fake-servers/whatsapp.ts:175`
  - body: `joinUrl` trims caller-provided URL parts with unbounded slash regexes, matching the CodeQL polynomial-regex review comment on this PR. Use URL construction or non-regex trimming so slash-heavy input cannot stall the mock socket path.
  - confidence: 0.72
