## Summary

Keep open: current main partly addresses the duplicate attachment insert symptom, but it still lacks the requested durable, queryable sync/import/write failure ledger, `failures` command, `doctor --recent-failures` JSON surface, and general row-context wrapping for write failures.

## What This Changes

The issue requests durable, queryable sync/import/write failure context with row identifiers and resolution status.

## Best Possible Solution

Design a narrow failure ledger and JSON read surface, then instrument sync, wiretap/import, media/embedding, and store write paths so successful later ingestion can resolve row-scoped failures.

## Reproduction Assessment

Yes for the missing requested surface: current-main source inspection shows no failure ledger, no `failures` command, and no `doctor --recent-failures` output. The original duplicate attachment incident is described, and the narrow duplicate-upsert symptom has since been partially addressed.

## Solution Assessment

Unclear until maintainers choose the persistent schema and lifecycle semantics. The likely best path is a durable ledger plus row-context wrappers before broader ingestion behavior changes.
