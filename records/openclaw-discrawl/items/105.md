## Summary

Keep open: current main and v0.11.3 only partially address the duplicate attachment insert symptom; they still lack the requested durable, queryable sync/import/write failure ledger and JSON `failures` or `doctor --recent-failures` surface.

## What This Changes

Request durable row-level failure context for sync/import/write errors, JSON read commands, retry/resolution state, and coverage output that distinguishes known failures from missing or unattempted rows.

## Best Possible Solution

Add a versioned failure ledger plus JSON read surfaces after maintainers choose the schema, retention policy, resolution semantics, and which write paths must record failures.

## Reproduction Assessment

Yes as a source-level gap: current main has no `failures` command, `doctor` rejects extra failure flags, and the schema has no general failure ledger. This is not a single runtime bug reproduction; it is a missing diagnostic capability backed by code inspection.

## Solution Assessment

No automated implementation path is best yet. The maintainable path is a maintainer-approved ledger design, then a focused implementation across the store, sync/import/write call sites, and JSON CLI output.
