## Summary

This issue remains open because current main still has `messages --sync` invoke the shared sync path before reading messages, while `tail` appears to hold the same sync lock for its lifetime. The report is narrow, recent, and source-reproducible enough to be a good focused repair candidate rather than a cleanup close.

## What This Changes

The issue asks Discrawl to avoid blocking or to fail/document clearly when `discrawl messages --sync` runs while `discrawl tail` already owns the sync lock.

## Best Possible Solution

Make `messages --sync` handle an already-running `tail` explicitly, preferably by falling back to the local read path or failing fast with operator-focused guidance while preserving the existing plain `messages` read-only behavior.

## Reproduction Assessment

Yes from source inspection, but not from a live run: `messages --sync` still enters `runSyncOnce`, and `tail` holds the same lock until exit. A full reproduction would require a configured Discord archive and two concurrent processes.

## Solution Assessment

Unclear which of the reporter's acceptable fixes is best because fallback, fail-fast, and tail lock narrowing have different operator semantics. The narrowest maintainable path is likely a targeted `messages --sync` lock-owner check with tests and help text if fallback is not chosen.
