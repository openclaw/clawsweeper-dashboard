## Label Justifications

- `P2`: This breaks a documented query workflow under a common tail-plus-query setup, but callers can still query the local archive by omitting `--sync`.
- `impact:other`: Automation can fail to retrieve otherwise-readable archived data because the command waits behind a long-lived lock.

## Summary

Current main still sends `messages --sync` through the shared writer lock, so it can wait behind a long-running tail even though the archive is locally readable. This is the narrow unresolved remainder of the earlier read-command lock fix, and a focused backward-compatible repair is viable without a broader product decision.

## What This Changes

Handle `messages --sync` gracefully when a live `tail` process already owns the shared sync lock.

## Best Possible Solution

Record enough lock-owner operation metadata to identify a live tail, then make only `messages --sync` fail fast with an actionable omit-`--sync` message while preserving existing serialization for every other writer.

## Reproduction Assessment

Yes from current source: `messages --sync` takes the shared writer-lock path, and existing tests establish that a concurrent lock owner blocks that path until cancellation. A live Discord run was not performed, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

Yes, a tail-specific fail-fast response is the narrowest maintainable fix: it preserves the explicit sync guarantee and ordinary writer serialization instead of silently converting `--sync` into a local-only read.
