## Label Justifications

- `P2`: The issue affects a real documented command workflow but is bounded to `messages --sync` under an active `tail` process.
- `impact:message-loss`: The reported lock wait prevents callers from retrieving archived messages through the documented freshness path while `tail` is running.

## Summary

Keep this issue open: current main still routes `messages --sync` through the shared writer/sync lock before reading messages, while the earlier lock fix only covers plain read commands. No open or merged PR now owns this narrower `tail` plus `messages --sync` behavior.

## What This Changes

The issue asks Discrawl to handle `messages --sync` more safely when a long-running `tail` process already owns the sync lock.

## Best Possible Solution

Define a documented non-hanging behavior for `messages --sync` under an active `tail`, then add regression coverage for a held `.discrawl-sync.lock` while invoking that command.

## Reproduction Assessment

Yes, at source level: current dispatch sends `messages --sync` through the locked service path, `runMessages` calls `syncMessagesQuery`, and that calls `Sync` before listing messages. A live Discord/tail reproduction was not run in this read-only review.

## Solution Assessment

Unclear; the issue describes several viable fixes, and maintainers should choose whether the permanent behavior is local fallback, fail-fast messaging, or a narrower tail lock before implementation.
