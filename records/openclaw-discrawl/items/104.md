## Summary

Keep open: current main still has only `channels list/show` plus loose id/name/partial query filters, but no resolver command, ambiguity candidate diagnostics, selected-channel resolution envelope, or merged docs covering numeric ids for repeatable workflows.

## What This Changes

The issue requests a CLI channel resolver, clearer ambiguity handling, selected-channel JSON context, and docs recommending numeric channel ids for repeatable Discrawl queries.

## Best Possible Solution

Define one shared resolver contract and use it from `channels`, `search`, and `messages`, while letting the workflow docs PR cover numeric-id guidance if maintainers accept it.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request for a resolver and output contract. Source inspection confirms current main lacks that resolver surface.

## Solution Assessment

Unclear until maintainers settle the CLI and JSON/error contract. A shared resolver reused by `channels`, `search`, and `messages` looks narrower and safer than adding separate ad hoc matching behavior per command.
