## Label Justifications

- `P2`: This is a normal-priority reliability bug affecting daemonized sync users rather than the entire CLI.
- `impact:message-loss`: The issue describes missed WhatsApp messages in downstream automations because the local store can go stale while sync appears healthy.

## Summary

Keep open: current main partially improves sync reliability by reconnecting after `StreamReplaced`, but it does not implement stale-follow freshness detection, stale health reporting, or an automatic reconnect/exit path when the process remains connected while store freshness stops advancing.

## What This Changes

The issue requests built-in detection or reporting for `sync --follow` processes that remain connected while the local store stops becoming fresh.

## Best Possible Solution

Add a bounded stale-follow health mechanism in the sync/app layer that can reconnect, emit a machine-readable event, or surface an unhealthy doctor status when connected follow mode stops advancing observable freshness.

## Reproduction Assessment

No high-confidence live reproduction is available from current main. Source inspection does show the missing guard: follow mode has no stale-freshness ticker and only reacts to disconnect-like events.

## Solution Assessment

Unclear until maintainers choose the desired failure mode. Reconnecting on stale follow is likely narrow, while adding doctor thresholds or a watchdog command changes the CLI contract.
