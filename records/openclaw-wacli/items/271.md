## What This Changes

This PR clones the default direct-media HTTP transport, adds a 10s response-header timeout, routes `downloadDirectBytes` through the dedicated client, and adds timeout/routing tests.

## Best Possible Solution

Land the focused transport-phase bound with its regression tests once maintainers accept the compatibility tradeoff and normal CI/merge checks are satisfactory.

## Reproduction Assessment

Yes, from source inspection rather than a live failing run. Current main routes read-only media downloads to `downloadDirectBytes`, which uses `http.DefaultClient.Do(req)` with no response-header-specific timeout, so a pre-header stall is only bounded by the caller context.

## Solution Assessment

Yes. Cloning `http.DefaultTransport`, adding only `ResponseHeaderTimeout`, and leaving `http.Client.Timeout` unset is a narrow maintainable fix; the remaining question is maintainer acceptance of the 10s compatibility cutoff.
