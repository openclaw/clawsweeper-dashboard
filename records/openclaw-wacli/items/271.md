## Summary

Keep open: the hardening direction is useful, but this PR still needs real direct-media behavior proof and the custom transport should preserve `http.DefaultTransport` HTTP/2 semantics before merge.

## What This Changes

The PR replaces direct WhatsApp media downloads' use of `http.DefaultClient` with a package-scoped phase-bounded HTTP client and adds tests for timeout and slow-body behavior.

## Best Possible Solution

Keep the phase-bound direct media client, preserve default transport protocol behavior by cloning `http.DefaultTransport` or opting HTTP/2 back in, and require redacted real direct-media proof before merge.

## Reproduction Assessment

Yes. Source inspection shows the read-only media command reaches `downloadDirectBytes`, which currently uses `http.DefaultClient.Do(req)`, so a pre-header stall is only bounded by the caller context on current main.

## Solution Assessment

No. Per-phase bounds and preserving the response-body budget are the right direction, but the implementation should preserve default HTTP/2 transport behavior before merge.
