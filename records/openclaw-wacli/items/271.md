## What This Changes

The PR adds a package-scoped direct media HTTP client cloned from `http.DefaultTransport`, sets a 10s response-header timeout, routes `downloadDirectBytes` through it, and adds focused timeout tests.

## Best Possible Solution

Land the phase-bound direct-media client after redacted real direct-media proof and explicit maintainer acceptance of the 10s pre-header cutoff, preserving the cloned default transport and no total body timeout.

## Reproduction Assessment

Yes, from source inspection rather than a live run. Current main routes read-only media downloads to `downloadDirectBytes`, which uses `http.DefaultClient.Do(req)` with no response-header-specific timeout, so a pre-header stall is only bounded by the caller context.

## Solution Assessment

Mostly yes. Cloning `http.DefaultTransport`, adding only `ResponseHeaderTimeout`, and leaving `http.Client.Timeout` unset is a narrow maintainable fix, but the 10s cutoff needs maintainer acceptance and real direct-media proof before merge.
