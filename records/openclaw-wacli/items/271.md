## Summary

Keep open: this is a useful direct-media hardening PR and the earlier total-download-timeout blocker was addressed, but it still lacks real behavior proof and the custom transport drops the HTTP/2 behavior provided by Go's default transport.

## What This Changes

The PR replaces `http.DefaultClient` in the direct WhatsApp media fallback with a dedicated phase-bounded HTTP client and adds regression tests for client configuration, stalled headers, and slow response bodies.

## Best Possible Solution

Keep the phase-bound client, but preserve default transport protocol behavior by cloning `http.DefaultTransport` or opting HTTP/2 back in, then require redacted real direct-media proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main reaches `downloadDirectBytes` from the read-only media command and uses `http.DefaultClient.Do(req)`, so a pre-header stall is not specifically bounded by a response-header timeout and falls back to the caller context.

## Solution Assessment

No. The per-phase timeout direction is good and the broad total timeout was removed, but the implementation should preserve default HTTP/2 transport behavior before merge.
