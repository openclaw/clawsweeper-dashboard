## Summary

Keep this PR open, but do not merge it as-is: the hardening is useful and current main still uses `http.DefaultClient`, but the proposed 30s `http.Client.Timeout` caps the entire media body download and can break legitimate large or slow direct media downloads.

## What This Changes

The PR adds a dedicated direct-media HTTP client with timeouts and idle connection limits, routes `downloadDirectBytes` through it, and adds regression tests for client configuration and stalled-server behavior.

## Best Possible Solution

Bound dial, TLS, and response-header stalls for direct media while preserving the parent command/context budget for valid response bodies, with regression tests and redacted real-run proof.

## Reproduction Assessment

Yes. Source inspection shows current main reaches `downloadDirectBytes` from the read-only media command and uses `http.DefaultClient.Do(req)`, so a pre-header stall is only bounded by the caller context.

## Solution Assessment

No. A dedicated client with per-phase timeouts is a good direction, but a 30s total client timeout is not the narrowest safe fix because it also caps legitimate response body downloads.

Full review comments:

- **[P1] Avoid capping full media downloads at 30 seconds:** `internal/wa/media.go:31`
  - body: `http.Client.Timeout` applies to the entire request, including reading the response body. Direct media downloads can be up to 100 MiB and currently run under the CLI's 5 minute command context, so this will make legitimate large downloads on slow links fail after 30s even after headers arrive. Keep the stall protection to dial/TLS/header phases, or use a body deadline that preserves the existing command budget.
