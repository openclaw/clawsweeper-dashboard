## Summary

Keep open. This is a fresh, well-scoped follow-up to the shipped call-events work: current main stores duration and outcome when WhatsApp delivers a call-log payload, but the `sync --once` path still has no dedicated post-call backfill and the only matching canonical work was the broader merged implementation.

## What This Changes

Request that `sync --once` populate duration and outcome for completed WhatsApp calls instead of leaving cron-based collectors with offer-only `call_events` rows.

## Best Possible Solution

Make completed call outcome and duration durable after reconnect when WhatsApp exposes the data, and document any protocol limit that prevents `sync --once` from recovering it.

## Reproduction Assessment

No, not with high confidence in this review. The reporter provides strong real-world evidence, and source inspection shows the relevant paths, but reproducing the failure requires a live WhatsApp call and a post-call `sync --once` run.

## Solution Assessment

Unclear. The current implementation is the right storage/parsing foundation, but the remaining fix depends on whether WhatsApp exposes completed call logs through an app-state/history path that `sync --once` can safely request.
