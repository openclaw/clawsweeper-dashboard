## Summary

Keep open: the report describes a plausible gap in the shipped call-events feature, and current main does not prove that `sync --once` backfills completed-call outcome or duration when only an offer event was captured. The parser and storage support exist, but the missing piece appears to be the sync/app-state delivery path, and a contributor has already commented that they are looking at it.

## What This Changes

Requests that `sync --once` and `wacli calls list` persist completed WhatsApp call outcome and duration instead of leaving offer-only rows.

## Best Possible Solution

Confirm which app-state or history path exposes completed call logs to linked devices, then either make `sync --once` reliably await/store those records with regression coverage or document the limitation if WhatsApp does not expose them.

## Reproduction Assessment

Unclear for a live current-main run: the reporter supplied concrete v0.9.2 output, and source inspection shows `sync --once` only stores completed-call details when WhatsApp emits a call-log message/app-state event. I did not reproduce against a live WhatsApp account.

## Solution Assessment

Unclear: the existing parser/storage path is the right foundation, but maintainers need to confirm whether the fix is to request/await the right call-log sync path or to document an inherent linked-device limitation.
