## Best Possible Solution

Land one cleaned presence fix that preserves caller cancellation for connect attempts, detaches only the websocket lifetime needed for graceful shutdown, serializes presence writes so unavailable is last, and leaves release notes to the release flow.

## Solution Assessment

- **[P2] Preserve connect cancellation when detaching the socket:** `internal/wa/client.go:146`
  - body: `Client.Connect` is used by non-sync commands under `withTimeout`/`--timeout`. Switching the shared `ConnectContext` call to `context.Background()` means a slow WhatsApp websocket connect no longer observes command timeout or cancellation while the process waits for the connect attempt to return. Please keep the websocket lifetime detached for graceful sync shutdown without dropping caller cancellation for connect attempts.
  - confidence: 0.82
- **[P2] Serialize available presence before cleanup:** `internal/app/sync_events.go:103-106`
  - body: These available-presence sends run in detached goroutines, while `Sync` later sends `PresenceUnavailable` synchronously after a successful run. On a short `sync --once` or shutdown path, cleanup can run first and a delayed goroutine can mark the linked device available again, defeating the PR's push-notification goal. Please serialize or drain these sends so unavailable is the last successful presence update.
  - confidence: 0.88
