## Summary

Current main still has the hybrid-search scoring defect described in the follow-up: when direct lexical recall wins deduplication, the retained entry can lack `embeddingId` and lose a vector score found for the same skill. Keep this issue open because https://github.com/openclaw/clawhub/pull/2591 is the specifically linked implementation candidate and must be reviewed or merged before the issue is resolved.

Likely related people
- @patrick91 — high confidence; recent contributor to the current search path and its June refactor.
- @vincentkoc — medium confidence; introduced the normalized prefix-recall behavior involved in this overlap.
- @steipete — medium confidence; introduced the earlier lexical fallback architecture that the current hybrid path builds on.

## What This Changes

Fix hybrid skill search so a direct lexical match retains the vector similarity score available for the same skill from vector recall.

## Best Possible Solution

Land a focused fix that indexes vector scores by skill identity without changing recall ordering, with regression coverage and a live `/api/v1/search?q=news` check showing lexical-retained results receive their vector contribution.

## Reproduction Assessment

Yes. The reporter provided a concrete production API query with derived component scores, and current-main source inspection independently confirms the exact deduplication and zero-vector-score path.

## Solution Assessment

Yes, the linked PR's direction is the narrowest maintainable fix: preserve vector scores by skill ID while retaining existing recall and merge ordering. It should include real Convex/API validation in addition to its focused regression test.
