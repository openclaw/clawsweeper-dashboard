## Summary

Current main clearly regenerates and promotes an embedding during publish, but neither the public API nor the supplied reproduction proves that `the-news` has a stale stored vector. The body-only keyword test is also inconclusive because current search requires query-token agreement with skill metadata before semantic candidates are returned, so production inspection and exact reproduction terms are still needed.

## What This Changes

Investigate whether publishing a new skill version regenerates and stores the skill embedding used by live search.

## Best Possible Solution

Add or use a maintainer-only diagnostic that compares the latest skill version with its embedding row, mapping, visibility, timestamp, and input fingerprint; repair or regenerate the vector only if those records disagree, while treating body-only semantic discovery as a separate documented product decision.

## Reproduction Assessment

No. Current source establishes the intended refresh path and live search shows the skill, but the supplied body-only keyword check conflicts with current search eligibility rules and the exact terms, versions, and score observations needed for a controlled reproduction are missing.

## Solution Assessment

Unclear. The narrowest first step is production-state diagnosis rather than adding a second refresh path; maintainers must separately decide whether queries should be allowed to match SKILL.md body content without metadata token agreement.
