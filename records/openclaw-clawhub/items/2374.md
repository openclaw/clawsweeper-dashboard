Keep open: current main builds and stores a new embedding during publish, but the reported public symptom still needs maintainer follow-up because live search behavior is consistent with the current metadata-gated relevance path and the stored production vector is not externally inspectable.

## What This Changes

The issue reports that publishing new versions of `the-news` with substantially different `SKILL.md` content does not appear to refresh the embedding or affect live search ranking.

## Best Possible Solution

A maintainer/operator should compare the latest stored embedding/version for `the-news`, then choose a narrow publish-path fix if stale or a documented search/diagnostics contract if the ranking behavior is intentional.

## Reproduction Assessment

Partly: live public API checks and current source reproduce the body-only-term symptom through the metadata/search eligibility gate. There is no high-confidence way from public data alone to prove the stored latest production embedding itself is stale.

## Solution Assessment

Unclear: the outside symptom cannot distinguish a stale vector from the current relevance contract or missing diagnostics. The safer path is to inspect stored production embedding state first, then decide between a publish-path repair, diagnostics, or documentation.
