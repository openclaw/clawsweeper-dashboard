## Summary

Keep open: current main generates a new embedding during each publish and marks the newest approved embedding searchable, but public evidence cannot establish whether the production `the-news` vector corresponds to its latest version. The reported body-term symptom is also inconclusive because current search discards vector candidates that lack a slug, display-name, capability-tag, or summary match.

## What This Changes

Investigate whether republishing a skill refreshes its stored search embedding and exposes changed skill content through search.

## Best Possible Solution

Have an operator compare the production skill’s latest version with its active `skillEmbeddings` and `embeddingSkillMap` records, repair any stale or zero-vector publish outcome with real Convex regression coverage, and handle semantic-only body matches as a separate search-contract decision.

## Reproduction Assessment

No, there is not yet a high-confidence reproduction of a stale vector. Source inspection identifies the complete publish/search path, but confirming the claim requires production embedding/version state plus the exact version changes and queries; body-only query misses can occur under the current eligibility rules even with a fresh vector.

## Solution Assessment

Unclear until production state is inspected. A mismatched, missing, or zero vector calls for a narrow publish-time repair, while a current vector would shift the remaining question to whether semantic-only SKILL.md body matches should be eligible search results.
