## Summary

The issue remains necessary: current main can retain the lexical copy of a duplicated skill and then assign it a zero vector contribution. The paired open PR is the canonical implementation path, so this issue should stay open until that PR is resolved.

## What This Changes

Fix hybrid skill search so direct lexical matches retain the vector similarity score computed for the same skill.

## Best Possible Solution

Land a focused skill-ID score-preservation fix with the regression test and a real `/api/v1/search?q=news` before/after check, without changing embedding generation or recall ordering.

## Reproduction Assessment

Yes, from source inspection: return one direct lexical entry without `embeddingId` and one vector-hydrated entry for the same skill; current de-duplication retains the lexical entry and scoring supplies zero vector similarity. The paired PR includes this exact regression setup, though a live current-main run was not performed in this read-only review.

## Solution Assessment

Yes. Preserving the already-computed vector score by skill ID is the narrowest maintainable repair because it leaves embedding generation, recall order, and the public response contract unchanged.
