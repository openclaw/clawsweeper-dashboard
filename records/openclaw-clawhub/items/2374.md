## Summary

Keep open: current main has a credible fresh-embedding write path, but neither public APIs nor the discussion prove that production `the-news` version 1.0.10 is linked to the correct latest-visible vector. The reproduced body-term symptom is not a reliable stale-vector test because current search deliberately rejects vector candidates without slug, display-name, summary, or capability-tag evidence.

## What This Changes

Investigate whether republishing changed `the-news` content refreshes the skill embedding used by live search.

## Best Possible Solution

Add or use a production operator diagnostic that compares the skill’s latest version ID with its latest-visible embedding row, timestamp, dimensions, and optionally a safe content fingerprint; repair or regenerate only inconsistent rows, and consider body-only semantic eligibility as a separate search-policy proposal.

## Reproduction Assessment

No, not for the claimed stale vector. Public search reproduces the missing body-term result, but current source and docs make that expected; confirming the alleged defect requires production Convex inspection of the latest version and embedding rows plus the reporter’s exact historical terms and versions.

## Solution Assessment

Unclear. A production integrity check and narrow row repair are the best path if linkage is stale; allowing pure SKILL.md-body semantic matches would be a separate product decision rather than the same bug fix.
