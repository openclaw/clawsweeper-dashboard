## Summary

Current main still reproduces the ranking defect: lexical-first deduplication can retain an entry without `embeddingId`, causing a zero vector contribution despite the same skill appearing in vector results. The issue should remain open because the author’s linked PR is still open and specifically owns the focused fix.

## What This Changes

Fix skill search scoring so a direct lexical match does not lose the vector similarity score from a duplicate vector-hydrated candidate.

## Best Possible Solution

Preserve vector similarity by skill ID when lexical and vector recall return the same skill, keep the existing recall ordering, and validate both the regression test and live `/api/v1/search` scoring through a real Convex deployment.

## Reproduction Assessment

Yes. On current main, a skill returned first by direct lexical recall can shadow its vector-hydrated duplicate, after which scoring sees no `embeddingId` and assigns a zero vector contribution.

## Solution Assessment

Yes. Mapping vector scores by skill ID is the narrowest maintainable repair because it restores the missing contribution without changing existing recall precedence, ranking weights, or public API shape.
