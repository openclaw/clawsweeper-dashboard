## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still contains the reported failure path: normal skill publish and manual package/plugin publish call the GitHub account-age gate, users missing cached `githubCreatedAt` trigger a GitHub API lookup, and 403/429 responses map to the exact rate-limit error.

## What This Changes

Publishing a skill should not fail with `GitHub API rate limit exceeded` during upload/publish.

## Best Possible Solution

Keep the account-age gate, verify production `GITHUB_TOKEN` and 403/429 logs, then reduce avoidable publish-time GitHub API dependence by persisting or backfilling `githubCreatedAt` for users missing it.

## Reproduction Assessment

Yes. Source inspection gives a concrete path for users without cached `githubCreatedAt`, and existing unit tests assert that 403/429 GitHub responses throw the same rate-limit error reached by skill and package publish.

## Solution Assessment

Unclear because no patch is proposed. The maintainable direction is to verify production configuration first, then add a targeted cache/backfill or error-handling mitigation rather than bypassing the account-age gate.
