## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-reproducible publish path where uncached GitHub account age triggers a GitHub API lookup whose 403/429 responses produce the reported rate-limit error; the Publish plugin comment aligns with package publish using the same gate.

## What This Changes

The issue asks for skill upload/publish to stop failing with `GitHub API rate limit exceeded` and to provide a usable recovery path.

## Best Possible Solution

Keep the account-age gate, verify production `GITHUB_TOKEN` and Convex 403/429 logs, then reduce avoidable publish-time GitHub lookups through safe caching, backfill, or clearer retry/reset handling without bypassing immutable-ID verification.

Yes, by source inspection. A normal publish by a user without cached `githubCreatedAt` calls `requireGitHubAccountAge`, and GitHub 403/429 responses throw the reported rate-limit text; I did not run a live production publish.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on production token/quota state and logs, and any code mitigation must preserve the security-sensitive GitHub account-age verification rather than bypass it.
