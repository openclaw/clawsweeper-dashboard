## Summary

Keep open. Current main still contains a source-reproducible publish path where missing cached GitHub account age triggers a GitHub API lookup, and GitHub 403/429 responses produce the reported rate-limit error; the Publish plugin comment matches the package publish path using the same gate.

## What This Changes

Publishing a skill should not fail with a GitHub API rate-limit error during ClawHub account-age verification.

## Best Possible Solution

Keep the account-age gate, verify production `GITHUB_TOKEN` and 403/429 logs, then reduce avoidable publish-time GitHub lookups through safe caching, backfill, or clearer retry/reset handling across web, API/import, and package publish without bypassing immutable-ID verification.

## Reproduction Assessment

Yes, by source inspection. A normal publish by a user without cached `githubCreatedAt` calls `requireGitHubAccountAge`, and GitHub 403/429 responses throw the reported rate-limit text; I did not run a live production publish.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on production token/quota state and logs, and any code mitigation must preserve the security-sensitive GitHub account-age verification rather than bypass it.
