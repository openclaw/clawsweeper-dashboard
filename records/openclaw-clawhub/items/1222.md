## Summary

Keep open. The ClawHub cleanup policy keeps issues open, and current main still has the reported source-reproducible publish failure path: normal skill publish reaches the GitHub account-age lookup, uncached users trigger a GitHub API request, and GitHub 403/429 responses map to the reported rate-limit error. The Publish plugin comment is also plausible because manual package publish uses the same account-age gate.

## What This Changes

The issue reports that publishing a new skill fails with a GitHub API rate-limit error during upload.

## Best Possible Solution

Keep the account-age gate intact, verify production GITHUB_TOKEN and 403/429 logs first, then reduce avoidable publish-time GitHub lookups or safely backfill/cache githubCreatedAt if maintainers choose a code mitigation.

## Reproduction Assessment

Yes, from source inspection rather than live production execution. Publish as a user without cached githubCreatedAt, have the GitHub account lookup return 403 or 429, and current main emits the reported rate-limit error.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on production token/quota state and logs, and any code change must preserve the security-sensitive GitHub account-age verification.
