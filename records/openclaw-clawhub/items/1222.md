## Summary

Keep open. Current main still has a source-reproducible publish path where uncached GitHub account age triggers a GitHub API lookup and 403/429 responses surface the reported rate-limit error; the related Publish plugin comment matches the package publish path sharing the same gate, and ClawHub cleanup policy keeps issues open for maintainer follow-up.

## What This Changes

Uploading or publishing a skill can fail with “GitHub API rate limit exceeded” during the GitHub account-age check.

## Best Possible Solution

Keep immutable-ID account-age verification, verify production `GITHUB_TOKEN` quota and Convex 403/429 logs, then reduce avoidable publish-time lookups through safe caching, backfill, or clearer retry/reset handling without weakening the anti-abuse gate.

## Reproduction Assessment

Yes, by source inspection. A normal publish by a user without cached `githubCreatedAt` calls `requireGitHubAccountAge`, and GitHub 403/429 responses throw the reported rate-limit error; I did not run a live production publish.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on production token/quota state and logs, and any code mitigation must preserve the security-sensitive account-age verification rather than bypassing it.
