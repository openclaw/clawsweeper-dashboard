## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-reproducible publish path where uncached GitHub account age triggers a GitHub API lookup and 403/429 responses produce the reported rate-limit error; the Publish plugin comment matches the manual package publish path using the same gate.

## What This Changes

The issue reports skill upload/publish, and a commenter reports plugin publish, failing with a GitHub API rate-limit error.

## Best Possible Solution

Keep the account-age gate, verify production GITHUB_TOKEN quota and Convex 403/429 logs, then reduce avoidable publish-time GitHub lookups through safe caching, backfill, or clearer retry/reset handling without weakening immutable-ID verification.

## Reproduction Assessment

Yes, by source inspection. A normal publish by a user without cached githubCreatedAt calls requireGitHubAccountAge, and existing tests show GitHub 403 or 429 responses throw the reported rate-limit text; I did not run a live production publish.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on production token/quota state and logs, and any code mitigation must preserve the security-sensitive account-age verification rather than bypassing it.
