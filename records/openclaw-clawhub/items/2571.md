## Summary

The PR is still necessary and the minimal API shape is reasonable, but the exact head retains a security-boundary bug: a blocked detail request sets the loaded markdown to `null` and then immediately falls back to the parsed long description. Real endpoint proof is also not yet strong enough for this external PR.

## What This Changes

The PR adds optional long descriptions and conservative `{key, required}` setup metadata to v1 skill list/detail responses, denormalizes descriptions into latest-version summaries, updates public schema packages, and adds regression tests.

## Best Possible Solution

Keep the minimal optional API contract, but branch explicitly on `descriptionAccessBlock` so no parsed-description fallback occurs when blocked, add a regression test for that exact case, and demonstrate hosted, GitHub-backed, and moderated responses against a real running Convex setup.

## Reproduction Assessment

Yes for the review finding: source inspection gives a high-confidence path where `descriptionAccessBlock` is truthy, `description` becomes null, and the response falls back to `latestVersion.parsed.description`. The broader feature itself is not a bug reproduction.

## Solution Assessment

No, not as currently written: the minimal schema is maintainable, but blocked access must override every description source rather than relying on a nullish fallback that conflates a blocked result with missing markdown.
