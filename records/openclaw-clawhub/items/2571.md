## Summary

Keep this PR open: the minimal API shape is sensible and moderation gating is preserved, but the setup builder can mislabel a mandatory environment key as optional, and the posted proof does not satisfy the repository’s real-Convex-runtime validation requirement.

## What This Changes

Adds long-form skill descriptions and conservative setup-key metadata to v1 skill list/detail responses, denormalized skill summaries, and public schema packages.

## Best Possible Solution

Make declarations under `requires.env` and `requires.config` unconditionally required, apply `envVars.required` only to envVars-only entries, and verify list/detail responses for fresh, existing, GitHub-backed, and moderated skills through a real Convex HTTP deployment.

## Reproduction Assessment

Yes. A skill declaring the same key in `requires.env` and in `envVars` with `required: false` deterministically produces `required: false` from the new builder, contrary to the documented format.

## Solution Assessment

No, not yet. The narrow `key`/`required` contract is a maintainable first release, but mandatory declarations must take precedence and the schema/API rollout needs real Convex validation.
