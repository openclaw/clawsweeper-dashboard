## Summary

Current main does not implement LaunchDarkly support, so this member-authored PR remains relevant if maintainers want that capability. It should not merge yet because it silently removes the existing env-backed Souls setting, sends stable account attributes to a new third-party service without a recorded policy decision, leaves identity promise failures unhandled, and lacks inspectable real-browser proof.

## What This Changes

Adds a LaunchDarkly React provider, typed feature-flag registry, authenticated and anonymous evaluation contexts, Souls link gating across header/home/footer, tests, setup documentation, and a new browser SDK dependency.

## Best Possible Solution

Keep LaunchDarkly as an optional remote override while retaining `VITE_FEATURE_SOULS` as the compatibility fallback, handle identity failures, minimize and document approved context attributes in durable repository intent, explicitly accept or replace the SDK dependency, and attach redacted browser evidence covering remote off/on/off plus the no-SDK upgrade path.

## Reproduction Assessment

Yes for the introduced compatibility defect: configure `VITE_FEATURE_SOULS=true` without `VITE_LAUNCHDARKLY_CLIENT_SIDE_ID`; current main resolves the flag true, while this branch resolves it false. The new capability itself is not a bug reproduction request.

## Solution Assessment

No, not in its current form. Central remote evaluation is plausible, but replacing a supported local setting, exporting account attributes without a durable policy decision, and leaving rejected identity calls unhandled are avoidable regressions.
