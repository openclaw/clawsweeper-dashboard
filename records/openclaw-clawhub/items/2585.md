## Summary

This PR remains necessary only if maintainers want LaunchDarkly as ClawHub’s browser flag backend, but it is not merge-ready: it breaks the existing Souls environment flag on upgrade, leaves identity failures unhandled, introduces an unapproved authenticated vendor-data and browser-SDK boundary, and lacks inspectable real behavior proof.

## What This Changes

Adds a LaunchDarkly React provider, maps the Souls flag to `clawhub-souls`, gates Souls links in the header, home page, and footer, and documents the new client-side configuration.

## Best Possible Solution

Preserve `VITE_FEATURE_SOULS` as the no-SDK and failure fallback, adopt a maintainer-approved pseudonymous minimal context, handle and retry failed identity transitions, record the security-sensitive vendor contract in `specs/`, and provide redacted real-browser proof before merge.

## Reproduction Assessment

Yes for the patch defects: source inspection shows that a deployment configured only with `VITE_FEATURE_SOULS=true` receives the false fallback, and both `identify()` promises lack rejection handling. The advertised live UI transitions are not independently reproducible from the evidence posted to this PR.

## Solution Assessment

No. A remote flag service may be useful, but replacing the supported environment path and exporting internal identity attributes is not the narrowest upgrade-safe implementation; compatibility and data minimization should be built into the provider boundary.
