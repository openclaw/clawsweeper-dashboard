## Summary

This member-authored PR should remain open for explicit maintainer judgment: it introduces a coherent remote-flag layer, but currently breaks the established `VITE_FEATURE_SOULS` upgrade path, leaves SDK identity failures unhandled, sends internal user identifiers and roles to a new browser vendor, and provides no inspectable real-browser artifact.

Adds a LaunchDarkly React provider, maps the Souls discovery flag to `clawhub-souls`, gates header/home/footer links, and documents the new client-side configuration.

## Best Possible Solution

Preserve the existing environment flag as the unconfigured/outage fallback, handle identity-update failures, approve and minimize the third-party context fields and dependency boundary, and attach real browser evidence for live off/on/off transitions before merge.

## Solution Assessment

No, not as currently proposed. A narrower and safer integration would retain the existing deployment control as a compatibility fallback and require an explicit decision before exporting authenticated context fields.
