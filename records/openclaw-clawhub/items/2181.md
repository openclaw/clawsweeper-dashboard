## Summary

Keep open for maintainer/operator follow-up: current main intentionally treats self-deleted accounts as deactivated and purged, and the normal unban path refuses those accounts, so this is a security-sensitive account recovery decision rather than conservative cleanup.

## What This Changes

The requester asks maintainers to clear a purged/deactivated GitHub-linked ClawHub account state so they can register or sign in again.

## Best Possible Solution

Define one supported deleted-account lifecycle: either keep deletion terminal with clear support messaging, or add an audited verified restore/unlink/recreate flow that preserves deletion, OAuth identity, moderation, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level. A user who runs deleteAccount gets deactivatedAt/purgedAt, later OAuth sign-in is rejected by handleDeletedUserSignIn, and the admin unban path throws for deactivated accounts; I did not verify the reporter's production row.

## Solution Assessment

No for a direct automated clear or ordinary unban. The maintainable solution needs maintainer security/product judgment and an audited account-recovery, identity-unlink, recreate, or terminal-support policy before changing purged/deactivated account behavior.
