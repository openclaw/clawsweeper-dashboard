## Summary

Keep open for maintainer/operator handling: this is an open ClawHub issue, current main intentionally makes self-deleted accounts deactivated/purged and non-restorable by normal login or unban, and resolving it requires production account access plus security/product judgment.

## What This Changes

The reporter asks maintainers to clear their GitHub-linked ClawHub account's purged/deactivated state so they can register again.

## Best Possible Solution

Define one supported deleted-account lifecycle: keep deletion terminal with clear support messaging, or add an audited verified restore/unlink/recreate flow that preserves OAuth identity, moderation, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level. A user who runs `deleteAccount` gets `deactivatedAt`/`purgedAt`, later OAuth sign-in is rejected by `handleDeletedUserSignIn`, and the admin unban path throws for deactivated accounts; I did not verify the reporter's production row.

## Solution Assessment

No for a direct automated clear or ordinary unban. The maintainable solution needs maintainer security/product judgment and an audited account-recovery, identity-unlink, recreate, or terminal-support policy before changing purged/deactivated account behavior.
