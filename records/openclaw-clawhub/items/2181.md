## Summary

Keep open for maintainer/operator handling: this is a security-sensitive production account recovery request, and current main intentionally treats self-deleted accounts as permanently deactivated rather than ordinary unban candidates.

## What This Changes

The requester asks ClawHub operators to clear the purged/deactivated state for their GitHub-linked account so they can register or sign in again.

## Best Possible Solution

Define one supported account-deletion lifecycle: either keep deletion terminal with clear support messaging, or add an audited verified restore/unlink/recreate flow that preserves identity, moderation, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level. A user who runs deleteAccount gets deactivatedAt/purgedAt, later OAuth sign-in is rejected by handleDeletedUserSignIn, and the admin unban path refuses deactivated accounts; I did not verify the reporter's production row.

## Solution Assessment

No for a direct unban or automated flag clear. The maintainable path needs maintainer security/product judgment and an audited account recovery, GitHub identity unlink, recreate, or terminal-support policy.
