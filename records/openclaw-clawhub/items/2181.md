## Summary

Keep open for maintainer/security follow-up: current main intentionally treats self-deleted ClawHub accounts as deactivated/purged, and the existing unban path refuses deactivated accounts, so this is a production account-recovery decision rather than conservative cleanup.

## What This Changes

The issue asks maintainers to clear a purged or deactivated GitHub-account mark so the reporter can register and use ClawHub again.

## Best Possible Solution

Define one supported deleted-account lifecycle: keep deletion terminal with clear support messaging, or add an audited verified restore/unbind/recreate flow that preserves deletion, OAuth identity, and moderation safeguards.

## Reproduction Assessment

Yes, at source level. A user who runs deleteAccount gets deactivatedAt/purgedAt, later OAuth sign-in is rejected by handleDeletedUserSignIn, and the admin unban path throws for deactivated accounts; I did not verify the reporter's production row.

## Solution Assessment

No for an automated direct clear. The maintainable solution needs maintainer security/product judgment and an audited account-recovery or terminal-support policy before changing purged/deactivated account behavior.
