Keep open for maintainer/operator follow-up: current main intentionally makes self-service deletion terminal and ordinary unban refuses deactivated accounts, so this is security-sensitive account recovery rather than conservative cleanup or an automated code fix.

## What This Changes

The reporter asks maintainers to clear the purged/deactivated state on their GitHub-linked ClawHub account so they can register or sign in again.

## Best Possible Solution

Define a supported deleted-account lifecycle: either keep deletion terminal with clearer support messaging, or add an audited, identity-verified restore/unlink/recreate flow that preserves deletion, OAuth, moderation, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level. A user who runs deleteAccount gets deactivatedAt and purgedAt, later OAuth sign-in is rejected by handleDeletedUserSignIn, and admin unban refuses deactivated accounts; this review did not verify the reporter's production row.

## Solution Assessment

No for a direct automated clear or ordinary unban. The maintainable solution needs maintainer security/product judgment plus audited identity-verified recovery, identity unlinking, account recreation, or an explicit terminal-support policy.
