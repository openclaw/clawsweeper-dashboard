## Summary

Keep open for maintainer/operator handling: current main intentionally makes user deletion terminal and blocks GitHub reauthentication for deactivated/purged accounts, while the requested clear/unban action would cross account recovery, OAuth identity, and deletion-policy boundaries.

## What This Changes

The issue asks maintainers to manually clear a purged/deactivated GitHub-linked ClawHub account so the reporter can register again.

## Best Possible Solution

Define a supported deleted-account lifecycle: either keep deletion terminal with clear support messaging, or add an audited, identity-verified restore/unlink/recreate flow that preserves deletion, OAuth, moderation, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level. A user who runs deleteAccount gets deactivatedAt and purgedAt, later OAuth sign-in is rejected by handleDeletedUserSignIn, and the admin unban path refuses deactivated accounts; I did not verify the reporter's production row.

## Solution Assessment

No for a direct automated clear or ordinary unban. The maintainable solution needs maintainer security/product judgment plus audited identity-verified recovery, identity unlinking, account recreation, or an explicit terminal-support policy.
