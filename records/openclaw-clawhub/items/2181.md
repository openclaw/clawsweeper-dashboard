## Summary

Keep open for maintainer/security follow-up: current main intentionally treats self-deleted ClawHub accounts as deactivated/purged, the unban path refuses deactivated accounts, and this issue asks for production account recovery rather than cleanup closure.

## What This Changes

The reporter asks maintainers to clear a purged/deactivated GitHub-linked ClawHub account so they can re-register and continue using the service.

## Best Possible Solution

Define one supported deleted-account lifecycle: keep deletion terminal with clear support messaging, or add an audited verified restore/unlink/recreate flow that preserves deletion, OAuth identity, and moderation safeguards.

## Reproduction Assessment

Yes, at source level. A user who runs deleteAccount gets deactivatedAt/purgedAt, later OAuth sign-in is rejected by handleDeletedUserSignIn, and admin unban throws for deactivated accounts; I did not verify the reporter's production row.

## Solution Assessment

No for an automated direct clear. The maintainable solution needs maintainer security/product judgment and an audited account-recovery or terminal-support policy before changing purged/deactivated account behavior.
