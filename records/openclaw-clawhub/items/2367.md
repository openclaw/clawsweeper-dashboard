## Label Justifications

- `P2`: The limitation blocks four known published packages from a consistent ownership state but is a bounded registry improvement rather than an active outage.
- `impact:security`: The scope-to-publisher rule is an authorization boundary preventing actors from publishing into namespaces they do not control.

## Summary

This remains a valid, intentionally unresolved publisher-identity limitation on current `main`; it must stay open because no canonical successor or implementation exists and the collaborator-authored, security-sensitive product decision requires explicit maintainer handling.

## What This Changes

Support npm-valid package scopes containing dots or underscores as secure ClawHub publisher identities, or provide a first-class equivalent mapping.

## Best Possible Solution

Choose one canonical publisher identity contract, document it in `specs/orgs.md`, then update validation, creation, routing, authorization, transfer, and migration behavior together with focused unit tests and a real Convex publish/transfer check for dotted and underscored scopes.

## Reproduction Assessment

Yes. Current-main validators, error handling, documentation, and regression tests deterministically reproduce rejection of dotted or underscored npm scopes.

## Solution Assessment

Unclear among the proposed options. Broadening the canonical handle grammar is simpler than aliases, but maintainers must first confirm URL, collision, migration, and ownership semantics; a local publish exception would not be sufficient.
