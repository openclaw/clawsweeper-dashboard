## Summary

Keep open: current main accepts npm-style package scopes with dots/underscores in package names, but publisher handles remain slug-only and scoped package ownership is enforced by exact handle match. The remaining work is an intentional publisher identity/security decision, not a cleanup close candidate.

## What This Changes

The issue asks ClawHub to decide how npm-valid package scopes containing dots or underscores should map to publisher handles so already-published plugin packages can be repaired without package renames.

## Best Possible Solution

Maintainers should choose one ownership model, then implement it consistently across publisher creation, publish and transfer validation, routing, docs, and regression tests for dot/underscore npm scopes.

## Reproduction Assessment

Yes at source level: current main accepts npm package scopes with dots/underscores but rejects matching publisher handles and enforces exact scope-owner matching. I did not run a live publish because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the identity model. Expanding publisher handles and adding first-class aliases are both plausible, but each needs URL, migration, and authorization design before implementation.
