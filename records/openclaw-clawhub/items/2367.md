## Summary

Keep open: current main still allows npm-style dotted/underscored package scopes while publisher handles remain hyphen-only and publish/transfer flows require exact scope-to-publisher ownership. The author is a collaborator, and the remaining work is a security-sensitive publisher identity decision rather than a cleanup close.

## What This Changes

The issue asks ClawHub to support or intentionally resolve npm-valid dotted/underscored package scopes that cannot currently map to matching ClawHub publisher handles.

## Best Possible Solution

Maintainers should choose one publisher identity model and implement it consistently across publisher creation, publish, transfer, routing, docs, and regression tests with explicit namespace authorization invariants.

## Reproduction Assessment

Yes, at source level: current main accepts package names like `@foo.bar/demo-plugin`, rejects matching publisher handles, and tests the invalid-handle guidance. I did not run a live publish in this read-only review.

## Solution Assessment

Unclear until maintainers choose the identity model. Expanding publisher handles and adding first-class aliases are both plausible, but either path needs URL, migration, and authorization design before implementation.
