## Summary

Current main and v0.20.2 still reject these npm-valid scopes, and no related PR implements an alternative. This issue should remain open for explicit maintainer judgment because it is collaborator-authored, protected by maintainer/security labels, and requires choosing a durable publisher identity contract. ## What This Changes

Decide whether npm scopes containing dots or underscores should be valid publisher handles, require package renaming, or map through explicit publisher aliases.

## Best Possible Solution

Define npm scope identity as a documented publisher invariant, preferably by safely accepting npm-compatible dots and underscores in publisher handles if collision and routing analysis permits; otherwise add a first-class, unique, auditable scope alias resolved consistently by every authorization path.

## Reproduction Assessment

Yes. Current main extracts the npm scope, requires exact equality with the selected publisher handle, and separately rejects dots and underscores through the publisher handle pattern.

## Solution Assessment

Unclear. The report accurately identifies the available directions, but maintainers must decide whether npm identity and canonical URL identity remain unified or gain an explicit mapping before a safe implementation can be selected.
