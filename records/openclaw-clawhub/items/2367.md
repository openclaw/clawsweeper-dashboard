## Summary

Keep open: current main still accepts npm-valid scoped package names containing dots/underscores while ClawHub publisher handles remain slug-only and scoped package ownership is enforced by exact owner match. The remaining fix is a publisher identity and authorization decision, so this is not a cleanup-close candidate.

## What This Changes

The issue asks ClawHub to support npm-valid scoped plugin packages whose scopes contain dots or underscores, either by expanding publisher handles, requiring package renames, or adding scope-to-publisher aliases.

## Best Possible Solution

Maintainers should choose one publisher identity model, then implement it consistently across publisher creation, publish and transfer validation, routing, docs, and regression tests for dot/underscore npm scopes.

## Reproduction Assessment

Yes, at source level: current main accepts package scopes with dots/underscores, rejects matching publisher handles, and enforces exact scope-to-owner matching. I did not run a live publish because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the identity model. Expanding publisher handles and adding first-class aliases are both plausible, but each needs URL, migration, and authorization design before implementation.
