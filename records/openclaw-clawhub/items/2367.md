## Summary

Keep open: current main still has the reported mismatch between npm-valid dotted/underscored package scopes and slug-only ClawHub publisher handles, and resolving it changes publisher identity plus publish authorization boundaries.

## What This Changes

Issue requests a decision and implementation path for npm-valid package scopes containing dots or underscores that cannot currently map to ClawHub publisher handles.

## Best Possible Solution

Maintainers should choose one publisher identity model, then implement it consistently across publisher creation, publish and transfer validation, routing, docs, and regression tests for dotted and underscored npm scopes.

## Reproduction Assessment

Yes, at source level: current main accepts npm-style dot/underscore scopes, rejects matching publisher handles, and enforces exact package-scope-to-owner matching. I did not run a live publish because this was a read-only review.

## Solution Assessment

Unclear until maintainers choose the identity model. Expanding publisher handles, requiring package renames, or adding first-class aliases are all plausible, but each carries different URL, migration, and authorization consequences.
