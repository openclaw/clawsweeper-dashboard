## Summary

Keep open: current main still has the reported npm-scope versus publisher-handle mismatch, and the remaining work is a publisher identity and namespace-authorization decision rather than a conservative cleanup close.

## What This Changes

Request to make scoped plugin package ownership work for npm-valid scopes containing dots or underscores, or provide a safe alias/rename path for already-published packages.

## Best Possible Solution

Choose one publisher identity model, document it, then implement it consistently across publisher creation, publish validation, transfers, package routing, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: current main accepts package scopes with dots or underscores, rejects matching publisher handles, and enforces exact scope-to-owner matching. I did not run a live publish because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the identity model. Expanding publisher handles, adding first-class aliases, or requiring package renames are all product/security choices with different URL and authorization consequences.
