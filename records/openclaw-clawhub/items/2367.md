## Summary

Current main intentionally rejects npm scopes that cannot be represented by ClawHub publisher handles, and no implemented alias or expanded-handle path was found. This collaborator-authored, security-sensitive issue still needs an explicit maintainer product decision because changing canonical publisher identity affects ownership, routing, transfers, and migration behavior.

## What This Changes

Support existing npm package scopes containing dots or underscores without weakening ClawHub's scope-to-publisher ownership enforcement.

## Best Possible Solution

Choose a single canonical publisher identity model, document its security and routing invariants in `specs/`, and implement collision-safe migration plus consistent authorization across publish, transfer, import, install, and public URL flows.

## Reproduction Assessment

Yes at source level. Current main explicitly requires exact scope-to-owner equality and tests the rejection path for scopes that are invalid ClawHub handles, although no live Convex reproduction was run during this read-only review.

## Solution Assessment

Unclear until maintainers select the identity contract. Expanded canonical handles, mandatory npm renames, and first-class aliases each solve the immediate block but have materially different routing, migration, and authorization costs.
