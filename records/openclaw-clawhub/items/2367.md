- `impact:security`: Package-scope-to-publisher matching controls who may publish executable plugin packages under a given npm identity.

## Summary

Current main still rejects npm scopes containing dots or underscores while requiring exact scope-to-publisher matching, so the four identified packages remain blocked. This is an intentional security-sensitive identity constraint rather than an already-fixed defect, and the collaborator-authored issue requires maintainer judgment on handles, aliases, or package renames.

## What This Changes

Allow npm-valid dotted or underscored package scopes to map to ClawHub publishers without renaming already-published packages.

## Best Possible Solution

Choose and document one canonical identity model, preserve unambiguous npm ownership, migrate the four affected packages, and validate publisher creation, publishing, transfer, API lookup, installation, and public URLs through real Convex flows.

## Reproduction Assessment

Yes at source level: dotted or underscored scopes fail publisher-handle validation, while choosing a similar slug-safe owner fails the exact scope-owner equality check. This read-only review did not execute a live Convex mutation.

## Solution Assessment

Unclear until maintainers select the identity contract: npm-compatible handles, explicit aliases, and forced package renames each have materially different URL, migration, and authorization consequences.
