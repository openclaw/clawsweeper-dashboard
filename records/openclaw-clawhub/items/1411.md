## Summary

Keep open. Current main still lacks editable-summary support, and the latest PR branch appears to have fixed the earlier invalid DOM concern but still has blocking backend authorization, moderation/search consistency, and validation issues; the related #1345 use case remains unresolved until a revised implementation lands.

## What This Changes

The PR adds a public `skills.updateSummary` mutation plus skill-detail/header UI and CSS for inline editing a skill summary from the detail page.

## Best Possible Solution

Land a revised editable-summary flow that treats the summary as validated public metadata, uses publisher-aware ownership checks, and keeps detail, search digest, API, future publish behavior, and moderation state consistent.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current main gives a high-confidence path: publisher members satisfy the current manage UI gate while the new mutation rejects them, and successful edits patch only the skill row while moderation and digest-derived surfaces remain stale.

## Solution Assessment

No. The PR writes public summary text directly; the safer maintainable path is a validated metadata update that either integrates with publish-derived summary semantics or explicitly models manual overrides while synchronizing moderation and search state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep summary edits on the moderation path:** `convex/skills.ts:4814-4818`
  - body: The publish path derives moderation flags and `isSuspicious` from the summary that gets persisted, but this mutation patches only `summary` and `updatedAt`. Owners could change visible searchable text after review while moderation fields still describe the old summary, so recompute moderation or define an explicit reviewed override flow before writing public metadata.
  - confidence: 0.9
- **[P1] Authorize publisher managers consistently:** `convex/skills.ts:4808-4810`
  - body: The edit control is exposed through the existing manage surface, which includes members of `skill.ownerPublisherId`, but `updateSummary` only allows the direct `ownerUserId` or a moderator. Publisher-owned skill managers can open the editor and then always fail on save; align the mutation with `assertCanManageOwnedResource` or narrow the UI gate.
  - confidence: 0.93
- **[P2] Sync the search digest when summaries change:** `convex/skills.ts:4814-4818`
  - body: Catalog, search, and API hydration read the copied `summary` from `skillSearchDigest`, while this mutation updates only the `skills` document. After saving, detail views can show the new value while list/search/API surfaces continue serving the old summary.
  - confidence: 0.88
- **[P2] Normalize and cap editable summaries:** `convex/skills.ts:4800-4803`
  - body: The new public mutation accepts raw `v.string()` and writes it into public skill metadata, unlike the existing summary pipeline that normalizes whitespace and caps summaries. Add server-side normalization and a matching textarea limit so malformed clients cannot save response-bloating text.
  - confidence: 0.84
