## Summary

Keep this issue open. Current main still has a clear source-level path where an ordinary non-trusted publisher update rewrites the skill row to hidden pending-scan state, and the public search/list/detail/API paths filter that row out instead of serving the last approved public version.

## What This Changes

The issue reports a previously published skill disappearing from search/listings after a web UI update attempt, with a server error and concern about preserving the slug/account ownership.

## Best Possible Solution

Preserve slug ownership and public continuity by serving the last approved public version while a new upload is pending scan/review, while still hiding malicious, quarantined, removed, manually locked, and malware-blocked content with owner-visible diagnostics.

## Reproduction Assessment

Yes, with high confidence from source inspection rather than a live production slug: update an active skill as an ordinary non-trusted publisher, observe insertVersion writing hidden pending-scan state to the skill row, then observe public search/list/detail/API paths filtering that row out.

## Solution Assessment

No patch is under review, and a simple preserve-active patch is not clearly the best fix because it can expose unscanned latest content. The safer maintainable direction is a last-approved-version continuity model with scanner/moderation state attached to the pending upload.
