## Summary

Keep this issue open. Current main still has a source-level path where an ordinary active-skill update writes hidden pending-scan state onto the skill row, and public search/list/detail paths filter that row out; the remaining fix needs maintainer moderation policy rather than cleanup closure.

## What This Changes

A published skill disappeared from search/listings after a web UI update returned a server error, with follow-up concern about preserving the existing slug and account ownership.

## Best Possible Solution

Serve the last approved public version for active skills while a new upload is pending scan or review, preserve slug ownership, and keep genuinely quarantined or malicious content hidden with owner-visible diagnostics.

## Reproduction Assessment

Yes, at source level. Update an active skill as an ordinary non-trusted publisher, observe `insertVersion` writing hidden pending-scan moderation state to the skill row, then observe public search/detail/API hydration filtering that row out.

## Solution Assessment

No patch is under review, and a blanket visibility-preservation patch would not be the safest fix. The maintainable direction is a last-approved-version model that keeps existing public reads stable while the new upload remains pending moderation.
