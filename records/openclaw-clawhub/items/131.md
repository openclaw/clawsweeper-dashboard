## Summary

Keep this issue open. Current main still has a credible publish-update path where a normal non-trusted publisher updates an active skill, the skill row is rewritten to hidden pending-scan moderation state, and public search/list/detail/API paths filter it out; ClawHub policy also keeps issues open for maintainer resolution.

## What This Changes

The issue reports that updating an already published skill can produce a server error and remove the active skill from public search/list/detail/API results while the new version is pending scan.

## Best Possible Solution

Preserve slug ownership and public continuity by serving the last approved public version while a new upload is pending scan or review, while still hiding static-malicious, quality-quarantined, user-moderated, removed, manually locked, and malware-blocked content with owner-visible diagnostics.

## Reproduction Assessment

Yes. A code-level reproduction path is clear: update an active skill as an ordinary non-trusted publisher, observe insertVersion writing hidden pending-scan state to the skill row, then observe public search/list/detail/API hydration filtering that row out.

## Solution Assessment

No patch is under review, and the narrowest maintainable solution is not simply preserving active status for the new upload. The safer direction is a last-approved-version model that keeps public reads stable while the new version remains pending moderation.
