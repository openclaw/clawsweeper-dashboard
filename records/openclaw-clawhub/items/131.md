## Summary

Keep this issue open. Current main still has a source-level path where a normal non-trusted publisher updating an active skill can move the skill row to hidden/pending moderation, and public search/list/detail/API paths then filter it out; the fix also needs maintainer moderation/security policy rather than conservative cleanup closure.

## What This Changes

The issue reports that updating a previously published skill can return a server error and make the skill disappear from search, listings, direct reads, and API consumers while slug ownership must be preserved.

## Best Possible Solution

Preserve slug ownership and public continuity by serving the last approved version while a new upload is pending scan/review, while still hiding malicious, quality-quarantined, manually moderated, removed, and malware-blocked content with clear owner diagnostics.

## Reproduction Assessment

Yes, at source level. Updating an active skill as an ordinary non-trusted publisher flows through insertVersion, writes hidden pending-scan moderation to the skill row, and public search/list/detail/API hydration rejects that row.

## Solution Assessment

No patch is under review. The safest fix is not wholesale public exposure of pending uploads, but a last-approved-version continuity model that keeps public reads stable while moderation evaluates the new version.
