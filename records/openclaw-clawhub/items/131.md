## Summary

Keep open. ClawHub cleanup policy keeps issues open in this pass, and current main still has a source-level path where an ordinary update can write hidden pending-scan state onto an existing skill row while public search/list/detail hydration filters hidden rows out; related production restorations do not verify the reporter’s private slug or establish a last-approved-version implementation on main.

## What This Changes

The issue asks to restore search/listing/API visibility and slug continuity for a previously published skill that disappeared after an update/publish attempt.

## Best Possible Solution

Implement a version-aware public projection that preserves slug ownership and serves the last approved public skill card, digest, searchable embedding, API metadata, and safe file surface while a new version remains gated by scan or moderation review.

## Reproduction Assessment

Yes, at source level: an ordinary non-trusted publisher updating an already-active skill can reach insertVersion, have the skill row patched to hidden pending scan, sync that state into search digest, and then be filtered out by public search/list/detail conversion. I did not reproduce the reporter’s private production row or the UI server-error toast directly.

## Solution Assessment

No patch is under review. The direct preserve-active approach in https://github.com/openclaw/clawhub/pull/274 was closed on moderation-policy grounds, so the safer direction is last-approved-version continuity while the new upload remains gated.
