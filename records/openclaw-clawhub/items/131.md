## Summary

Keep this issue open. Current main still has a source-backed path where a normal non-trusted update of an already published skill writes hidden pending-scan moderation state onto the canonical skill row, and public search/list/detail conversion then filters it out; ClawHub cleanup policy also keeps issues open for maintainer resolution.

## What This Changes

The issue reports that updating an already published skill via the web UI caused a server error and made the skill disappear from search/listings while the reporter wants slug ownership preserved.

## Best Possible Solution

Preserve slug ownership and public continuity by serving the last approved skill projection while a new upload is pending scan/review, then promote the new version only after moderation clears it, with owner-visible diagnostics for held content.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: update an active skill as an ordinary non-trusted publisher, observe `insertVersion` write hidden pending-scan state to the skill row, then observe public search/list/detail paths drop the row through `toPublicSkill`.

## Solution Assessment

No patch is under review. The narrow maintainable direction is a last-approved-version public projection, not simply leaving the new unscanned upload active or continuing to hide the whole skill during review.
