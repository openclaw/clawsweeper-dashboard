## Summary

Keep open. Current main still has internal per-version soft-delete state and public gates for withdrawn skill versions, but it does not expose an owner-facing CLI, HTTP API, or management workflow to hide/delete/restore one published skill version while leaving the skill live.

## What This Changes

Add owner-facing controls to hide, delete, unhide, or undelete an individual published skill version without deleting the whole skill.

## Best Possible Solution

Define the version-scoped lifecycle policy, then implement authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt, write version-specific audit logs, and keep version lists, detail/file/download endpoints, tags/latest, docs, tests, and restore behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because current main models and gates withdrawn versions, while the exposed owner API, CLI, docs, and management controls remain skill-scoped.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. The requested owner-facing version workflow is directionally right, but the maintainable fix must decide latest-version, tag, cache, download, restore, permission, and audit semantics first.
