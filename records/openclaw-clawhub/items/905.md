## Summary

Keep open. This issue is a valid feature/policy gap: current main can represent and hide soft-deleted skill versions internally, but there is still no owner-facing per-version CLI, HTTP API, or site workflow, and the repository policy for ClawHub keeps issues open for maintainer handling.

## What This Changes

The issue requests owner-facing controls to hide, delete, restore, or undelete an individual published skill version without deleting the whole skill.

## Best Possible Solution

Define the version lifecycle policy first, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt with audit logs and consistent public list/detail/file/download behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because the internal soft-delete state and public filtering exist, but the owner-facing per-version workflow is absent.

## Solution Assessment

Unclear until maintainers settle the lifecycle policy. A version-scoped owner API plus CLI/UI support is directionally right, but latest-version, tag, cache, download, restore, authorization, and audit semantics should be defined before implementation.
