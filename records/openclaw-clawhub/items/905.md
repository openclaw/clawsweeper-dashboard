## Summary

Keep open. Current main can model and publicly suppress soft-deleted skill versions, but it still lacks owner-facing CLI, HTTP API, docs, or UI controls to hide/delete/restore one published version while keeping the skill live; the feature also needs product and security policy.

## What This Changes

Request an owner-facing workflow to hide, delete, unhide, or undelete a single published skill version without soft-deleting the whole skill slug.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update `skillVersions.softDeletedAt`, write version-specific audit records, and keep lists, detail/file/download endpoints, tags/latest, docs, tests, and cache behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because version soft-delete state and public gates exist, while owner-facing per-version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. The requested version-scoped owner workflow is directionally right, but the maintainable implementation needs explicit latest, tag, download, cache, restore, permission, and audit semantics.
