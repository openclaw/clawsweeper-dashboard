## Summary

Keep open. Current main can represent and publicly suppress a soft-deleted individual skill version, but the owner-facing CLI, HTTP docs/API, and skill page controls still only hide/delete/restore whole skills; this also needs maintainer product and security policy before implementation.

## What This Changes

Request owner-facing controls to hide, delete, restore, or mark one published skill version as withdrawn while keeping the skill live.

## Best Possible Solution

Define the version-scoped lifecycle policy first, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt with version-specific audit logs and consistent list/detail/file/download/tag/latest behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the current gap because version soft-delete state and public gates exist while owner-facing per-version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must define latest, tag, download, cache, restore, permission, and audit semantics first.
