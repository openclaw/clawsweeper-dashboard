## Summary

Keep open. Current main still has the internal data model and public read gates for soft-deleted skill versions, but it does not expose owner-facing CLI, HTTP API, docs, or UI controls to hide/delete/restore one published version while leaving the skill live; the live discussion also treats this as the master policy tracker.

## What This Changes

Request owner-facing controls to hide, delete, restore, or delist a single published skill version while leaving the skill itself live.

## Best Possible Solution

Define the version-scoped lifecycle policy, then add authenticated owner/org-member/staff API, CLI, docs/tests, and optional owner UI that update `skillVersions.softDeletedAt` with version-specific audit and consistent public routing.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the current gap because version soft-delete state exists, public readers honor it, and the exposed owner CLI/API/docs remain skill-scoped.

## Solution Assessment

Unclear until maintainers define policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must cover latest, tags, downloads, cache behavior, restore rules, permissions, audit logs, docs, and tests.
