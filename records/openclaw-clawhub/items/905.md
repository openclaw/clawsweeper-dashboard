## Label Justifications

- `P2`: This is a normal priority feature gap with a clear publisher need but policy-sensitive lifecycle semantics.
- `impact:security`: Per-version withdrawal can be used to respond to accidentally published sensitive artifacts and must preserve audit and access boundaries.

## Summary

Keep this issue open. Current main can internally represent and publicly suppress a soft-deleted skill version, but the owner-facing CLI, HTTP API, docs, and management workflow still operate at whole-skill scope, and ClawHub issue policy keeps open issues for maintainer handling.

## What This Changes

Add an owner-facing way to hide, delete, unhide, or restore one published skill version without deleting the whole skill slug.

## Best Possible Solution

Define the version-scoped lifecycle policy first, then add authenticated owner/org-member/staff controls that patch skillVersions.softDeletedAt with version-specific audit logs and consistent public routing, download, cache, restore, docs, and tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because per-version soft-delete state and public gates exist while owner lifecycle controls remain skill-scoped.

## Solution Assessment

Unclear until maintainers settle policy. A version-scoped API plus CLI is directionally appropriate, but the maintainable solution must define latest, tag, download, cache, restore, permission, and audit semantics first.
