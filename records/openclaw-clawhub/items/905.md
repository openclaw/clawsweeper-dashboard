## Summary

Keep open. Current main still has internal version soft-delete state and public access gating, but no owner-facing CLI, HTTP API, or UI workflow to hide/delete/restore one published skill version while leaving the skill live; a collaborator also made this the master issue for that policy work.

## What This Changes

The issue requests owner-facing controls to hide, delete, restore, or mark one published skill version while keeping the skill live.

## Best Possible Solution

Define a version-scoped lifecycle policy, then expose consistent authenticated owner/org-member/staff controls across API, CLI, docs/tests, and possibly UI.

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the gap because current main has internal version soft-delete state and public filters, but exposed owner controls remain skill-scoped.

## Solution Assessment

Unclear until maintainers define policy. A version-scoped owner API plus CLI is directionally appropriate, but the durable solution must cover latest, tags, downloads, caches, restore rules, permissions, audit logs, docs, and tests.
