## Summary

Keep this open: current main still lacks owner-facing per-version hide/delete/restore controls, and a collaborator has explicitly designated this as the canonical policy thread for that remaining lifecycle work.

## What This Changes

The issue requests owner-facing controls to hide, delete, restore, or delist individual published skill versions while keeping the skill listing live.

## Best Possible Solution

Define an audited reversible per-version withdrawal lifecycle, then expose owner/org-admin/staff API, CLI, docs, and optional UI controls that keep version lists, direct access, downloads, tags, latest pointers, caches, and audit records consistent.

## Reproduction Assessment

Yes, at source level. Current main has internal version soft-delete state and public gates for that state, but no owner-facing CLI, HTTP API, or UI workflow to manage one published skill version.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. Raw deletion alone is not the best path; a reversible withdrawal workflow with explicit latest, tag, download, cache, restore, permission, and audit semantics is safer.
