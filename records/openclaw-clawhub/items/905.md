## Summary

Keep open: current main can represent and suppress a soft-deleted skill version, but the exposed owner CLI, HTTP API, docs, and version UI still manage whole skills rather than one published version, and this needs product/security policy before implementation.

## What This Changes

The issue requests owner-facing controls to hide, delete, restore, or otherwise withdraw one published skill version while keeping the skill live.

## Best Possible Solution

Define the version-scoped lifecycle policy first, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt with version-specific audit logs and consistent public routing, tags, downloads, caches, docs, and tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because version soft-delete state and public gates exist while owner-facing per-version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers settle policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must define latest, tag, download, cache, restore, permission, and audit semantics first.
