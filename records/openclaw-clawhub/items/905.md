## Label Justifications

- `impact:security`: Single-version withdrawal is used for leaked or unsafe artifact cleanup and affects authorization, public access, and audit evidence.

## Summary

Keep open. Current main can model and block soft-deleted skill versions, but the owner-facing CLI, HTTP API, docs, and management workflow still operate at whole-skill scope; a contributor also explicitly marked this as the master issue for the rolled-up per-version lifecycle requests.

## What This Changes

The issue requests owner-facing controls to hide, delete, restore, or deprecate a single published skill version while leaving the skill listing live.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt, write version-specific audit logs, and keep tags, latest pointers, public endpoints, downloads, docs, and restore semantics consistent.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because version soft-delete state and public gates exist while owner-facing per-version lifecycle controls are absent.

## Solution Assessment

Not applicable to a proposed patch. The request direction is valid, but the maintainable solution needs a policy-led API/CLI/UI design rather than an ad hoc version-row patch.
