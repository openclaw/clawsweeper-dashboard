## Summary

Current main has internal version soft-deletion and scanner quarantine behavior, but it still lacks the owner-facing per-version lifecycle workflow that a collaborator designated this issue to track; the remaining pointer, access, restoration, retention, and authority semantics require an explicit product and security decision.

## What This Changes

Add owner-facing controls to hide, withdraw, restore, or delete one published skill version while keeping the skill listing live.

## Best Possible Solution

Define an audited soft-withdrawal contract in `specs/`, then implement consistent authorization and public-read blocking across API, CLI, and owner UI, with deterministic latest/tag repair, moderator-only evidence access, cache behavior, and controlled restoration; reserve hard deletion for separately approved administrative cases.

## Reproduction Assessment

Yes at the source level: current main documents only whole-skill owner lifecycle commands, while version soft-deletion exists only in internal moderation/quarantine paths. This confirms the missing owner-facing capability without treating it as a runtime bug.

## Solution Assessment

No, the example commands and endpoints are not sufficient by themselves because they leave direct access, latest/tag pointers, retention, cache invalidation, restoration, and authority undefined. A policy-first soft-withdrawal model is the narrowest safe solution.

- Hard deletion can erase evidence needed for moderation, abuse investigation, incident response, audit history, and controlled restoration.
- Implementing only CLI, API, or UI would create inconsistent publisher expectations and uneven enforcement across lifecycle surfaces.
