## Summary

Keep open. Current main already stores and honors per-version soft-delete state in public readers, but owner-facing CLI, HTTP API, docs, and UI controls remain skill-scoped, and a recent maintainer comment explicitly makes this the master tracker for related per-version lifecycle requests.

## What This Changes

The issue requests owner-facing controls to hide, delete, restore, or delist individual published skill versions while keeping the skill live.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls across API, CLI, docs, tests, and any owner UI so version hiding is consistent with public reads and moderation audit requirements.

## Reproduction Assessment

Not applicable as a bug reproduction, but source inspection gives a high-confidence way to verify the feature gap: current main has version soft-delete state and public filtering, while exposed owner CLI/API/docs are only skill-scoped.

## Solution Assessment

Unclear until maintainers define policy. A maintainable fix should not be only a CLI wrapper; it needs coherent API authorization, latest/tag behavior, audit logs, restore semantics, docs, and regression coverage.
