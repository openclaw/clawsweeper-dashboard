## Summary

Keep open. Current main can internally soft-delete individual skill versions and public readers honor that state, but the requested owner-facing CLI/API/UI workflow for hiding, deleting, delisting, or restoring one version while keeping the skill live is still missing; a collaborator also confirmed this as the master policy issue.

## What This Changes

The issue requests owner-facing controls to hide, delete, delist, and restore a single published skill version without deleting the whole skill listing.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt with version-specific audit logs and consistent public list, detail, file, download, tag, latest, restore, docs, and test behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection give high confidence in the gap because internal version tombstones and public filters exist while owner-facing per-version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers define the lifecycle policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must cover latest, tags, downloads, caches, restore rules, permissions, audit logs, docs, and tests.
