## Summary

Keep open. Current main still lacks an owner-facing per-version lifecycle workflow: storage and public readers already understand hidden skill versions, but the exposed owner CLI, HTTP API, docs, and management surfaces remain skill-scoped, and ClawHub review policy keeps issues open for maintainer handling.

## What This Changes

Request an owner-facing CLI/API/UI workflow to hide, delete, restore, or delist one published skill version while keeping the skill live.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt with version-specific audit logs and consistent list, detail, file, download, tag/latest, cache, docs, and restore behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because current main models hidden versions and hides them from readers, but exposes only skill-scoped owner lifecycle controls.

## Solution Assessment

Unclear until maintainers define lifecycle policy. A version-scoped owner API plus CLI is directionally right, but the maintainable fix needs explicit latest-version, tag, cache, download, permission, restore, and audit semantics first.
