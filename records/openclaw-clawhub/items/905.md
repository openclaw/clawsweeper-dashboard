## Summary

Current main has internal per-version soft-deletion and public-read enforcement, but it still exposes only whole-skill owner lifecycle commands; the collaborator discussion also intentionally keeps this as the master issue for the unresolved policy-complete owner workflow.

## What This Changes

Add owner-facing controls to withdraw, hide, delete, and restore one published skill version without taking the whole skill offline.

## Best Possible Solution

Implement an audited owner withdrawal/restore state that atomically selects a valid latest version, repairs tags, blocks all public artifact paths and caches, and never lets owner restoration override moderator or security restrictions.

## Reproduction Assessment

Not applicable: this requests a new owner-facing lifecycle capability rather than reporting broken documented behavior. Source inspection establishes that the internal version state exists while the CLI exposes only whole-skill controls.

## Solution Assessment

No, the sample commands or endpoints alone are not a complete solution. Maintainers should first define one consistent withdrawal policy spanning latest/tag selection, every public read path, cache invalidation, restore authority, and audit retention.

## Risks / Open Questions

- A partial implementation could remove a version from history while leaving it reachable through a latest pointer, tag, direct version/file/download endpoint, search digest, or cache.
- Owner restoration could improperly revive a version hidden by moderation, scanning, a publisher ban, or a security incident unless authority and hide reasons are kept distinct.
- Hard deletion or incomplete pointer repair could lose audit evidence or leave a live skill with invalid tags and no valid latest version.
