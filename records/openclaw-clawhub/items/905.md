## Summary

Keep open. Current main still lacks an owner-facing per-version hide/delete/restore workflow: version soft-delete exists internally and public reads honor it, but the exposed CLI, HTTP API, docs, and management UI remain skill-scoped, and ClawHub issue cleanup policy keeps issues open for maintainer handling.

## What This Changes

The issue requests owner-facing controls to hide, delete, unhide, or undelete one published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then expose authenticated owner/org-member/staff controls that update version state, preserve audit records, and keep public list/detail/file/download/tag behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives high confidence in the feature gap: version soft-delete state exists, but owner-facing CLI/API/docs/UI workflows only manage whole skills.

## Solution Assessment

Unclear until maintainer policy is set. The requested direction is reasonable, but directly exposing the existing staff-only mutation would be too thin without latest-version, tag, cache, restore, download, and audit semantics.
