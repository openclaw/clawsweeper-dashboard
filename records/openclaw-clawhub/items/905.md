## Summary

Keep open. Current main can represent and publicly suppress soft-deleted skill versions, but still lacks owner-facing CLI/API/UI controls for withdrawing one skill version; this remains the canonical policy thread rather than an auto-close candidate.

## What This Changes

The issue requests owner-facing controls to hide, delete, unhide, or restore an individual published skill version while keeping the skill itself live.

## Best Possible Solution

Define the version-scoped lifecycle policy, then implement authenticated owner/org-member/staff controls that update `skillVersions.softDeletedAt`, write version-specific audit logs, and keep lists, detail/file/download endpoints, tags/latest, docs, and tests consistent.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because current main has version soft-delete state and public gates, but no owner-facing per-version lifecycle route or CLI command.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. The requested owner-facing per-version workflow is directionally right, but the maintainable solution needs explicit semantics for latest versions, tags, downloads, caches, restore permissions, and audit evidence.
