## Summary

Keep open. Current main can represent and publicly suppress soft-deleted skill versions, but still lacks owner-facing CLI/API/UI controls for withdrawing or restoring one published version, and this feature needs product and security policy before implementation.

## What This Changes

Owner-facing request to hide, delete, unhide, or undelete one published skill version while keeping the skill itself live.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update `skillVersions.softDeletedAt`, write version-specific audit records, and keep lists, detail/file/download endpoints, tags/latest, cache behavior, docs, and tests consistent.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the current gap because version soft-delete state and public gates exist, but owner-facing per-version API, CLI, docs, and UI controls are absent.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. The requested owner-facing version workflow is directionally right, but the maintainable solution needs explicit semantics for latest versions, tags, downloads, caches, restore permissions, and audit evidence.
