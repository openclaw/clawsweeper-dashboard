## Summary

Keep open. Current main still lacks an owner-facing per-version hide/delete/restore workflow: version soft-delete exists internally and public readers honor it, but exposed CLI, HTTP API, docs, and management UI controls remain skill-scoped; ClawHub cleanup policy also keeps issues open for maintainer handling.

## What This Changes

The issue requests owner-facing CLI/API/UI controls to hide, delete, restore, or delist one published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that resolve slug plus version, update skillVersions.softDeletedAt, write version-specific audit logs, and keep public lists, detail/file/download routes, tags/latest behavior, caches, docs, and restore semantics consistent.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives high confidence in the feature gap: version soft-delete state exists, but owner-facing CLI/API/docs/UI workflows still manage whole skills rather than individual versions.

## Solution Assessment

Unclear until maintainer policy is set. The requested direction is reasonable, but directly exposing the existing staff-only mutation would be too thin without latest-version, tag, cache, restore, download, and audit semantics.
