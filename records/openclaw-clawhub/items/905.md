## Summary

Keep open. Current main still lacks owner-facing per-version lifecycle controls: `skillVersions.softDeletedAt` exists and public readers honor it, but the exposed CLI, HTTP API, docs, and management flow remain skill-scoped, and the live discussion treats this as the master policy tracker.

## What This Changes

The issue requests owner-facing controls to hide, delete, restore, or delist individual published skill versions while keeping the skill live.

## Best Possible Solution

Define the version-scoped lifecycle policy, then add authenticated owner/org-member/staff API, CLI, docs/tests, and optional owner UI that update `skillVersions.softDeletedAt` with version-specific audit and consistent public routing.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the gap because version soft-delete state and public filters exist while owner-facing version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers define policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must cover latest, tags, downloads, cache behavior, restore rules, permissions, audit logs, docs, and tests.
