## Summary

Keep open. Current main still has internal version soft-delete state and public gates for hidden versions, but no owner-facing per-version lifecycle write path; a maintainer also explicitly named this as the master issue for that policy and implementation work.

## What This Changes

The issue asks for owner-facing controls to hide, delete, unhide, or undelete one published skill version while keeping the skill live.

## Best Possible Solution

Define the version-scoped lifecycle policy first, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt, write version-specific audit logs, and keep tags, latest pointers, public endpoints, downloads, caches, docs, and tests consistent.

## Reproduction Assessment

Not applicable as a runtime bug, but yes for the reported feature gap by source inspection. Current main has version soft-delete state and public gates, while CLI, HTTP API, docs, and tests expose only skill-level lifecycle write paths.

## Solution Assessment

Unclear until maintainers settle the lifecycle policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must define latest, tags, downloads, caches, restore, permission, and audit semantics first.
