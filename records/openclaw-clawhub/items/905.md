## Summary

Keep this open. Current main still has the building blocks for per-version withdrawal, but the exposed owner CLI, HTTP API, docs, and management flows remain whole-skill scoped, and a collaborator explicitly marked this as the master issue for the unresolved lifecycle policy.

## What This Changes

Request an owner-facing way to hide, delete, restore, or delist one published skill version while keeping the skill listing live.

## Best Possible Solution

Define an audited, reversible per-version withdrawal lifecycle first, then implement owner/org/staff API, CLI, UI, docs, and tests that keep latest pointers, tags, direct reads, downloads, caches, restore, and audit semantics consistent.

## Reproduction Assessment

Yes, source-level. Current main exposes whole-skill delete/hide/undelete paths and internal version soft-delete gates, but no owner-facing per-version CLI, API, UI, or docs workflow.

## Solution Assessment

No, raw owner deletion alone is not the best solution. The safer path is a reversible withdrawal state with explicit authority, latest/tag behavior, direct-access blocking, cache handling, restoration rules, and audit retention.
