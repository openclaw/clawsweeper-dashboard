## Summary

Keep open. Current main can represent and internally staff-hide individual skill versions, and public reads/downloads honor `softDeletedAt`, but owner-facing CLI, HTTP API, docs, and UI controls still operate at whole-skill level; ClawHub cleanup policy also keeps issues open for maintainer handling.

## What This Changes

The issue requests owner-facing controls to hide, delete, unhide, or undelete one published skill version without taking the whole skill offline.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update `skillVersions.softDeletedAt`, write version-specific audit logs, and keep public lists, detail/file/download endpoints, tags/latest behavior, cache invalidation, docs, and restore semantics consistent.

## Reproduction Assessment

Not applicable as a runtime bug, but current-main verification is high-confidence: the CLI only accepts skill slugs, the DELETE router rejects version paths, and docs list only read-only version endpoints plus whole-skill delete/undelete.

## Solution Assessment

Unclear until maintainer policy is set. The requested owner-facing workflow is the right product direction, but the maintainable fix should be gated by explicit latest-version, audit, restore, and cache/download rules rather than exposing a thin wrapper around the internal staff mutation.
