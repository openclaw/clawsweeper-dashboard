## Label Justifications

- `P2`: This is a valid publisher lifecycle improvement with limited blast radius but real policy and security implications.
- `impact:security`: The feature would let owners withdraw accidentally exposed or unsafe historical version content and must preserve moderation evidence.
- `impact:data-loss`: Version delete, hide, and restore controls affect published artifact availability and must avoid accidental irreversible loss.

## Summary

Keep open. Current main has per-version soft-delete state and public readers honor it, but owner-facing CLI, HTTP API, docs, and UI controls are still skill-scoped; a recent maintainer comment also explicitly made this the master tracker for per-version lifecycle policy.

## What This Changes

Add owner-facing controls to hide, delete, restore, or delist one published skill version while leaving the skill itself live.

## Best Possible Solution

Define a version-scoped lifecycle policy, then implement authenticated owner/org-member/staff API, CLI, docs/tests, and optional owner UI that updates skillVersions.softDeletedAt with version-specific audit logs and consistent public routing.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the gap because version soft-delete state and public filters exist while owner-facing version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers define policy. A version-scoped owner API plus CLI is directionally appropriate, but the durable fix needs coherent authorization, latest/tag behavior, audit logs, restore semantics, docs, and regression coverage.
