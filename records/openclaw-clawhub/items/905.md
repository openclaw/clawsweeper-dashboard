## Summary

Keep open. Current main still lacks an owner-facing per-version hide/delete/restore workflow for skills; the backend models and hides soft-deleted versions, but exposed CLI, HTTP API, docs, and management controls remain skill-scoped, and ClawHub policy keeps issues open for maintainer handling.

## What This Changes

Owners want a way to hide, delete, restore, or delist one published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt with audit logs and consistent public list, detail, file, download, tag/latest, cache, docs, and restore behavior.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives high confidence in the feature gap: version soft-delete state exists and public readers honor it, but owner-facing CLI/API/docs/UI workflows still manage whole skills rather than individual versions.

## Solution Assessment

Unclear until maintainer policy is set. The requested direction is reasonable, but a maintainable fix needs explicit latest-version, tag, cache, restore, download, and audit semantics rather than exposing a thin per-version toggle.
