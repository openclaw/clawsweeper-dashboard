## Summary

Keep open. Current main still has only skill-scoped owner lifecycle controls, while a maintainer comment explicitly made this the master tracker for per-version hide, delete, deprecate, and delist policy.

## What This Changes

The request is to add owner-facing controls to hide, delete, restore, or delist one published skill version while keeping the skill itself live.

## Best Possible Solution

Define the version-scoped lifecycle policy first, then add authenticated owner/org-member/staff API, CLI, docs/tests, and optional owner UI that updates skillVersions.softDeletedAt with version-specific audit and consistent public routing.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the gap because version soft-delete state and public filters exist while owner-facing version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers define policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must cover latest, tags, downloads, cache behavior, restore rules, permissions, audit logs, docs, and tests.
