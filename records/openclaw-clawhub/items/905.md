## Summary

Current main still lacks an owner-facing lifecycle for withdrawing one published skill version while keeping the skill live. The existing internal soft-delete field and hardened public read paths provide building blocks, but a repository collaborator explicitly designated this as the canonical issue because latest pointers, tags, direct downloads, restore/audit behavior, and owner-versus-moderator authority remain unresolved.

## What This Changes

Add owner-facing controls to hide, withdraw, restore, or otherwise manage one published skill version without deleting the whole skill.

## Best Possible Solution

Define and implement an audited, reversible per-version withdrawal state that consistently blocks public discovery and downloads, preserves moderator evidence, specifies latest/tag behavior, and reserves irreversible deletion for privileged incident handling.

## Reproduction Assessment

Yes, at source level. Current main exposes whole-skill owner delete/undelete and stores internal version soft-deletion state, but has no complete owner-facing single-version lifecycle across CLI, API, UI, and public read paths.

## Solution Assessment

No, raw owner deletion alone is not the best solution. A reversible withdrawal workflow with explicit authority, pointer, download, cache, retention, and audit semantics is narrower and safer.

## Risks / Open Questions

- A superficial history-only hide could leave withdrawn content reachable through direct version, file, archive, or download endpoints and caches.
- Automatically moving latest-version or tag pointers could silently retarget installs, while leaving them unchanged could make the skill unusable or expose a withdrawn release.
- Owner restoration must not override scanner, moderator, legal, malware, or security-redaction decisions.
- Hard deletion could destroy audit and incident-response evidence and requires separate retention and cache-purge policy.
