## Summary

Keep open. Current main can represent and hide soft-deleted skill versions internally, and public readers/downloads respect that state, but it still does not expose the requested owner-facing CLI, HTTP API, or UI workflow for hiding, deleting, or restoring one published version while keeping the skill live; a collaborator also marked this as the master issue for unresolved per-version lifecycle policy.

## What This Changes

The issue requests owner-facing controls to hide, delete, restore, or supersede one published skill version without deleting the whole skill listing.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt with audit logs and consistent public route, latest/tag, cache, restore, docs, and test behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the gap because internal version tombstones and public filters exist while owner-facing per-version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers define policy. A version-scoped owner API plus CLI is directionally appropriate, but the durable solution must cover latest, tags, downloads, caches, restore rules, permissions, audit logs, docs, and tests.
