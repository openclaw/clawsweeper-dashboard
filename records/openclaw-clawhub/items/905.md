## Summary

Keep this open. Current main still lacks an owner-facing CLI, HTTP API, or UI workflow to hide/delete/restore one published skill version while leaving the skill live; the existing code only provides internal version withdrawal state plus public read gates, so there is no implemented fix on main or in the latest release to close against.

## What This Changes

Add owner-facing controls to hide, delete, restore, or undelete a single published skill version while keeping the skill live.

## Best Possible Solution

Define a reversible, audited per-version withdrawal model and expose consistent owner/org/staff API, CLI, UI, docs, and tests across version lists, direct detail/file/download access, latest/tags, search, cache, and restore behavior.

## Reproduction Assessment

Yes, at source level. Current main exposes only skill-level owner delete/undelete in the CLI and HTTP API while version read paths only honor existing softDeletedAt state; no owner-facing single-version mutator is registered.

## Solution Assessment

No, a raw owner delete command by itself is not the best solution. The safer path is a policy-backed, reversible version withdrawal workflow with explicit latest/tag, direct-access, cache, restore, and audit semantics.
