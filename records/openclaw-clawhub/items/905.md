## Summary

Keep open. Current main can model and block soft-deleted skill versions, but it still lacks an owner-facing CLI, HTTP API, or UI workflow to hide/delete/restore one published version while keeping the skill live; a contributor has also marked this as the master issue for that policy surface.

## What This Changes

Add owner-facing controls to hide, delete, unhide, or undelete an individual published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt with version-specific audit logs and keep tags, latest behavior, public endpoints, downloads, docs, and restore semantics consistent.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because version soft-delete state and public gates exist while owner-facing per-version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must first define latest, tag, download, cache, restore, permission, and audit semantics.
