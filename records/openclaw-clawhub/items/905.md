## Summary

Keep open. Current main can represent and enforce per-version soft deletion internally, but the requested owner-facing CLI/API/UI workflow for hiding, deleting, or restoring one published version while the skill stays live is still missing; a collaborator also marked this as the master issue for that lifecycle policy.

## What This Changes

The issue asks for an owner-facing way to hide, soft-delete, restore, or otherwise withdraw one published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org/staff controls that update skillVersions.softDeletedAt with audit logs and consistent public route, latest/tag, restore, docs, and test behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection give high confidence in the gap because internal version tombstones and public filters exist while owner-facing per-version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers define policy. A version-scoped owner API plus CLI is directionally appropriate, but the durable solution must cover latest, tags, downloads, caches, restore rules, permissions, audit logs, docs, and tests.
