## Summary

Keep open. Current main can soft-delete skill versions internally and public readers honor that state, but it still lacks the requested owner-facing CLI, HTTP API, or UI workflow; a collaborator also routed this as the master issue for unresolved per-version lifecycle policy.

## What This Changes

The issue requests owner-facing controls to hide, delete, restore, or delist one published skill version while keeping the skill live.

## Best Possible Solution

Define the version-scoped lifecycle policy, then add authenticated owner/org/staff controls that update skillVersions.softDeletedAt with audit logs and consistent public version, file, download, tag, latest, cache, restore, docs, and test behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that the gap remains because internal version tombstones and public filters exist while owner-facing per-version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers define policy. A version-scoped owner API plus CLI/UI is directionally appropriate, but the durable solution must cover latest, tags, downloads, caches, restore rules, permissions, audit logs, docs, and tests.
