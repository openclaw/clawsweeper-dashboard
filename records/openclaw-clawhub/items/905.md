## Label Justifications

- `P2`: The missing lifecycle control affects publisher cleanup and incident response but does not make ClawHub's core runtime unusable.
- `impact:security`: Publishers may need to rapidly stop public access to a version containing accidentally disclosed sensitive material.
- `impact:other`: Publishers currently cannot withdraw one broken or inappropriate version while keeping the skill available.

## Summary

Current main has internal moderation logic for quarantining malicious versions and recently added whole-skill hide/unhide controls, but it still has no owner-facing operation for withdrawing one published version. A repository collaborator also designated this as the canonical issue, so the requested product work remains necessary.

## What This Changes

Add owner-facing controls to hide, withdraw, restore, or delete one published skill version while keeping the skill listing live.

## Best Possible Solution

Implement an audited per-version withdraw/restore lifecycle across API, CLI, and owner UI, defaulting to non-destructive withdrawal and consistently blocking withdrawn versions from public listing, resolution, files, and downloads.

## Reproduction Assessment

Yes, at the source-contract level. Current main exposes whole-skill lifecycle operations and internal moderation quarantine, but no owner-facing operation targets one published version.

## Solution Assessment

Yes for an audited withdraw/restore capability, but not necessarily for unrestricted hard deletion. Non-destructive withdrawal is the safer default unless maintainers explicitly define irreversible deletion, pinned-version, cache, and retention behavior.
