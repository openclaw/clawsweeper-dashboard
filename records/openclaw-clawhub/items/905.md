## Summary

Keep open. Current main and the latest tagged release still lack owner-facing per-version lifecycle controls; internal version soft-delete state exists, public readers honor it, and the maintainer discussion explicitly treats this issue as the master policy tracker.

## What This Changes

The issue requests owner-facing controls to hide, delete, restore, or delist individual published skill versions while keeping the skill listing live.

## Best Possible Solution

Define the version-scoped lifecycle policy first, then add authenticated owner/org-member/staff API and CLI support that updates `skillVersions.softDeletedAt`, writes version-specific audit logs, and keeps public reads, tags/latest, downloads, docs, and restore semantics consistent.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the gap because version soft-delete state and public filters exist while owner-facing version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers define policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must cover latest, tags, downloads, cache behavior, restore rules, permissions, audit logs, docs, and tests.
