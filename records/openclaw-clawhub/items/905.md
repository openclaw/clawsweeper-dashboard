## Summary

Keep open. Current main can internally soft-delete individual skill versions and public reads honor that state, but the exposed owner CLI, HTTP API, docs, and management UI remain skill-scoped; this is a real feature gap that needs maintainer policy review.

## What This Changes

Request an owner-facing workflow to hide, delete, unhide, or undelete one published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt, record version-specific audit logs, and keep public lists, details, files, downloads, tags/latest, docs, and restore behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug. Source verification is high-confidence: the internal version soft-delete state exists, but current owner CLI/API/docs expose only whole-skill delete/undelete flows.

## Solution Assessment

Unclear until maintainer policy is set. The requested owner-facing workflow is the right direction, but exposing it safely requires explicit latest-version, tag, audit, restore, and cache/download rules first.
