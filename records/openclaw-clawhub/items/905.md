## Summary

Keep open. Current main still lacks an owner-facing per-version lifecycle workflow; it can represent and enforce soft-deleted skill versions internally, but exposed CLI/API/docs remain skill-scoped, and the ClawHub review policy keeps issues open for maintainer handling.

## What This Changes

The issue requests owner-facing controls to hide, delete, unhide, or restore an individual published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt with audit logs and consistent list/detail/file/download/tag/cache/restore behavior.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives a high-confidence feature gap: version soft-delete state exists, but the exposed owner CLI/API/docs only manage whole skills.

## Solution Assessment

Unclear until maintainer policy is set. The requested direction is reasonable, but exposing the existing staff-only mutation directly would be too thin without latest-version, tag, cache, restore, and audit semantics.
