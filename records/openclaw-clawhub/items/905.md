## Summary

Keep this issue open. Current main can represent and hide soft-deleted skill versions internally, but the owner-facing CLI, HTTP API, docs, and UI controls are still skill-scoped; the ClawHub cleanup policy also keeps issues open for maintainer handling.

## What This Changes

Request owner-facing controls to hide, delete, unhide, or restore one published skill version while keeping the skill slug live.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt, write version-specific audit logs, and keep public lists, detail/file/download endpoints, tags/latest, cache invalidation, docs, and restore behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature gap. Code inspection gives a high-confidence verification path: current main exposes whole-skill delete/undelete publicly, while per-version deletion is internal and staff-only.

## Solution Assessment

Unclear until maintainers decide the lifecycle policy. A direct owner command/API is likely the right product direction, but it should be designed with latest-version, tags, audit, and security-incident behavior before implementation.
