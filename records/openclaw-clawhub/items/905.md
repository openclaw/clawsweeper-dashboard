## Summary

Keep open. Current main can represent and hide soft-deleted skill versions from public readers/downloads, but the owner-facing CLI, HTTP API, UI, and docs still expose only whole-skill lifecycle controls; ClawHub review policy also keeps issues open for maintainer handling.

## What This Changes

Request an owner-facing workflow to hide, delete, restore, or unhide one published skill version while keeping the skill slug live.

## Best Possible Solution

Define a version-scoped lifecycle policy, then expose authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt with version-specific audit logs and consistent tag, latest, download, cache, docs, and restore behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The high-confidence verification path is code/docs inspection: version soft-delete exists and public readers honor it, while owner-facing CLI/API/UI/docs are still skill-scoped.

## Solution Assessment

Unclear until maintainer policy is settled. The requested direction is appropriate, but the safest first implementation is likely hide/restore for non-latest versions with preserved audit history before adding stronger delete semantics.
