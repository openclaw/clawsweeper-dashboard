## Summary

Keep open. Current main can model and publicly suppress soft-deleted skill versions, but it still lacks owner-facing CLI, HTTP API, docs, or UI controls to hide/delete/restore one published skill version while keeping the skill live; related reports show this is still a real product and security-policy gap.

## What This Changes

The issue requests owner-facing CLI or API controls to hide, delete, restore, or otherwise delist one published skill version without deleting the whole skill.

## Best Possible Solution

Use this as the canonical policy thread, then add authenticated owner/org-member/staff version-scoped controls that update skillVersions.softDeletedAt, write version-specific audit logs, and keep public lists, detail/file/download routes, tags/latest behavior, docs, tests, and cache behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because version soft-delete state and public gates exist while owner-facing per-version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must define latest, tag, download, cache, restore, permission, and audit semantics first.
