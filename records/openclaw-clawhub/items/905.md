## Summary

Keep open. Current main still lacks an owner-facing version-scoped hide/delete/restore workflow for skill versions; the backend can represent hidden versions and public readers honor that state, but exposed CLI, HTTP API, docs, and management controls remain skill-scoped, and ClawHub policy keeps issues open for maintainer handling.

## What This Changes

Add an owner-facing way to hide, delete, restore, or delist a single published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then implement authenticated owner/org-member/staff controls that update skillVersions.softDeletedAt with version-specific audit logs and consistent list, detail, file, download, tag/latest, cache, docs, and restore behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap: version soft-delete state exists and public readers honor it, but owner-facing CLI/API/docs/UI workflows still manage whole skills rather than individual versions.

## Solution Assessment

Unclear until maintainers set lifecycle policy. A version-scoped owner API plus CLI matches the request directionally, but the maintainable fix needs explicit latest-version, tag, cache, restore, download, permission, and audit semantics first.
