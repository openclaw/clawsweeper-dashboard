## Summary

Keep open. Current main still has internal per-version soft-delete state and public gates for hidden versions, but it does not expose an owner-facing workflow to hide/delete/restore one published skill version; related reports point back to this issue as the canonical lifecycle-policy thread.

## What This Changes

Request an owner-facing CLI/API workflow to hide, delete, unhide, or undelete individual published skill versions without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update `skillVersions.softDeletedAt`, write version-specific audit logs, and keep public list/detail/file/scan/download/tag/latest behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because current main models and gates soft-deleted versions but exposes only skill-scoped owner lifecycle controls.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped owner API plus CLI is directionally right, but latest-version, tag, cache, download, restore, permission, and audit semantics should be decided first.
