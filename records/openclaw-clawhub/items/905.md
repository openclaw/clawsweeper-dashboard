## Summary

Keep open. Current main still has the requested per-version lifecycle gap: the backend can represent soft-deleted skill versions and public reads honor that state, but the owner-facing CLI, HTTP API, docs, and management UI remain skill-scoped, and a maintainer comment now treats this as the master policy tracker.

## What This Changes

Add owner-facing controls to hide, delete, delist, deprecate, restore, or otherwise manage one published skill version without deleting the whole skill.

## Best Possible Solution

Define the version-scoped lifecycle policy, then implement authenticated owner/org-member/staff API, CLI, docs/tests, and optional owner UI that update skillVersions.softDeletedAt with version-specific audit and consistent public routing.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the gap because version soft-delete state and public filters exist while owner-facing version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers define policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must cover latest, tags, downloads, cache behavior, restore rules, permissions, audit logs, docs, and tests.
