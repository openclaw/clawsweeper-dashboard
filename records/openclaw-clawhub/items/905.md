## Summary

Keep open. Current main still has only internal/staff/scanner-oriented version soft-delete behavior and public read gating; it does not expose the requested owner-facing CLI, HTTP API, or UI workflow for hiding, deleting, or restoring one published skill version while leaving the skill live.

## What This Changes

The issue asks for owner-facing controls to hide, delete, unhide, or restore a single published skill version without deleting the whole skill.

## Best Possible Solution

Define the version-scoped lifecycle policy, then add authenticated owner/org-member/staff API, CLI, docs/tests, and optional owner UI that update skillVersions.softDeletedAt with audit and consistent public-route behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the gap because internal version soft-delete state and public filters exist while owner-facing version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers define policy. A version-scoped owner API plus CLI is directionally appropriate, but the durable solution must cover latest, tags, downloads, caches, restore rules, permissions, audit logs, docs, and tests.
