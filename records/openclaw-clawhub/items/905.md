## Summary

Keep open. Current main still lacks an owner-facing per-version hide/delete/restore workflow: version rows can be soft-deleted internally and public reads honor that state, but the exposed CLI, HTTP API, docs, and owner UI remain skill-scoped, and ClawHub issue cleanup policy keeps issues open for maintainer handling.

## What This Changes

Request owner-facing CLI/API/UI support to hide, delete, unhide, or restore an individual published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update `skillVersions.softDeletedAt`, write version-specific audit logs, and keep public lists, detail/file/download endpoints, tags/latest behavior, cache invalidation, docs, and restore semantics consistent.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives a high-confidence verification path: the backend can hide individual versions internally, but the owner-facing CLI/API/docs/UI surfaces do not expose that workflow.

## Solution Assessment

Unclear until maintainer policy is set. The requested owner-facing workflow is the right product direction, but the maintainable fix should define latest-version, tag, audit, restore, cache, and download rules before exposing a thin wrapper around the internal staff mutation.
