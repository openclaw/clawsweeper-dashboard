## Summary

Keep open. Current main now covers local Plugin Inspector validation with `clawhub package validate`, but the requested supported pre-publish plugin scanner still includes hosted ClawScan/static/LLM security parity, credential/code findings, and stable severity semantics that are not exposed as a public local plugin CLI.

## What This Changes

The issue requests a supported local plugin scanner CLI that plugin authors can run before publishing to catch manifest, skill metadata, and security scan findings.

## Best Possible Solution

Keep this as the canonical tracker until maintainers define a supported pre-publish plugin scanner contract with stable human/JSON output, redaction, severity, and hosted-scan parity rules.

## Reproduction Assessment

Yes, at source level: current main supports `clawhub package validate` for Plugin Inspector findings and stored scan downloads, while local plugin ClawScan-style pre-publish scanning is absent from the supported CLI. This is a missing capability, not a runtime crash reproduction.

## Solution Assessment

Unclear. Extending `clawhub package validate` may be the right user surface, but maintainers first need to decide scanner parity, redaction, severity semantics, and whether LLM/static checks belong in the public CLI.
