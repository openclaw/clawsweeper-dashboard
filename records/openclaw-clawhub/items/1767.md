## What This Changes

The issue requests a local plugin scanner CLI or package that plugin authors can run before publishing to validate manifests, SKILL.md metadata, code-scan findings, JSON output, severity levels, and exit codes.

## Best Possible Solution

Define a maintainer-approved local scanner contract that reuses shared deterministic validation where possible, documents hosted-scan parity and divergence, and exposes a stable public CLI/package surface with human and JSON output.

## Security Review

Status: needs_attention

Summary: A public local scanner is security-sensitive because findings, severities, privacy expectations, and exit codes must not conflict with hosted ClawScan and moderation policy.

Concerns:

- **[medium] Define scanner parity before productizing local ClawScan:** `convex/lib/staticPublishScan.ts:22`
  - body: Hosted package scanning reads uploaded files from Convex storage and runs static moderation during publish/backfill, while the repo-local utility also calls an LLM with a local OpenAI key; a public CLI must define authoritative results, severity mapping, privacy expectations, and exit-code behavior.
  - confidence: 0.9
- **[medium] Decide the local LLM scan trust boundary:** `scripts/local-clawscan-dry-run.ts:107`
  - body: The repo-local dry-run script requires an OpenAI API key and evaluates artifacts through an LLM, so exposing it as supported publisher tooling needs explicit privacy, cost, authority, and output-stability decisions.
  - confidence: 0.86
- **[low] Reconcile requested code scanning with current package scan scope:** `specs/security-moderation.md:144`
  - body: The security-moderation spec currently says package static/LLM scan inputs are metadata-only and do not deep-scan every tarball file, so broader local code scanning needs a deliberate scope update.
  - confidence: 0.82
