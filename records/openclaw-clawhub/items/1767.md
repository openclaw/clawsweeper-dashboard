## Summary

Keep open. Current main now covers the author-facing Plugin Inspector part through `clawhub package validate`, but the requested supported local pre-publish scanner still includes ClawScan/static/LLM security parity, credential/code findings, and stable severity semantics that remain split between hosted scan paths and an internal repo script.

## What This Changes

The issue requests a local plugin scanner CLI or pre-publish lint mode that catches manifest, SKILL.md, and code/security findings before publishing.

## Best Possible Solution

Keep this as the canonical tracker until maintainers define a supported pre-publish plugin scanner contract with stable human/JSON output, redaction, severity, and hosted-scan parity rules.

## Reproduction Assessment

Yes, at source level: current main supports `clawhub package validate` for Plugin Inspector findings, while local ClawScan-style static/LLM security scanning remains a repo script and `clawhub scan` is for stored submitted-version reports. This is a missing supported capability, not a runtime crash reproduction.

## Solution Assessment

Unclear. Extending `clawhub package validate` may be the right user surface, but maintainers first need to decide scanner parity, redaction, severity semantics, and whether LLM/static checks belong in the public CLI.
