## Summary

Keep open. Current main now ships useful author-facing Plugin Inspector validation through `clawhub package validate`, but the requested local pre-publish scanner still includes ClawHub static/LLM scan parity and code/credential findings; that remains split between hosted publish-time scanning and an internal `clawscan:local` script rather than one supported plugin-author CLI contract.

## What This Changes

Requested a supported local plugin scanner or CLI lint/dry-run path that catches ClawHub pre-publish scanner findings before publication.

## Best Possible Solution

Define one supported pre-publish plugin scanner contract, likely by extending `clawhub package validate` or clearly documenting a sibling command, so Plugin Inspector and local static/LLM scan results have stable human and JSON output, severity semantics, redaction behavior, and documented parity with hosted publish checks.

## Reproduction Assessment

Yes, at source level: current main supports `clawhub package validate` for Plugin Inspector findings, but `clawhub scan` still rejects local paths and the local ClawScan-style static/LLM path is only an internal script requiring `OPENAI_API_KEY`. This is a capability gap rather than a failing runtime bug.

## Solution Assessment

No, closing as solved would be premature. `clawhub package validate` is the right starting point, but the best final shape needs maintainer-approved scanner parity and severity behavior rather than a second overlapping local scanner surface.
