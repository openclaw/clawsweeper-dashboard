## Summary

Keep open: current main and v0.21.0 still reject local scan/upload paths and only let authors scan or download already-submitted artifacts, so the requested pre-publish plugin scanner is not implemented.

## What This Changes

Request a local CLI or pre-publish scanner for OpenClaw plugins that validates manifest, SKILL metadata, and code findings before publishing to ClawHub.

## Best Possible Solution

Add a maintainer-approved pre-publish plugin scan or lint path that reuses shared ClawHub validators and static moderation logic, returns human and JSON severities, and never mutates public registry state.

## Reproduction Assessment

Yes, at source level: current main rejects `clawhub scan <path>` and backend local upload scan requests with explicit unsupported messages. The docs also direct plugin authors to upload first and download stored scan results afterward.

## Solution Assessment

Yes in direction, but the maintainable solution should extend or share the canonical ClawHub scan and validation pipeline rather than create a separate plugin-only scanner that can diverge from hosted enforcement.

## Risks / Open Questions

- A standalone local linter could drift from ClawHub's hosted scanner rules, severities, and blocking policy.
- Accepting arbitrary unpublished plugin bundles for analysis needs explicit authentication, size and rate limits, retention rules, abuse controls, and a guarantee that uploaded code is analyzed but not executed.
- The request mixes deterministic schema validation with heuristic security classification, so maintainers need to define stable exit codes and severity semantics.
