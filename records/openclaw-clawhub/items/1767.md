## Summary

Current main does not provide the requested pre-publish workflow: the shipped `clawhub scan` command explicitly rejects local folders and requires uploading a version before retrieving results. The issue remains useful, but implementing it requires a maintainer decision about which scanner stages may safely run on unpublished plugin code and how local findings relate to authoritative server scans.

## What This Changes

Add local, CI-friendly pre-publish validation for plugin manifests, SKILL.md metadata, and security scanner findings.

## Best Possible Solution

Extend `clawhub scan <path>` with a non-publishing plugin preflight that reuses canonical schema and deterministic static-analysis logic, emits structured severity and exit status, never executes the plugin, and labels server-only checks as unavailable rather than silently claiming parity.

## Reproduction Assessment

Yes, at source level. On current main, `clawhub scan <local-path>` deterministically reaches `rejectLocalScan()` and tells the author to upload a version first.

## Solution Assessment

Unclear. The existing CLI is the appropriate surface, but maintainers must approve the supported local scanner subset, plugin-versus-skill scope, and local/server authority boundary before implementation.

## Risks / Open Questions

- A local scanner could become a second, drifting source of truth unless it shares canonical package parsing, validators, and deterministic security rules with the registry.
- Scanning unpublished plugin code must not execute plugin code or expose local credentials, and any network-backed or privileged scanner stages need an explicit trust boundary.
- Local output must clearly identify checks that were not run so authors do not mistake advisory preflight results for an authoritative clean server scan.
