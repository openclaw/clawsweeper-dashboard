## Summary

Keep open. Current main partially covers the request with `clawhub package validate` for Plugin Inspector findings, but the requested supported pre-publish scanner still includes local static/LLM security scan parity, credential/code findings, and stable severity semantics that are split today between hosted publish-time scanning and an internal `clawscan:local` script.

## What This Changes

The issue requests a supported local pre-publish plugin scanner/linter CLI that validates manifest/SKILL metadata and code/security findings before authors publish.

## Best Possible Solution

Define one supported pre-publish plugin scanner contract, likely by extending `clawhub package validate` or documenting a sibling command with stable human/JSON output, redaction, severity, and hosted-scan parity rules.

## Reproduction Assessment

Yes, at source level: current main supports `clawhub package validate` for Plugin Inspector findings, while `clawhub scan` rejects local paths and ClawScan-style static/LLM local scanning remains an internal script requiring `OPENAI_API_KEY`. This is a capability gap rather than a failing runtime bug.

## Solution Assessment

Unclear. A supported local pre-publish scanner is useful, but maintainers should choose whether to extend `clawhub package validate`, promote the internal ClawScan dry-run, or define a separate command before implementation.
