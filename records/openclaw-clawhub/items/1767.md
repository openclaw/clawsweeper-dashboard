## Summary

Keep open. Current main partially covers author-facing Plugin Inspector validation with `clawhub package validate`, but the central requested local pre-publish scanner remains missing because `clawhub scan` rejects local paths and the approved scan design explicitly excludes unpublished plugin packages.

## What This Changes

Requested behavior is a local pre-publish plugin scanner CLI that checks manifests, skill metadata, code/credential heuristics, severities, JSON output, and publish-blocking parity before authors submit to ClawHub.

## Best Possible Solution

Define one maintainer-approved pre-publish scanner contract for plugins, reusing shared Plugin Inspector and deterministic scan logic while documenting what is and is not equivalent to hosted publish-time scanning.

## Reproduction Assessment

Yes, at source level: current main exposes `clawhub package validate` for Plugin Inspector checks, while `clawhub scan` rejects local paths and the local ClawScan helper is internal and API-key backed. This is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. A local pre-publish scanner is aligned and useful, but maintainers should choose whether to extend `package validate`, restore/localize `scan <path>`, or define a separate command before a PR implements it.
