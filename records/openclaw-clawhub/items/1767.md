## Summary

Keep open. Current main partially covers author-facing plugin validation through `clawhub package validate`, but the requested supported local pre-publish scanner still lacks local ClawScan/static/LLM security parity, credential/code findings, and stable severity semantics.

## What This Changes

The issue requests a supported local plugin scanner or CLI lint command that plugin authors can run before publishing to catch manifest, SKILL.md, and security-scan findings locally.

## Best Possible Solution

Define a supported public scanner contract that reuses Plugin Inspector and shared deterministic security logic, documents hosted-scan parity limits, and provides stable human and JSON output with redaction rules.

## Reproduction Assessment

Yes, at source level: current main supports `clawhub package validate` for Plugin Inspector findings, but `clawhub scan` rejects local paths and local ClawScan-style analysis remains an internal script.

## Solution Assessment

Unclear. Extending `clawhub package validate` may be the right surface, but maintainers first need to decide scanner parity, redaction, severity semantics, and whether local LLM/static checks belong in the public CLI.
