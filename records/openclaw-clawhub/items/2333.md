## Label Justifications

- `P1`: Affected users can be completely blocked from ClawHub sign-in after deleting their account, and the issue consolidates many duplicate reports.
- `impact:security`: The requested purge changes account-standing enforcement and must not bypass bans, malware autobans, disabled accounts, moderation holds, or abuse flags.
- `impact:auth-provider`: The failure mode centers on GitHub OAuth identity reuse and Convex Auth provider-account binding state after account deletion.

## Summary

Keep open: current main still treats owner-initiated deletion as irreversible, rejects later GitHub reauth for deleted/deactivated users, and does not provide the requested admin-only purge path for fresh signup with the same GitHub identity.

## What This Changes

The issue requests an auditable admin recovery operation that purges eligible self-deleted account identity bindings so the same GitHub identity can sign up again without restoring deleted content or bypassing bans.

## Best Possible Solution

Approve a narrow RFC-backed admin-only recovery flow that verifies eligibility, purges only self-deleted identity bindings, records an audit trail, and keeps security/account-standing blocks separate.

## Reproduction Assessment

Yes from source inspection: deleteAccount sets deactivatedAt/purgedAt and handleDeletedUserSignIn rejects that account on later GitHub reauth; auth tests assert the blocked path. I did not run a live GitHub OAuth reproduction because this is a security-sensitive hosted account flow.

## Solution Assessment

Unclear until maintainers decide the policy. The requested admin-only audited purge shape is plausible, but automatic fresh signup would conflict with the current irreversible-deletion spec and needs product/security approval.
