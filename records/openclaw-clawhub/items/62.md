## Summary

Keep open. This security-sensitive ClawHub issue is not eligible for auto-close under the repository cleanup policy, and current main still leaves an unresolved distribution-policy gap: malicious uploads are detected, hidden, and blocked from ZIP download, but malware-blocked detail/file surfaces remain public for transparency and the public UI still renders install affordances. AgentGuard from #75 is useful prior art, but that PR closed unmerged and is not shipped protection.

## What This Changes

The issue reports that ClawHub can distribute malicious skill content through a published skill page and asks for stronger core security handling.

## Best Possible Solution

Maintainers should define the intended blocked-skill posture, then enforce it consistently across detail pages, install prompts, readme/file APIs, ZIP downloads, package fallbacks, and CLI install flows while preserving any staff-only or redacted transparency path they choose.

## Reproduction Assessment

Yes, for the issue class from source inspection: a skill with blocked.malware remains readable through getBySlug and file access while SkillHeader still renders install commands. I did not run a live exploit against the specific external URL.

## Solution Assessment

Unclear. Current static scanning, auto-hide, autoban, and download blocking are useful mitigations, but deciding whether malware-blocked content should remain publicly inspectable or be redacted is a maintainer security/product decision.
