## Summary

Keep open. Current main has meaningful ClawHub-side mitigations for malicious skill distribution, but this is an open security issue and the repo cleanup policy for ClawHub issues does not allow auto-closing it; the remaining detail/file/install and OpenClaw runtime boundaries still need maintainer security judgment.

## What This Changes

The issue reports that a malicious published skill could distribute unsafe code through ClawHub/molthub, with one example URL provided.

## Best Possible Solution

Maintainers should define the intended security posture for malware-blocked skills, then enforce it consistently across detail pages, install prompts, readme/file APIs, ZIP/package download paths, and CLI/install surfaces while preserving an explicit staff or redacted transparency path.

## Reproduction Assessment

No, not for the exact example URL alone: the legacy page redirects but the canonical skill API returns Skill not found. Source inspection gives a medium-confidence current-main path for the policy gap because malware-blocked skills can remain visible/readable for transparency and the install card still renders while ZIP downloads are blocked.

## Solution Assessment

No, not as an automated cleanup or one-shot patch. The maintainable path is a maintainer security decision first, with scanner integration such as AgentGuard evaluated as prior art rather than treated as existing protection.
