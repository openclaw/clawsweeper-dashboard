## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and this security-sensitive report still has an unresolved distribution-policy question: current main detects and blocks important malicious-skill paths, but malware-blocked detail/readme/file surfaces can remain publicly inspectable and the public page still renders install affordances.

## What This Changes

The issue asks ClawHub to address malicious skill distribution risk demonstrated by a hosted skill link.

## Best Possible Solution

Maintainers should define the intended posture for malware-blocked skills, then enforce it consistently across skill detail pages, install prompts, readme/file APIs, ZIP/package downloads, and CLI/install flows while preserving any explicit staff or redacted transparency path.

## Reproduction Assessment

Yes, for the issue class: source and tests show the malicious install payload pattern is detected, and source inspection shows malware-blocked detail/file surfaces plus install prompts still remain reachable. I did not need a live exploit run of the originally linked artifact to verify the remaining policy gap.

## Solution Assessment

Unclear. The current static scanning, auto-hide, download blocking, and security UI are useful mitigations, but choosing transparency versus full redaction/blocking is a maintainer security/product decision rather than a narrow automated fix.
