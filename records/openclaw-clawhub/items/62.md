## Summary

Keep open. Current main has meaningful registry-side malware detection, auto-hide, download blocking, and regression coverage, but the security policy remains unresolved because malware-blocked skill detail/file surfaces are deliberately still readable and the public UI still renders install commands; the ClawHub cleanup policy for this pass only permits auto-closing pull requests that current main certainly implements.

## What This Changes

The issue asks ClawHub to address malicious skill distribution risk, with a concrete example link to a hosted suspicious skill.

## Best Possible Solution

Define the intended malware-blocked skill posture, then enforce it consistently across detail pages, install prompts, readme/file APIs, ZIP downloads, CLI install flows, and any explicit staff or redacted transparency path.

## Reproduction Assessment

Yes, for the issue class from source inspection and existing regression tests. Current main detects and auto-hides obfuscated install-payload publishes, while malware-blocked detail/readme/file surfaces and install affordances remain reachable by design.

## Solution Assessment

Unclear. The current scanner, auto-hide, download blocking, and moderation visibility are useful mitigations, but the remaining transparency-versus-redaction choice is a maintainer security/product decision rather than a narrow automated fix.
