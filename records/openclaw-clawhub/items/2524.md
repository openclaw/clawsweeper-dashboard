## Summary

Current main still selects the GitHub OAuth application entirely through production credentials, so no repository commit proves the reported personal branding has been corrected. The issue remains assigned, collaborator-authored, and protected for security review; it needs authorized production follow-up rather than cleanup.

## What This Changes

Replace the personal-branded GitHub OAuth client used by ClawHub web and CLI sign-in with an officially owned and branded ClawHub/OpenClaw OAuth app.

## Best Possible Solution

Use an official organization-owned GitHub OAuth app with the existing production callback contract, rotate both Convex credentials atomically, and verify fresh-browser web login plus CLI authentication before resolving the issue.

## Reproduction Assessment

Yes, from the supplied production screenshot and exact consent copy, with source confirming that the configured GitHub client controls this identity. A fresh private-browser authorization remains necessary to verify today's external production state.

## Solution Assessment

Yes. Transferring or replacing the OAuth app and rotating production credentials fixes the branding and ownership at their source; changing repository auth routing would be broader and would not change GitHub's displayed app owner.
