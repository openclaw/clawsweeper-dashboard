## Summary

This remains necessary: current main still delegates the consent identity to production `AUTH_GITHUB_ID`/`AUTH_GITHUB_SECRET`, and there is no repository change proving the official OAuth app has replaced the personally branded client. Because the report is collaborator-authored, security-labeled, assigned, and requires organization-level OAuth administration plus live verification, it should stay open for explicit human follow-up.

## What This Changes

Replace the personally branded GitHub OAuth application used by ClawHub web and CLI sign-in with an official organization-controlled ClawHub application.

## Best Possible Solution

Transfer or recreate the GitHub OAuth app under the official organization, rotate the Convex production credentials with rollback readiness, and record redacted fresh-browser and CLI authorization proof showing official branding and successful callbacks.

## Reproduction Assessment

Partly yes: the issue provides direct production screenshot evidence, and current source clearly identifies the OAuth credentials controlling the consent identity. This review did not complete a fresh authenticated consent flow, so current production branding is not independently reproduced at high confidence.

## Solution Assessment

Yes in direction: organization ownership and production credential rotation are the narrow root-cause solution, rather than adding another authentication implementation. The exact callback and rollback procedure must be verified before changing production.
