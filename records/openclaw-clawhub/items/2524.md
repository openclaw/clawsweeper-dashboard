## Summary

Keep open: current main and v0.23.0 still delegate GitHub consent-screen branding to the OAuth App selected by production credentials, and there is no merged PR, release note, or source evidence proving production moved to an official ClawHub/OpenClaw-owned app. The issue is collaborator-authored and security/auth-provider-sensitive, so cleanup closure is not appropriate.

## What This Changes

The issue asks for GitHub OAuth sign-in consent branding to use an official ClawHub/OpenClaw-owned OAuth App instead of a personal ClawdHub app.

## Best Possible Solution

Have a maintainer create or transfer the production GitHub OAuth App to the official OpenClaw/ClawHub owner, rotate AUTH_GITHUB_ID and AUTH_GITHUB_SECRET together, and verify web plus device-code sign-in against the live consent screen.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection confirms the reported web and device-code sign-in path reaches the same GitHub provider, but the actual consent branding depends on current production GitHub OAuth App state.

## Solution Assessment

Yes: changing or transferring the production OAuth App and rotating the existing Convex production credentials is the narrowest maintainable fix. A source-only PR would not change GitHub's visible consent branding unless production OAuth ownership/configuration changes too.
