## Summary

Current main still delegates the GitHub consent identity to externally administered production OAuth credentials, and there is no evidence that the app has been transferred or rebranded. This collaborator-authored, security-labeled, assigned operational issue requires explicit maintainer handling rather than repository cleanup or an automated fix PR.

## What This Changes

Replace or transfer the production GitHub OAuth client so web and CLI authorization show official ClawHub/OpenClaw ownership and branding.

## Best Possible Solution

Use an official organization-owned GitHub OAuth application with the existing production callback contract, rotate the Convex production credentials through the manual operations path, and retain before-and-after proof of successful fresh web and CLI authorization.

## Reproduction Assessment

Partly yes: source inspection establishes that the external OAuth client controls the consent identity, and the attached June 5, 2026 screenshot records the reported result. A fresh production web and CLI authorization is still required for high-confidence current-state reproduction.

## Solution Assessment

Yes. Correcting ownership and branding at the existing GitHub OAuth application boundary is narrower and more maintainable than changing ClawHub source, which cannot set GitHub's consent-screen owner label.
