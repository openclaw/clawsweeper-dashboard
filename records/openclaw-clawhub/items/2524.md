## Summary

Keep open: current main and v0.22.0 still route GitHub sign-in through the OAuth app selected by production `AUTH_GITHUB_ID`/`AUTH_GITHUB_SECRET`, and I found no source, release, or related PR evidence proving the live consent branding has moved to an official ClawHub/OpenClaw app. The issue is collaborator-authored and security/auth-provider-sensitive, so it should stay with maintainer/operator follow-up.

## What This Changes

The issue asks for production GitHub OAuth consent to show an official ClawHub/OpenClaw-owned app instead of a personal ClawdHub app.

## Best Possible Solution

Use an official ClawHub/OpenClaw-owned GitHub OAuth App for production, rotate the Convex production OAuth client ID and secret together, and verify both web sign-in and CLI device authorization against the live consent screen.

## Reproduction Assessment

No for the exact current live branding: the issue screenshot shows the reported copy, and source confirms the env-controlled OAuth app path, but a fresh private-browser or CLI production authorization check is still needed.

## Solution Assessment

Yes: replacing or transferring the production OAuth App and rotating the existing Convex production credentials is the narrowest maintainable fix. A source-only PR would not change GitHub's consent branding unless production OAuth ownership/configuration changes too.
