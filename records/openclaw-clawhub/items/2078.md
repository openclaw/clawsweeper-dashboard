## Summary

Keep open: the PR is still incorrect because an owner can first re-delete an already hidden skill, overwrite `hiddenBy` with their own user id, and then pass the new undelete gate; the required changelog entry is also still missing.

## What This Changes

The PR tightens skill owner undelete authorization in Convex, adds audit `actorRole` metadata and HTTP 403 formatting, resets VT hide provenance, and adds undelete gate tests.

## Best Possible Solution

Land the stricter owner restore only after owner deletes cannot overwrite protected hidden provenance, regression tests cover delete-to-undelete bypasses, and the changelog records the user-facing auth change.

## Reproduction Assessment

Yes. Source inspection shows current main lets owners restore hidden skills directly, and PR head still permits a delete-then-undelete path because owner delete rewrites `hiddenBy` before the restore gate runs.

## Solution Assessment

No. The direction is right, but the gate must also protect the owner delete path from changing moderation-owned hidden provenance before this is a complete fix.
