## Summary

Keep open: the latest PR head fixes the earlier moderator-hidden delete chain, but still lets an owner re-delete an `owner.merged` skill, clear that protected provenance, and then undelete it. The required changelog entry is also still missing.

## What This Changes

The PR tightens Convex skill soft-delete/undelete authorization, formats v1 HTTP authz errors, records audit actor-role metadata, adjusts VT hide provenance, removes an unused slug-validator export, and adds undelete-gate regression tests.

## Best Possible Solution

Preserve protected hide provenance for every non-restorable state, including owner-initiated merges, and land the authorization change with regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path on PR head: an `owner.merged` row has `hiddenBy` set to the owner, owner delete clears `moderationReason`, and the next owner undelete passes.

## Solution Assessment

No. The restore-side gate is the right direction, but the re-delete guard must also treat denied reasons like `owner.merged` as protected or avoid clearing provenance on already-hidden rows.
