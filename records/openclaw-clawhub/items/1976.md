## Summary

Keep open: this is a security-sensitive runtime approval-boundary report, and a MEMBER explicitly left it open for OpenClaw/runtime policy follow-up; current ClawHub main can catalog and scan Nova but does not execute `nova send` or enforce per-command approval.

## What This Changes

The issue reports that Nova's documented financial-action guardrail was bypassed when an agent ran `nova send` before confirming network, amount, destination, and explicit user approval.

## Best Possible Solution

Add or route to a runtime-enforced approval gate before irreversible financial commands such as `send`, `withdraw`, and claim-link creation can execute, while keeping ClawHub scan warnings and Nova documentation aligned.

## Reproduction Assessment

No. The trace and current published SKILL.md make the failure plausible, but ClawHub current main is a registry/scanner and does not execute Nova commands, so a current-main reproduction requires the OpenClaw runtime path outside this checkout.

## Solution Assessment

No. A ClawHub-only wording change is not the best fix because the report says the agent already bypassed written guardrails; the maintainable direction is an enforceable runtime approval boundary for final financial actions.
