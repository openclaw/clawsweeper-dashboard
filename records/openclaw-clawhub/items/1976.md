## Summary

Keeping this open because the maintainer discussion and current evidence point to an unresolved security-sensitive runtime approval-boundary issue. ClawHub is already keeping nova visible but flagged as suspicious/review-required; that does not enforce the required confirmation sequence before a live financial command runs.

## What This Changes

The issue requests stronger enforcement so nova `send`/`withdraw` financial final actions cannot run before network, amount, destination, and explicit user confirmations.

## Best Possible Solution

Keep nova flagged as review-required in ClawHub and implement/test a runtime approval gate that blocks nova financial final actions until the required confirmations are complete.

No. The reported trace and published nova skill semantics are concrete, but a high-confidence reproduction would require an OpenClaw runtime session and a safe harness for `nova send`, `nova withdraw`, or claim-link creation outside this checkout.

## Solution Assessment

No for a SKILL.md-only wording change. The maintainable fix is enforceable runtime approval before invoking the financial final commands, while ClawHub continues surfacing the suspicious/review-required state.
