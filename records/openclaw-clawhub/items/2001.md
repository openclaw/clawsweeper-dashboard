## Summary

This should stay open: the issue has a protected `security` label, and current `main` has a credible regression path where a scoped package name can retarget an existing publish to the scope publisher instead of the package's current owner.

## What This Changes

The issue reports that publishing a new version of the existing `@axonflow/openclaw` code-plugin fails as belonging to another publisher even though package inspect shows the logged-in user as owner.

## Best Possible Solution

Preserve cross-publisher protection while making existing package publishes default to the stored owner unless an explicit owner transfer or owner override is requested, with regression coverage for scoped names that do not match the current owner.

## Reproduction Assessment

Yes, from source inspection with high confidence: the reported command sends no explicit owner, current `main` infers `axonflow` from `@axonflow/openclaw`, and the release mutation rejects if that resolved publisher differs from the existing package owner. I did not run a live production publish in this read-only review.

## Solution Assessment

Yes, keeping this open is the right outcome. The likely fix should be a focused ownership-resolution change around `publishPackageImpl` and `insertReleaseInternal`, but the security label means maintainers should approve the exact authorization rule.
