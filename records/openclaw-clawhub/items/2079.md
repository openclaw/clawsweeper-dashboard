## Summary

This PR should stay open: it addresses a real unified-search mismatch, but the latest head can now show a hidden-suspicious-results notice for vector candidates that would not actually render in the unfiltered search results.

## What This Changes

The PR adds a `searchSkillsWithMeta` action, surfaces a hidden-suspicious-results notice in unified search and header typeahead, and adds regression tests for those paths.

## Best Possible Solution

Keep the disclosure approach, but make the hidden flag reflect the actual unfiltered search eligibility so it only appears when a suspicious result would otherwise render.

## Reproduction Assessment

Yes. Current main clearly reproduces the source-level mismatch by filtering unified search with `nonSuspiciousOnly: true` while `/skills` defaults to including suspicious skills; the remaining PR defect is also source-reproducible in the vector-pool probe.

## Solution Assessment

No. The chosen disclosure design is maintainable, but the vector hidden-match probe is broader than the actual search pipeline and should apply the same token/render eligibility before merge.

## Review Findings

Full review comments:

- **[P2] Check vector candidates against search tokens before flagging hidden results:** `convex/search.ts:771-773`

Apply the same query-token eligibility here or compare against an actual unfiltered result set.
