## Summary

Keep open: this member-authored PR is not implemented on main, and the proposed patch still misses an all-hidden-results case while adding hidden-count work to header typeahead searches that never display it.

## What This Changes

The PR adds a Convex `searchSkillsWithMeta` action, switches unified search to consume it, and renders a hidden-suspicious-results notice with a `/skills` link on the search page.

## Best Possible Solution

Keep the safer unified-search default, but disclose hidden suspicious matches in every search state where the count is positive and avoid doing that count work for consumers that do not render the disclosure.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main hardcodes `nonSuspiciousOnly: true` in unified search while `/skills` defaults it to false. The PR-specific empty-state regression is also source-reproducible when `items` is empty but `suspiciousHiddenCount` is positive.

## Solution Assessment

No, not yet: the direction is reasonable, but the notice must also appear when all visible results are filtered out, and metadata counting should be gated to `/search` rather than every shared hook consumer.
