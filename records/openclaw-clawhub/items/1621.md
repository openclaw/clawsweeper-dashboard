## Summary

Keep open. This is a corroborated production GitHub OAuth sign-in failure, current live auth-start routes still return HTTP 500, related OAuth/session reports remain open, and ClawHub cleanup policy does not auto-close open issues.

## What This Changes

The issue reports that first-time GitHub authorization completes but ClawHub fails to create a signed-in session and returns to the sign-in page or shows a generic failure.

## Best Possible Solution

Keep this in the canonical OAuth/session investigation until maintainers restore production GitHub sign-in for active first-time accounts, preserve explicit blocked-account rejection, and add focused callback/session regression coverage plus redacted production smoke proof.

## Reproduction Assessment

Yes for the central sign-in failure: on 2026-05-17, both the canonical ClawHub GitHub auth-start endpoint and the direct Convex auth-start endpoint returned HTTP 500 while the site root returned 200. The exact post-authorization no-session callback path was not independently replayed because production currently fails before redirecting into GitHub OAuth.

## Solution Assessment

Yes for keeping this open and routing it to maintainer/operator auth triage. A speculative source-only repair is not the narrowest maintainable solution until production logs, OAuth app settings, env values, cookies, and affected account state identify the failing branch.
