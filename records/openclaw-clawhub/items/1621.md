## Summary

Keep open: current production still fails at the GitHub OAuth start route with HTTP 500 on both the canonical site and direct Convex endpoint, and current main still routes sign-in through the same Convex Auth/provider surface.

## What This Changes

The report asks for GitHub OAuth sign-in to create a ClawHub session for first-time publishers instead of returning to the page or showing a generic failure.

## Best Possible Solution

Restore production GitHub OAuth end to end for active accounts, keep deleted/banned account rejection intact, and add redacted production smoke proof plus focused auth regression coverage for the specific fix.

## Reproduction Assessment

Yes. The central sign-in-start failure reproduces on the current deployed main because both https://clawhub.ai/api/auth/signin/github and the direct Convex endpoint return HTTP 500 instead of redirecting to GitHub OAuth; the later callback/no-session variant remains blocked by that earlier failure.

## Solution Assessment

Yes for keeping this open and routing it to maintainer/operator auth triage. A source-only automated repair is not the narrowest safe path until production logs, OAuth/env settings, cookie behavior, and account state identify the failing branch.
