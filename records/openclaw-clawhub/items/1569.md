## Summary

Keep open under the ClawHub issue policy and because the direct transfer-to-org path is still not implemented. Current main supports publish-time owner migration, but `clawhub transfer` and the transfer API remain user-recipient based, while the broader publisher/org transfer PR is still open and conflicting at https://github.com/openclaw/clawhub/pull/1635.

## What This Changes

The issue asks how to move existing personal skills into an organization so other org users can publish and update them after `clawhub transfer` reports `User not found`.

## Best Possible Solution

Keep `--owner` plus `--migrate-owner` as the immediate supported migration path, and land a publisher-targeted transfer model that covers user-to-org, org-to-user, and org-to-org with explicit role checks, approvals, aliases, stats, and audit records.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: direct transfers still accept `toUserHandle`, resolve only users or user-kind personal publishers, and store `toUserId`, so an org publisher handle resolves to null and produces `User not found`; I did not run a live authenticated transfer.

## Solution Assessment

Unclear as a final direct-transfer solution. Publish-time `--owner --migrate-owner` is the narrow current workaround, while direct org transfers need a publisher-targeted schema and authorization design rather than only changing handle lookup.
