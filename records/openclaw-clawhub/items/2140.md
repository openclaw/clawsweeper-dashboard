## Summary

Keep open: this PR targets a real source-reproducible search underrecall bug, but the patch needs a guard for Convex full-text query limits and still needs real behavior proof before merge.

## What This Changes

The PR adds full-text search indexes on `skillSearchDigest`, queries them from `directPrefixSkillMatches`, and extends Convex search tests for non-first-token skill matches.

## Best Possible Solution

Land a guarded full-text-index implementation that preserves the existing prefix/fallback paths, caps or skips over-limit full-text terms, and includes real Convex search proof for `yijian` and `vision`.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main only checks whole-field prefixes and first-token prefixes, so a non-first token in `baidu-yijian-vision` cannot match that direct search path. I did not run a live Convex reproduction.

## Solution Assessment

No, not as written: full-text indexes are a reasonable fix direction, but passing raw user queries into `.search(...)` is not safe without handling Convex's term limit. The maintainable path is to build a bounded full-text query from tokenized input or skip the full-text branches when over limit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Guard full-text searches against long queries:** `convex/search.ts:448`
  - body: Convex full-text search queries are limited to 16 terms, but this passes the raw user query into `.search(...)`. Since `searchSkills` only checks that tokenization produced at least one token and calls this internal query before vector/fallback search, a 17+ word search can now throw instead of returning the existing fallback results; cap the terms or skip the full-text branches when over the limit. ([docs.convex.dev](https://docs.convex.dev/search/text-search?utm_source=openai))
