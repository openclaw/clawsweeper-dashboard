## Summary

Keep open. Current main does not implement device-flow CLI auth, and the PR is not ready to merge because it exposes a user-facing `--device` login path before the required server endpoints, verification UI, and token-issuance controls exist. The earlier double-JSON-read review issue appears fixed in `c77aa402`, but the end-to-end auth surface still needs maintainer design.

## What This Changes

Adds a `--device` CLI login mode, client-side device-code request/polling helpers with tests, and routes `clawhub login` / `clawhub auth login` through device-flow token storage.

## Best Possible Solution

Complete #613 as an end-to-end rollout covering CLI flags, Convex HTTP endpoints, verification UI, token issuance controls, docs, changelog, and protocol/security tests.

## Reproduction Assessment

Yes. Source inspection of the PR diff is enough: `--device` calls `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not add those HTTP routes or a `/device` verification UI.

## Solution Assessment

No. The client helper is useful, but exposing the flag before the backend and verification surface exist is not the narrowest maintainable solution for a token-granting auth flow.
