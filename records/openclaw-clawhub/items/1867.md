## Summary

Keep open: current main still lacks device-flow auth, and the PR is a client-only implementation that exposes a login mode before the required server routes, verification UI, and auth controls exist.

## What This Changes

The PR adds a new device-auth helper/test and wires a `--device` flag into both ClawHub CLI login commands to request a device code and poll for a token.

## Best Possible Solution

Complete #613 as an end-to-end device-flow rollout covering CLI flags, Convex HTTP endpoints, verification UI, token issuance controls, docs, changelog, and protocol/security tests.

## Reproduction Assessment

Yes. Source inspection of the PR head is enough: `--device` routes into `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not add matching Convex HTTP routes or a `/device` verification UI.

## Solution Assessment

No. The client helper is useful, but enabling the public flag before the server and verification surface exists is not the narrowest maintainable solution; this should land as a complete auth flow or stay hidden until backend support is ready.
