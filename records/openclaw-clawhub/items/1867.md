## Summary

Keep open: the double JSON-read bug was fixed, but the branch still exposes a client-only device login path while the server routes, verification UI, and token-issuance controls needed for a working and reviewable auth flow are absent.

## What This Changes

The branch adds a `--device` CLI login option, a device auth request/polling helper, and unit tests for `/cli/device/code` and `/cli/device/token` polling.

## Best Possible Solution

Land or replace this with an end-to-end device-flow design covering CLI, Convex/server routes, verification UI, secure one-time code handling, token issuance, rate limiting, docs, and protocol tests; keep the flag hidden until those pieces are ready.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence check: the PR exposes `clawhub login --device` and calls `/cli/device/code` and `/cli/device/token`, while current main does not register those endpoints.

## Solution Assessment

No. The client helper direction is plausible, but exposing the flag before backend/UI/security support lands is not the narrowest maintainable solution.
