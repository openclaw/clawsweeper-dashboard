## Summary

Keep open. The earlier double body-read review was fixed, but current main still does not implement device-flow auth and this PR exposes a client-only `--device` login path without the required backend routes, verification UI, changelog entry, or server-side auth controls.

## What This Changes

Adds a `--device` CLI login option, client-side device-code request/polling helpers, and unit tests for Device Flow authentication.

## Best Possible Solution

Land device flow as an end-to-end auth feature covering backend code issuance/token exchange, authenticated verification UI, CLI behavior, docs, changelog, and regression tests together.

## Reproduction Assessment

Yes for the review findings. Source inspection of the PR head shows the CLI posts to `/cli/device/code` and `/cli/device/token`, while current main has no matching routes or user-code verification page.

## Solution Assessment

No. The client helper is a useful slice, but exposing the public flag before the backend and verification UI land is not the narrowest maintainable solution.
