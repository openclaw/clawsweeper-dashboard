## Summary

Keep open: current main does not implement device-flow auth, and the PR still exposes a client-only auth path before the required server routes and verification UI exist. The earlier double JSON parse issue appears addressed in the latest PR commit.

## What This Changes

The PR adds a `--device` CLI login option, device-code request and polling helpers, and unit tests for the client-side device flow.

## Best Possible Solution

Land or supersede this with an end-to-end device-flow implementation covering the CLI, Convex/server endpoints, verification UI, secure code storage, expiry and rate limiting, token issuance, docs, and tests; keep the flag hidden until those pieces are ready.

## Reproduction Assessment

Yes. A focused static check reproduces the issue: the PR’s `--device` path posts to `/cli/device/code`, while current main has no matching route and still documents token paste for headless auth.

## Solution Assessment

No, not as submitted. A client-only flag is not the narrowest maintainable solution for an auth feature; either include the server/UI/security controls in the same implementation or keep the option unavailable until they land.
