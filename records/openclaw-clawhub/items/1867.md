## Summary

Keep open. Current main does not implement device-flow auth, and this PR still exposes a public CLI device login path without the required server routes, verification page, and token-exchange controls.

## What This Changes

Adds a `--device` option to `clawhub login` and `clawhub auth login`, a client-side device-code request/polling helper, and unit tests for that helper.

## Best Possible Solution

Ship device flow end to end with CLI, Convex HTTP endpoints, authenticated verification UI, token issuance controls, docs/changelog, and security regression coverage, or keep the CLI flag hidden until that path is ready.

## Reproduction Assessment

Yes. Source inspection of the PR diff shows the new flag calling `/cli/device/code` and `/cli/device/token`, while current main has no matching routes or device-code verification page.

## Solution Assessment

No. The client helper is a useful slice, but exposing the public flag before the backend, verification UI, and token controls land is not the narrowest maintainable solution.
