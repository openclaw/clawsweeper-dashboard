## Summary

Keep open. Current main does not implement device-flow auth, and this PR still exposes a client-side `--device` path without the required server endpoints, verification UI, or token-issuance controls; the earlier double-read polling bug was addressed in the latest PR commit.

## What This Changes

The PR adds a ClawHub CLI `--device` login path, a device-code request/polling helper, and unit tests for polling `/cli/device/code` and `/cli/device/token`.

## Best Possible Solution

Land or replace this with an end-to-end device-flow auth design covering CLI, Convex/server routes, verification UI, secure one-time code handling, token creation, rate limiting, docs, and protocol tests; keep the flag hidden until those pieces are ready.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence check: current main lacks `--device` and device routes, while the PR would call missing `/cli/device/code` and `/cli/device/token` endpoints when `clawhub login --device` is used.

## Solution Assessment

No. The client helper direction is plausible, but exposing the CLI flag before backend/UI support lands is not the narrowest maintainable solution; either implement the flow end to end or hide the flag until the server side is ready.
