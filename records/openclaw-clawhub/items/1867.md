## Summary

Keep open: current main still has browser/token-only CLI auth, and this PR exposes only the client side of device flow. The earlier JSON double-read review was addressed, but the backend endpoints, verification UI, and token-security controls are still missing.

## What This Changes

The PR adds a client-side `--device` login mode, a device-auth polling helper with tests, and CLI auth command dispatch for RFC 8628-style authentication.

## Best Possible Solution

Ship device flow end to end with CLI support, Convex endpoints, a verification page, token issuance controls, docs/changelog, and protocol/security regression tests.

## Reproduction Assessment

Yes. Source inspection is enough: the PR posts to `/cli/device/code` and `/cli/device/token`, while current main registers no matching Convex routes and has no `/device` verification page.

## Solution Assessment

No. The client helper is a useful slice, but enabling the flag before backend and verification support exists is not the narrowest maintainable solution; the flow should land end to end or remain hidden until server support is ready.
