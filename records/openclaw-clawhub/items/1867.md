## Summary

Keep this PR open: current main does not implement device-flow auth, and the PR exposes a client flag before the required server/device-verification surface exists.

## What This Changes

The PR adds a `--device` CLI login mode, a device-flow polling client, and unit tests for device-code and token polling behavior.

## Best Possible Solution

Land device flow as a complete auth rollout with server endpoints, verification UI, token controls, CLI client, docs, changelog, and tests reviewed together.

## Reproduction Assessment

Yes. The PR client posts to `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not define those routes or a verification page.

## Solution Assessment

No. A client-only flag is not the best way to solve the headless login request; the maintainable path is to add the backend flow or keep the CLI flag unavailable until the backend is ready.
