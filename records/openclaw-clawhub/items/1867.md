## Summary

Keep open: current main does not already implement this PR, and the patch is a client-side slice that exposes a visible device login mode before the required server endpoints, verification UI, and auth controls exist. The prior double JSON-read concern was addressed in the follow-up commit.

## What This Changes

The PR adds a `--device` CLI login mode, a client-side device-code request/polling helper, and unit tests for headless Device Flow auth.

## Best Possible Solution

Ship device flow end to end with CLI support, Convex HTTP endpoints, an authenticated verification page, token issuance controls, docs/changelog, and protocol/security regression tests, or keep the flag hidden until that path is ready.

## Reproduction Assessment

Yes. Source inspection is enough: the PR head adds `--device` and posts to `/cli/device/code` and `/cli/device/token`, while current main has no matching Convex routes and `/cli/auth` remains loopback-only.

## Solution Assessment

No. The client helper is a useful slice, but enabling a public flag before backend, verification, and security controls land is not the narrowest maintainable solution.
