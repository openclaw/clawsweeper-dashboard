## Summary

Keep open: current main still has token/loopback CLI auth only, and this PR exposes a client-side `--device` flow whose required server endpoints and verification page are not present in main or in the branch.

## What This Changes

Adds a ClawHub CLI `--device` login mode, a device-code request/polling client, and unit tests for the client-side device flow.

## Best Possible Solution

Land device flow as an end-to-end auth feature with CLI flags, server endpoints, verification UI, bounded token issuance, docs/changelog, and protocol tests; keep the flag unavailable until that backend is ready.

## Reproduction Assessment

Yes. Source inspection is enough: the PR command posts to `/cli/device/code` and `/cli/device/token`, while current main has no matching routes or device verification page.

## Solution Assessment

No. A client-only `--device` flag is not the best fix for headless auth; the maintainable path is a complete reviewed rollout or holding the client flag until the server flow exists.
