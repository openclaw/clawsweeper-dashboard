## Summary

Keep this PR open: current main does not already implement device-flow auth, and the PR still exposes a user-facing CLI flag before the required backend endpoints and verification UI exist. The earlier double-JSON-read review concern appears fixed in the latest PR diff.

## What This Changes

The PR adds a `--device` option to both CLI login commands, a `deviceAuth` helper that requests and polls device-flow endpoints, and unit tests for that helper.

## Best Possible Solution

Ship device flow end to end with CLI support, Convex HTTP endpoints, verification UI, token issuance controls, docs/changelog, and protocol/security tests, or keep the flag hidden until the backend path is ready.

## Reproduction Assessment

Yes. Source inspection is enough: the PR registers `--device` and posts to `/cli/device/code` and `/cli/device/token`, while current main registers no routes for those paths and `/cli/auth` is loopback-only.

## Solution Assessment

No. A client-only device-flow flag is not the best landing shape; this should either ship end to end with backend and verification support or remain hidden until that path is ready.
