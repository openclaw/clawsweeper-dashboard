## Summary

Keep open: current main does not implement device-flow auth, and this PR is still a client-only login path with missing server endpoints, verification UI, changelog coverage, and security review needs before merge.

## What This Changes

The PR adds a `--device` option to ClawHub CLI login commands, a device-flow request/polling helper, and unit tests for the new helper.

## Best Possible Solution

Land device flow as a complete reviewed rollout covering CLI flags, Convex HTTP endpoints, verification UI, secure token issuance, docs, changelog, and protocol tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR posts to `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not define those routes or a device verification page.

## Solution Assessment

No. A client-only `--device` flag is not the best way to solve headless auth; the maintainable path is to complete the server-side RFC 8628 flow or keep the CLI path unavailable until backend support lands.
