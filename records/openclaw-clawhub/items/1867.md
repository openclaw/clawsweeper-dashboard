## Summary

Keep open: current main does not implement device-flow auth, and the PR still exposes a CLI login mode whose required server endpoints and verification UI are absent from both main and the branch.

## What This Changes

The PR adds a `--device` CLI login path, client-side device-code request and polling helpers, and unit tests for device-flow authentication.

## Best Possible Solution

Complete device-flow auth as one end-to-end rollout covering CLI flags, Convex HTTP endpoints, a verification UI, token issuance controls, docs, changelog, and protocol/security tests.

## Reproduction Assessment

Yes. Source inspection is enough: the PR routes `--device` into `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not add matching Convex routes or a `/device` verification UI.

## Solution Assessment

No. The client helper is useful, but enabling the flag before backend and verification support exists is not the narrowest maintainable merge boundary; this should land as a complete auth flow or remain hidden until server support exists.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Add the device endpoints before enabling the flag:** `packages/clawhub/src/cli.ts:136`
  - body: The new `--device` option routes users into a flow that posts to `/cli/device/code` and `/cli/device/token`, but neither current main nor this PR adds those HTTP routes or the verification page. Merging this would expose a login mode that cannot complete against the shipped app.
  - confidence: 0.93

  - body: `requestDeviceCode` validates the string fields but returns `expires_in` and `interval` unchecked, and the login command immediately uses them for display, timeout, and sleep math. A malformed response can produce `NaN` timing or invalid polling instead of a clear protocol error.
  - confidence: 0.84

  - confidence: 0.86
