## Summary

Keep open. Current main still has browser loopback and token-paste login only, while this PR is a useful but incomplete client-side slice that enables a device-flow flag before the required server endpoints, verification page, and token-issuance controls exist.

## What This Changes

Adds a CLI `--device` login mode, a `deviceAuth` polling helper with tests, and auth-command dispatch for a client-side device-code login flow.

## Best Possible Solution

Ship device flow end to end with CLI support, Convex endpoints, an authenticated verification page, secure token issuance controls, docs/changelog, and protocol/security regression tests; keep the flag hidden until that surface is ready.

## Reproduction Assessment

Yes. Source inspection is enough: the PR posts to `/cli/device/code` and `/cli/device/token`, while current main registers no matching routes and has no `/device` verification page.

## Solution Assessment

No. The client helper is a useful start, but enabling the flag before backend and verification support exists is not the narrowest maintainable solution; the auth flow should land end to end or remain unavailable.
