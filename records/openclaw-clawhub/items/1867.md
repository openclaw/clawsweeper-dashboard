## Summary

Keep this PR open: the client-side device-flow helper is useful and the double JSON-read bug was fixed, but the PR still enables a visible login mode without the server endpoints, verification UI, or auth/security controls needed for it to work safely.

## What This Changes

The PR adds a `--device` CLI login mode, a device-code request/polling helper, and unit tests for client-side device-flow authentication.

## Best Possible Solution

Complete #613 as an end-to-end auth rollout with CLI flags, Convex HTTP endpoints, a verification page, token issuance controls, docs, changelog, and protocol/security tests.

## Reproduction Assessment

Yes. Source inspection of the PR head is enough: `--device` routes into `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not add matching Convex HTTP routes or a `/device` verification UI.

## Solution Assessment

No. The client helper is useful, but enabling the flag before the server and verification surface exists is not the narrowest maintainable solution; this should land as a complete auth flow or stay hidden until backend support is ready.

## Review Findings

Overall correctness: patch is incorrect

  - body: The new `--device` option routes users into `cmdDeviceLogin`, which posts to `/cli/device/code` and `/cli/device/token`, but neither current main nor this PR adds those HTTP routes or a `/device` verification page.   - confidence: 0.84
