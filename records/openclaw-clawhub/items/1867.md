## Summary

Keep open: current main does not implement device-flow login, and this PR is not safe to close or merge as-is because it exposes a client-side `--device` path without the server endpoints, verification page, or token-issuance controls it depends on.

## What This Changes

Adds a `--device` login mode to the ClawHub CLI, a device-flow request/polling client, and unit tests for device-code and token polling behavior.

## Best Possible Solution

Land device flow as an end-to-end auth feature with CLI flags, server endpoints, verification UI, token controls, docs, changelog, and protocol tests, or keep the CLI flag unavailable until that backend is ready.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR command posts to `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not define those routes or a device verification page.

## Solution Assessment

No. A client-only `--device` flag is not the best fix for headless auth; the maintainable solution is a complete reviewed rollout or a hidden client until the server flow exists.
