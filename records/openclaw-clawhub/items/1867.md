## Summary

Keep this PR open: current main does not implement device-flow auth, and the PR still exposes a new CLI auth path before the required server endpoints, verification UI, and token-issuance controls exist.

## What This Changes

The PR adds a `--device` option to ClawHub CLI login commands, a device-code request/polling helper, and unit tests for the client-side device flow.

## Best Possible Solution

Land or supersede this with an end-to-end device-flow auth implementation covering CLI, Convex HTTP endpoints, verification UI, one-time code storage, expiry and rate limiting, token minting, docs, tests, and maintainer auth review; keep the CLI flag hidden until those pieces are ready.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the PR’s `--device` path posts to `/cli/device/code`, while current main’s HTTP router has no matching endpoint, so the command would fail before authentication can start.

## Solution Assessment

No. The client helper and tests are useful, but exposing the flag before backend/UI support exists is not the narrowest maintainable or secure solution; either ship the full flow together or keep the client path unreachable.
