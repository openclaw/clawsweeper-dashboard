## Summary

Keep open: current main does not implement device-flow auth, and the PR is still only the client half of the feature. The branch addresses the headless-auth request in #613 and also fixed the earlier double-JSON-read review comment, but it exposes a user-facing `--device` path before the required server endpoints and verification UI exist.

## What This Changes

The PR adds a ClawHub CLI `--device` login path, device-code request/polling helpers, and unit tests for client-side device-flow authentication.

## Best Possible Solution

Land or supersede this with an end-to-end device-flow auth implementation covering the CLI flag, Convex/server endpoints, verification UI, secure one-time code storage, expiry and rate limiting, token creation, docs, tests, and maintainer auth-security review; keep the flag hidden until those pieces are ready.

## Reproduction Assessment

Yes. On current main, the CLI exposes only token paste and browser login, while the PR adds calls to `/cli/device/code` and `/cli/device/token` that current routing does not provide.

## Solution Assessment

No. The client helper is a reasonable part of the solution, but exposing the flag before backend/UI support exists creates a user-visible dead path and leaves token issuance controls unreviewed.
