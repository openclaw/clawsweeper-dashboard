## Summary

Keep open: the PR is still useful work toward headless auth, but it exposes a CLI device-login path while the required server endpoints, verification UI, persistence, and token issuance controls are not present in current main or in the PR diff.

## What This Changes

The PR adds a ClawHub CLI `--device` login option, a client-side device-code request/poll helper, and unit tests for the proposed `/cli/device/*` flow.

## Best Possible Solution

Land or replace this with an end-to-end device-flow auth design covering CLI, Convex HTTP routes, verification UI, secure pending-code storage, expiry, rate limiting, replay prevention, token creation, docs, changelog, and protocol tests; keep the CLI flag hidden until those pieces are ready.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with the PR applied, `clawhub login --device` would POST to `/cli/device/code` and `/cli/device/token`, but current main has no matching routes or verification UI.

## Solution Assessment

No. The client helper is a plausible piece, but exposing it before backend/UI support lands is not the narrowest maintainable solution; either implement the flow end to end or keep the option unavailable until the server side is ready.
