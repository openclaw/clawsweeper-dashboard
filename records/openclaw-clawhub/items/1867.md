## Summary

Keep open: current main does not implement device-flow auth, and this PR is still a client-only rollout. The double JSON-read review comment was addressed in the latest PR diff, but the branch still exposes a login flag whose required server routes, verification UI, and token-issuance controls are absent.

## What This Changes

The PR adds a client-side `--device` CLI login mode, device-code request/poll helpers, and unit tests for `/cli/device/*` polling.

## Best Possible Solution

Complete the linked device-flow feature end to end: CLI flag, Convex HTTP endpoints, verification UI, token issuance controls, docs, changelog, and protocol/security tests should land together or behind an unavailable flag.

## Reproduction Assessment

Yes. Source inspection is enough: PR head wires `--device` to `/cli/device/code` and `/cli/device/token`, while current main has no matching Convex route or `/device` verification page.

## Solution Assessment

No. The client module is a useful start and the double-read bug is fixed, but exposing the flag before the server endpoints and security controls land is not the narrowest maintainable rollout.
