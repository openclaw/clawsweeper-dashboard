## Summary

Keep open for maintainer review: current main does not implement device auth, so this PR is not obsolete or closeable, but the branch exposes a user-facing auth flag before the required server endpoints, verification UI, and token-flow controls exist. The earlier double JSON parse concern was fixed in c77aa4028c49, leaving end-to-end auth/security blockers before merge.

## What This Changes

The branch adds a ClawHub CLI `--device` login path, a device-code request/polling client with unit tests, and wires both `login` commands to the new flow.

## Best Possible Solution

Land device flow as an end-to-end auth feature with CLI flags, Convex HTTP endpoints, verification UI, token issuance controls, docs, changelog, and protocol tests, or keep the CLI flag unavailable until that backend exists.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR's `--device` branch posts to `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not define those routes or a device verification page.

## Solution Assessment

No. A client-only `--device` flag is not the best way to solve headless auth; the maintainable path is a complete reviewed device-flow rollout or a hidden client until the server contract exists.
