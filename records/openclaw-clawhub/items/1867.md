## Summary

Keep open: current main does not implement device-flow auth, and this PR still exposes a user-facing CLI flag before the required server endpoints, verification page, and token controls exist. The earlier double JSON body-read review comment appears addressed by the PR’s second commit.

## What This Changes

The PR adds a `--device` option to CLI login commands, a device-code request/polling client, and unit tests for the new client module.

## Best Possible Solution

Ship device flow as a reviewed end-to-end auth feature with CLI flags, Convex/http endpoints, verification UI, expiring single-use codes, polling controls, docs, changelog, and protocol tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `clawhub login --device` on the PR branch posts to `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not define those routes or a device verification page.

## Solution Assessment

No. A client-only `--device` flag is not the best way to solve headless auth; the maintainable path is a complete backend-backed rollout or keeping the CLI flag unavailable until the server flow exists.
