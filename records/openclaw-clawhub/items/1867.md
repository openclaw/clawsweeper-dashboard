## Summary

Keep open. Current main does not implement device-flow login, and this PR is still a client-only auth path that exposes `--device` without the required server endpoints or verification UI; the earlier double-JSON-read review issue was fixed in `c77aa4028c497a610d940d3617ee780cda21557c`.

## What This Changes

The PR adds a `--device` login option, a device-code request/polling helper, and tests for client-side Device Flow authentication in the CLI.

## Best Possible Solution

Complete the linked device-flow feature as an end-to-end auth rollout covering CLI flags, Convex HTTP endpoints, the verification page, secure token issuance controls, docs, changelog, and protocol/security tests.

## Reproduction Assessment

Yes. Source inspection of the PR head is enough: `--device` routes into `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not add matching HTTP routes or a `/device` verification UI.

## Solution Assessment

No. The client helper is useful, but enabling a visible flag before the server and verification surface exists is not the narrowest maintainable solution; it should land as a complete auth flow or remain hidden until backend support is ready.

Overall correctness: patch is incorrect

  - body: The new option routes users into `cmdDeviceLogin`, which posts to `/cli/device/code` and `/cli/device/token`, but this PR does not add those HTTP routes or a `/device` verification page and current main does not have them.
