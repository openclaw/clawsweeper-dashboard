## Summary

Keep open; the PR is useful but not merge-ready because the new client path depends on device-code server endpoints and a verification UI that are absent from current main and from the PR, so the advertised login mode cannot complete.

## What This Changes

The PR adds a `--device` login mode, a CLI device-auth polling module, and unit tests for headless ClawHub authentication.

## Best Possible Solution

Land an end-to-end device-flow implementation covering CLI flags, server-side code issuance, token exchange, authenticated verification UI, secure API-token creation, rate limiting, protocol-state tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. The PR’s new client posts first to `/cli/device/code`, while current main and the PR diff contain no matching handler or verification page, so `clawhub login --device` would fail before authorization can begin.

## Solution Assessment

No. Client-side polling is directionally right, but exposing the flag before implementing the server/device verification side is not a complete or maintainable solution.
