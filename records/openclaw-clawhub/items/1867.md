## Summary

Keep open: current main does not implement the device-flow feature, and the PR still exposes a user-facing CLI flag before the required server endpoints, verification UI, and device-code controls exist. The earlier double JSON-read review has been addressed in the latest PR head.

## What This Changes

The branch adds a `--device` CLI login mode, a device-code request/polling helper with tests, and wiring from both login commands to `/cli/device/*` endpoints.

## Best Possible Solution

Complete #613 as an end-to-end device-flow rollout covering CLI flags, Convex HTTP endpoints, verification UI, token issuance controls, docs, changelog, and protocol/security tests.

## Reproduction Assessment

Yes. Source inspection is enough: PR head registers `--device` and calls `/cli/device/code` and `/cli/device/token`, while current main and the PR file list do not include matching Convex HTTP routes or a `/device` verification UI.

## Solution Assessment

No. A client-only flag is not the best way to solve headless auth; the maintainable path is to land the server-side device flow and keep the CLI entry point unavailable until that support exists.
