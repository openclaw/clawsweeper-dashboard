## Summary

This PR should stay open because current main does not implement device-flow login, and the proposed branch still exposes a CLI auth path whose required server endpoints and verification UI are absent; the earlier polling double-read bug appears fixed in the latest PR head.

## What This Changes

The PR adds a `--device` option to both ClawHub CLI login commands, a client-side device-code request and polling helper, and unit tests for that helper.

## Best Possible Solution

Land or supersede this with an end-to-end device-flow auth implementation covering the CLI flag, Convex/server endpoints, verification UI, secure one-time device-code storage, expiry and rate limiting, token creation, docs, tests, and maintainer auth-security review; keep the flag hidden until those pieces are ready.

## Reproduction Assessment

Yes. The focused check is source-level: the PR calls `/cli/device/code` and `/cli/device/token`, while current `convex/http.ts` and route searches show no matching server routes or verification page, so `clawhub login --device` cannot complete against current ClawHub.

## Solution Assessment

No. A client-only flag is not the narrowest maintainable solution for a new auth mechanism; the safer path is an end-to-end implementation or a hidden client helper that is not exposed until backend support is merged.
