## Summary

Keep this PR open. The client-side device-flow slice is useful and the earlier double-JSON read finding was fixed, but the PR still exposes a user-visible `--device` login mode that cannot complete against current main because the required Convex HTTP endpoints and verification UI do not exist.

## What This Changes

Adds a client-side `--device` login mode to the ClawHub CLI with device-code request/polling helpers and unit tests.

## Best Possible Solution

Implement the complete RFC 8628 path across CLI, Convex HTTP, authenticated verification UI, token issuance controls, docs, changelog, and regression tests before making `--device` public.

## Reproduction Assessment

Yes for the review findings. Source inspection shows the PR calls `/cli/device/code` and `/cli/device/token`, while current main has no matching routes or device-code verification page.

## Solution Assessment

No. The client helper is a useful slice, but exposing the flag before the backend, verification UI, and token controls land is not the narrowest maintainable solution.
