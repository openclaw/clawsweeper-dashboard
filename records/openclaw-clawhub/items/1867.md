## Summary

Keep open: the client-side device-flow work is useful, but current main does not implement the server endpoints or verification page needed for the PR to work end to end.

## What This Changes

The PR adds a `--device` CLI login option, a device-code request/polling helper, and unit tests for headless ClawHub authentication.

## Best Possible Solution

Ship device flow as an end-to-end auth feature covering the CLI flag/helper, Convex HTTP endpoints, verification page, token issuance controls, docs/changelog, and protocol/security tests.

## Reproduction Assessment

Yes. Source inspection of the PR head shows `--device` posts to `/cli/device/code` and `/cli/device/token`, while current main has no matching HTTP routes or `/device` verification route.

## Solution Assessment

No. The client helper is a useful slice, but enabling the flag before backend and verification support exists is not the narrowest maintainable solution; the flow should land end to end or remain hidden until server support is ready.
