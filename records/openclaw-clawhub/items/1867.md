## Summary

Keep open. Current main does not implement device flow, and this PR still exposes a client flag before the required `/cli/device/*` endpoints and verification UI exist, so it is not a cleanup-close candidate or merge-ready as-is.

## What This Changes

The branch adds a `--device` option to both CLI login commands, introduces device-code request/polling helpers and tests, and stores the returned token through the existing login verifier.

## Best Possible Solution

Ship device flow end to end: CLI flag/helper, Convex HTTP code/token endpoints, `/device` verification UI, token issuance controls, docs/changelog, and protocol/security tests.

## Reproduction Assessment

Yes. PR head source shows `--device` posts to `/cli/device/code` and `/cli/device/token`, while current main search and `convex/http.ts` show no matching endpoints or verification page.

## Solution Assessment

No. The client helper is useful, but exposing the flag before backend and verification support exists is not the narrowest maintainable solution; the flow should land end to end or stay hidden until server support is ready.
