## Summary

Keep open: current main does not implement device-flow auth, and the PR still exposes a nonfunctional auth path because the required server endpoints and verification UI are not part of the patch.

## What This Changes

The PR adds a client-side `--device` login mode for the ClawHub CLI with device-code request/polling helpers and unit tests.

## Best Possible Solution

Land or replace this with an end-to-end device-flow design covering CLI, Convex/server routes, verification UI, secure one-time code handling, token creation, rate limiting, docs, and protocol tests before exposing the flag.

## Reproduction Assessment

Yes. Source inspection shows current main lacks `--device` and device routes, while the PR diff would call missing `/cli/device/code` and `/cli/device/token` endpoints when `clawhub login --device` is used.

## Solution Assessment

No. The client helper is a plausible component, but exposing the public flag before backend/UI support and security controls land is not the narrowest maintainable solution.
