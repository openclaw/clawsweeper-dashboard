## Summary

Keep open: the intended device-flow feature is not on current main, and the PR still has merge blockers because it exposes an auth path before the required server routes, verification UI, and token controls exist.

## What This Changes

Adds a client-side `--device` login mode for `clawhub login` and `clawhub auth login`, with device-code request/polling helpers and unit tests.

## Best Possible Solution

Land device flow as a complete auth feature across the CLI, Convex HTTP routes, authenticated verification UI, token issuance controls, docs, changelog, and regression tests.

## Reproduction Assessment

Yes for the PR findings. Source inspection shows the PR calls `/cli/device/code` and `/cli/device/token`, while current main has no matching routes or device-code verification page.

## Solution Assessment

No. The client helper is a useful slice, but exposing the flag before backend and security controls exist is not the narrowest maintainable solution.
