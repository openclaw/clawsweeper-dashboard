## Summary

Keep open: current main does not implement device-flow login, and this PR is still a client-only slice that exposes a `--device` login mode before the required server endpoints, verification UI, and token-issuance controls exist. The earlier double body-read review issue appears fixed in the latest PR head.

## What This Changes

The PR adds a `--device` CLI login option, a device-flow request/polling helper, and unit tests for client-side device-code authentication.

## Best Possible Solution

Ship device flow end to end with CLI support, Convex HTTP endpoints, a verification page, token issuance controls, docs/changelog, and protocol/security regression tests.

## Reproduction Assessment

Yes. Source inspection is enough: the PR posts to `/cli/device/code` and `/cli/device/token`, while current main has no matching HTTP routes and no `/device` verification page.

## Solution Assessment

No. The client helper is a useful slice, but enabling the flag before backend and verification support exists is not the narrowest maintainable solution; the flow should land end to end or remain hidden until server support is ready.
