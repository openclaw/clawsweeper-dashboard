## Summary

Keep this PR open, but it is not merge-ready: current main does not implement the intended device-flow feature, and the PR exposes a client-side auth path before the required server routes, verification UI, storage, and token-issuance controls exist.

## What This Changes

The PR adds a ClawHub CLI `--device` login option, client-side device-code request/polling helpers, and unit tests for a proposed `/cli/device/*` flow.

## Best Possible Solution

Land or replace this with an end-to-end device-flow design covering CLI, Convex HTTP routes, verification UI, pending-code storage, expiry, rate limiting, replay prevention, token creation, docs, changelog, and protocol tests; keep the CLI flag unavailable until those pieces exist.

## Reproduction Assessment

Yes. With the PR applied, `clawhub login --device` would call `/cli/device/code` and `/cli/device/token`, but source inspection of current main shows no matching routes, route constants, schema, or verification UI.

## Solution Assessment

No. The client helper is a plausible component, but exposing it alone is not the best fix; the maintainable solution is either an end-to-end device flow or a hidden/gated flag until server support lands.
