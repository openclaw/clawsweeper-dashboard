## Summary

Keep open. Current main does not implement device-flow auth, so this PR is not obsolete; however the patch is incomplete because it exposes a CLI device-login flag before the required server endpoints, verification page, and token-issuance controls exist. The earlier double JSON-read review concern appears fixed in the latest PR diff.

## What This Changes

The PR adds a `--device` CLI login option, a client-side device-code request/polling helper, and unit tests for RFC 8628-style authentication.

## Best Possible Solution

Ship device flow end to end with CLI support, Convex HTTP endpoints, a verification page, token issuance controls, docs/changelog, and protocol/security regression tests, or keep the flag hidden until that path is ready.

## Reproduction Assessment

Yes. Source inspection is enough: the PR posts to `/cli/device/code` and `/cli/device/token`, while current main has no matching Convex HTTP routes and no `/device` verification page.

## Solution Assessment

No. The client helper is a useful slice, but enabling the CLI flag before backend and verification support exists is not the narrowest maintainable solution; the flow should land end to end or remain hidden until server support is ready.
