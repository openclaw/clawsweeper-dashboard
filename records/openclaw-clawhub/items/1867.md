## Summary

Keep open. The PR is still a useful implementation candidate for #613, and the double body-read review comment was addressed, but the branch remains client-only and would expose a `--device` login path before ClawHub has the required device-code endpoints or verification UI.

## What This Changes

Adds a `--device` login mode to the ClawHub CLI, a device-flow request/polling helper, and unit tests for that helper.

## Best Possible Solution

Land device flow as a complete reviewed rollout covering CLI flags, Convex HTTP endpoints, verification UI, token issuance controls, docs, changelog, and protocol tests.

## Reproduction Assessment

Yes. Source inspection and the PR diff give a high-confidence path: the new flag calls `/cli/device/code` and `/cli/device/token`, while current main and the PR do not add those routes or a verification page.

## Solution Assessment

No. A client-only flag that calls unimplemented endpoints is not the narrowest maintainable solution; the safer path is to complete backend support first or keep the CLI path unavailable until it exists.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Implement the device endpoints before exposing the flag:** `packages/clawhub/src/cli.ts:136`
  - body: The new `--device` option sends users into a flow that posts to `/cli/device/code` and `/cli/device/token`, but neither this PR nor current main defines those routes or a verification page. Merging this would expose a login mode that cannot succeed against ClawHub.
  - confidence: 0.91

  - body: `requestDeviceCode` only validates the string fields, but `cmdDeviceLogin` immediately uses `expires_in` and `interval` for display, deadline calculation, and polling delays. A malformed response can produce `NaN` output or invalid polling timing instead of failing early.
  - confidence: 0.82

  - body: This is a user-facing CLI feature, and the repo policy requires `feat` changes to have a `CHANGELOG.md` entry. Add an Unreleased entry before merge so the login change is covered in release notes.
  - confidence: 0.86
