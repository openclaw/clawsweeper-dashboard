## Summary

Keep open: current main does not implement device-flow auth, and the PR remains a client-only auth path with missing server endpoints/security review surface, numeric validation, and changelog coverage.

## What This Changes

Adds a `--device` CLI login flag plus a device-flow client module and tests that call `/cli/device/code` and `/cli/device/token`.

## Best Possible Solution

Land device flow as a complete reviewed auth feature with code/token endpoints, verification UI, token issuance controls, CLI client, docs, changelog, and tests, or keep the flag unavailable until those pieces are ready.

## Reproduction Assessment

Yes. The PR client posts to `/cli/device/code` and `/cli/device/token`, while current main and the PR diff contain no matching routes or verification page, so the new flag would hit missing endpoints once exposed.

## Solution Assessment

No. A client-only flag is not the best fix for #613; either include the server/device verification implementation in the rollout or keep the flag unavailable until backend support is ready.
