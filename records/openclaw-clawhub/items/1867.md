## Summary

Keep open: current main does not implement device-flow auth, and this PR still exposes a client-only login flag whose server routes, verification UI, and token controls are absent. The paired feature request #613 remains open, so this PR should continue through normal review rather than cleanup closure.

## What This Changes

The branch adds a ClawHub CLI `--device` login mode, a new `deviceAuth` request/polling helper with tests, and wiring from login commands to `/cli/device/code` and `/cli/device/token`.

## Best Possible Solution

Complete #613 as an end-to-end device-flow rollout covering CLI flags, Convex HTTP endpoints, verification UI, token issuance controls, docs, changelog, and protocol tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR registers `--device` and posts to `/cli/device/code` and `/cli/device/token`, while current main and the PR file list do not define those routes or a verification page.

## Solution Assessment

No. A client-only `--device` flag is not the best way to solve headless auth; the maintainable path is to land the server-side RFC 8628 flow and keep the CLI path unavailable until that support exists.
