## Summary

Keep open: current main does not implement device-flow auth, and this PR still exposes a client-only `--device` login path without the required server routes or verification UI. The related #613 remains open, so this PR should be reviewed as an implementation candidate rather than closed by cleanup.

## What This Changes

Adds a `--device` login option to the ClawHub CLI, a new device-flow request/polling helper, and unit tests for the client-side flow.

## Best Possible Solution

Land device flow as a complete reviewed rollout covering CLI flags, Convex HTTP endpoints, verification UI, token issuance controls, docs, changelog, and protocol tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR posts to `/cli/device/code` and `/cli/device/token`, while current main and the PR file list do not define those routes or a device verification page.

## Solution Assessment

No. A client-only `--device` flag is not the best way to solve headless auth; the maintainable path is to complete the server-side RFC 8628 flow or keep the CLI path unavailable until backend support lands.
