## Summary

Keep open: current main does not implement device-flow CLI auth, and this PR still exposes a client-only token-auth path whose required server endpoints and verification UI are absent from the branch.

## What This Changes

Adds a `--device` login option to the ClawHub CLI, a device-flow polling client, and unit tests for code request and token polling behavior.

## Best Possible Solution

Land device flow as a complete auth feature with server endpoints, verification UI, token controls, docs, changelog, and tests, or keep the CLI flag unavailable until that backend is ready.

## Reproduction Assessment

Yes. The PR client posts to `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not define those routes, so the exposed command path would fail against the current app.

## Solution Assessment

No. A client-only `--device` flag is not the narrowest maintainable solution for headless auth; the safer path is a complete reviewed rollout or a hidden client until the server flow exists.
