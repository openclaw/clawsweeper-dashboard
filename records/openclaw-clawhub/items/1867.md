## Summary

Keep open: current main does not implement this PR, and the branch exposes a user-facing device login mode that depends on server endpoints and a verification UI that are not present in main or in the PR diff.

## What This Changes

Adds a ClawHub CLI `--device` login path, a device-flow request/polling client, and unit tests for the client-side polling behavior.

## Best Possible Solution

Land device flow as an end-to-end auth feature with CLI flags, server code issuance, token polling/exchange, verification UI, security controls, docs, changelog, and protocol tests, or keep the CLI flag unavailable until that backend exists.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR posts to `/cli/device/code` and `/cli/device/token`, while current main and the PR diff do not define those routes or a device verification page.

## Solution Assessment

No. A client-only `--device` flag is not the best fix for headless auth; the maintainable solution is a complete reviewed rollout or a hidden client path until the server flow exists.
