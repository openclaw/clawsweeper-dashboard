## Summary

Keep open: current main covers the documented `metadata.openclaw.requires.env`, `primaryEnv`, `envVars`, and top-level array forms, but the live RiskState example still uses scalar `env: RISKSTATE_API_KEY`, and the current parser/env-signal helpers only consume arrays for that shorthand. The Agents Mail comment's nested `metadata.openclaw.requires.env` shape appears covered by current source, but the original scalar-env report is not definitely implemented.

## What This Changes

The issue reports that SKILL.md env frontmatter declarations are not carried into ClawHub registry/security metadata, causing required env vars to display as none and scanner trust mismatches.

## Best Possible Solution

Accept scalar top-level `env: FOO` as a required env declaration consistently in publish parsing and scanner env-signal extraction, while keeping the documented `metadata.openclaw.requires.env` path as the preferred format.

## Reproduction Assessment

Yes, source-reproducible: the live RiskState SKILL.md uses scalar `env`, while current main only parses array/object env declarations. I did not run a full publish/UI reproduction.

## Solution Assessment

Yes, a narrow parser/env-signal regression fix is the best path. It should not replace the documented `metadata.openclaw.requires.env` format; it should make the existing shorthand compatibility path handle the scalar case safely.
