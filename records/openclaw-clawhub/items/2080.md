## Summary

Keep open: current main still treats deleted package rows as reserved and has no package restore, owner repair, or hard-delete recovery path; the remaining action needs maintainer ownership policy and production verification, and this ClawHub issue is not eligible for cleanup close.

## What This Changes

The issue asks for supported recovery or clearer semantics when an org-owned plugin package deleted from the UI still reserves its package name and blocks republishing.

## Best Possible Solution

Define package deletion semantics, verify the `@yahaha-studio/kichi-forwarder` production ownership state, then add an audited restore/owner-repair/admin hard-delete path or make permanent reservation explicit in UI and docs.

## Reproduction Assessment

Yes for the central blocked-republish behavior: current source shows UI/API deletion sets `softDeletedAt`, visible reads hide the row, and publish rejects the same normalized package name. The exact production ownership mismatch needs live data verification.

## Solution Assessment

Unclear. A restore or audited recovery flow is likely safer than freeing names by default, but maintainers need to choose package delete, reservation, owner repair, transfer, and hard-delete semantics before implementation.
