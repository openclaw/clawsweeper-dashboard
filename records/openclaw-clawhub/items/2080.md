## Summary

Keep open: current main still allows a dashboard/CLI package delete to soft-delete the package while preserving the normalized package row, and publish then blocks reuse until a restore or owner-repair path exists. Because this is a ClawHub issue and the remaining recovery policy is namespace-sensitive, it should stay open for maintainer handling.

## What This Changes

The issue requests a supported recovery path when a UI-deleted plugin package remains reserved and blocks republishing under the same package name.

## Best Possible Solution

Define package deletion/reservation semantics, then implement an audited owner/admin recovery path for restore, ownership repair/transfer, or hard-delete with matching UI, CLI/API, tests, docs, and spec coverage.

## Reproduction Assessment

Yes. Source inspection shows dashboard deletion soft-deletes the package, public surfaces hide that row, and publish still checks the preserved normalizedName and ownership fields before rejecting reuse.

## Solution Assessment

Unclear. Blindly releasing the package name after delete is not the safest fix; the maintainable path is an explicit audited recovery policy with owner and publisher verification.
