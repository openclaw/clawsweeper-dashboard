## Summary

Keep open. Current main still soft-deletes package rows without a restore/reclaim path, and the publish/transfer/repair paths can keep a deleted or owner-mismatched package name blocked; ClawHub cleanup policy also keeps issues open for maintainer handling.

## What This Changes

The issue asks for a supported recovery path when republishing an org-owned plugin package fails after dashboard deletion because the package name or ownership mapping remains reserved.

## Best Possible Solution

A maintainer should inspect the production package record, choose the deleted-package lifecycle policy, then provide an audited restore, owner repair/transfer, hard-delete, or reclaim path across backend, API/CLI/UI, docs, specs, and tests.

## Reproduction Assessment

No for the exact org-owned package because that requires production records and an authenticated publish attempt. Source inspection does show the current-main branches that reserve soft-deleted package names and throw the reported ownership error when owner keys differ.

## Solution Assessment

Unclear until maintainers choose recovery semantics. The safer direction is an audited restore, transfer, owner repair, or hard-delete flow rather than allowing publish to implicitly overwrite a deleted or mismatched executable package record.
