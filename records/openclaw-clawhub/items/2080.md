## Summary

Keep open: current main still blocks republishing a deleted package name and has no owner-facing package restore or recovery path; the exact error text appears improved on main, but the recovery problem remains ownership-sensitive and needs maintainer policy plus production verification.

## What This Changes

Request a recovery or clarification path for org-owned plugin packages that disappear after UI delete but still block republishing the same package name.

## Best Possible Solution

Define package delete semantics, then add an audited owner/admin restore or recovery path and update the UI/docs so package name reservation is explicit.

## Reproduction Assessment

Yes for the central blocked-republish behavior: source inspection shows dashboard/API deletion sets softDeletedAt and current publish rejects the same normalized package name. The exact reported cross-publisher error may now differ on main because the soft-deleted branch runs before the ownership mismatch branch.

## Solution Assessment

Unclear. A restore/undelete flow is likely safer than releasing names by default, but maintainers need to choose whether package deletion reserves names, supports restore, supports hard-delete, or requires an admin repair flow.
