## Summary

Keep open: current main still preserves a soft-deleted package row and blocks publishing the same normalized package name, while the owner-facing restore/recovery path remains undefined and security-sensitive.

## What This Changes

The issue reports that deleting a plugin package from the UI hides it but leaves the package name reserved so publishing the same name again fails.

## Best Possible Solution

Define package delete semantics, then add an audited owner/admin package restore or recovery flow and update the UI, CLI, API docs, and deletion warning to make name reservation explicit.

## Reproduction Assessment

Yes for the central blocked-republish behavior: source inspection shows dashboard deletion sets `softDeletedAt`, public reads hide the package, and publish rejects the preserved normalized-name row. The exact original cross-publisher error may now differ because current main rejects soft-deleted rows before the ownership-mismatch branch.

## Solution Assessment

Unclear until maintainers choose the package lifecycle policy. A restore/undelete or audited recovery path is likely safer than freeing names by default, but the live record should be verified before any production action.
