## Summary

Keep open: current main still hides soft-deleted packages while preserving the package row/name reservation and rejecting republish, and the remaining recovery path is ownership-sensitive product work rather than cleanup-close material.

## What This Changes

The issue asks how org owners can recover or republish a plugin package name after deleting it from the ClawHub UI when the underlying package ownership/name reservation remains blocked.

## Best Possible Solution

Define package delete semantics, then add an audited package restore/recovery or ownership repair path with clear UI/CLI/API docs while preserving scoped namespace protections.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a package through the dashboard or package DELETE API, then publish the same normalized package name and hit the soft-deleted package rejection or owner-key mismatch while read/UI surfaces hide the package.

## Solution Assessment

Unclear. A restore/undelete or audited admin recovery path looks safer than silently releasing package names, but maintainers need to choose the package retention and takeover policy first.
