## Summary

Keep open: current main still treats package deletion as a soft-delete that hides the package while preserving the namespace record, and the reporter’s requested recovery path requires a maintainer policy and production support decision rather than conservative cleanup closure.

## What This Changes

The issue asks how an org package owner can recover or republish a package name after deleting the package in the ClawHub UI and then hitting a stale ownership/slug reservation error.

## Best Possible Solution

Define package delete semantics, then add an audited owner/admin restore or recovery flow that preserves namespace ownership and gives maintainers a safe way to resolve the affected production package.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a plugin package through the dashboard or package DELETE API, then publish the same normalized name and hit the soft-deleted package rejection branch while normal owner/read surfaces hide the package.

## Solution Assessment

Unclear. Restore or undelete is likely safer than releasing the slug outright, but maintainers first need to choose the product/security policy for reserved package names, ownership repair, and hard-delete recovery.
