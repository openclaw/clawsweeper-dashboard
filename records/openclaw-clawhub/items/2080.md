## Summary

Keep open: current main still hides soft-deleted packages from public/read paths while retaining the package row and rejecting publishes for deleted or differently owned package names. The live `@yahaha-studio/kichi-forwarder` and related `@xquik/tweetclaw` records need maintainer/admin verification before any restore, transfer, hard delete, or self-service recovery policy is safe.

## What This Changes

The issue requests an owner-safe recovery policy for package names that remain reserved or mapped to the wrong publisher after UI delete or owner-record drift.

## Best Possible Solution

Maintainers should inspect the live package and publisher records, verify requester authority, apply an audited restore, transfer, name repair, or hard delete only when appropriate, and document the approved delete/recovery semantics.

## Reproduction Assessment

Yes, at source level: current main hides soft-deleted packages from readable/public paths while retaining the global `normalizedName` row, and the publish path rejects soft-deleted or different-owner packages. The exact production state still requires admin data inspection.

## Solution Assessment

Unclear as an automatic fix: current main already has undelete, guarded transfer, and admin repair primitives, but choosing whether delete releases names, keeps them reserved, or exposes owner recovery is a product and security policy decision.
