## Summary

Keep open: current main still reserves deleted package rows by normalized name, blocks publish onto them, and has no package undelete, restore, transfer, or hard-delete recovery path for the reporter’s confirmed case.

## What This Changes

The issue asks for a supported recovery path or clearer semantics when a UI-deleted package still reserves its normalized package name and blocks republish.

## Best Possible Solution

Define durable package deletion and namespace-reservation semantics, then ship an audited restore, owner repair/transfer, or hard-delete recovery path with matching UI/API/CLI wording.

## Reproduction Assessment

Yes, as a source-level path: package deletion soft-deletes the row, publish resolves that same normalizedName and rejects it, and package transfer rejects soft-deleted rows. I did not run a live production-row reproduction in this read-only review.

## Solution Assessment

No; current main is only a partial solution because it improves validation and active-package transfer but does not let a deleted package be restored, transferred, reclaimed, or permanently removed.
