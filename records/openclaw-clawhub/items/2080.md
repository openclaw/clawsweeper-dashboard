## Summary

Keep open. Current main still treats UI-deleted package rows as reserved backend records and blocks publish/repair paths around soft-deleted package ownership, while the reporter's exact org-owned stale record needs maintainer/admin inspection and a security-sensitive package lifecycle decision.

## What This Changes

Request for a maintainer-supported recovery path when a UI-deleted org-owned plugin package is hidden but still blocks republishing the same package name.

## Best Possible Solution

A maintainer should inspect the production package record, choose the lifecycle policy, and provide an audited restore, owner repair, transfer, or hard-delete/reclaim path with matching UI/CLI/API docs and regression tests.

## Reproduction Assessment

No, not for the exact reporter sequence without production package records and an authenticated publish attempt. Source inspection does show current-main branches that reserve soft-deleted package names and throw the reported ownership error when owner keys differ.

## Solution Assessment

Unclear until maintainers choose deleted-package recovery semantics. The safest direction is an audited restore or owner-repair flow, not implicit publish overwrite of a deleted or mismatched package row.
