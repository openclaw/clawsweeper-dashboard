## Summary

Keep open. Current main still treats package deletion as recoverable reservation rather than slug release, and the concrete affected packages need maintainer/admin record inspection before choosing restore, transfer, identity repair, hard-delete, or an exception.

## What This Changes

The issue asks how ClawHub should recover package names or ownership when a UI-deleted or stale-owned plugin package blocks republishing.

## Best Possible Solution

A maintainer/admin should inspect the affected live records, apply the narrow audited recovery action that matches those records, and clarify owner-facing UI or docs that package delete reserves names unless a guarded recovery action changes that state.

## Reproduction Assessment

Yes, at source level: current main preserves package rows on soft delete and rejects publishing against soft-deleted or wrong-owner package rows. A live reproduction of the named packages requires admin data access.

## Solution Assessment

Unclear until maintainers inspect production records and choose the recovery policy. The safest direction is guarded, audited restore, transfer, identity repair, or admin recovery rather than automatic slug release on delete.
