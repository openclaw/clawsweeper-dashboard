## Summary

Keep open: current main still makes package deletion a soft-delete that hides the package while preserving the namespace record, blocks republish, and lacks an owner/admin package restore or recovery path for the reporter’s concrete org package.

## What This Changes

The issue asks how org owners can recover or republish a plugin package after the UI delete hides it but leaves the package name or ownership reservation blocking publish.

## Best Possible Solution

Define package delete semantics, then implement an audited owner/admin package restore or recovery flow across Convex, API, CLI, UI, docs, and production support for the affected package.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a package through the dashboard or package DELETE API, then publish the same normalized name and hit the soft-deleted package rejection branch while the package is hidden from owner/read surfaces.

## Solution Assessment

Unclear. Restore/undelete looks like the safest likely direction, but maintainers first need to choose whether deletion reserves names, releases names, supports hard-delete, or remains irreversible with clearer warnings.
