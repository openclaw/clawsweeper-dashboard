## Summary

Keep open: current main still has a package soft-delete recovery gap. Deleting a plugin hides the package and publish refuses the same normalized name, but there is no package restore route for the org owner to use.

## What This Changes

The issue asks for a recovery or clearer deletion policy when an org-owned plugin package is deleted in the UI but the package name remains reserved and publishing fails.

## Best Possible Solution

Define package deletion semantics, then add an audited owner/admin restore or recovery path that preserves namespace protections and updates the UI, CLI, API docs, and tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a package through the dashboard or package DELETE API, then publish the same normalized package name and hit the existing soft-deleted package rejection path while public read/search surfaces hide it.

## Solution Assessment

Unclear. A restore or admin recovery flow looks like the safest likely direction, but maintainers first need to choose whether package deletion reserves names, releases names, supports hard-delete, or remains irreversible with stronger warnings.
