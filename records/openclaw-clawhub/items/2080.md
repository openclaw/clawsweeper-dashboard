## Summary

Keep open under the ClawHub issue policy; current main still preserves soft-deleted package rows and owner keys while publish resolves those rows for ownership checks, so maintainers need to define the recovery and reservation policy.

## What This Changes

The issue asks for a supported recovery path or clearer reservation behavior after deleting an org-owned plugin package from the UI blocks publishing the same name again.

## Best Possible Solution

Define package deletion semantics, then implement an explicit restore/undelete, transfer/repair, or hard-delete recovery path and update dashboard, CLI, and API wording to match that policy.

## Reproduction Assessment

Yes, as a source-level path: UI/API delete soft-deletes the package, publish still resolves the row by normalized name, and the owner-key check can throw the reported publisher error. I did not run a live authenticated production publish, but the reporter confirmed this is their observed flow.

## Solution Assessment

Unclear until maintainers choose the intended reservation policy. The narrow maintainable direction is an explicit recovery or reservation contract, not silently creating a second package identity.
