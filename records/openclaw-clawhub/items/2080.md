## Summary

Keep open: current main has package soft-delete, undelete, transfer, and admin repair primitives, but this remains a live ownership-recovery case. The reported `@yahaha-studio/kichi-forwarder` deletion and the related `@xquik/tweetclaw` owner mismatch both need maintainer/admin verification before any restore, transfer, hard delete, or policy change is safe.

## What This Changes

The issue asks for a supported recovery path and clearer semantics when package publishing is blocked after UI deletion or stale package-owner/slug reservation state.

## Best Possible Solution

Maintainers should inspect the live package and publisher records, verify requester authority, apply an audited restore/transfer/name repair or hard-delete only when appropriate, and then document or build the approved recovery semantics for package deletion and stale owner records.

## Reproduction Assessment

Yes, at source level: current main hides soft-deleted packages from readable/public paths while retaining the `normalizedName` row, and the publish path rejects both soft-deleted packages and owner-key mismatches. The exact production state still requires admin data inspection.

## Solution Assessment

Unclear as an automatic code fix: undelete, package transfer, and admin repair primitives already exist, but the stale/deleted owner case needs maintainer security and product judgment before expanding self-service recovery.
