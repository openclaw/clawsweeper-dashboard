## Summary

Keep open. ClawHub issue-cleanup policy keeps issues open, and current main still leaves deleted package recovery unresolved: package delete is soft-delete, public reads hide the row, publish blocks deleted or mismatched-owner package names, and current transfer/repair paths reject soft-deleted package rows.

## What This Changes

The issue asks for a maintainer-supported recovery path when UI deletion hides an org-owned plugin package but publishing the same package name later fails with stale ownership or slug reservation.

## Best Possible Solution

A maintainer should inspect the production package record, choose the package lifecycle policy, then provide an audited restore, owner repair, transfer, hard-delete, or reclaim path with matching UI, CLI/API, docs, and tests.

## Reproduction Assessment

No, not for the exact org-owned package without production records and an authenticated publish attempt. Source inspection does show the current-main branches that reserve soft-deleted package names and throw the reported ownership error when owner keys differ.

## Solution Assessment

Unclear until maintainers choose recovery semantics. The safer direction is an audited restore or owner-repair flow, not letting publish implicitly overwrite a deleted or mismatched executable package record.
