## Summary

Keep open under the ClawHub issue policy; current main still preserves soft-deleted package name/owner rows and has no package restore path, so maintainers need to define recovery and reservation semantics.

## What This Changes

The issue asks for a maintainer-supported recovery path when a UI-deleted package name remains reserved and blocks republishing under an org publisher.

## Best Possible Solution

Define package deletion semantics, then add an explicit restore/undelete or maintainer recovery path and update dashboard/API/CLI wording to disclose whether deleted package names remain reserved.

## Reproduction Assessment

Yes, as a source-level path: UI/API delete soft-deletes the package row, publish still resolves that row by normalizedName, and the ownership check can throw the reported error for a stale or different ownerPublisherId. I did not run a live production-row reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the intended policy. A maintainable fix should make restore, transfer/repair, hard delete, or permanent reservation explicit rather than silently creating a second package identity.
