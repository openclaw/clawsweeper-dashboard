## Summary

Keep open: current main still soft-deletes UI-deleted plugin packages, hides them from public surfaces, and blocks publishing the same normalized package name without an owner-facing restore or recovery path. The related lifecycle issue is still open, and this issue adds a concrete production recovery case that needs maintainer handling.

## What This Changes

The issue asks ClawHub to define package delete semantics and provide recovery when a UI-deleted org-owned plugin package still blocks republishing the same name.

## Best Possible Solution

Define package deletion and reservation policy, then add an audited owner/admin restore or recovery path while verifying the reporter's production ownership before any data repair.

## Reproduction Assessment

Yes from source inspection: dashboard/API delete sets softDeletedAt, public reads hide the row, and publishing rejects the existing deleted normalized package. I did not run a live production publish for the reporter's package.

## Solution Assessment

Unclear. Releasing the package name by default is not clearly safe; an audited restore, owner repair, or staff recovery path is safer but needs a maintainer lifecycle decision first.
