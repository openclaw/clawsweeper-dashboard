## Summary

Keep open. Current main still allows the reported state: package deletion hides the package from public surfaces while preserving the package row and normalized name, and publish still rejects the preserved row. The related broader package lifecycle request at https://github.com/openclaw/clawhub/issues/1370 is also open, but this issue adds a concrete production recovery case that should stay visible for maintainer handling.

## What This Changes

The issue asks for a supported recovery path when a UI-deleted plugin package disappears from ClawHub surfaces but the package name remains blocked during publish.

## Best Possible Solution

Define package deletion and reservation semantics, then provide an audited restore, transfer, owner repair, or hard-delete recovery path after verifying control of the affected org publisher.

## Reproduction Assessment

Yes, from source inspection: dashboard deletion calls package soft delete, public surfaces hide soft-deleted packages, and publishing still checks the existing normalized package row. I did not run a live production publish against the reporter's specific package.

## Solution Assessment

Unclear. Automatically releasing the package name on delete is not clearly safe; the more maintainable direction is an explicit, audited recovery policy for restore, transfer, owner repair, or hard delete.
