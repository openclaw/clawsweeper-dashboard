## Summary

Keep open. ClawHub policy keeps issues open, and current main still leaves this as a security-sensitive production recovery case: soft delete hides the package while retaining the name, and the exact stale owner mismatch cannot be resolved safely without inspecting the affected production row.

## What This Changes

The issue asks how an org owner can recover when deleting a plugin package hides it from ClawHub but republishing the same package name fails due to retained ownership or slug reservation.

## Best Possible Solution

A maintainer should inspect the affected production package records, then use or add an audited recovery path that preserves reserved-name semantics while allowing authorized restore, owner repair/transfer, or hard delete.

## Reproduction Assessment

Unclear for the exact reported owner-mismatch error: the hidden-but-reserved state is source-reproducible, but the specific stale ownerPublisherId outcome depends on production data for @yahaha-studio/kichi-forwarder. Current public API checks show the package is hidden, not what retained backend owner it has.

## Solution Assessment

Yes for keeping this open and routing to maintainer support. The safe path is audited restore, transfer, identity repair, or hard delete after production inspection, not allowing publish to reclaim soft-deleted names implicitly.
