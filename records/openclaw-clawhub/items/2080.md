## Summary

Keep open: current main still soft-deletes packages, hides them from read/search surfaces, keeps the package name reserved, blocks republish, and has no package restore/undelete recovery path for the reporter’s case.

## What This Changes

The issue asks how org owners can recover when a UI-deleted plugin package remains reserved and republishing the same package name fails.

## Best Possible Solution

Define package delete semantics, then add an audited owner/admin restore or recovery flow with matching UI, CLI, API, docs, and production support for the reporter’s package.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a package through the dashboard or package delete API, then publish the same normalized name and hit the soft-deleted package rejection branch; I did not run the authenticated production publish flow.

## Solution Assessment

Unclear. An audited package restore/undelete flow looks like the safest likely direction, but maintainers first need to choose whether deletion reserves names, releases names, supports hard-delete, or remains irreversible with clearer warnings.
