## Summary

Keep open under the ClawHub issue policy; current main still preserves soft-deleted package rows and enforces the stored owner key during publish, so the remaining work needs maintainer policy and data-recovery judgment.

## What This Changes

The issue asks how org owners can recover or republish a package name after deleting a plugin package from the ClawHub UI.

## Best Possible Solution

Define package deletion and name-reservation semantics, then implement an explicit restore, transfer/repair, or hard-delete recovery path with matching dashboard/API/CLI wording.

## Reproduction Assessment

Yes, as a source-level path: UI delete soft-deletes the package row, publish still resolves that row by normalizedName, and the ownership check can throw the reported error. I did not run a live production-row reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the intended reservation policy. A maintainable fix should avoid silently creating a second package identity and should make restore, transfer, hard delete, or permanent reservation explicit.
