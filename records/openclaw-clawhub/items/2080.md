## Summary

Keep open. This is a recent external issue, and ClawHub cleanup policy says issues should stay open; source inspection also confirms a real lifecycle gap where UI package deletion soft-deletes and hides the package but leaves the package name reserved, while the reported ownership-mismatch recovery needs maintainer production-record review and policy choice.

## What This Changes

The issue asks how an org owner can recover or republish a plugin package after dashboard deletion hides it but later publish attempts fail with stale ownership or slug-reservation errors.

## Best Possible Solution

Maintain the reserved-name soft-delete semantics by default, then add or use an audited maintainer recovery path for restore, owner repair/transfer, or hard delete after inspecting the affected production package records.

## Reproduction Assessment

Unclear for the exact reported owner-mismatch error: source inspection reproduces the UI-hidden but still-reserved package state, but the specific 'belongs to another publisher' outcome depends on production ownerPublisherId data for the affected package.

## Solution Assessment

Yes for keeping this open. The narrow safe direction is a maintainer-approved, audited package recovery or ownership repair workflow rather than allowing publish to reclaim soft-deleted names implicitly.
