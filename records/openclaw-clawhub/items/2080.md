## Summary

Keep open under the ClawHub issue policy; current main still has a source-level path where a dashboard package delete soft-deletes the package but leaves the global name and owner mapping in place, so republish/recovery semantics need maintainer direction.

## What This Changes

The issue asks for a recovery or policy clarification when an org-owned plugin package is deleted in the UI but its package name remains reserved and blocks publishing.

## Best Possible Solution

Define package delete semantics, then implement an explicit same-owner recovery path with matching dashboard/API wording and regression coverage for org-owned package republish after deletion.

## Reproduction Assessment

Yes, as a source-level path: dashboard delete calls `softDeletePackage`, `softDeletePackageDoc` leaves the package name and owner mapping intact, and publish still resolves that row by `normalizedName` before enforcing owner identity. I did not run a live production Convex-row reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the intended reservation policy. A maintainable fix should avoid silently creating a second package identity and should make restore, transfer/repair, hard delete, or permanent reservation explicit.
