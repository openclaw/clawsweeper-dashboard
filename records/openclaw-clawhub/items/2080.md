## Summary

Keep open. Current main still has the reported source-level failure shape: package delete is soft-delete-only, publish refuses deleted or owner-mismatched package rows, and the available transfer/repair paths skip soft-deleted packages; ClawHub issue cleanup policy also does not auto-close issues.

## What This Changes

The issue asks for a supported recovery path when an org-owned plugin package is deleted from the UI and republishing the same package name later fails with stale ownership or slug-reservation errors.

## Best Possible Solution

Define an audited package recovery lifecycle, then implement maintainer/admin or owner-facing restore, transfer, reclaim, and hard-delete flows across Convex, HTTP API, CLI/UI, docs, specs, and tests.

## Reproduction Assessment

No for the exact production package without admin records and an authenticated publish attempt. Source inspection does show current-main branches that reserve soft-deleted package names and throw the reported ownership error when owner keys differ.

## Solution Assessment

Unclear until maintainers choose recovery semantics. The safer direction is an audited restore, transfer, owner repair, reclaim, or hard-delete path rather than implicit publish overwrite of deleted or mismatched executable package records.
