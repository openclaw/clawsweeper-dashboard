## Summary

Keep open under the ClawHub issue policy; current main still preserves deleted package rows and ownership keys while publish checks the retained global package name, and the reporter confirmed this matches their live package.

## What This Changes

The issue asks for a supported recovery or clearer reservation behavior when republishing an org-owned package after UI deletion leaves the package name blocked.

## Best Possible Solution

Define explicit soft-delete name reservation semantics, then implement an audited restore/undelete, transfer/repair, or hard-delete recovery path and align dashboard, API, and CLI wording with that policy.

## Reproduction Assessment

Yes, as a source-level path: dashboard delete calls softDeletePackage, which leaves the package row soft-deleted with its name and owner keys, and publish later resolves that row by normalizedName before owner-key checks. I did not run a live Convex production-row reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the intended reservation policy. A maintainable fix should avoid silently creating a second package identity and should make restore, transfer, hard delete, or permanent reservation explicit.
