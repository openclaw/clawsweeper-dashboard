## Summary

Keeping this open: current main still soft-deletes package rows, hides them from visible surfaces, and blocks same-name publishing with a restore-required error while no package restore path exists. The remaining work needs maintainer policy and likely a production repair path for the reporter's package.

## What This Changes

The issue requests a maintainer-supported recovery or clearer policy for org-owned plugin packages deleted from the UI that still reserve the package name during publish.

## Best Possible Solution

Define package deletion as a recoverable soft-delete operation, add an owner/admin package restore or repair path for deleted packages, and update dashboard/CLI/API copy so users know names remain reserved until restored or repaired.

## Reproduction Assessment

Yes, from source inspection: dashboard/API delete sets softDeletedAt and publish resolves by normalized package name, then rejects the tombstoned row before creating a replacement. I did not run a live Convex publish/delete flow in this read-only review.

## Solution Assessment

Unclear. A restore or maintainer repair flow looks safer than automatically releasing deleted names, but maintainers need to choose the reservation policy because package names and runtime IDs are registry identities.
