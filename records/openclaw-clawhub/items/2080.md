## Summary

Keep open: current main still reserves a deleted package name, blocks republish through the package insert path, and does not expose a package restore or undelete recovery path. The exact error text has changed on main, but the reporter’s core recovery problem remains a maintainer policy/ops question.

## What This Changes

Requests a supported recovery path or clearer delete semantics when deleting an org-owned plugin blocks republishing the same package name.

## Best Possible Solution

Define package deletion semantics, then ship one supported recovery path for soft-deleted packages plus UI/docs copy that accurately warns whether package names remain reserved.

## Reproduction Assessment

Yes, at source level: create or find a package, call the dashboard/API soft-delete path, then publish the same normalized package name. Current main still finds the soft-deleted row by `normalizedName` and blocks insertion.

## Solution Assessment

No current solution is complete: transfer only works for active packages, while the publish path now tells users to restore a package even though no restore path exists. The maintainable fix needs a deliberate recovery policy before implementation.
