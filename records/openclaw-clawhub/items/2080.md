Keep open. Current main still soft-deletes package rows without freeing the normalized package name, and the named production cases need a maintainer-chosen recovery action rather than cleanup closure.

## What This Changes

The issue asks for a supported recovery path when deleting a ClawHub plugin package from the UI leaves the package name or publisher ownership reserved and blocks future publishes.

## Best Possible Solution

Maintainers should inspect the affected package and publisher records, perform the narrow audited restore, transfer, identity repair, scoped exception, or hard-delete that matches those records, and clarify in owner-facing UI or docs that package delete reserves names unless a guarded recovery action changes that state.

## Reproduction Assessment

Yes, at source level: current main preserves package rows on soft delete and then rejects publishing against soft-deleted or wrong-owner package rows. I did not perform a live production reproduction because the concrete package records require admin data access.

## Solution Assessment

Unclear until maintainers inspect production records and decide the recovery policy. The safest direction is a guarded, audited restore, transfer, identity repair, or admin recovery path with equivalent authorization checks, not automatic slug release on delete.
