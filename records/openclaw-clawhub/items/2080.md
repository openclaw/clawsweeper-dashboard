## Summary

Keep open: current main still treats package deletion as a soft-delete that hides the package, preserves the namespace record, blocks republish, and has no owner-facing restore or recovery path for the reporter's confirmed org package.

## What This Changes

The issue requests a maintainer-supported recovery or clearer delete semantics when a deleted org plugin package remains reserved and blocks republishing.

## Best Possible Solution

Define package delete semantics, then implement an audited owner/admin package restore or recovery flow across Convex, API, CLI, UI, docs, and production support while preserving namespace protections.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the central blockage: delete a package through the dashboard or DELETE API, the package is hidden with softDeletedAt, and a later publish of the same normalized name is rejected; the exact production owner-mismatch error still needs database inspection.

## Solution Assessment

Unclear. Restore or undelete looks like the safest likely direction, but maintainers first need to decide whether package deletion reserves names, releases names, supports hard-delete, or remains irreversible with clearer warnings.
