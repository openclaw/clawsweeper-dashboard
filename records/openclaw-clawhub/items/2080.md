## Summary

Keep open under the ClawHub issue policy; current main still preserves soft-deleted package rows and owner keys while the reporter confirmed the same production symptom, so maintainers need to decide the recovery and reservation semantics.

## What This Changes

The issue asks ClawHub to define and support recovery when an org-owned package deleted from the UI remains reserved and blocks publishing the same package name.

## Best Possible Solution

Define package deletion and recovery semantics, then add an explicit restore, transfer/repair, hard-delete, or permanent-reservation flow with matching dashboard/API wording and regression coverage.

## Reproduction Assessment

Yes, as a source-level path: UI deletion soft-deletes the package, publish still resolves it by normalizedName, and the owner-key check can throw the reported error. I did not run a live production-row reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the intended policy. A maintainable fix should avoid silently creating a second package identity and should make restore, transfer, hard delete, or permanent reservation explicit.
