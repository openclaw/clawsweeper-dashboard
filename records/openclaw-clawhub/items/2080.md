## Summary

Keep open under the ClawHub issue policy; current main still preserves deleted package identity and owner keys while publish resolves the deleted row before ownership checks, so the recovery/reservation policy remains unresolved.

## What This Changes

The issue asks for a maintainer-supported recovery or clarification path when a UI-deleted org-owned plugin package still blocks republishing the same package name.

## Best Possible Solution

Define package deletion semantics, then implement an explicit package restore/undelete, moderated ownership repair/transfer, hard-delete recovery, or permanent reservation warning with matching docs and tests.

## Reproduction Assessment

Yes, as a source-level path: UI delete soft-deletes the package row while preserving its normalized name and owner key, and publish later resolves that row before throwing the reported ownership error on owner-key mismatch. I did not run a live production-row reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the intended policy. A maintainable fix should avoid accidental package takeover and make restore, transfer, hard delete, restore-on-publish, or permanent reservation explicit.
