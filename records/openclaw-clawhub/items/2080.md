## Summary

Keep open. Current main now has package delete/undelete and transfer surfaces, but this issue still needs maintainer production-record verification because the reported failure is the stale `ownerPublisherId` ownership branch, not just ordinary soft-delete recovery.

## What This Changes

The issue asks how an org owner can recover or republish a plugin package after UI deletion leaves the package name reserved or owned by a stale publisher record.

## Best Possible Solution

Use the shipped package undelete path when ownership matches; for this stale-owner case, staff should verify production package, release, publisher, and audit records, then apply or build an audited restore, transfer, identity repair, or hard-delete workflow that preserves namespace protections.

## Reproduction Assessment

Partly: source inspection shows the soft-delete-to-hidden-row path and the publish rejection branches on current main. The exact `belongs to another publisher` outcome depends on the production row's stored `ownerPublisherId`, which was not locally reproducible from the checkout.

## Solution Assessment

Keeping this open is the best review outcome. Current main has useful package undelete and transfer improvements, but silently republishing over a deleted or stale-owned package would be unsafe; an audited recovery decision is the narrower maintainable path.
