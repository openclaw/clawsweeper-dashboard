## Summary

Keep open: current main still reserves a soft-deleted package name, blocks republishing through the package insert path, and has no package restore/undelete path for org owners. ClawHub review policy also keeps issues open unless a maintainer resolves them explicitly.

## What This Changes

Request a supported recovery path or clearer delete semantics when a UI-deleted org-owned plugin package still reserves its package name and blocks republish.

## Best Possible Solution

Define package deletion semantics, then add one supported restore/ownership-repair path or explicit reserved-name warning/docs that org owners can rely on.

## Reproduction Assessment

Yes, at source level: delete a package through the dashboard/API, then publish the same normalized package name. Current main finds the soft-deleted package row by normalizedName and rejects the publish before a new release can be created.

## Solution Assessment

No current solution is complete: the publish path tells users to restore, but no package restore/undelete route exists and transfer/repair reject soft-deleted rows. A maintainable fix needs a deliberate namespace recovery policy before implementation.
