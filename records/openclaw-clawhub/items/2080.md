## Summary

Keep open: current main still hides soft-deleted package rows from public/dashboard reads while retaining the normalized package identity, and publish still rejects soft-deleted or owner-mismatched rows. The issue is not cleanup-ready because recovery semantics for live package namespaces need maintainer/admin policy and security review.

## What This Changes

The issue asks ClawHub to clarify and support recovery when a package deleted from the UI still reserves the package name or retains stale ownership that blocks future publishes.

## Best Possible Solution

Define and document package delete/recovery semantics, then expose a guarded owner/admin restore, transfer, repair, or hard-delete path that preserves namespace authority and auditability.

## Reproduction Assessment

Yes, at source level: soft-deleted package rows are hidden from public and dashboard reads but still exist by normalizedName, and publish rejects deleted or owner-mismatched rows. The specific production records still need admin inspection.

## Solution Assessment

Unclear as an automatic fix: existing restore, transfer, and admin repair primitives are safer than republish bypasses, but maintainers must choose whether deletion reserves names or supports verified release/reassignment.
