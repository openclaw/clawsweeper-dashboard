## Summary

Keep open: current main still has a source-visible gap between package soft-delete and package recovery, and ClawHub policy keeps issues open unless maintainers resolve them. Recent package ownership PRs improved scoped ownership and error text, but they do not provide a restore, transfer, or hard-delete path for soft-deleted package rows.

## What This Changes

The issue asks for a recovery path or clearer behavior when a UI-deleted org package still blocks publishing the same package name.

## Best Possible Solution

Define package soft-delete semantics and add a maintainer-supported recovery path, likely package undelete or audited admin repair that works for soft-deleted rows, with matching UI/CLI/API docs and support handling for the named package.

## Reproduction Assessment

Yes for the source-level workflow: dashboard/API delete sets softDeletedAt, hides the package from search, and publish blocks the same normalized package name afterward. The exact stale ownerPublisherId state needs maintainer access to production data to confirm.

## Solution Assessment

Unclear until maintainers choose the package deletion contract. The safest direction is not to silently release scoped package names, but to provide an audited restore or repair path that preserves namespace ownership.
