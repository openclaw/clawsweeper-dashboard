## Summary

Keep open: current main still soft-deletes plugin packages in a way that hides them from normal UI/search while preserving the package row and blocking republish, and the remaining recovery decision is ownership-sensitive maintainer work rather than a cleanup close.

## What This Changes

The issue asks for a supported recovery path when a UI-deleted org-owned plugin package still reserves its name or points at stale package ownership during publish.

## Best Possible Solution

Define package deletion semantics, then add an audited owner/admin restore or recovery path and update the UI/docs so package-name reservation after delete is explicit.

## Reproduction Assessment

Yes for the central blocked-republish behavior: current source shows dashboard/API deletion sets softDeletedAt and publish rejects the same normalized package name. The exact production ownership-mismatch message still needs data verification because current main now reaches the soft-deleted-package branch first.

## Solution Assessment

Unclear. Releasing the name by default is risky; an audited restore/undelete or admin recovery path is likely safer, but maintainers need to choose the intended delete and namespace-reservation policy.
