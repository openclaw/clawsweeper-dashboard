## Summary

Keep open: current main still reserves the deleted package row and blocks republishing the same normalized name, while the existing transfer and repair paths exclude soft-deleted packages.

## What This Changes

The issue asks for a maintainer-supported recovery path when a UI-deleted org package disappears from ClawHub but still blocks later publishes of the same package name.

## Best Possible Solution

Define package deletion semantics, then add an audited owner or admin recovery path and update the UI/docs so package name reservation is explicit.

## Reproduction Assessment

Yes, by source inspection: dashboard deletion sets softDeletedAt, public reads hide the row, and publishing the same normalized package name rejects the soft-deleted package. I did not run a live production publish, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear. Releasing deleted names by default may be unsafe; a restore or audited owner/admin recovery flow looks safer, but maintainers need to choose the intended deletion and namespace policy first.
