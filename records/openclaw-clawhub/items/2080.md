## Summary

Keep open: current main intentionally soft-deletes packages and hides them from public/read surfaces, but publishing the same name remains blocked and there is no package undelete/restore path for the reporter's recovery case.

## What This Changes

The issue asks for a recovery path or clearer semantics when a UI-deleted org-owned plugin package remains reserved and cannot be republished.

## Best Possible Solution

Define package delete semantics and add a maintainer-supported recovery path, likely an audited package undelete/restore flow plus UI/CLI/docs copy that explains whether names remain reserved.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: delete a package through the dashboard or package delete API, then publish the same normalized name; current main keeps the package row soft-deleted and rejects the new release. I did not run the authenticated production publish flow for the reporter's org package.

## Solution Assessment

Unclear; adding audited package undelete may be the narrowest maintainable recovery path, but maintainers need to choose whether deletion should reserve the name, release it, or remain irreversible with stronger warnings.
