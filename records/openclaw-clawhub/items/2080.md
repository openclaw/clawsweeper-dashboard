## Summary

Keep open. Current main still soft-deletes plugin package rows while preserving the package name and owner mapping, and the existing publish, transfer, and repair paths do not provide a safe self-service recovery for the reporter's deleted org-owned package.

## What This Changes

The issue asks for a supported recovery path when UI deletion hides an org-owned plugin package but later publishing the same package name is blocked by stale ownership or slug reservation state.

## Best Possible Solution

A ClawHub admin should inspect the production package record, choose the package lifecycle policy, then provide an audited restore, owner repair/transfer, hard-delete, or reclaim path with matching UI, CLI/API, docs, and tests.

## Reproduction Assessment

No for the exact reported org-owned package without production records and an authenticated publish attempt. Source inspection does show the current-main branches that reserve a soft-deleted package name and throw the reported ownership error when owner keys differ.

## Solution Assessment

Unclear until maintainers choose recovery semantics. The safest direction is an audited restore or owner-repair flow, not allowing publish to overwrite a deleted or mismatched executable package record implicitly.
