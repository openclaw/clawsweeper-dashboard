## Summary

Keep open. Current main still treats package deletion as soft-delete-only, while publish, transfer, and repair paths can leave a deleted or owner-mismatched package name blocked; recovery needs maintainer/admin policy and production record inspection.

## What This Changes

The issue asks how org owners can recover, transfer, restore, hard-delete, or understand a package name after UI deletion leaves the package slug blocked.

## Best Possible Solution

Define an audited package recovery lifecycle, then provide maintainer/admin tooling or owner-facing restore/transfer/reclaim flows with matching UI, CLI/API, docs, specs, and tests.

## Reproduction Assessment

No for the exact production package: that needs production records and an authenticated publish attempt. Source inspection does show current-main branches that reserve soft-deleted package names and throw the reported ownership error when owner keys differ.

## Solution Assessment

Unclear until maintainers choose recovery semantics. The safer direction is an audited restore, transfer, owner repair, reclaim, or hard-delete path rather than implicit publish overwrite of deleted or mismatched executable package records.
