## Label Justifications

- `P1`: This blocks real package publishers from releasing current plugin versions and affects a security-sensitive registry ownership path.
- `impact:security`: Package slug and publisher ownership recovery controls who can publish executable plugin packages under a namespace.

## Summary

Keep open: current main has package soft-delete and undelete primitives, but it still preserves the global package record and owner mapping, blocks publishing against soft-deleted or mismatched-owner package rows, and does not provide a settled recovery policy for stale package ownership.

## What This Changes

The issue asks for a supported recovery path when a deleted ClawHub package remains reserved or mapped to a stale publisher and future publishes fail.

## Best Possible Solution

Define an RFC-backed package namespace recovery policy, then provide an audited admin or owner-facing flow for restore, owner transfer, hard-delete, or name release with explicit authorization checks.

## Reproduction Assessment

Yes, at source level: a soft-deleted package is hidden from readable lookups but still found by normalizedName, and publish/transfer/repair paths reject deleted or mismatched-owner package rows. I did not run a live production publish because the exact stale owner mapping requires live records.

## Solution Assessment

No, the current undelete path alone is not the best complete solution for this issue because it only covers recoverable ownership states. The safer path is an explicit, audited recovery decision and implementation for stale package owner mappings.
