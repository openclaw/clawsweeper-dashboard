## Summary

Keep open under the ClawHub issue policy; current main now gives a clearer deleted-package publish error, but deleted package names remain reserved and there is still no owner-accessible restore, undelete, or stale-owner recovery path.

## What This Changes

Request a maintainer-supported recovery or clear reservation policy for org-owned plugin packages that were deleted in the UI but remain blocked on republish.

## Best Possible Solution

Define package deletion semantics, then ship an explicit recovery path and matching dashboard/API/CLI wording that makes reserved names and restore or transfer behavior clear.

## Reproduction Assessment

Yes, as a source-level path: dashboard delete soft-deletes the package while preserving the name and owner fields, and publish still resolves that row by normalizedName before rejecting it. I did not run a live production-row reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the intended policy. The narrow maintainable direction is an explicit restore/undelete or staff-supported recovery flow, not silently creating a second package identity or bypassing ownership checks.
