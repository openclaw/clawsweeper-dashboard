## Summary

Keep open. Current main shows package deletion is a soft-delete that removes packages from public/UI views while retaining the globally named package row, but the exact production ownership mismatch behind the reported publish error still needs maintainer verification and a supported recovery decision.

## What This Changes

The issue asks how an org package owner can recover or reuse a code-plugin package name after dashboard deletion leaves publish blocked by ownership or slug reservation.

## Best Possible Solution

Keep the issue open, verify the production package row, and provide an audited maintainer recovery path while the broader owner-facing lifecycle UX remains tracked at https://github.com/openclaw/clawhub/issues/1370.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection reproduces the core reservation behavior where dashboard delete soft-deletes a package but publish still consults the retained normalizedName row. The exact “belongs to another publisher” outcome after deletion depends on production ownerPublisherId state that was not publicly inspectable.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the recovery semantics. The safest direction is an audited restore, owner repair, or hard-delete path after production verification, not silent package-name reuse during publish.
