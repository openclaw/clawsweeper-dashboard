## Summary

Keep open: current main has clearer scoped-owner and soft-delete publish errors plus an active-package transfer flow, but deleted package rows still reserve the normalized name and there is no package undelete, restore, or hard-delete recovery path for the reporter’s case.

## What This Changes

The issue requests a supported way for org owners to republish or recover a plugin package after UI deletion leaves its package name or ownership mapping reserved.

## Best Possible Solution

Define package delete and reservation semantics, then add an audited package undelete, owner repair/transfer, or hard-delete recovery path and align the UI/API/CLI wording with that policy.

## Reproduction Assessment

Yes, as a source-level path: package deletion soft-deletes the row, publishing resolves the same normalizedName and rejects it, and the new transfer flow rejects soft-deleted packages. I did not run a live production-row reproduction in this read-only review.

## Solution Assessment

No, current main is only a partial solution: it improves ownership validation and transfer for active packages, but a deleted package still cannot be restored, transferred, or reclaimed. The maintainable fix needs an explicit, audited recovery policy before code changes.
