## Summary

Keep open. Current main still treats package deletion as a soft-delete-only lifecycle action, while publish, transfer, and repair paths can leave a deleted or owner-mismatched package name blocked; the reported org-owned package needs maintainer/admin recovery policy and production record inspection.

## What This Changes

The issue asks for a supported recovery or clearer lifecycle policy when an org-owned plugin package is deleted from the UI but the package name remains reserved during republish.

## Best Possible Solution

A maintainer should inspect the production package record, decide the deleted-package lifecycle policy, then provide an audited restore, owner repair/transfer, hard-delete, or reclaim path with matching UI, CLI/API, docs, specs, and tests.

## Reproduction Assessment

No for the exact reported org-owned package: that needs production records and an authenticated publish attempt. Source inspection does show the current-main branches that reserve soft-deleted package names and throw the reported ownership error when owner keys differ.

## Solution Assessment

Unclear until maintainers choose recovery semantics. The safer direction is an audited restore, transfer, owner repair, or hard-delete flow rather than letting publish implicitly overwrite a deleted or mismatched executable package record.
