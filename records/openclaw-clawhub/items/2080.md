## Summary

Keep open: current main still makes deleted plugin packages disappear from UI/search while preserving the package row/name reservation, and there is no owner-facing package restore or recovery path for republishing the same name.

## What This Changes

The issue asks how an org owner can recover or republish a plugin package after UI deletion leaves the package name blocked by stale ownership or slug reservation state.

## Best Possible Solution

Define package deletion semantics, then add an audited owner/admin package restore or recovery flow and make the UI/CLI/docs explicit about whether deletion reserves names.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: dashboard/CLI package delete sets softDeletedAt, read surfaces hide that package, and publish rejects the same normalizedName; the exact stale production owner mapping still needs live data verification.

## Solution Assessment

Unclear. Releasing names automatically is not obviously safe; an audited restore/undelete or staff recovery path with strong ownership checks is the safer maintainable direction.
