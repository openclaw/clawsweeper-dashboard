## Summary

Keep open: current main still hides soft-deleted package rows from reads while retaining the global package identity and owner mapping, and publish still rejects deleted or mismatched-owner rows. This remains a live package namespace recovery and policy issue, not a conservative cleanup close.

## What This Changes

The issue asks for a supported recovery or semantics path when deleting a ClawHub plugin package hides it but leaves the package name or owner mapping blocked for future publishes.

## Best Possible Solution

Define and document package delete/recovery semantics, then use audited owner/admin restore, transfer, repair, or hard-delete flows only after verifying package namespace authority.

## Reproduction Assessment

Yes, at source level: current main hides soft-deleted packages from readable/public paths while retaining the global normalizedName row, and publish rejects soft-deleted or different-owner rows. A live production reproduction still needs admin inspection of the affected records.

## Solution Assessment

Unclear as an automatic fix: existing restore, transfer, and admin repair primitives are safer than a republish bypass, but maintainers must choose whether deletion reserves names or supports verified release/reassignment.
