## Summary

Keep open under the ClawHub issue policy and because current main still has unresolved package-delete recovery semantics: deletion soft-deletes the package row, while publish continues to claim that row by normalized name before owner checks.

## What This Changes

The issue asks how org owners can recover or republish a package name after UI deletion leaves the visible package hidden but publish blocked by stale ownership or slug reservation state.

## Best Possible Solution

Choose the package deletion/name-reservation policy, then update Convex mutations, dashboard/API/CLI wording, and regression tests so deleted package names either remain intentionally reserved with a clear recovery path or can be safely republished by the rightful owner.

## Reproduction Assessment

Yes, as a source-level path: UI/API deletion leaves a soft-deleted package row, and publish still resolves that row by normalized name before enforcing the stored owner key. I did not run a live Convex reproduction against the reporter's production row.

## Solution Assessment

Unclear until maintainers choose the intended policy. The maintainable solution should avoid silently creating a second package identity and make restore, transfer, hard delete, or permanent reservation explicit.
