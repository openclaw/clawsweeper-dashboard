## Summary

Keep open: this is a current package-lifecycle and production recovery problem, not a cleanup close candidate. Current main still soft-deletes package rows, hides them from public package surfaces, and blocks publishing the same normalized package name again unless maintainers add or run an explicit recovery path; the broader lifecycle request at https://github.com/openclaw/clawhub/issues/1370 remains open and does not resolve this concrete case yet.

## What This Changes

The issue asks how org owners can recover or republish a plugin package after UI deletion leaves the package name or ownership mapping reserved.

## Best Possible Solution

Define package deletion and reservation semantics, then add or run an audited owner/admin recovery path for soft-deleted plugin packages after verifying control of the target org publisher.

## Reproduction Assessment

Yes, from source inspection. Dashboard/API deletion sets softDeletedAt, public surfaces hide the package, and publish rejects the existing normalized name; I did not run a live production publish against the reporter's specific row.

## Solution Assessment

Unclear. Releasing the package name by default is not clearly safe; an audited restore, transfer, owner repair, or admin recovery flow is the safer likely direction, but maintainers need to choose the lifecycle policy first.
