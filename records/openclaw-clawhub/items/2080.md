## Summary

Keep open: current main still has a real package-lifecycle recovery gap. UI/API deletion soft-deletes plugin packages and hides them from public package surfaces, while the publish path continues to reserve the normalized package name and the existing transfer paths do not recover soft-deleted rows; the related package lifecycle request at https://github.com/openclaw/clawhub/issues/1370 is still open and broader than this concrete production recovery case.

## What This Changes

The issue asks for a supported recovery path when a plugin package deleted from the UI remains reserved or stale-owned and blocks publishing the same package name again.

## Best Possible Solution

Define package deletion and reservation semantics, then provide an audited owner/admin recovery path for soft-deleted plugin packages after verifying control of the affected org publisher.

## Reproduction Assessment

Yes, from source inspection. Dashboard/API deletion sets softDeletedAt, public package surfaces hide the package, and publish/transfer paths continue to reject the existing normalized name; I did not run a live production publish against the reporter's specific row.

## Solution Assessment

Unclear. Releasing the package name automatically is not clearly safe; an audited restore, owner repair/transfer, or maintainer hard-delete flow is the safer likely direction, but maintainers need to choose the lifecycle policy first.
