## Summary

Keep open: current main still documents and enforces package deletion as a soft-delete that hides the package but reserves the package name, and this recent issue asks for a maintainer recovery policy for an affected production package.

## What This Changes

The issue asks how an org owner can recover or republish a package after deleting it in the ClawHub UI leaves the normalized package slug reserved.

## Best Possible Solution

Define package delete semantics, then add an audited owner/admin restore or recovery path that preserves namespace protections and documents whether deleted package names remain reserved.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a package through the dashboard or package DELETE API, then publish the same normalized package name and reach the soft-deleted package rejection branch while UI/read surfaces hide the package.

## Solution Assessment

Unclear. A restore or undelete flow looks like the safest likely direction, but maintainers first need to decide whether deletion reserves names, releases names, supports hard-delete, or remains irreversible with clearer UI/docs warnings.
