## Summary

Keep open under the ClawHub issue policy. Current main now fails soft-deleted package republishes with a clearer deleted-package message, but the central recovery problem remains: UI/CLI delete preserves the package row and owner keys, and there is no package undelete or deleted-package transfer path for the reporter’s org-owned package.

## What This Changes

The issue asks for a recovery or explicit reservation policy when an org-owned package is soft-deleted from the UI and republishing the same name is blocked.

## Best Possible Solution

Define the package deletion policy and add an audited recovery path, or make permanent name reservation explicit in the dashboard, CLI, HTTP docs, and publish error copy.

## Reproduction Assessment

Yes, as a source-level path: dashboard delete calls package soft-delete, the global name lookup still resolves the soft-deleted row, and publish rejects it before creating a new release. I did not run a live production publish, but the reporter confirmed the same user-visible case.

## Solution Assessment

Unclear until maintainers choose the policy. Current main’s clearer error is helpful but incomplete because package restore is not implemented; the maintainable fix should make restore, transfer/repair, hard delete, or permanent reservation explicit.
