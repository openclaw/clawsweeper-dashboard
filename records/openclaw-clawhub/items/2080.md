## Summary

Keep open: this is still an active package namespace recovery/admin problem, and ClawHub cleanup policy does not auto-close issues. Current main still hides soft-deleted packages while retaining the global package row and owner mapping, and publish still rejects soft-deleted or different-owner rows.

## What This Changes

The issue asks for a supported recovery path when package UI deletion or stale ownership hides a plugin package but leaves its package name or publisher mapping reserved so later publishes fail.

## Best Possible Solution

Define and document package delete/recovery semantics, then use audited admin or owner flows to restore, transfer, repair, or hard-delete verified package rows under that policy.

## Reproduction Assessment

Yes, at source level: current main hides soft-deleted packages from readable/public paths while retaining the global normalizedName row, and the publish path rejects soft-deleted or different-owner package rows. The exact production records still require admin inspection.

## Solution Assessment

Unclear as an automatic fix: current main has restore, transfer, and admin repair primitives, but choosing whether deletion releases names or preserves reserved namespaces is a maintainer product and security decision.
