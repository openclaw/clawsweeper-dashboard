## Summary

Keep open: current main improves ownership errors and active-package transfer, but deleted package rows still reserve the global package name and there is no restore, reclaim, or hard-delete recovery path for the reporter's deleted package.

## What This Changes

Org package owners need a recovery path or clearer delete semantics when UI deletion hides a plugin but the package name remains reserved.

## Best Possible Solution

Define package delete and reservation semantics, then implement an audited restore, owner repair/transfer, or hard-delete recovery path and align the UI/API/CLI wording with that policy.

## Reproduction Assessment

Yes, as a source-level path: dashboard/API deletion soft-deletes the package row, package publish resolves that same normalizedName and rejects it, and the transfer path rejects soft-deleted packages. I did not mutate production data or run a live production-row reproduction.

## Solution Assessment

No, current main is only a partial solution: active wrong-owner packages can be transferred, but deleted packages still cannot be restored, transferred, or reclaimed. The maintainable fix needs an explicit audited recovery policy first.
