## Summary

Keep open: ClawHub issue cleanup policy keeps issues open, and current main still has the central problem that UI/API package delete hides a soft-deleted package while preserving the normalized package record and blocking republish until a package recovery policy exists.

## What This Changes

Request a maintainer-supported recovery or clearer deletion policy when an org-owned plugin package is deleted in the UI and the package name remains reserved or stale-owned during republish.

## Best Possible Solution

Define package delete and recovery semantics, then add an audited owner/admin restore or recovery flow and update UI, CLI, API docs, and tests so package-name reservation behavior is explicit.

## Reproduction Assessment

Yes for the central failure path. Source inspection shows dashboard delete soft-deletes the package, dashboard/search hide it, and publish rejects the existing soft-deleted normalized package name; I did not run a live production publish or inspect the reporter's production row.

## Solution Assessment

Unclear until maintainers choose the product and security policy. A restore/undelete path with ownership checks and audit logging looks like the safest likely direction, but hard-delete or name-release semantics need explicit maintainer approval.
