## Summary

Keep open under the ClawHub issue policy. Current main partially improves the symptom by rejecting soft-deleted package publishes with an explicit deleted-package message and by adding active-package transfer support, but deleted package rows still preserve the package name and owner keys, and there is no restore, hard-delete, or deleted-row transfer recovery path for the reporter’s state.

## What This Changes

Request that package deletion and publishing define a recovery or reuse path when an org-owned plugin is deleted in the UI but later cannot be republished under the same package name.

## Best Possible Solution

Define package soft-delete reservation semantics, then add an explicit restore/undelete, admin hard-delete/reclaim, or deleted-package repair path and align dashboard, HTTP API, CLI, and docs with that policy.

## Reproduction Assessment

Yes, as a source-level current-main path: dashboard deletion sets softDeletedAt while preserving normalizedName and owner IDs, and publishing resolves the same by_name row before rejecting the publish. I did not run a live production Convex-row reproduction.

## Solution Assessment

No, not yet. Current main improves the error and adds transfer for active packages, but transfer rejects soft-deleted packages and there is no documented restore, undelete, hard-delete, or reclaim flow for the reported deleted package state.
