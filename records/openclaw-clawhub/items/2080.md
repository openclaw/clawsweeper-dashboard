Keep open. Current main has package soft-delete, undelete, transfer, and admin repair primitives, but the reported `ownerPublisherId` mismatch for `@yahaha-studio/kichi-forwarder` still needs production-row inspection and maintainer/admin recovery rather than conservative cleanup closure.

## What This Changes

The issue asks how org owners can recover a plugin package name that disappeared after UI deletion but still fails publish with a stale ownership or slug-reservation error.

## Best Possible Solution

Have a ClawHub admin verify the live package row and audit logs, then use the existing audited undelete, transfer, or identity repair path, adding UI/docs or reuse semantics only after the package namespace policy is explicit.

## Reproduction Assessment

No high-confidence reproduction for the exact production row without admin data access. Source inspection does reproduce the generic lifecycle: soft-deleted package names are hidden from public reads while retained in the table, and publish rejects either deleted rows or mismatched owner keys.

## Solution Assessment

Yes for keeping this open and routing it to maintainer/admin review. Current restore, transfer, and repair primitives are safer than letting publish implicitly reclaim deleted or mismatched names, but the affected row must be inspected before choosing one.
