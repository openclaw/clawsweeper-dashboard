## Summary

Keep open under the ClawHub issue policy; source inspection and reporter follow-up show current main preserves deleted package rows and ownership keys, so maintainers need to define recovery and reservation semantics.

## What This Changes

Request to define and provide a recovery path when a UI-deleted org-owned package remains reserved and blocks republishing under the same package name.

## Best Possible Solution

Define package delete semantics, then implement an explicit restore/undelete, transfer/repair, or hard-delete recovery path and align the dashboard/API wording with that policy.

## Reproduction Assessment

Yes, as a source-level path: UI delete leaves the package row soft-deleted, while publish still resolves that row by normalizedName and enforces the stored owner key. I did not run a live Convex production-row reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the intended policy. A maintainable fix should avoid silently creating a second package identity and should make restore, transfer, hard delete, or permanent reservation explicit.
