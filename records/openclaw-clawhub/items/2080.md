## Summary

Keep open: current main still preserves soft-deleted package records, hides them from read/UI surfaces, and blocks republishing the normalized package name until maintainers define a recovery or reservation policy. The reporter confirmed this source path matches their production case, so this needs maintainer follow-up rather than cleanup closure.

## What This Changes

The issue asks how org owners can recover or republish a package after deleting it in the UI when the hidden soft-deleted package row still blocks the package name.

## Best Possible Solution

Define package deletion semantics and add an audited owner/admin recovery path that preserves namespace protections while documenting whether deleted package names remain reserved.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a package through the dashboard or package DELETE API, then publish the same normalized package name and hit the reserved soft-deleted package row while UI/read paths hide it.

## Solution Assessment

Unclear. A restore or recovery flow is likely the safest direction, but maintainers need to decide whether package deletion should reserve names, release names, permit hard-delete, or remain irreversible with clearer warnings.
