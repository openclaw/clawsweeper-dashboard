Keep open: this is a recent ClawHub issue, and current main still preserves soft-deleted package records while blocking future publishes for the same normalized package name until a recovery or restore policy exists.

## What This Changes

The issue asks for a package recovery path or clearer deletion semantics after a UI-deleted org package remains reserved and blocks republish.

## Best Possible Solution

Define package delete semantics, then add an audited owner/admin restore or recovery path that preserves namespace protections and documents whether package names remain reserved after deletion.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a package through the dashboard or package DELETE API, then publish the same normalized package name and reach the existing soft-deleted package rejection branch while UI/read surfaces hide the package.

## Solution Assessment

Unclear. A restore or undelete flow looks like the safest likely implementation, but maintainers first need to decide whether deletion reserves names, releases names, supports hard-delete, or remains irreversible with stronger UI/docs warnings.
