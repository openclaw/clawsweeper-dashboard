## Summary

Keep open. Current main still soft-deletes package rows without exposing package restore/reclaim, and publish plus transfer/repair paths can block a deleted or mismatched package; the named org-owned package needs admin inspection and a package lifecycle policy decision.

## What This Changes

The issue asks for recovery from a UI-deleted org-owned plugin package whose package name remains blocked during republish.

## Best Possible Solution

A maintainer should inspect the production package record, choose the package lifecycle policy, then provide an audited restore, owner repair, transfer, hard-delete, or reclaim path with matching UI, CLI/API, docs, and tests.

## Reproduction Assessment

No, not for the exact reported org-owned package without production records and an authenticated publish attempt. Source inspection does show the current-main branches that reserve a soft-deleted package name and throw the reported ownership error when owner keys differ.

## Solution Assessment

Unclear until maintainers choose recovery semantics. The safest direction is an audited restore or owner-repair flow, not allowing publish to overwrite a deleted or mismatched executable package record implicitly.
