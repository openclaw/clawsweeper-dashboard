## Summary

Keep open: ClawHub cleanup policy keeps issues open, and current main still creates a stranded package state after UI/API delete: the package is hidden, the namespace row remains, republish is blocked, and transfer does not work for soft-deleted packages.

## What This Changes

The issue requests a maintainer-supported recovery path when a UI-deleted plugin package remains reserved and later publishes fail with a stale ownership or slug reservation error.

## Best Possible Solution

Define package lifecycle semantics, then add an audited owner/admin recovery flow that can restore, transfer, or deliberately release a soft-deleted package without weakening namespace ownership checks.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a package through the dashboard or package DELETE API, then publish the same normalized package name and hit the soft-deleted package rejection branch while the package is hidden from normal read surfaces.

## Solution Assessment

Unclear. Restore/undelete looks like the safest likely direction, but maintainers first need to decide whether package deletion reserves names, releases names, supports hard-delete, or remains irreversible with clearer warnings.
