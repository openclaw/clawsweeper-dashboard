## Summary

Keep open. Current main still has the reported recovery gap: package delete hides the plugin package but preserves the package row, and publishing continues to resolve and reject the existing normalized package name; this needs a namespace- and ownership-safe recovery policy, not cleanup closure.

## What This Changes

The issue asks for a supported recovery path when a ClawHub plugin package is deleted from the UI but the preserved package row still blocks publishing the same package name.

## Best Possible Solution

Define package deletion and reservation semantics, then provide an audited restore, transfer, owner repair, or hard-delete recovery path after verifying control of the affected org publisher and package row.

## Reproduction Assessment

Yes. From source inspection, dashboard deletion soft-deletes the package, public surfaces hide soft-deleted packages, and publishing still checks the preserved normalizedName row; I did not run a live production publish against the reporter's package.

## Solution Assessment

Unclear. Automatically releasing the package name on delete is not clearly safe; the narrower maintainable direction is an explicit audited recovery policy for restore, transfer, owner repair, or hard delete.
