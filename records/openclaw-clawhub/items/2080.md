## Summary

Keep open under the ClawHub issue policy; current main still preserves soft-deleted package identities while publish enforces the stored owner key, so the recovery and reservation policy remains unresolved.

## What This Changes

The issue asks for a maintainer-supported recovery or clearer reservation behavior when deleting an org-owned code-plugin package blocks republishing the same package name.

## Best Possible Solution

Define package delete semantics, then implement an explicit restore/undelete, transfer/repair, hard-delete, or permanent-reservation path and align dashboard/API/CLI wording with that policy.

## Reproduction Assessment

Yes, as a source-level path: UI/API delete soft-deletes the package row, while publish still resolves that row by normalizedName and enforces the stored owner key. I did not run a live production-row reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the intended reservation and recovery policy. Silently creating a second package identity would be risky; the maintainable path should make restore, transfer, hard delete, or permanent reservation explicit.
