## Summary

Keep open. Current main still has a package lifecycle gap: soft-deleted packages retain their package identity and ownership state, while publish blocks deleted names and owner-key mismatches instead of offering an owner recovery path.

## What This Changes

The issue requests maintainer-supported recovery or clearer deletion semantics when a UI-deleted org-owned package still blocks publishing the same package name.

## Best Possible Solution

Define and implement an audited package recovery lifecycle: owner/admin restore, verified ownership repair or transfer, and possibly hard-delete/reclaim semantics, with matching UI, CLI/API, docs, specs, and regression tests.

## Reproduction Assessment

No for the exact org-owned package, because that requires production records and an authenticated publish attempt. Source inspection and tests do show the current-main branches that reserve soft-deleted package names and throw the reported ownership error on owner-key mismatch.

## Solution Assessment

Unclear until maintainers choose deleted-package lifecycle semantics. The safer direction is an audited restore, transfer, owner repair, or hard-delete flow rather than allowing publish to implicitly overwrite a deleted or mismatched executable package record.
