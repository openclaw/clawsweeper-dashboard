## Summary

Keep open: current main still soft-deletes plugin packages, hides them from visible surfaces, and blocks publishing the same normalized name without exposing an owner-facing restore or recovery path. The related package lifecycle request remains open, but this report has concrete production recovery context that is not superseded.

## What This Changes

The issue asks for a recovery path or clearer semantics when an org-owned plugin package is deleted in the UI but its package name remains reserved during publish.

## Best Possible Solution

Define package delete and restore semantics, verify the specific @yahaha-studio/kichi-forwarder production row, then add an audited owner/admin restore or owner-repair path with UI and docs that make package name reservation explicit.

## Reproduction Assessment

Yes for the central current-main behavior: source inspection shows dashboard deletion sets softDeletedAt, public reads hide the package, and publish rejects the same normalized name. The exact production ownerPublisherId mismatch needs maintainer data access to verify.

## Solution Assessment

Unclear. Releasing deleted names by default is risky; an audited restore or owner-repair flow is likely safer, but maintainers need to choose the lifecycle policy before implementation.
