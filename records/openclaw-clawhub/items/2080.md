## Summary

Keep open under the ClawHub issue policy; current main still preserves deleted package rows and ownership keys, publish still resolves those rows by global package name, and the reporter confirmed that this matches their production failure.

## What This Changes

The issue requests a maintainer-supported recovery path and clear policy for org-owned package names that remain reserved after UI package deletion blocks republishing.

## Best Possible Solution

Define package deletion semantics, then implement an explicit package restore/undelete, soft-deleted transfer/repair, or hard-delete recovery path and align dashboard, CLI, and API wording with that policy.

## Reproduction Assessment

Yes, as a source-level path: UI/API delete leaves the package row soft-deleted while publish resolves that row by normalizedName and enforces the stored owner key. I did not prove the reporter's private production row state in this read-only review.

## Solution Assessment

Unclear until maintainers choose the package-name reservation policy. A maintainable fix should avoid silently creating a second package identity and make restore, transfer/repair, hard delete, or permanent reservation explicit.
