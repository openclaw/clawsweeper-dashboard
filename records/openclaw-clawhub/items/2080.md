## Summary

Keep open: source inspection confirms package rows are retained after delete and publish remains fail-closed for soft-deleted or wrong-owner package records, while the requested recovery path still needs maintainer/security policy rather than an automatic code cleanup. The related lifecycle issue at https://github.com/openclaw/clawhub/issues/1370 added package delete/undelete primitives, and the @xquik follow-up now appears to have a public owner of `xquik`, but this issue still needs a supported policy for stale owner or deleted-package recovery.

## What This Changes

The reporter asks for supported recovery and clear delete semantics when package publishing is blocked by stale package ownership or slug reservation after a plugin package is deleted from the UI.

## Best Possible Solution

Define and document the package recovery policy, then expose an audited restore/transfer/admin-repair path that preserves namespace authorization and makes soft-delete reservation behavior clear before deletion.

## Reproduction Assessment

Yes, source-level: an existing package with the same normalized name and a different owner key throws the reported ownership error, while a soft-deleted package is hidden from dashboard listings but still blocks reuse until restored. I did not run a live authenticated publish against production.

## Solution Assessment

Unclear until maintainers choose policy: an automatic hard-delete or name release would be unsafe for package namespaces. The maintainable direction is an audited restore/transfer/repair flow plus clear UI and docs for soft-delete reservation semantics.
