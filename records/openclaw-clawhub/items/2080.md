## Summary

Keep open. Current main intentionally soft-deletes package rows and keeps their normalized names reserved, while the exact stale owner mismatch behind this report still needs production record verification and a maintainer recovery decision; ClawHub cleanup policy also keeps issues open.

## What This Changes

The issue asks how org owners can recover when a plugin package deleted from the UI disappears publicly but later publishing the same package name fails with a stale ownership or slug-reservation error.

## Best Possible Solution

Have a ClawHub maintainer inspect the production package, release, publisher, and audit-log records, then use or build an audited restore, owner repair/transfer, or hard-delete path while coordinating self-service lifecycle semantics with https://github.com/openclaw/clawhub/issues/1370.

## Reproduction Assessment

Partly: source inspection gives a clear current-main path for UI delete hiding a package while leaving the normalized package row reserved. The exact reported `belongs to another publisher` outcome depends on production ownerPublisherId state that was not available in the checkout.

## Solution Assessment

Yes for keeping this open, but unclear for the eventual fix until maintainers choose recovery semantics. The safer direction is an audited recovery or repair workflow, not silent package-name reuse during publish.
