## Summary

Keeping this open for maintainer follow-up: current main still soft-deletes package rows, hides them from UI/search, and blocks same-name publishing without exposing a package restore or owner recovery flow. Recent merged work improved scoped ownership validation and the soft-deleted publish error, but it does not resolve the reporter's requested recovery path.

## What This Changes

The issue asks how org owners can recover or republish a package name after deleting a plugin in the UI when the backend still treats the package name as owned or reserved.

## Best Possible Solution

Define package deletion as a recoverable soft-delete operation, add a package restore or maintainer repair path for owners/admins, and update UI/CLI copy so users know package names remain reserved until restored or repaired.

## Reproduction Assessment

Yes from source inspection: package delete tombstones the package and removes it from visible lists, while publish still resolves the normalized-name row and refuses to create a replacement. I did not run a live Convex reproduction because this review is read-only.

## Solution Assessment

Unclear. A package undelete/recovery flow plus clearer delete copy looks like the safest direction, but maintainers should first confirm whether deleted package names are meant to stay reserved or become reclaimable.
