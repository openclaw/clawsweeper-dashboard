## Summary

Keep open: current main still soft-deletes package rows, hides them from public package surfaces, and blocks republishing the same normalized name unless maintainers add or run a recovery path; the broader lifecycle request at https://github.com/openclaw/clawhub/issues/1370 remains open and does not resolve this concrete production recovery case.

## What This Changes

The issue asks for a recoverable package lifecycle after an org-owned plugin is deleted in the UI and republishing the same package name remains blocked by stale ownership or reservation state.

## Best Possible Solution

Define package deletion and reservation semantics, then provide an audited owner/admin recovery path for soft-deleted plugin packages after verifying control of the target org publisher.

## Reproduction Assessment

Yes, from source inspection: dashboard/API deletion sets softDeletedAt, public package conversion hides the row, and release insertion rejects the existing normalized package name. I did not run a live production publish against the reporter's specific record.

## Solution Assessment

Unclear. Releasing deleted package names by default is not clearly safe; an audited restore, transfer, owner repair, or admin hard-delete path is the safer likely direction, but maintainers need to choose the lifecycle policy first.
