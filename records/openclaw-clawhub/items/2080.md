## Summary

Keep open: this issue is an active ownership-sensitive package recovery problem on current main, and repository policy for ClawHub issue cleanup does not allow auto-closing issues even when part of the behavior is understood.

## What This Changes

The issue requests a supported recovery path when a UI-deleted plugin package disappears from public surfaces but its package slug or owner mapping still blocks republishing.

## Best Possible Solution

Define package deletion and slug-reservation semantics, then add an audited restore, owner repair/transfer, or hard-delete recovery path after verifying control of the affected org publisher.

## Reproduction Assessment

Yes, from source inspection: the dashboard deletion path soft-deletes the package, public package conversion hides soft-deleted rows, and the publish path still checks the existing normalized package row. I did not run a live production publish against the reporter's exact package.

## Solution Assessment

Unclear. Releasing the package name automatically on delete is not clearly safe; the maintainable fix is an explicit, audited recovery policy for restore, transfer, owner repair, or hard delete.
