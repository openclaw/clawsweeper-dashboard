## Summary

Keep open: current main still soft-deletes package rows while retaining the global package identity and owner mapping, and publish still rejects deleted or mismatched-owner rows. This remains a live package namespace recovery issue that needs maintainer/admin policy and security review rather than conservative issue cleanup.

## What This Changes

Package owners report that deleting a plugin package from the UI hides it but leaves the package name or owner mapping reserved, so later publishes of the same package fail.

## Best Possible Solution

Define and document package delete/recovery semantics, then use audited owner/admin restore, transfer, repair, or hard-delete flows only after verifying package namespace authority.

## Reproduction Assessment

Yes, at source level: current main hides soft-deleted packages from readable/public paths while retaining the global normalizedName row, and the publish path rejects soft-deleted or different-owner rows. The exact production records still require admin inspection.

## Solution Assessment

Unclear as an automatic fix: existing restore, transfer, and admin repair primitives are safer than republish bypasses, but maintainers must choose whether deletion reserves names or supports verified release/reassignment.
