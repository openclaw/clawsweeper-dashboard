## Summary

Keep open under the ClawHub issue policy and because current main still lacks a defined recovery path: dashboard/API delete soft-deletes packages, publish still checks the preserved row and owner key, and the existing repair helpers reject soft-deleted packages.

## What This Changes

Org-owned code-plugin deletion leaves the package name and ownership state reserved, blocking republish and prompting a request for recovery semantics after dashboard delete.

## Best Possible Solution

Define and document package delete semantics, then add an audited restore/undelete, transfer/repair, or hard-delete recovery path for soft-deleted packages without weakening package ownership checks.

## Reproduction Assessment

Yes, as a source-level path: UI delete leaves a soft-deleted package row with its name and owner fields intact, while publish resolves that row and enforces the preserved owner key. I did not run a live production-row reproduction.

## Solution Assessment

Unclear until maintainers choose the intended policy. The safest direction is an explicit audited recovery path rather than silently creating a second package identity or bypassing ownership checks.
