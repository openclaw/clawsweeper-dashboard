## Summary

Keep open: current main still matches the central blocked-republish behavior after package soft delete, and there is no owner-facing package restore or recovery flow. The remaining decision is security-sensitive package lifecycle policy plus production ownership verification, not conservative cleanup.

## What This Changes

A package owner reports that deleting an org-owned plugin from the UI hides it but leaves the package name reserved, blocking republish and requiring restore, transfer, or hard-delete recovery.

## Best Possible Solution

Define package deletion and namespace-reservation semantics, then add an audited owner/admin restore or recovery path with UI/CLI/docs that make reservation behavior explicit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: dashboard/API delete sets softDeletedAt and hides the package, while publish rejects the same normalized name before a new release can be inserted.

## Solution Assessment

Unclear. A restore or audited admin recovery flow is likely safer than releasing names by default, but maintainers need to choose the package lifecycle policy before implementation.
