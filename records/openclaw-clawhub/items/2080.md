## Summary

Keep open: current main still treats package deletion as a soft delete, hides the package from read/UI surfaces, and blocks republishing the same normalized package name until a restore or recovery policy exists. The remaining work is ownership-sensitive and needs maintainer policy and production verification.

## What This Changes

Request a recovery or deletion policy for org-owned package names that remain reserved after dashboard deletion blocks republishing.

## Best Possible Solution

Define package delete semantics, then add an audited owner/admin restore or recovery path that preserves namespace protections and documents whether package names remain reserved after deletion.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a package through the dashboard or package DELETE API, then publish the same normalized package name and hit the soft-deleted package rejection while read/UI surfaces hide the package.

## Solution Assessment

Unclear. A restore or undelete flow is likely the safest direction, but maintainers need to decide whether deletion reserves names, releases names, supports hard-delete, or remains irreversible with stronger UI/docs warnings.
