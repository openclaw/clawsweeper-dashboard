## Summary

Keep open. Current main improves the old ownership failure by rejecting soft-deleted package publishes with an explicit restore-before-publish message, but it still does not provide the requested restore, transfer, hard-delete, or owner-repair recovery path.

## What This Changes

The issue asks how org owners can recover when a UI-deleted package disappears from ClawHub but the same package name remains blocked during later publish attempts.

## Best Possible Solution

Define deleted-package reservation semantics, then add an owner-safe recovery path and update dashboard, CLI, API docs, and tests to make that policy explicit.

## Reproduction Assessment

Yes, as a source-level path: dashboard delete soft-deletes the package while preserving the global name and owner fields, and current publish/transfer paths still reject that deleted row. I did not run a live production Convex recovery attempt in this read-only review.

## Solution Assessment

No, current main is only a partial solution: the error is clearer, but org owners still cannot recover the deleted package identity themselves. Maintainers should choose the intended deletion policy before code changes decide between restore, transfer repair, hard delete, or permanent reservation.
