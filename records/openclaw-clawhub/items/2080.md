## Summary

Keep open: current main still blocks republishing a soft-deleted package name and does not expose an owner-facing restore or recovery path; the exact error text may now be the newer restore-before-publishing message, but the reported recovery problem remains valid and security-sensitive.

## What This Changes

The issue asks how org owners can recover or republish a plugin package after UI deletion leaves the package name or ownership reservation in place.

## Best Possible Solution

Define package soft-delete semantics, then add an audited owner/admin restore or recovery path across Convex, HTTP API, CLI/UI, tests, and docs, coordinated with https://github.com/openclaw/clawhub/issues/1370.

## Reproduction Assessment

Yes for the central blocked-republish behavior: source inspection shows UI/API deletion sets softDeletedAt and current publish rejects the same normalized package name. The exact reported cross-publisher error may now differ because the soft-deleted branch runs before the ownership mismatch branch.

## Solution Assessment

Unclear. A restore or audited recovery flow is likely safer than releasing names by default, but maintainers need to choose the package deletion, reservation, transfer, and hard-delete policy first.
