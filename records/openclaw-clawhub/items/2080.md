## Summary

Keep open under the ClawHub issue policy; current main only partially mitigates the stale ownership symptom by returning a deleted-package error, while package delete still reserves the package name and there is no package restore or recovery path for org owners.

## What This Changes

The issue asks for a maintainer-supported way for org owners to recover or understand a package name that remains blocked after UI plugin deletion.

## Best Possible Solution

Define package deletion and name-reservation semantics, then ship an explicit recovery path and matching UI/API/CLI wording for deleted plugin packages.

## Reproduction Assessment

Yes, as a source-level path: dashboard/API delete soft-deletes the package while preserving its normalized name and owner fields, and publish still resolves that row and blocks new releases. I did not run a live Convex production-row reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the intended deletion policy. The current deleted-package error is a useful guard, but the maintainable solution should make restore, transfer/repair, hard delete, or permanent reservation explicit instead of inventing silent reuse semantics.
