## Summary

Keep open: this is an active package lifecycle and production recovery issue, and ClawHub policy keeps issues open unless a merge or explicit maintainer action resolves them. Current main still soft-deletes package rows, hides them from public surfaces, and keeps the normalized package name reserved during publish, so the reporter's recovery request remains real.

## What This Changes

The issue asks how an org owner can recover or republish a plugin package after UI deletion leaves the package slug or ownership reservation blocking publish.

## Best Possible Solution

Define package deletion and reservation semantics, then add or run an audited owner/admin recovery path that verifies control of the target org publisher before restoring, transferring, repairing, or deleting the stale package record.

## Reproduction Assessment

Yes, from source inspection. The dashboard delete path soft-deletes the package, public package surfaces hide soft-deleted rows, and the publish path still rejects an existing normalized package name; I did not run a live production publish against the reporter's specific row.

## Solution Assessment

Unclear. Releasing the package name automatically is not clearly safe; an audited restore, transfer, owner repair, or admin recovery flow is the safer likely direction, but maintainers need to choose the namespace reservation policy.
