## Summary

Keep open: current main still hides soft-deleted packages from UI/read paths while keeping the package name reserved, and there is no package restore/undelete recovery path for the reporter’s org-owned package case.

## What This Changes

The issue requests a maintainer-supported recovery path when deleting a plugin package from the UI leaves its package name reserved and later publishes blocked.

## Best Possible Solution

Define package deletion semantics, then add an audited owner/admin package restore or recovery flow across Convex, HTTP API, CLI, dashboard UI, docs, and production support for the affected package.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a package through the dashboard or package delete API, then publish the same normalized name and hit the soft-deleted package rejection branch; I did not run the authenticated production publish flow.

## Solution Assessment

Unclear. An audited restore/undelete path looks like the safest likely direction, but maintainers first need to decide whether package deletion reserves names, releases names, supports hard-delete, or remains irreversible with clearer warnings.
