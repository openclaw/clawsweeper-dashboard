## Summary

Keep open: current main still soft-deletes packages and hides them from public/read surfaces, while publishing the same normalized package name remains blocked and no package restore/undelete path exists for the reporter's recovery case.

## What This Changes

The issue asks ClawHub to clarify or repair package name reuse and ownership recovery after a UI-deleted org package blocks later publishing.

## Best Possible Solution

Define package deletion semantics and add an audited recovery path, likely package restore/undelete for authorized owners or maintainers plus UI/CLI/API/docs copy that makes reserved names explicit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete a package through the dashboard or package delete API, then publish the same normalized package name; current main keeps the package row soft-deleted and rejects the publish attempt.

## Solution Assessment

Unclear. A package restore/undelete flow looks like the narrowest maintainable recovery mechanism, but maintainers first need to choose the product policy for soft-deleted package names and stale ownership repair.
