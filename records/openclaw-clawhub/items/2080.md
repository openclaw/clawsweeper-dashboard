## Summary

Keep open: current main still soft-deletes the package row while leaving the normalized name reserved, and there is no package undelete, restore, or hard-delete recovery path for the reporter’s deleted-package case.

## What This Changes

The issue requests a recovery path or clearer semantics when a package deleted from the UI still reserves its package name and blocks republish.

## Best Possible Solution

Define package deletion and reservation semantics, then add an audited package restore, owner repair/transfer, or hard-delete recovery path and align the dashboard, API, CLI, docs, and specs with that policy.

## Reproduction Assessment

Yes, as a source-level path: dashboard delete soft-deletes the package row, UI/list paths hide soft-deleted packages, and publish resolves the same normalizedName and rejects it. I did not run a live production-row reproduction in this read-only review.

## Solution Assessment

No, the current main behavior is only a partial solution: it clarifies ownership validation and supports active-package transfer, but deleted packages still cannot be restored, transferred, reclaimed, or explicitly hard-deleted by an owner/admin flow.
