## Summary

Keep open under the ClawHub issue policy; source inspection supports that package deletion currently leaves the package row and owner mapping in place, so recovery semantics need a maintainer decision.

## What This Changes

The issue asks how org owners should recover when a package deleted from the UI remains reserved and later publishes fail with an ownership error.

## Best Possible Solution

Define package delete semantics, then implement an explicit recovery path for soft-deleted packages or update the UI/API to warn that deletion reserves the package name until an owner/admin restores or transfers it.

## Reproduction Assessment

Yes for the source-level stale reservation path: publish resolves soft-deleted packages by name and checks their stored owner mapping. I did not run a live Convex/database reproduction of the reporter's exact org state in this read-only review.

## Solution Assessment

Unclear until maintainers choose the intended policy. A maintainable fix should avoid silently creating a second package row and should make restore, transfer, or permanent reservation explicit.
