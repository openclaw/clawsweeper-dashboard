## Summary

Keep open: current main still soft-deletes package rows while retaining the global package identity and owner mapping, and publish still rejects deleted or mismatched-owner rows. This remains an active package namespace recovery/admin policy issue, and ClawHub cleanup policy does not auto-close issues.

## What This Changes

The issue requests a supported recovery policy and workflow when a plugin package deleted from the UI disappears publicly but its package slug or publisher ownership mapping still blocks future publishes.

## Best Possible Solution

Define and document package delete/recovery semantics, then use audited owner/admin restore, transfer, repair, or hard-delete flows only after verifying package namespace authority.

## Reproduction Assessment

Yes, at source level: current main hides soft-deleted packages from readable/public paths while retaining the global normalizedName row, and the publish path rejects soft-deleted or different-owner rows. The exact production records still require admin inspection.

## Solution Assessment

Unclear as an automatic fix: existing restore, transfer, and admin repair primitives are safer than republish bypasses, but maintainers must choose whether deletion reserves names, releases names, or supports verified reassignment.
