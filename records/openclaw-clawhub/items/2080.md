## Summary

Keep open: current main still hides UI-deleted plugin packages while preserving the package row/name ownership, and publishing the same normalized package name is rejected without an owner-facing package restore or recovery path.

## What This Changes

The issue requests a recovery path or clearer delete semantics after a UI-deleted org-owned plugin package still blocks republishing the same package name.

## Best Possible Solution

Define package deletion and reservation semantics, then add an audited owner/admin restore or recovery flow across Convex, HTTP/CLI/UI, docs, and tests, with production recovery only after verifying ownership.

## Reproduction Assessment

Yes. Source inspection shows dashboard/API delete sets softDeletedAt on package and release rows, public surfaces hide those rows, and publishing rejects the existing soft-deleted package name; I did not run a live production publish for the reporter's row.

## Solution Assessment

Unclear. Releasing the package name by default is not clearly safe; an audited restore/undelete or admin repair path looks safer, but maintainers need to choose the package lifecycle policy first.
