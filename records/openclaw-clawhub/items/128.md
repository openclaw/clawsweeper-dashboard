## Summary

Keep open. Current main still lacks `clawhub update --dry-run`, and ClawHub cleanup policy keeps issues open unless a specific close rule applies.

## What This Changes

Request to add a `--dry-run` flag to `clawhub update` so users can preview single-skill or bulk updates without changing installed files or lock state.

## Best Possible Solution

Add a no-write planning path to `cmdUpdate` that reuses existing registry, moderation, fingerprint, version, semver, and pin checks, then document and test the preview behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. The feature gap is source-checkable: current main has no update `--dry-run` flag, no `dryRun` option in `cmdUpdate`, and no preview-only branch before writes.

## Solution Assessment

Yes. Extending the existing update command is the narrow maintainable direction because it can reuse the current update checks instead of adding a separate checker that could drift.

## Repair Work Prompt

Implement `clawhub update --dry-run` for both `update <slug>` and `update --all`. Add the flag in CLI registration, thread a `dryRun` option into `cmdUpdate`, and create a preview-only branch that performs registry lookup, moderation checks, local fingerprint/version resolution, semver comparison, pin handling, and local-change detection but exits before `rm`, download/extract, origin metadata writes, or lockfile writes. Output should clearly separate would-update, up-to-date, skipped pinned, local-change, suspicious/malware-blocked, missing, and already-at-target cases. Preserve existing `--force`, `--version`, prompt, and pin semantics; do not change install, publish, sync, or package dry-run behavior. Add focused tests proving dry-run output and no mutation for single-skill and `--all`, update CLI docs/readmes, and add an Unreleased changelog entry for the user-facing CLI flag.
