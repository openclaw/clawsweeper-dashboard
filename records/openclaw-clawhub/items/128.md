## Summary

Keep open. Current main still lacks `clawhub update --dry-run`, the related dry-run work is for package publish rather than update, and ClawHub cleanup policy keeps issues open instead of auto-closing them.

## What This Changes

Requested a `--dry-run` mode for `clawhub update` that previews available skill updates without changing local state.

## Best Possible Solution

Add preview support to the existing update flow, reusing the same registry, moderation, fingerprint, version, and local-change checks while returning before any filesystem or lockfile writes.

## Reproduction Assessment

Yes for the feature gap: current main has no registered `--dry-run` flag, no `dryRun` option in `cmdUpdate`, and no preview-only branch before the mutating update steps. This is a new CLI capability request, not a broken existing contract.

## Solution Assessment

Yes. Extending the existing update command is the narrow maintainable solution because it can share the real update resolution and safety checks instead of adding a second checker with divergent behavior.

## Repair Work Prompt

Implement `clawhub update --dry-run` for https://github.com/openclaw/clawhub/issues/128. Add the Commander flag in `packages/clawhub/src/cli.ts`, thread a `dryRun?: boolean` option into `cmdUpdate`, and reuse the existing update resolution path for single-slug and `--all` modes. The dry-run path should report would-update, up-to-date, and skipped states, including pinned skills, suspicious or malware-blocked skills, local changes without `--force`, missing latest versions, and explicit `--version` handling. It must not call `rm`, `downloadZip`, `extractZipToDir`, `writeSkillOrigin`, or `writeLockfile`, including when fingerprint reconciliation would normally update the lockfile. Add focused tests in `packages/clawhub/src/cli/commands/skills.test.ts` for output and no-mutation behavior, and update `docs/cli.md` plus `packages/clawhub/README.md` if the README command reference needs the new flag. Do not change unrelated publish/sync dry-run behavior or Convex backend code. Changelog/release-note handling should follow maintainer release practice and should not add contributor credit boilerplate unless maintainers request it.
