## Summary

Keep open. Current main still lacks `--dry-run` on `clawhub update`, and ClawHub policy keeps issues open rather than auto-closing them; the request is a narrow CLI feature suitable for a focused repair PR.

## What This Changes

Add a `--dry-run` flag to `clawhub update` that previews available updates without changing installed skill files or lockfile state.

## Best Possible Solution

Add a no-mutation planning path to the existing update command, document the flag, add a changelog entry, and cover single-skill plus `--all` dry-run behavior in focused CLI tests.

## Repair Work Prompt

Implement `clawhub update --dry-run` as a focused CLI feature. Add the flag in `packages/clawhub/src/cli.ts`, extend `cmdUpdate` options in `packages/clawhub/src/cli/commands/skills.ts`, and reuse the existing registry, moderation, fingerprint, version-resolution, semver, and local-change checks to print what would happen for a single skill and for `--all`. The dry-run path must not remove directories, download/extract zips, write origin metadata, mutate the lock object in a way that is later persisted, or call `writeLockfile`; avoid prompts in dry-run and report skipped states clearly. Add focused tests in `packages/clawhub/src/cli/commands/skills.test.ts` for would-update, up-to-date, and no-write behavior; document the flag in `docs/cli.md` and user-facing CLI docs/examples where appropriate; add a `CHANGELOG.md` entry for the CLI feature. Do not change package publish or sync dry-run behavior except for shared helpers that keep their existing tests passing.
