## Summary

The branch’s central user problem remains unsolved in the CLI: owner-qualified refs still install and persist under the bare slug, so two publishers’ same-slug skills collide locally. The PR also needs current-main conflict resolution, durable ownership/compatibility intent, and real runtime proof before merge.

Likely related people
- Patrick-Erichsen — high confidence; introduced the branch and owns the recent current-main schema/CLI history around this surface.

## What This Changes

The PR makes skill slugs owner-scoped across backend publishing and lookup, legacy API ambiguity handling, ClawHub CLI commands and metadata, and owner-aware UI install/download links.

## Best Possible Solution

Use one canonical owner-qualified local identity for every installed skill, add a narrow migration from legacy slug-keyed state, preserve documented slug-only API compatibility, and validate duplicate-owner fresh installs plus upgrades in real Convex, HTTP, CLI, and browser flows.

## Reproduction Assessment

Yes for the PR defect at source level: installing two refs such as `@alice/demo` and `@bob/demo` reaches the same `demo` directory and `lock.skills.demo` entry. A real CLI run is still required as merge proof.

## Solution Assessment

No, not yet. Backend owner scoping is appropriate, but the CLI must use owner-qualified persisted identity and a tested legacy migration rather than retaining the bare slug as the local source of truth.
