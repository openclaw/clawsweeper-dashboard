## Label Justifications

- `P2`: This valid publish-path bug blocks same-slug publishing across publishers but has a bounded workflow blast radius.
- `impact:security`: Global slug reservation crosses publisher ownership boundaries and creates namespace-squatting or interference leverage.

## Summary

Current `main` still resolves version publishes through the global `skills.by_slug` index before checking publisher ownership, so a same-slug row under another publisher is treated as a migration/collision and the reported bug remains. The collaborator-authored issue is paired with the open, conflicted implementation at https://github.com/openclaw/clawhub/pull/2299, so it needs maintainer product/security review rather than a separate automated repair.

## What This Changes

Allow separate publishers to publish the same skill slug without treating another publisher’s row as an ownership migration.

## Best Possible Solution

Adopt one owner-scoped resolution contract: normal publishing looks up `(ownerPublisherId, slug)`, migration targets an existing row only with explicit intent and source/destination authority, and legacy slug-only ambiguity is deterministic, documented in the slug/security specs, and validated in a real Convex runtime.

## Reproduction Assessment

Yes. Current main globally selects the existing slug before comparing `ownerPublisherId`, then routes a different publisher into the migration guard and slug-taken error; this is a high-confidence source reproduction, though no runtime test was executed in this read-only review.

## Solution Assessment

Yes for the central fix shape: normal publishing should resolve by publisher plus slug. The exact landing still needs maintainer judgment because authorized migration, aliases, deleted-slug reservations, and legacy slug-only clients must remain safe and compatible.
