## Summary

Keep open: current main still resolves the skill publish write path through a global `skills.by_slug` lookup before the owner-migration guard, so the reported cross-publisher slug collision remains source-reproducible. The item is collaborator-authored and the broad owner-scoping implementation is still an open related PR, so this should stay with maintainer/security/product review.

## What This Changes

Request to stop skill publishing from treating slugs as globally unique when publishing under a specific publisher namespace.

## Best Possible Solution

Land a maintainer-approved owner-scoped publish fix that resolves normal publishes by `(ownerPublisherId, slug)` while preserving explicit `migrateOwner` admin checks and tested legacy unscoped ambiguity behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: publish a slug under publisher B when publisher A already owns that slug, and `insertVersion` finds publisher A through global `by_slug` before throwing the migration slug-taken error.

## Solution Assessment

Yes for the fix shape: normal publishing should resolve by publisher plus slug. The exact landing path still needs maintainer/security/product review because migration and legacy unscoped lookup semantics are involved.
