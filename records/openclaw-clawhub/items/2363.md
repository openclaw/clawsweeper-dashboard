## Summary

Keep this open: current main still resolves publish writes through the global slug index before owner scoping, and the related owner-scoped implementation remains open and conflicting. The issue is also collaborator-authored, so cleanup closure would require explicit maintainer handling even if the fix looked complete.

## What This Changes

Publishing a skill should resolve existing rows by `(ownerPublisherId, slug)` so another publisher's same slug does not block normal version creation.

## Best Possible Solution

Land a reviewed owner-scoped slug implementation that uses `(ownerPublisherId, slug)` for normal publishes, preserves explicit migration-only behavior for reclaiming existing rows, and records the ownership/API trust invariants in `specs/`.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create an existing skill under one `ownerPublisherId`, then publish the same normalized slug under another publisher and the global `by_slug` lookup reaches the cross-owner migration guard.

## Solution Assessment

Unclear: the narrow code fix is owner-scoped lookup in the publish path, but the linked implementation also defines legacy fallback and artifact trust policy that needs maintainer approval before merge.
