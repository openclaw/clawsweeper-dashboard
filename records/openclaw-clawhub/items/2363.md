## Summary

Current main still violates the repository’s publisher-scoped skill identity invariant: `insertVersion` resolves an existing skill by globally unique slug before applying ownership checks. The issue should remain open and paired with the open collaborator-authored implementation PR until maintainers settle legacy slug-only compatibility and land a fix.

## What This Changes

Make normal skill publishing resolve existing skills by publisher and slug so another publisher’s same slug does not block creation.

## Best Possible Solution

Enforce `(ownerPublisherId, slug)` for normal creation and updates, preserve explicit authorized migration or reclaim paths, and centralize deterministic or explicitly ambiguous handling for every legacy slug-only caller with real Convex upgrade validation.

## Reproduction Assessment

Yes, at source level with high confidence: current `insertVersion` globally resolves the slug and then treats a different publisher as an ownership mismatch, matching the reported rejection path. The eventual fix still needs a real Convex runtime check because index and mutation semantics are involved.

## Solution Assessment

Yes for the core owner-aware write direction, but not as an isolated lookup edit. The maintainable solution must also define legacy ambiguity behavior across reads, installs, aliases, moderation, and migration paths.
