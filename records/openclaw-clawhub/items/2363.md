## Summary

Keep open: current main still resolves normal skill publishing through a global slug lookup before the owner-migration guard, while the schema and org spec define skill uniqueness as publisher-scoped. The broad owner-scoped slug PR at https://github.com/openclaw/clawhub/pull/2299 closed unmerged, and the merged availability fix at https://github.com/openclaw/clawhub/pull/2179 does not repair this publish write path.

## What This Changes

Skill publishing should allow the same slug under different publishers by resolving normal publish targets with `(ownerPublisherId, slug)` rather than a globally unique slug.

## Best Possible Solution

Use `by_owner_publisher_slug` for ordinary publish target lookup, apply matching scoped alias handling where the publish path checks aliases, and keep cross-publisher ownership moves behind explicit migration checks with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows a different-publisher same-slug row is selected by the global `by_slug` query and then rejected by the migration guard; I did not run a live publish because this review is read-only.

## Solution Assessment

Yes. The schema and spec already define the scoped invariant, so the narrow maintainable fix is scoped publish lookup plus tests proving duplicate scoped slugs work while explicit owner migration remains guarded.
