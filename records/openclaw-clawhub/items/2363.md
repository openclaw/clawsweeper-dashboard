## Summary

Keep open: current main still resolves skill publishes through the global slug index before owner scoping, and the related owner-scoped implementation at https://github.com/openclaw/clawhub/pull/2299 is open, conflicting, and not a landed fix. The report is also collaborator-authored, so conservative cleanup should leave it for maintainer handling.

## What This Changes

Publishing a skill under one publisher should use the `(ownerPublisherId, slug)` target so another publisher's same slug does not block normal version creation.

## Best Possible Solution

Land a reviewed owner-scoped skill publishing implementation that uses `(ownerPublisherId, slug)` for normal publish/update targets while keeping explicit migration, alias, reservation, and legacy read compatibility safeguards intact.

## Reproduction Assessment

Yes, as a source-level reproduction: current `insertVersion` globally resolves `slug` with `skills.by_slug`, then treats a different `ownerPublisherId` as an owner migration and throws slug-taken without explicit migration authority. I did not run a live authenticated publish in this read-only review.

## Solution Assessment

Yes for the requested direction: normal version creation should resolve by `(ownerPublisherId, slug)` instead of bare slug. The final implementation still needs maintainer review because the durable fix must handle aliases, reservations, explicit migration, and legacy slug-only compatibility together.
