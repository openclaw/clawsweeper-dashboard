## Summary

The central bug remains on current main: normal version publishing still begins from a global slug match and treats a same-slug row under another publisher as an ownership migration. The issue is also collaborator-authored and paired with an open implementation PR, so it requires explicit maintainer handling rather than cleanup closure.

## What This Changes

Make normal skill publishing resolve identity by publisher and slug so separate publishers can use the same slug.

## Best Possible Solution

Use `(ownerPublisherId, slug)` for ordinary publish identity, retain explicit authorization for cross-publisher transfer, and define deterministic compatibility behavior for legacy unscoped APIs, aliases, and deleted records.

## Reproduction Assessment

Yes at source level: current `insertVersion` routes a different publisher’s same-slug row through the migration guard and slug-taken error. This read-only review did not run the required real Convex two-publisher publish flow.

## Solution Assessment

The owner-scoped lookup is the right central repair, but the broad paired PR still needs maintainer decisions for legacy slug-only ambiguity, transfers, aliases, and deleted-skill reservations.
