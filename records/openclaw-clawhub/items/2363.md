- `impact:security`: The issue concerns ownership and namespace boundaries for publishing skills across different publishers.

## Summary

Keep open: current main still resolves `insertVersion` through the global `skills.by_slug` index before the owner-migration guard, so a same-slug skill from another publisher can still become a publish collision. This issue is collaborator-authored and the related owner-scoped slug PR remains open and conflicting, so it needs maintainer handling rather than cleanup closure.

## What This Changes

Publishing a skill under one publisher should not be blocked by a same-slug skill owned by another publisher unless an explicit authorized owner migration is requested.

## Best Possible Solution

Use owner-publisher-scoped resolution for normal skill publish/update while retaining explicit, audited `migrateOwner` handling for cross-owner reclaim or migration, with regression tests and spec alignment.

## Reproduction Assessment

Yes, at source level: `insertVersion` still resolves the incoming slug through global `skills.by_slug` before checking owner migration, then rejects a different `ownerPublisherId` unless migration is explicitly authorized. I did not run a live Convex publish in this read-only review.

## Solution Assessment

Unclear as a landing path. Owner-scoped publish lookup is the right end state, but maintainers should choose whether to land the broad open PR or a narrow publish-path fix that preserves legacy compatibility and ownership checks.
