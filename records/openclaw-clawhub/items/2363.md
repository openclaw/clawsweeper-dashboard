## Summary

Current main still reproduces the reported cross-publisher collision: `insertVersion` chooses the target publisher but looks up the skill through global `by_slug` before applying cross-owner migration guards. The issue is collaborator-authored and the matching implementation remains open and conflicted, so it needs maintainer product/security review rather than cleanup or a second automated fix branch.

## What This Changes

Make normal skill publishing resolve existing skills by publisher and slug so different publishers can use the same slug without triggering ownership migration.

## Best Possible Solution

Adopt publisher-plus-slug as the first-class publish identity, retain separate explicitly authorized reclaim and migration paths, and define deterministic compatibility behavior for legacy slug-only reads with focused source tests and real Convex/HTTP publish validation.

## Reproduction Assessment

Yes. Current main derives the target publisher and then performs a global `by_slug` lookup, so a different publisher's same-slug row is selected and handled as a migration/collision instead of an independent publish target.

## Solution Assessment

Yes for the core direction, but not as an isolated lookup substitution. The maintainable fix must preserve aliases, deleted-slug reservations, authorized migration, ownership healing, and legacy slug-only compatibility as explicit paths.
