## Summary

Current main and the latest release still have the reported publish-path collision: normal publishing resolves the slug globally before owner-scoped handling. This should stay open because it is collaborator-authored, source-reproducible, and tied to https://github.com/openclaw/clawhub/pull/2299 as an open but conflicting broad fix candidate rather than an implemented main change.

## What This Changes

The issue reports that skill publishing should allow the same slug under different publisher owners instead of treating the slug as globally reserved.

## Best Possible Solution

Land reviewed owner-scoped slug handling that uses `(ownerPublisherId, slug)` for normal publishes while preserving explicit `migrateOwner` for true ownership transfers and legacy compatibility paths.

## Reproduction Assessment

Yes from source inspection: current main still resolves normal publishes through the global `skills.by_slug` lookup, then treats a different `ownerPublisherId` as an owner-migration collision unless `migrateOwner` and source authority are present. I did not run a live Convex runtime reproduction in this read-only review.

## Solution Assessment

Unclear as a full product path: owner-scoped publish lookup is the right direction, but the linked broad PR also changes legacy read/install ambiguity and CLI behavior, so maintainers should review the compatibility/security contract before landing it.
