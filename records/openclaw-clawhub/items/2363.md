## Summary

Keep open: current main still resolves skill publishes through a global slug lookup before owner scoping, and the linked owner-scoped slug PR is open and conflicting rather than a merged fix; the issue is also collaborator-authored, so it should not be auto-closed.

## What This Changes

The issue requests that skill publish/version creation resolve existing skills by `(ownerPublisherId, slug)` instead of treating `slug` as globally unique.

## Best Possible Solution

Land or narrow the owner-scoped slug fix so normal publishes resolve by publisher namespace while explicit migration/reclaim paths remain protected.

## Reproduction Assessment

Yes from source inspection: current main resolves the publish target through global `skills.by_slug`, and a different `ownerPublisherId` flows into the slug-taken migration guard. I did not run a live Convex mutation, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear as a final implementation path; owner-scoped lookup is necessary, but the linked PR changes many compatibility surfaces and is currently conflicting, so maintainers need to approve or narrow the design.
