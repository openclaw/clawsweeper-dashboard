- `impact:security`: The affected behavior is ownership-scoped publishing and cross-publisher authorization for skill namespaces.

## Summary

This should stay open: current main still resolves normal skill publishes through the global `skills.by_slug` lookup before the owner-migration guard, while the schema and org spec already define skill uniqueness as publisher-scoped.

## What This Changes

Publishing should use `(ownerPublisherId, slug)` for normal skill version creation so different publishers can publish the same local slug without triggering ownership migration.

## Best Possible Solution

Use the scoped `by_owner_publisher_slug` lookup for ordinary publish targets, apply the same scoped treatment to slug aliases where needed, and keep cross-publisher row reuse behind the existing explicit migration checks.

## Reproduction Assessment

Yes from source inspection, though I did not run a live publish: a different-publisher same-slug row is selected by the global `by_slug` lookup and then rejected by the migration guard.

## Solution Assessment

Yes; the schema and spec already define the desired invariant, so the maintainable fix is a narrow publish-path change plus regression coverage around scoped publish and explicit migration.

## Repair Work Prompt

Fix https://github.com/openclaw/clawhub/issues/2363. In `convex/skills.ts`, make normal `insertVersion` publishing resolve the target skill by `(ownerPublisherId, slug)` using `skills.by_owner_publisher_slug`, so different publishers can publish the same local slug independently. Preserve explicit owner migration semantics: only `migrateOwner: true` with source and destination authority may move an existing row from another publisher. Check slug alias handling too, because `skillSlugAliases` also has `by_owner_publisher_slug` and the current global alias lookup may otherwise keep scoped publishes blocked. Add regression tests for cross-publisher same-slug publish creating a new scoped skill, same-publisher republish updating the existing skill, explicit authorized migration still moving a row, and unauthorized or missing-migrate-owner attempts still failing. Do not broaden this into unrelated routing or UI work.
