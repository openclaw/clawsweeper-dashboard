## Label Justifications

- `P2`: This is a concrete backend publishing bug with limited blast radius and a clear repair surface, not a full service outage.
- `impact:security`: The issue affects publisher-scoped ownership and authorization boundaries by allowing another publisher's slug row to control the publish decision.

## Summary

Keep open: current main still resolves normal skill publishes through a global slug lookup before the owner-migration guard, so a same-slug skill under another publisher can still block a publisher-scoped publish. The merged related availability fix only addressed slug availability, and the broader owner-scoped PR is closed unmerged.

## What This Changes

Skill publishing should resolve existing publish targets by `(ownerPublisherId, slug)` so different publishers can reuse the same local skill slug without accidental ownership migration.

## Best Possible Solution

Use `by_owner_publisher_slug` for ordinary publish target lookup, scope publish-path alias checks consistently, and keep cross-publisher ownership changes behind explicit migration intent and authority with regression coverage.

## Reproduction Assessment

Yes from source inspection: a different-publisher same-slug row is selected by the global `by_slug` query and then rejected by the owner-migration guard. I did not run a live publish because this review is read-only.

## Solution Assessment

Yes; the schema and spec already define publisher-scoped uniqueness, so the narrow maintainable fix is scoped publish lookup plus tests proving explicit owner migration still requires intent and authority.
