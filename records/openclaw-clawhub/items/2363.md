## Label Justifications

- `impact:security`: The affected logic governs publisher ownership, cross-namespace slug claims, and squatting resistance.

## Summary

Current main still performs the initial `insertVersion` lookup through globally unique `by_slug`, so another publisher’s row is routed into the ownership-migration guard and blocks an ordinary namespaced publish. This issue is paired with an open collaborator-authored implementation PR and should remain open while maintainers resolve that PR’s compatibility and security-boundary decisions.

## What This Changes

Allow different publishers to publish the same skill slug by resolving normal version writes within the selected publisher namespace rather than globally.

## Best Possible Solution

Adopt publisher-scoped `(ownerPublisherId, slug)` identity for authoritative writes and owner-aware surfaces while retaining an explicit, tested migration path and controlled compatibility behavior for legacy slug-only clients.

## Reproduction Assessment

Yes, at source level with high confidence: current main globally resolves the slug at `convex/skills.ts:10585`, then treats another publisher’s row as an ownership-migration case at `convex/skills.ts:10633`. A real Convex two-publisher publishing scenario remains necessary for implementation validation.

## Solution Assessment

Yes directionally, but not as an isolated index substitution. The maintainable solution must scope authoritative writes by publisher while explicitly handling aliases, legacy slug-only reads and installs, and authorized ownership migration.
