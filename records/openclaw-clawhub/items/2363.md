## Summary

Current main still globally resolves a publish slug before checking the target publisher, contradicting the repository’s publisher-scoped uniqueness invariant. Keep this collaborator-authored issue open and paired with the broader open implementation at https://github.com/openclaw/clawhub/pull/2299; that PR is unmerged and currently conflicted, so it is not implementation evidence or a reason to close.

## What This Changes

Change normal skill publishing to resolve existing skills within the target publisher namespace rather than treating slugs as globally unique.

## Best Possible Solution

Implement publisher-scoped publish identity while preserving explicit authorized migration and defining deterministic, tested compatibility behavior for legacy slug-only API and CLI consumers.

## Reproduction Assessment

Yes at source level: current main resolves the slug globally and then routes a different `ownerPublisherId` into the collision/migration guard. This read-only review did not execute a real Convex publish, so the eventual patch must add runtime validation for two publishers using the same slug.

## Solution Assessment

The requested owner-scoped write lookup is necessary, but it is not sufficient by itself. The best maintainable solution must also preserve migration authorization and settle compatibility for all legacy slug-only consumers before duplicate slugs are admitted broadly.
