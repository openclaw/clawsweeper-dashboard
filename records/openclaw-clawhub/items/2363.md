## Summary

Keep open under the ClawHub issue policy and because current main still has the reported source-visible bug: normal skill publish resolution starts with a global slug lookup, while the schema, docs, and org spec define publisher-scoped skill ownership. The merged availability fix at https://github.com/openclaw/clawhub/pull/2179 covers slug availability, and the broader owner-scoped slug PR at https://github.com/openclaw/clawhub/pull/2299 closed unmerged, so this publish write-path issue remains.

## What This Changes

The issue asks for publisher-scoped skill publish lookup so same-slug skills under different publishers do not collide unless an explicit owner migration is requested.

## Best Possible Solution

Use `by_owner_publisher_slug` for ordinary publish target lookup, scope publish-time alias checks consistently, and keep cross-publisher ownership changes behind explicit migration intent and authority tests.

## Reproduction Assessment

Yes from source inspection: a different-publisher same-slug row can be selected by the global `by_slug` query and then rejected by the migration guard. I did not run a live publish because this review is read-only.

## Solution Assessment

Yes; the schema, docs, and org spec already define the invariant, so the narrow maintainable solution is scoped publish lookup plus regression coverage for normal scoped publish and explicit migration.
