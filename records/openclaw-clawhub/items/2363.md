- `impact:security`: The issue affects publisher ownership and slug squatting, which determine which principal controls a published skill namespace.

## Summary

Keep this open: current main still starts `insertVersion` with the global `skills.by_slug` lookup before owner scoping, while the owner-scoped implementation remains an open conflicting PR and this issue is collaborator-authored.

## What This Changes

Publishing a skill should resolve existing rows by `(ownerPublisherId, slug)` so another publisher's same slug does not block normal publishes unless explicit migration is requested.

## Best Possible Solution

Land a reviewed owner-scoped slug implementation that uses `(ownerPublisherId, slug)` for normal publishes, preserves explicit migration-only behavior for reclaiming existing rows, and records the ownership/API trust invariants in `specs/`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create an existing skill under one `ownerPublisherId`, then publish the same normalized slug under another publisher and the global `by_slug` lookup reaches the cross-owner migration guard.

## Solution Assessment

Unclear. The narrow repair is owner-scoped lookup in the publish path, but the linked implementation also defines legacy fallback and artifact trust policy that needs maintainer approval before merge.
