## Summary

Keep open: current main still resolves the skill publish write path through a global `skills.by_slug` lookup before the owner-migration guard, so the reported cross-publisher slug collision remains source-reproducible. ## What This Changes

The issue asks for normal skill publishing to resolve existing targets by publisher-scoped `(ownerPublisherId, slug)` instead of treating `slug` as globally unique.

## Best Possible Solution

Land a maintainer-approved scoped publish fix that looks up normal publish targets and aliases by target publisher plus slug, keeps `migrateOwner` as the only ownership-transfer path, and adds focused regression coverage for same-slug different-publisher publishing.

## Solution Assessment

Yes on direction, but maintainer approval is still needed: publisher-scoped lookup is the narrow fix, while alias, reservation, and migration compatibility need explicit review before changing this security-sensitive write path.
