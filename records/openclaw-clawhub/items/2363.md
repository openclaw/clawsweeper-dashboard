## Summary

Keep open: current main still resolves `insertVersion` through a global `skills.by_slug` lookup before the owner-migration guard, so the reported cross-publisher slug collision remains source-reproducible. The issue is collaborator-authored, security-sensitive, and the broad owner-scoped slug implementation remains open and conflicting, so it needs maintainer handling rather than cleanup close.

## What This Changes

Publishing a skill under one publisher should not be blocked by the same slug already existing under another publisher.

## Best Possible Solution

Land a maintainer-approved owner-scoped publish path that resolves normal creation by `(ownerPublisherId, slug)` while preserving explicit migration authority checks and tested legacy ambiguity behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: publish a slug under publisher B when publisher A already owns that slug, and `insertVersion` finds publisher A through global `by_slug` before throwing the migration slug-taken error.

## Solution Assessment

Yes for the fix shape: normal publishing should resolve by publisher plus slug. The exact landing path still needs maintainer/security/product review because migration and legacy unscoped lookup semantics are involved.
