## Label Justifications

- `P2`: The defect blocks legitimate publishing for colliding names across publisher namespaces but has a bounded workflow blast radius.
- `impact:security`: Global slug reservation crosses publisher ownership boundaries and enables namespace squatting against unrelated publishers.

## Summary

Current main still resolves the publish target through the globally unique `by_slug` index before applying publisher ownership checks, so the reported cross-publisher collision remains. The issue must also stay open because it is collaborator-authored and paired with the still-open implementation PR at https://github.com/openclaw/clawhub/pull/2299.

## What This Changes

Allow separate publishers to publish skills with the same slug while preserving explicit cross-owner migration controls.

## Best Possible Solution

Adopt publisher-plus-slug identity for normal publish and update flows, preserve an explicit authorized migration path, and define deterministic or controlled-ambiguity behavior for legacy slug-only APIs with focused runtime coverage.

## Reproduction Assessment

Yes, at source level with high confidence: current main’s global `.unique()` slug resolver conflicts with the schema’s publisher-scoped index and the owner-aware availability path. A real Convex two-publisher publish smoke is still needed as implementation validation, not to establish the code-path defect.

## Solution Assessment

The owner-scoped lookup direction is correct, but the active PR is broader than the write-path bug and requires maintainer judgment on legacy slug-only API ambiguity, aliases, and migration compatibility before merge.
