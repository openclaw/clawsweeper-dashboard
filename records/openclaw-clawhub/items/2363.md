- `P2`: The bug blocks legitimate publishing for colliding names but affects a bounded subset of publisher workflows.
- `impact:security`: Global slug reservation lets one publisher squat on and block the same name within another publisher’s namespace.

## Summary

Current main still performs a global slug lookup before applying publisher ownership checks, so a skill owned by another publisher can block a legitimate publisher-scoped creation. The linked implementation at https://github.com/openclaw/clawhub/pull/2299 remains open and conflicting, and both items are collaborator-authored, so this issue needs explicit maintainer handling rather than cleanup closure.

## What This Changes

Allow different publishers to publish the same skill slug without treating another publisher’s row as an ownership migration conflict.

## Best Possible Solution

Adopt a maintainer-approved publisher-qualified identity model: resolve normal publish/update targets by `(ownerPublisherId, slug)`, keep ownership transfers explicitly authorized, and define deterministic compatibility behavior for legacy slug-only APIs with real fresh-install and upgrade validation.

## Reproduction Assessment

Yes at source level with high confidence: current `insertVersion` performs the global lookup and sends a cross-publisher match into the migration/collision branch. This read-only review did not execute a live Convex publishing transaction.

## Solution Assessment

The publisher-scoped write lookup is the correct root direction, but the broad linked PR is not automatically the best landing shape until maintainers approve legacy slug resolution, alias uniqueness, reclaim semantics, and upgrade behavior.
