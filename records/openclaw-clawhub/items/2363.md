## Summary

Keep open: current main still routes normal skill publishing through a global slug lookup before the owner-migration guard, so same-slug skills owned by different publishers can still collide despite the schema and org spec defining publisher-scoped uniqueness. The broad related fix at https://github.com/openclaw/clawhub/pull/2299 closed unmerged, while the merged availability fix at https://github.com/openclaw/clawhub/pull/2179 only addressed slug availability/preflight behavior, not the publish write path.

## What This Changes

The issue requests making normal skill publish lookup publisher-scoped so same slugs under different publishers do not collide unless an explicit owner migration is requested.

## Best Possible Solution

Publisher-scoped publish should resolve `(ownerPublisherId, slug)` for ordinary version creation and only enter cross-publisher migration when the caller explicitly requests it with sufficient authority, with regression coverage for same-slug publishers and migration denial.

## Reproduction Assessment

Yes from source inspection: a different-publisher same-slug row is selected by the global `by_slug` query and then rejected by the migration guard. I did not run a live publish because this review is read-only.

## Solution Assessment

Yes; the schema and spec already define the invariant, so the narrow maintainable fix is scoped publish lookup plus tests that migration still requires explicit intent and authority.
