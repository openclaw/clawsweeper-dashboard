## Summary

Keep open: current main still resolves normal skill publishing through a global slug lookup before the owner-migration guard, so publisher-scoped slugs can still collide in the write path. The related scoped-slug PR closed unmerged, and the merged duplicate-slug fix covered availability checks rather than publishing.

## What This Changes

The issue requests publisher-scoped skill publish lookup so the same slug can exist under different publishers unless the caller explicitly migrates ownership.

## Best Possible Solution

Use publisher-scoped skill and alias lookup for ordinary publishes, keep explicit owner migration as the only cross-publisher reuse path, and add focused regression coverage.

## Reproduction Assessment

Yes from source inspection: create or assume an existing same-slug skill under another publisher, then publish that slug under a different `ownerPublisherId`; `insertVersion` selects the other publisher's row via `by_slug` and rejects through the migration guard. I did not run a live publish because this review is read-only.

## Solution Assessment

Yes; the schema and org spec already define scoped uniqueness, so the narrow maintainable fix is scoped publish lookup plus tests proving explicit migration still requires intent and authority.
