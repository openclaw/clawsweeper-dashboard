## Label Justifications

- `P2`: The bug blocks legitimate publishing when another publisher already owns the same slug, but its blast radius is limited to colliding namespaces.
- `impact:security`: The affected path governs publisher namespace ownership, slug squatting resistance, and authorization for cross-publisher transfers.

## Summary

Current `main` still performs a globally unique slug lookup before publisher ownership handling, so the reported collision remains real; keep this collaborator-authored issue open while maintainers resolve the compatibility-sensitive companion PR.

Likely related people: Patrick-Erichsen (high, recent skill-backend and owner-scoping contributor), Marko Anastasov (high, duplicate-slug availability contributor), and Momo (high, owner-migration hardening contributor).

## What This Changes

Allow independent skills with the same slug under different publishers while preserving explicit, authorized ownership migration and legacy slug-only compatibility.

## Best Possible Solution

Resolve the target publisher first and use `(ownerPublisherId, slug)` for ordinary creation and updates; enter a global reclaim/transfer path only with explicit migration intent and authority on both publishers, while returning controlled ambiguity for legacy ownerless reads and validating the behavior in a real Convex deployment.

## Reproduction Assessment

Yes—current-main source provides a high-confidence reproduction path: publish a slug already present under another `ownerPublisherId`, and the global `.unique()` lookup routes it into the cross-owner collision guard. This read-only review did not execute a live Convex mutation.

## Solution Assessment

Yes at the invariant level: publisher-plus-slug should identify ordinary publish targets, with a separate explicit migration path. The broad companion PR still needs maintainer judgment on legacy ownerless API and CLI ambiguity behavior.
