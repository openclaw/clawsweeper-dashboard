## Label Justifications

- `impact:security`: A globally reserved slug lets one publisher interfere with another publisher’s namespace and creates a concrete squatting and ownership-boundary concern.

## Summary

This remains an active core bug: current main still performs global-slug resolution in the publish mutation, while the comprehensive owner-scoped fix is open and conflicting. Because the reporter is a repository collaborator and the fix requires compatibility/security choices across legacy slug-only surfaces, this needs explicit maintainer handling rather than cleanup closure.

## What This Changes

Scope skill publish identity by publisher so a slug owned by one publisher does not block another publisher from publishing the same slug.

## Best Possible Solution

Make publisher plus slug the authoritative identity for owner-aware publish/update flows, while explicitly preserving deterministic legacy slug-only behavior and testing migration, aliases, deletion/reclaim, install, and security-sensitive lookup paths.

## Reproduction Assessment

Yes, at source level with high confidence: current main has the publisher-scoped index but the publish mutation still follows global-slug conflict and migration logic. This read-only review did not execute a live Convex mutation, so the status is source-reproducible rather than runtime-reproduced.

## Solution Assessment

The proposed owner-scoped direction is necessary, but changing only the initial lookup is not the best complete fix; maintainers must approve and validate the compatibility contract for every remaining slug-only consumer and reclaim path.
