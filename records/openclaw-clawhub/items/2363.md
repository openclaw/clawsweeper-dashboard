## Summary

This should stay open: current main still resolves normal skill publishes through the global `skills.by_slug` lookup before the owner-migration guard, while the schema and org spec define skill uniqueness as publisher-scoped.

## What This Changes

The issue reports that skill publishing still enforces globally unique slugs instead of allowing the same slug under different publishers.

## Best Possible Solution

Use the scoped `by_owner_publisher_slug` lookup for ordinary skill publish targets, apply owner-scoped alias handling where the publish path checks redirects, and keep cross-publisher row reuse behind the existing explicit migration checks.

## Reproduction Assessment

Yes from source inspection: a different-publisher same-slug row is selected by the global `by_slug` lookup and then rejected by the migration guard. I did not run a live publish because this review is read-only.

## Solution Assessment

Yes; the schema and spec already define the desired invariant, so the narrow maintainable fix is a scoped publish lookup plus regression coverage while preserving explicit migration authority checks.

## Repair Work Prompt

Fix https://github.com/openclaw/clawhub/issues/2363. In current main, `convex/skills.ts` `insertVersion` resolves the publish target with global `skills.by_slug`, so another publisher's same-slug skill is treated as the target row and rejected by the owner-migration guard. Change ordinary publish target resolution to use `(ownerPublisherId, slug)` via `by_owner_publisher_slug`, with the existing legacy personal-publisher fallback only where needed for pre-migration rows. Preserve the explicit `migrateOwner: true` source and destination authority checks for real ownership transfers. Apply the same scoped treatment to publish-path alias checks where a cross-publisher alias would otherwise block independent publication, and audit touched unscoped `.unique()` slug lookups so duplicate scoped slugs do not create new ambiguity crashes. Add focused regression coverage proving `/alice/example-skill` and `/bob/example-skill` can both publish, same-owner version publishing still updates the existing row, explicit owner migration still requires `migrateOwner: true` and admin/owner authority, and ambiguous legacy unscoped behavior is handled intentionally. Likely files are `convex/skills.ts`, `convex/skills.ownerMigration.test.ts`, `convex/skills.slugAvailability.test.ts`, and possibly `convex/httpApiV1.handlers.test.ts`. Do not broaden package behavior, redesign read APIs, or add a changelog entry unless maintainers explicitly request one.
