## Summary

Keep open: current main still resolves normal skill publishing through a global slug lookup before the owner-migration guard, while the schema and org spec define publisher-scoped skill uniqueness. The merged availability fix only covered slug availability, and the broader owner-scoped PR closed unmerged.

## What This Changes

The issue asks for normal skill publishing to resolve existing skills by publisher plus slug so different publishers can publish the same local slug without colliding globally.

## Best Possible Solution

Use publisher-scoped skill and alias lookup for ordinary publishes, keep explicit owner migration as the only cross-publisher reuse path, and add regression coverage for same-slug publishers plus migration rejection.

## Reproduction Assessment

Yes from source inspection: create or assume an existing same-slug skill under another publisher, then publish that slug under a different `ownerPublisherId`; `insertVersion` selects the other publisher's row via `by_slug` and rejects through the migration guard. I did not run a live publish because this review is read-only.

## Solution Assessment

Yes; the schema and org spec already define the invariant, so the narrow maintainable fix is scoped publish and alias lookup plus tests proving migration still requires explicit intent and authority.
