## Label Justifications

- `P2`: This is a real publish-path bug with limited but meaningful blast radius for duplicate skill slugs across publishers.
- `impact:security`: The issue affects publisher namespace ownership and slug squatting, which determine which principal controls a published skill name.

## Summary

Keep this open: current main still starts skill publish writes from the global slug index, so a same-slug skill owned by another publisher reaches the owner-migration guard instead of creating a separate publisher-scoped skill. The related owner-scoped implementation is open and conflicting, and this issue is collaborator-authored, so cleanup closure is not appropriate.

## What This Changes

Skill publishing should resolve existing skills by `(ownerPublisherId, slug)` so separate publishers can use the same local slug without unintended migration or slug-taken failures.

## Best Possible Solution

Land a reviewed owner-scoped slug implementation that uses `(ownerPublisherId, slug)` for normal publishes, preserves explicit migration-only behavior for reclaiming existing rows, and records the ownership/API trust invariants in `specs/`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create an existing skill under one `ownerPublisherId`, then publish the same normalized slug under another publisher and the global `by_slug` lookup reaches the cross-owner migration guard.

## Solution Assessment

Unclear. The narrow code fix is owner-scoped lookup in `insertVersion`, but the full legacy lookup, alias, migration, and artifact trust behavior needs maintainer product/security approval.
