## Summary

Keep open: current main still uses a global skill-slug lookup in the publish write path even though schema/spec intent is publisher-scoped uniqueness, and the related broad scoped-slug PR is closed unmerged while the merged availability PR only addressed preflight availability.

## What This Changes

Allow different publishers to publish the same skill slug under their own publisher namespace without falling into global slug collision or owner-migration rejection.

## Best Possible Solution

Use by_owner_publisher_slug for ordinary publish target lookup, apply equivalent scoped alias checks in the publish path, and keep cross-publisher ownership migration behind explicit migration intent plus regression coverage.

## Reproduction Assessment

Yes from source inspection: a different-publisher same-slug row is selected by the global by_slug query and then rejected by the migration guard. I did not run a live publish because this review is read-only.

## Solution Assessment

Yes; the schema and spec already define scoped uniqueness, so the narrow maintainable fix is scoped publish lookup plus tests that explicit migration still requires intent and authority.
