## Label Justifications

- `P2`: This is a real publish-path bug with limited but meaningful blast radius for duplicate skill slugs across publishers.
- `impact:security`: The bug affects publisher namespace ownership and slug squatting, which determines who controls a published skill identity.

## Summary

Keep this open: current main still starts `insertVersion` with a global `skills.by_slug` lookup, so a same-slug row owned by another publisher reaches the migration guard and can throw the slug-taken error. The issue is collaborator-authored and has a same-author open companion PR, so it needs maintainer review rather than cleanup closure.

## What This Changes

Skill publishing should resolve normal publishes by `(ownerPublisherId, slug)` so one publisher's slug does not block another publisher's namespace.

## Best Possible Solution

Land a reviewed owner-scoped slug implementation that uses `(ownerPublisherId, slug)` for normal publishes, keeps cross-owner reclaiming behind explicit migration authorization, and records the ownership/API trust invariants in `specs/`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create an existing skill under one `ownerPublisherId`, then publish the same normalized slug under another publisher and the global `by_slug` lookup reaches the cross-owner migration guard.

## Solution Assessment

Unclear. Owner-scoped lookup in the publish path is the narrow bug fix, but the broader legacy fallback and artifact-trust contract in the linked implementation requires maintainer approval.
