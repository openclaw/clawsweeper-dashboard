## Summary

Keep open: current main still resolves `insertVersion` through the global skill-slug index before the owner-migration guard, and the related owner-scoped slug PR is open and conflicting rather than merged. The issue is also collaborator-authored, which requires maintainer handling rather than cleanup closure.

## What This Changes

The issue reports that skill publishing still checks skill slugs globally instead of within the publisher namespace, blocking different publishers from using the same slug.

## Best Possible Solution

Use owner-publisher-scoped resolution for normal skill publish/update while keeping explicit, authorized migration for reclaiming an existing cross-owner row.

## Reproduction Assessment

Yes, at source level: the publish mutation still resolves `normalizedSlug` through global `skills.by_slug` before checking owner migration. I did not run a real Convex runtime reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation choice until maintainers resolve the owner-scoped slug PR. The narrow behavior should be owner-scoped publish lookup plus explicit migration-only reclaim semantics.
