## Summary

Keep open: current main still resolves normal skill publishing through a global `skills.by_slug` lookup before the owner-migration guard, while the schema, docs, and org spec define skill uniqueness as publisher-scoped. The merged availability fix at https://github.com/openclaw/clawhub/pull/2179 does not cover `insertVersion`, and the broader owner-scoped PR at https://github.com/openclaw/clawhub/pull/2299 closed unmerged.

## What This Changes

The issue asks for skill version publishing to use publisher-scoped slug lookup so different publishers can publish the same local slug without a global collision.

## Best Possible Solution

Use publisher-scoped skill and alias lookup for ordinary publishes, keep explicit owner migration as the only cross-publisher reuse path, and add focused regression coverage.

## Reproduction Assessment

Yes from source inspection: create or assume an existing same-slug skill under another publisher, then publish that slug under a different `ownerPublisherId`; `insertVersion` selects the other publisher's row via `by_slug` and rejects through the migration guard. I did not run a live publish because this review is read-only.

## Solution Assessment

Yes; the schema and org spec already define the invariant, so the narrow maintainable fix is scoped publish lookup plus tests proving migration still requires explicit intent and authority.
