## Summary

Keep open: current main still resolves normal skill publish targets through a global slug lookup before the owner-migration guard, while the schema, publishing docs, and org spec define owner-scoped publishing and `(ownerPublisherId, slug)` uniqueness. The merged availability fix at https://github.com/openclaw/clawhub/pull/2179 only covered slug availability, and the broader owner-scoped slug PR at https://github.com/openclaw/clawhub/pull/2299 closed unmerged.

## What This Changes

The issue asks for skill publishing to resolve existing targets by publisher and slug so different publishers can publish the same slug without being treated as ownership migration.

## Best Possible Solution

Use `by_owner_publisher_slug` for ordinary publish target lookup, scope publish-time alias collision checks to the selected publisher where appropriate, and keep cross-publisher ownership changes behind explicit migration intent and authority tests.

## Reproduction Assessment

Yes. Source inspection on current main shows a different-publisher same-slug row is selected by the global `by_slug` query and then rejected by the migration guard; I did not run a live publish because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to align the publish write path with the existing schema/spec invariant and add regression coverage for duplicate slugs plus explicit owner migration.
