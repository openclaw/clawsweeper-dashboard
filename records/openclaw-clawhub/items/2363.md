## Summary

This should stay open: current main still resolves normal publish paths through global slug lookup before the owner-migration guard, while the schema and org spec already define skill identity as publisher-scoped.

## What This Changes

The issue reports that skill publishing still treats slugs as globally unique instead of allowing the same local slug under different publishers.

## Best Possible Solution

Use `by_owner_publisher_slug` for ordinary publish target lookup, keep cross-publisher row reuse behind explicit migration checks, and add regression coverage for same-slug different-publisher publish plus migration behavior.

## Reproduction Assessment

Yes. From source inspection, a different-publisher same-slug row can be selected by the global `by_slug` publish lookup and then rejected by the owner-migration guard.

## Solution Assessment

Yes. The schema and spec already define the desired invariant, so the narrow maintainable fix is to make normal publish lookup publisher-scoped while preserving explicit migration checks.
