## Summary

Keep this issue open: current main still resolves ordinary skill publishes through the global `skills.by_slug` lookup before the owner-migration guard, even though the schema and org spec define skill names as publisher-scoped.

## What This Changes

The issue asks for normal skill publishing to treat `(ownerPublisherId, slug)` as the publish target so different publishers can use the same local slug.

## Best Possible Solution

Use `by_owner_publisher_slug` for ordinary `insertVersion` publish targets, make alias checks scoped where required, preserve explicit migration semantics for row reuse, and add regression coverage for same-slug different-publisher creation plus migration guards.

## Reproduction Assessment

Yes, from source inspection: a different-publisher same-slug row is selected by the global `by_slug` lookup and then rejected by the migration guard. I did not run a live publish in this read-only review.

## Solution Assessment

Yes; the schema and spec already define the desired invariant, so the maintainable fix is a narrow publish-path change plus regression coverage around scoped publish and explicit migration.
