## Summary

Keep open: current main still resolves normal skill publishing through global slug lookup before owner-scoped publish handling, so a different publisher’s same slug can still be treated as a collision or migration attempt. The merged availability fix only covered slug availability, and the broad owner-scoped PR closed unmerged.

## What This Changes

The issue asks for skill publishing to resolve existing rows by `(ownerPublisherId, slug)` so different publishers can publish the same local slug.

## Best Possible Solution

Use publisher-scoped skill and alias lookup for ordinary publishes, keep explicit authorized owner migration as the only cross-publisher reuse path, and add regression coverage for duplicate slugs across publishers.

## Reproduction Assessment

Yes from source inspection: create or assume an existing same-slug skill under another publisher, then publish that slug under a different `ownerPublisherId`; `insertVersion` selects the other publisher’s row via `by_slug` and rejects through the migration guard. I did not run a live publish because this review is read-only.

## Solution Assessment

Yes; the schema and org spec already define the invariant, so the narrow maintainable fix is scoped publish lookup plus tests proving migration still requires explicit intent and authority.
