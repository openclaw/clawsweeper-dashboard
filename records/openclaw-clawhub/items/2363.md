- `impact:security`: Skill slug resolution controls publisher namespace ownership, anti-squatting behavior, and which publisher can supply artifacts under a visible skill identity.

## Summary

Keep open: current main and v0.21.0 still resolve `insertVersion` by global slug before the owner-migration guard, while the related owner-scoped slug PR is open and conflicting rather than merged.

## What This Changes

The issue reports that skill publishing still checks slugs globally instead of resolving existing skills by publisher and slug.

## Best Possible Solution

Land a reviewed owner-scoped slug implementation that scopes normal publishes by `(ownerPublisherId, slug)`, keeps `migrateOwner` as an explicit transfer path, and updates legacy slug-only resolution, aliases, specs, and regression tests together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: publish a skill with a slug already owned by another `ownerPublisherId`, and `insertVersion` resolves the global `by_slug` row before throwing the slug-taken migration guard unless explicit migration is authorized.

## Solution Assessment

Yes on direction, but not as an isolated one-line lookup swap. The maintainable solution needs scoped publish lookup plus explicit handling for aliases, owner migration, and legacy slug-only ambiguity.
