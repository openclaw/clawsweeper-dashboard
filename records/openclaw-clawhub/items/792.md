## Summary

Current main still has no install/update-to-active-session handoff, and the related open reports cover distinct hook-governance and registration work rather than superseding this request. The feature is aligned with setup reliability, but it needs maintainer product and security judgment because newly installed instructions would influence a running agent.

## What This Changes

Notify an active OpenClaw session after a skill install or update so the agent can read the new SKILL.md and complete onboarding steps.

## Best Possible Solution

Define a consent-aware OpenClaw handoff that identifies the target session, presents the installed skill and onboarding actions as untrusted instructions, and requires explicit operator approval before privileged actions run.

## Reproduction Assessment

Yes. Current-main source shows install and update finish with filesystem, metadata, telemetry, and console work but no active-session event delivery, matching the reported behavior.

## Solution Assessment

No, not as unconditional instruction execution. A post-install handoff is appropriate, but the maintainable solution must make session targeting, trust labeling, consent, and privileged follow-up actions explicit.

## Risks / Open Questions

- Automatically instructing a running agent to act on newly installed SKILL.md content could convert installation into implicit command execution, configuration mutation, plugin activation, or service restart without a clear consent boundary.
- The ClawHub CLI lacks an active-session IPC and target-selection contract, so a CLI-only implementation could become coupled to OpenClaw internals or deliver the event to the wrong session.
- Hook governance and registration remain separately open in https://github.com/openclaw/clawhub/issues/707 and https://github.com/openclaw/clawhub/issues/2295; this design should not bypass their security and consent decisions.
