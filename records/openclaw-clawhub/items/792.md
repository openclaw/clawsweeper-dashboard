## Summary

Keep open. Current main still does not provide an active-session post-install/update handoff, and this ClawHub cleanup policy keeps issues open unless they are PRs already implemented on main; the request also needs maintainer product/security judgment.

## What This Changes

Request that `clawhub install` and `clawhub update` notify an active OpenClaw session so the agent reads `SKILL.md` and performs setup/onboarding work.

## Best Possible Solution

Maintainers should define the install/update boundary explicitly: either add a user-consented OpenClaw session handoff contract with focused CLI/session tests and docs, or document the inspectable copy-prompt flow as the supported setup path.

## Reproduction Assessment

Yes. Source inspection on current main is enough to verify the observable gap: install/update paths write skill files and metadata but have no session event, `SKILL.md` onboarding read, or handoff call.

## Solution Assessment

Unclear. The proposed event injection addresses the onboarding gap, but it is not clearly the best solution until maintainers define consent, trust, and the OpenClaw/ClawHub handoff boundary.

- Current ClawHub CLI code does not expose an active OpenClaw session IPC or handoff surface, so this likely needs a cross-project contract rather than a narrow CLI-only patch.
- The approved skill install surface currently chooses inspectable copyable prompts over direct handoff, so implementing this would revise an existing product direction.
