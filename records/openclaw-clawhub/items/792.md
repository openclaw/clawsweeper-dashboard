## Label Justifications

- `P2`: Complex skills can remain incompletely configured after installation, but users retain manual onboarding paths and the problem does not disable the core registry.
- `impact:security`: The proposed behavior allows newly installed third-party instructions to influence privileged agent actions and therefore changes an important trust boundary.
- `impact:session-state`: The requested capability directly injects new system context and follow-up work into the active agent session.

## Summary

Current main still completes skill installs and updates without notifying an active agent session, so the central capability remains necessary. The request is endorsed in the discussion and fits setup-reliability goals, but the exact automatic-execution wording needs a maintainer product and security decision because newly installed third-party instructions could initiate configuration changes, package installation, or gateway restarts.

## What This Changes

Notify the active agent session after a skill install or update so it reads the skill instructions and completes any remaining onboarding.

## Best Possible Solution

Define a structured OpenClaw-owned post-install/update receipt that identifies the installed version and declared onboarding actions, injects that receipt into the initiating active session when one exists, and requires explicit operator approval before privileged or mutating follow-up actions.

## Reproduction Assessment

Yes at source level. Current main's install and update paths end after filesystem, metadata, telemetry, and status-output work, with no active-session notification call; the report describes this missing capability accurately.

## Solution Assessment

No, not exactly as proposed. Session notification is a maintainable direction, but instructing automatic execution of arbitrary SKILL.md onboarding should be replaced by a structured, trust-aware receipt and explicit approval semantics.

## Risks / Open Questions

- An unconditional system instruction to execute any onboarding in a newly installed SKILL.md could let third-party content initiate package installs, configuration patches, plugin activation, health-check mutations, or gateway restarts without a clearly defined operator approval boundary.
- The complete solution crosses the ClawHub registry/install metadata boundary and the separate OpenClaw runtime/session boundary, so implementing only in this repository could create a partial or competing lifecycle path.
- The relationship among onboarding instructions, declared hooks, post-install commands, and hook registration remains a product-contract question; overlapping automatic mechanisms could produce duplicate actions or confusing authority.
