## Summary

Current main does not implement an active-session handoff after installation or update. The request remains useful and has explicit collaborator support, but the proposed automatic execution behavior creates a new ClawHub-to-OpenClaw trust boundary that requires maintainer product and security decisions.

## What This Changes

Add a post-install and post-update handoff that informs an active OpenClaw session about the changed skill and prompts it to process the skill’s onboarding instructions.

## Best Possible Solution

Define a durable, consent-aware ClawHub/OpenClaw handoff that emits structured install or update information, safely selects or declines a session target, presents requested setup actions, and requires explicit approval before consequential execution.

## Reproduction Assessment

Yes, at high confidence from source inspection: current main’s install completion has no gateway or session notification path, while the existing Install & Setup flow is only a copied prompt. This demonstrates the capability is absent, although it is a feature gap rather than broken documented behavior.

## Solution Assessment

No, not exactly as proposed. A structured notification is plausible, but unconditional instructions to execute any SKILL.md onboarding conflict with the project’s operator-control direction; maintainers should first define consent, session targeting, failure handling, and the boundary with hooks and install metadata.

## Risks / Open Questions

- A package-manager command targeting an active session needs authenticated session discovery and deterministic behavior when multiple or no sessions exist.
- The proposed instruction could convert publisher-controlled SKILL.md text into commands, configuration edits, plugin installation, or gateway restarts without a sufficiently explicit user decision.
- Installation and update may run outside an OpenClaw-hosted agent process, so failure to reach a gateway must not make local installation unreliable or leave ambiguous partial setup state.
- Declarative install metadata, hooks, and agent-level onboarding need a clear source of truth to avoid duplicate or conflicting setup execution.
