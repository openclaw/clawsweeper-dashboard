## Summary

Keep open. Current main still stops after installing or updating skill files and metadata; the existing copy-prompt setup flow is a partial workaround, not the requested active-session event, and collaborator feedback explicitly supports prioritizing the idea.

## What This Changes

Request to inject a post-install/update system event into an active OpenClaw session so the agent reads the installed skill's SKILL.md and handles setup/onboarding instructions.

## Best Possible Solution

Define a consent-aware ClawHub/OpenClaw install-update handoff contract, then implement active-session notification with explicit session targeting, trust/disclosure rules, tests, docs, and a fallback to the current copy-prompt path.

## Reproduction Assessment

Yes, at source level. Current cmdInstall and cmdUpdate complete file replacement and metadata writes without any active-session notification, though this is a missing feature rather than a broken documented contract.

## Solution Assessment

No, not exactly as proposed. A structured notification is plausible, but unconditional instructions to execute setup from SKILL.md need maintainer-approved consent, session targeting, trust, and failure-handling rules first.

## Risks / Open Questions

- A package-manager command needs authenticated session discovery and deterministic behavior when multiple or no OpenClaw sessions are active.
- Automatically directing an agent to execute SKILL.md onboarding could turn publisher-controlled text into commands, config edits, plugin activation, or gateway restarts without a clear operator decision.
- The approved install surface currently chooses inspectable copyable prompts over direct handoff, so maintainers need to revisit that product and security boundary before implementation.
- The narrower hook-registration work at https://github.com/openclaw/clawhub/issues/2295 should align with this handoff contract so hook setup and agent onboarding do not become competing setup paths.
