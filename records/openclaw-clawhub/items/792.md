## Label Justifications

- `P2`: Complex skill onboarding can remain incomplete after installation, but basic file installation and updating still work.
- `impact:security`: The requested handoff could cause powerful actions to run based on newly installed and potentially untrusted skill content.
- `impact:session-state`: The proposal injects installation-derived context into the active agent session and changes subsequent agent behavior.

## Summary

Current main still has no post-install or post-update handoff to an active agent session, and the related hook issues do not implement this distinct capability. The request remains useful and was explicitly endorsed by a repository collaborator, but its cross-repository execution and consent boundaries require maintainer product and security decisions.

## What This Changes

Add an active-session system event after skill installation or update so the agent reads and follows the installed skill’s onboarding instructions.

## Best Possible Solution

Define a typed, host-controlled install/update notification that identifies the installed skill and available onboarding guidance, while requiring explicit operator approval before privileged skill-authored actions execute.

## Reproduction Assessment

Yes, at source level. Current main’s install and update paths end with local file and metadata changes plus CLI output, with no active-session event, callback, or onboarding result.

## Solution Assessment

No, not exactly as proposed. A post-install handoff is appropriate, but an unconditional instruction to execute onboarding steps should be replaced by a typed host-controlled notification with explicit consent for privileged actions.

## Risks / Open Questions

- Directing an agent to execute newly installed SKILL.md instructions could implicitly authorize package installation, shell commands, configuration mutation, credential access, plugin activation, or gateway restarts.
- The active session is owned by OpenClaw rather than the standalone ClawHub CLI, so implementation needs a documented cross-repository event or result contract rather than a brittle output-parsing path.
- The trust model must be coordinated with https://github.com/openclaw/clawhub/issues/707 and https://github.com/openclaw/clawhub/issues/2295 so onboarding instructions and hooks do not become parallel lifecycle mechanisms with inconsistent consent rules.
