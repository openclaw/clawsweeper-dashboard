## Summary

Keep open. Current main still does not inject a post-install/update event into an active OpenClaw session, and this ClawHub issue needs maintainer product/security judgment rather than cleanup closure.

## What This Changes

Request to notify the active OpenClaw session after `clawhub install` or `clawhub update` so the agent reads `SKILL.md` and completes setup or onboarding steps.

## Best Possible Solution

Define the supported install/update handoff contract explicitly: either add a user-consented OpenClaw session event with security policy, tests, and docs, or document the inspectable copy-prompt flow as the intended boundary.

## Reproduction Assessment

Not applicable as a bug reproduction: this requests a new active-session handoff behavior rather than reporting a broken existing contract. Source inspection gives high confidence that current main lacks the requested install/update event injection.

## Solution Assessment

Unclear as proposed. A system event could solve the onboarding gap, but the maintainable path first needs a maintainer-approved consent and trust-boundary design aligned with the current inspectable prompt surface and the related hooks security RFC.

## Risks / Open Questions

- Automatic active-session injection would let newly installed or updated skill content influence an agent, so consent, moderation, trust, and prompt-injection boundaries need maintainer security review.
- The ClawHub CLI currently has no active OpenClaw session IPC or handoff contract, so the feature likely crosses the ClawHub/OpenClaw boundary rather than being a narrow CLI-only patch.
- The approved install surface intentionally uses inspectable copyable prompts, so implementing the request would revise an existing product decision.
- The open lifecycle hooks RFC at https://github.com/openclaw/clawhub/issues/707 overlaps post-install/update execution policy and should be considered before changing install/update behavior.
