- `impact:security`: The requested handoff could cause an active agent to act on newly installed or updated publisher-controlled setup instructions.

## Summary

Keep open. Current main and v0.23.0 still install/update skill files plus metadata without an active-session system event, while the existing install UX remains an inspectable copy-prompt flow; Patrick-Erichsen also signaled this should be prioritized, so this still needs maintainer product/security direction.

## What This Changes

Request to notify the active OpenClaw session after `clawhub install` or `clawhub update` so the agent reads the updated SKILL.md and performs approved setup/onboarding work.

## Best Possible Solution

Define a consent-aware ClawHub/OpenClaw install-update handoff contract with explicit session targeting, inspectable setup guidance, approval before consequential changes, docs, and CLI/session tests.

## Reproduction Assessment

Yes, at source level. Current `cmdInstall` and `cmdUpdate` write skill files and metadata and print success without an active-session system event or SKILL.md onboarding read; I did not run a live OpenClaw gateway reproduction.

## Solution Assessment

No, not exactly as proposed. A notification may be appropriate, but unconditional instructions to execute setup from SKILL.md should be replaced by a maintainer-approved consent, session-targeting, trust, and failure-handling contract.

## Risks / Open Questions

- Automatically injecting setup instructions from newly installed or updated SKILL.md content could let publisher-controlled text influence an active agent without a settled consent and trust boundary.
- The CLI does not expose a deterministic OpenClaw gateway/session target, so a narrow CLI-only patch could misroute or partially apply setup when multiple or no sessions are active.
- The related hook-registration work at https://github.com/openclaw/clawhub/issues/2295 should be aligned so hooks, declarative metadata, and agent onboarding do not become competing setup mechanisms.
