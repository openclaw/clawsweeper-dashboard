## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still only partially addresses the request: repository/source links exist on metadata-backed surfaces and `/u/$handle` redirects to `/p/$handle`, but owner badges and public user/publisher projections still do not expose a verified GitHub profile URL for username/name links.

## What This Changes

The issue requests GitHub profile links beside usernames and on user profile pages, plus repository/source links for projects.

## Best Possible Solution

Add trusted GitHub login/profile URL data to public user and publisher surfaces, render a separate external GitHub affordance beside internal owner/profile navigation, and keep repository links sourced from skill/package metadata.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a UI/data enhancement request. Source inspection on current main gives a high-confidence implementation gap: owner badges and public projections do not carry a verified GitHub profile URL, while the profile link that exists is derived from `publisher.handle`.

## Solution Assessment

Yes, with a verified-data constraint. The maintainable solution is a trusted external-link field and a separate UI affordance, not guessing GitHub URLs from editable handles or nesting anchors inside existing internal links.
