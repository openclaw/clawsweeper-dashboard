## Summary

Current main and shipped ClawHub releases resolve the reported failure by copying a bare, OpenClaw-compatible skill slug instead of an owner ID or owner-qualified path; the remaining owner-scoped slug proposal is broader follow-up work, not evidence that this exact bug remains.

## What This Changes

Correct the experience-package page’s copy action so it produces an installable command rather than a command containing a nonexistent author ID.

## Best Possible Solution

Keep the shipped bare-slug command as the compatibility fallback, and address future owner-qualified installs only through coordinated UI, CLI, API-resolution, documentation, and upgrade coverage.

## Reproduction Assessment

No. Current main has a direct source-and-test path showing that the copy action emits a bare skill slug rather than the reported nonexistent author ID.

## Solution Assessment

Yes. Using the slug accepted by the shipped OpenClaw install contract is the narrow maintainable fix and avoids fabricating an owner path that the client cannot resolve.
