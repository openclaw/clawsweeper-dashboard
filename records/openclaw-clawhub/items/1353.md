## Summary

Keep open. Current main still has no canonical alias collapse in the plugin catalog; production no longer reproduces the exact `q=wework` duplicate search result, but both WeWork package identities still resolve and the remaining fix needs maintainer data/product judgment.

## What This Changes

The issue asks the plugin catalog and public API to collapse historical or alias-equivalent plugin package records so the same WeWork plugin does not appear as duplicate cards.

## Best Possible Solution

Define a maintainer-approved canonical package and alias model, choose the canonical WeWork record, preserve intentional legacy detail/install locators, repair production data, and add backend/frontend regression coverage for search and browse.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not for the exact original `q=wework` production search, which now returns only one WeWork plugin record; source inspection still shows alias-equivalent package records would not be collapsed if they appear in the same catalog result set.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A catalog-only heuristic is not the safest fix; the maintainable path is a canonical package identity and data-repair policy approved by maintainers.
