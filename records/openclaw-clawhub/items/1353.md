## Summary

Keep open. The duplicate WeWork records still appear in the paginated production plugin catalog, and current main has no canonical package/alias collapse for browse results; search currently returns only the scoped WeWork record, so the exact UI search path has changed but the underlying duplicate catalog state remains.

## What This Changes

The issue asks for alias or historical WeWork plugin package records to collapse to one canonical catalog card.

## Best Possible Solution

Define canonical package identity and alias handling for plugin records, preserve legacy detail/install locators, collapse browse/search onto the canonical record, and add WeWork-style backend and frontend regression coverage.

## Reproduction Assessment

Yes. Production pagination for /api/v1/plugins still returns both WeWork package records, and current main’s browse/search source paths show no alias-equivalent collapse beyond exact package keys.

## Solution Assessment

No automated heuristic is clearly best yet. The maintainable fix needs a maintainer-approved canonical package and alias/data cleanup policy so legacy locators keep working without showing duplicate catalog cards.
