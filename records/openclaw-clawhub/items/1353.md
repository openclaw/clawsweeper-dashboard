## Summary

Keep open. The duplicate WeWork catalog result still reproduces in production, and current main still collapses only exact package family/name matches while browse and frontend paths pass package records through. The new package identity repair admin helper is useful, but it does not yet define canonical aliases or remove duplicate catalog rows.

## What This Changes

The issue reports duplicate WeWork plugin catalog cards from separate `@openclaw/wework` and `openclaw-wework` package records for the same displayed plugin.

## Best Possible Solution

Define a canonical package/alias model, choose the WeWork canonical record, preserve legacy detail/install locators, collapse browse and search on canonical identity, and add backend plus frontend regression coverage.

## Reproduction Assessment

Yes. Production `/api/v1/plugins/search?q=wework&limit=50` still returns the two WeWork records on 2026-05-02, and source inspection shows no canonical alias collapse in the current catalog paths.

## Solution Assessment

No for an automated heuristic. The maintainable fix should be a backend canonical identity and alias/data-cleanup path rather than a frontend-only or display-name de-dupe.
