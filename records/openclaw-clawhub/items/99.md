## Summary

Keep open. Repository policy keeps ClawHub issues open for maintainer judgment, and current main does not prove the browser-login/token-verification failure is fixed because the CLI still verifies `/api/v1/whoami` before persisting tokens and related Unauthorized reports remain open.

## What This Changes

The issue reports that `clawhub login` completes browser authentication but the CLI receives `Unauthorized` instead of storing a usable token.

## Best Possible Solution

Confirm browser-created and settings-created token verification in production, then land one focused fix for the canonical host, callback registry, or token verification path with regression coverage and consolidate the related Unauthorized reports under a maintainer-owned tracker.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from this review. Source inspection confirms the 401 surface after a loopback token is received, but proving the reported valid-token failure on current main requires a fresh production account/token.

## Solution Assessment

Unclear. A blanket switch to `www.clawhub.ai` is not currently the best proven fix because live host behavior now redirects `www` to non-www; maintainers should reproduce first and then fix the narrow failing layer.
