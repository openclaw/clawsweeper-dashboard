## Summary

Keep open. This is an open ClawHub issue, and current main plus live host checks do not prove the reported login/token verification failure is fixed; a May 12, 2026 comment reports the same symptoms on `clawhub@0.14.0` with a `users:ensure` Convex error.

## What This Changes

Reports that `clawhub login` and fresh web-created tokens complete browser auth but fail CLI verification with `Unauthorized`.

## Best Possible Solution

Confirm the production auth path with fresh browser-created and settings-created tokens, identify whether the root cause is host routing, token lookup, `users.ensure`, or account state, then land one narrow fix with regression coverage and consolidate related reports.

## Reproduction Assessment

No for a high-confidence live current-main reproduction: this review did not have a real production account/token to replay the failure. Source inspection does confirm the reported path: browser and manual login both verify `/api/v1/whoami` before storing credentials, and that endpoint returns Unauthorized for token/account failures.

## Solution Assessment

Unclear. A host-default-only fix is not the best standalone answer after current live redirect checks and the closed redirect PR; maintainers should reproduce the production failure and then patch the exact token, `users.ensure`, or account-state path.
