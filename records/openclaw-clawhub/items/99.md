## Summary

Keep open. Repository policy keeps ClawHub issues open, and current main still leaves the reported failure path unresolved: browser login and token login verify `/api/v1/whoami` before persisting credentials, while related Unauthorized reports remain open and the earlier redirect-only PR was closed unmerged.

## What This Changes

The issue reports that `clawhub login` and `clawhub login --token` can complete browser or token creation but then fail with `Unauthorized` instead of storing a usable token.

## Best Possible Solution

Confirm production browser-created and settings-created tokens against the canonical host, then land one focused fix in the registry, callback, token verification, or deployment layer and consolidate the related Unauthorized reports under a maintainer-owned tracker.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from this review. Source inspection shows the 401 verification surface after browser/token creation, but proving the reported valid-token failure on current main requires a fresh production account and token.

## Solution Assessment

Unclear. A blanket switch to `www.clawhub.ai` is not currently proven best because live checks now show www redirecting back to non-www; maintainers should reproduce first and fix the narrow failing layer.
