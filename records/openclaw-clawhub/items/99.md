## Summary

Keep open. This is a ClawHub issue, and the cleanup policy keeps issues open for maintainer judgment; current main also does not prove the reported browser-login/token-verification failure is fixed because the CLI still verifies `/api/v1/whoami` before persisting tokens and related Unauthorized reports remain open.

## What This Changes

The issue reports that `clawhub login` completes browser authentication but the CLI receives `Unauthorized` instead of storing a token.

## Best Possible Solution

Confirm the production browser-created and settings-created token paths, choose the canonical host behavior deliberately, add focused regression coverage, and consolidate the related Unauthorized reports under one maintainer-owned tracker or fix.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from this review. Source inspection confirms the reported verification path and 401 surface, but a fresh valid production token/account is still needed to reproduce the claimed valid-token failure on current main.

## Solution Assessment

Unclear. A blanket switch to `www.clawhub.ai` is not currently the best proven fix because live production now redirects `www` to the canonical non-www host; maintainers should reproduce first and then fix either host canonicalization or token verification narrowly.
