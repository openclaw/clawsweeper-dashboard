## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still routes browser and token login through `/api/v1/whoami` token verification; live checks only prove current canonical host and invalid-token behavior, not that fresh production tokens no longer fail.

## What This Changes

The issue asks for `clawhub login` and `clawhub login --token` to receive, verify, and store a valid CLI token instead of returning `Unauthorized` after web auth succeeds.

## Best Possible Solution

Maintainers should verify fresh browser-created and Settings-created tokens in production with the current published CLI, then consolidate the related Unauthorized reports or land one focused fix at the proven failing layer with regression coverage.

## Reproduction Assessment

No. Source inspection traces browser and token login into `/api/v1/whoami`, and live requests show current host routing plus invalid-token 401 behavior, but reproducing the reported valid-token failure requires a fresh production account/token.

## Solution Assessment

Unclear. The expected behavior is correct, but switching defaults to `https://www.clawhub.ai` as proposed in https://github.com/openclaw/clawhub/pull/101 is no longer proven best because current production redirects `www` back to the apex host.
