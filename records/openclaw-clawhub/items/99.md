## Summary

Keep open. This is an open ClawHub issue, and current main still does not prove the valid-token login failure is fixed: token login/browser login still verify through `/api/v1/whoami`, the old `www.clawhub.ai` workaround is now contradicted by live routing, and related Unauthorized reports remain open.

## What This Changes

Issue reports that browser and token CLI login complete token creation but fail with `Unauthorized` before the CLI stores usable credentials.

## Best Possible Solution

Maintainers should verify fresh browser-created and Settings-created tokens in production against the current CLI and canonical host, then consolidate the related Unauthorized reports or land one focused fix at the proven failing layer with regression coverage.

## Reproduction Assessment

No, not end to end with high confidence. Source inspection traces the reported symptom through `cmdLogin` into `/api/v1/whoami`, but reproducing the valid-token failure requires a real production account/token that this read-only review did not have.

## Solution Assessment

Unclear. The expected behavior is already defined, but the old `www.clawhub.ai` workaround is not the best proven fix because current source, docs, live discovery, and observed production routing are canonical non-www.
