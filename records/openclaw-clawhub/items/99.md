## Summary

Keep open. ClawHub issue policy keeps issues open, and current main still has a plausible verify-before-persist `/api/v1/whoami` failure path while the old `www.clawhub.ai` workaround is contradicted by current live routing and the related redirect PR was closed unmerged.

## What This Changes

The issue asks for `clawhub login` and `clawhub login --token` to accept browser-created or UI-created CLI tokens and store them instead of failing with `Unauthorized`.

## Best Possible Solution

Verify fresh production browser-created and Settings-created tokens against the latest CLI, then fix the proven failing layer across registry discovery/canonical host, token creation or lookup, account state, or deployment routing with focused regression coverage.

## Reproduction Assessment

No. Source inspection shows how the CLI can surface `Unauthorized` when `/api/v1/whoami` rejects the callback or `--token` token, and a dummy live host probe checked the redirect theory, but no fresh production token was available to reproduce the reported failure end to end.

## Solution Assessment

Unclear. The desired behavior is correct, but switching defaults to `https://www.clawhub.ai` is not currently proven safe because live `www` routing redirects back to non-www; maintainers should reproduce with real tokens before choosing a code or deployment fix.
