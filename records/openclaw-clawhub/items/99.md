## Summary

Keep open. This is a ClawHub issue, so the cleanup policy does not auto-close it, and current main still has the reported auth gate: browser and token login persist credentials only after `/api/v1/whoami` verifies the token, while related Unauthorized reports remain open.

## What This Changes

The issue reports `clawhub login` completing browser authentication but the CLI failing token verification with `Unauthorized` before credentials are stored.

## Best Possible Solution

Confirm fresh production browser-created and settings-created tokens, then land one focused auth/registry fix with regression coverage and consolidate the related Unauthorized reports under a canonical tracker if they share the same cause.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available. The source path is clear and unauthenticated `/api/v1/whoami` returns 401 on the canonical host, but reproducing the reported valid-token failure requires a real production ClawHub account and fresh token.

## Solution Assessment

Unclear. Changing the default registry to `www` is no longer the best proven fix because current source, discovery, and live production routing use the apex host; the narrow fix should follow real-token production verification.
