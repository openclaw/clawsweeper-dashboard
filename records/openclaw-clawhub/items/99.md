## Summary

Keep open. ClawHub issue policy prevents cleanup closure for issues, and current main still verifies browser and token login through `/api/v1/whoami` before storing credentials; the related `www` host PR was closed unmerged, and live routing now redirects `www.clawhub.ai` back to `clawhub.ai`, so this needs maintainer production-token verification.

## What This Changes

The issue reports that browser and token login should accept fresh ClawHub tokens and persist them instead of failing with `Unauthorized`.

## Best Possible Solution

Verify fresh browser-created and Settings-created tokens in production against the latest CLI, then either consolidate the related Unauthorized reports under one canonical tracker or land one focused fix at the proven failing layer with regression coverage.

## Reproduction Assessment

No. Source inspection traces the reported symptom through `cmdLogin` into `/api/v1/whoami`, but reproducing the actual valid-token failure requires a fresh production token and account; the dummy live probe only proves the current host routing and invalid-token 401 behavior.

## Solution Assessment

Unclear. The requested behavior is correct, but the earlier `www.clawhub.ai` default switch is not currently the safest fix because the related PR was closed unmerged and live `www` traffic now redirects back to non-www.
