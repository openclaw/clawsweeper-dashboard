## Summary

The exact head has a concrete authorization defect: all three direct URL preview/publish actions use a public-only repository validator instead of the branch’s immutable-account ownership guard. The demonstrated UI flow is compelling, but the PR also intentionally removes the existing third-party public URL workflow and currently conflicts with main, so it needs a repaired head and explicit maintainer approval of that compatibility boundary.

## What This Changes

Adds discovery, review, and batch publishing of skills from public GitHub repositories owned by the signed-in account, plus backend validation, dashboard and publish entry points, tests, docs, and a security specification.

## Best Possible Solution

Use one shared immutable GitHub-account ownership guard across discovery, preview, and publish, prove owned and third-party cases through a real Convex path, and land the change only after maintainers explicitly choose whether owned-only imports should replace the existing public URL contract.

## Reproduction Assessment

Yes. At the exact head, each URL action calls `requirePublicGitHubRepoForImport`, whose implementation neither receives caller identity nor compares owner ids, while the stronger ownership helper is unused on those paths.

## Solution Assessment

No. The UX direction is strong, but the implementation must consistently enforce the chosen server-side trust boundary and must not silently replace the established public URL workflow without explicit maintainer approval.
