## Summary

This remains a useful, unique feature PR, but the current head is not merge-ready: its three legacy URL actions accept any public repository while the new picker, spec, and PR description promise an owned-repository boundary, creating a direct authorization bypass. The inspected browser recording strongly proves the positive scan-to-publish flow, but the branch also has unresolved merge conflicts and needs negative runtime proof after the boundary repair.

## What This Changes

Adds discovery, review, and batch publishing of skills from public GitHub repositories associated with the signed-in account, plus dashboard/publish entry points, tests, docs, and a security specification.

## Best Possible Solution

Use one server-side authorization policy for every picker and URL entry point: verify the repository is eligible and its immutable GitHub owner ID matches the signed-in provider account before resolving or downloading content, preserve approved `SKILL.md`/legacy `skills.md` URL compatibility, and validate the rebased result with both positive and negative real-runtime flows.

## Reproduction Assessment

Yes, with high confidence from the exact head: authenticate and pass a public repository owned by another GitHub account to any URL-shaped preview/import action; each uses public-only validation and omits the immutable owner-ID comparison. A real Convex action test should confirm the failure before merge.

## Solution Assessment

No, not in its current form. The guided picker is a strong solution, but maintaining owned-only picker authorization alongside public-only direct actions creates a bypass; a single approved server-side policy is narrower and maintainable.
