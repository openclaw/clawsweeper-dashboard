## Summary

Keep open: the canonical ClawScan field direction is useful, but this diff currently weakens existing security enforcement by making static-malicious skill and package findings advisory before a ClawScan verdict, and it lacks real runtime/backfill proof.

## What This Changes

This PR widens ClawScan storage/API schema with canonical verdict/state fields and rewrites scanner, moderation, search, UI, CLI, maintenance, and docs paths around clean/review/warn/malicious.

## Best Possible Solution

Land the canonical ClawScan fields as a schema-widening change while preserving current static-malicious holds and package download blocks, then split any telemetry-only enforcement policy into a separately approved migration with runtime proof.

## Reproduction Assessment

## Solution Assessment

No. The canonical fields are a good direction, but mixing schema widening with a security-policy reversal is not the safest implementation path; preserve existing holds unless maintainers explicitly approve the enforcement change.
