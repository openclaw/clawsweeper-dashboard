- **[P1] Keep package scan status aligned with the release resolver:** `convex/lib/packageSecurity.ts:55`
  - body: This resolver now treats a release with static/verification `malicious` but no ClawScan verdict as `pending` when it has an artifact hash. Package publish still seeds `verification.scanStatus` and package `scanStatus` from the static-derived `malicious` value, and package access checks use package-level `scanStatus`, so a newly published static-malicious plugin can remain package-blocked while its release/download path is pending/advisory. Update the publish/sync path to seed package status through the same canonical ClawScan resolver and add coverage for a static-malicious, ClawScan-pending plugin publish.
  - confidence: 0.88
- **[P2] Re-apply the hash when late tabs become valid:** `src/components/SkillDetailPage.tsx:445`
  - body: This effect no longer re-reads `window.location.hash` when `validTabIds` changes. If a user opens a `#skill-card` URL before `generatedSkillCard` is present, the first pass falls back to `readme`; once the card appears, `readme` is still valid, so the Skill Card tab is never selected. The deleted test covered exactly that async case, so restore the hash preference when the hashed tab becomes valid.
  - confidence: 0.9
