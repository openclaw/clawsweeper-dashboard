## Summary

Keep open. This is an open ClawHub issue, and current evidence still shows an unresolved auth problem: main has the right stable GitHub numeric-ID/profile-sync foundation, but production GitHub sign-in returns HTTP 500 and the reporter's username-change verifier mismatch cannot be proven fixed from source alone.

## What This Changes

The issue reports that GitHub OAuth login fails after the reporter changed their GitHub username, with a Convex Auth OAuth verifier-token mismatch and a request to relink the account.

## Best Possible Solution

Restore production GitHub OAuth session creation for active accounts, keep identity anchored to immutable GitHub providerAccountId, and only add or perform an audited recovery/relink path after maintainer verification of the affected production auth records.

## Reproduction Assessment

Yes for the current production cannot-sign-in failure: on May 10, 2026, current main is deployed and both canonical and direct Convex GitHub sign-in endpoints returned HTTP 500. No for the username-change-specific verifier mismatch itself, because that requires live OAuth callback state and the affected production auth rows.

## Solution Assessment

Not applicable for a patch because this is an issue, not a proposed fix. The existing numeric-ID mapping and post-login profile sync are the right foundation, but the safest next step is maintainer/operator auth triage before any relink or source change.
