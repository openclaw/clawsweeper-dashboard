## Summary

Current main fixes the generic GitHub-rename path by keying OAuth to the stable numeric GitHub ID and synchronizing renamed profiles after sign-in, but this report remains an unresolved production account-recovery case. The reported OAuth verifier mismatch may prevent that post-login sync from running, and maintainers have not verified or recovered the reporter’s account; destructive deletion also still requires the reporter’s explicit consent because it removes owned skills and plugins.

## What This Changes

Restore access to an existing ClawHub account after its GitHub username changes without unsafe username-based relinking or unintended content loss.

## Best Possible Solution

Inspect the production user and provider-account records using the immutable GitHub numeric ID, repair stale or duplicate bindings while preserving owned content when safe, and use the audited purge path only if the account is actually self-deleted and the reporter explicitly confirms the destructive effects.

## Reproduction Assessment

No. Current main has a verified generic rename path, but the reporter’s production identity rows, OAuth verifier transaction, and possible duplicate/deactivated account state are unavailable, so this individual failure cannot be reproduced with high confidence from the checkout.

## Solution Assessment

No. Treating an apparently active account as a deletion case is not the safest first solution; immutable-ID inspection and content-preserving repair should precede any explicitly confirmed purge.
