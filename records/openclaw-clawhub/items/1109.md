Keep open. Current main has the generic GitHub-rename foundation, but this remains a production auth/account-recovery report with a `convexAuthOAuthVerifier` mismatch, live auth-start endpoints still return HTTP 500, and ClawHub cleanup policy keeps issues open for maintainer handling.

## What This Changes

The issue reports that GitHub OAuth login fails after the reporter changed their GitHub username and asks maintainers to relink the account.

## Best Possible Solution

Restore the production GitHub OAuth start/callback path, keep identity anchored to numeric `providerAccountId`, inspect the affected production account rows, and add a narrowly scoped recovery/admin path only if operator triage proves one is needed.

## Reproduction Assessment

No for the exact username-change `convexAuthOAuthVerifier` mismatch: reproducing that requires the reporter's affected production account state and Convex Auth logs. I did verify on May 18, 2026 that the live GitHub auth-start endpoints return HTTP 500, which supports the broader auth failure but not the exact account-relink state.

## Solution Assessment

No for relinking by the new username alone: the safer path is operator-backed verification of the immutable GitHub provider account id, OAuth verifier/session logs, and affected account rows before any recovery action. The current source-level numeric-ID mapping and post-login profile sync are the right foundation, but they do not prove this production report is resolved.
