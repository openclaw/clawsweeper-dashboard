## Summary

Current main already uses GitHub’s immutable numeric provider ID and refreshes mutable profile handles after successful authentication, but that does not resolve this reporter’s already-stuck account before identity matching completes. The canonical deletion-recovery work is shipped, yet the discussion establishes that applying it here requires account inspection and explicit consent because purging can remove the user’s skills and plugins.

## What This Changes

Restore access to an existing ClawHub account after the same GitHub identity changes its username.

## Best Possible Solution

Verify the reporter’s immutable GitHub provider ID against the stored auth binding and preserve the existing account and content through a narrowly audited rebind if possible; use the shipped purge workflow only as a confirmed last resort.

## Reproduction Assessment

Yes at the source-scenario level: create an existing GitHub provider binding, change only the GitHub login while retaining the numeric provider ID, and verify whether authentication reaches the profile-sync callback. This read-only review could not reproduce the reporter’s private production account state.

## Solution Assessment

No, deleting the account is not the best first-line solution when a content-preserving stable-ID rebind may be possible; the shipped purge is appropriate only after identity verification, account inspection, and explicit consent.
