## Summary

Keep open. Current main has the right generic GitHub-rename foundation, and the older generic report was closed as fixed, but this production account-recovery report still has an OAuth verifier/session symptom that cannot be resolved from source alone.

## What This Changes

The issue reports failed GitHub OAuth login after a GitHub username change, with a `convexAuthOAuthVerifier` token mismatch and a request to relink the affected account.

## Best Possible Solution

Restore the production GitHub OAuth start/callback path, keep identity anchored to numeric `providerAccountId`, inspect the affected production account rows, and add a narrowly scoped recovery/admin path only if operator triage proves one is needed.

## Reproduction Assessment

No for the exact username-change `convexAuthOAuthVerifier` mismatch: reproducing that requires the reporter's affected production account state and Convex Auth logs. I did verify on May 18, 2026 that the live GitHub auth-start endpoints return HTTP 500, which supports the broader auth failure but not the exact account-relink state.

## Solution Assessment

No for relinking by the new username alone: the safer path is operator-backed verification of the immutable GitHub provider account id, OAuth verifier/session logs, and affected account rows before any recovery action. The current numeric-ID mapping and post-login profile sync are the right foundation, but they do not prove this production report is resolved.
