## Summary

Keep open: current main still has no supported way to relink a ClawHub user or personal publisher from one immutable GitHub OAuth account to another. The request is valid account-recovery/admin work, but it needs staff identity verification and a policy decision before any ownership or publisher-control change.

## What This Changes

The issue asks staff to re-link the existing `@gingiris` ClawHub publisher identity from an inaccessible old GitHub OAuth account to the current `Gingiris-1031` GitHub account so updates can be published.

## Best Possible Solution

Define and use a guarded staff-only recovery workflow that verifies identity continuity, records private evidence and public-safe rationale, and atomically preserves publisher, skill, slug, and moderation history during any approved transfer.

## Reproduction Assessment

Yes at the source level: a different immutable GitHub `providerAccountId` maps to a different ClawHub user, and publishing rejects existing slugs unless the authenticated user owns or can manage the linked publisher. Live confirmation of the two production identity records remains staff-only.

## Solution Assessment

No, a direct username-based relink or staff batch-publishing bypass is not the best permanent solution. The maintainable path is a policy-backed, audited recovery operation with explicit identity evidence and atomic ownership preservation.
