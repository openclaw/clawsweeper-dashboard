## Summary

Current main has an audited admin ownership-transfer primitive for individual skills, but the public recovery policy is still draft and the requested account-level relink remains a security-sensitive staff decision; this issue is still necessary to verify the claimant and execute the appropriate recovery.

## What This Changes

Transfer the existing `@gingiris` publisher identity or its skills to the requester’s current GitHub OAuth account so updates can resume.

## Best Possible Solution

Staff should privately verify the claimant and both GitHub numeric account identities, inspect each skill’s moderation eligibility, then use an audited recovery operation to migrate the account identity if approved or transfer the eligible skills while preserving history and ownership records.

## Reproduction Assessment

Yes at the source level: ClawHub keys ownership to the authenticated user linked by immutable GitHub provider account ID, while the normal transfer flow requires the inaccessible current owner to initiate. The specific production identities and skill states still require staff verification.

## Solution Assessment

No, a simple handle relink is not safe; the narrow maintainable path is a guarded, audited recovery based on verified immutable identities, with per-skill transfer as the fallback.
