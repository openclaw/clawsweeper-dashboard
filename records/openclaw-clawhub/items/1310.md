## Summary

The requested public, comparative quality system is not implemented on current main: ClawHub exposes detailed security scan results and has an internal anti-spam/thin-content quality gate, but that gate measures prompt structure and publisher trust rather than claim accuracy, maintenance, dependency hygiene, or alternatives. The issue remains useful product input and overlaps with the open tooling proposal at https://github.com/openclaw/clawhub/issues/2042 and the moderation-policy RFC at https://github.com/openclaw/clawhub/issues/2321, but neither currently owns the full user-facing product decision.

## What This Changes

Add structured, user-facing skill quality signals covering documentation, claim accuracy, maintenance, dependency hygiene, complexity, and alternatives alongside existing security indicators.

## Best Possible Solution

Define an RFC for separate explainable quality dimensions, data provenance and freshness, author visibility/appeals, and whether each signal affects detail pages, discovery, or moderation; then prototype only dimensions with reliable evidence and keep security verdicts distinct.

## Reproduction Assessment

Not applicable; this issue proposes a new evaluation and discovery capability rather than reporting broken existing behavior.

## Solution Assessment

No, not yet: publishing one composite score is not sufficiently defined or demonstrably safer than separate explainable signals, so an RFC and evidence model should precede implementation.
