## Summary

Keep open: current main has security audit surfaces and a narrow publish-time quality gate, but it does not implement the requested public, multidimensional skill quality model. This remains product and security-sensitive trust work for maintainers.

## What This Changes

The issue asks ClawHub to add public skill quality signals beyond safety scanning, covering dimensions such as documentation, claim-vs-reality, maintenance, dependency hygiene, alternatives, and code-pattern analysis.

## Best Possible Solution

Define a product-owned quality-signal model distinct from security and moderation, coordinate it with https://github.com/openclaw/clawhub/issues/2042, then add approved schema, indexed API/search support, UI, backfill, operator docs, and evaluator data-boundary policy.

## Reproduction Assessment

Not applicable: this is a product and trust-model proposal rather than a runtime bug. Source inspection shows current main has only a narrow publish-time quality gate and existing security audits, not the requested public scoring workflow.

## Solution Assessment

Unclear: structured quality signals may be useful, but direct implementation is not yet established as the best path. Maintainers should first decide scoring semantics, data boundaries, rollout, and how quality stays separate from safety verdicts.
