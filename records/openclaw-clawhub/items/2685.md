## Summary

Keep open: current main has a merged install backfill for estimated all-time counts, but it does not restore or prove correctness for the reported `installsCurrent` reset, and the latest comment adds a second production example where old current-count data appears lost while new installs now accumulate.

## What This Changes

The issue asks ClawHub to investigate and restore current install counts after several published skills' `installsCurrent` values reset to zero.

## Best Possible Solution

Audit the affected production rows and, if `userSkillInstalls` is trustworthy, run a guarded dry-run-first repair that reconciles `statsInstallsCurrent` and denormalized search/publisher rows without changing unrelated install semantics.

## Reproduction Assessment

No high-confidence live reproduction is available from the checkout alone. Source inspection does show that if stored current counters drift to zero while an install row already exists, reinstalling only refreshes `lastSeenAt` and does not enqueue a new count event.

## Solution Assessment

Unclear as a pure code fix. The merged backfill is useful for estimated all-time install recovery, but it is not the best complete solution for this issue because it deliberately avoids `userSkillInstalls` and does not patch `statsInstallsCurrent`.
