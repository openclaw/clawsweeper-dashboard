## Label Justifications

- `P1`: Affected users can be locked out of ClawHub login and CLI-token access after deleting their account.
- `impact:security`: The unresolved behavior changes account deletion guarantees, identity reactivation, ownership, and token revocation boundaries.
- `impact:auth-provider`: The reported failure is in GitHub OAuth sign-in and ClawHub identity handling after account deletion.
- `impact:session-state`: The user can return from provider auth without an active ClawHub user state, leaving the UI in the logged-out sign-in branch.

## Summary

Keep open. Current main still intentionally blocks deleted or deactivated GitHub identities from becoming active users again, while this reported recovery/relink/recreate behavior is still a product and security policy decision with multiple related open reports.

## What This Changes

The issue asks ClawHub to allow a user who deleted their account to log back in with GitHub and regain access to account and CLI-token flows instead of staying in the signed-out button state.

## Best Possible Solution

End with one explicit deleted-account policy across OAuth callback handling, active-user gates, token creation, settings/header/CLI UI, docs, and regression tests: either irreversible deletion with clear support messaging or an audited restore/relink/recreate path.

## Reproduction Assessment

Yes, at source level. Deletion writes inactive/purged state and revokes tokens, OAuth reauth rejects deleted/deactivated users, active-user gates then return no user, and the header falls back to the sign-in button branch.

## Solution Assessment

No autonomous recreate or relink patch is the best next step yet. The maintainable fix requires an explicit product/security policy for restore, relink, recreate, or permanent rejection before code changes are safe.
