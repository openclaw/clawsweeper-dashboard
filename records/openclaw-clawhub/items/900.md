## Label Justifications

- `P1`: Deleted-account OAuth lockout blocks real users from ClawHub login and CLI token access, but the fix requires maintainer policy review rather than emergency runtime repair.
- `impact:security`: The requested recovery or relink behavior changes account deletion, identity verification, and token revocation boundaries.
- `impact:auth-provider`: The failure path is GitHub OAuth reauthentication for identities tied to deleted/deactivated ClawHub users.
- `impact:session-state`: The UI can have an authenticated provider session while users.me returns null, leaving the app in a logged-out active-user state.

## Summary

Keep open. Current main still blocks deleted/deactivated GitHub identities from becoming active users again; the visible error handling has improved, but recovery, relinking, recreation, or permanent rejection remains an unresolved product/security policy decision.

## What This Changes

The issue asks ClawHub to handle login after account deletion so GitHub OAuth no longer leaves the user locked out with the sign-in button still active.

## Best Possible Solution

Make one explicit deleted-account policy and enforce it across OAuth callback handling, active-user gates, CLI token creation, UI messaging, specs/docs, and regression tests.

## Reproduction Assessment

Yes, at source level. Deletion writes deactivatedAt/purgedAt and revokes tokens, OAuth reauth throws for deleted/deactivated users, users.me returns null through the active-user gate, and CLI token creation requires an active user; I did not run a live production OAuth deletion flow.

## Solution Assessment

No autonomous patch is the best next step yet. Restoring, relinking, recreating, or permanently rejecting deleted GitHub identities changes an auth identity boundary and needs maintainer product/security policy first.
