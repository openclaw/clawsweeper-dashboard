## Label Justifications

- `P1`: Affected users cannot regain ClawHub login or CLI-token access after deleting their account.
- `impact:security`: The unresolved decision changes identity reactivation, irreversible deletion, and token revocation boundaries.
- `impact:auth-provider`: The reported failure is in GitHub OAuth sign-in and ClawHub identity handling.
- `impact:session-state`: A provider-level auth attempt can return without an active ClawHub user state and keep the user in logged-out UI.

## Summary

Keep open. Current main still enforces irreversible deleted-account lockout for the same GitHub identity, while the requested restore/relink/recreate behavior remains a product and security policy decision shared by several open reports.

## What This Changes

The issue requests that a self-deleted ClawHub account can sign in with GitHub again or otherwise recover access instead of returning to a logged-out/sign-in state.

## Best Possible Solution

End with one explicit deleted-account policy: either preserve irreversible deletion with clear rejection/support messaging everywhere, or add an audited restore/relink/recreate flow that preserves ownership, purge, and token-revocation semantics.

## Reproduction Assessment

Yes, at source level. Deletion writes deactivatedAt/purgedAt and revokes tokens; OAuth reauth rejects deleted/deactivated users; active-user gates and CLI token creation require an active user, though I did not run a live production OAuth deletion flow.

## Solution Assessment

No autonomous recreate/relink patch is the best next step yet. Restoring, relinking, recreating, or permanently rejecting deleted GitHub identities is a product/security policy choice that must be applied consistently across backend gates, UI, CLI, specs, and tests.
