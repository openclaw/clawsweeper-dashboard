## Summary

Keep open. Current main still intentionally blocks self-deleted GitHub-backed accounts from becoming active users again, and ClawHub issue cleanup policy says issues stay open unless maintainers resolve them; the remaining work is a product/security account-lifecycle decision.

## What This Changes

The issue requests that a deleted GitHub-backed ClawHub account can sign in again or be recreated/relinked instead of returning to a logged-out sign-in-button state and blocking CLI token access.

## Best Possible Solution

Maintainers should choose one deleted-account identity policy, then implement it consistently across OAuth callback handling, auth account binding, active-user gates, token creation, settings/header/CLI UX, operator/support tooling, docs, and regression tests.

## Reproduction Assessment

Yes, at source level. Current main sets deactivatedAt/purgedAt on account deletion, rejects the same identity during the OAuth callback, returns no active users.me, and gates CLI token creation on an active user.

## Solution Assessment

No, not as an autonomous cleanup fix. Recreate, restore, relink, or support-only behavior changes the documented irreversible-deletion policy, so maintainers need to decide that account lifecycle first.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes GitHub identity reuse, deletion privacy, token revocation, and ownership semantics.

Concerns:

- **[medium] Define deleted-account recovery policy first:** `specs/security-moderation.md:132`
  - body: The spec currently says user-initiated deletion is irreversible and deleted accounts cannot be restored by logging in again, so allowing recovery or re-registration needs explicit maintainer approval.
  - confidence: 0.95
- **[medium] Preserve identity and ban safeguards:** `convex/auth.ts:37`
  - body: The OAuth callback checks that the incoming identity matches the existing user and separates banned/autobanned legacy records before migration; any recovery path needs equivalent provider-account safeguards.
  - confidence: 0.9
- **[medium] Protect token and ownership semantics:** `convex/users.ts:348`
  - body: Account deletion revokes tokens and purges profile/contact fields while published content remains public, so restore or purge tooling must define token, audit, publisher, and ownership behavior.
  - confidence: 0.88
