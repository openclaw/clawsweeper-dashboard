## Label Justifications

- `P1`: Affected users cannot regain ClawHub login or CLI-token access after account deletion.
- `impact:security`: The unresolved decision changes identity reactivation, deletion guarantees, and token revocation boundaries.
- `impact:auth-provider`: The reported failure is in GitHub OAuth sign-in and ClawHub identity handling.
- `impact:session-state`: A provider-level auth attempt can leave ClawHub without an active me state and keep the user in the sign-in UI.

## Summary

Keep open. Current main still deliberately blocks deleted/deactivated GitHub identities from becoming active users again; the UI now surfaces a permanent-deletion message, but the requested restore/relink/recreate behavior remains a product and security policy decision with several related reports still open.

## What This Changes

The issue requests that a user who deleted a ClawHub account can sign in again with GitHub, regain account state, and create CLI tokens instead of remaining in the sign-in flow.

## Best Possible Solution

End with one explicit deleted-account policy: either keep irreversible deletion with clear support/rejection messaging everywhere, or add an audited restore/relink/recreate flow that preserves ownership, purge, and token-revocation semantics.

## Reproduction Assessment

Yes, at source level. Deletion writes deactivatedAt/purgedAt and revokes tokens; OAuth reauth rejects deleted/deactivated users; active-user gates and CLI token creation require an active user, though I did not run a live production OAuth deletion flow.

## Solution Assessment

No autonomous patch is the best next step yet. Recreating, relinking, restoring, or permanently rejecting deleted GitHub identities is an account lifecycle and security policy choice that must be applied consistently across backend gates, UI, CLI, specs, and tests.
