## Label Justifications

- `P1`: Affected users are blocked from ClawHub sign-in and CLI-token access after account deletion, which is an urgent auth workflow problem.
- `impact:security`: The remaining decision changes deleted-account recovery, identity binding, privacy, audit, and token revocation boundaries.
- `impact:auth-provider`: The report is specifically about GitHub OAuth mapping back to a deleted ClawHub account.
- `impact:session-state`: Provider authentication can complete while ClawHub still has no active me/session state for the user.

## Summary

Keep open. Current v0.16.0 still intentionally blocks deleted/deactivated GitHub identities from becoming active ClawHub users again, and the remaining recovery/relink behavior is a product and security policy decision rather than conservative cleanup.

## What This Changes

The issue asks ClawHub to let a user sign in or recover/relink after deleting an account instead of leaving GitHub sign-in active and CLI tokens unavailable.

## Best Possible Solution

Define one maintainer-approved deleted-account policy, then implement it consistently across OAuth, account deletion, active-user gates, CLI token creation, UI messaging, docs/specs, audit, and tests.

## Reproduction Assessment

Yes, at source level. Deletion writes deactivatedAt/purgedAt, OAuth reauth rejects deleted/deactivated users, users.me returns null through the active-user gate, and CLI token creation requires an active user; I did not run a live production OAuth deletion flow.

## Solution Assessment

No autonomous patch is the best next step yet. Recreate, relink, restore, or permanently reject deleted GitHub identities is an account lifecycle and security policy choice that must be applied consistently.
