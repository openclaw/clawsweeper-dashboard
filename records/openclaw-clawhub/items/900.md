## Summary

Keep open. Current main still rejects deleted/deactivated accounts from OAuth re-entry and CLI token creation; the behavior is now documented as irreversible and partly surfaced with clearer auth-error UI, so resolving the report needs maintainer product/security judgment rather than cleanup close.

## What This Changes

The issue asks for a deleted GitHub-linked ClawHub account to be able to sign in again, relink, or be recreated so the user can access the account and CLI tokens.

## Best Possible Solution

Pick one explicit deleted-account policy, then implement it consistently: either a deliberate audited restore/relink/recreate flow, or clearer irreversible-deletion support messaging across OAuth, header/settings, CLI auth, and docs with regression coverage.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: deleteAccount sets `deactivatedAt`, OAuth reauth throws, active-user queries return no user, and CLI token creation requires `requireUser`. I did not run a live OAuth flow with the reporter's GitHub identity.

## Solution Assessment

No, directly allowing OAuth to recreate or relink the account is not the best automated fix without a policy decision. The safer path is a maintainer decision on deletion semantics followed by one coherent backend, UI, CLI, docs, and test update.
