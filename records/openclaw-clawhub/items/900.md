## Label Justifications

- `P1`: The issue blocks affected users from ClawHub login and CLI-token access after account deletion.
- `impact:security`: The remaining decision changes deleted-account recovery, identity reactivation, and token revocation boundaries.
- `impact:auth-provider`: The failure is in GitHub OAuth sign-in and ClawHub auth identity handling.
- `impact:session-state`: The user can be authenticated at the provider layer while ClawHub has no active `me` state.

## Summary

Keep open. Current main still enforces irreversible deleted-account lockout for GitHub OAuth and CLI token creation; the UI now surfaces a clearer deleted-account message, but the requested restore/relink/recreate behavior remains a product and security policy decision, with several related reports still open.

## What This Changes

The issue asks for deleted ClawHub accounts linked to GitHub to re-login or relink cleanly instead of returning to a live sign-in button and blocked CLI-token access.

## Best Possible Solution

End with one explicit deleted-account policy: either keep irreversible deletion with clear support/rejection messaging across OAuth, header, settings, CLI, docs, and specs, or add an audited restore/relink/recreate flow that preserves ownership, purge, and token-revocation semantics with regression tests.

## Reproduction Assessment

Yes, at source level. Deletion writes `deactivatedAt`/`purgedAt`, OAuth reauth rejects deleted/deactivated users, `users.me` returns null through the active-user gate, and CLI token creation requires an active user; I did not run a live production OAuth deletion flow.

## Solution Assessment

No autonomous patch is the best next step yet. Recreating, relinking, restoring, or permanently rejecting deleted GitHub identities is an account lifecycle and security policy choice that must be applied consistently across backend gates, UI, CLI, specs, and tests.
