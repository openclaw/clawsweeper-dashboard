## Summary

Keep open. Current main still treats user-initiated deletion as permanent and blocks the same GitHub identity from becoming an active user again; the newer UI/docs improve the blocked-account message, but the requested recovery/relink/re-registration behavior remains an unresolved account-lifecycle policy decision with several related reports still open.

## What This Changes

The issue requests that a self-deleted GitHub-backed account can log in again, or avoid the live sign-in loop and CLI-token lockout after OAuth returns.

## Best Possible Solution

Define one maintainer-approved deleted-account policy, then implement either a secure audited restore/relink/recreate path or clearer irreversible-deletion support routing across OAuth, CLI token creation, auth bindings, docs, operator tooling, and regression tests.

## Reproduction Assessment

Yes, at source level. Current main sets deactivatedAt/purgedAt on self-deletion, rejects deleted/deactivated users during OAuth completion, returns no active users.me, and keeps CLI token creation behind requireUser; I did not run a live production GitHub OAuth account deletion flow.

## Solution Assessment

No, not as an automatic source-only fix. Same-GitHub restore, relink, or re-registration changes a documented irreversible-deletion policy, so the maintainable path starts with maintainer product/security approval and an audited implementation plan.
