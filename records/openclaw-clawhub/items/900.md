## Summary

Keep open. Current main still blocks deleted/deactivated GitHub identities from becoming active users again, and the remaining recovery/relink behavior is a security-sensitive account-lifecycle policy decision with multiple related reports still open.

## What This Changes

The issue requests that a user who deleted their ClawHub account can sign in again with the same GitHub identity and reach account/CLI token features instead of staying in a sign-in loop.

## Best Possible Solution

Maintainers should choose and document one deleted-account recovery policy, then implement it consistently across OAuth callback handling, auth account binding, active-user gates, token creation, settings/header/CLI UX, specs/docs, operator tooling, and regression tests.

## Reproduction Assessment

Yes, at source level. Current main deactivates and purges the user on deletion, blocks the same identity during OAuth, returns no active users.me, and gates CLI token creation on an active user; I did not run a live production OAuth deletion flow.

## Solution Assessment

No autonomous code change is the best next step yet. Restoring, relinking, recreating, or support-routing deleted identities changes a security-sensitive irreversible-deletion policy, so maintainers should decide that policy first.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes identity reuse, deletion privacy, token revocation, and ownership semantics.

Concerns:

- **[medium] Define deleted-account recovery policy first:** `specs/security-moderation.md:139`
  - body: The internal spec currently says user-initiated deletion is irreversible and deleted accounts cannot be restored by logging in again, while the RFC leaves recovery as an open question; restore, relink, or re-registration needs explicit policy approval before implementation.
  - confidence: 0.95
- **[medium] Preserve OAuth identity and ban safeguards:** `convex/auth.ts:37`
  - body: The OAuth callback blocks deleted/deactivated identities and checks matching existingUserId and ban audit records; any recovery path needs equivalent identity-matching and abuse-prevention behavior.
  - confidence: 0.9
- **[medium] Define token, ownership, and privacy semantics:** `convex/users.ts:338`
  - body: Account deletion revokes API tokens, sets purged state, and clears profile/contact/GitHub-derived fields, so recovery must define whether tokens, publishers, audit history, and ownership are restored, recreated, or operator-managed.
  - confidence: 0.9
