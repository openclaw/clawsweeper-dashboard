## Label Justifications

- `P1`: The issue blocks affected users from ClawHub login and CLI-token access after account deletion.
- `impact:security`: Any recovery path changes deleted-account reactivation, identity ownership, and token revocation boundaries.
- `impact:auth-provider`: The failure is in GitHub OAuth sign-in and Convex/ClawHub auth identity handling.
- `impact:session-state`: The user can complete provider auth while ClawHub still has no active users.me state.

## Summary

Keep open. Current main still intentionally blocks deleted/deactivated GitHub identities from becoming active ClawHub users again; the UI has clearer auth-error messaging now, but the requested restore, relink, or recreate behavior remains an unresolved product and security decision with related reports still open.

## What This Changes

The issue requests that a user who self-deleted a ClawHub account can sign in again with GitHub, or at least no longer lands in a confusing logged-out/sign-in loop.

## Best Possible Solution

End with one explicit deleted-account policy: either keep irreversible deletion with clear support/rejection messaging everywhere, or add an audited restore/relink/recreate flow with ownership, purge, and token-revocation semantics covered by tests.

## Reproduction Assessment

Yes, at source level. Deletion writes deactivatedAt/purgedAt, OAuth reauth rejects deleted/deactivated users, users.me returns null through the active-user gate, and CLI token creation requires an active user; I did not run a live production OAuth deletion flow.

## Solution Assessment

No autonomous patch is the best next step yet. Recreating, relinking, restoring, or permanently rejecting deleted GitHub identities is an account lifecycle and security policy choice that must be applied consistently across backend gates, UI, CLI, specs, and tests.
