## Summary

Keep open: this member-authored draft is not implemented on current main and the PR still has a merge-blocking rate-limit model issue where the new route wrapper can consume quota before handlers that already enforce the correct principal-specific bucket.

## What This Changes

The PR adds a Convex HTTP route-registration rate-limit wrapper, token/IP/shard hardening, package download routing changes, regression tests, and rate-limit docs/spec notes.

## Best Possible Solution

Land a single-pass, principal-aware rate-limit model where each route consumes exactly one intended bucket, package publish tokens are evaluated before generic anonymous limits, and operator rollout notes match the deployed proxy topology.

## Reproduction Assessment

Yes, source inspection is enough to reproduce the review finding: the PR wrapper calls `applyRateLimit` before delegation, and representative V1 handlers still call `applyRateLimit` again. I did not run a live Convex HTTP smoke in this read-only review.

## Solution Assessment

No, not yet. A central wrapper is a plausible layer, but the current implementation must preserve single-pass, principal-aware consumption before it is the best fix.
