## Summary

Keep open: this member-authored draft is not an auto-close candidate, and the patch has a blocking rate-limit ownership issue because the new wrapper can consume generic quota before existing token-aware handlers and then consume again.

## What This Changes

The PR adds centralized HTTP route rate-limit wrapping, forwarded-IP trust changes, sharded counter hardening, and docs/tests for ClawHub API rate limits.

## Best Possible Solution

Use one principal-aware rate-limit path per route, preserving publish-token and authenticated-user buckets while adding pre-work fallback coverage only where handlers were missing it.

## Reproduction Assessment

Yes. Source inspection shows /api/v1/packages POST is wrapped by routeApi and then publishPackageV1Handler applies token-aware rate limiting again.

## Solution Assessment

No. A centralized wrapper is plausible, but it must either opt out already-limited handlers or carry the exact principal-specific options so each request consumes once.
