## Label Justifications

- `P1`: The PR touches public API rate-limiting security behavior and can block real package publish workflows before merge.
- `merge-risk: 🚨 compatibility`: Existing API clients can see lower effective quotas or new 429s because already-limited handlers are wrapped by another limiter.
- `merge-risk: 🚨 security-boundary`: The patch changes which principal owns rate-limit enforcement for public API routes and package publish tokens.
- `merge-risk: 🚨 availability`: Trusted-publisher package publishes can fail before reaching the token-aware publish bucket when the generic shared bucket is exhausted.

## Summary

Keep open. This is a useful security hardening draft and current main has not implemented the central route wrapper, but the proposed wrapper still precharges package publish requests with a generic write bucket before token-aware limiting, so it should not merge as-is.

## What This Changes

This draft PR centralizes ClawHub HTTP API rate limiting, hardens forwarded-IP and shard behavior, adds package publish-token bucket support, and documents/tests the intended rate-limit contract.

## Best Possible Solution

Land a single-pass, principal-aware rate-limit model that applies exactly one bucket per route and preserves publish-token behavior while documenting the operator rollout.

## Reproduction Assessment

Yes. Source inspection of the PR head shows POST /api/v1/packages is registered through routeApi without an override, rateLimitedHttpAction calls applyRateLimit before the handler, and publishPackageV1Handler then applies the token-aware limit.

## Solution Assessment

No. Centralized wrapping is plausible, but the best fix is a single principal-aware enforcement path per route so package publish tokens and already-limited handlers are not precharged by a generic bucket.
