## Label Justifications

- `P1`: The PR touches API rate-limiting security behavior and currently risks blocking real package publish workflows before merge.
- `merge-risk: 🚨 compatibility`: Existing API clients can receive lower effective quotas or new 429s because already-limited handlers are wrapped by another limiter.

- `merge-risk: 🚨 availability`: Trusted-publisher package publishes can fail before reaching the token-aware publish bucket when a shared or missing-IP bucket is exhausted.

## Summary

Keep open: this member-authored draft is not implemented on current main and the PR still has a merge-blocking rate-limit principal issue. Source inspection shows `POST /api/v1/packages` gets a generic route-level write limit before the token-aware package publish handler runs, so trusted-publisher publishes can be blocked by the wrong bucket and successful requests can consume quota twice.

## What This Changes

The branch wraps ClawHub HTTP API routes with default read/write rate limiting, adds token, forwarded-IP, and sharded-counter hardening, and documents/tests the new rate-limit contract.

## Best Possible Solution

Land a single-pass, principal-aware rate-limit model that applies exactly one bucket per route and preserves publish-token behavior while documenting the operator rollout.

## Reproduction Assessment

Yes. Source inspection of the PR head shows `routeApi` wraps `POST /api/v1/packages` with a default write limit before `publishPackageV1Handler` calls `applyRateLimit` with `allowPackagePublishToken`.

## Solution Assessment

No. A centralized wrapper is a plausible layer, but it must preserve exactly one principal-aware consumption per request or explicitly opt out handlers that already enforce the correct bucket.
