## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve repeated Set-Cookie headers when wrapping responses:** `convex/lib/httpRouteRateLimit.ts:103-106`
  - body: Because `installRateLimitedRoutes` is installed before `auth.addHttpRoutes(http)`, successful auth responses now pass through this wrapper. Rebuilding the response with `mergeHeaders(rate.headers, response.headers)` sends the existing `Headers` through `Object.fromEntries`, so repeated `Set-Cookie` values collapse to one header; OAuth or sign-out flows that set/clear multiple cookies can silently lose session state. Please preserve multi-value headers or bypass rebuilding cookie-bearing responses, and add a regression test with multiple `Set-Cookie` values.
  - confidence: 0.88

## Risks / Open Questions

- Successful metered auth responses can lose repeated Set-Cookie headers because the wrapper rebuilds responses through a record-shaped header merge.
- Anonymous callers without trusted forwarded headers will move to shared route-family fallback buckets, which is safer but can create new 429 behavior for deployments that previously depended on cf-connecting-ip by default.
- Persistent counter write contention now returns retryable 503 responses, so maintainers need to accept the fail-closed availability tradeoff for hot buckets.
- TRUST_FORWARDED_IPS remains an ops-sensitive switch: enabling it while the direct Convex site origin is reachable lets clients spoof forwarded client-IP headers.
