## Summary

Keep open: current main still renders and copies the bare slug install command even when owner data is available. The fix should not be UI-only because current docs/API/CLI surfaces still describe slug-only resolution and the specs point to both `owner/slug` and `@owner/name` as scoped locator shapes.

## What This Changes

The issue requests changing the skill page install command code block to use an owner-scoped skill locator.

## Best Possible Solution

Define one scoped skill locator contract, implement parser/resolver support with safe slug compatibility, then update the install UI, docs, and tests to use the canonical scoped command.

## Reproduction Assessment

Yes. Source and tests show the current skill command card renders and copies `openclaw skills install weather` even when `ownerHandle` is supplied.

## Solution Assessment

Unclear as requested. Owner-scoped install commands are the right direction, but the maintainable fix is a coordinated locator contract across UI, CLI/API parsing, docs, and compatibility behavior.
