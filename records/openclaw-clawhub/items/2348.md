- `impact:other`: The affected problem class is install-command correctness and ambiguity, which is maintainer-visible but outside the owned data-loss/security/session taxonomy.

## Summary

Keep open. Current main still renders and copies the skill install CLI command as a bare slug, and the related owner-scoped implementation at https://github.com/openclaw/clawhub/pull/2299 is still open and conflicting; this collaborator-authored issue should stay with maintainers.

## What This Changes

Request that the skill detail install code block and copy action use an owner-scoped skill target instead of a bare skill slug.

## Best Possible Solution

Agree on the `@owner/slug` install-locator contract, then land compatible UI, docs, tests, and resolver behavior through a revised https://github.com/openclaw/clawhub/pull/2299 or a narrower maintainer-sponsored follow-up.

## Reproduction Assessment

Yes. Source inspection on current main shows the compact install card formats and copies `openclaw skills install <slug>`, and tests assert that bare-slug behavior for a skill with owner `steipete`.

## Solution Assessment

Unclear as a UI-only change. The maintainable solution is to settle the scoped locator contract and make the displayed command match supported OpenClaw/ClawHub behavior, not just change the code block text.
