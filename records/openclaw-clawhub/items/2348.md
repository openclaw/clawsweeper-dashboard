## Label Justifications

- `P2`: The current command is compatible but ambiguous across publishers, making this a normal-priority install ergonomics and identity improvement rather than an outage.
- `impact:other`: The issue concerns trustworthy skill identity and copied installation instructions, which does not fit the more specific owned impact categories.

## Summary

Current main intentionally copies the CLI-compatible bare slug, so the requested owner-scoped command remains unimplemented; this issue should stay open because adopting `@owner/slug` requires coordinated OpenClaw CLI support and explicit maintainer agreement.

## What This Changes

Change the skill-page OpenClaw install code block from a bare slug to a canonical owner-scoped skill reference.

## Best Possible Solution

Add and validate one canonical owner-scoped reference—preferably the agreed plugin-style `@owner/slug` form—in the OpenClaw CLI resolver first, then align the ClawHub command card, prompts, help, public docs, design spec, and regression tests without removing bare-slug compatibility unnecessarily.

## Reproduction Assessment

Yes. Current main and its focused tests directly show that the copied command contains only the bare slug; prior issue https://github.com/openclaw/clawhub/issues/1807 also provides a concrete runtime reproduction showing why plain `owner/slug` was removed.

## Solution Assessment

No, changing only the code block is not the best solution because it could publish unsupported syntax. The maintainable path is to establish scoped resolution in the OpenClaw CLI and then update every ClawHub-facing command and document consistently.
