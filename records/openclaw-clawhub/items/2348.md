## Summary

Keep open: current main still uses the bare skill slug for the copyable CLI command, while owner-scoped install targets require a maintainer decision because the shipped CLI contract, prior fix history, and scoped-name specs do not agree yet.

## What This Changes

Requests that skill detail install commands use owner-qualified `openclaw skills install <owner-slug>/<skill-slug>` targets.

## Best Possible Solution

Choose one canonical scoped install locator, preserve compatible bare-slug behavior where needed, then update the UI command, copy behavior, CLI/API parsing, tests, and user docs together.

## Reproduction Assessment

Yes from source inspection: the skill header passes owner data into `SkillCommandLineCard`, but that component formats and copies the CLI command from the bare `slug`. I did not run the browser UI in this read-only review.

## Solution Assessment

Unclear until maintainers choose the install locator contract. A UI-only `owner/slug` change is not the best fix while current CLI/API paths reject slash-qualified skill install targets and the orgs spec points at `@owner/name`.
