## Summary

Keep open: current main still renders and copies the bare slug install command even when owner context is available, while the CLI/docs still describe slug-only install and the specs disagree on the canonical scoped locator shape. This needs maintainer product/API alignment before changing the UI copy alone.

## What This Changes

Request that skill detail install command code blocks use an owner-scoped command such as `openclaw skills install <owner-slug>/<skill-slug>` instead of a bare slug.

## Best Possible Solution

Define one scoped skill locator contract, implement parser/resolver support with safe slug compatibility and ambiguity handling, then update the install UI, docs, and tests to use that canonical command.

## Reproduction Assessment

Yes. Source and tests show the current skill command card renders and copies `openclaw skills install weather` even when `ownerHandle` is supplied.

## Solution Assessment

Unclear as requested. Owner-scoped install commands are the right direction, but the maintainable fix is a coordinated locator contract across UI, CLI/API parsing, docs, and compatibility behavior.
