## Summary

Keep open: this is a fresh, source-reproducible install UI/API contract issue, and ClawHub policy keeps issues open rather than auto-closing them. The reported page command still uses the bare skill slug even though nearby helper code and the install-surface spec already point toward owner-qualified targets.

## What This Changes

The issue requests owner-qualified skill install commands such as `openclaw skills install nickludlam/ghst` on skill pages.

## Best Possible Solution

Align the canonical install locator, then update the skill page command, copy behavior, tests, and user-facing docs so the visible command matches the supported scoped install contract.

## Reproduction Assessment

Yes from source inspection: the skill header passes owner data into `SkillCommandLineCard`, but that component formats and copies the CLI command from the bare `slug`. I did not run the browser UI in this read-only review.

## Solution Assessment

Unclear until maintainers confirm the canonical machine locator: the install-surface spec says `owner/slug`, the orgs spec says `@owner/name`, and current ClawHub CLI/API paths remain bare-slug based.
