## Label Justifications

- `P2`: This is a focused user-facing install-command and namespace-correctness issue with limited blast radius, but it needs product/API alignment before implementation.

## Summary

Keep open: current main still renders and copies the direct OpenClaw install code block as a bare slug, and changing it safely is broader than UI copy because the current CLI install path rejects slash locators and writes filesystem/lockfile state by bare slug.

## What This Changes

Request to render the skill-detail OpenClaw install code block as an owner-scoped skill locator such as `owner/skill` instead of a bare skill slug.

## Best Possible Solution

Define one scoped skill locator contract, preserve safe bare-slug compatibility where unambiguous, then update the install UI, CLI/API parser and resolver, docs, and regression tests together.

## Reproduction Assessment

Yes. Source and tests show the command card renders and copies `openclaw skills install weather` even when `ownerHandle` is supplied, so the requested scoped-codeblock behavior is absent on current main.

## Solution Assessment

No as a UI-only change. Owner-scoped install commands are a reasonable direction, but the maintainable fix must coordinate locator parsing, safe filesystem and lockfile behavior, API resolution, docs, and compatibility tests.
