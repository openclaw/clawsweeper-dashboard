## Summary

Keep open: current main and v0.20.0 still render and copy bare-slug install commands even when owner data is present. The scoped `@owner/slug` direction is in an open compatibility-sensitive PR, and this issue is collaborator-authored, so cleanup should not close it automatically.

## What This Changes

The issue asks for the skill detail install code block and copy action to use an owner-scoped `@owner/slug` install target instead of a bare skill slug.

## Best Possible Solution

Land one scoped locator contract for `@owner/slug` across UI copy, CLI/API resolution, docs, and regression tests while preserving unambiguous bare-slug compatibility.

## Reproduction Assessment

Yes. Source inspection shows current main renders and copies `openclaw skills install weather` even when owner props are present; I did not run a live browser flow in this read-only review.

## Solution Assessment

No, not as a codeblock-only copy change. The maintainable path is to align the scoped locator parser, ambiguity rules, docs, and UI tests together.
