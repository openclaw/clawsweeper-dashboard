## Label Justifications

- `P2`: The issue is a focused install-command and locator correctness problem with real user-facing confusion, but it needs product/API alignment rather than emergency handling.

## Summary

Keep open: current main still renders and copies the bare `openclaw skills install <slug>` command on skill pages, while owner-scoped install remains an unsettled CLI/API/product contract rather than a safe one-line UI cleanup.

## What This Changes

The issue requests that skill-page install commands use an owner-qualified locator such as `openclaw skills install <owner-slug>/<skill-slug>` instead of a bare skill slug.

## Best Possible Solution

Settle one canonical scoped install locator across OpenClaw, ClawHub API/CLI, and docs, then update the skill detail command, copy behavior, and tests to match that supported contract.

## Reproduction Assessment

Yes. Current main passes owner data into the command card, but the CLI tab formats and copies `openclaw skills install ${slug}`; I verified this by source and test inspection rather than by running the browser UI.

## Solution Assessment

Unclear. Owner-scoped installation is the right direction only after maintainers choose `owner/slug` versus `@owner/name` and make the CLI/API accept that locator; changing the page command alone would be brittle.
