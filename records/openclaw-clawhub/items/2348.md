## Summary

Keep open under the ClawHub issue policy: current main still renders and copies a bare-slug install command, while switching the UI to owner-scoped syntax would currently advertise a locator the CLI rejects and needs a product/API compatibility decision first.

## What This Changes

The issue requests that skill-page install code blocks use an owner-scoped locator like `<owner-slug>/<skill-slug>` instead of a bare skill slug.

## Best Possible Solution

Define one canonical scoped skill locator contract, keep bare-slug compatibility only where unambiguous, then update the install UI, CLI/API parser and resolver, docs, and regression tests together.

## Reproduction Assessment

Yes. Source and tests show the command card renders and copies `openclaw skills install weather` even when `ownerHandle` is supplied, while the current CLI normalizer rejects slash-containing install locators.

## Solution Assessment

No as a UI-only change. Owner-scoped install commands may be the right direction, but the maintainable fix must coordinate locator syntax, ambiguity handling, filesystem/lockfile behavior, API resolution, docs, and compatibility tests.
