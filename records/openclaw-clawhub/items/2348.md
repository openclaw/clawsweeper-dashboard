## Label Justifications

- `P2`: The issue is a focused user-facing install-command correctness and namespace problem, but it needs product/API alignment rather than emergency handling.

## Summary

Keep open: current main still renders and copies the bare skill slug in the install code block, while changing it to an owner-scoped locator would cross the UI, CLI/API, docs, compatibility, and ownership-boundary contract rather than being a safe UI-only edit.

## What This Changes

The issue requests that skill install code blocks use an owner-scoped locator such as `<owner-slug>/<skill-slug>` instead of a bare skill slug.

## Best Possible Solution

Settle one canonical scoped install locator across OpenClaw, ClawHub API/CLI, docs, and tests, then keep unscoped slug compatibility only where resolution remains intentional and unambiguous.

## Reproduction Assessment

Yes. Source and tests show current main still renders and copies `openclaw skills install ${slug}` even when owner data is available.

## Solution Assessment

Unclear as requested. Owner-scoped installation is likely the right direction, but the maintainable fix is a coordinated locator contract rather than changing the page command before CLI/API support and compatibility policy exist.
