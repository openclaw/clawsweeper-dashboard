## Summary

Keep open: current main still renders and copies a bare `openclaw skills install <slug>` command, while the requested `@owner/slug` behavior is only represented by a broad open/conflicting implementation candidate and needs maintainer product direction.

## What This Changes

The issue asks for the skill detail install code block and copy action to use an owner-scoped OpenClaw install target such as `@owner/slug` instead of a bare skill slug.

## Best Possible Solution

Land or replace the owner-scoped locator work so the skill detail UI, install resolver expectations, CLI/docs, and ambiguity handling all agree on one canonical command.

## Reproduction Assessment

Yes, at source level. Current main formats and tests the skill detail CLI code block as `openclaw skills install weather`, so the requested owner-scoped command is not present on main.

## Solution Assessment

No for a UI-only copy change. The maintainable solution is an end-to-end scoped locator decision that updates UI copy together with CLI/API resolver behavior, docs, and compatibility handling.
