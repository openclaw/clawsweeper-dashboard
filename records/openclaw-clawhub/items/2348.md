## Label Justifications

- `P2`: Ambiguous bare-slug install instructions can resolve the wrong publisher's skill or fail, but the impact is limited to the skill-install workflow.
- `impact:other`: The issue affects the correctness and trustworthiness of user-facing skill install instructions.

## Summary

Current main still displays and copies a bare skill slug, while the requested `@owner/slug` form is not yet supported end to end by the slug-only install resolver. The issue remains necessary, but it requires an explicit compatibility/product decision and must remain open because its author is a repository collaborator.

## What This Changes

Change the skill-page install code block from a bare slug to the canonical `@owner/slug` skill reference.

## Best Possible Solution

Implement `@owner/slug` as one tested, backward-compatible identity across the skill page, OpenClaw and ClawHub CLI resolution, API lookup, help text, and public docs, with an explicit ambiguity policy for legacy bare slugs.

## Reproduction Assessment

Yes. Current main constructs and tests `openclaw skills install <slug>` even though the component has owner data; source also shows the current install resolver is slug-only.

## Solution Assessment

No, a code-block-only string change is not the best complete solution. The UI should change only with compatible scoped lookup, CLI help, docs, ambiguity handling, and regression coverage.
