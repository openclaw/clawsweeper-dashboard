## Label Justifications

- `P2`: The install reference remains ambiguous or inconsistent for owner-namespaced skills, but the impact is bounded to install UX and compatibility.
- `impact:other`: The copied command can omit the canonical namespace marker and fail to identify the intended publisher unambiguously.

## Summary

Current main has partially moved the install surface toward owner-aware targets, but it still emits unprefixed `owner/slug` rather than the requested canonical `@owner/slug`, and the coordinated implementation remains an open, conflicting PR. This collaborator-authored issue should remain open for the shared CLI/API compatibility decision instead of spawning a separate UI-only fix.

## What This Changes

Change skill-page install code blocks and copied prompts to use the canonical owner-scoped `@owner/slug` OpenClaw install reference.

## Best Possible Solution

Land one documented `@owner/slug` contract across OpenClaw CLI parsing, ClawHub resolution, UI command copy, prompts, docs, and regression coverage, with explicit legacy behavior for clients that still send slug-only references.

## Reproduction Assessment

Yes at source level: current-main helpers and tests explicitly use unprefixed `owner/slug` and bare-slug forms rather than the requested `@owner/slug` reference.

## Solution Assessment

Yes directionally, but the narrow UI symptom should be solved through the coordinated owner-scoped reference contract rather than by changing display copy ahead of CLI and resolution support.
