## Label Justifications

- `P2`: This is a focused user-facing install-command and namespace correctness issue with limited blast radius, but it needs product/API alignment before implementation.

## Summary

Keep open under the ClawHub issue policy: current main still copies a slug-only OpenClaw install command, while owner-scoped install locators conflict with the current CLI/parser/docs and need a maintainer product/API compatibility decision.

## What This Changes

Request that the skill detail install code block use an owner-scoped command such as `openclaw skills install <owner-slug>/<skill-slug>`.

## Best Possible Solution

Define one canonical scoped skill locator contract, preserve bare-slug compatibility only where unambiguous, then update the install UI, CLI/API parser and resolver, docs, and regression tests together.

## Reproduction Assessment

Yes. Source and tests show the command card renders and copies `openclaw skills install weather` even when `ownerHandle` is supplied, while the CLI rejects slash-containing install locators.

## Solution Assessment

No as a UI-only change. Owner-scoped install commands may be the right direction, but the maintainable fix must coordinate locator parsing, ambiguity handling, filesystem/lockfile behavior, API resolution, docs, and compatibility tests.
