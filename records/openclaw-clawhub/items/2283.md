## Summary

Keep open: the PR is relevant and mostly aimed at the right trust boundary, but it introduces misleading package restore audit attribution and lacks required real behavior proof before merge.

## What This Changes

The PR hides/restores personal package resources during user ban/deactivation flows, revokes package publish tokens, adds `packages.softDeletedReason`, tests the new batch handlers, and updates moderation specs/changelog.

## Best Possible Solution

Keep the package cleanup design, but thread the unban actor through package restore batches for accurate audit logs and require redacted live proof before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: current main's user ban/delete/unban paths do not call package cleanup, and the PR diff exposes the new audit attribution bug without needing a runtime setup.

## Solution Assessment

Is this the best way to solve the issue? No, not as submitted: the package batch approach is maintainable, but restore audits should use the admin unban actor rather than the restored owner.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Record the unban actor for package restore audits:** `convex/packages.ts:3000-3001`
  - body: `restorePackageDoc` writes `params.actorUserId` to the `package.undelete` audit log, but the new unban batch passes the restored owner as that actor. Package audit history will say the previously banned owner restored their own package when the admin unban flow actually did it. Thread the unban actor through this batch and use that actor/role for the restore audit.
  - confidence: 0.87
