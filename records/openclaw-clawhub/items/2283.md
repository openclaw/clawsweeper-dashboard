## Summary

Keep open. Current main still lacks package-resource sanction handling, and the newest PR head has a blocking stale-continuation bug: scheduled package ban pages keep `allowActiveOwnerBeforeCommit`, so a delayed page can hide packages and revoke publish tokens after the user has already been unbanned.

## What This Changes

The PR adds package soft-delete reasons and sanction batches so personal package/plugin resources are hidden or restored with user bans/deletions, including token revocation, tests, spec text, changelog text, and management confirmation copy.

## Best Possible Solution

Expire the in-flight active-owner override before scheduled continuations can cross an unban boundary, add multi-page stale-after-unban coverage, and land only after maintainers accept the personal-versus-org package sanction policy.

## Reproduction Assessment

Yes, at source level. The PR test asserts scheduled continuations carry `allowActiveOwnerBeforeCommit`, and the handler treats an active owner as valid whenever that flag is present, so a later page after unban will not stale out.

## Solution Assessment

No. The overall direction is plausible, but the in-flight-ban flag must not survive into continuations that can run after unban; requiring the committed matching ban for later pages is the safer maintainable shape.
