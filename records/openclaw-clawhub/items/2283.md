## Summary

Keep this PR open: it addresses a real moderation/security gap and now has usable live Convex proof, but source review found blocking unban edge cases that can restore unrelated package releases or apply late ban pages after an unban.

## What This Changes

This PR adds package soft-delete reasons and ban/account-deletion batch mutations so personal package/plugin resources and package publish tokens are hidden or revoked with user sanctions while org-owned packages are preserved, plus tests, spec updates, UI confirmation copy, and a changelog entry.

## Best Possible Solution

Merge this after package unban restores are narrowed to resources hidden by the matching ban and scheduled ban pages cannot apply after an unban, while preserving the intended personal-resource sanction policy and org-owned package exception.

## Reproduction Assessment

Yes, from source inspection. A package with a release soft-deleted before the ban will have that unrelated release restored by the new unban path, and a multi-page ban can keep the active-owner allowance after an unban because scheduled args preserve the flag.

## Solution Assessment

No, not yet. The direction is maintainable, but the implementation needs tighter matching-ban restore semantics and stale-batch guards before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restore only matching-ban package releases:** `convex/packages.ts:3065-3070`
  - body: The unban batch first checks the package's ban timestamp, but then delegates to restorePackageDoc, whose release loop clears softDeletedAt on every release for that package. If one release was already hidden before the user ban, unbanning the account will publish that unrelated release too. Mirror the matching-timestamp behavior used by the autoban remediation path or add a restore mode that only clears releases hidden by this ban.
  - confidence: 0.91
- **[P1] Stop stale ban pages after unban:** `convex/packages.ts:3019`
  - body: scheduleNextBatchIfNeeded copies the full args object into follow-up pages, so allowActiveOwnerBeforeCommit remains true after the first page. If an admin unbans a user while a large package ban is still paging, owner.deletedAt becomes undefined and the active-before-commit branch still allows later pages to hide packages and revoke tokens after the unban. Drop or narrow that allowance once the original ban commit is visible and add a ban-then-unban pagination regression test.
  - confidence: 0.87
