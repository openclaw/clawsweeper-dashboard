## Label Justifications

- `P1`: The PR changes account-ban enforcement for package availability and has a merge-blocking moderation-boundary defect.
- `merge-risk: 🚨 compatibility`: Banned or deleted personal package owners would lose package catalog availability and package publish tokens under this new behavior.
- `merge-risk: 🚨 security-boundary`: The unban restore path can reactivate release rows that were soft-deleted for reasons outside the matching ban.

## Summary

Keep open: the PR has useful live Convex proof and the sanction-bound package behavior is plausible, but the current unban path can over-restore package releases that were hidden before the matching ban.

## What This Changes

The PR hides personal package resources and revokes package publish tokens during account ban/deletion, then restores matching ban-hidden personal packages on unban while preserving org-owned packages.

## Best Possible Solution

Land the sanction-bound personal package policy only with a release-bounded unban restore that preserves unrelated, malicious, quarantined, or revoked release soft-deletes and has focused regression coverage.

## Reproduction Assessment

Yes by source inspection. A package with softDeletedAt equal to the ban timestamp and softDeletedReason user.banned plus an older already-soft-deleted release will pass the new unban filter, then restorePackageDoc clears softDeletedAt on both releases.

## Solution Assessment

No, not yet. The product direction may be right, but the maintainable implementation should use timestamp and scan-bounded release restoration like the existing autoban remediation path instead of the generic package restore helper.
