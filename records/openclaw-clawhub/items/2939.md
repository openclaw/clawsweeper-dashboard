## Summary

Keep open: current main still has a plausible stats-freshness bug where the skill detail page can show a recent-period download total higher than the all-time counter. The issue is distinct from older downloads-vs-installs and frozen-counter reports because the current Downloads graph exists, but its daily-stat source can outpace the skill document all-time counter.

## What This Changes

The issue reports that a skill page can show an all-time download total lower than the recent-period download total because the all-time value appears stale.

## Best Possible Solution

Enforce the invariant that an all-time download total cannot appear lower than any recent-period total, preferably by improving skill stat document sync freshness/backlog handling and adding a focused detail-page regression guard.

## Reproduction Assessment

Yes for the visible symptom: the issue screenshots show 30d downloads greater than all-time downloads for the same skill, and the public API still reports the lower all-time value. Source inspection gives a plausible current-main cause because daily trend rows refresh separately from the slower all-time skill document counter.

## Solution Assessment

Unclear as an implementation choice: a UI lower-bound guard would fix the visible contradiction, but the more complete fix likely belongs in the backend stat-sync freshness/backlog path. The safest next step is maintainer selection of that repair shape before changing cron/drain behavior.
