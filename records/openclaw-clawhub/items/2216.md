## Summary

Keep open. This is a current product and backend architecture RFC, and current main still has only skill-specific trending rather than a shared Weekly/Monthly popularity contract for publishers, skills, and plugins.

## What This Changes

Add Weekly, Monthly, and All time popularity tabs for publishers, skills, and plugins backed by consistent temporal popularity data.

## Best Possible Solution

Settle the shared period semantics and attribution rules first, then add bounded package/publisher daily rollups, snapshot-pinned weekly/monthly leaderboards, and UI tabs backed by that contract.

## Reproduction Assessment

Not applicable. This is an RFC for new discovery behavior, and source inspection confirms the shared period contract is absent rather than an existing contract failing.

## Solution Assessment

Yes. The RFC-first direction is the maintainable path because a UI-only implementation would create misleading rankings before package and publisher temporal rollups exist.
