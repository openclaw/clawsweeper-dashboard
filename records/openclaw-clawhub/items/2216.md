## Summary

Keep open. Current main still lacks the shared weekly/monthly popularity contract across publishers, skills, and plugins; the issue is a product/data-model RFC rather than an already-implemented cleanup target.

## What This Changes

RFC proposes weekly, monthly, and all-time popularity tabs for publishers, skills, and plugins backed by a shared temporal popularity data contract.

## Best Possible Solution

Define the shared period semantics and attribution rules first, then implement package and publisher daily stats plus entity-specific snapshot-pinned leaderboards before adding the browse tabs.

## Reproduction Assessment

Not applicable. This is an RFC for a new discovery capability; source inspection confirms current main still lacks the requested shared weekly/monthly contract.

## Solution Assessment

Yes. The RFC-first direction is the maintainable path because a UI-only implementation would risk misleading publisher and plugin popularity rankings before the backend data contract exists.
