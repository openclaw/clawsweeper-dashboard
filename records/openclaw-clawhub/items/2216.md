## Summary

Current main still lacks weekly/monthly popularity for packages and publishers and has no unified period-aware browse contract, while no separate issue or PR currently owns the work; this remains a valid product RFC requiring maintainer decisions before implementation.

## What This Changes

Requests weekly, monthly, and all-time popularity across publisher, skill, and plugin browse surfaces, backed by shared temporal statistics and stable leaderboard pagination.

## Best Possible Solution

Approve one explainable period metric and ownership-attribution policy, persist indexed daily package and publisher rollups, add snapshot-stable temporal browse APIs, and only then expose semantically aligned tabs on all three pages.

## Reproduction Assessment

Not applicable: this requests new discovery behavior rather than reporting broken established behavior, and current-source inspection confirms the described capability is absent.

## Solution Assessment

Yes as a staged architectural direction, but it is not implementation-ready until maintainers decide scoring, attribution, backfill or warm-up policy, snapshot retention, and plugin-family semantics.
