## Summary

Keep open. This is a current product RFC for a new temporal discovery capability, and current main still lacks the shared period popularity contract needed across publishers, skills, and plugins.

## What This Changes

RFC requesting Weekly, Monthly, and All time popularity tabs for publishers, skills, and plugins, backed by consistent temporal ranking data.

## Best Possible Solution

Settle shared period semantics first, then add bounded package and publisher daily rollups plus snapshot-pinned weekly/monthly leaderboards before adding the UI tabs.

## Reproduction Assessment

Not applicable. This is an RFC for new discovery behavior; source inspection confirms current main lacks the shared period contract, but no existing documented behavior is broken.

## Solution Assessment

Yes. The RFC-first direction is the maintainable path because a UI-only implementation would risk misleading publisher and plugin popularity rankings before the backend data contract exists.
