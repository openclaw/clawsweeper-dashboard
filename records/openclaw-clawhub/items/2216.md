## Summary

Current main still lacks period-aware popularity across publishers, skills, and plugins: it has a skill-only trending foundation, while publisher discovery remains all-time and plugin browse remains non-temporal. This is a coherent, necessary product RFC, but its scoring, attribution, plugin identity, backfill, and pagination choices require explicit maintainer decisions before implementation.

## What This Changes

Add consistent weekly, monthly, and all-time popularity browsing for publishers, skills, and plugins using scalable temporal rollups and stable leaderboard pagination.

## Best Possible Solution

Choose one explainable V1 score and event-attribution policy, then land indexed package and publisher daily rollups plus entity-specific snapshot-pinned leaderboard APIs before adding coordinated period controls and cursor resets to all three browse pages.

## Reproduction Assessment

Not applicable. Current main follows its existing all-time and skill-trending behavior; the issue clearly requests a new cross-entity discovery capability rather than reporting a broken contract.

## Solution Assessment

Yes at the architectural level: establishing correct temporal data and pagination before exposing tabs is safer than a partial UI implementation. The exact score, attribution, plugin-family model, low-activity behavior, and backfill policy still need maintainer decisions.
