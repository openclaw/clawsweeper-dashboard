## Summary

Keep open. This is a current product RFC for a new temporal discovery capability; current main still has all-time or updated-date browse contracts and does not yet have the package, publisher, and shared leaderboard data model needed to make Weekly and Monthly mean the same thing across publishers, skills, and plugins.

## What This Changes

The issue proposes Weekly, Monthly, and All time popularity tabs for `/publishers`, `/skills`, and `/plugins`, backed by a shared temporal popularity foundation.

## Best Possible Solution

Settle the shared period semantics first, then implement bounded package and publisher daily rollups plus snapshot-pinned weekly/monthly leaderboards before adding the UI tabs.

## Reproduction Assessment

Not applicable. This is an RFC for new discovery behavior; source inspection confirms current main lacks the shared period contract, but no existing documented behavior is broken.

## Solution Assessment

Yes. The RFC-first direction is the maintainable path because a UI-only implementation would risk misleading publisher and plugin popularity rankings before the backend data contract exists.
