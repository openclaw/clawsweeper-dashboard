## Summary

Keep open. Current main still lacks the shared period-popularity contract, and this is a product/architecture RFC that needs maintainer direction before implementation.

## What This Changes

Request to add Weekly, Monthly, and All time popularity tabs for publishers, skills, and plugins, backed by a shared temporal popularity data contract.

## Best Possible Solution

Settle period semantics first, then add bounded daily package and publisher rollups plus snapshot-pinned weekly/monthly leaderboards before exposing UI tabs.

## Reproduction Assessment

Not applicable. This is an RFC for new discovery behavior; source inspection confirms the shared period contract is absent, but there is no broken existing contract to reproduce.

## Solution Assessment

Yes, the RFC-first direction is the best path. Shipping UI tabs before package and publisher temporal rollups exist would create misleading rankings.
