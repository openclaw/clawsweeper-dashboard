## Summary

Keep open. This is a current product/architecture RFC for new temporal discovery behavior, and current main still has only skill-only trending support rather than a shared Weekly/Monthly contract for publishers, skills, and plugins.

## What This Changes

Request Weekly, Monthly, and All time popularity tabs across publishers, skills, and plugins backed by consistent temporal ranking data.

## Best Possible Solution

Settle the shared period semantics first, then implement bounded package and publisher daily rollups, snapshot-pinned weekly/monthly leaderboards, and UI tabs backed by that contract.

## Reproduction Assessment

Not applicable. This is an RFC for new discovery behavior; source inspection confirms the shared period contract is absent, but there is no broken existing contract to reproduce.

## Solution Assessment

Yes, the RFC-first direction is the maintainable path. A UI-only implementation would risk misleading rankings because package and publisher temporal rollups do not exist yet.
