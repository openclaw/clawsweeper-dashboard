## Summary

Keep open. This is a current product and architecture RFC for new temporal popularity behavior, and current main still has only partial skill-only trending infrastructure rather than a shared Weekly/Monthly contract for publishers, skills, and plugins.

## What This Changes

Request adding Weekly, Monthly, and All time popularity periods to publisher, skills, and plugin browse surfaces after defining a shared temporal leaderboard data contract.

## Best Possible Solution

Settle shared period semantics first, then implement bounded daily package and publisher rollups, snapshot-pinned weekly/monthly leaderboards, and UI tabs backed by that contract.

## Reproduction Assessment

Not applicable. This is an RFC for new discovery behavior; source inspection confirms the requested shared period contract is absent rather than a broken existing behavior.

## Solution Assessment

Yes, the RFC-first direction is the maintainable path. A UI-only implementation would risk misleading rankings because package and publisher temporal rollups do not exist yet.
