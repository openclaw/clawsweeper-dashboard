## Summary

Keep open. This is an active product and architecture RFC for a new temporal popularity feature, and current main still lacks a shared Weekly/Monthly popularity contract for publishers, skills, and plugins.

## What This Changes

Request to add Weekly, Monthly, and All time popularity tabs across publisher, skill, and plugin browse surfaces backed by a consistent temporal ranking data model.

## Best Possible Solution

Settle the shared period semantics first, then implement bounded package and publisher daily rollups, snapshot-pinned weekly/monthly leaderboards, and UI tabs backed by that contract.

## Reproduction Assessment

Not applicable. This is an RFC for new discovery behavior; source inspection confirms the shared period contract is absent, but there is no broken existing contract to reproduce.

## Solution Assessment

Yes, the RFC-first direction is the maintainable path. A UI-only implementation would be narrower visually but would risk dishonest rankings because package and publisher temporal rollups do not exist yet.
