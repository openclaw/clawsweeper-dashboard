## Summary

Keep open. This is a current product/architecture RFC for a new temporal popularity feature, and current main still has only partial skill-only trending infrastructure rather than a shared Weekly/Monthly contract for publishers, skills, and plugins.

## What This Changes

The issue requests period-based Weekly, Monthly, and All time popularity tabs for publisher, skill, and plugin browse surfaces backed by correct temporal ranking data.

## Best Possible Solution

Settle shared period semantics first, then implement bounded daily package and publisher rollups, snapshot-pinned weekly/monthly leaderboards, and UI tabs backed by that contract.

## Reproduction Assessment

Not applicable. This is an RFC for new discovery behavior; source inspection confirms the requested shared period contract is absent, but there is no broken existing contract to reproduce.

## Solution Assessment

Yes, the RFC-first direction is the maintainable path. A UI-only implementation would risk misleading rankings because package and publisher temporal rollups do not exist yet.
