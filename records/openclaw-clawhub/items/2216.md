## Summary

Keep open. Current main still lacks a shared Weekly/Monthly popularity contract for publishers, skills, and plugins, and the remaining work is a product/backend RFC rather than a stale or already-implemented issue.

## What This Changes

The issue requests period-based popularity tabs and backing leaderboard/data contracts for publishers, skills, and plugins.

## Best Possible Solution

Settle the shared period semantics first, then implement bounded package and publisher daily rollups plus snapshot-pinned weekly/monthly leaderboards before adding UI tabs.

## Reproduction Assessment

Not applicable. This is an RFC for new discovery behavior; source inspection confirms current main lacks the shared period contract, but no documented existing behavior is broken.

## Solution Assessment

Yes. The RFC-first path is the best current direction because exposing period tabs before the shared data contract exists would risk misleading publisher and plugin popularity rankings.
