## Summary

Keep open: current main still lacks Weekly/Monthly UI tabs and a shared temporal popularity contract; the existing code has all-time publisher ranking, all-time skill/plugin browse plus a skill-only trending endpoint, and no package/publisher daily stats foundation.

## What This Changes

The RFC proposes Weekly, Monthly, and All time popularity tabs backed by temporal popularity data for publishers, skills, and plugins.

## Best Possible Solution

Keep this as the canonical product/architecture RFC until maintainers decide period semantics, scoring, attribution, and backfill policy, then land the backend leaderboard foundation before UI tabs.

## Reproduction Assessment

Not applicable: this is an RFC for a new discovery capability, not a report of broken existing behavior. Source inspection confirms the requested Weekly/Monthly surfaces are absent, but that is the requested feature gap rather than a reproducible bug.

## Solution Assessment

Yes, treating this as an RFC before implementation is the best current path. The source shows UI-only tabs would be misleading without package/publisher daily stats and period leaderboard contracts.
