## Summary

Keep open. This is a current product/architecture RFC, and current main still lacks package/publisher daily stats plus a shared paginated Weekly/Monthly leaderboard contract, so UI tabs would be premature and potentially misleading.

## What This Changes

Add Weekly, Monthly, and All time popularity periods across publisher, skill, and plugin browse surfaces using a shared temporal ranking contract.

## Best Possible Solution

Settle period semantics and attribution first, then add bounded package and publisher daily rollups plus snapshot-pinned weekly/monthly leaderboards before wiring tabs into the browse UIs.

Not applicable. This is an RFC for new discovery behavior; source inspection confirms the shared period contract is absent, but no existing documented behavior is broken.

## Solution Assessment

Yes. The RFC-first approach is the maintainable path because adding UI-only tabs before package and publisher temporal rollups would make popularity rankings misleading.
