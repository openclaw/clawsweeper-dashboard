## Summary

Current main still does not implement a shared Weekly/Monthly popularity contract for publishers, skills, and plugins. The RFC remains useful and well-scoped at the product level, but permanent scoring, attribution, backfill, and pagination choices require maintainer judgment before the cross-cutting backend and UI work should begin.

## What This Changes

Add Weekly, Monthly, and All time popularity modes with consistent ranking semantics and stable pagination across publisher, skill, and plugin browse pages.

## Best Possible Solution

Record one explainable period, scoring, ownership-attribution, and snapshot contract in `specs/`; then add indexed package and publisher daily rollups with safe backfills, build snapshot-pinned weekly/monthly APIs while preserving current all-time paths, and finally add the three browse-page controls and end-to-end validation.

## Reproduction Assessment

Not applicable: this is an RFC for a new discovery capability rather than broken established behavior. Source inspection gives a high-confidence check that the shared Weekly/Monthly contract and required package/publisher temporal data remain absent on current main.

## Solution Assessment

Yes at the architectural level: completing the shared temporal data contract before exposing tabs is safer and more maintainable than UI-only or skill-only ranking. The implementation cannot be finalized until maintainers choose the permanent scoring, attribution, backfill, threshold, and snapshot semantics.
