## Summary

Keep open. This is a current product RFC for a new temporal discovery capability; current main still lacks a shared period popularity contract for publishers, skills, and plugins, and the remaining choices are product/data-model decisions rather than cleanup-close material.

## What This Changes

The RFC requests Weekly, Monthly, and All time popularity tabs for publishers, skills, and plugins backed by consistent period-based popularity data.

## Best Possible Solution

Settle the shared period semantics first, then implement bounded package and publisher daily rollups plus snapshot-pinned weekly/monthly leaderboards before adding the UI tabs.

## Reproduction Assessment

Not applicable. This is an RFC for new discovery behavior; source inspection confirms current main lacks the shared period contract, but no existing documented behavior is broken.

## Solution Assessment

Yes. The RFC-first direction is the maintainable path because a UI-only implementation would risk misleading publisher and plugin popularity rankings before the backend data contract exists.
