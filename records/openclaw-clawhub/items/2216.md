## Summary

Current main still lacks period-based popularity across publishers, skills, and plugins: only skills have daily statistics and a leaderboard foundation, while publisher and plugin browsing remains based on all-time totals or update order. This is a valid product RFC that needs explicit metric, attribution, and rollout decisions before staged implementation.

## What This Changes

Add weekly, monthly, and all-time popularity modes to publisher, skill, and plugin browse surfaces using consistent temporal aggregation and stable pagination.

## Best Possible Solution

Approve and record the V1 ranking, attribution, backfill, and snapshot semantics in a focused spec, then land separate daily-rollup, leaderboard/API, and coordinated UI PRs with indexed queries and real Convex runtime validation.

## Reproduction Assessment

Not applicable; this RFC proposes a new discovery capability rather than reporting broken established behavior. Source inspection confirms that no common weekly or monthly contract currently exists across the three surfaces.

## Solution Assessment

Yes as an architectural direction: indexed daily rollups and precomputed leaderboards are safer than request-time scans. The implementation should wait until maintainers settle the open scoring, attribution, warm-up, and package-family semantics.
