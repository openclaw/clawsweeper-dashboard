## Summary

Keep open. This is a current feature/RFC that needs product and architecture decisions; current main still has only all-time or updated browse paths for publishers/plugins plus a separate limited skills trending path, not the shared period-based popularity contract requested here.

## What This Changes

Request to add weekly, monthly, and all-time popularity tabs across publisher, skill, and plugin browse surfaces with consistent temporal ranking data.

## Best Possible Solution

Settle the period semantics first, then add bounded daily package and publisher rollups, snapshot-pinned weekly/monthly leaderboards, and finally wire the three browse UIs to that shared contract.

## Reproduction Assessment

Not applicable. This is an RFC for new discovery behavior, and source inspection confirms the requested period contract is not present rather than exposing a broken existing contract to reproduce.

## Solution Assessment

Yes, the RFC-first direction is the maintainable path. A UI-only implementation would be misleading until the backend rollups, scoring, attribution, and pagination contract are decided.
