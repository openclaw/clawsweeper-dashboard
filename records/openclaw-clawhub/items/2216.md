## Summary

Keep open. Current main still lacks the shared weekly/monthly popularity contract across publishers, skills, and plugins, and this remains an RFC needing maintainer product/data-model decisions rather than cleanup that can close as implemented.

## What This Changes

RFC requests weekly/monthly/all-time popularity tabs for publishers, skills, and plugins backed by temporal stats, publisher/package rollups, and snapshot-pinned leaderboard pagination.

## Best Possible Solution

Define shared period semantics and attribution rules first, then implement package and publisher daily stats plus entity-specific snapshot-pinned leaderboards before adding browse tabs.

## Reproduction Assessment

Not applicable. This is an RFC for a new discovery capability, and source inspection gives high-confidence evidence that current main still lacks the requested shared weekly/monthly browse contract.

## Solution Assessment

Yes. The RFC-first direction is the best maintainable path because a UI-only implementation would risk misleading publisher and plugin popularity rankings before the backend data contract exists.
