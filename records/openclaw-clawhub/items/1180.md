## Summary

Keep open: current main and v0.23.0 still intentionally render the dashboard deployment-drift alert whenever frontend and Convex build SHAs are both present and unequal, so the reported warning remains a deploy-stamping or banner-policy decision rather than an implemented-away issue.

## What This Changes

The issue reports a persistent red dashboard deployment-drift warning when the frontend build SHA and Convex-reported backend build SHA differ even though the dashboard otherwise works.

## Best Possible Solution

Define the production deploy SHA contract first, then either stamp both deployables consistently or adjust the banner severity/copy for intentional partial deploys with matching tests and deploy docs.

## Reproduction Assessment

Yes at source level: current main deterministically sets hasMismatch and renders the alert when non-empty frontend and Convex SHAs differ. I did not recreate the historical production deployment state in a browser session.

## Solution Assessment

No automatic code change is proven best yet. Maintainers should first decide whether the mismatch represents a deploy-stamping violation or an expected partial-deploy state that needs different banner policy.
