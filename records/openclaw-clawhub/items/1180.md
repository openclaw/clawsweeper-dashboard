## Summary

Keep open. Current main still intentionally renders the dashboard deployment drift warning when frontend and Convex build SHAs differ, while the live production state has shifted to Convex reporting the current main SHA and the frontend runtime not exposing `VITE_APP_BUILD_SHA`; that makes this a maintainer/operator policy issue rather than cleanup-close evidence.

## What This Changes

The issue reports a persistent dashboard warning when the frontend build SHA and Convex `APP_BUILD_SHA` differ even though the dashboard works.

## Best Possible Solution

Keep the drift detector, make production SHA stamping coherent for both deployables, and adjust banner severity/copy/docs/tests only after maintainers define expected partial-deploy drift behavior.

## Reproduction Assessment

Yes for source-level behavior: current main renders a `role="alert"` when `VITE_APP_BUILD_SHA` and Convex `appBuildSha` are both nonempty and unequal, and tests cover that path. I did not reproduce the original live mismatch end to end; current live Convex reports the current main SHA while the frontend runtime omits `VITE_APP_BUILD_SHA`.

## Solution Assessment

Unclear. The current implementation is deliberate and tested against the documented deploy contract, so the best fix depends on maintainer/operator policy for SHA stamping and expected partial-deploy drift rather than unconditional banner removal.
