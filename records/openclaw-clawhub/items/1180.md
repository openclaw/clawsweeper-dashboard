## Summary

Keep open: current main and v0.21.0 still render the dashboard deployment-drift alert whenever frontend and Convex build SHAs are both present and unequal, and the remaining work is a maintainer decision about deploy stamping versus banner policy.

## What This Changes

The issue reports a persistent dashboard warning when the frontend expected one build SHA but Convex reported another while the dashboard otherwise worked.

## Best Possible Solution

Keep actionable drift detection, but make coordinated production deploys stamp one selected main SHA across Convex and Vercel or deliberately revise the banner severity, copy, tests, and deploy runbook for intentional partial deploys.

## Reproduction Assessment

Yes at source level: passing non-empty unequal frontend and Convex SHAs deterministically sets hasMismatch and renders the alert. I did not recreate the historical production deployment state in a browser session.

## Solution Assessment

No automatic code change is proven best yet. Maintainers should first decide whether production deploy stamping violated the same-SHA contract or whether the banner policy should change for intentional partial deploys.
