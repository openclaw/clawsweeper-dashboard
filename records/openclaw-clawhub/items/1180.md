## Summary

Current main still implements the reported exact-SHA warning, and the production runbook explicitly supports backend-only and frontend-only deployments that can legitimately leave those SHAs different. That makes this a real, source-reproducible operator UX problem, but resolving it requires maintainers to define which partial-deploy states or rollout windows are healthy before changing the alert.

## What This Changes

Change the dashboard deployment-drift warning so supported frontend/backend deployment differences do not produce a persistent false alarm.

## Best Possible Solution

Represent deployment compatibility or the selected deploy target explicitly, then alert only for an unexpectedly stale or incompatible pairing while keeping supported partial deployments quiet.

## Reproduction Assessment

Yes at source level: provide different non-empty `VITE_APP_BUILD_SHA` and Convex `APP_BUILD_SHA` values and current main deterministically renders the reported alert. The documented backend-only and frontend-only deployment targets provide a plausible real production path to that state, though this review did not perform a live production deployment.

## Solution Assessment

No, raw commit equality is not the best health criterion while the repository supports independently advancing frontend and backend deployments. Maintainers should first define compatible partial-deploy semantics, then implement and visibly validate that policy.
