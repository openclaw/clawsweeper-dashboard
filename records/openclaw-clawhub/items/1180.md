## Summary

Keep open. Current main still intentionally renders the deployment-drift alert when frontend and Convex build SHAs are both present and unequal, and ClawHub cleanup policy keeps issues open for maintainer/operator handling rather than auto-closing them.

## What This Changes

The issue reports a persistent dashboard deployment-drift warning where the frontend expected SHA `da76bf75543f421dd5c75a988c4a88ec3e704fa1` but Convex reported `fefb2340a85bc8058a07ac1dc157eaf82bcf90e5`.

## Best Possible Solution

Define and enforce the production deploy contract: coordinated deploys should stamp the same SHA, and expected partial-deploy drift should have explicit banner severity, copy, docs, and tests.

## Reproduction Assessment

Yes for the source path: provide different non-empty values for `VITE_APP_BUILD_SHA` and `appMeta.getDeploymentInfo().appBuildSha`, and current helper/component tests assert the warning. The exact live production SHA pair still needs operator verification.

## Solution Assessment

Unclear. A direct code suppression is not proven to be the best fix; maintainers should first verify production stamping and decide whether partial deploy drift should alert, warn differently, or be treated as expected.
