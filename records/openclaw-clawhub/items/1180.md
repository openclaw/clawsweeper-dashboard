## Label Justifications

- `P2`: The dashboard remains usable, but a persistent red operational warning can mislead deployers and erode trust in health signals.
- `impact:other`: The issue affects deployment-health observability and operator confidence outside the more specific owned impact categories.

## Summary

Current main still deliberately shows the warning whenever stamped frontend and Convex SHAs differ, while production supports independent frontend and backend deploys. The report remains necessary because maintainers must define when that skew is actionable versus expected.

## What This Changes

Adjust the dashboard’s persistent deployment-version drift warning when frontend and Convex build identifiers differ even though the dashboard remains functional.

## Best Possible Solution

Define a durable deployment-compatibility contract, then warn only for actionable or unexpectedly persistent skew using metadata that distinguishes intentional split rollouts from incompatible deployments, with coverage for full, backend-only, and frontend-only releases.

## Reproduction Assessment

Yes, at source level: non-empty unequal `VITE_APP_BUILD_SHA` and backend `APP_BUILD_SHA` values deterministically render the current alert. This review did not independently establish a live production mismatch.

## Solution Assessment

Unclear until maintainers define the intended behavior for supported split deployments; unconditionally hiding the banner is not the best fix because it would also remove detection of genuine incompatible skew.
