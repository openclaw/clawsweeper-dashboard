## Summary

Keep open: current main still intentionally renders the dashboard deployment-drift alert whenever frontend and Convex build SHAs are both present and unequal. Live production currently reports the Convex SHA as current main, but the frontend runtime env asset does not expose `VITE_APP_BUILD_SHA`, so the original red alert is not proven live and the remaining question is still deploy-stamping contract or banner policy, not an implemented-on-main cleanup.

## What This Changes

The issue reports a persistent dashboard warning because the frontend build SHA and Convex-reported deployment SHA differ even though dashboard functionality works.

## Best Possible Solution

Define the production deploy SHA contract, then either ensure coordinated frontend/backend stamping for normal deploys or intentionally downgrade/change the banner behavior for expected partial deploys with matching tests and deploy docs.

## Reproduction Assessment

Yes at source level: current main deterministically renders the alert when the frontend and Convex SHAs are both present and unequal, and tests assert that behavior. The current public frontend asset does not expose `VITE_APP_BUILD_SHA`, so I did not prove the original red warning in a live browser session.

## Solution Assessment

No automatic code change is proven best yet. Maintainers should first decide whether this is a deploy-stamping violation, a missing frontend stamp, or an intentional partial-deploy state that needs different banner policy.
