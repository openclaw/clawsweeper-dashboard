## Summary

Keep open: current main and the latest release still render the dashboard deployment-drift alert whenever frontend and Convex build SHAs are both present and unequal, and the remaining question is the intended deploy-stamping or banner policy.

## What This Changes

The issue reports a persistent dashboard warning because the frontend expected build SHA differs from the Convex-reported deployment SHA while the dashboard otherwise works.

## Best Possible Solution

Keep actionable drift detection, but define the production deploy SHA contract; if partial deploys are normal, adjust the banner severity, copy, tests, and deploy runbook together.

## Reproduction Assessment

Yes at source level: passing non-empty unequal frontend and Convex SHAs deterministically sets hasMismatch and renders the alert. I did not recreate the historical production deployment state in a browser session.

## Solution Assessment

No automatic code change is proven best yet. Maintainers should first decide whether production deploy stamping violated the same-SHA contract or whether the banner policy should change for intentional partial deploys.
