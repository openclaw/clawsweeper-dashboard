## Summary

Keep open: current main still deliberately renders the dashboard drift alert whenever the frontend and Convex build SHAs are both present and different, so the central warning behavior has not been implemented away.

## What This Changes

The issue reports a persistent dashboard deployment drift warning when the frontend expected SHA and Convex reported SHA differ.

## Best Possible Solution

Define the production SHA contract, then either ensure normal production deploys stamp both deployables with the same SHA or adjust the banner severity and copy for expected partial deploys with matching tests and deploy docs.

## Reproduction Assessment

Yes at source level: current main deterministically renders the alert when frontend and Convex SHAs are both present and unequal, and the component test asserts that behavior. I did not establish a live browser reproduction of the historical production mismatch.

## Solution Assessment

No automatic code change is proven best yet. Maintainers should first decide whether this is a deploy-stamping violation or whether the banner policy should change for intentional partial deploys.
