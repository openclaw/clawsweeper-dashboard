## Summary

Keep open. Current main still intentionally renders the deployment-drift alert when frontend and Convex build SHAs are both present and unequal, and the ClawHub cleanup policy for this review keeps issues open for maintainer/operator handling rather than auto-closing them.

## What This Changes

The issue reports a persistent dashboard deployment-version drift warning where the frontend expected SHA differs from the Convex reported SHA.

## Best Possible Solution

Keep the deployment contract explicit: coordinated production deploys should stamp the same main SHA on frontend and Convex, and any change to partial-deploy warning policy should update the banner copy, tests, deploy workflow expectations, and deploy spec together.

## Reproduction Assessment

Yes for the source path: set `VITE_APP_BUILD_SHA` and `appMeta.getDeploymentInfo().appBuildSha` to different non-empty values and current main renders the alert. The exact original live production warning was not reproduced here because this was a read-only source review rather than an operator check of deployed SHA stamping.

## Solution Assessment

Unclear. The current implementation matches the drift detector, tests, and deploy contract, so the right fix depends on whether production SHA stamping is wrong or maintainers want partial deploys to change warning policy.
