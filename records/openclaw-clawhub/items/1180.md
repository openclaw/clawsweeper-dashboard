Keep open. Current main still intentionally renders the deployment drift alert when frontend and Convex build SHAs are both present and different; current production evidence no longer confirms the original live SHA mismatch, so this remains maintainer/operator deploy-stamping and banner-policy triage rather than a cleanup close or automated fix.

## What This Changes

The issue reports a persistent dashboard deploy mismatch warning when the frontend expected SHA differed from the Convex reported SHA.

## Best Possible Solution

Maintain a deploy contract where frontend and Convex stamp the same main SHA; if partial deploys are expected, adjust banner severity/copy, tests, and deploy docs together.

## Reproduction Assessment

Yes for the source path: mismatched non-empty VITE_APP_BUILD_SHA and appMeta.getDeploymentInfo().appBuildSha values produce the alert, and tests assert that behavior. The original live mismatch was not reproduced today because Convex reports current main and the fetched frontend runtime env bundle has no VITE_APP_BUILD_SHA.

## Solution Assessment

Unclear. Current source matches the drift detector, tests, and deploy contract, so the right fix depends on whether production SHA stamping should be corrected or the warning policy should change for expected partial deploys.
