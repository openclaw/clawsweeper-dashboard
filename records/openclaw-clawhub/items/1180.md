## Summary

Keep open. Current main still intentionally renders the dashboard drift alert when frontend and Convex build SHAs are both present and unequal, while live production now appears to have a backend SHA stamped but no frontend `VITE_APP_BUILD_SHA` in the inspected runtime asset; the remaining decision is operator/product policy, not a cleanup close.

## What This Changes

The issue reports a persistent dashboard deployment-drift warning where the frontend and Convex build SHAs differ even though the dashboard works.

## Best Possible Solution

Keep drift detection, ensure coordinated production deploys stamp both deployables with the same SHA, and if partial-deploy drift is expected, adjust the banner severity/copy, tests, and deploy spec together.

## Reproduction Assessment

Yes for the source-level behavior: provide different non-empty `VITE_APP_BUILD_SHA` and `appMeta.getDeploymentInfo().appBuildSha` values and current main renders the warning. The exact original live production warning is not reproduced because the inspected live frontend runtime asset currently omits `VITE_APP_BUILD_SHA`.

## Solution Assessment

Unclear. Current source behavior matches the deployment contract, so the best fix depends on whether maintainers want to repair production SHA stamping or intentionally change warning policy for partial deploys.
