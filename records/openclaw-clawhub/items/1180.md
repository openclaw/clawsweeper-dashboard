## Summary

Keep open: current main and the latest release still intentionally render the dashboard drift alert when frontend and Convex build SHAs are both present and unequal, so this still needs a maintainer decision about deploy stamping versus banner policy.

## What This Changes

The issue reports a persistent dashboard deployment-drift warning when the frontend build SHA and Convex build SHA differ even though the dashboard otherwise works.

## Best Possible Solution

Keep actionable drift detection, but define the production deploy SHA contract; if partial deploys are normal, adjust the banner severity or copy and update the drift tests and deploy runbook together.

## Reproduction Assessment

Yes at source level: current main deterministically sets hasMismatch and renders the alert when non-empty frontend and Convex SHAs differ. I did not recreate the historical production deployment state in a browser session.

## Solution Assessment

No automatic code change is proven best yet. Maintainers should first decide whether this is a deploy-stamping failure or whether the banner policy should change for intentional partial deploys.
