## Summary

Current main still intentionally warns when the frontend and Convex report different non-empty build SHAs, so the reported behavior has not been implemented away; maintainers must determine whether production stamping violated the documented same-SHA contract or that contract should change.

## What This Changes

Resolve a persistent dashboard deployment-drift warning when the frontend and Convex report different build commit SHAs despite the dashboard otherwise working.

## Best Possible Solution

Make full production releases stamp and verify one selected main SHA across Convex and Vercel, while retaining the banner for genuine partial deploys and adding actionable remediation or deployment-age context if drift can temporarily be expected.

## Reproduction Assessment

Yes at source level: passing the two reported non-empty unequal SHAs deterministically enters the tested mismatch path and renders the alert. This read-only review did not recreate the exact historical production deployment state.

## Solution Assessment

No single automatic code change is yet proven best: removing the alert would weaken a documented rollout safeguard, so maintainers should first determine whether production stamping violated the same-SHA contract or whether that contract must be revised.
