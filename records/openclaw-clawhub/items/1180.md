## Summary

Current main still deliberately warns whenever the stamped frontend and Convex SHAs differ, while the deploy specification requires them to match; no merged fix or viable canonical item now owns whether partial deploys should restamp both sides or use different warning semantics.

## What This Changes

Resolve a persistent dashboard deployment-drift warning caused by different frontend and Convex build SHAs despite otherwise functional dashboard behavior.

## Best Possible Solution

Keep same-SHA drift detection for coordinated releases, but make partial-deploy behavior explicit by either restamping both deployment metadata surfaces to the selected main SHA or defining a lower-severity, actionable state with documented duration and recovery rules.

## Reproduction Assessment

Yes at source level: passing the reported non-empty unequal frontend and Convex SHAs deterministically enters the tested mismatch path and renders the alert. This review did not recreate the historical production deployment state in a live browser.

## Solution Assessment

No single automatic patch is proven best yet; maintainers must first decide whether production partial deploys violate the same-SHA contract or require intentionally different banner severity and recovery semantics.
