## Summary

Current main and v0.20.0 intentionally show the warning for any unequal stamped SHAs, while the dashboard otherwise remains usable. The issue is still necessary because no current code, release, or canonical follow-up defines when deployment skew is expected versus actionable.

## What This Changes

Adjust the dashboard’s persistent deployment-version drift warning so normal frontend/Convex version skew is not reported as an ongoing red alert.

## Best Possible Solution

Define and document the supported deployment-skew contract, then downgrade or delay expected transient skew while retaining a clear actionable alert for persistent or incompatible drift.

## Reproduction Assessment

Yes at source level: supplying the two reported nonempty unequal SHAs to `getDeploymentDriftInfo` deterministically yields `hasMismatch: true`, and the dashboard then renders the alert. A live production reproduction is not required to establish this current-main behavior.

## Solution Assessment

No single code change is yet clearly best because maintainers must first decide whether unequal SHAs are forbidden, transiently acceptable, or acceptable across independently deployed targets. The safest solution preserves detection but changes timing or severity according to that explicit policy.
