## Summary

Keep this open: current main partially addresses the report by sending best-effort authenticated telemetry immediately after `clawhub install`, but metric definitions remain incomplete and `clawhub explore` still hides the selected metric in text output. The collaborator-authored issue also requires an explicit product decision and cross-repository coordination. Likely related people: Patrick-Erichsen (high confidence), based on prior merged work implementing the affected ClawHub CLI paths.

## What This Changes

Clarify and align download, current-install, all-time-install, and trending semantics across ClawHub, OpenClaw, the API, documentation, and human-readable CLI output.

## Best Possible Solution

Define the four metrics in one durable contract, retain a single snapshot-based source of install truth unless maintainers intentionally approve a migration, display the chosen metric in `clawhub explore`, and align both CLIs plus public API/UI documentation with the selected semantics.

## Reproduction Assessment

Yes, at source level. Current main directly shows conditional immediate install telemetry, telemetry-derived install counters, and human explore output that omits the selected ranking metric.

## Solution Assessment

The issue identifies the correct surfaces, but the final implementation is not predetermined. Maintainers should first choose the canonical adoption signal; preserving one inventory-based source of truth and clarifying it is safer than adding a competing counter path.
