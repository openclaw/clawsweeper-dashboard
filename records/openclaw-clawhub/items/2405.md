## Summary

Current main still presents two distinct adoption signals without a single cross-product explanation: archive fetches use the download endpoint, install counts are documented as authenticated sync snapshots, and `clawhub explore` hides the selected metric in human output. The issue also requires explicit maintainer judgment because it proposes changing the install source of truth, spans the separate OpenClaw CLI, is collaborator-authored, and overlaps two open telemetry/ranking PRs without being fully resolved by either.

## What This Changes

Define and consistently present download, current-install, all-time-install, and trending semantics across ClawHub, its CLI, the OpenClaw CLI, and public API/UI surfaces.

## Best Possible Solution

Adopt one durable metric contract in a focused spec—preferably preserving distinct archive-fetch and adoption concepts—then coordinate the active telemetry work, display the selected sort metric in `clawhub explore`, update ClawHub docs/API/UI wording, and track the matching OpenClaw CLI changes explicitly.

## Reproduction Assessment

Yes. Current main directly shows the mismatch: installs are defined as authenticated sync snapshots, archive installation uses the download route, and human-readable sorted explore output omits the ranking metric.

## Solution Assessment

Not yet. Clarifying the existing terms and exposing the sort value are appropriate, but choosing immediate install events versus snapshot-derived adoption is a product/data-contract decision that should be settled before implementation.
