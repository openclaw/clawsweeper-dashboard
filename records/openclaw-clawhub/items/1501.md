## Summary

Keep open. Current main still source-reproduces the reported `toUserHandle required` failure through CLI double JSON serialization, and the account-specific ownership transfer may also depend on the broader org/publisher transfer work tracked separately.

## What This Changes

The issue requests transferring `baidu-netdisk-storage` to `@BaiduNetdiskAIBot` after `clawhub transfer request` returned `toUserHandle required`.

## Best Possible Solution

Land or extract the plain-object CLI payload fix tracked by https://github.com/openclaw/clawhub/issues/1300, then verify the recipient account type and use the existing user-transfer path, publish owner migration, or the publisher-targeted design tracked by https://github.com/openclaw/clawhub/issues/1675 as appropriate.

## Reproduction Assessment

Yes, source-reproducible. Current main sends a pre-stringified transfer request body into an HTTP helper that serializes POST bodies again, so the server receives a JSON string and cannot read `payload.toUserHandle`; I did not run a live authenticated transfer.

## Solution Assessment

Yes for the reported CLI error: pass plain object bodies from the CLI and update tests to assert that boundary. If the recipient is an org or non-user publisher, the safer solution is the publisher-targeted transfer model rather than broadening this user-only path ad hoc.
