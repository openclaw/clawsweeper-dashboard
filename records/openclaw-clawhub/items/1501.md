## What This Changes

A user asks to transfer `baidu-netdisk-storage` to `@BaiduNetdiskAIBot` after `clawhub transfer request` failed with `toUserHandle required`.

## Best Possible Solution

Land or extract the plain-object CLI payload fix tracked by https://github.com/openclaw/clawhub/issues/1300 and https://github.com/openclaw/clawhub/pull/1635, then verify the recipient account type and use either user-to-user transfer, publish owner migration, or the publisher-targeted transfer design tracked at https://github.com/openclaw/clawhub/issues/1675.

## Reproduction Assessment

Yes, source-reproducible. Current main still sends a pre-stringified transfer request body into an HTTP helper that serializes POST bodies again, so the server receives a JSON string and cannot read `payload.toUserHandle`.

## Solution Assessment

Yes for the reported CLI error: pass plain object bodies from the CLI and update tests to assert that boundary. If the recipient is an org/non-user publisher, the safer solution is the publisher-targeted transfer model rather than broadening this user-only path ad hoc.
