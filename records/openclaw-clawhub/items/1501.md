## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still source-reproduces the reported `toUserHandle required` transfer failure through double JSON serialization while possible org/publisher transfer semantics remain tracked separately.

## What This Changes

The issue requests transferring `baidu-netdisk-storage` to `@BaiduNetdiskAIBot` and reports the CLI failing with `toUserHandle required`.

## Best Possible Solution

Land or extract the plain-object CLI payload fix tracked by https://github.com/openclaw/clawhub/issues/1300, then handle any org/non-user publisher target through the publisher-transfer design tracked by https://github.com/openclaw/clawhub/issues/1675.

## Reproduction Assessment

Yes, source-reproducible. Current main sends a pre-stringified transfer request body into an HTTP helper that serializes POST bodies again, so the server receives a JSON string and cannot read `payload.toUserHandle`.

## Solution Assessment

Yes for the reported CLI error: send plain object bodies from the affected command handlers and update tests to assert that boundary. If the recipient is an org/non-user publisher, that should be solved through the publisher-targeted transfer design rather than broadening this user-only path ad hoc.
