## Summary

Keep open. Current main still source-reproduces the reported transfer failure: the CLI pre-serializes the transfer body, both HTTP transports serialize it again, and the server then cannot read `payload.toUserHandle`; any org/non-user transfer semantics remain separately tracked.

## What This Changes

The issue requests transferring `baidu-netdisk-storage` to `@BaiduNetdiskAIBot` with `clawhub transfer request`, which currently returns `toUserHandle required`.

## Best Possible Solution

Land the plain-object CLI payload fix tracked by https://github.com/openclaw/clawhub/issues/1300, and keep direct org/non-user publisher transfer design in https://github.com/openclaw/clawhub/issues/1675.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: the transfer command passes a JSON string, both HTTP transports stringify again, and the server requires `payload.toUserHandle` on an object. I did not run a live authenticated transfer.

## Solution Assessment

Yes for the reported CLI error: pass plain objects from affected command handlers and update tests to assert that boundary. If the target is an org/non-user publisher, that should be solved through the publisher-targeted transfer model rather than widening this user-only path ad hoc.
