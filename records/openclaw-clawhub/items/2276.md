## What This Changes

The branch moves requester active/deleted/deactivated validation before the ownership branch in `acceptTransferInternal` and updates an existing test mock for the unconditional requester fetch.

## Best Possible Solution

Land the accept-transfer guard with focused regression coverage and redacted after-fix runtime proof showing a banned or deactivated requester cannot transfer a still-owned skill.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a pending transfer whose skill is still owned by `transfer.fromUserId`, mark that requester `deletedAt` or `deactivatedAt`, then accept as the recipient; current main skips requester validation in that state.

## Solution Assessment

Yes. Moving the requester activity check to the shared accept gate is the narrowest maintainable fix for this race, with regression coverage and real behavior proof still needed before merge.
