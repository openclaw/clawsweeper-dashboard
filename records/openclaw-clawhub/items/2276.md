## Summary

Keep this PR open: current main has the reported transfer-acceptance gap, and the patch is a narrow guard in the right mutation, but contributor real-behavior proof is still missing before merge.

## What This Changes

This PR moves the requester active-user check in `acceptTransferInternal` so pending skill transfers are cancelled when the original requester has been deleted, banned, deactivated, or removed.

## Best Possible Solution

Land a narrow accept-transfer guard with regression coverage proving a pending transfer is cancelled when the requester is banned or deactivated while the skill is still owned by that requester.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a pending transfer whose skill is still owned by `transfer.fromUserId`, mark that requester `deletedAt` or `deactivatedAt`, then accept as the recipient on current main. The current branch at `convex/skillTransfers.ts:189` skips requester validation in that state.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix for this specific race because acceptance is the final gate before ownership changes. A regression test for the banned-requester case should accompany it before merge.
