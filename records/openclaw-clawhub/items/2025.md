## Summary

Keep open: current main intentionally treats user-initiated deletion as terminal, and resolving this GitHub OAuth lockout requires maintainer/operator identity verification plus a product and security policy decision rather than cleanup closure.

## What This Changes

The issue asks maintainers to restore a deleted ClawHub account or remove its GitHub OAuth binding so the same GitHub account can sign up again.

## Best Possible Solution

Define an audited account recovery or GitHub identity-unlink policy, then either add guarded support/admin tooling with tests or explicitly keep deletion terminal with clearer support guidance.

## Reproduction Assessment

Yes, by source inspection: settings calls users.deleteAccount, deleteAccount sets deactivatedAt/purgedAt, and the OAuth callback rejects deactivated users. I did not run a live OAuth flow or inspect the reporter's production record.

## Solution Assessment

No for a one-off direct restore or binding deletion. The maintainable path needs a policy-backed, audited recovery or unlink flow, or an explicit decision to keep deletion irreversible with support guidance.
