## Summary

Keep open: current main intentionally blocks sign-in after user-initiated deletion, and resolving this reporter's GitHub OAuth lockout requires maintainer/operator identity verification plus a security and product policy decision.

## What This Changes

The reporter cannot sign in with GitHub after deleting their account and asks either to restore the account or remove the GitHub OAuth binding so they can register again.

## Best Possible Solution

Define an audited account recovery or GitHub identity-unlink policy, then either add a guarded support/admin flow with tests or explicitly keep deletion terminal with clearer support guidance.

## Reproduction Assessment

Yes, by source inspection: settings calls users.deleteAccount, that mutation sets deactivatedAt/purgedAt, and the OAuth callback rejects deactivated users. I did not run a live OAuth flow or inspect the reporter's production account record.

## Solution Assessment

No for a direct autonomous restore or binding deletion. The maintainable path needs a policy-backed, audited recovery or unlink flow, or an explicit decision to keep deletion irreversible with support guidance.
