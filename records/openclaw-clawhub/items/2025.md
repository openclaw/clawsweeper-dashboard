## Summary

Keep open: current main intentionally blocks sign-in after user-initiated deletion, so this is production account support plus an auth/account-recovery policy decision rather than conservative cleanup.

## What This Changes

The reporter asks ClawHub to restore a deleted account or remove its GitHub OAuth binding so the same GitHub identity can sign up again.

## Best Possible Solution

A maintainer should verify the specific production account and decide whether to perform an audited support restore/unlink, then update code, docs, and tests only if ClawHub changes the irreversible deletion policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Settings deletion calls `users.deleteAccount`, the user row gets `deactivatedAt`/`purgedAt`, and the auth callback rejects later sign-in for that row; I did not run a live GitHub OAuth flow.

## Solution Assessment

Unclear as a code change. Restore/unlink conflicts with the current irreversible-deletion policy, so the narrow maintainable path is manual support/admin review first, followed by an explicit audited recovery policy only if maintainers choose it.
