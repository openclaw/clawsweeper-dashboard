## Summary

Keep open: current main intentionally blocks sign-in after user-initiated deletion, and this issue needs maintainer account-support/product-policy handling rather than conservative cleanup.

## What This Changes

The issue asks for restoration of a deleted account or removal of the GitHub OAuth binding so the same GitHub identity can sign up again.

## Best Possible Solution

A maintainer should handle the specific production account through support/admin tooling, then change code and docs only if ClawHub decides deleted GitHub identities should be unlinkable, restorable, or reusable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Settings deletion calls users.deleteAccount, the user row gets deactivatedAt, and the OAuth callback rejects later sign-in for that user; I did not run a live GitHub OAuth flow.

## Solution Assessment

Unclear as a code change. The requested restore/unlink conflicts with the documented irreversible-deletion policy, so the narrowest maintainable next step is maintainer support review before implementation.
