## Summary

Keep open: current main intentionally blocks GitHub OAuth reauth after user-initiated deletion, and ClawHub issue cleanup policy reserves auto-close for PRs already implemented on main. The remaining work is account support plus a product/security decision about restore, unlink, or re-registration semantics.

## What This Changes

The issue asks maintainers to restore a deleted account or remove its GitHub OAuth binding so the reporter can sign up again.

## Best Possible Solution

Maintainers should define the post-deletion lifecycle, then either provide an audited restore/unlink/re-registration path or keep deletion irreversible with clearer support/admin handling and OAuth messaging.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delete an account through settings so the user gets deactivatedAt/purgedAt, then complete GitHub OAuth for the same bound identity; the auth callback rejects that user and tests assert the rejection.

## Solution Assessment

Unclear as an immediate code change. The requested restore or binding removal conflicts with current docs and tests, so the narrowest maintainable next step is maintainer account-support review plus a product/security policy decision.
