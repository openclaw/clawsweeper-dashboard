## Summary

Keep open: current main intentionally blocks GitHub sign-in after permanent account deletion, and resolving this needs maintainer/operator identity-policy handling rather than conservative cleanup closure.

## What This Changes

A deleted GitHub-linked user requests account restoration or GitHub OAuth binding removal so they can sign up again.

## Best Possible Solution

Define an audited maintainer-approved recovery/unlink or support-only policy for self-deleted GitHub-linked accounts, preserving deletion, ownership, token, publisher, content, ban, and audit invariants before changing code or production data.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: settings calls users.deleteAccount, which sets deactivatedAt/purgedAt, and the next GitHub OAuth completion calls handleDeletedUserSignIn and rejects that user; I did not run a live OAuth flow.

## Solution Assessment

Unclear as a direct restore/unlink request. The narrow maintainable path is first deciding an audited recovery or unlink policy; directly removing the binding or restoring the row would bypass current irreversible-deletion semantics.
