## Summary

Keep open: current main intentionally treats user-initiated deletion as irreversible, while this request asks for account restoration or GitHub identity unlinking that needs maintainer/operator review and a security policy decision.

## What This Changes

The issue asks for a deleted GitHub-linked ClawHub account to be restored or for its GitHub OAuth binding to be removed so the user can sign up again.

## Best Possible Solution

Decide and document the account-recovery policy, then implement an audited restore or GitHub unlink operator flow only if recovery is allowed and ownership can be verified while preserving bans and purged-data boundaries.

## Reproduction Assessment

Yes. By source inspection, deleting an account sets `deactivatedAt`/`purgedAt`, and the GitHub OAuth callback rejects `deactivatedAt` users; I did not inspect the reporter's production record.

## Solution Assessment

No for an ad hoc restore or binding deletion. The maintainable path is a policy-backed, audited recovery or unlink flow with identity verification, or an explicit decision to keep deletion irreversible with clearer support guidance.

## Security Review

Status: needs_attention

Summary: Account restoration or GitHub OAuth unlinking is identity-sensitive and needs explicit recovery policy and operator safeguards.

Concerns:

- **[medium] Verify ownership before recovery:** `convex/auth.ts:42`
  - body: The OAuth path blocks deactivated users; any restore or unlink action must verify the requester still controls the relevant GitHub provider identity before re-enabling access or freeing a binding.
  - confidence: 0.92
- **[medium] Protect provider identity rebinding:** `convex/lib/githubIdentity.ts:17`
  - body: GitHub `providerAccountId` is the identity anchor for account-owned behavior, so unlinking or recreating auth bindings needs explicit safeguards against rebinding the wrong GitHub identity.
  - confidence: 0.88
- **[medium] Preserve purge and ban boundaries:** `convex/users.ts:348`
  - body: Permanent deletion clears user data while admin unban explicitly rejects `deactivatedAt` accounts, so a recovery flow must not bypass bans, malware blocks, or privacy semantics for purged accounts.
  - confidence: 0.9
