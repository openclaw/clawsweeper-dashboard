## Summary

Keep open: current main intentionally blocks sign-in after user-initiated account deletion, and ClawHub issue cleanup policy does not auto-close issues. This is a support/product-policy decision rather than a conservative source cleanup close.

## What This Changes

The issue asks maintainers to restore a deleted account or remove its GitHub OAuth binding so the reporter can sign up again.

## Best Possible Solution

Define one explicit post-deletion account lifecycle, then implement any chosen restore, unlink/re-register, or terminal-support behavior consistently across OAuth, admin tooling, user-facing errors, docs, and tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Settings deletion calls users.deleteAccount, which sets deactivatedAt and purgedAt, and handleDeletedUserSignIn rejects that user on later GitHub OAuth sign-in; I did not run a live OAuth flow.

## Solution Assessment

Unclear as a code change. The requested restore or unlink conflicts with the current documented irreversible-deletion policy, so the narrowest maintainable next step is support and product/security review before implementation.
