## Summary

Keep open: current main intentionally blocks GitHub OAuth sign-in after user-initiated account deletion, and this issue needs maintainer support/security policy rather than conservative cleanup closure.

## What This Changes

The reporter asks maintainers to restore a deleted GitHub-linked ClawHub account or remove its GitHub OAuth binding so they can sign up again.

## Best Possible Solution

Define a maintainer-approved account lifecycle policy, then either add an audited restore/unlink/re-registration flow that preserves abuse safeguards or document deletion as terminal with clear support guidance.

## Reproduction Assessment

Yes, by source inspection: settings calls deleteAccount, deleteAccount sets deactivatedAt/purgedAt, and the OAuth callback rejects that user on the next sign-in. I did not run a live OAuth flow or inspect the reporter's production record.

## Solution Assessment

No for a direct autonomous restore or binding deletion. The maintainable solution needs policy-backed identity verification, auditability, and separation between self-deleted accounts and enforcement-disabled accounts.
