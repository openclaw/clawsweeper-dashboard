## Summary

Keep open: current main intentionally blocks same-GitHub sign-in after user-initiated account deletion, and the remaining work is maintainer account-recovery policy plus production identity handling rather than cleanup closure.

## What This Changes

The reporter asks ClawHub maintainers to restore a deleted GitHub-linked account or remove its GitHub binding so they can sign up again.

## Best Possible Solution

Define and document an approved account recovery or re-registration policy, then implement an audited admin/support restore or GitHub-unlink flow only if that policy permits it.

## Reproduction Assessment

Yes, by source inspection: settings calls deleteAccount, that mutation sets deactivatedAt/purgedAt, and the next GitHub OAuth completion rejects the matched user. I did not run a live OAuth flow or inspect the reporter's production record.

## Solution Assessment

No for a direct autonomous restore or binding deletion. The maintainable path needs policy-backed, audited recovery or unlink semantics, or a documented decision that deletion remains irreversible with clearer support handling.
