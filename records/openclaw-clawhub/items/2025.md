## Summary

Keep open: this is a current account-recovery/support and policy issue, and ClawHub cleanup policy keeps issues open. Current main intentionally blocks sign-in after user-initiated deletion, so restoration or GitHub unlinking needs maintainer/operator identity verification and a security/product decision.

## What This Changes

The reporter cannot sign back in with GitHub after account deletion and asks for account restoration or removal of the GitHub OAuth binding.

## Best Possible Solution

Define an audited recovery or provider-unlink policy for deleted accounts, then either implement a guarded operator/admin flow with regression tests or document deletion as terminal with a clear support path.

## Reproduction Assessment

Yes, by source inspection: settings invokes users.deleteAccount, that mutation sets deactivatedAt/purgedAt, and the OAuth callback rejects deactivated users. I did not run a live OAuth callback or inspect the reporter's production record.

## Solution Assessment

No for an automated restore or binding deletion. The narrow maintainable path is a policy-backed, audited recovery or unlink flow, or an explicit decision to keep deletion irreversible with clearer support guidance.
