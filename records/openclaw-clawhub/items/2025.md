## Summary

Keep open: current main intentionally blocks sign-in after user-initiated deletion, so this needs maintainer account-support and product-policy handling rather than conservative cleanup.

## What This Changes

A deleted user asks ClawHub to restore their account or remove the GitHub OAuth binding so they can register again.

## Best Possible Solution

A maintainer should handle the production account through support/admin tooling, then update code and docs only if ClawHub decides deleted GitHub identities should be restorable, unlinkable, or reusable.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: Settings deletion calls users.deleteAccount, which sets deactivatedAt, and the auth callback rejects later sign-in for that user; I did not run a live GitHub OAuth flow.

## Solution Assessment

Unclear as a code change. The requested restore/unlink conflicts with the current documented irreversible-deletion policy, so the narrowest maintainable next step is maintainer support review before implementation.
