## Label Justifications

- `P2`: The missing explanation makes false-positive diagnosis difficult for a recently shipped classification feature but does not block publishing or core availability.
- `impact:security`: The issue concerns transparency and auditability for a classifier that determines whether a skill requires users to supply credentials.

## Summary

Current main still computes the classification rationale only as transient evaluator output and logs, while the stored skill-version model and user-facing badge expose only the boolean verdict. The assigned feature contributor also described the explanation as useful future work, so the request remains necessary and needs a product/security decision before implementation.

## What This Changes

Expose a user-auditable rationale explaining why a skill is classified as requiring an API key.

## Best Possible Solution

Define a versioned explanation model with stable reason codes and bounded sanitized evidence, persist it with the verdict, document public versus staff-only disclosure in a focused security-sensitive spec, and render the safe explanation on relevant skill and audit surfaces.

## Reproduction Assessment

Yes, at source level. Current main computes and logs a rationale but persists and renders only the boolean classification, directly reproducing the missing auditability described in the report.

## Solution Assessment

Yes in intent, but publishing raw model reasoning is not the safest implementation. Structured reason codes with sanitized, bounded evidence and an explicit disclosure policy are the narrowest maintainable solution.
