## Summary

Current main still discards the evaluator’s richer explanation before persistence and renders only a generic API-key-required badge. The request remains useful and aligned with the issue discussion, but the public explanation format and safe disclosure boundary need maintainer product judgment.

## What This Changes

Expose an auditable reason explaining why a skill version was classified as requiring an API key or equivalent secret.

## Best Possible Solution

Persist a concise structured reason and sanitized evidence summary alongside the verdict, render it on the skill detail surface, and document in a focused spec what may be public versus operational-only.

## Reproduction Assessment

Yes. Current main provides a high-confidence source reproduction: the evaluator generates rationale, deliberately reduces it to a boolean for storage, and the public badge has no rationale input.

## Solution Assessment

Yes in direction, but the safest maintainable implementation is a structured, sanitized explanation rather than publishing raw evaluator output.
