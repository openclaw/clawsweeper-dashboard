## Summary

Current main still discards the evaluator’s rationale and exposes only a boolean credential verdict, so the requested auditability is not implemented. The assigned contributor also treated the explanation as useful future work; this remains a valid product and security-design request needing an explicit disclosure policy.

## What This Changes

Expose an auditable explanation for why a skill is classified as requiring an API key or equivalent secret.

## Best Possible Solution

Persist a versioned explanation with normalized evidence such as literal environment-variable names and a short sanitized summary, update it atomically with the verdict, and render the safe portion in the existing security audit/details experience.

## Reproduction Assessment

Yes, at source level. Current main explicitly collapses the richer evaluator response into an optional boolean, and the public badge receives only that boolean, so no rationale can be audited through the current model or UI.

## Solution Assessment

Yes in direction, but not as raw model reasoning. A bounded structured evidence record plus a sanitized user-facing summary is the narrower and safer maintainable solution.
