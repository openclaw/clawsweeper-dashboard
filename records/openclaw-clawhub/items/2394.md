## Summary

Current main still exposes only a boolean badge even though the evaluator distinguishes multiple decision paths, so the requested auditability is not implemented and should remain open for maintainer product/security review.

## What This Changes

Expose an auditable explanation for why a skill version is classified as requiring an API key.

## Best Possible Solution

Persist a stable reason code, evaluator version, and concise sanitized explanation alongside the boolean, show the safe explanation to the skill publisher, and keep any deeper scanner audit detail restricted to authorized staff.

## Reproduction Assessment

Yes, at source level: the evaluator computes several distinct decision paths, but the mutation, schema, tests, and badge retain only the boolean result. The reporter did not provide a slug, so the specific false-positive classification is not reproduced, but the auditability gap is definitive.

## Solution Assessment

Yes in direction, but raw model rationale is not the safest implementation. A structured reason code plus a sanitized, audience-appropriate explanation is the narrowest maintainable solution.
