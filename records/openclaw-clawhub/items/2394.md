## Summary

Keep open: current main and the latest release still compute API-key classifier rationales only as transient action output/log text, while the stored model and user-facing badge expose only a boolean verdict. The discussion also identifies this as useful future work, so it needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue requests an auditable rationale explaining why a skill was classified as requiring an API key.

## Best Possible Solution

Define a versioned explanation model with stable reason codes and bounded sanitized evidence, then render the safe explanation on skill/audit surfaces with clear public versus staff-only disclosure rules.

## Reproduction Assessment

Yes, at source level. Current main computes rationale and env-var output, but stores, serializes, and renders only the `apiKeyRequired` boolean, so the missing audit trail is directly observable from source.

## Solution Assessment

Yes in intent, but raw model reasoning is not the safest implementation. Structured reason codes with sanitized bounded evidence and an explicit disclosure policy are the maintainable path.
