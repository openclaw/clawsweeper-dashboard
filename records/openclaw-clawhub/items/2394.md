## Summary

Keep open: current main and v0.21.0 still expose the API-key classifier as a boolean badge without persisted, user-auditable rationale, and maintainer discussion treats the explanation as future work rather than shipped behavior.

## What This Changes

The issue requests an auditable explanation for why a skill was marked “API key required.”

## Best Possible Solution

Define a security-reviewed explanation model with stable reason codes and bounded sanitized evidence, then persist and expose that explanation where publishers can diagnose the badge.

## Reproduction Assessment

Yes, at source level. Current main computes or receives rationale during evaluation but persists and renders only the boolean classification, matching the missing auditability described in the report.

## Solution Assessment

Yes in intent, but not as raw model text. The safer maintainable solution is structured reason codes plus sanitized bounded evidence with an explicit disclosure policy.
