## Summary

Keep open: current main still folds the API-key evaluator's richer result into a stored boolean, and the live discussion treats auditable rationale as useful future work rather than implemented behavior.

## What This Changes

The issue asks ClawHub to make the rationale behind an `API key required` scanner verdict auditable instead of exposing only a boolean badge.

## Best Possible Solution

Define a security-reviewed explanation model with stable reason codes and bounded sanitized evidence, then persist and surface the safe explanation where authors can diagnose the badge.

## Reproduction Assessment

Yes, at source level. Current main computes or receives rationale during evaluation but persists and renders only the boolean classification, matching the missing auditability described in the report.

## Solution Assessment

Yes in intent, but not as raw model text. The safer maintainable solution is structured reason codes plus sanitized bounded evidence with an explicit disclosure policy.
