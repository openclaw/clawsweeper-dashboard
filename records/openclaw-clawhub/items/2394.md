## Label Justifications

- `P2`: The missing explanation makes false-positive diagnosis difficult for a credential-related classifier, but it does not block publishing or core availability.
- `impact:security`: The issue concerns transparency and auditability for a classifier that tells users a skill appears to require credentials or equivalent secrets.

## Summary

Keep open: current main and the latest release still expose only the boolean `apiKeyRequired` verdict. The evaluator can produce transient rationale and env-var output, but that data is not persisted, denormalized, or rendered, and the discussion treats this as future work needing maintainer/security product decisions.

## What This Changes

Requester wants the API-key-required classifier to expose an auditable reason or rationale when a skill is flagged.

## Best Possible Solution

Define a security-reviewed, versioned explanation model with stable reason codes and bounded sanitized evidence, persist it with the verdict, and render the safe explanation on skill/audit surfaces.

## Reproduction Assessment

Yes, at source level. Current main computes or returns rationale during evaluation but persists and renders only the boolean classification, directly matching the missing auditability described in the report.

## Solution Assessment

Yes in intent, but not as raw model text. The safer maintainable solution is structured reason codes plus sanitized bounded evidence with an explicit public-versus-staff disclosure policy.
