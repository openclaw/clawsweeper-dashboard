## Label Justifications

- `P2`: The shipped classification works, but missing diagnostic context impairs false-positive investigation and appeals without breaking the core publishing workflow.

## Summary

Current main still reduces the analyzer’s richer result to a boolean and shows only a generic badge, so publishers cannot audit the classification. The request remains useful, but the public explanation format needs a maintainer product and security decision before implementation.

## What This Changes

Expose an auditable rationale for why a skill is classified as requiring an API key.

## Best Possible Solution

Persist a sanitized structured explanation—such as reason codes, relevant declared or referenced environment variables, classifier version, and scan timestamp—and expose it through the skill security audit or badge details without publishing raw chain-of-thought.

## Reproduction Assessment

Yes at the source level: the evaluator produces rationale and environment-variable evidence, then current main persists and renders only a boolean. The reporter’s particular false positive cannot be replayed because the requested skill slug was not supplied.

## Solution Assessment

Yes in goal, but raw model reasoning is not the safest contract; a structured, sanitized explanation with classifier provenance is the narrower maintainable solution.
