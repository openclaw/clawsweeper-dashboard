## Label Justifications

- `P2`: The classifier functions, but absent rationale makes credential-requirement false positives difficult to diagnose and audit.
- `impact:security`: The feature classifies credential requirements, and deciding what classification evidence may be disclosed is security-sensitive.

## Summary

Current main still collapses richer classification evidence into a boolean and shows only generic badge text. The feature author also agreed in discussion that rationale would be useful, so this remains valid product work requiring a security-disclosure decision.

## What This Changes

Requests an auditable rationale explaining why a skill is classified as requiring an API key.

## Best Possible Solution

Persist stable reason codes and sanitized evidence alongside the boolean, expose an appropriate explanation to skill owners and selected public surfaces, and record the disclosure policy in specs/security-moderation.md.

## Reproduction Assessment

Yes at the source level: current main explicitly folds richer evaluator output into apiKeyRequired, and the UI receives only that boolean. The reporter's particular false positive is not reproducible because the requested skill slug was not supplied.

## Solution Assessment

Yes in principle, but exposing raw agent prose is not the best permanent contract; stable reason codes with sanitized supporting evidence would be safer, testable, and consistently auditable.
