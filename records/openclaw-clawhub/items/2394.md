## Label Justifications

- `P2`: The missing explanation makes credential-classifier false positives hard to diagnose, but it does not block core availability or publishing outright.
- `impact:security`: The issue concerns transparency for a classifier that tells users a skill appears to require credentials or equivalent secrets.

## Summary

Keep open: current main and v0.21.0 still persist and render only the boolean `apiKeyRequired` verdict. The evaluator can produce rationale/env-var output, but that data is transient, and the issue discussion treats auditable rationale as useful future work needing maintainer/security product decisions.

## What This Changes

Request to make the `API key required` skill badge auditable by attaching a reason for why the analyzer marked a skill as requiring an API key or equivalent secret.

## Best Possible Solution

Define a security-reviewed explanation model with stable reason codes and bounded sanitized evidence, then persist and render the safe explanation where authors can diagnose the badge.

## Reproduction Assessment

Yes, at source level. Current main computes or receives rationale during evaluation but persists and renders only the boolean classification, matching the missing auditability described in the report.

## Solution Assessment

Yes in intent, but not as raw model text. The safer maintainable solution is structured reason codes plus sanitized bounded evidence with an explicit disclosure policy.
