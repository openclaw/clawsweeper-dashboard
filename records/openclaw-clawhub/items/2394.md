## Label Justifications

- `P2`: The missing explanation makes false-positive diagnosis difficult for a security-relevant classification but does not block publishing or service availability.
- `impact:security`: The issue concerns transparency and auditability for a classifier that represents whether a skill needs user credentials.

## Summary

Current main still exposes only the API-key-required verdict to users: the evaluator can produce a rationale, but it is not persisted in the public data model or rendered with the badge. The assigned contributor also described rationale as useful future work, so this remains a valid product and security-design request rather than completed functionality.

## What This Changes

Add an auditable explanation for why a published skill is classified as requiring an API key.

## Best Possible Solution

Define and persist a versioned explanation model with structured reason codes and bounded evidence, optionally accompanied by sanitized short text, then expose it on skill detail and appropriate audit surfaces while keeping raw scanner output in protected diagnostics.

## Reproduction Assessment

Yes, at source level. Current main computes and logs rationale internally but persists and displays only the API-key-required boolean, which directly reproduces the missing auditability described by the issue.

## Solution Assessment

Yes in intent, but raw model reasoning is not the safest implementation. A sanitized structured explanation contract, with explicit public and staff-only boundaries, is the narrowest maintainable solution.
