## Label Justifications

- `P2`: This is a normal-priority transparency improvement for a recently shipped classifier, with limited immediate blast radius but clear diagnostic value.
- `impact:security`: The badge asserts that users must provide credentials, so an unexplained false positive affects their understanding of a skill’s security and setup requirements.

## Summary

Current main still does not make the classification auditable: it generates a bounded rationale and detected environment-variable list during evaluation, then persists only an `apiKeyRequired` boolean and shows a generic badge. The feature introducer also treated rationale exposure as useful future work, so the request remains necessary and should stay open for a product/security decision.

## What This Changes

Request an auditable rationale explaining why a skill receives the “API key required” classification.

## Best Possible Solution

Persist a constrained evidence summary plus classifier provenance separately from the boolean, expose it to a clearly chosen audience, document the security intent in `specs/`, and define deliberate handling for previously classified versions.

## Reproduction Assessment

Yes at the source level: current main generates rationale internally but persists only a boolean and renders a generic badge, so the missing audit trail is directly verifiable without the reporter’s original skill slug. The related report at https://github.com/openclaw/clawhub/issues/2438 supplies a concrete false-positive case for later runtime validation.

## Solution Assessment

Yes in outcome, but the implementation design remains undecided. A bounded evidence summary is more maintainable than exposing raw model reasoning, and maintainers must choose its audience, redaction, retention, and historical-rescan policy.

- The related misclassification report may require classifier-policy changes beyond rationale display, so audit evidence should not be treated as a substitute for correcting false positives.
