## Summary

Keep open. Current main has an admin-only audited repair primitive, but publish still rejects code-plugin runtimeId retargeting and active runtimeId collisions; the public package API still shows the OpenViking package bound to the placeholder runtimeId while the intended runtimeId is held by another active package, so this remains a maintainer/admin product-security follow-up.

## What This Changes

Request a guarded way to correct or migrate code-plugin package runtimeId values when a placeholder package was published with the wrong id, including the OpenViking package.

## Best Possible Solution

Have a ClawHub admin verify both package rows and audit history, then perform an audited admin migration or add an explicit moderated transfer flow that preserves current authorization, collision, and audit checks.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show existing code-plugin packages reject runtimeId changes and active runtimeId collisions. The exact OpenViking publish retry and safe migration still require admin-side production record verification.

## Solution Assessment

Yes for the guarded admin/m oderated migration direction, but not for a blind runtimeId setter. The narrow maintainable path is to use the existing admin repair primitive after production verification or expose an equivalent deliberately moderated flow.
