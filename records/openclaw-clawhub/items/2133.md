## Summary

Keep open. Current main has an admin-only audited package identity repair mutation, but this issue asks for a safe exposed or maintainer-run runtimeId correction flow and the OpenViking case reportedly collides with another active package, so it needs maintainer/admin policy judgment rather than cleanup closure.

## What This Changes

The issue requests a guarded way to correct or migrate a code-plugin package runtimeId after a placeholder package was reserved with the wrong id.

## Best Possible Solution

Have a ClawHub admin verify the production package records, use the audited repair path if appropriate for the immediate migration, and only expose a broader CLI/API/dashboard flow after the ownership and collision policy is decided.

## Reproduction Assessment

No live production publish was run, but source inspection gives a clear current-main path: existing code-plugin packages reject runtimeId changes and active runtimeId collisions. The specific OpenViking data still needs admin-side verification.

## Solution Assessment

Yes in principle for a guarded audited flow, but not for a blind runtimeId edit. The narrow maintainable path is an admin-verified migration using or extending the existing repair mutation with the same collision and audit protections.
