## Summary

Keep open. Current main has an audited admin-only package identity repair mutation, but the requested owner/moderator-facing runtimeId correction flow is not exposed, and the reported OpenViking target runtimeId collision requires maintainer/admin judgment.

## What This Changes

The issue requests a guarded way to correct or migrate a code-plugin package runtimeId after a placeholder package was reserved with the wrong runtimeId.

## Best Possible Solution

Have a ClawHub admin verify the production package records, use the existing audited repair path if policy allows, and design any exposed runtimeId correction flow with explicit ownership, collision, transfer, and audit requirements.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show existing code-plugin packages reject runtimeId changes and active runtimeId collisions. The specific OpenViking production state still needs admin-side verification before a repair can be run.

## Solution Assessment

No for a blind runtimeId edit. The safest path is an audited admin migration using the existing repair mutation, or a deliberately designed guarded public/admin flow with equivalent collision and authorization checks.
