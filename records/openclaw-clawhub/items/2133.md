## Summary

Keep open: current main still enforces code-plugin runtimeId immutability on publish, and while an admin-only repair primitive exists, the explicit moderated/settings-page flow and the OpenViking one-off migration still need maintainer action.

## What This Changes

The issue requests a safe guarded way to correct or migrate a placeholder code-plugin package runtimeId, with an immediate OpenViking package migration need.

## Best Possible Solution

Use the existing admin repair primitive for the immediate OpenViking migration, then add an explicit moderated package identity repair flow in plugin settings or admin tooling with collision checks and audit logs.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from the checkout for the production OpenViking rows. Source inspection does reproduce the generic behavior: current main rejects runtimeId retargeting on existing code-plugin packages and active runtimeId collisions.

## Solution Assessment

Yes for an explicit moderated settings/admin flow, but no for the closed implicit publish-time exception. The safer path is to keep runtimeId transfer deliberate, audited, and collision-checked.
