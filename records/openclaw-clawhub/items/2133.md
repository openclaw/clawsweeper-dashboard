## Summary

Keep open. Current main has an audited admin-only package identity repair mutation and the live OpenViking package data matches the reported conflict, but exposing or executing this migration crosses package identity policy and needs maintainer/admin judgment.

## What This Changes

The issue requests a guarded way to correct or migrate a code-plugin package runtimeId after a placeholder package reserved the intended package name with the wrong id.

## Best Possible Solution

Have a ClawHub admin verify the live package records, use or extend the audited repair path for the specific migration, and only expose a CLI/API/dashboard flow after the moderated identity-transfer policy is explicit.

## Reproduction Assessment

Yes, from source inspection and live package metadata: publishing @openviking/openclaw-plugin with runtimeId openviking would hit the existing-package runtimeId mismatch path, while the target runtimeId is already active on @openclaw/openviking. I did not run a credentialed live publish.

## Solution Assessment

No for a blind runtimeId edit. The maintainable path is an audited admin migration or a deliberately exposed guarded flow with the same authorization, collision, and audit protections as the current internal mutation.
