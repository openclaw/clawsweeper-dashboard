## Summary

Keep open. Current main still rejects publish-time runtimeId retargeting, the existing repair path is admin-only, and the maintainer discussion rejected the implicit trusted-publish PR in favor of an explicit settings/admin repair flow plus a one-off OpenViking migration.

## What This Changes

The issue requests a guarded way to correct or migrate a code-plugin package runtimeId when a placeholder package reserved the wrong runtime/plugin id.

## Best Possible Solution

Verify the production package records, complete the agreed one-off admin move if still pending, then build an explicit settings/admin repair workflow with owner checks, collision checks, audit logs, and docs.

## Reproduction Assessment

Yes, at source level: current main rejects an existing code-plugin publish when the stored runtimeId and incoming runtimeId differ, and the tests cover that rejection. The exact OpenViking production rows still require admin database verification.

## Solution Assessment

No for the previously proposed publish-time correction path: maintainer feedback says it is too implicit. The safer solution is an explicit settings/admin repair flow, with a one-off admin migration for the current OpenViking blocker.
