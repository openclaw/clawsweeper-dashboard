## What This Changes

The issue requests a guarded way to migrate a code-plugin package runtimeId after a placeholder package reserved the wrong id, with an immediate OpenViking package repair.

## Best Possible Solution

Have a ClawHub admin verify the production records, then use the existing audited admin repair path or design an explicit moderated transfer flow with equivalent authorization, collision checks, and audit logging.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show existing code-plugin runtimeId retargeting and active runtimeId collisions are rejected. The exact OpenViking migration still requires production package-row verification by an admin.

## Solution Assessment

Yes for a guarded audited migration direction, but no for a blind runtimeId edit or broad owner-facing setter. The narrow maintainable path is the existing admin repair mutation or a deliberately exposed moderated flow with equivalent authorization, collision checks, and audit logging.
