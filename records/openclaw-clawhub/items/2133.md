## Summary

Keep open. Current main has an audited admin-only package identity repair path and still enforces runtimeId immutability/collision checks, but this request needs maintainer admin/product handling because the desired OpenViking runtimeId is already held by an active package.

## What This Changes

Requests a guarded ClawHub runtimeId correction or migration flow for code-plugin packages, with an immediate OpenViking package identity transfer case.

## Best Possible Solution

Have a ClawHub admin verify the production records, use or extend the audited repair path for the OpenViking migration, and expose broader runtimeId correction only after the authorization, collision, and audit policy is settled.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show runtimeId changes on existing code-plugin packages and active runtimeId collisions are rejected. The specific OpenViking migration still needs admin-side production verification before any repair is run.

## Solution Assessment

Yes for a guarded audited migration direction, but not for a blind runtimeId edit. The narrow maintainable path is the existing admin repair mutation or an intentionally exposed flow with equivalent authorization, collision, and audit guarantees.
