## Summary

Keep open. Current main still rejects runtimeId retargeting during publish, the existing repair primitive is admin-only, and the related publish-time correction PR was closed after maintainer feedback favored an explicit settings/admin repair flow plus a one-off OpenViking migration.

## What This Changes

Request a guarded way to correct or migrate a code-plugin package runtimeId when a placeholder reservation used the wrong id.

## Best Possible Solution

Verify the production records, perform the agreed one-off admin repair if still pending, then add an explicit settings/admin repair workflow with owner checks, active collision checks, audit logging, and documentation.

## Reproduction Assessment

Yes, at source level: current main rejects an existing code-plugin publish when stored runtimeId and incoming runtimeId differ, and tests assert that rejection. The exact OpenViking production state still requires admin verification.

## Solution Assessment

Unclear for a general product fix: the existing admin repair primitive matches the one-off migration need, while the previously proposed publish-time correction was rejected as too implicit. The maintainable direction is an explicit settings/admin repair flow, not silent publish-time retargeting.
