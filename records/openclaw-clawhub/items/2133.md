## What This Changes

The issue requests a guarded way to correct or migrate a code-plugin package runtimeId after a placeholder package reserved the wrong id.

## Best Possible Solution

Have a ClawHub admin verify the production package records, then use the existing audited admin repair path or implement an explicit moderated transfer flow with equivalent authorization, collision checks, and audit logging.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show code-plugin runtimeId retargeting and active runtimeId collisions are rejected. The exact OpenViking migration still needs admin-side production record verification.

## Solution Assessment

Yes for a guarded audited migration direction, but not for a blind runtimeId edit or broad owner-facing setter. The narrow maintainable path is the existing admin repair mutation or a deliberately exposed moderated flow with the same checks.
