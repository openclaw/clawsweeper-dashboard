## Summary

Keep open. Current main has an admin-only package identity repair primitive and intentionally rejects code-plugin runtimeId retargeting and active runtimeId collisions during publish, so this remains a maintainer/admin policy decision rather than a completed owner-facing recovery flow.

## What This Changes

The issue requests a guarded way to correct or migrate a code-plugin package runtimeId after a placeholder package reservation bound the package to the wrong id.

## Best Possible Solution

Have a ClawHub admin verify the production package records, then use the existing audited admin repair path or implement an explicit moderated transfer flow with equivalent checks.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show code-plugin runtimeId retargeting and active runtimeId collisions are rejected. The exact OpenViking production migration still needs admin access to verify the affected package rows.

## Solution Assessment

Yes for a guarded audited migration direction, but not for a blind runtimeId edit or broad owner-facing setter. The narrow maintainable path is the existing admin repair mutation or a deliberately exposed moderated flow with equivalent authorization, collision checks, and audit logging.
