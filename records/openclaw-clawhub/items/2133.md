## Summary

Keep open. Current main already has an audited admin-only repair mutation and still enforces runtimeId immutability plus active runtimeId collision checks, so the remaining work is maintainer/admin handling for the OpenViking transfer and a security-sensitive product decision about any exposed correction flow.

## What This Changes

Requests a safe way to correct or migrate a code-plugin package runtimeId that was reserved with a placeholder id, with OpenViking as the immediate case.

## Best Possible Solution

Have a ClawHub admin verify the production package records, perform an audited migration or transfer only under explicit ownership policy, then expose a broader correction flow only after authorization, collision, audit, and runtime identity semantics are documented and tested.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show existing code-plugin packages reject runtimeId changes and active runtimeId collisions. The exact OpenViking migration still needs admin-side production verification because another active package currently owns runtimeId openviking.

## Solution Assessment

Unclear whether a new owner/moderator-facing flow is the best final shape. The safe immediate path is an admin-run audited migration or transfer after verifying ownership and collision state, not a blind publish-time runtimeId overwrite.
