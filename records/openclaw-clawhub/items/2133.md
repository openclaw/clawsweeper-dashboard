## Summary

Keep open: the OpenViking production unblock was handled through an audited one-off admin repair, but current main still rejects runtimeId retargeting during publish and only exposes name/owner repair through operator tooling; the broader runtimeId correction workflow remains a security/product decision.

## What This Changes

The issue requests a guarded way to correct or migrate code-plugin package runtimeId values after placeholder reservations bind the wrong id.

## Best Possible Solution

Keep the shipped admin repair-name path for one-off migrations, and design any broader runtimeId correction as an explicit settings or admin workflow with authority checks, collision rejection, audit logging, dry-run behavior, and focused tests.

## Reproduction Assessment

Yes, at source level: current main rejects direct code-plugin runtimeId retargeting during publish, and tests assert that guard. The exact OpenViking production conflict was resolved through a one-off admin repair rather than reproduced locally.

## Solution Assessment
