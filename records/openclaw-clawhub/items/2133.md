## Summary

Keep open. Current main has an audited admin-only repair primitive and intentionally blocks publish-time runtimeId retargeting/collisions, so the remaining OpenViking request needs maintainer admin/product review rather than cleanup closure.

## What This Changes

Request a safe runtimeId correction or migration path for code-plugin packages accidentally reserved with a placeholder runtimeId.

## Best Possible Solution

Have a ClawHub admin verify the production records, use or extend the audited repair/transfer path for the OpenViking migration, and expose broader runtimeId correction only after authorization, collision, and audit policy is settled.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show existing code-plugin runtimeId changes and active runtimeId collisions are rejected. The exact OpenViking migration still needs admin-side production verification before any repair is run.

## Solution Assessment

Yes for a guarded audited migration direction, but no for a blind runtimeId edit. The narrow maintainable path is the existing admin repair mutation or an intentionally exposed flow with equivalent authorization, collision, and audit guarantees.
