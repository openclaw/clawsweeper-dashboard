## Summary

Current main has an audited admin-only package identity repair primitive and intentional publish-time runtimeId guards, but the concrete OpenViking migration still needs production record verification and a product/security decision before any owner/moderator-facing correction flow is exposed.

## What This Changes

Request a guarded way to correct or migrate a code-plugin package runtimeId, including moving `@openviking/openclaw-plugin` from a placeholder id to `openviking`.

## Best Possible Solution

Have a ClawHub admin verify the production package records, then use or extend the audited repair/transfer path with explicit authorization, collision, and audit policy.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show code-plugin runtimeId retargeting and active runtimeId collisions are rejected. The exact OpenViking migration still needs admin-side production verification.

## Solution Assessment

Yes for a guarded audited migration direction, but not for a blind runtimeId edit. The narrow maintainable path is the existing admin repair mutation or an intentionally exposed flow with equivalent authorization, collision, and audit guarantees.
