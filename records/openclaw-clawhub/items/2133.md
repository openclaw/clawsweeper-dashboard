## Summary

Keep open. Current main already has an audited admin-only repair hook and still enforces runtimeId immutability/collision checks, but this issue asks for production admin handling and possibly a new exposed migration flow, so it is not a conservative issue-close candidate.

## What This Changes

The issue requests a guarded way to correct or migrate a code-plugin package runtimeId when a placeholder reservation used the wrong id.

## Best Possible Solution

Have a ClawHub admin verify and resolve the OpenViking package records using the audited repair path or an equivalent intentional migration flow, then expose broader correction only with explicit authorization, collision, and audit policy.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show existing code-plugin packages reject runtimeId changes and active runtimeId collisions. The specific OpenViking migration still needs production admin verification before any repair is run.

## Solution Assessment

Yes for the guarded audited migration direction, but not for a blind runtimeId edit. The safest current path is the existing admin repair mutation or an intentionally exposed flow with equivalent authorization, collision, and audit guarantees.
