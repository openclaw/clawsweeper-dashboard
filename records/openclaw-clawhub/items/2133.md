## What This Changes

Request a guarded way to repair or migrate a code-plugin package runtimeId after a placeholder package was reserved or published with the wrong id.

## Best Possible Solution

Have a ClawHub admin verify the production package records, then use the existing audited admin repair path or implement an explicit moderated transfer flow with equivalent checks.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show code-plugin runtimeId retargeting and active runtimeId collisions are rejected. The exact OpenViking migration still needs production package-row verification by an admin.

## Solution Assessment

Yes for a guarded audited migration direction, but not for a blind runtimeId edit or broad owner-facing setter. The narrow maintainable path is the existing admin repair mutation or a deliberately exposed moderated flow with equivalent authorization, collision checks, and audit logging.
