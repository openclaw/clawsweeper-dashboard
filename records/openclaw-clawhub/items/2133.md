## Summary

Keep open. Current main has an audited admin-only package identity repair primitive and still blocks runtimeId changes and active runtimeId collisions during publish, but the requested OpenViking migration and any owner-facing correction flow need maintainer admin/product handling because the desired runtimeId is already claimed by another active package.

## What This Changes

The issue requests a guarded way to correct or migrate a code-plugin package runtimeId after an incorrect placeholder id was reserved or published.

## Best Possible Solution

Have a ClawHub admin verify the production package rows, then use or extend the audited repair path for the OpenViking migration and expose any broader correction flow only with settled authorization, collision, and audit policy.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show runtimeId changes on existing code-plugin packages and active runtimeId collisions are rejected. The exact OpenViking migration still needs production record inspection and admin-side verification.

## Solution Assessment

Yes for a guarded audited migration direction, but not for a blind runtimeId edit. The narrow maintainable path is the existing admin repair mutation or an intentionally exposed flow with equivalent authorization, collision prevention, and audit logging.
