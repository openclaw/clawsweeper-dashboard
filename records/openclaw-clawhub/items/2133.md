Keep open. Current main has an audited admin-only repair mutation and still enforces runtimeId immutability/collision checks, but this issue asks for admin/product handling and possibly a new exposed migration flow.

## What This Changes

Request a guarded way to correct or migrate a code-plugin package runtimeId, including the OpenViking placeholder package conflict.

## Best Possible Solution

Have a ClawHub admin verify the production package records, use or extend the audited repair path for the OpenViking migration, and expose broader runtimeId correction only after the authorization, collision, and audit policy is decided.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show existing code-plugin packages reject runtimeId changes and active runtimeId collisions. The specific OpenViking migration still needs admin-side production verification before any repair is run.

## Solution Assessment

Yes for the guarded audited migration direction, but not for a blind runtimeId edit. The safest path is the existing admin repair mutation or an intentionally exposed flow with equivalent authorization, collision, and audit guarantees.
