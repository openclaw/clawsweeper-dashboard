## Summary

Keep open. Current main has an audited admin-only package identity repair mutation, but the requested owner/moderator-facing runtimeId correction flow is not exposed, and the specific OpenViking target runtimeId is already claimed by another active package.

## What This Changes

Request a guarded way to correct or migrate a code-plugin package runtimeId after a placeholder package was reserved with the wrong id.

## Best Possible Solution

Have a ClawHub admin verify the production package records, use or extend the audited repair path for the OpenViking migration, and expose broader runtimeId correction only after the ownership and collision policy is decided.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show existing code-plugin packages reject runtimeId changes and active runtimeId collisions. The specific OpenViking migration still needs admin-side production verification before any repair is run.

## Solution Assessment

Yes for the guarded audited migration direction, but no for a blind runtimeId edit. The safest path is the existing admin repair mutation or a deliberately exposed flow with equivalent authorization, collision, and audit guarantees.
