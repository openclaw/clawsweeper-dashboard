## Summary

Keep open. Current main still rejects publish-time runtimeId retargeting, production still shows the OpenViking placeholder package with the wrong runtimeId while the intended runtimeId is held by the real package, and the existing repair primitive is admin-only after the publish-time PR was closed as too implicit.

## What This Changes

ClawHub should add a guarded way to correct or migrate code-plugin package runtimeId values when a placeholder package was reserved with the wrong id.

## Best Possible Solution

Verify the production rows, perform the agreed one-off admin repair if still pending, then add an explicit settings/admin repair workflow with owner or staff authorization, active collision checks, audit logging, and documentation.

## Reproduction Assessment

Yes, at source level: current main rejects an existing code-plugin publish when stored runtimeId and incoming runtimeId differ, and tests assert that rejection. The public package API also shows the OpenViking placeholder/runtimeId conflict still exists at review time.

## Solution Assessment

Yes for the explicit guarded repair direction, but not via the closed publish-time PR; maintainer feedback favored a settings/admin repair flow and a one-off admin migration over implicit latest-publish retargeting.
