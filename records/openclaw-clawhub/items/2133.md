## Summary

Keep open. Current main already has an audited admin-only repair primitive and still intentionally rejects runtimeId retargeting/collisions during publish, so the remaining OpenViking request is an admin/product decision rather than an auto-closeable source cleanup.

## What This Changes

Request a guarded way to correct or migrate a code-plugin package runtimeId, especially for @openviking/openclaw-plugin whose intended runtimeId is already claimed by @openclaw/openviking.

## Best Possible Solution

Have a ClawHub admin verify the production records, use or extend the audited repair/transfer path for the OpenViking migration, and expose broader runtimeId correction only after the authorization, collision, and audit policy is settled.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show existing code-plugin runtimeId changes and active runtimeId collisions are rejected. The exact OpenViking migration still needs admin-side production verification before any repair is run.

## Solution Assessment

Yes for a guarded audited migration direction, but no for a blind runtimeId edit. The narrow maintainable path is the existing admin repair mutation or an intentionally exposed flow with equivalent authorization, collision, and audit guarantees.
