## Summary

Keep open. Current main has an audited admin-only package identity repair mutation, but the requested owner/moderator-facing correction flow is not exposed, and the specific OpenViking migration reportedly collides with an existing active runtimeId claimant, so this needs maintainer/admin judgment rather than cleanup closure.

## What This Changes

The issue requests a guarded way to correct or migrate a code-plugin package runtimeId after a placeholder reservation used the wrong runtimeId.

## Best Possible Solution

Have a ClawHub admin verify the production package records, use or extend the audited repair path for this migration, and only expose CLI/API/dashboard runtimeId correction after the ownership and collision policy is decided.

## Reproduction Assessment

Yes for the generic current-main behavior: source and tests show existing code-plugin packages reject runtimeId changes and active runtimeId collisions. The specific OpenViking production state still needs admin-side verification before a repair can be run.

## Solution Assessment

No for a blind runtimeId edit. The safest path is an audited admin migration using the existing repair mutation or a deliberately designed guarded public/admin flow with equivalent collision and authorization checks.
