## Summary

Keep open. Current main still blocks deleted/deactivated GitHub-linked accounts from becoming active again, related same-GitHub recovery reports remain open, and this reporter-specific restore/purge request needs identity-sensitive maintainer/operator handling rather than conservative cleanup.

## What This Changes

The issue asks for account recovery or residual GitHub identity purge after a deleted ClawHub account can no longer log in or re-register with the same GitHub account.

## Best Possible Solution

Maintainers should verify GitHub ownership and production account state, then choose a documented deletion recovery policy and handle any approved restore, unlink, or purge through an audited operator path.

## Reproduction Assessment

Yes, at source level. Deletion writes `deactivatedAt`/`purgedAt`, OAuth rejects that state, active-user helpers hide the user, and CLI token creation requires an active user; the reporter's exact production row still needs operator verification.

## Solution Assessment

No autonomous source-only fix is the best next step. The maintainable path is verified operator handling plus an explicit product/security decision about whether deleted GitHub identities may be restored, unlinked, recreated, or support-only.
