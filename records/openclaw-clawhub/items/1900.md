## Summary

Keep open. The report is source-reproducible on current main, but restoring or purging a deleted GitHub-linked account is an account-specific, security-sensitive operator/product decision rather than conservative cleanup.

## What This Changes

The reporter asks ClawHub to restore their deleted account or purge residual GitHub identity data so the same GitHub account can sign up again.

## Best Possible Solution

Verify GitHub ownership and the production account row, then choose one documented policy: support-only irreversible deletion, audited restore, safe relink, or clean re-registration, with backend/UI/CLI/docs/tests updated to match.

## Reproduction Assessment

Yes, at source level. Deletion writes deactivatedAt/purgedAt and revokes tokens, OAuth rejects that state, active-user helpers hide the user, and CLI token creation requires an active user; the exact production account still needs operator verification.

## Solution Assessment

No autonomous source-only fix is the best next step. The maintainable path is verified operator handling plus an explicit product/security decision about restore, relink, recreate, or support-only deletion semantics.
