- `P1`: The issue blocks an affected user from ClawHub login and CLI-token access after account deletion, and several related reports show the same core workflow failure.
- `impact:security`: Any restore, purge, relink, or re-registration path changes identity binding, deletion finality, ban separation, and token revocation boundaries.

- `impact:session-state`: The user can return from provider auth without becoming an active ClawHub user, leaving website and CLI flows in logged-out or incomplete state.

## Summary

Keep open. Current main intentionally treats self-deleted/deactivated GitHub-linked accounts as inactive and non-restorable, while this account-specific restore/purge request needs verified operator handling plus a product/security decision; related same-root reports remain open rather than superseding this issue.

## What This Changes

The issue asks maintainers to restore or purge residual data for a deleted GitHub-linked ClawHub account so the same GitHub identity can sign in again.

## Best Possible Solution

Verify GitHub ownership and the production account row, then choose and document one deleted-account policy: support-only irreversible deletion, audited restore, secure relink, or clean re-registration with token, ownership, ban, and purge semantics covered.

## Reproduction Assessment

Yes, at source level. deleteAccount writes deactivatedAt/purgedAt and revokes tokens, handleDeletedUserSignIn rejects deactivated or legacy deleted users during OAuth completion, active-user helpers hide the user, and CLI token creation requires an active user; the specific production row still needs operator verification.

## Solution Assessment

No autonomous source-only fix is the best next step. The maintainable path is verified operator handling plus an explicit product/security decision about whether deleted GitHub identities may be restored, unlinked, recreated, or kept support-only.
