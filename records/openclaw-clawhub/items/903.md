## Summary

Keep open. Current main still contains the full-table global-stats recount paths and single-key rate-limit counters this PR is trying to replace, so the intended change is not implemented on main. The PR is useful but still needs maintainer review because the sharded rate-limit mutation can soften aggregate cap enforcement under concurrency and the countPublicSkills fallback tests remain aligned with the old behavior.

## What This Changes

The PR removes live global-stats recount fallbacks and the deprecated manual stats mutation, updates seeding docs to call the action reconciler, and changes API rate limiting from one per-window counter to sharded write counters with aggregate read status.

## Best Possible Solution

Land a revised, maintainer-approved patch that removes the live full-table global-stats scans, deletes the deprecated mutation and stale docs reference, updates tests for action-based reconciliation, and reduces rate-limit write contention while preserving or explicitly documenting aggregate cap semantics. If it resolves the user-facing throttling symptom, link the outcome back to #654.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve the aggregate rate-limit cap:** `convex/rateLimits.ts:69-73`
  - body: The status query sums all shards, but this mutation then chooses one random shard and increments only that shard. Concurrent requests near the cap can all observe allowed=true and write different shard documents without conflicting, so the window can admit more than args.limit requests. Keep a hard per-window reservation/cap or explicitly make this API policy a soft limit before using it on protected routes.
  - confidence: 0.87
- **[P2] Update countPublicSkills tests for the new fallback:** `convex/skills.ts:2753`
  - body: countPublicSkills now returns 0 when globalStats is missing, but the existing tests still assert the live-count fallback returns counted skills in that case. As-is, the branch should fail the current test suite until those assertions or equivalent reconciliation-path coverage are updated.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff changes API abuse-control semantics and may weaken hard per-window rate-limit enforcement under concurrent sharded writes.

Concerns:

- **[high] Sharded writes can bypass the aggregate cap:** `convex/rateLimits.ts:69`
  - body: Because the mutation writes only one random shard after a separate aggregate status query, concurrent requests near the limit can increment different shards without conflicting and admit more requests than args.limit. That matters for protected API routes that rely on this limiter as an abuse-control boundary.
  - confidence: 0.86
