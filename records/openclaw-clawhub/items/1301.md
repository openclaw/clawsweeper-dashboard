## Summary

The central request remains necessary: current main has administrator-controlled Official status, an internal `trustedPublisher` permission, and owner rescan tooling, but no public community verification application or eligibility path. The scan-delay report is now partly addressed by rescan guidance, while the trust badge question still requires an explicit security/product decision and overlaps the broader open discussions at https://github.com/openclaw/clawhub/issues/184 and https://github.com/openclaw/clawhub/issues/1994.

## What This Changes

Add a community-facing publisher verification path and clarify how code-plugin publishers should resolve scans that remain pending.

## Best Possible Solution

Adopt one documented publisher-trust model that clearly separates Official, internal publishing permission, automated scan results, and any community verification tier, then publish owner rescan and escalation expectations for scans that exceed the normal processing window.

## Reproduction Assessment

Partly yes: current source establishes with high confidence that no public community verification path exists, but reproducing the historical pending scan requires the affected ClawHub package URL, version, and current production scan record.

## Solution Assessment

No, adding a badge or exposing the internal trust flag alone would be unsafe; the maintainable solution starts with a security/product policy and then implements matching review, display, revocation, and operational escalation behavior.
