## Summary

Keep open: current main has package trust channels, release verification tiers, trusted-publisher plumbing, and package scan queues, but it still does not settle the verified community publisher policy or explain the named production scans that reportedly stayed pending.

## What This Changes

This issue asks for a public verified/trusted community publisher path for active code-plugin contributors and for investigation of pending security scans on several named published plugins.

## Best Possible Solution

Resolve the vetted-community trust model in the open RFC, document the approved policy across specs/API/admin/UI, and inspect production scan records for the named packages before scoping any source fix.

## Reproduction Assessment

No. The verified-publisher part is a policy/support request, and the pending-scan claim requires production packageReleases, securityScanJobs, scanner worker state, scanner env, and failure logs for the named packages.

## Solution Assessment

Yes for keeping this open for manual review. The existing trust and scan primitives are the right substrate, but a stronger community trust signal or manual scan-status action needs maintainer-approved semantics and production audit first.
