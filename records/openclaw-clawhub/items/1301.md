## Summary

Keep open. The named code plugins no longer appear stuck in `pending` in the public API, but the verified/community publisher path remains an unresolved trust-and-security policy question that should not be closed by source cleanup automation.

## What This Changes

The issue asks whether active community code-plugin publishers can obtain a verified/trusted status between `community` and `official`, and why several named plugins remained in `Scan: pending` for more than 24 hours.

## Best Possible Solution

Maintainers should answer the verified/vetted publisher policy alongside the open review-path RFC, inspect the named packages' production scan/audit history, and only open a narrower source fix if that inspection finds a scanner queue or status propagation defect.

## Reproduction Assessment

No, not for the pending-scan symptom: current public API checks show the named packages have concrete `clean` or `suspicious` statuses rather than `pending`. The verified-publisher path is a policy/admin question rather than a reproducible source bug.

## Solution Assessment

No automated source change is the best next step. Current main has trust-channel, verification-tier, trusted-publisher, and scan-scheduling primitives, but the remaining answer requires maintainer/security policy plus production scanner inspection.
