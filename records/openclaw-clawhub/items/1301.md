## Summary

Keep this issue open: current main and v0.20.2 provide an owner-initiated package rescan path, addressing the pending-scan recovery question, but they do not provide the requested intermediate verified-community publisher status. Related verification requests remain open, and defining that trust signal still requires an explicit product and security decision.

## What This Changes

Add a community publisher trust path for code plugins and clarify or recover plugin scans that remain pending.

## Best Possible Solution

Keep immutable per-release scan outcomes authoritative, document the shipped owner package-rescan path, and define any publisher reputation tier in one canonical trust-policy issue covering eligibility, evidence, scope, revocation, appeals, expiry, and unambiguous UI wording.

## Reproduction Assessment

Partly yes: current source clearly establishes the pending package-scan model and owner rescan path, but the named plugins' historical production delay cannot be reproduced without live package records and scanner job logs.

## Solution Assessment

No, not as one combined verification status. The narrow maintainable approach is to use the existing rescan mechanism for stuck releases while keeping any publisher identity or reputation signal distinct from release-specific security evidence.
