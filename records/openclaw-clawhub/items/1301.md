## Summary

Keep open. Current main and the latest release have package verification tiers, official/community channels, trusted-publishing configuration, and package scan queues; live package reads no longer show the named releases as pending. The missing part is still a maintainer security/product policy for a public community verified-publisher trust signal and any production explanation for the original scan delay.

## What This Changes

The issue asks whether community code-plugin contributors can get verified-publisher status between community and official, and why several named code plugins showed scans pending after more than 24 hours.

## Best Possible Solution

Maintainers should define and document the community publisher verification policy, its relationship to official publishers, trusted publishing, and release verification tiers, then inspect production scanner history for the named plugins and split any concrete scanner bug into a narrower issue.

## Reproduction Assessment

No high-confidence current failure reproduction: the original pending-scan symptom was not reproduced on the public package security endpoints for the named latest releases that responded. The verified-publisher portion is a product/security policy request, not a runtime bug reproduction.

## Solution Assessment

No, not as a code-only source fix right now. Current main has scanner and trust primitives, but the durable solution is a maintainer-approved verification policy plus production scanner inspection before any schema, UI, API, or docs change grants a new trust signal.
