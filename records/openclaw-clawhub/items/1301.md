## Summary

Current main now gives plugin owners a manual rescan path, but it still does not provide or document the requested community publisher verification program; publisher eligibility remains a trust-policy decision, and the named production scan records require authorized operational inspection.

## What This Changes

Define a community publisher verification path and clarify or resolve prolonged pending security scans for published code plugins.

## Best Possible Solution

Publish a transparent community publisher trust policy with explicit assurance and revocation semantics, while retaining owner-triggered rescans and documenting an operator escalation path for package jobs that remain pending.

## Reproduction Assessment

Yes for the verification gap: current source and policy show internal Official and trusted-publisher controls plus owner rescans, but no public community verification application path. The historical pending scans themselves need current production package and version records for a live reproduction.

## Solution Assessment

No narrow source repair is established yet; maintainers must first decide the verification program's eligibility and assurance contract, while an authorized operator should diagnose the named releases through the existing rescan and queue tooling.
