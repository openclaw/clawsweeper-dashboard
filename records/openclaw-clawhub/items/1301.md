## Summary

Keep open: current main provides code-plugin verification metadata, policy-assigned trust channels, internal publisher controls, scanner scheduling, and owner rescan flows, but it does not provide a documented application path for community publishers, and the historical production scan delays for the named plugins cannot be resolved from source alone. The overlapping open requests confirm that the remaining question is a maintainer security/product policy decision rather than a completed implementation.

## What This Changes

The issue asks for a community publisher-verification path and clarification of prolonged pending scans for published code plugins.

## Best Possible Solution

Adopt and document one maintainer-approved community verification policy—covering eligibility, evidence, verification scope, review, expiry, and revocation—then inspect production scan records for the named plugins and open a narrower scanner bug only if that inspection identifies a source defect.

## Reproduction Assessment

Unclear. The verification-path request is a feature/policy question rather than a reproducible bug, while reproducing the old pending-scan condition requires current production package versions, scan records, scheduler state, and deployment logs that are not present in the checkout.

## Solution Assessment

No, not as a single automatic code fix. Current rescan controls are the appropriate operational recovery path, but the verification tier requires an explicit trust-and-safety policy, and any actual scanner defect should be split into a focused issue after production diagnosis.
