## Summary

Current main now provides owner-initiated rescans for stalled plugin scans, but it explicitly has no public route to Official publisher status and still treats a separate trusted community verification cohort as an unresolved policy choice. This issue remains useful as the broad canonical discussion rather than being closed in favor of the narrower overlapping requests.

## What This Changes

Define a trust signal or verification path for active community code-plugin publishers and clarify how owners recover plugin scans that remain pending.

## Best Possible Solution

Keep Official status admin-scoped, document the existing package-owner rescan flow, and define a separate community verification tier only after specifying eligibility, verification scope, renewal, revocation, and its visual distinction from Official status.

## Reproduction Assessment

Yes for the central policy gap: current main explicitly has no generic public Official-status path and leaves trusted-cohort eligibility open. The original 24-hour production scan records cannot be reconstructed from the checkout, although the current owner rescan path is clear.

## Solution Assessment

No, treating Official status as an informal community verification path would conflate distinct trust claims. The narrow maintainable direction is a separately specified verification tier while using the existing owner rescan mechanism for stalled scans.
