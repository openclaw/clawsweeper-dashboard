## Summary

Keep open. Current main has Official publisher rows, release verification tiers, trusted publishing, and package scan/rescan plumbing, but it still does not provide a public community verified-publisher path; the named packages are no longer publicly pending, so the remaining scanner question needs production history rather than a source-tree fix.

## What This Changes

The issue asks whether active community code-plugin publishers can get a verified/trusted publisher status and why their code-plugin scans stayed pending after 24+ hours.

## Best Possible Solution

Define and document one maintainer-owned community publisher trust model, then inspect production scanner history for the named code plugins and open a narrower source bug only if that inspection finds a queue or worker defect.

## Reproduction Assessment

Partly: source confirms that the public self-serve publisher path does not grant trusted or official status, so the requested community verification path is missing. The original pending-scan symptom no longer appears on the public package detail endpoints for the named latest releases, but production history would be needed to explain why it happened.

## Solution Assessment

No, not as an ad hoc badge grant or code-only fix. The maintainable solution is a security/product policy that defines the trust signal before any UI, API, or moderation change grants it.
