## Summary

Keep open. This issue still needs maintainer/security policy work for a verified or vetted community plugin path, and any historical scan-pending explanation requires production scanner records even though the seven named packages no longer show `pending` in the public package security API.

## What This Changes

The issue asks whether active community code-plugin publishers can get a trust status between `community` and `official`, and why seven named plugins stayed `Scan: pending` after publication.

## Best Possible Solution

Decide the vetted-community publisher/plugin policy in the open RFC, document the trust semantics, and inspect production scanner history for the named packages only if maintainers still need a historical scan-state answer.

## Reproduction Assessment

No. The verified-publisher request is policy/admin work, and live public package security checks for all seven named packages now return `pending: false`, so the historical pending symptom is not reproducible from this checkout alone.

## Solution Assessment

No source-only fix is the best next step. Current main has trust, verification, scanner scheduling, and recovery primitives, but verified-community status needs maintainer security/product policy and scanner history needs production records.
