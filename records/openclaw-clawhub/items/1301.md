## Summary

Keep open: current main has package trust primitives and scanner queues, but it does not implement a public verified or vetted community-publisher path, and the named production scan-pending state still needs deployed Convex/scanner inspection.

## What This Changes

The issue asks whether active community code-plugin publishers can get verified/trusted status and why several published code plugins stayed at `Scan: pending` for more than 24 hours.

## Best Possible Solution

Maintainers should answer the trust-policy portion through https://github.com/openclaw/clawhub/issues/2272, then inspect production scanner records and logs for the named packages and open a narrower source bug only if that inspection finds a queue, worker, env, or scan-status defect.

## Reproduction Assessment

Source-reproducible for the missing public verification path: current specs, docs, and code expose Official/community/private plus admin-only trust primitives, not a community application flow. The named production pending-scan state is not reproducible from the source checkout because it requires deployed Convex records, scanner configuration, and worker logs.

## Solution Assessment

Yes, keeping this open for maintainer policy and operations review is the best path. A code PR before the trust contract and production scanner facts are known would either encode a premature security signal or guess at the wrong scanner failure mode.
