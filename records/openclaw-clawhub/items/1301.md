## Summary

Current main does not implement the requested intermediate community publisher verification path; Official remains a narrowly managed policy flag, while related open issues still seek the same trust signal. The scanner stack is now documented and implemented for plugins, but the report’s package-specific pending state needs current operational evidence rather than closure.

## What This Changes

Determine whether community code-plugin publishers have a verification path and why plugin security scans can remain pending.

## Best Possible Solution

Keep Official narrowly controlled, define a separate evidence-backed community publisher or audited-release signal with revocation and appeal semantics, and document an owner-visible code-plugin scan timeline plus retry/escalation path.

## Reproduction Assessment

## Solution Assessment

No single implementation is ready to apply. The maintainable solution requires an explicit trust-policy decision, while any scanner repair should follow package-specific live diagnostics rather than assuming the historical delay still occurs.
