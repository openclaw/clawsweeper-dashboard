## Summary

Current main implements package scanning, Official publisher status, and trusted CI publishing, but deliberately treats them as separate signals and does not provide the requested community verified-publisher tier. This remains a valid product and security-policy decision; the historical scan delay also lacks enough current production evidence to declare resolved or reproducible.

## What This Changes

The issue requests a community publisher-verification path and clarification of prolonged pending scans for code plugins.

## Best Possible Solution

Define a community publisher attestation that is visibly separate from Official status, trusted CI publishing, and per-version scan verdicts, with explicit eligibility, scope, renewal, revocation, appeals, and documented scan escalation timing.

## Reproduction Assessment

No, not for the reported 24-plus-hour delay: current source confirms a real pending state and worker-backed package scanning, but no live current deployment was exercised to reproduce prolonged pending scans. The publisher-verification request itself is a product-policy feature rather than a reproducible bug.

## Solution Assessment

No, existing Official status or trusted GitHub Actions publishing is not an adequate substitute because current policy intentionally assigns them different meanings. A separate, narrowly defined community identity attestation is the maintainable direction if maintainers approve its product and security contract.
