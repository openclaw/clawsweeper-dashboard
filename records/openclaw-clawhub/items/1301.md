## Summary

Keep open. This remains a live ClawHub trust-policy and scanner-operations question: current main has verification tiers, trusted-publisher gates, scan scheduling, and scan backfill, but it does not document a community verified-publisher path, and the original owner-visible pending-scan claim still needs production scanner inspection.

## What This Changes

The issue asks whether active community code-plugin publishers can get verified-publisher status and why several named code plugins showed pending scans after publication.

## Best Possible Solution

Maintainers should define and document the community verification or vetted-publisher policy, then inspect production scanner records for the named packages and open a narrower source issue only if that review finds a queue, backfill, config, API, or UI defect.

## Reproduction Assessment

Unclear. Public API checks against current production no longer show pending for the named latest releases, but owner-visible dashboard history and production scanner logs are needed to verify why the original publishes remained pending for 24+ hours.

## Solution Assessment

No for automation. Existing trust-channel, trusted-publisher, verification-tier, and scan-backfill code is the right substrate, but the requested status and historical scan explanation require maintainer/security policy plus production inspection before a maintainable source change can be scoped.
