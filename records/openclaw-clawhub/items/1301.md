## Summary

Current main implements plugin verification tiers, policy-controlled Official status, trusted automated publishing, scan scheduling, and owner-requested package rescans, but it does not provide a documented public application path for community verification. The named packages’ prolonged pending state also requires production scanner records and logs that the source checkout cannot supply.

## What This Changes

Establish a community publisher trust signal and clarify or repair prolonged pending scans for code plugins.

## Best Possible Solution

Define and document a staff-governed community verification program—or explicitly state that none is offered—while inspecting the named production package releases and scanner jobs; split any discovered queue, configuration, or rescan defect into a focused implementation issue.

## Reproduction Assessment

Partly. Current source gives a high-confidence reproduction of the missing generic public verification application path, but reproducing why the seven named packages remained pending requires production `packageReleases`, scheduler state, scanner configuration, insights, and failure logs.

## Solution Assessment

No, an automatic source patch or one-off publisher elevation is not the best solution. Maintainers should first decide the durable trust taxonomy and inspect production scanner health, then scope a narrow code or operations fix only if that inspection identifies one.
