## Summary

Keep open. The pending-scan symptom no longer reproduces in the public API, but the verified/community publisher path is still an unresolved trust and security policy question that needs maintainer review rather than source cleanup.

## What This Changes

The requester asks whether active community code-plugin publishers can get a verified status between community and official, and why several named code plugins stayed in Scan: pending for more than 24 hours.

## Best Possible Solution

Maintainers should resolve the vetted/verified community plugin policy through the open RFC, then inspect production scan history for the named plugins and only open a narrow source fix if that inspection finds a scanner queue or status propagation defect.

## Reproduction Assessment

No, not for the pending-scan symptom: current public package security responses show the named plugins have clean or suspicious results with pending=false. The verified-publisher path is a product and trust-policy question rather than a reproducible source bug.

## Solution Assessment

No automated source change is the best next step. Current main has trust channels, verification tiers, trusted-publisher gates, and scanner scheduling, but granting or defining verified community publisher status requires maintainer security and product policy.
