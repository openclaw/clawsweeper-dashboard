## Summary

Current main intentionally separates Official publisher designation, automated-publish permission, release verification, and scanner outcomes; it does not provide the requested community verified-publisher tier. The scan pipeline and owner rescan path exist, but the reported 24-hour pending state lacks current package/version evidence, so this remains a product and security-policy decision rather than a closeable implementation issue.

## What This Changes

Establish a community publisher-verification path and clarify or resolve prolonged pending scans for code plugins.

## Best Possible Solution

Adopt one documented trust model that separately labels publisher identity, artifact provenance, scanner verdict, and any human audit, with a governed application path and an observable rescan/status workflow.

## Reproduction Assessment

## Solution Assessment

No, not as an undifferentiated verified-publisher badge; the maintainable solution must specify whether each signal attests to identity, release provenance, automated scanning, or human review.
