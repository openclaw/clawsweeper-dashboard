## Summary

Keep open. Current main and v0.22.0 have official-publisher rows, trusted-publishing config, package verification fields, and package scan queues, but they still do not provide a public community verified-publisher path; the original scan-delay symptom also needs production scanner/job inspection rather than a source-only cleanup close.

## What This Changes

The issue asks whether active community code-plugin publishers can become verified and why the reporter's published plugins stayed in pending scan state after more than 24 hours.

## Best Possible Solution

Maintainers should define and document the community publisher verification policy and its relationship to Official publishers, trusted publishing, release verification tiers, and scan status, then split any confirmed production scanner bug into a narrower source issue.

## Reproduction Assessment

No. The public package API checks did not reproduce pending scan state for the named packages, and the verified-publisher portion is a policy/feature request rather than broken existing behavior.

## Solution Assessment

No, not as a source-only fix right now. The maintainable path is a maintainer-approved trust policy plus production scanner inspection before any schema, UI, API, or docs change grants a new public trust signal.
