## Summary

Keep open: current main provides official/community/private channels, artifact verification tiers, scan results, and internal trusted-publishing controls, but it still does not provide the requested public vetted-community publisher path. The historical pending-scan symptom is no longer the central unresolved problem; the trust-policy decision remains active in https://github.com/openclaw/clawhub/issues/2272, and draft https://github.com/openclaw/clawhub/pull/2411 explicitly does not add a vetted status.

## What This Changes

Define a public verification path for active community code-plugin publishers and clarify prolonged pending scan results.

## Best Possible Solution

Use the canonical RFC to define a documented vetted-community trust level with package-versus-publisher scope, review criteria, expiration and revocation rules, incident handling, and UI/API semantics, while retaining per-release scans and trusted publishing as separate evidence signals.

## Reproduction Assessment

Not applicable to the central verification-policy request because it asks for a new trust status and workflow. The old pending-scan observation would require production records for the named versions to reproduce as an operational scanner failure.

## Solution Assessment

No, an isolated publisher badge or manual exception from this issue would not be the safest solution. The narrower maintainable path is to settle the shared trust semantics in https://github.com/openclaw/clawhub/issues/2272 and then implement one documented workflow.
