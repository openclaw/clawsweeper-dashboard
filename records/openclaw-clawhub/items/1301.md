## Summary

Keep open. Current main has official publisher rows, trusted publishing, release verification tiers, and package scan scheduling, but it still lacks a public verified or vetted-community publisher path; the named pending code-plugin scans also need production scanner/job inspection rather than a source-only close.

## What This Changes

The issue asks whether active community code plugin publishers can get a verified trust signal and why several published code plugins remained at scan pending.

## Best Possible Solution

Use https://github.com/openclaw/clawhub/issues/2272 for the vetted-community policy decision, then inspect production scanner state for the named packages and open a narrower source bug only if the pending scans trace to queue, config, or worker failure.

## Reproduction Assessment

No. The verified-publisher portion is a policy/feature request, and the scan-pending portion needs production package records, queued job state, scanner environment, and failure logs for the named plugins before it can be reproduced with high confidence.

## Solution Assessment

No, not as a direct source fix right now. The maintainable path is a maintainer-approved vetted-community policy plus production scanner inspection, with code/docs changes only after the trust semantics or a concrete scanner failure are known.
