## Summary

Keep open: this remains a ClawHub marketplace trust and scanner-status support item, and the repo cleanup policy keeps issues open. Current main has community/official/private channels, release verification tiers, trusted-publisher plumbing, and package scan queues, but it still does not define a verified community publisher path; the named packages no longer appear pending in the public API, so production scanner history is needed to explain the original delay.

## What This Changes

The issue asks whether active community code-plugin publishers can receive a verified-publisher trust signal and why several published code plugins remained pending scan status after 24+ hours.

## Best Possible Solution

Resolve the vetted-community trust model in https://github.com/openclaw/clawhub/issues/2272, document the approved policy across specs/API/admin/UI, and inspect production scan records for the named packages before scoping any source fix.

## Reproduction Assessment

No. The verified-publisher part is a policy/support request, and the current public package API no longer shows the named plugins as pending; explaining the original delay requires production packageRelease, securityScanJobs, worker, env, and failure-log inspection.

## Solution Assessment

Yes for keeping this open for manual review. The current trust and scan primitives are the right substrate, but a stronger community trust signal or scan-status action needs maintainer-approved semantics and production audit first.
