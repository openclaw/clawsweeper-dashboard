## Summary

Current main provides package verification tiers, Official publisher policy, trusted CI publishing, and owner-accessible package rescans, but it still does not define a public community publisher-verification enrollment path. The historical pending scans also require production package/job inspection, so this remains a maintainer security/product and operations question rather than an implemented or safely automatable source fix.

## What This Changes

Define whether active community code-plugin contributors can obtain a visible verified-publisher status and investigate code-plugin scans that remain pending.

## Best Possible Solution

Define a public, evidence-based community verification policy that remains visibly separate from Official identity and CI trusted-publisher credentials, then inspect the named production package releases and scanner jobs; any confirmed queue or state-transition defect should become a narrower bug report with package versions and job evidence.

## Reproduction Assessment

Partly yes: current source provides no public community verification enrollment path, so that policy gap is source-reproducible. The reported 24-hour pending scans cannot be reproduced with high confidence without current package URLs, release versions, rescan results, and production scanner-job state.

## Solution Assessment

No, not as one undifferentiated change. Maintainers should define the trust signal and assurance contract first, while handling stalled scans through the existing package-rescan path and production diagnostics before deciding whether a source repair is needed.
