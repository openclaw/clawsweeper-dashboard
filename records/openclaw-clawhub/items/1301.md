## Summary

Keep open. Current main has controlled Official publisher designation, release verification tiers, trusted-publish plumbing, and async package scan/rescan paths, but it still does not provide the requested public community verified-publisher path, and the reported 24-hour pending scans need current production package/version evidence before they can be called fixed or reproducible.

## What This Changes

The issue asks whether community code-plugin publishers can obtain verified status and why several named code plugins stayed at `Scan: pending` for more than 24 hours.

## Best Possible Solution

Define one public trust model that separates publisher identity, release provenance, automated scanner verdicts, and any human audit, then inspect the named production package releases and document the author-facing status/rescan workflow.

## Reproduction Assessment

Unclear: source confirms there is no public community verified-publisher enrollment path and that package scans are asynchronous, but the 24-hour pending symptom needs current package URLs, exact versions, and production scan/job state.

## Solution Assessment

No, not as a code-only change or ad hoc badge grant. The maintainable path is a maintainer-owned trust policy plus production scanner inspection, followed by a narrower source/docs issue only if that inspection finds a concrete defect.
