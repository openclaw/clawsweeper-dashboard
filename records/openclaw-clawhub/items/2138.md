## Summary

Keep this PR open: it proposes a new bulk export API that is not already on current main, but the latest branch still needs security/product review, real behavior proof, and fixes for moderation-gate and rate-limit ordering issues before merge.

## What This Changes

The PR adds an admin-authenticated `GET /api/v1/skills/export` endpoint for paginated ZIP skill exports, with route/schema wiring, export auth, rate limiting, and ZIP assembly helpers.

## Best Possible Solution

Define a bulk export contract that reuses existing public skill visibility and download moderation gates, rate-limits auth failures before token lookup, and includes maintainer-approved mirror access policy plus redacted endpoint proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the latest PR head shows the export query can select rows that current public visibility and download code reject, and the handler checks auth before applying the export rate limit. I did not run the PR branch.

## Solution Assessment

No; the current patch is not the narrowest safe solution because it creates a new bulk blob path without reusing existing moderation gates and applies rate limiting only after successful export auth. A safer path is to enforce public visibility before blob reads and apply the export rate limit before auth checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Filter export rows through public moderation gates:** `convex/skills.ts:9639`
  - body: The export query pages `skillSearchDigest` with `by_active_created`, which only filters `softDeletedAt`. Hidden, pending-scan, or `blocked.malware` rows can still be selected and then have blobs bundled into the ZIP, bypassing the public/download gates that currently reject them. Filter through the same public visibility/moderation rules before reading version blobs.
  - confidence: 0.93
- **[P2] Apply export rate limits before auth:** `convex/httpApiV1/skillsV1.ts:1503-1506`
  - body: `requireExportAuth` runs before `applyRateLimit`, so missing/invalid bearer tokens and non-admin role probes return without consuming the export IP/user bucket while still doing token auth work. Match the existing protected v1 handler pattern by applying the export rate limit first and passing those headers into auth failures.
  - confidence: 0.82
