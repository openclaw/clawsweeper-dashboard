## Summary

Keep open: this is a new export surface that is not implemented on main, and the proposed patch currently bypasses existing moderation/download visibility gates while lacking concrete real-behavior proof.

## What This Changes

The PR adds an admin-authenticated `GET /api/v1/skills/export` endpoint for paginated ZIP exports, plus route wiring, auth/rate-limit helpers, a dev seed script, generated route output, mirror-sync docs, and an integration test script.

## Best Possible Solution

Define a public-registry export contract that reuses existing visibility and download moderation gates, has an explicit mirror/admin role and rate-limit policy, and ships with maintainable automated tests and docs.

## Reproduction Assessment

Yes for the review findings: source inspection shows the export query can select rows that current public visibility and download code would reject, and the test script hardcodes tokens that current seed code generates randomly. I did not run the PR branch.

## Solution Assessment

No; the current patch is not the narrowest safe solution because it creates a new bulk ZIP path without reusing the existing moderation/download gates and relies on fixed local bearer tokens. A safer approach is to filter through the same public registry contract before reading blobs and to derive test credentials from seed output or env vars.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter export rows through public moderation rules:** `convex/skills.ts:9398`
  - body: The export query scans `skillSearchDigest` by `by_active_created`, which only constrains `softDeletedAt`; hidden or `blocked.malware` rows can still have `softDeletedAt === undefined` and a latest version, so this new bulk ZIP path can bundle files that `/api/v1/download` and `toPublicSkill` currently block. Query/filter with the same public visibility rules before fetching version blobs.
  - confidence: 0.93
- **[P2] Load seeded tokens instead of hardcoding bearer values:** `test_export_api.sh:18-19`
  - body: `seedCliRoleHelpFixtures` returns fresh random tokens from `generateToken()` on each run, so these fixed `clh_...` values will not exist in a clean dev deployment and the auth tests will fail with 401/403. Read the returned tokens from the seed command or require env vars instead.
  - confidence: 0.88
