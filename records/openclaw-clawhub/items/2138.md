## Summary

Keep open: current main does not implement this endpoint, and the PR still adds a security-sensitive bulk export path that can bypass existing moderation/download gates, lacks real behavior proof, and needs maintainer product/security review for mirror support.

## What This Changes

The PR adds an admin-authenticated paginated `/api/v1/skills/export` ZIP endpoint plus route constants, auth/rate-limit helpers, ZIP merge helpers, and an internal date-range query.

## Best Possible Solution

Define an explicit supported mirror/export contract that reuses public visibility and download moderation gates, rate-limits unauthorized attempts, includes focused tests/spec/docs, and ships only after maintainer product/security approval plus redacted runtime proof.

## Reproduction Assessment

Yes for the review findings by source inspection: a non-soft-deleted hidden, removed, pending-scan, malware-blocked, or otherwise non-public skill in the requested date range can be selected by `by_active_created`, then the export handler reads its version blobs without the existing public/download gates. I did not run the PR branch.

## Solution Assessment

No; the current patch is not the best way to solve the mirror-sync problem because it adds a new bulk archive surface before the product trust model is settled and before reusing the existing moderation/download contract. The safer path is a maintainer-approved export API that filters through the same public registry rules and rate-limits auth failures before any token lookup or blob work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter export rows through public moderation gates:** `convex/skills.ts:9639`
  - body: `listByDateRange` pages `skillSearchDigest` with `by_active_created`, which only excludes soft-deleted rows. Hidden, removed, pending-scan, or `blocked.malware` skills can still have a latest version and will be bundled here, bypassing the gates that `toPublicSkill` and `/api/v1/download` currently enforce before blob reads.
  - confidence: 0.93
- **[P2] Rate-limit export requests before token auth:** `convex/httpApiV1/skillsV1.ts:1503-1507`
  - body: The handler authenticates first and only calls `applyRateLimit` after `requireExportAuth` succeeds, so missing or invalid bearer attempts never consume the new export/IP bucket. Existing HTTP API handlers rate-limit before API-token auth; move the limiter ahead of export auth and pass its headers into auth failures so this sensitive endpoint cannot be hammered with unbounded 401s and token-hash lookups.
  - confidence: 0.82
