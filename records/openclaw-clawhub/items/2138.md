## Best Possible Solution

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Page exports by updates, not skill creation:** `convex/skills.ts:10413`
  - body: The export is intended for hourly mirror sync, but this query only scans `by_active_created`. When an existing skill is republished, its files and `updatedAt` change while `createdAt` stays old, so incremental date windows will silently miss updates after the initial import. Use an update-time/change-feed cursor and add a regression for republishing an old skill.
  - confidence: 0.91
- **[P1] Preserve the public visibility boundary:** `convex/skills.ts:10413`
  - body: This raw digest page only constrains `softDeletedAt` and the handler then reads latest-version blobs directly. Current public APIs and downloads go through `isPublicSkillDoc`/moderation guards, so this can bulk-export hidden, malware-blocked, or otherwise non-public content to a mirror. Filter to the approved public export set or get an explicit maintainer-approved admin export boundary with tests.
  - confidence: 0.89
- **[P2] Rate-limit failed export auth attempts:** `convex/httpApiV1/skillsV1.ts:1571-1575`
  - body: `requireExportAuth` runs before `applyRateLimit`, so requests with missing or invalid bearer tokens return 401/403 without consuming the IP fallback bucket documented for invalid tokens. Put the export limiter before auth, or explicitly charge a failed-auth bucket before returning.
  - confidence: 0.88
- **[P2] Move export out of the slug route namespace:** `convex/http.ts:77-79`
  - body: Registering `/api/v1/skills/export` before the `/api/v1/skills/{slug}` router makes the documented detail route unusable for a skill named `export`, and that word is not currently reserved by the slug validator. Use a reserved namespace such as `/api/v1/skills/-/export` and reserve/test the route.
  - confidence: 0.86
- **[P2] Cap aggregate export size before buffering:** `convex/httpApiV1/skillsV1.ts:1740-1741`
  - body: The handler fetches all selected file blobs, materializes each `arrayBuffer`, and then builds another ZIP buffer for up to 1000 skills with no total byte or file-count budget. A few large skills can exhaust the action's memory or time budget; reject or paginate by an aggregate size limit before reading blobs.
  - confidence: 0.84
