## Best Possible Solution

## Reproduction Assessment

## Solution Assessment

No; a bulk export API may be useful, but this diff is not the best path until the mirror policy, visibility gates, rate-limit behavior, archive layout, and size budget are settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Filter export rows through public moderation gates:** `convex/skills.ts:9639-9645`
  - body: `listByDateRange` pages `skillSearchDigest` by active/deleted date keys only. Hidden, removed, or malware-blocked rows can still be returned and then have their blobs exported, bypassing the public list/download moderation gates.
  - confidence: 0.94
- **[P2] Rate-limit export auth failures before token lookup:** `convex/httpApiV1/skillsV1.ts:1503-1507`
  - body: The handler authenticates first and returns missing/invalid bearer responses before `applyRateLimit`, so those attempts never consume the export/IP bucket even though current API docs say invalid tokens fall back to IP enforcement.
  - confidence: 0.86
- **[P2] Avoid colliding with skill-owned metadata files:** `convex/httpApiV1/skillsV1.ts:1689-1690`
  - body: The export always writes `${slug}/_meta.json`, but a valid published skill can already contain `_meta.json`. That duplicate path makes `buildMergedExportZip` throw and turns a valid export page into a 500.
  - confidence: 0.82
- **[P2] Bound export size before reading blobs:** `convex/httpApiV1/skillsV1.ts:1625`
  - body: The endpoint reads blobs for up to 1000 skills before zipping. Since valid skill bundles can be much larger than the assumed average, a normal export page can exhaust action memory or response limits without a byte budget.
  - confidence: 0.78

## Security Review
