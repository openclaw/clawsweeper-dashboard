## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter export rows through the public visibility policy:** `convex/skills.ts:10411-10413`
  - body: `listByDateRange` only constrains `skillSearchDigest` by `softDeletedAt`, so hidden/pending-scan/moderation-blocked rows with no soft-delete can be exported even though the public API goes through `toPublicSkill` and the docs tell mirrors not to bypass public filters. Reuse the current public visibility path, or make the privileged staff-export contract explicit before exposing this route.
  - confidence: 0.93
- **[P1] Page incremental exports by update time:** `convex/skills.ts:10408-10413`
  - body: The PR describes hourly incremental mirror sync, but the range scan is on `by_active_created`. Any new version or metadata update for an older skill keeps the old `createdAt`, so later hourly exports will miss it; use an updatedAt-based contract or document this as created-only full traversal with a separate update path.
  - confidence: 0.87
- **[P1] Bound archive size before reading every blob:** `convex/httpApiV1/skillsV1.ts:1693-1741`
  - body: The handler accepts up to 1000 skills, fetches every file blob, and converts each blob to an ArrayBuffer before zipping. A valid page with large or many files can exhaust action memory and fail the backend path, so enforce per-export byte/file limits before reading blobs or reduce the page contract.
  - confidence: 0.9
- **[P2] Apply export rate limiting before token auth:** `convex/httpApiV1/skillsV1.ts:1571-1574`
  - body: This handler performs token lookup and admin authorization before consuming the export rate limit. Invalid-token traffic can therefore force repeated auth queries without hitting the endpoint bucket; apply the export limiter first, matching the current API handler pattern.
  - confidence: 0.86
