Overall confidence: 0.9

Full review comments:

- **[P1] Filter exports through public visibility gates:** `convex/skills.ts:10411-10413`
  - body: `listByDateRange` pages raw `skillSearchDigest` rows from `by_active_created` and the handler later zips their blobs without applying `toPublicSkill`/public owner checks or the download moderation checks. That lets non-active moderation rows or `blocked.malware` rows enter the mirror archive, contrary to the documented public API boundary.
  - confidence: 0.91
- **[P2] Rate-limit export requests before auth lookup:** `convex/httpApiV1/skillsV1.ts:1571-1574`
  - body: The handler calls `requireExportAuth` before `applyRateLimit`, so missing or invalid credentials return or do token-hash/user lookups without consuming the export IP bucket. The existing API contract says invalid tokens fall back to IP enforcement, and this order makes the new endpoint easier to brute-force or DoS.
  - confidence: 0.88
- **[P2] Cap exported bytes before fetching and zipping blobs:** `convex/httpApiV1/skillsV1.ts:1693`
  - body: A single request can select 1000 skills and then fetch every latest-version blob into memory before `buildMergedExportZip`, but skill bundles may be up to 50 MB each. Add an explicit total byte/file budget and fail early instead of relying on average catalog size.
  - confidence: 0.86
- **[P2] Reserve generated export filenames:** `convex/httpApiV1/skillsV1.ts:1757-1759`
  - body: Changing the synthetic metadata file to `_export_skill_meta.json` only moves the collision: uploaded skills can still contain that valid path, and `buildMergedExportZip` throws on duplicates. Reserve or namespace generated files before adding them so one skill cannot 500 the whole export page.
  - confidence: 0.82
