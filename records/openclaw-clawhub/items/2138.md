## Summary

Keep open: this is a new bulk export feature, not something already implemented on current main, and the latest PR head still has blocking security/correctness issues plus missing real behavior proof.

## What This Changes

The PR adds an admin-only `GET /api/v1/skills/export` endpoint for paginated ZIP bulk skill exports, with route/schema wiring, export auth/rate limits, and ZIP helper code.

## Best Possible Solution

Define a product-approved mirror export contract that reuses public visibility and download moderation gates, rate-limits before auth, and lands with regression coverage, docs, and redacted real endpoint proof.

## Reproduction Assessment

Yes for the review findings by source inspection: the latest PR head uses `by_active_created` for export rows and authenticates before rate limiting, while current main blocks hidden, pending-scan, removed, and malware-blocked skills in public/download paths. I did not run the PR branch.

## Solution Assessment

No; the current patch is not the best solution because it creates a new bulk archive path without reusing existing public/download gates and leaves auth failures outside the rate limiter. A safer approach is to build the endpoint around the existing public registry contract and an explicit mirror/admin policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter export rows through public visibility gates:** `convex/skills.ts:9639`
  - body: The export query pages `skillSearchDigest` by `by_active_created`, which only constrains non-soft-deleted rows by date. Hidden, pending-scan, or `blocked.malware` skills can still reach the handler, which then reads version blobs and puts them in the ZIP, bypassing the public/download moderation gates. Filter with the same visibility rules before any blob fetch.
  - confidence: 0.93
- **[P2] Apply rate limiting before export auth:** `convex/httpApiV1/skillsV1.ts:1503-1506`
  - body: `requireExportAuth` returns 401/403 before `applyRateLimit` runs, so missing, invalid, or non-admin bearer attempts never consume the anonymous or user rate-limit buckets. Existing v1 handlers rate-limit first and pass those headers into auth failures; do the same here to avoid unthrottled token probing against this admin-only endpoint.
  - confidence: 0.86
