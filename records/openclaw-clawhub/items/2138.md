## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move the export route out of the slug namespace:** `convex/http.ts:77-79`
  - body: `/api/v1/skills/export` now takes precedence over the documented `/api/v1/skills/{slug}` detail route, but `export` is still a valid skill slug. This can make an existing or future `export` skill unreachable at its public detail URL; use a reserved collection route such as `/api/v1/skills/-/export` or add an explicit slug migration/reservation plan.
  - confidence: 0.87
- **[P1] Throttle failed export auth attempts:** `convex/httpApiV1/skillsV1.ts:1571-1574`
  - body: The handler runs `requireExportAuth` before `applyRateLimit`, so missing, invalid, and non-admin tokens return without consuming the export/IP bucket. That leaves the expensive token/auth path unthrottled for the new endpoint; apply an anonymous limit before auth or ensure all auth failures consume a rate-limit bucket.
  - confidence: 0.9
- **[P1] Keep export rows inside the public mirror boundary:** `convex/skills.ts:10411-10414`
  - body: The export query scans `by_active_created` with only `softDeletedAt` constrained, so active suspicious/moderated rows can enter the mirror archive even though the docs tell third-party catalogs not to mirror moderation-blocked content outside public API filters. Reuse the public list visibility semantics or an approved export-specific visibility gate.
  - confidence: 0.86
- **[P2] Page mirror sync by updated content, not creation time:** `convex/skills.ts:10408-10413`
  - body: The PR describes hourly incremental mirror sync, but this query only windows on `createdAt`. Any skill republished or edited after the mirror has already passed its creation timestamp will not be exported in later hourly windows; use an updated-time cursor or another change-log style source.
  - confidence: 0.88
- **[P1] Add an aggregate export budget before materializing blobs:** `convex/httpApiV1/skillsV1.ts:1740-1741`
  - body: Each request can select up to 1000 skills, fetch every storage blob, materialize each blob with `arrayBuffer()`, and then ZIP the whole page with no total byte or file-count cap. Given existing skill publish limits, one request can attempt far more memory than an action should own; enforce an aggregate budget before reading blobs.
  - confidence: 0.86
