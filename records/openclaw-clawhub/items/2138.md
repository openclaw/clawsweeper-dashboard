Full review comments:

- **[P1] Avoid shadowing skill detail for the export slug:** `convex/http.ts:77-79`
  - body: Registering `/api/v1/skills/export` inside the skill detail namespace makes a valid public slug `export` unreachable at the documented `GET /api/v1/skills/{slug}` route. Use a reserved collection path such as `/api/v1/skills/-/export`, another namespace, or an explicit slug migration/reservation with compatibility tests.
  - confidence: 0.93
- **[P1] Keep exports behind the public visibility gates:** `convex/skills.ts:10411-10420`
  - body: `listByDateRange` returns raw digest rows constrained only by the active-created index window. Current public reads run through `toPublicSkill`/visibility logic, so this can export hidden, pending-scan, blocked, removed, or ownerless content and its blobs.
  - confidence: 0.92
- **[P1] Rate-limit export requests before auth:** `convex/httpApiV1/skillsV1.ts:1571-1575`
  - body: The handler calls `requireExportAuth` before `applyRateLimit`, so missing, invalid, and non-admin tokens can repeatedly hit auth/token lookup paths without consuming the export bucket. Apply the export limit first, then auth, matching the other HTTP handlers.
  - confidence: 0.9
- **[P2] Use updated pagination for mirror sync:** `convex/skills.ts:10408-10413`
  - body: The endpoint is described as supporting hourly mirror sync, but the query pages `by_active_created`, so updates to existing skills after their creation date are never included. Use an updatedAt-based index/cursor or clearly make this create-only and test that behavior.
  - confidence: 0.86
- **[P2] Bound aggregate export size before building the ZIP:** `convex/httpApiV1/skillsV1.ts:1693`
  - body: The handler can fetch up to 1000 skills, read every blob into memory, and then build the final ZIP without an aggregate byte or file-count budget. A few large skills can exceed action memory or stall the endpoint; track totals and return a bounded error before continuing.
  - confidence: 0.84
