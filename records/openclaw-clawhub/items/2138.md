## Summary

Keep open: the latest PR head still introduces a security-sensitive bulk export path that bypasses current public/download moderation gates, and it needs maintainer product/security review plus contributor-provided real behavior proof before merge.

## What This Changes

The PR adds an admin-authenticated `GET /api/v1/skills/export` endpoint that pages skill digests by date and builds a merged ZIP archive with manifest metadata, plus route, auth, rate-limit, and schema route wiring.

## Best Possible Solution

Define an explicit mirror/export contract that reuses the same public visibility and download gates, preserves documented rate-limit behavior, and ships with maintainable tests and operator docs after maintainer approval.

## Reproduction Assessment

Yes for the review findings: source inspection of PR head shows the export query can select rows that current public visibility and download code reject, and the auth path exits before rate limiting. I did not run the PR branch.

## Solution Assessment

No; the current PR is not the safest maintainable solution because it creates a new bulk archive path without reusing existing moderation/download gates or the documented auth-rate-limit order. A safer implementation should first settle the mirror policy, then filter through the established public registry contract before reading blobs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Filter exports through public visibility gates:** `convex/skills.ts:9398`
  - body: The export query pages `skillSearchDigest` through `by_active_created`, which only constrains `softDeletedAt`. Hidden, pending-scan, removed, or `blocked.malware` rows can still be selected and then zipped, bypassing the public/download gates that currently block those skills. Filter through the same public visibility rules before fetching version blobs.
  - confidence: 0.94
- **[P2] Rate-limit unauthenticated export attempts:** `convex/httpApiV1/skillsV1.ts:1505`
  - body: `requireExportAuth` runs before `applyRateLimit`, so missing or invalid bearer tokens return before consuming the documented IP fallback bucket. Existing HTTP handlers rate-limit first, then authenticate with those headers; keep that order here so failed export auth cannot bypass throttling.
  - confidence: 0.86
- **[P2] Avoid per-skill metadata path collisions:** `convex/httpApiV1/skillsV1.ts:1692`
  - body: A valid published skill can include `_meta.json`; this handler first adds user files under `${slug}/...` and then appends `${slug}/_meta.json`, while `buildMergedExportZip` throws on duplicate paths. One such skill would turn the whole export page into a 500 instead of exporting the catalog. Use a reserved non-colliding metadata path or record this as a per-skill export error.
  - confidence: 0.78
