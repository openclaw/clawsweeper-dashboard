## Best Possible Solution

Define an approved partner export contract first, then implement it behind public visibility/download checks with update/tombstone pagination, a non-conflicting route namespace, throttled auth, aggregate caps, audit docs, tests, and redacted real-run proof.

## Reproduction Assessment

Not applicable as a user-bug reproduction because this is a feature PR. The blocking review paths are source-reproducible from the diff: export scans raw digest rows, routes an exact slug path, performs auth before rate limiting, and buffers all blobs without aggregate caps.

## Solution Assessment

No. A privileged bulk mirror endpoint may be useful, but this implementation is not the best path until maintainers approve the product/security contract and the code reuses public visibility rules with safer sync and resource limits.

Full review comments:

- **[P1] Reuse public visibility checks for export rows:** `convex/skills.ts:10411-10413`
  - body: The export query returns raw `skillSearchDigest` rows from `by_active_created` and the handler reads latest-version blobs directly, so hidden, removed, or malware-blocked rows can be packaged even though docs require mirrors to stay inside public API filters. Build export pages from the same public/download visibility helpers or an explicitly approved privileged boundary.
  - confidence: 0.94
- **[P1] Use update and tombstone pagination for mirror sync:** `convex/skills.ts:10408-10413`
  - body: Paging by `createdAt` cannot support the PR's hourly incremental mirror claim: republished, renamed, moderated, removed, or restored skills keep their original creation time, so a mirror advancing a creation-date window will miss changes. The export contract needs updated-at pagination plus deletion/moderation tombstones or another approved full-sync contract.
  - confidence: 0.91
- **[P1] Avoid taking over the export slug detail route:** `convex/http.ts:76-79`
  - body: Registering an exact `/api/v1/skills/export` route makes a readable skill whose slug is `export` hit the export auth flow instead of the documented `GET /api/v1/skills/{slug}` detail endpoint. Reserve and migrate that slug first or move bulk export under a non-slug namespace such as `/api/v1/skills/-/export`.
  - confidence: 0.9
- **[P2] Rate-limit failed export auth attempts:** `convex/httpApiV1/skillsV1.ts:1571-1574`
  - body: The handler performs bearer-token lookup and role checks before consuming the new export rate-limit bucket, so missing, invalid, and non-admin attempts can hammer token auth without hitting this endpoint's limits. Apply an IP bucket before auth and then the authenticated bucket after a valid token if needed.
  - confidence: 0.88
- **[P2] Cap bytes before buffering the ZIP:** `convex/httpApiV1/skillsV1.ts:1740-1741`
  - body: An authorized request can read up to 1000 skills' blobs, copy every blob into `zipEntries`, and then create a second copied ZIP buffer without an aggregate byte or file-count cap. Add total byte/file limits and fail before materializing more data than Convex can safely hold.
  - confidence: 0.86
