## Summary

## What This Changes

The PR adds an admin-only `GET /api/v1/skills/export` endpoint that returns paginated ZIP archives of skill files and metadata for mirror synchronization.

## Best Possible Solution

Ship a designed bulk export surface only after maintainers choose the public-mirror versus privileged-staff contract, then enforce that boundary with update-time pagination, failed-auth throttling, aggregate size limits, focused tests, docs, and real runtime proof.

No: adding the endpoint as-is is not the best path because the public-versus-privileged export boundary and capacity model are unresolved.   - body: The export query pages `skillSearchDigest.by_active_created` and the handler then reads internal version blobs directly, so a mirror export can include active rows that current public list/download paths would hide for moderation, malware, or suspicious-state reasons.   - confidence: 0.95
- **[P1] Rate-limit failed export authentication:** `convex/httpApiV1/skillsV1.ts:1571-1574`
  - body: `requireExportAuth` runs before `applyRateLimit`, so missing or invalid bearer tokens return before consuming the export IP bucket and before rate-limit headers are attached. Apply the export rate limit first, like the existing v1 handlers, so failed-auth traffic cannot bypass the new throttle.
  - confidence: 0.91
- **[P1] Bound export size before building the ZIP:** `convex/httpApiV1/skillsV1.ts:1693-1780`
  - body: With a limit of 1000 skills, the handler fetches every blob, converts each blob to memory, and then `zipSync`s the entire archive without an aggregate byte cap. Since publishing allows large skill bundles, a valid export request can exhaust memory or stall the action; add an explicit total-size/page-size guard or a smaller streaming/batched design.
  - confidence: 0.9
