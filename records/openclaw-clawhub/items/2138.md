## Summary

This PR should remain open because it introduces a new privileged bulk export API whose contract, visibility boundary, sync semantics, throttling, and resource limits still need maintainer approval and repair before merge. ## What This Changes

The branch adds an admin-only `GET /api/v1/skills/export` API route that paginates `skillSearchDigest` rows by creation date, fetches latest skill blobs, and returns a merged ZIP with manifest metadata.

## Best Possible Solution

Define the mirror export contract first, then implement the endpoint with approved visibility rules, update-aware cursor semantics, failed-auth throttling, bounded ZIP generation, public/operator docs, and regression coverage.

## Solution Assessment

No: adding a privileged bulk export endpoint is not ready as proposed because the mirror contract and safety model are not settled. The maintainable path is a narrow, documented export feed with approved visibility semantics, update-aware pagination, and explicit resource limits.

Full review comments:

- **[P1] Keep exports inside approved visibility rules:** `convex/skills.ts:10411-10413`
  - body: This query pages `skillSearchDigest` directly for the export feed without applying the public catalog visibility rules or an explicit maintainer-approved privileged export contract. A bulk mirror endpoint can expose hidden, suspicious, or moderation-sensitive artifacts at scale unless the export layer first defines and enforces exactly which skill states are eligible.
  - confidence: 0.86
- **[P1] Rate-limit failed export auth attempts:** `convex/httpApiV1/skillsV1.ts:1571-1574`
  - body: The handler returns from `requireExportAuth` before calling `applyRateLimit`, so missing or invalid Bearer tokens are never charged against the export limiter. That leaves the new privileged endpoint with no IP fallback throttle for failed-auth traffic, even though the PR describes export-specific rate limiting as part of the safety boundary.
  - confidence: 0.92
- **[P1] Use update-aware pagination for mirror sync:** `convex/skills.ts:10411-10413`
  - body: The export query paginates by `by_active_created`, which only tracks initial creation ordering. Hourly mirror syncs need a cursor over changes or updated/publication state, otherwise republishes, metadata edits, restores, moderation transitions, and deletes after creation can be missed while the mirror still believes it is current.
  - confidence: 0.88
- **[P2] Bound exported bytes before building the ZIP:** `convex/httpApiV1/skillsV1.ts:1693`
  - body: The handler allows up to 1000 skills and fetches their files before building one merged archive, but there is no aggregate byte or file-count cap before materializing those blobs and the ZIP in memory. A single valid admin export over unusually large skills can exhaust the action budget or degrade service availability.
  - confidence: 0.83
