## Summary

Keep open: this is a new security-sensitive bulk export feature, not something already implemented on current main, and the current branch still has blocking issues around public moderation gates, failed-auth rate limiting, and missing real behavior proof.

## What This Changes

The PR adds an admin-authenticated `GET /api/v1/skills/export` endpoint that pages skill digests by date and returns merged ZIP archives, plus route constants, auth/rate-limit helpers, and ZIP path validation.

## Best Possible Solution

Define a maintainer-approved bulk export contract that reuses public visibility and download moderation gates, rate-limits all attempts before auth, includes regression coverage/docs/spec intent, and is backed by redacted runtime proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows the export query selects by `softDeletedAt` only and the handler authenticates before rate limiting. I did not run the PR branch.

## Solution Assessment

No; the branch is not the safest maintainable solution yet because it creates a bulk archive path without reusing existing moderation/download gates and does rate limiting after auth. A safer path is to approve the mirror contract, filter through the public registry rules before blob reads, and rate-limit before token/role checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter exports through public skill visibility:** `convex/skills.ts:9639`
  - body: The export query pages `skillSearchDigest.by_active_created`, which only constrains `softDeletedAt`. Hidden, removed, pending-scan, or `blocked.malware` rows can still have a latest version, so this bulk ZIP path can archive bytes that the public list and download paths currently exclude. Reuse the existing public visibility/download moderation rules before fetching blobs.
  - confidence: 0.93
- **[P2] Rate-limit export requests before auth:** `convex/httpApiV1/skillsV1.ts:1503-1506`
  - body: `requireExportAuth` runs before `applyRateLimit`, so invalid or non-admin bearer requests can hit token lookup/touch logic and return 401/403 without consuming the export rate bucket. Match the existing v1 pattern by applying the rate limit first and passing those headers into auth failure responses.
  - confidence: 0.84
