## Summary

This PR should stay open for maintainer review because it adds a privileged bulk export API that is not on current main and still has blocking security-boundary, sync-correctness, route-compatibility, and availability concerns.

## What This Changes

The PR adds an admin-only `GET /api/v1/skills/export` endpoint that pages through skill digests by date and returns skill files plus metadata in a merged ZIP archive for mirror synchronization.

## Best Possible Solution

Maintainers should first approve a mirror-export contract, then implement it with explicit public/moderation filtering, update/tombstone sync semantics, a non-conflicting route or slug reservation, pre-auth throttling, aggregate caps, audit/spec documentation, and focused tests.

## Reproduction Assessment

Not applicable as a user bug. The merge blockers are source-reproducible from the PR diff: the export query, route placement, auth-before-rate-limit order, created-date cursoring, and archive buffering path are visible in the proposed code.

## Solution Assessment

No. A bulk mirror export may be useful, but this implementation should not merge until maintainers define the mirror contract and the code preserves public/moderation boundaries, update semantics, compatibility, throttling, and size limits.

Full review comments:

- **[P1] Filter exported skills through the approved public boundary:** `convex/skills.ts:10411-10421`
  - body: The export query returns rows from `skillSearchDigest.by_active_created` without applying the public catalog/download checks for removed, hidden, suspicious, scan-blocked, or otherwise non-public skills. Because this endpoint archives files in bulk for an external mirror, it can expose content current public paths intentionally suppress unless maintainers explicitly approve a different mirror boundary.
  - confidence: 0.91
- **[P1] Use update-aware sync semantics instead of created-date paging:** `convex/skills.ts:10407-10418`
  - body: Paging by `createdAt` only exports skills created in the requested range, so republished files, metadata edits, moderation changes, removals, renames, and tombstones for older skills will not reach a mirror doing incremental sync. That makes the new API unsafe as the advertised hourly mirror synchronization contract.
  - confidence: 0.88
- **[P1] Avoid colliding with the skill slug detail route:** `convex/http.ts:76-80`
  - body: Registering `/api/v1/skills/export` creates a static route inside the same namespace as `/api/v1/skills/{slug}`. Current main does not reserve the slug `export`, so this can shadow or make ambiguous the documented detail endpoint for a valid skill slug unless the slug is reserved, the route is moved, and route-collision coverage is added.
  - confidence: 0.84
- **[P1] Rate-limit failed export auth before token lookup:** `convex/httpApiV1/skillsV1.ts:1570-1575`
  - body: `exportSkillsV1Handler` calls `requireExportAuth` before `applyRateLimit`, so requests with invalid bearer tokens still exercise token parsing, hashing, and lookup before the export limiter can reject them. For a privileged bulk endpoint, failed-auth attempts need a cheap pre-auth throttle or equivalent guard before touching token state.
  - confidence: 0.82
- **[P1] Cap aggregate export size before building the ZIP:** `convex/httpApiV1/skillsV1.ts:1643-1693`
  - body: The handler can collect up to 1000 skills worth of blobs and `buildMergedExportZip` creates the archive in memory, but there is no aggregate byte, file-count, or response-size cap. One authorized request can therefore create an oversized Convex action response or memory spike even if per-file reads are valid.
  - confidence: 0.86
