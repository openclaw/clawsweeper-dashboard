## Summary

Keep open. This is a ClawHub issue, and the cleanup policy for this pass does not auto-close issues; independently, current main still has a source-reproducible path where an ordinary update to an existing active skill can rewrite the skill row to hidden pending scan, after which public detail, search, catalog, download, and API paths filter it out.

## What This Changes

The issue reports that updating an already-published skill returns success or a `skills:publishVersion` server error while the skill disappears from inspect, search, and public listing surfaces.

## Best Possible Solution

Implement version-aware public continuity: keep the last approved skill card, search digest, API metadata, downloadable/file surface, and searchable embedding visible while a new version remains scanner/moderation-gated, then promote the new version only after those gates clear.

## Reproduction Assessment

Yes, at source level. An ordinary non-trusted publisher updating an existing active skill can reach `insertVersion`, have the skill row patched to hidden pending scan, and then be filtered out by public detail, search, catalog, API, and download paths; I did not run a live Convex publish in this read-only review.

## Solution Assessment

No patch is under review, and the old direct preserve-active-status approach is not the best fix because it can bypass pending-scan visibility gates. The safer solution is a last-approved public projection while the new version remains hidden until scanner and moderation finalization.
