## Summary

Keep open. Current main still has a source-reproducible path where an ordinary non-trusted update to an existing active skill writes hidden pending-scan state onto the skill row, and public detail/search/API paths then filter that row out; the related reports at https://github.com/openclaw/clawhub/issues/131 and https://github.com/openclaw/clawhub/issues/382 remain open.

## What This Changes

The issue reports that publishing a new version of an existing skill can return success or a server error while making the skill disappear from inspect, search, and public surfaces.

## Best Possible Solution

Implement version-aware public continuity: keep the last approved skill card, digest, searchable embedding, API metadata, and public file surface visible while the new upload remains scanner/moderation-gated, then promote the new version only after the gates clear.

## Reproduction Assessment

Yes, at source level. An ordinary non-trusted publisher updating an existing active skill reaches `insertVersion`, can write hidden pending-scan state to the skill row, and public conversion/search/detail/API paths then filter it out; I did not run a live Convex publish in this read-only review.

## Solution Assessment

No patch is under review, and the direct preserve-active approach is not the best fix because it was closed as a moderation bypass. The safer solution is last-approved-version continuity while the new version remains gated until scanner/moderation finalization.
