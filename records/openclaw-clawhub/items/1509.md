## Summary

Keep open. ClawHub cleanup policy keeps issues open, and although the broad-ignore manifest problem was fixed and released in `clawhub@0.12.3`, current main still has a source-reproducible path where `clawhub skill publish .` can resolve against a configured global workdir instead of the caller's skill folder; the related open report at https://github.com/openclaw/clawhub/issues/48 should be handled by the same CLI fix.

## What This Changes

The issue reports that `clawhub publish .` fails with `SKILL.md required` for a new skill folder even though `SKILL.md` exists and login succeeds.

## Best Possible Solution

Skill publish should treat user-entered relative source paths as cwd-relative when that resolves to the intended skill folder, while preserving absolute paths and configured-workdir behavior for non-publish workspace flows.

## Reproduction Assessment

Yes, from source inspection. Configure a Clawdbot/OpenClaw default workspace that differs from an unmarked cwd containing `SKILL.md`, then run `clawhub skill publish .`; current code can scan the configured workspace and hit `SKILL.md required`.

## Solution Assessment

Yes. The narrow maintainable repair is publish source-path resolution plus regression coverage; backend validation and the already-shipped root-manifest recovery should not be broadened for the remaining path.
