## Summary

Keep open. The broad-ignore manifest omission was fixed on main and released, but current main still has a source-reproducible `publish .` path where a configured global workdir can make the CLI inspect the wrong folder and then throw `SKILL.md required`; ClawHub cleanup policy also keeps issues open.

## What This Changes

The issue requests a fix for `clawhub publish .` failing with `SKILL.md required` when publishing a new skill whose manifest exists.

## Best Possible Solution

Skill publish should resolve user-entered relative source paths to the caller's intended skill folder when that folder contains `SKILL.md` or `SKILLS.md`, while preserving absolute paths and configured-workdir behavior for install, update, and sync workflows.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run `clawhub skill publish .` from a skill directory while the resolved global workdir points elsewhere, and publish can inspect the wrong folder before throwing `SKILL.md required`.

## Solution Assessment

Yes. The narrow maintainable fix is publish source-path resolution plus regression coverage, not backend validation changes or a broad rewrite of global workdir behavior.

## Repair Work Prompt

Fix the ClawHub CLI publish path-resolution bug tracked by https://github.com/openclaw/clawhub/issues/1509 and https://github.com/openclaw/clawhub/issues/48. In `packages/clawhub/src/cli/commands/publish.ts`, make `clawhub skill publish <relative-path>` resolve to the caller's intended local skill folder when that path exists from `process.cwd()` and contains `SKILL.md` or `SKILLS.md`, especially `publish .`, while preserving absolute paths, explicit `--workdir` behavior where intentional, and existing install/update/sync configured-workdir behavior. Add focused regression coverage in `packages/clawhub/src/cli/commands/publish.test.ts` for a cwd skill folder with a different configured/global workdir. Do not change backend `convex/lib/skillPublish.ts` validation or broaden ignored-file recovery beyond the existing root manifest behavior. Update specs/docs only if the intended precedence rule needs to be durable.
