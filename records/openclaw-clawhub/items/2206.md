## Summary

Keep open: the relevance and count-honesty direction is useful and backed by tests plus inspected UI screenshots, but the proposed pagination cap handling can leave Load More active after no more rows can be displayed.

## What This Changes

The PR tightens skill/package/plugin search eligibility and ranking, adds limit-plus-one count handling with grouped search UI/typeahead updates, and documents the search relevance contract.

## Best Possible Solution

Land the search relevance contract after making capped pagination stop implying additional rows beyond the supported backend window.

## Reproduction Assessment

Yes for the blocking review finding from source inspection: a query with more than 100 skill matches keeps `skillHasMore` true while the hook clamps the effective limit to 100, so later Load More clicks cannot add rows.

## Solution Assessment

No, not until the pagination cap is handled consistently. The narrow fix is to make `hasMore`/`canLoadMore` respect the backend maximum and cover that with hook or route tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Stop reporting hasMore after the search limit is capped:** `src/lib/useUnifiedSearch.ts:147-148`
  - body: When `/search` has loaded up to 100 results, this hook clamps `skillLimit`/`pluginLimit` to 100 but still requests one extra row and sets `skillHasMore` true whenever 101 skills are returned. The route can keep rendering Load More after each click even though the capped request is identical and no additional rows can appear. Gate `hasMore` on whether the effective limit can still increase, or hide Load More at the cap.
