## Summary

Keep open. Current main still has a source-reproducible GitHub import-to-publish path that can emit the reported rate-limit error, and the related publish rate-limit reports remain open pending production log/env triage.

## What This Changes

Publishing a skill from GitHub import should not remain stuck behind a GitHub API rate-limit failure that waiting does not resolve.

## Best Possible Solution

Verify production Convex failures and GITHUB_TOKEN configuration first, then either fix deployment configuration or refine the account-age lookup, caching, and error details without weakening the 14-day gate.

## Reproduction Assessment

Yes, as source-reproducible rather than live-reproduced: a user missing githubCreatedAt who imports from GitHub will hit publishVersionForUser, and a GitHub 403/429 from the account-age lookup produces the reported error. I did not have production Convex logs to prove the live trigger.

## Solution Assessment

Unclear. Current main already has GITHUB_TOKEN mitigation and tests for the behavior, so the narrowest fix depends on maintainer verification of production env/logs before choosing deployment configuration, caching, or error-handling changes.
