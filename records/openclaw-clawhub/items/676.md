Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-reproducible GitHub import-to-publish path that can surface the reported GitHub API rate-limit message when the account-age lookup receives a 403 or 429.

## What This Changes

Publishing a skill through GitHub import fails for affected users with a wrapped GitHub API rate-limit error that retrying does not resolve.

## Best Possible Solution

Keep this in the publish/import rate-limit cluster, verify production GITHUB_TOKEN and Convex failures first, then correct configuration or reduce uncached GitHub lookups with clearer retry detail while preserving the abuse gate.

Yes, source-level. On current main, GitHub import publishes through publishVersionForUser; for a user missing cached githubCreatedAt, a GitHub 403 or 429 in requireGitHubAccountAge produces the reported wrapped error.

## Solution Assessment

Unclear. The narrowest safe solution depends on production logs and quota state: it may be a Convex GITHUB_TOKEN deployment fix, or a code change to reduce uncached lookups and improve retry detail without bypassing the account-age gate.
