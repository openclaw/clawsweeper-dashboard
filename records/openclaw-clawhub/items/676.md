ClawHub cleanup policy keeps issues open, and current main still has a reachable import-to-publish path that emits the reported GitHub API rate-limit message when the account-age lookup receives a GitHub 403 or 429. ## What This Changes

Publishing a skill through GitHub import fails with a GitHub API rate-limit message that retrying does not resolve.

## Best Possible Solution

Keep this in the publish/import rate-limit cluster, verify production GITHUB_TOKEN and Convex failures first, then either correct deployment configuration or refine the GitHub account lookup so true rate limits are actionable and uncached first-publish lookups are reduced without weakening the abuse gate.

I did not run a live production reproduction because the missing data is the production GitHub quota/env state.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on production evidence: setting the existing GITHUB_TOKEN may solve an ops quota problem, while a code mitigation would need careful caching/backfill or retry detail without weakening the anti-abuse gate.
