## Summary

Current main still marks the policy Draft and leaves the central claim, dispute, deleted-namespace, recovery, and evidence-handling decisions unresolved; the existing fail-closed recovery and admin migration primitives do not replace the requested maintainer product/security decision.

## What This Changes

Define transparent policy for organization or owner namespace claims and disputes, deleted-owner namespace handling, and recovery after owner-initiated account deletion.

## Best Possible Solution

Approve one coherent set of ownership, dispute, deleted-namespace, recovery, audit, and confidentiality invariants in specs; publish safe user-facing guarantees in docs; keep sensitive verification steps in restricted reviewer guidance; then open narrow implementation follow-ups for any missing admin operations.

## Reproduction Assessment

Not applicable: this is an explicit policy RFC rather than a single failing execution path; the linked Mapbox, ClawSec, account-recovery, and deleted-handle reports provide concrete examples of the unresolved boundary.

## Solution Assessment

Yes in structure: one RFC followed by durable specs, public docs, and narrow implementation tasks is safer and more maintainable than embedding case-by-case policy inside admin operations, but maintainers still must choose the substantive rules.
