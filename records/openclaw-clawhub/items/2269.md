## Summary

Keep open. Current main and the latest release still count skill installs only through authenticated `clawhub sync` telemetry, while direct `clawhub install` downloads the skill and writes local state without reporting an install event; changing that is a product/privacy decision rather than conservative cleanup.

## What This Changes

The issue asks for direct `clawhub install` reporting so `installsAllTime` and `installsCurrent` reflect installed skills without requiring a later sync.

## Best Possible Solution

Maintainers should decide whether install stats remain sync-only telemetry or gain immediate install-event reporting, then update CLI/backend semantics, privacy docs/specs, and regression coverage together.

## Reproduction Assessment

Yes, source-level. Direct `cmdInstall` does not submit install telemetry, while authenticated `clawhub sync` is the current path that posts root snapshots used for install counters.

## Solution Assessment

No for an automatic direct-POST change. The maintainable path needs an explicit decision on privacy, opt-out, dedupe, auth, uninstall, and `installsCurrent` semantics before code changes.

## Risks / Open Questions

- Immediate install reporting could double-count users who later run `clawhub sync` unless both paths share a clear dedupe identity and event model.
- A one-shot install POST does not by itself define uninstall, reactivation, stale-root, update, or `installsCurrent` semantics.
- Direct `cmdInstall` currently works with optional auth; counting unique installs may require either logged-in-only reporting or a new anonymous/device identity policy.
- Changing install-time reporting would alter the documented sync-time telemetry and opt-out boundary unless maintainers explicitly choose that behavior.
