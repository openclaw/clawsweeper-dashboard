## Summary

Keep open. Current main still documents and implements skill install counters as authenticated `clawhub sync` telemetry, while direct `clawhub install` resolves, downloads, writes metadata, and updates the lockfile without posting an install event; the live example still shows non-zero downloads with zero install counters. The related stats discussion at https://github.com/openclaw/clawhub/issues/1712 closed package-stat work but explicitly treated direct skill install reporting as a separate telemetry/privacy decision.

## What This Changes

The issue asks for `clawhub install` to report install events so `installsAllTime` and `installsCurrent` update without requiring a later `clawhub sync`.

## Best Possible Solution

Maintainers should choose the install telemetry model first; if direct install reporting is adopted, it should share the sync identity, dedupe, opt-out, deletion, and current-install semantics, then update CLI, backend, docs/specs, and regression tests together.

## Reproduction Assessment

Yes. Source inspection shows direct `cmdInstall` has no install-telemetry POST while `cmdSync` is the path that posts telemetry snapshots, and the live example API still returns non-zero downloads with zero install counters.

## Solution Assessment

No for an automatic direct POST. The current implementation deliberately documents sync-time telemetry, so changing the install path needs a maintainer decision on privacy, deduplication, and current-install semantics.
