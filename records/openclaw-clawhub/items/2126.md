## Summary

Keep this PR open: it adds the requested plugin discovery surface, but the package API category validation currently rejects the documented family-scoped package list/search paths, and the contributor proof still does not demonstrate category-filtered behavior.

## What This Changes

The PR adds plugin category definitions, `/plugins` category and sort controls, plugin API category filters, denormalized category digest rows with a backfill, and related docs/tests/spec/changelog updates.

## Best Possible Solution

Fix the package endpoint validation for plugin-family-scoped category requests, keep the indexed category approach, and add proof showing selected category or API-filtered behavior after maintainer product review.

## Reproduction Assessment

Yes for the PR finding: source inspection shows `/api/v1/packages?family=code-plugin&category=data` and the matching search route are rejected because `includeSkills` is forced true for shared package handlers. I did not run the endpoint, but the rejection path is clear from the changed source.

## Solution Assessment

No; the indexed category design is plausible, but the current validation blocks a documented API path and the proof does not show category-filtered behavior. The narrower fix is to reject only truly mixed or skill-scoped category requests while allowing plugin-family-scoped package endpoints.

## Review Findings
