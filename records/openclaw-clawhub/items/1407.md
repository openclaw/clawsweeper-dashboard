## Summary

Keep open. The live ClawHub endpoints still reproduce the stale SKILL.md mismatch, and current main’s read path appears to serve the latest version row’s stored blob, so the remaining investigation belongs at the publish/upload and production-data boundary rather than cleanup closure.

## What This Changes

The issue reports that publishing new versions of `auth0-token-vault` does not update the SKILL.md content shown on the skill page or file API.

## Best Possible Solution

Trace the publish/upload payloads and production `skillVersions`/storage rows for the affected versions, repair bad rows if submitted bytes were correct, and add regression coverage for changed SKILL.md hashes across successive publishes.

## Reproduction Assessment

Yes. The live public API currently reproduces the mismatch: latest metadata is `0.4.0`, but later version rows still expose the same SKILL.md sha/size as the initial publish and the served file remains Gmail-focused.

## Solution Assessment

Unclear. A cache or page-read fix is not the best target based on current evidence; current main resolves the latest version row and serves that row’s stored blob, so the maintainable fix should start by tracing the publish/upload boundary.
