## Summary

Keep open. ClawHub cleanup policy keeps issues open, and the live public endpoints plus a maintainer production-data check show the latest version is selected correctly but the stored SKILL.md bytes are still stale across later publishes, so the remaining work is at the publish/upload boundary rather than issue cleanup closure.

## What This Changes

The issue requests that skill detail pages and file APIs show the SKILL.md content from the latest successfully published skill version.

## Best Possible Solution

Add publish/upload boundary instrumentation or validation that records the SKILL.md hash seen by the client and server for each publish, repair the affected production rows if the submitted bytes were wrong, and add a regression test that repeated publishes with changed SKILL.md content produce distinct stored hashes and rendered content.

## Reproduction Assessment

Yes. The public API gives a high-confidence reproduction: compare the versions endpoint, version-detail SKILL.md hashes for 0.2.1/0.3.0/0.4.0, and the 0.4.0 file content, which still lacks the expected later command sections.

## Solution Assessment

Unclear. A cache/read-path fix is not the best target now because current main and production evidence show the latest version row is being read; the maintainable fix should start at the upload/publish payload boundary and only then patch code or data at the confirmed failure point.
