## Summary

Keep open. ClawHub cleanup policy keeps issues open, and the live public endpoints still reproduce the stale SKILL.md mismatch even though current main's read path selects the latest version row and serves that row's stored file bytes.

## What This Changes

The issue asks ClawHub to render the SKILL.md content from the latest published skill version after repeated publishes and delete/recreate cycles.

## Best Possible Solution

Trace the publish/upload boundary with submitted SKILL.md hashes on both client and server, repair the affected production rows if the submitted bytes were correct, and add a regression that repeated publishes with changed SKILL.md content produce distinct stored hashes and rendered content.

## Reproduction Assessment

Yes. The live public API reproduces the issue: latest metadata is 0.4.0, but version details for later publishes still point at the same SKILL.md sha/size and the served 0.4.0 file remains Gmail-only beyond the frontmatter summary.

## Solution Assessment

Unclear. A cache/read-path fix is not the best target based on current evidence; the current read path serves the latest version's stored blob, so the maintainable fix should start by tracing the publish/upload payload and storage boundary.
