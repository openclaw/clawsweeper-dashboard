## Summary

Current main has complete star/download event recording, deduplication, canonical counter reads, and scheduled aggregation, but the visible skill document counters are intentionally synchronized only every six hours and no controlled production check proves the original or later freeze report is resolved. The closed cadence-only PR was not merged, so this issue remains useful as the canonical place to verify whether ingestion, backlog draining, or delayed display is still failing.

## What This Changes

The issue reports that starring skills and downloading them through the CLI or website does not increment the displayed star and download metrics.

## Best Possible Solution

Keep one canonical metric-pipeline issue, perform a controlled production star plus CLI and web download against a test skill, verify the dedupe and `skillStatEvents` records and both aggregation cursors, then either repair the failing stage or document the six-hour display contract and close with concrete runtime proof.

## Reproduction Assessment

No, we do not yet have a high-confidence failing current-main reproduction. The source provides a precise controlled path to test, but neither this report nor the later freeze report includes current production event, cursor, or counter observations.

## Solution Assessment

No, the closed proposal to shorten one cron interval is not established as the best fix. The narrow maintainable approach is to identify whether ingestion, deduplication, queue draining, or the intentional six-hour skill-document sync causes the observed result before changing cadence.
