## Label Justifications

- `P2`: Incorrect or excessively delayed public engagement counters reduce product trust but do not make ClawHub unavailable.
- `impact:data-loss`: Download events may be silently absent from canonical public totals when processing is delayed or stalled.

## Summary

Current main fixes the star side with synchronous counter updates, but downloads still rely on an intentional six-hour skill-document sync and there is no live evidence that production consistently drains queued events within that bound. ClawHub policy therefore requires keeping the combined report open for a freshness decision and production verification.

## What This Changes

Investigate skill star and download actions that fail to update their public counters promptly or at all.

## Best Possible Solution

Keep atomic star updates, define a user-visible download freshness SLO, and add production observability or alerting proving that queued download events update canonical skill counters within that bound.

## Reproduction Assessment

No, not at high confidence. Source proves downloads may remain invisible for up to six hours, but there is no timestamped current-production download and counter check or scheduler/backlog diagnostic showing failure beyond that bound.

## Solution Assessment

Partly. Synchronous star updates are the narrow maintainable fix for stars, but downloads require a maintainer-approved freshness target and real Convex validation before deciding whether cadence, batching, retries, or only monitoring should change.
