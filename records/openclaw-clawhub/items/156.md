## Summary

Keep open. Current main still records download and star events, but public counters are materialized through the delayed skillStatEvents sync path and the remaining question depends on production event backlog/logs plus a maintainer decision about counter freshness.

## What This Changes

The issue reports that starring skills and downloading skills from the CLI or website do not update the visible skill metrics.

## Best Possible Solution

Verify production skillStatEvents health first, then either repair stuck processing or define and implement a deliberate counter freshness model that preserves dedupe and avoids hot skill-document contention.

## Reproduction Assessment

No. Source inspection gives a clear stale-counter path through the delayed skill document sync, but I did not establish a current-main deployment where counters never update after the expected sync window.

## Solution Assessment

No for a cadence-only change. The maintainable solution is to inspect production health and backlog first, then adjust processing or freshness deliberately instead of blindly increasing cron frequency.
