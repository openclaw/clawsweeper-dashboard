## Summary

Keep open. Current main records download and star events, but visible counters still depend on asynchronous stat processing and materialized skill fields; without production queue/log evidence this is not a safe auto-close or repair, and ClawHub cleanup policy keeps issues open.

## What This Changes

The issue reports that starring skills and downloading them from the CLI or ClawHub website does not update visible download/star metrics.

## Best Possible Solution

Confirm production event queue and cron health, then either repair stuck event processing or choose a documented counter freshness model that avoids hot skill-document contention while making UI/API behavior understandable.

## Reproduction Assessment

No, not for the full “not updating” symptom. Source inspection gives a clear delayed-counter path after download/star events, but proving total failure needs production Convex logs, queue state, or an end-to-end run against the deployed service.

## Solution Assessment

Unclear. A blind cron-interval reduction is not the best fix because https://github.com/openclaw/clawhub/pull/176 was closed as not addressing the root cause; the safer path is to diagnose production processing first and then decide the freshness/performance tradeoff.
