## Summary

Keep open. Current main appears to have fixed the monorepo `--source-path` failure, but the remaining org-namespace behavior is an explicit product/security choice because publishing under an organization is currently driven by `--owner` plus backend membership checks, not inferred from package scope or repository owner.

## What This Changes

The issue requests that `clawhub package publish` support monorepo subdirectory publishing under an organization namespace without falling back to the authenticated user's personal namespace.

## Best Possible Solution

Keep the fixed GitHub subdirectory handling, then either document `--owner <org>` as the supported org-publish contract or design authorized org inference with explicit backend checks and regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection and existing tests show the original `--source-path` failure is fixed, while the remaining org-inference gap is visible because `ownerHandle` is only populated from `--owner`.

## Solution Assessment

Is this the best way to solve the issue? Unclear for automatic org inference: keeping `--owner` explicit is the narrowest current contract, while inference from package scope or repo metadata needs maintainer product and security approval.
