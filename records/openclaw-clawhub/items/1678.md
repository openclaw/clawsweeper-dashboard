## Summary

Keep open. Current main fixes the monorepo GitHub `--source-path` failure, but the org namespace request remains a product/security decision because publishing under an org is still explicit through `--owner`/`ownerHandle` rather than inferred from package scope or source repo.

## What This Changes

The issue asks package publish to support monorepo subdirectory publishing under an org namespace and to infer org ownership from scoped package/source metadata instead of defaulting to the authenticated personal owner.

## Best Possible Solution

Keep the shipped `--source-path` fix and have maintainers choose the owner contract: either document explicit `--owner <org>` as the supported path or implement authorized owner inference with backend membership validation and tests.

## Reproduction Assessment

Yes. The original GitHub shorthand subdirectory failure is covered by a focused regression test on current main, and code inspection shows owner inference still does not happen because `ownerHandle` only comes from the explicit CLI option.

## Solution Assessment

Unclear for the remaining ownership behavior. The source-path fix is the narrow maintainable fix for the monorepo bug, but owner inference needs maintainer product/security judgment before it becomes a safe implementation target.
