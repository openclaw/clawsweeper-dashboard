## Summary

Keep open. Current main has version soft-delete data and public read filtering, but it still lacks an owner-facing CLI or HTTP API workflow to delist, restore, or deprecate one published skill version; ClawHub cleanup policy also keeps issues open unless they are already resolved by a merged PR.

## What This Changes

Add owner-facing CLI/API support to deprecate, hide, or delist a single published skill version without deleting the whole skill.

## Best Possible Solution

Add owner/staff version-level delist and restore endpoints plus CLI commands backed by skillVersions.softDeletedAt, preserving audit history and latest/tag consistency while leaving separate deprecation semantics for a later decision.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence because current main exposes version reads and whole-skill lifecycle writes, but no owner-facing version-level write command or route.

## Solution Assessment

Partially yes: reusing the existing softDeletedAt model is the narrowest maintainable path for delist/hide/restore. Explicit deprecation should be kept separate unless maintainers define distinct semantics.
