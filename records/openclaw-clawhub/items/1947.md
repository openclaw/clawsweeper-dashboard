## Summary

Keep open. This is a valid feature gap also tracked by open #905: current main can represent and hide soft-deleted individual versions internally, but it still lacks an owner-facing CLI, HTTP API, UI, or docs workflow for delisting one published version; ClawHub policy keeps issues open for maintainer follow-up.

## What This Changes

Request an owner-facing command or API workflow to deprecate, hide, or delist a single published skill version without deleting the whole skill history.

## Best Possible Solution

Define a version-scoped lifecycle policy, then expose authenticated owner/org-member/staff API, CLI, UI, and docs that soft-delete or restore eligible versions while preserving audit logs and keeping public lists, detail, download, latest, and tag behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug. Current-main verification is high confidence: CLI commands only accept skill slugs, the DELETE router rejects version paths, and docs expose read-only version endpoints plus whole-skill delete/undelete.

## Solution Assessment

Unclear until maintainer policy is set. The requested command is directionally right, but the maintainable solution should be an explicit version lifecycle API/workflow rather than a thin CLI wrapper around the current staff-only internal mutation.
