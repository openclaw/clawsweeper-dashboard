## Summary

Keep open. Current main has whole-skill hide/delete and public per-version filtering for already soft-deleted versions, but it does not expose an owner-facing CLI/API path to delist, hide, restore, or deprecate one published version; ClawHub policy also keeps issues open during cleanup review.

## What This Changes

Request a CLI-supported way for owners to deprecate, hide, or delist one published version without removing the rest of the skill history.

## Best Possible Solution

Add an owner/staff skill-version delist and restore path that reuses `skillVersions.softDeletedAt`, keeps `latest` and tag pointers valid, documents the CLI/API, and leaves broader package-release lifecycle work to a separate scoped issue.

## Reproduction Assessment

Not applicable as a runtime bug, but the feature gap is high-confidence: current main exposes version reads and whole-skill lifecycle writes, with no version-level write command or route.

## Solution Assessment

Yes for delist/hide: using the existing version soft-delete model is the narrowest maintainable path. Per-version deprecation should be added only if maintainers choose explicit semantics beyond hiding from public resolution/downloads.

## Repair Work Prompt

Implement owner-facing skill version delist/restore support for ClawHub. Use the existing `skillVersions.softDeletedAt` semantics; add authenticated API routes for a skill owner or staff member to delist and restore a specific version by slug plus semver, update `latestVersionId`, `tags`, `latestVersionSummary`, capability tags, embeddings/search visibility, and audit logs safely when the affected version is currently tagged/latest.
