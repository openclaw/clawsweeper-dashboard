## Summary

Keep open. The request is valid and current main still lacks the owner-facing version deprecate/delist command or public API path; the code only has whole-skill delete/undelete plus a staff-only maintenance mutation for historical version soft-deletes.

## What This Changes

The issue requests a command to deprecate, hide, or delist one published version without deleting the rest of the skill/package history.

## Best Possible Solution

Define a single version lifecycle model, then expose audited owner/admin API and CLI support for version deprecation or delisting with clear latest/tag behavior, tests, and docs, coordinated with #905 and package-release lifecycle work in #1370.

## Reproduction Assessment

Not applicable as a feature request, but current-main source inspection gives a high-confidence gap check: the CLI and HTTP routes expose whole-skill deletion plus staff-only maintenance version soft-delete, not an owner-facing version command.

## Solution Assessment

Unclear. The proposed command direction is reasonable, but the best maintainable fix needs a maintainer decision on lifecycle semantics before adding a public CLI/API contract.
