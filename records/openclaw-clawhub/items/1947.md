## Summary

Keep open. Current main has internal version soft-delete plumbing and public read paths honor hidden versions, but there is still no publisher-facing CLI/API/docs workflow for delisting or deprecating a single skill version; related #905 remains the broader open tracker for the same lifecycle gap.

## What This Changes

The issue requests an owner-facing CLI/API workflow to mark one published skill version deprecated, hidden, or delisted without deleting the whole skill.

## Best Possible Solution

Triage this with #905, define the version lifecycle policy, then add authenticated owner/staff version-scoped CLI, HTTP API, docs, tests, and audit-preserving restore behavior consistently.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that internal version soft-delete exists but no owner-facing command/API/docs route is exposed.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers settle product policy. A version-scoped CLI/API matches the request directionally, but it should be specified together with #905 so authorization, latest-version, restore, and audit semantics are consistent.
