## Summary

This issue should stay open under the ClawHub cleanup policy, and the underlying feature gap still exists on current main: version rows can be soft-deleted internally and public reads respect that state, but there is no owner-facing CLI or v1 API route to delist a single published version.

## What This Changes

The issue requests an owner-facing command or API workflow to deprecate, hide, or delist one published skill version without deleting the whole skill.

## Best Possible Solution

Add authenticated owner/staff per-version delist and restore around skillVersions.softDeletedAt, with audit logging, clear latest-version behavior, CLI/API docs, and separate product handling for visible deprecation semantics.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence from source inspection: current main has internal version soft-delete state and public filtering, but no owner-facing CLI or HTTP write route for one version.

## Solution Assessment

Yes for delist/hide: reusing the existing soft-delete model is the narrowest maintainable implementation. A visible deprecation warning should be modeled separately only if maintainers want that user-facing status.
