## Summary

Keep open. Current main has the data model and read-side filtering for soft-deleted skill versions, but it still lacks an owner-facing CLI or HTTP API path to delist, hide, restore, or deprecate a single published version; ClawHub cleanup policy keeps issues open unless a PR is certainly implemented on main.

## What This Changes

Request an owner-facing command/API to deprecate, hide, or delist one published skill version without deleting the whole skill history.

## Best Possible Solution

Add owner/staff per-version delist and restore around skillVersions.softDeletedAt, keep latest/tag pointers consistent, and defer explicit deprecation semantics until maintainers define them.

## Reproduction Assessment

Not applicable as a runtime bug, but the feature gap is high-confidence. Current main exposes version reads and whole-skill lifecycle writes, with no owner-facing version-level write route or CLI command.

## Solution Assessment

Yes for delist/hide: reusing the existing softDeletedAt model is the narrowest maintainable path. Explicit per-version deprecation should be a separate design decision if it means more than hiding a version from public resolution/downloads.
