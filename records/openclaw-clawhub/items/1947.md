## Summary

Keep open. Current main already has version-level soft-delete state and public readers/downloads honor it, but there is still no owner-facing CLI, HTTP API, docs, or UI workflow to delist one published skill version; related #905 tracks the same product gap and ClawHub policy keeps issues open for maintainer handling.

## What This Changes

The issue requests a CLI command or equivalent API to deprecate, hide, or delist one published version without affecting the rest of the skill history.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update `skillVersions.softDeletedAt`, write version-specific audit logs, and keep public lists, detail/file/download routes, tags/latest behavior, docs, and restore semantics consistent.

## Reproduction Assessment

Not applicable as a runtime bug, but current-main verification is high confidence: the CLI only accepts skill slugs, the DELETE router rejects version paths, and docs list only read-only skill version endpoints plus whole-skill delete/undelete.

## Solution Assessment

Unclear until maintainer policy is set. The requested owner-facing workflow is directionally right, but the maintainable solution should not be a thin wrapper around the internal staff mutation without explicit latest, tag, audit, restore, and cache rules.
