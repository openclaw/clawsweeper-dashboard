## Summary

Keep open. Current main can represent and hide soft-deleted skill versions internally, but the owner-facing CLI, HTTP API, and docs still only expose whole-skill deletion/restoration; related #905 remains an open canonical companion for the same single-version lifecycle gap.

## What This Changes

The issue requests an owner-facing command or API to deprecate, delist, hide, or delete one published skill version without deleting the whole skill history.

## Best Possible Solution

Add authenticated owner/staff per-version delist and restore using `skillVersions.softDeletedAt`, with audit logs and clear latest/tag behavior; reserve visible deprecation status for a separate product decision.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence from source inspection: version endpoints are read-only, while lifecycle writes and CLI commands are whole-skill scoped.

## Solution Assessment

Yes for delist/hide: reusing the existing version `softDeletedAt` model is the narrowest maintainable solution. Visible deprecation should be a separate badge/status only if maintainers want a user-facing warning rather than hiding.

## Repair Work Prompt

Implement owner-facing per-version delist/hide and restore for published skills. Reuse `skillVersions.softDeletedAt`; add Convex mutation(s) with owner/org collaborator/staff authorization, audit logging, and a clear policy for latest-version/tag behavior, either refusing latest with guidance or atomically retargeting per maintainer preference. Add v1 HTTP routes such as `DELETE /api/v1/skills/{slug}/versions/{version}` and `POST /api/v1/skills/{slug}/versions/{version}/undelete` or equivalent hide/unhide aliases, plus CLI commands such as `clawhub version hide|delete|unhide|undelete <slug> <version> --yes`. Update schemas, docs, and tests, and add a `CHANGELOG.md` entry because this is user-facing. Keep hard deletion and visible deprecation warnings/status out of scope unless maintainers explicitly choose that policy.
