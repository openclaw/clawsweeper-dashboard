## Summary

Keep open: this is an unresolved ClawHub feature request, and repository policy keeps issues open for maintainer handling. Current main can represent and block soft-deleted skill versions, but owner-facing CLI/API lifecycle controls still operate on whole skills; the broader policy is already tracked at https://github.com/openclaw/clawhub/issues/905.

## What This Changes

Request an owner-facing CLI/API command to deprecate, hide, or delist one published skill version without affecting the rest of the skill history.

## Best Possible Solution

Use https://github.com/openclaw/clawhub/issues/905 as the canonical lifecycle-policy thread, then add authenticated version-scoped owner/staff API and CLI support that updates `skillVersions.softDeletedAt`, writes version-specific audit logs, and keeps public list/detail/file/download/tag/latest/restore behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because current main models and gates soft-deleted versions but exposes only skill-scoped owner lifecycle controls.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped owner API plus CLI is directionally right, but latest-version, tag, cache, download, restore, permission, and audit semantics should be decided together under the canonical lifecycle thread.
