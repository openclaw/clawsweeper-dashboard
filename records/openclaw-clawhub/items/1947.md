## Summary

Keep open. Current main already has the data model and public filtering for soft-deleted skill versions, but there is still no owner-facing CLI or HTTP API path to delist, restore, or deprecate one published version.

## What This Changes

Request a CLI/API command to deprecate, hide, or delist a single published skill version without deleting the whole skill history.

## Best Possible Solution

Add owner/staff per-version delist and restore using skillVersions.softDeletedAt, keep latest/tag pointers coherent, and document the CLI/API behavior while deferring separate deprecation semantics.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence because current main exposes version reads and whole-skill lifecycle writes, but no owner-facing version-level write command or route.

## Solution Assessment

Yes for delist/hide: reusing the existing soft-delete model is the narrowest maintainable path. Explicit deprecation should remain separate unless maintainers define new semantics.
