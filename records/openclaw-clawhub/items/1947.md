## Summary

Keep open. Current main has whole-skill hide/delete plus internal staff-only version soft-delete support, but it still lacks an owner-facing CLI/API/docs path to delist, restore, or deprecate a single published skill version; ClawHub cleanup policy also keeps issues open.

## What This Changes

Add a command/API flow to delist, hide, or deprecate one published skill version without removing the rest of the skill history.

## Best Possible Solution

Add an owner/staff per-version delist and restore flow backed by `skillVersions.softDeletedAt`, keep latest and tag pointers valid, document the CLI/API, and leave a separate per-version deprecation model to maintainer product choice.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence from source inspection: version reads and whole-skill lifecycle writes exist, but no owner-facing version-level write route or CLI command exists.

## Solution Assessment

Yes for delist/hide: reusing the existing version soft-delete model is the narrowest maintainable path. A separate deprecation flag should be added only if maintainers want visible-but-discouraged semantics instead of hiding from resolution/downloads.
