## Summary

Keep open. Current main still lacks an owner-facing command or public API to deprecate, delist, hide, or restore one published version; the exposed lifecycle commands delete or restore whole skills/packages, while version soft-delete is only available through staff/internal paths.

## What This Changes

Requested a CLI command to deprecate or delist one published version without affecting the rest of the skill or package history.

## Best Possible Solution

Define one version lifecycle policy, then add audited owner/staff API and CLI support that reuses existing soft-delete fields where appropriate, preserves latest/tag consistency, and documents separate deprecation semantics if maintainers want them.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence from source inspection: current main exposes read-only version inspection plus whole-skill/package lifecycle writes, not a version-level owner command or route.

## Solution Assessment

Unclear. Reusing `softDeletedAt` is likely the narrowest path for delist/hide, but the requested deprecation semantics and package-release scope need maintainer product/API decisions first.
