## Summary

Keep open. Current main can represent and internally soft-delete individual skill versions, and public readers honor that state, but there is still no owner-facing CLI/API/docs workflow for delisting one published version; ClawHub cleanup policy also keeps issues open for maintainer handling.

## What This Changes

The issue requests a version-scoped CLI or API command to deprecate, hide, or delist one published skill version without affecting the rest of the skill history.

## Best Possible Solution

Define a version-scoped lifecycle policy, then expose authenticated owner/org-member and staff controls that update version soft-delete state consistently across CLI, HTTP API, public readers, docs, audit logs, and restore behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main has internal version soft-delete support but no owner-facing version lifecycle command or write API.

## Solution Assessment

Unclear until maintainers settle the product policy. A version-scoped command/API is directionally right, but the maintainable solution should define authorization, latest-version handling, restore semantics, docs, and audit logging rather than exposing the internal staff mutation directly.
