## Summary

Keep open: this is an unresolved feature request, not an implemented cleanup candidate. Current main can represent and hide soft-deleted versions from public reads/downloads, but the owner-facing CLI and HTTP lifecycle controls remain skill/package scoped, and the closely related lifecycle-policy work is still open.

## What This Changes

Request an owner-facing CLI/API workflow to deprecate, hide, or delist a single published version without affecting the rest of the package or skill history.

## Best Possible Solution

Use https://github.com/openclaw/clawhub/issues/905 as the canonical skill-version lifecycle thread, coordinate the package side with https://github.com/openclaw/clawhub/issues/1370, then implement authenticated version-scoped owner/staff API, CLI, docs, tests, audit logging, and public-access consistency once policy is settled.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because current main stores and honors per-version soft-delete state but exposes only skill/package-scoped lifecycle controls.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped owner API plus CLI is directionally right, but the maintainable solution must define latest-version, tag, restore, cache, permission, and audit semantics first.
