## Summary

Keep open. Current main still lacks an owner-facing skill-version deprecate/delist workflow; the existing implementation has internal/staff-only version soft-delete plumbing and slug-level skill hide/delete commands, while #905 remains the broader open lifecycle-policy tracker.

## What This Changes

Request an owner-facing command or API to deprecate, hide, or delist one published skill version without removing the whole skill history.

## Best Possible Solution

Use #905 as the canonical lifecycle-policy discussion, then add version-scoped API, CLI, docs, tests, audit logging, and restore behavior once owner/staff semantics are defined.

## Reproduction Assessment

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped API and CLI command fit the request, but latest-version handling, restore behavior, and owner versus staff permissions should be defined first.
