## Summary

Keep open. Current main already stores and filters soft-deleted skill versions, but the exposed CLI and HTTP write paths are still whole-skill lifecycle controls, and ClawHub cleanup policy keeps issues open unless they are PRs already implemented on main.

## What This Changes

Request an owner-facing command/API to deprecate, hide, or delist one published skill version without deleting the rest of the skill history.

## Best Possible Solution

Add owner/staff per-version delist and restore around `skillVersions.softDeletedAt`, keep latest/tag pointers valid, document the CLI/API surface, and defer explicit deprecation semantics until maintainers define them.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence from source inspection: version reads exist and honor `softDeletedAt`, but no owner-facing version-level write command or route exists.

## Solution Assessment

Yes for delist/hide: reusing `skillVersions.softDeletedAt` is the narrowest maintainable primitive already supported by public reads. Treating deprecation as identical to delisting is unclear and should remain a separate product decision.
