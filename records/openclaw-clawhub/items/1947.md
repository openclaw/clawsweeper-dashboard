## Summary

Keep open. Current main already models and filters soft-deleted skill versions, but it still lacks an owner-facing CLI/API path to delist, hide, restore, or deprecate one published version; ClawHub cleanup policy also keeps issues open unless they are PRs already implemented on main.

## What This Changes

The issue requests a command/API to deprecate, hide, or delist a single published version without deleting the whole skill/package history.

## Best Possible Solution

Add owner/staff per-version delist and restore using skillVersions.softDeletedAt, keep latest/tag pointers valid, document the CLI/API surface, and defer explicit per-version deprecation semantics unless maintainers define them.

## Reproduction Assessment

Not applicable as a runtime bug, but the feature gap is high-confidence: current main exposes version reads and whole-skill lifecycle writes, with no owner-facing version-level write command or route.

## Solution Assessment

Yes for delist/hide: reusing the existing version soft-delete model is the narrowest maintainable path. Explicit deprecation should be a separate product decision if it needs semantics beyond hiding from public resolution and downloads.
