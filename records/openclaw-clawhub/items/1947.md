## Summary

Keep open. Current main has internal skill-version soft-delete storage and public read filtering, but it still lacks an owner-facing CLI/API command to delist, hide, or deprecate one published skill version; this is a new lifecycle feature and overlaps the broader open #905 policy thread.

## What This Changes

Request an owner-facing CLI/API command to deprecate, hide, or delist a specific published skill version without deleting the whole skill.

## Best Possible Solution

Use #905 as the canonical lifecycle-policy thread, then implement a version-scoped owner API and CLI with hide/delist/deprecate/restore semantics, audit logging, docs, tests, and clear latest-version rules.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection on current main shows read-only version endpoints and skill-scoped delete/undelete commands, but no owner-facing version-scoped write route or CLI command.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped owner API plus CLI matches the request directionally, but latest-version handling, restore/deprecation states, permissions, and audit semantics should be defined first.
