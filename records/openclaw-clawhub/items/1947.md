## Summary

Keep open. Current main models and enforces hidden skill versions internally, but the owner-facing API/CLI path to delist or restore one published skill version is still missing; current commands only hide or restore whole skills. Related #905 tracks the same owner-facing gap, so one focused implementation should cover both.

## What This Changes

Add a command/API for owners to deprecate, hide, or delist a single published version without deleting the whole skill.

## Best Possible Solution

Build version-level soft-delete and restore around the existing `skillVersions.softDeletedAt` model, with explicit latest-tag handling and owner/staff authorization.

## Reproduction Assessment

Not applicable as a feature request, but the current-main capability check is high-confidence: the schema/read paths understand hidden versions while CLI/API docs expose only whole-skill lifecycle commands.

## Solution Assessment

Yes, with clarified scope. The maintainable path is to surface authenticated skill-version soft-delete/restore, not to reuse package release moderation or whole-skill deletion.
