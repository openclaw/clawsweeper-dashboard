## Summary

Keep open: this remains a valid feature request and ClawHub issue cleanup policy does not auto-close issues. Current main can model and gate a soft-deleted skill version, but owner-facing API/CLI/docs still only expose whole-skill delete, hide, undelete, and unhide, and the same lifecycle policy is already being discussed at https://github.com/openclaw/clawhub/issues/905.

## What This Changes

Request an owner-facing CLI/API command to deprecate, hide, or delist one published skill version without affecting the rest of the skill history.

## Best Possible Solution

Use https://github.com/openclaw/clawhub/issues/905 as the canonical policy thread, then implement authenticated owner/org-member/staff version-scoped API and CLI support that updates `skillVersions.softDeletedAt`, writes version-specific audit records, and keeps list/detail/file/download/tag/latest behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because current main models and gates soft-deleted versions but exposes owner lifecycle controls only at the skill level.

## Solution Assessment

Unclear until maintainers settle the broader lifecycle policy in https://github.com/openclaw/clawhub/issues/905. A version-scoped owner API plus CLI is directionally appropriate, but implementing this issue alone would skip decisions about tags, latest versions, restores, caches, downloads, permissions, and audit evidence.
