## Summary

Keep open. Current main has the data model and public filtering needed for hidden skill versions, but the owner-facing CLI, HTTP API, docs, and tests still only expose read-only version access plus whole-skill hide/delete lifecycle writes.

## What This Changes

Request an owner-facing CLI/API command to delist, deprecate, hide, or restore a single published skill version without deleting the whole skill.

## Best Possible Solution

Add authenticated owner/staff per-version delist and restore using `skillVersions.softDeletedAt`, with audit logs and explicit latest/tag behavior; treat visible deprecation as a separate product semantic unless maintainers opt into it.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence from source inspection: version endpoints are read-only, public paths already honor `softDeletedAt`, and lifecycle writes remain whole-skill scoped.

## Solution Assessment

Yes for delist/hide: reusing existing `skillVersions.softDeletedAt` is the narrowest maintainable path. Physical deletion should remain out of scope, and visible deprecation should be added only with an explicit status/badge design.
