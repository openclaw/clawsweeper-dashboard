## Summary

Keep open. Current main supports internal, staff-only soft deletion of individual skill versions and public readers honor that state, but there is still no owner-facing CLI/API/docs workflow for delisting one published version; related #905 remains open for the same product gap.

## What This Changes

The issue requests an owner-facing command or API to deprecate, hide, or delist a single published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then expose authenticated owner/org-member/staff CLI and HTTP controls that preserve audit logs, protect latest/tag behavior, and document restore semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence the gap exists because current CLI/API lifecycle writes are skill-wide while version soft-delete remains internal and staff-only.

## Solution Assessment

Unclear until maintainers settle lifecycle semantics. Exposing the internal mutation directly would be too narrow because authorization, latest version handling, tags, audit logs, and restore behavior need to be defined together.
