## Summary

Keep open. This is a valid feature request for owner-facing single-version delist/deprecate controls; current main has internal staff-only version soft-delete plumbing and public readers honor it, but there is still no publisher-facing CLI/API/docs workflow, and ClawHub cleanup policy keeps issues open for maintainer handling.

## What This Changes

Request an owner-facing CLI/API command to deprecate, hide, or delist one published skill version without deleting the whole skill history.

## Best Possible Solution

Define a version-scoped lifecycle policy and expose it consistently through authenticated owner/org-member and staff API, CLI, docs, readers, restore behavior, and audit logs, likely consolidated with #905.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main lacks an owner-facing version delist/deprecate command or write API.

## Solution Assessment

Unclear until maintainers settle the product policy. A version-scoped command/API is aligned with the request, but the maintainable solution should define authorization, latest-version constraints, restore semantics, docs, and audit logging before exposure.
