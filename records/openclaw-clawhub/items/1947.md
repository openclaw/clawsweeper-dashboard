## Summary

Keep open. Current main has storage and public filtering for soft-deleted skill versions, but it still lacks an owner-facing CLI or HTTP API path to delist, restore, or explicitly deprecate one published version; ClawHub policy also keeps issues open for maintainer follow-up.

## What This Changes

Request a command or API flow to mark a specific published version as deprecated, hidden, or delisted without deleting the rest of the package or skill history.

## Best Possible Solution

Add owner/staff per-version delist and restore behavior using `skillVersions.softDeletedAt`, keep latest/tag pointers coherent, and document the CLI/API while treating visible deprecation as a separate product decision.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence because current main exposes version reads and whole-skill lifecycle writes, but no owner-facing version-level write command or route.

## Solution Assessment

Yes for delist/hide: reusing the existing soft-delete model is the narrowest maintainable path. Explicit deprecation should be separate unless maintainers want visible lifecycle semantics beyond hiding.

## Repair Work Prompt

Implement owner/staff skill-version delist and restore support for ClawHub. Use the existing `skillVersions.softDeletedAt` model; expose authenticated HTTP API write routes for a specific skill version, add matching CLI commands, update docs, and add focused tests. Keep latest-version and tag pointers coherent: either retarget to the newest non-deleted version with summary/tag updates or reject latest-version delist with a clear documented error, but do not leave `latestVersionId`, `tags.latest`, or public package aliases pointing at a hidden version.
