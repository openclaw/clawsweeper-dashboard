## Summary

Keep open. Current main still has storage and public filtering for soft-deleted skill versions, but the owner-facing CLI, HTTP API, and docs only support read-only version access and whole-skill lifecycle writes.

## What This Changes

Request an owner-facing command/API path to deprecate, hide, or delist a specific published skill version without deleting the whole skill history.

## Best Possible Solution

Add authenticated owner/staff per-version delist and restore using `skillVersions.softDeletedAt`, with audit logs, coherent latest/tag behavior, CLI/API docs, and separate product handling for visible deprecation semantics.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence from source inspection: version endpoints are read-only, while lifecycle writes are whole-skill scoped.

## Solution Assessment

Yes for delist/hide: reusing the existing soft-delete model is the narrowest maintainable path. Visible deprecation should be a separate status or badge only if maintainers want that user-facing semantic.

Use the existing `skillVersions.softDeletedAt` model; add authenticated HTTP API write routes under the skill version surface, add matching CLI commands, update response schemas, add CLI/API docs, and include a user-facing `CHANGELOG.md` entry.
