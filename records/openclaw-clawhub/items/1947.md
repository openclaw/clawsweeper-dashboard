## Summary

Keep open. Current main can represent and internally soft-delete individual skill versions, and public readers honor that state, but there is still no owner-facing CLI/API/docs workflow for delisting one published version; related #905 remains open for the same product gap and ClawHub policy keeps issues open for maintainer handling.

## What This Changes

Request an owner-facing CLI/API command to deprecate, hide, or delist a specific published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update version soft-delete state consistently across CLI, HTTP API, public readers, docs, audit logs, and restore behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Current-main source inspection gives high confidence that only whole-skill delete/undelete is exposed while version soft-delete remains internal and staff-only.

## Solution Assessment

Unclear until maintainers settle version lifecycle policy. The requested workflow is directionally right, but the maintainable solution should define authorization, latest/tag handling, audit logging, and restore semantics rather than wrapping the internal mutation directly.
