## Summary

Keep open. Current main still lacks an owner-facing version delist/deprecate workflow; it has internal staff-only version soft-delete plumbing and public readers honor that state, while #905 remains the broader overlapping lifecycle tracker.

## What This Changes

The issue requests a CLI/API command to deprecate, hide, or delist one published skill version without deleting the whole skill.

## Best Possible Solution

Triage this together with #905, define owner/staff semantics, then implement version-scoped API, CLI, docs, tests, audit logging, and restore behavior consistently.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection on current main shows version soft-delete state and public filtering, but no owner-facing version-scoped write API or CLI command.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped API/CLI command matches the requested direction, but latest-version handling, restore behavior, audit semantics, and owner versus staff permissions should be defined first.
