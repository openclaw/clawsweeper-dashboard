## Summary

Keep open. Current main still has storage and read-side filtering for soft-deleted skill versions, but no owner-facing CLI or HTTP write path to delist, restore, or visibly deprecate one published version.

## What This Changes

Request an owner-facing command/API path to deprecate, hide, or delist a specific published skill version without deleting the whole skill.

## Best Possible Solution

Add owner/staff per-version delist and restore behavior using `skillVersions.softDeletedAt`, keep latest/tag pointers coherent, and document the CLI/API while treating visible deprecation as a separate product semantic unless maintainers define it now.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence from source inspection: current main exposes version reads and whole-skill lifecycle writes, but no version-level owner write route or CLI command.

## Solution Assessment

Yes for delist/hide: reusing the existing soft-delete model is the narrowest maintainable path. Explicit deprecation should be separate unless maintainers want a visible lifecycle badge/status in addition to hiding.
