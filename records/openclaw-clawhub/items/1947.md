## Summary

Keep open. Current main has version-level soft-delete state, public filtering for already soft-deleted versions, and a staff-only internal withdrawal mutation, but it does not expose an owner-facing CLI/API/docs path to delist, hide, restore, or deprecate one published skill version; related #905 remains open for the same lifecycle gap.

## What This Changes

Request a command to deprecate or delist a single published skill version without deleting the rest of the skill history.

## Best Possible Solution

Define version-level delist/restore semantics, then expose authenticated owner/staff controls that update skillVersions.softDeletedAt, preserve audit logs, and keep public lists, tags, latest, file, scan, and download behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence from static inspection: current main exposes version reads and whole-skill lifecycle writes, while version-level writes are internal and staff-only.

## Solution Assessment

Unclear. Reusing skillVersions.softDeletedAt is the narrowest path for delist/hide, but deprecation semantics and latest/tag behavior need maintainer policy before implementation.
