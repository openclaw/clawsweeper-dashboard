## Summary

Keep open. Current main can internally soft-delete individual skill versions and public readers honor that state, but the requested owner-facing CLI/API workflow for delisting one published version is still absent; the related #905 remains open rather than merged or shipped.

## What This Changes

Request an owner-facing command or API to deprecate, hide, or delist one published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member and staff controls across CLI, HTTP API, docs, audit logs, and reader/download behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main lacks an owner-facing version lifecycle command/API despite internal soft-delete support.

## Solution Assessment

Unclear until maintainers settle the product policy. A version-scoped command/API is directionally right, but the maintainable solution should define permissions, latest-version constraints, restore semantics, docs, and audit logging first.
