## Summary

Keep open. This is a valid owner-facing feature gap: current main can represent and public-filter soft-deleted skill versions, but the CLI, HTTP API, docs, and OpenAPI surface only expose read-only version access plus whole-skill delete/undelete lifecycle writes.

## What This Changes

Add an owner-facing CLI/API workflow to delist, hide, or deprecate a single published skill version without deleting the entire skill.

## Best Possible Solution

Implement authenticated owner/staff per-version hide/delist and restore on top of skillVersions.softDeletedAt, with audit logs, clear latest/tag rules, CLI/API/docs coverage, and a separate decision for visible version deprecation semantics.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence from source inspection: version endpoints are read-only while lifecycle writes remain whole-skill scoped.

## Solution Assessment

Yes for hide/delist: reusing skillVersions.softDeletedAt is the narrowest maintainable solution. Visible deprecation should be a separate status or badge only if maintainers want that user-facing semantic.
