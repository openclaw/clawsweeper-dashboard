## Summary

Repository policy keeps ClawHub issues open, and current main still lacks an owner-facing command or API for delisting one published skill version. The code has soft-delete support for version records and public reads hide those versions, but the only version-level writer found is an internal staff/moderator maintenance mutation; this overlaps the broader open #905 lifecycle request.

## What This Changes

Request an owner-facing CLI/API workflow to deprecate, hide, or delist a single published skill version without deleting the whole skill.

## Best Possible Solution

Use #905 as the lifecycle-policy thread, then add a version-scoped owner API and CLI once latest-version, restore, audit, and permission semantics are defined.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection on current main shows read-only version endpoints and internal staff-only version soft-delete support, but no owner-facing version-scoped CLI/API workflow.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped owner API plus CLI command matches the request directionally, but latest-version handling, restore/deprecation states, permissions, and audit semantics should be defined first.
