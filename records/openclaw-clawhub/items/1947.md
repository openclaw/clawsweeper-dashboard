## Summary

Keep open. Current main has the storage and public filtering behavior for soft-deleted skill versions, but it still lacks an owner-facing CLI or HTTP API path to delist, restore, or explicitly deprecate one published version.

## What This Changes

The issue requests an owner-facing command or API to delist, hide, or deprecate one published version without deleting the whole skill.

## Best Possible Solution

Add owner/staff per-version delist and restore behavior using `skillVersions.softDeletedAt`, keep latest/tag pointers coherent, and document the CLI/API while deferring separate deprecation semantics unless maintainers define them.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence because current main exposes version reads and whole-skill lifecycle writes, but no owner-facing version-level write command or route.

## Solution Assessment

Yes for delist/hide: reusing the existing soft-delete model is the narrowest maintainable path. Explicit deprecation should be a separate status or badge only if maintainers want visible lifecycle semantics beyond hiding.
