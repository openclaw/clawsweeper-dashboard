## Summary

Keep open. Current main already has version-level soft-delete state and public read/download filtering, but the owner-facing CLI, HTTP API, and docs still only expose whole-skill hide/delete/restore; the existing ClawSweeper comment and related #905 report match this source inspection.

## What This Changes

The issue requests a command/API flow to deprecate, hide, or delist one published skill version without removing the rest of the skill history.

## Best Possible Solution

Expose authenticated owner/staff per-version delist and restore backed by `skillVersions.softDeletedAt`, preserve public filtering/download behavior and tag/latest invariants, and defer a separate deprecation model unless maintainers choose visible-but-discouraged semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes as a feature-gap check: current main has version read/filter behavior and whole-skill lifecycle writes, with no owner-facing version-level write path.

## Solution Assessment

Is this the best way to solve the issue? Yes for delist/hide if scoped to the existing soft-delete model; a new per-version deprecation field is not the narrowest fix without a maintainer decision.
