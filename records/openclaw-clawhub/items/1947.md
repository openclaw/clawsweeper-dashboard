## Summary

Repository policy keeps ClawHub issues open, and the requested owner-facing version lifecycle command still is not implemented on current main. The data model and public read paths already understand soft-deleted versions, but the available write surfaces are skill-scoped or internal staff-only, so this remains a feature/policy item rather than cleanup to close.

## What This Changes

Request an owner-facing CLI/API command to deprecate, hide, or delist a specific published skill version without deleting the whole skill.

## Best Possible Solution

Use #905 as the lifecycle-policy thread, then add a version-scoped owner API and CLI with restore/deprecation semantics, audit behavior, docs, and tests once latest-version and permission rules are defined.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection on current main shows read-only version endpoints and internal staff-only version soft-delete support, but no owner-facing version-scoped CLI/API workflow.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped owner API plus CLI command matches the request directionally, but latest-version handling, restore/deprecation states, permissions, and audit semantics should be defined first.
