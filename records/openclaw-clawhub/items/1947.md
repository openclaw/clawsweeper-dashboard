## Summary

Keep open. This is a valid feature/product request: current main can represent and hide soft-deleted skill versions internally, but it still exposes only whole-skill delete/hide workflows to publishers, and related #905 remains open for the same owner-facing version lifecycle gap.

## What This Changes

Request owner-facing CLI/API support to deprecate, hide, or delist one published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls across CLI, HTTP API, docs, audit logs, latest/tag handling, and restore behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main exposes only whole-skill delete/undelete while version soft-delete remains internal and staff-only.

## Solution Assessment

Unclear until maintainers settle the lifecycle policy. The requested direction is reasonable, but the maintainable solution should define authorization, latest/tag handling, audit logging, and restore semantics rather than wrapping the internal mutation directly.
