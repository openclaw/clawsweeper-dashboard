## Summary

Keeping this open: it is an unresolved feature request, and current main still lacks an owner-facing version-scoped deprecate/delist/hide workflow. The request overlaps open #905, but ClawHub cleanup policy keeps issues open unless a PR is certainly implemented on main.

## What This Changes

Add an owner-facing CLI/API workflow to deprecate, hide, or delist a single published skill version without affecting the rest of the skill history.

## Best Possible Solution

Use #905 as the canonical lifecycle-policy thread, then implement a version-scoped owner API and CLI once permissions, latest-version, restore, audit, and naming semantics are settled.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection shows internal staff-only version soft-delete support and public read hiding, but no owner-facing version-scoped CLI/API workflow.

## Solution Assessment

Unclear until maintainers settle the broader lifecycle policy in #905. A version-scoped owner API plus CLI matches the request directionally, but wiring the current internal admin-only mutation directly to owners would skip latest-version, restore, audit, and terminology decisions.
