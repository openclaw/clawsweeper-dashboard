## Summary

Keep open. The prior review correctly linked related #905, and current main still lacks an owner-facing CLI/API/docs/UI path to delist one published skill version even though internal version soft-delete state exists and public readers honor it; ClawHub policy keeps issues open for maintainer handling.

## What This Changes

Request an owner-facing command or API to deprecate, hide, or delist a single published skill version without deleting the whole skill.

## Best Possible Solution

Define a version-scoped lifecycle policy, then add authenticated owner/org-member/staff controls that update `skillVersions.softDeletedAt` with audit logs while keeping public lists, detail/file/download routes, tags/latest behavior, docs, and restore semantics consistent.

## Reproduction Assessment

Not applicable as a runtime bug. Current-main source verification is high confidence: the CLI delete/hide commands accept only slugs, and the HTTP DELETE router rejects version paths.

## Solution Assessment

Unclear until maintainers choose the version lifecycle policy. The requested owner-facing workflow is directionally right, but the maintainable solution should not simply expose the staff-only internal mutation without latest/tag, audit, restore, and cache rules.
