## Summary

Keep open. Current main has version-level `softDeletedAt` storage and public reads/downloads honor it, plus an internal staff-only maintenance mutation can soft-delete non-latest versions, but the owner-facing CLI/HTTP surface still only hides or restores whole skills. The overlapping #905 context also remains open, and ClawHub cleanup policy keeps issues open for maintainer handling.

## What This Changes

The issue requests an owner-facing command to deprecate, hide, or delist a single published skill version without removing the rest of the skill history.

## Best Possible Solution

Define one skill-version lifecycle model, then expose an owner-authorized HTTP API and CLI command that reuse the existing per-version visibility checks and document the behavior.

## Reproduction Assessment

Yes. Current-main inspection is enough to reproduce the feature gap: `clawhub` registers whole-skill hide/delete commands and the HTTP DELETE route only targets a skill slug, while version-specific reads merely honor existing `softDeletedAt` state.

## Solution Assessment

Unclear. A CLI command is likely the right UX, but the requested command name and behavior need maintainer decisions around deprecation versus delisting, latest-version refusal, tags, audit logs, and permissions.
