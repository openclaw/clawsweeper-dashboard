## Summary

Keep open: this is still a valid feature request and ClawHub policy keeps issues open for maintainer handling. Current main can represent and gate a soft-deleted skill version internally, but owner-facing API, CLI, and docs still expose only whole-skill delete/hide/undelete/unhide; the broader lifecycle policy is already tracked at https://github.com/openclaw/clawhub/issues/905.

## What This Changes

Request an owner-facing command or API to deprecate, hide, or delist one published skill version without affecting the rest of the skill history.

## Best Possible Solution

Use https://github.com/openclaw/clawhub/issues/905 as the canonical policy thread, then implement authenticated owner/org-member/staff version-scoped API and CLI support that updates `skillVersions.softDeletedAt`, writes version-specific audit records, and keeps list/detail/file/download/tag/latest behavior consistent.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because current main models and gates withdrawn versions, while exposed owner lifecycle controls remain skill-scoped.

## Solution Assessment

Unclear until maintainers settle the broader lifecycle policy in https://github.com/openclaw/clawhub/issues/905. A version-scoped owner command/API is directionally appropriate, but the durable fix must cover latest versions, tags, restores, caches, downloads, permissions, and audit evidence together.
