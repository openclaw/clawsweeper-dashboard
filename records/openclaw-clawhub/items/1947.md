## Summary

Keep open: this is an open ClawHub issue, and the requested per-version delist/deprecate workflow remains a valid security-sensitive feature gap. Current main can model and block soft-deleted versions internally, but owner-facing CLI/API/docs still only expose skill-level delete, hide, undelete, and unhide; the broader policy is already tracked at https://github.com/openclaw/clawhub/issues/905.

## What This Changes

Request a command/API workflow to deprecate or delist one published skill version without affecting the rest of the skill history.

## Best Possible Solution

Use https://github.com/openclaw/clawhub/issues/905 as the canonical policy thread, then implement authenticated owner/org-member/staff version-scoped controls with consistent API, CLI, docs, tests, public access gates, and version-specific audit records.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence in the gap because version soft-delete state and public gates exist, while owner-facing per-version lifecycle controls are absent.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped owner API plus CLI is directionally appropriate, but the maintainable solution must define latest, tag, cache, download, restore, permission, and audit semantics first.
