## Summary

Keep open. Current main has internal per-version soft-delete mechanics and public readers honor them, but there is still no owner-facing CLI/API/docs workflow to deprecate, hide, or delist one published skill version; related #905 remains open for the same product gap and ClawHub policy keeps issues open for maintainer handling.

## What This Changes

Request an owner-facing CLI/API command to deprecate, hide, or delist one published version without deleting the whole skill.

## Best Possible Solution

Define one version-scoped lifecycle policy, preferably through #905, then expose authenticated owner/org-member and staff controls consistently across CLI, HTTP API, docs, audit logs, public readers, and restore behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main only exposes whole-skill hide/delete while version soft-delete remains internal and staff-only.

## Solution Assessment

Unclear until maintainers settle version lifecycle policy. The requested command is directionally reasonable, but the maintainable solution needs explicit authorization, latest/tag handling, audit logging, and restore semantics rather than directly exposing the internal mutation.
