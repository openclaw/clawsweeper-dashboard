## Summary

Keep open. Current main can store and publicly filter soft-deleted skill versions, but the owner-facing CLI, HTTP API, and docs still only expose read-only version access plus whole-skill delete/undelete; related #905 tracks the same feature gap.

## What This Changes

Request an owner-facing CLI/API command to deprecate, hide, or delist a single published skill version without deleting the whole skill.

## Best Possible Solution

Add authenticated owner/staff per-version delist and restore around `skillVersions.softDeletedAt`, with audit logs, coherent latest/tag handling, CLI/API/docs coverage, and separate handling for any visible deprecation status.

## Reproduction Assessment

Not applicable as a runtime bug. The feature gap is high-confidence from source inspection: current version endpoints are read-only, while lifecycle writes are whole-skill scoped.

## Solution Assessment

Yes for delist/hide: reusing `skillVersions.softDeletedAt` is the narrowest maintainable path. Visible deprecation should be a separate status or badge only if maintainers want a user-facing warning semantic.

Implement owner/staff per-skill-version delist and restore for skills. Add authenticated API routes such as `DELETE /api/v1/skills/{slug}/versions/{version}` and `POST /api/v1/skills/{slug}/versions/{version}/undelete` or equivalent hide/unhide naming, backed by a Convex mutation that resolves slug/version via indexed lookups, authorizes the skill owner or moderator/admin, refuses the latest version unless maintainers choose a coherent retargeting policy, patches `skillVersions.softDeletedAt`, and writes audit logs.
