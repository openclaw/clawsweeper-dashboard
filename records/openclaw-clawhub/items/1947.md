## Summary

Current main still has internal skill-version soft-delete support and public read filtering, but no owner-facing CLI/API to delist or deprecate a single published version; this is a new lifecycle feature that should stay open for maintainer policy direction, especially alongside the broader open #905 thread.

## What This Changes

Request an owner-facing command or API to deprecate, hide, or delist one published version without deleting the rest of the skill history.

## Best Possible Solution

Use #905 as the lifecycle-policy thread, then implement a version-scoped owner API/CLI with restore/deprecation semantics, audit logging, docs, and tests once permissions and latest-version rules are defined.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection on current main shows read-only version endpoints and skill-scoped delete/undelete, but no public version-scoped write route or CLI command.

## Solution Assessment

Unclear until maintainers settle lifecycle policy. A version-scoped owner API plus CLI command matches the request directionally, but latest-version handling, restore/deprecation states, permissions, and audit semantics should be defined first.
