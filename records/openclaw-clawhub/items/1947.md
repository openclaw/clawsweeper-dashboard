## Summary

Keep open. This is a feature request for an owner-facing version delist/deprecate workflow; current main has internal staff-only version soft-delete plumbing and public readers honor it, but there is no CLI/API/docs path for publishers to delist one published version, and ClawHub cleanup policy keeps issues open for maintainer handling.

## What This Changes

Request an owner-facing command or API to deprecate, hide, or delist a single published version without removing the whole skill history.

## Best Possible Solution

Define a version-scoped lifecycle policy, then expose authenticated owner/org-member and staff controls consistently across CLI, HTTP API, docs, public readers, audit logs, and restore behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main has internal version soft-delete support but no owner-facing version lifecycle command or write API.

## Solution Assessment

Unclear until maintainers settle the product policy. A version-scoped command/API is directionally aligned with the request, but the maintainable solution should define authorization, latest-version constraints, restore semantics, docs, and audit logging before exposing it.
