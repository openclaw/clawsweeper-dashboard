## Summary

Keep open. ClawHub issue policy does not auto-close issues, and current main still lacks an owner-facing version delist/deprecate workflow: the data model and public readers understand soft-deleted versions, but the available write surfaces are internal staff-only or whole-skill delete/restore; open #905 tracks the broader same lifecycle gap.

## What This Changes

The issue requests an owner-facing command or API to deprecate, hide, or delist one published skill version without deleting the whole skill.

## Best Possible Solution

Triage this with open #905, define the version lifecycle policy, then add authenticated owner/staff API, CLI, docs, tests, audit logging, and restore behavior for non-latest published versions.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main has internal version soft-delete plumbing but no owner-facing version lifecycle command, write API, or docs workflow.

## Solution Assessment

Unclear until maintainers settle product policy. A version-scoped command/API matches the request directionally, but the maintainable solution needs explicit authorization, latest-version, restore, audit, and documentation semantics first.
