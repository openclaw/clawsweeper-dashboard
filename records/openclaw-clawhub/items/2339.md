## Summary

Keep this open: current main has owner-facing web deletion, but it still lacks a discoverable owner restore flow, and the remaining restore policy is security-sensitive and collaborator-authored.

## What This Changes

Add owner-facing web controls for whole-skill soft-delete and restore while preserving existing authorization, audit, slug reservation, and moderation/security restore gates.

## Best Possible Solution

Add a dashboard/settings restore section for eligible owner-deleted skills that delegates restore to the existing provenance-aware backend path and shows clear reservation expiry and denial states.

## Reproduction Assessment

Yes. Source inspection shows owner delete exists in settings, while the dashboard filters to active skills and no owner-facing web restore mutation or control exists.

## Solution Assessment

Yes in architecture, but the exact product policy is not settled. The narrow maintainable solution is to reuse the existing backend restore gate rather than create UI-only authorization rules.
