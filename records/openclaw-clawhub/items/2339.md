## Summary

Keep open: current main still lacks owner-facing web delete/restore controls, while the CLI/API and shared backend mutation already provide the underlying soft-delete and restore behavior. This is a current feature/product gap with security-sensitive restore boundaries, not a stale or implemented issue.

## What This Changes

Add owner-facing web controls for deleting and restoring manageable skills while preserving existing authorization, audit, slug-reservation, public-count, embedding, and moderation/security restore gates.

## Best Possible Solution

Implement a web flow on top of setSkillSoftDeletedInternal, with a discoverable owner-deleted restore surface and regression coverage for owner, publisher-admin, non-owner, and moderation/security-denial cases.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection verifies the product gap: current web settings expose rename and merge only, while CLI/API delete and undelete already exist.

## Solution Assessment

Yes, if maintainers accept the product change, reusing setSkillSoftDeletedInternal is the narrow maintainable path. A separate web-only delete/restore implementation would risk drifting from the existing authorization, audit, slug reservation, count, embedding, and moderation gates.
