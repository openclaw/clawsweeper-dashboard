## Summary

Keep open: current main still has the CLI/API/backend soft-delete and restore machinery, but the signed-in web UI does not expose owner-facing whole-skill delete or restore controls. This is a valid new lifecycle UI with ownership and moderation/security restore boundaries, so it needs maintainer product/security acceptance rather than cleanup closure.

## What This Changes

Add owner and publisher-admin web controls to soft-delete and restore whole skill listings, including 30-day slug reservation copy and moderation/security restore denials.

## Best Possible Solution

Implement the web flow on top of the existing soft-delete mutation semantics, with a discoverable owner-deleted restore surface and regression coverage for owner, publisher-admin, non-owner, and moderation/security-denial cases.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the product gap because current web settings and dashboard surfaces do not expose delete or restore while the CLI/API do.

## Solution Assessment

Yes, if maintainers accept the product behavior, reusing the existing server-side soft-delete path is the narrow maintainable solution. A separate web-only lifecycle implementation would risk drifting from the authorization, audit, slug reservation, count, and embedding behavior already in backend code.
