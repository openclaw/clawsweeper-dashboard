## Summary

Keep this open: current main has owner-facing web deletion, but it still lacks an owner-visible restore path, and the issue is collaborator-authored with a security-sensitive restore boundary.

## What This Changes

Add discoverable web UI controls for owners and publisher admins to soft-delete owned skills and restore eligible owner-deleted skills during the reservation window.

## Best Possible Solution

Add a dashboard/settings restore section for eligible owner-deleted skills that delegates to the existing provenance-aware backend restore path and clearly shows reservation expiry and denial states.

## Reproduction Assessment

Yes. Source inspection shows delete exists in settings, while the dashboard filters to active skills and the only owner web mutation sets deleted: true, so owner-deleted skills cannot be found and restored in the web UI.

## Solution Assessment

Yes in architecture, but the exact product policy needs maintainer decision. The maintainable path is to reuse the existing provenance-aware restore gate rather than duplicate authorization rules in the client.
