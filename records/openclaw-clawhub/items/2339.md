## Summary

Keep open: current main still has the CLI/API/backend skill delete and restore path, but the signed-in owner web UI does not expose discoverable whole-skill delete or restore controls, and the restore surface needs maintainer product/security acceptance before implementation.

## What This Changes

The issue requests owner-facing signed-in web controls to soft-delete and restore manageable skill listings while preserving existing authorization, audit, slug-reservation, count, embedding, and moderation/security restore gates.

## Best Possible Solution

Implement the owner web flow on top of the existing backend soft-delete semantics, with a discoverable owner-deleted restore surface and regression coverage for owner, publisher-admin, non-owner, and moderation/security denial cases.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the product gap because owner settings expose rename/merge but not delete/restore, and the dashboard lists only active skills.

## Solution Assessment

Yes, if maintainers accept the product behavior: reusing the existing backend lifecycle path is the narrow maintainable solution. A separate web-only implementation would risk drifting from the authorization, audit, slug reservation, count, and embedding behavior already present in backend code and tests.
