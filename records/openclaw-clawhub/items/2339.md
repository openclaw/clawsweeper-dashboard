## Summary

Keep this open: current main includes the owner-facing web delete flow from the merged settings PR, but it still does not expose an owner-visible web restore path, and the collaborator-authored/security-sensitive restore boundary needs maintainer review.

## What This Changes

The issue requests owner-facing web UI controls to soft-delete and restore whole skills from signed-in dashboard/settings surfaces.

## Best Possible Solution

Add an owner-visible restore section for eligible owner-deleted skills that shows reservation expiry, delegates restore to the existing provenance-aware backend path, and keeps protected hides staff-only.

## Reproduction Assessment

Yes, at source level. Current main renders the delete UI, but the web owner mutation only deletes and dashboard queries filter out soft-deleted skills, so owner-deleted skills are not discoverable or restorable in the web app.

## Solution Assessment

Yes in direction, but the exact UX needs maintainer product and security review. The maintainable path is to reuse the existing provenance-aware restore gate rather than duplicating restore eligibility in the client.
