## Summary

Current main implements owner-facing web deletion, but the requested restore discovery and control remain absent. That is a substantive acceptance criterion, and this collaborator-authored, security-sensitive product issue requires explicit maintainer handling.

## What This Changes

Add discoverable owner and publisher-admin web controls to soft-delete and restore skills while preserving existing authorization and moderation provenance restrictions.

## Best Possible Solution

Extend the accepted skill-settings or dashboard pattern with a server-filtered list of restore-eligible owner deletions and a server-backed restore action, excluding moderation, scanner, ban, merge, and security-redaction deletions while retaining explicit reservation timing.

## Reproduction Assessment

Yes, at source level. Current main directly shows the owner delete action while dashboard queries exclude deleted records and the web code exposes no equivalent restore path.

## Solution Assessment

Yes in architecture, but the implementation is incomplete. The remaining restore flow should reuse server-side authorization and provenance gates rather than introduce client-defined eligibility rules.
