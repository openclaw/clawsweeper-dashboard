## Summary

Keep open: current main still lacks owner-facing web delete/restore controls, while the CLI/API and backend soft-delete gates already exist; this is a new owner-facing lifecycle UI that needs maintainer product/security acceptance rather than cleanup closure.

## What This Changes

Add web UI controls for owners and publisher admins to soft-delete and restore whole skills using the existing delete/undelete semantics.

## Best Possible Solution

Implement the web flow on top of the existing soft-delete mutation semantics, with a discoverable owner-deleted restore surface and regression coverage for owner, publisher-admin, non-owner, and moderation/security-denial cases.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection verifies the product gap: current web settings and dashboard do not expose delete or restore while the CLI/API do.

## Solution Assessment

Yes, if maintainers accept the product behavior, reusing the existing server-side soft-delete path is the narrow maintainable solution. A separate web-only delete implementation would risk drifting from the authorization, audit, slug reservation, count, and embedding logic already covered in backend tests.
