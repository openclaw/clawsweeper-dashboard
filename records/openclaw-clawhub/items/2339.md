## Summary

Keep this open: current main has the owner delete UI via the merged owner-deletion work, but the owner-visible web restore/discovery path is still missing, and the remaining restore boundary is security-sensitive and collaborator-authored.

## What This Changes

Add signed-in web controls so skill owners can soft-delete and restore their own skills while preserving existing moderation and security restore gates.

## Best Possible Solution

Ship a discoverable owner restore surface for eligible owner-deleted skills that delegates to the provenance-aware backend restore path, shows reservation expiry, and keeps protected hides staff-only.

## Reproduction Assessment

Yes, at source level. Current main has a settings delete action, but dashboard queries filter out soft-deleted skills and source search found no owner-facing web restore control.

## Solution Assessment

Yes in architecture, but the exact UX still needs maintainer direction. The narrow maintainable path is to reuse the existing provenance-aware restore gate instead of duplicating authorization in the client.
