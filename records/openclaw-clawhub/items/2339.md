## Summary

Keep open: current main still has CLI/API/backend soft-delete and restore paths, but the signed-in owner settings and dashboard UI do not expose owner-facing delete or restore controls. This remains a new, security-sensitive lifecycle UI that needs maintainer product/security acceptance rather than cleanup closure.

## What This Changes

The issue requests owner-facing web controls for whole-skill soft delete and restore, reusing the existing CLI/API/backend lifecycle behavior.

## Best Possible Solution

Implement the web flow on top of the existing soft-delete semantics, with a discoverable owner-deleted restore surface and regression coverage for owner, publisher-admin, non-owner, and moderation/security denial cases.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the product gap because owner settings expose rename/merge but not delete/restore, and the dashboard lists only active skills.

## Solution Assessment

Yes, if maintainers accept the product behavior: reusing the existing backend soft-delete path is the narrow maintainable solution. A separate web-only lifecycle implementation would risk drifting from the authorization, audit, slug reservation, count, and embedding behavior already present in backend code and tests.
