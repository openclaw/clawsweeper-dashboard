## Summary

Keep open: current main has CLI/API/backend skill soft-delete and restore semantics, but the signed-in web dashboard/settings UI still lacks owner-facing delete and restore controls. The linked support issues are useful evidence for the product need, and this remains a security-sensitive feature request that needs maintainer product/security acceptance before implementation.

## What This Changes

Add discoverable web UI controls that let skill owners and publisher admins soft-delete owned skills and restore owner-deleted skills during the 30-day slug reservation window.

## Best Possible Solution

Implement the web flow on top of the existing soft-delete mutation/API semantics, with a discoverable owner-deleted restore surface and regression coverage for owner, publisher-admin, non-owner, and moderation/security denial cases.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the product gap because owner settings expose rename/merge but not delete/restore, and the dashboard lists only active skills.

## Solution Assessment

Yes, if maintainers accept the product behavior: reusing the existing backend soft-delete path is the narrow maintainable solution. A separate web-only lifecycle implementation would risk drifting from the authorization, audit, slug reservation, count, and embedding behavior already present in backend code and tests.
