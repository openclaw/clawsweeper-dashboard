## Summary

Keep open: current main has the CLI/API/backend soft-delete and restore path, but the signed-in owner web UI still lacks discoverable whole-skill delete and restore controls, and the remaining work crosses security-sensitive ownership and moderation boundaries.

## What This Changes

The issue requests owner-facing web controls to soft-delete and restore manageable skills while preserving existing backend authorization, audit, slug reservation, public count, embedding, and moderation/security gates.

## Best Possible Solution

Implement the web flow on top of the existing backend soft-delete semantics, with a discoverable owner-deleted restore surface and regression coverage for owner, publisher-admin, non-owner, and moderation/security denial cases.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence in the product gap because owner settings expose rename/merge but not delete/restore, and the dashboard lists only active skills.

## Solution Assessment

Yes, if maintainers accept the product behavior: reusing the existing backend lifecycle path is the narrow maintainable solution. A separate web-only implementation would risk drifting from authorization, audit, slug reservation, count, and embedding behavior already present in backend code and tests.
