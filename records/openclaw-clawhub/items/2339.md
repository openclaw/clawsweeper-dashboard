## Summary

Keep open: current main implements owner-facing web deletion, but owners still cannot discover or restore deleted skills in the web app. The remaining restore workflow is meaningful, and this collaborator-authored security-sensitive product boundary requires explicit maintainer handling.

## What This Changes

Add discoverable web controls for owners and publisher admins to soft-delete and restore manageable skills while preserving moderation and security restore restrictions.

## Best Possible Solution

Add a separate dashboard section or filter for eligible owner-deleted skills and a narrowly scoped public restore mutation that delegates authorization and provenance decisions to the existing internal restore path.

## Reproduction Assessment

Yes, at source level: after an owned skill is deleted through its settings page, dashboard queries exclude all rows with softDeletedAt set and the web app provides no restoration control. A live runtime reproduction was not performed during this read-only review.

## Solution Assessment

Yes in direction, subject to maintainer approval of the discovery UX. The narrowest maintainable implementation is to expose only eligible owner-deleted rows and reuse the established server restore gate rather than create web-specific restoration semantics.
