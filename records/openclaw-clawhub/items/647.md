## Summary

Keep open because this ClawHub issue is not an auto-close candidate and current main does not prove the reported production GitHub OAuth session-loss behavior is fixed for active accounts.

## What This Changes

After GitHub OAuth authorization on clawhub.ai, the app should establish an active session and update the signed-in UI instead of returning to login/logged-out state.

## Best Possible Solution

Keep one maintainer-owned OAuth/session-loop investigation that guarantees reliable active-account login, clear terminal messaging for intentionally blocked accounts, and regression coverage across callback handling, user bootstrap, and signed-in UI state.

## Reproduction Assessment

No, we do not have a high-confidence way to reproduce the full production active-account failure on current main. Source inspection does show a clear deleted/deactivated or missing-user path that can leave the UI effectively logged out after OAuth, but that only covers part of the report cluster.

## Solution Assessment

Unclear. A code-only fix is not yet clearly the best solution until maintainers correlate production logs, OAuth/cookie configuration, and affected account state, then separate active-account failures from intentional deleted-account blocking.
