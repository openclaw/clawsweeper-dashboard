## Summary

Keep open. This is an open ClawHub issue, and the repository cleanup policy only permits conservative auto-close for PRs that current main certainly implements. Current main has OAuth callback handling and user-facing auth errors, but the reported production session-loss behavior is not proven fixed; the active UI still depends on resolving an active app user, deleted/deactivated accounts are intentionally blocked, and multiple related OAuth/session-loop reports remain open.

## What This Changes

The issue reports that GitHub OAuth authorization completes successfully but ClawHub returns the user to a logged-out state without establishing or reflecting an active session.

## Best Possible Solution

Treat this as part of a canonical OAuth/session-loop investigation: active accounts should reliably establish a session and resolve a user document after GitHub OAuth, while deleted/deactivated accounts should get a clear terminal error or maintainer-approved recovery/support path with regression coverage for callback handling, user bootstrap, and signed-in UI state.
