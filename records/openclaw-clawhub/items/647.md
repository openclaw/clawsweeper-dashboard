Keep open. Current main has OAuth callback handling and blocked-account messaging, but it does not prove the reported production GitHub OAuth/session-loss behavior is fixed; related OAuth/session-loop reports remain open and the next step needs production auth/config/account-state investigation.

## What This Changes

The issue reports GitHub OAuth authorization returning to ClawHub without establishing an active web session, leaving users logged out after callback retries.

## Best Possible Solution

Treat this as part of a canonical OAuth/session-loop investigation: reliable session creation for active accounts, clear terminal/support messaging for blocked accounts, and regression coverage for callback handling, user resolution, and signed-in UI state.

## Reproduction Assessment

No high-confidence live reproduction for this exact reporter is available. Source inspection shows clear paths where deleted/deactivated or missing app users resolve to no active UI session, but active-account production failures still need OAuth/cookie/log correlation.

## Solution Assessment

No source-only fix is ready yet. The maintainable path is to split active-account session creation failures from intentional deleted/deactivated-account blocking, verify production config/logs, then patch the narrow failing path with tests.
