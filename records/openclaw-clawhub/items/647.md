## Summary

Keep open because this is an open ClawHub issue and current main does not prove the production GitHub OAuth callback/session-loss behavior is fixed. Current source now surfaces more auth failures and intentionally blocks deleted/deactivated accounts, but active-account failures still require production OAuth, cookie, log, and account-state triage.

## What This Changes

The issue reports that GitHub OAuth authorization succeeds but the ClawHub web app returns to a logged-out state without establishing a usable session.

## Best Possible Solution

Maintain one canonical OAuth/session-loop investigation that proves active-account login in production, separates deleted/deactivated-account blocking into a clear error or support flow, and adds regression coverage for callback handling, user bootstrap, and signed-in UI state.

## Reproduction Assessment

No. Source inspection shows plausible logged-out paths after OAuth when user bootstrap fails or the account is deleted/deactivated, but I did not establish a high-confidence current-main reproduction of the reported active-account production session loss.

## Solution Assessment

Unclear. A source-only repair is not yet the best path until maintainers separate active-account failures from intentional deleted/deactivated blocking and verify production OAuth, cookie, and account-state data.
