## Summary

Keep open. Current main still has a source-backed Settings Save failure path: the frontend awaits `users.updateProfile`, and that mutation can still roll back after a profile patch when personal-publisher synchronization throws a handle-conflict `ConvexError`.

## What This Changes

The issue reports that saving display name or bio from the Settings page fails with a Convex server error instead of persisting personal settings.

## Best Possible Solution

Persist personal displayName and bio updates while preserving publisher handle ownership invariants, with targeted backend and Settings form coverage for the conflict path.

## Reproduction Assessment

No, not for the reporter’s exact production account because the request row state is unavailable. Source inspection gives a focused reproduction scenario: call `users.updateProfile` for a user whose derived personal publisher handle conflicts with another publisher and observe the `ConvexError` rollback path.

## Solution Assessment

Yes, the narrowest maintainable fix belongs at the backend save/sync boundary. A frontend-only catch would still leave data unsaved, while swallowing every publisher sync error would hide real ownership failures.
