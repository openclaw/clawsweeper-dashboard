Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-backed failure path where Settings Save can roll back after `users.updateProfile` reaches personal-publisher synchronization and that helper throws a handle-conflict `ConvexError`.

## What This Changes

The issue reports that changing display name or bio on `/settings` and clicking Save fails with a `users.updateProfile` server error instead of persisting personal settings.

## Best Possible Solution

Personal display name and bio updates should persist while preserving publisher handle ownership invariants, with regression coverage for the mutation path and Settings form behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not for the reporter's exact production account, but source inspection gives a focused current-main reproduction: a user whose derived personal publisher handle conflicts with another publisher can trigger the `ConvexError` after `users.updateProfile` patches profile fields.

## Solution Assessment

Is this the best way to solve the issue? Yes, the fix belongs at the backend persistence/synchronization boundary so profile fields persist without weakening publisher ownership; a frontend-only catch would still leave the data unsaved.
