## Label Justifications

- `P2`: The report describes a concrete install/runtime integration gap with a workaround, but it needs product and security review rather than emergency handling.
- `impact:security`: Automatic hook registration would affect executable plugin code, user consent, and the ClawHub/OpenClaw trust boundary.

## Summary

Keep open. Current main still treats `clawhub install` as skill file extraction plus ClawHub metadata writes, while automatic OpenClaw hook registration would add a security-sensitive runtime handoff that needs maintainer policy first.

## What This Changes

The issue requests automatic registration of OpenClaw hook/plugin directories when `clawhub install` installs a skill that bundles `hooks/openclaw`.

## Best Possible Solution

Define a maintainer-approved hook registration contract first, then implement explicit detection, disclosure, consent, registration, docs, and tests in the chosen ClawHub/OpenClaw install path.

## Reproduction Assessment

Yes. Source inspection shows `cmdInstall` extracts the archive and writes origin/lock metadata without hook discovery or an OpenClaw plugin registration call; I did not run the command because this review is read-only.

## Solution Assessment

Unclear as proposed. Automatic registration would solve the symptom, but the maintainable path needs explicit maintainer policy for consent, scanning, disclosure, and OpenClaw gateway handoff.
