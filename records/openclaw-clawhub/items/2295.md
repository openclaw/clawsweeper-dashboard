## Summary

Current main still reproduces the core gap: skill installation copies and records the bundle but does not register embedded gateway code. The issue should remain open because automatic activation crosses the explicit skill/plugin trust boundary and needs a coordinated product and security decision rather than a mechanical CLI call.

## What This Changes

Add installation-time handling for skills that contain OpenClaw hooks so users are not left with an apparently installed but inactive hook-dependent skill.

## Best Possible Solution

Introduce an explicit, scanned companion-plugin declaration for hook-dependent skills, then prompt for informed installation consent and otherwise emit precise manual activation instructions; coordinate the contract with https://github.com/openclaw/clawhub/issues/707.

## Reproduction Assessment

Yes at source level: current main’s skill installer extracts and records the skill without any plugin-registration call, while the documented plugin flow is separate. This read-only review did not run a live gateway installation.

## Solution Assessment

No, the proposed unconditional `openclaw plugins install` call is not the best solution because it bypasses the current skill/plugin trust boundary; explicit declaration, scanning, consent, and lifecycle handling are required.
