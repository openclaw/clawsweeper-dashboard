## Label Justifications

- `P2`: Hook-dependent skills can appear installed while remaining inactive, but the affected surface is limited to skills that bundle OpenClaw hooks.
- `impact:security`: Registering the bundled hook activates executable skill-supplied code across an installation trust boundary.
- `impact:session-state`: The reported `agent:bootstrap` hook changes session initialization, so missing registration can leave expected agent context absent.

## Summary

Current main still installs skill files without registering bundled OpenClaw hooks. The report remains necessary, but automatic activation crosses an unresolved executable-code trust boundary also covered by the open hook-security and post-install lifecycle discussions.

## What This Changes

Add installation handling for skills that bundle OpenClaw hooks so users are not left with an apparently installed but inactive skill.

## Best Possible Solution

Define a declared and scanned hook capability that is visible before installation, requires explicit user approval, and has consistent install, update, uninstall, restart, and rollback semantics.

## Reproduction Assessment

Yes at the source level: current main copies and records the skill but contains no hook discovery or plugin-registration path, matching the reported CLI observation. This read-only review did not run a live OpenClaw gateway reproduction.

## Solution Assessment

No, unconditional automatic registration is not yet the best solution. The narrower maintainable direction is consent-gated activation after hook declaration, security scanning, transparency, and lifecycle behavior are defined.
