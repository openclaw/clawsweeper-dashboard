## Label Justifications

- `P2`: The issue reports a meaningful installer/setup reliability gap for hook-bearing skills, but it is bounded and needs product/security direction before implementation.
- `impact:security`: Automatic registration would cause skill-provided hook code to become executable OpenClaw behavior.
- `impact:session-state`: A hook-dependent skill can appear installed while the active agent/session never receives the expected lifecycle behavior.

## Summary

Keep open. Current main and v0.23.0 still install skill files and ClawHub metadata only; there is no hook-bundle discovery or OpenClaw gateway registration, and the remaining work is a product/security boundary rather than a stale cleanup target.

## What This Changes

The issue asks for `clawhub install` to detect skill hook bundles and automatically register them with the OpenClaw gateway.

## Best Possible Solution

Define a consent-aware hook capability contract, capture the security-sensitive intent in specs, then implement discovery, disclosure, scanner coverage, install/update registration, docs, and real OpenClaw gateway validation together.

## Reproduction Assessment

Yes, at source level. Current `cmdInstall` and `cmdUpdate` install files and metadata without hook discovery, gateway handoff, or plugin registration; I did not run a live OpenClaw gateway because this read-only source path is clear.

## Solution Assessment

Unclear as proposed. Automatic registration addresses the activation gap, but the maintainable solution should not be an unconditional hidden shell-out; it needs explicit consent, hook scanning/disclosure, and a ClawHub/OpenClaw gateway handoff policy first.
