## Label Justifications

## Summary

Current main still copies hook-bearing skills without registering their OpenClaw plugins, so the reported incomplete-install behavior remains. Keep this open for a maintainer decision because automatic activation would introduce a new executable-code trust boundary that the existing hooks and plugin plans have not settled.

## What This Changes

Add installation-time detection and activation of OpenClaw hooks bundled inside a ClawHub skill.

## Best Possible Solution

Define a typed hook or plugin declaration in the package format, scan and display its capabilities, require explicit consent for executable activation, and let OpenClaw register it through a supported native install contract with clear update, rollback, and uninstall semantics.

## Reproduction Assessment

Yes at source level: current main's `cmdInstall` extracts and records a skill but contains no hook discovery or plugin-registration path, matching the reported result. A live gateway run was not performed, but the missing integration path is explicit in the implementation.

## Solution Assessment

No, not exactly as proposed: unconditionally invoking `openclaw plugins install` would fix the immediate symptom but bypass the repository's planned family separation, trust disclosure, warning, and operator-consent model.
