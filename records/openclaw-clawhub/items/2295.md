## Label Justifications

- `P2`: This is a meaningful but bounded install-completeness and UX gap that requires design work rather than emergency response.
- `impact:security`: The requested behavior would automatically activate executable third-party hook code during an otherwise ordinary skill install.
- `impact:session-state`: The missing registration prevents intended bootstrap hooks from affecting agent initialization and session context.

## Summary

Current main still installs skills as files and records their origin without registering bundled OpenClaw hook code. The report remains necessary, but unconditional activation would cross the repository’s explicit executable-plugin consent boundary; the related hooks security RFC should settle scanning, disclosure, and permission policy before implementation.

## What This Changes

Extend `clawhub install` so a skill containing an OpenClaw hook is detected and activated rather than appearing installed while its hook remains inert.

## Best Possible Solution

Define one permissioned hook lifecycle with https://github.com/openclaw/clawhub/issues/707: detect executable hooks, scan and display their capabilities, require explicit operator consent, delegate activation to supported OpenClaw plugin machinery, and track update, uninstall, restart, and failure state.

## Reproduction Assessment

Yes, at source level with high confidence: current `cmdInstall` downloads and extracts the skill and records installation state, while no bundled-hook registration path exists. A live gateway reproduction was not run because the read-only review establishes the missing behavior from the current implementation and documented command boundary.

## Solution Assessment

No, silently running the equivalent plugin-install command is not the best solution because it bypasses the planned executable-code trust boundary. A capability-aware, consented handoff integrated with scanning and lifecycle tracking is safer and more maintainable.
