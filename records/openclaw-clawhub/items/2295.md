## Summary

Current main still installs skills as file bundles without registering nested OpenClaw hooks. Keep this open because automatic activation would cross the existing executable-code trust boundary, and the declaration, scanning, consent, and OpenClaw handoff contract remains an unresolved product decision tracked by related issues.

## What This Changes

The issue requests that `clawhub install` detect a bundled `hooks/openclaw` directory and automatically register it as an OpenClaw plugin.

## Best Possible Solution

Keep skill installation file-only until maintainers define a declared hook/plugin capability with scanning, visible permissions, explicit confirmation, and a supported OpenClaw activation or session-handoff contract.

## Reproduction Assessment

Yes, at source level: current main installs and tracks skill files but exposes plugin activation as a separate command, so installing a skill alone does not register its nested OpenClaw hook. A live gateway run was not necessary to establish this implementation boundary.

## Solution Assessment

No, unconditional automatic `openclaw plugins install` is not the best solution because it bypasses the current executable-code consent boundary. The declaration, scanning, permission UX, and OpenClaw handoff should be designed together with the related RFCs.
