## Label Justifications

- `P2`: Hook-dependent skills can appear installed while remaining inactive, but a manual registration workaround exists and final behavior requires policy design.
- `impact:security`: The requested automatic behavior would activate third-party executable hook code as part of installation.
- `impact:session-state`: The reported `agent:bootstrap` hook changes agent initialization and can leave expected session behavior inactive.

## Summary

Current main still installs skills as files and does not register nested OpenClaw hooks. The requested activation remains a valid setup gap, but it crosses the deliberate skill/code-plugin trust boundary and overlaps unresolved consent, scanning, disclosure, and lifecycle decisions, so it needs maintainer product and security direction rather than a narrow automated fix.

## What This Changes

Make skill installation detect bundled OpenClaw hooks and activate them so hook-dependent features work after installation.

## Best Possible Solution

Define one consent-gated and scanner-aware executable-hook contract, surface requested capabilities before activation, and manage install, update, uninstall, restart, and failure state without changing ordinary file-only skill installs.

## Reproduction Assessment

Yes at source level. Current main extracts and records skill files without any hook discovery or plugin-registration call, while executable plugins use a separate install surface; this read-only review did not execute the reporter's local OpenClaw setup.

## Solution Assessment

No, silently running the equivalent of `openclaw plugins install` is not yet the best solution. A maintainable fix must first settle artifact identity, security scanning, capability disclosure, consent, and lifecycle behavior with the related hook RFC.
