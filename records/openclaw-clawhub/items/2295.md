## Label Justifications

- `P2`: The missing activation silently leaves a bounded class of hook-dependent skills incomplete, but it does not make ClawHub generally unusable.
- `impact:security`: The requested behavior would activate arbitrary executable code distributed inside a skill bundle.
- `impact:session-state`: The reported `agent:bootstrap` hook is intended to alter session initialization and currently remains inactive.

## Summary

Current main still copies skill bundles without activating embedded OpenClaw hook plugins, but automatic activation would cross an executable-code trust boundary that requires an explicit maintainer-approved consent and lifecycle design.

## What This Changes

The issue requests automatic detection and registration of an embedded OpenClaw hook plugin during skill installation.

## Best Possible Solution

Define a security-reviewed hook manifest and lifecycle that discloses and scans executable hooks before installation, requests explicit approval, and reliably registers, updates, rolls back, and unregisters them with the owning skill.

## Reproduction Assessment

Yes at source level: current main completes skill file and metadata installation without any plugin-registration call, which directly explains why `openclaw hooks list` does not gain the bundled hook. This read-only review did not run a live gateway reproduction.

## Solution Assessment

No, unconditional automatic registration is not the best solution because it hides executable activation inside an otherwise declarative skill install. The safer path is explicit disclosure and approval with complete update and uninstall semantics.
