## Summary

This issue should stay open: current `clawhub install` only installs skill files and metadata, while the requested automatic OpenClaw hook registration crosses the skill/plugin execution boundary and needs an explicit maintainer product/security decision.

## What This Changes

Requests that `clawhub install` detect OpenClaw hook files in an installed skill and register the hook with the OpenClaw gateway automatically.

## Best Possible Solution

Define a consent-gated OpenClaw-native install/setup path for hook-bearing skills, or make `clawhub install` detect hook payloads and print an explicit follow-up command instead of silently auto-registering them.

## Reproduction Assessment

Yes, this is source-reproducible: the current `cmdInstall` path downloads, extracts, hashes, and records metadata without any hook registration step. I did not run a live OpenClaw gateway, so the exact runtime symptom remains unverified here.

## Solution Assessment

Unclear: automatic registration from `clawhub install` is not obviously the safest solution because current docs and specs keep OpenClaw execution explicit. A maintainer should choose between native OpenClaw install support, an opt-in registration flag/prompt, or a warning with exact manual steps.
