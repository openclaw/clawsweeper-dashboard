## Label Justifications

## Summary

Current main still installs skill files without registering bundled OpenClaw hooks, so the reported incomplete-install behavior remains. The issue should stay open because safe activation requires an explicit product and security decision coordinated with the broader hook-format RFC.

## What This Changes

Add installation behavior that detects bundled OpenClaw hooks and registers them so hook-dependent skills become active after installation.

## Best Possible Solution

Define one consented install flow that detects bundled OpenClaw hooks, presents their executable permissions and scanner status, registers approved hooks through a supported OpenClaw interface, reports restart or rollback requirements, and records the trust-boundary rationale in a focused spec.

## Reproduction Assessment

Yes at source level: current main has a concrete install path that copies and records the skill without any hook or plugin registration step, directly matching the reported command-level symptom. A live OpenClaw gateway was not needed to establish that missing path.

## Solution Assessment

No, not as proposed: silently invoking plugin installation is not the safest maintainable solution. Capability discovery, scanner coverage, explicit consent, activation status, restart handling, and rollback should be designed as one contract.
