## Label Justifications

- `P2`: The issue describes a concrete install/runtime integration gap with a workaround, but the fix needs product and security review rather than emergency handling.
- `impact:security`: Automatic hook registration affects executable plugin code, user consent, and the ClawHub/OpenClaw trust boundary.

## Summary

Keep open. Current main still installs skills as local files plus ClawHub metadata without detecting or registering `hooks/openclaw`, and the requested automatic registration crosses a plugin code-execution/session boundary that needs maintainer product and security policy first.

## What This Changes

Request that `clawhub install` detect skill-bundled OpenClaw hooks and register them with the gateway automatically after install.

## Best Possible Solution

Define the hook registration contract first, then implement explicit detection, disclosure, consent or opt-in policy, gateway registration, docs, and tests in the chosen ClawHub/OpenClaw install path.

## Reproduction Assessment

Yes, from source inspection: `cmdInstall` extracts the archive and writes origin/lock metadata without hook discovery or an OpenClaw plugin registration call. I did not execute the install command because this review is read-only.

## Solution Assessment

Unclear as proposed. Automatic registration would solve the symptom, but the maintainable path needs explicit maintainer policy for consent, scanning, disclosure, and the OpenClaw gateway handoff contract.
