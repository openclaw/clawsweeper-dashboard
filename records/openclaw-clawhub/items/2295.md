## Summary

Keep open. Current main still treats `clawhub install` as a skill file install, while gateway extension registration remains an explicit OpenClaw plugin-install path and the overlapping hook-security work is still open.

## What This Changes

The issue asks `clawhub install` to detect OpenClaw hook files bundled in a skill and register those hooks with the gateway automatically.

## Best Possible Solution

Add a policy-backed hook installation flow with explicit capability visibility, security scanning, and user/operator confirmation, likely through the OpenClaw install path rather than silent registration by `clawhub install`.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: install a skill with `hooks/openclaw`, and current `cmdInstall` only downloads/extracts/writes lock metadata without registering hooks. I did not run the CLI because this was a read-only review.

## Solution Assessment

No, automatic registration as stated is not yet the best safe fix. The maintainable path needs an explicit hook security and UX policy before implementation.
