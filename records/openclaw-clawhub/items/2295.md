## Label Justifications

## Summary

Current main still installs skill files without registering bundled OpenClaw hooks, so the reported activation gap remains. Keep this open for a maintainer product and security decision because silent registration would enable third-party executable behavior, while related open work already covers hook disclosure/scanning and broader post-install activation.

## What This Changes

Request that `clawhub install` detect bundled OpenClaw hooks and register them so hook-dependent skill behavior activates after installation.

## Best Possible Solution

Create one explicit hook-capability lifecycle that scans and displays declared hooks, obtains informed permission, registers through a stable OpenClaw interface, reports activation or restart status, and safely updates or removes registrations with the skill.

## Reproduction Assessment

Yes, at source level with high confidence: current `cmdInstall` completes file installation and lockfile recording without any hook or plugin registration step. A live gateway run was not performed, but the missing integration is explicit in the complete install path.

## Solution Assessment

No, unconditional automatic `openclaw plugins install` execution is not the best solution because it hides an executable-code decision and omits lifecycle safety. The narrower maintainable direction is an explicit, scanner-backed consent and registration contract shared with the related hook-security work.

## Risks / Open Questions

- Unconditional registration could silently enable third-party executable hooks and restart-sensitive runtime behavior without informed user consent.
- The design must coordinate with https://github.com/openclaw/clawhub/issues/707 so hook scanning and disclosure exist before installation begins activating code.
- Registration also needs defined update, uninstall, rollback, gateway-restart, non-interactive, and failure semantics; implementing only the install subprocess could leave stale or active hook state.
- Directly shelling out to `openclaw plugins install` would create a cross-project CLI/runtime dependency that needs an explicit supported contract.
