## Label Justifications

- `P2`: Hook-dependent skills can appear installed while remaining inactive, but the affected capability is optional and requires bounded cross-project design work.
- `impact:security`: The requested automatic registration would activate executable code supplied by an installed skill bundle.
- `impact:session-state`: The missing `agent:bootstrap` hook prevents the intended initialization of agent-session state.

## Summary

The requested behavior remains absent from current main, but it requires a deliberate product and security contract rather than a narrow bug fix: ClawHub currently installs skill files without activating nested executable extensions, while repository policy requires risky setup actions to remain explicit and operator-controlled.

## What This Changes

Add a post-install path that detects bundled OpenClaw hooks and makes them available for gateway registration after a skill install.

## Best Possible Solution

Define a declared and scanned hook capability with prominent pre-install disclosure, then offer an explicit confirmation-driven OpenClaw activation step instead of silently registering executable code.

## Reproduction Assessment

Yes at the source level: current main extracts the skill and records the installation but contains no plugin or gateway registration call, matching the reported outcome. A live gateway run was not performed, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

No; unconditional automatic plugin installation is not the safest or narrowest solution because it would bypass the repository's explicit operator-control boundary. The hook manifest, scanning, disclosure, consent, and OpenClaw handoff contract should be settled first.
