## Summary

Current main still installs skill files and lock metadata without detecting or registering bundled OpenClaw hooks. The report is valid, but automatic registration would create a new code-execution and session-influence boundary, so this needs a coordinated product and security decision rather than conservative cleanup or an automated fix.

## What This Changes

Detect OpenClaw hooks bundled in an installed skill and activate them automatically so hook-dependent behavior works after `clawhub install`.

## Best Possible Solution

Define a declared and scan-visible hook capability, show its permissions before installation, require explicit operator consent for activation, and make install, update, uninstall, rollback, and gateway reload reconcile the registered plugin state consistently.

## Reproduction Assessment

Yes, at high confidence from source inspection: current main’s `cmdInstall` extracts skill files and records installation metadata but has no path that discovers or registers `hooks/openclaw`. The exact gateway runtime was not executed because the missing call is directly observable in source.

## Solution Assessment

No, unconditional automatic `openclaw plugins install` is not yet the best solution. The maintainable path needs an approved hook declaration, scanning and disclosure model, explicit consent, and lifecycle reconciliation rather than silently escalating a passive skill install into code activation.

## Risks / Open Questions

- Automatically registering `hooks/openclaw` would turn a currently passive skill-file install into executable plugin activation, potentially granting unreviewed code persistent lifecycle access without informed consent.
- A complete design must reconcile install, update, pin, uninstall, failure rollback, gateway restart/reload, and stale-registration behavior; handling only first install could leave code and recorded state out of sync.
- The ClawHub CLI cannot safely assume every installed skill targets an available or compatible OpenClaw gateway, so non-interactive and non-OpenClaw workflows need a defined compatibility path.
