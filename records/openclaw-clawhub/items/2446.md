## Summary

Keep open: current main still rejects username-valued GitHub `providerAccountId` rows before publish, and the existing related report remains open for the same auth-provider failure. The current source intentionally uses immutable numeric GitHub IDs, so the remaining work is a maintainer-approved repair path for affected stored auth rows, not an auto-close cleanup.

## What This Changes

The issue reports that skill publishing fails when Convex Auth has stored the GitHub username in `authAccounts.providerAccountId` instead of the numeric GitHub account id expected by the publish gate.

## Best Possible Solution

Add a maintainer-approved repair or migration path that derives and stores the numeric GitHub id for affected auth account records while keeping publish-time account-age checks based on immutable numeric ids.

## Reproduction Assessment

Yes, as a source-level reproduction: use a user with no cached `githubCreatedAt` and an `authAccounts.providerAccountId` value like `stokkan-6ith`, then publish or call `requireGitHubAccountAge`; current main throws `GitHub account lookup failed` before fetching GitHub. I did not mutate a live database row in this read-only review.

## Solution Assessment

No, the issue's broad username-fallback option is not clearly the best fix because the code and spec intentionally require immutable numeric GitHub ids. The safer direction is a data repair or controlled migration that preserves the numeric-id publish gate.
