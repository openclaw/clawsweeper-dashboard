## Summary

Keep open: current production still shows npm-pack releases whose artifact metadata says the tarball has real implementation files while the public file/review surfaces only expose the two metadata files; current main fixes new ClawPack uploads but does not repair existing truncated rows or stale suspicious review verdicts.

## What This Changes

The issue requests that ClawPack/npm-pack package inspect and LLM review surfaces evaluate the actual artifact entries instead of only metadata files, so complete code plugins are not falsely flagged as missing implementation code.

## Best Possible Solution

Add a cursor-paginated repair that parses stored ClawPack tarballs, restores missing release.files entries for existing npm-pack rows, requeues stale package scans, and updates specs/tests so inspect, raw-file, compatibility ZIP, and review surfaces agree with artifact truth.

Yes. Public API checks reproduce the mismatch for two npm-pack releases, and current-main source shows detail, raw-file, compatibility ZIP, and package LLM surfaces still depend on persisted release.files for existing rows.

## Solution Assessment
