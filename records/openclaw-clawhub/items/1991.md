## Summary

Keep this open: current main appears to fix the publish path for newly uploaded ClawPacks by storing parsed tarball entries, but the reported historical releases still need the unmerged repair owned by https://github.com/openclaw/clawhub/pull/2583.

## What This Changes

The issue asks ClawHub to stop evaluating npm-pack ClawPack code-plugin releases from metadata-only `release.files` rows when the stored artifact contains implementation files.

## Best Possible Solution

Keep the new full-entry publish path and land the linked guarded historical repair after real Convex dry-run, targeted write, post-state, rescan, and idempotency proof.

## Reproduction Assessment

Yes. The reporter supplied exact live `clawhub package inspect`, `download`, and `verify` evidence, and source inspection confirms those stored release rows are what the LLM review reads.

## Solution Assessment

Yes in direction. Persisting artifact-derived file rows for new publishes plus a guarded historical migration is narrower than changing only the evaluator, because inspect/file surfaces also depend on the stored rows.
