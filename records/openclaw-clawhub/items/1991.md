## Summary

Keep open: current main partially addresses new ClawPack publishes, but historical npm-pack releases whose stored `release.files` only has metadata still feed incomplete data to package inspect, file lookup, static scan, and LLM review surfaces.

## What This Changes

Request ClawHub to have npm-pack/ClawPack code-plugin reviews inspect actual artifact entries instead of only stored metadata files.

## Best Possible Solution

Make package detail, file lookup, static scan, and LLM review share one consistent ClawPack file inventory for npm-pack releases, then refresh affected historical moderation records safely.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make detail, file lookup, CLI inspect, static scan, and LLM review see only those stored metadata entries.

## Solution Assessment

Partially. The current new-publish extraction path is a useful building block, but the issue needs historical release handling and an explicit moderation rescan policy before it is fully solved.
