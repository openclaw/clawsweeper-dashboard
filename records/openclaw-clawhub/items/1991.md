## Summary

Keep open: current main extracts ClawPack entries for new publishes, but historical npm-pack rows with metadata-only `release.files` still feed package detail, file lookup, static scan, and LLM review surfaces.

## What This Changes

The issue asks ClawHub to make npm-pack/ClawPack package inspection and LLM review use actual tarball entries rather than metadata-only `release.files` so complete code-plugin releases are not falsely flagged as missing implementation files.

## Best Possible Solution

Add a production-safe ClawPack artifact reparse/backfill or fallback-read path for historical releases, refresh affected static/LLM moderation, and align the security docs with the final scan scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: any existing npm-pack release whose stored `release.files` contains only metadata will still make package inspect, file lookup, static scan, and LLM review see only those metadata files.

## Solution Assessment

Partially. The current future-publish extraction is the right narrow code path for new releases, but it is not the full fix until historical artifacts are reparsed or the affected surfaces read selected files from the stored tarball and stale scans are refreshed.
