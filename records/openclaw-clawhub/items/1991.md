## Summary

Keep open: current main fixed new ClawPack publishes to store extracted file metadata, but live affected npm-pack releases still expose only two metadata files to version detail/raw-file/legacy LLM surfaces and no current repair backfill restores those persisted rows.

## What This Changes

The issue asks ClawHub to stop treating complete ClawPack npm-pack code-plugin artifacts as missing implementation files when package inspection and LLM review read truncated release file metadata.

## Best Possible Solution

Add a cursor-paginated admin repair that parses stored ClawPack tarballs, restores missing release.files entries, requeues stale package scan results, and updates tests/specs so inspect, raw-file, and review surfaces agree with artifact truth.

## Reproduction Assessment

Yes. Public API checks reproduce the mismatch for two npm-pack releases, and current main still drives the affected detail, raw-file, and legacy LLM surfaces from persisted release.files.

## Solution Assessment

Yes. Current main already stores full ClawPack entries for new uploads and the scan worker can inspect the tarball, so the narrow maintainable fix is a backfill/repair plus scan requeue for existing rows.
