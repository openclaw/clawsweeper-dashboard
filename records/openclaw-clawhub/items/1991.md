## Summary

Keep open: current main appears to fix new ClawPack publishes, but historical npm-pack releases with metadata-only `release.files` still flow through package detail, file lookup, static scan, and LLM review surfaces unchanged. ClawHub cleanup policy also keeps issues open rather than auto-closing them.

## What This Changes

The issue asks ClawHub to make ClawPack/npm-pack package inspection and LLM review use the actual tarball entries instead of only registry metadata files.

## Best Possible Solution

Add a cursor-based repair that reparses stored npm-pack tarballs into `release.files`, re-run affected static/LLM scans, and update security docs to describe the chosen scan scope.

## Reproduction Assessment

Yes, by source inspection and the reporter's live examples. Any npm-pack release whose `release.files` contains only metadata files will make package detail, `/file`, static scan, and LLM review omit artifact README/source/runtime entries because those paths all read `release.files`.

## Solution Assessment

Partially. Storing extracted entries for new ClawPack publishes is the right narrow code direction, but historical releases still need a backfill and re-evaluation decision before the issue is fully solved.
