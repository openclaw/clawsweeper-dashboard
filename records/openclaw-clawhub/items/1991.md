## Summary

Keep open: current main improves future ClawPack publishes, but historical npm-pack releases with metadata-only `release.files` can still feed incomplete data to inspect, file lookup, static scan, and LLM review; ClawHub cleanup policy also keeps issues open.

## What This Changes

The issue reports that npm-pack ClawPack releases expose only metadata files to package inspect and LLM/static review even when the downloadable artifact contains implementation files.

## Best Possible Solution

Centralize a canonical ClawPack file inventory for API inspect, file lookup, static scan, and LLM review, then run a maintainer-approved backfill/rescan for affected historical releases and update the security docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make inspect, file lookup, static scan, and LLM review see only those metadata entries.

## Solution Assessment

Unclear. Current main’s new-publish extraction is a useful partial fix, but the best complete solution depends on maintainer policy for historical file backfill versus tarball-backed lookup and moderation rescans.
