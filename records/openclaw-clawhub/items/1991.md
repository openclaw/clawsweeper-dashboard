## Summary

Keep open: current main fixes the future publish path by storing parsed ClawPack entries in release.files, but existing metadata-only npm-pack rows can still feed package detail/file APIs and LLM/static scans with only the old limited file list, and the scan-scope docs still need maintainer alignment.

## What This Changes

The issue reports that ClawPack/npm-pack releases are downloadable and verifiable with full tarball contents, but package detail/file APIs and the LLM review only see metadata files from release.files.

## Best Possible Solution

Add a maintainer-approved repair path for existing npm-pack releases that repopulates file metadata from stored ClawPack artifacts or explicitly documents that only future publishes get full file metadata, then rerun affected evaluations as appropriate.

## Reproduction Assessment

Yes. For affected existing rows, the reporter provided concrete live package examples and current source shows package detail, file lookup, and package LLM evaluation still depend on release.files; newly published ClawPacks on current main appear fixed.

## Solution Assessment

Partially. The current-main publish change is the right narrow fix for future releases, but the issue is not fully solved until old metadata-only rows are repaired or maintainers explicitly choose and document a forward-only policy.
