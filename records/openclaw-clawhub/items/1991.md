## Summary

Keep open: this is a valid, security-sensitive moderation bug on current main. New ClawPack publishes appear to store extracted artifact entries now, but historical npm-pack releases with metadata-only release.files still leave detail/file/static scan/LLM review paths depending on incomplete stored inventories.

## What This Changes

The issue reports that npm-pack ClawPack code-plugin releases can have complete downloadable artifacts while registry inspection and LLM review only see package.json and openclaw.plugin.json.

## Best Possible Solution

Use a single artifact-derived ClawPack inventory for package detail, raw file lookup, static scan, and LLM review, then run an explicit historical inventory repair and moderation rescan plan for affected npm-pack releases.

## Reproduction Assessment

Yes. Source inspection on current main shows the reported dependency chain: affected historical rows with metadata-only release.files would feed incomplete data to inspect, raw file lookup, static scan, and LLM review paths.

## Solution Assessment

Unclear. The new-publish extraction path is the right base, but the remaining historical repair needs maintainer choice between backfilling stored inventories, artifact-backed on-demand reads, and how to refresh moderation outcomes.
