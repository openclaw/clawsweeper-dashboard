## Summary

Keep open: current main appears to fix the new-publish path by storing parsed ClawPack entries in `release.files`, but existing npm-pack rows can still have metadata-only `release.files`, and the package detail, file, static scan, and LLM review paths still depend on that field.

## What This Changes

The issue asks package detail/file/LLM review surfaces for npm-pack ClawPack releases to evaluate the actual parsed artifact entries instead of only package metadata files.

## Best Possible Solution

Backfill or reparse existing ClawPack artifacts into `release.files`, refresh affected package scan verdicts, and keep the docs aligned with the chosen scan scope.

## Reproduction Assessment

Yes. A package release with `artifactKind: "npm-pack"`, a stored ClawPack, and metadata-only `release.files` will still reproduce the limited detail/file/LLM review surface because those paths read `release.files` rather than reparsing the artifact.

## Solution Assessment

Partially. The current-main new-publish extraction is the right narrow source fix for future uploads, but the issue is not fully solved until maintainers choose how to repair historical rows and refresh scan results.
