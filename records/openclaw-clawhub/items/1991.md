## Summary

Keep open: current main now stores extracted ClawPack entries for new uploads, but the API, CLI inspect/file paths, static scan refresh, and LLM review still depend on stored `release.files`, so existing metadata-only npm-pack releases remain affected.

## What This Changes

The issue asks ClawHub package inspect/file and LLM review surfaces to evaluate actual ClawPack/npm-pack artifact entries instead of metadata-only release records.

## Best Possible Solution

Choose a production-safe artifact reparse or tarball-backed lookup strategy, refresh affected static/LLM signals, and add regression coverage for historical metadata-only npm-pack releases.

## Reproduction Assessment

Yes, from source inspection. Any npm-pack release with a stored ClawPack artifact but metadata-only `release.files` will make inspect, file lookup, static scan refresh, and LLM review see only those stored metadata entries.

## Solution Assessment

Unclear. Persisting parsed entries for new uploads is a good building block, but the best complete fix depends on maintainer choice around historical artifact reparsing, API lookup behavior, and scan refresh policy.
