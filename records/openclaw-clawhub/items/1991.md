## Summary

Keep open: the live mismatch still reproduces, and current main still routes package detail/raw-file, static scan, and LLM review context through `release.files`. New publishes now store extracted ClawPack entries, but the current moderation spec says ClawPack static/LLM inputs are intentionally metadata-only for now, so changing historical review behavior needs maintainer policy rather than cleanup closure.

## What This Changes

The issue asks ClawHub to make npm-pack ClawPack package review use actual artifact entries, or clearly label metadata-only registry context, so complete code-plugin releases are not flagged as missing implementation files.

## Best Possible Solution

Choose the approved ClawPack review source of truth, then either backfill and rescan historical npm-pack inventories or make metadata-only APIs and prompts explicitly say they are not reviewing full tarball contents.

## Reproduction Assessment

Yes. Live public API checks reproduce the mismatch for `@openclaw/codex-claw@2026.5.6`, and current-main source shows file lookup, static scan, and LLM evaluation still consuming `release.files`.

## Solution Assessment

Unclear. Storing extracted entries for new uploads is useful, but the remaining change needs a maintainer decision between preserving metadata-only scope, backfilling and rescanning historical releases, or reading artifact entries on demand.
