## Summary

Keep open: current main partially fixes new ClawPack uploads, but the reported historical npm-pack failure mode remains source-reproducible because package detail, file lookup, static scan, and LLM review still depend on stored `release.files`.

## What This Changes

The issue requests ClawPack package inspect, file lookup, static scan, and LLM review to evaluate actual npm-pack artifact entries instead of metadata-only release files.

## Best Possible Solution

Make package detail, file lookup, static scan, and LLM review share one consistent ClawPack file inventory for npm-pack releases, then safely refresh affected historical moderation records.

## Reproduction Assessment

Yes. Source inspection shows that a ClawPack release with `clawpackStorageId` but metadata-only `release.files` will expose only those stored paths to inspect/file lookup, static scan, and LLM review; the issue comments provide live registry examples.

## Solution Assessment

Unclear. Extracting entries for new uploads is the right building block, but the complete fix needs a maintainer decision on historical backfill versus tarball-backed reads and how to replace completed moderation results.
