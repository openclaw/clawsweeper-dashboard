## Summary

Keep open: current main fixes future multipart ClawPack publishes, but metadata-only npm-pack releases still make package detail, file lookup, static scan, and LLM review consume an incomplete `release.files` surface.

## What This Changes

The issue reports that npm-pack ClawPack releases expose only metadata files to package inspect and LLM review even though the stored tarball contains implementation files.

## Best Possible Solution

Make package detail, file lookup, static scan, and LLM review share one consistent ClawPack file inventory, then safely refresh affected metadata-only npm-pack releases and moderation records.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a metadata-only npm-pack release with `clawpackStorageId` and full artifact metadata still drives inspect, file lookup, static scan, and LLM review from `release.files`; live registry calls reproduce that shape for two npm-pack releases.

## Solution Assessment

Unclear as a maintainer decision. The new-publish extraction fix is useful but incomplete; the durable solution needs a chosen historical repair and moderation rescan policy.
