## Summary

Keep open: this issue still reproduces on the live package surfaces, and ClawHub cleanup policy does not auto-close issues. Current main has mitigations for new uploads and false-positive review wording, but existing npm-pack rows still expose metadata-only inventories and stale suspicious LLM summaries until maintainers choose a backfill/rescan or scan-scope policy.

## What This Changes

The issue reports that ClawPack npm-pack code-plugin releases can be marked suspicious because package detail/file and LLM review paths see only package.json and openclaw.plugin.json instead of the full tarball contents.

## Best Possible Solution

Define the authoritative ClawPack review input for historical npm-pack releases, then either backfill and rescan affected package inventories or make metadata-only APIs and LLM summaries explicitly describe their limited review scope.

## Reproduction Assessment

Yes. Live API checks reproduce the metadata-only file inventory and stale missing-code LLM summaries, and current-main source shows historical rows are still consumed through release.files.

## Solution Assessment

Unclear. The new-upload storage fix and prompt mitigations are useful, but the remaining historical verdict problem needs a maintainer decision between backfill/rescan, artifact-on-demand review, or explicit metadata-only wording.
