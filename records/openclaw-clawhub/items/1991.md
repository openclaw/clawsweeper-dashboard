## Summary

Keep open: ClawHub policy keeps issues open, and source inspection shows the report remains actionable for historical npm-pack releases even though current main now stores extracted files for new multipart ClawPack publishes.

## What This Changes

Report that npm-pack/ClawPack package inspection and moderation review can use metadata-only `release.files` instead of actual tarball entries, producing false missing-code verdicts.

## Best Possible Solution

Unify npm-pack releases around one artifact-derived file inventory for detail, file lookup, inspect, static scan, and LLM review, then deliberately backfill and rescan affected historical releases.

## Reproduction Assessment

Yes by source inspection. An npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make version detail, `/file`, CLI inspect, static rescan, and LLM review miss README/source/runtime entries in the stored tarball.

## Solution Assessment

Partially. Current main's new-publish extraction path is a good building block, but the complete fix needs a maintainer-chosen historical storage or tarball-reader repair plus moderation rescan policy.
