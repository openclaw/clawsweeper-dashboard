Keep open: current main still has the reported ClawPack review blind spot, and ClawHub policy keeps issues open unless a merged fix resolves them. The upload path stores only metadata files for npm-pack releases, while package inspect/file and LLM review surfaces still read from that metadata-only `release.files` list.

## What This Changes

The issue reports that npm-pack ClawPack releases expose only metadata files to package inspect and LLM review even though the downloadable tarball contains implementation files.

## Best Possible Solution

Keep ClawPack artifact storage/download behavior intact, but make package inspect/file and LLM review surfaces explicitly distinguish metadata-only registry files from downloadable artifact contents and provide enough artifact facts or selected parsed entries to avoid false missing-code verdicts.

## Repair Work Prompt

Fix the ClawPack/npm-pack metadata-only review blind spot for package releases. Current main stores only `package.json` and `openclaw.plugin.json` in `release.files` for multipart ClawPack uploads, then package inspect/file and `evaluatePackageReleaseWithLlm` treat that list as the review/file surface, which can make complete tarballs look hollow. Keep artifact storage and download semantics intact. Add a focused fix that clearly distinguishes registry metadata files from actual ClawPack artifact contents and passes artifact facts such as `artifactKind`, `npmFileCount`, `npmUnpackedSize`, SHA/integrity, and/or selected parsed tarball entries into the package LLM/detail/file surface so complete npm-pack code plugins are not flagged solely for missing implementation files. Do not introduce a broad full-tarball scan unless maintainers explicitly choose that scope. Add regression coverage around a ClawPack containing `dist/index.js` and README/source entries where the LLM/package context no longer claims the package has only metadata without artifact qualification.
