## Summary

Keep open: current main has a future-publish extraction fix, but the reported historical ClawPack path is still valid because package detail, file lookup, static dataset export, and LLM review all consume the stored `release.files` surface.

## What This Changes

The issue asks ClawHub to stop treating complete npm-pack ClawPack releases as metadata-only inputs for package inspect and LLM review.

## Best Possible Solution

Choose a production-safe artifact reparse/backfill plan, refresh affected moderation signals, and update the API/security docs to match the final ClawPack scan and file-inspection scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: any npm-pack release with a stored ClawPack artifact but metadata-only `release.files` will still make inspect, file lookup, and LLM review see only those stored metadata entries.

## Solution Assessment

Partially. The current-main extraction path is the right narrow fix for future publishes, but it is not the complete solution until historical artifacts are reparsed or the affected surfaces read selected files from the stored tarball and the stale scans are refreshed.
