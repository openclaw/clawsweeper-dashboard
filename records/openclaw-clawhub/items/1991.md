## Summary

Keep open: the live registry still reproduces the ClawPack inventory gap for historical npm-pack releases, while current main only fixes parsed-file storage for new uploads and the spec still treats full-tarball LLM scan scope as a moderation policy decision.

## What This Changes

The issue asks ClawHub package detail/file and LLM review surfaces to distinguish metadata-only `release.files` from actual npm-pack ClawPack contents so complete code-plugin artifacts are not flagged as missing code.

## Best Possible Solution

Define the authoritative ClawPack inventory and scan scope, then either backfill and rescan affected releases or explicitly label historical file/detail and LLM surfaces as metadata-only.

## Reproduction Assessment

Yes. The live ClawHub API for `@openclaw/codex-claw@2026.5.6` shows an 11-file npm-pack artifact with only two stored files and a suspicious missing-code LLM summary, while the artifact download contains the implementation files and the README file endpoint returns 404.

## Solution Assessment

Unclear. Storing parsed tarball entries for new uploads is a good partial fix, but historical backfill, rescanning, and full-tarball LLM scope need maintainer moderation policy before a narrow implementation is safe.
