## Summary

Keep open: current main still has the reported ClawPack scanner blind spot, and ClawHub policy keeps issues open unless a merged fix resolves them. The upload path stores only metadata files for npm-pack ClawPack releases, while the package LLM evaluator and prompt treat that metadata-only file list as the review surface, which can produce the reported false missing-code verdict.

## What This Changes

The issue asks ClawHub to stop treating metadata-only ClawPack release files as the complete implementation surface during package LLM review.

## Best Possible Solution

Keep ClawPack artifact storage and download behavior unchanged, but make the package LLM scan context explicitly distinguish metadata-only release.files from downloadable artifact contents and include artifact facts so complete npm-pack plugins are not flagged as missing implementation.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: publish stores only two metadata files for ClawPack uploads, and the package LLM evaluator builds its file manifest and content from that same release.files array. I did not execute the live LLM path, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Yes, the narrowest maintainable fix is to clarify the package scan surface and pass artifact summary data into the prompt before considering any broader full-tarball scan. That addresses the false missing-code verdict without changing artifact storage or download semantics.
