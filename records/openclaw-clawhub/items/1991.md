## Summary

Keep open: current main now expands ClawPack entries for new uploads, but existing npm-pack releases with metadata-only `release.files` still feed incomplete file surfaces to inspect/file lookup, static scan, and LLM review.

## What This Changes

The issue asks ClawHub to evaluate and expose the actual files inside ClawPack/npm-pack artifacts instead of treating metadata-only `release.files` as the complete package.

## Best Possible Solution

Make package detail, file lookup, static scan, and LLM review share one ClawPack file inventory for npm-pack releases, then safely backfill or rescan affected historical releases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make package inspect, `/file`, static scan, and LLM review see only those stored metadata entries while artifact download can still stream the full tarball.

## Solution Assessment

Partially. The current new-publish extraction path is a useful building block, but it does not fix historical rows or define how completed moderation records should be refreshed.
