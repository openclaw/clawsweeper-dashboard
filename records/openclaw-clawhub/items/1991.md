## Summary

Keep open: current main fixes future ClawPack publishes by persisting extracted tarball entries into `release.files`, but existing metadata-only releases still need a maintainer-approved repair/re-evaluation path and the scan-scope docs still describe metadata-only LLM/static inputs.

## What This Changes

The issue asks ClawHub to stop false suspicious verdicts for ClawPack/npm-pack code plugins by exposing and evaluating actual tarball entries instead of only package metadata files.

## Best Possible Solution

Add a maintainer-approved repair path for existing npm-pack releases that repopulates file metadata from stored ClawPack artifacts, reruns affected evaluations, and aligns scan-scope documentation with the chosen policy.

## Reproduction Assessment

Yes for existing affected rows by source inspection and the provided live registry examples: package detail, file lookup, and LLM evaluation all still depend on `release.files`. I did not rerun the live CLI commands, and newly published ClawPacks on current main appear fixed.

## Solution Assessment

Partially. The current-main publish change is the right narrow fix for future releases, but the issue is not fully solved until existing metadata-only rows are repaired or maintainers explicitly document that only new publishes get full file metadata.
