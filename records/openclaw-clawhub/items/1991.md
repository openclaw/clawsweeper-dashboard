## Summary

Keep open: current main fixes new multipart ClawPack publishes by storing extracted entries, but live historical npm-pack releases still have metadata-only inventories and the detail, raw file, static scan, and LLM review paths still consume `release.files`. This also touches moderation policy because current specs say ClawPack static/LLM scan inputs are intentionally metadata-only for now.

## What This Changes

The issue reports a false suspicious package LLM verdict when ClawPack/npm-pack review sees metadata-only `release.files` instead of the complete downloadable artifact.

## Best Possible Solution

Define one approved ClawPack review source of truth, then either repair historical inventories and rescan them or make metadata-only review prompts explicitly distinguish registry metadata from artifact contents.

## Reproduction Assessment

Yes. Live API checks reproduce the mismatch for `@openclaw/codex-claw@2026.5.6`, and current-main source shows the file/detail/LLM/static paths still consume `release.files`.

## Solution Assessment

Unclear. The new-publish extraction path is the right building block, but the remaining fix needs a maintainer choice between historical backfill, on-demand tarball reads, and a narrower prompt/scope correction with rescans.
