## Summary

Keep open: new multipart ClawPack publishes on current main now store extracted artifact entries, but existing npm-pack releases can still have metadata-only `release.files`, and the detail, raw file, static scan, and LLM review paths still depend on that stored inventory.

## What This Changes

The issue asks for ClawPack npm-pack file-inspection and LLM review surfaces to use artifact-derived entries so complete code-plugin packages are not flagged as missing implementation.

## Best Possible Solution

Use one artifact-derived ClawPack inventory for package detail, raw file lookup, static scan, and LLM review, then repair affected historical npm-pack releases with an explicit audit and rescan plan.

## Reproduction Assessment

Yes. Source inspection plus live registry checks show npm-pack releases with complete artifacts but metadata-only `release.files`, and the current detail/file/LLM/static paths consume that stored inventory.

## Solution Assessment

Unclear. The current new-publish extraction fix is the right building block, but the remaining fix needs a maintainer decision between backfilling stored files, reading selected files from tarballs on demand, and how to update historical moderation state.
