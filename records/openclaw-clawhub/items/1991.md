## Summary

Keep open: current main still stores ClawPack npm-pack releases as exact downloadable artifacts while routing package inspect/file, static scan, and LLM review context through metadata-only `release.files`, so the reported false missing-code path remains source-reproducible.

## What This Changes

The issue reports that npm-pack ClawPack code-plugin reviews can falsely treat complete packages as missing implementation files because review context only includes registry metadata files.

## Best Possible Solution

Keep exact ClawPack tarball storage and download verification intact, but make package review context artifact-aware enough to separate registry metadata from the actual npm-pack contents.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: publish a code-plugin ClawPack with implementation files, then observe that `release.files` contains only metadata entries while LLM/static review consumes that same list; I did not run the live LLM.

## Solution Assessment

Yes. The narrowest maintainable repair is to pass artifact facts or selected parsed ClawPack entries into review context, or clearly label `release.files` as metadata-only, without changing exact-artifact download behavior.

Fix the ClawPack npm-pack package review context bug reported at https://github.com/openclaw/clawhub/issues/1991. Current main stores the exact `.tgz` artifact but only persists `package.json` and `openclaw.plugin.json` in `release.files`, while package LLM/static review and inspect/file surfaces treat that list as the review file surface. Implement a narrow repair so npm-pack code-plugin review no longer flags complete packages as missing implementation solely because `release.files` is metadata-only.
