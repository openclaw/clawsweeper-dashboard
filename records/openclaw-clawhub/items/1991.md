## Summary

Keep open: current main appears to fix future ClawPack publishes by storing parsed tarball entries in `release.files`, but existing npm-pack rows can still have metadata-only `release.files`, and the package detail, file, and LLM review paths still depend on that field.

## What This Changes

Ensure ClawPack/npm-pack package review, inspect, and file surfaces use actual tarball entries rather than only metadata files, so complete code-plugin releases are not flagged as missing code.

## Best Possible Solution

Both new and historical npm-pack releases should expose parsed artifact entries to package detail, file lookup, and LLM review surfaces, with security docs updated to match the chosen scan scope.

## Reproduction Assessment

Yes by source inspection and the reporter's live registry examples: current-main detail, file, and LLM review paths still depend on `release.files`, so historical rows containing only metadata files will reproduce the limited review surface. I did not rerun the live CLI commands in this read-only review.

## Solution Assessment

Partially. Storing extracted entries for new ClawPack publishes is the right narrow code direction, but the issue is not fully solved until existing metadata-only rows are repaired or maintainers explicitly document that only new publishes get full file metadata.
