## Summary

Keep open: current main fixes the new-publish path by storing parsed ClawPack entries, but already-published npm-pack rows with metadata-only `release.files` still make inspect, file lookup, static scan, and LLM review see incomplete inputs.

## What This Changes

The issue reports that npm-pack ClawPack package review and inspect surfaces use metadata-only release files instead of the full downloadable artifact inventory.

## Best Possible Solution

Use one artifact-derived file inventory for package detail, raw file lookup, static scan, and LLM review, then repair affected historical npm-pack releases and rerun moderation scans under an explicit audit policy.

## Reproduction Assessment

Yes. Source inspection shows an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make package detail, `/file`, CLI inspect, static scan, and LLM review consume only those stored metadata entries, matching the reporter's live examples.

## Solution Assessment

Unclear. The current new-publish extraction path is the right building block, but the remaining fix needs a maintainer choice between backfilling stored files, reading selected files from tarballs on demand, and how to rescan or update historical moderation state.
