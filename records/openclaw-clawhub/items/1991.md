## Label Justifications

- `P2`: The bug causes incorrect inspection and moderation for a bounded class of historical ClawPack releases without making the overall service unusable.
- `impact:data-loss`: Historical release metadata silently omits implementation entries that remain present in the canonical stored artifact.

## Summary

Current main materializes complete ClawPack file rows for new uploads, but it does not repair the historical metadata-only releases demonstrated by the report; the open closing PR at https://github.com/openclaw/clawhub/pull/2583 owns that remaining migration, so this issue is still necessary until the repair is validated and merged.

## What This Changes

Ensure ClawPack code-plugin inspection and LLM review use actual artifact entries, including repairing historical releases that stored only metadata files.

## Best Possible Solution

Land a guarded, cursor-based migration that rebuilds historical release file rows from verified stored artifacts, resets stale review state, queues fresh scans, and is proven first with a targeted real-Convex dry run and confirmed repair.

## Reproduction Assessment

Yes. Verified live artifacts have more entries than their stored historical release file rows, and current main’s inspect and LLM-review paths consume those incomplete rows.

## Solution Assessment

Yes. Rebuilding historical rows once from the verified stored artifact preserves a single downstream file source of truth and is safer than adding separate tarball-unpacking logic to every inspection and review consumer.
