## Summary

Keep open: this ClawHub issue is still valid and repository policy does not auto-close issues; current `main` stores parsed ClawPack entries for new uploads, but live historical npm-pack releases still expose metadata-only files to package/detail and LLM-review surfaces.

## What This Changes

The issue reports that npm-pack/ClawPack package review and file surfaces use metadata-only `release.files`, causing false missing-code suspicious verdicts when the downloadable artifact contains implementation files.

## Best Possible Solution

Define the authoritative ClawPack file inventory for public file/detail and LLM review surfaces, then backfill and rescan affected historical releases or explicitly label historical surfaces as metadata-only.

## Reproduction Assessment

Yes. The live ClawHub API still shows metadata-only `version.files` for affected npm-pack releases while artifact downloads contain the missing implementation files, and current source explains why LLM/file surfaces depend on that inventory.

## Solution Assessment

Unclear. Current main's parsed-entry storage is the right direction for new uploads, but historical releases need a maintainer choice between backfill/rescan, artifact-on-demand evaluation, or explicit metadata-only wording.
