## Summary

Keep open: current main now stores parsed ClawPack entries for new multipart publishes, but historical npm-pack releases with metadata-only `release.files` still feed incomplete data to package detail/file lookup, static scan, and LLM review.

## What This Changes

Requests that ClawPack/npm-pack package review and inspect surfaces use actual artifact entries rather than metadata-only `release.files` so complete code plugins are not flagged as missing implementation files.

## Best Possible Solution

Make package detail, file lookup, static scan, and LLM review share one consistent ClawPack file inventory for npm-pack releases, then safely refresh affected historical moderation records.

## Reproduction Assessment

Yes. Source inspection plus the reporter's live registry examples give a high-confidence path: an npm-pack release with `clawpackStorageId` but metadata-only `release.files` makes inspect/file lookup, static scan, and LLM review see only those metadata files.

## Solution Assessment

No. Current main's new-publish extraction is a useful partial fix, but the issue remains for historical releases until the file inventory and moderation rescan policy are handled explicitly.
