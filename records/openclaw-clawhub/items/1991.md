## Summary

Keep open: current main still has the reported ClawPack scanner blind spot. The upload path stores only metadata files for npm-pack ClawPack releases, while the package LLM evaluator and shared prompt treat that file set as the review surface, which can make a complete code-plugin artifact look like it has no implementation files.

## What This Changes

Make ClawPack/npm-pack code-plugin LLM review use parsed artifact entries or clearly distinguish registry metadata from downloadable tarball contents so complete packages are not flagged as missing code.

## Best Possible Solution

Keep ClawPack tarball storage/download behavior unchanged, but make package LLM scan context explicitly say when it is metadata-only and include artifact facts such as npm file count/unpacked size so missing-code findings are not inferred from release.files alone.

## Reproduction Assessment

Yes, from source inspection: publish stores only two metadata files for ClawPack uploads, and the package LLM evaluator builds its file manifest and contents from that same release.files array. I did not execute the live LLM path, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Yes, a narrow fix should clarify the package scan surface and pass artifact summary data into the prompt before attempting any broader full-tarball static/LLM scan. That addresses the false missing-code verdict without changing artifact storage or package download semantics.

## Repair Work Prompt

Fix the ClawPack npm-pack package LLM review false missing-code verdict. Current main stores only package.json and openclaw.plugin.json in release.files for ClawPack uploads while the package LLM evaluator treats release.files as the full review surface. Implement the narrowest repair: pass explicit artifact/scan-scope metadata into the package LLM prompt and update the prompt wording so metadata-only ClawPack package scans do not imply that the downloadable tarball lacks implementation files. Include npm-pack facts already stored on the release, such as artifact kind, npmFileCount, npmUnpackedSize, tarball name, and hashes where appropriate. Do not expand deep static/LLM scanning to every tarball file unless the existing code already provides a bounded helper; preserve artifact download, verification, and storage behavior. Add or update focused tests that cover a ClawPack with dist/index.js in the tarball but only metadata files in release.files, and assert the generated package scan prompt distinguishes registry metadata files from downloadable artifact contents. Add a CHANGELOG.md entry because this is user-facing scanner behavior.
