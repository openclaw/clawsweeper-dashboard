## Label Justifications

- `P2`: This is a normal-priority package review correctness bug with limited scope but real registry and moderation impact.
- `impact:data-loss`: Historical npm-pack ClawPack release rows can silently omit implementation file metadata even though the stored artifact contains those files.
- `impact:security`: The missing file rows feed package LLM/security review surfaces and can produce false suspicious missing-code verdicts.

## Summary

Keep open. Current main appears to fix future ClawPack publishes by storing parsed tarball entries, but historical npm-pack release rows can still have truncated file metadata and the open linked repair candidate has not merged.

## What This Changes

This issue asks ClawHub to stop package LLM/file review surfaces from treating npm-pack ClawPack code-plugin releases as metadata-only when the stored artifact contains implementation files.

## Best Possible Solution

Review and land https://github.com/openclaw/clawhub/pull/2583, or an equivalent maintainer-approved migration, so historical ClawPack releases get artifact-derived file rows and fresh scans.

## Reproduction Assessment

Yes, from reporter live-registry evidence plus current source inspection. I did not run the live CLI locally, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes for the repair direction, but it still needs maintainer approval and merge. Artifact-derived file rows plus rescan is narrower than making every evaluator read tarballs ad hoc.
