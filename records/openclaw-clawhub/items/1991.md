## Summary

Keep open: the report is still valid for existing npm-pack releases. Current main now stores parsed ClawPack entries for new multipart publishes, but historical releases with metadata-only `release.files` still feed incomplete file data to package inspect/file lookup, static scan, and LLM review paths.

## What This Changes

Fix ClawPack/npm-pack package review and file surfaces so code-plugin releases are evaluated against actual tarball entries instead of metadata-only `release.files`.

## Best Possible Solution

Adopt a production-safe artifact reparse/backfill or bounded tarball-backed lookup path for existing npm-pack releases, then refresh affected static/LLM moderation and API/CLI file surfaces so all ClawPack views use consistent artifact entries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will still make file lookup, CLI inspect, static scan, and LLM review see only those stored metadata files.

## Solution Assessment

No, current main is only a partial solution. Storing parsed entries for new publishes is necessary, but existing releases still need a maintainer-approved backfill, fallback, and moderation refresh policy.
