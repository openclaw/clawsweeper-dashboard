## Summary

Keep open: the live ClawPack/npm-pack rows still expose metadata-only file inventories to package detail, raw file lookup, LLM review, and static scan paths, while current specs say that metadata-only ClawPack scanning is intentional for now, so the remaining fix needs maintainer moderation/security policy.

## What This Changes

The issue reports that ClawPack/npm-pack code-plugin releases expose only registry metadata files to package inspection and LLM review even when the downloadable tarball contains implementation files.

## Best Possible Solution

Define the approved ClawPack review source of truth, then repair historical inventories or read selected tarball entries on demand and rescan/update prompts so metadata-only manifests are not presented as complete artifact reviews.

## Reproduction Assessment

Yes. The live detail API for `@openclaw/codex-claw@2026.5.6` shows `npmFileCount: 11` with only two stored files, the artifact download lists implementation files, and current main still routes package detail, raw file lookup, LLM review, and static scan through `release.files`.

## Solution Assessment

Unclear. The new-publish extracted-entry path is the right building block, but maintainers need to choose between historical backfill plus rescans, on-demand artifact reads, or narrower prompt/scope changes for metadata-only reviews.
