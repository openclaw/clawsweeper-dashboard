## Summary

Keep open: the live package still reproduces the historical ClawPack inventory gap, current main only fixes storage for new uploads, and the spec still makes full-tarball LLM scan scope a moderation policy decision.

## What This Changes

The issue asks ClawHub to stop evaluating ClawPack/npm-pack code-plugin releases from metadata-only release files when the downloadable tarball contains implementation files.

## Best Possible Solution

Define the authoritative ClawPack file and scan inventory, then either backfill and rescan affected releases or explicitly label historical file/detail and LLM surfaces as metadata-only.

## Reproduction Assessment

Yes. The live ClawHub API for @openclaw/codex-claw@2026.5.6 shows npmFileCount 11 with only two stored files and a suspicious missing-code LLM summary, while the downloaded artifact contains the implementation files.

## Solution Assessment

Unclear. Current main's parsed-entry storage is the right direction for new uploads, but historical releases need a maintainer choice between backfill/rescan, artifact-on-demand review, or explicit metadata-only wording.
