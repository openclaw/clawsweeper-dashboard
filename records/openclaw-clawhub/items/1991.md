## Summary

Keep open: current main improves future multipart ClawPack publishes, but metadata-only historical npm-pack releases can still feed incomplete file lists to inspect/file/static/LLM surfaces until backfill or artifact-backed review behavior is decided.

## What This Changes

The issue requests that ClawPack/npm-pack inspect, file lookup, and LLM review evaluate actual tarball entries instead of only metadata stored in `release.files`.

## Best Possible Solution

Make package detail, file lookup, static scan, and LLM review share one consistent ClawPack file inventory, then deliberately refresh affected historical moderation records.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make detail, file lookup, static scan, and LLM review see only those stored metadata entries.

## Solution Assessment

Unclear. The current new-publish extraction path is useful, but the issue is not fully solved without a maintainer decision on historical backfill versus artifact-backed lookup and moderation rescan behavior.
