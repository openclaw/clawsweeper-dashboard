## Summary

Keep open: current main only fixes the new multipart ClawPack publish path. Existing npm-pack releases with metadata-only `release.files` still feed incomplete data to package inspect/file lookup, static scan, and LLM review surfaces.

## What This Changes

The issue asks ClawHub to make ClawPack/npm-pack package review and inspect surfaces evaluate artifact entries instead of only registry metadata files.

## Best Possible Solution

Choose and document a production-safe way to expose selected stored artifact entries for existing npm-pack releases, then refresh affected static/LLM moderation verdicts.

## Reproduction Assessment

Yes. Source inspection and the issue’s live examples give a high-confidence path: a ClawPack release whose persisted `release.files` contains only metadata will make inspect, file lookup, static scan, and LLM review see only those files.

## Solution Assessment

Partially. The current new-publish extraction path is useful, but the full fix needs maintainer choice between historical artifact reparse/backfill, tarball-backed reads, and moderation rescan semantics.
