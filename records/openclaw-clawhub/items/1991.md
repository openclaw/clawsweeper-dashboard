## Summary

Keep open: current main fixes the upload path for new ClawPack publishes, but historical npm-pack releases with metadata-only `release.files` still feed incomplete inputs to package inspect, file lookup, static scan, and LLM review surfaces.

## What This Changes

The issue asks ClawPack/npm-pack package review surfaces to evaluate the actual artifact inventory instead of metadata-only `release.files`.

## Best Possible Solution

Unify package detail, file lookup, static scan, and LLM review around one ClawPack artifact-derived file inventory, then refresh affected historical releases with an explicit rescan/audit plan.

## Reproduction Assessment

Yes. Source inspection shows an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make package detail, `/file`, CLI inspect, static scan, and LLM review see only those stored metadata entries, matching the reporter's live examples.

## Solution Assessment

Unclear. The current new-publish extraction path is the right building block for future uploads, but the remaining fix requires maintainer choice between stored-file backfill, tarball-backed lookup, and moderation rescan policy.
