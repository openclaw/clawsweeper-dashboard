## What This Changes

The issue asks ClawHub to prevent false missing-code LLM verdicts for ClawPack/npm-pack code-plugin releases by giving review surfaces artifact-aware context instead of only metadata files.

## Best Possible Solution

Keep ClawPack tarball storage and download verification intact, but make package review context explicitly artifact-aware so metadata-only release.files cannot be interpreted as proof that a complete npm-pack artifact is missing implementation files.

## Reproduction Assessment

Yes. Source inspection shows ClawPack publishes only two metadata files into release.files while package inspect/file and LLM review consume that list; I did not rerun the live LLM model.

## Solution Assessment

Yes. The narrowest maintainable fix is to label the scan surface accurately and pass ClawPack artifact facts or selected parsed artifact text into review context, rather than changing artifact storage or broadening deep scans by default.

## Repair Work Prompt

Fix the source-reproducible false suspicious verdict for ClawPack/npm-pack package LLM review. On current main, multipart ClawPack publishes store the tarball artifact but only metadata files in release.files; evaluatePackageReleaseWithLlm and assembleEvalUserMessage then treat release.files as the full review surface. Implement a narrow fix that keeps tarball storage/download/verification semantics intact, but adds artifact-aware facts/context for npm-pack releases, such as artifact kind, npmFileCount/unpackedSize/digests and/or selected parsed tarball entries like README and runtime/source paths, plus prompt wording that says metadata-only inputs are not proof that implementation files are missing. Add regression coverage for package LLM prompt/context and package API/CLI behavior if file surfaces change. Add a CHANGELOG.md entry if the user-facing review or inspect behavior changes. Do not deep-scan every tarball file by default unless the code change explicitly preserves current scan-scope policy and resource limits.
