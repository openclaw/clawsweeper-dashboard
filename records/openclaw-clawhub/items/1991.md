## Summary

Keep open: current main fixes new ClawPack uploads by storing parsed entries, but historical npm-pack releases still expose metadata-only `release.files` and current main still derives package detail, file, and LLM review inputs from that field.

## What This Changes

The issue reports that npm-pack/ClawPack package review and file surfaces can see only metadata files, causing false missing-code suspicious verdicts for complete code-plugin artifacts.

## Best Possible Solution

Define the authoritative ClawPack inventory for package and ClawScan surfaces, then backfill or explicitly label historical metadata-only releases and rescan affected verdicts under that policy.

## Reproduction Assessment

Yes. The live codex-claw release shows npmFileCount 11 with only two stored files, README.md returns 404, the tarball contains source/runtime files, and current main still feeds detail/file/LLM surfaces from `release.files`.

## Solution Assessment

Unclear. Persisting parsed entries for new uploads is the right storage direction, but the remaining historical inventory and public ClawScan verdict repair needs an explicit backfill/rescan policy.
