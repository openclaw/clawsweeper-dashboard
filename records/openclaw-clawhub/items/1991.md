## Summary

Keep open: current main fixes the forward publish path for new multipart ClawPack uploads, but historical npm-pack releases can still have metadata-only `release.files`, and the package detail/file/LLM paths still depend on that field.

## What This Changes

The report asks ClawHub to make npm-pack ClawPack package file and LLM review surfaces see actual artifact entries instead of only metadata files.

## Best Possible Solution

Backfill or reparse historical npm-pack artifacts into release file metadata or targeted artifact-derived review context, refresh affected scan verdicts, and update docs/API expectations to match the chosen ClawPack scan scope.

## Reproduction Assessment

Yes. From source inspection, a historical npm-pack release with stored ClawPack bytes but metadata-only `release.files` will still make package detail, file lookup, static scan, and LLM review see only those metadata entries.

## Solution Assessment

Partially. The current-main extraction for new publishes is the right narrow forward fix, but the issue is not fully solved until maintainers choose and run a historical artifact reparse/backfill plus scan refresh path.
