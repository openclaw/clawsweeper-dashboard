## Summary

Keep open: the report still reproduces for live historical npm-pack releases, and current main still feeds package detail, raw-file lookup, static scan, and LLM review from `release.files` even when the stored ClawPack artifact contains more files.

## What This Changes

Ensure ClawPack/npm-pack code-plugin review and file inspection use artifact-derived entries instead of metadata-only `release.files` so complete packages are not flagged as missing code.

## Best Possible Solution

Use one artifact-derived ClawPack inventory for package detail, raw file lookup, static scan, and LLM review, then repair and rescan affected historical npm-pack releases under an explicit moderation policy.

## Reproduction Assessment

Yes. Live registry checks show complete npm-pack artifacts with metadata-only `version.files`, and current-main source inspection shows detail, raw-file, static-scan, and LLM-review paths consume that stored inventory.

## Solution Assessment

Unclear. The new-publish extraction path is the right building block, but maintainers still need to choose whether historical releases are backfilled into `release.files` or read from stored tarballs on demand, and how rescans update moderation state.
