## Summary

Keep open: current main fixes the new ClawPack publish path, but stored metadata-only npm-pack releases can still make package inspect, file lookup, static scan, and LLM review operate on an incomplete `release.files` surface.

## What This Changes

The issue requests that ClawPack/npm-pack code-plugin review and inspection use the actual artifact entries, or distinguish artifact contents from registry metadata files.

## Best Possible Solution

Make package detail, file lookup, static scan, and LLM review share one consistent ClawPack file inventory for npm-pack releases, then refresh affected historical moderation records safely.

## Reproduction Assessment

Yes from source inspection: an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make version detail, `/file`, CLI inspect, static scan, and LLM review see only those stored metadata entries. I did not rerun the live registry commands, but the issue comments provide concrete live examples.

## Solution Assessment

No, not fully yet. The current new-publish extraction path is a good partial fix, but the remaining issue needs historical release handling and an explicit moderation rescan policy.
