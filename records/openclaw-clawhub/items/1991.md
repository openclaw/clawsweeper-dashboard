## Summary

Keep open: current main fixes future multipart ClawPack publishes by storing parsed artifact entries, but live historical npm-pack releases still expose metadata-only `release.files`, and current detail, raw file, LLM, and static scan paths still consume that stored inventory.

## What This Changes

Ensure ClawPack npm-pack releases expose artifact-derived implementation files to package detail/raw file and package review surfaces so complete code plugins are not flagged as missing code.

## Best Possible Solution

Use one artifact-derived ClawPack inventory for package detail, raw file lookup, static scan, and LLM review, then repair affected historical npm-pack releases with an explicit audit and rescan plan.

## Reproduction Assessment

Yes. The live registry shows npm-pack releases whose artifact metadata and tarball contents include implementation files while `version.files` and raw file lookup expose only metadata files, and current source still drives detail/raw/LLM/static paths from `release.files`.

## Solution Assessment

Unclear. The current new-publish extraction fix is the right building block, but the remaining solution needs a maintainer decision between backfilling stored file inventories, reading selected files from tarballs on demand, and how to update historical moderation state.
