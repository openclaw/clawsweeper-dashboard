## Summary

Keep open: current main fixes the new multipart ClawPack publish path, but historical metadata-only npm-pack releases can still feed incomplete file data to inspect, file lookup, static scan, and LLM review surfaces.

## What This Changes

The issue requests that npm-pack/ClawPack package inspect and LLM review evaluate the actual artifact entries instead of only metadata files stored in `release.files`.

## Best Possible Solution

Define one shared ClawPack file inventory for package detail, file lookup, static scan, and LLM review, then repair or lazily resolve affected historical npm-pack releases and rescan them under an explicit moderation policy.

## Reproduction Assessment

Yes from source inspection. A historical npm-pack release with `clawpackStorageId` but metadata-only `release.files` will still make version detail, file lookup, static scan, and LLM evaluation consume only those stored metadata entries.

## Solution Assessment

Unclear. Storing extracted entries for new publishes is a good building block, but the remaining fix needs a maintainer choice between in-place backfill, tarball-backed lookup, and moderation rescan policy.
