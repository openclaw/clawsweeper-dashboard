## Summary

The issue should stay open: current main still has an actionable mismatch where npm-pack ClawPack artifacts can contain implementation files while package file lookup, static package scanning, and LLM review context are driven by `release.files`, which may only contain metadata files. The repository policy for ClawHub issues also says not to auto-close issues during cleanup.

## What This Changes

The issue reports that npm-pack ClawPack package review and file inspection use metadata-only `release.files` instead of the full artifact inventory, producing false missing-code moderation verdicts.

## Best Possible Solution

Define the approved ClawPack review source of truth, then either backfill/rescan npm-pack inventories so review/file APIs see selected artifact entries or explicitly label metadata-only review surfaces so they cannot imply full tarball coverage.

## Reproduction Assessment

Yes. The issue provides concrete live package evidence, and current-main source inspection shows package lookup, static scan, and LLM review inputs still depend on `release.files`.

## Solution Assessment

Unclear. The narrow technical fix is to stop treating metadata-only `release.files` as full artifact contents for npm-pack moderation, but maintainers need to choose whether to backfill historical rows, read tarballs on demand, or preserve metadata-only semantics with clearer wording.
