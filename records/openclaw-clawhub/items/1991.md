## Summary

Keep open: the report still reproduces for live npm-pack ClawPack releases, and current main still routes package file lookup, static scan, and LLM review through `release.files`. New multipart publishes now store extracted entries, but historical metadata-only inventories and the documented moderation scope still need maintainer policy before repair.

## What This Changes

The issue reports that ClawPack npm-pack code-plugin releases can receive false missing-code LLM verdicts because review and file surfaces see only metadata files instead of the full artifact contents.

## Best Possible Solution

Define the approved ClawPack review source of truth, then either repair and rescan historical npm-pack inventories or make metadata-only prompts and APIs explicitly distinguish registry metadata from artifact contents.

## Reproduction Assessment

Yes. Live API checks reproduce the mismatch, and current-main source shows the raw file, static scan, and LLM review paths still consuming `release.files`.

## Solution Assessment

Unclear. New-publish extraction is the right building block, but the remaining fix needs a maintainer choice between historical inventory backfill, on-demand tarball reads, prompt clarification, and rescans.
