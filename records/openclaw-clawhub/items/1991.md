## Summary

Keep open: current main fixes the new-publish path by storing parsed ClawPack entries, but historical npm-pack releases with metadata-only release.files still make package detail, file lookup, static scan, and LLM review consume incomplete inputs.

## What This Changes

The issue asks ClawHub to use artifact-derived ClawPack entries instead of metadata-only release.files when inspecting and reviewing npm-pack code-plugin releases.

## Best Possible Solution

Use one artifact-derived package file inventory for detail, raw file lookup, static scan, and LLM review, then repair affected historical npm-pack releases and rerun moderation under an explicit audit policy.

## Reproduction Assessment

Yes. Source inspection shows an npm-pack release with clawpackStorageId but metadata-only release.files will make package detail, raw file lookup, static scan, and LLM review see only those stored metadata entries, matching the reporter's live examples.

## Solution Assessment

Unclear. Persisting parsed tar entries for new publishes is the right building block, but the remaining fix needs a maintainer choice on historical backfill, rescan timing, and moderation verdict handling.
