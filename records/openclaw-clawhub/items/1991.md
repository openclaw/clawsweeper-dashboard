## Summary

Keep open: this is a valid ClawHub issue, and the live historical npm-pack release still appears to expose only metadata files to package/detail and LLM-review surfaces even though the artifact contains implementation files. Current main has coverage for storing parsed ClawPack entries on new uploads, but the remaining historical inventory/backfill/rescan behavior is still a maintainer moderation/product decision.

## What This Changes

The issue asks ClawHub to make npm-pack/ClawPack code-plugin LLM review use actual artifact entries, or clearly distinguish metadata-only files from downloadable package contents, so complete packages are not flagged as missing code.

## Best Possible Solution

Define the authoritative ClawPack file inventory for public file/detail and LLM review surfaces, then backfill/rescan affected releases or explicitly label historical surfaces as metadata-only.

## Reproduction Assessment

Yes. The live API for @openclaw/codex-claw@2026.5.6 still shows an npm-pack artifact with 11 files but only two stored `version.files`, while the downloaded artifact lists the missing implementation files.

## Solution Assessment

Unclear. Current main's parsed-entry storage is the right direction for new uploads, but historical releases need a maintainer choice between backfill/rescan, artifact-on-demand review, or explicit metadata-only wording.
