## Summary

Keep open: current main still routes ClawPack package inspect/file and LLM/static scan context through `release.files`, which for npm-pack publish contains only `package.json` and `openclaw.plugin.json`, so the false missing-code review path remains source-reproducible despite the documented metadata-only scan scope.

## What This Changes

Teach ClawPack/npm-pack package review surfaces to distinguish metadata-only release files from actual tarball contents so complete code plugins are not flagged as missing implementation.

## Best Possible Solution

Keep ClawPack storage/download behavior intact, but add explicit package-artifact review context and prompt wording that separates registry metadata files from actual npm-pack artifact facts or selected artifact entries.

## Reproduction Assessment

Yes. Source inspection shows the ClawPack publish path stores only two metadata files in `release.files`, while package inspect/file and LLM/static review consume that same list; I did not run the live LLM model.

## Solution Assessment

Yes. The narrowest maintainable fix is to label the scan surface accurately and pass ClawPack artifact facts or selected parsed artifact text into review context, rather than changing artifact storage or enabling broad deep scans by default.

Keep ClawPack tarball storage, artifact download/verify behavior, and the documented metadata-only deep-scan policy intact unless a test proves a narrower API change is required.
