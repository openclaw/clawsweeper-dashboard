## Summary

Keep open: the live mismatch still reproduces for historical npm-pack releases, and current main still routes raw-file, static scan, and LLM review context through `release.files`; resolving this needs maintainer moderation/security policy, not cleanup closure.

## What This Changes

The issue asks ClawHub to stop evaluating ClawPack npm-pack code-plugin releases as metadata-only when the downloadable tarball contains implementation files.

## Best Possible Solution

Maintainers should choose the approved ClawPack review source of truth, then either backfill and rescan historical npm-pack inventories or make metadata-only APIs and prompts explicitly say they are not reviewing full tarball contents.

## Reproduction Assessment

Yes. The live API for `@openclaw/codex-claw@2026.5.6` reports 11 artifact files but only two inventory files and an LLM summary claiming no bridge code, while the artifact download lists the implementation files.

## Solution Assessment

Unclear. Storing parsed entries for new uploads is useful, but the remaining work needs a maintainer decision between preserving metadata-only scope, backfilling and rescanning historical releases, or reading artifact entries on demand.
