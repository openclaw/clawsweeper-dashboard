## What This Changes

The issue requests that ClawPack/npm-pack review and inspect surfaces distinguish registry metadata files from actual artifact entries so complete code plugins are not flagged as missing implementation.

## Best Possible Solution

Keep ClawPack artifact storage and download behavior intact, but add explicit artifact facts or selected parsed ClawPack entry context plus prompt wording so the evaluator does not infer missing implementation from metadata-only release files.

## Repair Work Prompt

Fix the ClawPack/npm-pack package review false missing-code verdict. Keep ClawPack tarball storage/download behavior and the documented metadata-only broad scan policy intact. Update the package LLM/static review context so npm-pack releases clearly label release.files as registry metadata and include artifact facts such as artifactKind, npmFileCount, npmUnpackedSize, sha/integrity, and either selected parsed text entries or an artifact entry manifest sufficient to prevent claims that implementation files are absent when the tarball contains them. Do not make package publish upload every extracted file into release.files, and do not broaden scanning across every tarball file by default unless the change is explicitly bounded. Add regression coverage around assembleEvalUserMessage/evaluatePackageReleaseWithLlm and the ClawPack publish path; update CLI/API docs only if package inspect/file semantics change.
