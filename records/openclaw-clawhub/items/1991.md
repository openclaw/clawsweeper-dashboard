## Summary

Keep open: the live registry still reproduces the metadata-only inventory mismatch for npm-pack ClawPack releases, and current main still feeds package inspect/raw-file/static-scan/LLM review from `release.files`; because the spec currently says ClawPack static/LLM inputs are intentionally metadata-only, the remaining fix needs maintainer moderation policy rather than cleanup closure.

## What This Changes

The issue reports that npm-pack ClawPack releases expose only metadata files to package inspect and LLM review, producing false missing-code suspicious verdicts despite complete downloadable artifacts.

## Best Possible Solution

Define the approved ClawPack review source of truth, then either backfill/rescan historical npm-pack inventories or make metadata-only prompts and APIs explicitly distinguish registry metadata from artifact contents.

## Reproduction Assessment

Yes. The public API for @openclaw/codex-claw@2026.5.6 still exposes only metadata files while the artifact download contains implementation files, and current main routes inspect/raw-file/LLM/static scan through `release.files`.

## Solution Assessment

Unclear. Restoring extracted entries for new uploads is the right building block, but historical metadata-only rows and the documented metadata-only scan scope require a maintainer choice between backfill, on-demand artifact reads, prompt clarification, and rescans.
