## Summary

Keep open: current main appears to fix future ClawPack publishes by storing extracted artifact entries in `release.files`, but existing metadata-only releases may still need repair/re-evaluation, and ClawHub cleanup policy keeps issues open.

## What This Changes

The issue reports that ClawPack/npm-pack package review and inspect surfaces only see metadata files, causing complete code-plugin artifacts to receive false missing-code suspicious verdicts.

## Best Possible Solution

Repair or backfill existing affected ClawPack release file metadata from stored artifacts, rerun affected package evaluation, keep the current extracted-entry publish behavior, and update the scan-scope docs.

## Reproduction Assessment

Yes for existing affected rows: current code still evaluates and exposes package files from `release.files`, and no backfill was found for releases already saved with only metadata files. No for newly published ClawPacks on current main, because the upload path now stores parsed artifact entries.

## Solution Assessment

Partially. The current-main source change is the right direction for future publishes, but a complete resolution likely needs maintainer-approved production data repair/re-evaluation and docs alignment.
