## Summary

Keep open: current main fixes new ClawPack uploads, but historical npm-pack releases with truncated `release.files` still break inspect/file surfaces and can mislead package LLM review.

## What This Changes

The issue asks ClawHub to stop treating ClawPack npm-pack code plugins as metadata-only during package inspection and LLM review when the downloadable artifact contains implementation files.

## Best Possible Solution

Repair existing affected ClawPack rows so user-facing file inspection reflects artifact contents, and update package LLM context to acknowledge artifact file counts/metadata without violating the metadata-only scanner scope in `specs/security-moderation.md`.

## Reproduction Assessment

Yes, as a source-level reproduction: an npm-pack release with `clawpackStorageId`, `npmFileCount > release.files.length`, and only metadata files in `release.files` will list only those files and return `File not found` for README/source files. `evaluatePackageReleaseWithLlm` reads that same truncated array.

## Solution Assessment

Yes, with one constraint: a cursor-based repair/backfill plus targeted LLM context adjustment is the narrowest maintainable fix. It should not rewrite artifact bytes or silently expand ClawPack LLM scans beyond the documented scanner scope.

Fix the legacy ClawPack/npm-pack package release bug where `release.files` can contain only metadata files while the stored ClawPack artifact contains README/source/runtime entries. Add a cursor-based Convex repair/backfill over `packageReleases` rows with ClawPack storage and incomplete file metadata, parse the stored tarball with the existing ClawPack parser, store missing extracted file blobs, and patch `release.files` without changing `clawpackStorageId`, artifact bytes, SHA-256, npm integrity, or shasum. Make `package inspect --files` and `/api/v1/packages/{name}/file` work for repaired README/source/runtime files. For package LLM review, preserve the `specs/security-moderation.md` ClawPack metadata-only scanner intent: include artifact facts/manifest context or otherwise prevent false 'implementation missing' conclusions for intentionally limited ClawPack scan surfaces, but do not introduce unrestricted deep tarball LLM scanning unless the spec is updated. Add focused regression coverage for a metadata-only legacy ClawPack row and keep the existing new-upload extracted-file test passing. Do not change CLI upload behavior or artifact download bytes.
