## Summary

Keep open: current main improves the future publish path, but existing npm-pack releases with metadata-only `release.files` can still feed incomplete file data to package inspect, file lookup, static scan, and LLM review.

## What This Changes

The issue asks ClawHub package inspect and review surfaces to use or distinguish actual ClawPack tarball entries instead of treating metadata-only `release.files` as the full package.

## Best Possible Solution

Define the ClawPack file-surface source of truth, repair or reparse affected historical releases, refresh moderation signals, and align API/CLI/security docs with the final behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make inspect, file lookup, static scan, and LLM review see only the stored metadata files.

## Solution Assessment

Partially. The current new-publish extraction path is useful, but the complete fix needs a maintainer-approved historical repair or tarball-backed lookup strategy plus a moderation rescan policy.
