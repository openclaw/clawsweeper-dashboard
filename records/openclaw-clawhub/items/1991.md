## Summary

Keep open: this is a valid current-main ClawPack moderation/API inventory bug, and ClawHub cleanup policy keeps issues open unless a PR close condition applies. Source inspection shows package file lookup, static package scanning, and LLM evaluation still depend on `release.files`, while ClawPack upload currently stores only metadata entries for npm-pack artifacts.

## What This Changes

The issue requests that npm-pack/ClawPack package review and inspection use actual artifact entries, or clearly distinguish metadata-only `release.files` from downloadable tarball contents.

## Best Possible Solution

Define ClawPack artifact contents as the review/inspection source of truth, then add an indexed/backfillable inventory path or on-demand artifact reader plus an explicit rescan plan for affected historical releases.

## Reproduction Assessment

Yes. The live API for `@openclaw/codex-claw@2026.5.6` still exposes only metadata files while the artifact download contains implementation files, and current main routes file APIs and LLM review through `release.files`.

## Solution Assessment

Unclear. The reporter's direction is technically grounded, but maintainers need to choose the durable source of truth and historical rescan/backfill policy before code changes alter moderation results.
