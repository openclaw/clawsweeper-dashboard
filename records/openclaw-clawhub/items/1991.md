## Summary

Keep open: current main fixes extracted-file storage for new ClawPack publishes, but historical npm-pack releases can still have metadata-only inventories and the detail, raw file, static scan, and LLM review paths still consume that stored inventory.

## What This Changes

The issue requests artifact-derived ClawPack/npm-pack review inputs so complete code-plugin tarballs are not judged from metadata-only `release.files`.

## Best Possible Solution

Use one artifact-derived ClawPack inventory for detail, raw file lookup, static scan, and LLM review, then backfill affected historical npm-pack releases with explicit audit logging and rescans.

## Reproduction Assessment

Yes. Live API checks reproduce npm-pack releases with complete artifact metadata but metadata-only `version.files`, and current source inspection shows the review/detail/file/static paths consume that stored inventory.

## Solution Assessment

Unclear. The current new-publish extraction fix is the right building block, but the remaining historical repair needs a maintainer choice between backfilling inventories, parsing tarballs on demand, and how to update moderation outcomes.
