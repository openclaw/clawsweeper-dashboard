## Summary

Keep open: current main improves new ClawPack uploads and the new Codex scan worker can inspect the tarball, but stored npm-pack rows with metadata-only `release.files` still affect inspect/raw-file/static-scan surfaces and can leave old false scan results unrepaired.

## What This Changes

The issue asks ClawHub to evaluate ClawPack/npm-pack code-plugin releases from the actual artifact contents instead of treating metadata-only release files as the full package.

## Best Possible Solution

Backfill stored ClawPack npm-pack release file metadata from the saved tarballs, rescan repaired releases through the current ClawScan path, and update the security moderation spec to match the supported scan scope.

## Reproduction Assessment

Yes, source-level: create or locate an active npm-pack package release with `clawpackStorageId`, `npmFileCount` greater than `files.length`, and metadata-only `release.files`; inspect/raw-file/static-scan and the legacy LLM path consume the truncated list. I did not run a live production data query in this read-only review.

## Solution Assessment

Yes: a cursor-based artifact-file repair plus rescan is narrower than changing only prompts, because package inspect, raw-file, static scan, and old scan results all depend on the persisted file list. The current Codex worker already handles tarball extraction for new ClawScan jobs, so the remaining fix should repair stored rows and route rescans through that path.
