## Summary

Keep open: current main partially protects future ClawPack publishes, but source inspection still shows historical metadata-only release rows feeding the API/CLI inspect path and LLM/static scan inputs, while docs frame deep tarball scanning as a moderation-scope decision.

## What This Changes

Make ClawPack/npm-pack review and inspect surfaces use or accurately distinguish actual tarball entries instead of treating metadata-only release files as the complete package.

## Best Possible Solution

Use one consistent ClawPack artifact inventory for package detail/file lookup and moderation inputs, then repair or tarball-resolve affected historical releases and rescan moderation records under an explicit policy.

## Reproduction Assessment

Yes. Source inspection plus live examples show that an npm-pack release with a stored tarball but metadata-only release.files will make API/CLI inspect, file lookup, LLM review, and static scan paths iterate only those stored metadata files.

## Solution Assessment

Partially. Current main’s new-upload extraction path is useful for future publishes, but the best fix still needs a maintainer decision on historical release repair versus tarball-backed lookup and moderation rescan policy.
