## Label Justifications

- `P2`: The defect causes incorrect inspection and moderation verdicts for a bounded class of historical ClawPack releases without broadly disabling the registry.
- `impact:data-loss`: Implementation entries present in stored artifacts were omitted from the persisted release-file data consumed by inspection and review surfaces.

## Summary

Current main fixes newly published ClawPacks by persisting parsed artifact entries, but historical releases created by the earlier metadata-only path remain incomplete in `release.files`, which still drives inspection and LLM review. The open closing PR owns the necessary historical migration, so this issue remains necessary until that PR lands and the repair is validated and run.

## What This Changes

Make npm-pack ClawPack implementation entries available to package inspection and LLM review, including for historical releases stored with metadata-only file rows.

## Best Possible Solution

Retain the complete-entry upload path and land a dry-run-first, resumable, narrowly targetable historical migration that verifies the stored artifact, rebuilds release file rows and fingerprints, resets stale review state, and reruns package scans.

## Reproduction Assessment

Yes. The reporter provides a verified 11-entry artifact whose registry file view exposes only two entries, and current source confirms that historical `release.files` rows directly determine the LLM review context.

## Solution Assessment

Yes in direction. Repairing only incomplete historical rows from their verified stored artifacts is narrower and safer than changing the evaluator to maintain a second artifact-reading path, while the normal publication path is already corrected.
