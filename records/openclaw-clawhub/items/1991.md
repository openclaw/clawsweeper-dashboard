## Summary

Keep open: current main stores full parsed ClawPack entries for new multipart publishes, but historical npm-pack releases with metadata-only `release.files` can still make inspect, raw file lookup, static scan, and LLM review see incomplete inputs.

## What This Changes

The issue requests that ClawPack/npm-pack code-plugin review use the actual artifact file inventory, or clearly distinguish registry metadata files from downloadable tarball contents, to avoid false missing-code verdicts.

## Best Possible Solution

Use a single artifact-derived inventory for package detail, raw file lookup, static scan, and LLM review, then repair affected historical npm-pack releases with an explicit moderation audit/rescan plan.

## Reproduction Assessment

Yes. Source inspection shows an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make detail, `/file`, static scan, and LLM review consume only those stored metadata entries, matching the reporter's live registry examples.

## Solution Assessment

Unclear. The current new-publish extraction path is the right building block, but the remaining fix needs a maintainer choice between backfilling stored files, reading selected files from tarballs on demand, and how to update historical moderation state.
