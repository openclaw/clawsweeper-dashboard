## Summary

Keep open: current main now gives queued Codex ClawScan jobs the full ClawPack tarball and stores extracted entries for new uploads, but stored metadata-only npm-pack releases can still expose truncated inspect/raw-file data and existing false suspicious scan results are not automatically repaired.

## What This Changes

Fix ClawPack/npm-pack plugin review so ClawScan/LLM and package inspect surfaces use the actual artifact entries instead of only metadata files.

## Best Possible Solution

Add a cursor-based repair/backfill that reconstructs stored ClawPack file metadata from existing tarballs, reschedules affected scans through the tarball-aware worker, and keeps inspect/raw-file output consistent with artifact truth.

## Reproduction Assessment

Yes, source-level: seed or locate a package release with clawpackStorageId, npmFileCount greater than release.files.length, and a non-error suspicious llmAnalysis; inspect/raw-file and legacy LLM paths consume the truncated persisted files. I did not query production data in this read-only review.

## Solution Assessment

Yes for the remaining work: a targeted stored-release repair plus rescan is narrower than changing package publish UX, and current main already establishes full-tarball ClawScan as the intended review direction.
