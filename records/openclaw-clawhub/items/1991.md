## Summary

Keep open: ClawHub issues are not auto-closed by this cleanup pass, and current main still leaves existing npm-pack releases with metadata-only `release.files` driving package inspect, file lookup, static scan, and LLM review surfaces.

## What This Changes

The issue requests ClawPack/npm-pack review and file-inspection surfaces use actual tarball entries so complete code plugins are not falsely flagged as missing implementation files.

## Best Possible Solution

Use one artifact-aware file surface for package detail, raw file lookup, static scan, and LLM review, then safely repair or reparse existing npm-pack releases and refresh affected moderation results.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will still make inspect, file lookup, static scan, and LLM review see only the stored metadata files.

## Solution Assessment

Partially. The current extracted-file publish path is a useful part of the fix, but the best solution also needs historical release handling and an explicit moderation rescan policy.
