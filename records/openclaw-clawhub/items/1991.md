## Summary

Keep open: the live registry still shows the reported metadata-only inventory for affected ClawPack releases, and current main still feeds package file lookup, static rescans, and LLM review from the stored `release.files` inventory. Because the current moderation spec says ClawPack static/LLM inputs are intentionally metadata-only for now, the remaining fix needs maintainer security policy rather than cleanup closure.

## What This Changes

Request that ClawPack/npm-pack package review and file surfaces use, or accurately distinguish, full artifact entries instead of only metadata files so complete code plugins are not falsely flagged as missing code.

## Best Possible Solution

Set an explicit ClawPack review source of truth, then either backfill extracted files and rescan affected npm-pack releases or make APIs and prompts clearly metadata-only while artifact download and VirusTotal remain full-tarball paths.

## Reproduction Assessment

Yes. Live API checks reproduce the mismatch for `@openclaw/codex-claw@2026.5.6`, and current-main source shows file lookup, static rescans, and LLM evaluation still consuming stored `release.files`.

## Solution Assessment

Unclear. Current main improved new upload inventory storage, but the remaining best fix depends on maintainer choice between historical backfill/rescan, artifact-on-demand review, or explicit metadata-only labeling.
