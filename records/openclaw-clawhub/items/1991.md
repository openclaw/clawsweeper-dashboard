## Summary

Keep open: current main still has the reported ClawPack LLM review blind spot, and ClawHub policy keeps open issues open unless a merged fix resolves them. Source inspection shows metadata-only ClawPack release files are passed to a prompt that can describe that limited surface as the full included source.

## What This Changes

The issue requests that npm-pack/ClawPack code-plugin LLM review distinguish registry metadata files from actual downloadable artifact contents so complete packages are not flagged as missing code.

## Best Possible Solution

Keep ClawPack artifact storage and download behavior unchanged, but make package LLM scan context explicitly distinguish metadata-only release.files from downloadable artifact facts so complete npm-pack plugins are not flagged as hollow.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: ClawPack publish stores only two metadata files for scan inputs, and the package LLM evaluator builds its manifest and content from that same release.files array.

## Solution Assessment

Yes. The narrowest maintainable fix is to clarify the package scan surface and pass artifact summary data into the prompt before considering any broader full-tarball scan.

## Repair Work Prompt

Fix the ClawPack package LLM review blind spot reported in this issue. Keep ClawPack artifact storage, download, and metadata-only scan policy unchanged, but make the LLM review context accurately distinguish registry metadata files in release.files from the downloadable npm-pack artifact. Include stored artifact facts such as artifactKind, npmFileCount, npmUnpackedSize, npmIntegrity/npmShasum/sha256, and/or an explicit metadata-only scan note so complete code-plugin packages are not flagged as missing implementation solely because release.files lacks dist/src files. Update prompt tests and package publish/eval coverage as needed. Do not add a config option, do not implement broad full-tarball deep LLM/static scanning unless maintainers explicitly ask for that product change, and do not change artifact download semantics. Run the focused tests and repo checks listed in workValidation.
