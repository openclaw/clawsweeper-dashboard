## Summary

Keep open: the report still describes a valid ClawPack review-surface problem on current main. New multipart publishes now persist parsed tarball entries, but existing npm-pack releases whose stored `release.files` only contains metadata can still produce incomplete inspect/file/LLM/static-review inputs.

## What This Changes

The issue asks ClawHub to make npm-pack/ClawPack code-plugin review and inspection surfaces evaluate the actual artifact entries instead of only metadata files stored in `release.files`.

## Best Possible Solution

Pick a production-safe artifact reparse or tarball-backed lookup strategy, refresh affected moderation signals, and update public API/security docs to match the final ClawPack scan and file-inspection scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: any npm-pack release with a stored ClawPack artifact but metadata-only `release.files` will still make inspect, file lookup, static scan backfill, and LLM review see only those stored metadata entries.

## Solution Assessment

Partially. The current future-publish extraction path is a good building block, but the maintainable end state still needs a historical backfill or tarball-backed lookup plus refreshed scan results.
