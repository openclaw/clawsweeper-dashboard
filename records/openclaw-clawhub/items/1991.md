## Summary

Keep open: current main appears to fix new ClawPack publishes by storing extracted tarball entries in `release.files`, but existing metadata-only releases can still feed incomplete file surfaces into package detail, file lookup, static scan, and LLM review.

## What This Changes

The issue requests that ClawPack/npm-pack package review and package file surfaces evaluate actual tarball entries instead of metadata-only rows so complete code-plugin artifacts are not flagged as missing implementation code.

## Best Possible Solution

Add a maintainer-approved repair path that reparses stored npm-pack artifacts, repopulates affected release file metadata, reruns relevant package scans, and updates scan-scope docs to match the chosen policy.

## Reproduction Assessment

Yes for existing metadata-only rows: package detail, file lookup, static scan, and LLM review all still consume `release.files`, and the reporter supplied live affected package examples. Newly published ClawPacks on current main appear covered by the extracted-file publish path.

## Solution Assessment

Partially: the current-main publish change is the right narrow fix for future releases, but it does not repair or re-evaluate existing metadata-only ClawPack rows. The maintainable remaining fix is a bounded data repair plus scan/doc alignment, not a CLI-only workaround.
