## Summary

Keep open: current main fixes new ClawPack uploads, but live and source evidence show older npm-pack releases can still have metadata-only persisted file lists while inspect, raw-file, static scan, and LLM review paths read only those persisted rows.

## What This Changes

The issue requests that ClawPack/npm-pack code-plugin review and inspect surfaces use the full artifact entries rather than metadata-only release files to avoid false missing-code suspicious verdicts.

## Best Possible Solution

Add a cursor-based repair that parses stored ClawPack artifacts into full release file metadata for affected npm-pack rows, reschedules static and LLM scans, and updates the moderation spec to state the supported scan scope.

## Reproduction Assessment

Yes, source-level with live confirmation: seed or locate an npm-pack release whose artifact npmFileCount is greater than release.files.length and whose release.files contain only metadata, then the version, raw file, static scan, and LLM paths all consume the truncated list. I did not run a local Convex data mutation, but the live registry example reproduces the stored-row symptom exactly.

## Solution Assessment

Yes: repairing persisted file metadata from the stored artifact and then rescanning is the narrowest maintainable fix because changing only the LLM evaluator would leave package inspect, raw file, and static scan surfaces inconsistent.
