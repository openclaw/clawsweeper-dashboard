## Summary

Keep open: current main partially fixes new ClawPack publishes by persisting parsed tarball entries, but historical npm-pack releases whose stored `release.files` only contains metadata still feed incomplete inputs to inspect, file lookup, static scan, and LLM review surfaces.

## What This Changes

The issue requests that ClawPack/npm-pack code-plugin review and inspect surfaces evaluate actual tarball entries instead of only registry metadata files.

## Best Possible Solution

Make package detail, file lookup, static scan, and LLM review share one consistent ClawPack file inventory, then safely migrate or rescan affected metadata-only releases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make inspect/file, static scan, and LLM review see only those stored metadata entries.

## Solution Assessment

No. Current main's new-publish extraction is a useful partial fix, but it is not sufficient until historical releases and completed moderation records are repaired or explicitly rescanned.
