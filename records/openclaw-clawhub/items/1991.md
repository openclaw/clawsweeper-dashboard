## Summary

Keep open: this is a valid current-main ClawPack/npm-pack issue, and ClawHub policy keeps issues open for maintainer follow-up. Current main extracts file entries for future uploads, but existing metadata-only releases still feed package detail, file lookup, static scan/backfill, and LLM review from `release.files`.

## What This Changes

The issue asks ClawHub to make package inspect, file lookup, and LLM review evaluate actual ClawPack/npm-pack artifact contents instead of only metadata files stored in `release.files`.

## Best Possible Solution

Define and implement a production-safe artifact reparse/backfill for existing npm-pack releases, refresh affected moderation signals, and document the final ClawPack inspect and scan scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: any existing npm-pack release whose stored `release.files` contains only metadata will still make inspect, file lookup, static scan refreshes, and LLM review see only those metadata entries.

## Solution Assessment

Partially. The current future-upload extraction path is a good narrow fix for new publishes, but it does not solve historical releases until artifacts are reparsed or the affected surfaces read selected entries from stored tarballs and stale scan results are refreshed.
