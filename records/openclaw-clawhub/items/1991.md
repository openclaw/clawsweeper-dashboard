## Summary

Keep open: current main still routes ClawPack package review and file metadata through the metadata-only `release.files` list, so the false missing-code verdict remains source-reproducible.

## What This Changes

The issue asks ClawHub package review surfaces to distinguish npm-pack ClawPack artifact contents from the metadata-only release file list so complete code-plugin packages are not flagged as missing code.

## Best Possible Solution

Preserve the intentional metadata-only scan policy unless maintainers decide to expand it, but make the package review context explicitly include ClawPack artifact facts or bounded selected artifact entries so complete npm-pack releases are not treated as hollow.

## Reproduction Assessment

Yes. Source inspection against current main shows ClawPack publish stores only two metadata files in `release.files`, while package file/detail and LLM review surfaces consume that list; the live model was not run.

## Solution Assessment

Yes. The narrowest maintainable fix is to keep artifact storage semantics intact while adding accurate ClawPack artifact context and prompt wording, with tests covering the current false missing-code path.

## Repair Work Prompt

Fix issue #1991 on current main. The observable bug is that npm-pack/ClawPack code-plugin releases store only metadata files in `release.files`, but package LLM/static review and package inspect/file context can present that metadata-only list as if the downloadable artifact has no implementation files, causing false suspicious missing-code verdicts. Keep ClawPack download/storage behavior intact and do not add an unbounded deep tarball scan. Add a bounded representation of ClawPack artifact truth to review context: at minimum include artifact kind, npm file count/unpacked size/digests and prompt wording that distinguishes registry metadata files from actual npm-pack artifact entries; if parsing selected artifact entries, apply existing text/size limits and prompt-injection neutralization. Update tests to cover a ClawPack tarball containing `README.md`, `SKILL.md`, `dist/index.js`, and metadata files, and assert the LLM prompt/review context no longer implies the implementation is missing solely because `release.files` has two files. Consider whether `package inspect --files` output should explicitly label metadata files versus artifact facts without breaking legacy zip behavior. Include a `CHANGELOG.md` entry if the user-facing review or CLI/API output changes. Do not change unrelated package moderation policy, VirusTotal behavior, or trusted publish flow.
