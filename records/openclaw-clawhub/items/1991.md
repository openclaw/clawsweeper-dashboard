## Summary

Keep open: current main stores extracted entries for new multipart ClawPack publishes, but live historical npm-pack releases still have metadata-only inventories and the file, static scan, and LLM review paths still consume that stored `release.files` list.

## What This Changes

Make ClawPack/npm-pack package review and file/detail surfaces use artifact-derived entries so complete code-plugin packages are not flagged as missing implementation files.

## Best Possible Solution

Use one artifact-derived ClawPack inventory for package detail, raw file lookup, static scan, and LLM review, then backfill and rescan affected historical npm-pack releases under an explicit moderation plan.

## Reproduction Assessment

Yes. Live registry checks reproduce complete npm-pack artifacts with metadata-only file inventories, and source inspection shows the current detail, raw file, static scan, and LLM review paths still consume that stored inventory.

## Solution Assessment

Unclear. Current main's new-publish extraction is the right building block, but the remaining historical repair needs maintainer choice between backfilling stored files, reading selected tarball entries on demand, and how to update moderation state.
