## Summary

Keep open: current main now stores extracted entries for new ClawPack publishes, but live historical npm-pack releases still have metadata-only inventories and current detail, raw file, static scan, and LLM review paths consume those inventories.

## What This Changes

Request that ClawPack/npm-pack package review and file surfaces use artifact-derived entries instead of metadata-only release files so complete code plugins are not flagged as missing code.

## Best Possible Solution

Adopt a maintainer-approved artifact-derived ClawPack inventory for detail, raw file lookup, static scan, and LLM review, then repair and rescan affected historical releases with an audit trail.

## Reproduction Assessment

Yes. Live ClawHub API checks show npm-pack artifacts with 11 or 22 tarball files but only two stored release files, and current source inspection shows file/detail/LLM/static review paths consume release.files.

## Solution Assessment

Unclear. Artifact-derived inventory is likely the accurate fix, but current specs intentionally limit ClawPack static/LLM inputs and historical rescans can alter public moderation state, so maintainers should choose the repair policy first.
