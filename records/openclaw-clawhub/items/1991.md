## Summary

Keep open: the live mismatch still reproduces for published npm-pack releases, and current main still routes file lookup, static scan, and LLM context through `release.files`. New uploads now store extracted ClawPack entries, but historical rows and the current moderation spec still need maintainer security policy before changing public review outcomes.

## What This Changes

The issue requests that ClawPack/npm-pack code-plugin review surfaces evaluate enough tarball entries or artifact metadata to avoid false missing-code suspicious verdicts for complete packages.

## Best Possible Solution

Choose the approved ClawPack review source of truth, then either backfill/extract and rescan affected npm-pack releases or make APIs and prompts clearly state metadata-only scope.

## Reproduction Assessment

Yes. Live API checks for `@openclaw/codex-claw@2026.5.6` still show a two-file inventory while the artifact download contains implementation files, and current-main source shows the review surfaces consume `release.files`.

## Solution Assessment

Unclear. Storing extracted entries for new uploads is useful, but the remaining historical/public moderation behavior needs maintainer choice between backfill-and-rescan, on-demand artifact reads, or explicitly preserving metadata-only review scope.
