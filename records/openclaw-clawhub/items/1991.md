## Summary

Keep open: the live package still reproduces the ClawPack inventory gap for historical npm-pack rows, and current main only fixes parsed file storage for new uploads while scan scope/backfill remains a moderation policy decision.

## What This Changes

The issue reports that ClawPack npm-pack package review and file surfaces can see only metadata files even when the downloadable artifact contains implementation files, producing false missing-code suspicious verdicts.

## Best Possible Solution

Define the authoritative ClawPack file and scan inventory, then backfill stored file metadata and rescan affected releases or explicitly label historical detail/file/LLM surfaces as metadata-only.

## Reproduction Assessment

Yes. The live ClawHub API for @openclaw/codex-claw@2026.5.6 shows npmFileCount 11 with only two stored files and a suspicious missing-code LLM summary, while the artifact download contains the implementation files.

## Solution Assessment

Unclear. Persisting parsed entries for new uploads is the right partial fix, but the remaining historical inventory, file endpoint, and public moderation verdict behavior needs a maintainer decision before implementation.
