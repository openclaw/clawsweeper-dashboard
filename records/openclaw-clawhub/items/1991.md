## What This Changes

The issue asks ClawHub package detail/file and LLM review surfaces to evaluate actual npm-pack ClawPack artifact entries rather than only metadata files.

## Best Possible Solution

Define the authoritative ClawPack file inventory for package surfaces and moderation review, then backfill/rescan affected releases or clearly label historical metadata-only records.

Yes. The live API for @openclaw/codex-claw@2026.5.6 reports an npm-pack artifact with 11 files while version.files has only two metadata files, and current main’s file/detail and LLM paths read that stored release.files inventory.

## Solution Assessment

Unclear. Storing parsed entries for new uploads is the right direction, but changing historical inventories and moderation verdicts needs a maintainer decision on backfill, artifact-on-demand review, and rescans.
