## Summary

Keep open: current main appears to fix future ClawPack publishes by storing parsed tarball entries, but historical npm-pack rows with metadata-only `release.files` can still drive package detail, file lookup, static dataset, and LLM review surfaces into the false missing-code verdict described here.

## What This Changes

The issue asks ClawHub package detail/file/LLM review surfaces to evaluate actual npm-pack ClawPack artifact contents instead of metadata-only release files for code-plugin releases.

## Best Possible Solution

Backfill or reparse existing npm-pack ClawPack artifacts into release file metadata, refresh affected package scan verdicts, and update the security/API/CLI docs to match the chosen scan scope.

## Reproduction Assessment

Yes. A release with `artifactKind: "npm-pack"`, stored ClawPack bytes, and metadata-only `release.files` will still reproduce the limited package detail/file/LLM review surface because those paths read `release.files` rather than reparsing the artifact.

## Solution Assessment

Partially. The current-main extraction for new publishes is the right narrow source fix for future uploads, but the issue is not solved until maintainers choose and run a historical artifact reparse/backfill plus scan refresh path.
