## Summary

Keep open: this is source-reproducible on current main because future ClawPack publishes now store extracted entries, but historical metadata-only npm-pack releases still feed package detail, file lookup, and LLM review through `release.files`.

## What This Changes

Make ClawPack/npm-pack package inspection and LLM review evaluate artifact contents rather than metadata-only release files so complete code-plugin bundles are not flagged as missing implementation.

## Best Possible Solution

Choose a production-safe ClawPack artifact reparse/backfill or artifact-read plan, refresh affected moderation signals, and update API/security docs to match the final inspect and scan scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: any npm-pack release whose stored `release.files` contains only `package.json` and `openclaw.plugin.json` will make inspect, file lookup, and LLM review see only those entries.

## Solution Assessment

Unclear as a complete fix. The current new-publish extraction path is the right narrow direction for future releases, but the remaining solution needs maintainer choice on historical data repair and moderation refresh scope.
