## Summary

Keep open: current main stores full parsed ClawPack entries for new uploads, but historical npm-pack rows can still have metadata-only `release.files`, and package detail, file, and LLM review surfaces still read from that field. The remaining work affects public ClawScan verdicts and the current security-moderation policy, so it needs maintainer direction.

## What This Changes

The issue requests that ClawPack/npm-pack package review surfaces evaluate real artifact contents instead of metadata-only `release.files` so complete code-plugin packages are not falsely marked suspicious for missing code.

## Best Possible Solution

Define the authoritative ClawPack inventory for package detail, file serving, and ClawScan review, then backfill or clearly label historical metadata-only releases and rescan affected verdicts under that policy.

## Reproduction Assessment

Yes. The live registry reproduces the mismatch, and current main still shows package detail, file serving, and package LLM review deriving their inputs from `release.files`, which reproduces the failure mode for any historical npm-pack row whose stored files are metadata-only.

## Solution Assessment

Unclear. Persisting parsed ClawPack entries for new uploads is a good storage direction, but the remaining historical inventory repair and public ClawScan rescan behavior require a maintainer policy decision before a narrow implementation is safe.
