## Summary

Keep open: the live registry still reproduces the metadata-only ClawPack file inventory and stale missing-code LLM verdict for historical npm-pack releases, while current main only mitigates new uploads and leaves the historical rescan policy unresolved.

## What This Changes

Make ClawPack/npm-pack package file and LLM review surfaces evaluate actual tarball entries instead of only metadata files so complete code-plugin releases are not falsely flagged missing code.

## Best Possible Solution

Define the authoritative ClawPack review/file inventory policy, then backfill historical release.files and rescan affected package LLM verdicts or explicitly label historical surfaces as metadata-only.

## Reproduction Assessment

Yes. The live ClawHub API reproduces the mismatch for @openclaw/codex-claw@2026.5.6, and current main still routes historical file lookup and LLM review through release.files.

## Solution Assessment

Unclear. Persisting parsed ClawPack entries for new uploads is a useful partial fix, but historical backfill/rescan behavior and the metadata-only scan policy need maintainer direction before a repair PR changes public moderation outcomes.
