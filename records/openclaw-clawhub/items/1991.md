## Summary

Keep open: this is a reproduced ClawPack inventory and moderation gap, and current main only appears to fix parsed file storage for new uploads while historical package rows and rescan policy remain unresolved.

## What This Changes

The issue reports that ClawPack/npm-pack package file inspection and LLM review use metadata-only release files instead of the actual tarball entries, causing false missing-code suspicious verdicts.

## Best Possible Solution

Define the authoritative ClawPack file and scan inventory, then either backfill and rescan affected releases or explicitly label historical detail/file/LLM surfaces as metadata-only.

## Reproduction Assessment

Yes. The live registry shows @openclaw/codex-claw@2026.5.6 has an npm-pack artifact with 11 files while version.files exposes only two metadata files, the LLM verdict is suspicious for missing code, and README.md returns 404 through the file endpoint.

## Solution Assessment

Unclear. Current main's parsed-entry storage is the right direction for new uploads, but historical releases need a maintainer choice between inventory backfill/rescan, artifact-on-demand review, or explicit metadata-only wording.
