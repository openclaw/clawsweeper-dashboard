## Summary

Keep open: the live package surfaces still reproduce the metadata-only ClawPack inventory and stale missing-code LLM verdict for npm-pack releases, while current main only fixes extracted-file storage for new uploads and leaves historical scan/file policy unresolved.

## What This Changes

Ensure ClawPack/npm-pack package LLM review and file/detail surfaces evaluate or clearly distinguish actual artifact entries instead of treating metadata-only release files as complete package contents.

## Best Possible Solution

Define the authoritative ClawPack review/file source of truth, update the spec and public API wording, then either backfill and rescan affected npm-pack releases or explicitly label historical surfaces as metadata-only.

## Reproduction Assessment

Yes. The live ClawHub API shows @openclaw/codex-claw@2026.5.6 with npmFileCount 11, only two stored files, a missing-code suspicious LLM summary, and a README.md file 404 while the downloaded artifact contains the implementation files.

## Solution Assessment

Unclear. Current main’s extracted-file storage for new uploads is a useful partial fix, but the remaining historical inventory, file endpoint, and rescan behavior need a maintainer policy choice before a narrow implementation is safe.
