## Summary

Keep open: current main still uses `release.files` for package version files, raw file lookup, static scan inputs, and LLM prompt inputs, while live npm-pack releases still show metadata-only inventories and suspicious missing-code verdicts. The remaining fix is security/moderation policy work, not a conservative auto-close for a ClawHub issue.

## What This Changes

The issue asks ClawHub to prevent npm-pack ClawPack package review from treating metadata-only `release.files` as the full artifact contents.

## Best Possible Solution

Define the approved ClawPack review source of truth, then update file/detail/prompt behavior and rescan or clearly mark historical metadata-only rows so moderation verdicts cannot imply full artifact review when only metadata was inspected.

## Reproduction Assessment

Yes. The live API reproduces the mismatch for `@openclaw/codex-claw@2026.5.6`, and current main source shows package file lookup, static scan, and LLM review still consume `release.files`.

## Solution Assessment

Unclear. Storing parsed entries for new publishes is a useful building block, but the best complete fix needs maintainer choice between historical inventory backfill, on-demand tarball reads, prompt clarification, and rescans.
