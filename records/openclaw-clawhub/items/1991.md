## Summary

Keeping this open: live ClawPack package versions still expose only metadata files to the package detail/file and LLM review surfaces even though the npm-pack artifact has more files, and current specs say the scan input policy is intentionally metadata-only for now.

## What This Changes

The issue asks ClawHub to stop treating npm-pack ClawPack code-plugin releases as missing implementation files when the downloadable artifact contains the code.

## Best Possible Solution

Set the ClawPack scan/file-surface policy, then either make the package file/LLM surfaces artifact-aware or backfill affected npm-pack releases and rescan them without making metadata-only inputs look like missing code.

## Reproduction Assessment

Yes for the live registry and source path: live npm-pack releases expose only metadata files while the artifact reports a larger file count, and current package file/LLM code reads release.files. I did not establish a fresh failing current-main publish because current main also has tests showing new ClawPack uploads store parsed entries.

## Solution Assessment

Unclear as an automation target. The narrow technical paths are clear, but the spec explicitly documents metadata-only ClawPack scan inputs, so maintainers should first decide whether to change scan depth, prompt wording, API file behavior, or backfill existing releases.
