## Summary

Keep open: current main only partially addresses the report. New multipart ClawPack publishes now persist parsed tarball entries, but historical npm-pack releases whose stored `release.files` contains only metadata still feed incomplete data to inspect, file lookup, static scan, LLM review, and export surfaces.

## What This Changes

Request that ClawPack/npm-pack package inspection and review evaluate actual artifact contents or clearly distinguish registry metadata files from tarball entries.

## Best Possible Solution

Choose a production-safe artifact reparse or tarball-backed lookup path for existing npm-pack releases, refresh affected static/LLM moderation signals, and update API/CLI/security docs to match the final ClawPack file-surface semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an npm-pack release with `clawpackStorageId` but metadata-only `release.files` will still make detail, file lookup, CLI inspect, static scan, and LLM review see only those stored metadata entries.

## Solution Assessment

Partially. The current new-publish extraction path is a useful building block, but the remaining issue needs historical release handling and an explicit moderation rescan policy before it is fully solved.
