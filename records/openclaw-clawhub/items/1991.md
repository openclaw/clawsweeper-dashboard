## Summary

Keep open: current main stores parsed ClawPack entries for new uploads, but affected live npm-pack releases still have metadata-only inventories and the inspect, raw-file, static-scan, and LLM-review paths still depend on `release.files`.

## What This Changes

The issue asks ClawHub to make npm-pack ClawPack code-plugin inspection and LLM review evaluate the artifact file inventory instead of treating metadata-only `release.files` as the complete package.

## Best Possible Solution

Use a single artifact-derived ClawPack inventory for version detail, raw file lookup, static scan, and LLM review, then repair and audit affected historical npm-pack releases with an explicit rescan plan.

## Reproduction Assessment

Yes. Live registry checks show affected npm-pack releases whose artifacts contain implementation files while `version.files` exposes only metadata, and current main still routes inspect, raw file lookup, LLM review, and static rescan through `release.files`.

## Solution Assessment

Unclear. The new-upload extraction path is the right building block, but the remaining historical inventory repair and scanner rerun semantics need maintainer moderation policy before a safe code change.
