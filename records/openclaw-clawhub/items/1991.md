## Label Justifications

- `P2`: Historical ClawPack packages can receive false suspicious verdicts and incomplete inspection, while downloads and new-upload ingestion remain functional.
- `impact:data-loss`: Implementation entries present in stored artifacts are silently absent from historical release records consumed by inspection and moderation.

## Summary

Current main correctly materializes complete ClawPack file rows for new uploads, but it does not retroactively repair the historical metadata-only rows demonstrated in the report; the open closing PR at https://github.com/openclaw/clawhub/pull/2583 owns that remaining migration and must be validated and merged before this issue is no longer necessary.

## What This Changes

Repair ClawPack inspection and LLM review so npm-pack implementation files are represented instead of exposing only package metadata.

## Best Possible Solution

Land a dry-run-first, resumable migration that verifies stored artifacts, reconstructs complete historical release file rows, refreshes dependent fingerprints and extracted metadata, and queues fresh scans without changing the corrected ingestion path.

## Reproduction Assessment

Yes. The reporter provides verified live examples where 11- and 22-file artifacts expose only two stored files, and current source confirms those historical release.files rows are the inputs used by inspection and LLM review.

## Solution Assessment

Yes in direction. Preserving current artifact materialization for new uploads and repairing historical rows once through a guarded migration is narrower and less drift-prone than making every inspection and scanner consumer independently parse tarballs.
