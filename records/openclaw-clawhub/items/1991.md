## Label Justifications

- `P2`: The defect produces incomplete inspection and false suspicious moderation findings for npm-pack plugins, but its blast radius is bounded to ClawPack package surfaces.
- `impact:data-loss`: Implementation file entries present in verified artifacts are omitted from the persisted inventory used by inspection and review, effectively hiding package data from those surfaces.

## Summary

Current release v0.21.0 still has the reported split: npm-pack artifacts retain their full contents, but package LLM evaluation consumes the persisted `release.files` inventory, while the durable moderation spec explicitly limits ClawPack scan inputs to metadata. The issue remains necessary, and the open closing PR should be reviewed and either landed or replaced before this issue closes.

## What This Changes

Make ClawPack npm-pack implementation files available to package inspection and LLM review, including safe repair and rescanning of historical releases with metadata-only file rows.

## Best Possible Solution

Adopt one canonical artifact-entry policy for new ClawPack uploads and inspection/LLM review, then use a guarded, cursor-based migration to repair historical releases, recompute derived state, and rescan them with real Convex runtime validation.

## Reproduction Assessment

Yes. Current main explicitly keeps ClawPack scan inputs metadata-only and the evaluator reads `release.files`; combined with the reporter’s verified artifact-versus-inspection comparison, this is a high-confidence current-main reproduction.

## Solution Assessment

Yes in direction, but completion requires both forward ingest and historical repair. Persisting selected parsed artifact entries and safely resetting/rescanning affected releases is narrower and more maintainable than teaching each consumer to independently unpack stored tarballs.
