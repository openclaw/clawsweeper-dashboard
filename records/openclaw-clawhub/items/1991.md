## Summary

The issue remains necessary for historical ClawPack releases: current main stores full parsed entries for new uploads, but inspection and LLM review still consume persisted `release.files`, so older two-row releases remain incomplete. The open closing PR is the canonical repair path and must be reviewed rather than closing the issue before merge.

## What This Changes

Repair historical npm-pack ClawPack releases whose persisted file rows omit implementation files contained in the stored artifact, causing incomplete inspection and false LLM moderation findings.

## Best Possible Solution

Land a maintainer-approved, runtime-validated migration that reconstructs canonical historical file rows from each authoritative stored artifact, retains dry-run and resumable safeguards, and queues fresh moderation for repaired releases.

## Reproduction Assessment

Yes at high confidence from source and the reporter’s verified artifact comparison: historical `release.files` rows feed both inspection and evaluation while the stored tarball contains additional implementation entries. A real deployment run is needed to validate the migration, not to establish the defect.

## Solution Assessment

Yes in architectural direction: repairing the canonical persisted rows from the authoritative artifact avoids divergent bypass logic in every consumer. Maintainers still need to approve the historical rewrite, fingerprint update, moderation reset, and rollout contract.
