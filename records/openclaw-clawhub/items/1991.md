## Summary

Keep open: current main partially fixes future ClawPack publishes by storing parsed tarball entries, but existing npm-pack releases with metadata-only `release.files` still feed incomplete package inspect, file lookup, static scan, and LLM review surfaces.

## What This Changes

Report that ClawPack/npm-pack code-plugin releases can be falsely reviewed as missing implementation because registry and LLM review surfaces use metadata-only `release.files` instead of the full tarball contents.

## Best Possible Solution

Choose a production-safe ClawPack artifact reparse or tarball-backed lookup path, refresh affected static/LLM moderation results, and align API/CLI/docs with the final scan scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an existing npm-pack release with full artifact metadata but metadata-only `release.files` will still list only those files, fail inline source lookups, and feed only those files into current static/LLM review.

## Solution Assessment

Partially. The current extracted-entry path is the right direction for new publishes, but it does not repair historical rows or already-recorded review verdicts without a backfill, artifact-backed lookup, and rescan policy.
