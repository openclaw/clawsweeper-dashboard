## Summary

Keep open: current main fixes new multipart ClawPack publishes by storing extracted entries, but live historical npm-pack releases still have metadata-only inventories and the detail, raw file, static scan, and LLM paths still consume `release.files`.

## What This Changes

The issue asks ClawHub to make npm-pack ClawPack file inventory and LLM/static review use artifact contents instead of treating complete code-plugin tarballs as metadata-only bundles.

## Best Possible Solution

Adopt one artifact-derived ClawPack inventory for package detail, raw file lookup, static scan, and LLM review, then repair affected historical releases with an explicit audited rescan plan.

## Reproduction Assessment

Yes. Live API checks reproduce metadata-only `version.files` and a 404 raw README while the downloaded artifact contains implementation files, and current main shows the affected consumers still read `release.files`.

## Solution Assessment

Unclear. The new-publish extraction fix is the right building block, but maintainers still need to choose between backfilling stored inventories, reading selected files from tarballs on demand, and how to rescan historical moderation state.
