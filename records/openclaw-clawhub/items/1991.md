## Summary

Keep open: current main fixes future ClawPack publishes by storing extracted entries, but historical npm-pack releases with metadata-only `release.files` still feed incomplete data to package inspect/file lookup and package LLM/static scan paths; ClawHub policy also keeps issues open for maintainer handling.

## What This Changes

The issue requests that ClawPack/npm-pack releases expose actual artifact entries to package detail/file lookup and LLM review instead of only metadata files.

## Best Possible Solution

Make package detail, file lookup, static scan, and LLM review share one artifact-aware ClawPack file inventory, then apply a maintainer-approved repair and rescan plan for affected historical npm-pack releases.

## Reproduction Assessment

Yes. Source inspection shows the failure path for historical npm-pack rows: package detail/file lookup and LLM/static scans consume `release.files`, and the issue comments provide live registry examples where that stored list is metadata-only while the verified tarball is complete.

## Solution Assessment

No. Current main is only a partial fix for future publishes; the narrow maintainable solution still needs a shared ClawPack inventory plus an explicit historical repair and moderation refresh policy.
