## Summary

Keep open: current main partially fixes future ClawPack publishes by storing extracted entries, but historical npm-pack releases with metadata-only `release.files` still make package inspect/file lookup, static scan, and LLM review see an incomplete file surface.

## What This Changes

The issue reports that npm-pack ClawPack code-plugin releases are reviewed using only metadata entries instead of the full tarball contents, producing false missing-code suspicious verdicts.

## Best Possible Solution

Make package detail, file lookup, static scan, and LLM review share one authoritative ClawPack file inventory for npm-pack releases, then refresh affected historical moderation records under an explicit maintainer policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a ClawPack release with `clawpackStorageId` but metadata-only `release.files` will make version detail, `/file`, CLI inspect, static scan, and LLM review read only those stored metadata entries.

## Solution Assessment

No, not fully. The current new-upload extraction path is useful, but it does not repair historical rows or define how completed moderation results should be rescanned.
