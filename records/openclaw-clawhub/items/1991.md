## Summary

Keep open: current main partially fixes new multipart ClawPack publishes by storing parsed tar entries, but historical metadata-only npm-pack releases still drive inspect, file lookup, static scan, and LLM review from incomplete `release.files` data. The current moderation spec also needs maintainer reconciliation with the full-entry publish path before changing scan/backfill behavior.

## What This Changes

The issue asks ClawPack/npm-pack package review and inspect surfaces to distinguish registry metadata files from actual tarball entries so complete code-plugin artifacts are not marked suspicious for missing code.

## Best Possible Solution

Reconcile the ClawPack scan policy, then make package detail, file lookup, static scan, and LLM review use either artifact-derived inventories or explicit metadata-only semantics consistently, with an audited historical repair/rescan path.

## Reproduction Assessment

Yes. A current-main npm-pack release with `clawpackStorageId` but metadata-only `release.files` will make package detail, raw file lookup, static scan, and LLM review consume only those metadata entries, matching the reporter's live CLI examples.

## Solution Assessment

Unclear. Storing parsed entries for new uploads is a useful building block, but the durable moderation spec and historical release/backfill policy need maintainer resolution before this can be treated as a narrow code fix.
