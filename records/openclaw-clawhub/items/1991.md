## Summary

Keep open: current main repairs new ClawPack uploads by storing all parsed tarball entries, but there is still no source evidence that existing truncated npm-pack releases have their `release.files` rows backfilled or their stale review verdicts repaired.

## What This Changes

The issue reports that ClawPack/npm-pack package review and file-inspection surfaces expose only metadata files, causing complete code-plugin artifacts to be flagged as missing implementation files.

## Best Possible Solution

Backfill existing npm-pack release file metadata from stored ClawPack tarballs, keep the new-upload full-entry behavior, and rescan or clear stale review verdicts that were based on metadata-only rows.

## Reproduction Assessment

Yes from source: create or inspect an npm-pack release whose `release.files` contains only package.json and openclaw.plugin.json while the tarball has README/source/runtime entries; the file route and legacy package LLM evaluator consume `release.files`, so README/source lookups and prompt context stay truncated. The issue also provides live package examples matching that shape.

## Solution Assessment

No: the current main fix is only sufficient for newly uploaded ClawPack releases. The maintainable fix should also repair historical rows and stale review output instead of leaving affected published artifacts dependent on the old truncated metadata.
