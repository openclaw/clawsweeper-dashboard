## Summary

Keep open: current main still has the source-reproducible ClawPack npm-pack review blind spot, and there is no merged fix that makes package inspection or LLM evaluation distinguish metadata-only release files from the full downloadable artifact.

## What This Changes

The issue requests that ClawPack npm-pack package inspection and LLM review use artifact entries or artifact facts instead of treating `release.files` metadata as the complete package contents.

## Best Possible Solution

Keep ClawPack tarball storage and download behavior intact, but add an explicit ClawPack artifact review/file context that separates registry metadata from artifact entries and prevents metadata-only prompts from producing missing-code verdicts.

## Reproduction Assessment

Yes. Source inspection shows the ClawPack publish path stores only two metadata files in `release.files`, and package inspect/file plus LLM evaluation read from that same list; I did not run the live LLM path.

## Solution Assessment

Yes. The narrowest maintainable fix is to label the scan surface accurately and pass artifact entry facts or selected parsed artifact text into package review surfaces, rather than changing artifact storage or broadening deep scans by default.

## Repair Work Prompt

Fix the ClawPack npm-pack metadata-only review blind spot reported in this issue. On current main, multipart ClawPack publish stores only `package.json` and `openclaw.plugin.json` in `release.files`, but package inspect/file and `evaluatePackageReleaseWithLlm` treat that list as the package review/file surface. Add a narrow ClawPack artifact context that, for `artifactKind === "npm-pack"` with `clawpackStorageId`, parses or summarizes the stored tarball as needed to expose artifact entry facts and selected text entries to the package LLM/file surfaces. Preserve existing tarball storage, download, integrity, and VirusTotal behavior; do not add a deep static/LLM scan over every tarball file unless it is necessary for the narrow false-verdict fix. The LLM prompt must explicitly distinguish registry metadata files from ClawPack artifact entries and must not call metadata-only files the complete source. Add regression coverage for a ClawPack tarball containing README/source/runtime entries where `release.files` has only metadata but the review/file context no longer reports missing implementation incorrectly. Update docs only if the public inspect/review behavior changes.
