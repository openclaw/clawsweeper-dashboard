## Summary

Current main still lacks reliable plugin package search recall for display names or creator handles: the live API reproduces the Vibe Prospecting miss, and source only directly recalls normalized package name, runtime id, topic, and category while creator handle is not scored.

## What This Changes

The issue asks ClawHub plugin search to find a plugin by displayed plugin name or creator, not only by package/runtime id.

## Best Possible Solution

Add indexed/direct recall for plugin package `displayName` and `ownerHandle`, keep relevance ordering consistent with `specs/search-relevance.md`, and add regression coverage for the reported plugin queries.

Yes, with source-backed confidence: live `/api/v1/plugins/search` finds `vpai-plugin` but not `Vibe Prospecting` or `vibeprospecting`, and current main explains why those recall paths are missing. ## Solution Assessment

Yes, the narrow maintainable fix is in backend package/plugin search recall and tests, not UI-only filtering. If maintainers want creator display-name matching beyond `ownerHandle`, that should be added deliberately through digest denormalization and coverage.
