## Summary

Keep open. Current main intentionally treats publish-time `latest` as semver-managed for lower or prerelease versions, and changing `--tags latest` into an override conflicts with the backport guard from #1832, so this needs maintainer product judgment.

## What This Changes

Allow publish-time `--tags latest` to promote a prerelease or lower semver version to the displayed latest release.

## Best Possible Solution

Define the contract explicitly: either document publish-time `latest` as semver-managed with retagging as the override path, or add an intentional override flow with backend, CLI, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: publish a lower or prerelease semver with `tags: ["latest"]`, and `insertVersion` drops that tag unless the version is semver-greater; the backport test asserts the same behavior.

## Solution Assessment

Unclear. The requested behavior is plausible for rolling registries, but it conflicts with the intentional #1832 backport guard, so maintainers should choose between clearer docs and a deliberate override path.
