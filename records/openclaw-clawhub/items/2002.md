## Summary

Keep open. Current main intentionally preserves the semver-managed latest pointer on lower or prerelease publishes even when `latest` is supplied, so this needs maintainer product/docs judgment rather than conservative cleanup.

## What This Changes

The issue requests that `skill publish --tags latest` repoint the visible latest version to the published prerelease or lower version.

## Best Possible Solution

Define the contract explicitly: either document publish-time `latest` as semver-managed with a retag workflow, or add a deliberate override path with backend, CLI, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: publish a lower or prerelease semver with `tags: ['latest']`, and `insertVersion` skips that tag while tests assert the previous latest pointer remains selected.

## Solution Assessment

Unclear. The requested behavior conflicts with the intentional #1832 semver guard, so the maintainable fix depends on whether maintainers want publish-time overrides or clearer docs plus the existing retag flow.
