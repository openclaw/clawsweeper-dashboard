## Summary

Keep open. Current main intentionally preserves the semver-managed latest pointer on lower/prerelease publishes, even when `latest` is supplied in publish tags; changing that behavior needs maintainer product judgment because it conflicts with the backport guard from #1832.

## What This Changes

The issue requests that `skill publish --tags latest` move the displayed latest version to a prerelease/lower-semver publish instead of preserving the current stable latest pointer.

## Best Possible Solution

The durable end state is an explicit contract: either document publish-time `latest` as semver-managed and direct users to retagging, or add a deliberate override path with backend, CLI, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: publish a lower/prerelease semver with `tags: ['latest']`, and `insertVersion` skips that tag while tests assert the previous latest pointer remains selected.

## Solution Assessment

Unclear. The requested behavior conflicts with the intentional #1832 semver guard, so the best solution depends on whether maintainers want publish-time overrides or clearer docs plus the existing retag flow.
