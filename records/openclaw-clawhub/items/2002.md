## Summary

Keep open. Source inspection supports the reported behavior: publish currently skips caller-provided `latest` when the incoming version is not semver-greater, while docs and the tag-management path still describe `latest` as a movable tag; because existing tests frame the current behavior as backport/security protection, this needs maintainer policy judgment before a fix.

## What This Changes

The issue requests that `clawhub skill publish --tags latest` move the visible `latest` tag and displayed latest version to the newly published prerelease.

## Best Possible Solution

Maintainers should decide the intended `latest` policy, then either honor explicit publish tags with updated latest summary/embedding/moderation tests or document semver-protected publish behavior and provide a clear retag workflow.

## Reproduction Assessment

Yes, by source inspection: with current main, an existing latest of `2.7.1` and an incoming `2.7.0-2-g0d320d1` with `args.tags` containing `latest` will evaluate `isNewLatest` false and skip the explicit latest tag. I did not run an end-to-end publish because this review was read-only.

## Solution Assessment

Unclear as an implementation direction: honoring explicit `latest` matches the tag-pointer model and docs, but it conflicts with current backport/security tests. The narrow maintainable path starts with a maintainer policy decision, then code/docs/tests should be aligned to that decision.
