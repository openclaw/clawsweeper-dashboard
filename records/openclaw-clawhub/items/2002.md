## Summary

Keeping this open: current main clearly preserves the semver-selected latest version over an explicit publish-time `latest` tag, and existing tests intentionally protect that behavior, but the CLI/docs wording leaves the reporter's expectation plausible.

## What This Changes

The issue requests that `clawhub skill publish --tags latest` move the displayed latest version to the newly published prerelease even when semver ranks an older stable version higher.

## Best Possible Solution

Maintainers should choose one contract and codify it: either allow explicit publish-time `latest` overrides with updated backend tests and docs, or keep semver-protected publish behavior and document a separate retag/rollback path for moving `latest`.

## Reproduction Assessment

Yes, from source inspection: the CLI sends `tags: ["latest"]`, but `insertVersion` drops `latest` unless the new version wins the semver comparison, so a prerelease lower than the current stable remains visible only in history. I did not perform a live publish because this was a read-only review.

## Solution Assessment

Unclear. The reporter's requested behavior is coherent with tag-pointer semantics, but current code and tests intentionally reserve publish-time `latest` for semver promotion, so the narrow fix depends on a maintainer policy decision.
