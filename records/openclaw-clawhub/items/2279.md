## Summary

Keep open: current main still has static homepage chips, and this PR’s new public telemetry path needs security/privacy fixes plus stronger backend proof before merge.

## What This Changes

The PR adds dynamic homepage recommendation topics backed by public search telemetry, daily package stats, rebuild/prune crons, UI wiring, tests, and a search-relevance spec note.

## Best Possible Solution

Keep the dynamic recommendation direction, but gate or approve untrusted search telemetry before homepage publication, make retention cleanup reliable, define aggregate retention policy, and add backend runtime proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows a public/CORS-open POST telemetry path feeding homepage topics at a fixed threshold, and a daily prune job that deletes only one expired batch. The overall PR is a feature, not a bug reproduction.

## Solution Assessment

No. The feature is plausible, but the current patch is not the narrowest maintainable solution until telemetry integrity, retention, and real backend proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Gate telemetry before publishing homepage chips:** `convex/http.ts:69-72`
  - body: The new POST search endpoint accepts public telemetry and the topic builder promotes normalized search terms to homepage chips after only 10 counts. Because the shared response/preflight path allows any origin and dedupe is per client IP, an external site or distributed traffic can farm arbitrary filtered terms into the homepage. Add a first-party origin/referer gate, a diversity/approval threshold, or keep untrusted telemetry out of published chips.
  - confidence: 0.84
- **[P2] Drain all expired search dedupe rows:** `convex/searchTelemetry.ts:190-199`
  - body: The retention job deletes only the first batch of expired dedupe rows and then returns. Since the cron runs once per day with `batchSize: 500`, any expired backlog above that size keeps pseudonymous bucket rows past the intended 14-day boundary. Continue pruning when a full batch is deleted, or move the drain into a bounded action loop.
  - confidence: 0.9
- **[P3] Define retention for aggregate search terms:** `convex/schema.ts:1392-1401`
  - body: `searchQueryDailyStats` stores normalized/display query text and the rebuild reads only the recent window, but the PR adds no prune path or documented persistence policy for those aggregate rows. Private-looking filters reduce risk but do not make accepted user search text retention clear. Add explicit retention or document the intended aggregate lifetime.
  - confidence: 0.7
