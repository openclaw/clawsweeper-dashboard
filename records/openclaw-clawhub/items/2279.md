## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Use a non-reversible telemetry bucket:** `convex/httpApiV1/searchTelemetryV1.ts:45-47`
  - body: `getSearchTelemetryBucketKey` stores a deterministic SHA-256 hash of the raw client IP. For IPv4 this is cheaply enumerable if telemetry rows leak, so the dedupe table still contains a stable client identifier for the retention window despite the aggregate-only intent. Use a server-keyed or rotating bucket, or get explicit maintainer acceptance of that privacy tradeoff.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff includes strong endpoint and publication safeguards, but the dedupe bucket is a deterministic hash of the client IP and should be treated as privacy-sensitive.

Concerns:

- **[medium] Deterministic IP hash in telemetry dedupe:** `convex/httpApiV1/searchTelemetryV1.ts:47`
  - body: The dedupe table stores `sha256(search-telemetry:<ip>)`; that is a stable, unsalted identifier and IPv4 values can be brute-forced from leaked rows, so it is not a strong privacy boundary for telemetry storage.
