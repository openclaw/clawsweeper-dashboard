## Summary

Keep open. Current main still treats user-initiated account deletion as irreversible, blocks the same GitHub OAuth identity during sign-in, and has no audited restore or OAuth-unlink path for this support case; ClawHub cleanup policy also keeps issues open rather than auto-closing them.

## What This Changes

The issue requests restoring a deleted ClawHub account or removing the stale GitHub OAuth binding so the same GitHub account can sign in again.

## Best Possible Solution

Define an explicit, audited account-recovery or re-registration policy that can handle user-initiated deletions without weakening banned or abuse-disabled account blocks, then update Convex auth/account code, UI/docs, and regression tests to match it.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: deleteAccount sets deactivatedAt/purgedAt, and the OAuth afterUserCreatedOrUpdated callback rejects that user on the next GitHub sign-in. I did not run a live OAuth flow, but current tests assert the same blocking path.

## Solution Assessment

No, directly removing the OAuth binding or restoring the account is not the safest standalone fix because current specs intentionally make deletion irreversible and bans must remain blocked. The maintainable solution needs an explicit recovery/unlink policy plus audited implementation and tests.
