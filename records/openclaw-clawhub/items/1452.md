## Summary

Keep open. This is an issue in openclaw/clawhub, and the cleanup policy only permits auto-closing PRs; current main still treats user-initiated deletion as irreversible and blocks the same GitHub OAuth identity from signing in again, so resolving this requires maintainer account-recovery/security policy rather than conservative cleanup.

## What This Changes

The issue asks maintainers to remove a stale GitHub OAuth binding or restore a deleted ClawHub account so the same GitHub account can sign in again.

## Best Possible Solution

Define an audited recovery or re-registration flow that distinguishes self-deleted accounts from banned or abuse-disabled accounts, then update Convex auth/account code, operator UI/docs, and regression tests to enforce that policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: deleteAccount sets deactivatedAt/purgedAt, and the OAuth callback rejects that user on the next GitHub sign-in; I did not run a live OAuth flow.

## Solution Assessment

No. A one-off OAuth binding removal or account restoration is not the best durable fix because the repository currently documents deletion as irreversible and needs an audited policy split from banned-account handling.
