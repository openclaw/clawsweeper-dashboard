## Summary

Keep open. This is an open ClawHub issue, and current main still treats user-initiated deletion as irreversible: the OAuth callback rejects deactivated accounts instead of unlinking the GitHub binding or creating a fresh account, so resolving this needs maintainer account-recovery and security policy.

## What This Changes

The issue asks ClawHub maintainers to remove a stale GitHub OAuth binding or restore a deleted account so the same GitHub identity can sign in again.

## Best Possible Solution

Define an audited recovery or re-registration flow that can unlink/rekey only eligible self-deleted accounts while keeping banned and abuse-disabled identities blocked, then update Convex auth/account code, UI/docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection shows deleteAccount marks the user deactivated/purged and the OAuth callback rejects that matched user on the next sign-in; existing tests cover the same rejection path, though I did not run a live GitHub OAuth flow.

## Solution Assessment

No. One-off OAuth binding removal or account restoration is not the best durable fix because it bypasses the repository's irreversible-deletion policy and the necessary split between self-deleted, banned, and abuse-disabled accounts.
