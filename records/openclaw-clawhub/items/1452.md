## Summary

Keep open. Current main still intentionally blocks same-GitHub sign-in after user-initiated deletion, and openclaw/clawhub cleanup policy does not auto-close issues; resolving this needs maintainer account-recovery and security policy rather than conservative cleanup.

## What This Changes

The issue asks ClawHub maintainers to remove a stale GitHub OAuth binding or restore a deleted account so the same GitHub identity can sign in again.

## Best Possible Solution

Define an explicit audited recovery or re-registration flow that separates self-deleted accounts from banned/abuse-disabled accounts, then update Convex auth/account code, UI/docs, and regression tests to match.

## Reproduction Assessment

Yes. Source inspection shows deleteAccount sets deactivatedAt/purgedAt and the OAuth callback rejects that user on the next sign-in, with tests covering the same path; I did not run a live OAuth flow.

## Solution Assessment

No. Removing the OAuth binding or restoring the account as a one-off is not the safest durable fix because the current policy intentionally makes deletion irreversible and banned identities must remain blocked.
