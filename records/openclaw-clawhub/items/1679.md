## Summary

The issue remains necessary because production still reproduces the reported 404s for bundled skill names, and no merged or current-main ClawHub change makes absent registry records resolve. The evidence points to a real setup/CLI boundary problem: OpenClaw bundles the skills, while ClawHub correctly serves only published public records; maintainers should decide the canonical cross-repository behavior rather than create unowned aliases.

## What This Changes

Make installation of bundled OpenClaw skills avoid misleading ClawHub 404s and provide canonical spelling or availability guidance.

## Best Possible Solution

Define one explicit contract: bundled skills remain locally available through OpenClaw, registry installation is used only for published ClawHub packages, and the CLI reports bundled status plus spelling/search suggestions instead of a raw 404.

## Reproduction Assessment

Yes. Direct requests against the production API reproduce the four submitted 404s, and the successful `himalaya` response confirms that one submitted name is misspelled.

## Solution Assessment

No, adding ClawHub aliases alone is not the best demonstrated solution. The narrow maintainable outcome is to make OpenClaw distinguish its bundled skill inventory from registry packages and use ClawHub search only for genuine external installs.
