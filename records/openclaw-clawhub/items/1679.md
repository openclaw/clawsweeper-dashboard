## Summary

Keep open: current main still returns `404 Skill not found` when an exact public skill row or alias is absent, the ClawHub CLI still fetches `/api/v1/skills/<slug>` before install resolution, and production still returns 404 for the reported exact slugs.

## What This Changes

The issue asks for OpenClaw default or bundled skill slugs such as `coding-agent`, `summarize`, and `wacli` to install successfully instead of returning a ClawHub 404.

## Best Possible Solution

Define one install contract: bundled OpenClaw skills should resolve locally or give explicit guidance, while ClawHub should install published registry skills or maintainer-approved owner-backed records.

## Reproduction Assessment

Yes for the observable failure: production still returns 404 for the reported exact slugs, and current source returns 404 when no public registry skill or alias resolves. I did not run a local Convex instance with production registry data, so this is source-reproducible rather than fully current-main reproduced.

## Solution Assessment

Unclear. The narrowest maintainable solution depends on whether bundled OpenClaw skills should resolve locally, be published as owner-backed ClawHub records, or produce clearer CLI/search guidance.
