## Summary

The failure remains reproducible, but current evidence shows a cross-repository contract problem rather than a missing ClawHub lookup implementation: `coding-agent`, `summarize`, and `wacli` are bundled OpenClaw skills with no matching canonical ClawHub records, while `himayala` is misspelled and `himalaya` resolves. This issue should remain open for a maintainer decision on whether OpenClaw must stop routing bundled skills through ClawHub or ClawHub should reserve/publish canonical records.

## What This Changes

Clarify or correct installation behavior when users try to fetch bundled OpenClaw skills as ClawHub registry packages.

## Best Possible Solution

Make OpenClaw's setup and skill-install path recognize bundled skills locally, avoid a ClawHub fetch for those names, and return spelling or canonical-registry suggestions when a requested name is neither bundled nor published.

## Reproduction Assessment

Yes. Current production consistently returns the reported 404s, and current OpenClaw source confirms that the same names are bundled skill directories; `himalaya` additionally provides a positive control for the misspelling.

## Solution Assessment

No, adding generic ClawHub aliases is not yet the best demonstrated solution. The narrowest maintainable outcome is bundled-skill-aware OpenClaw installation, unless maintainers explicitly decide that official ClawHub records should become the canonical distribution source.
