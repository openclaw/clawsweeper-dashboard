## Summary

Current main still treats the reported endpoint as a published ClawHub registry lookup, and live production still returns 404 for the reported bundled-skill names. This should stay open because the unresolved work is a cross-repository product/support contract between OpenClaw bundled skills and ClawHub-published skills.

## What This Changes

The issue reports that installing bundled OpenClaw skill names through the ClawHub skill API returns 404 Skill not found.

## Best Possible Solution

Define one explicit contract: bundled skills remain locally available through OpenClaw, registry installation is for published ClawHub skills, and the CLI gives bundled-status plus spelling/search suggestions instead of a raw 404.

## Reproduction Assessment

Yes for the source and live API path: current main returns 404 when `/api/v1/skills/<slug>` or `/install` cannot resolve a public ClawHub skill row, and production still returns 404 for the reported bundled names. I did not run the OpenClaw CLI itself in this read-only ClawHub review.

## Solution Assessment

No, adding ClawHub aliases alone is not proven to be the best fix. The safer path is a maintainer-approved cross-repository contract that keeps ClawHub registry ownership clear while making OpenClaw CLI handle bundled skills and misspellings explicitly.
