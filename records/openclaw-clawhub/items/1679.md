## Summary

The issue remains necessary: production still returns 404 for three setup-facing names, while the fourth failure is the misspelling `himayala`; current ClawHub behavior is correct for absent exact slugs, but the user-facing bundled-versus-registry contract remains unresolved.

## What This Changes

Make setup-referenced skill names install successfully or clearly explain when a name is bundled, misspelled, renamed, or unavailable from ClawHub.

## Best Possible Solution

Establish a cross-project canonical-name contract: OpenClaw should recognize bundled skills without treating them as missing registry packages, and ClawHub should resolve only owner-approved exact records or explicit redirects while suggesting corrected names such as `himalaya`.

## Reproduction Assessment

Yes. Direct production requests reproduce the exact 404s for `coding-agent`, `summarize`, and `wacli`; corrected `himalaya` succeeds, confirming that one listed failure is a spelling error.

## Solution Assessment

No, adding unowned ClawHub aliases alone is not the best solution. The narrow maintainable outcome requires an explicit decision about bundled skill handling, canonical publishers, redirects, and CLI diagnostics.
