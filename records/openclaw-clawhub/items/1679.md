## Summary

Keep this issue open: production still returns `404 Skill not found` for `coding-agent`, `summarize`, and `wacli`, and current main only resolves existing published skill records or aliases. The `himayala` example is a typo because `himalaya` resolves, but that does not explain the other three failures. The remaining question is whether OpenClaw should install its baseline bundled skills locally or ClawHub should guarantee corresponding catalog entries, so no implemented-on-main close is supported.

## What This Changes

Resolve installation failures where several skills presented as OpenClaw defaults are requested from ClawHub under slugs that return 404.

## Best Possible Solution

Establish a single supported contract for baseline skills: OpenClaw should install bundled assets without requiring a registry record, or ClawHub should host stable canonical records or aliases with explicit ownership and lifecycle tests.

## Reproduction Assessment

Yes. Direct requests to the production API reproduce 404 responses for three correctly spelled reported slugs, while the corrected `himalaya` request returns 200 and isolates that example as a typo.

## Solution Assessment

Unclear. Adding isolated ClawHub records might mask the symptom, but maintainers first need to define whether baseline skills are bundled OpenClaw assets or registry-managed packages so the fix does not create a second source of truth.
