## What This Changes

The issue reports that `clawhub login --device` fails during browser approval because `cliDeviceAuth:approve` throws a Convex server error and the CLI never receives a token.

## Best Possible Solution

Make the documented device flow let a fresh GitHub-authenticated user approve a pending code and let CLI polling mint exactly one token, with real Convex runtime validation for the approve-to-token path.

## Reproduction Assessment

Yes for the reporter-provided production path, but I did not independently live-reproduce it. Source inspection confirms current main still routes browser approval through `api.cliDeviceAuth.approve` and leaves the CLI waiting on the matching token endpoint.

## Solution Assessment

Unclear because there is no proposed patch and the exact throw site needs production log correlation. The narrow maintainable path is to inspect the request IDs, fix the failing approve or fresh-user state transition, and add runtime coverage from approval through token polling.
