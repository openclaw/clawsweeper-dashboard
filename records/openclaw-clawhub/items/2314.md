## Summary

Keep open. Current main still has a plausible source path where skill publishing can abort on an embedding input-length error: publish builds a character-limited embedding payload, sends it directly to OpenAI, and converts embedding failure into a publish-blocking ConvexError.

## What This Changes

The issue reports that skill publish fails with an "8192 token" embedding error even though the submitted `SKILL.md` is small.

## Best Possible Solution

Make embedding enrichment bounded or non-blocking for publish: clamp by a conservative token budget or fall back to a safe embedding on known input-length errors while leaving upload validation, scans, and version insertion intact.

No high-confidence live reproduction yet because the report does not include the exact skill bundle or full logs. Source inspection does show a clear path where token-dense embedding input can make publish fail before the version insert completes.

Fix skill publish failures caused by embedding input exceeding the embedding model token budget. In current main, `publishVersionForUser` builds embedding text from `SKILL.md` plus selected text files, `buildEmbeddingText` caps by characters, and `generateEmbedding` throws non-retryable embedding API errors that abort publish. Implement a focused fix that either clamps embedding input using a conservative token-safe strategy before the API call or catches known input-length embedding errors and stores a safe fallback embedding so publish can complete. Add regression coverage for token-dense or over-budget embedding text and the publish/embedding error behavior. Do not change publish auth, upload limits, accepted file types, static scan/VT scheduling, ownership migration, or moderation semantics. Add a user-facing changelog/docs note only if maintainers require one for this publish-path fix.
