## Summary

Keep open under ClawHub issue policy; the report is credible because current main still calls `.unique()` on the global `by_slug` lookup in `checkSlugAvailability` before owner-scoped resolution, so duplicate or scoped same-slug rows can produce the reported Convex server error when the New Version page mounts.

## What This Changes

The issue reports that clicking a skill page’s New Version button opens an error screen from `skills:checkSlugAvailability` instead of the new-version publishing flow.

## Best Possible Solution

Make `checkSlugAvailability` owner-aware and duplicate-tolerant for the new-version preflight, with regression coverage for same-slug different-owner rows, while leaving the broader scoped-routing migration separate.

## Reproduction Assessment

Yes, source-level: create two skill rows with the same slug and open the New Version publish route for that slug; current main reaches `checkSlugAvailability`, whose global `.unique()` lookup can throw before owner resolution. The exact production slug is missing, so I did not confirm the reporter’s specific row.

## Solution Assessment

Yes, a narrow backend availability fix is the best path: avoid the throwing global `.unique()` in this preflight and select or reject by the requested owner. A broad scoped-route migration would be unnecessary for this issue.
