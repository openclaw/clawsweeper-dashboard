## Summary

Keep open: the feature is useful and now has inspectable UI screenshots, but the patch removes the current empty-page guard for filtered skill-list scans, which can make sparse category filters return an empty page with `hasMore: true` and repeatedly trigger the browse auto-loader.

## What This Changes

The branch adds inferred skill categories, category-filtered related skills on skill detail pages, category browse filtering, tests, styles, and a PR-template Behavioural Proof section.

## Best Possible Solution

Keep the category-aware related-skills direction, but preserve the no-empty-auto-page invariant by changing the server or client so filtered scans do not return an empty page that can auto-load again.

## Reproduction Assessment

Yes, from source inspection: the PR can return `page: []` with `hasMore: true` from `listPublicPageV4`, and the browse results component still mounts the load-more observer when the rendered list is empty. I did not run a browser reproduction because the checkout is read-only for this review.

## Solution Assessment

No: the feature should not remove the current empty-response guard unless the client/server gains another way to avoid repeated empty auto-loads. A safer fix is to keep scanning within a bounded server loop until a non-empty page or terminal end, or suppress auto-load on empty filtered pages.
