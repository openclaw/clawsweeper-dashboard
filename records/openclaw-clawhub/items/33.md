## Summary

Keep open under the ClawHub issue policy. Current source and the maintained `clawhub` npm package include `undici`, but the reported legacy `clawdhub@latest` path still resolves to `0.3.0` without that dependency, so the user-visible failure remains a registry/release decision rather than a source-only cleanup.

## What This Changes

The issue reports that running the legacy `clawdhub@latest` CLI fails because its published package metadata omits the runtime `undici` dependency.

## Best Possible Solution

A maintainer should resolve the legacy npm package state by publishing or retagging a stable patched `clawdhub` release, or by deprecating `clawdhub@0.3.0` with clear migration guidance to the current `clawhub` package, then close duplicate reports against the chosen canonical item.

## Reproduction Assessment

Yes. Live npm metadata shows `clawdhub@latest` resolves to `0.3.0` without `undici`, and the published tarball imports `undici`; reporter logs cover `npx`, `pnpm dlx`, and global install paths.

## Solution Assessment

No source-only fix is the best remaining solution. Current main and the maintained `clawhub` package already declare `undici`; maintainers need to handle the legacy package's registry state, dist-tag, release, or deprecation path.
