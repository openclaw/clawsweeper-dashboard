## Summary

Current main has a broad acceptable-usage policy, but this RFC still contains active unresolved questions about categorical crypto exclusions, spam versus harmless testing, scanner context, positive trust signals, and how categories affect installability and account enforcement. The issue is collaborator-authored, security-sensitive, and explicitly open for review, so it requires maintainer judgment rather than cleanup.

## What This Changes

Define content-policy categories and map them to review, discovery, registry inclusion, installability, and account-enforcement outcomes.

## Best Possible Solution

Keep this RFC as the canonical decision thread, incorporate the accepted SkillSpector boundary, separate enforcement categories from positive trust signals, and then record reviewer-only invariants in `specs/security-moderation.md` and publish finalized acceptable-use rules in `docs/acceptable-usage.md`.

Yes, keeping a canonical RFC open is the narrowest maintainable approach while category exceptions and enforcement mappings remain undecided; finalized decisions should then be split between internal specs and public policy docs.
