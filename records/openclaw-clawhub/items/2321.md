## Summary

Keep open: current main implements substantial moderation and scanner behavior, but it does not settle this RFC’s broader policy choices around duplicate or low-quality content, positive trust signals, and legitimate crypto or finance exceptions. The issue is collaborator-authored, actively discussed, and protected for maintainer, product, and security review, so explicit trust-and-safety judgment is required.

## What This Changes

Define content-policy categories and map each category to discovery, installability, human review, reporting, and account-enforcement outcomes.

## Best Possible Solution

Trust-and-safety owners should approve a reviewer-facing category-and-consequence matrix with explicit examples and exceptions, persist it in `specs/security-moderation.md`, and publish only stable user-facing rules in the acceptable-use and moderation docs.

## Reproduction Assessment

Not applicable; this is a policy and governance RFC rather than a report of broken runtime behavior. Current main can demonstrate existing moderation mechanics, but the remaining questions require normative decisions.

## Solution Assessment

Yes, keeping a canonical RFC open and then separating durable reviewer guidance in `specs/` from stable public rules in `docs/` is the narrowest maintainable approach. Automation should not choose the disputed categories or enforcement severity.
