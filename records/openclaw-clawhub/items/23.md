## Summary

Keep open: current `main` and the modern `clawhub` npm package declare `undici`, but the live legacy `clawdhub@latest` path still resolves to `0.3.0` without that dependency, so the reported user-visible failure remains until a maintainer handles npm release, dist-tag, deprecation, or migration state.

## What This Changes

The issue reports that the published legacy `clawdhub` CLI imports `undici` but does not declare it as a runtime dependency, causing clean Node.js installs to fail with `ERR_MODULE_NOT_FOUND`.

## Best Possible Solution

A maintainer should make the legacy package path unambiguous by publishing or promoting a fixed stable `clawdhub`, deprecating or migrating the broken legacy package toward `clawhub`, or otherwise ensuring legacy install commands no longer resolve to a package missing `undici`.

## Reproduction Assessment

Yes for the live distribution path: registry metadata and the `clawdhub@0.3.0` tarball show a package that imports `undici` without declaring it. Current main itself does not reproduce the source dependency omission because `packages/clawhub/package.json` already includes `undici`.

## Solution Assessment

No source-only dependency PR is the best remaining fix. The source manifest is already fixed, so the narrow maintainable solution is npm release, dist-tag, deprecation, or migration handling for the legacy package.
