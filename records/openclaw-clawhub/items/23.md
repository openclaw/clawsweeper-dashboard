## Summary

The source-side dependency fix is present, but the reported user-visible install path remains broken because `clawdhub@latest` still resolves to `0.3.0` without `undici`; ClawHub policy also keeps issues open for maintainer follow-up rather than auto-closing them as implemented.

## What This Changes

The issue reports that the legacy `clawdhub` npm package imports `undici` at runtime but published installs do not include it as a dependency.

## Best Possible Solution

A maintainer should make the legacy package path unambiguous by publishing or promoting a fixed stable `clawdhub`, deprecating or migrating users to `clawhub`, or otherwise ensuring legacy install commands no longer resolve to a package missing `undici`.

## Reproduction Assessment

Yes. The issue gives concrete Node/npm reproduction paths, and the live npm registry still shows `clawdhub@latest` resolving to `0.3.0` without `undici`, which matches the reported module-resolution failure.

## Solution Assessment

No source-only change is sufficient now. Current source and the modern `clawhub` package already include `undici`, so the narrow maintainable solution is maintainer-controlled npm distribution, dist-tag, deprecation, or migration handling for legacy `clawdhub`.
