## Summary

Keep open. Current `main` and the modern `clawhub` package declare `undici`, but the reported legacy `clawdhub@latest` path still resolves to `0.3.0` without that dependency while its tarball imports `undici`; ClawHub issue policy also keeps issues open for maintainer handling.

## What This Changes

Report that fresh Node/npm installs of legacy `clawdhub@0.3.0` fail because `dist/http.js` imports `undici` without declaring it as a dependency.

## Best Possible Solution

A maintainer should stop legacy install commands from resolving to the broken package by publishing or promoting a fixed `clawdhub`, deprecating/migrating it to `clawhub`, or updating npm dist-tags, then verify a clean Node smoke test.

## Reproduction Assessment

Yes for the live distribution path: registry metadata shows `clawdhub@latest` is `0.3.0` with no `undici`, and its tarball imports from `undici`. Current `main` itself does not reproduce the missing dependency because the modern manifest already declares it.

## Solution Assessment

No source-only PR is the best remaining fix. The dependency is already in current source and the modern package, so the narrow maintainable solution is maintainer-controlled npm release, dist-tag, deprecation, or migration handling for the legacy package.
