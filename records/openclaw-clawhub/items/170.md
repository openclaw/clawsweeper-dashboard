## Summary

Keep open. Current main has the right code-level foundation for renamed GitHub accounts, but this is an open ClawHub issue, related production OAuth/account-state reports remain open, and a live sign-in endpoint check currently fails before the username-change path can be isolated.

## What This Changes

GitHub OAuth sign-in should continue to recognize the same account after the user changes their GitHub username.

## Best Possible Solution

Keep identity anchored to numeric providerAccountId, restore production OAuth/session creation for active accounts, and add an audited maintainer recovery path only if production triage proves stale, duplicate, deleted, or verifier state still blocks renamed users.

## Reproduction Assessment

No. The report has plausible steps and current production auth is failing, but a high-confidence reproduction of the username-change-specific bug needs live GitHub OAuth state, browser storage/cookies, Convex Auth logs, and the affected production user/authAccounts rows.

## Solution Assessment

Unclear. Numeric GitHub provider ID lookup plus post-login profile sync is the right code foundation, but any remaining relink, verifier, duplicate-account, or production-config fix should be chosen only after maintainer/operator triage.
