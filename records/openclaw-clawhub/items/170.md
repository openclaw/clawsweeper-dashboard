## Summary

Keep open: current main has the stable GitHub numeric-ID and profile-sync implementation, and the unmerged #179 proposal was superseded by merged work in #293/#312, but ClawHub policy keeps issues open and related open report #1109 leaves production OAuth/account-state verification unresolved.

## What This Changes

The issue requests that GitHub sign-in keep recognizing the same account after a GitHub username change by using the immutable GitHub user ID instead of the mutable login.

## Best Possible Solution

Keep the numeric providerAccountId and profile-sync implementation, then have maintainers verify production auth state and add an audited recovery path only if triage shows a repeatable verifier, duplicate-account, or relink failure.

## Reproduction Assessment

No. The report gives plausible OAuth rename steps and #1109 adds a concrete convexAuthOAuthVerifier symptom, but a high-confidence current-main reproduction would require live GitHub OAuth callback state and production auth records.

## Solution Assessment

Yes for the code-level direction, but incomplete for the live account outcome. Numeric providerAccountId lookup plus post-login profile sync is the narrow maintainable source fix; any remaining relink or consolidation needs maintainer-owned identity verification.
