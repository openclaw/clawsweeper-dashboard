## Summary

Keep open. Current main has the correct numeric GitHub ID and post-login profile-sync foundation, but this is a ClawHub issue and the cleanup policy for this review only permits cleanup closure for PRs; the live production GitHub sign-in route also currently returns HTTP 500 and the related open report at https://github.com/openclaw/clawhub/issues/1109 keeps the account-state/verifier side unresolved.

## What This Changes

A user cannot sign in to ClawHub after changing their GitHub username and expects the account to remain linked by immutable GitHub user ID.

## Best Possible Solution

Restore the production GitHub OAuth redirect/session path, keep identity anchored to immutable providerAccountId, inspect affected production auth rows, and add a maintainer-owned recovery or consolidation path only if production triage proves one is needed.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the observable current-production GitHub sign-in failure: current HEAD is tagged prod/v2026.05.10.6 and both the canonical and direct Convex GitHub sign-in start endpoints returned HTTP 500; no for isolating the username-change-specific verifier/account-state failure without an affected renamed account and production auth rows.

## Solution Assessment

Is this the best way to solve the issue? Not applicable for a patch because this is an issue, not a proposed code change. The current numeric-ID mapping and profile sync are the right foundation, but any remaining fix should be scoped after operator triage of live OAuth and account state.
