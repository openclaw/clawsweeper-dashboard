## Summary

Keep open. Current main has the right stable GitHub ID and profile-sync foundation, but ClawHub policy does not auto-close issues, and live GitHub sign-in still returns HTTP 500 before the username-change path can be isolated.

## What This Changes

The issue reports that GitHub sign-in breaks after a username change and asks ClawHub to key account lookup on immutable GitHub user IDs while refreshing mutable username fields.

## Best Possible Solution

Keep identity anchored to numeric providerAccountId, restore the production OAuth redirect/session path, inspect the affected account rows, and add a maintainer-owned recovery path only if production triage proves one is needed.

## Reproduction Assessment

No. The issue has plausible steps and the live sign-in endpoint currently returns HTTP 500, but a high-confidence username-change-specific reproduction needs a renamed GitHub account, browser verifier/cookies, Convex logs, and affected users/authAccounts rows.

## Solution Assessment

Unclear. Numeric provider IDs plus post-login profile sync are the right code foundation, but the next fix should be scoped only after operator triage determines whether production config, duplicate rows, blocked account state, verifier state, or a recovery path is responsible.
