## Summary

Keep open. Current main has the numeric GitHub ID and profile-sync foundation for renamed accounts, but ClawHub issue cleanup does not auto-close issues, a related username-change report remains open, and live sign-in endpoints currently fail before the username-change path can be isolated.

## What This Changes

GitHub sign-in should continue to recognize the same account after the user changes their GitHub username.

## Best Possible Solution

Keep identity anchored to numeric providerAccountId, restore production OAuth/session creation for active accounts, and add an audited recovery path only if production triage proves stale, duplicate, deleted, deactivated, or verifier state still blocks renamed users.

## Reproduction Assessment

No. The reported username-change path needs live GitHub OAuth state plus affected production authAccounts/users rows; a current smoke check does show the GitHub sign-in endpoints returning HTTP 500 while the site root returns HTTP 200, but that does not isolate the username-change-specific bug.

## Solution Assessment

Unclear. Numeric GitHub provider ID lookup plus post-login profile sync is the right code foundation, but any remaining relink, verifier, duplicate-account, or production-config fix should be chosen only after maintainer/operator triage.
