## Summary

Keep open. Current main has the right numeric GitHub-ID and post-login profile-sync foundation, but this is a ClawHub issue under a policy that only allows cleanup closure for PRs, and the production GitHub sign-in start route still returns HTTP 500.

## What This Changes

The report asks ClawHub to preserve GitHub sign-in after a GitHub username change by keying account lookup on the stable numeric GitHub user ID rather than the mutable username.

## Best Possible Solution

Restore the production GitHub OAuth redirect/session path, keep identity anchored to numeric `providerAccountId`, and let maintainers repair or add a narrowly audited recovery path only after inspecting production auth records.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the current production GitHub sign-in failure: current main is production-tagged and both sign-in start endpoints return HTTP 500. No for isolating the username-change-specific verifier/account-state failure without an affected renamed account and production auth rows.

## Solution Assessment

Is this the best way to solve the issue? Not applicable for a patch because this is an issue. The numeric-ID mapping and post-login profile sync are the right foundation, but the remaining work needs maintainer/operator OAuth and account-state triage before any relink or source repair is scoped.
