## Summary

Keep open. Current main covers the core code-level GitHub rename path with numeric GitHub provider IDs and mutable profile sync, but ClawHub cleanup policy keeps issues open and the related #1109 login report still needs production/account-state triage.

## What This Changes

The issue reports that GitHub OAuth sign-in fails after a user changes their GitHub username and asks ClawHub to use the immutable GitHub user ID for account lookup while treating the username as mutable profile data.

## Best Possible Solution

Production should authenticate GitHub users by numeric `providerAccountId`, refresh mutable GitHub profile fields for active accounts, and provide a maintainer-owned recovery or consolidation path for stale, deleted/deactivated, duplicate, or verifier/session state that still blocks reauth.

## Reproduction Assessment

No: the current source path for a username rename is clear, but reproducing the reported live sign-in failure requires an affected production account and OAuth verifier/session state. The best available reproduction evidence is the issue report plus #1109's concrete `convexAuthOAuthVerifier token mismatch` symptom.

## Solution Assessment

Unclear as a code change: current main is the right source-level direction because it uses numeric provider IDs and syncs mutable profile data. The remaining path should be maintainer production/account-state triage before another automated code PR.
