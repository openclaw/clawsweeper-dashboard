## Summary

Keep open. Current main has the stable GitHub numeric-ID and profile-sync foundation, and related handle-sync work has shipped, but ClawHub cleanup policy keeps issues open and the remaining username-change login symptom is still represented by open production OAuth/verifier reports that need operator auth-state triage.

## What This Changes

The issue asks ClawHub sign-in to keep working after a GitHub account is renamed by using GitHub's immutable numeric account ID instead of the mutable login.

## Best Possible Solution

Keep identity anchored to numeric providerAccountId, verify the affected production auth state, and add a focused maintainer-owned recovery path only if production triage proves stale, duplicate, deleted/deactivated, or verifier state still blocks renamed users.

## Reproduction Assessment

No. The report has plausible steps and a related open verifier-mismatch report, but a high-confidence current-main reproduction needs live GitHub OAuth state, browser storage/cookies, production Convex logs, and the affected users/authAccounts rows.

## Solution Assessment

Partly. The suggested immutable GitHub ID direction is already implemented in source, but any remaining fix should be scoped after maintainer/operator verification of production OAuth/session and account-linkage state.
