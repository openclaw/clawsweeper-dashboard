## Summary

Keep open. Current main has the numeric GitHub ID and post-login profile-sync foundation, but the ClawHub cleanup policy keeps issues open, the live GitHub sign-in start route still returns HTTP 500, and the related open report at https://github.com/openclaw/clawhub/issues/1109 keeps the verifier/account-state side unresolved.

## What This Changes

The issue asks ClawHub sign-in to keep working after a GitHub username change by anchoring account lookup to the immutable GitHub user ID and refreshing mutable username/profile fields.

## Best Possible Solution

Restore the production GitHub OAuth redirect/session path, keep identity anchored to numeric providerAccountId, inspect affected production auth rows, and add a maintainer-owned recovery or consolidation path only if production triage proves one is needed.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue?
