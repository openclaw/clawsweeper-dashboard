## Summary

Keep open: current main has guarded admin repair primitives, but the live Apify package still has the reported `runtimeId: apify` while the upstream plugin manifest declares `apify-openclaw-plugin`, so the requested production correction is not done.

## What This Changes

The issue requests correcting or hard-deleting the ClawHub package row for `@apify/apify-openclaw-plugin` so future publishes can use runtimeId `apify-openclaw-plugin`.

## Best Possible Solution

Verify Apify publisher authority, confirm `apify-openclaw-plugin` has no active runtimeId collision, then perform an audited admin runtimeId repair from `apify` to `apify-openclaw-plugin`; use hard-delete only if repair cannot preserve the package safely.

## Reproduction Assessment

Yes. Live ClawHub still reports `runtimeId: apify` for `@apify/apify-openclaw-plugin`, the upstream manifest declares `apify-openclaw-plugin`, and current main rejects normal runtimeId changes on existing code-plugin packages.

## Solution Assessment

Yes for the repair shape: an audited admin runtimeId repair is narrower and safer than hard-delete because it can preserve package history while enforcing authority, collision checks, and audit logging. The remaining blocker is operational authorization and the exact admin execution path.
