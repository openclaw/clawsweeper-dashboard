## What This Changes

Requests an audited production repair or hard-delete for `@apify/apify-openclaw-plugin` because its ClawHub package runtimeId is `apify` while the upstream plugin id is `apify-openclaw-plugin`.

## Best Possible Solution

Verify Apify authority, confirm no active `apify-openclaw-plugin` runtimeId collision, then perform an audited admin runtimeId repair from `apify` to `apify-openclaw-plugin`; use hard-delete only if preserving the package row is not viable.

## Reproduction Assessment

Yes. The live package API still reports `runtimeId: apify`, the upstream manifest declares `apify-openclaw-plugin`, and current main's publish guard rejects normal runtimeId changes for an existing code-plugin package.

## Solution Assessment

Yes for the likely repair shape. An audited admin runtimeId repair is narrower and safer than hard-delete because it can preserve package history while enforcing admin authority, collision checks, and audit logging.
