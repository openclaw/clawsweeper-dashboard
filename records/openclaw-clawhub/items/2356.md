## Summary

Keep open: current main has guarded admin repair primitives, but the live Apify package row still reports `runtimeId: apify` while the upstream plugin manifest declares `apify-openclaw-plugin`, so the requested production identity repair has not happened yet. The related OpenViking issue is a useful precedent, but it does not resolve this package-specific request.

## What This Changes

The issue asks ClawHub to hard-delete or correct the `runtimeId` for `@apify/apify-openclaw-plugin` so publishes using the package's current plugin id are no longer rejected.

## Best Possible Solution

Verify Apify authority, confirm no active `apify-openclaw-plugin` runtimeId collision, then perform an audited admin runtimeId repair from `apify` to `apify-openclaw-plugin`; use hard-delete only if preserving the package row is not viable.

## Reproduction Assessment

Yes. The public package API still reports `runtimeId: apify` for `@apify/apify-openclaw-plugin`, the upstream manifest declares `apify-openclaw-plugin`, and current main rejects normal publishes that attempt runtimeId changes.

## Solution Assessment

Yes for the likely repair shape. An audited admin runtimeId repair is narrower and safer than hard-delete because it can preserve package history while enforcing admin authority, collision checks, and audit logging.
