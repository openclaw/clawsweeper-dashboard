## What This Changes

The issue asks ClawHub maintainers to correct or hard-delete the `@apify/apify-openclaw-plugin` package because its stored runtimeId remains `apify` while the plugin now publishes as `apify-openclaw-plugin`.

## Best Possible Solution

Verify Apify publisher authority, then perform an audited admin runtimeId repair from `apify` to `apify-openclaw-plugin` if collision checks pass; avoid hard-delete unless the repair path cannot preserve package history.

## Reproduction Assessment

Yes. Live ClawHub still reports `runtimeId: apify` for the package while the upstream manifest declares `apify-openclaw-plugin`, and current main rejects code-plugin runtimeId changes during normal publish.

## Solution Assessment

Unclear until maintainer review. The narrowest safe path appears to be an audited admin runtimeId repair rather than hard-delete, but a maintainer needs to verify authority and decide whether to use the internal mutation directly or expose a runtime-only operator route.
