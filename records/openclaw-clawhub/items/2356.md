## Summary

Keep open: the Apify production package still exposes the reported `runtimeId: apify` while the upstream plugin manifest declares `apify-openclaw-plugin`. Current main has guarded admin repair primitives, but the specific package row has not been corrected or deleted, so this still needs maintainer/admin handling.

## What This Changes

The issue asks ClawHub maintainers to update or hard-delete the `@apify/apify-openclaw-plugin` package entry so its runtimeId can change from `apify` to `apify-openclaw-plugin`.

## Best Possible Solution

Verify Apify publisher authority, confirm no active `apify-openclaw-plugin` runtimeId collision, then perform an audited admin runtimeId repair from `apify` to `apify-openclaw-plugin`; use hard-delete only if the safer repair path cannot preserve the package correctly.

## Reproduction Assessment

Yes. Live ClawHub still reports `runtimeId: apify` for `@apify/apify-openclaw-plugin`, while the upstream manifest declares `apify-openclaw-plugin` and current main rejects normal runtimeId changes on existing code-plugin packages.

## Solution Assessment

Yes for the likely repair shape: an audited admin runtimeId repair is narrower and safer than hard-delete because it can preserve package history while enforcing admin authority, collision checks, and audit logging. The remaining question is operational authorization, not a source-level fix.
