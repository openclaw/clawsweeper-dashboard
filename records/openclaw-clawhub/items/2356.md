## Summary

Keep open: current main has an audited internal admin primitive that can repair a package runtimeId, but live ClawHub still shows the Apify package stored as `runtimeId: apify` while the upstream manifest declares `apify-openclaw-plugin`, so the production publish blocker remains unresolved.

## What This Changes

The issue asks ClawHub maintainers to correct or remove the Apify package row because the stored runtimeId blocks publishing newer manifests that use `apify-openclaw-plugin`.

## Best Possible Solution

Verify Apify publisher authority, then perform an audited admin runtimeId repair from `apify` to `apify-openclaw-plugin` if collision checks pass; avoid hard-delete unless the repair path cannot preserve package history.

## Reproduction Assessment

Yes. Live ClawHub still reports `runtimeId: apify` for the package while the upstream manifest declares `apify-openclaw-plugin`, and current main rejects code-plugin runtimeId changes during normal publish.

## Solution Assessment

Unclear until maintainer review. The narrowest safe path appears to be the existing audited admin runtimeId repair rather than hard-delete or an implicit publish-time exception.
