## Summary

Keep open: current main has an audited internal repair primitive, but live ClawHub still shows the Apify package with `runtimeId: apify` while the upstream plugin declares `apify-openclaw-plugin`, so the specific production blocker remains unresolved.

## What This Changes

The issue asks for hard deletion or runtimeId migration of `@apify/apify-openclaw-plugin` because ClawHub stores the package runtimeId as `apify` while the plugin now publishes as `apify-openclaw-plugin`.

## Best Possible Solution

Verify Apify ownership, then use the audited admin runtimeId repair path to change `apify` to `apify-openclaw-plugin`; only add a runtimeId repair route/CLI if maintainers want this to become a repeatable operator workflow.

## Reproduction Assessment

Yes. Live ClawHub still returns `runtimeId: apify` for the package while the upstream manifest declares `apify-openclaw-plugin`, and current main still rejects normal publish-time runtimeId changes.

## Solution Assessment

Unclear until maintainer security review. The narrowest likely solution is verified, audited runtimeId repair rather than hard delete; a new route/CLI is only needed if maintainers want a repeatable operator surface.
