## Summary

Keep open: current main has an admin-only internal runtimeId repair primitive, but live ClawHub still shows `@apify/apify-openclaw-plugin` locked to `runtimeId: apify` while the upstream manifest now declares `apify-openclaw-plugin`, so the production publish blocker still needs maintainer/admin handling.

## What This Changes

The issue asks ClawHub maintainers to either update the Apify package runtimeId from `apify` to `apify-openclaw-plugin` or hard-delete the package row so it can be republished.

## Best Possible Solution

Verify Apify publisher authority, check runtimeId collisions, then use an audited admin identity repair to update the package runtimeId to `apify-openclaw-plugin`; avoid hard-delete unless repair cannot preserve history.

## Reproduction Assessment

Yes. Live ClawHub still reports `runtimeId: apify` for the package while the upstream manifest declares `apify-openclaw-plugin`, and current main rejects code-plugin runtimeId changes during normal publish.

## Solution Assessment

Unclear until maintainer review. The narrowest safe path appears to be an audited admin runtimeId repair rather than hard-delete, but a maintainer needs to verify authority and decide whether to use the internal mutation directly or expose a runtime-only operator route.
