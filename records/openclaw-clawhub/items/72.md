## Summary

Keep open. Current evidence does not prove that fresh Settings-created tokens now succeed with the default CLI registry; the old redirect-only workaround is stale, and this auth-sensitive issue needs maintainer production verification rather than cleanup closure.

## What This Changes

The issue reports that API tokens created in Settings return Unauthorized when used with `clawhub login --token --no-browser`.

## Best Possible Solution

Verify production with a fresh Settings-created token on the current published CLI, then fix the specific failing layer with focused CLI/backend tests while preserving bearer-token validation.

## Reproduction Assessment

No. Source inspection traces `login --token` to `/api/v1/whoami`, but a high-confidence failure reproduction requires a real fresh production token and account state that this read-only review did not have.

## Solution Assessment

Unclear. Changing defaults to `www.clawhub.ai` matched the original workaround, but current production redirects `www` back to the apex host, so maintainer production verification should choose the specific source, hosting, or account-state fix.
