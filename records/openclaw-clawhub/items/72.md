## Summary

Keep open. ClawHub cleanup policy keeps issues open unless they are PRs already implemented on main, and the current source plus live host checks do not prove the valid-token Unauthorized path is fixed. The earlier `www` workaround is useful context, but current production redirects `www.clawhub.ai` back to `clawhub.ai`, while token login still verifies through registry discovery and `/api/v1/whoami`, so this needs maintainer/operator verification with a fresh Settings-created token.

## What This Changes

Settings-created API tokens fail with Unauthorized when used with `clawhub login --token --no-browser`, and the requested behavior is for token login to succeed with the default registry.

## Best Possible Solution

The desired end state is a verified production auth path where a fresh Settings-created `clh_` token works with the default CLI registry for `login --token`, `whoami`, and a minimal publish or sync smoke path. If that still fails, the fix should target the confirmed layer with focused coverage around registry discovery/defaults, redirect/auth behavior, `/api/v1/whoami` token lookup, browser callback registry propagation, and user/publisher state.
