Keep open. This is an open ClawHub issue, and the repository cleanup policy only permits auto-closing PRs that current main certainly implements; the current code still routes token login through the default registry discovery path and `/api/v1/whoami`, while the thread and related reports show this auth-provider failure still needs a production token repro rather than cleanup closure.

## What This Changes

The issue reports that `clawhub login --token <clh_...> --no-browser` returns `Unauthorized` for fresh API tokens created in the ClawHub web UI.

## Best Possible Solution

End state should be a verified production auth path where a fresh Settings-created `clh_` token works with the default CLI registry for token login, `whoami`, and a minimal publish or sync smoke path, with the eventual code change targeted to the specific failing layer.

## Reproduction Assessment

No high-confidence live reproduction was established because this requires a real Settings-created production token. Source inspection confirms the reported path is clear: `login --token` resolves the default registry, calls `/api/v1/whoami`, and surfaces `Unauthorized` when token auth fails.

## Solution Assessment

Unclear. The old `www` default workaround was plausible in January, but current live redirects point the other direction, so the safer path is maintainer/operator production verification first and then a narrow fix at the proven failing layer.
