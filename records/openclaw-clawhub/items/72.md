## Summary

Keep open. This is an open ClawHub issue, and the current evidence does not prove the token-login path is fixed: source still verifies fresh tokens through the default registry and `/api/v1/whoami`, while live host probes no longer match the original `www` workaround enough to choose a safe automated fix.

## What This Changes

Fresh UI-created `clh_` tokens should let `clawhub login --token ... --no-browser` verify and store credentials with the default registry.

## Best Possible Solution

Maintainers should verify a fresh production token against the current published CLI; if it still fails, make one narrow fix at the proven failing layer across hosting, discovery, CLI auth, backend token lookup, and focused tests.

## Reproduction Assessment

No. The source path and historical user reports are strong, but this review did not have a real production token and the live dummy-token host probe no longer reproduces the originally reported redirect direction.

## Solution Assessment

Unclear. Switching defaults to `https://www.clawhub.ai` is not currently proven correct because live production now redirects `www` back to non-www; maintainers should verify with real tokens before choosing a source or hosting fix.
