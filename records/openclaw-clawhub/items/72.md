## Summary

Keep open. This is an open ClawHub issue, and current main does not prove the valid Settings-token failure is fixed; the old `www.clawhub.ai` workaround is now contradicted by source, live routing, and closed host-switch PR context.

## What This Changes

The issue reports that Settings-created `clh_` API tokens fail `clawhub login --token --no-browser` with `Unauthorized` despite being valid.

## Best Possible Solution

Verify fresh production Settings-created and browser-created tokens against the current published CLI and canonical host, then either consolidate the Unauthorized reports or land one focused fix at the proven failing layer with regression coverage.

## Reproduction Assessment

No. Historical reports and source inspection trace the path through `cmdLogin` and `/api/v1/whoami`, but high-confidence reproduction requires a real production Settings-created token that this read-only review did not have.

## Solution Assessment

Unclear. Switching defaults to `https://www.clawhub.ai` is not currently the best proven fix because current source, docs, live discovery, and observed production routing are canonical non-www.
