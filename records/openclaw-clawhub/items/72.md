## Summary

Keep open. This is a still-open ClawHub issue, and current main plus live host behavior do not prove fresh Settings-created tokens now work with the default CLI; the old `www` workaround is useful history but conflicts with today’s apex-host routing.

The issue reports that Settings-created `clh_` API tokens fail `clawhub login --token` with `Unauthorized` instead of being verified and stored.

## Best Possible Solution

No. The CLI and backend paths are clear, and dummy-token probes show current host behavior, but high-confidence reproduction requires a fresh production Settings-created token and real `login --token` plus `whoami` verification.

## Solution Assessment

Unclear. The historical `www` switch matched the original workaround, but current source and live redirects make the apex host canonical, so the fix should wait for real-token production verification rather than blindly changing defaults.
