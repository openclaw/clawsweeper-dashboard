Keep open. Current main still sends token login through registry discovery and `/api/v1/whoami`, while the original `www` workaround no longer matches live routing; a fresh production Settings token is needed before choosing a source or hosting fix.

The issue reports that web-created `clh_` API tokens are rejected by `clawhub login --token` with `Unauthorized`.

## Best Possible Solution

Maintainers should verify the current published CLI with a fresh production Settings token; if it still fails, fix the observed layer across registry discovery, redirect behavior, API-token lookup, or account state with focused tests, and otherwise consolidate this with the broader open auth reports.

No. The thread has credible January reproductions, but this review did not have a real production token; dummy live probes now show apex `/api/v1/whoami` responds directly and `www` redirects back to apex.

## Solution Assessment

Unclear. The original suggested `www` default is no longer the safest fix because current production routing points `www` back to the apex host; production-token verification should identify the failing layer first.
