Keep open. This is a credible CLI/API-token auth bug and ClawHub cleanup policy does not allow auto-closing issues; the specific old redirect theory appears fixed in production, but the valid-token failure still needs verification with a fresh Settings-created token.

## What This Changes

The issue asks for `clawhub login --token` to accept valid `clh_` tokens created in the ClawHub web UI instead of returning `Unauthorized`.

## Best Possible Solution

Verify the current production token-login path with a fresh Settings token, then fix the proven layer so default-registry token login, `whoami`, and a minimal publish or sync smoke path all work without a registry override.

## Reproduction Assessment

No high-confidence current-main reproduction is available because proving the reported valid-token failure requires a real fresh production Settings-created token. Source inspection confirms the relevant path: token login verifies through the default registry and `/api/v1/whoami` before storing the token.

## Solution Assessment

Unclear. Switching defaults to `https://www.clawhub.ai` matched early workaround reports, but the related redirect-specific report is closed as fixed and live probes now show the canonical apex API responding directly, so the next fix should follow fresh production-token evidence.
