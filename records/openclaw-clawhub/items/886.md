## Summary

Keep open: current main still blocks deleted-account GitHub sign-ins by design, and changing that behavior needs maintainer product/security direction rather than cleanup auto-close.

## What This Changes

The issue asks ClawHub to let a user re-register or be purged after deleting their account and then getting stuck signed out during GitHub OAuth.

## Best Possible Solution

Define a canonical deleted-account lifecycle and recovery policy, then implement the approved path in the auth callback, admin/support tooling, UI messaging, specs/docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: deletion sets deactivatedAt/purgedAt, GitHub OAuth reauth rejects that user, and active auth helpers return no active session; I did not run a live production deletion test.

## Solution Assessment

No. A one-off purge is not the best repo-level solution because deleted-account recovery needs an approved lifecycle policy that preserves OAuth identity binding, deletion guarantees, and ban boundaries.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it touches OAuth identity binding, irreversible deletion semantics, and moderation boundaries.

Concerns:

- **[medium] Preserve OAuth identity binding:** `convex/auth.ts:40`
  - body: handleDeletedUserSignIn checks the auth-linked existingUserId before handling deleted-account state; any recovery or purge tooling needs equivalent checks to avoid restoring or recreating the wrong GitHub-linked account.
  - confidence: 0.9

  - body: The admin unban path refuses permanently deleted account restoration, so any login recovery path must preserve ban and autoban boundaries instead of silently restoring blocked identities.
  - confidence: 0.86

  - body: The security/moderation spec says user-initiated deletion is irreversible and cannot be restored by logging in again, so any recovery behavior needs an explicit policy update.
  - confidence: 0.86
