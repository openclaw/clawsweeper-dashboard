## Summary

Keep open: current main intentionally blocks self-deleted GitHub-linked accounts during OAuth, and the remaining recovery or purge request needs maintainer product/security handling rather than cleanup closure.

## What This Changes

The issue asks for recovery, purge, or re-registration after a deleted ClawHub account gets stuck returning to sign-in after GitHub OAuth.

## Best Possible Solution

Decide and document the canonical deleted-account lifecycle, then ship the matching OAuth, admin/support, user-message, spec, and regression-test path while preserving GitHub identity and ban boundaries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: deleteAccount sets deactivatedAt/purgedAt, the OAuth callback rejects that linked user, and active auth helpers return no active session; I did not run a live production deletion test.

## Solution Assessment

No. A one-off purge is not the best repo-level solution because it can conflict with irreversible deletion policy, OAuth identity binding, and ban boundaries; maintainers need to choose the lifecycle and support policy first.
