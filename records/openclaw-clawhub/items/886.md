## Summary

Keep this issue open. Current main still blocks deleted/deactivated GitHub-linked accounts from signing in, and resolving the requested purge or re-registration path needs maintainer product, security, and production-support handling rather than conservative cleanup.

## What This Changes

A user asks for a purge or re-registration path after deleting a GitHub-linked ClawHub account and then being returned to sign-in after OAuth.

## Best Possible Solution

Define one account lifecycle policy for deleted GitHub-linked accounts, then implement the approved recovery or support path across Convex auth, deletion/admin tooling, UI messaging, docs/specs, and regression tests.

## Reproduction Assessment

Yes, from source inspection. deleteAccount sets deactivatedAt/purgedAt, the OAuth callback rejects deactivated users, and active auth helpers then expose no active user, matching the reported signed-out result after authorization.

## Solution Assessment

No, a one-off purge is not the best source-level solution. The maintainable path needs an explicit recovery/support policy that preserves OAuth identity binding, irreversible deletion semantics, and ban protections.
