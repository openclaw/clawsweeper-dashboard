## Summary

Keep this issue open. Current main intentionally rejects self-deleted GitHub-linked accounts during OAuth and now has clearer blocked-account messaging, but there is still no approved purge, restore, or re-registration path for affected users.

## What This Changes

The issue requests account recovery after deleting a ClawHub account and then getting stuck back at sign-in after GitHub authorization.

## Best Possible Solution

Define one canonical lifecycle policy for deleted GitHub-linked accounts, then implement the approved recovery or support path across Convex auth, admin tooling, UI messaging, specs/docs, and regression tests.

## Reproduction Assessment

Yes, from source inspection. Deleting an account sets deactivatedAt/purgedAt, the GitHub OAuth callback rejects that user on later sign-in, and active auth helpers then return no active user; I did not run a live OAuth flow.

## Solution Assessment

No, not as a one-off purge or silent recreate. The maintainable solution needs an approved account lifecycle and security policy before changing OAuth bindings, deletion state, recovery tooling, or support operations.
