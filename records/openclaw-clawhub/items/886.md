## Summary

Keep this issue open. Current main still treats self-deleted GitHub-linked accounts as terminal during OAuth, while the requested purge/re-registration path conflicts with the documented irreversible-deletion policy and needs maintainer support/security direction rather than conservative cleanup close.

## What This Changes

The issue asks for a deleted ClawHub account record to be purged or recovered so the same GitHub account can sign in or re-register instead of returning to the sign-in page.

## Best Possible Solution

Define one canonical deleted-account lifecycle policy, then implement the approved recovery or support path across OAuth callback handling, account deletion/admin tooling, UI messaging, specs/docs, and regression tests.

## Reproduction Assessment

Yes, from source inspection. deleteAccount marks the user deactivated/purged, the OAuth callback rejects that state, and active auth helpers expose no active user, matching an authorize-then-return-to-signed-out loop.

## Solution Assessment

No, a one-off purge is not the best source-level solution. The maintainable path needs an explicit recovery or support policy that preserves OAuth identity binding, irreversible deletion semantics, and ban protections.
