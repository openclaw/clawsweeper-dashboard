## Summary

Keep this issue open. Current main still treats self-deleted GitHub-linked ClawHub accounts as terminal during OAuth, and choosing a purge, restore, or re-registration path needs maintainer product, security, and production-support handling rather than conservative cleanup.

## What This Changes

The issue requests a purge or recovery path so a ClawHub account deleted from settings can re-register or sign in with the same GitHub identity after OAuth.

## Best Possible Solution

Define one canonical account lifecycle policy for deleted GitHub-linked accounts, then implement the approved recovery or support behavior across Convex auth, account deletion/admin tooling, UI messaging, specs/docs, and regression tests.

## Reproduction Assessment

Yes, from source inspection. deleteAccount sets deactivatedAt/purgedAt, the OAuth callback rejects deactivated users, and active auth helpers expose no active user, matching the reported signed-out result after authorization.

## Solution Assessment

No, a one-off purge is not the best source-level solution. The maintainable path needs an explicit recovery or support policy that preserves OAuth identity binding, irreversible deletion semantics, and ban protections.
