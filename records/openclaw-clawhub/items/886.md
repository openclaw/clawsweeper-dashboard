## Summary

Keep open. Current main still treats user-initiated ClawHub account deletion as terminal during GitHub OAuth, so the reported authorize-then-return-to-signed-out loop remains a real account-lifecycle issue; the ClawHub cleanup policy also keeps open issues in maintainer hands unless they are PRs already implemented on main.

## What This Changes

The issue asks for recovery, purge, or re-registration support when a GitHub-linked ClawHub account deletion leaves the user unable to sign back in.

## Best Possible Solution

Define one canonical account lifecycle policy for deleted GitHub-linked accounts, then implement the approved recovery behavior across the Convex auth callback, account deletion or admin tooling, UI error messaging, docs, and regression tests.

## Reproduction Assessment

Yes. Current main has a high-confidence source/test reproduction path: deleteAccount sets deactivatedAt/purgedAt, the OAuth callback rejects that user on reauth, and active auth helpers then hide the user from app sessions.

## Solution Assessment

No for a one-off purge as the general fix. The maintainable solution should be an explicit support or product-approved recovery path that keeps ban and OAuth identity protections intact.
