## Label Justifications

- `P1`: The issue blocks affected users from signing in after deletion and has repeated related reports, while the fix needs urgent maintainer policy handling.
- `impact:security`: Deleted-account recovery touches identity binding, ban boundaries, irreversible deletion semantics, and support/admin authority.
- `impact:auth-provider`: The failure occurs in the GitHub OAuth create/update callback and linked auth account flow.
- `impact:session-state`: The observed result is an OAuth-completes state where ClawHub active-session helpers still return no active user.

## Summary

Keep open: current main still treats self-deleted ClawHub accounts as terminal during GitHub OAuth, so the reported authorize-then-return-to-sign-in behavior is source-reproducible and still needs maintainer product/security direction.

## What This Changes

Request to recover or re-register a GitHub-linked ClawHub account after self-service account deletion causes a sign-in loop.

## Best Possible Solution

Define one canonical deleted-account lifecycle policy, then implement the approved recovery or support path across Convex auth, admin tooling, UI messaging, specs/docs, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: self-service deletion sets deactivatedAt/purgedAt, GitHub OAuth reauth rejects that user, and active auth helpers return no active session; I did not run a live production deletion test.

## Solution Assessment

No. A one-off purge is not the best repo-level solution because deleted-account recovery needs an approved lifecycle policy that preserves OAuth identity binding, deletion guarantees, and ban boundaries.
