## Summary

Keep open: current main still treats self-deleted ClawHub accounts as terminal during GitHub OAuth, and the remaining recovery path needs explicit account-lifecycle, support, and security policy.

## What This Changes

The issue asks ClawHub to recover or purge a deleted GitHub-linked account that now returns to the sign-in page after OAuth authorization.

## Best Possible Solution

Define one canonical deleted-account lifecycle policy, then implement the approved OAuth recovery or support path with admin tooling, UI messaging, specs/docs, and regression coverage for deleted, deactivated, banned, and legacy deletedAt users.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the lockout state: self-service deletion sets deactivatedAt/purgedAt, OAuth reauth rejects that user, and active auth helpers return no active session; I did not run a live production deletion test.

## Solution Assessment

No. A one-off purge is not the best repo-level solution because deleted-account recovery needs an approved lifecycle policy that preserves OAuth identity binding, deletion guarantees, and ban boundaries.
