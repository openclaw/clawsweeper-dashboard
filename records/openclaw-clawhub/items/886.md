## Summary

Keep this issue open. Current main still blocks self-deleted GitHub-linked accounts during OAuth, the named-account cleanup is production support work, and changing recovery behavior requires maintainer product/security policy rather than conservative issue cleanup.

## What This Changes

The issue requests production cleanup or a recovery path so a GitHub-linked ClawHub account deleted in settings can sign in or re-register instead of returning to Sign In after OAuth.

## Best Possible Solution

Define one canonical account lifecycle policy for deleted GitHub-linked accounts, then implement the approved recovery or support behavior across Convex auth, account deletion/admin tooling, UI messaging, specs/docs, and regression tests.

## Reproduction Assessment

Yes, from source inspection. deleteAccount sets deactivatedAt/purgedAt, the OAuth callback rejects deactivated users, and active auth helpers expose no active user, matching the reported signed-out result after authorization.

## Solution Assessment

No, a one-off purge of the named account is not the best source-level solution. The maintainable path needs an explicit recovery or support policy that preserves OAuth identity binding, irreversible deletion semantics, and ban protections.
