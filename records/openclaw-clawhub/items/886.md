## Summary

Keep this issue open. Current main still treats self-deleted GitHub-linked accounts as terminal during OAuth and tests that behavior, while the repository cleanup policy for ClawHub issues does not allow auto-closing open issues even when the implementation is well understood.

## What This Changes

The issue asks for a recovery, purge, or re-registration path after a user deletes their ClawHub account and can no longer complete GitHub sign-in.

## Best Possible Solution

Define one canonical lifecycle policy for deleted GitHub-linked accounts, then implement the approved recovery or support path across auth, admin tooling, user messaging, specs/docs, and regression tests.

## Reproduction Assessment

Yes, from source inspection. Deleting an account writes deactivatedAt/purgedAt, the GitHub OAuth callback rejects that account on later sign-in, and active auth helpers then hide the user from the app session.

## Solution Assessment

No, a one-off purge is not the best maintainable fix. The safer path is an approved account lifecycle policy with guarded recovery or support tooling that preserves OAuth binding and ban/deletion boundaries.
