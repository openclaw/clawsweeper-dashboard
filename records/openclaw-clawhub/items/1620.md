## Label Justifications

- `P1`: The issue blocks real users from accessing ClawHub through GitHub OAuth after account deletion.
- `impact:security`: Any recovery path touches account identity, irreversible deletion, ban boundaries, and ownership trust.
- `impact:auth-provider`: The lockout is specifically tied to GitHub OAuth identity binding and sign-in completion.
- `impact:session-state`: OAuth can return to ClawHub while the user remains unable to enter an authenticated account state.

## Summary

Keep open. Current main still intentionally blocks GitHub sign-in for self-deleted accounts, and resolving this support case would require a maintainer security/product decision rather than cleanup closure.

## What This Changes

The issue requests a way to sign in or re-register with GitHub after accidentally deleting a ClawHub account.

## Best Possible Solution

Define the account-recovery policy first, then either document irreversible deletion as the supported outcome or add an audited admin/support flow that can safely reactivate or unlink self-deleted GitHub identities without bypassing bans or ownership safeguards.

## Reproduction Assessment

Yes, at source level: deleteAccount writes deactivatedAt/purgedAt, and handleDeletedUserSignIn rejects that user on the next OAuth completion. I did not run a live OAuth flow, but current tests explicitly assert the blocked reauth path.

## Solution Assessment

No autonomous code fix is clearly best yet. The maintainable path depends on a maintainer decision about whether permanent deletion truly forbids recovery or whether ClawHub should add a constrained, audited support flow.
