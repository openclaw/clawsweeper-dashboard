Keep open; current main still implements permanent deleted-account lockout, the live issue has a second affected user plus related open reports, and restoration or GitHub identity unlinking is security-sensitive operator/product work outside cleanup closure.

## What This Changes

The issue requests account restoration or GitHub identity unlinking so the reporter can re-register after accidentally deleting a ClawHub account.

## Best Possible Solution

Define an audited support/admin recovery or GitHub unlink flow, or explicitly document permanent deletion and route affected users through a private operator process that preserves ban, purge, token, publisher, and audit boundaries.

## Reproduction Assessment

Yes for the general lockout: source inspection shows deleteAccount sets deactivatedAt/purgedAt and GitHub sign-in rejects deactivated users. This does not verify the reporter's production rows.

## Solution Assessment

No for ad hoc restoration or unlinking from a public issue. The maintainable path is a maintainer-approved, audited recovery/unlink policy or an explicit documented no-recovery decision.
