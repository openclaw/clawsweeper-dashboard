## Summary

Keep open. This is an active production GitHub OAuth/session failure: current main requires a callback `code` or `error`, the live sign-in endpoint still returns HTTP 500, and ClawHub issue-cleanup policy keeps issues open rather than auto-closing them.

## What This Changes

The issue reports that the production GitHub OAuth callback redirects back to the ClawHub app root without a frontend exchange code, leaving users unable to complete login.

## Best Possible Solution

Restore GitHub sign-in so valid active accounts receive a frontend exchange code and persisted session, while blocked/deleted accounts receive explicit safe errors and production env/callback alignment is covered by smoke checks.

## Reproduction Assessment

No. I did not replay a valid GitHub OAuth callback that redirects to the root without a code, but source inspection confirms that a no-code/no-error return cannot create a session and live production signin endpoints currently return HTTP 500.

## Solution Assessment

Unclear because there is no proposed patch and the failure crosses frontend callback handling, Convex Auth, deployed env, OAuth app settings, cookies, and account state. The narrow maintainable path is operator-backed diagnosis plus targeted regression coverage, not a blind source-only repair.
