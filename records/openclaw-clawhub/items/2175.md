## Summary

Keep open: the report is concrete, live production sign-in is still failing before GitHub OAuth, current main still requires a callback exchange code to create a frontend session, and related OAuth/session reports remain open for maintainer/operator triage.

## What This Changes

Reported production GitHub OAuth callback flow returns to the app root without `code` or `error`, leaving `AuthCodeHandler` unable to exchange a session.

## Best Possible Solution

Restore production GitHub sign-in end to end for active accounts, preserve deleted/deactivated/banned account rejection, choose a canonical OAuth/session tracker, and add redacted browser-level OAuth smoke proof plus targeted regression coverage for the specific fix.

## Reproduction Assessment

Yes for the broader production auth failure: on May 18, 2026, both the canonical and direct Convex GitHub sign-in endpoints returned HTTP 500. The exact callback-without-code stage was not replayed because sign-in currently fails earlier, but source confirms a root redirect without `code` cannot create a session.

## Solution Assessment

Yes for keeping this open and routing it to maintainer/operator auth triage. A source-only cleanup PR is not the best next step until production logs, OAuth settings, cookies, and account-state behavior identify the failing branch.
