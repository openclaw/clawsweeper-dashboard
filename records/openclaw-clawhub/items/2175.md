## Summary

Keep open. The current frontend still requires a callback `code` or `error` to complete session creation, production sign-in endpoints currently return HTTP 500 before a full callback replay, and related OAuth/session issues remain open under the ClawHub cleanup policy.

## What This Changes

The issue reports that GitHub OAuth redirects back to `clawhub.ai` without a frontend exchange `code` or `error`, so the app never creates a session.

## Best Possible Solution

Restore production GitHub OAuth for active accounts with import-safe diagnostics or deploy validation, preserve blocked-account rejection, and add focused regression coverage plus redacted live smoke proof.

## Reproduction Assessment

No high-confidence exact browser reproduction of the reported callback redirect was completed. Source inspection confirms a no-code redirect cannot create a session, and live production currently fails at `/api/auth/signin/github` with HTTP 500 before that callback path can be replayed.

## Solution Assessment

Unclear because there is no proposed patch and the failure crosses app code, Convex Auth, deployed env, OAuth app settings, cookies, and account state. The narrow maintainable path is maintainer/operator diagnosis with request-time or deploy validation rather than eager module-load auth failures.
