## Summary

Keep open. This remains an active GitHub OAuth/session failure: current main still requires a callback `code` or `error` to complete frontend session creation, live sign-in endpoints currently return HTTP 500, and ClawHub cleanup policy keeps issues open unless a merged fix resolves them.

## What This Changes

Report that GitHub OAuth redirects back to the ClawHub app root without `code` or `error`, leaving the frontend unable to exchange a code or persist a session.

## Best Possible Solution

Run an operator-backed OAuth/session investigation, restore active GitHub account session creation, preserve explicit deleted/banned-account rejection, add import-safe diagnostics for auth misconfiguration, and capture redacted production smoke proof.

## Reproduction Assessment

No for the exact callback redirect. Source inspection confirms that a no-code/no-error root redirect cannot create a session, and live production sign-in currently returns HTTP 500, but I did not complete a full valid GitHub OAuth callback replay in a browser.

## Solution Assessment

Unclear because there is no proposed patch and the failure crosses frontend callback handling, Convex Auth routes, OAuth provider configuration, production env, and account state. The safer direction is operator diagnosis plus import-safe request-time or deploy validation rather than eager module-load auth failure.
