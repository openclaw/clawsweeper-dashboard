## Summary

Keep open. This is still an active GitHub OAuth/session failure: current main depends on a callback `code` or `error`, production sign-in endpoints currently return HTTP 500, related auth reports remain open, and ClawHub issue cleanup policy does not auto-close issues without a merged fix.

## What This Changes

The issue reports that GitHub OAuth completes its Convex callback but redirects back to ClawHub without a frontend exchange code, leaving the app unable to create a session.

## Best Possible Solution

Run an operator-backed OAuth/session investigation, restore active GitHub account session creation, preserve explicit deleted/banned-account rejection, add import-safe diagnostics for auth misconfiguration, and capture redacted production smoke proof.

## Reproduction Assessment

No for the exact callback redirect. Source inspection confirms that a no-code/no-error root redirect cannot create a session, and live production sign-in currently returns HTTP 500, but I did not complete a full valid GitHub OAuth callback replay in a browser.

## Solution Assessment

Unclear because there is no proposed patch and the failure crosses frontend callback handling, Convex Auth routes, OAuth provider configuration, production env, and account state. The safest path is operator diagnosis plus import-safe request-time or deploy validation rather than an eager module-load auth failure.
