## Summary

Keep open. Current main has the detailed package `staticScan` data and the live version API already exposes the named finding, but `package moderation-status` and default human `package inspect` still show only aggregate scan state; the remaining work needs an explicit scanner-disclosure policy.

Request to expose scanner rule ID plus file:line detail in package moderation-status and inspect output, and to document server-side scanner rule IDs for false-positive triage.

## Best Possible Solution

Define a scanner finding disclosure contract, then expose sanitized rule/file/line detail consistently in `package moderation-status`, human `package inspect`, and docs while reusing the existing `staticScan` data shape.

## Reproduction Assessment

Yes. Source inspection shows the requested `moderation-status` and human `inspect` surfaces omit finding details, while the live package version endpoint for the reported version returns the exact rule and file:line finding.

## Solution Assessment

Unclear. Reusing the existing `staticScan` findings is the narrow technical path, but the maintainable fix needs maintainer approval for disclosure audience, evidence redaction, and whether to document rule IDs without full patterns.
