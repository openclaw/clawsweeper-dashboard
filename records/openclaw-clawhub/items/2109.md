## Summary

Keep open. Current main still has the owner-facing diagnostics gap: detailed static scan findings exist on version/security surfaces, but `package moderation-status` and normal package inspect output only expose aggregate status, and deciding how much scanner detail to disclose is a security/product decision.

## What This Changes

Expose scanner finding code/file/line in package moderation-status or inspect output and document enough server-side scanner rule detail for false-positive triage.

## Best Possible Solution

Expose a narrow authenticated owner/moderator path for existing staticScan code/file/line details in CLI/API output, document the intended disclosure boundary, and keep broader rule catalogs or raw scanner evidence behind explicit security approval.

## Reproduction Assessment

Yes for the output limitation: current source shows moderation-status and normal package inspect output omit scanner finding details, while the live version API exposes code, file, and line for the cited package. The false-positive safety judgment itself still needs moderator/security review because the aggregate verdict includes non-static scanner state.

## Solution Assessment

Unclear until maintainers set the disclosure boundary. The narrow maintainable direction is owner/moderator access to existing staticScan code/file/line fields; broader rule catalogs, regexes, raw evidence, LLM findings, or VT details need security/product approval first.
