## Summary

Keep open. Current main still does not expose package static scanner rule/file/line findings through `package moderation-status` or the human `package inspect` formatter, and the requested disclosure is security-sensitive enough to need maintainer policy before implementation.

## What This Changes

The issue requests owner-visible static scanner finding details, including rule IDs and file:line locations, in package moderation-status and inspect output.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, expose approved rule IDs and file:line details to authorized package owners and staff in moderation-status/inspect, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes, by source inspection. Current main stores package static findings with rule and location data, while the moderation-status schema/backend and human package inspect formatter still expose only aggregate scan and moderation fields.

## Solution Assessment

Unclear until maintainers set disclosure policy. The safest direction is owner/staff-scoped sanitized rule and file:line output, not broad raw evidence exposure across public surfaces.
