## Summary

Current main still leaves package owners with aggregate moderation status even though per-finding rule and location data is stored; the request remains necessary, but the durable moderation spec deliberately classifies static findings as internal ClawScan evidence, so maintainers must approve a narrower owner-safe disclosure contract before implementation. The earlier https://github.com/openclaw/clawhub/issues/731 covered a broader skill-scanner transparency and local-scan request and is closed, while no viable open or merged PR now owns this package-specific remainder.

## What This Changes

Expose authenticated package owners to scanner rule identifiers and file/line locations in package moderation or inspect output, and document the available server-side rule identifiers.

## Best Possible Solution

Document an owner-only scanner-diagnostic contract that exposes a stable rule code, normalized file path, and line while withholding matcher internals and sensitive evidence, then implement it through one canonical package API/CLI surface with authorization, rate limiting, redaction, and focused runtime coverage.

## Reproduction Assessment

Yes, at source level. Current main’s authenticated package moderation response returns aggregate scan state and reasons while omitting the stored per-finding code, file, and line fields.

## Solution Assessment

Unclear until maintainers approve the disclosure contract. The reporter’s owner-only rule/file/line proposal is a plausible narrow direction, but the supported command/API surface, redaction rules, rate limit, and treatment of existing public package-version data must be decided together.
