## Summary

Keep open. Current main still stores scanner finding detail but does not expose rule/file/line detail through the requested package moderation-status or human inspect paths, and the remaining work needs a maintainer security/product disclosure decision.

## What This Changes

Request to expose scanner rule IDs and file:line finding details in package moderation-status/inspect output and document server-side rule identifiers for false-positive triage.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, then expose approved rule IDs and file:line findings for authorized package owners and staff across moderation-status, package inspect, and public docs.

## Reproduction Assessment

Yes. Source inspection on current main shows staticScan findings carry code/file/line/evidence, while moderation-status schema/query and human CLI output return only aggregate scan and reason fields.

## Solution Assessment

Unclear. Surfacing sanitized rule/file/line detail is likely the right direction, but raw staticScan exposure is not the safest implementation without a maintainer-approved visibility and redaction policy.
