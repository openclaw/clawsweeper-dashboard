## Summary

Keep open. Current main already has structured static-scan findings and exposes them on the package version API, but the owner moderation-status response and normal human package inspect output still do not surface rule/file/line details, and broader scanner disclosure needs maintainer policy.

## What This Changes

Request to expose static scanner finding details such as rule ID, file, and line in package moderation-status and inspect output, plus document server-side rule IDs for publisher triage.

## Best Possible Solution

Define the scanner disclosure and redaction contract, then expose sanitized staticScan reasonCodes/findings consistently in owner/staff moderation-status, package inspect output, and public docs for stable rule IDs.

## Reproduction Assessment

Yes. Source inspection shows moderation-status and human package inspect omit finding detail, while the live package version API confirms rule/file/line data exists for the named package.

## Solution Assessment

Unclear until maintainers decide disclosure policy. Once approved, the maintainable path is to reuse the existing staticScan model instead of adding one-off strings to a single command.
