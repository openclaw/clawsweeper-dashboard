## Summary

Keep open. Current main and the live API can expose package version static-scan findings, including the reporter's exact rule and file:line, but `package moderation-status` and normal human `package inspect` still do not provide a deliberate, documented scanner-diagnostic contract; ClawHub cleanup policy also keeps issues open for maintainer follow-up.

## What This Changes

The issue asks ClawHub to expose static scanner finding details, such as rule ID and file:line, in CLI moderation-status or inspect output and to document server-side rule identifiers for publisher false-positive triage.

## Best Possible Solution

Use the existing package version `staticScan` data as the source of truth, define a sanitized owner/staff/public diagnostic contract, expose it consistently through `moderation-status` and human/JSON `inspect`, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes, by source inspection and live API checks. Current main stores and can expose static finding detail through package version JSON, while moderation-status and normal human package inspect output still omit a deliberate rule/file/line diagnostic view.

## Solution Assessment

Unclear until maintainers decide the disclosure contract. The requested direction is reasonable, but the maintainable solution should align existing version JSON, moderation-status, inspect output, docs, and redaction behavior instead of adding another inconsistent scanner surface.
