## Summary

Keep open. Current main stores and serves package static-scan finding detail on the version API, but `package moderation-status` and human `package inspect` still expose only aggregate scan state/reasons; exposing rule/file/line detail needs an explicit security/product disclosure policy.

## What This Changes

The issue requests rule ID and file:line scanner finding detail in package moderation-status and inspect output, plus documentation of server-side rule IDs for false-positive triage.

## Best Possible Solution

Define a scanner disclosure contract, then expose sanitized staticScan reasonCodes and rule/file/line findings consistently for authorized package owners and staff in moderation-status, package inspect, and docs.

## Reproduction Assessment

Yes. Source inspection shows `moderation-status` and human `package inspect` omit finding details, while the live package version endpoint for the named package returns concrete rule and file:line data.

## Solution Assessment

Unclear. Surfacing sanitized rule/file/line details is likely the right direction, but maintainers should first decide access boundaries and redaction policy for scanner findings.
