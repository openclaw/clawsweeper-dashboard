## Summary

Keep open. This is a recent ClawHub issue, and current main still does not expose scanner rule/file/line detail through the owner-facing moderation-status or human inspect paths; the remaining decision is a security/product disclosure contract rather than a cleanup-close candidate.

## What This Changes

Expose scanner finding detail such as rule ID and file:line in package moderation-status and inspect output, and document server-side scanner rule identifiers for false-positive triage.

## Best Possible Solution

Define a scanner disclosure/redaction contract, then expose sanitized staticScan reason codes and rule/file/line findings consistently for authorized package owners and staff in moderation-status, inspect, and public docs as appropriate.

## Reproduction Assessment

Yes. Source inspection on current main shows the stored static scan findings include rule and location data, while moderation-status and human inspect omit that detail and return only aggregate status/reason fields.

## Solution Assessment

Unclear. Surfacing sanitized rule/file/line detail is likely the right direction, but the best implementation depends on maintainer approval for who can see rule IDs, file paths, line numbers, evidence snippets, and server-side rule documentation.
