## Summary

Keep open. Current main has the underlying package static-scan finding data and even exposes it through package version JSON, but the requested `package moderation-status` response/human output and normal inspect flow still surface only aggregate scan status/reasons; changing scanner disclosure also needs maintainer security/product approval.

## What This Changes

Expose scanner finding details such as rule id and file:line in package moderation-status or inspect output, and document server-side scanner rule IDs for publisher triage.

## Best Possible Solution

Define the owner/staff/public scanner disclosure policy first, then expose sanitized package staticScan reasonCodes/findings through moderation-status and human inspect output, with docs for stable rule IDs and the existing version endpoint as a fallback path.

## Reproduction Assessment

Yes. Source inspection shows the moderation-status query/schema and CLI printer omit finding details, while the live package version endpoint confirms the underlying static finding exists for the named package.

## Solution Assessment

No, the current state is only a partial workaround. The maintainable solution is not to guess at every scanner rule in CLI code, but to approve a stable disclosure contract and reuse the stored staticScan finding model consistently across API, CLI, and docs.
