## Summary

Keep open. Current main still does not provide the requested owner-facing scanner rule and file:line diagnostics in `package moderation-status` or normal human `package inspect`, and the remaining work needs a security/product decision about what scanner detail is safe to disclose.

## What This Changes

Expose scanner finding details such as rule ID and file:line in moderation-status and inspect output, and document server-side rule identifiers for false-positive triage.

## Best Possible Solution

Define a sanitized owner/staff-scoped scanner diagnostics contract, expose it consistently through `moderation-status` and human/JSON `inspect`, and document intentionally public rule IDs without exposing evasion-sensitive regexes or raw evidence.

## Reproduction Assessment

Yes, by source inspection. Current main stores static finding rule/location data and can render it in security pages, while `package moderation-status` and normal human package inspect output still omit the requested rule ID and file:line detail.

## Solution Assessment

Unclear until maintainers decide the disclosure policy. The maintainable path is a redacted diagnostic contract with owner/staff/public boundaries, not an unfiltered dump of stored scanner evidence or private regexes.
