## Summary

Keep open. Current main still leaves the requested owner-facing diagnostic gap in `package moderation-status` and normal package inspect output, while the live version API shows that static scanner code/file/line data already exists; exposing it safely needs a maintainer security/product decision.

## What This Changes

Request to expose scanner rule and file:line details in package moderation-status and inspect output, and to document scanner rule IDs for false-positive triage.

## Best Possible Solution

Expose a narrow authenticated owner/moderator path for existing staticScan code/file/line/message details, document the supported fields and stable rule IDs, and keep raw evidence, regexes, LLM/VT internals, and broader rule catalogs behind explicit security approval.

## Reproduction Assessment

The full false-positive claim is only partly reproduced because the aggregate hosted verdict also includes LLM suspicious state.

## Solution Assessment

Yes directionally for a narrow owner-facing disclosure of existing staticScan fields. Publishing broader server-side rule catalogs, regexes, raw snippets, LLM findings, or VT details needs maintainer security/product approval first.
