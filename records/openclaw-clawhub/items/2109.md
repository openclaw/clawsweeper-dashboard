## Summary

Keep open. Current main still stores scanner finding detail but does not surface rule/file/line detail through the owner-facing package moderation-status path or the human package inspect summary; because this is a recent ClawHub issue and scanner-detail disclosure is a security/product contract, it is not a cleanup-close candidate.

## What This Changes

Expose scanner rule/file/line details and server-side rule documentation in package moderation-status and inspect output so publishers can triage false-positive scans.

## Best Possible Solution

Define a scanner disclosure and redaction contract, then expose sanitized staticScan reason codes plus rule/file/line findings consistently for authorized package owners and staff in moderation-status, inspect, and public docs where approved.

## Reproduction Assessment

Yes. Source inspection on current main shows package static findings store rule/location data, while moderation-status and the human inspect summary omit that detail and return only aggregate scan fields.

## Solution Assessment

Unclear. Surfacing sanitized rule/file/line detail is likely the right direction, but the exact solution depends on maintainer decisions about who can see rule IDs, file paths, line numbers, evidence snippets, and server-side rule documentation.
