## Summary

Keep open. Current main has structured static-scan findings and the version API can expose them, but `package moderation-status` and default human `package inspect` still show only aggregate scan state; exposing rule/file/line detail needs an explicit scanner-disclosure policy.

## What This Changes

Request to show scanner rule IDs and file:line finding detail in package moderation-status/inspect output and document server-side rules for publisher false-positive triage.

## Best Possible Solution

Define a scanner finding disclosure/redaction policy, then expose sanitized `staticScan.reasonCodes` and findings consistently in owner/staff moderation-status and package inspect output, with docs for stable rule IDs.

## Reproduction Assessment

Yes. Source inspection shows the moderation-status schema/query and CLI printer omit finding detail, while the live package version endpoint for the reported package contains rule/file/line staticScan data.

## Solution Assessment

No for the current aggregate-only output. The maintainable path is a maintainer-approved disclosure contract that reuses the existing staticScan model across API, CLI, and docs rather than adding ad hoc CLI text.
