## Summary

Keep open. Current main still leaves the owner-facing diagnostic gap: package moderation-status returns only aggregate scan/reason fields, while detailed static scan code/file/line data exists on the package version API and broader disclosure needs product/security approval.

## What This Changes

The issue asks ClawHub to show scanner rule and file/line details in package moderation-status or inspect output so publishers can triage false positives.

## Best Possible Solution

Expose a safe authenticated owner/moderator path for existing staticScan code/file/line/message details, document the intentionally disclosed fields, and keep broader rule catalogs or raw evidence behind security/product approval.

## Reproduction Assessment

Yes for the reported output limitation: current source shows package moderation-status and normal package inspect output omit scanner finding details, while the live package version API for the cited release returns code, file, and line. The false-positive safety judgment itself still needs moderator/security review because the aggregate verdict includes non-static scanner state.

## Solution Assessment

Yes directionally for a narrow owner-facing disclosure of existing staticScan fields. Publishing broader server-side rule catalogs, regexes, raw evidence, LLM findings, or VT details is not safe without maintainer security/product approval first.
