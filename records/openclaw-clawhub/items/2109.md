## Summary

Keep open. Current main still leaves the requested package scanner detail unresolved in `moderation-status` and default human `inspect` output, and this is a security/product disclosure decision rather than a conservative cleanup close.

## What This Changes

Request to expose scanner rule IDs and file:line findings in package `moderation-status`/`inspect` output, document server-side rules, and help triage a suspicious package false positive.

## Best Possible Solution

Define a scanner disclosure/redaction policy, then expose sanitized staticScan reasonCodes/findings consistently to authorized package owners/staff in moderation-status and human inspect output, with docs for stable rule IDs and separate handling for LLM/VT verdicts.

## Reproduction Assessment

Yes. Source inspection shows moderation-status query/schema/CLI omit finding details, while the package version API and live named package show rule/file/line staticScan data exists.

## Solution Assessment

No. The current aggregate-only output is not enough for false-positive triage; the maintainable path is a policy-approved disclosure contract that reuses the existing staticScan model instead of ad hoc CLI strings.
