## Summary

Keep open. Current main and production expose enough version-level staticScan data to identify the @lobstah finding, but package moderation-status and normal human package inspect output still lack a deliberate scanner rule/file/line diagnostic contract.

## What This Changes

Request to show scanner rule IDs and file:line locations in package moderation-status/inspect output and document public server-side rule identifiers for false-positive triage.

## Best Possible Solution

Define a sanitized owner/staff-scoped scanner diagnostic contract for rule ID, file:line, message, and redacted evidence, expose it consistently through moderation-status and package inspect, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes, by source and live-output inspection. Version JSON can show the @lobstah rule/file/line detail, while current main's moderation-status schema/backend/CLI and normal package inspect output do not expose that detail.

## Solution Assessment

Unclear until maintainers decide disclosure policy. The narrow maintainable path is a sanitized owner/staff diagnostic surface, not an unfiltered dump of stored evidence or private regex internals.
