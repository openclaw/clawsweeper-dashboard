## Summary

Keep open. Current main still leaves `package moderation-status` and human `package inspect` without the requested scanner rule/file/line diagnostics, while the stored package static-scan model already contains that data and raw evidence that needs a disclosure policy.

## What This Changes

The issue requests scanner finding details, including rule IDs and file:line locations, in package moderation-status or inspect output for false-positive triage.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, expose approved rule IDs and file:line details consistently to authorized publishers and staff, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes, by source inspection. Current main stores package static findings with code/file/line/evidence, while the package moderation-status schema/backend and human inspect formatter expose only aggregate scan and moderation information.

## Solution Assessment

Unclear until maintainers set disclosure policy. The maintainable direction is sanitized owner/staff-scoped rule and file:line output, not a raw staticScan dump in every surface.
