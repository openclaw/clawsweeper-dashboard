## Summary

Keep open. The request is still only partially covered on current main: package version/security pages can carry static scan details, but `package moderation-status` and normal human `package inspect` remain aggregate-only and the disclosure policy is security-sensitive.

## What This Changes

Expose sanitized scanner finding details, such as rule id and file:line, in package `moderation-status` and `inspect` output and document public server-side rule identifiers for publisher false-positive triage.

## Best Possible Solution

Define one sanitized scanner-disclosure contract, expose approved rule ids and file:line details to authorized publishers and staff in the relevant CLI/API surfaces, and document intentionally public rule identifiers without dumping raw snippets by default.

## Reproduction Assessment

Yes, by source inspection. Current main stores rule and file/line data and partially exposes it through version/security surfaces, while `package moderation-status` and normal human `package inspect` do not surface sanitized finding details.

## Solution Assessment

Unclear until maintainers decide the disclosure boundary. The maintainable path is owner/staff-scoped rule and file:line output with raw evidence redacted or explicitly gated, not a direct dump of every stored scanner field.
