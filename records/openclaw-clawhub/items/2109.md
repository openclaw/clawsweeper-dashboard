## Summary

Keep open. Current main already stores and serves package static-scan findings on the package version API, including the reported package's rule and file:line detail, but `package moderation-status` and human `package inspect` still expose only aggregate scan state; exposing more scanner detail needs maintainer security/product policy for access and redaction.

## What This Changes

Requested behavior is to expose scanner rule and file:line detail in `clawhub package moderation-status` and human `package inspect`, and to document server-side rule identifiers for publisher false-positive triage.

## Best Possible Solution

Define a scanner disclosure and redaction contract, then expose sanitized rule/file/line details consistently for authorized package owners and staff in `package moderation-status`, human `package inspect`, and docs.

## Reproduction Assessment

Yes. Source inspection shows moderation-status and human inspect omit finding details, while the live version endpoint for the reported package returns the exact rule and file:line detail.

## Solution Assessment

Unclear for an automated implementation. Surfacing sanitized rule/file/line detail is likely the right direction, but maintainers should first decide the access, redaction, and server-rule disclosure boundaries.
