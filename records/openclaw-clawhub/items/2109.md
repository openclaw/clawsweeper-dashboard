## Summary

Keep open. Current main stores static finding details and can expose some of them through version inspect JSON, but the owner-facing `package moderation-status` API/CLI still returns only compact status and reason fields, and documenting or exposing server-side scanner rules remains a security/product decision.

## What This Changes

Request owner-facing scanner diagnostics with rule/file/line detail in package moderation-status or inspect output, plus documentation of server-side rules for false-positive triage.

## Best Possible Solution

Define an owner-only diagnostic contract that returns redacted static finding code/file/line details where appropriate, updates CLI/docs/specs, and keeps public trust responses compact.

## Reproduction Assessment

Yes for the current source shape: the moderation-status schema, backend query, and CLI output all omit static finding rule/file/line detail. I did not run a live owner token flow, but the source path is direct.

## Solution Assessment

Unclear until maintainers set the disclosure boundary. Owner-only redacted code/file/line details look like the narrowest useful path, while publishing broader rule documentation needs security approval.
