Keep open. Current main still leaves the requested diagnostic gap in `package moderation-status` and normal package inspect output, while the cited package version JSON shows `staticScan` already has rule/file/line data; exposing or documenting that detail needs a security/product disclosure decision.

## What This Changes

The issue asks ClawHub to surface static scanner rule and file/line details in package moderation/inspect output and document stable server-side rule IDs for false-positive triage.

## Best Possible Solution

Expose a narrow owner/staff-safe path for existing `staticScan` code/file/line/message details, document stable supported fields and rule IDs, and keep regexes, raw evidence, LLM/VT internals, and broader rule catalogs behind explicit security approval.

The broader false-positive claim is only partly reproduced because the hosted aggregate verdict also includes LLM suspicious state.

## Solution Assessment

Yes directionally for a narrow owner/staff disclosure of existing `staticScan` fields. Publishing broader server-side rule catalogs, regexes, raw snippets, LLM findings, or VT details needs maintainer security/product approval first.
