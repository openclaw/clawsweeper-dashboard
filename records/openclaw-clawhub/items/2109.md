## Summary

Keep open. This issue asks ClawHub to add scanner finding disclosure to moderation-status/inspect, and current main still stores detailed findings but does not expose the requested rule/file/line detail through the owner moderation-status CLI/API path; because the request involves scanner disclosure policy, it needs maintainer product/security judgment rather than cleanup closure.

## What This Changes

Expose scanner rule IDs and file:line locations in package moderation-status or inspect output, document server-side rule identifiers, and help publishers triage false positives.

## Best Possible Solution

Define a sanitized owner/staff-scoped scanner finding contract, expose approved rule IDs and file:line details through moderation-status/inspect, and document intentionally public rule identifiers without leaking raw evidence unnecessarily.

## Reproduction Assessment

Yes. Source inspection on current main shows detailed findings are stored, while the moderation-status schema, backend, and CLI formatter expose only aggregate scan/moderation fields for this owner triage path.

## Solution Assessment

Unclear until maintainers set disclosure policy. The maintainable approach is sanitized owner/staff-scoped rule and file:line output, not broad raw evidence exposure on public package surfaces.
