## Summary

Keep open. Current main still has only partial scanner-detail exposure: stored/static web views can represent rule and location data, but package `moderation-status` and normal human `package inspect` output remain aggregate-only, and exposing scanner detail needs an explicit security/product disclosure policy.

## What This Changes

The issue requests scanner rule and file:line details in `clawhub package moderation-status` and `clawhub package inspect`, plus public documentation of server-side scanner rule IDs for false-positive triage.

## Best Possible Solution

Define a sanitized owner/staff-scoped scanner diagnostic contract for rule IDs, file:line, message, and redacted evidence, expose it consistently through moderation-status and inspect, and document intentionally public rule IDs without dumping evasion-sensitive internals.

## Reproduction Assessment

Yes, by source inspection. Current main stores static finding rule/location data and renders it in some web/version surfaces, while package moderation-status and normal human package inspect output still omit the requested scanner finding detail.

## Solution Assessment

Unclear until maintainers decide the disclosure policy. The maintainable path is a redacted owner/staff diagnostic contract and documented public rule identifiers, not an unfiltered dump of stored scanner evidence or regexes.
