## Summary

Keep open. Current main already persists package static-scan findings and exposes them through the package version API, but the owner moderation-status API/schema and CLI output still omit rule/file/line detail; broader scanner disclosure also needs a security/product policy.

## What This Changes

Request to expose scanner rule IDs and file:line finding details in package moderation-status/inspect output, document server-side scanner rules, and help triage a specific package false positive.

## Best Possible Solution

Define the scanner disclosure/redaction/audience policy, then expose sanitized staticScan reasonCodes/findings consistently in owner/staff moderation-status and package inspect output while documenting stable rule IDs separately from VT/LLM verdicts.

## Reproduction Assessment

Yes. Source inspection shows moderation-status and default inspect omit finding details, while the public package version endpoint currently exposes rule/file/line data for the named version.

## Solution Assessment

Yes, with a policy boundary. Reusing the existing staticScan model is the narrow maintainable path, but raw ad hoc disclosure should wait for maintainer decisions on redaction, audience, and rule-ID stability.
