## Summary

Keep open. Current main still omits scanner rule/file/line details from the owner package moderation-status endpoint and human package inspect output, while stored package scan data and the web/version static-analysis path already carry those details; exposing more of them needs maintainer security disclosure policy.

## What This Changes

Request to expose scanner rule IDs and file:line findings in package moderation-status/package inspect output and document server-side scanner rule IDs for false-positive triage.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, expose approved rule IDs and file:line details to authorized owners and staff in moderation-status/package inspect, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes. Source inspection shows staticScan findings are stored and rendered in version/web static-analysis paths, while package moderation-status and human package inspect still return only aggregate scanner/moderation fields.

## Solution Assessment

Unclear until maintainers set disclosure policy. The maintainable direction is likely sanitized owner/staff-scoped rule and file:line output, not raw evidence broadened across all public surfaces.
