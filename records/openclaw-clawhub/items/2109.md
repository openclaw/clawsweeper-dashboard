## Summary

Keep open. Current main stores detailed static scan findings and the package version API already exposes them, but `package moderation-status` and the normal human inspect path still surface only aggregate scan state; expanding scanner disclosure needs an explicit maintainer security/product decision.

## What This Changes

Request to expose static scanner rule IDs and file:line details in package moderation-status/inspect output and document server-side rule IDs for false-positive triage.

## Best Possible Solution

Define owner/staff/public scanner disclosure and redaction policy, then expose sanitized staticScan reasonCodes/findings consistently through moderation-status, inspect output, and docs for stable rule IDs.

## Reproduction Assessment

Yes. Source inspection shows moderation-status and normal human inspect omit finding details, while the live version endpoint confirms the underlying rule/file/line data exists for the reported package.

## Solution Assessment

Unclear until maintainers decide the disclosure policy. The maintainable implementation should reuse the stored staticScan model with explicit redaction/audience rules rather than duplicating scanner logic in the CLI.
