## Summary

Keep open. Current main stores package static-scan finding detail and exposes it in version/static-analysis surfaces, but the owner `package moderation-status` response and human `package inspect` output still omit rule/file/line detail; this is also a security-sensitive disclosure policy change.

## What This Changes

Request to expose scanner rule/file/line details in package moderation-status and package inspect output, and to document server-side rule IDs for false-positive triage.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, expose approved rule IDs and file:line details to authorized package owners and staff in moderation-status and package inspect, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes, from source inspection. Current main stores rule/file/line findings and shows them in version/static-analysis paths, while moderation-status and human package inspect still return only aggregate scanner and moderation fields.

## Solution Assessment

Unclear until maintainers set the disclosure policy. Surfacing sanitized rule/file/line details to owners is likely the right direction, but returning raw staticScan findings wholesale would broaden evidence snippets and rule detail without explicit redaction boundaries.
