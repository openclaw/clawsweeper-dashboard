## Summary

Keep open. Current main still leaves a publisher-facing diagnostic gap: package moderation-status returns only aggregate scan/moderation fields, while the static scanner model already stores code/file/line finding detail; exposing that detail safely needs maintainer security/product approval rather than cleanup closure.

## What This Changes

The issue asks ClawHub to expose scanner finding detail such as rule id and file:line in moderation-status or inspect output so publishers can triage suspicious false positives.

## Best Possible Solution

Expose a narrow owner/staff-supported path for existing staticScan code/file/line/message details, document the stable CLI/API fields and current JSON workaround, and keep raw evidence, rule logic, LLM findings, and VT internals behind an approved disclosure boundary.

## Reproduction Assessment

Yes for the requested output gap: current source shows moderation-status and human package inspect do not render staticScan findings, while the schema and scanner already have code/file/line finding data. The specific false-positive claim still needs hosted scanner evidence to fully adjudicate.

## Solution Assessment

Yes directionally for a narrow owner/staff disclosure of existing staticScan fields. Publishing broader rule catalogs, raw evidence, regexes, LLM findings, or VT details is not safe without maintainer security/product approval.
