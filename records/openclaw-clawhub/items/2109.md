## Summary

Keep open. Current main already stores and serves static-scan finding details through the package version API and web scanner page, and the live reported package now shows the exact static finding, but `package moderation-status`, default human `package inspect`, and CLI docs still expose only aggregate scan state; disclosing scanner details needs maintainer security/product policy before implementation.

## What This Changes

Request to expose scanner rule IDs and file:line finding details in `clawhub package moderation-status` / `package inspect`, and to document server-side rule IDs for publisher triage.

## Best Possible Solution

Define scanner disclosure and redaction policy, then expose sanitized staticScan reasonCodes/findings consistently to the approved audience in moderation-status, package inspect, API/schema docs, and CLI tests.

## Reproduction Assessment

Yes. Source inspection shows current main omits finding detail from moderation-status and human inspect output while the live package version endpoint returns the exact rule, file, line, and evidence for the reported package.

## Solution Assessment

No. The current aggregate-only output is not enough for false-positive triage; the maintainable path is a deliberate disclosure contract that reuses the existing staticScan model instead of adding ad hoc strings to one command.
