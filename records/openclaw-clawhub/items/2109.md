## Summary

Keep open. Current main stores package static-scan findings and exposes them on the package version/security path, but `package moderation-status` and human `package inspect` still expose only aggregate scan state/reasons; the remaining work needs maintainer security/product policy on disclosure boundaries.

## What This Changes

Expose scanner finding details such as rule ID and file:line in package moderation-status and inspect output, and document server-side rule identifiers for publisher false-positive triage.

## Best Possible Solution

Define a scanner disclosure/redaction contract, then expose sanitized staticScan reasonCodes/findings consistently for authorized package owners and staff in `package moderation-status`, human `package inspect`, and docs.

## Reproduction Assessment

Yes. Source inspection shows the moderation-status schema/query and human package inspect output omit finding details, while the live version endpoint for the named package returns the exact rule and file:line detail.

## Solution Assessment

Unclear. Surfacing sanitized rule/file/line detail is likely the right direction, but the maintainable fix needs maintainer approval for owner/staff/public access, redaction, and whether evidence snippets or only rule/file/line should be exposed.
