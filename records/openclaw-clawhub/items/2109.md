## Summary

This issue should stay open: current main already persists and exposes static scan findings on version detail surfaces, and the live package API gives the reporter's concrete finding, but `package moderation-status` and the default human `package inspect` output still do not surface rule/file/line details or document the intended disclosure boundary.

## What This Changes

Expose scanner finding details such as rule id and file:line in ClawHub package moderation/inspect outputs and document the server-side scanner rule identifiers enough for false-positive triage.

## Best Possible Solution

Keep the issue open, decide the scanner-detail disclosure boundary, then expose the already persisted `staticScan.findings` in the owner-facing CLI/API path and document the safe rule-id level.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature/disclosure request. Source inspection confirms the current moderation-status path omits finding details while the version detail path already has them.

## Solution Assessment

Partly: reusing existing `staticScan.findings` for owner-facing CLI output is the narrowest technical path, but documenting server-side rules needs a maintainer security/product decision so exact evasion-sensitive patterns are not overexposed.
