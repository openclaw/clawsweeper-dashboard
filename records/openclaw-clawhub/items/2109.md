## Summary

Keep open. Current main stores package static-scan findings with rule and location data, but owner-facing package moderation-status and human package inspect still omit that detail, and the remaining work needs a security/product disclosure decision rather than conservative cleanup closure.

## What This Changes

Expose scanner rule IDs and file/line findings in package moderation-status or inspect output, and document server-side scanner signals for false-positive triage.

## Best Possible Solution

Define an authorized scanner-detail disclosure contract, then expose sanitized rule/file/line findings consistently for package owners and staff in moderation-status, package inspect, and docs.

## Reproduction Assessment

Yes. Source inspection on current main shows staticScan findings store rule and file/line data, while the moderation-status schema/query and human package inspect output return only aggregate scan status and reasons.

## Solution Assessment

Unclear. Surfacing sanitized rule/file/line detail is likely the right direction, but maintainers first need to decide who can see rule IDs, file paths, line numbers, evidence snippets, and server-side rule documentation.
