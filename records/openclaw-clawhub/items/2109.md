## Summary

Keep open. Current main has the scanner detail model and even exposes it through the package version/security-detail surface, but `package moderation-status` and default human `package inspect` still show aggregate status/reasons instead of the requested rule and file:line details.

## What This Changes

The issue requests scanner finding details such as rule ID and file:line in CLI/API moderation or inspect output, plus documentation of stable server-side scanner rule IDs.

## Best Possible Solution

Define the scanner disclosure contract, then consistently expose sanitized staticScan reason codes and finding locations in owner/staff moderation-status and package inspect output, with docs for stable public rule IDs.

## Reproduction Assessment

Yes. Source inspection on current main shows moderation-status and default human inspect omit scanner finding details, while the live version endpoint for the named package shows the underlying rule and file:line data exists.

## Solution Assessment

Unclear until maintainers decide the disclosure contract. Reusing the existing staticScan model across API, CLI, and docs is the narrow maintainable direction, but documenting rule IDs and evidence needs policy signoff.
