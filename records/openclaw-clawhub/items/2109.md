## Summary

Keep open. Current main still leaves the reported owner-facing gap: detailed `staticScan` findings exist on the package version API, but `package moderation-status` and normal package inspect output do not show rule/file/line detail, and broader rule disclosure needs maintainer security/product judgment.

## What This Changes

Surface scanner rule IDs plus file/line details in owner-facing package moderation-status or inspect output, and document the safe rule-disclosure boundary.

## Best Possible Solution

Expose a safe authenticated owner/moderator path for existing staticScan code/file/line details, document which fields are intentionally disclosed, and keep broader rule catalogs or raw scanner evidence behind explicit security policy approval.

## Reproduction Assessment

Yes for the reported output limitation: current source shows moderation-status and default package inspect omit scanner finding details, while the live version API for the cited package returns code, file, and line. The false-positive safety judgment itself still requires moderator/security review because the aggregate verdict includes non-static scanner state.

## Solution Assessment

Yes directionally, but only for a narrow owner-facing disclosure of existing staticScan fields. Publishing broader server-side rule catalogs, regexes, raw evidence, LLM findings, or VT details needs maintainer security/product approval first.
