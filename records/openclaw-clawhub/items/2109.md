## Summary

Keep open. Current main stores scanner finding detail and exposes it on the package version API, but the requested `moderation-status` and human `package inspect` surfaces still omit rule/file/line detail; because this is a ClawHub issue and the gap is security-adjacent, it needs maintainer disclosure-policy follow-up rather than cleanup closure.

## What This Changes

The issue requests scanner finding details, such as rule ID plus file and line, in `clawhub package moderation-status`, `clawhub package inspect`, and related documentation for publisher false-positive triage.

## Best Possible Solution

Define a sanitized scanner disclosure contract, then expose authorized staticScan reasonCodes and findings consistently in moderation-status, package inspect, and public/operator docs.

## Reproduction Assessment

Yes. Source inspection shows moderation-status and human package inspect omit staticScan findings even though the schema and version API can carry rule/file/line detail; the live named package also demonstrates stored staticScan detail exists.

## Solution Assessment

Unclear. Surfacing sanitized rule/file/line detail is likely the right product direction, but the maintainable solution depends on maintainer approval for owner versus staff versus public access, redaction, and whether evidence snippets should be exposed.
