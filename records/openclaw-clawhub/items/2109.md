## Summary

Keep open. Current main stores and serves package static-scan finding detail on the version endpoint, but `package moderation-status` and human `package inspect` still expose only aggregate scan state/reasons; exposing rule/file/line detail needs an explicit security/product disclosure policy.

## What This Changes

Request to expose scanner rule IDs and file/line details in package moderation-status or inspect output and document server-side scanner rules for false-positive triage.

## Best Possible Solution

Define a scanner disclosure/redaction contract, then expose sanitized staticScan reasonCodes/findings consistently for authorized package owners and staff in moderation-status, package inspect, and docs.

## Reproduction Assessment

Yes. Source inspection shows the moderation-status query/schema and human package inspect output omit finding details, while the live version endpoint for the named package returns the exact rule and file:line detail.

## Solution Assessment

Unclear. Surfacing sanitized details is likely the right direction, but the maintainable solution needs maintainer approval for owner/staff/public access, redaction, and whether evidence snippets or only rule/file/line should be exposed.
