## Summary

Keep open. Current main already stores and exposes package static-scan findings on the version/security surfaces, and the live package version confirms the requested rule/file/line data exists, but `package moderation-status` and human `package inspect` still only show aggregate scan state. This needs maintainer security/product judgment before widening scanner-detail disclosure.

## What This Changes

Request to expose static-scan rule, file, and line details in package moderation-status and inspect output, document server-side rule identifiers, and help triage a specific suspicious plugin verdict.

## Best Possible Solution

Define a scanner disclosure/redaction contract, then expose sanitized staticScan rule/file/line details consistently for authorized package owners and staff in `package moderation-status`, human `package inspect`, and user-facing docs.

## Reproduction Assessment

Yes. Source inspection shows moderation-status and human package inspect omit finding details, while the live version endpoint for the named package returns the exact static-scan rule and file:line detail.

## Solution Assessment

Unclear. Surfacing sanitized rule/file/line detail is likely the right direction, but the best implementation depends on maintainer decisions about disclosure audience, redaction, and whether evidence snippets or only rule/file/line should be exposed.
