## Summary

Keep open. Current main still does not expose rule/file/line scanner details in `clawhub package moderation-status` or normal human `clawhub package inspect`, even though static scan findings can be stored and the public audit UI can show some findings.

## What This Changes

Request scanner rule and file:line diagnostics in package moderation-status/inspect output, plus documented server-side rule IDs, so publishers can triage false positives.

## Best Possible Solution

Approve a narrow owner/staff diagnostic contract, expose existing staticScan code/file/line/message details in moderation-status and inspect, document the stable rule identifiers, and keep sensitive scanner internals behind the agreed disclosure boundary.

## Reproduction Assessment

Yes for the requested output gap: source inspection shows moderation-status and normal human package inspect omit staticScan rule/file/line details while the model can store them. The specific package false-positive claim still depends on live production scan and moderation data.

## Solution Assessment

Unclear as written: exposing existing owner/staff staticScan code/file/line/message fields looks maintainable, but publishing server-side rule catalogs or raw evidence needs maintainer security/product approval first.
