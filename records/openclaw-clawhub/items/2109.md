## Summary

Keep open. This is a valid, recent feature/security-policy request: current main stores scanner findings with rule, file, line, message, and evidence, and some package/version surfaces can expose staticScan data, but the owner moderation-status CLI/API output still only returns aggregate status and reasons; deciding which scanner details to disclose needs maintainer policy rather than automated closure.

## What This Changes

The issue asks ClawHub to expose scanner finding details such as rule ID and file:line in package moderation-status and inspect output, document server-side scanner rule identifiers, and help triage a specific false positive.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, expose approved rule IDs and file:line details to authorized owners and staff in moderation-status/package inspect, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes. Source inspection shows detailed staticScan findings are stored and version/detail schemas can carry them, while the owner moderation-status schema, Convex query, and CLI formatter still return only aggregate scan/moderation fields.

## Solution Assessment

Unclear until maintainers decide disclosure policy. The likely maintainable approach is sanitized owner/staff-scoped rule and location output, not broadly exposing raw scanner evidence on public package surfaces.
