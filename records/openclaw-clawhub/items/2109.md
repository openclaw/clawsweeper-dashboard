Keep open. Current main still leaves the owner-facing scanner diagnostic gap in `package moderation-status` and normal human package inspect output, and the requested disclosure boundary is security-sensitive enough to require maintainer product/security review before implementation.

## What This Changes

## Best Possible Solution

Expose a narrow authenticated owner/moderator path for existing staticScan code/file/line/message details, document the intended fields, and keep broader rule catalogs or raw evidence behind explicit security policy approval.

Yes for the output limitation at source level: current moderation-status schema/query/CLI omit staticScan findings, while the live version API for the cited release returns code, file, and line. ## Solution Assessment

Yes directionally for a narrow owner-facing disclosure of existing staticScan fields. Publishing broader server-side rule catalogs, regexes, raw snippets, LLM findings, or VT details needs maintainer security/product approval first.
