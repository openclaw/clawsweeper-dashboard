## Summary

Keep open. Current main already stores package static-scan findings and exposes them through the package version JSON/web static-analysis path, but the requested owner moderation-status response and human package inspect output still omit rule/file/line detail; this also needs a maintainer disclosure policy before implementation.

## What This Changes

The issue requests scanner finding details, including rule IDs and file:line locations, in moderation-status and inspect output to support false-positive triage.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, expose approved rule IDs and file:line details to authorized package owners and staff in moderation-status and package inspect, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes, from source inspection. Current main stores staticScan findings with rule/file/line data and exposes them in package version/static-analysis paths, while moderation-status and human package inspect still return only aggregate scanner and moderation fields.

## Solution Assessment

Unclear until maintainers set the disclosure policy. Surfacing sanitized rule/file/line details to owners is likely the right direction, but raw findings and evidence snippets should not be broadened without explicit access-control and redaction rules.
