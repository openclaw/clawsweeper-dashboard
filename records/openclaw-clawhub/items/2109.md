## Summary

Keep open. Current main still omits scanner finding details from `package moderation-status` and human `package inspect` output even though package version data already carries static scanner code/file/line information; exposing more detail needs a maintainer security/product decision.

## What This Changes

Request to expose static scanner rule, file, and line details in package moderation-status and inspect output, and to document server-side rule IDs for false-positive triage.

## Best Possible Solution

Expose a narrow authenticated owner/moderator path for existing staticScan code/file/line/message details, document stable supported rule IDs, and keep raw evidence, regexes, LLM/VT internals, and broader rule catalogs behind explicit security approval.

## Reproduction Assessment

Yes for the output limitation: current source shows moderation-status and human package inspect do not render staticScan finding details. The live version API for the cited package also returns concrete code/file/line data, while the full false-positive claim remains partly policy-dependent because the aggregate hosted verdict includes LLM suspicious state.

## Solution Assessment

Yes directionally for narrow owner-facing disclosure of existing staticScan fields. Publishing broader server-side rule catalogs, regexes, raw snippets, LLM findings, or VT details is not clearly safe without maintainer security/product approval.
