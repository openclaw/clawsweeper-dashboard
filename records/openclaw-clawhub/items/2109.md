## Summary

Keep open. Current main still lacks owner-visible scanner rule/file/line details in `package moderation-status` and the human `package inspect` output, and the remaining work needs a maintainer-approved scanner disclosure and redaction policy.

## What This Changes

The issue asks ClawHub to surface scanner rule and file:line details in package moderation-status or inspect output, document useful server-side rule IDs, and help triage a reported false positive.

## Best Possible Solution

Define a sanitized scanner-disclosure contract for authorized owners and staff, expose approved rule IDs plus file:line details in moderation-status and human inspect output, and document intentionally public rule identifiers without leaking raw evidence by default.

## Reproduction Assessment

Yes. Source inspection on current main shows package static findings store rule/file/line data, while the moderation-status schema, backend, and CLI formatter return only aggregate scan and moderation fields; I did not run an authenticated live moderation-status request.

## Solution Assessment

Unclear until maintainers decide the disclosure policy. The narrow maintainable direction is owner/staff-scoped rule and file:line output with raw evidence redacted or omitted by default, plus docs for stable public rule IDs.
