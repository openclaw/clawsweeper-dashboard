## Summary

Keep open. This is a valid feature/security-policy request: current main stores and can render scanner findings in some version/security surfaces, but package moderation-status and normal human package inspect output still expose aggregate scan state rather than owner-friendly rule/file/line triage details.

## What This Changes

The issue requests scanner finding details such as rule ID and file:line in package moderation-status or inspect output, plus documentation of server-side scanner rules for false-positive triage.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, expose approved rule IDs and file:line details to authorized owners and staff in package moderation-status and human inspect output, and document intentionally public rule identifiers and any existing version-JSON workaround.

## Reproduction Assessment

Yes. Source inspection on current main shows package scanner findings are stored and can appear in version/security surfaces, while package moderation-status and normal human package inspect output still omit scanner finding detail.

## Solution Assessment

Unclear until maintainers decide disclosure policy. The maintainable path is owner/staff-scoped sanitized rule and file:line output, not broad raw evidence exposure without redaction boundaries.
