## Summary

Keep open: current main has partial diagnostic paths for scanner details, but `package moderation-status` still exposes only aggregate scan/reason state and the remaining request needs a maintainer security/product decision about how much rule detail to expose.

## What This Changes

Request to expose scanner rule IDs plus file/line finding detail in package moderation or inspection diagnostics so publishers can triage false positives.

## Best Possible Solution

Define one canonical owner-only diagnostic path for package scanner findings, then document whether publishers should use scan report downloads, package inspect JSON, or a new redacted moderation-status summary.

## Reproduction Assessment

Yes. Source inspection shows `package moderation-status` still returns and prints aggregate scan state and reasons without rule/file/line finding objects; current main also shows partial alternate diagnostic paths through package version JSON and scan report downloads.

## Solution Assessment

Unclear. A direct dump of server-side rule details into moderation-status may not be the safest long-term interface; maintainers should choose a bounded owner-only disclosure path and update docs/tests around that contract.
