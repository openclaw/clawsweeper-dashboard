## Summary

Keep open. Current main stores package static-scan findings and exposes them on the package version JSON path, but the requested owner moderation-status schema/query and human package inspect output still omit per-finding rule/file/line details; the remaining work also needs a maintainer security disclosure decision.

## What This Changes

Expose scanner rule IDs and file:line finding details in package moderation-status and inspect output, and document the server-side rule surface.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, expose approved rule IDs and file:line details to authorized package owners and staff in moderation-status and package inspect, and document what rule identifiers are intentionally public.

## Reproduction Assessment

Yes, from source inspection. Current main stores staticScan findings with rule/file/line data, while the moderation-status schema/query and human package inspect output return only aggregate scan and moderation fields.

## Solution Assessment

Unclear until maintainers set the disclosure policy. Surfacing sanitized rule/file/line details is likely the right direction, but raw staticScan evidence should not be broadened without explicit access-control and redaction rules.
