## Summary

Keep open. Current main already stores and exposes package static-scan findings through package version data and the plugin security page, but `package moderation-status` and default human `package inspect` still do not surface the requested rule/file/line details; this needs maintainer disclosure and docs policy before implementation.

## What This Changes

The issue asks ClawHub to surface package scanner rule/file/line details in moderation-status and inspect output, document server-side rule identifiers, and help triage a specific suspected false positive.

## Best Possible Solution

Define a package scanner disclosure contract, then expose sanitized staticScan reasonCodes/findings consistently in owner/staff moderation-status, package inspect output, and public docs while preserving separate handling for LLM and external scanner verdicts.

## Reproduction Assessment

Yes. Source inspection on current main shows the moderation-status query, schema, and CLI output omit staticScan findings while package version data and the web scanner page already have rule/file/line-capable staticScan data.

## Solution Assessment

No, the current aggregate-only CLI output is not enough for the reported false-positive triage. The maintainable solution is to approve a disclosure/redaction contract and reuse the existing staticScan model across CLI/API/docs rather than adding ad hoc strings to one command.
