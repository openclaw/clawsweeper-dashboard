## Summary

Keep open. Current main still has only partial scanner-detail exposure: static findings are modeled and can appear through version JSON/web scanner pages, but package moderation-status and normal human inspect output do not expose a deliberate sanitized rule/file/line diagnostic contract.

## What This Changes

Expose scanner rule and file:line details in package moderation-status/inspect output and document server-side rule identifiers for false-positive triage.

## Best Possible Solution

Define a sanitized owner/staff-scoped scanner diagnostic contract for rule IDs, file:line, message, and redacted evidence, expose it consistently through moderation-status and inspect, and document intentionally public rule identifiers without dumping evasion-sensitive internals.

## Reproduction Assessment

Yes, by source inspection. Current main stores static finding rule/location data and exposes some of it through version JSON/web pages, while moderation-status and normal human inspect output still lack the requested scanner finding detail.

## Solution Assessment

Unclear until maintainers decide disclosure policy. The maintainable path is an owner/staff-scoped, redacted diagnostic output and documented public rule IDs, not an unfiltered dump of stored scanner evidence or regexes.
