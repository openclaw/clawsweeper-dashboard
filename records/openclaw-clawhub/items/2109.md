## Summary

Keep open. Current main stores and can return static scanner code/file/line details through the package version API, and the cited package now shows the triggering static finding there, but `package moderation-status` and normal CLI output still expose only aggregate scan/reason fields; publishing broader server-side rule details also needs a security/product decision.

## What This Changes

The issue asks ClawHub to expose scanner rule and file/line details in publisher-facing package moderation or inspect output, document safe server-side rule identifiers, and help triage the cited `@lobstah/openclaw-provider` false-positive.

## Best Possible Solution

Expose or document a safe, consistent owner path for existing `staticScan` code/file/line details, and separately define how much rule-catalog detail ClawHub is willing to publish without weakening scanner effectiveness.

## Reproduction Assessment

Yes for the reported limitation: source inspection shows `package moderation-status` omits scanner finding details, while the live version API for the cited package returns the exact static rule and file/line. The broader false-positive claim still needs moderator judgment because aggregate suspicious state also includes LLM scan status.

## Solution Assessment

Yes directionally, but not as full server-side rule disclosure without policy review. The narrow maintainable path is to surface existing `staticScan` code/file/line details in owner-facing CLI/docs and avoid publishing regexes or sensitive evidence unless maintainers explicitly approve that boundary.
