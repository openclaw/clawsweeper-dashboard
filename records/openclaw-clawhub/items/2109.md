## Summary

Keep open. Current main stores structured static-scan findings and the public version API can expose them, but `package moderation-status` and the default human `package inspect` output still surface only aggregate scan state/reasons; expanding scanner detail is a security-sensitive feature/policy decision, not a cleanup close.

## What This Changes

Expose static scanner rule/file/line details in package `moderation-status` and `inspect` output, document rule IDs, and help identify the current false-positive trigger.

## Best Possible Solution

Define a scanner disclosure contract, then expose sanitized staticScan reasonCodes/findings consistently in owner/staff moderation-status and inspect output with public docs for stable rule IDs.

## Reproduction Assessment

Yes. Source inspection shows the moderation-status API/schema and CLI printers omit finding detail, while the live package version endpoint for the named package returns rule, file, and line data.

## Solution Assessment

Yes in direction, but not as an ad hoc CLI string. The maintainable solution is to approve a disclosure/redaction policy and reuse the existing PackageStaticScan model across API, CLI, tests, and docs.
