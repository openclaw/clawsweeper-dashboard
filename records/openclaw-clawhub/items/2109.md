## Summary

Keep open. Current main still leaves the owner-facing scanner diagnostic gap in `package moderation-status` and human `package inspect` output, and choosing which scanner details to disclose is a security/product decision rather than a safe automated cleanup.

## What This Changes

Request to show static scanner finding details such as rule id and file:line in package moderation-status/inspect output and document server-side rule identifiers for false-positive triage.

## Best Possible Solution

Expose a narrow authenticated owner/moderator path for existing staticScan `code`, `file`, `line`, and message details, document the intended fields, and keep broader rule catalogs or raw evidence behind explicit security policy approval.

## Reproduction Assessment

Yes for the owner-facing output limitation: current main's moderation-status type, query, schema, and CLI only return aggregate scan/moderation fields, while the cited live version API has a static finding with code, file, and line. I did not fully reproduce the false-positive judgment because the release also has LLM suspicious state.

## Solution Assessment

Yes directionally for a narrow authenticated disclosure of existing staticScan fields. Publishing broader server-side rule catalogs, regexes, raw evidence, LLM findings, or VirusTotal details needs maintainer security/product approval first.
