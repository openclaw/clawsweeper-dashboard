## Summary

Keep open. Current main stores package static-scan findings and can return them from the package version API, but the requested moderation-status contract and default human inspect output still expose only aggregate scan status and reasons; adding rule/file/line disclosure needs a maintainer security policy first.

## What This Changes

Expose scanner finding details such as rule ID and file/line in package moderation-status and inspect output, and document server-side rule IDs for false-positive triage.

## Best Possible Solution

Define a package scanner disclosure/redaction contract, then expose sanitized staticScan reasonCodes/findings consistently in owner/staff package moderation-status and inspect output with docs for stable rule IDs.

## Reproduction Assessment

Yes. Source inspection on current main shows the moderation-status query, schema, and CLI printer omit finding details, while the package version API and schema already carry staticScan rule/file/line data.

## Solution Assessment

Yes, with policy guardrails. Reusing the existing staticScan finding model is the narrowest maintainable direction, but maintainers should define owner/staff/public disclosure and redaction before expanding CLI/API output.
