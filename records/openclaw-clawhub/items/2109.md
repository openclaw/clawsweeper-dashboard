## Summary

Keep open: this ClawHub issue is not an auto-close candidate, and current main still lacks a deliberate owner/staff-visible scanner-detail contract for `package moderation-status` and human `package inspect` even though package static finding data exists.

## What This Changes

The issue requests scanner rule IDs and file:line locations in package moderation-status and inspect output, plus documentation of server-side rule identifiers for false-positive triage.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, expose approved rule IDs and file:line details to authorized owners and staff in package moderation-status/inspect, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes, by source inspection. Current main stores package static findings with rule/file/line data, while the moderation-status schema/backend and human package inspect formatter still expose only aggregate scan and moderation information.

## Solution Assessment

Unclear until maintainers set disclosure policy. The maintainable direction is sanitized owner/staff-scoped rule and file:line output, not broad raw evidence exposure across public surfaces.
