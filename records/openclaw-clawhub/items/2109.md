Keep open. Current main still does not expose package scanner rule/file/line findings through `package moderation-status` or the human `package inspect` formatter, and the request needs maintainer policy on owner/staff visibility, redaction, and rule documentation before implementation.

## What This Changes

The issue requests owner-visible scanner finding details and server-side rule documentation for package false-positive triage.

## Best Possible Solution

Define a sanitized owner/staff-scoped package scanner diagnostics contract, expose approved rule IDs and file:line details in moderation-status/inspect, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes, by source inspection. Current main stores package static findings with rule/file/line data, while the moderation-status schema/backend and human package inspect formatter expose only aggregate scan and moderation information.

## Solution Assessment

Unclear until maintainers set disclosure policy. The maintainable direction is sanitized owner/staff-scoped rule and file:line output, not broad raw evidence exposure across public surfaces.
