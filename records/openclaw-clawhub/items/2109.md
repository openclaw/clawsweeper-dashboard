## Summary

Keep open. Current main partially exposes static findings through version JSON and web security pages, but `package moderation-status` and normal human `package inspect` output still lack a deliberate sanitized rule/file/line diagnostic contract, and this ClawHub issue needs security/product policy before implementation.

## What This Changes

Request to expose scanner rule ID and file:line details in package moderation-status and inspect output, and to document server-side scanner rule identifiers for false-positive triage.

## Best Possible Solution

Define one sanitized owner/staff-scoped scanner diagnostic contract for rule IDs and file:line details, redact or gate raw evidence, expose it consistently through moderation-status and inspect, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes, by source inspection. Current main stores static finding rule/location data and exposes some detailed data through version JSON/web pages, while moderation-status and normal human inspect output do not expose a sanitized rule/file/line diagnostic contract.

## Solution Assessment

Unclear until maintainers decide disclosure policy. The maintainable path is an owner/staff-scoped, redacted diagnostic output and documented public rule IDs, not a direct dump of stored scanner evidence or regexes.
