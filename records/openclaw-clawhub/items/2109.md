## Summary

Keep open. Current main partially exposes package static-scan findings through version JSON and web security pages, but `package moderation-status` and normal human `package inspect` output still lack a deliberate sanitized rule/file/line diagnostic contract; this also needs maintainer security/product policy before implementation.

## What This Changes

The issue asks ClawHub to expose scanner rule and file:line details in package moderation-status and inspect output so publishers can triage false positives.

## Best Possible Solution

Define one sanitized owner/staff-scoped scanner diagnostic contract for rule IDs and file:line details, redact or gate raw evidence, expose it consistently through moderation-status and inspect, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes, by source inspection. Current main stores and partially exposes static finding data, while `package moderation-status` and normal human `package inspect` output still do not provide a sanitized rule/file/line diagnostic contract.

## Solution Assessment

Unclear until maintainers decide disclosure policy. The maintainable path is an owner/staff-scoped, redacted diagnostic output and documented public rule IDs, not a direct dump of stored scanner evidence or regexes.
