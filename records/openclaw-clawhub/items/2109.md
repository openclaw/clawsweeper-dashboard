## Summary

Keep open. The requested publisher-facing scanner detail is still not fully available in `package moderation-status` or the default human `package inspect` path, and exposing rule/file/line details needs a maintainer security policy decision rather than cleanup closure.

## What This Changes

Request to surface scanner rule and file:line details in package moderation-status/inspect output, document server-side scanner rule IDs, and help triage a specific suspicious plugin release.

## Best Possible Solution

Define an owner/staff-safe package scanner finding contract, reuse the existing staticScan finding model where appropriate, add redacted rule/file/line output to moderation-status and inspect, and document which rule IDs are stable enough for publishers.

## Reproduction Assessment

Yes for the CLI/API gap: source inspection shows `package moderation-status` and its schema expose aggregate scan state and reasons, not finding detail. I did not prove the live `@lobstah/openclaw-provider` false positive itself because that depends on production scanner evidence.

## Solution Assessment

Unclear as a direct implementation until maintainers decide disclosure and redaction policy. The narrow maintainable direction is to expose owner-scoped, redacted finding details from the existing `staticScan.findings` model rather than publishing raw scanner internals globally.
