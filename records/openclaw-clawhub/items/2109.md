## Summary

Keep open. Current main already stores static scanner detail and exposes it through package version data and the web security page, but the owner moderation-status endpoint and default human package inspect path still omit scanner findings; deciding how much rule/file/evidence detail to disclose is a security/product policy call.

## What This Changes

Expose static scanner rule IDs and file:line findings in package moderation-status/package inspect output and document server-side rule IDs for publisher false-positive triage.

## Best Possible Solution

Define a scanner disclosure/redaction policy, then expose sanitized staticScan reasonCodes/findings consistently to authorized owners/staff in moderation-status and inspect, with docs for stable rule IDs and separate LLM verdict handling.

## Reproduction Assessment

Yes. Source inspection shows the moderation-status query/schema and default CLI human output omit finding detail, while the live package version endpoint confirms rule/file/line data exists for the named package.

## Solution Assessment

Unclear until maintainers define disclosure boundaries. Once policy is set, reusing the existing staticScan reasonCodes/findings model across API, CLI, and docs is the narrowest maintainable path.
