## Summary

Keep open. Current main already stores and exposes static scanner finding details through the package version API and scanner UI, and the live cited package still has an actionable static finding, but `package moderation-status` and normal `package inspect` output still omit rule/file/line details; broader server-side rule disclosure needs maintainer security/product judgment.

## What This Changes

Request to expose static scanner rule and file:line details in package moderation-status/inspect output and document server-side scanner rules for false-positive triage.

## Best Possible Solution

Define an owner-safe scanner disclosure contract, then expose existing `staticScan` code/file/line/message details through owner-facing moderation-status/package inspect and docs while keeping unpublished rule logic behind a policy-approved boundary.

## Reproduction Assessment

Yes for the reported limitation: source inspection shows the moderation-status backend, schema, and CLI omit scanner finding details while the package version API and scanner UI can already carry them. This is not a failing documented contract so much as a source-reproducible missing owner-facing disclosure path.

## Solution Assessment

Yes directionally, but not as unrestricted rule disclosure. The narrow maintainable path is to expose the existing `staticScan` finding fields in owner-facing CLI/API/docs after maintainers approve the redaction and evasion boundary.
