## Summary

Keep open. Current main deliberately retains detailed static scanner findings as internal ClawScan evidence, while the authenticated package moderation-status contract still exposes only aggregate status and reason codes; recent public Plugin Inspector findings address a different validator surface and do not solve static-scanner false-positive triage.

## What This Changes

Expose actionable scanner rule identifiers and file/line locations to package owners through moderation-status or inspect output, and clarify which server-side scanner signals may be documented.

## Best Possible Solution

Define a durable owner-authenticated finding contract in `specs/security-moderation.md` that exposes actionable, redacted rule and location data while retaining sensitive heuristics/evidence internally, then use that same contract consistently in the package API, CLI inspect/moderation-status output, docs, and runtime tests.

## Reproduction Assessment

Yes, at source level. Current main stores detailed static findings but the authenticated moderation-status response and handler omit them by design; reproducing the reporter’s exact package match is unnecessary to establish that output gap.

## Solution Assessment

No, directly returning every raw static finding is not yet the best solution because it conflicts with the current internal-evidence policy. The narrow maintainable path is a curated owner-only diagnostic contract with explicit redaction and documentation rules.
