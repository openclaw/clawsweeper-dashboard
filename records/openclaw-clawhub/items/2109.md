## Summary

Keep open. Current main stores package static-scan findings and the public version endpoint can serve them, but `package moderation-status` and human `package inspect` still expose only aggregate scan state/reasons; deciding where to disclose rule/file/line/evidence needs maintainer security/product judgment.

## What This Changes

The issue requests scanner finding details such as rule ID and file:line in package moderation-status and inspect output to help plugin publishers triage false positives.

## Best Possible Solution

Define a scanner disclosure and redaction contract, then expose sanitized staticScan reasonCodes and findings consistently for authorized package owners and staff in moderation-status, package inspect, and public/operator docs.

## Reproduction Assessment

Yes. Source inspection shows the moderation-status schema/query and human package inspect output omit finding details, while the live version endpoint for the named package returns the rule and file:line detail.

## Solution Assessment

Unclear. Surfacing sanitized details is likely the right direction, but the maintainable fix needs maintainer approval for owner/staff/public access, redaction, and whether evidence snippets or only rule/file/line should be exposed.
