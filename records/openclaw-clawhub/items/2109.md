## Summary

Current main still omits structured static-scanner evidence from publisher moderation output, so the central false-positive triage request remains necessary; implementation should wait for an explicit security decision on which stored fields may be disclosed.

## What This Changes

Expose scanner rule identifiers and file/line locations in publisher-facing moderation-status or inspect output, and document the server-side scanner rule contract.

## Best Possible Solution

Add an authenticated package-owner projection containing stable static finding codes plus package-relative file and line locations, keep sensitive evidence and detector mechanics private, clearly distinguish these results from Plugin Inspector findings, and document the invariant in `specs/` with the public CLI/API behavior in `docs/`.

## Reproduction Assessment

Yes. Current main stores per-finding rule/file/line data, while the authenticated moderation response and CLI expose only aggregate statuses and reasons; the reporter's exact live package verdict does not need to remain unchanged to reproduce that output gap.

## Solution Assessment

Yes in outcome, but not as unrestricted disclosure: stable owner-only rule codes and locations are the narrowest maintainable solution, while raw evidence, messages, and detector definitions should remain redacted unless maintainers explicitly approve them.
