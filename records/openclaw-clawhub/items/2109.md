## Summary

Keep open. Current main still only partially exposes scanner detail: stored package static findings can include rule and location data, but package `moderation-status` and normal human `package inspect` output remain aggregate-only, and exposing more scanner detail needs an explicit security disclosure policy.

## What This Changes

Request to show scanner finding details such as rule ID and file:line in package moderation-status/inspect output and document server-side rule identifiers for false-positive triage.

## Best Possible Solution

Define a sanitized owner/staff-scoped scanner diagnostic contract for rule IDs, file:line, message, and redacted evidence, expose it consistently through moderation-status and inspect, and document intentionally public rule IDs without publishing evasion-sensitive internals.

## Reproduction Assessment

Yes, by source inspection. Current main stores static finding rule/location data and exposes it in some version/detail paths, while package moderation-status and normal human package inspect output still omit the requested scanner finding detail.

## Solution Assessment

Unclear until maintainers decide the disclosure policy. The maintainable path is a redacted owner/staff diagnostic contract and documented public rule identifiers, not an unfiltered dump of stored scanner evidence or regexes.
