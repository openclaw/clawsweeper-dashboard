## Summary

Keep open. Current main still has structured scanner finding data and a package-version JSON path that can carry it, but owner-facing `package moderation-status`, human `package inspect`, and the moderation-status docs still expose only aggregate scan/moderation state; deciding how much scanner detail to disclose needs maintainer security policy.

## What This Changes

Expose scanner finding details such as rule ID and file:line in owner-facing package moderation-status/inspect output and document server-side rule IDs for false-positive triage.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, expose approved rule IDs and file:line details to authorized owners and staff in moderation-status/package inspect, and document intentionally public rule identifiers.

## Reproduction Assessment

Yes. Source inspection on current main confirms stored package findings include code/file/line/evidence and version JSON can include `staticScan`, while moderation-status schema/backend and human CLI output still omit finding detail.

## Solution Assessment

Unclear until maintainers set disclosure policy. The maintainable direction is sanitized owner/staff-scoped rule and file:line output, not broad raw evidence exposure across public surfaces.
