## Summary

Keep open. Current main stores package static-scan findings and now treats static-suspicious findings as advisory, but the requested owner-facing `package moderation-status` schema/query and human `package inspect` output still do not surface rule/file/line details; the remaining change needs maintainer security disclosure policy.

## What This Changes

Surface scanner finding details such as rule ID and file:line in package moderation-status and inspect output, and document intentionally public server-side rule identifiers.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, expose approved rule IDs and file:line details to authorized package owners and staff in moderation-status and package inspect, and document which identifiers are intentionally public.

## Reproduction Assessment

Yes, from source inspection. Current main persists staticScan findings with rule and location data, while the moderation-status schema/query and human package inspect output still return only aggregate scan and moderation fields.

## Solution Assessment

Unclear until maintainers decide the disclosure policy. The requested direction is likely useful, but the maintainable implementation should expose sanitized owner/staff diagnostics rather than raw stored evidence by default.
