## Summary

Keep open. Current main has structured package static-scan findings, but the owner-facing moderation-status path and default human inspect output still expose only aggregate scan/moderation state; the requested disclosure also needs maintainer policy because scanner finding detail is security-sensitive.

## What This Changes

Request to expose scanner rule/file/line details in package moderation-status and inspect output, and to document server-side scanner rule IDs for false-positive triage.

## Best Possible Solution

Define a scanner disclosure and redaction contract, then expose sanitized staticScan reasonCodes/findings to authorized package owners and staff in moderation-status and inspect output, with docs for stable rule IDs separate from raw regexes.

## Reproduction Assessment

Yes. Source inspection shows the moderation-status schema/query/CLI omit staticScan/findings fields, while the package version endpoint/schema already includes staticScan findings with rule/file/line data.

## Solution Assessment

No, directly adding raw finding detail to one CLI printer is not the best first step. The maintainable path is to approve disclosure/redaction rules and then reuse the existing staticScan model consistently across API, CLI, and docs.
