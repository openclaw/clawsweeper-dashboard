## Summary

Keeping this open. Current main still has scanner findings stored and exposed on the version/security API surface, but `package moderation-status` and human `package inspect` do not show rule/file/line detail; because this is an issue in `openclaw/clawhub` and the disclosure boundary is security-sensitive, it needs maintainer follow-up rather than cleanup closure.

## What This Changes

Request to expose scanner finding rule and file:line details in CLI moderation/inspect output and document any server-side scanner rule identifiers for false-positive triage.

## Best Possible Solution

Define a scanner disclosure contract, then expose sanitized staticScan reasonCodes/findings for authorized package owners and staff in `package moderation-status`, human `package inspect`, and operator docs.

## Reproduction Assessment

Yes. Source inspection on current main shows moderation-status and human inspect omit rule/file/line findings while the stored staticScan object already carries that detail elsewhere.

## Solution Assessment

Unclear. Surfacing sanitized rule/file/line detail is likely the right direction, but maintainers should first decide which details are visible to owners, staff, public API consumers, and docs.
