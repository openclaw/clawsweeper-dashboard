## Summary

Keep open. Current main has partial scanner-detail exposure through static finding models, web scanner pages, and a new dataset-export sidecar, but package `moderation-status` and normal package inspect output still do not expose a deliberate owner/staff-scoped rule/file/line diagnostic contract.

## What This Changes

Request to expose scanner rule and file:line detail in package moderation-status and inspect output, document public server-side rule IDs, and help triage a specific flagged package.

## Best Possible Solution

Define a sanitized owner/staff-scoped scanner diagnostic contract for rule IDs, file:line, message, and redacted evidence, expose it consistently through package moderation-status and inspect, and document intentionally public rule identifiers without publishing evasion-sensitive internals.

## Reproduction Assessment

Yes, by source inspection. Current main stores and renders static finding rule/location data in some paths, while package moderation-status and normal package inspect output still lack the requested scanner finding detail.

## Solution Assessment

Unclear until maintainers decide disclosure policy. The maintainable path is an owner/staff-scoped, redacted diagnostic output and documented public rule IDs, not an unfiltered dump of stored scanner evidence or regexes.
