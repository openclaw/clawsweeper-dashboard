## Summary

Keep open. Current main stores and can expose package static-scan findings through the version API, but `package moderation-status` and human `package inspect` still omit rule/file/line detail, and the remaining disclosure contract needs maintainer security/product judgment.

## What This Changes

Request to expose scanner rule, file, and line details in package moderation-status / inspect output and document server-side rule identifiers for publisher triage.

## Best Possible Solution

Define a sanitized scanner disclosure contract for package owners and staff, then expose at least code/file/line consistently in `package moderation-status`, human `package inspect`, docs, and tests with explicit redaction rules for evidence snippets.

## Reproduction Assessment

Yes. Source inspection shows the requested CLI/API surfaces omit finding detail, while the live package version endpoint for the reported package returns the exact rule and file:line finding.

## Solution Assessment

Unclear until maintainers define the disclosure policy. Adding sanitized code/file/line for owners and staff is likely the narrow direction, but exposing evidence snippets or server-side rules needs explicit redaction and audience boundaries first.
