## Summary

Keep open. Current main stores and serves package static-scan finding detail on the package version endpoint, but `package moderation-status` and the human `package inspect` path still expose only aggregate scan state/reasons; adding rule/file/line output needs a maintainer security/product disclosure decision.

## What This Changes

Request to expose static scanner rule/file/line details in `package moderation-status` and human `package inspect`, and document server-side rule IDs for false-positive triage.

## Best Possible Solution

Define a scanner finding disclosure/redaction contract, then expose sanitized `staticScan` reason codes and rule/file/line findings consistently for authorized package owners and staff in moderation-status, package inspect, and docs.

## Reproduction Assessment

Yes. Source inspection shows the moderation-status schema/query and human package inspect path omit finding details, while the live package version endpoint for the named package returns the exact rule and file:line detail.

## Solution Assessment

Unclear. Surfacing sanitized detail is likely the right direction, but the maintainable implementation needs maintainer approval for owner/staff/public access, redaction, and whether evidence snippets or only rule/file/line should be exposed.
