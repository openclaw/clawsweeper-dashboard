## Summary

Keep open. Current main still does not expose scanner rule/file/line detail through the requested `package moderation-status` response or human `package inspect` output, although the data exists and the reporter's exact finding is visible through the version API.

## What This Changes

Request to expose static scanner finding details, document server-side rule IDs, and help triage a specific suspicious package verdict.

## Best Possible Solution

Define a sanitized scanner-disclosure contract, then expose approved rule IDs and file:line findings for authorized package owners and staff across `moderation-status`, `package inspect`, and public docs.

## Reproduction Assessment

Yes. Source inspection on current main shows static scan findings carry rule/file/line/evidence, while the requested moderation-status schema/query and human CLI output return only aggregate scan and reason fields.

## Solution Assessment

Unclear. Surfacing sanitized rule/file/line detail is likely the right direction, but dumping raw `staticScan` everywhere is not the safest implementation without a maintainer-approved visibility and redaction policy.
