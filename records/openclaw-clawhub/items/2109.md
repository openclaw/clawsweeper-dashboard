## Summary

Keep open. Current main stores package static-scan findings and exposes them on the package version endpoint, but `package moderation-status` and the default human `package inspect` path still surface only aggregate status/reasons; scanner finding disclosure needs an explicit security/product policy before implementation.

## What This Changes

Expose scanner finding details such as rule ID and file:line in package moderation-status and inspect output, and document server-side scanner rule IDs for publisher false-positive triage.

## Best Possible Solution

Define a scanner disclosure/redaction policy, then expose sanitized staticScan reasonCodes/findings consistently in owner/staff moderation-status and package inspect output, with docs for stable rule IDs and separate handling for LLM verdicts.

## Reproduction Assessment

Yes. Source inspection shows the moderation-status query, schema, and CLI printer omit finding details, while the live package version endpoint confirms rule/file/line data exists for the named package.

## Solution Assessment

No, the current aggregate-only output is only a partial workaround. The maintainable path is to approve a disclosure contract and reuse the existing staticScan finding model across API, CLI, and docs rather than adding ad hoc strings to one command.
