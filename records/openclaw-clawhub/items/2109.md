## Summary

Current main still leaves published package owners with aggregate scan status and reason codes rather than the verdict-producing rule and source-location details requested here. The newer `clawhub scan` reporting work is for skill scan requests and does not replace the package moderation path, while the repository’s security specification deliberately treats static findings as internal ClawScan context; this therefore remains a valid, product-sensitive feature requiring maintainer approval rather than a close candidate.

## What This Changes

Expose scanner rule identifiers and file/line locations to authenticated package owners in moderation-status or inspect output, and document the stable server-side rule identifiers.

## Best Possible Solution

Add an authenticated, owner-or-publisher-member package-version diagnostic response containing capped stable rule IDs and normalized file/line locations for the findings that produced the package verdict, keep matcher internals and public trust responses private, rate-limit retrieval, and document both the disclosure contract and stable rule identifiers.

## Reproduction Assessment

Yes, at high confidence from current source and the reporter’s current package output: the package owner path exposes aggregate scan state and reasons while structured findings remain outside that contract. A live private-owner request was not necessary to establish the missing response fields.

## Solution Assessment

Yes in direction, but not as unrestricted disclosure: an authenticated owner-only, capped projection on the package-version moderation path is narrower and safer than exposing findings publicly or treating the skill scan command as a substitute.
