## Summary

Keep open. Current main still leaves the owner-facing package moderation-status path and human package inspect output aggregate-only, and this recent security-transparency request needs a maintainer disclosure/redaction decision rather than cleanup closure.

## What This Changes

Expose scanner rule IDs and file:line finding detail in package moderation-status and inspect output, document server-side rule IDs, and help triage a reported plugin false positive.

## Best Possible Solution

Define an owner/staff/public scanner disclosure contract, then expose sanitized staticScan reason codes and rule/file/line findings in package moderation-status and inspect, with docs for the approved rule IDs.

## Reproduction Assessment

Yes. Source inspection on current main shows package staticScan findings store rule and location data, while moderation-status and human inspect omit that detail and return only aggregate scan/moderation fields.

## Solution Assessment

Unclear. Surfacing sanitized rule/file/line detail for authorized publishers is likely the right direction, but maintainers first need to decide who can see rule IDs, paths, line numbers, evidence snippets, and server-side rule documentation.
