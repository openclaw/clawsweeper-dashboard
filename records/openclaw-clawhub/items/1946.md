## Summary

Current main still only accepts changelog text while publishing a new version; it has no supported command or API for correcting an existing version’s changelog, and no related item clearly owns that post-publication capability. The request remains useful, but published-version mutability, authorization, audit history, and moderation behavior need an explicit maintainer decision before implementation.

## What This Changes

Add a CLI workflow for authorized publishers to edit the changelog of an already-published version.

## Best Possible Solution

Define a narrow metadata-correction contract, then add one authenticated API and CLI command that atomically updates the version and any latest-version mirror, records the actor and prior value, and applies explicit rules for moderated and historical releases.

## Reproduction Assessment

Not applicable. Source and CLI inspection establish that post-publication changelog editing is an absent capability, not a reproducible failure of an existing documented workflow.

## Solution Assessment

Unclear. A dedicated CLI command is a reasonable interface, but direct mutation is not yet the best proven design until maintainers define immutability, audit, authorization, and moderation semantics.
