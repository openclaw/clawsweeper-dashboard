## Summary

Current `main` still has no supported way to change a changelog after a version is published. The request is understandable, but it changes trusted package-history semantics and needs an explicit maintainer decision on immutability, authorization, and audit history before implementation.

## What This Changes

Add a command that lets publishers correct or update the changelog for an already-published version.

## Best Possible Solution

Define an audit-preserving correction model for published changelogs—recording editor, timestamp, prior text, and authorization—then expose the same approved behavior through the backend and CLI, with UI support only if maintainers want parity.

## Reproduction Assessment

Not applicable. Source inspection gives high confidence that post-publication changelog editing is absent, but the issue requests a new capability rather than reporting broken defined behavior.

## Solution Assessment

No, not as a CLI command alone. The maintainable solution must first establish backend authorization and audit/immutability semantics, then make the CLI a client of that policy.
