## Summary

Current main does not provide post-publish changelog editing: changelog text enters through publish/sync flows and is returned with version metadata, while owner-facing mutations do not expose a changelog update. The linked open report covers changelog generation and draft-overwrite defects, not this distinct capability, so the request remains necessary but needs an explicit product and trust-policy decision before implementation.

## What This Changes

Add an owner-authorized command and backend operation to edit the changelog of a specific already-published skill or package version.

## Best Possible Solution

Define one shared, auditable owner-authorized operation for updating only changelog text on an exact skill or package version, preserve artifact/provenance/scan fields unchanged, and expose that contract through the CLI after maintainers approve historical-metadata mutability.

## Reproduction Assessment

Not applicable; source inspection confirms this is an absent post-publish capability, not a failure of an existing documented edit workflow.

## Solution Assessment

No, the proposed CLI syntax alone is not the complete solution; the narrow maintainable path starts with a shared backend authorization and audit contract covering exact versions and then adds client commands.
