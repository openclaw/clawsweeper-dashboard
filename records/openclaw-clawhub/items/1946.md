## Summary

Current main has no post-publication changelog edit path, so this remains a real feature request. It should stay open for a maintainer product decision because changing historical release metadata affects provenance and needs an explicit authorization and audit policy.

## What This Changes

Add a CLI workflow for authorized publishers to correct the changelog of an already-published version.

## Best Possible Solution

Define historical changelogs as auditable metadata, then add an owner-authorized API and CLI correction flow that cannot alter package contents or version identity and preserves edit attribution and prior text.

## Reproduction Assessment

Not applicable. Source inspection confirms the current product accepts changelogs at publish time and exposes them afterward, while this issue requests a new mutation workflow.

## Solution Assessment

No, a CLI command by itself is not the best complete solution. The backend authorization boundary, immutable artifact guarantees, and audit/history semantics should be decided and implemented with the CLI surface.
