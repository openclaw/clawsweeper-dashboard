## Summary

Keep open: current main can set and display version changelogs, but it does not provide a CLI or authenticated API path to edit an existing version changelog after publish.

## What This Changes

The issue requests a CLI command for editing an existing version changelog, such as `clawhub changelog edit <version>`.

## Best Possible Solution

Add a resource-scoped, authenticated changelog edit flow that updates the stored version and all denormalized latest-version summaries consistently.

Not applicable as a runtime bug; this is a feature request. The current-main verification path is source/API inspection showing publish-time changelog writes and read-only version endpoints, with no edit command or update route.

## Solution Assessment

Yes, a dedicated command is a reasonable direction if it is backed by a server-side mutation; a safer shape is an explicit skill-scoped command rather than a version-only global command.

## Repair Work Prompt

Implement a narrow skill-version changelog edit flow for ClawHub. Add an authenticated backend mutation/API route that lets a skill owner, publisher member with suitable content permissions, moderator, or admin update the changelog for an existing non-deleted skill version. Keep the first implementation scoped to skills, not packages or souls, unless existing patterns make those trivial without broadening the PR. Prefer an explicit CLI shape such as `clawhub skill changelog edit <slug> <version> --changelog <text>` or an equivalent resource-scoped command; do not implement the ambiguous version-only command. The mutation must update `skillVersions.changelog` and `changelogSource`, and when the edited version is the skill's latest version it must also update `skills.latestVersionSummary` and the `skillSearchDigest` row. Add audit logging for the metadata edit. Update CLI schemas/types, command registration, focused CLI tests, Convex/API tests, docs, OpenAPI if the public API changes, generated Convex API/types if needed, and `CHANGELOG.md` because this is user-facing. Do not change file contents, version artifacts, tags, moderation status, or publish semantics.
