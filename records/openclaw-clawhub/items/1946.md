## Summary

Keep open: current main treats changelogs as publish-time metadata and read-only version history, so editing published release notes is a new mutable release-metadata feature that needs maintainer policy decisions.

## What This Changes

Add a CLI workflow for editing a specific published version changelog after publication.

## Best Possible Solution

Define an authenticated owner/staff mutation for approved versioned surfaces, update version rows and denormalized summaries consistently, audit edits, then expose it through CLI/UI/docs/tests.

## Reproduction Assessment

Not applicable. This is a feature request, and source inspection shows publish-time changelog input plus read-only changelog output rather than a broken current contract.

## Solution Assessment

No. A CLI command is a reasonable eventual surface, but the backend mutation, authorization, audit trail, supported object scope, and denormalized summary behavior should be designed first.
