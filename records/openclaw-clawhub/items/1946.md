## Summary

Keep open: current main can store and display version changelogs, but the CLI and HTTP API only write changelog text during publish/sync and do not expose a path to edit an already-published version changelog.

## What This Changes

Request a CLI workflow for editing an existing version changelog after publication.

## Best Possible Solution

Add an authenticated, resource-scoped changelog edit flow that updates the stored version and all latest-version summaries consistently.

## Solution Assessment

Yes, a dedicated command is a reasonable direction if it is backed by a server-side mutation; the maintainable shape is an explicit skill/package-scoped command rather than a bare version-only global command.

## Repair Work Prompt

Implement an authenticated, resource-scoped changelog edit flow for existing ClawHub skill versions. Add a CLI command under the existing skill command surface, such as `clawhub skill changelog edit <slug> <version> --message <text>` or the closest style consistent with the current CLI; do not add a bare global `clawhub changelog edit <version>` without a skill/package identifier. Add the matching schema/API route and Convex mutation. The server must verify the caller is the skill owner/publisher or staff, update `skillVersions.changelog` and `changelogSource`, and when the edited version is the current latest, update `skills.latestVersionSummary` and the `skillSearchDigest` mirror. Add focused tests for the CLI command, HTTP handler, authorization failure, and latest-version denormalization. Update public CLI/API docs and `CHANGELOG.md`. Avoid unrelated package registry, moderation, or publish-flow refactors.
