## Summary

Keep open: current main still lacks a CLI or authenticated API path to edit an existing skill version changelog after publish; changelogs are written during publish/sync and then exposed through read-only version endpoints.

## What This Changes

Request a CLI command to edit an already-published version changelog.

## Best Possible Solution

Add an authenticated, skill-scoped changelog edit flow that updates the version record and any latest-version denormalized summaries atomically with audit logging.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The high-confidence verification path is source inspection showing publish-time changelog writes, read-only version endpoints, and no edit command or route.

## Solution Assessment

Mostly yes: a dedicated CLI command is a maintainable user-facing shape if it is backed by a server-side mutation. The safer command/API design should require both slug and version rather than editing by version alone.

## Repair Work Prompt

Implement a skill version changelog edit flow. Add an authenticated API endpoint, for example a POST or PATCH under `/api/v1/skills/:slug/versions/:version/changelog`, that requires the caller to own the skill or hold the appropriate publisher role, validates the new changelog, updates `skillVersions.changelog` and `changelogSource`, and writes an audit log entry. If the edited version is the current latest version, update `skills.latestVersionSummary` and sync `skillSearchDigest` through the existing digest helper. Add a CLI command with explicit resource scope, such as `clawhub skill changelog edit <slug> <version> --changelog <text>` or an equivalent editor/stdin-friendly shape; do not implement a global version-only command. Update shared schemas/routes as needed, add focused backend and CLI tests, and add an Unreleased changelog entry for the user-facing CLI/API feature.
