## Summary

Keep open: current main still only captures changelogs during publish/sync and serves them as read-only version metadata, so the requested post-publish edit flow is a new release-metadata feature. The prior Codex review reached the same keep-open conclusion, and the adjacent open package changelog item at https://github.com/openclaw/clawhub/issues/1288 covers generation/race behavior rather than editing already-published version notes.

## What This Changes

Add a command such as `clawhub changelog edit <version>` to modify a published version changelog after release.

## Best Possible Solution

Define an authenticated owner/staff changelog edit flow with audit history and latest-summary synchronization, then expose it through CLI/API/UI/docs/tests once the product boundary is agreed.

## Reproduction Assessment

Not applicable. This is a feature request, and current source inspection shows publish-time changelog input plus read-only changelog output rather than a broken existing edit contract.

## Solution Assessment

No, not as stated. A CLI command may be useful, but the maintainable solution needs backend mutation scope, authorization, audit behavior, and denormalized latest-summary rules designed first.
