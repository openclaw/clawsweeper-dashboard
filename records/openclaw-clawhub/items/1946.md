## Summary

Keep open: current main supports changelog text only during publish/sync and exposes saved changelogs through read-only API/UI paths. Editing an already-published version changelog would be a new mutable release-metadata feature with policy decisions around scope, authorization, audit, and denormalized summaries.

## What This Changes

Request a CLI command, such as `clawhub changelog edit <version>`, for editing changelog text on already-published versions.

## Best Possible Solution

Define an authenticated, audited backend mutation for the supported versioned surfaces, keep version rows and latest summaries consistent, then expose the approved behavior through CLI/UI/docs/tests.

## Reproduction Assessment

Not applicable. This is a feature request, and source/docs inspection shows publish-time changelog input plus read-only version output rather than a broken existing contract.

## Solution Assessment

No. A CLI command is a plausible eventual surface, but the backend mutation, authorization, audit trail, supported object scope, and denormalized summary behavior should be designed first.
