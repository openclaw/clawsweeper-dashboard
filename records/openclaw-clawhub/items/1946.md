## Summary

Keep open: current main still has publish-time changelog input and read-only version changelog surfaces, but no post-publish edit command, API route, or owner UI flow. This is a new mutable-release-metadata feature that needs maintainer product/API judgment before implementation.

## What This Changes

The issue requests a dedicated CLI workflow for editing an existing version changelog after publish.

## Best Possible Solution

Define an authenticated owner/staff changelog update API that patches version rows and any affected latest summaries consistently, then add the agreed CLI command, docs, and focused regression coverage.

## Reproduction Assessment

Not applicable. This is a new feature request, and source inspection shows publish/read changelog paths but no documented existing edit workflow that is failing.

## Solution Assessment

No. A CLI command alone is not the best complete solution because the backend mutation, authorization, audit expectations, and denormalized latest-summary semantics need to be designed together.
