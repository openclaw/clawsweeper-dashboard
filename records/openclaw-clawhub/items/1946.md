## Summary

Keep open: current main only writes changelog text during publish/sync and exposes persisted changelogs through read-only API/UI paths. Editing an already-published version changelog would be a new mutable release-metadata feature that needs maintainer policy decisions.

## What This Changes

Add a ClawHub command, such as `clawhub changelog edit <version>`, for editing persisted version changelogs after publication.

## Best Possible Solution

Define authenticated owner/staff backend mutations for the supported versioned surfaces, update version rows plus any latest summaries and audit records consistently, then expose the approved behavior through CLI/UI/docs/tests.

## Reproduction Assessment

Not applicable. This is a feature request, and source inspection shows publish-time changelog input plus read-only changelog output rather than a broken current contract.

## Solution Assessment

No. A CLI command is a reasonable eventual surface, but the backend mutation, authorization, audit trail, supported object scope, and denormalized summary behavior should be designed first.
