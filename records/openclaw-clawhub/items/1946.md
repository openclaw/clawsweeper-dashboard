## Summary

Current main still supports changelog text only when publishing a new version; it has no supported command or API for correcting an existing version’s changelog. The request remains useful, but post-publication metadata mutability needs an explicit authorization, audit, and provenance decision before implementation.

## What This Changes

Add a CLI workflow for editing the changelog text of an already-published skill version.

## Best Possible Solution

Introduce an owner-authorized metadata-correction operation that records the actor, timestamp, previous and new text, and reason while keeping the version identifier and artifact contents immutable, then expose that operation consistently through the API and CLI.

## Reproduction Assessment

Not applicable: this is a request for a new capability rather than broken documented behavior. Current-main source inspection provides high confidence that no existing edit command or equivalent owner-facing mutation exists.

## Solution Assessment

No, the proposed command syntax alone is not the best complete solution. The repository first needs a deliberate policy for authorization, retained history, moderation visibility, and downstream metadata updates; the CLI should then wrap that guarded backend contract.
