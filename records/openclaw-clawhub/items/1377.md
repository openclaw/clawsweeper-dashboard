## Summary

Keep open. Current main and the live API still expose MIT-0 for the reported skill while the upstream package source declares MIT, and resolving that mismatch requires a maintainer/license-policy decision rather than cleanup.

## What This Changes

The issue asks ClawHub to display the reported skill's actual MIT license instead of the current MIT-0/no-attribution platform license.

## Best Possible Solution

Maintainers should choose one license model and implement it consistently: either hard-enforce MIT-0 with clear conflict handling for uploaded packages, or add per-skill SPDX support across schema, storage, API, UI, CLI, docs, and migration/backfill paths.

## Reproduction Assessment

Yes. Current main hardcodes, stores, and displays MIT-0 for published skills, and the live API returns MIT-0 for the reported skill while the upstream package source declares MIT.

## Solution Assessment

No. The requested label-only MIT display is not the best standalone fix because it would create conflicting sources of truth; the maintainable path is a license-model decision followed by a full-stack implementation.
