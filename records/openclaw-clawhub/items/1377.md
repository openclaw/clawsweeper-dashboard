## Summary

Current main still reproduces the mismatch, and the field is intentionally modeled as a platform-wide MIT-0 license rather than derived from package metadata. This needs a maintainer license-policy decision, not cleanup closure.

## What This Changes

Correct ClawHub’s license presentation so a skill declared as standard MIT is not shown to users as MIT-0.

## Best Possible Solution

Define and document separate author-declared and platform-publication license semantics, then update publish consent, storage/schema, API/CLI output, and skill-page copy together so attribution obligations are unambiguous.

## Reproduction Assessment

Yes. Current main hard-codes MIT-0 as the only platform skill license and propagates that value to publish and API/CLI surfaces, reproducing the reported presentation from source.

## Solution Assessment

No, changing only the visible label is not the best solution. Maintainers should first decide whether ClawHub displays the package’s declared license, a separate platform publication license, or both, then update the full contract consistently.
