## Summary

Keep open. Current main, the latest release, and production still present a fixed MIT-0 platform license for skills, while the reporter’s upstream package evidence shows MIT; the remaining work is a maintainer/legal product decision about platform-license versus upstream-source-license metadata, not an implemented fix.

## What This Changes

The issue requests that ClawHub stop showing the `openclaw-workspace-sync` listing license as MIT-0 when the upstream package declares MIT.

## Best Possible Solution

Make the platform license scope explicit in UI/API/docs, then add a separately named upstream/source license only if maintainers define its provenance, conflict behavior, migration plan, and legal/product wording.

## Reproduction Assessment

Yes for the observable output: current source renders the fixed `MIT-0` platform constant in the skill header, and the public API for the reported skill still returns `latestVersion.license: "MIT-0"`. It is not a strict bug reproduction because current docs and schemas intentionally define that platform-wide value.

## Solution Assessment

No, directly replacing MIT-0 with MIT is not the best solution because it would conflate the platform distribution grant with the author-declared upstream package license. The safer path is an explicit platform-license label plus a separate upstream-license model if maintainers want to support one.
