## Summary

Keep open. Current main and the live ClawHub API still expose this skill as MIT-0 while the upstream package sources report MIT, but the codebase also documents MIT-0 as an intentional platform-wide publishing policy, so the remaining work needs maintainer/legal product direction rather than cleanup closure.

## What This Changes

The issue asks ClawHub to stop displaying the `openclaw-workspace-sync` listing as MIT-0/no-attribution when the upstream package metadata and license file identify standard MIT.

## Best Possible Solution

Maintainers should choose one licensing model and implement it consistently: either reaffirm MIT-0 as the ClawHub redistribution license with stronger validation/copy for conflicting upstream metadata, or add per-skill SPDX support across schema, storage, public APIs, UI, CLI, docs, migrations, and tests.

## Reproduction Assessment

Yes, at source and hosted-service level: the live ClawHub API returns MIT-0 for this skill while the upstream package/GitHub/npm sources report MIT, and current main always writes and displays the platform MIT-0 constant. I did not run a local app instance, so this is marked source-reproducible rather than locally reproduced.

## Solution Assessment

No, changing the listing label alone is not the best fix because it would create a second license source of truth against the current MIT-0 publishing contract. The maintainable solution needs a product/legal policy choice first, then a coordinated implementation across all license surfaces.
