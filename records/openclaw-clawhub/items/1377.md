## Summary

Keep open. Current main intentionally uses MIT-0 as ClawHub’s mandatory platform license rather than deriving the displayed value from an upstream package, so the reporter’s requested MIT presentation remains unimplemented and requires a product/legal decision about supporting distinct upstream-license metadata. Earlier proposals at https://github.com/openclaw/clawhub/pull/594 and https://github.com/openclaw/clawhub/pull/603 closed without merge and do not supersede this issue.

## What This Changes

Display a skill’s actual upstream MIT license and attribution requirement instead of presenting ClawHub’s MIT-0 platform license as though it were package-derived metadata.

## Best Possible Solution

Keep the mandatory ClawHub platform license clearly labeled, and—if maintainers approve—add a separate optional upstream-source license field with defined precedence, validation, migration, API, UI, CLI, and documentation semantics.

## Reproduction Assessment

Yes. Current main hard-codes MIT-0 as the platform license, public documentation rejects per-skill overrides, and the live API returns MIT-0 for the reported skill.

## Solution Assessment

No, directly replacing the displayed value with package.json metadata is not the safest solution because the existing field represents ClawHub’s publishing contract. A separate, explicitly labeled upstream-license field is the narrower maintainable direction if the product should expose both concepts.
