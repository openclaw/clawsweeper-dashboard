## Summary

Keep open. ClawHub cleanup policy keeps issues open unless maintainers resolve them, and current main confirms this is an intentional deleted-account lifecycle boundary rather than a simple missed session write. The report remains useful because users experience the backend block as a login loop, and maintainers need to decide the support, UX, and policy path for re-registration after deletion.

## What This Changes

The issue requests that GitHub OAuth login after deleting a ClawHub account create a usable session and load the profile instead of leaving the UI logged out.

## Best Possible Solution

Maintainers should define the intended post-deletion lifecycle and make code, docs, support tooling, and UI agree: either keep permanent blocking with a clear user-facing support path, provide a verified administrative purge process, or implement a deliberate tombstone/re-registration flow with tests for OAuth callback, active-session queries, account deletion, and error display.

## Security Review

Status: needs_attention

Summary: Account deletion and re-registration semantics are privacy- and security-sensitive, so any fix needs explicit maintainer review before changing session creation behavior.

Concerns:

- **[medium] Preserve irreversible deletion semantics:** `docs/security.md:96`
  - body: The documented policy says user-initiated deletion is irreversible and cannot be restored by logging in again. Automatically recreating sessions or accounts for the same deleted identity would change that security/privacy contract.
  - confidence: 0.9
- **[medium] Avoid automatic account resurrection:** `convex/auth.ts:42`
  - body: The OAuth callback currently rejects deactivated users before normal post-login work. Reversing that behavior or auto-purging records needs maintainer-approved lifecycle design because it changes who can regain access after deletion.
  - confidence: 0.87
