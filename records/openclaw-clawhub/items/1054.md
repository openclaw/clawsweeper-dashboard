## Summary

Keep open. Current main intentionally treats user-initiated deletion as irreversible: deletion writes inactive tombstone fields, OAuth reauth is rejected, and active-user lookups return null, so changing this needs maintainer account-lifecycle and security policy.

## What This Changes

The issue asks for GitHub OAuth login after deleting a ClawHub account to create an authenticated session and load the user profile.

## Best Possible Solution

Maintainers should choose and document the post-deletion lifecycle, then implement either clear permanent-block messaging, a verified admin purge path, or a deliberate re-registration flow with tests preserving deletion and security invariants.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: deleteAccount marks the user deactivated/purged, the OAuth callback rejects that account via handleDeletedUserSignIn, and users.me returns null through getOptionalActiveAuthUserId.

## Solution Assessment

Unclear. Automatically creating a session after deletion is not the established best fix because docs and tests define deletion as irreversible; a maintainer-approved permanent-block UX, verified purge, or re-registration design is safer.
