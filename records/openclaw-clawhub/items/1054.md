## Summary

Keep open. The ClawHub cleanup policy keeps issues open, and current main shows the reported post-deletion login state is an intentional account-lifecycle boundary rather than a simple missing session write; the remaining work needs maintainer policy/support judgment.

## What This Changes

A user reports that after deleting a ClawHub account, GitHub OAuth completes and sets a cookie but the app remains logged out instead of creating a new session/profile.

## Best Possible Solution

Maintainers should define and document the post-deletion lifecycle, then align code, tests, support tooling, and UI around permanent blocking, verified administrative purge, or an explicit tombstone/re-registration flow.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence path: deleteAccount marks the user deactivated/purged, the OAuth callback rejects deleted users, and users.me returns null through the active-user helper.

## Solution Assessment

Unclear. Creating a new session automatically is not clearly the best fix because it conflicts with the documented irreversible deletion contract; a maintainer-approved lifecycle/support design is safer.

  - confidence: 0.9

  - body: The OAuth callback rejects deactivated users before normal post-login work, so bypassing that check or auto-purging records changes who can regain access after deletion.
  - confidence: 0.88
