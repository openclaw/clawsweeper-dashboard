## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still intentionally blocks deleted/deactivated accounts during OAuth and hides them from active-session helpers, so the remaining work is a maintainer account-lifecycle/support decision rather than a simple missing session write.

## What This Changes

A user reports that after deleting their ClawHub account, GitHub OAuth completes but no active ClawHub session/user is created and the UI remains logged out.

## Best Possible Solution

Maintainers should define the intended post-deletion lifecycle and make code, docs, support tooling, and UI agree: permanent blocking with a clear support path, verified administrative purge, or an explicit tombstone/re-registration flow with tests.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence reproduction path: `deleteAccount` sets `deactivatedAt`/`purgedAt`, the OAuth callback rejects that user, and `users.me` returns null via the active-user helper; I did not run live OAuth because this review is read-only.

## Solution Assessment

No. Automatically creating a new session after deletion is not the best fix without maintainer policy because it conflicts with the current irreversible-deletion contract; a deliberate lifecycle/support design is safer.

Concerns:

- **[medium] Preserve irreversible deletion semantics:** `docs/security.md:96`

  - confidence: 0.9
- **[medium] Avoid automatic account resurrection:** `convex/auth.ts:42`
  - body: The OAuth callback currently rejects deactivated users before scheduling normal post-login work. Reversing that behavior or purging records during sign-in would affect who can regain access after deletion.
  - confidence: 0.88
