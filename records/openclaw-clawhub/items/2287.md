## Summary

Keep open: the hardening is useful, but the PR still leaves the public hash-resolution endpoint outside the new same-skill/latest-version invariant, and the contributor proof is only tests/CI rather than live behavior evidence.

## What This Changes

The branch adds shared public skill-file moderation gates, same-skill version validation for tags/latest pointers, digest ownership markers, regression tests, a changelog note, and a moderation spec update.

## Best Possible Solution

Finish the hardening by validating the `latestVersion` returned from `resolveVersionByHash`, add a focused regression for `/api/v1/skills/resolve`, and require redacted live behavior proof before merge.

Yes, from source inspection: on the PR head, `/api/v1/skills/resolve` still returns `latestVersion` from `skill.latestVersionId` without the new same-skill or soft-delete validation. I did not execute tests because this review is read-only.

## Solution Assessment

No. The approach is directionally right and mostly scoped, but the invariant is incomplete until the public hash-resolution latest-version path uses the same validation and has a regression test.
