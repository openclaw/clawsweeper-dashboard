## Summary

Keep open. The PR is not implemented on main and the proposed hardening still leaves a blocking public-list metadata path: tags are ownership-filtered, but list responses can still serialize stale cross-skill latest-version summaries from the skill search digest.

## What This Changes

The PR adds shared Convex guards for moderated skill artifact access and same-skill version ownership across public file, download, package-compatibility, tag, test, changelog, and moderation spec surfaces.

## Best Possible Solution

Finish the invariant across both direct artifact/detail routes and public list/digest latest-version serialization, add a regression for stale cross-skill digest/list rows, and include live redacted API or Convex output showing the blocked paths.

## Reproduction Assessment

No live reproduction was run under the read-only review constraint. Source inspection gives a concrete path: a stale skillSearchDigest row with cross-skill latestVersionId/latestVersionSummary can still feed latestVersion in public list responses.

## Solution Assessment

No. The helper and route guards are the right direction, but the PR should also guard list/digest latest-version serialization and provide real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Filter list latest versions by skill ownership:** `convex/httpApiV1/skillsV1.ts:575`
  - body: Passing the skill ID into `resolveTagsBatch` only filters the tags map. The list response still serializes `item.latestVersion`, and the underlying list builders create that object from `skillSearchDigest.latestVersionSummary` plus `latestVersionId` without knowing whether the referenced version belongs to the skill. A stale row created before this fix can therefore still expose another skill's version metadata through `/api/v1/skills`, contrary to the new invariant; please clear or verify `latestVersion` for list/digest results too.
  - confidence: 0.84
