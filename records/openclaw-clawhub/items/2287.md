## Summary

Keep open: the PR is useful security hardening, but the live head still leaves the public skills audit feed dereferencing a digest latest-version ID without proving it belongs to the same skill, and the PR body only provides tests/CI rather than real behavior proof.

## What This Changes

The branch adds shared Convex moderation/file-access and same-skill version guards, denormalizes latest-version ownership into skillSearchDigest, updates affected public API routes and tests, adds changelog text, and records a moderation invariant spec.

## Best Possible Solution

Apply the same latest-version ownership guard to listAuditPage, add a stale-digest regression for that public audit route, and include redacted live API or Convex output showing the blocked paths.

Yes, source-reproducible: seed a skillSearchDigest row whose latestVersionId/latestVersionSummary point at another skill and whose latestVersionSkillId differs from skillId, then query api.skills.listAuditPage through the /audits skills feed. ## Solution Assessment

No. The shared helper and digest marker are the right direction, but the implementation needs to route listAuditPage through the same ownership check and provide real after-fix behavior proof before merge.
