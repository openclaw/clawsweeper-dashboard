## Summary

Keep this PR open: the force-pushed head appears to address the earlier monotonic marker finding, but it still needs contributor-provided real behavior proof and maintainer review before merge because it changes a security-sensitive moderation recovery path.

## What This Changes

The PR hardens paginated ban/unban skill cleanup by aborting stale unban restore batches, making ban markers monotonic, adding regression tests, updating the moderation spec, and adding a changelog entry.

## Best Possible Solution

Land the narrowed stale-batch fix after maintainer approval once the contributor adds redacted live backend evidence showing a stale unban page aborting after a later ban or deactivation.

## Reproduction Assessment

Yes, at source level. Current main captures the ban timestamp for unban restore and the paginated restore handler processes matching user.banned rows without first checking whether the owner has since been banned or deactivated again.

## Solution Assessment

Yes. The updated PR appears to be the narrowest maintainable fix: re-read the owner before unban restore pages, restrict restores to hidden user.banned rows, and only move ban markers forward.
