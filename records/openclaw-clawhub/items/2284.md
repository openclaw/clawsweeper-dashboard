## Summary

Keep this PR open: it targets a real moderation race, but the retimestamp change can move ban restore markers backward when an older scheduled ban page runs after a newer ban, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR updates Convex ban/unban skill batches to abort stale unban restores, retimestamp ban-hidden skills during later bans, add regression tests, and document the moderation invariant.

## Best Possible Solution

Keep the stale-unban abort, but retimestamp only from older ban markers to newer ones or guard stale ban pages, then add a regression case for old-batch-after-new-ban plus redacted runtime proof.

## Reproduction Assessment

Yes at source level. The current main race and the PR-introduced rollback can be traced through `banUserWithActor`, `unbanUserWithActor`, and the paginated skill batch handlers without needing speculative product behavior.

## Solution Assessment

No. The PR direction is maintainable, but the `!==` retimestamp check is too broad; the safer fix is monotonic retimestamping or a stale-ban-page guard before changing ban markers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep ban restore markers monotonic:** `convex/skills.ts:6404`
  - body: This retimestamps every existing ban-hidden row whenever the marker differs. If an older scheduled ban page runs after a later ban already moved a row to the newer `bannedAt`, this condition rewrites it back to the stale timestamp; the stale unban then aborts because the owner is banned, but a future unban for the current ban will no longer match and restore the skill. Only move the marker forward, or guard stale ban pages before retimestamping.
  - confidence: 0.88
