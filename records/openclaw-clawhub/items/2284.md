## Summary

Keep open: the PR fixes a real stale unban restore race and has sufficient runtime proof, but the current head still skips legacy `user.banned` rows with undefined `moderationStatus` during re-ban retimestamping, which can leave skills hidden after the final unban.

## What This Changes

The PR adds owner-state rechecks to paginated unban skill restore batches, monotonic ban-marker handling, removed-row skips, regression tests, a changelog entry, and moderation spec notes.

## Best Possible Solution

Land the owner re-read and fail-closed restore behavior after extending monotonic retimestamping to legacy no-status ban-hidden rows and covering the re-ban/final-unban path.

## Reproduction Assessment

Yes. Source inspection shows current main restores matching `user.banned` rows without re-reading owner state, and the PR-specific blocker is reproducible from source with a legacy undefined-status ban-hidden row.

## Solution Assessment

Not yet. The owner re-read and monotonic marker direction is the right narrow fix, but the retimestamp guard should include legacy no-status `user.banned` rows before merge.
