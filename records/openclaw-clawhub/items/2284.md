## Summary

Keep open: the patch appears to address a source-reproducible moderation trust race on current main, but this external PR still has only test/CI evidence and needs real backend behavior proof before merge.

## What This Changes

The PR hardens paginated user ban/unban skill batches by aborting stale unban restore pages, moving older ban-hidden markers forward, adding regression tests, and documenting the invariant.

## Best Possible Solution

Land the narrowed stale-batch fix after maintainer review once the contributor adds redacted live Convex/backend evidence that the stale unban restore path is blocked.

## Reproduction Assessment

Yes, at source level. Current main captures the ban timestamp for unban restore and the restore batch processes matching user.banned rows without re-reading owner state, so a delayed page can run after a later ban or deactivation.

## Solution Assessment

Yes. Re-reading owner state before every unban restore page, restoring only hidden user.banned rows, and moving ban markers forward only is the narrowest maintainable fix I found.
