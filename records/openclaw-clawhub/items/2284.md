## Label Justifications

- `P1`: A stale unban restore page can cross an account-moderation boundary by reactivating skills after the owner is banned again or deactivated.
- `merge-risk: ЁЯЪи compatibility`: The current diff can strand legacy ban-hidden rows during upgrade because the retimestamp path skips undefined `moderationStatus` rows.
- `merge-risk: ЁЯЪи security-boundary`: The PR changes the account-moderation restore boundary for delayed background pages, so maintainers should verify fail-closed behavior is intended.

## Summary

The PR addresses a real moderation race and includes strong live Convex proof, but it is not merge-ready because the new re-ban retimestamp path still skips legacy ban-hidden skills whose `moderationStatus` is undefined.

## What This Changes

The branch adds fail-closed owner-state checks to unban skill restore batches, monotonic ban marker retimestamping, removed-row skips, regression tests, a moderation spec note, and a changelog entry.

## Best Possible Solution

Land the fail-closed owner recheck after the retimestamp predicate also covers legacy `user.banned` rows with undefined `moderationStatus`, with a regression for the later-ban/final-unban path.

## Reproduction Assessment

Yes. Current main's restore handler lacks an owner-state recheck, and source inspection shows the remaining PR gap is reachable for legacy `moderationReason: "user.banned"` rows whose `moderationStatus` is undefined.

## Solution Assessment

No. The owner re-read and monotonic marker direction is the right narrow fix, but the retimestamp guard should include legacy no-status `user.banned` rows before merge.
