## Summary

Keep open: the PR addresses a real account-moderation race and includes good live proof, but current head still misses legacy ban-hidden rows with undefined `moderationStatus` during re-ban retimestamping, which can strand skills after a later final unban.

## What This Changes

The branch adds fail-closed owner-state checks to paginated unban skill restore batches, monotonic ban marker handling, removed-row skips, regression tests, a moderation spec note, and a changelog entry.

## Best Possible Solution

Land the owner re-read and fail-closed restore behavior after broadening monotonic retimestamping to legacy no-status `user.banned` rows while continuing to skip removed rows.

## Reproduction Assessment

Yes. Source inspection shows current main restores unban pages without re-reading owner state, and the PR head's remaining gap is traceable with a legacy `moderationReason: "user.banned"` row whose `moderationStatus` is undefined.

## Solution Assessment

No, not yet. The owner re-read and monotonic marker direction is the narrow maintainable fix, but the retimestamp guard should include legacy no-status `user.banned` rows before merge.
