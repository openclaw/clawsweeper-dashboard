## Summary

Keep open: the PR addresses a real source-reproducible moderation race, but the current diff still has a blocking marker rollback bug in a security-sensitive ban/unban path and lacks after-fix real behavior proof.

## What This Changes

The PR changes Convex ban/unban skill batch handling, adds mocked regression tests, updates the moderation spec, and adds a changelog entry.

## Best Possible Solution

Keep the stale-unban abort, make ban marker updates monotonic or guard stale ban pages before retimestamping, then add rollback regression coverage and redacted real backend proof.

## Reproduction Assessment

Yes, at source level. The race follows from current main’s paginated ban/unban handlers: unban restores by an old ban timestamp without re-reading owner state, while the PR’s changed ban path can retimestamp existing ban-hidden rows backward.

## Solution Assessment

No. The requested direction is appropriate, but the current implementation uses a broad `!==` retimestamp check; the safer solution is monotonic retimestamping or a stale-ban-page guard before changing existing ban markers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep ban restore markers monotonic:** `convex/skills.ts:6404`
  - body: This retimestamps every existing ban-hidden row whenever the marker differs. If an older scheduled ban page runs after a later ban already moved a row to the newer `bannedAt`, this condition rewrites it back to the stale timestamp; a future unban for the current ban will no longer match and restore the skill. Only move the marker forward, or guard stale ban pages before retimestamping.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff touches moderation recovery and introduces a concrete marker rollback risk in the ban/unban trust boundary.

- **[medium] Older ban pages can roll back newer ban markers:** `convex/skills.ts:6404`
  - body: Using `softDeletedAt !== args.bannedAt` lets a stale page from an older ban overwrite a row already retimestamped for a newer ban, which can prevent a later legitimate unban from restoring the owner's skills.
