## Summary

Keep open: the PR addresses a real moderation race, but the current diff still has a blocking ban-marker rollback bug, lacks after-fix real behavior proof, and is currently conflicting with main.

## What This Changes

The PR updates Convex ban/unban skill batches to abort stale unban restores, retimestamp ban-hidden rows during later bans, add regression tests, update the changelog, and document the moderation invariant.

## Best Possible Solution

Keep the stale-unban abort, but make ban markers monotonic or guard stale ban pages, resolve current-main conflicts, add coverage for old-ban-page-after-new-ban, and include redacted live Convex proof.

## Reproduction Assessment

Yes, at source level. The current-main race and the PR-introduced rollback are traceable through the paginated ban/unban mutations without needing speculative product behavior, though I did not execute a live Convex scenario in this read-only review.

## Solution Assessment

No. The direction is maintainable, but the `!==` retimestamp check is too broad; the safer fix is to move markers only forward or abort stale ban pages before patching rows.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep ban restore markers monotonic:** `convex/skills.ts:6404`
  - body: This retimestamps every existing ban-hidden row whenever the marker differs. If an older scheduled ban page runs after a later ban already moved a row to the newer `bannedAt`, this condition rewrites it back to the stale timestamp; a future unban for the current ban will no longer match and restore the skill. Only move the marker forward, or guard stale ban pages before retimestamping.
  - confidence: 0.88

## Security Review

  - body: Using `softDeletedAt !== args.bannedAt` lets a stale page from an older ban overwrite a row already retimestamped for a newer ban, which can prevent a later legitimate unban from restoring the owner's skills.
