## Summary

Keep this PR open: the core fix is useful and current main still lacks the stale unban restore guard, but the branch still misses legacy ban-hidden rows with undefined `moderationStatus` during later-ban retimestamping, which can leave skills hidden after the final unban.

## What This Changes

## Best Possible Solution

Land the owner re-read and fail-closed restore behavior after extending monotonic retimestamping to legacy no-status `user.banned` rows while still skipping removed rows.

## Reproduction Assessment

Yes. Source inspection shows current main restores unban pages without re-reading owner state, and the remaining PR-head gap is source-reproducible with a legacy `moderationReason: "user.banned"` row whose `moderationStatus` is undefined.

## Solution Assessment

Not yet. The owner re-read and monotonic marker direction is the right narrow fix, but the retimestamp guard should include legacy no-status `user.banned` rows before merge.
