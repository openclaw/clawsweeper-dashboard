## Summary

Keep open. The PR addresses a real current-main moderation race, but PR head still skips legacy `user.banned` rows with undefined `moderationStatus` during re-ban retimestamping, so a later final unban can leave those skills hidden.

## What This Changes

The branch adds fail-closed owner-state checks to paginated unban skill restore batches, monotonic ban-marker retimestamping, removed-row skips, regression tests, moderation spec notes, and a changelog entry.

## Best Possible Solution

Land the owner re-read and fail-closed restore behavior after extending monotonic retimestamping to legacy no-status `user.banned` rows while still skipping removed rows.

## Reproduction Assessment

Yes. Source inspection shows current main restores unban pages without re-reading owner state, and the remaining PR gap is source-reproducible with a legacy `moderationReason: "user.banned"` row whose `moderationStatus` is undefined.

## Solution Assessment

Not yet. The owner re-read and monotonic marker direction is narrow and maintainable, but the retimestamp guard should include legacy no-status `user.banned` rows before merge.
