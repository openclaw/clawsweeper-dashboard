## Summary

Keep open: the stale unban restore guard is useful and has live Convex proof, but the current head still skips legacy ban-hidden skills whose `moderationStatus` is undefined when retimestamping a later ban, which can leave those skills hidden after the final unban.

## What This Changes

The branch adds owner-state rechecks to paginated unban skill restore batches, monotonic ban-marker retimestamping, removed-row skips, regression tests, moderation spec notes, and a changelog entry.

## Best Possible Solution

Land the owner re-read and fail-closed restore behavior after extending monotonic retimestamping to legacy no-status ban-hidden rows and covering the re-ban/final-unban path.

## Reproduction Assessment

Yes. Source inspection shows current main restores unban pages without re-reading owner state, and the remaining PR gap is source-reproducible with a legacy `moderationReason: "user.banned"` row whose `moderationStatus` is undefined.

## Solution Assessment

Not yet. The owner re-read and monotonic marker direction is the right narrow fix, but the retimestamp guard should include legacy no-status `user.banned` rows before merge.
