## Summary

Keep this PR open. It fixes a real stale unban-restore moderation bug and has convincing live Convex proof, but the current head still misses the legacy no-status ban-hidden case, which can leave skills hidden after a later legitimate unban.

## What This Changes

The branch adds stale-owner checks to unban skill restore batches, retimestamps newer ban-hidden rows, skips removed rows, and adds tests, spec notes, and a changelog entry.

## Best Possible Solution

Land the fail-closed owner recheck after the retimestamp predicate also covers legacy `user.banned` rows with undefined `moderationStatus`, with a regression for the later-ban/final-unban path.

## Reproduction Assessment

Yes. Current main's restore handler lacks an owner-state recheck, and source inspection shows the remaining legacy retimestamp gap can be exercised with a `moderationReason: "user.banned"` row whose `moderationStatus` is undefined; the PR body supplies live Convex output for the after-fix stale-page abort path.

## Solution Assessment

Not yet. The owner re-read and monotonic ban marker direction is the right narrow fix, but the retimestamp guard should include legacy no-status `user.banned` rows before merge.
