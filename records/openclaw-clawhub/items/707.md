## Label Justifications

- `P1`: The unresolved design governs automatically executed skill code, blocking lifecycle events, and informed user consent across publishing and installation.
- `impact:security`: Lifecycle hooks expand the executable attack surface and current ClawHub metadata does not explicitly validate or disclose that capability.

## Summary

Current main and the latest v0.21.0 release still lack a defined lifecycle-hook metadata contract, hook-specific semantic analysis, and pre-install registry disclosure. This issue should remain open because the narrower install-registration report does not cover the broader schema, security policy, and permission-UX decisions, and a documentation-only change would prematurely endorse executable metadata without a complete trust boundary.

## What This Changes

Define, secure, document, and visibly disclose supported skill lifecycle hook declarations across ClawHub publishing and installation surfaces.

## Best Possible Solution

Create a durable security specification for a canonical, runtime-scoped hook model, then coordinate schema validation, declaration-to-file checks, semantic scanner findings, install-time consent or registration behavior, and prominent lifecycle-permission disclosure in the skill detail UI.

## Reproduction Assessment

Yes, at source level: current main and v0.21.0 omit hook declarations from the published field reference and contain no dedicated hook validation, semantic scanning, or registry disclosure model. The claimed third-party runtime execution behavior is credible context, but ClawHub’s missing contract is independently visible in source.

## Solution Assessment

No, the proposed documentation-first path is not sufficient by itself. The narrowest maintainable solution is to decide a canonical runtime scope and threat model first, then land documentation together with validation, scanning, consent, and UI disclosure rather than encouraging unsupported executable metadata.
