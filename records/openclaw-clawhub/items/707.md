## Summary

Keep this open: current main accepts generic frontmatter but does not define, validate, scan, or disclose hook-specific behavior, and the remaining security and consent choices require explicit maintainer product judgment.

## What This Changes

Define, validate, scan, and display declared skill hooks so lifecycle-triggered execution is explicit before installation.

## Best Possible Solution

Adopt an OpenClaw-owned hook declaration and threat model in the durable security specification, then add schema validation, scanner findings, and prominent skill-page disclosure while coordinating opt-in installation behavior through https://github.com/openclaw/clawhub/issues/2295.

## Reproduction Assessment

Yes at the source level: current main’s format reference, metadata parser, and moderation logic show no hook-specific supported contract or analysis path. This establishes the reported registry gap without requiring runtime execution.

## Solution Assessment

No, not as a documentation-only adoption of Claude Code syntax. The maintainable solution is a ClawHub/OpenClaw-specific schema and security policy that separates registry disclosure from automatic runtime registration.
