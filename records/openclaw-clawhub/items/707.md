## Summary

Current main and v0.21.0 still lack a supported lifecycle-hook metadata contract, hook-specific validation, install consent, and pre-install disclosure. This remains distinct from https://github.com/openclaw/clawhub/issues/2295, which covers automatic registration of OpenClaw hook bundles rather than the broader registry and security policy requested here. Likely related people: Patrick Erichsen is a medium-confidence routing candidate based on recent work across the format, publishing, scanning, and detail-page surfaces.

## What This Changes

Define, document, security-analyze, and disclose executable lifecycle hooks declared by skills.

## Best Possible Solution

Define a runtime-neutral executable lifecycle manifest with explicit runtime adapters, target validation, event and capability risk grading, install confirmation, and prominent pre-install disclosure before documenting any runtime-specific syntax as supported.

## Reproduction Assessment

Yes at the source level: current main stores arbitrary frontmatter and performs generic bundle scanning, while its public format and registry surfaces lack a supported hook contract and lifecycle disclosure. This is a verifiable capability gap rather than a failure of an existing documented contract.

## Solution Assessment

No, not exactly as proposed: adding Claude Code syntax to the documentation alone would imply portability and safety that ClawHub does not enforce. The narrowest maintainable direction is to decide the cross-runtime contract and consent model first, then implement documentation, validation, scanning, and UI together.
