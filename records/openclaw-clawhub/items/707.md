## Summary

The central request remains necessary on current main: ClawHub documents and displays declarative requirements and install metadata, but it has no supported skill lifecycle-hook schema, privilege analysis, or pre-install hook disclosure. The issue needs maintainer product and security decisions because Claude Code frontmatter hooks and OpenClaw hook packages have different runtime and registration semantics.

## What This Changes

Define, validate, scan, and disclose executable skill lifecycle metadata so users can understand and consent to hook behavior before installation.

## Best Possible Solution

Adopt a ClawHub-owned executable-capability model recording runtime, lifecycle event, hook type, target, privilege, and consent requirements; validate referenced files and scanner coverage; display declarations before installation; and document vendor-specific mappings without promising unsupported auto-registration.

## Reproduction Assessment

Yes. Current main provides a high-confidence source reproduction: the documented and parsed skill metadata and the install UI contain no lifecycle-hook capability or disclosure path.

## Solution Assessment

No, not exactly as proposed. The user problem is valid, but directly blessing Claude Code’s unrestricted syntax would conflate runtimes; the safer approach is a normalized ClawHub contract with explicit supported mappings, scanner rules, consent, and disclosure.
