## Label Justifications

- `P1`: Automatically triggered executable metadata is a high-priority registry trust-boundary concern that could affect users during ordinary agent activity.
- `impact:security`: The issue directly concerns arbitrary code execution, security analysis, permission disclosure, and user consent.

## Summary

Current main and v0.21.0 still lack a supported lifecycle-hook metadata contract and hook-specific validation, moderation, install consent, or pre-install disclosure. The related open issues address automatic OpenClaw hook registration and active-session handoff, not this broader registry trust boundary, so the RFC remains necessary but requires maintainer security and product decisions.

## What This Changes

Define and secure executable lifecycle-hook metadata across the skill format, publishing analysis, installation flow, and registry UI.

## Best Possible Solution

Define a durable cross-runtime hook trust-boundary specification, then implement typed validation, referenced-file checks, moderation signals, API and UI disclosure, and explicit installation consent as one coordinated capability.

## Reproduction Assessment

Yes, at source level. Current main has no typed hooks field or evident hook-specific validation, consent, and disclosure path, although live production prevalence was not measured.

## Solution Assessment

The end-to-end safety objective is appropriate, but documenting Claude Code syntax verbatim is not the best standalone fix; maintainers first need to define the supported ClawHub/OpenClaw runtime contract and restrictions.
