## Label Justifications

- `P1`: Executable lifecycle declarations can repeatedly run arbitrary code during privileged agent events, making the unresolved contract an urgent security and product concern.
- `impact:security`: The issue concerns code-execution declarations, scanning, informed consent, and permission disclosure for installed skills.

## Summary

Current main does not implement the requested executable skill-hook contract: the public format, canonical metadata schema, and parser omit lifecycle hooks, while the narrower automatic-registration behavior remains open at https://github.com/openclaw/clawhub/issues/2295. The security-transparency goal is valid, but this requires explicit product and security decisions rather than cleanup closure or an autonomous patch.

## What This Changes

Define, document, security-scan, and visibly disclose executable lifecycle hook metadata for published skills.

## Best Possible Solution

Approve a durable executable-metadata design covering supported runtimes, allowed events and types, referenced-file validation, scanner outcomes, page disclosure, install/update consent, and upgrade behavior, then implement the schema, scanner, UI, CLI, specs, and public documentation together while retaining https://github.com/openclaw/clawhub/issues/2295 for registration behavior.

## Reproduction Assessment

Yes, at the source level: current main's public format, shared metadata schema, and parser do not define the proposed hook field, and the related automatic-registration request remains open. The external Claude Code behavior supports the threat model but is not currently a ClawHub contract.

## Solution Assessment

No, a documentation-first addition is not the best solution. The maintainable path is a maintainer-approved security and portability contract followed by coordinated schema, scanner, disclosure, consent, runtime, spec, and documentation work.
