## Summary

Keep open. Current main still does not define a SKILL.md hook contract in the skill metadata path, hook-specific scanning, API/UI disclosure, or consent policy, so the security RFC remains active.

## What This Changes

This issue requests documenting SKILL.md `hooks`, adding hook-specific security analysis, and surfacing hook declarations before users install a skill.

## Best Possible Solution

Define the hook trust boundary in specs, then add a normalized schema/parser contract, docs, scanner checks, API/UI disclosure, and focused tests; keep install-time registration behavior separate from https://github.com/openclaw/clawhub/issues/2295.

## Reproduction Assessment

Yes. Source inspection on current main shows SKILL.md hooks are not documented, normalized into parsed metadata, surfaced in the skill UI/API, or checked by hook-specific scanner logic.

## Solution Assessment

Unclear until maintainers choose the hook policy. The broad direction is aligned, but the narrow implementation should start with a spec-level security decision rather than only adding format documentation.
