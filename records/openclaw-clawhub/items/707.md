## Summary

The request is still necessary: current main has no documented or surfaced skill-hook metadata contract, and the related open reports address adjacent install/session behavior rather than replacing this security and disclosure decision.

## What This Changes

Document hook metadata in the skill format, analyze its executable/security surface, and display declarations before installation.

## Best Possible Solution

Define a registry-owned executable-capability contract with an explicit runtime/dialect identifier, validated file targets, event/type policy, prominent pre-install disclosure, and operator consent, then split implementation into coordinated scanner/schema/UI work and the narrower installer behavior tracked separately.

## Solution Assessment

No, not as a documentation-first change. The safest maintainable solution is to choose the supported runtime and permission contract first, then add coordinated validation, security analysis, disclosure, and installer handling.

## Risks / Open Questions

- The RFC currently conflates Claude Code frontmatter hooks with OpenClaw plugin-hook bundles, so documenting a generic `hooks` field could promise compatibility that ClawHub and OpenClaw do not actually provide.
- Promoting executable hooks before defining supported runtimes, allowed event/type combinations, install-time consent, and scanner enforcement could increase users’ exposure to automatic code execution.
- The work spans durable security policy, public documentation, validation/scanning, stored metadata or derived parsing, UI disclosure, and possibly CLI behavior; treating it as a small documentation patch would leave the core risk unresolved.
