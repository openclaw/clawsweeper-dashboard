## Label Justifications

- `impact:security`: The issue concerns analysis, consent, and disclosure for lifecycle-triggered code supplied by untrusted skill packages.

## Summary

Current main still lacks a documented, normalized, and user-visible contract for executable skill hooks, and no merged replacement owns that work. The linked install reports are complementary rather than duplicates, so this security RFC remains necessary but needs an explicit maintainer product/security decision before implementation.

## What This Changes

Requests documentation, security analysis, and registry visibility for executable hooks declared by skills.

## Best Possible Solution

Define a supported executable-capability contract first, then persist normalized declarations, validate referenced files, classify security-sensitive events and types, show permission-style pre-install disclosure, and document only the runtime semantics ClawHub can accurately support.

## Reproduction Assessment

Yes at source level: current main’s format reference omits hooks, and the parser, scanner contract, and detail UI have no explicit lifecycle-hook model. Actual execution depends on the consuming agent runtime, so this is not a standalone ClawHub runtime failure reproduction.

## Solution Assessment

No, not as a documentation-only change; the narrowest safe solution is to decide the supported dialect and permission boundary first, then land parsing, scanning, disclosure, and documentation as a coordinated capability.
