## Summary

Keep this issue open: current main retains arbitrary SKILL.md frontmatter but does not define, validate, scan, install, or visibly disclose executable skill hooks, and no merged or viable canonical replacement owns that complete security boundary. The related https://github.com/openclaw/clawhub/issues/2295 covers registration behavior only, while this issue still requires an explicit maintainer product and security decision.

Likely related people
- Patrick-Erichsen — high confidence; recent owner of the security-audit UI and scanner integration surfaces.
- theonejvo — medium confidence; introduced the documented skill metadata and `metadata.openclaw` parsing model.
- Vincent Koc — medium confidence; introduced the current static publish scan helper and malware-scan parity work.

## What This Changes

Define, secure, document, and surface executable lifecycle hooks declared by skills.

## Best Possible Solution

Establish one canonical OpenClaw/ClawHub lifecycle capability contract and threat model, then add strict schema validation, referenced-file checks, event and hook-type risk classification, install-time consent, security-audit output, and skill-page disclosure before documenting the capability as supported.

## Reproduction Assessment

Yes, at source level: current main retains arbitrary frontmatter while the documented, normalized, scanned, and visible skill capability model has no hook field. Actual automatic registration is a separate runtime behavior tracked by https://github.com/openclaw/clawhub/issues/2295.

## Solution Assessment

No, a documentation-only acknowledgement of Claude Code syntax is not the safest solution. Maintainers should first choose the supported runtime contract and restrictions, then deliberately sequence the specification, security analysis, installation disclosure, and UI work.
