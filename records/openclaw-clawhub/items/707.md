## Summary

Keep open. Current main documents runtime requirements and scans uploaded skill contents, but it still does not define a runtime-scoped executable hook contract, validate hook declarations, expose hook permissions, or establish install consent; the requested trust-boundary work remains unresolved and is not superseded by the narrower open runtime issues.

## What This Changes

Define, validate, security-analyze, and disclose executable lifecycle hook declarations in skill metadata.

## Best Possible Solution

Define a specs-backed, runtime-namespaced executable-capability model that validates hook structure and referenced files, classifies privileged events and hook types, feeds scanner decisions, exposes API/UI permissions, and requires explicit install consent before documenting author-facing syntax.

## Reproduction Assessment

Yes at the source level: current main can be inspected directly and lacks a SKILL.md executable-hook contract, hook-specific validation, and user-facing permission disclosure. This is primarily a feature/security RFC rather than a failing established contract.

## Solution Assessment

No, a documentation-only addition is not the best solution because it would imply support before ClawHub defines runtime scope and enforcement. The narrowest maintainable path is a specs-first security contract followed by coordinated ingestion, scanning, disclosure, and consent changes.
