## Summary

Keep this issue open. Current main still lacks a supported SKILL.md hook contract with hook-specific validation, security analysis, and registry disclosure, and no merged PR or canonical related issue replaces that work.

## What This Changes

Define, secure, and disclose executable skill hook metadata across the format, scanner, installation boundary, and registry UI.

## Best Possible Solution

Keep this as the canonical registry-security RFC, specify one supported hook permission model in `specs/`, and coordinate schema validation, scanning, disclosure, and installation behavior with https://github.com/openclaw/clawhub/issues/2295.

## Reproduction Assessment

Yes. Current main's documented field table, metadata normalization, scanner model, and registry source provide a high-confidence source reproduction of the missing hook-specific contract.

## Solution Assessment

No, documenting Claude Code syntax alone is not the best solution. The maintainable path must define and enforce runtime ownership, schema, security policy, user consent, and registry disclosure together.
