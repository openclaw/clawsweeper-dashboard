## Summary

Keep open. Current main still lacks a SKILL.md lifecycle hook contract across docs, metadata parsing/schema, hook-specific moderation, API/UI disclosure, and install transparency; the existing `hooks` hits are package/plugin capability metadata, which is a separate surface.

## What This Changes

Request to document SKILL.md `hooks` frontmatter and add registry-side security analysis plus user-visible hook disclosure.

## Best Possible Solution

Define a maintainer-approved hook metadata and permission policy, then implement it consistently in the skill format docs, shared schema, parser/backfill path, public API, static and LLM moderation context, skill detail UI, and install warnings.

## Reproduction Assessment

Not applicable as a runtime bug. There is a high-confidence source verification path: current main can be inspected across docs, schema, parser, moderation, API, and UI, and the requested SKILL.md hook contract is still absent.

## Solution Assessment

No, a docs-only update is not the best way to solve this issue. The maintainable fix needs a policy-backed model that validates, scans, serializes, and surfaces hooks before encouraging authors to publish them.
