## Label Justifications

- `P1`: ClawHub currently lacks a governed disclosure and analysis path for skill metadata that may trigger automatic arbitrary code execution.
- `impact:security`: Lifecycle hooks cross a code-execution trust boundary through automatic commands, HTTP calls, prompts, or agent execution.

## Summary

Current main does not implement the RFC’s central capability: ClawHub has no documented, validated, or user-visible contract for executable lifecycle hooks in `SKILL.md`. The existing `hooks` field is plugin inventory metadata, not Claude Code-style automatic execution, so closing this would leave a real security and transparency gap; however, the exact schema, permitted events/types, consent flow, and relationship to installation require maintainer product and security decisions.

## What This Changes

Define, validate, security-scan, and disclose executable skill lifecycle hook metadata before installation.

## Best Possible Solution

Adopt a versioned, runtime-qualified executable-capability schema that keeps plugin hook inventory separate, allowlists supported events and hook types, validates referenced bundle files, produces scanner findings, requires explicit install consent, and displays permissions prominently on package pages.

## Reproduction Assessment

Yes, at the source level. Current main’s format, schema, registry derivation, and UI-facing model show that executable `SKILL.md` lifecycle metadata is not defined or analyzed; the existing `hooks` field is a different plugin inventory concept.

## Solution Assessment

No, the proposed documentation-first path is not sufficient by itself. The safest maintainable approach is to decide and specify ClawHub’s versioned capability, permission, and runtime contract before encouraging authors to ship automatic execution metadata.

## Risks / Open Questions

- Directly documenting Claude Code syntax could imply ClawHub support for a runtime-specific contract that OpenClaw does not currently parse or install.
- Advertising executable hooks before validation, event/type policy, permission prompts, and prominent disclosure are implemented could increase adoption of unreviewed automatic code execution.
- Automatic activation is separately tracked at https://github.com/openclaw/clawhub/issues/2295; combining activation with metadata governance without an explicit boundary could silently change installation security behavior.
- The related post-install agent workflow at https://github.com/openclaw/clawhub/issues/792 is adjacent but has different session-state and execution semantics.
