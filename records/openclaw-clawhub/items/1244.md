## Summary

Keep open. Current main still intentionally treats user-initiated account deletion as terminal: OAuth rejects deleted/deactivated users and active-user helpers exclude them; recent code improves blocked-auth error surfacing, but recovery or re-registration remains a production support plus account-lifecycle/security policy decision.

## What This Changes

The issue asks maintainers to purge or restore the reporter's GitHub-linked ClawHub account after deletion so OAuth sign-in can create an active session again.

## Best Possible Solution

Maintainers should choose the account lifecycle policy, then either add an audited recovery/re-registration or identity-unlink path with tests and operator tooling, or keep deletion terminal with clearer support messaging.

## Reproduction Assessment

Yes for a source-level path: after a GitHub-backed account is deleted, OAuth completion reaches handleDeletedUserSignIn, which rejects deactivatedAt users and migrates legacy deletedAt users to deactivatedAt/purgedAt before throwing. I did not verify the reporter's production row.

## Solution Assessment

No for a one-off purge as the general solution. A manual purge may unblock one account, but the maintainable fix is a policy-backed audited restore, re-registration, or identity-unlink flow, or an explicit terminal-deletion support path.
