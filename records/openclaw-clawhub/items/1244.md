## Summary

Keep open. This is a live ClawHub account-support and account-lifecycle policy issue: current main still treats user-initiated deletion as terminal, while the reporter needs production account triage and a security/product decision before any purge, restore, or re-registration behavior is safe.

## What This Changes

The reporter asks ClawHub maintainers to purge or reset a deleted GitHub-linked account so GitHub OAuth can create an active session again.

## Best Possible Solution

Maintainers should decide the deleted-account recovery policy, then add an audited restore, unlink/re-registration, or operator cleanup path with tests and docs, or keep deletion terminal while making the support/error path explicit.

## Reproduction Assessment

Yes for the source-level account lifecycle path: deleting an account writes deactivatedAt/purgedAt, OAuth re-entry rejects deleted/deactivated users, and active-session helpers exclude those records. I did not verify the reporter's production rows or run a live OAuth flow.

## Solution Assessment

No for an automatic one-off purge as a repository fix. The maintainable solution needs explicit account recovery/security policy before changing deletion, provider identity binding, audit, token, and operator semantics.
