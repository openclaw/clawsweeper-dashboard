## Summary

Keep open. Current main still explains the reported post-deletion OAuth loop: user-initiated deletion is terminal, OAuth completion rejects deleted/deactivated users, and active-user helpers exclude them, but this open ClawHub issue needs production support plus account-lifecycle security/product judgment rather than cleanup closure.

## What This Changes

The issue reports that GitHub OAuth completes after a prior ClawHub account deletion but no active ClawHub session is created, and asks for the reporter's account record to be purged or made re-registerable.

## Best Possible Solution

Maintainers should choose the deleted-account lifecycle policy, then either add an audited recovery or re-registration/unlink path with support tooling and tests, or keep deletion terminal while making the support route explicit in web and CLI login flows.

## Reproduction Assessment

Yes for the source-level path: current main deactivates deleted users, rejects those users during OAuth completion, and excludes them from active sessions. I did not verify the reporter's production account rows.

## Solution Assessment

No for the requested one-off purge as a general repository fix. The maintainable path needs an explicit recovery or terminal-deletion policy covering identity proof, privacy, ban separation, auditability, and operator behavior.
