## Summary

Keep open. Current main intentionally treats user-initiated account deletion as terminal, and the remaining request is production account support plus account-lifecycle/security policy; the ClawHub cleanup policy also says open issues should not be auto-closed. Merged auth-error UI work helps expose the blocked state, but it does not safely resolve re-registration or production record purging.

## What This Changes

The issue requests production support for a GitHub OAuth login loop after deleting a ClawHub account, including purging or unlinking the reporter’s deleted account record so they can re-register.

## Best Possible Solution

Maintainers should decide the deleted-account lifecycle policy, then either add audited recovery or identity-unlink tooling with tests and operator guidance, or keep deletion terminal with clearer blocked-login support messaging.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a deleted GitHub-backed account reaches handleDeletedUserSignIn during OAuth and is rejected when deactivatedAt is present or after legacy deletedAt migration. I did not verify the reporter's production row or perform a destructive account-deletion flow.

## Solution Assessment

No, a one-off public-requested purge is not the best general solution. The maintainable path needs policy-backed, audited recovery or identity-unlinking that preserves ban, ownership, token, and publisher invariants.
