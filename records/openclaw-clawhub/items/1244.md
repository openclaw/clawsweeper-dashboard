## Summary

Keep this open. Current main still treats user-initiated account deletion as terminal, and resolving the reported OAuth loop or requested purge needs production support access plus an account-lifecycle/security policy decision rather than conservative cleanup.

## What This Changes

The issue asks maintainers to purge or reset the reporter's deleted ClawHub account binding so GitHub OAuth can create an active session again.

## Best Possible Solution

Maintainers should decide the deleted-account recovery policy, then either add an audited restore/unlink/re-registration path with support tooling and tests or keep deletion irreversible while making the terminal state and support route clear in login flows.

## Reproduction Assessment

Yes, at source level: current main deactivates the user on account deletion and rejects later OAuth sign-in for that deleted user. I did not verify the reporter's specific production records because that requires privileged support access.

## Solution Assessment

No for the requested one-off purge as a repository fix: it may unblock this account, but the maintainable solution needs a policy-backed recovery or explicit irreversible-deletion path. Any code change should preserve banned/deleted identity boundaries and add regression coverage.
