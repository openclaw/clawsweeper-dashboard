## Summary

Keep open. Current main intentionally treats user-initiated ClawHub account deletion as terminal and now has UI/error handling for blocked sign-in, but this specific report still requires production support access plus an account-lifecycle/security policy decision rather than conservative cleanup closure.

## What This Changes

The issue asks maintainers to purge or reset the reporter's deleted GitHub-linked ClawHub account record so OAuth can create an active session again.

## Best Possible Solution

Maintainers should choose the deleted-account lifecycle policy, then either add audited recovery or identity-unlink tooling with tests and operator guidance, or keep deletion terminal with clearer support messaging for blocked sign-in.

## Reproduction Assessment

Yes for a source-level path: a deleted or deactivated GitHub-backed user reaches handleDeletedUserSignIn during OAuth and is rejected, while active-user helpers also exclude that record. I did not verify the reporter's production row.

## Solution Assessment

No for a one-off purge as the general solution. Manual cleanup may unblock this user, but the maintainable path needs a policy-backed, audited recovery or identity-unlink flow that preserves ban, deletion, and ownership invariants.
