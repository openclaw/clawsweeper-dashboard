## Summary

Keep open. Current main still treats self-service account deletion as terminal, and changing or purging a production OAuth-bound account needs maintainer support access plus account-lifecycle security/product approval.

## What This Changes

The reporter requests account recovery or a production purge so GitHub OAuth can create an active ClawHub session after a prior account deletion.

## Best Possible Solution

Maintainers should define the deleted-account lifecycle policy, then either add an audited recovery or re-registration path with support tooling and tests, or keep deletion terminal while making the support route explicit in login flows.

## Reproduction Assessment

Yes for the source-level path: current main deactivates deleted users, rejects those users during OAuth completion, and excludes them from active sessions. I did not verify the reporter's production account rows.

## Solution Assessment

No for the requested one-off purge as a general repository fix. The maintainable solution needs a defined recovery/support policy covering identity proof, deletion privacy, ban separation, auditability, and operator behavior.
