## Summary

Keep open. Current main still intentionally rejects deleted/deactivated ClawHub users during GitHub OAuth and active-user resolution, and this open ClawHub issue requires production account support plus account-lifecycle/security policy rather than conservative source cleanup.

## What This Changes

The issue requests recovery from a GitHub OAuth login loop after a prior ClawHub account deletion, likely by purging or unbinding a stale identity record.

## Best Possible Solution

Maintainers should decide the account lifecycle policy, then either add an audited restore/re-registration/unlink path with tests and support tooling, or keep deletion irreversible while making the terminal state and support route clear in web and CLI auth flows.

## Reproduction Assessment

Yes for the source-level behavior: a user document with deactivatedAt or legacy deletedAt is rejected by handleDeletedUserSignIn and excluded by active-user helpers, with tests covering that path. The reporter's specific production record still cannot be reproduced from the checkout alone.

## Solution Assessment

No, a one-off purge is not the best general fix unless it is done as verified support with an audit trail. The maintainable solution is a deliberate recovery or irreversible-deletion policy implemented consistently across auth, user management, tests, and support docs.
