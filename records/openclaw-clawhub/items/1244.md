## Label Justifications

- `P1`: The issue blocks a real user from signing in after GitHub OAuth and affects an account workflow required to use ClawHub.
- `impact:security`: Any purge, restore, or re-registration path changes deleted-account, ban, token, and identity-binding security boundaries.
- `impact:auth-provider`: The reported failure is tied to GitHub OAuth completion and the persisted provider account binding.
- `impact:session-state`: OAuth completes but ClawHub does not establish an active application session for the deleted/deactivated account.

## Summary

Keep open. Current main intentionally treats user-initiated ClawHub account deletion as terminal, and the source path explains the reported post-OAuth logged-out state, but this open ClawHub issue still needs production support plus account-lifecycle security/product judgment rather than conservative cleanup.

## What This Changes

The issue requests purging or unlinking a deleted ClawHub account's GitHub OAuth binding so the same GitHub user can re-register and establish a session.

## Best Possible Solution

Maintainers should choose the deleted-account lifecycle policy, then either add an audited recovery or re-registration/unlink path with support tooling and tests, or keep deletion terminal while making the support route explicit in web and CLI login flows.

## Reproduction Assessment

Yes for the source-level path: current main deactivates deleted users, rejects those users during OAuth completion, and excludes them from active sessions. I did not verify the reporter's production account rows.

## Solution Assessment

No for the requested one-off purge as a general repository fix. The maintainable path needs an explicit recovery or terminal-deletion policy covering identity proof, privacy, ban separation, auditability, and operator behavior.
