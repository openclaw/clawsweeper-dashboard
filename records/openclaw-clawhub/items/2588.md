## Summary

Current main still throws when a user has duplicate GitHub auth-account rows, so the affected account’s publishing workflow remains broken. The adjacent device-code fix does not repair or reconcile these identity records, and the safe canonicalization policy needs explicit maintainer judgment.

## What This Changes

Fix skill publishing for users with duplicate GitHub auth-account rows, prevent recurrence, and safely reconcile affected production identities.

## Best Possible Solution

Define and enforce one canonical GitHub identity mapping, add auditable and resumable reconciliation for existing records, prevent duplicate creation at the authentication boundary, and fail safely with actionable diagnostics if conflicting identity state reappears.

Yes, at source level: two GitHub authAccounts rows for one user make the current `.unique()` lookup deterministically throw, matching the stable reported error and fresh-account control. The exact path that created the duplicate records has not been reproduced.

## Solution Assessment

No; silently selecting the newest row is not established as safe because duplicates may represent conflicting identity relationships. Invariant enforcement plus audited reconciliation is the narrower security-preserving solution.
