## Summary

Current main still deterministically throws when a user has multiple GitHub auth-account rows, and no related fix owns reconciliation. The reported denial sequence is not proven as the creator, while production cleanup affects account ownership and authorization, so this remains open for maintainer-led security and data-policy review.

## What This Changes

Investigate and fix duplicate GitHub auth-account rows that cause authenticated skill publishing to fail after a device authorization denial error.

## Best Possible Solution

Define and enforce a single canonical GitHub identity invariant, audit and reconcile affected production records with an explicit ownership policy, prevent recurrence in the authentication lifecycle, and return an actionable fail-safe error if corrupt state reappears.

## Reproduction Assessment

Yes at the source level: multiple matching GitHub auth-account rows make the current `.unique()` lookup throw exactly as reported, reinforced by stable production row IDs and a successful fresh-account control. The path that created the duplicate rows has not been reproduced.

## Solution Assessment

No, silently selecting the newest row is not established as safe because duplicates may encode conflicting user, token, or ownership relationships; explicit reconciliation and invariant enforcement are safer.
