## Label Justifications

- `P1`: A valid authenticated first-time account is deterministically unable to publish, breaking a core contributor workflow in production.
- `impact:auth-provider`: The failure comes from duplicate GitHub provider-account identity records and affects authenticated publishing.

## Summary

Current main still contains the exact duplicate-sensitive identity lookup reported in production, and the issue provides deterministic CLI evidence from a valid authenticated account. The device deny mutation does not directly create auth accounts, so maintainers should correlate the supplied request ID and inspect the auth-linking lifecycle rather than assume the deny patch is the root write.

## What This Changes

Fix a deterministic skill-publish failure caused by duplicate GitHub `authAccounts` rows, repair the affected account, and investigate the device-login error that preceded the duplicates.

## Best Possible Solution

Preserve the single correct GitHub identity relationship, make account creation/linking idempotent, make lookups tolerate only equivalent legacy duplicates, and repair affected production data with dry-run output, batching/resume support, and post-repair CLI publish verification.

## Reproduction Assessment

Yes at source level: creating two GitHub `authAccounts` rows for one user makes the current `.unique()` lookup fail deterministically, exactly matching the supplied production error. The reported deny-then-authorize sequence has not independently established the duplicate creation mechanism.

## Solution Assessment

No, selecting the newest row alone is not the best complete solution because it can mask conflicting identity links. The safer fix combines validated duplicate handling, prevention in the account-linking path, a guarded data repair, and real Convex runtime verification.
