## Label Justifications

- `P1`: A deterministic production-side identity failure blocks an affected user’s core skill-publishing workflow on every attempt.

## Summary

This is a current, source-confirmed publishing blocker: duplicate GitHub account records make the identity lookup throw, and neither current main nor the latest release fixes the invalid auth state. The device-deny failure is plausible context but is not yet proven to have created the duplicates, so the issue should remain open for production cleanup and a security-aware root-cause fix.

## What This Changes

Prevent and safely repair duplicate GitHub auth-account records that cause authenticated skill publishing to fail after CLI device authorization.

## Best Possible Solution

Enforce one canonical GitHub provider account per identity, add a session-aware operator migration for existing duplicates, and validate deny, authorize, token issuance, and publishing against a real Convex runtime before deployment.

## Reproduction Assessment

Yes at source level: two GitHub `authAccounts` rows for one user deterministically make the current `.unique()` lookup throw exactly as reported. The sequence that creates those rows still requires reproduction in a real Convex authentication environment.

## Solution Assessment

No, making the lookup choose the newest row alone is not the best solution because it masks invalid security-sensitive state. Duplicate prevention, safe cleanup, and a defensive invariant response should be addressed together.
