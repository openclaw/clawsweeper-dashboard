## Label Justifications

- `P1`: A valid authenticated user is deterministically blocked from the core skill-publishing workflow until their identity data is repaired.
- `impact:security`: Canonicalizing or deleting duplicate authentication identities can alter account ownership and authorization boundaries.
- `impact:auth-provider`: The failure occurs while resolving the immutable GitHub provider identity used to authorize publishing.

## Summary

Current main partially addresses the reported deny-path failure through merged device-code collision handling, but it still deterministically throws when duplicate GitHub `authAccounts` rows reach the publishing identity lookup, and no canonical item or merged change repairs the affected identity data. The issue remains necessary because production reconciliation requires an explicit account-ownership policy rather than arbitrary duplicate selection.

## What This Changes

Restore publishing for accounts with duplicate GitHub authentication rows, prevent recurrence, and correct the related CLI device-denial failure.

## Best Possible Solution

Enforce one canonical immutable GitHub identity mapping, reconcile existing conflicts through audited resumable operator tooling, and return a fail-safe diagnostic rather than silently choosing among inconsistent ownership records.

## Reproduction Assessment

Yes at the source level: multiple GitHub account rows for the same user/provider deterministically make the current `.unique()` lookup throw exactly as reported. The specific lifecycle that originally created the duplicate authentication rows has not been reproduced.

## Solution Assessment

No; making the lookup choose the newest row would suppress the exception without proving that row represents the correct ownership relationship. Invariant enforcement, audited reconciliation, and safe conflict diagnostics are the narrower security-preserving solution.
