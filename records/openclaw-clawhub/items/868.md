## Summary

Keep open: ClawHub policy keeps issues open, and current main still has the source-level sync path where one selected skill can hit the backend slug-taken error and abort the remaining batch. The prior Codex comments are still consistent with current main; the central files have not changed since the last review checkpoint.

## What This Changes

`clawhub sync` should continue handling the remaining selected skills when one skill publish fails because the slug is already owned by another publisher.

## Best Possible Solution

Preserve backend slug uniqueness, but add sync-level per-skill slug-conflict handling that records the failed/skipped skill, continues later publishes, reports partial success accurately, and has regression coverage.

## Reproduction Assessment

Yes, via source inspection on current main. `/api/v1/resolve` can classify a slug-mismatched existing skill as `update`, `cmdSync` then awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken error with no per-skill catch in the sync loop.

## Solution Assessment

Yes. The narrow maintainable fix is in the CLI sync publish loop and tests; backend owner-scoped slug uniqueness should remain unchanged.
