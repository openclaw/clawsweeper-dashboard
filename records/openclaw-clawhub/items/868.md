## Summary

Keep this issue open. Current `main` still has the reported batch-abort path: `sync` selects changed skills, calls `cmdPublish` without a per-skill catch, and a backend slug-ownership conflict still throws the reported `Slug is already taken` error.

## What This Changes

The issue reports that `clawhub sync` aborts when one selected skill hits a backend slug-ownership conflict.

## Best Possible Solution

Add CLI-level per-skill handling for the known slug-taken publish error, continue remaining selected skills, and report an accurate partial result while preserving backend ownership rules.

## Reproduction Assessment

Yes, source-reproducible: make one selected `cmdPublish` call reject with the backend slug-taken message and current `cmdSync` will reject before later selected skills publish. I did not run the CLI because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is in CLI sync error handling, not in relaxing backend slug ownership or changing the resolve API first.
