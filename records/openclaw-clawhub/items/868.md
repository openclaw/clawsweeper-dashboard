## Summary

Keep open: current main still has a clear source-level path where one other-owner slug conflict during `clawhub sync` aborts the remaining selected publishes, and there is no closing PR or implemented fix for this issue.

## What This Changes

Fix `clawhub sync` so a selected skill whose slug is already owned by someone else does not prevent remaining eligible skills from syncing.

## Best Possible Solution

Add CLI-level per-skill handling around sync publishing that records the known slug-taken conflict, continues with remaining selected skills, and reports an accurate partial-failure result while preserving backend ownership checks.

## Reproduction Assessment

Yes, via source inspection on current main. `/api/v1/resolve` can classify the local skill as `update`, `cmdSync` then awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken error with no per-skill catch.

## Solution Assessment

Yes. The narrow maintainable fix is in the CLI sync publish loop and its regression tests; relaxing backend slug uniqueness or publisher ownership checks would be the wrong boundary.
