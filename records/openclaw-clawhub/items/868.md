## Summary

Keep open: ClawHub cleanup policy does not auto-close issues, and current main still has a source-level path where one slug-ownership conflict during `clawhub sync` aborts the remaining batch.

## What This Changes

The issue reports that `clawhub sync` aborts all skill updates when one selected local skill hits the backend `Slug is already taken` conflict.

## Best Possible Solution

Make sync accumulate per-skill publish results, continue past the known slug-taken conflict with clear skipped/failed output, and preserve backend slug ownership enforcement.

## Reproduction Assessment

Yes, via source inspection on current main. Resolve can classify a nonmatching existing slug as `update`, `cmdSync` awaits `cmdPublish` without a per-skill catch, and `cmdPublish` rethrows the backend slug-taken error.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the CLI sync publish loop: continue remaining selected skills for the known slug-taken conflict, report partial results, and leave backend ownership enforcement intact.
