## Summary

Keep open: current main still has a source-level path where one selected `clawhub sync` publish can abort the remaining batch on the backend slug-conflict error, and ClawHub cleanup policy keeps issues open rather than auto-closing them.

## What This Changes

The issue reports `clawhub sync` aborting when one selected skill hits `Slug is already taken`, preventing remaining skills from updating.

## Best Possible Solution

Add CLI-level per-skill slug-conflict handling around sync publishing while preserving backend slug ownership rules.

## Reproduction Assessment

Yes, via source inspection. `/api/v1/resolve` can classify a slug-mismatched skill as `update`, `cmdSync` then awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken error.

## Solution Assessment

Yes. The narrow maintainable fix is in the CLI sync publish loop; relaxing backend slug uniqueness or ownership checks is not needed for the reported batch-abort bug.
