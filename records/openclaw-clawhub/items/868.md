## Repair Work Prompt

Fix the `clawhub sync` batch-abort bug reported at https://github.com/openclaw/clawhub/issues/868. Current main can classify a different-owner slug as `update`, then `cmdSync` directly awaits `cmdPublish`; when the backend returns `Slug is already taken. Choose a different slug. Existing skill: /owner/slug`, the thrown error aborts the rest of the selected batch. Implement per-skill handling in the CLI sync publish loop so the known slug-taken conflict is recorded as a failed or skipped item, later selected skills still publish, and the final output clearly reports uploaded versus failed/skipped skills. Do not relax backend slug uniqueness, owner-scoped publishing, migration rules, or catch unrelated publish failures silently. Add focused regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts` where one mocked `cmdPublish` call rejects with the slug-taken message and a later selected skill is still attempted. Changelog expectation: no automated changelog entry unless there is an existing obvious unreleased CLI section; release-note wording can be maintainer-owned.
