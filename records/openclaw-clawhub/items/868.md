## Summary

Keep this issue open. Current main still has a clear source path where one selected skill’s slug-conflict publish error aborts the remaining `clawhub sync` batch.

## What This Changes

The issue reports that `clawhub sync` aborts the whole sync batch when one selected skill hits the backend “Slug is already taken” conflict.

## Best Possible Solution

Add CLI-level per-skill handling for the known slug-taken publish error, continue publishing remaining selected skills, and report an accurate partial-success result without relaxing backend ownership rules.

## Reproduction Assessment

Yes from source inspection: a selected skill classified as `update` can hit the backend slug-taken publish error, and that error is rethrown through an uncaught `await cmdPublish` in the sync loop. I did not run a live conflicting-account sync during this read-only review.

## Solution Assessment

Yes. A narrow CLI catch for the known slug-taken publish error is the maintainable fix; the backend uniqueness and owner checks should stay intact.

## Repair Work Prompt

Fix the `clawhub sync` batch behavior for known backend slug conflicts. On current main, `packages/clawhub/src/cli/commands/sync.ts` directly awaits `cmdPublish` for each selected skill, while `cmdPublish` rethrows backend errors; add focused per-skill handling so the known `Slug is already taken. Choose a different slug.` error, including the optional `Existing skill: /owner/slug` suffix, is recorded as a failed or skipped item and the remaining selected skills still publish. Print an accurate summary showing uploaded and failed/skipped skills, and make sure a sync with failures is not reported as fully successful. Do not relax backend slug uniqueness, owner-publisher checks, migration rules, or silently swallow unrelated publish failures. Add/update `packages/clawhub/src/cli/commands/sync.test.ts` with a regression case where one mocked `cmdPublish` call rejects with the slug-taken message and a later selected skill still publishes.
