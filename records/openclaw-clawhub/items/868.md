## Summary

Keep open: current main still has a source-level path where one other-owner slug conflict aborts the remaining `clawhub sync` publishes, and there is no merged/closing PR for this issue.

## What This Changes

Running `clawhub sync` should not prevent all remaining skill updates when one selected skill hits a backend slug-taken conflict.

## Best Possible Solution

Add CLI-level per-skill handling for known slug-taken publish failures, continue with remaining selected skills, and report a clear partial-failure result while preserving backend slug uniqueness.

## Reproduction Assessment

Yes, via source inspection. `/api/v1/resolve` can classify a slug-mismatched skill as `update`, then `cmdSync` awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken error.

## Solution Assessment

Yes. The narrow maintainable fix is in the CLI sync publish loop, with backend slug uniqueness and owner-scoped publishing left unchanged.

## Repair Work Prompt

Fix the `clawhub sync` batch behavior for known backend slug conflicts. In `packages/clawhub/src/cli/commands/sync.ts`, add per-skill handling around `cmdPublish` so an error matching `Slug is already taken. Choose a different slug.` is recorded as a failed or skipped item, includes the `Existing skill: /owner/slug` URL when present, and sync continues publishing later selected skills. Preserve backend slug uniqueness and owner-scoped publish rules; do not relax `convex/skills.ts`. Keep unrelated publish failures visible and do not silently swallow them. Print an accurate partial-success/partial-failure summary, and preserve a nonzero failure outcome when any selected skill failed or was skipped due to the conflict. Add regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts` that mocks one `cmdPublish` slug-taken rejection followed by another successful publish and asserts the later skill is attempted and the output does not claim full success.
