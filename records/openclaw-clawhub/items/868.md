Add CLI-level per-skill handling for known slug ownership conflicts in `clawhub sync`, continue publishing remaining selected skills, and report an accurate partial-failure result while preserving backend ownership checks.

## Repair Work Prompt

Fix the ClawHub CLI `sync` batch behavior for this issue. In `packages/clawhub/src/cli/commands/sync.ts`, add focused handling around each selected skill publish so the known backend slug conflict message (`Slug is already taken. Choose a different slug.` with optional `Existing skill: /owner/slug`) records that one skill as failed or skipped and continues publishing later selected skills. Preserve backend slug ownership rules and do not relax `convex/skills.ts`. Do not swallow unrelated publish errors silently; keep them visible. The final output must not claim all selected skills uploaded when any publish failed or was skipped, and should return a failure signal for partial failure unless an existing CLI pattern clearly dictates otherwise. Add regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts` showing a rejected `cmdPublish` for one selected skill does not prevent a later selected skill from publishing and that the summary reflects partial failure.
