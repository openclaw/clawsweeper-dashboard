## Repair Work Prompt

Fix https://github.com/openclaw/clawhub/issues/868. Current `clawhub sync` aborts the entire selected upload batch when one `cmdPublish` call fails with the backend slug conflict message `Slug is already taken. Choose a different slug. Existing skill: ...`. Add CLI-level handling around the sync publish loop so the known other-owner slug-conflict publish failure is recorded as a failed or skipped skill, remaining selected skills still publish, and the final output accurately reports partial success/failure. Do not relax backend slug uniqueness, ownership, or `migrateOwner` rules. Do not swallow unrelated publish failures silently; keep them visible and make partial failure exit semantics explicit. Add regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts` for a rejected `cmdPublish` slug conflict followed by a later selected skill still being attempted.
