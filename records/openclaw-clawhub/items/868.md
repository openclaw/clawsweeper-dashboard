## What This Changes

The issue reports that `clawhub sync` fails the whole update batch when one selected skill hits `Slug is already taken`.

## Best Possible Solution

Add CLI-level per-skill conflict handling in sync while preserving backend slug ownership and uniqueness rules.

## Reproduction Assessment

Yes, at source level. Current main can classify a nonmatching existing slug as `update`, then `cmdSync` awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken error without a per-skill catch.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the CLI sync publish loop: continue remaining selected skills for the known slug-taken conflict, report partial results, and leave backend ownership enforcement intact.

## Repair Work Prompt

Fix the `clawhub sync` batch behavior for backend slug-ownership conflicts. In `packages/clawhub/src/cli/commands/sync.ts`, wrap selected-skill publishing so the known `Slug is already taken. Choose a different slug. Existing skill: ...` publish failure is recorded for that skill and sync continues to later selected skills. Do not relax backend slug uniqueness or owner migration rules. Preserve visibility for unrelated publish failures, and make the final summary distinguish full success from partial failure; a nonzero failure after attempting remaining skills is acceptable if any selected skill failed. Add focused regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts` where one mocked `cmdPublish` rejects with the slug-taken message and a later selected skill is still published. Do not change ownership-transfer behavior or the separate transfer work.
