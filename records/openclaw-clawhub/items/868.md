## Summary

Keep this issue open: current main still has the reported batch-abort path, and the related ownership-transfer work does not cover CLI sync continuation after a per-skill slug conflict.

## What This Changes

The issue reports that `clawhub sync` aborts when one selected skill hits a backend slug ownership conflict, preventing later skills from updating.

## Best Possible Solution

Add CLI-level per-skill conflict handling around sync publishing, continue remaining selected uploads, summarize partial failures accurately, and preserve backend slug ownership checks.

## Reproduction Assessment

Yes, at source level: current main still selects slug-mismatched skills for publishing, `cmdPublish` rethrows backend publish errors, and the sync loop has no per-skill catch. I did not run a live Windows registry sync, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is in the CLI sync publish loop and regression tests; backend slug uniqueness and explicit owner-migration rules should stay unchanged.
