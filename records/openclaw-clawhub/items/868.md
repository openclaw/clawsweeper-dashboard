## Summary

Keep open: current main still lets a known backend slug-conflict publish error abort the whole `clawhub sync` batch before later selected skills publish, and ClawHub cleanup policy keeps issues open unless resolved by normal maintainer action.

## What This Changes

The issue asks `clawhub sync` to handle one per-skill slug ownership conflict without blocking updates for other selected skills.

## Best Possible Solution

Add CLI-level per-skill handling for the known slug-taken publish error, continue publishing remaining selected skills, and report accurate partial success/failure while preserving backend ownership rules.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to have resolve return an existing slug with no hash match, then make the mocked `cmdPublish` reject with the slug-taken message; current `cmdSync` awaits that rejection uncaught before later selected skills can publish.

## Solution Assessment

Yes. The narrowest maintainable solution is in the CLI sync loop and its regression tests, catching only the known conflict class and leaving backend slug ownership enforcement unchanged.
