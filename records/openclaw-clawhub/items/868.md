Keep this issue open. Current main still has a clear source path where one selected skill’s slug-conflict publish error aborts the remaining `clawhub sync` batch, and no linked or merged PR currently owns this exact fix.

## What This Changes

`clawhub sync` should handle a per-skill slug conflict without preventing the rest of the selected skills from syncing.

## Best Possible Solution

Add focused CLI sync handling that records the known slug-taken conflict for the affected skill, continues publishing remaining selected skills, and preserves backend slug ownership rules.

## Repair Work Prompt

Fix https://github.com/openclaw/clawhub/issues/868 in a narrow CLI PR. `clawhub sync` currently classifies slug-mismatched local skills as actionable, then directly awaits `cmdPublish`; if one selected skill hits the backend `Slug is already taken. Choose a different slug. Existing skill: /owner/slug` error, the whole batch aborts and later selected skills are not published. Add per-skill handling around the sync publish loop so the known other-owner slug-taken error is recorded as a failed or skipped skill, sync continues to later selected skills, and the final summary accurately reports partial success instead of `Uploaded N skill(s).` Do not relax backend slug uniqueness or ownership policy, and do not swallow unrelated publish failures. Add regression coverage in the sync CLI tests for one selected skill rejecting with the slug-taken message while a later selected skill still publishes.
