## Summary

Keep this issue open. Current main still has the reported failure shape: sync classifies a slug-mismatched remote skill as publishable, calls `cmdPublish` without a per-skill catch, and a backend slug conflict can still abort the rest of the selected batch.

## What This Changes

`clawhub sync` should keep updating other selected skills when one publish fails because that slug is already owned by another publisher.

## Best Possible Solution

Add CLI-level per-skill conflict handling in `clawhub sync`: detect the known slug-taken publish error, record that selected skill as failed or skipped with the existing owner URL when available, continue publishing the rest, then print an accurate partial-success summary and return a failure signal if any selected skill failed.

## Reproduction Assessment

Yes at source level: the current code path selects slug-mismatched skills for publishing, `cmdPublish` rethrows the backend slug-taken error, and `cmdSync` has no per-skill catch before continuing. I did not run a live Windows registry sync, but the source path is direct and matches the reporter's error.

## Solution Assessment

Yes. The narrow maintainable fix is in the CLI sync loop and its tests; the backend slug ownership rules should stay unchanged.
