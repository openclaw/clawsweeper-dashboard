## Summary

Keep open: current main still has a source-level path where one selected `clawhub sync` publish can abort the rest of the batch on the backend slug-taken error.

## What This Changes

The issue asks for `clawhub sync` to handle a slug conflict without preventing other selected skills from updating.

## Best Possible Solution

Add CLI-level per-skill handling around sync publishing that records known slug-taken conflicts, continues remaining selected skills, and reports an accurate partial-sync result while preserving backend ownership checks.

## Reproduction Assessment

Yes. Source inspection shows `/api/v1/resolve` can classify a mismatched existing slug as `update`, `cmdSync` then directly awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken error.

## Solution Assessment

Yes. The narrow maintainable repair is to handle the known slug-taken publish failure inside the sync loop; changing backend slug ownership or broadening resolve ownership semantics is unnecessary for the reported batch-abort bug.

## Repair Work Prompt

Fix issue #868 on current main. `clawhub sync` should not abort the entire selected batch when one `cmdPublish` call fails with the known backend slug-taken message (`Slug is already taken. Choose a different slug. Existing skill: ...`). Add per-skill handling in the sync publish loop that records that skill as failed or skipped, continues later selected skills, prints an accurate partial-success summary, and preserves visible/nonzero failure semantics for partial failures. Do not relax backend slug uniqueness, do not swallow unrelated publish errors, and do not mix in org-owned publish support from open PR #1840. Add focused regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts`; include a `CHANGELOG.md` Unreleased fix entry because this is user-facing CLI behavior.
