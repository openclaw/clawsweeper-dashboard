## Summary

Keep open: current main still has a source-level path where one other-owner slug conflict during `clawhub sync` aborts the remaining selected publishes, and there is no linked open or merged PR fixing the batch-continuation behavior.

## What This Changes

The issue requests that `clawhub sync` avoid failing the whole sync batch when one selected skill hits the backend “Slug is already taken” conflict.

## Best Possible Solution

Add CLI-level per-skill handling for the known slug-taken publish error, continue with remaining selected skills, preserve backend slug uniqueness, and report a clear partial-failure result.

## Reproduction Assessment

Yes. Source inspection shows `/api/v1/resolve` can classify a slug-mismatched skill as `update`, then `cmdSync` awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken error.

## Solution Assessment

Yes. The narrow maintainable fix is in the CLI sync publish loop; backend slug uniqueness and owner-scoped publishing should remain unchanged.
