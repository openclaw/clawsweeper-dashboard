## Summary

Keep open: current main still has a source-level path where one slug-ownership conflict during `clawhub sync` aborts the remaining selected publishes, and there is no closing PR linked to this issue.

## What This Changes

`clawhub sync` should not fail the whole batch when one selected skill hits the backend slug-ownership conflict.

## Best Possible Solution

Add CLI-level per-skill handling around sync publishing: record known slug-taken conflicts as skipped or failed, continue remaining selected skills, print an accurate summary, and keep backend ownership enforcement unchanged.

## Reproduction Assessment

Yes, via source inspection on current main: resolve can select a slug-mismatched skill for update, `cmdPublish` rethrows the backend slug-taken error, and `cmdSync` does not catch per selected skill. I did not run a live Windows CLI reproduction because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the CLI sync publish loop; relaxing backend slug uniqueness or owner-migration checks would solve the wrong boundary.

## Repair Work Prompt

Fix the `clawhub sync` batch publish behavior for slug-ownership conflicts. On current main, `packages/clawhub/src/cli/commands/sync.ts` directly awaits `cmdPublish` for each selected skill, so one backend error like `Slug is already taken. Choose a different slug. Existing skill: /owner/slug` aborts the whole sync. Add focused per-skill handling around the publish loop so the known slug-taken conflict is recorded as skipped or failed with the existing skill URL when present, sync continues to later selected skills, and the final output accurately reports uploaded vs skipped/failed work. Do not relax backend slug uniqueness, owner-scoped publishing, or owner-migration authorization in `convex/skills.ts`. Add regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts` that mocks one selected `cmdPublish` rejection followed by another selected publish and verifies continuation plus summary semantics. If the implementation would require a new config flag or a broad product decision, stop and report that instead.
