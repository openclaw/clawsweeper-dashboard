## Summary

Keep open: current main still has a source-reproducible CLI path where one slug-conflict publish aborts the remaining `clawhub sync` batch, and no merged PR or canonical issue resolves that CLI continuation gap.

## What This Changes

The issue asks for `clawhub sync` to tolerate a slug conflict for one skill without preventing updates for the rest of the batch.

## Best Possible Solution

Handle known backend slug-taken publish failures per selected skill in the CLI sync loop, continue remaining publishes, then print an accurate partial-failure summary while keeping backend slug ownership strict.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, via source inspection: `/api/v1/resolve` can classify a slug-mismatched skill as `update`, `cmdSync` directly awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken error.

## Solution Assessment

Is this the best way to solve the issue? Yes, the narrow maintainable fix is CLI-level per-skill handling in `cmdSync`; backend slug uniqueness and owner-scoped publishing should remain unchanged.

## Repair Work Prompt

Fix the `clawhub sync` batch behavior for known backend slug conflicts. In `packages/clawhub/src/cli/commands/sync.ts`, wrap selected-skill publishing so a `Slug is already taken. Choose a different slug.` publish failure for one skill is recorded as failed or skipped with the existing-skill URL when present, then continue publishing later selected skills. Keep backend uniqueness and ownership checks unchanged; do not relax `convex/skills.ts`. Do not swallow unrelated publish failures silently: either keep them fatal or collect them with a clear final failure. Print a summary that distinguishes uploaded, failed/skipped, and already-synced skills, and avoid `Uploaded N skill(s)` when N includes failed skills. Add focused regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts` where the first selected `cmdPublish` rejects with the slug-taken message and a later selected skill still publishes. Include a concise user-facing CHANGELOG.md fix entry if the project convention expects one for CLI behavior changes.
