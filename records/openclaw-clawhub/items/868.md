## Summary

Keep open: ClawHub cleanup policy keeps issues open, and current main still has a source-level path where one slug-conflict publish failure aborts the rest of `clawhub sync`.

## What This Changes

The issue reports that `clawhub sync` aborts when one selected skill hits the backend `Slug is already taken` publish error.

## Best Possible Solution

Add CLI-level per-skill conflict handling around sync publishing: detect the known slug-taken publish error, record that skill as failed or skipped, continue with remaining selected skills, and report partial success without weakening backend slug ownership.

## Reproduction Assessment

Yes, via source inspection. Current main can classify an existing slug with a mismatched hash as `update`, then `cmdSync` awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken publish error.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the CLI sync publish loop; backend slug uniqueness and explicit owner-migration policy should remain unchanged.
