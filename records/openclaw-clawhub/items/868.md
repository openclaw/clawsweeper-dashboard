## Summary

Keep open: current main still has the same source-level failure path, where one slug-conflict publish can stop the remaining `clawhub sync` batch.

## What This Changes

Running `clawhub sync` can abort all selected uploads when one skill hits a backend slug-ownership conflict.

## Best Possible Solution

Add CLI-level per-skill handling for the known slug-taken publish failure, continue later selected publishes, report the failed or skipped skill with the existing owner URL when available, and keep backend slug uniqueness unchanged.

## Reproduction Assessment

Yes, via source inspection rather than a live run: `/api/v1/resolve` can classify an existing nonmatching slug as `update`, `cmdSync` calls `cmdPublish` without a per-skill catch, and `cmdPublish` rethrows the backend slug-taken error.

## Solution Assessment

Yes. The narrow maintainable fix is CLI sync partial-failure handling for the known slug-taken publish error; changing backend ownership enforcement would be broader and unnecessary.
