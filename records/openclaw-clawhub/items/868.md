## Summary

Keep open: current main still lets a single slug-taken publish error abort the `clawhub sync` publish loop, and ClawHub cleanup policy keeps issues open unless a PR is already implemented on main.

## What This Changes

The issue requests that `clawhub sync` handle a per-skill slug collision without preventing other selected skills from syncing.

## Best Possible Solution

Add CLI-level partial-failure handling around sync publishing so known slug conflicts are recorded per skill, remaining selected skills continue, and the final output accurately reports uploaded, skipped, and failed items.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to have resolve return an existing slug with no hash match, then mock `cmdPublish` to reject with the slug-taken message; current `cmdSync` will rethrow before attempting later selected skills.

## Solution Assessment

Yes. The narrowest maintainable fix is in the CLI sync loop and regression tests, preserving backend ownership checks while making batch sync resilient to one known per-skill conflict.

## Repair Work Prompt

Fix `clawhub sync` so one selected skill that fails publish with the backend `Slug is already taken. Choose a different slug.` conflict does not abort the remaining selected skills. Scope the change to CLI sync behavior and tests: record the conflicted skill as failed or skipped, preserve the existing owner URL in user-facing output when present, continue publishing later selected skills, and print an accurate final summary for uploaded/failed/skipped counts. Do not relax backend slug ownership, do not change publish endpoint uniqueness rules, and do not swallow unrelated publish errors silently; unexpected failures should still be surfaced. Add focused coverage in `packages/clawhub/src/cli/commands/sync.test.ts` with multiple selected skills where the first or middle mocked `cmdPublish` rejects with the reported message and a later skill is still attempted.
