## Summary

Keep this issue open: current main still runs selected `clawhub sync` publishes through `cmdPublish` without per-skill error handling, so the backend slug-taken error can still abort the remaining batch, and no linked PR owns this exact CLI continuation bug.

## What This Changes

The issue asks for `clawhub sync` to avoid failing the entire skill update batch when one selected skill hits an existing slug owned by another publisher.

## Best Possible Solution

Add CLI-level per-skill handling around sync publishing for known slug ownership conflicts, continue remaining selected skills, and preserve backend slug ownership enforcement.

## Reproduction Assessment

Yes, at source level: `cmdSync` awaits `cmdPublish` inside the selected-skills loop with no per-skill catch, and `cmdPublish` rethrows the backend slug-taken error. I did not run a live Windows registry sync, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Yes. The narrow maintainable repair is in the CLI sync publish loop and its regression tests; the backend ownership and slug uniqueness checks should stay unchanged.

## Repair Work Prompt

Fix the `clawhub sync` batch-abort behavior reported in https://github.com/openclaw/clawhub/issues/868. In `packages/clawhub/src/cli/commands/sync.ts`, handle known `Slug is already taken. Choose a different slug.` publish failures per selected skill so the command records that skill as failed or skipped and continues with remaining selected skills. Preserve backend slug ownership and uniqueness semantics; do not relax `convex/skills.ts` checks. Add/update focused tests in `packages/clawhub/src/cli/commands/sync.test.ts` that mock `cmdPublish` rejecting for one selected skill, verify later selected skills still publish, and verify the final output/exit behavior does not claim full success. Keep unrelated publish failures visible rather than silently swallowing them. Update docs or changelog only if the touched CLI behavior requires it under local policy.
