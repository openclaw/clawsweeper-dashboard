`clawhub sync` should not prevent remaining selected skill uploads when one publish fails with an existing other-owner slug conflict.

## Best Possible Solution

Add CLI-level per-skill handling for the known slug-taken publish conflict, continue remaining selected skills, and preserve backend slug ownership rules.

## Repair Work Prompt

Fix the `clawhub sync` batch-abort bug for other-owner slug conflicts. In `packages/clawhub/src/cli/commands/sync.ts`, add per-selected-skill handling around `cmdPublish` so the known backend message `Slug is already taken. Choose a different slug.` records that skill as failed or skipped, includes the existing owner URL when present, continues publishing later selected skills, and prints an accurate partial-success summary. Do not relax backend slug uniqueness, owner migration, or publisher authorization rules. Do not hide unrelated publish failures; they should remain visible and should fail the command. Add focused regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts` for one selected skill rejecting with the slug-taken message while a later selected skill still publishes, plus summary/exit behavior that cannot be mistaken for full success. Update docs/specs only if the fix intentionally codifies new sync partial-failure behavior.
