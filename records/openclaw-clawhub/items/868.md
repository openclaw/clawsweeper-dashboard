## What This Changes

The issue reports that `clawhub sync` aborts the whole batch when one selected skill publish hits a backend slug-taken conflict.

## Best Possible Solution

Add CLI-level per-skill conflict handling around sync publishing while preserving backend slug uniqueness and owner-scoped publishing rules.

## Reproduction Assessment

Yes, via source inspection. `/api/v1/resolve` can classify a slug-mismatched skill as `update`, `cmdSync` then awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken error.

## Solution Assessment

Yes. The narrow maintainable fix is in the CLI sync publish loop and regression tests; relaxing backend slug uniqueness or ownership checks is not needed.

## Repair Work Prompt

Fix the `clawhub sync` batch-abort bug when one selected skill publish fails with the backend slug-taken conflict. Keep backend slug uniqueness and owner-scoped publishing unchanged. In `packages/clawhub/src/cli/commands/sync.ts`, handle the known `Slug is already taken. Choose a different slug.` publish failure per selected skill, record that skill as failed or skipped with the existing owner URL when available, continue publishing later selected skills, and print an accurate partial-success summary that cannot look like full success. Add focused regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts` by mocking `cmdPublish` to reject for one selected slug and asserting later selected skills still publish and the output reflects the conflict. Do not swallow unrelated publish errors.
