## Summary

Keep open: current main still has a source-level path where one slug-conflict publish during `clawhub sync` aborts the remaining selected skills, and this ClawHub review pass keeps issues open rather than auto-closing them.

## What This Changes

The issue requests that `clawhub sync` not fail the whole batch when one selected skill hits the backend slug-taken conflict.

## Best Possible Solution

Add CLI-level conflict handling around the sync publish loop so known slug-taken conflicts are recorded per skill, later selected skills still publish, and backend slug ownership rules remain unchanged.

## Reproduction Assessment

Yes, via source inspection. `/api/v1/resolve` can classify a slug-mismatched skill as `update`, `cmdSync` then awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken error.

## Solution Assessment

Yes. The narrow maintainable fix is in the CLI sync publish loop; relaxing backend slug uniqueness or ownership checks is not needed for the reported batch-abort bug.
