## Summary

Keep open: current main still has a source-level path where one backend slug-ownership conflict aborts the rest of `clawhub sync`, and ClawHub cleanup policy does not auto-close issues.

## What This Changes

The issue reports that `clawhub sync` should not prevent all selected skills from updating when one skill hits `Slug is already taken`.

## Best Possible Solution

Add sync-level per-skill handling for the known slug-taken publish error, continue remaining selected skills, and report partial results while keeping backend slug ownership unchanged.

## Reproduction Assessment

Yes, via source inspection. Current main can classify an existing nonmatching slug as `update`, then `cmdSync` awaits `cmdPublish`, and `cmdPublish` rethrows the backend slug-taken error without a per-skill catch.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the CLI sync publish loop; backend slug uniqueness and explicit owner-migration rules should remain unchanged.
