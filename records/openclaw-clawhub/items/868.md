## Summary

Keep open: current main still has the reported source-level failure path where a single slug-conflict publish can abort the rest of a `clawhub sync` batch.

## What This Changes

The issue asks for `clawhub sync` to avoid blocking all selected skill updates when one skill conflicts with an existing slug.

## Best Possible Solution

Add focused CLI sync conflict handling that records the slug-taken publish failure for that skill, continues later selected publishes, and reports a truthful partial result while preserving backend uniqueness rules.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: resolve can mark a conflicting existing slug as `update`, `cmdPublish` can throw the backend slug-taken error, and `cmdSync` currently has no per-skill catch before continuing the loop.

## Solution Assessment

Yes. The narrow maintainable fix is CLI-level per-skill handling around sync publishing for the known slug-conflict error, with backend slug ownership enforcement left unchanged.

## Repair Work Prompt

Fix the `clawhub sync` batch behavior for slug conflicts. When one selected skill publish fails with the known backend message `Slug is already taken. Choose a different slug.` optionally including `Existing skill: /owner/slug`, record that skill as failed or skipped, continue publishing remaining selected skills, and print an accurate partial-success/partial-failure summary. Preserve backend slug uniqueness and ownership checks, do not change `/api/v1/skills` publish semantics, and do not swallow unrelated publish failures. Add regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts` for a mocked `cmdPublish` rejection on one selected skill followed by a later selected skill still being attempted. Include a user-facing `CHANGELOG.md` entry for the CLI fix if this repo's current changelog format requires one.
