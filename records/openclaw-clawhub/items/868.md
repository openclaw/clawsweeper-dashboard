## Summary

Keep this issue open: current main still has a source-reproducible batch-abort path where one selected sync publish can throw the backend slug-conflict error before later selected skills are processed, and no canonical merged or open fix owns this exact behavior.

## What This Changes

The issue reports that `clawhub sync` aborts when one selected skill hits a backend slug conflict, preventing later skills from syncing.

## Best Possible Solution

Add CLI-level per-skill handling around sync publishing for the known slug-taken conflict, continue with remaining selected skills, and report an accurate partial-success result while preserving backend ownership checks.

## Reproduction Assessment

Yes, at source level: `cmdSync` directly awaits `cmdPublish`, `cmdPublish` rethrows backend publish errors, and the backend still produces the reported slug-taken conflict. I did not run a live Windows registry sync, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is in the CLI sync publish loop and its regression tests; backend slug uniqueness and owner-scoped publishing should stay unchanged.
