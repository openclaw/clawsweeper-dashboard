## Summary

Keep this issue open. Current main still has the reported source path where one selected `clawhub sync` publish can hit the backend slug-taken error and abort the rest of the batch; related owner-migration/org-publish fixes do not cover this sync continuation case.

## What This Changes

The issue asks for `clawhub sync` to keep updating other selected skills when one skill hits a slug ownership conflict.

## Best Possible Solution

Add CLI-level per-skill handling around sync publishing: detect the known slug-taken conflict, record the affected skill as failed or skipped with the existing owner URL when available, continue remaining uploads, and print an accurate partial-result summary while preserving backend ownership rules.

## Reproduction Assessment

Yes. Source inspection shows a selected skill can be classified for upload, `cmdPublish` rethrows the backend slug-taken error, and `cmdSync` has no per-skill catch before continuing the batch.

## Solution Assessment

Yes. The narrow maintainable fix is sync-level handling for the known publish conflict while leaving backend slug uniqueness and publisher authorization intact.
