## Summary

Keep this issue open. Current main still has a source-reproducible path where one selected `clawhub sync` publish can hit the backend slug-taken error and abort the remaining batch; related ownership-transfer work does not cover the CLI continuation case.

## What This Changes

The issue requests that `clawhub sync` continue handling other skills instead of aborting the whole batch when one selected skill hits a slug conflict.

## Best Possible Solution

Add sync-level per-skill handling for the known slug-taken publish conflict, continue the remaining selected uploads, and report an accurate partial-success result without changing backend ownership rules.

## Reproduction Assessment

Yes, at source level. Current main classifies slug-mismatched skills for upload, `cmdPublish` rethrows the backend slug-taken error, and `cmdSync` has no per-skill catch before the rest of the batch.

## Solution Assessment

Yes. The narrow maintainable fix is CLI sync batch handling for the known conflict while preserving backend slug ownership and explicit transfer/migration constraints.
