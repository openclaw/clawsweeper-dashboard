## Summary

Keep open: current main has already fixed the stale CLI schema for the flattened verify envelope, but the remaining reported workflow still reproduces because a clean, settled `fulcra-context@1.4.10` verify response is still `ok:false` with `card.missing`. Publisher-supplied `skill-card.md` is intentionally stripped/rejected, so the remaining fix should recover or backfill server-generated cards rather than accept uploaded cards.

## What This Changes

The issue reports that `clawhub skill verify` rejects the live verifier response schema and that publish strips `skill-card.md` while verify can fail with `card.missing`.

## Best Possible Solution

Preserve the server-generated Skill Card model, add a bounded recovery/backfill path for clean settled versions missing generated cards, and ship the already-fixed flattened CLI schema in the next CLI release.

## Reproduction Assessment

Yes. The live deployed endpoint for `fulcra-context@1.4.10` still returns `ok:false` with `card.missing` despite clean security signals, and current source shows the same condition is emitted when no server-generated card is available.

## Solution Assessment

No, accepting publisher-supplied `skill-card.md` is not the best fix because current code and docs make Skill Cards server-generated. The maintainable path is to recover missing server-generated cards and keep the flattened verify schema already on main.
