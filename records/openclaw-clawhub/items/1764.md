## Summary

Keep open: live production still returns HTTP 500 before the GitHub OAuth redirect, and the linked older report at https://github.com/openclaw/clawhub/issues/629 was closed after one improvement even though later comments and open related reports show the broader auth failure persists. This needs maintainer/operator auth triage, not source-only cleanup.

## What This Changes

Users report that GitHub OAuth login on clawhub.ai can fail or loop without establishing a ClawHub session.

## Best Possible Solution

Restore production GitHub OAuth end to end for active accounts, preserve deleted/banned/deactivated account rejection, and add only import-safe diagnostics or deploy validation after redacted production smoke proof.

## Reproduction Assessment

Yes. GET requests to both the canonical clawhub.ai GitHub sign-in endpoint and the direct Convex sign-in endpoint returned HTTP/2 500 on current production v0.16.0, while a public API control returned 200.

## Solution Assessment

Yes for keeping this open and routing it to maintainer/operator auth triage. The safer path is private production config/log inspection plus an import-safe diagnostic or deploy validation, not the already-rejected eager module-load env check.
