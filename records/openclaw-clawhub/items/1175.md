## Summary

Keep open. Current main still contains a concrete Convex pagination hazard in the owner-publisher search-digest rebuild helpers, and the repository cleanup policy for ClawHub keeps issues open rather than auto-closing them even when a likely fix path is clear.

## What This Changes

The issue reports `clawhub publish` failing during personal publisher creation with Convex's multiple-paginated-queries error.

## Best Possible Solution

Repair the owner-publisher package and skill digest rebuilds so each Convex mutation performs at most one paginated query and schedules continuation work until the rebuild completes.

## Reproduction Assessment

Yes from source inspection: publish still creates a personal publisher and schedules owner-publisher digest rebuilds, and those rebuild helpers still loop over `.paginate()` without a continuation argument. I did not run a live Convex publish against production, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

Yes, the best fix is a narrow Convex backend change that makes the digest rebuild helpers process one page per internal mutation and reschedule themselves with a cursor. Replacing pagination with broad `.collect()` would be riskier because it can reintroduce document-limit and bandwidth problems as owner data grows.

Fix the Convex multiple-paginated-queries failure reported for `clawhub publish` and related import/publish paths. In `convex/functions.ts`, change `syncPackageSearchDigestsForOwnerPublisherId` and `syncSkillSearchDigestsForOwnerPublisherId` so each internal mutation invocation performs at most one `.paginate()` call, accepts a continuation cursor in its args, and schedules the next invocation when `page.isDone` is false. Preserve indexed queries and existing digest update/delete behavior; do not replace the paginated reads with unbounded `.collect()` scans. Add or update focused tests for multi-page continuation behavior and publisher-trigger scheduling. Update `specs/` with the intended invariant that search digest rebuilds must be cursor-based and never run multiple paginated queries in one Convex function if no existing spec covers it. Keep the fix limited to the publish/search-digest rebuild path and consolidate the duplicate reports in the PR body.
