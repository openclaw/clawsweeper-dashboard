## Summary

Keep open: current main still leaves plugin README relative image URLs route-relative, and this PR is a useful implementation, but it has a blocking publish-warning mismatch where any non-empty commit is treated as rewrite-ready even though the renderer requires a 40-hex commit SHA.

## What This Changes

The branch rewrites plugin README package-relative images through a commit-pinned raw GitHub asset base, stores `verification.sourcePath`, adds publish-time README asset warnings, targeted tests, and an e2e fetch retry helper.

## Best Possible Solution

Align publish warning readiness with the renderer's exact GitHub repo and 40-hex commit requirements, then land the source-metadata rewrite with the existing focused tests and type/package gates.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: current main proxies only absolute image URLs and passes plugin READMEs to `MarkdownPreview` without an asset base, so relative README images remain route-relative. The PR-specific warning bug is also source-reproducible because the form checks only non-empty source fields while the renderer requires a 40-hex commit SHA.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes: commit-pinned GitHub raw URLs are narrower than turning ClawHub into a package asset CDN, but the publish UI must use the same validation as the renderer before merge.
