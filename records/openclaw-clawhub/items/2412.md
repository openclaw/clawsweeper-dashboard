## Summary

Keep open: the PR is a focused attempt at the right bug, but as written it would still fail the linked OpenViking example because it builds raw GitHub URLs from repo root only and does not have the source subpath needed for monorepo packages; it also lacks after-fix real behavior proof.

## What This Changes

The PR adds README image URL rewriting for plugin detail pages using GitHub source repo/commit metadata, plus a publish-form warning and focused tests.

## Best Possible Solution

Keep the PR open and resolve README assets through a version-stable package asset path or source repo plus commit plus source path, keep the publish warning aligned with the renderer’s metadata gate, and require real behavior proof for the affected plugin page.

## Reproduction Assessment

Yes, for the central failure mode: the live OpenViking README uses ./images paths, the PR helper would construct a repo-root raw URL that returns 404, and the source subpath URL returns 200. I did not run the PR app, but the failure follows directly from the new helper and live package metadata.

## Solution Assessment

No. A raw GitHub rewrite can be maintainable, but this version ignores source subpaths and hides warnings for metadata the renderer cannot use; the safer fix is to carry the package source path or use package asset serving and align publish validation with render behavior.
