## Summary

Keep open: current main still has the strict cursor-length guard that rejects the API's own full `getPage()` cursor keys, and this PR's code direction looks correct, but the external PR still lacks after-fix real API behavior proof before merge.

## What This Changes

The PR widens self-describing `GET /api/v1/skills` cursor decoding to accept `getPage()` full index keys, updates cursor tests, and adds a changelog entry.

## Best Possible Solution

Keep the self-describing cursor contract, accept full `getPage()` keys only for the matching index, preserve legacy raw-array and wrong-index guards, and merge after real API pagination proof is added.

## Reproduction Assessment

Yes, by source inspection: current main emits `getPage()` index keys as `nextCursor`, then rejects those full keys when the advertised cursor is supplied on the next request. The linked issue also provides a concrete curl reproduction against production.

## Solution Assessment

Yes for the code direction: the PR narrows the wider acceptance to self-describing cursors for the selected index and keeps legacy raw-array and wrong-index safeguards. It is not merge-ready until after-fix real API behavior proof is added.
