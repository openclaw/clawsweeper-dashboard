## Summary

Keep open. Current main still creates GitHub backup commits without passing any Git author or committer identity, and this ClawHub issue needs maintainer policy for attribution semantics and privacy-safe email handling before implementation.

## What This Changes

Request that skill addition/update commits in the skills backup repository use the owner’s GitHub identity rather than the bot identity.

## Best Possible Solution

Define a maintainer-approved attribution policy, then thread a privacy-safe Git author object through publish-time and bulk backup flows into both skill and metadata commit payloads with explicit bot fallback behavior and tests.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: the publish and bulk sync paths pass only ownerHandle, while both GitHub create-commit payloads omit author and committer, so generated backup commits default to the GitHub App/bot identity.

## Solution Assessment

Unclear until maintainers choose the attribution and email policy. The narrow implementation path is clear, but the safe solution should derive privacy-preserving author details from GitHub identity rather than publishing stored email blindly.

## Risks / Open Questions

- Maintainers still need to decide whether Git author attribution should represent the publishing actor, the personal owner, an organization/publisher owner, or a bot fallback for bulk sync and restore paths.
- Git author email is public in Git history, so using stored user email directly could expose private addresses.
- ownerHandle alone is mutable and can represent a user, org publisher, or fallback id, so it is not sufficient as a trustworthy Git author identity.
