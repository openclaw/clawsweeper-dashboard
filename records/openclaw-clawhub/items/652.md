## Summary

Keep open. Current main still creates skills-backup Git commits without author or committer metadata, and the publish and bulk-sync paths only pass an owner handle, so the requested GitHub-user attribution is not implemented.

## What This Changes

The issue asks ClawHub to attribute skill addition/update commits in the skills backup repository to the owner's GitHub account instead of the GitHub App bot.

## Best Possible Solution

Define the attribution policy first, then thread a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata commit payloads with explicit bot fallback behavior and focused tests.

## Reproduction Assessment

Yes. A static reproduction path is high confidence: publish-time and bulk backup code pass only ownerHandle, and the two GitHub commit payloads contain no author or committer fields.

## Solution Assessment

Unclear until maintainer policy is set. The requested direction is reasonable, but the maintainable fix needs an approved attribution model and privacy-safe email source before code changes are made.

## Risks / Open Questions

- Maintainers still need to decide whether Git author attribution should represent the publishing actor, the personal owner, an organization/publisher owner, or a bot fallback for sync paths.
- A naive implementation using stored user email as the Git author email could expose private addresses in public Git history; a GitHub noreply-style or otherwise approved email source likely needs policy first.
- ownerHandle is not sufficient as an author identity because it can be a publisher/org handle or user-id fallback rather than a GitHub account login.
