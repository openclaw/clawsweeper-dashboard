## Summary

Keep open. This is a valid ClawHub issue, and current main still creates skill backup commits without passing Git author or committer identity. The remaining work needs maintainer policy for attribution semantics and privacy-safe author email handling before an automated repair should be queued.

## What This Changes

Requests that skill addition/update commits mirrored to the skills backup repository use the owner's GitHub account details as the Git author instead of the ClawHub bot identity.

## Best Possible Solution

Define the attribution policy first, then thread a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata Git commit payloads, with explicit bot fallback behavior and tests for personal owners, org/publisher ownership, missing identity, and bulk sync.

## Risks / Open Questions

- Maintainers still need to decide whether Git author attribution should represent the publishing actor, the personal owner, an organization/publisher owner, or a bot fallback for bulk sync and restore paths.
- Using stored OAuth email directly as a Git author email could expose private addresses; the implementation needs an approved privacy-safe email source such as a GitHub noreply-style address.
- ownerHandle alone is not a complete author identity because it may refer to a handle fallback, a renamed account, or future organization/publisher ownership rather than the authenticated publishing actor.
