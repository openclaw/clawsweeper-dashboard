## Label Justifications

- `P2`: The active publication path has a meaningful attribution gap, but skill publishing itself remains functional.
- `impact:security`: Changing public Git identity attribution affects trust, consent, and impersonation boundaries.

## Summary

Current main still creates hosted-skill mirror commits without explicit author or committer metadata, leaving GitHub to attribute them to the App bot. The request remains useful, but maintainers need to define consent, identity verification, and provenance semantics before implementation.

## What This Changes

Use the authenticated ClawHub user’s GitHub identity as the author of skill addition and update commits mirrored to openclaw/skills instead of attributing every commit to clawdhub[bot].

## Best Possible Solution

Define a provenance-safe contract that retains the GitHub App as committer, uses only a verified and consented GitHub-linked identity as author, and records the originating ClawHub publication in durable metadata and tests.

## Reproduction Assessment

Yes. On current main, hosted skill publication schedules `backupSkillToGitHub`, and its Git commit request omits both `author` and `committer`, directly explaining the bot attribution.

## Solution Assessment

No, not as an unconditional replacement of the bot identity. The narrow maintainable design is a verified and consented user author combined with an explicit service committer and fallback policy.
