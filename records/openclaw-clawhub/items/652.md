## Label Justifications

- `P2`: This is a bounded provenance improvement to an active mirror workflow, with no outage, data loss, or urgent regression.
- `impact:security`: Generated commit identity is part of the publication audit trail, and an incorrect mapping could misrepresent or disclose publisher identity.

## Summary

Current main still creates generated skills-repository commits without explicit `author` or `committer` metadata, so GitHub attributes them to the App bot. The request remains necessary, but deciding whose identity represents personal, organization, bulk-sync, restore, and deletion activity is a provenance/privacy product decision rather than a safe automatic fix.

## What This Changes

Use the authenticated GitHub user’s identity as the author of generated skills-repository commits instead of attributing every commit to the ClawHub bot.

## Best Possible Solution

Keep the GitHub App as the service-controlled committer, define which authenticated publishing actor may be recorded as author, derive only an approved privacy-safe identity such as a verified GitHub noreply address, retain a bot fallback for non-user operations, document the trust semantics in `specs/`, and prove the resulting metadata through a real GitHub API backup run.

## Reproduction Assessment

Yes. Current main’s Git create-commit payloads omit `author` and `committer`, and the linked public repository exhibits the resulting App-bot attribution.

## Solution Assessment

No, not exactly as stated. User attribution is useful, but the narrow maintainable solution must preserve the service committer, identify the actual publishing actor rather than merely the current owner, protect email privacy, and define safe fallback behavior.

## Risks / Open Questions

- The intended actor is ambiguous for organization-owned skills, ownership transfers, scheduled bulk synchronization, restore operations, and deletions; assigning the current owner could produce historically inaccurate provenance.
- Using a stored or provider-returned email directly could expose a private address, while using only the mutable owner handle is insufficient to establish a verified Git author identity.
- Changing only the author field would not make the user the authenticated GitHub actor or cryptographically sign the commit; the UI must not imply stronger verification than the metadata provides.
