## Summary

Keep open. Current main still creates skill-backup Git commits without author or committer metadata, and ClawHub policy keeps issues open unless they are terminal close cases.

## What This Changes

Request that skill addition/update commits in the skills backup repository use the skill owner's GitHub identity as the Git commit author instead of the ClawHub bot.

## Best Possible Solution

Define a maintainer-approved attribution policy, then pass a privacy-safe Git author object through publish-time and bulk backup paths into both GitHub commit payloads with explicit bot fallback behavior and focused tests.

## Reproduction Assessment

Yes for the current behavior by source inspection. The publish and bulk backup paths pass only ownerHandle, and the GitHub create-commit payloads contain no author or committer fields.

## Solution Assessment

Unclear until maintainer policy is set. The requested direction is reasonable, but the maintainable fix needs an approved attribution model and privacy-safe email source before code changes are made.

## Risks / Open Questions

- Maintainers need to decide whether attribution should represent the publishing actor, the personal owner, an organization/publisher owner, or a bot fallback for bulk sync and restore-style paths.
- Using stored user email directly as a Git author email could expose private addresses; a safe implementation likely needs a GitHub noreply-style address or another approved source.
- ownerHandle alone is not sufficient as an author identity because it can represent org publishers, fallback ids, or display names rather than a personal GitHub account.
