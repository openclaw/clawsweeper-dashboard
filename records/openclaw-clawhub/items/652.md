## Summary

This remains necessary on current main: the active skill backup writer still creates GitHub commits without explicit author or committer metadata, so the backup repo continues to rely on the GitHub App/bot credential for commit attribution. The work is valid, but it needs maintainer product/security direction before implementation because Git author identity, publisher identity, private email handling, org ownership, and automated sync/restore commits are trust-boundary choices.

## What This Changes

Request to attribute skill addition/update commits in the openclaw/skills backup repository to the publishing GitHub user instead of the ClawHub bot identity.

## Best Possible Solution

Define a durable attribution policy first, then thread a privacy-safe Git author object through publish-time and bulk backup paths while keeping clear bot/App committer semantics and explicit fallbacks for org and automated operations.

## Reproduction Assessment

Yes, at source level. Current main creates skill backup commits without author/committer fields and never passes authenticated GitHub user author metadata into the backup action.

## Solution Assessment

Unclear until maintainers choose the trust model. Passing a Git author object is technically narrow, but the maintainable solution must first define identity source, privacy-safe email behavior, bot committer semantics, and fallbacks for org and automated operations.

## Risks / Open Questions

- Git author identity is a provenance and trust signal; setting it to a user could imply stronger verified authorship than ClawHub actually performed unless the GitHub App remains the committer or an equivalent audit principal is documented.
- Using stored OAuth email directly as a Git author email could expose private addresses, so maintainers need a privacy-safe policy such as GitHub noreply-style addresses or explicit fallback behavior.
- The policy must cover personal publishers, org publishers, deleted/renamed accounts, bulk sync, restore, delete, and reconciliation commits so backup history does not mix incompatible attribution meanings.
