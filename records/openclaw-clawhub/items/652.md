## Summary

Keep open. Current main still omits explicit Git author/committer fields from the skill backup commit payloads, and this unresolved ClawHub issue needs maintainer policy on attribution semantics and privacy-safe email handling before implementation.

## What This Changes

The issue requests that skill addition/update backup commits in the skills repository use the skill owner’s GitHub identity as the Git commit author instead of the ClawHub bot identity.

## Best Possible Solution

Define a maintainer-approved attribution policy, then thread a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata Git commit payloads with explicit fallback behavior and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature and policy request. Source inspection does confirm the current behavior: both publish-time and bulk sync backup paths omit explicit Git author and committer fields.

## Solution Assessment

Unclear until maintainers decide the attribution policy. The requested direction is reasonable, but the maintainable implementation should not simply reuse ownerHandle or raw stored email as Git author identity.

## Risks / Open Questions

- Maintainers still need to decide whether Git author attribution should represent the publishing actor, the personal owner, an organization/publisher owner, or a bot fallback for bulk sync and restore paths.
- Using stored user email directly as a Git author email could expose private addresses; a safe implementation likely needs GitHub noreply-style attribution or another approved public identity source.
- ownerHandle alone is mutable and can represent an org/publisher or fallback value, so it is not sufficient as a stable Git author identity.
