## Label Justifications

- `P2`: This is a meaningful but non-urgent provenance improvement whose implementation is bounded once the identity policy is decided.
- `impact:security`: The issue changes how authenticated publisher identity and provenance are represented in immutable Git history.

## Summary

This remains necessary on current main: ClawHub’s active backup path still lets the GitHub App identity supply commit attribution. The request is actionable, but maintainers must first decide whether authenticated publishers should appear as Git authors, be recorded through explicit trailers or metadata, and how organization and system-generated operations should be represented without overstating verified authorship.

## What This Changes

Use the authenticated GitHub publisher’s identity to attribute skill addition and update commits in the openclaw/skills repository instead of attributing every commit solely to clawdhub[bot].

## Best Possible Solution

Keep the GitHub App as the auditable committer and define a durable publisher-attribution policy that uses verified GitHub identity data for the author or explicit commit trailers, with documented fallbacks and coverage for personal, organization, deletion, restore, and reconciliation operations.

## Reproduction Assessment

Yes, at source level. Current main creates Git commit objects without author or committer metadata, so attribution defaults to the GitHub App credential rather than the authenticated ClawHub publisher.

## Solution Assessment

Unclear. Supplying the publisher as Git author is technically plausible, but it is not yet the best-defined solution because verified identity fields, bot-committer semantics, non-personal operations, and fallback behavior require a maintainer decision.

- Changing only publication commits could leave parallel backup operations with inconsistent or misleading attribution semantics.
