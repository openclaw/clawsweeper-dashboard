## Summary

Keep open. Current main still creates GitHub backup commits without explicit Git author or committer fields, and this ClawHub issue needs maintainer policy on attribution and privacy-safe author email semantics before implementation.

## What This Changes

Request for skill addition/update commits in the skills backup repository to use the owner’s GitHub identity as the Git author instead of the bot account.

## Best Possible Solution

Define the attribution policy first, then thread a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata commit payloads with explicit fallback behavior and regression coverage.

## Reproduction Assessment

Yes for source-level verification: the current backup commit payloads do not include author or committer fields on either the skill or metadata commit. This is a feature and policy request rather than a failing documented bug reproduction.

## Solution Assessment

Unclear until maintainers choose attribution and privacy semantics. The narrow code path is clear, but the correct identity source, email address, and fallback behavior need product/security policy first.

## Risks / Open Questions

- Maintainers still need to choose whether Git author attribution should represent the publishing actor, personal skill owner, organization publisher owner, or bot fallback for bulk sync and restore-style paths.
