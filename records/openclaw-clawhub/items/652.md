## Summary

Keep open. Current main still creates skills backup commits without explicit user author metadata, and the remaining work needs maintainer policy for attribution semantics and privacy-safe public email handling.

## What This Changes

Request skill addition and update commits in the public skills repository use the relevant GitHub user's author identity instead of the ClawHub bot identity.

## Best Possible Solution

Define a maintainer-approved attribution policy, then thread a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata commit payloads with explicit bot fallback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature and policy request. Source inspection gives high confidence that current main does not pass author or committer data into the GitHub backup commit payloads.

## Solution Assessment

Unclear until maintainers choose attribution and privacy semantics. The narrow maintainable direction is to derive a stable GitHub identity plus an approved public or noreply-style email value, with bot fallback behavior.

## Security Review

Status: needs_attention

Summary: Any implementation must avoid exposing private emails or treating mutable owner handles as durable public Git author identity.

Concerns:

- **[medium] Avoid publishing private stored emails:** `convex/schema.ts:117`
  - body: Mapping Git author.email from users.email or a raw OAuth profile email would expose private account addresses in public backup repository history; use a maintainer-approved public or noreply-style source.
  - confidence: 0.91

  - body: ownerHandle can represent a publisher handle, user handle, display name, or fallback value, so using it directly for Git author identity could misattribute public commits.
