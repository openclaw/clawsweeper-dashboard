## Summary

Keep open. The current main branch still creates skill backup commits without explicit Git author or committer identity, and this issue needs a maintainer policy decision for attribution and privacy-safe author email handling before implementation.

## What This Changes

Use the ClawHub skill owner's GitHub identity as the author for skill addition/update commits mirrored to the skills repository instead of leaving every commit attributed to the bot.

## Best Possible Solution

Define the attribution policy, then thread a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata commit payloads with explicit fallback behavior and tests.

## Reproduction Assessment

Yes for source-level verification: the current backup code path still has no explicit Git author or committer data in either commit payload. This is a feature request rather than a failing documented behavior reproduction.

## Solution Assessment

Unclear until maintainers choose attribution and email privacy semantics. The narrow implementation would pass an author object into the backup payloads, but the correct identity source and fallback behavior are product decisions.

## Risks / Open Questions

- Maintainers still need to choose whether attribution should represent the publishing actor, personal skill owner, organization publisher, or bot fallback for bulk sync and restore paths.
- Using a stored email directly as a Git author email could expose private addresses in public Git history; the design likely needs a GitHub noreply-style or otherwise approved public identity source.
- ownerHandle alone is mutable and can represent publisher/org handles or fallback values, so it is not enough as a durable author identity.
