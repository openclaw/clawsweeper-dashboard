## Label Justifications

- `P2`: The source-level provenance gap is meaningful but does not currently prevent publishing, backup, or installation.
- `impact:security`: Commit authorship affects identity provenance and could expose private email data or imply unverified user authorship if implemented incorrectly.

## Summary

Current main still creates GitHub backup commits without explicit author or committer metadata, so the requested attribution behavior remains missing. The issue should stay open because choosing the publishing actor, owner, organization, or bot as author—and selecting a privacy-safe public email source—requires an explicit maintainer product and security decision.

## What This Changes

Use the authenticated GitHub user's identity for ClawHub-generated repository commits instead of attributing every commit to the bot.

## Best Possible Solution

Define and document actor-aware attribution: direct authenticated publish or delete actions should use an approved public GitHub identity or noreply address, while organization-ambiguous, restore, bulk-sync, and other automated operations should retain explicit bot authorship.

## Reproduction Assessment

Yes at source level: current main omits explicit author and committer fields from both publication and deletion commit payloads, matching the reported bot attribution. This read-only review did not execute a live publication.

## Solution Assessment

Not yet as stated: user attribution is a reasonable goal, but always using the skill owner is incomplete because the publishing actor may differ from the personal or organization owner and a privacy-safe email source remains undefined.

## Risks / Open Questions

- Using an OAuth profile email directly could disclose a private address, while using only a mutable handle could create misleading or unstable Git provenance.
- Personal publications, organization-owned publications, deletions, restores, and automated synchronization do not necessarily share the same authoritative human actor.
- Git author metadata would represent asserted authorship, not necessarily a GitHub-verified or cryptographically signed user commit, so UI and documentation must avoid overstating verification.
