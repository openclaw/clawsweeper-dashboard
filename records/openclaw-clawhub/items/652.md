## Summary

Keep open. Current main still creates skill backup commits without passing Git author or committer identity, and the remaining decision is a maintainer policy choice about owner/actor attribution and privacy-safe email handling.

## What This Changes

Request that skill addition/update commits mirrored to the skills repository use the owner's GitHub account details instead of the ClawHub bot identity.

## Best Possible Solution

Define an attribution policy, then pass a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata commit payloads with explicit bot fallback behavior and tests.

## Reproduction Assessment

Yes. Static inspection of current main shows the publish and bulk backup paths pass only ownerHandle and both Git commit payloads omit author/committer fields.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation depends on maintainer policy for actor versus owner attribution and safe public email derivation.

## Risks / Open Questions

- Maintainers still need to decide whether Git author attribution should represent the publishing actor, the personal owner, an organization/publisher owner, or a bot fallback.
- Using stored user email directly as a Git author email could expose private addresses in public Git history.
- Bulk backup sync has no publish-time actor, so historical/backfill commits need an explicit fallback rule.
