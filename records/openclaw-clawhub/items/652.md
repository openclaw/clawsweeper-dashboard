## Summary

Keep open. This is a ClawHub issue, and current main still creates skill backup commits without explicit Git author or committer identity; the fix needs maintainer policy for attribution semantics and privacy-safe author email handling.

## What This Changes

Use skill owners' GitHub account details as Git author attribution for skill addition/update commits in the skills backup repository instead of the app/bot identity.

## Best Possible Solution

Define a maintainer-approved attribution policy, then thread a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata commit payloads with explicit bot fallback behavior and focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows both publish-time and bulk backup paths pass only ownerHandle, and both GitHub create-commit payloads omit author and committer fields.

## Solution Assessment

Unclear until maintainers decide the policy. The requested direction is reasonable, but the narrow maintainable fix depends on approved attribution semantics and a privacy-safe author email source.

## Risks / Open Questions

- Maintainers still need to choose whether attribution should represent the publishing actor, the personal owner, an organization/publisher owner, or a bot fallback for bulk sync paths.
- Using stored user.email directly could expose private addresses in public Git history; a safe noreply-style or otherwise approved public email source is needed.
- ownerHandle alone is mutable and can represent an org/publisher handle or fallback id, so it is not sufficient as a Git author identity.
