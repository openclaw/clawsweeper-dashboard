## Summary

Keep open. Current main still creates skill backup Git commits without author or committer metadata, and the remaining work needs maintainer policy for attribution semantics and privacy-safe author email handling.

## What This Changes

Request skill addition/update commits in the skills backup repository to use the owner or publishing GitHub user's commit author details instead of the ClawHub bot identity.

## Best Possible Solution

Define the attribution policy first, then thread a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata commit payloads with explicit bot fallback behavior and focused tests.

## Reproduction Assessment

Not applicable as a feature and policy request. Source inspection does give high confidence that current main lacks the requested behavior: publish and bulk backup paths pass only ownerHandle, and both Git commit payloads omit author/committer metadata.

## Solution Assessment

Unclear until maintainers choose the attribution model. The requested direction is reasonable, but the maintainable implementation needs an approved author identity and privacy-safe public email source.

## Risks / Open Questions

- Maintainers still need to decide whether attribution should represent the publishing actor, the personal owner, an organization/publisher owner, or a bot fallback for bulk sync and restore-like paths.
- Using stored user email directly as a Git author email could expose private addresses in public Git history.
- ownerHandle is not sufficient as an author identity because it can represent org publishers, fallback ids, or handles without an associated GitHub email policy.
