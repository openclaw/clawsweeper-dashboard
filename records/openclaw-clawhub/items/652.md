## Summary

Keep open. Current main still creates the GitHub backup commits without explicit author or committer metadata, and the remaining change needs maintainer policy on attribution and public email handling.

## What This Changes

The issue asks ClawHub to attribute skill add/update backup commits in the skills backup repository to the owning GitHub user instead of the ClawHub bot.

## Best Possible Solution

Define the attribution policy, then pass a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata commit payloads with tests for personal owners, org owners, missing identity, and bulk sync.

## Reproduction Assessment

Yes for the reported missing behavior. Source inspection shows both publish-time and bulk backup paths pass only ownerHandle, and both GitHub create-commit calls omit author and committer fields.

## Solution Assessment

Unclear until maintainers choose the attribution model. The requested direction is reasonable, but the maintainable fix needs an approved identity source and privacy-safe public email policy.

## Risks / Open Questions

- Maintainers need to decide whether attribution should use the publishing actor, personal owner, publisher/org owner, or bot fallback.
- Publishing stored user email directly as Git author metadata could expose private addresses; a noreply-style or otherwise approved public email source is needed.
- Bulk sync and backfill commits may not have a current authenticated actor, so fallback semantics need to be explicit.
