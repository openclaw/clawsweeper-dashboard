## Summary

Keep open. Current main still creates skill backup commits without author or committer identity fields, and the remaining work needs maintainer policy for attribution semantics and privacy-safe Git author email handling.

## What This Changes

Request skill addition and update backup commits use the owning GitHub user's author details instead of the ClawHub bot identity.

## Best Possible Solution

Define a maintainer-approved attribution policy, then thread a privacy-safe Git author object through publish-time and bulk backup paths into both commit payloads with explicit bot fallback behavior and focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows both publish-time and bulk backup paths pass only ownerHandle, and both GitHub create-commit payloads omit author and committer fields.

## Solution Assessment

Unclear until maintainers decide the policy. The requested direction is reasonable, but the narrow maintainable fix depends on approved attribution semantics and a privacy-safe author email source.

## Risks / Open Questions

- Maintainers still need to choose whether attribution should represent the publishing actor, personal owner, organization/publisher owner, or bot fallback for bulk sync and restore-style paths.
- Using stored user email directly could expose private addresses in public Git history; a noreply-style or other approved safe email source needs policy first.
- ownerHandle alone is not enough for author identity because it can be an organization/publisher handle or fallback id rather than a GitHub user account.
