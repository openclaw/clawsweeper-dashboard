## Summary

Keep open. The report is concrete and plausibly valid, but current ClawHub main is a registry/catalog app and does not contain the Anthropic/OpenClaw runtime transcript serializer needed to verify or fix the provider protocol bug here; target policy also keeps ClawHub issues open unless a PR is certainly implemented on main.

## What This Changes

The issue reports that multiple tool results are persisted as separate messages after one assistant multi-tool message, causing Anthropic-compatible providers to reject the transcript.

## Best Possible Solution

Route the report to the OpenClaw runtime/provider tracker and fix transcript conversion there so every tool_result block for one assistant multi-tool turn is emitted in the immediately following message, with a regression test based on the reported JSONL shape.

## Reproduction Assessment

Unclear in this repository. The issue provides a high-quality runtime repro and error transcript, but ClawHub does not contain the provider serializer needed to execute that repro against current main.

## Solution Assessment

No for this repository. Grouping all tool results into the immediately following Anthropic message is the maintainable runtime fix; forcing single-tool calls would be a narrower workaround with worse capability.
