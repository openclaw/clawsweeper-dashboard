Keep open. The report is concrete and plausibly valid, but current ClawHub main is a registry/catalog application and does not contain the Anthropic/OpenClaw runtime transcript serializer needed to verify or fix this here; target policy also keeps ClawHub issues open.

## What This Changes

The issue reports that OpenClaw persists multiple tool results from one assistant tool-use turn as separate messages, which Anthropic rejects because all corresponding tool_result blocks must be in the immediately following message.

## Best Possible Solution

Route the report to the OpenClaw runtime/provider tracker and fix the Anthropic transcript conversion there with a regression test based on the reported JSONL shape.

## Reproduction Assessment

No, there is no high-confidence way to reproduce the issue against current ClawHub main because the affected runtime/provider serializer is absent. The issue report itself provides a concrete external transcript and error, so the bug is plausible but not source-reproducible in this repository.

## Solution Assessment

Yes for the likely runtime fix: grouping all tool_result blocks for one assistant tool-use turn into the single immediately following provider message is the narrow Anthropic-compatible direction. It is not solvable inside ClawHub without the OpenClaw runtime serialization code.
