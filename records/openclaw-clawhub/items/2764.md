## Summary

Keep open: the homepage listing cache is a bounded useful optimization, but the patch has a source-reproducible regression where cached initial listings lose `hasMore` after mount, and the linked proof artifact could not be inspected.

## What This Changes

Adds a home route loader and shared listing-data helpers to seed and cache homepage Skills/Plugins listing tabs while keeping search fetch-on-query.

## Best Possible Solution

Keep the route-loader cache approach only after preserving listing `hasMore` independently of search cleanup and adding inspectable browser/network proof for initial data and tab-cache reuse.

## Reproduction Assessment

Yes for the PR regression by source inspection: mount `HomeListingSection` with `initialListing.hasMore: true` and no search, then the cache effect sets `hasMore` before the following non-search cleanup effect clears it. I did not execute the branch because this review is read-only.

## Solution Assessment

No as submitted; the caching direction is reasonable, but listing and search `hasMore` state need to stop clobbering each other. A narrow fix plus an inspectable browser/network proof would make this path much safer.
