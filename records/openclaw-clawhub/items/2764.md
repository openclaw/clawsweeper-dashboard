## Summary

Keep this PR open: the homepage listing cache direction is useful, but the patch can hide the Load more affordance for cached listings with more pages, and the linked browser proof is still not inspectable enough to clear the contributor proof gate.

## What This Changes

The branch adds a home route loader for the default Skills/Top listing, moves homepage listing fetch/filter helpers into a shared module, and seeds/reuses a client-side listing cache for homepage tabs.

## Best Possible Solution

Keep the route-loader cache approach only after preserving listing pagination state separately from search cleanup and adding inspectable browser/network proof for initial data and tab-cache reuse.

## Reproduction Assessment

Yes for the PR regression by source inspection: mount `HomeListingSection` with `initialListing.hasMore: true` and no search, then the cache effect sets `listingHasMore` before the non-search cleanup effect clears it.

## Solution Assessment

No as submitted; the route-loader cache is a reasonable layer, but listing and search pagination state need to stop clobbering each other.
