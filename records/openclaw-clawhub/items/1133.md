## Summary

Keep this PR open: current main still has no moderation-hold lift API, but the proposed restore path has blocking moderation safety and audit-accounting issues before it can merge.

## What This Changes

This PR adds public and internal Convex admin mutations to clear a user's moderation hold, batch-restore eligible owned skills, audit the action, add tests, and document the CLI workflow.

## Best Possible Solution

Land a maintainer-reviewed hold-lift implementation that clears user hold fields, restores only rows genuinely hidden by that hold, preserves malicious/manual/quality/soft-deleted/unresolved-scan protections, records complete audit data, and documents the admin CLI path.

## Reproduction Assessment

Yes. From source inspection, main creates holds and lacks a lift path, while the PR restore path can make not_found VT rows active and cannot distinguish pre-hold hidden rows after the existing hold batch overwrites their moderation state.

## Solution Assessment

No. The admin recovery capability is useful, but this implementation needs safer restore eligibility, unresolved-scan handling, and cumulative audit accounting before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Classify not_found VT status as unresolved:** `convex/skills.ts:4147-4148`
  - body: Current moderation maps `vtAnalysis.status === "not_found"` to `scanner.vt.pending`, and the pending queue retries non-final VT statuses. This restore only treats missing, `pending`, and `loading` as needing scan, so a held skill with `not_found` becomes active with `restored.moderation_lift` and drops out of the pending selectors.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4124-4128`
  - body: The existing hold batch overwrites every non-malicious owned skill to `hidden/user.moderation` with the hold timestamp. This restore then treats those rows as hold-owned, so content already hidden for quality or manual review before the hold can be reactivated; preserve prior state or avoid overwriting already-hidden rows.
  - confidence: 0.84
- **[P2] Report cumulative restored count after all batches:** `convex/users.ts:813`
  - body: The lift audit uses only the first restore batch's `restoredCount`. When `scheduledSkills` is true, continuation batches can restore more rows without updating the `user.moderation.lift` metadata or caller-visible count, leaving large recoveries with an incomplete security trail.
  - confidence: 0.86
