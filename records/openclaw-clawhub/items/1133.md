## Summary

Keep open: current main still lacks the moderation-hold lift surface, and this PR remains an active security-sensitive admin recovery change with blocking correctness findings, missing real behavior proof, and an explicit maintainer reinvestigation signal.

## What This Changes

This PR adds public and internal Convex admin mutations to clear user moderation holds, batch-restore eligible owned skills, add tests, and document the admin CLI recovery flow.

## Best Possible Solution

Land a maintainer-reviewed hold-lift implementation that clears user hold fields, restores only skills safely attributable to the lifted hold, preserves unrelated moderation states, keeps unresolved scans queued, records cumulative audit data, and documents the admin CLI/API path.

## Reproduction Assessment

Yes, for the PR review findings. Source inspection of current main and the PR head shows the VT not_found classification mismatch, the hold batch's overwrite of prior hidden states, and the first-batch-only audit count path.

## Solution Assessment

No. The admin recovery capability is a reasonable boundary, but this PR needs safer restore eligibility, unresolved-scan handling, cumulative audit accounting, maintainer/security policy review, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Treat not_found VT results as unresolved:** `convex/skills.ts:4147-4148`
  - body: The restore path only keeps absent, pending, or loading VT results queued. Current scanner semantics map `vtAnalysis.status === "not_found"` to `scanner.vt.pending`, so a lifted skill in that state can become active and leave the scan queue; include `not_found` or reuse the existing pending-status helper before publishing it.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4124-4129`
  - body: This restore treats post-hold `user.moderation` as proof that the skill is safe to reactivate, but the hold batch rewrites non-malicious hidden rows, including quality or manual hides, to that reason and a fresh `hiddenAt`. Store prior moderation state or skip ambiguous rows so lifting a user hold cannot publish independently quarantined content.
  - confidence: 0.84
- **[P2] Report cumulative restore counts:** `convex/users.ts:813`
  - body: The lift audit records only the first batch's `restoredCount`; when `scheduledSkills` is true, continuation batches can restore more skills without updating the `user.moderation.lift` metadata or returned count. Accumulate or separately audit continuation restores so large recoveries have a complete security trail.
  - confidence: 0.86
