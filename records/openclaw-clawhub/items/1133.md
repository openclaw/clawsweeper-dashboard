## Summary

Keep open: current main does not implement a moderation-hold lift API, and the reopened PR still has blocking restore-safety and audit-accounting issues before it is mergeable.

## What This Changes

The PR adds public and internal Convex admin mutations to clear user moderation holds, batch-restore eligible owned skills, write lift audit logs, add tests, and document the CLI workflow.

## Best Possible Solution

Land a maintainer-reviewed hold-lift implementation that clears user hold fields, restores only eligible hold-hidden skills, keeps malicious, manual/quality, soft-deleted, and unresolved-scan states protected, and records complete recovery audit data.

## Reproduction Assessment

Yes, for the PR review findings: source inspection shows current main treats VT not_found as pending, hold placement overwrites non-malicious skill reasons, and the PR records only the initial restore batch count.

## Solution Assessment

No. The recovery capability is useful, but the current PR needs safer restore eligibility, unresolved-scan handling, and cumulative audit accounting before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat not_found VT results as unresolved:** `convex/skills.ts:4147-4148`
  - body: The restore path only keeps missing, pending, or loading VT results in the scan queue. Current moderation maps vtAnalysis.status === "not_found" to scanner.vt.pending, so a lifted skill with that status can become active/restored and drop out of scanning; include not_found or reuse the scanner pending helper before publishing.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4128-4129`
  - body: This predicate trusts hidden/user.moderation as proof that the skill is safe to reactivate, but the hold batch rewrites every non-malicious owned skill to that reason, including rows already hidden for quality or manual review. Lifting a hold can therefore publish content quarantined before the hold; preserve prior state or skip ambiguous rows.
  - confidence: 0.82
- **[P2] Report all restored skills in the lift audit:** `convex/users.ts:813`
  - body: The caller records only the first batch's restoredCount. When scheduledSkills is true, continuation batches can restore additional skills without updating the user.moderation.lift metadata or returned count, leaving the recovery audit trail incomplete for large publishers.
  - confidence: 0.86
