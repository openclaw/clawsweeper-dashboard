## Summary

Keep open. Current main still has the moderation-hold placement path and no lift-hold API, so this PR is not obsolete; however, the proposed restore path is security-sensitive and still has blocking restore/audit semantics that should be fixed on the PR branch before merge.

## What This Changes

The PR adds public and internal Convex admin mutations to clear user moderation holds, batch-restore eligible owned skills, write audit logs, add tests, and document the admin CLI recovery workflow.

## Best Possible Solution

Land a maintainer-reviewed implementation that adds an admin-only moderation-hold lift operation, clears the user hold fields, restores only skills genuinely hidden by that hold, keeps malicious, quality/manual, soft-deleted, and unresolved-scan states hidden or queued, records accurate recovery audit data across batches, and documents the Convex CLI workflow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat not_found VT results as unresolved:** `convex/skills.ts:4147`
  - body: The restore path only keeps missing, pending, or loading VT results in the scan queue. Current moderation maps vtAnalysis.status === "not_found" to scanner.vt.pending and the queue retries non-final VT statuses, so a lifted not_found skill becomes active/restored.moderation_lift and drops out of scanning. Include not_found or reuse the existing final-status logic before making the skill public.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4128`
  - body: This predicate treats post-hold user.moderation as proof that a skill can be reactivated, but the existing hold batch overwrites every non-malicious owned skill with user.moderation, including rows already hidden for quality or manual review. Store the prior moderation state or avoid restoring rows that were locked before the hold.
  - confidence: 0.82
- **[P2] Report all restored skills in the lift audit:** `convex/users.ts:813`
  - body: The caller records only the first batch's restoredCount. When scheduledSkills is true, continuation batches can restore additional rows without updating the returned count or user.moderation.lift metadata, leaving large recoveries with an incomplete audit trail.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, install-script, or publishing changes were found, but the moderation restore path can expose unresolved or independently hidden content and needs security review before merge.

Concerns:

- **[high] Unresolved VT results can be published:** `convex/skills.ts:4147`
  - body: The PR treats not_found VT status as safe to restore even though current moderation classifies it as pending, allowing a skill to become public and leave the scan queue.
  - confidence: 0.92
- **[high] Independent moderation decisions can be cleared:** `convex/skills.ts:4128`

  - confidence: 0.82
