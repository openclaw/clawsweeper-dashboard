## Summary

Keep open: current main still lacks the hold-lift API, and the reopened PR is useful but not merge-ready because the restore path can expose unresolved or independently hidden content, undercount audit data, and lacks real behavior proof.

## What This Changes

Adds public and internal Convex admin mutations to lift user moderation holds, batch-restore eligible skills, write audit logs, add tests, and document CLI usage.

## Best Possible Solution

Land a maintainer-reviewed hold-lift implementation that clears user hold fields, restores only skills safely attributable to the lifted hold, preserves unrelated moderation states, keeps unresolved scans queued, and records complete audit data.

## Reproduction Assessment

Yes, for the PR review findings. Source inspection shows current main treats VT not_found as pending, the hold batch overwrites non-malicious hidden reasons, and the PR audit path records only the initial restore batch result.

## Solution Assessment

No. The admin recovery capability is a reasonable boundary, but the current PR needs safer restore eligibility, unresolved-scan handling, cumulative audit accounting, and real behavior proof before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat not_found VT results as unresolved:** `convex/skills.ts:4147-4148`
  - body: The restore path only treats pending/loading VT results as unresolved, but current scanner semantics map vtAnalysis.status === "not_found" to scanner.vt.pending. A lifted skill in that state can become active/restored and leave the scan queue; include not_found or reuse the existing pending-status helper before publishing it.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4128-4129`
  - body: This predicate trusts hidden/user.moderation as proof that a skill is safe to reactivate, but the hold batch rewrites non-malicious hidden skills to that reason, including rows already hidden for quality or manual review. Lifting a hold can therefore publish content quarantined before the hold; preserve prior state or skip ambiguous rows.
  - confidence: 0.82
- **[P2] Report all restored skills in the lift audit:** `convex/users.ts:813`
  - body: The caller records only the first batch's restoredCount. When scheduledSkills is true, continuation batches can restore more skills without updating the user.moderation.lift metadata or returned count, leaving the recovery audit trail incomplete for large publishers.
  - confidence: 0.86
