## Summary

Keep open. Current main still lacks the hold-lift API, and a maintainer reopened this PR for reinvestigation; the PR remains useful but the latest diff still has security-sensitive restore and audit correctness issues before merge.

## What This Changes

The PR adds public and internal Convex admin mutations to clear user moderation holds, batch-restore eligible owned skills, add tests, and document the CLI recovery flow.

## Best Possible Solution

Land a maintainer-reviewed hold-lift flow that clears user hold fields, restores only skills genuinely eligible after that hold, keeps malicious/manual/quality/soft-deleted/unresolved-scan states protected, records complete recovery audit data, and documents the admin CLI path.

## Reproduction Assessment

Yes, for the review findings. Source inspection of current main and the latest PR diff shows the not_found scan state, pre-hold moderation overwrite path, and first-batch-only audit count without needing to mutate the checkout.

## Solution Assessment

No. The recovery capability is useful, but this diff needs safer restore eligibility, unresolved VT handling, and cumulative audit accounting before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat not_found VT results as unresolved:** `convex/skills.ts:4147`
  - body: The restore path only re-queues missing, pending, or loading VT results. Current moderation maps `vtAnalysis.status === "not_found"` to `scanner.vt.pending`, so this can publish an unresolved scan as `restored.moderation_lift` and drop it out of the scan queue.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4128`
  - body: This predicate trusts the post-hold `user.moderation` reason, but the hold batch rewrites every non-malicious owned skill to that reason, including rows already hidden for quality or manual review. Lifting the hold can therefore publish content that was quarantined before the hold.
  - confidence: 0.82
- **[P2] Report all restored skills in the lift audit:** `convex/users.ts:813`
  - body: The audit metadata records only the first batch's `restoredCount`; when continuation batches are scheduled, later restored skills are omitted from the returned count and `user.moderation.lift` audit trail.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no dependency or workflow supply-chain change, but the moderation restore path can expose unresolved or independently hidden content.

Concerns:

- **[high] Unresolved VT results can be published:** `convex/skills.ts:4147`
  - body: A skill with VT `not_found` is treated as safe to restore even though current moderation classifies that status as pending, allowing it to become public and leave the scan queue.
  - confidence: 0.92
- **[high] Independent moderation decisions can be cleared:** `convex/skills.ts:4128`
  - body: Because the hold batch overwrites hidden non-malicious skills with `user.moderation`, the new restore path can reactivate skills that were hidden for manual or quality reasons before the user hold.
  - confidence: 0.82
- **[medium] Recovery audit can undercount batch restores:** `convex/users.ts:813`
  - body: Only the first batch count is written to the lift audit log, so large recoveries can omit later restored skills from the security trail.
  - confidence: 0.86
