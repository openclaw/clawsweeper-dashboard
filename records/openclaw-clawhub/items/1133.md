## Summary

Keep open: current main still lacks the admin moderation-hold lift surface, but this security-sensitive PR is not merge-ready because the restore path still has source-backed correctness issues and the external contributor has not provided real behavior proof.

## What This Changes

The PR adds public and internal Convex admin mutations to clear user moderation holds, restore eligible owned skills in batches, write audit logs, add tests, and document CLI usage.

## Best Possible Solution

Land a maintainer-reviewed hold-lift implementation that clears user hold fields, restores only skills safely attributable to the lifted hold, keeps malicious and unresolved-scan states hidden or queued, preserves unrelated moderation decisions, records complete audit data, and documents the admin CLI path.

## Reproduction Assessment

Yes, for the PR review findings. Source inspection shows current main treats VT not_found as pending, the hold batch overwrites non-malicious hidden reasons, and the PR audit path records only the initial restore batch result.

## Solution Assessment

No. The admin recovery capability is a reasonable boundary, but this PR needs safer restore eligibility, non-terminal scan handling, cumulative audit accounting, and real behavior proof before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat not_found VT results as unresolved:** `convex/skills.ts:4147-4148`
  - body: The restore path only keeps missing, pending, or loading VT results in the scan queue. Current moderation logic maps `vtAnalysis.status === "not_found"` to `scanner.vt.pending`, and rescan policy treats it as non-terminal, so a lifted skill in this state can become active and drop out of scanning.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4128-4129`
  - body: This restore treats the post-hold `user.moderation` reason as proof that the skill is safe to reactivate. The hold batch rewrites every non-malicious owned skill to that reason, including rows already hidden for independent reasons such as `quality.low`, so lifting the hold can publish content quarantined before the user hold.
  - confidence: 0.82
- **[P2] Report all restored skills in the lift audit:** `convex/users.ts:813`
  - body: The caller records only the first batch's `restoredCount`; when `scheduledSkills` is true, continuation batches can restore additional skills without updating the `user.moderation.lift` metadata or returned count. Large recoveries will therefore have incomplete audit data.
  - confidence: 0.86
