## Summary

Keep open: current main does not implement the moderation-hold lift path, and the PR still has blocking restore-safety and audit issues in a security-sensitive moderation flow.

## What This Changes

The PR adds Convex admin mutations to lift user moderation holds, batch-restore eligible skills, log the recovery, add tests, and document the CLI workflow.

## Best Possible Solution

Land a maintainer-reviewed hold-lift flow that clears the user hold, restores only skills genuinely eligible after that hold, preserves manual/quality/malicious/soft-deleted/unresolved-scan states, and records complete recovery audit data.

## Reproduction Assessment

Yes, for the review findings. Source inspection of the PR diff and current main shows the restore predicate trusts overwritten user.moderation rows, not_found is a non-terminal VT state, and the audit records only the first batch count.

## Solution Assessment

No. The recovery capability is useful, but this PR needs safer restore eligibility, unresolved-scan handling, and cumulative audit accounting before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat not_found VT results as unresolved:** `convex/skills.ts:4147-4148`
  - body: The restore path only keeps missing, pending, or loading VT results in the scan queue. Current main maps `vtAnalysis.status === "not_found"` to `scanner.vt.pending`, so a lifted skill with that status can become active and leave scanning; include `not_found` or reuse the existing scanner-state helper.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4128-4129`
  - body: This predicate treats `hidden/user.moderation` as proof that the hold hid the skill, but the hold batch overwrites every non-malicious owned skill to that reason and timestamp. Existing manual or quality-hidden rows can therefore be published when the hold is lifted; preserve prior state or avoid restoring rows that were already locked.
  - confidence: 0.82
- **[P2] Report all restored skills in the lift audit:** `convex/users.ts:813`
  - body: The audit metadata records only the first restore batch's `restoredCount`. When continuation batches are scheduled, later restored skills are omitted from the returned count and `user.moderation.lift` audit trail, which makes large recoveries incomplete.
  - confidence: 0.86
