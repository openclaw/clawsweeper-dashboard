## Summary

Keep open. Current main still lacks a moderation-hold lift API, the maintainer explicitly reopened this PR for reinvestigation, and the proposed restore path has source-visible security and audit issues that should be fixed before merge.

## What This Changes

The PR adds public and internal Convex admin mutations to clear user moderation holds, batch-restore eligible owned skills, write audit logs, add tests, and document the admin recovery command.

## Best Possible Solution

Land a maintainer-reviewed hold-lift implementation that clears user hold fields, restores only skills genuinely eligible after that hold, keeps malicious/manual/quality/soft-deleted/unresolved-scan states protected, records complete audit data, and documents the admin CLI workflow.

## Reproduction Assessment

Yes, for the review findings: source inspection shows the current hold batch overwrites non-malicious skill reasons, current scanner logic treats VT not_found as pending, and the PR records only the first restore batch count.

## Solution Assessment

No. The recovery capability is useful, but this PR needs safer restore eligibility, unresolved-scan handling, and cumulative audit accounting before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep not_found VT results in scan queue:** `convex/skills.ts:4147-4148`
  - body: The restore path only keeps missing, pending, or loading VT results hidden. Main classifies `vtAnalysis.status === "not_found"` as `scanner.vt.pending`, so a lifted skill with that status can become active/restored and leave the VT queue; include `not_found` or reuse the existing scanner reason helper before publishing.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4128-4129`
  - body: The restore predicate trusts `hidden/user.moderation`, but the hold batch rewrites every non-malicious owned skill to that reason, including rows already hidden for quality or manual review. Lifting a user hold can therefore publish content that was quarantined before the hold; preserve prior state or skip those rows.
  - confidence: 0.82
- **[P2] Report cumulative restore counts in audits:** `convex/users.ts:813`
  - body: The caller records only the first restore batch's `restoredCount`. When continuation batches run, later restored skills are omitted from returned metadata and the `user.moderation.lift` audit log; aggregate the count or defer the final audit until all batches complete.
  - confidence: 0.86
