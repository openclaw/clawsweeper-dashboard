## Summary

Keep open. Current main has improved parts of the publisher bootstrap and digest pagination path, but the API-token `whoami` path still accepts a handle-less user and the CLI still reports that as `unknown`; ClawHub cleanup policy also keeps issues open unless maintainers handle them explicitly.

## What This Changes

The issue reports that `clawhub login` succeeds but resolves the account as `unknown`, leaving the personal publisher missing and publish failing for the reporter.

## Best Possible Solution

Enforce a canonical GitHub handle and active personal publisher before token issuance, `whoami`, and publish can appear successful, then repair affected live account rows and add regression coverage.

## Reproduction Assessment

Yes for the source-level identity failure: a valid API token tied to a user whose `handle` is undefined reaches `whoamiV1Handler`, returns `null`, and the CLI prints `unknown`. I did not reproduce the reporter's exact production row or the exact current `Personal publisher not found` failure from the checkout.

## Solution Assessment

No, current main is only a partial solution because fallback publisher listing and cursorized digest sync do not make token creation or `whoami` enforce a valid GitHub handle and publisher invariant. The maintainable fix is to enforce the invariant in the auth/token path and pair it with ops repair for existing rows.
