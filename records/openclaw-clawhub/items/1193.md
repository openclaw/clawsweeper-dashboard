## Summary

Keep open. Current main still has a source-level path where a valid API-token user with a missing handle can be reported as `unknown`, and publishing still depends on personal-publisher repair; ClawHub cleanup policy also keeps issues open for maintainer handling.

## What This Changes

The issue reports successful ClawHub login followed by `whoami` resolving to `unknown` and publishing failing because the user's personal publisher is missing.

## Best Possible Solution

Make account normalization a hard invariant before token creation, `whoami`, and publish success, then repair existing affected production user/publisher rows and cover null or stale-handle accounts with regression tests.

## Reproduction Assessment

Yes for the source-level failure mode: a valid API token tied to a user whose `handle` is missing can produce a null-handle `whoami` response that the CLI displays as `unknown`. I did not reproduce the reporter's exact production row from the checkout.

## Solution Assessment

No current implementation fully solves the reported issue. Current main improves publisher healing and digest pagination, but the maintainable fix is a hard auth/publisher invariant plus ops repair for existing bad rows.
