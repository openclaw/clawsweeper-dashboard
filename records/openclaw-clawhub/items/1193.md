## Summary

Keep open. Current `main` still permits a valid API-token user with no canonical handle to appear as `unknown`, and the publish path still depends on personal-publisher repair; ClawHub cleanup policy also keeps issues open unless maintainers explicitly resolve them.

## What This Changes

The issue reports that `clawhub login` succeeds, but `whoami` resolves to `unknown` and publishing fails with `Personal publisher not found`.

## Best Possible Solution

Guarantee that every API-token user has a canonical GitHub handle and active personal publisher before login, `whoami`, or publish can appear successful, then repair affected production rows and add regression coverage for null or wrong-handle users.

## Reproduction Assessment

Yes for the source-level failure mode: a valid API token tied to a user with no handle can produce a null-handle `whoami` response that the CLI displays as `unknown`. The reporter's exact live production row was not reproducible from the checkout.

## Solution Assessment

No current implementation fully solves the issue. The narrow maintainable direction is an auth/token/publisher invariant plus ops repair for existing bad rows, not closing based on the digest pagination improvement alone.
