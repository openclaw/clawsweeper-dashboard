## Summary

Keep open. This is an open ClawHub issue, and current `main` still has a source-level path where an API-token user with no canonical handle is accepted, reported by the CLI as `unknown`, and publishing still relies on personal-publisher repair that cannot verify or repair the reporter's production account state from the repository alone.

## What This Changes

The issue reports that CLI login succeeds but authenticated identity resolves to an unknown user and publishing fails because the personal publisher is missing.

## Best Possible Solution

Guarantee that API-token login, `whoami`, and publish cannot appear successful until the user has a canonical GitHub handle and an active personal publisher, then repair affected production rows and add regression coverage.

## Reproduction Assessment

Yes for the source-level unknown-identity path: a valid API token whose user has no handle can flow through `whoamiV1Handler`, pass the client schema, and be displayed as `unknown` by the CLI. The reporter's exact production row and the full publish failure were not reproduced from the checkout.

## Solution Assessment

No, current main is not a complete solution because handle and publisher repair are background/best-effort around token creation and `whoami`. The safer direction is to make these invariants synchronous at login/whoami/publish boundaries and pair that with ops repair for existing rows.
