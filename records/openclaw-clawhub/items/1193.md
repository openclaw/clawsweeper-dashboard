## Summary

Keep open. This is an open ClawHub issue, and current main still has a source-level path where a valid API-token user with a missing handle is accepted by `whoami` and shown as `unknown`, while publish still depends on personal-publisher repair and live account state.

## What This Changes

The issue reports successful CLI login followed by `whoami` resolving to `unknown` and skill publish failing with `Personal publisher not found` for the reporter's GitHub account.

## Best Possible Solution

Enforce a canonical GitHub handle and active personal publisher before API-token login, `whoami`, or publish can report success, then repair affected production rows and add regression coverage for null or wrong-handle users.

## Reproduction Assessment

Yes for the source-level path: a valid API-token user whose `users.handle` is null can pass `/api/v1/whoami`, satisfy the CLI schema, and display as `unknown`. No live reproduction was done for the reporter's exact production rows or publish attempt.

## Solution Assessment

No complete source-only fix is established yet. Current main has useful pagination and personal-publisher healing mitigations, but the durable solution needs the identity and publisher invariants enforced at token, `whoami`, and publish boundaries plus production account repair.
