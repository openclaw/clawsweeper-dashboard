## Summary

Keep open. Current main still has a source-level path where a valid API-token user with a missing `handle` is accepted by `whoami` and displayed by the CLI as `unknown`; the related publisher-pagination repair is present, but the reporter's live handle/publisher state is not proven repaired, and ClawHub cleanup policy keeps issues open for maintainer handling.

## What This Changes

The issue asks for login, `whoami`, web upload, and publish to resolve the authenticated GitHub handle and personal publisher instead of returning `unknown` or `Personal publisher not found`.

## Best Possible Solution

Enforce that every authenticated API-token user has a canonical GitHub handle and active personal publisher before token creation, `whoami`, or publish can report success, then repair existing affected production rows and add regression coverage.

## Reproduction Assessment

Yes for the source-level identity path: a valid API token tied to a user whose `handle` is null flows through `/api/v1/whoami` and the CLI displays `unknown`. The reporter's exact production row and final publisher failure were not reproduced from the checkout.

## Solution Assessment

No, current main is not a complete solution because token creation and `whoami` do not synchronously enforce handle/publisher invariants. The safer direction is an invariant in the auth-token and publish paths plus explicit ops repair for affected rows.
