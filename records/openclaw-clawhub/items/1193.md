## Summary

Keep open. Current main still has a source-level path where an API-token user with a missing handle is accepted, reported by the CLI as `unknown`, and publishing still depends on personal-publisher repair; the digest pagination error path appears improved but does not prove the reporter's identity/publisher state is fixed.

## What This Changes

The issue reports successful `clawhub login` followed by `whoami` returning `unknown` and publish failing because the account does not resolve to a valid GitHub handle or personal publisher.

## Best Possible Solution

Guarantee that API-token login, `whoami`, and publish cannot appear successful until the user has a canonical GitHub handle and an active personal publisher, then repair existing bad production rows and add regression coverage.

## Reproduction Assessment

Yes for the source-level failure mode: a valid API token tied to a user whose `handle` is null flows through `whoamiV1Handler`, the client schema accepts it, and the CLI displays `unknown`. The reporter's exact production row was not reproduced from the checkout.

## Solution Assessment

No, current main is not a complete solution because background repair exists but token creation and `whoami` do not synchronously enforce handle and publisher invariants. The maintainable direction is a code invariant plus an ops repair for existing affected users.
