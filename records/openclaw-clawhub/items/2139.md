## Summary

Keep open. Current main does not implement this PR, and the proposed full-text path has a blocking correctness issue for multi-token queries plus no after-fix real behavior proof beyond test/checklist claims.

## What This Changes

The PR adds Convex full-text search indexes for skill display names and slugs, queries them alongside existing prefix indexes, and adds search helper tests.

## Best Possible Solution

Use an indexed full-text or token lookup path for non-first-token recall, but keep the existing all-query-token filtering semantics before merging candidates into search results and include real search output for representative queries.

## Reproduction Assessment

Yes, from source inspection: current main's direct indexed path only covers full-prefix and first-token prefix matches, while the fallback is bounded to recent digest windows. I did not run a live Convex deployment search, so this is source-reproducible rather than runtime-reproduced here.

## Solution Assessment

No. Full-text indexes are a reasonable primitive, but the PR should not merge raw any-word full-text results directly into direct matches; it should preserve the existing all-token matching contract before hydration/ranking.
