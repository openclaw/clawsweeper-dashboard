## Summary

Keep open: current main still omits package `openclaw.environment` env/config metadata from the package ClawScan prompt requirements path, and the related implementation PR was closed unmerged.

## What This Changes

Request that package ClawScan prompts include declared `package.json` `openclaw.environment` env vars and config paths, or avoid reporting them as none when `environment:declared` is present.

## Best Possible Solution

Add a sanitized package-environment-to-prompt mapper that forwards only declared env/config requirements into the existing prompt metadata shape, preserves existing requirements, and adds regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows package capability extraction reads `openclaw.environment`, while package LLM evaluation does not supply that metadata to the prompt path that renders env vars and config paths.

## Solution Assessment

Yes. The narrow maintainable fix is to normalize declared environment fields into prompt requirements while avoiding arbitrary package `openclaw` metadata forwarding.

## Repair Work Prompt

Fix https://github.com/openclaw/clawhub/issues/2013. On current main, package capability extraction derives `environment:declared` from `package.json` `openclaw.environment`, but `evaluatePackageReleaseWithLlm` does not map declared env/config metadata into the prompt requirements path that `assembleEvalUserMessage` reads, so ClawScan can say env vars/config paths are none. Add a conservative, sanitized mapper for package `openclaw.environment` into existing prompt metadata fields, preserving existing valid requirements; do not spread the full untrusted `openclaw` object or forward unrelated fields such as arbitrary `install`. Cover optional env vars, provider-pattern env vars, config paths, requirements merging, and malformed unrelated `openclaw` fields not crashing prompt assembly. Do not change package publish validation, capability tag derivation, or public package APIs beyond the prompt context unless tests prove it is necessary. Update specs only if the repair formalizes new environment aliases.
