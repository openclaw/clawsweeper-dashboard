## Summary

Keep open: this is still source-reproducible on current main, and ClawHub cleanup policy keeps issues open. Package releases can advertise `environment:declared`, but package ClawScan evaluation still does not pass parsed `package.json` `openclaw.environment` env/config metadata into the prompt fields that report requirements.

## What This Changes

The issue asks ClawScan package reviews to include declared `openclaw.environment` env vars and config paths in the LLM review context.

## Best Possible Solution

Add a sanitized package-environment-to-prompt mapper that forwards only normalized env/config declarations, preserves existing requirements, and covers malformed/unrelated package metadata with regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows package `environment:declared` tags are derived from `openclaw.environment`, while the package LLM eval context never supplies that metadata to the prompt fields that print env vars and config paths.

## Solution Assessment

Yes. The narrow maintainable fix is to map sanitized `openclaw.environment` env/config declarations into the existing `metadata.openclaw` prompt shape without forwarding the full untrusted package metadata object.

## Repair Work Prompt

Fix the ClawScan package prompt mismatch where package releases with `package.json` `openclaw.environment` metadata can produce `environment:declared` tags but still show env vars/config paths as `none` in package LLM review context. Implement a sanitized mapper in the package LLM eval path, likely in `convex/llmEval.ts`, that reads package metadata from the stored/extracted `package.json` or release files and forwards only normalized env/config prompt fields into the existing `metadata.openclaw` shape consumed by `assembleEvalUserMessage`. Preserve any valid existing requirements instead of overwriting `requires.env`, `bins`, or `anyBins`; do not spread the full untrusted `openclaw` object into prompt assembly. Add focused regression coverage for required and optional env vars, config paths, preservation of existing requirements, and malformed unrelated metadata. Use https://github.com/openclaw/clawhub/pull/2009 only as a reference for the intended behavior and known pitfalls, not as a patch to copy blindly. Update a focused spec note under `specs/` only if the accepted package `openclaw.environment` field aliases or invariants are clarified by the fix.
