## Summary

Keep open: current main still has the package ClawScan prompt mismatch, and ClawHub cleanup policy keeps issues open until a fix lands.

## What This Changes

ClawScan should include declared package `openclaw.environment` env/config metadata in package review prompt context, or avoid saying none when that metadata is declared.

## Best Possible Solution

Add a sanitized package-environment-to-prompt mapper that forwards only declared env/config requirements into the existing ClawScan prompt shape, preserves existing requirements, and covers malformed metadata with regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows package `environment:declared` tags are derived from `openclaw.environment`, while package LLM evaluation never supplies that metadata to the prompt fields that render env vars and config paths.

## Solution Assessment

Yes. The narrow maintainable fix is to normalize only declared env/config fields into `metadata.openclaw` requirements and preserve existing requirements, not to forward the full package `openclaw` object.

Add a sanitized mapper in the package LLM evaluation path that reads `release.extractedPackageJson` or the package.json artifact, normalizes declared env/config fields into the existing prompt shape (`metadata.openclaw.envVars`, `primaryEnv`, and/or `requires.env`/`requires.config` as appropriate), preserves any existing valid requirements, and does not forward unrelated or malformed `openclaw` fields such as arbitrary `install` values.
