## Summary

Keep open: current main still has the reported mismatch, because package review metadata does not pass parsed `package.json` `openclaw.environment` data into the ClawScan prompt path even though package capability tags are derived from that block.

## What This Changes

The issue asks ClawScan package reviews to include declared `openclaw.environment` env vars and config paths, or avoid claiming none are declared when environment metadata exists.

## Best Possible Solution

Add a sanitized package-environment-to-prompt mapper that forwards declared env vars/config paths into the existing ClawScan requirements context while preserving existing requirements and adding regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main derives `environment:declared` tags from package metadata but does not feed `openclaw.environment` into the prompt fields that print env vars and config paths.

## Solution Assessment

Yes. The maintainable fix is narrow: map only sanitized `openclaw.environment` env/config data into the existing prompt metadata shape and cover required, optional, malformed, and existing-requirements cases.

## Repair Work Prompt

Fix the package ClawScan prompt metadata gap reported in issue #2013. In `convex/llmEval.ts`, parse the release package.json during `evaluatePackageReleaseWithLlm` and pass a sanitized `metadata.openclaw` object into `assembleEvalUserMessage` containing only normalized prompt fields derived from `openclaw.environment` (required/optional env vars, primary credential when appropriate, and config paths). Do not spread or forward the full untrusted `openclaw` object. Preserve any existing `openclaw.requires` fields when adding config paths, and keep malformed unrelated metadata from crashing prompt assembly. Add focused tests in `convex/llmEval.test.ts` and/or `convex/lib/securityPrompt.test.ts` covering required env vars, optional env vars, config paths, existing `requires.env`/`bins` preservation, and malformed `install`/unrelated metadata. Do not change scanner verdict policy or package publish validation semantics.
