## Summary

Keep open: current main still appears to omit declared package `openclaw.environment` env/config metadata from the ClawScan package review prompt, even though package extraction derives environment capability tags from the same metadata. This is source-reproducible and remains actionable; the earlier implementation PR was closed unmerged, and the related open ClawScan refactor is not a closing fix for this issue.

## What This Changes

Include declared package `openclaw.environment` env var and config path metadata in ClawScan package review prompt requirements, or avoid saying none when environment metadata is declared.

## Best Possible Solution

Add a sanitized mapper from package `openclaw.environment` into the existing prompt requirements shape, preserving valid existing requirements and forwarding only normalized env/config fields with regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows package registry extraction reads `openclaw.environment`, while package LLM evaluation does not pass those declarations into the prompt requirements path that renders env vars and config paths.

## Solution Assessment

Yes. The narrow maintainable fix is to normalize only declared package environment fields into prompt requirements, rather than forwarding arbitrary package `openclaw` metadata or changing scanner policy.
