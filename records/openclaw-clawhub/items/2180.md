## Summary

Keep this PR open: the visual refresh is not on current main, the linked screenshot proof is useful, but the patch introduces an unrestricted server-side avatar fetch path that should block merge until it is constrained.

## What This Changes

The PR refreshes dynamic OG images for skills, plugins/packages, publisher profiles, and soul pages, adding plugin/profile OG routes and metadata wiring.

## Best Possible Solution

Keep the visual refresh as a PR, but land it only after the avatar image path is constrained to trusted sources or removed, the branch is rebased, and the visual direction is accepted.

## Reproduction Assessment

Yes for the introduced blocker: source inspection shows any new OG route can pass a public avatar URL into server-side fetchImageDataUrl. This is a feature PR rather than a reported current-main bug.

## Solution Assessment

No as currently proposed; the visual renderer direction is plausible, but arbitrary server-side avatar fetching should be replaced with an allowlisted/proxied image path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Do not fetch arbitrary avatar URLs server-side:** `server/og/fetchImageDataUrl.ts:8-14`
  - body: The public OG routes pass `avatar` query values into this helper, but the helper accepts any `http` or `https` URL and fetches it from the server. A request can make production reach private or metadata endpoints before rendering the PNG, and image responses can be embedded back into the output. Restrict this to trusted avatar origins or a safe image proxy that blocks private networks and redirects.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff introduces an SSRF-prone public OG avatar fetch path that needs to be constrained before merge.

Concerns:

- **[high] Unrestricted server-side avatar fetch:** `server/og/fetchImageDataUrl.ts:8`
  - body: fetchImageDataUrl accepts arbitrary http/https URLs, and the public skill/plugin/profile OG routes feed it a query-controlled avatar URL. That lets external callers make the server request untrusted hosts, including private network targets, during image rendering.
