## Summary

Keep this PR open for follow-up: the visual OG refresh is not on current main, but the current branch still has a blocking server-side avatar fetch issue and needs a rebase onto the canonical publisher profile route before visual-direction approval.

## What This Changes

The branch refreshes skill OG rendering and adds plugin/publisher dynamic OG routes, shared SVG renderers, metadata helpers, asset copying, a watermark asset, and tests.

## Best Possible Solution

Rebase onto current main, move publisher metadata to the canonical /user/$handle surface, constrain OG avatar loading to trusted public image sources or a safe proxy, then complete the requested visual-direction review.

## Reproduction Assessment

Not applicable for the visual feature itself. The blocking security issue is source-reproducible from the PR head because public OG routes forward query/meta avatar URLs into a helper that fetches any HTTP(S) URL.

## Solution Assessment

No. The renderer direction may be useful, but this implementation should not merge until the server-side avatar fetch is constrained and the publisher metadata is rebased onto the current canonical profile route.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict server-side avatar fetches:** `server/og/fetchImageDataUrl.ts:8-11`
  - body: The new public OG routes pass the `avatar` query value or stored owner/profile image URL into this helper, but the helper accepts any `http` or `https` URL before fetching it. That makes `/og/skill`, `/og/plugin`, and `/og/profile` a server-side fetch path to private or metadata endpoints that return an image content type, so avatar loading needs a trusted-host/public-IP guard or a safe proxy boundary before merge.
  - confidence: 0.89
- **[P2] Use the canonical publisher profile route:** `src/lib/og.ts:208`
  - body: Current main redirects `/p/$handle` to `/user/$handle`, but this helper still returns `/p` as the publisher canonical URL and the PR adds publisher head metadata to the legacy route. After rebasing, the visible publisher page would miss the new OG metadata and the canonical URL would point at a redirect; move this surface to `/user/$handle`.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The PR introduces an unrestricted server-side fetch path for public OG avatar URLs.

Concerns:

- **[high] Unrestricted OG avatar fetch can become SSRF:** `server/og/fetchImageDataUrl.ts:8`
  - body: `fetchImageDataUrl` only checks for `http` or `https` before fetching, and the new public OG routes pass requester-controlled or stored avatar URLs into it. A caller could make the server fetch private network or metadata URLs if they return an image-like response.
  - confidence: 0.88
