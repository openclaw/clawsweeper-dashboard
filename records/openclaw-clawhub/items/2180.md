## Summary

Keep open: current main does not implement the plugin/publisher OG refresh, and this PR is not merge-ready because it adds an unrestricted server-side avatar fetch path, targets publisher metadata at the legacy profile route, and is currently merge-conflicting.

## What This Changes

The PR refreshes dynamic OG image rendering for skills, plugins/packages, and publisher profiles with shared SVG renderers, new OG routes, metadata helpers, assets, and tests.

## Best Possible Solution

Land the visual refresh only after rebasing onto the /user route model and constraining avatar loading to trusted public-image sources or a safe proxy boundary.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the blocking security finding by source inspection: the PR diff shows public OG routes passing query or stored avatar URLs into an unrestricted HTTP(S) fetch helper. The visual refresh itself is a feature, so bug reproduction is otherwise not applicable.

## Solution Assessment

Is this the best way to solve the issue? No. The renderer direction is useful, but the implementation should not merge until avatar fetching is constrained, publisher metadata targets the canonical route, and the requested visual-direction review is complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict server-side avatar fetches:** `server/og/fetchImageDataUrl.ts:8-11`
  - body: The new public OG routes pass requester-controlled or stored avatar URLs into this helper, but the helper accepts any http or https URL before fetching it. That creates a server-side fetch path to private or metadata endpoints if they return an image-like response, so avatar loading needs a trusted-host/public-IP guard or safe proxy boundary before merge.
  - confidence: 0.89
- **[P2] Use the canonical publisher profile route:** `src/lib/og.ts:208`
  - body: Current main redirects /p/$handle to /user/$handle, but this helper still returns /p as the publisher canonical URL and the PR adds publisher head metadata to the legacy route. After rebasing, the visible publisher page would miss the new OG metadata and the canonical URL would point at a redirect.
  - confidence: 0.84
