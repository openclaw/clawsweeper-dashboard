## Summary

Keep this PR open, but do not merge it as-is: the visual OG refresh has real screenshot proof, yet the branch introduces an unsafe server-side avatar fetch path and is currently merge-conflicting with main’s publisher route changes.

## What This Changes

The PR refreshes dynamic OG images for skills, plugins/packages, publisher profiles, and related metadata routes.

## Best Possible Solution

Rebase onto current main, apply publisher metadata to the active profile route intentionally, and harden or remove remote avatar fetching before visual approval.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR accepts arbitrary http(s) avatar URLs and fetches them from the OG server path. I did not run a live SSRF probe because this was a read-only review.

## Solution Assessment

No; the visual refresh may be the right product direction, but the current implementation needs rebase work and a safer avatar-fetch boundary first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Constrain server-side avatar fetching:** `server/og/fetchImageDataUrl.ts:8-11`
  - body: The new OG routes pass `avatar` query params and profile/owner image URLs into this helper, which accepts any http(s) URL and fetches it from the server. That turns the public OG endpoint into a blind SSRF probe against internal or link-local services; timeout and byte limits only constrain response handling. Remove the public avatar override or restrict fetching to trusted image hosts while blocking private/link-local addresses before issuing the request.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The PR introduces a server-side fetch of attacker-controlled avatar URLs without host or private-network restrictions.

Concerns:

- **[high] Arbitrary server-side avatar fetch:** `server/og/fetchImageDataUrl.ts:8`
  - body: The public OG routes can pass query-provided or profile-provided URLs into `fetchImageDataUrl`, which accepts both http and https URLs and fetches them server-side. This creates SSRF exposure unless remote images are limited to trusted hosts and private/link-local targets are rejected before fetch.
