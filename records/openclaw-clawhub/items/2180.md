## Summary

This PR should stay open because the OG refresh is not on current main and the branch still has merge-blocking issues: unrestricted server-side avatar fetching in public OG routes and publisher metadata wired to the legacy `/p/$handle` route instead of the canonical `/user/$handle` surface.

## What This Changes

The branch refreshes dynamic OG rendering for skills, plugins/packages, and publisher profiles with new SVG renderers, OG routes, metadata helpers, asset copying, and tests.

## Best Possible Solution

Rebase onto current main, constrain OG avatar loading to trusted public image sources or another safe proxy boundary, put publisher metadata on `/user/$handle`, and then finish the requested visual-direction review.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the blocking security finding by source inspection: the PR diff shows public OG routes passing query or stored avatar URLs into an unrestricted HTTP(S) fetch helper. The visual refresh itself is a feature, so bug reproduction is otherwise not applicable.

## Solution Assessment

Is this the best way to solve the issue? No. The renderer direction may be useful, but the implementation should not merge until avatar fetching is constrained, publisher metadata targets the canonical route, and the requested visual direction review is complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict server-side avatar fetches:** `server/og/fetchImageDataUrl.ts:8-11`
  - body: The new public OG routes pass requester-controlled or stored avatar URLs into this helper, but the helper accepts any `http` or `https` URL before fetching it.   - confidence: 0.89

  - confidence: 0.83
