## Summary

This PR should stay open: current main does not include the refreshed OG image implementation, and the branch still has merge conflicts plus a concrete SSRF-class avatar-fetch issue before merge.

## What This Changes

The PR refreshes dynamic OG images for skills, plugins/packages, and publisher profiles by adding shared SVG renderers, new OG routes, metadata helpers, asset copying, and tests.

## Best Possible Solution

Rebase onto current main, constrain OG avatar loading to a trusted public-image boundary, move publisher metadata to the canonical /user/$handle surface, then finish the requested visual-direction review.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the blocking security finding by source inspection: public OG routes pass query or stored avatar URLs into an unrestricted HTTP(S) server-side fetch helper. The visual refresh itself is a feature, so bug reproduction is otherwise not applicable.

## Solution Assessment

Is this the best way to solve the issue? No. The renderer direction may be useful, but this implementation should not merge until avatar fetching is constrained, publisher metadata targets the canonical route, and the visual-direction question is resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict server-side avatar fetches:** `server/og/fetchImageDataUrl.ts:8-11`
  - body: The new public OG routes pass requester-controlled or stored avatar URLs into this helper, but it accepts any `http` or `https` URL before fetching it. That creates a server-side fetch path to private or metadata endpoints if they return an image-like response, so avatar loading needs a trusted-host/public-IP guard or safe proxy boundary before merge.
  - confidence: 0.89
- **[P2] Use the canonical publisher profile route:** `src/lib/og.ts:208`
  - body: Current main redirects `/p/$handle` to `/user/$handle`, but this helper still returns `/p` as the publisher canonical URL and the PR adds publisher head metadata to the legacy route. After rebasing, the visible publisher page would miss the new OG metadata and the canonical URL would point at a redirect.
  - confidence: 0.83
