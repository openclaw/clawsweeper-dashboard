## Summary

Keep open: the PR has useful visual proof, but it introduces a public server-side image fetch path that accepts arbitrary avatar URLs, which should be fixed before merge.

## What This Changes

The PR refreshes dynamic OG rendering for skills, plugins/packages, and publisher profiles, adds new OG routes and metadata wiring, copies a watermark asset, and updates related tests.

## Best Possible Solution

Keep the new OG image surfaces, but merge only after avatar fetching is made safe and the requested visual direction review is resolved.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable to the requested visual feature; yes for the blocking defect from source inspection, because the PR head routes read `query.avatar` and pass it to a helper that fetches arbitrary http/https URLs.

## Solution Assessment

Is this the best way to solve the issue? No: the OG refresh is directionally reasonable, but the server should not fetch arbitrary caller-supplied avatar URLs from public image routes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Constrain server-side avatar fetches:** `server/og/fetchImageDataUrl.ts:8-11`
  - body: `avatar` is a public query param on the OG routes, and all three routes pass it into this helper. Since the helper accepts any `http:` or `https:` URL and fetch follows redirects, callers can make the server request internal/private network URLs. Remove the public override or restrict fetching to trusted avatar origins and block private, loopback, and metadata targets before fetching.
  - confidence: 0.91
