## Summary

Keep open: the PR adds useful inspected OG image work, but it has blocking review findings around arbitrary server-side avatar fetching and a stale publisher-profile route change that conflicts with current main’s canonical `/user/$handle` routing.

## What This Changes

The PR refreshes dynamic OG image generation for skills, plugins/packages, and publisher profiles, adding shared renderers, new OG routes, metadata wiring, tests, and a watermark asset.

## Best Possible Solution

Harden OG avatar fetching before any network request and rebase publisher OG metadata onto `/user/$handle` while preserving `/p/$handle` as a legacy redirect.

## Reproduction Assessment

Yes for the review blockers: source inspection shows public OG routes pass `avatar` into an unrestricted server fetch helper, and current main/source specs show publisher profiles are canonical at `/user/$handle` while the PR reintroduces `/p/$handle` as the metadata route.

## Solution Assessment

No; the visual refresh direction may be useful, but this patch needs SSRF hardening and a current-main route rebase before it is the maintainable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Block private avatar fetch targets:** `server/og/fetchImageDataUrl.ts:8-11`
  - body: The public OG routes pass `avatar` query values into this helper, but the helper accepts any `http:` or `https:` URL and immediately fetches it. That lets any request to `/og/skill`, `/og/plugin`, or `/og/profile` make the server reach localhost, link-local, or internal hosts. Validate resolved/redirected targets against public image hosts or drop user-supplied avatar URLs before fetching.
  - confidence: 0.96
- **[P2] Keep publisher OG metadata on the canonical route:** `src/routes/p/$handle.tsx:32-54`
  - body: Current main makes `/p/$handle` a legacy redirect to `/user/$handle`, and the routing spec says `/user/<handle>` is canonical. This branch adds the publisher profile component/head back under `/p/$handle`, while the canonical `/user/$handle` route would not get the new OG metadata. Move the publisher head/meta to the canonical route and keep `/p` redirecting.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces server-side fetching of user-controlled avatar URLs without private-network or redirect-target controls.

Concerns:

- **[high] Arbitrary OG avatar URL fetch:** `server/og/fetchImageDataUrl.ts:8`
  - body: `fetchImageDataUrl` only checks the URL protocol before fetching, and public OG handlers feed it the `avatar` query parameter. This creates an SSRF surface against internal addresses unless targets and redirects are constrained to trusted public image origins or otherwise blocked before the request.
