## Summary

Keep open: this is an active feature PR that is not already on main, and the diff introduces a blocking security issue in the new avatar image fetch path despite adequate visual proof.

## What This Changes

The PR refreshes skill OG rendering and adds plugin/package and publisher/profile dynamic OG image routes, metadata tags, shared SVG renderers, watermark assets, and tests.

## Best Possible Solution

Land the OG refresh only after avatar fetching is limited to trusted/proxied image sources or omitted for untrusted URLs, then get the requested maintainer visual-direction signoff.

## Reproduction Assessment

Yes, for the review finding: source inspection shows the new public `/og/skill`, `/og/plugin`, and `/og/profile` handlers pass query-supplied `avatar` URLs to a helper that fetches any http(s) URL. I did not execute a live SSRF probe, but the vulnerable request path is explicit in the diff.

## Solution Assessment

No. The OG refresh direction is plausible, but the implementation should not directly fetch arbitrary avatar URLs from public image routes; use a trusted image proxy/allowlist, reject private and loopback targets before fetch, or omit avatars when the source is untrusted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Block arbitrary avatar fetches in OG routes:** `server/og/fetchImageDataUrl.ts:7-14`
  - body: The new helper is called from public OG endpoints with query-supplied `avatar` values, but it accepts any http(s) URL and performs the server-side fetch before any host safety check. A request such as `/og/skill?...&avatar=http://169.254.169.254/...` can make the server reach internal/private hosts, even if the response is later rejected by content type. Please restrict this path to trusted/proxied image hosts or reject private, loopback, and link-local targets before fetching.
  - confidence: 0.91
