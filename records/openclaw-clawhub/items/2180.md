## Summary

Keep open: current main does not contain the requested OG refresh, but this patch is not merge-ready because it adds an unrestricted public server-side image fetch path and targets publisher metadata at the legacy redirect route.

## What This Changes

The branch refreshes skill OG images and adds plugin/package and publisher OG image routes, metadata helpers, local assets, and route tests.

## Best Possible Solution

Rebase onto current main, remove or constrain arbitrary avatar fetching to a trusted public-image/proxy boundary, move publisher OG metadata and canonical URLs to /user/$handle, and then complete the requested visual-direction review.

## Reproduction Assessment

Yes for the blocking security finding by source inspection: public OG routes pass query or stored avatar URLs into a helper that fetches arbitrary http(s) URLs. The visual refresh itself is a feature, so bug reproduction is otherwise not applicable.

## Solution Assessment

No. The renderer direction may be useful, but the current implementation should not merge until avatar fetching is constrained, publisher metadata targets the canonical route, and the visual direction is approved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict server-side avatar fetches:** `server/og/fetchImageDataUrl.ts:8-11`
  - body: The new public OG routes pass requester-controlled or stored avatar URLs into this helper, but the helper accepts any `http:` or `https:` URL before fetching it. That creates a server-side fetch path to private or metadata endpoints if they return an image-like response, so avatar loading needs a trusted-host/public-IP guard or safe proxy boundary before merge.
  - confidence: 0.9
- **[P2] Put publisher OG metadata on the canonical route:** `src/lib/og.ts:208`
  - body: Current main makes `/user/$handle` the canonical publisher profile route and `/p/$handle` a redirect, but this helper still returns `/p/$handle` as `og:url`. After rebase, the visible canonical publisher page would miss the refreshed metadata and social crawlers would see a redirect as the canonical URL.
  - confidence: 0.84

## Security Review

Status: needs_attention
