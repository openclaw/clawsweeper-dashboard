## Summary

Keep open: current main does not implement the plugin/publisher OG refresh, and the PR still needs work before merge because the new public avatar fetch path is unrestricted, publisher metadata targets the legacy `/p` route after main switched to `/user`, the branch is merge-conflicting, and the body still requests visual-direction approval.

## What This Changes

The PR refreshes dynamic OG rendering for skills, plugin packages, and publisher profiles, adding shared SVG renderers, plugin/profile OG endpoints, metadata helpers, asset copying, a watermark asset, and tests.

## Best Possible Solution

Rebase onto current main, constrain OG avatar loading to trusted public image sources or a safe proxy boundary, move publisher metadata and canonical URLs to `/user/$handle`, then complete the requested visual review.

## Reproduction Assessment

Not applicable for the visual feature itself. The blocking security finding is source-reproducible on the PR head because public OG routes pass query or stored avatar URLs into a helper that fetches any `http` or `https` URL.

## Solution Assessment

No. The renderer direction may be useful, but this implementation should not merge until server-side avatar fetching is constrained and publisher metadata targets the current canonical profile route.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restrict server-side avatar fetches:** `server/og/fetchImageDataUrl.ts:8-11`
  - body: The new public OG routes pass the `avatar` query value or stored owner/profile image URL into this helper, but the helper accepts any `http` or `https` URL before fetching it. That makes `/og/skill`, `/og/plugin`, and `/og/profile` a server-side fetch path to private or metadata endpoints that return an image content type, so avatar loading needs a trusted-host/public-IP guard or safe proxy boundary before merge.
  - confidence: 0.89
- **[P2] Use the canonical publisher profile route:** `src/lib/og.ts:208`
  - body: Current main redirects `/p/$handle` to `/user/$handle`, but this helper still returns `/p` as the publisher canonical URL and the PR adds publisher head metadata to the legacy route. After rebasing, the visible publisher page would miss the new OG metadata and the canonical URL would point at a redirect; move this surface to `/user/$handle`.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The PR introduces an unrestricted server-side fetch path for public OG avatar URLs.
