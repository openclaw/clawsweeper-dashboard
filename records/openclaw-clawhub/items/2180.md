Keep open: this is a useful feature PR, but it is not merge-ready because the current head is merge-conflicting, the PR explicitly asks for visual-direction approval, and the diff introduces a public unrestricted server-side avatar fetch path.

## What This Changes

The PR refreshes dynamic OG image rendering for skills, plugins/packages, and publisher profiles, adds plugin/profile OG routes, updates metadata helpers, and includes related tests/assets.

## Best Possible Solution

Rebase onto the canonical /user/$handle profile route, keep the refreshed OG surfaces only after visual approval, and constrain avatar loading to a trusted/public image boundary.

## Reproduction Assessment

Not applicable for the visual feature itself; yes for the blocking security concern, because the diff shows public OG routes forwarding avatar URLs into a helper that fetches any http/https URL.

## Solution Assessment

No. The design direction may be acceptable, but the current patch needs a rebase, SSRF-safe avatar loading, and the requested maintainer visual decision before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Restrict server-side avatar fetches:** `server/og/fetchImageDataUrl.ts:8-11`
  - body: The new OG routes pass `avatar` query values and owner/profile image URLs into `fetchImageDataUrl`, but the helper accepts any `http` or `https` URL before fetching it server-side.
