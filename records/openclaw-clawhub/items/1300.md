## Summary

Keep open. Current main still has the reported CLI serialization bug: transfer request, skill rename, and skill merge pre-stringify request bodies before handing them to an HTTP helper that serializes JSON bodies again, and the server handlers then read missing object fields. The related focused fix PRs were closed without merging, while the open ownership-transfer PR only overlaps part of this issue and is currently conflicting.

## What This Changes

Fix CLI transfer request, skill rename, and skill merge so they send plain object bodies to `apiRequest` instead of double-encoded JSON strings.

## Best Possible Solution

Land a focused CLI fix that restores `apiRequest` as the single JSON serialization boundary, updates transfer and ownership command tests to assert plain object payloads, and covers any shared HTTP hardening in both runtime paths.

## Reproduction Assessment

Yes, by source inspection on current main. The affected commands pass JSON strings, both HTTP transports serialize request bodies again, and the server expects object fields, so the reported required-field failures follow without a live authenticated run.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix; optional HTTP tolerance is reasonable only if both Node and Bun/curl serialization paths receive regression coverage.
