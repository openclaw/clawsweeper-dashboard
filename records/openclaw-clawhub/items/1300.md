## Summary

Keep open under the ClawHub issue policy, even though current main fixes the reported CLI payload double-serialization path and the complete fix is tagged for CLI v0.15.0. No repair lane is needed unless someone reproduces the same transfer, rename, or merge failures on v0.15.0 or newer.

## What This Changes

The issue reports that CLI transfer request, skill rename, and skill merge pre-stringify JSON request bodies before the shared HTTP helper serializes them again, causing required API fields to be missing.

## Best Possible Solution

Keep the v0.15.0 object-body implementation and only reopen engineering work if a fresh report reproduces the same commands failing on v0.15.0 or newer with live output.

## Reproduction Assessment

No. On current main, the three affected command handlers pass object bodies and the command tests assert object payloads, so the reported double-serialization path is no longer present in source.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix for the reported failures; shared HTTP string-body tolerance would be broader optional hardening, not required for these commands.
