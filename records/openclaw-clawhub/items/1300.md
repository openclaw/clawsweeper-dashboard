## Summary

Keep open. Current main still source-reproduces the CLI double-serialization bug for `transfer request`, `skill rename`, and `skill merge`: the commands pass JSON strings while both HTTP transports serialize request bodies again, and the server handlers then read object fields. The linked focused fix PRs were closed without merging, and the broader ownership-transfer PR remains open and conflicting rather than shipped.

## What This Changes

The issue requests fixing CLI transfer, rename, and merge commands so JSON POST bodies are not double-serialized before reaching the API.

## Best Possible Solution

Land a focused CLI repair that sends plain object bodies from the affected commands, updates regression tests to assert object payloads, and keeps broader org/package ownership-transfer work separate.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: the affected commands pass JSON strings, both HTTP transports stringify again, and the server handlers require object fields.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix; optional HTTP string-body tolerance is acceptable only with coverage for both runtime serialization paths.
