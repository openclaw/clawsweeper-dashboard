## Summary

Keep open. Current main still sends pre-stringified request bodies from the reported CLI commands while the shared HTTP client serializes POST bodies again, so the server continues to see a JSON string instead of object fields.

## What This Changes

The issue requests fixing CLI transfer request, skill rename, and skill merge so their POST bodies are not double JSON-encoded before reaching the API.

## Best Possible Solution

Land a focused CLI fix that sends plain object bodies from transfer, rename, and merge, updates command tests to assert object payloads, and covers both HTTP runtime paths if shared defensive handling is added.

## Reproduction Assessment

Yes. The reporter provided concrete CLI commands and current main still contains the exact static path: command body strings are passed into an HTTP helper that stringifies them again before server handlers read object fields.

## Solution Assessment

Yes. Removing pre-serialization at the affected command call sites is the narrowest maintainable fix; shared HTTP string-body tolerance is acceptable only with tests for both runtime branches.
