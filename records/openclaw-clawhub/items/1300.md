## Summary

Keep open. Current main still source-reproduces the reported CLI double-serialization bug for transfer request, skill rename, and skill merge; the related focused fixes were closed unmerged, and the broader open transfer PR does not make this issue safe to close.

## What This Changes

The issue asks to stop pre-stringifying JSON request bodies for `transfer request`, `skill rename`, and `skill merge` so `apiRequest` remains the single JSON serialization boundary.

## Best Possible Solution

Land a focused CLI fix that sends plain object bodies from the three affected commands, updates command tests to assert object payloads, and only adds HTTP string-body tolerance if both runtime paths are covered.

## Reproduction Assessment

Yes, source-reproducible on current main. The affected commands pass strings to an HTTP helper that serializes POST bodies again, while the server handlers require object fields and return the exact required-field errors when those fields are missing.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix; optional HTTP helper hardening is acceptable only with coverage for both the Node and Bun/curl serialization paths.
