## Summary

Keep open. Current main still has the reported double JSON serialization path for transfer request, skill rename, and skill merge, and the related fix PRs were closed without merging; ClawHub cleanup policy also keeps issues open.

## What This Changes

The issue reports that transfer request, skill rename, and skill merge pass pre-stringified JSON bodies into an HTTP helper that serializes them again.

## Best Possible Solution

Land a focused CLI fix that restores plain object request bodies for the affected commands, updates regression tests, and keeps any shared HTTP hardening small and covered in both runtime paths.

## Reproduction Assessment

Yes. Source inspection on current main shows the affected commands pass JSON strings, the HTTP helper stringifies again, and the handlers require object fields that a double-encoded JSON string will not provide.

## Solution Assessment

Yes. The narrow maintainable repair is to pass plain objects at the affected call sites and update tests to assert that contract; defensive HTTP passthrough is acceptable only with coverage for both runtime paths.

## Repair Work Prompt

Fix the CLI double JSON serialization bug for transfer request, skill rename, and skill merge. In packages/clawhub/src/cli/commands/transfer.ts and packages/clawhub/src/cli/commands/ownership.ts, send plain object bodies to apiRequest instead of JSON.stringify strings. Update packages/clawhub/src/cli/commands/transfer.test.ts and packages/clawhub/src/cli/commands/ownership.test.ts to assert object payloads. If you defensively harden packages/clawhub/src/http.ts to tolerate pre-stringified bodies, cover both the Node fetch path and Bun/curl path; do not broaden this into org/package transfer behavior or the larger ownership-transfer feature work. Preserve the existing API routes and response schemas, and add only focused regression coverage for the reported commands.
