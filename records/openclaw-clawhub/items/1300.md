## Summary

Keep open under the ClawHub issue cleanup policy: current main and the v0.15.0 CLI tag have fixed all three reported double-serialization call sites, but this workflow does not auto-close ClawHub issues.

## What This Changes

The issue reports that transfer request, skill rename, and skill merge sent pre-stringified JSON bodies to apiRequest, causing required-field errors after a second serialization.

## Best Possible Solution

Keep the current object-body implementation and have maintainers manually close after they are satisfied that v0.15.0 or later is the supported retry path.

## Reproduction Assessment

No, not against current main. The original v0.9.0 report and later reproduction comments were credible, but current source and tests no longer contain the pre-stringified payload path for the three reported commands.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix for the required-field failures; defensive string-body handling in the shared HTTP helper is optional hardening rather than necessary for these current call sites.
