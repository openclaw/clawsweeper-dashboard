## Summary

Keep open. Current main still sends pre-stringified CLI request bodies into an HTTP helper that serializes JSON bodies again, so transfer request, skill rename, and skill merge still match the reported required-field failures; the related fix PRs were closed without merging.

## What This Changes

Fix CLI transfer request, skill rename, and skill merge so their POST bodies are not double JSON serialized.

## Best Possible Solution

Land a focused CLI fix that sends plain object bodies from the affected commands, updates regression tests to assert object payloads, and adds a changelog entry for the user-facing CLI fix.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: the affected commands pass JSON strings, both HTTP runtime paths stringify `args.body` again, and the server handlers then look for object fields. I did not run a live ownership-changing CLI command, but the current-main path deterministically produces the reported missing-field behavior.

## Solution Assessment

Yes, the narrow maintainable fix is to pass plain objects from `transfer.ts` and `ownership.ts` and update the tests to enforce that contract. Defensive string-body handling in `http.ts` is reasonable if it covers both runtime branches, but it should not replace fixing the buggy call sites.
