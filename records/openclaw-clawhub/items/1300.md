## Summary

Keep open. Current main still contains the reported double-serialization path for transfer request, skill rename, and skill merge; the earlier focused fixes were closed unmerged, and the open ownership-transfer PR only partially overlaps the transfer path rather than resolving the rename/merge command bodies.

## What This Changes

The issue asks to stop transfer request, skill rename, and skill merge from sending pre-stringified JSON bodies to `apiRequest`.

## Best Possible Solution

Use `apiRequest` as the single JSON serialization boundary for these CLI commands, with focused regression tests covering the object-body contract and both HTTP transports if the helper is hardened defensively.

## Reproduction Assessment

Yes. Current-main source shows the affected commands pass JSON strings, both HTTP transports stringify POST bodies again, and the server reads object fields that are absent after double encoding.

## Solution Assessment

Yes. Passing plain objects from the affected CLI handlers is the narrow maintainable fix; server-side parsing of nested JSON strings would mask the client boundary bug.
