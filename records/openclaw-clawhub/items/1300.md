## Summary

Keep open. Current main still has the reported CLI serialization bug: the transfer request, skill rename, and skill merge commands pass JSON strings into an HTTP helper that serializes bodies again, so the API receives a JSON string rather than the object fields its handlers read. ClawHub cleanup policy keeps issues open, and this remains a narrow repair candidate.

## What This Changes

The issue asks to stop double JSON serialization for `clawhub transfer request`, `clawhub skill rename`, and `clawhub skill merge`.

## Best Possible Solution

Land a focused CLI fix that restores `apiRequest` as the single JSON serialization boundary, updates transfer and ownership command tests to assert plain object payloads, and covers any shared HTTP hardening in both runtime paths.

## Reproduction Assessment

Yes, by source inspection on current main. The affected commands pass JSON strings, both HTTP transports stringify request bodies again, and the server handlers expect object fields, so the reported required-field failures follow without needing a live authenticated account.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix; optional HTTP tolerance is reasonable only if both Node and Bun/curl serialization paths receive regression coverage.
