## What This Changes

The issue reports that transfer request, skill rename, and skill merge pass pre-stringified JSON bodies into an HTTP helper that serializes request bodies again.

## Best Possible Solution

Land a focused CLI fix that sends plain object bodies from the affected commands, updates command tests to assert object payloads, and covers any shared HTTP hardening in both runtime paths.

## Solution Assessment

Yes. The narrow maintainable fix is to remove command-level pre-serialization and update regression tests; shared HTTP string-body tolerance is reasonable only if covered for both Node and Bun/curl paths.

Fix the CLI double JSON serialization bug reported for transfer request, skill rename, and skill merge. In packages/clawhub/src/cli/commands/transfer.ts and packages/clawhub/src/cli/commands/ownership.ts, pass plain object payloads to apiRequest instead of JSON.stringify strings. Update packages/clawhub/src/cli/commands/transfer.test.ts and packages/clawhub/src/cli/commands/ownership.test.ts so they assert object payload shape rather than string substrings. If you harden packages/clawhub/src/http.ts to tolerate pre-serialized string bodies, cover both the Node fetch path and Bun/curl path with tests in packages/clawhub/src/http.test.ts and packages/clawhub/src/http.bun.test.ts. Keep the fix scoped to this serialization contract; do not change API route semantics, auth, transfer policy, rename policy, merge policy, or unrelated CLI behavior. Include a CHANGELOG.md entry if this repo treats CLI user-facing fixes as changelog-worthy.
