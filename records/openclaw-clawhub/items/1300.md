## Summary

Keep open. Current main still contains the reported CLI double-serialization path: transfer request, skill rename, and skill merge pass JSON strings into an HTTP helper that serializes request bodies again, while the server handlers expect parsed object fields. The previously linked fixes remain closed and unmerged, so this issue still needs a focused repair rather than cleanup closure.

## What This Changes

The issue reports that transfer request, skill rename, and skill merge fail because their CLI handlers pre-stringify JSON bodies before passing them to the shared HTTP request helper.

## Best Possible Solution

Land a focused CLI fix that sends plain object request bodies from the affected commands, updates command tests to assert object payloads, and optionally hardens both HTTP serialization paths against pre-stringified JSON.

## Reproduction Assessment

Yes. Source inspection on current main shows the affected commands pass JSON strings, both HTTP runtime paths stringify request bodies again, and the API handlers read object fields that will be absent after double encoding.

## Solution Assessment

Yes. The narrow maintainable solution is to restore the plain-object apiRequest contract at the affected call sites, add regression coverage, and only add shared HTTP tolerance if covered in both runtime paths.
