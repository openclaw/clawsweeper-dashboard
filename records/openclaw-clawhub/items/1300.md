## Summary

Keep open under the ClawHub issue policy, but no repair lane is needed: current main sends plain object bodies for transfer, rename, and merge, and the complete reported fix is present in the v0.15.0 CLI tag.

## What This Changes

The issue reports CLI transfer request, skill rename, and skill merge passing pre-stringified POST bodies that `apiRequest` serializes a second time.

## Best Possible Solution

Keep the v0.15.0 object-body implementation and only reopen engineering work if someone reproduces the same failures on v0.15.0 or newer with live output.

## Reproduction Assessment

No. On current main, the three affected command handlers pass object bodies and the command tests assert object payloads, so the reported double-serialization path is no longer present in source.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix; shared HTTP string-body tolerance would be broader optional hardening, not required for these commands.
