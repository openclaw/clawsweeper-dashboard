## Summary

Keep this PR open: it addresses a real publisher OG-image bug and has good visual proof, but the new public HTTPS avatar fetch path is not safe enough to merge because it checks hostname text before `fetch` without proving the resolved target is public.

## What This Changes

The PR passes publisher avatar/kind/downloads into publisher OG image URLs, switches OG cards from Installs to Downloads with compact formatting, and broadens publisher OG avatar fetching to public HTTPS hosts.

## Best Possible Solution

Land the org-logo metadata fix with DNS-aware SSRF protections, and only change the cross-surface OG metric if maintainers confirm Downloads should supersede the current Installs behavior.

## Reproduction Assessment

Yes for the org-logo bug by source inspection: current main sends only `handle` to `/og/profile` and only fetches trusted avatar hosts. The Downloads-vs-Installs part is not a reproduction issue; current main intentionally shows Installs after a recent merged PR.

## Solution Assessment

No in its current form. Passing publisher metadata into OG URLs is the right layer, but the public-image fetch needs DNS-aware SSRF protection and the cross-surface metric reversal needs maintainer confirmation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate resolved image IPs before fetching:** `server/og/fetchImageDataUrl.ts:54`
  - body: This guard only checks the URL hostname text before calling `fetch`. A publisher avatar URL on an attacker-controlled public domain can pass this check and then resolve to 127.0.0.1, 169.254.169.254, or RFC1918 space when the server connects, so the new public HTTPS avatar path still creates an SSRF route. Please resolve and reject private/link-local/loopback/metadata targets before connecting, and apply the same check to redirects.
  - confidence: 0.86
