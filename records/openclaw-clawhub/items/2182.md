## Summary

Keep open: this is fresh account-specific reinstatement fallout involving production publisher attribution, OAuth login, and API-token state. Current main has relevant admin recovery paths and intentionally keeps revoked tokens revoked after unban, but the remaining failure depends on live account/auth/database state that cannot be safely resolved by source cleanup.

## What This Changes

The issue requests fixing a reinstated account whose public skills exist but are not attributed on the profile, cannot establish a login session, and cannot replace a revoked API token.

## Best Possible Solution

A trusted operator should inspect the production user/auth identity, publisher membership, skill `ownerUserId`/`ownerPublisherId` rows, token records, and audit/scanner history; if the account is approved, repair the publisher/skill linkage and remaining account-standing state, then have the user create a fresh API token after login succeeds.

No. The public HTML routes can be checked, but the reported login failure, token revocation, and attribution mismatch require the reporter’s auth flow and production database/audit rows that are not available in the repository checkout.
