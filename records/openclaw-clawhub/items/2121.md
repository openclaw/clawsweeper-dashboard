## Summary

Keep open: this fresh PR is not implemented on current main and the diff has a source-level trust UI bug plus tests-only proof, so it needs contributor/maintainer follow-up before merge.

## What This Changes

The PR exposes `isSuspicious` on public skill list/search payloads and hides the skills browse `Hide suspicious` checkbox unless suspicious results are loaded or the filter is active.

## Best Possible Solution

Publish a reliable public suspicious-state boolean using the existing fallback logic, then land the conditional toolbar only after maintainers accept the trust UX and the contributor provides real after-fix proof.

## Reproduction Assessment

Yes, from source inspection: current main always renders the toolbar checkbox, and the PR’s proposed failure mode is source-reproducible because `isSuspicious` is optional while the branch publishes it directly. I did not run the app or mutate the checkout.

## Solution Assessment

No as written: the UI direction may be acceptable, but the public payload should compute a fallback suspicious boolean for legacy digest/full skill rows instead of relying only on the optional denormalized field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Compute suspicious state for legacy digest rows:** `convex/lib/public.ts:149`
  - body: `isSuspicious` is still optional in the digest schema and has a backfill for undefined rows, but this publishes only `skill.isSuspicious`. A legacy digest/full skill with `moderationFlags: ['flagged.suspicious']` or a scanner suspicious reason and no denormalized flag will reach the browser without a true suspicious bit, so the newly conditional toolbar can hide the only visible way to enable the safety filter while a suspicious result is shown. Populate this field with the existing suspicious fallback when the denormalized value is missing.
  - confidence: 0.84
