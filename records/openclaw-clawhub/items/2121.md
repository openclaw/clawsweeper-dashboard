## Summary

Keep open. The branch is not implemented on main and still needs contributor real behavior proof plus a correctness fix before merge because the new trust-facing UI depends on an optional `isSuspicious` value without the established moderation fallback.

## What This Changes

The branch adds `isSuspicious` to public skill payloads, forwards it through digest hydration, and renders the skills browse `Hide suspicious` checkbox only when suspicious results are loaded or the filter is active.

## Best Possible Solution

Publish a reliable public suspicious-state boolean using the existing fallback logic, then land the conditional toolbar only after maintainers accept the trust UX and the contributor provides real after-fix proof.

## Reproduction Assessment

Yes, from source inspection: current main always renders the toolbar checkbox, and the branch would drive conditional rendering from `skill.isSuspicious`. The schema and backfill show that value can be undefined while moderation flags/reasons still mark a row suspicious.

## Solution Assessment

No as written. The UI direction may be acceptable, but the public payload should compute a fallback suspicious boolean instead of relying only on the optional denormalized field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Compute suspicious state with fallback:** `convex/lib/public.ts:149`
  - body: `isSuspicious` is still optional on both full skill and digest rows, but this publishes only `skill.isSuspicious`. A legacy row with `moderationFlags: ['flagged.suspicious']` or a scanner suspicious reason and no denormalized flag can reach the browser without a true suspicious bit, so the conditional toolbar can hide the only visible safety filter while suspicious results are shown. Populate this field with the existing suspicious fallback when the denormalized value is missing.
  - confidence: 0.84
