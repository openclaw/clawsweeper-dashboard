## Summary

Keep open: current main still renders the skills browse `Hide suspicious` control unconditionally, and the PR is not safe to merge as written because it exposes an optional suspicious-state field without the existing moderation fallback. The screenshots are valid visible UI proof for the clean-results state, but the backend fallback bug remains.

## What This Changes

The PR exposes `isSuspicious` on public skill payloads and hides the skills browse `Hide suspicious` checkbox unless suspicious results are loaded or the filter is already active.

## Best Possible Solution

Publish a reliable public suspicious-state boolean using the existing fallback logic, then land the conditional toolbar only if maintainers accept the trust UX.

## Reproduction Assessment

Yes for the PR review finding. Source inspection shows current main always renders the toolbar checkbox, while the branch can hide it for legacy suspicious rows because the public boolean is optional and published directly.

## Solution Assessment

No as written. The UI direction may be acceptable, but the public payload should compute `isSuspicious` from the denormalized value with the existing moderation fallback when that value is missing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Compute suspicious state for legacy rows:** `convex/lib/public.ts:149`
  - body: `isSuspicious` is optional on existing skill and digest rows, but this line publishes only the denormalized value. A legacy row with `flagged.suspicious` or a scanner suspicious reason can therefore reach the browser without `true`, causing the conditional toolbar to hide the only visible safety filter while a suspicious result is shown. Fall back to the existing suspicious-state helper when the denormalized field is missing.
  - confidence: 0.86
