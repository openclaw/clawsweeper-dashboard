## Summary

Keep this PR open: the UI direction is a useful follow-up to the merged browse toolbar work, but the patch has a blocking trust-data fallback issue and still lacks real after-fix behavior proof.

## What This Changes

The PR exposes `isSuspicious` on public skill payloads and hides the skills browse `Hide suspicious` checkbox unless suspicious results are loaded or the filter is active.

## Best Possible Solution

Publish a reliable suspicious-state boolean using the existing fallback logic, then land the conditional toolbar only after maintainers accept the trust UX and the contributor provides real after-fix proof.

## Reproduction Assessment

Yes. From source inspection, current main always renders the toolbar checkbox, and the PR’s introduced failure mode is source-reproducible because `isSuspicious` remains optional while the branch publishes it directly.

## Solution Assessment

No as written. The UI direction may be acceptable, but the public payload should compute a fallback suspicious boolean for legacy digest/full skill rows instead of relying only on the optional denormalized field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Compute suspicious state for legacy digest rows:** `convex/lib/public.ts:149`
  - body: `isSuspicious` is still optional in the digest schema and has a backfill for undefined rows, but this publishes only `skill.isSuspicious`. A legacy row with `moderationFlags: ['flagged.suspicious']` or a scanner suspicious reason and no denormalized flag can reach the browser without a true suspicious bit, so the conditional toolbar can hide the only visible way to enable the safety filter while a suspicious result is shown. Populate this field with the existing suspicious fallback when the denormalized value is missing.
  - confidence: 0.84
