## Summary

Keep this PR open: current main has not implemented the conditional control, and the branch still has a P2 trust/UI bug because it publishes the optional denormalized suspicious flag without the existing fallback.

## What This Changes

The PR hides the skills browse `Hide suspicious` checkbox unless loaded results include suspicious skills or the filter is active, and exposes `isSuspicious` on public skill payloads.

## Best Possible Solution

Keep the PR open, publish a reliable suspicious-state boolean using the existing fallback logic, add legacy-row regression coverage, and then review the trust UX choice.

## Reproduction Assessment

Yes. Source inspection shows current main always renders the checkbox, and the PR failure is reproducible from the branch diff because legacy suspicious rows can have moderation flags or scanner reasons while `isSuspicious` is undefined.

## Solution Assessment

No as written. The UI direction may be acceptable, but the public payload should compute a fallback suspicious boolean when the denormalized field is missing before the conditional toolbar depends on it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Compute suspicious state before publishing it:** `convex/lib/public.ts:149`
  - body: The new public payload returns only `skill.isSuspicious`, but both `skills` and `skillSearchDigest` keep that field optional while fallback state can still be represented by `moderationFlags` or `moderationReason`. A suspicious row can therefore render with `isSuspicious` undefined, making `showSuspiciousFilter` false and hiding the safety checkbox even though a suspicious result is loaded. Please use the existing fallback when the denormalized bit is missing and cover the legacy case.
  - confidence: 0.86
