## Summary

Keep this PR open: the UI change has direct screenshot proof, but the patch publishes an optional suspicious-state field directly, which can hide the safety control for legacy suspicious rows.

## What This Changes

The PR exposes `isSuspicious` on public skill payloads and hides the skills browse `Hide suspicious` checkbox unless the filter is active or loaded results include suspicious skills.

## Best Possible Solution

Compute a reliable public suspicious boolean with the existing fallback before conditionally hiding the toolbar control, then land the UX only after maintainer trust-UI approval.

## Reproduction Assessment

Yes for the patch defect. Source inspection shows a valid row can have suspicious moderation fields but undefined `isSuspicious`, and the PR’s UI only shows the control when the public value is exactly true.

## Solution Assessment

No as written. The narrow maintainable fix is to publish `skill.isSuspicious ?? isSkillSuspicious(skill)` or equivalent before relying on the client-side conditional.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Compute suspicious state before publishing it:** `convex/lib/public.ts:149`
  - body: `isSuspicious` is still optional on skills and digest rows, but this publishes only `skill.isSuspicious`. A legacy suspicious row represented by moderation flags or scanner reason can reach the browser with an undefined public bit, so `showSuspiciousFilter` hides the only visible safety control while a suspicious result is shown. Use the existing suspicious fallback when the denormalized value is missing.
