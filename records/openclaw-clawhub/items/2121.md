## Summary

Keep open: the PR has real UI screenshot proof and a narrow useful direction, but the patch can hide the safety control for legacy suspicious rows because it publishes the optional denormalized `isSuspicious` value without the existing moderation fallback.

## What This Changes

The PR exposes `isSuspicious` on public skill list payloads and hides the skills browse `Hide suspicious` checkbox unless suspicious results are loaded or the filter is active.

## Best Possible Solution

Keep the conditional toolbar behavior only if the public payload computes suspicious state reliably from the denormalized field with moderation-field fallback.

## Reproduction Assessment

Yes. Current main visibly renders the checkbox unconditionally in source, and the PR diff plus inspected screenshots show the intended conditional visible/hidden toolbar states.

## Solution Assessment

No as written. The UI direction may be acceptable, but the public suspicious boolean should use the existing fallback logic instead of relying only on an optional denormalized field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Compute suspicious state for legacy rows:** `convex/lib/public.ts:149`
  - body: `isSuspicious` is still optional on both `skills` and `skillSearchDigest`, but this publishes only `skill.isSuspicious`. A legacy row with `flagged.suspicious` or a scanner suspicious reason and no denormalized bit can reach the browser as not suspicious, causing the conditional toolbar to hide the only visible way to enable the safety filter while a suspicious result is loaded. Use the existing suspicious fallback when the denormalized value is missing.
  - confidence: 0.86
