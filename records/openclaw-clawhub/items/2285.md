## Summary

Keep open: the PR hardens several personal-publisher membership and transfer paths, but it still leaves the active publish target resolver trusting stale personal-publisher membership rows, and the contributor proof is tests/CI only.

## What This Changes

The PR changes Convex publisher membership, resource authorization, package transfer checks, tests, changelog, and org specs so personal publishers use `linkedUserId` instead of extra membership rows.

## Best Possible Solution

Apply the personal-publisher `linkedUserId` boundary to every publish, manage, and transfer authorization entry point, especially `resolvePublishTargetForUserInternal`, with focused regression coverage and redacted live proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a stale `publisherMembers` row for a `kind: "user"` publisher can satisfy `resolvePublishTargetForUserInternal`, and package/skill publish flows use that resolver before assigning owner targets; I did not execute a live Convex mutation in this read-only review.

## Solution Assessment

No, not yet. Splitting authorization by publisher kind is the right direction, but the PR misses the shared publish resolver; the narrower maintainable fix is to guard personal publishers there and add publish-path regressions.
