## What This Changes

The branch restricts personal publisher membership management and updates publisher, package, and skill authorization paths so personal publishers rely on `linkedUserId` while org publishers continue using memberships.

## Best Possible Solution

Centralize a publisher-kind-aware owner access helper and apply it to every skill/package dashboard, read, appeal, publish, transfer, and manage path, with stale-personal negative tests and org-positive regressions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a stale `publisherMembers` row for a `kind: "user"` publisher and call the skill detail/dashboard/appeal or package dashboard owner-publisher paths as the stale member.

## Solution Assessment

No. The direction is correct, but the implementation should use a shared publisher-kind-aware helper so every personal-publisher authority check rejects stale memberships consistently.
