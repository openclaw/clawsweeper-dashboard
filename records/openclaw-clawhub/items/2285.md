## Summary

Keep open: this is a real security hardening PR, but it is not merge-ready because stale personal-publisher membership rows still authorize two security-sensitive paths and the proof is only test/CI output.

## What This Changes

The branch restricts personal publisher membership management and adjusts publisher/package authorization so personal publishers rely on `linkedUserId` while org publishers continue to use memberships.

## Best Possible Solution

Centralize a publisher-kind-aware authorization helper and use it for every personal-publisher membership boundary, with stale-personal negative tests and org-positive tests for publish, transfer, read, status, and cleanup flows.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: create a stale `publisherMembers` row on a `kind: "user"` publisher and the remaining skill destination and package owner-access helpers still accept that row.

## Solution Assessment

No. The PR fixes several important paths, but the safer maintainable solution is a shared publisher-kind-aware authorization helper applied consistently across all membership-based personal-publisher checks.
