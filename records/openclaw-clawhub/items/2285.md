## Summary

Keep this PR open: the direction is useful for a security boundary, but the patch is not ready because package ownership transfer still trusts personal-publisher membership rows and the PR has no after-fix real behavior proof.

## What This Changes

The branch blocks member add/update flows on personal publishers, treats `linkedUserId` as authoritative in publisher helpers, allows linked users to remove stale extra personal-publisher memberships, and adds tests/spec/changelog coverage.

## Best Possible Solution

Apply the same personal-publisher `linkedUserId` rule to every ownership-management path, especially package transfers, then add focused regression coverage and real runtime proof for the protected flow.

## Reproduction Assessment

Yes, from source: create a user-kind source publisher with a stale admin `publisherMembers` row for a non-linked actor, then call `transferPackageOwnerForUserInternal`; `canManageSource` still becomes true through that row. I did not run it, so this is source-reproducible rather than executed.

## Solution Assessment

No; the helper and member-management changes are the right shape, but the security invariant is incomplete while package transfer bypasses those helpers and still honors stale personal-publisher memberships.
