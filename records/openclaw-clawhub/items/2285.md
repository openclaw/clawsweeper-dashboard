## Summary

Keep open: the patch moves in the right direction, but stale personal-publisher membership rows are still trusted by active publish and skill-transfer paths, and the PR has only test/CI output rather than real after-fix behavior proof.

## What This Changes

The PR changes Convex publisher membership helpers, member mutations, package transfer authorization, tests, changelog, and org specs so personal publishers use `linkedUserId` rather than collaborative membership rows.

## Best Possible Solution

Apply the personal-publisher `linkedUserId` boundary to every publish, manage, and transfer authorization entry point, add focused regressions for package and skill paths, and provide redacted live proof of stale-membership denial.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a stale `publisherMembers` row on another user's `kind: "user"` publisher can still satisfy the active publish resolver and skill transfer destination helper; I did not run live Convex mutations in this read-only review.

## Solution Assessment

No. Splitting authorization by publisher kind is the right direction, but the patch misses shared publish and skill-transfer entry points; the narrow fix is to guard personal publishers there and cover those paths with regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard personal publishers in the publish resolver:** `convex/publishers.ts:822-823`
  - body: The PR hardens other helpers, but package and skill publishes still call `resolvePublishTargetForUserInternal`. That resolver accepts an allowed `publisherMembers` row after resolving a requested handle, so a stale membership on another user's personal publisher can still select that publisher as the publish owner. Add the same `kind === "user"` / `linkedUserId` boundary here and cover package and skill publish paths.
  - confidence: 0.9
- **[P1] Guard personal destinations in skill transfers:** `convex/skills.ts:8280-8281`
  - body: Skill owner transfers still call `canManagePublisherDestination`, which falls through to admin membership rows even when the destination publisher is personal. That leaves the same stale-membership boundary open for transferring a skill to someone else's personal publisher; split this helper by publisher kind like the package transfer path.
  - confidence: 0.84
