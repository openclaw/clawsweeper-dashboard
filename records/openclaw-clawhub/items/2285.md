## Summary

Keep open: the PR is aimed at a real authorization gap, but the hardening is incomplete and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR restricts Convex publisher membership management and publisher-role helpers so personal publishers use `linkedUserId`, adds regression tests, updates `specs/orgs.md`, and records a changelog entry.

## Best Possible Solution

Centralize personal-publisher authorization so every role, target-resolution, file-read, resource-management, and membership-management path treats `linkedUserId` as authoritative for `kind: "user"`, then cover stale-row cases with regression tests and real behavior proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: create a personal publisher with an extra `publisherMembers` row, then current main membership checks authorize that non-linked user in several paths. I did not execute the scenario because this was a read-only review.

## Solution Assessment

Not yet. The PR is directionally right and narrow, but the best fix must also harden remaining membership-role consumers such as `getMemberRoleInternal` before merge.
