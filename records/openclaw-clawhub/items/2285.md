## Summary

The PR is useful security hardening but is not merge-ready: it closes membership-management and package-transfer paths while leaving the shared publish-target resolver able to trust stale personal-publisher membership rows, and it still lacks after-fix real behavior proof beyond tests and CI output.

## What This Changes

The PR changes publisher membership helpers, membership mutations, package ownership transfer authorization, tests, changelog, and org specs so personal publishers use `linkedUserId` boundaries instead of collaborative membership rows.

## Best Possible Solution

Apply the personal-publisher `linkedUserId` boundary to every publish, manage, and transfer authorization entry point, especially `resolvePublishTargetForUserInternal`, then merge only after focused regressions and redacted live proof show stale-membership denial plus linked-owner success.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a stale `publisherMembers` row for a `kind: "user"` publisher can satisfy `resolvePublishTargetForUserInternal`, and package/skill publish flows use that resolver before assigning owner targets; I did not execute a live Convex mutation in this read-only review.

## Solution Assessment

No, not yet. Splitting authorization by publisher kind is the right direction, but the PR misses the shared publish resolver; the narrow maintainable fix is to guard personal publishers there and add publish-path regressions.
