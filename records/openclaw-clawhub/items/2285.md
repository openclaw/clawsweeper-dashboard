## Summary

Keep this PR open: the direction is useful security hardening, but the latest head still leaves stale personal-publisher membership rows trusted in publish and skill-transfer authorization paths, and the PR has only test/CI output rather than real after-fix behavior proof.

## What This Changes

The PR restricts personal-publisher member management, updates shared publisher authorization helpers and package transfer checks, adds regression tests, documents the intended org/personal publisher boundary, and updates the changelog.

## Best Possible Solution

Apply the personal-publisher linkedUserId boundary to every publish, manage, and transfer authorization entry point, then add focused regressions and redacted live proof for stale-membership denial.

## Reproduction Assessment

Yes, by source inspection: a stale publisherMembers row for a kind: "user" publisher can still satisfy resolvePublishTargetForUserInternal, and publish flows use that resolver before assigning owner targets. I did not execute a live Convex mutation in this read-only review.

## Solution Assessment

No, not yet. Splitting authorization by publisher kind is the right direction, but the patch needs the same boundary in the active publish resolver and skill transfer destination path before it closes the security gap.
