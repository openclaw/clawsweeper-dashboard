## Summary

Keep this PR open: the validation-error direction addresses the related CLI/API symptom, but the patch also exposes arbitrary unknown 500 exception text at a public HTTP boundary and still lacks after-fix real behavior proof.

## What This Changes

The PR changes HTTP API v1 soft-delete error mapping so validation failures return 400 with a cleaned message and unknown 500 responses include the cleaned underlying message.

## Best Possible Solution

Keep the new explicit 400 mapping for known validation errors, preserve a generic body for unexpected 500s, add focused regression coverage for the invalid package delete path, and require redacted live CLI or HTTP proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: CLI package delete sends the raw name, the package delete route passes it to softDeletePackageInternal, normalizePackageName rejects bare scope/name values, and current main falls through to a generic 500 body. I did not run a live HTTP reproduction.

## Solution Assessment

No, not as written. The 400 validation mapping is the right narrow direction, but unexpected 500 responses should remain generic unless maintainers explicitly approve exposing internal exception text.
