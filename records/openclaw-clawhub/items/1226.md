## Summary

Keep open. Current main still routes relative package publish sources through the resolved global workdir, so a configured workspace can make a plugin directory with a valid manifest fail as though `openclaw.plugin.json` were missing; ClawHub cleanup policy also keeps issues open.

## What This Changes

Publishing a local code-plugin package should not fail with `openclaw.plugin.json required` when the manifest exists in the caller's plugin directory.

## Best Possible Solution

Fix local package-publish source resolution so relative sources use the caller’s intended package directory, then add regression coverage for configured-workspace versus plugin-cwd behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run `clawhub package publish . --family code-plugin` from a plugin directory without `.clawhub` markers while the CLI resolves a configured Clawdbot/OpenClaw workspace elsewhere, and the manifest check runs against the wrong folder.

## Solution Assessment

Yes. A focused resolver fix with regression tests is the narrow maintainable solution; weakening `openclaw.plugin.json` validation would hide that the CLI read the wrong package directory.

## Repair Work Prompt

Fix the ClawHub CLI package-publish local source resolution bug reported in https://github.com/openclaw/clawhub/issues/1226. Current behavior can make `clawhub package publish . --family code-plugin` from a plugin directory without `.clawhub` markers resolve `.` under a configured Clawdbot/OpenClaw workspace, then fail with `openclaw.plugin.json required` even though the caller's plugin directory has the manifest. Update the resolver or package-publish call boundary so implicit relative local package publish sources use the caller's intended package directory, while preserving explicit `--workdir` and `CLAWHUB_WORKDIR` semantics, GitHub URL/shorthand precedence, existing local-directory precedence, and ClawPack `.tgz` behavior. Add regression coverage for configured-workspace versus plugin-cwd behavior, and ensure a GuardClaw-shaped fixture with a manifest reaches the compatibility metadata validation rather than manifest-missing. Do not weaken `openclaw.plugin.json` validation and do not add plugin search/install/list features in this PR. Update CLI docs only if the user-facing behavior text needs clarification.
