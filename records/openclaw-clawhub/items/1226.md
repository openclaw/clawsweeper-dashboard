## Summary

Keep open. This is a ClawHub issue, and current main still leaves the reported failure source-reproducible because local package sources are resolved from the CLI's resolved workdir, which can differ from the plugin directory where the command was run.

## What This Changes

The issue reports that `clawhub package publish . --family code-plugin` rejects an existing root `openclaw.plugin.json` with `openclaw.plugin.json required`.

## Best Possible Solution

Fix local package-publish source resolution so relative local sources use the caller's intended package directory when no explicit workdir override is supplied, while keeping manifest and compatibility validation intact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run `clawhub package publish . --family code-plugin` from a plugin directory without `.clawhub` markers while Clawdbot/OpenClaw config points the resolved workdir elsewhere, causing manifest validation to inspect the wrong folder.

## Solution Assessment

Yes. A focused source-base resolver fix with regression tests is the narrow maintainable solution; weakening `openclaw.plugin.json` validation would hide that the CLI read the wrong package directory.
