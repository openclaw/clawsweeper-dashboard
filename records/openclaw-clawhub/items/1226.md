The issue reports that `clawhub package publish . --family code-plugin` rejects an existing root `openclaw.plugin.json` with a missing-manifest error.

## Best Possible Solution

Fix local package-publish source resolution so relative local sources use the caller's intended package directory when no explicit workdir override is supplied, while preserving explicit workdir/env and GitHub source behavior.

Yes, source inspection gives a high-confidence path: run `clawhub package publish . --family code-plugin` from a plugin directory without ClawHub markers while Clawdbot/OpenClaw config points the CLI workdir elsewhere. The manifest check then inspects the configured workspace instead of the plugin folder.

## Solution Assessment

Yes. A focused resolver fix with regression tests is the narrow maintainable solution; weakening `openclaw.plugin.json` validation would only hide that the CLI read the wrong package directory.

## Repair Work Prompt

Fix the ClawHub CLI package publish resolver bug reported in https://github.com/openclaw/clawhub/issues/1226. Local relative package sources such as `.` must resolve to the caller's intended package directory when no explicit workdir override is supplied, instead of blindly using a configured Clawdbot/OpenClaw workspace. Preserve explicit `--workdir` / `CLAWHUB_WORKDIR` semantics, GitHub URL and `owner/repo` shorthand behavior, existing local-directory precedence, and the `openclaw.plugin.json` validation itself. Add regression coverage for a plugin cwd with `openclaw.plugin.json` while the resolved global workdir points elsewhere, and make sure packages that lack newer OpenClaw compatibility metadata still fail with the metadata error after the path is fixed. Consider related source-resolution reports https://github.com/openclaw/clawhub/issues/48 and https://github.com/openclaw/clawhub/issues/1509 only where the same helper or precedence rule can be applied without broadening beyond publish source resolution.
