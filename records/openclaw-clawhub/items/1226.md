## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-backed path to the reported false `openclaw.plugin.json required` error when `package publish .` is resolved against the CLI's configured workdir instead of the caller's plugin directory.

## What This Changes

The issue reports that `clawhub package publish . --family code-plugin` rejects a plugin repository with `openclaw.plugin.json required` even though the manifest exists at the repository root.

## Best Possible Solution

Update CLI package-publish source resolution so implicit local sources use the caller's intended package directory while preserving explicit workdir/env and GitHub source behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run `clawhub package publish . --family code-plugin` from a plugin directory without `.clawhub` markers while a configured Clawdbot/OpenClaw workspace points elsewhere, and `.` is resolved under that workspace before the manifest check.

## Solution Assessment

Yes. A focused CLI source-resolution fix with regression tests is the narrowest maintainable solution; loosening `openclaw.plugin.json` validation would hide the wrong-folder read.

## Repair Work Prompt

Fix current-main CLI package publish path resolution for local code-plugin sources. When a user runs `clawhub package publish . --family code-plugin` from a plugin repository, the source package directory should be the caller's intended local path rather than an unrelated Clawdbot/OpenClaw default workspace selected as the global CLI workdir. Preserve explicit `--workdir` / `CLAWHUB_WORKDIR` behavior, GitHub URL/shorthand behavior, and existing local directory precedence for `owner/repo` inputs. Add regression coverage for configured workspace != plugin cwd and keep `openclaw.plugin.json required` for genuinely missing manifests. Add a `CHANGELOG.md` entry for the user-facing CLI fix.
