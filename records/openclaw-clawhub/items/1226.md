Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-backed path where `package publish .` can inspect the configured workspace instead of the caller's plugin directory before emitting `openclaw.plugin.json required`.

## What This Changes

The issue reports that `clawhub package publish . --family code-plugin` rejects a plugin repository with `openclaw.plugin.json required` even though the manifest exists in the caller's directory.

## Best Possible Solution

Implement a focused CLI source-resolution fix so implicit local package publish sources use the caller's intended package directory, with regression coverage for configured-workspace versus plugin-cwd behavior.

## Repair Work Prompt

Fix the ClawHub CLI package-publish source resolution bug reported in https://github.com/openclaw/clawhub/issues/1226. Current main resolves `clawhub package publish . --family code-plugin` through the global resolved `opts.workdir`, which can be a configured Clawdbot/OpenClaw workspace rather than the directory where the user ran the command, causing a false `openclaw.plugin.json required` error. Keep the fix scoped to CLI source resolution for package publish and related tests. Preserve explicit `--workdir`/`CLAWHUB_WORKDIR` behavior, GitHub source parsing and shorthand precedence, existing local-directory precedence, `.tgz` ClawPack publish behavior, and manifest/compatibility validation. Add regression coverage for a plugin cwd without `.clawhub` markers while the configured workspace points elsewhere, proving `package publish .` reads the plugin folder manifest. Do not loosen `openclaw.plugin.json` or external-code-plugin compatibility validation. No changelog entry is required unless the repo's current policy requires one for this user-facing CLI fix.
