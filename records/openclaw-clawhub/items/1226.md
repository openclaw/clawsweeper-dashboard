## Summary

Keep open. ClawHub issue policy keeps issues open, and current main still has a source-backed path where `package publish .` can resolve `.` under the CLI's resolved workspace instead of the caller's plugin directory before emitting `openclaw.plugin.json required`.

## What This Changes

The issue requests that local code-plugin publishing accept the manifest in the caller's plugin repository instead of failing with a missing `openclaw.plugin.json` error.

## Best Possible Solution

Fix package publish local-source resolution so implicit local inputs use the caller's intended package directory when no explicit workdir override is supplied, while keeping manifest and compatibility validation intact.

## Reproduction Assessment

Yes by source inspection. Run `clawhub package publish . --family code-plugin` from a plugin directory without `.clawhub` markers while Clawdbot/OpenClaw config has a default workspace elsewhere; `.` is resolved through `opts.workdir` and the manifest check can run against the wrong folder.

## Solution Assessment

Yes. A focused source-resolution fix with regression coverage is the narrowest maintainable path; weakening `openclaw.plugin.json` or code-plugin compatibility validation would hide the wrong-directory read.

Preserve explicit workdir/env behavior, GitHub URL/shorthand handling, local `owner/repo` directory precedence, `.tgz` ClawPack behavior, monorepo `--source-path`, and all `openclaw.plugin.json` plus code-plugin compatibility/content validation. Add focused regression coverage for configured-workspace versus plugin-cwd behavior and include a changelog entry if the fix is user-facing. Stop if the implementation would require a new CLI option or a product policy change.
