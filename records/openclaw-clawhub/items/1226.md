## Summary

Keep open under the ClawHub issue cleanup policy, and current main still leaves the reported missing-manifest failure source-reproducible when local package sources are resolved under the CLI workdir instead of the caller's plugin directory.

## What This Changes

The issue reports that `clawhub package publish . --family code-plugin` rejects a plugin repository with `openclaw.plugin.json required` even though the manifest exists.

## Best Possible Solution

Fix local package-publish source resolution so relative local sources use the caller's intended package directory when no explicit workdir override is supplied, while preserving explicit workdir/env and GitHub source behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run `clawhub package publish . --family code-plugin` from a plugin directory without `.clawhub` markers while Clawdbot/OpenClaw config points the CLI workdir at another workspace, causing the manifest check to inspect the wrong folder.

## Solution Assessment

Yes. A focused resolver fix with regression tests is the narrow maintainable solution; weakening `openclaw.plugin.json` validation would hide that the CLI read the wrong package directory.
