## Summary

Keep open. The ClawHub cleanup policy keeps issues open, and current `main` still leaves the reported false manifest error plausible because local publish sources are resolved under the CLI's resolved global workdir before `openclaw.plugin.json` validation. The related comment about plugin browse/install/list commands is useful adjacent CLI catalog feedback, but it does not supersede this publish-path bug.

## What This Changes

Publishing a local code plugin with `clawhub package publish . --family code-plugin` reports that `openclaw.plugin.json` is missing even when the manifest exists in the plugin root.

## Best Possible Solution

Fix local package source resolution so relative `package publish` paths use the caller's intended package directory when no explicit workdir override is supplied, while preserving explicit workdir/env behavior, GitHub source handling, and manifest/compatibility validation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a plugin cwd without `.clawhub` markers plus a configured Clawdbot/OpenClaw default workspace that lacks `openclaw.plugin.json`; `package publish .` resolves `.` under that configured workspace and reaches the manifest error.

## Solution Assessment

Yes. The narrowest maintainable fix is to adjust local source resolution for the package publish command and add regression coverage for configured workspace versus plugin cwd; weakening manifest validation would hide real invalid packages.
