## Summary

Keep open. Current main still leaves the reported CLI failure source-reproducible: relative local `package publish .` sources are resolved against the CLI workdir, which can differ from the plugin directory, and the reported GuardClaw commit does contain a root `openclaw.plugin.json`.

## What This Changes

The issue asks for `clawhub package publish . --family code-plugin` to publish a valid local code-plugin package instead of reporting its manifest missing.

## Best Possible Solution

Fix local package-publish source resolution so relative local sources use the caller's intended package directory when no explicit workdir override is supplied, while preserving manifest validation and GitHub source behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run `clawhub package publish . --family code-plugin` from a plugin directory without `.clawhub` markers while Clawdbot/OpenClaw config points the resolved workdir elsewhere, causing the manifest check to inspect the wrong folder.

## Solution Assessment

Yes. A focused resolver fix with regression tests is the narrow maintainable solution; weakening `openclaw.plugin.json` validation would hide that the CLI read the wrong package directory.
