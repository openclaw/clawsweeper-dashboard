Keep open. The supplied ClawHub policy keeps issues open, and current main still has a credible source-level path to the reported false `openclaw.plugin.json required` error when `package publish .` is resolved against a configured CLI workdir rather than the caller's plugin directory; the later plugin browse/install comment is a separate CLI feature request, not a fix for this publish failure.

## What This Changes

The issue asks for `clawhub package publish . --family code-plugin` to accept a valid root `openclaw.plugin.json` instead of failing with a missing-manifest error.

## Repair Work Prompt

Fix the CLI package publish local-source resolution bug reported here. `clawhub package publish . --family code-plugin` should read the plugin directory the caller intended when no explicit `--workdir` or `CLAWHUB_WORKDIR` override is supplied, even if a configured Clawdbot/OpenClaw default workspace exists elsewhere. Preserve explicit workdir/env behavior, GitHub shorthand/URL handling, local-directory-over-GitHub-shorthand precedence, and `--source-path` behavior. Do not relax `openclaw.plugin.json` or `openclaw.compat.pluginApi` / `openclaw.build.openclawVersion` validation; after the path fix, GuardClaw-style packages that lack compatibility metadata should fail with the metadata error instead of a missing manifest. Add focused regression tests simulating a plugin cwd without `.clawhub` markers and a configured default workspace elsewhere. Add a `CHANGELOG.md` entry if this is treated as a user-facing CLI fix.
