## Summary

Keep open. The ClawHub cleanup policy does not auto-close issues, and current main still has a credible false manifest-missing path because local publish sources are resolved from the CLI workdir before code-plugin manifest validation.

## What This Changes

The issue requests that `clawhub package publish . --family code-plugin` accept a valid root `openclaw.plugin.json` instead of failing with a false manifest-missing error.

## Best Possible Solution

Fix local package publish resolution so relative local sources use the caller’s intended package directory when no explicit workdir override is supplied, while preserving explicit workdir/env and GitHub source behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to run `clawhub package publish . --family code-plugin` from a plugin directory without a ClawHub marker while Clawdbot/OpenClaw config points the default workspace elsewhere; source inspection shows `.` will be validated under that workspace.

## Solution Assessment

Yes. The narrow maintainable fix is in CLI source resolution plus regression coverage; server-side package validation and the newer compatibility metadata checks should remain unchanged.

## Repair Work Prompt

Fix the ClawHub CLI bug where `clawhub package publish . --family code-plugin` can validate a configured Clawdbot/OpenClaw workspace instead of the plugin directory the user invoked from. Keep the change scoped to local package publish source resolution and tests. Preserve explicit `--workdir`, `CLAWHUB_WORKDIR`/legacy env behavior, absolute paths, `~` paths, existing-local-directory precedence over GitHub shorthand, GitHub URL/shorthand fetching, and `--source-path` behavior for GitHub sources. Add regression coverage for a cwd/plugin folder that lacks ClawHub markers while the global/default workspace points elsewhere; the command should inspect the plugin folder and then proceed to the next real validation error or dry-run output. Do not relax code-plugin requirements for `openclaw.plugin.json`, source metadata, semver, or `openclaw.compat.pluginApi`/`openclaw.build.openclawVersion`. Update docs only if user-facing invocation semantics change.
