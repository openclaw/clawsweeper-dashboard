## Summary

Keep open. Current main still resolves relative `package publish` sources through the CLI's resolved workdir, so a configured Clawdbot/OpenClaw workspace can make a valid plugin directory fail as though `openclaw.plugin.json` were missing; ClawHub cleanup policy also keeps issues open.

## What This Changes

The issue requests that `clawhub package publish . --family code-plugin` publish a valid code-plugin directory instead of failing with `openclaw.plugin.json required`.

## Best Possible Solution

Fix local package-publish source resolution so relative local sources use the caller's intended package directory when no explicit workdir override is supplied, then add regression coverage for configured-workspace versus plugin-cwd behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run `clawhub package publish . --family code-plugin` from a plugin directory without `.clawhub` markers while the CLI resolves a configured Clawdbot/OpenClaw workspace elsewhere, causing the manifest check to read the wrong folder.

## Solution Assessment

Yes. A focused resolver fix with regression tests is the narrow maintainable solution; weakening `openclaw.plugin.json` validation would hide that the CLI read the wrong package directory.
