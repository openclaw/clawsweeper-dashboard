## Summary

Keep open. Current main still has a credible source-level path to the reported false `openclaw.plugin.json required` error because local `package publish .` is resolved against the CLI workdir, which can be a configured Clawdbot/OpenClaw workspace rather than the caller's plugin directory; the later browse/install comment is a separate CLI feature discussion.

## What This Changes

Fix code-plugin publishing so a valid root `openclaw.plugin.json` is honored when publishing a local plugin with `clawhub package publish .`.

## Best Possible Solution

Update local package-publish source resolution so implicit `.` uses the caller's intended package directory while preserving explicit workdir/env overrides and GitHub source behavior, with regression tests for the configured-workspace case.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run from a plugin directory without `.clawhub` markers while a Clawdbot/OpenClaw default workspace points elsewhere, then `package publish .` resolves the source under that configured workspace and can throw the missing manifest error.

## Solution Assessment

Yes. A focused CLI source-resolution fix plus tests is the narrowest maintainable solution; loosening manifest or compatibility validation would mask the wrong folder being read.
