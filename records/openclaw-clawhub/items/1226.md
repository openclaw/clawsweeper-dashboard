## Summary

Keep open. This is an open ClawHub issue, and current main still leaves the reported CLI failure source-reproducible: local package publish resolves relative sources against the global CLI workdir, which can differ from the plugin directory where the command was run.

## What This Changes

The issue reports that `clawhub package publish . --family code-plugin` rejects a local code-plugin package with `openclaw.plugin.json required` even though that manifest exists at the package root.

## Best Possible Solution

Fix local package-publish source resolution so relative local sources use the caller's intended package directory when no explicit workdir override is supplied, while preserving explicit workdir/env behavior, GitHub source parsing, and manifest compatibility validation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run `clawhub package publish . --family code-plugin` from a plugin directory without `.clawhub` markers while Clawdbot/OpenClaw config points the resolved global workdir elsewhere, causing validation to inspect the wrong folder.

## Solution Assessment

Yes. A focused resolver fix with regression tests is the narrow maintainable solution; weakening `openclaw.plugin.json` validation would hide that the CLI read the wrong package directory.
