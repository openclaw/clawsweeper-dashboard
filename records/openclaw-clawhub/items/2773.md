## Summary

Keep open: the exact public plugin catalog shape still returns HTTP 500 in production, and current main has not changed the relevant catalog files since the merged pagination fixes. Source tracing points to the official-first category wrapper still being able to invoke the underlying paginated package list twice in one Convex query when official results are exhausted or underfilled.

## What This Changes

The issue reports that plugin category browse should return category-filtered plugins instead of a 500 when category filters, `officialFirst=true`, and scan-status exclusions are combined.

## Best Possible Solution

Update the official-first category pager so each Convex query invocation performs at most one underlying `.paginate()` call, returning a resumable phase cursor before switching from official to community results, with regression coverage for the exact scan-status exclusion shape.

## Reproduction Assessment

Yes: the live production endpoint returns HTTP 500 for the reported API shape, and current main routes the same filters into the same official-first category pager. I did not run a local Convex instance during this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is in the backend pagination state machine, not a UI workaround, because docs/specs define category filtering and official-first browse as backend catalog behavior.
