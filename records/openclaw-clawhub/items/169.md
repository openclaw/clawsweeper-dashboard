## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still shows a credible fingerprint-contract bug where installed files can hash differently from the registry’s publish-time fingerprint inputs.

## What This Changes

The issue asks for `clawhub update` to recognize freshly installed skills as clean instead of reporting `local changes (no match)`.

## Best Possible Solution

Define one canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve so clean installs match while real local edits still block non-forced updates.

## Reproduction Assessment

Yes, by source inspection. A downloaded skill can include generated `_meta.json`, local hashing includes root JSON files, and backend resolve compares against publish-time `version.files` fingerprints, so a clean install can resolve to no match without running the CLI here.

## Solution Assessment

Yes. The maintainable fix is to make the fingerprint inputs canonical across the CLI and backend, not to bypass resolve or ignore broad file classes.

Define a single canonical fingerprint contract across CLI local hashing, downloaded zip contents, publish-time fingerprints, sync, and `/api/v1/resolve`; fresh installs must resolve to their installed version, while real local edits must still block non-forced updates. Use related reports #178 and #937 as supporting reproduction context. Likely files include `packages/clawhub/src/skills.ts`, `packages/clawhub/src/cli/commands/skills.ts`, `packages/clawhub/src/cli/commands/sync.ts`, `packages/clawhub/src/cli/commands/syncHelpers.ts`, `convex/downloads.ts`, `convex/lib/skillZip.ts`, `convex/skills.ts`, and related tests/docs.
