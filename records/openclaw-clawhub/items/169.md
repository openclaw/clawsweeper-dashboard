## Summary

Keep open under the ClawHub issue policy, but current main now covers the reported fresh-install `update` false positive: installs record a local fingerprint and update trusts that stored fingerprint when registry resolution misses.

## What This Changes

The issue reports `clawhub update` falsely detecting local changes immediately after a clean `clawhub install` because origin metadata did not store a content fingerprint.

## Best Possible Solution

Keep the current stored-fingerprint fallback for `update`, and track any broader legacy migration or `sync` fingerprint-contract cleanup in a separate canonical item if maintainers want it.

## Reproduction Assessment

No, not against current main: the fresh-install path now writes an origin fingerprint, and update can match that stored fingerprint even if the registry hash resolver misses. The original v0.5.0 report remains credible for older released behavior.

## Solution Assessment

Yes. For the reported clean-install update warning, storing the installed fingerprint and using it as a local baseline is a narrow maintainable fix that preserves local-edit detection for changed bytes.
