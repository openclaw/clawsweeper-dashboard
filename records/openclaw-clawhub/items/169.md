## Repair Work Prompt

Fix the false `local changes (no match)` behavior tracked by openclaw/clawhub#169 and related reports #178/#937. On current main, `clawhub update` hashes installed text files and resolves that fingerprint, while downloads can install generated root `_meta.json` and backend resolve compares publish-time fingerprints from `version.files`. Define a canonical skill-content fingerprint contract shared across install/update/sync, download zip creation, publish, and resolve. A fresh install or `clawhub update --force` followed immediately by non-forced update should report up to date or update cleanly; a real edit to skill content should still block without `--force`. Do not broadly ignore all JSON or generated-looking files; either avoid writing registry-generated `_meta.json` into installed skill content or deliberately exclude only that generated registry metadata from all relevant content-hash paths. Add focused regression tests for CLI update and backend zip/resolve behavior, and add a user-facing CHANGELOG.md entry if behavior changes.
