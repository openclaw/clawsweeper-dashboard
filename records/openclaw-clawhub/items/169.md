## Summary

Keep open under the ClawHub issue policy, but current main and the shipped v0.15.0 CLI appear to address the originally reported fresh-install false positive by persisting an installed fingerprint and trusting it when registry hash resolution misses.

## What This Changes

The issue reports that `clawhub update` treats freshly installed skills as locally modified because install metadata lacked a clean content fingerprint baseline.

## Best Possible Solution

Keep the shipped origin-fingerprint fallback, and if users still report update mismatches, collect a fresh live transcript that distinguishes legacy pre-fingerprint installs from registry resolve or sync-version bugs.

## Reproduction Assessment

No high-confidence failing reproduction remains on current main from source inspection: fresh installs now write a content fingerprint, and update falls back to that stored fingerprint before printing the local-changes warning. I did not execute a live install/update against the registry in this read-only pass.

## Solution Assessment

Yes. Persisting the installed fingerprint and using it only when the slug and local fingerprint still match is a narrow fix for the reported clean-install case without broadly ignoring JSON or generated files.
