## Summary

Keep open under the ClawHub issue policy, but current main and v0.15.0+ appear to cover the reported clean-install false positive with stored install fingerprints and regression coverage.

## What This Changes

The issue reports that `clawhub update` flags freshly installed skills as `local changes (no match)` because install metadata did not record a clean baseline fingerprint.

## Best Possible Solution

Keep the shipped stored-fingerprint fallback, and let maintainers manually close this once they are satisfied with the live CLI smoke path and older-install baseline note.

## Reproduction Assessment

No. Source inspection shows current main writes a post-extract fingerprint into origin metadata and uses it as a clean baseline when registry resolve returns no match, so the reported current-main false positive is not present by source review.

## Solution Assessment

Yes. Recording and trusting the post-install fingerprint for the same slug is a narrow fix for the reported clean-install false positive while still detecting changed local files when the stored baseline no longer matches.
