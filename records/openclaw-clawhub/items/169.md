Fix `clawhub update` so freshly installed or force-updated skills resolve as unchanged instead of reporting `local changes (no match)`.

## Best Possible Solution

Define one canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve, with regression coverage proving fresh installs resolve while real edits still block non-forced updates.

Yes. A shared fingerprint contract is the maintainable fix; bypassing resolve or broadly excluding file classes would be riskier than aligning the generated metadata, install extraction, stored baseline, and backend fingerprints.

## Repair Work Prompt

Implement a focused fix for this issue: make freshly installed and freshly force-updated skills resolve as unchanged instead of reporting `local changes (no match)`. Define a canonical fingerprint contract across `publish`, skill `download`, CLI `install`, CLI `update`, `sync`, and `/api/v1/resolve`. Preserve tamper/local-edit detection: do not disable the resolve check, do not broadly ignore all JSON files, and do not weaken detection for real edits. Prefer either avoiding generated root `_meta.json` in installed skill content or excluding only the generated metadata from the skill-content fingerprint contract, and add/update any Convex maintenance/backfill path needed for stored fingerprints. Include regression tests for fresh install/update and real local modification behavior. Do not touch unrelated package download behavior or plugin package ZIP layout.
