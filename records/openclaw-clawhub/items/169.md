## Summary

Keep open. The ClawHub cleanup policy keeps issues open, and current main still has a credible false-positive update path: installed/downloaded skill content can include generated root metadata that is hashed locally, while resolve-by-hash compares against publish-time version files and stored version fingerprints.

## What This Changes

The issue reports that `clawhub update` rejects freshly installed skills as `local changes (no match)` because the local integrity fingerprint cannot be matched to a published version.

## Best Possible Solution

Define one canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve. Fresh installs and force-updates should immediately resolve to their installed version, while real local edits should still block non-forced updates; add regression tests and a maintenance/backfill step if production fingerprint rows need repair.

## Repair Work Prompt

Fix the ClawHub CLI/backend fingerprint mismatch that makes freshly installed or force-updated skills report `local changes (no match)` during `clawhub update`. Treat related reports #178 and #937 as the same cluster. Define one canonical fingerprint contract across publish, download, install, update, sync, and resolve: downloaded installed content should resolve to the installed published version immediately, while real local edits must still block non-forced updates. Investigate generated root `_meta.json`, stale or missing `skillVersionFingerprints`, and the lack of an installed-content baseline in origin metadata. Do not broadly ignore all JSON or generated-looking files, do not weaken local tamper detection, and do not add or publish skill content directly. Add focused regression tests for fresh install/update and force-update behavior, plus backend tests for any changed download/resolve/backfill path.
