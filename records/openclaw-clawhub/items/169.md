## Summary

Keep open. Current main still has a source-reproducible clean-install fingerprint mismatch path, and the ClawHub cleanup policy for this repository keeps issues open unless a PR is already implemented on main.

## What This Changes

The issue asks for `clawhub update` to stop reporting `local changes (no match)` for freshly installed or force-updated skills with unchanged content.

## Best Possible Solution

Align the canonical fingerprint inputs across download, install, update, sync, publish, and resolve so generated ClawHub metadata cannot cause false local-change warnings while authored files remain protected.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: install/update extracts a download zip containing root `_meta.json`, local hashing includes root JSON files, and server resolve compares against published-version files that do not include that generated file. I did not run a live registry install/update because this review was read-only.

## Solution Assessment

Yes. A focused shared fingerprint-contract repair is the narrow maintainable solution; bypassing resolve, relying on `--force`, or ignoring broad file classes would weaken local-edit detection.

Fix the false `clawhub update` local-change warning for clean installed or force-updated skills. Align the fingerprint contract used by local install/update/sync, downloaded skill zips, publish-time fingerprints, and `/api/v1/resolve` so ClawHub-generated root metadata such as download `_meta.json` does not make a clean skill look modified. Preserve integrity detection for real authored files, including authored JSON; do not broadly ignore all JSON or all metadata-looking files. Cover the current open report and related reports at https://github.com/openclaw/clawhub/issues/178 and https://github.com/openclaw/clawhub/issues/937. Add regression tests showing install or force-update followed immediately by normal update resolves as up to date, and add coverage that a real local edit still blocks without `--force`. Keep the change scoped to CLI skill hashing/install/update/sync and Convex download/publish/resolve helpers; do not change unrelated plugin/package flows or release metadata.
