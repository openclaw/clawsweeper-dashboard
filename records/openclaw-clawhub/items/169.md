## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-reproducible false-positive path: downloaded skills can include generated root `_meta.json`, local update hashing includes root JSON files, and registry resolution compares against published-version fingerprints that omit generated download metadata. A related closed report had useful maintainer context about `.clawhub/origin.json`, but that does not cover the remaining root `_meta.json` mismatch here.

## What This Changes

The issue reports that `clawhub update` falsely reports `local changes (no match)` immediately after a clean skill install or force update.

## Best Possible Solution

Define one canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve so generated ClawHub metadata does not create false positives while real local edits still block non-forced updates.

## Reproduction Assessment

Yes. Source inspection shows downloaded skill zips can install generated root `_meta.json`, local update hashing includes root JSON files, and registry resolve compares against published file fingerprints that omit that generated metadata.

## Solution Assessment

Yes. The narrow maintainable fix is to align the compared fingerprint inputs across CLI and backend paths, not to require `--force`, manual `.clawhubignore` workarounds, or broad weakening of local-change detection.
