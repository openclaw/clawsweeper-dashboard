## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-level path where clean installed content can fail registry fingerprint resolution and produce the reported `local changes (no match)` warning; related reports #178 and #937 describe the same symptom cluster.

## What This Changes

The issue requests that `clawhub update` recognize freshly installed or force-updated skills as matching the registry instead of reporting `local changes (no match)`.

## Best Possible Solution

Define a canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve so clean installs and force-updates resolve immediately while genuine local edits still block non-forced updates.

## Reproduction Assessment

Yes, from source inspection: `update` hashes installed files and refuses when the registry cannot resolve that hash, while downloaded zips can include generated root metadata outside the published-version fingerprint. I did not execute the CLI because this review was read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to align the fingerprint contract across client and server; storing an origin hash alone may help, but should not bypass registry matching or weaken local-change detection.
