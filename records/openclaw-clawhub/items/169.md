## Summary

Keep open. Current main still has a source-confirmed clean-install fingerprint mismatch path, and the ClawHub review policy says issues should stay open while this needs a focused CLI/backend repair.

## What This Changes

The issue reports that `clawhub update` falsely treats freshly installed or force-updated skills as locally modified because the local fingerprint cannot resolve to a known published version.

## Best Possible Solution

Land one focused CLI/backend integrity fix that makes install/update local hashing, downloaded zip contents, publish fingerprints, and registry resolve use the same canonical file set while preserving detection of real local edits.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: install/update extracts a zip that can contain root `_meta.json`, local hashing includes root JSON files, and registry resolve compares only published-version file fingerprints.

## Solution Assessment

Yes. A shared fingerprint-contract fix is the narrow maintainable solution; relying on `--force` or excluding all JSON would weaken the local-change guard.

## Repair Work Prompt

Fix the false `clawhub update` local-change warning for clean installs and immediate force-updates. Define one canonical fingerprint contract shared by publish, download, install/update, sync, and `/api/v1/resolve`: ClawHub-generated root metadata such as `_meta.json` must not make a clean installed skill differ from the published version, while authored JSON/text edits must still block non-forced updates. Coordinate the related reports at https://github.com/openclaw/clawhub/issues/169, https://github.com/openclaw/clawhub/issues/178, https://github.com/openclaw/clawhub/issues/937, and https://github.com/openclaw/clawhub/issues/416. Prefer a targeted exclusion or no-extract/no-inject behavior for generated root metadata over broad JSON ignores. Add regression coverage for fresh install or force-update followed by normal update reporting up to date, plus a real local edit still producing the warning. Update `specs/` or CLI docs if the fingerprint contract is clarified, and add a changelog entry if the repo’s release process requires one for this user-facing CLI fix.
