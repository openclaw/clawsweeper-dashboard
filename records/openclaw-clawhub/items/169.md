Keep open under the ClawHub issue policy: current main covers the reported fresh-install false positive with stored origin fingerprints and regression coverage, so no repair lane is needed for this issue.

## What This Changes

The issue reports that `clawhub update` falsely flags freshly installed skills as `local changes (no match)` because install metadata lacks a clean-content fingerprint baseline.

## Best Possible Solution

Keep the stored-origin fingerprint fallback for clean installs, and track any remaining legacy-origin migration or sync-reporting behavior as narrower follow-ups.

## Reproduction Assessment

No, not against current main: fresh installs now persist a computed origin fingerprint, and update can match that stored fingerprint when registry resolution returns no match. I verified this by source inspection and existing regression coverage rather than a live registry run.

## Solution Assessment

Yes. The current stored-fingerprint fallback is the narrow maintainable fix for the reported clean-install path because it preserves real local-edit detection while avoiding the false warning for bytes installed by the CLI.
