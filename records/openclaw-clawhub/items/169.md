## Summary

Keep open. Current main still has a source-reproducible clean-install fingerprint mismatch path: generated root `_meta.json` from downloads is installed and hashed locally, while registry resolve compares against published-version fingerprints that do not include that generated file.

## What This Changes

Fix `clawhub update` so clean installs and force-updates are recognized as matching published versions instead of reported as local changes.

## Best Possible Solution

Define one canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve so generated ClawHub metadata is excluded or never installed while authored files still protect against real local edits.

## Reproduction Assessment

Yes. Source inspection shows a metadata-bearing skill download installs root `_meta.json`, local update hashing includes root JSON files, and registry resolve compares against published fingerprints that omit generated metadata.

## Solution Assessment

Yes. The narrow maintainable fix is to align the compared fingerprint inputs across CLI and backend paths, not to require `--force`, manual `.clawhubignore`, or a broad JSON ignore.

Related context: https://github.com/openclaw/clawhub/issues/178, https://github.com/openclaw/clawhub/issues/416, and https://github.com/openclaw/clawhub/issues/937. Define a shared fingerprint boundary for published skill content across publish, download, install, update, sync, and resolve. Prefer a small shared helper or explicit contract over ad hoc ignores. Add regression coverage for a metadata-bearing downloaded archive extracted to a skill directory, update/sync fingerprint resolution, and a real authored local edit still blocking non-forced update. Update `specs/` if the fingerprint contract is clarified.
