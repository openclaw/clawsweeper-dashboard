## Summary

Keep open under the ClawHub cleanup policy for issues, even though current main and the shipped CLI now cover the reported fresh-install path by storing and trusting an install fingerprint when registry resolution returns no match.

## What This Changes

The issue reports that `clawhub update` falsely prints `local changes (no match)` immediately after a fresh `clawhub install`, forcing users to use `--force`.

## Best Possible Solution

Keep the shipped install-fingerprint fallback for fresh installs and force-updates; track any legacy no-fingerprint self-healing separately if maintainers want to remove the one-time force-update escape hatch.

## Reproduction Assessment

No high-confidence current-main reproduction path remains for the reported fresh-install case. Source inspection and regression coverage show fresh installs now write a fingerprint and update trusts that stored fingerprint when registry resolution returns no match.

## Solution Assessment

Yes. The current implementation is the narrow maintainable solution for the central symptom because it preserves registry fingerprint matching while adding a local clean-install baseline instead of weakening content hashing.
