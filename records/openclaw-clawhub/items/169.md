## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-level clean-install mismatch path: update hashes installed files and refuses when resolve cannot match that fingerprint, while downloads can add generated root metadata outside the publish-time fingerprint inputs.

## What This Changes

The issue requests that `clawhub update` stop reporting `local changes (no match)` for freshly installed or freshly force-updated skills.

## Best Possible Solution

Define one canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve so clean installs resolve while real local edits still block non-forced updates.

## Reproduction Assessment

Yes. The issue gives concrete install/update steps, and current main still shows a source-level path where downloaded/generated metadata can be included in local hashing but absent from resolve-side publish fingerprints; I did not run the CLI because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable direction is to align the fingerprint contract across the CLI and backend. Bypassing resolve or broadly ignoring file classes would be riskier because it could weaken local-change detection.

## Repair Work Prompt

Fix the false `local changes (no match)` result from `clawhub update` after a fresh install or force update. Define and implement one canonical skill-content fingerprint contract across CLI local hashing, downloaded skill zips, publish-time stored fingerprints/indexes, resolve API, and sync. Account for generated root `_meta.json` and `.clawhub/origin.json` without broadly ignoring user-owned JSON or weakening tamper detection. Fresh install or force-update followed immediately by update should resolve to the installed/latest version and report up to date; editing a real skill file such as `SKILL.md` should still block non-forced updates. Add focused regression coverage. Do not change moderation blocking, registry auth, package download behavior, or unrelated lockfile semantics.
