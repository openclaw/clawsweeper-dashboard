## Summary

Keep open: current main still has the reported slug-conflict path, but this PR should not merge as-is because it turns publish permission into ownership-transfer permission for publisher-only org members. The MEMBER-authored, security-sensitive ownership path needs maintainer review.

## What This Changes

The PR adds a backend owner-migration branch in `insertVersion` plus tests for preserving a skill while republishing it under a different owner.

## Best Possible Solution

Keep the bug fix direction, but require documented ownership-transfer authority on both source and destination publishers and add regression tests for publisher-only denial cases.

## Reproduction Assessment

Yes. Current main source shows the web publish path passes the selected owner, then `insertVersion` throws the slug-taken error whenever the existing skill belongs to a different publisher.

## Solution Assessment

No. The backend-only approach targets the right failure, but the proposed permission check is too broad for an ownership transfer; admin/owner-level authority should be required or explicitly approved by maintainers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Require admin rights before moving ownership:** `convex/skills.ts:7747`
  - body: This branch lets any `publisher` member of the current owner publisher move the skill to another publisher during a normal version publish. The org ownership spec reserves transfer control for owner/admin-level authority, and package transfers already require admin access on both sides, so a publisher-only member of an org could move an org skill into their personal publisher by publishing a version. Require transfer-level rights on both source and destination before patching ownership.
  - confidence: 0.93

## Security Review

Status: needs_attention
