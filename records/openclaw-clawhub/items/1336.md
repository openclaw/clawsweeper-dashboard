## Summary

Keep open: ClawHub cleanup policy keeps issues open, and current production-tagged main still shows the GitHub OAuth start route failing with HTTP 500 instead of redirecting to GitHub.

## What This Changes

The issue reports that the ClawHub “Sign in with GitHub” button does not respond, suspected to be caused by missing or incorrect GitHub OAuth Client ID configuration.

## Best Possible Solution

Restore production GitHub OAuth so the sign-in start route redirects to GitHub, callback completion establishes a session for valid active users, failed starts show a clear error, and deleted or banned users remain blocked.

## Reproduction Assessment

Yes. A live read-only check against production-tagged current main reproduced the start-route failure: the site root is reachable, but the canonical and direct Convex GitHub sign-in endpoints return HTTP 500 instead of redirecting to GitHub OAuth.

## Solution Assessment

Not applicable because there is no proposed patch. The narrowest maintainable path is operator auth/config triage first, followed by a focused source fix only if logs prove the code is mishandling failed starts, callbacks, or sessions.
