Keep open. Current main still does not prove the reported GitHub sign-in no-op/configuration failure is fixed: the header path can still ignore a resolved failed sign-in start, OAuth remains deployment-env driven, and related callback/session reports are still open.

## What This Changes

The issue reports that the ClawHub “Sign in with GitHub” button repeatedly does nothing, possibly because the GitHub OAuth client ID is missing or misconfigured.

## Best Possible Solution

Trace the production GitHub OAuth flow end to end, then either fix the deployment OAuth/Convex/Vercel configuration or align the header/callback/session handling so failed starts and failed completions surface a clear user-facing error.

## Reproduction Assessment

No high-confidence production reproduction is available from this issue alone. Source inspection does show a header path where signIn resolving with { signingIn: false } produces no visible error, but production OAuth traces are still missing.

## Solution Assessment

Unclear. A narrow header fix would be maintainable for the silent failed-start path, but maintainer/operator validation should determine first whether the real fix is code, OAuth configuration, Convex auth config, or browser/session callback handling.
