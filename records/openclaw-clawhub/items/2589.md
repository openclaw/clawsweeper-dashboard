## Summary

Keep open: the patch fixes the immediate duplicate-row exception, but it leaves other active rows with the same user code authorizable, so one browser code can approve multiple CLI device attempts; it also lacks the real Convex and live device-flow proof required for this runtime-sensitive auth change.

## What This Changes

The PR replaces unique CLI device user-code lookup with duplicate-aware newest-row selection for approval and denial and adds three mocked regression tests.

## Best Possible Solution

Keep the user code single-use by deterministically selecting one active attempt, atomically invalidating every competing live row sharing its hash, and proving through real Convex create/approve/poll coverage that only the selected device can receive a token.

## Reproduction Assessment

Yes at source level: inserting two rows with the same `userCodeHash` makes current main’s indexed `.unique()` lookup throw, and the linked production report supplies repeatable symptoms and request IDs. This review did not independently execute the live production flow.

## Solution Assessment

No as currently written: duplicate-tolerant lookup is the correct narrow direction, but the selected authorization must also invalidate competing active rows and be verified through the real Convex create-to-poll path.
