## Label Justifications

- `P1`: The bug blocks the CLI device-login workflow, preventing affected users and headless agents from authenticating and publishing.
- `merge-risk: 🚨 auth-provider`: The patch changes which duplicate device-auth record receives approval or denial when a user code collides.

## Summary

Current main still uses `.unique()` for device user-code approval and denial, so this PR remains necessary; the proof-backed patch is focused and no discrete correctness defect is evident.

## What This Changes

The PR replaces unique CLI device user-code lookup with bounded newest-first duplicate handling, expires stale pending rows, and adds focused approval and denial regression tests.

## Best Possible Solution

Land the bounded indexed duplicate handling with its regression coverage, while retaining the documented newest-active-pending collision policy and monitoring for evidence that duplicates require a separate data-integrity repair.

## Reproduction Assessment

Yes. A disposable real Convex deployment with duplicate `userCodeHash` rows reproduced the relevant runtime state and demonstrated approval, polling, expiration, terminal preservation, and denial after the fix.

## Solution Assessment

Yes. Replacing `.unique()` with a bounded indexed read and deterministic state selection directly fixes the crash while preserving existing device-flow states and avoiding an unrelated schema migration.
