Keep open. ClawHub policy does not allow auto-closing issues in this repository, and current main still contains the reported dashboard-to-plugin-detail path where an owner-visible package can land on the generic `Plugin not found` state. The affected package URL/name, account, sign-in state, and package-detail API response are still needed before this can become a safe code fix.

## What This Changes

The issue reports that a plugin the reporter published is visible to them but opens to a `Plugin not found` page.

## Best Possible Solution

Collect the affected package URL/name, publisher account, signed-in state, and package-detail API response; if reproduced, update the package detail API/UI to distinguish owner-visible private, pending, blocked, auth-failed, fetch-failed, and truly missing states while preserving public 404 behavior for unreadable packages.

No. Source inspection confirms the route that turns a package-detail 404 into `Plugin not found`, but the report lacks the specific package/account/API response needed to reproduce the reporter's failure against current main.

## Solution Assessment

Unclear. Improving owner-visible package-detail diagnostics is likely the maintainable direction after a maintainer gathers the missing package details, but changing 404 semantics without that data could hide the wrong underlying cause.
