## Summary

Keep open: the PR is useful but not merge-ready because GitHub-backed verification scans only the first 256 KiB of each text file, while the new detector only reports after seeing later content, so a larger padding gap can still hide the later payload. The branch also intentionally maps padding-only suspicious findings to clean installability, which needs maintainer sign-off.

## What This Changes

The branch adds a static moderation reason for extreme context padding, wires it into GitHub-backed skill verification, retains structured moderation evidence, adds regression tests, and updates moderation specs.

## Best Possible Solution

Keep the static signal, fix the GitHub-backed scan-boundary gap, then merge only if maintainers explicitly accept the clean installability exception with retained moderation evidence.

## Reproduction Assessment

Yes for the changed behavior: the PR body includes copied runStaticPublishScan output for a padded SKILL.md, and the branch adds focused tests for within-cap blank-line and same-line padding. Source inspection also gives a high-confidence reproduction for the untested GitHub-backed cap edge: header plus more than 256 KiB of whitespace plus later content is truncated before the detector sees the later content.

## Solution Assessment

No, not yet: the detector is narrow and useful, but the GitHub-backed scan-cap edge should be fixed before maintainers decide the clean installability exception. The safer alternative is full or truncation-aware scanning for this signal plus explicit policy acceptance of any padding-only clean mapping.
