## Summary

The context-padding detector is useful and well demonstrated, but the GitHub compatibility workaround rewrites padding-only results to `clean`, discarding the reviewer-facing security signal that the change is intended to preserve.

## What This Changes

The PR adds deterministic detection of extreme blank-line or whitespace padding, introduces a new moderation reason code, documents the security intent, and keeps padding-only GitHub-backed skills promoted by normalizing their verification status to clean.

## Best Possible Solution

Separate GitHub promotion eligibility from moderation evidence: keep context-padding-only artifacts installable while persisting the reason and finding in reviewer-visible state for ClawScan.

## Solution Assessment

No, not as currently represented. The detector is appropriately narrow, but compatibility should be separated from evidence persistence instead of replacing the suspicious result with clean.
