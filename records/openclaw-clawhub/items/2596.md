## What This Changes

The branch pins package-publish examples to the v0.20.2 commit, adds Dependabot guidance, and automates post-release documentation updates through a new workflow, updater script, tests, and release skill.

## Best Possible Solution

Use the selected GitHub App token for checkout and every branch push, preserve the primary/fallback token behavior, and demonstrate that both initial creation and retrying an existing generated PR trigger the expected checks.

## Reproduction Assessment

Yes, from source inspection: checkout persists the default workflow credential, the branch push occurs with that credential, and the app token is only assigned to the later PR command. The problematic case is retrying or updating an already-open generated PR.

## Solution Assessment

No, not yet. Pinning the reusable workflow to a full SHA is appropriate, but branch pushes must use the GitHub App token so initial and repeated docs-PR updates reliably trigger validation.

Full review comments:

- **[P2] Push docs updates with the GitHub App token:** `.github/workflows/clawhub-cli-package-publish-docs.yml:28-31`
  - body: `actions/checkout` persists `github.token`, so the later `git push` does not use either app token; those tokens are only exposed to `gh pr create/edit`. Initial PR creation may start checks through the app-authenticated `opened` event, but retrying an existing PR pushes with `GITHUB_TOKEN`, which can suppress its `synchronize` workflows and leave stale checks. Create/select the app token before checkout and use it for checkout or explicitly reconfigure the remote before pushing.
