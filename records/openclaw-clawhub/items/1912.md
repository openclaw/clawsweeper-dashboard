## Summary

Keep open: current main has partial scanner coverage for the originally reported Bash pattern, but the live published skill remains visible as v2.0.16 with clean static analysis while still uploading local file bytes through the Python implementation.

## What This Changes

The issue reports a published PaddleOCR skill that can read arbitrary local files, base64-encode them, and upload them to a configured third-party OCR API with a bearer token.

## Best Possible Solution

Keep the issue open until maintainers settle the publisher remediation and ClawHub scanner/moderation policy for Python local-file-to-external-API uploads, then either flag/hide the current version or document/guard it to the accepted standard.

## Reproduction Assessment

Yes. The current published v2.0.16 path is statically reproducible: --file-path reaches _load_file_as_base64(), which reads bytes and base64-encodes them before httpx posts the JSON with the bearer token.

## Solution Assessment

No. The current main change is a useful partial mitigation for the original shell wrapper, but it does not resolve the live Python implementation or the clean public scan state.
