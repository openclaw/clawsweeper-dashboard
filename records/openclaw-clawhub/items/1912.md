This should stay open: current main has useful scanner coverage for the original Bash upload pattern, but the live published skill still exposes a Python local-file-to-external-API upload path and a maintainer explicitly kept the issue open for publisher-side remediation/review.

## What This Changes

The issue reports that the published paddleocr-doc-parsing skill can read arbitrary local file paths, base64-encode the bytes, and upload them to an external OCR API with bearer-token auth.

## Best Possible Solution

Keep the issue open until publisher remediation and ClawHub scanner/moderation policy for Python local-file uploads are both reviewed and reflected in the live skill state.

## Reproduction Assessment

Yes. Static inspection of current live v2.0.16 shows --file-path flows to _load_file_as_base64(), then Path.read_bytes()/base64.b64encode(), and finally httpx.Client.post() with an Authorization token.

## Solution Assessment

No. The current main mitigation covers the original Bash wrapper shape, but it does not fully resolve the live Python implementation or the public null static-scan/moderation state.
