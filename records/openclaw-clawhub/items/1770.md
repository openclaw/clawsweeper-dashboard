Keep this open for maintainer follow-up. Current main still rejects owner renames when the target slug is held by a different `skills.by_slug` row, including soft-deleted rows, and the requested resolution is production-data administration with ownership verification rather than a source cleanup.

## What This Changes

The issue requests purging or releasing the soft-deleted `creator-scraper-cv` slug so the organization-owned skill can be renamed to that slug.

## Best Possible Solution

A maintainer should verify production slug state and requester control of both accounts, then use the least destructive admin path to release the old slug and confirm the org-owned rename succeeds.

## Reproduction Assessment

Yes for the general source path: current main rejects owner rename when a different `skills.by_slug` row already has the target slug, and the schema keeps soft-deleted rows in that index. The exact production row for `creator-scraper-cv` was not verified through read-only inspection.

## Solution Assessment

Yes for this support request: maintainer-controlled production administration is the narrowest safe path. Changing global rename semantics to ignore soft-deleted rows would be broader than this named slug request and would need separate policy review.
