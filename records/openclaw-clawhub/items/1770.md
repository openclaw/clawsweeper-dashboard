## Summary

Keep this open for maintainer follow-up. Current main still rejects owner renames when the target slug is held by any different `skills.by_slug` row, including soft-deleted rows, but the requested purge is ownership-sensitive production administration rather than a source cleanup that this workflow should close.

## What This Changes

The requester wants the soft-deleted personal-account slug `creator-scraper-cv` purged so an organization-owned skill can be renamed to that slug.

## Best Possible Solution

A maintainer should verify production slug state and requester control of both accounts, then release the slug through the least destructive admin path and confirm the organization-owned rename succeeds.

## Reproduction Assessment

Yes for the general source path: current main rejects an owner rename when any different `skills.by_slug` row already has the target slug, and the schema keeps soft-deleted rows in that index. The exact production row for `creator-scraper-cv` was not verified through read-only inspection.

## Solution Assessment

Yes. Maintainer-controlled production administration is the narrowest maintainable solution for this specific slug release; changing global rename semantics for all soft-deleted slugs would be broader and would need policy decisions.
