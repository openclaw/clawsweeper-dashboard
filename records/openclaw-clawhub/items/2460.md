## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Make the feed data query internal:** `convex/feeds.ts:59`
  - body: Registering `rootFeed` with `query` makes raw digest rows callable directly, bypassing the HTTP feed shape, attestation, cache headers, and any route-level safeguards. Use an `internalQuery` behind the HTTP handler or move the sanitization boundary into the query result itself.
  - confidence: 0.92
- **[P1] Rate-limit feed reads before querying:** `convex/httpApiV1/feedsV1.ts:405-408`
  - body: The feed handler calls `ctx.runQuery` without the shared read limiter used by existing public catalog endpoints. Apply `applyRateLimit(ctx, request, "read")` before the database work and return its headers on success and error responses.
  - confidence: 0.9
- **[P1] Use catalog visibility guards for feed entries:** `convex/feeds.ts:111-112`
  - body: Filtering digest rows with only local feed checks can drift from the canonical catalog rules for public skill/package visibility, owner resolution, and public conversion. Build feed entries through the existing catalog helpers or the same visibility checks so hidden or unresolved-owner rows cannot leak.
  - confidence: 0.86
- **[P2] Bound feed reads to the requested page size:** `convex/feeds.ts:72-97`
  - body: Even `limit=1` currently reads up to 5,000 skills and up to 5,000 rows from each plugin family before filtering and slicing. Use paginated catalog helpers or page-sized batches so small feed requests do not burn large read budgets.
  - confidence: 0.88
