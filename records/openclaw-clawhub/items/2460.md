## What This Changes

The PR adds five public ClawHub catalog-feed endpoints, lane-specific skill/plugin selection, deterministic feed serialization with SHA-256 hashes, route constants, handler tests, and HTTP API documentation.

## Best Possible Solution

Define the feed schema and lane guarantees in `specs/`, share canonical catalog eligibility and trust helpers with existing APIs, paginate/index by lane so every matching entry is reachable, and validate representative live responses and load behavior on real Convex state.

## Reproduction Assessment

Yes for the review finding: source inspection shows that each family is capped before lane predicates run, so more than 5,000 newer nonmatching rows can hide older matching entries. The broader feature itself is a new capability, not a bug report.

## Solution Assessment

No in its current form; a feed adapter is plausible, but it must use lane-complete indexed pagination and canonical ClawHub trust/visibility policy rather than a capped parallel selection engine.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Filter each lane before applying the 5,000-row cap:** `convex/feeds.ts:74-78`
  - body: The query takes only the newest 5,000 rows from each family and then applies `official`/`community`/`reviewed` predicates. Once a family exceeds that size, newer nonmatching rows can crowd out older matching entries, so even `limit=5000` is not a complete root lane and consumers can never page to the omitted entries. Add indexed lane fields/pagination or otherwise continue reading until the requested number of matching rows is obtained.
  - confidence: 0.96
