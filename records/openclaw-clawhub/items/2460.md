## Summary

This draft is still necessary only if maintainers want ClawHub to produce feeds, but it is not merge-ready: the emitted JSON is incompatible with the linked OpenClaw consumer, the unauthenticated query reads up to 15,000 digest documents per request regardless of the default 500-entry limit, the branch conflicts with current main, and no real runtime proof is provided.

## What This Changes

Adds five public HTTP endpoints that expose all, official, community, and reviewed ClawHub skill/plugin catalog feeds with SHA-256 hashes, tests, schema routes, and API documentation.

## Best Possible Solution

Define one versioned feed schema shared by producer and consumer, implement indexed or cursor-based lane retrieval that scales with the requested page, centralize current public-safety predicates, record reviewed-lane trust invariants in `specs/`, and prove the rebased endpoints with real Convex and OpenClaw consumption.

Yes for the review defects: source comparison deterministically shows the producer/consumer field mismatch, and the query code shows three independent 5,000-row reads before slicing. Real runtime behavior of the new endpoints has not been demonstrated.

## Solution Assessment

No, not in its current form. A ClawHub feed producer may be useful, but it should share a versioned contract with its consumer and use scalable indexed or cursor-based retrieval before becoming a public API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.98

Full review comments:

- **[P1] Make the feed document match the linked consumer schema:** `convex/httpApiV1/feedsV1.ts:121-127`
  - body: The linked OpenClaw parser requires root `id` and entry fields such as `name` and `sourceUrl`, but this producer emits root `feedId`, entry `title`, and a nested `source` object. As a result, OpenClaw rejects the document for the missing feed id or drops the producer-specific fields, so the advertised shared feed stack cannot interoperate. Define one schema and add a producer-to-consumer fixture test.
  - confidence: 0.99
- **[P1] Avoid reading 15,000 digests for every feed request:** `convex/feeds.ts:72-78`
  - body: The query always takes 5,000 skills plus 5,000 rows from each plugin family before applying the lane filters and requested limit. Thus a default `limit=500` unauthenticated request still performs up to 15,000 document reads, and repeated requests can hit Convex bandwidth limits or degrade the public API. Add lane-selective indexes/denormalized fields or cursor-based bounded retrieval so reads scale with the requested result size.
  - confidence: 0.97
