Full review comments:

- **[P1] Do not expose the root feed builder as a public query:** `convex/feeds.ts:59`
  - body: `rootFeed` is registered with the public `query` wrapper while it can read thousands of catalog digest rows before filtering. That lets callers invoke the expensive feed builder directly outside the HTTP rate-limit, stored-publication, and cache-validator protections; make this internal or serve a stored/quota-protected publication instead.
  - confidence: 0.91
- **[P1] Serve root feeds from published snapshots:** `convex/httpApiV1/feedsV1.ts:405-408`
  - body: The handler builds every public feed response by running the live root-feed query. Current main's hosted feed contract serves stored bytes with ETag/304 caching and explicitly says not to make feed requests request-time dynamic, so this path can bypass the feed availability model.
  - confidence: 0.9
- **[P1] Use a distinct ID for the root official feed:** `convex/httpApiV1/feedsV1.ts:18`
  - body: `clawhub-official` is already the ID for the hosted plugin feed at `/api/v1/feeds/plugins` with schema version 2 and a different install-candidate contract. Reusing it here can make clients that pin, cache, or compare feeds by ID treat incompatible feeds as the same source.
  - confidence: 0.88
- **[P2] Publish the full root feed API contract:** `docs/http-api.md:139`
  - body: The PR documents and exposes new public v1 feed routes, but the published OpenAPI document is not changed and the Convex generated API declarations do not include the new `feeds` module. Clients and deploy-time generated surfaces should not diverge from the documented API.
  - confidence: 0.84
