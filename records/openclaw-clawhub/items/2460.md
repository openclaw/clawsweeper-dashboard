## Summary

This PR remains useful but is not merge-ready: the reviewed lane bypasses canonical trust signals, the public query performs a large fixed read regardless of requested limit, and `sourceRevision` does not identify content revisions. The branch also lacks real deployed endpoint proof and is currently conflict-marked.

## What This Changes

Adds five public ClawHub HTTP endpoints that publish an index plus all, official, community, and reviewed catalog feeds with deterministic entries and SHA-256 attestations.

## Best Possible Solution

Define the feed contract in `specs/`, derive lane membership from shared canonical visibility, installability, official-publisher, and moderation helpers, make retrieval bounded or precomputed, emit a content-derived revision, and validate representative live requests on a real Convex deployment.

- **[P1] Use canonical trust checks for the reviewed lane:** `convex/feeds.ts:29`
  - body: Do not classify a skill as reviewed from only `deprecated` and `isSuspicious`.   - confidence: 0.94
- **[P1] Bound reads by the requested feed size:** `convex/feeds.ts:78`
  - body: The query always takes 5,000 skills plus 5,000 entries from each plugin family before filtering, so even `limit=1` can read 15,000 documents. This public endpoint will become an avoidable availability and billing hotspot as the catalog grows; use indexed lane data, cursor pagination, or a precomputed feed so work is bounded by the requested page size.
  - confidence: 0.97
- **[P1] Make sourceRevision change with feed contents:** `convex/httpApiV1/feedsV1.ts:354`
  - body: The revision string only encodes feed id, limit, and truncation, so two different catalog snapshots return the same `sourceRevision`. Clients using this field for update detection or lifecycle tooling will miss changes even though the attestation hash changes; derive the revision from the canonical content/hash or a real monotonic catalog revision.
  - confidence: 0.96
