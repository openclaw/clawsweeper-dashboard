## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Make the feed data query internal:** `convex/feeds.ts:59`
  - body: This registers `rootFeed` with `query`, so callers can invoke the Convex function directly and receive raw digest rows without the HTTP feed shape, attestation, cache headers, or route-level safeguards. Register this as an internal query and call it through `internal`, or move the sanitization boundary into the query itself.
  - confidence: 0.92
- **[P1] Rate-limit feed reads before querying:** `convex/httpApiV1/feedsV1.ts:405-408`
  - body: The feed handler calls `ctx.runQuery` without the shared read limiter used by the existing public catalog endpoints. Anonymous requests can repeatedly trigger the expensive feed query; apply `applyRateLimit(ctx, request, "read")` first and return its headers on success/error responses.
  - confidence: 0.9
- **[P1] Reuse the catalog skill visibility guard:** `convex/feeds.ts:111-112`
  - body: `matchesSkillFeed` only checks soft-delete/moderation flags, but the current skill catalog also requires `toPublicSkill(...)` plus resolved public owner info before a row is visible. Feed skills should be built through `listPackageCatalogPage` or the same visibility helper so deleted/deactivated or unresolved owners cannot leak through the new feed.
  - confidence: 0.86
- **[P2] Bound feed reads to the requested page size:** `convex/feeds.ts:78`
  - body: Even `limit=1` currently reads up to 5,000 skills and up to 5,000 rows from each plugin family before filtering and slicing. Use paginated catalog helpers or page-sized batches so small feed requests do not burn large read budgets.
  - confidence: 0.88
