## Summary

This PR should stay open for maintainer review: it adds useful official-publisher badge/policy work that is not on current main, but the current trust predicate has a security-boundary blocker around self-service ownership of the `openclaw` handle.

## What This Changes

The branch derives Official publisher/package state from the `openclaw` org/member policy, replaces Verified UI with Official badges across publisher/skill/plugin surfaces, and documents the narrowed policy.

## Best Possible Solution

Gate Official status on an admin-reserved or admin-created `openclaw` publisher, and make the trustedPublisher-to-OpenClaw-only migration explicit before merge.

## Reproduction Assessment

Yes, source-level: on a deployment without an existing `@openclaw` publisher, an authenticated user can call the self-service org creation path with handle `openclaw`, and the new helper then treats that org as Official. I did not execute a live Convex mutation.

## Solution Assessment

No, not yet. Deriving Official from OpenClaw membership is maintainable once the `openclaw` publisher is protected, but handle-only trust is not the narrowest safe security boundary.
