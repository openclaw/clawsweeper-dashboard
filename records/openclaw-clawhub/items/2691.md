## Summary

Keep open: current main and the live public page show the all-time install count as the primary `Installs` metric, but they do not show downloads or explain in place why downloads can be much higher, and no open canonical item owns this exact public-page clarification.

## What This Changes

The issue asks for the public skill page to distinguish downloads/archive fetches from telemetry-derived current and all-time installs when those stats diverge.

## Best Possible Solution

Keep installs as the primary public adoption metric, then add a maintainer-approved accessible explanation or secondary metric on skill detail pages that defines all-time/current installs and distinguishes them from archive downloads.

## Reproduction Assessment

Yes, from source and live page inspection: the public page renders `Installs` with value `1`, and current main maps that label to `formattedStats.installsAllTime` without showing downloads in the same metadata block. I did not run a local browser instance.

## Solution Assessment

Unclear until maintainers choose the UX policy. A narrow accessible explanation or secondary metric is safer than reverting downloads to the primary adoption count.
