## Summary

Keep open: current main intentionally renders the public skill detail metric as all-time telemetry-backed `Installs`, but it still does not show downloads or explain on the detail page why downloads can be much higher. The remaining work is a maintainer UX/product decision, not an already-implemented fix.

## What This Changes

The issue requests that public skill detail pages distinguish archive/download counts from telemetry-backed current and all-time install counts.

## Best Possible Solution

Keep installs as the primary public adoption metric if that remains the product direction, then add maintainer-approved visible downloads context or accessible explanatory copy on detail pages and align adjacent public surfaces as needed.

## Reproduction Assessment

Yes, at source level and with live API data: current main renders `Installs` from all-time install stats, while the reported skill still has downloads far above install counters. I did not run a local browser because this review is read-only.

## Solution Assessment

Unclear until maintainers decide the public metric hierarchy. A narrow explanation or secondary downloads metric is safer than silently reverting installs back to the primary adoption count.
