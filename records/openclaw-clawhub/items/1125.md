## Summary

Keep this PR open. Current main does not implement install-time OS compatibility checks, but the proposed patch has correctness issues around OS value normalization, version-scoped metadata, and backward-compatible schema parsing that should be addressed before merge.

## What This Changes

The PR adds `metadata.os` parsing to the CLI skill detail schema, rejects `clawhub install` on host OS mismatch unless `--force` is used, and updates the install `--force` help text.

## Best Possible Solution

Revise the PR so install compatibility uses the same normalized OS vocabulary as docs/UI, validates platform metadata for the selected version, preserves backward-compatible parsing when metadata is absent, and adds focused CLI tests for compatible, incompatible, alias, `--version`, and `--force` cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Normalize OS aliases before rejecting installs:** `packages/clawhub/src/cli/commands/skills.ts:100`
  - body: The new guard compares `metadata.os` directly with `platform()`, but documented skill metadata uses values like `macos` and `windows` while Node reports `darwin` and `win32`. On those hosts, supported skills can be rejected unless both sides are normalized before the membership check.
  - confidence: 0.94
- **[P2] Validate the selected version's OS metadata:** `packages/clawhub/src/cli/commands/skills.ts:100`
  - body: The check runs before `versionFlag` is resolved and reads metadata from the skill detail response, which current main derives from the latest version. Installing an older version after the latest release changes its OS list can reject a compatible version or allow an incompatible one; validate metadata for `resolvedVersion` instead.
  - confidence: 0.88
- **[P2] Keep metadata optional in the response schema:** `packages/clawhub/src/schema/schemas.ts:203-206`
  - body: The added `metadata` field is required by the schema, but older registries or responses can omit this newly consumed key. The install code already treats it as optional, so the schema should allow the field to be absent as well as null.
  - confidence: 0.82
