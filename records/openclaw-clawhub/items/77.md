## Summary

Keep open. Current main and v0.22.0 still accept any non-empty skill display name across the backend publish path, web publish form, GitHub import path, and CLI/API schema; implementing a cap still needs a maintainer product decision on the exact limit and legacy behavior.

## What This Changes

The issue requests a maximum length policy for skill titles so displayed skill names stay short and concise.

## Best Possible Solution

Define the public skill display-name policy first, then enforce the chosen maximum in shared backend publish validation with matching web, GitHub import, CLI/API schema, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main accepts any non-empty displayName in the shared publish, web publish, GitHub import, schema, and CLI/API schema surfaces. I did not run a live publish because the validators and entry points already show the missing policy.

## Solution Assessment

Unclear until maintainers choose the maximum length and legacy handling. The maintainable path is a shared backend rule with UI/API/CLI/docs/test alignment, not a presentation-only clamp.
