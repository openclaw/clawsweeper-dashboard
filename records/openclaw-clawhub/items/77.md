## Summary

Current main still accepts any non-empty skill display name, so the requested constraint remains necessary; the unresolved maximum and grandfathering behavior are product decisions rather than a safely inferred bug fix.

## What This Changes

Define and enforce a maximum length for skill titles/display names so catalog entries remain concise.

## Best Possible Solution

Define a documented Unicode-aware display-name maximum, enforce it through one shared validator across every publish/import entry point with a clear error, and explicitly grandfather existing records unless maintainers approve a migration.

## Reproduction Assessment

Yes at source level: on current main, any non-empty trimmed `displayName` passes the canonical publish validation and can be persisted. No runtime exercise is needed to establish that the requested limit is absent.

## Solution Assessment

No, the issue alone is not yet an implementation-ready solution because it leaves the threshold and upgrade behavior undefined. A shared backend validator with explicit legacy handling is narrower and more consistent than UI-only truncation.
