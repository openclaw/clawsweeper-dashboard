## Summary

Keep open. Current main still accepts any non-empty skill display name through the shared server publish path and stores it without a maximum-length constraint, so the requested policy is not implemented; maintainers still need to choose the limit and compatibility treatment for existing long titles.

## What This Changes

Add an enforced maximum length for skill display titles so catalog names remain concise.

## Best Possible Solution

Define one documented display-name maximum in shared server validation, apply matching feedback across web, GitHub import, CLI/API paths, and grandfather unchanged legacy over-limit names unless maintainers explicitly choose a migration.

## Reproduction Assessment

Yes at source level: current main’s shared publish path rejects only empty display names, and the persisted skill field has no maximum-length validator. A live oversized publish was not attempted because this was a read-only review.

## Solution Assessment

Not fully specified yet. A centralized server rule with matching client feedback is the maintainable approach, but maintainers must select the maximum and decide whether existing over-limit names are grandfathered or migrated.
