## Summary

Keep open. Current main still lacks REST/CLI owner-targeted skill publishing, so this PR is not obsolete; the proposed approach matches the existing server-authorized web publish path, with a required changelog cleanup before merge.

## What This Changes

The PR adds optional `ownerHandle`/`--owner` support across skill publish schemas, v1 JSON and multipart parsing, server-side publisher resolution, CLI payload generation, generated schema artifacts, regression tests, and a small maintenance lint cleanup.

## Best Possible Solution

Land this PR or an equivalent focused patch after adding the required changelog entry, preserving server-side publisher resolution and regression coverage for schema parsing, JSON publish, multipart publish, and CLI payload generation.

## Reproduction Assessment

Yes. Inspecting current main gives a high-confidence reproduction path: publish an existing org-owned skill slug through `/api/v1/skills` or the CLI, and the request cannot carry an owner target, so it falls back to the actor’s personal publisher path described in #1814.

## Solution Assessment

Yes. Resolving `ownerHandle` server-side with `minimumRole: "publisher"` and passing only `ownerPublisherId` into `publishVersionForUser` is the narrow maintainable fix; the branch still needs the required release-note entry before merge.
