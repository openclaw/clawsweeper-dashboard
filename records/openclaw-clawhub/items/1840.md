## Summary

Keep open. Current main still lacks REST/CLI owner-targeted skill publishing, and this unmerged PR remains the focused implementation candidate for the linked org-owned publish gap.

## What This Changes

The PR threads optional ownerHandle/--owner support through skill publish schemas, JSON and multipart REST parsing, server-side publisher resolution, CLI payload generation, generated schema artifacts, and regression tests.

## Best Possible Solution

Review and merge this PR, or land an equivalent focused patch, so /api/v1/skills and both CLI publish commands can target an org/user publisher handle through server-side publisher-role authorization with JSON, multipart, CLI, and schema regression coverage.

## Reproduction Assessment

Yes. Current-main source inspection shows ownerHandle cannot be accepted, preserved, resolved, or sent by the REST/CLI skill publish path; the linked report’s Slug is already taken failure follows from that personal-publisher fallback.

## Solution Assessment

Yes. Resolving ownerHandle server-side with the existing publisher-role resolver before calling publishVersionForUser matches the web action’s authorization model and is the narrow maintainable fix; adding an auth failure-path test would further strengthen the patch.
