## Summary

Keep open. Current main still lacks org-owner targeting for REST/CLI skill publishes, while this PR is the focused implementation candidate and needs maintainer review, rebase/mergeability handling, and final CI rather than cleanup closure.

## What This Changes

The PR threads optional `ownerHandle`/`--owner` through publish schemas, CLI payloads, REST JSON and multipart parsing, server-side publisher resolution, generated schema artifacts, and regression tests.

## Best Possible Solution

Land this PR or an equivalent rebased patch that adds `ownerHandle`/`--owner`, resolves the publisher server-side with publisher-role authorization, and keeps paired schema, JSON, multipart, and CLI regression coverage.

## Reproduction Assessment

Yes. Read-only inspection of current main shows REST/CLI publish payloads cannot carry `ownerHandle`, so an org-owned slug update still reaches `publishVersionForUser` without `ownerPublisherId` and follows the existing slug ownership check.

## Solution Assessment

Yes, with maintainer review. The PR mirrors the existing web action pattern by resolving `ownerHandle` through the internal publisher resolver and passing only the derived `ownerPublisherId`; a failure-path test and rebase are the main remaining hardening steps.
