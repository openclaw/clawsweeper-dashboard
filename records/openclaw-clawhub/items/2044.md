## Summary

Keep this open: current main still has different suspicious-skill defaults between unified search and the skills directory, and choosing the intended visibility policy needs maintainer product/security judgment.

## What This Changes

Make skill search results consistent between homepage/unified search and `/skills`, or visibly disclose the default suspicious-skill filter difference.

## Best Possible Solution

Define one shared moderation-visibility rule for public search surfaces, then enforce it consistently with visible UI state and focused regression coverage.

## Reproduction Assessment

Yes for the source-level root cause: current main has unified search and `/skills` calling the same backend action with different `nonSuspiciousOnly` defaults. The exact result-count mismatch still depends on production data for the reported query.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable fix is either to align the defaults through a shared search contract or keep the split and make the active filter visible in unified search.
