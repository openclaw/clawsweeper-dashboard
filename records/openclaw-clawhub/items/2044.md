## Summary

This should stay open under the ClawHub issue policy, and current main still shows the reported split: unified search hides suspicious skills by default while `/skills` does not unless the visible filter is enabled.

## What This Changes

The issue asks ClawHub to align skill-search results across homepage/unified search and `/skills`, or visibly explain the different suspicious-skill filter defaults.

## Best Possible Solution

Define one product rule for suspicious-skill visibility across global and directory search, then encode it through shared search UI state and regression tests.

## Reproduction Assessment

Yes, for the root cause: current main shows the two search entry points calling the same backend action with different `nonSuspiciousOnly` defaults. The exact 1-versus-6 count still depends on production data for the reported query.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable fix is either to align the defaults through a shared contract or keep the split but make the active filter visible in unified search.
