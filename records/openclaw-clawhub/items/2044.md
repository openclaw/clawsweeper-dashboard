## Summary

Keep open: current main still has the public search split, and choosing whether suspicious skills should be hidden everywhere or visibly exposed as a filter needs maintainer product/security judgment.

## What This Changes

The issue asks for homepage/unified search and `/skills` search to return consistent skill results or clearly show when different suspicious-skill filters apply.

## Best Possible Solution

Define one shared moderation-visibility rule for public skill search, enforce it consistently across both entry points, and add focused tests plus visible UI state where the filter can differ.

## Reproduction Assessment

Yes for the source-level mismatch on current main: unified search passes `nonSuspiciousOnly: true`, while `/skills` defaults that flag to false and the backend honors it. The exact 1-versus-6 result count remains production-data-dependent.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable solution is either a shared default or an explicit visible filter/explanation in unified search.
