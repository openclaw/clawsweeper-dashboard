## Summary

This issue should stay open under the ClawHub issue policy. Current main still has two skill-search entry points using different suspicious-skill defaults, and live API mode checks for the reported query show that the flag changes the result count, so the remaining decision is a maintainer moderation/UX policy choice.

## What This Changes

The issue asks for homepage/unified search and the /skills directory to return consistent skill results for the same query or visibly disclose differing suspicious-skill filters.

## Best Possible Solution

Define one moderation visibility rule for global and directory search, then enforce it through shared frontend search state, visible filter/explanation text, and regression tests.

## Reproduction Assessment

Yes for the root cause: current main shows `/search` passing `nonSuspiciousOnly: true` while /skills defaults it to false, and production API checks return 1 versus 6 results for the reported query depending on that flag. I did not run a full browser reproduction against current main, so this is source-reproducible rather than an end-to-end local repro.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable fix is either to align the defaults through a shared contract or keep the split but make the active suspicious-skill filtering visible in unified search.
