## Summary

Keep this open: current main still has different suspicious-skill visibility defaults between unified search and the skills directory, and the reported production query now confirms the filtered/unfiltered result-count split. The remaining decision is a maintainer product/security choice, not conservative cleanup closure.

## What This Changes

Request to align or clearly disclose the default suspicious-skill filtering between homepage/unified search and `/skills` search.

## Best Possible Solution

Define one shared public-search moderation visibility rule, then enforce it consistently through shared defaults or visible filter state with focused regression coverage.

## Reproduction Assessment

Yes for the root cause: current main has unified search hard-coded to `nonSuspiciousOnly: true` while `/skills` defaults it to false, and the live API returns one versus six results for the reported query under those filter settings. The exact browser UI was not run locally.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable path is either a shared default across search surfaces or a visible active-filter indication in unified search, without silently weakening suspicious-skill filtering.
