## Summary

Keep this open: current main still has the reported split between unified search and the skills directory, and the safe fix depends on maintainer policy for suspicious-skill visibility.

## What This Changes

The issue asks for consistent skill search results, or visible explanation of differing default filters, between homepage/unified search and `/skills`.

## Best Possible Solution

Define one shared public-search moderation rule, then enforce it consistently across unified search and `/skills` with visible UI state and focused regression coverage.

## Reproduction Assessment

Yes for the root cause: current main has two search entry points calling the same backend action with different `nonSuspiciousOnly` defaults. The exact result-count mismatch still depends on production data for the reported query.

## Solution Assessment

Unclear until maintainers choose the intended suspicious-skill visibility policy. The narrow maintainable fix is either to align both callers through a shared search contract or keep the split and make the active filter visible in unified search.
