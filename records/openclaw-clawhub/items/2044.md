## Summary

Keep open: current main still uses different suspicious-skill defaults between unified search and the skills directory, and the intended public moderation visibility needs maintainer judgment.

## What This Changes

The issue requests consistent skill search results between homepage/unified search and `/skills`, or visible UI disclosure when different moderation filters apply.

## Best Possible Solution

Define one shared moderation-visibility rule for public skill search surfaces, then enforce it consistently with visible UI state and focused regression coverage.

## Reproduction Assessment

Yes for the root cause: current main has two search entry points calling the same backend action with different `nonSuspiciousOnly` defaults. The exact production count mismatch still depends on live data for the reported query.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable fix is either to align the defaults through a shared search contract or keep the split and make the active filter visible in unified search.
