## Summary

Keep this open: current main still has the reported split, and choosing whether public unified search should hide or disclose suspicious-skill filtering needs maintainer product/security judgment.

## What This Changes

The issue asks for consistent skill results between homepage/unified search and `/skills`, or visible UI disclosure when different suspicious-skill filters apply.

## Best Possible Solution

Define one shared public-search moderation visibility rule, apply it consistently across `/search` and `/skills`, and make any active suspicious-skill filter visible with focused regression coverage.

## Reproduction Assessment

Yes for the source and data-path split: current main sends different `nonSuspiciousOnly` defaults, and the live API returned 6 unfiltered results versus 1 filtered result for the reported query. I did not run a browser session, but source plus API behavior matches the report.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable implementation is to centralize the search visibility contract, then either align the defaults or expose the active filter clearly in unified search.
