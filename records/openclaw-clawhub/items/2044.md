## Summary

This should stay open under the ClawHub issue policy, and current main still contains the reported split: unified search hides suspicious skills by default while /skills leaves that filter off unless selected.

## What This Changes

The issue asks for consistent search results between homepage/unified search and /skills, or a visible explanation when default filters differ.

## Best Possible Solution

Define one product rule for suspicious-skill visibility across global and directory search, then enforce it with shared search defaults, visible UI state, and regression tests.

## Reproduction Assessment

Yes for the root cause: current main shows the two entry points passing different nonSuspiciousOnly defaults into the same backend search action. The exact 1-versus-6 result count still depends on production data for the reported query.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable fix is either to align the defaults through a shared search contract or keep the split but make the active filter visible in unified search.
