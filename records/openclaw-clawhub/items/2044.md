## Summary

Keep this open: current main still has the reported source-level split between unified search and the skills directory, and resolving it requires maintainer policy judgment about suspicious-skill visibility.

## What This Changes

The issue asks for homepage/unified search and `/skills` searches to return consistent skill results for the same query, or to visibly explain differing suspicious-skill filters.

## Best Possible Solution

Define one public-search moderation visibility policy, then enforce it through a shared search contract with either matching defaults or a visible filter/explanation in unified search.

## Reproduction Assessment

Yes for the source-level root cause: current main sends `nonSuspiciousOnly: true` from unified search and `false` by default from `/skills`. The exact production count mismatch still depends on live data for the reported query.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable path is to centralize the visibility rule and cover both search entry points with focused regression tests.
