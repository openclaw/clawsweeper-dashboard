## Summary

Keep this open: current main still has two public search surfaces using different suspicious-skill defaults, and choosing whether to align or disclose that filter needs maintainer product/security judgment.

## What This Changes

The issue requests consistent skill search behavior, or a visible explanation, between homepage/unified search and the `/skills` directory for the same query.

## Best Possible Solution

Define one shared moderation-visibility rule for public skill search, then either align both entry points to that rule or show the active suspicious-skill filter consistently, with focused regression coverage.

## Reproduction Assessment

Yes for the source-level mismatch: current main still sends `nonSuspiciousOnly: true` from unified search and `false` by default from `/skills`. The exact production result-count mismatch for `国泰海通` was not locally reproduced.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable path is either a shared search visibility contract or a visible filter/disclosure on unified search.
