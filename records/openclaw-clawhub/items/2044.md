## Summary

Keep this open: current main still has the reported source-level split between unified search and the skills directory, and the right fix depends on maintainer policy for suspicious-skill visibility.

## What This Changes

The issue asks to make homepage/unified search and `/skills` return consistent skill results for the same query or visibly explain the suspicious-skill filter difference.

## Best Possible Solution

Define a shared suspicious-skill visibility contract for public search surfaces, then enforce it consistently with focused UI and regression coverage.

## Reproduction Assessment

Yes for the root cause: current main has two entry points calling the same backend search action with different `nonSuspiciousOnly` defaults. The exact production result-count mismatch still depends on live skill data for the reported query.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable path is either to align the defaults through a shared search contract or keep the split and make the active filter visible in unified search.
