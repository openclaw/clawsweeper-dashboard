## Summary

Keep open: current main still has the unified-search versus skills-directory default split, and changing it affects suspicious-skill visibility, so maintainers need to choose the intended public search policy.

## What This Changes

The issue asks for homepage/unified search and `/skills` to return consistent skill results for the same query or to visibly explain the suspicious-skill filter difference.

## Best Possible Solution

Define one shared public-search moderation visibility rule and enforce it consistently, either by sharing defaults or by showing the active suspicious-skill filter on unified search with regression coverage.

## Reproduction Assessment

Yes for the source-level mismatch: current main hardcodes `nonSuspiciousOnly: true` in unified search while `/skills` defaults it to false and visibly exposes `Hide suspicious`. The exact reported count still depends on production data that was not queried locally.

## Solution Assessment

Unclear until maintainers choose the moderation policy. The narrow maintainable fix is to share the search filter contract or visibly surface the filter on `/search`; dropping the filter without UX/security context is riskier.
