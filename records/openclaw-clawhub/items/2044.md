## Summary

Keep open under the ClawHub issue policy; current main still has divergent suspicious-skill defaults between unified search and /skills, and resolving that split needs maintainer moderation and UX policy.

## What This Changes

The report asks for homepage/unified search and /skills search to return consistent skill results for the same query, or to clearly disclose differing suspicious-skill filters.

## Best Possible Solution

Define a shared suspicious-skill visibility rule for unified search and /skills, then enforce it with visible UI state and regression tests.

## Reproduction Assessment

Yes for the source-level root cause: current main has unified search hardcoding `nonSuspiciousOnly: true` while /skills defaults it from `search.nonSuspicious ?? false`. The exact 1-versus-6 result count still depends on production data for `国泰海通`.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable fix is either a shared search visibility contract or visible unified-search filter state, not silently exposing flagged skills without context.
