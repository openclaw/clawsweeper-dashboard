## Summary

Keep open: current main still has the reported source-level split between unified search and `/skills`, and the repair needs maintainer moderation-policy judgment before changing suspicious-skill visibility.

## What This Changes

The issue asks for homepage/unified search and `/skills` to return consistent skill results for the same query or visibly disclose different suspicious-skill filters.

## Best Possible Solution

Define one shared product rule for suspicious-skill visibility across global and directory search, then encode it in UI state and regression tests.

## Reproduction Assessment

Yes, for the root cause: current main shows two search entry points calling the same backend action with different `nonSuspiciousOnly` defaults. The exact 1-versus-6 count still depends on production data for the reported query.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable solution is either to align the defaults through a shared search contract or keep the split while making the active filter visible in unified search.
