## Summary

Keep open: current main still has an unlabelled split between unified search and the skills directory, and deciding whether suspicious skills should be hidden or merely labeled in public search needs maintainer product/security judgment.

## What This Changes

The issue requests consistent skill search results, or visible explanation of default moderation filters, between homepage/unified search and `/skills`.

## Best Possible Solution

Define one shared moderation-visibility policy for public skill search, then enforce it consistently or show the active filter on every search surface with focused regression coverage.

## Reproduction Assessment

Yes for the source-level inconsistency: current main calls the same backend search action with different `nonSuspiciousOnly` defaults across unified search and `/skills`. The exact production count mismatch for `国泰海通` still depends on hosted data.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable fix is either to align the defaults through a shared search contract or keep the split and make the active safety filter visible in unified search.
