## Summary

This should stay open under the ClawHub issue policy, and current main still shows the reported split: unified search filters suspicious skills by default while /skills does not unless the user enables the visible filter.

## What This Changes

The issue requests consistent skill-search visibility, or a visible explanation of differing default filters, between homepage/unified search and /skills.

## Best Possible Solution

Define one product rule for suspicious-skill visibility across global and directory search, then enforce that shared rule with visible UI state and regression tests.

## Reproduction Assessment

Yes for the root-cause split: current main shows two search entry points calling the same backend action with different nonSuspiciousOnly defaults. The exact 1-versus-6 count still depends on production data for the reported query.

## Solution Assessment

Unclear until maintainers choose the intended moderation policy. The narrow maintainable fix is either to align the defaults through a shared search contract or keep the filtering split but make the active filter visible in unified search.
