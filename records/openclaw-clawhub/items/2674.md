## Summary

Keep this PR open for maintainer review. The install-sort plumbing is useful, but the patch currently turns the existing public `sort=downloads` compatibility alias into a 400 while the local package API helper can still send that value.

## What This Changes

The PR updates package/plugin catalog HTTP handlers, plugin route loading, tests, and API docs to support install sorting, default plugin catalog requests to recommended sorting, and reject `sort=downloads`.

## Best Possible Solution

Keep `sort=installs` support and the plugin default-sort cleanup, but preserve or explicitly deprecate the `sort=downloads` compatibility path across server, docs, tests, and client helpers before merge.

## Reproduction Assessment

Yes for the compatibility issue from source inspection: current main maps `sort=downloads` to `installs`, while the PR removes that accepted value and asserts a 400. I did not run live HTTP requests because this review is read-only.

## Solution Assessment

No. Adding install sort support fits the existing catalog endpoints, but removing the existing public compatibility alias is not the narrowest maintainable solution without an explicit deprecation/client update path.
