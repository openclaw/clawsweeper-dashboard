## Summary

Keep this open because current main and v0.23.0 still document and implement lowercase/plural skill readme compatibility, and I found no case-collision rejection or ClawHub `PROTOCOL.md` package contract. The reporter's alias-preserving duplicate-case suggestion is useful, but it needs maintainer policy before implementation.

## What This Changes

The issue requests enforcing a single canonical uppercase `SKILL.md`/`PROTOCOL.md` spelling, or at least rejecting packages that contain conflicting uppercase and lowercase variants.

## Best Possible Solution

Define one casing policy in the skill format spec, then enforce it consistently across CLI scan/sync/publish, browser upload, GitHub import, server publish, API file resolution, and any external content cleanup plan.

## Reproduction Assessment

Yes, at source level. Current main still accepts lowercase/plural skill readme aliases and lacks case-collision rejection in the inspected publish/import/upload paths, though I did not recreate the external Windows checkout failure.

## Solution Assessment

Unclear until maintainers choose the policy. The reporter's compatibility-preserving duplicate-case rejection is narrower than uppercase-only enforcement, but `PROTOCOL.md` still needs a ClawHub scope decision.
