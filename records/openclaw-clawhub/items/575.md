## Summary

Keep this open. Current main still accepts lowercase and plural skill readme variants without a case-insensitive duplicate rejection rule, while strict uppercase-only enforcement would conflict with the documented compatibility contract and `PROTOCOL.md` is not currently a ClawHub package contract.

## What This Changes

The issue requests enforcement that skill packages do not contain conflicting uppercase and lowercase `SKILL.md` or `PROTOCOL.md` files.

## Best Possible Solution

Maintainers should define the accepted readme casing and `PROTOCOL.md` scope, then add one consistent case-insensitive path-collision rule across docs, CLI scan/sync/publish, browser upload, GitHub import, server publish, and read paths.

## Reproduction Assessment

Yes for the ClawHub validation gap: source inspection shows current docs and publish/import/read paths accept lowercase or plural skill readme variants without a case-insensitive duplicate rejection rule. I did not reproduce the external Windows checkout failure in `openclaw/skills`.

## Solution Assessment

No for strict uppercase-only enforcement because it conflicts with the current documented compatibility contract. The better path is the reporter's narrowed collision rule after maintainers decide accepted readme variants and whether `PROTOCOL.md` belongs in ClawHub validation.
