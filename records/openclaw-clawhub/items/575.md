## Summary

Keep this issue open. Current main still accepts lowercase and plural skill readme variants, and the latest reporter comment narrows the request into a compatibility-preserving case-collision policy that still needs maintainer judgment.

## What This Changes

Request to enforce canonical casing or case-insensitive collision handling for `SKILL.md`/`skill.md`, `skills.md`, and possibly `PROTOCOL.md` in skill packages.

## Best Possible Solution

Define a compatibility-preserving case-insensitive duplicate-path policy, decide whether `PROTOCOL.md` belongs in the package contract, then enforce it consistently across CLI scan/sync/publish, browser upload, GitHub import, server publish, stored readme resolution, and content cleanup.

## Reproduction Assessment

Yes for the ClawHub validation side by source inspection: current main accepts `skill.md` / `skills.md` and lacks a case-insensitive duplicate-path rejection in the inspected publish/import paths. I did not reproduce the separate Windows checkout failure from the content repository in this read-only review.

## Solution Assessment

No for the original uppercase-only framing because it conflicts with documented lowercase compatibility. The reporter's later duplicate-path rule is the safer direction, but the exact policy still needs maintainer judgment for accepted variants, `PROTOCOL.md` scope, and existing content cleanup.
