## Summary

Current `main` still permits case-colliding bundle paths and selects the first case-insensitive manifest match, so the reported cross-platform ambiguity remains. Uppercase-only enforcement needs maintainer judgment because lowercase `skill.md` is currently documented as supported.

Likely related people: `Patrick-Erichsen` is connected through the latest history of the central validation files (medium confidence), while `steipete` appears in earlier publishing and manifest-history commits (medium confidence).

## What This Changes

Reject conflicting case variants of `SKILL.md` and `PROTOCOL.md`, while deciding whether a single lowercase manifest remains supported.

## Best Possible Solution

Add shared, server-authoritative rejection of duplicate case-folded paths across publish and import paths, with client diagnostics and tests; separately preserve lowercase-only manifests or introduce an explicit documented deprecation policy.

## Reproduction Assessment

Yes. The report provides a concrete Windows checkout symptom, and current source accepts colliding variants before selecting the first case-insensitive manifest match.

## Solution Assessment

No, unconditional uppercase-only enforcement is broader than the narrow compatible fix. Case-folded collision rejection should be implemented first, with canonical casing decided explicitly by maintainers.
