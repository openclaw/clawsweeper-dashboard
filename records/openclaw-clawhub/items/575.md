Keep this issue open. Current main still allows lowercase and plural skill readme variants and has no case-insensitive duplicate-path rejection across the inspected ClawHub publish/import surfaces, while the narrowed request still needs a maintainer package-format decision.

## What This Changes

The issue requests enforcing a single canonical casing or collision rule for skill package files such as `SKILL.md`/`skill.md` and `PROTOCOL.md`/`protocol.md`.

## Best Possible Solution

Define a compatibility-preserving case-insensitive collision policy for accepted skill files, decide whether `PROTOCOL.md` belongs in the skill package contract, then update docs, validation, tests, and content cleanup guidance together.

## Reproduction Assessment

Yes by source inspection for the ClawHub validation gap: current main accepts lowercase/plural skill readme variants and has no case-insensitive duplicate-path rejection in the inspected CLI, browser upload, GitHub import, server publish, and readme-resolution paths. I did not reproduce the separate Windows checkout failure in the external skills repository.

## Solution Assessment

No for strict uppercase-only enforcement because current docs and code deliberately allow lowercase and plural readme variants. The better path is the reporter's narrowed collision rule after maintainers decide the accepted variants, `PROTOCOL.md` scope, and cleanup behavior.
