## Label Justifications

- `P2`: This is a valid package-integrity and cross-platform compatibility bug with a bounded affected surface.
- `impact:other`: Case-colliding paths make affected packages unusable on case-insensitive filesystems and can make entry-file parsing order-dependent.

## Summary

The issue remains necessary on current `main`: skill publishing accepts and stores case-colliding paths, then selects the first case-insensitive entry-file match. The reported Windows failure is valid, but enforcing uppercase-only names would unnecessarily break documented lowercase compatibility, so maintainer confirmation of the narrower collision rule is needed.

## What This Changes

Prevent packages from containing uppercase/lowercase filename variants such as both `SKILL.md` and `skill.md`, with the issue proposing canonical uppercase names.

## Best Possible Solution

Preserve one `SKILL.md` or `skill.md` entry file, but reject any two skill paths that collide after separator normalization and case folding in the shared backend publish boundary; document the invariant and cover representative publish/import paths.

## Reproduction Assessment

Yes. Submit a skill bundle containing both `SKILL.md` and `skill.md`; current `main` accepts the paths, selects the first case-insensitive entry match, and retains the collision that causes the reported Windows checkout behavior.

## Solution Assessment

No, not as titled. Uppercase-only enforcement is broader than the bug; case-folded duplicate-path rejection at the shared backend boundary is the narrowest maintainable solution while preserving documented compatibility.
