## Summary

Keep this open: current main and the latest v0.23.0 release still allow lowercase/plural skill metadata filenames in the documented and implemented publish/import paths, and there is no implemented `PROTOCOL.md` package contract or duplicate-case rejection policy yet.

## What This Changes

The issue requests canonical uppercase `SKILL.md`/`PROTOCOL.md` enforcement, or at least rejection of packages containing both case variants, to avoid case-insensitive filesystem breakage.

## Best Possible Solution

Maintainers should define the canonical casing and duplicate-case policy, decide whether `PROTOCOL.md` belongs in the skill package contract, then update docs, validation, tests, and any content cleanup plan consistently.

## Reproduction Assessment

Yes, at source level. Current docs and publish/import paths still accept lowercase or plural skill readme aliases, and the inspected validation paths do not reject case-colliding duplicates.

## Solution Assessment

Unclear until maintainers choose the format policy. The reporter's compatibility-preserving duplicate-case rejection is narrower than uppercase-only enforcement, but `PROTOCOL.md` still needs a ClawHub scope decision.
