## Summary

Keep this issue open. Current main still documents and implements lowercase/plural skill readme compatibility and does not reject case-conflicting `SKILL.md`/`skill.md` pairs, so the requested enforcement remains a product and compatibility decision rather than an implemented cleanup.

## What This Changes

Request to enforce canonical uppercase skill package filenames and prevent conflicting uppercase/lowercase manifest files that break case-insensitive checkouts.

## Best Possible Solution

Define the filename casing policy first, then update docs and validation consistently across CLI scan/sync/publish, browser upload, GitHub import, server publish, stored readme resolution, and any external content cleanup plan.

## Reproduction Assessment

Yes, at source level: current CLI, browser, GitHub import, and server publish paths accept lowercase or plural readme variants and do not reject case-conflicting file pairs. I did not run a Windows checkout reproduction, but the current validation gap is clear from source inspection.

## Solution Assessment

Unclear until maintainers decide the package-format policy. The safer likely direction is to preserve documented lowercase compatibility while rejecting case-fold filename collisions, rather than enforcing uppercase-only unconditionally.
