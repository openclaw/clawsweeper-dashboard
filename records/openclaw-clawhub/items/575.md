## Summary

Keep this issue open. Current main still documents and implements lowercase/plural skill readme compatibility in several publish/import paths, and the narrower duplicate-case policy plus the requested `PROTOCOL.md` contract still need maintainer policy judgment.

## What This Changes

The issue requests a canonical skill/protocol file casing policy that prevents conflicting uppercase and lowercase files from breaking case-insensitive checkouts.

## Best Possible Solution

Define the canonical case-insensitive file policy first, then update docs and validation consistently across CLI scan/sync/publish, browser upload, GitHub import/sync, server publish, readme resolution, and any external content cleanup plan.

## Reproduction Assessment

Yes for the source-level gap: current main still accepts lowercase readme variants and lacks a consistent duplicate-case guard in the central publish/import paths. I did not recreate the Windows checkout state locally, but the reported failure mode is consistent with case-insensitive filesystem behavior.

## Solution Assessment

Unclear until maintainers decide the policy. Preserving lowercase compatibility while rejecting duplicate case-insensitive paths is narrower than uppercase-only enforcement and matches the reporter's later suggestion.
