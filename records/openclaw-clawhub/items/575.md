## Summary

Keep this open. Current main still documents and implements lowercase/plural skill readme compatibility and does not reject case-insensitive duplicate paths; the remaining work needs a maintainer package-format decision, especially for `PROTOCOL.md`.

## What This Changes

The issue asks ClawHub to enforce a single casing contract for skill package files and prevent conflicting variants such as `SKILL.md` plus `skill.md` or `PROTOCOL.md` plus `protocol.md`.

## Best Possible Solution

Define the skill package casing and `PROTOCOL.md` contract first, then add one consistent case-insensitive path-collision rule across docs, CLI scan/sync/publish, browser upload, GitHub import, server publish, and read paths.

## Reproduction Assessment

Yes for the ClawHub validation gap: source inspection shows current docs and publish/import/read paths accept lowercase or plural skill readme variants without a case-insensitive duplicate rejection rule. I did not reproduce the external Windows checkout failure in `openclaw/skills`.

## Solution Assessment

No for strict uppercase-only enforcement because it conflicts with the current documented compatibility contract. The better path is the reporter's narrowed collision rule after maintainers decide accepted readme variants and whether `PROTOCOL.md` belongs in ClawHub validation.
