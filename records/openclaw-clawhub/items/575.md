## Summary

Keep this issue open. Current main still documents and implements lowercase/plural skill readme compatibility across docs, CLI discovery, CLI publish, browser publish, GitHub import, and server-side publish validation; changing that is a breaking package-format policy decision rather than an implemented cleanup.

## What This Changes

Request to enforce canonical uppercase `SKILL.md` and `PROTOCOL.md` names so case-colliding skill packages do not break on case-insensitive filesystems.

## Best Possible Solution

Maintainers should decide the canonical casing policy and whether `PROTOCOL.md` belongs in the package contract, then update docs, specs, validators, import/upload/publish paths, and any content migration plan together.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a package-format policy request. Source inspection gives high confidence that current main still intentionally allows lowercase and plural readme variants.

## Solution Assessment

No for an automated uppercase-only fix: enforcing this would change the public package contract and add an undefined `PROTOCOL.md` rule. The maintainable path is a maintainer policy decision followed by coordinated validation, docs, tests, and content cleanup.
