## Summary

Keep this open. Current main and v0.20.0 still document and implement lowercase/plural skill marker compatibility, and the inspected publish/import/scan paths do not reject case-insensitive duplicate marker paths; the `PROTOCOL.md` part of the request is still an undefined ClawHub package contract.

## What This Changes

The issue requests enforcing canonical uppercase `SKILL.md` and `PROTOCOL.md` casing, or at least rejecting skill bundles that contain case-insensitive duplicate marker files.

## Best Possible Solution

Define the canonical case-insensitive path policy, then reject normalized duplicate marker paths consistently across CLI scan/sync/publish, browser/API publish and scan, GitHub import, docs, and specs while handling existing corpus cleanup separately.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the missing ClawHub enforcement: current publish/import/scan paths accept lowercase/plural markers and do not reject normalized duplicate marker paths. I did not recreate the reporter’s Windows checkout, but their transcript is concrete supporting evidence for the filesystem failure mode.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. Rejecting case-insensitive duplicate paths is the maintainable end state, but uppercase-only behavior and `PROTOCOL.md` scope should be decided before implementation.
