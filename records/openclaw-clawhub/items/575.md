## Label Justifications

- `impact:other`: Affected packages cannot faithfully represent both paths and can remain perpetually modified on Windows and other case-insensitive filesystems.

## Summary

Current main does not enforce portable case-folded uniqueness for multipart skill uploads, so the Windows checkout failure remains valid. The issue should stay open, but the narrow fix is collision rejection rather than requiring uppercase filenames because ClawHub intentionally documents and accepts lowercase manifests.

## What This Changes

Reject skill packages containing file paths that differ only by case, especially simultaneous `SKILL.md`/`skill.md` or `PROTOCOL.md`/`protocol.md` entries.

## Best Possible Solution

Add a shared portable-path validator that rejects every duplicate after slash normalization and case folding at the server upload boundary, mirror the same check in the CLI for early feedback, preserve lowercase manifest compatibility, and document the collision rule in the format docs and an upload-gating spec.

## Reproduction Assessment

Yes, at source-reproduction confidence: the issue provides a concrete Windows checkout transcript, and current main accepts case-insensitive manifest names without a case-folded uniqueness guard in multipart skill publishing. A live Windows checkout was not rerun during this read-only review.

## Solution Assessment

No, uppercase-only enforcement is not the best fix because it conflicts with the documented lowercase compatibility contract. Portable case-collision rejection for all paths is narrower and directly addresses the failure.
