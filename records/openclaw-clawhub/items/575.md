## Summary

Keep this issue open. Current main still allows lowercase and plural skill readme variants and has no case-insensitive duplicate-path rejection rule; the latest reporter suggestion narrows the request, but it still needs a maintainer package-format decision, especially for `PROTOCOL.md`.

## What This Changes

The issue requests enforcement around `SKILL.md` and `PROTOCOL.md` casing so skill packages cannot contain conflicting uppercase and lowercase files that break on case-insensitive filesystems.

## Best Possible Solution

Define a compatibility-preserving case-insensitive duplicate-path policy, decide whether `PROTOCOL.md` belongs in the skill package contract, then enforce the policy consistently across CLI, browser upload, GitHub import, server publish, docs, and content cleanup.

## Reproduction Assessment

Yes for the ClawHub validation gap by source inspection: current main accepts lowercase and plural skill readme variants and lacks a case-insensitive duplicate-path rejection rule in the inspected publish/import paths. I did not reproduce the separate Windows checkout failure from the external content repository.

## Solution Assessment

No for strict uppercase-only enforcement because it conflicts with documented compatibility. The safer direction is the reporter's narrowed duplicate-case rule after maintainers decide accepted variants, `PROTOCOL.md` scope, and cleanup behavior.
