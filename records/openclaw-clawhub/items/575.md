## Summary

Keep this issue open. Current main still accepts lowercase and plural skill readme variants without a case-insensitive duplicate-path rejection rule, and the remaining request changes the package-format contract rather than being a cleanup-close candidate for this repository.

## What This Changes

Request to enforce canonical casing or reject simultaneous case-conflicting files such as `SKILL.md`/`skill.md` and `PROTOCOL.md`/`protocol.md` to avoid case-insensitive filesystem breakage.

## Best Possible Solution

Adopt a compatibility-preserving case-insensitive path-collision policy for accepted skill files, decide `PROTOCOL.md` scope, then update docs, CLI/browser/GitHub/server validation, tests, and any content cleanup plan together.

Yes by source inspection for the ClawHub validation gap: current main accepts lowercase/plural readme variants and has no case-insensitive duplicate-path check in the inspected CLI, browser upload, GitHub import, and server publish paths. I did not reproduce the separate Windows checkout failure in the external skills repository.

## Solution Assessment

No for strict uppercase-only enforcement because current docs and code deliberately accept lowercase and plural variants. The safer path is the reporter’s narrowed collision rule after maintainers decide accepted variants, `PROTOCOL.md` scope, and cleanup behavior.
