## Summary

Keep this issue open. Current main still intentionally accepts lowercase and plural skill readme variants and lacks case-insensitive duplicate-path enforcement, while the latest reporter comment narrows the request into a compatibility-preserving package-format policy that needs maintainer judgment.

## What This Changes

The issue asks ClawHub to enforce collision-safe casing for `SKILL.md` and `PROTOCOL.md` so skill packages do not break on case-insensitive filesystems.

## Best Possible Solution

Define a compatibility-preserving case-insensitive duplicate-path policy, decide whether `PROTOCOL.md` belongs in the package contract, then enforce it consistently across CLI scan/sync/publish, browser upload, GitHub import, server publish, stored readme resolution, and content cleanup.

## Reproduction Assessment

Yes for the ClawHub validation side by source inspection: current main accepts `skill.md` / `skills.md` and lacks a case-insensitive duplicate-path rejection in the inspected publish/import paths. I did not reproduce the separate Windows checkout failure from the content repository.

## Solution Assessment

No for strict uppercase-only enforcement because it conflicts with current documented compatibility. The reporter's newer duplicate-path policy is the safer direction, but maintainers still need to decide accepted variants, `PROTOCOL.md` scope, and cleanup behavior.
