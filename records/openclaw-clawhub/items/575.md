Keep this issue open. Current main still documents and implements lowercase/plural skill readme compatibility and does not reject case-insensitive duplicate package paths, while the reporter’s latest comment narrows the ask to collision enforcement that needs a package-format policy decision.

## What This Changes

The issue asks ClawHub to prevent skill packages from containing case-conflicting `SKILL.md`/`skill.md` and `PROTOCOL.md`/`protocol.md` variants.

## Best Possible Solution

Adopt a compatibility-preserving case-insensitive collision rule for accepted skill package paths, decide whether `PROTOCOL.md` belongs in the contract, then update docs, CLI scan/sync/publish, browser upload, GitHub import, server publish, read paths, and regression tests consistently.

## Reproduction Assessment

Yes for the ClawHub validation gap: source inspection shows CLI scan/publish, browser publish, GitHub import, server publish, and read paths accept lowercased variants and do not reject case-insensitive duplicates. I did not reproduce the separate Windows checkout failure in the external skills repository.

## Solution Assessment

No for strict uppercase-only enforcement, because current docs and code deliberately allow lowercase and plural readme variants. The safer path is the reporter’s narrowed collision rule after maintainers decide accepted variants, `PROTOCOL.md` scope, and cleanup behavior.
