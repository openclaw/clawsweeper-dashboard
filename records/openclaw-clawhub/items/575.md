## Summary

Keep this issue open. Current main still documents and implements lowercase/plural skill readme compatibility and has no case-insensitive duplicate rejection rule, while the narrowed collision rule and original `PROTOCOL.md` scope still need a maintainer package-format decision.

## What This Changes

Request to enforce canonical casing for skill package `SKILL.md`/`PROTOCOL.md` and reject uppercase/lowercase duplicates that break case-insensitive file systems.

## Best Possible Solution

Define the accepted readme casing and `PROTOCOL.md` contract first, then add one consistent case-insensitive path-collision rule across docs, CLI scan/sync/publish, browser upload, GitHub import, server publish, and read paths.

## Reproduction Assessment

Yes for the ClawHub validation gap: source inspection shows current main accepts lowercase/plural skill readme variants and lacks case-insensitive duplicate rejection. I did not reproduce the external Windows checkout failure in `openclaw/skills`.

## Solution Assessment

No for strict uppercase-only enforcement because it conflicts with current documented compatibility. The better path is the reporter's narrowed collision rule after maintainers decide accepted readme variants and whether `PROTOCOL.md` belongs in ClawHub validation.
