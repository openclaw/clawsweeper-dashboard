## Summary

Keep this open. Current main and v0.20.0 still document lowercase skill marker compatibility and the publish/import paths accept lowercase or plural marker names without rejecting case-insensitive duplicate paths; `PROTOCOL.md` enforcement still needs a maintainer compatibility decision.

## What This Changes

Require ClawHub validation to reject conflicting case variants such as `SKILL.md`/`skill.md` and `PROTOCOL.md`/`protocol.md` so skill repositories remain usable on case-insensitive filesystems.

## Best Possible Solution

Define the canonical case-insensitive file path policy, then add duplicate-normalized-path rejection across CLI publish/sync/scan, API publish/scan, GitHub import, and docs/specs while preserving the chosen compatibility contract.

## Reproduction Assessment

Yes, from source inspection and the reporter's concrete Windows checkout transcript, though I did not run a Windows filesystem reproduction. Current main still accepts lowercase/plural skill markers and lacks a normalized duplicate-path guard in the inspected publish/import paths.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. Rejecting case-insensitive duplicate paths is the right end state, but uppercase-only enforcement and `PROTOCOL.md` scope need product direction first.
