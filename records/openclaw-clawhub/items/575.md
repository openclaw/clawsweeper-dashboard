## Label Justifications

- `P2`: Case-colliding package paths cause real cross-platform checkout and consumption failures, but affect a limited set of malformed packages rather than the whole service.
- `impact:other`: The issue affects package validity and Windows-compatible repository checkout, which is meaningful maintainer impact outside the specific owned categories.

## Summary

The reported cross-platform failure remains possible because current main sanitizes file paths without rejecting case-insensitive duplicates. However, the requested uppercase-only rule conflicts with existing lowercase manifest compatibility, so maintainers should first confirm the intended package contract.

## What This Changes

Prevent skill packages from containing case-colliding metadata paths such as `SKILL.md` with `skill.md` or `PROTOCOL.md` with `protocol.md`.

## Best Possible Solution

Reject case-insensitive duplicate paths server-side for all skill and package ingestion while retaining lowercase entrypoint compatibility unless maintainers explicitly adopt and document an uppercase-only format migration.

## Reproduction Assessment

Yes, at source level: current publishing accepts differently cased paths as distinct entries, while such a tree collides on common case-insensitive filesystems, matching the reported Windows checkout behavior.

## Solution Assessment

No, not exactly as titled. Unconditionally requiring uppercase names would break an existing compatibility path; rejecting case-insensitive duplicates is the narrower solution unless maintainers intentionally revise the package format.
