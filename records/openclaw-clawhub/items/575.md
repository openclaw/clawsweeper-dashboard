## Label Justifications

- `P2`: Case-colliding files are a normal-priority cross-platform package-integrity bug with a bounded validation fix and limited runtime blast radius.
- `impact:other`: Affected skill packages or corpus checkouts can become persistently dirty or unusable on Windows and other case-insensitive filesystems.

## Summary

This remains necessary: current main intentionally accepts lowercase skill markers, while archive and repository ingestion do not establish a universal case-folded path uniqueness invariant, so case-only duplicates can remain valid on Linux but break Windows checkouts. The reporter’s follow-up usefully narrows the compatible fix to rejecting simultaneous case variants rather than removing lowercase support.

## What This Changes

Reject packages or imported skill trees containing case-colliding SKILL.md or PROTOCOL.md paths, with a decision on whether lowercase-only marker files remain supported.

## Best Possible Solution

Add one shared case-folded path-collision validator across multipart publish, ClawPack, and GitHub import/sync boundaries, keep a lone lowercase marker readable for compatibility, document uppercase `SKILL.md` as canonical, and update the package-integrity spec; include `PROTOCOL.md` only if it is confirmed as a governed ClawHub metadata filename.

## Reproduction Assessment

Yes at source level: dual-case marker discovery plus exact-case duplicate detection permits both `SKILL.md` and `skill.md` on a case-sensitive host, matching the reported Windows collision. This read-only review did not clone the external corpus on Windows.

## Solution Assessment

No, strict uppercase-only enforcement is not the best compatible fix because current code deliberately accepts lowercase markers. Rejecting case-folded duplicate paths everywhere is narrower; canonical casing and any deprecation of lowercase-only files require a maintainer decision.
