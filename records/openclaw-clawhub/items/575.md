## Summary

Keep this issue open. Current main still permits lowercase and plural skill readme variants and does not reject case-insensitive duplicate package paths; the latest reporter comment narrows the ask to collision enforcement, which still needs a maintainer format-policy decision.

## What This Changes

Request to enforce a collision-safe casing policy for skill package files, especially `SKILL.md`/`skill.md` and `PROTOCOL.md` variants.

## Best Possible Solution

Define a compatibility-preserving case-insensitive collision rule for accepted skill package paths, decide the `PROTOCOL.md` contract, then update docs, CLI scan/sync/publish, browser upload, GitHub import, server publish, read paths, and regression tests consistently.

## Reproduction Assessment

Yes for the ClawHub validation gap: source inspection shows the CLI, browser publish route, GitHub import, server publish, and read paths accept lowercased variants and do not reject case-insensitive duplicates. I did not reproduce the separate Windows checkout failure in the external skills repository.

## Solution Assessment

No for strict uppercase-only enforcement, because current docs and code deliberately allow lowercase and plural readme variants. The safer path is the reporter’s narrowed collision rule after maintainers decide accepted variants, `PROTOCOL.md` scope, and cleanup behavior.
