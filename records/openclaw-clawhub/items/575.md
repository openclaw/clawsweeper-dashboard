## Summary

Keep this issue open. Current main still documents and implements lowercase and plural skill readme compatibility in the public format, CLI, browser upload, GitHub import, server publish validation, and stored readme resolution, so uppercase-only enforcement is not implemented and needs maintainer format-policy review.

## What This Changes

Require canonical uppercase `SKILL.md` and `PROTOCOL.md` files and reject lowercase or mixed-case duplicates that break case-insensitive checkouts.

## Best Possible Solution

Maintainers should define the canonical casing policy and `PROTOCOL.md` scope, then update docs, CLI scan/sync/publish, browser upload, GitHub import, server validation, readme resolution, and any content cleanup plan consistently.

## Reproduction Assessment

Yes for the policy gap: current main can be checked by reading the documented and implemented lowercase/plural acceptance paths. The exact Windows checkout failure is external to this repository because the colliding content lives in `openclaw/skills`.

## Solution Assessment

Unclear until maintainers decide the package-format policy. Uppercase-only validation may be the right long-term guard, but it is a breaking change from current documented compatibility and needs a migration plan plus a decision on whether `PROTOCOL.md` belongs in the contract.
