## Summary

Keep this open. Current main still documents and implements lowercase/plural skill readme compatibility and does not define `PROTOCOL.md` as a ClawHub skill-package contract, so the requested casing enforcement remains a maintainer policy decision rather than implemented cleanup.

## What This Changes

Request to enforce a canonical or non-conflicting casing policy for skill package files, especially `SKILL.md`/`skill.md` and `PROTOCOL.md`/`protocol.md`, to avoid case-insensitive filesystem breakage.

## Best Possible Solution

Maintainers should define the casing policy first, then apply duplicate-case rejection consistently across publish, sync, GitHub import, browser upload, server validation, docs, and tests while preserving any chosen compatibility aliases.

## Reproduction Assessment

Yes, at source level. Current main accepts lowercase/plural skill readme aliases in multiple import and publish paths without a global case-collision rejection, though I did not rerun the Windows checkout reproduction against the external skills repository.

## Solution Assessment

Unclear until maintainers choose the policy. The reporter's compatibility-preserving duplicate-case rejection is narrower than uppercase-only enforcement, but `PROTOCOL.md` still needs a ClawHub scope decision.
