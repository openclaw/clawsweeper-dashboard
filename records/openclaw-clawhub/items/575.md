## Summary

Keep this issue open. Current main still accepts lowercase/plural skill readme variants and lacks case-insensitive duplicate rejection, while the narrowed collision rule and `PROTOCOL.md` scope need a maintainer package-format decision.

## What This Changes

The issue asks ClawHub to reject skill packages that include case-colliding marker files such as both `SKILL.md` and `skill.md`, and also raises `PROTOCOL.md` casing.

## Best Possible Solution

Define the skill package casing and `PROTOCOL.md` contract first, then add one consistent case-insensitive path-collision rule across docs, CLI scan/sync/publish, browser upload, GitHub import, server publish, and read paths.

## Reproduction Assessment

Yes for the ClawHub validation gap: source inspection shows current docs and publish/import/read paths still accept lowercase or plural skill readme variants without a case-insensitive duplicate rejection rule. I did not reproduce the external Windows checkout failure in `openclaw/skills`.

## Solution Assessment

No for strict uppercase-only enforcement because it conflicts with current documented compatibility. The better path is the reporter's narrowed collision rule, after maintainers decide accepted readme variants and whether `PROTOCOL.md` belongs in ClawHub validation.
