## Summary

Keep open: current main and v0.17.0 still accept lowercase/plural skill readme variants and do not reject case-insensitive duplicate paths; the remaining work needs a package-format decision, especially for `PROTOCOL.md`.

## What This Changes

The issue requests enforcing canonical casing for `SKILL.md` and `PROTOCOL.md`, later narrowed to preserving lowercase readme compatibility while rejecting simultaneous uppercase/lowercase variants.

## Best Possible Solution

Define the skill package casing and `PROTOCOL.md` contract, then add one consistent case-insensitive path-collision rule across docs, CLI scan/sync/publish, browser upload, GitHub import, server publish, and read paths.

## Reproduction Assessment

Yes for the ClawHub validation gap: source inspection shows docs and publish/import/read paths accept lowercase or plural readme variants without a case-insensitive duplicate rejection rule. I did not reproduce the external Windows checkout failure in `openclaw/skills`.

## Solution Assessment

No for strict uppercase-only enforcement because it conflicts with the current documented compatibility contract. The safer path is the reporter's narrowed collision rule after maintainers decide accepted readme variants and whether `PROTOCOL.md` belongs in ClawHub validation.
