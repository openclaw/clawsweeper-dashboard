## Summary

Current main still accepts case-insensitive manifest names and does not reject case-folded duplicate package paths, so the Windows portability bug remains. The issue should stay open for a maintainer decision because enforcing uppercase-only names would conflict with the documented and shipped lowercase compatibility contract; the reporter’s follow-up proposing duplicate-only rejection is the safer direction.

## What This Changes

Reject skill packages containing paths that differ only by filename case, with a decision on whether single lowercase `skill.md` and `protocol.md` variants remain supported.

## Best Possible Solution

Define case-folded path uniqueness as the package invariant, preserve a single supported lowercase manifest where compatibility is desired, reject ambiguous uploads in both CLI and server validation, and separately clean existing colliding corpus entries with a documented keep-one-file rule.

## Reproduction Assessment

Yes at source level: a case-sensitive source folder can contain both filename variants, the CLI uploads both, and server validation does not reject their case-folded collision. The reporter also provides a concrete Windows checkout transcript showing the resulting filesystem failure.

## Solution Assessment

No, unconditional uppercase-only enforcement is not the best current formulation because it conflicts with documented lowercase compatibility. Duplicate case-folded path rejection plus an explicit legacy-manifest policy is narrower and safer.
