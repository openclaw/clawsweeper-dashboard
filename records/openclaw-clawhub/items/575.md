## Summary

Keep this issue open. Current main still documents and implements lowercase/plural skill readme compatibility, and the latest requester comment narrows this into a compatibility-preserving case-collision policy that still needs maintainer product judgment.

## What This Changes

The issue asks ClawHub to prevent skill packages or repositories from containing both case variants of package marker files such as `SKILL.md`/`skill.md` and `PROTOCOL.md`/`protocol.md`.

## Best Possible Solution

Define a compatibility-preserving case-collision policy, decide whether `PROTOCOL.md` belongs in the package contract, then enforce and document it consistently across CLI scan/sync/publish, browser upload, GitHub import, server publish, readme resolution, and content cleanup.

## Reproduction Assessment

Yes for the ClawHub validation side by source inspection: current main accepts `skill.md`/`skills.md` and lacks a case-insensitive duplicate marker rejection in the inspected publish/import paths. I did not reproduce the separate Windows checkout failure from the external skills content repository.

## Solution Assessment

No for the original uppercase-only framing; the requester’s later compatibility-preserving duplicate-path rule is safer, but maintainers still need to decide accepted variants, `PROTOCOL.md` scope, and existing content cleanup.
