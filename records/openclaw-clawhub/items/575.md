## Summary

Keep this issue open. Current main still accepts lowercase/plural skill readme variants and does not reject case-insensitive duplicate package paths; the latest narrowed proposal is plausible, but it remains a package-format policy and migration decision, especially for `PROTOCOL.md` and existing external skill content.

## What This Changes

Request ClawHub enforce skill package filename casing/collision rules for `SKILL.md`/`skill.md` and `PROTOCOL.md`/`protocol.md`, now narrowed to reject simultaneous case variants while preserving lowercase compatibility.

## Best Possible Solution

Adopt a documented no case-insensitive duplicate path rule while preserving accepted readme variants, implement the same validation and error messaging across CLI scan/sync/publish, web upload, GitHub import, and server publish, and handle existing skill-content cleanup separately.

## Reproduction Assessment

Yes for the ClawHub enforcement gap by source inspection: current docs and publish/import paths accept lowercase/plural readme filenames and do not reject case-insensitive duplicates. I did not recreate the separate Windows checkout of `openclaw/skills`.

## Solution Assessment

Unclear until maintainers decide the package policy. The latest proposal is safer than uppercase-only because it preserves compatibility, but the collision rule, `PROTOCOL.md` scope, and migration/autofix behavior need explicit maintainer direction.
