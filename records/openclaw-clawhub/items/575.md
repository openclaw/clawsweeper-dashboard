## Summary

Keep this open. Current main still permits lowercase/plural skill readme variants in several publish/import paths and does not define `PROTOCOL.md` as ClawHub package metadata, so the requested casing policy is not implemented yet.

## What This Changes

The issue asks ClawHub to enforce canonical uppercase `SKILL.md`/`PROTOCOL.md` casing, or at least reject simultaneous case variants that break case-insensitive filesystems.

## Best Possible Solution

Define the casing and `PROTOCOL.md` policy first, then consistently reject case-insensitive duplicate package paths across hosted upload, CLI publish/sync, GitHub import/source sync, server validation, docs, tests, and any content cleanup plan.

## Reproduction Assessment

Yes, at source level. Current main explicitly lowercases and accepts skill readme aliases in multiple publish/import paths without a global case-collision rejection; I did not run a Windows checkout reproduction.

## Solution Assessment

Unclear until maintainers choose the policy. The reporter's later conflict-only proposal is safer than uppercase-only enforcement, but lowercase/plural compatibility and `PROTOCOL.md` scope still need a product decision.
