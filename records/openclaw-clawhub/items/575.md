## Label Justifications

- `P2`: This is a reproducible cross-platform packaging defect with a bounded blast radius and no evidence of data loss or service outage.
- `impact:other`: Case-colliding package entries can make repositories and installed skill archives unusable or unstable on Windows and other case-insensitive filesystems.

## Summary

Current main still accepts case-distinct package paths without a shared collision check, so the Windows portability problem remains; however, uppercase-only enforcement would also remove intentionally supported lowercase and legacy manifest aliases and needs maintainer judgment.

## What This Changes

Reject skill packages containing case-colliding `SKILL.md` or `PROTOCOL.md` paths and decide whether lone lowercase manifest aliases remain supported.

## Best Possible Solution

Define one shared invariant that rejects distinct normalized relative paths with the same case-folded value before publishing, importing, restoring, hashing, or extracting, while preserving lone lowercase aliases unless maintainers deliberately approve a documented breaking change.

## Reproduction Assessment

Yes, at source level: current publishing and extraction paths accept case-distinct entries without a case-folded uniqueness check, and the reporter supplied a concrete Windows checkout transcript showing the collision. A live Windows run was not performed during this read-only review.

## Solution Assessment

No, uppercase-only enforcement is not the narrowest safe solution because it breaks existing alias compatibility. Rejecting ambiguous case-colliding paths while retaining one unambiguous supported manifest is the maintainable repair.
