## Summary

Keep open. Current source and the live public API no longer show the named skill stuck in a pending state, but ClawHub cleanup policy keeps issues open and the requested owner-facing queue/heartbeat/retry visibility is still a product and security decision.

## What This Changes

The issue reports the DarkPhoenix2704/nocodb-official skill staying pending even after VirusTotal passed, with follow-up asking for queue position, worker heartbeat, retry timing, and per-item state transitions.

## Best Possible Solution

Keep the existing recovery/rescan plumbing, then add a narrowly scoped owner/admin scan-status surface only after maintainers decide which per-item scanner fields are safe to expose.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The live API for the named skill now returns a suspicious moderation verdict rather than pending, and source review cannot inspect the exact historical scanner row or worker logs.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The current recovery path is reasonable, but the requested queue/heartbeat/retry visibility should be solved as an authenticated per-item status design rather than exposing raw worker internals.
