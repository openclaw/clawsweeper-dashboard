## Summary

## What This Changes

The PR adds financial and paid-service capability tags, narrows crypto/wallet inference away from generic purchases and transaction signing, updates UI labels and API docs, and adds capability-tag regression tests.

## Best Possible Solution

Keep the financial versus crypto split, but make the final sensitive-credential implication depend on the emitted `requires-wallet` tag and add a regression assertion for the inferred crypto transaction-signing case.

## Solution Assessment

No, not as currently patched. The direction is maintainable, but the implementation should preserve the existing invariant that wallet-requiring capability labels imply sensitive-credential risk.
