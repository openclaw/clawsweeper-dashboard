## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-reproducible publish path that can surface the reported GitHub 403/429 rate-limit message; the deleted-account sign-in half now matches documented permanent-deletion behavior, but the remaining recovery and publish wording decisions are security-sensitive.

## What This Changes

The issue asks ClawHub to recover gracefully from GitHub API rate limits during skill publishing and to handle sign-in after a user deletes their account.

## Best Possible Solution

Keep the permanent-deletion policy unless maintainers approve a recovery path, and triage the publish-rate-limit cluster with production evidence before adding retry, cache, backoff, or wording changes that preserve the account-age gate.

## Reproduction Assessment

Yes, by source inspection. Normal web and API publish paths call requireGitHubAccountAge, which throws the reported GitHub 403/429 rate-limit message, and deleted-account OAuth reaches handleDeletedUserSignIn, which rejects permanent deletion; I did not run a live production OAuth or publish flow.

## Solution Assessment

Unclear. The deleted-account behavior is documented current policy, but publish-rate-limit recovery depends on production quota/log evidence and must not weaken the GitHub account-age anti-abuse gate.
