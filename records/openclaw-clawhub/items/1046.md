## Summary

Keep open. Current main still matches both reported surfaces: deleted/deactivated GitHub-linked accounts are rejected on reauth, and publish can still surface the exact GitHub 403/429 account-age rate-limit error; the remaining work needs maintainer account-lifecycle and publish-gate policy, not conservative cleanup closure.

## What This Changes

The issue reports a failed skill publish after a GitHub API rate-limit error plus inability to sign back in after deleting a GitHub-linked ClawHub account.

## Best Possible Solution

Define the deleted-account recovery policy first, then handle publish 403/429 recovery separately through production token/quota verification or a narrow retry/cache/backoff change that preserves the GitHub account-age gate.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: account deletion sets `deactivatedAt`/`purgedAt` and OAuth reauth rejects that user, while publish as a user without cached `githubCreatedAt` emits the reported error when the GitHub lookup returns 403 or 429.

## Solution Assessment

No patch is proposed, and an immediate automated repair is not the best path. The maintainable solution is to split the account-lifecycle policy decision from the narrower publish-rate-limit mitigation instead of doing a one-off purge, restore, or gate bypass.
