## Summary

Keep open. Current main appears to intentionally handle the deleted-account/sign-in half with a user-facing permanent-deletion block, but the publish side still has a source-reproducible path where GitHub 403/429 from the account-age lookup surfaces as the reported rate-limit failure; because this is an open ClawHub issue with security/auth labels and maintainer comments leaving publish recovery unresolved, it should remain with maintainer review.

## What This Changes

The issue asks ClawHub to recover gracefully from GitHub API rate-limit failures during skill publishing and to handle GitHub sign-in after a user deletes their ClawHub account.

## Best Possible Solution

Keep the permanent-deletion policy unless maintainers approve a controlled recovery path, and separately improve publish-rate-limit recovery or messaging around the GitHub account-age gate using production quota/log evidence.

## Reproduction Assessment

Yes, by source inspection. Normal publish calls can still hit requireGitHubAccountAge and emit the reported GitHub API rate-limit message, while deleted-account OAuth reaches handleDeletedUserSignIn and is rejected by the current permanent-deletion policy.

## Solution Assessment

Unclear. The deleted-account behavior now appears intentionally documented in code and covered by tests, but publish-rate-limit recovery needs maintainer policy/security judgment and production GitHub quota evidence before changing the anti-abuse gate.
