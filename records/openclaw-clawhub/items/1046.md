## Summary

Keep open. Current main still has a source-reproducible publish path that can emit the reported GitHub 403/429 rate-limit error, while the deleted-account sign-in half now matches the documented permanent-deletion behavior and the latest human triage explicitly leaves publish-flow recovery unresolved.

## What This Changes

The issue asks ClawHub to recover more clearly from a GitHub API rate-limit failure during skill publish and to handle sign-in or recreation after user-initiated account deletion.

## Best Possible Solution

Keep this open, preserve the documented permanent-deletion behavior unless maintainers approve a recovery policy, and separately triage publish 403/429 recovery with production quota/log evidence before changing retry, cache, config, or wording behavior.

## Reproduction Assessment

Yes, by source inspection. Normal publish calls requireGitHubAccountAge, which throws the exact 403/429 rate-limit message, and deleted-account OAuth reaches handleDeletedUserSignIn, which rejects permanent deletion; I did not run a live production OAuth or publish flow.

## Solution Assessment

Unclear. The current deleted-account behavior is the documented policy, but publish-rate-limit recovery is not proven fixed, and choosing retry, cache, deployment config, or copy changes needs production evidence and anti-abuse review.
