## Summary

Keep open. The deleted-account sign-in half is now intentional blocked-account behavior with user-facing copy, but a maintainer explicitly left this issue open for the separate publish-rate-limit recovery/wording surface, and current main still routes normal publish through the GitHub account-age lookup that emits the reported 403/429 message.

## What This Changes

The reporter asks for skill publish to recover or guide better when GitHub rate limits are hit, and for deleted GitHub-linked account sign-in or recreation to work cleanly.

## Best Possible Solution

Keep permanent deletion unless maintainers approve a controlled recovery path, and separately improve publish 403/429 recovery or guidance around the GitHub account-age gate, ideally aligned with https://github.com/openclaw/clawhub/issues/1222.

## Reproduction Assessment

Yes, by source inspection. Normal publish calls can still hit requireGitHubAccountAge and emit the reported GitHub rate-limit message, while deleted-account OAuth reaches handleDeletedUserSignIn and is rejected by the current permanent-deletion policy; I did not run a live OAuth or production quota test.

## Solution Assessment

Unclear. The deleted-account behavior is deliberately documented and tested, but the publish-rate-limit wording or retry path needs production GitHub quota evidence and maintainer security/product judgment before changing the anti-abuse gate.
