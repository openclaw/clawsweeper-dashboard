## Summary

Keep open. Current main still matches both reported surfaces, and this ClawHub issue is not eligible for cleanup auto-close: deleted GitHub-linked accounts are intentionally rejected on reauth, while publish still surfaces GitHub 403/429 from the account-age gate as the reported rate-limit error.

## What This Changes

The issue asks for clearer/recoverable handling of GitHub API rate limits during skill publish and a clean sign-in or recovery path after deleting a GitHub-linked ClawHub account.

## Best Possible Solution

Maintainers should choose the deleted-account lifecycle policy first, then implement either clearer irreversible-deletion/support guidance or an audited restore/relink/recreate path; publish 403/429 recovery should be handled separately without bypassing the GitHub account-age gate.

## Reproduction Assessment

Yes, from source inspection. deleteAccount sets deactivatedAt/purgedAt, the OAuth callback rejects that account on later GitHub sign-in, and publishVersionForUser still reaches requireGitHubAccountAge, which throws the reported error for GitHub 403/429 responses.

## Solution Assessment

No patch is proposed, and a one-off purge or blind retry is not the best solution. The maintainable path is to split the account-lifecycle policy decision from a narrower publish-rate-limit mitigation while preserving anti-abuse checks.
