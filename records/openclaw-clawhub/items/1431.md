## Summary

Keep open. Current main still exposes an owner selector for publishing while rejecting existing skills whose stored publisher differs from the selected publisher, and the dedicated transfer path is still user-targeted; https://github.com/openclaw/clawhub/pull/1635 is the active implementation candidate, not merged main behavior.

## What This Changes

The issue requests allowing an existing skill to move from a personal publisher to an organization publisher instead of failing as a slug collision.

## Best Possible Solution

Land a maintainer-approved publisher-targeted transfer flow, likely through https://github.com/openclaw/clawhub/pull/1635 or a successor, and route existing-skill owner changes through that flow instead of silently re-homing during publish.

## Reproduction Assessment

Yes, from source inspection. Selecting an organization submits ownerHandle, publishVersion resolves it to ownerPublisherId, and insertVersion rejects an existing skill when the stored ownerPublisherId differs from the selected publisher.

## Solution Assessment

No, a narrow insertVersion bypass is not the best direction. Maintainer discussion on https://github.com/openclaw/clawhub/pull/1562 chose an explicit transfer flow, and the docs plus https://github.com/openclaw/clawhub/pull/1635 point to publisher-targeted transfer semantics.
