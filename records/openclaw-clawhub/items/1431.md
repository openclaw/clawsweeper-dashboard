## Summary

Keep open under the ClawHub cleanup policy for issues. Current main fixes the reported publish-flow slug collision through explicit owner migration, but direct organization-targeted transfer remains separate, authorization-sensitive work with open tracking and an open implementation candidate.

## What This Changes

The issue asks ClawHub to let an existing skill move from a personal publisher to an organization publisher instead of failing with a slug-collision error.

## Best Possible Solution

Keep the explicit publish-time owner migration as the supported immediate path, then consolidate any direct organization-transfer command/API work in the publisher-targeted transfer track at https://github.com/openclaw/clawhub/issues/1675 or a maintainer-selected successor to https://github.com/openclaw/clawhub/pull/1635.

## Reproduction Assessment

No for the original publish-flow slug collision on current main: source inspection shows the UI sends a confirmed migrateOwner flag and the backend migrates instead of treating the owner change as a collision. The remaining direct transfer-to-org gap is source-reproducible because the transfer API still requires toUserHandle and stores toUserId.

## Solution Assessment

Yes for the reported publish-flow failure: explicit opt-in migration with source and destination authorization is the narrow maintainable fix. Direct organization transfer should be handled through a publisher-targeted transfer design rather than by only changing handle lookup.
