## Summary

Keep open. Current main still has the reported source path, and the ClawHub cleanup policy for this review keeps issues open; this is a security-sensitive ownership-transfer product gap rather than conservative cleanup.

## What This Changes

Allow an existing skill owner to move a published skill from a personal publisher to an organization publisher, either while publishing a new version or through a dedicated transfer flow.

## Best Possible Solution

Adopt explicit publisher-target transfer semantics with source and target role checks, alias/API/CLI/docs updates, and adjust the publish UI so existing-skill owner changes follow the supported flow.

## Reproduction Assessment

Yes, from source inspection rather than a live deployment run. Selecting an organization in the publish UI submits ownerHandle, publishVersion resolves it to a publisher, and insertVersion rejects the existing skill when the stored ownerPublisherId differs.

## Solution Assessment

Unclear. A narrow insertVersion change would address the symptom, but the maintainable direction appears to be a maintainer-approved publisher-target transfer flow, or hiding existing-skill owner changes until that flow exists.
