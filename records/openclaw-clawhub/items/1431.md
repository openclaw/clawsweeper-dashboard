## Summary

Keep open under the ClawHub cleanup policy for issues. Current main now fixes the original publish-flow owner move with explicit `migrateOwner` handling, but the direct transfer command/API remains user-targeted and the remaining org-transfer direction needs maintainer product/security judgment.

## What This Changes

Allow an existing skill owner to move a skill from a personal publisher to an organization owner without a slug-taken failure.

## Best Possible Solution

Keep `--owner` plus `--migrate-owner` as the supported publish-time owner migration path, then have maintainers decide whether remaining direct transfer-to-org work belongs under https://github.com/openclaw/clawhub/issues/1675 and https://github.com/openclaw/clawhub/pull/1635.

## Reproduction Assessment

No for the original publish-flow failure: current source shows a confirmed owner change sends `migrateOwner` and the backend migrates instead of throwing the slug-taken collision. The separate direct transfer-to-org path is source-reproducible because transfer still accepts and stores user recipients only.

## Solution Assessment

Unclear as a final issue disposition. The merged publish migration is a narrow maintainable fix for the reported UI path, but direct publisher-targeted transfers are separate authorization-sensitive product work.
