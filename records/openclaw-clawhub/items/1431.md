## Summary

Keep open. Current main still matches the reported product gap: the publish UI can submit a different organization owner for an existing skill, but the backend rejects an existing skill whose stored publisher differs from the selected publisher; the dedicated transfer flow is still user-to-user.

## What This Changes

Allow an existing skill owner to move a skill from a personal publisher to an organization publisher instead of receiving a slug-taken error.

## Best Possible Solution

Land a maintainer-approved publisher-target ownership-transfer design, or disable existing-skill owner changes in the publish UI until that supported transfer path exists.

## Reproduction Assessment

Yes. The source-level path is clear: selecting an organization owner submits ownerHandle, publishVersion resolves it to ownerPublisherId, and insertVersion rejects the existing skill when that publisher differs from the stored one.

## Solution Assessment

Unclear. Allowing republish to move ownership is possible, but a dedicated publisher-transfer flow with explicit role checks and audit logging may be the safer maintainable solution.
