## Summary

Keep open. Current main still matches the reported product gap: the publish UI can submit an organization owner for an existing skill, but `insertVersion` rejects an existing skill when its stored `ownerPublisherId` differs from the selected publisher; the supported transfer path is still user-to-user and personal-publisher based. The ClawHub cleanup policy also keeps issues open, and related org-transfer PRs are closed unmerged while #1569 remains open.

## What This Changes

Allow an existing skill owner to move a skill from a personal publisher to an organization publisher, either during republish or through a dedicated transfer flow.

## Best Possible Solution

Land a maintainer-approved publisher-target transfer design that updates skills, aliases, public URLs, HTTP API, CLI behavior, docs, and regression tests consistently; if republish is not the transfer mechanism, hide or constrain owner changes for existing skills.

## Reproduction Assessment

Yes. A source-level reproduction is clear: publish a skill under a personal publisher, select an org publisher while republishing the same slug, and current `insertVersion` rejects the `ownerPublisherId` mismatch as a slug collision.

## Solution Assessment

No current solution is in place. The maintainable path is either a dedicated publisher-target transfer flow with audited role checks or a clearly authorized republish ownership move, not the current UI/backend mismatch.
