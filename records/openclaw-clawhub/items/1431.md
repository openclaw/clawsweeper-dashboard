## Summary

Keep open. Current main still matches the source-level failure path, and the ClawHub cleanup policy for this pass keeps open issues open; the remaining fix is an authorization-sensitive publisher-transfer design rather than conservative cleanup.

## What This Changes

The issue requests a way to move an existing skill from a personal publisher to an organization publisher without hitting the slug-collision publish path.

## Best Possible Solution

Implement a maintainer-approved publisher-target transfer flow, or disable owner changes for existing-skill publish until that flow exists, with skill aliases, URLs, HTTP API, CLI, docs, audit logs, and regression tests kept consistent.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: choose an organization owner while republishing an existing personal skill, publishVersion resolves that publisher, and insertVersion rejects the existing skill because ownerPublisherId differs.

## Solution Assessment

Unclear. Allowing republish to mutate ownerPublisherId would address the symptom, but docs and related work point toward a publisher-target transfer model with explicit authorization and audit rules as the safer maintainable direction.
