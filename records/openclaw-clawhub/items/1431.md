## Summary

Keep open. The reported path still exists on current main: the publish form can submit a different owner handle for an existing skill, while the publish backend rejects an existing skill whose stored publisher differs from the selected publisher; the dedicated transfer path remains user-targeted rather than org-publisher-targeted. ClawHub cleanup policy also keeps issues open even when they look valid.

## What This Changes

The issue requests a supported way to move an existing skill from a personal publisher to an organization publisher without hitting the slug-collision error.

## Best Possible Solution

Land a maintainer-approved publisher-target ownership-transfer design that updates the publish UI, backend transfer model, slug aliases, HTTP API, CLI behavior, docs, and regression tests consistently.

## Reproduction Assessment

Yes, from source inspection. Selecting an organization owner on the publish page submits ownerHandle, publishVersion resolves it to a publisher, and insertVersion rejects the existing skill when the stored ownerPublisherId differs.

## Solution Assessment

Unclear. A narrow bypass in insertVersion would address the immediate error, but ownership transfer is authorization-sensitive and likely needs an explicit maintainer-approved transfer flow or a UI change that hides owner changes for existing skills until that flow exists.
