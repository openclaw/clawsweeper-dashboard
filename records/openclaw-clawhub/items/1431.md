## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still matches the reported gap: the publish UI can submit an organization owner for an existing skill, but the backend rejects a changed `ownerPublisherId` as a slug collision while the transfer system remains user-to-user.

## What This Changes

Allow an existing personal-owned skill to move to an organization publisher instead of failing with a slug-taken error during republish or transfer.

## Best Possible Solution

Land a maintainer-approved publisher-target transfer flow that updates skill ownership, slug aliases, public/API/CLI behavior, docs, audit logs, and regression coverage consistently.

## Reproduction Assessment

Yes. The source-level path is clear on current main: select an organization owner in the publish UI for an existing personal skill, and `insertVersion` rejects the existing skill because its stored publisher differs from the selected publisher.

## Solution Assessment

Unclear. Allowing republish to move ownership may solve the visible failure, but a dedicated publisher-target transfer flow with explicit role checks and audit logging is likely the safer maintainable direction.
