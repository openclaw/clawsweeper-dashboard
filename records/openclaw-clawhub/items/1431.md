## What This Changes

The issue asks for an existing skill to be movable from a personal publisher to an organization publisher instead of failing with a slug-collision error during republish.

## Best Possible Solution

Keep the explicit publish-time owner migration as the supported solution for this report, and let maintainers decide whether to land or narrow a separate publisher-targeted transfer flow.

## Reproduction Assessment

No for the original publish-flow failure on current main: source inspection shows the UI now sends a confirmed `migrateOwner` flag and the backend migrates instead of treating the owner change as a slug collision. I did not run a live authenticated publish.

## Solution Assessment

Yes for the reported publish-flow failure: explicit opt-in migration with source and destination authority checks is the narrow maintainable fix. A direct org-transfer command/API should stay in the separate transfer design because it changes publishing authority.
