## Summary

Keep open. Current main still matches the reported gap: the publish UI can submit an organization owner for an existing skill, but the backend rejects an existing skill whose stored publisher differs from the selected publisher, and the dedicated transfer flow is still user-to-user rather than publisher-targeted.

## What This Changes

Allow an existing skill to move from a personal publisher to an organization publisher through republish or a dedicated ownership-transfer flow.

## Best Possible Solution

Land a publisher-targeted transfer design that consistently updates skills, slug aliases, public/API/CLI behavior, docs, and tests, or hide owner changes during republish and route users to that transfer flow.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: publish a skill under a personal publisher, select an organization owner for the same slug in the publish form, and the backend reaches the `ownerPublisherId` mismatch guard that throws the slug-taken error.

## Solution Assessment

Unclear. Allowing republish to implicitly transfer ownership may solve the symptom, but a dedicated publisher-transfer flow is likely safer unless maintainers explicitly approve republish-time reassignment semantics.
