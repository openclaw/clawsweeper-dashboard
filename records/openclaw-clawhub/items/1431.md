## Summary

Keep open. Current main still matches the reported gap: the publish UI can send an organization owner for an existing skill, but the backend rejects the existing slug when the stored publisher differs, and the supported transfer flow is still user-to-user; related org-transfer PRs are closed unmerged while #1569 remains open.

## What This Changes

Request to let a skill owner move an existing skill from a personal publisher to an organization publisher instead of receiving a slug-taken error on republish.

## Best Possible Solution

Land a maintainer-approved publisher-target transfer design that updates skills, slug aliases, public/API/CLI behavior, docs, audit logs, and tests consistently; if republish should not transfer ownership, hide owner changes for existing skills and route users to that flow.

## Reproduction Assessment

Yes, source-level. Selecting an organization owner for an existing personal skill sends ownerHandle through publishVersion, and insertVersion rejects the existing skill when ownerPublisherId differs.

## Solution Assessment

Unclear. The suggested insertVersion escape hatch may fix the immediate error, but ownership transfer is authorization-sensitive and likely needs a dedicated audited publisher-transfer path or an explicit maintainer decision to make republish perform transfers.
