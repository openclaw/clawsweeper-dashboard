## Summary

Keep open. Current main still matches the reported product gap: the publish flow can submit an organization owner for an existing skill, but the backend rejects a changed publisher as a slug collision, and the dedicated transfer path remains user-to-user rather than publisher-targeted.

## What This Changes

Allow a skill owner with the required organization role to move an existing skill from a personal publisher to an organization publisher.

## Best Possible Solution

Land a maintainer-approved publisher-target transfer design that updates skills, slug aliases, public URLs, HTTP API, CLI behavior, docs, audit logs, and regression tests consistently.

## Reproduction Assessment

Yes. The source-level path is high confidence: select an org owner for an existing skill in the publish UI, then `insertVersion` rejects the existing skill because its stored `ownerPublisherId` differs from the selected publisher.

## Solution Assessment

Unclear. Directly mutating ownership during republish may be too broad; the repo design doc points toward publisher-target ownership transfers with explicit role checks and audit behavior.
