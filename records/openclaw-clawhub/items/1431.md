## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still matches the reported gap: the publish UI can submit an organization owner for an existing skill, but the backend rejects a changed publisher as a slug collision while the transfer path remains user-to-user.

## What This Changes

The issue requests a supported way to move an existing skill from a personal publisher to an organization publisher without hitting a slug-taken error.

## Best Possible Solution

Land a maintainer-approved publisher-target transfer flow that updates skills, slug aliases, public URLs, HTTP API, CLI behavior, docs, audit logs, and regression tests consistently.

## Reproduction Assessment

Yes. The source-level path is clear: select an organization owner in the publish UI for an existing personal skill, publishVersion resolves that owner, and insertVersion rejects the existing skill when ownerPublisherId differs.

## Solution Assessment

Unclear. Allowing republish to move ownership is one possible fix, but a dedicated publisher-transfer flow with explicit role checks and audit logging may be safer for long-term maintenance.
