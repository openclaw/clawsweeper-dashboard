## Summary

Keep open under the ClawHub cleanup policy for issues. Current main appears to fix the reported publish-flow owner move with explicit `migrateOwner` handling, while direct transfer-to-org remains separate product/security work tracked by related open transfer items.

## What This Changes

Allow an existing skill owner to move a skill from a personal publisher to an organization publisher when publishing a new version, or through an equivalent dedicated transfer action.

## Best Possible Solution

Keep the explicit publish-time migration path as the supported fix for the reported UI flow, then have maintainers either manually close this issue or move any remaining direct transfer-to-org requirement to https://github.com/openclaw/clawhub/issues/1675 and https://github.com/openclaw/clawhub/pull/1635.

## Reproduction Assessment

No for the original publish-flow failure: current main requires confirmation, sends `migrateOwner`, and has a backend migration branch plus regression coverage for personal-to-org moves. The separate direct transfer-to-org command/API gap is source-reproducible but is broader work tracked by related transfer items.

## Solution Assessment

Yes for the original publish-flow bug: explicit opt-in plus admin/owner checks is a maintainable fix and avoids silent ownership changes. Unclear for a dedicated transfer-to-org command/API, which needs maintainer product and security decisions before implementation.
