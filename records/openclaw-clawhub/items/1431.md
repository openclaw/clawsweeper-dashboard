## Summary

Keep open under the ClawHub cleanup policy for issues. Current main fixes the reported publish-flow slug collision with explicit owner migration, while direct org-targeted transfer remains separate authorization-sensitive work tracked by related open issues and an open transfer PR.

## What This Changes

The issue asks for an existing skill to be movable from a personal publisher to an organization publisher without hitting a slug-collision error.

## Best Possible Solution

Keep the explicit publish-time `--owner` plus `--migrate-owner` path as the supported answer for this report, and consolidate any direct org-transfer command/API work in the existing publisher-transfer track.

## Reproduction Assessment

No for the original publish-flow slug-collision failure on current main: source inspection shows the UI sends a confirmed `migrateOwner` flag and the backend migrates instead of treating the owner change as a slug collision. Direct transfer-to-org is source-traceable as missing, but that is a separate transfer path.

## Solution Assessment

Yes for the reported publish-flow failure: explicit opt-in migration with source and destination authorization is the narrow maintainable fix. A direct org-transfer command/API should stay in the separate transfer design because it changes future publishing authority.
