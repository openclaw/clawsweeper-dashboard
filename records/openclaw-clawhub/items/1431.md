## Summary

Keep open. Current main still matches the reported failure path, and this ClawHub cleanup pass keeps issues open; the remaining work is an authorization-sensitive publisher-transfer product decision.

## What This Changes

The issue requests a supported way to move an existing skill from a personal publisher to an organization publisher without hitting the existing-skill slug collision path.

## Best Possible Solution

Land a maintainer-approved publisher-target transfer flow for skills, or hide owner changes for existing-skill publish until that flow exists.

## Reproduction Assessment

Yes. Source inspection gives a current-main path: the publish UI can submit an organization `ownerHandle`, the backend resolves it to an `ownerPublisherId`, and `insertVersion` rejects the existing skill when that publisher differs.

## Solution Assessment

Unclear. Mutating `ownerPublisherId` during republish would address the symptom, but a dedicated publisher-target transfer flow with explicit authorization and audit rules is likely the safer maintainable direction.
