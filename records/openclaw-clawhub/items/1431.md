## Summary

Keep open. Current main still matches the reported gap: the publish form can submit an organization publisher for an existing skill, but the backend rejects an existing skill whose stored publisher differs from the selected publisher as a slug collision, and the dedicated transfer flow remains user-to-user/personal-publisher based. The ClawHub cleanup policy for this pass also keeps issues open unless they are PRs already implemented on main.

## What This Changes

Allow an existing skill owner to move a skill from a personal publisher to an organization publisher through republish or a supported transfer flow instead of receiving a slug-taken error.

## Best Possible Solution

Land a maintainer-approved publisher-target transfer design that updates skills, slug aliases, public URL/API behavior, CLI behavior, docs, audit logs, and regression tests consistently. If republish should not move ownership, the existing-skill publish UI should stop offering owner changes and point users to the supported transfer flow.
