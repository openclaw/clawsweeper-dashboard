## Summary

Keep open. Current main still matches the reported gap: the publish UI can submit a different publisher for an existing skill, but the backend treats the existing skill under another ownerPublisherId as a slug collision, and the dedicated transfer flow remains user-to-user only.

## What This Changes

The issue requests allowing an existing skill owner to move a skill from a personal publisher to an organization publisher through republish or a supported transfer flow.

## Best Possible Solution

Land a maintainer-approved publisher-target ownership transfer design that updates the publish UI, transfer API/CLI, slug aliases, public URLs, docs, audit logs, and regression coverage consistently.

## Reproduction Assessment

Yes. Source inspection shows the publish UI can submit an organization ownerHandle for an existing skill, publishVersion resolves that publisher, and insertVersion rejects the different ownerPublisherId with the slug-taken path.

## Solution Assessment

Unclear. Allowing republish to change owner would address the reported UI path, but a dedicated transfer flow may be safer because publisher ownership controls release authority and public URLs.
