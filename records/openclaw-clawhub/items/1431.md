## Summary

Keep open. Current main still follows the reported source path, and the ClawHub cleanup policy keeps issues open; the remaining work is a security-sensitive ownership-transfer decision rather than conservative cleanup.

## What This Changes

Allow an existing personal skill owner to move that skill to an organization publisher, either through republishing with a selected owner or a dedicated transfer action.

## Best Possible Solution

Implement a maintainer-approved publisher-target transfer flow, or disable owner changes for existing-skill publish until that flow exists, with aliases, URLs, API, CLI, docs, audit logs, and regression tests kept consistent.

## Reproduction Assessment

Yes. Source inspection shows a concrete path: select an organization owner while republishing an existing personal skill, publishVersion resolves that owner, and insertVersion rejects the existing skill because ownerPublisherId differs.

## Solution Assessment

Unclear. Allowing republish to mutate ownerPublisherId would address the symptom, but a dedicated publisher-transfer flow with explicit authorization and audit rules may be the more maintainable product direction.
