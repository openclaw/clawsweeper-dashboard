## Summary

Keep open under the ClawHub cleanup policy for issues. Current main fixes the reported publish-form owner migration path with explicit `migrateOwner` handling, while broader direct transfer-to-org behavior is still tracked separately at https://github.com/openclaw/clawhub/issues/1675 and https://github.com/openclaw/clawhub/pull/1635.

## What This Changes

The issue asks for an existing skill owned by a personal publisher to be movable to an organization publisher without hitting the existing-slug collision.

## Best Possible Solution

Keep the guarded publish-time migration as the supported immediate route, then let maintainers decide whether direct org-transfer support should be extracted or replaced under the existing ownership-transfer track.

## Reproduction Assessment

No for the original publish-flow failure: current source shows a confirmed owner change sends `migrateOwner` and the backend migrates instead of throwing the slug-taken collision. The separate direct transfer-to-org path is source-visible as still user-targeted, but that is broader remaining product work.

## Solution Assessment

Yes for the reported publish path: the explicit `migrateOwner` flow is a narrow maintainable fix with role checks, audit logging, alias preservation, and regression coverage. A direct transfer-to-org command/API would need a separate publisher-targeted transfer design rather than a small resolver-only change.
