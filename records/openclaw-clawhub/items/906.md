## Summary

Keep this issue open. It reports a core GitHub OAuth login failure, related OAuth/session reports are still open, and current deployed main still fails the GitHub sign-in endpoint with HTTP 500 before the exact callback/no-session variant can be re-tested.

## What This Changes

The issue asks for GitHub OAuth to return from authorization with an active ClawHub session instead of landing back on the upload page logged out in Chrome.

## Best Possible Solution

Restore production GitHub OAuth so active accounts receive a durable Convex Auth session after callback, preserve deleted/banned/deactivated account blocking, and add focused regression or smoke coverage once the failing layer is identified.

## Reproduction Assessment

Yes for the central current-main GitHub sign-in failure: deployed HEAD prod/v2026.05.10.6 serves the site and API but returns HTTP 500 from both GitHub sign-in endpoints. No for the exact callback-succeeds/no-session variant, because current production fails before authorization can complete.
