## Summary

Keep open: current main still relies on a dedicated, guarded recovery path for partial CLI releases, and the linked preflight does not prove that recovery remains possible after this change.

## What This Changes

Removes the standalone CLI GitHub Release create/repair workflow and its recovery instructions from the deployment specification.

## Best Possible Solution

Keep a guarded release-only recovery path—standalone or incorporated as an explicit non-publishing mode—and preserve an exact, owner-approved partial-release procedure in `specs/deploy.md`.

## Reproduction Assessment

Yes for the operational failure boundary. Current-main source shows the normal workflow rejects an already-published package before reaching GitHub Release creation, while the proposed deletion removes the separate recovery path.

## Solution Assessment

No. Deleting the recovery workflow without an equivalent no-publish mode or an approved manual procedure is not the safest maintainable solution.
