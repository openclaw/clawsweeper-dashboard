## Summary

This focused cleanup is not ready to merge: current main publishes to npm before creating the GitHub Release, and the deleted workflow is the documented recovery mechanism when that later step fails. The collaborator-authored PR also requires explicit maintainer handling rather than automated closure.

## What This Changes

The PR removes the standalone CLI GitHub Release create/repair workflow and deletes its partial-release recovery instructions from the deployment specification.

## Best Possible Solution

Consolidate recovery into an explicit no-publish mode in the npm release workflow with equivalent proof validation and existing-release safeguards, or retain the standalone workflow if maintainers prefer the current separation.

Yes at source level: workflow ordering establishes the partial-success state, and current main explicitly assigns its recovery to the workflow being deleted. This read-only review did not intentionally induce a failed production release.
