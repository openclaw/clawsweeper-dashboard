## Summary

Current main still uses the standalone workflow for a distinct partial-release recovery case: the primary workflow publishes to npm before creating the GitHub Release, while the deleted workflow can repair that later step without republishing. Removing both the workflow and its runbook therefore creates an operational regression, and this collaborator-authored automation change requires explicit owner judgment.

## What This Changes

The PR deletes the standalone CLI GitHub Release create/repair workflow and removes its partial-release recovery instructions from `specs/deploy.md`.

## Best Possible Solution

Keep an idempotent, documented no-republish GitHub Release recovery mechanism, preferably by adding an explicit repair-only mode to the npm release workflow before deleting the standalone workflow.

## Reproduction Assessment

Not applicable as a conventional bug reproduction; source inspection does establish the operational failure boundary because npm publishing precedes GitHub Release creation and the deleted workflow is the documented recovery mechanism.

## Solution Assessment

No; deleting the recovery mechanism without an equivalent no-publish replacement or an explicitly approved manual contract is not the safest maintainable solution.
