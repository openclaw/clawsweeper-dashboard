## Label Justifications

- `P2`: This is a bounded release-automation change with a material but infrequent partial-release recovery impact.
- `merge-risk: 🚨 automation`: Merging can strand a CLI release after npm succeeds by removing the automated GitHub Release repair path.

## Summary

Current main still relies on this standalone workflow as the guarded recovery path when npm publishing completes but GitHub Release creation fails; deleting it without an approved equivalent would make partial CLI releases harder and less safely recoverable. The PR is collaborator-authored and touches a security-operations-owned workflow, so it also requires explicit maintainer judgment rather than cleanup closure.

## What This Changes

The PR deletes the standalone CLI GitHub Release create/repair workflow and removes its recovery instructions from the deployment specification.

## Best Possible Solution

Retain a guarded no-npm-publish GitHub Release repair mode—either standalone or integrated into the main release workflow—and document and exercise the partial-release recovery procedure before removing the current path.

## Reproduction Assessment

Yes, from current-main source: npm publication and GitHub Release creation are sequential, and the standalone workflow is the documented route for repairing the second step without republishing. This read-only review did not intentionally trigger a failed production release.

## Solution Assessment

No. Outright deletion is not the narrowest safe solution unless release and security-operations owners explicitly approve and prove an equivalent guarded or manual recovery contract.
