## Summary

The report is still necessary because current main supports proxy environments but continues to present underlying Node transport failures as the same generic `fetch failed` message, while the affected macOS setups lack enough detail to identify whether DNS, TLS, proxy, runtime, or local networking is responsible. The related https://github.com/openclaw/clawhub/issues/861 shows the same unresolved login symptom, so there is no implemented-on-main or safe duplicate-close basis yet.

## What This Changes

Investigate ClawHub CLI search and login commands failing with a generic `fetch failed` error on macOS.

## Best Possible Solution

Preserve the existing proxy support, add redacted transport cause diagnostics to the shared CLI error path, and validate search and login on an affected current-version macOS environment before deciding whether a deeper network fix is needed.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction of the network failure on current main. Source inspection does reproduce the diagnostic limitation: an underlying transport cause can reach the CLI as only the generic top-level `fetch failed` message.

## Solution Assessment

Yes, the narrowest maintainable next step is safe cause-chain diagnostics plus an affected-environment check; treating every instance as a proxy failure would be unsupported.
