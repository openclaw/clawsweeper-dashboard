## Summary

Keep open. This issue remains actionable under the ClawHub policy for issues: current main has proxy support and docs for one common cause, but search and token login still surface low-level transport failures through a raw shared formatter, matching the repeated `fetch failed` reports.

## What This Changes

The issue reports ClawHub CLI commands returning raw `fetch failed` errors during search and login.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error boundary for DNS, TLS/certificate, proxy, timeout/abort, and curl failures while preserving existing proxy support and HTTP semantics.

Yes for the observable diagnostics gap: an `Error("fetch failed")` thrown from `apiRequest` reaches `cmdSearch` or `cmdLogin`, and `formatError` renders only the raw message. The exact reporter network root cause remains unclear.

## Solution Assessment

Yes. A shared HTTP/error-formatting fix is the narrowest maintainable solution because search, login, and other CLI commands already converge on the same boundary.
