## Summary

Keep open. Current main still has a source-supported path where CLI search and token login can surface low-level transport failures as raw `fetch failed`, and ClawHub issue cleanup policy does not auto-close issues that remain actionable.

## What This Changes

The issue asks for ClawHub CLI commands such as `search` and token `login` to handle `fetch failed` failures with actionable behavior instead of raw transport output.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance without changing request semantics.

## Reproduction Assessment

Yes for a source-level path: inject an Undici-style `TypeError('fetch failed')` into `fetchWithTimeout` or a non-zero curl result into the Bun path, and current `cmdSearch`/`cmdLogin` still print only `formatError(error)`. I did not recreate the reporters' exact macOS network environment.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, package, and sync commands already converge through that boundary.
