## Summary

Keep open. This is a ClawHub issue, so cleanup policy does not auto-close it, and current main still has a source-supported CLI path where search/login transport failures can surface as raw `fetch failed` without actionable diagnostics.

## What This Changes

The issue reports `clawhub search "web"` failing with raw `fetch failed` output, with later comments showing similar failures for token login.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error boundary, with focused search/login coverage and troubleshooting guidance for DNS, TLS, proxy, timeout/abort, Undici, and curl failures.

## Reproduction Assessment

Yes for the raw diagnostics path: source inspection shows fetch/curl `Error` messages flow through `apiRequest` into search and login, then `formatError` emits only the message. The exact reporter network root cause was not reproduced from the available data.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, and related CLI commands already converge through the same request and formatter code.

## Repair Work Prompt

Add centralized, redacted transport-error formatting at the shared HTTP/error boundary so search, token login, browser-login verification, install/download, publish, and Bun curl paths produce actionable guidance for DNS, TLS/certificate, proxy, timeout/abort, Undici, and curl failures while preserving existing request semantics. Add focused regression coverage for `formatError`/transport classification plus search/login paths, and update troubleshooting docs if the CLI guidance changes. Preserve existing HTTP status handling, retries, rate-limit hints, timeout behavior, and proxy env support from https://github.com/openclaw/clawhub/pull/363.
