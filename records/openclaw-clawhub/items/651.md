## Summary

Keep open. Current main still has a source-level path where `clawhub search` and token login transport failures are printed through the generic formatter as raw `fetch failed`, while the merged proxy and Node 22 fixes cover only specific root causes; ClawHub cleanup policy also keeps issues open until a fix lands.

## What This Changes

The issue reports the ClawHub CLI printing a raw `fetch failed` error for `clawhub search`, with later comments showing the same raw message for token login.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error-formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance while preserving request semantics.

## Reproduction Assessment

Yes for the source-level raw-message path: make `fetchWithTimeout` receive an Undici-style `TypeError('fetch failed')` or make the Bun curl path fail, then `cmdSearch` and `cmdLogin` still render `formatError(error)`. The original reporters' exact network environments were not independently reproduced.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, and related CLI commands already converge through this boundary.
