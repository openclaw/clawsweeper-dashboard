## Summary

Keep open. Current main has proxy support and troubleshooting docs, but `search`, token login, and browser-login token verification can still surface transport failures through the shared formatter as only the top-level `fetch failed` message; ClawHub cleanup policy also keeps issues open rather than auto-closing them.

## What This Changes

The issue reports `clawhub search` and related CLI commands failing with an unhelpful `fetch failed` transport error.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error-formatting boundary, with search/login coverage and troubleshooting updates, while preserving existing request semantics.

## Reproduction Assessment

Yes for the source-level failure path: if `apiRequest` receives an Undici `TypeError('fetch failed')` or curl transport failure, search and login still print only `formatError(error)`. The reporters' exact network root cause was not reproduced from the available environment data.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, and related CLI commands already converge through that boundary.
