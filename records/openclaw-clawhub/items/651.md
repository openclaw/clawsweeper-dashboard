## Summary

Keep open. Current main still routes search/login transport failures through the shared CLI formatter as raw `Error.message`, and ClawHub cleanup policy keeps issues open rather than auto-closing them.

## What This Changes

Investigate `clawhub search` and login surfacing raw `fetch failed` transport errors without actionable diagnostics.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error-formatting boundary for DNS, TLS/certificate, proxy, timeout/abort, Undici, and curl failures.

## Reproduction Assessment

Yes for the raw diagnostics path: source inspection shows fetch/curl `Error` messages flow through `apiRequest` into search and login, then `formatError` emits only the message. The exact reporter network root cause was not reproduced from the available environment data.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, and related CLI commands already converge through the same request and formatter code.

## Repair Work Prompt

Fix the ClawHub CLI transport diagnostics for raw `fetch failed` errors. Add centralized classification and redaction at the shared HTTP/error-formatting boundary, likely in `packages/clawhub/src/http.ts` and `packages/clawhub/src/cli/ui.ts`, so search, login/whoami, install, publish, and related CLI commands produce actionable messages for DNS, TLS/certificate, proxy, timeout/abort, Undici, and curl failures. Preserve HTTP status errors, retry/rate-limit behavior, successful requests, EnvHttpProxyAgent behavior, Bun curl fallback, and auth semantics. Do not add a new config option. Redact Authorization headers, API tokens, token query parameters, and proxy credentials. Add focused regression coverage for `formatError`/transport errors plus the `cmdSearch` and token-login paths, and update troubleshooting docs or changelog only if user-facing wording changes.
