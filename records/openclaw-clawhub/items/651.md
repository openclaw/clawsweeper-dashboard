## Summary

Keep open. Current main still has a source-supported CLI path where `search` and token/login verification can surface transport failures as the raw top-level `fetch failed` message, and ClawHub policy keeps issues open unless they are resolved by a PR/implementation decision.

## What This Changes

The issue reports `clawhub search "web"` failing with only `fetch failed` and no actionable CLI guidance.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance while preserving request semantics.

## Reproduction Assessment

Yes, for a source-level path: make `fetchWithTimeout` receive an Undici-style `TypeError('fetch failed')` or make the Bun curl path fail, then `cmdSearch` and token `cmdLogin` still render `formatError(error)`. The exact reporter networks were not independently reproduced.

## Solution Assessment

Yes. A shared HTTP/error-formatting diagnostics repair is the narrowest maintainable fix because search, login, install, publish, and related CLI commands already converge through that boundary.
