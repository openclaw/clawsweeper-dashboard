## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still has a source-supported path where CLI transport failures in search or login can surface only the raw `fetch failed` message despite the merged proxy and Node 22 fixes.

## What This Changes

The issue reports ClawHub CLI commands such as `search` and token login failing with the generic `fetch failed` message.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error-formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance while preserving request semantics.

## Reproduction Assessment

Yes for a source-level path: make `fetchWithTimeout` receive an Undici-style `TypeError('fetch failed')` or make the Bun curl path fail, then `cmdSearch` and `cmdLogin` still render `formatError(error)`. The exact reporter network environments were not independently reproduced.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, and related CLI commands already converge through this boundary.
