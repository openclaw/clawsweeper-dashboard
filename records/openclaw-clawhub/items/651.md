## Summary

Keep open. Current main still has a source-supported path where CLI search and token login transport failures surface as raw `fetch failed`, and ClawHub cleanup policy does not auto-close issues that remain actionable.

## What This Changes

The issue asks why `clawhub search "web"` fails with raw `fetch failed` output, with later comments reporting the same symptom for token login.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance without changing request semantics.

## Reproduction Assessment

Yes for a source-level path: make `fetchWithTimeout` throw an Undici-style `TypeError('fetch failed')` or make the Bun curl path return non-zero, and current `cmdSearch`/`cmdLogin` still print only `formatError(error)`. I did not recreate the reporters' exact macOS network environment.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, package, and sync already converge through that boundary.
