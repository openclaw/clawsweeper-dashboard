## Summary

Keep open. Current main still has a source-supported CLI path where search and token login can render low-level transport failures as the raw top-level `fetch failed` message, and ClawHub cleanup policy keeps issues open unless there is a stronger terminal reason.

## What This Changes

The issue reports `clawhub search "web"` failing with a raw `fetch failed` message in the ClawHub CLI.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance while preserving existing request semantics.

## Reproduction Assessment

Yes for a source-level path: make `fetchWithTimeout` receive an Undici-style `TypeError('fetch failed')` or make a Bun curl path fail, then `cmdSearch` and token `cmdLogin` still render `formatError(error)`. The exact reporter environments were not independently reproduced.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, package, and sync commands already converge through this boundary.
