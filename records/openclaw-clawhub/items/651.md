## Summary

Keep open: current main still routes CLI search and token-login transport failures through a generic formatter that can print only `fetch failed`, and ClawHub cleanup policy does not auto-close issues.

## What This Changes

The issue reports `clawhub search "web"` and later login commands failing with a generic `fetch failed` message.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error-formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance while preserving existing request and token-verification semantics.

## Reproduction Assessment

Yes, at source level: inject an Undici-style `TypeError('fetch failed')` or a nonzero curl transport failure into the HTTP path used by `cmdSearch` or token login, and current main prints `formatError(error)`. I did not recreate the reporters' exact network environments.

## Solution Assessment

Yes. A shared, redacted HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, package, and sync already converge through that boundary.
