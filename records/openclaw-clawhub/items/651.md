## Summary

Keep open. Current main has proxy support and troubleshooting docs, but the shared CLI error path can still surface low-level transport failures as raw messages such as `fetch failed`; under the ClawHub cleanup policy, issues should stay open unless maintainers decide otherwise.

## What This Changes

The issue reports `clawhub search "web"` failing with the unhelpful CLI message `fetch failed`.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error boundary for DNS, TLS/certificate, proxy, timeout/abort, connection, and curl failures while preserving existing request semantics.

## Reproduction Assessment

Yes for the observable diagnostics gap: source inspection shows `Error("fetch failed")` or curl stderr can flow from the HTTP layer into `cmdSearch` or `cmdLogin` and be rendered unchanged by `formatError`. The exact user network root cause is still unclear from the report data.

## Solution Assessment

Yes. A shared HTTP/error-formatting fix is the narrowest maintainable solution because search, login, install, publish, and related CLI commands already converge through the same request and formatting paths.
