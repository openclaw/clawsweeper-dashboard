## Summary

Keep open. Current main still has a source-supported path where CLI search and token-login transport failures are surfaced as raw `fetch failed` messages, and the merged proxy/Node fixes cover specific causes rather than the remaining diagnostics gap.

## What This Changes

The issue asks for the ClawHub CLI to handle `clawhub search` and related login failures that currently print only `fetch failed`.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance while preserving existing request semantics and login token verification.

## Reproduction Assessment

Yes for a source-level path: inject an Undici-style `TypeError('fetch failed')` or a non-zero curl result into the HTTP path used by `cmdSearch` or `cmdLogin`, and current main prints only `formatError(error)`. I did not reproduce the reporters' exact macOS network environment.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, package, and sync already converge through that boundary.
