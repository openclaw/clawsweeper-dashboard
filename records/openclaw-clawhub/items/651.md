## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still leaves the reported CLI transport-diagnostics gap: `search` and token login can surface low-level transport failures as raw messages.

## What This Changes

Report that `clawhub search` and related login flows can fail with an unhelpful `fetch failed` CLI message.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error formatting boundary while preserving existing proxy support and HTTP semantics.

## Reproduction Assessment

Yes for the observable diagnostics gap: an `Error("fetch failed")` from the shared HTTP path is printed unchanged by `formatError` in search and login. The exact external network condition from the reporters remains environment-specific.

## Solution Assessment

Yes. The maintainable fix is a shared HTTP/error-formatting improvement rather than per-command handling, because the same transport path is used by search, login, and other CLI commands.

## Repair Work Prompt

Implement centralized ClawHub CLI transport diagnostics for the reported raw `fetch failed` failures. In `packages/clawhub/src/http.ts` and/or `packages/clawhub/src/cli/ui.ts`, classify common transport failures from Node fetch/Undici and Bun curl into concise actionable messages for DNS, TLS/certificate, proxy, timeout/abort, and generic curl failures. Preserve existing HTTP status errors, retries, rate-limit hints, timeout behavior, and proxy support from #363. Redact Authorization headers, API tokens, token query parameters, and credentials embedded in proxy URLs or environment variables. Add focused unit tests for Node fetch throwing `Error("fetch failed")` with causes/codes, timeout/abort behavior, and Bun curl stderr. Add a `CHANGELOG.md` entry because this is a user-facing CLI fix. Do not add live network calls, change registry API semantics, or print secrets.
