## Summary

Keep open. Current main has proxy support and troubleshooting docs, but `search` and token login can still display raw transport failures through the shared formatter, so the repeated `fetch failed` reports remain actionable.

## What This Changes

The issue reports `clawhub search "web"` failing with a raw `fetch failed` CLI error, with later comments reporting the same symptom for search and token login.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error boundary for DNS, TLS/certificate, proxy, timeout/abort, connection, and curl failures while preserving existing request semantics.

## Reproduction Assessment

Yes for the observable diagnostics gap: source inspection shows `Error("fetch failed")` or curl stderr can flow from the HTTP layer into `cmdSearch` or `cmdLogin` and be rendered unchanged by `formatError`. The exact user network root cause is still not reproducible from the report data alone.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable direction because search, login, install, publish, and related CLI commands already converge through the same request and formatting paths.
