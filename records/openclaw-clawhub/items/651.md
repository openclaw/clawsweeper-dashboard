## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still has a source-supported CLI path where search/login transport failures can be shown as raw `fetch failed` despite the merged proxy and Node transport fixes.

## What This Changes

The issue reports `clawhub search "web"` failing with a raw `fetch failed` CLI error.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error-formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance while preserving request semantics.

## Reproduction Assessment

Yes, for a source-level path: make `fetchWithTimeout` receive an Undici-style `TypeError('fetch failed')` or make the Bun curl path fail, then `cmdSearch` and token `cmdLogin` still render `formatError(error)`. The exact reporter environments were not independently reproduced.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, and related CLI commands already converge through that boundary.

Add centralized, redacted transport-error classification so Undici fetch failures, DNS/TLS/certificate/proxy/timeout/abort cases, and Bun curl failures produce actionable CLI guidance without changing HTTP status handling, retries, rate-limit messages, successful requests, or token verification semantics. Include regression coverage for `cmdSearch`, token login or shared formatter behavior, Node fetch errors with nested causes, and Bun curl stderr.
