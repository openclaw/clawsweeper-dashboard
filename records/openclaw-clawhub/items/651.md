## Summary

Keep open. Current main includes proxy support and troubleshooting docs, but the shared CLI error path still renders transport failures as raw `Error.message`, so the reported `fetch failed` experience remains an actionable CLI diagnostics issue.

## What This Changes

The issue reports `clawhub search "web"` failing with a raw `fetch failed` error from the ClawHub CLI.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error boundary for DNS, TLS/certificate, proxy, timeout/abort, connection, and curl failures while preserving existing proxy support and HTTP semantics.

## Reproduction Assessment

Yes for the observable diagnostics gap: a transport-level `Error("fetch failed")` from `apiRequest` reaches `cmdSearch` or `cmdLogin` and is rendered unchanged by `formatError`. The exact reporter network root cause is still unclear because the environment details are incomplete.

## Solution Assessment

Yes. A shared HTTP/error-formatting fix is the narrowest maintainable path because search, login, and other CLI commands already converge there; special-casing only search or assuming proxy configuration would leave the same raw output elsewhere.
