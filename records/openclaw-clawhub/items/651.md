## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-supported CLI diagnostics gap where search and token login can surface raw `fetch failed` transport errors despite proxy support and troubleshooting docs.

## What This Changes

The issue reports `clawhub search` failing with `fetch failed`, with later discussion showing the same generic failure for search and token login.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error-formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance while preserving existing request semantics.

## Reproduction Assessment

Yes for the source-level failure path: make `apiRequest` receive an Undici `TypeError('fetch failed')` or curl transport failure, and `cmdSearch`/token login still print `formatError(error)`. The reporters' exact network root cause was not reproduced from the available environment data.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, and related CLI commands already converge through that boundary.
