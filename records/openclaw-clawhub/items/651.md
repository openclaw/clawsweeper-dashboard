## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-supported path where search and token login surface transport failures as only the raw top-level `fetch failed` message; the merged proxy work at https://github.com/openclaw/clawhub/pull/363 fixed one common cause but did not close the shared diagnostics gap.

## What This Changes

The issue reports `clawhub search` and later token/browser login failing with raw `fetch failed` output.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error-formatting boundary, with login/search coverage and troubleshooting updates, while preserving existing request semantics.

## Reproduction Assessment

Yes for the source-level diagnostics path: if `apiRequest` receives an Undici `TypeError('fetch failed')` or curl transport failure, search and token login still print only `formatError(error)`. The reporters' exact network root cause was not reproduced from the available environment data.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, and related CLI commands already converge through that boundary.
