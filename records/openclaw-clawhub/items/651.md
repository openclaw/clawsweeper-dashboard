## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still sends CLI transport failures through a raw shared formatter; the merged proxy support from #363 narrows one root cause but does not provide actionable diagnostics for the reported `fetch failed` path.

## What This Changes

Improve the ClawHub CLI’s handling of `fetch failed` during search and login so transport failures produce actionable diagnostics instead of raw error text.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error boundary for DNS, TLS/certificate, proxy, timeout/abort, connection, and curl failures while preserving the existing proxy support and HTTP semantics.

## Reproduction Assessment

Yes for the observable diagnostics gap: a transport-level `Error("fetch failed")` thrown from `apiRequest` reaches `cmdSearch` or `cmdLogin` and is rendered unchanged by `formatError`. The exact reporter network root cause is unclear because the environment details are incomplete.

## Solution Assessment

Yes, a shared HTTP/error-formatting fix is the narrowest maintainable path because search, login, and other CLI commands already converge there. Special-casing only `search` or assuming proxy configuration would leave the same raw output elsewhere.
