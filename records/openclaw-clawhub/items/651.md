## Summary

Keep open. Under the ClawHub cleanup policy this issue should not be auto-closed, and current main still routes transport failures through shared CLI paths that can print raw messages such as `fetch failed` for search and login.

## What This Changes

The issue reports `clawhub search "web"` failing with a raw `fetch failed` CLI error, with later comments reporting the same symptom for search and token login.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error boundary for DNS, TLS/certificate, proxy, timeout/abort, connection, and curl failures while preserving existing request semantics.

## Reproduction Assessment

Yes for the observable diagnostics gap: source inspection shows a transport `Error("fetch failed")` can propagate from `fetchWithTimeout` into `cmdSearch` or `cmdLogin` and be rendered unchanged by `formatError`. The original users' network root cause is not fully reproducible from the issue data alone.

## Solution Assessment

Yes. A shared HTTP/error-formatting fix is the narrowest maintainable solution because search, login, install, publish, and other CLI commands already converge through the same request and formatting paths.

Preserve existing HTTP status errors, retries, timeout values, rate-limit hints, proxy support, and Bun curl behavior.
