## Summary

Keep open. The repeated CLI `fetch failed` reports still map to a current shared error path where search and login can display only the raw transport message, and ClawHub cleanup policy keeps issues open unless they are PR cleanup candidates.

## What This Changes

The issue reports `clawhub search "web"` failing with a raw `fetch failed` CLI error, with later comments showing the same error for search and token login.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error-formatting boundary so search, login, install, and publish failures explain likely DNS, TLS, proxy, timeout/abort, and curl causes while preserving existing request semantics.

## Reproduction Assessment

Yes for the source-level diagnostics path: if `apiRequest` receives a fetch or curl transport failure, search and token login still print only `formatError(error)`, which returns the top-level `Error.message`. The reporters' exact network root cause was not reproduced from the available environment data.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because the affected search, login, install, publish, and related CLI commands already converge through that boundary.
