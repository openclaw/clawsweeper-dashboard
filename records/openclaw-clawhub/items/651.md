## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a valid CLI diagnostics gap: proxy and Node 22 fixes exist, but search and login can still surface raw transport failures as `fetch failed` instead of actionable guidance.

## What This Changes

The issue reports `clawhub search "web"` failing with a generic `fetch failed` CLI error, with later comments showing the same symptom for search and token login.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error-formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance without changing request semantics.

## Reproduction Assessment

Yes for the source-level failure path: if `apiRequest` receives an Undici-style `TypeError('fetch failed')` or curl transport failure, search and token login still print the shared formatted message. The reporters' exact network root cause was not reproduced from their macOS environments.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, and related CLI commands already converge through that boundary.
