## Summary

Keep open. Current main still lets CLI transport failures from search and token login surface as raw `fetch failed`, so this remains an actionable CLI diagnostics bug rather than a cleanup-close candidate.

## What This Changes

The issue reports the ClawHub CLI printing only `fetch failed` for `clawhub search` and related login/network commands.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error-formatting boundary so search and login failures produce actionable DNS, TLS, proxy, timeout, abort, and curl guidance while preserving existing request semantics.

## Reproduction Assessment

Yes at source level: inject an Undici-style `TypeError('fetch failed')` or non-zero curl transport failure into the HTTP path used by `cmdSearch` or `cmdLogin`, and current main prints only `formatError(error)`. I did not recreate the reporters' exact network environments.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, install, publish, package, and sync already converge through that boundary.
