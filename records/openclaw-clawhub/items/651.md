## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still lets transport failures from the CLI HTTP layer surface as raw messages like `fetch failed`; the merged proxy support in #363 covers one likely root cause but not the shared diagnostics gap.

## What This Changes

The issue reports that `clawhub search "web"` fails with a raw `fetch failed` CLI error.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error boundary for DNS, TLS/certificate, proxy, timeout/abort, connection, and curl failures while preserving existing proxy support and HTTP semantics.

## Reproduction Assessment

Yes for the observable diagnostics path: source inspection shows a transport-level `Error("fetch failed")` can reach `cmdSearch` or token login and be rendered unchanged by `formatError`. The exact reporter network root cause is not reproducible from the issue because environment details are missing.

## Solution Assessment

Yes. A shared HTTP/error-formatting repair is the narrowest maintainable solution because search, login, and other CLI commands already converge on `apiRequest` plus `formatError`; special-casing only `search` would leave the same raw output elsewhere.

## Repair Work Prompt

Fix the CLI diagnostics gap reported here: when shared HTTP requests fail at the transport layer in Node fetch/Undici or Bun curl, the CLI should emit a redacted, actionable message instead of only `fetch failed` or raw curl stderr. Implement centrally in `packages/clawhub/src/http.ts` and/or `packages/clawhub/src/cli/ui.ts`; cover search and token-login behavior through shared tests. Recognize DNS, TLS/certificate, proxy, timeout/abort, connection refused/reset, and curl failure details where feasible. Preserve existing HTTP status, retry, rate-limit, timeout, and #363 proxy-env behavior. Redact Authorization headers, API tokens, token query parameters, and credentials in proxy URLs/env values. Add a user-facing `CHANGELOG.md` entry if behavior changes.
