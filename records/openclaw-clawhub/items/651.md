## Summary

Keep open. Current main still has a clear CLI path where search and login transport failures can be surfaced as the raw top-level `fetch failed` message, and ClawHub policy does not auto-close issues in this cleanup pass.

## What This Changes

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics in the shared HTTP/error formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance without changing successful request behavior.

## Repair Work Prompt

Fix ClawHub CLI raw transport diagnostics for search/login failures like this issue. On current main, `cmdSearch` and `cmdLogin` catch `apiRequest` failures and display `formatError(error)`, while `formatError` returns only the top-level `Error.message`; `fetchWithTimeout` rethrows Undici `Error` objects unchanged and Bun curl paths throw raw stderr or `curl failed`. Add centralized, redacted diagnostics in the shared HTTP/UI boundary for DNS, TLS/certificate, proxy, timeout/abort, curl, and nested Undici cause failures. Preserve successful request behavior, HTTP status/rate-limit messages, retries, timeout durations, Bun curl behavior, and `/api/v1/whoami` token verification. Do not add config flags or broaden product behavior. Add focused regression coverage for Node and Bun transport failures plus search/login formatting, and update troubleshooting docs if the new guidance changes user-facing diagnostics. Related refs: https://github.com/openclaw/clawhub/issues/861, https://github.com/openclaw/clawhub/pull/363, https://github.com/openclaw/clawhub/pull/245. This is a user-facing CLI fix, so note that maintainers may add a changelog entry at landing; do not make contributor credit/changelog text the core repair.
