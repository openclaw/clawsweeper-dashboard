Keep open. Current main still uses the non-www registry/site defaults, the live dummy-token probe no longer matches the originally reported non-www to www redirect, and ClawHub issue policy leaves issues open for maintainer verification rather than auto-closing them.

## What This Changes

The issue asks ClawHub to fix CLI token authentication failing when requests to the default non-www registry are redirected to a www host and lose the Authorization header.

## Best Possible Solution

Maintainers should verify the current published CLI with a real token, then either close or consolidate with #99 if non-www works, or make one canonical-domain fix across CLI defaults, discovery/docs, and hosting if redirect auth still fails.

## Solution Assessment
