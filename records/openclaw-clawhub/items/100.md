## Summary

Keep open. ClawHub cleanup policy keeps issues open for maintainer judgment, and current source plus live routing no longer match the original non-www to www redirect theory closely enough to close or automate a fix without real production-token verification.

## What This Changes

The issue reports CLI token auth failing because requests to the default `https://clawhub.ai` registry allegedly redirect to `https://www.clawhub.ai` and lose the bearer Authorization header.

## Best Possible Solution

Maintainers should verify fresh production tokens against the current canonical non-www flow, then either consolidate this with the broader Unauthorized tracker or make one narrow fix at the proven failing layer with registry/auth regression coverage.

## Reproduction Assessment

No. Source inspection confirms the CLI auth path is redirect-sensitive, but the live dummy-token check no longer reproduces the reported `clawhub.ai` to `www.clawhub.ai` redirect and this review had no real production token for a full login/whoami check.

## Solution Assessment

Unclear. Switching defaults to `https://www.clawhub.ai` is not the best proven fix today because live `www` redirects back to non-www; maintainers should verify the real-token failure mode before changing source or hosting configuration.
