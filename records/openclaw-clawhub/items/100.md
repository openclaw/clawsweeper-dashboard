## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main plus live routing do not prove the reported valid-token Unauthorized path is fixed or still caused by the old apex-to-www redirect.

## What This Changes

The issue reports that CLI token authentication fails when the default registry crosses from `clawhub.ai` to `www.clawhub.ai`, losing the bearer Authorization header.

## Best Possible Solution

Maintainers should verify fresh production browser-created and Settings-created tokens against the current released CLI and canonical host, then either consolidate the Unauthorized cluster or land one focused auth/domain fix with regression coverage.

## Reproduction Assessment

No. Source inspection confirms the redirect-sensitive token verification path, and dummy live probes show the original redirect direction no longer occurs, but a fresh real production token is required to reproduce or clear the reported Unauthorized failure with high confidence.

## Solution Assessment

Unclear. Switching defaults to `https://www.clawhub.ai` was a plausible historical workaround, but current source, discovery, docs, live routing, and the closed unmerged host-switch PR all point at the apex host as canonical now.
