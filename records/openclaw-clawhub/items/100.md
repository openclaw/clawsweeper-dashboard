## Summary

Keep open under the ClawHub issue policy. Current main and live discovery still use the non-www canonical host, the originally reported non-www to www redirect no longer reproduces with a dummy bearer request, and the remaining auth question needs maintainer verification with a real production token.

## What This Changes

The issue reports that CLI authentication fails because authenticated requests to `https://clawhub.ai` redirect to `https://www.clawhub.ai` and lose the bearer token.

## Best Possible Solution

Maintainers should verify the current published CLI against production with a fresh real token; if the non-www flow works, consolidate or close this with the broader auth tracker, and if it fails, make one narrow canonical-domain/auth fix across hosting, CLI defaults, discovery, tests, and docs.

## Reproduction Assessment

No. Source inspection confirms a redirect-sensitive auth path, but the live dummy-token probe did not reproduce the reported `clawhub.ai` to `www.clawhub.ai` redirect and no real production token was available for a full CLI login and `whoami` check.

## Solution Assessment

Unclear. Switching defaults to `https://www.clawhub.ai` is not currently proven correct because live production now redirects www back to non-www; maintainers should reproduce with real tokens before choosing a source or hosting fix.
