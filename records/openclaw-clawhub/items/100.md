## Summary

Keep open under the ClawHub issue policy. Current main still uses `https://clawhub.ai` as the CLI/discovery default, the originally reported non-www to www redirect no longer reproduces in a live dummy-token probe, and the remaining auth question needs maintainer verification with a real production token plus coordination with the related Unauthorized report.

## What This Changes

The issue reports that CLI token auth fails when the default `https://clawhub.ai` registry redirects to `https://www.clawhub.ai` and strips the bearer `Authorization` header.

## Best Possible Solution

Maintainers should verify the current published CLI with a fresh real token against `https://clawhub.ai`; if it works, close or consolidate this with `https://github.com/openclaw/clawhub/issues/99`, and if it fails, land one narrow canonical-domain/auth fix across hosting, discovery, CLI defaults, tests, and docs.

## Reproduction Assessment

No. The originally reported non-www to www redirect did not reproduce with a dummy bearer request, and this read-only review did not have a valid production token to verify the full CLI login and `whoami` flow.

## Solution Assessment

No. Switching defaults to `https://www.clawhub.ai` is not currently the best proven fix because production now redirects `www` back to non-www; maintainers should reproduce with real tokens before choosing a source or hosting change.
