## Summary

Keep open. The original redirect theory no longer matches current source or live routing, but ClawHub issue policy and the lack of a real-token production check mean this should stay with maintainers, likely alongside the broader Unauthorized tracker.

## What This Changes

Report that CLI token authentication fails because the default registry request to clawhub.ai is redirected to www.clawhub.ai and loses the bearer Authorization header.

## Best Possible Solution

Maintain a verified non-www canonical auth flow, consolidate with https://github.com/openclaw/clawhub/issues/99 if that is the active tracker, and make a narrow code, docs, or hosting change only if real-token production verification still fails.

## Reproduction Assessment

No. A dummy bearer request to the current non-www endpoint returns 401 without a redirect and www redirects back to non-www, but I did not have a real production token to verify the full CLI auth flow.

## Solution Assessment

No. The older default switch to www is not currently the safest fix because live www redirects back to non-www and the linked PR was closed as obsolete; maintainer production-token verification should come first.
