## Summary

Keep open. The report and workaround were valid enough to trace, but ClawHub cleanup policy keeps issues open, the current source still uses `https://clawhub.ai` as canonical, and live routing now redirects `www.clawhub.ai` back to non-www, so this needs maintainer verification with a real production token rather than cleanup closure.

## What This Changes

The issue reports that CLI token authentication returns Unauthorized when the default `clawhub.ai` registry crosses a `www.clawhub.ai` redirect that strips the Authorization header.

## Best Possible Solution

Maintainers should verify fresh production API tokens against the current CLI and canonical non-www host, then either consolidate this under the broader Unauthorized tracker or land one focused auth/domain fix at the proven failing layer with regression coverage.

## Reproduction Assessment

No. Source inspection confirms a redirect-sensitive auth path, but the current live dummy-token probe did not reproduce the reported `clawhub.ai` to `www.clawhub.ai` redirect and a real production token is needed for a high-confidence login and `whoami` reproduction.

## Solution Assessment

Unclear. Switching defaults to `https://www.clawhub.ai` is not currently proven best because current source, docs, live discovery, and observed production routing are non-www canonical, and the related host-switch PR was closed unmerged for that reason.
