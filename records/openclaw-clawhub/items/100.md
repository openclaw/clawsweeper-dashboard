## Summary

Keep open. ClawHub cleanup policy keeps issues open for maintainer judgment, and current production routing no longer matches the reported non-www to www redirect even though the CLI auth path remains redirect-sensitive and was not verified with a real production token.

## What This Changes

The issue reports that CLI token auth fails because requests to `https://clawhub.ai/api/v1/whoami` allegedly redirect to `www.clawhub.ai` and lose the bearer Authorization header.

## Best Possible Solution

Maintainers should verify fresh production browser-created and Settings-created tokens against the current CLI, then fix the proven failing layer or consolidate this under the broader Unauthorized issue cluster.

## Reproduction Assessment

No. The redirect-specific repro does not currently reproduce with a dummy bearer probe: non-www returns 401 without a Location header and www redirects back to non-www, but a real production token is still needed to verify the full auth flow.

## Solution Assessment

Unclear. Switching defaults to `https://www.clawhub.ai` is not currently the best proven fix because current production redirects www back to non-www; the safer path is maintainer real-token verification before choosing a source or hosting change.
