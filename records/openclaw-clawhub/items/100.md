## Summary

Keep open. Repository policy for openclaw/clawhub keeps issues open, and current main plus live dummy-token routing still do not show the originally reported apex-to-www redirect; the remaining Unauthorized cluster needs maintainer verification with a real production token.

## What This Changes

The issue reports CLI token authentication failing because authenticated requests to `https://clawhub.ai` allegedly redirect to `https://www.clawhub.ai` and lose the bearer header.

## Best Possible Solution

Maintainers should verify the current published CLI with a fresh production token on the apex host, then either consolidate this into the broader Unauthorized cluster or make one narrow canonical-domain/auth fix across CLI defaults, discovery, tests, docs, and hosting configuration.

## Reproduction Assessment

No. The specific apex-to-www redirect no longer reproduces with a dummy bearer probe, and proving the remaining auth failure requires a valid production token and account state that were not available in this read-only review.

## Solution Assessment

Unclear. Switching the CLI to `www.clawhub.ai` matched the original workaround, but current main and live routing treat the apex host as canonical, so maintainer production-token verification should come before a source or hosting change.
