## Summary

Keep open. Current main and live discovery still use the non-www canonical host, the original non-www to www redirect does not reproduce with a dummy bearer probe, and ClawHub policy keeps issues open unless maintainers verify the real production-token flow and canonical-domain direction.

## What This Changes

The issue reports that CLI token authentication fails when the default registry uses `https://clawhub.ai` because a redirect to `https://www.clawhub.ai` drops the bearer Authorization header.

## Best Possible Solution

Maintainers should verify the current published CLI with a real production token, then either close or consolidate with #99 if non-www works, or make one consistent canonical-domain fix across hosting, CLI defaults, discovery, docs, and tests.

## Reproduction Assessment

Unclear. The original non-www to www redirect did not reproduce with a dummy bearer request on May 4, 2026, but a real production token and current published CLI run are needed to verify the actual auth flow.

## Solution Assessment

Unclear. The closed #101 direction of switching defaults to `www.clawhub.ai` is not clearly the best fix now that source/docs/live discovery use non-www and live traffic redirects `www` back to non-www.
