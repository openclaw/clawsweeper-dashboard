## Summary

Keep open. Current main and live discovery still use the non-www host, while the originally reported non-www to www redirect no longer reproduces with a dummy bearer probe; because this is a ClawHub issue and the remaining check requires a real production token plus canonical-domain judgment, it should stay with maintainers.

## What This Changes

The issue reports that authenticated CLI requests to the default `https://clawhub.ai` registry lose `Authorization` during a non-www to www redirect and fail with `Unauthorized`.

## Best Possible Solution

The desired end state is a verified canonical-domain auth flow: either confirm the current non-www production path works with a real token and consolidate or close this alongside #99, or make one consistent domain fix across hosting, CLI defaults, discovery, docs, and tests.

## Reproduction Assessment

Unclear. A dummy bearer probe on May 3, 2026 returned 401 directly from `https://clawhub.ai/api/v1/whoami` with no `Location` header, but no valid production token was available to run the reported `clawhub login --token` and `clawhub whoami` flow end to end.

## Solution Assessment

Unclear. The closed #101 approach of switching defaults to `www.clawhub.ai` matched the original report, but it conflicts with current source, docs, live discovery, and the current live redirect direction.
