## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current evidence does not prove the real-token CLI auth flow is fixed: source still uses non-www as canonical, the original non-www to www redirect no longer reproduces, and related #99 still tracks broader Unauthorized login failures.

## What This Changes

The issue reports that CLI token authentication fails when `https://clawhub.ai` redirects to `https://www.clawhub.ai`, stripping the bearer authorization header.

## Best Possible Solution

Maintainers should verify current production with a real token, then either close or consolidate this with #99 if non-www works, or land one canonical-domain auth fix across hosting, CLI defaults, discovery, docs, and tests.

## Reproduction Assessment

No. The original non-www to www redirect does not reproduce with a dummy bearer request on 2026-05-03, but a real production token is required to verify the reported authenticated CLI failure.

## Solution Assessment

Unclear. The requested `www.clawhub.ai` default is not clearly the best fix now that live production redirects www back to non-www and current source/discovery advertise `https://clawhub.ai`.
