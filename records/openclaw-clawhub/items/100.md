## Summary

Keep open. Current main still uses non-www `https://clawhub.ai` for CLI defaults and discovery, the auth path remains sensitive to host redirects, and the live dummy-token probe now shows `www` redirecting back to non-www rather than the originally reported direction; maintainers need real-token production verification and a canonical-host decision, especially with related #99 still open.

## What This Changes

The issue asks for CLI authentication to work without bearer tokens being stripped by redirects between `clawhub.ai` and `www.clawhub.ai`.

## Best Possible Solution

Maintain one verified canonical-domain auth flow: if real-token production auth works on non-www, close or consolidate this with #99; if it fails, make a coordinated fix across CLI defaults, discovery, hosted redirects, docs, and focused auth tests.

## Reproduction Assessment

No high-confidence reproduction is available from this review. The original steps are clear, but a dummy-token host probe on 2026-05-02 no longer reproduces the reported non-www to `www` redirect, and a real production token is needed to verify the CLI failure.

## Solution Assessment

Unclear. The closed unmerged #101 direction of switching defaults to `www` is not currently the narrowest maintainable fix because current main and live traffic canonicalize non-www; maintainers should verify the real-token flow and domain policy first.
