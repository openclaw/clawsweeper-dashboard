## Summary

Keep open under the ClawHub issue policy. Current main still uses the non-www canonical host, the original non-www to www redirect no longer reproduces in a live dummy-token probe, and the remaining auth question needs maintainer verification with a real production token.

## What This Changes

The issue reports that CLI token authentication fails when the default registry redirects from `clawhub.ai` to `www.clawhub.ai` and drops the bearer token.

## Best Possible Solution

Maintainers should verify the current published CLI with a fresh real token against `https://clawhub.ai`; if it works, close or consolidate this with `https://github.com/openclaw/clawhub/issues/99`, and if it fails, land one narrow canonical-domain/auth fix across hosting, discovery, CLI defaults, tests, and docs.

## Reproduction Assessment

No. The originally reported non-www to www redirect no longer reproduces with a dummy bearer request, and this read-only review did not have a valid production token to verify the full CLI auth flow.

## Solution Assessment

No. The earlier `https://www.clawhub.ai` default change is no longer clearly the best fix because current production redirects www back to non-www; maintainers should reproduce with real tokens before changing code or hosting policy.
