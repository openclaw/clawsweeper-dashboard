## Summary

Keep open. Current main already contains the core href-only README link fix, but this draft PR now proposes broader markdown URL rewriting for links, images, and definitions, and that broader patch is not fully implemented on main and has a concrete image-serving correctness issue.

## What This Changes

The draft branch replaces the current href-only skill README URL rewrite with a remark plugin that rewrites markdown link, image, and definition URLs to the skill file API, adds focused tests, and includes a small lint-only maintenance cleanup.

## Best Possible Solution

Land a focused markdown URL rewrite that keeps skill-internal document links safe, and only rewrite images after ClawHub has a binary-safe asset response path or an explicit decision to support image files through this viewer.

## Reproduction Assessment

Yes. The current-main behavior is inspection-reproducible because `SkillDetailTabs` rewrites only `href`, and the PR's new image path is visible in the raw PR helper while the target file endpoint reads blobs as text.

## Solution Assessment

No, not as written. The link and definition rewrite direction is reasonable, but routing markdown image nodes to the raw text file API is not the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid routing README images to the text file endpoint:** `src/lib/skillReadmeLinks.ts:21`
  - body: The remark plugin now includes `image` nodes, so `![Logo](img/logo.png)` becomes an `<img src="/api/v1/skills/.../file?...">`. That endpoint is documented and implemented as raw text (`blob.text()` plus `safeTextFileResponse`), so packaged PNG/JPEG assets will not be served as binary images. Either leave images out of this rewrite or add a binary-safe asset endpoint before pointing image `src` values at it.
  - confidence: 0.86
