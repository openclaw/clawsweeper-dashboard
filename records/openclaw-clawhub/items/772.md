## Summary

Keep open. This is an open ClawHub issue, and current main still has a plausible disabled-button path because `/cli/auth` disables the GitHub sign-in button directly from Convex auth loading; a read-only production WebSocket probe no longer reproduced HTTP 307, but it did not verify the full browser OAuth and loopback token flow.

## What This Changes

The issue reports that `clawhub login` opens `/cli/auth` with the GitHub sign-in button disabled because the Convex WebSocket connection returns HTTP 307.

## Best Possible Solution

Production CLI login should complete end to end; if Convex auth/connectivity cannot resolve, `/cli/auth` should show a recoverable error or headless-token fallback instead of leaving sign-in permanently disabled.

## Reproduction Assessment

Unclear. The reporter provided a concrete `npx clawhub login` path and current source preserves the disabled-button mechanism, but the read-only WebSocket probe returned 101 rather than 307 and did not exercise the full browser OAuth plus loopback callback.

## Solution Assessment

Unclear. The maintainable path is production verification first, then either a Convex/Vercel/OAuth configuration fix or a narrow `/cli/auth` fallback with regression coverage for the unauthenticated loading/error branch.
