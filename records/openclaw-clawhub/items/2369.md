## Summary

Keep open for maintainer security/product routing: the report is reproducible against the third-party `@clawnify/clawflow` artifact, while current ClawHub/OpenClaw behavior treats dynamic-code install blocks as a safety boundary rather than an automatic fix target.

## What This Changes

The issue reports that installing the third-party `@clawnify/clawflow` plugin is blocked because OpenClaw detects dynamic code execution in the published artifact.

## Best Possible Solution

Keep the scanner block as the default, ask the plugin owner to remove or sandbox dynamic evaluation, and require explicit maintainer security approval before any exception or ingestion change.

## Reproduction Assessment

Yes for source/artifact reproduction, but not from a live installer run in this review. The npm 1.2.2 artifact contains `new Function` at the reported runner line, and current main's scanner flags that pattern as dynamic code execution.

## Solution Assessment

Unclear as an in-repo fix: changing the scanner or install gate is not the narrow maintainable first move. The safer path is plugin-owner remediation or a maintainer-approved sandbox/exception policy for this specific plugin design.
