## Summary

Keep open for maintainer security/product routing: the reported block is for a third-party plugin artifact with dynamic evaluation, and current ClawHub/OpenClaw safety surfaces intentionally treat scan and trust signals as install gates rather than something ClawSweeper should relax automatically.

## What This Changes

The issue reports that installing the third-party `@clawnify/clawflow` plugin is blocked because OpenClaw detects dynamic code execution in the package artifact.

## Best Possible Solution

Keep the scanner default intact, have the plugin author remove or sandbox dynamic evaluation and publish a refreshed package, and make any exception or policy relaxation an explicit maintainer security decision.

## Reproduction Assessment

No high-confidence ClawHub bug reproduction was run against the external installer, but the source path is reproducible from current code: the scanner has a dynamic-execution rule and emits the exact reported message. The reporter provided a concrete command and installed-version context, while live package state suggests the failing artifact may be outside the current ClawHub release record.

## Solution Assessment

Unclear as an in-repo fix: changing the scanner or install gate is not the narrow maintainable first move. The safer solution is plugin-author remediation or a maintainer-approved sandbox/exception policy for this specific dynamic-expression design.
