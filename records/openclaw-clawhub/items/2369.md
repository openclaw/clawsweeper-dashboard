Keep open for maintainer security/product routing: the reported block is tied to dynamic evaluation in a third-party plugin artifact, and current ClawHub safety surfaces intentionally treat scan/trust signals as install gates rather than something ClawSweeper should relax automatically.

## What This Changes

The issue reports OpenClaw blocking installation of `@clawnify/clawflow` because dynamic JavaScript execution was detected in the plugin artifact.

## Best Possible Solution

Keep the scanner and install gate intact, have the plugin author remove or sandbox the dynamic evaluation and publish a refreshed package, and make any exception an explicit maintainer security decision.

## Reproduction Assessment

Yes for the source-level cause: the npm artifact contains `new Function` at the reported line and current main has a scanner rule that flags that pattern. I did not run a full OpenClaw installer repro against the user’s exact environment.

## Solution Assessment

Unclear as an in-repo fix: changing the scanner or install gate is not the narrow maintainable first move. The safer path is plugin-author remediation or a maintainer-approved sandbox/exception policy for this specific dynamic-expression design.
