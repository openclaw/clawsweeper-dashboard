## Summary

Keep open: current main does not yet include this recovery path, and the PR is a viable merge candidate with no blocking code-review findings. The remaining maintainer concerns are the intentional compatibility and security-boundary risks of transferring an established personal publisher after staff identity verification.

## What This Changes

The PR adds an admin-only personal publisher recovery HTTP endpoint and CLI command, with immutable GitHub ID checks, dry-run/apply behavior, audit logging, docs/spec updates, and regression coverage.

## Best Possible Solution

Land the guarded recovery path with the documented staff verification procedure, using dry-run output before apply and a separate resumable migration for publishers that exceed the transaction bound.

## Reproduction Assessment

Yes. Current main keys GitHub OAuth users by immutable providerAccountId and authorizes personal publishers by linkedUserId, so a replacement GitHub principal cannot publish under the legacy linked personal publisher without a recovery operation.

## Solution Assessment

Yes. An admin-only recovery keyed by both immutable GitHub provider IDs, with dry-run/apply separation, explicit staff verification, fail-closed conflict checks, owner migration, and audit logging, is narrower and safer than rewriting Convex Auth rows or relinking by username.
