## Summary

The guarded recovery design and real Convex proof are strong, but the reviewed head does not transfer the active reserved-handle authority, so a later normal identity synchronization can still disagree with the recovered user’s protected handle.

## What This Changes

Adds an admin-only, dry-run-first workflow to transfer a personal publisher and its legacy resource ownership to a verified replacement GitHub principal.

## Best Possible Solution

Atomically reassign the recovered handle’s active reservation to the verified replacement user, or fail closed on an unexpected reservation, then prove recovery followed by normal GitHub profile synchronization against a real Convex backend.

## Reproduction Assessment

Yes, from source inspection: create an active reservation for the legacy handle owned by the previous user, apply recovery, then run the normal GitHub profile synchronization path for the replacement user and inspect the conflicting reservation authority. The submitted live proof establishes the surrounding recovery path but does not execute this final synchronization step.

## Solution Assessment

No on the reviewed head. An admin-only, provider-ID-guarded, dry-run-first recovery is the right narrow design, but the transaction must include every persisted authority for the protected handle and validate ordinary behavior after recovery.
