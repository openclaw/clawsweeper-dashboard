## Summary

Keep this PR open for a focused repair: the primary publisher/resource migration and exact-head live proof are strong, but the recovery transaction can leave the recovered handle reserved to the previous principal, allowing normal user synchronization to move the replacement account away from the recovered identity.

## What This Changes

The PR adds an admin-only API and CLI workflow to dry-run or apply a verified personal-publisher transfer between GitHub OAuth principals, including bounded legacy owner migration, audit logging, documentation, and regression coverage.

## Best Possible Solution

Atomically reconcile the recovered handle's active reservation: transfer it when its rightful owner is the verified previous principal, fail closed when it belongs to a third user, and prove that a subsequent normal user ensure preserves the recovered handle and publisher linkage.

## Reproduction Assessment

Yes. Current main clearly lacks recovery, the branch behavior has exact-head live Convex proof, and the remaining defect is source-reproducible by creating an active reservation for the recovered handle owned by the previous user and then running normal user ensure after recovery.

## Solution Assessment

No, not yet. The bounded publisher and resource transfer is a maintainable approach, but it must reconcile the repository's existing reserved-handle source of truth in the same transaction to avoid competing identity state.
