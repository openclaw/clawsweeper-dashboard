## Label Justifications

- `P1`: An incomplete identity recovery can leave an inaccessible or prior OAuth principal authoritative in ownerUserId-based workflows after staff believe control was transferred.
- `merge-risk: 🚨 compatibility`: The PR changes persistent user, publisher, handle, and membership state without migrating legacy resource ownership fields still consumed on current main.
- `merge-risk: 🚨 security-boundary`: The new admin operation transfers control of published resources between immutable OAuth principals and must not leave split authorization.

## Summary

This is a useful, carefully guarded recovery design, but it is not merge-ready: the mutation transfers the publisher link without migrating legacy resource ownership fields, so several current-main paths still treat the inaccessible account as owner, and the external PR provides no real Convex behavior proof.

## What This Changes

The PR adds an admin-only API and CLI workflow to dry-run or apply a verified transfer of a personal publisher between immutable GitHub OAuth principals, with audit logging and destination-resource safeguards.

## Best Possible Solution

Implement an auditable, resumable transfer that makes the replacement user authoritative across both publisher-based authorization and every still-supported legacy ownerUserId surface, then prove dry-run, apply, rejection, and post-transfer publishing in a real Convex deployment.

## Reproduction Assessment

Yes at source level: the linked issue supplies the two-account scenario, and current main has no supported way for a replacement GitHub principal to become the linked owner of the legacy personal publisher. This review did not execute the production-like recovery path.

## Solution Assessment

No, not yet: immutable-provider verification, dry-run, audit logging, and fail-closed destination checks are sound directions, but transferring only publisher links creates split ownership while current code still relies directly on resource ownerUserId.
