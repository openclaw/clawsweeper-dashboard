## Summary

Keep open: current main still lacks the recovery endpoint, and this PR remains the active implementation candidate. I found no blocking line-level patch defect; the remaining merge decision is the security-sensitive operator procedure and bounded migration behavior.

## What This Changes

Adds an admin-only personal publisher recovery HTTP endpoint, admin CLI command, API schema/docs/spec updates, and tests for transferring a verified legacy personal publisher between immutable GitHub OAuth principals.

## Best Possible Solution

Land the guarded recovery path with the documented staff verification procedure, dry-run before apply, audit logging, and a separate resumable migration path for publishers that exceed the per-table bound.

## Reproduction Assessment

Yes. Source inspection shows current main has no publisher-recovery route and personal publisher access is linkedUserId-based, so a replacement GitHub principal remains unable to publish under the legacy personal publisher.

## Solution Assessment

Yes. An admin-only transfer keyed by both immutable GitHub provider IDs, with dry-run, explicit staff verification, conflict rejection, bounded migration, and audit logging, is the narrowest maintainable path compared with rewriting Convex Auth rows or relinking by mutable usernames.
