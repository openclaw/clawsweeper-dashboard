## Summary

Current main still supports only one-skill-at-a-time installation and has no publisher-wide installation path. The request is product-aligned, but skills, code plugins, and bundle plugins have distinct installation and trust contracts; because this collaborator-authored issue also carries an explicit product-decision signal, it requires maintainer direction rather than automatic implementation or closure.

## What This Changes

Add a publisher-scoped command or workflow for installing all skills and/or plugins owned by that publisher.

## Best Possible Solution

Provide a preview-first publisher workflow with explicit package-family filters, exact version and safety metadata, and confirmation, then delegate each selected item to its established family-specific installer while reporting successes, skips, and failures individually.

## Reproduction Assessment

Not applicable; this requests a new publisher-wide installation capability rather than reporting broken documented behavior. Source inspection establishes with high confidence that current main accepts one skill slug per installation.

## Solution Assessment

No, not as an unqualified mixed-package install-all shortcut. The safer maintainable direction is a previewable, family-aware flow that preserves each package type's existing trust, compatibility, and lifecycle checks.

## Risks / Open Questions

- A combined “all skills/plugins” operation could obscure package-specific moderation, execution, compatibility, integrity, capability, and permission decisions.
- Publisher catalogs are paginated and mutable, so the workflow needs explicit preview or snapshot semantics if packages change while the operation runs.
- Partial success across many packages needs a defined exit status, retry policy, and per-package result report.
- Plugin artifacts do not yet share the skill installer's destination, lockfile, update, and uninstall lifecycle, so treating them identically would create an unsupported installation contract.
