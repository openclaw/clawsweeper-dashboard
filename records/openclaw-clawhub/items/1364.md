Keep open: current main still has the source-confirmed double JSON serialization path for `clawhub transfer request`, and ClawHub cleanup policy keeps issues open rather than auto-closing them. ## What This Changes

The issue reports that `clawhub transfer request <slug> <handle>` sends a double-serialized body, causing the API to reject the request as missing `toUserHandle`.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from transfer, rename, and merge commands, updates tests to assert object payloads, and keeps `apiRequest` as the JSON serialization boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows the command passes a JSON string, both HTTP transports stringify again, and the server requires `payload.toUserHandle` on an object; I did not run a live authenticated transfer.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix; server-side tolerance for double-encoded strings would mask the client boundary bug, though defensive HTTP coverage can be added if desired.
