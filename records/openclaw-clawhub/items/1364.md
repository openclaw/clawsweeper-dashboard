## What This Changes

## Best Possible Solution

Land one focused CLI repair for the double-JSON bug across transfer, rename, and merge, with tests asserting object payloads and `apiRequest` remaining the serialization boundary.

## Reproduction Assessment

Yes, by source inspection: current main passes a JSON string from the transfer command into HTTP helpers that stringify bodies again, while the server expects an object with `toUserHandle`. I did not run a live authenticated transfer in this read-only review.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix; server-side tolerance for double-encoded strings would mask the client boundary bug.

## Repair Work Prompt

Fix the ClawHub CLI JSON body boundary for skill ownership operations. Current main has `cmdTransferRequest` passing `body: JSON.stringify({ toUserHandle, message })` to `apiRequest`, while `apiRequest` and the Bun curl fallback already serialize `args.body`; `cmdRenameSkill` and `cmdMergeSkill` in `ownership.ts` use the same pre-stringified pattern. Change these command handlers to pass plain objects, update transfer and ownership command tests to assert object payloads, and keep `apiRequest` as the single JSON serialization boundary. Coordinate with the overlapping reports at https://github.com/openclaw/clawhub/issues/1300 and https://github.com/openclaw/clawhub/issues/1364 plus prior PR https://github.com/openclaw/clawhub/pull/1251, but do not fold in the broader org/package ownership-transfer feature from https://github.com/openclaw/clawhub/pull/1635. If adding defensive HTTP string passthrough, cover both Node fetch and Bun/curl paths.
