## Summary

Keep open: ClawHub cleanup policy keeps issues open, and current main still source-reproduces the reported CLI serialization bug. The transfer command sends a JSON string into `apiRequest`, both HTTP transports serialize request bodies again, and the API handler expects an object field named `toUserHandle`.

## What This Changes

The issue reports that `clawhub transfer request <slug> <handle>` fails because the positional handle is not delivered to the API as `toUserHandle`.

## Best Possible Solution

Land one focused CLI serialization repair for transfer, rename, and merge, with command tests asserting object payloads and `apiRequest` remaining the JSON serialization boundary.

## Reproduction Assessment

Yes, by source inspection on current main: the transfer command passes a JSON string into HTTP helpers that serialize bodies again, while the server reads `parsed.payload.toUserHandle` from an object. I did not run a live authenticated transfer in this read-only review.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix; server-side tolerance for double-encoded strings would hide the client serialization-boundary bug.

Fix the CLI double-JSON request-body bug for ownership operations. On current main, `clawhub transfer request <slug> <handle>` passes `body: JSON.stringify(...)` to `apiRequest`, and both fetch and Bun/curl HTTP transports serialize `args.body` again, causing the API to see a JSON string instead of an object with `toUserHandle`. Update transfer, skill rename, and skill merge command handlers to pass plain object bodies. Keep `apiRequest` as the serialization boundary; do not broaden this repair into org/package ownership-transfer product work. Update focused tests so command tests assert object bodies and HTTP tests cover the serialization behavior. Preserve existing transfer semantics except for the body-shape fix.
