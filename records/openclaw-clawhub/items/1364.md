## What This Changes

The issue reports that `clawhub transfer request <slug> <handle>` sends a malformed API payload instead of mapping the positional handle to `toUserHandle`.

## Best Possible Solution

Land one focused CLI serialization repair for transfer, rename, and merge, with tests asserting object payloads and `apiRequest` remaining the JSON serialization boundary.

## Reproduction Assessment

Yes, by source inspection: current main passes a JSON string from the transfer command into HTTP helpers that stringify bodies again, while the server expects an object with `toUserHandle`. I did not run a live authenticated transfer in this read-only review.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix; optional HTTP string-body tolerance is acceptable only with coverage for both Node fetch and Bun/curl serialization paths.

## Repair Work Prompt

Fix the CLI double-JSON serialization bug tracked by this issue and https://github.com/openclaw/clawhub/issues/1300. On current main, `cmdTransferRequest`, `cmdRenameSkill`, and `cmdMergeSkill` pass `JSON.stringify(...)` into `apiRequest`, while `apiRequest` serializes JSON bodies for both Node fetch and Bun/curl paths. Change the affected command handlers to pass plain object bodies, update command tests to assert object payloads, and keep the shared HTTP helper as the serialization boundary. If you add defensive string-body handling in `packages/clawhub/src/http.ts`, cover both runtime paths. Do not add org/package transfer behavior, management approval, or backend authorization changes in this repair.
