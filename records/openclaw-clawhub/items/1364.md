## Summary

Keep open: current main still contains the source-confirmed double-serialization path, and the consolidated ownership-transfer PR that carries an object-body repair remains open and conflicting rather than landed.

## What This Changes

The issue requests that `clawhub transfer request <slug> <handle>` send the positional handle as `toUserHandle` instead of losing it through a double-serialized JSON body.

## Best Possible Solution

Land or extract the object-body CLI fix from the ownership-transfer work, update command tests to assert plain object payloads, and keep `apiRequest` as the single JSON serialization layer.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: the command sends a JSON string, both HTTP transports stringify it again, and the server requires `payload.toUserHandle` on an object.

## Solution Assessment

Yes. Passing a plain object from the CLI is the narrow maintainable fix; reparsing double-encoded request strings server-side would mask the client boundary bug.
