## Summary

Keep open: current main still source-reproduces the CLI double-serialization bug, and ClawHub policy keeps issues open for maintainer follow-up rather than auto-closing them.

## What This Changes

The transfer request CLI should send the positional recipient handle as the `toUserHandle` API payload field.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from transfer, rename, and merge command handlers while keeping `apiRequest` as the single JSON serialization layer.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: the CLI passes a JSON string, both HTTP transports stringify it again, and the API then sees no `toUserHandle` property.

## Solution Assessment

Yes. Passing a plain object from the command layer is the narrow maintainable repair; teaching the server to accept double-encoded request strings would preserve a broken client contract.

## Repair Work Prompt

Fix the CLI JSON body boundary for ownership-transfer commands on current main. `cmdTransferRequest` must pass a plain object body with `toUserHandle` and optional `message` to `apiRequest`; apply the same object-body pattern to `cmdRenameSkill` and `cmdMergeSkill` for `newSlug` and `targetSlug` so the shared HTTP helper remains the only JSON serialization layer. Update command tests to assert object bodies, and add or adjust HTTP/client tests only if needed to preserve the single-serialization contract. Keep the repair focused; do not implement the broader ownership-transfer workflow from #1635, change API semantics, or make the server accept double-encoded JSON strings. Add a changelog entry if repo policy treats this CLI fix as user-facing.
