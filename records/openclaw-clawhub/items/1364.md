## Summary

Keep open: current main still has the source-confirmed double-serialization path for `clawhub transfer request`, and the only overlapping implementation is the broader conflicting ownership-transfer PR rather than a merged fix.

## What This Changes

The issue reports that `clawhub transfer request <slug> <handle>` fails because the positional handle is not delivered as the `toUserHandle` API payload field.

## Best Possible Solution

Keep a single JSON serialization boundary: CLI commands pass plain object bodies, `apiRequest` serializes them for both transports, and regression tests assert object payloads for transfer, rename, and merge.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: the command sends a JSON string, both HTTP transports stringify it again, and the server requires `payload.toUserHandle` on an object.

## Solution Assessment

Yes. Passing plain objects from the affected CLI commands and updating tests is the narrow maintainable fix; parsing double-encoded strings server-side would mask the client boundary bug.

## Repair Work Prompt

Fix the CLI double JSON serialization bug on current main. `cmdTransferRequest` in `packages/clawhub/src/cli/commands/transfer.ts` and the related `cmdRenameSkill` / `cmdMergeSkill` paths in `packages/clawhub/src/cli/commands/ownership.ts` pass `JSON.stringify(...)` bodies to `apiRequest`, while `packages/clawhub/src/http.ts` serializes `args.body` in both the Node fetch path and Bun/curl path. Change the CLI command call sites to pass plain objects, update `transfer.test.ts` and `ownership.test.ts` to assert object bodies, and keep the HTTP helper as the single serialization boundary. Only harden `apiRequest` defensively if both `http.test.ts` and `http.bun.test.ts` cover the behavior. Do not expand into org ownership transfer, management approval, package transfer, or other product scope from https://github.com/openclaw/clawhub/pull/1635.
