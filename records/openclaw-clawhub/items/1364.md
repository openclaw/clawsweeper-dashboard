## Summary

Keep open. Current main still contains the reported CLI serialization bug, and ClawHub policy keeps issues open for maintainer follow-up rather than auto-closing them.

## What This Changes

The issue reports that `clawhub transfer request <slug> <handle>` fails because the CLI sends a pre-serialized request body instead of mapping the positional handle to the API's `toUserHandle` field.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from transfer, rename, and merge command handlers to `apiRequest`, updates command tests to assert object bodies, and keeps the HTTP helper as the single JSON serialization layer.

## Reproduction Assessment

Yes: current source provides a high-confidence reproduction path. `cmdTransferRequest` sends a string body, both HTTP transports stringify it again, and the server then looks for `toUserHandle` on an object payload.

## Solution Assessment

Yes: the reporter's suggested command-side object-body fix is the narrowest maintainable direction. Teaching the server to accept double-encoded JSON would mask the client boundary bug instead of fixing it.

Fix the CLI request body serialization bug reported for `clawhub transfer request <slug> <handle>`. In `packages/clawhub/src/cli/commands/transfer.ts`, pass a plain object body `{ toUserHandle, message }` to `apiRequest` instead of a pre-serialized JSON string. Audit and fix the same command-side pattern in `packages/clawhub/src/cli/commands/ownership.ts` for rename and merge. Update `packages/clawhub/src/cli/commands/transfer.test.ts` and `packages/clawhub/src/cli/commands/ownership.test.ts` to assert object bodies, not serialized string contents. Keep `packages/clawhub/src/http.ts` as the only JSON serialization boundary; do not change the server to accept double-encoded JSON strings. Add a `CHANGELOG.md` Unreleased Fixes entry for the user-facing CLI fix, crediting the reporter if following existing project convention.
