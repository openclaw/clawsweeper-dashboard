## What This Changes

The issue reports that `clawhub transfer request <slug> <handle>` fails because the CLI sends a double-encoded JSON body instead of an object with `toUserHandle`.

## Best Possible Solution

Land a focused CLI serialization fix that sends plain object bodies from the affected command handlers, updates tests to assert object payloads, and keeps org/package ownership-transfer product work in the separate consolidated thread.

## Reproduction Assessment

Yes, by source inspection rather than a live authenticated transfer: current main passes a string body from `cmdTransferRequest`, both HTTP runtime paths serialize it again, and the server requires `payload.toUserHandle` on an object.

## Solution Assessment

Yes. Passing plain objects from the affected CLI commands and updating regression tests is the narrow maintainable fix; reparsing double-encoded strings server-side would hide the client boundary bug.

## Repair Work Prompt

Fix the current-main double JSON serialization bug for ClawHub CLI JSON POST bodies. In `packages/clawhub/src/cli/commands/transfer.ts`, pass a plain object for `cmdTransferRequest` instead of `JSON.stringify(...)`. Also fix the same pre-serialized request-body pattern in `packages/clawhub/src/cli/commands/ownership.ts` for `skill rename` and `skill merge`, so this repair covers the canonical broader report. Update `transfer.test.ts` and `ownership.test.ts` to assert object bodies, not serialized substrings. Keep `apiRequest` as the JSON serialization boundary; do not change the server to accept double-encoded request strings, and do not implement org/package ownership-transfer product changes from the consolidated PR. If you harden `http.ts` for string passthrough, include coverage for both Node fetch and Bun/curl paths; otherwise keep the fix at the command call sites. No user-facing docs change is expected for this bug fix.
