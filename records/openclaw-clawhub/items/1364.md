## Summary

Keep open. Current main still contains the reported double JSON serialization path, and ClawHub policy keeps issues open even when related reports exist; the bug is narrow enough for a focused repair PR.

## What This Changes

The issue reports that `clawhub transfer request <slug> <handle>` sends the positional handle in the wrong payload shape, causing the API to reject the request as missing `toUserHandle`.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from transfer, rename, and merge command handlers, updates the command tests, and records the user-facing CLI fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction: the command sends a JSON string, the HTTP layer serializes it again, and the server validates object fields that are therefore missing.

## Solution Assessment

Yes. The narrow maintainable solution is to keep `apiRequest` as the single JSON serialization boundary and change the affected command handlers/tests to use object bodies.

## Repair Work Prompt

Fix the double JSON serialization bug for ClawHub CLI ownership operations. In `packages/clawhub/src/cli/commands/transfer.ts`, pass a plain object body to `apiRequest` for transfer requests instead of `JSON.stringify(...)`. In `packages/clawhub/src/cli/commands/ownership.ts`, do the same for `skill rename` and `skill merge`. Update `packages/clawhub/src/cli/commands/transfer.test.ts` and `packages/clawhub/src/cli/commands/ownership.test.ts` to assert object bodies rather than serialized JSON strings. Keep `packages/clawhub/src/http.ts` as the only JSON serialization layer; do not teach the server to accept double-encoded JSON strings. Add a concise `CHANGELOG.md` entry for the user-facing CLI fix. Coordinate with #1300 and this issue; do not roll in the broad package/org ownership-transfer behavior from #1635.
