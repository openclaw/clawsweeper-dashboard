## Summary

Keep open: this repo’s cleanup policy keeps issues open, and current main still contains the reported CLI/API request-body mismatch for transfer requests.

## What This Changes

The issue reports that `clawhub transfer request <slug> <handle>` fails because the positional handle is not delivered to the API as `toUserHandle`.

## Best Possible Solution

Ship a focused CLI serialization-boundary fix with regression tests, leaving broader ownership-transfer product changes to the existing transfer-work branch.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: the command passes a JSON string, both HTTP transports serialize it again, and the API then sees no `toUserHandle` property.

## Solution Assessment

Yes. Passing plain object bodies from command handlers while keeping `apiRequest` as the single JSON serialization layer is the narrow maintainable fix; server-side reparsing of double-encoded strings would preserve an invalid client contract.

## Repair Work Prompt

Fix the reported `clawhub transfer request <slug> <handle>` serialization bug. Keep `apiRequest` as the single JSON serialization boundary and change the transfer command to pass a plain object body with `toUserHandle` and optional `message`; do not teach the server to accept double-encoded JSON strings. Audit the same command-side pre-serialization pattern in `packages/clawhub/src/cli/commands/ownership.ts` for rename/merge and include it only if it is the same boundary fix. Update focused CLI tests to assert object bodies, add or adjust HTTP coverage if needed, and add a `CHANGELOG.md` entry if the repo policy treats this as a user-facing CLI fix. Coordinate with #1635 and do not implement package/org approval-flow changes in this repair.
