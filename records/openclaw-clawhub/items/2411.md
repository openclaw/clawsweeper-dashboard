## Summary

This remains useful work because current main has no consolidated plugin verification endpoint, but the draft is not safe to merge: version-specific responses can inherit package-level verification from another release, the exported schema does not validate most of the security payload, and the only behavior evidence is mocked.

## What This Changes

The draft adds `GET /api/v1/plugins/{name}/verify` with version/tag/latest resolution, aggregated plugin trust evidence, schema exports, HTTP documentation, a trust-boundary spec, and mocked handler coverage.

## Best Possible Solution

Reimplement the narrow endpoint on current main using release-only provenance for version-scoped claims, fully typed nested response schemas, explicit non-endorsement semantics, and live Convex-backed proof for latest, tag, historical version, blocked, pending, stale, and clean releases.

## Reproduction Assessment

Yes for the review defect: request a historical release whose `release.verification` is absent while `pkg.verification` describes the current release; source inspection shows the historical response inherits the package summary. Real Convex execution has not been demonstrated.

## Solution Assessment

No, not as submitted. A consolidated evidence endpoint is reasonable, but version-scoped evidence must never fall back to another release’s verification, and a versioned public security schema should validate its complete contract.
