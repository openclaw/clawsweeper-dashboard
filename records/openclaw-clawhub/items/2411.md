## Summary

Keep open: current main does not implement this endpoint, but this PR is not merge-ready because blocked public plugins can be hidden before the verify response can report a failed trust decision, and the new public trust semantics still need maintainer/security review.

## What This Changes

The PR adds a public plugin verification JSON endpoint, package schema export/dist artifacts, HTTP docs, tests, and an internal trust-semantics spec.

## Best Possible Solution

Keep the endpoint version-scoped and evidence-only, but make its lookup match the existing public trust path so private packages stay hidden while blocked public releases return a failed JSON envelope with stable reasons.

## Reproduction Assessment

Yes, from source inspection: the candidate verify route calls `getByNameForViewerInternal`, while current main's `getReadablePackageByName` can return null for blocked public packages. The existing security endpoint uses a separate trust-readable lookup for this reason.

## Solution Assessment

No. The endpoint shape is plausible, but it should reuse or extend the exact-release public trust semantics and add blocked-package coverage before becoming a stable API.
