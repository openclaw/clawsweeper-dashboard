## Summary

Keep open: current main does not have this endpoint, and the draft PR is not merge-ready because the verify route can hide blocked public plugins before returning a failed evidence envelope and still lacks real local HTTP/Convex proof.

## What This Changes

The PR adds a public `GET /api/v1/plugins/{name}/verify` JSON evidence endpoint, schema export/dist updates, HTTP docs, a trust-semantics spec, and handler tests.

## Best Possible Solution

Keep the endpoint version-scoped and evidence-only, but implement it on the trust-readable package/release path, preserve private-package hiding, add blocked-plugin coverage, and require real local HTTP proof before stabilizing the API.

## Reproduction Assessment

Yes, from source inspection: the candidate verify route calls `getByNameForViewerInternal`, while current main's normal readable-package path can return null for blocked public packages. The existing security endpoint uses a separate trust-readable lookup for this reason.

## Solution Assessment

No. The endpoint shape is plausible, but it should reuse or extend the exact-release public trust semantics and add blocked-package coverage before becoming a stable API.
