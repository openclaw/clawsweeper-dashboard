## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep verify metadata release-scoped:** `convex/httpApiV1/packagesV1.ts:653`
  - body: This endpoint is advertised as version-scoped, but `?version=` requests can copy `params.pkg.verification`, `params.pkg.compatibility`, and `params.pkg.capabilities` from the package summary when the selected release lacks those fields. That summary represents the latest package state, so an older release can incorrectly claim a newer source commit, provenance tier, hooks, or API range that clients may use for trust decisions. Use only the selected release's evidence here, or return `null` when it is unavailable, and add a regression test where latest and requested versions differ.
  - confidence: 0.91
- **[P2] Use the public trust lookup for blocked plugins:** `convex/httpApiV1/packagesV1.ts:3336-3338`
  - body: The verify route calls `getByNameForViewerInternal`, which uses normal readability rules and can return `null` for a public viewer when a package is blocked. That makes `/verify` return 404 instead of the moderation/download-blocking evidence this endpoint is meant to expose; the existing package security endpoint uses `getPackageReadableForPublicTrust` and annotates `publicDownloadBlocked`. Mirror that path so blocked releases produce machine-readable trust evidence rather than disappearing.
  - confidence: 0.86
