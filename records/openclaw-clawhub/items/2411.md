## Label Justifications

- `merge-risk: 🚨 compatibility`: The versioned response becomes a client contract, while current fallbacks can give selected versions inconsistent meanings.

## Summary

Current main does not provide the proposed plugin verification endpoint, so this PR remains useful, but it is not merge-ready: selected-version responses can inherit package-level metadata, blocked packages can be filtered out before their fail evidence is returned, the branch conflicts with main, and the contributor proof is mock-only.

## What This Changes

This PR adds a public version-scoped plugin verification evidence endpoint with schema exports, handler tests, HTTP documentation, and a trust-semantics specification.

## Best Possible Solution

Land a refreshed endpoint only after every artifact-dependent field is selected-release-scoped, blocked plugins return a minimal safe failure envelope, the public schema has precise nested types, and real local Convex requests demonstrate both passing and blocked releases.

## Reproduction Assessment

Yes for the patch defects: source inspection gives a high-confidence path where a version request uses package-level fallbacks and where a blocked package is filtered before evidence construction. The proposed endpoint itself is a new feature, not a reported current-main bug.

## Solution Assessment

No; consolidating evidence is reasonable, but this implementation is not the safest solution until release scoping, blocked-state lookup, schema precision, and the permanent public trust semantics are resolved.
