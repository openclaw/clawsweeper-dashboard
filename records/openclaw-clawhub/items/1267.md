## Summary

Keep open. ClawHub cleanup policy keeps open issues in maintainer hands, and current main still intentionally rejects deleted/deactivated user sign-in, documents deletion as irreversible, and lacks a self-service or support-admin account recovery/relink path; the related login-loop cluster remains open.

## What This Changes

The issue requests a way to recover, purge, or cleanly recreate a GitHub-linked ClawHub account after account deletion causes OAuth login to return to a logged-out state.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, auth/user records, support/admin tooling, user-facing errors, documentation, and tests.

## Reproduction Assessment

Yes for the source-level behavior: delete a ClawHub account, then reauthenticate with the same GitHub provider identity and the auth callback rejects the deactivated user while app helpers treat the user as inactive. The exact reporter account records were not inspected.

## Solution Assessment

No, a one-off manual purge is not the best long-term solution because it leaves identity, deletion, and abuse semantics undefined. The safer path is an explicit recovery, purge/recreate, unlink/relink, or terminal-state policy with audited implementation.
