## Summary

Keep open. Current main still intentionally treats user-initiated account deletion as permanent and blocks later OAuth sign-in for the deleted identity; UI error handling has improved, but there is still no recovery, relink, or support-admin purge workflow, and the related login-loop cluster remains open.

## What This Changes

The issue requests recovery or server-side cleanup for a user locked out of ClawHub login after deleting one GitHub-linked account and trying to sign in with another GitHub account.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked users, then implement it consistently across OAuth callbacks, user/auth account records, support/admin tooling, user-facing errors, docs, and tests.

## Reproduction Assessment

No for the exact reporter account state; the production records and cross-account zocase/wakeband path were not inspected. Yes for the clustered source-level path where a deleted GitHub-linked user reauthenticates and current main rejects the identity as permanently deleted.

## Solution Assessment

No, a one-off manual purge is not the best maintainable fix. The safer solution is an explicit maintainer-approved lifecycle: restore, purge/recreate, unlink/relink, or a terminal state with a clear support route.
