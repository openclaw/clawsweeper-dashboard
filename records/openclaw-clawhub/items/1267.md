## Summary

Keep this open. Current main still treats user-initiated deletion as terminal and rejects later GitHub auth for deleted/deactivated users; the clustered reports show this needs a maintainer product and security decision, not an auto-close or narrow repair lane.

## What This Changes

The issue asks ClawHub to unblock GitHub OAuth after account deletion by purging, relinking, or otherwise recovering stale account state so the reporter can sign in again.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs/specs, and tests.

## Reproduction Assessment

Yes, from source inspection: deleteAccount sets deactivatedAt/purgedAt, the GitHub auth callback rejects deactivated or legacy deleted users, and active-user helpers refuse to load those records. I did not verify the named production records or run a live OAuth flow.

## Solution Assessment

Unclear until maintainers choose the deletion policy. A one-off purge may unblock this reporter, but the maintainable solution is an explicit restore, purge/recreate, unlink/relink, or terminal support route with security review.
