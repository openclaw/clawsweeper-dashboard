## Summary

Keep open because current main still intentionally blocks OAuth sign-in for deleted/deactivated users, while the requested purge, recovery, or relink path needs an account-lifecycle and security policy decision rather than cleanup closure.

## What This Changes

The issue asks ClawHub to resolve a post-deletion GitHub OAuth login loop by purging, recreating, restoring, or relinking affected account records.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs/specs, and regression tests.

## Reproduction Assessment

No for the exact named production accounts because live data was not inspected. Yes at the source level for the broader deleted-account OAuth lockout: current auth, deletion, access, docs, and tests intentionally reject deleted or deactivated users after GitHub OAuth.

## Solution Assessment

No for a one-off purge as the repository-level solution. The maintainable path is a maintainer-approved recovery, purge/recreate, unlink/relink, or explicitly terminal support flow with provider-account verification, privacy rules, audit logging, and abuse safeguards.
