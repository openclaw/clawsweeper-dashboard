## Summary

Keep open. Current main still treats user-initiated ClawHub account deletion as terminal and blocks deleted/deactivated users during OAuth, while the reporter's named-account cleanup requires production support access plus an account-lifecycle/security policy decision rather than conservative cleanup closure.

## What This Changes

The issue asks ClawHub to recover or purge stale GitHub-linked account state after account deletion so the reporter can sign in and publish skills.

## Best Possible Solution

Maintainers should choose one supported post-deletion lifecycle, then implement it consistently across OAuth callbacks, provider bindings, user records, support/admin tooling, user-facing errors, specs, docs, and regression tests.

## Reproduction Assessment

No, not for the exact wakeband/zocase production records because this review did not inspect live account data. The broader deleted-account OAuth lockout is source-reproducible: deletion writes deactivatedAt/purgedAt, OAuth calls handleDeletedUserSignIn, and active-user helpers reject the record.

## Solution Assessment

No for a one-off purge as the repository-level solution. The maintainable path is a maintainer-approved restore, purge/recreate, unlink/relink, or explicitly terminal support route with identity, privacy, audit, and abuse safeguards.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes OAuth identity reuse, deletion privacy guarantees, and ban-bypass boundaries.

Concerns:

- **[medium] Define recovery authority before changing sign-in:** `convex/auth.ts:40`
  - body: handleDeletedUserSignIn currently checks the incoming identity relationship and blocks deleted users; any restore, purge, recreate, or relink path needs equivalent provider-account verification and audit logging.
  - confidence: 0.9
- **[medium] Preserve ban and autoban separation:** `convex/auth.ts:51`
  - body: The reauth path checks ban audit records before migrating legacy deleted users, so recovery must not let banned or malware-blocked identities regain access through deleted-account handling.
  - confidence: 0.88
- **[medium] Keep deletion privacy semantics explicit:** `convex/users.ts:486`
  - body: deleteAccount revokes tokens and purges profile/contact/GitHub-derived fields, so recovery or re-registration needs clear rules for restored data, published content ownership, audit records, and token state.
  - confidence: 0.86
