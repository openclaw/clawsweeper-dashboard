## Summary

Keep open because current main still intentionally blocks deleted or deactivated ClawHub users from OAuth sign-in, and the requested recovery, purge, or relink path is a security-sensitive account-lifecycle decision rather than conservative cleanup.

## What This Changes

The issue requests account cleanup or recovery so a user can sign in after deleting a ClawHub account and switching GitHub accounts.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs, specs, and regression tests.

## Reproduction Assessment

No for the exact wakeband/zocase production records because live data was not inspected. Yes at the source level for the broader deleted-account lockout: current auth, deletion, access helpers, docs/specs, and tests intentionally reject deleted or deactivated users after OAuth.

## Solution Assessment

No for a one-off purge as the repository-level solution. The maintainable path is a maintainer-approved recovery, purge/recreate, unlink/relink, or explicitly terminal support flow with provider-account verification, privacy rules, audit logging, and abuse safeguards.

Status: needs_attention
