## Summary

Keep open. Current main still treats user-initiated ClawHub account deletion as permanent and blocks deleted/deactivated users during GitHub OAuth, while this clustered account lifecycle report needs maintainer product, security, and support policy rather than cleanup closure.

## What This Changes

Request to purge, recover, unlink, relink, or recreate GitHub-linked ClawHub auth state so the reporter can sign in after account deletion and a GitHub account change.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user/auth records, admin or support tooling, user-facing errors, docs, audit logs, and regression tests.

## Reproduction Assessment

No for the exact wakeband/zocase production records because the live database was not inspected. The broader deleted-account path is source-reproducible: current main records permanent deactivation and rejects deleted/deactivated users during OAuth and active-user lookup.

## Solution Assessment

Unclear. A one-off purge could unblock one account, but the maintainable solution is a maintainer-approved account lifecycle and identity recovery policy rather than ad hoc database cleanup.
