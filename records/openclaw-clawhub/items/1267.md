## Label Justifications

- `P1`: The issue blocks affected users from ClawHub sign-in and publishing after account deletion, with multiple related reports showing active workflow impact.
- `impact:security`: Recovery or relinking after deletion affects identity binding, deletion guarantees, abuse controls, and account ownership verification.
- `impact:auth-provider`: The failure is in the GitHub OAuth account-binding and post-callback sign-in path.
- `impact:session-state`: Users report OAuth success followed by no active ClawHub session, and current active-user helpers intentionally drop deleted/deactivated records.

## Summary

Keep open: current main still treats user-initiated deletion as terminal and has no supported recovery, purge/recreate, unlink/relink, or support-admin restore path. The source behavior is clear, but the remaining work is a product and security decision around account lifecycle rather than conservative cleanup.

## What This Changes

Investigate a ClawHub web authentication loop after account deletion and GitHub account switching, with a request for server-side cleanup or recovery.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs/specs, and tests.

## Reproduction Assessment

Yes, at source level: deletion writes deactivatedAt/purgedAt, OAuth callback handling rejects those records, and active-user helpers refuse them after callback. I did not run a live OAuth flow or inspect the named production rows.

## Solution Assessment

No for the one-off purge request: it may unblock one account, but the maintainable solution is a maintainer-approved restore, purge/recreate, unlink/relink, or explicitly terminal support route with security review.
