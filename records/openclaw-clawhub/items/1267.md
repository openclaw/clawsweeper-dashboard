## Summary

Keep open. Current main intentionally blocks sign-in for deleted/deactivated accounts and has no purge, recreate, unlink/relink, or recovery path for this account-lifecycle case; the related report cluster remains open and this needs maintainer product, security, and support judgment rather than cleanup closure.

## What This Changes

The issue requests a server-side cleanup or recovery path after account deletion leaves GitHub OAuth sign-in returning to the logged-out state.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, support/admin tooling, user-facing errors, specs/docs, and regression tests.

## Reproduction Assessment

No for the exact wakeband/zocase production records because live database state was not inspected. Yes at the source level for the broader deleted-account OAuth lockout: current deletion code writes deactivatedAt/purgedAt and the auth callback rejects deleted/deactivated users after GitHub OAuth.

## Solution Assessment

No for a one-off purge as the repository-level solution. The maintainable path is a maintainer-approved recovery, purge/recreate, unlink/relink, or explicitly terminal support flow with provider-account verification, privacy rules, audit logging, and abuse safeguards.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes OAuth identity reuse, deletion guarantees, token/audit semantics, and ban-bypass boundaries.

Concerns:

- **[medium] Define recovery authority before changing sign-in:** `convex/auth.ts:40`
  - body: handleDeletedUserSignIn currently verifies the incoming identity relationship and blocks deactivated users; any restore, purge, recreate, or relink behavior needs equivalent provider-account verification and audit logging.
  - confidence: 0.9
- **[medium] Preserve ban and autoban separation:** `convex/auth.ts:51`
  - body: The legacy deleted-user reauth path checks ban audit records before migration, so recovery must not let banned or malware-blocked identities regain access through deleted-account handling.
  - confidence: 0.88

  - body: deleteAccount purges profile/contact/GitHub-derived fields and revokes tokens, so recovery or re-registration needs clear rules for restored data, published content ownership, audit records, and token state.
