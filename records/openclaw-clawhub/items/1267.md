## Summary

Keep open. Current main still treats user-initiated ClawHub deletion as a permanent inactive account state, and this account-specific OAuth/recovery report needs maintainer product, security, and production-support handling rather than cleanup closure.

## What This Changes

The issue requests recovery or server-side cleanup for a GitHub OAuth login loop after deleting one ClawHub account and attempting to use another GitHub account.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement the chosen restore, purge/recreate, unlink/relink, or terminal-support path consistently across OAuth, account records, admin tooling, docs, audit logs, and tests.

## Reproduction Assessment

No for the exact wakeband/zocase production records because this review did not inspect the live database. The broader deleted-account OAuth lockout is source-reproducible: deleteAccount writes deactivatedAt/purgedAt, OAuth calls handleDeletedUserSignIn, and active-user helpers reject that account.

## Solution Assessment

No for a one-off purge as the general solution. The maintainable path is a maintainer-approved account lifecycle and identity-recovery policy with explicit safety checks, not ad hoc database cleanup from an automated review.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery or GitHub identity rebinding is security-sensitive and needs explicit maintainer policy before code or production data changes.

Concerns:

- **[medium] Define audited recovery before purge or relink:** `convex/auth.ts:37`
  - body: The sign-in path currently verifies the incoming identity and blocks deleted users; any restore, purge, recreate, or relink flow needs equivalent provider-account verification and audit logging to prevent the wrong GitHub identity from reclaiming an account or publisher namespace.
  - confidence: 0.9
- **[medium] Preserve ban and autoban boundaries:** `convex/auth.ts:48`
  - body: The reauth path checks ban audit records before migrating legacy deleted users, so recovery must not let banned or malware-blocked identities regain access through deleted-account handling.
  - confidence: 0.88

  - body: deleteAccount revokes tokens and purges profile/contact/GitHub-derived fields; any recovery or re-registration behavior needs clear rules for restored data, token state, published content ownership, and audit history.
  - confidence: 0.86
