## Summary

Keep open. Current main still deliberately treats user-initiated account deletion as a terminal auth state, related deleted-account OAuth reports remain open, and resolving recovery, purge, or relink behavior requires maintainer product/security judgment rather than conservative cleanup.

## What This Changes

Request a recovery or cleanup path for a ClawHub user locked out of GitHub OAuth after account deletion and a GitHub account change.

## Best Possible Solution

Define one maintainer-approved post-deletion lifecycle, then implement it consistently across OAuth callbacks, account state, auth bindings, operator/audit tooling, user-facing errors, docs, and regression tests.

## Reproduction Assessment

Yes, from source inspection for the deleted-account OAuth path: deleteAccount writes deactivatedAt/purgedAt and handleDeletedUserSignIn rejects that account during OAuth completion. I did not live-reproduce the named wakeband/zocase production records.

## Solution Assessment

No for an automatic purge, restore, or unlink patch as cleanup. The maintainable path first needs a product/security decision on whether deletion is terminal, restorable, relinkable, or operator-only, and what invariants must be preserved.
