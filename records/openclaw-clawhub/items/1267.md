## Summary

Keep open. Current main still makes user-initiated account deletion permanent and blocks later GitHub OAuth sign-in for the linked inactive account, while the requested recovery or purge path needs maintainer account-lifecycle, security, and production-support judgment.

## What This Changes

The issue requests recovery or server-side cleanup after account deletion so GitHub OAuth can create or restore a usable ClawHub login.

## Best Possible Solution

Choose one supported post-deletion lifecycle, then implement it consistently across OAuth callbacks, auth account bindings, deleted-user state, admin/support tooling, user-facing errors, docs/specs, and regression tests.

## Reproduction Assessment

No for the exact wakeband/zocase production records because live data was not inspected. Yes at the source level for the broader deleted-account OAuth lockout: current auth, deletion, access, specs, and tests all reject deactivated users after GitHub OAuth.

## Solution Assessment

No for a one-off purge as the repository-level source solution. The maintainable path is a maintainer-approved recovery, purge/recreate, unlink/relink, or terminal support flow with explicit identity, privacy, audit, and abuse safeguards.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes OAuth identity reuse, deletion privacy guarantees, token/audit semantics, and ban-bypass boundaries.

Concerns:

- **[medium] Define recovery authority before changing sign-in:** `convex/auth.ts:40`
  - body: handleDeletedUserSignIn currently verifies the incoming identity relationship and blocks deactivated users; any restore, purge, recreate, or relink path needs equivalent provider-account verification and audit logging.
  - confidence: 0.9
- **[medium] Preserve ban and autoban separation:** `convex/auth.ts:51`
  - body: The legacy deleted-user reauth path checks ban audit records before migration, so recovery must not let banned or malware-blocked identities regain access through deleted-account handling.
  - confidence: 0.88
- **[medium] Keep deletion privacy semantics explicit:** `convex/users.ts:486`
  - body: deleteAccount purges profile/contact/GitHub-derived fields and revokes tokens, so recovery or re-registration needs clear rules for restored data, published content ownership, audit records, and token state.
  - confidence: 0.86
