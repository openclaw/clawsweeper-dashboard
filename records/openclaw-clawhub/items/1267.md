## Summary

Keep open. Current main intentionally blocks sign-in for user-deleted accounts and now surfaces clearer blocked-account errors, but it still has no supported purge/recreate, unlink/relink, restore, or support recovery path for this clustered account lifecycle problem.

## What This Changes

The issue requests recovery or purge/recreation for a user who deleted one ClawHub account and then could not complete GitHub sign-in with another account.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs, specs, audit logging, and regression tests.

## Reproduction Assessment

No, not for the exact named `wakeband`/`zocase` production records because the live database was not inspected. The deleted-account path is source-reproducible: current auth code and tests reject deleted/deactivated users during sign-in and active-user lookup.

## Solution Assessment

Unclear. A one-off purge could unblock a single user, but the maintainable solution is a deliberate account lifecycle and identity recovery policy rather than an ad hoc database cleanup path.
