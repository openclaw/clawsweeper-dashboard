## Summary

Keep open because current main still treats user-initiated deletion as irreversible and blocks later OAuth sign-in, while the requested recovery, purge, or relink path is a security-sensitive account-lifecycle decision rather than cleanup closure.

## What This Changes

The issue requests recovery, purge, or relink support for a ClawHub account after deletion leaves GitHub OAuth sign-in in a refresh loop.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs/specs, and regression tests.

## Reproduction Assessment

No for the exact wakeband/zocase production records because live data was not inspected. Yes at the source level for the broader deleted-account OAuth lockout: current auth, deletion, access, docs/specs, and tests intentionally reject deleted or deactivated users after GitHub OAuth.

## Solution Assessment

No for a one-off purge as the repository-level solution. The maintainable path is a maintainer-approved recovery, purge/recreate, unlink/relink, or explicitly terminal support flow with provider-account verification, privacy rules, audit logging, and abuse safeguards.
