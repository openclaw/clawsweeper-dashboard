## Label Justifications

- `P1`: Affected users are blocked from ClawHub sign-in and publishing after account deletion, and the timeline links multiple similar active reports.
- `impact:security`: Recovery or relinking after deletion affects identity binding, deletion guarantees, abuse controls, and ownership verification.
- `impact:auth-provider`: The failure is in the GitHub OAuth account-binding and post-callback sign-in path.
- `impact:session-state`: Users report OAuth success followed by no active ClawHub session, and current active-user helpers intentionally drop deleted or deactivated records.

## Summary

Keep open: current main still intentionally treats user-initiated account deletion as terminal, and the clustered login-loop reports need a maintainer product/security decision for recovery, purge/recreate, unlink/relink, or clearer terminal handling rather than conservative cleanup.

## What This Changes

The issue asks ClawHub to recover or purge stale GitHub-linked account state after account deletion so a different GitHub account can sign in without an OAuth refresh loop.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs/specs, and regression tests.

## Reproduction Assessment

Yes, at source level: current main records deleted accounts as deactivated/purged, rejects those records during OAuth handling, and active-user helpers then return no usable session. I did not run a live GitHub OAuth flow or inspect the named production records.

## Solution Assessment

No for a one-off purge-only fix: that may unblock one account, but the maintainable solution needs a maintainer-approved restore, purge/recreate, unlink/relink, or explicitly terminal support route with security review.
