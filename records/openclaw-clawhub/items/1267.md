## Label Justifications

- `P1`: The issue blocks affected users from ClawHub sign-in and publishing after account deletion, with multiple related reports showing real workflow impact.
- `impact:security`: Recovery or relinking after deletion affects identity binding, deletion guarantees, abuse controls, and account ownership verification.
- `impact:auth-provider`: The failure is in the GitHub OAuth account-binding and post-callback sign-in path.
- `impact:session-state`: Users report OAuth success followed by no active ClawHub session, and current active-user helpers intentionally drop deleted/deactivated records.

## Summary

Keep open: current main still intentionally blocks deleted or deactivated users from GitHub sign-in, and there is no supported recovery, purge/recreate, unlink/relink, or support-admin restore path. The repeated related reports make this an account-lifecycle product and security decision, not a conservative cleanup close.

## What This Changes

The issue asks ClawHub to clear or recover stale deleted-account and GitHub identity state so the reporter can sign in with an established GitHub account.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs/specs, and tests.

## Reproduction Assessment

Yes, at source level: account deletion writes deactivatedAt/purgedAt, OAuth callback handling rejects those users, and active-user helpers refuse the resulting record. I did not run a live OAuth flow or inspect the named production rows.

## Solution Assessment

No for the one-off purge request: it may unblock one user, but the maintainable solution is a maintainer-approved restore, purge/recreate, unlink/relink, or explicitly terminal support route with security review.
