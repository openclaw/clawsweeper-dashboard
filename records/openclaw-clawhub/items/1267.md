## Summary

Keep this open. Current main still rejects deleted ClawHub accounts during GitHub auth, the related deleted-account login cluster remains open, and choosing recovery, purge/recreate, or relink behavior needs maintainer product and security direction.

## What This Changes

Request a supported way to sign in, recover, relink, or purge/recreate a GitHub-linked ClawHub account after account deletion and GitHub account changes.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs/specs, and tests.

## Reproduction Assessment

Yes, from source inspection for the deleted-account path: deactivatedAt users are rejected during GitHub auth, legacy deletedAt users are migrated to deactivatedAt/purgedAt and rejected, and active-user helpers refuse to load deleted/deactivated records. I did not verify the named production records or run a live OAuth trace.

## Solution Assessment

Unclear until maintainers choose the deletion policy. A one-off database purge might unblock a single reporter, but the maintainable path is an explicit restore, purge/recreate, unlink/relink, or terminal support process.
