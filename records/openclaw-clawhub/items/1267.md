## Summary

Keep open. Current main intentionally rejects deleted/deactivated GitHub-backed users and now surfaces clearer blocked-account errors, but the requested purge/recreate/relink recovery path is still unresolved and needs maintainer product/security handling.

## What This Changes

The issue requests server-side cleanup or recovery so a user can sign in to ClawHub after deleting an account tied to one GitHub account and switching to another.

## Best Possible Solution

Define a maintainer-approved post-deletion account lifecycle, then implement recovery, unlink/relink, purge/recreate, or an explicit terminal support path consistently across OAuth, user records, operator tooling, docs, specs, audit logs, and tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level deleted-account path: current main rejects deleted/deactivated users during OAuth completion and active-user lookup. No for the exact named production accounts, because their live database rows were not inspected.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A one-off purge could unblock one user, but the maintainable solution is a deliberate account lifecycle and identity recovery policy rather than ad hoc database cleanup.
