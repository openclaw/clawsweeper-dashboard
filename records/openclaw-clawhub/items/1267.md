## Summary

Keep this open. Current main still treats self-deleted ClawHub accounts as a terminal auth state, the clustered reports remain open, and resolving the reported login loop requires a maintainer product/security decision about account recovery rather than conservative cleanup.

## What This Changes

The issue requests a server-side way to recover, purge, or recreate login access after deleting a ClawHub account and then attempting GitHub OAuth with another account.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs, and regression tests.

## Reproduction Assessment

Yes from source inspection: a user record with deactivatedAt or legacy deletedAt is rejected during GitHub auth, and active-user helpers then refuse to load that account. I did not inspect production data for the named accounts, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear until maintainers choose the deletion policy. A narrow code patch should wait for a decision on whether deleted accounts are restorable, purged/recreated, relinkable, or intentionally terminal with a clear support path.
