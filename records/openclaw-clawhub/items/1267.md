## Summary

Keep this open. Current main still treats user-initiated deletion as a terminal auth state and has no recovery, purge/recreate, unlink/relink, or admin restore path; the related deleted-account OAuth cluster is still open and the ClawHub cleanup policy keeps issues open for maintainer handling.

## What This Changes

The issue asks for ClawHub account recovery or server-side cleanup when GitHub login loops after account deletion and a later GitHub login attempt.

## Best Possible Solution

Choose one supported post-deletion lifecycle for GitHub-linked users, then implement it consistently across OAuth callbacks, user records, support/admin tooling, user-facing errors, specs/docs, and regression tests.

## Reproduction Assessment

Yes for the clustered deleted-account path from source inspection: deactivated and legacy deleted users are rejected during OAuth and excluded by active-user helpers. The exact wakeband/zocase production state still needs maintainer access to verify.

## Solution Assessment

No for an ad-hoc purge as the durable solution. The maintainable path is a policy-backed lifecycle decision: restore, purge/recreate, unlink/relink, or an explicitly terminal state with a clear support route.
