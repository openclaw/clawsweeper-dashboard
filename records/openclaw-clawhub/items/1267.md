## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still treats user-initiated account deletion as terminal rather than providing recovery, purge/recreate, or GitHub unlink/relink support.

## What This Changes

The issue asks ClawHub to recover or purge stale GitHub-linked auth/account state after account deletion so login can succeed again.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user/auth records, admin or support tooling, user-facing errors, docs, and tests.

## Reproduction Assessment

Unclear for the exact two-GitHub-account wakeband/zocase scenario because production auth records were not inspected. There is a high-confidence source-level reproduction for the broader deletion loop class: delete an account, then sign in again with a GitHub identity mapped to that deactivated user and observe the auth rejection plus inactive app session.

## Solution Assessment

No, a one-off manual purge is not the best durable solution. The maintainable path is a product and security decision on whether deleted accounts are restorable, purgeable/recreatable, relinkable, or terminal with a clear support route.
