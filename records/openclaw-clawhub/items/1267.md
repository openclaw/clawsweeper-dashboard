## Summary

Keep this open. Current main still treats user-initiated account deletion as a terminal auth state, related deleted-account OAuth reports remain open, and resolving the loop needs a maintainer product/security decision rather than conservative cleanup.

## What This Changes

The issue requests fixing a ClawHub login loop after account deletion and GitHub account switching so the user can regain access or have stale auth records cleaned up.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, support/admin tooling, user-facing errors, docs/specs, and regression tests.

## Reproduction Assessment

Yes from source inspection for the clustered deleted-account path: a user with deactivatedAt or legacy deletedAt is rejected during GitHub auth, and active-user helpers refuse to load that account. I did not inspect production data for the named accounts, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear until maintainers choose the deletion policy. A narrow code patch should wait for a decision on whether deleted accounts are restorable, purged/recreated, unlinkable/relinkable, or intentionally terminal with a clear support route.
