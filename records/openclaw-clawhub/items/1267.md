## Summary

Keep open. Current main still treats user-initiated ClawHub account deletion as irreversible and blocks later GitHub OAuth reauthentication rather than providing purge, recreate, restore, unlink, or relink recovery; the related login-loop cluster also remains open and ClawHub cleanup policy keeps issues in maintainer hands.

## What This Changes

The issue requests server-side recovery or purge for a ClawHub user stuck in a GitHub OAuth login loop after deleting one account and trying to sign in with another GitHub account.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user/auth records, admin/support tooling, user-facing errors, docs, and tests.

## Reproduction Assessment

Yes for the same-account post-deletion path: current source and tests show deleteAccount sets deactivatedAt/purgedAt and later OAuth reauth is rejected while active-user helpers hide the account. The exact two-GitHub-account production state is not reproducible from the repository alone.

## Solution Assessment

No patch is proposed, and a one-off manual purge is not the best durable product fix. The maintainable path is a maintainer-approved account lifecycle and recovery policy keyed on immutable GitHub identity, with clear handling for user deletion versus bans.
