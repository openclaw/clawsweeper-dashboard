## Summary

Keep this open. Current main still intentionally makes user-initiated account deletion terminal, blocks later GitHub OAuth sign-in for deleted/deactivated users, and has no restore, purge/recreate, unlink/relink, or support-admin recovery path; the clustered reports point to a product and security decision rather than cleanup closure.

## What This Changes

Allow users who deleted a GitHub-linked ClawHub account to regain access through a defined recovery, purge/recreate, or identity relink path.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs/specs, and tests.

## Reproduction Assessment

Yes, at source level: deleteAccount writes deactivatedAt/purgedAt, handleDeletedUserSignIn rejects those users during GitHub auth, and active-user helpers refuse to load them. I did not run a live OAuth flow or inspect the named production records.

## Solution Assessment

Unclear until maintainers choose the deletion policy. A one-off purge may unblock one reporter, but the maintainable fix is an explicit restore, purge/recreate, unlink/relink, or terminal support route with security review.
