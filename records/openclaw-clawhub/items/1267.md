## Summary

Keep this open. Current main still treats user-initiated deletion as a terminal auth state, the related deleted-account OAuth reports remain open, and choosing restore, purge/recreate, unlink/relink, or terminal-support behavior needs maintainer product and security judgment.

## What This Changes

The issue requests a way to recover or clean up a ClawHub GitHub login loop after account deletion and a subsequent GitHub account change.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs/specs, and regression tests.

## Reproduction Assessment

Yes for the clustered deleted-account path from source inspection: deactivatedAt and legacy deletedAt records are rejected during GitHub auth and active-user helpers refuse them. The exact wakeband/zocase production record state was not verified, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear until maintainers choose the deletion policy. A one-off purge may unblock a user, but it is not the best general fix unless the product chooses purge/recreate or unlink/relink as the supported recovery model.
