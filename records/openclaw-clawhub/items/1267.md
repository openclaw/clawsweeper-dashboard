## Summary

Keep open. Current main still treats user-initiated ClawHub account deletion as irreversible and rejects later OAuth sign-in for the same deleted account state, while the remaining recovery/purge/relink behavior needs maintainer account-lifecycle and security policy rather than cleanup closure.

## What This Changes

The issue requests resolving a ClawHub GitHub OAuth login loop after account deletion, including possible purge, recreate, restore, or relink handling for the affected GitHub identities.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, specs/docs, and regression tests.

## Reproduction Assessment

No for the exact wakeband/zocase production records because live data was not inspected. The broader deleted-account OAuth lockout is source-reproducible: deletion writes deactivatedAt/purgedAt, OAuth calls handleDeletedUserSignIn, and active-user helpers reject deactivated users.

## Solution Assessment

No for a one-off purge as the repository-level solution. The maintainable path is a maintainer-approved restore, purge/recreate, unlink/relink, or terminal support flow with explicit identity, privacy, audit, and abuse safeguards.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes OAuth identity reuse, irreversible deletion guarantees, and ban-bypass boundaries.

Concerns:

- **[medium] Define recovery authority before changing sign-in:** `convex/auth.ts:40`
  - body: handleDeletedUserSignIn currently verifies the incoming identity relationship and blocks deleted users; any restore, purge, recreate, or relink path needs equivalent provider-account verification and audit logging.
  - confidence: 0.9
- **[medium] Preserve ban and autoban separation:** `convex/auth.ts:51`
  - body: The reauth path checks ban audit records before migrating legacy deleted users, so recovery must not let banned or malware-blocked identities regain access through deleted-account handling.
  - confidence: 0.88
- **[medium] Keep deletion privacy semantics explicit:** `convex/users.ts:486`
  - body: deleteAccount revokes tokens and purges profile/contact/GitHub-derived fields, so recovery or re-registration needs clear rules for restored data, published content ownership, audit records, and token state.
  - confidence: 0.86
