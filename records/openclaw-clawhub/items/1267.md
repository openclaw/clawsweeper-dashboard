## Summary

Keep open. Current main intentionally blocks re-auth for user-deleted accounts and now surfaces clearer blocked-account errors, but it still has no supported purge/recreate, unlink/relink, or deleted-account recovery path; the clustered reports need maintainer policy and support-design decisions.

## What This Changes

Request account cleanup or recovery for a GitHub-linked ClawHub account that remains unable to sign in after user-initiated account deletion and OAuth with another GitHub account.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, admin/support tooling, user-facing errors, docs, specs, and regression tests.

## Reproduction Assessment

No high-confidence reproduction for the exact named `wakeband`/`zocase` production records because the live database was not inspected. The clustered deleted-account state is source-reproducible: current auth code and tests reject deleted/deactivated users during sign-in and then treat them as inactive.

## Solution Assessment

Unclear. The current implementation is internally consistent with irreversible deletion and now has clearer auth errors, but the requested purge/recreate or unlink/relink behavior is a product and security-policy choice rather than a narrow bug fix.
