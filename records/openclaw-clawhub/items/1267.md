## Summary

Keep this open. Current main still rejects deleted/deactivated ClawHub users during GitHub auth, and the remaining recovery, purge, or relink behavior needs maintainer product and security direction before code changes are safe.

## What This Changes

The issue requests a way for a user who deleted a GitHub-linked ClawHub account to sign in again, recreate or relink the account, or have stale server-side auth records purged.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then implement it consistently across OAuth callbacks, user records, support/admin tooling, user-facing errors, specs/docs, and tests.

## Reproduction Assessment

Yes, from source inspection: a user record with deactivatedAt or legacy deletedAt is rejected by handleDeletedUserSignIn during GitHub auth, and active-user helpers then refuse to load that user. I did not run a live OAuth flow or inspect the reporter's production rows.

## Solution Assessment

Unclear until maintainers choose the deletion policy. A one-off purge might unblock the reporter, but the maintainable solution needs an explicit restore, purge/recreate, unlink/relink, or terminal support route with security review.
