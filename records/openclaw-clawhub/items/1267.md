## Summary

Keep this open. Current main still treats user-initiated account deletion as terminal and rejects later GitHub auth for deactivated users; the related reports and current labels point to a platform account-lifecycle decision, not a cleanup close.

## What This Changes

The issue asks for account recovery or server-side cleanup when GitHub OAuth loops after deleting a ClawHub account and then trying to sign in with another GitHub account.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked ClawHub users, then align OAuth callbacks, user records, admin/support tooling, user-facing errors, docs/specs, and tests around that policy.

## Reproduction Assessment

Yes, at source level: account deletion writes deactivatedAt/purgedAt, the OAuth callback rejects deactivated users, and active-user helpers refuse to load them. I did not run a live OAuth flow or inspect the named production records.

## Solution Assessment

Unclear until maintainers choose the deletion policy. A one-off purge may unblock one user, but the maintainable solution is an explicit restore, purge/recreate, unlink/relink, or terminal support route with security review.
