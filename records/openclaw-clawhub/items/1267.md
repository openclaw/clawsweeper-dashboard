## Summary

Keep open. Current main intentionally rejects deleted/deactivated users during OAuth and now surfaces clearer blocked-account messages, but the requested recovery/relink/purge behavior still needs maintainer account-lifecycle policy, and ClawHub issue cleanup policy does not auto-close open issues.

## What This Changes

The issue asks for recovery or server-side cleanup after a deleted ClawHub account leaves GitHub sign-in unable to create a usable session.

## Best Possible Solution

Define one approved post-deletion lifecycle, then implement it consistently across OAuth callbacks, auth account bindings, deletion, support/admin tooling, user-facing errors, specs/docs, and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the exact wakeband/zocase production records; yes for the broader source path where deleted/deactivated users are rejected during OAuth and by active-user helpers.

## Solution Assessment

Is this the best way to solve the issue? No for a one-off purge as the repository-level fix; the maintainable solution is a maintainer-approved restore, purge/recreate, unlink/relink, or terminal support path with explicit security and privacy rules.
