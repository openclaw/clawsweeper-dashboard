## Summary

Keep open because current main still treats user-initiated deletion as terminal and rejects later OAuth sign-in, while the requested purge/relink/recovery behavior is an account-lifecycle, product, and security decision rather than conservative cleanup.

## What This Changes

The report requests a purge, relink, or recovery path so a user can sign in to ClawHub after deleting a GitHub-linked account and trying another GitHub account.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked users, then implement it consistently across OAuth callbacks, user records, support/admin tooling, user-facing errors, docs, specs, and regression tests.

## Reproduction Assessment

No for the exact wakeband/zocase production records because live data was not inspected. Yes at the source level for the broader deleted-account lockout: current auth, deletion, access helpers, specs, docs, and tests intentionally reject deleted or deactivated users after OAuth.

## Solution Assessment

No for a one-off purge as the repository-level solution. The maintainable path is a maintainer-approved recovery, purge/recreate, unlink/relink, or explicitly terminal support flow with provider-account verification, privacy rules, audit logging, and abuse safeguards.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes OAuth identity reuse, deletion guarantees, token/audit semantics, and ban-bypass boundaries.

  - confidence: 0.9

  - body: deleteAccount clears profile/contact/GitHub-derived fields and revokes tokens, so recovery or re-registration needs explicit rules for restored data, published content ownership, audit records, and token state.
