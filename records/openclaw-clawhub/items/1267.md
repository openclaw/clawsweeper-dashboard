## Summary

Keep this open. Current main still treats user-initiated deletion as terminal and blocks later GitHub auth for deleted/deactivated users; the clustered reports show this needs a maintainer product and security decision rather than cleanup closure.

## What This Changes

The issue requests a way to recover from or clean up a ClawHub GitHub login loop after deleting an account and attempting to sign in with GitHub again.

## Best Possible Solution

Define one supported post-deletion lifecycle for GitHub-linked users, then align OAuth callbacks, user records, admin/support tooling, user-facing errors, docs/specs, and tests around that policy.

## Reproduction Assessment

Yes, at source level: deleteAccount writes deactivatedAt/purgedAt, handleDeletedUserSignIn rejects those users during GitHub auth, and active-user helpers refuse to load them. I did not run a live OAuth flow or inspect the named production records.

## Solution Assessment

Unclear until maintainers choose the deletion policy. A one-off purge may unblock a reporter, but the maintainable solution is an explicit restore, purge/recreate, unlink/relink, or terminal support route with security review.
