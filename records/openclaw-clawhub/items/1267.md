## Summary

Keep open. Current main intentionally blocks deleted/deactivated ClawHub accounts from completing GitHub OAuth sign-in, the related deleted-account OAuth loop cluster still has multiple open reports, and changing this needs maintainer account-recovery/security policy rather than cleanup closure.

## What This Changes

The issue requests account cleanup or recovery so a user who deleted one GitHub-linked ClawHub account can sign in or recreate access with another GitHub account.

## Best Possible Solution

Maintainers should choose and document one post-deletion lifecycle, then implement it consistently across OAuth, account deletion, auth bindings, support/admin tooling, user-facing errors, specs/docs, and regression tests.

## Reproduction Assessment

Unclear for the exact wakeband/zocase production records because live data was not inspected. The broader deleted-account OAuth lockout is source-reproducible: deletion records deactivatedAt/purgedAt, OAuth calls handleDeletedUserSignIn, and active-user helpers reject the user.

## Solution Assessment

No for a one-off purge as the repository-level solution. The maintainable path is a maintainer-approved restore, purge/recreate, unlink/relink, or explicit terminal-support policy with identity, privacy, audit, and abuse safeguards.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes OAuth identity reuse, deletion privacy guarantees, token revocation, and ban-bypass boundaries.

Concerns:

- **[medium] Define recovery authority before changing sign-in:** `convex/auth.ts:37`
  - body: handleDeletedUserSignIn currently verifies the incoming identity relationship and blocks deleted users; any restore, purge, recreate, or relink path needs equivalent provider-account verification and audit logging.
  - confidence: 0.9
- **[medium] Preserve deletion privacy and token semantics:** `convex/users.ts:348`
  - body: deleteAccount revokes tokens and clears profile/contact/GitHub-derived fields, so recovery must define what data can return and must not revive revoked credentials unexpectedly.
  - confidence: 0.86
- **[medium] Keep ban and deletion handling separate:** `convex/auth.ts:48`
  - body: The reauth path checks ban audit records before migrating legacy deleted users, so recovery must not let banned or malware-blocked identities regain access through deleted-account handling.
  - confidence: 0.88
