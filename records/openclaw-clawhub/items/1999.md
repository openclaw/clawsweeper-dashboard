## Summary

Keep open: this is an open ClawHub issue, and current production deployed from current main still fails the GitHub OAuth sign-in start route with HTTP 500 while the site and public skills API serve normally. The source has Convex Auth routes and frontend callback handling, so the next step is maintainer/operator auth triage rather than cleanup.

## What This Changes

The issue reports a production GitHub OAuth flow where authorization returns to clawhub.ai without persisting Convex Auth JWT or refresh tokens in browser storage.

## Best Possible Solution

Restore production GitHub OAuth so active accounts receive durable Convex Auth sessions, preserve deleted and banned account rejection, and add focused smoke or regression coverage after the failing layer is identified.

## Reproduction Assessment

Yes for the current production sign-in availability failure: current main is tagged prod/v2026.05.10.4, the site and skills API return HTTP 200, but both canonical and direct Convex GitHub sign-in endpoints return HTTP 500. No for the exact callback-succeeds/no-JWT variant, because the flow currently fails one step earlier.

## Solution Assessment

Not applicable because there is no proposed patch. The narrowest maintainable path is maintainer/operator auth triage using production Convex logs, OAuth app settings, cookie/storage traces, and affected account records before choosing a config, dependency, or source fix.
