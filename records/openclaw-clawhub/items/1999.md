## Summary

Keep open: ClawHub policy does not auto-close issues, and the production GitHub OAuth path still fails before a normal session can be established. Current main delegates the affected route to Convex Auth and has no source-level proof that the reported callback/no-JWT path is fixed.

## What This Changes

The issue reports that GitHub OAuth returns to `clawhub.ai` without persisting Convex Auth session tokens, leaving users logged out after sign-in.

## Best Possible Solution

Restore production GitHub OAuth so active accounts receive a Convex Auth browser session, preserve deleted/banned-account rejection semantics, and add focused regression or smoke coverage once the failing layer is identified.

## Reproduction Assessment

Yes for the central production sign-in outage: both the canonical ClawHub route and direct Convex GitHub sign-in endpoint currently return HTTP 500. No for the exact callback-succeeds/no-JWT variant, because the live flow now fails one step earlier.

## Solution Assessment

Unclear because there is no patch and the failure boundary likely spans production Convex Auth configuration, OAuth callback behavior, cookies, and logs. The narrowest maintainable path is maintainer/operator triage followed by a verified source, dependency, or deployment-config repair.
