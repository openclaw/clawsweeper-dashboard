## Summary

Keep open. Current main still leaves the reported discovery ambiguity in place: `clawhub search` and `/api/v1/search` omit owner identity, while `clawhub install` still accepts only bare slugs.

## What This Changes

Request to make ClawHub skill discovery less ambiguous by showing owner identity in search results or supporting clearer owner-qualified install targets.

## Best Possible Solution

Expose owner identity additively in the public v1 skill search response, update both schema copies, render owner information in `clawhub search`, and document the output/API change while deferring owner-qualified install syntax.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI output, public search response, and both search schemas are ownerless even though internal search entries already carry owner fields.

## Solution Assessment

Yes for the narrow owner-disambiguated search/API display path. Adding full owner-qualified install parsing in the same change is not the best first fix because current install normalization intentionally rejects slash-containing inputs before filesystem and lockfile side effects.
