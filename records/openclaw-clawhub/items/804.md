## Summary

Keep open. Current main still reproduces the reported owner ambiguity in `clawhub search`: internal search entries carry owner identity, but the public v1 search response, schemas, and CLI renderer still omit it.

## What This Changes

The issue asks ClawHub to make skill discovery/install targets less ambiguous by showing owner identity or owner-qualified locators for similarly named skills.

## Best Possible Solution

Expose owner identity additively in `/api/v1/search`, update both schema copies, render owner information in `clawhub search`, and leave owner-qualified install locators for a separate maintainer decision.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: internal search entries include `ownerHandle` and `owner`, but the v1 handler, schemas, and CLI renderer drop that identity before `clawhub search` output.

## Solution Assessment

Yes for the narrow owner-disambiguated search/API display path. Adding full owner-qualified install syntax in the same pass is not the best first fix because current install validation deliberately rejects slash-containing locators before filesystem and lockfile operations.
