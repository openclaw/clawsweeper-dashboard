## Summary

Keep open. Current main still leaves the reported owner ambiguity in CLI search and the public v1 search response, even though internal search entries already carry owner identity.

## What This Changes

The issue asks ClawHub skill discovery and install surfaces to disambiguate skills by owner, such as owner/slug syntax, skill-page guidance, or an owner column in CLI search.

## Best Possible Solution

Expose owner identity additively in `/api/v1/search`, update shared and CLI schemas plus docs, render owner information in `clawhub search`, and keep owner-qualified install locator semantics as a separate decision.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: internal search entries include owner identity, but the public v1 handler, schemas, and CLI renderer drop it before `clawhub search` output.

## Solution Assessment

Yes for the narrow owner-disambiguated search/API display path. Full owner-qualified install syntax should not be folded into the same automated repair because it changes path validation, extraction targets, and lockfile semantics.
