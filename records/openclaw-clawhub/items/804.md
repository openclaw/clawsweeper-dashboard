## Summary

Keep open under the repository policy for ClawHub issue cleanup, even though the central owner-disambiguated search behavior is now present on current main and production-tagged history. No ClawSweeper repair lane is needed unless maintainers decide to split out the broader owner-qualified install syntax as a separate product item.

## What This Changes

The issue asks for ClawHub skill search/install discovery to show enough owner identity to distinguish similarly named skills such as `steipete/weather` from other `weather` results.

## Best Possible Solution

Keep the shipped owner-aware search/API/CLI/docs behavior; if maintainers want `clawhub install owner/slug`, track that separately with explicit filesystem, lockfile, and compatibility semantics.

## Reproduction Assessment

No for the original ownerless search ambiguity on current main: source inspection shows `/api/v1/search` returns owner metadata and `clawhub search` renders it. The broader owner-qualified install syntax is a separate product decision, not a current reproduction failure.

## Solution Assessment

Yes for the reported ambiguity: adding minimal owner metadata and rendering it as a separate owner column is the narrowest maintainable fix. It avoids teaching users to paste `owner/slug` into an install path that still validates bare slugs only.
