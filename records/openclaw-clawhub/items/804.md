## Summary

Keep open under the ClawHub issue policy: current main has shipped owner-aware search/API/docs, so the original ownerless CLI ambiguity no longer reproduces, but the remaining owner-qualified install locator is a maintainer product decision.

## What This Changes

The issue asks for clearer owner-qualified skill identity in ClawHub search/install flows, such as owner/name locators, install-page guidance, or an owner column in CLI search results.

## Best Possible Solution

Keep the shipped owner-aware search behavior; if maintainers want scoped installs, implement @owner/name as a deliberate CLI/API/filesystem/lockfile migration aligned with specs/orgs.md.

## Reproduction Assessment

No. The original ownerless `clawhub search` ambiguity no longer reproduces by source inspection because current main prints owner handles and returns owner metadata; only unsupported owner-qualified install syntax remains observable, and that is not part of the current install contract.

## Solution Assessment

Yes for the shipped discovery fix: owner-aware search resolves the confusing result list without teaching an unsupported locator. Full owner-qualified install support should be handled through the scoped identity migration rather than a narrow CLI-only patch.
