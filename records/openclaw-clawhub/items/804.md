## Summary

Keep open. Current main still leaves the reported CLI discovery ambiguity unresolved: public v1 search and `clawhub search` omit owner identity, while `clawhub install` still accepts only bare slugs and rejects slash-qualified locators.

## What This Changes

Request clearer owner/slug identity in ClawHub discovery and install flows so similarly named skills can be distinguished by publisher.

## Best Possible Solution

Expose owner identity additively in public v1 skill search, update both schema copies and docs, and render owner information in `clawhub search` while keeping install locator semantics unchanged for now.

## Reproduction Assessment

Yes. Source inspection on current main shows the search response/schema and CLI output omit owner fields, and slash-qualified install targets are rejected before lookup.

## Solution Assessment

Yes for the narrowed owner-display repair; it is the smallest maintainable fix for the reporter's discovery problem. Full owner-qualified install syntax should be handled as a separate maintainer-reviewed locator design.

## Repair Work Prompt

Implement a narrow additive fix for the ClawHub skill search owner ambiguity reported at https://github.com/openclaw/clawhub/issues/804. Add nullable owner identity, preferably `ownerHandle`, to public v1 skill search results by plumbing the existing internal search `ownerHandle` through `convex/httpApiV1/skillsV1.ts`; update both schema copies in `packages/schema/src/schemas.ts` and `packages/clawhub/src/schema/schemas.ts`; update `docs/http-api.md`; and render owner information in `clawhub search` so users can distinguish entries like `steipete/weather` from other `weather` slugs. Add focused tests for the handler response, schema parsing, and CLI output. Keep `clawhub install` semantics, slash rejection, extraction layout, and bare-slug resolution unchanged; stop and ask for maintainer review if the implementation requires owner-qualified install locators or filesystem path changes.
