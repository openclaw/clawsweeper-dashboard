## Summary

Keep open: current main has fixed the original ownerless `clawhub search` ambiguity, but the owner-scoped install locator and skill-page install command behavior remains an open product/API decision already represented by https://github.com/openclaw/clawhub/issues/2348.

## What This Changes

The issue asks ClawHub to disambiguate skill search and install identifiers so users can tell which publisher owns a skill and avoid installing a similarly named bare slug.

## Best Possible Solution

Keep the shipped owner metadata in search, and resolve the remaining install identity through the owner-scoped install/codeblock design tracked in https://github.com/openclaw/clawhub/issues/2348 with an explicit compatibility path for existing bare-slug installs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the remaining behavior: current search exposes owners, while `clawhub install` still rejects slash-containing locators and the compact install command still renders a bare slug. I did not run a live install because the review is read-only.

## Solution Assessment

No, closing this as fully implemented would be premature. The search-owner fix is the right narrow repair for the original ambiguity, but owner-scoped install/codeblock behavior should be decided through the open canonical follow-up before this issue is swept away.
