## Summary

Keep open. Current main still leaves the reported owner ambiguity in the CLI search path: backend search entries already know the owner, but the public v1 search response, both schema copies, and `clawhub search` output omit that identity; ClawHub cleanup policy also keeps issues open for maintainer handling.

## What This Changes

The issue asks for clearer owner-qualified skill identity in ClawHub search/install flows so users can distinguish similarly named skills by publisher.

## Best Possible Solution

Expose owner identity additively in `/api/v1/search`, update both schema copies, render owner information in `clawhub search`, and keep full owner-qualified install locators as a separate follow-up.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: internal search results carry `ownerHandle` and `owner`, but the v1 response, schemas, and CLI renderer drop that identity before `clawhub search` output.

## Solution Assessment

Yes for the narrow owner-disambiguated search/API display path. Adding full `owner/slug` install syntax in the same pass is not the best first fix because current install validation deliberately rejects slash-containing locators before filesystem and lockfile operations.
