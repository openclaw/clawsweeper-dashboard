## Summary

Keep open under the ClawHub issue policy and because the remaining scoped install locator question still needs maintainer product judgment. Current main has already shipped owner-aware search output and API metadata, but `clawhub install owner/slug` or `user_name/skill_name:version` remains unsupported and touches compatibility-sensitive install contracts.

## What This Changes

The issue asks ClawHub to reduce ambiguity between same-named skills by showing publisher identity in search/install workflows or supporting owner-qualified skill locators.

## Best Possible Solution

Keep the shipped owner-aware search/API/docs behavior, and handle any future scoped install locator as an explicit install-contract design with upgrade tests for bare slugs, lockfiles, API resolution, and extraction paths.

## Reproduction Assessment

No. Current source and regression tests show `clawhub search` now includes owner metadata; the remaining `owner/slug` or `name:version` install syntax is an unsupported new locator feature rather than a reproducible break in documented behavior.

## Solution Assessment

Yes for the shipped discovery fix: showing owner metadata separately resolves the reported ambiguity without teaching users to paste an unsupported locator. Full scoped install support should go through the broader install identity contract rather than a narrow CLI-only patch.
