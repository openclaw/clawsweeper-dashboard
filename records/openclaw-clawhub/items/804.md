## Summary

Keep open under the ClawHub issue policy: current main has fixed the original ownerless CLI/API search ambiguity, but full owner-qualified install syntax remains a broader product and compatibility decision.

## What This Changes

The issue asks for clearer owner-qualified skill identity in ClawHub search/install flows so users can distinguish skills with similar names or slugs.

## Best Possible Solution

Keep the shipped owner-aware search/API/docs behavior; if scoped installs are desired, implement `@owner/name` through the planned orgs/scoped-identity migration with compatibility tests for bare slugs and existing lockfiles.

## Reproduction Assessment

No. The original ownerless `clawhub search` ambiguity no longer reproduces by source inspection and regression tests; only unsupported owner-qualified install syntax remains, and that is future scoped-locator work rather than current broken behavior.

## Solution Assessment

Yes for the shipped discovery fix: showing owner metadata in search resolves the reported ambiguity without teaching an unsupported locator. Full scoped install support should be handled as the broader `@owner/name` migration, not as a piecemeal CLI-only change.
