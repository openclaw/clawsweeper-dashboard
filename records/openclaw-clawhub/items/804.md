## Summary

Keep open under the ClawHub issue policy and because the remaining meaningful work is a maintainer product decision around scoped install locators. Current main already resolves the reported search ambiguity by showing owner metadata in CLI/API/docs, and https://github.com/openclaw/clawhub/pull/2183 was closed after equivalent behavior landed on main.

## What This Changes

The issue asks for clearer skill identity in ClawHub search/install flows, including owner-visible search output or a user/skill-style install locator.

## Best Possible Solution

Keep the shipped owner-aware search/API/docs behavior, and only add scoped install locators through an explicit install identity design with compatibility tests for bare slugs, lockfiles, origin metadata, resolution, extraction paths, and docs.

## Reproduction Assessment

No. Current source and regression tests show `clawhub search` now includes owner metadata, while the remaining `owner/slug` or versioned locator syntax is unsupported new behavior rather than a broken documented path.

## Solution Assessment

Yes for the shipped discovery fix: owner metadata in search resolves the ambiguity without teaching users to paste an unsupported locator. Full scoped install support should go through the broader install identity contract rather than a narrow CLI-only patch.
