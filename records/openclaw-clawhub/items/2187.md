## Summary

Keep open: current main matches the reported inconsistency, and resolving it needs an API contract decision rather than conservative cleanup. Several v1 handlers still return `text/plain` errors while OpenAPI mostly lists non-2xx responses by status/description only.

## What This Changes

The issue asks maintainers to clarify whether public v1 API errors should remain plain text, be documented that way, or gain a stable JSON error envelope.

## Best Possible Solution

Make an explicit v1 API compatibility decision, then align handlers, tests, CLI/client parsing, OpenAPI, and public docs with that chosen error contract.

## Reproduction Assessment

Yes. Source inspection shows the reported 400/404/429 paths return plain text today, and tests assert some of those raw text bodies; I did not need to run live production requests to confirm the code path.

## Solution Assessment

Unclear until maintainers choose the contract. Documenting the current plain-text behavior is the narrowest low-risk fix, while `Accept: application/json` negotiation or a JSON envelope is a user-facing API change that should be intentional.
