## Summary

The exact-version revocation direction is coherent and current main does not already provide it, but this PR should remain open because its public version listing can hide active versions behind revoked rows, the branch is currently conflicted, and the security-sensitive behavior lacks real Convex proof.

## What This Changes

The PR adds moderator-controlled exact skill-version revocation, blocks public delivery of revoked artifacts, exposes revocation in bulk security verdicts, and extends the moderation CLI, schemas, tests, docs, and security specification.

## Best Possible Solution

Keep one exact-version, auditable revocation record and shared availability predicate, but make public listing pagination skip revoked rows without truncating pages, then validate revoke and restore end to end on a real Convex deployment before coordinating the ClawHub-first rollout with the OpenClaw consumer.

## Reproduction Assessment

Yes for the review finding: source inspection shows `listVersionsPage` paginates first and filters revoked rows afterward, so a fetched page containing only revoked versions returns empty despite later active rows. The broader feature itself is new behavior rather than a current-main bug reproduction.

## Solution Assessment

No in its current form. A shared exact-version availability predicate is the right foundation, but public pagination must preserve usable page semantics and the security contract needs real Convex evidence before merge.
