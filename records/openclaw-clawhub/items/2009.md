## Summary

Keep open: current main still omits package `openclaw.environment` metadata from package ClawScan prompts, and this PR addresses the right bug but needs fixes before merge.

## What This Changes

The PR parses package.json `openclaw.environment` during package LLM evaluation and injects normalized env vars/config paths into ClawScan prompt metadata with coverage.

## Best Possible Solution

Land a sanitized mapper that forwards only normalized environment-derived prompt fields, preserves valid existing requirements, ignores malformed unrelated metadata, and adds regression coverage for those cases.

## Reproduction Assessment

Yes, by source inspection: current main derives `environment:declared` tags but does not pass package `openclaw.environment` into the metadata that `assembleEvalUserMessage` reads. I did not execute tests because this review is read-only.

## Solution Assessment

No, not as written: the direction is right, but spreading the full package `openclaw` object creates avoidable prompt/crash surface and drops existing `requires` data. The narrower maintainable solution is to construct a sanitized metadata object for env/config only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Sanitize forwarded OpenClaw metadata before prompting:** `convex/llmEval.ts:433`
  - body: `buildOpenClawMetadataForEval` spreads the full package `openclaw` object into prompt metadata. If a package includes malformed `openclaw.install` such as a string, `assembleEvalUserMessage` treats it as an array and calls `.map`, throwing before package LLM analysis is stored. Build a sanitized object with only normalized env/config fields, or normalize unrelated fields before forwarding.
  - confidence: 0.9
- **[P2] Preserve existing requirements when adding config paths:** `convex/llmEval.ts:79`
  - body: When `environment.configPaths` is present, this replaces any existing `openclaw.requires` object with `{ config }`. Packages that also declare `requires.env`, `bins`, or `anyBins` would lose those signals in the ClawScan prompt, so merge config paths into a sanitized requirements object instead.
  - confidence: 0.84
