## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still has a source-supported path where browser login receives a token, then `/api/v1/whoami` verification can fail through the shared HTTP client and display only the generic `fetch failed` message; the broader shared diagnostics issue at https://github.com/openclaw/clawhub/issues/651 is also still open.

## What This Changes

The issue asks ClawHub CLI login to handle a browser-auth `fetch failed` failure instead of leaving the user with only the generic error.

## Best Possible Solution

Keep token verification intact, consolidate with the open shared diagnostics work where practical, and add redacted transport diagnostics plus login-specific troubleshooting and fallback guidance.

## Reproduction Assessment

Yes for a high-confidence source-level path: after browser auth returns a token, make the `/api/v1/whoami` verification fetch throw `TypeError('fetch failed')`, and current code reports only `formatError(error)`. I did not reproduce the reporter's exact macOS/network environment.

## Solution Assessment

Yes. The narrow maintainable fix is sanitized diagnostics at the shared HTTP/error boundary plus login-specific docs, not bypassing `/api/v1/whoami` verification or saving unverified tokens.
