## Summary

Keep open. Current main still has a source-supported CLI path where browser login can receive a token and then fail `/api/v1/whoami` verification with only the raw `fetch failed` transport message; the related open report shows the same shared diagnostics gap.

## What This Changes

The issue reports `npx clawhub login` opening the browser auth URL and then failing with `fetch failed`.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics plus login-specific proxy and `clawhub login --token` fallback guidance, with focused regression coverage and troubleshooting docs.

## Reproduction Assessment

Yes for the source-level failure path: after browser auth returns a token, make the `/api/v1/whoami` verification fetch throw `TypeError('fetch failed')` and the CLI can still print only `fetch failed`. The reporter's exact network environment was not independently reproduced.

## Solution Assessment

Yes. A shared HTTP/error-formatting diagnostics repair is the narrowest maintainable solution because login, search, install, publish, and related CLI commands already converge through that boundary.
