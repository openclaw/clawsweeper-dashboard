## Summary

Keep open: ClawHub policy does not auto-close issues, and current main still has a source-supported CLI login path where browser auth can succeed but `/api/v1/whoami` verification can fail through the shared HTTP transport and surface only the generic `fetch failed` message. The related open report at https://github.com/openclaw/clawhub/issues/651 points to the same shared CLI transport-diagnostics gap.

## What This Changes

The issue reports `npx clawhub login` opening browser auth and then failing with a generic `fetch failed` error.

## Best Possible Solution

Preserve `/api/v1/whoami` token verification, but add centralized redacted transport diagnostics plus login-specific proxy and `clawhub login --token` fallback guidance with regression tests and troubleshooting docs.

## Reproduction Assessment

Yes for a source-level path: inject an Undici-style `TypeError('fetch failed')` or a non-zero curl result into the HTTP path used by `cmdLogin`, and current main prints only `formatError(error)`. I did not recreate the reporter's exact network environment.

## Solution Assessment

Yes. A shared, redacted HTTP/error-formatting repair with login-specific guidance is the narrowest maintainable fix because login, search, install, publish, package, and sync already converge through the same boundary.
