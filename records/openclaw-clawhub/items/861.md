## Summary

Keep open: current main still has a source-supported CLI login path where browser-auth token verification can fail through shared HTTP transport and surface only the generic `fetch failed` message; proxy support and docs help one class of failures but do not solve the generic diagnostics gap.

## What This Changes

The issue reports `npx clawhub login` opening browser auth and then failing with the generic `fetch failed` message.

## Best Possible Solution

Keep browser-token verification intact, but add centralized redacted transport diagnostics plus login-specific proxy and `clawhub login --token` fallback guidance with regression coverage.

## Reproduction Assessment

Yes, at source level: inject an Undici-style `TypeError('fetch failed')` or equivalent transport failure into the HTTP path used by browser-token verification, and current main routes it through `formatError(error)`. I did not recreate the reporter's exact network environment.

## Solution Assessment

Yes. The narrow maintainable repair is shared sanitized HTTP/error formatting plus login-specific docs and tests; it should not bypass `/api/v1/whoami` verification or save unverified browser tokens.
