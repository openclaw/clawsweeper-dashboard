## Summary

Keep this open under ClawHub issue cleanup policy. Current main still has a source-supported CLI login diagnostics gap: browser auth hands off to token verification, and a thrown fetch error can still surface as only the generic top-level `fetch failed` message.

## What This Changes

The issue reports `npx clawhub login` opening browser auth and then failing with `fetch failed`.

## Best Possible Solution

Keep token verification intact, but add login-specific sanitized network-cause diagnostics, proxy and `clawhub login --token` fallback guidance, regression tests, and troubleshooting docs.

## Reproduction Assessment

No, not as a high-confidence live reproduction because the reporter's network environment is missing. Source inspection does give a focused reproduction path by making the post-browser `/api/v1/whoami` verification fetch throw `TypeError('fetch failed')`.

## Solution Assessment

Yes: a narrow CLI diagnostics/docs/test fix is the best direction because it improves the reported failure while preserving token verification and existing auth semantics.
