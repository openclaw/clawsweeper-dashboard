## Summary

Keep open. Current main still has a narrow CLI login diagnostics gap where browser auth can hand off a token, then `/api/v1/whoami` verification can fail with only the generic top-level `fetch failed` message.

## What This Changes

The issue reports `npx clawhub login` opening browser auth and then failing with `fetch failed`.

## Best Possible Solution

Keep token verification intact, but add sanitized login-specific network diagnostics plus proxy and `clawhub login --token` fallback guidance with focused regression coverage.

## Repair Work Prompt

Fix the ClawHub CLI login diagnostics gap where browser auth receives a token, then `/api/v1/whoami` verification fails with a generic `fetch failed`. Keep verification mandatory and do not save unverified tokens. Add a sanitized helper or command-level handling so login verification fetch failures include useful network/proxy guidance and mention `clawhub login --token` as a fallback without leaking tokens, Authorization headers, callback URLs, or full request details. Update troubleshooting docs for this exact post-browser verification path and add focused regression tests for the login verification fetch failure and error formatting behavior. Do not broaden auth semantics or add new config flags.
