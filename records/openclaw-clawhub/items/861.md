## Summary

Keep open: current main still has a source-supported CLI login path where browser auth can succeed, `/api/v1/whoami` token verification can fail at transport level, and the user-facing output can remain the generic top-level `fetch failed` message.

## What This Changes

The issue reports that `npx clawhub login` opens browser auth and then fails with a generic `fetch failed` error.

## Best Possible Solution

Keep `/api/v1/whoami` verification before config writes, but add centralized redacted transport diagnostics plus login-specific remediation guidance and regression coverage.

## Reproduction Assessment

Yes for a source-level path: inject an Undici-style `TypeError('fetch failed')` or Bun curl failure into the shared HTTP path used by `cmdLogin`, and current main prints only `formatError(error)`. I did not recreate the reporter's exact network environment.

## Solution Assessment

Yes. A shared, redacted HTTP/error-formatting repair with login-specific guidance is the narrowest maintainable fix because login, search, install, publish, package, and sync already converge through the same boundary.

## Repair Work Prompt

Fix the ClawHub CLI generic `fetch failed` diagnostics, especially `clawhub login` after browser auth and the shared search/token HTTP paths. Keep `cmdLogin` verifying `/api/v1/whoami` before `writeGlobalConfig`; do not save unverified browser tokens and do not change registry discovery, request semantics, retry behavior, HTTP status handling, proxy selection, or Bun curl behavior. Implement centralized redacted transport diagnostics in `packages/clawhub/src/http.ts` and/or `packages/clawhub/src/cli/ui.ts` for Node/Undici `TypeError('fetch failed')` with nested causes, DNS/TLS/proxy/timeout/abort failures, and non-zero curl stderr. Redact Authorization headers, bearer/API tokens, `clh_` tokens, query/hash token values, callback URLs, proxy credentials, and private endpoints before displaying details. Add login-specific guidance for proxy/firewall checks and `clawhub login --token` / device login fallback where appropriate. Add focused regression coverage in the likely test files and update troubleshooting docs for the browser-auth verification fetch failure. Include a changelog entry if repo policy treats the CLI diagnostics change as user-facing release work.
