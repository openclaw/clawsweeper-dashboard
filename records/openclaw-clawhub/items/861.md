## Summary

Current main removes the reported browser-callback login path: `clawhub login` now defaults to device-code login, and the old `/cli/auth` token handoff page no longer mints callback tokens. The broader raw `fetch failed` transport-diagnostics concern remains better tracked separately at https://github.com/openclaw/clawhub/issues/651.

## What This Changes

The issue reports `npx clawhub login` opening `https://clawhub.ai/cli/auth?...` and then failing with a generic `fetch failed` error.

## Best Possible Solution

Keep the current device-code login implementation, ship it in the next CLI npm release, and leave the broader shared transport-diagnostics work to https://github.com/openclaw/clawhub/issues/651.

## Reproduction Assessment

No current-main reproduction remains for the reported browser-auth symptom: source inspection shows `clawhub login` now uses device-code login instead of opening `/cli/auth`. Historical v0.21.0 source and the linked OpenClaw report explain how the old `Opening browser ... fetch failed` path occurred.

## Solution Assessment

Yes. Moving default CLI login to device-code flow and retiring the token callback route removes the failing browser loopback handoff while preserving explicit token verification.
