## Summary

This issue should stay open: current `main` still has a source-supported CLI login path where browser auth reaches `/api/v1/whoami` verification and can surface a raw `fetch failed` transport error, and ClawHub cleanup policy does not auto-close issues without a stronger terminal reason.

## What This Changes

The issue reports `npx clawhub login` opening browser auth and then failing with a generic `fetch failed` message.

## Best Possible Solution

Preserve `/api/v1/whoami` token verification, but add centralized redacted transport diagnostics plus login-specific proxy and `clawhub login --token` fallback guidance with regression coverage and troubleshooting docs.

## Reproduction Assessment

Yes for a source-level path: inject an Undici-style `TypeError('fetch failed')` or Bun curl failure into the shared HTTP path used by `cmdLogin`, and current `main` prints only `formatError(error)`. The reporter's exact network environment was not recreated live.

## Solution Assessment

Yes. A shared, redacted HTTP/error-formatting repair with login-specific guidance is the narrowest maintainable solution because login, search, install, publish, and sync already converge through this boundary.
