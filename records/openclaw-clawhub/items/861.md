## Summary

Keep open: current main still has a source-supported CLI login path where browser-token verification can fail through the shared HTTP transport and surface only the generic `fetch failed` message. The related open report at https://github.com/openclaw/clawhub/issues/651 shows the same raw transport-failure class, so one focused diagnostics/docs/test repair should cover the cluster.

## What This Changes

The issue reports `npx clawhub login` opening browser auth and then failing with `fetch failed` instead of completing login or showing actionable diagnostics.

## Best Possible Solution

Preserve `/api/v1/whoami` token verification, then add centralized redacted transport diagnostics with login-specific guidance, focused regression tests, and troubleshooting docs for the browser-auth verification step.

## Reproduction Assessment

Yes, at source level: inject an Undici-style `TypeError('fetch failed')` or a nonzero curl transport failure into the HTTP path used by `cmdLogin`, and current main routes it through `formatError(error)`. I did not recreate the reporter's exact macOS/network environment.

## Solution Assessment

Yes. The narrow maintainable repair is shared, redacted HTTP/error formatting plus login-specific docs and tests; it should not bypass `/api/v1/whoami` verification or save unverified browser tokens.
