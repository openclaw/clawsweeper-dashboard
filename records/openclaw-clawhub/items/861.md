## Summary

This should stay open: current main still has a source-supported CLI path where browser login reaches `/api/v1/whoami` token verification and can surface a low-level transport failure as the generic `fetch failed` message, and ClawHub cleanup policy does not auto-close issues.

## What This Changes

The issue reports `npx clawhub login` opening browser auth and then failing with a generic `fetch failed` error.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics for shared HTTP failures, with focused login verification guidance, while preserving `/api/v1/whoami` token verification before config writes.

## Reproduction Assessment

Yes for a source-level path: inject an Undici-style `TypeError('fetch failed')` or Bun curl failure into the shared HTTP path used by `cmdLogin`, and current main prints only `formatError(error)`. The reporter's exact network environment was not recreated live.

## Solution Assessment

Yes. A shared, redacted HTTP/error-formatting repair with login-specific guidance is the narrowest maintainable fix because login, search, install, publish, and sync already converge through the same boundary.
