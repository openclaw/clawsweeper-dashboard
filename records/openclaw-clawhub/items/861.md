## Summary

Keep open: current main still has a source-supported CLI login path where browser auth can complete, `/api/v1/whoami` verification can fail through the shared HTTP transport, and the user sees only the generic `fetch failed` message. The related open report at https://github.com/openclaw/clawhub/issues/651 shows the same raw transport-failure class, and no open PR currently owns the fix.

## What This Changes

The issue asks ClawHub to fix or improve `npx clawhub login` when browser auth opens and the CLI then exits with `fetch failed`.

## Best Possible Solution

Keep browser-token verification intact, but add centralized sanitized transport diagnostics plus login-specific proxy, `--device`, and `clawhub login --token` fallback guidance with regression coverage.

## Reproduction Assessment

Yes, at source level: inject an Undici-style `TypeError('fetch failed')` or equivalent transport failure into the HTTP path used by browser-token verification, and current main routes it through `formatError(error)` as a generic message. I did not recreate the reporter's exact network environment.

## Solution Assessment

Yes. The narrow maintainable repair is shared sanitized HTTP/error formatting plus login-specific docs and tests; it should not bypass `/api/v1/whoami` verification or save unverified browser tokens.
