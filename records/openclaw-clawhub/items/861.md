## Summary

Current main still has a source-level path for the reported symptom: browser login can reach token verification, and rejected network requests can ultimately surface as the generic runtime message `fetch failed`. The report is actionable as a narrow error-handling and diagnostics improvement, but it does not identify whether DNS, TLS, proxy, redirect, or endpoint reachability caused the original failure, so the repair should stay root-cause-neutral. The open duplicate-device-code PR addresses a different server-side collision and does not supersede this issue.

## What This Changes

Improve ClawHub CLI login so network failures during browser authentication and token verification produce safe, actionable diagnostics instead of only `fetch failed`.

## Best Possible Solution

Keep the existing browser and device authentication protocols, but convert rejected login requests into operation-specific, sanitized errors that identify the public host and preserve the underlying cause when available, with regression coverage and a real redacted login smoke.

## Reproduction Assessment

Yes at source level: current main has fetch-based login and token-verification paths where a rejected transport request can produce the reported generic message. The reporter's particular DNS, TLS, proxy, or connectivity condition was not reproduced live.

## Solution Assessment

Yes, a narrow sanitized error-handling repair at the existing request boundary is the best direction. A new auth mode, automatic proxy policy, or endpoint fallback would exceed the evidence and require a separate product decision.

Preserve the current browser/device authentication protocols, discovery behavior, token storage, retry policy, and proxy behavior.
