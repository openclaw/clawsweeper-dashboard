## Summary

Keep open: current main moved default CLI login to device flow and retired the old browser callback, but that change is current-main-only relative to the latest npm release and does not prove the reporter’s broader login `fetch failed` problem is fixed because device-code requests, polling, and token verification still rely on network fetches that can surface raw messages.

## What This Changes

The issue reports `npx clawhub login` opening the ClawHub browser auth URL and then failing with a raw `fetch failed` error.

## Best Possible Solution

Keep the device-flow default, update CLI/troubleshooting docs, and add sanitized login-specific fetch diagnostics if current-release retesting still shows raw network failures.

## Reproduction Assessment

No. The old browser-auth path no longer appears to be the default on current main, but source still shows generic network fetch failures can surface during login without a live retest in the affected environment.

## Solution Assessment

Unclear. The new device-flow default may avoid the reported browser-auth failure mode, but it is not enough evidence to close because login still depends on network fetches that can report the same generic message.
