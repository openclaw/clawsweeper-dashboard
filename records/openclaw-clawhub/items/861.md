## Summary

Keep open. Current main still has a source-supported CLI path where browser login receives a token, verifies it with `/api/v1/whoami`, and can surface an Undici `fetch failed` as only the generic top-level message; https://github.com/openclaw/clawhub/issues/651 remains open for the same shared transport-diagnostics gap.

## What This Changes

The issue reports `npx clawhub login` opening browser auth and then failing with `fetch failed`.

## Best Possible Solution

Add redacted, actionable transport diagnostics at the shared HTTP/error boundary and document the login verification fallback path while keeping token verification intact.

## Reproduction Assessment

Yes for a focused source-level path: after browser auth returns a token, make the `/api/v1/whoami` verification fetch throw `TypeError('fetch failed')`; current code reports only `formatError(error)`. I did not reproduce the reporter's exact macOS/network environment.

## Solution Assessment

Yes. The narrow maintainable fix is sanitized transport diagnostics plus login-specific troubleshooting, not bypassing `/api/v1/whoami` or saving unverified tokens.
