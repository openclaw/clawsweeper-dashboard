## Summary

Keep open: ClawHub cleanup policy does not auto-close issues, and current main still has a source-supported CLI login path where browser-token verification can fail through the shared HTTP transport and surface only the generic `fetch failed` message. The related open report at https://github.com/openclaw/clawhub/issues/651 shows the same raw transport-failure class, so one focused diagnostics/docs/test repair should cover the cluster.

## What This Changes

The issue requests investigation of `npx clawhub login` opening browser auth and then failing with `fetch failed`.

## Best Possible Solution

Keep browser-token verification intact, but make transport failures produce centralized, sanitized cause diagnostics plus login-specific proxy and `clawhub login --token` fallback guidance, with regression tests and troubleshooting docs.

## Reproduction Assessment

Yes, at source level: inject an Undici-style `TypeError('fetch failed')` or equivalent transport failure into the HTTP path used by browser-token verification, and current main routes it through `formatError(error)` as a generic message. I did not recreate the reporter's exact network environment.

## Solution Assessment

Yes. The narrow maintainable repair is shared sanitized HTTP/error formatting plus login-specific docs and tests; it should not bypass `/api/v1/whoami` verification or save unverified browser tokens.
