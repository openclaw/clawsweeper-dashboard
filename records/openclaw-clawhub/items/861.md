This issue should stay open. Current main still has a source-supported CLI path where browser login reaches token verification and then renders low-level transport failures through the generic `fetch failed` message, and ClawHub cleanup policy does not auto-close issues.

## What This Changes

The report asks ClawHub to investigate and fix `npx clawhub login` failing after opening browser auth with `fetch failed`.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics for login verification and other shared HTTP failures while preserving token verification before config writes.

## Solution Assessment

Yes. A shared, redacted HTTP/error-formatting repair with focused login guidance is the narrowest maintainable solution because login, search, install, publish, and sync already converge through the same boundary.

Add regression coverage for login verification failure, shared formatter/search behavior, HTTP/Bun curl diagnostics, and redaction. Update troubleshooting or CLI/auth docs with login-verification proxy guidance and token/device fallback where appropriate. Do not add a new CLI flag or weaken token verification.
