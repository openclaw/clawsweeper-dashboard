## Summary

Keep open: current main still has a source-supported CLI login path where browser auth can succeed, `/api/v1/whoami` token verification can fail at transport level, and the CLI can still show only the generic `fetch failed` message. ClawHub cleanup policy also does not auto-close issues that remain actionable.

## What This Changes

The issue asks for `npx clawhub login` to stop failing with a generic `fetch failed` after opening browser auth.

## Best Possible Solution

Keep browser-token verification intact, but add centralized redacted transport diagnostics plus login-specific proxy and `clawhub login --token` fallback guidance, with regression coverage and troubleshooting docs.

## Reproduction Assessment

Yes for a source-level path: inject an Undici-style `TypeError('fetch failed')` or a Bun curl failure into the shared HTTP path used by `cmdLogin`, and current main prints only `formatError(error)`. I did not recreate the reporter's exact network environment.

## Solution Assessment

Yes. A shared, redacted HTTP/error-formatting repair with login-specific guidance is the narrowest maintainable fix because login, search, install, publish, package, and sync already converge through the same boundary.
