## Summary

Current main still turns a network rejection during post-browser token verification into the generic `fetch failed` message, so the reported user problem is not definitely implemented; the issue should remain open for a narrow diagnostics and validation fix.

## What This Changes

Improve the ClawHub CLI login failure path when browser authorization is followed by a generic network `fetch failed` error.

## Best Possible Solution

Keep browser token verification intact, but report the failing authentication stage and a sanitized underlying network cause, clearly suggest the existing device-login fallback, and validate both behavior and secret redaction in a real CLI run.

## Reproduction Assessment

No, not at high confidence: current source clearly permits the reported generic failure during token verification, but the report lacks the environment and nested network error needed to reproduce its exact cause.

## Solution Assessment

Yes, a narrow diagnostics, fallback-guidance, and regression-coverage change is the most maintainable direction; changing authentication semantics or assuming the older HTTP/2 defect is the only cause would be less safe.
