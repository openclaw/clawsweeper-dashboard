## Summary

Keep open. Current main still has a source-supported CLI login path where browser auth can succeed, then token verification against `/api/v1/whoami` can fail through the shared HTTP client and surface only the generic top-level `fetch failed` message.

## What This Changes

The issue reports `npx clawhub login` opening browser authentication and then failing with a generic `fetch failed` message.

## Best Possible Solution

Keep browser-token verification intact, but add centralized redacted transport diagnostics plus login-specific proxy and `clawhub login --token` or device-login guidance with focused tests and troubleshooting docs.

## Reproduction Assessment

Yes for the source-level path: after browser auth returns a token, make the `/api/v1/whoami` verification fetch throw `TypeError('fetch failed')` and current main can still print only `fetch failed`. The reporter's exact network environment was not independently reproduced.

## Solution Assessment

Yes. A shared HTTP/error-formatting diagnostics repair is the narrowest maintainable solution because login, search, install, publish, and related CLI commands already converge through this boundary.
