## Summary

Current release v0.21.0 resolves two onboarding concerns—CLI publishing exists and `.git` is automatically excluded—but it still rejects local pre-upload scans, and the abuse/sandbox/remediation requests remain policy decisions. The narrower scanner gap is already tracked by https://github.com/openclaw/clawhub/issues/1767, but that open item does not dispose of the broader policy scope here.

## What This Changes

Request CLI skill publishing, automatic Git metadata handling, pre-upload security feedback, clearer abuse rules, and a contributor-remediation path for suspicious skills.

## Best Possible Solution

Preserve the shipped CLI publishing and automatic `.git` exclusion, implement the local scanner under the canonical scanner issue after approving its ephemeral-upload security contract, and split abuse, appeal, sandbox, and contributor-remediation rules into a focused moderation-policy item with corresponding specs and public docs.

## Reproduction Assessment

Yes. On current v0.21.0, a local-path scan deterministically reaches `rejectLocalScan()` and fails before scanning; source also confirms CLI publishing and automatic `.git` exclusion already work.

## Solution Assessment

No as one umbrella implementation. The narrow scanner should reuse the hosted scan pipeline through a reviewed ephemeral artifact flow, while enforcement and remediation policy should be decided and documented separately.
