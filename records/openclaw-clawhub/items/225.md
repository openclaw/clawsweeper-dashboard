## Summary

Current main resolves the repository-packaging and abuse-policy questions and provides actionable post-submission scan retrieval, but it intentionally does not provide the requested pre-publication local scan. The issue remains useful if narrowed to browser onboarding and contributor remediation, while the unpublished-source scanner contract stays with the related canonical issue.

## What This Changes

Improve skill-author onboarding with pre-upload validation, repository-packaging guidance, abuse-policy clarity, and contributor remediation for suspicious findings.

## Best Possible Solution

Keep unpublished-source scanning under https://github.com/openclaw/clawhub/issues/1767, and narrow this issue to a browser onboarding flow that explains ignored files, publication state, scan-result access, and a safe fix-and-resubmit path.

## Reproduction Assessment

Yes. On current main, providing a local path to the scan command deterministically reaches an explicit rejection, and the backend independently rejects local-upload scans.

## Solution Assessment

No, not as one umbrella change. The production scanner should remain the single analysis source, with its unpublished-input contract decided in the canonical scanner issue and browser-specific onboarding handled as a focused UX change.
