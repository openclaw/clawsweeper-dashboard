- `P2`: The missing local validation loop is a meaningful but non-emergency publisher workflow improvement with bounded impact.
- `impact:security`: The issue concerns pre-publication security findings, suspicious submissions, remediation, and enforcement boundaries.

## Summary

Current main implements CLI publishing and post-publication scan retrieval, but it still rejects local path scans, so the central request for pre-upload security feedback remains necessary; the enforcement and sandboxing suggestions also require product judgment.

## What This Changes

Improve publisher onboarding with CLI uploads, local pre-upload security feedback, Git-repository import guidance, and clearer abuse/remediation handling.

## Best Possible Solution

Keep the existing publish workflow, implement authenticated ephemeral local scanning through the canonical scanner work, clarify whether repository metadata such as `.git` is excluded, and document that good-faith remediation uploads are distinct from repeated clearly abusive submissions; use existing hidden or review states unless maintainers define a complete sandbox contract.

## Reproduction Assessment

Yes at source level: `clawhub scan <local-path>` reaches an explicit local-scan rejection on current main, while published-package scans are supported. This is a missing capability rather than a regression in an established contract.

## Solution Assessment

No as one umbrella implementation: the local scanner should follow the focused authenticated, ephemeral scan design, while repository exclusions and remediation/enforcement semantics should be handled through narrow documentation and product-policy decisions.
