## Label Justifications

- `P1`: The issue affects a core GitHub sign-in/account recovery workflow for multiple real users after account deletion.
- `impact:security`: Any recovery or relinking flow changes identity binding, deletion privacy, ban separation, and credential revocation semantics.
- `impact:auth-provider`: The reported failure is in GitHub OAuth completion and Convex Auth user binding after deletion.
- `impact:session-state`: Affected users return from OAuth without an active session and remain in a logged-out loop.

## Summary

Keep open. Current main still deliberately blocks same-GitHub sign-in after user-initiated deletion, and allowing re-signup/relinking is a product and security decision rather than conservative cleanup.

## What This Changes

The issue requests a way to re-sign up or regain GitHub sign-in after deleting a ClawHub account instead of being returned to the login flow.

## Best Possible Solution

Maintainers should choose and document the account deletion policy, then either implement a secure same-GitHub recreate/relink flow or keep irreversible deletion while making the OAuth result and support path explicit.

## Reproduction Assessment

Yes, by source inspection: deleteAccount sets deactivatedAt/purgedAt and handleDeletedUserSignIn rejects deactivated or legacy deleted users during OAuth completion. I did not run a live GitHub OAuth flow or inspect production rows.

## Solution Assessment

No for an automatic cleanup fix. The maintainable path needs a maintainer-approved deletion policy: either a secure relink/recreate flow or an explicit irreversible-deletion UX and support path.
