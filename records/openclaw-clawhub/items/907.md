## Summary

Keep open. Current main still intentionally blocks same-GitHub re-signup after user-initiated deletion; the old silent-loop symptom is partly mitigated by user-facing auth errors, but the requested recovery or relink behavior needs maintainer product and security review.

## What This Changes

Request to let a user sign up or sign in again with the same GitHub account after deleting a previous ClawHub account, instead of getting stuck after GitHub OAuth.

## Best Possible Solution

Maintainers should choose and document the account deletion policy, then either implement a secure audited same-GitHub recreate/relink flow or keep deletion irreversible with an explicit OAuth/support path.

## Reproduction Assessment

Yes, by source inspection for the blocked re-signup path: deleteAccount deactivates and purges the user, handleDeletedUserSignIn rejects deactivated or legacy deleted users, and tests assert that rejection. I did not reproduce the old silent browser OAuth loop in a live environment; current main now has a visible blocked-auth error path.

## Solution Assessment

No as an automatic fix: allowing same-GitHub re-signup would change the current irreversible-deletion contract. The maintainable path is a maintainer-approved product/security policy followed by focused auth, user lifecycle, docs/spec, and regression-test changes.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes identity reuse, deletion guarantees, OAuth bindings, ownership, and ban-bypass protections.

Concerns:

- **[medium] Define recovery authority before changing sign-in:** `specs/security-moderation.md:141`
  - body: The spec currently says user-initiated deletion is irreversible and deleted accounts cannot be restored by logging in again, so restore, relink, or recreate behavior needs explicit policy approval before implementation.
  - confidence: 0.95
- **[medium] Preserve ban and identity safeguards:** `convex/auth.ts:37`
  - body: The OAuth callback checks deleted/deactivated state, mismatched existingUserId, and ban audit records before allowing sign-in; any recovery path needs equivalent identity matching and abuse-prevention checks.
  - confidence: 0.9
- **[medium] Preserve purge and token semantics:** `convex/users.ts:338`
  - body: Account deletion revokes API tokens and clears profile/contact/GitHub-derived fields, so recovery must define whether credentials, private profile data, publishers, packages, skills, and audit history are restored or recreated.
  - confidence: 0.9
