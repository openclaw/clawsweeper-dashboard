## Summary

Keep open. Current main intentionally blocks same-GitHub sign-in after user-initiated deletion and only improves the error message; for ClawHub issues this needs maintainer product/security judgment rather than cleanup closure.

## What This Changes

The issue asks ClawHub to allow re-signup or sign-in with the same GitHub account after a previous account deletion instead of leaving the user in an OAuth loop.

## Best Possible Solution

Maintainers should decide and document the account deletion policy; if re-signup is allowed, implement a secure same-GitHub recreate or relink flow, otherwise keep deletion irreversible and make the blocked-account/support path explicit.

## Reproduction Assessment

Yes, source-reproducible: current main rejects deleted/deactivated users during OAuth completion, and auth tests assert that behavior including fresh login of a legacy deleted user. I did not run a live GitHub OAuth browser flow, so the exact visual page-refresh loop is not runtime-reproduced here.

## Solution Assessment

No, not as a narrow automatic bug fix. The requested re-signup path changes deletion and OAuth binding semantics, so the maintainable solution starts with maintainer product/security judgment.
