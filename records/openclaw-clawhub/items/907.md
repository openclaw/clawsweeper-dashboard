## Label Justifications

- `P1`: The issue affects a core GitHub sign-in/account recovery workflow for multiple real users after account deletion.
- `impact:security`: Any recovery or relinking flow changes identity binding, deletion privacy, ban separation, and credential revocation semantics.
- `impact:auth-provider`: The reported failure occurs during GitHub OAuth completion and Convex Auth user binding after deletion.
- `impact:session-state`: Affected users return from OAuth without an active session and remain logged out.

## Summary

Keep open. Current main still does not provide same-GitHub re-signup or relinking after account deletion; the implementation now treats user deletion as irreversible and rejects deleted accounts during OAuth, so the remaining change needs maintainer product and security review rather than conservative cleanup.

## What This Changes

The issue requests that a user who deleted their ClawHub account can sign up or sign in again with the same GitHub account instead of returning to a logged-out loop.

## Best Possible Solution

Maintainers should decide and document the deletion policy, then either implement a secure same-GitHub recreate or relink flow that preserves abuse protections, or keep irreversible deletion while making the OAuth result and support path explicit.

## Reproduction Assessment

Yes, by source inspection: deleteAccount sets deactivatedAt/purgedAt and handleDeletedUserSignIn rejects deactivated or legacy deleted users during OAuth completion. I did not run a live GitHub OAuth flow or inspect production rows.

## Solution Assessment

No for an automated cleanup fix. The maintainable path requires a maintainer-approved deletion policy: either a secure relink/recreate flow or an explicit irreversible-deletion UX and support path.
