## Summary

Keep open. Current main still blocks same-GitHub sign-in after user-initiated account deletion; the old silent-loop symptom is partly mitigated by visible auth errors, but re-signup or relinking would change account-deletion and identity policy.

## What This Changes

The issue requests the ability to re-sign up or log in with the same GitHub account after deleting a ClawHub account instead of getting stuck in the OAuth sign-in loop.

## Best Possible Solution

Maintainers should choose and document the deletion policy, then either implement a secure audited same-GitHub recreate/relink flow or keep deletion irreversible with a clear OAuth/support outcome.

## Reproduction Assessment

Yes, by source inspection for the blocked re-signup path: deleteAccount deactivates and purges the user, handleDeletedUserSignIn rejects deactivated or legacy deleted users, and tests assert that rejection. I did not run a live browser OAuth flow, and current main now has a visible blocked-auth error path rather than only a silent loop.

## Solution Assessment

No as an automatic fix: same-GitHub re-signup or relinking changes the current irreversible-deletion contract. The maintainable path is a maintainer-approved product/security policy followed by focused auth, user lifecycle, docs/spec, and regression-test changes.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes identity reuse, OAuth bindings, deletion guarantees, and ban-bypass protections.

Concerns:

- **[medium] Define recovery authority before changing sign-in:** `specs/security-moderation.md:141`
  - body: The spec currently says user-initiated deletion is irreversible and deleted accounts cannot be restored by logging in again, so restore, relink, or recreate behavior needs explicit policy approval before implementation.
  - confidence: 0.95
- **[medium] Preserve OAuth identity and ban safeguards:** `convex/auth.ts:37`
  - body: The OAuth callback checks deleted/deactivated state, mismatched existingUserId, and ban audit records before allowing sign-in; any recovery path needs equivalent identity matching and abuse-prevention checks.
  - confidence: 0.9
- **[medium] Preserve purge and token semantics:** `convex/users.ts:338`
  - body: Account deletion revokes API tokens and clears profile, contact, and GitHub-derived fields, so recovery must define whether credentials, private profile data, publishers, packages, skills, and audit history are restored or recreated.
  - confidence: 0.9
