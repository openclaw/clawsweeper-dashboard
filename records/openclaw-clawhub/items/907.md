## Summary

Keep open. Current main intentionally rejects GitHub sign-in after user-initiated deletion and now surfaces a blocked-account message, but the requested same-GitHub re-signup/relinking path remains a product and security decision rather than a cleanup-close candidate.

## What This Changes

The issue requests allowing a user who deleted a ClawHub account to sign up or sign in again with the same GitHub account instead of refreshing or looping.

## Best Possible Solution

Maintainers should decide and document the deletion policy; if re-signup is allowed, implement a secure same-GitHub recreate/relink flow, otherwise keep irreversible deletion and make the OAuth/support outcome explicit.

## Reproduction Assessment

Yes, from source inspection: current main explicitly rejects deleted or deactivated users during OAuth completion, and tests assert that behavior. I did not run a live GitHub OAuth flow, so the browser loop itself is source-reproducible rather than runtime-reproduced.

## Solution Assessment

No, not as a narrow bug fix yet. The requested re-signup path changes deletion and OAuth binding semantics, so the maintainable solution starts with a maintainer product/security decision before code changes.
