## Summary

Keep open. Current main still intentionally blocks same-GitHub sign-in after user-initiated deletion and only surfaces an error, while ClawHub cleanup policy keeps issues open unless maintainers decide the account-lifecycle policy.

## What This Changes

The issue asks ClawHub to allow re-signup or recovery after deleting an account and then signing in again with the same GitHub identity.

## Best Possible Solution

Maintainers should choose and document the deletion policy, then either implement a secure same-GitHub recreate/relink flow or keep deletion irreversible with a clear OAuth/support path.

## Reproduction Assessment

Yes for the account lockout: current main source shows deletion writes deactivatedAt/purgedAt and OAuth completion rejects those users, with tests asserting rejection. I did not run a live GitHub OAuth browser flow, so the exact historical silent refresh loop is not runtime-reproduced.

## Solution Assessment

No as an automatic bug fix. Same-GitHub re-signup changes the current irreversible-deletion policy, so the maintainable path starts with maintainer product and security approval.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes identity reuse, deletion guarantees, token revocation, and ownership behavior.

Concerns:

- **[medium] Define recovery policy before changing auth:** `specs/security-moderation.md:130`
  - body: The spec currently says user-initiated deletion is irreversible and deleted accounts cannot be restored by logging in again, so restore/relink/recreate behavior needs explicit maintainer approval.
  - confidence: 0.95
- **[medium] Preserve identity and ban safeguards:** `convex/auth.ts:37`
  - body: The OAuth callback checks deleted/deactivated state and guards mismatched existingUserId before blocking, so any recovery path needs equivalent identity and ban/autoban protections.
  - confidence: 0.9
- **[medium] Define token and ownership semantics:** `convex/users.ts:338`
  - body: Account deletion revokes tokens and purges profile fields while published content remains public, so recovery must define whether tokens, publishers, packages, skills, and audit history are restored or recreated.
