## Summary

Keep open. Current main still treats user-initiated deletion as irreversible and rejects same-GitHub sign-in afterward; for openclaw/clawhub, issue cleanup should not auto-close this, and the remaining work needs a product/security decision rather than conservative closeout.

## What This Changes

The issue requests that users who deleted a ClawHub account can sign up or sign in again with the same GitHub account instead of looping after OAuth.

## Best Possible Solution

Maintainers should choose and document the deletion policy, then either implement a secure same-GitHub recreate/relink flow or keep deletion irreversible with a clear OAuth/support path that avoids a silent loop.

## Reproduction Assessment

Yes, by source inspection: current main rejects deleted/deactivated users during GitHub OAuth completion and tests assert that rejection. I did not run a live OAuth browser flow, so the exact old page-refresh symptom was not directly reproduced here.

## Solution Assessment

No as an automatic fix: allowing same-GitHub re-signup would change the documented irreversible-deletion contract and auth binding behavior. The maintainable path is a maintainer-approved account lifecycle policy followed by focused auth, user deletion, docs/spec, and regression-test changes.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes identity reuse, deletion guarantees, OAuth bindings, ownership, and ban-bypass protections.

Concerns:

- **[medium] Define recovery authority before changing sign-in:** `specs/security-moderation.md:180`
  - body: The spec currently says user-initiated deletion is irreversible and deleted accounts cannot be restored by logging in again, so restore, relink, or recreate behavior needs explicit policy approval before implementation.
  - confidence: 0.95
- **[medium] Preserve identity and ban safeguards:** `convex/auth.ts:40`
  - body: The OAuth callback checks deleted/deactivated state, mismatched existingUserId, and ban audit records before allowing sign-in; any recovery path needs equivalent identity matching and abuse-prevention checks.
  - confidence: 0.9
- **[medium] Define purge and token semantics:** `convex/users.ts:475`
  - body: Account deletion revokes API tokens and clears profile/contact/GitHub-derived fields, so recovery must define whether credentials, private profile data, publishers, packages, skills, and audit history are restored or recreated.
