## Label Justifications

- `P1`: The issue affects a core GitHub sign-in/account recovery workflow for multiple real users after account deletion.
- `impact:security`: Any recovery or relinking path changes identity binding, deletion privacy, ban separation, and credential revocation semantics.
- `impact:auth-provider`: The reported failure occurs during GitHub OAuth completion and Convex Auth user binding after deletion.
- `impact:session-state`: Affected users return from OAuth without an active ClawHub session and remain logged out.

## Summary

Keep open. Current main still treats user-initiated deletion as irreversible and blocks the same GitHub identity during OAuth completion; changing that behavior would require a maintainer product and security decision, and ClawHub cleanup policy does not auto-close issues.

## What This Changes

The reporter wants to sign up or sign in again with the same GitHub account after deleting the previous ClawHub account instead of landing in a refresh/sign-in loop.

## Best Possible Solution

Maintainers should decide the account deletion policy, then either implement a secure same-GitHub recreate/relink flow that preserves abuse protections and ownership invariants or make the irreversible-deletion/support path explicit enough that users are not left in a loop.

## Reproduction Assessment

Yes, by source inspection: deleteAccount sets deactivatedAt/purgedAt and handleDeletedUserSignIn rejects that user during OAuth completion. I did not run a live GitHub OAuth flow or inspect production rows.

## Solution Assessment

No for an automated cleanup fix. The maintainable path requires a maintainer-approved deletion policy and security design before changing re-registration or relinking behavior.
