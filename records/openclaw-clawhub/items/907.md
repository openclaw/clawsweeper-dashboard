## Label Justifications

- `P1`: The issue affects a core GitHub sign-in/account recovery workflow for multiple real users after account deletion.
- `impact:security`: Any recovery or relinking path changes identity binding, deletion privacy, ban separation, and credential revocation semantics.
- `impact:auth-provider`: The failure occurs during GitHub OAuth completion and Convex Auth user binding after deletion.
- `impact:session-state`: Affected users return from OAuth without an active ClawHub session and remain logged out.

## Summary

Keep open. Current main still blocks same-GitHub sign-in after user-initiated deletion, and allowing re-signup would change the documented irreversible-deletion policy, so this needs maintainer product and security review rather than conservative cleanup.

## What This Changes

The issue requests the ability to sign up or sign in again with the same GitHub account after deleting the previous ClawHub account.

## Best Possible Solution

Maintainers should decide and document the deletion policy, then either implement a secure same-GitHub relink/recreate flow that preserves abuse protections or make irreversible deletion and the OAuth support path explicit.

## Reproduction Assessment

Yes, by source inspection: deleteAccount sets deactivatedAt/purgedAt and handleDeletedUserSignIn rejects that user during OAuth completion. I did not run a live GitHub OAuth flow or inspect production rows.

## Solution Assessment

No for an automated cleanup fix. The maintainable path requires a maintainer-approved deletion policy and security design before changing re-registration or relinking behavior.
