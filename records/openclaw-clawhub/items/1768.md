## Summary

Keep open. Current main still lacks native per-skill pinning, and #1806 is already the open unmerged implementation candidate with closing syntax, so this issue should stay open for that PR’s maintainer review rather than be closed or duplicated.

## What This Changes

The issue requests native CLI per-skill pinning so targeted, forced, and all-skill updates honor an operator freeze.

## Best Possible Solution

Review and land #1806, or an equivalent maintainer-approved patch, so pin/unpin is lockfile-backed, visible in list output, and enforced for direct, forced, and all-skill updates.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an installed skill lock entry has only `version`/`installedAt`, and `cmdUpdate` can reach `rm`, download, extract, and lockfile rewrite without any pin check; I did not run a real update because that would mutate the checkout/environment.

## Solution Assessment

Yes. The requested lockfile-backed pin/unpin surface is the narrow maintainable fix because it puts the freeze decision in the same state that update already reads, while still allowing `update --all` to proceed for unpinned skills.
