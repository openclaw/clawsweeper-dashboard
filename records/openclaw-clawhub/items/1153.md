## Summary

Keep open for maintainer/operator handling. Current main still treats owner-facing deletion as soft-delete while preserving version rows, and the request is an irreversible production data action that needs ownership verification, purge scope, and coordination with related open request #1167.

## What This Changes

The requester asks maintainers to hard-delete all versions of `JasonStarlight/phosor-ai` so the skill can be republished from a clean `v1.0.0`.

## Best Possible Solution

A maintainer should verify requester ownership of the production owner account and target skill, then either run the existing audited admin hard-delete flow with an explicit purge scope or design a documented owner reset/version-purge workflow if ClawHub wants to support this regularly.

## Reproduction Assessment

Not applicable as a source-level bug reproduction; this is a production data-deletion request. The source path is clear: public and CLI deletion soft-delete the skill, while existing `skillVersions` rows still block republishing the same version.

## Solution Assessment

No, an automated hard-delete from this issue is not the best way to solve it. The maintainable path is operator verification first, followed by an audited admin purge or an intentional product workflow for safe version reset.
