## Summary

Keep open for maintainer/operator handling. Current source still makes owner-facing CLI/API deletion a soft-delete, same-version republishes are blocked while old version rows remain, and the available hard-delete path is admin-only and destructive; ClawHub policy also keeps issues open rather than auto-closing them.

## What This Changes

The issue requests hard-deleting all published versions of `JasonStarlight/phosor-ai` so the owner can republish from a clean `v1.0.0`.

## Best Possible Solution

A maintainer should verify requester ownership and the exact purge scope, then use the existing admin hard-delete path if that scope is acceptable or design a documented owner reset/version-purge workflow for recurring cases.

## Reproduction Assessment

Yes for the source-level behavior: current main documents and implements owner deletion as a soft-delete, preserves `skillVersions`, and rejects republishing an existing version. The production account ownership and actual purge cannot be reproduced from the checkout.

## Solution Assessment

No automated source fix is the best immediate solution. The narrow maintainable path is operator verification plus the existing admin-only hard-delete if appropriate; any owner self-service reset would need a product/security policy decision first.
