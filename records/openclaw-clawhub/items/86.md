## Summary

Current main does not implement the central request: ClawHub can categorize existing packages as Automation, but its publishing and installation contracts remain limited to skills, code plugins, and bundle plugins. This remains a valid product proposal requiring maintainer decisions about the artifact model and scheduled-execution trust boundary; the related UI report at https://github.com/openclaw/clawhub/issues/1112 does not supersede it.

## What This Changes

Add a browsable, previewable, and installable catalog of reusable OpenClaw cron job templates.

## Best Possible Solution

Define a declarative cron-template package contract that reuses ClawHub discovery and verification, paired with an explicit OpenClaw preview-and-confirm import API that never silently enables recurring actions.

## Reproduction Assessment

Not applicable: this requests a new artifact type, catalog, and installation workflow rather than reporting broken existing behavior.

## Solution Assessment

No, the high-level one-click proposal is not yet a sufficiently safe implementation plan; the narrowest maintainable direction is to reuse package infrastructure while defining a declarative, previewable runtime import boundary.

## Risks / Open Questions

- A one-click import could schedule repeated privileged or externally visible agent actions, so untrusted templates require explicit preview, capability disclosure, operator consent, moderation, and safe handling of credentials and destinations.
- A separate automation registry could duplicate existing package discovery, verification, ownership, and moderation systems unless the product model deliberately reuses them.
- The OpenClaw runtime currently lacks an established ClawHub-facing contract for importing declarative scheduled jobs, so implementing the catalog alone would not solve the requested installation workflow.
