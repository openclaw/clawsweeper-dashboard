## Label Justifications

- `P1`: The issue blocks affected users from signing in after account deletion and is clustered across multiple active reports.
- `impact:security`: Deleted-account recovery changes deletion guarantees, auditability, token revocation, and ban-bypass boundaries.
- `impact:auth-provider`: The failure occurs during GitHub OAuth reauthorization and identity reuse for a deleted ClawHub account.
- `impact:session-state`: OAuth can complete while ClawHub still refuses to establish an active usable session for the affected identity.

## Summary

Keep open. Current main no longer treats this as an unexplained loading loop, but it still intentionally blocks deleted-account GitHub OAuth re-entry, so the remaining request is a security-sensitive account lifecycle/product decision rather than a cleanup-close candidate for a ClawHub issue.

## What This Changes

The issue asks ClawHub to let a previously deleted GitHub-linked account reauthorize and regain access, or otherwise reset the deleted account state so sign-in no longer loops.

## Best Possible Solution

Maintainers should choose and document the deleted-account lifecycle, then implement either an audited recovery/re-registration/admin-reset path or keep deletion irreversible with a clear post-OAuth error and support workflow.

## Reproduction Assessment

Yes, at the source level: deletion marks the user deactivated and purged, OAuth re-entry calls handleDeletedUserSignIn and rejects that record, and active-user helpers hide the deleted identity from session state. I did not run a live production OAuth flow or inspect the reporter's database row.

## Solution Assessment

No, the requested hard-delete or automatic recreate path is not yet clearly the best fix because it bypasses unresolved deletion, privacy, audit, token, and ban policy. The maintainable solution is an explicit account-lifecycle decision plus audited implementation or support tooling.
