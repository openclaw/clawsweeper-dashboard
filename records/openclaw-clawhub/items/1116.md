## Label Justifications

- `P1`: The issue blocks affected users from signing in or publishing after account deletion and is clustered across multiple reports.
- `impact:security`: Deleted-account recovery changes deletion guarantees, auditability, token revocation, and ban-bypass boundaries.
- `impact:auth-provider`: The failure occurs during GitHub OAuth reauthorization and identity reuse for a deleted ClawHub account.
- `impact:session-state`: OAuth can complete while ClawHub does not establish an active usable session for the affected identity.

## Summary

Keep open. Current main still blocks deleted/deactivated ClawHub accounts during GitHub OAuth re-entry, and the remaining ask is a clustered account-lifecycle and support-policy decision rather than a conservative cleanup close.

## What This Changes

The issue requests allowing a user who deleted a ClawHub account to sign back in or re-register with the same GitHub OAuth identity instead of looping or remaining unauthorized.

## Best Possible Solution

Maintainers should choose and document the deleted-account lifecycle, then implement either a controlled audited recovery/re-registration/admin-reset path or keep terminal deletion with clear OAuth and support messaging.

## Reproduction Assessment

Yes, at the source level: deleteAccount sets deactivatedAt/purgedAt, OAuth re-entry calls handleDeletedUserSignIn and rejects deleted/deactivated records, and active-user helpers return no user for those records. I did not run a live production OAuth flow or inspect the reporter's database row.

## Solution Assessment

No, a one-off hard delete or automatic recreate is not the best immediate source fix because it bypasses unresolved deletion, privacy, audit, token, and ban policy. The maintainable path is an explicit account-lifecycle decision plus audited implementation or support tooling.
