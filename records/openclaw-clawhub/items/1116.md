## Summary

Keep open. Current main treats deleted ClawHub accounts as permanently deactivated and blocks GitHub OAuth reauth instead of recreating or relinking the account; this remains a clustered account-lifecycle/support problem, and the ClawHub cleanup policy keeps issues open for maintainer resolution rather than auto-closing them.

## What This Changes

The issue asks ClawHub to let a user recover, reset, or re-register after deleting an account and later authorizing the same GitHub identity again.

## Best Possible Solution

Maintainers should decide and document the post-deletion lifecycle: keep permanent deletion with clearer support UX, provide a verified administrative purge/reset process, or implement a guarded same-provider re-registration flow with coverage for active, deleted, legacy-deleted, banned/autobanned, and mismatched-identity cases.

## Reproduction Assessment

Yes for the current blocked reauthorization path by source inspection: deletion writes deactivatedAt/purgedAt, OAuth rejects that identity, and active-user helpers hide it. The original silent infinite-loading/no-error symptom is less certain on current main because the OAuth UI now surfaces blocked/no-session errors.

## Solution Assessment

No for an automated account reset or fresh-account recreation. The requested recovery behavior conflicts with the current irreversible-deletion contract, so restore, purge, recreate, relink, or support-reset behavior needs explicit maintainer security and product approval first.
