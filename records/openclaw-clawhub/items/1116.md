## Label Justifications

- `P1`: The clustered issue blocks affected users from signing in to ClawHub after account deletion and needs maintainer direction.
- `impact:security`: Deleted-account recovery changes deletion guarantees, auditability, token revocation, and ban-bypass boundaries.
- `impact:auth-provider`: The failure occurs during GitHub OAuth reauthorization and identity reuse for a deleted ClawHub account.
- `impact:session-state`: OAuth can complete while ClawHub refuses to establish an active usable session for the affected identity.

## Summary

Keep open. Current main no longer looks like a pure infinite-spinner bug because it now maps deleted-account auth failures to a user-facing error, but it still intentionally rejects deleted-account GitHub re-entry and has no audited re-registration or admin-reset path; ClawHub issue cleanup policy keeps this open for maintainer decision.

## What This Changes

The issue asks for a deleted ClawHub account to be reset or recreated so the same GitHub identity can sign in again after OAuth reauthorization.

## Best Possible Solution

Maintainers should choose the deleted-account lifecycle, then implement either controlled audited re-registration/admin reset or keep deletion irreversible with explicit support guidance while preserving token revocation, data purge, and ban protections.

## Reproduction Assessment

Yes, at the source level. Deletion sets deactivatedAt/purgedAt, OAuth re-entry calls handleDeletedUserSignIn and rejects those rows, and active-user helpers hide deactivated users; I did not run a live production OAuth flow or inspect the reporter's database row.

## Solution Assessment

No, a one-off hard delete or automatic recreate is not the safest source-level fix without a policy decision. The maintainable path is an explicit account-lifecycle decision plus audited recovery, reset, or clearer irreversible-deletion support behavior.
