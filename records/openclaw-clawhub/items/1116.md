## Summary

Keep open. Current main still treats user-initiated ClawHub deletion as terminal during GitHub OAuth; it now surfaces a clearer blocked-account error, but it does not provide the requested reset, purge, re-registration, or relinking path, and ClawHub cleanup policy keeps issues open for maintainer handling.

## What This Changes

The issue requests a hard reset or recovery path so a GitHub account can sign in again after its linked ClawHub account was deleted.

## Best Possible Solution

Maintainers should choose and document a canonical deleted-account recovery policy, then implement it consistently across Convex Auth, provider identity binding, admin/support tooling, UI copy, docs, and regression tests.

## Reproduction Assessment

Yes, from source inspection rather than a live OAuth run: deleting an account sets `deactivatedAt`/`purgedAt`, the GitHub OAuth callback calls `handleDeletedUserSignIn`, and active auth helpers then expose no signed-in user. Current tests assert this rejection path.

## Solution Assessment

No for an automated source-only fix. The requested purge, recreate, or relink behavior conflicts with the current irreversible-deletion contract and needs maintainer product and security approval first.
