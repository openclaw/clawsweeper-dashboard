## Label Justifications

- `P1`: The issue blocks affected users from signing in after account deletion and is clustered across multiple active reports.
- `impact:security`: Deleted-account recovery changes deletion guarantees, auditability, token revocation, and ban-bypass boundaries.
- `impact:auth-provider`: The failure occurs during GitHub OAuth reauthorization and identity reuse for a deleted ClawHub account.
- `impact:session-state`: OAuth can complete while ClawHub still refuses to establish an active usable session for the affected identity.

## Summary

Keep open. Current main still treats deleted ClawHub accounts as permanently deactivated during GitHub OAuth re-entry, and repository cleanup policy keeps issues open for maintainer resolution rather than auto-closing them.

## What This Changes

The issue asks ClawHub to let a user sign in again, or get an administrative reset, after deleting a ClawHub account tied to the same GitHub identity.

## Best Possible Solution

Maintainers should explicitly choose the deleted-account lifecycle, then implement either an audited recovery/re-registration/admin-reset path or keep deletion irreversible with clear OAuth error handling and support guidance.

## Reproduction Assessment

Yes, at the source level: deletion sets deactivatedAt/purgedAt, OAuth re-entry calls handleDeletedUserSignIn and rejects those records, and active-user helpers hide them. I did not run a live production OAuth flow or inspect the reporter's database row.

## Solution Assessment

No, a one-off hard delete or automatic recreate is not the safest immediate source fix because it bypasses unresolved deletion, privacy, audit, token, and ban policy. The maintainable path is an explicit account-lifecycle decision plus audited implementation or support tooling.
