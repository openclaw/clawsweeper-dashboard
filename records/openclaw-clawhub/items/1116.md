## Label Justifications

- `P1`: The issue blocks affected users from signing in after account deletion and is clustered across multiple active OAuth reports.
- `impact:security`: Deleted-account recovery changes deletion guarantees, auditability, token revocation, and ban-bypass boundaries.
- `impact:auth-provider`: The failure occurs during GitHub OAuth reauthorization and identity reuse for a deleted ClawHub account.
- `impact:session-state`: OAuth can complete while ClawHub refuses to establish an active usable session for the affected identity.

## Summary

Keep open. Current main still treats deleted ClawHub accounts as permanently deactivated during GitHub OAuth re-entry, and the remaining path needs maintainer policy/security direction rather than cleanup closure.

## What This Changes

The issue requests that a user who deleted a ClawHub account can re-authorize the same GitHub identity and get an active account/session again.

## Best Possible Solution

Maintainers should choose and document the deleted-account lifecycle, then implement either an audited re-registration/admin-reset path or keep deletion irreversible with clear OAuth error handling and support guidance.

## Reproduction Assessment

Yes, at the source level: deletion sets deactivatedAt/purgedAt, OAuth re-entry calls handleDeletedUserSignIn and rejects those records, and active-user helpers hide them. I did not run a live production OAuth flow or inspect the reporter's database row.

## Solution Assessment

No, a one-off hard delete or automatic recreate is not the safest source fix before maintainers settle deletion, privacy, audit, token, and ban policy. The maintainable path is an explicit account-lifecycle decision plus audited implementation or support tooling.
