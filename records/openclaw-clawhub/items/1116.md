## Summary

Keep open. Current main still treats user-initiated account deletion as terminal during GitHub OAuth, the related deleted-account login cluster remains open, and ClawHub policy keeps issues open for maintainer handling rather than cleanup-closing them.

## What This Changes

The issue asks ClawHub to recover, recreate, or reset a GitHub-linked account after user-initiated deletion instead of leaving OAuth in a loading or logged-out loop.

## Best Possible Solution

Maintainers should choose one post-deletion lifecycle, then implement that restore, purge/recreate, unlink/relink, or explicitly terminal support path consistently across OAuth, user records, admin tooling, user-facing errors, docs/specs, and tests.

## Reproduction Assessment

Yes at the source level for the current blocked deleted-account OAuth path: deletion writes deactivatedAt/purgedAt, the OAuth callback rejects that state, and active-auth helpers hide the user. The exact older infinite-loading presentation was not reproduced because current frontend tests now cover blocked/no-session error surfacing.

## Solution Assessment

No for an automatic hard-delete or recreate change as the next step. The maintainable fix needs an explicit account-lifecycle, security, and production-support decision before code changes alter deleted-account recovery semantics.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes OAuth identity reuse, deletion guarantees, token/audit semantics, and ban-bypass boundaries.

Concerns:

- **[medium] Define recovery authority before changing sign-in:** `convex/auth.ts:40`
  - body: handleDeletedUserSignIn currently checks the incoming identity relationship before handling deleted users; any restore, purge, recreate, or relink path needs equivalent provider-account verification and audit logging.
  - confidence: 0.9
- **[medium] Preserve deletion privacy and token revocation semantics:** `convex/users.ts:469`
  - body: deleteAccount revokes API tokens and clears profile/contact/GitHub-derived fields, so recovery must avoid unexpectedly restoring private data or revoked credentials.
  - confidence: 0.88
- **[medium] Keep recovery separate from bans:** `convex/users.ts:899`
  - body: The ban and unban paths distinguish deactivated accounts from banned deletedAt accounts, so recovery tooling must not let banned or autobanned identities regain access through deleted-account handling.
  - confidence: 0.86
