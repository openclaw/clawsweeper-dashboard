## Summary

Keep open. Current main deliberately blocks OAuth sign-in for deleted/deactivated accounts and now surfaces a deleted-account error, while this user-facing recovery/re-registration problem remains clustered with several open reports and needs maintainer account-lifecycle policy rather than cleanup closure.

## What This Changes

The issue asks ClawHub to recover or recreate a GitHub-linked account after account deletion leaves later OAuth sign-in stuck, unauthorized, or unable to establish a session.

## Best Possible Solution

Maintainers should choose and document the deleted-account lifecycle, then implement either a controlled recovery/re-registration/admin reset flow or an explicit irreversible-deletion support path with regression coverage for active, deleted, legacy-deleted, banned, and autobanned identities.

## Reproduction Assessment

Yes, for the blocked reauthorization/re-registration path from source inspection rather than a live production account. `deleteAccount` marks the user deactivated and purged, the next OAuth callback calls `handleDeletedUserSignIn`, and active-user helpers reject that linked user identity.

## Solution Assessment

No for an automated fix. The requested fresh-account behavior conflicts with the current irreversible-deletion contract, so recovery, hard purge, relink, or support reset behavior needs maintainer security/product approval first.
