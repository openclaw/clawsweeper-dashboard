## Summary

Keep open. Current main still treats user-initiated account deletion as a terminal auth state, while this issue and the related open cluster show real users blocked from re-registering after GitHub OAuth; for ClawHub issues, this should remain open for maintainer product/security handling rather than cleanup closure.

## What This Changes

Issue requests a recovery, reset, or re-registration path for a deleted ClawHub account whose GitHub OAuth identity currently returns to a logged-out/loading state.

## Best Possible Solution

Maintainers should choose and document the deleted-account lifecycle, then implement the chosen restore, unlink/relink, purge/recreate, or explicitly terminal support path across OAuth, user records, admin tooling, user-facing errors, docs/specs, and tests.

## Reproduction Assessment

Yes at the source level: current main writes deactivatedAt/purgedAt on deletion, rejects deleted/deactivated users in the OAuth callback, and hides those records from active-auth helpers. I did not run a live OAuth flow for the reporter's production account.

## Solution Assessment

No for an automatic hard-delete or recreate change as the next step. The maintainable fix needs an explicit account-lifecycle, security, privacy, and production-support decision before code changes alter deleted-account recovery semantics.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes OAuth identity reuse, deletion guarantees, token/audit semantics, and ban-bypass boundaries.

Concerns:

- **[medium] Define recovery authority before changing sign-in:** `convex/auth.ts:40`
  - body: handleDeletedUserSignIn currently verifies the incoming identity relationship before handling deleted users; any restore, purge, recreate, or relink path needs equivalent provider-account verification and audit logging.
  - confidence: 0.9
- **[medium] Preserve deletion privacy and token revocation semantics:** `convex/users.ts:469`
  - body: deleteAccount revokes API tokens and clears profile, contact, and GitHub-derived fields, so recovery must avoid unexpectedly restoring private data or revoked credentials.
  - confidence: 0.88
- **[medium] Keep recovery separate from bans:** `convex/users.ts:899`
  - body: Ban and unban paths distinguish deactivated accounts from banned deletedAt accounts; recovery tooling must not let banned or autobanned identities regain access through deleted-account handling.
  - confidence: 0.86
