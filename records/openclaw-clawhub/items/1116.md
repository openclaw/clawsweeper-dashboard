## Summary

Keep open. Current main still treats user-initiated deletion as an irreversible auth state, while this issue and related open reports show users blocked after GitHub OAuth; for ClawHub issues this should stay with maintainers for product/security handling rather than cleanup closure.

## What This Changes

Allow users who deleted a ClawHub account to sign in again with the same GitHub OAuth identity instead of looping or remaining unauthorized.

## Best Possible Solution

Maintainers should choose and document the deleted-account lifecycle, then implement either a controlled recovery/admin reset path or an explicitly terminal support/error path with security, docs/specs, and regression coverage updated together.

## Reproduction Assessment

Yes at the source level: current main writes deactivatedAt/purgedAt on deletion, rejects deleted/deactivated users during OAuth handling, and hides those records from active-auth helpers. I did not run a live OAuth flow against the reporter's production account.

## Solution Assessment

No for the requested one-off hard-delete or automatic recreate behavior as the immediate product-wide fix. The maintainable path needs an explicit lifecycle, privacy, security, ban, and support decision before code changes alter deleted-account recovery semantics.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes OAuth identity reuse, deletion guarantees, token/audit semantics, and ban-bypass boundaries.

Concerns:

- **[medium] Define recovery authority before changing sign-in:** `convex/auth.ts:40`
  - body: handleDeletedUserSignIn verifies the incoming identity relationship before handling deleted users; any restore, purge, recreate, or relink path needs equivalent provider-account verification and audit logging.
  - confidence: 0.9
- **[medium] Preserve deletion privacy and token revocation semantics:** `convex/users.ts:469`
  - body: deleteAccount revokes API tokens and clears profile, contact, and GitHub-derived fields, so recovery must not unexpectedly restore private data or revoked credentials.
  - confidence: 0.88
- **[medium] Keep recovery separate from bans:** `convex/users.ts:910`
  - body: Ban and unban paths distinguish deactivated accounts from banned deletedAt accounts; recovery tooling must not let banned or autobanned identities regain access through deleted-account handling.
  - confidence: 0.86
