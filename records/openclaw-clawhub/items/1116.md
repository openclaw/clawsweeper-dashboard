## Summary

Keep open. Current main still treats user-initiated deletion as irreversible and blocks later GitHub OAuth sign-in for deleted/deactivated users; newer UI code surfaces auth errors, but it does not recreate, relink, or administratively reset the account state. ClawHub cleanup policy also keeps issues open unless they are PRs already implemented on main.

## What This Changes

The issue requests re-registration or administrator reset for a GitHub OAuth identity that cannot sign in after its ClawHub account was deleted.

## Best Possible Solution

Maintainers should choose and document the deleted-account lifecycle, then implement either a controlled re-registration/admin reset flow or an explicit irreversible-deletion error with a support workflow and regression coverage for active, deleted, legacy-deleted, banned, autobanned, and reauthorized identities.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: deleteAccount sets deactivatedAt/purgedAt, OAuth sign-in calls handleDeletedUserSignIn, and tests assert that deactivated users are rejected. I did not run a live OAuth flow during this read-only review.

## Solution Assessment

Unclear. The current implementation is internally consistent with the irreversible-deletion docs and now has UI error handling, but it does not satisfy the requested account recreation/reset behavior; the safest repair depends on maintainer policy for deletion, privacy, and abuse prevention.
