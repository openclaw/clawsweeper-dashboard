## Summary

This PR addresses a real gap that current main still has, but it is conflicted, lacks after-fix proof, and misses an existing GitHub sign-in entry point, so the diagnostic would work inconsistently.

## What This Changes

The branch records outbound GitHub OAuth redirects, surfaces returns without a code, error, or session, and disambiguates CLI device user codes from OAuth callback codes.

## Best Possible Solution

Rebase a single shared GitHub sign-in helper onto current main so every entry point records redirects consistently, retain current retry and banned-account behavior, and validate normal OAuth, no-code fallback, and CLI device authorization in a real setup.

## Reproduction Assessment

Yes from source and the reported live redirect: current main has no handler for a return with neither code nor error, and the CLI route’s device `code` can be consumed as an OAuth code. A full live current-main reproduction was not independently run.

## Solution Assessment

No in its current form; the redirect-attempt approach is plausible, but it must cover every sign-in entry point, be rebased onto the canonical retry path, and have its timeout and CLI behavior proven live.
