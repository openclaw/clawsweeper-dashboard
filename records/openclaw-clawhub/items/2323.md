## Summary

This PR addresses a distinct OAuth failure that current main still does not surface: main retries code-bearing callbacks that fail to create a session, but a callback returning with neither `code` nor `error` still falls through to the ordinary signed-out UI. The branch should remain open because that useful behavior is not implemented on main, but it is not merge-ready: it is conflicted with newer auth handling, omits one production GitHub sign-in caller from redirect tracking, and provides no after-fix real-auth proof. Likely related people: Patrick-Erichsen (high confidence, current callback retry owner) and Loïc Françoix (medium confidence, original auth provider path contributor).

## What This Changes

The PR records GitHub OAuth redirect attempts in session storage, reports callback returns with no code/error/session, and separates CLI device user codes from OAuth callback codes.

## Best Possible Solution

Rebase a centralized redirect-attempt mechanism onto current main so every production GitHub sign-in entry point participates, preserve the merged retry/toast/account-state behavior, retain backward-compatible CLI device entry, and prove the no-code failure plus successful browser and CLI auth flows in a real environment.

## Reproduction Assessment

Yes at source level: current main has no handler when a GitHub callback returns with neither `code` nor `error`, matching the reported stranded signed-out state. This review did not execute a live provider callback, so confidence remains medium rather than a fully reproduced result.

## Solution Assessment

No, not yet: redirect-attempt tracking is a plausible fix, but implementing it in selected callers leaves another production sign-in path uncovered and creates future drift. A shared sign-in wrapper or complete centralized instrumentation is the narrower maintainable approach.
