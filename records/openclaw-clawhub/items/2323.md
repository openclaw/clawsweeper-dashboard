## Summary

The no-code callback failure remains unsolved on main, but this branch is not ready to land because redirect tracking misses a direct GitHub sign-in caller and its timeout-based failure inference lacks real-runtime validation.

## What This Changes

The PR adds session-scoped GitHub OAuth redirect tracking, reports callbacks returning without code, error, or session, and separates CLI device user codes from OAuth callback codes.

## Best Possible Solution

Rebase the diagnostic onto the current centralized callback handler, route every GitHub sign-in entry point through one marker-aware helper, preserve existing CLI links, and validate no-code, normal OAuth, and CLI device flows in a real browser.

## Reproduction Assessment

Yes, at source level: current main only enters callback handling when `code` or an error is present, matching the reported live return with neither value. I did not independently reproduce the OAuth provider failure in a real deployment.

## Solution Assessment

No, not in its current form; the diagnostic is useful, but the narrow maintainable solution must compose with current main, cover all sign-in callers, and avoid an unproven fixed timeout and CLI query migration.
