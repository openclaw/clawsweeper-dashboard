## Summary

Keep open: the PR addresses a real auth diagnostic gap, but the proposed fallback can misclassify normal OAuth code/error callbacks after the existing handlers strip those params, and the PR does not include after-fix real behavior proof.

## What This Changes

The branch records GitHub sign-in redirects in sessionStorage and adds a global fallback handler plus tests to show an auth diagnostic when the callback returns to the same URL without code, error, or session.

## Best Possible Solution

Keep the fallback, but make it explicitly clear or bypass the pending marker when a legitimate code/error callback is present or has just been handled, then prove the empty-return diagnostic in a redacted real auth run.

## Reproduction Assessment

Yes for the review finding: with a pending redirect marker and a callback URL containing code or error, the existing handlers run first and strip those params before the new fallback checks the URL. I did not reproduce the original live Convex/GitHub empty-return path in a browser.

## Solution Assessment

No as written. The fallback direction is maintainable, but it must be isolated from legitimate code/error callbacks and should include coverage for those callback paths with a pending marker.
