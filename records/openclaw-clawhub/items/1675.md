## Summary

Keeping this open. Current main still models skill transfers as user-recipient transfers, while org publishers and partial publish-as-org support exist elsewhere, so moving an existing personal skill to an org remains a real product/backend gap.

## What This Changes

Allow existing skills owned by personal handles to be transferred or otherwise re-homed to organization publishers while preserving stats and canonical owner URLs.

## Best Possible Solution

Implement a publisher-targeted skill transfer flow that supports user and org publishers, authorizes acceptance through org membership, preserves stats and aliases, and keeps existing user-to-user clients compatible.

## Reproduction Assessment

Yes. The source path is high confidence: an org handle reaches getActiveUserByHandleOrPersonalPublisher, which intentionally returns null for org publishers, causing requestTransferInternal to throw User not found before a transfer can be created.

## Solution Assessment

Unclear. Transfer-to-org matches the documented publisher-transfer direction, but the maintainable fix should update the transfer model around publisher targets rather than bolting org lookup onto the existing user-only toUserHandle schema.
