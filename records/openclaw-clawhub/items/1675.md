## Summary

Keeping this open. Current main still has the reported product gap: skill transfer requests resolve only user recipients, while org/publisher ownership exists and the org design notes call out publisher-targeted transfers as unfinished work.

## What This Changes

Request support for moving existing skills from personal ownership to organization ownership, or an equivalent publish-as-org/re-home path that preserves stats and existing URLs.

## Best Possible Solution

Implement publisher-targeted skill transfers that preserve stats, aliases, existing install/page URLs, and user-to-user compatibility while authorizing org acceptance through publisher membership.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: an org handle is normalized into toUserHandle, then the recipient helper rejects non-user publishers and requestTransferInternal throws User not found before any transfer can be created.

## Solution Assessment

Unclear. Transfer-to-org matches the documented direction, but the maintainable fix should update transfers around publisher targets rather than bolt org lookup onto the existing user-only toUserHandle/toUserId model.
