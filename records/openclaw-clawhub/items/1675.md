## Summary

Keeping this open under the ClawHub issue-cleanup policy. Current main now supports the publish-time owner migration path, but the direct `clawhub transfer request` and transfer API still target users rather than organization publishers, and the broader publisher-transfer work remains in the open PR at https://github.com/openclaw/clawhub/pull/1635.

## What This Changes

The issue requests a supported way to move an existing personally owned skill under an organization owner without deleting and republishing.

## Best Possible Solution

Keep the explicit publish owner-migration route as the supported immediate path, then either land or extract a publisher-targeted transfer flow with org-role checks if direct transfer commands remain a product requirement.

## Reproduction Assessment

Yes. Source inspection shows the direct transfer path still accepts `toUserHandle`, resolves only users or user-kind personal publishers, and stores `toUserId`, so an organization publisher handle remains an invalid transfer destination.

## Solution Assessment

Unclear. The merged `ownerHandle` plus `migrateOwner` publish flow is a maintainable way to re-home a skill while publishing, but a direct transfer-to-org command/API would need a separate publisher-targeted transfer design rather than a small resolver tweak.
