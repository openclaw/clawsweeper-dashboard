## Summary

Keep open for maintainer follow-up. Current main now supports publish-based owner migration, but the direct `clawhub transfer request` and transfer API path are still user-targeted; ClawHub cleanup policy also keeps issues open rather than auto-closing them.

## What This Changes

Request support for moving skills from personal ownership to org ownership through direct transfer, publish-as-org, or re-home-on-publish behavior.

## Best Possible Solution

Keep the explicit `--owner` plus `--migrate-owner` publish path as the supported migration route, then either document that as the resolution or implement a separate publisher-targeted transfer flow with matching authorization, API, CLI, docs, and tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the direct transfer API/CLI still uses `toUserHandle` and `toUserId`, and the recipient resolver only returns users or user-kind personal publishers, so an org publisher handle cannot be resolved as a transfer target.

## Solution Assessment

Unclear. The current publish migration is a maintainable solution for users who can republish with `--owner --migrate-owner`, but it does not make the direct transfer API/CLI publisher-targeted; maintainers should decide whether that remaining flow is required.
