## Summary

Keeping this open. Current main has org publishers and web publish-as-org support, but the existing skill transfer workflow is still user-recipient based and cannot re-home an existing skill to an org while preserving the current skill record.

## What This Changes

The issue requests support for organization-owned skills, especially transferring an existing personally owned skill to an org owner without losing stats or canonical URLs.

## Best Possible Solution

Implement publisher-targeted skill transfers that store source and target publisher ids, authorize acceptance by target owner/admin membership, preserve stats and slug aliases, and update the API/CLI/UI contracts compatibly.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an org handle reaches the transfer endpoint as toUserHandle, but recipient resolution only returns active users or user-kind personal publishers, then the transfer table and accept path require user ids.

## Solution Assessment

No, a small org lookup added to the current toUserHandle flow is not the best fix. The maintainable solution is the publisher-targeted transfer model already described in docs/orgs.md, because schema, listing, acceptance, audit, and compatibility are all currently user-centric.
