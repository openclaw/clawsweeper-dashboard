## Summary

Keep open under the ClawHub issue policy. Current main now has publish-time owner migration and a direct HTTP org-transfer path, but the exact CLI command can still fail because the transfer command pre-serializes its JSON body before the shared HTTP client serializes it again; that narrow CLI bug is already tracked by https://github.com/openclaw/clawhub/issues/1300.

## What This Changes

The issue requests support for moving a skill from an individual owner to an organization owner through transfer, publish-as-org, or publish-time re-homing while preserving existing skill state.

## Best Possible Solution

Keep the current publisher-aware transfer and publish-time migration paths, repair the shared CLI object-body bug through https://github.com/openclaw/clawhub/issues/1300, and make any pending acceptance or approval semantics an explicit maintainer decision.

## Reproduction Assessment

Yes for the reported CLI command error: source inspection shows cmdTransferRequest passes a pre-serialized JSON string and both HTTP transports stringify it again, so the server sees a string payload and can return toUserHandle required. The direct curl/API org-transfer path is no longer source-reproducible on current main because org handles are routed to transferSkillOwnerForUserInternal.

## Solution Assessment

Unclear as a final product direction. Current main's immediate admin-authorized org transfer plus publish-time migration is maintainable, but the CLI serialization fix belongs in the canonical double-JSON issue and any pending acceptance or management approval flow needs maintainer judgment.
