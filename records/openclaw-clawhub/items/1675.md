## Summary

Keep this issue open. Current main has partial org publisher support, including web publish owner selection, but skill transfers still use user-to-user recipient fields and the CLI/API skill publish path still lacks a publisher owner payload, so the reported no-loss personal-to-org transfer remains unresolved.

## What This Changes

Support transferring or publishing skills under org publishers so existing personal-owned skills can move to organization ownership without deleting them or losing stats and canonical URLs.

## Best Possible Solution

Implement a core publisher-targeted skill transfer flow that resolves user and org publisher handles, authorizes source and target publisher roles, preserves stats and aliases, keeps old user-to-user clients compatible, and updates API/CLI schemas plus regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: POSTing a transfer to an org handle still enters the toUserHandle path, the resolver excludes org publishers, and the handler returns User not found; no live production call was needed for that conclusion.

## Solution Assessment

Unclear as proposed because the issue lists several possible product paths. The narrowest maintainable end state appears to be explicit publisher-targeted transfer rather than implicit re-home-on-publish, but maintainers should confirm the contract before implementation.
