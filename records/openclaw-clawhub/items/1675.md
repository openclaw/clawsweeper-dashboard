## Summary

Keeping this open. Current main still has the reported gap: skill transfer requests are modeled as user-to-user, while org publishers and publish-as-org support exist separately and the org design doc still lists publisher-targeted transfers as unfinished work.

## What This Changes

Support moving an existing skill from a personal owner to an org owner without deleting, republishing, losing stats, or breaking existing URLs.

## Best Possible Solution

Implement publisher-targeted skill transfers that preserve stats, aliases, existing install URLs, and user-to-user compatibility, with org acceptance authorized through publisher membership.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an org handle is normalized as toUserHandle, getActiveUserByHandleOrPersonalPublisher returns null for org publishers, and requestTransferInternal throws "User not found" before creating a transfer.

## Solution Assessment

Unclear. Transfer-to-org matches the documented direction, but the maintainable solution should migrate the transfer model to publisher targets instead of bolting org lookup onto the current user-only schema.
