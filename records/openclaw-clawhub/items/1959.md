## Summary

Keep open: current main still has the reported scanner-dispute loop, and the linked #1958 report is a concrete production example from the same non-collaborator. This is also a ClawHub issue involving analyzer/security triage policy, so it is not eligible for cleanup closure under the repository policy.

## What This Changes

Report that ClawHub’s scanner-dispute auto-response closes static-analysis appeal issues while telling non-collaborators to reopen them, creating a duplicate-report loop.

## Best Possible Solution

Define a non-collaborator follow-up path for scanner disputes, then update the rescan-guidance automation, docs, and tests to use that path while preserving secops review for security findings.

## Reproduction Assessment

Yes. The linked #1958 report is a concrete production example, and current main deterministically closes `r: rescan-guidance` issues while telling reporters to reopen them after rescanning.

## Solution Assessment

Unclear. There is no proposed patch, and the narrow code change depends on maintainer/secops policy: keep these reports open, redirect to another writable channel, or add a reviewed analyzer acknowledgement/suppression path.
