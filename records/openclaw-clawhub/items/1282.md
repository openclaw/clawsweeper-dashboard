## Summary

Keep open: current main still treats self-service account deletion as irreversible and blocks later GitHub OAuth sign-in for the same deactivated account, while the requested restore, unbind, or hard-delete path remains a maintainer product/security and production-support decision. ClawHub cleanup policy also reserves auto-close here for PRs that are already implemented on main, not open account-support issues.

## What This Changes

The issue asks ClawHub maintainers to restore, unbind, or hard-delete a deleted GitHub-linked account so the reporter can sign in and publish again.

## Best Possible Solution

Define the deleted-account lifecycle explicitly: either keep deletion irreversible with a documented support response, or add a narrow audited restore/unbind flow that verifies GitHub ownership and preserves moderation safeguards.

## Reproduction Assessment

Yes, at source level. A GitHub-linked user who runs deleteAccount gets deactivatedAt and purgedAt, and a later OAuth callback invokes handleDeletedUserSignIn and rejects that user; I did not verify the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved account lifecycle policy with identity verification, audit requirements, and ban/deletion separation.
