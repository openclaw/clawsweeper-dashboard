## Summary

Keep open: current main still intentionally blocks GitHub OAuth sign-in after user-initiated account deletion, and the requested restore/unbind/hard-delete path is a security-sensitive ClawHub account-lifecycle decision rather than a cleanup close.

## What This Changes

The issue asks ClawHub to restore, unbind, or hard-delete a deleted GitHub-linked account so the same GitHub identity can sign in again.

## Best Possible Solution

Define the deleted-account lifecycle explicitly: either keep deletion irreversible with a documented support response, or add a narrow audited restore/unbind flow that verifies GitHub ownership and preserves moderation, privacy, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level: a GitHub-linked user who runs `deleteAccount` gets `deactivatedAt` and `purgedAt`, and a later OAuth callback invokes `handleDeletedUserSignIn` and rejects that user. I did not verify the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved lifecycle policy with identity verification, audit requirements, and ban/deletion separation.
