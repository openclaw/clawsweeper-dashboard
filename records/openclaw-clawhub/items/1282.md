## Summary

Keep open. Current main deliberately rejects deleted-account GitHub reauth and the requested restore, unlink, or hard-delete path would change account lifecycle and security policy; ClawHub cleanup policy also keeps issues open rather than auto-closing them as implemented.

## What This Changes

The issue requests re-enabling GitHub login or re-registration after the reporter deleted a ClawHub account, by restoring or unbinding/hard-deleting the old GitHub-linked record.

## Best Possible Solution

Define a maintainer-approved account lifecycle: either keep deletion irreversible with a documented support-only process, or add a narrow audited restore/unlink flow that verifies GitHub identity ownership, preserves moderation safeguards, and covers web plus CLI login behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `deleteAccount` sets `deactivatedAt`/`purgedAt`, GitHub OAuth calls `handleDeletedUserSignIn`, and tests assert deleted users are rejected; I did not exercise production OAuth for nisxu.

## Solution Assessment

No for ad hoc unbinding or hard deletion as the durable repo fix. The maintainable path needs an explicit recovery policy with identity verification, auditability, and ban/deletion separation before code or operator cleanup proceeds.
