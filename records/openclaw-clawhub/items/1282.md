## Summary

Keep open: current main still intentionally blocks GitHub OAuth sign-in after user-initiated account deletion, and the requested restore/unbind/hard-delete path needs maintainer product, security, and production-support decisions rather than cleanup auto-close.

## What This Changes

The issue requests recovery, hard deletion, or GitHub OAuth unbinding after a deleted ClawHub account blocks sign-in.

## Best Possible Solution

Define the deleted-account lifecycle explicitly: keep deletion irreversible with a documented support response, or add a narrow audited restore/unbind flow with GitHub ownership verification, moderation safeguards, docs, and tests.

## Reproduction Assessment

Yes, at source level. A GitHub-linked user who runs deleteAccount gets deactivatedAt/purgedAt, and a later OAuth callback rejects that user through handleDeletedUserSignIn; I did not verify the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved account lifecycle policy with audited recovery authority, identity verification, and ban/deletion separation.
