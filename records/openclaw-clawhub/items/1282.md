## Summary

Keep open: current main still implements irreversible self-deletion and rejects later GitHub OAuth sign-in for the same deactivated account, while the requested restore/unbind path is a security-sensitive product and production-support decision rather than a narrow source bug fix.

## What This Changes

The issue asks for account recovery or OAuth unbinding so a GitHub-linked account can sign in again after self-deleting its ClawHub account.

## Best Possible Solution

Define one supported deleted-account lifecycle: keep deletion terminal with a documented support response, or add a narrow audited restore/unlink/recreate flow with GitHub ownership verification, moderation safeguards, docs, and tests.

## Reproduction Assessment

Yes, at source level. A GitHub-linked user who runs deleteAccount gets deactivatedAt/purgedAt, and a later OAuth callback rejects that user via handleDeletedUserSignIn; I did not verify the reporter's production row.

## Solution Assessment

No for ad hoc hard deletion or OAuth unbinding as the durable solution. The safer path is a maintainer-approved account lifecycle policy with audited recovery authority, identity verification, and ban/deletion separation.
