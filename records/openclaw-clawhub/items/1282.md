## Summary

Keep open: current main still rejects GitHub OAuth sign-in for permanently deleted users, and the requested restore, unbind, or hard-delete path remains a product/security/support decision rather than a cleanup-close candidate for a ClawHub issue.

## What This Changes

Allow a user who deleted a GitHub-linked ClawHub account to regain access by restoring, unbinding, or hard-deleting the old auth record.

## Best Possible Solution

Define the deleted-account lifecycle explicitly: keep deletion irreversible with a documented support response, or add a narrow audited restore/unbind flow that verifies GitHub ownership and preserves moderation safeguards.

## Reproduction Assessment

Yes, at source level. A GitHub-linked user who runs deleteAccount gets deactivatedAt and purgedAt, and a later OAuth callback invokes handleDeletedUserSignIn and rejects that user; I did not verify the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved lifecycle policy with identity verification, audit requirements, and ban/deletion separation.
