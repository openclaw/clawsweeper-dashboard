## Summary

Keep open. Current main still implements irreversible self-deletion and rejects GitHub OAuth sign-in for deleted/deactivated accounts, while related account-deletion sign-in reports such as https://github.com/openclaw/clawhub/issues/504 remain open; this needs maintainer account-lifecycle and production-support handling, not conservative cleanup.

## What This Changes

The issue requests restoration, hard deletion, or GitHub identity unbinding for a deleted ClawHub account so the same GitHub account can sign in again.

## Best Possible Solution

Define the account lifecycle explicitly: keep deletion irreversible with a documented support-only process, or add a narrow audited restore/unlink flow that verifies GitHub ownership and preserves moderation protections.

## Reproduction Assessment

Yes by source inspection rather than live OAuth: deleteAccount writes deactivatedAt/purgedAt, handleDeletedUserSignIn rejects deleted/deactivated users, and tests assert the blocked sign-in path.

## Solution Assessment

No for ad hoc unbinding or hard deletion as the durable solution. The safer path is a maintainer-approved account recovery policy or audited support flow that does not weaken deletion, identity, or moderation boundaries.
