## Summary

Keep open: current main still deliberately treats self-deleted GitHub-backed accounts as permanently inactive and rejects later OAuth sign-in, while this issue asks for a security-sensitive restore, unbind, or hard-delete account lifecycle decision that maintainers must choose.

## What This Changes

The reporter asks ClawHub maintainers to restore, hard-delete, or unbind a deleted account's GitHub identity so the same GitHub account can log in or re-register.

## Best Possible Solution

Define the deleted-account lifecycle explicitly: either keep deletion irreversible with a documented support response, or add a narrow audited restore/unbind flow that verifies GitHub ownership and preserves moderation, privacy, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level: a GitHub-linked user who runs deleteAccount is marked deactivated/purged, and a later OAuth callback for that same user is rejected by handleDeletedUserSignIn. I did not verify the reporter's production database row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved lifecycle policy with identity verification, audit requirements, and ban/deletion separation.

## Risks / Open Questions

- Manual GitHub identity unbinding or hard deletion would change a production auth binding and should verify current control of the GitHub account before any database edit.
