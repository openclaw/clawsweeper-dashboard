## Label Justifications

- `P1`: The reporter is blocked from ClawHub sign-in after account deletion, and several related open reports show the workflow affects real users.
- `impact:security`: Recovering or unlinking a deleted account affects OAuth identity ownership, deletion guarantees, audit history, and moderation bypass boundaries.
- `impact:auth-provider`: The reported failure is specifically tied to GitHub OAuth identity binding after account deletion.
- `impact:session-state`: The user cannot move from OAuth completion to an active ClawHub session because the linked account is inactive.

## Summary

Keep open: current main still implements irreversible account deletion and rejects the next GitHub OAuth sign-in for that deleted identity, so the requested restore/unbind path remains unresolved and needs maintainer handling.

## What This Changes

The issue requests account recovery or GitHub OAuth unbinding after a deleted ClawHub account blocks re-registration with the same GitHub identity.

## Best Possible Solution

Define the deleted-account lifecycle explicitly: keep deletion irreversible with a documented support response, or add a narrow audited restore/unbind flow that verifies GitHub ownership and preserves moderation, privacy, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level: a GitHub-linked user who runs deleteAccount gets deactivatedAt and purgedAt, and a later OAuth callback invokes handleDeletedUserSignIn and rejects that user. I did not inspect the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved lifecycle policy with identity verification, audit requirements, and ban/deletion separation.
