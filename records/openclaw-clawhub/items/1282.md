## Summary

Keep open: current main still intentionally makes self-service deletion terminal and blocks later GitHub OAuth sign-in for the same account, so resolving this needs a maintainer account-lifecycle/security decision and production identity handling rather than cleanup closure.

## What This Changes

The issue asks ClawHub staff to restore, hard-delete, or unbind a deleted account's GitHub OAuth identity so the same GitHub user can sign in again.

## Best Possible Solution

Define a maintainer-approved deleted-account lifecycle: keep deletion terminal with documented support guidance, or build a narrow audited restore/unbind flow with GitHub ownership verification, moderation safeguards, docs, and tests.

## Reproduction Assessment

Yes, at source level. A GitHub-linked user who runs deleteAccount gets deactivatedAt/purgedAt, and the later OAuth callback rejects that user through handleDeletedUserSignIn; I did not verify the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved lifecycle policy with audited recovery authority, identity verification, and ban/deletion separation.
