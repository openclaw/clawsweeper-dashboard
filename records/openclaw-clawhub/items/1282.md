## Summary

Keep open: current main intentionally keeps user-initiated deletion terminal and rejects later GitHub OAuth sign-in for the same deleted account, so the reporter's restore/unbind request remains unresolved and needs maintainer product/security handling rather than conservative cleanup.

## What This Changes

The issue asks maintainers to unbind or hard-delete a deleted GitHub-linked ClawHub account so the same GitHub account can sign in again.

## Best Possible Solution

Define one supported post-deletion lifecycle: keep deletion terminal with a documented support response, or add an audited restore/unbind/recreate flow with GitHub ownership verification, privacy rules, moderation safeguards, and web/CLI coverage.

## Reproduction Assessment

Yes at source level. A GitHub-linked user who runs deleteAccount receives deactivatedAt/purgedAt, and a later OAuth callback calls handleDeletedUserSignIn, which rejects that user instead of creating or restoring an active account.

## Solution Assessment

No for ad hoc manual hard deletion or unbinding as the durable solution. The safer path is a maintainer-approved account lifecycle policy with audited identity verification and explicit ban/deletion separation.
