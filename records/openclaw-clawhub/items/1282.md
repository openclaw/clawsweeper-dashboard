## Summary

Keep open: current main intentionally treats user-initiated deletion as irreversible and rejects later GitHub OAuth sign-in for the same deleted account, while the requested restore/unlink path still needs security, product, and production-support decisions.

## What This Changes

The issue asks ClawHub to restore, unlink, or hard-delete a deleted GitHub-bound account so the same GitHub identity can log in again.

## Best Possible Solution

Define a maintainer-approved deleted-account lifecycle: keep deletion terminal with a documented support response, or build a narrow audited restore/unlink flow with GitHub ownership verification, moderation safeguards, docs, and tests.

## Reproduction Assessment

Yes, at source level. A GitHub-linked user who runs deleteAccount gets deactivatedAt/purgedAt, and a later OAuth callback rejects that user through handleDeletedUserSignIn; I did not verify the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved account lifecycle policy with audited recovery authority, identity verification, and ban/deletion separation.
