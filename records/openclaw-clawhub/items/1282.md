## Summary

Keep open. Current main still intentionally rejects GitHub sign-in for permanently deleted ClawHub accounts, several related reports remain open, and the repository cleanup policy does not auto-close ClawHub issues; the remaining work is a maintainer account-lifecycle and security decision.

## What This Changes

The issue requests manual unbinding, hard deletion, or restoration so the same GitHub account can use ClawHub again after account deletion.

## Best Possible Solution

The desired end state is a maintainer-approved account lifecycle that either documents irreversible deletion with support-only handling or provides an audited restore/unlink flow with tests and moderation safeguards.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `deleteAccount` sets `deactivatedAt`/`purgedAt`, GitHub OAuth calls `handleDeletedUserSignIn`, and tests assert deleted/deactivated users are rejected; I did not run a live production OAuth flow for this reporter’s account.

## Solution Assessment

No for ad hoc unbinding or hard deletion as the durable repo fix. A maintainable solution needs a maintainer-approved recovery policy that distinguishes self-deletion from bans or abuse-disablement and records the action audibly.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes GitHub identity binding, deletion permanence, and moderation bypass boundaries.

Concerns:

- **[medium] Define recovery authority before changing OAuth sign-in:** `convex/auth.ts:42`
  - body: Restoring, recreating, or unlinking a deleted GitHub-backed account during OAuth would override the current irreversible-deletion contract unless recovery authority and audit requirements are decided first.
  - confidence: 0.94
- **[medium] Preserve ban and deletion separation:** `convex/auth.ts:48`
  - body: The sign-in path still checks ban audit records for legacy deleted users, so any recovery path must keep banned and malware-autobanned identities from re-registering through the same provider identity.
  - confidence: 0.9
- **[medium] Audit provider-account rebinding:** `convex/lib/githubIdentity.ts:13`
  - body: GitHub `providerAccountId` is used as an identity anchor for ownership healing, so unlinking or rebinding it needs explicit identity verification and audit records.
  - confidence: 0.86
- **[medium] Preserve deletion privacy semantics:** `convex/users.ts:348`
  - body: Account deletion clears profile/contact/GitHub-derived fields and revokes API tokens, so recovery must avoid reviving private data or credentials unexpectedly.
  - confidence: 0.86
