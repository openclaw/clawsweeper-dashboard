## Label Justifications

- `P1`: The reporter is blocked from signing in to ClawHub after deletion, and multiple related open reports show the same auth workflow is affecting real users.
- `impact:security`: Restoring or unlinking a deleted account affects OAuth identity ownership, deletion guarantees, audit history, and moderation bypass boundaries.
- `impact:auth-provider`: The failure is specifically tied to GitHub OAuth identity binding after account deletion.
- `impact:session-state`: OAuth completion cannot produce an active ClawHub session because the linked user record is inactive.

## Summary

Keep open: current main still treats user-initiated deletion as irreversible and blocks later GitHub OAuth sign-in for that linked account, while the requested restore/unbind path needs a maintainer security and product decision. I did not find a merged fix or maintainer-selected canonical issue that safely owns the recovery policy yet.

## What This Changes

The issue requests restoration, hard deletion, or unbinding of a deleted ClawHub account's GitHub OAuth identity so the reporter can sign in and publish again.

## Best Possible Solution

Define the account lifecycle explicitly: either keep deletion irreversible with a documented support response, or add a narrow audited restore/unbind flow that verifies GitHub ownership and preserves moderation, privacy, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level: a GitHub-linked user who runs deleteAccount gets deactivatedAt and purgedAt, and a later OAuth callback invokes handleDeletedUserSignIn and rejects that user. I did not inspect the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved lifecycle policy with identity verification, audit requirements, and ban/deletion separation.
