## Summary

Keep open: current main intentionally blocks OAuth sign-in for permanently deleted GitHub-linked accounts, the requested restore/unbind/hard-delete path remains unresolved, and ClawHub policy keeps issues open for maintainer handling.

## What This Changes

The reporter asks ClawHub to restore, hard-delete, or unbind a deleted GitHub-linked account so the same GitHub identity can sign in again.

## Best Possible Solution

Define the deleted-account lifecycle explicitly: either keep deletion irreversible with a documented support response, or add a narrow audited restore/unbind flow that verifies GitHub ownership and preserves moderation, privacy, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level: a GitHub-linked user who runs `deleteAccount` gets `deactivatedAt` and `purgedAt`, and a later OAuth callback invokes `handleDeletedUserSignIn` and rejects that user. I did not inspect the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved lifecycle policy with identity verification, audit requirements, and ban/deletion separation.
