## Summary

Keep open. Current main still intentionally treats user-initiated deletion as irreversible and rejects the next GitHub OAuth sign-in, while the reporter’s requested restore/unbind path remains a production account-lifecycle policy question rather than conservative cleanup.

## What This Changes

The issue requests restoring, unbinding, or hard-deleting a deleted GitHub-backed ClawHub account so the same GitHub identity can log in or re-register.

## Best Possible Solution

Define the intended account lifecycle explicitly, then implement either a documented support-only recovery process or a narrow audited restore/unbind flow that verifies GitHub identity ownership and preserves moderation protections.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: deleteAccount sets deactivatedAt/purgedAt, the OAuth callback calls handleDeletedUserSignIn, and the active-user helpers keep the auth-linked deleted user from becoming an active session.

## Solution Assessment

No for an ad hoc unbind or hard delete. The maintainable solution needs a maintainer-approved security and privacy policy plus an audited implementation across OAuth, CLI token creation, auth account binding, operator tooling, docs, and tests.
