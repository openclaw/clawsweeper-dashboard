## Summary

Keep open. Current main still deliberately blocks GitHub reauthentication for deleted or deactivated ClawHub accounts, the related account-recovery cluster remains open, and the ClawHub cleanup policy does not auto-close issues.

## What This Changes

The issue requests manual unbinding, hard deletion, or restoration so the reporter can re-register or sign in with the same GitHub account after deleting their ClawHub account.

## Best Possible Solution

Finalize the account-recovery policy, then implement either support-only docs/operator runbook or an audited restore, unlink, or re-registration flow across Convex auth, account deletion, auth bindings, CLI token creation, docs/specs, and regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: deleteAccount sets deactivatedAt/purgedAt, GitHub OAuth calls handleDeletedUserSignIn, and active-user helpers refuse sessions tied to deleted records; I did not run a live OAuth flow for this reporter’s production account.

## Solution Assessment

No for ad hoc unbinding or hard deletion. The maintainable solution needs a maintainer-approved recovery/privacy/security policy before code chooses restore, rebind, hard delete, or support-only behavior.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery is security-sensitive because it changes OAuth identity binding, deletion permanence, moderation bypass boundaries, and audit expectations.

  - confidence: 0.9
