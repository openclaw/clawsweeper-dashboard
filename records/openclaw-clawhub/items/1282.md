## Summary

Keep open: current main still intentionally rejects GitHub OAuth sign-in for permanently deleted/deactivated accounts, and the requested restore/unbind path remains a security-sensitive account-lifecycle decision. ClawHub cleanup policy also keeps issues open rather than auto-closing them from source inspection alone.

## What This Changes

The reporter asks ClawHub staff to restore, unbind, or hard-delete a deleted GitHub-linked account so the same GitHub identity can sign in again.

## Best Possible Solution

Define one supported deleted-account lifecycle: either keep deletion terminal with a documented support response, or add a narrow audited restore/unlink/recreate flow with GitHub ownership verification, moderation safeguards, docs, and regression tests.

## Reproduction Assessment

Yes, at source level. A GitHub-linked user who runs deleteAccount gets deactivatedAt/purgedAt, and the later OAuth callback rejects that user through handleDeletedUserSignIn; I did not inspect the reporter's production row.

## Solution Assessment

No for ad hoc hard deletion or unbinding as the durable solution. The safer path is a maintainer-approved account lifecycle policy with audited recovery authority, identity verification, and ban/deletion separation.
