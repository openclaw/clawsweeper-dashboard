## Label Justifications

- `P1`: The reporter is blocked from ClawHub sign-in after deletion, and several related open reports show this affects real users.
- `impact:security`: Restoring or unlinking a deleted account affects OAuth identity ownership, deletion guarantees, audit history, and moderation bypass boundaries.
- `impact:auth-provider`: The reported failure is specifically tied to GitHub OAuth identity binding after account deletion.
- `impact:session-state`: The user cannot move from OAuth completion to an active ClawHub session because the linked account is inactive.

## Summary

Keep open: current main still intentionally blocks GitHub sign-in for permanently deleted ClawHub accounts, the requested restore/unbind path is unresolved, and ClawHub cleanup policy reserves auto-close for implemented PRs rather than issues.

## What This Changes

The issue requests restoring, hard-deleting, or unbinding a deleted ClawHub account so the same GitHub identity can sign in again.

## Best Possible Solution

Define the account lifecycle explicitly: keep deletion irreversible with a documented support response, or add a narrow audited restore/unbind flow that verifies GitHub ownership and preserves moderation, privacy, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level: a GitHub-linked user who runs deleteAccount gets deactivatedAt and purgedAt, and a later OAuth callback invokes handleDeletedUserSignIn and rejects that user. I did not inspect the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved lifecycle policy with identity verification, audit requirements, and ban/deletion separation.
