## Summary

Keep open: current main still intentionally blocks GitHub OAuth reauth for permanently deleted accounts, and the requested restore/unbind/hard-delete path is a security-sensitive production account-lifecycle decision rather than a cleanup close for a ClawHub issue.

## What This Changes

The reporter requests restoration, hard deletion, or unbinding of their deleted ClawHub account's GitHub identity so they can sign in again.

## Best Possible Solution

Define the deleted-account lifecycle explicitly: keep deletion irreversible with a documented support response, or add a narrow audited restore/unbind flow that verifies GitHub ownership and preserves moderation, privacy, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level: a GitHub-linked user who runs `deleteAccount` gets `deactivatedAt` and `purgedAt`, and a later OAuth callback invokes `handleDeletedUserSignIn` and rejects that user. I did not verify the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved lifecycle policy with identity verification, audit requirements, and ban/deletion separation.
