## Summary

Keep open. Current main does not implement the security-category moderation path, so this PR is not obsolete, but the patch is not merge-ready because it adds an unrelated npm lockfile and changes a security-sensitive moderation trust boundary that needs maintainer review.

## What This Changes

This PR adds `category: security` moderation metadata handling, rewrites selected static reason codes to `info.security_context.*`, passes category context to LLM evaluation, updates docs/tests, and adds `package-lock.json`.

## Best Possible Solution

Rebase onto current main, remove the npm lockfile, preserve current VT/static/LLM safeguards, and land the category system only after maintainers approve the trust model and tests/docs match that decision.

## Reproduction Assessment

Not applicable as a feature PR rather than a runtime bug. Current-main verification is high confidence: source and docs searches show the category system is absent, while the PR diff context shows it being added.

## Solution Assessment

No, this is not yet the best mergeable solution. The feature direction may address security-tool false positives, but this branch needs rebase work, lockfile removal, and explicit maintainer/security agreement on fail-open category behavior.
