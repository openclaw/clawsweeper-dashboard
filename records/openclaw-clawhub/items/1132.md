## Summary

Keep open: current main does not implement publisher-declared security categories in the moderation pipeline, and this PR is still a live feature/security-policy proposal rather than obsolete cleanup. The patch is not merge-ready because it adds an unrelated npm lockfile to a Bun-only repo and needs maintainer/security review after rebasing onto the current moderation engine.

## What This Changes

This PR adds a `category: security` moderation context that rewrites selected static scan reason codes to `info.security_context.*`, passes category context into LLM evaluation, updates docs/tests, and adds a `package-lock.json`.

## Best Possible Solution

Rebase this feature onto the current moderation engine, remove the npm lockfile, and have maintainers explicitly approve the category trust model before landing focused tests and docs.

## Reproduction Assessment

Not applicable as a current-main bug reproduction; this is a feature PR. The merge blocker is source-reproducible from the PR diff: `package-lock.json` is added even though the repository uses `bun.lock` and enforces Bun installs.

## Solution Assessment

No. The proposed direction may be useful, but this PR should not merge until it is rebased onto current main, the npm lockfile is removed, and maintainers settle the security policy for category-based contextualization.
