## Summary

Keep open. Current main does not implement the publisher-declared security-category moderation path, and this PR still changes a security-sensitive scanner trust boundary while remaining conflicted, missing real behavior proof, and carrying an unrelated lockfile.

## What This Changes

This PR adds a `category: security` skill metadata path that rewrites selected static moderation reason codes to `info.security_context.*`, passes category context to the LLM evaluator, updates docs/tests, and adds `package-lock.json`.

## Best Possible Solution

Maintainers should either bless the shipped static-advisory/review-bucket model as the canonical fix, or land a rebased category/allowlist design with narrow safety floors, matching docs, focused tests, and real behavior proof.

## Reproduction Assessment

Not applicable as a current-main bug reproduction; this is a feature PR for new moderation metadata and policy. Source inspection shows current main lacks the proposed category path, while related false-positive examples are now partly mitigated by different ClawScan advisory behavior.

## Solution Assessment

No. The direction may be useful, but this branch is not the best mergeable form as-is because it changes a security policy boundary, conflicts with current main, includes an unrelated lockfile, has documentation that does not fully describe the LLM confirmation gate, and lacks real behavior proof.
