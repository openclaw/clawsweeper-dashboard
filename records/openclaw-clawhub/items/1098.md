## Summary

Keep open. The canonical `https://clawhub.ai` API is healthy on current main, but the reported legacy hostname still returns Vercel `DEPLOYMENT_NOT_FOUND`, and source inspection shows `clawhub explore` can still fail if the resolved registry is that host.

## What This Changes

The issue reports that `clawhub explore` stopped working because the skills list was being requested through `registry.clawhub.ai`, which returns `DEPLOYMENT_NOT_FOUND`.

## Best Possible Solution

Restore or redirect `registry.clawhub.ai` to the canonical ClawHub deployment, then consider a narrow CLI compatibility fallback for non-explicit legacy registry values while preserving intentional custom registries.

## Reproduction Assessment

Yes. The live legacy endpoint still returns `DEPLOYMENT_NOT_FOUND`, and current source shows `cmdExplore` will request `/api/v1/skills` from whichever registry is resolved; I did not run the CLI itself because this review is read-only.

## Solution Assessment

Yes for the primary direction. Restoring or redirecting the legacy hostname is the narrow compatibility fix; any CLI fallback is secondary and needs policy judgment around explicit custom registry settings.
