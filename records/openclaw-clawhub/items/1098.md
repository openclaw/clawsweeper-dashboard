## Summary

Keep open. Current main and the live canonical API now use `https://clawhub.ai`, but the legacy `registry.clawhub.ai` host reported in the issue still returns Vercel `DEPLOYMENT_NOT_FOUND`, and ClawHub issue cleanup should stay open for maintainer follow-up rather than auto-close.

## What This Changes

Report that `clawhub explore` stopped working because registry traffic reached `registry.clawhub.ai`, which returns Vercel `DEPLOYMENT_NOT_FOUND`, while `clawhub search` still works.

## Best Possible Solution

Restore or redirect `registry.clawhub.ai` to the canonical ClawHub deployment, then consider a narrow CLI compatibility patch for cached non-explicit legacy registry values.

## Reproduction Assessment

Yes for the legacy-host failure: `registry.clawhub.ai/api/v1/skills?limit=3` returns HTTP 404 with `DEPLOYMENT_NOT_FOUND`, and source shows explore depends on the resolved registry. I did not run the CLI because this review is read-only and current defaults now point at the canonical host.

## Solution Assessment

No patch is proposed, and current main only mitigates the default path by using `https://clawhub.ai`. The most maintainable fix is domain restoration/redirect first, with any CLI legacy-cache behavior handled as a small compatibility decision.
