## Summary

Keep open. The canonical `https://clawhub.ai` API is healthy and current main defaults discovery there, but the reported legacy `registry.clawhub.ai` hostname still returns Vercel `DEPLOYMENT_NOT_FOUND`, and explicit legacy registry overrides can still make `clawhub explore` hit that broken host.

## What This Changes

The issue reports that `clawhub explore` stopped working because it reaches `registry.clawhub.ai`, which returns Vercel `DEPLOYMENT_NOT_FOUND`, while search still works.

## Best Possible Solution

Restore or redirect `registry.clawhub.ai` to the canonical ClawHub deployment for backward compatibility, then consider a narrow CLI migration for non-explicit legacy cached values while preserving intentional custom registries.

## Reproduction Assessment

Yes. The live legacy endpoint still returns `DEPLOYMENT_NOT_FOUND`, and source inspection shows `cmdExplore` builds `/api/v1/skills` from the resolved registry, so an explicit legacy registry setting will reproduce the failure path.

## Solution Assessment

Yes for the primary direction. Restoring or redirecting the legacy hostname is the narrow compatibility fix; CLI fallback behavior is secondary and needs maintainer judgment around cached versus explicit registry overrides.
