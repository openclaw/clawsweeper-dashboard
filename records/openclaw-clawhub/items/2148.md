## Summary

Keep open: the UI consolidation is useful, but the PR introduces a source-reproducible /import auth-loading regression and GitHub reports the branch as merge-conflicting.

## What This Changes

The PR adds a shared SignInPrompt component, rewires signed-out auth-gated routes and comment prompts to use it, and adds component/stars route tests.

## Best Possible Solution

Update the branch against current main and keep the shared SignInPrompt only where it preserves each route's auth-loading and error behavior, with a focused /import loading-state test.

## Reproduction Assessment

Yes, source-reproducible: /import currently distinguishes auth loading from signed-out using isLoading, but the PR removes that branch and renders SignInPrompt whenever isAuthenticated is false.

## Solution Assessment

No: the shared component is a maintainable direction, but /import must retain a loading branch or SignInPrompt must support loading/disabled behavior before this is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve import auth loading state:** `src/routes/import.tsx:222`
  - body: On `/import`, `useAuthStatus()` can report `isAuthenticated: false` while auth is still loading; main uses `isLoading` here to show `Loading...` and withhold the sign-in button. This change drops that distinction, so already signed-in users can briefly see an enabled sign-in prompt before auth resolves. Keep the loading branch or pass a disabled/loading action until auth settles.
