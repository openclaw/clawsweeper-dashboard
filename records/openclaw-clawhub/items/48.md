## Summary

Keep open: ClawHub cleanup policy does not auto-close issues, and current main still has a source-reproducible path where `skill publish .` can resolve against a configured global workdir instead of the caller's skill folder. The related open report at https://github.com/openclaw/clawhub/issues/1509 covers the same user-visible failure and should be fixed with the same focused CLI change.

## What This Changes

The issue reports that `clawhub publish .` fails with `SKILL.md required` even when the target skill directory contains a valid `SKILL.md`.

## Best Possible Solution

Skill publish should treat user-entered relative source paths as relative to the caller's current directory when that resolves to a skill folder, while preserving absolute paths and existing configured-workdir behavior for install/update/sync flows.

## Repair Work Prompt

Fix the remaining `clawhub skill publish .` / legacy `clawhub publish .` relative-path bug tracked by https://github.com/openclaw/clawhub/issues/48 and https://github.com/openclaw/clawhub/issues/1509. Current `cmdPublish` resolves `folderArg` with `resolve(opts.workdir, folderArg)`, so a configured Clawdbot/OpenClaw default workspace can make `.` scan the wrong folder and throw `SKILL.md required` even when the caller's cwd contains `SKILL.md`. Implement a conservative helper for publish source resolution only: absolute paths should stay absolute; relative publish source paths that resolve to an existing skill folder from `process.cwd()` should use that cwd-relative folder; preserve configured-workdir semantics for install/update/sync and avoid broad global workdir changes. Add regression coverage for a cwd skill folder with a valid `SKILL.md` while `opts.workdir` points elsewhere, plus any alias coverage needed for the legacy top-level publish command. Do not undo the existing root-manifest recovery from the closed manifest PR, and do not broaden ignored-file upload behavior beyond the required root manifest. Update specs only if the intended publish path precedence is clarified beyond existing docs.
