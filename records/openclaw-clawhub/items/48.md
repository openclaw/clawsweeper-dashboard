## Summary

Keep open: current main still has a source-reproducible CLI path where skill publish can scan the resolved global workdir instead of the caller’s skill folder, and repository policy keeps ClawHub issues open rather than auto-closing them. The related open report at https://github.com/openclaw/clawhub/issues/1509 appears to share the same user-visible failure.

## What This Changes

The issue reports CLI publish failing with `SKILL.md required` even when the selected source directory contains a valid `SKILL.md`.

## Best Possible Solution

Skill publish should resolve user-entered relative source paths against the caller’s cwd when that points to a skill folder, while preserving absolute paths and existing configured-workdir semantics for install/update/sync flows.

## Reproduction Assessment

Yes, by source inspection. With a configured Clawdbot/OpenClaw default workspace different from an unmarked cwd, `clawhub skill publish .` can scan the configured workspace and throw `SKILL.md required` even when cwd contains `SKILL.md`.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust only skill-publish source-path resolution and add regression coverage for a cwd skill folder with a different configured/global workdir, without changing sync/install workdir behavior.
