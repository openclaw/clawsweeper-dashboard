## Summary

## What This Changes

The issue asks for `clawhub`/`clawdhub publish .` to publish a valid skill directory instead of failing with `SKILL.md required` when `SKILL.md` exists.

## Best Possible Solution

Skill publish should resolve user-entered relative source paths from the caller's cwd when they identify a skill folder, preserve existing workspace-driven flows, and add focused regression coverage.

## Reproduction Assessment

Yes, from source inspection. With a configured Clawdbot/OpenClaw default workspace that differs from cwd and no cwd marker, `skill publish .` can scan the configured workspace and throw `SKILL.md required` even when cwd contains `SKILL.md`.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust publish source-path resolution and add regression tests for cwd skill folders with a different configured/global workdir, without changing install/update/sync semantics broadly.

## Repair Work Prompt

Fix the CLI skill publish path-resolution bug for the current issue and related #1509. On current main, `cmdPublish` resolves `folderArg` with `resolve(opts.workdir, folderArg)`, so `skill publish .` can scan a configured Clawdbot/OpenClaw workspace rather than `process.cwd()` after the user cd's into a skill folder. Implement a narrow publish-only resolver for user-entered relative sources: when cwd-relative resolution points to an existing skill folder containing `SKILL.md`/`skill.md`/`skills.md`, publish that folder; preserve absolute paths, existing workdir-relative publish behavior where needed, sync calls that pass absolute `skill.folder`, and package publish semantics. Add regression coverage for a cwd skill folder with a different configured/global workdir plus existing workdir-relative behavior. Do not change install/update/sync workspace semantics broadly, and note if the absolute-path timeout needs a separate follow-up.
