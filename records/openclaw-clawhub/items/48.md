## Summary

Keep open: ClawHub cleanup policy keeps issues open, and current `main` still has a credible source-level path to the reported `SKILL.md required` failure when `skill publish .` is run from a skill folder while the resolved global workdir points elsewhere.

## What This Changes

The issue reports that CLI skill publishing fails with `SKILL.md required` even when the target skill directory contains a valid `SKILL.md`.

## Best Possible Solution

Skill publish should treat user-entered relative source paths as relative to the caller's current directory when that resolves to a skill folder, while preserving absolute paths and existing configured-workdir flows.

## Reproduction Assessment

Yes, from source inspection. With a configured Clawdbot/OpenClaw default workspace that differs from cwd and no cwd marker, `skill publish .` can scan the configured workspace and throw `SKILL.md required` even when cwd contains `SKILL.md`.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust publish source-path resolution and add regression tests for cwd skill folders with a different configured/global workdir, without changing install/update/sync semantics broadly.

## Repair Work Prompt

Fix the CLI skill publish path-resolution bug reported here and in #1509. `clawhub skill publish .` and legacy `clawhub publish .` should publish the skill folder in the caller's current directory when that relative path resolves to a folder containing `SKILL.md`, even if `resolveWorkdir` selected a configured Clawdbot/OpenClaw workspace elsewhere. Preserve absolute path behavior, `--workdir` / `CLAWHUB_WORKDIR` semantics where intentionally used, and `sync` flows that pass absolute skill folders. Add focused regression coverage in `packages/clawhub/src/cli/commands/publish.test.ts`; update docs only if the resulting precedence rule needs clarification. Do not broaden install/update/sync workdir semantics beyond what is necessary for publish source paths.
