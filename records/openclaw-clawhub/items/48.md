## Summary

Keep open: current main still has a source-reproducible CLI path where `skill publish .` can inspect the resolved global workdir instead of the caller's skill directory. The related broad-ignore manifest fix landed in `clawhub@0.12.3` via https://github.com/openclaw/clawhub/pull/1856, but that does not change this relative-path/workdir behavior, and ClawHub policy keeps issues open for maintainer follow-up.

## What This Changes

Fix CLI publish so running `clawhub`/`clawdhub publish .` from a skill directory containing `SKILL.md` does not fail with `SKILL.md required`.

## Best Possible Solution

Skill publish should resolve a user-entered relative source path against the caller's cwd when that path is an existing skill folder, while preserving absolute paths and configured-workdir behavior for install/update/sync flows, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, from source inspection. With a configured Clawdbot/OpenClaw default workspace that differs from cwd and no cwd marker, `clawhub skill publish .` can scan the configured workspace and throw `SKILL.md required` even when cwd contains `SKILL.md`.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust only skill-publish source-path resolution and add regression coverage for a cwd skill folder with a different configured/global workdir, without broad workdir or sync behavior changes.
