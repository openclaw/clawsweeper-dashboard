## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still has a source-reproducible CLI path where `skill publish .` resolves the source folder under the global workdir rather than the caller's skill directory.

## What This Changes

The issue reports `clawhub`/`clawdhub publish .` throwing `SKILL.md required` even when the current directory contains a valid `SKILL.md`.

## Best Possible Solution

Fix `skill publish` to resolve user-entered relative source paths to the caller's intended skill folder when that folder contains a root manifest, while preserving absolute paths and non-publish workdir semantics, then add regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: run `clawhub skill publish .` from an unmarked skill directory with `SKILL.md` while a configured Clawdbot/OpenClaw default workspace points elsewhere, and publish can inspect the wrong folder before throwing `SKILL.md required`.

## Solution Assessment

Yes. The narrow maintainable fix is publish-specific source-path resolution plus regression coverage, not backend validation changes or a broad rewrite of global workdir behavior.

## Repair Work Prompt

Fix the ClawHub CLI publish bug where `clawhub skill publish .` or legacy `clawhub publish .` can throw `SKILL.md required` even though the caller's current directory contains `SKILL.md`, because `cmdPublish` resolves relative paths under `opts.workdir`. Keep the fix publish-specific: preserve absolute paths, preserve explicit/env workdir behavior where intentionally used, do not change install/update/sync global workdir semantics, and keep the existing `ensureRootManifestFile` broad-ignore recovery. Add a regression test where the process cwd is a skill folder with `SKILL.md` and `GlobalOpts.workdir` points at another directory lacking a manifest; publishing `.` should upload the cwd manifest. Include an Unreleased Fixes changelog entry if this remains user-facing release work. Stop and ask for maintainer input if the implementation would require a new flag, config option, or broad workdir policy change.
