## Summary

Keep open: this remains a valid, narrow CLI bug on current main, and ClawHub cleanup policy does not auto-close issues. The current publish flow can still resolve `.` under a configured global workspace instead of the caller's skill directory, which explains the reported `SKILL.md required` failure.

## What This Changes

Fix CLI publish so a folder containing a valid `SKILL.md` is not rejected with `SKILL.md required`.

## Best Possible Solution

Skill publish should resolve user-entered relative source paths against the caller's current directory when that identifies a skill folder, while preserving absolute paths and existing configured-workdir flows elsewhere.

## Reproduction Assessment

Yes, from source inspection. With a configured Clawdbot/OpenClaw default workspace that differs from cwd and no cwd marker, `skill publish .` can scan the configured workspace and throw `SKILL.md required` even when cwd contains `SKILL.md`.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust publish source-path resolution and add regression coverage for cwd skill folders with a different configured/global workdir, without changing install/update/sync semantics broadly.

## Repair Work Prompt

Fix the CLI skill publish path-resolution bug reported here and in openclaw/clawhub#1509. When a user runs `clawhub publish .` or `clawhub skill publish .` from a directory containing `SKILL.md`, publish should scan that cwd skill folder even if the resolved global workdir comes from Clawdbot/OpenClaw config. Preserve absolute path behavior, preserve explicit configured-workdir flows where they are intentionally used by install/update/sync, and avoid broad global workdir rewrites. Add regression coverage for a cwd skill folder with a different configured/default workspace and for existing workdir-relative publish behavior. Add a user-facing `CHANGELOG.md` fix entry. Do not change package publishing or plugin detection semantics.
