## Summary

Keep open. Current main still has a source-backed path to the reported `SKILL.md required` failure: publish resolves relative source paths under the resolved global workdir, which can differ from the caller's skill directory. Related #1509 remains open with the same user-visible failure.

## What This Changes

The issue asks for `clawhub publish .` or the legacy `clawdhub` alias to publish a folder containing a valid `SKILL.md` instead of failing with `SKILL.md required`.

## Best Possible Solution

Publish should resolve a user-entered relative source path against the caller's current directory when that path points to a skill folder, with regression coverage for a cwd skill folder and a different configured Clawdbot/OpenClaw workdir.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run publish from a directory containing `SKILL.md` while the resolved global workdir points elsewhere, and the command can inspect the wrong folder before throwing `SKILL.md required`.

## Solution Assessment

Yes. A focused publish-only path resolution fix with regression tests is narrower than changing global workdir behavior across install, update, sync, and configured workspace flows.

## Repair Work Prompt

Fix the CLI publish path-resolution bug where `clawhub publish .` / `clawhub skill publish .` can throw `SKILL.md required` even when the caller's current directory contains a valid `SKILL.md`, because the publish command resolves relative source paths under the resolved global workdir. Keep the change scoped to publish source-folder resolution: relative publish arguments that point to a skill folder in the caller's cwd should be honored, absolute paths should keep working, and install/update/sync/global workdir semantics should not be broadly changed. Add or update regression tests covering a cwd skill folder with `SKILL.md` while the configured Clawdbot/OpenClaw default workspace or command workdir differs. Add an appropriate `CHANGELOG.md` entry for the user-facing CLI fix. Mention related issue #1509 in the PR summary, and stop for maintainer input if the repair would require a new flag or broad product-wide precedence change.
