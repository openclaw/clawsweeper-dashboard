## Summary

Keep open: current main still has a source-reproducible CLI path where `skill publish .` can resolve `.` under the resolved global workdir instead of the caller's skill directory, then throw `SKILL.md required`. ClawHub cleanup policy also keeps issues open, and the related open report at https://github.com/openclaw/clawhub/issues/1509 appears to share the same user-visible failure.

## What This Changes

The issue requests a fix for `clawhub`/`clawdhub publish` reporting `SKILL.md required` even when the skill folder contains a valid root `SKILL.md`.

## Best Possible Solution

Skill publish should resolve a user-entered relative source path against the caller's current directory when that points to a skill folder, while preserving absolute paths and existing configured-workdir behavior for install/update/sync flows.

## Reproduction Assessment

Yes, by source inspection. With a configured Clawdbot/OpenClaw default workspace different from an unmarked cwd, `clawhub skill publish .` scans the configured workspace and can throw `SKILL.md required` even when cwd contains `SKILL.md`.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust only skill-publish source-path resolution and add regression coverage for a cwd skill folder with a different configured/global workdir, without changing sync/install workdir semantics.

Preserve absolute path behavior and avoid broad changes to install/update/sync or package publish workdir semantics.
