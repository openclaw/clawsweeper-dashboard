## Summary

Keep open: current main still has a source-reproducible CLI path where relative skill publish paths are resolved under the resolved global workdir, so a configured Clawdbot/OpenClaw workspace can make `skill publish .` scan the wrong folder and throw `SKILL.md required`. The 0.12.3 broad-ignore manifest recovery is present, but it does not address this relative-path/workdir behavior.

## What This Changes

The issue reports that `clawhub publish .` fails with `SKILL.md required` even when the caller's directory contains a valid `SKILL.md`.

## Best Possible Solution

Skill publish should resolve a user-entered relative source path against the caller's cwd when that path is an existing skill folder, while preserving absolute paths and configured-workdir behavior for install/update/sync flows, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection shows that with a configured Clawdbot/OpenClaw default workspace different from cwd and no cwd marker, `clawhub skill publish .` can scan the configured workspace and throw `SKILL.md required` even when cwd contains `SKILL.md`.

## Solution Assessment

Yes. The narrow maintainable solution is to adjust only skill-publish source-path resolution and add regression coverage for a cwd skill folder with a different configured/global workdir, without changing broad workdir semantics for sync or package publish.
