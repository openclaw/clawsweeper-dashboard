## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-backed path to the reported `SKILL.md required` failure because relative publish paths are resolved under the global workdir instead of necessarily under the caller's skill directory.

## What This Changes

The issue reports that `clawhub publish .` fails with `SKILL.md required` even when `SKILL.md` exists in the current skill folder.

## Best Possible Solution

Skill publish should resolve user-entered relative source paths against the caller's cwd when that path points to a skill folder, while preserving absolute paths and configured-workdir behavior for other CLI flows.

## Reproduction Assessment

Yes. A high-confidence source-backed path is a cwd containing `SKILL.md` plus a configured Clawdbot/OpenClaw default workspace elsewhere; current main resolves `publish .` under `opts.workdir` and can scan the wrong folder.

## Solution Assessment

Yes. A focused publish-only path resolution fix with regression coverage is narrower and safer than changing global workdir semantics across install, update, and sync.

## Repair Work Prompt

Fix the `clawhub skill publish .` / legacy `clawhub publish .` path-resolution bug reported here and in #1509. The observable failure is `SKILL.md required` even though the caller's current directory contains `SKILL.md`, because `cmdPublish` resolves relative source paths against `opts.workdir`, which may be a configured Clawdbot/OpenClaw workspace. Keep the change focused to skill publish: resolve relative publish source paths against the caller's cwd when that resolves to a skill folder, preserve absolute paths, and do not broadly change install/update/sync workdir semantics. Add regression coverage for a cwd skill folder while global/configured workdir points elsewhere. Add a user-facing `CHANGELOG.md` Unreleased fix entry. Do not change package publish behavior or registry upload semantics.
