## What This Changes

The issue asks for `clawhub sync` to stop showing proposed semver-bumped upload versions as if they were available registry updates.

## Best Possible Solution

Update the sync planning output and upload-selection label so unmatched local content is described as local changes relative to the registry latest, with the bumped version labeled only as the proposed upload version, and add regression coverage for that state.

Yes. A high-confidence source-level reproduction exists: mock registry resolution with `latestVersion` present and `match: null`, then observe `formatActionableLine` render `UPDATE <latest> → <semver.inc(latest)>`.

## Repair Work Prompt

Fix the `clawhub sync` reporting bug for installed local skills whose `/api/v1/resolve` result has `latestVersion` but `match: null`. In `packages/clawhub/src/cli/commands/syncHelpers.ts`, stop presenting `semver.inc(latest, bump)` as an available registry update in the `To sync` output and the interactive upload-selection label; describe the local content as differing from the registry latest and label the bumped version only as the proposed upload version. Preserve `resolvePublishMeta` and the actual upload semver bump behavior. Add regression tests for dry-run output and prompt labels using the existing `update-skill` mock state, and ensure the output no longer implies that `1.0.1` exists when the registry latest is `1.0.0`. Keep the dashboard drift warning tracked separately in #1180. Add a `CHANGELOG.md` fix entry for the CLI-facing output change.
