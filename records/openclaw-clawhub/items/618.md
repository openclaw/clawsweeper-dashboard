## Summary

Keep open. Current main still has the reported CLI presentation bug: unmatched local content is treated as an update candidate and the sync report shows a semver-bumped proposed upload version as if it were an available registry version.

## What This Changes

The issue asks for `clawhub sync` to report actual registry latest versions instead of semver-bumped proposed upload versions.

## Best Possible Solution

Update the sync planning output and interactive upload-selection label so unmatched local content is described as local changes relative to registry latest, with the bumped version clearly labeled as a proposed upload version.

Yes. A high-confidence source-level reproduction exists by mocking `/api/v1/resolve` with `latestVersion` present and `match: null`, then observing `formatActionableStatus` render `UPDATE <latest> → <semver.inc(latest)>`.

## Solution Assessment

Yes. The narrowest maintainable fix is to change the CLI reporting and prompt wording plus add regression assertions for this state, without changing publish metadata resolution or registry APIs.

## Repair Work Prompt

Fix the `clawhub sync` version display bug for unmatched local skill content. In current main, `/api/v1/resolve` with `latestVersion` present and `match: null` is classified as `update`; the sync report and upload-selection label then render `UPDATE <latest> → <semver.inc(latest)>`, which makes the proposed upload version look like an available registry update. Keep the publish/upload behavior and semver bump calculation intact, but change the planning output and interactive label so the registry latest and proposed upload version have distinct roles. Add regression coverage for the `latestVersion` present plus `match: null` case, including non-interactive report output and interactive selection label if feasible. Do not change the dashboard drift warning tracked separately in #1180, registry API schemas, or publish metadata resolution.
