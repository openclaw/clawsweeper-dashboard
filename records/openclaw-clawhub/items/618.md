## What This Changes

## Best Possible Solution

Keep the issue open until the CLI report and interactive selection label distinguish registry latest versions from proposed upload versions, with regression coverage for `latestVersion` present and `match: null`.

Yes. Source inspection gives a high-confidence path: a resolve response with `latestVersion: 1.0.0` and `match: null` becomes an update candidate, then sync formats it as `UPDATE 1.0.0 -> 1.0.1`.

Yes. The narrow maintainable fix is to change the sync report and picker label semantics while preserving the semver bump that is later used for publishing.

When `/api/v1/resolve` returns a registry `latestVersion` but `match: null`, the sync planning output and interactive upload picker must not present `semver.inc(latestVersion, bump)` as an available registry update.
