## Summary

Keep open. Current main still treats user-initiated deletion as permanent and blocks GitHub OAuth reauth for deactivated accounts; the merged UI error work only surfaces that failure, while the requested recovery/re-registration path remains a security-sensitive product decision.

## What This Changes

Request recovery from a deleted-account GitHub OAuth loop by restoring, re-linking, or allowing re-registration so web and CLI login work again.

## Best Possible Solution

Keep this as the canonical deleted-account GitHub OAuth recovery triage item and implement a maintainer-approved recovery policy consistently across web OAuth, CLI token creation, account deletion, auth binding, operator flows, user-facing errors, docs, and regression tests.

## Reproduction Assessment

Yes. The source-level path is clear: deleteAccount sets deactivatedAt/purgedAt, then OAuth completion calls handleDeletedUserSignIn and rejects the user, so CLI token creation never proceeds because the app remains unauthenticated.

## Solution Assessment

No. Automatic re-registration or identity unlinking is not the best fix until maintainers approve the security, privacy, audit, and support semantics for deleted-account recovery.
