## Summary

Keep open. Current main still treats user-initiated deletion as terminal and blocks GitHub OAuth reauth, while the discussion and related open reports show a real account-recovery/support problem that needs security-sensitive product and operator policy rather than cleanup closure.

## What This Changes

The issue requests that users who deleted their ClawHub account can sign in again with GitHub, either by recreating, relinking, restoring, or unlinking the prior account state.

## Best Possible Solution

Keep this as the canonical deleted-account GitHub OAuth recovery triage item and implement a maintainer-approved policy across web OAuth, deletion, auth account binding, CLI token creation, operator flow, docs/specs, and regression tests.

## Reproduction Assessment

Yes, from current source inspection rather than a live production account. Deletion writes `deactivatedAt`/`purgedAt`, OAuth rejects that state, and the CLI token route cannot proceed without an authenticated user.

## Solution Assessment

No for an automated source-only repair. Recreating, restoring, or unlinking accounts changes deleted-account recovery and GitHub identity-binding semantics, so the narrowest maintainable fix first needs explicit product/security policy.
