## Summary

Keep open. Current main intentionally blocks deleted-account reauth, the issue has repeated affected-user reports and an open related cluster, and ClawHub cleanup policy keeps issues open; the remaining work is a security/product decision about account recovery rather than conservative cleanup.

## What This Changes

Users want GitHub sign-in to recover, unlink, or cleanly recreate a ClawHub account after user-initiated account deletion.

## Best Possible Solution

Define a maintainer-approved account lifecycle policy and implement it consistently across OAuth callbacks, account deletion, auth account bindings, operator tools, docs/specs, user-facing errors, CLI token creation, and regression tests.

## Reproduction Assessment

Yes, by source inspection: deleting an account writes `deactivatedAt`/`purgedAt`, OAuth re-entry calls `handleDeletedUserSignIn`, and that helper rejects the user before a usable `me` state or CLI token path exists. I did not run a live OAuth flow or inspect production rows.

## Solution Assessment

No, treating this as a narrow OAuth loop fix is not the best path. The maintainable solution needs an explicit recovery/unlink/re-registration policy before changing identity binding, deletion privacy, audit, and token semantics.
