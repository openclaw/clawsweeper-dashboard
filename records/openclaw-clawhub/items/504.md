## Summary

Keep open. Current main still intentionally rejects OAuth sign-in for deleted/deactivated accounts, the issue has multiple affected-user confirmations and related open reports, and ClawHub cleanup policy keeps issues open unless a PR is certainly implemented on main.

## What This Changes

Request that GitHub OAuth sign-in after user-initiated ClawHub account deletion restore, relink, or recreate a usable account instead of returning to a logged-out state.

## Best Possible Solution

Define and implement a maintainer-approved deleted-account recovery or re-registration policy across OAuth callback handling, user deletion state, auth account bindings, operator tooling, user-facing errors, docs/specs, and regression coverage.

## Reproduction Assessment

Yes for the core blocked path by source inspection: a deleted/deactivated user signing in through GitHub reaches `handleDeletedUserSignIn`, which rejects the session and prevents CLI token creation. I did not run a live OAuth flow with production account state, but the code and tests make the behavior high-confidence.

## Solution Assessment

Unclear. Auto-restoring or auto-recreating accounts is not obviously the safest fix because current specs make deletion irreversible; maintainers should first choose the security/privacy policy and then implement the narrow flow that matches it.
