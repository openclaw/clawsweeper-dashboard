## Summary

Keep open. Current main still blocks deleted/deactivated accounts from completing GitHub sign-in, and the requested recovery or re-registration path changes security-sensitive account deletion policy rather than being conservative cleanup.

## What This Changes

The issue asks ClawHub to let a GitHub OAuth identity sign in, re-link, or re-register after the linked ClawHub account was deleted.

## Best Possible Solution

Define and implement a maintainer-approved recovery path, or an explicit no-recovery operator process, across OAuth, deletion state, authAccounts, CLI token creation, user-facing errors, docs, audit, and regression tests.

## Reproduction Assessment

Yes by source inspection: deleteAccount sets deactivatedAt/purgedAt and handleDeletedUserSignIn rejects the later GitHub OAuth callback before a usable session exists. I did not run a live production OAuth flow or inspect reporter account rows.

## Solution Assessment

No for an automatic one-off restore, auth binding deletion, or fresh-account recreation without policy. The maintainable solution needs explicit recovery authority, ban separation, privacy semantics, audit behavior, and operator handling.
