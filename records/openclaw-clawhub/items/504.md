## Summary

Keep open. Current main still blocks deleted/deactivated account reauth by design, related reports remain open, and the remaining work is a security-sensitive account lifecycle decision rather than conservative cleanup.

## What This Changes

The issue requests recovery, re-linking, or clean re-registration after a user deletes a ClawHub account tied to GitHub OAuth.

## Best Possible Solution

Keep this as the canonical deleted-account GitHub OAuth recovery item and land a maintainer-approved account lifecycle policy across auth, deletion, CLI tokens, docs/specs, operator flows, and regression tests.

## Reproduction Assessment

Yes, by source inspection: deleting an account writes `deactivatedAt`/`purgedAt`, OAuth re-entry calls `handleDeletedUserSignIn`, and the helper rejects the user before `me` or CLI token creation can succeed. I did not run a live OAuth flow or inspect production rows.

## Solution Assessment

No, an OAuth-only restore patch is not the best path yet. The maintainable solution needs an explicit recovery, unlink, or re-registration policy that preserves identity proof, deletion privacy, token revocation, audit, and banned-account boundaries.
