## Summary

Keep open. Current main still rejects deleted/deactivated account reauth, tests and docs encode that behavior, and the affected-user thread is asking for a security-sensitive account recovery or identity-unlink policy rather than conservative cleanup.

## What This Changes

The issue asks for users who deleted a ClawHub account to recover, unlink, or re-register with the same GitHub OAuth identity instead of remaining logged out after OAuth completes.

## Best Possible Solution

Keep this as the canonical deleted-account GitHub OAuth recovery item and implement a maintainer-approved account lifecycle policy across OAuth, deletion, CLI tokens, auth binding, operator flows, docs/specs, and regression tests.

## Reproduction Assessment

Yes, by source inspection: deleting an account writes deactivatedAt/purgedAt, OAuth re-entry calls handleDeletedUserSignIn, and active-user helpers reject deleted or deactivated users before web or CLI token access can succeed. I did not run a live OAuth flow or inspect production rows.

## Solution Assessment

No, a narrow OAuth-only restore patch is not the best path yet. The maintainable solution needs an explicit recovery, unlink, or re-registration policy that preserves identity proof, deletion privacy, token revocation, audit, and banned-account boundaries.
