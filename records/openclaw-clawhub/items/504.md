## Label Justifications

- `P1`: Multiple users are blocked from GitHub sign-in and CLI token access after account deletion, which is an urgent auth workflow issue.
- `impact:security`: The remaining decision changes deleted-account recovery, identity binding, token revocation, audit, and privacy boundaries.
- `impact:auth-provider`: The issue is specifically about GitHub OAuth reauthentication mapping back to a deleted ClawHub account.
- `impact:session-state`: Affected users complete OAuth but do not get an active ClawHub session or usable CLI token state.

## Summary

Keep open. Current main and the latest v0.16.0 tag still treat user-initiated account deletion as irreversible and reject GitHub OAuth reauth for deleted/deactivated users, while the live issue has multiple same-symptom reports and related open issues.

## What This Changes

The issue requests that GitHub OAuth sign-in recover, relink, or recreate access after a user deletes their ClawHub account.

## Best Possible Solution

Define a maintainer-approved deleted-account recovery or re-registration policy and implement it consistently across OAuth, auth account binding, deletion, CLI tokens, operator/audit flows, docs, specs, and regression tests.

## Reproduction Assessment

Yes, by source inspection: deletion sets deactivatedAt/purgedAt, OAuth re-entry calls handleDeletedUserSignIn, and active-user helpers reject deleted or deactivated users before web or CLI token access can succeed. I did not run a live OAuth flow or inspect production rows.

## Solution Assessment

No, a narrow OAuth-only recreate patch is not the best path yet. The maintainable solution needs an explicit recovery, unlink, or re-registration policy that preserves identity proof, deletion privacy, token revocation, audit, and banned-account boundaries.
