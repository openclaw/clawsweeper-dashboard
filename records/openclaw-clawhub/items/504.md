## Summary

Keep open. Current main still treats user-initiated deletion as permanent and blocks GitHub reauth for deleted/deactivated users, while the live discussion shows multiple affected users; the remaining work is a security-sensitive recovery or unlink policy, not conservative cleanup.

## What This Changes

The issue asks ClawHub to allow GitHub sign-in, recovery, unlinking, or clean re-registration after a user deletes their ClawHub account and reauthorizes with GitHub.

## Best Possible Solution

Keep this as the canonical deleted-account GitHub OAuth recovery triage item and implement a maintainer-approved policy across OAuth, deletion, CLI tokens, auth binding, operator flows, user-facing errors, specs, docs, and regression tests.

## Reproduction Assessment

Yes, by source inspection: deletion sets deactivatedAt/purgedAt, OAuth re-entry calls handleDeletedUserSignIn, and active-user helpers reject deleted or deactivated users before web or CLI token access can succeed. I did not run a live OAuth flow or inspect production rows.

## Solution Assessment

No, a narrow OAuth-only restore patch is not the best path yet. The maintainable solution needs an explicit recovery, unlink, or re-registration policy that preserves identity proof, deletion privacy, token revocation, audit, and banned-account boundaries.
