## Summary

Keep open. Current main still encodes account deletion as irreversible and blocks GitHub reauth for deleted/deactivated users, while the live discussion and related open reports show multiple affected users; for ClawHub issues this should remain open for maintainer policy review rather than cleanup closure.

## What This Changes

The issue requests a way to log back in, re-link, or recreate a ClawHub account with GitHub after deleting the prior account, instead of ending in an OAuth loop with no active session or CLI tokens.

## Best Possible Solution

Keep this as the canonical deleted-account GitHub OAuth recovery item and implement a maintainer-approved policy across OAuth, deletion, CLI tokens, auth binding, operator flows, user-facing errors, specs, docs, and regression tests.

## Reproduction Assessment

Yes, by source inspection: deletion sets deactivatedAt/purgedAt, OAuth re-entry calls handleDeletedUserSignIn, and active-user helpers reject deleted or deactivated users before web or CLI token access can succeed. I did not run a live OAuth flow or inspect production rows.

## Solution Assessment

No, a narrow OAuth-only restore patch is not the best path yet. The maintainable solution needs an explicit recovery, unlink, or re-registration policy that preserves identity proof, deletion privacy, token revocation, audit, and banned-account boundaries.
