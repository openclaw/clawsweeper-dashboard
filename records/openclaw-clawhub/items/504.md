## Summary

Keep open. Current main still deliberately rejects OAuth sign-in for deleted/deactivated accounts, this issue has multiple affected-user reports and duplicate open reports, and ClawHub issue cleanup policy does not auto-close issues; the remaining change is a security/product decision about account recovery, GitHub identity binding, and deletion semantics.

## What This Changes

The issue asks ClawHub to allow users who deleted their ClawHub account to sign back in with the same GitHub identity or otherwise recover/re-register so CLI tokens and account access work again.

## Best Possible Solution

Adopt a maintainer-approved account recovery policy and implement it consistently across OAuth callbacks, account deletion, auth account bindings, audit/operator tooling, user-facing errors, docs/specs, CLI token access, and regression tests.

## Reproduction Assessment

Yes, by source inspection: deleting an account writes deactivatedAt/purgedAt, OAuth re-entry calls handleDeletedUserSignIn, and that helper rejects the user before a usable session and CLI token path exist. I did not run a live OAuth flow or inspect affected production rows.

## Solution Assessment

No, an automatic restore or purge is not clearly the best fix yet. The maintainable solution needs an explicit account recovery/security policy before changing deletion, provider identity binding, audit, and token semantics.
