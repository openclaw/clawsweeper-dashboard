## Summary

Keep open. Current main still deliberately blocks OAuth reauthentication for deleted/deactivated ClawHub accounts, and the issue discussion plus related open reports show multiple users still need an approved recovery, unlink, or operator-cleanup path.

## What This Changes

The issue asks ClawHub to recover, recreate, or relink GitHub sign-in after a user deletes their ClawHub account and then tries to authenticate again.

## Best Possible Solution

Keep this as the canonical deleted-account GitHub OAuth recovery triage item, with a maintainer-approved policy implemented consistently across OAuth, CLI tokens, deletion, auth-account binding, admin/operator flows, user-facing errors, docs, and regression tests.

## Reproduction Assessment

Yes. The source path is deterministic: delete an account with `users.deleteAccount`, then a later GitHub OAuth callback reaches `handleDeletedUserSignIn`, sees `deactivatedAt`, and rejects; `convex/auth.test.ts` directly covers that path.

## Solution Assessment

Unclear. Automatically recreating or relinking the account may satisfy affected users, but the safest maintainable solution needs a product/security decision on deletion, recovery, identity rebinding, and audit semantics before coding.
