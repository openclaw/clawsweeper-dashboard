## Summary

Keep open: this is a fresh Settings redesign PR that is not already implemented on current main, and the diff introduces a blocking org-member management regression plus a smaller Convex read-amplification issue. The screenshot proof is sufficient for the visible UI portions of the change.

## What This Changes

The branch replaces Settings with query-driven account, organization, API token, and deletion views, moves Stars into the avatar menu, updates responsive styling/tests, and adds a changelog entry.

## Best Possible Solution

Land the Settings redesign after restoring owner-member management semantics, skipping org-member reads outside the Organizations view, and adding focused regression coverage for the new navigation and org-member behavior.

## Reproduction Assessment

Yes. Source inspection of the PR branch shows owner rows always hide the Remove action, while current main and the backend allow org owners to remove other owners except the last owner.

## Solution Assessment

No. The redesign direction is maintainable, but the member-row condition should reflect the viewer's role and backend safeguards, and listMembers should be skipped outside the Organizations view.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve owner removal for org owners:** `src/routes/settings.tsx:814`
  - body: The redesigned member list hides `Remove` for every owner row. Current Settings exposes removal to org managers and the backend already permits an org owner to remove another owner while protecting the last owner, so this removes a valid owner-management path from the UI. Gate owner-row removal on the viewer role instead of the target role alone.
  - confidence: 0.91
- **[P3] Skip member loading outside the org view:** `src/routes/settings.tsx:186-189`

  - confidence: 0.84
