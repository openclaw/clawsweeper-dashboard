## Summary

This is a current feature PR, not stale cleanup and not already implemented on main, so it should stay open. I found two concrete PR-introduced issues in the Settings redesign that should be fixed before merge: an unlabeled organization selector and token-copy success feedback that can fire when copying failed.

## What This Changes

The PR redesigns the web Settings page into account, organizations, API tokens, and account deletion views, moves Stars into the signed-in avatar menu, adjusts shared select styling, and updates tests and the changelog.

## Best Possible Solution

Keep the redesign under review, fix the accessibility and copy-result handling defects, then let maintainers decide whether the new Settings UX is the desired product direction.

## Reproduction Assessment

Yes for the review findings: source inspection gives a high-confidence path for both the unlabeled organization selector and the token copy handler that reports success without knowing whether copying succeeded. The redesign itself is a feature change, not a bug reproduction target.

## Solution Assessment

No for the current patch as written: the Settings redesign may be a reasonable direction, but the org selector should be explicitly labeled and token copying should reuse the existing fallback/error-aware copy helper or equivalent handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Label the organization selector:** `src/routes/settings.tsx:505`
  - body: This `SelectTrigger` is the control that switches which organization is being managed, but it is not associated with a label or `aria-label`; its accessible name is only the selected org text. Screen reader users get the current value without the purpose of the combobox, so wrap it in the existing `Field` component or add an explicit accessible label.
  - confidence: 0.87
- **[P2] Handle clipboard copy failures before showing success:** `src/routes/settings.tsx:971-973`
  - body: This click handler fires `navigator.clipboard.writeText(newToken)` and immediately shows `Token copied`. In unsupported browsers, insecure contexts, denied permission, or rejected writes, users still get success feedback even though the token was not copied. Await/catch the operation or reuse `copyText()` so success is shown only after a real copy.
  - confidence: 0.93
