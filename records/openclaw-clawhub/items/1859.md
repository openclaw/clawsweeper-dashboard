## Summary

Keep open. This draft PR is not obsolete because current main still lacks the intended docs and CLI schema alignment, but the branch needs a correctness fix: the new moderation regression covers direct `metadata.envVars` while the docs it adds instruct authors to use nested `metadata.openclaw.envVars`, and the current static scanner still misses that nested raw metadata shape.

## What This Changes

This draft PR documents optional skill env vars as `envVars` declarations with `required: false`, adds the same shape to the CLI schema copy, updates security prompt wording and requirements output, and adds schema/moderation tests.

## Best Possible Solution

Land or replace this PR after the docs, CLI schema copy, prompt summary, static scanner, and tests all consistently support `metadata.openclaw.envVars` entries with `required: false` as declared optional env access, while keeping existing undeclared-env and broad-env security detections intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Cover the documented nested envVars shape:** `convex/lib/moderationEngine.test.ts:123`
  - body: The new regression passes `metadata.envVars` directly, but the docs added in this PR tell authors to declare optional variables under `metadata.openclaw.envVars`. The publish/rescan path passes raw frontmatter and metadata into the static scan, and `collectDeclaredEnvNames` only reads direct `envVars`/`requires.env`, so a skill following the new docs can still be flagged for undeclared env access. Please update the scanner and regression to cover `metadata: { openclaw: { envVars: [...] } }`.
