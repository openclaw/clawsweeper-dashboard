## Summary

This remains a useful but non-merge-ready feature: the new target set is defined by stored Markdown bytes rather than ClawScan’s canonical prepared-content character boundary, so operators could receive misleading security-remediation coverage; the head also conflicts with main and lacks real Convex/operator proof.

## What This Changes

This PR adds a configurable `truncation-risk-latest` mode to bulk ClawScan rescans across the backend, admin API, moderation CLI, schemas, tests, docs, and moderation specification.

## Best Possible Solution

Define eligibility with ClawScan’s shared prepared-content truncation calculation or a persisted canonical truncation signal, retain dry-run sampling and broad-mode compatibility, then prove the conflict-resolved flow on a real development deployment.

## Reproduction Assessment

Yes for the patch defect: source inspection establishes that eligibility uses stored bytes while scanner truncation uses prepared-content characters, so divergent target sets are possible without executing the branch.

## Solution Assessment

No, not as proposed: extending the existing bulk-rescan path is appropriate, but a separate byte heuristic is not the narrowest reliable definition of ClawScan truncation risk.
