## Summary

Keep open: current main does not implement the PR, and the latest head still has a blocking CLI correctness/security issue where version flags are treated as help and can trigger a tokened whoami call before local version output.

## What This Changes

The PR changes the ClawHub CLI help gate to resolve the stored user's role, hide moderator/admin root and package commands by audience, and expand admin help tests.

## Best Possible Solution

Land the role-aware help gate after exempting CLI version flags from help-role lookup, covering that with regression tests, and adding an Unreleased changelog entry.

## Reproduction Assessment

Yes from source inspection: current main exposes package moderation/operator commands in normal help, while the PR head makes -V/--cli-version fall into help-role lookup. I did not execute the PR branch because this review is read-only.

## Solution Assessment

No as currently patched: the role-aware audience split fits the existing help-gating direction, but option-only version flags need a narrow bypass before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep version flags out of help-role lookup:** `packages/clawhub/src/cli/adminHelp.ts:86`
  - body: `isHelpRequest` now treats any argv without a known command path as help. Since `clawhub -V` and `clawhub --cli-version` have no command path and `resolveHelpRole()` runs before Commander prints the version, logged-in version checks can read config and call `/api/v1/whoami` with the stored token for a local/offline command. Exempt the version flags or classify them as known non-help requests.
  - confidence: 0.88
- **[P3] Add the CLI fix to the changelog:** `CHANGELOG.md:5`
  - body: This is a user-facing CLI fix, but the PR head leaves `CHANGELOG.md` unchanged. Repo policy requires user-facing fixes to be listed under `Unreleased`, so please add a concise CLI fix entry before merge.
  - confidence: 0.9
