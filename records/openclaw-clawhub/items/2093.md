## Summary

Keep open: current main does not implement this PR, and the branch still has merge-blocking review issues around dev admin auth, malicious-appeal cleanup behavior, missing durable spec updates, and absent real behavior proof.

## What This Changes

The PR changes skill/package moderation appeals to malicious-only, adds appeal cleanup/review-verdict metadata and moderation decision export tooling, and adds a local dev persona switcher.

## Best Possible Solution

Land a revised PR that hardens or removes shared-deployment dev auth, keeps malicious appeals open while cleaning only non-malicious Review cases, updates the moderation spec, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, for the PR-introduced findings: source inspection shows the dev credentials provider can mint an admin persona when DEV_AUTH_ENABLED is set, and the cleanup helper maps malicious appeals to rejected. I did not run the PR branch or a live Convex deployment.

## Solution Assessment

No. The malicious-only appeal direction may be acceptable, but this implementation currently broadens dev auth and closes malicious appeals; the safer direction is to harden dev auth to truly local/private use and scope cleanup to non-malicious Review cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate dev persona auth to local-only deployments:** `convex/auth.ts:95-104`
  - body: This provider is reachable from the Convex auth backend whenever `DEV_AUTH_ENABLED=1` and the deployment name is not prod. Since the personas include an admin user, any shared preview or staging Convex deployment with that env var set would allow public admin sign-in; the frontend localhost check does not protect this backend endpoint. Please require a truly local/private deployment guard or remove the admin-capable provider from shared deployments.
  - confidence: 0.88
- **[P2] Leave malicious appeals open during cleanup:** `convex/skills.ts:3358-3361`
  - body: The PR summary says malicious appeals should stay open, but this helper rejects every matching open appeal when the effective status is `malicious`; package cleanup has the same mapping. Because rescans and moderator state changes call this path, an appeal can be auto-rejected exactly when the artifact becomes malicious, instead of remaining available for owner review.
  - confidence: 0.9
- **[P2] Update the moderation spec for appeal policy changes:** `convex/skills.ts:3483-3489`
  - body: This changes the security-sensitive appeal contract from suspicious/scanner-flagged outcomes to malicious-only, but `specs/security-moderation.md` still records the old expected behavior. Please update the durable spec in the same PR so future moderation work preserves the intended policy.
  - confidence: 0.86
- **[P3] Preserve repo-local skill ignore exceptions:** `.gitignore:38`
  - body: Removing the `.agents/skills` exceptions leaves `*/skills/*` in place, so new Convex or blacksmith repo-local skill files under `.agents/skills/**` will be ignored by default. That conflicts with the repo policy that allows these developer skills while keeping top-level `skills/` reserved.
  - confidence: 0.82
