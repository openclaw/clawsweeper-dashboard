## Summary

Keep open: current main does not implement the malicious-only appeal policy, and this PR still has blocking correctness, security, typecheck, spec, and proof issues before it can merge.

## What This Changes

The PR changes skill/package appeals to malicious-only, adds review metadata cleanup/export tooling, adds local dev persona auth/UI, and refactors artifact dashboard/detail surfaces.

## Best Possible Solution

Land a revised PR that enforces malicious-only appeals consistently, keeps malicious appeals open for moderator resolution, gates dev persona auth to a truly local backend, updates the durable moderation spec/docs, and includes redacted runtime proof.

## Reproduction Assessment

Yes. Source inspection of the PR head reproduces the blockers: admin-capable dev auth is gated only by env/deployment string, malicious reconciliation rejects open appeals, and the package appeal path leaves an unused local under noUnusedLocals.

## Solution Assessment

No. The malicious-only direction may be acceptable after maintainer approval, but the current patch is not the narrowest maintainable solution because it leaves security, appeal lifecycle, spec, typecheck, and proof gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate dev persona auth to local deployments:** `convex/auth.ts:18-21`
  - body: This backend provider is enabled solely by `DEV_AUTH_ENABLED=1` plus a non-production-looking `CONVEX_DEPLOYMENT`, and it can mint an admin persona. A shared preview, staging, or dev backend with that env var would expose the credentials endpoint even though the frontend button is localhost-only, so the backend needs a true local-only guard or no admin persona outside local dev.
  - confidence: 0.9
- **[P2] Keep malicious skill appeals open during reconciliation:** `convex/skills.ts:3374-3377`
  - body: The helper sets `appealStatus` to `rejected` whenever the effective status is malicious. Rescans, moderator final actions, and manual overrides call this helper, so an open malicious appeal can be auto-rejected even though malicious artifacts are the only ones the new policy keeps appealable.
  - confidence: 0.9
- **[P2] Keep malicious package appeals open during reconciliation:** `convex/packages.ts:2562-2565`
  - body: Package reconciliation has the same malicious-to-rejected mapping. Bulk cleanup skips malicious appeals, but moderator and rescan paths still call this helper and can close the appeal contrary to the malicious-only appeal policy.
  - confidence: 0.88
- **[P2] Remove the unused package moderation state:** `convex/packages.ts:3072`
  - body: After replacing the appealability check with `scanStatus !== "malicious"`, `moderationState` is no longer read. The root `tsconfig.json` enables `noUnusedLocals`, so `bunx tsc --noEmit` will fail until this is removed or used.
  - confidence: 0.87
- **[P2] Record the appeal policy in the moderation spec:** `convex/skills.ts:3499-3505`
  - body: This changes the security-sensitive appeal contract to malicious-only, but `specs/security-moderation.md` still records the old broader skill appeal behavior. Update the durable spec in the same PR so future moderation work preserves the intended policy.
  - confidence: 0.86
