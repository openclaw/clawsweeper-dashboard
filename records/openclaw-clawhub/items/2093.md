## Summary

Keep open: current main does not implement the PR, and the latest PR head still has blocking security, moderation lifecycle, typecheck, spec, UI regression, and proof gaps.

## What This Changes

The PR changes moderation appeals to malicious-only, adds cleanup/export tooling and review metadata, introduces local dev persona auth/UI, refreshes artifact dashboard/detail UI, and updates repo-local Convex agent guidance.

## Best Possible Solution

Land a revised PR that consistently enforces malicious-only appeals, keeps malicious appeals open for moderator resolution, gates dev persona auth to a truly local backend, preserves existing detail-page capabilities unless intentionally removed, updates durable specs/docs, and includes redacted runtime proof.

## Reproduction Assessment

Yes. Source inspection of the latest PR head reproduces the blockers in `convex/auth.ts`, `convex/skills.ts`, `convex/packages.ts`, `specs/security-moderation.md`, and `src/components/SkillDetailTabs.tsx` without needing a seeded runtime.

## Solution Assessment

No. The malicious-only direction may be acceptable after maintainer approval, but this patch is not merge-ready because it leaves security, appeal lifecycle, spec, typecheck, UI regression, and proof gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate dev persona auth to local deployments:** `convex/auth.ts:18-21`
  - body: This backend provider is enabled by `DEV_AUTH_ENABLED=1` plus a non-production-looking deployment string and can mint an admin persona. A shared preview, staging, or dev backend with that env var would expose admin sign-in even though the frontend button is localhost-only, so the backend needs a true local-only guard or no admin persona outside local dev.
  - confidence: 0.9
- **[P2] Keep malicious skill appeals open during reconciliation:** `convex/skills.ts:3374-3377`
  - body: This maps malicious effective status to `rejected` for every open matching appeal. Rescans, moderator decisions, and manual overrides can therefore close the only appeals the new policy still allows, before a moderator resolves them.
  - confidence: 0.9
- **[P2] Keep malicious package appeals open during reconciliation:** `convex/packages.ts:2562-2565`
  - body: Package reconciliation has the same malicious-to-rejected mapping. Bulk cleanup skips malicious appeals, but moderator and rescan paths still call this helper and can reject an appeal that should remain open under the malicious-only policy.
  - confidence: 0.88
- **[P2] Remove the unused package moderation state:** `convex/packages.ts:3075`
  - body: After replacing the appealability check with `scanStatus !== "malicious"`, `moderationState` is never read. The root `tsconfig.json` enables `noUnusedLocals`, so `bunx tsc --noEmit` will fail until this local is removed or used.
  - confidence: 0.87
- **[P2] Record the appeal policy in the moderation spec:** `convex/skills.ts:3499-3505`
  - body: This changes the security-sensitive appeal contract to malicious-only, but the durable moderation spec still documents the old broader skill and package appeal behavior. Update the spec in the same PR so future moderation work preserves the intended policy.
  - confidence: 0.86
- **[P2] Preserve skill version history and compare access:** `src/components/SkillDetailTabs.tsx:16`
  - body: The PR removes the `compare` and `versions` tabs from skill details and deletes the diff/version panels, leaving users with only SKILL.md and Files. Unless maintainers explicitly decided to remove version history and diffing, this regresses existing skill-detail functionality and should be restored or replaced.
  - confidence: 0.82
