## Summary

Keep open: current main still allows broader skill and package appeals, and the PR is not merge-ready because source inspection shows an admin-capable dev auth provider, malicious appeals being auto-rejected during reconciliation, stale moderation-spec intent, removed repo-local skill ignore exceptions, insufficient real behavior proof, and failing static/unit checks.

## What This Changes

The PR changes skill/package moderation to malicious-only appeals, adds cleanup/export and review metadata tooling, adds localhost dev persona auth, refreshes artifact/detail UI, and updates repo-local Convex guidance.

## Best Possible Solution

Land a revised PR that consistently enforces malicious-only appeals across backend/API/CLI, keeps malicious appeals open for moderator review, gates dev persona auth to a truly local backend, preserves repo-local skill ignore exceptions, updates the durable moderation spec, and includes redacted runtime proof.

## Reproduction Assessment

Yes, for the PR-introduced blockers: PR-head source inspection shows the admin-capable dev auth provider and the malicious appeal auto-rejection paths. Current-main source also shows the requested malicious-only policy is not already implemented.

## Solution Assessment

No. The malicious-only appeal direction may be acceptable after maintainer approval, but this patch still needs auth hardening, appeal reconciliation fixes, spec and ignore-rule cleanup, green checks, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate dev persona auth to local-only deployments:** `convex/auth.ts:18-22`
  - body: This provider runs whenever `DEV_AUTH_ENABLED=1` and the deployment string is not production-like, and the allowed personas include admin. A shared preview, staging, or dev backend with that env var could mint an admin session; hiding the frontend control on non-localhost pages does not protect the credentials endpoint.
  - confidence: 0.9
- **[P2] Keep malicious skill appeals open during reconciliation:** `convex/skills.ts:3374-3377`
  - body: The PR goal says only malicious skills remain appealable, but this helper rejects matching open appeals whenever the effective status is malicious. Rescans and moderator state changes call this path, so a malicious appeal can be auto-closed instead of staying available for moderator review.
  - confidence: 0.9
- **[P2] Keep malicious package appeals open during reconciliation:** `convex/packages.ts:2562-2565`
  - body: Package reconciliation has the same malicious-to-rejected mapping. Bulk cleanup skips malicious appeals, but rescan and moderator-state paths still call this helper and can close package appeals contrary to the malicious-only appeal policy.
  - confidence: 0.88
- **[P2] Record the appeal policy in the moderation spec:** `convex/skills.ts:3500-3504`
  - body: This changes the security-sensitive appeal contract to malicious-only, while the durable moderation spec on the PR head still records the old broader skill appeal behavior. Update the spec in the same PR so future moderation work preserves the intended policy.
  - confidence: 0.86
- **[P3] Preserve repo-local skill ignore exceptions:** `.gitignore:38-39`
  - body: Removing these exceptions leaves `*/skills/*` in place, so new repo-local developer skills under `.agents/skills` are ignored by default. That conflicts with the repository policy allowing Convex and blacksmith developer skills there while keeping top-level `skills/` reserved.
  - confidence: 0.82
