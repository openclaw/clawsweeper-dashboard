## Summary

Keep open: current main has not implemented this PR, and the PR still has blocking security/correctness/spec issues plus missing real behavior proof. Source inspection shows the dev credentials provider can mint admin personas when DEV_AUTH_ENABLED is set on any non-production deployment, and reconciliation helpers can reject malicious appeals despite the stated policy to keep them open.

## What This Changes

The PR branch changes skill/package moderation appeals to malicious-only, adds appeal cleanup/review metadata and moderation decision export tooling, and adds a local dev persona switcher.

## Best Possible Solution

Land a revised PR that hardens or removes shared-deployment dev auth, keeps malicious appeals open during reconciliation, updates the durable moderation spec, preserves repo-local skill ignore exceptions, and includes redacted real behavior proof.

## Reproduction Assessment

Yes, for the PR-introduced findings: source inspection of the PR head shows the admin-capable dev credentials provider and the malicious-to-rejected cleanup mappings. I did not need a live Convex deployment to identify those code paths.

## Solution Assessment

No. The malicious-only appeal direction may be acceptable after maintainer approval, but this implementation needs security hardening, cleanup semantics fixes, a durable spec update, and proof before it is the maintainable path.

Full review comments:

  - body: This provider is reachable from the Convex auth backend whenever DEV_AUTH_ENABLED=1 and the deployment string is not production.   - confidence: 0.9

  - body: Package reconciliation has the same malicious-to-rejected mapping. Bulk cleanup skips malicious appeals, but rescan and moderator-state paths still call this helper, so package appeals can be closed contrary to the PR's malicious-only appeal policy.
  - confidence: 0.87
- **[P2] Update the moderation spec for the appeal policy:** `convex/skills.ts:3483-3489`
  - body: This changes the security-sensitive appeal contract from suspicious/scanner-flagged outcomes to malicious-only, but the durable moderation spec still records the old behavior. Please update the spec in the same PR so future moderation work preserves the intended policy.
  - confidence: 0.86
