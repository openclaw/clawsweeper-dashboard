## Summary

Keep open: this is a draft MEMBER-authored PR, current main does not yet implement the ClawPack API/schema surface, and the proposed branch has a blocking publish consistency issue.

## What This Changes

Adds a ClawPack artifact ecosystem for plugin packages, including Convex schema/API/backfill/moderation, CLI publish/download/verify/admin commands, management UI, plugin detail pages, docs, and tests.

## Best Possible Solution

Keep the PR open until the artifact build/attach path is made transactionally safe from a product perspective and the draft branch is ready for maintainer review.

## Reproduction Assessment

Yes for the review finding: the proposed `publishPackageImpl` commits `insertReleaseInternal` before storage reads, hash checks, ClawPack build/storage, and attachment. A failure in any of those later steps reproduces the inconsistent committed-release path by source inspection.

## Solution Assessment

No: the proposed sequencing is not the best way to solve publishing because it makes the release visible before the required ClawPack artifact succeeds. A safer path is to build/attach before exposing the release, or add an explicit compensating cleanup/failed-artifact state around the post-insert work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Make ClawPack publish failures roll back the release:** `convex/packages.ts:3538-3552`
  - body: The new flow commits `insertReleaseInternal` before building, storing, and attaching the ClawPack. If `storage.get`, hash validation, `buildClawPack`, `storage.store`, or the attach mutation fails, the action throws after the package/release rows and tags/latest state have already committed, and the later scan scheduling is skipped. Build/attach before making the release visible or add a compensating cleanup/failed state around this block.
