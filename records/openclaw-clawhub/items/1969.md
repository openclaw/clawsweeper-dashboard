## Summary

Keep this draft member-authored PR open: current main does not implement the ClawPack ecosystem, and the proposed branch still has a blocking publish consistency issue in the artifact path.

## What This Changes

The PR adds a ClawPack plugin artifact ecosystem across Convex package storage/API/backfills/moderation, CLI package commands, management and publishing UI, docs, schema types, and tests.

## Best Possible Solution

Keep the PR open until ClawPack artifact creation and attachment are made transactionally safe and the draft product surface is ready for maintainer review.

## Reproduction Assessment

Yes. By source inspection, a publish that succeeds through `insertReleaseInternal` and then fails during storage reads, hash validation, ClawPack build/storage, or artifact attach reproduces a committed release without its expected ClawPack artifact and before later scan scheduling.

## Solution Assessment

No. The proposed publish sequencing is not the best solution because it exposes the release before the required artifact work succeeds; a safer design would build/attach before making the release visible or add an explicit failed-artifact state with compensating cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Make ClawPack publish failures roll back the release:** `convex/packages.ts:3538-3552`
  - body: The new flow commits `insertReleaseInternal` before building, storing, and attaching the ClawPack. If `storage.get`, hash validation, `buildClawPack`, `storage.store`, or the attach mutation fails, the action throws after package/release rows and latest/tag state are already committed, and scan scheduling is skipped. Build/attach before exposing the release or add compensating cleanup/failed-artifact state around this block.
  - confidence: 0.88
