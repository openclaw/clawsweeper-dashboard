## Summary

Keep this PR open: the storage-derived metadata direction is useful, but the current patch still reads supplied storage blobs before enforcing trusted size limits, and the member-authored PR needs maintainer handling rather than cleanup closure.

## What This Changes

The PR changes skill publishing to derive stored file size, content type, and sha256 during publish and adds regression tests for forged caller metadata.

## Best Possible Solution

Derive sha256, size, and content type from trusted storage data while rejecting over-limit stored bytes before downloading blobs for text processing or hashing.

## Reproduction Assessment

Yes from source inspection. Current main trusts caller-supplied publish metadata for size/fingerprint, and the PR tests demonstrate the forged-metadata case; I did not execute tests in this read-only review.

## Solution Assessment

No as written. Deriving metadata from storage is the right direction, but the implementation should enforce trusted stored size limits before downloading blobs.
