## Summary

Current main still trusts caller-supplied publish metadata, while this focused PR moves validation and hashing to stored bytes; the patch appears correct, but the member-authored PR must remain open for ordinary maintainer review.

## What This Changes

The PR derives publish file size, content type, and SHA-256 from stored blobs, enforces limits against stored bytes, and adds regression tests for mismatched caller metadata.

## Best Possible Solution

Land the shared storage-authoritative validation after maintainers are satisfied that supported JSON, multipart, staged, import, and restore publish paths retain their existing payload compatibility.

## Reproduction Assessment

Yes, from source inspection: current main enforces type and byte limits using caller-provided metadata before reading storage, so those values can disagree with the stored bytes. This review did not execute a dedicated mismatched-metadata publish against a live Convex deployment.

## Solution Assessment

Yes. Deriving metadata in `publishVersionForUser` is the narrowest maintainable solution because the UI, CLI, HTTP, import, and restore publishing paths converge on that shared boundary.
