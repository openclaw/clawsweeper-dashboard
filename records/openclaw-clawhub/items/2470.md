## Summary

Keep this PR open: deriving size and hashes from stored bytes is the correct direction, but empty storage MIME values still fall back to publisher-controlled metadata, leaving part of the trust-boundary bug unresolved. The PR also requires real Convex storage proof before merge, and its MEMBER-authored status requires explicit maintainer handling.

## What This Changes

The PR derives published skill file size, content type, and SHA-256 from stored blobs before enforcing limits and persisting metadata, with two focused regression tests.

## Best Possible Solution

Use storage-owned metadata when available and otherwise infer text eligibility from the sanitized path or inspect the stored bytes, never from publisher-supplied MIME metadata; then prove mismatched size, hash, MIME, and empty-MIME cases through a real Convex publish.

## Reproduction Assessment

Yes at source level: current main validates size, hash, and MIME metadata supplied by the caller rather than independently deriving all authoritative values from storage. The PR does not include a real Convex reproduction or after-fix demonstration.

## Solution Assessment

No, not yet: storage-authoritative size and hashing are correct, but falling back to caller MIME leaves the fix incomplete; path-derived or byte-verified classification is safer.
