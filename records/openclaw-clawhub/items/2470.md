## Summary

This PR should remain open because deriving publish metadata from storage fixes a real trust-boundary bug, but the branch fetches every referenced blob before enforcing the authoritative 10 MB file and 50 MB bundle limits. That ordering creates an avoidable availability exposure, and the PR provides only mocked tests rather than the repository-required real Convex storage validation.

## What This Changes

The PR changes skill publishing to derive file size, content type, and SHA-256 from stored blobs and adds regression tests for caller metadata that disagrees with storage.

## Best Possible Solution

Read authoritative `_storage` metadata first, reject missing, non-text, per-file, and aggregate violations before blob downloads, then fetch and hash only accepted files while preserving the existing publish input contract.

## Reproduction Assessment

Yes, at source level: submit many individually acceptable storage IDs whose authoritative total exceeds 50 MB, and the proposed loop calls `storage.get()` for every file before aggregate rejection. A real Convex run is still needed to measure the runtime effect.

## Solution Assessment

No. Trusting storage rather than caller metadata is the correct fix, but authoritative metadata should gate downloads so the validation itself cannot consume resources beyond the publish limit.
