## Summary

This PR remains necessary because current main still validates caller-supplied publish metadata rather than the referenced stored bytes. The implementation is focused and no correctness defect was identified, but the storage-dependent behavior should receive the repository-required real Convex runtime validation before merge.

## What This Changes

The PR derives published skill file size, content type, and SHA-256 from Convex storage blobs, enforces limits against those stored bytes, and adds regression tests for mismatched caller metadata.

## Best Possible Solution

Keep the shared publish boundary authoritative by deriving metadata from storage, then merge after a real Convex publish smoke verifies MIME normalization, byte hashing, missing-blob handling, and both file and aggregate size limits on the final merge result.

Yes at the source level: current main validates caller-provided size, content type, and hash while later reading the referenced storage Blob, allowing those inputs to disagree. No live current-main Convex reproduction was supplied or run during this read-only review.

## Solution Assessment

Yes in direction: deriving authoritative metadata once at the shared publication boundary is the narrowest maintainable fix and protects all callers. Real Convex validation is still needed to confirm the runtime storage semantics.
