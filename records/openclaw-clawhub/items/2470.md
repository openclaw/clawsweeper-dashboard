## Summary

Keep this member-authored PR open for revision: it addresses a real caller-metadata trust flaw, but the proposed MIME derivation still falls back to untrusted publish metadata when the stored Blob has no type, and the PR lacks real Convex behavior proof.

## What This Changes

This PR derives skill publish file size, content type, and SHA-256 from stored blobs, enforces limits against those bytes, and adds regression tests for forged metadata.

## Best Possible Solution

Make stored metadata or conservative server-side inference the sole authority for file type, document the publish trust invariant in `specs/`, and validate forged metadata plus near-limit bundles through a real Convex publish flow.

## Reproduction Assessment

Yes. Current-main source clearly shows caller-supplied size, content type, and digest values entering publish validation, and the supported multipart path can store a Blob with an empty MIME type.

## Solution Assessment

No, not completely. Deriving size and digest from stored bytes is the correct root-cause direction, but falling back to caller MIME metadata preserves part of the trust boundary the PR intends to remove.
