## Summary

Keep this member-authored PR open, but require changes before merge: deriving metadata from storage fixes a real trust problem, while the current helper downloads every submitted blob before enforcing the authoritative aggregate limit and lacks real Convex runtime proof.

## What This Changes

The PR replaces caller-supplied publish size, content type, and SHA-256 with values derived from stored blob bodies and adds regression tests for mismatched metadata.

## Best Possible Solution

Read authoritative `_storage` metadata through a narrow internal Convex function, reject missing or oversized submissions before blob downloads, then fetch only accepted blobs for content validation.

## Reproduction Assessment

Yes, from source inspection: current main trusts caller metadata, and the proposed branch clearly fetches every stored blob before aggregate-size rejection. No runtime execution was performed during this read-only review.

## Solution Assessment

No, not in its current form. Storage-authoritative metadata is the correct direction, but authoritative size checks must precede blob-body reads.
