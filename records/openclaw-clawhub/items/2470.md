## Summary

This PR addresses a real publish-integrity gap that remains on current main, but it needs correction before merge because an empty stored MIME type still falls back to caller metadata; durable security-boundary documentation and real Convex behavior proof are also missing. Likely related people: @Patrick-Erichsen (high confidence, recent publish-path contributor), @vincentkoc (high confidence, publish-limit feature owner), and @steipete (medium confidence, module refactor author).

## What This Changes

The branch derives published file size, content type, and SHA-256 from stored blobs before validation and adds mocked regressions for mismatched caller metadata and oversized stored files.

## Best Possible Solution

Make stored bytes and server-derived path rules authoritative without a publish-request MIME fallback, document the invariant in a focused upload or publishing spec, and verify mismatched metadata plus near-limit bundles on a real Convex deployment.

## Reproduction Assessment

Yes from source inspection: current main trusts caller size, content type, and SHA-256 metadata, and the branch’s mocked tests demonstrate mismatches with stored bytes. The remaining empty-MIME fallback is also directly source-reproducible, although no real Convex reproduction was supplied.

## Solution Assessment

No, not in its current form: deriving size and digest from storage is the right narrow solution, but falling back to publish-request MIME metadata leaves the intended trust boundary incomplete. Remove that fallback, preserve the invariant in specs, and validate the complete path on real Convex.
