## Summary

The PR should stay open: its trusted-metadata direction fixes a real integrity gap, but the current implementation downloads every submitted blob before enforcing the aggregate publish limit, and it lacks real Convex runtime proof.

## What This Changes

The PR derives published skill file size, content type, and SHA-256 from stored blobs instead of caller metadata and adds focused mismatch and oversize tests.

## Best Possible Solution

Preflight trusted Convex `_storage` records for existence, per-file size, aggregate size, SHA-256, and content type before downloading blob bodies, then fetch only an accepted bundle for text-content checks and publishing.

## Reproduction Assessment

Yes from source inspection: submit valid storage IDs whose actual combined size exceeds 50MB while caller sizes are understated; the proposed helper fetches every blob before rejecting the trusted total.

## Solution Assessment

No, not yet; deriving metadata from trusted storage is the correct boundary, but trusted storage metadata must enforce the size budget before blob retrieval.
