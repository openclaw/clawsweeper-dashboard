Keep open under the supplied ClawHub cleanup policy because this item is an issue, not a PR. The reported HTTP API regression appears fixed on current main and live production: the route now calls the V4 listing query and returns non-empty items, while the old empty query remains only as a deprecated stub.

## What This Changes

The issue reports that `GET /api/v1/skills` returns an empty list because the HTTP handler still calls the deprecated `skills.listPublicPage` stub instead of the V4 listing query.

## Best Possible Solution

Keep the V4-backed HTTP route as the supported implementation for `GET /api/v1/skills`; a maintainer can manually close this issue as implemented, with optional follow-up cleanup for the deprecated stub.

## Reproduction Assessment

Yes. The issue provides a focused curl path, and the same live production request now returns non-empty items while current main routes the endpoint through the V4 query.

## Solution Assessment

Yes. Routing the HTTP endpoint to `listPublicPageV4` is the narrow maintainable fix for the reported regression; removing or hardening the deprecated stub would be a separate preventive cleanup.
