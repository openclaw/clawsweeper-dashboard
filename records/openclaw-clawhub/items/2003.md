## Summary

Keep open. Current main still has a credible static-scanner false-positive class: manifest files are checked for raw-IP URLs, so docker-compose metadata using loopback service URLs can produce `suspicious.install_untrusted_source`; changing that rule needs maintainer/security policy judgment.

## What This Changes

The issue asks ClawHub to refine static analysis so docker-compose-based skills are not misleadingly classified as `suspicious.install_untrusted_source` for expected compose/container-install patterns.

## Best Possible Solution

Adopt a narrow compose-manifest scanner policy that treats expected loopback service metadata differently while preserving warnings for URL shorteners, public raw-IP install sources, mutable remote compose fetches, and weak container provenance.

## Reproduction Assessment

Yes. Source inspection shows current main applies the raw-IP install-source heuristic to `.yml` manifests, and public scan data confirms InvestorClaw 4.1.22 was flagged while its compose file contained matching `127.0.0.1` URLs.

## Solution Assessment

No for a broad verified-registry allowlist as the first fix; registry verification alone does not prove immutability, signing, or publisher integrity. The safer path is a narrow manifest-context refinement with regression coverage after maintainer/security sign-off.
