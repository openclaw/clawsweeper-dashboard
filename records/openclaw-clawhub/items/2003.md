## Summary

Keep open: ClawHub policy says issues are not cleanup-close candidates, and current main still has a manifest-level raw-IP heuristic that plausibly explains this docker-compose false-positive class. The remaining fix is security-policy-sensitive because it must distinguish expected loopback compose metadata from unsafe install sources and weak container provenance.

## What This Changes

The issue asks ClawHub to reduce false positives where docker-compose-based skills are flagged as `suspicious.install_untrusted_source` despite local compose files and public GHCR images.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that permits expected loopback service metadata while preserving useful warnings for unsafe install sources and weak container provenance.

## Reproduction Assessment

Yes, by source inspection plus live API evidence. Current main scans all YAML/TOML/JSON manifests for raw IPv4 URLs, and the affected 4.1.22 compose manifest contains loopback URLs while the live scan still reports `suspicious.install_untrusted_source`.

## Solution Assessment

Unclear until maintainers set scanner policy. A broad verified-registry allowlist or metadata escape hatch is riskier than a targeted compose-manifest exception paired with explicit container provenance rules.
