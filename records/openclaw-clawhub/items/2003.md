## Summary

Keep open. Current main still applies the `install_untrusted_source` manifest rule to raw-IP URLs inside YAML manifests, and the affected compose file contains loopback `127.0.0.1` service metadata that matches that rule. A fix is plausible, but changing install-source scanner policy is security-sensitive and needs maintainer approval rather than conservative cleanup.

## What This Changes

The issue asks ClawHub to refine its static moderation scanner so docker-compose based skills are not blanket-flagged as `suspicious.install_untrusted_source` for local compose files and public container images.

## Best Possible Solution

Codify a narrow manifest-scanner policy that exempts expected loopback docker-compose service metadata while preserving warnings for URL shorteners, public raw-IP install sources, mutable remote compose fetches, and weak container provenance.

## Reproduction Assessment

Yes. Current main applies `RAW_IP_URL_PATTERN` to `.yml` manifests, and the affected 4.1.22 compose file contains loopback URLs that match that regex; I did not run the full Convex test suite in this read-only review.

## Solution Assessment

No for the broad verified-registry allowlist suggested in the issue; that is too coarse for a supply-chain scanner. The safer path is a narrow manifest-context refinement with regression tests after maintainer/security sign-off.
