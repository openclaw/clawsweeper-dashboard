## Summary

Keep open: source inspection supports a real static-moderation false-positive path, but the fix changes an abuse-detection heuristic and needs maintainer/security approval before implementation.

## What This Changes

The issue asks to refine the static analyzer so docker-compose-based skills are not incorrectly flagged as `suspicious.install_untrusted_source`.

## Best Possible Solution

Add a narrow loopback-aware manifest heuristic with regression tests, while preserving warnings for public raw-IP and URL-shortener install sources; defer registry allowlists or new metadata knobs unless maintainers choose that policy.

## Reproduction Assessment

Yes, by source inspection: a manifest file containing a loopback raw-IP URL matches `RAW_IP_URL_PATTERN` and flows through `scanManifestFile` to `suspicious.install_untrusted_source`. I did not run a generated repro test because this review is read-only.

## Solution Assessment

Unclear pending maintainer/security approval. The narrowest maintainable fix appears to be distinguishing loopback/local endpoints from public raw-IP install sources, rather than adding broad verified-registry allowlists or a new installation-type field immediately.
