## Summary

Keep open. Current main still applies `suspicious.install_untrusted_source` to any YAML/JSON/TOML manifest containing dotted-quad HTTP(S) URLs, and the public InvestorClaw 4.1.22 data shows that finding on a compose file with loopback metadata; this is a real scanner-calibration issue, but the fix is security policy work rather than conservative cleanup.

## What This Changes

Refine ClawHub's static scanner so docker-compose-based skills are not misclassified as `install_untrusted_source` when the compose file is local and the risky source signal is expected container or loopback metadata.

## Best Possible Solution

Define a narrow, context-aware static scanner policy for docker-compose manifests that permits expected loopback service metadata while preserving warnings for URL shorteners, public raw-IP install sources, remote compose fetches, mutable tags, and weak container provenance.

## Reproduction Assessment

Yes. Source inspection on current main shows the manifest scanner still emits `install_untrusted_source` for raw-IP URLs in YAML, and the public 4.1.22 scan/file endpoints show that finding alongside loopback compose metadata.

## Solution Assessment

Unclear until maintainers set the scanner policy. A broad verified-registry allowlist is not the best first fix; a narrower compose-manifest exception for expected loopback metadata plus stronger container provenance rules is safer.
