## Summary

Keep open: current main still contains the manifest raw-IP heuristic that can emit `suspicious.install_untrusted_source` for docker-compose YAML with loopback service URLs, and the live affected version still reports that static rule. The problem is localized and source-reproducible, but changing compose/container trust classification is security-policy-sensitive and needs maintainer direction.

## What This Changes

Request to refine ClawHub static moderation so docker-compose-based skills with bundled compose files and container images are not misclassified as `suspicious.install_untrusted_source`.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that permits expected loopback service metadata while retaining warnings for unsafe install sources and weak container provenance, then add regression tests and a security spec note.

## Reproduction Assessment

Yes, by source inspection plus live API evidence. Current main scans every manifest file for raw IPv4 URLs, and InvestorClaw 4.1.22 still reports `suspicious.install_untrusted_source` while its bundled compose/install YAML contains loopback URLs.

## Solution Assessment

Unclear until maintainers set scanner policy. A targeted compose-manifest exception with explicit container provenance rules is safer than a broad verified-registry allowlist or metadata escape hatch.
