## Summary

Keep open: current main still contains the raw-IP manifest heuristic that explains the reported false positive, and the live moderation endpoint still reports `suspicious.install_untrusted_source` for the affected skill. Because this is security moderation tuning, the next step should be maintainer/security review rather than cleanup closure.

## What This Changes

Request refinement of the static moderation analyzer so docker-compose-based skills are not over-flagged as `suspicious.install_untrusted_source` for expected local compose/service patterns.

## Best Possible Solution

Teach the moderation engine to treat localhost/loopback manifest URLs as local service endpoints while preserving warnings for public raw-IP and shortener install sources, with regression coverage for both cases.

## Reproduction Assessment

Yes. The public moderation endpoint still shows `suspicious.install_untrusted_source`, and current main's manifest scanner deterministically applies the raw-IP URL heuristic to the published loopback service URLs.

## Solution Assessment

Yes, but narrower than the original registry-trust hypothesis. A loopback/localhost exception in manifest scanning is more maintainable than adding verified-registry allowlists or a new installation-type field first.
