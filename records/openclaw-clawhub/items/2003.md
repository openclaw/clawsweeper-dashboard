## Summary

Keep open: current main still supports a real static-scan false positive, and this is an issue rather than an implemented PR, so no ClawHub cleanup close applies. The strongest evidence points to loopback raw-IP URLs in bundled manifests, not the GHCR image line alone, and scanner tuning should get maintainer/security review.

## What This Changes

The issue asks for the static moderation analyzer to stop classifying docker-compose-based skill manifests as `suspicious.install_untrusted_source` when they use intended local compose files and trusted container images.

## Best Possible Solution

Narrow the manifest raw-IP heuristic so localhost and loopback service URLs in bundled compose/config files do not trigger `install_untrusted_source`, while preserving warnings and tests for public raw-IP and URL-shortener install sources.

## Reproduction Assessment

Yes. The public moderation endpoint still reports the finding for the affected skill, and current main's source deterministically applies the manifest raw-IP heuristic to loopback URLs in scanned text files.

## Solution Assessment

Yes, but the best fix is narrower than the initial registry allowlist proposal. A loopback/localhost exception with regression coverage is more maintainable than adding trusted-registry policy or a new installation-type metadata field first.
