## Summary

Keep open: the report is still valid, but changing Docker Compose and container provenance heuristics is a security-policy decision, not conservative cleanup. Current main still has the manifest raw-IP rule that can produce `suspicious.install_untrusted_source`, and the live affected version still reports that static reason.

## What This Changes

The issue asks ClawHub to reduce false-positive `suspicious.install_untrusted_source` results for docker-compose-based skills that use local compose files and public container images.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that permits expected loopback service metadata while preserving warnings for URL shorteners, remote compose fetches, public raw-IP install sources, mutable tags, and weak container provenance.

## Reproduction Assessment

Yes. The live scan API reproduces the reported reason for `investorclaw` 4.1.22, and current main source shows the manifest raw-IP rule that matches the compose file's loopback URLs.

## Solution Assessment

Unclear. A broad verified-registry allowlist is not the safest fix; a narrower compose-manifest exception plus explicit container provenance rules would be more maintainable.
