## Summary

Keep open: this is a plausible static-moderation false-positive class, but current main points to manifest raw-IP/loopback URL matching rather than GHCR image pulls. Because the fix would change a trust-and-safety heuristic, it needs maintainer/security approval before implementation.

## What This Changes

The issue asks ClawHub to refine static moderation so docker-compose-based skill manifests are not incorrectly flagged as `suspicious.install_untrusted_source`.

## Best Possible Solution

Refine the manifest scanner to exempt localhost/loopback install endpoints while preserving warnings for public raw-IP URLs and URL shorteners, with regression tests for compose/install YAML.

## Reproduction Assessment

Yes, by source inspection. A YAML/JSON/TOML manifest containing `http://127.0.0.1:<port>/...` still matches `RAW_IP_URL_PATTERN` in `scanManifestFile` and emits `suspicious.install_untrusted_source`; the GHCR image string itself is not matched by that current path.

## Solution Assessment

No, not as broadly proposed. A verified-registry allowlist or new `installation_type` metadata field is broader than the concrete source-reproducible path; a loopback-aware manifest matcher is the narrower maintainable fix unless maintainers choose a larger policy change.
