## Summary

Keep open. Current main still has the static manifest rule that flags raw-IP URLs in YAML/TOML/JSON as `suspicious.install_untrusted_source`, and public scan data shows the affected InvestorClaw version was flagged while later publisher-side loopback URL changes cleared the static result; the remaining fix is a scanner-policy/security decision, not conservative cleanup.

## What This Changes

The issue requests a refinement to the static analyzer so docker-compose-based skills are not over-flagged as `suspicious.install_untrusted_source` for expected local compose/MCP metadata or trusted container image usage.

## Best Possible Solution

Codify a narrow manifest-scanner policy that exempts expected loopback compose/MCP service metadata while preserving warnings for URL shorteners, public raw-IP install sources, mutable remote compose fetches, and weak container provenance, with regression coverage and a spec note for the scanner boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows `.yml` manifests are scanned with the raw-IP URL pattern, and public scan/file endpoints show InvestorClaw 4.1.22 was flagged while containing `127.0.0.1` compose metadata that later changed to `localhost` in a clean version.

## Solution Assessment

No for a broad verified-registry allowlist; that would be too coarse for a supply-chain scanner. The safer solution is a narrow context-aware manifest refinement after maintainer/security sign-off.
