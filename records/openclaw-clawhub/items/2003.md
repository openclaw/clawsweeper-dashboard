## Summary

Keep open: current main still has a source-reproducible false-positive path for `install_untrusted_source` in manifest files, but the reported GHCR image-pull hypothesis is not supported by the static scanner code and security heuristic tuning needs maintainer review.

## What This Changes

The issue asks for the static analyzer to stop flagging benign docker-compose-based skills that use bundled compose files and public verified-registry images as `suspicious.install_untrusted_source`.

## Best Possible Solution

Add a loopback-aware manifest install-source heuristic with regression tests for local compose endpoints and remote raw-IP/shortener cases, after maintainer/security approval.

## Reproduction Assessment

Yes, by source inspection for the narrowed false-positive class: a YAML/JSON/TOML manifest containing a loopback raw-IP URL matches `RAW_IP_URL_PATTERN` and flows to `suspicious.install_untrusted_source`. I did not find a current-main matcher for GHCR image pulls themselves.

## Solution Assessment

Yes, but narrower than the issue hypothesis: a loopback-aware manifest matcher is the maintainable fix boundary. Verified-registry allowlists or a new `installation_type` metadata field would need separate product/security policy approval.
