## Summary

Keep open: current main still has a manifest-scanner path that can emit `suspicious.install_untrusted_source` for raw-IP URLs in YAML, and the affected 4.1.22 compose file still shows that reason code in live output. The remaining fix is a scanner trust-policy decision for Docker Compose/container provenance, not conservative cleanup.

## What This Changes

Refine the ClawHub static analyzer so docker-compose-based skills with local compose files and trusted container images are not misleadingly classified as `suspicious.install_untrusted_source`, or classify that signal more precisely.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that permits expected loopback service metadata while preserving warnings for URL shorteners, public raw-IP install sources, remote compose fetches, mutable tags, and weak container provenance.

## Reproduction Assessment

Yes. Source inspection plus live affected-version output show that current main still has a manifest scanner path for raw-IP URLs and the affected 4.1.22 compose file still carries the reported static reason code.

## Solution Assessment

Unclear until maintainers set the scanner trust policy. A broad verified-registry allowlist is not the safest default; a narrower compose-manifest exception plus explicit container provenance rules would be more maintainable.
