## Summary

Keep open: current main still has a source-visible false-positive path for `suspicious.install_untrusted_source`, but changing a moderation scanner heuristic needs maintainer/security approval before implementation.

## What This Changes

Refine the static analyzer so docker-compose-style skills are not incorrectly flagged as `suspicious.install_untrusted_source` for intended compose/install patterns.

## Best Possible Solution

Refine the static scan with a maintainer-approved, regression-tested exception for known-local compose/install endpoints while preserving public raw-IP and shortener warnings.

## Reproduction Assessment

Yes, for a concrete current-main path by source inspection: a manifest containing `http://127.0.0.1:<port>/...` matches `RAW_IP_URL_PATTERN` and flows through `scanManifestFile` to `suspicious.install_untrusted_source`. I did not find a current matcher that specifically flags a GHCR image directive alone.

## Solution Assessment

No, not as the broad verified-registry allowlist or new metadata-field request. A narrower loopback-aware manifest matcher with regression tests appears safer unless maintainers explicitly choose a larger trust-policy change.
