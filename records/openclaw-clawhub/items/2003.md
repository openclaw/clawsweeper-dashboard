## Summary

Keep open: current main still contains a manifest-level raw IPv4 heuristic that can emit `suspicious.install_untrusted_source` for docker-compose YAML containing loopback service URLs, and the live affected version still reports that static rule. The issue is localized and source-reproducible, but changing compose/container trust classification is security-policy-sensitive and should not be swept closed.

## What This Changes

The issue asks ClawHub to reduce false positives from `suspicious.install_untrusted_source` on docker-compose-based skills that pull public registry images.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that permits expected loopback service metadata while preserving warnings for unsafe install sources and weak container provenance, then add regression tests and a security spec note.

## Reproduction Assessment

Yes, by source inspection plus live API evidence. Current main scans manifest files for raw IPv4 URLs, and the affected version still reports `suspicious.install_untrusted_source` while its compose manifest contains loopback URLs.

## Solution Assessment

Unclear until maintainers set scanner policy. A targeted compose-manifest exception with explicit container provenance rules is safer than a broad verified-registry allowlist or metadata escape hatch.
