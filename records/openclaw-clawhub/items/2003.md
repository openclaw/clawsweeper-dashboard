## Summary

Keep open. Current main reduced the user-facing impact by making static suspicious findings advisory, but the manifest scanner can still emit `suspicious.install_untrusted_source` for raw-IP URLs in compose YAML, and changing that scanner trust boundary needs maintainer security-policy judgment.

## What This Changes

The issue requests reducing false positives from `suspicious.install_untrusted_source` on docker-compose-based skills that use bundled compose manifests and public registry images.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that permits expected loopback service metadata while preserving warnings for URL shorteners, public raw-IP install sources, remote compose fetches, mutable tags, and weak container provenance.

## Reproduction Assessment

Yes, from source inspection and live affected-version output. Current main still has a manifest scanner path that emits `install_untrusted_source` for raw-IP URLs in YAML, and the published 4.1.22 compose file contains loopback URLs with that static finding.

## Solution Assessment

Unclear. A broad verified-registry allowlist is not the best default; a narrower compose-manifest exception for expected loopback metadata plus explicit container provenance rules would be more maintainable.
