## Summary

Keep open: this still looks like a valid, security-sensitive moderation-scanner false-positive class, but the concrete source path on current main points to manifest loopback raw-IP URLs rather than GHCR image pulls.

## What This Changes

The issue asks to reduce `suspicious.install_untrusted_source` false positives for docker-compose skills that pull public container images from trusted registries.

## Best Possible Solution

Have a maintainer/security reviewer approve a narrow loopback-aware manifest matcher that preserves public raw-IP and shortener warnings, then add regression tests for both sides.

## Reproduction Assessment

Yes, by source inspection. A YAML/JSON/TOML manifest containing `http://127.0.0.1:<port>/...` flows through `runStaticPublishScan` into `scanManifestFile`, matches `RAW_IP_URL_PATTERN`, and emits `suspicious.install_untrusted_source`.

## Solution Assessment

Unclear for the original verified-registry allowlist proposal; the maintainable fix appears narrower. A loopback/private-address exception in the manifest matcher would address the observed false-positive path without adding new metadata or broad registry trust policy.
