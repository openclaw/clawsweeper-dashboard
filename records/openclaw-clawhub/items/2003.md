## Summary

Keep open. The issue is actionable, but current evidence points to the manifest raw-IP heuristic over-firing on loopback service URLs in compose/install YAML rather than the GHCR image reference, and security-scanner policy should get maintainer review before changing.

## What This Changes

Refine the static moderation analyzer so docker-compose skills with bundled compose files and public container images are not incorrectly flagged as `suspicious.install_untrusted_source`.

## Best Possible Solution

Exempt loopback/local service URLs from manifest install-source findings while preserving warnings for external raw-IP and shortener install sources, then add an InvestorClaw-style compose/install regression fixture.

## Reproduction Assessment

Yes. Source inspection shows the current manifest scanner flags any raw IPv4 URL, and live scan data for InvestorClaw 4.1.22 shows the exact static reason code while the bundle contains `http://127.0.0.1` compose/install URLs; I did not execute local tests because this review is read-only.

## Solution Assessment

No, the broad GHCR verified-publisher allowlist or new `installation_type` field suggested in the report is not clearly the narrowest fix. The safer path is a tightly scoped loopback/local URL exception in manifest scanning, with separate maintainer policy if container provenance should become an informational signal.
