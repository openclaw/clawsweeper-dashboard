## Summary

Keep open: current main still has a source-reproducible false-positive path for `suspicious.install_untrusted_source`, but the concrete scanner path appears to be manifest raw-IP matching rather than GHCR container image pulls. Because this changes an abuse-detection heuristic, it needs maintainer/security approval before implementation.

## What This Changes

The issue asks ClawHub to reduce `suspicious.install_untrusted_source` false positives for docker-compose-based skills that use bundled compose files and public registry images.

## Best Possible Solution

Refine the manifest install-source heuristic to exempt loopback/local development endpoints while preserving warnings for public raw-IP URLs and URL shorteners; handle verified-registry allowlisting or new install metadata as a separate product/security policy if maintainers want it.

## Reproduction Assessment

Yes, by source inspection. A YAML/YML/TOML/JSON manifest containing `http://127.0.0.1:<port>/...` still reaches `scanManifestFile`, matches `RAW_IP_URL_PATTERN`, and emits `suspicious.install_untrusted_source`; the GHCR image string itself did not match the current pattern in a focused regex check.

## Solution Assessment

Yes, but narrower than the issue's suggested verified-registry allowlist or new metadata field. A loopback-aware manifest matcher with regression tests is the smallest maintainable fix unless maintainers explicitly choose broader registry trust policy.
