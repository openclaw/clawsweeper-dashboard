## Summary

Keep open: this is a current scanner-policy issue, and the ClawHub cleanup policy for this review only auto-closes pull requests, not issues. Current main still has the broad manifest URL heuristic that can emit `suspicious.install_untrusted_source`, and tightening Docker Compose/container trust handling needs maintainer security-policy judgment.

## What This Changes

The issue asks ClawHub to refine static moderation heuristics so docker-compose-based skills are not broadly flagged as `suspicious.install_untrusted_source` for expected compose and container-image references.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that allows expected loopback service metadata while preserving warnings for remote compose fetches, URL shorteners, public raw-IP install sources, mutable container tags, and weak container provenance.

## Reproduction Assessment

Yes. Current source shows the manifest rule that emits `suspicious.install_untrusted_source` for raw-IP URL matches, and the live scan plus published compose file for `investorclaw` 4.1.22 confirm the affected loopback compose shape.

## Solution Assessment

Unclear. A broad verified-registry allowlist is not the safest fix; a narrower compose-manifest exception plus explicit container provenance rules would preserve useful scanner signals better.
