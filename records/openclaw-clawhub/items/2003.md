## Summary

Keep open. Current main still contains the manifest-level raw-IP/shortener rule that can flag docker-compose YAML with loopback service metadata, the affected public version reproduces `suspicious.install_untrusted_source`, and changing scanner trust heuristics is security-policy work rather than conservative cleanup.

## What This Changes

Request refinement of the static moderation analyzer so docker-compose-based skill manifests are not incorrectly flagged as `suspicious.install_untrusted_source` for intentional compose usage.

## Best Possible Solution

Define a narrow compose-aware manifest scanner policy that exempts benign loopback service metadata while preserving warnings for URL shorteners, public raw-IP install sources, mutable remote compose fetches, and weak container provenance.

## Reproduction Assessment

Yes. Source inspection on current main shows manifest YAML with any dotted-quad HTTP(S) URL can emit `suspicious.install_untrusted_source`, and the public 4.1.22 scan/file endpoints show the exact result with `127.0.0.1` compose metadata.

## Solution Assessment

No for a broad verified-registry allowlist or new metadata flag as the first fix; that is too coarse for a supply-chain scanner. The safer path is a narrow compose-manifest policy reviewed as security/product policy.
