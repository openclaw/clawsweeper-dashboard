## Summary

Keep open: this is a ClawHub issue, and the current scanner still has a manifest-level raw-IP heuristic that plausibly explains the docker-compose false-positive class. The remaining work is security-policy-sensitive because it must distinguish expected loopback compose metadata from unsafe install sources and weak container provenance.

## What This Changes

The issue asks ClawHub to reduce false-positive `suspicious.install_untrusted_source` scanner findings for docker-compose-based skills using bundled compose manifests and registry images.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that permits expected loopback service metadata while preserving warnings for unsafe install sources and weak container provenance, then cover it with regression tests and a spec note.

## Reproduction Assessment

Yes, by source inspection plus live API evidence. Current main scans manifest files for raw IPv4 URLs, and the affected 4.1.22 compose manifest contains loopback URLs while the live scan still reports `suspicious.install_untrusted_source`.

## Solution Assessment

Unclear until maintainers set scanner policy. A broad verified-registry allowlist or metadata escape hatch is riskier than a targeted compose-manifest exception paired with explicit container provenance rules.
