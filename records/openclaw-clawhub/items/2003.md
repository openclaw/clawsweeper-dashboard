## Summary

Keep open: this is a valid moderation-scanner false-positive class, but current evidence points to manifest loopback raw-IP URLs rather than GHCR image pulls. Changing this heuristic affects abuse detection, so it needs maintainer/security approval before implementation.

## What This Changes

The issue asks for the static analyzer to stop reporting `suspicious.install_untrusted_source` for docker-compose-based skills that pull expected container images.

## Best Possible Solution

Refine the manifest install-source heuristic to exempt clearly local loopback endpoints while preserving warnings for URL shorteners and public raw-IP install sources, with regression coverage for allowed and blocked manifest cases.

## Reproduction Assessment

Yes, by source inspection and live affected-version data. A YAML/JSON/TOML manifest containing `http://127.0.0.1:<port>/...` still matches `RAW_IP_URL_PATTERN` and flows through `scanManifestFile` to `suspicious.install_untrusted_source`; the GHCR image-pull hypothesis is not reproduced from current source.

## Solution Assessment

Yes, but narrower than the issue hypothesis. A loopback-aware manifest matcher is more maintainable than adding a verified-registry allowlist or new `installation_type` metadata field unless maintainers explicitly choose that policy.
