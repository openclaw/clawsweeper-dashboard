## Summary

Keep open: source inspection supports a current false-positive path for manifest loopback raw-IP URLs, while the GHCR image-pull hypothesis is not supported by the current static scanner code. Because this is a ClawHub issue involving security moderation policy, it needs maintainer/security follow-up rather than cleanup closure.

## What This Changes

The report asks the static moderation scanner to stop classifying docker-compose-based skills that pull verified registry images as `suspicious.install_untrusted_source`, or to classify that risk more precisely.

## Best Possible Solution

After maintainer/security approval, refine manifest scanning to exempt localhost/loopback install endpoints while preserving URL-shortener and public raw-IP findings, with regression tests for both safe and risky manifest cases.

## Reproduction Assessment

Yes, by source inspection. A YAML/JSON/TOML manifest containing `http://127.0.0.1:<port>/...` still matches `RAW_IP_URL_PATTERN` in `scanManifestFile` and emits `suspicious.install_untrusted_source`; I did not run tests because this review is read-only.

## Solution Assessment

No for a broad verified-registry allowlist or new metadata field as the immediate fix. The narrower maintainable path is loopback-aware manifest matching, but that still needs maintainer/security approval because it changes abuse-detection behavior.
