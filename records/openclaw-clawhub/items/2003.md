## Summary

Keep open: current main still has a manifest scanner path that can emit `suspicious.install_untrusted_source` for loopback/raw-IP URLs in YAML, and the affected live skill version still carries that static reason code. Refining Docker Compose/container provenance scanning is a security-policy decision, so this should stay with maintainers rather than be closed by cleanup.

## What This Changes

Refine ClawHub static scanning so docker-compose-based skills are not over-flagged as `suspicious.install_untrusted_source` when they use bundled compose files and expected local service URLs/container images.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that treats expected loopback MCP/dashboard health URLs as non-install sources, while preserving warnings for URL shorteners, public raw-IP install sources, remote compose fetches, mutable tags, and weak container provenance.

## Reproduction Assessment

Yes. Current main still scans manifest YAML for raw-IP URLs and live API output for `investorclaw` 4.1.22 still shows the reported static reason code; I did not run a local scanner fixture, so this is source-reproducible rather than test-reproduced.

## Solution Assessment

Unclear. A broad verified-registry allowlist is not the safest repair; a narrower compose-manifest exception plus explicit container provenance rules would be more maintainable.
