## Summary

Keep open: current main still has a source-reproducible false-positive path for loopback raw-IP URLs in manifest files, while the GHCR image-pull hypothesis is not supported by the deterministic scanner source. Changing this moderation heuristic is security-sensitive and needs maintainer/security approval.

## What This Changes

Requests tuning the static moderation analyzer so docker-compose skills with local compose files and registry images are not flagged as `suspicious.install_untrusted_source`.

## Best Possible Solution

Refine the manifest install-source heuristic to exempt localhost and loopback URLs while preserving warnings for public raw-IP URLs and shorteners, with regression tests for both allowed and warned cases.

## Reproduction Assessment

Yes. Source inspection shows a YAML/JSON/TOML manifest containing `http://127.0.0.1:<port>/...` matches `RAW_IP_URL_PATTERN` and flows through `scanManifestFile` to `suspicious.install_untrusted_source`; the public 4.1.22 scan shows the same reason code.

## Solution Assessment

No for the broad registry-allowlist or new metadata-field direction as the first step. A narrower loopback-aware manifest matcher is the most maintainable fix unless maintainers explicitly choose a verified-registry trust policy.
