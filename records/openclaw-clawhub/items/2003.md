## Summary

Keep open. Current main still contains the scanner path that can classify loopback URLs in YAML compose manifests as `suspicious.install_untrusted_source`, and changing that boundary is security scanner policy work rather than conservative cleanup.

## What This Changes

Request to refine ClawHub's static analyzer so docker-compose-based skills are not incorrectly flagged as `install_untrusted_source` for expected local compose metadata or trusted container-image use.

## Best Possible Solution

Codify a narrow scanner policy that exempts expected loopback compose service metadata while preserving warnings for URL shorteners, public raw-IP install sources, mutable remote compose fetches, and weak container provenance.

## Reproduction Assessment

Yes. Source inspection shows current main applies `RAW_IP_URL_PATTERN` to `.yml` manifests, and the public scan endpoint shows InvestorClaw 4.1.22 was flagged with `suspicious.install_untrusted_source` while its compose file contained matching `127.0.0.1` URLs.

## Solution Assessment

No for a broad verified-registry allowlist as the first fix; that is too coarse for a supply-chain scanner. The safer path is a narrow manifest-context refinement with regression coverage after maintainer/security sign-off.
