## Summary

Keep open. Current main still contains the manifest-scanner path that can classify loopback URLs in docker-compose YAML as `suspicious.install_untrusted_source`, and changing that rule is a security scanner policy decision rather than conservative cleanup.

## What This Changes

The issue asks to reduce a static analyzer false positive where docker-compose-based skills are flagged as `suspicious.install_untrusted_source`.

## Best Possible Solution

Codify a narrow manifest-scanner policy that exempts expected loopback compose service metadata while preserving warnings for URL shorteners, public raw-IP install sources, mutable remote compose fetches, and weak container provenance.

## Reproduction Assessment

Yes. Source inspection shows current main applies the raw-IP URL pattern to `.yml` manifests, and public scan data shows InvestorClaw 4.1.22 was flagged with the same reason while its compose file contained `127.0.0.1` service URLs.

## Solution Assessment

No for a broad verified-registry allowlist as the first fix; that is too coarse for a supply-chain scanner. The safer path is a narrow manifest-context refinement with regression coverage after maintainer/security sign-off.
