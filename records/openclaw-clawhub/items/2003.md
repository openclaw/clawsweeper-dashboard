## Summary

Keep open: current main still contains the manifest rule that can emit `suspicious.install_untrusted_source` for raw-IP/loopback URLs in YAML, live affected-version output shows the reported reason code, and changing Docker Compose/container provenance trust policy needs maintainer security judgment.

## What This Changes

The issue asks to reduce or reclassify static scanner `install_untrusted_source` findings for Docker Compose skills that pull registry images and expose localhost service metadata.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that permits expected loopback service metadata while preserving warnings for URL shorteners, public raw-IP install sources, remote compose fetches, mutable tags, and weak container provenance.

## Reproduction Assessment

Yes. Source inspection plus live API output show that current main still has a manifest scanner path for raw-IP URLs and the affected 4.1.22 compose file still carries the reported static reason code.

## Solution Assessment

Unclear. A broad verified-registry allowlist is not the safest default; a narrower compose-manifest exception plus explicit container provenance rules would be more maintainable.
