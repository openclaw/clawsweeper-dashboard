## Summary

Keep open. Current main still has a manifest scanner path that can emit `suspicious.install_untrusted_source` for raw-IP URLs in YAML/compose files, while recent main changes only made static suspicious findings advisory; changing the rule needs maintainer security-policy judgment.

## What This Changes

Request to reduce or clarify static `install_untrusted_source` findings for docker-compose skills that use local compose metadata and pull known container images.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that permits expected loopback service metadata while preserving warnings for URL shorteners, public raw-IP install sources, remote compose fetches, mutable tags, and weak container provenance.

## Reproduction Assessment

Yes, from source inspection plus live affected-version output. Current main still has a manifest scanner path that emits `install_untrusted_source` for raw-IP URLs in YAML, and the published 4.1.22 compose file contains loopback URLs with that static finding.

## Solution Assessment

Unclear. A broad verified-registry allowlist is not the best default; a narrower compose-manifest exception for expected loopback metadata plus explicit container provenance rules would be more maintainable.
