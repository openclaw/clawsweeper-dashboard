## Summary

Keep open for maintainer/security follow-up. Current main and public scan data support a real scanner over-fire path, but the evidence points to loopback raw-IP URLs in compose manifests rather than the GHCR image reference, and ClawHub policy keeps issues open.

## What This Changes

Request to reduce false positives from the static analyzer’s `suspicious.install_untrusted_source` rule for docker-compose-based skills that use local compose files and public registry images.

## Best Possible Solution

Codify a narrow manifest-scanner policy that exempts loopback compose service metadata while preserving warnings for URL shorteners, raw public-IP install sources, mutable remote compose fetches, and unpinned container provenance.

## Reproduction Assessment

Yes. Source inspection shows current main applies the raw-IP URL pattern to YAML manifests, and the public 4.1.22 scan/file endpoints show a matching compose manifest plus the `suspicious.install_untrusted_source` result.

## Solution Assessment

No for the broad verified-registry allowlist suggested in the issue; that would be too coarse for a supply-chain scanner. A narrower context-aware manifest exemption with regression tests is the safer direction after maintainer/security approval.
