## Summary

Keep open. Current main still has a source-reproducible path where YAML manifest loopback URLs can emit `suspicious.install_untrusted_source`, and changing that scanner rule is security-sensitive policy work rather than cleanup-close material.

## What This Changes

Request refinement of the static analyzer so docker-compose-based skill manifests do not receive misleading `install_untrusted_source` warnings for expected compose metadata.

## Best Possible Solution

Codify a narrow manifest-scanner policy that exempts expected loopback compose service metadata while preserving warnings for URL shorteners, public raw-IP install sources, mutable remote compose fetches, and weak container provenance.

## Reproduction Assessment

Yes. Source inspection shows current main applies `RAW_IP_URL_PATTERN` to YAML manifests, and the public InvestorClaw 4.1.22 compose file plus scan endpoint show the affected `127.0.0.1` compose URLs alongside `suspicious.install_untrusted_source`.

## Solution Assessment

No for a broad verified-registry allowlist as the first fix. The safer maintainable direction is a narrow manifest-context refinement after maintainer/security approval.
