## Summary

Keep open for maintainer/security follow-up. Current main and public scan data support a real scanner over-fire path, but the evidence points to loopback raw-IP URLs in compose manifests rather than the GHCR image reference, and ClawHub policy keeps issues open.

## What This Changes

Request a static analyzer refinement so docker-compose based skills with local compose files and container images are not misclassified as `suspicious.install_untrusted_source`.

## Best Possible Solution

Codify a narrow manifest-scanner policy that exempts loopback compose service metadata while preserving warnings for URL shorteners, raw public-IP install sources, mutable remote compose fetches, and unpinned container provenance.

## Reproduction Assessment

Yes by source inspection and public API evidence: current main applies `RAW_IP_URL_PATTERN` to `.yml` manifests, and the affected 4.1.22 compose file contains matching `127.0.0.1` service URLs. The public scan endpoint reports the matching static reason code for that version.

## Solution Assessment

No for a broad registry allowlist as the first move; that is too coarse for a supply-chain scanner. The safer solution is a narrow manifest-context refinement with regression tests after maintainer/security sign-off.
