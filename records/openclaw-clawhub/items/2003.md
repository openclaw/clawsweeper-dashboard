## Summary

Keep open: current main still has a source-reproducible false-positive path for manifest YAML/JSON/TOML containing loopback raw-IP URLs, while the GHCR image-pull hypothesis is not supported by the scanner code path I found. Because this changes a moderation/security heuristic, it should get maintainer or security approval before implementation.

## What This Changes

The issue asks to reduce false positives from the static moderation analyzer when docker-compose-based skills use bundled compose/install manifests and public container images.

## Best Possible Solution

Refine manifest install-source detection to exempt localhost and loopback install endpoints while preserving warnings for public raw-IP URLs and URL shorteners, with regression tests covering both safe and unsafe manifest cases.

## Reproduction Assessment

Yes, by source inspection. `runStaticPublishScan` feeds text manifests into `runStaticModerationScan`, and `scanManifestFile` applies a raw IPv4 URL matcher to YAML/JSON/TOML without a loopback exclusion, though I did not run a live publish.

## Solution Assessment

Yes, but narrower than the issue hypothesis. A loopback-aware manifest matcher is more maintainable than a verified-registry allowlist or new installation-type metadata unless maintainers explicitly choose that product policy.
