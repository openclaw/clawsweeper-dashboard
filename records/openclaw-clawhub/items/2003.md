## Summary

Keep open. Current main still contains the source-level path that can classify loopback Docker Compose service metadata as `suspicious.install_untrusted_source`; the affected publisher worked around it in later releases, but the scanner policy itself is not fixed.

## What This Changes

The issue requests narrowing the static analyzer's `install_untrusted_source` heuristic for docker-compose-based skills so legitimate local compose metadata and container-image pulls are not treated like untrusted shell installs.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that exempts expected loopback service metadata while preserving warnings for URL shorteners, mutable remote compose fetches, public raw-IP install sources, and weak container provenance.

## Reproduction Assessment

Yes, source-level. Current main applies `RAW_IP_URL_PATTERN` to YAML manifests, and the published InvestorClaw 4.1.22 compose file contains `http://127.0.0.1` URLs while its static scan reports `suspicious.install_untrusted_source`.

## Solution Assessment

No for a broad verified-registry allowlist as the first fix. The safer path is a narrow manifest-context refinement for loopback compose metadata after maintainer/security approval.
