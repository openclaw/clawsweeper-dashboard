## Summary

Keep open. Current main still contains a manifest raw-IP/shortener rule that can emit `suspicious.install_untrusted_source` for compose YAML, the public affected version still shows that static finding, and changing this scanner trust boundary needs maintainer security-policy review rather than cleanup closure.

## What This Changes

The issue asks ClawHub to reduce false positives from the static analyzer's `suspicious.install_untrusted_source` rule for docker-compose-based skills that pull public container images.

## Best Possible Solution

Define a narrow compose-manifest scanner policy that exempts expected loopback service metadata while preserving warnings for URL shorteners, public raw-IP install sources, mutable remote compose fetches, and weak container provenance.

## Reproduction Assessment

Yes. Source inspection shows the current manifest scanner applies the raw-IP rule to YAML manifests, and the public 4.1.22 scan/file endpoints show `suspicious.install_untrusted_source` alongside loopback compose metadata; I did not run the scanner locally.

## Solution Assessment

No for a broad verified-registry allowlist as the first fix, because registry identity does not prove immutability or provenance. A narrower compose-manifest rule with maintainer-approved security policy is the safer path.
