## Summary

Keep open: current main still rejects live package publishes whenever a skill or skill alias resolves to the same normalized package name, before the existing same-owner package update checks can run. The related issues cover runtime-id migration or publisher access, not this same-owner skill/package slug collision.

## What This Changes

A package publisher reports that publishing a new version of an existing code-plugin package is blocked when the same owner also has a skill with the same slug/name.

## Best Possible Solution

Allow a same-owner existing package to publish new versions despite a matching owned skill slug, while preserving cross-owner skill/alias collision protection and adding regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create an existing package and a skill/alias with the same normalized name, then a real package publish hits the collision guard before release insertion. I did not run a live publish because it requires the reporter's account/package state.

## Solution Assessment

Yes, the narrow maintainable fix is to make the collision guard aware of same-owner existing package version publishes while keeping cross-owner and alias protections. An admin-only migration path may still help edge cases, but it would not be the smallest fix for this reported blocker.
