## Summary

Keep this open: current main still rejects package publishes whose normalized name resolves to a skill slug or alias, the latest release still contains the same guard, and the linked fix PR was closed unmerged after collaborator feedback that the namespace policy needs a different maintainer-approved shape.

## What This Changes

The issue requests that an existing package owner be able to publish a new package version even when the same owner also has a skill with the same slug/name.

## Best Possible Solution

Maintainers should decide and document the typed skill/package namespace policy, then update the package publish guard, regression tests, and relevant specs while preserving reserved alias and ownership protections.

## Reproduction Assessment

Yes. Source inspection on current main shows the normalized package name still hits an unconditional skill-or-alias collision guard before package version insertion can complete; I did not run a live publish because that would require account and package fixture state.

## Solution Assessment

Unclear. A same-owner existing-package exception would address the reporter's concrete upgrade, but collaborator discussion points toward a broader typed-namespace policy that maintainers should settle first.
