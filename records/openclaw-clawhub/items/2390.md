Keep this open: current main still has the unconditional package-vs-skill slug collision guard, the latest release tag still contains the same guard, and the linked same-owner exception PR was closed unmerged after collaborator feedback that the namespace policy needs maintainer review.

## What This Changes

The issue asks to unblock publishing a new version of an existing code-plugin package when the same owner also has a skill with the same slug, or to provide a workable migration path.

## Best Possible Solution

Maintainers should decide and document the skill/package namespace policy, then update `convex/packages.ts`, focused publish tests, and the relevant specs while preserving reserved alias, first-publish, and ownership protections.

## Reproduction Assessment

Yes. Source inspection shows a real package publish whose normalized name matches any skill slug or alias still reaches the unconditional collision guard before existing-package owner checks; I did not run a live publish because that requires account/package fixture state.

## Solution Assessment

Unclear. A narrow same-owner existing-package exception would address the reporter's concrete upgrade, but collaborator discussion and conflicting specs point to a broader typed-namespace policy that maintainers should settle first.
