## Summary

Keep open. Current main still rejects package publish whenever a skill has the same slug, without exempting an existing same-owner package version publish; the rename symptom is already covered by the older CLI serialization fix tracked at https://github.com/openclaw/clawhub/issues/1300.

## What This Changes

The issue requests that package version publishing not be blocked by a same-owner legacy skill with the same slug, or that the CLI/server provide a usable migration path.

## Best Possible Solution

Permit a new version only for an already-existing package controlled by the authenticated publisher when the colliding skill is owned by the same publisher, while preserving rejection for first publishes and different-owner collisions.

## Reproduction Assessment

Yes from source inspection, though not by executing a live publish: current main still throws on any skill slug matching the package name before inserting the release. The reporter's production error points at that same branch.

## Solution Assessment

Yes for the package publish half: a narrow existing-package same-owner exception is the maintainable fix boundary. The `newSlug required` symptom should not drive this issue's fix because current main already sends `newSlug` correctly and https://github.com/openclaw/clawhub/issues/1300 tracks that older CLI serialization bug as implemented.
