## Summary

Keep this open: current main and the latest release still reject real package publishes when the normalized package name resolves to any skill slug or alias, and the linked fix PR was closed unmerged after collaborator feedback that the namespace policy needs a maintainer decision.

## What This Changes

The issue reports that publishing an existing package version is blocked when the same account also owns a skill with the same slug.

## Best Possible Solution

Maintainers should decide and document the typed skill/package namespace policy, then update the publish guard, regression tests, and relevant specs together while preserving reserved alias and ownership protections.

## Reproduction Assessment

Yes from source. Current main still looks up a skill by normalized package name and rejects any match before completing the real package publish path; I did not run a live Convex publish because this read-only review cannot create the required account/package state.

## Solution Assessment

Unclear. A narrow same-owner existing-package exception would address the reporter’s concrete upgrade, but collaborator discussion points toward a broader typed-namespace policy that maintainers should settle first.
