## Summary

Keep open: current main still rejects package publishes when the normalized package name resolves to any skill slug, and the direct fix PR was closed unmerged after collaborator feedback that the namespace policy needs a different shape.

## What This Changes

The issue reports that package version publishing is blocked when the package name matches an existing skill slug, even for an existing package owned by the same account.

## Best Possible Solution

Settle and document the typed skill/package namespace policy, then update package publish validation, regression tests, and specs so legitimate package upgrades are not blocked while reserved and cross-owner protections remain intact.

## Reproduction Assessment

Yes from source: current main still calls the skill slug lookup during package publish and throws on any matching skill slug. I did not run a live Convex publish path because this review is read-only.

## Solution Assessment

Unclear. A narrow same-owner existing-package exception would address the concrete report, but collaborator feedback points toward an explicit typed-namespace policy rather than the closed PR's narrow shape.
