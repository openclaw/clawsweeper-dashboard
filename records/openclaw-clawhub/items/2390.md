- `impact:security`: The requested policy change affects artifact identity, anti-squatting protections, and trust signals across skills and packages.

## Summary

Current main still contains the reported package-versus-skill collision guard, and the only linked fix was closed unmerged because a collaborator preferred a broader typed-namespace decision. This issue remains the canonical open discussion and needs explicit product and security approval rather than cleanup closure.

## What This Changes

Unblock publishing upgrades to an existing package when its name matches a skill slug, or provide a reliable migration path for renaming the skill.

## Best Possible Solution

Approve and document either typed skill/package namespaces or a clear grandfathered migration rule, preserve official aliases and reserved-name protections, and add real Convex/CLI validation for both an existing-package upgrade and positional skill rename.

## Reproduction Assessment

Yes at source level. Current main contains the exact unconditional collision guard matching the reported publish error; the rename subproblem still needs a current-CLI runtime check.

## Solution Assessment

No. The closed same-owner exception would introduce owner-dependent namespace semantics; an explicit typed-namespace or grandfathered-migration policy is more coherent and maintainable.
