## Summary

Keep open: current main still rejects any package publish whose normalized name matches a skill slug, and the linked fix PR was closed unmerged after collaborator feedback that the namespace policy needs a different shape.

## What This Changes

Publishing a new package version should not be blocked solely because a same-named skill exists when the package already exists under that package name.

## Best Possible Solution

Document the typed skill/package namespace policy, then update package publish validation and regression tests so legitimate package upgrades are not blocked by skill slugs while ownership and reservation protections remain intact.

## Reproduction Assessment

Yes. Current main still rejects a package publish when `getSkillBySlugInternal({ slug: name })` returns a skill before the release path can apply existing-package owner checks; I did not run a live publish because this read-only review does not have the reporter's authenticated package fixture state.

## Solution Assessment

Unclear. A narrow same-owner existing-package exception would address the reported upgrade case, but collaborator feedback points toward a broader typed-namespace decision rather than the closed PR's shape.
