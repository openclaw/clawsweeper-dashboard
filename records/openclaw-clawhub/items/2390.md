## Summary

Keep this open: current main and the latest release still reject any package publish whose normalized name matches a skill slug, and the linked same-owner exception PR was closed unmerged while maintainers discuss the broader skill/package namespace policy.

## What This Changes

The issue requests unblocking package version publishes when an existing package owner also owns a skill with the same slug, or providing an actionable migration path.

## Best Possible Solution

Maintainers should decide and document the skill/package namespace policy, then update the package publish guard, regression tests, and durable specs to cover first publishes, existing package upgrades, aliases, routes, and ownership checks.

## Reproduction Assessment

Yes. Source inspection of current main shows the package publish path still looks up a skill by normalized package name and rejects any match; I did not run a live Convex publish in this read-only review.

## Solution Assessment

Unclear. A narrow existing-package exception would address the reporter's concrete failure, but collaborator discussion and specs indicate maintainers should first choose the long-term typed namespace policy.
