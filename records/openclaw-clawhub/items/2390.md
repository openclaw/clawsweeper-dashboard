## Summary

Keep open: current main and v0.20.0 still reject the reported package upgrade, and the available rename flow retains the old skill slug as an alias that continues triggering the same guard. The linked fix closed unmerged after collaborator feedback established that the correct cross-type namespace policy still needs maintainer approval.

## What This Changes

Unblock package version publishing when a skill or retained skill alias shares the package name, while preserving package identity protections.

## Best Possible Solution

Adopt and document an explicit typed skill/package namespace contract, preserve official aliases, route reservations, historical-name protections, and package authorization, then validate first publish, existing-package upgrades, and legacy collision migration against a real Convex deployment.

## Reproduction Assessment

Yes. Current main and v0.20.0 contain the exact unconditional rejection path, and source inspection also shows why renaming cannot clear it because the old slug remains an alias.

## Solution Assessment

No settled implementation is currently best. The closed same-owner exception is narrower than collaborator feedback, while a typed-namespace fix must explicitly preserve security-sensitive aliases, reservations, and ownership checks.
