## Summary

Keep open. Current main still matches the reported dashboard gap: the UI can scope queries by publisher but auto-selects the personal publisher and exposes no visible control to switch to an org publisher, while the backend already supports org-scoped package and skill dashboard reads.

## What This Changes

The issue requests that `/dashboard` expose organization-owned packages for signed-in publisher members after transfer or publish.

## Best Possible Solution

Add a Dashboard publisher selector backed by `api.publishers.listMine`, scope both skills and packages to the selected publisher, pass the selected owner into creation actions where supported, and cover personal-plus-org publishers in route tests.

## Reproduction Assessment

Yes, by source inspection. A signed-in user with both a personal publisher and an org membership lands on `/dashboard`, the effect selects the personal publisher, and no rendered control calls `setSelectedPublisherId` to switch to the org.

## Solution Assessment

Yes. A Dashboard owner selector is the narrow maintainable fix because backend package and skill queries already accept `ownerPublisherId` and enforce publisher membership.
