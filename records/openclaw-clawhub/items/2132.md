## Summary

Keep open: current main still leaves organization-owned dashboard content unreachable from the UI because `/dashboard` selects the personal publisher and renders no publisher switcher, while the backend already supports owner-publisher scoped dashboard reads.

## What This Changes

The issue requests a dashboard publisher selector so signed-in members can view and publish against organization-owned packages and skills.

## Best Possible Solution

Add a visible dashboard publisher selector backed by `api.publishers.listMine`, keep skill/package reads scoped to the selected publisher, align publish actions with the selected handle, and add personal-plus-org dashboard coverage.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: sign in as a user with both a personal publisher and an org publisher, load `/dashboard`, and the code selects the personal publisher while rendering no control that can switch to the org publisher.

## Solution Assessment

Yes. A visible dashboard publisher selector is the narrowest maintainable fix because the backend owner-publisher reads and membership checks already exist; grouping and persistence can be follow-ups.
