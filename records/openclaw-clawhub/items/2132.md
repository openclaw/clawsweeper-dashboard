## Summary

Keep open: current main still appears to hide org-owned dashboard content because `/dashboard` defaults to the personal publisher and has no rendered control that can switch the selected publisher to an org.

## What This Changes

The issue requests dashboard visibility and owner switching for packages and skills owned by organization publishers the signed-in user can manage.

## Best Possible Solution

Add a dashboard publisher selector backed by `api.publishers.listMine`, keep skills and packages scoped to the selected publisher, align publish actions with that owner handle, and cover the personal-plus-org case in dashboard tests.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: sign in as a user with both personal and org publishers, load `/dashboard`, and current code selects the personal publisher while exposing no UI path to set the org publisher id.

## Solution Assessment

Yes. A visible publisher selector is the narrowest maintainable fix because the backend already supports membership-checked `ownerPublisherId` dashboard reads; grouped sections and persistence can be follow-ups.
