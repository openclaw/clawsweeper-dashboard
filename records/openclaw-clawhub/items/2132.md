## Summary

Keep open: current main still defaults `/dashboard` to the personal publisher and exposes no UI control to switch to an organization publisher, while the backend already supports membership-checked org dashboard reads.

## What This Changes

Expose a Dashboard publisher/org selector so signed-in members can view organization-owned packages and publish actions can target the selected publisher.

## Best Possible Solution

Add a `/dashboard` publisher selector backed by `api.publishers.listMine`, keep personal as the default, scope skills/packages and quick publish actions to the selected publisher, and cover personal-plus-org behavior in tests.

## Reproduction Assessment

Yes. A high-confidence source path exists: sign in as a user with personal and org publishers, load `/dashboard`, and current code auto-selects personal without rendering any control that can set the org publisher id.

## Solution Assessment

Yes. A visible dashboard publisher selector is the narrowest maintainable fix because the backend already accepts `ownerPublisherId`, publish pages already use owner-selector patterns, and the orgs spec already defines dashboard owner selection.
