## Summary

Keep open: current main still defaults `/dashboard` to the personal publisher and provides no UI path to switch to an organization publisher, while the backend already supports membership-checked org dashboard reads and the orgs spec calls for dashboard owner selection.

## What This Changes

The issue requests that `/dashboard` expose organization publishers so a member can view org-owned packages, including the reported `@clawkit` packages, after transfer or publish.

## Best Possible Solution

Add a `/dashboard` publisher selector backed by `api.publishers.listMine`, keep personal as the default, scope skills/packages and publish quick actions to the selected publisher, and cover personal-plus-org behavior in tests.

## Reproduction Assessment

Yes. A high-confidence source path exists: sign in as a user with personal and org publishers, load `/dashboard`, and current code auto-selects the personal publisher without rendering a control that can set the org publisher id.

## Solution Assessment

Yes. A visible publisher selector is the narrowest maintainable fix because existing queries already accept `ownerPublisherId`, publish pages already use owner-selector patterns, and the spec already defines dashboard owner selection.
