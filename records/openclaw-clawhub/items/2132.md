## Summary

Keep open: current main still defaults the dashboard to the personal publisher and exposes no control to switch to an organization publisher, while the backend scoped dashboard reads and the org spec already support this flow.

## What This Changes

The dashboard should expose organization publishers so members can view org-owned skills and packages after transfer or publish.

## Best Possible Solution

Add a dashboard publisher selector backed by api.publishers.listMine, keep skills and packages scoped to the selected publisher, align quick publish actions with that owner handle, and cover the personal-plus-org case in dashboard tests.

## Reproduction Assessment

Yes. A high-confidence source path exists: a signed-in user with both personal and org publishers loads /dashboard, the effect selects the personal publisher, and the rendered UI exposes no path to set the org publisher id.

## Solution Assessment

Yes. A visible publisher selector is the narrowest maintainable fix because the backend already has membership-checked ownerPublisherId dashboard reads; grouped sections or persistence can be follow-ups.
