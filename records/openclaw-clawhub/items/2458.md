## Summary

This useful feature is not merge-ready: the branch predates current main’s database-backed official-publisher registry and introduces separate handle-based authorization in both backend and CLI paths. The PR is also conflict-blocked and lacks real Convex publish/retrieval proof.

## What This Changes

The PR allows official organization publishers to bundle a root `skill-card.md` and preserves or trusts that card through publishing, generation, public API, verification, and CLI flows.

## Best Possible Solution

Rebase and authorize bundled cards exclusively from the server’s database-backed official-publisher row, keep authorization policy out of the CLI, document the trusted-content invariant in `specs/`, and prove both allowed and rejected flows on real local Convex.

## Reproduction Assessment

Yes at source level: designate an official organization in the database that is absent from the CLI’s two-handle list, and the CLI strips its card before the authoritative server can evaluate it. The inverse policy-drift risk is also clear from the compiled backend helper.

## Solution Assessment

No. The capability may be appropriate, but duplicating official organization handles in the client and backend is not the safest solution after current main established a database-backed registry.
