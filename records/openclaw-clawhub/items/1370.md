## Summary

Keep open. This remains a valid product gap: current main has partial package soft-delete plumbing, but not an end-to-end owner-facing hide/delete/undelete or version delist workflow for plugin packages, and the discussion ties it to a real install-blocking case plus related open lifecycle requests #905 and #1744.

## What This Changes

Add owner/admin self-service hide, delete, undelete, or delist controls for ClawHub plugin packages across API, CLI, website UI, and docs.

## Best Possible Solution

Define the package and package-release lifecycle policy, then ship owner/admin API, CLI, dashboard/settings UI, tests, and docs that let publishers withdraw and restore a broken plugin package or selected release without maintainer intervention.

## Reproduction Assessment

Not applicable as a feature request, but the product gap is high-confidence from static current-main inspection. Package-wide DELETE exists, while restore, hide/unhide, version delist, owner UI, CLI, and docs paths are absent or incomplete.

## Solution Assessment

Unclear. A package-wide DELETE alone is not the best complete solution because restore behavior and release-level delisting need maintainer lifecycle decisions before implementation.
