## Summary

Current main and release v0.20.0 implement the requested artifact-first enforcement model, including candidate blocking, clean-version preservation, publisher remediation, two-distinct-artifact escalation, and three-attempt escalation. This issue must nevertheless remain open because it is collaborator-authored and defines security enforcement policy requiring explicit maintainer disposition.

## What This Changes

Soften malicious scanner enforcement by blocking candidate artifacts first and reserving account bans for repeated or broader malicious behavior.

## Best Possible Solution

Keep the released artifact-first enforcement model and have a maintainer explicitly confirm that its escalation policy is the intended permanent security boundary before closing this issue.

## Reproduction Assessment

Yes at the source level: current main has explicit malicious-finding thresholds, quarantine call paths, focused tests, and a local-auth end-to-end scenario covering the enforcement journey.

## Solution Assessment

Yes. Candidate-only quarantine, preservation of clean published versions, actionable remediation email, and thresholded account enforcement are a coherent, spec-backed solution; only explicit maintainer policy confirmation remains.
