## Summary

Current main implements the central artifact-first enforcement and exact numeric thresholds for both skills and plugins. This should remain open because it is collaborator-authored and the remaining evasion, abuse, and human-confirmation clauses require explicit maintainer security-policy confirmation.

## What This Changes

Soften malicious scanner enforcement by blocking submitted artifact versions, preserving prior clean versions, notifying publishers, and escalating accounts only at defined repeat-offense thresholds.

## Best Possible Solution

Keep the shipped artifact-first flow and shared two-artifact/three-attempt thresholds, then have security owners either codify the remaining exceptional escalation triggers in `specs/security-moderation.md` with focused runtime coverage or confirm they are already handled by separate account-enforcement paths.

## Reproduction Assessment

Yes at source level: current main has a concrete shared enforcement path with explicit two-distinct-artifact and three-attempt thresholds, plus dedicated malicious publish-flow coverage. A fresh runtime reproduction was not run because this review is read-only.

## Solution Assessment

Mostly yes: one shared audit-backed escalation helper for skills and plugins is the narrowest maintainable solution. Maintainers still need to decide and document whether evasion, abuse, and human-confirmed findings bypass the numeric thresholds.
