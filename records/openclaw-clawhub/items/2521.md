## Label Justifications

## Summary

Current main implements and production-shipped the central artifact-first policy: malicious candidates are blocked, prior clean releases remain live, publishers receive remediation guidance, and account autobans wait for two distinct artifacts or three attempts on one artifact. This collaborator-authored issue should remain open because the proposed immediate escalation rules for evasion, abuse, and human-confirmed malicious behavior are not codified and require explicit security-policy judgment.

## What This Changes

Soften scanner-triggered malicious publish enforcement so malicious candidates are blocked first and account bans occur only after defined escalation thresholds or exceptional abuse.

## Best Possible Solution

Preserve the shipped candidate-version quarantine and thresholded autoban model, then explicitly approve, reject, or define immediate account-ban paths for evasion, abuse, and human-confirmed malicious conduct in the moderation specification and corresponding enforcement entry points.

## Reproduction Assessment

Yes at source level: current main clearly exposes the two-distinct-artifact and three-attempt thresholds, candidate quarantine behavior, notification flow, and account-ban side effects, with focused regression coverage. This is primarily a policy question rather than a runtime failure requiring a new reproduction.

## Solution Assessment

Yes for ordinary scanner findings: artifact-level quarantine with thresholded account escalation is the narrowest maintainable approach and is already implemented. The exceptional immediate-ban criteria still require explicit maintainer security-policy judgment rather than an autonomous code change.
