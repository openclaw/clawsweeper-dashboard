## Summary

Current main already implements the central artifact-first enforcement model and shipped it in v0.20.0, including shared skill/plugin thresholds of two distinct artifacts or three findings for one artifact. This issue should nevertheless remain open for explicit maintainer disposition because its author is a repository collaborator and the requested local-folder remediation was intentionally replaced with the safer stored-report download workflow.

## What This Changes

Soften malicious scanner enforcement by blocking candidate skill/plugin versions first, preserving prior clean releases, notifying publishers, and escalating repeated findings to account bans.

## Best Possible Solution

Keep the shipped shared artifact-first enforcement and stored-report remediation flow, document its security invariants in a focused moderation spec, and explicitly close this issue once maintainers confirm that policy.

## Reproduction Assessment

Yes. Current source exposes the complete shared finding-count and autoban path, and the merged related pull request reports a real local-auth flow covering clean publication, three malicious retries, artifact emails, retained clean latest state, and eventual account disablement.

## Solution Assessment

Yes, with one deliberate adjustment: shared candidate-version blocking plus thresholded escalation is the narrow maintainable solution, while stored scan-report download is safer than restoring local-folder scanner-oracle uploads.
