## Summary

Current main and v0.20.0 implement the central artifact-first policy: malicious candidates are blocked, prior clean releases remain live, publishers receive remediation guidance without an appeal CTA, and account autobans wait for two distinct artifacts or three attempts on one artifact. This issue must nevertheless remain open because it is collaborator-authored and the requested exceptional escalation rules for evasion, spam/rate-limit abuse, or human-confirmed malicious behavior still require explicit maintainer confirmation.

## What This Changes

Soften scanner-triggered enforcement so malicious publish attempts block the submitted skill or plugin version first and escalate to account bans only at defined repeat-abuse thresholds.

## Best Possible Solution

Keep the shipped artifact-first enforcement, document or confirm that exceptional evasion and human-confirmed abuse use existing manual/rate-limit ban paths, then let a maintainer explicitly close the issue if no additional policy work remains.

## Solution Assessment

Yes for the central problem: the shared current implementation is narrower and more maintainable than immediate account autobans, while the exceptional-abuse branch still needs a maintainer policy confirmation.
