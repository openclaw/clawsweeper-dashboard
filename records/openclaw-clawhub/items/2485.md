Keep this PR open: it contains useful shared metering work, but the patch can silently stop existing public download counters on upgraded deployments where the new metering HMAC secret is not configured.

## What This Changes

The PR adds shared Convex download metering for skill and package/plugin downloads with HMAC identity hashing, daily dedupe rows, weekly snapshots, cron jobs, tests, and an internal spec note.

## Best Possible Solution

Keep the shared metering design, but preserve or explicitly gate the legacy counter path during upgrade so existing download stats cannot silently stall.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: `hashDownloadIdentity` throws without `DOWNLOAD_METERING_HMAC_SECRET`, and both download handlers catch the whole metric/counter write as best effort. I did not run tests because this review is read-only.

## Solution Assessment

No; the shared metering design is plausible, but making existing public counters depend on a newly required secret without an upgrade-safe fallback or deployment gate is not the safest implementation.
