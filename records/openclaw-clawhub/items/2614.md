## Summary

Keep this member-authored security PR open. It correctly targets a source-reproducible bearer-token disclosure path and adds layered allowlist/CSP protection, but the loopback gate still fails open for any defined `NODE_ENV` other than exactly `production`, the required durable trust-boundary spec is absent, and the submitted proof does not exercise the real authenticated handoff.

## What This Changes

Restricts docs-auth loopback return origins to development, removes loopback CSP form targets, and adds environment-specific regression tests.

## Best Possible Solution

Permit loopback callbacks only under an explicit local-development signal, retain the production CSP restriction, record the docs-auth token-handoff invariant under `specs/`, and demonstrate production rejection plus local acceptance with redacted real-flow diagnostics.

## Reproduction Assessment

Yes. Current main unconditionally accepts loopback return origins, and the docs-auth route automatically submits the signed-in user's bearer token to the resulting callback.

## Solution Assessment

No, not yet. Splitting trusted production origins from local callbacks and tightening CSP is the correct design, but the loopback condition must be explicitly local-only and supported by durable specification and real-flow proof.
