## Label Justifications

- `P0`: The current behavior can disclose a signed-in user's reusable bearer token across a production browser trust boundary.
- `merge-risk: 🚨 security-boundary`: The proposed runtime classification directly determines whether authenticated tokens may be posted to loopback processes.

## Summary

This member-authored security PR addresses a real bearer-token exposure with a focused diff, but it should remain open because the loopback gate still trusts any `NODE_ENV` other than `production`, the authenticated handoff lacks real before/after runtime proof, and the repository-required trust-boundary invariant is not recorded in `specs/`.

## What This Changes

Restricts docs-auth loopback return origins to development, removes loopback CSP form targets, and adds environment-specific regression tests.

## Best Possible Solution

Enable loopback callbacks only from an affirmative local-development signal, add fail-closed tests for staging/test/unknown values, keep the deployed CSP aligned, document the token-destination invariant under `specs/`, and demonstrate the authenticated handoff in both production and local development.

## Reproduction Assessment

Yes from source inspection: current main accepts loopback origins unconditionally, and the docs-auth route builds its token callback from the accepted origin. The branch's residual staging/test failure is also directly reproducible by setting `NODE_ENV` to a non-production value.

## Solution Assessment

No, not yet; the narrow allowlist/CSP approach is appropriate, but local development must be recognized affirmatively rather than by trusting every environment that is not exactly production.
