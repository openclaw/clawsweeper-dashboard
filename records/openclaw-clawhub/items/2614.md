## Label Justifications

- `P0`: The current flow can disclose a signed-in ClawHub bearer token to an attacker-controlled local process.
- `merge-risk: 🚨 security-boundary`: Merging the branch as written can preserve loopback bearer-token delivery in publicly reachable non-production runtimes.

## Summary

Current main still exposes the docs-auth bearer token to allowlisted loopback callbacks, so this security fix remains necessary. The branch is not merge-ready because its reused runtime helper trusts every non-`production` environment, the application/CSP invariant is not persisted in `specs/`, and the submitted proof does not exercise the real authenticated POST.

## What This Changes

The PR gates docs-auth loopback return origins by runtime, removes loopback destinations from the deployed CSP, and adds development/production unit tests.

## Best Possible Solution

Permit loopback callbacks only under an affirmative local-development signal, keep the deployed CSP aligned, record the bearer-token destination invariant in a focused `specs/` note, and demonstrate the authenticated handoff's production rejection and local-development success with redacted diagnostics.

## Reproduction Assessment

Yes. Current-main source provides a high-confidence path: a loopback `return_to` is accepted, converted to the callback action, and receives the auto-submitted authenticated token; the branch's staging/test residual is reproducible by setting a non-production `NODE_ENV`.

## Solution Assessment

No, not yet. Runtime gating and CSP cleanup are the narrow correct direction, but loopback must require affirmative local development and the cross-file security invariant must be specified and proven end to end.
