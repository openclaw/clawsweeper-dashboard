## Summary

Keep open: current main still has a source-level cursor handling bug for non-trending `GET /api/v1/skills`, and an open linked PR already targets the fix, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The report asks for `GET /api/v1/skills` cursor pagination to advance instead of returning the same first page and `nextCursor`.

## Best Possible Solution

Review and land the linked cursor-decoder fix after real API pagination proof is added, preserving the non-trending cursor contract and tracking any trending pagination request separately.

## Reproduction Assessment

Yes. A live `clawhub.ai` round trip reproduced byte-identical first and second pages, and current main source shows the emitted full helper cursor key is rejected when supplied back.

## Solution Assessment

Yes for the active fix direction. The narrow maintainable path is to accept full self-describing `getPage()` keys for the matching index while preserving stale/wrong-index guards, as the linked PR attempts.
