## Summary

Keep open: the pagination bug still reproduces against the live public API and current main still contains the source-level mismatch where `listPublicApiPageV1` emits full `getPage()` index-key cursors that `decodePublicListCursor` rejects as too long. The open linked fix PR https://github.com/openclaw/clawhub/pull/2286 is the right active implementation path, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Fix `GET /api/v1/skills` so replaying a returned `nextCursor` advances to the next page instead of repeating the first page.

## Best Possible Solution

Review and land https://github.com/openclaw/clawhub/pull/2286 or an equivalent narrow patch that accepts full self-describing `getPage()` cursors for non-trending sorts while preserving stale-index guards.

## Reproduction Assessment

Yes. A live API round trip returned byte-identical first and second pages, and current main source shows the decoder rejects the full cursor key shape that the API emits.

## Solution Assessment

Yes for the central bug. The narrow maintainable fix is to accept the full emitted self-describing cursor for the matching index while continuing to reject stale or wrong-index cursors; `trending` pagination should remain separate unless maintainers choose to extend the API contract.
