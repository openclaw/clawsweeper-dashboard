## Summary

Keep open under the ClawHub cleanup policy even though the supported pagination bug is fixed and shipped: current main and production advance non-`trending` cursor replays, while `trending` remains outside the documented cursor contract.

## What This Changes

The issue reports that `GET /api/v1/skills` repeats the first page and returns the same `nextCursor` when the server-emitted cursor is replayed across sort modes.

## Best Possible Solution

Keep the merged non-`trending` cursor decoder fix as the canonical implementation, and track any desired `trending` cursor support as a separate API design item.

## Reproduction Assessment

No. The original curl replay no longer reproduces for `sort=updated`: live production returns different slugs and a different cursor, and current main has focused regression coverage for replaying the returned full cursor.

## Solution Assessment

Yes for the documented non-`trending` bug. Accepting full self-describing `getPage()` keys only when the index and equality prefix match is a narrow fix that preserves stale-cursor safeguards.
