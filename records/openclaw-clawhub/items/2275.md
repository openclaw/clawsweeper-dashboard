## Best Possible Solution

Keep the merged cursor-decoder implementation as the canonical fix for documented non-trending pagination, and track any desired `trending` cursor support separately.

## Reproduction Assessment

No, not against current `main` or the live non-`trending` production path. Replaying the server-emitted `nextCursor` now advances to different slugs, and the regression test covers the full `getPage()` cursor replay path.

## Solution Assessment

Yes. The merged decoder change is the narrow maintainable fix because it accepts the full cursor shape the API already emits while preserving index and equality-prefix stale-cursor guards.
