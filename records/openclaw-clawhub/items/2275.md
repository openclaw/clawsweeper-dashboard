## Summary

Keep open: current main still reproduces the public skills API cursor bug, and the issue is already paired with an open closing PR that targets the same decoder path.

## What This Changes

The issue requests that `GET /api/v1/skills` honor the returned `nextCursor` instead of repeating the first page.

## Best Possible Solution

Land the linked cursor-decoder fix after real API pagination proof is added, and track trending cursor support separately if maintainers want that API behavior.

## Reproduction Assessment

Yes. A live production curl replay reproduced byte-identical first and second pages, and current main source shows the API emits full helper cursors that the decoder can reject.

## Solution Assessment

Yes. The narrow maintainable fix is to accept full self-describing `getPage()` keys for the matching non-trending index while preserving stale/wrong-index and legacy raw-array guards.
