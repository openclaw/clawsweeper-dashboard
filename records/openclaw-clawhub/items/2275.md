Keep open: current main still has the cursor decode path that can reject the self-describing cursor shape emitted by the public skill-list API, and an open linked PR already targets the fix. The issue should remain open until that implementation lands or is replaced.

## What This Changes

The issue reports that `GET /api/v1/skills` returns the first page again when the API's own `nextCursor` is supplied back as `cursor`.

## Best Possible Solution

Review and land the linked cursor decoder fix, then close this issue after the non-trending public API cursor round trip works on main and track trending separately if needed.

## Reproduction Assessment

Yes. A live API round trip reproduced the repeated page, and current main source shows the emitted helper cursor shape can be rejected on decode, restarting the skill list at the first page.

## Solution Assessment

Yes. The narrow maintainable path is to accept the full self-describing `getPage()` cursor for matching non-trending indexes while preserving stale or wrong-index guards.
