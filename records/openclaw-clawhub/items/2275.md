## Summary

This issue should stay open: current main still has a source-level cursor reuse bug for non-trending `GET /api/v1/skills` pages, and https://github.com/openclaw/clawhub/pull/2286 is already open with closing syntax for the fix. The linked PR is the right place to finish review rather than closing the issue early.

## What This Changes

The issue requests that `GET /api/v1/skills` honor its returned `nextCursor` so subsequent requests advance instead of returning the same page.

## Best Possible Solution

Review and land or close https://github.com/openclaw/clawhub/pull/2286, keeping the fix focused on accepting the self-describing cursor shape the API already emits while preserving stale-cursor safety and regression coverage.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: request a non-trending public skill list page, pass back the emitted self-describing cursor with helper tie-breakers, and current validation can reject it and restart from page one. The reporter also provided production curl output showing byte-identical pages.

## Solution Assessment

Yes, the linked PR’s direction appears to be the narrowest maintainable fix: accept full `getPage()` keys only for self-describing cursors on the matching index while keeping legacy raw-array and wrong-index guards strict.
