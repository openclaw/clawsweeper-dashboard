## Summary

Keep open: the documented non-trending API pagination bug is still present in current main/source and reproduced on production, and the linked fixing PR is still open rather than merged.

## What This Changes

The issue reports that GET /api/v1/skills ignores the server-provided nextCursor and repeats the first page for paginated sort modes.

## Best Possible Solution

Review and land https://github.com/openclaw/clawhub/pull/2286 once it has sufficient after-fix real API proof, keeping trending pagination separate unless maintainers intentionally expand the contract.

## Reproduction Assessment

Yes. Production repeats the first page when the advertised nextCursor is reused, and current main still emits getPage index keys that the decoder rejects because of their full tie-breaker shape.

## Solution Assessment

Yes. The narrow maintainable fix is to accept full self-describing getPage keys for the selected index and equality prefix while preserving wrong-index and legacy raw-array guards.
