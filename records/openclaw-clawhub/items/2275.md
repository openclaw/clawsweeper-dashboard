## Summary

Keep open: current main still has the public skill-list cursor decoder path that rejects the full self-describing cursor shape the API emits, and the live API still repeats the first page when that cursor is replayed. The linked fix PR is open, so this issue should stay open until that implementation lands or is replaced.

## What This Changes

The issue reports that `GET /api/v1/skills` returns the same page and `nextCursor` when clients replay the advertised pagination cursor.

## Best Possible Solution

Land or replace the linked PR’s narrow decoder fix for supported non-trending sorts, then close this issue after a real API cursor round trip advances to the next page.

## Reproduction Assessment

Yes. A live API round trip reproduced the repeated page, and current main source shows the emitted 4-field `getPage()` cursor can be rejected by the decoder and restart at the first page.

## Solution Assessment

Yes. The narrow maintainable fix is to accept matching self-describing `getPage()` cursors with tie-breaker fields while preserving wrong-index and legacy raw-cursor guards.
