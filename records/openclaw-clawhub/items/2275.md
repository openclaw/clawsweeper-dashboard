## Summary

Keep open: the public API pagination bug still reproduces on the current released main, and an open linked PR already targets the cursor decoder fix, so this issue should remain open until that PR lands or is replaced.

## What This Changes

`GET /api/v1/skills` should honor the returned `nextCursor` so subsequent non-trending requests advance instead of repeating the first page.

## Best Possible Solution

Review and land the linked cursor-decoder PR with after-fix real API pagination proof for at least one non-trending sort, and track trending pagination separately if maintainers want that behavior.

## Reproduction Assessment

Yes. A live request against the current released site reproduced byte-identical first and second pages, and current main source shows the API emits full helper index keys while the decoder rejects longer keys on the next non-trending request.

## Solution Assessment

Yes for the active direction. The narrow maintainable fix is to accept full self-describing `getPage()` keys for the matching index while preserving wrong-index and legacy raw-array guards.
