## Summary

Keep open under the ClawHub issue policy even though the central non-trending `/api/v1/skills` cursor replay bug is fixed on current `main`, shipped to production, and covered by regression tests.

## What This Changes

The issue reports that `GET /api/v1/skills` returns the same first page and `nextCursor` when the advertised cursor is replayed across public API sort modes.

## Best Possible Solution

Keep the merged non-trending cursor fix as the canonical implementation and track any desired `sort=trending` cursor support as a separate public API enhancement.

## Reproduction Assessment

No, not for the documented non-trending cursor contract on current main or production. Source inspection, regression coverage, and a live read-only check all show replayed `nextCursor` values now advance for supported sorts.

## Solution Assessment

Yes for the non-trending bug: accepting the full self-describing `getPage()` cursor while preserving index and prefix guards is the narrow maintainable fix. Trending cursor support is outside the current documented contract.
