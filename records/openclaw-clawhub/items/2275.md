## Summary

Keep open: this appears to be a real current API pagination bug for the documented non-trending `/api/v1/skills` sorts. Production repeats the same page when its own cursor is reused, and current source explains the failure by rejecting emitted cursor keys as too long and restarting at the first page.

## What This Changes

The issue reports that `GET /api/v1/skills` ignores the returned cursor, causing paginated requests to repeat the first page.

## Best Possible Solution

Fix the non-trending V1 skills API cursor path so every `nextCursor` emitted by `listPublicApiPageV1` is accepted on the next request, while leaving `trending` as documented unless maintainers choose to add that capability separately.

## Reproduction Assessment

Yes for production and source-level reproduction: the live endpoint repeats the first page when its own cursor is reused, and current source has the same decode-and-fallback path. I did not run a local current-main server because this review is read-only.

## Solution Assessment

Yes: the narrow maintainable fix is to make the cursor decoder and tests match the full `getPage` index-key shape, preserving stale or wrong-index cursor recovery. Adding `trending` pagination should be handled separately if desired.

## Repair Work Prompt

Fix `GET /api/v1/skills` pagination for documented non-trending sorts. The endpoint currently emits `nextCursor` values from `getPage`, but `decodePublicListCursor` rejects those full index keys as too long and `listPublicApiPageV1` falls back to the first page. Keep the public response shape unchanged (`items`, `nextCursor`) and do not change unrelated package/plugin pagination. Update `convex/skills.ts` so cursors emitted by `listPublicApiPageV1` are accepted for `updated`, `createdAt`/`newest`, `downloads`, `installsAllTime`/`installs`, `stars`, and `name`; preserve stale/wrong-index cursor recovery. Add regression tests using realistic `getPage` index keys with tie-breaker fields so reusing `nextCursor` sets `startInclusive: false` and starts from the decoded key. Do not add `trending` pagination unless maintainers explicitly scope that product change.
