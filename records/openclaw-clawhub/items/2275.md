## Summary

Keep open: live production and current main still reproduce the non-trending cursor reset, and the linked closing PR remains open and conflicting, so this should stay open until a fix merges and deploys.

## What This Changes

GET /api/v1/skills cursor pagination repeats the first page because the API emits full getPage index-key cursors that current main rejects when replayed.

## Best Possible Solution

Land or replace the linked cursor-decoder fix so matching self-describing getPage cursor keys work for non-trending API sorts while stale wrong-index and legacy cursor guards remain intact.

## Reproduction Assessment

Yes. A live 2026-05-18 clawhub.ai round trip for sort=updated&limit=10 reproduced byte-identical first and second pages, and current main source shows the endpoint emits full getPage keys then rejects them on replay.

## Solution Assessment

Yes for the active direction. The narrow maintainable fix is to accept full self-describing cursors for the matching index while preserving stale, wrong-index, and legacy raw-array guards; trending pagination should stay separate.
