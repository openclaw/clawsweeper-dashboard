## Summary

This should stay open because current main still matches the reported pagination failure and an open linked PR already targets the fix with closing syntax. The source path and live API check both show the advertised cursor being rejected back to the first page for supported non-trending sorts; `trending` is separately documented as not cursor-paginated.

## What This Changes

The issue reports that `GET /api/v1/skills` ignores the returned `nextCursor` and repeats the first page across sort modes.

## Best Possible Solution

Review and land the linked fix PR, or apply an equivalent narrow fix that accepts the API's own self-describing `getPage()` cursors without weakening wrong-index or legacy cursor guards.

## Reproduction Assessment

Yes, by source inspection and live API confirmation. Current main emits `getPage()` index keys as `nextCursor`, the decoder rejects overlong emitted keys, and a read-only production check returned identical first and second `updated` pages when the advertised cursor was reused.

## Solution Assessment

Yes for the active fix direction. The linked PR narrows the change to self-describing cursors for the selected index while preserving stale/wrong-index and legacy raw-array guards, so the issue should remain open until that PR or an equivalent fix lands.
