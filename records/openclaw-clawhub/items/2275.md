## Summary

Keep open only because ClawHub cleanup policy does not auto-close issues; the documented non-trending `/api/v1/skills` cursor replay bug is fixed on current `main`, covered by regression tests, and deployed to production.

## What This Changes

The issue reports that replaying `nextCursor` from `GET /api/v1/skills` returns the same page instead of advancing pagination.

## Best Possible Solution

Keep the merged non-trending cursor decoder fix as the canonical implementation, and track any desired cursor support for `trending` as a separate API design item.

## Reproduction Assessment

No. On current `main` and production, replaying a non-`trending` `nextCursor` advances to a different page; the focused regression test also covers the same full-cursor replay path.

## Solution Assessment

Yes. Accepting full self-describing `getPage()` keys only when the index and equality prefix match is the narrow maintainable fix for the opaque cursor contract while keeping stale-cursor guards intact.
