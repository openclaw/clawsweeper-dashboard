## Summary

Keep open: current main still emits full `getPage()` index-key cursors from `GET /api/v1/skills` and then rejects those same full keys for non-trending sorts, while an open linked PR already targets the fix.

## What This Changes

The issue reports that `GET /api/v1/skills` does not advance when the advertised `nextCursor` is supplied back to the API.

## Best Possible Solution

Review and land the linked cursor-decoder PR after redacted real API pagination proof is added, preserving non-trending cursor behavior and tracking trending pagination separately if needed.

## Reproduction Assessment

Yes. The current-main source path is enough to reproduce the bug: the API emits full helper index keys as `nextCursor`, then the decoder rejects those full keys and restarts from the first page.

## Solution Assessment

Yes for the active direction. The narrow maintainable fix is to accept full self-describing `getPage()` keys for the matching selected index while preserving raw-array and wrong-index cursor guards.
