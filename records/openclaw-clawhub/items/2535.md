## Summary

This PR needs maintainer resolution before merge: the final head intentionally makes install telemetry activation-only, so removed installations can remain counted indefinitely, and it also makes compatibility-sensitive route and ranking changes while a linked PR proposes the opposite ranking policy.

## What This Changes

The PR replaces sync-snapshot install accounting with explicit install events, removes installs from search and recommended ranking, adds a high-volume install/download-ratio abuse signal, and updates the CLI, schemas, docs, specs, and tests.

## Best Possible Solution

Choose one maintainer-approved definition for current installs and discovery ranking, retain explicit install events for immediacy, add bounded indexed deactivation or reconciliation for removed installs, and stage legacy-route and cursor compatibility through the deployment transition.

## Reproduction Assessment

Yes at source level: report an install and then remove it locally; the final head has no uninstall event, snapshot reconciliation, or stale-root expiry path that deactivates the resulting active install row.

## Solution Assessment

No. Explicit install events improve prompt counting, but an activation-only event model cannot maintain a reliable current-install count after removal or missed telemetry, and the ranking direction still requires a canonical product decision.
