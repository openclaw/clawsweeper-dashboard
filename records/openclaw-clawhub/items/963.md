## Summary

Keep open. Current main still lacks the PR's digest-based scan and lazy version-file fetch, and the PR branch has confirmed owner correctness issues in the GitHub backup write path that should be revised before merge.

## What This Changes

This PR changes GitHub backup sync to paginate `skillSearchDigest`, use denormalized owner/version metadata during scanning, fetch version files lazily before backup writes, remove the missing-owner stat, update backup pagination tests, and add GitHub ref fast-forward retry handling.

## Best Possible Solution

Land a revised digest-based backup scan that keeps the scan phase lightweight, preserves owner-id fallback for handle-less owners, validates live owner eligibility or guarantees skill digest refresh on owner-state changes, resolves the current backup owner path, fetches version files only after a version mismatch, and keeps the fast-forward retry behavior already present on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve owner fallback for handle-less users:** `convex/githubBackups.ts:75`
  - body: The digest writer stores `ownerHandle: ''` for visible owners with no handle, while current main backs those skills up under `owner._id`. This guard skips those rows entirely, so their public skills stop being mirrored. Resolve a fallback owner path instead of treating empty string as ineligible.
  - confidence: 0.94
- **[P1] Revalidate owner state before trusting digest rows:** `convex/githubBackups.ts:72-75`
  - body: The previous page builder read live users and excluded deleted or deactivated owners. This replacement only checks denormalized digest fields, while user handle/deleted/deactivated changes do not refresh skill digests by owner user id, so stale rows can be backed up and then pruned repeatedly. Recheck live owner state or refresh skill digests on owner-state changes before enqueueing backups.
  - confidence: 0.88
- **[P2] Resolve the current owner path before backup lookup:** `convex/githubBackups.ts:84`
  - body: Using `digest.ownerHandle` directly can target an old owner path after handle changes for stale or legacy skill digests. Prune compares against the live owner path, so the backup can churn or disappear until the digest refreshes. Resolve the current owner handle or fallback before GitHub lookup/write calls.
  - confidence: 0.8
