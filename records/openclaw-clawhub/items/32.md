## Summary

Keep open. Current main intentionally blocks deleted/deactivated GitHub sign-in and now explains the failure, but this issue and the related open reports still need a maintainer account-lifecycle decision: irreversible deletion with clearer support guidance, or a guarded recovery/GitHub-binding reset path.

## What This Changes

Investigate login failure after deleting a ClawHub account and attempting to re-authenticate with the same GitHub identity.

## Best Possible Solution

Maintainers should decide and document the deleted-account lifecycle, then either keep irreversible deletion with clearer support UX or implement an audited same-provider recovery/reset path with ban safeguards and regression coverage.

## Reproduction Assessment

Yes, by source inspection: settings deletion reaches users.deleteAccount, which writes deactivatedAt/purgedAt, and GitHub OAuth completion calls handleDeletedUserSignIn, which rejects that identity. I did not run a live production OAuth flow or inspect the reporter's account row.

## Solution Assessment

No for an automated one-off restore or GitHub unlink. The maintainable solution needs a maintainer-approved policy for irreversible deletion versus verified recovery, then a guarded implementation and tests.
