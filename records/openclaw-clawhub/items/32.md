## Summary

Keep open. Current main intentionally blocks deleted/deactivated GitHub sign-in and now surfaces deleted-account messaging, but the remaining request is account recovery or GitHub binding reset, which needs maintainer product/security handling rather than cleanup closure.

## What This Changes

Users who deleted a ClawHub account want to sign in or re-register again with the same GitHub account instead of returning to a logged-out state.

## Best Possible Solution

Maintainers should decide whether deletion remains irreversible; if recovery is allowed, build an audited same-provider recovery or GitHub binding reset path with ban/deletion safeguards, docs, UI messaging, and regression coverage.

## Reproduction Assessment

Yes, for current-main behavior by source inspection: settings deletion reaches users.deleteAccount, which marks the user deactivated/purged, and the OAuth callback rejects deactivated or legacy deleted users. I did not run a live production OAuth flow or inspect the reporter's account row.

## Solution Assessment

No for an automated restore or one-off binding deletion. The maintainable path needs a product/security decision: either keep deletion non-restorable with clear support guidance, or implement a guarded recovery/unlink flow.
