## Summary

Keep this open. Current main intentionally treats user-initiated deletion as irreversible and now surfaces deleted-account sign-in errors, but this ClawHub issue has repeated affected-user reports and the remaining choice is account recovery versus permanent deletion, which needs maintainer product/security judgment.

## What This Changes

Users who self-delete their ClawHub account want to sign back in with the same GitHub account instead of being returned to a signed-out state.

## Best Possible Solution

Maintainers should decide whether self-deletion is intentionally non-restorable; if yes, keep the current policy and improve support/UX messaging, and if no, implement scoped same-provider recovery with regression coverage for deleted, deactivated, banned, autobanned, legacy, and identity-mismatch cases.

## Reproduction Assessment

Yes from source inspection and tests, not from an end-to-end OAuth run. Current main writes deactivatedAt/purgedAt on deletion and handleDeletedUserSignIn rejects deactivated or legacy deleted users, with regression tests covering those paths.

## Solution Assessment

No for an automated fix. Current main is internally consistent with the irreversible-deletion docs, but changing the outcome to account restoration is a product/security policy decision rather than a narrow bug fix.
