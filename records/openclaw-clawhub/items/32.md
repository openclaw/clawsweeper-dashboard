## Label Justifications

- `P1`: Multiple real users are locked out of GitHub sign-in after account deletion and the issue is ready for maintainer handling.
- `impact:security`: Changing deleted-account recovery affects identity relinking, bans, token revocation, audit trails, and purged data semantics.
- `impact:auth-provider`: The symptom occurs during GitHub OAuth completion and depends on ClawHub's provider identity matching for deleted accounts.

## Summary

Keep open. Current main intentionally treats user-initiated account deletion as irreversible, but the issue discussion has multiple affected users asking for recovery or re-registration, so this needs maintainer product/security handling rather than cleanup closure.

## What This Changes

Users who deleted their ClawHub account want to sign back in or reset the GitHub binding so they can re-register.

## Best Possible Solution

Maintainers should choose the account-deletion lifecycle explicitly, then either keep irreversible deletion with clearer OAuth/support guidance or implement a same-provider recovery/recreate flow with ban, purge, and audit safeguards.

## Reproduction Assessment

Yes, at source level: deleteAccount marks users deactivated/purged, and handleDeletedUserSignIn rejects deleted or deactivated users during OAuth. I did not run a live production OAuth flow.

## Solution Assessment

Unclear until maintainers choose the deletion lifecycle policy. Current main is internally consistent with irreversible deletion, but the requested recovery behavior would need a security-reviewed product path.

## Security Review

Status: needs_attention

Summary: Account recovery after deletion is security-sensitive because it changes identity, ban, purge, token, and audit semantics.
