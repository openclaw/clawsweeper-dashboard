## Summary

Keep this open: current main now intentionally treats user-initiated deletion as irreversible and surfaces blocked-account sign-in errors, but the issue discussion and prior merged restoration attempts show that account recovery after self-deletion still needs maintainer product/security triage. ClawHub cleanup policy also keeps issues open and only cleanup-closes PRs that are certainly implemented on main.

## What This Changes

The issue requests that users who soft-delete their ClawHub account can authenticate again with the same GitHub account instead of returning to the homepage signed out.

## Best Possible Solution

Maintainers should either retain irreversible deletion with clear support/sign-in guidance, or implement a scoped same-provider recovery/recreation flow with regression coverage for self-deleted, deactivated, banned/autobanned, legacy deletedAt, fresh-login, and identity-mismatch cases.

## Reproduction Assessment

Yes for the current source path: after deleteAccount writes deactivatedAt/purgedAt, the OAuth afterUserCreatedOrUpdated callback rejects that user with the permanent deleted-account message. I did not run an end-to-end GitHub OAuth flow in this read-only review.

## Solution Assessment

Unclear. Current main is maintainable if irreversible deletion is intended, but restoring or recreating accounts after deletion is not the best path without explicit product and security approval.
