## Summary

Keep open. Current main intentionally blocks deleted/deactivated GitHub OAuth sign-in and now shows a deleted-account error, but the report and follow-up comments describe account recovery expectations; for openclaw/clawhub this issue should stay with maintainers for product/security handling rather than cleanup closure.

## What This Changes

The issue reports that users who soft-deleted their ClawHub account are redirected back to Sign in after OAuth and cannot re-authenticate with the same GitHub provider.

## Best Possible Solution

Maintainers should decide whether self-deletion remains non-restorable; if not, implement an audited same-provider recovery or GitHub identity unlink flow with explicit ban/deletion safeguards, docs, UI messaging, and regression coverage.

## Reproduction Assessment

Yes, by source inspection: settings calls users.deleteAccount, that mutation sets deactivatedAt/purgedAt, and the OAuth callback rejects deactivated users with tests covering the path. I did not run a live GitHub OAuth flow or inspect the reporter's production account record.

## Solution Assessment

No for an automated restore or one-off binding deletion. The maintainable solution needs a policy-backed, audited recovery or unlink path, or an explicit decision to keep deletion irreversible with clearer support guidance.
