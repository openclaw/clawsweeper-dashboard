## Summary

Keep open. Current main intentionally blocks deleted/deactivated users during OAuth and now documents that deletion is irreversible, while the remaining requested outcome is account recovery, re-registration, or GitHub-binding reset, which needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue asks for deleted users to be able to sign in or re-register with the same GitHub OAuth account after account deletion.

## Best Possible Solution

Maintainers should choose and document the deleted-account lifecycle, then either keep deletion terminal with clear OAuth/support messaging or implement an audited same-provider recovery, re-registration, or operator reset path with ban and privacy safeguards.

## Reproduction Assessment

Yes, at the source level: deleteAccount sets deactivatedAt/purgedAt and clears profile data, and afterUserCreatedOrUpdated calls handleDeletedUserSignIn, which rejects deactivated or legacy deleted users. I did not run a live production OAuth flow or inspect affected account rows.

## Solution Assessment

No for an automated one-off restore, purge, or unlink. The maintainable path requires an explicit product/security decision about irreversible deletion versus same-provider recovery before code or operator changes alter identity and privacy boundaries.
