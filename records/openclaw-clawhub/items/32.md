## Summary

Keep open. Current main treats self-deleted accounts as terminal during OAuth and surfaces that policy to users, while the report and related comments ask for recovery, re-registration, or a GitHub binding reset after deletion; that remaining choice is an auth/security product decision for maintainers, not conservative cleanup.

## What This Changes

Users who deleted their ClawHub account cannot sign in again with the same GitHub account and are asking for re-authentication, restoration, or a GitHub binding reset.

## Best Possible Solution

Maintainers should choose and document the deleted-account lifecycle, then either keep deletion terminal with clear OAuth/support guidance or implement an audited same-provider recovery, re-registration, or operator reset path with ban and privacy safeguards.

## Reproduction Assessment

Yes, at the source level: deletion writes deactivatedAt/purgedAt and the OAuth callback rejects deactivated or legacy deleted users. I did not run a live production OAuth flow or inspect affected account rows.

## Solution Assessment

No for an automated restore, purge, or unlink. The maintainable solution needs an explicit account-lifecycle policy that covers same-provider identity matching, bans, privacy purging, and audit history before code or operator action changes deleted-account semantics.

## Security Review

Status: needs_attention

Summary: Deleted-account recovery or GitHub-binding reset is security-sensitive because it changes identity, ban, deletion, and privacy boundaries.

Concerns:

- **[medium] Verify provider identity before recovery:** `convex/auth.ts:40`
