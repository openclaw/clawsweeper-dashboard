## Summary

Keep open: current main intentionally blocks deleted/deactivated account sign-in and surfaces a deleted-account error, but the report asks for same-account re-authentication after self-deletion and has multiple affected-user confirmations. ClawHub cleanup policy keeps issues open, and changing this behavior needs maintainer product/security judgment.

## What This Changes

Users report that after deleting a ClawHub account they cannot re-authenticate with the same GitHub account and are returned to the sign-in state.

## Best Possible Solution

Maintainers should decide whether deletion is intentionally non-restorable; then either keep irreversible deletion with clearer UX/support guidance or implement a scoped same-GitHub-identity restoration or recreation flow with regression coverage.

## Reproduction Assessment

Yes. A code-level reproduction is clear: after deleteAccount writes deactivatedAt/purgedAt, OAuth completion calls handleDeletedUserSignIn and throws the deleted-account error; convex/auth.test.ts covers both deactivated and legacy deletedAt cases.

## Solution Assessment

Unclear. Current main is internally consistent for irreversible deletion and improves error surfacing, but it does not satisfy the requested same-account re-authentication and should not be changed without a maintainer security/product decision.
