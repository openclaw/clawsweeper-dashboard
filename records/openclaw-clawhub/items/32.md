## Summary

Keep open. Current main intentionally rejects GitHub re-authentication for deleted/deactivated ClawHub accounts and now surfaces a deleted-account error, but this issue is the oldest account-recovery report in an active cluster and ClawHub cleanup policy keeps issues open for maintainer product/security handling rather than auto-closing them.

## What This Changes

The issue reports that users who soft-deleted their ClawHub account cannot sign back in with the same GitHub OAuth account and are returned to the homepage sign-in state.

## Best Possible Solution

Maintainers should choose and document the deleted-account lifecycle, then either keep deletion irreversible with clearer post-OAuth/support guidance or implement an audited same-provider recovery/unlink/re-registration flow with ban, autoban, legacy-deletion, and mismatched-identity safeguards.

## Reproduction Assessment

Yes, by source inspection: settings deletion calls `users.deleteAccount`, the mutation records `deactivatedAt`/`purgedAt`, the OAuth callback rejects that account on reauth, and active-user helpers treat the linked user as signed out. I did not run a live GitHub OAuth flow or inspect the reporter’s production row.

## Solution Assessment

No for an automated restore, hard delete, or one-off binding deletion. The maintainable path needs a maintainer-approved account lifecycle and identity-verification policy before code chooses recovery, unlinking, re-registration, or support-only behavior.
