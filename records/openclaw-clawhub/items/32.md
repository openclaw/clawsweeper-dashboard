## Summary

Keep open. Current main intentionally blocks deleted/deactivated users during OAuth and now surfaces a deleted-account message, but the issue and related reports ask whether self-deleted accounts should be recoverable, re-registerable, or resettable; that remaining choice is product and security policy, not conservative cleanup closure.

## What This Changes

Users want GitHub OAuth sign-in, re-registration, restoration, or reset to work after user-initiated ClawHub account deletion.

## Best Possible Solution

Maintainers should choose and document the deleted-account lifecycle, then either keep terminal deletion with clear OAuth/support messaging or implement a scoped same-provider recovery/re-registration path with ban, purge, audit, and identity safeguards.

## Reproduction Assessment

Yes, at source level: deleteAccount writes deactivatedAt/purgedAt, OAuth calls handleDeletedUserSignIn, and tests assert deleted users are rejected. I did not run a live production OAuth flow or inspect individual account rows.

## Solution Assessment

Unclear until maintainers choose the deletion lifecycle policy. A one-off restore or hard-delete from a public report is not the best source fix because recovery needs identity verification, audit, privacy, and ban safeguards.
