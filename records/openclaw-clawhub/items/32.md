## Summary

Keep this open. Current main intentionally blocks deleted/deactivated users during OAuth and now surfaces a deleted-account error, but the issue and related reports still need a maintainer product/security decision about irreversible deletion versus scoped account recovery.

## What This Changes

A user who self-deleted their ClawHub account expects to sign in or re-register again with the same GitHub account instead of returning to a signed-out state.

## Best Possible Solution

Maintainers should choose the account-deletion policy explicitly: keep irreversible deletion with clearer OAuth/settings/support messaging, or implement a scoped same-provider reactivation or re-registration flow with regression coverage for self-deleted, deactivated, banned/autobanned, legacy deletedAt, fresh-login existingUserId null, and mismatched existingUserId cases.

## Reproduction Assessment

Yes, at source level. Current main reproduces the core lockout path because deleteAccount writes deactivatedAt/purgedAt and the OAuth callback rejects deleted/deactivated users; I did not perform a live GitHub OAuth run.

## Solution Assessment

Unclear until maintainers decide the policy. The current permanent-deletion implementation is internally consistent, but restoring access would require an explicit account-recovery design rather than a narrow automated bug fix.
