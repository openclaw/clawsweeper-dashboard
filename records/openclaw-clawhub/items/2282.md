## Summary

Keep open: the direct-transfer guard is directionally useful, but the patch now documents an all-transfer moderation invariant while the recipient-accepted transfer path can still move a moderated skill, and the contributor has not provided real behavior proof beyond tests/CI.

## What This Changes

Adds a moderation-state guard to direct skill owner transfers, regression tests, a changelog entry, and an org ownership spec note.

## Best Possible Solution

Apply one shared moderation-transfer eligibility rule to every skill ownership transfer write path, after authorization and before ownership patches, with specs and tests matching that scope.

## Reproduction Assessment

Yes. Source inspection on current main shows the direct transfer path reaches the ownership patch without a moderation check, and the existing accepted-transfer path still patches ownership without such a guard.

## Solution Assessment

No. Guarding the direct path is useful, but the current patch should either cover request/accept transfer paths too or narrow the new spec language to direct transfers only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Cover accepted transfers before claiming every path is blocked:** `specs/orgs.md:167-168`
  - body: The new spec says no ownership transfer path should move a hidden, removed, suspicious, or malicious skill, but the recipient-accepted flow in `convex/skillTransfers.ts` still patches `ownerUserId`/`ownerPublisherId` on accept without this guard. A pending user-to-user transfer can therefore still move a moderated skill; please either apply the guard to request/accept as well or narrow this spec text to direct transfers.
