## Summary

Keep open: current main still has the desktop Files tab height cap, and this PR's head is not ready because it introduces an unreachable-file case on mid-width viewports.

## What This Changes

The PR removes the desktop Files tab height cap, adds mobile truncation with a See all affordance, polishes file viewer styling, updates a focused test, and adds a changelog entry.

## Best Possible Solution

Align the component and CSS mobile breakpoints so desktop stays uncapped and every truncated viewport has a visible, tested expansion control.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: view this PR at 481px-899px with more than eight files, where the component renders only the first eight but the CSS keeps the See all overlay hidden.

## Solution Assessment

No. The intended direction is reasonable, but the implementation should first make the JS mobile cutoff and CSS media query agree and add a mid-width regression check.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Match the hidden-file CSS to the mobile cutoff:** `src/styles.css:8599-8605`
  - body: The component treats every viewport up to 899px as mobile and slices `visibleFiles` to eight, but these CSS rules only run under `@media (max-width: 480px)`. At 481px-899px, files after the eighth are not rendered and the See all button stays `display: none`, so users cannot reach the rest of the list. Move the rules to the same breakpoint or lower the JS cutoff consistently, and cover a mid-width viewport.
  - confidence: 0.92
