## Summary

Keep this PR open because current main still has the desktop Files tab height cap and the PR head still has a blocking breakpoint mismatch that can hide files at tablet widths.

## What This Changes

The PR removes the desktop Files tab height cap, adds a mobile preview/See all affordance, refines file viewer styling, adds a component test, and updates the changelog.

## Best Possible Solution

Align the component and CSS breakpoints so desktop remains uncapped and every truncated mobile/tablet viewport has a visible, tested expansion control.

## Reproduction Assessment

Yes. Source inspection of PR head gives a high-confidence path: at any viewport from 481px through 899px with more than eight files, JS renders only eight rows while the CSS that displays the See all control remains scoped to `max-width: 480px`.

## Solution Assessment

No. The intended UI direction is reasonable, but the implementation should first make the JS mobile cutoff and CSS media query agree and add a mid-width regression check.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Match the hidden-file CSS to the mobile cutoff:** `src/styles.css:8599-8605`
  - body: The component treats every viewport up to 899px as mobile and slices `visibleFiles` to eight, but the CSS that shows `.file-list-see-all-wrap` only lives in the `@media (max-width: 480px)` block. On widths from 481px through 899px, files after the eighth are not rendered and the See all button remains `display: none`, so users cannot reach the rest of the list. Move these rules to the same 899px breakpoint or lower the JS cutoff consistently, and cover a mid-width viewport.
  - confidence: 0.92
