## Summary

Keep open: the PR adds a useful moderation email path, but it still needs real Resend behavior proof and a security fix because unrecognized manual ban reasons can fall through into outbound email payloads.

## What This Changes

The branch adds Resend-backed ban and account-restoration emails to Convex moderation flows, updates tests, docs/specs, env examples, and adds the `resend` dependency.

## Best Possible Solution

Ship ban/restoration emails only after outgoing ban copy is derived from allowlisted public reasons or a dedicated public-notice field, with regression tests and redacted runtime proof for Resend dispatch.

## Reproduction Assessment

Yes. By source inspection, a manual ban with an unrecognized free-form reason reaches the formatter fallback and is serialized into the Resend text/html payload; I did not execute runtime email sending in this read-only review.

## Solution Assessment

No. Ban/restoration notices are a reasonable feature, but the safest implementation needs public-safe reason mapping or a separate public notice field before moderator-entered text leaves ClawHub.
