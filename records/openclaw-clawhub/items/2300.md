## Summary

Keep open: this is a useful moderation-notification feature, but the latest head still lets unrecognized manual ban reasons flow into outbound email copy and lacks real Resend behavior proof.

## What This Changes

The PR adds Resend-backed ban and restoration notification emails to Convex moderation flows, documents the required email env setup, and adds unit coverage for manual bans, malware autobans, unbans, and autoban remediation.

## Best Possible Solution

Ship ban/restoration emails only after outgoing copy is derived from allowlisted public reasons or a dedicated public-notice field, with regression tests and redacted runtime proof for Resend dispatch.

## Reproduction Assessment

Yes. By source inspection, a manual ban with an unrecognized free-form reason reaches the formatter fallback and is serialized into the Resend text/html payload; I did not execute runtime email sending in this read-only review.

## Solution Assessment

No, not as currently proposed. The notification feature is directionally useful, but the maintainable path is to separate internal ban reasons from public email copy and prove the real Resend dispatch path before merge.
