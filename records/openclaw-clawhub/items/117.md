## Summary

Keep this open. Current main has owner soft-delete/restore through backend, HTTP API, CLI, and docs, but the signed-in web app still lacks a delete/restore workflow and the owner dashboard filters soft-deleted skills before they could be restored.

## What This Changes

Skill authors should be able to delete their own published ClawHub skills.

## Best Possible Solution

Add a signed-in owner web delete/restore workflow that reuses existing soft-delete semantics, confirms destructive actions, and lets owners find and restore their soft-deleted skills.

## Reproduction Assessment

Yes. On current main, the settings panel only renders rename/merge controls, dashboard row actions only render settings/rescan, and api.skills.list filters softDeletedAt rows for owner scopes.

## Solution Assessment

Yes. The narrow maintainable solution is to expose the existing soft-delete primitive through authenticated web mutations and owner UI/query support; hard-delete or client-supplied identity would be the wrong direction.
