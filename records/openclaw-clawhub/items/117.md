## Summary

Keep this issue open. Current main has backend, HTTP API, CLI, docs, and tests for owner soft-delete/restore, but the signed-in web app still lacks a discoverable owner delete/restore workflow and owner dashboard lists filter out soft-deleted skills before they can be restored.

## What This Changes

Skill authors should be able to delete their own published skills without maintainer intervention.

## Best Possible Solution

Add authenticated web delete/restore controls that reuse soft-delete semantics, require confirmation, derive identity server-side, and let owners find deleted skills for restoration.

## Reproduction Assessment

Yes. On current main, source inspection shows owner settings renders rename/merge only, dashboard row actions render settings/rescan only, and api.skills.list filters softDeletedAt rows before owner dashboard rendering.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse the existing soft-delete/restore primitive through authenticated web mutations and owner-visible query/UI support; hard-delete for normal owners would be the wrong direction.
