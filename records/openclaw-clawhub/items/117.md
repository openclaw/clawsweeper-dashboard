## Summary

Keep this open. Current main has owner soft-delete/restore plumbing in backend, HTTP API, CLI, and docs, but the signed-in web app still lacks a discoverable owner delete/restore workflow and filters deleted skills out of the dashboard path needed for restoration.

## What This Changes

The issue requests self-service deletion for a skill author's own published ClawHub skills.

## Best Possible Solution

Add a signed-in web delete/restore flow that reuses existing soft-delete semantics, confirms destructive actions, and lets owners find and restore deleted skills without adding hard-delete behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, by current-main source inspection: owner settings render rename/merge only, dashboard row actions render settings/rescan only, and dashboard queries exclude soft-deleted skills.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable fix is to reuse the existing soft-delete primitive behind authenticated web mutations and add owner-visible delete/restore UI; hard delete or client-supplied user identity would be the wrong direction.
