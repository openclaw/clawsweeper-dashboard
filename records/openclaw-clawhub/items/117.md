## Summary

Keep open. Current main has owner soft-delete/restore support in backend, HTTP API, CLI, docs, and tests, but the signed-in web owner surfaces still expose only rename/merge/settings actions and hide soft-deleted rows before an owner can restore them; ClawHub policy also keeps issues open rather than auto-closing them.

## What This Changes

The issue asks for skill authors to be able to delete their own published skills from ClawHub.

## Best Possible Solution

Add a signed-in web owner delete/restore workflow that reuses existing soft-delete semantics, derives identity server-side, authorizes direct owners and eligible publisher members consistently, confirms destructive actions, and lets owners find and restore owner-hidden skills.

## Reproduction Assessment

Yes. Source inspection on current main shows owner settings render rename/merge only, dashboard row actions omit skill deletion, and owner skill queries filter out soft-deleted rows before any restore UI could show them.

## Solution Assessment

Yes. The narrow maintainable solution is a web owner delete/restore flow over the existing soft-delete primitive; hard delete, client-supplied actor identity, or bypassing moderation provenance would be the wrong boundary.
