## Summary

Keep this issue open. Current main has owner soft-delete/restore primitives in the backend, HTTP API, CLI, docs, and tests, but the signed-in web app still lacks owner-facing skill delete/restore controls and the dashboard filters soft-deleted skills out before owners could restore them.

## What This Changes

Allow skill authors to delete their own published skills.

## Best Possible Solution

Add a signed-in web delete/restore workflow that reuses existing soft-delete semantics, derives identity server-side, supports direct owners and publisher members consistently, and avoids normal-owner hard delete.

## Reproduction Assessment

Yes. Source inspection is enough: owner settings has rename/merge only, dashboard skill rows lack a delete action, and the owner dashboard query filters out soft-deleted rows before rendering.

## Solution Assessment

Yes. The narrow maintainable path is to add owner-facing web controls around the existing soft-delete/undelete helper and tests, not to change CLI/API semantics or introduce hard delete for normal owners.
