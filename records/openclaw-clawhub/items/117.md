## Summary

Keep this issue open. Current main has owner soft-delete/restore through backend, HTTP API, CLI, docs, and tests, but the signed-in web app still lacks a discoverable owner delete/restore flow and the dashboard filters soft-deleted skills before an owner can restore them.

## What This Changes

The issue requests a way for skill authors to delete their own published skills.

## Best Possible Solution

Add a signed-in web owner delete/restore workflow that reuses existing soft-delete semantics, exposes an owner-visible deleted/held view, and keeps moderation/security hides non-restorable by normal owners.

## Reproduction Assessment

Yes, as a source-level gap: owner settings renders summary, publisher note, rename, and merge only, while dashboard owner queries use active indexes that exclude soft-deleted rows. I did not run a live browser flow.

## Solution Assessment

Yes, if maintainers want web self-service; reusing setSkillSoftDeletedInternal and adding an owner-visible restore surface is safer than adding hard-delete behavior or a second deletion contract.
