## Summary

Keep open. Current main supports owner soft-delete/restore through CLI and API, but the signed-in web app still exposes only rename/merge/settings-style skill owner controls and filters soft-deleted skills out of the dashboard, so the self-service website workflow remains unfinished.

## What This Changes

The issue asks for skill authors to be able to delete their own published skills.

## Best Possible Solution

Add a signed-in web soft-delete and restore workflow that reuses current soft-delete semantics, derives identity server-side, preserves moderation/security gates, and gives owners a way to find and restore eligible deleted skills.

## Reproduction Assessment

Yes, from source inspection rather than a live run. A signed-in owner reaches owner tools and dashboard row menus without a skill delete/restore action, and deleted skills are filtered out of the dashboard query before restore UI can appear.

## Solution Assessment

Yes. The narrow maintainable path is to reuse the existing soft-delete/restore model in web UI and tests, not add owner hard-delete or weaken moderation restore gates.
