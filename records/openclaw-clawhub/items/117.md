## Summary

Keep this issue open. Current main supports owner soft-delete/restore through backend, HTTP API, CLI, docs, and tests, but the signed-in web app still lacks a discoverable owner delete/restore workflow, and live comments plus related open removal requests show the user need is still active.

## What This Changes

The issue asks for skill authors to be able to delete their own published skills.

## Best Possible Solution

Add a signed-in web delete/restore workflow that reuses existing soft-delete semantics, derives identity server-side, exposes owner-deleted skills for restore, and keeps moderation/security hides non-restorable by owners.

## Reproduction Assessment

Yes, as a source-level gap: owner settings renders summary, publisher note, rename, and merge only, while dashboard owner queries use active indexes that exclude soft-deleted rows. I did not run a live browser flow.

## Solution Assessment

Unclear until maintainers decide the web self-service policy. If approved, reusing setSkillSoftDeletedInternal behind a narrow authenticated web mutation is the maintainable path; hard-delete or a second deletion contract would be riskier.
