## Summary

Keep this issue open. Current main has owner soft-delete/restore through backend, HTTP API, CLI, and docs, but the signed-in web app still lacks a discoverable owner skill delete/restore workflow; a prior closed PR proposed that exact web flow but was not merged.

## What This Changes

The issue asks for skill authors to be able to delete their own published skills.

## Best Possible Solution

Add a signed-in web delete/restore workflow that reuses existing soft-delete semantics, derives identity server-side, supports direct owners and publisher members consistently, and lets owners find owner-deleted skills for restore without exposing hard delete.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction of the web gap: owner settings renders only rename/merge, dashboard row actions delete plugins only, and dashboard skill queries exclude soft-deleted rows before owners could restore them. I did not run a browser session because this was a read-only cleanup review.

## Solution Assessment

Yes. The narrowest maintainable path is to add web controls over the existing soft-delete/restore API and keep hard-delete and moderation override behavior in staff/admin paths.
