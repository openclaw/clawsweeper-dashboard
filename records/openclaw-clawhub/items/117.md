## Summary

Keep this open. Current main has owner soft-delete/restore through the backend, HTTP API, CLI, and docs, but the signed-in web app still lacks a visible owner delete/restore workflow and dashboard queries only page active skills.

## What This Changes

Allow skill authors to delete or restore their own published skills without maintainer intervention.

## Best Possible Solution

Add a signed-in owner web delete/restore workflow that reuses existing soft-delete semantics, confirms destructive actions, and lets owners find and restore soft-deleted skills.

## Reproduction Assessment

Yes. Source inspection on current main shows owner settings expose rename/merge only, dashboard row actions expose settings/rescan only, and dashboard pagination filters soft-deleted rows before owners can restore them.

## Solution Assessment

Yes. The narrow maintainable path is to reuse the existing soft-delete primitive through authenticated web mutations and owner-visible UI/query support; hard-delete or client-supplied identity would be the wrong direction.
