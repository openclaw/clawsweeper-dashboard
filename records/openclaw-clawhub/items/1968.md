## Summary

Keep open. Current source and the maintainer production recheck point to a moderation-state follow-up rather than a stale search-index bug: global search intentionally hides suspicious skills, while the skills browse page can show them unless the user enables the non-suspicious filter.

## What This Changes

The issue asks staff to investigate why the published `skillpick` and `threeperson` skills are absent from `/search` after the reporter believed moderation had cleared.

## Best Possible Solution

Keep the global-search suspicious filter in place and resolve the named skills through staff moderation review or an explicit override; improve owner-facing diagnostics only if the API still presents suspicious records as cleared.

## Reproduction Assessment

Yes for the observed visibility split: current source shows `/search` forces `nonSuspiciousOnly: true`, `/skills` defaults that filter off, and the maintainer production recheck says both named skills were still suspicious after fresh rescans. I did not establish a failing current-main source bug because the observed behavior matches the configured moderation boundary.

## Solution Assessment

No for a search-index refresh or source filter change as the primary fix. The narrow maintainable path is staff moderation review or override of the production records while preserving the public-search safety filter.
