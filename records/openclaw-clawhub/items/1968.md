## Summary

Keep open. Current production and current main both support the reported split: the named skills are searchable when suspicious filtering is not applied, but global web search applies `nonSuspiciousOnly: true`, and production still reports both skills as `suspicious.llm_suspicious`; ClawHub cleanup policy also does not auto-close open issues.

## What This Changes

The issue asks why the published `skillpick` and `threeperson` skills are visible through CLI/detail flows but absent from global web search after the reporter believed moderation had cleared.

## Best Possible Solution

Keep the issue open for a moderator or maintainer to either clear the production suspicious moderation state for the named skills or make an explicit product decision about whether global web search should include active suspicious skills.

## Reproduction Assessment

Yes. Current production reproduces the filtering path: unfiltered search returns both slugs, `nonSuspiciousOnly=true` hides them, and both detail/moderation endpoints report `isSuspicious: true`.

## Solution Assessment

Unclear. If these are false positives, a moderation override or rescan is the narrowest fix; changing global search to include suspicious skills would need product/security judgment.
