## Summary

Keep open. Current main and production show a real visibility split: unified web search applies the suspicious-skill filter, while the skills browse page does not by default, and production still reports both named skills as suspicious rather than cleared.

## What This Changes

The issue asks maintainers to investigate why the published `skillpick` and `threeperson` skills appear in CLI/API detail surfaces but not in global web search after claimed moderation clearance.

## Best Possible Solution

Moderation staff should first confirm whether the two skills should be cleared, then maintainers should make global search and `/skills` browse intentionally consistent with documented tests and UI copy.

## Reproduction Assessment

Yes. Production public API calls reproduce the core split: normal search returns the named slugs, but the same search with `nonSuspiciousOnly=true` returns no results, and current main shows global web search always applies that filter.

## Solution Assessment

Unclear. Refreshing the search index alone is not the best fix if the records remain suspicious; removing the filter from unified search would align the UI surfaces but requires a deliberate security/moderation policy choice.
