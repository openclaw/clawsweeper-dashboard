## Summary

Keep open. Current main and production still reproduce the reported split: global `/search` forces the non-suspicious skills filter, `/skills` exposes that filter as optional, and the named production records still report suspicious moderation rather than cleared moderation.

## What This Changes

The issue requests that `skillpick` and `threeperson` become visible in global web search or that maintainers explain which moderation/search flag is still hiding them.

## Best Possible Solution

Maintainers should decide whether these two skills should be manually cleared, remain suspicious, or whether global search should expose the same suspicious-content control and documentation as the skills directory.

## Reproduction Assessment

Yes. Compare `/api/v1/search?q=<slug>` with the same request plus `nonSuspiciousOnly=true` for `skillpick` and `threeperson`, then inspect `/api/v1/skills/<slug>`; current production shows the unfiltered results, filtered empties, and `moderation.isSuspicious: true`.

## Solution Assessment

Unclear. A blind code change to remove global filtering would change a security/moderation policy, so the safer path is maintainer moderation review first and only then a focused UX/docs/test change if policy says global search should behave like `/skills`.
