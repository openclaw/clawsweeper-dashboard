## Summary

Keep open. The behavior is reproducible against current production and current main explains the split: global `/search` forces the suspicious-content filter, while `/skills` only applies it when the optional filter is enabled; the named skills currently still report suspicious moderation in the public API.

## What This Changes

The issue reports that published skills visible through direct pages, CLI search, and `/skills` search are missing from global `/search` results after the reporter expected moderation to be cleared.

## Best Possible Solution

Maintainers should decide whether these skills should remain suspicious, be manually cleared, or whether global `/search` should expose the same visible filter behavior as `/skills` with matching docs/tests.

## Reproduction Assessment

Yes. Current production reproduces the split by comparing `/api/v1/search?q=<slug>` with the same request plus `nonSuspiciousOnly=true`, and the global web search path hard-codes that filtered mode.

## Solution Assessment

Unclear. The current implementation is internally consistent with hiding suspicious skills from global search, but deciding whether that is the right user-facing behavior requires moderation/product judgment.
