## Summary

Keep open. Current main explains the web/CLI visibility split through intentional suspicious-skill filtering, and the maintainer production recheck says the two skills are still flagged, so this is a moderation follow-up rather than an implemented or stale search-index bug.

## What This Changes

The reporter asks why `skillpick` and `threeperson` are visible through CLI/direct skill surfaces but absent from unified web search after they believed moderation was cleared.

## Best Possible Solution

Keep the global-search suspicious filter in place, complete staff moderation review or an explicit override for the production skills, and improve diagnostics only if owner/API views still imply they are cleared.

## Reproduction Assessment

Yes for the visibility split from source: unified web search sets `nonSuspiciousOnly: true`, while `/skills` defaults that filter off and CLI search does not send it. No for the cleared-moderation premise, which depends on production records and was contradicted by the maintainer rescan.

## Solution Assessment

No for a search-index refresh or source change that bypasses the filter. The narrow maintainable path is staff moderation review or override while preserving the suspicious-skill public-search boundary.
