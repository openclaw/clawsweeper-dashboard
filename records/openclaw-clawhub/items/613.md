## Summary

Keep open. Current main still lacks device-flow authentication, the ClawHub issue cleanup policy keeps issues open, and open PR #1867 is already paired with this request but has not merged.

## What This Changes

Add GitHub-style device-flow authentication for headless `clawhub login` so a CLI can display a code, poll for approval, and store an issued token without a localhost redirect.

## Best Possible Solution

Keep this issue open and land or replace PR #1867 with an end-to-end device-flow implementation covering CLI flags, server code issuance, polling/token exchange, verification UI, secure token creation, rate limiting, and protocol-state tests.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: `clawhub login --no-browser` without `--token` fails, there is no `--device` flag, and the documented headless path is manual token paste.

## Solution Assessment

Yes in direction, but only if implemented end to end. A client-only device flow is not sufficient without server-side issuance, verification, polling semantics, and security controls.
