## Summary

Keeping open: the security-labeled report is source-reproducible against the hosted ClawHub skill artifact, and ClawHub issue policy requires maintainer handling rather than cleanup closure.

## What This Changes

Report that the published `context-restore` skill’s `--confirm` flow auto-approves when stdin is unavailable.

## Best Possible Solution

Moderators should triage the hosted skill report, require the skill owner to fail closed when `--confirm` cannot read user input, and consider a scanner rule if this confirmation-bypass pattern should be caught at publish time.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the hosted script’s `--confirm` branch catches `EOFError`/`OSError` from `input()` and assigns `confirmation = 'y'`. I did not execute the artifact, but the direct branch matches the reported noninteractive stdin case.

## Solution Assessment

Unclear for repository automation: the narrow content fix is to make the skill fail closed or require an explicit noninteractive approval flag, while any ClawHub scanner/moderation change needs maintainer security judgment.
