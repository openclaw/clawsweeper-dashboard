## Label Justifications

- `P1`: Supported legacy clients can reject valid package installs because the advertised integrity metadata describes different bytes from the legacy download.
- `impact:security`: The issue concerns the package-integrity trust boundary between advertised hashes and executable plugin archive bytes.

## Summary

Keep open: current main still exposes stored package `sha256hash` separately from the ZIP digest returned by legacy package `/download`, and the only related canonical item is still open with no merged PR implementing the compatibility fix.

## What This Changes

The issue asks ClawHub to keep legacy package version `sha256hash` consistent with the bytes returned by legacy `/download` for npm-pack package versions.

## Best Possible Solution

Define one exact-bytes contract: artifact metadata identifies npm-pack tarballs, while legacy metadata identifies the exact ZIP returned by legacy `/download`, with tests for both client paths.

## Reproduction Assessment

Yes at source level: current main exposes stored `sha256hash` separately from the ZIP digest computed by legacy `/download`. A live production check also reproduces the same artifact/hash split for the reported package, though I did not run an old OpenClaw install locally.

## Solution Assessment

No final implementation is selected yet. The exact-bytes invariant is the right direction, but maintainers need to choose ZIP-digest mapping or a proven old-client fallback before repair automation changes the compatibility contract.

## Risks / Open Questions

- Mapping `sha256hash` to generated ZIP bytes likely requires computing the ZIP digest during version response or persisting separate legacy download metadata without weakening npm-pack artifact and scanner metadata.
- Omitting `sha256hash` is safe only if OpenClaw 2026.4.x clients are proven to fall back to secure per-file ZIP verification.
- The broader related report covers the same artifact split, so fixes should be coordinated to avoid two inconsistent integrity contracts.
