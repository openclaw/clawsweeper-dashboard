## Summary

Keep open: current main still exposes legacy `sha256hash` separately from the ZIP digest returned by legacy package `/download` for npm-pack releases. The related report at https://github.com/openclaw/clawhub/issues/2104 corroborates the same artifact split, but it is still open and does not supersede this narrower old-client compatibility contract.

## What This Changes

The issue asks ClawHub to keep the legacy version `sha256hash` field consistent with the bytes returned by the legacy `/download` endpoint for npm-pack package versions.

## Best Possible Solution

Define one exact-bytes contract: artifact fields continue identifying the uploaded npm tarball, while legacy metadata either identifies the exact ZIP returned by `/download` or is omitted only after old-client fallback verification is proven and documented.

## Reproduction Assessment

Yes. Source inspection shows the version response exposes stored `sha256hash` while legacy package `/download` generates a ZIP with a separate digest, and a live production check reproduced the two different hashes for the reported package.

## Solution Assessment

The requested invariant is the right direction, but the proposed alternatives are not interchangeable. Maintainers should choose the ZIP-digest mapping or a proven old-client fallback contract, then cover both legacy and artifact-aware install paths.
