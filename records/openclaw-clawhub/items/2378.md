## Label Justifications

- `P1`: A legacy install-time integrity mismatch can block real OpenClaw plugin installs for versions that advertise compatibility with older clients.
- `impact:security`: The issue concerns public package artifact digest metadata and install integrity verification boundaries.

## Summary

Keep this open: current main still leaves the reported legacy integrity split in place. The npm-pack scan/artifact path uses the uploaded tarball digest, while the legacy package `/download` route still generates ZIP bytes with a separate ZIP digest, so the requested legacy `sha256hash` invariant is not implemented.

## What This Changes

The issue asks ClawHub to keep legacy version `sha256hash` metadata consistent with the bytes returned by the legacy package `/download` endpoint for npm-pack-backed package versions.

## Best Possible Solution

Keep the artifact-aware npm-pack resolver intact and add a maintainer-approved legacy integrity contract so old clients never compare `/download` ZIP bytes against tarball metadata.

## Reproduction Assessment

Yes. Source inspection shows npm-pack scanning records the tarball digest into `sha256hash`, while the legacy package `/download` route returns generated ZIP bytes with its own response digest.

## Solution Assessment

Unclear. The requested invariant is valid, but changing or splitting a public integrity field needs maintainer/security direction before a repair chooses ZIP-specific, nullable, or explicit legacy metadata semantics.
