## Label Justifications

- `P1`: The deterministic hash mismatch blocks plugin installation on older hosts that package metadata declares supported.
- `impact:security`: Legacy clients are given an integrity digest for an artifact different from the bytes they download and verify.

## Summary

Current main still serializes the stored release `sha256hash` while the legacy download handler independently builds a deterministic ZIP, and no merged PR addresses the old-client mismatch. This issue should remain open because it captures a concrete compatibility invariant not yet implemented, even though the broader related issue tracks the same artifact split.

## What This Changes

Keep legacy `sha256hash` aligned with the ZIP bytes returned by `/api/v1/packages/{name}/download` for npm-pack-backed package versions.

## Best Possible Solution

Implement one canonical package API fix that preserves npm tarball integrity metadata, supplies a ZIP-specific legacy hash or a proven-safe null fallback, and adds regression coverage for both artifact-aware clients and OpenClaw 2026.4.x legacy installs.

## Reproduction Assessment

Yes, at high confidence from source and the supplied production digests: current main returns the stored release hash while independently generating ZIP bytes for the legacy download route. A full old-client process was not run, but the comparison path and failure are concrete.

## Solution Assessment

The requested invariant is correct, but the suggested alternatives are not interchangeable: maintainers should choose between computing a legacy ZIP digest and omitting the field only after confirming the old-client fallback contract.
