## Summary

Keep open: current main still exposes npm-pack version `sha256hash` as the tarball digest while the legacy package `/download` route returns ZIP bytes with a different digest. Live ClawHub production also still reproduces the reported split, so this needs maintainer/security direction rather than cleanup closure.

## What This Changes

The issue requests that legacy package version `sha256hash` stay consistent with the ZIP bytes served by `/api/v1/packages/{name}/download` for npm-pack package releases.

## Best Possible Solution

Define one legacy integrity contract: old clients should either receive a ZIP digest that matches `/download` bytes or no misleading `sha256hash`, while artifact-aware clients keep exact npm-pack tarball verification.

## Reproduction Assessment

Yes. Source inspection on current main shows the version metadata and legacy download route still describe different artifacts, and live production on 2026-06-10 confirms the reported tarball-vs-ZIP SHA-256 mismatch.

## Solution Assessment

Unclear. The requested invariant is the right target, but maintainers need to choose whether to remap legacy `sha256hash`, null it for old clients, add explicit legacy download metadata, or use another compatibility-preserving contract.
