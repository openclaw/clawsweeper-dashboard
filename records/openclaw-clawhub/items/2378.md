## Label Justifications

- `P1`: A compatible plugin install path fails for older OpenClaw clients because executable package integrity metadata disagrees with the downloaded bytes.
- `impact:security`: The issue concerns public executable plugin artifact digest fields disagreeing across ClawHub resolver paths.

## Summary

Keep this open: current main and live production still expose the npm-pack tarball digest through legacy version metadata while legacy `/download` serves generated ZIP bytes, so older OpenClaw clients can still hit the reported integrity mismatch.

## What This Changes

The issue asks ClawHub to keep the legacy `sha256hash` field internally aligned with the bytes returned by the legacy package `/download` endpoint for npm-pack-backed versions.

## Best Possible Solution

Define and document a stable compatibility contract: legacy version responses consumed by older OpenClaw clients should expose a digest matching the legacy ZIP or omit the legacy field, while artifact-aware fields continue to carry npm-pack tarball integrity and scan provenance, with regression coverage for both paths.

## Reproduction Assessment

Yes. Live production reproduces the mismatch for `@openviking/openclaw-plugin@2026.5.21`, and current main source shows the same split between stored `sha256hash` and ZIP bytes from legacy `/download`; I did not run a local Convex fixture.

## Solution Assessment

Unclear. The requested invariant is correct, but the permanent fix needs maintainer/security approval because `sha256hash` is also used for scan and provenance state on npm-pack artifacts.
