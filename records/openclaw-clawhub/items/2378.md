## Label Justifications

- `P1`: The issue breaks a real older OpenClaw plugin install path for a package that declares compatibility and fails at integrity verification.
- `impact:security`: The report is about executable package artifact integrity metadata disagreeing across public install and download paths.

## Summary

Keep this open: the report is valid and security-sensitive, and ClawHub policy keeps issues open rather than auto-closing them. Current source plus the live current-main deployment show npm-pack version metadata exposes the tarball hash in `sha256hash` while legacy `/download` still serves a generated ZIP with a different digest, so older OpenClaw clients can fail integrity checks.

## What This Changes

Align the legacy `sha256hash` contract with the bytes served by legacy package `/download` for npm-pack-backed package versions.

## Best Possible Solution

Define a stable compatibility contract where legacy responses either expose a digest matching the legacy ZIP or omit the legacy digest for old clients, while artifact-aware fields keep the npm-pack digest and tests cover both download paths plus existing-row repair.

## Reproduction Assessment

Yes. The live current deployment and current source show the same npm-pack release exposes a tarball digest in version metadata while legacy `/download` returns ZIP bytes with a different digest; I did not run an older OpenClaw client locally.

## Solution Assessment

Unclear. The requested invariant is the right direction, but the permanent fix needs maintainer/security approval because `sha256hash` also feeds scan and provenance surfaces; separating legacy ZIP integrity from npm-pack artifact integrity is the safer path.

## Risks / Open Questions

- Changing `sha256hash` directly could move VirusTotal, security UI, or provenance consumers from the scanned npm-pack tarball hash to a generated ZIP hash.
- Existing npm-pack release rows may need a backfill, cached legacy ZIP digest, or deliberate on-demand compatibility behavior; a forward-only publish fix would leave old compatible releases broken.
- The broader open artifact-integrity issue at https://github.com/openclaw/clawhub/issues/2104 overlaps this area, so maintainers should choose one public resolver/integrity contract before implementation.
