## Label Justifications

- `P1`: The issue breaks a real plugin install path for older compatible OpenClaw clients with an integrity mismatch on executable package downloads.
- `impact:security`: The report is about artifact integrity metadata disagreeing across public executable-plugin download paths.

## Summary

Keep this open: current source and the live example show npm-pack `version.sha256hash` can identify the uploaded `.tgz` while the legacy `/download` endpoint returns a generated ZIP with a different digest, so older OpenClaw clients can fail a valid install. This overlaps with https://github.com/openclaw/clawhub/issues/2104, but this report is narrower around the legacy `sha256hash` contract and still needs maintainer/security judgment.

## What This Changes

Ensure legacy package version metadata stays internally consistent with the legacy ZIP returned by `/api/v1/packages/{name}/download` for npm-pack releases.

## Best Possible Solution

Define one compatibility contract: legacy version responses consumed by OpenClaw 2026.4.x should expose a digest that matches the legacy ZIP or omit that legacy field, while artifact-aware fields continue to carry the npm-pack digest and scan provenance, with regression coverage for both download paths.

## Reproduction Assessment

Yes for the live example and source path: the reported package currently returns a tarball hash in version metadata while legacy `/download` returns ZIP bytes with another hash. I did not run a local Convex fixture against current main, so this is source-reproducible rather than a local current-main test reproduction.

## Solution Assessment

Unclear: the requested invariant is correct, but the permanent fix needs maintainer/security approval because `sha256hash` also represents scan/provenance state for npm-pack artifacts. A safer direction is to explicitly separate legacy ZIP integrity from npm-pack artifact integrity and test both contracts.

## Risks / Open Questions

- Changing `sha256hash` without a clear compatibility contract could move VirusTotal/security UI and provenance consumers from the scanned npm-pack tarball hash to a generated ZIP hash.
- Existing npm-pack rows may need a backfill, cached legacy ZIP digest, or deliberately on-demand computation; otherwise old releases can remain broken even after new publishes are fixed.
- The related open artifact-integrity issue at https://github.com/openclaw/clawhub/issues/2104 already flags this area as public API and security-sensitive, so the fix should be reviewed as an integrity contract change.
