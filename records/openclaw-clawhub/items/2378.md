## Label Justifications

- `P1`: Supported older clients can reject valid package installs because legacy integrity metadata describes different bytes from the legacy download.
- `impact:security`: The issue concerns the package integrity trust boundary between advertised hashes and executable plugin archive bytes.

## Summary

Keep open: current main still exposes `sha256hash` from stored release metadata while legacy `/download` returns a generated ZIP with its own digest. The related report corroborates the same artifact split, but it is still open and current main has not implemented the legacy compatibility contract.

## What This Changes

The issue asks ClawHub to keep the legacy `sha256hash` field consistent with the exact bytes returned by legacy package `/download` for npm-pack releases.

## Best Possible Solution

Define one exact-bytes integrity contract: artifact fields identify the uploaded npm tarball, while legacy metadata identifies the generated ZIP returned by `/download` or uses a documented, proven old-client fallback.

## Reproduction Assessment

Yes. Source inspection shows the version response exposes stored `sha256hash` while legacy `/download` builds and hashes a ZIP, and a live endpoint check reproduced different `.tgz` and ZIP hashes for the reported package.

## Solution Assessment

Unclear until maintainers choose the backward-compatible legacy contract. The invariant is correct, but remapping versus omitting `sha256hash` needs old-client security validation.
