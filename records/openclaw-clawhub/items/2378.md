## Label Justifications

- `P1`: Older supported clients can reject valid package installs because advertised integrity metadata describes different bytes from the legacy download.
- `impact:security`: The issue concerns the package-integrity trust boundary between advertised hashes and executable plugin archive bytes.

## Summary

Keep open: current main still exposes package version `sha256hash` separately from the ZIP digest returned by legacy package `/download` for npm-pack releases, and the open related report corroborates the same artifact split without superseding this narrower old-client compatibility contract.

## What This Changes

The issue asks ClawHub to keep legacy package version `sha256hash` consistent with the ZIP bytes returned by legacy `/download` for npm-pack releases while preserving npm tarball metadata for artifact-aware clients.

## Best Possible Solution

Define one exact-bytes contract: artifact fields identify the uploaded npm tarball, while legacy metadata either identifies the exact ZIP returned by `/download` or is omitted only after old-client fallback verification is proven and documented.

## Reproduction Assessment

Yes, at source level. Current main exposes stored `sha256hash` from the version response while legacy `/download` generates ZIP bytes with a separate digest, and live production reproduces the reported hash split for the affected package.

## Solution Assessment

Yes, the requested exact-bytes invariant is the right direction, but the alternatives are not interchangeable. Maintainers should choose ZIP-digest mapping or a proven old-client fallback contract, then cover both legacy and artifact-aware install paths.
