## Label Justifications

- `P1`: The digest mismatch blocks real legacy OpenClaw plugin installs for package versions that advertise compatibility with older clients.
- `impact:security`: The issue concerns public package digest metadata used to verify executable plugin download bytes.

## Summary

Keep open: current main still exposes package version `sha256hash` separately from the explicit artifact metadata while the legacy package `/download` route builds ZIP bytes with its own digest. Live production still reproduces the reported tarball-vs-ZIP mismatch, so the remaining work needs maintainer/security direction before implementation.

## What This Changes

The issue requests keeping legacy package `sha256hash` consistent with the ZIP bytes served by `/api/v1/packages/{name}/download` for npm-pack-backed versions.

## Best Possible Solution

Define the legacy integrity contract so old clients either compare ZIP bytes to a ZIP digest or do not receive a mismatched `sha256hash`, while artifact-aware clients keep exact npm-pack tarball verification.

## Reproduction Assessment

Yes. Source inspection on current main shows version metadata and legacy download routes still describe different artifacts, and live production on June 10, 2026 reproduces the exact tarball-vs-ZIP SHA-256 mismatch for the reported package version.

## Solution Assessment

Unclear. The requested invariant is the right target, but the permanent fix needs maintainer/security choice between remapping legacy `sha256hash`, nulling it for old clients, adding explicit legacy download metadata, or another compatibility-preserving API contract.
