## Label Justifications

- `P1`: The current mismatch blocks an established plugin installation workflow for older OpenClaw clients even when the package advertises compatibility with them.
- `impact:security`: The issue concerns disagreement between cryptographic integrity metadata and the executable package bytes actually downloaded.

## Summary

Keep open: current main still gives npm-pack releases an artifact digest identity while the legacy download path independently generates ZIP bytes with a different digest, so older compatible OpenClaw clients can still reject valid packages.

## What This Changes

Ensure legacy `sha256hash` metadata matches the ZIP bytes returned by the legacy package download endpoint for npm-pack-backed versions.

## Best Possible Solution

Represent the npm tarball and generated legacy ZIP as distinct integrity identities, expose a ZIP-aligned digest to legacy clients, preserve existing npm-pack and scanner hashes, and validate both old-client and artifact-aware install paths against real Convex endpoints.

## Reproduction Assessment

Yes. Current main clearly uses the npm-pack artifact hash in metadata while the legacy endpoint independently constructs a ZIP and computes a different digest, establishing the reported failure path from source.

## Solution Assessment

The requested consistency invariant is correct, but globally redefining `sha256hash` is not necessarily the safest implementation; a separate legacy ZIP digest or narrowly scoped legacy serializer avoids conflating modern artifact and scanner identities.
