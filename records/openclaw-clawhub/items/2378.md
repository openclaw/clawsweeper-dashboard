## Label Justifications

- `P1`: A compatible legacy OpenClaw install path fails because public package integrity metadata disagrees with the bytes served by legacy download.
- `impact:security`: The issue concerns digest metadata for executable plugin artifacts disagreeing across public install/download paths.

## Summary

Keep this open: current production still shows the reported npm-pack tarball digest versus legacy ZIP digest split, and current main still emits stored `sha256hash` separately from the ZIP bytes generated by legacy `/download`. The related issue remains broader; this one is the narrower legacy digest invariant.

## What This Changes

Request is to keep legacy package version `sha256hash` internally consistent with legacy `/download` for npm-pack releases while preserving artifact-aware tarball metadata.

## Best Possible Solution

Define and document a stable compatibility contract where legacy responses either expose a digest matching legacy ZIP bytes or omit that legacy field, while artifact-aware fields continue to carry npm-pack tarball integrity and scan provenance.

## Reproduction Assessment

Yes. The live affected package still returns the npm-pack digest in `version.sha256hash` while legacy `/download` returns different ZIP bytes, and current main source shows the same route split.

## Solution Assessment

Unclear. The requested invariant is correct, but the permanent implementation needs maintainer/security choice because `sha256hash` is also used for scan and provenance state on npm-pack artifacts.
