## Label Justifications

- `P1`: The mismatch blocks real legacy OpenClaw plugin installs for packages that advertise compatibility with older clients.
- `impact:security`: The issue concerns digest metadata used to verify executable plugin download bytes served by ClawHub.

## Summary

Keep open: current main and live production still expose a tarball `sha256hash` for npm-pack versions while legacy `/download` returns ZIP bytes with a different digest, so the legacy install integrity mismatch remains unresolved.

## What This Changes

The issue asks ClawHub to keep legacy package version `sha256hash` consistent with the bytes returned by legacy `/api/v1/packages/{name}/download` for npm-pack-backed releases.

## Best Possible Solution

Define a single legacy integrity contract: old clients should either receive a ZIP digest that matches legacy `/download` bytes or no legacy digest to compare, while artifact-aware clients keep exact npm-pack tarball verification.

## Reproduction Assessment

Yes. Current source shows the version metadata and legacy download route still describe different artifacts, and live production on June 10, 2026 reproduces the exact tarball-versus-ZIP SHA-256 split for the reported package version.

## Solution Assessment

Unclear. The requested invariant is the right end state, but the durable fix needs maintainer/security choice between remapping legacy `sha256hash`, omitting it for old clients, adding explicit legacy download metadata, or another compatibility-preserving contract.
