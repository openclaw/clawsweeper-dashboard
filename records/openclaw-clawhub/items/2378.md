## Summary

Keep open: current main and live production still expose a tarball `sha256hash` for the reported npm-pack version while legacy `/download` returns ZIP bytes with a different digest, so the legacy install integrity mismatch remains unresolved.

## What This Changes

The issue asks ClawHub to keep legacy package version `sha256hash` internally consistent with legacy `/download` bytes for npm-pack package versions.

## Best Possible Solution

Define a single legacy integrity contract: older clients should receive a ZIP digest matching legacy `/download` or no legacy digest to compare, while artifact-aware clients keep npm-pack tarball verification.

## Reproduction Assessment

Yes. Current production and current source both show npm-pack metadata using the tarball SHA-256 while legacy `/download` returns generated ZIP bytes with a different SHA-256.

## Solution Assessment

Unclear. The requested invariant is the right end state, but maintainers need to choose the public compatibility contract before changing `sha256hash`, adding explicit legacy digest metadata, or backfilling existing rows.
