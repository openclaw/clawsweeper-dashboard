## Summary

Keep this open: current main and the live example still show the reported integrity-contract split, and ClawHub repository policy keeps issues open unless a PR-only close path applies. The broader related issue remains open, but this report is narrower around the legacy `sha256hash` versus legacy ZIP download invariant and should stay available for maintainer/security review.

## What This Changes

The issue asks ClawHub to keep legacy package `sha256hash` internally consistent with the bytes returned by the legacy `/download` endpoint for npm-pack-backed package versions.

## Best Possible Solution

Define and document a stable compatibility contract: legacy version responses consumed by older OpenClaw clients should expose a digest matching the legacy ZIP or omit that legacy field, while artifact-aware fields continue to carry npm-pack tarball integrity and scan provenance, with regression coverage for both paths.

## Reproduction Assessment

Yes. Live production reproduces the mismatch for `@openviking/openclaw-plugin@2026.5.21`, and current main source shows why: version metadata exposes the stored npm-pack hash while legacy `/download` builds a ZIP with a different digest.

## Solution Assessment

Unclear. The requested invariant is correct, but the permanent fix needs maintainer/security approval because the same `sha256hash` field is also used for scan and provenance state on npm-pack artifacts.

## Risks / Open Questions

- Changing `sha256hash` without a clear compatibility contract could move VirusTotal, security UI, or provenance consumers from the scanned npm-pack tarball hash to a generated ZIP hash.
- Existing npm-pack rows may need a backfill, cached legacy ZIP digest, or deliberate on-demand legacy digest computation; otherwise already published compatible versions can remain broken.
- The broader open issue at https://github.com/openclaw/clawhub/issues/2104 overlaps, but closing this one as a duplicate would lose the narrower legacy field invariant that older clients rely on.
