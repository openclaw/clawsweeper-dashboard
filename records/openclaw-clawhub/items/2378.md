## Label Justifications

- `P1`: A real compatible plugin install path fails for older OpenClaw clients because executable package integrity metadata disagrees with the downloaded bytes.
- `impact:security`: The issue concerns public executable plugin artifact digest fields disagreeing across ClawHub resolver paths.

## Summary

Keep this open: live production still reproduces the legacy integrity split, and current main still emits npm-pack scan/artifact hash metadata while legacy `/download` serves generated ZIP bytes. The broader related issue remains open, but this report is the narrower legacy `sha256hash` versus legacy ZIP invariant.

## What This Changes

The issue requests that legacy package version metadata keep `sha256hash` internally consistent with bytes returned by legacy `/download` for npm-pack-backed package versions.

## Best Possible Solution

Define and document a stable compatibility contract: legacy version responses consumed by older OpenClaw clients should expose a digest matching the legacy ZIP or omit the legacy field, while artifact-aware fields continue to carry npm-pack tarball integrity and scan provenance, with regression coverage for both paths.

## Reproduction Assessment

Yes. Live production reproduces the mismatch for `@openviking/openclaw-plugin@2026.5.21`, and current main source shows the same split: version JSON exposes stored `sha256hash` while legacy `/download` builds and hashes ZIP bytes.

## Solution Assessment

Unclear. The requested invariant is correct, but the permanent fix needs maintainer/security approval because `sha256hash` is also used for scan and provenance state on npm-pack artifacts.
