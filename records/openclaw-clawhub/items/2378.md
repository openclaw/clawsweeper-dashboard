- `impact:security`: The issue concerns cryptographic integrity metadata being compared against bytes from a different downloadable artifact.

## Summary

Keep open: current main and the latest v0.20.0 release still expose one stored `sha256hash` while serving npm tarball and generated legacy ZIP artifacts with different digests, so older clients can reject valid npm-pack releases. The related report at https://github.com/openclaw/clawhub/issues/2104 independently confirms the same artifact mismatch.

## What This Changes

Ensure the legacy `sha256hash` for npm-pack releases matches the generated ZIP returned by the legacy download endpoint while preserving separate npm tarball integrity metadata.

## Best Possible Solution

Define separate, durable integrity metadata for the stored npm tarball and generated legacy ZIP, keep legacy `sha256hash` aligned with `/download`, and cover both historical-row upgrades and fresh npm-pack publishing with real API and legacy-client install validation.

## Reproduction Assessment

Yes at source level: current main returns a stored `sha256hash` in version metadata but independently builds and hashes ZIP bytes for `/download`, matching two concrete reports of cross-artifact integrity failures. A live legacy-client run was not performed during this read-only review.

## Solution Assessment

The requested invariant is correct, but the exact repair needs maintainer agreement: explicit per-artifact metadata plus a migration is safer than silently replacing or removing a shared legacy field without proving old- and new-client upgrade behavior.
