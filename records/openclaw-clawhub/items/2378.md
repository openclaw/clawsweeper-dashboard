- `P1`: The issue breaks a real plugin install path for older compatible OpenClaw clients with an integrity mismatch on executable package downloads.
- `impact:security`: The report is about executable package artifact integrity metadata disagreeing across public install/download paths.

## Summary

Keep this open: current main and live production still show npm-pack version metadata exposing the tarball digest while the legacy package download route returns ZIP bytes with a different digest, so older compatible OpenClaw clients can fail valid installs. The overlap with https://github.com/openclaw/clawhub/issues/2104 is real, but this issue is the narrower legacy `sha256hash` contract and needs maintainer/security judgment.

## What This Changes

The issue asks ClawHub to keep legacy package-version `sha256hash` consistent with the bytes returned by the legacy `/download` endpoint for npm-pack-backed package versions.

## Best Possible Solution

Define one compatibility contract: legacy responses consumed by OpenClaw 2026.4.x should expose a digest that matches the legacy ZIP or omit that legacy field, while artifact-aware fields keep the npm-pack digest and scan provenance, with regression coverage for both download paths.

## Reproduction Assessment

Yes for the live example and source path: production currently returns a tarball hash in version metadata while legacy `/download` returns ZIP bytes with another hash. I did not run a local Convex fixture against current main, so this is source-reproducible rather than a local current-main reproduction.

## Solution Assessment

Unclear: the requested invariant is directionally correct, but the permanent fix needs maintainer/security approval because `sha256hash` also represents scan and provenance state for npm-pack artifacts. A safer direction is to explicitly separate legacy ZIP integrity from npm-pack artifact integrity and test both contracts.

## Risks / Open Questions

- Changing `sha256hash` without a clear contract could move VirusTotal/security UI and provenance consumers from the scanned npm-pack tarball digest to a generated ZIP digest.
- Existing npm-pack rows may need a backfill, cached legacy ZIP digest, or intentionally on-demand computation; otherwise already-published compatible releases can remain broken for old clients.
- The related open artifact-integrity report at https://github.com/openclaw/clawhub/issues/2104 shows this area is broader than one field, so the fix should be reviewed as a public integrity contract change.
