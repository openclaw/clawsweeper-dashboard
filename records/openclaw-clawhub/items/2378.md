## Label Justifications

- `P1`: A legacy install-time integrity mismatch can block real OpenClaw plugin installs for package versions that advertise compatibility with older clients.
- `impact:security`: The issue concerns public package digest metadata used to verify executable plugin download bytes.

## Summary

Keep open: current main still exposes the stored package `sha256hash` separately from the npm-pack artifact metadata while the legacy package download route rebuilds and serves ZIP bytes with a different digest. The related broader artifact mismatch report remains open, so this narrower legacy integrity contract still needs maintainer/security direction.

## What This Changes

The issue requests that legacy package `sha256hash` remain internally consistent with the bytes served by the legacy package `/download` endpoint for npm-pack releases.

## Best Possible Solution

Keep the issue open until maintainers define a legacy integrity contract, then update the package API so old clients never compare legacy ZIP bytes against npm-pack tarball metadata while artifact-aware clients retain exact tarball verification.

## Reproduction Assessment

Yes. Source inspection shows npm-pack metadata and the legacy package download route still describe different artifacts with different SHA-256 sources; I did not run a live Convex HTTP smoke in this read-only review.

## Solution Assessment

Unclear. The requested invariant is valid, but the exact fix should be a maintainer/security decision between ZIP-specific legacy metadata, nulling legacy `sha256hash`, or adding explicit legacy download metadata.
