## Summary

Keep this open: live endpoints still reproduce the digest split, and current main has the same source shape. Legacy version JSON exposes `sha256hash` from the npm-pack artifact/scan path while legacy `/download` builds and hashes ZIP bytes, so a maintainer/security decision is needed before changing this public executable-artifact integrity contract.

## What This Changes

The issue asks ClawHub to keep legacy package `sha256hash` consistent with the bytes returned by legacy `/download` for npm-pack-backed package versions.

## Best Possible Solution

Define and document the compatibility contract so legacy version responses either expose a digest matching legacy ZIP bytes or omit the legacy field, while artifact-aware fields keep npm-pack integrity and tests cover both resolver paths.

## Reproduction Assessment

Yes. Live production still shows the npm-pack artifact hash and legacy ZIP hash differ for the affected version, and current main source explains the mismatch through separate version metadata and ZIP download paths.

## Solution Assessment

Unclear. The invariant is correct, but the durable fix needs maintainer/security choice because `sha256hash` also carries scan/provenance state for npm-pack artifacts.

## Risks / Open Questions

- Changing `sha256hash` directly could move VirusTotal, security UI, or provenance consumers from the scanned npm-pack tarball hash to a generated ZIP hash without an approved compatibility contract.
- Existing npm-pack rows may need cached legacy ZIP digests, a backfill, or a deliberate omission strategy; a route-only change may not repair already published versions for older clients.
- The broader open issue at https://github.com/openclaw/clawhub/issues/2104 overlaps, but closing this one as a duplicate would lose the narrower legacy-field invariant that older OpenClaw clients rely on.
