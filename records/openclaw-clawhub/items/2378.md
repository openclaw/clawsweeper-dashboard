## Label Justifications

- `P1`: A digest mismatch blocks real legacy OpenClaw plugin installs for package versions that advertise compatibility with older clients.
- `impact:security`: The issue concerns public package digest metadata used to verify executable plugin download bytes.

## Summary

Keep open: current main and live production still expose the npm-pack tarball digest through version `sha256hash` while legacy package `/download` returns ZIP bytes. The broader related report remains open, but it does not replace this narrower legacy integrity contract.

## What This Changes

The issue asks ClawHub to keep legacy package version `sha256hash` internally consistent with the bytes returned by legacy `/download` for npm-pack-backed package versions.

## Best Possible Solution

Define and implement a legacy integrity contract so old clients either compare ZIP bytes to a ZIP digest or do not receive a mismatched `sha256hash`, while artifact-aware clients keep exact npm-pack tarball verification.

## Reproduction Assessment

Yes. Source inspection on current main shows the version metadata and legacy download route still describe different artifacts, and live production on 2026-06-10 reproduces the tarball-vs-ZIP SHA-256 mismatch for the reported package version.

## Solution Assessment

Unclear. The requested invariant is the right target, but the permanent fix needs maintainer/security choice between remapping legacy `sha256hash`, nulling it for old clients, adding explicit legacy download metadata, or another compatibility-preserving API contract.
