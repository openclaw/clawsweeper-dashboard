## Label Justifications

- `P1`: A legacy install-time integrity mismatch can block real OpenClaw plugin installs for versions that advertise compatibility with older clients.
- `impact:security`: The issue concerns public package artifact digest metadata used for executable plugin install integrity verification.

## Summary

Keep this open: current main still exposes npm-pack artifact metadata separately from the legacy ZIP bytes returned by `/download`, so the requested legacy `sha256hash` invariant is not implemented. The broader related artifact-integrity report at https://github.com/openclaw/clawhub/issues/2104 is also still open, so there is no merged canonical fix to close against.

## What This Changes

Ensure legacy package version `sha256hash` matches the bytes returned by legacy `/download` for npm-pack package versions so older OpenClaw clients do not fail integrity checks.

## Best Possible Solution

Keep the artifact-aware npm-pack resolver intact, then make legacy version metadata internally consistent with the ZIP bytes returned by `/download`, with focused HTTP handler coverage and public API docs for the chosen contract.

## Reproduction Assessment

Yes, by source inspection. Current main still returns stored `sha256hash` in version metadata while legacy `/download` returns generated ZIP bytes with a separate ZIP digest.

## Solution Assessment

Unclear. The requested invariant is valid, but the exact fix needs maintainer/security direction before changing a public integrity field to ZIP-specific, nullable, or explicit legacy metadata semantics.
