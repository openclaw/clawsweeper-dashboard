## Summary

Keep this open: source inspection and the live package example still show a real legacy integrity mismatch for npm-pack releases, and ClawHub issue policy does not auto-close issues even when the shape is clear.

## What This Changes

The issue asks ClawHub to keep legacy `version.sha256hash` consistent with the ZIP bytes returned by `/api/v1/packages/{name}/download` for npm-pack package versions so older OpenClaw clients do not fail integrity checks.

## Best Possible Solution

Define a compatibility contract where legacy responses consumed by OpenClaw 2026.4.x either expose a digest matching the legacy ZIP or omit the legacy field, while artifact-aware fields keep the npm-pack digest and regression tests cover both download paths.

## Reproduction Assessment

Yes for the live example and source path: production metadata exposes the npm-pack digest in `version.sha256hash` while legacy `/download` returns ZIP bytes with a different digest. I did not run a local Convex fixture against current main, so this is source-reproducible rather than a local current-main test reproduction.

## Solution Assessment

Unclear: the requested invariant is sound, but the best implementation needs maintainer/security approval because `sha256hash` also carries scan/provenance meaning for npm-pack artifacts. A safer direction is to separate legacy ZIP integrity from npm-pack artifact integrity and repair existing rows deliberately.
