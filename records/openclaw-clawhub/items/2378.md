## Label Justifications

- `P1`: Older supported clients can reject valid package installs because advertised integrity metadata describes different bytes from the legacy download.
- `impact:security`: The issue concerns the package-integrity trust boundary between advertised hashes and executable plugin archive bytes.

## Summary

Keep open: current main still exposes the stored npm-pack tarball digest as legacy `sha256hash` while the legacy `/download` route returns a generated ZIP with a different digest. The related report at https://github.com/openclaw/clawhub/issues/2104 corroborates the artifact split, but it remains open and does not supersede this old-client compatibility contract.

## What This Changes

The issue asks ClawHub to keep the legacy `sha256hash` field consistent with the bytes served by the legacy `/download` endpoint for npm-pack-backed package versions.

## Best Possible Solution

Define one backward-compatible exact-bytes contract: artifact fields continue identifying the uploaded npm tarball, while legacy metadata either identifies the exact ZIP returned by `/download` or is omitted only after old-client fallback verification is proven and documented.

## Reproduction Assessment

Yes. Current source shows the version response exposes `result.version.sha256hash`, npm-pack scanning stores the `.tgz` digest there, and legacy `/download` builds a ZIP with a separate digest; a live production check reproduced the two different hashes for the reported package.

## Solution Assessment

The requested invariant is correct, but the proposed alternatives are not interchangeable. Maintainers should choose the ZIP-digest mapping or a proven old-client fallback contract, then cover both legacy and artifact-aware install paths.
