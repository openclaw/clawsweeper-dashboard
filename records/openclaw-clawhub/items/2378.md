## Summary

Keep open: current main and live production still expose the npm-pack tarball digest through package version `sha256hash` while the legacy package `/download` path returns ZIP bytes with a different digest. The broader related report at https://github.com/openclaw/clawhub/issues/2104 remains open, but this narrower legacy integrity contract still needs maintainer/security direction.

## What This Changes

The issue asks ClawHub to keep legacy package `sha256hash` internally consistent with the bytes returned by legacy `/api/v1/packages/{name}/download` for npm-pack package versions.

## Best Possible Solution

Define and implement a legacy integrity contract so old clients either compare ZIP bytes to a ZIP digest or do not receive a mismatched `sha256hash`, while artifact-aware clients keep exact npm-pack tarball verification.

## Reproduction Assessment

Yes. Source inspection on current main shows the version metadata and legacy download route still describe different artifacts, and live production on June 10, 2026 reproduces the exact tarball-vs-ZIP SHA-256 mismatch for the reported package version.

## Solution Assessment

Unclear. The requested invariant is the right target, but the permanent fix needs maintainer/security choice between remapping legacy `sha256hash`, nulling it for old clients, adding explicit legacy download metadata, or another compatibility-preserving API contract.
