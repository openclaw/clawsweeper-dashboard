## Summary

Current main and v0.21.0 still expose the npm-pack tarball digest as legacy `sha256hash` while the legacy download endpoint independently generates a ZIP with a different digest. This issue remains necessary because the related broader report is also open and does not replace the specific old-client compatibility decision.

## What This Changes

Keep legacy package-version integrity metadata aligned with the generated ZIP served by `/download` while retaining separate npm-pack artifact hashes for artifact-aware clients.

## Best Possible Solution

Define and document separate legacy-ZIP and npm-pack integrity contracts, preserve strong verification for both client generations, and validate them through real Convex HTTP responses plus old- and current-client install smokes.

## Reproduction Assessment

Yes at source level: current main exposes the stored npm-pack digest in version metadata but calculates a different digest over generated ZIP bytes for `/download`. The same contract is present in v0.21.0, although this review did not execute a live old-client installation.

## Solution Assessment

Unclear. Correcting the inconsistent legacy contract is necessary, but maintainers must choose between digest remapping, explicit legacy metadata, or omission without weakening integrity checks or confusing newer consumers.

## Risks / Open Questions

- Changing `sha256hash` to mean the generated ZIP digest could surprise artifact-aware consumers that still read the legacy field.
- Omitting `sha256hash` may weaken whole-archive verification on older clients unless their per-file fallback is confirmed and tested.
- The generated ZIP digest is currently calculated only during download, so exposing it in version metadata may require durable storage, deterministic recomputation, or an explicitly versioned response contract.
