## Label Justifications

- `P1`: The mismatch blocks an established plugin installation workflow for older clients that package metadata still declares compatible.
- `impact:security`: A legacy integrity field currently identifies different bytes from those delivered by the corresponding download endpoint.

## Summary

Current main still exposes the release’s stored npm-pack hash as legacy `sha256hash` while `/download` remains a separately generated ZIP for old clients, so the reported integrity mismatch remains necessary to track. The broader open report covers the same artifact split, but this issue adds a distinct, documented compatibility invariant for OpenClaw 2026.4.x and ClawHub policy requires keeping unresolved items open.

## What This Changes

Keep the legacy `sha256hash` consistent with the ZIP bytes returned by the legacy package download endpoint for npm-pack versions.

## Best Possible Solution

Define and document one legacy contract where every integrity value consumed by old clients authenticates the exact deterministic ZIP returned by `/download`, while artifact-aware fields continue to authenticate the npm-pack tgz, then cover both routes with runtime and old-client install validation.

## Reproduction Assessment

Yes, at source level with high confidence: current main serializes the stored release hash while the documented legacy endpoint independently returns a generated ZIP, reproducing the two-artifact comparison described in the report. A live old-client install was not run during this read-only review.

## Solution Assessment

Unclear. The requested invariant is correct, but maintainers must choose whether to expose a computed deterministic-ZIP hash or omit the legacy field only after proving the old-client fallback verifies the same ZIP safely.
