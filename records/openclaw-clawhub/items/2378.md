## Label Justifications

- `P1`: Older supported clients reject valid packages because the advertised integrity digest covers different bytes from the legacy download.
- `impact:security`: The report concerns the package-integrity trust boundary between advertised digests and downloaded artifact bytes.

## Summary

Current main and v0.21.0 still serialize the stored npm-pack tarball digest as legacy `sha256hash` while the legacy download handler independently returns a deterministic ZIP with its own digest. The report therefore remains necessary; the broader related issue confirms the same artifact split but does not replace this issue’s specific old-client compatibility requirement.

## What This Changes

Ensure legacy `sha256hash` matches the bytes returned by the legacy package download endpoint for npm-pack versions.

## Best Possible Solution

Define and document a single backward-compatible contract: artifact-specific fields continue identifying the uploaded npm tarball, while legacy metadata either identifies the exact deterministic ZIP returned by `/download` or is omitted only after proving supported legacy clients securely verify its files.

## Reproduction Assessment

Yes, at high confidence from current source and the report’s exact production hashes: version metadata exposes the stored digest while the legacy endpoint constructs and hashes different ZIP bytes. An independent report also confirms the same artifact split.

## Solution Assessment

The requested invariant is correct, but the proposed alternatives are not interchangeable. Maintainers should select the ZIP-digest or proven file-verification fallback contract and coordinate it with https://github.com/openclaw/clawhub/issues/2104.
