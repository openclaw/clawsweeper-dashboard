## Summary

Keep this open: live production and current source still show npm-pack version metadata exposing the tarball SHA-256 through `sha256hash` while the legacy `/download` route returns ZIP bytes with a different digest, which can break older compatible OpenClaw installs. This overlaps with https://github.com/openclaw/clawhub/issues/2104, but this issue is the narrower legacy digest-contract report and still needs maintainer/security judgment.

## What This Changes

The issue asks ClawHub to keep legacy `sha256hash` metadata internally consistent with the legacy `/download` artifact for npm-pack package versions.

## Best Possible Solution

Define a compatibility contract where legacy 2026.4.x version responses either expose a digest matching the legacy ZIP or omit the legacy field, while artifact-aware fields keep the npm-pack digest and both paths get regression coverage.

## Reproduction Assessment

Yes. The live example reproduces the mismatched tarball and ZIP digests, and current main source explains the split; I did not run a local Convex fixture, so this is source-reproducible rather than a local current-main fixture reproduction.

## Solution Assessment

Unclear. The requested invariant is directionally correct, but the permanent fix needs maintainer/security approval because `sha256hash` is also used for npm-pack scan/provenance state.

## Risks / Open Questions

- Changing `sha256hash` without a clear compatibility contract could move VirusTotal/security UI and provenance consumers from the scanned npm-pack tarball hash to a generated ZIP hash.
- Existing npm-pack rows may need a backfill, cached legacy ZIP digest, or deliberately on-demand computation; otherwise already-published compatible releases can remain broken for older clients.
- The broader open report at https://github.com/openclaw/clawhub/issues/2104 covers adjacent artifact-route behavior, so maintainers should keep the two contracts aligned rather than fixing this field in isolation.
