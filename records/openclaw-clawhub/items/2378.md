## Label Justifications

- `P1`: Older supported clients can reject valid package installs because advertised integrity metadata describes different bytes from the legacy download path.
- `impact:security`: The issue concerns the package-integrity trust boundary between advertised hashes and executable plugin archive bytes.

## Summary

Keep open: current main still exposes stored release `sha256hash` while legacy package `/download` builds a separate ZIP for npm-pack releases. The related report at https://github.com/openclaw/clawhub/issues/2104 is also open, and there is no merged PR or current-main implementation that owns this compatibility fix.

## What This Changes

The issue asks ClawHub to keep legacy package `sha256hash` consistent with the ZIP bytes served by legacy `/api/v1/packages/{name}/download` for npm-pack releases.

## Best Possible Solution

Define one exact-bytes contract: artifact fields keep identifying the uploaded npm tarball, while legacy metadata identifies the exact ZIP returned by `/download` or is omitted only after old-client fallback verification is proven.

## Reproduction Assessment

Yes, at source level. Current main returns stored `sha256hash` in version metadata, streams npm-pack bytes on the artifact resolver path, and builds ZIP bytes with a separate digest on the legacy download path; I did not run a full current-main server smoke in this read-only pass.

## Solution Assessment

The requested exact-byte invariant is the right direction, but the two proposed compatibility options are not interchangeable. Maintainers should choose ZIP-digest mapping or a proven old-client fallback contract, then cover both legacy and artifact-aware install paths.
