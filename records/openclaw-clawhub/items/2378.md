## Label Justifications

- `P1`: Valid npm-pack releases remain un-installable through the legacy path used by older supported OpenClaw clients.
- `impact:security`: The issue concerns which downloaded package bytes are authenticated by the published integrity hash.

## Summary

Current main still returns a stored npm-pack hash in the legacy version response while the legacy package download route independently generates ZIP bytes, so the reported old-client integrity mismatch remains necessary to resolve. The broader https://github.com/openclaw/clawhub/issues/2104 describes the same artifact split but remains open and does not supersede this issue’s narrower backward-compatibility invariant.

## What This Changes

Keep legacy `sha256hash` aligned with the ZIP served by `/download` for npm-pack-backed versions while preserving tarball integrity metadata for artifact-aware clients.

## Best Possible Solution

Define separate canonical tarball and legacy ZIP integrity metadata, ensure every legacy hash authenticates exactly its legacy download bytes, document the invariant in `specs/`, and add real HTTP plus old-client install regression coverage.

## Reproduction Assessment

Yes, from current source with high confidence: the version response returns the stored `sha256hash`, while `/download` constructs separate deterministic ZIP bytes. The supplied real package hashes and the related report provide a concrete external example, although this read-only review did not execute the old OpenClaw client.

## Solution Assessment

The requested invariant is the right compatibility goal, but the exact implementation needs a maintainer decision: compute and expose the legacy ZIP hash without losing the canonical tarball hash, or omit the legacy hash only if old-client fallback behavior is verified safe.
