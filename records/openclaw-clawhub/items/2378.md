## Summary

Keep this open: the live package example still shows `version.sha256hash` matching the npm-pack tarball while legacy `/download` returns ZIP bytes with a different digest, and current main still has the same split between version metadata and legacy ZIP generation. This overlaps with https://github.com/openclaw/clawhub/issues/2104, but this issue is the narrower legacy digest contract and needs maintainer/security judgment.

## What This Changes

Requests that legacy package version metadata keep `sha256hash` consistent with the ZIP bytes returned by legacy `/download` for npm-pack-backed package versions.

## Best Possible Solution

Define a single compatibility contract that keeps artifact-aware fields tied to the npm-pack tarball while legacy OpenClaw 2026.4.x responses either expose a ZIP-matching legacy digest or omit the legacy field, with regression coverage and any required existing-row repair.

## Reproduction Assessment

Yes. The live example still reproduces the tarball-vs-ZIP digest mismatch, and current main source shows the version response and legacy ZIP download paths that produce it; I did not run a local Convex fixture, so this is source-reproducible rather than a local current-main fixture reproduction.

## Solution Assessment

Unclear. The requested invariant is correct, but the permanent fix needs maintainer/security choice because `sha256hash` also carries scan/provenance meaning for npm-pack artifacts; explicit legacy ZIP integrity metadata is the safer direction.

## Risks / Open Questions

- Changing `sha256hash` directly could move VirusTotal/security UI and provenance consumers from the scanned npm-pack tarball hash to a generated ZIP hash unless the contract is explicitly separated.
- Existing npm-pack rows may need cached legacy ZIP digest metadata, an on-demand compatibility path, or a backfill so already-published versions do not remain broken.
- The broader open report at https://github.com/openclaw/clawhub/issues/2104 covers related artifact URL/provenance behavior, so maintainers should coordinate the two fixes rather than landing conflicting integrity semantics.
