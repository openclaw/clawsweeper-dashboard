## Label Justifications

- `P1`: A real older-client install path for a compatible executable plugin fails because public integrity metadata disagrees with the legacy download bytes.
- `impact:security`: The issue is about executable package artifact integrity fields disagreeing across public resolver and legacy download paths.

## Summary

Keep this open: current source and live production still show npm-pack version metadata exposing the uploaded tarball hash while the legacy `/download` path serves a generated ZIP with a different digest, so older compatible clients can fail valid installs. The report is security-sensitive and needs maintainer agreement on the legacy digest contract before implementation.

## What This Changes

Ensure legacy ClawHub `sha256hash` stays internally consistent with the bytes served by legacy `/api/v1/packages/{name}/download` for npm-pack package versions.

## Best Possible Solution

Define an explicit dual-integrity contract: legacy responses should expose a digest matching the legacy ZIP or omit the legacy field, artifact-aware fields should keep the npm-pack digest, and existing rows should be repaired with regression coverage for both paths.

## Reproduction Assessment

Yes. Live production reproduces the mismatch for the reported package, and current main source explains why `sha256hash` can track npm-pack tarball bytes while legacy `/download` returns ZIP bytes; I did not run a local Convex fixture.

## Solution Assessment

Unclear. The requested invariant is right, but the permanent fix needs maintainer/security approval because `sha256hash` also carries scan and provenance meaning for npm-pack artifacts.

## Risks / Open Questions

- Changing `sha256hash` without an explicit contract could move VirusTotal, security UI, or provenance consumers from the scanned npm-pack tarball hash to a generated ZIP hash.
- Existing npm-pack rows may need a backfill, cached legacy ZIP digest, or intentional on-demand computation, otherwise old releases can remain inconsistent after new publishes are fixed.
- The related open artifact-integrity issue at https://github.com/openclaw/clawhub/issues/2104 shows this area affects public installer behavior and should be reviewed as an integrity contract change.
