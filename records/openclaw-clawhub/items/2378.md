## Label Justifications

- `P1`: A legacy install-time integrity mismatch can block real OpenClaw plugin installs for package versions that advertise compatibility with older clients.
- `impact:security`: The issue concerns public package digest metadata used to verify executable plugin download bytes.

## Summary

Keep open: current main still exposes `version.sha256hash` separately from npm-pack artifact metadata while legacy `/download` rebuilds and serves ZIP bytes with its own digest. This remains a security-sensitive compatibility bug for older installers, and the exact public legacy digest contract still needs maintainer/security choice before a repair changes semantics.

## What This Changes

The issue asks ClawHub to keep legacy package `sha256hash` consistent with the bytes returned by legacy `/api/v1/packages/{name}/download` for npm-pack-backed versions.

## Best Possible Solution

Define the legacy package integrity contract, then update the package API so old clients never compare ZIP bytes against npm-pack tarball metadata while artifact-aware clients retain exact tarball verification.

## Reproduction Assessment

Yes. Source inspection on current main shows npm-pack scan metadata can set `sha256hash` from tarball bytes while the legacy package download route serves generated ZIP bytes with a separate digest; I did not run a live Convex HTTP smoke in this read-only review.

## Solution Assessment

Unclear. The requested invariant is valid, but maintainers should choose between ZIP-specific legacy metadata, nulling legacy `sha256hash`, or adding explicit legacy download metadata before implementation.
