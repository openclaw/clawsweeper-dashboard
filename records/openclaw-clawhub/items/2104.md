## Label Justifications

- `P1`: A real plugin install path can fail integrity verification because an artifact-looking package URL returns bytes different from the manifest-backed artifact.
- `impact:security`: Package artifact URLs and digest verification are a supply-chain boundary for installable plugin code.

## Summary

Keep open: current main still routes extra package URL segments into the package handler and the broad version-detail branch still accepts `/versions/{version}/tarball`; live production also still returns JSON bytes at that artifact-looking URL while the explicit artifact download route returns the canonical tarball bytes.

## What This Changes

The issue reports that a package tarball-looking HTTP API URL returns different bytes than the CLI-resolved artifact, causing plugin install integrity verification to fail.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs intentionally serve or redirect to exact stored artifact bytes, or fail clearly without a digest-mismatched 200 response; validate the source-commit fallback separately if confirmed.

## Reproduction Assessment

Yes, at source level and against live production. Current main preserves trailing package segments and lets `/versions/{version}/tarball` fall into the version JSON branch, while production returns JSON bytes at that URL and tarball bytes at the explicit artifact route.

## Solution Assessment

Unclear until maintainers choose the API compatibility policy. Rejecting unknown version subpaths is the narrow defensive fix, while serving or redirecting `/tarball` to exact artifact bytes may be safer for existing installers that already use that URL shape.
