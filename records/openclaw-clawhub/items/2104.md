## Summary

Keep open: this is a ClawHub issue, and the reported install-facing artifact mismatch still reproduces against the hosted endpoint while current main still explains the bad route behavior.

## What This Changes

The issue reports that a package install URL under the HTTP API returns bytes whose SHA-256 does not match the manifest/CLI-downloaded package artifact.

## Best Possible Solution

Make package artifact resolution unambiguous, keep digest-checked ClawPack tarball bytes as the install source of truth, add route regression coverage, and correct or clarify source commit provenance.

## Reproduction Assessment

Yes. The hosted `clawhub.ai` endpoint currently returns different byte streams and digests for the tarball-looking version subpath versus the explicit ClawPack artifact download path, and current-main source inspection explains the fallthrough.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The narrow maintainable fix is either to reject unsupported version subpaths clearly or intentionally route compatible tarball aliases to the verified ClawPack artifact.
