## Summary

Keep open: the existing ClawSweeper comment is still accurate after a fresh check against current main and the live API. The artifact-looking `/tarball` subpath still returns version JSON with a mismatched digest, while the explicit artifact download returns the expected uploaded tarball bytes; this issue needs maintainer handling rather than cleanup closure.

## What This Changes

The issue reports that the package HTTP API `/versions/0.1.0/tarball` path returns bytes with a different digest from the CLI/downloaded artifact and also notes bad source-commit attribution for the affected release.

## Best Possible Solution

Make package artifact routes digest-safe by rejecting unsupported version subpaths or explicitly aliasing `/tarball` to the exact stored ClawPack artifact, then separately repair or document the affected release provenance.

## Reproduction Assessment

Yes. The live `/versions/0.1.0/tarball` URL returns JSON bytes with a different SHA-256 from the explicit artifact download, and current main still contains the broad version-detail branch that explains the fallthrough.

## Solution Assessment

Unclear. Rejecting unknown subpaths is the narrowest defensive fix, but maintainers should choose whether `/tarball` should 404, redirect, or become a supported artifact alias because external installers may already be trying that path.
