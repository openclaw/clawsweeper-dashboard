## Summary

Keep open: the reported artifact mismatch still reproduces on production, current main still has the route fallthrough that explains it, and ClawHub cleanup policy keeps issues open until a maintainer resolves them or a fixing PR lands.

## What This Changes

The issue reports that the `/versions/{version}/tarball` HTTP path returns different bytes and digest than the exact ClawPack artifact returned by the supported CLI/download routes.

## Best Possible Solution

Make package artifact routes digest-safe by rejecting or explicitly supporting unknown version subpaths with exact artifact bytes, then separately correct or document the affected release provenance.

## Reproduction Assessment

Yes. Production still returns JSON bytes with a different digest for the reported `/tarball` URL, and current main still contains the broad version-detail route that accepts that extra subpath.

## Solution Assessment

Unclear. Rejecting unknown subpaths is the narrowest defensive fix, but maintainers should decide the public route contract before choosing 404, redirect, or a supported tarball alias.
