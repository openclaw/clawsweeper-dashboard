## Label Justifications

- `P1`: A plugin install integrity mismatch breaks a real package install workflow and affects trust in executable plugin distribution.
- `impact:security`: The issue is about executable package artifact integrity and source provenance disagreeing across public install paths.

## Summary

Keep this open: live production still returns three different successful bodies for the affected package paths, and current main still has the same route split between exact ClawPack artifact bytes, generated legacy ZIP bytes, and an unguarded extra-suffix version JSON response. The open related issue at https://github.com/openclaw/clawhub/issues/2378 covers the narrower legacy `sha256hash` contract, but this issue also carries the `/tarball` fallthrough and source-provenance concern.

## What This Changes

The issue reports that plugin install integrity can fail because ClawHub HTTP package routes expose different artifact bytes and digests than the CLI/package manifest path.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from `/artifact/download` or the npm mirror, legacy `/download` stays internally consistent as ZIP compatibility, unsupported version suffixes do not return successful metadata bytes, and existing bad provenance is repaired through an approved moderation or backfill path.

## Reproduction Assessment

Yes. Live production currently reproduces the mismatched successful bodies, and current main source explains the `/tarball` fallthrough plus ZIP-only legacy route; I did not run a local Convex fixture, so I am marking this source-reproducible rather than a local current-main fixture reproduction.

## Solution Assessment

Unclear. The bug is concrete, but the permanent behavior needs maintainer choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.
