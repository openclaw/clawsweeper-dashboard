## Label Justifications

- `P1`: A real plugin install workflow can fail integrity verification because an artifact-looking package URL returns bytes different from the manifest-backed artifact.
- `impact:security`: Package artifact URLs and digest verification are a supply-chain security boundary for installable plugin code.

## Summary

Keep open: current main and live production still show the artifact-looking `/tarball` package URL returning JSON bytes instead of the manifest-backed tarball, and this security-sensitive route behavior needs maintainer policy review before repair.

## What This Changes

The issue reports a ClawHub package install integrity mismatch where `/versions/{version}/tarball` returns different bytes while the resolver and npm mirror paths return the manifest-matching `.tgz`.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs either serve or redirect to the exact stored artifact bytes by intentional compatibility policy, or fail clearly without returning digest-mismatched JSON, with regression coverage for the chosen behavior.

## Reproduction Assessment

Yes, at source level and against live production. Current main routes package paths by prefix and the broad version-detail branch accepts extra `/versions/{version}/...` segments, while live production returns JSON bytes for the reported `/tarball` path and the correct `.tgz` bytes for resolver/npm paths.

## Solution Assessment

Unclear until maintainers decide the compatibility contract. Rejecting unknown version subpaths is the narrow defensive fix, but serving or redirecting `/tarball` to exact artifact bytes may be safer if installers already depend on that shape.
