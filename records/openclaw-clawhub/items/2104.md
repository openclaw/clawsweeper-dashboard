## Summary

Keep open: the reported install-integrity mismatch still reproduces in production, current main still has the broad version-detail route that explains the JSON response on an artifact-looking subpath, and this needs maintainer route-contract/security handling rather than cleanup closure.

## What This Changes

The issue asks ClawHub to resolve an install-time integrity mismatch where `/api/v1/packages/{name}/versions/{version}/tarball` returns bytes that differ from the package artifact manifest, with a related source-commit provenance concern.

## Best Possible Solution

Harden package version routing so unknown version subpaths cannot return digest-relevant JSON, then explicitly choose 404, redirect, or documented exact-byte alias behavior and correct affected release provenance if the source commit is wrong.

## Reproduction Assessment

Yes. Live production returns JSON bytes with the wrong digest for the reported tarball-looking URL, while the supported npm mirror URL returns the expected 4678-byte artifact; current main source explains the fallthrough path.

## Solution Assessment

Unclear. The narrow defensive fix is to reject unknown version subpaths, but maintainers should choose the public route contract and decide how to handle the affected release provenance.
