The issue asks ClawHub to fix an install-facing package tarball path that returns bytes different from the CLI-resolved artifact and to clarify the source commit provenance fallback.

## Best Possible Solution

Define a digest-safe package artifact route contract with regression coverage for unsupported or aliased version subpaths, then handle source-commit fallback display as an explicit provenance policy.

## Reproduction Assessment

Yes. The live ClawHub API still returns metadata JSON from the reported `/tarball` path while the explicit artifact routes return the expected tarball bytes, and current main contains the broad version-detail route that explains the behavior.

## Solution Assessment

Unclear. The narrow code fix is likely to reject unknown `/versions/{version}/...` subpaths or intentionally alias/redirect `/tarball`, but maintainers should choose that API contract and decide whether the source-provenance concern belongs in the same change.
