## Label Justifications

- `P1`: The production mismatch blocks plugin installation for clients using the legacy HTTP path.
- `impact:security`: Artifact digests and source revisions are supply-chain trust signals for executable plugin packages.

## Summary

The issue is reproducible in production, and current main does not implement the requested compatibility and provenance behavior. The artifact-aware tgz path is internally consistent, but the legacy ZIP is a distinct byte stream and the questionable source commit remains public.

## What This Changes

Align legacy integrity metadata with legacy ZIP bytes while preserving npm-pack digests, and stop exposing unverified CLI build hashes as package source commits.

## Best Possible Solution

Define and document separate verified digest contracts for npm-pack and legacy ZIP representations, keep legacy fields compatible with legacy clients, and expose source commits only when backed by publisher-provided or trusted-publisher provenance.

## Reproduction Assessment

Yes. On June 11, 2026, the production artifact endpoint and tgz download agreed on SHA-256 5cf6706388a5770f043f7237e50502f7e2f8d498f6b402308a7b07c80b01b90d, while the legacy ZIP hashed to 1b36f01e16d5b826085d95dd28c6a770381536a742622bbd8f506b1591b1276f; the questionable sourceCommit was also still visible.

## Solution Assessment

No single field substitution is proven best yet. The maintainable solution is format-specific integrity metadata plus an explicit backward-compatibility contract and validated source provenance.
