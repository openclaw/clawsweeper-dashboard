## Summary

Keep open: current production still reproduces the digest split, and current main still allows the tarball-looking subpath to fall through to version metadata instead of returning exact artifact bytes or a clear unsupported-route response.

## What This Changes

The issue asks ClawHub to stop serving mismatched package artifact bytes and digests for the tarball-looking HTTP API path versus the CLI artifact download, with source-commit provenance also called out.

## Best Possible Solution

Make package artifact resolution unambiguous and digest-safe for install clients, with tests/docs for the chosen route behavior and a clear source provenance rule for publishes.

## Reproduction Assessment

Yes. Current production on the deployed current-main tag returns different byte streams and SHA-256 digests for the tarball-looking path, explicit artifact download, npm mirror, and legacy ZIP route, and source inspection explains the fallthrough.

## Solution Assessment

Unclear. The narrow code surface is identifiable, but maintainers should first decide whether the unsupported tarball-looking route should alias to exact ClawPack bytes or fail clearly, and how strict source provenance should be.

## Security Review

Status: needs_attention

Summary: This is supply-chain sensitive because an install-facing package URL can return bytes that do not match the advertised ClawPack artifact digest.
