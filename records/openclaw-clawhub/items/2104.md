## Summary

Keep open: the live install-integrity mismatch still reproduces, and current main still has the broad package version-detail route that can return JSON for the undocumented artifact-looking `/tarball` subpath. This is security-sensitive package install behavior and needs maintainer handling rather than cleanup closure.

## What This Changes

The issue reports that an artifact-looking package tarball URL returns different bytes than the CLI-supported artifact download, causing plugin install integrity checks to fail.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs cannot return non-artifact JSON, then handle any affected published release provenance separately.

## Reproduction Assessment

Yes. The live `/tarball` URL still returns JSON bytes with a different digest than the supported artifact download, and current main contains the same broad version-detail route that explains the behavior.

## Solution Assessment

Unclear. Rejecting unknown version subpaths is the narrowest defensive fix, but maintainers should decide whether the public contract should be 404, redirect, or a supported exact-byte tarball alias.
