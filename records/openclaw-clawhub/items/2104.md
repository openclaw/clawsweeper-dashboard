## Summary

Keep open: current main and live production still show an integrity-sensitive route mismatch where `/versions/{version}/tarball` returns version JSON instead of artifact bytes or a clear 404, while the supported npm mirror path returns the exact stored tarball.

## What This Changes

The issue reports a plugin install integrity failure where an HTTP package tarball URL serves bytes different from the CLI-verified artifact, with a related source-commit provenance concern on the listing.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs either serve the exact stored artifact bytes through a documented compatibility alias or fail clearly without returning digest-mismatched JSON, then separately verify listing source-commit provenance if still wrong.

## Reproduction Assessment

Yes, at source level and against live production. Current main still accepts extra version subpaths in the broad version-detail route, and the live `/tarball` URL returns JSON bytes while the documented npm tarball path returns the expected artifact digest.

## Solution Assessment

Unclear. Rejecting unknown version subpaths is the narrow defensive fix, but supporting `/tarball` as a compatibility alias may be better if installers already depend on that URL shape.
