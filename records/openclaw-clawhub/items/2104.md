## Summary

Keep open: current main still has the loose package-version subpath routing that explains the reported artifact-looking URL returning metadata JSON, and live clawhub.ai checks still reproduce distinct byte streams and digests across the tarball-looking, explicit ClawPack, and legacy ZIP paths. This is an install-path and supply-chain-relevant issue, and ClawHub cleanup policy keeps issues open rather than auto-closing them.

## What This Changes

The issue reports a ClawHub package install integrity mismatch where a tarball-looking HTTP API path serves different bytes than the CLI-verified ClawPack artifact, plus suspicious source commit provenance.

## Best Possible Solution

Make package artifact resolution unambiguous, keep digest-checked ClawPack tarball bytes as the install source of truth, add regression coverage for unsupported version subpaths, and validate or repair source commit provenance.

## Reproduction Assessment

Yes. Live clawhub.ai endpoint checks reproduce the different byte streams and digests, and current-main source inspection shows how the tarball-looking subpath falls through to metadata JSON without running a local Convex server.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The narrow maintainable fix is either to reject unsupported version subpaths clearly or intentionally route compatible tarball requests to the exact verified ClawPack artifact, with provenance validation handled in the publish path.
