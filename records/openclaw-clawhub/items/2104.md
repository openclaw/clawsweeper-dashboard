## Summary

Keep open: this is a live install-integrity bug on an install-facing package artifact path, and current main still lets a tarball-looking version subpath return version metadata bytes instead of either artifact bytes or a clear unsupported-route failure.

## What This Changes

The issue reports that a plugin install path hitting `/api/v1/packages/{name}/versions/{version}/tarball` receives metadata bytes instead of the advertised tarball and fails integrity verification, with a related source commit provenance concern.

## Best Possible Solution

Define a single digest-safe artifact contract for package installs, make ambiguous version subpaths either stream the advertised artifact or return a clear unsupported-route response, and add regression coverage for artifact URL semantics plus source provenance handling.

## Reproduction Assessment

Yes. The live reported tarball-looking URL returns 5,332 bytes of JSON with a mismatched digest, while the explicit artifact download returns the 4,678-byte package artifact with the expected digest, and current main's route fallthrough explains the behavior.

## Solution Assessment

Unclear. A narrow code fix is likely, but maintainers should first choose whether `/versions/{version}/tarball` is a supported artifact alias or an invalid route, and define the sourceCommit fallback policy separately.
