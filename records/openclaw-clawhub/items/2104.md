## Label Justifications

- `P1`: The issue affects a real plugin install workflow with package artifact integrity failures and a clear current-main route path.
- `impact:security`: Package artifact URLs and digest verification are supply-chain security boundaries for installable plugin code.

## Summary

Keep open: current main and live production still let `/versions/{version}/tarball` fall through to version JSON while the documented artifact download route returns the stored tarball bytes, so install clients can hit digest mismatches on plugin code.

## What This Changes

The issue asks ClawHub to stop serving digest-mismatched content for plugin tarball-looking package URLs and align the HTTP install path with the CLI artifact bytes.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs either serve or redirect to the exact stored artifact bytes through a documented compatibility path, or fail clearly without returning digest-mismatched JSON; add regression coverage for the chosen contract.

## Reproduction Assessment

Yes. A live curl to the reported `/tarball` URL returns JSON bytes with a different digest while `/artifact/download` returns the expected tarball, and source inspection shows why the extra segment falls through.

## Solution Assessment

Unclear. Rejecting unknown version subpaths is the narrow defensive fix, but serving or redirecting `/tarball` to exact artifact bytes may be the better compatibility choice if existing installers depend on that shape.
