## Summary

Keep this open: current main still leaves the artifact contract split in place, and live ClawHub endpoints still return different successful bodies for the explicit npm-pack artifact, legacy ZIP download, and unsupported tarball suffix.

## What This Changes

The issue reports that ClawHub package install paths can serve bytes whose digest does not match the package metadata expected by installers.

## Best Possible Solution

Define one package artifact contract: explicit artifact and npm mirror paths serve exact ClawPack bytes, legacy ZIP downloads expose matching legacy integrity or fail clearly, and unsupported suffixes such as /tarball do not return successful metadata bytes.

## Reproduction Assessment

Yes. Source inspection on current main shows separate tarball metadata and legacy ZIP download paths, and live production on 2026-06-10 still returns different successful bodies for the affected package routes.

## Solution Assessment

Unclear. The reported problem is real, but the permanent fix needs maintainer/security choice between remapping legacy sha256hash, adding explicit legacy download metadata, redirecting, or failing unsupported routes clearly.
