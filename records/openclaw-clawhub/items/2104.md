## Summary

Keep this open: live production at the current main deploy still returns successful but different bodies for the affected package paths, and current source still has the version-metadata, explicit artifact, legacy ZIP, and unsupported suffix behavior that explains the integrity mismatch. The related issue at https://github.com/openclaw/clawhub/issues/2378 overlaps on the legacy `sha256hash` invariant, but this issue also carries the unsupported `/tarball` success response and source-provenance repair concern.

## What This Changes

The issue reports that npm-pack plugin releases expose and download mismatched artifacts across HTTP paths, causing install integrity failures and misleading source provenance.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from `/artifact/download` or the npm mirror, legacy `/download` is internally consistent or fails clearly, unsupported suffixes do not return successful metadata bytes, and bad provenance is repaired through an approved backfill or moderation path.

## Reproduction Assessment

Yes. Live production deployed from current main reproduces mismatched successful bodies for the affected package paths, and source at `963b0a5719431a182bef07ed1cd4ff09cc6fc1c9` explains the `/tarball` fallthrough plus ZIP-only legacy route.

## Solution Assessment

Unclear. The reported invariant is valid, but the permanent fix needs maintainer choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.
