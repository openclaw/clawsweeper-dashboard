## Label Justifications

- `P1`: A plugin install integrity mismatch breaks a real package install workflow and affects trust in executable plugin distribution.
- `impact:security`: The issue is about executable package artifact integrity and source provenance disagreeing across public install paths.

## Summary

Keep this open: current production still reproduces the artifact mismatch, and current main still has the route shape that can return version JSON for the reported `/tarball` suffix while the documented artifact resolver and legacy ZIP routes return different byte streams. The related open issue at https://github.com/openclaw/clawhub/issues/2378 covers the narrower legacy `sha256hash` contract, but this issue also includes the `/tarball` fallthrough and source-provenance concern.

## What This Changes

The issue reports that ClawHub's public HTTP package paths can serve bytes whose digest differs from the CLI-downloaded npm-pack plugin artifact.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from `/artifact/download` or the npm mirror, legacy `/download` stays internally consistent as ZIP compatibility, unsupported version suffixes do not return successful metadata bytes, and existing bad provenance is repaired through an approved moderation or backfill path.

## Reproduction Assessment

Yes. Live production reproduces the mismatched successful bodies, and current main source explains the `/tarball` fallthrough plus ZIP-only legacy route; I did not run a local Convex fixture, so this is source-reproducible rather than a local current-main fixture reproduction.

## Solution Assessment

Unclear. The bug is concrete, but the permanent behavior needs maintainer choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.

## Risks / Open Questions

- Maintainers still need to choose whether `/api/v1/packages/{name}/versions/{version}/tarball` should fail closed, redirect, or become a compatibility alias for external installers.
- Changing `sha256hash` without a clear compatibility contract could move security, VirusTotal, and provenance consumers from the scanned npm-pack tarball hash to a generated ZIP hash.
- Existing published rows such as `zerogpu-router@0.1.0` may need a moderation or backfill repair for source metadata; a route-only fix will not prove or correct already published provenance.
