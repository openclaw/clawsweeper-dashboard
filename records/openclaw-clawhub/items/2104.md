## Summary

Keep open: live production still reproduces the package artifact mismatch, and current main still routes the artifact-looking `/tarball` subpath into version JSON instead of exact artifact bytes. This is an install-integrity/security-sensitive issue, and ClawHub policy keeps issues open for maintainer handling.

## What This Changes

The issue asks ClawHub to resolve an integrity mismatch where `/versions/{version}/tarball` returns different bytes than the CLI-resolved package artifact.

## Best Possible Solution

Make package artifact routes digest-safe by rejecting unknown version subpaths or deliberately serving exact artifact bytes for a supported tarball alias, then document the chosen route contract.

## Reproduction Assessment

Yes. The live `/versions/0.1.0/tarball` URL still returns JSON bytes with a different size and digest from the exact artifact download, and current main contains the route shape that explains that behavior.

## Solution Assessment

Unclear. The narrow defensive fix is to stop unknown version subpaths from returning version JSON, but maintainers should decide whether the public contract should be 404, redirect, or a supported tarball alias.
