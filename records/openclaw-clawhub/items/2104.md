## Summary

Keep open: the current source and live service still support the reported package install integrity mismatch, and the decision about whether `/versions/{version}/tarball` should alias artifact bytes or reject clearly is supply-chain-sensitive enough for maintainer review.

## What This Changes

The issue reports that a package install path using the HTTP tarball-looking URL returns bytes whose SHA-256 differs from the manifest/CLI-downloaded artifact for `zerogpu-router@0.1.0`.

## Best Possible Solution

Define one digest-safe package artifact contract, implement it consistently in the HTTP API and CLI expectations, then add regression coverage and docs for route behavior and source provenance.

## Reproduction Assessment

Yes. The live tarball-looking URL still returns 5,332 bytes with the reporter's mismatched digest, while the explicit artifact route returns 4,678 bytes with the expected manifest/CLI digest.

## Solution Assessment

Unclear. The route surface is narrow, but maintainers should decide whether the unsupported tarball-looking path must alias exact artifact bytes for compatibility or fail clearly so clients use `/artifact`.
