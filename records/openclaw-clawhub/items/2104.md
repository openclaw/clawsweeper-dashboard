## Summary

Keep this open: current source still explains the reported mismatch, live production reproduces different successful responses for the exact artifact, legacy ZIP, and `/tarball` paths, and the related open report at https://github.com/openclaw/clawhub/issues/2378 only covers the narrower legacy `sha256hash` contract rather than the full `/tarball` and provenance concern.

## What This Changes

The issue reports that installing `zerogpu-router` through an HTTP package tarball/download path fails integrity verification because the API returns bytes or metadata that differ from the CLI artifact resolver.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes should come from `/artifact/download` or the npm mirror, legacy `/download` should stay internally consistent as ZIP compatibility, unknown version suffixes should not return successful metadata bytes, and existing bad provenance should be repaired through a maintainer-approved moderation/backfill path.

## Reproduction Assessment

Yes. Live production reproduces the mismatched successful bodies, and current main source explains the `/tarball` fallthrough plus the ZIP-only legacy route; I did not run a local Convex fixture, so this is source-reproducible rather than a local current-main test reproduction.

## Solution Assessment

Unclear. The bug is narrow, but the best public behavior depends on a maintainer choice between strict 404, a `/tarball` compatibility redirect, requiring external clients to use `/artifact/download`, and any existing-row provenance repair.
