## Summary

Keep this open: live production still returns different successful bodies for the affected artifact paths, and current main still has the route and metadata split that explains the install-time integrity failure. The related issue at https://github.com/openclaw/clawhub/issues/2378 covers the narrower legacy `sha256hash` contract, but this issue also carries the unsupported `/tarball` success response and source-provenance concern.

## What This Changes

The issue reports that public HTTP plugin artifact paths can return bytes whose digest differs from the CLI/package manifest artifact, causing plugin install integrity mismatches.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from `/artifact/download` or the npm mirror, legacy `/download` is internally consistent or fails clearly, unsupported suffixes do not return successful metadata bytes, and bad provenance is repaired through an approved backfill or moderation path.

## Reproduction Assessment

Yes. Live production on May 24, 2026 reproduced three different successful bodies for `zerogpu-router@0.1.0`, and current-main source explains the `/tarball` fallthrough plus ZIP-only legacy route.

## Solution Assessment

Unclear. The reported invariant is valid, but the permanent fix needs maintainer choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.
