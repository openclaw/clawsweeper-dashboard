## Summary

Keep this open: current production still reproduces the artifact mismatch, current main still has the `/versions/{version}/tarball` fallthrough and legacy ZIP-only `/download` split, and the open related issue only covers the narrower legacy `sha256hash` contract.

## What This Changes

The issue reports that ClawHub package install paths can serve different artifact bytes and integrity hashes for the same plugin version, causing install-time integrity mismatches and confusing source provenance.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from `/artifact/download` or the npm mirror, legacy `/download` stays internally consistent for old clients, unsupported `/tarball` suffixes do not return successful metadata bytes, and existing bad provenance is repaired through an approved moderation or backfill path.

## Reproduction Assessment

Yes. Live production for the current deployed main SHA reproduces the mismatched successful bodies, and current main source explains the `/tarball` fallthrough plus ZIP-only legacy route.

## Solution Assessment

Unclear. The bug is concrete, but the permanent behavior needs maintainer choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.
