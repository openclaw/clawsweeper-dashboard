## Label Justifications

- `P1`: A plugin install integrity mismatch breaks a real package install workflow and affects trust in executable plugin distribution.
- `impact:security`: The issue is about executable package artifact integrity and source provenance disagreeing across public install paths.

## Summary

Keep this open: production still returns mismatched successful bodies for the affected package, and current main still has the route shape that lets an unsupported `/tarball` suffix return version JSON while the documented artifact and legacy ZIP routes return different artifacts. The related open issue at https://github.com/openclaw/clawhub/issues/2378 covers the narrower legacy `sha256hash` contract, but this issue still carries separate `/tarball` fallthrough and source-provenance evidence.

## What This Changes

Ensure ClawHub package install/download artifact endpoints and integrity metadata agree for npm-pack plugins, including the affected zerogpu-router release and published source provenance.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from `/artifact/download` or the npm mirror, legacy `/download` remains internally consistent ZIP compatibility, unsupported version suffixes do not return successful metadata bytes, and bad existing provenance is repaired through an approved moderation or backfill path.

## Reproduction Assessment

Yes. Live production reproduces the mismatched successful bodies, and current main source explains the `/tarball` fallthrough plus ZIP-only legacy route; I did not run a local Convex fixture, so this is source-reproducible rather than a local current-main fixture reproduction.

## Solution Assessment

Unclear. The bug is concrete, but the permanent behavior needs maintainer choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.
