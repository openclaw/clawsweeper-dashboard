## Label Justifications

- `P1`: A real plugin install path can fail integrity verification because an artifact-looking package URL returns bytes different from the manifest-backed artifact.
- `impact:security`: Package artifact URLs and digest verification are a supply-chain boundary for installable plugin code.

## Summary

Keep open: the reported install integrity mismatch is still reproducible against current deployed main, and the source router still lets an artifact-looking package URL return version JSON bytes under HTTP 200.

## What This Changes

The issue asks ClawHub to stop serving digest-mismatched bytes from a plugin package tarball-looking HTTP API URL used by install clients.

## Best Possible Solution

Define one package API policy for /versions/{version}/tarball, then make artifact-looking URLs either serve or redirect to the exact stored artifact bytes or fail clearly without returning digest-mismatched JSON.

## Reproduction Assessment

Yes. Current deployed main returns HTTP 200 JSON bytes for /versions/0.1.0/tarball while the explicit artifact download route returns the expected 4678-byte tarball, and the current source routing explains that behavior.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. Rejecting unknown version subpaths is the narrow defensive fix, while serving or redirecting /tarball may be better if existing installers already depend on that URL shape.
