## Summary

Keep open: the documented artifact resolver and npm mirror now serve the correct ClawPack bytes, but the reported `/versions/{version}/tarball` path still returns version JSON with HTTP 200, which explains the install-time integrity mismatch for clients using that path.

## What This Changes

The issue asks ClawHub to stop serving non-matching bytes for plugin install tarball downloads and to fix the bad source-commit attribution on the affected package release.

## Best Possible Solution

Define the resolver contract explicitly: either add a tested `/versions/{version}/tarball` compatibility path that streams the same ClawPack bytes and headers as the npm mirror, or make unsupported version subpaths return 404 and update the affected installer/client guidance; separately repair or republish the bad source attribution.

## Reproduction Assessment

Yes. The current live deployment reproduces the mismatch: the reported `/tarball` path returns JSON with a different hash, while the documented artifact route and npm mirror return the expected 4678-byte tarball.

## Solution Assessment

Unclear until maintainers decide the compatibility contract. The narrow code fix is straightforward, but the product/security choice is whether to support the legacy-looking tarball URL or reject it clearly and update clients.
