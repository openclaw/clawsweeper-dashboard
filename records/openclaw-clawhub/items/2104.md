## Label Justifications

- `P1`: A real install workflow can fail integrity verification when an artifact-looking package URL returns bytes different from the manifest-backed artifact.
- `impact:security`: Package artifact URLs and digest verification are a supply-chain security boundary for installable plugin code.

## Summary

Keep open: current main still has a route shape where the artifact-looking package path `/versions/{version}/tarball` can return version JSON instead of exact artifact bytes or a clear miss, and live production still returns a different digest from the manifest-backed tarball.

## What This Changes

The issue reports an install-time integrity mismatch where a package tarball-looking HTTP API URL serves different bytes than the CLI-resolved artifact, plus incorrect source-commit metadata on the package listing.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs either intentionally stream or redirect to the exact stored artifact bytes, or fail clearly without returning digest-mismatched JSON; handle the bad source provenance as a separate data correction if needed.

## Reproduction Assessment

Yes, at source level and against live production. Current main routes package paths by prefix and accepts extra version subpath segments, while production currently returns JSON bytes for the reported `/tarball` URL and exact tarball bytes for the documented npm artifact URL.

## Solution Assessment

Unclear. Rejecting unknown version subpaths is the narrow defensive fix, but serving or redirecting `/versions/{version}/tarball` to exact artifact bytes may be the better compatibility choice if installers already use that shape.
