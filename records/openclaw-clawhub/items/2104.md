## Summary

Keep open: this issue is still current and security-sensitive. Current main still routes unknown package version subpaths into the version-detail JSON response, so an artifact-looking install URL can return bytes that do not match the artifact digest.

## What This Changes

The issue reports that a package version tarball URL returns different bytes from the CLI/package artifact download path, causing install integrity verification to fail.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs either serve the exact stored artifact bytes through a documented alias or fail clearly without returning digest-mismatched JSON.

## Reproduction Assessment

Yes. Source inspection on current main shows the same route shape described by the live reproduction: extra version subpaths are accepted by the broad version-detail branch instead of being rejected or routed to artifact streaming.

## Solution Assessment

Unclear. The narrow defensive fix is to reject unknown subpaths, but supporting /tarball as an artifact alias may be better for compatibility if existing installers already depend on that URL shape.
