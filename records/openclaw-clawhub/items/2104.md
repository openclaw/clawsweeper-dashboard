## Summary

Keep open: the live route and current main still show the tarball-looking package URL returning version JSON while the supported artifact resolver advertises the smaller ClawPack tarball, so the install integrity report remains valid and needs maintainer contract follow-up.

## What This Changes

The issue asks ClawHub to resolve a package install integrity mismatch where a tarball-looking HTTP API URL returns different content than the CLI artifact resolver, and to clarify source commit provenance.

## Best Possible Solution

Define one digest-safe package artifact contract, implement the chosen reject-or-alias behavior for unsupported tarball-looking paths, and add regression coverage plus source provenance policy docs/tests.

## Reproduction Assessment

Yes. A live request to the reported tarball-looking URL still returns 5,332-byte JSON with the mismatched digest, and current main source explains it through the broad /versions/{version} fallback.

## Solution Assessment

Unclear. The code surface is narrow, but maintainers should first decide whether this unsupported route should become an exact artifact alias or a clear error, and how strict non-trusted source provenance should be.
