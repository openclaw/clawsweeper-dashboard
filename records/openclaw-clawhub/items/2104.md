## Summary

Keep open: current main still has the package route shape that can return version JSON from an artifact-looking `/versions/{version}/tarball` path, and live production still shows the mismatch against the explicit artifact download path.

## What This Changes

The issue asks to fix an install-time integrity mismatch where an HTTP package tarball path returns different bytes than the CLI artifact resolver and to investigate misleading source commit metadata.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs either serve or redirect to the exact stored artifact bytes through a documented compatibility path, or fail clearly without returning digest-mismatched JSON; validate the source commit metadata separately if needed.

## Reproduction Assessment

Yes, at source level and against live production. Current main routes the extra `tarball` segment into the broad version-detail JSON branch, while the live explicit artifact download returns the expected tarball digest.

## Solution Assessment

Unclear. Rejecting unknown version subpaths is the narrow defensive fix, but serving or redirecting `/tarball` to exact artifact bytes may be the better compatibility choice if installers already depend on that shape.
