## Summary

Keep this open: current main still has the artifact-contract split, and live ClawHub endpoints still return different successful bodies for the affected tarball suffix, explicit artifact download, and legacy ZIP download paths.

## What This Changes

The issue reports that ClawHub install paths can compare integrity against different artifact bytes when the HTTP API serves a generated ZIP while package metadata expects the npm-pack tarball.

## Best Possible Solution

Define one artifact contract: exact ClawPack bytes come from the explicit artifact or npm mirror path, legacy ZIP downloads expose matching legacy integrity or fail clearly, and unsupported suffixes such as /tarball do not return successful metadata bytes.

## Reproduction Assessment

Yes. Live production on June 10, 2026 still returns three successful but different bodies for the affected tarball, explicit artifact, and legacy download paths, and current main source explains the route split.

## Solution Assessment

Unclear. The route strictness piece is narrow, but the permanent digest behavior needs maintainer/security choice between fail-closed behavior, compatibility aliasing, redirecting, ZIP-specific legacy metadata, and existing-row provenance repair.
