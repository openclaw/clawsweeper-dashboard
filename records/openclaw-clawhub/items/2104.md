## Summary

Keep open: the hosted mismatch still reproduces, current main still lets extra version subpaths return JSON metadata, and ClawHub cleanup policy keeps issues open for maintainer follow-up.

## What This Changes

The issue reports that a package install path fetches bytes from a tarball-looking HTTP API URL that do not match the manifest-backed ClawPack artifact downloaded by the CLI.

## Best Possible Solution

Make package artifact resolution unambiguous: install clients should receive digest-checked ClawPack bytes from a supported resolver/npm route or a clear unsupported-route error, with regression coverage and a source-provenance clarification.

## Reproduction Assessment

Yes. Hosted endpoint checks reproduce the reported byte and digest mismatch, and current-main source inspection shows why an extra `tarball` segment reaches the version metadata JSON response.

## Solution Assessment

Unclear until maintainers choose the route compatibility contract. The narrow maintainable fix is either to alias the tarball-looking path to exact ClawPack bytes or to reject unsupported version subpaths clearly, then cover that behavior in tests.
