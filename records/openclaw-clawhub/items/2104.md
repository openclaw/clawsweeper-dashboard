## Summary

Keep open: the reported install integrity mismatch still reproduces against the deployed current-main package routes, and the remaining fix needs maintainer judgment on artifact URL semantics and source provenance policy.

## What This Changes

The issue reports that a package install path can fetch bytes from a tarball-looking HTTP API URL that do not match the manifest artifact bytes downloaded by the CLI.

## Best Possible Solution

Define one digest-safe package artifact contract, implement that behavior in the HTTP API and CLI expectations, and add regression coverage and docs for the chosen route and source provenance rule.

## Reproduction Assessment

Yes. Live requests to the current deployed route reproduce the reported 5,332-byte JSON response and digest, while the explicit artifact download and npm mirror return the 4,678-byte ClawPack artifact with the expected digest.

## Solution Assessment

Unclear. The code surface is narrow, but maintainers should first decide whether the tarball-looking v1 subpath should alias exact artifact bytes or return a clear unsupported-route error, and how strict source metadata inference should be.
