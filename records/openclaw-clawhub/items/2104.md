- `impact:security`: The issue concerns executable package artifact digest metadata and install integrity verification boundaries.

## Summary

Keep this open: current main still leaves the reported artifact-contract split in place. The documented explicit artifact path returns the expected npm-pack tarball, but the affected `/tarball` suffix still returns successful JSON and legacy `/download` still returns ZIP bytes with a different digest; the related issue at https://github.com/openclaw/clawhub/issues/2378 only covers the narrower legacy `sha256hash` invariant.

## What This Changes

The issue reports that ClawHub HTTP install paths can serve bytes whose digest differs from the published package artifact, causing plugin installation integrity failures.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from `/artifact/download` or the npm mirror, legacy `/download` is internally consistent or fails clearly for npm-pack versions, and unsupported suffixes such as `/tarball` do not return successful metadata bytes.

## Reproduction Assessment

Yes. Live production on June 10, 2026 still returns three successful but different bodies for the affected tarball, explicit artifact, and legacy download paths, and current main source explains the permissive `/versions/{version}` fallthrough.

## Solution Assessment

Unclear. The route strictness piece is narrow, but the permanent fix needs maintainer/security choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.
