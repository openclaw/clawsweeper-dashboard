## Summary

This remains a valid, unresolved package-integrity bug. Current main intentionally provides exact npm-pack tarballs and separately generated legacy ZIP downloads, but no current-main source or release provenance proves that the legacy integrity field is aligned with the ZIP bytes; the newer related report covers the shared old-client contract but remains open rather than superseding this evidence.

## What This Changes

Ensure plugin integrity metadata matches the exact archive bytes served through ClawHub’s modern artifact and legacy HTTP download paths.

## Best Possible Solution

Define one explicit backward-compatible contract: npm-pack fields must verify the exact uploaded `.tgz`, legacy fields must verify the generated ZIP returned by legacy routes, and real HTTP tests must cover both artifact-aware and older-client installation.

## Reproduction Assessment

Yes, at source level. The current dual-artifact contract plus the reporter’s concrete byte sizes and hashes provides a high-confidence reproduction path, although this read-only review did not rerun the production install.

## Solution Assessment

The endpoint-specific integrity invariant is the correct direction, but maintainers must decide the permanent backward-compatible meaning of legacy `sha256hash` before implementation.
