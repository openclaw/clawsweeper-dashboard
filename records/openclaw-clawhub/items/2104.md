## Summary

Keep open: repository policy does not auto-close ClawHub issues, and current main still has the source-level route shape that can return version JSON for artifact-looking package subpaths such as `/versions/{version}/tarball`. The report is security-sensitive because install clients can receive bytes that do not match the advertised artifact digest.

## What This Changes

The issue reports that the HTTP package tarball path can serve bytes different from the CLI-resolved package artifact, causing install-time integrity mismatches.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs either serve or redirect to the exact stored artifact bytes through a documented compatibility path, or fail clearly without returning digest-mismatched JSON; add regression coverage for the chosen contract.

## Reproduction Assessment

Yes, at source level. Current main routes package subpaths through a prefix router, preserves the extra `tarball` segment, and then accepts any `/versions/{version}/...` path in the version-detail branch after the explicit artifact branch is skipped.

## Solution Assessment

Unclear. Rejecting unknown subpaths is the narrow defensive fix, but serving or redirecting `/tarball` to exact artifact bytes may be the better compatibility choice if existing installers depend on that shape.
