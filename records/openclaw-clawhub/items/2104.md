## Summary

Keep open: current hosted behavior still reproduces the artifact digest mismatch, and current main still contains a broad package version route that can answer a tarball-like subpath with metadata. ClawHub policy keeps issues open for maintainer handling, and this one is supply-chain relevant.

## What This Changes

The issue reports that plugin installation through an HTTP package tarball path receives bytes whose digest differs from the CLI-downloaded artifact and manifest.

## Best Possible Solution

Make package artifact resolution unambiguous, keep digest-checked ClawPack artifact bytes as the install source of truth, add regression coverage for unsupported version subpaths, and clarify source commit provenance when publishing with explicit source metadata.

## Reproduction Assessment

Yes. The hosted endpoints currently return different byte streams and digests for the tarball-looking path, the explicit artifact download, and the legacy download route, and current-main source inspection explains the version-subpath fallthrough.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The narrow maintainable direction is to make unsupported artifact-like subpaths fail clearly or intentionally serve the exact verified ClawPack artifact bytes, with regression tests for the chosen behavior.
