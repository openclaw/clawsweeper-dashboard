- `P1`: A plugin install integrity mismatch breaks a real package install workflow and affects trust in executable plugin distribution.
- `impact:security`: The issue concerns executable plugin artifact integrity and source provenance disagreeing across public install paths.

## Summary

Keep this open: production still reproduces the artifact mismatch, and current main still has separate npm-pack artifact, legacy ZIP download, and unguarded version-suffix behavior that can serve successful non-artifact bytes to installer-style clients.

## What This Changes

The issue reports that ClawHub package HTTP install paths can return bytes whose digest differs from the CLI/published plugin artifact, plus incorrect source provenance on the affected listing.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from /artifact/download or the npm mirror, legacy /download stays internally consistent as ZIP compatibility, unsupported suffixes do not return successful metadata bytes, and bad published provenance is repaired through an approved moderation or backfill path.

## Reproduction Assessment

Yes. Live production reproduces the mismatched successful bodies, and current main source explains the /tarball fallthrough plus separate tarball and ZIP byte streams; I did not run a local Convex fixture, so this is source-reproducible rather than a local current-main fixture reproduction.

## Solution Assessment

Unclear. The bug is concrete, but the permanent fix needs maintainer/security choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.
