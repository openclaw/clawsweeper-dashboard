## Summary

Keep open: the report is reproducible and current main still accepts the tarball-looking subpath as version metadata, causing a digest mismatch on an install-facing path; no duplicate or merged fix was found.

## What This Changes

The issue asks ClawHub to fix package install integrity mismatches where the `/versions/{version}/tarball` HTTP path returns bytes that differ from the CLI-resolved artifact, and to clarify incorrect source commit provenance.

## Best Possible Solution

Make package artifact URL handling digest-safe, add regression coverage for unsupported or aliased artifact subpaths, and define how source commit provenance should be displayed and trusted.

## Reproduction Assessment

Yes. The live reported tarball-looking URL returns JSON with the reporter's mismatched digest while the explicit artifact download returns the expected artifact bytes, and current main's route fallthrough explains the behavior.

## Solution Assessment

Unclear. A narrow fix is likely, but maintainers should first decide whether `/versions/{version}/tarball` is a supported artifact alias or an invalid route, and handle the sourceCommit report as a separate provenance policy check.
