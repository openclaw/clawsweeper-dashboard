## Summary

Keep open: the hosted endpoints still return different byte streams for artifact-looking package URLs, and current main still has a version-subpath fallthrough that can explain the mismatch. ClawHub cleanup policy keeps issues open, and this one is install-path and supply-chain relevant.

## What This Changes

The issue reports that OpenClaw/KiloClaw plugin installs receive HTTP package artifact bytes that do not match the ClawHub CLI artifact manifest, with suspect source commit metadata.

## Best Possible Solution

Make package artifact resolution unambiguous, keep digest-checked ClawPack tarball bytes as the install source of truth, add regression coverage for unsupported version subpaths, and correct or clarify source commit provenance.

## Reproduction Assessment

Yes. Hosted curl checks reproduce the digest mismatch, and current-main source inspection shows the /versions/{version} catchall that lets an artifact-looking extra segment return metadata JSON.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The narrow maintainable fix is either to make unsupported version subpaths fail clearly or to route compatible tarball requests to the exact verified ClawPack artifact, with regression tests for the chosen behavior.
