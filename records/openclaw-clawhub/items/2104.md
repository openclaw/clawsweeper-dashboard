## Label Justifications

- `P1`: A real plugin install path can fail integrity verification because an artifact-looking package URL returns bytes different from the manifest-backed artifact.
- `impact:security`: Package artifact URLs and digest verification are a supply-chain boundary for installable plugin code.

## Summary

Keep open: current main still preserves the trailing tarball segment and lets the reported artifact-looking URL fall through to version JSON, while live production returns JSON bytes at that URL and exact tarball bytes at the explicit artifact download route.

## What This Changes

The issue reports that an artifact-looking package HTTP URL returns bytes with a different digest than the CLI-resolved artifact, causing plugin install integrity verification to fail.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs either intentionally serve or redirect to the exact stored artifact bytes, or fail clearly without returning digest-mismatched JSON.

## Reproduction Assessment

Yes, at source level and against live production. Current main routes package subpaths by prefix, preserves trailing segments, and lets /versions/{version}/tarball fall into the version JSON branch instead of exact artifact bytes.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. Rejecting unknown version subpaths is the narrow defensive fix, while serving or redirecting /tarball to exact artifact bytes may be better if installers already depend on that URL shape.
