<details>
<summary>Evidence reviewed</summary>

What I checked:

- **Legacy response serialization:** The package-version response returns `result.version.sha256hash` directly (or null), rather than a digest derived from the generated legacy ZIP. ([`convex/httpApiV1/packagesV1.ts:3518`](https://github.com/openclaw/clawhub/blob/d6500794c618/convex/httpApiV1/packagesV1.ts#L3518), [d6500794c618](https://github.com/openclaw/clawhub/commit/d6500794c618))
- **Legacy download bytes:** The legacy package download handler builds and returns a deterministic ZIP from release files, so its bytes have a distinct integrity identity from an npm-pack tarball. ([`convex/httpApiV1/packagesV1.ts:3608`](https://github.com/openclaw/clawhub/blob/d6500794c618/convex/httpApiV1/packagesV1.ts#L3608), [d6500794c618](https://github.com/openclaw/clawhub/commit/d6500794c618))
- **Compatibility history:** Commit 4c69f2af made package hashes nullable and commit 199e6a0 exposed legacy ZIP aliases, but current main still does not bind the legacy hash to the generated ZIP bytes. ([`packages/schema/src/packages.ts:454`](https://github.com/openclaw/clawhub/blob/4c69f2af2e2d/packages/schema/src/packages.ts#L454), [4c69f2af2e2d](https://github.com/openclaw/clawhub/commit/4c69f2af2e2d))
