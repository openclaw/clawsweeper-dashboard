## Label Justifications

- `P1`: The unresolved mismatch prevents valid plugin installation for affected clients and remains observable on the live service.
- `impact:security`: The API can cause executable artifact verification to compare trusted integrity metadata with bytes from a different archive.

## Summary

Current main and the live service still expose divergent artifact behavior: the supported artifact download returns the stored 4,678-byte npm tarball, the legacy download returns a generated ZIP, and the unsupported `/tarball` suffix incorrectly returns version metadata with HTTP 200. The related issue covers legacy digest compatibility but does not fully own the unsupported route or source-provenance concern, so this issue remains necessary for maintainer judgment.

## What This Changes

Fix plugin installation integrity failures caused by artifact URLs, hashes, or source provenance describing different content than the bytes actually served.

## Best Possible Solution

Define a durable per-artifact contract: canonical npm-pack URLs and digests identify the exact stored `.tgz`, legacy ZIP responses publish a digest for those exact ZIP bytes or fail explicitly, unknown version suffixes return a non-success response, and source commits are supplied and validated rather than populated from unrelated hashes.

## Reproduction Assessment

Yes. Live requests on June 11, 2026 reproduce distinct tarball, legacy ZIP, and erroneous JSON responses, while current source explains the separate artifact-aware and legacy paths.

## Solution Assessment

Unclear as a final implementation choice. The integrity invariant and unsupported-route correction are well defined, but maintainers must explicitly choose the backward-compatible relationship among legacy `sha256hash`, generated ZIP downloads, canonical tarballs, and historical source metadata.
