## Label Justifications

- `P1`: The mismatch currently breaks plugin installation for affected clients and concerns a production artifact-distribution path.
- `impact:security`: Integrity and source-provenance metadata currently do not reliably identify the exact executable artifact bytes delivered through every supported path.

## Summary

Current production and current main still expose mismatched artifact identities: npm-pack metadata identifies the stored `.tgz`, while the legacy download endpoint generates different ZIP bytes. The related issue covers that compatibility invariant, but this report also identifies misleading source-commit provenance that remains visible and is not tracked there, so this issue is not fully superseded.

## What This Changes

Align package integrity metadata with the artifact bytes served to each installer and correct misleading source-commit provenance when publishing without an explicit commit.

## Best Possible Solution

Define and document separate canonical identities for the stored npm tarball and generated legacy ZIP, ensure each exposed digest matches its corresponding download bytes, and leave source provenance unavailable unless the publisher supplies or ClawHub independently resolves a verified repository commit.

## Reproduction Assessment

Yes. On June 11, 2026, the live legacy endpoint returned ZIP digest `1b36f01e…` while the same release's artifact endpoint reported npm-tarball digest `5cf67063…`, and the current-main handlers explain that split.

## Solution Assessment

No single mechanical fix is established yet. The narrowest maintainable solution is to preserve explicit tarball metadata for artifact-aware clients while making the legacy hash and download contract internally consistent, with a separate verified rule for source attribution.
