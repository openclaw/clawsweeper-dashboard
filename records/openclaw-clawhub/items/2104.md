## What This Changes

The issue reports that a package tarball-looking HTTP API URL returns bytes that do not match the package manifest digest while the CLI artifact download returns the expected artifact.

## Best Possible Solution

Define digest-safe package artifact routing with regression coverage for unsupported or aliased artifact subpaths, then handle the displayed source-commit trust semantics explicitly.

## Reproduction Assessment

Yes. The live URL still returns the reported JSON payload and digest while the explicit artifact download returns the expected tarball bytes, and current main's route structure explains the same behavior from source inspection.

## Solution Assessment

Unclear. The narrow code fix is probably to reject unknown /versions/{version}/... subpaths or intentionally alias /tarball, but maintainers should first choose the API contract and whether source provenance belongs in the same fix.
