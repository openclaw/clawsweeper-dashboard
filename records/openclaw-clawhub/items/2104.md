## Label Justifications

- `P1`: The mismatch currently prevents affected clients from installing otherwise valid plugins through ClawHub.
- `impact:security`: The issue concerns cryptographic artifact verification and provenance at the package supply-chain boundary.

## Summary

Current production, which matches the reviewed v0.21.0/main SHA, still exposes the npm-pack tarball hash as `sha256hash` while the legacy download returns different ZIP bytes, so the reported installation failure remains real. The related issue tracks the same compatibility defect, but this issue also contains distinct evidence about incorrect source-commit provenance and should not be closed automatically.

## What This Changes

Make package integrity metadata match the bytes served to each install path and ensure displayed source commits reflect the actual package source.

## Best Possible Solution

Define and test separate integrity contracts for artifact-aware npm tarballs and legacy ZIP downloads, expose a digest matching each returned byte stream, and repair or remove unverifiable source-commit metadata for affected historical releases.

## Reproduction Assessment

Yes. The live v0.21.0 deployment returns an npm-pack hash in version metadata and a different digest for the legacy ZIP download of the reported package, directly reproducing the failure condition.

## Solution Assessment

No final implementation is selected yet. The narrowest maintainable solution is server-side integrity consistency for both client generations, without weakening client verification, plus a separate decision on repairing historical provenance metadata.
