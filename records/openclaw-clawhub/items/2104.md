## Label Justifications

- `P1`: The production mismatch blocks plugin installation for clients using the legacy package path and remains reproducible today.
- `impact:security`: The issue concerns cryptographic artifact identity not matching downloaded bytes and potentially misleading source-provenance metadata.

## Summary

Current production still exposes incompatible artifact identities: version metadata identifies the stored npm-pack tarball, while the legacy install endpoint generates a different ZIP. The report also contains a distinct, still-visible source-provenance defect, so the related integrity issue does not fully supersede this issue.

## What This Changes

Ensure package integrity metadata matches each served artifact and package source verification records the actual source revision rather than an unrelated inferred commit.

## Best Possible Solution

Preserve distinct npm-pack and legacy ZIP identities, expose a ZIP-specific integrity value compatible with legacy clients, and record source provenance only from an explicit or correctly discovered package-repository revision.

## Reproduction Assessment

Yes. Live production responses reproduce the differing tarball and ZIP hashes and the suspect source commit, while current main exposes the corresponding server and publication paths.

## Solution Assessment

The integrity invariant proposed by https://github.com/openclaw/clawhub/issues/2378 is appropriate, but maintainers must select and validate the legacy compatibility strategy; the source-provenance defect also needs an explicit fix or canonical tracker.
