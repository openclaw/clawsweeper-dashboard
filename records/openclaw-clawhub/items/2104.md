## Label Justifications

- `P1`: This is a current install workflow failure for plugin packages with a supply-chain integrity dimension.
- `impact:security`: The issue concerns package artifact integrity and whether install clients receive bytes matching the advertised digest.

## Summary

Keep open: current main/v0.16.0 still returns 200 application/json for the reported `/versions/0.1.0/tarball` URL, and the source still routes unknown package version subpaths into version-detail JSON instead of artifact bytes or a clear miss.

## What This Changes

The issue reports that a package version `/tarball` HTTP API URL returns bytes with a different digest than the CLI-resolved artifact, breaking plugin installs with an integrity mismatch.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs either serve the exact stored artifact bytes through a documented compatibility alias or fail clearly without returning digest-mismatched JSON.

## Reproduction Assessment

Yes. The live v0.16.0/current-main endpoint still returns JSON for the reported artifact-looking URL, and source inspection shows the broad version-detail branch accepts the extra `tarball` segment.

## Solution Assessment

Unclear. Rejecting unknown subpaths is the narrow defensive fix, but a documented `/tarball` alias or redirect may be better if existing installers already depend on that URL shape.
