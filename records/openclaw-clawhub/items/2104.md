## Summary

Keep open: this is a ClawHub issue, and the report still maps to a live install-facing artifact-route mismatch on current main; cleanup policy only auto-closes ClawHub PRs when they are already implemented, not issues.

## What This Changes

The issue reports that an install-facing package tarball URL serves bytes that do not match the artifact digest returned by the ClawHub CLI/resolver path.

## Best Possible Solution

Make package artifact resolution unambiguous and digest-checked across install clients, with regression coverage for unsupported version subpaths and the canonical artifact download route.

## Reproduction Assessment

Yes. The hosted endpoints reproduce the byte and digest mismatch, and current-main source inspection shows a version-subpath fallthrough that can return metadata JSON for an archive-looking URL.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The maintainable fix is either to alias these tarball-looking paths to exact ClawPack artifact bytes or reject them clearly, then cover that behavior in route tests.
