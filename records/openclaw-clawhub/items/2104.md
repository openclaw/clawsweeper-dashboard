## Summary

Keep open: the hosted package endpoints still reproduce the reported artifact/digest split, and current main still has the version-subpath fallthrough that can return version metadata JSON for an artifact-looking URL. ClawHub policy keeps issues open, and this one is install-path and supply-chain relevant.

## What This Changes

The issue reports that a plugin install path fetches different artifact bytes than the ClawHub CLI’s verified artifact resolver, causing an integrity mismatch and showing questionable source commit provenance.

## Best Possible Solution

Make package artifact resolution unambiguous, keep digest-checked ClawPack tarball bytes as the install source of truth, add regression coverage for unsupported version subpaths, and correct or clarify source commit provenance.

## Reproduction Assessment

Yes. Hosted endpoint checks on May 9, 2026 reproduce different byte streams and digests for the tarball-looking path, explicit ClawPack artifact download, npm mirror tarball, and legacy ZIP route, and current-main source inspection explains the fallthrough.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The narrow maintainable fix is either to make unsupported version subpaths fail clearly or to intentionally route compatible tarball requests to the exact verified ClawPack artifact, with regression tests and a provenance cleanup path.
