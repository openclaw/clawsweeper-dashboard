## Label Justifications

- `P1`: Affected clients cannot install valid plugins because the integrity check compares metadata with different downloaded bytes.
- `impact:security`: The issue concerns whether integrity and provenance metadata authenticate the exact executable package artifact delivered to users.

## Summary

Keep this open: current main intentionally serves npm-pack tarball bytes to artifact-aware clients while retaining a generated ZIP legacy path, and it does not yet establish the reported invariant that each legacy integrity value authenticates the exact legacy bytes delivered. The narrower integrity work is also tracked in https://github.com/openclaw/clawhub/issues/2378, but this issue additionally raises source-commit provenance, so it is not a complete duplicate.

## What This Changes

Make ClawHub package metadata and HTTP downloads advertise integrity and source-provenance values that correspond to the exact artifact and source they describe.

## Best Possible Solution

Give the immutable npm-pack tarball and generated legacy ZIP separate explicit identities, guarantee that every advertised hash matches its associated download bytes, preserve old-client behavior with focused compatibility tests, and omit source-commit provenance unless it was supplied or verifiably derived from the package source.

## Reproduction Assessment

Yes at source level: current main exposes separate npm-pack tarball and legacy ZIP paths for one release, and the supplied endpoint hashes demonstrate the resulting mismatch. This review did not independently download the production artifact, so the status is source-reproducible rather than a fresh runtime reproduction.

## Solution Assessment

No, one ambiguous legacy hash should not represent both the stored npm tarball and generated ZIP. The safest narrow direction is separate artifact identities with backward-compatible legacy metadata and real HTTP/client validation for both paths.
