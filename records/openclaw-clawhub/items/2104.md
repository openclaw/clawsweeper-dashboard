## Label Justifications

- `P1`: Affected plugin installation workflows fail integrity verification even though the stored npm artifact itself is valid.
- `impact:security`: The report concerns cryptographic package integrity metadata being compared against bytes from a different artifact representation.

## Summary

Current main and production still reproduce the central artifact-identity problem: the reported version `tarball` URL returns version JSON rather than the advertised npm archive, while the legacy download produces a separately generated ZIP. The related open issue covers the legacy hash contract but not necessarily the broken `tarball` alias or source-provenance concern, so this issue remains necessary pending a coordinated maintainer decision.

## What This Changes

Ensure ClawHub package download URLs, integrity metadata, and source provenance consistently identify the artifact clients actually receive.

## Best Possible Solution

Make every documented or compatibility download URL return its named artifact, bind each integrity value to those exact bytes, retain explicit legacy ZIP metadata for old clients, and omit source-commit provenance unless it can be verified from the publisher's source repository.

## Reproduction Assessment

Yes. Production currently returns JSON from the reported `tarball` URL, a correct 4,678-byte archive from the canonical npm URL, and a differently hashed ZIP from legacy `/download`, providing a high-confidence live reproduction.

## Solution Assessment

No, treating this only as registry-mirror repacking is not the best diagnosis. The maintainable solution must define compatible integrity semantics for three observable surfaces: the version route, canonical npm artifact, and generated legacy ZIP.
