## Summary

Keep open: this is still a reproduced install-integrity bug, and current main still falls through from a tarball-looking package version subpath to JSON metadata instead of artifact bytes or a clear unsupported-route response.

## What This Changes

The issue reports that an install-facing package tarball URL returns bytes whose digest does not match the advertised package artifact.

## Best Possible Solution

Define one digest-safe package artifact contract, make ambiguous version subpaths either stream the advertised artifact or fail clearly, and add regression coverage for artifact URL semantics plus source provenance handling.

## Reproduction Assessment

Yes. The live reported URL returns JSON bytes with the reported mismatched digest, while the explicit artifact download returns the expected 4,678-byte artifact; current main’s route source explains the fallthrough.

## Solution Assessment

Unclear. A narrow code fix is likely, but maintainers should first choose whether /versions/{version}/tarball is an artifact alias or an unsupported path, and separately define the sourceCommit fallback policy.
