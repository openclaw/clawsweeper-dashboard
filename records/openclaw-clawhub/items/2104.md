## Summary

Keep open: the hosted API still reproduces the reported artifact mismatch, and current main still has a broad version route that can treat extra subpaths like `tarball` as version metadata instead of an artifact response. The fix needs maintainer choice because the documented resolver, legacy ZIP route, and undocumented compatibility path currently imply different contracts.

## What This Changes

The issue asks ClawHub to eliminate plugin install integrity mismatches where HTTP API tarball or legacy download paths return bytes that differ from the CLI-resolved ClawPack artifact.

## Best Possible Solution

Make package artifact resolution unambiguous: installers should consume `/versions/{version}/artifact` and its `downloadUrl`, while ClawHub either rejects unknown tarball subpaths or intentionally maps them to the exact ClawPack artifact, with regression coverage and source metadata validation.

## Reproduction Assessment

Yes. The hosted API returns the reported mismatched hashes today, and current-main source inspection shows why `/versions/{version}/tarball` can fall through to version JSON instead of an artifact response.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The narrow maintainable direction is to keep the explicit artifact resolver as source of truth and add deliberate behavior for unknown or legacy tarball paths.
