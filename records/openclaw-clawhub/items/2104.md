## Label Justifications

- `P1`: The mismatch blocks otherwise valid plugin installations on supported client paths and affects a core registry trust boundary.
- `impact:security`: Executable plugin bytes can be compared against integrity metadata describing a different archive.

## Summary

Current production still serves two different archives for the same release, and the legacy version metadata does not provide a ZIP-specific integrity value; the related issue covers that compatibility defect, while this issue also preserves a distinct source-provenance concern.

## What This Changes

Correct package integrity and source-provenance mismatches where HTTP downloads, CLI resolution, and metadata can identify different archives or commits.

## Best Possible Solution

Implement one coordinated package API correction that exposes a digest for each endpoint’s exact bytes, proves both legacy and artifact-aware installation paths, and records source provenance only when an actual repository commit is supplied.

## Reproduction Assessment

Yes. A current production check reproduces distinct tarball and ZIP hashes for the same release, and current-main source shows the stored release hash and generated legacy ZIP are not bound to one artifact identity.

## Solution Assessment

The exact-byte, endpoint-specific digest invariant is the best fix; however, maintainers must choose and validate the backward-compatible `sha256hash` behavior and separately resolve the source-commit fallback.
