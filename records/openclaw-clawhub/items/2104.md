- `impact:security`: The issue is about package artifact integrity and source provenance for executable plugin artifacts.

## Summary

Keep this open: the exact ClawPack artifact path now works, but the reported installer path still gets a successful response with bytes that do not match the manifest, and the legacy ZIP-vs-tarball contract needs maintainer/security judgment before changing public API behavior.

## What This Changes

The issue reports that a plugin install path fetches bytes whose digest does not match the published manifest because the HTTP API path serves a different artifact from the CLI download path.

## Best Possible Solution

Set one public artifact contract for ClawPack installs: exact bytes should come from /artifact/download or the npm mirror, unknown version subpaths should not return successful metadata bytes, and any bad published provenance should be corrected through a maintainer-approved moderation/backfill path.

## Reproduction Assessment

Yes. Live production still returns different successful bodies for the exact artifact path, legacy download path, and reported /tarball path, and current source explains the /tarball fallthrough through the broad versions/{version} route.

## Solution Assessment

Unclear. The narrow technical fix is small, but the best public behavior depends on a maintainer decision between strict 404, redirecting /tarball to the exact artifact, or requiring external clients to update to /artifact/download.
