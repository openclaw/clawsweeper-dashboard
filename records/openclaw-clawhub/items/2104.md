## Summary

Keep open: the reported mismatch still reproduces on the live API, and current main still has the broad package-version route that explains why an artifact-looking subpath can return metadata JSON. This is also a ClawHub issue, and repository cleanup policy keeps issues open for maintainer handling.

## What This Changes

The issue reports that the HTTP package tarball path for `zerogpu-router@0.1.0` serves bytes with a different SHA-256 digest than the CLI-downloaded artifact, causing installs through that HTTP path to fail integrity verification.

## Best Possible Solution

Define a digest-safe package artifact route contract with regression coverage for unsupported or aliased artifact subpaths, then handle source-commit provenance display as an explicit trust-policy decision.

## Reproduction Assessment

Yes. The live ClawHub API still returns the reported mismatched JSON and artifact digests, and current main’s route structure provides a high-confidence source-level reproduction path.

## Solution Assessment

Unclear. The narrow code fix is probably to reject unknown `/versions/{version}/...` subpaths or intentionally alias `/tarball`, but maintainers should first choose the API contract and decide whether source provenance belongs in the same fix.
