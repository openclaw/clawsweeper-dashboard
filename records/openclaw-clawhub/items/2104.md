## Summary

Keep open: the hosted API still reproduces the artifact mismatch, and current main still has route behavior that lets a tarball-looking version subpath return version JSON instead of unambiguous artifact bytes or a clear miss. This is supply-chain relevant and needs maintainer direction on the compatibility contract.

## What This Changes

The issue reports that a plugin install path fetches package bytes from an HTTP API endpoint whose size and digest differ from the published ClawPack artifact.

## Best Possible Solution

Make package artifact resolution unambiguous: keep the explicit artifact resolver as source of truth, add regression coverage for unknown or legacy tarball-looking subpaths, and tighten source metadata publishing so explicit repo information cannot be paired with an unrelated inferred commit.

## Reproduction Assessment

Yes. The hosted API returns different byte streams and digests for the tarball-looking path, the explicit artifact download, and the legacy ZIP path; current-main source inspection shows why the extra `tarball` segment falls through to version metadata.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The narrow maintainable direction is to make unsupported subpaths fail clearly or deliberately stream/redirect to the exact ClawPack artifact, with tests for both resolver and legacy ZIP behavior.
