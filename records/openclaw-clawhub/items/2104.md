## Summary

Keep this open: current main and live production still reproduce the artifact-integrity split. The explicit artifact resolver serves the npm-pack tarball, legacy `/download` serves a generated ZIP with a different digest, and the unsupported `/versions/{version}/tarball` suffix still returns version JSON instead of artifact bytes or a clear failure.

## What This Changes

The issue reports that ClawHub package install paths can fetch different artifacts and digests for the same plugin version, causing integrity verification failures.

## Best Possible Solution

Define one public artifact integrity contract: artifact-aware clients get exact npm-pack bytes, legacy ZIP responses are internally consistent or fail clearly, unsupported suffixes do not return successful JSON bodies, and any existing bad provenance is repaired through an approved backfill or moderation path.

## Reproduction Assessment

Yes. Live production still returns different successful bodies and SHA-256 digests for the explicit artifact path, legacy download path, and unsupported tarball suffix, and current main source explains the same routing split.

## Solution Assessment

Unclear. The diagnosis is strong, but the durable fix needs maintainer/security choice between fail-closed routing, redirect or alias behavior, legacy digest semantics, and existing-row provenance repair.
