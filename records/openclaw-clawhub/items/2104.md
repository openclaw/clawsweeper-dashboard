## Summary

Keep open: current production still reproduces the reported package-artifact digest split, and current main still routes extra `/versions/{version}/...` subpaths to version metadata JSON instead of artifact bytes or a clear unsupported-route error.

## What This Changes

The issue asks ClawHub to make plugin package install/artifact HTTP paths return bytes that match the advertised artifact integrity and to correct confusing source-commit provenance.

## Best Possible Solution

Make package artifact resolution unambiguous and digest-safe for install clients, then fix or clearly document source provenance rules for published plugin metadata.

## Reproduction Assessment

Yes. Current production, which contains current main, still returns different byte streams and digests for the tarball-looking API path versus the supported artifact download path, and source inspection shows why the extra subpath reaches metadata JSON.

## Solution Assessment

Unclear. The narrow fix is source-local, but maintainers should first choose whether the unsupported tarball-looking route becomes a compatibility alias or a hard 404/400 so install clients fail clearly.
