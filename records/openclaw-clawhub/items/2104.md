## Summary

Keep open: this is a live install integrity bug in a supply-chain-sensitive package artifact path, and current main still accepts a tarball-looking version subpath as metadata rather than serving artifact bytes or failing clearly.

## What This Changes

The issue reports that the package HTTP API tarball path returns bytes with a different digest than the artifact advertised by the CLI/resolver for `zerogpu-router` 0.1.0.

## Best Possible Solution

Define one digest-safe artifact contract for package installs, make ambiguous version subpaths either stream the advertised artifact or fail clearly, and add regression coverage for artifact URL semantics plus source provenance handling.

## Reproduction Assessment

Yes. The live reported tarball-looking URL returns 5,332 bytes of JSON with the mismatched digest, while the explicit artifact download returns the 4,678-byte package artifact with the expected digest.

## Solution Assessment

Unclear. A narrow code fix is likely, but maintainers should first choose whether `/versions/{version}/tarball` is a supported artifact alias or an invalid route, and define the sourceCommit fallback policy separately.
