## Summary

Keep open: current production still reproduces the artifact digest split, and current main still lets `/versions/{version}/tarball` fall through to version metadata JSON instead of exact artifact bytes or a clear unsupported-route response.

## What This Changes

The issue reports that a tarball-looking package HTTP API path returns different bytes and SHA-256 than the ClawHub CLI-supported artifact download, with a secondary concern about source commit provenance.

## Best Possible Solution

Make package artifact resolution unambiguous and digest-safe for install clients, then tighten or document source provenance rules for published plugin metadata.

## Reproduction Assessment

Yes. Current production at the current main deployment returns different byte streams and SHA-256 digests for the tarball-looking API path versus the supported artifact download and npm mirror paths, and source inspection explains the fallthrough.

## Solution Assessment

Unclear. The route should not serve unrelated bytes, but maintainers should first decide whether the unsupported tarball-looking path should alias to exact ClawPack bytes or fail clearly.
