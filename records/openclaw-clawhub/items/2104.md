## Summary

Keep open: current production still reproduces the package artifact digest split, and current main still accepts extra `/versions/{version}/...` subpaths as version metadata instead of artifact bytes or a clear unsupported-route response.

## What This Changes

The issue reports that an install-facing package URL returns bytes and digests that differ from the ClawHub CLI artifact, plus possibly misleading source commit metadata.

## Best Possible Solution

Make package artifact resolution unambiguous and digest-safe for install clients, then fix or document the source provenance rules for published plugin metadata.

## Reproduction Assessment

Yes. Current production, which contains current main, still returns different byte streams and digests for the tarball-looking API path versus the supported artifact download path, and source inspection shows the route fallthrough.

## Solution Assessment

Unclear. The source-local repair is narrow, but maintainers should first choose whether the unsupported tarball-looking route becomes a compatibility alias or a hard 404/400.
