## Summary

Keep open: live checks and current source still show the mismatched install-facing artifact path, and the route/provenance policy needs maintainer security/product judgment before code changes.

## What This Changes

The issue reports that a tarball-looking package HTTP API URL returns bytes whose SHA-256 does not match the advertised `zerogpu-router` artifact while the CLI download path returns the expected artifact.

## Best Possible Solution

Define a digest-safe package artifact contract, make ambiguous version subpaths either stream the advertised artifact or reject clearly, and add regression coverage for artifact URL semantics plus source provenance handling.

## Reproduction Assessment

Yes. The live URL still returns 5,332 bytes of JSON with the reporter’s mismatched digest while the explicit artifact download returns the expected 4,678-byte tarball, and current-main source shows the route fallthrough that causes it.

## Solution Assessment

Unclear. A narrow fix is likely, but maintainers should first choose whether `/versions/{version}/tarball` is a supported artifact alias or an invalid route, and define sourceCommit fallback policy separately.
