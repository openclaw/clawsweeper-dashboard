## Summary

Keep open: the hosted mismatch is reproducible, and current main still has a broad package-version route that treats extra subpaths like `tarball` as version metadata instead of exact artifact bytes. The fix should wait for maintainer choice on the compatibility contract for tarball-looking paths and legacy ZIP behavior.

## What This Changes

The issue reports a ClawPack plugin install integrity mismatch where HTTP package paths serve bytes different from the manifest-backed CLI resolver artifact, plus misleading source commit metadata.

## Best Possible Solution

Make package artifact resolution unambiguous: keep the resolver path as the source of truth, either reject or intentionally alias tarball-looking subpaths to exact artifact bytes, and prevent explicit source repos from being paired with unrelated inferred commits.

## Reproduction Assessment

Yes. The hosted API currently returns different byte streams and hashes for the reported tarball-looking path, exact artifact path, npm mirror path, and legacy ZIP path; current-main source inspection explains the `/versions/{version}/tarball` fallthrough.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The narrow maintainable fix is to make unknown/legacy tarball paths explicit instead of letting them return unrelated metadata or ZIP bytes.
