## Summary

Keep open: the live API still reproduces the artifact-looking `/tarball` path returning JSON bytes with a different digest from the explicit artifact download, current main still has the broad version-detail route fallthrough, and no canonical fixing PR or issue supersedes this report.

## What This Changes

The issue reports that `/api/v1/packages/zerogpu-router/versions/0.1.0/tarball` returns bytes that fail the expected package artifact integrity while the CLI/explicit artifact path returns the correct tarball, with a secondary source-commit metadata concern.

## Best Possible Solution

Make package artifact routes digest-safe by ensuring unknown version subpaths do not return metadata bytes, then separately correct or document the bad source-commit provenance for the affected release.

## Reproduction Assessment

Yes. Curling the live `/versions/0.1.0/tarball` URL returns JSON bytes with a different digest from the explicit artifact download, and current main still contains the broad version-detail route that explains the fallthrough.

## Solution Assessment

Unclear. Rejecting unknown subpaths is the narrowest defensive fix, but maintainers should choose the public API contract before deciding between 404, redirect, or a supported `/tarball` alias.
