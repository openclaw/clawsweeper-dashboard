## Summary

Keep open: the reported integrity mismatch is reproducible on the hosted API, and current main still has a package version route that can treat extra subpaths like `tarball` as ordinary version metadata instead of artifact bytes. The fix needs maintainer choice because documented resolver downloads, legacy ZIP downloads, and undocumented compatibility paths currently have different contracts.

## What This Changes

The issue asks ClawHub to make HTTP package artifact downloads match the CLI-resolved ClawPack artifact and to avoid misleading source commit metadata.

## Best Possible Solution

Make package artifact resolution unambiguous: installers should consume `/versions/{version}/artifact` and its `downloadUrl`, while ClawHub should either reject legacy/unknown tarball subpaths clearly or intentionally redirect/stream the exact ClawPack artifact, with regression tests and source metadata validation.

## Reproduction Assessment

Yes for the hosted API: the reported paths return different byte streams and hashes today. Current-main source inspection also explains the `/versions/{version}/tarball` fallthrough, though I did not execute the local handler in this read-only review.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The narrow maintainable direction is to keep the artifact resolver as the source of truth and add explicit behavior for unknown or legacy tarball paths instead of letting them return unrelated metadata or ZIP bytes.
