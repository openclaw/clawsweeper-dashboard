## Summary

Keep open: the live API and current source still show the artifact-looking `/tarball` path returning version JSON instead of tarball bytes, no merged canonical fix owns this exact problem, and ClawHub issue policy keeps valid issues open for maintainer handling.

## What This Changes

The issue reports that an artifact-looking package HTTP API path returns bytes that differ from the CLI-resolved tarball, causing install-time integrity verification to fail for `zerogpu-router`.

## Best Possible Solution

Define a digest-safe package artifact route contract with regression coverage for unsupported or aliased artifact subpaths, then handle source-commit provenance display as an explicit trust-policy decision.

## Reproduction Assessment

Yes. The live ClawHub API still returns metadata JSON from the reported tarball-looking path while the explicit artifact routes return the expected tarball bytes, and current main’s route structure explains the same fallthrough.

## Solution Assessment

Unclear. The narrow code change is likely to reject unknown `/versions/{version}/...` subpaths or intentionally alias/redirect `/tarball`, but maintainers should first choose the API contract and decide whether source provenance belongs in the same fix.
