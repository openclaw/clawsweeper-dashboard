## What This Changes

The issue reports package install integrity failure because an artifact-looking HTTP API tarball path returns bytes that do not match the CLI-resolved package artifact.

## Best Possible Solution

Make package artifact routes digest-safe by rejecting unsupported version subpaths or serving exact artifact bytes through an explicit compatibility alias, then separately handle the reported release provenance mismatch.

## Reproduction Assessment

Yes. Live production still returns JSON bytes with a different digest for the reported `/tarball` URL, and current main still contains the route fallthrough that accepts that extra subpath.

## Solution Assessment

Unclear. Rejecting unknown subpaths is the narrowest defensive fix, but maintainers should decide whether the public contract should be 404, redirect, or a supported tarball alias.
