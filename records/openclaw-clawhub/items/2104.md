## What This Changes

The issue asks ClawHub to fix a package install integrity mismatch where an artifact-looking HTTP API subpath returns different bytes than the CLI-resolved artifact.

## Best Possible Solution

Make package artifact routes digest-safe by rejecting unknown version subpaths or explicitly serving exact artifact bytes, then handle affected release provenance separately.

## Reproduction Assessment

Yes. Live production still returns JSON bytes with a different digest for the reported /tarball URL, and current main contains the broad version-detail route that accepts the extra subpath.

## Solution Assessment

Unclear. The narrowest defensive fix is to stop unknown version subpaths from returning version JSON, but maintainers should decide the public route contract before choosing 404, redirect, or a supported tarball alias.
