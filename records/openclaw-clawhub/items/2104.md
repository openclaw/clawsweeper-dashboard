## Summary

Keep open: the reported integrity mismatch is reproducible against the live API and explained by current main routing; ClawHub cleanup policy keeps issues open, and this is a narrow repair candidate.

## What This Changes

The issue reports that a tarball-looking package API URL returns bytes that do not match the package artifact manifest, causing plugin install integrity checks to fail.

## Best Possible Solution

Make tarball-like package artifact URLs resolve explicitly to the exact stored ClawPack bytes, and tighten the generic version metadata route so unknown suffixes cannot return JSON with a 200 status.

## Reproduction Assessment

Yes for the live public API and source path. I did not run a local server, so this is marked source-reproducible rather than fully reproduced against a local current-main deployment.

## Solution Assessment

Yes. The narrow maintainable fix is in the package HTTP router: make `/versions/{version}` exact, keep `/artifact` and `/artifact/download` behavior, and route `/tarball` explicitly to the existing ClawPack stream helper or a clear non-200 error.

## Repair Work Prompt

Fix the package artifact route mismatch reported in https://github.com/openclaw/clawhub/issues/2104. Current main lets `GET /api/v1/packages/{name}/versions/{version}/tarball` fall through to the generic version metadata JSON response because the `/versions/{version}` branch in `convex/httpApiV1/packagesV1.ts` does not require an exact segment count. Add a focused router fix so `/versions/{version}` only returns metadata for the exact route, `/versions/{version}/artifact` keeps returning resolver metadata, `/versions/{version}/artifact/download` keeps streaming exact artifact bytes, and `/versions/{version}/tarball` no longer returns JSON metadata. Prefer making `/tarball` a compatibility alias for npm-pack releases by delegating to the existing `streamClawPackRelease` helper, preserving rate-limit bucket, security blocking, content-disposition, ETag, SHA-256, npm integrity, and shasum headers; if legacy ZIP support is needed, redirect to the legacy download URL consistently with `/artifact/download`. Add handler regression coverage proving the tarball-looking URL returns exact ClawPack bytes or a clear non-200 response, and add an assertion that unknown extra suffixes under `/versions/{version}` do not return metadata JSON. Do not change publish/sourceCommit attribution in this PR unless a separate failing test is added and maintainer scope is explicit. Update `docs/http-api.md` only if `/tarball` becomes a supported alias.
