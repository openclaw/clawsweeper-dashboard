## Summary

Keep open: the reported mismatch still reproduces on production and current main still has the permissive version-route fallthrough, so this needs maintainer follow-up rather than cleanup closure.

## What This Changes

The issue reports that an artifact-looking package HTTP route returns bytes whose SHA-256 does not match the manifest-backed CLI artifact download.

## Best Possible Solution

Make package artifact routes digest-safe by preventing unknown version subpaths from returning metadata bytes, then separately correct or document the bad source-commit provenance for the affected release.

## Reproduction Assessment

Yes. Live production reproduces the mismatched bytes and current main has the route fallthrough that explains the same behavior for `/versions/{version}/tarball`.

## Solution Assessment

Unclear. Rejecting unknown version subpaths is the narrowest defensive fix, but maintainers should choose whether the public contract is 404, redirect, or a supported alias.
