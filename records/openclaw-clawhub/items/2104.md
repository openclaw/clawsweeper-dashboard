## Summary

Keep open. The report is valid: current production and current main still distinguish the supported ClawPack resolver/npm routes from legacy or unknown paths, and clients using the reported HTTP path can receive non-tarball bytes that fail integrity verification.

## What This Changes

The issue reports that a plugin install path gets bytes whose hash does not match the ClawPack artifact used by the CLI, plus misleading source-commit metadata on the published package.

## Best Possible Solution

Add an explicit compatibility decision for the `tarball` path, keep legacy ZIP and ClawPack tarball routes unambiguous, and repair the affected release metadata if maintainers confirm the source commit is wrong.

## Reproduction Assessment

Yes. Live curl checks reproduce the mismatched bytes, and current main explains it: the supported resolver/npm routes serve the exact 4678-byte ClawPack tarball while the reported `tarball` suffix falls through to JSON and the legacy download path serves ZIP bytes.

## Solution Assessment

Unclear. The narrow code shape is clear, but the best permanent behavior needs maintainer choice: support the undocumented `tarball` alias for compatibility or reject it and require clients to use `/artifact` or `/api/npm`.
