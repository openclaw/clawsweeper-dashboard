## Summary

Keep open: this is a current install-facing artifact mismatch with supply-chain implications, and ClawHub policy keeps issues open rather than auto-closing them. Hosted checks still reproduce different bytes and digests, and current main still has a version-subpath fallthrough that lets a tarball-looking URL return version metadata JSON instead of artifact bytes.

## What This Changes

The issue reports that a plugin install path receives HTTP API artifact bytes that do not match the published manifest digest, while the CLI package download path receives the expected artifact.

## Best Possible Solution

Make package artifact resolution unambiguous and digest-checked across install-facing routes, with regression tests for the chosen compatibility behavior and a separate provenance correction for source commit metadata.

## Reproduction Assessment

Yes. Hosted endpoint checks reproduce three different byte streams and digests for the reported package/version, and current-main source inspection explains the tarball-looking subpath fallthrough.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The narrow maintainable fix is either to alias the tarball-looking route to the exact ClawPack artifact bytes or to reject unsupported version subpaths clearly, then cover that choice in tests.
