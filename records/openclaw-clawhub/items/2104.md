## Label Justifications

- `P1`: The currently reproducible integrity mismatch prevents affected clients from installing otherwise valid plugin releases.
- `impact:security`: Cryptographic package integrity metadata identifies different bytes from the archive returned through the legacy install path.

## Summary

Production and current main still reproduce the legacy integrity mismatch: artifact-aware downloads return the stored npm tgz correctly, while the legacy download returns a generated ZIP but version metadata advertises the tgz hash. The related issue narrows that compatibility defect, but this issue also contains a distinct source-commit provenance concern, so it should remain open pending maintainer consolidation and contract decisions.

## What This Changes

Make ClawHub package integrity metadata match the exact artifact served to each client and avoid presenting fallback values as verified source commits.

## Best Possible Solution

Define and document separate legacy ZIP and npm-pack artifact identities so each integrity value matches its associated download bytes, preserve current artifact-aware verification, and only expose source commits as verified provenance when they were explicitly supplied or reliably inferred.

## Reproduction Assessment

Yes. On June 11, 2026, production returned a tgz hash of `5cf670…b01b90d` from `/artifact/download` and a ZIP hash of `1b36f0…b1276f` from legacy `/download`, while version metadata exposed the tgz hash as `sha256hash`.

## Solution Assessment

No, treating the metadata-like `/tarball` URL as the archive endpoint is not the narrowest framing. The maintainable solution is the explicit legacy compatibility invariant tracked by https://github.com/openclaw/clawhub/issues/2378, combined with separate validation of source-commit provenance.
