## Summary

Keep open: current production still reproduces the reported package artifact digest split, and current main still accepts an unsupported `/versions/{version}/tarball` suffix as version metadata instead of exact artifact bytes or a clear route error.

## What This Changes

The reporter asks ClawHub to resolve a package install integrity mismatch where a tarball-looking HTTP API URL returns different bytes than the CLI/explicit ClawPack artifact download, with related source-commit provenance cleanup.

## Best Possible Solution

Make package artifact resolution unambiguous and digest-safe for install clients, with route tests/docs for the chosen behavior and a clear source provenance rule for publishes.

## Reproduction Assessment

Yes. Current production on the current-main prod tag returns different byte streams and SHA-256 digests for the tarball-looking path, explicit artifact download, and legacy ZIP route, and source inspection explains the fallthrough.

## Solution Assessment

Unclear. The code surface is narrow, but maintainers should first decide the supported route contract for `/versions/{version}/tarball` and the strictness of source provenance metadata.
