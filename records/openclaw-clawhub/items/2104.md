## Summary

Keep open: the live production mismatch still reproduces, current `main` still has the broad version-detail route that explains it, and the item needs maintainer handling because it touches install integrity and route contract behavior.

## What This Changes

The issue reports that an artifact-looking package tarball URL returns different bytes than the CLI-supported artifact download path, causing install integrity verification to fail.

## Best Possible Solution

Make package artifact routes digest-safe by rejecting unknown version subpaths or explicitly serving exact artifact bytes, then handle the reported release provenance mismatch separately.

## Reproduction Assessment

Yes. Live production still returns HTTP 200 JSON for the reported `/tarball` URL with a different SHA-256 than the supported artifact download endpoints, and current `main` contains the broad route branch that explains it.

## Solution Assessment

Unclear. The narrowest defensive fix is to stop unknown version subpaths from returning version JSON, but maintainers should choose the public route contract before deciding between 404, redirect, or a supported tarball alias.
