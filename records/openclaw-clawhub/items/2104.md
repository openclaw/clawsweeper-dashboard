## Summary

Keep this open: live production still returns mismatched successful bodies for the affected package paths, and current main still has the route/metadata split that explains the install-time integrity failure. The related issue at https://github.com/openclaw/clawhub/issues/2378 covers the narrower legacy `sha256hash` contract, but this issue also carries the unsupported `/tarball` fallthrough and source-provenance concern.

## What This Changes

The issue asks ClawHub to make package HTTP artifact/install paths expose consistent bytes, hashes, and source provenance for plugin installs.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from `/artifact/download` or the npm mirror, legacy `/download` remains internally consistent as ZIP compatibility or fails clearly, unsupported version suffixes do not return successful metadata bytes, and existing bad provenance is repaired through an approved moderation or backfill path.

## Reproduction Assessment

Yes. Live production reproduces the mismatched successful bodies, and current main source explains the `/tarball` fallthrough plus ZIP-only legacy route; I did not run a local Convex fixture, so this is source-reproducible rather than a local current-main fixture reproduction.

## Solution Assessment

Unclear. The bug is concrete, but the permanent behavior needs maintainer choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.
