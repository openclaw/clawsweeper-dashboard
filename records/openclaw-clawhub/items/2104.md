## Summary

Keep open: current main still lets an artifact-looking `/versions/{version}/tarball` package URL fall through to version-detail JSON rather than exact artifact bytes or a clear failure, matching the reported integrity-mismatch failure mode.

## What This Changes

The issue reports that a package tarball HTTP URL returns bytes that differ from the manifest/CLI artifact, causing plugin install integrity failures.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs either stream or redirect to the exact stored artifact through an intentional compatibility alias, or fail clearly without returning digest-mismatched JSON; handle the source commit metadata as a separate provenance fix if it remains wrong.

## Reproduction Assessment

Yes, at source level: the router preserves `tarball` as an extra segment and the broad version-detail branch accepts it as JSON. I did not run production downloads during this read-only review, but the issue body and prior ClawSweeper comment provide live mismatch evidence.

## Solution Assessment

Unclear. A fail-closed length check is the narrowest defensive fix, but an explicit `/tarball` compatibility alias may be the better product choice because an installer is already using that shape.
