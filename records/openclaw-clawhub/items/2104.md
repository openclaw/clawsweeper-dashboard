## Summary

Keep this open: current `main` still leaves the reported artifact-contract split in place, and the related report only owns the narrower legacy `sha256hash` versus `/download` invariant.

## What This Changes

The issue reports that ClawHub HTTP package install paths can serve artifact bytes whose digest differs from the CLI-downloaded artifact and published metadata.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from `/artifact/download` or the npm mirror, legacy `/download` is internally consistent or fails clearly for npm-pack versions, and unsupported suffixes such as `/tarball` do not return successful metadata bytes.

## Reproduction Assessment

Yes. Live production still returns successful but different bodies for the affected `/tarball`, explicit artifact download, and legacy download paths, and current `main` source explains the same route split and fallthrough.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The route strictness piece is narrow, but the permanent fix needs a maintainer/security choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.
