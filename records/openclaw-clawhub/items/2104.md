## Label Justifications

- `P1`: A plugin install integrity mismatch breaks a real package install workflow and affects trust in executable plugin distribution.
- `impact:security`: The issue concerns executable package artifact integrity and source provenance disagreeing across public install paths.

## Summary

Keep this open: production at the current main deploy still returns mismatched successful bodies for the affected package, and current main still has the route and metadata split that explains the install-time integrity failure. The related issue at https://github.com/openclaw/clawhub/issues/2378 covers the narrower legacy `sha256hash` contract, but this issue also carries the unsupported `/tarball` success response and source-provenance repair concern.

## What This Changes

The issue asks ClawHub package HTTP APIs to serve and describe plugin artifacts consistently so install clients do not verify one artifact while downloading another.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from `/artifact/download` or the npm mirror, legacy `/download` is internally consistent or fails clearly, unsupported suffixes do not return successful metadata bytes, and bad provenance is repaired through an approved backfill or moderation path.

## Reproduction Assessment

Yes. Live production at the current deployed main reproduces mismatched successful bodies for the affected package paths, and source at `963b0a5719431a182bef07ed1cd4ff09cc6fc1c9` explains the `/tarball` fallthrough and ZIP-only legacy route.

## Solution Assessment

Unclear. The reported invariant is valid, but the permanent fix needs maintainer choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.
