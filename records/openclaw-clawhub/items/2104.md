## Summary

Keep open: the live API still returns JSON from the artifact-looking `/tarball` path while the explicit artifact resolver streams the expected tarball bytes, and current main still has the broad version route that explains the mismatch. This is a security-sensitive ClawHub issue, and repository cleanup policy keeps ClawHub issues open for maintainer handling.

## What This Changes

The issue reports that an install-facing package tarball URL serves bytes whose digest differs from the CLI/manifest artifact, with a secondary source-commit provenance mismatch.

## Best Possible Solution

Define a digest-safe package artifact route contract with regression coverage for unsupported or aliased artifact subpaths, then handle source-commit provenance display as an explicit trust-policy decision.

## Reproduction Assessment

Yes. The live ClawHub API still returns metadata JSON from the reported tarball-looking path while the explicit artifact route returns the expected tarball bytes, and current main’s route structure provides a high-confidence source-level reproduction path.

## Solution Assessment

Unclear. The narrow code fix is probably to reject unknown `/versions/{version}/...` subpaths or intentionally alias/redirect `/tarball`, but maintainers should first choose the API contract and decide whether source provenance belongs in the same fix.
