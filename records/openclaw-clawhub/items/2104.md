## Summary

Keep open: ClawHub policy keeps issues open, and the reported install-facing artifact mismatch is reproducible against the hosted endpoints and source-reproducible on current main. The current package API still has separate exact ClawPack artifact routes, a legacy ZIP route, and a tarball-looking version subpath that falls through to JSON metadata.

## What This Changes

The issue reports that a plugin install path can receive HTTP API bytes whose digest differs from the manifest and from the ClawHub CLI download path.

## Best Possible Solution

Make package artifact resolution unambiguous: install clients should receive digest-checked ClawPack bytes from the resolver/npm artifact route or a clear unsupported-route error, with regression coverage for the chosen route contract and source-provenance behavior.

## Reproduction Assessment

Yes. The hosted endpoint checks reproduce the byte and digest split, and current-main source inspection shows the ambiguous route path that makes an archive-looking version subpath return metadata JSON instead of artifact bytes.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. The narrow maintainable fix is either to alias tarball-looking version paths to exact ClawPack artifact bytes or to reject unsupported subpaths clearly, then cover that behavior in tests.
