## Label Justifications

- `P1`: A real install workflow can fail integrity verification when an artifact-looking package URL returns bytes different from the manifest-backed artifact.
- `impact:security`: Package artifact URLs and digest verification are a supply-chain security boundary for installable plugin code.

## Summary

Live production still reproduces the byte mismatch, current main still routes the reported `/tarball` suffix into JSON version-detail handling, and no canonical PR or duplicate issue owns the remaining fix, so this should stay open for maintainer/security follow-up.

## What This Changes

Issue reports that an artifact-looking package tarball HTTP URL returns bytes different from the manifest-backed CLI artifact, causing plugin install integrity verification to fail.

## Best Possible Solution

Constrain package version subpaths so artifact-looking URLs either serve or redirect to the exact stored artifact bytes through an intentional compatibility route, or fail clearly without returning digest-mismatched JSON.

## Reproduction Assessment

Yes at source level, with live production confirmation. Current main has a source-level reproduction path, and live production still returns JSON bytes for `/versions/0.1.0/tarball` while the npm mirror tarball returns the correct gzip artifact bytes.

## Solution Assessment

Unclear. Rejecting unknown version subpaths is the narrow defensive fix, but serving or redirecting `/tarball` to exact artifact bytes may be the better compatibility choice if installers already depend on that shape.
