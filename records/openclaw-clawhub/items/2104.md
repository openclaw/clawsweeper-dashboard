## Summary

Keep open: the artifact mismatch remains reproducible and current main still lets the tarball-looking version subpath fall through to version metadata instead of exact artifact bytes or a clear unsupported-route response.

## What This Changes

The issue reports that a package install URL returns bytes with a different digest than the CLI-resolved ClawPack artifact for `zerogpu-router@0.1.0`.

## Best Possible Solution

Make package artifact resolution unambiguous and digest-safe for install clients, with regression tests/docs for the chosen route behavior and a clear source provenance rule for publishes.

## Reproduction Assessment

Yes. Production currently returns different byte streams and digests for the tarball-looking path, explicit artifact download, npm mirror, and legacy ZIP route, and current main source explains the `/tarball` fallthrough.

## Solution Assessment

Unclear. The explicit artifact resolver is the maintainable base, but maintainers should decide whether the tarball-looking path is a compatibility alias or an unsupported route before a repair PR changes install-facing behavior.
