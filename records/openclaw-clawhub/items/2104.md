## Summary

Keep open: current main and live ClawHub still expose an npm-pack digest through the legacy `sha256hash` field while the compatibility `/download` route returns a separately generated ZIP, so older clients can compare the digest and bytes of different artifacts. The open narrower report at https://github.com/openclaw/clawhub/issues/2378 confirms that the remaining work needs an explicit compatibility and security decision rather than cleanup.

## What This Changes

Correct package download and integrity metadata so each advertised digest matches the exact artifact bytes fetched by modern and legacy clients.

## Best Possible Solution

Define separate, unambiguous metadata for the exact uploaded npm-pack and generated legacy ZIP, ensure every digest is bound to its associated download URL, preserve modern artifact verification, and document/test the supported behavior for older clients.

## Reproduction Assessment

Yes. Current source defines separate npm-pack and legacy ZIP streams, and live requests reproduce distinct successful bodies and hashes while version metadata retains the npm-pack `sha256hash`.

## Solution Assessment

No single proposed repair is yet established as best. The narrowest durable solution is to bind separate modern and legacy digests to their exact routes rather than treating the unsupported `/tarball` suffix or one archive format in isolation.
