## Label Justifications

- `P1`: The reproducible mismatch blocks plugin installation for affected legacy clients and compromises a core package-integrity check.
- `impact:security`: The report concerns integrity verification and potentially misleading source provenance for executable plugin packages.

## Summary

Production still reproduces the central integrity failure: npm-pack metadata advertises the tarball hash while the legacy install URL returns a separately generated ZIP, so legacy clients can reject valid packages. The narrower related issue owns that compatibility invariant, but this report also contains a live, distinct provenance defect where an unrelated commit is presented as the package source commit; closing this issue now would discard actionable security context.

## What This Changes

Align plugin artifact integrity metadata with the exact bytes served to each client path and prevent unrelated source commits from being shown as package provenance.

## Best Possible Solution

Make every legacy integrity field describe the exact ZIP returned by legacy `/download`, preserve npm tarball integrity exclusively in artifact-aware fields, and represent absent source provenance as absent or unverified rather than substituting an unrelated commit.

## Reproduction Assessment

Yes. Live production requests reproduce distinct tarball and ZIP hashes, and the current version response exposes the tarball-derived legacy hash alongside the ZIP download path; the reported source-commit value is also visible in production metadata.

## Solution Assessment

Partly: https://github.com/openclaw/clawhub/issues/2378 is the narrowest place to resolve legacy hash compatibility, but the provenance fallback needs an explicit companion fix or canonical tracker before this broader issue can close.
