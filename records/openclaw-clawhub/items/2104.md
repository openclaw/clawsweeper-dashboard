Keep open: current production still serves JSON from the reported artifact-looking `/tarball` URL while the explicit artifact resolver serves the expected tarball bytes, and current `main` still has the broad version route that explains the mismatch. ClawHub cleanup policy also keeps issues open for maintainer handling rather than auto-closing them.

## What This Changes

The issue reports that an install path for `zerogpu-router` receives bytes from the HTTP API that do not match the published artifact digest returned by the CLI-backed artifact path.

## Best Possible Solution

Define a digest-safe package artifact route contract with regression coverage for unsupported or aliased artifact subpaths, then handle source-commit provenance display as an explicit trust-policy decision.

## Reproduction Assessment

Yes. The live ClawHub API still returns metadata JSON from the reported `/tarball` path while the explicit artifact route returns the expected tarball bytes, and current `main` has the route fallthrough that explains the behavior.

## Solution Assessment

Unclear. The narrow technical fix is likely to reject unknown `/versions/{version}/...` subpaths or intentionally alias/redirect `/tarball`, but maintainers should choose the API contract and decide whether the source-provenance concern belongs in the same change.
