## Summary

Keep open: the live API still serves JSON at an artifact-looking `/tarball` path while the documented artifact download and npm mirror paths serve the expected ClawPack bytes, and current main still has the generic version route that explains the mismatch.

## What This Changes

The issue reports that an install client using a package HTTP tarball path receives bytes whose digest differs from the published ClawPack artifact and also questions the displayed source commit provenance.

## Best Possible Solution

Define a digest-safe package artifact route contract with regression coverage for unsupported or aliased artifact subpaths, then handle source-commit provenance display as a separate trust-policy fix if needed.

## Reproduction Assessment

Yes. The live ClawHub API still returns JSON bytes at the reported artifact-looking `/tarball` path while the documented artifact download and npm mirror paths return the expected 4678-byte ClawPack artifact; current main’s generic version route makes the same failure path source-reproducible.

## Solution Assessment

Unclear. The narrow fix is likely to reject unknown `/versions/{version}/...` subpaths or deliberately alias `/tarball`, but maintainers should choose the API contract and decide whether the source-provenance report belongs in the same change.
