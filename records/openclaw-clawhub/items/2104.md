## Summary

Keep this open: the live package still returns three different successful bodies across the exact artifact, legacy ZIP, and reported `/tarball` paths, and current main still has the route shape that explains the mismatch. The related open issue at https://github.com/openclaw/clawhub/issues/2378 covers the narrower legacy `sha256hash` contract, but this issue also carries the `/tarball` fallthrough and existing provenance concern.

## What This Changes

This issue reports that public ClawHub install paths return mismatched artifact bytes, digests, and provenance for an npm-pack plugin release, causing integrity checks to fail.

## Best Possible Solution

Define one public artifact contract: exact ClawPack bytes come from `/artifact/download` or the npm mirror, legacy `/download` stays internally consistent as ZIP compatibility, unknown version suffixes stop returning successful metadata bytes, and existing bad provenance is repaired through an approved moderation/backfill path.

Unclear. The bug is concrete, but the permanent behavior needs maintainer choice between fail-closed, redirect, compatibility alias, legacy digest repair, and existing-row provenance repair.
