## Summary

Keep open: the issue still reproduces on live production and current main, and this ClawHub cleanup pass should not auto-close issues before a maintainer resolves them or a fixing PR lands.

## What This Changes

The issue reports that an artifact-looking HTTP package tarball path serves bytes that do not match the CLI-validated package artifact, with a related source-commit provenance mismatch on the listing.

## Best Possible Solution

Make package artifact routes digest-safe by preventing unknown version subpaths from returning metadata bytes, then handle the source-commit provenance mismatch as a separate cleanup if needed.

## Reproduction Assessment

Yes. The live `/versions/0.1.0/tarball` URL returns JSON bytes with a different SHA-256 digest from the supported `/artifact/download` route, and current main still contains the broad version-detail fallthrough that explains it.

## Solution Assessment

Unclear. Rejecting unknown subpaths is the narrowest defensive fix, but maintainers should choose the public API contract before deciding between 404, redirect, or a supported `/tarball` alias.
