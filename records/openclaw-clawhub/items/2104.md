## Summary

Keep open: the reported digest split still reproduces on the hosted API, and current main still has the broad version route that explains why `/versions/{version}/tarball` returns metadata JSON instead of artifact bytes or a clear unsupported-route response.

## What This Changes

The issue reports that a tarball-looking package API URL serves bytes that do not match the manifest or CLI artifact download, causing plugin install-time integrity failures.

## Best Possible Solution

Make install-facing package artifact URLs unambiguous and digest-safe, then separately clarify or fix source provenance metadata for publishes without explicit source commits.

## Reproduction Assessment

Yes. The hosted API still returns different byte streams and SHA-256 digests for the tarball-looking URL and the documented artifact download URL, and source inspection shows the same broad version-subpath route remains on current main.

## Solution Assessment

Unclear. A narrow code fix is likely possible, but maintainers should first decide whether the unsupported tarball-looking route should become a compatibility alias or a hard 404/400 so install clients fail clearly.
