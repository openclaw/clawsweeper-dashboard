## Summary

Keep open: this is a recent non-maintainer issue, and ClawHub cleanup policy does not auto-close issues. The reported install-path mismatch is still source-reproducible on current main because the package router accepts extra `/versions/{version}/...` subpaths as version metadata while the documented artifact download route streams the uploaded ClawPack bytes.

## What This Changes

The issue reports that a tarball-looking package HTTP API URL returns bytes whose digest differs from the CLI-downloaded ClawPack artifact, causing plugin installation integrity checks to fail.

## Best Possible Solution

Make package artifact resolution unambiguous by serving digest-checked ClawPack bytes from supported artifact resolver routes or returning a clear unsupported-route error for tarball-looking version subpaths, with regression coverage and a source-provenance clarification.

## Reproduction Assessment

Yes for source reproduction: current main still allows an extra `/versions/{version}/tarball` path to reach version metadata instead of artifact bytes, while documented artifact routes stream the ClawPack artifact. I did not need to mutate local state or run tests to establish the route-level mismatch.

## Solution Assessment

Unclear until maintainers choose the route compatibility contract. The narrow maintainable fix is either to alias the tarball-looking path to the exact ClawPack artifact bytes or to reject unsupported version subpaths before metadata generation, then cover that chosen behavior in endpoint tests.
