## Label Justifications

- `P1`: The PR changes the clean-scan trust boundary and can also stop benign scan completion in a production worker workflow.
- `merge-risk: 🚨 security-boundary`: Unsupported range-kind assertions can currently satisfy the new verifier and allow an inadequately substantiated benign result to become clean.
- `merge-risk: 🚨 availability`: The mandatory output schema and fail-closed verifier may reject real benign worker results when coverage output is malformed or incomplete.

## Summary

The hardening goal is valuable and not present on current main, but the verifier accepts coverage labels whose coordinates are entirely null, so a benign result can still claim full or head/tail inspection without identifying any inspected interval; the PR also lacks after-fix evidence from a real Codex worker.

## What This Changes

The PR requires Codex scan workers to report file-level artifact coverage before benign scan results can be stored as clean.

## Best Possible Solution

Require every accepted coverage range to contain at least one complete, ordered, non-empty line or byte interval, retain incomplete diagnostics for operators, and prove with redacted real-worker output that valid benign coverage succeeds while incomplete coverage is rejected.

## Reproduction Assessment

Yes. The PR's accepted test fixture constructs coordinate-free `full`, `head`, and `tail` ranges, and source inspection shows the verifier accepts them based only on their kind labels.

## Solution Assessment

No, not yet. Requiring coverage at the worker boundary is appropriately narrow, but accepted coverage needs mechanically meaningful intervals and real-worker validation before it can safely protect the clean-scan decision.
