## Label Justifications

- `P1`: The PR changes the production ClawScan clean-result trust boundary and can make live scan workers retry or fail if coverage output is unreliable.
- `merge-risk: 🚨 security-boundary`: Merging changes the conditions under which ClawScan may store a clean artifact result.

## Summary

Keep open: this PR is useful security hardening with sufficient live proof, but the new verifier rejects the natural full-coverage byte interval for zero-byte files, and maintainers still need to accept the stricter fail-closed ClawScan worker contract.

## What This Changes

This PR requires `artifact_coverage` in Codex scan worker output, prompts and verifies submitted-file coverage before storing benign ClawScan results, adds worker tests, and records the invariant in `specs/security-moderation.md`.

## Best Possible Solution

Fix zero-byte coverage handling, then land after maintainer security review accepts the fail-closed clean-result gate and coordinates with https://github.com/openclaw/clawhub/pull/2492 as needed.

## Reproduction Assessment

Yes. Source inspection shows current main lacks `artifact_coverage`, the PR body includes live acceptance/rejection output, and the zero-byte defect follows from the data model allowing `size: v.number()` while the PR rejects `endByte === startByte`.

## Solution Assessment

No, not yet. The worker/schema boundary is the right place to enforce clean-result coverage, but the verifier should handle zero-byte submitted files before maintainers accept the fail-closed policy change.

Full review comments:

- **[P2] Accept empty-file full coverage:** `scripts/security/run-codex-scan-worker.ts:998-999`
  - body: Artifact file sizes are stored as plain `v.number()` with no minimum, and publish paths do not reject zero-byte files. For a zero-byte submitted file, the honest full byte interval is `startByte: 0, endByte: 0`, but this check rejects every byte interval where `end === start` before comparing it to `file.size`. That can make benign scans containing empty files fail or retry even though there is no file content left uninspected; allow the `0..0` full range for `file.size === 0` and add a regression test.
  - confidence: 0.86
