## Summary

This PR should remain open because its verifier accepts coverage range labels without checking any line or byte bounds, allowing a benign result to be stored without concrete proof of what content was inspected; it also lacks real Codex worker evidence for the new fail-closed output contract.

## What This Changes

The PR requires structured artifact-coverage output from the Codex scan worker, validates per-file coverage before storing benign results, adds regression tests, and documents the security invariant.

## Best Possible Solution

Require every accepted `full`, `head`, and `tail` range to contain at least one concrete, ordered line or byte interval, retain the per-file size checks, and validate the finalized contract with redacted output from a real Codex worker execution.

## Reproduction Assessment

Yes at source level: pass a benign payload containing required `full`, `head`, and `tail` range kinds with all line and byte coordinates null through `assertCodexArtifactCoverageForVerdict`. The PR's successful fixture already exercises this path.

## Solution Assessment

No, not yet. The worker/schema boundary is appropriate, but semantic range labels without concrete bounds do not prove inspection, and the stricter hosted-model contract still needs real execution evidence.
