## Summary

The latest PR head still accepts `full`, `head`, and `tail` claims with every coordinate null, so the proposed security proof remains structurally assertive rather than mechanically meaningful; real worker proof is also absent.

## What This Changes

The PR requires Codex scan workers to provide file-level artifact coverage proof before a benign result can be stored as clean.

## Best Possible Solution

Require each accepted coverage range to contain a valid non-empty ordered line or byte interval, preserve suspicious and malicious handling, add malformed-range regression tests, and demonstrate both benign rejection and acceptance with redacted live worker output.

No, not yet. The worker boundary is the correct location, but coverage proof must identify meaningful intervals and be exercised against a real Codex worker before it safely supports clean results.
