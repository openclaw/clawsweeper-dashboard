## Summary

Keep open: the hardening is useful, but the verifier still accepts kind-only ranges with null coordinates, and the contributor evidence is test-command claims rather than a real Codex worker run.

## What This Changes

The PR adds `artifact_coverage` to the Codex scan output schema, prompts the worker to report submitted-file coverage, rejects benign results without adequate coverage, adds tests, and records the invariant in the security moderation spec.

## Best Possible Solution

Tighten the verifier so range proof includes concrete non-null line or byte bounds for full/head/tail coverage, then show a real Codex worker run where an incomplete benign result fails and a covered benign result stores cleanly.

Yes from source inspection: a benign result with `ranges` containing `head` and `tail` kinds but null coordinates reaches the success path because the verifier only checks range kind membership. I did not run the test suite in this read-only review.

## Solution Assessment

No. The direction is maintainable, but the verifier should require concrete range bounds and the PR needs real worker proof before this is the best merge path.
