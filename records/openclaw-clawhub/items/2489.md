## Label Justifications

- `P1`: The patch modifies the clean-scan security boundary but currently accepts coverage claims that do not identify any inspected content.
- `merge-risk: 🚨 security-boundary`: The new verifier controls whether a model’s benign verdict becomes a trusted clean scan, and coordinate-free claims can bypass its intended assurance.
- `merge-risk: 🚨 availability`: The required output contract fails closed and can prevent benign scans from completing when actual Codex output does not conform.

## Summary

Keep this PR open: the worker-boundary design is appropriately scoped and current main does not provide equivalent coverage enforcement, but the proposed verifier accepts range labels with every coordinate set to null, so a benign result can still claim `full` or `head`/`tail` coverage without identifying any inspected content. The contributor also has not supplied real Codex-worker evidence for this security-sensitive fail-closed contract.

## What This Changes

The PR adds mandatory artifact-coverage output and rejects benign Codex scan results unless every submitted file claims full coverage or, for large files, head-and-tail coverage.

## Best Possible Solution

Require each accepted coverage claim to identify a valid non-empty line or byte interval appropriate to its kind, test malformed and coordinate-free ranges, and demonstrate the final contract with redacted real-worker output before merge.

## Reproduction Assessment

Yes. The new fixture itself constructs `full`, `head`, and `tail` ranges with every coordinate null, and the verifier accepts them based only on the range-kind strings.

## Solution Assessment

No, not yet. Worker-boundary enforcement is the right architecture, but meaningful coverage proof requires validated non-empty coordinates rather than trusting labels alone, followed by real worker compatibility evidence.
