- `merge-risk: 🚨 security-boundary`: The patch changes the clean-verdict trust boundary for ClawScan and currently leaves a contradictory coverage-proof acceptance path.

## Summary

Keep open: the security hardening is useful and not already on main, but the proposed validator can still accept contradictory metadata-only or unread coverage when a range claims full/head-tail coverage, and the PR body only provides test/type/static validation rather than real worker proof.

## What This Changes

This PR adds `artifact_coverage` to the Codex scan output schema and prompt, rejects benign worker results without submitted-file coverage proof, adds worker tests, and documents the invariant in `specs/security-moderation.md`.

## Best Possible Solution

Land after the validator rejects metadata-only and unread coverage regardless of ranges, regression tests cover contradictory coverage records, and the PR body includes redacted real Codex worker proof for both a covered benign result and a rejected incomplete result.

## Reproduction Assessment

Yes. Source inspection shows a benign result with `coverage: "metadata_only"` or `"not_inspected"` can pass if `ranges` includes `full` or head/tail, because the validator treats ranges as sufficient at the new fallback checks.

## Solution Assessment

No. The schema/prompt direction is sound, but the validator should reject unread and metadata-only coverage enum values before any range fallback, then prove the real worker path with redacted output.
