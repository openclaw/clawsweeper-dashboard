## Summary

This security hardening should remain open because its verifier accepts coverage labels without concrete inspected intervals, and the external contribution still lacks evidence from a real Codex worker run.

## What This Changes

The PR requires Codex scan worker output to report per-file artifact coverage and rejects benign results that omit submitted files or lack required full/head-tail coverage kinds.

## Best Possible Solution

Keep the stricter benign-verdict gate, require concrete ordered line or byte bounds for every accepted coverage range, and prove the contract with redacted live worker output showing both successful acceptance and fail-closed rejection.

## Reproduction Assessment

Yes. PR-head source provides a high-confidence path: the accepted fixture creates full/head/tail ranges with all coordinates null, while the verifier checks only their kind labels.

## Solution Assessment

No, not yet. The worker boundary is the correct enforcement point, but the implementation must validate concrete intervals and demonstrate compatibility with output from a real worker run.
