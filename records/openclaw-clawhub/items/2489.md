## Label Justifications

- `P2`: This is a bounded security-hardening improvement with meaningful scanner impact but no demonstrated active emergency or released regression.
- `merge-risk: 🚨 security-boundary`: The new verifier decides whether a Codex result may be persisted as clean ClawScan state, and coordinate-free claims currently pass.

## Summary

Current `main` does not implement artifact-coverage proof, so the PR remains useful, but it is not merge-ready: the verifier trusts `full`, `head`, and `tail` labels even when every coordinate is null, and the contributor has not shown a real Codex worker producing the new required contract.

## What This Changes

The PR requires per-file `artifact_coverage` in Codex scan output, rejects insufficient benign-verdict coverage before storage, adds worker regression tests, and documents the scanner invariant.

## Best Possible Solution

Keep the fail-closed worker-boundary design, but require each accepted content range to contain a valid ordered line or byte interval and demonstrate compatible output from a real Codex scan-worker run before merge.

## Reproduction Assessment

Yes. The PR’s own `coverageRange` fixture supplies `full`, `head`, and `tail` entries with all coordinates null, and source inspection shows those entries satisfy the verifier.

## Solution Assessment

No, not yet. The worker/schema boundary is the correct narrow location, but symbolic range kinds must not substitute for concrete intervals, and production-like output compatibility needs direct evidence.
