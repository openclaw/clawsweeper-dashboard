## Label Justifications

- `P1`: The PR changes the clean-scan trust boundary and can fail production scan jobs if live Codex output cannot satisfy the new schema.
- `merge-risk: 🚨 security-boundary`: The proposed proof can still accept coordinate-free range labels as evidence for clean-scan artifact coverage.
- `merge-risk: 🚨 availability`: The mandatory structured coverage contract can fail otherwise benign worker jobs when live Codex responses are incomplete or malformed.

## Summary

Keep open. The hardening is useful and not already on main, but PR head can still trust coordinate-free `full`/`head`/`tail` range labels as clean-scan coverage and the contributor has not provided live Codex worker proof for the stricter schema.

## What This Changes

The branch adds `artifact_coverage` to Codex worker output, prompts Codex to prove submitted-file coverage, rejects benign results with missing or incomplete coverage, adds tests, and documents the invariant in `specs/security-moderation.md`.

## Best Possible Solution

Keep the worker-level coverage contract, but validate non-empty ordered line or byte intervals for accepted ranges and require redacted live Codex worker proof before maintainers accept the stricter fail-closed behavior.

## Reproduction Assessment

Yes by source inspection. PR head's positive fixture accepts null coordinates, and `assertCodexArtifactCoverageForVerdict` only checks accepted range-kind membership.

## Solution Assessment

No, not yet. The worker boundary is the right place, but clean-scan proof must validate meaningful inspected intervals and include live worker evidence before this is a safe fail-closed contract.
