## Summary

Keep open: the security hardening direction is useful, but the patch still lets a benign result prove coverage with range kinds that have no line or byte bounds, and the external PR only provides test/CI validation rather than real worker behavior proof.

## What This Changes

The PR adds an `artifact_coverage` field to Codex scan worker output, prompts the worker to report submitted-file coverage, rejects incomplete benign coverage before clean storage, adds tests, and records the invariant in `specs/security-moderation.md`.

## Best Possible Solution

Keep the benign-only coverage gate, but require measurable line or byte bounds for accepted ranges, cover that case with regression tests, and add redacted real worker output showing both accepted complete coverage and rejected incomplete coverage.

## Reproduction Assessment

Yes for the review finding: source inspection shows `coverageRangeKinds` only reads range `kind`, and the accepted test fixture uses all-null line and byte coordinates. I did not run the tests because this review is read-only.

## Solution Assessment

No, not yet. The direction is the right security boundary, but the implementation should validate measurable range bounds and include real worker proof before maintainers rely on it.
