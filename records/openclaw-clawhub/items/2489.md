## Summary

Keep open: current main and the latest release still lack Codex artifact coverage proof, and this PR is useful security hardening, but the patch is not merge-ready because the verifier still accepts label-only ranges with null coordinates and the external PR has no real worker proof.

## What This Changes

The PR adds `artifact_coverage` to the Codex scan worker schema and prompt, validates benign worker outputs against submitted-file coverage, adds tests, and records the invariant in `specs/security-moderation.md`.

## Best Possible Solution

Tighten the verifier so full/head/tail coverage ranges include concrete non-null line or byte intervals, then add redacted real worker output showing incomplete benign coverage fails and covered benign output stores cleanly.

## Reproduction Assessment

Yes. From source inspection, a benign result with `ranges` containing `head` and `tail` kinds but null coordinates reaches the success path because the verifier checks only kind membership; I did not run tests in this read-only review.

## Solution Assessment

No. The direction is maintainable, but the verifier should require concrete coverage bounds and the PR needs real worker proof before this is the best merge path.
