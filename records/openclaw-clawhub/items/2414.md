## Summary

Keep open. The security direction is useful and current head preserves the released CLI multipart field names, but web publishing appears broken because it sends a Convex session token to an HTTP publish endpoint that still accepts only API or trusted publish tokens; real behavior proof and maintainer rollout/security approval are still missing.

## What This Changes

This PR makes package publish multipart-only, rejects caller-supplied file and artifact metadata, derives upload metadata server-side, updates CLI/web publish flows, and adds docs, specs, and tests.

## Best Possible Solution

Land the package publish hardening only after the browser publish auth path, multipart-only rollout, and 18MB cutoff are explicitly approved, tested, documented, and proven against real publish/rejection flows.

## Solution Assessment

No. The server-derived metadata invariant is the right security direction, but this implementation should not merge until the browser auth boundary and compatibility rollout are fixed or explicitly approved.
