## Summary

Keep open. This is an open ClawHub issue, current production GitHub sign-in still returns HTTP 500 before it can redirect into GitHub OAuth, and the source shows the callback/issuer path is environment-driven rather than a hardcoded localhost callback.

## What This Changes

The issue reports that production GitHub OAuth redirects to a 127.0.0.1 callback instead of the production Convex Auth callback.

## Best Possible Solution

Restore production GitHub OAuth credentials and callback/site URL alignment, verify web and CLI login end to end, add production diagnostics or smoke checks for this path, then consolidate related OAuth reports once the root cause is clear.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the current production OAuth failure: current main is production-tagged and both the canonical and direct Convex GitHub sign-in endpoints return HTTP 500. The exact reported 127.0.0.1 callback cannot be confirmed until sign-in reaches GitHub authorization again.

## Solution Assessment

Is this the best way to solve the issue? Yes for maintainer/operator repair, not for hardcoding a callback in source. The narrow path is to fix GitHub OAuth App plus Convex/Vercel env alignment and add request-time diagnostics or a deployment smoke check without weakening Convex Auth state/session handling.
