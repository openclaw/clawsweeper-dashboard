## Summary

Keep open. This is an open ClawHub issue, and current production-tagged main still returns HTTP 500 from both GitHub sign-in endpoints, so the auth failure is not proven fixed; the exact 127.0.0.1 callback variant cannot be reconfirmed until sign-in reaches GitHub again.

## What This Changes

Report that production GitHub OAuth sends users to a loopback callback after authorization, leaving login broken on clawhub.ai.

## Best Possible Solution

Restore production GitHub OAuth app, Convex env, Vercel/site URL, and callback alignment, verify web and CLI login end to end, then add an import-safe diagnostic or deploy smoke check for 500s and localhost callback targets.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the current production sign-in failure: the current production-tagged SHA returns HTTP 500 from both GitHub sign-in endpoints while public API reads return 200. The exact 127.0.0.1 post-authorization redirect is not reconfirmed because the flow now fails before reaching GitHub.

## Solution Assessment

Is this the best way to solve the issue? Partly: operator-side OAuth callback and env alignment is the right repair boundary, but it should be verified with production logs and smoke checks rather than hardcoding a callback or reviving the eager module-load validation rejected in https://github.com/openclaw/clawhub/pull/1720.
