## Summary

Keep open. This is an open ClawHub issue, cleanup policy does not auto-close issues, and the live production GitHub sign-in path is still failing before the reported 127.0.0.1 callback can be proven fixed.

## What This Changes

The issue reports production GitHub OAuth redirecting to localhost/127.0.0.1 after authorization instead of the production Convex callback.

## Best Possible Solution

Restore production GitHub OAuth credential, callback, issuer, and site URL alignment, verify web and CLI login end to end, then add an import-safe diagnostic or deploy smoke check for auth 500s and localhost callback targets.

## Reproduction Assessment

Partly: live GETs to both production GitHub sign-in endpoints returned HTTP/2 500 while the public skills API returned HTTP/2 200. The exact 127.0.0.1 post-authorization redirect is not currently reproducible because the flow fails before GitHub authorization.

## Solution Assessment

Yes for routing: keeping this open for maintainer/operator auth triage is the narrowest maintainable path. A source-only repair is not enough until production OAuth app settings, Convex/Vercel env, cookies, and auth logs are verified; any diagnostic should follow the request-time or deploy/env validation direction from https://github.com/openclaw/clawhub/pull/1720.
