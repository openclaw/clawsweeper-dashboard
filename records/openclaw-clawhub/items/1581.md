## Summary

Keep open. Current ClawHub cleanup policy does not auto-close issues, and the production GitHub sign-in path is still failing: both the canonical site and direct Convex auth endpoint return HTTP 500 while the public skills API is healthy. Current source routes auth through Convex and environment-driven GitHub credentials rather than a hardcoded web callback, so this needs production OAuth/Convex/Vercel operator follow-up.

## What This Changes

The issue reports production GitHub OAuth redirecting to `127.0.0.1` instead of the production Convex callback after authorization.

## Best Possible Solution

Restore production GitHub OAuth credential, callback, issuer, site URL, and cookie alignment, then verify web sign-in and CLI token creation end to end against `clawhub.ai` with redacted evidence.

## Reproduction Assessment

Yes for the central production auth failure: current read-only curl checks show both canonical and direct Convex GitHub sign-in endpoints return HTTP 500 while the public skills API returns HTTP 200. The later `127.0.0.1` callback symptom is not currently replayable because sign-in fails before GitHub authorization.

## Solution Assessment

Yes for keeping this open and routing to maintainer/operator auth triage.
