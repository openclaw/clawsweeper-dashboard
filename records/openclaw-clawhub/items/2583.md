## Summary

This PR addresses a real historical ClawPack data defect with a well-bounded admin migration, but it is not merge-ready: the apply path can orphan newly staged or superseded storage blobs, and the posted evidence remains mocked rather than real Convex runtime proof.

## What This Changes

The PR adds an admin-only, dry-run-first Convex migration that reconstructs incomplete historical ClawPack release file rows from stored artifacts, resets derived review state, requeues scans, adds regression tests, and updates the moderation specification.

## Best Possible Solution

Keep the explicit targeted migration, but make blob ownership transactional at the application level by cleaning staged blobs on non-commit/error and deleting only superseded prior blobs after a successful swap, then validate a narrow dry run and confirmed apply on real Convex before rollout.

Yes. The linked published release has fewer stored file rows than verified artifact entries, and current main consumes those stored rows for inspection/review; the proposed storage leak is also directly reproducible from the store-before-conditional-swap source path.

No, not yet. Separating destructive historical repair from ordinary scan backfill is the right architecture, but the implementation needs explicit blob cleanup and real Convex runtime verification.
