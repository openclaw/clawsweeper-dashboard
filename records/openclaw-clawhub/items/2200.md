## Summary

Keep open. Current main still has a source-visible deleted-slug collision path: rename rejects any different row with the target slug and can tell the user to merge, while merge refuses soft-deleted source rows; publish and availability have partial 30-day reservation handling, but that does not fully cover the reported personal/org rename workflow.

## What This Changes

The issue asks for deleted skill slugs to be reusable, releasable, or reclaimable across owners when the user can manage both entities.

## Best Possible Solution

Keep the 30-day owner-delete reservation, then add a publisher-aware audited path that releases expired owner-unpublished slug rows consistently for rename/publish and gives clear UI/CLI guidance for in-window or cross-publisher cases.

## Reproduction Assessment

Yes, from source inspection: soft-delete one manageable skill, then rename another manageable skill to that slug; rename sees the deleted row in skills.by_slug and rejects before any expired-reservation release, and merge rejects soft-deleted sources. I did not run a live Convex reproduction or inspect production data.

## Solution Assessment

Unclear for the full request: immediate deleted-slug reuse or cross-publisher reclaim before the 30-day TTL needs maintainer product/security judgment. The narrow maintainable code direction is to align rename with the existing expired owner-unpublish release contract and improve the merge/rename message when the slug is still reserved.
