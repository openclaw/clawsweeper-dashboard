## Summary

Keep open. Current main still has a source-level lifecycle gap: availability/publish can treat expired owner-unpublished slugs as claimable, but rename rejects the soft-deleted slug row first and merge rejects deleted rows, while the requested cross-owner reclaim behavior needs maintainer ownership/security policy judgment.

## What This Changes

The issue requests a way to reuse, release, merge, or reclaim a soft-deleted skill slug across owners when the same user has permission for both entities.

## Best Possible Solution

Maintainers should define the same-actor cross-owner reclaim rules, then align rename, merge, publish, availability, UI copy, specs/docs, and regression tests around one deleted-slug lifecycle.

## Reproduction Assessment

Yes for the source-level mismatch: a rename to a slug held by a different soft-deleted skills.by_slug row reaches the collision branch, while merge rejects deleted source or target rows. The exact live personal/org production state was not verified.

## Solution Assessment

Unclear. A narrow fix could make rename share the documented post-expiry release path, but deleted-row merge and pre-expiry cross-owner reclaim require explicit ownership and abuse-prevention policy first.
