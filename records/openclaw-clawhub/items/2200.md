## Summary

Keep open: current main still has a source-level mismatch where expired owner-deleted slugs can be shown as claimable and reused by publish, but rename rejects the soft-deleted row first and merge rejects soft-deleted sources. The broader personal/org reclaim behavior needs explicit ownership and security policy before implementation.

## What This Changes

The issue asks for an authorized way to reuse or release a deleted skill slug, including personal/org ownership cases where rename suggests merge but merge cannot use the deleted source.

## Best Possible Solution

Keep the 30-day owner-delete reservation, align rename with the existing post-expiry publish release path, and add any in-window or cross-publisher reclaim only behind explicit audited authority checks.

## Reproduction Assessment

Yes, from source inspection: after an owner-deleted skill's reservation expires, availability and publish can treat the slug as claimable, but rename rejects the soft-deleted row first and merge rejects a soft-deleted source. I did not run a live Convex reproduction or inspect production data.

## Solution Assessment

Unclear for the full request: immediate reuse or cross-publisher reclaim before the 30-day TTL needs maintainer product/security judgment. The narrow maintainable fix is to align rename with the existing expired owner-unpublish release contract and improve guidance while a slug is still legitimately reserved.
