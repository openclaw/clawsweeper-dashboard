## Summary

Keep open. Current main still has a source-visible deleted-slug collision path: availability and publish partially honor expired owner-delete reservations, but rename rejects the soft-deleted slug row first and merge rejects deleted sources, so the reported workflow remains unresolved and needs maintainer policy/security review.

## What This Changes

The issue asks for a way to reuse, release, merge, or reclaim a soft-deleted skill slug across owners when the user has authority for both publishers.

## Best Possible Solution

Keep the 30-day deleted-slug reservation, make rename honor the existing post-TTL lazy-release contract, and route in-window or cross-publisher reclaim through audited publisher authority checks.

## Reproduction Assessment

Yes, from source inspection: after a manageable skill is soft-deleted, renaming another manageable skill to that slug hits the skills.by_slug collision path before any expired-reservation release, while merge rejects the deleted source. I did not run a live Convex reproduction against production data.

## Solution Assessment

Unclear for the full request because cross-publisher reclaim before or during the reservation window is a product/security decision. The narrow maintainable fix is to make rename honor the existing expired-reservation release contract and require explicit authority checks for any cross-publisher reclaim path.
