## Summary

Keep open. Current main still has a source-visible deleted-slug gap: availability and publish handle expired owner-unpublish reservations, but owner rename rejects the soft-deleted `skills.by_slug` row first and merge refuses soft-deleted source/target rows; cross-publisher reclaim also needs maintainer security/product policy.

## What This Changes

The issue requests a way to reuse, release, merge, or reclaim a deleted skill slug across owners when the actor has permission for the affected entities.

## Best Possible Solution

Keep the 30-day owner-delete reservation, align rename/publish/availability after expiry, and route in-window or cross-publisher reclaim through audited authority checks for both source and target publishers.

## Reproduction Assessment

Yes, at source level: an expired owner-unpublish deleted slug can appear available and can be claimed through publish, but rename rejects the existing soft-deleted `skills.by_slug` row and merge rejects soft-deleted source/target rows. I did not run a live Convex reproduction or inspect production data.

## Solution Assessment

Unclear for the full request: immediate reuse or cross-publisher reclaim before the 30-day TTL needs maintainer product/security judgment. The narrow maintainable direction is to make rename honor the existing expired owner-unpublish contract and keep cross-publisher reclaim explicitly authorized and audited.
