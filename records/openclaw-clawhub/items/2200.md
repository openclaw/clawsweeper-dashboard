## Summary

Keep open: current main partially handles expired owner-unpublished slug release on publish and has an admin-only reclaim path, but the reporter's rename/merge-across-owners workflow is still not implemented. Rename still rejects any different skill row with the requested slug, merge still rejects soft-deleted source or target rows, and self-service cross-owner reclaim remains an authorization-sensitive product decision.

## What This Changes

The issue asks for a way to reuse, release, merge, or reclaim a deleted skill slug across owners when the actor has permission for both entities.

## Best Possible Solution

Define a maintainer-approved self-service reclaim/release policy for expired owner-unpublished slugs, then implement it in backend and UI with explicit owner authorization, TTL handling, audit provenance, and tests that preserve moderator/security hides.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: after an owner-deleted slug reservation expires, availability and publish can treat it as claimable, but rename still rejects the existing soft-deleted row and merge rejects soft-deleted source or target rows.

## Solution Assessment

No current implementation fully solves it. The safest path is an explicit owner-authorized reclaim/release policy rather than broadly ignoring soft-deleted slug rows in rename or merge.
