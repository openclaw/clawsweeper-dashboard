## Summary

Keep open. Current main still has the deleted-slug collision path: rename rejects a different `skills.by_slug` row before release handling, while merge refuses soft-deleted sources; the remaining cross-owner reclaim behavior needs product and security review.

## What This Changes

Deleted skill slugs remain unavailable for rename or merge/reclaim across owners even when the actor can manage both publishers.

## Best Possible Solution

Keep the 30-day owner-delete reservation, then add a publisher-aware audited path that consistently releases expired owner-unpublished slug rows for rename/publish and gives clear guidance for in-window or cross-publisher cases.

## Reproduction Assessment

Yes, from source inspection: soft-delete one manageable skill, then rename another manageable skill to that slug; rename sees the deleted row through `skills.by_slug` and rejects before release handling, while merge rejects soft-deleted sources. I did not run a live Convex reproduction or inspect production data.

## Solution Assessment

Unclear for the full request: immediate deleted-slug reuse or cross-publisher reclaim before the 30-day TTL needs maintainer product/security judgment. The narrow maintainable code direction is to align rename with the existing expired owner-unpublish release contract and improve messages for still-reserved slugs.
