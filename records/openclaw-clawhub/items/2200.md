## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a real product gap around reusing a deleted slug through rename/merge when source and target are managed through different owner entities.

## What This Changes

The issue requests a way to reuse, release, merge, or reclaim a deleted skill slug when the user has permission for both involved owners.

## Best Possible Solution

Define a permission-aware deleted-slug reclaim or rename flow that preserves the 30-day reservation/audit model, supports users who administer both the source and destination publishers, and avoids telling users to merge with a deleted or unavailable source.

## Reproduction Assessment

Yes, at source level: rename rejects any different skills.by_slug row, including a soft-deleted one, while merge rejects soft-deleted source/target skills and the UI lists only active skills for the current owner scope. I did not verify a live production slug row for this specific reporter.

## Solution Assessment

Unclear. Reusing a deleted slug immediately conflicts with the documented 30-day reservation policy, so maintainers should first decide whether the supported path is post-TTL rename release, explicit source restore/transfer, admin support reclaim, or a new self-service cross-publisher reclaim flow.
