## Summary

Keep open: current main still only releases expired owner-unpublished slugs on the publish path, while rename and merge continue to reject soft-deleted slug rows. The remaining cross-owner reclaim behavior is authorization-sensitive and needs a maintainer policy decision.

## What This Changes

Request a way to reuse or release a soft-deleted skill slug, or merge/reclaim it across personal and organization owners when the actor controls both.

## Best Possible Solution

Define the cross-publisher deleted-slug reclaim policy first, then implement a narrow backend/UI flow that honors the 30-day owner-unpublish reservation, excludes moderation/security hides, and records audit provenance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: after an owner-deleted slug reservation expires, availability and publish can treat it as claimable, but rename still rejects the existing soft-deleted row and merge rejects soft-deleted source or target rows.

## Solution Assessment

No current implementation fully solves it. The safest path is an explicit owner-authorized reclaim/release policy, not broadly ignoring soft-deleted rows in rename or merge.
