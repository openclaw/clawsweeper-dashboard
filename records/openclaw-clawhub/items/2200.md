## Summary

Keep open. Current main still has a source-visible deleted-slug collision path: availability and publish have partial post-TTL release handling, but rename rejects the soft-deleted row first and merge refuses deleted sources, so the reported workflow is not fully solved.

## What This Changes

Allow a deleted skill slug to be reused, released, or reclaimed or merged across publishers when the actor has appropriate permissions.

## Best Possible Solution

Keep the documented 30-day owner-delete reservation, align rename/publish/availability after expiry, and route in-window or cross-publisher reclaim through audited publisher authority checks.

## Reproduction Assessment

Yes, from source inspection: soft-delete a manageable skill, wait past the owner-unpublish reservation if testing post-TTL behavior, then rename another manageable skill to that slug; rename sees the deleted row in skills.by_slug and rejects before any expired-reservation release, while merge rejects soft-deleted sources. I did not run a live Convex reproduction or inspect production data.

## Solution Assessment

Unclear for the full request: immediate reuse or cross-publisher reclaim before the 30-day TTL needs maintainer product/security judgment. The narrow maintainable code direction is to align rename with the existing expired owner-unpublish release contract and improve guidance when a slug is still legitimately reserved.
