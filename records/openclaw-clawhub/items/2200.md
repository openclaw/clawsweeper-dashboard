## Summary

Keep open. Current main still has a source-visible deleted-slug collision path: availability and publish know about expired owner-unpublish reservations, but owner rename rejects the soft-deleted `skills.by_slug` row first and merge refuses a soft-deleted source; cross-owner reclaim also needs maintainer security/product judgment.

## What This Changes

The issue asks ClawHub to let authorized users reuse or release a deleted skill slug, or merge/reclaim it across personal and organization owners.

## Best Possible Solution

Keep the documented 30-day owner-delete reservation, align rename/publish/availability after expiry, and route in-window or cross-publisher reclaim through audited publisher authority checks for both source and target owners.

## Reproduction Assessment

Yes, from source inspection: after a manageable skill is soft-deleted, renaming another manageable skill to that slug still hits the soft-deleted `skills.by_slug` row and fails before any expired-reservation release logic, while merge rejects a soft-deleted source. I did not run a live Convex reproduction or inspect production data.

## Solution Assessment

Unclear for the full request: immediate reuse or cross-publisher reclaim before the 30-day TTL needs maintainer product/security judgment. The narrow maintainable direction is to align rename with the existing expired owner-unpublish release contract and improve guidance while a slug is still legitimately reserved.
