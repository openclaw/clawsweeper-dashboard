## Summary

Keep open: current main still has a source-visible deleted-slug mismatch where availability and publish understand expired owner-delete reservations, but rename rejects the soft-deleted row first and merge cannot use a soft-deleted source.

## What This Changes

The issue asks for deleted skill slugs to become reusable, releasable, or reclaimable across owners when the actor has authority over both entities.

## Best Possible Solution

Keep the 30-day owner-delete reservation, align rename and publish after expiry, and route in-window or cross-publisher reclaim through audited authority checks for both publishers.

## Reproduction Assessment

Yes, from source inspection: after a manageable skill is soft-deleted, renaming another manageable skill to that slug still hits the soft-deleted skills.by_slug row before any expired-reservation release logic, while merge rejects a soft-deleted source. I did not run a live Convex reproduction or inspect production data.

## Solution Assessment

Unclear for the full request: immediate reuse or cross-publisher reclaim before the 30-day TTL needs maintainer product/security judgment. The narrow maintainable fix is to align rename with the existing expired owner-unpublish release contract and improve guidance while a slug is still legitimately reserved.
