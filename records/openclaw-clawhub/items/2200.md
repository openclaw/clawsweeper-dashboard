## Summary

Keep this issue open. Current main still has the reported workflow gap: owner deletes reserve the slug, rename treats the deleted row as a collision, and merge/transfer reject soft-deleted source skills; fixing cross-publisher reclaim needs an explicit ownership and abuse-prevention decision.

## What This Changes

The issue requests a way to reuse, release, merge, or reclaim a deleted skill slug across owners when the same actor can manage both publishers.

## Best Possible Solution

Maintainers should define a publisher-aware reclaim path that verifies authority over both the deleted slug's publisher and the destination publisher, preserves audit history, and documents how it interacts with the 30-day reservation model.

## Reproduction Assessment

Yes for the source behavior: delete sets a 30-day unpublished slug reservation, rename rejects any different global slug row, and merge/transfer reject soft-deleted source skills. I did not verify the reporter's exact production records.

## Solution Assessment

Unclear as an implementation path. The safest next step is a maintainer decision between admin-only support, a self-service publisher-aware release, or a soft-deleted transfer/merge flow with strict authorization and audit behavior.
