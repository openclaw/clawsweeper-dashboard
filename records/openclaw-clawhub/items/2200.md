## Summary

Current source still has the reported collision: availability and publish know about expired owner-unpublish releases, but rename rejects the soft-deleted slug row first and merge rejects deleted sources, so this needs maintainer policy follow-up.

## What This Changes

The issue asks for deleted skill slugs to be reusable or reclaimable across owners when the actor has authority over both publishers.

## Best Possible Solution

Keep the 30-day owner-delete reservation, make rename and publish consistently release expired owner-unpublished collisions, and require an audited publisher-authority decision for in-window or cross-publisher reclaim.

## Reproduction Assessment

Yes, from source inspection: soft-delete a manageable skill, wait past the owner-unpublish reservation if testing the post-TTL path, then rename another manageable skill to that slug; rename rejects the soft-deleted by_slug row before lazy release, and merge rejects a soft-deleted source.

## Solution Assessment

Unclear for the full request because cross-publisher self-service reclaim is an ownership policy decision; the narrow maintainable fix is to align rename with the existing expired-reservation release contract and gate any broader reclaim path behind verified publisher authority and audit logs.
