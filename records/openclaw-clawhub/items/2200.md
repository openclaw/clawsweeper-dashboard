## Summary

Keep open: current main still has only the publish-time lazy release for expired owner-unpublished slugs; rename rejects the existing soft-deleted slug row first, and merge refuses soft-deleted source/target rows, so the cross-owner reclaim path remains unresolved and authorization-sensitive.

## What This Changes

The issue asks for a way to reuse, release, merge, or reclaim a deleted skill slug across personal and organization owners when the same user has permission for both.

## Best Possible Solution

Define the cross-publisher deleted-slug reclaim policy first, then implement the narrow backend and UI paths that honor the 30-day owner-unpublish TTL, exclude moderation/security hides, and write audit provenance.

## Reproduction Assessment

Yes, from source inspection: create an owner-deleted skill row with an expired owner-unpublish reservation, then try to rename another manageable skill to that slug or merge from the deleted source; current main rejects those paths. I did not run the live Convex scenario.

## Solution Assessment

No current implementation fully solves it. The maintainable path is to decide the cross-publisher reclaim policy, then wire rename, merge guidance, and tests to that policy instead of broadly ignoring soft-deleted rows.
