## Summary

Keep open. Current main documents and implements owner-deleted skill slugs as reserved for a cooldown, while rename/merge still cannot reclaim a soft-deleted cross-owner row; the requested self-service release/reclaim path needs maintainer product and security judgment.

## What This Changes

The issue requests a way to reuse, release, merge, or reclaim a deleted skill slug across owners when the user has permission for both entities.

## Best Possible Solution

Define a publisher-aware deleted-slug reclaim policy, then implement a guarded owner/admin workflow that verifies authority on both source and destination, preserves audit/aliases, and respects the reservation model; use existing admin support paths for individual production cases meanwhile.

## Reproduction Assessment

Yes, from source inspection: owner delete sets a reservation, rename rejects any different global slug row, and merge rejects soft-deleted source/target skills. I did not run a live Convex reproduction.

## Solution Assessment

No as a mechanical one-line fix. The safe solution needs an explicit policy for when a deleted slug may move across publishers, then a guarded implementation rather than globally ignoring soft-deleted rows.
