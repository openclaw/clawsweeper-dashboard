## Summary

Current main still leaves a reproducible gap: rename treats a soft-deleted canonical skill as occupied before applying cooldown logic, while merge requires both skills to be live. Keep this open for a publisher-aware reclaim/transfer policy that preserves namespace security.

## What This Changes

Allow an authorized user to reclaim a deleted skill slug or consolidate it across personal and organization ownership.

## Best Possible Solution

Implement one publisher-aware, auditable reclaim or transfer workflow that honors the cooldown, verifies the caller administers the relevant publishers, preserves redirects and history, and rejects unrelated claimants.

## Reproduction Assessment

Yes. Current main deterministically finds and rejects the soft-deleted canonical row before reservation expiry can permit reuse, and the management merge flow is designed around live owned listings.

## Solution Assessment

No, globally releasing deleted slugs would weaken identity protections; the narrow maintainable fix is a publisher-authorized restore, transfer, or merge operation with explicit cooldown and audit semantics.
