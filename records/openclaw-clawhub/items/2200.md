## Summary

Keep open: current main still has a real source-level mismatch for deleted slug reuse. Availability and publish can treat an expired owner-deleted slug as claimable, but rename rejects the soft-deleted row first and merge cannot use a soft-deleted source; the cross-publisher reclaim part still needs ownership/security policy review.

## What This Changes

The issue requests a way to reuse, release, merge, or reclaim a deleted skill slug across personal and organization owners when the actor has authority over both.

## Best Possible Solution

Keep the 30-day owner-delete cooldown, align rename with the existing post-expiry lazy release contract, and add any in-window or cross-publisher reclaim path only behind explicit audited authority checks.

## Reproduction Assessment

Yes, from source inspection: create or assume an owner-deleted skill whose reservation has expired, then rename another owned skill to that slug. Availability and publish can treat the slug as claimable, but rename rejects the soft-deleted row first and merge rejects a soft-deleted source.

## Solution Assessment

Unclear for the full request: immediate reuse or cross-publisher reclaim before the 30-day TTL needs maintainer product/security judgment. The narrow maintainable repair is to align rename with the existing expired owner-unpublish release contract and improve guidance while a slug is still legitimately reserved.
