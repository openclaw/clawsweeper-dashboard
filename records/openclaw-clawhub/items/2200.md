## Summary

Keep open: current main still only handles expired owner-unpublished slug release on availability/fresh publish paths, while rename rejects the soft-deleted slug row and merge rejects soft-deleted source or target rows. The remaining cross-publisher reclaim behavior is authorization-sensitive and needs maintainer policy review.

## What This Changes

The issue requests a way to reuse, release, merge, or reclaim a deleted skill slug when the user controls both involved publishers.

## Best Possible Solution

Define and implement an auditable slug reclaim path that only releases or transfers deleted skill slugs when the caller is authorized for both affected publishers and the reservation policy permits it.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: after the owner-unpublish reservation expires, availability and publish can treat the slug as claimable, but rename still rejects the soft-deleted row and merge still refuses soft-deleted source or target rows.

## Solution Assessment

No current implementation fully solves it. The safest solution is an explicit owner-authorized release or reclaim policy, not broadly ignoring soft-deleted rows in rename or merge.
