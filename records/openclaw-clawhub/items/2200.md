## Summary

Keep open: current main still leaves rename and merge unable to consume a soft-deleted skill slug, even though publish-time release of expired owner-unpublished slugs exists. The remaining cross-owner reclaim behavior is ownership-sensitive and needs a maintainer policy decision before implementation.

## What This Changes

The issue requests a way to reuse, release, merge, or reclaim a deleted skill slug across owners when the same user has permission for both entities.

## Best Possible Solution

Define the cross-publisher deleted-slug reclaim policy first, then implement a narrow backend/UI flow that honors the 30-day owner-unpublish reservation, excludes moderation/security hides, and records audit provenance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create an owner-deleted skill row with an expired unpublished slug reservation, then attempt to rename another manageable skill to that slug or merge from the deleted source; current main rejects those paths before reclaim can happen.

## Solution Assessment

No current implementation fully solves it. The narrow maintainable path is not to ignore soft-deleted rows broadly, but to add an explicit owner-authorized reclaim/release policy and then wire rename, merge guidance, and tests to that policy.
