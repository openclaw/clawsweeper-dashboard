## Summary

Keep open. Current main still has a source-visible deleted-slug collision path in rename and merge, and the cross-publisher reclaim part is ownership-sensitive enough to need maintainer product/security judgment before changing behavior.

## What This Changes

The issue asks for deleted skill slugs to be reusable, releasable, or reclaimable across personal and organization owners when the actor has permission for both sides.

## Best Possible Solution

Keep the 30-day owner-delete reservation, align rename with the existing post-TTL lazy release contract, and require an explicit maintainer-approved authorization/audit policy for in-window or cross-publisher reclaim.

## Reproduction Assessment

Yes, from source inspection for the post-TTL path: after an owner soft-deletes a skill and its reservation expires, rename still rejects the soft-deleted by_slug row before release logic can run, while merge refuses the soft-deleted source. I did not run a live Convex reproduction or inspect production data.

## Solution Assessment

Unclear for the full request because cross-publisher reclaim is an authorization policy decision. The narrow maintainable fix is to make rename follow the same expired owner-unpublish release contract as availability and publish, then separately decide any self-service reclaim flow.
