## Summary

Keep open: current main still only releases expired owner-unpublished slugs on the publish path, while rename still rejects the soft-deleted slug row and merge still rejects soft-deleted source or target rows. The remaining cross-owner reclaim behavior is authorization-sensitive and needs maintainer policy review.

## What This Changes

The issue requests a way to reuse, release, merge, or reclaim a soft-deleted skill slug across owners when the actor has permission for both entities.

## Best Possible Solution

Define the cross-publisher deleted-slug reclaim policy first, then implement a narrow backend/UI flow that honors the 30-day owner-unpublish reservation, excludes moderation/security hides, and records audit provenance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: after an owner-deleted slug reservation expires, availability and publish can treat it as claimable, but rename still rejects the existing soft-deleted row and merge rejects soft-deleted source or target rows.

## Solution Assessment

No current implementation fully solves it. The safest path is an explicit owner-authorized reclaim or release policy, not broadly ignoring soft-deleted rows in rename or merge.
