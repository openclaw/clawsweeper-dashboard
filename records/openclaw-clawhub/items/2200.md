## Summary

Keep open: current main still has a source-level mismatch where an expired owner-deleted slug can be shown as claimable and publish can lazily release it, but rename rejects the existing soft-deleted row and merge cannot use a soft-deleted source. The personal/org reclaim part remains an ownership and security policy decision.

## What This Changes

Request to make deleted skill slugs reusable or reclaimable across personal and organization owners when rename and merge currently cannot resolve the collision.

## Best Possible Solution

Keep the 30-day owner-delete reservation, make rename follow the existing post-expiry publish release contract, and add any in-window or cross-publisher reclaim path only with explicit audited authority checks.

## Reproduction Assessment

Yes, from source inspection: after an owner-deleted skill's reservation expires, availability and publish can treat the slug as claimable, but rename rejects the soft-deleted row first and merge rejects a soft-deleted source. I did not run a live Convex reproduction or inspect production data.

## Solution Assessment

Unclear for the full request: immediate reuse or cross-publisher reclaim before the 30-day TTL needs maintainer product/security judgment. The narrow maintainable fix is to align rename with the existing expired owner-unpublish release contract and improve guidance while a slug is still legitimately reserved.
