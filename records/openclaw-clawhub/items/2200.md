## Summary

Keep open. Current main still has a source-visible deleted-slug collision path: availability and publish honor the expired owner-unpublish reservation contract, but rename rejects the soft-deleted `skills.by_slug` row before release logic and merge refuses soft-deleted sources; cross-owner reclaim also needs explicit security/product policy.

## What This Changes

The issue requests a way to reuse, release, merge, or reclaim a deleted skill slug across owners when the actor has authority over both entities.

## Best Possible Solution

Keep the 30-day owner-delete reservation, align rename with the existing post-expiry publish/availability release behavior, and handle in-window or cross-publisher reclaim only through audited publisher authority checks or a staff support flow.

## Reproduction Assessment

Yes, from source inspection: an expired owner-unpublished deleted slug can be reported available and released on publish, but rename still collides with the soft-deleted `by_slug` row and merge rejects a soft-deleted source. I did not run a live Convex reproduction or inspect production data.

## Solution Assessment

Unclear for the full requested self-service reclaim path because cross-publisher ownership transfer is security-sensitive. The narrow maintainable fix is to reuse the existing expired-reservation release contract for rename and reserve broader in-window reclaim for explicit maintainer policy.
