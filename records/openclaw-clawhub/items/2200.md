## Summary

Keep open. Current main still has a source-visible deleted-slug gap: availability and publish recognize expired owner-unpublish reservations, but rename rejects the soft-deleted slug row before release logic and merge rejects soft-deleted sources; the cross-publisher reclaim part also needs explicit security/product policy.

## What This Changes

Allow a deleted skill slug to be reused, released, or reclaimed/merged across personal and organization publishers when the user has the right permissions.

## Best Possible Solution

Keep the documented 30-day owner-delete reservation, align rename/publish/availability after expiry, and route in-window or cross-publisher reclaim through audited authority checks for both source and target publishers.

## Reproduction Assessment

Yes, from source inspection: after a manageable skill is soft-deleted, renaming another manageable skill to that expired slug still hits the soft-deleted skills.by_slug row and fails before release logic, while merge rejects a soft-deleted source. I did not run a live Convex reproduction or inspect production data.

## Solution Assessment

Unclear for the full request: immediate reuse or cross-publisher reclaim before the 30-day TTL needs maintainer product/security judgment. The narrow maintainable fix is to align rename with the existing expired owner-unpublish release contract and improve guidance while a slug is still legitimately reserved.
