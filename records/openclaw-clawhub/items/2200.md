## Summary

Keep open: current main still has only a publish-time lazy release for expired owner-deleted slugs; rename still rejects the soft-deleted row and merge rejects soft-deleted source/target rows. The remaining cross-owner reclaim behavior is security-sensitive publisher ownership policy, and the broader owner-scoped slug PR is still open and not a safe superseding merge path yet.

## What This Changes

The issue requests a self-serve way to reuse, release, merge, or reclaim a deleted skill slug across personal and organization owners when the actor has permission for both entities.

## Best Possible Solution

Define the cross-owner deleted-slug reclaim policy first, then implement a scoped backend/UI flow that honors owner authority, 30-day reservations, moderation/security exclusions, and audit provenance.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: availability and publish can treat an expired owner-unpublished slug as claimable, but rename still rejects the existing soft-deleted row and merge rejects soft-deleted source or target rows.

## Solution Assessment

Unclear. A reclaim/release flow is plausible, but the best implementation depends on maintainer policy for cross-owner authorization, TTLs, protected moderation states, owner-scoped slug work, and audit records.
