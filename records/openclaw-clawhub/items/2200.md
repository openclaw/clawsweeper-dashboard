## Summary

Keep open. Current main has a documented 30-day owner-unpublish reservation and a post-expiry lazy release path for new publishes, but rename still blocks on soft-deleted slug rows and merge cannot consume the deleted row, so the report exposes a real lifecycle gap that needs maintainer policy judgment.

## What This Changes

The issue asks for deleted skill slugs to become reusable or reclaimable across owners when the user can manage both owners.

## Best Possible Solution

Maintainers should define whether same-actor cross-owner reclaim is allowed during the 30-day hold, then make rename, merge, publish, availability, UI copy, docs, and tests follow one deleted-slug lifecycle.

## Reproduction Assessment

Yes for the source-level mismatch: a rename to a slug held by a different soft-deleted `skills.by_slug` row reaches the collision branch, while merge rejects deleted source or target rows. The exact live personal/org ownership state from the report was not verified.

## Solution Assessment

Unclear. A narrow code fix could make rename honor the documented post-expiry release path, but immediate cross-owner reclaim before the 30-day reservation needs maintainer product/security judgment.
