## Summary

Keep open: source inspection still supports the report, and ClawHub cleanup policy keeps issues open. The browser publish form can block an org member at slug preflight even though the backend publish path authorizes publisher/admin/owner membership for the selected org.

## Plan

Fix issue 2171: in the browser skill publish route, an authenticated member with publisher/admin/owner rights on the selected current org owner must be able to publish a new version of an existing org-owned skill even when they are not skill.ownerUserId. Current main calls api.skills.checkSlugAvailability with only the slug, receives a taken result, adds it to validation, and disables submit before publishVersion can authorize the org membership. Keep the backend publishVersion/insertVersion checks as the final authority; do not loosen owner migration, migrateOwner confirmation, or admin requirements for moving a skill between publishers. Prefer a publisher-aware slug preflight for the selected owner or an update-mode same-owner exemption that is backed by server authorization and tests. Add regression coverage showing same-owner org republish works for a publisher/admin member, unrelated slug collisions still block, and owner migration still requires explicit confirmation and the existing migration role checks. Update specs/docs only if the intended behavior wording changes.
