# Coding Plan for openclaw/clawhub#2171: \"ORG publisher/admin\" user cannot publish skills in ORG

Source report: [records/openclaw-clawhub/items/2171.md](records/openclaw-clawhub/items/2171.md)

## Summary

Keep open: ClawHub issue policy prevents auto-closing issues, and source inspection supports the report. The browser publish form still blocks org members in slug-availability preflight even though the backend publish path accepts publisher/admin membership for the selected org owner.

## Plan

Fix the browser skill publish preflight so org publisher/admin members can publish a new version of an existing org-owned skill after transfer. Current main allows publishVersion to authorize the selected ownerHandle via publisher membership, but src/routes/skills/publish.tsx calls convex/skills.checkSlugAvailability with only the slug, and convex/skills.ts treats an existing slug as available only for skill.ownerUserId or duplicate GitHub identity. Implement the narrowest safe fix, likely by making slug availability publisher-aware for the selected/current owner or by suppressing the collision only in update mode when the selected owner matches the existing skill owner and the actor has publish rights. Do not weaken insertVersion/publishVersion ownership migration checks, do not allow strangers to claim existing slugs, and preserve explicit migrateOwner semantics. Add regression coverage for an org-owned skill where a member with role publisher and a member with role admin can pass the publish preflight for the same owner, plus a non-member collision case that remains blocked. Update specs/docs only if the implementation changes intended behavior beyond the existing documented org publishing model.
