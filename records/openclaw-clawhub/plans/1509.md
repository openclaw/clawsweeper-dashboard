# Coding Plan for openclaw/clawhub#1509: Bug: clawhub publish fails with 'SKILL.md required' for new skills

Source report: [records/openclaw-clawhub/items/1509.md](records/openclaw-clawhub/items/1509.md)

## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a source-reproducible publish path where a relative source can be resolved under the configured global workdir rather than the caller's skill folder.

## Plan

Fix the remaining `clawhub skill publish .` / legacy `clawhub publish .` path-resolution bug for issues https://github.com/openclaw/clawhub/issues/1509 and https://github.com/openclaw/clawhub/issues/48. Current `cmdPublish` resolves relative sources with `resolve(opts.workdir, folderArg)`, so when the global workdir comes from Clawdbot/OpenClaw config and differs from the caller's cwd, publish can inspect the wrong folder and throw `SKILL.md required`. Implement a conservative publish-specific resolution rule that treats user-entered relative source paths as relative to `process.cwd()` when that resolves to an existing skill folder with root `SKILL.md` or `SKILLS.md`, while preserving absolute paths, explicit invalid-path failures, plugin-folder detection, and non-publish workdir behavior. Add focused regression coverage for a cwd skill folder with a different configured/global workdir, and keep the existing broad-ignore manifest recovery test intact. Do not change backend publish validation or general install/update/sync workdir semantics.
