# Coding Plan for openclaw/clawhub#2132: Dashboard does not expose org publisher packages after transfer/publish

Source report: [records/openclaw-clawhub/items/2132.md](records/openclaw-clawhub/items/2132.md)

## Summary

Keep open: current main still defaults `/dashboard` to the personal publisher and provides no UI path to switch to an organization publisher, while the backend already supports membership-checked org dashboard reads and the orgs spec calls for dashboard owner selection.

## Plan

Fix org publisher visibility on `/dashboard` for users who belong to both a personal publisher and one or more organization publishers. Add a visible publisher selector when `api.publishers.listMine` returns multiple publishers, keep the personal publisher as the default, update `selectedPublisherId` from that selector, and ensure dashboard skills, packages, empty states, and `New Skill` / `New Plugin` quick actions use the selected publisher handle/id. Reuse the existing owner-selector patterns from `src/routes/plugins/publish.tsx` and `src/routes/skills/publish.tsx`. Add focused regression coverage in `src/routes/-dashboard.test.tsx` for a personal-plus-org publisher list: the default personal query remains intact, switching to the org passes `ownerPublisherId` to skills/packages queries, and the plugin publish link carries the org owner handle. Do not change backend authorization, package search/indexing, publisher membership rules, or add a config option. Update specs only if the implementation changes the documented org-dashboard intent.
