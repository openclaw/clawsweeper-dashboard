# Coding Plan for openclaw/clawhub#651: why

Source report: [records/openclaw-clawhub/items/651.md](records/openclaw-clawhub/items/651.md)

## Summary

Keep open. Current main has proxy support and device-code login, but search, token login, device-flow fetches, and registry discovery can still expose raw transport failures, so the central CLI `fetch failed` problem is not fully implemented on main.

## Plan

Fix the remaining raw ClawHub CLI transport failure surface reported here: `clawhub search` and login-related commands can still show only `fetch failed` when the underlying network request fails. Keep the scope narrow to CLI HTTP/discovery/device-auth diagnostics and shared transport behavior. Prefer routing device-flow and registry-discovery requests through the same timeout/proxy/error-normalization path used by `packages/clawhub/src/http.ts`, or add a small shared helper that turns network `TypeError: fetch failed`, DNS/TLS/proxy/timeouts, and curl transport failures into actionable CLI messages mentioning registry/site, proxy environment variables, and troubleshooting docs. Preserve existing HTTP status handling, retries, rate-limit messages, token verification, and config behavior. Do not add new config flags or change API contracts. Update focused tests for search/login/device/discovery error messages and docs only if the user-facing remediation changes.
