# Coding Plan for openclaw/clawhub#651: why

Source report: [records/openclaw-clawhub/items/651.md](records/openclaw-clawhub/items/651.md)

## Summary

Keep open. Current main still has a source-supported path where CLI search and token login can surface low-level transport failures as raw `fetch failed`, and ClawHub issue cleanup policy does not auto-close issues that remain actionable.

## Plan

Fix the ClawHub CLI `fetch failed` diagnostics for issue https://github.com/openclaw/clawhub/issues/651 and sibling https://github.com/openclaw/clawhub/issues/861. Keep request behavior unchanged, but add centralized, redacted transport-error classification in the shared HTTP/error-formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance for `search`, token login, browser-login verification, and other CLI commands. Do not assume every `fetch failed` is a proxy issue. Preserve `/api/v1/whoami` verification before writing tokens. Add focused regression coverage for Node fetch errors, Bun curl failures, formatter redaction, login verification, and search display. Update troubleshooting docs if user guidance changes. If adding a changelog entry, keep it concise and do not add a bot Thanks line.
