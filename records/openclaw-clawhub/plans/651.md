# Coding Plan for openclaw/clawhub#651: why

Source report: [records/openclaw-clawhub/items/651.md](records/openclaw-clawhub/items/651.md)

## Summary

Keep open. Current main has proxy support and device-code login, but the reported `search` and token-login paths can still surface low-level transport failures as raw `fetch failed`, so the central CLI diagnostics problem is not certainly implemented on main.

## Plan

Fix the ClawHub CLI `fetch failed` UX for search and login by adding centralized, sanitized transport error classification in the shared HTTP/error formatting boundary. Preserve existing proxy support, device-code login default, token verification, retries, timeout semantics, rate-limit/HTTP status handling, and Bun curl behavior. Cover DNS, TLS/certificate, proxy, timeout/abort, and curl stderr cases with actionable redacted messages; do not print Authorization headers, API tokens, callback URLs, proxy credentials, or private endpoints. Related refs: https://github.com/openclaw/clawhub/issues/651, https://github.com/openclaw/clawhub/issues/861, https://github.com/openclaw/clawhub/pull/363, and https://github.com/openclaw/clawhub/pull/245.
