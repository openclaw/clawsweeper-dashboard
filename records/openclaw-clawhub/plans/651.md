## Summary

Keep open: current main and the latest CLI release still send search and token-login transport failures through `formatError`, which returns raw `Error.message`, so the reported `fetch failed` UX remains possible even after proxy support landed.

## Plan

Fix the ClawHub CLI raw `fetch failed` diagnostics reported here and in related fetch-failure reports. Add centralized, redacted transport diagnostics in the shared HTTP/error-formatting boundary so Node/Undici and Bun curl failures for search, token login, whoami, install/download, publish/package, and sync produce actionable messages for DNS, TLS/certificate, proxy, timeout/abort, and curl transport failures. Preserve existing request semantics, HTTP status handling, retry/rate-limit behavior, proxy behavior, Bun curl behavior, and `/api/v1/whoami` token verification. Do not add a new config option or assume every `fetch failed` is a proxy issue. Redact Authorization headers, API tokens, callback token/state parameters, proxy credentials, and private endpoints. Add focused regression coverage for the formatter/HTTP path and relevant CLI command surfaces; update docs only if the emitted guidance changes user-facing troubleshooting.
