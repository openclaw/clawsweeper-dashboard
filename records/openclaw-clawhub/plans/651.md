## Summary

Keep open. Current main still has a source-supported CLI path where search and token login can render low-level transport failures as the raw top-level `fetch failed` message, and ClawHub cleanup policy keeps issues open unless there is a stronger terminal reason.

## Plan

Fix the ClawHub CLI raw `fetch failed` diagnostics for issue https://github.com/openclaw/clawhub/issues/651, clustered with https://github.com/openclaw/clawhub/issues/861. Add centralized transport-error classification/redaction in the shared HTTP/error formatting boundary so search and login failures from Undici, DNS, TLS/certificates, proxy configuration, timeout/abort, and Bun curl stderr produce actionable guidance instead of only `fetch failed`. Preserve successful request behavior, HTTP status handling, retries, rate-limit hints, timeout semantics, Bun curl behavior, and token verification before config writes. Redact API tokens, Authorization headers, token query parameters, callback URLs, proxy credentials, and private endpoints. Add focused regression coverage for the formatter/HTTP path and affected search/login command output. Update troubleshooting docs only for user-facing guidance that is not already covered. Do not add a new config option or change registry/proxy semantics; leave changelog entry handling to maintainers unless explicitly requested.
