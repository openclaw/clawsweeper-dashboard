## Summary

Keep open. Current main still has a clear CLI path where search/login transport failures can surface as the raw top-level `fetch failed` message, and ClawHub cleanup policy does not auto-close issues.

## Plan

Fix the ClawHub CLI raw `fetch failed` diagnostics for search/login and the shared HTTP transport path. Add centralized, redacted transport-error classification so Undici fetch failures, DNS/TLS/certificate/proxy/timeout/abort cases, and Bun curl failures produce actionable CLI guidance without changing HTTP status handling, retries, rate-limit messages, successful requests, or token verification semantics. Include regression coverage for `cmdSearch`, token login or shared formatter behavior, Node fetch errors with nested causes, and Bun curl stderr. Update troubleshooting docs only if the user-facing guidance changes. Do not assume every `fetch failed` is a proxy issue, and do not expose Authorization headers, API tokens, token query parameters, callback URLs, proxy credentials, or private endpoint details.
