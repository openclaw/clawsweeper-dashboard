# Coding Plan for openclaw/clawhub#651: why

Source report: [records/openclaw-clawhub/items/651.md](records/openclaw-clawhub/items/651.md)

## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still has a source-supported path where CLI search and login transport failures can surface only as raw `fetch failed` messages.

## Plan

Fix the ClawHub CLI raw `fetch failed` diagnostics reported by https://github.com/openclaw/clawhub/issues/651 and related https://github.com/openclaw/clawhub/issues/861. Add centralized, redacted transport-error classification in the shared HTTP/error formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and Bun curl failures produce actionable guidance for search and login without changing successful request behavior. Preserve `/api/v1/whoami` token verification and do not write unverified browser tokens. Do not add a new config option or assume every failure is a proxy issue. Redact API tokens, Authorization headers, callback URL tokens, query/hash token values, proxy credentials, and private endpoints. Add focused regression tests for the formatter/HTTP path and login/search-facing behavior, and update troubleshooting docs if user-facing guidance changes. Include a CHANGELOG entry only if maintainers require it for release preparation.
