## Summary

Keep open. Current main includes proxy support and a device-code login default, but `search`, token login, and device-flow requests can still surface transport failures through raw `fetch failed`/curl messages, so the central CLI problem is not certainly implemented on main.

## Plan

Fix this issue by improving ClawHub CLI transport failure diagnostics without changing auth or registry semantics. Add a centralized redacted formatter/helper for Node undici `fetch failed` causes, timeout/abort errors, DNS/TLS/proxy connection codes, and Bun curl stderr; use it from shared command error formatting and device-flow fetch failures so `clawhub search`, `clawhub login --token`, and default device login produce actionable guidance. Preserve existing HTTP status handling, retries, rate-limit messages, proxy support, and token verification; do not store unverified tokens and do not print API tokens, Authorization headers, callback URLs, proxy credentials, or private endpoints. Update troubleshooting/auth docs only where they describe current behavior, including the current device-code default if touched.
