## Summary

Keep open: current main still routes `search` and token-login transport failures through the shared formatter as raw `Error.message`, so users can still see only `fetch failed` despite proxy support and docs already landing.

## Plan

Implement centralized, redacted ClawHub CLI transport diagnostics for network failures without changing request semantics. Classify Undici-style `TypeError('fetch failed')` nested causes and Bun curl stderr/status for DNS, TLS/certificate, proxy/refused, timeout, and abort failures; include concise actionable hints for proxy env vars and troubleshooting. Preserve HTTP status errors, retry/rate-limit behavior, timeout behavior, proxy support, Bun curl behavior, and `/api/v1/whoami` verification before token config writes. Do not print Authorization headers, `clh_` tokens, callback URL token/state params, proxy credentials, or private endpoint details. Cover `search` and login/token-verification paths with focused tests and update troubleshooting docs only if user-facing guidance changes.
