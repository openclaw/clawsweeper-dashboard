## Summary

Keep open. Current main still routes CLI search and token-login transport failures through a formatter that returns only the raw top-level message, so the reported `fetch failed` output remains a narrow actionable diagnostics bug rather than a cleanup-close candidate.

## Plan

Fix the ClawHub CLI `fetch failed` diagnostics reported in this issue and the related login/search cluster. Add centralized, redacted transport-error classification at the shared HTTP/error-formatting boundary so search, token login, browser-login verification, and other CLI network commands surface actionable guidance for DNS, TLS/certificate, proxy, timeout/abort, and curl failures without changing request semantics. Preserve `/api/v1/whoami` verification and existing HTTP status/rate-limit behavior. Add regression coverage for Undici-style `TypeError('fetch failed')`, non-Error throws, curl stderr/`curl failed`, and redaction of tokens, Authorization headers, callback URL tokens, proxy credentials, and private endpoints. Update troubleshooting docs only where they explain current CLI behavior; do not add a new config option or skip auth verification.
