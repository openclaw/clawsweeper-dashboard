Keep open. Current main still has a source-supported path where CLI search and token-login transport failures are surfaced as raw `fetch failed` messages, and the merged proxy/Node fixes cover specific causes rather than the remaining diagnostics gap.

## Plan

Fix the ClawHub CLI `fetch failed` diagnostics without changing HTTP request semantics. Add a centralized, redacted transport-error formatter/classifier in the shared CLI HTTP/error boundary so `cmdSearch`, `cmdLogin`, browser/device login verification, install, publish, package, and sync failures can surface actionable hints for DNS, TLS/certificate, proxy, timeout/abort, and curl failures. Preserve `/api/v1/whoami` verification and do not write unverified tokens. Redact Authorization headers, `clh_`-style tokens, token query/hash values, callback URLs, proxy credentials, and private endpoint details. Include regression coverage for Node fetch `TypeError('fetch failed')` with nested causes, Bun curl stderr/non-zero failures, search output, token-login/browser-login verification output, and existing HTTP status/rate-limit behavior. Keep the change focused; do not introduce a new config flag or alter proxy routing behavior.
