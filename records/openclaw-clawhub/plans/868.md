## Plan

Fix the `clawhub sync` batch publish behavior for slug-ownership conflicts. On current main, `packages/clawhub/src/cli/commands/sync.ts` directly awaits `cmdPublish` for each selected skill, so one backend error like `Slug is already taken. Choose a different slug. Existing skill: /owner/slug` aborts the whole sync. Add focused per-skill handling around the publish loop so the known slug-taken conflict is recorded as skipped or failed with the existing skill URL when present, sync continues to later selected skills, and the final output accurately reports uploaded vs skipped/failed work. Add regression coverage in `packages/clawhub/src/cli/commands/sync.test.ts` that mocks one selected `cmdPublish` rejection followed by another selected publish and verifies continuation plus summary semantics.
