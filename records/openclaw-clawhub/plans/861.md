Keep open. Current main still has a source-supported CLI login path where browser auth can succeed, then token verification against `/api/v1/whoami` can fail through the shared HTTP client and surface only the generic top-level `fetch failed` message.

## Plan

Fix the ClawHub CLI raw `fetch failed` diagnostics gap for login verification and shared transport failures. Preserve the security invariant that `cmdLogin` verifies the browser/token/device token with `/api/v1/whoami` before writing config; do not save unverified tokens. Add centralized, redacted transport diagnostics for Node fetch errors and Bun curl failures so DNS, TLS/certificate, proxy, timeout/abort, and generic Undici `fetch failed` causes produce actionable guidance without leaking tokens, Authorization headers, callback URLs, proxy credentials, or private endpoints. Coordinate with related report https://github.com/openclaw/clawhub/issues/651 and keep the fix narrow; do not add a new config flag or bypass existing retries/status/rate-limit behavior.
