# Coding Plan for openclaw/clawhub#861: fetch fail problem

## Summary

Keep open. Current main still has a source-supported CLI path where browser login can receive a token and then fail `/api/v1/whoami` verification with only the raw `fetch failed` transport message; the related open report shows the same shared diagnostics gap.

## Plan

Fix the ClawHub CLI raw `fetch failed` diagnostics gap for browser login verification and the shared HTTP path. Preserve browser-token `/api/v1/whoami` verification and do not write unverified tokens to config. Add a centralized redacted transport-error formatter/classifier for Undici fetch failures and Bun curl failures so DNS, TLS/certificate, proxy, timeout/abort, and generic curl failures produce actionable guidance without changing request semantics, retries, HTTP status handling, rate-limit hints, or successful responses. Add login-specific guidance that points users to proxy troubleshooting and `clawhub login --token` as a fallback when browser auth verification cannot reach the registry. Update focused tests for `formatError`/HTTP failure handling plus auth/login verification failure behavior, and update troubleshooting docs. Do not add new config flags or bypass authentication verification.
