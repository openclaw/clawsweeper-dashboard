# Coding Plan for openclaw/clawhub#1300: Bug: transfer request, skill rename, and skill merge fail with double JSON serialization

Source report: [records/openclaw-clawhub/items/1300.md](records/openclaw-clawhub/items/1300.md)

## Summary

Keep open. Current main still has the reported CLI serialization bug: transfer request, skill rename, and skill merge pre-stringify request bodies before handing them to an HTTP helper that serializes JSON bodies again, and the server handlers then read missing object fields. The related focused fix PRs were closed without merging, while the open ownership-transfer PR only overlaps part of this issue and is currently conflicting.

## Plan

Fix the current-main CLI double JSON serialization bug for `clawhub transfer request`, `clawhub skill rename`, and `clawhub skill merge`. In `packages/clawhub/src/cli/commands/transfer.ts` and `packages/clawhub/src/cli/commands/ownership.ts`, pass plain object bodies to `apiRequest` instead of pre-stringified JSON. Update `packages/clawhub/src/cli/commands/transfer.test.ts` and `packages/clawhub/src/cli/commands/ownership.test.ts` so they assert object payloads, not serialized substrings. If you harden `packages/clawhub/src/http.ts` to tolerate pre-serialized strings, use one shared serializer and add/update coverage in both `packages/clawhub/src/http.test.ts` and `packages/clawhub/src/http.bun.test.ts`; otherwise keep `apiRequest` as the single serialization layer. Coordinate with overlapping open PR https://github.com/openclaw/clawhub/pull/1635, but do not change org-transfer approval/product semantics while fixing this bug. Add a concise `CHANGELOG.md` fix entry if the current changelog format covers CLI fixes, and add a focused spec note only if you formalize a new CLI/HTTP request-body contract.
