## What This Changes

The PR rewrites `skills.checkSlugAvailability` to avoid global `.unique()` slug lookups, adds owner+slug indexes for skills and slug aliases, and expands slug-availability regression tests.

## Best Possible Solution

Land a narrow backend fix that returns normal availability results for duplicate scoped slug data, after contributor-provided real behavior proof and maintainer validation of the deploy/index path.

## Reproduction Assessment

Yes, from source inspection: current main uses Convex `.unique()` on slug indexes in `checkSlugAvailability`, which throws when duplicate slug rows exist. The linked reports provide production request IDs for the same failing query, but I did not run a live Convex reproduction in this read-only pass.

## Solution Assessment

Yes, for the backend availability-query crash: owner+slug indexes plus bounded `take(2)` queries are a maintainable way to avoid Convex `unique()` exceptions without scanning. The PR should still wait for real behavior proof and the broader project gates before merge.
