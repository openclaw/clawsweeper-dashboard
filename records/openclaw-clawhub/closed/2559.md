## What This Changes

The issue reports that `clawhub install` and forced updates can write different `installedAt` values to `.clawhub/origin.json` and `.clawhub/lock.json`, causing installed skills to be flagged invalid.

## Best Possible Solution

Make `installedAt` a single shared value for each install/update write, preserve existing public metadata formats and pin metadata, and add regression tests that assert exact timestamp equality between origin metadata and the lockfile.

## Reproduction Assessment

Yes by source inspection: forced update writes the preserved origin timestamp and a fresh lockfile timestamp on current main. I did not run the CLI because this review was read-only, but the forced-update mismatch is clear from the code path.

## Solution Assessment

Yes: the narrowest maintainable fix is to choose one `installedAt` value per affected skill operation and pass it to both `writeSkillOrigin` and `withPinnedMetadata`, with tests that compare the exact values.

## Repair Work Prompt

Fix the `installedAt` mismatch reported in https://github.com/openclaw/clawhub/issues/2559. In `packages/clawhub/src/cli/commands/skills.ts`, make `clawhub install` and forced update paths choose one `installedAt` value per affected skill and write that exact value to both `.clawhub/origin.json` and `.clawhub/lock.json`. Preserve pin metadata, existing update semantics, and public file formats; if origin metadata preserves the original install time, the lockfile should receive the same preserved value. Add focused regression coverage in `packages/clawhub/src/cli/commands/skills.test.ts` for fresh install and forced update that compares the exact `installedAt` values written to `writeSkillOrigin` and `writeLockfile` instead of only using `expect.any(Number)`. Do not change unrelated CLI behavior, telemetry, package APIs, or docs unless a small fixture update is required.
