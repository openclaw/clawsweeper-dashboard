## Summary

Keep open: current main still maps package moderation queue timestamps from `release.createdAt` directly, so this PR is not implemented on main. The proposed fallback to Convex `_creationTime` looks like the right narrow fix, but the PR still needs after-fix real behavior proof and the required changelog entry before merge.

## What This Changes

The PR makes package moderation queue items fall back to Convex `_creationTime` when a legacy package release row lacks `createdAt`, and adds regression coverage for that path.

## Best Possible Solution

Land the narrow `_creationTime` fallback with regression coverage, an Unreleased changelog entry, and contributor-supplied after-fix CLI/API output proving the moderation queue command succeeds against a real setup.

## Reproduction Assessment

Yes from source inspection and the PR body: current main maps `release.createdAt` directly into a response field that the public schema and CLI require to be numeric, and the reporter observed real CLI server errors for `blocked` and `manual` queue status before the fix. I did not run a live Convex reproduction in this read-only review.

## Solution Assessment

Yes, the proposed queue-item fallback to Convex `_creationTime` is the narrowest maintainable fix for legacy release rows in this moderation queue path. Separate package status surfaces that also read release timestamps directly should be handled only if they get their own reproduction or scope.
