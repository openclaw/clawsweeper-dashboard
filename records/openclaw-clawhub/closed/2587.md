## Summary

Current main still throws when more than one device authorization row has the same user-code hash, matching the reported failure. The issue should remain open because https://github.com/openclaw/clawhub/pull/2589 explicitly claims it and is still unmerged; review and validate that PR rather than opening a competing fix.

## What This Changes

Make CLI device approval and denial tolerate duplicate server-side user-code rows so the CLI can receive its authentication token.

## Best Possible Solution

Land the linked narrow fix after real Convex and end-to-end CLI validation proves it selects only the newest active pending request, safely handles stale duplicates, and issues exactly one token.

## Reproduction Assessment

Yes at source level. Duplicate `userCodeHash` rows deterministically cause Convex `.unique()` to throw in both approval and denial, matching the supplied production request IDs and error path.

## Solution Assessment

Yes in direction: replacing the invalid uniqueness assumption with deterministic active-row handling is narrow and maintainable. The linked PR still needs real Convex and end-to-end device-login proof because this is an authentication boundary.
