## Summary

Keep open: the direction is useful, but the router predicate in the PR never matches ClawScan finding hashes because hash strings include the leading `#`, so the duplicate scroll-restoration opt-out described by the PR does not take effect. The PR also needs inspectable real behavior proof before merge.

## What This Changes

The PR removes ClawScan’s component-level hash scroll effect, adds a router scroll-restoration opt-out for `/security/clawscan` finding hashes, and removes the old `scrollIntoView` test.

## Best Possible Solution

Fix the router hash predicate to include `#`, keep or add focused regression coverage for the scroll-restoration opt-out, and attach inspectable browser proof for direct ClawScan finding hash navigation.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the blocking PR bug: TanStack’s parsed hash keeps the leading `#`, while the PR checks for a prefix without it. I did not run the browser preview in this read-only review.

## Solution Assessment

No. The intended router-level opt-out is a reasonable direction, but the current predicate disables none of the intended ClawScan finding hashes; the narrow fix is to include the leading `#` and cover that behavior.
