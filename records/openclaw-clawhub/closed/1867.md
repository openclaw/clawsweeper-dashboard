## Summary

Keep open. Current main does not implement device-flow auth, and this PR is still a client-only slice that exposes a public login mode before the required server endpoints, verification UI, security controls, changelog, and real behavior proof exist.

## What This Changes

Adds a `--device` login option to the ClawHub CLI, client-side device-code request/polling helpers, and unit tests for the helper.

## Best Possible Solution

Land device flow as an end-to-end auth feature covering CLI behavior, backend device-code issuance and token exchange, an authenticated verification page, docs, changelog, and regression coverage.

## Reproduction Assessment

Yes for the review findings from source inspection: the PR head exposes `--device` and posts to `/cli/device/code` and `/cli/device/token`, while current main has no matching routes or verification page. I did not run the flow live because the missing server surface is clear in source.

## Solution Assessment

No. The client helper is a useful slice, but exposing the flag before backend issuance, polling exchange, verification UI, and security controls land is not the narrowest maintainable solution.
