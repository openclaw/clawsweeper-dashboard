## Summary

Keep open: the intended timestamp fix is useful, but the branch is not merge-ready because the shared helper change would also suppress legitimate publish/update timestamps for manually overridden skills.

## What This Changes

The PR changes scanner moderation sync so skill-level moderation patches do not update the public skill `updatedAt`, and adds regression tests for LLM scan sync with and without manual overrides.

## Best Possible Solution

Keep scanner refresh patches timestamp-free at scanner/moderation boundaries while preserving caller-provided `updatedAt` for user-visible publishes, moderator actions, and other real skill updates.

## Reproduction Assessment

Yes from source inspection: current main scanner sync paths write `updatedAt: now`, and the PR’s changed helper would strip `updatedAt` from any manual-override skill patch. I did not run tests because this review is read-only, but the source path is direct.

## Solution Assessment

No, not as written. The scanner timestamp preservation should be scoped to scanner refreshes, not implemented by removing `updatedAt` inside a shared helper used by publishing and other user-visible updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep publish updates from losing updatedAt:** `convex/skills.ts:427-428`
  - body: This helper is also used by the skill publish/update path after it builds a real content patch with `updatedAt: now`. Stripping `updatedAt` here means a publisher can release a new latest version for a skill with a manual override and the public timestamp will not move, breaking recency sorting and “Last updated” surfaces. Please scope the timestamp suppression to scanner refreshes, or otherwise preserve caller-provided `updatedAt` for user-visible updates.
  - confidence: 0.88
