## Summary

Keep open. The report identifies a real public API contract gap on current main: many v1 error paths still return plain text, OpenAPI often documents only status descriptions, and choosing plain text, negotiated JSON, or a future-version envelope needs maintainer compatibility judgment.

## What This Changes

Request to clarify or standardize public v1 API error responses, including whether failures stay text/plain or gain a stable JSON envelope and error-code taxonomy.

## Best Possible Solution

Define the v1 error contract in specs, docs, and OpenAPI first, then implement either documented plain-text compatibility or a negotiated/versioned JSON envelope with regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction: this is an API contract clarification request, not a broken already-defined behavior. Source inspection does confirm the reported current behavior on main for skill, package, resolve, download, and 429 error paths.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. The narrow maintainable path is to decide and document the contract before changing runtime responses, because text/plain, Accept-negotiated JSON, and a future-version JSON envelope all have different client-compatibility costs.
