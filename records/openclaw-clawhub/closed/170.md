## Summary

Keep open. Current main has stable numeric GitHub ID mapping and post-login profile sync, but this is an open ClawHub issue and the live GitHub sign-in start route still returns HTTP 500, so the username-change account-state failure remains unresolved and needs maintainer/operator triage.

## What This Changes

Maintain GitHub OAuth sign-in across GitHub username changes by anchoring account identity to the immutable GitHub user ID and refreshing mutable username/profile fields.

## Best Possible Solution

Restore the production GitHub OAuth redirect/session path, keep identity anchored to numeric providerAccountId, inspect affected production auth rows, and add a maintainer-owned recovery or consolidation path only if triage proves one is needed.

## Reproduction Assessment

Yes, partially: I reproduced the currently observable production GitHub sign-in start failure with both sign-in endpoints returning HTTP 500 on May 10, 2026 while HEAD is tagged prod/v2026.05.10.8. I did not isolate the username-change-specific verifier/relink failure without an affected account and production auth rows.
