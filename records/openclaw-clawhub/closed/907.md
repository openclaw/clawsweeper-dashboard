## Label Justifications

- `P1`: Multiple real users are blocked from a core GitHub sign-in/account recovery workflow after account deletion.
- `impact:security`: Any recovery path changes identity binding, deletion privacy, token revocation, and ban separation semantics.
- `impact:auth-provider`: The failure occurs during GitHub OAuth completion and Convex Auth user binding for a previously deleted identity.
- `impact:session-state`: Affected users return from auth without an active ClawHub session and remain logged out.

## Summary

Keep open. Current main still treats user-initiated deletion as irreversible and blocks the same GitHub identity during OAuth completion; changing that requires maintainer product and security review rather than conservative cleanup.

## What This Changes

The issue asks for users to be able to sign up or log in again with the same GitHub account after deleting a prior ClawHub account.

## Best Possible Solution

Maintainers should decide and document the deletion policy, then either implement a secure same-GitHub recreate/relink flow that preserves abuse protections or keep irreversible deletion and make the OAuth/support outcome explicit.

## Reproduction Assessment

Yes, by source inspection: deleteAccount sets deactivatedAt/purgedAt and handleDeletedUserSignIn rejects deactivated or legacy deleted users during OAuth completion. I did not run a live GitHub OAuth flow against production data.

## Solution Assessment

No for an automated cleanup fix. Same-GitHub re-signup is a policy and security-sensitive recovery feature; if maintainers keep irreversible deletion, the current code direction should instead focus on clearer user-facing support/error handling.
