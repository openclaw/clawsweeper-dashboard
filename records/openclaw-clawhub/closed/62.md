## Summary

Keep open. This is a protected security issue, and current main still shows a real policy gap: ClawHub blocks malware ZIP/package downloads and has static malicious-install detection, but malware-blocked skill detail/file visibility and install UI behavior remain a maintainer security decision.

## What This Changes

The issue asks ClawHub to address malicious skill distribution risk exposed by a linked ClawHub skill example.

## Best Possible Solution

Maintainers should define the intended posture for malware-blocked skills, then enforce it consistently across detail pages, install prompts, README/file APIs, ZIP/package downloads, downstream mirrors, and OpenClaw install clients while preserving any explicit staff or redacted transparency path.

## Reproduction Assessment

No high-confidence live reproduction for the exact reported skill is available because the canonical v1 API now returns `Skill not found`. The remaining source-level policy gap is reproducible from current main: malware-blocked skills are still publicly readable for transparency while downloads are blocked and install UI is not gated at the call site.

## Solution Assessment

No, this is not best handled as an automated one-shot code change. The narrow maintainable path is a maintainer security decision first, followed by consistent enforcement across registry, file/API, install, mirror, and client surfaces.
