## Summary

Keep this PR open: the Worktrunk routing is useful local-dev tooling, but the latest diff still makes the documented worktree command depend on an undeclared `wt` executable, moves dependency preparation out of the non-Worktrunk seed path, and has no real runtime proof attached.

## What This Changes

The branch adds Worktrunk config and `.worktreeinclude`, switches `dev:worktree` to `wt --yes dev`, moves dependency setup into Worktrunk hooks, and adjusts local worktree process/env handling.

## Best Possible Solution

Make Worktrunk either optional with the existing repo-local fallback or explicitly mandatory with install/version checks and docs, keep dependency preparation on every documented path, and require redacted runtime proof before merge.

## Reproduction Assessment

Yes, at source level: the PR head hard-calls `wt --yes dev` from `package.json` without a repo dependency or documented prerequisite, and `seed:dev` bypasses the new Worktrunk dependency hook. I did not spawn local services during this read-only review.

## Solution Assessment

No: routing through Worktrunk can be a maintainable direction, but this patch is not the best merge shape until compatibility, seed setup, and runtime proof are handled.
