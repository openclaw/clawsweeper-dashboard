## Summary

Current main fixes the reported public API cursor replay for the documented non-`trending` `/api/v1/skills` sorts, and live production now advances the original curl pattern; `trending` is explicitly outside the cursor contract and returns no cursor.

## What This Changes

The issue reports that `GET /api/v1/skills` returns the same page and cursor when replaying `nextCursor`, across supported sorts and `trending`.

## Best Possible Solution

Keep the merged non-`trending` cursor decoder fix as the canonical implementation, and treat any future `trending` cursor support as a separate API feature request.

## Reproduction Assessment

No. The original replay pattern no longer reproduces for the documented cursor-based sorts on live production, and current main has focused source coverage for replaying the returned full cursor.

## Solution Assessment

Yes. Accepting full self-describing `getPage()` keys only for the matching index and equality prefix is the narrow maintainable fix, while leaving `trending` outside cursor pagination matches the documented API contract.
