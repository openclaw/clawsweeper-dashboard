## Summary

This should stay open: current main still rejects the reported personal-to-org owner move as a slug collision, and the protected security label means the ownership-transfer behavior needs explicit maintainer handling rather than cleanup closure.

## What This Changes

The issue asks for an existing skill to be movable from a personal publisher to an organization publisher through the web publish flow while preserving identity, metadata, stats, and canonical URL behavior.

## Best Possible Solution

Add an explicit publisher-target skill transfer path for web, CLI, and API with role checks, audit logs, alias/redirect handling, and regression coverage, then decide whether publish-time transfer should call that path or be disallowed with clearer UI copy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence current-main reproduction path even though I did not run the deployed UI: the form submits the selected owner handle, but `insertVersion` rejects the existing slug when the selected publisher differs from the current owner publisher.

## Solution Assessment

No, implicit ownership transfer as a side effect of publishing is not clearly the safest standalone solution; the maintainable path is an explicit publisher-target transfer workflow or a maintainer-approved publish-time call into that workflow.
