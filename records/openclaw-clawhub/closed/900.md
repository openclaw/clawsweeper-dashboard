## Label Justifications

- `P1`: Affected users cannot regain ClawHub login or CLI-token access after deleting their account, and multiple open reports describe the same blocker.
- `impact:security`: The unresolved behavior changes identity reactivation, deletion guarantees, purge semantics, and token revocation boundaries.
- `impact:auth-provider`: The reported failure happens in GitHub OAuth sign-in and ClawHub identity handling.
- `impact:session-state`: OAuth can return without an active ClawHub user document, leaving the UI and CLI auth flow in a logged-out state.

## Summary

Keep open. Current main still rejects deleted/deactivated GitHub identities from signing in again, while this issue asks for recovery, relink, or recreation behavior that conflicts with the documented irreversible deletion policy and needs maintainer product/security judgment.

## What This Changes

The issue requests a way for users who deleted their ClawHub account to sign in with GitHub again and regain account and CLI-token access.

## Best Possible Solution

Make one explicit deleted-account policy and implement it consistently across OAuth callback handling, active-user gates, token creation, account/settings UI, CLI auth, docs/specs, and regression tests.

## Reproduction Assessment

Yes, at source level. The deletion mutation writes inactive state, OAuth reauth rejects deleted/deactivated users, active-user helpers return no user, and the header/CLI auth surfaces require an active me document.

## Solution Assessment

No autonomous recreate/relink patch is the best next step yet. Maintainers need to choose whether deleted GitHub identities may be restored, relinked, recreated, or only clearly rejected/support-routed before implementation.
