## Best Possible Solution

Keep the settings delete feature, but make org-owned owner deletes restorable by current publisher owners/admins across admin churn while still blocking moderator, security, ban, removed, and malicious hides.

## Reproduction Assessment

Yes. From source inspection, delete an org-owned skill as org admin A through the new settings mutation, remove or deactivate admin A, then restore as current org admin B; the restore gate follows `hiddenBy` to admin A and rejects before treating the current publisher admin as valid.

## Solution Assessment

No. The settings delete flow is a reasonable feature direction, but the restore provenance should survive org-admin churn at the publisher boundary instead of depending on one active deleter account.
