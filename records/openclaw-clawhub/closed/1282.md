## Label Justifications

- `P1`: The issue blocks a real user from ClawHub sign-in and matches several open reports of the same GitHub OAuth account-recovery failure.
- `impact:security`: Restoring or unlinking a deleted account affects identity ownership, deletion guarantees, audit history, and moderation bypass boundaries.
- `impact:auth-provider`: The failure is specifically tied to GitHub OAuth identity binding after a ClawHub account deletion.
- `impact:session-state`: OAuth completion cannot produce an active ClawHub session because the linked user record is inactive.

## Summary

Keep open: current main intentionally treats user-initiated deletion as irreversible and blocks later GitHub OAuth sign-in for that linked account, so the reporter's recovery/unbind request is still unresolved and needs maintainer security/product handling rather than cleanup closure.

## What This Changes

The issue asks for account recovery or GitHub OAuth unbinding so a deleted ClawHub account can sign in or register again.

## Best Possible Solution

Define the account lifecycle explicitly: either keep deletion irreversible with a documented support response, or add a narrow audited restore/unbind flow that verifies GitHub ownership and preserves moderation, privacy, token, and audit safeguards.

## Reproduction Assessment

Yes, at source level: deleting an authenticated GitHub-backed user sets deactivatedAt/purgedAt, and a later OAuth callback rejects that same linked user through handleDeletedUserSignIn. I did not inspect the reporter's production row.

## Solution Assessment

No, ad hoc hard deletion or OAuth unbinding is not the best durable solution. The safer path is a maintainer-approved account lifecycle policy with identity verification, audit requirements, and ban/deletion separation.
