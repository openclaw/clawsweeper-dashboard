## Summary

Keep this issue open. ClawHub cleanup policy keeps issues open, and current main still has a source-backed Settings save failure path where `users.updateProfile` can roll back after personal-publisher synchronization throws on a handle conflict.

## What This Changes

Issue reports that saving display name or bio from the Settings page fails with a Convex `users:updateProfile` server error.

## Best Possible Solution

Persist personal display name and bio updates even when personal-publisher synchronization hits a recoverable handle conflict, while preserving publisher ownership and uniqueness invariants.

## Reproduction Assessment

Yes by source inspection: a Settings form save calls `users.updateProfile`, and that mutation can still throw through personal-publisher synchronization after patching the user. I did not run a live production mutation because that would mutate account data.

## Solution Assessment

Yes. The narrow maintainable fix is to decouple personal profile-field persistence from recoverable personal-publisher sync conflicts, not to change only the Settings UI or weaken publisher handle uniqueness.
