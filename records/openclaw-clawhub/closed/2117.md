## Summary

Keep open: the report is credible against current main and live ClawHub scan data, and the protected `security` label requires maintainer handling instead of cleanup closure.

## What This Changes

The issue asks ClawHub to stop `suspicious.exposed_secret_literal` from flagging non-literal identifier or function-call assignments as exposed secrets.

## Best Possible Solution

Add a focused scanner fix and regression tests so secret-literal detection requires an actual literal value while preserving true positives for hardcoded keys, then let owners rescan the affected skill versions.

## Reproduction Assessment

Yes: source inspection plus a direct regex check shows the current main matcher captures `_fetch_auth_credential` function/member expressions as secret values, and the live ClawHub scan shows the same rule firing on the affected skills.

## Solution Assessment

Yes in direction: the narrow maintainable fix is to guard the RHS as a real literal and add regression coverage; because this is a protected security scanner path, maintainers should own the final false-positive/false-negative threshold.
