## Summary

Keep open under the ClawHub issue cleanup policy, though current main appears to address both reported publish paths: GitHub `--source-path` is applied before fetching, and scoped package names now infer an org owner through the backend membership-checked publisher path.

## What This Changes

The issue reports that `clawhub package publish` could not publish an org-scoped plugin from a monorepo subdirectory under the org namespace because local-folder publishes used the personal namespace and `owner/repo@ref --source-path` failed.

## Best Possible Solution

Keep the current source-path handling and membership-validated scoped-name owner inference, then have a maintainer verify production rollout and update docs or manually close the issue if no follow-up remains.

## Reproduction Assessment

No, not against current main. Source inspection and regression tests show the old `--source-path` failure and personal-namespace fallback path are now covered by current code rather than reproducible as failures.

## Solution Assessment

Yes for the implemented behavior. Applying `--source-path` before fetch and inferring org ownership server-side from scoped package names while validating publisher membership is the narrow maintainable implementation; remaining docs or rollout confirmation should be handled explicitly.
