## Summary

Keep this draft PR open: the performance direction is useful, but the proposed summary-first pagination can regress legacy or unbackfilled publisher rows and the external PR has only test/CI evidence, not real after-fix behavior proof.

## What This Changes

The PR changes `publishers.listPublicPage` to filter/count/sort lightweight publisher summaries, hydrate only the final page slice, and adds regression tests for large publisher lists and no-match searches.

## Best Possible Solution

Keep the summary-first optimization, but preserve the missing-stat fallback before counts, sorting, and pagination or complete a migration-safe backfill before relying on denormalized publisher summaries.

## Reproduction Assessment

Yes, by source inspection of the PR: a publisher without aggregate stat fields can enter `itemSummaries`, consume a page slot, and then be dropped during hydration after slicing. A focused regression test with an unbackfilled empty publisher before a valid publisher would reproduce the short-page/count issue.

## Solution Assessment

No, not yet. Hydrating only the final slice is the right optimization target, but the implementation must not paginate or count missing-stat summaries before resolving whether they actually have published content.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the missing-stats fallback before slicing:** `convex/publishers.ts:1057`
  - body: `publishedSkills` and `publishedPackages` are still optional, and `hasPublisherListContent` keeps missing-stat publishers in `itemSummaries`. Because this slice is taken before `hydratePublisherListSummaries` recomputes and drops empty publishers, a legacy empty publisher at the front of the index can consume a page slot and return a short or empty page with incorrect counts where current main would hydrate/filter before pagination. Please hydrate or otherwise resolve missing-stat summaries before count/sort/slice, or make the denormalized fields migration-safe first.
  - confidence: 0.89
