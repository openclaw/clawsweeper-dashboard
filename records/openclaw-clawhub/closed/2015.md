## Summary

Keep open. Current main still has the cross-kind bucket collision the PR targets, but this draft maintainer PR also changes security-sensitive write throttling and leaves public rate-limit docs inconsistent with the new behavior.

## What This Changes

The PR changes HTTP rate-limit keys to include the request kind for anonymous and authenticated buckets, raises several rate-limit constants, and updates related tests.

## Best Possible Solution

Land the kind-scoped bucket fix after separating or explicitly approving the write-throttle policy change, and update the public rate-limit documentation with the new quotas and bucket semantics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: on current main, two request kinds using the same user or known-IP key are counted together because the stored count is keyed only by key and window. I did not run tests because this review was read-only.

## Solution Assessment

No, not as submitted. Kind-scoped keys are the narrow fix, but raising the anonymous write IP throttle is a separate security/product decision and the docs need to move with the public behavior change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep anonymous write throttling conservative:** `convex/lib/httpRateLimit.ts:11`
  - body: `write.ip` is used when a write route has no valid bearer token, because valid API-token requests go through the user bucket. Raising this from 45/min to 300/min therefore weakens throttling for unauthenticated or invalid-token POST/DELETE traffic, but it does not provide the stated authenticated release or CI headroom. Please keep this bucket conservative or split the policy change out for explicit approval.
  - confidence: 0.86
- **[P3] Update the public rate-limit docs:** `convex/lib/httpRateLimit.ts:108-109`
  - body: The patch changes the published quota values and also changes missing-IP read/write traffic from one shared `ip:unknown` bucket to per-kind buckets. `docs/http-api.md` and the troubleshooting/deploy guidance still describe the old limits and shared unknown-bucket behavior, so users would get headers and 429 behavior that disagree with the docs.
  - confidence: 0.84
