## What This Changes

The PR adds a changelog-section extractor, a manual CLI GitHub Release create/repair workflow, npm release workflow release-note validation and GitHub Release creation, plus deploy runbook docs.

## Best Possible Solution

Land the release automation after write-scoped release creation is isolated from npm publish execution and every release-create/repair path enforces main-ref tooling.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows the publish job now has `contents: write` before release-tag code runs, and the added repair workflow has no main-ref guard while using `${{ github.ref }}` tooling.

## Solution Assessment

No: the automation direction is maintainable, but the safer solution is to isolate write-scoped GitHub Release creation and enforce main-ref release tooling before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep the publish token read-only until release creation:** `.github/workflows/clawhub-cli-npm-release.yml:195-198`
  - body: Changing the whole npm publish job to `contents: write` means checkout and the release-tag publish script run with a write-scoped repository token available through the job checkout state. Current main kept this job read-only; move the GitHub Release create/edit step into a separate dependent job, or otherwise disable persisted credentials and grant write only where `gh release` runs.
  - confidence: 0.9
- **[P2] Require main tooling for the release repair workflow:** `.github/workflows/clawhub-cli-github-release.yml:45-49`
  - body: This new `contents: write` workflow checks out release tooling from `${{ github.ref }}` and never enforces `refs/heads/main`, unlike the npm publish workflow's real-publish guard. Dispatching it from a branch can create or edit GitHub Releases with unmerged tooling, so add an early main-ref check or pin the tooling checkout to main.
  - confidence: 0.86
