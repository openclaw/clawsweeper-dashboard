## Summary

Current main removes the reported browser-callback login path: `clawhub login` now defaults to device-code login, and the legacy `/cli/auth` route no longer mints callback tokens. Broader raw `fetch failed` transport diagnostics can remain separate from this exact browser-auth failure.

## What This Changes

The issue reports `npx clawhub login` opening the `/cli/auth` browser flow and then failing with a raw `fetch failed` error.

## Best Possible Solution

Keep the current device-code login implementation, close this exact browser-callback login failure as implemented on main, and track generic CLI transport diagnostics in a separate scoped report.

## Reproduction Assessment

No current-main reproduction remains for the reported browser-auth symptom: current `cmdLoginFlow` calls device login by default instead of opening `/cli/auth`. The old `Opening browser` behavior is visible in the `v0.21.0` source snapshot, so this is an implemented-on-main cleanup rather than an unreproducible mystery.

## Solution Assessment

Yes. Moving default CLI login to device-code flow and retiring callback token handoff is the narrowest maintainable fix for the reported `/cli/auth` browser-login failure; broader transport diagnostics should be handled separately.
