## Summary

## What This Changes

This PR shards the Codex security scan workflow across eight scheduled workers, lowers the per-shard batch size, raises the global active scan cap, changes queued-job claim ordering, adds an index, and extends tests.

## Best Possible Solution

Land the throughput increase only after the scanner preserves priority semantics, the secret-bearing runner choice is approved or reverted, and a redacted real worker run shows sharded claims completing within the global cap.

## Reproduction Assessment

Yes for the priority-ordering defect from source inspection: active jobs preserve priority while allowing source overwrite, but the new claim path buckets by source without reapplying priority. The runner-secret concern is a static workflow security review rather than a runtime reproduction.

## Solution Assessment

No; sharding and a higher cap are plausible, but the maintainable path needs to preserve existing priority semantics and avoid changing the Production secret runner boundary without explicit approval and proof.
