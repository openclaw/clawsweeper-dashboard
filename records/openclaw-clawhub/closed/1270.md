## Summary

Keep open: current main covers documented `metadata.openclaw.requires.env`, `primaryEnv`, `envVars`, and top-level env arrays, but the reported scalar `env: RISKSTATE_API_KEY` shape is still ignored before registry/scanner metadata consumes it. The protected `security` label also makes this unsuitable for cleanup-close or automated repair.

## What This Changes

The issue asks ClawHub to parse SKILL.md env frontmatter declarations into registry metadata so skill pages and scanner analysis do not report required env vars as none.

## Best Possible Solution

Keep the issue open and land a narrow, security-reviewed parser/helper fix that normalizes scalar top-level `env` as one required env declaration while preserving the documented `metadata.openclaw.requires.env`, `primaryEnv`, and `envVars` paths.

## Reproduction Assessment

Yes. Source inspection shows scalar top-level `frontmatter.env` is routed only through array-only helpers, so a SKILL.md with `env: RISKSTATE_API_KEY` will not become required-env metadata; I did not run tests because this review is read-only.

## Solution Assessment

Mostly yes. Since current code already accepts top-level env arrays as a fallback, extending that normalization to a scalar value with regression coverage is the narrowest maintainable fix while docs continue to prefer `metadata.openclaw.requires.env`.
