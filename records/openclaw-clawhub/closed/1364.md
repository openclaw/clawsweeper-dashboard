## Summary

Keep open: current main still source-reproduces the reported CLI double-serialization bug, and ClawHub cleanup policy keeps issues open during this pass. The broader open report at https://github.com/openclaw/clawhub/issues/1300 is the canonical cluster for transfer, rename, and merge, while the matching fix PRs found in history were closed without merging.

## What This Changes

The issue reports that `clawhub transfer request <slug> <handle>` sends a pre-serialized JSON body so the API never receives the positional handle as `toUserHandle`.

## Best Possible Solution

Land a focused CLI repair that passes plain object bodies from transfer, rename, and merge command handlers, updates tests to assert object payloads, and keeps `apiRequest` as the serialization boundary with runtime coverage if defensive string-body tolerance is added.

## Reproduction Assessment

Yes. Source inspection on current main shows the transfer command passes a JSON string, both HTTP transports stringify bodies again, and the server reads `parsed.payload.toUserHandle` from an object.

## Solution Assessment

Yes. Passing plain objects from the affected command handlers is the narrow maintainable fix; broader org/package ownership-transfer behavior should remain separate from this serialization bug.
