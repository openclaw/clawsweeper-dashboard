## Summary

Keep this PR open: the central UI change is not on current main, but the proposed patch has a blocking correctness issue because it exposes an optional `isSuspicious` value directly and can hide the safety control for legacy suspicious rows.

## What This Changes

The PR exposes `isSuspicious` in public skill payloads and renders the skills browse `Hide suspicious` checkbox only when loaded results include suspicious skills or the filter is active, with tests and a changelog entry.

## Best Possible Solution

Publish a reliable public suspicious-state boolean using the existing moderation fallback, then land the conditional toolbar only if maintainers accept the trust UI behavior.

## Reproduction Assessment

Yes, for the review finding. Source inspection shows the branch can publish `undefined` for legacy suspicious rows while existing fallback logic would identify them as suspicious; I did not run the app in this read-only pass.

## Solution Assessment

No as written. The UI direction may be acceptable, but the public payload should compute a fallback boolean before the browser uses it to hide or show the safety filter.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Compute suspicious state before publishing it:** `convex/lib/public.ts:149`
  - body: `isSuspicious` is still optional on skills and digest rows during migration, but this publishes only `skill.isSuspicious`. A legacy row with `flagged.suspicious` or a scanner suspicious reason and no denormalized value can render as not suspicious, which hides the only visible way to enable the safety filter while a suspicious result is shown. Populate this public field with the existing suspicious fallback when the denormalized value is missing.
  - confidence: 0.86
