## Summary

Keep open: current main still has the transfer-acceptance race and this PR is a plausible narrow security fix, but it is not implemented on main and still lacks required real behavior proof; GitHub also reports the branch as dirty against main.

## What This Changes

The branch adds accept-transfer cancellation guards for inactive requesters and moderated skills, maps committed cancellation failures to HTTP errors, and adds regression/spec/changelog coverage.

## Best Possible Solution

Land a reconciled accept-transfer hardening patch that cancels pending transfers before reporting inactive-requester or moderation rejections, with regression coverage for both the mutation and public HTTP mapping.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a pending transfer whose skill is still owned by `transfer.fromUserId`, mark that requester deleted or deactivated, then accept as the recipient on current main; the requester check is skipped in that state.

## Solution Assessment

Yes. Guarding `acceptTransferInternal` is the narrowest maintainable fix because it is the final shared gate before ownership changes, and returning a typed failure lets Convex commit the cancellation before the HTTP layer maps the rejection.
