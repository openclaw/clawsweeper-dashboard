## Summary

Keep open. Current main still has a source-level publish-blocking embedding path, and the linked implementation PR is open and unmerged, so this issue should stay open until that fix lands or is replaced.

## What This Changes

Publishing valid skill bundles should not fail with an OpenAI 8192-token embedding error when SKILL.md is small but auxiliary text files make the embedding input dense.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/clawhub/pull/2337 with a token- or byte-safe embedding input cap that preserves validation and version insertion for valid bundles.

## Reproduction Assessment

Yes at source level: current main builds a 12,000-character embedding string from SKILL.md plus selected text files and sends it to generateEmbedding, where an OpenAI 4xx input-limit response aborts publish. I did not live-replay the exact reporter bundle because its contents were not attached.

## Solution Assessment

Yes, the narrow maintainable path is to bound publish-time embedding input by a provider-safe byte or token budget. The current issue should stay open while the linked PR is reviewed and proven.
