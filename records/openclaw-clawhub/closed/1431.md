## Summary

Keep open under the ClawHub cleanup policy for issues. Current main fixes the reported publish-flow slug collision through explicit owner migration, but the separate direct organization-targeted transfer flow is still user-targeted and is already represented by https://github.com/openclaw/clawhub/issues/1569 and the open implementation candidate at https://github.com/openclaw/clawhub/pull/1635.

## What This Changes

The issue asks for an existing skill to move from a personal publisher to an organization publisher when publishing a new version or through an equivalent supported transfer flow.

## Best Possible Solution

Keep the explicit publish-time owner migration path for the reported UI bug, and consolidate any standalone org-targeted transfer work in the open publisher-transfer track rather than adding another narrow lookup-only fix.

## Reproduction Assessment

No for the original publish-flow failure on current main: source inspection shows the UI sends a confirmed migrateOwner flag and the backend migrates instead of treating the owner change as a slug collision. The separate direct org-transfer path remains source-traceable as missing because it still accepts toUserHandle and stores toUserId.

## Solution Assessment

Yes for the reported publish-flow bug: explicit opt-in migration with source and destination authority checks is the narrow maintainable fix. The remaining direct org-transfer flow should be solved through a publisher-targeted transfer design with maintainer-approved authorization semantics.
