## Summary

Keep open: current main still restores an unset prior execute/fix gate to `1`, which contradicts the default-closed gate contract and can leave write-mode automation enabled after an `--open-execute-window` run.

## What This Changes

The issue reports that `--open-execute-window` restores previously unset `CLOWNFISH_ALLOW_EXECUTE` and `CLOWNFISH_ALLOW_FIX_PR` variables to `1` instead of restoring a closed or absent state.

## Best Possible Solution

Restore absent prior gate values to a closed state, or delete variables that were absent, in both self-heal and requeue helpers with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: an absent variable becomes `""`, `openGate()` stores it, and the restore expression maps it to `"1"`; I did not run the live mutation path.

## Solution Assessment

Yes. A narrow restore/delete change in both open-window helpers is the maintainable fix, with security review deciding the exact `0` versus delete semantics.
