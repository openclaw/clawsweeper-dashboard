## Summary

Keep open: current main still restores an unset prior execute/fix gate value to `1`, contradicting the documented default-closed gate contract and leaving a security-relevant automation gate open after an `--open-execute-window` run.

## What This Changes

The issue requests restoring `--open-execute-window` gate variables to their prior closed or unset state instead of leaving them set to `1` after dispatch.

## Best Possible Solution

Restore empty prior gate values to a closed state, or delete variables that were absent, across both self-heal and requeue helpers with focused regression coverage for unset prior values.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an absent variable becomes `""`, `openGate()` stores it, and the restore expression maps it to `"1"`; a live run would require mutating Actions variables.

## Solution Assessment

Yes. Restoring or deleting unset prior gate values is the narrow maintainable fix, and the parallel requeue helper should be handled at the same time to avoid a partial repair.
