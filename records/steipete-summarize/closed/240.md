## Summary

Keep open: current main still has a source-level mismatch where local transcript cache reads include fileMtime but the CLI SQLite cache write key omits it, so unchanged local audio can miss the cache.

## What This Changes

Repeated `--extract` runs for unchanged local audio are reported to miss the transcript cache because local-file cache writes omit `fileMtime`.

## Best Possible Solution

Update the CLI SQLite transcript cache write key to include `fileMtime`, then add a real `createCacheStore` regression test that writes and reads the same local-file URL with matching and changed mtimes.

## Reproduction Assessment

Yes. The provided two-run `--extract` path has a high-confidence source reproduction because local media passes `fileMtime`, reads key by it, and the real SQLite write path ignores it; I did not execute a live transcription provider during this read-only review.

## Solution Assessment

Yes. Including `fileMtime` in the CLI store write key and covering the real store is the narrowest maintainable fix because the core interface and read path already carry the value.
