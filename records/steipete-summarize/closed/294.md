## What This Changes

The PR serializes media-cache index read/modify/write operations with a lock file, stages slow media copy/hash work before the index lock, adds stale-lock recovery, and covers concurrent writes in tests.

## Best Possible Solution

Land this focused locking fix after normal maintainer review so the daemon and CLI share one consistent media-cache index under concurrent writers.

## Reproduction Assessment

Yes. Current main reads, mutates, and rewrites the shared media-cache `index.json` without synchronization, and the PR body gives concrete concurrent same-process and multi-process failure counts; I did not run the reproduction because this was a read-only review.

## Solution Assessment

Yes. Locking the shared index read/modify/write section while staging copy/hash work outside the lock is a narrow maintainable fix for the observed lost-entry race.
