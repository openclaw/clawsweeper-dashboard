## Summary

The PR is small, well-scoped, and has sufficient real CLI proof, but it intentionally changes malformed `file:` URL handling from permissive coercion to Node parser failures; maintainers should explicitly accept that compatibility break before merge.

## What This Changes

The PR replaces manual `file:` URL pathname decoding in CLI input resolution with `fileURLToPath` and adds regression tests for POSIX host rejection, encoded separators, and Windows drive/UNC conversion.

## Best Possible Solution

Merge after maintainer acceptance that `file:` inputs should follow Node's URL validation, keeping the regression tests that document the new contract.

## Reproduction Assessment

Yes from source inspection: current main decodes `parsed.pathname` and resolves it, while the PR routes the same URLs through Node `fileURLToPath`, and the PR body shows real CLI before/after-style malformed URL failures. I did not run tests because this review is read-only and the provided proof plus diff establish the path.

## Solution Assessment

Yes, if maintainers accept Node's `file:` URL contract as the CLI contract; using `fileURLToPath` is the narrow standard-library fix and the new tests cover the changed edge cases. The only safer alternative is preserving permissive parsing as a documented compatibility mode, which is a product decision rather than a mechanical defect.
