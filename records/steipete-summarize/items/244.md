## Summary

Keep open: current main still bypasses summary-cache reads and writes for binary image attachments because their prompt content is empty and the cache path requires a non-null text-derived content hash. No open or merged PR currently owns this exact fix.

## What This Changes

Cache repeated local image summaries using a stable binary-content identity while retaining the existing prompt, model, length, language, and format-version key dimensions.

## Best Possible Solution

Use a hash of already-loaded attachment bytes as the fallback content identity when inline prompt content is absent, preserving existing text cache keys and adding hit and invalidation coverage.

## Reproduction Assessment

Yes at the source level: image bytes are attached outside the prompt, the tagged content is empty, and the cache path requires a non-null prompt-content hash. A provider-backed CLI run was not performed in this read-only review.

## Solution Assessment

Yes. Hashing the already-loaded binary bytes is the narrowest robust correction and avoids the aliasing and invalidation weaknesses of path, size, or mtime identity.
