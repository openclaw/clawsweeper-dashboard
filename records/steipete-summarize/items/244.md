## Summary

Keep open: current main still skips summary cache reads and writes for binary image attachments because image prompts have an empty content block and the asset summary cache path requires a non-null content hash. This is a narrow cache bug, separate from https://github.com/steipete/summarize/issues/240 and https://github.com/steipete/summarize/pull/241.

## What This Changes

Cache repeated summaries for identical local JPG/PNG inputs using a stable binary content identity alongside the existing prompt, model, length, language, and cache-format keys.

## Best Possible Solution

Add a binary attachment content identity, preferably a SHA-256 hash of the image bytes, to the asset summary cache key path and cover repeated local-image runs with a regression test.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: local images are binary attachments, their prompt content block is empty, and the asset summary cache is skipped when `contentHash` is null. I did not run the two-command CLI reproduction during this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to derive the asset summary `contentHash` from binary attachment bytes when prompt content is empty, while keeping the existing prompt/model/length/language invalidation behavior.
