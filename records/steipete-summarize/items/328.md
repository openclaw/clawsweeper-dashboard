## Summary

Keep open. This is an OWNER-authored draft, and the current patch still has a merge-blocking compatibility concern: Chrome Direct-provider requests configured for `http://127.0.0.1` can be routed into the Summarize native daemon bridge instead of the user's local provider; the changed extension E2E lane is also currently failing.

## What This Changes

The PR migrates the Chrome extension daemon companion from loopback fetches to optional Native Messaging, adds managed-policy UX/docs, updates daemon host install/uninstall support, and revises extension tests/CI.

## Best Possible Solution

Land the Native Messaging migration only after preserving or explicitly accepting the 127.0.0.1 Direct-provider compatibility break, documenting the Windows Daemon limitation, and getting a fresh green Chrome extension E2E signal on current main.

## Reproduction Assessment

Yes from source inspection for the main compatibility concern: direct summary/chat requests receive the background fetch wrapper, and that wrapper diverts 127.0.0.1 URLs to the daemon bridge. I did not run the extension E2E because this is a read-only review, but GitHub currently reports that check failing.

## Solution Assessment

No for the 127.0.0.1 path; routing all 127.0.0.1 requests through Native Messaging is broader than the daemon-only boundary and can replace configured direct local providers. A narrower daemon-origin fetch path or explicit origin-permission design is safer unless maintainers deliberately choose the stricter boundary.
