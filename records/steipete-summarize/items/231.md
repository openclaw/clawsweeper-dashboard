Full review comments:

- **[P2] Keep agy model lookup provider-specific:** `src/model-spec.ts:196`
  - body: Accepting `cli/agy` here exposes the new provider to `resolveModelSelection`, but `src/run/run-models.ts` still falls through unknown CLI providers to `cli?.copilot?.model`, and `src/model-auto-cli.ts` falls through to the Claude model. With an existing Copilot or Claude model configured, bare `cli/agy` stops using agy's runtime default and is invoked with another provider's model string, so add explicit agy branches and regression tests before accepting the provider.
  - confidence: 0.89
- **[P2] Wire agy through config parsing:** `src/config/types.ts:39`
  - body: The new `cli.agy` type is exposed, but `parseCliProvider` still rejects `agy` in provider lists and `parseCliConfig` never reads `value.agy`. That means users cannot put `agy` in `cli.enabled`/`cli.autoFallback.order`, and any `cli.agy` binary/model/extraArgs config is ignored, unlike the other supported providers.
  - confidence: 0.88
- **[P2] Include agy in bare --cli selection:** `src/run/help.ts:144`
  - body: The help text now lists `agy`, but `resolveConfigState` still builds the default enabled provider list for a bare `--cli` from the old providers only. `summarize --cli` therefore will not consider agy unless the user names it explicitly or config parsing is fixed, which contradicts the advertised provider list.
  - confidence: 0.82
