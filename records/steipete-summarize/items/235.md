## Summary

Keep open: current main still forces chat completions for any custom OpenAI base URL, so an explicit false setting cannot select the Responses API for custom gateways.

## What This Changes

The issue asks `OPENAI_USE_CHAT_COMPLETIONS=false` and `openai.useChatCompletions=false` to prevent `/chat/completions` from being forced when OpenAI uses a custom base URL.

## Best Possible Solution

Carry an explicit tri-state OpenAI chat-completions preference through run config and provider resolution so absent config keeps the current custom-base default, while explicit false uses Responses API endpoints for custom OpenAI-compatible gateways.

## Reproduction Assessment

Yes from source inspection: false is parsed, but a custom OpenAI base URL still makes `resolveOpenAiClientConfig` set `useChatCompletions` true. I did not run a live gateway reproduction.

## Solution Assessment

Yes: the narrow maintainable repair is to preserve explicit false as an override instead of adding another flag or changing all custom-base defaults.
