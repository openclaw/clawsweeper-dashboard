## Summary

Keep open for maintainer review: the sink-side cap is narrow and appears correct, but this external security PR still needs real behavior proof and deliberately turns oversized remote podcast downloads into a fail-closed path.

## What This Changes

The PR adds `MAX_REMOTE_MEDIA_BYTES` enforcement to podcast remote-media temp-file downloads and adds regression tests for streaming and non-streaming oversized responses.

## Best Possible Solution

Land the sink-side cap after contributor proof, with media documentation aligned to the enforced 512 MB remote podcast download limit and the compatibility tradeoff explicitly accepted.

## Reproduction Assessment

Yes from source inspection: current main can reach `downloadToFile()` after a missing or under-reported HEAD length and then write streamed bytes without a max cap. I did not execute tests because this review is read-only.

## Solution Assessment

Yes, with one documentation caveat: enforcing the existing `MAX_REMOTE_MEDIA_BYTES` policy inside the file-download sink is the narrowest maintainable fix for the disk-exhaustion path. The docs should be updated so the newly reliable fail-closed behavior is not surprising.
