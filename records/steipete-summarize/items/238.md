## Summary

Keep this PR open, but it is not merge-ready: it improves literal loopback and redirect handling for RSS transcript URLs, yet the security boundary remains incomplete for hostnames that resolve or rebind to private addresses and it duplicates the stronger daemon URL guard.

## What This Changes

The PR adds RSS podcast transcript URL validation, manual redirect handling, and two mocked SSRF regression tests for literal loopback and redirect-to-loopback transcript URLs.

## Best Possible Solution

Use one shared URL-fetch security boundary for RSS transcript fetching that resolves DNS, rejects private resolved addresses, pins fetch addresses, revalidates redirects, and has focused DNS/rebinding regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a feed-controlled hostname such as a public-looking domain that resolves to a private address passes the PR's string-only hostname checks before runtime fetch resolves it.

## Solution Assessment

No. Literal IP and redirect validation is useful, but the maintainable fix should reuse or share the existing daemon-grade DNS resolution and address-pinning policy rather than adding a weaker parallel guard.
