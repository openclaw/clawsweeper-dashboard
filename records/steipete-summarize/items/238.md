## Label Justifications

- `P1`: This PR is trying to close a transcript-fetch SSRF boundary, and the remaining DNS-resolution bypass is security-sensitive enough to need prompt maintainer attention.
- `merge-risk: 🚨 security-boundary`: Merging as-is could leave transcript SSRF protection incomplete because private DNS answers and rebinding are not rejected before fetch.

## Summary

Keep open: the hardening direction is useful, but the patch is not merge-ready because it validates only literal hostnames before fetching, while the existing daemon guard already resolves and pins DNS addresses; DNS-resolved private hosts would still be reachable.

## What This Changes

The PR adds RSS transcript URL validation, manual redirect revalidation, and regression tests for direct loopback and redirect-to-loopback transcript targets.

## Best Possible Solution

Land a core-safe transcript fetch guard that resolves and pins public hostnames for initial and redirect targets, reuses or extracts the daemon guard’s DNS validation model without importing CLI-only code into core, and includes DNS-resolution regression coverage plus real proof.

## Reproduction Assessment

Yes for the review finding from source inspection: a public hostname that resolves to a private address passes `assertSafeTranscriptUrl` because only the URL hostname string is checked before `fetchImpl` runs DNS. I did not run a live network reproduction because the review checkout is read-only.

## Solution Assessment

No. The direction is right, but the maintainable fix is to reuse or extract the existing DNS resolution and pinning model instead of adding a literal-only transcript-specific guard.
