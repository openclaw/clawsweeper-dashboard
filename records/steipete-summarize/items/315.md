## Summary

This should stay open: current main and the latest release still generate a Scheduled Task XML declaration with `encoding="UTF-8"`, the current test asserts that declaration, and no open fix PR or duplicate issue appears to own the schtasks encoding failure.

## What This Changes

Windows daemon install should generate `schtasks /Create /XML` input that does not fail with “Cannot switch encoding” on non-UTF-8 Windows ANSI code pages.

## Best Possible Solution

Write the Scheduled Task XML in a schtasks-compatible encoding for localized Windows, cover the generated XML with focused tests including non-ASCII principal/path values, and validate `summarize daemon install` on a non-UTF-8 Windows code page.

## Reproduction Assessment

No, not with a live current-main run here. The issue is source-reproducible because current main emits the reported UTF-8 XML declaration, writes it as UTF-8, and passes it to `schtasks /Create /XML`, but the localized Windows code-page failure still needs live confirmation.

## Solution Assessment

Unclear as proposed. Removing the encoding declaration may solve the reporter's ASCII-path case, but the generated XML can include non-ASCII user principals or profile paths, so the repair should choose and test an encoding strategy that stays correct for those values.

Do not edit `CHANGELOG.md`; put release-note context in the PR body instead.
