## Summary

Keep open: the no-throw fix is well-scoped, but it is not merge-ready because dropping opaque origins to an empty list makes inline cookie reads unfiltered, and the PR's own after-proof returns an `example.com` cookie for a `file://` URL.

## What This Changes

This PR guards `normalizeOrigins()` against opaque parsed origins and adds a util regression test for `file://` and `about:blank` inputs.

## Best Possible Solution

Preserve the shared no-throw guard, add public `getCookies()` coverage, and make opaque or unusable origins fail closed for inline filtering before running `pnpm test` and `pnpm check`.

## Reproduction Assessment

Yes. I reproduced current-main behavior with the checked-in dist: `getCookies()` throws `TypeError: Invalid URL` for the `file://` inline payload path, and Node's URL API confirms the opaque origin is `null`.

## Solution Assessment

No, not as currently submitted. The shared normalizer guard is the right direction for the throw, but the implementation also disables inline host filtering for opaque URLs; the safer fix keeps no-throw behavior while returning no unrelated cookies.
