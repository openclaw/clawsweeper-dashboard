## Summary

Keep open: current main still has Pushover/profile notification settings and realtime unread handling, but no browser Notification API, service-worker, PushManager, or PWA notification path for incoming web messages.

## What This Changes

Add a web/PWA browser notification path for incoming `message.created` realtime events, separate from server-side Pushover.

## Best Possible Solution

Ship a narrow browser-notification opt-in in the web app that handles permission from an explicit user action, skips own and active-visible messages, and focuses/navigates to the relevant channel or DM on click.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path: current main handles incoming `message.created` realtime events but has no browser notification or service-worker primitives. I did not run a live browser permission scenario.

## Solution Assessment

Unclear: the proposed ChatApp hook is a plausible minimal implementation, but maintainers should decide the permission UX, first shipped scope, and whether notification bodies should stay generic for privacy.
