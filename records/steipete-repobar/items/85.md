## Best Possible Solution

Get explicit maintainer sponsorship for core GitLab support first; if sponsored, land a narrower HTTPS-by-default implementation with provider-isolated credentials, complete GitLab menu/CLI routing, preserved regression coverage, and redacted runtime proof.

## Solution Assessment

No. The safer path is maintainer-sponsored provider support that preserves transport security by default, keeps credentials scoped to their provider, updates every visible GitLab menu/CLI route, preserves regression coverage, and includes real GitLab runtime proof.

## Review Findings

Full review comments:

- **[P1] Remove the global ATS bypass:** `Scripts/package_app.sh:123-126`
  - body: This writes `NSAllowsArbitraryLoads=true` into the packaged app while the new GitLab PAT flow accepts `http://` hosts and sends the PAT in `PRIVATE-TOKEN`. That allows cleartext token traffic for any URLSession in the app; keep HTTPS as the default or add a narrow maintainer-approved exception instead.
  - confidence: 0.93
- **[P1] Keep GitLab tokens out of GitHub legacy paths:** `Sources/RepoBar/App/AppState+Accounts.swift:164-166`
  - body: This guard only covers the fresh-login path, but switching to or falling back from a GitLab account still mirrors active-account credentials to legacy keys, and the GitHub token provider reads the active account token. A GitLab PAT can therefore be used by GitHub-only clients; gate all legacy mirroring and GitHub token-provider paths on `.github`.
  - confidence: 0.87
- **[P2] Use GitLab web routes beyond CI:** `Sources/RepoBar/Support/RepoWebURLBuilder.swift:43-49`
  - body: The provider-aware URL work only special-cases CI, leaving issue and merge-request actions to append GitHub paths like `/issues` and `/pulls`. GitLab projects should use provider-specific web paths such as `/-/merge_requests` and the corresponding GitLab routes for the visible menu actions.
  - confidence: 0.84
- **[P2] Populate GitLab issue and merge request counts:** `Sources/RepoBarCore/Models/Repository+Factory.swift:18-21`
  - body: Every GitLab project is mapped with `openIssues: 0` and `openPulls: 0`, so repository cards and submenu badges will report no issues or MRs even when the GitLab API has items. Fetch or map those counts before presenting GitLab projects as supported.
  - confidence: 0.82
