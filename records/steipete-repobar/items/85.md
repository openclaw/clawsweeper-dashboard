Full review comments:

- **[P1] Remove the global ATS bypass:** `Scripts/package_app.sh:123-126`
  - body: This writes `NSAllowsArbitraryLoads=true` into the packaged app while the new GitLab PAT flow accepts `http://` hosts and sends the PAT in `PRIVATE-TOKEN`. That allows cleartext token traffic for any URLSession in the app; keep HTTPS as the default or add a narrow maintainer-approved exception instead.
  - confidence: 0.93
- **[P1] Keep GitLab tokens out of GitHub legacy paths:** `Sources/RepoBar/App/AppState+Accounts.swift:164-166`
  - body: This guard only covers the fresh-login path, but switching to or falling back from a GitLab account still mirrors active-account credentials to legacy keys, and the GitHub token provider reads the active account token. A GitLab PAT can therefore be used by GitHub-only clients; gate all legacy mirroring and GitHub token-provider paths on `.github`.
  - confidence: 0.87
- **[P2] Use the active provider host for checkout:** `Sources/RepoBar/StatusBar/StatusBarMenuManager.swift:63-65`
  - body: The PR makes browser URL building provider-aware here, but `cloneURL(for:)` still builds checkout remotes from `session.settings.githubHost`. With a GitLab repo selected, Checkout Repo can try to clone from GitHub instead of the active GitLab instance.
  - confidence: 0.84
- **[P2] Use GitLab paths for issue and MR menus:** `Sources/RepoBar/Support/RepoWebURLBuilder.swift:43-49`
  - body: The provider-aware URL work only special-cases CI; issue and pull-request actions still append GitHub paths like `/issues` and `/pulls`. For GitLab projects these should route to `/-/issues` and `/-/merge_requests`, otherwise visible menu actions open the wrong pages.
  - confidence: 0.78
