## Summary

This PR fixes a real parser inconsistency for signed, decimal, and exponent literals in `where`, but the shared `parseValue` change also changes top-level `limit` parsing from an integer-normalized path to direct decimal/exponent numbers that flow into SQL `LIMIT`; that needs a narrow repair before merge.

## What This Changes

The PR broadens GraphQL literal parsing from unsigned integers to signed, decimal, and exponent numbers and adds a unit test for numeric `where` values.

## Best Possible Solution

Keep the widened numeric handling for `where` values while preserving integer-only normalization for top-level `limit`, with a regression test covering fractional or exponent `limit` inputs.

## Reproduction Assessment

Yes for the parser inconsistency from source and the PR's real-output proof: current main only treats unsigned integers as numbers. The merge blocker is also source-reproducible because `limit` uses the same widened `parseValue` path and later flows into SQL.

## Solution Assessment

No, not as currently scoped: widening numeric parsing is the right fix for `where`, but applying it through the shared helper without preserving `limit` integer semantics is too broad. A small targeted repair should keep `where` numeric consistency and guard top-level `limit`.
