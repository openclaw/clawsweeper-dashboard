## What This Changes

## Best Possible Solution

Land a rebased, compatible provider design that keeps `tailscale_enabled` working as an alias/default, adds focused upgrade validation, and treats Netbird sudo/firewall permissions as an explicit opt-in security decision.

## Reproduction Assessment

Yes for the Tailscale authkey gap by source inspection: current docs/defaults expose `tailscale_authkey`, while the task only prints manual instructions and never runs `tailscale up --authkey`. Netbird support is a new feature rather than a reproduced bug.

## Solution Assessment

No. Adding a provider switch is plausible, but replacing `tailscale_enabled` outright is not the safest upgrade path; preserve the old variable as an alias while adding Netbird behind the new provider setting.
