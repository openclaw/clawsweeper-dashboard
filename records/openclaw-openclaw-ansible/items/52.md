## Summary

Keep this PR open: it is a real candidate fix for the linked Tailscale issue and has strong terminal proof, but the latest patch still broadens the service account's Tailscale sudo boundary and leaves manual Tailscale setup outside the automatic operator grant.

## What This Changes

The PR wires the documented `tailscale_authkey` into Linux Tailscale setup, grants the OpenClaw service user Tailscale operator rights after user creation, and expands scoped sudoers for Tailscale Serve.

## Best Possible Solution

Set the Tailscale operator after user creation for both auth-key and manual Tailscale flows, keep Serve out of broad wildcard sudoers unless maintainers explicitly approve that boundary, and retain redacted auth-key handling.

## Reproduction Assessment

Yes. Source inspection and the linked report show current main documents `tailscale_authkey` without using it and lacks the service-user operator grant needed for `tailscale serve`; I did not run a fresh VM locally.

## Solution Assessment

No. Auth-key auto-connect and moving the operator grant after user creation are good directions, but the patch should cover manual Tailscale setups and avoid wildcard root-run Serve sudoers unless maintainers explicitly accept that boundary.
