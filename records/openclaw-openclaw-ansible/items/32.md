## Label Justifications

- `P0`: The installer grants the service account Docker daemon access that can undermine the documented security boundary if that account is compromised.
- `impact:security`: The issue is about privilege escalation and host compromise through Docker daemon access.

## Summary

Keep open: current main still appends `openclaw_user` to the `docker` group and has no rootless-Docker or socket-mediation path, so the reported security boundary gap remains valid and no merged related fix supersedes it.

## What This Changes

The issue reports that the Linux installer adds the OpenClaw service user to the `docker` group, giving root-equivalent Docker daemon access despite scoped-sudo and hardening claims.

## Best Possible Solution

Remove root-equivalent Docker daemon access from the long-lived OpenClaw account while preserving one-command install and service management, then update security docs and VM verification to prove the account cannot control the host through Docker.

## Reproduction Assessment

Yes. Source inspection shows the Linux install path appends `openclaw_user` to the `docker` group; I did not run a VM exploit, but the code path is direct and current.

## Solution Assessment

Not yet. The report identifies the unsafe path, but the maintainable fix needs a maintainer-owned Docker access design, such as rootless Docker or a root-owned service/helper model with upgrade validation.
