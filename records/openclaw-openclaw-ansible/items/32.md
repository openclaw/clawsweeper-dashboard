## Label Justifications

- `P0`: The report concerns a root-equivalent host compromise path through Docker daemon access from the OpenClaw runtime user.
- `impact:security`: The issue is about a privilege boundary and host-compromise path for the OpenClaw runtime user.

## Summary

Keep open: current main still appends `openclaw_user` to the `docker` group, while the repo’s hardening docs still present scoped sudo and non-root execution as security boundaries; no merged related fix or current Debian/Ubuntu rootless/socket-mediation path supersedes the issue.

## What This Changes

The issue asks to remove or redesign `openclaw_user` Docker daemon access because docker-group membership can grant root-equivalent host control on typical Linux hosts.

## Best Possible Solution

Remove root-equivalent Docker daemon access from the long-lived OpenClaw account while preserving one-command install and service lifecycle management, then update docs and VM security checks to prove the account cannot control the host through Docker.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection of current main confirms the Linux Docker task still appends `openclaw_user` to the `docker` group, which is the reported boundary gap.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The report identifies the unsafe access path, but the maintainable fix needs a maintainer-owned Docker control design such as rootless Docker, a root-owned service/helper model, or another constrained control path with upgrade proof.
