## Summary

Keep open: current main still appends the OpenClaw service account to the rootful Docker daemon’s `docker` group, and I found no rootless-Docker or socket-mediation replacement. The report remains a live security-boundary issue that needs maintainer security/design review rather than cleanup closure.

## What This Changes

The issue asks maintainers to remove or redesign `openclaw_user` docker-group membership because it grants root-equivalent Docker daemon access despite the installer’s hardening claims.

## Best Possible Solution

Choose and document a Docker control model that removes root-equivalent daemon access from the long-lived OpenClaw account while preserving one-command install, then add VM upgrade/security verification proving the account cannot control the host through Docker.

## Reproduction Assessment

Yes. Source inspection on current main shows the Linux install path appends `openclaw_user` to the `docker` group; I did not run a VM exploit, but the code path is direct and high-confidence.

## Solution Assessment

Unclear. The report correctly identifies the unsafe path, but the maintainable fix needs a maintainer-owned Docker access design such as rootless Docker, a root-owned service/helper model, or another mediated control path with upgrade proof.
