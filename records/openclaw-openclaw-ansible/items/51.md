## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enable the Quadlet user service before relying on it:** `roles/openclaw/tasks/quadlet-redhat.yml:110-114`
  - body: This task only runs `systemctl --user start` or `restart`, so the generated unit is not enabled into the user's `default.target`; after reboot the OpenClaw gateway can remain down even though installation appeared successful. Enable the user service as well as starting it, keeping the existing user and `XDG_RUNTIME_DIR` environment.
  - confidence: 0.93
- **[P1] Avoid fixed overlapping subordinate ID ranges:** `roles/openclaw/tasks/podman-redhat.yml:33-47`
  - body: These tasks write the configured `/etc/subuid` and `/etc/subgid` ranges for `openclaw` without checking whether `100000:65536` is already assigned to another local user. Preserve an existing `openclaw` allocation or choose a free non-overlapping range before asserting rootless Podman isolation.
  - confidence: 0.9
- **[P1] Raise the Podman floor or use v4.4 Quadlet keys:** `roles/openclaw/vars/RedHat.yml:4`
  - body: The RedHat vars allow Podman 4.4.0, but the generated Quadlet uses newer keys such as `UserNS=` and `Pull=` that are absent from the Podman 4.4 Quadlet docs. A supported fresh host can pass preflight and then fail to generate or start `openclaw.service`, so either make the template v4.4-compatible or raise the minimum to a version that supports those keys.
  - confidence: 0.88
- **[P1] Install ansible-core on RHEL-family bootstrap:** `install.sh:67`
  - body: On supported RHEL-family hosts without EPEL, this dnf/yum path tries to install `ansible git`, but the minimal package that provides `ansible-playbook` is `ansible-core`. The advertised one-command install can exit before collection installation, so use `ansible-core` for dnf/yum or fall back to it when `ansible` is unavailable.
  - confidence: 0.86
- **[P2] Remove host CLI setup commands from Podman docs:** `docs/installation.md:114-119`
  - body: The Fedora/RHEL section still tells users to run `openclaw providers login` and `openclaw configure` as the host `openclaw` user, but the Podman path intentionally skips host CLI installation. Replace this with the supported container-native configuration/login flow or install a real host CLI for that path.
  - confidence: 0.84

## Security Review

Status: needs_attention

  - body: The PR writes rootless Podman subuid/subgid ranges from fixed defaults; on hosts where another rootless user already owns that range, namespace mappings and storage ownership can collide.
