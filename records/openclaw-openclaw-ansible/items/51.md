## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enable the Quadlet user service before relying on it:** `roles/openclaw/tasks/quadlet-redhat.yml:110-114`
  - body: This task only runs `systemctl --user start` or `restart`, so the generated unit is never enabled into the user's `default.target`; after a reboot the OpenClaw gateway can stay down even though the install appeared successful. Enable the user service as well as starting it, for example with an `enable --now` sequence that keeps the existing user/XDG runtime environment.
  - confidence: 0.91
- **[P1] Avoid fixed overlapping subordinate ID ranges:** `roles/openclaw/tasks/podman-redhat.yml:33-47`
  - body: These tasks write the configured `/etc/subuid` and `/etc/subgid` ranges for `openclaw` without checking whether `100000:65536` is already assigned to another local user. Preserve an existing `openclaw` allocation or choose a free non-overlapping range before asserting rootless Podman isolation.
  - confidence: 0.88
- **[P2] Gate the RedHat welcome path away from host CLI commands:** `playbooks/install.yml:24`
  - body: This line makes all RedHat-family hosts supported, but the same playbook later writes a one-time welcome message that still tells users to run host `openclaw onboard`, `configure`, provider login, daemon, status, and log commands. Since the Podman path skips host CLI installation, split the welcome content by runtime or provide a real supported CLI path.
  - confidence: 0.84
- **[P2] Stop auto-pulling a mutable default image:** `roles/openclaw/vars/RedHat.yml:9-11`
  - body: The default RedHat image still resolves to `:latest`, and the Quadlet uses `Pull=newer`, so a service start can move an installed host to a different image digest without a reviewed playbook change. Use a versioned/digest default or disable automatic pulling unless maintainers explicitly accept that upgrade policy.
  - confidence: 0.82

## Security Review

Status: needs_attention
