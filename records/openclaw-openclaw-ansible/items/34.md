## Summary

Keep open: current main still defines and documents Tailscale auto-connect and runs OpenClaw as the openclaw user, but it does not consume `tailscale_authkey` or set the openclaw user as Tailscale operator for `tailscale serve`.

## What This Changes

The issue reports that `tailscale_enabled: true` leaves OpenClaw unable to run `tailscale serve` as the openclaw user and that `tailscale_authkey` is ignored during provisioning.

## Best Possible Solution

Implement a narrow Tailscale-only fix that uses `tailscale_authkey` securely and grants only the minimal operator/serve capability needed for the openclaw service user, with VM validation and documentation updates.

## Reproduction Assessment

Yes from source inspection: current main defines and documents the auth key but never consumes it, and the service runs as openclaw without a task setting that user as Tailscale operator. I did not run a live Tailscale VM reproduction during this read-only review.

## Solution Assessment

Unclear until maintainers accept the permission boundary, but the narrowest maintainable path is a focused Tailscale authkey/operator fix rather than the broader NetBird/provider migration in the open overlapping PR.
