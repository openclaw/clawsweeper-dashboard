## Summary

Keep open. Current main still does not set the Tailscale operator for the OpenClaw service user and does not consume the documented auth key variable; the related VPN PR only partially overlaps and is not a clean canonical fix.

## What This Changes

Issue reports that Tailscale serve mode fails for the OpenClaw service user until `tailscale set --operator=openclaw` is run manually, and that `tailscale_authkey` is documented but unused.

## Best Possible Solution

Add a security-reviewed Tailscale setup path that uses `tailscale_authkey` with `no_log` and deliberately handles serve-mode operator permission without broad sudo.

## Reproduction Assessment

Yes, by source inspection rather than a VM: current main declares and documents `tailscale_authkey`, but the Tailscale task only prints manual setup instructions and the sudoers allowlist lacks the reported operator/serve path.

## Solution Assessment

Unclear until maintainers choose the permission boundary. A narrow fix should preserve scoped sudo, redact auth-key handling, and either grant the Tailscale operator deliberately after the service user exists or document that serve mode requires a manual operator command.
