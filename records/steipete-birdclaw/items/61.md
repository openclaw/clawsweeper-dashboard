## Summary

Keep open: current main and v0.8.3 still split backup JSONL with Node readline while export can write literal Unicode line separators, and the linked open fix PR has not merged yet.

## What This Changes

The issue asks backup import/sync to preserve JSONL records containing U+2028 or U+2029 inside strings and asks export to avoid writing those literals.

## Best Possible Solution

Land a focused fix that reads backup JSONL by physical LF records, escapes U+2028/U+2029 on export, and preserves round-trip backup coverage.

## Reproduction Assessment

Yes, at source level: current main feeds backup JSONL through Node readline, and a Node probe shows readline splits a single JSON record at U+2028 before JSON.parse fails. I did not run a full CLI import in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to change the shared JSONL reader to physical LF records and escape U+2028/U+2029 in backup export, with focused backup round-trip coverage.
