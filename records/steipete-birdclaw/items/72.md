Keep open: current main still uses only a subtle shared active class for the Today period buttons, exposes no pressed/selected ARIA state, and has no Today UI for custom ranges even though the digest backend can accept explicit windows.

## What This Changes

The issue asks for clearer selected-state styling on the Today view time range control, with custom time ranges as a follow-up enhancement.

## Best Possible Solution

Land a focused active styling and ARIA-state fix for the existing period buttons, then treat richer custom ranges as a separately scoped UI enhancement using the existing since/until backend if maintainers want it.

## Reproduction Assessment

Source inspection gives a good reproduction path for the UI state: switch the period buttons and inspect that only the subtle active class changes, with no pressed/selected ARIA state. I did not run a browser proof, and the custom-range part is an enhancement rather than a failing behavior reproduction.

## Solution Assessment

Yes for a narrow active-state fix: strengthen the selected styling and expose semantic pressed/selected state in the existing control. The custom range and saved preset ideas should not be mixed into that first repair without maintainer product direction.
