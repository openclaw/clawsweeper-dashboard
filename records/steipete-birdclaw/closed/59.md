## What This Changes

The PR raises SQLite busy timeouts to 30 seconds, starts top-level transactions with `begin immediate`, skips completed startup backfills, adds cross-process lock regression tests, and updates the changelog.

## Best Possible Solution

Land this after the maintainer accepts the longer SQLite wait behavior and live validation passes; if the wait is too broad, keep the contention tests and narrow the timeout or `begin immediate` scope.

## Solution Assessment

Is this the best way to solve the issue? Yes, with an explicit availability tradeoff: acquiring the writer slot before transaction reads and increasing the busy timeout directly targets SQLite single-writer contention, while the backfill checks reduce unnecessary startup writes.
