## Summary

Keep open: current main still resolves OpenCode Go usage for one workspace at a time, while this request asks for a new multi-workspace fan-out and menu presentation. The related multiple-account discussion is useful context, but it does not supersede this same-account, multiple-workspace feature request.

## What This Changes

The issue requests showing usage for multiple OpenCode Go workspaces under one authenticated account instead of manually swapping a single workspace ID.

## Best Possible Solution

Add a maintainer-approved OpenCode Go multi-workspace model that fetches each workspace with the shared cookie and renders labeled per-workspace usage without mixing provider identity data.

## Reproduction Assessment

Yes from source inspection: current main resolves one OpenCode Go workspace ID and returns one UsageSnapshot, so it cannot show all workspaces at once. No live provider probe was run because the repository policy avoids real account/cookie validation unless explicitly requested.

## Solution Assessment

No implementation is proposed in this issue; the requested direction is plausible but needs maintainer UI/product sign-off before code should fan out over all workspaces. The narrowest maintainable solution would extend the provider data model and menu rendering rather than adding another manual single-workspace setting.
