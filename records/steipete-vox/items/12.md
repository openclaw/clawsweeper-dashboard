Keep open: current main still processes tool outputs from every `response.done`, and the PR’s production guard is narrow and likely correct, but this external PR needs real behavior proof before merge and has one ineffective disk assertion in the new regression test.

## What This Changes

The PR adds an early `response.status === "cancelled"` return in `handleResponseDone` and a regression test for cancelled `response.done` tool outputs.

## Best Possible Solution

Land the narrow cancelled-response guard with regression coverage that verifies both OpenAI follow-ups and report-file side effects in the actual logger directory, after the contributor adds redacted runtime proof.

## Reproduction Assessment

Yes from source inspection: current main enables `interrupt_response` and `handleResponseDone` processes `response.output` without checking `response.status`, so a cancelled `response.done` carrying function calls will reach the tool handlers. I did not run tests because this review is read-only.

## Solution Assessment

Yes, returning before output parsing is the narrowest maintainable production fix for cancelled responses. The test should inspect the actual generated logger directory, and the PR still needs real runtime proof before merge.
