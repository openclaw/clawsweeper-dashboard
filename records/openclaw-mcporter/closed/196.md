## Label Justifications

- `impact:auth-provider`: The issue concerns OAuth scope selection, refresh tokens, and provider compatibility.
- `impact:session-state`: The reported user impact is lost authentication continuity after access-token expiry.

## Summary

Current main and the latest release no longer match the reported 0.7.3 behavior: OAuth metadata omits the hardcoded `mcp:tools` scope by default, supports an explicit `oauthScope` escape hatch, and has non-interactive refresh-token use for expired cached OAuth tokens.

## What This Changes

The issue asks mcporter to stop hardcoding OAuth scope `mcp:tools`, use advertised or configured scopes so refresh tokens can be issued, and keep OAuth sessions alive across access-token expiry.

## Best Possible Solution

Keep the current metadata-derived scope behavior and `oauthScope` override; users still on 0.7.x should upgrade to the current release rather than patching installed files.

## Reproduction Assessment

No current-main failing reproduction remains: the reported 0.7.3 hardcoded-scope line is absent, and current source only sets `scope` when `oauthScope` is configured. The old behavior is source-verifiable in the older release tree, but not in current main.

## Solution Assessment

Yes. The current implementation is the narrow maintainable fix: omit the conflicting default so the SDK can derive server scopes, keep `oauthScope` for non-discoverable providers, and refresh cached OAuth tokens when a refresh token exists.
