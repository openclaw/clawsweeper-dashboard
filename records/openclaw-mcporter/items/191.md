## Repair Work Prompt

Fix the daemon process lifecycle race without changing the documented per-login detached daemon model. In `src/daemon/client.ts`, make daemon liveness PID-aware by reading metadata, checking the metadata PID is alive when possible, requesting `status`, and requiring the responder PID/socket/config freshness to match before treating the daemon as ready. Replace or wrap the per-instance `startingPromise` start path with a cross-process lock using the existing `withFileLock` helper, and re-check liveness while holding the lock before launching. In `src/daemon/host.ts`, avoid blindly replacing a healthy daemon for the same config/socket; preserve the existing atomic metadata write path. Do not add a new dependency, do not add a new `daemon cleanup` command, and do not make the daemon die merely because the launching CLI/gateway exits. Add focused tests for concurrent independent clients starting the same daemon, metadata PID mismatch, stale socket/metadata recovery, and the changed fresh-metadata status-preflight behavior. Keep release notes out of `CHANGELOG.md`. Related refs: https://github.com/openclaw/mcporter/issues/191 and https://github.com/openclaw/mcporter/issues/188.
