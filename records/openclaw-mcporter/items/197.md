## Summary

Keep open: current main still has the reported cache/OAuth coupling, and there is no `disableOAuth` or equivalent cache-friendly OAuth suppression option. The request is aligned with MCPorter daemon/repeated-call reliability, but it needs a maintainer API decision before implementation.

## What This Changes

The issue requests an additive `disableOAuth` or equivalent connect option so callers can suppress interactive OAuth while still using cached auth and pooled connections.

## Best Possible Solution

Add a maintainer-approved, cache-friendly OAuth suppression mode and cover repeated cached-auth connects plus close/reset cleanup behavior with focused regression tests.

## Reproduction Assessment

Yes at source level: repeated `connect()` calls with `maxOAuthAttempts: 0` take the uncached path on current main, while cleanup only walks `this.clients`. I did not run a live repro because this review was required to stay read-only.

## Solution Assessment

Unclear until maintainers choose the public API shape. A separate cache-friendly OAuth-off option or mode is likely the narrow maintainable direction, but adding it is a new runtime option surface.
