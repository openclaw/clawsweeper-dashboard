## Summary

Keep this PR open: it addresses a real source-reproducible config-loading bug, but the submitted fix can drop a usable duplicate import because it catches normalization errors only after the first imported entry has already claimed the name.

## What This Changes

The PR changes `src/config.ts` so imported server normalization errors are caught and skipped while local server normalization errors still throw.

## Best Possible Solution

Keep the PR open and move the skip earlier in the import pipeline, or retry alternates, so invalid imported entries do not claim names before a valid normalized definition is chosen; local config should still fail fast.

## Reproduction Assessment

Yes, by source inspection. On current main, an imported entry containing an unresolved `${workspaceFolder}` in a non-deferred field reaches `normalizeServerEntry`, which throws during `loadServerDefinitions`; I did not run tests because this was a read-only review.

## Solution Assessment

No. The intended imported-server skip is reasonable, but catching after the first-wins merge means a skipped primary cannot fall through to a later valid duplicate, so the repair needs to normalize or retry before claiming the name.
