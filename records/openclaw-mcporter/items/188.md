## Summary

Keep open: the current daemon/listing path is still source-plausible for repeated keep-alive catalog enumeration to bypass cached reuse, and there is no current-main regression test proving repeated daemon `listTools` reuses or reaps a single stdio process.

## What This Changes

The issue reports that `lifecycle.mode: "keep-alive"` is accepted but repeated daemon catalog enumeration spawns unreused stdio child processes that accumulate until the host is memory pressured.

## Best Possible Solution

Add a current-main regression for repeated daemon `listTools`/catalog enumeration on a keep-alive stdio server, then make that path reuse the daemon-managed connection or reliably reap exactly one transient process without weakening non-interactive auth behavior.

## Reproduction Assessment

No, not with high confidence yet: the 0.7.3 process leak was not rerun against current main. Source inspection does show a plausible current-main path where daemon catalog/listTools requests keep `autoAuthorize: false`, which causes `Runtime.listTools` to skip cached reuse and close a fresh transport after each enumeration.

## Solution Assessment

Unclear until a focused regression is written, but the likely best fix is narrow: align daemon list/catalog handling with the documented keep-alive lifecycle while preserving non-interactive auth behavior for `mcporter list`. The newer serve bridge and cached-auth fixes are useful adjacent work, but they do not prove repeated catalog enumeration is fixed.
