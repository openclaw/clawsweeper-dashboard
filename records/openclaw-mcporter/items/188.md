## Summary

Keep open: current main has not proven the reported keep-alive catalog path is fixed; the CLI/daemon list path can still pass `autoAuthorize: false` into `Runtime.listTools`, which disables cached reuse, while existing tests cover adjacent daemon behavior but not repeated keep-alive catalog enumeration process reuse.

## What This Changes

The issue reports that `lifecycle: { mode: "keep-alive", idleTimeoutMs }` is accepted but repeated daemon catalog enumeration spawns unreused or unreaped stdio server processes.

## Best Possible Solution

Add a current-main regression for repeated daemon `listTools`/catalog enumeration on a keep-alive stdio server, then make that path reuse the daemon-managed connection or reliably reap exactly one transient process without weakening non-interactive auth behavior.

## Reproduction Assessment

No, not with high confidence from a live run; this read-only pass did not execute the reported daemon scenario against current main. Source inspection does show a plausible path where daemon/list requests keep `autoAuthorize: false`, causing `Runtime.listTools` to skip cached reuse and close a fresh transport after listing.

## Solution Assessment

Unclear until a focused regression pins the process behavior, but the likely best fix is narrow: align daemon list/catalog handling with the documented keep-alive lifecycle while preserving non-interactive auth behavior for `mcporter list`.
