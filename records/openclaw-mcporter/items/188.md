## Summary

Keep open: current main has substantial daemon keep-alive and idle-reaping code, plus newer serve-bridge improvements, but the reported catalog-enumeration failure is still source-plausible for non-interactive list paths and lacks a current-main regression test proving repeated keep-alive list enumeration reuses or reaps one process.

## What This Changes

The issue reports that `lifecycle.mode: "keep-alive"` is accepted but mcporter 0.7.3 still spawns unreaped stdio server processes instead of reusing one persistent process.

## Best Possible Solution

Make daemon-managed keep-alive catalog/listTools enumeration reuse the managed stdio process, or explicitly reap exactly one process, while preserving non-interactive auth behavior and idle-timeout cleanup.

## Reproduction Assessment

No, not yet with high confidence: the 0.7.3 process leak was not rerun against current main, but source inspection shows a plausible current non-interactive list path that bypasses keep-alive reuse. A focused current-main regression should exercise repeated daemon `listTools`/catalog enumeration for a keep-alive stdio server.

## Solution Assessment

Unclear until that regression is written, but the likely best fix is narrow: align daemon list/catalog handling with the documented keep-alive lifecycle without changing the user-facing auth policy for `mcporter list`. The newer serve bridge is useful but is not enough evidence to close this issue for every catalog enumeration path.
