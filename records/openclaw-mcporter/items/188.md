## Summary

Keep open: the report matches a documented keep-alive daemon contract, and current source still has a plausible daemon catalog/listTools path that can bypass cached reuse when `autoAuthorize: false` is forwarded into `Runtime.listTools`. The exact 0.7.3 process leak has not been rerun against current `main`, so this should become a focused regression/fix rather than a cleanup close.

## What This Changes

The issue reports that a configured `lifecycle: { mode: "keep-alive", idleTimeoutMs }` stdio server is accepted but repeated daemon catalog enumeration spawns unreused, unreaped child processes.

## Best Possible Solution

Add a current-main regression for repeated daemon `listTools` or catalog enumeration on a keep-alive stdio server, then make that path either reuse the daemon-managed connection or reliably reap exactly one transient process while preserving non-interactive auth behavior.

## Reproduction Assessment

No, not with high confidence: I did not run the 0.7.3 or current-main daemon leak scenario. Source inspection does show a current-main path where daemon listTools can preserve `autoAuthorize: false`, causing `Runtime.listTools` to skip cached reuse and close a fresh transport after each enumeration.

## Solution Assessment

Unclear until a focused regression is written, but the likely best fix is narrow: align daemon list/catalog handling with the documented keep-alive lifecycle without weakening non-interactive OAuth behavior for normal `mcporter list` usage.
