- `impact:data-loss`: The reported failure silently drops the user's configured mcporter server registry from discovery when XDG points elsewhere.

## Summary

Keep open: current main still makes an absolute `XDG_CONFIG_HOME` the only implicit home-config root, so a populated legacy `~/.mcporter/mcporter.json` is ignored and the registry can become empty with only `Unknown MCP server` as the call-time symptom. No mcporter PR or canonical issue currently owns the fix, and the compatible behavior needs a maintainer choice between fallback, a dedicated mcporter env var, and fail-loud diagnostics.

## What This Changes

Make mcporter avoid or clearly diagnose an empty server registry when an embedder sets `XDG_CONFIG_HOME` for an unrelated downstream tool while the user's legacy `~/.mcporter` config exists.

## Best Possible Solution

Adopt a maintainer-approved compatibility policy for mcporter's own config discovery, then add a focused regression test for the unrelated-embedder `XDG_CONFIG_HOME` case plus docs or diagnostics that make an ignored legacy registry impossible to miss.

## Reproduction Assessment

Yes by source inspection: set an absolute `XDG_CONFIG_HOME` with no `mcporter/` entry while the server exists only in legacy `~/.mcporter`, and current discovery will not probe the legacy candidates before constructing an empty registry. I did not execute the CLI because this review is read-only, but the path from `mcporterDir()` through `loadConfigLayers()` to the runtime error is direct.

## Solution Assessment

Unclear: the report offers several plausible fixes, but fallback, a dedicated mcporter home env var, and warning-only diagnostics make different compatibility tradeoffs. Maintainers should choose the policy first, then the implementation can stay narrow.
