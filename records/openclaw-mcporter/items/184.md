## Label Justifications

- `P1`: The report describes a recent compatibility regression that can silently break a real embedder's MCP-backed memory search workflow.
- `impact:data-loss`: The issue is about mcporter silently ignoring an existing user config registry and loading an empty server set from the caller's XDG environment.

## Summary

Keep open: current main still treats an absolute `XDG_CONFIG_HOME` as mcporter's only home config root, so an embedder-owned XDG directory with no `mcporter/` config can leave the runtime with an empty server registry and only the downstream `Unknown MCP server` error. The older XDG support work at https://github.com/openclaw/mcporter/issues/155 explains how this behavior arrived, but it does not cover the compatibility guard or fail-loud diagnostic requested here.

## What This Changes

The issue requests a safer config-discovery boundary so embedders can set `XDG_CONFIG_HOME` for another tool without silently hiding mcporter's populated legacy registry.

## Best Possible Solution

Keep XDG support, then add a tested compatibility guard that makes the empty-XDG-config plus populated-legacy-config case impossible to miss, with maintainer choice on whether that guard should be fallback, dedicated mcporter path override, or fail-loud diagnostic.

## Reproduction Assessment

Yes, at source level: set an absolute `XDG_CONFIG_HOME` with no `mcporter/mcporter.json[c]` while `~/.mcporter/mcporter.json` contains the server, and current discovery never considers the legacy candidate before the runtime throws `Unknown MCP server`. I did not run a live `qmd` server in this read-only sweep.

## Solution Assessment

Unclear: the report identifies a real compatibility bug, but the best fix shape is a maintainer product choice between fallback semantics, a new mcporter-specific path override, and a diagnostic-only fail-loud path.
