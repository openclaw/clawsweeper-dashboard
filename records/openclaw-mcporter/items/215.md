## Summary

Keep open: this PR targets the right active bug, but it is not merge-ready because the new stdout-flush callback can wait without a cap if stdout never drains, and the PR still lacks real pipe-output proof.

## What This Changes

The PR changes `closeRuntimeAfterCommand()` so the forced-exit timer waits for an empty `process.stdout.write` callback before calling `process.exit`.

## Best Possible Solution

Land a bounded stdout-flush fix with real terminal proof and regression coverage for the greater-than-64KB pipe case while preserving the anti-hang cap and exit-code behavior.

## Reproduction Assessment

Yes. The linked issue gives a concrete greater-than-64KB pipe repro, and current main still emits large JSON before a timed hard exit with no drain handling; this read-only review did not execute the repro.

## Solution Assessment

Not yet. Waiting for stdout before exit is the right narrow direction, but the current PR should bound that wait and prove the 64KB pipe behavior with regression coverage before merge.
