- `merge-risk: 🚨 availability`: The diff changes forced-exit shutdown timing and can expose an unhandled stdout stream error during cleanup.
- `merge-risk: 🚨 compatibility`: Broken-pipe or early-closing stdout consumers are existing shell-pipeline setups that could see a new cleanup-time failure.

## Summary

Keep open: the PR now has direct proof, focused regression coverage, and a bounded stdout flush, but the cleanup-time empty write still needs error handling so broken stdout pipes cannot turn a successful command into an unhandled stream error.

## What This Changes

This PR makes `closeRuntimeAfterCommand()` wait for an empty stdout write with a 2s fallback before forced exit and adds POSIX integration coverage for delayed-pipe output and stalled readers.

## Best Possible Solution

Land one bounded, error-tolerant forced-exit flush implementation for stdout, ideally including stderr too, while preserving this PR's direct >64KB pipe proof and resolving the overlap with https://github.com/openclaw/mcporter/pull/216.

## Reproduction Assessment

Yes, source-reproducible: current main writes large JSON through stdout and then schedules a hard exit after 50ms without drain handling. The PR body also supplies before/after byte counts for a >64KB delayed pipe; I did not run the live shell repro in this read-only review.

## Solution Assessment

Not quite. Waiting for stdout with a bounded fallback is the right narrow fix, but the cleanup write should treat stdout errors/EPIPE as completion and then exit, or maintainers should land the overlapping error-tolerant flush from https://github.com/openclaw/mcporter/pull/216.
