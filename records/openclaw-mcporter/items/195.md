## Summary

Keep this PR open for maintainer review: current main still lets direct daemon host starts rebind the socket, while this patch adds a narrow host-side guard with status verification, metadata matching, and focused tests. I did not find a blocking correctness defect, but the change intentionally alters daemon startup arbitration and should be accepted by a maintainer as an availability/compatibility tradeoff.

## What This Changes

The PR adds a daemon host bind lock, live status probe, metadata-match guard, and daemon host/integration tests to prevent duplicate direct starts from orphaning a running daemon.

## Best Possible Solution

Merge the host-side arbitration fix if maintainers accept the startup behavior change, keeping the client-side lifecycle reconciliation on main and the new host-side tests together.

## Reproduction Assessment

Yes. Source inspection shows direct foreground starts call runDaemonHost outside the DaemonClient lock, and the contributor provided main-vs-branch macOS race counts; this read-only review did not run the race locally.

## Solution Assessment

Yes. The PR is a narrow host-side complement to the existing client lock and now verifies a daemon status response plus matching metadata before skipping the bind, while preserving hung or stale socket recovery.
