## Summary

Keep open: the status probe now handles hung, dead, and foreign listeners, but the early-exit guard can still strand client-launched startup when a live daemon owns the socket and the metadata file is missing or mismatched.

## What This Changes

The PR adds a host-side daemon bind lock/status probe in `runDaemonHost` and covers duplicate foreground-start races with daemon host and integration tests.

## Best Possible Solution

Keep the host-side bind lock, but make the early-exit path use the same metadata, pid, and socket invariant as `DaemonClient` or repair metadata before returning, with regression coverage for missing and mismatched metadata.

## Reproduction Assessment

Yes from source inspection and the PR's manual repro notes for the duplicate foreground-start race. The remaining blocker is also source-reproducible: client readiness requires verified metadata, while the new host guard can exit without writing or repairing it.

## Solution Assessment

No, not yet. The host-side guard is the right layer for direct daemon starts, but it needs to preserve the existing metadata verification contract before it is the narrow maintainable fix.
