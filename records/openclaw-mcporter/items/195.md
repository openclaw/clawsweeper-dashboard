## Summary

Keep open: the force-pushed PR fixes the previous connect-only liveness issue and includes useful live daemon proof, but the new host guard can still exit while leaving `DaemonClient` unable to verify metadata, so it needs one narrow repair before merge.

## Best Possible Solution

Keep the host-side singleton guard, but make the early-exit path either verify matching metadata or atomically repair it before the child exits, with regression coverage for missing or mismatched metadata.

## Solution Assessment

No, not quite. The host-side bind guard is the right repair boundary, but it must preserve or repair the metadata/PID verification contract that current `DaemonClient` requires before skipping the bind.
