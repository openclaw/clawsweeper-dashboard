## Summary

Keep open: this PR addresses a real remaining daemon singleton gap, but the host guard exits on any socket listener instead of a verified daemon status response, which can block recovery from stale, hung, or non-daemon sockets.

## What This Changes

The PR adds a host-side daemon socket bind guard, a foreground-start precheck, and race regression coverage to prevent duplicate daemon processes.

## Best Possible Solution

Keep the host-side singleton guard, but make the early exit depend on verified daemon status and matching PID/socket metadata; unverified listeners should fall through to the existing rebind recovery path.

## Reproduction Assessment

Yes, the duplicate foreground-start race has a high-confidence source path and the PR body reports real macOS survivor counts; I did not run the macOS race in this read-only review. The new blocker is also source-reproducible because an accepting socket is treated as live without the existing verified-status checks.

## Solution Assessment

No as written: the host-side guard is the right layer, but accepting a socket connection is weaker than the current daemon liveness contract. The narrow fix is to verify daemon status and metadata before exiting early.
