## Summary

Keep open: current main has the earlier 50 ms forced-exit grace window, but it still hard-exits without any stdout/stderr drain handling, so the refined pipe-truncation bug remains actionable.

## What This Changes

The issue asks MCPorter to stop truncating large CLI stdout output when stdout is piped by flushing or draining output before the forced exit.

## Best Possible Solution

Implement a bounded stdout/stderr flush or drain step in the forced-exit shutdown path, then strengthen regression coverage with a payload over 64 KiB through a pipe.

## Reproduction Assessment

Yes. The issue gives a concrete pipe-based repro over 64 KiB, and current source still emits large output through stdout before a timed hard `process.exit()` with no drain handling; I did not run it in this read-only review.

## Solution Assessment

Yes. A bounded drain/flush before the forced exit is narrower and safer than disabling forced exit or adding a new config knob, as long as the timeout preserves the existing anti-hang guarantee.
