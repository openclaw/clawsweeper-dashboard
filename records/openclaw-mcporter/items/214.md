## Summary

Keep open: current main still emits large CLI output through stdout and then schedules a hard process exit after runtime cleanup without waiting for stdio to drain. Two open pull requests are candidate fixes, so this issue should remain the canonical tracker until one lands or maintainers choose a different repair.

## What This Changes

The issue reports silent truncation of large CLI stdout sent through a pipe because forced process exit can run before stdout drains.

## Best Possible Solution

Review and land one bounded stdout/stderr flush before the forced-exit fallback, with regression coverage for large piped output, stalled readers, and broken-pipe behavior.

## Reproduction Assessment

Yes. The report gives concrete pipe-versus-redirect commands, current source still has one-shot stdout output followed by a timed hard exit with no drain handling, and the linked contributor PR includes before/after byte-count proof.

## Solution Assessment

Yes. A bounded stdout/stderr flush before forced exit is the narrow maintainable fix, provided the anti-hang fallback and focused regression coverage remain in the chosen landing branch.
