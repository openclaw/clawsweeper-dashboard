## Summary

Keep open: current main still emits large CLI output in one shot and then schedules a 50 ms hard `process.exit` without waiting for stdio to drain. The issue is already paired with open candidate fixes at https://github.com/openclaw/mcporter/pull/215 and https://github.com/openclaw/mcporter/pull/216, so it should close only after one implementation lands.

## What This Changes

Ensure large CLI stdout written to a pipe drains before the forced-exit fallback so machine-readable output is not silently truncated.

## Best Possible Solution

Land one verified bounded stdout/stderr drain implementation with delayed-pipe, stalled-reader, and broken-pipe regression coverage, then let the linked PR close this issue.

## Reproduction Assessment

Yes: the report gives pipe-versus-file commands for output above 64 KiB, and current source still writes via `console.log` before a 50 ms hard exit without stdio drain handling. I did not run a live pipe reproduction during this read-only review.

## Solution Assessment

Yes: a bounded stdout/stderr flush before forced exit is the narrow maintainable fix while preserving the anti-hang fallback. The remaining choice is which open implementation and timeout behavior maintainers want to land.
