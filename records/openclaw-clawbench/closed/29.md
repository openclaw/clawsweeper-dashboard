## Label Justifications

- `P2`: This is a concrete execution-check correctness and hardening bug with limited benchmark-surface blast radius and no reported production outage.
- `impact:security`: Unquoted runtime values in shell-mode commands can be interpreted as shell syntax when values contain metacharacters.
- `impact:other`: Whitespace in runtime paths can make deterministic execution checks fail even when the agent produced the expected artifact.

## Summary

Keep open. Current main still renders shell-mode execution-check commands with plain placeholder substitution and passes the result to `create_subprocess_shell`; the related fix at https://github.com/openclaw/clawbench/issues/28 only covered the `shell=False` argv path.

## What This Changes

The issue asks ClawBench to make `shell=True` execution checks handle runtime placeholder values with whitespace or shell metacharacters safely.

## Best Possible Solution

Keep this open and implement a compatibility-conscious shell-mode templating fix with regression coverage for whitespace and metacharacter values in both verifier implementations.

## Reproduction Assessment

Yes, at source level: current main renders placeholders with `str(value)` and sends the raw command to `/bin/sh -c` for default `shell=True` checks. A direct shell check also shows `cat report 2026.json` splits the filename and fails.

## Solution Assessment

Unclear. Quoting substituted shell values is likely the right direction, but maintainers should decide how to handle existing templates that already quote placeholders or intentionally rely on shell expansion.
